## Overview

### What is EC2? When Do I Use It?

> **Real-World Analogy:** EC2 is like renting a computer in the cloud. Instead of buying a physical server, plugging it in, and maintaining it, you click a button and get a virtual machine in seconds. It's like renting an apartment vs buying a house — flexibility without the ownership burden.

**Why does this matter?** EC2 is the most fundamental AWS service. Even if you end up using serverless (Lambda) or containers (ECS/EKS), understanding EC2 is essential because many other services run on EC2 under the hood. It's also the service where you'll spend the most money — choosing the right instance type and purchasing option can save 60-90% on your bill.

**Quick decision: Do I need EC2?**
- Need full OS access, custom software, GPU? → **EC2**
- Simple web app, just deploy code? → App Runner or Beanstalk
- Short-running code triggered by events? → Lambda
- Running containers? → ECS/EKS (may still use EC2 underneath)

Amazon EC2 (Elastic Compute Cloud) is AWS's virtual machine service. It's the backbone of AWS compute — every time you need a server (web server, database, CI/CD runner, dev machine), you launch an EC2 instance. This chapter covers everything about creating, configuring, and managing EC2 instances.

## Part 5: Instance Metadata Service (IMDS)

```
┌─────────────────────────────────────────────────────────────────────┐
│           INSTANCE METADATA (IMDS)                                    │
├─────────────────────────────────────────────────────────────────────┤
│                                                                       │
│ What: HTTP endpoint available FROM within the instance at          │
│ http://169.254.169.254/latest/meta-data/                           │
│                                                                       │
│ Provides: Instance info without needing API calls                  │
│ ├── Instance ID, type, AZ, region                                 │
│ ├── Public/private IP                                              │
│ ├── IAM role credentials (temporary!)                             │
│ ├── Security groups                                                │

│ └── MAC address, network interfaces                               │
│                                                                       │
│ IMDSv1 (legacy — insecure):                                        │
│ curl http://169.254.169.254/latest/meta-data/instance-id           │
│ → i-0abc123def456                                                  │
│                                                                       │
│ IMDSv2 (required — secure):                                        │
│ # Get token first                                                   │
│ TOKEN=$(curl -X PUT \                                               │
│   "http://169.254.169.254/latest/api/token" \                      │
│   -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")               │
│                                                                       │
│ # Use token for requests                                            │
│ curl -H "X-aws-ec2-metadata-token: $TOKEN" \                      │
│   http://169.254.169.254/latest/meta-data/instance-id              │
│                                                                       │
│ Why IMDSv2:                                                          │
│ ├── Prevents SSRF attacks (Server-Side Request Forgery)          │
│ ├── Token required (PUT request — can't be triggered by SSRF)   │
│ ├── Token has TTL (expires)                                       │
│ └── ⚠️ ALWAYS use IMDSv2 only. Block v1.                          │
│                                                                       │
│ Common metadata paths:                                               │
│ ├── /meta-data/instance-id: i-0abc123def456                      │
│ ├── /meta-data/instance-type: m7g.large                           │
│ ├── /meta-data/ami-id: ami-0abc123456                            │
│ ├── /meta-data/local-ipv4: 10.0.1.23                             │
│ ├── /meta-data/public-ipv4: 54.123.45.67                        │
│ ├── /meta-data/placement/availability-zone: ap-south-1a          │
│ ├── /meta-data/iam/security-credentials/role-name: temp creds   │
│ └── /user-data: Your bootstrap script                             │
│                                                                       │
└─────────────────────────────────────────────────────────────────────┘

## Part 6: EBS Volumes

```
┌─────────────────────────────────────────────────────────────────────┐
│           EBS (ELASTIC BLOCK STORE)                                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                       │
│ What: Network-attached block storage for EC2 instances.            │
│ Think of it as a virtual hard drive.                               │
│                                                                       │
│ Key facts:                                                           │
│ ├── Network attached (not physically on the host)                 │
│ ├── Persists independently from EC2 (can outlive instance)       │
│ ├── One EBS volume → one EC2 instance (usually)                  │
│ │   (Multi-attach io2 supports multiple instances)               │
│ ├── Bound to ONE AZ (can't move, must snapshot + restore)       │
│ ├── Can be detached and re-attached to another EC2 (same AZ)    │
│ ├── Can resize on the fly (grow, never shrink)                   │
│ └── Should ALWAYS be encrypted                                    │
│                                                                       │
│ Volume types:                                                        │
│ ┌──────────┬──────────┬──────────┬───────────┬───────────────────┐│
│ │ Type     │ IOPS     │ Thruput  │ Cost/GB/mo│ Use case          ││
│ ├──────────┼──────────┼──────────┼───────────┼───────────────────┤│
│ │ gp3      │ 3000     │ 125 MB/s │ $0.08     │ Most workloads   ││
│ │ (SSD)    │ (free)   │ (free)   │           │ ⚡ DEFAULT         ││
│ │          │ up to    │ up to    │           │                   ││
│ │          │ 16,000   │ 1000 MB/s│           │                   ││
│ │          │          │          │           │                   ││
│ │ gp2      │ Burstable│ 250 MB/s │ $0.10     │ Legacy, use gp3  ││
│ │ (SSD)    │ 3/GB     │          │           │                   ││
│ │          │ max 16K  │          │           │                   ││
│ │          │          │          │           │                   ││
│ │ io2      │ Up to    │ Up to    │ $0.125 +  │ High-perf DB     ││
│ │ (SSD)    │ 64,000   │ 1000 MB/s│ $0.065/   │ (MySQL, Postgres,││
│ │          │          │          │ IOPS      │ Oracle, SAP)      ││
│ │          │          │          │           │                   ││
│ │ io2 BE   │ Up to    │ Up to    │ $0.125 +  │ Largest databases││
│ │ (SSD)    │ 256,000  │ 4000 MB/s│ $0.065/   │ SAP HANA          ││
│ │          │          │          │ IOPS      │                   ││
│ │          │          │          │           │                   ││
│ │ st1      │ 500 MB/s │ 500 MB/s │ $0.045    │ Big data, logs   ││
│ │ (HDD)    │ thruput  │          │           │ (sequential I/O) ││
│ │          │          │          │           │                   ││
│ │ sc1      │ 250 MB/s │ 250 MB/s │ $0.015    │ Cold storage,    ││
│ │ (HDD)    │ thruput  │          │           │ infrequent access││
│ └──────────┴──────────┴──────────┴───────────┴───────────────────┘│
│                                                                       │
│ ⚡ 99% of the time, use gp3. It's cheaper and faster than gp2.     │
│    Use io2 only for production databases needing consistent IOPS.  │
│                                                                       │
│ EBS Snapshots:                                                       │
│ ├── Point-in-time backup of an EBS volume                         │
│ ├── Stored in S3 (managed by AWS, you don't see the bucket)     │
│ ├── Incremental (only changed blocks saved after first snapshot) │
│ ├── Can create volume from snapshot (any AZ in same region)      │
│ ├── Can copy snapshot to another region (DR)                     │
│ ├── Can share snapshot with other accounts                        │
│ └── Automate with AWS Backup or DLM (Data Lifecycle Manager)    │
│                                                                       │
│ Create snapshot:                                                     │
│ EC2 → Volumes → Select → Actions → Create snapshot                │
│   Description: [prod-web-data-2026-05-16]                          │
│   Tags: backup-type=daily, retention=30d                           │
│                                                                       │
│ Encryption:                                                          │
│ ├── Encrypted volumes → encrypted snapshots                      │
│ ├── Unencrypted → encrypted: Create snapshot → copy with        │
│ │   encryption → create volume from encrypted snapshot           │
│ ├── Default encryption: EC2 → Settings → EBS encryption →      │
│ │   "Always encrypt new EBS volumes" ☑                           │
│ └── ⚡ ENABLE default encryption account-wide!                    │
│                                                                       │
└─────────────────────────────────────────────────────────────────────┘
```

## Part 9: Instance Lifecycle

```
┌─────────────────────────────────────────────────────────────────────┐
│           INSTANCE LIFECYCLE                                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                       │
│ States:                                                              │
│                                                                       │
│ pending → running ←→ stopping → stopped                            │
│             │                      │                                  │
│             │                      └── Start → pending → running   │
│             │                                                        │
│             └── Terminate → shutting-down → terminated              │
│                                                                       │
│ ┌──────────────┬───────────────────────────────────────────────┐    │
│ │ Action       │ What happens                                  │    │
│ ├──────────────┼───────────────────────────────────────────────┤    │
│ │ Stop         │ Instance halts. EBS volumes preserved.       │    │
│ │              │ Public IP changes on restart!                 │    │
│ │              │ No charges for compute (EBS still charged).  │    │
│ │              │ Instance may move to different physical host.│    │
│ │              │                                               │    │
│ │ Start        │ Boots up. New public IP (unless Elastic IP). │    │
│ │              │ User data does NOT re-run.                   │    │
│ │              │                                               │    │
│ │ Reboot       │ Same as OS reboot. Keeps same host.         │    │
│ │              │ Keeps public IP. Instance store preserved.   │    │
│ │              │                                               │    │
│ │ Terminate    │ Instance + root EBS deleted (if set).        │    │
│ │              │ ⚠️ IRREVERSIBLE!                               │    │
│ │              │ Enable termination protection for production.│    │
│ │              │                                               │    │
│ │ Hibernate    │ RAM contents saved to EBS, then stopped.     │    │
│ │              │ On start: RAM restored, fast resume.         │    │
│ │              │ Root EBS must be encrypted.                   │    │
│ │              │ Root EBS must be large enough for RAM.       │    │
│ │              │ Limited instance types/sizes.                │    │
│ └──────────────┴───────────────────────────────────────────────┘    │
│                                                                       │
│ Termination protection:                                              │
│ EC2 → Instance → Actions → Instance settings →                    │
│ Change termination protection → ☑ Enable                           │
│ (Prevents accidental termination via console/API)                  │
│                                                                       │
│ Stop protection:                                                     │
│ Similar — prevents accidental stop.                                 │
│ (Enable for always-on production servers)                          │
│                                                                       │
│ Elastic IP:                                                          │
│ ├── Static public IPv4 address that doesn't change               │
│ ├── Associate with instance or ENI                                │
│ ├── Persists through stop/start                                   │
│ ├── Free while associated with running instance                  │
│ ├── ⚠️ Charged when NOT associated ($0.005/hr) — release unused   │
│ ├── Limited to 5 per region (request increase)                   │
│ └── ⚡ Better: Use ALB/NLB DNS name instead of Elastic IP         │
│                                                                       │
└─────────────────────────────────────────────────────────────────────┘

## Troubleshooting: Common EC2 Issues

### "I can't SSH/RDP into my instance"

```
Checklist:
1. ☐ Is the instance running? (not stopped or terminated)
   Console → EC2 → Instances → Instance state

2. ☐ Does the instance have a public IP?
   Check: Public IPv4 address column
   (Instances in private subnets won't have one — use a bastion or SSM)

3. ☐ Does the Security Group allow inbound SSH (22) or RDP (3389)?
   Source should be YOUR IP, not 0.0.0.0/0

4. ☐ Are you using the correct key pair?
   Error: "Permission denied (publickey)" = wrong key or wrong username
   Default usernames: ec2-user (Amazon Linux), ubuntu (Ubuntu), admin (Debian)

5. ☐ Are the key file permissions correct?
   Linux/Mac: chmod 400 my-key.pem
   Windows: Properties → Security → Remove all users except you

6. ☐ Does the subnet route table have 0.0.0.0/0 → IGW?
```

### "My instance is running but my app isn't working"

```
1. ☐ Did User Data script run successfully?
   Connect via SSM/SSH and check: cat /var/log/cloud-init-output.log

2. ☐ Is the application actually running?
   systemctl status nginx (or your app service)

3. ☐ Is the Security Group allowing traffic on the app port?
   (e.g., port 80, 443, 3000, 8080)
```

### Common Mistakes

| Mistake | Impact | Fix |
|---------|--------|-----|
| Losing key pair file | Can't SSH ever again | Use SSM Session Manager (no key needed) |
| Not stopping dev instances | Paying 24/7 for idle servers | Use a schedule or Lambda to stop at night |
| Wrong instance type for workload | Poor performance or wasted money | Start with t3.micro, upgrade based on CloudWatch |
| EBS volume left after termination | Paying for orphaned storage | Enable "Delete on Termination" |
| User Data not working | App not installed | Check cloud-init logs, ensure script is bash-compatible |
