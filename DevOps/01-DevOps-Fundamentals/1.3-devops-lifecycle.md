# 1.3 — DevOps Lifecycle — Plan, Code, Build, Test, Release, Deploy, Operate, Monitor

> **One Line Answer:** The DevOps Lifecycle is an **infinite loop** of 8 phases that continuously deliver software from idea to production and back again — **Plan → Code → Build → Test → Release → Deploy → Operate → Monitor → (repeat forever)**.

---

## 🎯 Why You Need to Know This

The DevOps lifecycle is the **backbone** of everything you'll learn in DevOps. Every tool, every practice, every concept fits into one of these phases. When someone says "CI/CD" — that's Build + Test + Release + Deploy. When someone says "IaC" — that's part of Deploy + Operate.

Understanding this lifecycle means you'll always know **where** a tool or practice fits in the big picture.

---

## ♾️ The Infinite Loop

The most important thing about the DevOps lifecycle is that it's an **infinity loop (∞)**, not a straight line. It never ends. After monitoring, you go back to planning. Always.

```
              ┌─────────────────────────────────────────┐
              │           DEVOPS LIFECYCLE               │
              │                                         │
              │         ∞ INFINITY LOOP ∞               │
              │                                         │
              │    Plan → Code → Build → Test           │
              │      ↑                    ↓             │
              │    Monitor ← Operate ← Deploy ← Release │
              │                                         │
              └─────────────────────────────────────────┘
```

This loop is also divided into **two big halves**:

```
        LEFT SIDE (Dev)              RIGHT SIDE (Ops)
   ┌──────────────────────┐    ┌──────────────────────┐
   │  Plan                │    │  Release              │
   │  Code                │    │  Deploy               │
   │  Build               │    │  Operate              │
   │  Test                │    │  Monitor              │
   └──────────────────────┘    └──────────────────────┘
   
   Continuous Integration        Continuous Delivery/
                                 Deployment
```

Now let's deep dive into each phase 👇

## Phase 1: 📋 PLAN

### What Happens Here?

This is where **everything starts**. Before writing a single line of code, you figure out:

- **What** to build (features, bug fixes, improvements)
- **Why** to build it (business value, customer feedback)
- **Who** will build it (team assignments)
- **When** it should be done (timelines, sprint planning)
- **How** to build it (architecture decisions, tech choices)

### Activities in This Phase:

```
✅ Gather requirements from stakeholders and customers
✅ Write user stories / feature requests
✅ Create and prioritize the product backlog
✅ Sprint planning (if using Agile/Scrum)
✅ Architecture and design discussions
✅ Estimate effort and plan sprints
✅ Review monitoring data and customer feedback from the previous cycle
✅ Plan infrastructure changes needed
```

### Tools Used:

| Tool | Purpose |
|------|---------|
| **Jira** | Issue tracking, sprint planning, backlog management |
| **Azure Boards** | Microsoft's project management tool |
| **Trello** | Kanban boards for task management |
| **Confluence** | Documentation, decision records |
| **GitHub Issues / Projects** | Issue tracking integrated with code |
| **GitLab Issues / Boards** | Same but in GitLab ecosystem |
| **Slack / Microsoft Teams** | Team communication |
| **Miro / Lucidchart** | Architecture diagrams, brainstorming |

### Real-World Example:

```
Sprint Planning Meeting:

Product Owner: "Customers are complaining that checkout is slow. 
                Let's optimize it this sprint."

Dev Team: "We'll need to:
           1. Profile the checkout API endpoint
           2. Add caching for product prices
           3. Optimize the database query
           Let's estimate: 8 story points"

Ops/DevOps: "We'll also need to:
              1. Set up performance monitoring for checkout
              2. Update the auto-scaling rules
              Let's estimate: 3 story points"

Total sprint capacity: 40 points. This takes 11. We have room for more.
```

### Key Takeaway:

> In DevOps, planning **includes Ops concerns** from the start. How will this be deployed? How will we monitor it? What infrastructure changes are needed? This is NOT just a developer planning meeting.

## Phase 2: 💻 CODE

### What Happens Here?

Developers **write the actual code** for the features, bug fixes, or improvements planned in Phase 1. This also includes writing Infrastructure as Code (IaC) and configuration files.

### Activities in This Phase:

```
✅ Write application code
✅ Write unit tests alongside the code (TDD)
✅ Write Infrastructure as Code (Terraform, Ansible, CloudFormation)
✅ Write Dockerfiles and Kubernetes manifests
✅ Write CI/CD pipeline configurations
✅ Code reviews via Pull Requests / Merge Requests
✅ Follow coding standards and best practices
✅ Commit code to version control (Git)
```

### Tools Used:

| Tool | Purpose |
|------|---------|
| **VS Code / IntelliJ / Vim** | Code editors/IDEs |
| **Git** | Version control |
| **GitHub / GitLab / Bitbucket** | Remote repositories + code review |
| **Pre-commit hooks** | Automated checks before committing |
| **Linters (ESLint, pylint, etc.)** | Code quality checks |
| **SonarQube** | Code quality + security scanning |

### Real-World Example — The Code Phase in Action:

```bash
# Developer creates a new feature branch
git checkout -b feature/optimize-checkout

# Writes code, tests, and infrastructure changes
vim src/checkout/service.py          # Application code
vim tests/test_checkout.py           # Unit tests
vim terraform/cache.tf               # Infrastructure (Redis cache)
vim k8s/checkout-deployment.yaml     # Kubernetes manifest
vim Dockerfile                       # Container definition
vim .github/workflows/ci.yml         # CI/CD pipeline

# Runs tests locally
python -m pytest tests/

# Commits with a meaningful message
git add .
git commit -m "feat: add Redis cache to checkout for 3x faster response"

# Pushes and creates a Pull Request
git push origin feature/optimize-checkout
# → Opens a Pull Request on GitHub for team review
```

### The Pull Request (Code Review) Flow:

```
Developer pushes code
       ↓
Creates Pull Request (PR)
       ↓
Automated checks run (lint, tests, security scan)
       ↓
Team members review the code
       ↓
Feedback → Developer makes changes → Re-review
       ↓
PR approved ✅
       ↓
Merge to main branch → Triggers BUILD phase
```

### Key Takeaway:

> In DevOps, "code" isn't just application code. It includes **infrastructure code** (Terraform), **pipeline code** (Jenkinsfile, GitHub Actions), **container definitions** (Dockerfile), and **configuration** (K8s manifests). EVERYTHING is code. Everything is in Git.

## Phase 3: 🔨 BUILD

### What Happens Here?

The source code is **compiled, packaged, and turned into a deployable artifact**. Think of it as: raw ingredients (code) → finished product (artifact).

### What Is an "Artifact"?

An artifact is the **output of a build** — the thing you actually deploy.

| Language/Tech | What Gets Built (Artifact) |
|--------------|---------------------------|
| Java | `.jar` or `.war` file |
| Node.js | `node_modules` + bundled JS |
| Python | `.whl` file or packaged app |
| Go | Binary executable |
| Docker | Docker Image |
| Frontend | Minified HTML/CSS/JS bundle |

### Activities in This Phase:

```
✅ Compile source code (if compiled language)
✅ Resolve and download dependencies
✅ Run linters and static analysis
✅ Build Docker images
✅ Tag artifacts with version numbers
✅ Store artifacts in a registry/repository
```

### Tools Used:

| Tool | Purpose |
|------|---------|
| **Maven / Gradle** | Build Java applications |
| **npm / yarn** | Build Node.js applications |
| **pip / poetry** | Build Python applications |
| **Docker** | Build container images |
| **Jenkins / GitHub Actions / GitLab CI** | Orchestrate the build process |
| **Docker Hub / ECR / ACR / Harbor** | Store built Docker images |
| **JFrog Artifactory / Nexus** | Store all types of artifacts |

### Real-World Example — Automated Build:

Here's what a build looks like in a **GitHub Actions** CI pipeline:

```yaml
# .github/workflows/build.yml
name: Build Application

on:
  push:
    branches: [main]     # Trigger on push to main branch
  pull_request:
    branches: [main]     # Trigger on PR to main branch

jobs:
  build:
    runs-on: ubuntu-latest   # Use a Linux machine
    
    steps:
      # Step 1: Get the code
      - name: Checkout code
        uses: actions/checkout@v4
      
      # Step 2: Set up Python
      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      
      # Step 3: Install dependencies
      - name: Install dependencies
        run: pip install -r requirements.txt
      
      # Step 4: Run linting (code quality check)
      - name: Lint code
        run: pylint src/
      
      # Step 5: Build Docker image
      - name: Build Docker image
        run: docker build -t myapp:${{ github.sha }} .
      
      # Step 6: Push to registry
      - name: Push to Docker Hub
        run: docker push myapp:${{ github.sha }}
```

**What this does:**
- Every time someone pushes code or creates a PR → this runs automatically
- It checks out the code, installs dependencies, checks code quality, builds a Docker image, and pushes it to a registry
- All without any human pressing a button!

### Key Takeaway:

> The build phase **converts code into a deployable artifact** and is **fully automated** in DevOps. No human should be running build commands manually. Push code → build happens automatically.

## Phase 4: 🧪 TEST

### What Happens Here?

The built artifact is **tested automatically** to make sure it works correctly, performs well, and is secure. In DevOps, testing is **automated and continuous** — not a separate phase at the end.

### Types of Tests (The Testing Pyramid):

```
                    ╱╲
                   ╱  ╲
                  ╱ E2E╲         ← Few, slow, expensive
                 ╱──────╲          (End-to-End / UI tests)
                ╱        ╲
               ╱Integration╲     ← Medium amount
              ╱──────────────╲     (API tests, service tests)
             ╱                ╲
            ╱   Unit Tests     ╲  ← Many, fast, cheap
           ╱────────────────────╲   (Test individual functions)
```

### Each Type Explained:

| Type | What It Tests | Speed | Example |
|------|--------------|-------|---------|
| **Unit Tests** | Individual functions/methods | Very fast (ms) | "Does the `calculateTotal()` function return the right value?" |
| **Integration Tests** | Multiple components together | Medium (seconds) | "Does the checkout API correctly talk to the database?" |
| **End-to-End (E2E)** | Entire user workflow | Slow (minutes) | "Can a user add item to cart, checkout, and receive confirmation?" |
| **Performance Tests** | Speed and load handling | Varies | "Can the system handle 10,000 users at once?" |
| **Security Tests** | Vulnerabilities | Medium | "Are there any SQL injection vulnerabilities?" |
| **Smoke Tests** | Basic functionality | Fast | "Does the app start? Does the homepage load?" |

### Tools Used:

| Tool | Purpose |
|------|---------|
| **pytest / JUnit / Jest** | Unit & integration testing |
| **Selenium / Cypress / Playwright** | End-to-end / UI testing |
| **JMeter / k6 / Locust** | Performance / load testing |
| **SonarQube** | Code quality & static analysis |
| **Trivy / Snyk** | Security vulnerability scanning |
| **OWASP ZAP** | Dynamic security testing |

### Real-World Example — Automated Testing in Pipeline:

```yaml
# Continuing our GitHub Actions pipeline
  test:
    runs-on: ubuntu-latest
    needs: build                  # Runs AFTER build completes
    
    steps:
      - name: Checkout code
        uses: actions/checkout@v4
      
      - name: Install dependencies
        run: pip install -r requirements.txt
      
      # Run unit tests
      - name: Run Unit Tests
        run: python -m pytest tests/unit/ --verbose
      
      # Run integration tests
      - name: Run Integration Tests
        run: python -m pytest tests/integration/ --verbose
      
      # Run security scan
      - name: Security Scan
        run: trivy image myapp:${{ github.sha }}
      
      # Run code coverage
      - name: Code Coverage
        run: python -m pytest --cov=src/ --cov-report=html
```

### Key Takeaway:

> In DevOps, testing is **automated, continuous, and fast**. Every code change triggers tests automatically. If tests fail, the pipeline stops — broken code never reaches production. The phrase is: **"Shift left"** — test as early as possible.

## Phase 5: 📦 RELEASE

### What Happens Here?

The tested artifact is **approved and prepared for deployment**. This is the gate between "it works" and "let's put it in production." The release phase decides **when** and **what version** goes to production.

### Activities in This Phase:

```
✅ Version tagging (v1.2.3 using Semantic Versioning)
✅ Generate release notes / changelog
✅ Approval gates (manual or automated)
✅ Store release artifact in production-ready registry
✅ Create release branch (if using Git Flow)
✅ Notify stakeholders that a release is ready
```

### Semantic Versioning (SemVer):

```
Version format: MAJOR.MINOR.PATCH

  v1.0.0 → v1.0.1    PATCH: Bug fix (backward compatible)
  v1.0.0 → v1.1.0    MINOR: New feature (backward compatible)
  v1.0.0 → v2.0.0    MAJOR: Breaking change (NOT backward compatible)

Examples:
  v1.0.0  → Initial release
  v1.0.1  → Fixed a login bug
  v1.1.0  → Added "forgot password" feature
  v1.2.0  → Added social login
  v2.0.0  → Completely redesigned the auth system (breaking change)
```

### Tools Used:

| Tool | Purpose |
|------|---------|
| **Git Tags** | Mark specific commits as releases |
| **GitHub Releases / GitLab Releases** | Create releases with notes |
| **Semantic Release** | Automated versioning and changelog |
| **Helm** | Package Kubernetes apps for release |
| **Nexus / Artifactory** | Store release-ready artifacts |

### Real-World Example:

```bash
# Tag a release
git tag -a v1.2.0 -m "Release v1.2.0: Added checkout caching"

# Push the tag
git push origin v1.2.0

# This triggers the deployment pipeline automatically
# → The v1.2.0 Docker image is promoted to the production registry
```

### Key Takeaway:

> The release phase is about **deciding what's ready** for production. In mature DevOps organizations, this is **automated** — if all tests pass, the release is automatically approved and moves to deployment.

## Phase 6: 🚀 DEPLOY

### What Happens Here?

The released artifact is **deployed to the production environment** (or staging/pre-prod first). This is where your code actually reaches real users.

### Deployment Strategies:

| Strategy | How It Works | Risk Level |
|----------|-------------|------------|
| **Big Bang** | Deploy everything at once, replace old version | 🔴 High |
| **Rolling Update** | Gradually replace old instances with new ones | 🟡 Medium |
| **Blue/Green** | Run two identical environments, switch traffic | 🟢 Low |
| **Canary** | Deploy to a small percentage of users first, then expand | 🟢 Very Low |
| **A/B Testing** | Different versions for different user groups | 🟢 Very Low |

### Visual Explanation of Each:

```
ROLLING UPDATE:
Server 1: [v1] → [v2] ✅
Server 2: [v1] → [v2] ✅    (one at a time)
Server 3: [v1] → [v2] ✅
Server 4: [v1] → [v2] ✅

BLUE/GREEN:
Blue  (current): [v1] [v1] [v1] [v1]  ← Currently serving traffic
Green (new):     [v2] [v2] [v2] [v2]  ← Ready, tested
                       ↓
Switch traffic from Blue → Green instantly
If problems → Switch back to Blue instantly

CANARY:
[v1] [v1] [v1] [v1] [v1] [v1] [v1] [v1] [v1] [v2]  ← 10% traffic to v2
Monitor... looks good? ✅
[v1] [v1] [v1] [v1] [v1] [v2] [v2] [v2] [v2] [v2]  ← 50% traffic to v2
Monitor... looks good? ✅
[v2] [v2] [v2] [v2] [v2] [v2] [v2] [v2] [v2] [v2]  ← 100% traffic to v2
```

### Tools Used:

| Tool | Purpose |
|------|---------|
| **Kubernetes** | Container orchestration and deployment |
| **Helm** | Kubernetes package manager |
| **ArgoCD / Flux** | GitOps-based deployment |
| **Jenkins / GitHub Actions** | Pipeline-driven deployment |
| **AWS CodeDeploy** | AWS-native deployment |
| **Spinnaker** | Multi-cloud deployment |
| **Ansible** | Configuration and deployment automation |
| **Terraform** | Infrastructure deployment |

### Real-World Example — Kubernetes Deployment:

```yaml
# k8s/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: checkout-service
spec:
  replicas: 4                      # Run 4 copies of our app
  strategy:
    type: RollingUpdate             # Use rolling update strategy
    rollingUpdate:
      maxSurge: 1                   # Add 1 extra pod during update
      maxUnavailable: 0             # Never have 0 available pods
  selector:
    matchLabels:
      app: checkout
  template:
    metadata:
      labels:
        app: checkout
    spec:
      containers:
        - name: checkout
          image: myapp:v1.2.0       # The version we're deploying
          ports:
            - containerPort: 8080
          resources:
            requests:
              memory: "128Mi"
              cpu: "250m"
            limits:
              memory: "256Mi"
              cpu: "500m"
```

```bash
# Deploy with kubectl
kubectl apply -f k8s/deployment.yaml

# Watch the rolling update happen
kubectl rollout status deployment/checkout-service

# Something wrong? Rollback instantly!
kubectl rollout undo deployment/checkout-service
```

### Key Takeaway:

> Deployment in DevOps is **automated, repeatable, and reversible**. You should be able to deploy with a single command (or automatically) and rollback just as easily. No SSH-ing into servers and copying files manually!

## Phase 7: ⚙️ OPERATE

### What Happens Here?

The application is **running in production** and the team **maintains and manages** it. This includes keeping servers healthy, scaling up during traffic, managing incidents, and ensuring everything runs smoothly.

### Activities in This Phase:

```
✅ Infrastructure management (servers, networks, databases)
✅ Auto-scaling based on traffic
✅ Incident management (respond to outages)
✅ Patching and updates (OS, dependencies, security patches)
✅ Backup and disaster recovery
✅ Configuration management
✅ Cost optimization
✅ On-call rotations
✅ Runbook maintenance
```

### Tools Used:

| Tool | Purpose |
|------|---------|
| **Kubernetes** | Container orchestration, self-healing, auto-scaling |
| **Terraform** | Infrastructure management |
| **Ansible / Chef / Puppet** | Configuration management |
| **AWS / Azure / GCP** | Cloud infrastructure |
| **PagerDuty / Opsgenie** | Incident alerting and on-call management |
| **Confluence / Notion** | Runbooks and documentation |

### Real-World Example — Auto-Scaling:

```yaml
# Kubernetes Horizontal Pod Autoscaler
# Automatically scales pods based on CPU usage
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: checkout-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: checkout-service
  minReplicas: 2          # At least 2 pods always running
  maxReplicas: 20         # Scale up to 20 during high traffic
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70    # Scale up when CPU > 70%
```

```
Normal day:     [pod1] [pod2]                          (2 pods, low traffic)
Black Friday:   [pod1] [pod2] [pod3] ... [pod20]       (20 pods, HIGH traffic!)
After sale:     [pod1] [pod2]                          (scales back down)

All automatic! No human intervention needed. 🎉
```

### Key Takeaway:

> Operations in DevOps is **proactive, not reactive**. Instead of waiting for things to break and manually fixing them, you build **self-healing, auto-scaling systems** and automate routine operations.

## Phase 8: 📊 MONITOR

### What Happens Here?

**Watch everything.** Collect data on how the application and infrastructure are performing. Detect problems **before users notice them**. This is also where you gather feedback that feeds back into the **PLAN** phase.

### The Three Pillars of Observability:

```
┌──────────────────────────────────────────────────────┐
│                 OBSERVABILITY                         │
│                                                      │
│   📊 METRICS        📝 LOGS          🔗 TRACES      │
│   (Numbers)         (Events)         (Request flow)  │
│                                                      │
│   CPU: 75%          "User 123        Request went:   │
│   Memory: 4GB        logged in       Frontend →      │
│   Requests: 500/s    at 14:32"       API → DB →      │
│   Errors: 0.1%      "Payment         Cache → API →   │
│   Latency: 120ms     failed for      Frontend        │
│                       order 456"     (took 230ms)    │
└──────────────────────────────────────────────────────┘
```

| Pillar | What It Answers | Example Tools |
|--------|----------------|---------------|
| **Metrics** | "How much? How fast? How often?" | Prometheus, Datadog, CloudWatch |
| **Logs** | "What happened? What went wrong?" | ELK Stack, Loki, Splunk |
| **Traces** | "How did a request flow through the system?" | Jaeger, Zipkin, OpenTelemetry |

### What Do We Monitor?

```
Application Monitoring:
  ├── Response time (latency)
  ├── Error rate
  ├── Request throughput (requests/second)
  ├── Availability (uptime)
  └── Business metrics (orders/minute, signups/day)

Infrastructure Monitoring:
  ├── CPU usage
  ├── Memory usage
  ├── Disk I/O and space
  ├── Network traffic
  └── Container health

Pipeline Monitoring:
  ├── Build success/failure rate
  ├── Deployment frequency
  ├── Lead time for changes
  └── Mean Time to Recovery (MTTR)
```

### Tools Used:

| Tool | Purpose |
|------|---------|
| **Prometheus** | Metrics collection and alerting |
| **Grafana** | Beautiful dashboards and visualization |
| **ELK Stack (Elasticsearch, Logstash, Kibana)** | Log aggregation and analysis |
| **Loki** | Lightweight log aggregation |
| **Jaeger** | Distributed tracing |
| **Datadog / New Relic** | All-in-one monitoring platforms |
| **PagerDuty / Opsgenie** | Alert routing and on-call |
| **Slack / Teams** | Alert notifications |

### Real-World Example — Prometheus Alert:

```yaml
# Alert rule: Notify when error rate is too high
groups:
  - name: checkout-alerts
    rules:
      - alert: HighErrorRate
        expr: rate(http_requests_total{status="500", service="checkout"}[5m]) > 0.05
        for: 2m                      # Must be true for 2 minutes
        labels:
          severity: critical
        annotations:
          summary: "Checkout service has high error rate"
          description: "More than 5% of checkout requests are failing for the last 2 minutes"
```

**What this does in plain English:**
- Watches the checkout service
- If more than 5% of requests are returning error (HTTP 500) for 2 minutes straight
- Fires a **critical alert** → goes to PagerDuty → wakes up the on-call engineer
- The engineer uses Grafana dashboards and logs to diagnose and fix the issue

### The Feedback Loop (Monitor → Plan):

```
Monitor detects: "Checkout latency increased 300% after last deployment"
    ↓
Team investigates: "The new database query is slow"
    ↓
Feeds back to PLAN phase: "Sprint task: Optimize checkout DB query"
    ↓
And the cycle begins again... ♾️
```

### Key Takeaway:

> Monitoring **completes the loop**. Without monitoring, you're flying blind. The data from monitoring feeds directly back into planning, making the whole DevOps lifecycle a **continuous feedback loop** of improvement.


### The DORA Metrics (Measuring DevOps Success):

The **DevOps Research and Assessment (DORA)** team at Google identified **4 key metrics** to measure how well your DevOps lifecycle is performing:

| Metric | What It Measures | Elite Performance |
|--------|-----------------|-------------------|
| **Deployment Frequency** | How often you deploy to production | Multiple times per day |
| **Lead Time for Changes** | Time from code commit to production | Less than 1 hour |
| **Change Failure Rate** | Percentage of deployments causing failure | 0-15% |
| **Mean Time to Recovery (MTTR)** | How fast you recover from failure | Less than 1 hour |
