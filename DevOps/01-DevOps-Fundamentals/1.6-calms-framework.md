# 1.6 — CALMS Framework (Culture, Automation, Lean, Measurement, Sharing)

> **One Line Answer:** CALMS is a **5-pillar framework** that helps you understand, assess, and improve your DevOps adoption — it's like a checklist that tells you: "Are you really doing DevOps, or just pretending?"

## 🤔 Why Do We Need a Framework?

So we said DevOps is a "culture." But that's kind of vague, right?

- How do you **measure** culture?
- How do you know if you're doing DevOps **correctly**?
- How do you explain DevOps to your **boss** in a structured way?

That's where **CALMS** comes in.

CALMS was created by **Jez Humble** (co-author of *The DevOps Handbook* and *Continuous Delivery*) as a way to assess whether an organization is truly adopting DevOps or just doing "DevOps theater" (pretending to do DevOps while nothing really changes).

```
C.A.L.M.S.

C — Culture
A — Automation
L — Lean
M — Measurement
S — Sharing

Each letter represents a PILLAR of DevOps. 
If you're weak in any one of these, your DevOps is incomplete.
```

Let's break down each pillar in detail.

---

## 🔵 C — Culture

### What It Means

Culture is about **how people in your organization work together, communicate, and handle failures.**

It's the most important pillar — and the hardest to change. You can install Jenkins in a day. Changing culture takes months or years.

### Key Aspects of DevOps Culture

| Aspect | Description |
|--------|-------------|
| **Collaboration** | Dev, Ops, QA, Security — all work together as ONE team, not in isolated silos |
| **Trust** | Teams trust each other. No blame games. No finger-pointing |
| **Shared responsibility** | Everyone owns the product end-to-end. "You build it, you run it" |
| **Psychological safety** | People feel safe to admit mistakes, raise concerns, and suggest ideas |
| **Blameless postmortems** | When things break, we fix the system — not blame the person |
| **Growth mindset** | Failure is a learning opportunity, not a career-ending event |

### How to Assess Your Culture

Ask yourself these questions:

```
□ Can a developer deploy to production without waiting for another team?
□ When production goes down, do teams collaborate or blame each other?
□ Are postmortems blameless? Or does someone always get punished?
□ Do Dev and Ops sit together (physically or in shared channels)?
□ Does leadership encourage experimentation and accept failure?
□ Are goals shared across teams, or does each team have conflicting KPIs?

If most answers are "No" → Your culture needs work.
```

### Example: A Company with Good DevOps Culture

```
Scenario: A deployment breaks production at 2 PM

What happens at a company with GOOD DevOps culture:
1. Monitoring automatically detects the issue → alerts the on-call engineer
2. On-call engineer (who could be a Dev or Ops person) acknowledges the alert
3. They open an incident channel in Slack → anyone who can help joins voluntarily
4. Team collaborates in real-time to diagnose and fix
5. Service is restored
6. Next day: blameless postmortem → action items created → systems improved
7. Postmortem shared with entire organization → everyone learns

Notice: No yelling. No blame. No "who did this?!" Just teamwork and learning.
```

## 🟢 A — Automation

### What It Means

Automation means **replacing manual, repetitive, error-prone work with automated processes.**

If a human is doing the same task over and over — AUTOMATE IT.

### What Should Be Automated?

```
The DevOps Automation Spectrum:

Everything that a human does more than once should be automated:

┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│  Code → Build → Test → Package → Deploy → Monitor → Respond    │
│    ↑       ↑       ↑       ↑        ↑        ↑         ↑       │
│   Git   Build   Unit    Docker   CI/CD   Prometheus  Auto-     │
│        Tools   Tests   Images   Pipeline Grafana     scaling   │
│                Integ.  Helm              Alerts      Self-     │
│                Tests   Charts                        healing   │
│                                                                 │
│  ALL of this should be automated.                              │
└─────────────────────────────────────────────────────────────────┘
```

### Areas of Automation in DevOps

| Area | Manual Way (Bad) | Automated Way (Good) | Tools |
|------|-----------------|---------------------|-------|
| **Code Integration** | Copy files to shared folder | Git push → auto merge | Git, GitHub, GitLab |
| **Building** | Run build command manually | Auto-build on every commit | Jenkins, GitHub Actions, GitLab CI |
| **Testing** | Tester manually clicks through the app | Automated test suite runs on every build | Selenium, Jest, pytest, JUnit |
| **Infrastructure** | Click around in AWS console to create servers | Write code that creates infrastructure | Terraform, CloudFormation, Pulumi |
| **Configuration** | SSH into each server and edit config files | Push config to all servers automatically | Ansible, Chef, Puppet, SaltStack |
| **Deployment** | Manually copy files to production server | One click (or zero click) deployment | ArgoCD, Spinnaker, Flux, Jenkins |
| **Monitoring** | Check logs manually | Auto-alerts when something goes wrong | Prometheus, Grafana, Datadog, PagerDuty |
| **Security** | Manual security review before release | Auto-scan code for vulnerabilities | Snyk, SonarQube, Trivy, OWASP ZAP |
| **Scaling** | Add more servers manually when traffic spikes | Auto-scale based on traffic | Kubernetes HPA, AWS Auto Scaling |

### Why Automation Matters

```
Manual Process:
- Slow (humans are slow)
- Error-prone (humans make mistakes)
- Inconsistent (different person = different result)
- Not scalable (can't manually deploy to 1000 servers)
- Boring (nobody wants to do the same thing 100 times)

Automated Process:
- Fast (runs in seconds/minutes)
- Reliable (same result every time)
- Consistent (no "works on my machine" problems)
- Scalable (same effort for 1 server or 1000 servers)
- Frees humans to do creative, valuable work
```

### A Simple Automation Example: CI/CD Pipeline

Here's what an automated CI/CD pipeline looks like in a tool like GitHub Actions:

```yaml
# .github/workflows/deploy.yml
# This file tells GitHub to automatically build, test, and deploy your code

name: Build and Deploy

# WHEN should this run?
on:
  push:
    branches: [main]  # Every time someone pushes code to the main branch

# WHAT should it do?
jobs:
  build-and-deploy:
    runs-on: ubuntu-latest  # Use a fresh Linux machine
    
    steps:
      # Step 1: Get the code
      - name: Checkout code
        uses: actions/checkout@v4
      
      # Step 2: Set up the programming language
      - name: Set up Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
      
      # Step 3: Install dependencies
      - name: Install dependencies
        run: npm install
      
      # Step 4: Run tests
      - name: Run tests
        run: npm test
      
      # Step 5: Build the application
      - name: Build
        run: npm run build
      
      # Step 6: Deploy to production
      - name: Deploy
        run: npm run deploy
```

**What this does:** Every time you push code to the `main` branch, GitHub **automatically** checks out your code, installs dependencies, runs tests, builds the app, and deploys it. No human involvement needed!

### The Golden Rule of Automation

> **"If you have to do it twice, automate it."** — Google SRE

But also:

> **Don't automate a broken process.** Fix the process first, THEN automate it. Otherwise, you just do the wrong thing faster.

## 🟡 L — Lean

### What It Means

Lean comes from **Lean Manufacturing** (originally from Toyota in the 1950s). The core idea is:

> **Eliminate waste. Maximize value. Deliver faster.**

In DevOps, "Lean" means focusing on **what delivers value to the customer** and removing everything else.

### What is "Waste" in Software?

Lean identifies several types of waste. Here's how they apply to software:

| Type of Waste | In Manufacturing | In Software/DevOps |
|--------------|-----------------|-------------------|
| **Waiting** | Parts waiting on the assembly line | Code waiting for approval, deployment, or review |
| **Overproduction** | Making too many parts | Building features nobody asked for or will use |
| **Defects** | Broken products | Bugs, security vulnerabilities, outages |
| **Extra Processing** | Unnecessary steps in production | Unnecessary approvals, meetings, documentation |
| **Motion** | Workers walking too far | Developers switching between too many tools/tasks |
| **Inventory** | Too many parts sitting in warehouse | Unfinished features sitting in branches, not deployed |
| **Transportation** | Moving parts between factories | Manual handoffs between teams (Dev → QA → Ops) |

### Lean Practices in DevOps

#### 1. Small Batch Sizes

Instead of working on a huge feature for 6 months and deploying it all at once, break it into tiny pieces and deploy each one:

```
❌ Big Batch (Risky, Slow):
   Work for 6 months → 10,000 lines of code → Deploy → 💥 Probably breaks

✅ Small Batch (Safe, Fast):
   Work for 1 day → 50 lines of code → Deploy → ✅ Easy to verify
   Work for 1 day → 30 lines of code → Deploy → ✅ Easy to verify
   Work for 1 day → 40 lines of code → Deploy → ✅ Easy to verify

Smaller batches = Less risk = Faster feedback = Easier debugging
```

#### 2. Limit Work in Progress (WIP)

Don't try to work on 10 things at once. Focus on **finishing** a few things rather than **starting** many things.

```
❌ Bad: 10 tasks started, 0 finished
   Task 1: 30% done
   Task 2: 50% done
   Task 3: 10% done
   Task 4: 70% done
   ... and so on

✅ Good: 2 tasks started, 8 in queue
   Task 1: ✅ DONE
   Task 2: 80% done (almost done!)
   Task 3: Waiting (that's OK — it'll get focused attention next)
```

Why? Because **unfinished work is waste.** A feature that's 90% done delivers ZERO value until it's 100% done and deployed.

#### 3. Value Stream Mapping

Map out every step from "idea" to "running in production" and find the bottlenecks:

```
Value Stream Example:

Idea → Write Code → Code Review → Merge → Build → Test → Deploy → Monitor
       (2 hours)    (3 DAYS! 😱)  (1 hr)  (5 min) (10 min) (1 hr)  (always)

Total lead time: ~4 days
Bottleneck: Code Review (3 days!)

Fix: Add more reviewers, set review SLAs, use automated code analysis
New lead time: ~1 day 🎉
```

#### 4. Continuous Improvement (Kaizen)

Always ask: "How can we make this better?" Never say "good enough."

- Hold regular retrospectives
- Measure everything (see the next section: Measurement)
- Make small improvements constantly

## 🟠 M — Measurement

### What It Means

> **"You can't improve what you can't measure."**

Measurement means **tracking the right metrics** so you know:
- Are we getting faster?
- Are we getting more reliable?
- Are we delivering more value?
- Where are our bottlenecks?

### The Four Key Metrics (DORA Metrics)

The **DORA (DevOps Research and Assessment)** team at Google identified **4 key metrics** that separate high-performing DevOps teams from low-performing ones:

```
The 4 DORA Metrics:

1. Deployment Frequency      → How often do you deploy to production?
2. Lead Time for Changes     → How long from code commit to running in production?
3. Change Failure Rate       → What % of deployments cause failures?
4. Mean Time to Recovery     → When something breaks, how fast do you fix it?
```

| Metric | Elite Teams | High | Medium | Low |
|--------|-------------|------|--------|-----|
| **Deployment Frequency** | On-demand (multiple/day) | Weekly to monthly | Monthly to every 6 months | Every 6+ months |
| **Lead Time for Changes** | Less than 1 hour | 1 day to 1 week | 1 week to 1 month | 1 to 6 months |
| **Change Failure Rate** | 0-15% | 16-30% | 16-30% | 16-30% |
| **Mean Time to Recovery (MTTR)** | Less than 1 hour | Less than 1 day | 1 day to 1 week | More than 6 months |

### What Else Should You Measure?

Beyond DORA, here are other important metrics:

| Category | Metric | Why It Matters |
|----------|--------|---------------|
| **Speed** | Build time | How long does it take to build your app? |
| **Speed** | Test execution time | How long do automated tests take? |
| **Quality** | Code coverage | What % of code is covered by tests? |
| **Quality** | Bug escape rate | How many bugs reach production? |
| **Reliability** | Uptime (SLA/SLO) | What % of time is your service available? |
| **Reliability** | Error rate | What % of requests result in errors? |
| **Efficiency** | Infrastructure cost | How much are you spending on cloud? |
| **Efficiency** | Resource utilization | Are servers overprovisioned or underused? |
| **People** | Developer satisfaction | Are developers happy with tools and processes? |
| **People** | On-call burden | How often are people getting woken up at night? |

### How to Measure: Dashboards

Teams typically create dashboards that show these metrics in real-time. Here's what that looks like conceptually:

```
┌─────────────────────────────────────────────────────────────┐
│                    DevOps Dashboard                         │
├─────────────────┬───────────────────┬───────────────────────┤
│ Deploy Freq     │ Lead Time         │ Change Failure Rate   │
│ 12/day  📈 ✅   │ 45 min  📉 ✅     │ 3%  📉 ✅            │
├─────────────────┼───────────────────┼───────────────────────┤
│ MTTR            │ Uptime            │ Error Rate            │
│ 15 min  📉 ✅   │ 99.95%  📈 ✅     │ 0.1%  📉 ✅          │
├─────────────────┴───────────────────┴───────────────────────┤
│ Pipeline Status: ✅ Build #4521 passed (2 min ago)         │
│ Active Incidents: 0                                         │
│ On-Call: Sarah (Primary), Mike (Secondary)                  │
└─────────────────────────────────────────────────────────────┘
```

### Common Measurement Anti-Patterns

| Anti-Pattern | Problem |
|-------------|---------|
| **Measuring vanity metrics** | "We have 1000 builds!" — So what? Did they deliver value? |
| **Measuring too many things** | Information overload. Nobody looks at dashboards with 200 charts |
| **Using metrics to punish people** | "Your change failure rate is high → you're fired." Metrics become gamed |
| **Not acting on data** | Measuring is pointless if you don't use the data to improve |

## 🔴 S — Sharing

### What It Means

Sharing is about **open communication, transparency, and spreading knowledge** across teams and the organization.

> In DevOps, knowledge shouldn't be locked inside one person's head or one team's wiki. It should flow freely.

### What Should Be Shared?

| What | How |
|------|-----|
| **Code** | Open source internally. Any team can see any team's code (InnerSource) |
| **Postmortem reports** | Published to the entire organization — not hidden in team folders |
| **Monitoring dashboards** | Accessible to everyone, not just Ops |
| **Documentation** | Wikis, runbooks, and guides are written collaboratively and openly |
| **Knowledge** | Tech talks, lunch & learns, internal blogs, pair programming |
| **Tools** | Shared platforms and tools — don't let each team build their own pipeline |
| **On-call experiences** | What did you learn from that 3 AM incident? Share it! |
| **Feedback** | Both positive and constructive. Regularly and openly |

### Sharing Practices

#### 1. ChatOps

Run operations through chat (Slack, Teams) so everyone can **see** what's happening:

```
In a Slack channel:

🤖 DeployBot: [14:00] Deploying v2.3.1 to production...
🤖 DeployBot: [14:02] ✅ Deployment successful. All health checks passing.
👩 Sarah:     Nice! Looks good in monitoring.
🤖 AlertBot:  [14:15] ⚠️ Error rate increased by 5% in /api/users endpoint
👨 Mike:      Looking into it. Might be the new caching change.
👨 Mike:      [14:20] Found it. Cache TTL was too low. Pushing fix now.
🤖 DeployBot: [14:22] Deploying hotfix v2.3.2...
🤖 DeployBot: [14:24] ✅ Deployment successful.
👩 Sarah:     Error rate back to normal. 👍

Everyone can see what happened. Full transparency.
```

#### 2. Internal Tech Talks & Communities of Practice

- Weekly "lunch & learn" sessions where people present what they learned
- Internal blogs where engineers share solutions to problems
- "Guilds" or "Communities of Practice" — cross-team groups focused on specific topics (e.g., Kubernetes Guild, Security Guild)

#### 3. Documentation as Code

Treat documentation like code:
- Store it in Git (version controlled)
- Review it through pull requests
- Keep it close to the code it documents
- If code changes, docs must change too

#### 4. InnerSource

Apply open-source practices **inside** the company:
- Any team can see any other team's code
- Any team can submit pull requests to any other team's codebase
- Best practices spread across the organization naturally

### Why Sharing Matters

```
Without Sharing:                        With Sharing:

Team A solves Problem X                 Team A solves Problem X
Team B hits Problem X later             Team A shares the solution
Team B spends 3 days solving it again   Team B finds it in 5 minutes
Team C hits Problem X even later        Team C finds it in 5 minutes
Team C spends 3 days solving it again   Team D finds it in 5 minutes

Total time wasted: 9 days              Total time wasted: 0 days

Sharing = Compounding knowledge.
Every solution shared saves time for EVERYONE, FOREVER.
```

## 🧩 Putting CALMS Together

Here's how all five pillars work together:

```
                        ┌───────────┐
                        │  CULTURE  │
                        │  (Trust,  │
                        │ Teamwork) │
                        └─────┬─────┘
                              │
              ┌───────────────┼───────────────┐
              │               │               │
        ┌─────┴─────┐  ┌─────┴─────┐  ┌──────┴────┐
        │AUTOMATION │  │   LEAN    │  │  SHARING  │
        │(Eliminate │  │(Eliminate │  │ (Spread   │
        │ manual    │  │  waste)   │  │ knowledge)│
        │ work)     │  │           │  │           │
        └─────┬─────┘  └─────┬─────┘  └──────┬────┘
              │               │               │
              └───────────────┼───────────────┘
                              │
                        ┌─────┴─────┐
                        │MEASUREMENT│
                        │  (Track   │
                        │ progress) │
                        └───────────┘

Culture is the foundation.
Automation, Lean, and Sharing are the practices.
Measurement tells you if it's working.
```

### CALMS Self-Assessment

Use this checklist to see where your organization stands:

```
CULTURE:
  □ Teams collaborate across boundaries
  □ Blameless postmortems are standard
  □ People feel safe to experiment and fail
  □ Shared goals across Dev, Ops, QA, Security
  Score: __/4

AUTOMATION:
  □ CI/CD pipeline automates build, test, deploy
  □ Infrastructure is managed as code
  □ Monitoring and alerting are automated
  □ Security scanning is part of the pipeline
  Score: __/4

LEAN:
  □ Small batch sizes (frequent small releases)
  □ WIP limits are enforced
  □ Value stream is mapped and bottlenecks identified
  □ Continuous improvement is practiced (retrospectives)
  Score: __/4

MEASUREMENT:
  □ DORA metrics are tracked
  □ Dashboards are visible and accessible
  □ Data drives decisions (not gut feeling)
  □ Metrics are used to improve, not punish
  Score: __/4

SHARING:
  □ Knowledge is documented and accessible
  □ Postmortems are shared organization-wide
  □ ChatOps provides visibility into operations
  □ Internal tech talks and knowledge sharing happen regularly
  Score: __/4

Total: __/20
  0-5:   Just getting started — focus on culture first
  6-10:  Making progress — identify weakest pillar and focus there
  11-15: Good DevOps maturity — keep pushing!
  16-20: Elite level — you're a DevOps role model 🏆
```

## 📝 Key Takeaways

```
✅ CALMS = Culture, Automation, Lean, Measurement, Sharing
✅ Created by Jez Humble to assess DevOps adoption
✅ Culture: Trust, collaboration, blameless, shared responsibility
✅ Automation: Automate everything that's manual and repetitive
✅ Lean: Eliminate waste, small batches, limit WIP, continuous improvement
✅ Measurement: Track DORA metrics, use data to improve (not punish)
✅ Sharing: Share knowledge, tools, postmortems, dashboards openly
✅ If you're weak in ANY one pillar, your DevOps is incomplete
✅ Culture is the foundation — start there if you're starting from scratch
```
