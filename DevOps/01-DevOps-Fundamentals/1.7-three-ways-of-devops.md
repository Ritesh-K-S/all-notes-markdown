# 1.7 — Three Ways of DevOps (Flow, Feedback, Continuous Learning)

> **One Line Answer:** The Three Ways are the **fundamental principles** behind every DevOps practice — they explain **WHY** DevOps works, not just **HOW** to do it. If CALMS is the "what," the Three Ways are the "why."

---

## 🤔 Where Did the Three Ways Come From?

The Three Ways come from the legendary book **"The Phoenix Project"** by Gene Kim, Kevin Behr, and George Spafford (published in 2013). This book is a must-read for anyone in DevOps — it's written as a **novel** (yes, a story!) about an IT manager trying to save his company from disaster.

Gene Kim later expanded on these ideas in **"The DevOps Handbook"** (2016).

The Three Ways describe the **underlying principles** from which all DevOps behaviors and patterns are derived.

```
The Three Ways:

┌─────────────────────────────────────────────────────────┐
│                                                         │
│  First Way:  FLOW          (Left to Right)  →→→→→→→    │
│  Second Way: FEEDBACK      (Right to Left)  ←←←←←←←    │
│  Third Way:  CONTINUOUS LEARNING & EXPERIMENTATION      │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

Let's dive deep into each one.

---

## 🔵 The First Way: Flow (Systems Thinking)

### The Core Idea

> **Make work flow from left to right (from Development to Operations to the Customer) as fast and smoothly as possible.**

Think of software delivery as a **river**. You want the water (your work) to flow smoothly from the source (an idea) all the way to the ocean (production/customer). Any dam, blockage, or whirlpool slows everything down.

```
The Flow:

IDEA → CODE → BUILD → TEST → DEPLOY → OPERATE → CUSTOMER
  ════════════════════════════════════════════════════►
                    FLOW (Left to Right)

Goal: Make this flow as FAST and SMOOTH as possible.
Remove every bottleneck, every wait, every blockage.
```

### What Blocks Flow? (The Enemies)

| Blocker | Example | Impact |
|---------|---------|--------|
| **Long wait times** | Code review takes 5 days | Work sits idle, value not delivered |
| **Large batch sizes** | Deploy 6 months of work at once | Huge risk, hard to debug if it breaks |
| **Manual processes** | Manual testing, manual deployment | Slow, error-prone, can't scale |
| **Handoffs between teams** | Dev throws code to QA, QA throws to Ops | Each handoff adds delay and information loss |
| **Rework/defects** | Bugs found late in the process | Fix takes 10x more time than if caught early |
| **Multitasking** | Working on 5 things at once | Nothing gets finished, context switching is costly |
| **Unplanned work** | Constant firefighting, emergencies | Planned work keeps getting pushed |

### How to Improve Flow

#### 1. Make Work Visible

You can't improve what you can't see. Use **Kanban boards** to visualize all work:

```
┌─────────────┬──────────────┬───────────────┬───────────────┬──────────┐
│  Backlog    │  In Progress │  In Review    │  Testing      │   Done   │
│             │  (WIP: 3)    │  (WIP: 2)     │  (WIP: 2)     │          │
├─────────────┼──────────────┼───────────────┼───────────────┼──────────┤
│ Feature E   │ Feature B    │ Feature A     │ Bug Fix #3    │Feature Z │
│ Feature F   │ Bug Fix #5   │               │               │Feature Y │
│ Feature G   │ Feature C    │               │               │Bug Fix#1 │
│ Feature H   │              │               │               │          │
└─────────────┴──────────────┴───────────────┴───────────────┴──────────┘

Now you can SEE:
- Where is work piling up? (Bottleneck in "In Review"? Need more reviewers!)
- Is anyone working on too many things? (WIP limits help)
- How long does work take from left to right? (Lead time)
```

#### 2. Limit Work in Progress (WIP)

The #1 counterintuitive insight:

> **Starting more work does NOT make you faster. Finishing work makes you faster.**

```
❌ Start 10 tasks → context switch constantly → finish 2 in a week
✅ Start 3 tasks → focus deeply → finish 3 in a week AND do them better

The math:
- 10 tasks at 10% each = 0 delivered value
- 3 tasks at 100% each = 3 delivered values

WIP limits FORCE you to finish before starting new work.
```

#### 3. Reduce Batch Sizes

The smaller the change, the easier it is to:
- **Review** it (50 lines vs 5000 lines)
- **Test** it (does this one small thing work?)
- **Deploy** it (lower risk)
- **Debug** it (if it breaks, you know exactly what changed)
- **Roll back** (undo just that small change)

```
Big Batch Deployment:

Commit 1 ─┐
Commit 2 ─┤
Commit 3 ─┤
Commit 4 ─┼──→ Big Deploy → 💥 Something broke!
Commit 5 ─┤                    Which commit caused it? 🤷
Commit 6 ─┤                    Could be ANY of them!
Commit 7 ─┘

Small Batch Deployment:

Commit 1 ──→ Deploy → ✅ Works!
Commit 2 ──→ Deploy → ✅ Works!
Commit 3 ──→ Deploy → 💥 Broke! → We KNOW it's Commit 3 → Fix in minutes
Commit 4 ──→ Deploy → ✅ Works!

Small batches = Fast diagnosis = Fast recovery
```

#### 4. Automate Everything in the Path

Every manual step in the pipeline is a speed bump. Automate the entire path:

```
Manual Pipeline (Slow):

Developer commits code
    ↓ (waits for someone to notice)
Someone manually triggers a build       → 30 min wait
    ↓ (waits for QA to be available)
QA manually tests the feature            → 2 days wait
    ↓ (waits for change approval meeting)
Change Advisory Board approves (weekly)  → 5 days wait
    ↓ (waits for Ops team to schedule)
Ops manually deploys on maintenance day  → 3 days wait

Total lead time: ~10 days 😫

Automated Pipeline (Fast):

Developer commits code
    ↓ (automatically triggers)
CI server builds the code                → 2 minutes
    ↓ (automatically runs)
Automated test suite passes              → 10 minutes
    ↓ (automatically deploys)
Deployed to staging                      → 5 minutes
    ↓ (automatically verifies)
Smoke tests pass                         → 2 minutes
    ↓ (automatically promotes)
Deployed to production                   → 5 minutes

Total lead time: ~24 minutes 🚀
```

#### 5. Reduce Handoffs

Every time work moves from one team to another, information is lost and delays are added:

```
Handoffs are like the "Telephone Game":

Dev tells QA: "Here's the feature. Test it."
QA tells Ops: "It passed tests. Deploy it."
Ops tells Support: "It's deployed. Support it."

At each step, context is lost. 
By the time Support gets a question, nobody remembers WHY something was built that way.

Solution: Cross-functional teams that handle the entire flow.
```

### Real-World First Way Example: Amazon

Amazon's deployment pipeline is a perfect example of The First Way:

- Code commit to production: **minutes** (not days or weeks)
- Deployments per day: **~150,000** across all services
- How? Fully automated pipeline, tiny batch sizes (microservices), no manual gates
- Each team owns their pipeline end-to-end (no handoffs)

## 🟢 The Second Way: Feedback (Amplify Feedback Loops)

### The Core Idea

> **Create fast, constant feedback loops from right to left — from Operations back to Development — so problems are found and fixed immediately.**

The First Way moves work forward fast. The Second Way sends **information backward** fast — so you know if what you shipped actually works!

```
The Feedback Loop:

IDEA → CODE → BUILD → TEST → DEPLOY → OPERATE → CUSTOMER
  ════════════════════════════════════════════════════►
                    FLOW (Left to Right)
  
  ◄════════════════════════════════════════════════════
                    FEEDBACK (Right to Left)

"Is it working? Are customers happy? Is anything broken?"
```

### Why Feedback Matters

Without feedback, you're flying blind:

```
Without Feedback:                           With Feedback:

Deploy code → 🤷 "Hope it works"           Deploy code → Monitor → Alert in 30 sec
Customer complaint after 2 weeks            Fix before customer even notices
Dev: "What? We deployed that 2 weeks ago"   Dev: "Got it, fixing now"
Takes 3 days to even find the bug           Fix deployed in 15 minutes
Takes 2 more days to fix it                 Customer impact: near zero
Customer impact: 2+ weeks of bad experience
```

### Types of Feedback in DevOps

#### 1. Automated Testing (Fastest Feedback)

Tests give you feedback **before** code even reaches production:

```
Types of Tests (from fastest to slowest):

Unit Tests (milliseconds)
├── Test individual functions
├── "Does this function add two numbers correctly?"
└── Run: thousands per minute

Integration Tests (seconds)
├── Test how components work together
├── "Does the API correctly save data to the database?"
└── Run: hundreds per minute

End-to-End Tests (minutes)
├── Test the entire user journey
├── "Can a user sign up, log in, and buy a product?"
└── Run: tens per minute

Each layer catches different types of bugs.
```

Here's a simple unit test example:

```python
# calculator.py — the code we want to test
def add(a, b):
    return a + b

def divide(a, b):
    if b == 0:
        raise ValueError("Cannot divide by zero")
    return a / b
```

```python
# test_calculator.py — the test that gives us feedback
import pytest
from calculator import add, divide

def test_add_positive_numbers():
    """Does add() work with positive numbers?"""
    assert add(2, 3) == 5      # ✅ If this passes, we're confident!

def test_add_negative_numbers():
    """Does add() work with negative numbers?"""
    assert add(-1, -1) == -2   # ✅ Edge case covered!

def test_divide_normal():
    """Does divide() work with normal numbers?"""
    assert divide(10, 2) == 5  # ✅ Basic case works!

def test_divide_by_zero():
    """Does divide() handle division by zero?"""
    with pytest.raises(ValueError):
        divide(10, 0)          # ✅ Error case handled!

# Run these tests: pytest test_calculator.py
# Result in 0.1 seconds: 4 passed ✅
# FEEDBACK: Your code works correctly!
```

**The point:** Within **0.1 seconds** of running tests, you KNOW if your code works. That's incredibly fast feedback!

#### 2. Monitoring & Alerting (Production Feedback)

Once code is in production, you need to know if it's actually working for real users:

```
Monitoring Setup (Conceptual):

Your Application
    │
    ├── Collects metrics (response time, error rate, CPU usage)
    │       │
    │       └──→ Prometheus (stores metrics)
    │                │
    │                └──→ Grafana (shows dashboards)
    │
    ├── Collects logs (what's happening inside the app)
    │       │
    │       └──→ ELK Stack / Loki (stores and searches logs)
    │
    └── Collects traces (follows a request through the system)
            │
            └──→ Jaeger / Zipkin (visualizes request flow)


Alert Rules:
- If error rate > 1% for 5 minutes → Alert on-call engineer
- If response time > 2 seconds → Alert the team
- If CPU > 90% for 10 minutes → Alert infrastructure team
```

Here's what a simple monitoring alert looks like:

```yaml
# Prometheus alert rule
# This automatically watches your application and alerts if something is wrong

groups:
  - name: application-alerts
    rules:
      # Alert if error rate goes above 1%
      - alert: HighErrorRate
        expr: rate(http_requests_total{status=~"5.."}[5m]) / rate(http_requests_total[5m]) > 0.01
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "High error rate detected!"
          description: "Error rate is {{ $value | humanizePercentage }} (threshold: 1%)"

      # Alert if response time is too slow
      - alert: SlowResponseTime
        expr: histogram_quantile(0.95, rate(http_request_duration_seconds_bucket[5m])) > 2
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Response time is slow!"
          description: "95th percentile response time is {{ $value }}s (threshold: 2s)"
```

**What this does:** Prometheus automatically watches your app. If errors spike above 1% or response time exceeds 2 seconds, it sends an alert. You find out about problems in **minutes**, not days.

#### 3. Peer Code Reviews (Human Feedback)

Before code is merged, another developer reviews it:

```
Code Review Flow:

Developer writes code
    ↓
Creates a Pull Request (PR)
    ↓
Another developer reviews:
    - Is the logic correct?
    - Are there edge cases missed?
    - Is it readable and maintainable?
    - Are there security concerns?
    ↓
Reviewer provides feedback:
    ✅ "Approved — looks great!"
    or
    💬 "Consider handling the null case on line 42"
    ↓
Developer addresses feedback
    ↓
Code gets merged

This feedback catches bugs BEFORE they reach production.
Much cheaper to fix a bug in code review than in production!
```

#### 4. Customer Feedback

The ultimate feedback: does the customer actually like what you built?

```
Feature Flags for Customer Feedback:

Instead of deploying a feature to ALL users at once:

1. Deploy to 1% of users (canary release)
2. Monitor: Are they using it? Do they like it? Any errors?
3. If good → roll out to 10%, then 50%, then 100%
4. If bad → turn it off instantly (no deployment needed!)

This gives you REAL customer feedback before full rollout.
```

### Feedback Speed Matters

The faster you get feedback, the cheaper it is to fix problems:

```
Cost of Fixing a Bug at Different Stages:

Found during coding (IDE warnings)        → $1          (seconds to fix)
Found during code review                  → $10         (minutes to fix)
Found during automated testing            → $100        (minutes to fix)
Found during integration testing          → $1,000      (hours to fix)
Found in staging/pre-production           → $5,000      (hours to fix)
Found in production (by monitoring)       → $10,000     (hours to fix)
Found by customer complaint               → $100,000    (days to fix + reputation damage)

The further right a bug travels, the MORE EXPENSIVE it becomes!

That's why The Second Way says:
"Push feedback as far LEFT as possible" (Shift Left)
```

### The Concept of "Shift Left"

"Shift Left" means doing things **earlier** in the pipeline:

```
Traditional (Problems found late):

CODE → BUILD → TEST → SECURITY SCAN → DEPLOY → MONITOR
                                 ↑
                           Problems found HERE
                           (expensive to fix!)

Shift Left (Problems found early):

CODE → BUILD → TEST → DEPLOY → MONITOR
  ↑       ↑       ↑
  │       │       └── Automated tests catch bugs
  │       └────────── Build catches compilation errors
  └────────────────── IDE catches syntax errors, linter catches style issues
                      Security scanning happens HERE
                      
Problems found early = cheap to fix!
```

## 🟡 The Third Way: Continuous Learning & Experimentation

### The Core Idea

> **Create a culture where everyone is constantly learning, experimenting, and improving — and where failure is seen as a learning opportunity, not a disaster.**

The First Way makes things fast. The Second Way catches problems. The Third Way makes the **whole system better over time** through learning and experimentation.

```
The Third Way:

         ┌──────────────────────────────────────┐
         │   CONTINUOUS LEARNING LOOP            │
         │                                       │
         │   Try Something New                   │
         │        ↓                              │
         │   Did it Work?                        │
         │     ├── Yes → Make it standard        │
         │     └── No  → Learn WHY → Try again   │
         │        ↓                              │
         │   Share What You Learned              │
         │        ↓                              │
         │   Repeat Forever ♻️                   │
         │                                       │
         └──────────────────────────────────────┘
```

### Key Practices of the Third Way

#### 1. Allocate Time for Learning and Improvement

If your team spends 100% of their time on feature work and firefighting, they'll never improve. You need to intentionally set aside time:

```
How top teams allocate time:

┌────────────────────────────────────────────┐
│ ██████████████████████████████░░░░░░░░░░░  │
│ ├── 70% Feature Work ──────┤├ 20% Tech ─┤ │
│                              │  Debt &    │ │
│                              │  Improve   │ │
│                              ├────────────┤ │
│                              │10% Learning│ │
│                              │& Experiment│ │
└────────────────────────────────────────────┘

Google's famous "20% time" policy:
Engineers spend 20% of their time on side projects.
Gmail, Google News, and AdSense all came from 20% time!
```

#### 2. Blameless Postmortems (Again!)

Postmortems are SO important that they appear in both The Third Way AND in CALMS AND in DevOps Culture (Chapter 1.5). That's because they are the **primary mechanism** for organizational learning.

```
The Postmortem Learning Cycle:

Incident Occurs
    ↓
Immediate Response (fix the problem)
    ↓
Postmortem Meeting (understand WHY)
    ↓
Action Items (prevent it from happening again)
    ↓
Implementation (actually make the changes)
    ↓
Share Learnings (so other teams benefit too)
    ↓
System is now STRONGER than before

Each incident makes the system MORE resilient!
This is called "antifragility" — getting stronger from stress.
```

#### 3. Experimentation (Safe to Fail)

Create an environment where it's safe to try new things:

```
Safe Experimentation Practices:

1. Feature Flags: Deploy new code but keep it "off" until you're ready
   
   if feature_flags.is_enabled("new_checkout_flow"):
       use_new_checkout()    # New experimental code
   else:
       use_old_checkout()    # Safe, proven code

2. Canary Deployments: Roll out changes to a tiny % of users first
   
   100 servers total:
   - 2 servers get the new version (canary)
   - 98 servers keep the old version
   - Monitor the canary: if it's healthy → roll out more
   - If it's broken → roll back just 2 servers (minimal impact!)

3. Blue-Green Deployments: Run two identical environments
   
   ┌──────────────┐     ┌──────────────┐
   │ BLUE (v1)    │     │ GREEN (v2)   │
   │ (currently   │     │ (new version │
   │  live)       │     │  being       │
   │              │     │  tested)     │
   └──────┬───────┘     └──────┬───────┘
          │                    │
          └──── Load Balancer ─┘
               (switch traffic when ready)

   If v2 works → switch traffic to Green
   If v2 fails → instantly switch back to Blue (zero downtime!)
```

#### 4. Game Days and Chaos Engineering

Deliberately **break things** to learn how your system responds:

```
Chaos Engineering:

Normal Day: "Our system can handle server failures!"
Game Day:   "Let's PROVE it. Kill a server RIGHT NOW."

Netflix's Chaos Monkey:
- Randomly kills virtual machines in production
- Forces teams to build resilient systems
- If your service can't handle a random server dying,
  you find out during a game day — not at 3 AM on Black Friday!

Game Day Process:
1. Define the hypothesis: "If we kill Database Server 2, 
   traffic should automatically route to Server 1 and Server 3"
2. Run the experiment: Actually kill Database Server 2
3. Observe: Did traffic reroute? Were there errors? How long?
4. Learn: "It took 45 seconds to reroute. Can we make it 5?"
5. Improve: Update failover configuration
6. Repeat: Run the same test again — now it takes 3 seconds ✅
```

#### 5. Communities of Practice & Knowledge Sharing

```
Internal Learning Structures:

1. Tech Talks (weekly/monthly)
   - Engineers present what they learned
   - "How I reduced our build time by 80%"
   - "What I learned from last week's incident"

2. Guilds / Communities of Practice
   - Cross-team groups focused on a topic
   - Example: "Kubernetes Guild" — anyone interested in K8s joins
   - Share best practices, solve common problems together

3. Internal Documentation & Wikis
   - Runbooks: "How to handle X incident"
   - Architecture Decision Records (ADRs): "Why we chose Y"
   - Onboarding guides: "New engineer? Start here"

4. Pair Programming / Mob Programming
   - Two or more people write code together
   - Knowledge transfers naturally
   - Senior teaches junior, junior brings fresh perspective

5. Hack Days / Innovation Sprints
   - Dedicate 1-2 days to work on anything
   - Build prototypes, try new tools, fix annoying problems
   - Some of the best ideas come from hack days!
```

### The Learning Organization

The Third Way transforms a company into a **Learning Organization** — one that gets better every day:

```
Regular Organization:                    Learning Organization:

Makes a mistake                          Makes a mistake
    ↓                                        ↓
Hides it                                 Shares it openly
    ↓                                        ↓
Same mistake happens again               Conducts postmortem
    ↓                                        ↓
Hides it again                           Creates prevention measures
    ↓                                        ↓
Eventually: catastrophic failure         Shares learning across teams
                                             ↓
                                         Same mistake NEVER happens again
                                             ↓
                                         Eventually: incredibly resilient system
```

## 🧩 How the Three Ways Work Together

The Three Ways aren't separate — they build on each other:

```
┌─────────────────────────────────────────────────────────────────────────┐
│                                                                         │
│   FIRST WAY (Flow):                                                     │
│   Make work flow fast from Dev → Ops → Customer                        │
│   "Ship things quickly and smoothly"                                   │
│                                                                         │
│       Enables ↓                                                        │
│                                                                         │
│   SECOND WAY (Feedback):                                                │
│   Get information back fast from Customer → Ops → Dev                  │
│   "Know if what you shipped actually works"                            │
│                                                                         │
│       Enables ↓                                                        │
│                                                                         │
│   THIRD WAY (Learning):                                                 │
│   Use that feedback to continuously improve                            │
│   "Get better every day"                                               │
│                                                                         │
│       Which improves ↓                                                 │
│                                                                         │
│   FIRST WAY (Flow) gets even faster...                                 │
│   And the cycle repeats, getting better each time!                     │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### A Day in the Life (Three Ways in Action)

Let's see how all three ways work together in a real scenario:

```
9:00 AM — Developer Sarah writes a new feature (small batch — First Way)
9:30 AM — She pushes code → CI pipeline auto-builds and tests (First Way: automation)
9:35 AM — Tests pass ✅ → Auto-deployed to staging (First Way: flow)
9:40 AM — Automated smoke tests pass ✅ → Deployed to production via canary (First Way)
9:45 AM — Monitoring shows: response time increased by 200ms (Second Way: feedback!)
9:50 AM — Alert fires → Sarah investigates (Second Way: fast feedback)
10:00 AM — Root cause found: inefficient database query (Second Way)
10:15 AM — Sarah pushes a fix → auto-deployed (First Way)
10:20 AM — Response time back to normal ✅ (Second Way confirms)

Next Day:
10:00 AM — Postmortem: "Why did a slow query reach production?" (Third Way)
           Action item: Add query performance test to CI pipeline (Third Way)
           Shared with all teams: "Here's how to test for slow queries" (Third Way)

Next Week:
           The CI pipeline now catches slow queries automatically (First Way improved!)
           This problem can NEVER happen again.

The system is now STRONGER than before the incident! 💪
```

## 📊 Three Ways vs CALMS — How Do They Relate?

| Three Ways | CALMS Pillar | Connection |
|-----------|-------------|------------|
| First Way (Flow) | Automation, Lean | Automation enables flow. Lean removes waste from flow |
| Second Way (Feedback) | Measurement | You need measurement to create feedback loops |
| Third Way (Learning) | Culture, Sharing | Learning requires psychological safety (culture) and knowledge sharing |
| All Three Ways | Culture | All three require a supportive, collaborative culture as the foundation |

```
Think of it this way:

CALMS tells you WHAT to do:
  → Build a collaborative culture
  → Automate things
  → Be lean
  → Measure stuff
  → Share knowledge

Three Ways tell you WHY:
  → Flow: Because fast delivery creates business value
  → Feedback: Because catching problems early saves money and reputation
  → Learning: Because continuous improvement is how you stay ahead
```

```
✅ The Three Ways come from "The Phoenix Project" by Gene Kim
```
