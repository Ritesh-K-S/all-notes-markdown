## 🤔 Why Do You Need Networking Commands?

Imagine you deployed your awesome web app on a server, but users say **"I can't access it!"**

What do you do? Panic? No! You use networking commands to figure out:

- Is the server even connected to the network?
- Is the right port open?
- Can the server reach the database?
- Is DNS resolving the domain name correctly?
- Where exactly is the connection failing?

**As a DevOps engineer, networking troubleshooting is your bread and butter.** You'll use these commands every single day.

---

## 🧱 The Big Picture — What Each Command Does

Before we dive deep, here's a quick cheat sheet:

| Command | What It Does | Think of it as... |
|---------|-------------|-------------------|
| `ip` | Show/configure network interfaces & routing | Your **network Swiss Army knife** (modern) |
| `ifconfig` | Show/configure network interfaces | The **old version** of `ip` |
| `netstat` | Show network connections, ports, stats | **Who's talking to whom?** (old) |
| `ss` | Show network connections, ports, stats | **Who's talking to whom?** (modern, faster) |
| `nslookup` | Query DNS to resolve domain names | **What's the IP of google.com?** |
| `dig` | Advanced DNS queries | **nslookup on steroids** |
| `curl` | Transfer data to/from a URL | **Talk to a website from terminal** |
| `wget` | Download files from the web | **Download manager for terminal** |
| `traceroute` | Trace the path packets take to a destination | **Follow the breadcrumbs across the internet** |
| `ping` | Test if a host is reachable | **"Hey, are you alive?"** |

## 1️⃣ `ip` — The Modern Network Tool

The `ip` command is the **modern replacement** for `ifconfig`, `route`, and several other old commands. It's part of the `iproute2` package and is available on virtually every modern Linux distro.

### Show All Network Interfaces

```bash
ip addr show
# or the shorthand
ip a
```

**Output explained:**

```
1: lo: <LOOPBACK,UP,LOWER_UP> mtu 65536
    inet 127.0.0.1/8 scope host lo
    inet6 ::1/128 scope host

2: eth0: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500
    inet 192.168.1.100/24 brd 192.168.1.255 scope global eth0
    inet6 fe80::1/64 scope link
```

Let's break this down:

| Part | Meaning |
|------|---------|
| `lo` | **Loopback interface** — the machine talking to itself (127.0.0.1) |
| `eth0` | **Ethernet interface** — your actual network connection |
| `inet 192.168.1.100/24` | Your **IPv4 address** is 192.168.1.100, and `/24` means the subnet mask is 255.255.255.0 |
| `brd 192.168.1.255` | **Broadcast address** — sends to all devices on the same network |
| `UP` | The interface is **active and running** |
| `mtu 1500` | **Maximum Transmission Unit** — max packet size in bytes |

### Show Only IPv4 Addresses

```bash
ip -4 addr show
```

### Show Only a Specific Interface

```bash
ip addr show eth0
```

### Assign an IP Address to an Interface

```bash
# Add an IP address
sudo ip addr add 192.168.1.200/24 dev eth0

# Remove an IP address
sudo ip addr del 192.168.1.200/24 dev eth0
```

> ⚠️ These changes are **temporary**! They'll be gone after a reboot. For permanent changes, edit the network config files (like `/etc/netplan/` on Ubuntu or `/etc/sysconfig/network-scripts/` on CentOS).

### Bring an Interface Up or Down

```bash
# Bring the interface DOWN (disconnect from network)
sudo ip link set eth0 down

# Bring the interface UP (reconnect to network)
sudo ip link set eth0 up
```

### Show the Routing Table

The routing table tells Linux **where to send packets**.

```bash
ip route show
# or shorthand
ip r
```

**Output:**

```
default via 192.168.1.1 dev eth0 proto dhcp metric 100
192.168.1.0/24 dev eth0 proto kernel scope link src 192.168.1.100
```

| Part | Meaning |
|------|---------|
| `default via 192.168.1.1` | **Default gateway** — if the destination isn't on the local network, send it here (usually your router) |
| `192.168.1.0/24 dev eth0` | Packets for the 192.168.1.x network go through `eth0` directly |

### Add/Delete a Route

```bash
# Add a route: "To reach 10.0.0.0/8 network, go through 192.168.1.1"
sudo ip route add 10.0.0.0/8 via 192.168.1.1

# Delete a route
sudo ip route del 10.0.0.0/8
```

### Show ARP Table (Who's on My Network?)

```bash
ip neigh show
# or shorthand
ip n
```

This shows the **MAC addresses** of devices on your local network.

## 2️⃣ `ifconfig` — The Classic (Old But Still Around)

`ifconfig` was the go-to command for decades, but it's now **deprecated** in favor of `ip`. However, you'll still see it everywhere — in old scripts, tutorials, and on older systems.

```bash
ifconfig
```

**Output:**

```
eth0: flags=4163<UP,BROADCAST,RUNNING,MULTICAST>  mtu 1500
        inet 192.168.1.100  netmask 255.255.255.0  broadcast 192.168.1.255
        ether 08:00:27:1a:2b:3c  txqueuelen 1000  (Ethernet)
        RX packets 15234  bytes 12345678 (12.3 MB)
        TX packets 8765  bytes 7654321 (7.6 MB)
```

| Part | Meaning |
|------|---------|
| `inet 192.168.1.100` | Your IP address |
| `netmask 255.255.255.0` | Subnet mask |
| `ether 08:00:27:1a:2b:3c` | MAC address (hardware address) |
| `RX packets` | Packets **received** |
| `TX packets` | Packets **transmitted** |

### Common `ifconfig` Commands

```bash
# Show a specific interface
ifconfig eth0

# Assign an IP address
sudo ifconfig eth0 192.168.1.200 netmask 255.255.255.0

# Bring interface up/down
sudo ifconfig eth0 up
sudo ifconfig eth0 down
```

### Install `ifconfig` (if missing)

On newer systems, `ifconfig` might not be installed:

```bash
# Ubuntu/Debian
sudo apt install net-tools

# CentOS/RHEL
sudo yum install net-tools
```

### `ip` vs `ifconfig` — Which One to Use?

| Feature | `ifconfig` | `ip` |
|---------|-----------|------|
| Status | ❌ Deprecated | ✅ Actively maintained |
| Speed | Slower | Faster |
| Features | Basic | Full networking control |
| Used in | Old scripts, legacy systems | Modern systems, new scripts |

**Bottom line:** Learn `ip`, but know `ifconfig` because you'll encounter it in the wild.

## 3️⃣ `netstat` — Network Statistics (Old School)

`netstat` shows you **active network connections, listening ports, routing tables**, and more. It's been replaced by `ss` but is still widely used.

### Show All Active Connections

```bash
netstat -a
```

### Show Only Listening Ports (The Most Useful One!)

```bash
netstat -tlnp
```

Let's break down those flags:

| Flag | Meaning |
|------|---------|
| `-t` | Show **TCP** connections only |
| `-l` | Show only **listening** ports (servers waiting for connections) |
| `-n` | Show **numbers** instead of names (192.168.1.1 instead of hostname) |
| `-p` | Show the **process** (PID) that owns each connection |

**Output:**

```
Proto  Local Address          Foreign Address        State       PID/Program
tcp    0.0.0.0:22             0.0.0.0:*              LISTEN      1234/sshd
tcp    0.0.0.0:80             0.0.0.0:*              LISTEN      5678/nginx
tcp    0.0.0.0:3306           0.0.0.0:*              LISTEN      9012/mysqld
```

This tells you:
- **SSH** is listening on port 22
- **Nginx** is listening on port 80
- **MySQL** is listening on port 3306

### Show All TCP Connections (Who's Connected Right Now?)

```bash
netstat -tnp
```

### Show UDP Connections

```bash
netstat -ulnp
```

### Show Routing Table

```bash
netstat -r
```

### Show Network Statistics

```bash
netstat -s
```

### Install `netstat` (if missing)

```bash
# Same package as ifconfig
sudo apt install net-tools      # Ubuntu/Debian
sudo yum install net-tools      # CentOS/RHEL
```

## 4️⃣ `ss` — Socket Statistics (Modern Replacement for netstat)

`ss` is **faster and more powerful** than `netstat`. It pulls data directly from the kernel, making it significantly quicker, especially on servers with thousands of connections.

### Show All Listening Ports

```bash
ss -tlnp
```

Same flags as `netstat`:

| Flag | Meaning |
|------|---------|
| `-t` | TCP |
| `-l` | Listening |
| `-n` | Numeric (don't resolve names) |
| `-p` | Show process |

**Output:**

```
State   Recv-Q  Send-Q  Local Address:Port   Peer Address:Port   Process
LISTEN  0       128     0.0.0.0:22           0.0.0.0:*           users:(("sshd",pid=1234,fd=3))
LISTEN  0       511     0.0.0.0:80           0.0.0.0:*           users:(("nginx",pid=5678,fd=6))
```

### Show All Established Connections

```bash
ss -tnp
```

### Show All Connections (TCP + UDP)

```bash
ss -tunap
```

| Flag | Meaning |
|------|---------|
| `-u` | UDP connections |
| `-a` | All (listening + established) |

### Filter by Port

```bash
# Show everything on port 80
ss -tlnp | grep :80

# Or use the built-in filter (more elegant)
ss -tlnp 'sport = :80'
```

### Filter by State

```bash
# Show only ESTABLISHED connections
ss state established

# Show only TIME-WAIT connections (useful for debugging)
ss state time-wait
```

### Count Connections by State

```bash
ss -s
```

**Output:**

```
Total: 256
TCP:   45 (estab 12, closed 5, orphaned 0, timewait 3)
```

This is super useful for quick health checks!

### `ss` vs `netstat` — Quick Comparison

| Feature | `netstat` | `ss` |
|---------|----------|------|
| Speed | Slow (reads /proc) | Fast (reads kernel directly) |
| Status | ❌ Deprecated | ✅ Modern standard |
| Filter support | Basic (use grep) | Built-in powerful filters |
| Info detail | Good | More detailed |

## 5️⃣ `ping` — Is This Host Alive?

The simplest and most used networking command. `ping` sends ICMP echo request packets and waits for replies.

```bash
ping google.com
```

**Output:**

```
PING google.com (142.250.190.78) 56(84) bytes of data.
64 bytes from 142.250.190.78: icmp_seq=1 ttl=117 time=12.3 ms
64 bytes from 142.250.190.78: icmp_seq=2 ttl=117 time=11.8 ms
64 bytes from 142.250.190.78: icmp_seq=3 ttl=117 time=12.1 ms
```

| Part | Meaning |
|------|---------|
| `64 bytes` | Size of the reply packet |
| `icmp_seq=1` | Sequence number (packet number) |
| `ttl=117` | **Time To Live** — how many hops (routers) the packet can pass through before dying |
| `time=12.3 ms` | **Round-trip time** — how long the packet took to go there and come back |

### Useful Options

```bash
# Send only 5 pings (instead of infinite)
ping -c 5 google.com

# Set timeout to 2 seconds
ping -W 2 google.com

# Ping with a specific packet size (useful for MTU testing)
ping -s 1400 google.com

# Flood ping (sends as fast as possible — use carefully, for testing only!)
sudo ping -f google.com
```

### What Does "Request Timed Out" Mean?

```
From 192.168.1.1 icmp_seq=1 Destination Host Unreachable
```

This could mean:
- The host is **down** or doesn't exist
- A **firewall** is blocking ICMP packets
- There's a **routing problem** between you and the host

> 💡 **Fun fact:** Many servers (like some cloud VMs) disable ICMP replies for security. So `ping` failing doesn't always mean the server is down!

## 6️⃣ `nslookup` — DNS Lookup (Simple)

`nslookup` queries DNS servers to find out what **IP address a domain name resolves to** (or vice versa).

### Basic Lookup

```bash
nslookup google.com
```

**Output:**

```
Server:    127.0.0.53
Address:   127.0.0.53#53

Non-authoritative answer:
Name:    google.com
Address: 142.250.190.78
```

| Part | Meaning |
|------|---------|
| `Server: 127.0.0.53` | The DNS server that answered your query |
| `Non-authoritative answer` | The answer came from a **cache**, not directly from Google's DNS servers |
| `Address: 142.250.190.78` | The IP address of google.com |

### Reverse Lookup (IP → Domain)

```bash
nslookup 8.8.8.8
```

**Output:**

```
8.8.8.8.in-addr.arpa    name = dns.google.
```

### Query a Specific DNS Server

```bash
# Ask Google's DNS (8.8.8.8) instead of your default DNS
nslookup google.com 8.8.8.8
```

### Query Specific Record Types

```bash
# MX records (mail servers)
nslookup -type=MX google.com

# NS records (name servers)
nslookup -type=NS google.com

# TXT records (SPF, DKIM, verification records)
nslookup -type=TXT google.com
```

## 7️⃣ `dig` — DNS Lookup (Advanced & Powerful)

`dig` (Domain Information Groper) is the **power tool** for DNS queries. It gives you much more detail than `nslookup` and is preferred by sysadmins and DevOps engineers.

### Basic Lookup

```bash
dig google.com
```

**Output (simplified):**

```
;; QUESTION SECTION:
;google.com.                    IN      A

;; ANSWER SECTION:
google.com.             179     IN      A       142.250.190.78

;; Query time: 12 msec
;; SERVER: 127.0.0.53#53(127.0.0.53)
```

| Part | Meaning |
|------|---------|
| `QUESTION SECTION` | What you asked for |
| `ANSWER SECTION` | The answer — google.com's IP is 142.250.190.78 |
| `179` | **TTL** — this DNS record will be cached for 179 more seconds |
| `IN` | Internet class |
| `A` | Record type (A = IPv4 address) |
| `Query time: 12 msec` | How long the DNS lookup took |

### Get Just the Answer (Clean Output)

```bash
dig +short google.com
```

**Output:**

```
142.250.190.78
```

Super clean! Great for scripting.

### Query Specific Record Types

```bash
# MX records (mail servers)
dig MX google.com

# AAAA records (IPv6 address)
dig AAAA google.com

# CNAME records (aliases)
dig CNAME www.google.com

# ALL records
dig ANY google.com

# NS records (name servers)
dig NS google.com
```

### Use a Specific DNS Server

```bash
# Query Cloudflare's DNS (1.1.1.1)
dig @1.1.1.1 google.com
```

### Trace DNS Resolution Path

This shows you the **entire chain** of DNS lookups — from root servers to the final answer:

```bash
dig +trace google.com
```

This is invaluable for debugging DNS propagation issues!

### `nslookup` vs `dig`

| Feature | `nslookup` | `dig` |
|---------|-----------|-------|
| Output | Simple, human-readable | Detailed, scriptable |
| Power | Basic lookups | Full DNS debugging |
| Recommended for | Quick checks | Serious DNS troubleshooting |

## 8️⃣ `curl` — Transfer Data from URLs

`curl` (Client URL) is one of the **most versatile commands** in Linux. It lets you make HTTP requests, download files, interact with APIs — basically anything that involves talking to a URL.

### Simple GET Request

```bash
curl https://google.com
```

This downloads and displays the HTML of google.com.

### Get Just the HTTP Headers

```bash
curl -I https://google.com
```

**Output:**

```
HTTP/2 200
content-type: text/html; charset=ISO-8859-1
date: Tue, 15 Jan 2025 10:30:00 GMT
server: gws
```

This is super useful for checking:
- Is the server responding? (200 OK)
- What server software is running? (gws = Google Web Server)
- Content type, caching headers, etc.

### Follow Redirects

```bash
curl -L https://google.com
```

Without `-L`, if the server sends a redirect (301/302), `curl` stops. With `-L`, it follows the redirect to the final destination.

### Show Detailed Info (Verbose)

```bash
curl -v https://google.com
```

This shows the **entire conversation**: DNS lookup, TCP connection, TLS handshake, request headers, response headers, and body.

### Download a File

```bash
# Save with the remote filename
curl -O https://example.com/file.zip

# Save with a custom filename
curl -o myfile.zip https://example.com/file.zip
```

### Send a POST Request (Interact with APIs)

```bash
# Send JSON data
curl -X POST https://api.example.com/users \
  -H "Content-Type: application/json" \
  -d '{"name": "John", "email": "john@example.com"}'
```

Let's break this down:

| Flag | Meaning |
|------|---------|
| `-X POST` | Use the POST HTTP method |
| `-H` | Add a **header** |
| `-d` | Send **data** (the request body) |

### Other HTTP Methods

```bash
# PUT request (update a resource)
curl -X PUT https://api.example.com/users/1 \
  -H "Content-Type: application/json" \
  -d '{"name": "Jane"}'

# DELETE request
curl -X DELETE https://api.example.com/users/1

# PATCH request (partial update)
curl -X PATCH https://api.example.com/users/1 \
  -H "Content-Type: application/json" \
  -d '{"email": "newemail@example.com"}'
```

### Authentication

```bash
# Basic auth
curl -u username:password https://api.example.com/private

# Bearer token (common in APIs)
curl -H "Authorization: Bearer YOUR_TOKEN_HERE" https://api.example.com/data
```

### Check Response Code Only

```bash
curl -s -o /dev/null -w "%{http_code}" https://google.com
# Output: 200
```

| Flag | Meaning |
|------|---------|
| `-s` | Silent mode (no progress bar) |
| `-o /dev/null` | Discard the response body |
| `-w "%{http_code}"` | Print only the HTTP status code |

This is **perfect for health check scripts!**

### Timing a Request

```bash
curl -s -o /dev/null -w "DNS: %{time_namelookup}s\nConnect: %{time_connect}s\nTLS: %{time_appconnect}s\nTotal: %{time_total}s\n" https://google.com
```

**Output:**

```
DNS: 0.012s
Connect: 0.025s
TLS: 0.067s
Total: 0.089s
```

This is amazing for debugging **slow connections** — you can see exactly where the time is spent!

## 9️⃣ `wget` — Download Files from the Web

`wget` is specifically designed for **downloading files**. While `curl` can do this too, `wget` has some unique advantages.

### Download a File

```bash
wget https://example.com/file.zip
```

### Download and Save with a Custom Name

```bash
wget -O custom-name.zip https://example.com/file.zip
```

### Resume a Failed Download

```bash
wget -c https://example.com/large-file.iso
```

The `-c` flag is a lifesaver! If your download gets interrupted, just run the same command again and it **picks up where it left off**.

### Download an Entire Website (Mirror)

```bash
wget --mirror --convert-links --page-requisites https://example.com
```

| Flag | Meaning |
|------|---------|
| `--mirror` | Download the whole site recursively |
| `--convert-links` | Convert links to work offline |
| `--page-requisites` | Download CSS, images, etc. |

### Download in the Background

```bash
wget -b https://example.com/huge-file.iso
```

The download continues in the background. Check progress with:

```bash
tail -f wget-log
```

### Limit Download Speed

```bash
# Limit to 500 KB/s
wget --limit-rate=500k https://example.com/file.zip
```

### `curl` vs `wget` — When to Use Which?

| Feature | `curl` | `wget` |
|---------|--------|--------|
| Download files | ✅ | ✅ (better for this) |
| API interactions | ✅ (much better) | ❌ Limited |
| Resume downloads | ✅ (`-C -`) | ✅ (`-c`) — easier |
| Recursive download | ❌ | ✅ (can mirror entire sites) |
| Protocol support | 25+ protocols | HTTP, HTTPS, FTP |
| Scripting | Great for APIs | Great for batch downloads |

**Rule of thumb:** Use `curl` for APIs and HTTP requests. Use `wget` for downloading files.

## 🔟 `traceroute` — Trace the Path to a Destination

`traceroute` shows you **every router (hop)** that your packets pass through on the way to a destination. It's like GPS navigation for your network packets!

### Basic Usage

```bash
traceroute google.com
```

**Output:**

```
traceroute to google.com (142.250.190.78), 30 hops max, 60 byte packets
 1  192.168.1.1 (192.168.1.1)  1.234 ms  1.123 ms  1.089 ms
 2  10.0.0.1 (10.0.0.1)  5.432 ms  5.321 ms  5.234 ms
 3  isp-router.example.net (203.0.113.1)  12.345 ms  12.234 ms  12.123 ms
 4  * * *
 5  google-router.net (142.250.1.1)  15.678 ms  15.567 ms  15.456 ms
 6  142.250.190.78 (142.250.190.78)  16.789 ms  16.678 ms  16.567 ms
```

| Part | Meaning |
|------|---------|
| Hop 1 (`192.168.1.1`) | Your **router/gateway** |
| Hop 2 (`10.0.0.1`) | Your ISP's first router |
| Hop 3 (`203.0.113.1`) | Another ISP router |
| Hop 4 (`* * *`) | This router **doesn't respond** to traceroute (firewall blocking) |
| Hop 5-6 | Getting closer to Google's servers |
| Three time values | Three probes are sent to each hop — you see the round-trip time for each |

### Use TCP Instead of UDP (Better for Firewalled Networks)

```bash
# Many firewalls block UDP but allow TCP
sudo traceroute -T google.com
```

### Use ICMP (Like ping)

```bash
sudo traceroute -I google.com
```

### Limit the Number of Hops

```bash
traceroute -m 10 google.com
```

### Install `traceroute` (if missing)

```bash
# Ubuntu/Debian
sudo apt install traceroute

# CentOS/RHEL
sudo yum install traceroute
```

### Alternative: `mtr` (My Traceroute) — The Best of Both Worlds

`mtr` combines `ping` and `traceroute` into a **single, real-time, continuously updating tool**:

```bash
mtr google.com
```

It shows you a live view of each hop with packet loss and latency. It's absolutely amazing for network debugging. Install it with:

```bash
sudo apt install mtr        # Ubuntu/Debian
sudo yum install mtr        # CentOS/RHEL
```

## 🛠️ Real-World Troubleshooting Scenarios

Let's put it all together with real scenarios you'll face as a DevOps engineer:

### Scenario 1: "The Website is Down!"

```bash
# Step 1: Can you even reach the server?
ping -c 3 myserver.com

# Step 2: Is DNS resolving correctly?
dig +short myserver.com

# Step 3: Is the web server listening on port 80/443?
ss -tlnp | grep -E ':80|:443'

# Step 4: Can you get a response from the web server?
curl -I https://myserver.com

# Step 5: Where is the connection failing?
traceroute myserver.com
```

### Scenario 2: "The App Can't Connect to the Database"

```bash
# Step 1: Is the database port open? (usually 3306 for MySQL, 5432 for PostgreSQL)
ss -tlnp | grep :3306

# Step 2: Can you reach the database server from the app server?
ping -c 3 db-server.internal

# Step 3: Can you actually connect to the port?
curl -v telnet://db-server.internal:3306

# Step 4: Check the app server's routing
ip route show
```

### Scenario 3: "DNS Isn't Working"

```bash
# Step 1: What DNS server are you using?
cat /etc/resolv.conf

# Step 2: Can you reach the DNS server?
ping -c 3 8.8.8.8

# Step 3: Does DNS resolution work with a known-good DNS server?
dig @8.8.8.8 google.com

# Step 4: Compare with your default DNS
dig google.com

# Step 5: Trace the full DNS resolution chain
dig +trace mysite.com
```

## 📋 The Ultimate Cheat Sheet

```bash
# ===== IP & INTERFACES =====
ip a                           # Show all interfaces & IPs
ip r                           # Show routing table
ip link set eth0 up/down       # Enable/disable interface

# ===== CONNECTIONS & PORTS =====
ss -tlnp                       # Show listening TCP ports with process info
ss -tunap                      # Show ALL connections (TCP + UDP)
ss -s                          # Connection statistics summary

# ===== DNS =====
dig +short example.com         # Quick DNS lookup
dig MX example.com             # Mail server records
dig @8.8.8.8 example.com      # Query specific DNS server
nslookup example.com           # Simple DNS lookup

# ===== HTTP & DOWNLOADS =====
curl -I https://example.com    # Get HTTP headers only
curl -v https://example.com    # Verbose (see everything)
curl -s -o /dev/null -w "%{http_code}" URL  # Get status code only
wget -c URL                    # Download with resume support

# ===== CONNECTIVITY =====
ping -c 5 host                 # Test reachability
traceroute host                # Trace packet path
mtr host                       # Live traceroute + ping
```

## ✅ Key Takeaways

1. **`ip` replaces `ifconfig`** — use `ip` for all interface and routing work
2. **`ss` replaces `netstat`** — use `ss` for checking ports and connections
3. **`dig` is better than `nslookup`** — use `dig` for DNS troubleshooting
4. **`curl` is for APIs, `wget` is for downloads** — both are essential
5. **`traceroute`/`mtr` shows the network path** — invaluable for debugging connectivity issues
6. **Combine these tools** to systematically troubleshoot any networking problem
7. Always follow a **systematic approach**: check DNS → check connectivity → check ports → check application
