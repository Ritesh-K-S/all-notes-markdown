## 📦 Part 1: Arrays in Bash

### What is an Array?

An array is just a **list of values stored in a single variable**.

Think of it like this:

```
Normal variable → ONE box with ONE item
Array            → ONE shelf with MANY boxes, each holding ONE item
```

### Indexed Arrays (Numbered List)

These are arrays where each item has a **number (index)** starting from 0.

#### Creating an Array

```bash
# Method 1: Direct assignment
fruits=("apple" "banana" "cherry" "mango")

# Method 2: One by one
colors[0]="red"
colors[1]="green"
colors[2]="blue"

# Method 3: Using declare
declare -a servers=("web01" "web02" "db01")
```

#### Accessing Array Elements

```bash
fruits=("apple" "banana" "cherry" "mango")

# Access single element (index starts at 0!)
echo ${fruits[0]}    # apple
echo ${fruits[1]}    # banana
echo ${fruits[3]}    # mango

# Access ALL elements
echo ${fruits[@]}    # apple banana cherry mango
echo ${fruits[*]}    # apple banana cherry mango

# Get the LENGTH of the array (how many items)
echo ${#fruits[@]}   # 4

# Get the LENGTH of a specific element
echo ${#fruits[0]}   # 5  (because "apple" has 5 characters)
```

**Explanation:**
- `${fruits[0]}` — Get the first item (index 0)
- `${fruits[@]}` — Get ALL items (the `@` means "everything")
- `${#fruits[@]}` — The `#` before the name gives you the **count**

#### Modifying Arrays

```bash
fruits=("apple" "banana" "cherry")

# Add an element to the end
fruits+=("mango")
echo ${fruits[@]}    # apple banana cherry mango

# Change an element
fruits[1]="blueberry"
echo ${fruits[@]}    # apple blueberry cherry mango

# Delete an element
unset fruits[2]
echo ${fruits[@]}    # apple blueberry mango
# ⚠️ Note: This leaves a gap! Index 2 is now empty, but 0, 1, 3 still exist.
```

#### Looping Through Arrays

```bash
servers=("web01" "web02" "db01" "cache01")

# Loop through values
for server in "${servers[@]}"; do
    echo "Checking server: $server"
done

# Loop through indices (index numbers)
for i in "${!servers[@]}"; do
    echo "Index $i → ${servers[$i]}"
done
```

**Output:**
```
Index 0 → web01
Index 1 → web02
Index 2 → db01
Index 3 → cache01
```

**Explanation:**
- `"${servers[@]}"` — Expands to all values. The quotes are important — they handle values with spaces correctly.
- `"${!servers[@]}"` — The `!` gives you the **indices** (0, 1, 2, 3) instead of the values.

#### Slicing Arrays

```bash
fruits=("apple" "banana" "cherry" "mango" "grape")

# Get a slice — ${array[@]:start:count}
echo ${fruits[@]:1:3}    # banana cherry mango
#                ↑ start at index 1, take 3 items

echo ${fruits[@]:2}      # cherry mango grape
#                ↑ start at index 2, take everything after
```

### Associative Arrays (Key-Value Pairs / Dictionaries)

These are like dictionaries — instead of numbers, each item has a **name (key)**.

> ⚠️ Associative arrays require Bash **version 4+**. Check with `bash --version`.

#### Creating Associative Arrays

```bash
# You MUST use declare -A (capital A!) for associative arrays
declare -A user_info

user_info[name]="Ritesh"
user_info[role]="DevOps Engineer"
user_info[city]="Bangalore"

# Or create all at once
declare -A server_ips=(
    [web01]="192.168.1.10"
    [web02]="192.168.1.11"
    [db01]="192.168.1.20"
)
```

#### Accessing Associative Arrays

```bash
declare -A server_ips=(
    [web01]="192.168.1.10"
    [web02]="192.168.1.11"
    [db01]="192.168.1.20"
)

# Access a value by key
echo ${server_ips[web01]}    # 192.168.1.10

# Get ALL values
echo ${server_ips[@]}        # 192.168.1.10 192.168.1.11 192.168.1.20

# Get ALL keys
echo ${!server_ips[@]}       # web01 web02 db01

# Get the count
echo ${#server_ips[@]}       # 3
```

#### Looping Through Associative Arrays

```bash
declare -A services=(
    [nginx]="running"
    [mysql]="stopped"
    [redis]="running"
)

for service in "${!services[@]}"; do
    echo "$service is ${services[$service]}"
done
```

**Output:**
```
nginx is running
mysql is stopped
redis is running
```

**Explanation:**
- `"${!services[@]}"` gives us all the **keys** (nginx, mysql, redis)
- `${services[$service]}` gives us the **value** for each key

---

### Real-World Example: Check if Servers are Reachable

```bash
#!/bin/bash

servers=("google.com" "github.com" "nonexistent.server.xyz")

for server in "${servers[@]}"; do
    if ping -c 1 -W 2 "$server" &>/dev/null; then
        echo "✅ $server is reachable"
    else
        echo "❌ $server is NOT reachable"
    fi
done
```

**What this does:**
- Loops through a list of servers
- Pings each one (`-c 1` = send 1 packet, `-W 2` = wait 2 seconds max)
- `&>/dev/null` hides the ping output (we only care about success/failure)
- Reports which are reachable and which are not

## 🪤 Part 2: Traps — Cleaning Up After Yourself

### What is a Trap?

A **trap** is a way to say: *"Hey Bash, if THIS signal/event happens, run THIS command."*

Think of it like a **safety net**. Even if your script crashes, gets killed, or someone presses Ctrl+C — you can still run cleanup code.

### Why Do You Need Traps?

Imagine your script:
1. Creates a temporary file
2. Does some processing
3. Deletes the temporary file

But what if it **crashes at step 2**? The temp file is left behind! Traps fix this.

### Basic Trap Syntax

```bash
trap 'commands_to_run' SIGNAL
```

### The Most Common Trap — Cleanup on EXIT

```bash
#!/bin/bash

# Create a temporary file
TMPFILE=$(mktemp /tmp/myapp.XXXXXX)
echo "Created temp file: $TMPFILE"

# Set a trap — when the script exits (for ANY reason), delete the temp file
trap 'echo "Cleaning up..."; rm -f "$TMPFILE"' EXIT

# Do some work
echo "Doing important work..."
echo "Some data" > "$TMPFILE"
cat "$TMPFILE"

# Even if we don't explicitly delete the file, the trap will handle it!
echo "Script finished."
```

**Output:**
```
Created temp file: /tmp/myapp.abc123
Doing important work...
Some data
Script finished.
Cleaning up...
```

**Explanation:**
- `mktemp /tmp/myapp.XXXXXX` — Creates a unique temporary file (the `XXXXXX` gets replaced with random characters)
- `trap '...' EXIT` — The `EXIT` signal fires whenever the script ends — whether it finishes normally, hits an error, or gets killed
- The cleanup command runs **automatically** at the end

### Trap with a Cleanup Function (Better for Complex Cleanup)

```bash
#!/bin/bash

TMPDIR=$(mktemp -d /tmp/myapp.XXXXXX)
LOGFILE="$TMPDIR/app.log"

cleanup() {
    echo "🧹 Cleaning up temporary files..."
    rm -rf "$TMPDIR"
    echo "✅ Cleanup complete."
}

# Register the cleanup function to run on EXIT
trap cleanup EXIT

# Do work
echo "Working in $TMPDIR"
echo "Log entry 1" >> "$LOGFILE"
echo "Log entry 2" >> "$LOGFILE"

echo "Work done!"
# No need to manually call cleanup — the trap handles it!
```

**Explanation:**
- Instead of putting commands directly in the trap, we defined a `cleanup()` function
- This is cleaner and easier to maintain when you have multiple things to clean up
- `mktemp -d` creates a temporary **directory** instead of a file

### Trapping Multiple Signals

```bash
#!/bin/bash

trap 'echo "Caught SIGINT (Ctrl+C)! Exiting gracefully..."; exit 1' SIGINT
trap 'echo "Caught SIGTERM! Someone killed me..."; exit 1' SIGTERM
trap 'echo "Script is exiting. Final cleanup..."; rm -f /tmp/mylock' EXIT

echo "Script is running. Press Ctrl+C to test..."
echo "PID: $$"

# Create a lock file
touch /tmp/mylock

# Simulate a long-running process
while true; do
    echo "Working... ($(date))"
    sleep 2
done
```

**What each signal means:**
| Signal | Triggered By | Number |
|--------|-------------|--------|
| `SIGINT` | User presses **Ctrl+C** | 2 |
| `SIGTERM` | `kill <pid>` command (default signal) | 15 |
| `SIGKILL` | `kill -9 <pid>` — **Cannot be trapped!** | 9 |
| `SIGHUP` | Terminal is closed | 1 |
| `EXIT` | Script finishes (any reason) | N/A |

> ⚠️ **Important:** You **cannot** trap `SIGKILL` (kill -9). It's the operating system's "nuclear option" — it kills the process immediately, no questions asked.

## 🚦 Part 3: Signal Handling — Graceful Shutdowns

### Why Care About Signals?

When you're running a script that:
- Has a database connection open
- Is writing to a file
- Has child processes running
- Holds a lock file

You don't want it to just **die** suddenly. You want it to **shut down gracefully** — close connections, finish writes, release locks, stop child processes.

### Building a Graceful Shutdown Handler

```bash
#!/bin/bash

# Global flag to track if we should keep running
KEEP_RUNNING=true

# Graceful shutdown function
graceful_shutdown() {
    echo ""
    echo "⚠️  Received shutdown signal!"
    echo "🔄 Finishing current task..."
    KEEP_RUNNING=false
}

# Final cleanup
cleanup() {
    echo "🧹 Releasing resources..."
    # In real life: close DB connections, remove lock files, etc.
    rm -f /tmp/myapp.lock
    echo "✅ Shutdown complete."
}

# Set up traps
trap graceful_shutdown SIGINT SIGTERM
trap cleanup EXIT

# Create lock file
touch /tmp/myapp.lock
echo "🚀 Application started (PID: $$)"

# Main loop
counter=0
while $KEEP_RUNNING; do
    counter=$((counter + 1))
    echo "Processing batch #$counter..."
    sleep 3  # Simulate work

    # Check if we should stop
    if ! $KEEP_RUNNING; then
        echo "Finishing batch #$counter before shutting down..."
    fi
done

echo "Main loop ended."
# cleanup will run automatically via the EXIT trap
```

**How this works:**
1. When the script receives `SIGINT` (Ctrl+C) or `SIGTERM` (kill), it sets `KEEP_RUNNING=false`
2. The `while` loop checks this flag and stops gracefully
3. The current batch finishes before the script exits
4. The `EXIT` trap calls `cleanup()` automatically

This is exactly how **production services** handle shutdown signals — they finish what they're doing before stopping.

### Handling Child Processes

```bash
#!/bin/bash

CHILD_PID=""

cleanup() {
    if [[ -n "$CHILD_PID" ]]; then
        echo "Stopping child process (PID: $CHILD_PID)..."
        kill "$CHILD_PID" 2>/dev/null
        wait "$CHILD_PID" 2>/dev/null
        echo "Child process stopped."
    fi
}

trap cleanup EXIT SIGINT SIGTERM

# Start a background process
echo "Starting background worker..."
sleep 1000 &    # This is our "child process"
CHILD_PID=$!     # $! = PID of the last background process

echo "Parent script running (PID: $$)"
echo "Child process running (PID: $CHILD_PID)"
echo "Press Ctrl+C to stop..."

# Wait for the child process
wait "$CHILD_PID"
```

**Explanation:**
- `sleep 1000 &` — The `&` runs it in the background
- `$!` — Special variable that holds the PID of the last background process
- `wait "$CHILD_PID"` — Pauses the parent script until the child finishes
- When you press Ctrl+C, the trap catches it and kills the child process cleanly

## 🐛 Part 4: Debugging Bash Scripts

### The Problem

Bash scripts can be **really tricky** to debug. There are no breakpoints, no fancy IDE debuggers. But Bash gives you some powerful built-in tools.

### Method 1: `set -x` — Trace Mode (The Most Useful!)

This makes Bash **print every command before executing it**.

```bash
#!/bin/bash
set -x    # Turn on tracing

name="DevOps"
greeting="Hello, $name!"
echo "$greeting"

set +x    # Turn off tracing
echo "This line won't be traced"
```

**Output:**
```
+ name=DevOps
+ greeting='Hello, DevOps!'
+ echo 'Hello, DevOps!'
Hello, DevOps!
+ set +x
This line won't be traced
```

**Explanation:**
- Lines starting with `+` are the **trace** — they show what Bash is actually executing
- You can see how variables get expanded (`$name` → `DevOps`)
- `set -x` turns it ON, `set +x` turns it OFF
- You can also run a script with tracing from outside: `bash -x myscript.sh`

### Method 2: `set -e` — Exit on Error

By default, Bash **keeps running** even when a command fails. This can cause cascading disasters!

```bash
#!/bin/bash
set -e    # Exit immediately if any command fails

echo "Step 1: Creating directory..."
mkdir /some/directory/that/doesnt/exist    # This will fail!

echo "Step 2: This will NEVER run because step 1 failed"
```

**Without `set -e`:** Both steps would try to run (step 2 would run even though step 1 failed).
**With `set -e`:** The script stops immediately at the failed `mkdir`.

### Method 3: `set -u` — Treat Unset Variables as Errors

```bash
#!/bin/bash
set -u    # Error on unset variables

echo "My name is $NAME"    # ERROR! $NAME was never defined
```

**Output:**
```
bash: NAME: unbound variable
```

**Without `set -u`:** Bash would silently use an empty string for `$NAME`, which can cause subtle bugs.

### The Holy Trinity: `set -euo pipefail`

Most production scripts start with this:

```bash
#!/bin/bash
set -euo pipefail

# -e → Exit on any error
# -u → Error on undefined variables
# -o pipefail → If any command in a pipeline fails, the whole pipeline fails
```

#### What Does `pipefail` Do?

```bash
# Without pipefail:
false | true | echo "hello"
echo $?    # 0 (success!) — But the first command failed!

# With pipefail:
set -o pipefail
false | true | echo "hello"
echo $?    # 1 (failure!) — Now the pipeline correctly reports the failure
```

**Explanation:**
- Normally, the exit status of a pipeline is the exit status of the **last** command
- With `pipefail`, if **any** command in the pipeline fails, the whole pipeline reports failure
- This is crucial for catching errors in commands like `cat file.txt | grep "pattern" | wc -l`

### Method 4: Custom Debug Function

```bash
#!/bin/bash

# Set DEBUG=1 to enable, DEBUG=0 to disable
DEBUG=${DEBUG:-0}

debug() {
    if [[ "$DEBUG" -eq 1 ]]; then
        echo "[DEBUG] $(date '+%H:%M:%S') — $*" >&2
    fi
}

debug "Script starting..."
debug "Processing files..."

for file in /etc/*.conf; do
    debug "Found config file: $file"
done

debug "Script finished."
```

**Run normally (no debug output):**
```bash
./myscript.sh
```

**Run with debug enabled:**
```bash
DEBUG=1 ./myscript.sh
```

**Output (with DEBUG=1):**
```
[DEBUG] 14:30:01 — Script starting...
[DEBUG] 14:30:01 — Processing files...
[DEBUG] 14:30:01 — Found config file: /etc/resolv.conf
[DEBUG] 14:30:01 — Found config file: /etc/sysctl.conf
[DEBUG] 14:30:01 — Script finished.
```

**Explanation:**
- `${DEBUG:-0}` — Use the value of `$DEBUG` if it's set, otherwise default to `0`
- `>&2` — Print debug messages to **stderr** (standard error) so they don't mix with normal output
- You can enable/disable debug without changing the script!

### Method 5: Using `trap` for Debugging with `ERR`

```bash
#!/bin/bash
set -e

trap 'echo "❌ ERROR at line $LINENO: Command \"$BASH_COMMAND\" failed with exit code $?"' ERR

echo "Step 1: OK"
echo "Step 2: OK"
ls /nonexistent/path    # This will fail
echo "Step 3: This won't run"
```

**Output:**
```
Step 1: OK
Step 2: OK
❌ ERROR at line 7: Command "ls /nonexistent/path" failed with exit code 2
```

**Explanation:**
- `$LINENO` — The line number where the error occurred
- `$BASH_COMMAND` — The actual command that failed
- `$?` — The exit code of the failed command
- This gives you a **beautiful error message** that tells you exactly what went wrong and where

### Method 6: Bash Debug Trap (Step-by-Step Execution)

```bash
#!/bin/bash

trap 'read -p "[$LINENO] $BASH_COMMAND → Press Enter to continue..."' DEBUG

x=10
y=20
z=$((x + y))
echo "Result: $z"
```

**How it works:**
- The `DEBUG` trap fires **before every single command**
- It pauses and shows you the line number and command about to run
- You press Enter to execute it — like a **breakpoint debugger!**

> 💡 This is super useful for complex scripts. It's like stepping through code line by line.

## 📋 Quick Reference — Common Array Operations

| Operation | Syntax |
|-----------|--------|
| Create array | `arr=("a" "b" "c")` |
| Access element | `${arr[0]}` |
| All elements | `${arr[@]}` |
| Array length | `${#arr[@]}` |
| Add element | `arr+=("d")` |
| Delete element | `unset arr[1]` |
| Slice | `${arr[@]:1:2}` |
| All indices | `${!arr[@]}` |
| Associative array | `declare -A arr` |
| Check if key exists | `[[ -v arr[key] ]]` |

## 📋 Quick Reference — Signals

| Signal | Number | Meaning | Trappable? |
|--------|--------|---------|------------|
| `SIGHUP` | 1 | Terminal closed / hangup | ✅ Yes |
| `SIGINT` | 2 | Ctrl+C pressed | ✅ Yes |
| `SIGQUIT` | 3 | Ctrl+\ pressed (core dump) | ✅ Yes |
| `SIGKILL` | 9 | Force kill — instant death | ❌ No |
| `SIGTERM` | 15 | Polite kill (default `kill`) | ✅ Yes |
| `SIGSTOP` | 19 | Pause process (Ctrl+Z) | ❌ No |
| `SIGCONT` | 18 | Resume paused process | ✅ Yes |
| `EXIT` | N/A | Script exits (Bash-specific) | ✅ Yes |
| `ERR` | N/A | Command returns non-zero (Bash) | ✅ Yes |
| `DEBUG` | N/A | Before each command (Bash) | ✅ Yes |

## 📋 Quick Reference — Debug Flags

| Flag | Meaning |
|------|---------|
| `set -x` | Print every command before executing (trace mode) |
| `set +x` | Turn off trace mode |
| `set -e` | Exit immediately on error |
| `set -u` | Error on undefined variables |
| `set -o pipefail` | Pipeline fails if any command fails |
| `set -euo pipefail` | All of the above — **the gold standard** |
| `bash -x script.sh` | Run script with tracing from outside |
| `bash -n script.sh` | Check syntax without running (dry run) |

## 🎯 Key Takeaways

1. **Arrays** let you store lists of values — use indexed arrays for ordered data, associative arrays for key-value pairs
2. **Traps** are your safety net — always clean up temp files, lock files, and child processes
3. **Signal handling** lets your scripts shut down gracefully — finish current work, release resources, then exit
4. **`set -euo pipefail`** should be at the top of every production script — it catches errors early
5. **`set -x`** is your best friend for debugging — it shows you exactly what Bash is doing
6. **The ERR trap** with `$LINENO` and `$BASH_COMMAND` gives you beautiful error messages
7. **Never forget:** `SIGKILL` (kill -9) cannot be trapped — it's the last resort
