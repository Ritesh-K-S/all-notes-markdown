# 2.14 — Cron Jobs & Task Scheduling (crontab, at, systemd timers)

> **One Line Summary:** Need something to run automatically at a specific time — every day, every hour, or just once? That's what cron jobs, `at`, and systemd timers are for. They're like alarm clocks for your Linux system.

---

## 🤔 Why Do We Need Task Scheduling?

As a DevOps engineer, you'll have tons of repetitive tasks:

- Take database backups **every night at 2 AM**
- Clear old log files **every Sunday**
- Check disk space **every 30 minutes**
- Send a report email **every Monday morning**
- Rotate SSL certificates **every 60 days**
- Run health checks **every 5 minutes**

You don't want to wake up at 2 AM to run a backup command manually, right? 😄

That's where **task scheduling** comes in. You tell Linux: *"Run this command at this time, automatically."*

There are **three main ways** to schedule tasks in Linux:

| Tool | Best For | Type |
|------|----------|------|
| **cron / crontab** | Recurring tasks (every X minutes/hours/days) | Repeating |
| **at** | One-time tasks (run once at a specific time) | One-time |
| **systemd timers** | Modern replacement for cron (more features) | Both |

Let's learn all three!

## ⏰ Part 1: Cron & Crontab — The Classic Scheduler

### What is Cron?

**Cron** is a background service (daemon) that runs on every Linux system. It wakes up **every minute**, checks if there are any scheduled tasks to run, and executes them.

**Crontab** (cron table) is the file where you define your scheduled tasks.

### How Cron Works

```
Every minute, the cron daemon wakes up:
    ↓
Reads the crontab files for all users
    ↓
Checks: "Is it time to run any of these tasks?"
    ↓
If yes → Runs the command
If no  → Goes back to sleep for 1 minute
```

### Managing Your Crontab

```bash
# View your current cron jobs
crontab -l

# Edit your cron jobs (opens in your default editor)
crontab -e

# Remove ALL your cron jobs (careful!)
crontab -r

# View cron jobs for a specific user (need root)
sudo crontab -u username -l

# Edit cron jobs for a specific user
sudo crontab -u username -e
```

### The Cron Syntax — Understanding the 5 Stars

This is the most important thing to learn. Every cron job has this format:

```
┌───────────── Minute (0-59)
│ ┌───────────── Hour (0-23)
│ │ ┌───────────── Day of Month (1-31)
│ │ │ ┌───────────── Month (1-12)
│ │ │ │ ┌───────────── Day of Week (0-7, where 0 and 7 = Sunday)
│ │ │ │ │
* * * * *  command_to_run
```

Each `*` means **"every"**. So `* * * * *` means "every minute of every hour of every day of every month on every day of the week" — basically **every single minute**.

### Let's Learn with Examples (This is the Best Way!)

```bash
# ──────────────────────────────────────────────────
# Every minute
* * * * * /home/user/check-health.sh

# ──────────────────────────────────────────────────
# Every 5 minutes
*/5 * * * * /home/user/check-disk.sh
# The */5 means "every 5th minute" (0, 5, 10, 15, 20...)

# ──────────────────────────────────────────────────
# Every hour (at minute 0)
0 * * * * /home/user/hourly-report.sh
# Runs at 1:00, 2:00, 3:00, 4:00... etc.

# ──────────────────────────────────────────────────
# Every day at 2:30 AM
30 2 * * * /home/user/backup.sh
# Minute=30, Hour=2, rest are * (every day)

# ──────────────────────────────────────────────────
# Every Monday at 9:00 AM
0 9 * * 1 /home/user/weekly-report.sh
# Day of week: 0=Sun, 1=Mon, 2=Tue, 3=Wed, 4=Thu, 5=Fri, 6=Sat

# ──────────────────────────────────────────────────
# Every weekday (Mon-Fri) at 6 PM
0 18 * * 1-5 /home/user/end-of-day.sh
# 1-5 means Monday through Friday

# ──────────────────────────────────────────────────
# 1st of every month at midnight
0 0 1 * * /home/user/monthly-cleanup.sh

# ──────────────────────────────────────────────────
# Every 15 minutes between 9 AM and 5 PM
*/15 9-17 * * * /home/user/business-hours-check.sh

# ──────────────────────────────────────────────────
# Every January 1st at midnight (Happy New Year! 🎉)
0 0 1 1 * /home/user/new-year-task.sh

# ──────────────────────────────────────────────────
# Multiple specific times — At 8 AM, 12 PM, and 6 PM daily
0 8,12,18 * * * /home/user/three-times-daily.sh
# The comma means "and" — hour 8 AND 12 AND 18
```

### Special Cron Shortcuts

Instead of the 5-field syntax, cron supports these shortcuts:

```bash
@reboot    /home/user/startup.sh      # Run once at system boot
@yearly    /home/user/yearly.sh       # Same as: 0 0 1 1 *
@annually  /home/user/yearly.sh       # Same as @yearly
@monthly   /home/user/monthly.sh      # Same as: 0 0 1 * *
@weekly    /home/user/weekly.sh       # Same as: 0 0 * * 0
@daily     /home/user/daily.sh        # Same as: 0 0 * * *
@midnight  /home/user/midnight.sh     # Same as @daily
@hourly    /home/user/hourly.sh       # Same as: 0 * * * *
```

### Real-World Example: Database Backup Cron Job

```bash
# Edit crontab
crontab -e

# Add this line — backup MySQL every night at 2 AM
0 2 * * * /usr/bin/mysqldump -u root -p'MyPassword123' --all-databases | gzip > /backups/db-$(date +\%Y\%m\%d).sql.gz 2>> /var/log/backup.log
```

**Line-by-line explanation:**
- `0 2 * * *` — At 2:00 AM, every day
- `/usr/bin/mysqldump` — Full path to the backup command (always use full paths in cron!)
- `-u root -p'MyPassword123'` — Database credentials
- `--all-databases` — Back up everything
- `| gzip >` — Compress the backup
- `/backups/db-$(date +\%Y\%m\%d).sql.gz` — Save with today's date in the filename
- `2>> /var/log/backup.log` — Append errors to a log file

> ⚠️ **Note the `\%`!** In crontab, the `%` character has special meaning (it becomes a newline). So you must escape it with `\%` when using it in `date` formats.

> ⚠️ **Security Note:** Storing passwords directly in crontab is not recommended. Use a `.my.cnf` credentials file or environment variables instead.

### Important Cron Tips & Gotchas

#### 1. Always Use Full Paths

```bash
# ❌ BAD — cron doesn't know where "python3" is
* * * * * python3 /home/user/script.py

# ✅ GOOD — use the full path
* * * * * /usr/bin/python3 /home/user/script.py

# 💡 Find the full path with:
which python3    # /usr/bin/python3
```

**Why?** Cron runs with a very **minimal PATH**. It doesn't load your `.bashrc` or `.bash_profile`. So commands that work in your terminal might not work in cron.

#### 2. Redirect Output (or Cron Will Email You!)

```bash
# ❌ BAD — output goes to email (or gets lost)
0 2 * * * /home/user/backup.sh

# ✅ GOOD — redirect output to a log file
0 2 * * * /home/user/backup.sh >> /var/log/backup.log 2>&1

# ✅ GOOD — discard output entirely (if you don't need it)
0 2 * * * /home/user/backup.sh > /dev/null 2>&1
```

**Explanation of `>> /var/log/backup.log 2>&1`:**
- `>>` — Append normal output (stdout) to the log file
- `2>&1` — Redirect errors (stderr) to the same place as stdout
- This means BOTH normal output and errors go to the log file

#### 3. Environment Variables in Cron

```bash
# You can set environment variables at the top of your crontab
SHELL=/bin/bash
PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
MAILTO=admin@example.com
HOME=/home/user

# Now your cron jobs below will use these settings
0 2 * * * /home/user/backup.sh
```

#### 4. System-Wide Cron Directories

Besides per-user crontabs, Linux has system-wide cron directories:

```bash
/etc/crontab              # System-wide crontab file (has an extra "user" field)
/etc/cron.d/              # Drop cron files here for system jobs
/etc/cron.daily/          # Scripts here run daily
/etc/cron.hourly/         # Scripts here run hourly
/etc/cron.weekly/         # Scripts here run weekly
/etc/cron.monthly/        # Scripts here run monthly
```

Just drop your script into `/etc/cron.daily/` and it'll run once a day automatically!

```bash
# Make sure it's executable!
sudo cp my-cleanup.sh /etc/cron.daily/
sudo chmod +x /etc/cron.daily/my-cleanup.sh
```

### Viewing and Debugging Cron

```bash
# Check if cron service is running
sudo systemctl status cron       # Debian/Ubuntu
sudo systemctl status crond      # RHEL/CentOS

# View cron logs
sudo grep CRON /var/log/syslog          # Debian/Ubuntu
sudo grep CRON /var/log/cron            # RHEL/CentOS
sudo journalctl -u cron --since today   # With systemd

# Test your cron expression (check what times it would run)
# Use websites like crontab.guru — it's amazing!
```

> 💡 **Pro Tip:** Use [crontab.guru](https://crontab.guru) to test and visualize cron expressions. Just paste your cron schedule and it tells you in plain English what it means!

## 🎯 Part 2: `at` — One-Time Scheduled Tasks

### What is `at`?

While cron is for **recurring** tasks, `at` is for **one-time** tasks. You say: *"Run this command at 3 PM tomorrow"* — and it runs it once, then forgets about it.

### Installing `at`

```bash
# Debian/Ubuntu
sudo apt install at

# RHEL/CentOS
sudo yum install at

# Start and enable the at daemon
sudo systemctl start atd
sudo systemctl enable atd
```

### Basic `at` Usage

```bash
# Schedule a command for a specific time
at 3:00 PM
> /home/user/backup.sh          # Type your command
> Ctrl+D                         # Press Ctrl+D to save and exit

# Schedule for a specific date and time
at 10:00 AM Dec 25
> echo "Merry Christmas!" | mail -s "Holiday" admin@example.com
> Ctrl+D

# Schedule relative to now
at now + 30 minutes
> /home/user/restart-service.sh
> Ctrl+D

at now + 2 hours
> echo "Reminder: Check deployment" | mail -s "Reminder" me@example.com
> Ctrl+D

at now + 1 day
> /home/user/cleanup.sh
> Ctrl+D
```

### Using `at` with a Script

```bash
# Pass commands via echo and pipe
echo "/home/user/backup.sh" | at 2:00 AM

# Run multiple commands
at now + 10 minutes << 'EOF'
echo "Starting maintenance..."
systemctl stop nginx
/home/user/update.sh
systemctl start nginx
echo "Maintenance complete."
EOF
```

**Explanation:**
- `<< 'EOF'` is a **here document** — everything between `EOF` markers is passed as input to the `at` command
- The single quotes around `'EOF'` prevent variable expansion inside the block

### Managing `at` Jobs

```bash
# List all pending at jobs
atq
# Output:
# 3   Wed Jun 15 14:00:00 2025 a user
# 4   Thu Jun 16 02:00:00 2025 a user

# View the details of a specific job (job number 3)
at -c 3

# Remove/cancel a job
atrm 3     # Remove job number 3
```

### `at` Time Formats

```bash
at 2:00 PM                    # Today at 2 PM
at 14:00                      # Same thing in 24-hour format
at 2:00 PM tomorrow           # Tomorrow at 2 PM
at 2:00 PM next week          # Next week, same day
at now + 30 minutes           # 30 minutes from now
at now + 2 hours              # 2 hours from now
at now + 3 days               # 3 days from now
at midnight                   # Tonight at midnight
at noon                       # Today at 12 PM
at teatime                    # Today at 4 PM (yes, really! 🫖)
at 10:00 AM Jul 4             # Specific date
```

### Controlling Who Can Use `at`

```bash
/etc/at.allow     # If this file exists, ONLY users listed here can use at
/etc/at.deny      # Users listed here are DENIED access to at

# If neither file exists, only root can use at (on some systems, everyone can)
```

## ⏱️ Part 3: Systemd Timers — The Modern Alternative

### Why Systemd Timers Over Cron?

Systemd timers are the **modern replacement** for cron. Here's why:

| Feature | Cron | Systemd Timer |
|---------|------|---------------|
| Logging | Manual (redirect to file) | Automatic (journalctl) |
| Dependencies | None | Can depend on other services |
| Missed runs | Lost if system was off | Can catch up on missed runs |
| Accuracy | Minute-level only | Can go down to seconds/microseconds |
| Resource control | None | CPU/Memory limits via cgroups |
| Status checking | Hard to check | `systemctl status` shows everything |
| Environment | Minimal, often causes issues | Full systemd environment |

### How Systemd Timers Work

You need **two files**:

1. **A `.service` file** — Defines WHAT to run
2. **A `.timer` file** — Defines WHEN to run it

They must have the **same name** (e.g., `backup.service` and `backup.timer`).

### Example: Daily Backup with Systemd Timer

#### Step 1: Create the Service File

```bash
sudo nano /etc/systemd/system/backup.service
```

```ini
[Unit]
Description=Daily Database Backup
# Optional: only run after network is available
After=network.target

[Service]
Type=oneshot
ExecStart=/home/user/backup.sh
User=user
Group=user

# Optional: Set environment variables
Environment="BACKUP_DIR=/backups"
Environment="DB_HOST=localhost"

# Optional: Resource limits
MemoryMax=512M
CPUQuota=50%
```

**Explanation:**
- `Type=oneshot` — This is a one-shot task (runs and exits), not a long-running service
- `ExecStart` — The script/command to run
- `User/Group` — Run as this user (not root, for security)
- `After=network.target` — Wait for the network to be ready before running
- `MemoryMax/CPUQuota` — Resource limits (cron can't do this!)

#### Step 2: Create the Timer File

```bash
sudo nano /etc/systemd/system/backup.timer
```

```ini
[Unit]
Description=Run backup every day at 2 AM

[Timer]
OnCalendar=*-*-* 02:00:00
# Translation: Every year, every month, every day at 02:00:00

Persistent=true
# If the system was off at 2 AM, run the backup as soon as it boots

RandomizedDelaySec=300
# Add a random delay of up to 5 minutes (to avoid thundering herd)

[Install]
WantedBy=timers.target
```

**Explanation:**
- `OnCalendar` — When to run (we'll cover the syntax below)
- `Persistent=true` — **This is amazing!** If your system was off at the scheduled time, it will run the task when the system comes back up. Cron can't do this!
- `RandomizedDelaySec=300` — Add a random delay up to 300 seconds. This is useful when multiple servers have the same timer — it prevents them all from hitting the database at the exact same second.

#### Step 3: Enable and Start the Timer

```bash
# Reload systemd to pick up the new files
sudo systemctl daemon-reload

# Enable the timer (so it starts on boot)
sudo systemctl enable backup.timer

# Start the timer now
sudo systemctl start backup.timer

# Check the timer status
sudo systemctl status backup.timer

# List ALL active timers
sudo systemctl list-timers --all
```

**Output of `systemctl list-timers`:**
```
NEXT                         LEFT          LAST                         PASSED   UNIT             ACTIVATES
Wed 2025-06-18 02:00:00 IST  11h left      Tue 2025-06-17 02:00:23 IST  12h ago  backup.timer     backup.service
```

This tells you:
- **NEXT** — When it will run next
- **LEFT** — How much time until the next run
- **LAST** — When it last ran
- **PASSED** — How long ago it ran
- **UNIT** — The timer name
- **ACTIVATES** — The service it triggers

### OnCalendar Syntax (Systemd Timer Schedule)

The format is: `DayOfWeek Year-Month-Day Hour:Minute:Second`

```bash
# ──────────────────────────────────────────────────
# Every day at 2 AM
OnCalendar=*-*-* 02:00:00

# ──────────────────────────────────────────────────
# Every hour
OnCalendar=*-*-* *:00:00
# Or use the shorthand:
OnCalendar=hourly

# ──────────────────────────────────────────────────
# Every 15 minutes
OnCalendar=*-*-* *:00/15:00
# 00/15 means: starting at 0, every 15 (0, 15, 30, 45)

# ──────────────────────────────────────────────────
# Every Monday at 9 AM
OnCalendar=Mon *-*-* 09:00:00

# ──────────────────────────────────────────────────
# Every weekday at 6 PM
OnCalendar=Mon..Fri *-*-* 18:00:00

# ──────────────────────────────────────────────────
# First of every month at midnight
OnCalendar=*-*-01 00:00:00

# ──────────────────────────────────────────────────
# Every January 1st
OnCalendar=*-01-01 00:00:00

# ──────────────────────────────────────────────────
# Every 5 minutes
OnCalendar=*-*-* *:00/5:00

# ──────────────────────────────────────────────────
# Shorthands
OnCalendar=minutely
OnCalendar=hourly
OnCalendar=daily
OnCalendar=weekly
OnCalendar=monthly
OnCalendar=yearly
```

### Testing Your OnCalendar Expression

```bash
# Super useful! Validate your timer expression:
systemd-analyze calendar "*-*-* 02:00:00"

# Output:
#   Original form: *-*-* 02:00:00
#   Normalized form: *-*-* 02:00:00
#   Next elapse: Wed 2025-06-18 02:00:00 IST
#   (in UTC): Tue 2025-06-17 20:30:00 UTC
#   From now: 11h 23min left

# Check when it would run next (multiple times):
systemd-analyze calendar --iterations=5 "*-*-* 02:00:00"
```

### Monotonic Timers (Relative Time)

Instead of calendar-based schedules, you can use **relative time**:

```ini
[Timer]
# Run 10 minutes after boot
OnBootSec=10min

# Run 1 hour after the timer is started
OnActiveSec=1h

# Run every 30 minutes after the last run completed
OnUnitActiveSec=30min

# Run every 15 minutes after the last time the service was activated
OnUnitInactiveSec=15min
```

**When to use which:**
- `OnCalendar` — When you want **exact times** (2 AM daily, every Monday, etc.)
- `OnBootSec` — When you want to run **X time after boot**
- `OnUnitActiveSec` — When you want to run **every X minutes** (like cron's `*/5`)

### Viewing Timer Logs

```bash
# View logs for your timer's service
sudo journalctl -u backup.service

# View recent logs only
sudo journalctl -u backup.service --since today

# Follow logs in real-time
sudo journalctl -u backup.service -f

# View the last 50 lines
sudo journalctl -u backup.service -n 50
```

### Real-World Example: Log Cleanup Timer

#### The Cleanup Script

```bash
sudo nano /usr/local/bin/cleanup-logs.sh
```

```bash
#!/bin/bash
set -euo pipefail

LOG_DIR="/var/log/myapp"
MAX_AGE_DAYS=30

echo "$(date) — Starting log cleanup..."
echo "Removing logs older than $MAX_AGE_DAYS days from $LOG_DIR"

# Find and delete old log files
deleted_count=$(find "$LOG_DIR" -name "*.log" -type f -mtime +$MAX_AGE_DAYS -delete -print | wc -l)

echo "Deleted $deleted_count old log files."
echo "$(date) — Cleanup complete."
```

```bash
sudo chmod +x /usr/local/bin/cleanup-logs.sh
```

#### The Service File

```bash
sudo nano /etc/systemd/system/cleanup-logs.service
```

```ini
[Unit]
Description=Clean up old application logs

[Service]
Type=oneshot
ExecStart=/usr/local/bin/cleanup-logs.sh
```

#### The Timer File

```bash
sudo nano /etc/systemd/system/cleanup-logs.timer
```

```ini
[Unit]
Description=Run log cleanup every Sunday at 3 AM

[Timer]
OnCalendar=Sun *-*-* 03:00:00
Persistent=true

[Install]
WantedBy=timers.target
```

#### Enable It

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now cleanup-logs.timer

# Verify it's scheduled
sudo systemctl list-timers | grep cleanup
```

## 🔍 Cron vs `at` vs Systemd Timers — When to Use What?

| Scenario | Use |
|----------|-----|
| Recurring task, simple schedule | **Cron** |
| One-time task at a specific time | **`at`** |
| Recurring task with logging, dependencies, resource limits | **Systemd Timer** |
| Need to catch up on missed runs | **Systemd Timer** (Persistent=true) |
| Quick and easy, don't want to create files | **Cron** |
| Enterprise/production workloads | **Systemd Timer** |
| Run something after system boot | **Systemd Timer** (OnBootSec) or **Cron** (@reboot) |

### General Recommendation:

- For **simple, quick tasks** → Use **cron** (it's faster to set up)
- For **production/important tasks** → Use **systemd timers** (better logging, reliability)
- For **one-time tasks** → Use **`at`**

## 📋 Quick Reference — Cron Syntax

```
* * * * * command
│ │ │ │ │
│ │ │ │ └── Day of Week  (0-7, Sun=0 or 7)
│ │ │ └──── Month        (1-12)
│ │ └────── Day of Month  (1-31)
│ └──────── Hour          (0-23)
└────────── Minute        (0-59)

Special characters:
  *     Every value
  */n   Every nth value
  n-m   Range from n to m
  n,m   Value n AND m
```

## 📋 Quick Reference — Common Cron Schedules

| Schedule | Cron Expression |
|----------|----------------|
| Every minute | `* * * * *` |
| Every 5 minutes | `*/5 * * * *` |
| Every hour | `0 * * * *` |
| Every day at midnight | `0 0 * * *` |
| Every day at 2:30 AM | `30 2 * * *` |
| Every Monday at 9 AM | `0 9 * * 1` |
| Weekdays at 6 PM | `0 18 * * 1-5` |
| 1st of month at midnight | `0 0 1 * *` |
| Every 15 min, 9 AM–5 PM | `*/15 9-17 * * *` |
| At reboot | `@reboot` |

## 🎯 Key Takeaways

1. **Cron** is the classic scheduler — learn the 5-star syntax, it's used everywhere
2. Always use **full paths** in cron jobs — cron has a minimal PATH
3. Always **redirect output** in cron — otherwise output goes to email or gets lost
4. Use `\%` in crontab instead of `%` — the percent sign is special in crontab
5. **`at`** is for one-time tasks — schedule it and forget it
6. **Systemd timers** are the modern way — better logging, can catch up on missed runs, resource limits
7. Use `crontab.guru` to test cron expressions and `systemd-analyze calendar` to test timer expressions
8. For production workloads, prefer **systemd timers** over cron
