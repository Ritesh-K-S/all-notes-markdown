# 2.16 — Kernel Basics — Modules, Parameters, Boot Process (GRUB, initramfs)

> **One Line Answer:** The Linux **kernel** is the brain of your operating system — it manages hardware, loads drivers via **modules**, can be tuned using **parameters**, and the whole system comes alive through a well-orchestrated **boot process** involving GRUB and initramfs.

---

## 🤔 Why Should a DevOps Engineer Care About the Kernel?

You might think — *"I just deploy containers and write pipelines, why do I need to know about the kernel?"*

Here's why:

| Situation | Why Kernel Knowledge Helps |
|-----------|---------------------------|
| Container won't start | Could be a missing kernel module (like `overlay` or `br_netfilter`) |
| Network issues in Kubernetes | Might need to tune kernel parameters (`net.ipv4.ip_forward`) |
| Performance problems | Kernel tuning (sysctl) can make or break your server |
| Custom AMI / VM image building | You need to understand boot process and initramfs |
| Debugging server boot failures | GRUB and initramfs knowledge is essential |
| Security hardening | Kernel parameters control security behavior |

Bottom line — the kernel is the **foundation** of everything. Understanding it makes you a **10x better** troubleshooter.

## 🧠 What is the Linux Kernel?

### The Simple Explanation

Think of the kernel as the **middleman** between your software and your hardware.

Your applications can't directly talk to the CPU, RAM, or disk. They need someone to translate — and that someone is the **kernel**.

```
┌──────────────────────────────────────┐
│          Your Applications           │
│   (nginx, docker, python, java...)   │
├──────────────────────────────────────┤
│            System Calls              │
│  (open, read, write, fork, exec...) │
├──────────────────────────────────────┤
│         🧠 LINUX KERNEL 🧠          │
│   ┌──────────┬────────────────┐     │
│   │ Process  │   Memory       │     │
│   │ Manager  │   Manager      │     │
│   ├──────────┼────────────────┤     │
│   │ File     │   Network      │     │
│   │ System   │   Stack        │     │
│   ├──────────┼────────────────┤     │
│   │ Device Drivers (Modules)  │     │
│   └───────────────────────────┘     │
├──────────────────────────────────────┤
│            Hardware                  │
│   (CPU, RAM, Disk, NIC, GPU...)      │
└──────────────────────────────────────┘
```

### What the Kernel Does

| Responsibility | What It Means |
|---------------|---------------|
| **Process Management** | Creates, schedules, and kills processes (every running program) |
| **Memory Management** | Decides how RAM is allocated and freed |
| **File System Management** | Handles reading/writing to disks (ext4, xfs, btrfs...) |
| **Device Management** | Talks to hardware through drivers (loaded as modules) |
| **Networking** | Manages TCP/IP stack, routing, firewalls (iptables/nftables) |
| **Security** | Enforces permissions, capabilities, namespaces, cgroups |

## 🔌 Kernel Modules — Plug and Play for the Kernel

### What Are Kernel Modules?

Here's the thing — the Linux kernel would be **enormous** if it included drivers for every possible piece of hardware ever made. So instead, Linux uses a clever approach: **modules**.

Think of modules as **plugins** for the kernel. They can be:
- **Loaded** when needed (e.g., you plug in a USB drive)
- **Unloaded** when not needed (to save memory)
- **Built into** the kernel itself (always available)

```
┌─────────────────────────────────────────────┐
│                Linux Kernel                  │
│  ┌─────────────────────────────────────┐    │
│  │    Core (always loaded)             │    │
│  │    - Process scheduling             │    │
│  │    - Memory management              │    │
│  │    - Basic file system (VFS)        │    │
│  └─────────────────────────────────────┘    │
│                                              │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐    │
│  │ Module:  │ │ Module:  │ │ Module:  │    │
│  │ ext4     │ │ nvidia   │ │ overlay  │    │
│  │ (fs)     │ │ (gpu)    │ │(docker)  │    │
│  └──────────┘ └──────────┘ └──────────┘    │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐    │
│  │ Module:  │ │ Module:  │ │ Module:  │    │
│  │ iptables │ │ br_net   │ │ bonding  │    │
│  │(firewall)│ │ filter   │ │(network) │    │
│  └──────────┘ └──────────┘ └──────────┘    │
│   ↑ These can be loaded/unloaded anytime    │
└─────────────────────────────────────────────┘
```

### Working with Kernel Modules — Commands

#### 1. List All Currently Loaded Modules

```bash
lsmod
```

**Output looks like:**
```
Module                  Size  Used by
overlay                151552  20
br_netfilter            32768   1
bridge                 311296   1 br_netfilter
ip_tables              32768   3
ext4                   815104   1
```

**What the columns mean:**
- `Module` — Name of the loaded module
- `Size` — How much memory it uses (in bytes)
- `Used by` — How many things depend on it (and what)

#### 2. Get Information About a Module

```bash
modinfo overlay
```

**Output:**
```
filename:       /lib/modules/5.15.0-91-generic/kernel/fs/overlayfs/overlay.ko
license:        GPL
description:    Overlay filesystem
author:         Miklos Szeredi <miklos@szeredi.hu>
depends:
retpoline:      Y
intree:         Y
```

This tells you everything: where the file is, who wrote it, what license it uses, and what it depends on.

#### 3. Load a Module

```bash
# Load a module
sudo modprobe overlay

# modprobe is smart — it automatically loads any dependencies too!
```

**Why `modprobe` and not `insmod`?**

```bash
# insmod — the "dumb" way (you must specify the full path, no dependency handling)
sudo insmod /lib/modules/5.15.0-91-generic/kernel/fs/overlayfs/overlay.ko

# modprobe — the "smart" way (finds the module automatically, handles dependencies)
sudo modprobe overlay
```

**Always use `modprobe`.** It's smarter and safer.

#### 4. Unload a Module

```bash
# Unload a module
sudo modprobe -r overlay

# Or use rmmod (but modprobe -r is preferred)
sudo rmmod overlay
```

> ⚠️ **Warning:** You can only unload a module if nothing is using it. If `Used by` shows a number > 0 in `lsmod`, you can't unload it.

#### 5. Load Modules Automatically at Boot

If you need a module to load every time the system starts:

```bash
# Create a config file
echo "br_netfilter" | sudo tee /etc/modules-load.d/kubernetes.conf
echo "overlay" | sudo tee -a /etc/modules-load.d/kubernetes.conf
```

**This is super common in Kubernetes setups!** You'll see this in almost every k8s installation guide:

```bash
cat <<EOF | sudo tee /etc/modules-load.d/k8s.conf
overlay
br_netfilter
EOF

# Load them right now (without rebooting)
sudo modprobe overlay
sudo modprobe br_netfilter
```

#### 6. Where Are Module Files Stored?

```bash
# All modules for your current kernel
ls /lib/modules/$(uname -r)/

# Output:
# build  kernel  modules.alias  modules.builtin  modules.dep  ...

# The actual .ko files (kernel objects) are in the kernel/ subdirectory
find /lib/modules/$(uname -r)/kernel -name "*.ko" | head -5
```

## ⚙️ Kernel Parameters — Tuning the Kernel at Runtime

### What Are Kernel Parameters?

Kernel parameters are like **settings** or **knobs** you can turn to change how the kernel behaves — **without recompiling or rebooting**.

Think of it like this: your car has settings for seat position, mirror angle, AC temperature. Similarly, the kernel has hundreds of settings you can adjust.

### Viewing Kernel Parameters

```bash
# View ALL kernel parameters (there are hundreds!)
sudo sysctl -a

# This will show something like:
# net.ipv4.ip_forward = 0
# vm.swappiness = 60
# kernel.hostname = myserver
# fs.file-max = 9223372036854775807
# ... hundreds more ...

# View a specific parameter
sysctl net.ipv4.ip_forward
# Output: net.ipv4.ip_forward = 0

# You can also read directly from /proc/sys/
cat /proc/sys/net/ipv4/ip_forward
# Output: 0
```

**Notice the pattern?**
- `sysctl` uses dots: `net.ipv4.ip_forward`
- `/proc/sys/` uses slashes: `/proc/sys/net/ipv4/ip_forward`
- They're the **same thing**, just different ways to access it!

### Setting Kernel Parameters

#### Temporarily (Lost After Reboot)

```bash
# Enable IP forwarding (needed for Docker, Kubernetes, routing)
sudo sysctl -w net.ipv4.ip_forward=1

# Or write directly to /proc/sys
echo 1 | sudo tee /proc/sys/net/ipv4/ip_forward
```

#### Permanently (Survives Reboot)

```bash
# Add to /etc/sysctl.conf or create a file in /etc/sysctl.d/
cat <<EOF | sudo tee /etc/sysctl.d/99-kubernetes.conf
net.bridge.bridge-nf-call-iptables = 1
net.bridge.bridge-nf-call-ip6tables = 1
net.ipv4.ip_forward = 1
EOF

# Apply the changes immediately (without rebooting)
sudo sysctl --system
```

### Most Important Kernel Parameters for DevOps

Here are the parameters you'll use most often:

| Parameter | What It Does | Common Value |
|-----------|-------------|--------------|
| `net.ipv4.ip_forward` | Allows packet forwarding (needed for containers/routing) | `1` |
| `net.bridge.bridge-nf-call-iptables` | Bridge traffic goes through iptables (needed for k8s) | `1` |
| `vm.swappiness` | How aggressively the system uses swap (0 = avoid, 100 = aggressive) | `10` for servers |
| `vm.overcommit_memory` | Controls memory overcommit policy | `1` for Redis |
| `fs.file-max` | Maximum number of open files system-wide | `2097152` |
| `fs.inotify.max_user_watches` | Max filesystem watches (for file watchers like webpack) | `524288` |
| `net.core.somaxconn` | Max socket connection backlog | `65535` |
| `net.ipv4.tcp_max_syn_backlog` | Max SYN queue length | `65535` |
| `kernel.pid_max` | Maximum process ID number | `4194304` |

### Real-World Example: Kubernetes Node Setup

Every Kubernetes installation guide includes something like this:

```bash
# Step 1: Load required modules
cat <<EOF | sudo tee /etc/modules-load.d/k8s.conf
overlay
br_netfilter
EOF

sudo modprobe overlay
sudo modprobe br_netfilter

# Step 2: Set required kernel parameters
cat <<EOF | sudo tee /etc/sysctl.d/k8s.conf
net.bridge.bridge-nf-call-iptables  = 1
net.bridge.bridge-nf-call-ip6tables = 1
net.ipv4.ip_forward                 = 1
EOF

# Step 3: Apply without reboot
sudo sysctl --system

# Step 4: Verify
sysctl net.bridge.bridge-nf-call-iptables
sysctl net.bridge.bridge-nf-call-ip6tables
sysctl net.ipv4.ip_forward
```

Now you know exactly what this does and **why**!

## 🔄 The Linux Boot Process — From Power Button to Login Screen

This is one of the most important things to understand. When you press the power button, a **lot** happens before you see a login prompt. Let's walk through it step by step.

### The Boot Process Overview

```
┌──────────────────────────────────────────────────────────────────┐
│                    LINUX BOOT PROCESS                            │
│                                                                  │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐  │
│  │  Step 1  │───▶│  Step 2  │───▶│  Step 3  │───▶│  Step 4  │  │
│  │ BIOS /   │    │   GRUB   │    │ initramfs│    │  systemd  │  │
│  │  UEFI    │    │(Bootldr) │    │ (initrd) │    │  (init)   │  │
│  └──────────┘    └──────────┘    └──────────┘    └──────────┘  │
│       │               │               │               │         │
│  Power-on self   Loads the       Temporary root    Real OS      │
│  test, find      kernel from     filesystem to     starts up    │
│  bootloader      disk            load drivers      services     │
│                                                                  │
│  ┌──────────┐                                                   │
│  │  Step 5  │                                                   │
│  │  Login   │                                                   │
│  │  Prompt  │                                                   │
│  └──────────┘                                                   │
│  You're in! 🎉                                                  │
└──────────────────────────────────────────────────────────────────┘
```

Let's look at each step in detail.

---

### Step 1: BIOS / UEFI — "Wake Up, Hardware!"

When you press the power button:

1. **BIOS** (Basic Input/Output System) or **UEFI** (Unified Extensible Firmware Interface) wakes up
2. It runs **POST** (Power-On Self-Test) — checks if CPU, RAM, and basic hardware are working
3. It looks for a **bootable device** (hard disk, USB, network) based on boot order
4. It finds the **bootloader** on the boot device and hands over control

**BIOS vs UEFI:**

| Feature | BIOS (Legacy) | UEFI (Modern) |
|---------|--------------|----------------|
| Age | 1975+ | 2005+ |
| Partition Table | MBR (max 2TB disks, 4 partitions) | GPT (no practical limits) |
| Boot Speed | Slower | Faster |
| Security | None | Secure Boot (verifies bootloader signature) |
| Interface | Text-based | Graphical |

> Most modern servers use **UEFI**. But you'll still hear people say "BIOS" out of habit.

---

### Step 2: GRUB — The Bootloader

**GRUB** (GRand Unified Bootloader) is the most common Linux bootloader. Its job is to:

1. Show you a menu to choose which OS/kernel to boot
2. Load the selected kernel into memory
3. Load the **initramfs** into memory
4. Hand over control to the kernel

#### Where GRUB Lives

```bash
# GRUB config file
/boot/grub/grub.cfg       # Auto-generated — DON'T edit this directly!

# GRUB settings you CAN edit
/etc/default/grub          # Your settings go here

# After editing /etc/default/grub, regenerate the config:
sudo update-grub           # Debian/Ubuntu
# or
sudo grub2-mkconfig -o /boot/grub2/grub.cfg   # RHEL/CentOS
```

#### Understanding /etc/default/grub

```bash
cat /etc/default/grub
```

```bash
# Which menu entry to boot by default (0 = first entry)
GRUB_DEFAULT=0

# How long to show the menu (in seconds) before auto-booting
GRUB_TIMEOUT=5

# Extra parameters passed to the kernel at boot
GRUB_CMDLINE_LINUX_DEFAULT="quiet splash"

# Parameters for ALL boot entries (including recovery mode)
GRUB_CMDLINE_LINUX=""
```

**What do those kernel command-line parameters mean?**

| Parameter | What It Does |
|-----------|-------------|
| `quiet` | Don't show detailed boot messages |
| `splash` | Show a graphical splash screen instead |
| `nomodeset` | Disable kernel video driver (useful for GPU issues) |
| `init=/bin/bash` | Boot directly into a bash shell (emergency recovery!) |
| `single` or `1` | Boot into single-user mode (maintenance) |
| `rd.break` | Break into initramfs (for password reset on RHEL/CentOS) |

#### Real Scenario: Recovering a Server That Won't Boot

```bash
# 1. At GRUB menu, press 'e' to edit the boot entry
# 2. Find the line starting with 'linux' or 'linux16'
# 3. Add 'init=/bin/bash' at the end
# 4. Press Ctrl+X or F10 to boot

# You're now in a root shell! You can:
mount -o remount,rw /         # Make filesystem writable
passwd root                    # Reset root password
exec /sbin/init                # Continue normal boot
```

### Step 3: initramfs — The Temporary Root Filesystem

#### The Problem initramfs Solves

Here's the chicken-and-egg problem:

```
🐔 The kernel needs to mount the root filesystem (/) to start the system
🥚 But to mount the root filesystem, it might need drivers (like RAID, LVM, encryption)
🤔 Those drivers are stored ON the root filesystem!

Solution: initramfs — a tiny temporary filesystem loaded into RAM
that contains just enough drivers to mount the REAL root filesystem
```

#### What Is initramfs?

**initramfs** (initial RAM filesystem) is a compressed archive (like a zip file) that contains:
- A minimal filesystem
- Essential kernel modules (drivers)
- Scripts to detect and mount the real root filesystem

```
┌─────────────────────────────────────────────┐
│              initramfs Contents              │
│                                              │
│  /bin/        — Basic commands (busybox)     │
│  /sbin/       — System tools                │
│  /lib/modules — Essential kernel modules     │
│  /etc/        — Minimal config               │
│  /init        — The startup script          │
│                                              │
│  This whole thing is ~30-80MB               │
│  Loaded entirely into RAM                    │
└─────────────────────────────────────────────┘
```

#### Working with initramfs

```bash
# See your initramfs file(s)
ls -lh /boot/initrd*
# -rw-r--r-- 1 root root  56M Jan 15 10:00 /boot/initrd.img-5.15.0-91-generic

# Rebuild initramfs (Debian/Ubuntu)
sudo update-initramfs -u

# Rebuild initramfs (RHEL/CentOS)
sudo dracut --force

# See what's inside initramfs (peek inside the archive)
lsinitramfs /boot/initrd.img-$(uname -r) | head -20

# Or on RHEL/CentOS
lsinitrd /boot/initramfs-$(uname -r).img | head -20
```

#### When Do You Need to Rebuild initramfs?

| Scenario | Why |
|----------|-----|
| Added a new kernel module | Module needs to be included in initramfs |
| Changed root filesystem type | New FS driver must be in initramfs |
| Changed disk configuration (RAID, LVM) | New drivers needed at boot |
| Updated kernel | Usually done automatically by package manager |
| Encrypted root filesystem | Encryption tools must be in initramfs |

---

### Step 4: Kernel Initialization + systemd

Once the kernel has the real root filesystem mounted:

```
1. Kernel initializes all its subsystems
   └── Memory management
   └── Process scheduler
   └── Device drivers

2. Kernel runs /sbin/init (which is systemd on modern systems)
   └── systemd reads its configuration
   └── systemd starts services in parallel
   └── Services: networking, SSH, Docker, logging, etc.

3. systemd reaches the "default target"
   └── multi-user.target (server — text login)
   └── graphical.target (desktop — GUI login)
```

```bash
# Check what init system you're using
ps -p 1 -o comm=
# Output: systemd

# Check the default target
systemctl get-default
# Output: multi-user.target

# See boot time breakdown
systemd-analyze
# Startup finished in 3.456s (kernel) + 12.789s (userspace) = 16.245s

# See which services took the longest to start
systemd-analyze blame | head -10
```

---

### Step 5: Login Prompt

Finally, systemd starts the login service:
- **Text login** → `getty` service provides a terminal login prompt
- **SSH login** → `sshd` service listens on port 22
- **GUI login** → Display manager (GDM, LightDM) shows graphical login

## 🔍 Kernel Troubleshooting Commands — Quick Reference

```bash
# ─── Kernel Info ───
uname -r                          # Kernel version
uname -a                          # Full system info
cat /proc/version                 # Detailed kernel version

# ─── Kernel Modules ───
lsmod                             # List loaded modules
modinfo <module>                  # Module details
sudo modprobe <module>            # Load a module
sudo modprobe -r <module>         # Unload a module
cat /etc/modules-load.d/*.conf    # Modules loaded at boot

# ─── Kernel Parameters ───
sysctl -a                         # Show all parameters
sysctl <param>                    # Show specific parameter
sudo sysctl -w <param>=<value>    # Set temporarily
cat /etc/sysctl.d/*.conf          # Persistent parameters
sudo sysctl --system              # Reload all sysctl configs

# ─── Boot & GRUB ───
cat /etc/default/grub             # GRUB settings
sudo update-grub                  # Regenerate GRUB config (Debian)
sudo grub2-mkconfig -o /boot/grub2/grub.cfg  # RHEL/CentOS

# ─── initramfs ───
ls /boot/initrd*                  # List initramfs files
sudo update-initramfs -u          # Rebuild (Debian/Ubuntu)
sudo dracut --force               # Rebuild (RHEL/CentOS)

# ─── Boot Analysis ───
systemd-analyze                   # Total boot time
systemd-analyze blame             # Per-service boot time
journalctl -b                     # Logs from current boot
journalctl -b -1                  # Logs from previous boot
dmesg                             # Kernel ring buffer (hardware messages)
dmesg | grep -i error             # Find kernel errors
```

## 🧪 Interview / Real-World Scenarios

### Q: A server isn't booting after a kernel upgrade. How do you fix it?

**Answer:**
1. Reboot and go to **GRUB menu** (hold Shift or press Esc)
2. Select the **previous working kernel** from the list
3. Boot into it
4. Investigate what went wrong with the new kernel
5. If needed, set the old kernel as default in `/etc/default/grub`
6. Run `sudo update-grub` to apply

### Q: Kubernetes pods can't communicate across nodes. What kernel parameters should you check?

**Answer:**
```bash
# Check these three critical parameters:
sysctl net.ipv4.ip_forward                    # Must be 1
sysctl net.bridge.bridge-nf-call-iptables     # Must be 1
sysctl net.bridge.bridge-nf-call-ip6tables    # Must be 1

# Also check if br_netfilter module is loaded
lsmod | grep br_netfilter
```

### Q: Docker says "overlay filesystem not supported." What do you do?

**Answer:**
```bash
# Load the overlay module
sudo modprobe overlay

# Make it persistent
echo "overlay" | sudo tee /etc/modules-load.d/docker.conf

# Verify
lsmod | grep overlay
```

## 📝 Key Takeaways

| Concept | Remember This |
|---------|---------------|
| **Kernel** | The core of Linux — manages hardware, processes, memory, networking |
| **Modules** | Loadable plugins for the kernel (drivers). Use `modprobe` to manage |
| **Parameters** | Runtime kernel settings. Use `sysctl` to view/change |
| **BIOS/UEFI** | Firmware that starts the boot process |
| **GRUB** | Bootloader that loads the kernel. Config at `/etc/default/grub` |
| **initramfs** | Temporary RAM filesystem with essential drivers to mount real root |
| **systemd** | The init system that starts all services after kernel boots |
| **Boot order** | BIOS/UEFI → GRUB → initramfs → Kernel → systemd → Login |
