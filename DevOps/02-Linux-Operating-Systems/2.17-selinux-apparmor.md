# 2.17 — SELinux & AppArmor

> **One Line Answer:** SELinux and AppArmor are **Mandatory Access Control (MAC)** security systems built into the Linux kernel that go **beyond** traditional file permissions — they restrict what processes can do, even if they're running as root.

---

## 🤔 Why Should a DevOps Engineer Care About SELinux & AppArmor?

Here's a scenario that will make you care:

> You deploy a web app. It works perfectly in development. You push it to production.
> **It breaks.** The logs say "Permission denied."
>
> You check file permissions — everything looks fine. `chmod 777`? Still broken.
>
> **What's happening?** SELinux (or AppArmor) is blocking your app from doing something it's not supposed to do.

This happens **all the time**. Here's why you need to understand these tools:

| Situation | Why It Matters |
|-----------|---------------|
| App can't write to a directory | SELinux/AppArmor might be blocking it |
| Container won't start | Security policy conflict |
| Nginx can't bind to non-standard port | SELinux blocks it by default |
| You `chmod 777` but it still doesn't work | MAC overrides DAC (traditional permissions) |
| Security audit / compliance | You need to know how to configure and audit these tools |
| Production debugging | "It works on my machine but not in production" |

---

## 🔐 Understanding Access Control — DAC vs MAC

Before diving into SELinux and AppArmor, let's understand the two types of access control:

### DAC — Discretionary Access Control (Traditional)

This is the regular Linux permission system you already know (`chmod`, `chown`):

```bash
ls -l myfile.txt
# -rw-r--r-- 1 ritesh devops 1024 Jan 15 10:00 myfile.txt
#  ↑           ↑       ↑
#  permissions  owner   group
```

**The problem with DAC:**
- The **owner** of a file decides who can access it (that's the "discretionary" part)
- If a process runs as **root**, it can do **anything** — read any file, kill any process, bind to any port
- If a hacker exploits a root-level service, they own **everything**

```
DAC says: "Root can do whatever it wants 🤷"

Hacker exploits nginx (running as root):
  ├── Read /etc/shadow? ✅ Sure!
  ├── Modify /etc/passwd? ✅ No problem!
  ├── Access the database? ✅ Go ahead!
  └── Install a backdoor? ✅ Why not!
```

### MAC — Mandatory Access Control (SELinux / AppArmor)

MAC adds a **second layer** of security on top of DAC:

```
MAC says: "Even root must follow the rules 🛡️"

Hacker exploits nginx (running as root):
  ├── Read /etc/shadow? ❌ DENIED (nginx policy doesn't allow it)
  ├── Modify /etc/passwd? ❌ DENIED
  ├── Access the database? ❌ DENIED (nginx can only access web files)
  └── Install a backdoor? ❌ DENIED
  
nginx can ONLY:
  ├── Read files in /var/www/ ✅
  ├── Listen on ports 80/443 ✅
  ├── Write to /var/log/nginx/ ✅
  └── That's it. Nothing else.
```

**This is the power of MAC** — every process is locked in a **cage** and can only do what its policy explicitly allows.

```
┌────────────────────────────────────────────────────┐
│              Security Decision Flow                │
│                                                    │
│  Process wants to access a resource                │
│         │                                          │
│         ▼                                          │
│  ┌──────────────┐                                  │
│  │  DAC Check   │  (Traditional permissions)       │
│  │  chmod/chown │                                  │
│  └──────┬───────┘                                  │
│         │ Allowed?                                 │
│         ▼                                          │
│  ┌──────────────┐                                  │
│  │  MAC Check   │  (SELinux or AppArmor)           │
│  │  Policy      │                                  │
│  └──────┬───────┘                                  │
│         │ Allowed?                                 │
│         ▼                                          │
│    Access Granted ✅                               │
│                                                    │
│  ⚠️ BOTH must say "yes" for access to happen       │
└────────────────────────────────────────────────────┘
```

## 🛡️ SELinux — Security-Enhanced Linux

### What Is SELinux?

SELinux was developed by the **NSA** (yes, that NSA) and is the default MAC system on:
- **RHEL** (Red Hat Enterprise Linux)
- **CentOS**
- **Fedora**
- **Rocky Linux**
- **AlmaLinux**
- **Amazon Linux**

### How SELinux Works — The Label System

SELinux works by putting **labels** (called "contexts") on everything:
- Every **file** has a label
- Every **process** has a label
- Every **port** has a label
- Every **user** has a label

Then **policies** define which labels are allowed to interact with which other labels.

```
┌──────────────────────────────────────────────────────┐
│                SELinux Label System                    │
│                                                       │
│  Process: httpd_t  ──────  Can it access  ──────▶     │
│  (Apache web server)        this file?                │
│                                                       │
│                             File: httpd_sys_content_t │
│                             (/var/www/html/index.html)│
│                                                       │
│  Policy says: httpd_t CAN read httpd_sys_content_t    │
│  Result: ✅ ALLOWED                                   │
│                                                       │
│  ─────────────────────────────────────────────────    │
│                                                       │
│  Process: httpd_t  ──────  Can it access  ──────▶     │
│  (Apache web server)        this file?                │
│                                                       │
│                             File: shadow_t            │
│                             (/etc/shadow)             │
│                                                       │
│  Policy says: httpd_t CANNOT read shadow_t            │
│  Result: ❌ DENIED                                    │
└──────────────────────────────────────────────────────┘
```

### SELinux Modes

SELinux has three modes:

| Mode | What Happens | When to Use |
|------|-------------|-------------|
| **Enforcing** | Blocks AND logs violations | Production (this is what you want!) |
| **Permissive** | Only logs violations, doesn't block | Debugging / Testing |
| **Disabled** | Completely off | Never in production! |

```bash
# Check current SELinux mode
getenforce
# Output: Enforcing

# Or get more detail
sestatus
# Output:
# SELinux status:                 enabled
# SELinuxfs mount:                /sys/fs/selinux
# SELinux root directory:         /etc/selinux
# Loaded policy name:             targeted
# Current mode:                   enforcing
# Mode from config file:          enforcing
```

#### Changing SELinux Mode

```bash
# ─── Temporarily change mode (lost after reboot) ───

# Switch to Permissive (for debugging)
sudo setenforce 0

# Switch back to Enforcing
sudo setenforce 1

# Verify
getenforce

# ─── Permanently change mode (survives reboot) ───

# Edit the config file
sudo vi /etc/selinux/config

# Change this line:
# SELINUX=enforcing
# to:
# SELINUX=permissive
```

> ⚠️ **IMPORTANT:** Changing from `disabled` to `enforcing` requires a **reboot** and relabeling of the entire filesystem (can take 10+ minutes). Don't do this on a production server without planning!

### SELinux Contexts (Labels) — Understanding Them

Every file and process has a security context with 4 parts:

```
user:role:type:level

Example: system_u:object_r:httpd_sys_content_t:s0
         ────────  ────────  ──────────────────  ──
            │         │            │               │
         SELinux    SELinux      Type            Security
          user       role      (MOST             Level
                              IMPORTANT!)       (MLS/MCS)
```

**The `type` is what you'll work with 99% of the time.** The rest you can mostly ignore.

#### Viewing SELinux Contexts

```bash
# See file contexts
ls -Z /var/www/html/
# -rw-r--r--. root root system_u:object_r:httpd_sys_content_t:s0 index.html
#                                          ^^^^^^^^^^^^^^^^^^^^^^
#                                          This is the SELinux type

# See process contexts
ps -eZ | grep httpd
# system_u:system_r:httpd_t:s0    1234 ?  00:00:01 httpd
#                   ^^^^^^^^
#                   Process type

# See port contexts
sudo semanage port -l | grep http
# http_port_t      tcp      80, 81, 443, 488, 8008, 8009, 8443, 9000
```

### Common SELinux Scenarios & Solutions

#### Scenario 1: Nginx Can't Serve Files from a Custom Directory

```bash
# You moved your website to /data/website/ instead of /var/www/html/
# Nginx returns 403 Forbidden. WHY?

# Check the SELinux context of the files
ls -Z /data/website/
# -rw-r--r--. root root unconfined_u:object_r:default_t:s0 index.html
#                                              ^^^^^^^^^
#                                          Wrong type! Should be httpd_sys_content_t

# FIX: Change the context
sudo semanage fcontext -a -t httpd_sys_content_t "/data/website(/.*)?"
sudo restorecon -Rv /data/website/

# Verify
ls -Z /data/website/
# -rw-r--r--. root root unconfined_u:object_r:httpd_sys_content_t:s0 index.html
#                                              ^^^^^^^^^^^^^^^^^^^^^^ ✅ Fixed!
```

**What those commands do:**
- `semanage fcontext -a` — Add a new file context rule (persistent)
- `-t httpd_sys_content_t` — Set the type to httpd web content
- `"/data/website(/.*)?"` — Apply to this directory and everything inside it (regex)
- `restorecon -Rv` — Apply the new labels to existing files (`-R` = recursive, `-v` = verbose)

#### Scenario 2: Nginx Can't Bind to Port 8888

```bash
# You configured nginx to listen on port 8888
# But it fails to start!

# Check what ports are allowed for http
sudo semanage port -l | grep http_port_t
# http_port_t   tcp   80, 81, 443, 488, 8008, 8009, 8443, 9000

# Port 8888 is NOT in the list. Add it:
sudo semanage port -a -t http_port_t -p tcp 8888

# Verify
sudo semanage port -l | grep 8888
# http_port_t   tcp   80, 81, 443, 488, 8008, 8009, 8443, 8888, 9000

# Now restart nginx — it works!
sudo systemctl restart nginx
```

#### Scenario 3: Quick Fix with SELinux Booleans

SELinux has **booleans** — simple on/off switches for common policies:

```bash
# List all booleans
sudo getsebool -a

# Common ones:
sudo getsebool httpd_can_network_connect
# httpd_can_network_connect --> off

# Nginx needs to connect to a backend API? Enable it:
sudo setsebool -P httpd_can_network_connect on
# -P means "persistent" (survives reboot)

# Other useful booleans:
sudo setsebool -P httpd_can_network_connect_db on   # Allow web server to connect to DB
sudo setsebool -P httpd_enable_homedirs on           # Allow serving from home directories
sudo setsebool -P container_manage_cgroup on         # Allow containers to manage cgroups
```

### Troubleshooting SELinux Denials

When SELinux blocks something, it logs it. Here's how to find and fix denials:

```bash
# ─── Method 1: Check the audit log ───
sudo ausearch -m AVC -ts recent
# or
sudo cat /var/log/audit/audit.log | grep denied

# Example output:
# type=AVC msg=audit(1642281600.123:456): avc: denied { read } for
#   pid=1234 comm="nginx" name="index.html" dev="sda1" ino=789
#   scontext=system_u:system_r:httpd_t:s0
#   tcontext=system_u:object_r:default_t:s0
#   tclass=file permissive=0

# Translation: nginx (httpd_t) tried to read a file (default_t) and was DENIED

# ─── Method 2: Use sealert (if available) ───
sudo sealert -a /var/log/audit/audit.log
# This gives you human-readable explanations AND suggested fixes!

# ─── Method 3: Use audit2why ───
sudo ausearch -m AVC -ts recent | audit2why
# This tells you WHY the denial happened
```

### The "I'll Just Disable SELinux" Trap

> 🚨 **DON'T DO THIS:** `setenforce 0` or `SELINUX=disabled`

Every DevOps beginner's instinct is to disable SELinux when something doesn't work. **Resist this urge!**

Instead:
1. Check the logs (`ausearch -m AVC`)
2. Understand what's being denied
3. Fix the context (`semanage fcontext` + `restorecon`)
4. Or flip a boolean (`setsebool -P`)
5. Or use `audit2allow` to create a custom policy

```bash
# Generate a custom policy module from denials
sudo ausearch -m AVC -ts recent | audit2allow -M my_custom_policy

# Install the policy
sudo semodule -i my_custom_policy.pp
```

## 🛡️ AppArmor — Application Armor

### What Is AppArmor?

AppArmor is the MAC system used by:
- **Ubuntu** (default since 7.10)
- **Debian** (default since Buster 10)
- **SUSE / openSUSE**

It's generally considered **simpler** than SELinux, which is both its strength and weakness.

### SELinux vs AppArmor — Key Differences

| Feature | SELinux | AppArmor |
|---------|---------|----------|
| **Approach** | Labels everything (files, processes, ports) | Uses file paths |
| **Default on** | RHEL, CentOS, Fedora | Ubuntu, Debian, SUSE |
| **Complexity** | More complex, more powerful | Simpler, easier to learn |
| **Policy style** | Type enforcement (label-based) | Path-based profiles |
| **Granularity** | Very fine-grained | Good but less granular |
| **Learning curve** | Steep | Moderate |
| **Performance** | Slightly more overhead | Slightly less overhead |
| **Works on renamed files** | Yes (labels stay with the file) | No (path changes = different rules) |

### How AppArmor Works — The Profile System

AppArmor uses **profiles** — text files that define what a program can and cannot do.

```
┌──────────────────────────────────────────────────┐
│              AppArmor Profile for nginx           │
│                                                   │
│  This program CAN:                                │
│  ├── Read files in /var/www/html/       ✅        │
│  ├── Read /etc/nginx/                    ✅       │
│  ├── Write to /var/log/nginx/            ✅       │
│  ├── Listen on ports 80, 443             ✅       │
│  └── Read shared libraries in /lib/      ✅       │
│                                                   │
│  This program CANNOT:                             │
│  ├── Read /etc/shadow                    ❌       │
│  ├── Write to /etc/                      ❌       │
│  ├── Execute /bin/bash                   ❌       │
│  ├── Access /home/                       ❌       │
│  └── Anything not explicitly listed      ❌       │
└──────────────────────────────────────────────────┘
```

### AppArmor Modes

Just like SELinux, AppArmor has modes (but for individual profiles):

| Mode | What Happens | Use Case |
|------|-------------|----------|
| **Enforce** | Blocks and logs violations | Production |
| **Complain** | Only logs, doesn't block | Debugging / building new profiles |
| **Disabled** | Profile is not loaded | Troubleshooting |

### Basic AppArmor Commands

```bash
# ─── Check AppArmor Status ───
sudo apparmor_status
# or
sudo aa-status

# Output:
# apparmor module is loaded.
# 34 profiles are loaded.
# 34 profiles are in enforce mode.
#    /usr/sbin/nginx
#    /usr/bin/man
#    ...
# 0 profiles are in complain mode.
# 12 processes have profiles defined.
# 12 processes are in enforce mode.
```

```bash
# ─── List All Profiles ───
sudo aa-status

# ─── Put a Profile in Complain Mode (for debugging) ───
sudo aa-complain /usr/sbin/nginx
# Now nginx violations are logged but NOT blocked

# ─── Put a Profile Back in Enforce Mode ───
sudo aa-enforce /usr/sbin/nginx
# Now nginx violations are blocked AND logged

# ─── Disable a Profile Temporarily ───
sudo aa-disable /usr/sbin/nginx
# Profile is unloaded completely

# ─── Reload All Profiles ───
sudo systemctl reload apparmor
```

### Where Are AppArmor Profiles Stored?

```bash
# Profile files
ls /etc/apparmor.d/
# Output:
# usr.sbin.nginx
# usr.sbin.mysqld
# usr.bin.docker
# ...

# The filename pattern is the path with dots instead of slashes:
# /usr/sbin/nginx → usr.sbin.nginx
```

### Understanding an AppArmor Profile

Let's look at a simplified example:

```bash
cat /etc/apparmor.d/usr.sbin.nginx
```

```bash
# AppArmor profile for nginx
#include <tunables/global>

/usr/sbin/nginx {
  # Include common abstractions
  #include <abstractions/base>
  #include <abstractions/nameservice>

  # Capabilities nginx needs
  capability net_bind_service,    # Bind to ports < 1024 (80, 443)
  capability setuid,              # Change user ID
  capability setgid,              # Change group ID
  capability dac_override,        # Override file permissions

  # What nginx can READ
  /etc/nginx/** r,                # Read nginx config files
  /var/www/html/** r,             # Read website files
  /usr/share/nginx/** r,          # Read default pages

  # What nginx can WRITE
  /var/log/nginx/** w,            # Write to log files
  /run/nginx.pid rw,              # Read/write PID file

  # What nginx can EXECUTE
  /usr/sbin/nginx mr,             # Execute itself (main + workers)

  # Network access
  network inet stream,            # TCP connections (IPv4)
  network inet6 stream,           # TCP connections (IPv6)
}
```

**Profile syntax explained:**

| Symbol | Meaning |
|--------|---------|
| `r` | Read access |
| `w` | Write access |
| `a` | Append access |
| `k` | Lock access |
| `m` | Memory map executable |
| `x` | Execute |
| `l` | Link |
| `**` | All files and subdirectories (recursive) |
| `*` | All files in directory (not recursive) |

### Creating a Custom AppArmor Profile

Let's create a profile for a custom application:

```bash
# Step 1: Generate a basic profile
sudo aa-genprof /usr/local/bin/myapp

# This starts an interactive wizard:
# 1. It runs your app in a special mode
# 2. It watches what files/resources the app accesses
# 3. It asks you to Allow or Deny each access
# 4. It generates a profile based on your answers

# Step 2: Or create one manually
sudo vi /etc/apparmor.d/usr.local.bin.myapp
```

```bash
# /etc/apparmor.d/usr.local.bin.myapp

#include <tunables/global>

/usr/local/bin/myapp {
  #include <abstractions/base>

  # Allow reading its own config
  /etc/myapp/** r,

  # Allow writing to its own log
  /var/log/myapp/** w,

  # Allow reading data files
  /data/myapp/** r,

  # Allow network access
  network inet stream,

  # Deny everything else (implicit)
}
```

```bash
# Step 3: Load the profile
sudo apparmor_parser -r /etc/apparmor.d/usr.local.bin.myapp

# Step 4: Verify it's loaded
sudo aa-status | grep myapp
```

### Troubleshooting AppArmor

```bash
# ─── Check for denials in system log ───
sudo dmesg | grep apparmor
# or
sudo journalctl -k | grep apparmor
# or
sudo cat /var/log/syslog | grep apparmor

# Example denial log:
# [  123.456] audit: type=1400 audit(1642281600.123:456):
#   apparmor="DENIED" operation="open" profile="/usr/sbin/nginx"
#   name="/data/website/index.html" pid=1234 comm="nginx"
#   requested_mask="r" denied_mask="r"

# Translation: nginx tried to READ /data/website/index.html and was DENIED

# ─── Fix: Add the path to nginx's profile ───
# Edit /etc/apparmor.d/usr.sbin.nginx and add:
#   /data/website/** r,

# Reload the profile
sudo apparmor_parser -r /etc/apparmor.d/usr.sbin.nginx
```

### AppArmor and Docker/Containers

Docker uses AppArmor by default on Ubuntu/Debian systems:

```bash
# Check Docker's default AppArmor profile
docker info | grep apparmor
# Security Options: apparmor

# Run a container with default AppArmor profile
docker run -it ubuntu bash
# The container is restricted by the docker-default profile

# Run a container with NO AppArmor profile (not recommended in production!)
docker run --security-opt apparmor=unconfined -it ubuntu bash

# Run with a custom profile
docker run --security-opt apparmor=my-custom-profile -it ubuntu bash
```

## 🔄 SELinux vs AppArmor — Quick Comparison Cheat Sheet

| Task | SELinux | AppArmor |
|------|---------|----------|
| Check status | `getenforce` / `sestatus` | `sudo aa-status` |
| Enable debug mode | `sudo setenforce 0` (Permissive) | `sudo aa-complain /path/to/program` |
| Re-enable protection | `sudo setenforce 1` (Enforcing) | `sudo aa-enforce /path/to/program` |
| View denials | `sudo ausearch -m AVC` | `sudo dmesg \| grep apparmor` |
| Fix file context | `restorecon -Rv /path` | Edit profile, add path rule |
| Allow port | `semanage port -a -t type -p tcp PORT` | Add network rule to profile |
| Config location | `/etc/selinux/config` | `/etc/apparmor.d/` |
| Reload | `sudo semodule -i policy.pp` | `sudo apparmor_parser -r profile` |

## 🧪 Interview / Real-World Scenarios

### Q: You deployed an app on RHEL and it can't write to /opt/myapp/data/. File permissions are correct (777). What's wrong?

**Answer:**
```bash
# 1. Check if SELinux is blocking it
getenforce
# Enforcing

# 2. Check recent denials
sudo ausearch -m AVC -ts recent | grep myapp
# Found a denial!

# 3. Check the file context
ls -Z /opt/myapp/data/
# default_t — wrong context!

# 4. Set the correct context
sudo semanage fcontext -a -t myapp_data_t "/opt/myapp/data(/.*)?"
# (or use a generic writable type if no specific type exists)
sudo semanage fcontext -a -t httpd_sys_rw_content_t "/opt/myapp/data(/.*)?"
sudo restorecon -Rv /opt/myapp/data/
```

### Q: How would you temporarily debug an AppArmor issue?

**Answer:**
```bash
# 1. Put the profile in complain mode
sudo aa-complain /usr/sbin/myapp

# 2. Reproduce the issue — it will work now

# 3. Check logs for what was being denied
sudo dmesg | grep apparmor | grep ALLOWED
# (In complain mode, blocked actions show as ALLOWED with a note)

# 4. Update the profile to allow the denied actions

# 5. Put it back in enforce mode
sudo aa-enforce /usr/sbin/myapp
```

### Q: Which is better — SELinux or AppArmor?

**Answer:**

Neither is universally "better" — it depends on your use case:

| Choose SELinux When | Choose AppArmor When |
|--------------------|---------------------|
| You need maximum security | You want simplicity |
| You're on RHEL/CentOS | You're on Ubuntu/Debian |
| You need fine-grained control | You need quick setup |
| Compliance requires it (government, military) | You're new to MAC systems |
| Files move between directories | File paths are stable |

## 📝 Key Takeaways

| Concept | Remember This |
|---------|---------------|
| **DAC** | Traditional permissions (chmod/chown). Owner decides access. Root bypasses all. |
| **MAC** | Mandatory Access Control. Even root is restricted by policy. |
| **SELinux** | Label-based MAC. Default on RHEL/CentOS. More powerful, more complex. |
| **AppArmor** | Path-based MAC. Default on Ubuntu/Debian. Simpler, easier to learn. |
| **Never disable** | Don't disable SELinux/AppArmor. Learn to work WITH them. |
| **Debug mode** | SELinux: `setenforce 0` (Permissive). AppArmor: `aa-complain`. |
| **Check denials** | SELinux: `ausearch -m AVC`. AppArmor: `dmesg \| grep apparmor`. |
| **Fix contexts** | SELinux: `semanage fcontext` + `restorecon`. AppArmor: edit profile. |
