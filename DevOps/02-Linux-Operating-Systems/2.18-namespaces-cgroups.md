> **One Line Answer:** **Namespaces** give processes their own isolated view of the system (separate network, PIDs, filesystems), while **Cgroups** limit how much resources (CPU, RAM, disk I/O) a process can use — and together, they are the **two pillars** that make Docker and Kubernetes containers possible.

## 🤔 Why Should a DevOps Engineer Care About Namespaces & Cgroups?

This is arguably the **most important topic** in this chapter for DevOps engineers. Here's why:

**Containers are NOT virtual machines.** They're just regular Linux processes with namespaces and cgroups applied. That's it. There's no magic.

```
Virtual Machine:                        Container:
┌──────────────────┐                   ┌──────────────────┐
│   Your App       │                   │   Your App       │
├──────────────────┤                   ├──────────────────┤
│   Guest OS       │                   │   Namespaces     │ ← Isolation
│   (Full Linux)   │                   │   + Cgroups      │ ← Resource limits
├──────────────────┤                   ├──────────────────┤
│   Hypervisor     │                   │   Host OS Kernel │
│   (VMware/KVM)   │                   │   (Shared!)      │
├──────────────────┤                   ├──────────────────┤
│   Host OS        │                   │   Hardware       │
├──────────────────┤                   └──────────────────┘
│   Hardware       │
└──────────────────┘

Heavy (GBs), slow to start              Light (MBs), starts in seconds
```

| Situation | Why This Knowledge Helps |
|-----------|------------------------|
| Understanding Docker internals | Docker = namespaces + cgroups + union filesystem |
| Container security | Know what's isolated and what's NOT |
| Kubernetes resource limits | `resources.limits.cpu` and `resources.limits.memory` use cgroups |
| Debugging container issues | "Why can my container see the host's processes?" |
| Container escapes / security | Understanding namespace boundaries |
| Performance tuning containers | Cgroup settings directly affect performance |

## 📦 Part 1: Linux Namespaces — Isolation

### What Are Namespaces?

A namespace **wraps a global system resource** in an abstraction that makes it appear to processes within the namespace that they have their **own isolated instance** of that resource.

In simple terms: namespaces let you create a **private world** for a process where it thinks it's the only thing running on the system.

### Types of Namespaces

Linux has **8 types** of namespaces:

| Namespace | Flag | What It Isolates | Example |
|-----------|------|-----------------|---------|
| **PID** | `CLONE_NEWPID` | Process IDs | Container sees its own PID 1 |
| **Network** | `CLONE_NEWNET` | Network interfaces, IPs, routes, ports | Container has its own IP address |
| **Mount** | `CLONE_NEWNS` | Filesystem mount points | Container sees its own `/` |
| **UTS** | `CLONE_NEWUTS` | Hostname and domain name | Container has its own hostname |
| **IPC** | `CLONE_NEWIPC` | Inter-process communication (shared memory, semaphores) | Processes can't share memory across containers |
| **User** | `CLONE_NEWUSER` | User and group IDs | Root inside container ≠ root on host |
| **Cgroup** | `CLONE_NEWCGROUP` | Cgroup root directory | Container sees its own cgroup hierarchy |
| **Time** | `CLONE_NEWTIME` | System clocks (since kernel 5.6) | Container can have different clock offsets |

Let's understand each one with examples:

---

### 1. PID Namespace — "You're the Only One Here"

The PID namespace makes a process think it has its own set of process IDs, starting from PID 1.

```
HOST VIEW:                          CONTAINER VIEW:
┌──────────────────┐               ┌──────────────────┐
│ PID 1: systemd   │               │ PID 1: nginx     │  ← Thinks it's PID 1!
│ PID 2: kthreadd  │               │ PID 2: worker    │
│ PID 345: sshd    │               │ PID 3: worker    │
│ PID 789: dockerd │               │                  │
│ PID 1001: nginx  │ ◄─── same ──▶│ PID 1: nginx     │
│ PID 1002: worker │ ◄─── same ──▶│ PID 2: worker    │
│ PID 1003: worker │ ◄─── same ──▶│ PID 3: worker    │
└──────────────────┘               └──────────────────┘
Host sees ALL processes             Container only sees ITS processes
```

**Let's try it!**

```bash
# Create a new PID namespace and run bash inside it
sudo unshare --pid --fork --mount-proc bash

# Now inside the new namespace:
ps aux
# You'll only see bash and ps — nothing else!
# PID 1 is bash (normally PID 1 is systemd)

# Exit the namespace
exit
```

**What happened here:**
- `unshare` — creates new namespaces
- `--pid` — create a new PID namespace
- `--fork` — fork a new process (required for PID namespace)
- `--mount-proc` — mount a new /proc so `ps` shows the right PIDs

### 2. Network Namespace — "You Have Your Own Network"

Each container gets its own network stack: interfaces, IP addresses, routing table, ports.

```
HOST:                              CONTAINER:
┌──────────────────┐              ┌──────────────────┐
│ eth0: 10.0.0.5   │              │ eth0: 172.17.0.2 │
│ docker0: 172.17. │              │ lo: 127.0.0.1    │
│         0.1      │              │                  │
│ lo: 127.0.0.1    │◄── veth ───▶│ Routes:          │
│                  │   pair       │ default via      │
│ Port 80: nginx   │              │   172.17.0.1     │
│ Port 22: sshd    │              │                  │
└──────────────────┘              │ Port 80: app     │
                                  │ (same port, no   │
                                  │  conflict!)       │
                                  └──────────────────┘
```

**Let's create one:**

```bash
# Create a new network namespace called "mynet"
sudo ip netns add mynet

# List network namespaces
ip netns list
# mynet

# Run a command inside the namespace
sudo ip netns exec mynet ip addr
# Only shows 'lo' (loopback) — no other interfaces!

# Let's bring up loopback
sudo ip netns exec mynet ip link set lo up

# Create a virtual ethernet pair (like a virtual cable)
sudo ip link add veth0 type veth peer name veth1

# Move one end into the namespace
sudo ip link set veth1 netns mynet

# Assign IP addresses
sudo ip addr add 192.168.100.1/24 dev veth0
sudo ip link set veth0 up

sudo ip netns exec mynet ip addr add 192.168.100.2/24 dev veth1
sudo ip netns exec mynet ip link set veth1 up

# Now you can ping between host and namespace!
ping 192.168.100.2
sudo ip netns exec mynet ping 192.168.100.1

# Clean up
sudo ip netns delete mynet
```

**This is EXACTLY how Docker networking works!** Docker creates a veth pair for each container, connecting it to the `docker0` bridge.

### 3. Mount Namespace — "You Have Your Own Filesystem"

Each container sees its own filesystem tree. Changes inside don't affect the host.

```bash
# Create a new mount namespace
sudo unshare --mount bash

# Mount a tmpfs at /mnt inside the namespace
mount -t tmpfs tmpfs /mnt
echo "hello from namespace" > /mnt/test.txt

# This file exists inside the namespace
cat /mnt/test.txt
# hello from namespace

# Exit the namespace
exit

# Check on the host — the file doesn't exist!
cat /mnt/test.txt
# cat: /mnt/test.txt: No such file or directory
```

### 4. UTS Namespace — "You Have Your Own Hostname"

```bash
# Create a new UTS namespace
sudo unshare --uts bash

# Set a custom hostname
hostname my-container

# Verify
hostname
# my-container

# Exit
exit

# Host hostname is unchanged
hostname
# your-original-hostname
```

### 5. User Namespace — "Root Inside, Nobody Outside"

This is a **security game-changer**. A process can be root (UID 0) inside the container but mapped to an unprivileged user (like UID 65534/nobody) on the host.

```
INSIDE CONTAINER:                    ON THE HOST:
┌──────────────────┐                ┌──────────────────┐
│ UID 0 (root)     │  ──maps to──▶ │ UID 100000       │
│ UID 1 (bin)      │  ──maps to──▶ │ UID 100001       │
│ UID 1000 (app)   │  ──maps to──▶ │ UID 101000       │
└──────────────────┘                └──────────────────┘

"I'm root! I can do anything!"      "Lol no, you're user 100000"
```

```bash
# Create a user namespace (no sudo needed!)
unshare --user --map-root-user bash

# Check who you are inside the namespace
id
# uid=0(root) gid=0(root) groups=0(root)
# You're "root" inside!

# But on the host, you're still your regular user
# Open another terminal and check:
ps -eo pid,uid,cmd | grep unshare
# Shows your regular UID, not 0
```

### Viewing Namespaces of a Process

```bash
# Every process's namespaces are visible in /proc
ls -la /proc/self/ns/
# lrwxrwxrwx 1 root root 0 Jan 15 10:00 cgroup -> cgroup:[4026531835]
# lrwxrwxrwx 1 root root 0 Jan 15 10:00 ipc -> ipc:[4026531839]
# lrwxrwxrwx 1 root root 0 Jan 15 10:00 mnt -> mnt:[4026531840]
# lrwxrwxrwx 1 root root 0 Jan 15 10:00 net -> net:[4026531969]
# lrwxrwxrwx 1 root root 0 Jan 15 10:00 pid -> pid:[4026531836]
# lrwxrwxrwx 1 root root 0 Jan 15 10:00 user -> user:[4026531837]
# lrwxrwxrwx 1 root root 0 Jan 15 10:00 uts -> uts:[4026531838]

# The numbers in brackets are namespace IDs
# Processes in the same namespace share the same ID

# Compare two processes:
sudo ls -la /proc/1/ns/    # systemd's namespaces
sudo ls -la /proc/$$/ns/   # Your shell's namespaces
# If the IDs match, they're in the same namespace
```

```bash
# For Docker containers, you can check namespaces too:
# Get the container's PID on the host
docker inspect --format '{{.State.Pid}}' <container_id>
# Output: 12345

# View its namespaces
sudo ls -la /proc/12345/ns/

# Enter a container's namespaces (this is what 'docker exec' does!)
sudo nsenter --target 12345 --mount --uts --ipc --net --pid bash
```

## 📊 Part 2: Cgroups — Resource Limits

### What Are Cgroups?

**Cgroups** (Control Groups) let you limit, account for, and isolate the resource usage of a group of processes.

Think of it like this:

```
Without Cgroups:                    With Cgroups:
┌──────────────────┐               ┌──────────────────┐
│  Process A: "I   │               │  Process A:      │
│  want ALL the    │               │  ├── Max CPU: 50% │
│  CPU and RAM!"   │               │  ├── Max RAM: 256M│
│                  │               │  └── Max I/O: 100M│
│  Process B: "Me  │               │                   │
│  too! Fight me!" │               │  Process B:       │
│                  │               │  ├── Max CPU: 25% │
│  Process C: "I'm │               │  ├── Max RAM: 128M│
│  starving... 😢" │               │  └── Max I/O: 50M │
│                  │               │                   │
│  CHAOS! One bad  │               │  Fair & controlled│
│  process kills   │               │  Each process gets│
│  everything      │               │  its fair share   │
└──────────────────┘               └──────────────────┘
```

### Cgroups v1 vs Cgroups v2

There are two versions of cgroups. Most modern systems use v2, but you'll still encounter v1.

| Feature | Cgroups v1 | Cgroups v2 |
|---------|-----------|-----------|
| Structure | Multiple hierarchies (one per controller) | Single unified hierarchy |
| Filesystem | `/sys/fs/cgroup/<controller>/` | `/sys/fs/cgroup/` |
| Controllers | Each mounted separately | All in one tree |
| Complexity | More complex | Simpler |
| Default on | Older systems (Ubuntu < 21.10, RHEL 7) | Modern systems (Ubuntu 22.04+, RHEL 9) |

```bash
# Check which version you're using
stat -fc %T /sys/fs/cgroup/
# cgroup2fs → You're on cgroups v2
# tmpfs → You're on cgroups v1 (or mixed)

# Or check mounted cgroup filesystems
mount | grep cgroup
```

### What Resources Can Cgroups Control?

| Controller | What It Controls | Example Use |
|-----------|-----------------|-------------|
| **cpu** | CPU time allocation | Limit container to 0.5 CPU cores |
| **cpuset** | Which CPU cores to use | Pin container to cores 0-3 |
| **memory** | RAM usage | Limit container to 512MB RAM |
| **blkio / io** | Disk I/O bandwidth | Limit container to 100MB/s disk read |
| **pids** | Number of processes | Max 100 processes per container |
| **devices** | Device access | Block container from accessing /dev/sda |
| **freezer** | Pause/resume processes | Pause a container without killing it |
| **hugetlb** | Huge page memory | Limit huge page allocations |

### Working with Cgroups v2 — Hands-On

Let's create a cgroup and limit a process's resources:

#### Step 1: Explore the Cgroup Filesystem

```bash
# The cgroup filesystem is mounted here
ls /sys/fs/cgroup/

# See available controllers
cat /sys/fs/cgroup/cgroup.controllers
# cpu io memory pids

# See what controllers are active for child groups
cat /sys/fs/cgroup/cgroup.subtree_control
# cpu io memory pids
```

#### Step 2: Create a New Cgroup

```bash
# Create a new cgroup called "my_limited_group"
sudo mkdir /sys/fs/cgroup/my_limited_group

# See the files inside (automatically created by the kernel)
ls /sys/fs/cgroup/my_limited_group/
# cgroup.controllers
# cgroup.events
# cgroup.procs         ← PIDs of processes in this group
# cpu.max              ← CPU limit
# cpu.stat             ← CPU usage statistics
# memory.current       ← Current memory usage
# memory.max           ← Memory limit
# memory.high          ← Memory high watermark
# pids.current         ← Current number of processes
# pids.max             ← Max processes limit
# ... and more
```

#### Step 3: Set Resource Limits

```bash
# ─── Limit CPU ───
# Format: $MAX $PERIOD (in microseconds)
# "Allow 50000 microseconds of CPU time per 100000 microsecond period" = 50% of one core
echo "50000 100000" | sudo tee /sys/fs/cgroup/my_limited_group/cpu.max

# 100% of one core
echo "100000 100000" | sudo tee /sys/fs/cgroup/my_limited_group/cpu.max

# 200% (= 2 cores)
echo "200000 100000" | sudo tee /sys/fs/cgroup/my_limited_group/cpu.max

# No limit
echo "max 100000" | sudo tee /sys/fs/cgroup/my_limited_group/cpu.max

# ─── Limit Memory ───
# Set max memory to 256MB
echo $((256 * 1024 * 1024)) | sudo tee /sys/fs/cgroup/my_limited_group/memory.max
# Or simply:
echo "268435456" | sudo tee /sys/fs/cgroup/my_limited_group/memory.max

# Set a "high" watermark (kernel will try to reclaim memory above this)
echo $((200 * 1024 * 1024)) | sudo tee /sys/fs/cgroup/my_limited_group/memory.high

# ─── Limit Number of Processes ───
echo "50" | sudo tee /sys/fs/cgroup/my_limited_group/pids.max
```

#### Step 4: Add a Process to the Cgroup

```bash
# Add a process by writing its PID to cgroup.procs
echo $$ | sudo tee /sys/fs/cgroup/my_limited_group/cgroup.procs
# $$ is the PID of the current shell

# Or start a new process directly in the cgroup
sudo cgexec -g cpu,memory:my_limited_group stress --cpu 4 --timeout 30
# (cgexec requires the cgroup-tools package)
```

#### Step 5: Monitor the Cgroup

```bash
# See current CPU usage
cat /sys/fs/cgroup/my_limited_group/cpu.stat
# usage_usec 1234567
# user_usec 1000000
# system_usec 234567

# See current memory usage
cat /sys/fs/cgroup/my_limited_group/memory.current
# 15728640  (= ~15MB)

# See processes in this cgroup
cat /sys/fs/cgroup/my_limited_group/cgroup.procs
# 1234
# 1235
```

#### Step 6: Clean Up

```bash
# First, move all processes out of the cgroup
# (Move them to the root cgroup)
for pid in $(cat /sys/fs/cgroup/my_limited_group/cgroup.procs); do
    echo $pid | sudo tee /sys/fs/cgroup/cgroup.procs
done

# Then remove the cgroup directory
sudo rmdir /sys/fs/cgroup/my_limited_group
```

### Real-World Demo: Limiting a CPU-Hungry Process

Let's see cgroups in action by limiting a CPU-intensive process:

```bash
# Step 1: Create a cgroup with 25% CPU limit
sudo mkdir /sys/fs/cgroup/cpu_demo
echo "25000 100000" | sudo tee /sys/fs/cgroup/cpu_demo/cpu.max

# Step 2: Run a CPU-intensive command WITHOUT the limit
# (In terminal 1)
stress --cpu 1 --timeout 60 &
top -p $!
# CPU usage: ~100% 🔥

kill %1   # Stop it

# Step 3: Run the SAME command WITH the limit
# (Write the PID to the cgroup)
stress --cpu 1 --timeout 60 &
echo $! | sudo tee /sys/fs/cgroup/cpu_demo/cgroup.procs
top -p $!
# CPU usage: ~25% ✅  (cgroup is limiting it!)

# Step 4: Clean up
kill %1
sudo rmdir /sys/fs/cgroup/cpu_demo
```

## 🐳 Part 3: How Docker Uses Namespaces & Cgroups

Now let's connect everything to Docker. When you run `docker run`, here's what happens:

```bash
docker run -d \
  --name myapp \
  --cpus="0.5" \
  --memory="256m" \
  --pids-limit 100 \
  -p 8080:80 \
  nginx
```

```
What Docker does behind the scenes:
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  1. Creates NAMESPACES for isolation:                        │
│     ├── PID namespace    → nginx gets its own PID 1          │
│     ├── Network namespace → gets its own IP (172.17.0.x)     │
│     ├── Mount namespace  → sees its own filesystem           │
│     ├── UTS namespace    → has hostname = container ID       │
│     ├── IPC namespace    → isolated shared memory            │
│     └── User namespace   → (optional, rootless mode)         │
│                                                              │
│  2. Creates CGROUP with resource limits:                     │
│     ├── CPU: 50% of one core     (--cpus="0.5")             │
│     ├── Memory: 256MB max        (--memory="256m")           │
│     └── PIDs: max 100            (--pids-limit 100)          │
│                                                              │
│  3. Sets up NETWORKING:                                      │
│     ├── Creates veth pair                                    │
│     ├── Connects one end to docker0 bridge                   │
│     ├── Moves other end into container's network namespace   │
│     └── Sets up port mapping (8080→80) via iptables          │
│                                                              │
│  4. Prepares FILESYSTEM:                                     │
│     ├── Uses overlay filesystem (layers from image)          │
│     ├── Mounts volumes                                       │
│     └── Sets up in mount namespace                           │
│                                                              │
│  5. RUNS the process (nginx) inside all these namespaces     │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### Inspecting Docker's Cgroups

```bash
# Run a container
docker run -d --name test --cpus="0.5" --memory="256m" nginx

# Find its cgroup
# Docker creates cgroups under /sys/fs/cgroup/system.slice/ (systemd) or /sys/fs/cgroup/docker/

# Method 1: Docker inspect
docker inspect test --format '{{.HostConfig.CgroupParent}}'

# Method 2: Find the container's PID
PID=$(docker inspect --format '{{.State.Pid}}' test)

# Check its cgroup
cat /proc/$PID/cgroup
# 0::/system.slice/docker-<container_id>.scope

# See the CPU limit
cat /sys/fs/cgroup/system.slice/docker-*/cpu.max
# 50000 100000  (= 50% of one CPU)

# See the memory limit
cat /sys/fs/cgroup/system.slice/docker-*/memory.max
# 268435456  (= 256MB)
```

### Inspecting Docker's Namespaces

```bash
# Get container PID
PID=$(docker inspect --format '{{.State.Pid}}' test)

# See all namespaces
sudo ls -la /proc/$PID/ns/
# Each one points to a different namespace than the host's PID 1

# Compare with host
sudo ls -la /proc/1/ns/
# Different namespace IDs = different namespaces = isolation!

# Enter the container's namespaces (same as docker exec)
sudo nsenter --target $PID --mount --uts --ipc --net --pid bash
# You're now "inside" the container!
```

## ☸️ Part 4: Kubernetes and Cgroups

When you set resource limits in Kubernetes, it translates directly to cgroup settings:

```yaml
# kubernetes-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: myapp
spec:
  containers:
  - name: app
    image: nginx
    resources:
      requests:           # Minimum guaranteed resources
        cpu: "250m"       # 250 millicores = 25% of one core
        memory: "128Mi"   # 128 MiB
      limits:             # Maximum allowed resources
        cpu: "500m"       # 500 millicores = 50% of one core
        memory: "256Mi"   # 256 MiB
```

**How Kubernetes translates this to cgroups:**

| Kubernetes Setting | Cgroup Setting | Value |
|-------------------|---------------|-------|
| `limits.cpu: "500m"` | `cpu.max` | `50000 100000` |
| `limits.memory: "256Mi"` | `memory.max` | `268435456` |
| `requests.cpu: "250m"` | `cpu.weight` | Proportional weight |
| `requests.memory: "128Mi"` | Used for scheduling (not cgroup) | — |

**What happens when limits are exceeded?**

| Resource | What Happens |
|----------|-------------|
| CPU limit exceeded | Process is **throttled** (slowed down, not killed) |
| Memory limit exceeded | Process is **OOM killed** (killed by Out Of Memory killer) |

```bash
# You've probably seen this dreaded message:
# OOMKilled: true

# This means the container tried to use more memory than its cgroup limit
# The kernel's OOM killer stepped in and killed the process

# Check in Kubernetes:
kubectl describe pod myapp | grep -A 5 "State"
# State: Terminated
#   Reason: OOMKilled
#   Exit Code: 137
```

## 🔬 Building a Container from Scratch (No Docker!)

Let's build a simple container using just namespaces and cgroups to truly understand what's happening:

```bash
#!/bin/bash
# simple-container.sh — Build a container from scratch!

set -e

ROOTFS="/tmp/my-container"
CGROUP_PATH="/sys/fs/cgroup/my_container"

echo "=== Step 1: Create a minimal root filesystem ==="
mkdir -p $ROOTFS
# Using debootstrap to create a minimal Debian filesystem
# (Install with: sudo apt install debootstrap)
sudo debootstrap --variant=minbase stable $ROOTFS http://deb.debian.org/debian/

echo "=== Step 2: Create a cgroup with resource limits ==="
sudo mkdir -p $CGROUP_PATH
echo "50000 100000" | sudo tee $CGROUP_PATH/cpu.max         # 50% CPU
echo "134217728" | sudo tee $CGROUP_PATH/memory.max          # 128MB RAM
echo "50" | sudo tee $CGROUP_PATH/pids.max                   # Max 50 processes

echo "=== Step 3: Run a process with namespaces ==="
sudo unshare \
  --pid \                    # New PID namespace
  --net \                    # New network namespace
  --mount \                  # New mount namespace
  --uts \                    # New UTS namespace (hostname)
  --ipc \                    # New IPC namespace
  --fork \                   # Fork (required for PID namespace)
  --mount-proc \             # Mount new /proc
  chroot $ROOTFS /bin/bash   # Change root to our filesystem

# Inside this "container", you'll see:
# - PID 1 is bash
# - Hostname is isolated
# - Network is isolated
# - Filesystem is isolated
# - Resource usage is limited by cgroups
```

> 🎉 **Congratulations!** You just built a container. Docker is essentially a polished, automated version of exactly this.

## 🧪 Interview / Real-World Scenarios

### Q: What's the difference between a container and a VM?

**Answer:**

```
VM = Hardware virtualization (full OS, separate kernel)
Container = OS-level virtualization (shared kernel, namespaces + cgroups)

VM:
  App → Guest OS → Hypervisor → Host OS → Hardware
  Overhead: HIGH (GBs of RAM, minutes to start)

Container:
  App → Namespaces + Cgroups → Host OS Kernel → Hardware
  Overhead: LOW (MBs of RAM, seconds to start)
```

### Q: A Kubernetes pod keeps getting OOMKilled. How do you diagnose?

**Answer:**
```bash
# 1. Check the pod events
kubectl describe pod myapp | grep -A 10 "Events"

# 2. Check the actual memory usage vs limit
kubectl top pod myapp
# NAME    CPU(cores)   MEMORY(bytes)
# myapp   100m         245Mi          ← Close to 256Mi limit!

# 3. Check cgroup memory stats on the node
PID=$(docker inspect --format '{{.State.Pid}}' <container_id>)
cat /proc/$PID/cgroup
# Find the cgroup path
cat /sys/fs/cgroup/<path>/memory.current    # Current usage
cat /sys/fs/cgroup/<path>/memory.max        # The limit
cat /sys/fs/cgroup/<path>/memory.events     # OOM events count

# 4. Solution: Increase the memory limit or optimize the app
```

### Q: Can a process inside a container see host processes?

**Answer:**

By default, **no** — because of the PID namespace. But there are exceptions:
- If you run with `--pid=host` (Docker) or `hostPID: true` (Kubernetes), the container shares the host's PID namespace
- This is sometimes needed for monitoring tools but is a **security risk**

```bash
# Normal container — isolated PIDs
docker run --rm -it alpine ps aux
# Only sees its own processes

# With host PID namespace — sees EVERYTHING
docker run --rm -it --pid=host alpine ps aux
# Sees all host processes!
```

### Q: What happens if you set CPU limit but no CPU request in Kubernetes?

**Answer:**
- **Request** = how much CPU is guaranteed (used for scheduling)
- **Limit** = maximum CPU the container can use (enforced by cgroup `cpu.max`)
- If you set limit but no request, Kubernetes sets request = limit
- If you set neither, the container can use unlimited CPU (dangerous!)

## 📝 Key Takeaways

| Concept | Remember This |
|---------|---------------|
| **Namespaces** | Provide **isolation** — what a process can see |
| **Cgroups** | Provide **resource limits** — what a process can use |
| **Container** | = Namespaces + Cgroups + Union Filesystem. That's it! |
| **PID Namespace** | Each container has its own PID 1 |
| **Network Namespace** | Each container has its own IP, ports, routes |
| **Mount Namespace** | Each container has its own filesystem view |
| **User Namespace** | Root in container ≠ root on host |
| **Cgroups v2** | Modern, unified hierarchy. Check with `stat -fc %T /sys/fs/cgroup/` |
| **CPU cgroup** | Throttles (slows down) when exceeded |
| **Memory cgroup** | OOM kills when exceeded |
| **Docker** | Automates namespace + cgroup creation for you |
| **Kubernetes resources** | `requests` = scheduling, `limits` = cgroups enforcement |
