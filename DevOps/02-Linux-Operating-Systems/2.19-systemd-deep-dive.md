# 2.19 — Systemd Deep Dive — Units, Targets, Sockets, Timers

> **One Line Answer:** Systemd is the **init system and service manager** that controls everything in modern Linux — from booting your system to starting services, scheduling tasks, and managing sockets — and understanding it deeply is a **superpower** for any DevOps engineer.

---

## 🤔 Why Should a DevOps Engineer Care About Systemd?

Here's the truth — if you're managing **any** modern Linux server, you're already using systemd. You might just not realize how powerful it is.

| What You Need to Do | Systemd Handles It |
|---|---|
| Start/Stop services (nginx, docker, postgres) | ✅ `systemctl start/stop` |
| Make services start on boot | ✅ `systemctl enable` |
| Schedule tasks (replace cron) | ✅ Systemd Timers |
| Manage network sockets | ✅ Socket Units |
| Control the boot process | ✅ Targets |
| View logs | ✅ `journalctl` |
| Run containers | ✅ Some container runtimes use systemd |
| Manage temporary files | ✅ `systemd-tmpfiles` |
| Set hostnames & time | ✅ `hostnamectl`, `timedatectl` |

Almost every major Linux distro uses systemd today:

- **Ubuntu** (since 15.04)
- **CentOS / RHEL** (since 7)
- **Debian** (since 8)
- **Fedora** (since 15)
- **Arch Linux**
- **SUSE**

The only major exceptions are Alpine Linux (uses OpenRC) and some minimal/embedded distros.

## 📖 What is Systemd? — The Big Picture

### The Simple Explanation

When you turn on your Linux computer, **something** needs to:

1. Start the kernel
2. Start all the services (networking, logging, SSH, etc.)
3. Get you to a login screen or command prompt

That "something" is called the **init system** — and **systemd** is the most popular init system in Linux today.

But systemd is **way more** than just an init system. It's like a Swiss Army knife for system management:

```
┌───────────────────────────────────────────────────┐
│                    SYSTEMD                         │
│                                                   │
│  ┌──────────┐  ┌──────────┐  ┌──────────────┐    │
│  │  Init    │  │ Service  │  │   Logging    │    │
│  │ System   │  │ Manager  │  │ (journald)   │    │
│  └──────────┘  └──────────┘  └──────────────┘    │
│                                                   │
│  ┌──────────┐  ┌──────────┐  ┌──────────────┐    │
│  │  Timer   │  │  Socket  │  │   Device     │    │
│  │ Manager  │  │ Manager  │  │   Manager    │    │
│  └──────────┘  └──────────┘  └──────────────┘    │
│                                                   │
│  ┌──────────┐  ┌──────────┐  ┌──────────────┐    │
│  │  Mount   │  │  Login   │  │  Hostname    │    │
│  │ Manager  │  │ Manager  │  │  & Time Mgr  │    │
│  └──────────┘  └──────────┘  └──────────────┘    │
└───────────────────────────────────────────────────┘
```

### Before Systemd — A Quick History

Before systemd, Linux used other init systems:

```
Timeline of Linux Init Systems:
──────────────────────────────────────────
1983  │  SysVinit (System V Init)
      │  └── Used shell scripts in /etc/init.d/
      │  └── Services started sequentially (SLOW boot)
      │  └── Runlevels (0-6) to define system states
      │
2006  │  Upstart (by Ubuntu/Canonical)
      │  └── Event-based service starting
      │  └── Faster than SysVinit
      │  └── Used in Ubuntu for several years
      │
2010  │  Systemd (by Lennart Poettering & Kay Sievers)
      │  └── Parallel service startup (FAST boot)
      │  └── Dependency-based service management
      │  └── Unified configuration format
      │  └── Became the standard for most distros
──────────────────────────────────────────
```

**Why did systemd win?**

| Feature | SysVinit | Upstart | Systemd |
|---|---|---|---|
| Parallel startup | ❌ No | ⚠️ Partial | ✅ Yes |
| Dependency management | ❌ Manual | ⚠️ Basic | ✅ Full |
| Socket activation | ❌ No | ❌ No | ✅ Yes |
| Unified logging | ❌ No | ❌ No | ✅ Yes (journald) |
| Cgroup integration | ❌ No | ❌ No | ✅ Yes |
| Uniform config format | ❌ Shell scripts | ⚠️ Custom | ✅ INI-style unit files |

## 🧱 Systemd Unit Files — The Building Blocks

### What is a Unit?

A **unit** is the basic object that systemd manages. Think of it as a "thing" that systemd knows how to control.

Every unit has a **unit file** — a simple configuration file that tells systemd:
- What this thing is
- How to start/stop it
- What it depends on
- When it should run

### Types of Units

Systemd has **12 types** of units. Here are the most important ones:

| Unit Type | File Extension | What It Does | Example |
|---|---|---|---|
| **Service** | `.service` | Manages a process/daemon | `nginx.service` |
| **Socket** | `.socket` | Manages a network/IPC socket | `sshd.socket` |
| **Timer** | `.timer` | Schedules tasks (like cron) | `backup.timer` |
| **Target** | `.target` | Groups units together | `multi-user.target` |
| **Mount** | `.mount` | Manages mount points | `home.mount` |
| **Automount** | `.automount` | Auto-mounts on access | `nas.automount` |
| **Device** | `.device` | Manages devices | `dev-sda.device` |
| **Swap** | `.swap` | Manages swap space | `swapfile.swap` |
| **Path** | `.path` | Watches files/directories | `myapp.path` |
| **Slice** | `.slice` | Manages cgroups resource groups | `user.slice` |
| **Scope** | `.scope` | Manages externally created processes | `session-1.scope` |
| **Snapshot** | `.snapshot` | Saves systemd state | (deprecated) |

### Where Are Unit Files Stored?

This is **super important** to understand:

```
Priority (highest to lowest):
───────────────────────────────────────────────────

/etc/systemd/system/          ← HIGHEST PRIORITY
  │                             Admin/user customizations
  │                             YOU edit files here
  │
/run/systemd/system/          ← RUNTIME (temporary)
  │                             Created at runtime
  │                             Lost on reboot
  │
/usr/lib/systemd/system/      ← LOWEST PRIORITY
  │                             Package defaults
  │                             Installed by apt/yum
  │                             NEVER edit files here directly
───────────────────────────────────────────────────
```

**Golden Rule:** Never edit files in `/usr/lib/systemd/system/`. Always copy them to `/etc/systemd/system/` first, or use `systemctl edit` to create overrides.

## ⚙️ Service Units — Deep Dive

Service units are the most common type. Let's break them down completely.

### Anatomy of a Service Unit File

```ini
# /etc/systemd/system/myapp.service

[Unit]
Description=My Awesome Application
Documentation=https://myapp.example.com/docs
After=network.target postgresql.service
Wants=postgresql.service
Requires=network-online.target

[Service]
Type=simple
User=myapp
Group=myapp
WorkingDirectory=/opt/myapp
Environment=NODE_ENV=production
EnvironmentFile=/etc/myapp/env
ExecStartPre=/opt/myapp/pre-start.sh
ExecStart=/usr/bin/node /opt/myapp/server.js
ExecStartPost=/opt/myapp/post-start.sh
ExecStop=/bin/kill -SIGTERM $MAINPID
ExecReload=/bin/kill -SIGHUP $MAINPID
Restart=on-failure
RestartSec=5
TimeoutStartSec=30
TimeoutStopSec=30
StandardOutput=journal
StandardError=journal
SyslogIdentifier=myapp
LimitNOFILE=65535

[Install]
WantedBy=multi-user.target
```

Let's break down **every single section**:

### [Unit] Section — Metadata & Dependencies

This section describes the unit and defines its relationships with other units.

```ini
[Unit]
# Human-readable description — shows up in systemctl status
Description=My Awesome Application

# Link to documentation
Documentation=https://myapp.example.com/docs

# Start this service AFTER these units are started
# (ordering only — doesn't require them to exist)
After=network.target postgresql.service

# These units SHOULD be started alongside this service
# (weak dependency — service starts even if these fail)
Wants=postgresql.service

# These units MUST be running for this service to start
# (strong dependency — service fails if these fail)
Requires=network-online.target

# If these units fail, stop this service too
BindsTo=docker.service

# If ANY of these are not started, don't start this service
# (stronger than Requires — checks active state)
Requisite=network-online.target

# Opposite of After — start BEFORE these units
Before=other-app.service

# Conflicts — stop this unit if the conflicting unit starts
Conflicts=other-conflicting-app.service
```

**Understanding Dependency Types:**

```
Wants=      →  "I'd LIKE this to run too, but I'll survive without it"
Requires=   →  "I NEED this to run. If it fails at startup, I fail too"
BindsTo=    →  "I'm BOUND to this. If it stops anytime, stop me too"
Requisite=  →  "CHECK if this is already running. If not, don't even try me"
Conflicts=  →  "If this other thing starts, STOP me (and vice versa)"

After=      →  "Start me AFTER this" (ordering only, not dependency)
Before=     →  "Start me BEFORE this" (ordering only, not dependency)
```

**Common Mistake:** Using `After=` alone doesn't create a dependency! You usually need both:

```ini
# CORRECT: Start after AND depend on postgresql
After=postgresql.service
Requires=postgresql.service

# WRONG: Only ordering, no dependency — if postgresql isn't installed,
# this service will still try to start
After=postgresql.service
```

### [Service] Section — How to Run the Service

This is the heart of a service unit file.

#### Service Types

The `Type=` directive tells systemd how your service behaves:

```ini
# ── simple (default) ──
# The process started by ExecStart IS the main service process
# Systemd considers it "started" immediately after ExecStart runs
Type=simple
ExecStart=/usr/bin/myapp

# ── forking ──
# The process forks a child and the parent exits
# Systemd waits for the parent to exit, then tracks the child
# Common for traditional daemons (Apache, MySQL old-style)
Type=forking
PIDFile=/var/run/myapp.pid
ExecStart=/usr/sbin/myapp --daemon

# ── oneshot ──
# The process does its job and exits
# Systemd waits for it to finish before considering it "started"
# Great for setup/initialization scripts
Type=oneshot
ExecStart=/usr/local/bin/setup-firewall.sh
RemainAfterExit=yes    # Consider service "active" even after process exits

# ── notify ──
# Like simple, but the service sends a notification to systemd
# when it's ready (using sd_notify)
# Best for services that need time to initialize
Type=notify
ExecStart=/usr/bin/myapp

# ── dbus ──
# Like notify, but uses D-Bus for readiness notification
Type=dbus
BusName=org.example.myapp

# ── idle ──
# Like simple, but waits until all other jobs are done
# Useful for services whose output you want to see at boot
Type=idle
ExecStart=/usr/bin/myapp
```

**Which Type Should You Use?**

```
Is your app a one-time script?          → Type=oneshot
Does your app fork/daemonize itself?    → Type=forking
Does your app use sd_notify()?          → Type=notify
Is your app a regular long-running app? → Type=simple  ← Most common
```

#### Exec Directives

```ini
[Service]
# Run BEFORE the main process starts
ExecStartPre=/opt/myapp/check-config.sh

# The main command to start the service
ExecStart=/usr/bin/myapp --config /etc/myapp/config.yml

# Run AFTER the main process starts successfully
ExecStartPost=/usr/bin/notify-monitoring "myapp started"

# Command to reload configuration (without full restart)
ExecReload=/bin/kill -HUP $MAINPID

# Command to stop the service
ExecStop=/bin/kill -SIGTERM $MAINPID

# Run AFTER the service stops
ExecStopPost=/usr/local/bin/cleanup.sh
```

#### Restart Policies

This is **critical** for production services:

```ini
[Service]
# When to restart the service:
Restart=no              # Never restart (default)
Restart=always          # Always restart, no matter what
Restart=on-success      # Only restart if exited with code 0
Restart=on-failure      # Restart on non-zero exit, signal, timeout, watchdog
Restart=on-abnormal     # Restart on signal, timeout, watchdog (not clean exit)
Restart=on-abort        # Restart only on signal (crash)
Restart=on-watchdog     # Restart only on watchdog timeout

# How long to wait before restarting
RestartSec=5

# How many times to try restarting within a time window
StartLimitBurst=5
StartLimitIntervalSec=60
# (if it restarts 5 times within 60 seconds, stop trying)
```

**For production services, you almost always want:**

```ini
Restart=on-failure
RestartSec=5
StartLimitBurst=5
StartLimitIntervalSec=60
```

#### User & Group

```ini
[Service]
# Run the service as a specific user (not root!)
User=myapp
Group=myapp

# Or create a dynamic user (systemd creates it temporarily)
DynamicUser=yes
```

**Security Best Practice:** Never run services as root unless absolutely necessary.

#### Environment Variables

```ini
[Service]
# Set individual variables
Environment=NODE_ENV=production
Environment=PORT=3000

# Or load from a file
EnvironmentFile=/etc/myapp/env
# The env file looks like:
#   NODE_ENV=production
#   PORT=3000
#   DB_HOST=localhost
```

#### Resource Limits

```ini
[Service]
# Maximum open files
LimitNOFILE=65535

# Maximum processes
LimitNPROC=4096

# CPU quota (50% of one core)
CPUQuota=50%

# Memory limit
MemoryMax=512M

# I/O weight (relative to other services, 1-10000)
IOWeight=500
```

### [Install] Section — When to Start

```ini
[Install]
# This service should be started when multi-user.target is reached
# (multi-user.target = normal boot without GUI)
WantedBy=multi-user.target

# Or if you want it to start in graphical mode:
WantedBy=graphical.target

# Create an alias for this service
Alias=my-other-name.service

# Also enable these units when this unit is enabled
Also=myapp-worker.service
```

**What does `WantedBy=` actually do?**

When you run `systemctl enable myapp.service`, systemd creates a symbolic link:

```
/etc/systemd/system/multi-user.target.wants/myapp.service 
    → /etc/systemd/system/myapp.service
```

This tells systemd: "When you reach multi-user.target during boot, also start myapp.service."

## 🎯 Targets — System States (Replacing Runlevels)

### What is a Target?

A **target** is a group of units that represent a system state. Targets replaced the old **runlevels** from SysVinit.

Think of targets like checkpoints during boot:

```
poweroff.target        = System is off
  │
rescue.target          = Single-user mode (for emergencies)
  │
multi-user.target      = Full multi-user mode (CLI, no GUI)
  │                       Most servers stop here
graphical.target       = Full multi-user mode + GUI desktop
  │                       Desktop systems stop here
reboot.target          = System is rebooting
```

### Old Runlevels vs New Targets

| Runlevel | Target | Description |
|---|---|---|
| 0 | `poweroff.target` | Halt the system |
| 1 | `rescue.target` | Single user mode |
| 2 | `multi-user.target` | Multi-user (no networking in some distros) |
| 3 | `multi-user.target` | Multi-user with networking |
| 4 | `multi-user.target` | Custom (usually same as 3) |
| 5 | `graphical.target` | Multi-user with GUI |
| 6 | `reboot.target` | Reboot |

### Working with Targets

```bash
# See the current default target
systemctl get-default
# Output: multi-user.target  (for servers)
# Output: graphical.target   (for desktops)

# Set the default target
sudo systemctl set-default multi-user.target

# Switch to a different target RIGHT NOW
sudo systemctl isolate rescue.target     # Go to single-user mode
sudo systemctl isolate multi-user.target # Go to multi-user mode
sudo systemctl isolate graphical.target  # Go to GUI mode

# List all targets
systemctl list-units --type=target

# See what units a target wants
systemctl list-dependencies multi-user.target
```

### How Boot Works with Targets

```
BIOS/UEFI → GRUB Bootloader → Linux Kernel → Systemd (PID 1)
                                                  │
                                                  ▼
                                          default.target
                                       (usually multi-user.target)
                                                  │
                                 ┌────────────────┼────────────────┐
                                 │                │                │
                            basic.target    network.target   other targets
                                 │                │
                           ┌─────┼──────┐    ┌────┼─────┐
                           │     │      │    │    │     │
                        sysinit paths  slices sshd nginx postgres
                        .target               .svc .svc  .svc
```

### Creating a Custom Target

You can create your own target to group services:

```ini
# /etc/systemd/system/myapp-stack.target
[Unit]
Description=My Application Stack
Requires=multi-user.target
After=multi-user.target

[Install]
WantedBy=multi-user.target
```

Then make your services part of this target:

```ini
# In your service files, change Install section to:
[Install]
WantedBy=myapp-stack.target
```

Now you can start/stop your entire application stack at once:

```bash
# Start everything
sudo systemctl start myapp-stack.target

# Stop everything
sudo systemctl stop myapp-stack.target

# Check status of everything
systemctl list-dependencies myapp-stack.target
```

## 🔌 Socket Units — On-Demand Service Activation

### What is Socket Activation?

Socket activation is one of the **coolest features** of systemd. Here's the idea:

**Without socket activation:**
```
Boot → Start nginx immediately → nginx uses RAM even if nobody visits
Boot → Start sshd immediately → sshd uses RAM even if nobody SSHs in
```

**With socket activation:**
```
Boot → Systemd listens on port 80 (not nginx)
     → Someone visits the website
     → Systemd says "Hey nginx, wake up! Someone's here!"
     → Systemd starts nginx and hands it the connection
     → nginx handles the request
```

This means:
- **Faster boot** — services start only when needed
- **Less memory** — idle services don't consume resources
- **No ordering issues** — systemd holds connections until the service is ready

### How Socket Activation Works

```
Step 1: Systemd creates the socket (e.g., port 80)
         ┌─────────────┐
         │   Systemd   │
         │ listening on │
         │   port 80   │
         └──────┬──────┘
                │
Step 2: Client connects
         ┌─────────────┐
         │   Client    │──── connects to port 80
         └─────────────┘
                │
Step 3: Systemd starts the service and passes the socket
         ┌─────────────┐     ┌─────────────┐
         │   Systemd   │────▶│   nginx     │
         │  passes fd  │     │ handles req  │
         └─────────────┘     └─────────────┘
```

### Creating a Socket Unit

A socket unit always comes in a **pair** — a `.socket` file and a matching `.service` file:

```ini
# /etc/systemd/system/myapp.socket
[Unit]
Description=My App Socket

[Socket]
# Listen on a TCP port
ListenStream=8080

# Or listen on a Unix socket
# ListenStream=/run/myapp.sock

# Number of connections to queue
Backlog=128

# Set socket permissions (for Unix sockets)
# SocketUser=myapp
# SocketGroup=myapp
# SocketMode=0660

# Accept connections or pass the listening socket?
Accept=no
# Accept=no: Pass the listening socket to one service instance (most common)
# Accept=yes: Spawn a new service instance for each connection (like inetd)

[Install]
WantedBy=sockets.target
```

```ini
# /etc/systemd/system/myapp.service
# (the name MUST match the socket file name)
[Unit]
Description=My App Service
Requires=myapp.socket

[Service]
Type=simple
ExecStart=/usr/bin/myapp
User=myapp

[Install]
WantedBy=multi-user.target
```

Now enable and start the socket (not the service!):

```bash
sudo systemctl enable myapp.socket
sudo systemctl start myapp.socket

# The service is NOT running yet
systemctl status myapp.service  # inactive (dead)

# But the socket IS listening
systemctl status myapp.socket   # active (listening)
ss -tlnp | grep 8080           # systemd is listening on 8080

# Now connect to it...
curl http://localhost:8080

# And the service starts automatically!
systemctl status myapp.service  # active (running)
```

### Real-World Socket Activation Example: SSH

Many systems use socket activation for SSH:

```ini
# /usr/lib/systemd/system/sshd.socket
[Unit]
Description=OpenSSH Server Socket

[Socket]
ListenStream=22
Accept=yes

[Install]
WantedBy=sockets.target
```

```ini
# /usr/lib/systemd/system/sshd@.service
# The @ means it's a template — one instance per connection
[Unit]
Description=OpenSSH Per-Connection Server

[Service]
ExecStart=-/usr/sbin/sshd -i
StandardInput=socket
```

## ⏰ Systemd Timers — The Modern Cron

### Why Timers Over Cron?

Systemd timers are a **better alternative** to cron jobs:

| Feature | Cron | Systemd Timers |
|---|---|---|
| Logging | ❌ Manual (redirect output) | ✅ Automatic (journald) |
| Dependencies | ❌ None | ✅ Full dependency support |
| Missed runs | ❌ Lost forever | ✅ `Persistent=true` catches up |
| Resource control | ❌ None | ✅ Cgroups, CPU/memory limits |
| Monitoring | ❌ Hard | ✅ `systemctl status` |
| Random delay | ❌ Not built-in | ✅ `RandomizedDelaySec=` |
| Calendar events | ⚠️ Limited syntax | ✅ Rich calendar expressions |
| Security sandboxing | ❌ None | ✅ Full sandboxing support |

### Creating a Timer

A timer always comes in a **pair** — a `.timer` file and a matching `.service` file:

**Step 1: Create the Service (what to run)**

```ini
# /etc/systemd/system/backup.service
[Unit]
Description=Run Daily Backup

[Service]
Type=oneshot
ExecStart=/usr/local/bin/backup.sh
User=backup
Nice=10
IOSchedulingClass=idle
```

Note: No `[Install]` section needed — the timer will trigger this service.

**Step 2: Create the Timer (when to run)**

```ini
# /etc/systemd/system/backup.timer
[Unit]
Description=Run Backup Daily at 2 AM

[Timer]
# When to run (calendar-based — like cron)
OnCalendar=*-*-* 02:00:00

# Or use relative time (monotonic):
# OnBootSec=15min           # 15 minutes after boot
# OnUnitActiveSec=1h        # 1 hour after last activation
# OnUnitInactiveSec=1h      # 1 hour after service becomes inactive
# OnStartupSec=5min         # 5 minutes after systemd starts

# If the system was off when the timer should have fired, run it when it comes back
Persistent=true

# Add a random delay (up to 5 minutes) to avoid thundering herd
RandomizedDelaySec=5min

# Send accuracy (how precise the timing needs to be)
# Lower = more precise but more wake-ups
AccuracySec=1min

[Install]
WantedBy=timers.target
```

**Step 3: Enable and Start**

```bash
# Reload systemd to pick up new files
sudo systemctl daemon-reload

# Enable the timer (not the service!)
sudo systemctl enable backup.timer

# Start the timer NOW
sudo systemctl start backup.timer

# Check timer status
systemctl status backup.timer

# List all timers
systemctl list-timers --all
```

### OnCalendar Syntax — The Time Language

The `OnCalendar=` syntax is **much more powerful** than cron:

```
Format: DayOfWeek Year-Month-Day Hour:Minute:Second

Examples:
──────────────────────────────────────────────────────
*-*-* 02:00:00           = Every day at 2:00 AM
Mon *-*-* 09:00:00       = Every Monday at 9:00 AM
Mon,Fri *-*-* 09:00:00   = Every Monday and Friday at 9:00 AM
*-*-01 00:00:00          = First day of every month at midnight
*-01-01 00:00:00         = January 1st every year at midnight
*-*-* *:00:00            = Every hour on the hour
*-*-* *:*:00             = Every minute
*-*-* *:00/15:00         = Every 15 minutes (0, 15, 30, 45)
*-*-* 09..17:00:00       = Every hour from 9 AM to 5 PM
Mon..Fri *-*-* 08:00:00  = Every weekday at 8:00 AM

Special shortcuts:
──────────────────────────────────────────────────────
minutely      = Every minute
hourly        = Every hour
daily         = Every day at midnight
weekly        = Every Monday at midnight
monthly       = First day of each month at midnight
yearly        = January 1st at midnight
quarterly     = Jan, Apr, Jul, Oct 1st at midnight
semiannually  = Jan, Jul 1st at midnight
```

**Testing your calendar expression:**

```bash
# This is SUPER useful — test before deploying!
systemd-analyze calendar "Mon..Fri *-*-* 09:00:00"
# Output:
#   Original form: Mon..Fri *-*-* 09:00:00
#   Normalized form: Mon..Fri *-*-* 09:00:00
#   Next elapse: Mon 2024-01-15 09:00:00 UTC
#        (in UTC): Mon 2024-01-15 09:00:00 UTC
#        From now: 2 days left

# See the next 5 trigger times:
systemd-analyze calendar --iterations=5 "Mon..Fri *-*-* 09:00:00"
```

### Monotonic Timers (Relative Time)

Instead of calendar-based, you can use relative timers:

```ini
[Timer]
# 15 minutes after boot
OnBootSec=15min

# 1 hour after the timer unit was last activated
OnUnitActiveSec=1h

# 30 minutes after the triggered service became inactive (finished)
OnUnitInactiveSec=30min

# 10 minutes after systemd started
OnStartupSec=10min
```

**When to use which:**

```
Calendar (OnCalendar)       → "Run every day at 2 AM"
OnBootSec                   → "Run 5 minutes after every boot"
OnUnitActiveSec             → "Run every 4 hours" (from activation)
OnUnitInactiveSec           → "Run 30 min after the last run finishes"
```

### Replacing Common Cron Jobs

**Cron: Run a script every 5 minutes**

```
# Old cron way:
*/5 * * * * /usr/local/bin/check-health.sh
```

```ini
# Systemd timer way:
# /etc/systemd/system/health-check.service
```

## 🔧 Essential systemctl Commands

### Service Management

```bash
# ── Starting & Stopping ──
sudo systemctl start nginx.service      # Start a service
sudo systemctl stop nginx.service       # Stop a service
sudo systemctl restart nginx.service    # Stop then start
sudo systemctl reload nginx.service     # Reload config without restart
sudo systemctl reload-or-restart nginx  # Reload if possible, else restart

# ── Enable & Disable (boot behavior) ──
sudo systemctl enable nginx.service     # Start on boot
sudo systemctl disable nginx.service    # Don't start on boot
sudo systemctl enable --now nginx       # Enable AND start immediately
sudo systemctl disable --now nginx      # Disable AND stop immediately

# ── Status & Information ──
systemctl status nginx.service          # Detailed status
systemctl is-active nginx.service       # Just "active" or "inactive"
systemctl is-enabled nginx.service      # Just "enabled" or "disabled"
systemctl is-failed nginx.service       # Just "failed" or not
systemctl show nginx.service            # Show all properties
systemctl show -p MainPID nginx         # Show specific property
systemctl cat nginx.service             # Show the unit file contents

# ── Listing ──
systemctl list-units                    # All loaded units
systemctl list-units --type=service     # All loaded services
systemctl list-units --failed           # All failed units
systemctl list-unit-files               # All installed unit files
systemctl list-unit-files --type=service # All installed service files
systemctl list-dependencies nginx       # Show dependency tree
```

### Editing Unit Files

```bash
# ── The RIGHT way to customize a unit ──

# Create an override file (opens your editor)
sudo systemctl edit nginx.service
# This creates: /etc/systemd/system/nginx.service.d/override.conf
# Add ONLY the sections/directives you want to change

# Example override — increase memory limit:
# [Service]
# MemoryMax=1G

# If you want to edit the FULL unit file (not just override):
sudo systemctl edit --full nginx.service
# This copies the full file to /etc/systemd/system/ and opens it

# After ANY edit to unit files:
sudo systemctl daemon-reload    # ALWAYS run this!

# See what overrides exist:
systemd-delta

# Revert to the original (remove overrides):
sudo systemctl revert nginx.service
```

### Masking Services

**Masking** is stronger than disabling — it makes it **impossible** to start a service:

```bash
# Mask a service (creates a symlink to /dev/null)
sudo systemctl mask nginx.service
# Now even 'systemctl start nginx' will fail!

# Unmask it
sudo systemctl unmask nginx.service
```

**When to mask?**
- You want to prevent a service from **ever** starting accidentally
- You're replacing a service with another (e.g., iptables → nftables)

## 📝 journalctl — Systemd's Logging System

Systemd has its own logging daemon called **journald** that collects logs from all services.

### Basic Log Viewing

```bash
# View all logs (oldest first)
journalctl

# View all logs (newest first)
journalctl -r

# Follow logs in real time (like tail -f)
journalctl -f

# Show only the last 50 lines
journalctl -n 50

# Show logs since last boot
journalctl -b

# Show logs from previous boot
journalctl -b -1

# Show logs from 2 boots ago
journalctl -b -2
```

### Filtering Logs

```bash
# ── By Service ──
journalctl -u nginx.service               # Logs for nginx
journalctl -u nginx.service -u php-fpm    # Multiple services
journalctl -u nginx.service -f            # Follow nginx logs

# ── By Time ──
journalctl --since "2024-01-15 09:00:00"
journalctl --since "1 hour ago"
journalctl --since "today"
journalctl --since "yesterday" --until "today"
journalctl --since "2024-01-15" --until "2024-01-16"

# ── By Priority ──
journalctl -p err                 # Errors and above
journalctl -p warning             # Warnings and above
# Priority levels: emerg, alert, crit, err, warning, notice, info, debug

# ── By Process ──
journalctl _PID=1234              # By PID
journalctl _UID=1000              # By user ID
journalctl _COMM=nginx            # By command name

# ── Kernel Messages ──
journalctl -k                     # Kernel messages (like dmesg)
journalctl -k -b                  # Kernel messages from this boot
```

### Output Formats

```bash
# Default (human-readable)
journalctl -u nginx

# JSON format (great for parsing)
journalctl -u nginx -o json

# JSON pretty-printed
journalctl -u nginx -o json-pretty

# Short with timestamps
journalctl -u nginx -o short-iso

# Verbose (all fields)
journalctl -u nginx -o verbose

# Only the message (no metadata)
journalctl -u nginx -o cat
```

### Log Management

```bash
# See how much disk space logs are using
journalctl --disk-usage

# Clean up old logs — keep only last 2 weeks
sudo journalctl --vacuum-time=2weeks

# Clean up old logs — keep only 500MB
sudo journalctl --vacuum-size=500M

# Clean up old logs — keep only last 5 log files
sudo journalctl --vacuum-files=5

# Configure log retention in /etc/systemd/journald.conf:
# [Journal]
# SystemMaxUse=500M        # Max disk usage
# SystemMaxFileSize=50M    # Max size per file
# MaxRetentionSec=2weeks   # Max retention time
# Compress=yes             # Compress old logs
```

## 🔒 Systemd Security Hardening

Systemd provides **powerful sandboxing** features to lock down services:

```ini
[Service]
# ── File System ──
ProtectSystem=strict          # Make /usr, /boot, /efi read-only
ProtectHome=yes               # Make /home, /root, /run/user inaccessible
ReadWritePaths=/var/lib/myapp # Allow writes only to specific paths
PrivateTmp=yes                # Give the service its own /tmp
ReadOnlyPaths=/etc            # Make /etc read-only for this service

# ── Network ──
PrivateNetwork=yes            # No network access (use for local-only services)
RestrictAddressFamilies=AF_INET AF_INET6  # Only IPv4 and IPv6

# ── Capabilities ──
NoNewPrivileges=yes           # Can't gain new privileges
CapabilityBoundingSet=        # Drop ALL capabilities
AmbientCapabilities=CAP_NET_BIND_SERVICE  # Allow binding to ports < 1024

# ── System Calls ──
SystemCallFilter=@system-service  # Only allow common system calls
SystemCallArchitectures=native    # Only native arch (no 32-bit compat)

# ── Other ──
ProtectKernelTunables=yes     # Protect /proc/sys, /sys
ProtectKernelModules=yes      # Prevent module loading
ProtectControlGroups=yes      # Make cgroup hierarchy read-only
ProtectClock=yes              # Prevent changing system clock
LockPersonality=yes           # Lock the execution domain
MemoryDenyWriteExecute=yes    # Prevent W+X memory mappings
RestrictRealtime=yes          # Prevent real-time scheduling
RestrictSUIDSGID=yes          # Prevent SUID/SGID files
PrivateDevices=yes            # No access to physical devices
```

### Checking Service Security

```bash
# Analyze security of a service (gives a score)
systemd-analyze security nginx.service

# Output example:
# → Overall exposure level for nginx.service: 4.1 MEDIUM 😐
#
# NAME                        DESCRIPTION                    EXPOSURE
# PrivateNetwork=             Service has access to network  0.5
# User=/DynamicUser=          Service runs as root           0.4
# CapabilityBoundingSet=      Service has broad capabilities 0.3
# ...

# Get security scores for ALL services
systemd-analyze security
```

## 🛠️ Practical Examples

### Example 1: Deploy a Node.js Application

```ini
# /etc/systemd/system/nodeapp.service
[Unit]
Description=Node.js Production Application
Documentation=https://github.com/myorg/myapp
After=network.target

[Service]
Type=simple
User=nodeapp
Group=nodeapp
WorkingDirectory=/opt/nodeapp

# Environment
Environment=NODE_ENV=production
Environment=PORT=3000
EnvironmentFile=-/opt/nodeapp/.env
# The "-" means "don't fail if file doesn't exist"

# Process
ExecStart=/usr/bin/node /opt/nodeapp/server.js
ExecReload=/bin/kill -HUP $MAINPID

# Restart policy
Restart=on-failure
RestartSec=10
StartLimitBurst=3
StartLimitIntervalSec=60

# Logging
StandardOutput=journal
StandardError=journal
SyslogIdentifier=nodeapp

# Resource limits
LimitNOFILE=65535
MemoryMax=512M
CPUQuota=80%

# Security hardening
NoNewPrivileges=yes
ProtectSystem=strict
ProtectHome=yes
ReadWritePaths=/opt/nodeapp/logs /opt/nodeapp/uploads
PrivateTmp=yes
ProtectKernelTunables=yes
ProtectControlGroups=yes

[Install]
WantedBy=multi-user.target
```

```bash
# Deploy it:
sudo systemctl daemon-reload
sudo systemctl enable --now nodeapp.service
systemctl status nodeapp.service
journalctl -u nodeapp.service -f
```

### Example 2: Automated Database Backup with Timer

```ini
# /etc/systemd/system/db-backup.service
[Unit]
Description=PostgreSQL Database Backup
After=postgresql.service
Requires=postgresql.service

[Service]
Type=oneshot
User=postgres
ExecStart=/bin/bash -c '/usr/bin/pg_dumpall | gzip > /backups/db-$(date +%%Y%%m%%d-%%H%%M%%S).sql.gz'
ExecStartPost=/usr/local/bin/cleanup-old-backups.sh 30
# Note: %% is needed to escape % in systemd unit files
```

```ini
# /etc/systemd/system/db-backup.timer
[Unit]
Description=Daily Database Backup at 3 AM

[Timer]
OnCalendar=*-*-* 03:00:00
Persistent=true
RandomizedDelaySec=15min

[Install]
WantedBy=timers.target
```

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now db-backup.timer
systemctl list-timers | grep db-backup
```
