## 🤔 Why Should a DevOps Engineer Master Linux Troubleshooting?

Here's a hard truth: **things will break.** Servers will crash, services will hang, disks will fill up, and networks will go down. The question isn't *if* — it's *when*.

What separates a **great** DevOps engineer from an average one is how fast they can:

1. **Detect** the problem
2. **Diagnose** the root cause
3. **Fix** it
4. **Prevent** it from happening again

| Scenario | How Often It Happens |
|---|---|
| Service crashes at 3 AM | Very often |
| Disk full, deployments failing | Weekly at many companies |
| Server running slow, users complaining | Constant |
| Network issues, services can't communicate | Regular |
| Memory leak crashing an app | Periodically |
| Permission denied errors blocking workflows | Daily |
| Boot failures after an update | Occasionally |

**Every single one** of these can be diagnosed with the tools and techniques in this chapter.

## 🧠 The Troubleshooting Mindset

Before we jump into tools, let's talk about **how to think** when something breaks. This framework will save you hours:

### The S.O.S. Method

```
S — Symptoms       → What EXACTLY is happening? (not what you THINK is happening)
O — Observe         → Gather data. Logs, metrics, error messages. Don't guess.
S — Solve           → Make ONE change at a time. Test. Repeat.
```

### The Golden Rules of Troubleshooting

```
Rule 1: Don't Panic
        └── Take a breath. Read the error message carefully.
        └── 80% of problems are in the error message itself.

Rule 2: Reproduce the Problem
        └── Can you make it happen again?
        └── If yes, you're halfway to fixing it.

Rule 3: Change ONE Thing at a Time
        └── If you change 5 things and it works, you don't know what fixed it.
        └── You'll be back here next week.

Rule 4: Check the Obvious First
        └── Is the server running? Is the network cable plugged in?
        └── Is there disk space? Is the service actually started?
        └── You'd be surprised how often the answer is simple.

Rule 5: Read the Logs
        └── Logs are your best friend. Always check logs FIRST.
        └── Most people skip this step. Don't be most people.

Rule 6: Document What You Did
        └── Write down what you tried and what worked.
        └── Future-you will thank present-you.
```

## 🔍 System Overview — First Things First

When you SSH into a broken server, **start here**. Get the big picture before diving into specifics.

### Quick Health Check (Run These First!)

```bash
# ── Who am I and where am I? ──
whoami                    # What user am I?
hostname                  # What server am I on?
uname -a                  # Kernel version, architecture
cat /etc/os-release       # OS name and version

# ── How long has this server been running? ──
uptime
# Output: 10:42:07 up 45 days, 3:21, 2 users, load average: 2.50, 1.80, 0.95
#                   ↑                           ↑
#              Server uptime              Load averages (1min, 5min, 15min)

# ── What's the current date/time? ──
date                      # Is the clock correct?
timedatectl               # Timezone, NTP sync status

# ── Any critical messages right now? ──
dmesg -T --level=err,crit,alert,emerg | tail -20
# Shows kernel errors with human-readable timestamps

# ── Failed services? ──
systemctl --failed
# This alone will tell you if something crashed
```

### Understanding Load Average

The `uptime` command shows **load average** — one of the most important numbers on a Linux system:

```
load average: 2.50, 1.80, 0.95
              ↓      ↓      ↓
           1 min  5 min  15 min

What does it mean?
─────────────────────────────────────────────
The load average is the average number of processes that are:
  1. Running on a CPU, OR
  2. Waiting for a CPU

How to interpret it:
─────────────────────────────────────────────
First, find out how many CPU cores you have:
  $ nproc
  4    ← This server has 4 cores

Now compare load average to core count:

Load Average  │  Cores  │  Meaning
──────────────┼─────────┼────────────────────
0.50          │  4      │  ✅ System is idle (12.5% busy)
2.00          │  4      │  ✅ Moderate load (50% busy)
4.00          │  4      │  ⚠️ Fully loaded (100% busy)
8.00          │  4      │  🔴 Overloaded! (2x capacity)
16.00         │  4      │  🔥 SEVERELY overloaded!

Rule of thumb:
  load_average < num_cores     → System is fine
  load_average == num_cores    → System is at capacity
  load_average > num_cores     → System is overloaded
```

## 💾 Disk Issues — "No Space Left on Device"

This is probably the **#1 most common** issue in DevOps. Let's master it.

### Diagnosing Disk Space Issues

```bash
# ── Check overall disk usage ──
df -h
# Output:
# Filesystem      Size  Used Avail Use% Mounted on
# /dev/sda1        50G   48G  2.0G  96% /          ← 🔴 Almost full!
# /dev/sda2       200G   50G  150G  25% /data       ← ✅ Fine
# tmpfs            16G  500M   16G   4% /tmp        ← ✅ Fine

# ── Which directory is eating all the space? ──
# Start from root and find the biggest offenders
sudo du -sh /* 2>/dev/null | sort -rh | head -10
# Output:
# 30G   /var
# 10G   /home
# 5G    /usr
# 2G    /opt

# Drill down into /var
sudo du -sh /var/* 2>/dev/null | sort -rh | head -10
# Output:
# 25G   /var/log       ← Aha! Logs are eating the disk!
# 3G    /var/lib
# 1G    /var/cache

# Keep drilling down
sudo du -sh /var/log/* 2>/dev/null | sort -rh | head -10
# Output:
# 20G   /var/log/nginx/access.log   ← Found the culprit!
```

### Finding Large Files

```bash
# Find files larger than 100MB
sudo find / -type f -size +100M -exec ls -lh {} \; 2>/dev/null | sort -k5 -rh

# Find files larger than 1GB
sudo find / -type f -size +1G 2>/dev/null

# Find the top 20 largest files on the system
sudo find / -type f -exec du -h {} + 2>/dev/null | sort -rh | head -20

# Find recently modified large files (changed in last 24 hours)
sudo find / -type f -size +100M -mtime -1 2>/dev/null
```

### Inode Exhaustion — The Sneaky Disk Problem

Sometimes `df -h` shows free space, but you still get "No space left on device." This means you've run out of **inodes** (file slots):

```bash
# Check inode usage
df -i
# Output:
# Filesystem      Inodes   IUsed   IFree IUse% Mounted on
# /dev/sda1      3276800 3276800       0  100% /   ← 🔴 No inodes left!

# This happens when you have MILLIONS of tiny files.
# Common culprits: session files, cache files, tmp files

# Find directories with the most files
sudo find / -xdev -printf '%h\n' 2>/dev/null | sort | uniq -c | sort -rn | head -10
# Output:
# 2500000  /var/lib/php/sessions    ← 2.5 million session files!
# 500000   /tmp/cache

# Clean them up
sudo find /var/lib/php/sessions -type f -mtime +1 -delete
```

### Fixing Disk Space Issues

```bash
# ── Quick Wins ──

# 1. Clean package manager cache
sudo apt clean                          # Debian/Ubuntu
sudo yum clean all                      # CentOS/RHEL
sudo dnf clean all                      # Fedora

# 2. Remove old kernels (keep the current one!)
sudo apt autoremove --purge             # Debian/Ubuntu

# 3. Clean journal logs
sudo journalctl --vacuum-size=100M
sudo journalctl --vacuum-time=7d

# 4. Find and remove old log files
sudo find /var/log -name "*.gz" -mtime +30 -delete
sudo find /var/log -name "*.log.*" -mtime +30 -delete

# 5. Clean /tmp
sudo find /tmp -type f -mtime +7 -delete

# 6. Check for deleted files still held open
sudo lsof +L1
# This shows files that have been deleted but are still being held open
# by a running process. The disk space won't be freed until the process
# is restarted!
# Fix: restart the process holding the file
sudo systemctl restart nginx    # If nginx was holding a deleted log file

# 7. Docker cleanup (if using Docker)
docker system prune -af                 # Remove unused images, containers, networks
docker volume prune -f                  # Remove unused volumes
```

### Monitoring Disk Space Proactively

```bash
# Simple disk space check script
#!/bin/bash
THRESHOLD=85
PARTITION="/"

USAGE=$(df -h "$PARTITION" | awk 'NR==2 {print $5}' | tr -d '%')

if [ "$USAGE" -gt "$THRESHOLD" ]; then
    echo "WARNING: Disk usage on $PARTITION is ${USAGE}%!"
    # Send alert (email, Slack, PagerDuty, etc.)
fi
```

## 🧠 Memory Issues — "Out of Memory" / OOM Killer

### Understanding Memory in Linux

```bash
# ── Check memory usage ──
free -h
# Output:
#               total        used        free      shared  buff/cache   available
# Mem:           16G         12G        500M        200M        3.5G        3.2G
# Swap:          4G          2G          2G

# Important: "free" doesn't mean wasted!
# Linux uses free RAM for disk cache (buff/cache).
# "available" = memory that CAN be freed for applications
#
# Think of it like this:
#  total    = 16G (your RAM sticks)
#  used     = 12G (actively used by apps + kernel)
#  free     = 500M (truly unused — this being low is NORMAL)
#  buff/cache = 3.5G (used for disk cache — can be freed if needed)
#  available = 3.2G (THIS is what matters — real free memory)
```

```
When to worry:
─────────────────────────────────────────────
"available" > 20% of total    → ✅ Fine
"available" = 10-20% of total → ⚠️ Getting tight
"available" < 10% of total    → 🔴 Low memory!
"available" ≈ 0 AND swap used → 🔥 Danger zone — OOM killer coming!
```

### Finding Memory-Hungry Processes

```bash
# ── Top memory consumers ──
# Sort by memory usage (%MEM column)
ps aux --sort=-%mem | head -15

# More readable format
ps -eo pid,ppid,user,%mem,%cpu,rss,comm --sort=-%mem | head -15
# RSS = Resident Set Size (actual physical memory used, in KB)

# ── Live monitoring ──
top
# Press 'M' to sort by memory usage
# Press 'P' to sort by CPU usage
# Press 'q' to quit

# Better alternative:
htop
# (install with: sudo apt install htop)

# ── Memory usage by process name ──
# How much memory is nginx using in total?
ps -C nginx -o rss= | awk '{sum += $1} END {print sum/1024 " MB"}'

# ── Detailed process memory info ──
cat /proc/PID/status | grep -i mem
# VmSize = virtual memory (includes shared libs, mapped files)
# VmRSS  = physical memory (what's actually in RAM)
# VmSwap = memory that's been swapped to disk
```

### The OOM Killer

When Linux completely runs out of memory (RAM + swap), the kernel's **OOM Killer** (Out of Memory Killer) activates and kills processes to free up memory.

```bash
# ── Check if OOM Killer has killed anything ──
dmesg | grep -i "oom"
# Or with timestamps:
dmesg -T | grep -i "oom"

# Check journal
journalctl -k | grep -i "oom"
# Look for lines like:
# "Out of memory: Killed process 1234 (nginx) total-vm:500000kB"

# ── See which processes the OOM Killer would target ──
# Lower score = less likely to be killed
# Higher score = more likely to be killed
for proc in /proc/[0-9]*/; do
    if [ -f "$proc/oom_score" ]; then
        echo "$(cat $proc/oom_score) $(cat $proc/comm 2>/dev/null)" 
    fi
done | sort -rn | head -15

# ── Protect a process from OOM Killer ──
# Set oom_score_adj to -1000 (never kill this process)
echo -1000 > /proc/PID/oom_score_adj

# Or in a systemd service file:
# [Service]
# OOMScoreAdjust=-1000
```

### Diagnosing Memory Leaks

```bash
# Watch a process's memory over time
watch -n 5 'ps -o pid,rss,vsz,comm -p PID'

# If RSS keeps growing over time without stopping → memory leak!

# More detailed tracking with smem (install: sudo apt install smem)
smem -rs pss -t | tail -15
# PSS = Proportional Set Size (fair share of shared memory)
# USS = Unique Set Size (memory unique to this process)

# Check if swap is being heavily used
vmstat 1 5
# Look at 'si' (swap in) and 'so' (swap out) columns
# If they're consistently > 0, the system is swapping
# Swapping is VERY slow and indicates memory pressure
```

## 🔥 CPU Issues — "Server is Slow"

### Diagnosing CPU Problems

```bash
# ── Quick CPU overview ──
# Check CPU count and current load
nproc                     # Number of CPU cores
uptime                    # Load averages
mpstat                    # CPU usage breakdown (install: sudo apt install sysstat)

# ── Find CPU-hungry processes ──
# Sort by CPU usage
ps aux --sort=-%cpu | head -15

# Live view
top
# Press '1' to show individual CPU cores
# Press 'P' to sort by CPU
# Press 'H' to show threads

# Better:
htop

# ── Detailed CPU stats ──
mpstat -P ALL 1 5
# Shows per-core usage every 1 second, 5 times
# Look at:
#   %usr    = user-space applications
#   %sys    = kernel/system
#   %iowait = waiting for disk I/O
#   %idle   = doing nothing
```

### Understanding CPU Usage Patterns

```
High %usr (user)
  └── Your applications are using lots of CPU
  └── Find the process: ps aux --sort=-%cpu | head
  └── Common causes: computation-heavy code, infinite loops, regex backtracking

High %sys (system)
  └── The kernel is busy
  └── Common causes: lots of context switches, many network connections, 
      heavy disk I/O, too many processes

High %iowait
  └── CPUs are idle, waiting for slow disk I/O
  └── This is a DISK problem, not a CPU problem!
  └── Common causes: slow disk, too much read/write, swap usage

High %steal (in VMs)
  └── The hypervisor is taking CPU time away from your VM
  └── Common causes: noisy neighbor, oversold host, need bigger VM
```

### CPU Profiling

```bash
# ── strace: See what system calls a process is making ──
# This is INCREDIBLY useful for debugging slow or stuck processes

# Trace a running process
sudo strace -p PID

# Trace with timestamps (see where time is spent)
sudo strace -p PID -T

# Count system calls (summary)
sudo strace -c -p PID
# Let it run for a while, then Ctrl+C to see the summary:
# % time     seconds  calls    syscall
# ------ ----------- -------- --------
#  85.00    2.500000   500000  write    ← Spending 85% of time writing!
#  10.00    0.300000   100000  read
#   5.00    0.150000    50000  stat

# Trace a new command
strace -o output.txt ls -la /tmp

# ── perf: Advanced CPU profiling ──
# (install: sudo apt install linux-tools-common)

# Record CPU usage for 30 seconds
sudo perf record -a -g -- sleep 30

# Show the results
sudo perf report

# Quick summary of what's happening on the system
sudo perf top
```

## 🌐 Network Issues — "Can't Connect"

Network problems are some of the trickiest to debug. Here's a systematic approach.

### The Network Debugging Checklist

```
Layer 1: Physical/Link    → Is the interface up? Is there a link?
Layer 2: IP Address       → Does the server have the right IP?
Layer 3: Routing          → Can it reach the gateway?
Layer 4: DNS              → Can it resolve hostnames?
Layer 5: Port/Service     → Is the service listening on the right port?
Layer 6: Firewall         → Is the firewall blocking connections?
Layer 7: Application      → Is the application working correctly?
```

### Step-by-Step Network Debugging

```bash
# ═══════════════════════════════════════════
# STEP 1: Is the network interface up?
# ═══════════════════════════════════════════
ip link show
# Look for "UP" in the output:
# 2: eth0: <BROADCAST,MULTICAST,UP,LOWER_UP> ← Good!
# 2: eth0: <BROADCAST,MULTICAST>              ← DOWN! Bad!

# If down, bring it up:
sudo ip link set eth0 up

# ═══════════════════════════════════════════
# STEP 2: Does the server have an IP address?
# ═══════════════════════════════════════════
ip addr show
# Look for an inet address:
# inet 192.168.1.100/24 brd 192.168.1.255 scope global eth0

# If no IP, try getting one via DHCP:
sudo dhclient eth0

# ═══════════════════════════════════════════
# STEP 3: Can we reach the gateway?
# ═══════════════════════════════════════════
# Check the routing table
ip route show
# default via 192.168.1.1 dev eth0 ← This is your gateway

# Ping the gateway
ping -c 3 192.168.1.1

# If gateway ping fails → Layer 2 problem (cable, switch, VLAN)
# If gateway ping works → Move to next step

# ═══════════════════════════════════════════
# STEP 4: Can we reach the internet?
# ═══════════════════════════════════════════
# Ping by IP (bypass DNS)
ping -c 3 8.8.8.8

# If this fails → Routing or firewall problem
# If this works → DNS might be the issue

# ═══════════════════════════════════════════
# STEP 5: Is DNS working?
# ═══════════════════════════════════════════
# Check DNS configuration
cat /etc/resolv.conf
# nameserver 8.8.8.8
# nameserver 8.8.4.4

# Test DNS resolution
nslookup google.com
dig google.com
host google.com

# If DNS fails:
# - Check /etc/resolv.conf has valid nameservers
# - Try a different DNS server: dig google.com @8.8.8.8
# - Check if systemd-resolved is running: systemctl status systemd-resolved

# ═══════════════════════════════════════════
# STEP 6: Is the service listening?
# ═══════════════════════════════════════════
# Check what ports are open
sudo ss -tlnp
# -t = TCP, -l = listening, -n = numeric, -p = show process

# Output:
# State  Local Address:Port  Process
# LISTEN 0.0.0.0:80          nginx
# LISTEN 0.0.0.0:22          sshd
# LISTEN 127.0.0.1:5432      postgres  ← Only localhost!

# If service is listening on 127.0.0.1 → Only accessible locally
# If service is listening on 0.0.0.0   → Accessible from anywhere

# ═══════════════════════════════════════════
# STEP 7: Is the firewall blocking?
# ═══════════════════════════════════════════
# Check iptables rules
sudo iptables -L -n -v

# Check nftables
sudo nft list ruleset

# Check firewalld (CentOS/RHEL)
sudo firewall-cmd --list-all

# Check ufw (Ubuntu)
sudo ufw status verbose

# Temporarily disable firewall to test (BE CAREFUL in production!):
# sudo ufw disable
# sudo systemctl stop firewalld

# ═══════════════════════════════════════════
# STEP 8: Test the connection end-to-end
# ═══════════════════════════════════════════
# Can we reach the port?
telnet 192.168.1.100 80
# or
nc -zv 192.168.1.100 80
# Connection succeeded! → Service is reachable

# Test HTTP
curl -v http://192.168.1.100:80
# -v for verbose output showing headers and errors

# Trace the network path
traceroute 192.168.1.100
# or
mtr 192.168.1.100    # (better, live view)
```

### Common Network Issues & Fixes

```bash
# ── Issue: "Connection refused" ──
# Meaning: Nothing is listening on that port
# Fix: Start the service, check the port binding
sudo systemctl start nginx
sudo ss -tlnp | grep 80

# ── Issue: "Connection timed out" ──
# Meaning: Packets aren't getting through (firewall, routing, down host)
# Debug:
traceroute TARGET_IP
sudo iptables -L -n | grep PORT

# ── Issue: "Name or service not known" ──
# Meaning: DNS can't resolve the hostname
# Fix:
cat /etc/resolv.conf                   # Check DNS config
dig example.com @8.8.8.8              # Try different DNS
sudo systemctl restart systemd-resolved # Restart DNS service

# ── Issue: "Network is unreachable" ──
# Meaning: No route to the destination
# Fix:
ip route show                          # Check routing table
sudo ip route add default via GATEWAY  # Add default route

# ── Issue: High latency / packet loss ──
# Debug:
mtr TARGET_IP                          # Live traceroute with stats
ping -c 100 TARGET_IP                  # Send 100 pings, check loss %
```

## ⚙️ Service & Process Issues

### Service Won't Start

```bash
# ── Step 1: Check the status ──
systemctl status myservice.service
# Look for:
#   Active: failed (Result: exit-code)
#   → Check the error message below

# ── Step 2: Check the logs ──
journalctl -u myservice.service -n 50 --no-pager
# or for real-time:
journalctl -u myservice.service -f

# ── Step 3: Try starting manually ──
# Run the ExecStart command directly to see the error
sudo -u serviceuser /path/to/binary --config /etc/myapp/config.yml
# This often gives you much clearer error messages

# ── Step 4: Check the unit file ──
systemctl cat myservice.service
# Look for typos, wrong paths, wrong user

# ── Step 5: Verify the unit file ──
systemd-analyze verify /etc/systemd/system/myservice.service

# ── Step 6: Check dependencies ──
systemctl list-dependencies myservice.service
# Are all dependencies running?

# ── Common Causes: ──
# 1. Binary not found (wrong ExecStart path)
# 2. Permission denied (wrong User/Group)
# 3. Config file error (check app's own logs)
# 4. Port already in use (another service on same port)
# 5. Missing dependency (database not running)
# 6. File/directory permissions
```

### Process is Stuck / Zombie Processes

```bash
# ── Find zombie processes ──
ps aux | grep 'Z'
# Or more specifically:
ps -eo pid,ppid,stat,comm | grep Z

# Zombies are dead processes whose parent hasn't "reaped" them
# They use NO resources but take up a PID slot

# To fix: kill the PARENT process (not the zombie)
# Find the parent PID (PPID)
ps -eo pid,ppid,stat,comm | grep Z
# Then kill the parent:
kill PPID
# If that doesn't work:
kill -9 PPID

# ── Find stuck/uninterruptible processes ──
ps aux | grep 'D'
# 'D' state = uninterruptible sleep (usually waiting for I/O)
# These can't be killed with kill -9!
# Common cause: NFS mount that's unreachable, disk failure

# ── Process won't die with kill ──
# Step 1: Try graceful kill
kill PID               # Sends SIGTERM

# Step 2: Wait a few seconds, then force kill
kill -9 PID            # Sends SIGKILL (can't be caught)

# Step 3: If even kill -9 doesn't work
# The process is in 'D' state (uninterruptible sleep)
# You may need to fix the underlying I/O issue or reboot
```

## 📊 Performance Monitoring Tools

### The Essential Performance Commands

```bash
# ═══════════════════════════════════════════
# vmstat — Virtual Memory Statistics
# ═══════════════════════════════════════════
vmstat 1 10
# Print stats every 1 second, 10 times
#
# procs -----------memory---------- ---swap-- -----io---- -system-- ------cpu-----
#  r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa st
#  2  0   1024 500000 200000 300000    0    0   100   200  500  800 25  5 65  5  0
#
# Key columns:
#   r   = processes running (high = CPU bottleneck)
#   b   = processes blocked on I/O (high = disk bottleneck)
#   si  = swap in (high = memory pressure)
#   so  = swap out (high = memory pressure)
#   wa  = I/O wait (high = slow disk)
#   id  = idle (low = busy system)

# ═══════════════════════════════════════════
# iostat — I/O Statistics
# ═══════════════════════════════════════════
iostat -xz 1 5
# Print extended stats for busy disks, every 1 second, 5 times
#
# Key columns:
#   %util   = disk utilization (>80% = saturated)
#   await   = average I/O wait time in ms (high = slow disk)
#   r/s     = reads per second
#   w/s     = writes per second

# ═══════════════════════════════════════════
# sar — System Activity Reporter
# ═══════════════════════════════════════════
# (install: sudo apt install sysstat)

# CPU usage for the day
sar -u

# Memory usage for the day
sar -r

# Disk I/O for the day
sar -d

# Network usage for the day
sar -n DEV

# Historical data from a specific day
sar -u -f /var/log/sysstat/sa15   # Data from the 15th

# ═══════════════════════════════════════════
# iotop — Who's doing disk I/O?
# ═══════════════════════════════════════════
sudo iotop
# Shows per-process I/O usage in real-time
# Press 'o' to only show processes doing I/O
# Press 'a' for accumulated I/O

# ═══════════════════════════════════════════
# nethogs — Who's using network bandwidth?
# ═══════════════════════════════════════════
sudo nethogs
# Shows per-process network usage in real-time

# ═══════════════════════════════════════════
# nmon — All-in-one monitor
# ═══════════════════════════════════════════
nmon
# Press: c=CPU, m=Memory, d=Disk, n=Network, t=Top processes
```

### The USE Method (Brendan Gregg)

A systematic approach to performance analysis. For each resource, check:

- **U** — Utilization (how busy is it?)
- **S** — Saturation (is work queuing up?)
- **E** — Errors (are there error conditions?)

```bash
# ── CPU ──
# Utilization: mpstat -P ALL 1
# Saturation:  vmstat 1 (check 'r' column > CPU count)
# Errors:      dmesg | grep -i "mce"  (machine check errors)

# ── Memory ──
# Utilization: free -h (check 'available')
# Saturation:  vmstat 1 (check 'si'/'so' for swap activity)
# Errors:      dmesg | grep -i "oom"

# ── Disk ──
# Utilization: iostat -xz 1 (check '%util')
# Saturation:  iostat -xz 1 (check 'avgqu-sz' queue length)
# Errors:      dmesg | grep -i "error"

# ── Network ──
# Utilization: sar -n DEV 1
# Saturation:  ss -s (check for drops), netstat -s | grep -i "drop"
# Errors:      ip -s link show (check errors/drops)
```

## 📄 Log Analysis — Your Best Friend

### Where Are the Logs?

```
/var/log/                      ← Main log directory
├── syslog (or messages)       ← General system log
├── auth.log (or secure)       ← Authentication events (SSH, sudo)
├── kern.log                   ← Kernel messages
├── dmesg                      ← Boot/hardware messages
├── nginx/                     ← Nginx logs
│   ├── access.log
│   └── error.log
├── apache2/ (or httpd/)       ← Apache logs
├── mysql/ (or mariadb/)       ← Database logs
├── apt/                       ← Package manager logs
├── journal/                   ← Systemd journal (binary)
├── cloud-init.log             ← Cloud instance init log
├── cron.log                   ← Cron job logs
└── fail2ban.log               ← Failed login ban logs
```

### Essential Log Commands

```bash
# ── Real-time log monitoring ──
tail -f /var/log/syslog                    # Follow syslog
tail -f /var/log/nginx/error.log           # Follow nginx errors
tail -f /var/log/auth.log                  # Watch for SSH attempts

# Follow multiple log files at once
tail -f /var/log/nginx/error.log /var/log/nginx/access.log

# Or use multitail (install: sudo apt install multitail)
multitail /var/log/nginx/error.log /var/log/syslog

# ── Searching logs ──
# Find errors in syslog
grep -i "error" /var/log/syslog

# Find errors in the last hour (combine with date filtering)
grep "$(date '+%b %d %H')" /var/log/syslog | grep -i error

# Count occurrences of each error type
grep -i "error" /var/log/syslog | awk '{print $5, $6, $7}' | sort | uniq -c | sort -rn

# Find SSH login failures
grep "Failed password" /var/log/auth.log | tail -20

# Count failed login attempts by IP
grep "Failed password" /var/log/auth.log | awk '{print $(NF-3)}' | sort | uniq -c | sort -rn

# ── Systemd journal (journalctl) ──
# Most recent errors across ALL services
journalctl -p err --since "1 hour ago"

# Logs for a specific service
journalctl -u nginx.service --since today

# Kernel messages
journalctl -k

# Boot messages
journalctl -b
```

### Log Analysis One-Liners

```bash
# ── Top IP addresses hitting your web server ──
awk '{print $1}' /var/log/nginx/access.log | sort | uniq -c | sort -rn | head -20

# ── Top requested URLs ──
awk '{print $7}' /var/log/nginx/access.log | sort | uniq -c | sort -rn | head -20

# ── HTTP response codes breakdown ──
awk '{print $9}' /var/log/nginx/access.log | sort | uniq -c | sort -rn

# ── Requests per hour ──
awk '{print $4}' /var/log/nginx/access.log | cut -d: -f1-2 | sort | uniq -c

# ── Find 500 errors ──
awk '$9 == 500' /var/log/nginx/access.log

# ── Find slow requests (>5 seconds) ──
awk '$NF > 5.0' /var/log/nginx/access.log
```

## 🔐 Permission & Access Issues

### Common Permission Errors

```bash
# ── "Permission denied" ──
# Check file permissions
ls -la /path/to/file
# Output: -rw-r----- 1 root root 1234 Jan 15 10:00 file.conf
#         ↑          ↑ ↑
#         perms    owner group

# Check who you're running as
whoami
id

# Fix: Change ownership or permissions
sudo chown myuser:mygroup /path/to/file
sudo chmod 644 /path/to/file    # rw-r--r--

# ── "Operation not permitted" (even as root!) ──
# File might have immutable attribute
lsattr /path/to/file
# ----i--------e-- file.conf   ← 'i' = immutable!

# Remove immutable flag
sudo chattr -i /path/to/file

# ── SELinux denials ──
# Check if SELinux is blocking something
getenforce                      # Enforcing, Permissive, Disabled
sudo ausearch -m AVC --ts recent # Recent SELinux denials
sudo sealert -a /var/log/audit/audit.log  # Human-readable denials

# Temporarily set SELinux to permissive (for testing):
sudo setenforce 0

# Fix SELinux context:
sudo restorecon -Rv /path/to/directory

# ── Capabilities ──
# Some binaries need specific capabilities to work
# (e.g., binding to ports below 1024)
getcap /path/to/binary

# Set a capability
sudo setcap 'cap_net_bind_service=+ep' /path/to/binary
```

### Debugging Permission Issues Systematically

```bash
# ── The Permission Debugging Flowchart ──

# 1. What's the error?
cat /var/log/syslog | grep -i "permission\|denied\|forbidden"

# 2. What user/group is the process running as?
ps aux | grep myservice
# Or check the systemd unit:
systemctl show myservice -p User -p Group

# 3. What are the file permissions?
ls -la /path/to/file
namei -l /path/to/file    # Shows permissions for EVERY directory in the path!
# This is SUPER useful! Example:
# f: /opt/myapp/data/file.txt
# drwxr-xr-x root  root  /
# drwxr-xr-x root  root  opt
# drwx------ root  root  myapp    ← AHA! myapp dir is 700, only root can enter!
# drwxr-xr-x myapp myapp data
# -rw-r--r-- myapp myapp file.txt

# 4. Fix the permissions along the path
sudo chmod 755 /opt/myapp    # Let others read and traverse
```

## 🚨 Boot Issues — System Won't Start

### Common Boot Problems

```bash
# ── System boots to emergency/rescue mode ──
# You'll see: "Welcome to emergency mode!"
# Common causes:
#   - Bad entry in /etc/fstab (can't mount a filesystem)
#   - Corrupted filesystem
#   - Missing /boot files

# Step 1: Check what went wrong
journalctl -xb    # Logs from this boot
systemctl --failed # Which services failed?

# Step 2: If /etc/fstab is the problem
# Edit it and fix/remove the bad entry
vi /etc/fstab
# Then try remounting:
mount -a    # Mount everything in fstab

# ── GRUB bootloader issues ──
# If GRUB is broken, boot from a live USB and:
sudo mount /dev/sda1 /mnt          # Mount root partition
sudo mount --bind /dev /mnt/dev
sudo mount --bind /proc /mnt/proc
sudo mount --bind /sys /mnt/sys
sudo chroot /mnt
grub-install /dev/sda
update-grub
exit
sudo umount -R /mnt
reboot

# ── Kernel panic ──
# Usually means hardware issue, corrupted kernel, or bad driver
# Boot into an older kernel via GRUB menu:
#   1. Reboot
#   2. Hold Shift (BIOS) or Esc (UEFI) to get GRUB menu
#   3. Select "Advanced options"
#   4. Choose a previous kernel version
```

### Recovery Mode Operations

```bash
# ── Booting into Single User / Rescue Mode ──
# From running system:
sudo systemctl rescue        # Rescue mode (minimal services)
sudo systemctl emergency     # Emergency mode (even more minimal)

# From GRUB:
#   1. Edit the boot entry (press 'e')
#   2. Find the line starting with 'linux'
#   3. Add 'single' or 'rescue' at the end
#   4. Press Ctrl+X to boot

# ── Reset root password (if forgotten) ──
#   1. Boot into GRUB
#   2. Edit boot entry, add: init=/bin/bash
#   3. Press Ctrl+X
#   4. Mount root as read-write:
mount -o remount,rw /
#   5. Change password:
passwd root
#   6. Reboot:
exec /sbin/init
```

## 🧪 Real-World Troubleshooting Scenarios

### Scenario 1: "The Website Is Down!"

```bash
# Step 1: Is the server reachable?
ping webserver.example.com

# Step 2: Is the web service running?
systemctl status nginx
# → "Active: inactive (dead)"  ← FOUND IT!

# Step 3: Why did it stop?
journalctl -u nginx -n 50
# → "[emerg] bind() to 0.0.0.0:80 failed (98: Address already in use)"
# Another process is using port 80!

# Step 4: Find what's using port 80
sudo ss -tlnp | grep :80
# → LISTEN  0  128  0.0.0.0:80  users:(("apache2",pid=1234))
# Apache is running on port 80!

# Step 5: Fix it
sudo systemctl stop apache2
sudo systemctl disable apache2
sudo systemctl start nginx
```

### Scenario 2: "The Server Is Really Slow!"

```bash
# Step 1: Get the big picture
uptime
# → load average: 45.00, 40.00, 35.00   ← WAY too high for 4 cores!

# Step 2: What's causing the load?
top
# Press 'P' to sort by CPU
# → PID USER  %CPU  COMMAND
# → 1234 www  99.0  php-fpm     ← PHP is eating all the CPU!

# Step 3: Which request is causing it?
strace -p 1234
# → Lots of "read" system calls on a huge file

# Step 4: Check the PHP error log
tail -100 /var/log/php-fpm/error.log
# → "Allowed memory size exhausted" for /var/www/report.php

# Step 5: Fix it
# The report page is processing too much data
# → Optimize the code or add pagination
# → Temporarily: kill the stuck process
kill 1234
```

### Scenario 3: "Disk Full, Can't Deploy!"

```bash
# Step 1: Confirm disk is full
df -h
# → /dev/sda1  50G  50G  0  100%  /

# Step 2: Find what's using space
sudo du -sh /* | sort -rh | head
# → 40G /var

sudo du -sh /var/* | sort -rh | head
# → 35G /var/log

sudo du -sh /var/log/* | sort -rh | head
# → 30G /var/log/app/debug.log    ← ONE LOG FILE!

# Step 3: Emergency fix (free space NOW)
# Truncate the log file (don't delete — the app might be writing to it)
sudo truncate -s 0 /var/log/app/debug.log

# Step 4: Prevent it from happening again
# Add log rotation
sudo cat > /etc/logrotate.d/myapp << 'EOF'
/var/log/app/*.log {
    daily
    rotate 7
    compress
    missingok
    notifempty
    create 0640 www-data www-data
    postrotate
        systemctl reload myapp
    endscript
}
EOF

# Step 5: Set up monitoring
# Add a disk space alert (cron or systemd timer)
```

### Scenario 4: "SSH Connection Refused!"

```bash
# From another machine or console:

# Step 1: Is SSH running?
systemctl status sshd
# → "Active: active (running)"  ← It's running...

# Step 2: Is it listening on the right port?
sudo ss -tlnp | grep ssh
# → LISTEN 0 128 *:2222  ← It's listening on port 2222, not 22!

# Step 3: Check the SSH config
grep "Port" /etc/ssh/sshd_config
# → Port 2222

# Step 4: Fix it
sudo sed -i 's/Port 2222/Port 22/' /etc/ssh/sshd_config
sudo systemctl restart sshd

# Or connect on the right port:
ssh -p 2222 user@server
```
