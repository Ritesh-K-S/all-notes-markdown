> **One Line Answer:** Linux organizes **everything** as files inside a single tree structure starting from `/` (root), and every directory has a specific purpose — knowing this hierarchy is like knowing the map of a city before you start driving.

---

## 🤔 Why Do You Need to Know This?

As a DevOps engineer, you'll constantly be:

- Finding where config files live (`/etc/nginx/nginx.conf`)
- Checking logs when things break (`/var/log/syslog`)
- Managing application deployments (`/opt/myapp/`)
- Troubleshooting disk space issues (`/var` is full again!)
- Writing scripts that read/write files in the right places
- Setting up Docker volumes, mount points, and backups

If you don't know the file system hierarchy, you'll waste hours looking for files that are sitting right where they're supposed to be.

## 🌳 The Big Picture — Everything is a Tree

In Windows, you have different drives: `C:\`, `D:\`, `E:\`

In Linux, there are **no drive letters**. Everything starts from a single point called **`/`** (pronounced "root"). Everything is inside this tree.

```
/                          ← Root of EVERYTHING
├── bin/                   ← Essential command binaries
├── boot/                  ← Boot loader files (kernel, grub)
├── dev/                   ← Device files (hardware)
├── etc/                   ← System configuration files
├── home/                  ← User home directories
│   ├── ritesh/            ← Your stuff
│   └── john/              ← John's stuff
├── lib/                   ← Essential shared libraries
├── lib64/                 ← 64-bit libraries
├── media/                 ← Removable media mount points (USB, CD)
├── mnt/                   ← Temporary mount points
├── opt/                   ← Optional/third-party software
├── proc/                  ← Virtual filesystem (process info)
├── root/                  ← Root user's home directory
├── run/                   ← Runtime data (PIDs, sockets)
├── sbin/                  ← System administration binaries
├── srv/                   ← Service data (web, FTP)
├── sys/                   ← Virtual filesystem (kernel/hardware info)
├── tmp/                   ← Temporary files (cleared on reboot)
├── usr/                   ← User programs and data
│   ├── bin/               ← User command binaries
│   ├── lib/               ← User libraries
│   ├── local/             ← Locally installed software
│   │   ├── bin/
│   │   ├── lib/
│   │   └── etc/
│   ├── sbin/              ← Non-essential system binaries
│   └── share/             ← Shared data (docs, man pages)
└── var/                   ← Variable data (logs, caches, databases)
    ├── log/               ← Log files
    ├── cache/             ← Application caches
    ├── lib/               ← Variable state data
    ├── mail/              ← User mailboxes
    ├── spool/             ← Print/mail queues
    └── tmp/               ← Temporary files (persist across reboots)
```

This structure follows the **FHS (Filesystem Hierarchy Standard)** — a set of rules that most Linux distros follow so that everything is in a predictable place.

## 🗂️ "Everything is a File" — The Linux Philosophy

This is one of the most important concepts in Linux:

> **In Linux, EVERYTHING is a file.** Regular files, directories, hardware devices, running processes, network sockets — ALL represented as files.

```
Your keyboard?        → /dev/input/event0       (a file!)
Your hard disk?       → /dev/sda                (a file!)
Your RAM info?        → /proc/meminfo           (a file!)
Your CPU info?        → /proc/cpuinfo           (a file!)
Network interfaces?   → /sys/class/net/eth0     (a file!)
A running process?    → /proc/1234/             (a directory of files!)
```

This means you can use the same tools (`cat`, `echo`, `read`) to interact with almost anything in the system. Super powerful!

## 📁 Now Let's Deep Dive Into Each Important Directory

---

### `/` — The Root Directory

```
/
```

- The **top-level** directory. Everything lives under here.
- Only the **root user** (administrator) can write directly to `/`.
- Do NOT create random files here. Each subdirectory has a purpose.

```bash
# List the root directory
ls /

# Output:
# bin  boot  dev  etc  home  lib  lib64  media  mnt  opt  proc  root  run  sbin  srv  sys  tmp  usr  var
```

**Analogy:** `/` is like the front door of a building. Everything is inside, organized into rooms (directories).

---

### `/etc` — System Configuration Files ⚙️

```
/etc    (pronounced "et-see" or "etcetera")
```

This is where **all system-wide configuration files** live. If you need to configure any service on Linux, you'll find its config file here.

**This is the directory you'll visit the MOST as a DevOps engineer.**

```
/etc/
├── nginx/
│   └── nginx.conf              ← Nginx web server config
├── ssh/
│   └── sshd_config             ← SSH server configuration
├── docker/
│   └── daemon.json             ← Docker daemon config
├── systemd/
│   └── system/                 ← Systemd service files
├── crontab                     ← System cron jobs
├── fstab                       ← Filesystem mount table
├── hosts                       ← Local DNS (hostname → IP mapping)
├── hostname                    ← This machine's hostname
├── passwd                      ← User account information
├── shadow                      ← Encrypted passwords (restricted access)
├── group                       ← Group information
├── sudoers                     ← Sudo (admin) permissions
├── resolv.conf                 ← DNS resolver configuration
├── environment                 ← System-wide environment variables
├── profile                     ← Login shell settings
├── bashrc                      ← Bash shell settings (system-wide)
└── os-release                  ← OS identification info
```

**Practical examples — things you'll do constantly:**

```bash
# 1. Check what Linux distro you're running
cat /etc/os-release

# 2. Edit the SSH config (e.g., disable root login)
sudo nano /etc/ssh/sshd_config
# Find: PermitRootLogin yes
# Change to: PermitRootLogin no
# Save and restart SSH:
sudo systemctl restart sshd

# 3. Add a hostname mapping (like a local DNS entry)
sudo nano /etc/hosts
# Add a line:
# 192.168.1.100  myserver.local

# 4. Check what filesystems are mounted at boot
cat /etc/fstab

# 5. Check all user accounts on the system
cat /etc/passwd

# Example line from /etc/passwd:
# ritesh:x:1000:1000:Ritesh Singh:/home/ritesh:/bin/bash
# ↑       ↑      ↑     ↑            ↑              ↑
# user  passwd  UID   GID       home dir         shell

# 6. Check all groups
cat /etc/group

# 7. View DNS resolver settings
cat /etc/resolv.conf
# nameserver 8.8.8.8
# nameserver 8.8.4.4

# 8. View the hostname
cat /etc/hostname

# 9. View system-wide environment variables
cat /etc/environment
```

**⚠️ Golden Rule:** ALWAYS back up config files before editing them!

```bash
# Before editing any config file:
sudo cp /etc/ssh/sshd_config /etc/ssh/sshd_config.backup

# If something goes wrong, restore:
sudo cp /etc/ssh/sshd_config.backup /etc/ssh/sshd_config
```

### `/home` — User Home Directories 🏠

```
/home
```

Every regular user gets a personal directory under `/home`.

```
/home/
├── ritesh/           ← Your personal directory
│   ├── .bashrc       ← Your personal bash settings
│   ├── .ssh/         ← Your SSH keys
│   │   ├── id_rsa            ← Private SSH key
│   │   ├── id_rsa.pub        ← Public SSH key
│   │   └── authorized_keys   ← Keys allowed to SSH in
│   ├── .profile      ← Login settings
│   ├── .bash_history ← Command history
│   ├── Documents/
│   ├── Downloads/
│   └── projects/
├── john/
└── alice/
```

**Key facts about `/home`:**

| Fact | Detail |
|------|--------|
| **Each user gets one** | `/home/username` |
| **Users can only modify their own** | `ritesh` can't write to `/home/john/` |
| **The `~` shortcut** | `~` always refers to YOUR home directory |
| **Dot files** | Files starting with `.` are hidden (config files) |

```bash
# Go to your home directory (all three do the same thing)
cd ~
cd /home/ritesh
cd $HOME

# Print your home directory path
echo $HOME
# /home/ritesh

# List all files INCLUDING hidden ones
ls -la ~

# The tilde (~) is a shortcut:
# ~ = /home/your_username
# ~john = /home/john
```

**Important dot files (hidden config files):**

```bash
~/.bashrc           # Bash settings (aliases, prompt, functions)
~/.profile           # Login shell settings
~/.bash_history      # Your command history
~/.ssh/              # SSH keys and config
~/.gitconfig         # Git configuration
~/.vimrc             # Vim editor settings
~/.config/           # App configurations (newer standard)
```

**💡 DevOps Tip:** When you SSH into a server, you land in the home directory of the user you logged in as. That's why `~` and `cd` (with no arguments) take you "home."

### `/var` — Variable Data (Logs, Caches, Databases) 📊

```
/var    (stands for "variable")
```

This directory stores data that **changes frequently** during normal system operation. Logs grow, caches fill up, databases change — all of that goes here.

**For DevOps, `/var/log` is your best friend** — when something breaks, the answer is almost always in the logs.

```
/var/
├── log/                    ← 🔥 LOG FILES (check here when things break!)
│   ├── syslog              ← Main system log (Ubuntu/Debian)
│   ├── messages            ← Main system log (RHEL/CentOS)
│   ├── auth.log            ← Authentication logs (who logged in, sudo usage)
│   ├── kern.log            ← Kernel messages
│   ├── dmesg               ← Boot and hardware messages
│   ├── dpkg.log            ← Package installation history (Debian/Ubuntu)
│   ├── yum.log             ← Package installation history (RHEL/CentOS)
│   ├── nginx/
│   │   ├── access.log      ← Nginx access logs
│   │   └── error.log       ← Nginx error logs
│   ├── apache2/
│   │   ├── access.log      ← Apache access logs
│   │   └── error.log       ← Apache error logs
│   └── journal/            ← Systemd journal logs
├── cache/                  ← Application caches
│   └── apt/                ← Downloaded .deb packages (Ubuntu)
├── lib/                    ← Variable state data
│   ├── docker/             ← Docker images, containers, volumes
│   ├── mysql/              ← MySQL database files
│   └── postgresql/         ← PostgreSQL database files
├── mail/                   ← User mailboxes
├── spool/                  ← Print queues, mail queues
│   └── cron/               ← User crontab files
├── tmp/                    ← Temporary files (survive reboots, unlike /tmp)
└── www/                    ← Web server content (sometimes)
    └── html/               ← Default website root
        └── index.html
```

**Practical examples — the logs you'll check constantly:**

```bash
# 1. View the main system log (real-time, follow mode)
sudo tail -f /var/log/syslog          # Ubuntu/Debian
sudo tail -f /var/log/messages        # RHEL/CentOS

# 2. Check who logged into the system
sudo cat /var/log/auth.log            # Ubuntu/Debian
sudo cat /var/log/secure              # RHEL/CentOS

# 3. Check kernel messages (hardware issues, driver problems)
dmesg | tail -20
# or
sudo cat /var/log/kern.log

# 4. Check nginx errors
sudo tail -f /var/log/nginx/error.log

# 5. Check recent package installations
cat /var/log/dpkg.log | tail -20      # Ubuntu/Debian
cat /var/log/yum.log | tail -20       # RHEL/CentOS

# 6. Use journalctl for systemd-managed services
sudo journalctl -u nginx              # Logs for nginx service
sudo journalctl -u docker             # Logs for Docker service
sudo journalctl -f                    # Follow all logs in real-time
sudo journalctl --since "1 hour ago"  # Logs from the last hour

# 7. Check disk usage of /var/log (logs can fill up your disk!)
sudo du -sh /var/log/
sudo du -sh /var/log/* | sort -rh | head -10
```

**⚠️ Common DevOps Problem:** `/var/log` filling up the entire disk!

```bash
# Check if disk is full
df -h
# If /var is at 100%, logs might be the culprit

# Find the biggest log files
sudo du -sh /var/log/* | sort -rh | head -10

# Quick fix: Clear a log file (keeps the file, empties content)
sudo truncate -s 0 /var/log/syslog

# Better fix: Set up log rotation (already configured by default)
cat /etc/logrotate.conf
ls /etc/logrotate.d/        # Per-service rotation configs
```

**Where Docker stores everything:**
```bash
# Docker data directory (images, containers, volumes)
ls /var/lib/docker/

# /var/lib/docker/
# ├── containers/    ← Container data
# ├── image/         ← Image layers
# ├── volumes/       ← Docker volumes
# ├── overlay2/      ← Image/container filesystem layers
# └── network/       ← Network configuration

# Check Docker disk usage
docker system df
```

### `/tmp` — Temporary Files 🗑️

```
/tmp
```

A place for **temporary files** that any user or program can create. Files here are **automatically deleted on reboot** (on most distros).

```bash
# Create a temporary file
echo "hello" > /tmp/mytest.txt

# Anyone can write to /tmp
ls -la /tmp/

# Check the sticky bit (security feature)
ls -ld /tmp/
# drwxrwxrwt 10 root root 4096 Jun  1 12:00 /tmp/
#          ↑
#          t = sticky bit
#          Anyone can create files, but ONLY the owner can delete their own files
```

**Key facts about `/tmp`:**

| Fact | Detail |
|------|--------|
| **World-writable** | Any user can create files here |
| **Cleared on reboot** | Files are deleted when the system restarts |
| **Sticky bit enabled** | User A can't delete User B's files (even though both can write) |
| **Used by programs** | Apps create temp files here during operation |
| **NOT for important data** | Never store anything important in `/tmp` |

**`/tmp` vs `/var/tmp`:**
```
/tmp/       → Cleared on reboot
/var/tmp/   → NOT cleared on reboot (persists across restarts)
```

**💡 DevOps Use:** You'll often download scripts or files to `/tmp` for quick one-time use:
```bash
# Download a script temporarily
curl -o /tmp/install.sh https://example.com/install.sh
chmod +x /tmp/install.sh
/tmp/install.sh
# It'll be gone after reboot — no cleanup needed!
```

### `/opt` — Optional / Third-Party Software 📦

```
/opt    (stands for "optional")
```

This is where **third-party software** and **add-on applications** are installed — stuff that doesn't come with the OS and isn't managed by the package manager.

```
/opt/
├── google/
│   └── chrome/              ← Google Chrome
├── containerd/              ← Container runtime
├── cni/                     ← Container Network Interface plugins
├── mycompany-app/           ← Your company's custom application
│   ├── bin/
│   ├── lib/
│   ├── config/
│   └── logs/
└── prometheus/              ← Prometheus monitoring
    ├── prometheus
    ├── prometheus.yml
    └── data/
```

**When to use `/opt`:**

| Install In `/opt` | Don't Use `/opt` |
|-------------------|------------------|
| Third-party software not in repos | Software from `apt`/`yum` repos |
| Custom company applications | Standard Linux tools |
| Software you download as tarballs | Python packages (`pip`) |
| Self-contained applications | System libraries |

**Example: Installing a tool manually to `/opt`:**
```bash
# Download Prometheus
cd /tmp
wget https://github.com/prometheus/prometheus/releases/download/v2.45.0/prometheus-2.45.0.linux-amd64.tar.gz

# Extract to /opt
sudo tar xzf prometheus-2.45.0.linux-amd64.tar.gz -C /opt/

# Rename for cleanliness
sudo mv /opt/prometheus-2.45.0.linux-amd64 /opt/prometheus

# Now prometheus lives in /opt/prometheus/
ls /opt/prometheus/
# prometheus  promtool  prometheus.yml  consoles/  console_libraries/

# Run it
/opt/prometheus/prometheus --config.file=/opt/prometheus/prometheus.yml
```

**💡 DevOps Tip:** Many DevOps tools (Jenkins, Grafana, Prometheus) when installed manually end up in `/opt/`. When installed via package managers, they spread across `/usr/bin`, `/etc/`, `/var/lib/`.

### `/usr` — User Programs & Shared Data 📚

```
/usr    (stands for "Unix System Resources" — NOT "user"!)
```

This is one of the **largest directories**. It contains the majority of user-facing programs, libraries, and documentation.

```
/usr/
├── bin/                ← User command binaries (most commands live here)
│   ├── python3
│   ├── git
│   ├── docker
│   ├── curl
│   ├── vim
│   └── ...hundreds more
├── sbin/               ← System admin binaries (need sudo)
│   ├── iptables
│   ├── useradd
│   └── ...
├── lib/                ← Libraries for /usr/bin and /usr/sbin
├── lib64/              ← 64-bit libraries
├── local/              ← Locally compiled/installed software
│   ├── bin/            ← Local binaries (manually installed)
│   ├── lib/            ← Local libraries
│   └── etc/            ← Local configs
├── share/              ← Architecture-independent data
│   ├── man/            ← Manual pages
│   ├── doc/            ← Documentation
│   └── zoneinfo/       ← Timezone data
└── include/            ← C/C++ header files
```

**The `/bin` vs `/usr/bin` confusion:**

Historically:
```
/bin/       → Essential commands needed to boot and fix the system
/usr/bin/   → Everything else

But modern Linux (Ubuntu 20.04+, Fedora, etc.) has MERGED them:
/bin is now a symlink → /usr/bin
/sbin is now a symlink → /usr/sbin
```

```bash
# Check if they're the same
ls -la /bin
# lrwxrwxrwx 1 root root 7 /bin -> usr/bin

# So these are the same thing:
/bin/ls
/usr/bin/ls
```

**`/usr/local/` — Where YOUR custom stuff goes:**
```bash
# When you compile and install software from source code:
./configure
make
sudo make install
# It installs to /usr/local/bin/, /usr/local/lib/, etc.

# This keeps your custom software separate from OS-managed software
which python3          # /usr/bin/python3 (from apt)
which my-custom-tool   # /usr/local/bin/my-custom-tool (manually installed)
```

### `/proc` — Process & Kernel Information (Virtual) 🧬

```
/proc    (stands for "process")
```

This is a **virtual filesystem** — it doesn't exist on disk. It's generated by the kernel in real-time and gives you a window into the running system.

```bash
# CPU information
cat /proc/cpuinfo

# Memory information
cat /proc/meminfo

# Kernel version
cat /proc/version

# System uptime (in seconds)
cat /proc/uptime

# Currently mounted filesystems
cat /proc/mounts

# Each running process has a directory:
ls /proc/1/          # Process with PID 1 (init/systemd)
ls /proc/1234/       # Process with PID 1234

# Inside a process directory:
# /proc/1234/cmdline   ← What command started this process
# /proc/1234/status    ← Process status (memory, state)
# /proc/1234/environ   ← Environment variables
# /proc/1234/fd/       ← Open file descriptors
```

**Practical DevOps uses:**
```bash
# Quick way to check total RAM
grep MemTotal /proc/meminfo
# MemTotal:       16384000 kB

# Check number of CPU cores
grep -c ^processor /proc/cpuinfo
# 4

# Check current kernel parameters
cat /proc/sys/net/ipv4/ip_forward
# 0 = disabled, 1 = enabled (needed for Docker/K8s networking)

# Enable IP forwarding (temporarily)
echo 1 | sudo tee /proc/sys/net/ipv4/ip_forward
```

### `/dev` — Device Files 🔌

```
/dev    (stands for "devices")
```

Hardware devices are represented as files here.

```bash
# Common device files:
/dev/sda        ← First hard disk
/dev/sda1       ← First partition of first disk
/dev/sdb        ← Second hard disk
/dev/nvme0n1    ← NVMe SSD
/dev/null       ← "Black hole" — discards anything written to it
/dev/zero       ← Infinite stream of zero bytes
/dev/random     ← Random data generator
/dev/tty        ← Current terminal

# The famous /dev/null — discard output
command_that_produces_noise 2>/dev/null    # Throw away errors
command > /dev/null 2>&1                    # Throw away ALL output
```

### `/boot` — Boot Loader Files 🚀

```
/boot
```

Contains everything needed to **boot the system** — the Linux kernel, initial RAM disk, and GRUB bootloader.

```bash
ls /boot/
# config-5.15.0-91-generic        ← Kernel configuration
# initrd.img-5.15.0-91-generic    ← Initial RAM disk
# vmlinuz-5.15.0-91-generic       ← THE Linux kernel (compressed)
# grub/                            ← GRUB bootloader configs
```

**⚠️ Don't mess with `/boot` unless you know exactly what you're doing.** Deleting files here can make your system unbootable.

## 🔍 Useful Commands for Navigating the File System

```bash
# Print current directory
pwd
# /home/ritesh

# Change directory
cd /etc/nginx          # Go to specific directory
cd ~                   # Go home
cd ..                  # Go up one level
cd -                   # Go back to previous directory

# List files
ls                     # Basic list
ls -la                 # Detailed list with hidden files
ls -lh                 # Human-readable file sizes
ls -R                  # Recursive list (all subdirectories)

# Show directory tree (install: sudo apt install tree)
tree /etc/nginx
# /etc/nginx/
# ├── nginx.conf
# ├── sites-available/
# │   └── default
# └── sites-enabled/
#     └── default -> ../sites-available/default

tree -L 2 /            # Show root, 2 levels deep

# Find files
find / -name "nginx.conf"                  # Find by name
find /var/log -name "*.log" -size +100M    # Find log files bigger than 100MB
find /etc -type f -name "*.conf"           # Find all .conf files in /etc

# Find which package owns a file
dpkg -S /usr/bin/curl        # Debian/Ubuntu
rpm -qf /usr/bin/curl        # RHEL/CentOS

# Find where a command lives
which docker                  # /usr/bin/docker
whereis nginx                 # Shows binary, source, and man page locations
type ls                       # ls is aliased to 'ls --color=auto'

# Check disk usage by directory
du -sh /var/log               # Total size of /var/log
du -sh /*                     # Size of each top-level directory
du -sh /var/log/* | sort -rh | head -10   # Top 10 biggest items in /var/log

# Check filesystem disk space
df -h                         # Disk space for all mounted filesystems
df -h /var                    # Disk space for /var specifically
```

## 🆚 Linux vs Windows File System Comparison

If you're coming from Windows, this comparison will help:

| Concept | Windows | Linux |
|---------|---------|-------|
| **Root** | `C:\` | `/` |
| **Path separator** | `\` (backslash) | `/` (forward slash) |
| **User files** | `C:\Users\username\` | `/home/username/` |
| **Program files** | `C:\Program Files\` | `/usr/bin/`, `/opt/` |
| **System config** | Registry + `C:\Windows\` | `/etc/` |
| **Temp files** | `C:\Windows\Temp\` | `/tmp/` |
| **Logs** | Event Viewer | `/var/log/` |
| **Drive letters** | `C:\`, `D:\`, `E:\` | No drive letters. Everything under `/` |
| **Hidden files** | File attribute | Filename starts with `.` |
| **Case sensitive** | ❌ No (`File.txt` = `file.txt`) | ✅ Yes (`File.txt` ≠ `file.txt`) |
| **File extensions** | Required (`.exe`, `.txt`) | Optional (Linux uses permissions, not extensions) |

**⚠️ Important:** Linux file paths are **case-sensitive**!
```bash
# These are THREE DIFFERENT files in Linux:
/home/ritesh/readme.txt
/home/ritesh/Readme.txt
/home/ritesh/README.txt
```

## 📝 Key Takeaways

```
✅ Everything in Linux starts from / (root) — no drive letters
✅ "Everything is a file" — hardware, processes, network = all files
✅ /etc = Configuration files (you'll live here as a DevOps engineer)
✅ /home = User personal directories (~/ is a shortcut to yours)
✅ /var = Variable data (logs in /var/log, databases in /var/lib)
✅ /var/log = First place to check when things break
✅ /tmp = Temporary files (cleared on reboot)
✅ /opt = Third-party/optional software
✅ /usr/bin = Most user commands; /usr/local/bin = manually installed
✅ /proc = Virtual filesystem with live system/process info
✅ /dev = Device files (hardware represented as files)
✅ Linux paths are case-sensitive! /Home ≠ /home
✅ Always back up config files before editing: cp file file.backup
✅ Know the "Where Do I Find...?" cheat sheet — it saves hours
```
