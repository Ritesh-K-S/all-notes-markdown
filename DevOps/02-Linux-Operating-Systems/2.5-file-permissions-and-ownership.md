# 2.5 — File Permissions & Ownership — chmod, chown, umask, ACLs

> **One Line Answer:** Every file and directory in Linux has **permissions** (who can read, write, or execute it) and **ownership** (who owns it). Understanding this is **critical for security** — get it wrong, and you either lock people out or leave doors wide open for hackers.

---

## 🎯 Why File Permissions Matter

Imagine a shared office building:

- **Your desk drawers** → Only YOU can open them (private files)
- **The meeting room** → Your team can use it (group access)
- **The lobby** → Anyone can walk in (public access)

Linux file permissions work **exactly** like this. Every file has rules about:

1. **Who** can access it (owner, group, everyone else)
2. **What** they can do (read, write, execute)

```
Why This Matters in DevOps:

  ❌ Web server config readable by everyone → Security risk!
  ❌ Script not executable → "Permission denied" errors
  ❌ SSH key with wrong permissions → SSH refuses to connect
  ❌ Log file not writable → Application crashes
  ✅ Correct permissions → Everything works securely
```

---

## 📖 Understanding the Permission System

### The Three Permission Types

| Symbol | Permission | For Files | For Directories |
|--------|-----------|-----------|-----------------|
| `r` | **Read** | Can view file contents | Can list directory contents (`ls`) |
| `w` | **Write** | Can modify file contents | Can create/delete files inside |
| `x` | **Execute** | Can run the file as a program | Can enter the directory (`cd`) |
| `-` | **No permission** | Can't do that action | Can't do that action |

### The Three User Categories

Every file has permissions set for **three categories** of users:

| Category | Symbol | Who? |
|----------|--------|------|
| **Owner (User)** | `u` | The user who owns the file |
| **Group** | `g` | Users who belong to the file's group |
| **Others** | `o` | Everyone else on the system |

### Reading Permission Strings

When you run `ls -l`, you see something like this:

```bash
$ ls -l
-rwxr-xr-- 1 ritesh devops 4096 May 27 10:00 deploy.sh
```

Let's break down that permission string: `-rwxr-xr--`

```
-  rwx  r-x  r--
│  │    │    │
│  │    │    └── Others: r-- (read only)
│  │    └─────── Group:  r-x (read + execute)
│  └──────────── Owner:  rwx (read + write + execute)
└─────────────── Type:   - (regular file)

File Types:
  -  = Regular file
  d  = Directory
  l  = Symbolic link
  b  = Block device
  c  = Character device
```

### Let's Read Some Real Examples:

```bash
drwxr-xr-x  →  Directory | Owner: rwx | Group: r-x | Others: r-x
# Owner can do everything. Group and others can read and enter, but can't modify.

-rw-r--r--  →  File | Owner: rw- | Group: r-- | Others: r--
# Owner can read and write. Everyone else can only read.

-rwx------  →  File | Owner: rwx | Group: --- | Others: ---
# Only the owner can do anything. Nobody else has any access.

-rw-rw----  →  File | Owner: rw- | Group: rw- | Others: ---
# Owner and group can read/write. Others have no access.

drwx------  →  Directory | Owner: rwx | Group: --- | Others: ---
# Only the owner can enter and see this directory.
```

## 🔢 Numeric (Octal) Permission System

Instead of typing `rwx`, Linux also lets you use **numbers**. This is actually what most DevOps engineers use because it's faster.

### The Number System:

Each permission has a number value:

```
r (read)    = 4
w (write)   = 2
x (execute) = 1
- (nothing) = 0
```

You **add them up** for each category:

```
rwx = 4 + 2 + 1 = 7  (full access)
rw- = 4 + 2 + 0 = 6  (read + write)
r-x = 4 + 0 + 1 = 5  (read + execute)
r-- = 4 + 0 + 0 = 4  (read only)
-wx = 0 + 2 + 1 = 3  (write + execute)
-w- = 0 + 2 + 0 = 2  (write only)
--x = 0 + 0 + 1 = 1  (execute only)
--- = 0 + 0 + 0 = 0  (no access)
```

### Three Digits = Owner, Group, Others:

```
755 = rwxr-xr-x  → Owner: full | Group: read+execute | Others: read+execute
644 = rw-r--r--  → Owner: read+write | Group: read | Others: read
700 = rwx------  → Owner: full | Nobody else
600 = rw-------  → Owner: read+write | Nobody else
777 = rwxrwxrwx  → Everyone can do everything (⚠️ DANGEROUS!)
000 = ---------  → Nobody can do anything (even owner needs sudo)
```

### The Most Common Permission Numbers You'll Use:

```
Files:
  644 → Default for regular files (owner rw, everyone else read)
  755 → Executable scripts (owner rwx, everyone else rx)
  600 → Private files like SSH keys (only owner rw)
  400 → Read-only private files (only owner read)

Directories:
  755 → Default for directories (owner full, everyone else can read/enter)
  700 → Private directories (only owner can enter)
  775 → Shared team directories (owner and group full, others read/enter)
```

## 🔧 `chmod` — Change File Permissions

`chmod` = **Ch**ange **Mod**e. This is how you change permissions on files and directories.

### Method 1: Numeric Mode (Most Common)

```bash
# Set permissions using numbers
$ chmod 755 deploy.sh
# Owner: rwx (7), Group: r-x (5), Others: r-x (5)

$ chmod 644 config.yml
# Owner: rw- (6), Group: r-- (4), Others: r-- (4)

$ chmod 600 ~/.ssh/id_rsa
# Owner: rw- (6), Group: --- (0), Others: --- (0)
# SSH keys MUST be 600 or SSH will refuse to use them!

$ chmod 700 ~/private-scripts/
# Only you can enter and use this directory
```

### Method 2: Symbolic Mode (More Readable)

```bash
# Syntax: chmod [who][action][permission] file

# who:    u = owner, g = group, o = others, a = all
# action: + = add, - = remove, = = set exactly
# perm:   r = read, w = write, x = execute

# Add execute permission for the owner
$ chmod u+x script.sh

# Remove write permission for others
$ chmod o-w config.yml

# Add read permission for group
$ chmod g+r document.txt

# Set exact permissions for all
$ chmod a=r file.txt
# Everyone gets read-only

# Combine multiple changes
$ chmod u+x,g+r,o-w file.txt
# Owner: add execute
# Group: add read
# Others: remove write

# Make a script executable by everyone
$ chmod a+x script.sh
# or equivalently:
$ chmod +x script.sh
# When you don't specify who, it applies to all
```

### Recursive Permissions:

```bash
# Apply permissions to a directory AND everything inside it
$ chmod -R 755 /var/www/html/
# -R = Recursive

# ⚠️ Be careful with recursive chmod!
# You usually want different permissions for files vs directories:

# Set directories to 755 and files to 644 (common pattern)
$ find /var/www/html -type d -exec chmod 755 {} \;
$ find /var/www/html -type f -exec chmod 644 {} \;
# This is the CORRECT way to set web server permissions
```

### Real-World DevOps Permission Scenarios:

```bash
# Scenario 1: SSH key has wrong permissions
$ ssh -i mykey.pem server
@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
@         WARNING: UNPROTECTED PRIVATE KEY FILE!          @
@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
Permissions 0644 for 'mykey.pem' are too open.

# Fix:
$ chmod 600 mykey.pem    # Now it works!

# Scenario 2: Script won't execute
$ ./deploy.sh
bash: ./deploy.sh: Permission denied

# Fix:
$ chmod +x deploy.sh     # Now it works!

# Scenario 3: Nginx can't read website files
# Fix:
$ chmod -R 755 /var/www/html/
$ chmod -R 644 /var/www/html/*.html

# Scenario 4: Log file can't be written
# Fix:
$ chmod 666 /var/log/myapp.log
# Or better — change ownership (see chown below)
```

## 👤 `chown` — Change File Ownership

`chown` = **Ch**ange **Own**ership. Every file has an **owner** (a user) and a **group**. `chown` lets you change both.

### Basic Usage:

```bash
# Change the owner of a file
$ chown ritesh file.txt
# Now ritesh owns the file

# Change the owner AND the group
$ chown ritesh:devops file.txt
# Owner: ritesh, Group: devops

# Change only the group (two ways)
$ chown :devops file.txt     # Using chown with :group
$ chgrp devops file.txt       # Using chgrp command

# Recursive (change ownership of everything inside a directory)
$ chown -R ritesh:devops /var/www/html/
# Now ritesh:devops owns the directory and ALL its contents
```

### Why Ownership Matters — Real Stories:

```bash
# Story 1: Nginx says "403 Forbidden"
# Problem: Nginx runs as the "www-data" user, but files are owned by "root"
$ ls -la /var/www/html/
-rw-r--r-- 1 root root 1024 index.html    # ← root owns it!

# Nginx (running as www-data) can't read root's files!
# Fix:
$ chown -R www-data:www-data /var/www/html/
# Now Nginx can read the files ✅

# Story 2: Application can't write to its log directory
# Problem: App runs as user "appuser" but /var/log/myapp/ is owned by root
$ ls -ld /var/log/myapp/
drwxr-xr-x 2 root root 4096 /var/log/myapp/

# Fix:
$ chown appuser:appuser /var/log/myapp/
# Now the app can write logs ✅

# Story 3: Docker builds fail because of file ownership
# Docker often needs specific ownership
$ chown -R 1000:1000 /app/data/
```

### Checking Ownership:

```bash
# See who owns a file
$ ls -la myfile.txt
-rw-r--r-- 1 ritesh devops 4096 May 27 10:00 myfile.txt
#              ^^^^^^ ^^^^^^
#              Owner  Group

# See who owns a directory and its contents
$ ls -la /var/www/html/

# Check ownership of all files recursively
$ find /var/www -exec ls -ld {} \;
```

## 🎭 `umask` — Default Permission Mask

When you create a **new file or directory**, what permissions does it get? That's controlled by `umask`.

### How umask Works:

Think of umask as a **filter** that **removes** permissions from the defaults.

```
Default permissions for NEW files:       666 (rw-rw-rw-)
Default permissions for NEW directories: 777 (rwxrwxrwx)

umask SUBTRACTS from these defaults.
```

```
Example with umask 022 (the most common default):

For a new FILE:
  Default:  666 (rw-rw-rw-)
  umask:    022
  Result:   644 (rw-r--r--)  ← Owner rw, group r, others r

For a new DIRECTORY:
  Default:  777 (rwxrwxrwx)
  umask:    022
  Result:   755 (rwxr-xr-x)  ← Owner rwx, group rx, others rx
```

### Working with umask:

```bash
# Check your current umask
$ umask
0022

# The leading 0 is for "special permissions" (we'll cover those later)
# Just focus on the last 3 digits: 022

# Set a new umask
$ umask 077
# Now new files will be 600 (rw-------) — only owner can access
# New directories will be 700 (rwx------) — only owner can access

# Test it:
$ umask 077
$ touch secret.txt
$ ls -l secret.txt
-rw------- 1 ritesh ritesh 0 May 27 10:00 secret.txt
# Only owner can read/write ✅

$ mkdir private-dir
$ ls -ld private-dir
drwx------ 2 ritesh ritesh 4096 May 27 10:00 private-dir
# Only owner can access ✅
```

### Common umask Values:

```
umask 022  → Files: 644, Dirs: 755  (Default on most systems)
umask 027  → Files: 640, Dirs: 750  (Group can read, others nothing)
umask 077  → Files: 600, Dirs: 700  (Only owner — most restrictive)
umask 002  → Files: 664, Dirs: 775  (Group gets write access too)
```

### Making umask Permanent:

```bash
# umask set in terminal only lasts for that session
# To make it permanent, add it to your shell profile:

$ echo "umask 027" >> ~/.bashrc
$ source ~/.bashrc

# For ALL users system-wide:
$ echo "umask 027" >> /etc/profile
```

### Real-World DevOps Usage:

```bash
# On a production server, you might want restrictive defaults
$ umask 077
# Now any config files created accidentally won't be readable by others

# In a shared development environment
$ umask 002
# Team members in the same group can read/write each other's files
```

## 🌟 Special Permissions — SUID, SGID, Sticky Bit

These are **advanced permissions** that go beyond the basic rwx. You'll encounter them in DevOps, so let's understand them.

### 1. SUID (Set User ID) — Run as File Owner

When SUID is set on an executable, **anyone** who runs it will execute it with the **owner's permissions**, not their own.

```bash
# Classic example: the passwd command
$ ls -l /usr/bin/passwd
-rwsr-xr-x 1 root root 68208 /usr/bin/passwd
#  ^
#  s ← That's SUID! (notice the 's' instead of 'x' in owner permissions)

# Why? Every user needs to change their own password.
# But the password file (/etc/shadow) is owned by root.
# SUID lets the passwd command run as root — so it can modify /etc/shadow.
# Without SUID, normal users couldn't change their passwords!
```

```bash
# Setting SUID:
$ chmod u+s myprogram         # Symbolic way
$ chmod 4755 myprogram        # Numeric way (4 = SUID)

# Removing SUID:
$ chmod u-s myprogram
```

> **⚠️ Security Warning:** SUID on the wrong file is a **major security risk**. If you set SUID on a shell (`/bin/bash`), any user could become root!

### 2. SGID (Set Group ID) — Run as File's Group / Inherit Group

**On a file:** Similar to SUID but for the group. The file runs with the group's permissions.

**On a directory:** Much more useful! New files created inside will **inherit the directory's group** instead of the creator's primary group.

```bash
# Example on a directory:
$ mkdir /shared-project
$ chown :devops /shared-project
$ chmod g+s /shared-project      # Set SGID on directory
# or: chmod 2775 /shared-project  (2 = SGID)

$ ls -ld /shared-project
drwxrwsr-x 2 root devops 4096 /shared-project
#       ^
#       s ← SGID is set

# Now when ANYONE creates a file in /shared-project:
$ touch /shared-project/newfile.txt
$ ls -l /shared-project/newfile.txt
-rw-r--r-- 1 ritesh devops 0 newfile.txt
#                   ^^^^^^
#                   Group is "devops" (inherited from directory!)
#                   Without SGID, it would be the user's primary group
```

This is **super useful** for shared team directories where everyone needs to work on the same files.

### 3. Sticky Bit — Only Owner Can Delete

When set on a directory, only the **file owner** (or root) can delete files inside it — even if others have write access.

```bash
# Classic example: /tmp directory
$ ls -ld /tmp
drwxrwxrwt 10 root root 4096 /tmp
#         ^
#         t ← Sticky bit! (notice the 't' at the end)

# Everyone can write to /tmp (create files)
# But you can only delete YOUR OWN files — not someone else's
# Without the sticky bit, anyone could delete anyone's temp files!
```

```bash
# Setting sticky bit:
$ chmod +t /shared-folder         # Symbolic
$ chmod 1777 /shared-folder       # Numeric (1 = sticky bit)

# Removing sticky bit:
$ chmod -t /shared-folder
```

### Special Permissions Summary:

```
Numeric Prefix:
  4 = SUID     (4xxx)
  2 = SGID     (2xxx)
  1 = Sticky   (1xxx)

Can be combined:
  6 = SUID + SGID (6xxx)
  3 = SGID + Sticky (3xxx)
  7 = All three (7xxx)

How to spot them in ls -l:
  SUID:    s in owner's execute position     -rwsr-xr-x
  SGID:    s in group's execute position     -rwxr-sr-x
  Sticky:  t in others' execute position     drwxrwxrwt

  If the underlying execute is not set:
  SUID:    S (capital S)                     -rwSr-xr-x
  SGID:    S (capital S)                     -rwxr-Sr-x
  Sticky:  T (capital T)                     drwxrwxrwT
```

## 📋 ACLs — Access Control Lists (Fine-Grained Permissions)

Regular permissions only let you set rules for **owner**, **group**, and **others**. But what if you want to give a **specific user** or a **specific group** special access? That's where ACLs come in.

### The Problem ACLs Solve:

```
Scenario: You have a file owned by ritesh:devops with permissions 640.
  - ritesh (owner) can read/write ✅
  - devops group can read ✅
  - others have no access ✅

BUT: You need to give user "alice" read access too.
  - Alice is NOT in the devops group
  - You don't want to give ALL others read access
  - You can't add a second group owner in basic permissions

Solution: Use an ACL to give alice specific access!
```

### Working with ACLs:

```bash
# Check if a file has ACLs
$ ls -l file.txt
-rw-r-----+ 1 ritesh devops 4096 file.txt
#         ^
#         + sign means this file has ACLs!

# View ACLs
$ getfacl file.txt
# file: file.txt
# owner: ritesh
# group: devops
user::rw-
user:alice:r--        ← Alice has read access (ACL entry)
group::r--
mask::r--
other::---

# Set an ACL — Give user "alice" read access
$ setfacl -m u:alice:r file.txt
# -m = modify
# u:alice:r = user alice gets read permission

# Give group "qa" read+write access
$ setfacl -m g:qa:rw file.txt

# Remove an ACL entry
$ setfacl -x u:alice file.txt
# Removes alice's ACL entry

# Remove ALL ACLs from a file
$ setfacl -b file.txt
# -b = remove all ACLs

# Set default ACLs for a directory (new files inherit these)
$ setfacl -d -m u:alice:rw /shared-project/
# -d = default (applied to new files created inside)
# Now every new file in /shared-project/ gives alice rw access

# Apply ACLs recursively
$ setfacl -R -m g:devops:rx /var/www/html/
# -R = recursive
```

### ACL Examples in DevOps:

```bash
# Give the CI/CD user access to deployment directories
$ setfacl -R -m u:jenkins:rwx /var/www/html/
$ setfacl -R -d -m u:jenkins:rwx /var/www/html/

# Give the monitoring user read access to log files
$ setfacl -R -m u:prometheus:r /var/log/myapp/

# Give multiple teams access to a shared config directory
$ setfacl -m g:devops:rwx /etc/myapp/
$ setfacl -m g:qa:rx /etc/myapp/
$ setfacl -m g:security:r /etc/myapp/
```

### The Mask:

```bash
# The mask sets the MAXIMUM permissions for ACL entries
$ getfacl file.txt
user::rw-
user:alice:rwx       #effective:r--    ← Alice asked for rwx but mask limits it!
group::r--
mask::r--            ← The mask says "max is read only"
other::---

# Set the mask
$ setfacl -m m::rw file.txt
# Now the maximum for ACL entries is read+write
```

## 🔒 Security Best Practices for Permissions

### The Golden Rules:

```bash
# 1. SSH keys MUST have strict permissions
$ chmod 700 ~/.ssh
$ chmod 600 ~/.ssh/id_rsa           # Private key: owner read/write ONLY
$ chmod 644 ~/.ssh/id_rsa.pub       # Public key: everyone can read
$ chmod 600 ~/.ssh/authorized_keys  # Auth keys: owner read/write ONLY
$ chmod 644 ~/.ssh/config           # SSH config: everyone can read

# 2. Web server files
$ chown -R www-data:www-data /var/www/html/
$ find /var/www/html -type d -exec chmod 755 {} \;
$ find /var/www/html -type f -exec chmod 644 {} \;

# 3. Config files with passwords/secrets
$ chmod 600 /etc/myapp/database.yml   # Only root/owner can read
$ chmod 600 .env                       # Environment files with secrets

# 4. Scripts that need to execute
$ chmod 755 /usr/local/bin/deploy.sh   # Everyone can execute

# 5. Sensitive directories
$ chmod 700 /root                      # Only root can access root's home
$ chmod 750 /var/log/myapp/            # Owner full, group read, others nothing
```

### Finding Permission Problems:

```bash
# Find files with dangerous 777 permissions
$ find / -type f -perm 777 2>/dev/null

# Find SUID files (potential security risk)
$ find / -type f -perm -4000 2>/dev/null

# Find SGID files
$ find / -type f -perm -2000 2>/dev/null

# Find world-writable files (anyone can modify — risky!)
$ find / -type f -perm -o+w 2>/dev/null

# Find files with no owner (orphaned files — suspicious)
$ find / -nouser -o -nogroup 2>/dev/null
```

## 📝 Quick Reference Cheat Sheet

### Common Permission Numbers:

| Number | Permissions | Typical Use |
|--------|------------|-------------|
| `777` | rwxrwxrwx | ⚠️ AVOID! Everyone can do everything |
| `755` | rwxr-xr-x | Executables, public directories |
| `750` | rwxr-x--- | Semi-private directories |
| `700` | rwx------ | Private directories, `.ssh/` |
| `644` | rw-r--r-- | Normal files, public configs |
| `640` | rw-r----- | Config files readable by group |
| `600` | rw------- | SSH keys, secrets, `.env` files |
| `400` | r-------- | Read-only secrets |

### Command Summary:

| Command | What It Does | Example |
|---------|-------------|---------|
| `chmod 755 file` | Set permissions (numeric) | `chmod 755 deploy.sh` |
| `chmod u+x file` | Add permission (symbolic) | `chmod u+x script.sh` |
| `chmod -R 644 dir/` | Recursive permissions | `chmod -R 644 /var/www/` |
| `chown user:group file` | Change ownership | `chown www-data:www-data index.html` |
| `chown -R user:group dir/` | Recursive ownership | `chown -R nginx:nginx /var/www/` |
| `chgrp group file` | Change group only | `chgrp devops config.yml` |
| `umask 022` | Set default permission mask | `umask 077` |
| `getfacl file` | View ACLs | `getfacl secret.conf` |
| `setfacl -m u:user:rwx file` | Set ACL | `setfacl -m u:alice:r file` |

## 🧠 Key Takeaways

1. **Every file has owner, group, and others** — each with read, write, execute permissions
2. **Numeric mode is faster** — memorize 755, 644, 600, 700
3. **`chmod`** changes permissions, **`chown`** changes ownership — know the difference
4. **SSH keys MUST be 600** or SSH will refuse to work
5. **Never use 777** — it's a security disaster
6. **umask** controls default permissions for new files
7. **ACLs** give you fine-grained control when basic permissions aren't enough
8. **Always check permissions** when something says "Permission denied"
