> **One Line Answer:** Linux is a **multi-user system** — multiple people (or services) can use the same machine at the same time. Managing **who** these users are, **what groups** they belong to, and **what they're allowed to do** is a fundamental DevOps skill.

---

## 🎯 Why Users & Groups Matter

Think of a Linux server like an **apartment building**:

- Each **user** has their own apartment (home directory) with their own key (password)
- **Groups** are like clubs — "Residents Club", "Gym Members", "Pool Access"
- Belonging to a group gives you access to shared resources
- The **building manager (root)** has the master key to everything

```
Why This Matters in DevOps:

  🔧 Every service runs as a specific user (nginx, mysql, jenkins)
  🔐 Security depends on users having the RIGHT level of access
  🚫 The principle of "least privilege" — give only what's needed
  👥 Teams need shared access to certain files and directories
  🤖 Automation (CI/CD, scripts) runs as specific service accounts
```

## 📁 The Important Files — Where User Data Lives

Linux stores all user and group information in a few key files. Let's understand each one.

### 1. `/etc/passwd` — The User Database

This file contains **basic information** about every user on the system. Despite the name, it does NOT contain actual passwords anymore (those are in `/etc/shadow`).

```bash
$ cat /etc/passwd
root:x:0:0:root:/root:/bin/bash
daemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin
www-data:x:33:33:www-data:/var/www:/usr/sbin/nologin
ritesh:x:1000:1000:Ritesh Singh:/home/ritesh:/bin/bash
jenkins:x:997:997:Jenkins CI:/var/lib/jenkins:/bin/bash
```

Each line has **7 fields** separated by colons (`:`):

```
ritesh : x : 1000 : 1000 : Ritesh Singh : /home/ritesh : /bin/bash
  │      │    │      │          │              │              │
  │      │    │      │          │              │              └── 7. Login shell
  │      │    │      │          │              └───────────────── 6. Home directory
  │      │    │      │          └──────────────────────────────── 5. Comment/Full name (GECOS)
  │      │    │      └─────────────────────────────────────────── 4. Primary Group ID (GID)
  │      │    └────────────────────────────────────────────────── 3. User ID (UID)
  │      └─────────────────────────────────────────────────────── 2. Password placeholder (x = stored in /etc/shadow)
  └────────────────────────────────────────────────────────────── 1. Username
```

#### Key Things to Know:

```
UID 0        = root (the superuser, the god of the system)
UID 1-999    = System/service users (nginx, mysql, jenkins, etc.)
UID 1000+    = Regular human users (you, me, your teammates)

Shell /bin/bash          = Can log in and use the terminal
Shell /usr/sbin/nologin  = CANNOT log in (used for service accounts)
Shell /bin/false         = CANNOT log in (another way to disable login)
```

```bash
# Quick ways to read /etc/passwd

# Show just usernames
$ cut -d':' -f1 /etc/passwd

# Show users with a login shell (actual humans or services that can log in)
$ grep "/bin/bash" /etc/passwd

# Show service accounts (can't log in)
$ grep "nologin\|/bin/false" /etc/passwd

# Count total users
$ wc -l /etc/passwd
```

### 2. `/etc/shadow` — The Password File (Secret!)

This is where the **actual password hashes** are stored. Only **root** can read it.

```bash
$ sudo cat /etc/shadow
root:$6$abc123...:19500:0:99999:7:::
ritesh:$6$xyz789...:19500:0:99999:7:::
jenkins:!:19500:0:99999:7:::
```

Each line has **9 fields**:

```
ritesh : $6$xyz789... : 19500 : 0 : 99999 : 7 : : :
  │          │            │     │     │      │  │ │ │
  │          │            │     │     │      │  │ │ └── 9. Reserved
  │          │            │     │     │      │  │ └──── 8. Expiration date
  │          │            │     │     │      │  └────── 7. Inactivity period
  │          │            │     │     │      └───────── 6. Warning days before expiry
  │          │            │     │     └──────────────── 5. Maximum password age (days)
  │          │            │     └────────────────────── 4. Minimum password age (days)
  │          │            └──────────────────────────── 3. Last password change (days since Jan 1, 1970)
  │          └───────────────────────────────────────── 2. Password hash
  └──────────────────────────────────────────────────── 1. Username
```

#### Password Hash Field Special Values:

```
$6$...     = SHA-512 hash (most common, most secure)
$5$...     = SHA-256 hash
$1$...     = MD5 hash (old, insecure — don't use!)
!          = Account is LOCKED (can't log in with password)
*          = Account has never had a password (system account)
!! or empty = No password set (can't log in with password)
```

> **⚠️ Security Note:** Never share or display `/etc/shadow` contents. The hashes can be brute-forced by attackers.

### 3. `/etc/group` — The Group Database

```bash
$ cat /etc/group
root:x:0:
sudo:x:27:ritesh
www-data:x:33:
devops:x:1001:ritesh,alice,bob
docker:x:998:ritesh
```

Each line has **4 fields**:

```
devops : x : 1001 : ritesh,alice,bob
  │      │    │           │
  │      │    │           └── 4. Group members (comma-separated)
  │      │    └────────────── 3. Group ID (GID)
  │      └─────────────────── 2. Group password (x = not used)
  └────────────────────────── 1. Group name
```

```bash
# Show all groups
$ cat /etc/group

# See what groups a user belongs to
$ groups ritesh
ritesh : ritesh sudo docker devops

# See your own groups
$ groups
ritesh sudo docker devops

# Alternative: id command (more detailed)
$ id ritesh
uid=1000(ritesh) gid=1000(ritesh) groups=1000(ritesh),27(sudo),998(docker),1001(devops)
```

### 4. `/etc/gshadow` — Group Password File (Rarely Used)

Similar to `/etc/shadow` but for groups. Rarely used in practice — just know it exists.

## 👤 Creating Users — `useradd` and `adduser`

### `useradd` — The Low-Level Command

`useradd` creates a user but does **minimal setup** by default. You need to specify everything yourself.

```bash
# Create a basic user
$ sudo useradd john
# This creates the user BUT:
# - No password set
# - Home directory might not be created (depends on distro)
# - Default shell might be /bin/sh
```

```bash
# Create a user with ALL the options you typically want
$ sudo useradd -m -s /bin/bash -c "John Doe" -G sudo,docker john

# Let's break down each flag:
# -m              = Create home directory (/home/john)
# -s /bin/bash    = Set login shell to bash
# -c "John Doe"  = Set the comment/full name
# -G sudo,docker = Add to additional groups (sudo and docker)

# Now set a password
$ sudo passwd john
New password: ********
Retype password: ********
passwd: password updated successfully
```

#### All Useful `useradd` Flags:

```bash
# -m         → Create home directory
# -M         → Do NOT create home directory
# -s /shell  → Set login shell
# -c "name"  → Set comment/description
# -G groups  → Add to supplementary groups (comma-separated)
# -g group   → Set primary group
# -u 1500    → Set specific UID
# -d /path   → Set custom home directory path
# -e date    → Set account expiration date (YYYY-MM-DD)
# -r         → Create a SYSTEM user (UID below 1000, no home dir)

# Examples:

# Create a developer user
$ sudo useradd -m -s /bin/bash -c "Alice Developer" -G devops,docker alice

# Create a system/service user (for running an application)
$ sudo useradd -r -s /usr/sbin/nologin -c "My App Service" myapp
# -r = system user (low UID, no home directory)
# -s /usr/sbin/nologin = can't log in interactively
# This is how you create users for services like nginx, mysql, etc.

# Create a user with a specific UID
$ sudo useradd -m -u 1500 -s /bin/bash specialuser

# Create a temporary user that expires
$ sudo useradd -m -s /bin/bash -e 2026-12-31 contractor
# Account automatically disabled after Dec 31, 2026
```

On Debian-based systems (Ubuntu, etc.), `adduser` is an **interactive, friendlier** version that asks you questions.

```bash
    Full Name []: John Doe
    Room Number []:
    Work Phone []:
    Home Phone []:
    Other []:

```

### `useradd` vs `adduser` — Which to Use?

```
useradd:
  ✅ Works on ALL Linux distributions
  ✅ Better for scripts and automation (non-interactive)
  ❌ Does minimal setup — you need to specify everything
  📌 Use for: Automation, Dockerfiles, CI/CD, shell scripts

adduser:
  ✅ Interactive and user-friendly
  ✅ Automatically creates home dir, sets shell, prompts for password
  ❌ Only available on Debian/Ubuntu (not on RHEL/CentOS by default)
  📌 Use for: Manually adding human users on Debian/Ubuntu
```

## ✏️ Modifying Users — `usermod`

`usermod` changes properties of an **existing** user.

```bash
# Change a user's login shell
$ sudo usermod -s /bin/zsh ritesh
# Now ritesh's shell is zsh instead of bash

# Add a user to additional groups (KEEP existing groups)
$ sudo usermod -aG docker ritesh
# -a = APPEND (crucial! Without -a, it REPLACES all groups!)
# -G = supplementary groups

# ⚠️ DANGEROUS — WITHOUT -a:
$ sudo usermod -G docker ritesh
# This REMOVES ritesh from ALL other groups and ONLY keeps docker!
# ALWAYS use -aG, never just -G

# Add a user to multiple groups at once
$ sudo usermod -aG sudo,docker,devops ritesh

# Change username
$ sudo usermod -l newname oldname
# Changes login name from oldname to newname

# Change home directory
$ sudo usermod -d /new/home/path -m ritesh
# -d = new home directory
# -m = MOVE contents from old home to new home

# Lock a user account (disable login)
$ sudo usermod -L ritesh
# Adds ! to the password hash — can't log in with password

# Unlock a user account
$ sudo usermod -U ritesh

# Set account expiration date
$ sudo usermod -e 2026-12-31 contractor
# Account will be disabled after this date

# Change user's comment/full name
$ sudo usermod -c "Ritesh Singh - Senior DevOps" ritesh
```

### Real-World DevOps Scenarios:

```bash
# New team member needs Docker access
$ sudo usermod -aG docker newdev

# Developer needs sudo access
$ sudo usermod -aG sudo newdev

# Contractor's access needs to be extended
$ sudo usermod -e 2027-06-30 contractor

# Employee left — lock their account immediately
$ sudo usermod -L former-employee
# Later, you can fully delete the account

# Service needs to run as a different user
$ sudo usermod -s /usr/sbin/nologin serviceuser
```

## 🗑️ Deleting Users — `userdel`

```bash
# Delete a user (keeps their home directory)
$ sudo userdel john
# User is deleted, but /home/john still exists

# Delete a user AND their home directory
$ sudo userdel -r john
# -r = remove home directory and mail spool

# Force delete (even if user is logged in)
$ sudo userdel -f john
# ⚠️ Use with caution!
```

### Before Deleting — Best Practices:

```bash
# 1. Check if user has any running processes
$ ps -u john
# Kill them first if needed

# 2. Check if user owns important files
$ find / -user john 2>/dev/null

# 3. Backup their home directory first
$ tar -czf /backup/john-home-backup.tar.gz /home/john

# 4. Lock the account first (safer than immediate deletion)
$ sudo usermod -L john

# 5. Then delete when you're sure
$ sudo userdel -r john
```

## 👥 Managing Groups

### Creating Groups — `groupadd`

```bash
# Create a new group
$ sudo groupadd devops

# Create a group with a specific GID
$ sudo groupadd -g 2000 engineering

# Create a system group (GID below 1000)
$ sudo groupadd -r appgroup
```

### Modifying Groups — `groupmod`

```bash
# Rename a group
$ sudo groupmod -n newname oldname

# Change group's GID
$ sudo groupmod -g 2001 devops
```

### Deleting Groups — `groupdel`

```bash
# Delete a group
$ sudo groupdel devops
# ⚠️ Can't delete a group that is someone's PRIMARY group
# You must change their primary group first
```

### Managing Group Membership:

```bash
# Add a user to a group
$ sudo usermod -aG devops ritesh
# or
$ sudo gpasswd -a ritesh devops

# Remove a user from a group
$ sudo gpasswd -d ritesh devops

# List members of a group
$ getent group devops
devops:x:1001:ritesh,alice,bob

# See all groups a user belongs to
$ groups ritesh
$ id ritesh
```

## 🔐 Password Management

### Setting and Changing Passwords:

```bash
# Set/change password for another user (as root)
$ sudo passwd john
New password: ********
Retype new password: ********

# Change your OWN password
$ passwd
Current password: ********
New password: ********
Retype new password: ********

# Force a user to change password at next login
$ sudo passwd -e john
# -e = expire the password immediately
# Next time john logs in, he'll be forced to set a new password
```

### Password Policies:

```bash
# View password aging info for a user
$ sudo chage -l ritesh
Last password change                                    : May 27, 2026
Password expires                                        : never
Password inactive                                       : never
Account expires                                         : never
Minimum number of days between password change          : 0
Maximum number of days between password change          : 99999
Number of days of warning before password expires       : 7

# Set password to expire every 90 days
$ sudo chage -M 90 ritesh

# Set minimum days between password changes (prevent rapid changes)
$ sudo chage -m 7 ritesh

# Force password change on next login
$ sudo chage -d 0 ritesh

# Set account expiration date
$ sudo chage -E 2026-12-31 contractor

# Set all policies at once
$ sudo chage -M 90 -m 7 -W 14 ritesh
# -M 90  = Max age: 90 days
# -m 7   = Min age: 7 days
# -W 14  = Warning: 14 days before expiry
```

## 🛡️ The `sudo` System — Running Commands as Root

### What is sudo?

`sudo` = **S**uper **U**ser **Do**. It lets a permitted user run commands as **root** (or another user) without actually logging in as root.

```bash
# Run a single command as root
$ sudo apt update

# Switch to root shell
$ sudo -i
# or
$ sudo su -
# Now you ARE root. Be careful!

# Run a command as a different user
$ sudo -u postgres psql
# Runs psql as the postgres user

# Check what sudo commands you're allowed to run
$ sudo -l
User ritesh may run the following commands on server:
    (ALL : ALL) ALL
```

### The sudoers File — `/etc/sudoers`

This file controls WHO can use sudo and WHAT they can do. **NEVER edit it directly** — always use `visudo`.

```bash
# Edit sudoers safely
$ sudo visudo
# visudo checks for syntax errors before saving — this prevents you from
# locking yourself out of sudo!
```

#### sudoers File Format:

```bash
# Format: WHO WHERE=(AS_WHOM) WHAT

# Full access (can do anything as anyone)
ritesh    ALL=(ALL:ALL) ALL
# ritesh can run ALL commands as ALL users on ALL hosts

# Let's break this down:
# ritesh       → username
# ALL          → on any host (machine)
# (ALL:ALL)    → as any user:any group
# ALL          → can run any command
```

```bash
# Allow a user to run specific commands only
john    ALL=(ALL) /usr/bin/systemctl restart nginx, /usr/bin/systemctl status nginx
# john can ONLY restart/check nginx — nothing else with sudo

# Allow a group to use sudo (note the % prefix for groups)
%devops    ALL=(ALL) ALL
# Anyone in the "devops" group has full sudo access

# Allow sudo WITHOUT password (useful for automation, risky for humans)
jenkins    ALL=(ALL) NOPASSWD: ALL
# jenkins can run ANY command without entering a password
# ⚠️ Use this ONLY for service accounts, not for human users

# Allow specific commands without password
ritesh     ALL=(ALL) NOPASSWD: /usr/bin/docker, /usr/bin/systemctl
# ritesh can run docker and systemctl without password
# But other sudo commands still require password
```

### Best Practices for sudo:

```bash
# 1. Use sudoers.d directory instead of editing main sudoers file
$ sudo visudo -f /etc/sudoers.d/devops-team
# Add rules here. Each file in /etc/sudoers.d/ is included automatically.
# This is cleaner and easier to manage.

# Example /etc/sudoers.d/devops-team:
%devops ALL=(ALL) ALL

# 2. Don't give NOPASSWD to human users
# 3. Limit commands when possible (principle of least privilege)
# 4. Use groups instead of individual users
# 5. Always use visudo — never edit sudoers with a regular editor
```

## 🔄 Switching Users — `su`

```bash
# Switch to another user
$ su john
Password: ********
# Now you're john. Type 'exit' to go back.

# Switch to another user WITH their full environment
$ su - john
# The dash (-) loads john's full environment (PATH, home dir, etc.)
# WITHOUT the dash, you keep your current environment (often causes issues)
# ALWAYS use "su -" (with the dash)

# Switch to root
$ su -
Password: ********
# Now you're root

# Run a single command as another user
$ su - john -c "whoami"
john
```

### `su` vs `sudo`:

```
su:
  - Switches to another user's account entirely
  - Requires the TARGET user's password
  - You "become" that user
  - Less auditable (hard to track who did what)

sudo:
  - Runs a single command as another user (usually root)
  - Requires YOUR OWN password
  - You stay as yourself, just elevated for one command
  - Fully auditable (logged in /var/log/auth.log)
  - Can be fine-tuned (limit which commands are allowed)

Winner: sudo (almost always preferred over su in modern DevOps)
```

## 🤖 Service Users — Users for Applications

In DevOps, applications and services run as **dedicated users** — not as root, not as a human user.

### Why Service Users?

```
Security Principle: Least Privilege

If nginx runs as root and gets hacked:
  → Attacker has ROOT ACCESS to everything! 💀

If nginx runs as www-data (service user with limited access):
  → Attacker can only access what www-data can access
  → Damage is contained! 🛡️
```

### Creating Service Users:

```bash
# Create a service user for your application
$ sudo useradd -r -s /usr/sbin/nologin -d /var/lib/myapp -c "My App Service" myapp

# Breakdown:
# -r                    = System user (UID < 1000)
# -s /usr/sbin/nologin  = Cannot log in interactively
# -d /var/lib/myapp     = Home directory (for app data)
# -c "My App Service"   = Description
# myapp                 = Username

# Create the home directory and set ownership
$ sudo mkdir -p /var/lib/myapp
$ sudo chown myapp:myapp /var/lib/myapp

# Verify
$ id myapp
uid=997(myapp) gid=997(myapp) groups=997(myapp)

$ grep myapp /etc/passwd
myapp:x:997:997:My App Service:/var/lib/myapp:/usr/sbin/nologin
```

### Common Service Users You'll See:

```bash
$ grep -E "nginx|www-data|mysql|postgres|jenkins|docker|prometheus|grafana" /etc/passwd

www-data:x:33:33:www-data:/var/www:/usr/sbin/nologin           # Nginx/Apache
mysql:x:27:27:MySQL Server:/var/lib/mysql:/bin/false            # MySQL
postgres:x:26:26:PostgreSQL Server:/var/lib/pgsql:/bin/bash     # PostgreSQL
jenkins:x:997:997:Jenkins CI:/var/lib/jenkins:/bin/bash         # Jenkins
```

## 🏗️ Practical Scenarios — Putting It All Together

### Scenario 1: Setting Up a New Developer's Account

```bash
# Step 1: Create the user
$ sudo useradd -m -s /bin/bash -c "Alice Johnson - Developer" alice

# Step 2: Set a temporary password
$ sudo passwd alice

# Step 3: Force password change on first login
$ sudo passwd -e alice

# Step 4: Add to necessary groups
$ sudo usermod -aG sudo,docker,devops alice

# Step 5: Verify
$ id alice
uid=1001(alice) gid=1001(alice) groups=1001(alice),27(sudo),998(docker),1001(devops)

$ ls -la /home/alice/
```

### Scenario 2: Creating a Shared Project Directory

```bash
# Step 1: Create the devops group (if not exists)
$ sudo groupadd devops

# Step 2: Add team members to the group
$ sudo usermod -aG devops ritesh
$ sudo usermod -aG devops alice
$ sudo usermod -aG devops bob

# Step 3: Create the shared directory
$ sudo mkdir -p /projects/shared

# Step 4: Set ownership
$ sudo chown root:devops /projects/shared

# Step 5: Set permissions — group can read/write, SGID for inheritance
$ sudo chmod 2775 /projects/shared
# 2 = SGID (new files inherit the devops group)
# 775 = Owner rwx, Group rwx, Others r-x

# Step 6: Verify
$ ls -ld /projects/shared
drwxrwsr-x 2 root devops 4096 /projects/shared

# Now any file created by ritesh, alice, or bob in this directory
# will automatically belong to the devops group ✅
```

### Scenario 3: Setting Up a Jenkins Service Account

```bash
# Step 1: Create the service user
$ sudo useradd -r -m -d /var/lib/jenkins -s /bin/bash -c "Jenkins CI" jenkins

# Step 2: Add jenkins to docker group (so it can build images)
$ sudo usermod -aG docker jenkins

# Step 3: Set up SSH key for Jenkins (passwordless auth)
$ sudo -u jenkins ssh-keygen -t ed25519 -f /var/lib/jenkins/.ssh/id_ed25519 -N ""

# Step 4: Set correct permissions
$ sudo chmod 700 /var/lib/jenkins/.ssh
$ sudo chmod 600 /var/lib/jenkins/.ssh/id_ed25519

# Step 5: Give jenkins limited sudo access (no password)
$ sudo visudo -f /etc/sudoers.d/jenkins
# Add this line:
# jenkins ALL=(ALL) NOPASSWD: /usr/bin/systemctl restart myapp, /usr/bin/docker

# Step 6: Verify
$ id jenkins
$ sudo -u jenkins docker ps
```

### Scenario 4: Employee Offboarding (Someone Leaves)

```bash
# Step 1: Lock the account IMMEDIATELY
$ sudo usermod -L alice
# Alice can't log in anymore

# Step 2: Kill any active sessions
$ sudo pkill -u alice
# Terminates all processes running as alice

# Step 3: Check what alice owns
$ sudo find / -user alice 2>/dev/null > /tmp/alice-files.txt

# Step 4: Backup their home directory
$ sudo tar -czf /backup/alice-home-$(date +%Y%m%d).tar.gz /home/alice

# Step 5: Remove from all groups
$ sudo gpasswd -d alice devops
$ sudo gpasswd -d alice docker
$ sudo gpasswd -d alice sudo

# Step 6: Delete the account (after management approval and backup confirmation)
$ sudo userdel -r alice

# Step 7: Verify they're gone
$ id alice
id: 'alice': no such user ✅
```

```bash
ritesh:x:1000:1000:Ritesh Singh:/home/ritesh:/bin/bash

# Search for a group
$ getent group devops
devops:x:1001:ritesh,alice,bob
```

## 📁 The `/etc/skel` Directory — Template for New Users

When you create a new user with `-m` (create home directory), Linux copies files from `/etc/skel` into their new home directory.

```bash
$ ls -la /etc/skel
-rw-r--r-- 1 root root  220 .bash_logout
-rw-r--r-- 1 root root 3771 .bashrc
-rw-r--r-- 1 root root  807 .profile
```

You can customize this to set up a default environment for all new users:

```bash
# Add a welcome message for all new users
$ sudo sh -c 'echo "Welcome to the server! Read /docs/onboarding.md" > /etc/skel/README'

# Add a default .vimrc
$ sudo cp /path/to/standard-vimrc /etc/skel/.vimrc

# Add a default Git config
$ sudo cp /path/to/gitconfig /etc/skel/.gitconfig

# Now every new user created with -m will get these files automatically!
```

| Command | What It Does | Example |

| `useradd -m -s /bin/bash user` | Create user | `sudo useradd -m -s /bin/bash alice` |

## 🧠 Key Takeaways

1. **Every user has a UID, primary group, home directory, and shell** — stored in `/etc/passwd`
2. **Passwords are stored as hashes** in `/etc/shadow` — never in plain text
3. **Always use `useradd -m -s /bin/bash`** to create users with a home dir and proper shell
4. **ALWAYS use `-aG` with usermod** — forgetting the `-a` removes all existing groups!
5. **Service accounts should use `/usr/sbin/nologin`** shell — they should never log in interactively
6. **Use `sudo` instead of `su`** — it's more secure and auditable
7. **Never edit `/etc/sudoers` directly** — always use `visudo`
8. **Lock accounts before deleting them** — gives you time to backup and verify
9. **Groups are the key to managing team access** — add users to groups, set group ownership on directories
