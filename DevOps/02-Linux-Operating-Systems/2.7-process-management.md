# 2.7 — Process Management — ps, top, htop, kill, systemctl, journalctl

> **One Line Answer:** Process management is all about **seeing what's running on your Linux system, how much resources it's eating, and controlling it** — starting, stopping, or killing processes when needed.

---

## 🤔 What is a Process?

Let's start super simple.

Every time you **run a command, open an app, or start a service** on Linux, the system creates a **process**. A process is basically a **running instance of a program**.

Think of it like this:

```
Program = A recipe written in a cookbook (sitting on disk, doing nothing)
Process = You actually cooking that recipe in the kitchen (alive, using resources)
```

So when you type `ls` in the terminal, Linux creates a process for `ls`, it runs, shows you the files, and then the process **dies** (because its job is done).

But some processes **keep running** — like a web server (nginx), a database (MySQL), or your SSH connection. These are called **long-running processes** or **daemons** (services).

---

## 🧠 Key Concepts Before We Dive In

| Concept | Meaning |
|---------|---------|
| **PID** | Process ID — a unique number assigned to every process |
| **PPID** | Parent Process ID — the PID of the process that started this one |
| **Daemon** | A background process that runs continuously (like nginx, sshd) |
| **Foreground** | A process running in your terminal — you can see its output |
| **Background** | A process running behind the scenes — your terminal is free |
| **Zombie** | A dead process whose parent hasn't cleaned it up yet |
| **Orphan** | A process whose parent has died — adopted by PID 1 (init/systemd) |
| **Signal** | A message sent to a process (like "stop", "pause", "die") |

### Process Lifecycle:

```
Program on disk
      ↓
You run it (./app or command)
      ↓
Kernel creates a process (assigns PID)
      ↓
Process runs (uses CPU, RAM, I/O)
      ↓
Process finishes OR you kill it
      ↓
Process dies → Resources freed
```

### Every Process Has a Parent:

```
systemd (PID 1) ← the FIRST process (grandparent of everything)
   ├── sshd (PID 500)
   │     └── bash (PID 1200) ← your terminal shell
   │           └── python app.py (PID 1500) ← you ran this command
   ├── nginx (PID 600)
   └── cron (PID 700)
```

If you want to see this tree, run:

```bash
pstree
```

## 📋 1. `ps` — View Running Processes (Snapshot)

`ps` stands for **"Process Status"**. It shows you a **snapshot** of currently running processes at that exact moment. It doesn't update live — it's a one-time photo.

### Basic Usage:

```bash
ps
```

This shows only processes **in your current terminal session**:

```
  PID TTY          TIME CMD
 1200 pts/0    00:00:00 bash
 1550 pts/0    00:00:00 ps
```

- **PID** — Process ID
- **TTY** — Terminal associated with the process
- **TIME** — Total CPU time used
- **CMD** — The command that started this process

### Show ALL Processes (The Most Common Command):

```bash
ps aux
```

This is the **most used `ps` command in the DevOps world**. Let's break down the flags:

| Flag | Meaning |
|------|---------|
| `a` | Show processes from **all users** |
| `u` | Show **user-oriented** format (with CPU%, MEM%, etc.) |
| `x` | Show processes that are **not attached to a terminal** (daemons/services) |

Output looks like:

```
USER       PID %CPU %MEM    VSZ   RSS TTY      STAT START   TIME COMMAND
root         1  0.0  0.1 169316 13092 ?        Ss   May01   0:05 /usr/lib/systemd/systemd
root       500  0.0  0.0  15432  4520 ?        Ss   May01   0:00 /usr/sbin/sshd
www-data   600  0.0  0.2  45628 18340 ?        S    May01   0:10 nginx: worker process
ritesh    1200  0.0  0.0  21468  5184 pts/0    Ss   10:30   0:00 -bash
ritesh    1550  0.0  0.0  37120  3312 pts/0    R+   10:45   0:00 ps aux
```

Let's understand each column:

| Column | What It Means |
|--------|--------------|
| **USER** | Who owns this process |
| **PID** | Process ID |
| **%CPU** | Percentage of CPU being used |
| **%MEM** | Percentage of RAM being used |
| **VSZ** | Virtual memory size (total memory the process *could* use) |
| **RSS** | Resident Set Size (actual physical memory being used RIGHT NOW) |
| **TTY** | Terminal. `?` means no terminal (it's a daemon/background process) |
| **STAT** | Process state (see below) |
| **START** | When the process started |
| **TIME** | Total CPU time consumed |
| **COMMAND** | The full command that started the process |

### Understanding STAT (Process States):

| State | Meaning |
|-------|---------|
| `R` | Running — actively using CPU right now |
| `S` | Sleeping — waiting for something (I/O, signal, etc.) |
| `D` | Uninterruptible Sleep — usually waiting for disk I/O (can't be killed!) |
| `Z` | Zombie — process finished but parent hasn't collected its exit status |
| `T` | Stopped — paused by a signal (Ctrl+Z) |
| `+` | Running in foreground |
| `s` | Session leader |
| `l` | Multi-threaded |

### Another Popular Format — Using `-ef`:

```bash
ps -ef
```

| Flag | Meaning |
|------|---------|
| `-e` | Show **every** process |
| `-f` | **Full format** (shows PPID, which `aux` doesn't) |

Output:

```
UID        PID  PPID  C STIME TTY          TIME CMD
root         1     0  0 May01 ?        00:00:05 /usr/lib/systemd/systemd
root       500     1  0 May01 ?        00:00:00 /usr/sbin/sshd
```

Notice you get **PPID** (Parent Process ID) here — useful for finding who spawned a process.

### Most Useful `ps` Recipes:

```bash
# Find a specific process (e.g., nginx)
ps aux | grep nginx

# Show processes sorted by CPU usage (highest first)
ps aux --sort=-%cpu | head -20

# Show processes sorted by memory usage (highest first)
ps aux --sort=-%mem | head -20

# Show process tree (parent-child relationships)
ps auxf

# Find the PID of a specific process
ps aux | grep sshd | grep -v grep

# Better way to find PID — use pgrep
pgrep nginx
# Returns just the PID(s): 600

# Even better — pidof
pidof nginx
```

### `ps aux` vs `ps -ef` — When to Use Which?

| `ps aux` | `ps -ef` |
|----------|----------|
| Shows %CPU, %MEM | Shows PPID (parent PID) |
| BSD-style syntax | Unix/POSIX-style syntax |
| More common in DevOps/sysadmin | More common in scripting |
| Use when you care about **resource usage** | Use when you care about **process relationships** |

## 📊 2. `top` — Real-Time Process Monitor

While `ps` gives you a snapshot, `top` gives you a **live, continuously updating view** of what's happening on your system. Think of it as a **task manager** for Linux.

### Basic Usage:

```bash
top
```

This opens a full-screen, auto-refreshing display:

```
top - 10:45:23 up 26 days,  3:12,  2 users,  load average: 0.52, 0.38, 0.30
Tasks: 215 total,   1 running, 213 sleeping,   0 stopped,   1 zombie
%Cpu(s):  5.2 us,  2.1 sy,  0.0 ni, 91.8 id,  0.5 wa,  0.0 hi,  0.4 si,  0.0 st
MiB Mem :   7963.2 total,   1234.5 free,   3456.7 used,   3272.0 buff/cache
MiB Swap:   2048.0 total,   2048.0 free,      0.0 used.   4100.5 avail Mem

  PID USER      PR  NI    VIRT    RES    SHR S  %CPU  %MEM     TIME+ COMMAND
 1234 mysql      20   0 1456320 345612  12340 S  12.3   4.3   5:23.45 mysqld
  567 www-data   20   0  456780 123456   8900 S   5.6   1.5   2:10.12 nginx
  890 root       20   0   45678  12345   6789 R   2.3   0.2   0:05.67 top
```

### Understanding the Header:

**Line 1 — System Overview:**

```
top - 10:45:23 up 26 days, 3:12, 2 users, load average: 0.52, 0.38, 0.30
        │           │            │              │
        │           │            │              └── CPU load (1min, 5min, 15min)
        │           │            └── Number of logged-in users
        │           └── System uptime (running for 26 days)
        └── Current time
```

**Load Average** is super important:
- It shows **how busy the CPU is** over 1 minute, 5 minutes, and 15 minutes
- If you have **4 CPUs**, a load average of **4.0 means 100% utilized**
- Load average of **8.0 on a 4-CPU system** = system is **overloaded** (processes are waiting)
- **Rule of thumb:** Load average should be **less than the number of CPUs**

```bash
# Check how many CPUs you have:
nproc
# Output: 4

# So load average should ideally stay below 4.0
```

**Line 2 — Tasks:**

```
Tasks: 215 total, 1 running, 213 sleeping, 0 stopped, 1 zombie
```

- **Running** — actively using CPU
- **Sleeping** — waiting (normal, most processes sleep most of the time)
- **Stopped** — paused
- **Zombie** — dead but not cleaned up (if you see many zombies, something is wrong!)

**Line 3 — CPU Usage:**

```
%Cpu(s): 5.2 us, 2.1 sy, 0.0 ni, 91.8 id, 0.5 wa, 0.0 hi, 0.4 si, 0.0 st
```

| Code | Meaning |
|------|---------|
| `us` | User space — CPU time used by user applications |
| `sy` | System/kernel — CPU time used by the Linux kernel |
| `ni` | Nice — CPU time for low-priority (nice) processes |
| `id` | Idle — CPU doing nothing (higher is better!) |
| `wa` | I/O Wait — CPU waiting for disk I/O (high = disk bottleneck!) |
| `hi` | Hardware interrupts |
| `si` | Software interrupts |
| `st` | Steal time — CPU stolen by hypervisor (only in VMs/cloud) |

**Key insight:** If `wa` (I/O wait) is high (>20%), your **disk is the bottleneck**, not the CPU!

**Line 4-5 — Memory:**

```
MiB Mem :  7963.2 total, 1234.5 free, 3456.7 used, 3272.0 buff/cache
MiB Swap: 2048.0 total, 2048.0 free,    0.0 used. 4100.5 avail Mem
```

- **buff/cache** — Memory used by Linux for caching (this is GOOD, Linux uses free RAM as cache)
- **avail Mem** — Memory actually available for new processes (this is the number you should look at)
- **Swap** — If swap is being used, you're **running low on RAM**

### Interactive Commands Inside `top`:

Once `top` is running, you can press these keys:

| Key | What It Does |
|-----|-------------|
| `q` | Quit top |
| `h` | Show help |
| `P` (shift+p) | Sort by **CPU usage** (default) |
| `M` (shift+m) | Sort by **Memory usage** |
| `T` (shift+t) | Sort by **Time** |
| `k` | Kill a process (prompts for PID) |
| `r` | Renice a process (change priority) |
| `1` (number one) | Show **individual CPU cores** |
| `c` | Show **full command path** |
| `f` | Choose which **fields/columns** to display |
| `d` | Change refresh **delay** (default 3 seconds) |
| `u` | Filter by a specific **user** |
| `W` | Save current settings |

### Useful `top` One-Liners:

```bash
# Run top for a specific user
top -u ritesh

# Run top and show only 10 processes
top -n 1 -b | head -17  
# -n 1 = run once (batch mode)
# -b   = batch mode (for scripting, no interactive screen)

# Sort by memory in batch mode (useful for scripting/logging)
top -b -n 1 -o %MEM | head -20
```

## 💀 4. `kill` — Sending Signals to Processes

Sometimes a process misbehaves — it freezes, uses too much CPU, or you simply need to stop it. That's when you **kill** it.

But `kill` doesn't always mean "destroy." It actually **sends a signal** to a process. Different signals tell the process different things.

### Understanding Signals:

```
Signal = A message from you (or the system) to a process

"Hey process, please stop gracefully" → SIGTERM (15)
"Hey process, I don't care — DIE NOW" → SIGKILL (9)
"Hey process, pause for a moment" → SIGSTOP (19)
"Hey process, resume" → SIGCONT (18)
```

### The Most Important Signals:

| Signal | Number | Meaning | Can Process Ignore It? |
|--------|--------|---------|----------------------|
| `SIGHUP` | 1 | Hang up — terminal closed. Also used to **reload config** | Yes |
| `SIGINT` | 2 | Interrupt — same as pressing **Ctrl+C** | Yes |
| `SIGQUIT` | 3 | Quit — similar to SIGINT but creates a core dump | Yes |
| `SIGKILL` | 9 | **FORCE KILL** — process is immediately destroyed | **NO! Cannot be ignored!** |
| `SIGTERM` | 15 | **Graceful termination** — "please shut down cleanly" (DEFAULT) | Yes |
| `SIGSTOP` | 19 | **Pause** the process (like pressing Ctrl+Z) | **NO!** |
| `SIGCONT` | 18 | **Resume** a stopped process | Yes |
| `SIGUSR1` | 10 | User-defined signal 1 (apps can use for custom actions) | Yes |
| `SIGUSR2` | 12 | User-defined signal 2 | Yes |

### Using `kill`:

```bash
# Default kill (sends SIGTERM = signal 15 = graceful shutdown)
kill 1234
# This says: "Process 1234, please shut down gracefully"

# Force kill (SIGKILL = signal 9 = immediate death)
kill -9 1234
# This says: "Process 1234, you're DEAD. No questions asked."

# Send a specific signal by name
kill -SIGTERM 1234
kill -SIGKILL 1234
kill -SIGHUP 1234

# Send a specific signal by number
kill -15 1234
kill -9 1234
kill -1 1234
```

### ⚠️ IMPORTANT: Always Try SIGTERM Before SIGKILL!

```
SIGTERM (kill 1234):
  → Process receives the signal
  → Process can clean up (close files, save data, close connections)
  → Process shuts down gracefully
  → ✅ Data is safe

SIGKILL (kill -9 1234):
  → Process is IMMEDIATELY destroyed by the kernel
  → Process gets NO chance to clean up
  → Open files may be corrupted
  → Database transactions may be lost
  → ❌ Data may be lost!

GOLDEN RULE:
  1. First try:  kill <PID>        (SIGTERM — graceful)
  2. Wait 5-10 seconds
  3. If still alive: kill -9 <PID> (SIGKILL — force)
```

### `killall` — Kill by Process Name:

```bash
# Kill all nginx processes
killall nginx

# Force kill all nginx processes
killall -9 nginx

# Kill all processes owned by a user
killall -u ritesh
```

### `pkill` — Kill by Pattern:

```bash
# Kill processes whose name matches a pattern
pkill nginx

# Kill processes matching a pattern owned by a specific user
pkill -u ritesh python

# Force kill
pkill -9 java
```

### `xkill` — Kill a GUI Window (Desktop):

```bash
# On desktop Linux, if a window freezes:
xkill
# Your cursor turns into an X — click on the frozen window to kill it
```

### View All Available Signals:

```bash
kill -l
```

Output:

```
 1) SIGHUP       2) SIGINT       3) SIGQUIT      4) SIGILL       5) SIGTRAP
 6) SIGABRT      7) SIGBUS       8) SIGFPE       9) SIGKILL     10) SIGUSR1
11) SIGSEGV     12) SIGUSR2     13) SIGPIPE     14) SIGALRM     15) SIGTERM
... (and more)
```

## ⚙️ 5. `systemctl` — Managing Services (systemd)

In modern Linux (Ubuntu 16+, CentOS 7+, Debian 8+), **systemd** is the **init system** — it's the very first process (PID 1) that starts when your computer boots, and it manages ALL services.

`systemctl` is the command you use to **control systemd** — start services, stop them, enable them to start on boot, check their status, etc.

### Why This Matters for DevOps:

As a DevOps engineer, you'll constantly manage services like:
- `nginx` (web server)
- `docker` (container engine)
- `sshd` (SSH server)
- `mysql` / `postgresql` (databases)
- `prometheus` (monitoring)
- `jenkins` (CI/CD)

You NEED to know `systemctl` inside and out.

### Basic Service Management:

```bash
# Check the status of a service
sudo systemctl status nginx
```

Output:

```
● nginx.service - A high performance web server
   Loaded: loaded (/lib/systemd/system/nginx.service; enabled; vendor preset: enabled)
   Active: active (running) since Mon 2025-05-01 10:30:00 UTC; 26 days ago
     Docs: man:nginx(8)
 Main PID: 600 (nginx)
    Tasks: 5 (limit: 4915)
   Memory: 12.3M
   CGroup: /system.slice/nginx.service
           ├─600 nginx: master process /usr/sbin/nginx
           └─601 nginx: worker process

May 01 10:30:00 server01 systemd[1]: Starting A high performance web server...
May 01 10:30:00 server01 systemd[1]: Started A high performance web server.
```

Let's understand the key parts:

| Part | Meaning |
|------|---------|
| `Loaded: loaded` | The service unit file exists and was loaded |
| `enabled` | Will **auto-start on boot** |
| `Active: active (running)` | Currently running right now |
| `Main PID: 600` | The main process ID |
| `Memory: 12.3M` | RAM being used |
| `CGroup` | Shows all processes belonging to this service |

### All the `systemctl` Commands You Need:

```bash
# Start a service
sudo systemctl start nginx

# Stop a service
sudo systemctl stop nginx

# Restart a service (stop + start)
sudo systemctl restart nginx

# Reload configuration WITHOUT restarting (zero downtime!)
sudo systemctl reload nginx

# Reload OR restart (reload if supported, otherwise restart)
sudo systemctl reload-or-restart nginx

# Enable service to start on boot
sudo systemctl enable nginx

# Disable service from starting on boot
sudo systemctl disable nginx

# Enable AND start in one command
sudo systemctl enable --now nginx

# Disable AND stop in one command
sudo systemctl disable --now nginx

# Check if a service is running
sudo systemctl is-active nginx
# Output: active (or inactive)

# Check if a service is enabled (starts on boot)
sudo systemctl is-enabled nginx
# Output: enabled (or disabled)

# Check if a service failed
sudo systemctl is-failed nginx
# Output: active (or failed)

# Mask a service (completely prevent it from starting — even manually!)
sudo systemctl mask nginx

# Unmask a service
sudo systemctl unmask nginx
```

### Understanding Enable vs Start:

```
start   → Starts the service RIGHT NOW (but won't survive reboot)
enable  → Sets the service to auto-start on BOOT (but doesn't start it now)

So you usually need BOTH:
  sudo systemctl enable nginx    ← start on boot
  sudo systemctl start nginx     ← start right now

Or combine:
  sudo systemctl enable --now nginx   ← does both!
```

### Listing Services:

```bash
# List ALL services (running, stopped, failed, etc.)
sudo systemctl list-units --type=service

# List only running services
sudo systemctl list-units --type=service --state=running

# List only failed services (VERY useful for troubleshooting!)
sudo systemctl list-units --type=service --state=failed

# List all enabled services (that start on boot)
sudo systemctl list-unit-files --type=service --state=enabled
```

### Creating Your Own systemd Service:

Sometimes you have a custom application (like a Python app, Node.js server, or Go binary) that you want to run as a service. Here's how:

**Step 1: Create a service unit file:**

```bash
sudo nano /etc/systemd/system/myapp.service
```

**Step 2: Add the configuration:**

```ini
[Unit]
Description=My Custom Application
After=network.target
Wants=network-online.target

[Service]
Type=simple
User=appuser
Group=appuser
WorkingDirectory=/opt/myapp
ExecStart=/usr/bin/python3 /opt/myapp/main.py
Restart=always
RestartSec=5
StandardOutput=journal
StandardError=journal
Environment=APP_ENV=production
Environment=APP_PORT=8080

[Install]
WantedBy=multi-user.target
```

Let's understand every line:

| Section | Directive | Meaning |
|---------|-----------|---------|
| `[Unit]` | `Description` | Human-readable name |
| | `After` | Start this service AFTER networking is ready |
| | `Wants` | Soft dependency (nice to have) |
| `[Service]` | `Type=simple` | The process started by ExecStart is the main process |
| | `User` | Run as this user (don't run as root!) |
| | `WorkingDirectory` | cd to this directory before starting |
| | `ExecStart` | The actual command to run |
| | `Restart=always` | Restart the service if it crashes |
| | `RestartSec=5` | Wait 5 seconds before restarting |
| | `StandardOutput=journal` | Send stdout to journald (so you can read it with journalctl) |
| | `Environment` | Set environment variables |
| `[Install]` | `WantedBy=multi-user.target` | Enable this service for normal multi-user mode (standard servers) |

**Step 3: Enable and start:**

```bash
# Reload systemd to pick up the new file
sudo systemctl daemon-reload

# Enable and start
sudo systemctl enable --now myapp

# Check status
sudo systemctl status myapp
```

**Step 4: If you edit the service file later:**

```bash
# After editing the .service file, ALWAYS reload:
sudo systemctl daemon-reload

# Then restart the service:
sudo systemctl restart myapp
```

### Common `Restart` Options:

| Value | When Does It Restart? |
|-------|-----------------------|
| `no` | Never (default) |
| `always` | Always restart, no matter why it stopped |
| `on-failure` | Only restart if it exited with a non-zero exit code |
| `on-abnormal` | Restart on signal, timeout, or watchdog failure |
| `on-abort` | Restart on unclean signal (crash) |

## 📓 6. `journalctl` — Reading System Logs

`journalctl` is the command to **read logs from systemd's journal** (called `journald`). Every service managed by systemd sends its logs here.

This is your **#1 tool for debugging services** that won't start or are behaving weirdly.

### Basic Usage:

```bash
# Show ALL logs (very long output!)
sudo journalctl

# Show logs and scroll to the BOTTOM (most recent)
sudo journalctl -e

# Follow logs in real-time (like tail -f)
sudo journalctl -f
```

### View Logs for a Specific Service (MOST COMMON):

```bash
# Show all logs for nginx
sudo journalctl -u nginx

# Show logs for nginx — most recent at bottom
sudo journalctl -u nginx -e

# Follow nginx logs in real-time
sudo journalctl -u nginx -f

# Show only the last 50 lines
sudo journalctl -u nginx -n 50

# Show only the last 100 lines and follow
sudo journalctl -u nginx -n 100 -f
```

### Filter by Time:

```bash
# Logs from today only
sudo journalctl -u nginx --since today

# Logs from the last 1 hour
sudo journalctl -u nginx --since "1 hour ago"

# Logs from the last 30 minutes
sudo journalctl -u nginx --since "30 min ago"

# Logs between two dates
sudo journalctl -u nginx --since "2025-05-01 00:00:00" --until "2025-05-02 00:00:00"

# Logs since last boot
sudo journalctl -u nginx -b
```

### Filter by Priority (Severity):

```bash
# Show only ERROR and above
sudo journalctl -u nginx -p err

# Show only WARNINGS and above
sudo journalctl -u nginx -p warning

# Show only CRITICAL
sudo journalctl -u nginx -p crit
```

Priority levels (from most to least severe):

| Priority | Number | Meaning |
|----------|--------|---------|
| `emerg` | 0 | System is unusable |
| `alert` | 1 | Action must be taken immediately |
| `crit` | 2 | Critical conditions |
| `err` | 3 | Error conditions |
| `warning` | 4 | Warning conditions |
| `notice` | 5 | Normal but significant |
| `info` | 6 | Informational |
| `debug` | 7 | Debug-level messages |

### Output Formats:

```bash
# JSON output (useful for parsing with scripts or jq)
sudo journalctl -u nginx -o json-pretty

# Short format (default, one line per entry)
sudo journalctl -u nginx -o short

# Verbose (show ALL metadata)
sudo journalctl -u nginx -o verbose
```

### Check Disk Usage of Logs:

```bash
# How much disk space are logs using?
sudo journalctl --disk-usage
# Output: Archived and active journals take up 1.2G in the file system.

# Clean up old logs — keep only last 7 days
sudo journalctl --vacuum-time=7d

# Clean up old logs — keep only 500M
sudo journalctl --vacuum-size=500M
```

### Real-World Debugging Example:

Let's say nginx won't start. Here's your debugging workflow:

```bash
# Step 1: Check the status
sudo systemctl status nginx
# You see: Active: failed (Result: exit-code)

# Step 2: Read the logs
sudo journalctl -u nginx -n 50 --no-pager
# You see the error: "bind() to 0.0.0.0:80 failed (98: Address already in use)"

# Step 3: Find what's using port 80
sudo ss -tlnp | grep :80
# Output: LISTEN  0  128  0.0.0.0:80  *  users:(("apache2",pid=1234,fd=4))

# Step 4: Apache is using port 80! Stop it.
sudo systemctl stop apache2

# Step 5: Start nginx again
sudo systemctl start nginx

# Step 6: Verify
sudo systemctl status nginx
# Active: active (running) ✅
```

## 🔄 Foreground vs Background Processes

This is a very practical concept you'll use daily.

### Running a Process in the Background:

```bash
# Add & at the end to run in background
./long-script.sh &
# Output: [1] 12345  (job number and PID)

# Your terminal is free — you can type other commands!
```

### Moving Between Foreground and Background:

```bash
# Start a process normally (foreground)
./long-script.sh

# Press Ctrl+Z to PAUSE it and send it to background
# Output: [1]+  Stopped  ./long-script.sh

# Resume it in the BACKGROUND
bg
# Output: [1]+ ./long-script.sh &

# Bring it back to FOREGROUND
fg

# If you have multiple background jobs, list them
jobs
# Output:
# [1]+  Running   ./long-script.sh &
# [2]-  Stopped   python3 server.py

# Bring a specific job to foreground
fg %2   # brings job [2] to foreground
```

### `nohup` — Keep Running After Logout:

Normally, when you close your terminal or SSH session, all background processes get killed. `nohup` prevents that:

```bash
# Run a process that survives logout
nohup ./long-script.sh &

# Output goes to nohup.out by default
# Redirect to a custom file:
nohup ./long-script.sh > /var/log/myapp.log 2>&1 &
```

What does `> /var/log/myapp.log 2>&1 &` mean?

```
> /var/log/myapp.log   → Redirect stdout to this file
2>&1                   → Redirect stderr to the same place as stdout
&                      → Run in background
```
