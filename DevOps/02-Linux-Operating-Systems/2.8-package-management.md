## 🤔 What is a "Package"?

Let's start simple. When you want to install software on Windows, you download a `.exe` file and double-click it. On macOS, you download a `.dmg` file.

On Linux, software comes in **packages**. A package is basically a **compressed archive** that contains:

```
A Package Contains:
  ├── The actual program files (binaries, libraries)
  ├── Configuration files
  ├── Documentation
  ├── Metadata (name, version, description)
  └── Dependency info (what OTHER packages this needs to work)
```

Think of a package like a **shipping box** from Amazon — it contains the product, the manual, and a label that says what else you need.

## 🧠 Key Concepts

### 1. Package vs Package Manager

| Term | What It Is | Example |
|------|-----------|---------|
| **Package** | A single software bundle | `nginx_1.24.0-1_amd64.deb` |
| **Package Manager** | The tool that installs/removes packages | `apt`, `yum`, `dnf` |
| **Repository (Repo)** | An online server/storage that holds packages | Ubuntu's repos, EPEL |

```
Repository (online server)
    │
    │ ← Package manager downloads from here
    ↓
Package Manager (apt, yum, dnf)
    │
    │ ← Installs to your system
    ↓
Your Linux System
```

### 2. Dependencies

Most software **depends on other software** to work. For example:

```
You want to install: WordPress
  WordPress needs: PHP
    PHP needs: libxml2
      libxml2 needs: zlib

This chain is called a "dependency tree"
```

The **package manager automatically handles dependencies** for you! When you install WordPress, it installs PHP, libxml2, and zlib automatically. This is the **magic** of package managers.

### 3. Two Families of Linux Distros

The Linux world is mainly divided into **two big families**, and each uses a **different package format**:

| Family | Distros | Package Format | Low-Level Tool | High-Level Tool |
|--------|---------|---------------|---------------|-----------------|
| **Debian-based** | Ubuntu, Debian, Linux Mint, Kali | `.deb` | `dpkg` | `apt` (or older `apt-get`) |
| **Red Hat-based** | RHEL, CentOS, Fedora, Amazon Linux, Rocky Linux, AlmaLinux | `.rpm` | `rpm` | `yum` or `dnf` |

```
Think of it like:

Debian Family:
  dpkg   = the low-level mechanic (installs individual .deb files)
  apt    = the smart manager (handles repos, dependencies, updates)

Red Hat Family:
  rpm    = the low-level mechanic (installs individual .rpm files)
  yum/dnf = the smart manager (handles repos, dependencies, updates)
```

**You almost always use the high-level tool** (`apt` or `yum`/`dnf`). The low-level tools (`dpkg`, `rpm`) are for special cases.

## 📦 1. `apt` — Debian/Ubuntu Package Manager

`apt` (Advanced Package Tool) is the **primary package manager** for Ubuntu, Debian, and their derivatives. It's the one you'll use most often if you're working with Ubuntu servers (which are VERY common in DevOps).

> **Note:** `apt` is the modern, user-friendly version. `apt-get` is the older version. Both work, but `apt` is recommended for interactive use.

### Updating Package Lists:

Before installing anything, you should **update the package list** — this tells your system about the latest available packages:

```bash
sudo apt update
```

What this does:

```
Your system contacts all configured repositories:
  → "Hey repo, what packages do you have? What versions?"
  
Repo responds:
  → "Here's my complete list of packages and versions"
  
Your system saves this list locally.

NOW your system knows what's available to install.

IMPORTANT: This does NOT install or upgrade anything!
            It just refreshes the "menu" (package list).
```

### Installing Packages:

```bash
# Install a single package
sudo apt install nginx

# Install multiple packages at once
sudo apt install nginx curl wget vim

# Install a specific version
sudo apt install nginx=1.24.0-1ubuntu1

# Install without asking for confirmation (-y = yes to all)
sudo apt install -y nginx

# Install a package and its recommended extras
sudo apt install --install-recommends nginx
```

When you run `sudo apt install nginx`, here's what happens behind the scenes:

```
Step 1: apt checks its local package list (from apt update)
Step 2: Finds the nginx package and all its dependencies
Step 3: Downloads nginx + all dependencies from the repo
Step 4: Installs dependencies first (in the right order)
Step 5: Installs nginx
Step 6: Runs any post-install scripts (creates configs, users, etc.)
Step 7: Done! nginx is ready to use ✅
```

### Removing Packages:

```bash
# Remove a package (keeps config files)
sudo apt remove nginx

# Remove a package AND its config files (complete removal)
sudo apt purge nginx

# Remove unused dependencies (packages that were installed automatically 
# but are no longer needed by any installed package)
sudo apt autoremove

# Purge + autoremove (cleanest removal)
sudo apt purge nginx && sudo apt autoremove
```

Difference between `remove` and `purge`:

```
remove:
  ✅ Deletes the program files
  ❌ Keeps config files in /etc/ (in case you reinstall later)

purge:
  ✅ Deletes the program files
  ✅ Deletes config files too
  
Use purge when you want a CLEAN removal.
Use remove when you might reinstall and want to keep your configs.
```

### Upgrading Packages:

```bash
# Upgrade ALL installed packages to their latest versions
sudo apt upgrade

# Upgrade with automatic handling of dependencies 
# (may install new packages or remove old ones if needed)
sudo apt full-upgrade

# Update + Upgrade in one go (most common maintenance command)
sudo apt update && sudo apt upgrade -y
```

Difference between `upgrade` and `full-upgrade`:

```
upgrade:
  → Upgrades packages to latest version
  → Will NOT remove any packages
  → Will NOT install new packages
  → "Safe" upgrade

full-upgrade (previously dist-upgrade):
  → Upgrades packages to latest version
  → MAY remove packages if needed
  → MAY install new dependencies
  → Needed for major version upgrades
```

### Searching for Packages:

```bash
# Search for a package by name or description
apt search nginx

# Show detailed info about a package
apt show nginx

# List all installed packages
apt list --installed

# List all upgradable packages
apt list --upgradable

# Check if a specific package is installed
apt list --installed | grep nginx
# OR
dpkg -l | grep nginx
```

### Managing Repositories:

Sometimes the software you need isn't in the default repos. You need to **add a new repository**.

```bash
# Add a new repository (PPA — Personal Package Archive)
sudo add-apt-repository ppa:ondrej/php

# After adding a repo, ALWAYS update:
sudo apt update

# Remove a repository
sudo add-apt-repository --remove ppa:ondrej/php

# List all configured repositories
cat /etc/apt/sources.list
ls /etc/apt/sources.list.d/
```

### `apt` Configuration Files:

```
/etc/apt/sources.list              → Main repository list
/etc/apt/sources.list.d/           → Additional repository files (one per file)
/var/cache/apt/archives/           → Downloaded .deb packages (cache)
/var/lib/apt/lists/                → Package list data (from apt update)
```

### Cleaning Up:

```bash
# Remove downloaded .deb files from cache
sudo apt clean
# Frees up disk space, doesn't affect installed packages

# Remove only outdated .deb files from cache
sudo apt autoclean

# Remove unused dependencies
sudo apt autoremove
```

## 📦 2. `dpkg` — Low-Level Debian Package Tool

`dpkg` is the **low-level tool** that directly installs `.deb` files. `apt` actually uses `dpkg` behind the scenes. You use `dpkg` when you have a `.deb` file downloaded locally.

### When Would You Use `dpkg`?

```
Scenario: You downloaded Google Chrome's .deb file from the website
          (Chrome is not in Ubuntu's repos)

File: google-chrome-stable_current_amd64.deb

You can't use apt install for a local file → use dpkg!
```

### Common `dpkg` Commands:

```bash
# Install a local .deb file
sudo dpkg -i package.deb
# Example:
sudo dpkg -i google-chrome-stable_current_amd64.deb

# If dpkg fails due to missing dependencies:
sudo apt install -f
# This tells apt: "Fix my broken dependencies"

# Remove a package
sudo dpkg -r package-name

# Purge a package (remove + delete config)
sudo dpkg -P package-name

# List ALL installed packages
dpkg -l

# Check if a specific package is installed
dpkg -l | grep nginx
# OR
dpkg -s nginx

# List all files installed by a package
dpkg -L nginx
# Shows:
# /usr/sbin/nginx
# /etc/nginx/nginx.conf
# /var/log/nginx/
# ... etc

# Find which package owns a specific file
dpkg -S /usr/sbin/nginx
# Output: nginx-core: /usr/sbin/nginx

# Show info about a .deb file (without installing)
dpkg -I package.deb

# List contents of a .deb file (without installing)
dpkg -c package.deb
```

### The `-i` then `-f` Pattern (Very Common!):

```bash
# Step 1: Try to install the .deb
sudo dpkg -i some-package.deb
# ERROR: dependency problems - leaving unconfigured

# Step 2: Fix the broken dependencies automatically
sudo apt install -f
# apt downloads and installs all missing dependencies
# Then finishes configuring the package

# ✅ Done!
```

This is such a common pattern that you should memorize it.

## 📦 3. `yum` — Red Hat/CentOS Package Manager (Older)

`yum` (Yellowdog Updater, Modified) is the package manager for **RHEL, CentOS 7, and older Red Hat-based** distros. It's similar to `apt` but for the Red Hat world.

> **Note:** `yum` has been replaced by `dnf` in newer systems (Fedora 22+, RHEL 8+, CentOS 8+). But many servers still run CentOS 7, so knowing `yum` is important.

### Basic `yum` Commands:

```bash
# Update package list AND upgrade all packages
sudo yum update
# Note: unlike apt, yum's "update" does BOTH (refresh list + upgrade packages)

# Install a package
sudo yum install nginx

# Install without confirmation
sudo yum install -y nginx

# Remove a package
sudo yum remove nginx

# Search for a package
yum search nginx

# Get info about a package
yum info nginx

# List all installed packages
yum list installed

# List available updates
yum list updates

# Check which package provides a file
yum provides /usr/sbin/nginx
# OR
yum whatprovides */nginx

# Clean cached data
sudo yum clean all

# View yum history (see past installs/updates)
sudo yum history
sudo yum history info <ID>

# Undo a previous yum transaction (rollback!)
sudo yum history undo <ID>
```

### Managing Repositories with `yum`:

```bash
# List all enabled repositories
yum repolist

# List all repos (including disabled)
yum repolist all

# Enable a repo
sudo yum-config-manager --enable epel

# Disable a repo
sudo yum-config-manager --disable epel

# Add the EPEL repo (Extra Packages for Enterprise Linux)
sudo yum install epel-release
```

### `yum` Group Install:

```bash
# Install a group of related packages
sudo yum groupinstall "Development Tools"
# This installs gcc, make, autoconf, and all development tools

# List available groups
yum grouplist
```

### `yum` Config Files:

```
/etc/yum.conf                → Main yum configuration
/etc/yum.repos.d/            → Repository files (one .repo file per repo)
/var/cache/yum/              → Package cache
```

## 📦 4. `dnf` — The Modern Replacement for yum

`dnf` (Dandified YUM) is the **next-generation package manager** for Red Hat-based systems. It replaces `yum` and is faster, more efficient, and handles dependencies better.

Used in: **Fedora 22+, RHEL 8+, CentOS Stream, Rocky Linux 8+, AlmaLinux 8+**

> **Good news:** `dnf` commands are almost **identical to `yum`** commands! If you know `yum`, you know `dnf`.

### Basic `dnf` Commands:

```bash
# Update all packages
sudo dnf update
# OR (same thing in dnf)
sudo dnf upgrade

# Install a package
sudo dnf install nginx

# Install without confirmation
sudo dnf install -y nginx

# Remove a package
sudo dnf remove nginx

# Search for a package
dnf search nginx

# Get info
dnf info nginx

# List installed packages
dnf list installed

# List available updates
dnf list updates
# OR
dnf check-update

# Clean cache
sudo dnf clean all

# Autoremove unused dependencies
sudo dnf autoremove

# View transaction history
sudo dnf history
sudo dnf history info <ID>
sudo dnf history undo <ID>
```

### What's Different in `dnf` vs `yum`:

| Feature | `yum` | `dnf` |
|---------|-------|-------|
| Speed | Slower | Faster (uses libsolv) |
| Dependency resolution | Sometimes fails | Better algorithm |
| Python dependency | Python 2 based | Python 3 based |
| Memory usage | Higher | Lower |
| `autoremove` | Not built-in | Built-in |
| Module support | No | Yes (dnf module) |
| API | Old | Modern |

### `dnf` Modules (RHEL 8+):

Modules let you install **different versions** of software from the same repo:

```bash
# List available modules
dnf module list

# List available streams for a module (e.g., different Node.js versions)
dnf module list nodejs
# Output:
#  nodejs  14  [default]  development, minimal, default
#  nodejs  16              development, minimal, default
#  nodejs  18              development, minimal, default

# Enable a specific version
sudo dnf module enable nodejs:18

# Install it
sudo dnf module install nodejs:18

# Reset module (to switch versions)
sudo dnf module reset nodejs
```

## 📦 5. `rpm` — Low-Level Red Hat Package Tool

`rpm` is to `yum`/`dnf` what `dpkg` is to `apt`. It's the **low-level tool** for working directly with `.rpm` files.

### Common `rpm` Commands:

```bash
# Install a local .rpm file
sudo rpm -ivh package.rpm
#  -i = install
#  -v = verbose (show progress)
#  -h = hash marks (show progress bar)

# Upgrade a package (install new version, remove old)
sudo rpm -Uvh package.rpm

# Remove a package
sudo rpm -e package-name

# List ALL installed packages
rpm -qa
# -q = query
# -a = all

# Check if a specific package is installed
rpm -qa | grep nginx
# OR
rpm -q nginx

# List files installed by a package
rpm -ql nginx
# -q = query
# -l = list files

# Find which package owns a file
rpm -qf /usr/sbin/nginx
# Output: nginx-1.24.0-1.el8.x86_64

# Show info about an installed package
rpm -qi nginx

# Show info about an .rpm file (without installing)
rpm -qip package.rpm

# List contents of an .rpm file (without installing)
rpm -qlp package.rpm

# Verify a package (check if files have been modified)
rpm -V nginx
# No output = everything is fine
# S.5....T = file size, checksum, or timestamp changed
```

### The `rpm` Query Flags Explained:

```
-q   = query mode (I want to ask about a package)
  -a = all installed packages
  -l = list files
  -i = info
  -f = find package owning a file
  -p = query a package FILE (not installed package)
  
Combine them:
  rpm -qa         → query all installed
  rpm -ql nginx   → query + list files for nginx
  rpm -qi nginx   → query + info about nginx
  rpm -qip file.rpm → query + info about a .rpm file
```

## 📦 6. `snap` — Universal Package Manager

`snap` is a **universal package manager** developed by Canonical (the company behind Ubuntu). The key idea: **one package works on any Linux distro**.

### Why snap Exists:

```
The Problem:
  Ubuntu uses .deb
  Fedora uses .rpm
  Arch uses .pkg.tar
  
  Software developers have to package their software
  DIFFERENTLY for each distro. That's a lot of work!

The Solution (snap):
  Create ONE package (a "snap") that works EVERYWHERE.
  A snap contains the app + ALL its dependencies bundled together.
```

### How snap Works:

```
Traditional Package (.deb/.rpm):
  App depends on system libraries → may conflict with other apps
  
Snap Package:
  App + ALL libraries bundled in a single file
  Runs in its own isolated sandbox
  No conflicts with other apps or system libraries
  
Think of it like shipping containers:
  .deb = loose cargo on a truck (can interact with other cargo)
  .snap = sealed container (isolated, self-contained)
```

### Basic `snap` Commands:

```bash
# Install snapd (the snap daemon) — needed first
sudo apt install snapd      # Ubuntu (usually pre-installed)
sudo yum install snapd      # CentOS/RHEL

# Install a snap package
sudo snap install vlc

# Install a "classic" snap (runs without sandbox — needs --classic)
sudo snap install code --classic
# VS Code, some IDEs, and dev tools need --classic

# List installed snaps
snap list

# Search for a snap
snap find nginx

# Get info about a snap
snap info vlc

# Update all snaps
sudo snap refresh

# Update a specific snap
sudo snap refresh vlc

# Remove a snap
sudo snap remove vlc

# Revert to previous version
sudo snap revert vlc

# List available versions/channels
snap info vlc
# Channels:
#   stable:    3.0.18
#   candidate: 3.0.19
#   beta:      3.0.20
#   edge:      3.0.21

# Install from a specific channel
sudo snap install vlc --channel=beta
```

### Snap vs apt — When to Use What?

| Aspect | `apt` | `snap` |
|--------|-------|--------|
| **Speed** | Faster (native) | Slower to start (sandboxed) |
| **Disk usage** | Smaller | Larger (bundles dependencies) |
| **Updates** | Manual (`apt upgrade`) | Automatic (background updates) |
| **Isolation** | Shares system libraries | Sandboxed (isolated) |
| **Compatibility** | Distro-specific | Works on any distro |
| **Best for** | System tools, servers | Desktop apps, dev tools |

> **DevOps tip:** For servers, stick with `apt`/`yum`/`dnf`. Snap is more popular for desktop Linux and some developer tools.

| Operation | `apt` (Debian/Ubuntu) | `yum` (CentOS 7) | `dnf` (RHEL 8+/Fedora) |

## 🏢 Real-World DevOps Scenarios

### Scenario 1: Setting Up a Web Server on Ubuntu

```bash
# Step 1: Update package list
sudo apt update

# Step 2: Install nginx
sudo apt install -y nginx

# Step 3: Start and enable nginx
sudo systemctl enable --now nginx

# Step 4: Verify
sudo systemctl status nginx
curl http://localhost
```

### Scenario 2: Setting Up a Web Server on CentOS/RHEL

```bash
# Step 1: Install EPEL repo (for extra packages)
sudo dnf install -y epel-release

# Step 2: Install nginx
sudo dnf install -y nginx

# Step 3: Start and enable
sudo systemctl enable --now nginx

# Step 4: Open firewall (CentOS has firewalld)
sudo firewall-cmd --permanent --add-service=http
sudo firewall-cmd --reload

# Step 5: Verify
curl http://localhost
```

### Scenario 3: Find Out What's Using Disk Space

```bash
# See all installed packages sorted by size (Ubuntu)
dpkg-query -Wf '${Installed-Size}\t${Package}\n' | sort -rn | head -20

# See all installed packages sorted by size (RHEL/CentOS)
rpm -qa --queryformat '%{SIZE}\t%{NAME}\n' | sort -rn | head -20
```

### Scenario 4: Downgrade a Package

```bash
# Ubuntu — install a specific older version
apt list -a nginx    # see all available versions
sudo apt install nginx=1.18.0-1ubuntu1

# RHEL/CentOS — downgrade
sudo dnf downgrade nginx
```

### Scenario 5: Pin/Hold a Package Version (Prevent Auto-Upgrade)

```bash
# Ubuntu — hold a package at current version
sudo apt-mark hold nginx
# Now "apt upgrade" will SKIP nginx

# Unhold (allow upgrades again)
sudo apt-mark unhold nginx

# List held packages
apt-mark showhold

# RHEL/CentOS — exclude from updates (in /etc/yum.conf or /etc/dnf/dnf.conf)
# Add this line:
# exclude=nginx*
```

## 🔒 Security Best Practices

```
1. ALWAYS run "apt update" (or equivalent) before installing
   → Ensures you get the latest (most secure) version

2. Only add repos from TRUSTED sources
   → Random PPAs/repos can contain malware

3. Regularly update your system
   → Security patches are released as package updates
   
4. Use "apt upgrade" or "dnf update" regularly
   → Unpatched servers are the #1 attack vector

5. Remove packages you don't need
   → Less software = smaller attack surface

6. On production servers, prefer LTS (Long Term Support) repos
   → More stable, get security updates longer
```
