# 2.9 — Disk Management — df, du, lsblk, mount, fdisk, LVM

> **One Line Answer:** Disk management is about **seeing how much storage you have, what's using it, adding new disks, partitioning them, and even resizing storage on the fly** — a critical skill because a full disk can crash your entire server.

---

## 🤔 Why Disk Management Matters in DevOps

Here's a real scenario that happens ALL the time:

```
3 AM — Your phone rings 📱
Alert: "CRITICAL — Server disk 95% full!"

If the disk fills to 100%:
  ❌ Databases crash (can't write data)
  ❌ Logs can't be written (can't debug anything)
  ❌ Applications crash
  ❌ Services refuse to start
  ❌ SSH might not even work!

Your job: Find out what's eating the disk and fix it — FAST.
```

That's why every DevOps engineer MUST know disk management inside and out.

## 🧠 Key Concepts — How Linux Storage Works

Before diving into commands, let's understand the storage hierarchy:

```
Physical Disk (e.g., /dev/sda — 500 GB hard drive)
    │
    ├── Partition 1 (/dev/sda1) — 500 MB — /boot (boot files)
    ├── Partition 2 (/dev/sda2) — 50 GB  — / (root filesystem)
    ├── Partition 3 (/dev/sda3) — 8 GB   — swap (virtual memory)
    └── Partition 4 (/dev/sda4) — 442 GB — /home (user files)
```

### Key Terms:

| Term | Meaning | Analogy |
|------|---------|---------|
| **Disk** | A physical or virtual storage device | The entire plot of land |
| **Partition** | A logical division of a disk | Dividing the land into rooms |
| **Filesystem** | The way data is organized on a partition (ext4, xfs, etc.) | Shelving system inside a room |
| **Mount Point** | The directory where a partition is attached | The door to enter the room |
| **Block Device** | Any storage device accessed in blocks | Hard drives, SSDs, USB drives |
| **Inode** | Metadata about a file (permissions, size, location) | The catalog entry for a file |

### Device Naming Convention:

```
/dev/sda   → First SCSI/SATA disk
/dev/sdb   → Second SCSI/SATA disk
/dev/sdc   → Third disk
...

/dev/sda1  → First partition on first disk
/dev/sda2  → Second partition on first disk
/dev/sda3  → Third partition on first disk

/dev/nvme0n1     → First NVMe SSD
/dev/nvme0n1p1   → First partition on first NVMe
/dev/nvme0n1p2   → Second partition on first NVMe

/dev/vda   → Virtual disk (in cloud VMs like AWS, GCP)
/dev/xvda  → Xen virtual disk (older AWS instances)
```

### Common Filesystems:

| Filesystem | Used By | Notes |
|-----------|---------|-------|
| **ext4** | Ubuntu/Debian default | Most common, very reliable |
| **xfs** | RHEL/CentOS default | Great for large files, can't shrink |
| **btrfs** | SUSE, Fedora (optional) | Advanced features (snapshots, compression) |
| **swap** | All Linux | Virtual memory (not really a filesystem) |
| **tmpfs** | All Linux | RAM-based filesystem (super fast, temporary) |
| **vfat/fat32** | USB drives | Cross-platform compatibility |
| **ntfs** | Windows drives | Can be mounted on Linux for dual-boot |

## 📊 1. `df` — Disk Free (Check Available Space)

`df` shows you how much disk space is **used and available** on each mounted filesystem. This is your **go-to command** when someone says "the disk is full!"

### Basic Usage:

```bash
df
```

Output:

```
Filesystem     1K-blocks      Used Available Use% Mounted on
/dev/sda2       51475068  18234560  30602892  38% /
/dev/sda1         523248     78432    444816  15% /boot
/dev/sda4      461456320 125678900 312345420  29% /home
tmpfs            4081640         0   4081640   0% /dev/shm
```

But those numbers in kilobytes are hard to read. Use `-h` for **human-readable**:

```bash
df -h
```

Output:

```
Filesystem      Size  Used Avail Use% Mounted on
/dev/sda2        50G   18G   30G  38% /
/dev/sda1       512M   77M  435M  15% /boot
/dev/sda4       440G  120G  298G  29% /home
tmpfs           3.9G     0  3.9G   0% /dev/shm
```

**MUCH better!** Now you can see:
- `/` (root) has 50 GB, 18 GB used, 30 GB free
- `/boot` has 512 MB, 77 MB used
- `/home` has 440 GB, 120 GB used

### Understanding the Columns:

| Column | Meaning |
|--------|---------|
| **Filesystem** | The device/partition |
| **Size** | Total size of the partition |
| **Used** | How much is used |
| **Avail** | How much is available |
| **Use%** | Percentage used (watch for >80%!) |
| **Mounted on** | Where it's attached in the directory tree |

### Useful `df` Options:

```bash
# Human-readable (GB, MB instead of blocks)
df -h

# Show filesystem type
df -hT
# Output adds a "Type" column: ext4, xfs, tmpfs, etc.

# Show only specific filesystem types
df -hT -t ext4
# Shows only ext4 filesystems

# Exclude specific filesystem types (hide tmpfs, devtmpfs etc.)
df -hT -x tmpfs -x devtmpfs -x squashfs

# Show inode usage (instead of disk space)
df -i
# Important! You can run out of inodes even if you have disk space!
# This happens when you have millions of tiny files
```

### What is an Inode Problem?

```
Normal disk full:
  → Disk space (bytes) is used up
  → Solution: Delete files or add more space

Inode exhaustion:
  → Disk has space BUT no more inodes (file entries) available
  → This happens when you create MILLIONS of small files
  → Common culprit: /tmp full of session files, mail queue, old logs
  → Solution: Delete the tiny files

How to check:
  df -i
  
  Filesystem      Inodes   IUsed   IFree IUse% Mounted on
  /dev/sda2      3276800 3276800       0  100% /         ← FULL! No inodes!
```

## 📏 2. `du` — Disk Usage (What's Using Space?)

While `df` tells you how much space is left, `du` tells you **what's eating the space**. It shows the disk usage of files and directories.

### Basic Usage:

```bash
# Show size of current directory and all subdirectories
du -h

# Show only the total size of the current directory
du -sh
# -s = summary (total only)
# -h = human-readable

# Show size of a specific directory
du -sh /var/log
# Output: 2.3G    /var/log

# Show sizes of all items in a directory (one level deep)
du -sh /var/*
```

Output:

```
4.0K    /var/backups
234M    /var/cache
16K     /var/crash
2.3G    /var/log
156M    /var/lib
4.0K    /var/local
0       /var/lock
12K     /var/mail
0       /var/opt
56K     /var/spool
20K     /var/tmp
```

Now you can see `/var/log` is eating 2.3 GB!

### The Most Useful `du` Commands:

```bash
# Find the TOP 10 largest directories (SUPER useful!)
du -h --max-depth=1 / 2>/dev/null | sort -rh | head -10

# Explanation:
#  du -h --max-depth=1 /   → Size of each directory one level under /
#  2>/dev/null              → Hide permission errors
#  sort -rh                 → Sort by size, largest first
#  head -10                 → Show only top 10

# Output:
#  120G    /home
#  18G     /var
#  5.2G    /usr
#  2.1G    /opt
#  512M    /boot
#  ...

# Find the TOP 10 largest files on the entire system
sudo find / -type f -exec du -h {} + 2>/dev/null | sort -rh | head -10

# Find the TOP 10 largest files in a specific directory
find /var -type f -exec du -h {} + 2>/dev/null | sort -rh | head -10

# Show disk usage and sort by size (another way)
du -ah /var/log | sort -rh | head -20
```

### Real-World Disk Space Investigation:

When the disk is full, follow this workflow:

```bash
# Step 1: See which partition is full
df -h
# Result: /dev/sda2  50G  48G  0G  100% /    ← ROOT IS FULL!

# Step 2: Find which top-level directory is biggest
sudo du -h --max-depth=1 / 2>/dev/null | sort -rh | head -10
# Result: 
#   30G   /var        ← /var is suspicious!
#   12G   /home
#   5G    /usr

# Step 3: Drill into /var
sudo du -h --max-depth=1 /var | sort -rh | head -10
# Result:
#   28G   /var/log    ← LOGS are eating everything!

# Step 4: Find the biggest log files
sudo find /var/log -type f -size +100M -exec ls -lh {} \;
# Result:
#   -rw-r----- 1 syslog adm 25G May 27 10:30 /var/log/syslog.1

# Step 5: Clean it up
sudo truncate -s 0 /var/log/syslog.1   # Empty the file (keeps it, sets size to 0)
# OR
sudo rm /var/log/syslog.1               # Delete the file

# Step 6: Verify
df -h
# Result: /dev/sda2  50G  20G  28G  42% /   ← FIXED! ✅
```

### `ncdu` — A Better `du` (Interactive):

`ncdu` (NCurses Disk Usage) is an **interactive, visual version of `du`**. It's AMAZING for exploring disk usage.

```bash
# Install
sudo apt install ncdu      # Ubuntu/Debian
sudo dnf install ncdu      # RHEL/Fedora

# Run it on root
sudo ncdu /

# Run it on a specific directory
sudo ncdu /var
```

It shows a beautiful interactive display where you can navigate with arrow keys:

```
ncdu 1.15.1 ~ Use the arrow keys to navigate, press ? for help

  120.5 GiB [##########] /home

    2.0 GiB [          ] /opt
  512.0 MiB [          ] /boot

```

Press `Enter` to drill into a directory, `d` to delete a file/directory.

## 🧱 3. `lsblk` — List Block Devices

`lsblk` shows all **block devices** (disks, partitions, USB drives) in a nice tree format. It's the best way to get an **overview of your storage layout**.

### Basic Usage:

```bash
lsblk
```

Output:

```
NAME    MAJ:MIN RM   SIZE RO TYPE MOUNTPOINT
sda       8:0    0   500G  0 disk 
├─sda1    8:1    0   512M  0 part /boot
├─sda2    8:2    0    50G  0 part /
├─sda3    8:3    0     8G  0 part [SWAP]
└─sda4    8:4    0 441.5G  0 part /home
sdb       8:16   0   100G  0 disk 
└─sdb1    8:17   0   100G  0 part 
sr0      11:0    1  1024M  0 rom  
```

This tells you:

```
Disk sda (500 GB) has 4 partitions:
  sda1 (512 MB) → mounted at /boot
  sda2 (50 GB)  → mounted at /
  sda3 (8 GB)   → swap
  sda4 (441 GB) → mounted at /home

Disk sdb (100 GB) has 1 partition:
  sdb1 (100 GB) → NOT mounted (no mount point shown)

sr0 = CD/DVD drive (ignore on servers)
```

### Useful `lsblk` Options:

```bash
# Show filesystem type
lsblk -f

# Output:
# NAME    FSTYPE  LABEL  UUID                                 MOUNTPOINT
# sda                                                         
# ├─sda1  ext4          a1b2c3d4-e5f6-7890-abcd-ef1234567890 /boot
# ├─sda2  ext4          b2c3d4e5-f6a7-8901-bcde-f12345678901 /
# ├─sda3  swap          c3d4e5f6-a7b8-9012-cdef-123456789012 [SWAP]
# └─sda4  ext4          d4e5f6a7-b8c9-0123-defa-234567890123 /home

# Show everything (size, type, mount, etc.)
lsblk -a

# Show only disks (not partitions)
lsblk -d

# Show specific columns
lsblk -o NAME,SIZE,TYPE,FSTYPE,MOUNTPOINT
```

### `fdisk -l` — Another Way to List Disks:

```bash
sudo fdisk -l
```

This gives you even more detail — partition tables, sector sizes, disk labels, etc. Useful for low-level disk work.

## 🔗 4. `mount` and `umount` — Attaching and Detaching Storage

In Linux, a disk partition isn't automatically accessible. You need to **mount** it — which means **attach it to a directory** so you can read/write files there.

### Understanding Mount:

```
Think of it like a USB drive:

Windows:
  Plug in USB → appears as D: drive automatically

Linux:
  Plug in USB → it's just /dev/sdb1 (a device, no directory)
  You need to MOUNT it:
    mount /dev/sdb1 /mnt/usb
  Now you can access files at /mnt/usb/

  When done:
    umount /mnt/usb
  Now /mnt/usb is empty again
```

### Mounting a Partition:

```bash
# Create a directory to mount to (the "mount point")
sudo mkdir -p /mnt/data

# Mount a partition to that directory
sudo mount /dev/sdb1 /mnt/data

# Verify
df -h | grep sdb1
# /dev/sdb1  100G  10G  90G  10% /mnt/data

# Now you can use /mnt/data like any directory:
ls /mnt/data
cd /mnt/data
cp /home/ritesh/file.txt /mnt/data/
```

### Unmounting:

```bash
# Unmount a partition
sudo umount /mnt/data
# Note: it's "umount" NOT "unmount" (yes, the 'n' is missing — classic Linux 😄)

# If you get "device is busy" error:
sudo umount -l /mnt/data
# -l = lazy unmount (detaches filesystem, cleans up when it's no longer busy)

# Force unmount (use only if lazy doesn't work)
sudo umount -f /mnt/data

# Find who's using the mount point:
sudo lsof /mnt/data
# OR
sudo fuser -mv /mnt/data
```

### Making Mounts Permanent with `/etc/fstab`:

When you reboot, all manually created mounts disappear. To make them **permanent**, add them to `/etc/fstab` (**F**ile **S**ystem **TAB**le).

```bash
# View current fstab
cat /etc/fstab
```

Example `/etc/fstab`:

```
# <filesystem>                              <mount>  <type>  <options>        <dump> <pass>
UUID=a1b2c3d4-e5f6-7890-abcd-ef1234567890  /boot    ext4    defaults         0      2
UUID=b2c3d4e5-f6a7-8901-bcde-f12345678901  /        ext4    defaults,errors=remount-ro 0 1
UUID=c3d4e5f6-a7b8-9012-cdef-123456789012  none     swap    sw               0      0
UUID=d4e5f6a7-b8c9-0123-defa-234567890123  /home    ext4    defaults         0      2
```

Let's understand each column:

| Column | Meaning | Example |
|--------|---------|---------|
| **filesystem** | Device or UUID | `UUID=a1b2c3...` or `/dev/sdb1` |
| **mount point** | Where to mount it | `/mnt/data` |
| **type** | Filesystem type | `ext4`, `xfs`, `swap` |
| **options** | Mount options | `defaults`, `noatime`, `ro` |
| **dump** | Backup flag (0=no, 1=yes) | Usually `0` |
| **pass** | fsck order (0=skip, 1=root, 2=others) | Root=`1`, others=`2` |

### Adding a New Permanent Mount:

```bash
# Step 1: Find the UUID of the partition
sudo blkid /dev/sdb1
# Output: /dev/sdb1: UUID="e5f6a7b8-c9d0-1234-efab-567890123456" TYPE="ext4"

# Step 2: Create the mount point
sudo mkdir -p /mnt/data

# Step 3: Add entry to /etc/fstab
# Add this line to /etc/fstab:
UUID=e5f6a7b8-c9d0-1234-efab-567890123456  /mnt/data  ext4  defaults  0  2

# Step 4: Test the fstab entry (IMPORTANT — a bad fstab can prevent boot!)
sudo mount -a
# This mounts everything in fstab that isn't already mounted
# If no errors, you're good!

# Step 5: Verify
df -h | grep data
```

> ⚠️ **WARNING:** Always use `UUID` instead of device names (`/dev/sdb1`) in `/etc/fstab`. Device names can change across reboots (sdb can become sdc), but UUIDs never change!

### Common Mount Options:

| Option | Meaning |
|--------|---------|
| `defaults` | rw, suid, dev, exec, auto, nouser, async (standard options) |
| `ro` | Read-only |
| `rw` | Read-write (default) |
| `noexec` | Can't execute binaries (security) |
| `nosuid` | Ignore SUID/SGID bits (security) |
| `noatime` | Don't update access time (performance boost for SSDs) |
| `nofail` | Don't fail boot if this device is missing |
| `auto` | Mount automatically at boot (default) |
| `noauto` | Don't mount at boot (manual only) |

## 🔧 5. `fdisk` — Partition Management

`fdisk` is the tool for **creating, deleting, and managing partitions** on a disk. It's a low-level tool — you use it when setting up a new disk.

> ⚠️ **WARNING:** `fdisk` can DESTROY DATA. Always make sure you're working on the right disk!

### The Typical Workflow — Adding a New Disk:

Let's say you added a new 100 GB disk to your server (maybe an EBS volume in AWS). Here's the full process:

```bash
# Step 1: See the new disk
lsblk
# You see:
# sda      500G  ← existing disk
# sdb      100G  ← NEW disk (no partitions!)

# Step 2: Partition the new disk
sudo fdisk /dev/sdb
```

`fdisk` opens an interactive prompt. Here are the commands:

```
Command (m for help): m          ← show help

Commands:
  n   add a new partition
  d   delete a partition
  p   print the partition table
  t   change partition type
  w   write table to disk and exit
  q   quit without saving

Command (m for help): n          ← create new partition
Partition type:
   p   primary (0 primary, 0 extended, 4 free)
Select: p                        ← primary partition
Partition number (1-4): 1        ← partition 1
First sector (2048-209715199): [Enter]    ← accept default (start)
Last sector: [Enter]             ← accept default (use entire disk)

Command (m for help): p          ← verify the partition
# Shows: /dev/sdb1  100G  Linux

Command (m for help): w          ← WRITE and EXIT
# The partition table has been altered!
```

```bash
# Step 3: Create a filesystem on the new partition
sudo mkfs.ext4 /dev/sdb1
# This formats /dev/sdb1 with ext4 filesystem
# WARNING: This ERASES all data on the partition!

# For XFS filesystem:
# sudo mkfs.xfs /dev/sdb1

# Step 4: Create a mount point
sudo mkdir -p /mnt/data

# Step 5: Mount it
sudo mount /dev/sdb1 /mnt/data

# Step 6: Verify
df -h | grep sdb1
# /dev/sdb1  100G  44K  95G  1% /mnt/data  ✅

# Step 7: Make it permanent (add to fstab)
sudo blkid /dev/sdb1    # get the UUID
# Add to /etc/fstab:
# UUID=xxxxx  /mnt/data  ext4  defaults,nofail  0  2

sudo mount -a            # test fstab
```

### `parted` — Modern Alternative to fdisk:

`parted` handles **GPT partition tables** (needed for disks larger than 2 TB) and is more modern:

```bash
# List partitions
sudo parted /dev/sdb print

# Create a partition (non-interactive)
sudo parted /dev/sdb mklabel gpt
sudo parted /dev/sdb mkpart primary ext4 0% 100%

# Then create filesystem
sudo mkfs.ext4 /dev/sdb1
```

## 🏗️ 6. LVM — Logical Volume Manager

LVM is the **most powerful disk management feature** in Linux. It adds a layer of abstraction between physical disks and filesystems, giving you **flexibility to resize, extend, and manage storage** without downtime.

### Why LVM Exists (The Problem):

```
WITHOUT LVM:
  You have a 50 GB partition for /var
  It fills up to 100%
  
  To resize it you need to:
    1. Shut down the server ❌
    2. Boot from a live CD
    3. Resize the partition
    4. Resize the filesystem
    5. Pray nothing breaks 🙏
    6. Reboot
    
  Total downtime: 30-60 minutes 😱

WITH LVM:
  You have a 50 GB logical volume for /var
  It fills up to 100%
  
  To resize:
    1. Add a new disk (or use free space)
    2. lvextend -L +20G /dev/vg_data/lv_var
    3. resize2fs /dev/vg_data/lv_var
    
  Total downtime: 0 minutes! ✅ (done while system is running)
```

### LVM Architecture — Three Layers:

```
┌─────────────────────────────────────────┐
│          Logical Volumes (LV)           │  ← What you mount (like /var, /home)
│  ┌──────────┐  ┌──────────┐            │
│  │ lv_root  │  │ lv_data  │            │
│  │  30 GB   │  │  100 GB  │            │
│  └──────────┘  └──────────┘            │
├─────────────────────────────────────────┤
│          Volume Group (VG)              │  ← A pool of storage
│             vg_main                     │
│              200 GB                     │
├─────────────────────────────────────────┤
│        Physical Volumes (PV)            │  ← Actual disks/partitions
│  ┌──────────┐  ┌──────────┐            │
│  │ /dev/sda │  │ /dev/sdb │            │
│  │  100 GB  │  │  100 GB  │            │
│  └──────────┘  └──────────┘            │
└─────────────────────────────────────────┘
```

Think of it like this:

```
Physical Volumes (PV) = Individual water tanks (your disks)
Volume Group (VG)     = Connecting all tanks together into one big pool
Logical Volumes (LV)  = Pipes drawing water from the pool to different places
```

The beauty: you can **add more tanks (disks) to the pool** and **extend any pipe (logical volume)** without stopping anything!

### LVM Terminology:

| Term | Abbreviation | Meaning |
|------|-------------|---------|
| Physical Volume | PV | A disk or partition prepared for LVM |
| Volume Group | VG | A pool made from one or more PVs |
| Logical Volume | LV | A "virtual partition" carved from a VG |
| Physical Extent | PE | The smallest unit of allocation (default 4 MB) |

### Setting Up LVM — Complete Walkthrough:

Let's say you have two new 100 GB disks (`/dev/sdb` and `/dev/sdc`) and want to create a 180 GB logical volume for `/data`.

```bash
# Step 1: Create Physical Volumes (PV)
sudo pvcreate /dev/sdb /dev/sdc

# Verify:
sudo pvs
# Output:
#  PV         VG    Fmt  Attr PSize   PFree
#  /dev/sdb         lvm2 ---  100.00g 100.00g
#  /dev/sdc         lvm2 ---  100.00g 100.00g

# More detailed view:
sudo pvdisplay
```

```bash
# Step 2: Create a Volume Group (VG) from both PVs
sudo vgcreate vg_data /dev/sdb /dev/sdc

# Verify:
sudo vgs
# Output:
#  VG       #PV #LV #SN Attr   VSize   VFree
#  vg_data    2   0   0 wz--n- 199.99g 199.99g

# 200 GB total! (100 + 100) — the two disks are now one pool
```

```bash
# Step 3: Create a Logical Volume (LV)
sudo lvcreate -L 180G -n lv_data vg_data
# -L 180G    = size 180 GB
# -n lv_data = name the LV "lv_data"
# vg_data    = create it from the volume group "vg_data"

# Verify:
sudo lvs
# Output:
#  LV      VG      Attr       LSize   Pool Origin Data%  Meta%
#  lv_data vg_data -wi-a----- 180.00g

# Your LV is at: /dev/vg_data/lv_data
```

```bash
# Step 4: Create a filesystem on the LV
sudo mkfs.ext4 /dev/vg_data/lv_data

# Step 5: Mount it
sudo mkdir -p /data
sudo mount /dev/vg_data/lv_data /data

# Step 6: Verify
df -h | grep data
# /dev/mapper/vg_data-lv_data  180G  44K  171G  1% /data  ✅

# Step 7: Make it permanent
echo '/dev/vg_data/lv_data  /data  ext4  defaults  0  2' | sudo tee -a /etc/fstab
sudo mount -a
```

### Extending a Logical Volume (The Real Magic!):

Your `/data` is getting full. You have 20 GB free in the volume group. Let's add it:

```bash
# Check free space in the VG
sudo vgs
# VFree = 19.99g  ← we have 20 GB free

# Extend the LV by 20 GB
sudo lvextend -L +20G /dev/vg_data/lv_data
# +20G means ADD 20 GB (total becomes 200 GB)

# Now resize the filesystem to use the new space
sudo resize2fs /dev/vg_data/lv_data   # for ext4
# OR
sudo xfs_growfs /data                  # for xfs

# Verify
df -h | grep data
# /dev/mapper/vg_data-lv_data  200G  170G  30G  85% /data  ✅
# Notice: size went from 180G to 200G!
```

### What If You Need Even More Space?

Add a new disk to the VG!

```bash
# You attached a new 200 GB disk: /dev/sdd

# Step 1: Create a PV
sudo pvcreate /dev/sdd

# Step 2: Add it to the existing VG
sudo vgextend vg_data /dev/sdd

# Step 3: Check — now you have 200 GB more
sudo vgs
# VFree = 200g

# Step 4: Extend the LV
sudo lvextend -L +200G /dev/vg_data/lv_data

# Step 5: Resize the filesystem
sudo resize2fs /dev/vg_data/lv_data

# Done! Your /data went from 200G to 400G — with ZERO downtime! 🎉
```

### Shrinking a Logical Volume (ext4 only — XFS can't shrink!):

```bash
# ⚠️ WARNING: Back up data first! Shrinking is risky!

# Step 1: Unmount the filesystem
sudo umount /data

# Step 2: Check the filesystem
sudo e2fsck -f /dev/vg_data/lv_data

# Step 3: Shrink the filesystem first
sudo resize2fs /dev/vg_data/lv_data 100G

# Step 4: Then shrink the LV
sudo lvreduce -L 100G /dev/vg_data/lv_data

# Step 5: Remount
sudo mount /dev/vg_data/lv_data /data
```

> ⚠️ **IMPORTANT:** Always shrink the **filesystem first**, then the LV. If you shrink the LV first, you'll corrupt the filesystem!

### LVM One-Liner — Extend with Filesystem Resize in One Command:

```bash
# Extend LV and resize filesystem in one step
sudo lvextend -L +20G --resizefs /dev/vg_data/lv_data
# The --resizefs flag handles both steps automatically!
```

### LVM Command Reference:

| Layer | Create | Display (short) | Display (detail) | Remove |
|-------|--------|-----------------|-------------------|--------|
| **PV** | `pvcreate /dev/sdb` | `pvs` | `pvdisplay` | `pvremove /dev/sdb` |
| **VG** | `vgcreate vg01 /dev/sdb` | `vgs` | `vgdisplay` | `vgremove vg01` |
| **LV** | `lvcreate -L 50G -n lv01 vg01` | `lvs` | `lvdisplay` | `lvremove /dev/vg01/lv01` |

| Operation | Command |
|-----------|---------|
| Extend VG | `vgextend vg01 /dev/sdc` |
| Extend LV | `lvextend -L +20G /dev/vg01/lv01` |
| Extend LV to fill all free space | `lvextend -l +100%FREE /dev/vg01/lv01` |
| Resize ext4 filesystem | `resize2fs /dev/vg01/lv01` |
| Resize XFS filesystem | `xfs_growfs /mountpoint` |
| Extend + resize in one step | `lvextend -L +20G --resizefs /dev/vg01/lv01` |

## 🏢 Real-World DevOps Scenarios

### Scenario 1: AWS EC2 — Expanding the Root Volume

```bash
# You expanded the EBS volume from 20G to 50G in AWS Console
# But the OS still shows 20G. You need to:

# Step 1: Check current layout
lsblk
# xvda     50G
# └─xvda1  20G   ← partition is still 20G!

# Step 2: Grow the partition
sudo growpart /dev/xvda 1
# Now xvda1 is 50G

# Step 3: Resize the filesystem
sudo resize2fs /dev/xvda1    # for ext4
# OR
sudo xfs_growfs /             # for xfs

# Step 4: Verify
df -h
# /dev/xvda1  50G  ← ✅
```

### Scenario 2: Finding What's Eating Disk Space

```bash
# Quick investigation workflow:
df -h                                                    # Which partition is full?
sudo du -h --max-depth=1 / 2>/dev/null | sort -rh | head # Top offenders
sudo du -h --max-depth=1 /var | sort -rh | head          # Drill down
sudo find /var/log -type f -size +100M -exec ls -lh {} \; # Find big files
```

### Scenario 3: Docker Eating All Disk Space

```bash
# Docker images, containers, and volumes can fill your disk fast!

# Check Docker disk usage
docker system df

# Clean up unused Docker data
docker system prune -af --volumes
# This removes:
#   - All stopped containers
#   - All unused images
#   - All unused volumes
#   - All unused networks
```
