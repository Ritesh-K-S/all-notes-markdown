## 1. What is a CDN?

A **Content Delivery Network (CDN)** is a geographically distributed network of servers that delivers web content to users based on their location.

Think of it like this:

> Imagine you own a pizza shop in New York. A customer in Tokyo orders a pizza. Shipping it from New York would take forever, and the pizza would arrive cold. **A CDN is like having a branch of your pizza shop in Tokyo** — the customer gets a fresh, hot pizza quickly because it's served locally.

In technical terms:
- Your **origin server** (the main server) sits in one location (say, US-East).
- The CDN places **copies of your content** on servers all around the world (called **edge servers** or **Points of Presence / PoPs**).
- When a user in India requests your website, the CDN serves the content from the nearest edge server (say, Mumbai) instead of the origin server in the US.

**Result?** Faster load times, less load on your origin server, and a better user experience.

## 2. Why Do We Need CDNs?

### The Problem Without a CDN

```
User in Australia ──────────────── (12,000 km) ──────────────── Origin Server in USA
                                                                 
    ⏱️ High latency (200-400 ms round trip)
    📦 All traffic hits one server
    💥 Server overload during traffic spikes
    🌐 Poor experience for global users
```

### The Solution With a CDN

```
User in Australia ──── (50 km) ──── Edge Server in Sydney ──── Origin Server in USA
                                                                 
    ⏱️ Low latency (5-20 ms)
    📦 Traffic distributed across edge servers
    💥 Can handle massive traffic spikes
    🌐 Great experience worldwide
```

### Key Benefits

| Benefit | Explanation |
|---------|-------------|
| **Speed** | Content is served from the closest server → faster page loads |
| **Reduced Latency** | Less physical distance = fewer network hops = lower latency |
| **Scalability** | CDN handles traffic spikes (Black Friday, viral content) without crashing your origin |
| **Reliability** | If one edge server goes down, traffic is rerouted to the next closest one |
| **Bandwidth Savings** | Origin server sends content once to the CDN; CDN handles all the user requests |
| **Security** | CDNs provide DDoS protection, WAF (Web Application Firewall), and SSL/TLS |
| **SEO Boost** | Google ranks faster sites higher — CDN directly helps your SEO |

## 3. How a CDN Works — Step by Step

Let's walk through what happens when a user visits `www.example.com` with a CDN in front:

### Step-by-Step Flow

```
1. User types www.example.com in their browser
                    │
                    ▼
2. DNS resolves www.example.com → CDN's edge server IP (not origin IP)
                    │
                    ▼
3. CDN checks: "Do I have this content cached on this edge server?"
                    │
            ┌───────┴───────┐
            ▼               ▼
     4a. CACHE HIT      4b. CACHE MISS
     (Content found!)   (Content not found)
            │               │
            ▼               ▼
     Serve directly    Fetch from origin server
     to the user       ──► Cache it on edge
                       ──► Serve to user
                    │
                    ▼
5. Next request for the same content → CACHE HIT (fast!)
```

### What is a Cache Hit vs Cache Miss?

- **Cache Hit:** The edge server already has a copy of the requested content → serves it instantly. ✅
- **Cache Miss:** The edge server does NOT have the content → it goes to the origin, fetches it, caches it, then serves it. ❌ → ✅ (next time it'll be a hit)

### Cache Hit Ratio

```
Cache Hit Ratio = (Cache Hits / Total Requests) × 100

Example: 950 hits out of 1000 requests = 95% cache hit ratio
```

A good CDN setup aims for **90-99%** cache hit ratio.

## 4. Key CDN Concepts

### 4.1 Origin Server

The **original server** where your content lives. This is your actual web server (e.g., an EC2 instance, an S3 bucket, your Nginx server, etc.).

### 4.2 Edge Server (Edge Location)

A server placed at a **Point of Presence (PoP)** close to the end users. It caches and serves content.

### 4.3 Point of Presence (PoP)

A **physical location** (usually a data center) where edge servers are placed. Major CDNs have 200-300+ PoPs worldwide.

```
Example PoP locations:
├── North America: New York, Los Angeles, Chicago, Dallas, Toronto
├── Europe: London, Frankfurt, Paris, Amsterdam, Stockholm
├── Asia: Tokyo, Singapore, Mumbai, Seoul, Hong Kong
├── South America: São Paulo, Buenos Aires
├── Oceania: Sydney, Melbourne
└── Africa: Johannesburg, Cape Town
```

### 4.4 TTL (Time to Live)

How long the CDN should keep a cached copy before checking the origin for a fresh version.

```
TTL = 3600  →  Cache for 1 hour
TTL = 86400 →  Cache for 24 hours
TTL = 0     →  Don't cache (always fetch from origin)
```

### 4.5 Cache Invalidation / Purge

Manually telling the CDN to **delete** its cached copy so it fetches fresh content from the origin.

```bash
# AWS CloudFront — Invalidate a specific path
aws cloudfront create-invalidation \
  --distribution-id E1A2B3C4D5E6F7 \
  --paths "/images/logo.png" "/css/style.css"

# Invalidate everything
aws cloudfront create-invalidation \
  --distribution-id E1A2B3C4D5E6F7 \
  --paths "/*"
```

### 4.6 Anycast

A networking technique where **multiple servers share the same IP address**, and the network routes the user to the **nearest one**. CDNs like Cloudflare use Anycast heavily.

```
Same IP address: 104.16.132.229

User in Tokyo     → Routed to Tokyo PoP
User in London    → Routed to London PoP
User in São Paulo → Routed to São Paulo PoP
```

## 6. CDN Architecture

### Pull CDN vs Push CDN

| Feature | Pull CDN | Push CDN |
|---------|----------|----------|
| **How it works** | Edge server **pulls** content from origin on first request | You **push** content to the CDN manually |
| **When content is cached** | On first user request (lazy loading) | Before any user requests (proactive) |
| **Best for** | Websites with many pages, unpredictable traffic | Large files (videos, software downloads) |
| **Example** | Cloudflare, most CDN setups | AWS S3 + CloudFront with pre-warming |
| **Pros** | Simple to set up, automatic | Guaranteed availability from the start |
| **Cons** | First request is slow (cache miss) | Requires manual management |

### CDN Architecture Diagram

```
                    ┌─────────────────────┐
                    │   Origin Server     │
                    │  (Your main server) │
                    └──────────┬──────────┘
                               │
                    ┌──────────┴──────────┐
                    │   CDN Core Network  │
                    │ (Smart routing,     │
                    │  load balancing)    │
                    └──────────┬──────────┘
                               │
          ┌────────────────────┼────────────────────┐
          │                    │                    │
   ┌──────┴──────┐     ┌──────┴──────┐     ┌──────┴──────┐
   │  PoP: USA   │     │ PoP: Europe │     │  PoP: Asia  │
   │  Edge Server│     │ Edge Server │     │ Edge Server │
   └──────┬──────┘     └──────┬──────┘     └──────┬──────┘
          │                    │                    │
     Users in USA       Users in Europe       Users in Asia
```

## 7. Caching in CDNs

### How CDN Caching Works

The CDN decides what to cache and for how long using **HTTP Cache Headers** sent by your origin server.

### Important Cache Headers

```http
# Cache-Control — The most important header
Cache-Control: public, max-age=86400
# "public" = CDN can cache it
# "max-age=86400" = cache for 24 hours (in seconds)

Cache-Control: private, no-cache
# "private" = Only the user's browser can cache, NOT the CDN
# "no-cache" = Always check with origin before serving

Cache-Control: no-store
# Don't cache at all — not even the browser

# ETag — A fingerprint of the content
ETag: "abc123def456"
# If the ETag changes, the CDN knows the content has been updated

# Expires — Older way to set cache expiry
Expires: Thu, 01 Jan 2027 00:00:00 GMT

# Last-Modified — When the content was last changed
Last-Modified: Mon, 15 May 2026 12:00:00 GMT
```

### Cache-Control Cheat Sheet

| Directive | Meaning |
|-----------|---------|
| `public` | Any cache (CDN, browser) can store this |
| `private` | Only the user's browser can cache (not the CDN) |
| `max-age=N` | Cache for N seconds |
| `s-maxage=N` | Like max-age, but specifically for CDN/shared caches |
| `no-cache` | Cache it, but always revalidate with origin before serving |
| `no-store` | Don't cache at all |
| `must-revalidate` | Once expired, must check with origin (don't serve stale) |
| `stale-while-revalidate=N` | Serve stale content for N seconds while fetching fresh copy in background |

### Setting Cache Headers in Nginx

```nginx
server {
    listen 80;
    server_name example.com;

    # Static assets — cache for 1 year
    location ~* \.(css|js|png|jpg|jpeg|gif|svg|woff2|ico)$ {
        expires 1y;
        add_header Cache-Control "public, max-age=31536000, immutable";
    }

    # HTML files — don't cache (always serve fresh)
    location ~* \.html$ {
        add_header Cache-Control "no-cache, no-store, must-revalidate";
    }

    # API responses — cache for 5 minutes on CDN
    location /api/ {
        add_header Cache-Control "public, s-maxage=300, max-age=0";
    }
}
```

### Cache Key

The CDN uses a **cache key** to identify unique content. By default, the cache key is the **URL**.

```
Cache Key: https://example.com/images/logo.png
```

But you can customize the cache key to include:
- Query strings: `?version=2`
- Headers: `Accept-Language: en`
- Cookies: `user_region=asia`

## 8. CDN & DNS — How They Work Together

CDNs rely heavily on DNS to route users to the nearest edge server.

### How DNS Routes to CDN

```
1. User types: www.example.com
2. DNS lookup happens:
   
   www.example.com → CNAME → cdn.example.cloudfront.net
                                    │
                                    ▼
                     CloudFront DNS resolves to nearest edge IP
                                    │
                                    ▼
                     Edge Server IP: 13.225.78.42 (nearest PoP)
```

### DNS Record Setup for CDN

```dns
; Without CDN (direct to origin)
www.example.com    A     203.0.113.50

; With CDN — CNAME to CDN provider
www.example.com    CNAME d1234abcdef.cloudfront.net

; With Cloudflare — Proxy mode (orange cloud)
www.example.com    A     203.0.113.50   ; Cloudflare proxies this automatically
```

### GeoDNS

Some CDNs use **GeoDNS** — DNS that returns **different IPs** based on the user's geographic location:

```
User in USA    → DNS returns 192.0.2.1  (US edge server)
User in Europe → DNS returns 192.0.2.2  (EU edge server)
User in Asia   → DNS returns 192.0.2.3  (Asia edge server)
```

## 10. AWS CloudFront — Deep Dive

### What is CloudFront?

AWS CloudFront is Amazon's CDN service. It integrates tightly with other AWS services like S3, EC2, ALB, API Gateway, and Lambda.

### Key CloudFront Concepts

```
┌─────────────────────────────────────────────────────────────┐
│                     CloudFront Distribution                 │
│                                                             │
│  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐  │
│  │  Behavior 1  │    │  Behavior 2  │    │  Behavior 3  │  │
│  │  /images/*   │    │  /api/*      │    │  /* (default) │  │
│  │  → S3 Bucket │    │  → ALB       │    │  → EC2       │  │
│  └──────────────┘    └──────────────┘    └──────────────┘  │
│                                                             │
│  Origins: S3, ALB, EC2, API Gateway, Custom HTTP Server    │
│  Edge Locations: 400+ worldwide                             │
└─────────────────────────────────────────────────────────────┘
```

### CloudFront Distribution Components

| Component | Description |
|-----------|-------------|
| **Distribution** | The main CDN configuration — maps to a domain name |
| **Origin** | Where CloudFront fetches content from (S3, ALB, EC2, etc.) |
| **Behavior** | Rules for how CloudFront handles requests (path pattern → origin) |
| **Edge Location** | Server that caches and serves content (400+ worldwide) |
| **Regional Edge Cache** | A larger cache between edge locations and the origin |

### Setting Up CloudFront with S3 (AWS CLI)

```bash
# Step 1: Create an S3 bucket for your static site
aws s3 mb s3://my-awesome-website-2026

# Step 2: Upload your website files
aws s3 sync ./my-website/ s3://my-awesome-website-2026/ \
  --cache-control "public, max-age=86400"

# Step 3: Create a CloudFront distribution
aws cloudfront create-distribution \
  --origin-domain-name my-awesome-website-2026.s3.amazonaws.com \
  --default-root-object index.html
```

### CloudFront with S3 — Origin Access Control (OAC)

To make your S3 bucket **private** but still accessible through CloudFront:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCloudFrontAccess",
      "Effect": "Allow",
      "Principal": {
        "Service": "cloudfront.amazonaws.com"
      },
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-awesome-website-2026/*",
      "Condition": {
        "StringEquals": {
          "AWS:SourceArn": "arn:aws:cloudfront::123456789012:distribution/E1A2B3C4D5E6F7"
        }
      }
    }
  ]
}
```

### Lambda@Edge — Run Code at the Edge

Lambda@Edge lets you run AWS Lambda functions at CloudFront edge locations. This is powerful for:

- URL rewrites / redirects
- A/B testing
- Authentication at the edge
- Adding security headers
- Personalizing content

```javascript
// Lambda@Edge function — Add security headers
exports.handler = async (event) => {
    const response = event.Records[0].cf.response;
    const headers = response.headers;

    // Add security headers
    headers['strict-transport-security'] = [{
        key: 'Strict-Transport-Security',
        value: 'max-age=63072000; includeSubDomains; preload'
    }];
    
    headers['x-content-type-options'] = [{
        key: 'X-Content-Type-Options',
        value: 'nosniff'
    }];
    
    headers['x-frame-options'] = [{
        key: 'X-Frame-Options',
        value: 'DENY'
    }];
    
    headers['content-security-policy'] = [{
        key: 'Content-Security-Policy',
        value: "default-src 'self'"
    }];

    return response;
};
```

## 11. Cloudflare — Deep Dive

### What is Cloudflare?

Cloudflare is a CDN + Security + Performance company. It's popular because:
- **Generous free tier** (unlimited bandwidth!)
- Built-in **DDoS protection**
- **DNS hosting** (fastest public DNS resolver: 1.1.1.1)
- **Cloudflare Workers** (serverless edge compute)

### How Cloudflare Works

Unlike traditional CDNs, Cloudflare acts as a **reverse proxy** — ALL traffic goes through Cloudflare:

```
              Traditional CDN:
User → DNS → Edge Server (cached content) or → Origin

              Cloudflare:
User → DNS → Cloudflare Proxy → (cached? serve!) or → Origin
              │
              ├── DDoS protection
              ├── WAF (firewall)
              ├── SSL/TLS termination
              ├── Bot protection
              └── Performance optimization
```

### Setting Up Cloudflare

```
Step 1: Sign up at cloudflare.com
Step 2: Add your domain (e.g., example.com)
Step 3: Cloudflare scans your DNS records
Step 4: Change your nameservers to Cloudflare's:
         - ada.ns.cloudflare.com
         - bob.ns.cloudflare.com
Step 5: Configure caching, security, and performance settings
Step 6: Done! All traffic now flows through Cloudflare.
```

### Cloudflare Proxy Modes

```
☁️ DNS Only (Gray Cloud):
   - Cloudflare only provides DNS resolution
   - No CDN, no security features
   - User → Your Origin Server directly

🟠 Proxied (Orange Cloud):
   - All traffic flows through Cloudflare
   - CDN caching + DDoS protection + WAF + SSL
   - User → Cloudflare → Your Origin Server
```

### Cloudflare Page Rules (Legacy) / Cache Rules

```
Example rules:

1. Cache everything for /static/*
   URL: example.com/static/*
   Setting: Cache Level = Cache Everything, Edge TTL = 1 month

2. Bypass cache for /admin/*
   URL: example.com/admin/*
   Setting: Cache Level = Bypass

3. Always use HTTPS
   URL: http://example.com/*
   Setting: Always Use HTTPS
```

### Cloudflare Workers — Edge Computing

Cloudflare Workers let you run JavaScript/TypeScript at the edge (in every Cloudflare PoP):

```javascript
// worker.js — A/B testing at the edge
addEventListener('fetch', event => {
    event.respondWith(handleRequest(event.request));
});

async function handleRequest(request) {
    const url = new URL(request.url);
    
    // Simple A/B test — 50/50 split
    const variant = Math.random() < 0.5 ? 'A' : 'B';
    
    if (variant === 'A') {
        // Serve version A
        url.pathname = '/landing-page-a.html';
    } else {
        // Serve version B
        url.pathname = '/landing-page-b.html';
    }
    
    const response = await fetch(url.toString());
    
    // Add a header so we know which variant was served
    const newResponse = new Response(response.body, response);
    newResponse.headers.set('X-AB-Variant', variant);
    
    return newResponse;
}
```

## 13. CDN Security Features

CDNs provide several built-in security features:

### DDoS Protection

```
Normal Traffic:       1,000 requests/second → Your Server handles it ✅

DDoS Attack:     1,000,000 requests/second → Your Server crashes 💥

With CDN:        1,000,000 requests/second → CDN absorbs the attack
                 Only legitimate traffic reaches your server ✅
```

### Web Application Firewall (WAF)

A WAF inspects HTTP requests and blocks malicious ones:

```
Blocked by WAF:
├── SQL Injection:     ' OR 1=1 --
├── XSS:               <script>alert('hack')</script>
├── Path Traversal:    ../../etc/passwd
├── Command Injection: ; rm -rf /
└── Bad Bots:          Scrapers, credential stuffers
```

### SSL/TLS Termination

The CDN handles SSL/TLS encryption so your origin doesn't have to:

```
User ──── HTTPS ────► CDN Edge ──── HTTP (or HTTPS) ────► Origin

Benefits:
├── CDN handles certificate management
├── Faster SSL handshake (edge is closer to user)
├── Reduced CPU load on origin server
└── Free SSL certificates (Cloudflare, Let's Encrypt)
```

### Bot Protection

```
Good Bots (allow):          Bad Bots (block):
├── Googlebot               ├── Scrapers
├── Bingbot                 ├── Credential stuffers
├── Social media crawlers   ├── Inventory hoarders
└── Monitoring tools        └── DDoS bots
```

### Signed URLs / Signed Cookies

Restrict access to CDN content — only users with a valid signature can access it:

```bash

  --url "https://d1234.cloudfront.net/premium-video.mp4" \

```

## 14. CDN Performance Optimization

### Best Practices for CDN Performance

#### 1. Optimize Cache Hit Ratio

```
Goal: 95%+ cache hit ratio

How:
├── Set proper Cache-Control headers
├── Use consistent URLs (avoid random query strings)
├── Use versioned filenames instead of cache busting
│   ❌ style.css?v=random123
│   ✅ style.abc123.css (content hash in filename)
└── Set long TTLs for static assets
```

#### 2. Enable Compression

```nginx
# Nginx — Enable gzip compression
gzip on;
gzip_types text/plain text/css application/json application/javascript 
           text/xml application/xml application/xml+rss text/javascript
           image/svg+xml;
gzip_min_length 256;
gzip_comp_level 6;
```

```
# Brotli (better than gzip — 15-25% smaller)
# Most CDNs support Brotli automatically
# Cloudflare enables it by default
```

#### 3. Use HTTP/2 or HTTP/3

```
HTTP/1.1: One request at a time per connection (slow)
HTTP/2:   Multiple requests on one connection (multiplexing) — fast!
HTTP/3:   Uses QUIC (UDP-based) — even faster, especially on mobile

Most CDNs support HTTP/2 by default. Cloudflare supports HTTP/3.
```

#### 4. Image Optimization

```
CDN image optimization features:
├── Auto WebP/AVIF conversion (smaller file sizes)
├── Responsive images (serve different sizes based on device)
├── Lazy loading
└── Quality adjustment
```

#### 5. Prefetching & Preloading

```html
<!-- DNS Prefetch — resolve CDN domain early -->
<link rel="dns-prefetch" href="https://cdn.example.com">

<!-- Preconnect — establish connection early -->
<link rel="preconnect" href="https://cdn.example.com" crossorigin>

<!-- Preload — fetch critical resources early -->
<link rel="preload" href="https://cdn.example.com/fonts/main.woff2" 
      as="font" type="font/woff2" crossorigin>
```

## 15. CDN in DevOps Workflows

### CDN + CI/CD Pipeline

```yaml
# GitHub Actions — Deploy to S3 + Invalidate CloudFront
name: Deploy Website

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Build
        run: npm run build
      
      - name: Deploy to S3
        run: |
          aws s3 sync ./build/ s3://my-website-bucket/ \
            --delete \
            --cache-control "public, max-age=31536000, immutable"
          
          # HTML files — short cache
          aws s3 cp ./build/index.html s3://my-website-bucket/ \
            --cache-control "no-cache"
      
      - name: Invalidate CloudFront Cache
        run: |
          aws cloudfront create-invalidation \
            --distribution-id ${{ secrets.CLOUDFRONT_DIST_ID }} \
            --paths "/index.html" "/manifest.json"
```

### CDN + Terraform (Infrastructure as Code)

```hcl
# Terraform — CloudFront Distribution
resource "aws_cloudfront_distribution" "website" {
  enabled             = true
  default_root_object = "index.html"
  
  origin {
    domain_name = aws_s3_bucket.website.bucket_regional_domain_name
    origin_id   = "S3-Website"
    
    origin_access_control_id = aws_cloudfront_origin_access_control.oac.id
  }
  
  default_cache_behavior {
    allowed_methods  = ["GET", "HEAD"]
    cached_methods   = ["GET", "HEAD"]
    target_origin_id = "S3-Website"
    
    forwarded_values {
      query_string = false
      cookies {
        forward = "none"
      }
    }
    
    viewer_protocol_policy = "redirect-to-https"
    min_ttl                = 0
    default_ttl            = 86400    # 1 day
    max_ttl                = 31536000 # 1 year
    compress               = true
  }
  
  restrictions {
    geo_restriction {
      restriction_type = "none"
    }
  }
  
  viewer_certificate {
    cloudfront_default_certificate = true
  }
  
  tags = {
    Environment = "production"
    ManagedBy   = "terraform"
  }
}
```

## 17. CDN Troubleshooting

### Common CDN Issues & Fixes

| Problem | Cause | Fix |
|---------|-------|-----|
| Content not updating | Stale cache | Invalidate/purge cache, reduce TTL |
| CORS errors | CDN stripping headers | Configure CDN to forward Origin header |
| 403 Forbidden | Origin access misconfigured | Check OAC/OAI settings, bucket policy |
| Mixed content warnings | HTTP resources on HTTPS page | Update all URLs to HTTPS |
| High origin load | Low cache hit ratio | Check Cache-Control headers, increase TTL |
| Slow first load | Cache miss (cold cache) | Pre-warm cache, use push CDN |

### Debugging CDN with Headers

```bash
# Check if content is served from CDN cache
curl -I https://example.com/style.css

# Look for these headers:
# CloudFront:
#   X-Cache: Hit from cloudfront     ← Cache hit!
#   X-Cache: Miss from cloudfront    ← Cache miss
#   X-Amz-Cf-Pop: IAD89             ← Edge location (Washington DC)

# Cloudflare:
#   CF-Cache-Status: HIT             ← Cache hit!
#   CF-Cache-Status: MISS            ← Cache miss
#   CF-Cache-Status: DYNAMIC         ← Not cached (dynamic content)
#   CF-Cache-Status: BYPASS          ← Cache bypassed (by rule)
#   CF-Ray: 7f1234567890abcd-IAD     ← Request ID + PoP location

# General:
#   Age: 3600                         ← Content has been cached for 3600 seconds
#   Cache-Control: public, max-age=86400
```

## 18. When NOT to Use a CDN

CDNs aren't always the answer. Skip the CDN when:

```
❌ Your users are all in one geographic location
   → A server in the same region is fast enough

❌ All your content is highly dynamic and personalized
   → CDN caching won't help much (though edge compute can)

❌ You're dealing with real-time data (WebSocket-heavy apps)
   → CDN adds an extra hop; direct connection may be faster

❌ Your site has very low traffic
   → CDN overhead isn't worth it for 100 visitors/day

❌ You're in development/staging
   → CDN caching makes debugging harder

❌ Regulatory requirements demand data stays in one region
   → CDN distributes data globally (some offer geo-restrictions though)
```

## 19. Summary

```
CDN Cheat Sheet:

What:     Network of edge servers that cache and serve content close to users
Why:      Speed, scalability, reliability, security
How:      DNS routes user to nearest edge → cache hit (fast) or cache miss (fetch from origin)

Key Concepts:
├── Origin Server     → Your main server
├── Edge Server       → CDN server near users
├── PoP               → Physical location of edge servers
├── TTL               → How long to cache
├── Cache Invalidation→ Force CDN to fetch fresh content
└── Anycast           → Same IP, nearest server

Providers:
├── CloudFront → Best for AWS users
├── Cloudflare → Best free tier + security
├── Akamai     → Best for enterprises
└── Fastly     → Best for dynamic content

Cache Headers:
├── Cache-Control: public, max-age=86400
├── ETag: "fingerprint"
├── Expires: <date>
└── Last-Modified: <date>

DevOps Integration:
├── CI/CD → Deploy + invalidate cache
├── IaC   → Terraform / CloudFormation for CDN setup
└── Monitoring → Cache hit ratio, latency, error rates
```
