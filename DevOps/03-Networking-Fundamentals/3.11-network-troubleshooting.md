## 1. Network Troubleshooting Mindset

Before you start typing commands, think systematically. Network issues can happen at any layer, so we follow a **bottom-up** or **top-down** approach.

### The Troubleshooting Layers

```
Layer 7 — Application    : Is the app responding? (curl, browser)
Layer 4 — Transport      : Is the port open? (ss, netstat, telnet)
Layer 3 — Network        : Can I reach the IP? (ping, traceroute)
Layer 2 — Data Link      : Is the MAC/ARP working? (arp, ip neigh)
Layer 1 — Physical       : Is the cable plugged in? (ethtool, ip link)
```

### Golden Rule of Troubleshooting

```
1. Can I reach myself?          → ping localhost / ping 127.0.0.1
2. Can I reach my gateway?      → ping <gateway-ip>
3. Can I reach the internet?    → ping 8.8.8.8
4. Can I resolve DNS?           → ping google.com / nslookup google.com
5. Can I reach the target?      → ping <target-ip>
6. Can I reach the service?     → curl http://<target>:<port>
```

If it fails at step 3, you have a **routing or connectivity** issue.  
If it fails at step 4, you have a **DNS** issue.  
If it fails at step 6 but step 5 works, you have a **firewall or application** issue.

## 2. The Troubleshooting Toolkit — Overview

| Tool | What It Does | When to Use |
|------|-------------|-------------|
| `ping` | Tests basic connectivity (ICMP echo) | "Can I reach this host?" |
| `traceroute` | Shows the path packets take | "Where is the packet getting stuck?" |
| `mtr` | Continuous ping + traceroute | "Where is packet loss happening?" |
| `tcpdump` | Captures raw packets | "What exactly is going over the wire?" |
| `Wireshark` | Visual packet analysis (GUI) | "I need to deeply analyze captured packets" |
| `nslookup/dig` | DNS queries | "Is DNS resolving correctly?" |
| `netstat/ss` | Shows active connections and ports | "What's listening on my server?" |
| `curl` | HTTP testing | "Is the web service responding?" |
| `nmap` | Port scanning | "Which ports are open on this host?" |
| `iperf` | Bandwidth testing | "How fast is my network?" |
| `ip/ifconfig` | Network interface info | "What's my IP? Is the interface up?" |
| `arp` | ARP table (IP → MAC mapping) | "Is Layer 2 working?" |

## 3. ping — Is the Host Alive?

### What is ping?

`ping` sends **ICMP Echo Request** packets to a target and waits for **ICMP Echo Reply** packets. It's the simplest and most commonly used network troubleshooting tool.

Think of it like knocking on someone's door:
> "Hey, are you there?" → *knock knock* (ICMP Echo Request)  
> "Yes, I'm here!" → *door opens* (ICMP Echo Reply)

### Basic Usage

```bash
# Ping a host
ping google.com

# Output:
PING google.com (142.250.80.46) 56(84) bytes of data.
64 bytes from lax17s55-in-f14.1e100.net (142.250.80.46): icmp_seq=1 ttl=117 time=12.3 ms
64 bytes from lax17s55-in-f14.1e100.net (142.250.80.46): icmp_seq=2 ttl=117 time=11.8 ms
64 bytes from lax17s55-in-f14.1e100.net (142.250.80.46): icmp_seq=3 ttl=117 time=12.1 ms
^C
--- google.com ping statistics ---
3 packets transmitted, 3 received, 0% packet loss, time 2003ms
rtt min/avg/max/mdev = 11.811/12.073/12.301/0.200 ms
```

### Understanding ping Output

```
64 bytes from 142.250.80.46: icmp_seq=1 ttl=117 time=12.3 ms
│         │                  │           │       │
│         │                  │           │       └── Round Trip Time
│         │                  │           │            (lower = better)
│         │                  │           └── Time to Live (hops remaining)
│         │                  └── Sequence number (increases each packet)
│         └── IP address that replied
└── Size of the reply packet
```

### Key Metrics

| Metric | What It Means | Good Value |
|--------|--------------|------------|
| **time (RTT)** | Round trip time — how long the packet took to go and come back | < 50ms (same continent), < 150ms (intercontinental) |
| **ttl** | Time to Live — max hops before the packet is discarded | 64 (Linux), 128 (Windows), 255 (Network device) |
| **packet loss** | Percentage of packets that didn't get a reply | 0% (ideal), < 1% (acceptable) |
| **mdev/jitter** | Variation in response times | Low is better (consistent connection) |

### Common ping Options

```bash
# Ping a specific number of times (Linux: -c, Windows: -n)
ping -c 5 google.com        # Linux — send 5 pings
ping -n 5 google.com        # Windows — send 5 pings

# Ping with a specific packet size
ping -s 1400 google.com     # Send 1400-byte packets (test MTU issues)

# Ping with no DNS resolution (faster)
ping -n 8.8.8.8             # Don't resolve hostname (Linux uses different flag)

# Flood ping (requires root — sends packets as fast as possible)
sudo ping -f google.com     # ⚠️ Use carefully — can be seen as an attack

# Ping with specific interval
ping -i 0.5 google.com      # Ping every 0.5 seconds (default is 1 second)

# Ping with a timeout
ping -W 2 google.com        # Wait max 2 seconds for each reply

# Ping with timestamp
ping -D google.com           # Show timestamp for each reply
```

### Interpreting ping Failures

```bash
# 1. "Destination Host Unreachable"
#    → The router can't find a path to the host
#    → Check: Is the host up? Is the IP correct? Is there a route?

# 2. "Request timed out"
#    → The host didn't respond within the timeout
#    → Check: Firewall blocking ICMP? Host is down? Network congestion?

# 3. "Name or service not known"
#    → DNS can't resolve the hostname
#    → Check: DNS settings, try pinging the IP directly

# 4. High packet loss (e.g., 30%)
#    → Network congestion, bad cable, faulty network device
#    → Use mtr to find WHERE the loss is happening

# 5. High latency (e.g., 500ms for a local server)
#    → Network congestion, routing issue, overloaded server
#    → Use traceroute to find the slow hop
```

### Important Note About ping

> **Some hosts block ICMP (ping).** For example, some servers, firewalls, and cloud security groups block ICMP. This means `ping` fails, but the service is actually running fine. Don't conclude a host is down just because ping fails — always try other methods like `curl` or `telnet`.

## 4. traceroute / tracert — What Path Do Packets Take?

### What is traceroute?

`traceroute` (Linux/Mac) or `tracert` (Windows) shows you **every router (hop)** your packet passes through on its way to the destination. It helps you find **where** a problem is occurring in the network path.

Think of it like tracking a package:
> Your package goes from Warehouse → Regional Hub → Airport → Destination Airport → Local Post Office → Your Door.  
> `traceroute` shows you each stop along the way and how long the package stayed at each one.

### How traceroute Works (Under the Hood)

```
traceroute uses TTL (Time to Live) cleverly:

Packet 1:  TTL=1 → First router receives it, TTL reaches 0, 
                    router sends back "Time Exceeded" message
                    → Now we know Router 1's IP and latency

Packet 2:  TTL=2 → Passes Router 1, reaches Router 2, TTL=0,
                    Router 2 sends back "Time Exceeded"
                    → Now we know Router 2's IP and latency

Packet 3:  TTL=3 → Passes Router 1 & 2, reaches Router 3...

...continues until we reach the destination.
```

### Basic Usage

```bash
# Linux/Mac
traceroute google.com

# Windows
tracert google.com

# Output:
traceroute to google.com (142.250.80.46), 30 hops max, 60 byte packets
 1  gateway (192.168.1.1)           1.234 ms  1.112 ms  1.089 ms
 2  isp-router.example.com (10.0.0.1)  5.678 ms  5.432 ms  5.321 ms
 3  core-router.isp.net (172.16.0.1)   10.234 ms  10.112 ms  9.876 ms
 4  * * *                                                          
 5  edge-router.google.com (72.14.238.1)  11.567 ms  11.432 ms  11.234 ms
 6  lax17s55-in-f14.1e100.net (142.250.80.46)  12.345 ms  12.123 ms  11.987 ms
```

### Understanding traceroute Output

```
 3  core-router.isp.net (172.16.0.1)   10.234 ms  10.112 ms  9.876 ms
 │  │                    │              │           │           │
 │  │                    │              └───────────┴───────────┘
 │  │                    │                 3 round-trip time measurements
 │  │                    └── IP address of the router
 │  └── Hostname of the router (if DNS can resolve it)
 └── Hop number (how many routers deep)
```

### What Do the Stars (* * *) Mean?

```
 4  * * *

This means the router at hop 4 didn't respond. Possible reasons:
├── Router is configured to NOT reply to traceroute (common for security)
├── Firewall is blocking ICMP/UDP responses
├── Router is too busy to respond
└── MPLS network (encapsulated traffic — routers don't see the TTL)

★ Stars don't always mean a problem! Many routers intentionally hide themselves.
   If the packets reach the destination, the path is fine.
```

### traceroute Options

```bash
# Use ICMP instead of UDP (sometimes gets through firewalls)
sudo traceroute -I google.com

# Use TCP SYN (port 80) — best for getting through firewalls
sudo traceroute -T -p 80 google.com

# Set max hops (default is 30)
traceroute -m 15 google.com

# Don't resolve hostnames (faster)
traceroute -n google.com

# Set wait time for each reply
traceroute -w 3 google.com

# Send a specific number of probes per hop (default is 3)
traceroute -q 5 google.com

# Windows — similar options
tracert -d google.com         # Don't resolve hostnames
tracert -h 15 google.com     # Max 15 hops
tracert -w 3000 google.com   # Timeout in milliseconds
```

### Reading traceroute Like a Pro

```
Scenario 1: Everything looks great ✅
 1  192.168.1.1      1 ms    1 ms    1 ms
 2  10.0.0.1         5 ms    5 ms    5 ms
 3  172.16.0.1       10 ms   10 ms   10 ms
 4  142.250.80.46    12 ms   12 ms   12 ms    ← Destination reached
 Latency increases gradually — this is normal.

Scenario 2: Sudden latency spike 🔍
 1  192.168.1.1      1 ms    1 ms    1 ms
 2  10.0.0.1         5 ms    5 ms    5 ms
 3  172.16.0.1       150 ms  148 ms  155 ms   ← Sudden jump!
 4  142.250.80.46    152 ms  150 ms  153 ms
 The problem is between hop 2 and hop 3 (ISP issue likely).

Scenario 3: Packet loss at a specific hop ⚠️
 1  192.168.1.1      1 ms    1 ms    1 ms
 2  10.0.0.1         5 ms    *       5 ms     ← One packet lost
 3  172.16.0.1       10 ms   10 ms   10 ms
 4  142.250.80.46    12 ms   12 ms   12 ms
 One star at hop 2 is usually okay — could be rate limiting.
 If ALL subsequent hops show loss, there's a real problem.

Scenario 4: Packets never reach destination 💥
 1  192.168.1.1      1 ms    1 ms    1 ms
 2  10.0.0.1         5 ms    5 ms    5 ms
 3  * * *
 4  * * *
 5  * * *
 Packets are being dropped/blocked after hop 2.
 Could be a firewall, routing issue, or the destination is down.
```

## 5. mtr — ping + traceroute Combined

### What is mtr?

`mtr` (My Traceroute) combines `ping` and `traceroute` into one tool. It **continuously** sends packets and shows you real-time statistics for **every hop** in the path.

It's the **best tool for finding intermittent network problems** because it runs over time and shows packet loss and latency at each hop.

### Installation

```bash
# Ubuntu/Debian
sudo apt install mtr

# CentOS/RHEL
sudo yum install mtr

# macOS
brew install mtr

# Windows — WinMTR (GUI version)
# Download from: https://winmtr.net
```

### Basic Usage

```bash
# Interactive mode (text-based UI — updates in real-time)
mtr google.com

# Report mode (runs N cycles and prints a report)
mtr -r -c 100 google.com

# Output:
HOST: myserver                Loss%   Snt   Last   Avg  Best  Wrst StDev
  1.|-- gateway (192.168.1.1)   0.0%   100    1.2   1.3   0.9   2.1   0.3
  2.|-- isp-router (10.0.0.1)   0.0%   100    5.4   5.5   4.8   7.2   0.5
  3.|-- core.isp.net            2.0%   100   10.1  10.3   9.5  15.4   1.2
  4.|-- google-peer             0.0%   100   11.8  11.9  11.2  13.1   0.4
  5.|-- lax17s55 (142.250.80)   0.0%   100   12.3  12.4  11.8  13.5   0.3
```

### Understanding mtr Output

```
HOST: myserver                Loss%   Snt   Last   Avg  Best  Wrst StDev
  3.|-- core.isp.net            2.0%   100   10.1  10.3   9.5  15.4   1.2
                                │       │     │     │     │     │     │
                                │       │     │     │     │     │     └── Standard Deviation
                                │       │     │     │     │     │          (consistency of latency)
                                │       │     │     │     │     └── Worst latency (ms)
                                │       │     │     │     └── Best latency (ms)
                                │       │     │     └── Average latency (ms)
                                │       │     └── Last packet latency (ms)
                                │       └── Packets Sent
                                └── Packet Loss percentage
```

### Key Metrics to Watch

| Metric | Healthy | Warning | Problem |
|--------|---------|---------|---------|
| **Loss%** | 0% | 1-2% (at intermediate hops only) | > 2% at destination or > 5% at any hop |
| **Avg latency** | Increases gradually hop by hop | Sudden big jump between hops | Consistently high |
| **StDev** | Low (< 5) | Medium (5-20) | High (> 20) — means jitter |
| **Wrst** | Close to Avg | 2-3x Avg | 10x+ Avg — intermittent spikes |

### mtr Options

```bash
# Report mode — run 100 cycles and print summary
mtr -r -c 100 google.com

# Report mode — wide format (shows full hostnames)
mtr -rw -c 100 google.com

# Use TCP instead of ICMP (gets through more firewalls)
mtr -T -P 443 google.com

# Use UDP
mtr -u google.com

# Show IP addresses only (no DNS resolution — faster)
mtr -n google.com

# Show both IPs and hostnames
mtr -b google.com

# Set packet size
mtr -s 1400 google.com

# Set interval between packets (seconds)
mtr -i 0.5 google.com

# JSON output (great for scripting)
mtr -j -c 50 google.com
```

### Reading mtr Like a Pro

```
Key Insight: Packet loss at intermediate hops but NOT at the destination
is usually NOT a real problem. Many routers deprioritize ICMP.

Example — False alarm:
  1.|-- gateway           0.0%   100    1.2   1.3   ...
  2.|-- isp-router        0.0%   100    5.4   5.5   ...
  3.|-- core-router       15.0%  100   10.1  10.3   ...  ← Looks bad!
  4.|-- google-peer        0.0%  100   11.8  11.9   ...
  5.|-- destination        0.0%  100   12.3  12.4   ...  ← But destination is fine!
  
  → Hop 3 is just rate-limiting ICMP responses. NOT a real issue.

Example — Real problem:
  1.|-- gateway           0.0%   100    1.2   1.3   ...
  2.|-- isp-router        0.0%   100    5.4   5.5   ...
  3.|-- core-router       15.0%  100   10.1  10.3   ...  ← Loss starts here
  4.|-- google-peer       14.0%  100   11.8  11.9   ...  ← Continues
  5.|-- destination       13.0%  100   12.3  12.4   ...  ← Still present!
  
  → Real packet loss starting at hop 3. Contact your ISP.
```

## 6. tcpdump — Capture Raw Network Packets

### What is tcpdump?

`tcpdump` is a **command-line packet analyzer**. It captures network packets in real-time, letting you see exactly what's going over the wire. It's the most powerful CLI network debugging tool.

Think of it like:
> Putting a wiretap on your network cable and reading every letter (packet) that passes through.

### Installation

```bash
# Usually pre-installed on Linux. If not:
sudo apt install tcpdump     # Debian/Ubuntu
sudo yum install tcpdump     # CentOS/RHEL
```

### Basic Usage

```bash
# Capture all packets on the default interface (requires root)
sudo tcpdump

# Output:
# 14:23:45.123456 IP 192.168.1.100.43210 > 142.250.80.46.443: Flags [S], seq 123456, ...
# 14:23:45.135678 IP 142.250.80.46.443 > 192.168.1.100.43210: Flags [S.], seq 654321, ...
# 14:23:45.135789 IP 192.168.1.100.43210 > 142.250.80.46.443: Flags [.], ack 654322, ...
```

### Understanding tcpdump Output

```
14:23:45.123456 IP 192.168.1.100.43210 > 142.250.80.46.443: Flags [S], seq 123456, win 65535, length 0
│               │  │                │     │               │   │          │           │          │
│               │  │                │     │               │   │          │           │          └── Packet data length
│               │  │                │     │               │   │          │           └── Window size
│               │  │                │     │               │   │          └── Sequence number
│               │  │                │     │               │   └── TCP Flags
│               │  │                │     │               └── Destination port
│               │  │                │     └── Destination IP
│               │  │                └── Source port
│               │  └── Source IP
│               └── Protocol (IP, ARP, etc.)
└── Timestamp
```

### TCP Flags Cheat Sheet

```
Flag   Symbol   Meaning
────   ──────   ───────
SYN    [S]      Starting a connection (handshake step 1)
SYN-ACK [S.]   Accepting connection (handshake step 2)
ACK    [.]      Acknowledging received data (handshake step 3)
FIN    [F]      Closing connection gracefully
RST    [R]      Forcefully resetting/closing connection
PSH    [P]      Push — send data immediately (don't buffer)
URG    [U]      Urgent data

Common Combinations:
[S]    → SYN (new connection request)
[S.]   → SYN-ACK (connection accepted)
[.]    → ACK (acknowledgment)
[P.]   → PSH-ACK (data being sent)
[F.]   → FIN-ACK (closing connection)
[R.]   → RST-ACK (connection reset)
```

### Essential tcpdump Filters

```bash
# ─── Filter by host ───
sudo tcpdump host 192.168.1.100           # Traffic to/from this IP
sudo tcpdump src host 192.168.1.100       # Traffic FROM this IP
sudo tcpdump dst host 192.168.1.100       # Traffic TO this IP

# ─── Filter by port ───
sudo tcpdump port 80                       # HTTP traffic
sudo tcpdump port 443                      # HTTPS traffic
sudo tcpdump port 22                       # SSH traffic
sudo tcpdump portrange 8000-9000           # Range of ports

# ─── Filter by protocol ───
sudo tcpdump tcp                           # Only TCP packets
sudo tcpdump udp                           # Only UDP packets
sudo tcpdump icmp                          # Only ICMP (ping) packets
sudo tcpdump arp                           # Only ARP packets

# ─── Filter by network interface ───
sudo tcpdump -i eth0                       # Specific interface
sudo tcpdump -i any                        # All interfaces

# ─── Combine filters with AND, OR, NOT ───
sudo tcpdump host 192.168.1.100 and port 80
sudo tcpdump src host 192.168.1.100 or dst host 192.168.1.200
sudo tcpdump not port 22                   # Exclude SSH traffic
sudo tcpdump 'host 192.168.1.100 and (port 80 or port 443)'

# ─── Filter by packet size ───
sudo tcpdump greater 1000                  # Packets larger than 1000 bytes
sudo tcpdump less 100                      # Packets smaller than 100 bytes
```

### Saving and Reading Captures

```bash
# Save captured packets to a file (PCAP format)
sudo tcpdump -w capture.pcap -c 1000 host 192.168.1.100
# -w: write to file
# -c: capture only 1000 packets then stop

# Read a saved capture file
tcpdump -r capture.pcap

# Read with filters
tcpdump -r capture.pcap port 80

# Save to file AND show on screen
sudo tcpdump -w capture.pcap -c 1000 host 192.168.1.100 | tee /dev/stderr
```

### Useful tcpdump Options

```bash
# Show packet contents in ASCII
sudo tcpdump -A port 80

# Show packet contents in hex and ASCII
sudo tcpdump -XX port 80

# Don't resolve hostnames (faster output)
sudo tcpdump -n port 80

# Don't resolve hostnames OR port names
sudo tcpdump -nn port 80

# Verbose output (more details)
sudo tcpdump -v port 80      # Verbose
sudo tcpdump -vv port 80     # Very verbose
sudo tcpdump -vvv port 80    # Maximum verbose

# Limit capture to N packets
sudo tcpdump -c 50 port 80

# Show timestamps in readable format
sudo tcpdump -tttt port 80

# Set snapshot length (capture more of each packet)
sudo tcpdump -s 0 port 80    # Capture entire packet (default is 262144)
sudo tcpdump -s 100 port 80  # Capture first 100 bytes only
```

### Real-World tcpdump Examples

```bash
# 1. Capture a TCP 3-way handshake
sudo tcpdump -nn -c 3 'tcp[tcpflags] & (tcp-syn|tcp-ack) != 0' and host google.com

# 2. Capture HTTP GET requests
sudo tcpdump -A -s 0 'tcp port 80 and (((ip[2:2] - ((ip[0]&0xf)<<2)) - ((tcp[12]&0xf0)>>2)) != 0)'

# 3. Capture DNS queries
sudo tcpdump -nn port 53

# 4. Capture all traffic on port 443 (HTTPS) from a specific host
sudo tcpdump -nn -i eth0 host 10.0.0.50 and port 443 -w https_traffic.pcap

# 5. Find who's talking to a specific server
sudo tcpdump -nn dst host 10.0.0.100 -c 100 | awk '{print $3}' | sort | uniq -c | sort -rn

# 6. Capture only SYN packets (new connection attempts)
sudo tcpdump -nn 'tcp[tcpflags] == tcp-syn'

# 7. Capture packets with RST flag (connection resets — useful for debugging)
sudo tcpdump -nn 'tcp[tcpflags] & tcp-rst != 0'

# 8. Capture traffic between two specific hosts
sudo tcpdump -nn host 192.168.1.100 and host 10.0.0.50

# 9. Watch for ARP traffic (useful for network discovery issues)
sudo tcpdump -nn arp

# 10. Rotate capture files (useful for long-running captures)
sudo tcpdump -w capture_%Y%m%d_%H%M%S.pcap -G 3600 -W 24 port 80
# -G 3600: Rotate every 3600 seconds (1 hour)
# -W 24: Keep only 24 files (24 hours of captures)
```

## 7. Wireshark — Visual Packet Analysis

### What is Wireshark?

Wireshark is a **GUI-based packet analyzer** — it's like tcpdump but with a beautiful visual interface. It can capture live traffic or analyze PCAP files saved by tcpdump.

### Installation

```bash
# Ubuntu/Debian
sudo apt install wireshark

# CentOS/RHEL
sudo yum install wireshark-gnome

# macOS
brew install --cask wireshark

# Windows
# Download from: https://www.wireshark.org/download.html
```

### Wireshark Interface Overview

```
┌─────────────────────────────────────────────────────────┐
│                    Menu & Toolbar                       │
├─────────────────────────────────────────────────────────┤
│ Display Filter: [  http.request.method == "GET"      ] │ ← Type filters here
├─────────────────────────────────────────────────────────┤
│ No.  Time      Source         Dest          Protocol   │
│ 1    0.000     192.168.1.1    8.8.8.8       DNS       │ ← Packet List
│ 2    0.015     8.8.8.8        192.168.1.1   DNS       │    (all captured packets)
```

### Wireshark Display Filters

Display filters let you filter the **already captured** packets to focus on what matters:

```
# ─── Basic Filters ───
ip.addr == 192.168.1.100          # Any traffic involving this IP
ip.src == 192.168.1.100           # Traffic FROM this IP
ip.dst == 192.168.1.100           # Traffic TO this IP

# ─── Protocol Filters ───
tcp                                # Only TCP packets
udp                                # Only UDP packets
dns                                # Only DNS packets
http                               # Only HTTP packets
tls                                # Only TLS/SSL packets
arp                                # Only ARP packets
icmp                               # Only ICMP (ping) packets

# ─── Port Filters ───
tcp.port == 80                     # TCP port 80 (HTTP)
tcp.port == 443                    # TCP port 443 (HTTPS)
tcp.dstport == 22                  # Destination port 22 (SSH)
udp.port == 53                     # UDP port 53 (DNS)

# ─── HTTP Specific Filters ───
http.request                       # Only HTTP requests
http.response                      # Only HTTP responses
http.request.method == "GET"       # Only GET requests
http.request.method == "POST"      # Only POST requests
http.response.code == 200          # Only 200 OK responses
http.response.code >= 400          # Only error responses (4xx, 5xx)
http.host == "example.com"         # Requests to specific host
http.request.uri contains "api"    # URIs containing "api"

# ─── TCP Specific Filters ───
tcp.flags.syn == 1                 # SYN packets (connection starts)
tcp.flags.rst == 1                 # RST packets (connection resets)
tcp.flags.fin == 1                 # FIN packets (connection closes)
tcp.analysis.retransmission        # Retransmitted packets (network issues!)
tcp.analysis.duplicate_ack         # Duplicate ACKs (congestion indicator)
tcp.analysis.zero_window           # Zero window (receiver buffer full)

# ─── DNS Specific Filters ───
dns.qry.name == "google.com"       # DNS queries for google.com
dns.flags.response == 1            # DNS responses only
dns.qry.type == 1                  # A record queries
dns.qry.type == 28                 # AAAA record queries

# ─── TLS/SSL Filters ───
tls.handshake                      # TLS handshake packets
tls.handshake.type == 1            # Client Hello
tls.handshake.type == 2            # Server Hello
tls.handshake.extensions.server_name == "example.com"  # SNI filter

# ─── Combining Filters ───
ip.addr == 192.168.1.100 and tcp.port == 80
http.request or dns
tcp.port == 443 and ip.src == 10.0.0.50
not arp and not dns                # Exclude ARP and DNS
(http.request or tls.handshake) and ip.addr == 192.168.1.100

# ─── Time-Based Filters ───
frame.time >= "2026-05-27 14:00:00"   # Packets after a specific time
frame.time_delta > 1                   # Packets with > 1 second gap
```

### Wireshark Capture Filters vs Display Filters

```
Capture Filters:  Applied BEFORE capturing — limits what packets are captured
                  Uses BPF (Berkeley Packet Filter) syntax (same as tcpdump)
                  Example: host 192.168.1.100 and port 80

Display Filters:  Applied AFTER capturing — filters the view of captured packets
                  Uses Wireshark's own filter syntax
                  Example: ip.addr == 192.168.1.100 && tcp.port == 80

★ Use capture filters to reduce file size when capturing for a long time.
★ Use display filters for analysis (they're more powerful and flexible).
```

## 8. nslookup & dig — DNS Troubleshooting

### nslookup — Simple DNS Queries

```bash
# Basic lookup
nslookup google.com
# Server:    8.8.8.8
# Address:   8.8.8.8#53
# Non-authoritative answer:
# Name: google.com
# Address: 142.250.80.46

# Query a specific record type
nslookup -type=MX google.com          # Mail servers
nslookup -type=NS google.com          # Name servers
nslookup -type=TXT google.com         # TXT records
nslookup -type=CNAME www.example.com  # CNAME records
nslookup -type=SOA google.com         # SOA (Start of Authority)

# Use a specific DNS server
nslookup google.com 8.8.8.8           # Query Google's DNS
nslookup google.com 1.1.1.1           # Query Cloudflare's DNS

# Reverse lookup (IP → hostname)
nslookup 142.250.80.46
```

### dig — Advanced DNS Queries (Preferred Over nslookup)

```bash
# Basic lookup
dig google.com

# Output explained:
# ;; QUESTION SECTION:
# ;google.com.                    IN      A       ← What we asked
#
# ;; ANSWER SECTION:
# google.com.             300     IN      A       142.250.80.46   ← The answer
#                         │              │        │
#                         TTL (seconds)  Type     IP address
#
# ;; Query time: 12 msec                         ← How long the query took
# ;; SERVER: 8.8.8.8#53                          ← Which DNS server answered

# Short answer only
dig +short google.com
# 142.250.80.46

# Query specific record types
dig MX google.com               # Mail servers
dig NS google.com               # Name servers
dig TXT google.com              # TXT records
dig AAAA google.com             # IPv6 address
dig ANY google.com              # All records (may not work with all DNS servers)

# Use a specific DNS server
dig @8.8.8.8 google.com         # Google DNS
dig @1.1.1.1 google.com         # Cloudflare DNS

# Trace the full DNS resolution path
dig +trace google.com
# Shows: Root → TLD (.com) → Authoritative → Answer

# Reverse lookup
dig -x 142.250.80.46

# Check DNSSEC
dig +dnssec google.com

# Show only the answer section
dig +noall +answer google.com

# Check specific nameserver for a domain
dig @ns1.google.com google.com

# Check if a domain exists (NXDOMAIN = doesn't exist)
dig nonexistent-domain-12345.com
# ;; ->>HEADER<<- ... status: NXDOMAIN
```

### DNS Troubleshooting Checklist

```
1. Is DNS resolving at all?
   dig google.com
   → If no response: DNS server might be down

2. Is it resolving to the RIGHT IP?
   dig +short mysite.com
   → Compare with expected IP

3. Is the TTL too high? (Old cached record)
   dig mysite.com | grep -i ttl
   → High TTL = old cache; wait or flush DNS

4. Are nameservers correct?
   dig NS mysite.com
   → Should point to your DNS provider

5. Is propagation complete? (After DNS change)
   dig @8.8.8.8 mysite.com       # Google
   dig @1.1.1.1 mysite.com       # Cloudflare
   dig @9.9.9.9 mysite.com       # Quad9
   → All should return the same, new IP

6. Flush local DNS cache
   # Linux (systemd-resolved)
   sudo systemd-resolve --flush-caches
   
   # macOS
   sudo dscacheutil -flushcache; sudo killall -HUP mDNSResponder
   
   # Windows
   ipconfig /flushdns
```

## 11. nmap — Port Scanning & Network Discovery

### What is nmap?

`nmap` (Network Mapper) is a powerful tool for **network discovery and security auditing**. It scans hosts to find open ports, detect services, and identify operating systems.

> **⚠️ Legal Warning:** Only scan networks you own or have explicit permission to scan. Unauthorized scanning is illegal in many jurisdictions.

### Installation

```bash
sudo apt install nmap       # Debian/Ubuntu
sudo yum install nmap       # CentOS/RHEL
brew install nmap           # macOS
# Windows: Download from https://nmap.org/download.html
```

### Basic Usage

```bash
# Scan a single host — find open ports
nmap 192.168.1.100

# Output:
# PORT     STATE SERVICE
# 22/tcp   open  ssh
# 80/tcp   open  http
# 443/tcp  open  https
# 3306/tcp open  mysql

# Scan multiple hosts
nmap 192.168.1.100 192.168.1.101 192.168.1.102
nmap 192.168.1.100-110              # Range
nmap 192.168.1.0/24                 # Entire subnet

# Scan specific ports
nmap -p 80 192.168.1.100            # Single port
nmap -p 80,443,8080 192.168.1.100   # Multiple ports
nmap -p 1-1000 192.168.1.100        # Range of ports
nmap -p- 192.168.1.100              # ALL 65535 ports (slow but thorough)

# Service version detection
nmap -sV 192.168.1.100
# PORT    STATE SERVICE VERSION
# 22/tcp  open  ssh     OpenSSH 8.9p1 Ubuntu 3ubuntu0.1
# 80/tcp  open  http    nginx 1.24.0
# 443/tcp open  https   nginx 1.24.0

# OS detection
sudo nmap -O 192.168.1.100
# OS details: Linux 5.15 - 5.19

# Aggressive scan (OS detection + version + scripts + traceroute)
sudo nmap -A 192.168.1.100

# Quick scan (top 100 most common ports)
nmap -F 192.168.1.100

# Ping sweep — find all live hosts on a network
nmap -sn 192.168.1.0/24
# 192.168.1.1 is up
# 192.168.1.100 is up
# 192.168.1.105 is up

# TCP SYN scan (stealthy — doesn't complete the handshake)
sudo nmap -sS 192.168.1.100

# TCP connect scan (full handshake — doesn't require root)
nmap -sT 192.168.1.100

# UDP scan (slow but important)
sudo nmap -sU 192.168.1.100

# Save results
nmap -oN scan_results.txt 192.168.1.100       # Normal output
nmap -oX scan_results.xml 192.168.1.100       # XML output
nmap -oG scan_results.gnmap 192.168.1.100     # Grepable output
```

### Port States in nmap

```
open       → Port is accepting connections (service is running)
closed     → Port is accessible but no service is listening
filtered   → Firewall is blocking — nmap can't determine if open or closed
unfiltered → Port is accessible but nmap can't determine if open or closed
open|filtered   → nmap can't determine which
closed|filtered → nmap can't determine which
```

## 14. arp — Address Resolution Protocol

### What is ARP?

ARP maps **IP addresses to MAC addresses** on a local network. When your computer wants to send data to another device on the same network, it needs the destination's MAC address.

```
Your Computer:  "Who has 192.168.1.1? Tell me your MAC address."  (ARP Request — broadcast)
Router:         "192.168.1.1 is at AA:BB:CC:DD:EE:FF"            (ARP Reply — unicast)
```

### ARP Commands

```bash
# Show ARP table
arp -a
# Or
ip neigh show

# Output:
# 192.168.1.1 dev eth0 lladdr aa:bb:cc:dd:ee:ff REACHABLE
# 192.168.1.105 dev eth0 lladdr 11:22:33:44:55:66 STALE

# ARP states:
# REACHABLE  → Recently confirmed (MAC is valid)
# STALE      → Not confirmed recently (might be outdated)
# DELAY      → Waiting to confirm
# FAILED     → ARP request got no reply

# Manually add an ARP entry
sudo arp -s 192.168.1.200 aa:bb:cc:dd:ee:ff

# Delete an ARP entry
sudo arp -d 192.168.1.200

# Clear ARP cache (force re-discovery)
sudo ip neigh flush all
```

## 15. Troubleshooting Flowchart

Use this flowchart when you encounter a network issue:

```
START: "I can't reach a service"
│
├── Step 1: Can you ping yourself?
│   $ ping 127.0.0.1
│   ├── NO  → Network stack is broken. Check if interface is up: ip link show
│   └── YES → Continue
│
├── Step 2: Can you ping your gateway?
│   $ ip route show default  → Get gateway IP
│   $ ping <gateway-ip>
│   ├── NO  → Local network issue. Check cable, WiFi, IP config, DHCP
│   └── YES → Continue
│
├── Step 3: Can you ping an external IP?
│   $ ping 8.8.8.8
│   ├── NO  → Routing issue. Check: ip route, firewall rules, ISP status
│   └── YES → Continue
│
├── Step 4: Can you resolve DNS?
│   $ nslookup google.com
│   ├── NO  → DNS issue. Check /etc/resolv.conf, try different DNS server
│   └── YES → Continue
│
├── Step 5: Can you ping the target host?
│   $ ping <target-ip-or-hostname>
│   ├── NO  → Host might be down, or ICMP blocked
│   │        Run: traceroute <target> to find where it stops
│   │        Run: mtr <target> for detailed path analysis
│   └── YES → Continue
│
├── Step 6: Can you reach the specific port?
│   $ telnet <target> <port>   OR   $ curl <target>:<port>
│   ├── NO  → Port is blocked (firewall) or service not running
│   │        Check on target: ss -tlnp | grep <port>
│   │        Check firewall: iptables -L, ufw status, security groups
│   └── YES → Continue
│
├── Step 7: Is the application responding correctly?
│   $ curl -v http://<target>:<port>/health
│   ├── NO  → Application issue. Check logs on the server
│   │        $ journalctl -u <service-name> -f
│   │        $ tail -f /var/log/nginx/error.log
│   └── YES → The service is working! The issue is elsewhere.
│
└── Still stuck? → Capture packets with tcpdump/Wireshark for deeper analysis
```

```bash
ss -tlnp | grep 8080
# Nothing? → Service is not running. Start it.

# 2. Is it listening on the right interface?
ss -tlnp | grep 8080
# 127.0.0.1:8080 → Only listening on localhost! 
# Change to 0.0.0.0:8080 for external access

# 3. Is the firewall blocking?
sudo iptables -L -n | grep 8080
sudo ufw status
# Check cloud security group rules if on AWS/GCP/Azure

# 4. Verify with curl from the server itself
curl localhost:8080              # Works? → Firewall or network issue
curl 10.0.0.50:8080              # Doesn't work? → App config issue
```

### Scenario 3: "DNS is Not Resolving"

```bash
# 1. Check current DNS config
cat /etc/resolv.conf
# nameserver 192.168.1.1  → Is this DNS server working?

# 2. Test with different DNS servers
dig @8.8.8.8 example.com        # Google DNS
dig @1.1.1.1 example.com        # Cloudflare DNS
dig @192.168.1.1 example.com    # Local DNS

# 3. If external DNS works but local doesn't
# → Your local DNS server (192.168.1.1) is the problem
# Fix: Add working DNS to /etc/resolv.conf
echo "nameserver 8.8.8.8" | sudo tee /etc/resolv.conf

# 4. Flush DNS cache
sudo systemd-resolve --flush-caches   # Linux
ipconfig /flushdns                     # Windows
sudo dscacheutil -flushcache           # macOS
```

### Scenario 4: "Intermittent Packet Loss"

```bash
# 1. Run mtr for extended analysis (100 packets)
mtr -rw -c 100 target-host

# 2. Look for packet loss pattern
#    - Loss at ONE hop but not at destination → Router rate-limiting (not a problem)
#    - Loss at a hop AND all subsequent hops → Real problem at that hop
#    - Loss only at destination → Target server issue

# 3. Check during different times of day
#    (Congestion is often time-based)

# 4. Capture packets to see what's happening
sudo tcpdump -nn -c 1000 host target-host -w packet_loss_debug.pcap
# Analyze in Wireshark — look for retransmissions
```

## 17. Summary & Quick Reference

### Tool Decision Matrix

```
"Can I reach this host?"           → ping
"Where is the packet getting stuck?" → traceroute
"Where is packet loss happening?"  → mtr
"What's on the wire?"             → tcpdump
"Deep packet analysis?"           → Wireshark
"Is DNS working?"                 → dig / nslookup
"What's listening on my server?"  → ss / netstat
"Is the HTTP service responding?" → curl
"Which ports are open?"           → nmap
"How fast is my network?"         → iperf
"What's my IP / interface?"       → ip addr / ifconfig
"Is Layer 2 (ARP) working?"      → arp / ip neigh
```

### Quick Reference Cheat Sheet

```bash
# ─── Essential Commands ───
ping -c 5 host                    # Test connectivity
traceroute host                   # Trace path
mtr -rw -c 50 host              # Detailed path analysis
dig +short host                   # DNS lookup
ss -tlnp                          # Show listening ports
curl -vI https://host             # Test HTTP + headers
sudo tcpdump -nn port 80 -c 100  # Capture 100 packets on port 80
nmap -sV host                     # Scan ports + detect services

# ─── "Is it DNS?" ───
dig @8.8.8.8 hostname            # Yes if this works but normal dig doesn't

# ─── "Is the port open?" ───
ss -tlnp | grep PORT             # Check locally
nmap -p PORT host                # Check remotely

# ─── "What's happening on the wire?" ───
sudo tcpdump -A -nn port 80 host HOST -c 50

# ─── "How slow is the network?" ───
mtr -rw host                     # Path analysis
iperf3 -c server                 # Bandwidth test
curl -o /dev/null -s -w "%{time_total}\n" https://host  # HTTP response time
```
