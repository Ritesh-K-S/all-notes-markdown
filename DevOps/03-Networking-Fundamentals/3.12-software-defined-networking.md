## 1. What is Software Defined Networking (SDN)?

**Software Defined Networking (SDN)** is an approach to networking where the **control of the network is done through software**, separate from the physical hardware that forwards the traffic.

Let's break that down with a simple analogy:

### The Restaurant Analogy

> **Traditional Networking** is like a restaurant where every waiter decides on their own where to seat customers, what to recommend, and how to handle complaints. Each waiter makes independent decisions. If you want to change the seating policy, you have to go to **every single waiter** and tell them individually.
>
> **SDN** is like having a **manager** who makes ALL the decisions — where to seat people, what the specials are, how to handle complaints — and the waiters **just follow orders**. Want to change the policy? Tell the manager once, and ALL waiters follow the new rules instantly.

### Technical Definition

In SDN:
- The **brain** (control plane) that makes routing decisions is **separated** from the **muscles** (data plane) that actually forwards packets.
- The brain is a centralized **software controller** that programs all the network devices.
- Network devices (switches, routers) become **dumb forwarding devices** that just do what the controller tells them.

```
Traditional Network:
┌──────────────────┐    ┌──────────────────┐    ┌──────────────────┐
│ Switch 1         │    │ Switch 2         │    │ Switch 3         │
│ ┌──────────────┐ │    │ ┌──────────────┐ │    │ ┌──────────────┐ │
│ │ Control Plane│ │    │ │ Control Plane│ │    │ │ Control Plane│ │
│ │ (Brain)      │ │    │ │ (Brain)      │ │    │ │ (Brain)      │ │
│ ├──────────────┤ │    │ ├──────────────┤ │    │ ├──────────────┤ │
│ │ Data Plane   │ │    │ │ Data Plane   │ │    │ │ Data Plane   │ │
│ │ (Forwarding) │ │    │ │ (Forwarding) │ │    │ │ (Forwarding) │ │
│ └──────────────┘ │    │ └──────────────┘ │    │ └──────────────┘ │
└──────────────────┘    └──────────────────┘    └──────────────────┘
Each device has its OWN brain. Configuration is per-device.

SDN:
                    ┌──────────────────────┐
                    │   SDN Controller     │
                    │   (Centralized Brain)│
                    └──────┬───────────────┘
                           │ Southbound API (OpenFlow, etc.)
              ┌────────────┼────────────────┐
              │            │                │
       ┌──────┴─────┐ ┌───┴──────────┐ ┌───┴──────────┐
       │ Switch 1   │ │ Switch 2     │ │ Switch 3     │
       │ Data Plane │ │ Data Plane   │ │ Data Plane   │
       │ (Forward)  │ │ (Forward)    │ │ (Forward)    │
       └────────────┘ └──────────────┘ └──────────────┘
ONE brain controls ALL devices. Configuration is centralized.
```

## 2. Why Was SDN Created? — The Problem with Traditional Networking

### Problems with Traditional Networking

| Problem | Explanation |
|---------|-------------|
| **Manual configuration** | Every switch/router must be configured individually via CLI. 100 switches? 100 separate SSH sessions. |
| **Vendor lock-in** | Each vendor (Cisco, Juniper, Arista) has its own CLI, OS, and proprietary features. Hard to mix and match. |
| **Slow to change** | Adding a new VLAN, firewall rule, or routing policy means touching multiple devices. Takes hours or days. |
| **Error-prone** | Humans manually typing commands on 100 devices = guaranteed typos and inconsistencies. |
| **No global view** | Each device only knows about its immediate neighbors. No single place to see the entire network. |
| **Hard to automate** | Traditional devices weren't designed for API-driven automation. Screen scraping CLI output is fragile. |
| **Doesn't scale for cloud** | Cloud needs to spin up/down networks in seconds. Traditional networking takes days. |

### What SDN Solves

```
Problem:                          SDN Solution:
──────────────────               ──────────────────
Manual CLI on each device    →   Centralized controller with APIs
Vendor lock-in               →   Open standards (OpenFlow), white-box switches
Slow changes                 →   Instant programmable changes
Error-prone                  →   Automated, consistent configuration
No global view               →   Controller has a complete network map
Hard to automate             →   REST APIs, Python SDKs, Infrastructure as Code
Doesn't scale for cloud      →   Virtual networks created in milliseconds
```

## 3. SDN Architecture — The Three Layers

SDN has a clean three-layer architecture:

```
┌─────────────────────────────────────────────────────────────┐
│                    APPLICATION LAYER                         │
│                                                             │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────────┐  │
│  │ Firewall │ │ Load     │ │ Network  │ │ Monitoring   │  │
│  │ App      │ │ Balancer │ │ Monitor  │ │ Dashboard    │  │
│  └──────────┘ └──────────┘ └──────────┘ └──────────────┘  │
│                                                             │
│  These are network applications that use the controller's   │
│  APIs to implement network functions.                       │
├────────────────── Northbound API (REST) ────────────────────┤
│                                                             │
│                    CONTROL LAYER                             │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │              SDN Controller                          │   │
│  │                                                      │   │
│  │  ┌──────────┐ ┌──────────┐ ┌──────────┐            │   │
│  │  │ Topology │ │ Routing  │ │ Stats    │            │   │
│  │  │ Manager  │ │ Engine   │ │ Collector│            │   │
│  │  └──────────┘ └──────────┘ └──────────┘            │   │
│  │                                                      │   │
│  │  The "brain" — knows the entire network topology     │   │
│  │  and makes all forwarding decisions.                  │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
├────────────── Southbound API (OpenFlow, etc.) ──────────────┤
│                                                             │
│                 INFRASTRUCTURE LAYER                         │
│                                                             │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐     │
│  │ Switch 1 │ │ Switch 2 │ │ Switch 3 │ │ Switch 4 │     │
│  │ (OVS)    │ │ (OVS)    │ │ (HW)     │ │ (HW)     │     │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘     │
│                                                             │
│  Physical or virtual switches that forward packets based    │
│  on flow rules from the controller.                         │
└─────────────────────────────────────────────────────────────┘
```

### Layer 1: Infrastructure Layer (Data Plane)

```
What:    Physical or virtual network devices (switches, routers)
Role:    Forward packets based on rules received from the controller
Examples: Open vSwitch (OVS), white-box switches, hardware SDN switches
Key:     These devices are "dumb" — they don't make decisions on their own
```

### Layer 2: Control Layer (Control Plane)

```
What:    The SDN Controller — the centralized brain
Role:    Computes forwarding rules and pushes them to switches
         Maintains a global view of the entire network topology
Examples: OpenDaylight, ONOS, Floodlight, Ryu, POX
Key:     This is where the intelligence lives
```

### Layer 3: Application Layer

```
What:    Network applications that consume the controller's APIs
Role:    Implement specific network functions (firewall, load balancer, etc.)
Examples: Network monitoring dashboards, traffic engineering apps, 
         security policies, QoS management
Key:     Applications talk to the controller via the Northbound API (usually REST)
```

### The APIs

| API | Direction | Purpose | Protocol |
|-----|-----------|---------|----------|
| **Northbound API** | Apps ↔ Controller | Applications request network services from the controller | REST API, Python SDK |
| **Southbound API** | Controller ↔ Switches | Controller programs flow rules into switches | OpenFlow, NETCONF, gRPC |
| **East-West API** | Controller ↔ Controller | Multiple controllers sync with each other (distributed SDN) | Proprietary, OVSDB |

## 4. How SDN Works — Step by Step

Let's trace what happens when a new packet enters an SDN network:

### Scenario: Host A (10.0.0.1) sends a packet to Host B (10.0.0.2)

```
Step 1: Host A sends a packet to Host B
        The packet arrives at Switch 1.

Step 2: Switch 1 checks its flow table:
        "Do I have a rule for this packet?"

        ┌─────────────────────────────────────────────┐
        │ Flow Table (Switch 1)                       │
        │                                             │
        │ Match                    → Action           │
        │ ─────                      ──────           │
        │ (empty — no rules yet!)                     │
        └─────────────────────────────────────────────┘

Step 3: No matching rule! → TABLE MISS
        Switch 1 sends the packet to the SDN Controller.
        (This is called a "Packet-In" message)

Step 4: SDN Controller receives the packet:
        - Looks at its global network topology
        - Calculates the best path: Switch 1 → Switch 2 → Switch 3
        - Creates flow rules for each switch in the path

Step 5: Controller pushes flow rules to all switches in the path:
        (These are called "Flow-Mod" messages)

        Switch 1: "If dst=10.0.0.2, forward to port 3"
        Switch 2: "If dst=10.0.0.2, forward to port 5"
        Switch 3: "If dst=10.0.0.2, forward to port 1"

Step 6: Switches install the flow rules in their flow tables:
        
        ┌─────────────────────────────────────────────┐
        │ Flow Table (Switch 1) — UPDATED             │
        │                                             │
        │ Match              → Action                 │
        │ ─────                ──────                 │
        │ dst_ip=10.0.0.2    → Forward to port 3     │
        └─────────────────────────────────────────────┘

Step 7: The original packet (and all future packets to 10.0.0.2) 
        are forwarded along the path WITHOUT going to the controller again.

Step 8: Future packets matching the same rule are handled 
        DIRECTLY by the switches — no controller involved.
        This is called "fast path" or "data plane forwarding."
```

### The Key Insight

```
First packet:  Switch → Controller → "Here's the rule" → Switch forwards
               (Slow — involves the controller)

All subsequent: Switch forwards directly using the installed rule
                (Fast — hardware speed, no controller involvement)
```

## 5. SDN vs Traditional Networking — Side by Side

| Aspect | Traditional Networking | SDN |
|--------|----------------------|-----|
| **Control** | Distributed (each device decides) | Centralized (controller decides) |
| **Configuration** | CLI per device (SSH/Telnet) | API-driven, programmatic |
| **Intelligence** | In each device's firmware | In the software controller |
| **Vendor lock-in** | High (proprietary CLI, OS) | Low (open standards, white-box) |
| **Network view** | Local (each device sees neighbors) | Global (controller sees everything) |
| **Change speed** | Hours/days (manual work) | Seconds (API call) |
| **Automation** | Difficult (screen scraping) | Native (REST APIs, Python SDKs) |
| **Cost** | Expensive (vendor hardware) | Cheaper (commodity/white-box switches) |
| **Scalability** | Complex (each device configured separately) | Simpler (controller handles it) |
| **Troubleshooting** | SSH into each device | Centralized dashboard, single view |
| **Innovation** | Vendor controls features | Anyone can write applications |

### When to Use Traditional Networking

```
✅ Small networks (< 20 devices) where manual management is feasible
✅ Legacy environments that can't be easily migrated
✅ When you need specific vendor features (Cisco ACI, etc.)
✅ When your team's expertise is in traditional networking
```

### When to Use SDN

```
✅ Data centers with hundreds/thousands of switches
✅ Cloud environments (AWS, GCP, Azure all use SDN internally)
✅ Kubernetes clusters needing programmable networking
✅ Multi-tenant environments needing network isolation
✅ When you need rapid, automated network changes
✅ When you want to treat the network as code (Infrastructure as Code)
```

## 6. Key SDN Concepts

### 6.1 Flow

A **flow** is a set of packets that share common characteristics (like same source IP, destination IP, protocol, port, etc.).

```
Example flows:
├── All HTTP traffic from 10.0.0.0/24 to the internet
├── All DNS queries from any host to 8.8.8.8
├── All SSH traffic to server 10.0.0.50
└── All traffic from VLAN 100 to VLAN 200
```

### 6.2 Flow Table

The **flow table** is the forwarding table inside each SDN switch. It contains rules that tell the switch what to do with packets.

```
┌────────────────────────────────────────────────────────────────┐
│                    Flow Table                                  │
│                                                                │
│ Priority │ Match Fields           │ Action          │ Counters │
│ ──────── │ ────────────           │ ──────          │ ──────── │
│ 100      │ dst_ip=10.0.0.2       │ Forward port 3  │ 1523 pkts│
│ 90       │ dst_ip=10.0.0.0/24    │ Forward port 5  │ 8932 pkts│
│ 80       │ tcp, dst_port=80      │ Forward port 7  │ 456 pkts │
│ 10       │ *  (match anything)   │ Send to controller│ 12 pkts│
│ 0        │ *  (table miss)       │ Drop            │ 3 pkts  │
└────────────────────────────────────────────────────────────────┘

Higher priority rules are checked first.
First match wins (like firewall rules).
```

### 6.3 Flow Rule Components

Each flow rule has three parts:

```
1. MATCH FIELDS — What packets does this rule apply to?
   ├── Source/Destination IP
   ├── Source/Destination MAC
   ├── Source/Destination Port
   ├── Protocol (TCP, UDP, ICMP)
   ├── VLAN ID
   ├── Input port
   └── And more...

2. ACTIONS — What to do with matching packets?
   ├── Forward to a specific port
   ├── Forward to the controller
   ├── Drop the packet
   ├── Modify headers (change VLAN, rewrite IP/MAC)
   ├── Send to a group (multicast/load balance)
   ├── Set QoS queue
   └── Meter (rate limiting)

3. COUNTERS — Statistics
   ├── Packet count
   ├── Byte count
   └── Duration (how long the rule has existed)
```

### 6.4 Proactive vs Reactive Flow Installation

```
Reactive (On-Demand):
├── No rules pre-installed
├── First packet of a flow → TABLE MISS → Controller
├── Controller installs rule → Subsequent packets handled at line speed
├── Pros: Simple, only rules for active flows
├── Cons: First packet delay, controller can be bottleneck
└── Good for: Small/medium networks

Proactive (Pre-Programmed):
├── Controller installs rules BEFORE traffic arrives
├── ALL packets are handled at line speed (no TABLE MISS)
├── Pros: No first-packet delay, no controller bottleneck
├── Cons: More complex, need to predict all traffic patterns
└── Good for: Large data centers, production networks
```

## 7. OpenFlow — The Original SDN Protocol

### What is OpenFlow?

OpenFlow is the **first and most well-known** southbound protocol in SDN. It defines how the controller communicates with switches to program flow tables.

Think of OpenFlow as the **language** the controller speaks to the switches:
> Controller (in OpenFlow): "Hey Switch 1, if you see a packet going to 10.0.0.2, send it out port 3."
> Switch 1: "Got it, boss."

### OpenFlow Architecture

```
┌──────────────────────────┐
│    SDN Controller        │
│                          │
│  Sends OpenFlow messages │
│  to manage flow tables   │
└────────────┬─────────────┘
             │ OpenFlow Protocol (TCP port 6653)
             │
┌────────────┴─────────────┐
│    OpenFlow Switch       │
│                          │
│  ┌────────────────────┐  │
│  │ Flow Table 0       │  │
│  ├────────────────────┤  │
│  │ Flow Table 1       │  │  ← Multiple tables for pipeline processing
│  ├────────────────────┤  │
│  │ Flow Table N       │  │
│  └────────────────────┘  │
│                          │
│  ┌────────────────────┐  │
│  │ Group Table        │  │  ← For multicast, load balancing
│  └────────────────────┘  │
│                          │
│  ┌────────────────────┐  │
│  │ Meter Table        │  │  ← For rate limiting
│  └────────────────────┘  │
└──────────────────────────┘
```

### Key OpenFlow Messages

| Message | Direction | Purpose |
|---------|-----------|---------|
| **Packet-In** | Switch → Controller | "I got a packet I don't know what to do with" |
| **Packet-Out** | Controller → Switch | "Send this specific packet out of port X" |
| **Flow-Mod** | Controller → Switch | "Install/modify/delete this flow rule" |
| **Port-Status** | Switch → Controller | "A port went up/down" |
| **Stats-Request** | Controller → Switch | "Give me your flow table statistics" |
| **Stats-Reply** | Switch → Controller | "Here are my statistics" |
| **Hello** | Both | "Let's establish a connection" |
| **Echo** | Both | "Are you still alive?" (keepalive) |
| **Features-Request/Reply** | Both | "What capabilities do you have?" |

### OpenFlow Flow Rule Example

```
A flow rule that says: "All HTTP traffic from subnet 10.0.0.0/24 
going to server 10.0.1.100, forward to port 5"

Match:
  ├── in_port: any
  ├── eth_type: 0x0800 (IPv4)
  ├── ip_src: 10.0.0.0/24
  ├── ip_dst: 10.0.1.100
  ├── ip_proto: 6 (TCP)
  └── tcp_dst: 80

Action:
  └── output: port 5

Priority: 100
Timeout:  idle_timeout=300, hard_timeout=0
```

### OpenFlow Versions

```
Version    Year    Key Features
───────    ────    ────────────
OF 1.0     2009    Basic flow tables, single table per switch
OF 1.1     2011    Multiple flow tables, groups, VLAN support
OF 1.2     2011    IPv6 support, extensible match
OF 1.3     2012    Meters (rate limiting), per-port queues ← Most widely supported
OF 1.4     2013    Synchronized tables, optical ports
OF 1.5     2014    Egress tables, copy-field action

★ OpenFlow 1.3 is the most commonly used version in production.
```

## 8. SDN Controllers — The Brain of the Network

### What Does a Controller Do?

```
1. Maintains a global view of the network topology
2. Discovers switches and hosts (LLDP, ARP)
3. Computes optimal paths (shortest path, load-balanced, etc.)
4. Installs flow rules in switches
5. Handles exceptions (table misses, new hosts)
6. Provides APIs for network applications
7. Collects statistics and monitoring data
```

### Popular SDN Controllers

| Controller | Language | Maintained By | Best For |
|------------|----------|---------------|----------|
| **OpenDaylight (ODL)** | Java | Linux Foundation | Enterprise, production |
| **ONOS** | Java | Open Networking Foundation | Service providers, telecoms |
| **Ryu** | Python | Community (NTT Labs) | Learning, small deployments |
| **Floodlight** | Java | Big Switch Networks | Learning, moderate deployments |
| **POX** | Python | Community | Education, prototyping |
| **Faucet** | Python | Community | Production-ready, simple |

### Controller Comparison

```
For Learning & Prototyping:
├── Ryu     → Python, simple, great documentation, easy to write apps
├── POX     → Python, extremely simple, perfect for beginners
└── Mininet → Not a controller, but great for testing with any controller

For Production:
├── OpenDaylight → Feature-rich, enterprise-grade, large community
├── ONOS         → Designed for service providers, high availability
└── Faucet       → Simple, production-ready, uses YAML for config
```

### Writing a Simple SDN Application (Ryu Controller)

```python
# simple_switch.py — A basic L2 learning switch using Ryu
# This controller learns MAC addresses and forwards packets intelligently

from ryu.base import app_manager
from ryu.controller import ofp_event
from ryu.controller.handler import CONFIG_DISPATCHER, MAIN_DISPATCHER
from ryu.controller.handler import set_ev_cls
from ryu.ofproto import ofproto_v1_3
from ryu.lib.packet import packet, ethernet

class SimpleSwitch(app_manager.RyuApp):
    """
    A simple L2 learning switch.
    
    How it works:
    1. When a packet arrives and we don't know where to send it,
       we flood it to all ports (like a dumb hub).
    2. But we LEARN: "MAC address X is reachable via port Y"
    3. Next time we see a packet destined for MAC X, 
       we send it directly to port Y (smart switch behavior).
    """
    
    OFP_VERSIONS = [ofproto_v1_3.OFP_VERSION]
    
    def __init__(self, *args, **kwargs):
        super(SimpleSwitch, self).__init__(*args, **kwargs)
        # MAC address table: {dpid: {mac_address: port_number}}
        # dpid = switch ID
        self.mac_to_port = {}
    
    @set_ev_cls(ofp_event.EventOFPSwitchFeatures, CONFIG_DISPATCHER)
    def switch_features_handler(self, ev):
        """
        Called when a switch connects to the controller.
        We install a default rule: "If no match, send to controller"
        """
        datapath = ev.msg.datapath          # The switch object
        ofproto = datapath.ofproto          # OpenFlow protocol constants
        parser = datapath.ofproto_parser    # Message parser
        
        # Install a table-miss flow entry:
        # Match: everything (no specific match = match all)
        # Action: send to controller
        # Priority: 0 (lowest — only matches if nothing else does)
        match = parser.OFPMatch()
        actions = [parser.OFPActionOutput(
            ofproto.OFPP_CONTROLLER,        # Send to controller
            ofproto.OFPCML_NO_BUFFER        # Send full packet, don't buffer
        )]
        self.add_flow(datapath, 0, match, actions)
        self.logger.info("Switch %s connected!", datapath.id)
    
    def add_flow(self, datapath, priority, match, actions):
        """Helper: Install a flow rule in a switch"""
        ofproto = datapath.ofproto
        parser = datapath.ofproto_parser
        
        # Build the instruction: "Apply these actions"
        inst = [parser.OFPInstructionActions(
            ofproto.OFPIT_APPLY_ACTIONS, actions
        )]
        
        # Build the Flow-Mod message
        mod = parser.OFPFlowMod(
            datapath=datapath,
            priority=priority,
            match=match,
            instructions=inst
        )
        
        # Send the rule to the switch
        datapath.send_msg(mod)
    
    @set_ev_cls(ofp_event.EventOFPPacketIn, MAIN_DISPATCHER)
    def packet_in_handler(self, ev):
        """
        Called when a switch sends us a packet (TABLE MISS).
        This is where the magic happens:
        1. Learn the source MAC → port mapping
        2. If we know where the destination MAC is, forward directly
        3. If not, flood to all ports
        """
        msg = ev.msg
        datapath = msg.datapath
        ofproto = datapath.ofproto
        parser = datapath.ofproto_parser
        in_port = msg.match['in_port']      # Which port the packet came in on
        
        # Parse the packet to get source and destination MAC
        pkt = packet.Packet(msg.data)
        eth = pkt.get_protocol(ethernet.ethernet)
        
        dst = eth.dst    # Destination MAC address
        src = eth.src    # Source MAC address
        dpid = datapath.id  # Switch ID
        
        # Initialize MAC table for this switch if needed
        self.mac_to_port.setdefault(dpid, {})
        
        # LEARN: "MAC src is reachable via in_port"
        self.mac_to_port[dpid][src] = in_port
        self.logger.info("Learned: Switch %s, MAC %s → Port %s", dpid, src, in_port)
        
        # DECIDE: Do we know where the destination MAC is?
        if dst in self.mac_to_port[dpid]:
            # YES! We know which port to use → Forward directly
            out_port = self.mac_to_port[dpid][dst]
            self.logger.info("Known destination %s → Port %s", dst, out_port)
        else:
            # NO! We don't know → Flood to all ports
            out_port = ofproto.OFPP_FLOOD
            self.logger.info("Unknown destination %s → Flooding", dst)
        
        actions = [parser.OFPActionOutput(out_port)]
        
        # If we know the destination, install a flow rule so future packets
        # are handled by the switch directly (no controller involvement)
        if out_port != ofproto.OFPP_FLOOD:
            match = parser.OFPMatch(in_port=in_port, eth_dst=dst, eth_src=src)
            self.add_flow(datapath, 1, match, actions)
        
        # Send this specific packet out
        data = msg.data
        out = parser.OFPPacketOut(
            datapath=datapath,
            buffer_id=msg.buffer_id,
            in_port=in_port,
            actions=actions,
            data=data
        )
        datapath.send_msg(out)
```

```bash
# Output:

# instantiating app simple_switch.py of SimpleSwitch
# Switch 1 connected!
# Learned: Switch 1, MAC 00:00:00:00:00:01 → Port 1
# Unknown destination 00:00:00:00:00:02 → Flooding
# Learned: Switch 1, MAC 00:00:00:00:00:02 → Port 2
# Known destination 00:00:00:00:00:01 → Port 1
```

## 9. Types of SDN

### 9.1 Open SDN (Pure SDN)

```
Uses open protocols (OpenFlow) between controller and switches.
Controller has full control over the data plane.

Controller ──── OpenFlow ──── White-box switches (Open vSwitch, etc.)

Pros: Full control, vendor neutral, open standards
Cons: Requires OpenFlow-compatible switches, complexity
```

### 9.2 API-Based SDN (Overlay SDN)

```
Uses the existing network as an underlay and creates virtual 
networks on top using tunneling protocols (VXLAN, GRE, STT).

┌─────────────────────────────────────┐
│ Virtual Network (Overlay)           │
│ 10.0.0.0/24 ←→ 10.0.1.0/24       │
│ (Managed by SDN controller)         │
├─────────────────────────────────────┤
│ Physical Network (Underlay)         │
│ 192.168.0.0/16                     │
│ (Regular switches — no OpenFlow)    │
└─────────────────────────────────────┘

Pros: Works with existing hardware, no switch replacement needed
Cons: Overhead from encapsulation, doesn't optimize the physical network
Examples: VMware NSX, Nuage Networks, Flannel (Kubernetes)
```

### 9.3 Hybrid SDN

```
Some traffic is handled by traditional protocols (OSPF, BGP), 
and some traffic is handled by the SDN controller.

Best of both worlds — gradual migration from traditional to SDN.

Pros: Incremental adoption, compatibility with legacy network
Cons: Complexity of managing two paradigms
Examples: Cisco ACI, many enterprise deployments
```

## 10. Network Function Virtualization (NFV)

### What is NFV?

NFV takes **network functions that traditionally ran on dedicated hardware** and runs them as **software on commodity servers**.

```
Traditional:
┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐
│ Hardware │ │ Hardware │ │ Hardware │ │ Hardware │
│ Firewall │ │ Load     │ │ IDS/IPS  │ │ Router   │
│ ($50K)   │ │ Balancer │ │ ($30K)   │ │ ($20K)   │
│          │ │ ($40K)   │ │          │ │          │
└──────────┘ └──────────┘ └──────────┘ └──────────┘
Total: $140K+ in hardware

NFV:
┌──────────────────────────────────────────────────┐
│ Commodity Server ($5K)                           │
│                                                  │
│ ┌──────────┐ ┌──────────┐ ┌──────────┐         │
│ │ Virtual  │ │ Virtual  │ │ Virtual  │         │
│ │ Firewall │ │ Load     │ │ IDS/IPS  │ ...     │
│ │ (VM/CNT) │ │ Balancer │ │ (VM/CNT) │         │
│ └──────────┘ └──────────┘ └──────────┘         │
│                                                  │
│ Running as VMs or containers on standard x86     │
└──────────────────────────────────────────────────┘
Total: $5K + software licenses
```

### SDN vs NFV — They're Different But Complementary

| Aspect | SDN | NFV |
|--------|-----|-----|
| **Focus** | How traffic is routed/controlled | Where network functions run |
| **What it virtualizes** | The network control plane | Network appliances (firewall, LB, etc.) |
| **Key technology** | Centralized controller + OpenFlow | VMs/Containers running network functions |
| **Replaces** | Manual switch configuration | Dedicated hardware appliances |
| **Works together** | SDN can manage the network connecting NFV functions |

### Common NFV Functions (VNFs)

```
Virtual Network Functions (VNFs):
├── vFirewall    → Virtual firewall (pfSense, OPNsense)
├── vRouter      → Virtual router (VyOS, FRRouting)
├── vLoad Balancer → Virtual load balancer (HAProxy, Nginx, Envoy)
├── vIDS/IPS     → Virtual intrusion detection (Suricata, Snort)
├── vDNS         → Virtual DNS server (BIND, CoreDNS)
├── vWAN Optimizer → Virtual WAN optimization
└── vCPE         → Virtual Customer Premises Equipment
```

```
When you do this in AWS:
```

```hcl
    from_port   = 0
    to_port     = 0
```

## 12. SDN in Kubernetes and Containers

### Container Networking is SDN

When you deploy containers in Kubernetes, the **Container Network Interface (CNI)** plugins are basically SDN implementations.

```
Kubernetes Networking Requirements (SDN solves all of these):
1. Every pod gets its own IP address
2. Pods can communicate with each other without NAT
3. Services load-balance traffic across pods
4. Network policies control which pods can talk to each other

All of this is done through SDN — virtual networks, flow rules, 
and programmable overlays.
```

### CNI Plugins = SDN for Containers

| CNI Plugin | SDN Type | How It Works |
|------------|----------|-------------|
| **Flannel** | Overlay (VXLAN) | Creates an overlay network on top of the physical network |
| **Calico** | L3 Routing (BGP) or Overlay | Programs routes using BGP or VXLAN overlays |
| **Cilium** | eBPF-based SDN | Uses Linux kernel eBPF for high-performance networking |
| **Weave** | Overlay (custom) | Mesh overlay with encryption |
| **Canal** | Flannel + Calico | Flannel for networking + Calico for network policies |
| **OVN-Kubernetes** | Open Virtual Network | Uses OVS/OVN for virtual networking |

### Kubernetes Network Policy — SDN Flow Rules

```yaml
# This Kubernetes NetworkPolicy is essentially an SDN flow rule!
# It says: "Only allow traffic to the 'api' pods from 'frontend' pods on port 8080"

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-api
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: api                    # Apply to pods labeled "app: api"
  
  policyTypes:
    - Ingress                     # Control incoming traffic
  
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: frontend       # Only allow from pods labeled "app: frontend"
      ports:
        - protocol: TCP
          port: 8080              # Only on port 8080

# What happens behind the scenes (SDN):
# The CNI plugin (e.g., Calico) translates this to:
# iptables rules, eBPF programs, or OVS flow rules
# that enforce the policy at the kernel level.
```

### Calico — SDN Flow Rules Example

```bash
# Calico translates Kubernetes NetworkPolicies into iptables rules:

# Allow traffic from frontend to api on port 8080
iptables -A FORWARD -s 10.244.1.0/24 -d 10.244.2.0/24 \
  -p tcp --dport 8080 -j ACCEPT

# Block everything else to the api pods
iptables -A FORWARD -d 10.244.2.0/24 -j DROP

# (In reality, Calico uses more sophisticated rules with ipsets and chains)
```

### Cilium — eBPF-Based SDN

```
Cilium uses eBPF (extended Berkeley Packet Filter) instead of iptables.
eBPF programs run INSIDE the Linux kernel, making them extremely fast.

Traditional (iptables-based CNI):
Packet → Kernel → iptables chain 1 → chain 2 → ... → chain N → Forward
         (Slow when you have thousands of rules)

Cilium (eBPF-based):
Packet → Kernel → eBPF program → Forward
         (Fast! O(1) lookup regardless of rule count)
```

## 13. SD-WAN — SDN for Wide Area Networks

### What is SD-WAN?

**SD-WAN (Software Defined Wide Area Network)** applies SDN principles to **WAN connections** — the links between your offices, data centers, and cloud providers.

```
Traditional WAN:
┌──────────┐                                    ┌──────────┐
│ Office A │ ────── Expensive MPLS line ──────── │ Office B │
│          │       ($3,000/month, 50 Mbps)       │          │
└──────────┘                                    └──────────┘
One path, one provider, expensive, inflexible.

SD-WAN:
┌──────────┐   ┌─── MPLS (reliable, expensive) ────┐   ┌──────────┐
│ Office A │ ──┤                                     ├── │ Office B │
│  SD-WAN  │   ├─── Broadband Internet (cheap) ────┤   │  SD-WAN  │
│  Device  │   │                                     │   │  Device  │
│          │   └─── 4G/5G LTE (backup) ────────────┘   │          │
└──────────┘                                            └──────────┘

SD-WAN controller intelligently routes traffic:
├── Video calls → MPLS (needs reliability)
├── Web browsing → Broadband (cheaper, good enough)
├── Cloud apps → Direct Internet (bypass the data center)
└── Backup → 4G/5G (if MPLS and broadband are down)
```

### SD-WAN Benefits

```
1. Cost Savings:     Use cheap broadband instead of expensive MPLS
2. Performance:      Intelligent path selection based on real-time conditions
3. Cloud-Ready:      Direct cloud access from branches (no backhauling)
4. Visibility:       Centralized dashboard for all WAN links
5. Automation:        Zero-touch provisioning for new sites
6. Redundancy:       Multiple paths = automatic failover
```

### SD-WAN Providers

| Provider | Type | Notes |
|----------|------|-------|
| **Cisco Viptela/Meraki** | Enterprise | Largest market share |
| **VMware VeloCloud** | Enterprise | Strong integration with VMware stack |
| **Palo Alto Prisma SD-WAN** | Security-focused | Strong security features |
| **Fortinet Secure SD-WAN** | Security-focused | Integrated FortiGate security |
| **Cloudflare Magic WAN** | Cloud-native | Part of Cloudflare One |
| **Zscaler** | Cloud-native | Security-first approach |

## 14. SDN Security

### Security Benefits of SDN

```
1. Centralized Policy Enforcement
   → One place to define security rules, applied everywhere instantly

2. Network Segmentation (Micro-segmentation)
   → Isolate workloads at the VM/container level, not just at the VLAN level

3. Dynamic Threat Response
   → Detect an attack → Controller instantly reprograms the network to block it

4. Complete Network Visibility
   → Controller sees ALL traffic → better anomaly detection

5. Consistent Security
   → No "forgot to configure the firewall on Switch 47" problems
```

### Security Risks of SDN

```
1. Controller is a Single Point of Failure
   → If the controller is compromised, the entire network is compromised
   → Mitigation: Run redundant controllers, secure the controller

2. Southbound API Attacks
   → Attacker could forge OpenFlow messages to inject malicious flow rules
   → Mitigation: TLS encryption on OpenFlow connections

3. Controller Overload (DoS)
   → Flood the network with unknown packets → every packet goes to controller
   → Mitigation: Rate limiting on Packet-In messages, proactive flow rules

4. Application Layer Attacks
   → Malicious SDN application with API access could disrupt the network
   → Mitigation: Application sandboxing, RBAC for northbound API
```

### Micro-segmentation with SDN

```
Traditional Segmentation:            SDN Micro-segmentation:
                                     
┌─────────────────────┐              ┌─────────────────────┐
│ VLAN 100 (Web)      │              │ Web Tier            │
│ ┌───┐ ┌───┐ ┌───┐  │              │ ┌───┐ ┌───┐ ┌───┐  │
│ │ A │ │ B │ │ C │  │              │ │ A │ │ B │ │ C │  │
│ └───┘ └───┘ └───┘  │              │ └─┬─┘ └─┬─┘ └─┬─┘  │
│ A, B, C can talk    │              │   │     │     │     │
│ freely to each other│              │   Each pod has its  │
└─────────────────────┘              │   own security rules│
                                     │   A can only talk   │
                                     │   to DB server      │
                                     │   B can only talk   │
                                     │   to Cache server   │
                                     └─────────────────────┘
```

## 16. Hands-On: SDN with Open vSwitch (OVS)

### What is Open vSwitch?

Open vSwitch (OVS) is a **production-quality virtual switch** that supports OpenFlow. It's used in most cloud platforms and container networking solutions.

### Installation

```bash
# Ubuntu/Debian
sudo apt install openvswitch-switch

# CentOS/RHEL
sudo yum install openvswitch

# Verify installation
sudo ovs-vsctl show
```

### Basic OVS Commands

```bash
# Create a virtual switch (bridge)
sudo ovs-vsctl add-br br0

# Show all bridges
sudo ovs-vsctl show

# Add a port to the bridge
sudo ovs-vsctl add-port br0 eth1
sudo ovs-vsctl add-port br0 veth0

# List ports on a bridge
sudo ovs-vsctl list-ports br0

# Delete a port
sudo ovs-vsctl del-port br0 eth1

# Delete a bridge
sudo ovs-vsctl del-br br0

# Set OpenFlow controller for a bridge
sudo ovs-vsctl set-controller br0 tcp:192.168.1.100:6653

# Show OpenFlow flow tables
sudo ovs-ofctl dump-flows br0

# Add a flow rule manually
sudo ovs-ofctl add-flow br0 \
  "priority=100, in_port=1, dl_dst=00:00:00:00:00:02, actions=output:2"

# Add a flow rule: drop all traffic from a specific MAC
sudo ovs-ofctl add-flow br0 \
  "priority=200, dl_src=aa:bb:cc:dd:ee:ff, actions=drop"

# Delete a flow rule
sudo ovs-ofctl del-flows br0 "in_port=1"

# Show port statistics
sudo ovs-ofctl dump-ports br0

# Set a port as a trunk port (carry multiple VLANs)
sudo ovs-vsctl set port eth1 trunks=100,200,300

# Set a port as an access port (single VLAN)
sudo ovs-vsctl set port eth1 tag=100

# Create a GRE tunnel between two OVS bridges
sudo ovs-vsctl add-port br0 gre0 -- \
  set interface gre0 type=gre options:remote_ip=192.168.1.200

# Create a VXLAN tunnel
sudo ovs-vsctl add-port br0 vxlan0 -- \
  set interface vxlan0 type=vxlan options:remote_ip=192.168.1.200 \
  options:key=5000
```

### Example: Create a Simple SDN Network with OVS

```bash
# Create a virtual network with 2 hosts connected via OVS

# 1. Create the bridge
sudo ovs-vsctl add-br br0

# 2. Create two network namespaces (simulating two hosts)
sudo ip netns add host1
sudo ip netns add host2

# 3. Create virtual ethernet pairs (veth)
sudo ip link add veth-h1 type veth peer name veth-h1-br
sudo ip link add veth-h2 type veth peer name veth-h2-br

# 4. Move one end of each veth into the namespace
sudo ip link set veth-h1 netns host1
sudo ip link set veth-h2 netns host2

# 5. Connect the other end to the OVS bridge
sudo ovs-vsctl add-port br0 veth-h1-br
sudo ovs-vsctl add-port br0 veth-h2-br

# 6. Configure IP addresses inside the namespaces
sudo ip netns exec host1 ip addr add 10.0.0.1/24 dev veth-h1
sudo ip netns exec host1 ip link set veth-h1 up
sudo ip netns exec host1 ip link set lo up

sudo ip netns exec host2 ip addr add 10.0.0.2/24 dev veth-h2
sudo ip netns exec host2 ip link set veth-h2 up
sudo ip netns exec host2 ip link set lo up

# 7. Bring up the bridge ports
sudo ip link set veth-h1-br up
sudo ip link set veth-h2-br up

# 8. Test connectivity!
sudo ip netns exec host1 ping 10.0.0.2
# PING 10.0.0.2 (10.0.0.2) 56(84) bytes of data.
# 64 bytes from 10.0.0.2: icmp_seq=1 ttl=64 time=0.234 ms

# 9. View the flow table
sudo ovs-ofctl dump-flows br0
# NXST_FLOW reply:
#  cookie=0x0, table=0, priority=0 actions=NORMAL

# 10. Add a custom flow rule: block host2 from pinging host1
sudo ovs-ofctl add-flow br0 \
  "priority=100, ip, nw_src=10.0.0.2, nw_dst=10.0.0.1, icmp, actions=drop"

# 11. Test again — ping should fail now!
sudo ip netns exec host2 ping -c 3 10.0.0.1
# 3 packets transmitted, 0 received, 100% packet loss

# 12. Remove the block
sudo ovs-ofctl del-flows br0 "nw_src=10.0.0.2, nw_dst=10.0.0.1"

# 13. Clean up
sudo ovs-vsctl del-br br0
sudo ip netns del host1
sudo ip netns del host2
```

### Python Script: Custom Mininet Topology

```python
#!/usr/bin/env python3
"""
custom_topo.py — Create a custom SDN topology with Mininet

Topology:
    h1 ──── s1 ──── s2 ──── h3
              │      │
             h2     h4
"""

from mininet.net import Mininet
from mininet.node import Controller, OVSSwitch, RemoteController
from mininet.cli import CLI
from mininet.log import setLogLevel
from mininet.link import TCLink

def create_network():
    # Create the network
    net = Mininet(
        controller=Controller,    # Use default controller
        switch=OVSSwitch,         # Use Open vSwitch
        link=TCLink               # Use links with traffic control
    )
    
    # Add controller
    c0 = net.addController('c0')
    
    # Add switches
    s1 = net.addSwitch('s1')
    s2 = net.addSwitch('s2')
    
    # Add hosts
    h1 = net.addHost('h1', ip='10.0.0.1/24')
    h2 = net.addHost('h2', ip='10.0.0.2/24')
    h3 = net.addHost('h3', ip='10.0.0.3/24')
    h4 = net.addHost('h4', ip='10.0.0.4/24')
    
    # Add links (with bandwidth and delay)
    net.addLink(h1, s1, bw=100, delay='1ms')   # h1 ↔ s1: 100 Mbps, 1ms delay
    net.addLink(h2, s1, bw=100, delay='1ms')   # h2 ↔ s1
    net.addLink(s1, s2, bw=1000, delay='5ms')  # s1 ↔ s2: 1 Gbps, 5ms delay
    net.addLink(h3, s2, bw=100, delay='1ms')   # h3 ↔ s2
    net.addLink(h4, s2, bw=100, delay='1ms')   # h4 ↔ s2
    
    # Start the network
    net.start()
    
    print("\n=== Network Topology ===")
    print("h1 (10.0.0.1) ──── s1 ──── s2 ──── h3 (10.0.0.3)")
    print("                    │       │")
    print("                h2 (10.0.0.2)  h4 (10.0.0.4)")
    print("========================\n")
    
    # Test connectivity
    print("Testing connectivity...")
    net.pingAll()
    
    # Test bandwidth between h1 and h3
    print("\nBandwidth test h1 → h3:")
    h1, h3 = net.get('h1', 'h3')
    net.iperf((h1, h3))
    
    # Start CLI for interactive use
    CLI(net)
    
    # Stop the network
    net.stop()

if __name__ == '__main__':
    setLogLevel('info')
    create_network()
```

```bash

sudo python3 custom_topo.py
```

## 20. Summary

```
SDN Cheat Sheet:

WHAT:    Separating the network's brain (control plane) from the muscles (data plane)
WHY:     Traditional networking is manual, slow, vendor-locked, and can't scale for cloud
HOW:     A centralized controller programs dumb switches via protocols like OpenFlow

Architecture:
┌──────────────────────────┐
│ Application Layer        │  ← Network apps (firewall, monitor, LB)
│ (Northbound API: REST)   │
├──────────────────────────┤
│ Control Layer            │  ← SDN Controller (the brain)
│ (Southbound API: OpenFlow│
├──────────────────────────┤
│ Infrastructure Layer     │  ← Switches (the muscles)
└──────────────────────────┘

Key Concepts:
├── Flow          → A set of packets matching certain criteria
├── Flow Table    → Rules in a switch: Match → Action
├── Controller    → Centralized brain (OpenDaylight, ONOS, Ryu)
├── OpenFlow      → Protocol between controller and switches
├── Overlay       → Virtual network on top of physical (VXLAN)
└── NFV           → Network functions as software, not hardware

SDN in Your World:
├── Cloud (AWS VPC, Azure VNet)      = SDN
├── Kubernetes (CNI plugins)         = SDN
├── Docker networks                  = SDN
├── Security Groups / Network ACLs   = SDN flow rules
└── Terraform networking resources   = SDN automation

Tools:
├── Open vSwitch (OVS) → Virtual switch for SDN
├── Mininet            → SDN network emulator
├── Ryu / ONOS / ODL   → SDN Controllers
├── Cilium / Calico    → Kubernetes SDN (CNI)
└── Wireshark          → Analyze OpenFlow traffic
```
