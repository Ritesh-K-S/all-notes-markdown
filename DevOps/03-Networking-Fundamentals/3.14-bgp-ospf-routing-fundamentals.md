## 1. What is Routing?

Let's start from the very beginning.

When you type `google.com` in your browser, your computer needs to send a packet to Google's server. But your computer doesn't have a direct cable to Google — the packet has to travel through **many networks** to get there.

**Routing** is the process of **finding the best path** for a packet to travel from its source to its destination across multiple networks.

Think of it like GPS navigation:
- You want to go from your house to a restaurant
- There are many possible roads
- The GPS picks the best route based on distance, traffic, tolls, etc.
- If a road is blocked, it recalculates

**Routers** are the devices that make these decisions. Every time a packet hits a router, the router looks at the destination IP address and decides: *"Where should I send this next?"*

### The Key Question Routing Answers:

> "I have a packet for IP address `142.250.190.46` — which of my network interfaces (or neighbor routers) should I forward it to?"

## 2. Static vs Dynamic Routing

There are two fundamental approaches to routing:

### Static Routing

You **manually tell** the router where to send traffic. You literally type in the rules yourself.

```
# Linux static route example
ip route add 10.20.0.0/16 via 192.168.1.1
```

This says: "To reach the 10.20.0.0/16 network, send packets to the router at 192.168.1.1"

**Advantages:**
- Simple to set up
- No extra CPU/memory overhead
- Very predictable — you know exactly what path traffic takes
- More secure — no routing protocol messages to intercept

**Disadvantages:**
- Doesn't scale — imagine manually configuring routes for 10,000 networks!
- No automatic failover — if a link goes down, you have to manually update routes
- High maintenance — every network change requires manual updates

**When to use:** Small networks, default gateways, very specific traffic paths, simple setups.

### Dynamic Routing

Routers **automatically discover** routes by talking to each other using **routing protocols**. They exchange information about which networks they know about and automatically update routes when things change.

**Advantages:**
- Scales to millions of routes (the entire internet runs on dynamic routing!)
- Automatic failover — if a link dies, routers recalculate automatically
- Low maintenance — routers figure it out themselves
- Adapts to network changes in real time

**Disadvantages:**
- Uses CPU, memory, and bandwidth
- More complex to configure initially
- Can have convergence delays (time to recalculate after a change)
- Potential for routing loops or instability if misconfigured

**When to use:** Any network with more than a few routers, production environments, the internet.

### Quick Comparison:

| Feature | Static | Dynamic |
|---------|--------|---------|
| Configuration | Manual | Automatic |
| Scalability | Poor | Excellent |
| Failover | Manual | Automatic |
| CPU/Memory | Low | Higher |
| Best for | Small networks | Large networks |
| Example | `ip route add` | OSPF, BGP |

## 3. Routing Tables — The Core Concept

Every device that does routing (routers, Linux servers, your laptop) has a **routing table**. This is the master list that tells the device: "For this destination, send the packet HERE."

### Viewing the Routing Table on Linux:

```bash
# Method 1: ip route (modern, preferred)
ip route show

# Example output:
default via 192.168.1.1 dev eth0 proto dhcp metric 100
10.0.0.0/8 via 10.1.1.1 dev eth1 proto ospf metric 20
172.16.0.0/16 via 10.1.1.2 dev eth1 proto bgp metric 30
192.168.1.0/24 dev eth0 proto kernel scope link src 192.168.1.100

# Method 2: route (older, still works)
route -n

# Method 3: netstat (legacy)
netstat -rn
```

### Understanding Each Field:

```
10.0.0.0/8 via 10.1.1.1 dev eth1 proto ospf metric 20
│           │            │         │          │
│           │            │         │          └── Metric (cost/priority - lower is better)
│           │            │         └── Protocol that added this route
│           │            └── Which network interface to use
│           └── Next hop (the next router to send to)
└── Destination network (CIDR notation)
```

### How Route Matching Works — Longest Prefix Match:

When a router receives a packet, it needs to find the best matching route. It uses **longest prefix match**:

```
Routes in table:
  10.0.0.0/8      → via Router A
  10.20.0.0/16    → via Router B
  10.20.30.0/24   → via Router C
  0.0.0.0/0       → via Default Gateway

Incoming packet for: 10.20.30.5

Which route matches?
  10.0.0.0/8      → YES (matches /8)
  10.20.0.0/16    → YES (matches /16)
  10.20.30.0/24   → YES (matches /24)  ← WINNER! Most specific match
  0.0.0.0/0       → YES (matches everything)

Result: Send to Router C (longest/most specific prefix wins)
```

The router always picks the **most specific** (longest prefix) match. If nothing matches, the **default route** (`0.0.0.0/0`) catches everything — like a catch-all rule.

### Administrative Distance (Route Preference):

When multiple routing protocols suggest routes to the same destination, the router needs to pick one. **Administrative Distance (AD)** is the trust level:

| Source | AD | Trust Level |
|--------|-----|------------|
| Directly connected | 0 | Highest |
| Static route | 1 | Very high |
| eBGP | 20 | High |
| OSPF | 110 | Medium |
| RIP | 120 | Low |
| iBGP | 200 | Lower |
| Unknown/unreachable | 255 | Never used |

Lower AD = more trusted. If OSPF (AD 110) and RIP (AD 120) both know about `10.0.0.0/8`, the OSPF route wins.

## 4. Interior vs Exterior Gateway Protocols

Routing protocols are divided into two major categories:

### IGP — Interior Gateway Protocol

Used **inside** a single organization's network (called an **Autonomous System** or **AS**).

- Think: routing within your company, your data center, your campus
- Examples: **OSPF**, **EIGRP**, **RIP**, **IS-IS**
- Focus: Find the **fastest/shortest** path within the organization

### EGP — Exterior Gateway Protocol

Used **between** different organizations/networks (between Autonomous Systems).

- Think: routing between ISPs, between your company and the internet, between cloud providers
- Example: **BGP** (it's the only one used today)
- Focus: **Policy-based** routing — not just shortest path, but business relationships

### What's an Autonomous System (AS)?

An **Autonomous System** is a collection of networks under a single administrative control that presents a common routing policy to the internet.

- Your company's entire network = 1 AS
- Each ISP = 1 AS
- AWS = 1 AS (actually they have multiple)
- Google = 1 AS

Each AS gets a unique number called an **ASN** (Autonomous System Number):
- Google: AS15169
- Amazon: AS16509
- Cloudflare: AS13335

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  ┌──────────────┐    IGP (OSPF)    ┌──────────────┐     │
│  │   Router 1   │◄───────────────►│   Router 2   │     │
│  │  10.1.0.0/16 │                 │  10.2.0.0/16 │     │
│  └──────┬───────┘                 └──────┬───────┘     │
│         │           YOUR NETWORK         │              │
│         │           (AS 65001)           │              │
│  ┌──────┴───────┐                 ┌──────┴───────┐     │
│  │   Router 3   │◄───────────────►│   Router 4   │     │
│  │  10.3.0.0/16 │    IGP (OSPF)   │  10.4.0.0/16 │     │
│  └──────┬───────┘                 └──────────────┘     │
│         │                                               │
└─────────┼───────────────────────────────────────────────┘
          │
          │ BGP (eBGP) ← Exterior, between different organizations
          │
┌─────────┼───────────────────────────────────────────────┐
│         │              ISP NETWORK                       │
│  ┌──────┴───────┐      (AS 65002)                       │
│  │   ISP Router │                                       │
│  └──────────────┘                                       │
└─────────────────────────────────────────────────────────┘
```

## 5. OSPF — Open Shortest Path First

### What is OSPF?

OSPF is the most widely used **Interior Gateway Protocol (IGP)**. It's used to route traffic **within** an organization's network.

Think of OSPF like this: Every router in your network creates a **complete map** of the entire network. Then each router independently calculates the **shortest path** to every destination using that map.

### Key Characteristics:

| Feature | Detail |
|---------|--------|
| **Type** | Link-State Protocol |
| **Algorithm** | Dijkstra's Shortest Path First (SPF) |
| **Metric** | Cost (based on bandwidth) |
| **Standard** | Open standard (RFC 2328) |
| **Convergence** | Fast (seconds) |
| **Supports** | VLSM, CIDR, authentication |
| **Protocol** | IP protocol 89 (not TCP or UDP!) |
| **Multicast** | 224.0.0.5 (AllSPFRouters), 224.0.0.6 (AllDRRouters) |

### How OSPF Works — The Big Picture:

```
Step 1: DISCOVER — Routers discover neighbors by sending Hello packets
Step 2: EXCHANGE — Routers exchange their link-state information (LSAs)
Step 3: BUILD   — Each router builds a complete map (Link-State Database / LSDB)
Step 4: CALCULATE — Each router runs Dijkstra's algorithm on the map
Step 5: INSTALL — Best routes go into the routing table
Step 6: MAINTAIN — Routers keep sending Hellos; if topology changes, re-run SPF
```

### Step-by-Step Breakdown:

**Step 1: Hello Packets (Discovery)**

Every OSPF router periodically sends **Hello packets** out all OSPF-enabled interfaces:
- "Hi, I'm Router 1, my router ID is 1.1.1.1, I'm in Area 0"
- If another router hears the Hello and the parameters match, they become **neighbors**

Hello packets contain:
- Router ID
- Area ID
- Hello interval (default: 10 seconds)
- Dead interval (default: 40 seconds — if no Hello in 40s, neighbor is dead)
- Authentication info
- Network mask
- List of known neighbors

**Step 2: Exchange LSAs (Link-State Advertisements)**

Once neighbors are established, routers exchange **LSAs** — these describe:
- What networks the router is connected to
- What other routers it's connected to
- The cost (bandwidth) of each link

**Step 3: Build the LSDB (Link-State Database)**

Every router collects all LSAs and builds a **complete map** of the network topology. All routers in the same area should have **identical LSDBs**.

**Step 4: Run SPF (Dijkstra's Algorithm)**

Each router uses the LSDB as input to Dijkstra's algorithm to calculate the **shortest path tree** — the best path to every destination.

**Step 5: Install Routes**

The best paths from the SPF tree are installed into the routing table.

### OSPF Cost Calculation:

OSPF uses **cost** as its metric. Cost is calculated based on bandwidth:

```
Cost = Reference Bandwidth / Interface Bandwidth
```

Default reference bandwidth = 100 Mbps (10^8)

| Interface | Bandwidth | Cost |
|-----------|-----------|------|
| Fast Ethernet (100 Mbps) | 100,000,000 | 100M/100M = **1** |
| Gigabit Ethernet (1 Gbps) | 1,000,000,000 | 100M/1000M = **0.1 → 1** (minimum) |
| 10 Gigabit | 10,000,000,000 | 100M/10000M = **0.01 → 1** |
| T1 (1.544 Mbps) | 1,544,000 | 100M/1.544M = **64** |
| 56k modem | 56,000 | 100M/0.056M = **1785** |

> ⚠️ Notice the problem? With the default reference bandwidth, anything 100 Mbps or faster gets cost 1 — the router can't tell the difference between Gigabit and 10 Gigabit! That's why in modern networks, you should increase the reference bandwidth:

```
# Cisco router
router ospf 1
  auto-cost reference-bandwidth 10000    # 10 Gbps reference

# Now:
# 10 GigE = cost 1
# 1 GigE = cost 10
# 100 Mbps = cost 100
```

The total cost of a path = sum of all link costs along the path. The **lowest total cost** path wins.

## 6. OSPF Areas & Hierarchical Design

### Why Areas?

In a small network with 10 routers, every router can have a complete map of everything — no problem. But in a large network with 1,000 routers, things get messy:
- The LSDB becomes enormous
- Every change triggers SPF recalculation on ALL routers
- Hello/LSA traffic consumes bandwidth

**Areas** solve this by breaking the network into smaller regions:

```
┌──────────────────────────────────────────────────┐
│                    Area 0                         │
│                (Backbone Area)                    │
│                                                  │
│    ┌────┐        ┌────┐        ┌────┐           │
│    │ABR1│        │ R5 │        │ABR2│           │
│    └──┬─┘        └────┘        └──┬─┘           │
│       │                           │              │
└───────┼───────────────────────────┼──────────────┘
        │                           │
   ┌────┼──────┐              ┌────┼──────┐
   │    │      │              │    │      │
   │  ┌─┴──┐  │              │  ┌─┴──┐  │
   │  │ R1 │  │              │  │ R3 │  │
   │  └────┘  │              │  └────┘  │
   │  ┌────┐  │              │  ┌────┐  │
   │  │ R2 │  │              │  │ R4 │  │
   │  └────┘  │              │  └────┘  │
   │  Area 1  │              │  Area 2  │
   └──────────┘              └──────────┘
```

### Area Types:

| Area Type | Description |
|-----------|-------------|
| **Area 0 (Backbone)** | The core area — all other areas MUST connect to it |
| **Regular Area** | Normal OSPF area (Area 1, 2, etc.) |
| **Stub Area** | No external routes — uses default route instead |
| **Totally Stubby Area** | Even more restricted — only default route |
| **NSSA (Not-So-Stubby Area)** | Stub area that can import some external routes |

### Key Area Rules:

1. **Area 0 is mandatory** — it's the backbone, everything connects through it
2. **All areas must connect to Area 0** — either directly or via virtual links
3. **ABRs (Area Border Routers)** sit between two areas — they summarize routes between areas
4. **ASBRs (Autonomous System Boundary Routers)** connect OSPF to external routing domains (like BGP)

### Benefits of Areas:

- **Smaller LSDBs** — routers only need the full map of their own area
- **Fewer SPF calculations** — changes in one area don't affect other areas
- **Route summarization** — ABRs can summarize routes between areas
- **Faster convergence** — smaller scope = quicker recalculation

## 7. OSPF Configuration Examples

### On a Linux Router (using FRRouting / FRR):

FRRouting (FRR) is the most popular open-source routing suite for Linux. It's what you'll use in DevOps/cloud environments.

```bash
# Install FRRouting on Ubuntu/Debian
sudo apt update
sudo apt install frr frr-pythontools -y

# Enable OSPF daemon
sudo sed -i 's/ospfd=no/ospfd=yes/' /etc/frr/daemons

# Restart FRR
sudo systemctl restart frr

# Enter the FRR shell (similar to Cisco CLI)
sudo vtysh
```

```
# Inside vtysh - Configure OSPF
configure terminal

! Set router ID (usually a loopback IP)
router ospf
  ospf router-id 1.1.1.1

! Add networks to OSPF
  network 10.0.1.0/24 area 0
  network 10.0.2.0/24 area 0
  network 192.168.1.0/24 area 1

! Optional: change reference bandwidth for modern networks
  auto-cost reference-bandwidth 10000

! Optional: set interface cost manually
interface eth0
  ip ospf cost 10

! Optional: authentication
interface eth1
  ip ospf authentication message-digest
  ip ospf message-digest-key 1 md5 MySecretKey

! Save configuration
end
write memory
```

### Verify OSPF:

```bash
# Inside vtysh
show ip ospf neighbor           # See OSPF neighbors
show ip ospf database           # See the LSDB
show ip ospf route              # See OSPF-learned routes
show ip ospf interface          # See OSPF-enabled interfaces
show ip route ospf              # See OSPF routes in the routing table
```

Example output of `show ip ospf neighbor`:


## 8. OSPF Packet Types & Neighbor States

### The 5 OSPF Packet Types:

| Type | Name | Purpose |
|------|------|---------|
| 1 | **Hello** | Discover and maintain neighbors |
| 2 | **DBD (Database Description)** | Summary of LSDB contents for comparison |
| 3 | **LSR (Link-State Request)** | "Send me the full details of these LSAs" |
| 4 | **LSU (Link-State Update)** | "Here are the full LSA details you asked for" |
| 5 | **LSAck (Link-State Acknowledgment)** | "I received your LSA update" |

### OSPF Neighbor States (in order):

```
Down → Init → 2-Way → ExStart → Exchange → Loading → Full
```

| State | What's Happening |
|-------|-----------------|
| **Down** | No Hello received from neighbor |
| **Init** | Hello received, but neighbor hasn't acknowledged us yet |
| **2-Way** | Both routers see each other in Hello packets — bidirectional communication confirmed |
| **ExStart** | Negotiating master/slave for database exchange |
| **Exchange** | Trading DBD packets — comparing LSDBs |
| **Loading** | Requesting missing LSAs with LSR, receiving with LSU |
| **Full** | LSDBs are synchronized — fully adjacent! |

> 💡 The **2-Way** state is where **DR/BDR election** happens on broadcast networks (like Ethernet). The **Full** state means the routers are fully synchronized and can route traffic properly.

### DR and BDR (Designated Router / Backup Designated Router):

On a broadcast network (Ethernet), if 10 routers are connected to the same switch, do they ALL exchange LSAs with ALL others? That would be 10×9 = 90 adjacencies!

Instead, OSPF elects:
- **DR (Designated Router)** — the "hub" that all other routers send LSAs to
- **BDR (Backup Designated Router)** — takes over if DR fails

All other routers form Full adjacency only with DR and BDR (not with each other — they stay in 2-Way).

**Election rules:**
1. Highest OSPF priority wins (default: 1, range: 0-255, 0 = never become DR)
2. If priorities are equal, highest Router ID wins

## 9. BGP — Border Gateway Protocol

### What is BGP?

If OSPF is the routing protocol for inside your house, **BGP is the routing protocol for the entire internet**. It's what makes the internet work.

> **BGP is the protocol that glues the entire internet together.**

Every ISP, every cloud provider, every large enterprise uses BGP to exchange routing information with each other. When you access a website in another country, BGP is what figured out the path your packets should take across dozens of networks.

### Why is BGP Special?

BGP is fundamentally different from OSPF:

| Feature | OSPF | BGP |
|---------|------|-----|
| **Scope** | Inside one organization (IGP) | Between organizations (EGP) |
| **Goal** | Shortest path | Policy-based routing |
| **Scale** | Thousands of routes | ~1 million+ routes (full internet table) |
| **Protocol** | IP Protocol 89 | TCP port 179 |
| **Algorithm** | Dijkstra (SPF) | Best Path Selection (many attributes) |
| **Trust** | Everyone in the area cooperates | "Trust no one" — verify everything |
| **Speed** | Fast convergence (seconds) | Slow convergence (can take minutes) |
| **Metric** | Cost (bandwidth-based) | AS path, local pref, MED, etc. |

### BGP Version:

The current version is **BGP-4** (RFC 4271), which has been the standard since 1995. When people say "BGP," they mean BGP-4.

### Key BGP Concepts:

**1. Autonomous System (AS)**
- A network or group of networks under a single administrative control
- Identified by a unique ASN (Autonomous System Number)
- 2-byte ASN: 1-65535 (running out)
- 4-byte ASN: 1-4294967295 (the new range)
- Private ASNs: 64512-65534 (2-byte), 4200000000-4294967294 (4-byte)

**2. BGP Peers / Neighbors**
- Two BGP routers that have established a TCP session with each other
- BGP runs over **TCP port 179** — it's one of the few routing protocols that uses TCP
- Peers must be **manually configured** — BGP doesn't auto-discover like OSPF
- Each peer relationship is explicitly defined

**3. Prefixes / NLRI (Network Layer Reachability Information)**
- The routes (network prefixes) that BGP advertises
- Example: "I can reach 203.0.113.0/24 through my network"

**4. AS-PATH**
- The list of ASes a route has traveled through
- Used for loop prevention and path selection
- Example: AS-PATH = {65001, 65002, 65003} means the route went through 3 ASes

## 10. How BGP Works — Step by Step

### Step 1: Establish TCP Session

BGP routers establish a TCP connection on **port 179**. Unlike OSPF, BGP doesn't use multicasting to discover neighbors — you must **explicitly configure** each peer.

```
Router A (AS 65001) ──── TCP:179 ────► Router B (AS 65002)
```

### Step 2: Open Message

Once TCP is connected, routers exchange **OPEN messages** containing:
- BGP version (4)
- ASN
- Router ID
- Hold time (default: 180 seconds — if no message in 180s, session is dead)
- Capabilities (address families, route refresh, etc.)

### Step 3: Keepalive Messages

Routers exchange **KEEPALIVE messages** every 60 seconds (1/3 of hold time) to confirm they're still alive.

### Step 4: Update Messages

This is where the actual routing information is exchanged. **UPDATE messages** contain:
- **New routes** (NLRI) — "Hey, I can now reach these networks"
- **Withdrawn routes** — "I can no longer reach these networks, remove them"
- **Path attributes** — AS-PATH, next-hop, local preference, MED, etc.

### Step 5: Ongoing Operation

BGP only sends **incremental updates** — unlike OSPF, it does NOT periodically resend the entire routing table. It only sends changes (new routes or withdrawn routes).

### BGP Message Types:

| Message | Purpose |
|---------|---------|
| **OPEN** | Establish session, exchange parameters |
| **UPDATE** | Advertise new routes / withdraw old routes |
| **KEEPALIVE** | "I'm still here" (every 60 seconds) |
| **NOTIFICATION** | Error detected, closing the session |

### BGP Finite State Machine (Session States):

```
Idle → Connect → Active → OpenSent → OpenConfirm → Established
```

| State | What's Happening |
|-------|-----------------|
| **Idle** | BGP process just started, waiting to initiate connection |
| **Connect** | Trying to establish TCP connection to peer |
| **Active** | TCP connection failed, retrying (this is NOT a good state despite the name!) |
| **OpenSent** | TCP connected, OPEN message sent, waiting for peer's OPEN |
| **OpenConfirm** | OPEN received, parameters match, waiting for KEEPALIVE |
| **Established** | Session is up! Routes can be exchanged |

> ⚠️ **Common confusion:** The **Active** state is actually BAD — it means the router is actively trying (and failing) to connect. If you see a BGP session stuck in Active, there's a problem (usually TCP connectivity or misconfigured peer IP).

## 11. iBGP vs eBGP

### eBGP (External BGP)

BGP sessions between routers in **different** Autonomous Systems.

```
┌───────────┐          eBGP          ┌───────────┐
│  AS 65001 │◄───────────────────────►│  AS 65002 │
│  Router A │     TCP port 179       │  Router B │
└───────────┘                        └───────────┘
```

**Key eBGP Rules:**
- Peers are usually **directly connected** (TTL = 1 by default)
- eBGP routes have AD = 20 (highly trusted)
- The AS-PATH is incremented (your ASN is added to the path)
- Routes received from eBGP peers are **advertised to all other peers** by default

### iBGP (Internal BGP)

BGP sessions between routers in the **same** Autonomous System.

```
┌───────────────────────────────────────────────┐
│                   AS 65001                     │
│                                               │
│  ┌────────┐      iBGP       ┌────────┐       │
│  │Router A│◄───────────────►│Router B│       │
│  └────────┘                 └────────┘       │
│                                               │
└───────────────────────────────────────────────┘
```

**Why do we need iBGP?** Because when Router A learns an external route via eBGP, it needs to distribute that route to other routers within its own AS. That's what iBGP does.

**Key iBGP Rules:**
- Peers do NOT need to be directly connected (they can be many hops away, connected via OSPF/IGP)
- iBGP routes have AD = 200 (less trusted than eBGP)
- The AS-PATH is NOT incremented for iBGP
- **Critical Rule:** Routes learned from one iBGP peer are **NOT advertised to other iBGP peers** (to prevent loops)

### The iBGP Full Mesh Problem:

Because iBGP doesn't re-advertise routes to other iBGP peers, every iBGP router must peer with **every other iBGP router** = **full mesh**.

With 5 routers: 5×4/2 = 10 sessions (manageable)
With 50 routers: 50×49/2 = 1,225 sessions (painful)
With 500 routers: 500×499/2 = 124,750 sessions (impossible!)

**Solutions to the full mesh problem:**

1. **Route Reflectors** — One router acts as a "hub" and reflects routes to other iBGP peers
2. **Confederations** — Split the AS into sub-ASes internally

### Route Reflectors (Most Common Solution):

```
               ┌──────────┐
          ┌────► Route     ◄────┐
          │    │ Reflector │    │
          │    └─────┬─────┘    │
          │          │          │
     ┌────┴──┐  ┌───┴───┐  ┌──┴─────┐
     │Client1│  │Client2│  │Client3 │
     └───────┘  └───────┘  └────────┘
```

Instead of full mesh, all iBGP routers peer with the Route Reflector. The Route Reflector "reflects" routes from one client to all others. Problem solved!

## 12. BGP Path Selection — How Routes are Chosen

When BGP has multiple paths to the same destination (learned from different peers), it needs to pick the **best** one. BGP uses a very specific selection algorithm with many attributes, evaluated in order:

### The BGP Best Path Selection Algorithm (in order):

```
1.  Highest WEIGHT                  (Cisco proprietary, local to router)
2.  Highest LOCAL_PREF              (Within AS, default: 100)
3.  Locally originated route         (network command, redistribute, aggregate)
4.  Shortest AS_PATH                (fewer ASes = better)
5.  Lowest ORIGIN type              (IGP < EGP < Incomplete)
6.  Lowest MED                     (Multi-Exit Discriminator)
7.  eBGP over iBGP                  (external routes preferred)
8.  Lowest IGP metric to next-hop   (closest exit point)
9.  Oldest eBGP route               (stability)
10. Lowest Router ID of peer        (tiebreaker)
11. Lowest peer IP address           (final tiebreaker)
```

Let's understand the important ones:

### LOCAL_PREF (Local Preference)

- Only used within an AS (iBGP)
- **Higher is better** (opposite of most metrics!)
- Default: 100
- Used to tell your entire AS which exit point to prefer

```
Example: You have two ISP connections. Set LOCAL_PREF 200 on routes
from the preferred ISP, and LOCAL_PREF 100 (default) on the backup.
All routers in your AS will prefer the ISP with LOCAL_PREF 200.
```

### AS_PATH

- The list of ASes the route has passed through
- **Shorter is better** — fewer hops = more direct
- Also prevents loops — if a router sees its own ASN in the path, it rejects the route

```
Route 1: AS_PATH = {65002, 65003}           → 2 ASes (preferred)
Route 2: AS_PATH = {65004, 65005, 65006}    → 3 ASes
```

### MED (Multi-Exit Discriminator)

- Used to tell a **neighboring AS** which of your entry points to prefer
- **Lower is better**
- Only compared between routes from the **same neighboring AS**

```
You have two connections to ISP-X. You set MED 100 on link 1
and MED 200 on link 2. ISP-X will prefer link 1 (lower MED).
```

### Quick Memory Aid — "We Love Oranges AS Oranges Mean Pure Refreshment"

```
W - Weight (highest)
L - Local Preference (highest)
O - Originated locally
AS - AS Path (shortest)
O - Origin (lowest: IGP < EGP < ?)
M - MED (lowest)
P - Prefer eBGP over iBGP
R - Router ID (lowest)
```

## 14. BGP Communities & Route Filtering

### What are BGP Communities?

Communities are **tags** you attach to routes. They're like labels that say "treat this route in a special way." ISPs and large networks use communities extensively to implement routing policies.

A community is a 32-bit value, typically written as `ASN:value`:

```
65001:100    → "This is a customer route"
65001:200    → "This is a peer route"
65001:666    → "Blackhole this route" (used for DDoS mitigation!)
```

### Well-Known Communities:

| Community | Meaning |
|-----------|---------|
| `no-export` | Don't advertise this route to eBGP peers |
| `no-advertise` | Don't advertise this route to ANY peer |
| `internet` | Advertise everywhere (default behavior) |
| `no-peer` | Don't advertise to bilateral peers |

### Route Filtering — Why It Matters:

Route filtering is **critical** for BGP security and policy. Without it:
- A small ISP could accidentally announce they own Google's IP space (this has happened!)
- Your router could accept 1 million routes and crash from memory overload
- Invalid routes could redirect traffic through malicious networks

### Types of Filters:

**1. Prefix Lists — Filter by network prefix:**

```
! Only accept routes for 10.0.0.0/8 and its subnets
ip prefix-list ALLOWED-IN seq 10 permit 10.0.0.0/8 le 24
ip prefix-list ALLOWED-IN seq 20 deny 0.0.0.0/0 le 32

! Apply to neighbor
router bgp 65001
  neighbor 203.0.113.1 prefix-list ALLOWED-IN in
```

**2. AS-Path Filters — Filter by AS path:**

```
! Only accept routes that originated in AS 65002
bgp as-path access-list 1 permit ^65002$

! Apply to neighbor
router bgp 65001
  neighbor 203.0.113.1 filter-list 1 in
```

**3. Route Maps — Complex policies:**

```
! Set local preference 200 for routes from AS 65002
route-map PREFER-ISP-A permit 10
  match as-path 1
  set local-preference 200

route-map PREFER-ISP-A permit 20
  ! Allow everything else with default values

router bgp 65001
  neighbor 203.0.113.1 route-map PREFER-ISP-A in
```

### BGP Security — RPKI (Resource Public Key Infrastructure):

RPKI helps prevent **route hijacking** — where someone announces IP prefixes they don't own.

```
How RPKI works:
1. IP address owners create ROAs (Route Origin Authorizations)
   "AS 65001 is authorized to announce 203.0.113.0/24"
2. BGP routers validate routes against ROAs
3. Invalid routes are rejected or de-preferred

States:
  Valid    → ROA exists and matches
  Invalid  → ROA exists but doesn't match (REJECT this!)
  Unknown  → No ROA exists (accept but be cautious)
```

## 15. OSPF vs BGP — When to Use What

| Criteria | OSPF | BGP |
|----------|------|-----|
| **Use case** | Inside your network | Between networks / internet |
| **Scale** | Up to ~10,000 routes | Millions of routes |
| **Protocol** | Link-State (IP 89) | Path-Vector (TCP 179) |
| **Speed** | Fast convergence | Slower convergence |
| **Policy** | Limited (just cost) | Rich policy control |
| **Configuration** | Simpler | Complex |
| **Auto-discovery** | Yes (Hello packets) | No (manual peers) |
| **Loop prevention** | SPF algorithm | AS-PATH |

### Decision Matrix:

```
Routing WITHIN your data center?          → OSPF (or IS-IS)
Routing WITHIN your company WAN?          → OSPF
Connecting to the internet?               → BGP (mandatory)
Connecting to a cloud provider?           → BGP (via VPN/Direct Connect)
Multi-homed (multiple ISP connections)?   → BGP
Peering at an internet exchange?          → BGP
Inside a Kubernetes cluster?              → Neither (use CNI)
Between Kubernetes clusters?              → BGP (Calico uses BGP!)
```

### Can You Use Both Together?

Absolutely! In fact, most real networks use BOTH:

```
                Internet
                   │
              ┌────┴────┐
              │  Border  │  ← BGP here (eBGP to ISPs)
              │  Router  │
              └────┬────┘
                   │
              ┌────┴────┐
              │  Core    │  ← OSPF here (internal routing)
              │  Router  │
              └────┬────┘
                   │
         ┌────────┴────────┐
    ┌────┴────┐      ┌────┴────┐
    │  Dist   │      │  Dist   │  ← OSPF here
    │ Router 1│      │ Router 2│
    └────┬────┘      └────┬────┘
         │                │
    ┌────┴────┐      ┌────┴────┐
    │ Access  │      │ Access  │  ← OSPF here
    │Switch 1 │      │Switch 2 │
    └─────────┘      └─────────┘
```

BGP is used at the edge (connecting to ISPs and the internet), while OSPF runs internally. The border router **redistributes** routes between BGP and OSPF.

## 16. Other Routing Protocols (RIP, EIGRP, IS-IS)

### RIP (Routing Information Protocol)

The **oldest** dynamic routing protocol. Simple but very limited.

| Feature | Detail |
|---------|--------|
| Type | Distance-Vector |
| Metric | Hop count (max 15 — 16 = unreachable) |
| Updates | Full table every 30 seconds (wasteful!) |
| Convergence | Slow (minutes) |
| Use today? | Almost never — legacy only |

```
How RIP works (simplified):
Router A: "I can reach 10.0.0.0/8 in 1 hop"
Router B: "I can reach 10.0.0.0/8 in 2 hops (through A)"
Router C: "I can reach 10.0.0.0/8 in 3 hops (through B)"

Problem: Max 15 hops → can't use in large networks
```

### EIGRP (Enhanced Interior Gateway Routing Protocol)

Cisco's proprietary protocol (recently opened as an informational RFC, but still 99% Cisco).

| Feature | Detail |
|---------|--------|
| Type | Advanced Distance-Vector / Hybrid |
| Metric | Composite (bandwidth + delay + reliability + load) |
| Algorithm | DUAL (Diffusing Update Algorithm) |
| Convergence | Very fast (sub-second!) |
| Use today? | Only in Cisco-only networks |

**Advantage:** Extremely fast convergence and easy to configure on Cisco gear.
**Disadvantage:** Cisco-proprietary, doesn't work on non-Cisco equipment.

### IS-IS (Intermediate System to Intermediate System)

The "other" link-state protocol. Used by very large ISPs and service providers.

| Feature | Detail |
|---------|--------|
| Type | Link-State (like OSPF) |
| Metric | Configurable cost (default: 10 per interface) |
| Protocol | Runs directly on Layer 2 (not IP!) |
| Convergence | Fast |
| Use today? | Large ISPs, service providers, some data centers |

**Why some prefer IS-IS over OSPF:**
- Slightly simpler to configure at massive scale
- More stable (runs on L2, not affected by IP issues)
- Used by Facebook, Google, and other hyperscalers internally
- Better multi-topology support

**For most DevOps engineers:** You'll use OSPF (simpler, more common in enterprise). IS-IS is mainly for ISP/carrier networks.

## 18. Routing in Kubernetes & Container Networking

### Kubernetes Uses Routing Heavily!

Every Kubernetes cluster is a complex routed network:

```
┌─────────────────────────────────────────────────────────────┐
│                     Kubernetes Cluster                        │
│                                                              │
│  Node 1 (10.0.1.0/24)          Node 2 (10.0.2.0/24)        │
│  ┌──────────────────┐          ┌──────────────────┐         │
│  │ Pod A: 10.0.1.5  │          │ Pod C: 10.0.2.5  │         │

│  └────────┬─────────┘          └────────┬─────────┘         │
│           │                             │                    │
│           └─────────── Routes ──────────┘                    │
│                                                              │
│  Each node knows: "To reach 10.0.2.0/24, go via Node 2"    │
└─────────────────────────────────────────────────────────────┘
```

### Calico Uses BGP!

**Calico**, one of the most popular Kubernetes CNI plugins, uses **BGP** to distribute pod network routes between nodes:

```bash
# Check Calico BGP peers
sudo calicoctl node status

# Output:
# IPv4 BGP status
# +--------------+-------------------+-------+----------+-------+
# | PEER ADDRESS |     PEER TYPE     | STATE |  SINCE   | INFO  |
# +--------------+-------------------+-------+----------+-------+
# | 10.0.0.2     | node-to-node mesh | up    | 12:00:00 | Established |
# | 10.0.0.3     | node-to-node mesh | up    | 12:00:00 | Established |
# +--------------+-------------------+-------+----------+-------+
```

Each Kubernetes node runs a BGP speaker (via Calico's BIRD daemon) and exchanges pod CIDR routes with other nodes. **You're running BGP without even knowing it!**

### Viewing Routes on a Kubernetes Node:

```bash
# SSH into a Kubernetes node and check routes
ip route show

# You'll see routes like:
# 10.0.1.0/24 dev cali123 scope link    ← Local pod network
# 10.0.2.0/24 via 192.168.1.2 dev eth0  ← Route to pods on Node 2
# 10.0.3.0/24 via 192.168.1.3 dev eth0  ← Route to pods on Node 3
# blackhole 10.0.1.0/24                  ← Prevent loops for local CIDR
```

### MetalLB — BGP for Kubernetes LoadBalancer Services:

In bare-metal Kubernetes clusters (no cloud), **MetalLB** uses BGP to announce LoadBalancer IP addresses:

```yaml
# MetalLB BGP configuration
apiVersion: metallb.io/v1beta2
kind: BGPPeer
metadata:
  name: upstream-router
  namespace: metallb-system
spec:
  myASN: 64512
  peerASN: 65001
  peerAddress: 10.0.0.1
---
apiVersion: metallb.io/v1beta1
kind: IPAddressPool
metadata:
  name: production-pool
  namespace: metallb-system
spec:
  addresses:
    - 203.0.113.100-203.0.113.200
---
apiVersion: metallb.io/v1beta1
kind: BGPAdvertisement
metadata:
  name: production-advertisement
  namespace: metallb-system
spec:
  ipAddressPools:
    - production-pool
```

This tells MetalLB: "Use BGP to announce IPs from 203.0.113.100-200 to the upstream router."

## 19. Troubleshooting Routing Issues

### Common Routing Problems & Solutions:

### Problem 1: "Can't reach a network"

```bash
# Step 1: Check if route exists
ip route show | grep <destination-network>

# Step 2: Check the path packets take
traceroute <destination-ip>
# or
mtr <destination-ip>       # Better — shows packet loss per hop

# Step 3: Check if the interface is up
ip link show
ip addr show

# Step 4: Check if the next-hop is reachable
ping <next-hop-ip>

# Step 5: Check ARP table
ip neigh show
```

### Problem 2: "OSPF neighbors not forming"

```bash
# Check OSPF neighbors
sudo vtysh -c "show ip ospf neighbor"

# Common causes:
# 1. Mismatched area IDs
# 2. Mismatched Hello/Dead intervals
# 3. Mismatched authentication
# 4. Mismatched subnet masks
# 5. Firewall blocking OSPF (IP protocol 89)
# 6. Interface not in OSPF

# Debug OSPF
sudo vtysh -c "debug ospf packet hello"
sudo vtysh -c "show ip ospf interface"
```

### Problem 3: "BGP session stuck in Active"

```bash
# Check BGP session state
sudo vtysh -c "show bgp summary"

# Common causes:
# 1. TCP port 179 blocked by firewall
# 2. Wrong peer IP address configured
# 3. Wrong ASN configured
# 4. Can't reach peer (routing issue to the peer itself)
# 5. Authentication mismatch (MD5 password)

# Debug BGP
sudo vtysh -c "show bgp neighbors <peer-ip>"
# Look for "Last error" and "BGP state"

# Check TCP connectivity
nc -zv <peer-ip> 179

# Check firewall
sudo iptables -L -n | grep 179
```

### Problem 4: "Route learned but not installed in routing table"

```bash
# Check if route exists in protocol database
sudo vtysh -c "show bgp ipv4 unicast"    # for BGP
sudo vtysh -c "show ip ospf route"       # for OSPF

# Common causes:
# 1. Better route exists from another source (lower AD)
# 2. Next-hop unreachable
# 3. Route filtered by policy
# 4. Maximum prefix limit reached (BGP)

# Check routing table
ip route show
sudo vtysh -c "show ip route"
```

### Problem 5: "Asymmetric routing" (traffic goes out one path, comes back another)

```bash
# Trace the forward path
traceroute <destination>

# Then from the destination, trace back
traceroute <source>

# If paths are different → asymmetric routing
# This can cause issues with stateful firewalls and NAT

# Solutions:
# - Adjust OSPF costs to make paths symmetric
# - Use BGP LOCAL_PREF or MED to influence path
# - Configure firewall to handle asymmetric routing
```

### Useful Troubleshooting Commands Summary:

```bash
# General routing
ip route show                    # View routing table
ip route get <destination>       # Which route would be used for this destination?
traceroute <destination>         # Show the path
mtr <destination>               # Continuous traceroute with packet loss stats

# OSPF (via FRR/vtysh)
show ip ospf neighbor            # OSPF adjacencies
show ip ospf database            # Link-State Database
show ip ospf route               # OSPF routing table
show ip ospf interface           # Interface OSPF status
debug ospf packet all            # Debug OSPF packets (careful — verbose!)

# BGP (via FRR/vtysh)
show bgp summary                 # BGP session overview
show bgp neighbors               # Detailed peer info
show bgp ipv4 unicast            # Full BGP table
show bgp ipv4 unicast <prefix>   # Specific prefix info
show ip route bgp                # BGP routes in routing table
show bgp ipv4 unicast community  # Routes with communities
debug bgp updates                # Debug BGP updates

# Network connectivity
ping <ip>                        # Basic connectivity
nc -zv <ip> <port>              # TCP port check
ss -tlnp                        # Listening ports
tcpdump -i eth0 port 179        # Capture BGP traffic
tcpdump -i eth0 proto ospf      # Capture OSPF traffic
```

## 20. Summary

### Routing at a Glance:

| Concept | Key Point |
|---------|-----------|
| **Routing** | Finding the best path for packets across networks |
| **Static Routing** | Manually configured, simple, doesn't scale |
| **Dynamic Routing** | Automatic, scales, adapts to changes |
| **Routing Table** | Master list of "destination → next hop" |
| **Longest Prefix Match** | Most specific route wins |
| **Administrative Distance** | Which protocol is more trusted |

### OSPF at a Glance:

| Concept | Key Point |
|---------|-----------|
| **Type** | Interior Gateway Protocol (IGP) |
| **Algorithm** | Dijkstra (Shortest Path First) |
| **Metric** | Cost (bandwidth-based) |
| **How it works** | Build a map (LSDB), calculate shortest paths |
| **Areas** | Hierarchical design for scalability |
| **Area 0** | Backbone — mandatory, all areas connect to it |
| **Convergence** | Fast (seconds) |
| **When to use** | Inside your organization/data center |

### BGP at a Glance:

| Concept | Key Point |
|---------|-----------|
| **Type** | Exterior Gateway Protocol (EGP) |
| **Protocol** | TCP port 179 |
| **Focus** | Policy-based routing between organizations |
| **eBGP** | Between different ASes (AD = 20) |
| **iBGP** | Within the same AS (AD = 200) |
| **Key attribute** | AS-PATH (list of ASes traversed) |
| **Path selection** | Weight → Local Pref → AS Path → MED → etc. |
| **When to use** | Internet, multi-ISP, cloud connectivity |
| **DevOps relevance** | Calico (K8s), MetalLB, cloud VPNs, Direct Connect |

### The Complete Picture:

```
┌────────────────────────────────────────────────────────────┐
│                        THE INTERNET                         │
│                     (BGP everywhere)                        │
│                                                            │
│    AS 65001          AS 65002          AS 65003            │
│  ┌─────────┐      ┌─────────┐      ┌─────────┐          │
│  │  ISP-A  │◄────►│  ISP-B  │◄────►│ Google  │          │
│  └────┬────┘      └─────────┘      └─────────┘          │
│       │ eBGP                                              │
└───────┼──────────────────────────────────────────────────┘
        │
  ┌─────┴──────────────────────────────────────────┐
  │            YOUR NETWORK (AS 64512)              │
  │                                                 │
  │  Border Router ← BGP (eBGP to ISP)            │
  │       │                                         │
  │  Core Router ← OSPF (internal routing)         │
  │       │                                         │
  │  ┌────┴──────┐                                 │
  │  │  K8s      │ ← BGP (Calico between nodes)   │
  │  │  Cluster  │ ← Network Policies (pod firewall)│
  │  └───────────┘                                  │
  └─────────────────────────────────────────────────┘
```

> **Bottom line:** OSPF handles routing inside your network (fast, automatic, shortest path). BGP handles routing between networks and across the internet (policy-based, scalable, the glue of the internet). As a DevOps engineer, you'll encounter BGP in Kubernetes (Calico, MetalLB), cloud connectivity (VPNs, Direct Connect), and multi-cloud architectures. OSPF shows up in data center and enterprise internal routing.
