## 🤔 Why Should a DevOps Engineer Care About IP Addressing?

IP addresses are literally **everywhere** in your DevOps life:

| What You Do | IP Knowledge Needed |
|-------------|-------------------|
| SSH into a server | You need the server's IP address |
| Set up a VPC in AWS/Azure/GCP | You define IP ranges (CIDR blocks) |
| Configure Kubernetes networking | Pods, Services, Nodes all have IPs |
| Set up a database connection | Connection string includes an IP or hostname |
| Write firewall rules | Allow/block traffic by IP |
| Set up a load balancer | Route traffic to backend server IPs |
| Debug "connection refused" errors | Is the IP correct? Is the host reachable? |
| Design a multi-region architecture | Subnetting across regions |

If you don't understand IP addressing, you're essentially trying to deliver mail without understanding how addresses work.

## 📬 What is an IP Address?

### The Simple Explanation

An **IP address** (Internet Protocol address) is like a **home address for your computer** on a network. Just like your house needs a unique address so the postman can deliver mail to you, every device on a network needs a unique IP address so data packets can find it.

### Two Versions

There are two versions of IP addresses in use today:

| Version | Example | Created | Status |
|---------|---------|---------|--------|
| **IPv4** | `192.168.1.100` | 1981 | Still dominant (~80% of internet traffic) |
| **IPv6** | `2001:0db8:85a3:0000:0000:8a2e:0370:7334` | 1998 | Growing (~30% and increasing) |

**Why two versions?** Because IPv4 ran out of addresses! (We'll explain this in detail.)

## 🔢 IPv4 — The Workhorse of the Internet

### How IPv4 Addresses Look

An IPv4 address is a **32-bit number** written as **4 groups of numbers** separated by dots. Each group is called an **octet** (because it's 8 bits).

```
IPv4 Address:   192  .  168  .  1  .  100
                 │       │      │     │
                 │       │      │     └─ 4th Octet (8 bits)
                 │       │      └─────── 3rd Octet (8 bits)
                 │       └────────────── 2nd Octet (8 bits)
                 └────────────────────── 1st Octet (8 bits)

Total: 4 octets × 8 bits = 32 bits
```

### The Binary Behind It

Each octet is actually an 8-bit binary number (0-255):

```
Decimal:    192    .    168    .      1    .    100
Binary:  11000000  .  10101000  .  00000001  .  01100100

How to convert 192 to binary:
128  64  32  16  8  4  2  1
 1    1   0   0  0  0  0  0  = 128 + 64 = 192 ✅

How to convert 168 to binary:
128  64  32  16  8  4  2  1
 1    0   1   0  1  0  0  0  = 128 + 32 + 8 = 168 ✅
```

### How Many IPv4 Addresses Exist?

```
32 bits → 2^32 = 4,294,967,296 addresses (about 4.3 billion)
```

Sounds like a lot, right? **Nope.** There are about 8 billion people on Earth, plus billions of devices (phones, laptops, servers, IoT devices, smart fridges...). We literally ran out of IPv4 addresses in **2011** when the last blocks were allocated.

That's why we need IPv6 (more on that later) and why techniques like **NAT** were invented (to share one public IP among many devices).

---

### IPv4 Address Classes

IPv4 addresses are divided into **5 classes** (A through E). This is the original way IP addresses were organized.

```
┌───────┬─────────────────────────┬────────────────────────┬───────────────────┐
│ Class │    IP Range             │  Default Subnet Mask   │    Purpose        │
├───────┼─────────────────────────┼────────────────────────┼───────────────────┤
│   A   │ 1.0.0.0 - 126.0.0.0    │ 255.0.0.0 (/8)         │ Large networks    │
│   B   │ 128.0.0.0 - 191.255.0.0│ 255.255.0.0 (/16)      │ Medium networks   │
│   C   │ 192.0.0.0 - 223.255.255.0│ 255.255.255.0 (/24)  │ Small networks    │
│   D   │ 224.0.0.0 - 239.255.255.255│ N/A                 │ Multicast         │
│   E   │ 240.0.0.0 - 255.255.255.255│ N/A                 │ Experimental      │
└───────┴─────────────────────────┴────────────────────────┴───────────────────┘
```

**How to quickly identify the class:**

```
First octet:
  1 - 126   → Class A
  128 - 191  → Class B
  192 - 223  → Class C
  224 - 239  → Class D (Multicast)
  240 - 255  → Class E (Experimental)

Note: 127.x.x.x is reserved for loopback (127.0.0.1 = localhost)
```

> **Important:** Classful addressing is **outdated**. Modern networking uses **CIDR** (Classless Inter-Domain Routing) instead. But you need to know classes for interviews and to understand the history.

### Private vs Public IP Addresses

This is a **must-know** concept!

Not all IP addresses are routable on the public internet. Some are reserved for **private/internal** use:

```
┌──────────────────────────────────────────────────────────────────┐
│                    PRIVATE IP RANGES                              │
├────────┬─────────────────────────────┬───────────────────────────┤
│ Class  │ Range                       │ How Many Addresses?       │
├────────┼─────────────────────────────┼───────────────────────────┤
│ A      │ 10.0.0.0 - 10.255.255.255  │ 16,777,216 addresses      │
│ B      │ 172.16.0.0 - 172.31.255.255│ 1,048,576 addresses       │
│ C      │ 192.168.0.0 - 192.168.255.255│ 65,536 addresses        │
└────────┴─────────────────────────────┴───────────────────────────┘
```

**The difference in simple words:**

```
PUBLIC IP:
  → Unique across the entire internet
  → Assigned by your ISP
  → Used to communicate on the internet
  → Example: 203.45.67.89

PRIVATE IP:
  → Only unique within YOUR local network
  → Assigned by your router (via DHCP)
  → Cannot be accessed directly from the internet
  → Example: 192.168.1.100

Your network looks like this:

                    Internet
                       │
                 ┌─────┴─────┐
                 │  Router    │  Public IP: 203.45.67.89
                 │  (NAT)    │
                 └─────┬─────┘
           ┌───────────┼───────────┐
           │           │           │
     ┌─────┴─────┐ ┌──┴────┐ ┌───┴─────┐
     │  Laptop   │ │ Phone │ │  PC     │
     │192.168.1.2│ │.1.3   │ │.1.4    │
     └───────────┘ └───────┘ └────────┘

All three devices share ONE public IP (203.45.67.89) using NAT
(Network Address Translation)
```

**DevOps Relevance:**
- **Cloud VPCs** use private IP ranges (usually `10.0.0.0/8` or `172.16.0.0/12`)
- **Kubernetes Pods** get private IPs from the cluster's CIDR range
- **Docker containers** get private IPs from the Docker network (usually `172.17.0.0/16`)
- **Bastion hosts** have public IPs; backend servers use private IPs

### Special IP Addresses You Must Know

| IP Address | Purpose |
|-----------|---------|
| `127.0.0.1` | **Loopback** — refers to "this machine itself" (localhost) |
| `0.0.0.0` | **All interfaces** — listen on every available IP |
| `255.255.255.255` | **Broadcast** — send to all devices on the network |
| `169.254.x.x` | **APIPA** — auto-assigned when DHCP fails |
| `8.8.8.8` | Google's public DNS server |
| `1.1.1.1` | Cloudflare's public DNS server |

```bash
# Check your IP addresses
ip addr show

# Find your public IP (what the internet sees)
curl ifconfig.me
# or
curl icanhazip.com

# Loopback test
ping 127.0.0.1
ping localhost

# Listen on all interfaces (0.0.0.0)
# Example: running a web server
python3 -m http.server 8080 --bind 0.0.0.0
```

## 🎭 Subnet Mask — What is It and Why Do We Need It?

A **subnet mask** tells you which part of an IP address is the **network** and which part is the **host**.

### The Concept

Think of an IP address like a phone number:

```
Phone number:   (555)  -  1234
                 │          │
                 Area Code  │
                 (Network)  │
                            │
                   Local Number
                   (Host)

IP address:     192.168.1    .   100
                 │                │
                 Network part     │
                                  │
                             Host part
```

The **subnet mask** defines where the split happens.

### How Subnet Masks Work

```
IP Address:     192.168.1.100
Subnet Mask:    255.255.255.0

In binary:
IP:     11000000.10101000.00000001.01100100
Mask:   11111111.11111111.11111111.00000000
        ├───── Network (1s) ─────┤├─ Host ─┤

Where the mask has 1s = Network portion
Where the mask has 0s = Host portion

So:
  Network address: 192.168.1.0
  Host part:       .100
  Broadcast:       192.168.1.255
  Usable range:    192.168.1.1 to 192.168.1.254
```

### Common Subnet Masks

| Subnet Mask | CIDR | Binary | Network Bits | Host Bits | Usable Hosts |
|-------------|------|--------|-------------|-----------|-------------|
| 255.0.0.0 | /8 | 11111111.00000000.00000000.00000000 | 8 | 24 | 16,777,214 |
| 255.255.0.0 | /16 | 11111111.11111111.00000000.00000000 | 16 | 16 | 65,534 |
| 255.255.255.0 | /24 | 11111111.11111111.11111111.00000000 | 24 | 8 | 254 |
| 255.255.255.128 | /25 | 11111111.11111111.11111111.10000000 | 25 | 7 | 126 |
| 255.255.255.192 | /26 | 11111111.11111111.11111111.11000000 | 26 | 6 | 62 |
| 255.255.255.224 | /27 | 11111111.11111111.11111111.11100000 | 27 | 5 | 30 |
| 255.255.255.240 | /28 | 11111111.11111111.11111111.11110000 | 28 | 4 | 14 |
| 255.255.255.248 | /29 | 11111111.11111111.11111111.11111000 | 29 | 3 | 6 |
| 255.255.255.252 | /30 | 11111111.11111111.11111111.11111100 | 30 | 2 | 2 |
| 255.255.255.255 | /32 | 11111111.11111111.11111111.11111111 | 32 | 0 | 1 |

> **Formula:** Usable hosts = 2^(host bits) - 2
> (Subtract 2 because the first address is the **network address** and the last is the **broadcast address**)

## 🧮 CIDR — Classless Inter-Domain Routing

CIDR (pronounced "cider") is the **modern way** of defining IP address ranges. Instead of using classes (A, B, C), we use a **slash notation** to specify exactly how many bits are for the network.

### CIDR Notation

```
192.168.1.0/24
  │            │
  │            └── /24 means "the first 24 bits are the network part"
  │                which means 32 - 24 = 8 bits for hosts
  │                = 2^8 - 2 = 254 usable host addresses
  │
  └── Base network address

Some more examples:

10.0.0.0/8      → 10.0.0.0 to 10.255.255.255     (16 million+ hosts!)
172.16.0.0/16   → 172.16.0.0 to 172.16.255.255   (65,534 hosts)
192.168.1.0/24  → 192.168.1.0 to 192.168.1.255   (254 hosts)
192.168.1.0/28  → 192.168.1.0 to 192.168.1.15    (14 hosts)
10.0.0.5/32     → Just 10.0.0.5 (a single host)
0.0.0.0/0       → ALL IP addresses (default route / "everything")
```

### Why CIDR Matters for DevOps

You will see CIDR notation **constantly** in DevOps:

```yaml
# AWS VPC CIDR block (Terraform)
resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"    # 65,534 IPs for our entire VPC
}

# Subnets within the VPC
resource "aws_subnet" "public" {
  cidr_block = "10.0.1.0/24"   # 254 IPs for public subnet
}

resource "aws_subnet" "private" {
  cidr_block = "10.0.2.0/24"   # 254 IPs for private subnet
}
```

```yaml
# Kubernetes Pod CIDR
apiVersion: kubeadm.k8s.io/v1beta3
kind: ClusterConfiguration
networking:
  podSubnet: "10.244.0.0/16"      # IP range for all pods
  serviceSubnet: "10.96.0.0/12"   # IP range for all services
```

```yaml
# Security Group / Firewall Rule
# Allow SSH only from your office IP
ingress:
  - from_port: 22
    to_port: 22
    cidr_blocks: ["203.45.67.0/24"]  # Only your office's IP range

# Allow traffic from anywhere
ingress:
  - from_port: 80
    to_port: 80
    cidr_blocks: ["0.0.0.0/0"]      # Open to the entire internet
```

## 🧩 Subnetting — Dividing a Network into Smaller Parts

Subnetting is the process of **splitting a large network into smaller, more manageable sub-networks** (subnets).

### Why Do We Subnet?

```
Without Subnetting:                    With Subnetting:
──────────────────                     ─────────────────

One huge network:                      Split into logical subnets:
10.0.0.0/16                           
(65,534 devices all in                 10.0.1.0/24 → Web Servers (254 IPs)
 one flat network 😱)                  10.0.2.0/24 → App Servers (254 IPs)
                                       10.0.3.0/24 → Databases (254 IPs)
                                       10.0.4.0/24 → Management (254 IPs)

Problems:                              Benefits:
❌ Security: everyone can see everyone ✅ Security: control traffic between subnets
❌ Performance: too much broadcast      ✅ Performance: less broadcast traffic
❌ Management: chaos                    ✅ Organization: logical grouping
❌ Hard to troubleshoot                 ✅ Easy to troubleshoot
```

### Subnetting Example — Step by Step

Let's subnet `192.168.1.0/24` into **4 smaller subnets**.

```
Original Network: 192.168.1.0/24
  → 256 total IPs (254 usable)
  → We want 4 subnets

Step 1: How many extra bits do we need?
  → 4 subnets = 2^2 → We need 2 extra bits for the network
  → New subnet mask: /24 + 2 = /26

Step 2: How many hosts per subnet?
  → /26 means 32 - 26 = 6 host bits
  → 2^6 - 2 = 62 usable hosts per subnet

Step 3: Calculate the subnets
  → Block size = 256 / 4 = 64 (or 2^6 = 64)

┌─────────────┬─────────────────────┬───────────────────────┬───────────────┐
│   Subnet #  │   Network Address   │   Usable Range        │   Broadcast   │
├─────────────┼─────────────────────┼───────────────────────┼───────────────┤
│   Subnet 1  │   192.168.1.0/26    │   192.168.1.1 - .62   │ 192.168.1.63  │
│   Subnet 2  │   192.168.1.64/26   │   192.168.1.65 - .126 │ 192.168.1.127 │
│   Subnet 3  │   192.168.1.128/26  │   192.168.1.129 - .190│ 192.168.1.191 │
│   Subnet 4  │   192.168.1.192/26  │   192.168.1.193 - .254│ 192.168.1.255 │
└─────────────┴─────────────────────┴───────────────────────┴───────────────┘
```

### Real-World Cloud Subnetting Example

Here's how you might subnet a VPC in AWS:

```
VPC CIDR: 10.0.0.0/16 (65,534 total IPs)

├── Availability Zone A
│   ├── Public Subnet:   10.0.1.0/24  (254 IPs) — Load Balancers, NAT Gateway
│   ├── Private Subnet:  10.0.10.0/24 (254 IPs) — App Servers
│   └── Database Subnet: 10.0.20.0/24 (254 IPs) — RDS, ElastiCache
│
├── Availability Zone B
│   ├── Public Subnet:   10.0.2.0/24  (254 IPs) — Load Balancers, NAT Gateway
│   ├── Private Subnet:  10.0.11.0/24 (254 IPs) — App Servers
│   └── Database Subnet: 10.0.21.0/24 (254 IPs) — RDS, ElastiCache
│
└── Availability Zone C
    ├── Public Subnet:   10.0.3.0/24  (254 IPs) — Load Balancers, NAT Gateway
    ├── Private Subnet:  10.0.12.0/24 (254 IPs) — App Servers
    └── Database Subnet: 10.0.22.0/24 (254 IPs) — RDS, ElastiCache
```

This is what you do in real life when building cloud architectures!

## 🚀 IPv6 — The Future of IP Addressing

### Why Do We Need IPv6?

```
IPv4: 2^32 = ~4.3 billion addresses    → RAN OUT in 2011! 😱
IPv6: 2^128 = 340 undecillion addresses → That's 340,282,366,920,938,463,463,374,607,431,768,211,456

To put IPv6 into perspective:
  → Every grain of sand on Earth could have BILLIONS of IPv6 addresses
  → We will NEVER run out
```

### How IPv6 Addresses Look

```
Full IPv6 Address:
  2001:0db8:85a3:0000:0000:8a2e:0370:7334

  8 groups of 4 hexadecimal digits, separated by colons

  Each group = 16 bits
  8 groups × 16 bits = 128 bits total
```

### IPv6 Shortening Rules

IPv6 addresses are long, so there are rules to shorten them:

```
Rule 1: Remove leading zeros in each group
  2001:0db8:0042:0000:0000:8a2e:0370:7334
  → 2001:db8:42:0:0:8a2e:370:7334

Rule 2: Replace consecutive groups of all zeros with ::  (only ONCE!)
  2001:db8:42:0:0:8a2e:370:7334
  → 2001:db8:42::8a2e:370:7334

More examples:
  Full:       fe80:0000:0000:0000:0000:0000:0000:0001
  Shortened:  fe80::1

  Full:       0000:0000:0000:0000:0000:0000:0000:0001
  Shortened:  ::1    (This is the IPv6 loopback! Like 127.0.0.1)
```

### IPv4 vs IPv6 Comparison

| Feature | IPv4 | IPv6 |
|---------|------|------|
| **Address size** | 32 bits | 128 bits |
| **Address format** | Dotted decimal (192.168.1.1) | Hexadecimal colons (2001:db8::1) |
| **Total addresses** | ~4.3 billion | ~340 undecillion |
| **Example** | `192.168.1.100` | `2001:db8:85a3::8a2e:370:7334` |
| **Loopback** | `127.0.0.1` | `::1` |
| **All interfaces** | `0.0.0.0` | `::` |
| **Configuration** | Manual or DHCP | Auto-configuration (SLAAC) |
| **NAT needed?** | Yes (due to address shortage) | No (enough addresses for everyone) |
| **IPsec** | Optional | Built-in |
| **Header size** | Variable (20-60 bytes) | Fixed (40 bytes) |
| **Broadcast** | Yes | No (uses multicast instead) |

### Special IPv6 Addresses

| Address | Purpose |
|---------|---------|
| `::1` | Loopback (same as `127.0.0.1`) |
| `::` | All interfaces (same as `0.0.0.0`) |
| `fe80::/10` | Link-local (auto-assigned, like `169.254.x.x` in IPv4) |
| `fc00::/7` | Unique Local Address (like private IPs in IPv4) |
| `2000::/3` | Global Unicast (public, routable on the internet) |
| `ff00::/8` | Multicast |

```bash
# View IPv6 addresses on your system
ip -6 addr show

# Ping using IPv6
ping6 ::1
# or
ping -6 google.com

# View IPv6 routing table
ip -6 route show

# DNS lookup for IPv6
dig AAAA google.com

# curl using IPv6
curl -6 http://[::1]:8080
```

## 🧠 DHCP — How Devices Get Their IP Automatically

DHCP (Dynamic Host Configuration Protocol) automatically assigns IP addresses to devices on a network.

### The DHCP Process — DORA

```
Client                                DHCP Server
  │                                       │
  │ ── 1. DISCOVER (broadcast) ─────────► │  "Hey, anyone got an IP for me?"
  │                                       │
  │ ◄── 2. OFFER ──────────────────────── │  "Sure! How about 192.168.1.50?"
  │                                       │
  │ ── 3. REQUEST ─────────────────────► │  "Yes, I'll take 192.168.1.50!"
  │                                       │
  │ ◄── 4. ACKNOWLEDGE ───────────────── │  "It's yours! Lease: 24 hours"
  │                                       │
  │     Now client has IP: 192.168.1.50   │

DORA = Discover → Offer → Request → Acknowledge
```

**DevOps Relevance:**
- Cloud instances get private IPs via DHCP from the cloud provider
- Docker containers get IPs via Docker's built-in DHCP
- Kubernetes pods get IPs from the CNI (Container Network Interface) plugin
- **Static IPs** are used for servers that need a permanent address (databases, load balancers)

```bash
# Release and renew DHCP lease (Linux)
sudo dhclient -r       # Release current IP
sudo dhclient           # Get a new IP

# View DHCP lease info
cat /var/lib/dhcp/dhclient.leases
```

## 🔥 NAT — Network Address Translation

NAT allows **multiple devices** to share **one public IP address**. This is how your home/office works!

```
Without NAT:                              With NAT:
──────────                                ────────
Every device needs                        All devices share ONE
its own public IP                         public IP address
(impossible — not                         
enough IPv4 addresses!)                   Private IPs → Router (NAT) → Public IP

┌──────────────────────────────────────────────────────────────┐
│                         NAT in Action                         │
│                                                               │
│  Laptop (192.168.1.2)  ──┐                                    │
│  Phone  (192.168.1.3)  ──┼── Router/NAT ── 203.45.67.89 ── Internet
│  PC     (192.168.1.4)  ──┘   (Translates                     │
│                              private → public)                │
│                                                               │
│  Outgoing: 192.168.1.2:54321 → NAT → 203.45.67.89:54321     │
│  Incoming: 203.45.67.89:54321 → NAT → 192.168.1.2:54321     │
└──────────────────────────────────────────────────────────────┘
```

### Types of NAT

| Type | Description | Use Case |
|------|------------|----------|
| **SNAT** (Source NAT) | Changes the source IP of outgoing packets | Internal devices accessing the internet |
| **DNAT** (Destination NAT) | Changes the destination IP of incoming packets | Port forwarding, load balancing |
| **PAT** (Port Address Translation) | Multiple devices share one IP using different ports | Home routers (most common) |

**DevOps Relevance:**
- **AWS NAT Gateway** — allows private subnet instances to access the internet
- **Docker** uses NAT for port mapping (`-p 8080:80`)
- **Kubernetes NodePort** uses NAT to forward traffic from node to pod
- **iptables** is the Linux tool that implements NAT

```bash
# View NAT rules on Linux (iptables)
sudo iptables -t nat -L -n -v

# Docker's NAT rules (created automatically)
sudo iptables -t nat -L DOCKER -n -v

# AWS NAT Gateway (Terraform example)
resource "aws_nat_gateway" "main" {
  allocation_id = aws_eip.nat.id
  subnet_id     = aws_subnet.public.id
}
```

## 🎯 Interview Questions — IP Addressing

### Q1: What's the difference between a public and private IP?

> **Public IPs** are globally unique and routable on the internet (assigned by ISPs). **Private IPs** (10.x.x.x, 172.16-31.x.x, 192.168.x.x) are only unique within a local network and can't be accessed directly from the internet. NAT is used to translate between them.

### Q2: What is CIDR? Give an example.

> CIDR (Classless Inter-Domain Routing) is a method of defining IP address ranges using slash notation. For example, `10.0.0.0/24` means the first 24 bits are the network part, giving us 256 addresses (254 usable). It replaced the old classful addressing system for more flexible allocation.

### Q3: How do you subnet 10.0.0.0/24 into 4 equal subnets?

> Borrow 2 bits from the host portion (/24 → /26). This creates 4 subnets, each with 62 usable hosts:
> - 10.0.0.0/26 (10.0.0.1 - 10.0.0.62)
> - 10.0.0.64/26 (10.0.0.65 - 10.0.0.126)
> - 10.0.0.128/26 (10.0.0.129 - 10.0.0.190)
> - 10.0.0.192/26 (10.0.0.193 - 10.0.0.254)

### Q4: What is NAT and why is it needed?

> NAT (Network Address Translation) translates private IPs to public IPs and vice versa. It's needed because there aren't enough IPv4 addresses for every device. NAT allows many devices to share a single public IP. In cloud environments, NAT Gateways let private subnet instances access the internet.

### Q5: What is 0.0.0.0/0 in a route table or security group?

> `0.0.0.0/0` represents **all IP addresses** — it's the entire internet. In a route table, it's the default route. In a security group, it means "allow traffic from/to anywhere." Use it carefully — it opens access to the world.

### Q6: What's the difference between IPv4 and IPv6?

> IPv4 uses 32-bit addresses (~4.3 billion total) and is running out. IPv6 uses 128-bit addresses (340 undecillion — practically infinite). IPv6 has built-in security (IPsec), no need for NAT, and simpler headers. IPv4 uses dotted decimal (192.168.1.1); IPv6 uses hexadecimal colons (2001:db8::1).

## 🧪 Hands-On Practice

```bash
# ─── View your IP addresses ───
ip addr show                    # All interfaces with IPs
hostname -I                     # Just the IPs
curl ifconfig.me                # Your public IP

# ─── Subnetting in practice ───
# Install ipcalc for subnet calculations
sudo apt install ipcalc         # Debian/Ubuntu
sudo yum install ipcalc         # RHEL/CentOS

# Calculate subnet info
ipcalc 192.168.1.0/24
ipcalc 10.0.0.0/16
ipcalc 192.168.1.0/26

# ─── Test connectivity ───
ping -c 4 8.8.8.8              # Ping Google DNS (IPv4)
ping6 -c 4 ::1                 # Ping loopback (IPv6)

# ─── View routing ───
ip route show                   # IPv4 routes
ip -6 route show               # IPv6 routes

# ─── Docker networking ───
docker network inspect bridge   # See Docker's default subnet
docker run --rm alpine ip addr  # See container's IP

# ─── Check for IP conflicts ───
arping -D 192.168.1.100        # Check if IP is already in use
```

## 📝 Summary

| Concept | Key Takeaway |
|---------|-------------|
| **IP Address** | Unique identifier for a device on a network (like a home address) |
| **IPv4** | 32-bit, dotted decimal (192.168.1.1), ~4.3 billion addresses, running out |
| **IPv6** | 128-bit, hex colons (2001:db8::1), practically unlimited addresses |
| **Private IPs** | 10.x.x.x, 172.16-31.x.x, 192.168.x.x — internal network only |
| **Public IPs** | Globally unique, routable on the internet |
| **Subnet Mask** | Defines network vs host portion of an IP |
| **CIDR** | Modern notation for IP ranges (e.g., /24 = 254 hosts) |
| **Subnetting** | Splitting a network into smaller, organized sub-networks |
| **NAT** | Translates private IPs to public IPs (sharing one public IP) |
| **DHCP** | Automatically assigns IP addresses to devices |
| **0.0.0.0/0** | All IP addresses (the entire internet) |
| **127.0.0.1** | Loopback / localhost |
