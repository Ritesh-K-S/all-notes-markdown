# 3.3 — DNS — How It Works, Records (A, AAAA, CNAME, MX, TXT, NS, SOA)

> **One Line Answer:** DNS (Domain Name System) is the **phonebook of the internet** — it translates human-friendly domain names like `google.com` into machine-friendly IP addresses like `142.250.80.46` so your computer knows where to send the request.

---

## 🤔 Why Should a DevOps Engineer Care About DNS?

DNS is one of the **most critical** pieces of infrastructure. When DNS breaks, **everything breaks**.

| Situation | DNS Knowledge Needed |
|-----------|---------------------|
| Website is down | Is the DNS record pointing to the right IP? |
| Email not working | Are the MX records correct? |
| SSL certificate won't issue | Is the TXT validation record set? |
| Migrating to a new server | Need to update DNS records + handle TTL |
| Setting up a load balancer | Create DNS record pointing to the LB |
| Kubernetes ingress not working | Is the domain resolving to the ingress controller? |
| Setting up a CDN (CloudFront, Cloudflare) | Need CNAME or alias records |
| CI/CD webhook not firing | Is the DNS resolving correctly from the CI server? |
| Multi-region failover | DNS-based routing (Route 53, Cloudflare) |

There's a famous saying in the industry:

> **"It's always DNS."** — Every DevOps engineer, at least once a week.

Seriously. When something is broken and you can't figure out why — check DNS first. It's almost always DNS. 😄

## 📖 What is DNS?

### The Problem

Computers communicate using **IP addresses** (like `142.250.80.46`). But humans are terrible at remembering numbers. Imagine if you had to type `142.250.80.46` every time you wanted to visit Google!

### The Solution

DNS is a **distributed naming system** that maps human-readable domain names to IP addresses.

```
What YOU type:        google.com
What your COMPUTER needs:  142.250.80.46

DNS is the translator between these two.

It's like a phonebook:
  ┌─────────────────────────────────────────┐
  │           DNS "Phonebook"               │
  ├─────────────────┬──────────────────────┤
  │  Name           │  IP Address          │
  ├─────────────────┼──────────────────────┤
  │  google.com     │  142.250.80.46       │
  │  github.com     │  140.82.121.3        │
  │  netflix.com    │  54.74.73.31         │
  │  facebook.com   │  157.240.1.35        │
  └─────────────────┴──────────────────────┘
```

But DNS is much more than just mapping names to IPs. It also handles:
- **Email routing** (MX records)
- **Domain verification** (TXT records)
- **Aliases** (CNAME records)
- **Load balancing** (multiple IPs per domain)
- **Failover** (health-checked routing)
- **CDN routing** (geographic routing)

## 🏗️ How DNS Works — The Complete Journey

When you type `www.google.com` in your browser, here's **exactly** what happens behind the scenes:

### Step-by-Step DNS Resolution

```
You type: www.google.com

Step 1: Browser Cache
  → Browser checks: "Have I looked this up recently?"
  → If YES → Use the cached IP. Done! ✅
  → If NO → Move to Step 2.

Step 2: OS Cache
  → OS checks its local DNS cache
  → Also checks /etc/hosts file (Linux) or C:\Windows\System32\drivers\etc\hosts (Windows)
  → If found → Done! ✅
  → If not → Move to Step 3.

Step 3: Resolver (Recursive DNS Server)
  → Your computer asks the DNS resolver (usually your ISP's DNS or 8.8.8.8)
  → "Hey, what's the IP for www.google.com?"
  → The resolver checks its own cache
  → If found → Returns the IP. Done! ✅
  → If not → The resolver starts the lookup journey...

Step 4: Root Name Server
  → Resolver asks a Root Name Server: "Where can I find .com domains?"
  → Root server says: "Go ask the .com TLD server at 192.5.6.30"
  → (There are 13 root server clusters worldwide: a.root-servers.net through m.root-servers.net)

Step 5: TLD (Top-Level Domain) Name Server
  → Resolver asks the .com TLD server: "Where can I find google.com?"
  → TLD server says: "Go ask Google's authoritative name server at ns1.google.com"

Step 6: Authoritative Name Server
  → Resolver asks Google's authoritative server: "What's the IP for www.google.com?"
  → Authoritative server says: "It's 142.250.80.46! Here you go. TTL: 300 seconds."

Step 7: Response Returns
  → Resolver caches the answer (for 300 seconds / 5 minutes)
  → Returns the IP to your computer
  → Your browser connects to 142.250.80.46
  → Google's homepage loads! 🎉
```

## 🌳 The DNS Hierarchy — Domain Name Structure

Domain names have a **hierarchical structure** — they read from **right to left**, from most general to most specific:

```
            www . google . com .
             │      │      │   │
             │      │      │   └── ROOT (the invisible dot — always there)
             │      │      └────── TLD (Top-Level Domain)
             │      └───────────── SLD (Second-Level Domain)
             └──────────────────── Subdomain

Full hierarchy:

                    . (ROOT)
                    │
         ┌──────────┼──────────┐
         │          │          │
       .com       .org       .io      ... (TLDs)
         │
    ┌────┼────┐
    │         │
  google    github               ... (SLDs)
    │
  ┌─┼──┐
  │    │
 www  mail  docs  drive          ... (Subdomains)
```

### Parts of a Domain Name

| Part | Example | Description |
|------|---------|-------------|
| **Root** | `.` (invisible) | The starting point of all DNS lookups |
| **TLD** (Top-Level Domain) | `.com`, `.org`, `.io`, `.dev` | Highest level in the hierarchy |
| **SLD** (Second-Level Domain) | `google`, `github`, `netflix` | The name you buy/register |
| **Subdomain** | `www`, `mail`, `api`, `docs` | You create these yourself (free, unlimited) |
| **FQDN** (Fully Qualified Domain Name) | `www.google.com.` | Complete path including root dot |

### Types of TLDs

| Type | Examples | Description |
|------|---------|-------------|
| **gTLD** (Generic) | `.com`, `.org`, `.net`, `.info` | Open to anyone |
| **ccTLD** (Country Code) | `.uk`, `.de`, `.jp`, `.in`, `.io` | Country-specific |
| **New gTLDs** | `.dev`, `.app`, `.cloud`, `.ninja` | Newer, specialized |
| **Infrastructure** | `.arpa` | Used for reverse DNS |

## 📋 DNS Record Types — The Complete Guide

DNS records are **instructions** stored on authoritative DNS servers. Each record type serves a different purpose. Let's cover every important one:

### A Record (Address Record) ⭐

Maps a domain name to an **IPv4 address**. This is the most common DNS record.

```
Type:    A
Name:    google.com
Value:   142.250.80.46
TTL:     300

What it means:
  "When someone asks for google.com, give them IP 142.250.80.46"
```

**You can have multiple A records for the same domain** (DNS round-robin load balancing):

```
google.com    A    142.250.80.46
google.com    A    142.250.80.47
google.com    A    142.250.80.48

→ DNS will rotate between these IPs, distributing traffic
```

```bash
# Look up A records
dig google.com A
# or
nslookup google.com

# Output:
# google.com.    300    IN    A    142.250.80.46
```

---

### AAAA Record (IPv6 Address Record) 🌐

Same as an A record, but for **IPv6 addresses**.

```
Type:    AAAA
Name:    google.com
Value:   2607:f8b0:4004:800::200e
TTL:     300

What it means:
  "When someone asks for google.com via IPv6, give them this IPv6 address"
```

Why "AAAA"? Because IPv6 addresses are **4 times longer** than IPv4. A = IPv4 (32-bit), AAAA = IPv6 (128-bit = 4 × 32-bit).

```bash
# Look up AAAA records
dig google.com AAAA

# Output:
# google.com.    300    IN    AAAA    2607:f8b0:4004:800::200e
```

---

### CNAME Record (Canonical Name) 🔗

Creates an **alias** — points one domain name to another domain name (NOT to an IP).

```
Type:    CNAME
Name:    www.example.com
Value:   example.com
TTL:     3600

What it means:
  "www.example.com is just an alias for example.com.
   Go look up example.com's A record to get the actual IP."

The chain:
  www.example.com → CNAME → example.com → A → 93.184.216.34
```

**Common uses:**

```
# Point www to the root domain
www.example.com     CNAME    example.com

# Point to a CDN
cdn.example.com     CNAME    d1234.cloudfront.net

# Point to a load balancer
app.example.com     CNAME    my-alb-1234.us-east-1.elb.amazonaws.com

# Point to an external service
blog.example.com    CNAME    mycompany.ghost.io
status.example.com  CNAME    mycompany.statuspage.io
```

**Important CNAME Rules:**

```
✅ CNAME can point to another domain
❌ CNAME CANNOT coexist with other records for the same name
❌ CNAME CANNOT be used on the root/apex domain (example.com)
   → Use an ALIAS or ANAME record instead (provider-specific)
   → Or use an A record directly

Why can't CNAME be on the root domain?
  → Because root domains MUST have SOA and NS records
  → CNAME says "ignore all other records, just follow this alias"
  → This conflicts with required SOA/NS records
```

```bash
# Look up CNAME records
dig www.github.com CNAME

# Output:
# www.github.com.    3600    IN    CNAME    github.com.
```

### MX Record (Mail Exchange) 📧

Tells the world **which mail servers handle email** for your domain.

```
Type:    MX
Name:    example.com
Priority: 10
Value:   mail1.example.com
TTL:     3600

What it means:
  "When someone sends email to user@example.com,
   deliver it to mail1.example.com"
```

**MX records have a priority number** — lower number = higher priority:

```
example.com    MX    10    mail1.example.com    (Primary — tried first)
example.com    MX    20    mail2.example.com    (Secondary — backup)
example.com    MX    30    mail3.example.com    (Tertiary — last resort)

If mail1 is down → try mail2
If mail2 is down → try mail3
```

**Using Google Workspace / Gmail for your domain:**

```
example.com    MX    1     ASPMX.L.GOOGLE.COM
example.com    MX    5     ALT1.ASPMX.L.GOOGLE.COM
example.com    MX    5     ALT2.ASPMX.L.GOOGLE.COM
example.com    MX    10    ALT3.ASPMX.L.GOOGLE.COM
example.com    MX    10    ALT4.ASPMX.L.GOOGLE.COM
```

```bash
# Look up MX records
dig example.com MX

# Output:
# example.com.    3600    IN    MX    10 mail1.example.com.
# example.com.    3600    IN    MX    20 mail2.example.com.
```

### TXT Record (Text Record) 📝

Stores **arbitrary text data** in DNS. Used heavily for **verification and security**.

```
Type:    TXT
Name:    example.com
Value:   "v=spf1 include:_spf.google.com ~all"
TTL:     3600
```

**Common uses of TXT records:**

#### 1. SPF (Sender Policy Framework) — Email Security

Tells the world which servers are **allowed to send email** on behalf of your domain.

```
example.com    TXT    "v=spf1 include:_spf.google.com include:sendgrid.net ~all"

Translation:
  → "Only Google's and SendGrid's servers can send emails from @example.com"
  → ~all = soft fail (mark as spam if it's from anyone else)
  → -all = hard fail (reject if it's from anyone else)
```

#### 2. DKIM (DomainKeys Identified Mail) — Email Authentication

Adds a digital signature to verify emails weren't tampered with.

```
google._domainkey.example.com    TXT    "v=DKIM1; k=rsa; p=MIIBIjANBgkq..."
```

#### 3. DMARC (Domain-based Message Authentication) — Email Policy

Tells receiving servers what to do with emails that fail SPF/DKIM checks.

```
_dmarc.example.com    TXT    "v=DMARC1; p=quarantine; rua=mailto:dmarc@example.com"
```

#### 4. Domain Verification

When you add a domain to services like Google, AWS, or Let's Encrypt, they ask you to add a TXT record to prove you own the domain.

```
example.com    TXT    "google-site-verification=abc123xyz789"
example.com    TXT    "aws:domainVerification=def456uvw012"

# Let's Encrypt (ACME DNS challenge)
_acme-challenge.example.com    TXT    "ghi789rst345"
```

```bash
# Look up TXT records
dig example.com TXT

# Look up SPF specifically
dig example.com TXT | grep spf

# Look up DMARC
dig _dmarc.example.com TXT
```

### NS Record (Name Server) 🏢

Tells the world **which DNS servers are authoritative** for your domain — i.e., which servers have the "official" DNS records.

```
Type:    NS
Name:    example.com
Value:   ns1.example.com
TTL:     86400

Typical setup:
example.com    NS    ns1.example.com    (Primary nameserver)
example.com    NS    ns2.example.com    (Secondary nameserver)
```

**Using different DNS providers:**

```
# If you use Cloudflare:
example.com    NS    ada.ns.cloudflare.com
example.com    NS    bob.ns.cloudflare.com

# If you use AWS Route 53:
example.com    NS    ns-123.awsdns-45.com
example.com    NS    ns-678.awsdns-90.co.uk
example.com    NS    ns-111.awsdns-22.net
example.com    NS    ns-333.awsdns-44.org

# If you use Google Cloud DNS:
example.com    NS    ns-cloud-a1.googledomains.com
example.com    NS    ns-cloud-a2.googledomains.com
```

**When you buy a domain**, the first thing you do is set the NS records to point to your DNS provider (Cloudflare, Route 53, etc.). Then you manage all other records (A, CNAME, MX, etc.) on that provider.

```bash
# Look up NS records
dig example.com NS

# Output:
# example.com.    86400    IN    NS    ns1.example.com.
# example.com.    86400    IN    NS    ns2.example.com.
```

### SOA Record (Start of Authority) 📜

The **most important record** in any DNS zone. Contains metadata about the zone — who manages it, how often it should be refreshed, etc.

```
Type:    SOA
Name:    example.com

Fields:
  MNAME:    ns1.example.com          (Primary nameserver)
  RNAME:    admin.example.com        (Admin email — the @ is replaced with .)
  Serial:   2024010101               (Version number — increment on every change)
  Refresh:  3600                     (How often secondary NS checks for updates)
  Retry:    600                      (How soon to retry if refresh fails)
  Expire:   604800                   (When secondary NS stops serving if it can't refresh)
  Min TTL:  300                      (Default TTL for negative caching)
```

**The Serial number is important!** Every time you change any DNS record, the serial number should increase. Secondary nameservers compare serial numbers to know if they need to update.

```bash
# Look up SOA record
dig example.com SOA

# Output:
# example.com.  86400  IN  SOA  ns1.example.com. admin.example.com. (
#                            2024010101  ; serial
#                            3600        ; refresh
#                            600         ; retry
#                            604800      ; expire
#                            300         ; minimum TTL
#                            )
```

### PTR Record (Pointer Record) — Reverse DNS 🔄

Does the **opposite** of an A record — maps an **IP address back to a domain name**.

```
Type:    PTR
Name:    46.80.250.142.in-addr.arpa    (IP address reversed + .in-addr.arpa)
Value:   google.com
TTL:     3600

What it means:
  "If someone asks 'what domain owns 142.250.80.46?',
   the answer is google.com"
```

**Why is reverse DNS important?**
- Email servers check PTR records to verify sender legitimacy (anti-spam)
- Log analysis — convert IPs to hostnames for readability
- Security auditing

```bash
# Reverse DNS lookup
dig -x 8.8.8.8

# Output:
# 8.8.8.8.in-addr.arpa.    3600    IN    PTR    dns.google.

# Using nslookup
nslookup 8.8.8.8

# Using host command
host 8.8.8.8
```

---

### SRV Record (Service Record) 🔧

Specifies the **host and port** for a specific service. Used by protocols that need to discover services.

```
Type:    SRV
Name:    _sip._tcp.example.com
Priority: 10
Weight:   60
Port:     5060
Target:   sip1.example.com

Format: _service._protocol.domain    SRV    priority weight port target
```

**Used by:**
- Microsoft Active Directory
- SIP (VoIP)
- XMPP (chat)
- Minecraft servers (yes, really!)
- Kubernetes external DNS

```bash
# Look up SRV records
dig _sip._tcp.example.com SRV
```

---

### CAA Record (Certificate Authority Authorization) 🔒

Specifies which **Certificate Authorities** are allowed to issue SSL/TLS certificates for your domain.

```
Type:    CAA
Name:    example.com
Flag:    0
Tag:     issue
Value:   "letsencrypt.org"

example.com    CAA    0 issue "letsencrypt.org"
example.com    CAA    0 issue "amazon.com"
example.com    CAA    0 iodef "mailto:security@example.com"

Translation:
  → Only Let's Encrypt and Amazon can issue certificates for our domain
  → Email security@example.com if someone tries to issue an unauthorized cert
```

```bash
# Look up CAA records
dig example.com CAA
```

## ⏱️ TTL (Time to Live) — DNS Caching

**TTL** is how long (in seconds) a DNS record is **cached** before it must be looked up again.

```
TTL Values:
  300     =  5 minutes    (good for frequently changing records)
  3600    =  1 hour       (standard for most records)
  86400   =  24 hours     (for stable records that rarely change)
  604800  =  7 days       (for very stable records like NS)

Example:
  example.com    A    93.184.216.34    TTL: 3600

  → After looking up this record, caches will store the answer for 1 hour
  → After 1 hour, they'll ask again
```

### TTL Strategy for Migrations

This is a **real-world DevOps trick**:

```
BEFORE migration (1-2 days ahead):
  → Lower TTL from 3600 to 60 seconds
  → Wait 24-48 hours for old TTL to expire everywhere

DURING migration:
  → Change the A record to point to the new server IP
  → Because TTL is 60 seconds, the change propagates in ~1 minute!

AFTER migration (once stable):
  → Raise TTL back to 3600 seconds
  → Reduces DNS query load

Timeline:
  Day 1:  TTL = 3600 → Change to TTL = 60
  Day 3:  Change IP from 1.2.3.4 → 5.6.7.8 (propagates fast!)
  Day 4:  Confirm everything works → Change TTL back to 3600
```

This is **extremely common** in production migrations!

## 🔧 DNS Tools and Commands — Hands-On

### `dig` — The Swiss Army Knife of DNS

```bash
# Basic lookup (A record by default)
dig google.com

# Look up specific record types
dig google.com A          # IPv4 address
dig google.com AAAA       # IPv6 address
dig google.com MX         # Mail servers
dig google.com NS         # Name servers
dig google.com TXT        # Text records
dig google.com SOA        # Start of Authority
dig google.com ANY        # All records (not always supported)

# Short answer only
dig google.com +short
# Output: 142.250.80.46

# Trace the full DNS resolution path (like following the journey)
dig google.com +trace

# Query a specific DNS server
dig @8.8.8.8 google.com          # Ask Google's DNS
dig @1.1.1.1 google.com          # Ask Cloudflare's DNS
dig @ns1.google.com google.com   # Ask Google's authoritative server

# Reverse DNS lookup
dig -x 8.8.8.8

# Check TTL
dig google.com | grep -A1 "ANSWER SECTION"
```


## 🏗️ DNS in the Real World — DevOps Scenarios

### Scenario 1: Setting Up a New Website

```
1. Buy domain "myapp.com" from a registrar (Namecheap, GoDaddy, Route 53)

2. Set NS records to your DNS provider:
   myapp.com    NS    ns1.cloudflare.com
   myapp.com    NS    ns2.cloudflare.com

3. Create DNS records:
   myapp.com       A      93.184.216.34        (server IP)
   www.myapp.com   CNAME  myapp.com            (www → root)
   api.myapp.com   A      93.184.216.35        (API server)
   
4. Set up email (Google Workspace):
   myapp.com       MX     1  ASPMX.L.GOOGLE.COM
   myapp.com       TXT    "v=spf1 include:_spf.google.com ~all"

5. Set up SSL (Let's Encrypt):
   _acme-challenge.myapp.com  TXT  "verification-string-here"
```

### Scenario 2: DNS with AWS Route 53

```bash
# Create a hosted zone
aws route53 create-hosted-zone --name myapp.com --caller-reference "unique-string-123"

# Create an A record
aws route53 change-resource-record-sets --hosted-zone-id Z123456 --change-batch '{
  "Changes": [{
    "Action": "CREATE",
    "ResourceRecordSet": {
      "Name": "myapp.com",
      "Type": "A",
```

## 🛡️ DNS Security

### DNS-Related Attacks

| Attack | Description | Protection |
|--------|------------|-----------|
| **DNS Spoofing/Cache Poisoning** | Attacker injects fake DNS records into a resolver's cache | DNSSEC |
| **DNS Hijacking** | Attacker changes your DNS settings (at registrar or router) | Two-factor auth on registrar, lock domain |
| **DNS Amplification DDoS** | Attacker uses DNS servers to amplify traffic towards a victim | Rate limiting, response rate limiting |
| **DNS Tunneling** | Attacker hides malicious data inside DNS queries | DNS monitoring, firewall rules |
| **Typosquatting** | Attacker registers similar domain names (gooogle.com) | Monitor for similar domains, DMARC |

### DNSSEC (DNS Security Extensions)

DNSSEC adds **digital signatures** to DNS records to prevent tampering:

```
Without DNSSEC:
  Your computer asks for google.com
  → Attacker intercepts and sends fake IP
  → You connect to attacker's server 😱

With DNSSEC:
  Your computer asks for google.com
  → Attacker intercepts and sends fake IP
  → But the response is NOT digitally signed (or signed with wrong key)
  → Your computer rejects the fake response ✅
  → Tries again and gets the real, signed response
```

```bash
# Check if a domain uses DNSSEC
dig google.com +dnssec

# Check DNSKEY records
dig google.com DNSKEY
```

## 🔍 The `/etc/resolv.conf` File — DNS Client Configuration

This file tells your Linux system **which DNS servers to use**.

```bash
cat /etc/resolv.conf

# Typical content:
nameserver 8.8.8.8          # Primary DNS (Google)
nameserver 8.8.4.4          # Secondary DNS (Google)
nameserver 1.1.1.1          # Tertiary DNS (Cloudflare)
search example.com          # Default domain suffix

# The "search" directive means:
# If you type "myserver", Linux will try "myserver.example.com" first
```

**In Docker containers:**

```bash
# Docker copies the host's DNS settings into the container
docker run --rm alpine cat /etc/resolv.conf

# Override DNS in a container
docker run --dns 8.8.8.8 --rm alpine cat /etc/resolv.conf

# In Docker Compose:
services:
  app:
    dns:
      - 8.8.8.8
      - 1.1.1.1
```

## 🎯 Interview Questions — DNS

### Q1: What is DNS and how does it work?

> DNS (Domain Name System) translates domain names to IP addresses. When you type a URL, your browser checks its cache, then the OS cache, then asks a recursive DNS resolver. The resolver queries root servers → TLD servers → authoritative servers to find the IP. The result is cached based on the TTL value.

### Q2: What's the difference between A and CNAME records?

> An **A record** maps a domain directly to an **IP address** (google.com → 142.250.80.46). A **CNAME record** maps a domain to **another domain name** (www.google.com → google.com). CNAME is an alias; it can't exist on the root/apex domain and can't coexist with other records of the same name.

### Q3: What DNS records are needed for email?

> **MX records** specify which mail servers handle email for the domain (with priority). **SPF** (TXT record) defines which servers can send email on behalf of the domain. **DKIM** (TXT record) adds a digital signature for authentication. **DMARC** (TXT record) defines the policy for handling failed authentication.

### Q4: What is TTL in DNS? How would you handle a server migration?

> TTL (Time to Live) is how long a DNS record is cached (in seconds). For a migration: lower the TTL to 60 seconds 24-48 hours before the migration. Change the IP when ready. The change propagates in ~1 minute instead of hours. Raise the TTL back afterward.

### Q5: How does Kubernetes handle DNS?

> Kubernetes runs **CoreDNS** as a cluster add-on. Every Service gets a DNS name: `<service>.<namespace>.svc.cluster.local`. Pods can reach services by name instead of IP. DNS is configured automatically — pods use CoreDNS as their resolver via `/etc/resolv.conf`.

### Q6: What is "It's always DNS"? Give an example.

> It's a running joke (and truth) in DevOps. Example: your application can't connect to `api.internal.com` in production. You check the code, check the firewall, check the network — everything looks fine. Then you discover the DNS record was never created, or the TTL cached an old IP after a migration. DNS is often the last thing you check but should be the first!

## 🧪 Hands-On Practice

```bash
# ─── Basic DNS Lookups ───
dig google.com                     # Default A record lookup
dig google.com +short              # Just the IP
dig google.com AAAA +short         # IPv6 address
dig google.com MX +short           # Mail servers
dig google.com NS +short           # Name servers
dig google.com TXT                 # Text records
dig google.com SOA                 # Start of Authority

# ─── Trace DNS Resolution Path ───
dig google.com +trace              # Follow the full resolution chain

# ─── Query Specific DNS Servers ───
dig @8.8.8.8 google.com           # Ask Google's DNS
dig @1.1.1.1 google.com           # Ask Cloudflare's DNS

# ─── Reverse DNS ───
dig -x 8.8.8.8                    # What domain owns this IP?
host 1.1.1.1                      # Same but simpler output

# ─── Check Propagation ───
# Query multiple DNS servers to see if a change has propagated
dig @8.8.8.8 myapp.com +short     # Google
dig @1.1.1.1 myapp.com +short     # Cloudflare
dig @208.67.222.222 myapp.com +short  # OpenDNS

# ─── DNS Security ───
dig google.com +dnssec             # Check DNSSEC
dig _dmarc.google.com TXT          # Check DMARC policy

# ─── Local DNS ───
cat /etc/hosts                     # View local DNS overrides
cat /etc/resolv.conf               # View DNS resolver config
systemd-resolve --status           # DNS resolver status (systemd systems)

# ─── Flush DNS Cache ───
# On Linux (systemd-resolved):
sudo systemd-resolve --flush-caches

# On macOS:
# sudo dscacheutil -flushcache; sudo killall -HUP mDNSResponder

# On Windows:
# ipconfig /flushdns
```

## 📝 Summary

| Concept | Key Takeaway |
|---------|-------------|
| **DNS** | Translates domain names to IP addresses (phonebook of the internet) |
| **DNS Resolution** | Browser cache → OS cache → Resolver → Root → TLD → Authoritative |
| **A Record** | Maps domain → IPv4 address (most common) |
| **AAAA Record** | Maps domain → IPv6 address |
| **CNAME Record** | Alias: domain → another domain (not for apex domains) |
| **MX Record** | Which servers handle email (with priority) |
| **TXT Record** | Verification, SPF, DKIM, DMARC (text data) |
| **NS Record** | Which DNS servers are authoritative for the domain |
| **SOA Record** | Zone metadata (admin, serial, refresh, TTL) |
| **PTR Record** | Reverse DNS: IP → domain name |
| **TTL** | How long records are cached (lower before migrations!) |
| **DNSSEC** | Digital signatures to prevent DNS spoofing |
| **"It's always DNS"** | When in doubt, check DNS first 😄 |
