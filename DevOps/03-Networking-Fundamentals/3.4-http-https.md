# 3.4 — HTTP/HTTPS — Methods, Status Codes, Headers, TLS/SSL Handshake

> **One Line Answer:** HTTP is the **language that your browser and web servers speak** to exchange data — and HTTPS is the **secure, encrypted version** that makes sure nobody can eavesdrop on the conversation.

---

## 🤔 Why Should a DevOps Engineer Care About HTTP/HTTPS?

Every single thing you do in DevOps involves HTTP/HTTPS in some way:

| What You Do | How HTTP/HTTPS is Involved |
|------------|---------------------------|
| Deploy an application | The app serves content over HTTP/HTTPS |
| Set up a load balancer | It routes HTTP/HTTPS requests to backend servers |
| Configure a reverse proxy (Nginx, HAProxy) | It handles HTTP headers, methods, and TLS termination |
| Monitor applications | Health checks use HTTP endpoints (`/health`, `/status`) |
| Work with APIs | REST APIs communicate over HTTP |
| Set up CI/CD pipelines | Webhooks trigger builds via HTTP POST requests |
| Manage Kubernetes Ingress | Ingress controllers route HTTP/HTTPS traffic |
| Debug production issues | You'll read HTTP status codes and headers all day long |

If you don't understand HTTP deeply, you'll struggle with **debugging, configuration, and architecture** — all core DevOps skills.

## 🌐 What is HTTP?

### The Simple Explanation

**HTTP** stands for **HyperText Transfer Protocol**.

It's a set of rules that defines **how your browser (client) asks for data** and **how the server sends it back**. Every time you open a website, your browser sends an HTTP request, and the server responds with an HTTP response.

Think of it like ordering food at a restaurant:

```
You (Client/Browser)              Waiter (HTTP)              Kitchen (Server)
       │                              │                            │
       │── "I want a pizza" ──────────>│                            │
       │   (HTTP Request)              │── delivers order ──────────>│
       │                               │                            │
       │                               │<── pizza is ready ─────────│
       │<── here's your pizza ─────────│   (HTTP Response)          │
       │                               │                            │
```

That's literally it. HTTP is just a **request-response protocol**.

### The Technical Definition

HTTP is an **application-layer protocol** (Layer 7 of the OSI model) that operates on top of TCP. It is:

- **Stateless** — Each request is independent; the server doesn't remember previous requests
- **Text-based** — Requests and responses are human-readable text (mostly)
- **Client-Server model** — The client initiates, the server responds
- **Connectionless (in HTTP/1.0)** — Connection closes after each request (improved in HTTP/1.1 with keep-alive)

## 📜 A Brief History of HTTP

```
1989 — Tim Berners-Lee invents the World Wide Web at CERN
    └── He also creates HTTP as the protocol to transfer web pages

1991 — HTTP/0.9
    └── Super simple — only GET method, no headers, only HTML responses
    └── Example: GET /index.html

1996 — HTTP/1.0 (RFC 1945)
    └── Added headers, status codes, POST method
    └── Problem: new TCP connection for EVERY single request (slow!)

1997 — HTTP/1.1 (RFC 2068, updated in RFC 2616)
    └── Keep-alive connections (reuse the same TCP connection)
    └── Chunked transfer encoding
    └── Host header (multiple websites on one IP)
    └── This is STILL the most widely used version

2015 — HTTP/2 (RFC 7540)
    └── Binary protocol (not text-based anymore)
    └── Multiplexing (multiple requests over a single connection)
    └── Header compression (HPACK)
    └── Server push
    └── Much faster than HTTP/1.1

2022 — HTTP/3 (RFC 9114)
    └── Built on QUIC (uses UDP instead of TCP!)
    └── Even faster — no head-of-line blocking
    └── Built-in encryption (TLS 1.3)
    └── Growing adoption (Google, Facebook, Cloudflare already use it)
```

## 📨 Anatomy of an HTTP Request

When your browser (or `curl` or any client) talks to a server, it sends an **HTTP Request**. Let's break it apart piece by piece.

### Structure

```
┌────────────────────────────────────────────────────┐
│  Request Line    →  METHOD  PATH  HTTP/VERSION     │
├────────────────────────────────────────────────────┤
│  Headers         →  Key: Value                     │
│                     Key: Value                     │
│                     Key: Value                     │
├────────────────────────────────────────────────────┤
│  (Empty Line)    →  separates headers from body    │
├────────────────────────────────────────────────────┤
│  Body (optional) →  data sent with the request     │
└────────────────────────────────────────────────────┘
```

### Real Example

```http
POST /api/users HTTP/1.1
Host: www.example.com
Content-Type: application/json
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
Accept: application/json
Content-Length: 52

{"name": "Ritesh", "role": "DevOps Engineer"}
```

Let's break this down:

| Part | Value | What It Means |
|------|-------|---------------|
| **Method** | `POST` | I want to CREATE something |
| **Path** | `/api/users` | I want to access the users API |
| **HTTP Version** | `HTTP/1.1` | I'm speaking HTTP version 1.1 |
| **Host header** | `www.example.com` | The server I'm talking to |
| **Content-Type** | `application/json` | I'm sending JSON data |
| **Authorization** | `Bearer eyJ...` | Here's my auth token |
| **Accept** | `application/json` | I want JSON back |
| **Content-Length** | `52` | My body is 52 bytes |
| **Body** | `{"name": "Ritesh"...}` | The actual data I'm sending |

## 📬 Anatomy of an HTTP Response

When the server processes your request, it sends back an **HTTP Response**.

### Structure

```
┌────────────────────────────────────────────────────┐
│  Status Line     →  HTTP/VERSION  STATUS_CODE  MSG │
├────────────────────────────────────────────────────┤
│  Headers         →  Key: Value                     │
│                     Key: Value                     │
├────────────────────────────────────────────────────┤
│  (Empty Line)                                      │
├────────────────────────────────────────────────────┤
│  Body            →  the actual content             │
└────────────────────────────────────────────────────┘
```

### Real Example

```http
HTTP/1.1 201 Created
Content-Type: application/json
Date: Tue, 27 May 2025 10:30:00 GMT
Server: nginx/1.24.0
Content-Length: 78

{"id": 42, "name": "Ritesh", "role": "DevOps Engineer", "created": true}
```

## 🔧 HTTP Methods (Verbs) — The Complete Guide

HTTP methods tell the server **what action you want to perform**. Think of them as verbs in a sentence.

### The Main Methods

| Method | What It Does | Analogy | Has Body? | Safe? | Idempotent? |
|--------|-------------|---------|-----------|-------|-------------|
| **GET** | Retrieve/read data | "Show me the menu" | No | ✅ Yes | ✅ Yes |
| **POST** | Create new data | "I want to place a new order" | Yes | ❌ No | ❌ No |
| **PUT** | Replace/update entire resource | "Replace my entire order with this new one" | Yes | ❌ No | ✅ Yes |
| **PATCH** | Partially update a resource | "Just change the drink in my order" | Yes | ❌ No | ❌ No |
| **DELETE** | Delete a resource | "Cancel my order" | Optional | ❌ No | ✅ Yes |
| **HEAD** | Same as GET but only returns headers (no body) | "Is the restaurant open?" | No | ✅ Yes | ✅ Yes |
| **OPTIONS** | Ask what methods are allowed | "What can I order here?" | No | ✅ Yes | ✅ Yes |

### Wait — What Do "Safe" and "Idempotent" Mean?

**Safe** = The request **doesn't change anything** on the server. Calling it 100 times has no side effects. GET is safe — it just reads data.

**Idempotent** = Calling the request **once or 100 times** produces the **same result**. PUT is idempotent — if you send the same update 10 times, the resource still ends up the same. POST is NOT idempotent — sending the same POST 10 times might create 10 duplicate records.

### Practical Examples with `curl`

#### GET — Fetch data

```bash
# Get a list of users
curl -X GET https://api.example.com/users

# Short form (GET is the default)
curl https://api.example.com/users
```

#### POST — Create data

```bash
# Create a new user
curl -X POST https://api.example.com/users \
  -H "Content-Type: application/json" \
  -d '{"name": "Ritesh", "email": "ritesh@example.com"}'
```

**Explanation:**
- `-X POST` — Use the POST method
- `-H "Content-Type: application/json"` — Set a header telling the server we're sending JSON
- `-d '...'` — The data (body) to send

#### PUT — Replace data

```bash
# Replace user 42 entirely
curl -X PUT https://api.example.com/users/42 \
  -H "Content-Type: application/json" \
  -d '{"name": "Ritesh Singh", "email": "ritesh@newmail.com", "role": "SRE"}'
```

#### PATCH — Partially update data

```bash
# Only update the role of user 42
curl -X PATCH https://api.example.com/users/42 \
  -H "Content-Type: application/json" \
  -d '{"role": "Platform Engineer"}'
```

#### DELETE — Remove data

```bash
# Delete user 42
curl -X DELETE https://api.example.com/users/42
```

#### HEAD — Check without downloading

```bash
# Check if a file exists and get its size (without downloading it)
curl -I https://example.com/large-file.zip
```

**Explanation:** `-I` is the shortcut for HEAD. It returns only the headers — useful for checking Content-Length, Content-Type, or if a URL exists.

#### OPTIONS — Check what's allowed

```bash
# What methods does this endpoint support?
curl -X OPTIONS https://api.example.com/users -i
```

This is often used in **CORS (Cross-Origin Resource Sharing)** — the browser sends an OPTIONS request before making a cross-origin request to check if it's allowed.

## 📊 HTTP Status Codes — The Complete Reference

Status codes are **3-digit numbers** that the server sends back to tell you what happened. They're grouped into 5 categories:

```
1xx — Informational    → "Hold on, I'm working on it"
2xx — Success          → "Here you go, everything worked!"
3xx — Redirection      → "What you want is somewhere else"
4xx — Client Error     → "YOU messed up"
5xx — Server Error     → "I (the server) messed up"
```

### 1xx — Informational (Rare, but good to know)

| Code | Name | What It Means |
|------|------|---------------|
| 100 | Continue | "I got your headers, go ahead and send the body" |
| 101 | Switching Protocols | "OK, let's switch to WebSocket/HTTP2 as you asked" |
| 102 | Processing | "I'm working on it, give me a moment" (WebDAV) |

### 2xx — Success ✅

| Code | Name | What It Means | When You See It |
|------|------|---------------|-----------------|
| **200** | OK | "Here's what you asked for" | GET request succeeded |
| **201** | Created | "I made the new thing you requested" | POST request created a resource |
| **202** | Accepted | "I got your request, I'll process it later" | Async operations (queued jobs) |
| **204** | No Content | "Done! But I have nothing to send back" | DELETE succeeded, no body |

### 3xx — Redirection 🔀

| Code | Name | What It Means | When You See It |
|------|------|---------------|-----------------|
| **301** | Moved Permanently | "This resource has moved FOREVER to a new URL" | Old URL redirects to new one (SEO-friendly) |
| **302** | Found (Temporary Redirect) | "This resource is temporarily at another URL" | Login redirects, temporary maintenance |
| **304** | Not Modified | "Nothing changed since last time, use your cache" | Browser caching |
| **307** | Temporary Redirect | Like 302, but MUST keep the same HTTP method | POST redirect that stays POST |
| **308** | Permanent Redirect | Like 301, but MUST keep the same HTTP method | Permanent redirect that keeps POST as POST |

### 4xx — Client Error ❌ (YOUR fault)

| Code | Name | What It Means | Common Cause |
|------|------|---------------|-------------|
| **400** | Bad Request | "I can't understand what you sent me" | Malformed JSON, invalid parameters |
| **401** | Unauthorized | "Who are you? Show me your ID" | Missing or invalid authentication |
| **403** | Forbidden | "I know who you are, but you're NOT allowed" | Valid auth but insufficient permissions |
| **404** | Not Found | "That thing doesn't exist" | Wrong URL, deleted resource |
| **405** | Method Not Allowed | "You can't use that method here" | POST to a GET-only endpoint |
| **408** | Request Timeout | "You took too long to send your request" | Slow client |
| **409** | Conflict | "Your request conflicts with current state" | Duplicate entry, version conflict |
| **413** | Payload Too Large | "Your request body is too big" | File upload exceeds limit |
| **415** | Unsupported Media Type | "I don't understand that content type" | Sending XML when server expects JSON |
| **422** | Unprocessable Entity | "I understand your request but can't process it" | Validation errors |
| **429** | Too Many Requests | "Slow down! You're hitting me too fast" | Rate limiting |

### 5xx — Server Error 💥 (SERVER's fault)

| Code | Name | What It Means | Common Cause |
|------|------|---------------|-------------|
| **500** | Internal Server Error | "Something broke on my side, I don't know what" | Unhandled exception, bug in code |
| **502** | Bad Gateway | "I'm a proxy/gateway and the backend gave me garbage" | Upstream server crashed |
| **503** | Service Unavailable | "I'm overloaded or under maintenance" | Server overwhelmed, deploy in progress |
| **504** | Gateway Timeout | "I'm a proxy and the backend didn't respond in time" | Backend is too slow or hung |

### DevOps Pro Tip — Status Codes You'll See Every Day

As a DevOps engineer, these are the ones you'll deal with constantly:

```
200 — Everything is fine, your health check passed
301 — You set up a redirect correctly
403 — IAM/permissions misconfigured
404 — Your Nginx location block or API route is wrong
500 — The application crashed — check the app logs
502 — Your backend (upstream) server is down — check if it's running
503 — Your service is overwhelmed — scale up or check resources
504 — Your backend is too slow — check for database deadlocks, CPU issues
```

## 📋 HTTP Headers — The Complete Guide

Headers are **key-value pairs** that carry extra information about the request or response. Think of them as **metadata** — they don't change the content, but they tell the other side important things.

### Request Headers (Client → Server)

| Header | Example | What It Does |
|--------|---------|-------------|
| `Host` | `Host: www.example.com` | Which website you want (required in HTTP/1.1) |
| `User-Agent` | `User-Agent: Mozilla/5.0...` | What browser/tool is making the request |
| `Accept` | `Accept: application/json` | What content types you want back |
| `Accept-Language` | `Accept-Language: en-US` | What language you prefer |
| `Accept-Encoding` | `Accept-Encoding: gzip, br` | What compression you support |
| `Authorization` | `Authorization: Bearer abc123` | Your auth credentials/token |
| `Content-Type` | `Content-Type: application/json` | Format of the data you're SENDING |
| `Content-Length` | `Content-Length: 348` | Size of the body in bytes |
| `Cookie` | `Cookie: session_id=abc123` | Cookies sent to the server |
| `Cache-Control` | `Cache-Control: no-cache` | How caching should work |
| `If-None-Match` | `If-None-Match: "etag123"` | Conditional request (return data only if changed) |
| `X-Forwarded-For` | `X-Forwarded-For: 203.0.113.50` | Real client IP (when behind a proxy) |
| `X-Request-ID` | `X-Request-ID: uuid-1234` | Unique ID for request tracing |

### Response Headers (Server → Client)

| Header | Example | What It Does |
|--------|---------|-------------|
| `Content-Type` | `Content-Type: text/html; charset=utf-8` | Format of the data being sent back |
| `Content-Length` | `Content-Length: 1234` | Size of the response body |
| `Content-Encoding` | `Content-Encoding: gzip` | Compression used on the response |
| `Set-Cookie` | `Set-Cookie: session_id=abc; HttpOnly` | Server tells browser to store a cookie |
| `Cache-Control` | `Cache-Control: max-age=3600` | How long to cache this response |
| `ETag` | `ETag: "abc123"` | Version identifier for the resource |
| `Location` | `Location: /new-url` | Where to redirect (used with 3xx codes) |
| `Server` | `Server: nginx/1.24` | What server software is running |
| `X-Content-Type-Options` | `X-Content-Type-Options: nosniff` | Security: prevent MIME type sniffing |
| `Strict-Transport-Security` | `HSTS: max-age=31536000` | Security: force HTTPS for this domain |
| `Access-Control-Allow-Origin` | `ACAO: https://app.example.com` | CORS: which origins can access this API |

### Seeing Headers in Action

```bash
# See all request and response headers with curl
curl -v https://www.google.com 2>&1 | head -30

# Only see response headers
curl -I https://www.google.com

# Send custom headers
curl -H "X-Custom-Header: my-value" \
     -H "Authorization: Bearer my-token" \
     https://api.example.com/data
```

**Explanation:**
- `-v` (verbose) — Shows the full conversation including headers
- `-I` — HEAD request, shows only response headers
- `-H` — Add a custom header to the request

## 🔒 HTTP vs HTTPS — What's the Difference?

### The Problem with HTTP

Regular HTTP sends everything in **plain text**. This means anyone sitting between you and the server can read everything:

```
YOU ──────── [Plain Text] ──────── SERVER
        │
        │  🕵️ Attacker can see:
        │  - Your passwords
        │  - Your credit card numbers
        │  - Your personal messages
        │  - Your cookies (session hijacking!)
        │  - Everything!
```

This is called a **Man-in-the-Middle (MITM) Attack**.

### The Solution: HTTPS

**HTTPS** = HTTP + **TLS** (Transport Layer Security)

It wraps the HTTP communication in an **encrypted tunnel** so nobody can read it:

```
YOU ──────── [🔐 Encrypted Data] ──────── SERVER
        │
        │  🕵️ Attacker sees:
        │  - @#$%^&*! (gibberish)
        │  - Can't read anything
        │  - Can't modify anything
        │  - Knows you're talking to the server, but not WHAT you're saying
```

### HTTP vs HTTPS — Quick Comparison

| Feature | HTTP | HTTPS |
|---------|------|-------|
| **Port** | 80 | 443 |
| **Encryption** | ❌ None | ✅ TLS/SSL |
| **Data Visible to Attackers** | ✅ Yes | ❌ No |
| **Certificate Required** | ❌ No | ✅ Yes (SSL/TLS cert) |
| **URL prefix** | `http://` | `https://` |
| **Performance** | Slightly faster | Tiny overhead (negligible today) |
| **SEO** | Penalized by Google | Boosted by Google |
| **Browser** | Shows "Not Secure" ⚠️ | Shows 🔒 padlock |

## 🔐 TLS/SSL — How HTTPS Actually Works

### Quick Terminology

| Term | What It Is |
|------|-----------|
| **SSL** | Secure Sockets Layer — the OLD encryption protocol (SSL 1.0, 2.0, 3.0). **Deprecated and insecure.** |
| **TLS** | Transport Layer Security — the NEW, better version. Current: **TLS 1.3** (2018). |
| **SSL Certificate** | Despite the name, modern "SSL certs" actually use TLS. The name stuck. |
| **CA** | Certificate Authority — a trusted organization that issues SSL/TLS certificates (Let's Encrypt, DigiCert, etc.) |

> **Important:** When people say "SSL," they almost always mean TLS. SSL is dead. We use TLS now. But the term "SSL certificate" is still used everywhere.

### How Does TLS Encryption Work? (The Handshake)

Before any encrypted data flows, the client and server need to agree on HOW to encrypt. This negotiation is called the **TLS Handshake**.

#### TLS 1.2 Handshake (Still Common)

```
Client (Browser)                                Server (Nginx/Apache)
       │                                               │
  1.   │──── ClientHello ─────────────────────────────>│
       │     "Hi! I support TLS 1.2, here are my      │
       │      supported cipher suites, and here's      │
       │      a random number (Client Random)"         │
       │                                               │
  2.   │<──── ServerHello ────────────────────────────│
       │      "Hi! Let's use TLS 1.2 with             │
       │       AES-256-GCM cipher, here's my           │
       │       random number (Server Random)"          │
       │                                               │
  3.   │<──── Certificate ────────────────────────────│
       │      "Here's my SSL/TLS certificate           │
       │       (contains my public key)"               │
       │                                               │
  4.   │ Client verifies the certificate:              │
       │   - Is it signed by a trusted CA?             │
       │   - Is it expired?                            │
       │   - Does the domain match?                    │
       │                                               │
  5.   │──── ClientKeyExchange ───────────────────────>│
       │     "Here's a Pre-Master Secret               │
       │      (encrypted with your public key)"        │
       │                                               │
  6.   │  Both sides now compute the SAME              │
       │  Session Key using:                           │
       │   - Client Random                             │
       │   - Server Random                             │
       │   - Pre-Master Secret                         │
       │                                               │
  7.   │──── ChangeCipherSpec ────────────────────────>│
       │     "I'm switching to encrypted mode now"     │
       │                                               │
  8.   │<──── ChangeCipherSpec ───────────────────────│
       │      "Me too! Let's go encrypted"             │
       │                                               │
  9.   │<════ 🔐 Encrypted HTTP Traffic ═════════════>│
       │      All data from now on is encrypted        │
       │      with the shared Session Key              │
```

This takes **2 round trips** (2-RTT) — that's extra latency.

#### TLS 1.3 Handshake (Modern & Faster)

TLS 1.3 is simpler and faster — it reduces the handshake to **1 round trip** (1-RTT):

```
Client (Browser)                                Server
       │                                               │
  1.   │──── ClientHello + KeyShare ──────────────────>│
       │     "Hi! Here are my supported groups          │
       │      AND my key share (I'm guessing            │
       │      which key exchange you'll pick)"          │
       │                                               │
  2.   │<──── ServerHello + KeyShare + Certificate ───│
       │      "Here's my key share, my cert,            │
       │       and let's start encrypting!"             │
       │                                               │
  3.   │<════ 🔐 Encrypted HTTP Traffic ═════════════>│
       │      Done! Only 1 round trip!                 │
```

**TLS 1.3 also supports 0-RTT** — if you've connected before, the next connection can send encrypted data immediately (no handshake at all!).

### Key Exchange — Asymmetric + Symmetric

TLS uses **two types of encryption**:

```
┌─────────────────────────────────────────────────────────────┐
│                    ASYMMETRIC ENCRYPTION                     │
│              (Public Key + Private Key)                       │
│                                                              │
│  Used ONLY during the handshake to securely exchange         │
│  the session key.                                            │
│                                                              │
│  - Slow but secure                                           │
│  - RSA, ECDHE (Elliptic Curve Diffie-Hellman)               │
│  - Public key: everyone can see it (in the certificate)      │
│  - Private key: only the server has it (NEVER shared)        │
└─────────────────────────────────────────────────────────────┘
                          │
                          │ exchanges the session key
                          ▼
┌─────────────────────────────────────────────────────────────┐
│                    SYMMETRIC ENCRYPTION                      │
│                   (One Shared Key)                            │
│                                                              │
│  Used for ALL actual data transfer after the handshake.      │
│                                                              │
│  - Fast and efficient                                        │
│  - AES-256-GCM, ChaCha20-Poly1305                           │
│  - Same key used by both client and server                   │
│  - This is the "Session Key"                                 │
└─────────────────────────────────────────────────────────────┘
```

**Why both?** Asymmetric is secure but SLOW. Symmetric is FAST but you need to share the key securely. So TLS uses asymmetric to securely share a symmetric key, then uses symmetric for everything else. Best of both worlds!

## 📜 SSL/TLS Certificates — What Are They?

An SSL/TLS certificate is like a **digital ID card** for a website. It proves:

1. **Identity** — "I am really google.com, not a fake site"
2. **Public Key** — "Here's my public key for encryption"
3. **Trust Chain** — "A trusted authority (CA) vouches for me"

### What's Inside a Certificate?

```
┌─────────────────────────────────────────────┐
│           SSL/TLS Certificate                │
├─────────────────────────────────────────────┤
│  Subject:    CN=www.example.com              │
│  Issuer:     Let's Encrypt Authority X3      │
│  Valid From: 2025-01-01                      │
│  Valid To:   2025-03-31                      │
│  Public Key: RSA 2048-bit                    │
│  SANs:       example.com, www.example.com    │
│  Signature:  SHA256withRSA (by the CA)       │
└─────────────────────────────────────────────┘
```

| Field | What It Means |
|-------|---------------|
| **Subject (CN)** | The domain this cert is for |
| **Issuer** | Who issued/signed this certificate (the CA) |
| **Valid From/To** | When the cert is valid |
| **Public Key** | The server's public key for encryption |
| **SANs** | Subject Alternative Names — additional domains covered |
| **Signature** | The CA's digital signature proving it's legit |

### Types of Certificates

| Type | What It Proves | Example |
|------|---------------|---------|
| **DV (Domain Validation)** | You own the domain | Let's Encrypt (free!) |
| **OV (Organization Validation)** | You own the domain + you're a real organization | DigiCert |
| **EV (Extended Validation)** | You own the domain + full company verification | Big banks, government sites |
| **Wildcard** | Covers all subdomains (`*.example.com`) | `*.example.com` |
| **SAN/Multi-domain** | Covers multiple different domains | `example.com`, `example.org` |

### Getting a Free Certificate with Let's Encrypt

```bash
# Install Certbot (the Let's Encrypt client)
sudo apt update
sudo apt install certbot python3-certbot-nginx

# Get a certificate for your domain (Nginx)
sudo certbot --nginx -d example.com -d www.example.com

# Get a certificate for your domain (Apache)
sudo certbot --apache -d example.com -d www.example.com

# Get a certificate without modifying web server config (standalone)
sudo certbot certonly --standalone -d example.com

# Auto-renew certificates (Let's Encrypt certs expire every 90 days)
sudo certbot renew --dry-run    # Test renewal
sudo crontab -e
# Add this line to auto-renew:
# 0 3 * * * certbot renew --quiet
```

**Explanation:**
- `certbot --nginx` — Gets a cert AND configures Nginx automatically
- `certonly --standalone` — Gets a cert but you configure the web server yourself
- `renew --dry-run` — Tests that renewal will work without actually renewing
- Cron job runs at 3 AM daily to check if renewal is needed

### Checking a Certificate

# Check a website's certificate
openssl s_client -connect example.com:443 -servername example.com < /dev/null 2>/dev/null | openssl x509 -text -noout

# Quick check — see expiry date

# Check a local certificate file

## ⚙️ Configuring HTTPS on Nginx

Here's a typical Nginx HTTPS configuration:

```nginx
# Redirect all HTTP traffic to HTTPS
server {
    listen 80;
    server_name example.com www.example.com;
    return 301 https://$host$request_uri;
}

# HTTPS server
server {
    listen 443 ssl http2;
    server_name example.com www.example.com;

    # SSL Certificate files
    ssl_certificate     /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

    # TLS configuration (secure settings)
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers 'ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384';
    ssl_prefer_server_ciphers off;

    # HSTS (force HTTPS for 1 year)
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;

    # OCSP Stapling (faster certificate validation)
    ssl_stapling on;
    ssl_stapling_verify on;

    # Your application
    location / {
        proxy_pass http://localhost:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

**Line-by-line explanation:**

| Line/Block | What It Does |
|-----------|-------------|
| `listen 80` | Listen on port 80 (HTTP) |
| `return 301 https://...` | Permanently redirect all HTTP to HTTPS |
| `listen 443 ssl http2` | Listen on port 443 with SSL and HTTP/2 |
| `ssl_certificate` | Path to the public certificate |
| `ssl_certificate_key` | Path to the private key (KEEP THIS SECRET!) |
| `ssl_protocols TLSv1.2 TLSv1.3` | Only allow TLS 1.2 and 1.3 (disable old, insecure versions) |
| `ssl_ciphers` | Which encryption algorithms to use |
| `Strict-Transport-Security` | Tell browsers to ALWAYS use HTTPS for this domain |
| `ssl_stapling on` | Speed up cert verification by stapling OCSP response |
| `proxy_pass` | Forward requests to your app running on port 3000 |
| `X-Forwarded-For` | Pass the real client IP to your backend |
| `X-Forwarded-Proto` | Tell the backend whether the original request was HTTP or HTTPS |

## 🔄 HTTP/1.1 vs HTTP/2 vs HTTP/3

| Feature | HTTP/1.1 | HTTP/2 | HTTP/3 |
|---------|---------|--------|--------|
| **Year** | 1997 | 2015 | 2022 |
| **Transport** | TCP | TCP | QUIC (UDP) |
| **Format** | Text-based | Binary | Binary |
| **Multiplexing** | ❌ (one request at a time per connection) | ✅ Multiple requests over one connection | ✅ Even better (no HOL blocking) |
| **Header Compression** | ❌ | ✅ HPACK | ✅ QPACK |
| **Server Push** | ❌ | ✅ | ❌ (removed) |
| **Head-of-Line Blocking** | ✅ Problem | ✅ Still an issue (at TCP level) | ❌ Solved |
| **Connection Setup** | TCP + TLS = 3 round trips | TCP + TLS = 3 round trips | QUIC = 1 round trip (0-RTT possible) |
| **TLS Required** | ❌ Optional | Practically required | ✅ Built-in (TLS 1.3) |

### What is Head-of-Line (HOL) Blocking?

Imagine a single-lane road:

```
HTTP/1.1 (one lane, one car at a time):
  🚗 ──── 🚗 ──── 🚗 ──── 🚗
  If the first car stops, EVERYONE waits.

HTTP/2 (multiple lanes, but same road/TCP):
  🚗 ──── 🚗
  🚙 ──── 🚙
  🚕 ──── 🚕
  If a TCP packet is lost, ALL lanes wait for retransmission.

HTTP/3 (separate roads for each car/QUIC):
  🚗 ──── 🚗    (independent stream)
  🚙 ──── 🚙    (independent stream)
  🚕 ──── 🚕    (independent stream)
  If one stream has a lost packet, only THAT stream waits. Others continue!
```

## 🧪 Debugging HTTP with Common Tools

### curl — The Swiss Army Knife

```bash
# Basic GET request
curl https://example.com

# See response headers + body
curl -i https://example.com

# See FULL conversation (request + response headers)
curl -v https://example.com

# Only see response headers
curl -I https://example.com

# Follow redirects
curl -L http://example.com    # Will follow 301/302 to https://

# POST with JSON data
curl -X POST https://api.example.com/data \
  -H "Content-Type: application/json" \
  -d '{"key": "value"}'

# Download a file
curl -O https://example.com/file.zip

# See timing information
curl -w "DNS: %{time_namelookup}s\nConnect: %{time_connect}s\nTLS: %{time_appconnect}s\nTotal: %{time_total}s\n" -o /dev/null -s https://example.com
```

### httpie — A User-Friendly Alternative

```bash
# Install
sudo apt install httpie

# GET request (beautiful formatted output)
http GET https://api.example.com/users

# POST with JSON (automatic Content-Type)
http POST https://api.example.com/users name=Ritesh role="DevOps"

# Custom headers
http GET https://api.example.com/data Authorization:"Bearer my-token"
```

### Checking HTTPS Configuration

```bash
# Test SSL/TLS configuration of a site
curl -vI https://example.com 2>&1 | grep -E "SSL|TLS|subject|issuer|expire"

# Detailed SSL test (what protocols/ciphers are supported)
nmap --script ssl-enum-ciphers -p 443 example.com

# Check certificate expiry
echo | openssl s_client -connect example.com:443 2>/dev/null | openssl x509 -noout -enddate
```

## 🧠 Summary — Key Takeaways

| Concept | Key Point |
|---------|-----------|
| **HTTP** | Request-response protocol for the web (Layer 7) |
| **HTTPS** | HTTP + TLS encryption — always use this |
| **Methods** | GET (read), POST (create), PUT (replace), PATCH (update), DELETE (remove) |
| **Status Codes** | 2xx = success, 3xx = redirect, 4xx = client error, 5xx = server error |
| **Headers** | Metadata about the request/response |
| **TLS Handshake** | Client & server negotiate encryption, verify certificate, establish session key |
| **TLS 1.3** | Faster (1-RTT), more secure, removes weak algorithms |
| **Certificates** | Digital ID proving a server's identity (get free ones from Let's Encrypt) |
| **HTTP/2** | Binary, multiplexing, header compression — big upgrade |
| **HTTP/3** | Uses QUIC (UDP), eliminates HOL blocking — the future |
