# 3.6 — Ports, Sockets & Firewalls (iptables, ufw, firewalld, nftables)

> **One Line Answer:** Ports are like **apartment numbers** on a building (IP address), sockets are the **actual doorways** where conversations happen, and firewalls are the **security guards** who decide who gets in and who gets kicked out.

---

## 🤔 Why Should a DevOps Engineer Care About Ports, Sockets & Firewalls?

This is one of the **most practical** topics you'll ever learn. You'll use this knowledge literally every day:

| Task | Why This Matters |
|------|-----------------|
| Deploy an application | Need to expose the right port |
| Debug "connection refused" errors | Wrong port? Firewall blocking? Socket not listening? |
| Secure a server | Close unnecessary ports, set up firewall rules |
| Configure Kubernetes Services | Port, targetPort, nodePort — you must get these right |
| Set up Docker port mapping | `-p 8080:80` — what does this even mean? |
| Troubleshoot networking | "Is the port open? Is something listening? Is the firewall blocking?" |
| Pass security audits | Auditors will check your open ports and firewall rules |
| Handle production incidents | "Port 443 is not responding!" — you need to fix this fast |

## 🚪 Part 1: Ports — The Complete Guide

### What is a Port?

An IP address identifies a **machine** on the network. But a single machine can run **many services** — a web server, a database, SSH, etc. How does the machine know which service a packet is for?

**That's what ports are for.**

A port is a **16-bit number** (0 to 65535) that identifies a specific **service or application** on a machine.

### The Analogy

Think of it like a big apartment building:

```
IP Address = Building Address (123 Main Street)
Port       = Apartment Number

┌──────────────────────────────────────────┐
│           192.168.1.100                   │
│          (The Building)                   │
├──────────────────────────────────────────┤
│                                           │
│  Apt 22  = SSH Server         🔐          │
│  Apt 80  = Web Server (HTTP)  🌐          │
│  Apt 443 = Web Server (HTTPS) 🔒          │
│  Apt 3306 = MySQL Database    🗄️          │
│  Apt 6379 = Redis Cache       ⚡          │
│  Apt 9090 = Prometheus        📊          │
│                                           │
└──────────────────────────────────────────┘
```

When someone sends a packet to `192.168.1.100:443`, it's like saying "deliver this to apartment 443 at 123 Main Street."

### Port Ranges

The 65,536 ports (0-65535) are divided into three ranges:

| Range | Name | Who Uses Them |
|-------|------|--------------|
| **0 - 1023** | Well-Known Ports | Reserved for standard services (HTTP=80, SSH=22). Need **root/admin** to bind to these. |
| **1024 - 49151** | Registered Ports | Used by specific applications (MySQL=3306, PostgreSQL=5432). Any user can bind. |
| **49152 - 65535** | Dynamic/Ephemeral Ports | Temporary ports assigned by the OS for outgoing connections. |

### Working with Ports — Essential Commands

#### Check What's Listening on Which Ports

```bash
# Show all listening ports with process names (MOST USEFUL COMMAND)
sudo ss -tlnp

# Explanation:
# -t = TCP only
# -l = only listening sockets
# -n = show port numbers (don't resolve to names)
# -p = show the process using the port
```

Output looks like:

```
State   Recv-Q  Send-Q  Local Address:Port   Peer Address:Port  Process
LISTEN  0       128     0.0.0.0:22           0.0.0.0:*          users:(("sshd",pid=1234))
LISTEN  0       511     0.0.0.0:80           0.0.0.0:*          users:(("nginx",pid=5678))
LISTEN  0       128     127.0.0.1:3306       0.0.0.0:*          users:(("mysqld",pid=9012))
```

**Reading this output:**
- `0.0.0.0:22` — SSH is listening on **all interfaces** on port 22
- `0.0.0.0:80` — Nginx is listening on **all interfaces** on port 80
- `127.0.0.1:3306` — MySQL is listening **only on localhost** (can't be accessed from outside)

```bash
# Show all listening ports (TCP + UDP)
sudo ss -tulnp

# Same thing with netstat (older but still common)
sudo netstat -tulnp

# Check if a specific port is in use
sudo ss -tlnp | grep :8080

# Find which process is using port 80
sudo lsof -i :80

# Check all connections (not just listening)
ss -tan

# Check UDP ports
sudo ss -ulnp
```

#### Check if a Remote Port is Open

```bash
# Using netcat
nc -zv remote-server.com 443
# Output: Connection to remote-server.com port 443 [tcp/https] succeeded!

# Using curl (for HTTP/HTTPS ports)
curl -I https://remote-server.com

# Using telnet (old school but works)
telnet remote-server.com 22

# Using nmap (scan multiple ports)
nmap -p 22,80,443,3306 remote-server.com

# Scan a range of ports
nmap -p 1-1000 remote-server.com

# Quick scan of common ports
nmap remote-server.com
```

### Ephemeral Ports — The Ports You Don't Think About

When your machine makes an **outgoing connection**, the OS picks a random **ephemeral port** as the source port:

```
Your Machine                                     Web Server
192.168.1.100:52431 ─────────────────────> 93.184.216.34:443
                 │                                      │
                 │                                      │
          Source Port                           Destination Port
        (ephemeral — OS                        (well-known — 443
         picked 52431                           is HTTPS)
         randomly)
```

You can see the ephemeral port range on your system:

```bash
# Check ephemeral port range
cat /proc/sys/net/ipv4/ip_local_port_range
# Output: 32768   60999

# Change ephemeral port range (if needed)
sudo sysctl -w net.ipv4.ip_local_port_range="10000 65535"
```

## 🔌 Part 2: Sockets — The Complete Guide

### What is a Socket?

A port is just a number. A **socket** is the actual **endpoint** where communication happens. It's the combination of:

```
Socket = IP Address + Port + Protocol

Example:
  TCP socket → 192.168.1.100:80 (TCP)
  UDP socket → 192.168.1.100:53 (UDP)
```

A socket is like a **phone line plugged in and ready to receive calls**. The port is the phone number, but the socket is the actual physical connection.

### A Connection is Defined by Two Sockets

Every network connection is uniquely identified by a **5-tuple**:

```
┌──────────────────────────────────────────────────────────────┐
│                   A Network Connection                        │
├──────────────────────────────────────────────────────────────┤
│  1. Protocol:     TCP                                         │
│  2. Source IP:    192.168.1.100                                │
│  3. Source Port:  52431                                        │
│  4. Dest IP:      93.184.216.34                                │
│  5. Dest Port:    443                                          │
└──────────────────────────────────────────────────────────────┘
```

This means the **same port** can have thousands of connections — as long as the 5-tuple is different:

```
Client A: 192.168.1.100:52431 → 93.184.216.34:443   ← Connection 1
Client A: 192.168.1.100:52432 → 93.184.216.34:443   ← Connection 2 (different source port)
Client B: 10.0.0.50:48721     → 93.184.216.34:443   ← Connection 3 (different source IP)
```

All three hit port 443, but they're **different connections** because the 5-tuples are different.

### Socket Types

| Type | Protocol | Description |
|------|----------|-------------|
| **Stream Socket** (SOCK_STREAM) | TCP | Reliable, ordered, connection-oriented |
| **Datagram Socket** (SOCK_DGRAM) | UDP | Unreliable, connectionless |
| **Raw Socket** (SOCK_RAW) | IP | Direct access to lower-level protocols (used by ping, traceroute) |

### How a TCP Server Socket Works

```
Server                                          Client
   │                                               │
   │  1. socket() — Create a socket                │
   │  2. bind() — Attach to IP:Port                │
   │  3. listen() — Start listening                 │
   │  4. accept() — Wait for connections            │
   │                                               │
   │<──── Client connects ────────────────────────│
   │                                               │
   │  5. accept() returns a NEW socket             │
   │     for this specific client                  │
   │                                               │
   │  Original socket keeps listening              │
   │  New socket handles this client               │
   │                                               │
   │  6. read()/write() — Exchange data            │
   │  7. close() — Close when done                 │
```

**Key insight:** The `listen` socket stays put and keeps accepting new connections. Each new connection gets its **own socket**. That's how a web server handles thousands of connections on port 80 — one listening socket, thousands of connected sockets.

### Socket States — Same as TCP States

When you run `ss` or `netstat`, the states you see are actually **socket states**:

```bash
# See all socket states
ss -tan

# You'll see:
# LISTEN      — Waiting for connections (server socket)
# ESTABLISHED — Active connection
# TIME_WAIT   — Connection closing (waiting for stale packets)
# CLOSE_WAIT  — Other side closed, waiting for app to close
# FIN_WAIT_1  — Sent FIN, waiting for ACK
# FIN_WAIT_2  — Got ACK for FIN, waiting for other side's FIN
```

### Socket Limits and Tuning

A single server has limits on how many sockets (connections) it can handle:

```bash
# Check max number of open file descriptors (sockets are file descriptors in Linux!)
ulimit -n
# Default: usually 1024 — WAY too low for a web server!

# Increase for current session
ulimit -n 65535

# Increase permanently (edit /etc/security/limits.conf)
# Add these lines:
# * soft nofile 65535
# * hard nofile 65535

# Check max connections system-wide
cat /proc/sys/net/core/somaxconn
# Default: 4096

# Increase it
sudo sysctl -w net.core.somaxconn=65535

# Check max number of file descriptors system-wide
cat /proc/sys/fs/file-max
```

### Socket Binding — 0.0.0.0 vs 127.0.0.1

This is a **critical** security concept:

```
┌─────────────────────────────────────────────────────────────────┐
│  Binding to 127.0.0.1:3306                                      │
│  (localhost only)                                                │
│                                                                  │
│  ✅ Accessible from the same machine                            │
│  ❌ NOT accessible from outside                                 │
│  📌 Use for: databases, internal services                        │
│                                                                  │
│  Example: MySQL listening on 127.0.0.1:3306                      │
│  → Only local apps can connect                                   │
│  → Secure by default!                                            │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  Binding to 0.0.0.0:80                                          │
│  (all interfaces)                                                │
│                                                                  │
│  ✅ Accessible from everywhere (local + network + internet)     │
│  ⚠️ Make sure your firewall is configured!                      │
│  📌 Use for: web servers, APIs, public services                  │
│                                                                  │
│  Example: Nginx listening on 0.0.0.0:80                          │
│  → Anyone can connect                                            │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  Binding to 10.0.1.5:8080                                       │
│  (specific interface)                                            │
│                                                                  │
│  ✅ Only accessible via that specific network interface         │
│  📌 Use for: services that should only be on certain networks    │
└─────────────────────────────────────────────────────────────────┘
```

## 🔥 Part 3: Firewalls — The Complete Guide

### What is a Firewall?

A firewall is a **security system** that monitors and controls incoming and outgoing network traffic based on a set of **rules**. It decides whether to **allow**, **deny**, or **drop** traffic.

```
Internet                Firewall                    Your Server
   │                       │                            │
   │── SSH (port 22) ────>│── ✅ ALLOW ──────────────>│  (You need SSH)
   │                       │                            │
   │── HTTP (port 80) ───>│── ✅ ALLOW ──────────────>│  (Web traffic)
   │                       │                            │
   │── MySQL (3306) ─────>│── ❌ DROP ────────────────>│  (Don't expose DB!)
   │                       │                            │
   │── Random (31337) ──>│── ❌ DROP ────────────────>│  (Suspicious port)
   │                       │                            │
```

### REJECT vs DROP — What's the Difference?

| Action | What Happens | When to Use |
|--------|-------------|-------------|
| **ACCEPT** | Packet is allowed through | For legitimate traffic |
| **DROP** | Packet is silently discarded (no response) | Default for most blocked traffic. Attacker doesn't know if the port exists |
| **REJECT** | Packet is blocked AND sender gets an error message | When you want the sender to know explicitly (internal networks) |

```
DROP:    Attacker sends packet → nothing happens → attacker doesn't know if server exists
REJECT:  Attacker sends packet → gets "Connection Refused" → attacker knows server exists
```

**Best practice:** Use **DROP** for external traffic (don't give attackers any info) and **REJECT** for internal traffic (so internal tools get clear error messages).

## 🔧 iptables — The Classic Linux Firewall

`iptables` is the **traditional** Linux firewall tool. It's been around since 2001 and is still widely used. Understanding iptables is essential because:
- It's the **foundation** that other tools (ufw, firewalld) are built on top of
- Docker uses iptables for container networking
- Kubernetes uses iptables (or its successor nftables) for service routing

### How iptables Works — Tables, Chains & Rules

```
┌──────────────────────────────────────────────────────────────┐
│                      iptables Architecture                    │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  Tables (collections of chains):                              │
│  ┌─────────────────────────────────────────┐                  │
│  │  filter (default) — Accept/drop packets │                  │
│  │  nat — Network Address Translation       │                  │
│  │  mangle — Modify packet headers          │                  │
│  │  raw — Bypass connection tracking        │                  │
│  └─────────────────────────────────────────┘                  │
│                                                               │
│  Chains (within each table):                                  │
│  ┌─────────────────────────────────────────┐                  │
│  │  INPUT    — Packets coming IN to server │                  │
│  │  OUTPUT   — Packets going OUT from server│                  │
│  │  FORWARD  — Packets passing THROUGH      │                  │
│  │             (server acting as router)    │                  │
│  └─────────────────────────────────────────┘                  │
│                                                               │
│  Each chain has a list of RULES checked in order:             │
│  Rule 1 → Rule 2 → Rule 3 → ... → Default Policy             │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

### The Packet Flow

```
                    Incoming Packet
                         │
                         ▼
                    ┌──────────┐
                    │ PREROUTING│ (nat, mangle)
                    └────┬─────┘
                         │
                    Is it for this    
                      machine?        
                    ╱          ╲      
                  Yes           No    
                  ╱               ╲   
            ┌────┴────┐     ┌────┴────┐
            │  INPUT   │     │ FORWARD │
            │ (filter) │     │ (filter)│
            └────┬────┘     └────┬────┘
                 │               │
            ┌────┴────┐     ┌────┴────┐
            │  Local   │     │POSTROUTING│
            │ Process  │     │  (nat)   │
            └────┬────┘     └────┬────┘
                 │               │
            ┌────┴────┐          ▼
            │ OUTPUT   │    Outgoing Packet
            │ (filter) │    (forwarded)
            └────┬────┘
                 │
            ┌────┴─────┐
            │POSTROUTING│ (nat)
            └────┬─────┘
                 ▼
           Outgoing Packet
```

### Essential iptables Commands

#### Viewing Rules

```bash
# List all rules (most useful command)
sudo iptables -L -n -v

# Explanation:
# -L = list rules
# -n = show numbers instead of resolving hostnames (faster)
# -v = verbose (show packet/byte counters)

# List rules with line numbers (useful for deleting specific rules)
sudo iptables -L -n -v --line-numbers

# List rules for a specific chain
sudo iptables -L INPUT -n -v

# List NAT rules (important for Docker!)
sudo iptables -t nat -L -n -v
```

#### Adding Rules

```bash
# Allow SSH (port 22) from anywhere
sudo iptables -A INPUT -p tcp --dport 22 -j ACCEPT

# Explanation:
# -A INPUT     = Append to the INPUT chain
# -p tcp       = Protocol is TCP
# --dport 22   = Destination port is 22
# -j ACCEPT    = Jump to ACCEPT (allow the packet)
```

```bash
# Allow HTTP and HTTPS
sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 443 -j ACCEPT

# Allow from a specific IP only
sudo iptables -A INPUT -p tcp --dport 22 -s 10.0.1.100 -j ACCEPT
# -s 10.0.1.100 = Source IP is 10.0.1.100

# Allow from a subnet
sudo iptables -A INPUT -p tcp --dport 3306 -s 10.0.1.0/24 -j ACCEPT
# -s 10.0.1.0/24 = Any IP in the 10.0.1.x range

# Allow established connections (CRITICAL — without this, responses won't come back!)
sudo iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# Allow loopback (localhost) traffic
sudo iptables -A INPUT -i lo -j ACCEPT

# Allow ping (ICMP)
sudo iptables -A INPUT -p icmp -j ACCEPT
```

#### Blocking Traffic

```bash
# Block a specific IP
sudo iptables -A INPUT -s 203.0.113.50 -j DROP

# Block a port from outside
sudo iptables -A INPUT -p tcp --dport 3306 -j DROP

# Block everything except what's explicitly allowed (set default policy)
sudo iptables -P INPUT DROP
sudo iptables -P FORWARD DROP
sudo iptables -P OUTPUT ACCEPT    # Usually ACCEPT — allow outgoing traffic

# ⚠️ WARNING: Always add your SSH ACCEPT rule BEFORE setting default policy to DROP!
# Otherwise you'll lock yourself out!
```

#### Deleting Rules

```bash
# Delete by rule number (see numbers with --line-numbers)
sudo iptables -D INPUT 3    # Delete rule #3 from INPUT chain

# Delete by exact rule specification
sudo iptables -D INPUT -p tcp --dport 8080 -j ACCEPT

# Flush (delete) ALL rules — ⚠️ DANGEROUS
sudo iptables -F
```

### A Complete iptables Setup Script

```bash
#!/bin/bash
# Basic server firewall setup

# Flush existing rules
iptables -F
iptables -X

# Set default policies — drop everything by default
iptables -P INPUT DROP
iptables -P FORWARD DROP
iptables -P OUTPUT ACCEPT

# Allow loopback (localhost)
iptables -A INPUT -i lo -j ACCEPT

# Allow established and related connections
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# Allow SSH (port 22)
iptables -A INPUT -p tcp --dport 22 -j ACCEPT

# Allow HTTP (port 80) and HTTPS (port 443)
iptables -A INPUT -p tcp --dport 80 -j ACCEPT
iptables -A INPUT -p tcp --dport 443 -j ACCEPT

# Allow ping
iptables -A INPUT -p icmp --icmp-type echo-request -j ACCEPT

# Allow DNS (UDP port 53) — for outgoing DNS resolution
iptables -A INPUT -p udp --sport 53 -j ACCEPT

# Log dropped packets (optional — useful for debugging)
iptables -A INPUT -j LOG --log-prefix "IPTables-Dropped: " --log-level 4

# Drop everything else (already handled by default policy, but explicit is nice)
iptables -A INPUT -j DROP

echo "Firewall rules applied!"
```

**Line-by-line explanation:**

| Line | What It Does |
|------|-------------|
| `iptables -F` | Flush (clear) all existing rules |
| `iptables -X` | Delete all custom chains |
| `-P INPUT DROP` | Default: drop all incoming traffic |
| `-P FORWARD DROP` | Default: don't forward packets (not a router) |
| `-P OUTPUT ACCEPT` | Default: allow all outgoing traffic |
| `-i lo -j ACCEPT` | Allow traffic on loopback (localhost-to-localhost) |
| `--state ESTABLISHED,RELATED` | Allow responses to connections WE initiated |
| `--dport 22 -j ACCEPT` | Allow incoming SSH |
| `--dport 80 -j ACCEPT` | Allow incoming HTTP |
| `--dport 443 -j ACCEPT` | Allow incoming HTTPS |
| `--icmp-type echo-request` | Allow incoming ping |
| `-j LOG` | Log any packets that reach this point (about to be dropped) |

### Saving iptables Rules

iptables rules are **lost on reboot** unless saved!

```bash
# On Debian/Ubuntu
sudo apt install iptables-persistent
sudo netfilter-persistent save
sudo netfilter-persistent reload

# On CentOS/RHEL
sudo service iptables save
# Rules saved to /etc/sysconfig/iptables

# Manual save/restore
sudo iptables-save > /etc/iptables.rules
sudo iptables-restore < /etc/iptables.rules
```

## 🛡️ ufw — The Uncomplicated Firewall

`ufw` is a **user-friendly frontend** for iptables. It's the default firewall tool on **Ubuntu** and is much easier to use than raw iptables.

### Think of It This Way

```
iptables = Manual transmission car (full control, complex)
ufw      = Automatic transmission car (easy to drive, does the hard stuff for you)

Under the hood, ufw is just generating iptables rules for you!
```

### Essential ufw Commands

#### Enable/Disable

```bash
# Enable the firewall
sudo ufw enable

# Disable the firewall
sudo ufw disable

# Check status
sudo ufw status
sudo ufw status verbose        # More detail
sudo ufw status numbered        # With rule numbers (for deleting)
```

#### Allow Traffic

```bash
# Allow SSH (ALWAYS do this BEFORE enabling ufw!)
sudo ufw allow ssh
# or by port number:
sudo ufw allow 22/tcp

# Allow HTTP and HTTPS
sudo ufw allow http
sudo ufw allow https
# or:
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp

# Allow a port range
sudo ufw allow 8000:9000/tcp

# Allow from specific IP
sudo ufw allow from 10.0.1.100

# Allow from specific IP to specific port
sudo ufw allow from 10.0.1.100 to any port 3306

# Allow from a subnet
sudo ufw allow from 10.0.1.0/24 to any port 22

# Allow both TCP and UDP
sudo ufw allow 53        # Both TCP and UDP for DNS
```

#### Deny/Block Traffic

```bash
# Block a specific port
sudo ufw deny 3306

# Block a specific IP
sudo ufw deny from 203.0.113.50

# Block a specific IP from a specific port
sudo ufw deny from 203.0.113.50 to any port 22
```

#### Delete Rules

```bash
# Delete by rule number
sudo ufw status numbered
sudo ufw delete 3        # Delete rule #3

# Delete by specification
sudo ufw delete allow 8080/tcp
```

#### Set Default Policies

```bash
# Default: deny incoming, allow outgoing (RECOMMENDED)
sudo ufw default deny incoming
sudo ufw default allow outgoing
```

### Complete ufw Setup Example

```bash
# 1. Set defaults
sudo ufw default deny incoming
sudo ufw default allow outgoing

# 2. Allow SSH (DO THIS FIRST!)
sudo ufw allow ssh

# 3. Allow web traffic
sudo ufw allow http
sudo ufw allow https

# 4. Allow your app port
sudo ufw allow 8080/tcp

# 5. Allow MySQL only from app server
sudo ufw allow from 10.0.1.20 to any port 3306

# 6. Enable the firewall
sudo ufw enable

# 7. Verify
sudo ufw status verbose
```

Output:

```
Status: active
Logging: on (low)
Default: deny (incoming), allow (outgoing), disabled (routed)

To                         Action      From
--                         ------      ----
22/tcp                     ALLOW IN    Anywhere
80/tcp                     ALLOW IN    Anywhere
443/tcp                    ALLOW IN    Anywhere
8080/tcp                   ALLOW IN    Anywhere
3306                       ALLOW IN    10.0.1.20
```

### ufw Application Profiles

ufw has pre-configured profiles for common applications:

```bash
# List available profiles
sudo ufw app list

# Output:
# Available applications:
#   Nginx Full
#   Nginx HTTP
#   Nginx HTTPS
#   OpenSSH
#   Apache Full

# Allow Nginx (both HTTP and HTTPS)
sudo ufw allow "Nginx Full"

# Allow only Nginx HTTPS
sudo ufw allow "Nginx HTTPS"

# See what ports a profile includes
sudo ufw app info "Nginx Full"
# Output:
# Profile: Nginx Full
# Ports: 80,443/tcp
```

## 🔥 firewalld — The RHEL/CentOS Firewall

`firewalld` is the default firewall on **Red Hat, CentOS, Rocky Linux, AlmaLinux, and Fedora**. It uses the concept of **zones** to manage rules.

### Zones — The Key Concept

Instead of writing individual rules, firewalld groups interfaces into **zones** with predefined trust levels:

```
┌──────────────────────────────────────────────────────┐
│                   firewalld Zones                     │
├──────────────────────────────────────────────────────┤
│                                                       │
│  drop     → DROP everything, no response              │
│  block    → REJECT everything with error message      │
│  public   → Default. Only selected services allowed   │
│  external → For NAT/masquerading (external network)   │
│  dmz      → Limited access for DMZ servers            │
│  work     → Trust work network, more services allowed │
│  home     → Trust home network, even more allowed     │
│  internal → Trust internal network                    │
│  trusted  → ACCEPT everything (full trust)            │
│                                                       │
└──────────────────────────────────────────────────────┘

Least Trust ←──────────────────────────────→ Most Trust
  drop  block  public  external  dmz  work  home  internal  trusted
```

### Essential firewalld Commands

The main command is `firewall-cmd`:

#### Check Status

```bash
# Check if firewalld is running
sudo firewall-cmd --state

# See the default zone
sudo firewall-cmd --get-default-zone

# List all zones
sudo firewall-cmd --get-zones

# See active zones (which zone each interface is in)
sudo firewall-cmd --get-active-zones

# List all rules in the default zone
sudo firewall-cmd --list-all

# List rules for a specific zone
sudo firewall-cmd --zone=public --list-all
```

#### Allow Services

```bash
# Allow SSH (already allowed by default in public zone)
sudo firewall-cmd --zone=public --add-service=ssh --permanent

# Allow HTTP and HTTPS
sudo firewall-cmd --zone=public --add-service=http --permanent
sudo firewall-cmd --zone=public --add-service=https --permanent

# Allow a specific port
sudo firewall-cmd --zone=public --add-port=8080/tcp --permanent

# Allow a port range
sudo firewall-cmd --zone=public --add-port=8000-9000/tcp --permanent

# ⚠️ IMPORTANT: --permanent makes the change survive reboots
# Without --permanent, the rule is TEMPORARY (lost on restart)

# After adding --permanent rules, reload to apply:
sudo firewall-cmd --reload
```

**Key point about `--permanent`:**
- Without `--permanent` → Rule takes effect immediately but disappears on restart
- With `--permanent` → Rule is saved but doesn't take effect until reload
- Best practice: Use `--permanent` and then `--reload`

#### Remove Rules

```bash
# Remove a service
sudo firewall-cmd --zone=public --remove-service=http --permanent

# Remove a port
sudo firewall-cmd --zone=public --remove-port=8080/tcp --permanent

# Reload after changes
sudo firewall-cmd --reload
```

#### Rich Rules — Advanced Firewall Rules

Rich rules give you iptables-like flexibility within firewalld:

```bash
# Allow MySQL only from a specific IP
sudo firewall-cmd --zone=public --add-rich-rule='
  rule family="ipv4"
  source address="10.0.1.20"
  port protocol="tcp" port="3306"
  accept' --permanent

# Block a specific IP
sudo firewall-cmd --zone=public --add-rich-rule='
  rule family="ipv4"
  source address="203.0.113.50"
  drop' --permanent

# Rate limit SSH (prevent brute force)
sudo firewall-cmd --zone=public --add-rich-rule='
  rule service name="ssh"
  accept
  limit value="5/m"' --permanent

# Don't forget to reload!
sudo firewall-cmd --reload

# List rich rules
sudo firewall-cmd --zone=public --list-rich-rules
```

### Complete firewalld Setup Example

```bash
# 1. Start and enable firewalld
sudo systemctl start firewalld
sudo systemctl enable firewalld

# 2. Set default zone to public
sudo firewall-cmd --set-default-zone=public

# 3. Allow necessary services
sudo firewall-cmd --zone=public --add-service=ssh --permanent
sudo firewall-cmd --zone=public --add-service=http --permanent
sudo firewall-cmd --zone=public --add-service=https --permanent

# 4. Allow app port
sudo firewall-cmd --zone=public --add-port=8080/tcp --permanent

# 5. Allow MySQL only from app server
sudo firewall-cmd --zone=public --add-rich-rule='
  rule family="ipv4"
  source address="10.0.1.20"
  port protocol="tcp" port="3306"
  accept' --permanent

# 6. Reload to apply
sudo firewall-cmd --reload

# 7. Verify
sudo firewall-cmd --zone=public --list-all
```

Output:

```
public (active)
  target: default
  icmp-block-inversion: no
  interfaces: eth0
  sources:
  services: ssh http https
  ports: 8080/tcp
  protocols:
  masquerade: no
  forward-ports:
  source-ports:
  icmp-blocks:
  rich rules:
    rule family="ipv4" source address="10.0.1.20" port port="3306" protocol="tcp" accept
```

## 🆕 nftables — The Modern Replacement

`nftables` is the **successor to iptables**. It replaces iptables, ip6tables, arptables, and ebtables with a single, unified framework. It's the default in newer Linux kernels (3.13+) and is already the default backend for firewalld and Docker in modern distros.

### Why nftables?

| Feature | iptables | nftables |
|---------|---------|---------|
| **Syntax** | Separate commands per rule | Clean, scripting-friendly syntax |
| **Performance** | Linear rule matching | Better performance with sets and maps |
| **IPv4 + IPv6** | Separate tools (iptables, ip6tables) | Single tool handles both |
| **Atomic Updates** | Replace rules one by one | Load entire ruleset at once |
| **Kernel Support** | Legacy | Modern — uses the same netfilter framework but better |

### nftables Concepts

```
iptables                          nftables
─────────                         ────────
Tables (filter, nat, mangle)  →   Tables (you name them anything)
Chains (INPUT, OUTPUT, FORWARD)→  Chains (you create them with hooks)
Rules                          →   Rules (cleaner syntax)
Matches                        →   Expressions
Targets (ACCEPT, DROP)         →   Verdicts (accept, drop)
```

### Essential nftables Commands

#### Viewing Rules

```bash
# List all rules
sudo nft list ruleset

# List a specific table
sudo nft list table inet my_firewall

# List a specific chain
sudo nft list chain inet my_firewall input
```

#### Creating a Firewall from Scratch

```bash
# Create a table (inet = both IPv4 and IPv6)
sudo nft add table inet my_firewall

# Create chains with hooks
sudo nft add chain inet my_firewall input { type filter hook input priority 0 \; policy drop \; }
sudo nft add chain inet my_firewall forward { type filter hook forward priority 0 \; policy drop \; }
sudo nft add chain inet my_firewall output { type filter hook output priority 0 \; policy accept \; }
```

**Explanation:**
- `inet` — Handles both IPv4 and IPv6
- `my_firewall` — Table name (you choose it)
- `type filter` — This is a filtering chain
- `hook input` — Attached to the INPUT hook (incoming traffic)
- `priority 0` — Standard priority
- `policy drop` — Default action: drop everything

#### Adding Rules

```bash
# Allow loopback
sudo nft add rule inet my_firewall input iif lo accept

# Allow established connections
sudo nft add rule inet my_firewall input ct state established,related accept

# Allow SSH
sudo nft add rule inet my_firewall input tcp dport 22 accept

# Allow HTTP and HTTPS (using a set — nftables superpower!)
sudo nft add rule inet my_firewall input tcp dport { 80, 443 } accept

# Allow ping
sudo nft add rule inet my_firewall input icmp type echo-request accept
sudo nft add rule inet my_firewall input icmpv6 type echo-request accept

# Allow from specific IP
sudo nft add rule inet my_firewall input ip saddr 10.0.1.20 tcp dport 3306 accept

# Log dropped packets
sudo nft add rule inet my_firewall input log prefix "nft-dropped: " drop
```

#### Using an nftables Configuration File

The real power of nftables is using a config file — much cleaner than one command at a time:

```bash
# Create/edit the nftables config file
sudo nano /etc/nftables.conf
```

```
#!/usr/sbin/nft -f

# Flush existing rules
flush ruleset

# Define the firewall table
table inet my_firewall {
    
    # Input chain — incoming traffic
    chain input {
        type filter hook input priority 0; policy drop;
        
        # Allow loopback
        iif lo accept
        
        # Allow established/related connections
        ct state established,related accept
        
        # Drop invalid connections
        ct state invalid drop
        
        # Allow ping (rate limited)
        icmp type echo-request limit rate 5/second accept
        icmpv6 type echo-request limit rate 5/second accept
        
        # Allow SSH
        tcp dport 22 accept
        
        # Allow HTTP and HTTPS
        tcp dport { 80, 443 } accept
        
        # Allow custom app port
        tcp dport 8080 accept
        
        # Allow MySQL from app server only
        ip saddr 10.0.1.20 tcp dport 3306 accept
        
        # Log everything else before dropping
        log prefix "nft-dropped: " counter drop
    }
    
    # Forward chain — traffic passing through
    chain forward {
        type filter hook forward priority 0; policy drop;
    }
    
    # Output chain — outgoing traffic
    chain output {
        type filter hook output priority 0; policy accept;
    }
}
```

```bash
# Apply the config
sudo nft -f /etc/nftables.conf

# Enable nftables to start at boot
sudo systemctl enable nftables

# Verify
sudo nft list ruleset
```

### nftables Sets — A Powerful Feature

Sets let you group IPs, ports, or networks — way cleaner than multiple rules:

```
# Define a set of allowed IPs
table inet my_firewall {
    set trusted_ips {
        type ipv4_addr
        elements = { 10.0.1.20, 10.0.1.21, 10.0.1.22 }
    }
    
    set web_ports {
        type inet_service
        elements = { 80, 443, 8080, 8443 }
    }
    
    chain input {
        type filter hook input priority 0; policy drop;
        
        # Allow trusted IPs to access MySQL
        ip saddr @trusted_ips tcp dport 3306 accept
        
        # Allow web traffic on all web ports
        tcp dport @web_ports accept
    }
}
```

You can even **dynamically add/remove** from sets:

```bash
# Add an IP to the trusted set
sudo nft add element inet my_firewall trusted_ips { 10.0.1.30 }

# Remove an IP
sudo nft delete element inet my_firewall trusted_ips { 10.0.1.30 }
```

|---------|---------|-----|-----------|---------|
| **Distro** | All Linux | Ubuntu/Debian | RHEL/CentOS/Fedora | Modern Linux (all) |

```
├── Need complex rules? → nftables or raw iptables
```

## 🐛 Troubleshooting — Common Port & Firewall Issues

### "Connection Refused" vs "Connection Timed Out"

```
"Connection Refused" (immediate error)
  → The port is reachable but NOTHING is listening on it
  → OR the firewall is using REJECT (not DROP)
  → Fix: Start the service, check the port binding

"Connection Timed Out" (long wait, then error)
  → The packet is being DROPPED by a firewall
  → OR the host is unreachable
  → Fix: Check firewall rules, check network connectivity
```

### Debugging Checklist

```bash
# 1. Is the service running?
sudo systemctl status nginx
sudo systemctl status mysql

# 2. What port is it listening on?
sudo ss -tlnp | grep nginx
sudo ss -tlnp | grep mysql

# 3. Is it bound to the right interface?
# 0.0.0.0 = all interfaces (accessible from outside)
# 127.0.0.1 = localhost only (NOT accessible from outside)

# 4. Is the local firewall allowing it?
sudo iptables -L INPUT -n -v
# or
sudo ufw status
# or
sudo firewall-cmd --list-all

# 5. Can you reach the port from another machine?
nc -zv target-server 443

# 6. Is there a security group (cloud) blocking it?
# Check AWS/GCP/Azure security group settings

# 7. Check for firewall DROP logs
sudo tail -f /var/log/syslog | grep -i "drop\|reject\|denied"
# or
sudo dmesg | grep -i "nft\|iptables"
```

## 🧠 Summary — Key Takeaways

| Concept | Key Point |
|---------|-----------|
| **Ports** | Numbers (0-65535) identifying services on a machine |
| **Well-Known Ports** | 0-1023 (need root): HTTP=80, HTTPS=443, SSH=22 |
| **Sockets** | IP + Port + Protocol = actual connection endpoint |
| **5-Tuple** | Protocol + Src IP + Src Port + Dst IP + Dst Port = unique connection |
| **0.0.0.0 vs 127.0.0.1** | All interfaces (public) vs localhost only (private) |
| **iptables** | Classic Linux firewall — tables, chains, rules |
| **ufw** | Easy frontend for iptables (Ubuntu) |
| **firewalld** | Zone-based firewall (RHEL/CentOS) |
| **nftables** | Modern iptables replacement — cleaner, faster, unified |
| **DROP vs REJECT** | DROP = silent, REJECT = sends error back |
| **Connection Refused** | Port reachable but nothing listening |
| **Connection Timed Out** | Firewall dropping packets |

```bash
# === PORTS ===
```
