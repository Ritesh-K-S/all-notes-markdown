# 3.7 — Load Balancing — L4 vs L7, Algorithms & Everything You Need to Know

> **Level:** Intermediate
> **Chapter:** 03 — Networking Fundamentals

---

## What is Load Balancing?

Imagine you own a very popular restaurant. On a Friday night, 200 customers walk in. If you have only **one waiter**, that waiter will be overwhelmed, and customers will wait forever. But if you have **5 waiters** and a **host at the door** who sends each customer to the least busy waiter — everyone gets served quickly.

**That host at the door is a Load Balancer.**

In the tech world:

- Your **waiters** = servers (backend servers)
- Your **customers** = client requests (users hitting your website)
- Your **host** = Load Balancer (distributes incoming traffic across multiple servers)

### Formal Definition

> A **Load Balancer** is a device or software that distributes incoming network traffic across multiple backend servers to ensure no single server is overwhelmed, improving **availability**, **reliability**, and **performance**.

---

## Why Do We Need Load Balancing?

| Problem Without Load Balancing | How Load Balancing Solves It |
|---|---|
| Single server handles all traffic → crashes under load | Traffic is spread across multiple servers |
| If one server dies, the whole app goes down | Load balancer detects the dead server and routes traffic to healthy ones |
| Uneven load — one server is overloaded, others are idle | Traffic is distributed evenly using algorithms |
| Scaling is hard — you're stuck with one server | Just add more servers behind the load balancer |
| Maintenance requires downtime | Take one server offline, load balancer routes to the rest |

---

## Where Does a Load Balancer Sit?

```
                    ┌──────────────┐
  Users ──────────► │ Load Balancer │
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              ▼            ▼            ▼
         ┌────────┐   ┌────────┐   ┌────────┐
         │Server 1│   │Server 2│   │Server 3│
         └────────┘   └────────┘   └────────┘
```

The load balancer sits **between the client and the servers**. All requests hit the load balancer first, and it decides which server should handle each request.

---

## Types of Load Balancers

### 1. Hardware Load Balancers

- Physical devices (appliances) placed in data centers
- Examples: **F5 BIG-IP**, **Citrix ADC (NetScaler)**
- Very fast, but **very expensive** ($10,000–$100,000+)
- Used in traditional enterprise data centers

### 2. Software Load Balancers

- Software running on regular servers or VMs
- Examples: **Nginx**, **HAProxy**, **Traefik**, **Envoy**
- Much cheaper and flexible
- The standard in modern DevOps and cloud

### 3. Cloud Load Balancers

- Managed services from cloud providers
- Examples:
  - **AWS:** ALB (Application Load Balancer), NLB (Network Load Balancer), CLB (Classic)
  - **Azure:** Azure Load Balancer, Application Gateway
  - **GCP:** Cloud Load Balancing
- Fully managed — no infrastructure to maintain
- Auto-scales with your traffic

## The Big Concept: L4 vs L7 Load Balancing

This is one of the **most important concepts** in networking and DevOps interviews. Let's break it down.

### Quick Refresher: OSI Model Layers

| Layer | Name | What It Deals With |
|-------|------|-------------------|
| Layer 4 | Transport Layer | TCP/UDP, ports, connections |
| Layer 7 | Application Layer | HTTP, HTTPS, URLs, headers, cookies |

---

### Layer 4 (L4) Load Balancing — Transport Layer

L4 load balancing works at the **Transport Layer**. It makes routing decisions based on:

- **Source IP address**
- **Destination IP address**
- **Source port**
- **Destination port**
- **Protocol** (TCP or UDP)

**It does NOT look inside the packet.** It has no idea whether the request is for `/api/users` or `/images/logo.png`. It just sees "a TCP packet from IP X going to port 80."

#### How L4 Works (Step by Step)

```
1. Client sends a TCP SYN packet to the Load Balancer's IP
2. Load Balancer picks a backend server (say Server 2)
3. Load Balancer forwards the packet to Server 2
4. Server 2 responds — the response goes back through the Load Balancer to the client
5. For the entire TCP connection, all packets go to the same server
```

#### Characteristics of L4

| Feature | Details |
|---------|---------|
| Speed | Very fast — just looks at packet headers |
| Intelligence | Low — can't inspect content |
| Protocols | Works with ANY TCP/UDP protocol (HTTP, FTP, SMTP, databases, etc.) |
| SSL Termination | No — passes encrypted traffic as-is |
| Session Persistence | Based on IP + port only |
| Use Case | High-performance, protocol-agnostic balancing |

#### Example: AWS NLB (Network Load Balancer)

AWS NLB is a classic L4 load balancer. It can handle **millions of requests per second** with ultra-low latency because it operates at the transport layer.

---

### Layer 7 (L7) Load Balancing — Application Layer

L7 load balancing works at the **Application Layer**. It makes routing decisions based on:

- **URL path** (`/api/v1/users` vs `/static/images`)
- **HTTP headers** (`Host: api.example.com` vs `Host: web.example.com`)
- **HTTP method** (GET, POST, PUT, DELETE)
- **Cookies** (session cookies for sticky sessions)
- **Query parameters**
- **Request body** (in some advanced cases)

**It opens the packet, reads the HTTP content, and makes smart decisions.**

#### How L7 Works (Step by Step)

```
1. Client sends an HTTP request: GET /api/users HTTP/1.1
2. Load Balancer receives the full HTTP request
3. Load Balancer inspects the URL path → "/api/users"
4. Based on rules:
   - /api/*    → route to API server pool
   - /static/* → route to CDN or static server pool
   - /admin/*  → route to admin server pool
5. Load Balancer forwards the request to the right server
6. Server processes and responds through the Load Balancer
```

#### Characteristics of L7

| Feature | Details |
|---------|---------|
| Speed | Slower than L4 — needs to parse HTTP content |
| Intelligence | High — content-based routing |
| Protocols | HTTP/HTTPS only (application-level protocols) |
| SSL Termination | Yes — can decrypt HTTPS, inspect, then re-encrypt or forward |
| Session Persistence | Cookie-based, header-based, or URL-based |
| Use Case | Microservices, URL-based routing, A/B testing, canary deployments |

#### Example: AWS ALB (Application Load Balancer)

AWS ALB is a classic L7 load balancer. You can set rules like:

- If path is `/api/*` → send to target group A
- If host is `admin.example.com` → send to target group B
- If header `X-Custom: beta` → send to canary target group

---

### L4 vs L7 — Head-to-Head Comparison

| Feature | L4 (Transport) | L7 (Application) |
|---------|----------------|-------------------|
| OSI Layer | Layer 4 | Layer 7 |
| Decision Based On | IP, Port, Protocol | URL, Headers, Cookies, Body |
| Speed | Very fast | Slower (parses content) |
| Intelligence | Dumb — just forwards packets | Smart — understands content |
| Protocol Support | Any (TCP, UDP, etc.) | HTTP/HTTPS |
| SSL Termination | No | Yes |
| Content-Based Routing | No | Yes |
| WebSocket Support | Yes (passthrough) | Yes (aware) |
| Cost | Generally cheaper | More expensive |
| Example (AWS) | NLB | ALB |
| Example (Software) | HAProxy (TCP mode) | Nginx, HAProxy (HTTP mode), Traefik |

### When to Use What?

| Scenario | Use |
|----------|-----|
| You need raw speed and handle millions of TCP connections | L4 |
| You're load balancing a database (MySQL, PostgreSQL) | L4 |
| You need URL-based routing for microservices | L7 |
| You want SSL termination at the load balancer | L7 |
| You need to do A/B testing or canary deployments | L7 |
| You're routing gRPC traffic | L7 |
| You're load balancing a game server (UDP) | L4 |
| You want to route based on HTTP headers or cookies | L7 |

## Load Balancing Algorithms

Now that we know **where** the load balancer works, let's understand **how** it decides which server to pick. These are called **Load Balancing Algorithms**.

---

### 1. Round Robin

**How it works:** Requests are distributed to servers **in order**, one by one, cycling through the list.

```
Request 1 → Server A
Request 2 → Server B
Request 3 → Server C
Request 4 → Server A  ← back to the start
Request 5 → Server B
...
```

**Pros:**
- Dead simple to implement
- Works great when all servers are identical

**Cons:**
- Doesn't consider server load — if Server A is already handling a heavy request, it still gets the next one
- Assumes all servers have equal capacity

**Best For:** Stateless applications where all servers are identical.

---

### 2. Weighted Round Robin

**How it works:** Same as Round Robin, but servers with **higher weights get more requests**.

```
Server A (weight: 5) → gets 5 requests
Server B (weight: 3) → gets 3 requests
Server C (weight: 2) → gets 2 requests
...then repeat
```

**Use Case:** When you have servers with different capacities — a beefy 32-core server should handle more traffic than a small 4-core one.

#### Nginx Config Example:

```nginx
upstream backend {
    server 192.168.1.10 weight=5;   # Powerful server — gets more traffic
    server 192.168.1.11 weight=3;   # Medium server
    server 192.168.1.12 weight=2;   # Smaller server
}

server {
    listen 80;
    location / {
        proxy_pass http://backend;
    }
}
```

**Explanation:**
- `upstream backend { ... }` — Defines a group of backend servers called "backend"
- `server 192.168.1.10 weight=5;` — This server gets 5 out of every 10 requests
- `proxy_pass http://backend;` — Forward all incoming requests to this group

---

### 3. Least Connections

**How it works:** The load balancer sends the next request to the server with the **fewest active connections**.

```
Server A: 12 active connections
Server B: 5 active connections   ← next request goes here!
Server C: 8 active connections
```

**Pros:**
- Great when requests have **varying processing times** (some requests take 10ms, others take 5 seconds)
- Prevents any server from being overwhelmed

**Cons:**
- Slightly more overhead — load balancer must track active connections

**Best For:** Applications where request processing times vary greatly (e.g., file uploads, complex API calls).

#### Nginx Config Example:

```nginx
upstream backend {
    least_conn;                    # Use least connections algorithm
    server 192.168.1.10;
    server 192.168.1.11;
    server 192.168.1.12;
}
```

---

### 4. Weighted Least Connections

**How it works:** Combines **Least Connections** with **weights**. It considers both the number of active connections AND the server's capacity.

```
Formula: (Active Connections) / Weight = Effective Load

Server A: 10 connections, weight 5 → 10/5 = 2.0
Server B: 6 connections, weight 3  → 6/3 = 2.0
Server C: 3 connections, weight 2  → 3/2 = 1.5 ← next request goes here!
```

**Best For:** Mixed server capacities with varying request processing times.

---

### 5. IP Hash

**How it works:** The load balancer takes the **client's IP address**, runs it through a hash function, and the result determines which server handles the request.

```
hash("192.168.1.100") % 3 = 1 → always goes to Server B
hash("10.0.0.50") % 3 = 0    → always goes to Server A
hash("172.16.0.25") % 3 = 2  → always goes to Server C
```

**Key Property:** The **same client always goes to the same server** (as long as the server pool doesn't change). This is called **session persistence** or **sticky sessions**.

**Pros:**
- No need for a shared session store — user's session lives on one server
- Simple way to achieve stickiness

**Cons:**
- If a server goes down, all its clients get reassigned (sessions lost)
- Can lead to uneven distribution if many clients come from the same IP range (e.g., corporate networks behind NAT)

#### Nginx Config Example:

```nginx
upstream backend {
    ip_hash;                       # Use IP hash algorithm
    server 192.168.1.10;
    server 192.168.1.11;
    server 192.168.1.12;
}
```

---

### 6. Least Response Time

**How it works:** The load balancer sends the request to the server with the **fastest response time AND fewest active connections**.

```
Server A: avg response 20ms, 5 connections
Server B: avg response 10ms, 5 connections  ← picked! (faster response)
Server C: avg response 30ms, 3 connections
```

**Pros:**
- Considers actual server performance, not just connections
- Adapts to real-time conditions

**Cons:**
- More overhead — needs to track response times
- Not available in all load balancers

**Best For:** When servers have different performance characteristics and you want optimal user experience.

---

### 7. Random

**How it works:** Exactly what it sounds like — randomly picks a server for each request.

```
Request 1 → Server C (random)
Request 2 → Server A (random)
Request 3 → Server A (random)
Request 4 → Server B (random)
```

**Pros:**
- Extremely simple, almost no overhead
- Statistically, distributes evenly over time (with large numbers)

**Cons:**
- Can be uneven with small numbers of requests
- Doesn't consider server load at all

**Best For:** Simple setups where perfect distribution isn't critical.

---

### Algorithm Comparison Table

| Algorithm | How It Chooses | Stickiness | Server Capacity Aware | Load Aware | Best For |
|-----------|---------------|------------|----------------------|------------|----------|
| Round Robin | In order, one by one | No | No | No | Equal servers, stateless apps |
| Weighted Round Robin | In order, weighted | No | Yes | No | Unequal servers |
| Least Connections | Fewest connections | No | No | Yes | Variable request times |
| Weighted Least Conn | Fewest conn + weight | No | Yes | Yes | Unequal servers + variable times |
| IP Hash | Hash of client IP | Yes | No | No | Session persistence needed |
| Least Response Time | Fastest response | No | No | Yes | Performance-critical apps |
| Random | Random choice | No | No | No | Simple setups |

## Health Checks

A load balancer isn't useful if it keeps sending traffic to a **dead server**. That's where **health checks** come in.

### What Are Health Checks?

The load balancer periodically sends a "are you alive?" request to each server. If a server doesn't respond (or responds with an error), the load balancer **marks it as unhealthy** and stops sending traffic to it.

### Types of Health Checks

| Type | How It Works | Example |
|------|-------------|---------|
| **TCP Health Check** | Tries to open a TCP connection to the server | Can I connect to port 80? |
| **HTTP Health Check** | Sends an HTTP GET request and checks the status code | GET /health → expects 200 OK |
| **Custom Health Check** | Checks specific application logic | GET /health → checks DB connection, cache, etc. |

### Nginx Health Check Example:

```nginx
upstream backend {
    server 192.168.1.10 max_fails=3 fail_timeout=30s;
    server 192.168.1.11 max_fails=3 fail_timeout=30s;
    server 192.168.1.12 max_fails=3 fail_timeout=30s;
}
```

**Explanation:**
- `max_fails=3` — If a server fails to respond 3 times in a row, mark it as down
- `fail_timeout=30s` — After marking it down, wait 30 seconds before trying again

### HAProxy Health Check Example:

```haproxy
backend web_servers
    option httpchk GET /health       # Send HTTP GET to /health endpoint
    http-check expect status 200     # Expect a 200 OK response

    server web1 192.168.1.10:80 check inter 5s fall 3 rise 2
    server web2 192.168.1.11:80 check inter 5s fall 3 rise 2
    server web3 192.168.1.12:80 check inter 5s fall 3 rise 2
```

**Explanation:**
- `option httpchk GET /health` — Use HTTP health checks on the `/health` path
- `check` — Enable health checking for this server
- `inter 5s` — Check every 5 seconds
- `fall 3` — Mark as down after 3 consecutive failures
- `rise 2` — Mark as up after 2 consecutive successes

## SSL/TLS Termination

### What Is It?

When your load balancer handles HTTPS traffic, it can **decrypt the SSL/TLS** at the load balancer level. This is called **SSL Termination**.

```
                           SSL Terminated Here
                                  ↓
Client ──── HTTPS (encrypted) ──► Load Balancer ──── HTTP (plain) ──► Backend Servers
```

### Why Do This?

1. **Offloads CPU work** from backend servers (encryption/decryption is CPU-intensive)
2. **Centralized certificate management** — only the load balancer needs the SSL cert
3. **Easier debugging** — traffic between LB and servers is plain HTTP
4. **Better performance** — backend servers can focus on business logic

### SSL Passthrough vs SSL Termination

| Feature | SSL Termination | SSL Passthrough |
|---------|----------------|-----------------|
| Where SSL ends | At the Load Balancer | At the Backend Server |
| LB can inspect traffic? | Yes (it's decrypted) | No (encrypted throughout) |
| Backend handles SSL? | No | Yes |
| Security | Less secure between LB and backend | End-to-end encryption |
| Use Case | Most web applications | When end-to-end encryption is required (compliance, PCI-DSS) |

## Session Persistence (Sticky Sessions)

### The Problem

Imagine a user logs into your website. Their session (login state, cart items, etc.) is stored on Server A. On the next request, the load balancer sends them to Server B. Server B has no idea who this user is — they're logged out!

### The Solution: Sticky Sessions

Sticky sessions ensure that **a user's requests always go to the same server**.

### Methods of Sticky Sessions

| Method | How It Works |
|--------|-------------|
| **Cookie-based** | Load balancer sets a cookie (e.g., `SERVERID=server1`) — all future requests with this cookie go to server1 |
| **IP-based** | Use IP hash algorithm (discussed above) |
| **URL-based** | Encode server info in the URL (less common) |

### Nginx Sticky Session Example:

```nginx
upstream backend {
    ip_hash;    # Sticky sessions using IP hash
    server 192.168.1.10;
    server 192.168.1.11;
    server 192.168.1.12;
}
```

### Better Approach: Don't Use Sticky Sessions

In modern architectures, the **better approach** is to make your application **stateless**:

- Store sessions in a **shared session store** (Redis, Memcached, database)
- Use **JWT tokens** (stateless authentication)
- This way, ANY server can handle ANY request — no stickiness needed

## Real-World Load Balancer Tools

### 1. Nginx (Most Popular)

```nginx
# Basic Nginx Load Balancer Configuration
http {
    upstream myapp {
        least_conn;
        server 10.0.0.1:8080;
        server 10.0.0.2:8080;
        server 10.0.0.3:8080;
    }

    server {
        listen 80;
        server_name myapp.example.com;

        location / {
            proxy_pass http://myapp;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
        }
    }
}
```

**Line-by-line explanation:**
- `upstream myapp { ... }` — Defines a pool of 3 backend servers
- `least_conn;` — Use Least Connections algorithm
- `listen 80;` — Listen for HTTP traffic on port 80
- `proxy_pass http://myapp;` — Forward requests to the "myapp" upstream group
- `proxy_set_header Host $host;` — Pass the original Host header to the backend
- `proxy_set_header X-Real-IP $remote_addr;` — Pass the client's real IP to the backend
- `proxy_set_header X-Forwarded-For ...` — Standard header for tracking client IPs through proxies
- `proxy_set_header X-Forwarded-Proto $scheme;` — Tell the backend whether the original request was HTTP or HTTPS

### 2. HAProxy (High Performance)

```haproxy
# /etc/haproxy/haproxy.cfg

global
    log /dev/log local0
    maxconn 4096

defaults
    mode http
    timeout connect 5s
    timeout client 30s
    timeout server 30s
    option httplog

frontend http_front
    bind *:80
    default_backend web_servers

backend web_servers
    balance roundrobin
    option httpchk GET /health
    server web1 10.0.0.1:8080 check
    server web2 10.0.0.2:8080 check
    server web3 10.0.0.3:8080 check
```

**Explanation:**
- `global` — Global settings (logging, max connections)
- `defaults` — Default settings for all frontends/backends
- `frontend http_front` — Defines where HAProxy listens for incoming connections (port 80)
- `backend web_servers` — Defines the pool of servers and routing rules
- `balance roundrobin` — Use Round Robin algorithm
- `check` — Enable health checks on each server

### 3. Traefik (Cloud-Native, Auto-Discovery)

Traefik is popular in **Docker and Kubernetes** environments because it can automatically discover services.

```yaml
# docker-compose.yml with Traefik
version: '3'
services:
  traefik:
    image: traefik:v2.10
    command:
      - "--providers.docker=true"
      - "--entrypoints.web.address=:80"
    ports:
      - "80:80"
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock

  myapp:
    image: myapp:latest
    labels:
      - "traefik.http.routers.myapp.rule=Host(`myapp.example.com`)"
    deploy:
      replicas: 3    # 3 instances — Traefik auto-balances across them
```

## DNS Load Balancing

A simpler (but less smart) form of load balancing done at the **DNS level**.

```
myapp.example.com  →  A record →  10.0.0.1
                   →  A record →  10.0.0.2
                   →  A record →  10.0.0.3
```

When someone resolves `myapp.example.com`, DNS returns one of these IPs (rotating). This is how services like **Route 53** do basic load balancing.

**Limitations:**
- No health checks (DNS doesn't know if a server is down)
- DNS caching means changes are slow to propagate
- No content-based routing
- Limited algorithms (basically just Round Robin)

---

## Global Server Load Balancing (GSLB)

For applications running across **multiple geographic regions**:

```
User in India    → Routed to Mumbai data center
User in USA      → Routed to Virginia data center
User in Europe   → Routed to Frankfurt data center
```

This is called **Global Server Load Balancing**. Tools:
- AWS Route 53 (Geolocation/Latency-based routing)
- Cloudflare Load Balancing
- Azure Traffic Manager

## Quick Summary

| Concept | Key Takeaway |
|---------|-------------|
| Load Balancer | Distributes traffic across multiple servers |
| L4 | Fast, dumb — works with IP/port (TCP/UDP level) |
| L7 | Smart — works with HTTP content (URLs, headers, cookies) |
| Round Robin | Simple rotation through servers |
| Least Connections | Routes to server with fewest active connections |
| IP Hash | Same client always goes to same server |
| Weighted | Gives more traffic to more powerful servers |
| Health Checks | Automatically detects and avoids dead servers |
| SSL Termination | Decrypt HTTPS at the load balancer |
| Sticky Sessions | Keep a user on the same server (prefer stateless design instead) |

## Key Interview Questions

1. **What's the difference between L4 and L7 load balancing?**
   → L4 works at transport layer (IP/port), L7 works at application layer (HTTP content)

2. **When would you choose L4 over L7?**
   → When you need raw speed, protocol-agnostic balancing (databases, game servers)

3. **What's the best algorithm for servers with different capacities?**
   → Weighted Round Robin or Weighted Least Connections

4. **How does a load balancer know a server is down?**
   → Health checks (TCP, HTTP, or custom)

5. **What's SSL termination?**
   → Decrypting HTTPS at the load balancer and forwarding plain HTTP to backends

6. **How do you handle sessions with a load balancer?**
   → Sticky sessions OR (better) make the app stateless with shared session store/JWT
