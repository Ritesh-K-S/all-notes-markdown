# 3.8 — Proxy vs Reverse Proxy vs Forward Proxy

> **Level:** Intermediate
> **Chapter:** 03 — Networking Fundamentals

---

## The Word "Proxy" — What Does It Even Mean?

Before diving into forward proxy, reverse proxy, or any of that — let's understand the word **proxy** itself.

> **Proxy = Someone or something that acts on behalf of someone else.**

Real-life example: You can't attend a meeting, so you send your colleague to represent you. Your colleague is your **proxy**.

In networking:

> A **Proxy Server** is a server that sits **between a client and a destination server**, acting as an intermediary. It receives requests from one side, forwards them to the other, and passes back the response.

---

## The Two Types: Forward Proxy vs Reverse Proxy

Here's the simplest way to remember them:

| Type | Who does it help? | Where does it sit? |
|------|-------------------|-------------------|
| **Forward Proxy** | Helps the **client** (user) | In front of clients |
| **Reverse Proxy** | Helps the **server** | In front of servers |

That's it. The **direction** of who's being helped is the key difference.

---

## Forward Proxy — In Detail

### What Is a Forward Proxy?

A Forward Proxy sits **between the client and the internet**. The client sends its request to the proxy, and the proxy forwards it to the destination server on behalf of the client.

```
                    Forward Proxy
                         │
Client ───request──►  [Proxy]  ───request──►  Internet / Destination Server
                         │
Client ◄──response───  [Proxy]  ◄──response───  Internet / Destination Server
```

The **destination server** sees the proxy's IP address, NOT the client's IP address. The server has no idea who the actual client is.

### Real-Life Analogy

Imagine you want to buy a car, but you don't want the dealer to know who you are (maybe to negotiate better). So you send your friend to buy it for you. The dealer only sees your friend — not you.

**Your friend = Forward Proxy**

### Why Use a Forward Proxy?

| Use Case | Explanation |
|----------|-------------|
| **Privacy / Anonymity** | Hide the client's real IP address from the server |
| **Content Filtering** | Companies use proxies to block employees from accessing social media, gambling sites, etc. |
| **Bypass Geo-Restrictions** | Access content blocked in your country by routing through a proxy in another country |
| **Caching** | Cache frequently accessed content to reduce bandwidth (e.g., all 500 employees accessing google.com goes through one cached copy) |
| **Logging & Monitoring** | Companies can monitor what websites employees are visiting |
| **Security** | Scan outgoing traffic for malware or data leaks |

### Examples of Forward Proxies

| Tool | Description |
|------|-------------|
| **Squid** | One of the oldest and most popular forward proxy/caching servers |
| **Privoxy** | Focused on privacy and ad-blocking |
| **Charles Proxy** | Popular among developers for debugging HTTP traffic |
| **Corporate Proxies** | Many companies set up internal proxy servers |
| **VPN** | Technically acts like a forward proxy (more on this in 3.9) |

### Squid Forward Proxy — Basic Setup

```bash
# Install Squid on Ubuntu
sudo apt update
sudo apt install squid -y

# Main config file
sudo nano /etc/squid/squid.conf
```

```squid
# /etc/squid/squid.conf — Basic Forward Proxy

# Define the network that's allowed to use this proxy
acl localnet src 192.168.1.0/24          # Only allow local network

# Define blocked sites
acl blocked_sites dstdomain .facebook.com .instagram.com .tiktok.com

# Rules
http_access deny blocked_sites           # Block social media
http_access allow localnet               # Allow local network
http_access deny all                     # Deny everyone else

# Listen on port 3128 (default Squid port)
http_port 3128

# Enable caching
cache_dir ufs /var/spool/squid 100 16 256
```

**Explanation:**
- `acl localnet src 192.168.1.0/24` — Defines an access control list (ACL) for IPs in the 192.168.1.x range
- `acl blocked_sites dstdomain ...` — Defines a list of destination domains to block
- `http_access deny blocked_sites` — Blocks requests to those domains
- `http_access allow localnet` — Allows requests from the local network
- `http_access deny all` — Denies everything else (security best practice)
- `http_port 3128` — Squid listens on port 3128
- `cache_dir ...` — Sets up a 100MB disk cache

#### Using the Proxy from a Client

```bash
# Set proxy for curl
curl -x http://proxy-server:3128 http://example.com

# Set proxy as environment variable (Linux)
export http_proxy="http://proxy-server:3128"
export https_proxy="http://proxy-server:3128"

# Now all HTTP/HTTPS commands will use this proxy
curl http://example.com    # Goes through the proxy automatically
wget http://example.com    # Also goes through the proxy
```

## Reverse Proxy — In Detail

### What Is a Reverse Proxy?

A Reverse Proxy sits **between the internet and your backend servers**. The client sends its request to the reverse proxy (thinking it's the actual server), and the reverse proxy forwards it to one of the backend servers.

```
                              Reverse Proxy
                                   │
Internet / Client ───request──►  [Proxy]  ───request──►  Backend Server 1
                                   │                      Backend Server 2
Internet / Client ◄──response──  [Proxy]  ◄──response──  Backend Server 3
```

The **client** sees only the reverse proxy's IP address. The client has no idea how many backend servers exist or what their IPs are.

### Real-Life Analogy

You walk into a restaurant and talk to the **receptionist** (reverse proxy). You place your order. The receptionist takes it to the **kitchen** (backend servers). You never see the kitchen, you don't know how many chefs are working, or which chef prepared your food. You just get your meal.

**Receptionist = Reverse Proxy**

### Why Use a Reverse Proxy?

| Use Case | Explanation |
|----------|-------------|
| **Load Balancing** | Distribute traffic across multiple backend servers (discussed in 3.7) |
| **SSL Termination** | Handle HTTPS encryption/decryption — backends get plain HTTP |
| **Security** | Hide backend server IPs and architecture from the public |
| **Caching** | Cache static content (images, CSS, JS) to reduce backend load |
| **Compression** | Compress responses (gzip/brotli) before sending to clients |
| **DDoS Protection** | Absorb and filter malicious traffic before it hits your servers |
| **URL Rewriting** | Change URLs before forwarding to backends |
| **Rate Limiting** | Limit how many requests a client can make per second |
| **Authentication** | Handle authentication before passing requests to backends |
| **A/B Testing** | Route a percentage of traffic to different backend versions |

### Examples of Reverse Proxies

| Tool | Description |
|------|-------------|
| **Nginx** | The most popular reverse proxy in the world |
| **HAProxy** | High-performance reverse proxy and load balancer |
| **Traefik** | Cloud-native reverse proxy, popular with Docker/Kubernetes |
| **Envoy** | Modern proxy built for microservices (used by Istio service mesh) |
| **Caddy** | Auto-HTTPS reverse proxy — gets SSL certs automatically |
| **Apache httpd** | Classic web server that can also act as a reverse proxy |
| **AWS ALB/ELB** | Cloud-managed reverse proxies |
| **Cloudflare** | Acts as a global reverse proxy (CDN + security) |

### Nginx Reverse Proxy — Basic Setup

```nginx
# /etc/nginx/conf.d/myapp.conf

# Define backend servers
upstream backend_servers {
    server 10.0.0.1:8080;
    server 10.0.0.2:8080;
    server 10.0.0.3:8080;
}

server {
    listen 80;
    server_name myapp.example.com;

    # Reverse proxy — forward all requests to backend servers
    location / {
        proxy_pass http://backend_servers;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # Cache static files
    location /static/ {
        proxy_pass http://backend_servers;
        proxy_cache_valid 200 1d;    # Cache 200 responses for 1 day
    }
}
```

**Line-by-line explanation:**
- `upstream backend_servers { ... }` — Defines a pool of 3 backend servers
- `listen 80;` — Reverse proxy listens on port 80
- `server_name myapp.example.com;` — Only handle requests for this domain
- `proxy_pass http://backend_servers;` — Forward requests to the backend pool
- `proxy_set_header Host $host;` — Pass the original Host header to backends
- `proxy_set_header X-Real-IP $remote_addr;` — Pass the client's real IP
- `proxy_set_header X-Forwarded-For ...` — Chain of all proxy IPs the request has passed through
- `proxy_cache_valid 200 1d;` — Cache successful responses for 1 day

### Nginx Reverse Proxy with SSL Termination

```nginx
server {
    listen 443 ssl;
    server_name myapp.example.com;

    # SSL Certificate
    ssl_certificate /etc/ssl/certs/myapp.crt;
    ssl_certificate_key /etc/ssl/private/myapp.key;

    # Strong SSL settings
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;

    location / {
        proxy_pass http://backend_servers;    # Forward as plain HTTP
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-Proto https;
    }
}

# Redirect HTTP to HTTPS
server {
    listen 80;
    server_name myapp.example.com;
    return 301 https://$host$request_uri;
}
```

**What happens here:**
1. Client connects via **HTTPS** (port 443) — encrypted
2. Nginx **terminates SSL** — decrypts the traffic
3. Nginx forwards the request to backend servers via **plain HTTP** — unencrypted (but within private network)
4. Backend server processes and responds
5. Nginx encrypts the response and sends it back to the client

The `return 301` block redirects any HTTP request to HTTPS automatically.

### Caddy — Automatic Reverse Proxy (The Simple Way)

```
# Caddyfile — that's it. This is the entire config.
myapp.example.com {
    reverse_proxy 10.0.0.1:8080 10.0.0.2:8080 10.0.0.3:8080
}
```

Caddy automatically:
- Gets SSL certificates from Let's Encrypt
- Enables HTTPS
- Redirects HTTP to HTTPS
- Load balances across the 3 servers

**Yes, that 3-line config does all of that.** That's why Caddy is popular for simple setups.

## Forward Proxy vs Reverse Proxy — The Complete Comparison

| Feature | Forward Proxy | Reverse Proxy |
|---------|--------------|---------------|
| **Who it helps** | Client (user) | Server (backend) |
| **Where it sits** | Between client and internet | Between internet and servers |
| **Who knows about it?** | Client knows (configured to use it) | Client doesn't know (thinks proxy IS the server) |
| **Hides what?** | Hides client's identity from the server | Hides server's identity from the client |
| **Server sees** | Proxy's IP (not client's IP) | Client's IP (passed via headers) |
| **Client sees** | Destination server's response | Proxy's IP (not backend server's IP) |
| **Main use cases** | Privacy, content filtering, caching | Load balancing, SSL, security, caching |
| **Example tools** | Squid, Privoxy, corporate proxies | Nginx, HAProxy, Traefik, Cloudflare |
| **Configuration** | Client must be configured to use it | Client needs no configuration |
| **Direction** | Client → Proxy → Internet | Internet → Proxy → Backend Servers |

### Visual Comparison

```
FORWARD PROXY:
==============
                    ┌──────────────┐
  Client A ─────►   │              │
  Client B ─────►   │ Forward Proxy│ ─────►  Internet (any server)
  Client C ─────►   │              │
                    └──────────────┘
  Protects CLIENTS                    Servers don't know who clients are


REVERSE PROXY:
==============
                    ┌──────────────┐
                    │              │  ─────►  Server 1
  Internet ─────►   │ Reverse Proxy│  ─────►  Server 2
  (any client)      │              │  ─────►  Server 3
                    └──────────────┘
                    Protects SERVERS    Clients don't know which server responded
```

## Open Proxy vs Transparent Proxy

### Open Proxy

An **open proxy** is a forward proxy that's available to **anyone on the internet** — no authentication required.

- Free proxy lists on the internet are open proxies
- **Dangerous!** The proxy owner can see ALL your traffic
- Often used for malicious purposes
- Never use open proxies for sensitive activities

### Transparent Proxy

A **transparent proxy** intercepts traffic **without the client knowing**. The client doesn't need to configure anything — the proxy is invisible.

```
Client ──────►  [Transparent Proxy] ──────►  Internet
                (client doesn't know)
```

**Use Cases:**
- ISPs use them to cache content and reduce bandwidth
- Companies use them to monitor/filter traffic without configuring every device
- Hotels/airports use them for their captive portals (the login page before you can use WiFi)

**How it works:** The transparent proxy sits on the network path and intercepts all traffic on port 80/443 using network-level redirection (iptables, router config).

## Reverse Proxy Patterns in Modern Architecture

### 1. API Gateway Pattern

An API Gateway is essentially a specialized reverse proxy for microservices:

```
                    ┌──────────────┐
                    │              │  ──►  User Service (/api/users)
Client ─────────►   │  API Gateway │  ──►  Order Service (/api/orders)
                    │ (Rev. Proxy) │  ──►  Product Service (/api/products)
                    │              │  ──►  Auth Service (/api/auth)
                    └──────────────┘
```

The API Gateway handles:
- Routing requests to the correct microservice
- Authentication & authorization
- Rate limiting
- Request/response transformation
- Logging & monitoring

Popular API Gateways: **Kong**, **AWS API Gateway**, **Apigee**, **Traefik**, **Nginx Plus**

### 2. Sidecar Proxy Pattern (Service Mesh)

In Kubernetes service meshes, each service gets its own **sidecar proxy**:

```
┌─────────────────────────┐
│  Pod                    │
│  ┌──────────┐  ┌─────┐ │
│  │  Your App │◄─│Envoy│ │ ◄── Sidecar Proxy
│  └──────────┘  └─────┘ │
└─────────────────────────┘
         ▲
         │ All traffic goes through the sidecar
         ▼
┌─────────────────────────┐
│  Pod                    │
│  ┌──────────┐  ┌─────┐ │
│  │ Other App │◄─│Envoy│ │ ◄── Sidecar Proxy
│  └──────────┘  └─────┘ │
└─────────────────────────┘
```

The sidecar proxy handles: mTLS encryption, load balancing, retries, circuit breaking, observability — all without changing your app code.

### 3. Edge Proxy Pattern (CDN)

Services like **Cloudflare** act as a reverse proxy at the **edge** (closest to the user):

```
User in India ──► Cloudflare Edge (Mumbai) ──► Your Origin Server (US)
                  ↓
                  Cached content served from Mumbai
                  (no need to go to US)
```

## Practical Example: Setting Up Nginx as Both Forward and Reverse Proxy

### Nginx as a Forward Proxy

```nginx
# Forward Proxy (less common for Nginx, but possible)
server {
    listen 8888;

    # Allow CONNECT method for HTTPS
    proxy_connect;
    proxy_connect_allow 443 563;

    location / {
        resolver 8.8.8.8;                       # Use Google DNS
        proxy_pass http://$http_host$request_uri; # Forward to requested URL
    }
}
```

### Nginx as a Reverse Proxy (The Common Use Case)

```nginx
# Reverse Proxy with multiple features
upstream api_servers {
    least_conn;
    server 10.0.0.1:3000;
    server 10.0.0.2:3000;
}

upstream static_servers {
    server 10.0.0.5:80;
    server 10.0.0.6:80;
}

server {
    listen 443 ssl;
    server_name myapp.example.com;

    ssl_certificate /etc/ssl/certs/myapp.crt;
    ssl_certificate_key /etc/ssl/private/myapp.key;

    # API requests → API server pool
    location /api/ {
        proxy_pass http://api_servers;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;

        # Rate limiting
        limit_req zone=api burst=20 nodelay;
    }

    # Static files → Static server pool
    location /static/ {
        proxy_pass http://static_servers;
        expires 30d;                      # Browser cache for 30 days
        add_header Cache-Control "public";
    }

    # WebSocket support
    location /ws/ {
        proxy_pass http://api_servers;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
}
```

**What this does:**
- `/api/*` requests → go to API servers (with rate limiting)
- `/static/*` requests → go to static file servers (with caching)
- `/ws/*` requests → go to API servers with WebSocket upgrade support
- All traffic is SSL terminated at Nginx

## Quick Summary

| Concept | Key Takeaway |
|---------|-------------|
| **Proxy** | An intermediary between two parties |
| **Forward Proxy** | Sits in front of **clients** — hides clients from servers |
| **Reverse Proxy** | Sits in front of **servers** — hides servers from clients |
| **Forward Proxy Use** | Privacy, content filtering, bypassing restrictions |
| **Reverse Proxy Use** | Load balancing, SSL, security, caching |
| **Transparent Proxy** | Intercepts traffic without client's knowledge |
| **API Gateway** | Specialized reverse proxy for microservices |
| **Sidecar Proxy** | Per-service proxy in service mesh (Envoy, Istio) |

## Key Interview Questions

1. **What's the difference between a forward proxy and a reverse proxy?**
   → Forward proxy helps clients (hides client identity), reverse proxy helps servers (hides server identity)

2. **Give a real-world example of a forward proxy.**
   → Corporate proxy that blocks social media sites for employees

3. **Give a real-world example of a reverse proxy.**
   → Nginx sitting in front of 10 web servers, distributing traffic and handling SSL

4. **Is a load balancer a reverse proxy?**
   → Yes! A load balancer is a type of reverse proxy that specifically focuses on distributing traffic

5. **What's a transparent proxy?**
   → A proxy that intercepts traffic without the client's knowledge or configuration

6. **Why would you put a reverse proxy in front of your application?**
   → Security (hide backend IPs), SSL termination, load balancing, caching, rate limiting, compression

7. **What is an API Gateway?**
   → A specialized reverse proxy for microservices that handles routing, auth, rate limiting, and more
