## Why This Topic Matters

Every time you deploy an application to the cloud (AWS, Azure, GCP), you're working inside a **virtual network**. Understanding VPCs, subnets, NAT, and gateways is **non-negotiable** for any DevOps engineer. Without this knowledge, you can't:

- Secure your infrastructure
- Connect services to the internet
- Set up private networks for databases
- Troubleshoot connectivity issues
- Design production-ready architectures

Let's break each concept down one by one.

## VPN — Virtual Private Network

### What Is a VPN?

Imagine you're sitting in a coffee shop using public WiFi. Anyone on that network could potentially snoop on your traffic. Now imagine you could create a **secret, encrypted tunnel** from your laptop to a trusted server far away — and ALL your internet traffic goes through that tunnel. Nobody in the coffee shop can see what you're doing.

**That encrypted tunnel is a VPN.**

> A **VPN (Virtual Private Network)** creates a secure, encrypted connection over a public network (like the internet), making it appear as if you're directly connected to a private network.

### How Does a VPN Work?

```
Without VPN:
  Your Laptop ──── (open internet, anyone can see) ────► Website

With VPN:
  Your Laptop ════ [Encrypted Tunnel] ══════► VPN Server ────► Website
                   Nobody can see                 ↑
                   what's inside               Your traffic
                                              appears to come
                                              from VPN server's IP
```

**Step by step:**
1. You connect to a VPN server (using a VPN client)
2. A secure, encrypted tunnel is created between you and the VPN server
3. ALL your internet traffic goes through this tunnel
4. The VPN server decrypts your traffic and sends it to the destination
5. Responses come back to the VPN server, get encrypted, and sent back to you through the tunnel

### VPN Use Cases

| Use Case | Explanation |
|----------|-------------|
| **Remote Work** | Employees connect to the company's internal network from home |
| **Privacy** | Hide your browsing activity from ISPs and public WiFi snoopers |
| **Bypass Geo-Restrictions** | Access content blocked in your country |
| **Site-to-Site Connectivity** | Connect two office networks securely over the internet |
| **Cloud Access** | Securely connect your on-premises data center to cloud VPCs |

### Types of VPNs

#### 1. Remote Access VPN (Client-to-Site)

A single user connects to a remote network.

```
Employee's Laptop ════ [VPN Tunnel] ════► Company Network
  (at home)                                (office servers, databases)
```

**Tools:** OpenVPN, WireGuard, Cisco AnyConnect, FortiClient

#### 2. Site-to-Site VPN

Two entire networks are connected to each other over the internet.

```
Office A Network ════ [VPN Tunnel] ════► Office B Network
(192.168.1.0/24)                         (10.0.0.0/24)
  All devices in                          All devices in
  Office A can talk                       Office B can talk
  to Office B                             to Office A
```

**Tools:** IPsec, AWS VPN Gateway, Azure VPN Gateway

#### 3. Cloud VPN

Connects your on-premises network to a cloud VPC.

```
Your Data Center ════ [VPN Tunnel] ════► AWS VPC
(on-premises)                            (cloud)
```

**Tools:** AWS Site-to-Site VPN, Azure VPN Gateway, GCP Cloud VPN

### VPN Protocols

| Protocol | Speed | Security | Notes |
|----------|-------|----------|-------|
| **WireGuard** | Very Fast | Very Strong | Modern, simple, recommended |
| **OpenVPN** | Good | Very Strong | Open source, battle-tested, widely used |
| **IPsec/IKEv2** | Fast | Strong | Built into most OSes, great for mobile |
| **L2TP/IPsec** | Moderate | Strong | Older, being phased out |
| **PPTP** | Fast | Weak | Old and insecure — **never use this** |

### Setting Up a Basic WireGuard VPN

#### Server Side (Ubuntu)

```bash
# Install WireGuard
sudo apt update
sudo apt install wireguard -y

# Generate server key pair
wg genkey | tee server_private.key | wg pubkey > server_public.key

# View keys
cat server_private.key
cat server_public.key
```

```ini
# /etc/wireguard/wg0.conf — Server Configuration

[Interface]
Address = 10.10.0.1/24               # VPN server's IP in the tunnel network
ListenPort = 51820                    # Port WireGuard listens on
PrivateKey = <SERVER_PRIVATE_KEY>     # Server's private key

# Enable IP forwarding and NAT
PostUp = iptables -A FORWARD -i wg0 -j ACCEPT; iptables -t nat -A POSTROUTING -o eth0 -j MASQUERADE
PostDown = iptables -D FORWARD -i wg0 -j ACCEPT; iptables -t nat -D POSTROUTING -o eth0 -j MASQUERADE

[Peer]
PublicKey = <CLIENT_PUBLIC_KEY>       # Client's public key
AllowedIPs = 10.10.0.2/32            # Client's IP in the tunnel
```

**Explanation:**
- `Address = 10.10.0.1/24` — The server gets IP 10.10.0.1 inside the VPN tunnel
- `ListenPort = 51820` — WireGuard uses UDP port 51820 by default
- `PrivateKey` — The server's private key (keep this secret!)
- `PostUp` — Commands to run when the tunnel starts (enables forwarding and NAT)
- `PostDown` — Commands to run when the tunnel stops (cleans up rules)
- `[Peer]` — Defines a client that's allowed to connect
- `AllowedIPs = 10.10.0.2/32` — This client gets IP 10.10.0.2 in the tunnel

```bash
# Start WireGuard
sudo wg-quick up wg0

# Enable on boot
sudo systemctl enable wg-quick@wg0

# Check status
sudo wg show
```

#### Client Side

```bash
# Install WireGuard
sudo apt install wireguard -y

# Generate client key pair
wg genkey | tee client_private.key | wg pubkey > client_public.key
```

```ini
# /etc/wireguard/wg0.conf — Client Configuration

[Interface]
Address = 10.10.0.2/24               # Client's IP in the tunnel
PrivateKey = <CLIENT_PRIVATE_KEY>     # Client's private key
DNS = 1.1.1.1                        # Use Cloudflare DNS

[Peer]
PublicKey = <SERVER_PUBLIC_KEY>       # Server's public key
Endpoint = 203.0.113.1:51820         # Server's public IP and port
AllowedIPs = 0.0.0.0/0               # Route ALL traffic through VPN
PersistentKeepalive = 25             # Keep connection alive every 25 seconds
```

**Explanation:**
- `AllowedIPs = 0.0.0.0/0` — Route **all** traffic through the VPN (full tunnel)
- `Endpoint` — The server's public IP address and port
- `PersistentKeepalive = 25` — Send a packet every 25 seconds to keep the connection alive (important for NAT traversal)

```bash
# Connect to VPN
sudo wg-quick up wg0

# Check your public IP (should show VPN server's IP)
curl ifconfig.me

# Disconnect
sudo wg-quick down wg0
```

## VPC — Virtual Private Cloud

### What Is a VPC?

When you sign up for AWS (or any cloud provider), you don't just throw your servers onto the internet randomly. You first create a **Virtual Private Cloud (VPC)** — your own isolated, private network in the cloud.

> A **VPC** is your own logically isolated section of the cloud where you can launch resources (EC2 instances, databases, etc.) in a virtual network that **you** define and control.

### Real-Life Analogy

Think of a cloud provider (like AWS) as a massive apartment complex with thousands of apartments. When you rent an apartment, you get your own private space — with your own rooms, walls, doors, and locks. You can't see or access other tenants' apartments, and they can't access yours.

**Your apartment = Your VPC**

### Key Properties of a VPC

| Property | Description |
|----------|-------------|
| **Isolation** | Your VPC is completely isolated from other customers' VPCs |
| **IP Range** | You choose the IP address range (CIDR block) for your VPC |
| **Subnets** | You divide your VPC into smaller networks (subnets) |
| **Route Tables** | You control how traffic flows within and outside the VPC |
| **Security** | You define who can access what using Security Groups and NACLs |
| **Internet Access** | You decide which resources can access the internet |

### VPC Architecture Overview

```
┌─────────────────────────────── AWS Cloud ───────────────────────────────┐
│                                                                        │
│  ┌──────────────────────── Your VPC (10.0.0.0/16) ──────────────────┐ │
│  │                                                                    │ │
│  │  ┌─── Availability Zone A ───┐  ┌─── Availability Zone B ───┐    │ │
│  │  │                           │  │                           │    │ │
│  │  │ ┌── Public Subnet ──────┐ │  │ ┌── Public Subnet ──────┐│    │ │
│  │  │ │  10.0.1.0/24          │ │  │ │  10.0.3.0/24          ││    │ │
│  │  │ │  • Web Server         │ │  │ │  • Web Server         ││    │ │
│  │  │ │  • NAT Gateway        │ │  │ │                       ││    │ │
│  │  │ └───────────────────────┘ │  │ └───────────────────────┘│    │ │
│  │  │                           │  │                           │    │ │
│  │  │ ┌── Private Subnet ─────┐ │  │ ┌── Private Subnet ─────┐│    │ │
│  │  │ │  10.0.2.0/24          │ │  │ │  10.0.4.0/24          ││    │ │
│  │  │ │  • App Server         │ │  │ │  • App Server         ││    │ │
│  │  │ │  • Database           │ │  │ │  • Database           ││    │ │
│  │  │ └───────────────────────┘ │  │ └───────────────────────┘│    │ │
│  │  │                           │  │                           │    │ │
│  │  └───────────────────────────┘  └───────────────────────────┘    │ │
│  │                                                                    │ │
│  └────────────────────────────────────────────────────────────────────┘ │
│                                                                        │
└────────────────────────────────────────────────────────────────────────┘
```

### Creating a VPC (AWS CLI)

```bash
# Create a VPC with CIDR block 10.0.0.0/16 (65,536 IP addresses)
aws ec2 create-vpc \
    --cidr-block 10.0.0.0/16 \
    --tag-specifications 'ResourceType=vpc,Tags=[{Key=Name,Value=my-production-vpc}]'
```

**Explanation:**
- `--cidr-block 10.0.0.0/16` — This VPC will use IP addresses from 10.0.0.0 to 10.0.255.255 (65,536 IPs)
- `--tag-specifications` — Gives the VPC a human-readable name "my-production-vpc"

### VPC CIDR Block — How Many IPs?

| CIDR | Number of IPs | Common Use |
|------|--------------|------------|
| /16 | 65,536 | Large VPC (most common for production) |
| /20 | 4,096 | Medium VPC |
| /24 | 256 | Small VPC or individual subnet |
| /28 | 16 | Very small (minimum allowed in AWS) |

**AWS reserves 5 IPs in each subnet:**
- First IP (10.0.1.0) — Network address
- Second IP (10.0.1.1) — VPC router
- Third IP (10.0.1.2) — DNS server
- Fourth IP (10.0.1.3) — Reserved for future use
- Last IP (10.0.1.255) — Broadcast address

So a /24 subnet (256 IPs) actually gives you **251 usable IPs**.

## Subnets — Dividing Your VPC

### What Is a Subnet?

A **Subnet** (sub-network) is a smaller network **inside** your VPC. It's like dividing your apartment into rooms — each room has a purpose.

> A subnet is a range of IP addresses within your VPC. You launch resources (like EC2 instances) into specific subnets.

### Why Subnets?

1. **Organization** — Group related resources together
2. **Security** — Apply different security rules to different subnets
3. **Availability** — Spread subnets across multiple Availability Zones for redundancy
4. **Access Control** — Public subnets for web servers, private subnets for databases

### Public Subnet vs Private Subnet

This is **super important**:

| Feature | Public Subnet | Private Subnet |
|---------|--------------|----------------|
| **Internet Access** | Yes — has a route to the Internet Gateway | No direct internet access |
| **Public IPs** | Resources can have public IPs | Resources only have private IPs |
| **What goes here** | Web servers, load balancers, NAT Gateways | Databases, app servers, internal services |
| **Security** | More exposed — needs strict security groups | More secure — hidden from the internet |
| **Route Table** | Routes 0.0.0.0/0 → Internet Gateway | Routes 0.0.0.0/0 → NAT Gateway (if any) |

### Creating Subnets (AWS CLI)

```bash
# Create a PUBLIC subnet in Availability Zone A
aws ec2 create-subnet \
    --vpc-id vpc-0abc123def456 \
    --cidr-block 10.0.1.0/24 \
    --availability-zone us-east-1a \
    --tag-specifications 'ResourceType=subnet,Tags=[{Key=Name,Value=public-subnet-a}]'

# Create a PRIVATE subnet in Availability Zone A
aws ec2 create-subnet \
    --vpc-id vpc-0abc123def456 \
    --cidr-block 10.0.2.0/24 \
    --availability-zone us-east-1a \
    --tag-specifications 'ResourceType=subnet,Tags=[{Key=Name,Value=private-subnet-a}]'

# Create a PUBLIC subnet in Availability Zone B (for high availability)
aws ec2 create-subnet \
    --vpc-id vpc-0abc123def456 \
    --cidr-block 10.0.3.0/24 \
    --availability-zone us-east-1b \
    --tag-specifications 'ResourceType=subnet,Tags=[{Key=Name,Value=public-subnet-b}]'

# Create a PRIVATE subnet in Availability Zone B
aws ec2 create-subnet \
    --vpc-id vpc-0abc123def456 \
    --cidr-block 10.0.4.0/24 \
    --availability-zone us-east-1b \
    --tag-specifications 'ResourceType=subnet,Tags=[{Key=Name,Value=private-subnet-b}]'
```

**Explanation:**
- We create 4 subnets across 2 Availability Zones (for redundancy)
- Each subnet gets a /24 CIDR block (251 usable IPs)
- Public subnets for web-facing resources, private for internal resources

### A Subnet is NOT Public or Private by Default

**Important:** A subnet doesn't magically become "public" or "private" just because you name it that way. What makes it public or private is the **Route Table** attached to it:

- If the route table has a route to an **Internet Gateway** → it's a **public subnet**
- If the route table does NOT route to an Internet Gateway → it's a **private subnet**

## Internet Gateway (IGW)

### What Is an Internet Gateway?

An **Internet Gateway** is the door between your VPC and the public internet. Without it, nothing in your VPC can reach the internet, and nothing from the internet can reach your VPC.

> An **Internet Gateway (IGW)** is a horizontally scaled, redundant, and highly available VPC component that allows communication between your VPC and the internet.

### How It Works

```
                              Internet Gateway
                                    │
Internet ◄───────────────────►  [  IGW  ]  ◄───────► Public Subnet
                                                       (Web Servers)
```

1. A user on the internet sends a request to your web server's public IP
2. The request hits the **Internet Gateway**
3. The IGW forwards it to the web server in the public subnet
4. The web server responds, and the response goes back through the IGW to the user

### Key Facts About Internet Gateways

- **One per VPC** — Each VPC can have only one IGW attached
- **Free** — No additional cost for the IGW itself
- **Highly Available** — AWS manages redundancy automatically
- **Two-way** — Handles both inbound and outbound traffic
- **No bandwidth limits** — Scales automatically

### Setting Up an Internet Gateway (AWS CLI)

```bash
# Step 1: Create an Internet Gateway
aws ec2 create-internet-gateway \
    --tag-specifications 'ResourceType=internet-gateway,Tags=[{Key=Name,Value=my-igw}]'

# Step 2: Attach it to your VPC
aws ec2 attach-internet-gateway \
    --internet-gateway-id igw-0abc123 \
    --vpc-id vpc-0abc123def456

# Step 3: Create a route table for public subnets
aws ec2 create-route-table \
    --vpc-id vpc-0abc123def456 \
    --tag-specifications 'ResourceType=route-table,Tags=[{Key=Name,Value=public-rt}]'

# Step 4: Add a route to the Internet Gateway
# This is what makes a subnet "public"!
aws ec2 create-route \
    --route-table-id rtb-0abc123 \
    --destination-cidr-block 0.0.0.0/0 \
    --gateway-id igw-0abc123

# Step 5: Associate the route table with public subnets
aws ec2 associate-route-table \
    --route-table-id rtb-0abc123 \
    --subnet-id subnet-public-a

aws ec2 associate-route-table \
    --route-table-id rtb-0abc123 \
    --subnet-id subnet-public-b
```

**Explanation:**
- `0.0.0.0/0` — This means "any IP address" (all internet traffic)
- `--gateway-id igw-0abc123` — Route it through the Internet Gateway
- By associating this route table with a subnet, that subnet becomes **public**

### Route Table Comparison

**Public Subnet Route Table:**

| Destination | Target | Meaning |
|-------------|--------|---------|
| 10.0.0.0/16 | local | Traffic within the VPC stays within the VPC |
| 0.0.0.0/0 | igw-xxx | Everything else goes to the Internet Gateway |

**Private Subnet Route Table:**

| Destination | Target | Meaning |
|-------------|--------|---------|
| 10.0.0.0/16 | local | Traffic within the VPC stays within the VPC |
| 0.0.0.0/0 | nat-xxx | Everything else goes through the NAT Gateway |

## NAT — Network Address Translation

### What Is NAT?

**NAT** is the process of translating private IP addresses to public IP addresses (and vice versa) so that devices on a private network can communicate with the internet.

> **NAT (Network Address Translation)** modifies the source or destination IP address of packets as they pass through a router or gateway.

### Why Do We Need NAT?

**The Problem:** Your database server sits in a **private subnet** (no public IP, no Internet Gateway route). But it needs to:
- Download software updates (`apt update`)
- Pull Docker images
- Send logs to an external service
- Make API calls to third-party services

**The Challenge:** The database shouldn't be directly accessible FROM the internet (security risk), but it needs to reach OUT to the internet.

**The Solution:** NAT Gateway

### NAT Gateway

A **NAT Gateway** allows resources in a **private subnet** to initiate outbound connections to the internet, while preventing the internet from initiating inbound connections to those resources.

```
                                         Internet
                                            │
                                     ┌──────┴──────┐
                                     │     IGW     │
                                     └──────┬──────┘
                                            │
                              ┌─────────────┼─────────────┐
                              │                           │
                    ┌─────────┴─────────┐       ┌────────┴────────┐
                    │  Public Subnet    │       │  Public Subnet  │
                    │  ┌─────────────┐  │       │                 │
                    │  │ NAT Gateway │  │       │  Web Server     │
                    │  └──────┬──────┘  │       │                 │
                    └─────────┼─────────┘       └─────────────────┘
                              │
                    ┌─────────┴─────────┐
                    │  Private Subnet   │
                    │                   │
                    │  Database Server  │  ← Can reach internet via NAT
                    │  App Server       │  ← But internet can't reach them
                    │                   │
                    └───────────────────┘
```

**How it works (step by step):**

1. Database server (10.0.2.50) wants to download updates from the internet
2. It sends the packet to the NAT Gateway (via route table)
3. NAT Gateway **replaces** the source IP (10.0.2.50) with its own **public IP** (e.g., 52.1.2.3)
4. The packet goes to the internet through the IGW
5. The response comes back to the NAT Gateway's public IP (52.1.2.3)
6. NAT Gateway **translates** the destination back to 10.0.2.50
7. The packet is delivered to the database server

**The internet only sees the NAT Gateway's IP — never the database server's private IP.**

### Types of NAT

| Type | Description | Use Case |
|------|-------------|----------|
| **NAT Gateway** (AWS managed) | Fully managed, highly available, auto-scales | Production workloads |
| **NAT Instance** (self-managed) | An EC2 instance configured to do NAT | Dev/test (cheaper but you manage it) |
| **Static NAT** (1:1) | One private IP maps to one public IP | When you need a fixed public IP |
| **Dynamic NAT** | Pool of public IPs shared among private IPs | Multiple devices sharing IPs |
| **PAT (Port Address Translation)** | Many private IPs share ONE public IP using different ports | Home routers! (your WiFi does this) |

### NAT at Home — You're Already Using It!

Your home router is doing NAT right now:

```
Your Laptop (192.168.1.10) ──┐
Your Phone (192.168.1.11)  ──┤──► Router (NAT) ──► Internet
Smart TV (192.168.1.12)    ──┘    Public IP: 203.0.113.5
```

All your devices share ONE public IP (from your ISP). The router uses **PAT (Port Address Translation)** to keep track of which device made which request:

```
192.168.1.10:54321 → 203.0.113.5:54321 → google.com:443
192.168.1.11:12345 → 203.0.113.5:12345 → youtube.com:443
192.168.1.12:33333 → 203.0.113.5:33333 → netflix.com:443
```

Each device gets a unique port number on the public IP.

### Setting Up NAT Gateway (AWS CLI)

```bash
# Step 1: Allocate an Elastic IP (NAT Gateway needs a public IP)
aws ec2 allocate-address --domain vpc

# Step 2: Create the NAT Gateway in a PUBLIC subnet
aws ec2 create-nat-gateway \
    --subnet-id subnet-public-a \
    --allocation-id eipalloc-0abc123 \
    --tag-specifications 'ResourceType=natgateway,Tags=[{Key=Name,Value=my-nat-gw}]'

# Step 3: Update the PRIVATE subnet's route table
# Route all internet traffic through the NAT Gateway
aws ec2 create-route \
    --route-table-id rtb-private \
    --destination-cidr-block 0.0.0.0/0 \
    --nat-gateway-id nat-0abc123
```

**Explanation:**
- NAT Gateway must be placed in a **public subnet** (it needs internet access via IGW)
- It needs an **Elastic IP** (static public IP)
- The private subnet's route table sends all outbound internet traffic (`0.0.0.0/0`) through the NAT Gateway

### NAT Gateway vs NAT Instance

| Feature | NAT Gateway | NAT Instance |
|---------|-------------|-------------|
| **Managed by** | AWS (fully managed) | You (self-managed EC2) |
| **Availability** | Highly available in one AZ | You must set up redundancy |
| **Bandwidth** | Up to 100 Gbps | Depends on instance type |
| **Maintenance** | None — AWS handles patches | You must patch and update |
| **Cost** | ~$0.045/hr + data processing | EC2 instance cost |
| **Security Groups** | Not supported (uses NACLs) | Supports Security Groups |
| **Best For** | Production | Dev/test, learning |

## Putting It All Together — Complete VPC Architecture

Here's a real-world production VPC setup using Terraform (Infrastructure as Code):

```hcl
# main.tf — Complete VPC setup with Terraform

# 1. Create the VPC
resource "aws_vpc" "main" {
  cidr_block           = "10.0.0.0/16"
  enable_dns_support   = true
  enable_dns_hostnames = true

  tags = {
    Name = "production-vpc"
  }
}

# 2. Create Internet Gateway
resource "aws_internet_gateway" "igw" {
  vpc_id = aws_vpc.main.id

  tags = {
    Name = "production-igw"
  }
}

# 3. Create Public Subnets (2 AZs for high availability)
resource "aws_subnet" "public_a" {
  vpc_id                  = aws_vpc.main.id
  cidr_block              = "10.0.1.0/24"
  availability_zone       = "us-east-1a"
  map_public_ip_on_launch = true      # Auto-assign public IPs

  tags = {
    Name = "public-subnet-a"
  }
}

resource "aws_subnet" "public_b" {
  vpc_id                  = aws_vpc.main.id
  cidr_block              = "10.0.3.0/24"
  availability_zone       = "us-east-1b"
  map_public_ip_on_launch = true

  tags = {
    Name = "public-subnet-b"
  }
}

# 4. Create Private Subnets
resource "aws_subnet" "private_a" {
  vpc_id            = aws_vpc.main.id
  cidr_block        = "10.0.2.0/24"
  availability_zone = "us-east-1a"

  tags = {
    Name = "private-subnet-a"
  }
}

resource "aws_subnet" "private_b" {
  vpc_id            = aws_vpc.main.id
  cidr_block        = "10.0.4.0/24"
  availability_zone = "us-east-1b"

  tags = {
    Name = "private-subnet-b"
  }
}

# 5. Elastic IP for NAT Gateway
resource "aws_eip" "nat" {
  domain = "vpc"

  tags = {
    Name = "nat-eip"
  }
}

# 6. NAT Gateway (in public subnet)
resource "aws_nat_gateway" "nat" {
  allocation_id = aws_eip.nat.id
  subnet_id     = aws_subnet.public_a.id     # NAT goes in PUBLIC subnet

  tags = {
    Name = "production-nat"
  }

  depends_on = [aws_internet_gateway.igw]
}

# 7. Route Table for Public Subnets
resource "aws_route_table" "public" {
  vpc_id = aws_vpc.main.id

  route {
    cidr_block = "0.0.0.0/0"
    gateway_id = aws_internet_gateway.igw.id   # Route to Internet Gateway
  }

  tags = {
    Name = "public-rt"
  }
}

# 8. Route Table for Private Subnets
resource "aws_route_table" "private" {
  vpc_id = aws_vpc.main.id

  route {
    cidr_block     = "0.0.0.0/0"
    nat_gateway_id = aws_nat_gateway.nat.id    # Route to NAT Gateway
  }

  tags = {
    Name = "private-rt"
  }
}

# 9. Associate Route Tables with Subnets
resource "aws_route_table_association" "public_a" {
  subnet_id      = aws_subnet.public_a.id
  route_table_id = aws_route_table.public.id
}

resource "aws_route_table_association" "public_b" {
  subnet_id      = aws_subnet.public_b.id
  route_table_id = aws_route_table.public.id
}

resource "aws_route_table_association" "private_a" {
  subnet_id      = aws_subnet.private_a.id
  route_table_id = aws_route_table.private.id
}

resource "aws_route_table_association" "private_b" {
  subnet_id      = aws_subnet.private_b.id
  route_table_id = aws_route_table.private.id
}
```

**What this Terraform code does:**
1. Creates a VPC with a large IP range (10.0.0.0/16)
2. Creates an Internet Gateway and attaches it to the VPC
3. Creates 2 public subnets and 2 private subnets across 2 Availability Zones
4. Creates a NAT Gateway in a public subnet with an Elastic IP
5. Creates route tables — public routes to IGW, private routes to NAT
6. Associates each subnet with its appropriate route table

## Security Groups vs NACLs

Both control traffic, but at different levels:

### Security Groups (Instance Level)

- Act like a **firewall for individual instances**
- **Stateful** — if you allow inbound, the response is automatically allowed outbound
- **Allow rules only** — you can only specify what to ALLOW
- Applied at the **instance/ENI level**

```bash
# Create a Security Group
aws ec2 create-security-group \
    --group-name web-server-sg \
    --description "Security group for web servers" \
    --vpc-id vpc-0abc123

# Allow HTTP from anywhere
aws ec2 authorize-security-group-ingress \
    --group-id sg-0abc123 \
    --protocol tcp \
    --port 80 \
    --cidr 0.0.0.0/0

# Allow SSH from your IP only
aws ec2 authorize-security-group-ingress \
    --group-id sg-0abc123 \
    --protocol tcp \
    --port 22 \
    --cidr 203.0.113.50/32    # Your specific IP
```

### Network ACLs (Subnet Level)

- Act like a **firewall for entire subnets**
- **Stateless** — you must explicitly allow both inbound AND outbound
- **Allow AND Deny rules** — you can specify both
- Applied at the **subnet level**
- Rules are evaluated in **order** (lowest number first)

```bash
# Example NACL rules for a public subnet:

# Inbound Rules:
# Rule 100: Allow HTTP (port 80) from anywhere
# Rule 110: Allow HTTPS (port 443) from anywhere
# Rule 120: Allow SSH (port 22) from your IP
# Rule *:   Deny everything else

# Outbound Rules:
# Rule 100: Allow all traffic out
# Rule *:   Deny everything else
```

### Security Groups vs NACLs Comparison

| Feature | Security Group | Network ACL |
|---------|---------------|-------------|
| Level | Instance | Subnet |
| Stateful? | Yes | No |
| Rule Type | Allow only | Allow and Deny |
| Rule Order | All rules evaluated | Evaluated in number order |
| Default | Deny all inbound, allow all outbound | Allow all inbound and outbound |
| Applied To | Individual instances | All instances in the subnet |

## VPC Peering

### What Is VPC Peering?

**VPC Peering** connects two VPCs so resources in each VPC can communicate using private IP addresses — as if they were in the same network.

```
┌─── VPC A (10.0.0.0/16) ───┐     ┌─── VPC B (172.16.0.0/16) ───┐
│                            │     │                              │
│  App Server (10.0.1.10)   ◄─────►  Database (172.16.1.20)     │
│                            │     │                              │
└────────────────────────────┘     └──────────────────────────────┘
                    VPC Peering Connection
```

**Key Rules:**
- CIDR blocks must NOT overlap (10.0.0.0/16 and 10.0.0.0/16 would fail)
- **Not transitive** — if VPC A peers with VPC B, and VPC B peers with VPC C, A cannot automatically reach C
- Can peer across regions and even across AWS accounts

---

## AWS Transit Gateway — The Better Way

For many VPCs, peering gets messy. **Transit Gateway** acts as a central hub:

```
                      ┌───────────────┐
            ┌────────►│Transit Gateway│◄────────┐
            │         └───────┬───────┘         │
            │                 │                 │
     ┌──────┴──────┐  ┌──────┴──────┐  ┌──────┴──────┐
     │    VPC A    │  │    VPC B    │  │    VPC C    │
     └─────────────┘  └─────────────┘  └─────────────┘
                              │
                      ┌───────┴───────┐
                      │ On-Premises   │
                      │ Data Center   │
                      └───────────────┘
```

All VPCs connect to the Transit Gateway, and the Transit Gateway routes traffic between them. Much simpler than managing dozens of peering connections.

## Quick Summary

| Concept | Key Takeaway |
|---------|-------------|
| **VPN** | Encrypted tunnel over the internet — connects remote users or sites to a private network |
| **VPC** | Your isolated private network in the cloud — you control the IPs, subnets, and routing |
| **Subnet** | A smaller network inside your VPC — public (internet-facing) or private (internal) |
| **Internet Gateway** | The door between your VPC and the public internet |
| **NAT Gateway** | Lets private subnet resources reach the internet without being reachable FROM the internet |
| **Route Table** | Decides where traffic goes — IGW for public subnets, NAT for private subnets |
| **Security Group** | Instance-level firewall (stateful, allow-only rules) |
| **NACL** | Subnet-level firewall (stateless, allow + deny rules) |
| **VPC Peering** | Connects two VPCs privately |
| **Transit Gateway** | Central hub connecting multiple VPCs and on-premises networks |

```

      │

```

## Key Interview Questions

1. **What is a VPC?**
   → An isolated virtual network in the cloud where you launch resources. You control the IP range, subnets, routing, and security.

2. **What's the difference between a public and private subnet?**
   → A public subnet has a route to the Internet Gateway. A private subnet does not — it routes through a NAT Gateway for outbound internet access.

3. **What is a NAT Gateway and why do we need it?**
   → NAT Gateway allows private subnet resources to reach the internet (for updates, API calls) without being directly accessible from the internet.

4. **Where does the NAT Gateway go — public or private subnet?**
   → **Public subnet.** It needs internet access via the IGW to forward traffic from private resources.

5. **What's the difference between Security Groups and NACLs?**
   → Security Groups are stateful (instance-level, allow-only). NACLs are stateless (subnet-level, allow + deny).

6. **What is VPC Peering?**
   → A connection between two VPCs that allows private communication. CIDRs must not overlap, and it's not transitive.

7. **What makes a subnet public?**
   → A route in its route table that sends 0.0.0.0/0 traffic to an Internet Gateway.

8. **What's the difference between a VPN and VPC Peering?**
   → VPN connects over the public internet (encrypted). VPC Peering connects within AWS's private network (faster, lower latency, no encryption needed).
