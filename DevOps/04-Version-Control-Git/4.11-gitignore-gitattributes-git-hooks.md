## 1. .gitignore — Telling Git What to Ignore

A `.gitignore` file tells Git: **"Hey, don't track these files. Pretend they don't exist."**

Think of it like this:

> Imagine you're packing for a vacation. You want to bring your clothes, passport, and camera. But you DON'T want to bring your dirty laundry, empty water bottles, or used tissues. A `.gitignore` is your **"do NOT pack" list** — it tells Git which files to leave behind.

### What goes in `.gitignore`?

Files that should **never** be in your Git repository:

| Type | Examples | Why Ignore? |
|------|----------|-------------|
| **Dependencies** | `node_modules/`, `vendor/`, `venv/` | Huge folders — reinstall from package file |
| **Build outputs** | `dist/`, `build/`, `*.class`, `*.o` | Can be regenerated from source |
| **Environment files** | `.env`, `.env.local` | Contain secrets (API keys, passwords) |
| **IDE settings** | `.vscode/`, `.idea/`, `*.swp` | Personal editor preferences |
| **OS files** | `.DS_Store`, `Thumbs.db` | OS junk files |
| **Logs** | `*.log`, `npm-debug.log` | Temporary debugging output |
| **Compiled files** | `*.pyc`, `*.exe`, `*.dll` | Binary files generated from source |

---

## 2. Why Do We Need .gitignore?

### Without .gitignore:

```
$ git status
Changes not staged for commit:
  modified:   src/app.js          ← Your actual code change
  modified:   .DS_Store            ← macOS junk
  modified:   node_modules/...     ← 50,000 dependency files
  modified:   .env                 ← YOUR API KEYS AND PASSWORDS! 🚨
  modified:   dist/bundle.js       ← Auto-generated file
```

Problems:
1. **Security risk** — `.env` files with secrets get pushed to GitHub (public repos = leaked passwords!)
2. **Huge repository** — `node_modules` can be 500MB+. Nobody wants to download that.
3. **Noisy diffs** — Auto-generated files create meaningless changes in every commit.
4. **Merge conflicts** — OS-specific files cause conflicts between Windows and Mac developers.

### With .gitignore:

```
$ git status
Changes not staged for commit:
  modified:   src/app.js          ← Only your actual code change! ✅
```

Clean. Simple. Safe.

## 3. .gitignore Syntax & Patterns

Create a `.gitignore` file in the **root of your repository**:

```bash
touch .gitignore
```

### Basic Syntax

```gitignore
# This is a comment (lines starting with # are ignored)

# Ignore a specific file
secret.txt

# Ignore all files with a specific extension
*.log
*.tmp
*.pyc

# Ignore a specific folder (and everything inside it)
node_modules/
dist/
build/

# Ignore files in a specific folder
logs/*.log

# Ignore all files in all subdirectories with a pattern
**/*.class

# Negate a pattern (DON'T ignore this file)
!important.log

# Ignore everything in a folder EXCEPT one file
temp/*
!temp/.gitkeep
```

### Pattern Reference Table

| Pattern | Meaning | Example Matches |
|---------|---------|-----------------|
| `*.log` | Any file ending in `.log` | `error.log`, `debug.log` |
| `debug.log` | A specific file named `debug.log` | `debug.log` (in root) |
| `debug?.log` | `?` matches any single character | `debug0.log`, `debugA.log` |
| `debug[0-9].log` | `[ ]` matches a range of characters | `debug0.log`, `debug9.log` |
| `logs/` | A directory called `logs` | `logs/` and everything inside |
| `logs` | A file or directory called `logs` | `logs` file or `logs/` directory |
| `**/logs` | `logs` folder anywhere in the repo | `a/logs/`, `a/b/logs/` |
| `logs/**` | Everything inside `logs` | `logs/file.txt`, `logs/sub/file.txt` |
| `**/logs/*.log` | `.log` files in any `logs` folder | `a/logs/error.log` |
| `!important.log` | Negate — DON'T ignore this file | `important.log` is tracked |
| `/debug.log` | Only in the root directory | `debug.log` but NOT `sub/debug.log` |

### Precedence Rules

When patterns conflict, **later rules override earlier ones**:

```gitignore
# Ignore all .log files
*.log

# But DON'T ignore important.log
!important.log
```

Result: All `.log` files are ignored **except** `important.log`.

**But watch out!** You can't un-ignore a file inside an ignored directory:

```gitignore
# This WON'T work:
logs/
!logs/important.log    # ❌ Can't un-ignore — parent directory is ignored

# Instead, do this:
logs/*                 # Ignore files inside logs (not the directory itself)
!logs/important.log    # ✅ Now this works!
```

## 5. Global .gitignore

Some files (like `.DS_Store` on macOS or `Thumbs.db` on Windows) should be ignored in **every** repository. Instead of adding them to each project's `.gitignore`, create a **global** one:

### Step 1: Create the Global .gitignore File

```bash
# Create the file
touch ~/.gitignore_global
```

### Step 2: Add Your Global Ignores

```bash
# Edit the file
nano ~/.gitignore_global
```

```gitignore
# macOS
.DS_Store
.AppleDouble
.LSOverride

# Windows
Thumbs.db
ehthumbs.db
Desktop.ini

# Linux
*~
.directory

# IDEs
.idea/
.vscode/
*.swp
*.swo
*.sublime-workspace
*.sublime-project

# Compiled files
*.com
*.class
*.dll
*.exe
*.o
*.so
```

### Step 3: Tell Git to Use It

```bash
git config --global core.excludesfile ~/.gitignore_global
```

Now these files will be ignored in **every** Git repository on your machine — no need to add them to each project!

## 6. Ignoring Already Tracked Files

**Common problem:** You added `.env` to `.gitignore`, but Git still tracks it! That's because `.gitignore` only affects **untracked** files. If Git is already tracking a file, adding it to `.gitignore` won't stop tracking it.

### How to Fix It

```bash
# Step 1: Remove the file from Git's tracking (but keep the file on disk)
git rm --cached .env

# Step 2: Commit the change
git commit -m "Stop tracking .env file"

# Step 3: Make sure .env is in .gitignore
echo ".env" >> .gitignore
git add .gitignore
git commit -m "Add .env to .gitignore"
```

**What does `--cached` do?** It tells Git: "Remove the file from tracking, but DON'T delete it from my hard drive."

### Removing an Entire Folder from Tracking

```bash
# Stop tracking node_modules (if accidentally committed)
git rm -r --cached node_modules/
git commit -m "Remove node_modules from tracking"
```

### Important Warning! ⚠️

Even after removing a file from tracking, **it still exists in Git history!** If you accidentally committed a password or API key, anyone can see it by looking at old commits.

To truly remove sensitive data from Git history, you need:

```bash
# Using git-filter-repo (recommended)
pip install git-filter-repo
git filter-repo --path .env --invert-paths

# Or using BFG Repo-Cleaner
java -jar bfg.jar --delete-files .env
git reflog expire --expire=now --all
git gc --prune=now --aggressive
```

> **Best practice:** Never commit secrets in the first place. Use environment variables or a secrets manager.

## 7. Debugging .gitignore Issues

### "Why is Git still tracking this file?!"

Use `git check-ignore` to debug:

```bash
# Check if a file is ignored
git check-ignore -v .env

# Output:
# .gitignore:3:.env    .env
# ↑ file       ↑ line  ↑ pattern   ↑ matched file
```

This tells you:
- **Which `.gitignore` file** contains the rule
- **Which line** the rule is on
- **Which pattern** matched

### "Why is my file NOT being ignored?"

```bash
# Verbose check — shows even non-ignored files
git check-ignore -v --no-index myfile.txt

# If nothing is printed, no rule matches this file
```

### List All Ignored Files

```bash
# Show all files that are currently being ignored
git status --ignored

# Or more compact
git ls-files --others --ignored --exclude-standard
```

## 8. .gitattributes — Telling Git How to Handle Files

While `.gitignore` tells Git **what to ignore**, `.gitattributes` tells Git **how to handle files**.

Think of it like this:

> `.gitignore` = "Don't pack these items"  
> `.gitattributes` = "Pack the clothes folded, the shoes in bags, and the camera in bubble wrap"

It controls:
- **Line endings** (LF vs CRLF — the #1 cross-platform headache)
- **Diff behavior** (how to show differences for binary files)
- **Merge strategy** (how to resolve conflicts for specific files)
- **Git LFS** (track large files separately)
- **Language statistics** (what GitHub shows in your repo's language bar)
- **Export behavior** (what to include/exclude in archives)

---

## 9. Why Do We Need .gitattributes?

### The Line Ending Problem

This is the **#1 reason** `.gitattributes` exists:

- **Windows** uses `CRLF` (Carriage Return + Line Feed: `\r\n`)
- **Linux/macOS** uses `LF` (Line Feed: `\n`)

When a Windows developer and a Mac developer work on the same repo:

```
Without .gitattributes:
├── Windows dev commits with CRLF
├── Mac dev pulls and sees every single line changed
├── Mac dev commits with LF  
├── Windows dev pulls and sees every single line changed
├── Both developers are confused and angry 😤
└── Repeat forever...
```

With `.gitattributes`, you define **one consistent behavior** for the entire team.

## 11. Line Ending Nightmares (CRLF vs LF)

Let's understand this properly so you never have this problem again.

### What Are Line Endings?

When you press **Enter** in a text file, an invisible character is inserted:

```
Windows:    Hello\r\n     (CRLF — Carriage Return + Line Feed — 2 characters)
Linux/Mac:  Hello\n       (LF — Line Feed — 1 character)
Old Mac:    Hello\r       (CR — Carriage Return — extinct, nobody uses this)
```

### Why Does This Cause Problems?

```bash
# Windows developer creates a file:
echo "Hello World" > file.txt
# File contains: Hello World\r\n

# Linux developer opens the same file and sees:
cat -A file.txt
# Hello World^M$
#              ↑ This ^M is the \r character — it shouldn't be there on Linux!
```

Shell scripts with CRLF line endings **will fail** on Linux:

```bash
# A script with CRLF line endings:
#!/bin/bash\r\n     ← Linux reads this as: interpreter is "/bin/bash\r"
echo "Hello"\r\n    ← which doesn't exist, so the script crashes!

# Error: /bin/bash^M: bad interpreter: No such file or directory
```

### The Solution: `.gitattributes` + `core.autocrlf`

#### Option 1: `.gitattributes` (Recommended — project-level)

```gitattributes
# Normalize all text files to LF in the repo
# On checkout, convert to OS-native line endings
* text=auto
```

#### Option 2: `core.autocrlf` (Per-developer setting)

```bash
# On Windows — convert CRLF to LF on commit, LF to CRLF on checkout
git config --global core.autocrlf true

# On Linux/macOS — don't convert on checkout, convert CRLF→LF on commit
git config --global core.autocrlf input

# Disable conversion entirely (not recommended for teams)
git config --global core.autocrlf false
```

### How It Works Together

```
                  Working Directory    →    Git Repository    →    Working Directory
                  (your file system)        (stored commits)       (colleague's machine)

Windows dev:      file.txt (CRLF)    →    file.txt (LF)     →    file.txt (CRLF)
Linux dev:        file.txt (LF)      →    file.txt (LF)     →    file.txt (LF)
                                     ↑
                              Always stored as LF!
                           (consistent for everyone)
```

### Fix Line Endings in an Existing Repo

If your repo already has mixed line endings:

```bash
# Step 1: Add .gitattributes
echo "* text=auto" > .gitattributes
git add .gitattributes
git commit -m "Add .gitattributes for line ending normalization"

# Step 2: Renormalize all files
git add --renormalize .
git commit -m "Normalize line endings"
```

## 12. Git LFS (Large File Storage) with .gitattributes

Git is designed for text files (code). Large binary files (videos, images, datasets) make your repo **huge** and **slow**. Git LFS solves this by storing large files on a separate server.

### Install Git LFS

```bash
# Ubuntu/Debian
sudo apt install git-lfs

# macOS
brew install git-lfs

# Windows
# Download from https://git-lfs.com

# Initialize LFS in your repo
git lfs install
```

### Track Large Files

```bash
# Tell Git LFS to handle Photoshop files
git lfs track "*.psd"

# Track video files
git lfs track "*.mp4"
git lfs track "*.mov"

# Track large datasets
git lfs track "*.csv"
git lfs track "data/*.zip"
```

This **automatically adds entries to `.gitattributes`**:

```gitattributes
*.psd filter=lfs diff=lfs merge=lfs -text
*.mp4 filter=lfs diff=lfs merge=lfs -text
*.mov filter=lfs diff=lfs merge=lfs -text
*.csv filter=lfs diff=lfs merge=lfs -text
data/*.zip filter=lfs diff=lfs merge=lfs -text
```

### How It Works

```
Without LFS:
Git repo contains:  code files + HUGE binary files (500MB video)
Clone size:         2 GB 😱

With LFS:
Git repo contains:  code files + tiny pointer files (100 bytes each)
LFS server stores:  HUGE binary files (downloaded on demand)
Clone size:         50 MB 😊
```

### Important Commands

```bash
# See which files are tracked by LFS
git lfs ls-files

# See LFS status
git lfs status

# Pull LFS files (if you cloned without them)
git lfs pull

# Migrate existing files to LFS
git lfs migrate import --include="*.psd"
```

## 13. What Are Git Hooks?

**Git Hooks** are **scripts that run automatically** when specific Git events happen. They let you automate tasks, enforce rules, and catch problems before they cause damage.

Think of it like this:

> Imagine a nightclub with a bouncer at the door. Before you enter (commit), the bouncer (hook) checks your ID, makes sure you're dressed properly, and you're on the guest list. If anything fails, you're not getting in. **Git hooks are bouncers for your code.**

### What Can You Automate with Hooks?

| Event | Hook | What You Can Do |
|-------|------|-----------------|
| Before committing | `pre-commit` | Run linter, formatter, spell-check |
| After writing commit message | `commit-msg` | Validate commit message format |
| Before pushing | `pre-push` | Run tests, check for secrets |
| After merging | `post-merge` | Install dependencies, rebuild |
| Before receiving a push (server) | `pre-receive` | Enforce branch protection rules |

---

## 14. Where Do Git Hooks Live?

Git hooks live in the `.git/hooks/` directory of every Git repository:

```bash
ls .git/hooks/

# Output:
# applypatch-msg.sample
# commit-msg.sample
# post-update.sample
# pre-applypatch.sample
# pre-commit.sample
# pre-merge-commit.sample
# pre-push.sample
# pre-rebase.sample
# prepare-commit-msg.sample
# update.sample
```

**Notice:** They all end in `.sample`. Git provides **example hooks** but they're disabled by default. To activate a hook, **remove the `.sample` extension** and make it executable.

```bash
# Activate the pre-commit hook
mv .git/hooks/pre-commit.sample .git/hooks/pre-commit
chmod +x .git/hooks/pre-commit
```

### Important Facts

| Fact | Detail |
|------|--------|
| **Location** | `.git/hooks/` directory |
| **Language** | Any executable script (Bash, Python, Node.js, Ruby, etc.) |
| **Activation** | Remove `.sample` extension, make executable (`chmod +x`) |
| **Shared?** | ❌ NOT shared via `git push` (`.git/` is local only) |
| **Bypass** | `git commit --no-verify` skips hooks (but don't do this!) |
| **Exit code** | Return `0` = success (proceed), return `1` = failure (abort) |

## 15. Types of Git Hooks — Complete List

### Client-Side Hooks (Run on Developer's Machine)

| Hook | When It Runs | Common Use |
|------|-------------|------------|
| **`pre-commit`** | Before a commit is created | Lint code, format code, check for secrets |
| **`prepare-commit-msg`** | After default commit message is created, before editor opens | Auto-add ticket number to message |
| **`commit-msg`** | After user writes commit message | Validate message format |
| **`post-commit`** | After commit is completed | Send notifications, update dashboard |
| **`pre-push`** | Before push starts | Run tests, check branch rules |
| **`post-checkout`** | After `git checkout` | Install dependencies, rebuild |
| **`post-merge`** | After `git merge` | Install dependencies, rebuild |
| **`pre-rebase`** | Before `git rebase` | Prevent rebasing published commits |

### Server-Side Hooks (Run on Git Server)

| Hook | When It Runs | Common Use |
|------|-------------|------------|
| **`pre-receive`** | Before accepting a push | Enforce rules (branch protection, code review) |
| **`update`** | Like pre-receive, but per branch | Per-branch access control |
| **`post-receive`** | After push is accepted | Trigger CI/CD, send notifications |

## 16. pre-commit Hook — Deep Dive

The `pre-commit` hook runs **before every commit**. If it exits with a non-zero code, the commit is **aborted**.

### Example 1: Block Commits with Debug Statements

```bash
#!/bin/bash
# .git/hooks/pre-commit
# Block commits that contain console.log or debugger statements

echo "🔍 Checking for debug statements..."

# Check staged files for debug statements
if git diff --cached --name-only --diff-filter=ACM | xargs grep -l 'console\.log\|debugger\|print(' 2>/dev/null; then
    echo ""
    echo "❌ COMMIT BLOCKED!"
    echo "Found debug statements (console.log, debugger, or print()) in your code."
    echo "Please remove them before committing."
    echo ""
    echo "Files with debug statements:"
    git diff --cached --name-only --diff-filter=ACM | xargs grep -n 'console\.log\|debugger\|print(' 2>/dev/null
    echo ""
    echo "To bypass this check (not recommended): git commit --no-verify"
    exit 1
fi

echo "✅ No debug statements found. Proceeding with commit."
exit 0
```

Let's break this down:

| Part | Meaning |
|------|---------|
| `#!/bin/bash` | This hook is a Bash script |
| `git diff --cached` | Look at files that are **staged** (added with `git add`) |
| `--name-only` | Only show file names |
| `--diff-filter=ACM` | Only check **Added**, **Copied**, or **Modified** files |
| `xargs grep -l` | Search for patterns in those files |
| `exit 1` | **Abort the commit** (non-zero = failure) |
| `exit 0` | **Allow the commit** (zero = success) |

### Example 2: Auto-Format Code Before Commit

```bash
#!/bin/bash
# .git/hooks/pre-commit
# Auto-format staged files with Prettier

echo "✨ Running Prettier on staged files..."

# Get staged JavaScript/TypeScript files
STAGED_FILES=$(git diff --cached --name-only --diff-filter=ACM | grep -E '\.(js|jsx|ts|tsx|json|css|md)$')

if [ -z "$STAGED_FILES" ]; then
    echo "No files to format."
    exit 0
fi

# Format each staged file
echo "$STAGED_FILES" | while read FILE; do
    npx prettier --write "$FILE"
    git add "$FILE"    # Re-stage the formatted file
done

echo "✅ Formatting complete!"
exit 0
```

### Example 3: Check for Secrets Before Commit

```bash
#!/bin/bash
# .git/hooks/pre-commit
# Prevent committing files with potential secrets

echo "🔐 Scanning for secrets..."

# Patterns that look like secrets
PATTERNS=(
    'password\s*=\s*["\x27]'
    'api[_-]?key\s*=\s*["\x27]'
    'secret\s*=\s*["\x27]'
    'AWS_ACCESS_KEY_ID'
    'AWS_SECRET_ACCESS_KEY'
    'PRIVATE[_-]?KEY'
    'BEGIN RSA PRIVATE KEY'
    'BEGIN OPENSSH PRIVATE KEY'
)

FOUND=0

for pattern in "${PATTERNS[@]}"; do
    RESULTS=$(git diff --cached --name-only --diff-filter=ACM | xargs grep -inE "$pattern" 2>/dev/null)
    if [ -n "$RESULTS" ]; then
        echo "⚠️  Potential secret found matching pattern: $pattern"
        echo "$RESULTS"
        FOUND=1
    fi
done

if [ $FOUND -eq 1 ]; then
    echo ""
    echo "❌ COMMIT BLOCKED! Potential secrets detected in staged files."
    echo "Please remove secrets and use environment variables instead."
    exit 1
fi

echo "✅ No secrets found. Proceeding with commit."
exit 0
```

## 17. commit-msg Hook — Enforce Commit Message Format

The `commit-msg` hook runs after you write your commit message. It receives the commit message file path as argument `$1`. You can validate the message format and reject badly formatted commits.

### Example: Enforce Conventional Commits

```bash
#!/bin/bash
# .git/hooks/commit-msg
# Enforce Conventional Commits format:
# type(scope): description
# 
# Examples:
#   feat: add dark mode
#   fix(auth): resolve login crash
#   docs: update README
#   chore(deps): upgrade express to v5

COMMIT_MSG_FILE="$1"
COMMIT_MSG=$(cat "$COMMIT_MSG_FILE")

# Conventional Commits regex
PATTERN="^(feat|fix|docs|style|refactor|perf|test|build|ci|chore|revert)(\(.+\))?: .{1,100}$"

# Skip merge commits
if echo "$COMMIT_MSG" | grep -qE "^Merge "; then
    exit 0
fi

if ! echo "$COMMIT_MSG" | head -1 | grep -qE "$PATTERN"; then
    echo ""
    echo "❌ INVALID COMMIT MESSAGE!"
    echo ""
    echo "Your message:   \"$COMMIT_MSG\""
    echo ""
    echo "Expected format: type(scope): description"
    echo ""
    echo "Valid types:"
    echo "  feat:     A new feature"
    echo "  fix:      A bug fix"
    echo "  docs:     Documentation changes"
    echo "  style:    Code style (formatting, semicolons, etc.)"
    echo "  refactor: Code refactoring (no feature or bug fix)"
    echo "  perf:     Performance improvements"
    echo "  test:     Adding or updating tests"
    echo "  build:    Build system changes (npm, webpack, etc.)"
    echo "  ci:       CI/CD configuration changes"
    echo "  chore:    Other changes (updating dependencies, etc.)"
    echo "  revert:   Revert a previous commit"
    echo ""
    echo "Examples:"
    echo "  feat: add user authentication"
    echo "  fix(api): handle null response from payment gateway"
    echo "  docs: update installation guide"
    echo ""
    exit 1
fi

echo "✅ Commit message looks good!"
exit 0
```

## 18. pre-push Hook — Deep Dive

The `pre-push` hook runs **before pushing code** to a remote. If it fails, the push is aborted.

### Example 1: Run Tests Before Push

```bash
#!/bin/bash
# .git/hooks/pre-push
# Run tests before allowing a push

echo "🧪 Running tests before push..."

# Run the test suite
npm test

# Check if tests passed
if [ $? -ne 0 ]; then
    echo ""
    echo "❌ PUSH BLOCKED! Tests are failing."
    echo "Fix the failing tests before pushing."
    echo ""
    echo "To bypass (NOT recommended): git push --no-verify"
    exit 1
fi

echo "✅ All tests passed! Pushing..."
exit 0
```

### Example 2: Prevent Pushing to Main/Master

```bash
#!/bin/bash
# .git/hooks/pre-push
# Prevent direct pushes to main or master

PROTECTED_BRANCHES="^(main|master|production)$"
CURRENT_BRANCH=$(git symbolic-ref HEAD | sed -e 's,.*/\(.*\),\1,')

if echo "$CURRENT_BRANCH" | grep -qE "$PROTECTED_BRANCHES"; then
    echo ""
    echo "❌ PUSH BLOCKED!"
    echo ""
    echo "Direct push to '$CURRENT_BRANCH' is not allowed."
    echo "Please create a feature branch and submit a Pull Request."
    echo ""
    echo "Steps:"
    echo "  1. git checkout -b feature/my-feature"
    echo "  2. git push origin feature/my-feature"
    echo "  3. Create a Pull Request on GitHub/GitLab"
    echo ""
    exit 1
fi

exit 0
```

### Example 3: Check for Large Files Before Push

```bash
#!/bin/bash
# .git/hooks/pre-push
# Prevent pushing files larger than 5MB

MAX_SIZE=5242880  # 5MB in bytes

echo "📏 Checking for large files..."

# Get list of files being pushed
LARGE_FILES=$(git diff --stat --name-only @{u}..HEAD 2>/dev/null | while read FILE; do
    if [ -f "$FILE" ]; then
        SIZE=$(wc -c < "$FILE")
        if [ "$SIZE" -gt "$MAX_SIZE" ]; then
            echo "  $FILE ($(numfmt --to=iec $SIZE))"
        fi
    fi
done)

if [ -n "$LARGE_FILES" ]; then
    echo ""
    echo "❌ PUSH BLOCKED! Large files detected (>5MB):"
    echo "$LARGE_FILES"
    echo ""
    echo "Consider using Git LFS for large files:"
    echo "  git lfs track '*.psd'"
    echo "  git add .gitattributes"
    echo ""
    exit 1
fi

echo "✅ No large files found. Pushing..."
exit 0
```

## 19. Sharing Hooks with Your Team

**Problem:** Git hooks live in `.git/hooks/`, which is NOT committed to the repository. So how do you share hooks with your team?

### Solution 1: Custom Hooks Directory

```bash
# Create a hooks directory in your project
mkdir -p .githooks

# Move your hooks there
cp .git/hooks/pre-commit .githooks/pre-commit
cp .git/hooks/pre-push .githooks/pre-push

# Tell Git to use this directory (Git 2.9+)
git config core.hooksPath .githooks

# Commit the hooks
git add .githooks/
git commit -m "Add shared Git hooks"
```

Now add this to your README:

```markdown
## Setup

After cloning, configure Git hooks:

​```bash
git config core.hooksPath .githooks
​```
```

### Solution 2: Makefile Setup

```makefile
# Makefile
.PHONY: setup
setup:
	git config core.hooksPath .githooks
	chmod +x .githooks/*
	@echo "✅ Git hooks installed!"
```

Then new developers just run:

```bash
make setup
```

### Solution 3: Use a Tool (Husky, pre-commit)

This is the **recommended approach** for teams. See sections below.

## 20. Husky — The Modern Way to Manage Git Hooks

**[Husky](https://typicode.github.io/husky/)** is the most popular tool for managing Git hooks in **Node.js projects**. It makes hooks shareable, easy to configure, and version-controlled.

### Install Husky

```bash
# Install Husky as a dev dependency
npm install --save-dev husky

# Initialize Husky (creates .husky/ directory)
npx husky init
```

This creates:
```
.husky/
├── _/
│   └── husky.sh       # Husky internals
└── pre-commit         # Default pre-commit hook
```

And adds a `prepare` script to `package.json`:

```json
{
  "scripts": {
    "prepare": "husky"
  }
}
```

### Add a pre-commit Hook

```bash
# Create a pre-commit hook that runs the linter
echo "npx lint-staged" > .husky/pre-commit
```

### Add a commit-msg Hook

```bash
# Create a commit-msg hook that validates message format
echo 'npx --no -- commitlint --edit "$1"' > .husky/commit-msg
```

### Add a pre-push Hook

```bash
# Create a pre-push hook that runs tests
echo "npm test" > .husky/pre-push
```

### Husky + lint-staged (The Perfect Combo)

**lint-staged** runs linters **only on staged files** — not the entire project. This makes hooks fast!

```bash
# Install lint-staged
npm install --save-dev lint-staged
```

Add to `package.json`:

```json
{
  "lint-staged": {
    "*.{js,jsx,ts,tsx}": [
      "eslint --fix",
      "prettier --write"
    ],
    "*.{json,md,css,scss}": [
      "prettier --write"
    ]
  }
}
```

Now when you `git commit`:
1. Husky triggers the `pre-commit` hook
2. `lint-staged` runs ESLint and Prettier **only on the files you changed**
3. If there are errors, the commit is blocked
4. If everything passes, the commit goes through

### Husky + commitlint (Enforce Commit Messages)

```bash
# Install commitlint
npm install --save-dev @commitlint/cli @commitlint/config-conventional
```

Create `commitlint.config.js`:

```javascript
module.exports = {
  extends: ['@commitlint/config-conventional'],
  rules: {
    'type-enum': [2, 'always', [
      'feat', 'fix', 'docs', 'style', 'refactor',
      'perf', 'test', 'build', 'ci', 'chore', 'revert'
    ]],
    'subject-max-length': [2, 'always', 100],
  }
};
```

### Full Setup — Complete Example

```json
// package.json
{
  "name": "my-project",
  "scripts": {
    "prepare": "husky",
    "lint": "eslint src/",
    "format": "prettier --write src/",
    "test": "jest"
  },
  "devDependencies": {
    "husky": "^9.0.0",
    "lint-staged": "^15.0.0",
    "@commitlint/cli": "^19.0.0",
    "@commitlint/config-conventional": "^19.0.0",
    "eslint": "^9.0.0",
    "prettier": "^3.0.0"
  },
  "lint-staged": {
    "*.{js,jsx,ts,tsx}": ["eslint --fix", "prettier --write"],
    "*.{json,md,css}": ["prettier --write"]
  }
}
```

```

├── pre-commit       # Contains: npx lint-staged
├── commit-msg       # Contains: npx --no -- commitlint --edit "$1"
└── pre-push         # Contains: npm test

## 21. pre-commit Framework (Python)

For **Python projects** (or multi-language projects), the **[pre-commit](https://pre-commit.com/)** framework is the standard. It's language-agnostic and supports hundreds of ready-made hooks.

### Install

```bash
pip install pre-commit
```

### Configure

Create `.pre-commit-config.yaml` in your project root:

```yaml
# .pre-commit-config.yaml
repos:
  # General hooks
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v4.5.0
    hooks:
      - id: trailing-whitespace          # Remove trailing whitespace
      - id: end-of-file-fixer            # Ensure files end with newline
      - id: check-yaml                   # Validate YAML syntax
      - id: check-json                   # Validate JSON syntax
      - id: check-added-large-files      # Prevent large files (>500KB)
        args: ['--maxkb=500']
      - id: check-merge-conflict         # Check for merge conflict markers
      - id: detect-private-key           # Detect private keys

  # Python formatting
  - repo: https://github.com/psf/black
    rev: 24.3.0
    hooks:
      - id: black                        # Auto-format Python code

  # Python linting
  - repo: https://github.com/PyCQA/flake8
    rev: 7.0.0
    hooks:
      - id: flake8                       # Lint Python code

  # Python import sorting
  - repo: https://github.com/PyCQA/isort
    rev: 5.13.0
    hooks:
      - id: isort                        # Sort Python imports

  # Shell script linting
  - repo: https://github.com/shellcheck-py/shellcheck-py
    rev: v0.10.0.1
    hooks:
      - id: shellcheck                   # Lint shell scripts

  # Dockerfile linting
  - repo: https://github.com/hadolint/hadolint
    rev: v2.12.0
    hooks:
      - id: hadolint                     # Lint Dockerfiles

  # Terraform formatting
  - repo: https://github.com/antonbabenko/pre-commit-terraform
    rev: v1.86.0
    hooks:
      - id: terraform_fmt               # Format Terraform files
      - id: terraform_validate          # Validate Terraform
```

### Install the Hooks

```bash
# Install the hooks (downloads and sets up everything)
pre-commit install

# Also install commit-msg hook
pre-commit install --hook-type commit-msg
```

### Usage

```bash
# Hooks run automatically on git commit
git add .
git commit -m "feat: add new feature"
# Output:
# Trim Trailing Whitespace.........................................Passed
# Fix End of Files.................................................Passed
# Check Yaml......................................................Passed
# Check JSON......................................................Passed
# Check for added large files......................................Passed
# Detect Private Key..............................................Passed
# black...........................................................Passed
# flake8..........................................................Passed
# isort...........................................................Passed

# Run hooks manually on all files
pre-commit run --all-files

# Run a specific hook
pre-commit run black --all-files

# Update hooks to latest versions
pre-commit autoupdate

# Skip hooks temporarily (not recommended)
git commit --no-verify -m "emergency fix"
```

## 22. Common Mistakes & Troubleshooting

### Problem 1: ".gitignore isn't working!"

**Cause:** The file is already tracked by Git.

**Fix:**

```bash
git rm --cached filename
git commit -m "Stop tracking filename"
```

### Problem 2: "My Git hooks aren't running!"

**Check these things:**

```bash
# 1. Is the hook executable?
ls -la .git/hooks/pre-commit
# Should show: -rwxr-xr-x  (the 'x' means executable)

# Fix:
chmod +x .git/hooks/pre-commit

# 2. Does the hook have the correct name? (no .sample extension!)
ls .git/hooks/
# Should be: pre-commit  (NOT pre-commit.sample)

# 3. Does the hook have the correct shebang?
head -1 .git/hooks/pre-commit
# Should be: #!/bin/bash  (or #!/usr/bin/env python3, etc.)

# 4. Is core.hooksPath pointing elsewhere?
git config core.hooksPath
# If set, hooks are looked up from that directory instead of .git/hooks/
```

### Problem 3: "Line endings keep changing in my diffs!"

**Fix:**

```bash
# 1. Add .gitattributes
echo "* text=auto" > .gitattributes
git add .gitattributes
git commit -m "Add .gitattributes"

# 2. Renormalize everything
git add --renormalize .
git commit -m "Normalize line endings"
```

### Problem 4: "My hook works but my teammate's doesn't!"

**Cause:** Git hooks aren't shared by default.

**Fix:** Use Husky (Node.js) or pre-commit framework (Python) — see sections above.

### Problem 5: "Someone is bypassing hooks with --no-verify!"

**Solution:** Add **server-side hooks** or CI/CD checks. Client-side hooks can always be bypassed. Never rely on them as your only line of defense.

```yaml
# GitHub Actions — runs the same checks as your hooks, but on the server
name: Code Quality
on: [push, pull_request]
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run lint
      - run: npm test
```

### Key Takeaways

1. **`.gitignore`** keeps junk, secrets, and dependencies out of your repo.
2. **`.gitattributes`** prevents line-ending wars between Windows and Linux/macOS.
3. **Git hooks** automate quality checks — they're your last line of defense before bad code gets in.
4. Always have **both** client-side hooks AND server-side CI/CD checks — don't rely on just one.
5. Use **Husky** (Node.js) or **pre-commit** (Python) to share hooks with your team.
6. **Never commit secrets** — use `.gitignore`, environment variables, and secrets scanners.
