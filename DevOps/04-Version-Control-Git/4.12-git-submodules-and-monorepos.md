## 1. The Problem: Managing Multiple Repositories

As your project grows, you'll face a common challenge:

> "I have multiple projects (frontend, backend, shared libraries, infrastructure) that depend on each other. How do I organize them?"

### The Scenario

```
Company: SuperApp Inc.

Projects:
├── frontend (React app)
├── backend (Node.js API)
├── shared-utils (helper functions used by both)
├── mobile-app (React Native)
├── infrastructure (Terraform configs)
└── docs (Documentation site)
```

These projects **depend on each other**:
- Frontend imports functions from `shared-utils`
- Backend imports functions from `shared-utils`
- Mobile app uses the same API types as frontend
- Infrastructure deploys both frontend and backend

**How do you manage this?** You have three main options:

```
Option 1: Polyrepo      → Each project in its own repository
Option 2: Submodules     → Separate repos, but linked together
Option 3: Monorepo       → Everything in ONE repository
```

## 2. Two Approaches: Submodules vs Monorepos

Before diving deep, here's the quick comparison:

| Feature | Git Submodules | Monorepo |
|---------|---------------|----------|
| **Number of repos** | Multiple repos linked together | One single repo |
| **Complexity** | High (submodule commands are tricky) | Low (just folders) |
| **Dependency sharing** | Hard (must publish packages) | Easy (import directly) |
| **Versioning** | Each repo has own versions | Everything versioned together |
| **CI/CD** | Separate pipelines per repo | One pipeline (with selective builds) |
| **Team ownership** | Clear boundaries per repo | Must define code ownership |
| **Best for** | External libraries, vendor code | Internal projects that share code |

## 3. What Are Git Submodules?

A **Git submodule** is a way to include one Git repository inside another Git repository. The inner repository keeps its own history, branches, and commits.

Think of it like this:

> Imagine you're building a house (your main project). Instead of building the kitchen cabinets yourself, you order them from a cabinet company (an external library). The cabinets arrive as a pre-built unit — you install them in your house, but you didn't build them, and they have their own product version and history. **A submodule is like installing a pre-built component** inside your project.

### Visual

```
my-project/                 ← Main repository
├── src/
│   ├── app.js
│   └── config.js
├── libs/
│   └── shared-utils/       ← THIS IS A SUBMODULE (another repo!)
│       ├── .git/           ← Its own Git history
│       ├── index.js
│       └── helpers.js
├── .gitmodules             ← File that tracks submodule info
└── .git/                   ← Main project's Git history
```

The key point: `shared-utils/` is **not just a folder** — it's a **separate Git repository** embedded inside your project. It has its own commits, branches, and remote URL.

## 4. How Submodules Work Internally

When you add a submodule, Git creates two things:

### 1. The `.gitmodules` File

This file tracks which submodules exist and where they come from:

```ini
[submodule "libs/shared-utils"]
    path = libs/shared-utils
    url = https://github.com/company/shared-utils.git
    branch = main
```

### 2. A Special Entry in the Git Tree

Git doesn't store the submodule's files directly. Instead, it stores a **pointer** — just a commit hash.

```bash
# What Git actually stores for the submodule:
160000 commit abc123def456... libs/shared-utils
```

This means: "At this path, use commit `abc123def456` from the `shared-utils` repo."

### How It Looks Conceptually

```
Main Repo (my-project):
  commit: xyz789
  ├── src/app.js           → actual file content
  ├── src/config.js        → actual file content
  └── libs/shared-utils    → POINTER to commit abc123 in shared-utils repo

Submodule Repo (shared-utils):
  commit: abc123
  ├── index.js             → actual file content
  └── helpers.js           → actual file content
```

**Important:** The main repo doesn't track the submodule's files — it only tracks **which commit** to use. This is why submodules can feel confusing at first.

## 5. Adding a Submodule

### Basic Command

```bash
# Add a submodule
git submodule add <repository-url> <path>
```

### Step-by-Step Example

```bash
# Navigate to your main project
cd my-project

# Add the shared-utils repo as a submodule in the libs/ directory
git submodule add https://github.com/company/shared-utils.git libs/shared-utils
```

What just happened? Let's check:

```bash
# 1. A new .gitmodules file was created
cat .gitmodules
# [submodule "libs/shared-utils"]
#     path = libs/shared-utils
#     url = https://github.com/company/shared-utils.git

# 2. The submodule directory was created with the repo's content
ls libs/shared-utils/
# index.js  helpers.js  package.json  README.md

# 3. Git knows about the new submodule
git status
# new file:   .gitmodules
# new file:   libs/shared-utils

# 4. Commit the submodule addition
git add .gitmodules libs/shared-utils
git commit -m "Add shared-utils as a submodule"
```

### Specifying a Branch

```bash
# Add submodule and track a specific branch
git submodule add -b main https://github.com/company/shared-utils.git libs/shared-utils
```

### What the Main Repo Actually Stores

```bash
# Check what Git stores for the submodule
git ls-tree HEAD libs/

# Output:
# 160000 commit abc123def456789... libs/shared-utils
#   ↑                ↑
# special mode    commit hash from the submodule repo
```

`160000` is a special Git mode that means "this is a submodule, not a regular file or directory."

## 6. Cloning a Repo with Submodules

When someone clones a repo that has submodules, the submodule directories will be **empty** by default!

### The Wrong Way (Empty Submodule Folders)

```bash
# Clone the repo
git clone https://github.com/company/my-project.git
cd my-project

# Check the submodule directory
ls libs/shared-utils/
# (empty!) 😱
```

### The Right Way — Option 1: Clone with `--recurse-submodules`

```bash
# Clone and initialize submodules in one command
git clone --recurse-submodules https://github.com/company/my-project.git
cd my-project

# Check the submodule directory
ls libs/shared-utils/
# index.js  helpers.js  package.json  README.md  ✅
```

### The Right Way — Option 2: Initialize After Cloning

```bash
# Clone normally
git clone https://github.com/company/my-project.git
cd my-project

# Initialize and fetch all submodules
git submodule init
git submodule update

# Or combine both commands:
git submodule update --init

# For nested submodules (submodules inside submodules):
git submodule update --init --recursive
```

### Make It the Default (Recommended!)

Tired of forgetting `--recurse-submodules`? Make Git do it automatically:

```bash
# Always recurse submodules on clone
git config --global submodule.recurse true

# Now this works automatically:
git clone https://github.com/company/my-project.git
# Submodules are automatically initialized and fetched! ✅
```

## 7. Updating Submodules

### Scenario: The Submodule Got New Updates

The `shared-utils` team pushed new commits. You want those updates in your project.

### Update a Specific Submodule

```bash
# Go into the submodule directory
cd libs/shared-utils

# Pull the latest changes
git pull origin main

# Go back to the main project
cd ../..

# Check what changed
git status
# modified:   libs/shared-utils (new commits)

# The main project now knows the submodule has moved to a newer commit
git diff libs/shared-utils
# -Subproject commit abc123def456...
# +Subproject commit def789abc012...

# Commit the update
git add libs/shared-utils
git commit -m "Update shared-utils to latest version"
```

### Update All Submodules at Once

```bash
# Fetch and update all submodules to their latest commits
git submodule update --remote

# Or with merge
git submodule update --remote --merge

# Or with rebase
git submodule update --remote --rebase
```

### The Difference: `update` vs `update --remote`

| Command | What It Does |
|---------|-------------|
| `git submodule update` | Updates to the commit that the **main repo points to** (the pinned commit) |
| `git submodule update --remote` | Updates to the **latest commit** on the submodule's tracked branch |

Think of it this way:
- `update` = "Give me the version my project expects"
- `update --remote` = "Give me the newest version available"

```
Main repo says: "Use commit abc123 of shared-utils"

git submodule update          → Checks out abc123 (what the parent expects)
git submodule update --remote → Checks out xyz789 (the very latest commit)
```

## 8. Making Changes Inside a Submodule

Sometimes you need to fix something in the submodule and want the change to appear in both the submodule repo and your main project.

### Step-by-Step

```bash
# 1. Go into the submodule
cd libs/shared-utils

# 2. Make sure you're on a branch (submodules default to "detached HEAD"!)
git checkout main

# 3. Make your changes
echo "export function newHelper() { ... }" >> helpers.js

# 4. Commit inside the submodule
git add helpers.js
git commit -m "feat: add newHelper function"

# 5. Push the submodule's changes to its own repo
git push origin main

# 6. Go back to the main project
cd ../..

# 7. The main project detects the submodule has changed
git status
# modified:   libs/shared-utils (new commits)

# 8. Update the main project to point to the new submodule commit
git add libs/shared-utils
git commit -m "Update shared-utils with new helper function"

# 9. Push the main project
git push origin main
```

### Common Mistake: Detached HEAD in Submodules

When you `git submodule update`, the submodule is checked out at a **specific commit** — not on a branch. This means you're in "detached HEAD" state.

```bash
cd libs/shared-utils
git status
# HEAD detached at abc123  ← This means you're NOT on any branch!
```

**If you make commits in detached HEAD, you might lose them!** Always checkout a branch first:

```bash
cd libs/shared-utils
git checkout main          # ← Now you're on the main branch
# NOW make your changes
```

## 9. Removing a Submodule

Removing a submodule is surprisingly annoying. There's no single `git submodule remove` command (as of Git 2.x, `git rm` works, but here's the full process):

### The Modern Way (Git 2.12+)

```bash
# Remove the submodule
git rm libs/shared-utils

# Remove the submodule entry from .git/config
git config --remove-section submodule.libs/shared-utils 2>/dev/null

# Remove the cached submodule directory
rm -rf .git/modules/libs/shared-utils

# Commit the removal
git commit -m "Remove shared-utils submodule"
```

### The Manual Way (Older Git)

```bash
# 1. Remove from .gitmodules
git config -f .gitmodules --remove-section submodule.libs/shared-utils

# 2. Remove from .git/config
git config --remove-section submodule.libs/shared-utils

# 3. Remove the submodule directory from Git
git rm --cached libs/shared-utils

# 4. Remove the actual directory
rm -rf libs/shared-utils

# 5. Remove the submodule's Git data
rm -rf .git/modules/libs/shared-utils

# 6. Commit everything
git add .gitmodules
git commit -m "Remove shared-utils submodule"
```

## 11. When to Use Submodules (and When NOT To)

### Use Submodules When ✅

| Scenario | Example |
|----------|---------|
| **Using an external library** | Including a third-party SDK in your project |
| **Shared component across unrelated projects** | A design system used by 5 different apps |
| **Vendor code you don't control** | Company uses a fork of an open-source tool |
| **Different teams, different release cycles** | Backend team releases weekly, frontend releases daily |
| **Large binary assets separate from code** | Game assets in one repo, game code in another |

### Don't Use Submodules When ❌

| Scenario | Better Alternative |
|----------|-------------------|
| **Projects that are tightly coupled** | Use a monorepo |
| **Shared code between your own apps** | Use a monorepo with workspaces |
| **You want simple dependency management** | Use a package manager (npm, pip) |
| **Your team is small** | A monorepo is simpler to manage |
| **Projects that are always deployed together** | Use a monorepo |

### The Honest Truth About Submodules

Submodules have a reputation for being **confusing and painful**. Common complaints:

- "I cloned the repo but the submodule folder is empty!"
- "I'm in detached HEAD and lost my commits!"
- "I updated the submodule but my teammate didn't, so their build is broken!"
- "Removing a submodule requires 5 manual steps!"

If your projects are tightly coupled and developed by the same team, a **monorepo** is usually a better choice.

## 12. What is a Monorepo?

A **monorepo** (mono = one, repo = repository) is a strategy where you keep **all your related projects in a single Git repository**.

Think of it like this:

> Instead of each family member having their own house (polyrepo), everyone lives in **one big house** with separate rooms (monorepo). You share the kitchen, the living room, and the Wi-Fi — but everyone has their own bedroom.

### Monorepo vs Polyrepo Visual

```
POLYREPO (Multiple Repositories):
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│ frontend/    │  │ backend/     │  │ shared/      │
│  .git/       │  │  .git/       │  │  .git/       │
│  src/        │  │  src/        │  │  src/        │
│  package.json│  │  package.json│  │  package.json│
└──────────────┘  └──────────────┘  └──────────────┘
  3 separate repos, 3 separate Git histories

MONOREPO (Single Repository):
┌─────────────────────────────────────────────┐
│ my-company/                                  │
│  .git/          ← ONE Git history            │
│  packages/                                   │
│    frontend/                                 │
│      src/                                    │
│      package.json                            │
│    backend/                                  │
│      src/                                    │
│      package.json                            │
│    shared/                                   │
│      src/                                    │
│      package.json                            │
│  package.json   ← Root config                │
└─────────────────────────────────────────────┘
  1 repo, 1 Git history, multiple projects
```

## 13. Monorepo vs Polyrepo — The Great Debate

### Monorepo Advantages ✅

| Advantage | Explanation |
|-----------|-------------|
| **Shared code is easy** | Import directly between packages — no publishing or versioning |
| **Atomic changes** | One commit can change frontend, backend, AND shared library together |
| **Consistent tooling** | Same linter, formatter, and test runner across all projects |
| **Easier refactoring** | Rename a function in shared code and update all consumers in one PR |
| **Single CI/CD pipeline** | One place to define builds, tests, and deployments |
| **Better discoverability** | New devs can see all code in one place |
| **Simplified dependency management** | One lock file, consistent dependency versions |

### Monorepo Disadvantages ❌

| Disadvantage | Explanation |
|--------------|-------------|
| **Repo size grows** | Over time, the repo gets large (slow clones) |
| **Longer CI/CD runs** | Without optimization, every push triggers all builds |
| **Access control is harder** | Can't easily restrict access to specific folders |
| **Git performance degrades** | Large repos with long history become slow |
| **Tooling required** | You need tools like Nx, Turborepo, or Bazel to manage builds |
| **Merge conflicts increase** | More people working in the same repo = more conflicts |

### Polyrepo Advantages ✅

| Advantage | Explanation |
|-----------|-------------|
| **Independent deployments** | Each service deploys on its own schedule |
| **Clear ownership** | Each team owns their repo |
| **Smaller repos** | Fast clones, fast operations |
| **Fine-grained access** | Easy to restrict who can see/commit to each repo |
| **Technology freedom** | Each repo can use different languages/frameworks |

### Polyrepo Disadvantages ❌

| Disadvantage | Explanation |
|--------------|-------------|
| **Cross-repo changes are painful** | Changing shared code means PRs in multiple repos |
| **Dependency hell** | Version mismatches between repos |
| **Code duplication** | Same utility code copied across repos |
| **Inconsistent tooling** | Each repo might use different linting rules, test frameworks |
| **Discovery is hard** | "Where is this code?" — could be in any of 50 repos |

## 17. Nx — Monorepo Tool Deep Dive

**[Nx](https://nx.dev)** is one of the most powerful monorepo tools. It works with any language but is especially popular with JavaScript/TypeScript projects.

### Install Nx

```bash
# Create a new Nx workspace
npx create-nx-workspace@latest my-company

# You'll be asked:
# ? What to create in the new workspace: apps
# ? Application name: web
# ? Bundler to use: vite
# ? Default stylesheet format: css
# ? CI provider: github
```

### Key Nx Features

#### 1. Project Graph — Understand Dependencies

```bash
# See how your projects depend on each other
npx nx graph
```

This opens a visual dependency graph in your browser:

```
                    ┌─────────┐
                    │  shared  │
                    │  utils   │
                    └────┬─────┘
                    ┌────┴─────┐
              ┌─────┤          ├─────┐
              │     └──────────┘     │
        ┌─────▼───┐            ┌─────▼───┐
        │ frontend │            │ backend  │
        └─────────┘            └─────────┘
```

#### 2. Affected Commands — Only Build/Test What Changed

```bash
# Only run tests for projects affected by your changes
npx nx affected --target=test

# Only build projects affected by your changes
npx nx affected --target=build

# Only lint affected projects
npx nx affected --target=lint
```

Example: If you only changed `shared-utils`, Nx knows to test `shared-utils`, `frontend`, AND `backend` (because they depend on it), but NOT `mobile-app` (if it doesn't use shared-utils).

#### 3. Computation Caching — Never Build the Same Thing Twice

```bash
# First run: takes 30 seconds
npx nx build frontend
# > Building frontend... Done in 30s

# Second run (nothing changed): instant!
npx nx build frontend
# > Nx read the output from the cache instead of running the command for 1 out of 1 tasks.
# > Done in 0.5s  ← Cached! 🚀
```

#### 4. Remote Caching — Share Cache Across Team

```bash
# Connect to Nx Cloud for remote caching
npx nx connect
```

## 18. Turborepo — Fast Monorepo Builds

**[Turborepo](https://turbo.build)** (by Vercel) focuses on one thing: **making monorepo builds fast** through intelligent caching and parallel execution.

### Install Turborepo

```bash
# Create a new Turborepo workspace
npx create-turbo@latest my-company

# Or add to existing project
npm install turbo --save-dev
```

### Configuration: `turbo.json`

```json
{
  "$schema": "https://turbo.build/schema.json",
  "tasks": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": ["dist/**", ".next/**"]
    },
    "test": {
      "dependsOn": ["build"]
    },
    "lint": {},
    "dev": {
      "cache": false,
      "persistent": true
    }
  }
}
```

Let's break this down:

| Setting | Meaning |
|---------|---------|
| `"dependsOn": ["^build"]` | Build dependencies first (^ means "packages this depends on") |
| `"outputs": ["dist/**"]` | Cache these output folders |
| `"cache": false` | Don't cache `dev` tasks (they run continuously) |
| `"persistent": true` | Task runs indefinitely (dev servers) |

### Using Turborepo

```bash
# Build all packages (with caching and parallelism)
npx turbo build

# Run tests for all packages
npx turbo test

# Run dev servers for all packages
npx turbo dev

# Build only what changed (using filters)
npx turbo build --filter=frontend

# Build a package and its dependencies
npx turbo build --filter=frontend...

# Build only packages that changed since the last commit
npx turbo build --filter=[HEAD^1]
```

### Turbo Output Example

```bash
$ npx turbo build

• Packages in scope: @company/frontend, @company/backend, @company/shared-utils
• Running build in 3 packages
• Remote caching enabled

 @company/shared-utils:build: cache hit, replaying output ← Already built, using cache!
 @company/frontend:build: cache miss, executing          ← Changed, needs rebuild
 @company/backend:build: cache hit, replaying output     ← Already built, using cache!

 Tasks:    3 successful, 3 total
 Cached:   2 cached, 3 total
 Time:     3.2s  ← Only rebuilt what changed!
```

## 20. Monorepo with npm/yarn/pnpm Workspaces

You don't always need a fancy tool. **Package manager workspaces** give you basic monorepo support out of the box.

### npm Workspaces

Root `package.json`:

```json
{
  "name": "my-company",
  "private": true,
  "workspaces": [
    "packages/*",
    "apps/*"
  ]
}
```

```bash
# Install dependencies for all packages at once
npm install

# Run a script in a specific workspace
npm run build --workspace=packages/shared-utils
npm run build -w packages/shared-utils      # shorthand

# Run a script in all workspaces
npm run build --workspaces
npm run build -ws                            # shorthand

# Add a dependency to a specific workspace
npm install express --workspace=apps/backend

# Add a dependency to the root
npm install typescript --save-dev
```

### pnpm Workspaces (Recommended!)

pnpm is the fastest and most disk-efficient package manager. Create `pnpm-workspace.yaml`:

```yaml
# pnpm-workspace.yaml
packages:
  - 'packages/*'
  - 'apps/*'
```

```bash
# Install all dependencies
pnpm install

# Run a script in a specific package
pnpm --filter @company/frontend build

# Run in all packages
pnpm -r build

# Run only in packages that changed
pnpm --filter "...[HEAD~1]" build

# Add a dependency to a specific package
pnpm --filter @company/backend add express

# Add a shared dependency to the root
pnpm add -D typescript -w
```

### Referencing Internal Packages

In `apps/frontend/package.json`:

```json
{
  "name": "@company/frontend",
  "dependencies": {
    "@company/shared-utils": "workspace:*",
    "@company/shared-types": "workspace:*",
    "react": "^18.2.0"
  }
}
```

The `"workspace:*"` protocol tells the package manager: "Use the local version of this package, not one from npm."

Now in your frontend code, you can import directly:

```javascript
// apps/frontend/src/App.js
import { formatDate, capitalize } from '@company/shared-utils';
import type { User, Product } from '@company/shared-types';

function App() {
  const user = { name: capitalize('john'), joinedAt: formatDate(new Date()) };
  return <div>Welcome, {user.name}!</div>;
}
```

No publishing, no versioning — just import and use!

## 21. Monorepo CI/CD Strategies

The biggest challenge with monorepos is CI/CD. You don't want to build and deploy **everything** when only one package changed.

### Strategy 1: Affected/Changed Detection

Only build and test what's been affected by the current change:

```yaml
# GitHub Actions with Turborepo
name: CI
on: [push, pull_request]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0    # Need full history for change detection

      - uses: actions/setup-node@v4
        with:
          node-version: 20

      - run: npm ci

      # Only build/test affected packages
      - run: npx turbo build test lint --filter="...[origin/main]"
```

### Strategy 2: Path-Based Triggers

Run different workflows based on which files changed:

```yaml
# .github/workflows/frontend.yml
name: Frontend CI
on:
  push:
    paths:
      - 'apps/frontend/**'
      - 'packages/shared-utils/**'    # Also trigger if shared code changes
      - 'packages/shared-types/**'

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npx turbo build --filter=@company/frontend...
```

```yaml
# .github/workflows/backend.yml
name: Backend CI
on:
  push:
    paths:
      - 'apps/backend/**'
      - 'packages/shared-utils/**'

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npx turbo build test --filter=@company/backend...
```

### Strategy 3: Nx Affected in CI

```yaml
# GitHub Actions with Nx
name: CI
on: [push, pull_request]

jobs:
  main:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - uses: actions/setup-node@v4
        with:
          node-version: 20

      - run: npm ci

      # Set the base and head SHAs for affected detection
      - uses: nrwl/nx-set-shas@v4

      # Only run affected targets
      - run: npx nx affected --target=lint
      - run: npx nx affected --target=test
      - run: npx nx affected --target=build
```

## 22. Git Sparse Checkout (For Huge Monorepos)

If your monorepo is massive (thousands of projects), cloning the entire thing is slow. **Sparse checkout** lets you clone **only the folders you need**.

### How It Works

```bash
# 1. Clone without checking out files
git clone --no-checkout https://github.com/company/monorepo.git
cd monorepo

# 2. Enable sparse checkout
git sparse-checkout init --cone

# 3. Specify which folders you want
git sparse-checkout set apps/frontend packages/shared-utils packages/shared-types

# 4. Check out (only downloads the specified folders)
git checkout main
```

Now your working directory looks like:

```
monorepo/
├── apps/
│   └── frontend/        ✅ Downloaded
├── packages/
│   ├── shared-utils/    ✅ Downloaded
│   └── shared-types/    ✅ Downloaded
└── (other folders not downloaded — saves time and disk space!)
```

### Useful Commands

```bash
# See what's currently checked out
git sparse-checkout list

# Add another folder
git sparse-checkout add apps/backend

# Disable sparse checkout (get everything)
git sparse-checkout disable
```

### Combine with Shallow Clone for Maximum Speed

```bash
# Clone only recent history + only specific folders
git clone --depth=1 --no-checkout https://github.com/company/monorepo.git
cd monorepo
git sparse-checkout init --cone
git sparse-checkout set apps/frontend packages/shared-utils
git checkout main

# Result: Lightning-fast clone of a massive monorepo! ⚡
```

## 23. Git Subtree — Alternative to Submodules

**Git Subtree** is another way to include one repository inside another. Unlike submodules, it **merges the external repo's code directly into your repo's history**.

Think of it like this:

> **Submodule** = A framed photo on your wall (it's there but it's a separate thing with its own history).  
> **Subtree** = A photo printed directly onto your wallpaper (it's part of your wall now).

### Adding a Subtree

```bash
# Add an external repo as a subtree
git subtree add --prefix=libs/shared-utils \
  https://github.com/company/shared-utils.git main --squash
```

| Part | Meaning |
|------|---------|
| `--prefix=libs/shared-utils` | Where to put the code in your repo |
| `https://...` | URL of the external repo |
| `main` | Branch to pull from |
| `--squash` | Squash the external repo's history into one commit |

### Pulling Updates from the External Repo

```bash
git subtree pull --prefix=libs/shared-utils \
  https://github.com/company/shared-utils.git main --squash
```

### Pushing Changes Back to the External Repo

```bash
git subtree push --prefix=libs/shared-utils \
  https://github.com/company/shared-utils.git main
```

### Subtree vs Submodule

| Feature | Submodule | Subtree |
|---------|-----------|---------|
| **External repo visible?** | Yes (separate `.git`) | No (merged into main repo) |
| **Cloning complexity** | Must init/update submodules | Just `git clone` — everything is there |
| **History** | Separate (link to specific commit) | Merged into main repo's history |
| **Updating** | `git submodule update --remote` | `git subtree pull` |
| **Detached HEAD issues** | Yes (common complaint) | No |
| **Removing** | Complex (multiple steps) | Just delete the folder |
| **Repo size** | Smaller (pointer only) | Larger (contains all the code) |

---

## 24. Comparison: Submodules vs Subtree vs Monorepo

| Feature | Submodules | Subtree | Monorepo |
|---------|-----------|---------|----------|
| **Complexity** | High | Medium | Low |
| **External repos** | ✅ Best for this | ✅ Works | ❌ Not applicable |
| **Internal projects** | ❌ Overkill | ❌ Overkill | ✅ Best for this |
| **Shared code** | Hard | Medium | Easy |
| **Atomic changes** | ❌ Multiple commits | ❌ Multiple commits | ✅ One commit |
| **CI/CD** | Separate per repo | Complex | One pipeline |
| **Team confusion** | High | Medium | Low |
| **When to use** | External vendor code | Including OSS forks | Your own projects |

### Decision Flowchart

```
Do you need to include an EXTERNAL repository?
├── YES → Is it a repo you also contribute to?
│         ├── YES → Use Git Subtree
│         └── NO  → Use Git Submodule
│
└── NO  → Are the projects tightly coupled?
          ├── YES → Use a Monorepo
          └── NO  → Do they share code?
                    ├── YES → Use a Monorepo (or publish packages)
                    └── NO  → Use Polyrepo (separate repos)
```

## 25. Common Mistakes & Troubleshooting

### Problem 1: "Submodule folder is empty after clone!"

**Cause:** Submodules aren't initialized by default.

**Fix:**

```bash
git submodule update --init --recursive

# Or prevent this forever:
git config --global submodule.recurse true
```

### Problem 2: "I'm in detached HEAD inside a submodule!"

**Cause:** `git submodule update` checks out a specific commit, not a branch.

**Fix:**

```bash
cd libs/shared-utils
git checkout main        # Switch to a branch before making changes
```

### Problem 3: "My monorepo is too slow to clone!"

**Fix:** Use sparse checkout + shallow clone:

```bash
git clone --depth=1 --no-checkout https://github.com/company/monorepo.git
cd monorepo
git sparse-checkout init --cone
git sparse-checkout set apps/my-app packages/shared-utils
git checkout main
```

### Problem 4: "CI takes forever in our monorepo!"

**Fix:** Only build/test affected packages:

```bash
# Turborepo
npx turbo build test --filter="...[origin/main]"

# Nx
npx nx affected --target=build --target=test
```

### Problem 5: "Someone changed shared code and broke my app!"

**Fix:** Add proper CI checks:

```yaml
# When shared-utils changes, test ALL consumers
- run: npx turbo test --filter="...@company/shared-utils"
```

### Problem 6: "How do I manage different Node versions per package in a monorepo?"

**Fix:** Use a `.nvmrc` or `engines` field in each package:

```json
// packages/frontend/package.json
{
  "engines": {
    "node": ">=20.0.0"
  }
}
```

### Key Takeaways

1. **Git Submodules** include external repos inside your repo — useful for vendor code and shared libraries across organizations.
2. **Git Subtree** merges external repo code into your history — simpler than submodules, but increases repo size.
3. **Monorepos** put all related projects in one repo — best for tightly coupled internal projects.
4. **Use workspace tools** (pnpm, npm, yarn) for basic monorepo support.
5. **Use build tools** (Turborepo, Nx) for caching, affected detection, and parallel builds.
6. **Sparse checkout** lets you clone only the parts of a monorepo you need.
7. If your projects are tightly coupled and maintained by the same team, a **monorepo is almost always better** than submodules.
