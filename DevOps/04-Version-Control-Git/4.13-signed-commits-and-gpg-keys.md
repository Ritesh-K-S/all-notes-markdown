## 1. Why Do We Need Signed Commits?

Let me tell you a scary story.

Anyone in the world can make a Git commit that says it's from **you**. Seriously. Git doesn't verify identity — it just trusts whatever name and email you put in `git config`.

```bash
# Anyone can do this on their machine
git config user.name "Linus Torvalds"
git config user.email "torvalds@linux-foundation.org"

# Now every commit they make will LOOK like it came from Linus!
git commit -m "Totally legit commit from Linus"
```

**That commit will show up as if Linus Torvalds wrote it.** There's no warning, no error, no verification. Git just takes your word for it.

### Why is this dangerous?

| Scenario | Risk |
|----------|------|
| Someone impersonates a senior developer | Malicious code gets merged because it "looks legit" |
| Supply chain attack | Attacker pushes code pretending to be a trusted maintainer |
| Compliance & auditing | You can't prove who actually wrote the code |
| Open source projects | Anyone can submit PRs pretending to be someone else |
| Legal disputes | "I didn't write that code!" — but the commit says you did |

**Signed commits solve this.** They add a cryptographic proof that says: "This commit was REALLY made by this person, and the content hasn't been tampered with."

## 2. The Problem — Git Identity is Easy to Fake

Let's understand the problem more clearly.

### How Git identifies you (the default — insecure way):

```bash
git config --global user.name "Your Name"
git config --global user.email "your@email.com"
```

That's it. Git just stamps this name and email onto every commit. There is **zero verification**.

### A real attack scenario:

```
Step 1: Attacker finds out your name and email from your public GitHub profile
Step 2: Attacker sets their git config to match yours
Step 3: Attacker creates a malicious commit — it looks exactly like YOUR commit
Step 4: If they get access to push to a repo, the commit history shows YOU did it
```

### What signed commits add:

```
Regular commit:                          Signed commit:
┌──────────────────────┐                ┌──────────────────────┐
│ Author: ritesh        │                │ Author: ritesh        │
│ Email: ritesh@xyz.com │                │ Email: ritesh@xyz.com │
│ Message: fix bug      │                │ Message: fix bug      │
│                       │                │                       │
│ No verification ❌    │                │ GPG Signature: ✅     │
│                       │                │ ─────────────────     │
│ Anyone could have     │                │ Cryptographic proof   │
│ made this commit!     │                │ that ritesh's private │
│                       │                │ key signed this.      │
└──────────────────────┘                │                       │
                                         │ Tampering = detected! │
                                         └──────────────────────┘
```

## 3. What is GPG? — A Simple Explanation

**GPG** stands for **GNU Privacy Guard** (also called **GnuPG**). It's a free tool for encryption and digital signatures.

Think of GPG like a digital version of a wax seal on a letter:

```
Medieval times:                          Digital age (GPG):
┌─────────────────┐                     ┌─────────────────┐
│                  │                     │                  │
│  Dear King,      │                     │  git commit:     │
│  Send troops.    │                     │  "fix auth bug"  │
│                  │                     │                  │
│  - General Smith │                     │  - ritesh         │
│                  │                     │                  │
│  🔴 [WAX SEAL]  │                     │  🔐 [GPG SIG]   │
│                  │                     │                  │
│  • Only General  │                     │  • Only ritesh's  │
│    Smith has the │                     │    private key    │
│    seal stamp    │                     │    can create it  │
│  • If broken,   │                     │  • If commit is   │
│    letter was    │                     │    modified, sig  │
│    tampered with │                     │    becomes invalid│
└─────────────────┘                     └─────────────────┘
```

### GPG uses a **key pair**:

| Key | What It Is | Who Has It | What It Does |
|-----|-----------|------------|-------------|
| **Private Key** | Your secret key | Only YOU | Signs (creates) the signature |
| **Public Key** | Shared openly | Everyone | Verifies the signature is real |

### How signing works (simplified):

```
1. You write a commit
2. GPG takes the commit content + your private key
3. GPG creates a unique digital signature
4. The signature is attached to the commit
5. Anyone with your public key can verify it

         commit content
              │
              ▼
    ┌─────────────────┐
    │   GPG Engine     │ ◄── Your Private Key (secret!)
    │                  │
    │   Creates a      │
    │   unique digital │
    │   signature      │
    └────────┬─────────┘
             │
             ▼
      Digital Signature
   (attached to commit)
```

```
│     └──────────┬───────────────┘                         │
│                │                                          │
│                ▼                                          │
│     ┌──────────────────────────┐                         │
│     │ GPG signs it with your   │                         │
│     │ PRIVATE key              │                         │
│     └──────────┬───────────────┘                         │
│                │                                          │
│                ▼                                          │
│     ┌──────────────────────────┐                         │
│     │ Signed commit stored     │                         │
```

## 6. Generating Your GPG Key Pair

Now let's create your personal key pair. This is a one-time setup.

### Step 1: Generate the key

```bash
gpg --full-generate-key
```

You'll be asked several questions. Here's what to choose:

```
Please select what kind of key you want:
   (1) RSA and RSA
   (2) DSA and Elgamal
   (3) DSA (sign only)
   (4) RSA (sign only)
   (9) ECC (sign and encrypt) *default*
  (10) ECC (sign only)

Your selection? 1          # ← Choose RSA and RSA (most compatible)
```

```
RSA keys may be between 1024 and 4096 bits long.
What keysize do you want? (3072) 4096    # ← Choose 4096 for maximum security
```

```
Please specify how long the key should be valid.
         0 = key does not expire
      <n>  = key expires in n days
      <n>w = key expires in n weeks
      <n>m = key expires in n months
      <n>y = key expires in n years

Key is valid for? (0) 2y    # ← 2 years is a good practice (you can extend later)
```

```
Is this correct? (y/N) y
```

```
Real name: Your Full Name          # ← Must match your Git config name
Email address: your@email.com      # ← Must match your Git config email  
Comment:                            # ← Can leave empty, press Enter

You selected this USER-ID:
    "Your Full Name <your@email.com>"

Change (N)ame, (C)omment, (E)mail or (O)kay/(Q)uit? O
```

You'll then be asked to set a **passphrase**. This is the password that protects your private key. **Choose a strong one and remember it!**

```
┌──────────────────────────────────────────────┐
│ Please enter the passphrase to               │
│ protect your new key                         │
│                                               │
│ Passphrase: ******************************** │
│                                               │
│ Repeat:     ******************************** │
└──────────────────────────────────────────────┘
```

After this, GPG generates your key pair. It might take a few seconds.

```
gpg: key 3AA5C34371567BD2 marked as ultimately trusted
gpg: revocation certificate stored as '/home/user/.gnupg/openpgp-revocs.d/...'
public and secret key created and signed.

pub   rsa4096 2026-05-27 [SC] [expires: 2028-05-27]
      3AA5C34371567BD2F9B1234567890ABCDEF12345
uid                      Your Full Name <your@email.com>
sub   rsa4096 2026-05-27 [E] [expires: 2028-05-27]
```

> **Important:** The long hex string `3AA5C34371567BD2F9B1234567890ABCDEF12345` is your key's **fingerprint**. The last 16 characters are your **Key ID**.

### Quick method (non-interactive):

```bash
# Generate a key with defaults in one shot
gpg --batch --generate-key <<EOF
Key-Type: RSA
Key-Length: 4096
Subkey-Type: RSA
Subkey-Length: 4096
Name-Real: Your Full Name
Name-Email: your@email.com
Expire-Date: 2y
%commit
EOF
```

## 7. Listing and Managing Your GPG Keys

### List your keys:

```bash
# List public keys
gpg --list-keys
# OR
gpg -k

# List secret (private) keys
gpg --list-secret-keys
# OR
gpg -K

# List keys with their Key IDs in long format (this is what Git needs)
gpg --list-secret-keys --keyid-format=long
```

### Understanding the output:

```bash
$ gpg --list-secret-keys --keyid-format=long

/home/user/.gnupg/pubring.kbx
------------------------------
sec   rsa4096/3AA5C34371567BD2 2026-05-27 [SC] [expires: 2028-05-27]
      3AA5C34371567BD2F9B1234567890ABCDEF12345
uid                 [ultimate] Your Full Name <your@email.com>
ssb   rsa4096/1234567890ABCDEF 2026-05-27 [E] [expires: 2028-05-27]
```

Let's break this down:

```
sec   → "secret" — this is your PRIVATE key
rsa4096 → Algorithm and key size
/3AA5C34371567BD2 → Your KEY ID (this is what you'll tell Git to use!)
2026-05-27 → Date created
[SC] → Can be used for Signing and Certification
[expires: 2028-05-27] → Expiry date

uid → User ID (your name and email)
[ultimate] → Trust level (you trust yourself completely!)

ssb → "secret sub-key" — used for encryption
[E] → Can be used for Encryption
```

### Export your public key (you'll need this for GitHub/GitLab):

```bash
# Replace KEY_ID with your actual key ID
gpg --armor --export 3AA5C34371567BD2
```

This outputs your public key in ASCII format — looks like:

```
-----BEGIN PGP PUBLIC KEY BLOCK-----

mQINBGZ...
...lots of characters...
...
-----END PGP PUBLIC KEY BLOCK-----
```

> Copy this entire block (including the `BEGIN` and `END` lines). You'll paste it into GitHub/GitLab.

### Delete a key (be careful!):

```bash
# Delete a public key
gpg --delete-key KEY_ID

# Delete a private key (you MUST delete private key first if both exist)
gpg --delete-secret-key KEY_ID
```

### Export / Backup your private key (keep this SAFE!):

```bash
# Export private key to a file
gpg --armor --export-secret-keys 3AA5C34371567BD2 > my-private-key.asc

# Import it back (on another machine, for example)
gpg --import my-private-key.asc
```

> ⚠️ **NEVER share your private key.** Never commit it to Git. Never paste it online. Treat it like a password.

## 8. Configuring Git to Use Your GPG Key

Now let's tell Git to use your GPG key for signing.

### Step 1: Get your Key ID

```bash
gpg --list-secret-keys --keyid-format=long
# Look for the line that starts with "sec" and find the key ID after the "/"
# Example: sec   rsa4096/3AA5C34371567BD2 → Key ID is 3AA5C34371567BD2
```

### Step 2: Tell Git which key to use

```bash
# Set your signing key
git config --global user.signingkey 3AA5C34371567BD2
```

### Step 3: (Optional) Auto-sign ALL commits

```bash
# If you want EVERY commit to be automatically signed (recommended!)
git config --global commit.gpgsign true

# And every tag too
git config --global tag.gpgsign true
```

If you set `commit.gpgsign = true`, you don't need to manually add `-S` to every commit. Every `git commit` will be automatically signed.

### Step 4: Tell Git where GPG is installed (Windows users especially)

```bash
# On Windows, if Git can't find GPG:
git config --global gpg.program "C:/Program Files/Git/usr/bin/gpg.exe"

# Or if you installed Gpg4win:
git config --global gpg.program "C:/Program Files (x86)/GnuPG/bin/gpg.exe"
```

### Step 5: (macOS) Configure pinentry for passphrase prompt

```bash
# Add this to ~/.gnupg/gpg-agent.conf
echo "pinentry-program /opt/homebrew/bin/pinentry-mac" >> ~/.gnupg/gpg-agent.conf

# Restart the GPG agent
gpgconf --kill gpg-agent
```

### Verify your Git config:

```bash
git config --global --list | grep -i gpg
# Should show:
# user.signingkey=3AA5C34371567BD2
# commit.gpgsign=true
# tag.gpgsign=true
```

## 9. Signing Commits

Now the fun part — let's actually sign commits!

### Method 1: Sign a single commit with `-S` flag

```bash
# The -S flag (capital S) tells Git to sign this commit
git commit -S -m "feat: add user authentication"
```

Git will ask for your GPG passphrase (the one you set when creating the key). Enter it, and the commit is signed!

### Method 2: Auto-sign all commits (if configured)

```bash
# If you set commit.gpgsign = true, just commit normally
git commit -m "feat: add user authentication"
# It's automatically signed!
```

### Check if a commit is signed:

```bash
# Show the signature of the last commit
git log --show-signature -1
```

Output for a signed commit:

```
commit abc123def456... (HEAD -> main)
gpg: Signature made Tue May 27 2026 10:30:00 AM IST
gpg:                using RSA key 3AA5C34371567BD2F9B1234567890ABCDEF12345
gpg: Good signature from "Your Full Name <your@email.com>" [ultimate]
Author: Your Full Name <your@email.com>
Date:   Tue May 27 10:30:00 2026 +0530

    feat: add user authentication
```

### Show signatures in log format:

```bash
# Show signatures for all commits
git log --show-signature

# Show a compact format with signature status
git log --pretty="format:%h %G? %aN %s"
```

The `%G?` format shows signature status:

| Symbol | Meaning |
|--------|---------|
| `G` | Good (valid) signature |
| `B` | Bad signature |
| `U` | Good signature but unknown trust |
| `X` | Good signature that has expired |
| `Y` | Good signature made by an expired key |
| `R` | Good signature made by a revoked key |
| `E` | Signature cannot be checked (missing key) |
| `N` | No signature |

### Amend a commit with a signature:

```bash
# If you forgot to sign a commit, amend it
git commit --amend -S --no-edit
```

### Sign multiple past commits (rebase):

```bash
# Re-sign the last 5 commits
git rebase --exec 'git commit --amend --no-edit -n -S' -i HEAD~5
```

> ⚠️ **Warning:** This rewrites commit history! Only do this on branches that haven't been pushed, or coordinate with your team.

## 10. Signing Tags

Tags can also be signed, and in fact, **signing tags is even more important than signing commits** for releases.

### Create a signed tag:

```bash
# -s flag (lowercase s) creates a signed tag
git tag -s v1.0.0 -m "Release version 1.0.0"
```

### Verify a signed tag:

```bash
git tag -v v1.0.0
```

Output:

```
object abc123...
type commit
tag v1.0.0
tagger Your Full Name <your@email.com> 1716789000 +0530

Release version 1.0.0
gpg: Signature made Tue May 27 2026 10:30:00 AM IST
gpg:                using RSA key 3AA5C34371567BD2
gpg: Good signature from "Your Full Name <your@email.com>" [ultimate]
```

### Why sign tags?

When you release software (like v2.1.0), a signed tag proves that the release was actually created by a trusted maintainer — not someone who hacked into the repo and created a fake release with malicious code.

```
Unsigned tag:                            Signed tag:
┌──────────────────┐                    ┌──────────────────┐
│ Tag: v1.0.0      │                    │ Tag: v1.0.0      │
│ By: maintainer   │                    │ By: maintainer   │
│                   │                    │                   │
│ Could be fake! ❌│                    │ GPG Signature ✅ │
│ Anyone with push │                    │ Cryptographically │
│ access could     │                    │ proven to be from │
│ have created it  │                    │ the real          │
│                   │                    │ maintainer        │
└──────────────────┘                    └──────────────────┘
```

## 11. Verifying Signed Commits and Tags

### Verify a specific commit:

```bash
# Verify the last commit
git verify-commit HEAD

# Verify a specific commit by hash
git verify-commit abc123def

# Verify and show raw signature data
git verify-commit --raw HEAD
```

### Verify a merge commit:

```bash
# When merging, require that the commits being merged are signed
git merge --verify-signatures feature-branch
```

If any commit in the feature branch is unsigned or has an invalid signature, the merge is **rejected**:

```
fatal: Commit abc123 does not have a GPG signature.
```

This is a powerful way to enforce signing in your workflow!

### Verify a tag:

```bash
git verify-tag v1.0.0
```

### Verify during pull:

```bash
# Verify signatures when pulling
git pull --verify-signatures
```

```bash

gpg --armor --export 3AA5C34371567BD2 | xclip -selection clipboard
```

```
1. Go to GitHub.com → Settings (click your profile picture → Settings)
2. In the left sidebar, click "SSH and GPG keys"
3. Click "New GPG key"
4. Paste your entire public key (including the BEGIN/END lines)
5. Click "Add GPG key"
```

```

2. Scroll down to "Vigilant mode"
3. Check "Flag unsigned commits as unverified"
```

With vigilant mode, **all** your unsigned commits will be marked as "Unverified" — this adds peer pressure to always sign your commits!

## 14. The "Verified" Badge — What It Means

When you push a signed commit to GitHub/GitLab, you'll see badges:

```
GitHub commit list:
┌─────────────────────────────────────────────────────────────┐
│ ┌──────────┐                                                │
│ │ Verified │  fix: resolve auth bug (#234)     abc123       │
│ │   ✅     │  ritesh committed 2 hours ago                  │
│ └──────────┘                                                │
│                                                              │
│ ┌──────────────┐                                            │
│ │ Unverified   │  update readme                 def456       │
│ │      ⚠️      │  someone committed 3 hours ago             │
│ └──────────────┘                                            │
│                                                              │
│                  chore: update deps             ghi789       │
│                  someone committed 5 hours ago               │
│                  (No badge — unsigned)                       │
└─────────────────────────────────────────────────────────────┘
```

### What do the badges mean?

| Badge | Meaning |
|-------|---------|
| **Verified** ✅ | The commit is signed AND the signature matches a GPG key uploaded to this account |
| **Partially Verified** | The commit is signed but there's an issue (e.g., key expired) |
| **Unverified** ⚠️ | The commit is signed but the key doesn't match any known key |
| No badge | The commit is not signed at all |

### When does a commit show "Verified"?

All of these must be true:
1. The commit is signed with a GPG key
2. The GPG key's email matches a verified email on your GitHub/GitLab account
3. The public key is uploaded to your GitHub/GitLab account
4. The key is not expired or revoked

## 15. Signing with SSH Keys (Modern Alternative)

Starting from **Git 2.34+**, you can use SSH keys for signing instead of GPG! This is simpler because most developers already have SSH keys.

### Why SSH signing?

| Feature | GPG Signing | SSH Signing |
|---------|------------|-------------|
| Setup complexity | Medium-High | Low |
| Key management | GPG keyring | Just SSH keys |
| Already have keys? | Probably not | Probably yes! |
| GitHub support | ✅ Yes | ✅ Yes (since 2022) |
| GitLab support | ✅ Yes | ✅ Yes |
| Key expiry | Built-in | Manual |
| Web of Trust | Yes | No |
| Best for | Enterprise, open source | Individual devs, teams |

### How to set up SSH signing:

```bash
# Step 1: Tell Git to use SSH for signing
git config --global gpg.format ssh

# Step 2: Tell Git which SSH key to use
git config --global user.signingkey ~/.ssh/id_ed25519.pub

# Step 3: Enable auto-signing (optional)
git config --global commit.gpgsign true
git config --global tag.gpgsign true
```

### Create a commit with SSH signing:

```bash
git commit -S -m "feat: add dashboard page"
```

### Verify SSH-signed commits locally:

You need an **allowed signers file** — this is a list of trusted SSH public keys.

```bash
# Create the allowed signers file
echo "your@email.com $(cat ~/.ssh/id_ed25519.pub)" > ~/.config/git/allowed_signers

# Tell Git where to find it
git config --global gpg.ssh.allowedSignersFile ~/.config/git/allowed_signers
```

Now you can verify:

```bash
git log --show-signature -1
```

### Add SSH signing key to GitHub:

```
1. Go to Settings → SSH and GPG keys
2. Click "New SSH key"
3. Change the "Key type" dropdown to "Signing key" (not Authentication!)
4. Paste your public SSH key
5. Click "Add SSH key"
```

> You can use the **same SSH key** for both authentication AND signing, or use separate keys. GitHub lets you choose.

## 16. S/MIME Signing — The Enterprise Option

For completeness, there's a third option: **S/MIME** (using X.509 certificates).

This is mainly used in **enterprise environments** where companies have their own Certificate Authority (CA) and issue certificates to employees.

```bash
# Configure Git to use S/MIME
git config --global gpg.format x509

# Set your certificate
git config --global user.signingkey <your-certificate-id>
```

| Feature | GPG | SSH | S/MIME |
|---------|-----|-----|--------|
| Best for | Open source, general use | Simple setup | Enterprise |
| Trust model | Web of Trust | Allowed signers file | Certificate Authority |
| Key management | GPG keyring | SSH keys | X.509 certificates |
| Complexity | Medium | Low | High |
| GitHub support | ✅ | ✅ | ✅ |

**For most DevOps engineers, GPG or SSH signing is the way to go.** S/MIME is only needed in specific corporate environments.

## 17. Enforcing Signed Commits in a Team

### On GitHub — Branch Protection Rules:

```
1. Go to your repo → Settings → Branches
2. Click "Add rule" next to Branch protection rules
3. Set the branch name pattern (e.g., "main")
4. Check "Require signed commits"
5. Click "Create" or "Save changes"
```

Now, any unsigned commit pushed to `main` will be **rejected**!

```bash
# This will be rejected if the commit isn't signed:
git push origin main
# Error: remote: Commits must be signed.
```

### On GitLab — Push Rules:

```
1. Go to your project → Settings → Repository
2. Expand "Push Rules"
3. Check "Reject unsigned commits"
4. Click "Save Push Rules"
```

### In a CI/CD Pipeline:

You can also verify signatures in your CI pipeline:

```yaml
# .github/workflows/verify-signatures.yml
name: Verify Commit Signatures

on: [pull_request]

jobs:
  verify:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0    # Need full history for verification

      - name: Verify all commits are signed
        run: |
          # Check each commit in the PR
          for commit in $(git rev-list origin/main..HEAD); do
            if ! git verify-commit "$commit" 2>/dev/null; then
              echo "❌ Commit $commit is not signed!"
              exit 1
            fi
          done
          echo "✅ All commits are signed!"
```

### Team GPG key server (for verifying each other's signatures):

```bash
# Share public keys within the team
# Export your key
gpg --armor --export your@email.com > your-public-key.asc

# Teammates import your key
gpg --import your-public-key.asc

# After importing, they can verify your commits locally
git log --show-signature
```

## 18. GPG Key Best Practices

### DO:

| Practice | Why |
|----------|-----|
| Use RSA 4096-bit or ED25519 | Strong security |
| Set an expiry date (1-2 years) | Limits damage if key is compromised |
| Use a strong passphrase | Protects the private key |
| Backup your private key securely | Don't lose access to your signing ability |
| Use a separate signing subkey | If compromised, you can revoke just the subkey |
| Generate a revocation certificate | In case your key is compromised |
| Match email with Git config & GitHub | Otherwise, "Verified" badge won't show |

### DON'T:

| Practice | Why |
|----------|-----|
| Don't share your private key | Defeats the entire purpose |
| Don't use 1024-bit keys | Too weak, can be cracked |
| Don't skip the passphrase | If someone steals your key file, they can sign as you |
| Don't commit your private key | Obvious, but people do it! |
| Don't use the same key forever | Keys should be rotated periodically |

### Generate a revocation certificate (do this immediately after creating your key!):

```bash
gpg --gen-revoke 3AA5C34371567BD2 > revocation-certificate.asc
```

Store this file safely (e.g., on a USB drive in a safe). If your key is ever compromised, you can use this certificate to tell the world "this key is no longer valid."

### Extend an expiring key:

```bash
# Edit the key
gpg --edit-key 3AA5C34371567BD2

# In the GPG prompt:
gpg> expire
# Choose new expiry date
# Confirm

gpg> save
```

Then re-upload the updated public key to GitHub/GitLab.

## 19. Troubleshooting Common Issues

### Problem: "error: gpg failed to sign the data"

This is the most common issue. Here are the fixes:

```bash
# Fix 1: Make sure GPG agent is running
gpgconf --kill gpg-agent
gpg-agent --daemon

# Fix 2: Set the GPG_TTY environment variable
export GPG_TTY=$(tty)
# Add this to your ~/.bashrc or ~/.zshrc to make it permanent:
echo 'export GPG_TTY=$(tty)' >> ~/.bashrc

# Fix 3: Test if GPG itself is working
echo "test" | gpg --clearsign
# If this fails, GPG itself has an issue

# Fix 4: Make sure Git is using the right GPG
git config --global gpg.program $(which gpg)

# Fix 5: On WSL2, you might need:
echo "pinentry-program /usr/bin/pinentry-tty" >> ~/.gnupg/gpg-agent.conf
gpgconf --kill gpg-agent
```

### Problem: Commit shows as "Unverified" on GitHub

```bash
# Check 1: Does the key email match your GitHub email?
gpg --list-keys
git config user.email
# These MUST match, AND the email must be verified on GitHub

# Check 2: Did you upload the PUBLIC key (not private) to GitHub?

# Check 3: Is the key expired?
gpg --list-keys
# Look for [expires: ...] — if it's past, extend it

# Check 4: Are you using the right key ID in Git config?
git config user.signingkey
gpg --list-secret-keys --keyid-format=long
# Make sure they match
```

### Problem: Passphrase prompt doesn't appear (hangs)

```bash
# On macOS:
brew install pinentry-mac
echo "pinentry-program /opt/homebrew/bin/pinentry-mac" > ~/.gnupg/gpg-agent.conf
gpgconf --kill gpg-agent

# On Linux (headless/SSH session):
echo "pinentry-program /usr/bin/pinentry-tty" > ~/.gnupg/gpg-agent.conf
export GPG_TTY=$(tty)
gpgconf --kill gpg-agent

# On Windows:
# Make sure Git is using the right gpg.exe
git config --global gpg.program "C:/Program Files/Git/usr/bin/gpg.exe"
```

### Problem: "No secret key" error

```bash
# Make sure your private key exists
gpg --list-secret-keys

# If empty, you need to import or generate a key
# If you have a backup:
gpg --import my-private-key.asc
```

## 20. Real-World Scenarios — When Signing Matters

### Scenario 1: Open Source Software Supply Chain

```
Linux kernel, Kubernetes, Node.js — all major open source projects
require signed commits/tags from maintainers.

Why? Because a single malicious commit in these projects could
affect MILLIONS of servers worldwide.

The SolarWinds hack (2020) showed what happens when the software
supply chain is compromised — 18,000+ organizations were affected.
```

### Scenario 2: Compliance & Regulatory Requirements

```
Industries like finance, healthcare, and government often require:
- Traceability: WHO made each change
- Non-repudiation: Proof that the person can't deny making the change
- Integrity: Proof that the code hasn't been tampered with

Signed commits satisfy all three requirements.
```

### Scenario 3: Automated Deployments

```
If your CI/CD pipeline automatically deploys code that's pushed to main,
you REALLY want to verify that the code is actually from a trusted developer.

Without signing: Someone compromises a developer's account → pushes malicious
                 code → pipeline deploys it → production is compromised

With signing: Even if the account is compromised, the attacker can't sign
              commits (they don't have the GPG private key) → pipeline
              rejects unsigned commits → production is safe
```
