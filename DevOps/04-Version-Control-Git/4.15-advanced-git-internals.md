## 1. Why Learn Git Internals?

You might be thinking: *"I already know `git add`, `git commit`, `git push` — why should I care what's happening under the hood?"*

Great question! Here's why:

| Reason | Example |
|--------|---------|
| **Fix broken repos** | "My repo is corrupted — how do I recover?" |
| **Understand weird behaviors** | "Why does `git status` show changes I didn't make?" |
| **Performance tuning** | "My repo is 10 GB — how do I shrink it?" |
| **Advanced operations** | "How do I remove a file from ALL of Git history?" |
| **Interview questions** | "How does Git store data internally?" |
| **Deeper confidence** | Know what `git reset`, `rebase`, `cherry-pick` REALLY do |

Think of it like driving a car: you can drive without knowing how the engine works, but when something breaks down, the mechanic who understands the engine can fix it. **You want to be the mechanic.**

## 2. Git is a Content-Addressable File System

This is the most important thing to understand about Git:

> **Git is NOT a version control system at its core. It's a content-addressable file system with a version control system built on top.**

What does "content-addressable" mean?

```
Regular file system:                    Content-addressable (Git):
─────────────────────                   ─────────────────────────
You find files by NAME:                 You find files by CONTENT:

/documents/report.txt                   SHA-1 hash of the content
/photos/vacation.jpg                    = unique ID for that content

If two files have the same              If two files have the same
name, confusion!                        CONTENT, they get the SAME hash
                                        = only stored ONCE! (efficient!)
```

### Simple analogy:

```
Regular library:                        Git library:
────────────────                        ─────────────
Books organized by shelf location:      Books organized by fingerprint:

"Go to shelf 3, row 2"                 "Content fingerprint: abc123"

If someone moves the book,              The fingerprint never changes.
you can't find it!                      You can ALWAYS find the book
                                        by its content fingerprint.
```

Every piece of data in Git is stored and looked up by its **SHA-1 hash** — a 40-character hexadecimal string that is uniquely determined by the content.

```bash
# Example: Git hashes the content "Hello World"
echo "Hello World" | git hash-object --stdin
# Output: 557db03de997c86a4a028e1ebd3a1ceb225be238

# The hash is ALWAYS the same for the same content, on any computer, forever!
```

## 3. Inside the .git Directory

When you run `git init`, Git creates a hidden `.git` directory. **This IS your repository.** Everything Git knows about your project lives here.

```bash
# Create a new repo and look inside
git init my-project
cd my-project
ls -la .git/
```

```
.git/
├── HEAD                 # Points to the current branch
├── config               # Repository-specific configuration
├── description          # Used by GitWeb (rarely used)
├── hooks/               # Client-side and server-side scripts
│   ├── pre-commit.sample
│   ├── pre-push.sample
│   └── ...
├── info/
│   └── exclude          # Like .gitignore but not committed
├── objects/             # THE OBJECT DATABASE (all your data!)
│   ├── info/
│   └── pack/            # Packed objects (compressed)
├── refs/                # REFERENCES (branches, tags)
│   ├── heads/           # Branch pointers
│   └── tags/            # Tag pointers
├── index                # The staging area (binary file)
├── logs/                # Reflog history
│   ├── HEAD
│   └── refs/
└── packed-refs          # Packed references (for performance)
```

Let's understand each critical part:

```
┌──────────────────────────────────────────────────────────────┐
│                    THE .git DIRECTORY                          │
│                                                               │
│  ┌────────────┐    "Where am I?"                             │
│  │    HEAD     │    Points to current branch                 │
│  └────────────┘    (usually refs/heads/main)                 │
│                                                               │
│  ┌────────────┐    "Where are the branches?"                 │
│  │   refs/     │    Branch and tag pointers                   │
│  │  heads/     │    heads/main → commit SHA                  │
│  │  tags/      │    tags/v1.0 → commit/tag SHA               │
│  └────────────┘                                               │
│                                                               │
│  ┌────────────┐    "Where is ALL the data?"                  │
│  │  objects/   │    Every blob, tree, commit, tag             │
│  │   ab/       │    stored as files named by SHA-1            │
│  │   cd/       │    (first 2 chars = directory)              │
│  │   pack/     │    packed versions for efficiency            │
│  └────────────┘                                               │
│                                                               │
│  ┌────────────┐    "What's staged for commit?"               │
│  │   index     │    Binary file listing staged files          │
│  └────────────┘                                               │
└──────────────────────────────────────────────────────────────┘
```

## 4. The Four Types of Git Objects

**Everything** in Git is stored as one of four object types:

```
┌──────────────────────────────────────────────────────────────┐
│                   GIT OBJECT TYPES                            │
│                                                               │
│  ┌──────────┐  BLOB                                          │
│  │  blob    │  = File content (just the data, no filename!)  │
│  └──────────┘                                                 │
│                                                               │
│  ┌──────────┐  TREE                                          │
│  │  tree    │  = Directory listing (filenames + blob refs)   │
│  └──────────┘                                                 │
│                                                               │
│  ┌──────────┐  COMMIT                                        │
│  │  commit  │  = Snapshot (points to tree + metadata)        │
│  └──────────┘                                                 │
│                                                               │
│  ┌──────────┐  TAG (annotated)                               │
│  │  tag     │  = Named pointer with metadata                 │
│  └──────────┘                                                 │
└──────────────────────────────────────────────────────────────┘
```

### How they relate:

```
                    ┌──────────────┐
                    │   COMMIT     │
                    │              │
                    │ tree: abc123 │──────┐
                    │ parent: ...  │      │
                    │ author: ...  │      │
                    │ message: ... │      │
                    └──────────────┘      │
                                          │
                                          ▼
                              ┌──────────────────┐
                              │      TREE        │
                              │                  │
                              │ blob abc → app.js│──► BLOB (file content)
                              │ blob def → pkg.js│──► BLOB (file content)
                              │ tree ghi → src/  │──► TREE (subdirectory)
                              └──────────────────┘        │
                                                           │
                                                           ▼
                                               ┌──────────────────┐
                                               │    TREE (src/)   │
                                               │                  │
                                               │ blob xyz → main.│──► BLOB
                                               │ blob uvw → util.│──► BLOB
                                               └──────────────────┘
```

## 5. Blob Objects — Storing File Content

A **blob** (Binary Large Object) stores the **content** of a single file. That's it — just the raw content. No filename, no permissions, no metadata.

### Creating a blob manually:

```bash
# Hash a string and store it as a blob in the object database
echo "Hello, Git Internals!" | git hash-object -w --stdin
# Output: a1b2c3d4e5f6... (SHA-1 hash)
# -w = actually WRITE it to the object database (not just calculate the hash)
```

### Looking at a blob:

```bash
# See the content of a blob
git cat-file -p a1b2c3d4e5f6
# Output: Hello, Git Internals!

# See the type of an object
git cat-file -t a1b2c3d4e5f6
# Output: blob

# See the size of an object
git cat-file -s a1b2c3d4e5f6
# Output: 22 (bytes)
```

### Key insight — blobs are content-addressed:

```bash
# Create two files with the SAME content
echo "Hello World" > file1.txt
echo "Hello World" > file2.txt

git add file1.txt file2.txt

# Both files point to the SAME blob object!
# Git stores the content only ONCE, no matter how many files have it.
# This is one reason Git repos are so space-efficient.
```

### What's actually stored in a blob:

```
The blob object contains:
┌────────────────────────────────────────┐
│ blob <content-size>\0<content>         │
│                                         │
│ Example:                                │
│ blob 22\0Hello, Git Internals!\n       │
│                                         │
│ \0 = null byte (separator)             │
│ The SHA-1 is calculated from this      │
│ entire string                          │
└────────────────────────────────────────┘
```

## 6. Tree Objects — Storing Directory Structure

A **tree** object represents a **directory**. It contains a list of entries, where each entry is either a blob (file) or another tree (subdirectory).

### Looking at a tree:

```bash
# Look at the tree of the latest commit
git cat-file -p main^{tree}
```

Output:

```
100644 blob a1b2c3d4...  README.md
100644 blob d5e6f7a8...  package.json
040000 tree b9c0d1e2...  src
100755 blob f3a4b5c6...  deploy.sh
```

### Understanding the fields:

```
100644 blob a1b2c3d4...  README.md
  │      │      │           │
  │      │      │           └── Filename
  │      │      └── SHA-1 hash of the object
  │      └── Object type (blob = file, tree = directory)
  └── File mode (permissions)
```

### File modes:

| Mode | Meaning |
|------|---------|
| `100644` | Regular file (not executable) |
| `100755` | Executable file |
| `120000` | Symbolic link |
| `040000` | Subdirectory (tree) |
| `160000` | Submodule (gitlink) |

### A tree pointing to a subdirectory:

```bash
# If "src" is a subdirectory, its tree entry points to ANOTHER tree:
git cat-file -p b9c0d1e2    # The SHA from the "src" tree entry
```

```
100644 blob 1a2b3c4d...  index.js
100644 blob 5e6f7a8b...  utils.js
040000 tree 9c0d1e2f...  components
```

### Visual representation:

```
Root tree (abc123):
├── 100644 blob → README.md
├── 100644 blob → package.json
├── 100755 blob → deploy.sh
└── 040000 tree → src/ (def456)
                   ├── 100644 blob → index.js
                   ├── 100644 blob → utils.js
                   └── 040000 tree → components/ (ghi789)
                                      ├── 100644 blob → Header.jsx
                                      └── 100644 blob → Footer.jsx
```

### Creating a tree manually (advanced):

```bash
# You rarely need to do this, but it shows how Git works internally

# First, create some blobs
echo "console.log('hello')" | git hash-object -w --stdin
# Returns: abc123...

# Then create a tree entry
# (Git provides git mktree for this, but it's complex)
# Usually, `git add` + `git write-tree` handles this automatically

# Write the current index as a tree
git write-tree
# Returns the SHA-1 of the new tree object
```

## 7. Commit Objects — Storing Snapshots in Time

A **commit** object is the "snapshot" you create when you run `git commit`. It contains:

1. A pointer to the **tree** (the state of all files at that moment)
2. A pointer to the **parent commit(s)** (what came before)
3. The **author** and **committer** info
4. The **commit message**

### Looking at a commit object:

```bash
git cat-file -p HEAD
```

Output:

```
tree 8c7e5a2b3d4f5e6a7b8c9d0e1f2a3b4c5d6e7f8a
parent 1a2b3c4d5e6f7a8b9c0d1e2f3a4b5c6d7e8f9a0b
author Ritesh Singh <ritesh@example.com> 1716789000 +0530
committer Ritesh Singh <ritesh@example.com> 1716789000 +0530

feat: add user authentication module
```

### Breaking it down:

```
tree 8c7e5a2b...
│
├── Points to the ROOT TREE of this commit
│   (the complete snapshot of ALL files at this point)
│
parent 1a2b3c4d...
│
├── Points to the PREVIOUS commit (the one before this)
│   (a merge commit has TWO parents)
│   (the very first commit has NO parent)
│
author Ritesh Singh <ritesh@example.com> 1716789000 +0530
│
├── WHO wrote the code + WHEN
│   (1716789000 = Unix timestamp)
│   (+0530 = timezone offset)
│
committer Ritesh Singh <ritesh@example.com> 1716789000 +0530
│
├── WHO committed it + WHEN
│   (Usually same as author, but different in cases
│    like cherry-pick, rebase, or applying patches)
│
feat: add user authentication module
│
└── The commit message
```

### A commit chain looks like this:

```
Commit C (HEAD)                Commit B                   Commit A (first)
┌──────────────┐              ┌──────────────┐           ┌──────────────┐
│ tree: tree3   │              │ tree: tree2   │           │ tree: tree1   │
│ parent: B     │──────────►  │ parent: A     │─────────►│ parent: none  │
│ author: ...   │              │ author: ...   │           │ author: ...   │
│ msg: "fix..." │              │ msg: "add..." │           │ msg: "init"   │
└──────────────┘              └──────────────┘           └──────────────┘

Each commit points to its parent.
Git follows this chain to reconstruct history.
This is why Git history is a LINKED LIST (or more precisely, a DAG).
```

### A merge commit has TWO parents:

```bash
git cat-file -p <merge-commit-sha>
```

```
tree 9a8b7c6d...
parent 1111111...     ← First parent (the branch you were on)
parent 2222222...     ← Second parent (the branch you merged in)
author ...
committer ...

Merge branch 'feature-x' into main
```

```
         ┌──── Commit X (feature)
         │
    ─────┼─── Commit Y (feature)
         │            \
    ─────┼─────────── Merge Commit (two parents!)
         │            /
    ─────┼─── Commit Z (main)
         │
```

### Creating a commit manually (to understand internals):

```bash
# Step 1: Create a blob
echo "Hello World" | git hash-object -w --stdin
# → abc123

# Step 2: Create a tree with that blob
# (using git update-index and git write-tree)
git update-index --add --cacheinfo 100644 abc123 hello.txt
TREE_SHA=$(git write-tree)
echo $TREE_SHA

# Step 3: Create a commit pointing to that tree
COMMIT_SHA=$(echo "My first manual commit" | git commit-tree $TREE_SHA)
echo $COMMIT_SHA

# Step 4: Point a branch to that commit
git update-ref refs/heads/main $COMMIT_SHA

# Now you have a working commit that you created entirely manually!
git log --oneline
```

This is **exactly** what `git commit` does behind the scenes — just automated!

## 8. Tag Objects — Named Points in History

There are two types of tags in Git:

### Lightweight tags (just a ref — no object):

```bash
# Creates a ref file, not an object
git tag v1.0.0

# This simply creates a file:
# .git/refs/tags/v1.0.0 → contains the commit SHA
```

### Annotated tags (a full object):

```bash
# Creates a tag OBJECT with metadata
git tag -a v1.0.0 -m "Release version 1.0.0"
```

```bash
# Look at the tag object
git cat-file -p v1.0.0
```

Output:

```
object 1a2b3c4d5e6f...        ← The commit this tag points to
type commit                    ← What type of object it points to
tag v1.0.0                     ← Tag name
tagger Ritesh Singh <ritesh@example.com> 1716789000 +0530

Release version 1.0.0         ← Tag message
```

### Visual comparison:

```
Lightweight tag:                  Annotated tag:
──────────────────                ──────────────────

refs/tags/v1.0.0 ──► Commit      refs/tags/v1.0.0 ──► Tag Object ──► Commit
                                                        │
                                                        ├── tagger info
                                                        ├── date
                                                        ├── message
                                                        └── (optional) GPG signature
```

> **Best practice:** Use **annotated tags** for releases (they have metadata and can be signed). Use lightweight tags for personal/temporary markers.

## 9. How a Commit Actually Works — Putting It All Together

Let's trace through what happens when you run `git commit`:

### Before the commit:

Imagine your project has these files:
```
my-project/
├── README.md         (content: "# My Project")
├── src/
│   └── app.js        (content: "console.log('hi')")
└── package.json      (content: "{name: 'my-project'}")
```

### What Git creates:

```
Step 1: Create blob objects for each file
─────────────────────────────────────────
blob aaa111 → "# My Project"          (README.md content)
blob bbb222 → "console.log('hi')"     (app.js content)
blob ccc333 → "{name: 'my-project'}"  (package.json content)

Step 2: Create tree objects for each directory
──────────────────────────────────────────────
tree ddd444 (src/):
  100644 blob bbb222  app.js

tree eee555 (root):
  100644 blob aaa111  README.md
  040000 tree ddd444  src
  100644 blob ccc333  package.json

Step 3: Create the commit object
─────────────────────────────────
commit fff666:
  tree eee555
  parent (previous commit SHA, or none if first commit)
  author Ritesh Singh <ritesh@example.com> ...
  committer Ritesh Singh <ritesh@example.com> ...

  Initial commit
```

### The complete picture:

```
                 commit fff666
                 ┌───────────────┐
                 │ tree: eee555  │
                 │ parent: none  │
                 │ author: ...   │
                 │ msg: "Init"   │
                 └───────┬───────┘
                         │
                         ▼
                  tree eee555 (root)
                 ┌───────────────────────────────┐
                 │ blob aaa111  README.md         │
                 │ tree ddd444  src/              │
                 │ blob ccc333  package.json      │
                 └──┬──────┬──────────────┬──────┘
                    │      │              │
           ┌────────┘      │              └────────┐
           ▼               ▼                       ▼
    blob aaa111     tree ddd444 (src/)      blob ccc333
    ┌──────────┐    ┌─────────────────┐    ┌──────────────────┐
    │ # My     │    │ blob bbb222     │    │ {name:           │
    │ Project  │    │   app.js        │    │  'my-project'}   │
    └──────────┘    └────────┬────────┘    └──────────────────┘
                             │
                             ▼
                      blob bbb222
                      ┌──────────────────┐
                      │ console.log('hi')│
                      └──────────────────┘
```

### Now, if you modify ONE file and commit again:

```bash
echo "console.log('hello world')" > src/app.js
git add src/app.js
git commit -m "Update greeting"
```

Git creates:
- **1 new blob** (for the changed `app.js`)
- **1 new tree** (for `src/` — because its contents changed)
- **1 new tree** (for root — because `src/` tree changed)
- **1 new commit** (pointing to the new root tree)

But it **reuses** the blobs for `README.md` and `package.json` — they didn't change!

```
                 commit ggg777 (new)
                 ┌────────────────┐
                 │ tree: hhh888   │──────────────┐
                 │ parent: fff666 │              │
                 └────────────────┘              │
                                                  ▼
                                        tree hhh888 (new root)
                                   ┌───────────────────────────┐
                             ┌─────│ blob aaa111  README.md    │  ← REUSED!
                             │     │ tree iii999  src/          │─┐
                             │     │ blob ccc333  package.json  │ │ ← REUSED!
                             │     └───────────────────────────┘ │
                             │                                    │
                             ▼                                    ▼
                      blob aaa111 (reused)              tree iii999 (new)
                      ┌──────────┐                  ┌─────────────────────┐
                      │ # My     │                  │ blob jjj000  app.js │
                      │ Project  │                  └──────────┬──────────┘
                      └──────────┘                             │
                                                               ▼
                                                       blob jjj000 (new)
                                                  ┌─────────────────────────┐
                                                  │ console.log('hello      │
                                                  │  world')                │
                                                  └─────────────────────────┘
```

This is why Git is so efficient — it **shares** unchanged objects between commits!

## 10. SHA-1 Hashing — How Git Names Everything

Every object in Git is identified by a **SHA-1 hash** — a 40-character hexadecimal string.

### How SHA-1 works:

```bash
# Git calculates: SHA-1( "<type> <size>\0<content>" )

# For a blob:
echo -n "blob 14\0Hello, World!\n" | sha1sum
# (This is a simplified example)

# The actual Git way:
echo "Hello, World!" | git hash-object --stdin
# Output: 8ab686eafeb1f44702738c8b0f24f2567c36da6d
```

### Properties of SHA-1:

```
Input: "Hello World"       → SHA-1: 0a4d55a8d778e5022fab701977c5d840bbc486d0
Input: "Hello World!"      → SHA-1: 2ef7bde608ce5404e97d5f042f95f89f1c232871
Input: "hello world"       → SHA-1: 2aae6c35c94fcfb415dbe95f408b9ce91ee846ed

Notice:
1. Same input → ALWAYS same output (deterministic)
2. Tiny change → COMPLETELY different hash (avalanche effect)
3. Fixed length → Always 40 hex characters (160 bits)
4. One-way → You can't reverse-engineer the input from the hash
5. Collision-resistant → It's practically impossible for two different
   inputs to produce the same hash
```

### Why SHA-1 is perfect for Git:

```
┌────────────────────────────────────────────────────────────┐
│  BENEFIT                     │  HOW SHA-1 PROVIDES IT      │
├──────────────────────────────┼─────────────────────────────┤
│  Content addressing          │  Hash IS the address         │
│  Data integrity              │  Any change = different hash │
│  Deduplication               │  Same content = same hash    │
│  Efficient comparison        │  Compare 40 chars, not files │
│  Distributed consistency     │  Same hash on every machine  │
└──────────────────────────────┴─────────────────────────────┘
```

### Abbreviated hashes:

```bash
# Full SHA-1: 2ef7bde608ce5404e97d5f042f95f89f1c232871
# Git usually shows abbreviated versions:

git log --oneline
# 2ef7bde  fix: resolve auth bug
# 0a4d55a  feat: add login page
# 8ab686e  initial commit

# You can use abbreviated hashes in commands:
git show 2ef7bde
# Git will find the full hash automatically
# (as long as the abbreviation is unique in your repo)
```

## 11. The Object Database — How Git Stores Objects

Objects are stored in `.git/objects/` as files, using the first 2 characters of the SHA-1 as a directory name and the remaining 38 characters as the filename.

```
.git/objects/
├── 2e/
│   └── f7bde608ce5404e97d5f042f95f89f1c232871   ← object 2ef7bd...
├── 0a/
│   └── 4d55a8d778e5022fab701977c5d840bbc486d0   ← object 0a4d55...
├── 8a/
│   └── b686eafeb1f44702738c8b0f24f2567c36da6d   ← object 8ab686...
├── info/
└── pack/
```

### Why the 2-character directory split?

```
Without split:                         With split (what Git does):
.git/objects/                          .git/objects/
├── 2ef7bde608ce540...                 ├── 2e/
├── 0a4d55a8d778e50...                 │   └── f7bde608ce540...
├── 8ab686eafeb1f44...                 ├── 0a/
├── ... (thousands more)               │   └── 4d55a8d778e50...
│                                      ├── 8a/
│ One directory with                   │   └── b686eafeb1f44...
│ thousands of files =                 │
│ SLOW on many filesystems!            │ 256 directories, each with
                                       │ fewer files = FAST!
```

### Looking at raw object files:

```bash
# Object files are zlib-compressed
# You can decompress them with Python:
python3 -c "
import zlib
with open('.git/objects/2e/f7bde608ce5404e97d5f042f95f89f1c232871', 'rb') as f:
    print(zlib.decompress(f.read()))
"
# Output: b'blob 14\x00Hello, World!\n'
```

The format is: `<type> <size>\0<content>` compressed with zlib.

## 12. Refs — Branches, Tags, and HEAD

**Refs** (references) are simply files that contain a SHA-1 hash pointing to a commit. They're how Git gives human-readable names to commits.

### Branches are just refs:

```bash
# A branch is literally a file containing a commit SHA
cat .git/refs/heads/main
# Output: 2ef7bde608ce5404e97d5f042f95f89f1c232871

cat .git/refs/heads/feature-login
# Output: 9a8b7c6d5e4f3a2b1c0d9e8f7a6b5c4d3e2f1a0b
```

That's it. **A branch is a 41-byte file** (40 hex chars + newline) that points to a commit. When you "create a branch," Git just creates a new file.

```bash
# These two commands are equivalent:
git branch new-feature
# IS THE SAME AS:
echo "2ef7bde608ce5404e97d5f042f95f89f1c232871" > .git/refs/heads/new-feature
```

### Tags are also refs:

```bash
# Lightweight tag
cat .git/refs/tags/v1.0.0
# Output: 2ef7bde608ce5404e97d5f042f95f89f1c232871
# (Points directly to a commit)

# Annotated tag
cat .git/refs/tags/v2.0.0
# Output: 5e6f7a8b9c0d1e2f3a4b5c6d7e8f9a0b1c2d3e4f
# (Points to a tag OBJECT, which then points to a commit)
```

### Remote refs:

```bash
# Remote tracking branches
cat .git/refs/remotes/origin/main
# Output: 2ef7bde608ce5404e97d5f042f95f89f1c232871
```

### The ref hierarchy:

```
.git/refs/
├── heads/                  ← Local branches
│   ├── main               → commit SHA
│   ├── develop            → commit SHA
│   └── feature-login      → commit SHA
├── tags/                   ← Tags
│   ├── v1.0.0             → commit SHA (lightweight) or tag object SHA (annotated)
│   └── v2.0.0             → tag object SHA
└── remotes/                ← Remote tracking branches
    └── origin/
        ├── main           → commit SHA
        └── develop        → commit SHA
```

### Low-level ref commands:

```bash
# Read a ref
git rev-parse main
# Output: 2ef7bde608ce5404e97d5f042f95f89f1c232871

# Update a ref (careful! this moves the branch pointer!)
git update-ref refs/heads/main abc123def

# Delete a ref
git update-ref -d refs/heads/old-branch

# List all refs
git show-ref
```

### Packed refs (optimization):

When you have many refs, Git can "pack" them into a single file for efficiency:

```bash
cat .git/packed-refs
```

```
# pack-refs with: peeled fully-peeled sorted
2ef7bde608ce5404e97d5f042f95f89f1c232871 refs/heads/main
9a8b7c6d5e4f3a2b1c0d9e8f7a6b5c4d3e2f1a0b refs/heads/develop
5e6f7a8b9c0d1e2f3a4b5c6d7e8f9a0b1c2d3e4f refs/tags/v1.0.0
```

Instead of hundreds of tiny files, Git stores all refs in one file. Individual ref files in `refs/` override entries in `packed-refs`.

## 13. HEAD — Where Are You Right Now?

**HEAD** is a special ref that tells Git which branch (or commit) you're currently on.

### Normal state (attached HEAD):

```bash
cat .git/HEAD
# Output: ref: refs/heads/main
# This means: "You are currently on the 'main' branch"
```

HEAD is **not** a commit SHA — it's a **symbolic reference** that points to a branch, which then points to a commit:

```
HEAD → refs/heads/main → commit abc123
```

### When you switch branches:

```bash
git checkout develop
cat .git/HEAD
# Output: ref: refs/heads/develop
# HEAD now points to 'develop' instead of 'main'
```

### Detached HEAD state:

```bash
git checkout abc123    # Checkout a specific commit (not a branch)
cat .git/HEAD
# Output: abc123def456789...
# HEAD points DIRECTLY to a commit, not to a branch!
```

```
Normal (attached):              Detached HEAD:
                                
HEAD ──► refs/heads/main        HEAD ──► abc123 (commit directly!)
              │                 
              ▼                 No branch involved.
         commit abc123          Commits you make here are "orphaned"
                                unless you create a branch.
```

### Why does this matter?

```
In detached HEAD, if you make commits and then switch to a branch,
those commits have NO branch pointing to them → they become
unreachable → eventually garbage collected → LOST!

Solution: Create a branch before leaving detached HEAD:
$ git checkout -b my-new-branch
```

## 14. The Reflog — Git's Safety Net

The **reflog** records every change to HEAD and branch tips. It's your "undo history" — even if you mess up badly, the reflog can save you.

```bash
# See the reflog
git reflog
```

Output:

```
abc123 HEAD@{0}: commit: feat: add auth
def456 HEAD@{1}: checkout: moving from develop to main
ghi789 HEAD@{2}: commit: fix: typo in readme
jkl012 HEAD@{3}: commit (amend): update config
mno345 HEAD@{4}: rebase: update title
pqr678 HEAD@{5}: reset: moving to HEAD~3
```

### The reflog saves you from:

```bash
# Scenario: You accidentally reset and lost commits
git reset --hard HEAD~5    # Oh no! 5 commits gone!

# The reflog still has them!
git reflog
# Shows: pqr678 HEAD@{1}: reset: moving to HEAD~5

# Recover!
git reset --hard HEAD@{1}    # Go back to before the reset
# Your commits are back! 🎉
```

### Reflog for specific branches:

```bash
git reflog show main
git reflog show develop
```

### Reflog expiry:

```bash
# Reflog entries expire after 90 days by default
# Unreachable entries expire after 30 days

# Configure expiry:
git config gc.reflogExpire 180.days
git config gc.reflogExpireUnreachable 60.days
```

> **Important:** The reflog is LOCAL only. It's not pushed to remotes. If you clone a repo, you don't get the other developers' reflogs.

## 15. The Index (Staging Area) — How It Really Works

The **index** (also called the **staging area** or **cache**) is a binary file at `.git/index`. It holds the "proposed next commit."

```bash
# See what's in the index
git ls-files --stage
```

Output:

```
100644 aaa111bbb222ccc333ddd444eee555fff666aaa1 0	README.md
100644 bbb222ccc333ddd444eee555fff666aaa111bbb2 0	package.json
100644 ccc333ddd444eee555fff666aaa111bbb222ccc3 0	src/app.js
```

### The three trees of Git:

```
┌────────────────┐    ┌────────────────┐    ┌────────────────┐
│  Working Dir    │    │     INDEX      │    │     HEAD       │
│  (your files)   │    │  (staging area)│    │  (last commit) │
│                 │    │                │    │                │
│  README.md  ◄───┼────┼── README.md   │    │  README.md     │
│  app.js     ◄───┼────┼── app.js      │    │  app.js        │
│  new.txt        │    │               │    │                │
│                 │    │                │    │                │
└─────────┬───────┘    └───────┬────────┘    └────────────────┘
          │                    │
     git add ──────────►       │
                          git commit ──────────►
```

### What `git add` really does:

```bash
git add README.md
```

1. Creates a **blob object** for `README.md`'s current content
2. Updates the **index** to point to that new blob
3. Does NOT create a commit yet!

### What `git commit` really does:

1. Takes the **index** as-is
2. Creates a **tree** from the index
3. Creates a **commit** object pointing to that tree
4. Updates the current branch ref to point to the new commit

### What `git status` really does:

```
git status compares THREE things:

1. HEAD tree vs INDEX    →  "Changes to be committed"
   (what's staged but not committed)

2. INDEX vs Working Dir  →  "Changes not staged for commit"
   (what's changed but not staged)

3. Working Dir files not in INDEX  →  "Untracked files"
```

## 16. Loose Objects vs Pack Files

When Git first creates objects, they're stored as individual files — called **loose objects**.

```
Loose objects:                     Pack files:
──────────────                     ───────────
.git/objects/                      .git/objects/pack/
├── 2e/                            ├── pack-abc123.idx   (index)
│   └── f7bde608...               └── pack-abc123.pack  (data)
├── 0a/
│   └── 4d55a8d7...               Single file containing
├── 8a/                            MANY objects, compressed
│   └── b686eafe...               with delta encoding!
├── ...
│
Many individual files,
each zlib-compressed
```

### When does Git pack objects?

```
1. Automatically when there are too many loose objects (~6,700)
2. When you run `git gc` (garbage collection)
3. When you `git push` (creates a pack for transfer)
4. When you `git clone` (you receive a pack)
```

## 17. Pack Files — How Git Compresses Data

Pack files are one of Git's most clever optimizations. They take many loose objects and compress them into a single file using **delta compression**.

### Looking at pack files:

```bash
# List pack files
ls .git/objects/pack/

# Output:
# pack-abc123def456.idx     ← Index (lookup table)
# pack-abc123def456.pack    ← Data (compressed objects)
```

### Pack file structure:

```
Pack File (.pack):
┌──────────────────────────────────────────────┐
│ HEADER                                        │
│ ├── Magic: "PACK"                            │
│ ├── Version: 2                               │
│ └── Object count: 1547                       │
├──────────────────────────────────────────────┤
│ OBJECT 1 (full)                              │
│ ├── Type: commit                             │
│ ├── Size: 234                                │
│ └── Data: (zlib compressed)                  │
├──────────────────────────────────────────────┤
│ OBJECT 2 (delta)                             │
│ ├── Type: OFS_DELTA                          │
│ ├── Base object offset: -4567               │
│ ├── Size: 45                                 │
│ └── Delta: (instructions to recreate)        │
├──────────────────────────────────────────────┤
│ ... more objects ...                          │
├──────────────────────────────────────────────┤
│ CHECKSUM: SHA-1 of entire pack               │
└──────────────────────────────────────────────┘
```

### Index file (.idx):

```
The index file is a lookup table that lets Git quickly find
any object in the pack by its SHA-1 hash.

Without the index, Git would have to scan the entire pack file
sequentially to find an object — way too slow!

With the index, it's a binary search: O(log n) lookups.
```

### Inspect pack files:

```bash
# Verify integrity of pack files
git verify-pack -v .git/objects/pack/pack-abc123.idx
```

Output:

```
abc123def456 commit 234 150 12         ← Full object
bbb222ccc333 tree   89  75  300        ← Full object
ccc333ddd444 blob   1024 489 450       ← Full object
ddd444eee555 blob   45   39  940 3 ccc333ddd444   ← DELTA! (based on ccc333)
eee555fff666 blob   38   33  990 3 ccc333ddd444   ← DELTA! (also based on ccc333)
```

The columns are: SHA-1, type, actual-size, pack-size, offset, depth, base-SHA

## 18. Delta Compression — The Clever Trick

Delta compression is how Git keeps pack files small. Instead of storing every version of a file in full, Git stores the **differences** (deltas) between similar objects.

### Example:

```
Version 1 of app.js (1000 lines):
┌──────────────────────────────────────────┐
│ function main() {                         │
│   console.log("Hello");                  │
│   // ... 998 more lines ...              │
│ }                                         │
└──────────────────────────────────────────┘
Size: 20 KB

Version 2 of app.js (1001 lines — added 1 line):
┌──────────────────────────────────────────┐
│ function main() {                         │
│   console.log("Hello World");            │  ← Changed this line
│   console.log("New line");               │  ← Added this line
│   // ... 998 more lines ...              │
│ }                                         │
└──────────────────────────────────────────┘
Size: 20.1 KB
```

### Without delta compression:

```
Stored:
├── blob v1: 20 KB (full content)
├── blob v2: 20.1 KB (full content)
│
Total: 40.1 KB
```

### With delta compression:

```
Stored:
├── blob v2: 20.1 KB (FULL — the "base" object)
├── blob v1: 0.2 KB (DELTA — "take v2, remove line 3, change line 2")
│
Total: 20.3 KB  ← Almost 50% savings!
```

### How delta instructions look (simplified):

```
Delta for v1 (based on v2):
┌──────────────────────────────────────────┐
│ COPY bytes 0-25 from base               │  ← "function main() {\n"
│ INSERT "  console.log(\"Hello\");\n"    │  ← Replace the changed line
│ COPY bytes 52-20100 from base           │  ← Copy rest unchanged
└──────────────────────────────────────────┘
```

### Interesting detail — Git stores the NEWEST version in full:

```
Why? Because you access recent versions more often than old ones.

If the base object is the newest version:
- Checking out HEAD → fast (read the base directly)
- Checking out an old commit → slower (apply deltas backward)

This is the opposite of what you might expect!
```

### Delta chains:

```
Sometimes deltas reference other deltas (delta chains):

Object D → delta of C → delta of B → delta of A (base)

To reconstruct D, Git must:
1. Read A (base)
2. Apply delta to get B
3. Apply delta to get C
4. Apply delta to get D

Git limits chain depth (default: 50) to prevent this from getting too slow.
```

```bash
# Configure maximum delta chain depth
git config pack.depth 50        # Default: 50
git config pack.window 250      # How many objects to consider for delta pairing
```

## 19. Git GC — Garbage Collection

**`git gc`** (garbage collection) is Git's cleanup tool. It:

1. Packs loose objects into pack files
2. Removes unreachable objects (dangling blobs, orphaned commits)
3. Packs refs
4. Prunes reflog entries older than the expiry

```bash
# Run garbage collection manually
git gc

# Aggressive GC (more thorough, slower)
git gc --aggressive

# See what GC would do without doing it
git gc --dry-run
```

### When does GC run automatically?

```bash
# Git runs `git gc --auto` in the background when:
# - There are more than 6,700 loose objects
# - There are more than 50 pack files

# Configure thresholds:
git config gc.auto 6700          # Loose object threshold
git config gc.autoPackLimit 50   # Pack file threshold
```

### Pruning unreachable objects:

```bash
# Objects become unreachable when:
# - You amend a commit (old commit is orphaned)
# - You rebase (old commits are orphaned)
# - You delete a branch with unmerged commits
# - You reset to an earlier commit

# Prune unreachable objects
git prune

# By default, GC won't prune objects less than 2 weeks old
# (safety net in case you need them)
git config gc.pruneExpire "2.weeks.ago"
```

### Reducing repo size:

```bash
# Step 1: See current size
git count-objects -vH
# Output shows: size-pack, count, size, in-pack, etc.

# Step 2: Find large files in history
git rev-list --objects --all | \
git cat-file --batch-check='%(objecttype) %(objectname) %(objectsize) %(rest)' | \
grep '^blob' | \
sort -t' ' -k3 -n -r | \
head -20

# Step 3: Run aggressive GC
git gc --aggressive --prune=now

# Step 4: Check new size
git count-objects -vH
```

## 20. Git Transfer Protocols — How Push/Pull Works

When you `git push` or `git pull`, Git uses a **transfer protocol** to move objects between repositories.

### The smart HTTP protocol (most common):

```
git push origin main:

1. Client → Server: "I want to push. Here's what I have"
   (sends list of refs and their SHAs)

2. Server → Client: "Here's what I have"
   (sends its refs and SHAs)

3. Client: Calculates what the server DOESN'T have
   (objects reachable from new refs but not from server's refs)

4. Client → Server: Sends a PACK FILE with just those objects

5. Server: Unpacks the objects and updates refs
```

```
┌──────────┐                         ┌──────────┐
│  Client   │                         │  Server   │
│           │   "I have ref A→abc"   │           │
│           │ ──────────────────────► │           │
│           │                         │           │
│           │   "I have ref A→xyz"   │           │
│           │ ◄────────────────────── │           │
│           │                         │           │
│           │   Pack file with new    │           │
│           │   objects               │           │
│           │ ──────────────────────► │           │
│           │                         │           │
│           │   "OK, refs updated"    │           │
│           │ ◄────────────────────── │           │
└──────────┘                         └──────────┘
```

### Protocols Git supports:

| Protocol | URL Format | Auth | Use Case |
|----------|-----------|------|----------|
| HTTPS | `https://github.com/user/repo.git` | Username/token | Most common |
| SSH | `git@github.com:user/repo.git` | SSH keys | Developer workflow |
| Git | `git://github.com/user/repo.git` | None | Read-only, rare |
| Local | `/path/to/repo.git` | Filesystem | Local clones |

### What happens during `git clone`:

```bash
git clone https://github.com/user/repo.git
```

```
1. Connect to server
2. Receive list of refs (branches, tags)
3. Server creates a pack file with ALL objects
4. Client downloads the pack file
5. Client unpacks objects (or keeps as pack)
6. Client creates local refs based on remote refs
7. Client checks out the default branch (usually main)
```

### Shallow clones:

```bash
# Only get the last N commits (not full history)
git clone --depth 1 https://github.com/user/repo.git

# Useful in CI/CD to speed up checkout!
# The pack file is MUCH smaller
```

## 21. SHA-256 — The Future of Git Hashing

Git is gradually moving from SHA-1 to SHA-256 for improved security.

### Why move away from SHA-1?

```
In 2017, Google demonstrated a SHA-1 collision:
Two different PDF files with the SAME SHA-1 hash!

While this hasn't been exploited against Git yet,
it theoretically allows an attacker to create a
malicious object with the same hash as a legitimate one.

SHA-256 provides much stronger collision resistance.
```

### Current state (as of 2026):

```bash
# Git 2.42+ supports SHA-256 repositories
git init --object-format=sha256 my-new-repo

# Check your repo's hash format
git rev-parse --show-object-format
# Output: sha1  or  sha256
```

### SHA-1 vs SHA-256:

| Feature | SHA-1 | SHA-256 |
|---------|-------|---------|
| Hash length | 160 bits (40 hex chars) | 256 bits (64 hex chars) |
| Security | Broken (collisions found) | Currently secure |
| Performance | Slightly faster | Slightly slower |
| Git support | Default, universal | Experimental, growing |
| Interop | Works with everything | Not compatible with SHA-1 repos |

```
SHA-1:   2ef7bde608ce5404e97d5f042f95f89f1c232871
SHA-256: a1b2c3d4e5f6789012345678901234567890abcdef1234567890abcdef12345678
```

> **For now:** Most repos still use SHA-1, and Git has hardening against known collision attacks. SHA-256 will become the default eventually.

## 23. Debugging with Git Internals

### Finding lost commits:

```bash
# List all unreachable commits
git fsck --unreachable --no-reflogs

# Find dangling commits (not referenced by any branch)
git fsck --dangling

# Find lost commits via reflog
git reflog --all

# Recover a lost commit
git branch recover-branch <lost-commit-sha>
```

### Checking repository integrity:

```bash
# Full integrity check
git fsck --full

# Output (if healthy):
# Checking object directories: 100% (256/256), done.
# Checking objects: 100% (1547/1547), done.

# Output (if corrupted):
# error: sha1 mismatch abc123...
# missing blob abc123...
```

### Finding large files in history:

```bash
# Find the 10 largest blobs in the repo
git rev-list --objects --all | \
git cat-file --batch-check='%(objecttype) %(objectname) %(objectsize) %(rest)' | \
grep '^blob' | \
sort -t' ' -k3 -n -r | \
head -10
```

### Removing sensitive data from history:

```bash
# If you accidentally committed a password or API key:

# Method 1: git filter-branch (older, slower)
git filter-branch --force --index-filter \
  'git rm --cached --ignore-unmatch path/to/secret-file' \
  --prune-empty --tag-name-filter cat -- --all

# Method 2: git filter-repo (modern, faster — recommended!)
# Install first: pip install git-filter-repo
git filter-repo --path path/to/secret-file --invert-paths

# After either method:
git push origin --force --all
git push origin --force --tags

# Also: Rotate the exposed credentials immediately!
```

## 24. Summary

```
┌─────────────────────────────────────────────────────────────────┐
│              GIT INTERNALS CHEAT SHEET                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  FOUR OBJECT TYPES:                                              │
│  ──────────────────                                              │
│  blob   = file content (no name, just data)                     │
│  tree   = directory listing (names + blob/tree refs)            │
│  commit = snapshot (tree + parent + author + message)           │
│  tag    = annotated tag (name + commit ref + metadata)          │
│                                                                  │
│  REFS:                                                           │
│  ─────                                                           │
│  Branch = file in .git/refs/heads/ containing a commit SHA      │
│  Tag    = file in .git/refs/tags/                               │
│  HEAD   = file in .git/HEAD pointing to current branch          │
│                                                                  │
│  STORAGE:                                                        │
│  ────────                                                        │
│  Loose objects  = individual compressed files in .git/objects/  │
│  Pack files     = many objects in one file with delta encoding   │
│  git gc         = converts loose → packed, removes garbage      │
│                                                                  │
│  KEY COMMANDS:                                                   │
│  ─────────────                                                   │
│  git cat-file -t SHA     Show object type                       │
│  git cat-file -p SHA     Show object content                    │
│  git cat-file -s SHA     Show object size                       │
│  git hash-object -w      Create a blob object                   │
│  git ls-files --stage    Show index (staging area)              │
│  git count-objects -vH   Object database stats                  │
│  git verify-pack -v      Inspect pack files                     │
│  git fsck                Check integrity                        │
│  git reflog              View ref history (safety net!)         │
│  git gc                  Run garbage collection                 │
│                                                                  │
│  THE BIG PICTURE:                                                │
│  ────────────────                                                │
│  Git = content-addressable filesystem                           │
│  Everything identified by SHA-1 hash                            │
│  Branches are just pointers (tiny files)                        │
│  Commits don't store diffs — they store full snapshots          │
│  (but Git is efficient because it shares unchanged objects)     │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```
