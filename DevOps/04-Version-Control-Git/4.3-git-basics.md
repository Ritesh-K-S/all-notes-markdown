## 🤔 The Big Picture — How These Commands Work Together

Before diving into each command individually, let's see how they all fit together. Remember the three areas of Git from the last section?

```
┌─────────────────┐                 ┌─────────────────┐                ┌─────────────────┐
│                 │    git add      │                 │   git commit   │                 │
│   WORKING       │ ─────────────→  │   STAGING AREA  │ ────────────→  │   REPOSITORY    │
│   DIRECTORY     │                 │   (Index)       │                │   (.git folder) │
│                 │                 │                 │                │                 │
│  You edit files │                 │  "Ready to be   │                │  Permanent      │
│  here           │                 │   committed"    │                │  history        │
│                 │                 │                 │                │                 │
└─────────────────┘                 └─────────────────┘                └─────────────────┘

git init    → Creates the repository (the .git folder)
git status  → Shows what's happening in all three areas
git diff    → Shows exactly what changed in your files
git log     → Shows the history of all commits in the repository
```

**The workflow every developer follows, every single day:**

```
1. Edit files in your Working Directory
2. git add — move changes to Staging Area (select what to commit)
3. git commit — save the staged changes permanently to the Repository
4. Repeat!
```

Now let's learn each command in detail.

## 1️⃣ `git init` — Initialize a New Repository

### What It Does

`git init` creates a **new Git repository** in your current directory. It creates a hidden `.git` folder that Git uses to track everything.

### Syntax

```bash
git init
```

### Let's Try It!

```bash
# Step 1: Create a new project directory
mkdir my-first-repo
cd my-first-repo

# Step 2: Initialize Git
git init
```

**Output:**
```
Initialized empty Git repository in /home/ritesh/my-first-repo/.git/
```

### What Just Happened?

Let's look at what Git created:

```bash
ls -la
```

**Output:**
```
total 12
drwxr-xr-x 3 ritesh ritesh 4096 Jan 15 10:00 .
drwxr-xr-x 5 ritesh ritesh 4096 Jan 15 10:00 ..
drwxr-xr-x 7 ritesh ritesh 4096 Jan 15 10:00 .git    ← This is the magic!
```

That `.git` folder is where Git stores **everything** — all commits, branches, history, configuration. Let's peek inside:

```bash
ls .git/
```

**Output:**
```
HEAD        ← Points to the current branch
config      ← Repository-level configuration
description ← Used by GitWeb (rarely used)
hooks/      ← Scripts that run on certain events (we'll cover later)
info/       ← Additional info (rarely used)
objects/    ← Where Git stores all data (commits, trees, blobs)
refs/       ← Where branches and tags are stored
```

> **Important:** Never manually edit anything inside `.git/` unless you know exactly what you're doing. Corrupting this folder means losing your entire Git history.

### Common Mistake: `git init` in the Wrong Directory

```bash
# ❌ DON'T do this — never init in your home directory!
cd ~
git init     # Now Git tracks EVERYTHING in your home folder 😱

# ✅ DO this — init in your project folder
cd ~/projects/my-app
git init
```

If you accidentally init in the wrong place:
```bash
# Just remove the .git folder to "un-Git" a directory
rm -rf .git
```

### Reinitializing an Existing Repo

Running `git init` in a folder that already has a `.git` folder is **safe**:

```bash
git init
# Output: Reinitialized existing Git repository in /home/ritesh/my-repo/.git/
```

It won't delete any existing history. It just reinitializes the configuration.

## 2️⃣ `git status` — Check What's Going On

### What It Does

`git status` shows you the **current state** of your working directory and staging area. It tells you:
- Which files have been **modified**
- Which files are **staged** (ready to commit)
- Which files are **untracked** (new files Git doesn't know about)

### Syntax

```bash
git status
```

### Let's See It in Action

Right after `git init`, with no files:

```bash
git status
```

**Output:**
```
On branch main
No commits yet
nothing to commit (create/copy files and use "git add" to track)
```

Git is telling us: *"Hey, the repo is empty. There's nothing to commit. Create some files!"*

### Now Let's Create a File

```bash
echo "Hello, Git!" > hello.txt
git status
```

**Output:**
```
On branch main

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)

        hello.txt

nothing added to commit but untracked files present (use "git add" to track)
```

**What this means:**

```
Untracked files:
        hello.txt       ← Git sees this file but is NOT tracking it yet
```

The file is in your **Working Directory** but Git doesn't know about it. It's "untracked." We need to use `git add` to tell Git to start tracking it.

### The Different Statuses a File Can Have

```
┌──────────────────────────────────────────────────────────────────────┐
│                                                                      │
│   Untracked  →  Git doesn't know about this file at all             │
│                  (new files that were never added)                    │
│                                                                      │
│   Tracked    →  Git is watching this file                           │
│       │                                                              │
│       ├── Unmodified  →  File hasn't changed since last commit      │
│       │                  (not shown in git status — everything OK)    │
│       │                                                              │
│       ├── Modified    →  File has been changed but NOT staged       │
│       │                  (shown in RED in git status)                 │
│       │                                                              │
│       └── Staged      →  File changes are ready to be committed     │
│                          (shown in GREEN in git status)              │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

### Short Status (Compact View)

```bash
git status -s
# or
git status --short
```

**Output:**
```
?? hello.txt          ← ?? means untracked
A  staged-file.txt    ← A means newly staged (Added)
M  modified-file.txt  ← M means modified
MM both-file.txt      ← staged AND modified again after staging
```

The symbols:
| Symbol | Position | Meaning |
|--------|----------|---------|
| `??` | — | Untracked file |
| `A` | Left | New file staged for commit |
| `M` | Left | Modified and staged |
| `M` | Right | Modified but NOT staged |
| `D` | Left | Deleted and staged |
| `R` | Left | Renamed |

## 3️⃣ `git add` — Stage Your Changes

### What It Does

`git add` moves changes from your **Working Directory** to the **Staging Area**. Think of it as **selecting** which changes you want to include in the next commit.

### Why Not Just Commit Everything?

Great question! The staging area gives you **control**. Maybe you changed 5 files, but you only want to commit 3 of them right now. `git add` lets you pick exactly what goes into each commit.

```
Example:

You changed:  file1.py, file2.py, file3.py, file4.py, file5.py

But you only want to commit the login feature (file1 and file2):

  git add file1.py file2.py    ← Only these go to staging
  git commit -m "Add login feature"

Then commit the other files separately:
  git add file3.py file4.py file5.py
  git commit -m "Add dashboard feature"
```

This creates a **clean, organized** history instead of one giant "I changed everything" commit.

### Syntax & Examples

```bash
# Add a single file
git add hello.txt

# Add multiple files
git add file1.txt file2.txt file3.txt

# Add all files in a specific directory
git add src/

# Add ALL changes (new, modified, deleted files)
git add .

# Add all changes (alternative)
git add -A
# or
git add --all

# Add all modified and deleted files (but NOT new untracked files)
git add -u
```

### Let's See It in Action

```bash
# Create some files
echo "Hello, Git!" > hello.txt
echo "print('Hello')" > app.py

# Check status — both are untracked
git status
```

**Output:**
```
Untracked files:
        app.py
        hello.txt
```

```bash
# Stage only hello.txt
git add hello.txt

# Check status again
git status
```

**Output:**
```
Changes to be committed:
  (use "git rm --cached <file>..." to unstage)
        new file:   hello.txt          ← GREEN — staged and ready!

Untracked files:
        app.py                         ← RED — still not tracked
```

See the difference? `hello.txt` is now in the **Staging Area** (shown in green), while `app.py` is still untracked.

### `git add .` vs `git add -A` vs `git add -u`

| Command | New Files | Modified Files | Deleted Files | Scope |
|---------|-----------|---------------|---------------|-------|
| `git add .` | ✅ | ✅ | ✅ | Current directory and subdirectories |
| `git add -A` | ✅ | ✅ | ✅ | Entire repository |
| `git add -u` | ❌ | ✅ | ✅ | Entire repository (only tracked files) |

**Most common usage:** `git add .` (adds everything in the current directory). Use this when you want to stage all your changes.

### Unstaging a File (Undo `git add`)

Oops, you staged a file by mistake? No problem:

```bash
# Unstage a specific file (keep the file, just remove from staging)
git restore --staged hello.txt

# Older syntax (still works)
git reset HEAD hello.txt
```

### Interactive Staging (Stage Parts of a File)

What if you changed 10 lines in a file but only want to commit 5 of them? Git can do that!

```bash
git add -p hello.txt
# or
git add --patch hello.txt
```

Git will show you each chunk of changes and ask:

```
Stage this hunk [y,n,q,a,d,s,e,?]?
```

| Option | Meaning |
|--------|---------|
| `y` | Yes, stage this chunk |
| `n` | No, skip this chunk |
| `q` | Quit (don't stage anything else) |
| `a` | Stage this and all remaining chunks |
| `s` | Split this chunk into smaller pieces |
| `e` | Manually edit this chunk |

This is an **advanced** feature, but it's incredibly useful for creating clean, focused commits.

## 4️⃣ `git commit` — Save Your Changes Permanently

### What It Does

`git commit` takes everything in the **Staging Area** and creates a permanent **snapshot** (commit) in the **Repository**. It's like pressing "Save Game" in a video game.

### Syntax

```bash
# Basic commit with a message
git commit -m "Your commit message here"

# Commit with a detailed message (opens editor)
git commit

# Stage all tracked files AND commit in one step (skip git add for MODIFIED files)
git commit -am "Your message"
# Note: -am does NOT work for NEW (untracked) files — you still need git add first!
```

### Let's Make Our First Commit!

```bash
# Make sure we have a file staged
git add hello.txt

# Create the commit
git commit -m "Add hello.txt with greeting message"
```

**Output:**
```
[main (root-commit) a1b2c3d] Add hello.txt with greeting message
 1 file changed, 1 insertion(+)
 create mode 100644 hello.txt
```

**Let's break down this output:**

| Part | Meaning |
|------|---------|
| `[main` | The commit was made on the `main` branch |
| `(root-commit)` | This is the very first commit in the repo |
| `a1b2c3d]` | The first 7 characters of the commit's unique SHA hash |
| `1 file changed` | One file was part of this commit |
| `1 insertion(+)` | One line was added |
| `create mode 100644` | A new file was created with regular file permissions |

### Anatomy of a Commit

Every commit contains:

```
┌─────────────────────────────────────────────────────────┐
│                     COMMIT OBJECT                       │
│                                                         │
│  Hash:      a1b2c3d4e5f6789...  (unique SHA-1 ID)     │
│  Author:    Ritesh Singh <ritesh@example.com>           │
│  Date:      Mon Jan 15 10:30:00 2025 +0530             │
│  Message:   "Add hello.txt with greeting message"       │
│  Parent:    (none — this is the first commit)           │
│  Snapshot:  [tree object pointing to all files]         │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### Writing Good Commit Messages

This is **way more important** than most beginners think. Good commit messages save your future self (and your team) hours of confusion.

**Bad Commit Messages:** ❌
```
git commit -m "fix"
git commit -m "update"
git commit -m "changes"
git commit -m "asdfgh"
git commit -m "WIP"
git commit -m "stuff"
```

**Good Commit Messages:** ✅
```
git commit -m "Add user authentication with JWT tokens"
git commit -m "Fix login page crash when password is empty"
git commit -m "Update Dockerfile to use Node 20 base image"
git commit -m "Remove deprecated API endpoint /v1/users"
git commit -m "Refactor database connection pool for better performance"
```

### The Convention for Commit Messages

Most teams follow this format:

```
<type>: <short description>

[optional body — more details]

[optional footer — references to issues]
```

**Common types:**

| Type | When to Use |
|------|-------------|
| `feat` | A new feature |
| `fix` | A bug fix |
| `docs` | Documentation changes |
| `style` | Formatting, missing semicolons (no code change) |
| `refactor` | Code restructuring (no behavior change) |
| `test` | Adding or fixing tests |
| `chore` | Build process, dependencies, tooling changes |

**Examples:**
```bash
git commit -m "feat: add user registration form"
git commit -m "fix: resolve null pointer in payment processing"
git commit -m "docs: update API documentation for v2 endpoints"
git commit -m "chore: upgrade Express.js from 4.x to 5.x"
```

This is called **Conventional Commits** — and many CI/CD tools can automatically generate changelogs from these!

### Multi-Line Commit Messages

For more detailed messages, just run `git commit` without `-m`:

```bash
git commit
```

This opens your configured editor. Write:

```
feat: add user authentication system

- Implement JWT-based token authentication
- Add login and register endpoints
- Add password hashing with bcrypt
- Add middleware for protected routes

Closes #42
```

Save and close the editor — Git captures the full message.

### The `-am` Shortcut

```bash
# Instead of two commands:
git add .
git commit -m "Update homepage"

# You can do:
git commit -am "Update homepage"
```

**⚠️ Important caveat:** `-am` only works for files that are **already tracked** (modified files). It does **NOT** stage new (untracked) files. For new files, you still need `git add` first.

### Amending the Last Commit

Made a typo in your commit message? Forgot to include a file? You can fix the **most recent** commit:

```bash
# Fix the commit message
git commit --amend -m "Correct commit message here"

# Add a forgotten file to the last commit
git add forgotten-file.txt
git commit --amend --no-edit    # --no-edit keeps the original message
```

> **⚠️ Warning:** Only amend commits that haven't been **pushed** to a remote repository yet. Amending pushed commits rewrites history and can cause problems for your team.

## 5️⃣ `git log` — View Commit History

### What It Does

`git log` shows you the **history of all commits** in the repository, from newest to oldest.

### Basic Usage

```bash
git log
```

**Output:**
```
commit a1b2c3d4e5f6789012345678901234567890abcd (HEAD -> main)
Author: Ritesh Singh <ritesh@example.com>
Date:   Mon Jan 15 10:30:00 2025 +0530

    Add hello.txt with greeting message
```

### Let's Create More Commits to See a Better Log

```bash
# Create and commit more files
echo "print('Hello World')" > app.py
git add app.py
git commit -m "feat: add main application file"

echo "# My Project" > README.md
git add README.md
git commit -m "docs: add README file"

echo "*.pyc" > .gitignore
git add .gitignore
git commit -m "chore: add gitignore for Python"
```

Now let's see the log:

```bash
git log
```

**Output:**
```
commit d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0c1d2e3 (HEAD -> main)
Author: Ritesh Singh <ritesh@example.com>
Date:   Mon Jan 15 11:00:00 2025 +0530

    chore: add gitignore for Python

commit b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0c1

```

### Useful `git log` Options

There are **many** ways to customize the log output. Here are the most useful ones:

#### One-Line Log (Compact View)

```bash
git log --oneline
```

**Output:**
```
d4e5f6a (HEAD -> main) chore: add gitignore for Python
b2c3d4e docs: add README file
9a0b1c2 feat: add main application file
a1b2c3d Add hello.txt with greeting message
```

Much cleaner! Each commit is on one line with just the short hash and message.

#### Limit Number of Commits

```bash
# Show only last 3 commits
git log -3

# Show only last 5 commits in one-line format
git log --oneline -5
```

#### Show What Files Changed in Each Commit

```bash
git log --stat
```

**Output:**
```
commit d4e5f6a (HEAD -> main)
Author: Ritesh Singh <ritesh@example.com>
Date:   Mon Jan 15 11:00:00 2025 +0530

    chore: add gitignore for Python

 .gitignore | 1 +
 1 file changed, 1 insertion(+)
```

#### Show the Actual Changes (Patch)

```bash
git log -p
# or
git log --patch
```

This shows the full **diff** for each commit — every line that was added or removed.

#### Pretty Graph View

```bash
git log --oneline --graph --all --decorate
```

**Output:**
```
* d4e5f6a (HEAD -> main) chore: add gitignore for Python
* b2c3d4e docs: add README file
* 9a0b1c2 feat: add main application file
* a1b2c3d Add hello.txt with greeting message
```

When you have branches, this shows a beautiful ASCII graph:

```
* e5f6a7b (HEAD -> main) Merge feature-login
|\
| * c3d4e5f (feature-login) Add login page
| * b2c3d4e Add auth module
|/
* 9a0b1c2 Add homepage
* a1b2c3d Initial commit
```

Remember the alias we set up in the previous section? This is it:
```bash
git lg    # If you set up the alias
```

#### Search Commits by Author

```bash
git log --author="Ritesh"
```

#### Search Commits by Message

```bash
git log --grep="login"
```

Shows only commits whose messages contain "login".

#### Show Commits in a Date Range

```bash
git log --after="2025-01-01" --before="2025-02-01"
```

#### Show Commits for a Specific File

```bash
git log -- app.py
git log --oneline -- app.py
```

Shows only commits that affected `app.py`.

### Quick Reference: `git log` Options

| Command | What It Shows |
|---------|--------------|
| `git log` | Full log (hash, author, date, message) |
| `git log --oneline` | Compact one-line per commit |
| `git log -5` | Last 5 commits |
| `git log --stat` | Files changed in each commit |
| `git log -p` | Full diff for each commit |
| `git log --graph` | ASCII branch graph |
| `git log --author="name"` | Commits by a specific author |
| `git log --grep="text"` | Commits with "text" in message |
| `git log -- filename` | Commits affecting a specific file |
| `git log --oneline --graph --all` | The "ultimate" overview |

## 6️⃣ `git diff` — See Exactly What Changed

### What It Does

`git diff` shows the **exact changes** made to files — line by line. It compares different versions and shows what was added, removed, or modified.

### Understanding Diff Output

Before we look at the commands, let's understand how to **read** a diff:

```bash
# Let's modify a file
echo "Hello, Git! Welcome to version control." > hello.txt

# Now see the diff
git diff
```

**Output:**
```diff
diff --git a/hello.txt b/hello.txt
index 8d0e412..b94e238 100644
--- a/hello.txt
+++ b/hello.txt
@@ -1 +1 @@
-Hello, Git!
+Hello, Git! Welcome to version control.
```

**Let's break this down piece by piece:**

| Line | Meaning |
|------|---------|
| `diff --git a/hello.txt b/hello.txt` | Comparing the old version (a/) with the new version (b/) of hello.txt |
| `index 8d0e412..b94e238 100644` | Internal Git object IDs and file permissions |
| `--- a/hello.txt` | The old version of the file (before changes) |
| `+++ b/hello.txt` | The new version of the file (after changes) |
| `@@ -1 +1 @@` | Shows which lines are affected. `-1` = line 1 of old file, `+1` = line 1 of new file |
| `-Hello, Git!` | Line **removed** (shown with a minus sign, usually in RED) |
| `+Hello, Git! Welcome to version control.` | Line **added** (shown with a plus sign, usually in GREEN) |

### A More Complex Diff Example

```bash
# Create a file with multiple lines
cat > app.py << 'EOF'
def greet(name):
    print(f"Hello, {name}!")

def farewell(name):
    print(f"Goodbye, {name}!")

greet("World")
EOF

# Stage and commit
git add app.py
git commit -m "Add greeting functions"

# Now modify the file
cat > app.py << 'EOF'
def greet(name):
    print(f"Hi, {name}! Welcome!")

def farewell(name):
    print(f"See you later, {name}!")

def thanks(name):
    print(f"Thank you, {name}!")

greet("World")
thanks("World")
EOF

# See the diff
git diff
```

**Output:**
```diff
diff --git a/app.py b/app.py
index 1234567..abcdefg 100644
--- a/app.py
+++ b/app.py
@@ -1,7 +1,10 @@
 def greet(name):
-    print(f"Hello, {name}!")
+    print(f"Hi, {name}! Welcome!")
 
 def farewell(name):
-    print(f"Goodbye, {name}!")
+    print(f"See you later, {name}!")
+
+def thanks(name):
+    print(f"Thank you, {name}!")
 
 greet("World")
+thanks("World")
```

**Reading this diff:**

- Lines starting with a **space** (no symbol) = **unchanged** (context lines)
- Lines starting with **`-`** (red) = **removed** from the old version
- Lines starting with **`+`** (green) = **added** in the new version
- `@@ -1,7 +1,10 @@` means: old file starts at line 1 showing 7 lines, new file starts at line 1 showing 10 lines

### Different Types of `git diff`

This is where it gets really useful — you can compare different things:

```
┌─────────────────┐              ┌─────────────────┐              ┌─────────────────┐
│   WORKING       │   git diff   │   STAGING AREA  │ git diff     │   REPOSITORY    │
│   DIRECTORY     │ ◄──────────► │   (Index)       │ --staged     │   (Last Commit) │
│                 │              │                 │ ◄──────────► │                 │
└─────────────────┘              └─────────────────┘              └─────────────────┘
                                                     
       git diff HEAD ◄──────────────────────────────────────────►
       (Working Directory vs Last Commit)
```

#### 1. Working Directory vs Staging Area (Default)

```bash
git diff
```

Shows changes in your working directory that are **NOT yet staged**.

#### 2. Staging Area vs Last Commit

```bash
git diff --staged
# or (older syntax, same thing)
git diff --cached
```

Shows what's in the staging area that will be included in the **next commit**.

#### 3. Working Directory vs Last Commit

```bash
git diff HEAD
```

Shows **ALL** changes (both staged and unstaged) compared to the last commit.

#### 4. Compare Two Specific Commits

```bash
git diff a1b2c3d b4c5d6e
```

Shows the differences between two commits.

#### 5. Diff for a Specific File

```bash
git diff hello.txt
git diff --staged app.py
git diff HEAD README.md
```

### Practical Example: Understanding the Three Diffs

```bash
# Start with a clean state
echo "Line 1" > test.txt
git add test.txt
git commit -m "Add test.txt"

# Now modify the file
echo "Line 2" >> test.txt

# git diff → Shows "Line 2" was added (Working vs Staging)
git diff
# Output: +Line 2

# Stage the change
git add test.txt

# git diff → Shows NOTHING (Working = Staging now)
git diff
# Output: (empty — no difference between working dir and staging)

# git diff --staged → Shows "Line 2" was added (Staging vs Last Commit)
git diff --staged
# Output: +Line 2

# Now modify again WITHOUT staging
echo "Line 3" >> test.txt

# git diff → Shows "Line 3" (Working vs Staging)
git diff
# Output: +Line 3

# git diff --staged → Still shows "Line 2" (Staging vs Last Commit)
git diff --staged
# Output: +Line 2

# git diff HEAD → Shows BOTH "Line 2" AND "Line 3" (Working vs Last Commit)
git diff HEAD
# Output: +Line 2
#         +Line 3
```

This example really shows the power of the staging area!

### Useful `git diff` Options

| Command | What It Compares |
|---------|-----------------|
| `git diff` | Working Directory vs Staging Area |
| `git diff --staged` | Staging Area vs Last Commit |
| `git diff HEAD` | Working Directory vs Last Commit |
| `git diff commit1 commit2` | Two specific commits |
| `git diff branch1..branch2` | Two branches |
| `git diff -- filename` | Diff for a specific file |

#### Stat View (Summary Only)

```bash
git diff --stat
```

**Output:**
```
 app.py     | 5 +++--
 hello.txt  | 2 +-
 2 files changed, 4 insertions(+), 3 deletions(-)
```

Shows just a summary — which files changed and how many lines were added/removed.

#### Word-Level Diff

```bash
git diff --word-diff
```

Instead of showing entire lines as changed, this highlights only the **specific words** that changed:

```
Hello, [-Git!-]{+Git! Welcome to version control.+}
```

- `[-...-]` = removed words (red)
- `{+...+}` = added words (green)

#### Name-Only Diff

```bash
git diff --name-only
```

Shows only the **file names** that changed:

```
app.py
hello.txt
```

#### Name and Status Diff

```bash
git diff --name-status
```

**Output:**
```
M  app.py        ← Modified
A  new-file.txt  ← Added
D  old-file.txt  ← Deleted
```

```bash
# =============================================

# =============================================
# Step 6: Commit!

# Todo App

A simple command-line todo application built with Python.

## Features
- Add tasks
- View all tasks

git commit -m "docs: add README with project description"

# =============================================
# Step 8: Modify app.py — add a delete function
# =============================================
cat > app.py << 'EOF'
# Simple Todo App

todos = []

def add_todo(task):
    todos.append(task)
    print(f"Added: {task}")

def delete_todo(index):
    if 0 < index <= len(todos):
        removed = todos.pop(index - 1)
        print(f"Deleted: {removed}")
    else:
        print("Invalid index!")

def show_todos():
    if not todos:
        print("No tasks yet!")
        return
    for i, task in enumerate(todos, 1):
        print(f"{i}. {task}")

# Main
add_todo("Learn Git")
add_todo("Practice commits")
show_todos()
EOF

# =============================================
# Step 9: See what changed
# =============================================
git diff
# Shows the exact changes to app.py

git diff --stat
# Output: app.py | 14 ++++++++++++--

# =============================================
# Step 10: Stage and commit the changes
# =============================================
git add app.py
git diff --staged    # Verify what we're about to commit
git commit -m "feat: add delete function and improve show_todos"

# =============================================
# Step 11: View the full history
# =============================================
git log --oneline
# Output:
# f6g7h8i (HEAD -> main) feat: add delete function and improve show_todos

# a1b2c3d feat: initial todo app with add and show functions

git log --oneline --stat
# Shows which files changed in each commit
```

## 🚫 `.gitignore` — Telling Git What to Ignore

While we're talking about basics, let's cover `.gitignore`. This file tells Git which files or folders to **completely ignore** — don't track them, don't show them in `git status`.

### Why Do We Need This?

Some files should **never** be in your repository:

| File Type | Why Ignore It |
|-----------|--------------|
| `node_modules/` | Dependencies — too large, reinstall with `npm install` |
| `__pycache__/` | Python compiled bytecode — auto-generated |
| `.env` | Environment variables — **contains secrets!** |
| `*.log` | Log files — too large, not useful in Git |
| `.DS_Store` | macOS folder metadata — useless in Git |
| `build/`, `dist/` | Build output — regenerated from source |
| `*.pyc` | Python compiled files |
| `.idea/`, `.vscode/` | IDE settings (sometimes shared, sometimes not) |

### Creating a `.gitignore` File

```bash
cat > .gitignore << 'EOF'
# Python
__pycache__/
*.pyc
*.pyo
*.egg-info/
dist/
build/
venv/
.env

# Node.js
node_modules/
npm-debug.log

# IDE
.idea/
.vscode/
*.swp
*.swo

# OS
.DS_Store
Thumbs.db

# Logs
*.log

# Compiled output
*.o
*.so
*.exe
EOF

git add .gitignore
git commit -m "chore: add gitignore file"
```

### `.gitignore` Pattern Rules

| Pattern | What It Matches |
|---------|----------------|
| `*.log` | Any file ending in `.log` |
| `build/` | The `build` directory and everything inside |
| `!important.log` | **Don't** ignore this file (exception) |
| `doc/*.txt` | `.txt` files in `doc/` but not in subdirectories |
| `doc/**/*.txt` | `.txt` files in `doc/` and ALL subdirectories |
| `/TODO` | Only the `TODO` file in the root (not `subdir/TODO`) |
| `temp-*` | Any file starting with `temp-` |

### Pro Tip: Global Gitignore

For files you always want to ignore (like `.DS_Store` on macOS), set a global gitignore:

```bash
git config --global core.excludesfile ~/.gitignore_global
```

Then add your global ignores to `~/.gitignore_global`:
```
.DS_Store
Thumbs.db
*.swp
.idea/
```

### Already Tracked a File That Should Be Ignored?

If you added a file to Git before adding it to `.gitignore`, Git will keep tracking it. To fix this:

```bash
# Remove from Git tracking (but keep the file on disk)
git rm --cached filename.txt

# For a directory
git rm -r --cached node_modules/

# Then commit
git commit -m "chore: remove tracked files that should be ignored"
```

## 📝 Quick Summary

| Command | What It Does | Most Common Usage |
|---------|-------------|-------------------|
| `git init` | Creates a new Git repository | `git init` (run once per project) |
| `git status` | Shows current state of working dir & staging | `git status` or `git status -s` |
| `git add` | Stages changes for the next commit | `git add .` or `git add filename` |
| `git commit` | Saves staged changes as a permanent snapshot | `git commit -m "message"` |
| `git log` | Shows commit history | `git log --oneline` |
| `git diff` | Shows exact changes made to files | `git diff` or `git diff --staged` |
| `.gitignore` | Tells Git which files to never track | Create once, update as needed |

### The Daily Workflow (You'll Do This 100 Times)

```bash
# 1. Make changes to files
# 2. Check what changed
git status
git diff

# 3. Stage the changes
git add .

# 4. Commit with a good message
git commit -m "feat: describe what you did"

# 5. Check the history
git log --oneline
```
