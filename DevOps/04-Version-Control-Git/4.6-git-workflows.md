# 4.6 — Git Workflows — Feature Branch, Git Flow, Trunk-Based Development

> **One Line Answer:** A Git workflow is a **recipe or strategy** your team agrees on for how to use branches, when to merge, and how to release code — picking the right one keeps your team organized, avoids chaos, and makes deployments smooth.

---

## 🤔 Why Do You Need a Git Workflow?

Git is incredibly flexible — you can branch, merge, and tag however you want. But that flexibility is a **double-edged sword**.

Without an agreed-upon workflow:

| Problem | What Happens |
|---------|-------------|
| Everyone pushes to `main` directly | 💥 Broken code in production |
| No one knows which branch to work on | Confusion, duplicated effort |
| Releases are chaotic | "Wait, which commits go in this release?" |
| Merge conflicts everywhere | Multiple people working on the same branch |
| No code review process | Bugs slip into production |
| Hotfixes are messy | "Do we branch from main? develop? release?" |

A Git workflow is like **traffic rules** — without them, everyone crashes into each other. With them, traffic flows smoothly.

---

## 📚 The Four Most Common Git Workflows

Here's a quick overview before we deep-dive into each one:

| Workflow | Complexity | Best For | Key Idea |
|----------|-----------|----------|----------|
| **Feature Branch** | Simple | Small teams, any project | Every feature gets its own branch |
| **Git Flow** | Complex | Large teams, versioned releases | Multiple long-lived branches with strict rules |
| **Trunk-Based Development** | Simple | CI/CD-heavy teams, fast releases | Everyone commits to `main` (trunk) frequently |
| **Forking Workflow** | Moderate | Open source projects | Everyone works on their own fork |

Let's explore each one in detail.

---

## 1️⃣ Feature Branch Workflow

### The Idea

This is the **simplest and most popular** workflow. The rule is simple:

> **Every new feature, bugfix, or change gets its own branch. Nobody works directly on `main`.**

That's it. That's the whole workflow.

### How It Works

```
main:      ─────●─────────────●─────────────●─────── (always stable)
                 ↘             ↗             ↗
feature-A:        ●────●──────              /
                       ↘                   /
feature-B:              ●────●────●───────
```

**The Steps:**

```bash
# 1. Start from main (always get the latest)
git switch main
git pull origin main

# 2. Create a feature branch
git switch -c feature-user-dashboard

# 3. Do your work — make commits
git add .
git commit -m "Add dashboard layout"
git add .
git commit -m "Add charts to dashboard"

# 4. Push your branch to the remote
git push -u origin feature-user-dashboard

# 5. Create a Pull Request (PR) on GitHub/GitLab
#    → Team reviews your code
#    → Suggest changes if needed
#    → Approve when ready

# 6. Merge the PR into main (usually done on GitHub UI)

# 7. Clean up
git switch main
git pull origin main
git branch -d feature-user-dashboard
```

### Rules of the Feature Branch Workflow

| Rule | Why |
|------|-----|
| `main` is always deployable | You can deploy from `main` at any time |
| Every change starts with a new branch | Keeps `main` clean |
| Use Pull Requests for merging | Enables code review |
| Branch names should be descriptive | `feature-login` not `my-branch` |
| Delete branches after merging | Keeps the repo clean |
| Pull latest `main` before branching | Avoids outdated code |

### When to Use It

✅ Small to medium teams (2-20 developers)
✅ Projects with continuous deployment
✅ When you want simplicity
✅ When every commit to `main` can be deployed

### Branch Naming Conventions

```
feature/user-authentication
feature/payment-integration
bugfix/login-redirect-loop
hotfix/critical-security-fix
docs/api-documentation
chore/update-dependencies
test/add-unit-tests
refactor/database-queries
```

## 2️⃣ Git Flow Workflow

### The Idea

Git Flow is a **more structured and formal** workflow created by **Vincent Driessen** in 2010. It uses **multiple long-lived branches** with strict rules about when and how to merge.

> **Git Flow is designed for projects that have scheduled releases** (like v1.0, v2.0, etc.)

### The Branch Structure

Git Flow has **five types of branches**:

| Branch | Lifetime | Purpose |
|--------|----------|---------|
| `main` | Permanent | Production-ready code. Every commit here is a release. |
| `develop` | Permanent | Integration branch. All features merge here first. |
| `feature/*` | Temporary | Individual features. Branches off `develop`, merges back into `develop`. |
| `release/*` | Temporary | Preparing a release. Branches off `develop`, merges into `main` AND `develop`. |
| `hotfix/*` | Temporary | Emergency production fixes. Branches off `main`, merges into `main` AND `develop`. |

### Visual Diagram

```
main:      ─────v1.0────────────────────v2.0──────────v2.0.1───
                  ↑                       ↑               ↑
                  │                       │               │
release:          │            release/2.0─●──●           │
                  │           ↗                ↘          │
develop:   ───●───●──●──●──●────●──●──●─────────●──●──●──●───
                     ↗  ↗     ↗       ↗               ↗
features:    feat-A──  feat-B─  feat-C─  feat-D──     │
                                                       │
hotfix:                                        hotfix/2.0.1─
```

### How It Works — Step by Step

#### Setting Up Git Flow

```bash
# Initialize a project with Git Flow
# (You need the git-flow extension, or you can do it manually)
git flow init

# Or manually: just create the develop branch
git switch main
git switch -c develop
git push -u origin develop
```

#### Working on a Feature

```bash
# 1. Start from develop (NOT main!)
git switch develop
git pull origin develop

# 2. Create a feature branch
git switch -c feature/shopping-cart

# 3. Do your work
git add .
git commit -m "Add shopping cart functionality"
git add .
git commit -m "Add cart item counter"

# 4. Push your feature branch
git push -u origin feature/shopping-cart

# 5. Create a PR to merge into DEVELOP (not main!)

# 6. After PR is approved and merged, clean up
git switch develop
git pull origin develop
git branch -d feature/shopping-cart
```

#### Creating a Release

When `develop` has enough features for a release:

```bash
# 1. Branch off from develop
git switch develop
git pull origin develop
git switch -c release/2.0.0

# 2. On this branch: bump version numbers, update changelogs, final bug fixes
echo "2.0.0" > VERSION
git add .
git commit -m "Bump version to 2.0.0"

# 3. Fix any last-minute bugs on this branch
git add .
git commit -m "Fix typo in release notes"

# 4. Merge into MAIN (this is the actual release!)
git switch main
git merge release/2.0.0
git tag -a v2.0.0 -m "Release version 2.0.0"
git push origin main --tags

# 5. Also merge back into DEVELOP (so develop gets the fixes too)
git switch develop
git merge release/2.0.0
git push origin develop

# 6. Delete the release branch
git branch -d release/2.0.0
git push origin --delete release/2.0.0
```

#### Creating a Hotfix

Something broke in production? Emergency fix!

```bash
# 1. Branch off from MAIN (the production code)
git switch main
git pull origin main
git switch -c hotfix/fix-payment-crash

# 2. Fix the bug
git add .
git commit -m "Fix payment processing crash"

# 3. Merge into MAIN
git switch main
git merge hotfix/fix-payment-crash
git tag -a v2.0.1 -m "Hotfix: payment crash"
git push origin main --tags

# 4. Also merge into DEVELOP (so the fix is in future development too)
git switch develop
git merge hotfix/fix-payment-crash
git push origin develop

# 5. Clean up
git branch -d hotfix/fix-payment-crash
```

### Git Flow Summary Table

| Action | Branch From | Merge Into | Who Does It |
|--------|------------|-----------|-------------|
| Start a feature | `develop` | `develop` | Developer |
| Start a release | `develop` | `main` + `develop` | Release manager |
| Start a hotfix | `main` | `main` + `develop` | On-call developer |

### When to Use Git Flow

✅ Large teams with multiple developers
✅ Projects with **scheduled, versioned releases** (v1.0, v2.0)
✅ Enterprise software, mobile apps
✅ When you need strict separation between development and production

### When NOT to Use Git Flow

❌ Small teams that deploy continuously
❌ Web apps that deploy multiple times a day
❌ When it feels like too much overhead (it often is!)

### Honest Opinion

Git Flow was revolutionary in 2010, but many teams today find it **too complex**. If you're doing CI/CD and deploying frequently, **Trunk-Based Development** (below) is usually a better choice. Git Flow shines when you have distinct release versions (like mobile apps or installable software).

## 3️⃣ Trunk-Based Development (TBD)

### The Idea

Trunk-Based Development is the **opposite of Git Flow** in terms of complexity. The core principle:

> **Everyone commits to `main` (the "trunk") frequently. Branches live for hours or 1-2 days, not weeks.**

This is the workflow used by **Google, Facebook, Netflix, Amazon** — basically all the big tech companies that deploy hundreds of times per day.

### How It Works

```
main (trunk):  ──●──●──●──●──●──●──●──●──●──●──●──●──●──●──
                  ↑↗  ↑↗  ↑↗       ↑↗  ↑↗       ↑↗
features:        ●   ●   ●──●     ●   ●──●     ●──●
              (tiny, short-lived branches — merged within hours/1-2 days)
```

Notice how different this looks from Git Flow — there's no `develop`, no `release`, no complex branch hierarchy. Just `main` and tiny feature branches.

### The Steps

```bash
# 1. Pull the latest main
git switch main
git pull origin main

# 2. Create a SHORT-LIVED branch
git switch -c feature/add-search-bar

# 3. Do a SMALL, focused change (not a week-long feature!)
git add .
git commit -m "Add search bar component"

# 4. Push and create a PR
git push -u origin feature/add-search-bar

# 5. Get it reviewed and merged — SAME DAY ideally

# 6. The CI/CD pipeline automatically tests and deploys
```

### Key Principles of Trunk-Based Development

| Principle | Explanation |
|-----------|-------------|
| **Small, frequent commits** | Commit multiple times a day, not once a week |
| **Short-lived branches** | Branches live for hours, maybe 1-2 days max |
| **Feature flags** | Big features are hidden behind flags until ready |
| **CI/CD is mandatory** | Every commit triggers automated tests |
| **`main` is always deployable** | If tests pass, it can go to production |
| **No long-lived branches** | No `develop`, no `release` branches |

### Feature Flags — The Secret Sauce

"But wait," you might ask, "how do I work on a big feature that takes weeks if I have to merge into `main` daily?"

The answer: **Feature Flags** (also called feature toggles).

```python
# Example: Feature flag in code
if feature_flags.is_enabled("new-search-bar"):
    show_new_search_bar()
else:
    show_old_search_bar()
```

```javascript
// JavaScript example
if (featureFlags.isEnabled('newCheckoutFlow')) {
  renderNewCheckout();
} else {
  renderOldCheckout();
}
```

**How it works:**
1. You merge your incomplete feature into `main` every day
2. The feature is hidden behind a flag (turned off in production)
3. You keep adding to it daily, always behind the flag
4. When it's ready, you flip the flag ON
5. If something breaks, flip the flag OFF instantly — no rollback needed!

Feature flag services: **LaunchDarkly, Unleash, Flagsmith, Split.io**, or a simple config file.

### When to Use Trunk-Based Development

✅ Teams doing **continuous deployment** (deploying multiple times a day)
✅ Teams with **strong CI/CD pipelines** and automated testing
✅ Web applications and SaaS products
✅ Teams that want simplicity and speed
✅ Organizations that want to reduce merge conflicts

### When NOT to Use Trunk-Based Development

❌ Teams without automated tests (you'll break `main` constantly)
❌ Teams without CI/CD infrastructure
❌ Products that need versioned releases (use Git Flow instead)
❌ Open source projects with many external contributors

## 4️⃣ Forking Workflow

### The Idea

The Forking Workflow is mainly used in **open source projects**. Instead of everyone having write access to one repository, each contributor:

1. **Forks** the original repo (creates their own copy on GitHub)
2. Works on their fork
3. Submits a **Pull Request** to the original repo

> **Nobody except maintainers can push to the original repository.**

### How It Works

```
Original Repo (upstream):   ──●──●──●──●──●──●──
                                        ↑
                              Pull Request │
                                        │
Your Fork (origin):         ──●──●──●──●──●──
                                  ↗
Your Local Machine:         ──●──●──●──
```

### The Steps

```bash
# 1. FORK the original repo on GitHub (click the "Fork" button)

# 2. CLONE your fork to your local machine
git clone git@github.com:YOUR-USERNAME/project.git
cd project

# 3. ADD the original repo as "upstream"
git remote add upstream https://github.com/ORIGINAL-OWNER/project.git

# 4. Verify your remotes
git remote -v
# origin    git@github.com:YOUR-USERNAME/project.git (fetch)
# origin    git@github.com:YOUR-USERNAME/project.git (push)
# upstream  https://github.com/ORIGINAL-OWNER/project.git (fetch)
# upstream  https://github.com/ORIGINAL-OWNER/project.git (push)

# 5. Create a feature branch (on your fork)
git switch -c fix-typo-in-readme

# 6. Make your changes
git add .
git commit -m "Fix typo in README.md"

# 7. Push to YOUR FORK (origin), NOT the original (upstream)
git push origin fix-typo-in-readme

# 8. Go to GitHub → Create a Pull Request from your fork to the original repo

# 9. The maintainers review and merge (or request changes)
```

### Keeping Your Fork Up to Date

The original repo keeps getting new commits from other contributors. You need to sync your fork:

```bash
# 1. Fetch the latest from the original repo (upstream)
git fetch upstream

# 2. Switch to your main branch
git switch main

# 3. Merge upstream's main into your main
git merge upstream/main

# 4. Push the updates to your fork (origin)
git push origin main
```

**Do this regularly** to avoid your fork falling behind!

### When to Use the Forking Workflow

✅ Open source projects (Linux kernel, Kubernetes, React, etc.)
✅ When you don't have write access to the original repo
✅ Large projects with many external contributors
✅ When maintainers want to gatekeep what goes in

## 🔄 Comparing All Workflows — Side by Side

| Feature | Feature Branch | Git Flow | Trunk-Based | Forking |
|---------|---------------|----------|-------------|---------|
| **Complexity** | Low | High | Low | Medium |
| **Main branches** | `main` | `main` + `develop` | `main` (trunk) | `main` (per fork) |
| **Branch lifetime** | Days to weeks | Days to weeks | Hours to 1-2 days | Days to weeks |
| **Release process** | Deploy from `main` | Release branches + tags | Deploy every commit | PR to upstream |
| **Best for** | Small-mid teams | Large teams, versioned | CI/CD teams, SaaS | Open source |
| **Merge frequency** | Weekly | Per feature | Multiple times/day | Per contribution |
| **Requires CI/CD?** | Recommended | Recommended | **Mandatory** | Optional |
| **Feature flags?** | Optional | Not needed | **Essential** | Not needed |
| **Code review** | Via PRs | Via PRs | Via PRs (fast) | Via PRs to upstream |

## 🛡️ Branch Protection Rules — Enforcing Your Workflow

No matter which workflow you choose, you should **protect your important branches** (like `main`). GitHub, GitLab, and Bitbucket all support this.

### Setting Up Branch Protection on GitHub

Go to: **Repository → Settings → Branches → Add branch protection rule**

| Rule | What It Does | Recommended? |
|------|-------------|--------------|
| **Require pull request reviews** | No one can push directly — must go through a PR | ✅ Yes |
| **Require status checks** | CI/CD must pass before merging | ✅ Yes |
| **Require conversation resolution** | All review comments must be addressed | ✅ Yes |
| **Require signed commits** | Commits must be GPG-signed | Optional |
| **Include administrators** | Even admins follow the rules | ✅ Yes |
| **Restrict who can push** | Only specific people/teams can push | For large teams |
| **Require linear history** | No merge commits (rebase only) | For trunk-based |

### Example: Minimal Protection for `main`

```
✅ Require a pull request before merging
    ✅ Require 1 approval
✅ Require status checks to pass before merging
    ✅ Select your CI/CD checks
✅ Do not allow bypassing the above settings
```

This ensures:
1. Nobody can push directly to `main`
2. At least one person reviews the code
3. All automated tests pass
4. Not even admins can skip the rules

## 📋 Quick Reference — Workflow Cheat Sheet

### Feature Branch Workflow

```bash
git switch main && git pull
git switch -c feature/my-feature
# ... work, commit ...
git push -u origin feature/my-feature
# Create PR → Review → Merge → Delete branch
```

### Git Flow

```bash
# Feature
git switch develop && git pull
git switch -c feature/my-feature
# ... work, commit ...
# PR into develop

# Release
git switch develop
git switch -c release/1.0.0
# ... final fixes ...
# Merge into main + develop, tag

# Hotfix
git switch main
git switch -c hotfix/urgent-fix
# ... fix ...
# Merge into main + develop, tag
```

### Trunk-Based Development

```bash
git switch main && git pull
git switch -c small-change
# ... one small change ...
git push -u origin small-change
# PR → Review → Merge (same day!)
# CI/CD auto-deploys
```

### Forking Workflow

```bash
# Fork on GitHub, then:
git clone git@github.com:YOU/project.git
git remote add upstream https://github.com/OWNER/project.git
git switch -c fix-something
# ... work, commit ...
git push origin fix-something
# Create PR to upstream repo
```

## ⚡ Tips & Best Practices

| Tip | Why |
|-----|-----|
| Start with Feature Branch, evolve later | Don't over-engineer your workflow |
| Automate what you can | Branch protection, CI/CD, linting |
| Document your workflow | Write it down so everyone follows the same process |
| Keep branches short-lived | Long branches = painful merges |
| Review PRs quickly | Don't let PRs sit for days |
| Use conventional commit messages | `feat:`, `fix:`, `docs:`, `chore:` |
| Protect `main` (and `develop` in Git Flow) | Prevent accidental pushes |
| Use feature flags for big changes | Deploy incomplete code safely |
| Rebase before merging (when appropriate) | Cleaner history |
| Agree as a team | The best workflow is the one everyone follows |

---

## 🚫 Common Mistakes & How to Avoid Them

| Mistake | What Happens | How to Fix |
|---------|-------------|-----------|
| No agreed workflow | Everyone does their own thing → chaos | Pick one, write it down, enforce it |
| Using Git Flow for everything | Overhead kills speed for small teams | Use Feature Branch or Trunk-Based |
| Long-lived feature branches | Merge hell, tons of conflicts | Merge daily, use feature flags |
| No branch protection | Someone pushes broken code to `main` | Set up protection rules |
| Skipping code reviews | Bugs, bad patterns slip in | Require PR approvals |
| Not syncing with `main` regularly | Your branch diverges too far | Merge or rebase `main` into your branch daily |
| Mixing workflows | Confusion — "do we branch from `main` or `develop`?" | Stick to ONE workflow |
| No CI/CD with Trunk-Based | Broken code gets deployed | CI/CD is mandatory for TBD |

## 🎓 Key Takeaways

1. **A Git workflow = team agreement** on how to use branches and releases.
2. **Feature Branch Workflow** — Simple, universal, great starting point. Every change = a branch + PR.
3. **Git Flow** — Structured, uses `main` + `develop` + release/hotfix branches. Best for versioned releases.
4. **Trunk-Based Development** — Everyone merges to `main` frequently. Fast, requires strong CI/CD + feature flags.
5. **Forking Workflow** — Each contributor has their own copy. Standard for open source.
6. **Start simple** — Feature Branch is enough for most teams. Add complexity only when needed.
7. **Protect your `main` branch** — require PRs, reviews, and passing CI checks.
8. **Short-lived branches** — no matter the workflow, merge early and often.
9. **Feature flags** unlock Trunk-Based Development for big features.
10. **The best workflow is the one your team actually follows consistently.**
