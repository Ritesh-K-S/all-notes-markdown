# 4.7 — Pull Requests / Merge Requests — Code Reviews Best Practices

> **One Line Answer:** A Pull Request (PR) or Merge Request (MR) is a **formal way to propose, review, discuss, and approve code changes** before they get merged into the main branch — it's the backbone of collaborative development and quality control.

---

## 🤔 Why Should You Care About Pull Requests?

Let's say you're working in a team of 5 developers. Everyone is writing code. Now imagine **everyone just pushes code directly to the `main` branch** without anyone reviewing it.

What happens?

```
Developer A pushes broken code → 💥 Production crashes
Developer B pushes a security flaw → 🔓 Data breach
Developer C overwrites Developer D's code → 😡 Work lost
Nobody knows what changed or why → 🤷 Chaos
```

**Pull Requests solve this problem.** They create a structured process where:

- Someone **proposes** changes
- Others **review** the code
- The team **discusses** any concerns
- Only **approved** code gets merged

Think of it like an **editor reviewing a newspaper article** before it gets published. No one publishes raw, unedited work. Same with code.

---

## 📝 What Exactly is a Pull Request?

A **Pull Request (PR)** — called **Merge Request (MR)** in GitLab — is a request that says:

> "Hey team, I've made some changes on my branch. Can you **review** them and, if they look good, **merge** them into the main branch?"

### The Basic Idea

```
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│   You work on your branch:   feature/add-login-page                │
│                                                                     │
│   You finish your work → You create a Pull Request                 │
│                                                                     │
│   Your team reviews your code → They suggest changes or approve    │
│                                                                     │
│   Once approved → Your branch gets merged into main                │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### Real-World Analogy

Imagine you wrote an essay. Instead of submitting it directly:

1. You give it to a **friend to review** (the reviewer)
2. They **highlight mistakes**, suggest improvements
3. You **fix** the issues
4. They say **"looks good!"**
5. You **submit** the final version

That's exactly what a PR does for code.

---

## 🏷️ PR vs MR — What's the Difference?

| Platform | What They Call It | Abbreviation |
|----------|------------------|-------------|
| **GitHub** | Pull Request | PR |
| **GitLab** | Merge Request | MR |
| **Bitbucket** | Pull Request | PR |
| **Azure DevOps** | Pull Request | PR |

**They are the EXACT same thing** — just different names. The concept, workflow, and purpose are identical. GitLab chose "Merge Request" because it describes what happens at the end (merging), while GitHub's "Pull Request" describes what you're asking (pull my changes in).

Throughout this note, we'll use **PR** — but everything applies to MRs too.

## 🔄 The Complete Pull Request Workflow

Let's walk through the entire lifecycle of a Pull Request, step by step.

### Step 1: Create a Feature Branch

```bash
# Make sure you're on the latest main branch
git checkout main
git pull origin main

# Create a new branch for your work
git checkout -b feature/user-authentication
```

### Step 2: Make Your Changes and Commit

```bash
# Make your changes...
# Edit files, write code, etc.

# Stage your changes
git add .

# Commit with a clear message
git commit -m "feat: add user authentication with JWT tokens"
```

### Step 3: Push Your Branch to Remote

```bash
git push origin feature/user-authentication
```

### Step 4: Create the Pull Request

Now go to your Git platform (GitHub/GitLab/Bitbucket) and create a PR.

**On GitHub:**
- Go to your repository
- You'll see a banner: "feature/user-authentication had recent pushes — Compare & pull request"
- Click it!

**Or manually:**
- Click "Pull requests" tab → "New pull request"
- Select **base:** `main` ← **compare:** `feature/user-authentication`
- Click "Create pull request"

**Using GitHub CLI:**

```bash
# Create a PR directly from the terminal
gh pr create --title "feat: add user authentication" \
             --body "Added JWT-based authentication for users" \
             --base main \
             --head feature/user-authentication
```

### Step 5: Fill in the PR Details

A good PR includes:

```markdown
## Title
feat: add user authentication with JWT tokens

## Description
- Added login and signup endpoints
- Implemented JWT token generation and validation
- Added middleware for protected routes
- Added unit tests for auth service

## Type of Change
- [x] New feature
- [ ] Bug fix
- [ ] Breaking change
- [ ] Documentation update

## How to Test
1. Run `npm install`
2. Run `npm test` to see all tests pass
3. Start the server with `npm start`
4. Hit POST /api/login with test credentials

## Screenshots (if applicable)
N/A

## Checklist
- [x] Code follows project style guidelines
- [x] Self-reviewed the code
- [x] Added tests
- [x] All tests pass
- [x] Updated documentation
```

### Step 6: Code Review Happens

- Reviewers look at your code
- They leave comments, suggestions, and questions
- They might request changes

### Step 7: Address Feedback

```bash
# Make changes based on review feedback
# Edit your files...

git add .
git commit -m "fix: address review feedback - add input validation"
git push origin feature/user-authentication
```

The PR **automatically updates** with your new commits. No need to create a new PR!

### Step 8: Approval & Merge

Once reviewers approve:

```
Reviewer clicks "Approve" 
     ↓
All checks pass (CI/CD, tests, linting)
     ↓
Click "Merge pull request"
     ↓
Your code is now in main! 🎉
```

### Step 9: Clean Up

```bash
# Switch back to main
git checkout main

# Pull the latest (with your merged changes)
git pull origin main

# Delete your local branch (it's merged, you don't need it)
git branch -d feature/user-authentication

# Delete the remote branch (optional — many platforms do this automatically)
git push origin --delete feature/user-authentication
```

## 🔀 Merge Strategies — How Your PR Actually Gets Merged

When you click "Merge", there are **three common options**. Understanding them matters because they affect your Git history.

### 1. Merge Commit (Default)

```bash
# What happens behind the scenes
git merge feature/user-authentication
```

```
main:     A --- B --- C --------- M  (merge commit)
                       \         /
feature:                D --- E
```

- Creates a **merge commit** (M) that ties the branches together
- **Preserves the full history** — you can see exactly when the branch was created and merged
- **Best for:** Teams that want complete history
- **Downside:** Can make the history look cluttered with many merge commits

### 2. Squash and Merge

```bash
# What happens behind the scenes
git merge --squash feature/user-authentication
git commit -m "feat: add user authentication"
```

```
Before squash:
feature:  D --- E --- F  (3 commits)

After squash merge:
main:     A --- B --- C --- S  (one single squashed commit)
```

- **Combines all your branch commits into ONE commit** on main
- Makes the main branch history **clean and easy to read**
- **Best for:** When your branch has messy "work in progress" commits
- **Downside:** You lose the individual commit history from the branch

### 3. Rebase and Merge

```bash
# What happens behind the scenes
git rebase main feature/user-authentication
git checkout main
git merge feature/user-authentication  # fast-forward
```

```
Before rebase:
main:     A --- B --- C
                       \
feature:                D --- E

After rebase and merge:
main:     A --- B --- C --- D' --- E'  (linear history)
```

- **Replays your commits on top of main** — creates a straight-line history
- No merge commits at all
- **Best for:** Teams that love a clean, linear history
- **Downside:** Rewrites commit hashes (D and E become D' and E')

### Which One to Choose?

| Strategy | When to Use |
|----------|------------|
| **Merge Commit** | You want complete, unmodified history |
| **Squash and Merge** | Your branch has messy/WIP commits and you want a clean main |
| **Rebase and Merge** | You want a perfectly linear history |

> 💡 **Most teams use "Squash and Merge"** because it keeps `main` clean while individual branch work can be messy.

## 👀 Code Review — The Heart of Pull Requests

A PR without a code review is just a formality. The **real value** is in the review process.

### What is a Code Review?

A code review is when **one or more team members read through your code changes**, looking for:

| What They Check | Why |
|----------------|-----|
| **Bugs** | Does the code have logical errors? |
| **Security** | Is there a SQL injection? Hardcoded secrets? |
| **Performance** | Will this query be slow with 1 million records? |
| **Readability** | Can someone else understand this code in 6 months? |
| **Best Practices** | Does it follow the team's coding standards? |
| **Edge Cases** | What happens with empty input? Null values? |
| **Tests** | Are there enough tests? Do they cover important cases? |
| **Architecture** | Does this fit the overall system design? |

### The Reviewer's Toolkit (Types of Comments)

When reviewing code on GitHub/GitLab, you can leave different types of feedback:

**1. Comment** — Just a note, no action required

```
💬 "Nice approach! I like how you handled the edge case here."
```

**2. Suggestion** — A specific code change recommendation

On GitHub, you can use the **suggestion** feature:

```suggestion
// Instead of this
const result = data.filter(item => item.active == true);

// Suggest this
const result = data.filter(item => item.active === true);
```

The author can apply this suggestion with **one click**!

**3. Request Changes** — Something that MUST be fixed before merging

```
🔴 "This endpoint doesn't validate user input. We need to add 
    validation to prevent injection attacks."
```

**4. Approve** — Everything looks good!

```
✅ "LGTM (Looks Good To Me)! Great work."
```

## ✅ Code Review Best Practices — For REVIEWERS

### 1. Be Kind and Constructive

```
❌ Bad:  "This code is terrible. Did you even test this?"
✅ Good: "I think we could improve this by adding input validation. 
         What do you think about using a library like Joi for this?"
```

### 2. Explain the "Why," Not Just the "What"

```
❌ Bad:  "Use === instead of =="
✅ Good: "Use === instead of == because == does type coercion, which 
         can lead to unexpected bugs. For example, '0' == false is 
         true, but '0' === false is false."
```

### 3. Differentiate Between Must-Fix and Nice-to-Have

Use labels in your comments:

```
🔴 [MUST FIX] This has a SQL injection vulnerability
🟡 [SUGGESTION] Consider renaming this variable for clarity
🟢 [NIT] Minor formatting issue — extra space on line 42
❓ [QUESTION] Why did we choose this approach over using a queue?
```

### 4. Review in Small Batches

```
❌ 2000-line PR → "Looks good" (you definitely didn't review it properly)
✅ 200-line PR → Thorough, detailed review
```

**The golden rule:** The quality of a code review is inversely proportional to the size of the PR.

### 5. Check the Big Picture First, Then Details

```
Review Order:
1. Read the PR description — understand WHAT and WHY
2. Look at the file structure — does the organization make sense?
3. Check the architecture — does this fit the system design?
4. Review the logic — is the code correct?
5. Check edge cases — what could go wrong?
6. Look at tests — are important cases covered?
7. Check style — formatting, naming, conventions
```

### 6. Time-Box Your Reviews

- Don't spend more than **60-90 minutes** on a single review
- If it takes longer, the PR is probably **too big**
- Take **breaks** between reviews to stay fresh

## ✅ Code Review Best Practices — For AUTHORS (PR Creators)

### 1. Keep PRs Small and Focused

```
❌ Bad:  One PR that adds authentication + fixes 3 bugs + refactors the database layer
✅ Good: One PR per logical change:
         - PR #1: Add authentication
         - PR #2: Fix login bug
         - PR #3: Refactor database layer
```

**Ideal PR size: 200-400 lines of code changes.** Anything above 500 should be split.

### 2. Write a Clear Description

Your PR description should answer:

- **WHAT** did you change?
- **WHY** did you change it?
- **HOW** should someone test it?
- **ANYTHING** they should pay extra attention to?

### 3. Self-Review Before Requesting Review

Before asking others to review, review your OWN code first:

```bash
# Look at your changes
git diff main...feature/your-branch

# Check for common issues:
# - Leftover console.log / print statements?
# - Hardcoded values?
# - Missing error handling?
# - Commented-out code?
# - TODO comments that should be resolved?
```

### 4. Respond to Every Comment

```
Reviewer: "Can you add error handling for the case when the API is down?"

❌ Bad:  (no response, just pushes code)
✅ Good: "Great catch! Added a try-catch with retry logic. See commit abc123."
```

### 5. Don't Take Feedback Personally

Code review is about making the **code** better, not criticizing **you**.

```
Reviewer: "I think this function is doing too many things. Can we split it?"

❌ Bad:  "Are you saying I don't know how to code?"
✅ Good: "Good point! I split it into three focused functions — 
          processInput(), validateData(), and saveResult()."
```

## 🛡️ Branch Protection Rules

Most teams set up rules so that code **can't** be pushed directly to `main`. This forces everyone to go through the PR process.

### GitHub Branch Protection Settings

Go to: **Settings → Branches → Branch protection rules → Add rule**

| Setting | What It Does | Recommendation |
|---------|-------------|---------------|
| **Require pull request reviews** | Code must be reviewed before merging | ✅ Enable — set to 1-2 required reviewers |
| **Require status checks to pass** | CI/CD tests must pass before merging | ✅ Enable — ensures no broken code gets in |
| **Require conversation resolution** | All review comments must be resolved | ✅ Enable — prevents ignored feedback |
| **Require signed commits** | Commits must be GPG-signed | Optional — good for security-critical repos |
| **Include administrators** | Even admins must follow the rules | ✅ Enable — no one should bypass the process |
| **Restrict who can push** | Only certain people/teams can push | Enable for critical branches |
| **Require linear history** | No merge commits (rebase only) | Optional — depends on team preference |

### GitLab Equivalent

In GitLab, these are called **Protected Branches** and **Merge Request Approvals**:

- Go to **Settings → Repository → Protected branches**
- Set **Allowed to merge** and **Allowed to push**
- Go to **Settings → Merge requests → Approvals**
- Set required approvers

## 📋 PR Templates — Standardize Your PRs

You can create templates so that every new PR starts with a pre-filled structure.

### GitHub PR Template

Create a file at `.github/PULL_REQUEST_TEMPLATE.md`:

```markdown
## 📝 Description
<!-- What does this PR do? Why is it needed? -->

## 🔄 Type of Change
- [ ] 🐛 Bug fix
- [ ] ✨ New feature
- [ ] 💥 Breaking change
- [ ] 📝 Documentation update
- [ ] ♻️ Refactoring
- [ ] 🧪 Test addition

## 🧪 How to Test
<!-- Steps to test this PR -->
1. 
2. 
3. 

## 📸 Screenshots (if applicable)
<!-- Add screenshots here -->

## ✅ Checklist
- [ ] I have self-reviewed my code
- [ ] I have added tests that prove my fix/feature works
- [ ] All existing tests pass
- [ ] I have updated the documentation (if needed)
- [ ] My changes don't generate new warnings

## 🔗 Related Issues
<!-- Link related issues: Fixes #123, Relates to #456 -->
```

### GitLab MR Template

Create a file at `.gitlab/merge_request_templates/Default.md` with similar content.

Now, every time someone creates a new PR, they'll see this template pre-filled. No more lazy, empty PRs!

## 🤖 Automating PR Checks with CI/CD

A truly effective PR process isn't just humans reviewing code — it's also **automated checks** running on every PR.

### Common Automated Checks

```
PR Created
    ↓
┌─────────────────────────────────────────────┐
│  Automated Checks Run:                      │
│                                             │
│  ✅ Linting (ESLint, Pylint, etc.)          │
│  ✅ Unit Tests                              │
│  ✅ Integration Tests                       │
│  ✅ Code Coverage (must be > 80%)           │
│  ✅ Security Scan (Snyk, SonarQube)         │
│  ✅ Build succeeds                          │
│  ✅ Docker image builds successfully        │
│  ✅ No merge conflicts                      │
│                                             │
│  All pass? → Ready for human review         │
│  Any fail? → PR blocked until fixed         │
└─────────────────────────────────────────────┘
```

### Example: GitHub Actions for PR Checks

```yaml
# .github/workflows/pr-checks.yml
name: PR Checks

on:
  pull_request:
    branches: [main, develop]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: npm ci
      - run: npm run lint

  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: npm ci
      - run: npm test

  build:
    runs-on: ubuntu-latest
    needs: [lint, test]  # Only build if lint and tests pass
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: npm ci
      - run: npm run build
```

**What this does:**
- Every time a PR is created or updated against `main` or `develop`
- It automatically runs linting, tests, and build
- If any step fails, the PR shows a ❌ and can't be merged
- If all pass, it shows a ✅ and is ready for review

## 🏢 Real-World PR Workflow in a Team

Here's how a professional team typically handles PRs:

```
1. Developer picks up a Jira/GitHub issue
2. Creates a branch: feature/PROJ-123-add-user-dashboard
3. Writes code, commits regularly with clear messages
4. Pushes branch and creates a PR
5. Fills in the PR template
6. Assigns 1-2 reviewers
7. CI/CD runs automatically (tests, lint, build)
8. Reviewers review the code
9. Developer addresses feedback
10. Reviewers approve
11. PR is merged (usually squash-merge)
12. Branch is deleted
13. Deployment pipeline runs on main
14. Issue is marked as "Done"
```

## 📊 PR Metrics — Measuring Your Team's Health

Tracking PR metrics helps identify bottlenecks:

| Metric | What It Measures | Healthy Range |
|--------|-----------------|--------------|
| **PR Size** | Lines of code changed | < 400 lines |
| **Time to First Review** | How long until someone starts reviewing | < 4 hours |
| **Time to Merge** | Total time from PR creation to merge | < 24 hours |
| **Review Iterations** | Number of review rounds | 1-3 rounds |
| **PR Throughput** | Number of PRs merged per week per dev | 3-5 PRs |
| **Approval Rate** | Percentage of PRs approved on first review | > 70% |

If PRs are taking days to get reviewed, that's a sign of a process problem.

## ⚡ Tips for Faster PR Reviews

| Tip | Details |
|-----|---------|
| **Keep PRs small** | Smaller PRs get reviewed faster and more thoroughly |
| **Add context** | Screenshots, GIFs, before/after comparisons |
| **Tag the right reviewers** | Don't just tag everyone — tag people who know that area |
| **Use draft PRs** | Mark a PR as "Draft" if it's not ready for review yet |
| **Respond quickly** | When reviewers leave feedback, address it promptly |
| **Use CODEOWNERS** | Automatically assign reviewers based on file paths |

### CODEOWNERS File

Create `.github/CODEOWNERS`:

```
# Each line is a file pattern followed by the owner(s)

# Default owner for everything
*                       @team-lead

# Frontend code
/src/frontend/          @frontend-team
*.css                   @frontend-team
*.tsx                   @frontend-team

# Backend code
/src/backend/           @backend-team
/src/api/               @backend-team

# Infrastructure
/terraform/             @devops-team
/kubernetes/            @devops-team
Dockerfile              @devops-team

# Documentation
/docs/                  @tech-writers
*.md                    @tech-writers
```

Now when someone creates a PR that changes a Terraform file, the `@devops-team` is **automatically** assigned as reviewers!
