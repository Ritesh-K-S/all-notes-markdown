# 4.8 — Rebasing vs Merging — When to Use What

> **One Line Answer:** Both `git merge` and `git rebase` combine work from one branch into another, but **merge preserves history as it happened** (with merge commits), while **rebase rewrites history to make it look linear and clean** — and knowing when to use each one is a critical Git skill.

---

## 🤔 Why Does This Topic Matter?

This is one of the **most debated topics** in the Git world. Teams argue about it. Engineers have strong opinions. Job interviewers ask about it.

Here's the thing — **neither is "better."** They're tools for different situations. The key is understanding what each one does so you can make the right choice.

Let's break it down from the very basics.

---

## 📚 The Problem Both Solve

Imagine two developers working on the same project:

```
Timeline:
─────────────────────────────────────────────

1. Both start from the same point on "main"
2. Alice creates branch "feature-A" and makes 2 commits
3. Bob pushes 2 new commits to "main"
4. Alice wants to include Bob's changes in her branch

Question: HOW does Alice bring Bob's changes into her branch?
```

```
     Bob's new commits
          ↓   ↓
main:     A --- B --- C --- D
                       \
feature-A:              E --- F
                        ↑   ↑
                  Alice's commits
```

Alice has two choices:
1. **Merge** main into her branch
2. **Rebase** her branch on top of main

Let's see what each one does.

---

## 🔀 Git Merge — Combining With History

### What Merge Does

`git merge` takes two branches and creates a **new "merge commit"** that ties them together. It's like saying, "Here's where these two paths came back together."

### How to Do It

```bash
# Alice is on feature-A branch
git checkout feature-A

# She merges main into her branch
git merge main
```

### What Happens Visually

```
BEFORE merge:
main:       A --- B --- C --- D
                         \
feature-A:                E --- F

AFTER merge:
main:       A --- B --- C --- D
                         \       \
feature-A:                E --- F --- M  (merge commit)
```

That `M` is the **merge commit**. It has **two parents** — commit `D` (from main) and commit `F` (from feature-A). It's a special commit that says "these two lines of history came together here."

### A Real Example

```bash
# Let's walk through a complete example

# Start on main
git checkout main
echo "Initial content" > file.txt
git add file.txt
git commit -m "A: initial file"

# Create feature branch
git checkout -b feature-A
echo "Feature work 1" >> file.txt
git commit -am "E: feature work 1"
echo "Feature work 2" >> file.txt
git commit -am "F: feature work 2"

# Meanwhile, someone else updates main
git checkout main
echo "Main update 1" >> README.md
git add README.md
git commit -m "C: main update 1"
echo "Main update 2" >> README.md
git commit -am "D: main update 2"

# Now Alice merges main into her feature branch
git checkout feature-A
git merge main
# This opens an editor for the merge commit message
# Default message: "Merge branch 'main' into feature-A"
```

```bash
# Check the log — you'll see the merge commit
git log --oneline --graph

# Output:
# *   M  Merge branch 'main' into feature-A
# |\
# | * D  main update 2
# | * C  main update 1
# * | F  feature work 2
# * | E  feature work 1
# |/
# * B  ...
# * A  initial file
```

### Merge Pros and Cons

| ✅ Pros | ❌ Cons |
|---------|---------|
| **Non-destructive** — never changes existing commits | Creates extra merge commits that can clutter history |
| **Complete history** — you can see exactly when branches diverged and merged | Git log with many branches can look like a "train track" |
| **Safe** — no risk of losing work or breaking others' branches | Harder to read a linear story of what happened |
| **Easy** — hard to mess up | Can make `git bisect` slightly harder |

---

## 📏 Git Rebase — Rewriting for a Clean History

### What Rebase Does

`git rebase` takes your commits and **replays them on top of another branch**, as if you started your work from a later point. It literally rewrites your commit history to create a straight line.

### How to Do It

```bash
# Alice is on feature-A branch
git checkout feature-A

# She rebases onto main
git rebase main
```

### What Happens Visually

```
BEFORE rebase:
main:       A --- B --- C --- D
                         \
feature-A:                E --- F

AFTER rebase:
main:       A --- B --- C --- D
                                \
feature-A:                       E' --- F'
```

Notice something important: the commits are now `E'` and `F'` (with a prime/apostrophe). They're **new commits**! They have the same changes and the same messages, but they have **different commit hashes** because they now have a different parent (D instead of B).

**Rebase literally rewrites history.**

### A Real Example

```bash
# Same setup as before...
# Alice is on feature-A with commits E and F
# Main has new commits C and D

git checkout feature-A
git rebase main
```

What Git does behind the scenes:

```
Step 1: Git finds the common ancestor (B)
Step 2: Git "detaches" your commits (E, F) and saves them temporarily
Step 3: Git moves your branch pointer to the tip of main (D)
Step 4: Git replays your commits one by one on top of D

Internally:
1. Start at D (tip of main)
2. Apply E → creates E' (same changes, new hash, parent is D)
3. Apply F → creates F' (same changes, new hash, parent is E')
4. Move feature-A branch pointer to F'
```

```bash
# Check the log after rebase
git log --oneline --graph

# Output:
# * F'  feature work 2
# * E'  feature work 1
# * D   main update 2
# * C   main update 1
# * B   ...
# * A   initial file

# Beautiful! A perfectly straight line! No merge commit!
```

### Rebase Pros and Cons

| ✅ Pros | ❌ Cons |
|---------|---------|
| **Clean, linear history** — easy to read and follow | **Rewrites history** — changes commit hashes |
| **No merge commits** — the log tells a straight story | **Dangerous on shared branches** — can cause chaos for others |
| **Easier to use `git bisect`** | **Conflicts can be painful** — you resolve per-commit, not once |
| **Cleaner `git log`** | **Not safe if others have your old commits** |

## ⚠️ The Golden Rule of Rebasing

> **NEVER rebase a branch that other people are working on.**

This is the **most important rule** about rebase. Let me explain why.

### The Problem

```
Scenario: Alice and Bob are both working on feature-A

1. Alice and Bob both have commits E and F on feature-A

    Alice:   A --- B --- E --- F     (feature-A)
    Bob:     A --- B --- E --- F     (feature-A)

2. Alice rebases feature-A onto main (which has new commits C and D)

    Alice:   A --- B --- C --- D --- E' --- F'  (feature-A — rebased!)

3. Alice force pushes: git push --force origin feature-A

4. Now Bob tries to pull:
   
    Bob has:    E --- F      (old commits)
    Remote has: E' --- F'    (new commits — same changes, different hashes!)
    
    Git is CONFUSED! It sees commits it doesn't recognize.
    Bob's local history and the remote history have DIVERGED.

5. 💥 Chaos. Bob might lose work or create duplicate commits.
```

### The Rule Explained Simply

```
✅ SAFE to rebase:
   - Your OWN local branch that nobody else has
   - Before you've pushed (or before anyone has pulled your branch)

❌ NEVER rebase:
   - main, develop, or any shared branch
   - A branch that other people have pulled and are working on
   - After pushing, unless you're the ONLY one using that branch
```

Think of it this way:

- **Rebase = rewriting history**
- If you rewrite history that only YOU have → no problem
- If you rewrite history that OTHERS have → you just confused everyone

## 🔄 Side-by-Side Comparison

Let's see the exact same scenario handled by merge vs. rebase:

### Starting Point

```
main:       A --- B --- C
                   \
feature:            D --- E
```

### After Merge

```bash
git checkout feature
git merge main
```

```
main:       A --- B --- C
                   \       \
feature:            D --- E --- M
```

- Total commits on feature: D, E, M (3 commits)
- History shows: feature branched from B, main continued to C, then merged
- Commit hashes: D and E are **unchanged**

### After Rebase

```bash
git checkout feature
git rebase main
```

```
main:       A --- B --- C
                         \
feature:                  D' --- E'
```

- Total commits on feature: D', E' (2 commits)
- History shows: feature started from C (looks like it was always based on the latest main)
- Commit hashes: D and E are **changed** to D' and E'

## 🤼 Handling Conflicts

Both merge and rebase can cause **conflicts** when the same lines were changed in both branches. But they handle conflicts differently.

### Merge Conflicts

```bash
git checkout feature
git merge main
# CONFLICT! Git stops and asks you to resolve

# You see this in the file:
<<<<<<< HEAD (feature)
console.log("Hello from feature");
=======
console.log("Hello from main");
>>>>>>> main

# Fix the conflict by choosing what you want:
console.log("Hello from both feature and main");

# Then complete the merge:
git add conflicted-file.js
git commit  # Creates the merge commit
```

With merge, you resolve conflicts **once** — in the merge commit.

### Rebase Conflicts

```bash
git checkout feature
git rebase main
# CONFLICT! Git stops at the FIRST commit being replayed

# You see the same conflict markers:
<<<<<<< HEAD (main's latest)
console.log("Hello from main");
=======
console.log("Hello from feature - commit D");
>>>>>>> D (your first commit being replayed)

# Fix the conflict:
console.log("Hello from main and feature");

# Continue the rebase:
git add conflicted-file.js
git rebase --continue

# If there's ANOTHER conflict with the next commit (E), 
# you have to resolve THAT too!
```

With rebase, you might have to resolve conflicts **for each commit** being replayed. If you have 10 commits, you might resolve conflicts 10 times!

### Conflict Comparison

| Aspect | Merge | Rebase |
|--------|-------|--------|
| **How many times** | Once (in the merge commit) | Once per commit being replayed |
| **Difficulty** | Usually easier — one big resolution | Can be repetitive with many commits |
| **Abort** | `git merge --abort` | `git rebase --abort` |
| **Risk** | Low | Medium — resolving wrong can lose changes |

> 💡 **Tip:** If you have many small commits that might each conflict, consider **squashing** your commits first (into one or a few) before rebasing. This way you only resolve conflicts once.

## 🎯 Interactive Rebase — The Power Tool

Interactive rebase (`git rebase -i`) is one of the **most powerful Git features**. It lets you edit, reorder, squash, and clean up your commits before merging.

### How to Use It

```bash
# Interactive rebase for the last 4 commits
git rebase -i HEAD~4

# Or rebase interactively onto main
git rebase -i main
```

This opens your editor with something like:

```
pick abc1234 feat: add login form
pick def5678 fix: typo in login form
pick ghi9012 WIP: working on validation
pick jkl3456 feat: complete form validation

# Commands:
# p, pick = use commit as-is
# r, reword = use commit, but edit the commit message
# e, edit = use commit, but stop for amending
# s, squash = meld commit into previous commit (keep both messages)
# f, fixup = like squash, but discard this commit's message
# d, drop = remove commit entirely
# x, exec = run command
```

### Common Interactive Rebase Actions

**1. Squash messy commits into one clean commit:**

```
pick abc1234 feat: add login form
squash def5678 fix: typo in login form
squash ghi9012 WIP: working on validation
squash jkl3456 feat: complete form validation

# Result: All 4 commits become ONE commit
# You'll be asked to write a new combined commit message
```

**2. Reword a commit message:**

```
pick abc1234 feat: add login form
reword def5678 fix: tpyo in login form    ← fix this typo!
pick ghi9012 add validation

# Git will stop and let you edit the message for def5678
```

**3. Reorder commits:**

```
# Original order:
pick abc1234 feat: add login form
pick def5678 docs: update README
pick ghi9012 feat: add signup form

# Rearranged (just move the lines!):
pick abc1234 feat: add login form
pick ghi9012 feat: add signup form
pick def5678 docs: update README
```

**4. Drop a commit entirely:**

```
pick abc1234 feat: add login form
drop def5678 WIP: debugging stuff       ← remove this
pick ghi9012 feat: add signup form
```

### Real-World Interactive Rebase Workflow

Here's how many developers use interactive rebase before creating a PR:

```bash
# 1. You've been working on a feature with messy commits:
git log --oneline
# jkl3456 fix lint errors
# ghi9012 oops, forgot a file
# def5678 WIP - almost done
# abc1234 feat: start user dashboard

# 2. Clean them up before the PR:
git rebase -i HEAD~4

# 3. In the editor, squash everything into one clean commit:
pick abc1234 feat: start user dashboard
fixup def5678 WIP - almost done
fixup ghi9012 oops, forgot a file
fixup jkl3456 fix lint errors

# 4. Result: ONE clean commit
git log --oneline
# xyz9999 feat: start user dashboard

# 5. Now create your PR — clean and professional!
```

## 🏢 What Do Real Teams Use?

Different companies and teams have different preferences. Here's what's common:

### Strategy 1: "Always Merge" (Conservative)

```
Workflow:
1. Create feature branch from main
2. Work on feature branch
3. When ready, create PR
4. Merge (with merge commit) into main
5. Delete feature branch

Teams that use this: Beginners, teams that want complete history
```

### Strategy 2: "Rebase + Merge" (Balanced)

```
Workflow:
1. Create feature branch from main
2. Work on feature branch
3. Before PR, rebase onto latest main:
   git checkout feature
   git rebase main
4. Create PR
5. Merge (with merge commit) into main

Why: Your branch is up-to-date, and the merge commit marks when it was integrated
Teams that use this: Many open-source projects
```

### Strategy 3: "Rebase + Squash Merge" (Clean History)

```
Workflow:
1. Create feature branch from main
2. Work on feature branch (commit freely, messy is fine)
3. Create PR
4. Use "Squash and Merge" on the platform
5. Result: one clean commit on main per feature

Why: Main branch has a perfectly clean, one-commit-per-feature history
Teams that use this: Many professional teams, recommended for most projects
```

### Strategy 4: "Rebase Only" (Strict Linear History)

```
Workflow:
1. Create feature branch from main
2. Work on feature branch
3. Rebase onto latest main
4. Squash commits locally: git rebase -i main
5. Fast-forward merge into main (no merge commit)

Why: Perfectly linear history, no merge commits at all
Teams that use this: Teams with strict history requirements
```

## 📋 Quick Decision Guide

```
Should I merge or rebase?
          │
          ▼
Is it a shared branch (main, develop)?
          │
     ┌────┴────┐
    YES        NO
     │          │
     ▼          ▼
  ALWAYS     Is it just YOUR branch?
  MERGE       │
   (never    ┌┴──────┐
   rebase   YES      NO (shared feature branch)
   shared    │        │
   branches) ▼        ▼
          REBASE    MERGE
          is safe   (or coordinate with teammates)
```

### The Simple Rules

| Situation | Use |
|-----------|-----|
| Updating your local feature branch with latest main | **Rebase** — `git rebase main` |
| Merging a finished feature into main (via PR) | **Merge** (or squash merge) |
| Cleaning up your commits before a PR | **Interactive rebase** — `git rebase -i` |
| Combining a shared branch into another shared branch | **Merge** — never rebase |
| You're unsure and don't want to risk anything | **Merge** — it's always safe |

## 🧪 Practice Scenarios

### Scenario 1: Update Feature Branch

```bash
# You're on feature-A, and main has new commits you need

# Option A: Merge (safe, adds merge commit)
git checkout feature-A
git merge main

# Option B: Rebase (clean, rewrites history)
git checkout feature-A
git rebase main
# If conflicts: resolve → git add . → git rebase --continue

# If you already pushed feature-A before rebasing:
git push --force-with-lease origin feature-A
# --force-with-lease is SAFER than --force (it checks if someone else pushed)
```

### Scenario 2: Clean Up Before PR

```bash
# Your branch has 7 messy commits, clean them up

git rebase -i main
# Squash/fixup the messy ones
# Reword the remaining ones to be clear

# Then push (force if already pushed)
git push --force-with-lease origin feature-A
```

### Scenario 3: Undo a Bad Rebase

```bash
# Oh no, the rebase went wrong! 

# Use reflog to go back in time
git reflog
# Find the commit hash BEFORE the rebase started

git reset --hard HEAD@{5}  # or whatever number from reflog
# You're back to before the rebase! Crisis averted.
```

## 🔑 `--force` vs `--force-with-lease`

After rebasing, you need to force push because the history has changed. But there are two ways:

```bash
# ❌ Dangerous — overwrites remote no matter what
git push --force origin feature-A

# ✅ Safer — only overwrites if no one else has pushed
git push --force-with-lease origin feature-A
```

| Command | Safety | What It Does |
|---------|--------|-------------|
| `--force` | 🔴 Dangerous | Overwrites remote branch unconditionally |
| `--force-with-lease` | 🟢 Safe | Only overwrites if remote hasn't changed since your last fetch |

**Always use `--force-with-lease`** instead of `--force`. It protects you from accidentally overwriting someone else's work.

---

## 📝 Quick Summary

| Concept | Merge | Rebase |
|---------|-------|--------|
| **What it does** | Creates a merge commit tying two branches together | Replays your commits on top of another branch |
| **History** | Preserves complete, true history | Rewrites history to be linear |
| **Commit hashes** | Unchanged | Changed (new hashes) |
| **Safety** | Always safe | Dangerous on shared branches |
| **Conflicts** | Resolve once | Resolve per commit |
| **When to use** | Integrating finished work into shared branches | Updating your local feature branch |
| **Git log** | Can look messy with many merge commits | Clean, linear, easy to read |
| **Golden Rule** | N/A | NEVER rebase shared branches |
