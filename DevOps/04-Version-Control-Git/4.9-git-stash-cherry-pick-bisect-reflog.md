# 4.9 — Git Stash, Cherry-Pick, Bisect, Reflog

> **One Line Answer:** These four commands are **Git's power tools** — `stash` saves work temporarily without committing, `cherry-pick` grabs a specific commit from another branch, `bisect` finds which commit introduced a bug using binary search, and `reflog` is your safety net that tracks every move you've made — even ones `git log` doesn't show.

---

## 🤔 Why Learn These Commands?

These aren't commands you use every day, but when you need them, **nothing else will do**. They solve very specific problems that every developer faces:

| Problem | Solution |
|---------|----------|
| "I need to switch branches but my work isn't ready to commit" | **git stash** |
| "I need ONE specific commit from another branch, not the whole branch" | **git cherry-pick** |
| "Something is broken and I have no idea which of the 200 commits caused it" | **git bisect** |
| "I accidentally deleted my branch / reset my commits / broke something" | **git reflog** |

Let's master each one.

# 📦 Part 1: Git Stash — Your Temporary Shelf

---

## What is Git Stash?

Imagine you're in the middle of cooking dinner. Suddenly, someone rings the doorbell and you need to answer it. You can't just leave the stove on — so you **put your cooking aside temporarily**, handle the door, and then **come back and continue** where you left off.

`git stash` does exactly this for your code. It **temporarily saves your uncommitted changes** and gives you a clean working directory — so you can switch branches, pull updates, or handle something urgent — and then **bring your changes back** when you're ready.

---

## When Do You Need Stash?

### Scenario 1: Urgent Branch Switch

```
You're working on feature-A, halfway through some changes.
Your boss says: "Drop everything! There's a critical bug on main. Fix it NOW!"

Problem: You can't switch to main because you have uncommitted changes.
         And your work isn't ready to commit.

Solution: STASH your changes → switch to main → fix the bug → 
          switch back → UNSTASH your changes → continue working!
```

### Scenario 2: Wrong Branch

```
You started writing code... then realized you're on the WRONG branch!

Solution: STASH your changes → switch to the correct branch → 
          UNSTASH your changes → continue on the right branch!
```

### Scenario 3: Pull Before Commit

```
You want to pull the latest changes from remote, but Git won't let you
because you have local uncommitted changes that might conflict.

Solution: STASH → pull → UNSTASH
```

---

## Basic Stash Commands

### Save Your Changes to the Stash

```bash
# Stash all modified tracked files
git stash

# Output:
# Saved working directory and index state WIP on feature-A: abc1234 last commit message
```

Your working directory is now **clean** — all your changes are safely saved in the stash.

### What Exactly Gets Stashed?

```bash
# By default, git stash saves:
# ✅ Modified tracked files (files Git already knows about)
# ✅ Staged changes (files you've git add-ed)
# ❌ Untracked files (new files Git doesn't know about)
# ❌ Ignored files (.gitignore'd files)
```

### Stash with Untracked Files

```bash
# Include untracked (new) files
git stash -u
# or
git stash --include-untracked

# Include EVERYTHING (even ignored files)
git stash -a
# or
git stash --all
```

### Stash with a Message

```bash
# Default stash has a generic message — not helpful when you have multiple stashes
git stash
# → WIP on feature-A: abc1234 some commit message  (not very descriptive)

# Better: add your own message
git stash save "halfway through login form validation"
# or (newer syntax)
git stash push -m "halfway through login form validation"
```

---

## Getting Your Changes Back

### Apply the Most Recent Stash

```bash
# Apply the stash (keeps it in the stash list)
git stash apply

# Apply AND remove from stash list
git stash pop
```

**Difference between `apply` and `pop`:**

| Command | Applies Changes | Removes from Stash |
|---------|----------------|-------------------|
| `git stash apply` | ✅ Yes | ❌ No (stash stays in the list) |
| `git stash pop` | ✅ Yes | ✅ Yes (stash is removed) |

> 💡 Use `apply` when you might want to apply the same stash to multiple branches. Use `pop` when you're done with the stash.

---

## Managing Multiple Stashes

You can have **multiple stashes** stacked on top of each other, like a stack of plates.

### List All Stashes

```bash
git stash list

# Output:
# stash@{0}: On feature-A: halfway through login form validation
# stash@{1}: WIP on main: abc1234 fix header
# stash@{2}: On feature-B: working on API integration
```

The newest stash is always `stash@{0}`.

### Apply a Specific Stash

```bash
# Apply the second stash (index 1)
git stash apply stash@{1}

# Pop a specific stash
git stash pop stash@{2}
```

### See What's in a Stash

```bash
# Show the diff of the most recent stash
git stash show

# Output:
# src/login.js  | 15 +++++++++------
# src/utils.js  |  3 +++
# 2 files changed, 12 insertions(+), 6 deletions(-)

# Show the full diff (actual code changes)
git stash show -p
# or
git stash show -p stash@{1}  # for a specific stash
```

### Delete Stashes

```bash
# Drop a specific stash
git stash drop stash@{1}

# Drop the most recent stash
git stash drop

# Delete ALL stashes (careful!)
git stash clear
```

---

## Stash with Branches

Sometimes your stashed changes conflict with the current branch state. A neat trick is to create a new branch from the stash:

```bash
# Create a new branch and apply the stash to it
git stash branch new-feature-branch

# This does three things:
# 1. Creates a new branch from the commit where you stashed
# 2. Applies the stash to that branch
# 3. Drops the stash (if applied successfully)
```

This is super useful when your stash conflicts with the current branch — start fresh on a new branch!

---

## Complete Stash Workflow Example

```bash
# 1. You're working on feature-A
git checkout feature-A
# ... editing files ...
echo "new login code" >> login.js
echo "new helper function" >> utils.js

# 2. Urgent bug fix needed! But your work isn't ready to commit.
git status
# modified: login.js
# modified: utils.js

# 3. Stash your work with a descriptive message
git stash push -m "login form - validation logic in progress"

# 4. Your directory is clean now
git status
# nothing to commit, working tree clean

# 5. Switch to main and fix the bug
git checkout main
git checkout -b hotfix/critical-bug
# ... fix the bug ...
git commit -am "fix: resolve critical null pointer in payment flow"
git checkout main
git merge hotfix/critical-bug
git push origin main

# 6. Go back to your feature branch
git checkout feature-A

# 7. Get your work back
git stash pop

# 8. Continue where you left off!
git status
# modified: login.js
# modified: utils.js
# Your changes are back! 🎉
```

---

## Stash Quick Reference

| Command | What It Does |
|---------|-------------|
| `git stash` | Stash tracked modified + staged changes |
| `git stash -u` | Also stash untracked files |
| `git stash push -m "msg"` | Stash with a descriptive message |
| `git stash list` | List all stashes |
| `git stash show` | Show summary of latest stash |
| `git stash show -p` | Show full diff of latest stash |
| `git stash apply` | Apply latest stash (keep in list) |
| `git stash pop` | Apply latest stash (remove from list) |
| `git stash apply stash@{n}` | Apply a specific stash |
| `git stash drop stash@{n}` | Delete a specific stash |
| `git stash clear` | Delete ALL stashes |
| `git stash branch <name>` | Create branch from stash |

# 🍒 Part 2: Git Cherry-Pick — Grab Just What You Need

---

## What is Cherry-Pick?

Imagine a tree full of cherries. You don't want to pick the whole tree — you just want **one specific cherry**. That's cherry-pick!

`git cherry-pick` lets you take **one specific commit** from any branch and apply it to your current branch. Not the whole branch — just that one commit.

---

## When Do You Need Cherry-Pick?

### Scenario 1: A Bug Fix on the Wrong Branch

```
A developer fixed a critical bug on the "develop" branch.
But you need that fix on "main" RIGHT NOW.
You can't merge all of develop (it has unfinished features).

Solution: Cherry-pick just the bug fix commit to main.
```

### Scenario 2: Sharing a Specific Feature

```
Your teammate built a utility function on their branch.
You need that function on YOUR branch, but you don't want 
all their other changes.

Solution: Cherry-pick just the commit with the utility function.
```

### Scenario 3: Backporting

```
You fixed something in version 3.0 (main branch).
But customers on version 2.0 (release-2.0 branch) need the fix too.

Solution: Cherry-pick the fix commit from main to release-2.0.
```

---

## How to Cherry-Pick

### Basic Cherry-Pick

```bash
# Step 1: Find the commit hash you want to cherry-pick
git log --oneline feature-branch

# Output:
# f7e3a1b feat: add payment gateway
# c4d2e9a fix: resolve timeout issue in API    ← I want this one!
# a1b2c3d feat: add user dashboard
# 9x8y7z6 initial setup

# Step 2: Switch to the branch where you want the commit
git checkout main

# Step 3: Cherry-pick the commit
git cherry-pick c4d2e9a

# Output:
# [main abc4567] fix: resolve timeout issue in API
#  1 file changed, 5 insertions(+), 2 deletions(-)
```

That's it! The commit is now on `main` as a **new commit** with a **new hash** but the **same changes and message**.

```
BEFORE cherry-pick:
main:       A --- B --- C
feature:    A --- B --- D --- E --- F
                              ↑
                        (this commit: E)

AFTER cherry-pick:
main:       A --- B --- C --- E'  (cherry-picked!)
feature:    A --- B --- D --- E --- F
                              ↑
                        (original still here too)
```

### Cherry-Pick Multiple Commits

```bash
# Cherry-pick multiple specific commits
git cherry-pick c4d2e9a f7e3a1b

# Cherry-pick a range of commits (D through F, NOT including D)
git cherry-pick D..F

# Cherry-pick a range including the start commit
git cherry-pick D^..F
```

### Cherry-Pick Without Committing

Sometimes you want to apply the changes but not create a commit right away:

```bash
# Apply changes but don't commit (stages them instead)
git cherry-pick --no-commit c4d2e9a
# or
git cherry-pick -n c4d2e9a

# Now you can:
# - Modify the changes further
# - Combine with other changes
# - Commit when you're ready
git commit -m "backport: fix timeout issue from feature branch"
```

---

## Handling Cherry-Pick Conflicts

Sometimes the commit you're cherry-picking conflicts with the code on your current branch.

```bash
git cherry-pick c4d2e9a

# CONFLICT! Git stops and tells you:
# error: could not apply c4d2e9a... fix: resolve timeout issue
# hint: After resolving the conflicts, mark the corrected paths
# hint: with 'git add <paths>' and run 'git cherry-pick --continue'

# Step 1: Open the conflicting files and resolve the conflicts
# (same as merge conflicts — look for <<<<<<< markers)

# Step 2: Stage the resolved files
git add resolved-file.js

# Step 3: Continue the cherry-pick
git cherry-pick --continue

# OR abort if you changed your mind
git cherry-pick --abort
```

---

## Cherry-Pick Best Practices

| Practice | Why |
|----------|-----|
| **Use sparingly** | Cherry-picking too much creates duplicate commits across branches |
| **Prefer merge/rebase** | If you need many commits, merge or rebase is better |
| **Document it** | Add a note in the commit message: "Cherry-picked from feature-X (original: abc123)" |
| **Be aware of dependencies** | If commit E depends on commit D, cherry-picking only E might break things |
| **Test after cherry-picking** | Always run tests — the commit might not work in a different context |

# 🔍 Part 3: Git Bisect — Find the Bug with Binary Search

---

## What is Git Bisect?

Imagine your app was working fine 2 weeks ago. Today, it's broken. There have been **200 commits** in those 2 weeks. Which commit broke it?

You could check each commit one by one. That's 200 checks. Or...

`git bisect` uses **binary search** to find the broken commit in just **~8 checks** (log₂ 200 ≈ 8). It's like a detective tool that asks you "is this commit good or bad?" and narrows down the suspect with each answer.

### Binary Search Visual

```
200 commits:
[1 ---- 100 ---- 200]

Step 1: Check commit 100 → "It's good" (bug not here yet)
[101 ---- 150 ---- 200]

Step 2: Check commit 150 → "It's bad" (bug exists)
[101 ---- 125 ---- 150]

Step 3: Check commit 125 → "It's good"
[126 ---- 137 ---- 150]

Step 4: Check commit 137 → "It's bad"
[126 ---- 131 ---- 137]

...continue until you find the EXACT commit that introduced the bug!

Total checks: ~8 instead of 200! 🎯
```

---

## How to Use Git Bisect

### Step-by-Step

```bash
# Step 1: Start bisecting
git bisect start

# Step 2: Tell Git the current commit is BAD (it has the bug)
git bisect bad

# Step 3: Tell Git a known GOOD commit (where the bug didn't exist)
# This can be a commit hash, tag, branch, or relative reference
git bisect good v2.0.0
# or
git bisect good abc1234
# or
git bisect good HEAD~50
```

Now Git enters **bisect mode**. It checks out a commit in the middle and waits for your verdict.

```bash
# Git says:
# Bisecting: 100 revisions left to test after this (roughly 7 steps)
# [def5678] feat: add user notification system

# Step 4: Test the current code
# Run your tests, check the behavior, etc.
npm test
# or manually check the bug

# Step 5: Tell Git if this commit is good or bad
git bisect good   # Bug is NOT present in this commit
# or
git bisect bad    # Bug IS present in this commit

# Git checks out the next commit to test...
# Repeat steps 4-5 until Git finds the culprit!
```

### The Final Result

After enough steps, Git tells you:

```bash
# Output:
# abc4567 is the first bad commit
# commit abc4567
# Author: John Doe <john@example.com>
# Date:   Mon Dec 1 14:30:00 2025
#
#     feat: optimize database queries
#
# file1.js | 30 ++++++++++++------
# file2.js | 10 +++++
# 2 files changed, 28 insertions(+), 12 deletions(-)
```

**Found it!** Commit `abc4567` introduced the bug. Now you know exactly what change caused the problem and who made it.

### End Bisect

```bash
# When you're done, end the bisect session
git bisect reset

# This takes you back to where you were before bisecting
```

---

## Complete Bisect Example

Let's walk through a real scenario:

```bash
# Your website login is broken. It was working in version 1.5.0
# There have been 100 commits since then.

# 1. Start bisecting
git bisect start

# 2. Current HEAD is broken
git bisect bad

# 3. Version 1.5.0 was working
git bisect good v1.5.0

# Output: Bisecting: 50 revisions left to test (roughly 6 steps)
# Git checks out commit in the middle

# 4. Test the login
npm test
# Tests pass! Login works here.
git bisect good

# Output: Bisecting: 25 revisions left to test (roughly 5 steps)

# 5. Test again
npm test
# Tests fail! Login is broken here.
git bisect bad

# Output: Bisecting: 12 revisions left to test (roughly 4 steps)

# 6. Continue...
npm test
git bisect good  # or bad

# ... after ~7 steps total:

# Output:
# a1b2c3d is the first bad commit
# Author: Jane Smith
# Date: Nov 15, 2025
# Message: "refactor: change session handling logic"

# 7. Found it! End bisect.
git bisect reset

# 8. Now look at that commit to understand what went wrong
git show a1b2c3d
```

---

## Automated Bisect (The Cool Way)

If you have a test script that can determine if a commit is good or bad, you can **automate the entire process**!

```bash
# Start bisect
git bisect start
git bisect bad HEAD
git bisect good v1.5.0

# Run bisect automatically with a test script
git bisect run npm test

# Git will:
# 1. Check out a commit
# 2. Run "npm test"
# 3. If tests pass (exit code 0) → mark as "good"
# 4. If tests fail (exit code 1) → mark as "bad"
# 5. Repeat until it finds the first bad commit
# 6. Print the result

# All done automatically! 🤖
```

You can use **any script or command**:

```bash
# Using a custom script
git bisect run ./test-login.sh

# Using a specific test
git bisect run python -m pytest tests/test_login.py

# Using a make command
git bisect run make test
```

### Writing a Custom Bisect Script

```bash
#!/bin/bash
# test-login.sh — script for git bisect run

# Build the project
npm install && npm run build

# Check if login page loads
if curl -s http://localhost:3000/login | grep -q "Login Form"; then
    exit 0  # GOOD — login page works
else
    exit 1  # BAD — login page is broken
fi
```

```bash
# Make it executable and use it
chmod +x test-login.sh
git bisect run ./test-login.sh
```

---

## Bisect Tips

| Tip | Details |
|-----|---------|
| **Use tags** | Tag your releases so you can easily say `git bisect good v2.0` |
| **Skip broken commits** | If a commit doesn't compile, use `git bisect skip` |
| **Automate when possible** | `git bisect run` is faster and removes human error |
| **Reset if confused** | `git bisect reset` and start over |
| **Log your bisect** | `git bisect log` shows all your good/bad markings |
| **Replay a bisect** | `git bisect log > bisect.log` then `git bisect replay bisect.log` |

# 🕰️ Part 4: Git Reflog — Your Time Machine & Safety Net

---

## What is Reflog?

`git reflog` (reference log) is Git's **secret diary**. It records **every single thing your HEAD has pointed to** — every commit, checkout, merge, rebase, reset, amend, stash, and more.

Even when `git log` can't see a commit (because you deleted the branch, did a hard reset, etc.), **reflog still has it**.

### The Difference Between `log` and `reflog`

```bash
# git log shows: The commit history of the current branch
# Think of it as: "What's the history of this branch?"

# git reflog shows: Every place HEAD has been
# Think of it as: "What have I been doing in this repo?"
```

```
git log:
* abc1234 (HEAD → main) feat: add login
* def5678 fix: header alignment
* ghi9012 initial commit

git reflog:
abc1234 HEAD@{0}: commit: feat: add login
def5678 HEAD@{1}: checkout: moving from feature to main
xyz9999 HEAD@{2}: commit: WIP on feature     ← This commit isn't in log!
abc1234 HEAD@{3}: reset: moving to abc1234
jkl3456 HEAD@{4}: commit: something I "lost" ← This too!
```

**Reflog sees EVERYTHING.**

---

## Why Reflog is Your Safety Net

### Scenario: "I accidentally deleted my branch!"

```bash
# Oops! I deleted my branch that had important work!
git branch -D feature-important-work
# Deleted branch feature-important-work (was abc1234).

# Don't panic! Reflog has the commit hash.
git reflog
# ... find the commit from your branch ...
# abc1234 HEAD@{5}: commit: important work on feature

# Recover it!
git checkout -b feature-important-work abc1234
# Branch restored! 🎉
```

### Scenario: "I did a hard reset and lost commits!"

```bash
# I accidentally reset and lost my last 3 commits
git reset --hard HEAD~3
# My last 3 commits are GONE from git log!

# But reflog has them:
git reflog
# abc1234 HEAD@{1}: commit: my third important commit
# def5678 HEAD@{2}: commit: my second important commit
# ghi9012 HEAD@{3}: commit: my first important commit
# xyz0000 HEAD@{4}: reset: moving to HEAD~3

# Recover!
git reset --hard abc1234
# All 3 commits are back! 🎉
```

### Scenario: "A rebase went wrong!"

```bash
# I rebased and everything is messed up
git rebase main
# ... conflicts, wrong resolutions, chaos ...

# Find the state before the rebase
git reflog
# ... 
# def5678 HEAD@{7}: rebase (start): checkout main
# abc1234 HEAD@{8}: commit: my last good state   ← before rebase!

# Go back to before the rebase
git reset --hard abc1234
# Back to normal! 🎉
```

---

## How to Read Reflog

```bash
git reflog

# Output format:
# <hash> HEAD@{n}: <action>: <description>

# Example output:
abc1234 HEAD@{0}: commit: feat: add user dashboard
def5678 HEAD@{1}: checkout: moving from feature to main
ghi9012 HEAD@{2}: commit (amend): fix: update header styles
jkl3456 HEAD@{3}: commit: fix: update header styles
mno7890 HEAD@{4}: rebase (finish): returning to refs/heads/feature
pqr2345 HEAD@{5}: rebase (pick): feat: add login form
stu6789 HEAD@{6}: rebase (start): checkout main
vwx0123 HEAD@{7}: commit: feat: add login form
```

### Understanding the Parts

```
abc1234 HEAD@{0}: commit: feat: add user dashboard
  │        │         │            │
  │        │         │            └── Description of what happened
  │        │         └── What action was taken
  │        └── Position (0 = most recent)
  └── Commit hash at that point
```

### Common Actions You'll See in Reflog

| Action | What Happened |
|--------|--------------|
| `commit:` | A new commit was made |
| `commit (amend):` | A commit was amended |
| `checkout:` | Switched branches |
| `merge:` | A merge was performed |
| `rebase (start):` | A rebase started |
| `rebase (pick):` | A commit was replayed during rebase |
| `rebase (finish):` | A rebase completed |
| `reset:` | A reset was performed |
| `pull:` | A pull was performed |
| `cherry-pick:` | A cherry-pick was performed |

---

## Reflog with Time

You can reference reflog entries by **time** instead of position:

```bash
# Where was HEAD 30 minutes ago?
git show HEAD@{30.minutes.ago}

# Where was HEAD yesterday?
git show HEAD@{yesterday}

# Where was HEAD 2 weeks ago?
git show HEAD@{2.weeks.ago}

# What did my branch look like 3 days ago?
git diff main@{3.days.ago} main
```

This is incredibly useful for questions like "what was the state of the code yesterday before I broke everything?"

---

## Reflog for Specific Branches

Reflog isn't just for HEAD — each branch has its own reflog:

```bash
# Reflog for HEAD (default)
git reflog

# Reflog for a specific branch
git reflog show main
git reflog show feature-A

# Useful to see the history of a specific branch
```

---

## How Long Does Reflog Keep Data?

Reflog entries **expire** over time:

```bash
# Default expiry:
# - Reachable commits (still in some branch): 90 days
# - Unreachable commits (lost/dangling): 30 days
```

After those periods, Git's garbage collection (`git gc`) may clean them up.

```bash
# Check your reflog expiry settings
git config gc.reflogExpire          # default: 90 days
git config gc.reflogExpireUnreachable  # default: 30 days

# If you want to keep reflog entries longer:
git config gc.reflogExpire 180.days
git config gc.reflogExpireUnreachable 90.days
```

> 💡 **Important:** Don't rely on reflog for long-term backup. It's a safety net for recent mistakes, not a permanent archive. If something is important, make sure it's on a branch or a tag.

---

## Complete Recovery Example

Here's a full "disaster recovery" scenario using reflog:

```bash
# 1. You're working on a feature branch
git checkout -b feature/awesome-feature
echo "amazing code" > awesome.js
git add awesome.js
git commit -m "feat: build something awesome"

echo "more amazing code" >> awesome.js
git commit -am "feat: add more awesomeness"

echo "final touches" >> awesome.js
git commit -am "feat: final polish"

# 2. You check the log — 3 beautiful commits
git log --oneline
# ccc3333 feat: final polish
# bbb2222 feat: add more awesomeness
# aaa1111 feat: build something awesome

# 3. DISASTER: You accidentally reset hard!
git reset --hard HEAD~3

# 4. Check git log — your commits are GONE!
git log --oneline
# Only old commits here... your work is gone! 😱

# 5. DON'T PANIC! Check reflog:
git reflog
# abc0000 HEAD@{0}: reset: moving to HEAD~3
# ccc3333 HEAD@{1}: commit: feat: final polish
# bbb2222 HEAD@{2}: commit: feat: add more awesomeness
# aaa1111 HEAD@{3}: commit: feat: build something awesome

# 6. RECOVER! Reset to before the disaster:
git reset --hard ccc3333

# 7. Check log again:
git log --oneline
# ccc3333 feat: final polish
# bbb2222 feat: add more awesomeness
# aaa1111 feat: build something awesome

# ALL YOUR WORK IS BACK! 🎉🎉🎉
```

## Reflog Quick Reference

| Command | What It Does |
|---------|-------------|
| `git reflog` | Show reflog for HEAD |
| `git reflog show <branch>` | Show reflog for a specific branch |
| `git show HEAD@{n}` | Show the commit at reflog position n |
| `git show HEAD@{time}` | Show commit at a specific time |
| `git diff HEAD@{0} HEAD@{5}` | Diff between two reflog positions |
| `git reset --hard HEAD@{n}` | Reset to a reflog position |
| `git checkout -b <name> HEAD@{n}` | Create branch from reflog position |

| **git stash** | Temporarily save uncommitted changes | Use `-u` for untracked files |
| **git stash pop** | Restore and remove the latest stash | Use `apply` to keep it in the list |
| **git stash list** | Show all saved stashes | Numbered from 0 (newest) |
| **git cherry-pick \<hash\>** | Apply a specific commit to current branch | Creates a new commit with new hash |
| **git cherry-pick -n \<hash\>** | Apply without committing | Good for combining changes |
| **git bisect start** | Begin binary search for a bug | Follow with `bad` and `good` |
| **git bisect run \<script\>** | Automate bisect with a test | Script returns 0 (good) or 1 (bad) |
| **git bisect reset** | End bisect session | Returns to your original branch |
| **git reflog** | Show everywhere HEAD has been | Your safety net — sees "lost" commits |
| **git reset --hard HEAD@{n}** | Go back to reflog position | Nuclear option — use carefully |
