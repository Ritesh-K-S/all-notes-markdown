```

Docker Compose                 → YAML
GitHub Actions / GitLab CI     → YAML
```

## 2. JSON — JavaScript Object Notation

JSON (pronounced "jay-son") was created by Douglas Crockford in the early 2000s. Despite having "JavaScript" in the name, it's used by EVERY programming language.

### Why JSON Exists

Before JSON, people used XML for data exchange:

```xml
<!-- XML — Verbose and painful -->
<person>
    <name>Ritesh</name>
    <age>25</age>
    <skills>
        <skill>Docker</skill>
        <skill>Kubernetes</skill>
    </skills>
</person>
```

```json
// JSON — Clean and simple
{
    "name": "Ritesh",
    "age": 25,
    "skills": ["Docker", "Kubernetes"]
}
```

JSON is **lighter**, **easier to read**, and **faster to parse** than XML.

### The Core Idea

```
JSON is just a way to represent data as text.
It uses two structures:

1. Objects  → { "key": "value" }     (like a dictionary)
2. Arrays   → [ "item1", "item2" ]   (like a list)

That's it. Everything in JSON is built from these two things.
```

## 3. JSON Syntax — The Complete Rules

JSON has very strict syntax. One tiny mistake and it breaks. Here are ALL the rules:

### Rule 1: Objects Use Curly Braces `{}`

```json
{
    "name": "nginx",
    "version": "1.25"
}
```

- An object is a collection of **key-value pairs**.
- Keys MUST be **strings** (in double quotes).
- Key and value are separated by a **colon** `:`.
- Pairs are separated by **commas** `,`.

### Rule 2: Arrays Use Square Brackets `[]`

```json
["Docker", "Kubernetes", "Terraform"]
```

- An array is an **ordered list** of values.
- Values are separated by commas.
- Values can be of any type (strings, numbers, objects, arrays, etc.).

### Rule 3: Strings Must Use Double Quotes

```json
{
    "correct": "Hello World",
    "also_correct": "He said \"hello\"",
    "WRONG": 'single quotes are NOT allowed'
}
```

**Only double quotes `"` are valid in JSON.** No single quotes. Ever.

### Rule 4: No Trailing Commas

```json
{
    "name": "nginx",
    "version": "1.25",     ← This trailing comma is INVALID
}
```

```json
{
    "name": "nginx",
    "version": "1.25"      ← Correct! No comma after the last item
}
```

### Rule 5: No Comments

```json
{
    // This comment is INVALID in JSON
    "name": "nginx",
    /* This is also INVALID */
    "version": "1.25"
}
```

**JSON does NOT support comments.** This is one of its biggest limitations for configuration files (and one reason YAML is more popular for configs).

> **Note:** Some tools like VS Code support "JSONC" (JSON with Comments) for their own config files like `settings.json`. But standard JSON? No comments.

## 4. JSON Data Types

JSON supports exactly **6 data types**:

```json
{
    "string":  "Hello World",              // Text in double quotes
    "number":  42,                         // Integer or decimal (no quotes!)
    "decimal": 3.14,                       // Floating point number
    "boolean": true,                       // true or false (no quotes, lowercase!)
    "null":    null,                        // Represents "nothing" or "empty"
    "object":  { "key": "value" },         // Nested object
    "array":   [1, 2, 3]                   // List of values
}
```

### Important Details

```
Data Type     Valid Examples                   Invalid Examples
──────────    ──────────────                   ────────────────
String        "hello", "123", ""              'hello' (no single quotes)
Number        42, 3.14, -10, 1.5e10          042 (no leading zeros)
Boolean       true, false                     True, FALSE, "true" (wrong!)
Null          null                            None, NULL, nil (wrong!)
Object        {"a": 1}                        {a: 1} (keys need quotes)
Array         [1, "two", true]                (1, 2, 3) (wrong brackets)
```

### Nesting — Building Complex Structures

```json
{
    "server": {
        "host": "10.0.1.5",
        "port": 8080,
        "ssl": true,
        "allowed_ips": ["10.0.0.0/8", "192.168.1.0/24"],
        "features": {
            "caching": true,
            "compression": false,
            "rate_limiting": {
                "enabled": true,
                "requests_per_minute": 100
            }
        }
    }
}
```

You can nest objects and arrays as deep as you want. But try to keep it readable!

## 7. Working with JSON on the Command Line

### Pretty Printing JSON

```bash
# Raw JSON from an API (hard to read)
curl -s https://api.github.com/users/torvalds
# Output: {"login":"torvalds","id":1024025,"node_id":"MDQ6VXNl..."}

# Pretty print with python
curl -s https://api.github.com/users/torvalds | python3 -m json.tool

# Pretty print with jq (much better — install first!)
curl -s https://api.github.com/users/torvalds | jq .
```

### JSON from kubectl

```bash
# Get a pod's full details as JSON
kubectl get pod nginx -o json

# Get just specific fields
kubectl get pod nginx -o jsonpath='{.metadata.name}'
kubectl get pod nginx -o jsonpath='{.status.phase}'
kubectl get pod nginx -o jsonpath='{.spec.containers[0].image}'

# Get all pod names in a namespace
kubectl get pods -o jsonpath='{.items[*].metadata.name}'
```

### JSON from Docker

```bash
# Get a container's IP address
docker inspect my-container | jq '.[0].NetworkSettings.IPAddress'

# Get the container's state
docker inspect my-container | jq '.[0].State'

# Get the image name
docker inspect my-container --format '{{.Config.Image}}'
```

## 8. jq — The JSON Swiss Army Knife

**jq** is the most important JSON tool you'll ever use. It lets you slice, filter, map, and transform JSON data from the command line.

### Installing jq

```bash
# Ubuntu/Debian
sudo apt install jq

# CentOS/RHEL
sudo yum install jq

# Mac
brew install jq

# Windows (via Chocolatey)
choco install jq
```

### Basic jq Usage

```bash
# Sample JSON (saved as data.json)
echo '{
    "name": "web-app",
    "version": "2.1.0",
    "replicas": 3,
    "containers": [
        {"name": "nginx", "port": 80},
        {"name": "redis", "port": 6379},
        {"name": "app", "port": 8080}
    ]
}' > data.json

# Get a single field
jq '.name' data.json
# Output: "web-app"

# Get a nested field
jq '.containers[0].name' data.json
# Output: "nginx"

# Get all container names
jq '.containers[].name' data.json
# Output:
# "nginx"
# "redis"
# "app"

# Get all ports as an array
jq '[.containers[].port]' data.json
# Output: [80, 6379, 8080]

# Filter containers where port > 1000
jq '.containers[] | select(.port > 1000)' data.json
# Output: {"name": "redis", "port": 6379}
#         {"name": "app", "port": 8080}

# Transform data — create new structure
jq '{app_name: .name, container_count: (.containers | length)}' data.json
# Output: {"app_name": "web-app", "container_count": 3}

# Get raw values (without quotes)
jq -r '.name' data.json
# Output: web-app  (no quotes!)
```

### jq with Kubernetes

```bash
# List all images used across all pods
kubectl get pods -A -o json | jq -r '.items[].spec.containers[].image' | sort -u

# Find pods in CrashLoopBackOff
kubectl get pods -A -o json | jq -r '
    .items[] |
    select(.status.containerStatuses[]?.state.waiting?.reason == "CrashLoopBackOff") |
    "\(.metadata.namespace)/\(.metadata.name)"
'

# Get resource requests for all containers
kubectl get pods -o json | jq '.items[].spec.containers[] | {
    name: .name,
    cpu_request: .resources.requests.cpu,
    mem_request: .resources.requests.memory
}'
```

### jq Cheat Sheet

```
Expression          What It Does                    Example
──────────          ─────────────                   ───────
.field              Access a field                  .name → "nginx"
.field.nested       Access nested field             .metadata.name
.[0]                Access array element            .items[0]
.[]                 Iterate over array              .items[]
| select(cond)      Filter                          select(.port > 80)
| length            Count items                     .items | length
| keys              Get object keys                 . | keys
| map(expr)         Transform each element          map(.name)
| sort_by(.field)   Sort by field                   sort_by(.age)
| unique            Remove duplicates               unique
| group_by(.field)  Group by field                  group_by(.status)
-r                  Raw output (no quotes)          jq -r '.name'
-c                  Compact output (one line)       jq -c '.'
```

## 9. YAML — YAML Ain't Markup Language

YAML (originally "Yet Another Markup Language," later changed to "YAML Ain't Markup Language") is the **king of DevOps configuration**.

### Why YAML is Everywhere in DevOps

```
YAML was designed to be:
 ✅ Human-readable (much easier to read than JSON)
 ✅ Human-writable (no closing braces to match)
 ✅ Supports comments (unlike JSON!)
 ✅ Cleaner for large configurations
 ✅ A superset of JSON (valid JSON is valid YAML!)
```

### YAML vs JSON — First Look

```json
// JSON
{
    "apiVersion": "v1",
    "kind": "Pod",
    "metadata": {
        "name": "nginx",
        "labels": {
            "app": "nginx"
        }
    },
    "spec": {
        "containers": [
            {
                "name": "nginx",
                "image": "nginx:1.25",
                "ports": [
                    {
                        "containerPort": 80
                    }
                ]
            }
        ]
    }
}
```

```yaml
# YAML — Same thing, but MUCH cleaner
apiVersion: v1
kind: Pod
metadata:
  name: nginx
  labels:
    app: nginx
spec:
  containers:
    - name: nginx
      image: nginx:1.25
      ports:
        - containerPort: 80
```

See how much cleaner the YAML version is? No curly braces, no square brackets, no quotes everywhere. Just clean, indented text.

## 10. YAML Syntax — The Complete Rules

### Rule 1: Indentation with Spaces (NEVER Tabs!)

```yaml
# CORRECT — using spaces (2 spaces per level is standard)
parent:
  child: value
  another_child:
    grandchild: value

# WRONG — using tabs (YAML will throw an error!)
parent:
	child: value    # ← This is a TAB character. YAML will REJECT this.
```

**This is the #1 source of YAML errors.** Configure your editor to use spaces, not tabs.

> **Pro tip:** In VS Code, set `"editor.insertSpaces": true` and `"editor.tabSize": 2` for YAML files.

### Rule 2: Key-Value Pairs

```yaml
# Simple key-value pairs
name: nginx
version: "1.25"
port: 80
debug: true

# The space after the colon is REQUIRED
name:nginx    # WRONG — no space after colon
name: nginx   # CORRECT
```

### Rule 3: Comments Use `#`

```yaml
# This is a comment
name: nginx  # This is also a comment (inline)

# Comments can be anywhere!
# They're ignored by the YAML parser
```

Unlike JSON, YAML supports comments! This is huge for documenting your configs.

### Rule 4: Documents Start with `---` and End with `...`

```yaml
---
# First document
name: app1
version: "1.0"
---
# Second document
name: app2
version: "2.0"
...
```

- `---` marks the start of a new YAML document.
- `...` marks the end (optional, rarely used).
- This lets you put **multiple documents** in one file (Kubernetes uses this heavily).

```yaml
# Kubernetes: multiple resources in one file
---
apiVersion: v1
kind: Service
metadata:
  name: my-service
spec:
  ports:
    - port: 80
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-deployment
spec:
  replicas: 3
```

## 11. YAML Data Types

YAML automatically detects data types (unlike JSON where you have to be explicit).

### Strings

```yaml
# All of these are strings
name: nginx                    # Unquoted string
name: "nginx"                  # Double-quoted string
name: 'nginx'                  # Single-quoted string

# When to use quotes:
special: "true"                # Without quotes, YAML thinks this is a boolean!
version: "1.25"                # Without quotes, YAML thinks this is a number!
colon_value: "key: value"      # Strings containing colons need quotes
hash_value: "# not a comment"  # Strings starting with # need quotes

# Escape characters (only work in double quotes)
newline: "line1\nline2"        # \n becomes an actual newline
tab: "col1\tcol2"              # \t becomes a tab
```

### Numbers

```yaml
# Integers
count: 42
negative: -10
hex: 0xFF          # Hexadecimal → 255
octal: 0o77        # Octal → 63

# Floating point
pi: 3.14159
scientific: 1.5e+10
infinity: .inf
not_a_number: .nan
```

### Booleans

```yaml
# All of these are TRUE
enabled: true
enabled: True
enabled: TRUE
enabled: yes        # YAML 1.1 (careful — removed in YAML 1.2!)
enabled: on         # YAML 1.1 (careful — removed in YAML 1.2!)

# All of these are FALSE
enabled: false
enabled: False
enabled: FALSE
enabled: no         # YAML 1.1
enabled: off        # YAML 1.1
```

**⚠️ Warning:** The `yes/no/on/off` boolean values catch many people off guard:

```yaml
# This is a COMMON bug!
country: no        # You think it's the string "no" but YAML reads it as boolean false!
country: "no"      # This is what you actually want — the string "no"

# Norway problem — famously causes issues:
countries:
  - us
  - uk
  - no             # This is false, not "Norway"!
  - "no"           # This is the string "no" (for Norway)
```

### Null

```yaml
# All of these represent null/empty
value: null
value: ~
value:              # Empty value = null
```

## 12. YAML Collections — Lists and Dictionaries

### Lists (Arrays)

```yaml
# Block style (most common in DevOps)
fruits:
  - apple
  - banana
  - cherry

# Flow style (compact — like JSON)
fruits: [apple, banana, cherry]

# List of objects
containers:
  - name: nginx
    image: nginx:1.25
    port: 80
  - name: redis
    image: redis:7
    port: 6379
```

### Dictionaries (Maps)

```yaml
# Block style
metadata:
  name: my-app
  namespace: production
  labels:
    app: my-app
    tier: frontend

# Flow style (compact)
metadata: {name: my-app, namespace: production}

# Nested dictionaries
server:
  database:
    host: db.example.com
    port: 5432
    credentials:
      username: admin
      password: secret123
```

### Mixed — Lists of Dictionaries (This is What Kubernetes Uses!)

```yaml
spec:
  containers:              # List of container objects
    - name: web            # First container
      image: nginx:1.25
      ports:               # List of ports for this container
        - containerPort: 80
          protocol: TCP
        - containerPort: 443
          protocol: TCP
      env:                 # List of environment variables
        - name: DB_HOST
          value: "postgres.default.svc"
        - name: DB_PORT
          value: "5432"
    - name: sidecar        # Second container
      image: fluentd:latest
      volumeMounts:
        - name: logs
          mountPath: /var/log
```

## 13. YAML Multi-Line Strings

This is a powerful feature that JSON doesn't have. There are two styles:

### Literal Block Scalar `|` — Preserves Newlines

```yaml
# The | character means: "keep all newlines exactly as they are"
description: |
  This is line 1
  This is line 2
  This is line 3

# Result: "This is line 1\nThis is line 2\nThis is line 3\n"
# (each line break is preserved)
```

### Folded Block Scalar `>` — Folds Newlines into Spaces

```yaml
# The > character means: "join lines with spaces (like a paragraph)"
description: >
  This is a very long description
  that spans multiple lines in the YAML file
  but will be rendered as a single paragraph.

# Result: "This is a very long description that spans multiple lines in the YAML file but will be rendered as a single paragraph.\n"
# (newlines become spaces, except blank lines which become \n)
```

### Controlling the Trailing Newline

```yaml
# |  → Keep trailing newline (default)
text: |
  hello
# Result: "hello\n"

# |- → Strip trailing newline (chomp)
text: |-
  hello
# Result: "hello"

# |+ → Keep ALL trailing newlines
text: |+
  hello

# Result: "hello\n\n"
```

### Real-World Example — Shell Script in YAML

```yaml
# Kubernetes ConfigMap with an embedded script
apiVersion: v1
kind: ConfigMap
metadata:
  name: init-script
data:
  setup.sh: |
    #!/bin/bash
    echo "Starting initialization..."
    
    # Wait for database
    until pg_isready -h $DB_HOST -p $DB_PORT; do
      echo "Waiting for database..."
      sleep 2
    done
    
    echo "Database is ready!"
    python manage.py migrate
    python manage.py collectstatic --noinput
```

### Real-World Example — Nginx Config in YAML

```yaml
# ConfigMap with embedded Nginx configuration
apiVersion: v1
kind: ConfigMap
metadata:
  name: nginx-config
data:
  nginx.conf: |
    server {
        listen 80;
        server_name example.com;
        
        location / {
            proxy_pass http://backend:8080;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
        }
        
        location /health {
            return 200 'OK';
        }
    }
```

## 14. YAML Anchors & Aliases — DRY Configuration

YAML has a built-in way to avoid repeating yourself: **anchors** (`&`) and **aliases** (`*`).

### Basic Anchor & Alias

```yaml
# Define an anchor with & (like saving a template)
defaults: &default_settings
  timeout: 30
  retries: 3
  log_level: info

# Use the anchor with * (like pasting the template)
development:
  <<: *default_settings     # Merge all keys from default_settings
  log_level: debug          # Override just this one key

staging:
  <<: *default_settings
  timeout: 60               # Override timeout for staging

production:
  <<: *default_settings
  retries: 5                # More retries for production
  log_level: warn           # Less verbose logging
```

This is equivalent to:

```yaml
development:
  timeout: 30
  retries: 3
  log_level: debug

staging:
  timeout: 60
  retries: 3
  log_level: info

production:
  timeout: 30
  retries: 5
  log_level: warn
```

### Docker Compose Example

```yaml
# docker-compose.yml with anchors
x-common-env: &common_env
  DB_HOST: postgres
  DB_PORT: "5432"
  REDIS_HOST: redis

x-common-deploy: &common_deploy
  restart: unless-stopped
  logging:
    driver: json-file
    options:
      max-size: "10m"
      max-file: "3"

services:
  api:
    image: my-api:latest
    <<: *common_deploy
    environment:
      <<: *common_env
      SERVICE_NAME: api
      PORT: "8080"
    ports:
      - "8080:8080"

  worker:
    image: my-worker:latest
    <<: *common_deploy
    environment:
      <<: *common_env
      SERVICE_NAME: worker
      QUEUE: default
```

**`<<:`** is the merge key — it tells YAML to merge the anchor's keys into the current mapping.

## 15. YAML vs JSON — Side-by-Side Comparison

```
Feature                  YAML                          JSON
─────────────────────    ─────                         ─────
Readability              Very high                     Medium
Comments                 ✅ Yes (#)                    ❌ No
Trailing commas          N/A (no commas)               ❌ No
Multi-line strings       ✅ Yes (| and >)              ❌ No (use \n)
Anchors/Aliases          ✅ Yes (&, *)                 ❌ No
Data types               Auto-detected                 Explicit
Strictness               Lenient                       Very strict
Whitespace               Significant (indentation!)    Not significant
Quotes required          Usually no                    Always for strings
Parsing speed            Slower                        Faster
Human writing            Easier                        Harder
Machine parsing          Harder                        Easier
File extension           .yml or .yaml                 .json
```

### When to Use Which?

```
Use YAML for:                          Use JSON for:
──────────────                         ──────────────
Configuration files                    API request/response bodies
Kubernetes manifests                   Data exchange between services
CI/CD pipelines                        Lock files (package-lock.json)
Docker Compose                         Terraform state files
Ansible playbooks                      VS Code settings
Helm values files                      Machine-generated configs
Anything humans write/read often       Anything machines write/read
```

### The Golden Rule

```
YAML = Human-first (configs, pipelines, manifests)
JSON = Machine-first (APIs, data storage, state files)
```

## 16. Converting Between YAML and JSON

### Using Python (Built-in)

```bash
# YAML to JSON
python3 -c "
import yaml, json, sys
data = yaml.safe_load(open(sys.argv[1]))
print(json.dumps(data, indent=2))
" input.yaml > output.json

# JSON to YAML
python3 -c "
import yaml, json, sys
data = json.load(open(sys.argv[1]))
print(yaml.dump(data, default_flow_style=False))
" input.json > output.yaml
```

### Using yq (Like jq, but for YAML)

```bash
# Install yq
# Ubuntu/Debian
sudo wget -qO /usr/local/bin/yq https://github.com/mikefarah/yq/releases/latest/download/yq_linux_amd64
sudo chmod +x /usr/local/bin/yq

# Mac
brew install yq

# YAML to JSON
yq -o=json eval '.' input.yaml

# JSON to YAML
yq -P eval '.' input.json

# Query YAML (like jq for JSON)
yq '.metadata.name' deployment.yaml
yq '.spec.containers[0].image' pod.yaml

# Edit YAML in-place
yq -i '.spec.replicas = 5' deployment.yaml
```

### Using kubectl

```bash
# Get resource as JSON
kubectl get deployment nginx -o json

# Get resource as YAML
kubectl get deployment nginx -o yaml

# Apply JSON as YAML (kubectl accepts both!)
kubectl apply -f config.json
kubectl apply -f config.yaml
```

## 18. Common YAML Gotchas and Mistakes

These are the mistakes that catch EVERYONE — beginners and experts alike.

### Gotcha 1: Tabs vs Spaces

```yaml
# ERROR: TabError - YAML doesn't allow tabs!
server:
→   port: 8080    # ← That → is a tab character
```

**Fix:** Configure your editor to use spaces. In VS Code:
```json
{
    "[yaml]": {
        "editor.insertSpaces": true,
        "editor.tabSize": 2
    }
}
```

### Gotcha 2: The Boolean Trap

```yaml
# These are NOT strings — they're booleans!
answer: yes       # → true
answer: no        # → false
answer: on        # → true  
answer: off       # → false

# Fix: Use quotes!
answer: "yes"     # → string "yes"
country: "no"     # → string "no" (not boolean false!)
```

### Gotcha 3: The Number Trap

```yaml
# These are NOT strings — they're numbers!
version: 1.0      # → number 1.0 (not string "1.0")
zipcode: 01onal   # → error (starts like a number)
phone: 1234567890 # → number 1234567890

# Fix: Use quotes!
version: "1.0"
zipcode: "01onal"
phone: "1234567890"
```

### Gotcha 4: Colons in Strings

```yaml
# ERROR: YAML thinks this is a key-value pair
message: Error: Connection refused

# Fix: Use quotes
message: "Error: Connection refused"
```

### Gotcha 5: Special Characters at Start of String

```yaml
# These characters are special in YAML: & * ! | > ' " % @ `
name: *important     # ← YAML thinks this is an alias!
name: "*important"   # ← This is the string "*important"

command: !unsafe     # ← YAML thinks this is a tag!
command: "!unsafe"   # ← This is the string "!unsafe"
```

### Gotcha 6: Inconsistent Indentation

```yaml
# WRONG — mixed indentation levels
parent:
  child1: value
    child2: value     # ← 4 spaces! Inconsistent with child1 (2 spaces)

# CORRECT — consistent indentation
parent:
  child1: value
  child2: value       # ← Both children at 2 spaces
```

### Gotcha 7: Missing Space After Colon

```yaml
# WRONG
key:value             # ← No space after colon

# CORRECT
key: value            # ← Space after colon is required
```

### The YAML Gotcha Cheat Sheet

```
Problem                              Solution
─────────                            ────────
Tabs in indentation                  Use spaces (2 per level)
yes/no/on/off interpreted as bool    Quote them: "yes", "no"
Numbers like 1.0 or 0123            Quote them: "1.0", "0123"
Strings with colons                  Quote them: "key: value"
Strings starting with *!&%@         Quote them
Empty value treated as null          Use "" for empty string
Inconsistent indentation             Pick 2 spaces and stick with it
```

## 19. YAML Linting and Validation

Always lint your YAML before using it. One wrong space can break your entire deployment!

### yamllint — The Standard YAML Linter

```bash
# Install
pip install yamllint

# Lint a file
yamllint deployment.yaml

# Lint with strict mode
yamllint -s deployment.yaml

# Lint all YAML files in a directory
yamllint -d relaxed k8s/
```

### Custom yamllint Configuration

```yaml
# File: .yamllint.yml
---
rules:
  braces:
    min-spaces-inside: 0
    max-spaces-inside: 0
  brackets:
    min-spaces-inside: 0
    max-spaces-inside: 0
  commas:
    max-spaces-before: 0
    min-spaces-after: 1
  comments:
    min-spaces-from-content: 2
  document-start: disable          # Don't require --- at start
  empty-lines:
    max: 2
  indentation:
    spaces: 2                      # 2 spaces per indent level
    indent-sequences: true
  line-length:
    max: 120                       # Allow up to 120 characters per line
  truthy:
    allowed-values: ['true', 'false']  # Only allow true/false (not yes/no)
```

### kubectl Validation

```bash
# Dry-run validation — check if K8s would accept your YAML

# Server-side validation (more thorough)
kubectl apply -f deployment.yaml --dry-run=server

kubectl diff -f deployment.yaml
```

## 20. JSON Schema — Validating Your Data

JSON Schema is a way to define the **structure** your JSON/YAML must follow — like a blueprint.

### Simple Example

```json
{
    "$schema": "http://json-schema.org/draft-07/schema#",
    "title": "Server Configuration",
    "type": "object",
    "required": ["host", "port"],
    "properties": {
        "host": {
            "type": "string",
            "description": "Server hostname or IP"
        },
        "port": {
            "type": "integer",
            "minimum": 1,
            "maximum": 65535,
            "description": "Port number"
        },
        "ssl": {
            "type": "boolean",
            "default": false
        },
        "environment": {
            "type": "string",
            "enum": ["development", "staging", "production"]
        }
    }
}
```

### Valid and Invalid Data

```json
// ✅ VALID
{
    "host": "10.0.1.5",
    "port": 8080,
    "ssl": true,
    "environment": "production"
}

// ❌ INVALID — port is a string
{
    "host": "10.0.1.5",
    "port": "8080"
}

// ❌ INVALID — missing required field "host"
{
    "port": 8080
}

// ❌ INVALID — environment not in allowed values
{
    "host": "10.0.1.5",
    "port": 8080,
    "environment": "testing"
}
```

### VS Code Integration

```json
// .vscode/settings.json — Enable schema validation for your project
{
    "yaml.schemas": {
        "https://raw.githubusercontent.com/compose-spec/compose-spec/master/schema/compose-spec.json": "docker-compose*.yml",
        "https://json.schemastore.org/github-workflow.json": ".github/workflows/*.yml"
    }
}
```

This gives you **autocomplete** and **error highlighting** for your YAML files right in VS Code!

## 21. TOML and HCL — Other Formats You'll Encounter

While YAML and JSON dominate, you'll see two other formats in DevOps:

### TOML — Tom's Obvious, Minimal Language

```toml
# Used by: Rust (Cargo.toml), Python (pyproject.toml), Hugo, Prometheus Alertmanager

[server]
host = "0.0.0.0"
port = 8080
debug = false

[database]
host = "localhost"
port = 5432
name = "myapp"

[database.pool]
min_connections = 5
max_connections = 20

[[servers]]         # Array of tables
name = "web-1"
ip = "10.0.1.1"

[[servers]]
name = "web-2"
ip = "10.0.1.2"
```

```
TOML is like an INI file on steroids.
Good for: Simple config files with clear sections.
Not great for: Deeply nested structures (gets ugly).
```

### HCL — HashiCorp Configuration Language

```hcl
# Used by: Terraform, Vault, Consul, Nomad, Packer

resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.micro"
  
  tags = {
    Name        = "web-server"
    Environment = "production"
  }
  
  root_block_device {
    volume_size = 20
    volume_type = "gp3"
  }
}

variable "region" {
  type        = string
  default     = "us-east-1"
  description = "AWS region"
}

output "public_ip" {
  value = aws_instance.web.public_ip
}
```

```
HCL is designed specifically for infrastructure configuration.
Good for: Terraform and other HashiCorp tools.
Not used for: General config files (it's HashiCorp-specific).
```

## 22. Best Practices

### YAML Best Practices

```yaml
# 1. Always use 2-space indentation (industry standard)

# 2. Always quote strings that could be misinterpreted
version: "1.0"          # Not 1.0 (number)
enabled: "yes"          # Not yes (boolean)
country: "no"           # Not no (boolean)

# 3. Use comments to explain WHY, not WHAT
replicas: 5  # Scale to 5 for Black Friday traffic (revert after)

# 4. Use --- to separate multiple documents
---
apiVersion: v1
kind: Service
---
apiVersion: apps/v1
kind: Deployment

# 5. Keep nesting shallow (max 4-5 levels)

# 6. Use consistent key naming (kebab-case or snake_case, pick one)
app-name: my-app        # kebab-case (Kubernetes convention)
app_name: my-app        # snake_case (Python/Ansible convention)

# 7. Always lint before committing
# yamllint .
# kubeconform k8s/
```

### JSON Best Practices

```json
{
    "practices": [
        "1. Use 2 or 4 space indentation (be consistent)",
        "2. Use meaningful key names (camelCase for JS/TS, snake_case for Python)",
        "3. Validate against schemas when possible",
        "4. Use jq for command-line processing",
        "5. Never manually edit large JSON files — use tools",
        "6. Keep JSON files in version control",
        "7. Use JSONC or comments-before-file for documentation"
    ]
}
```
