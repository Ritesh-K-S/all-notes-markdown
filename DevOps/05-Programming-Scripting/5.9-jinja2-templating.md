## 1. What is Templating? (And Why Should You Care?)

Imagine you need to create Nginx configuration files for 50 servers. Each server has a different hostname, IP address, and port. Would you:

**Option A:** Write 50 separate config files by hand (copy-paste, change values, pray you don't miss anything) ❌

**Option B:** Write ONE template and let a tool fill in the values automatically ✅

That's what templating is. Let's see it in action:

```
THE TEMPLATE (write once):
─────────────────────────
server {
    listen {{ port }};
    server_name {{ hostname }};
    
    location / {
        proxy_pass http://{{ backend_ip }}:{{ backend_port }};
    }
}

THE DATA (changes per server):
──────────────────────────────
Server 1: { port: 80, hostname: "web1.example.com", backend_ip: "10.0.1.1", backend_port: 8080 }
Server 2: { port: 80, hostname: "web2.example.com", backend_ip: "10.0.1.2", backend_port: 8080 }
Server 3: { port: 443, hostname: "api.example.com", backend_ip: "10.0.2.1", backend_port: 3000 }

THE OUTPUT (generated automatically):
─────────────────────────────────────
Server 1:                          Server 2:                          Server 3:
server {                           server {                           server {
    listen 80;                         listen 80;                         listen 443;
    server_name web1.example.com;      server_name web2.example.com;      server_name api.example.com;
    location / {                       location / {                       location / {
        proxy_pass http://10.0.1.1:8080;   proxy_pass http://10.0.1.2:8080;   proxy_pass http://10.0.2.1:3000;
    }                                  }                                  }
}                                  }                                  }
```

**One template + data = unlimited config files.** No copy-paste errors. No missed values. Pure automation.

## 2. What is Jinja2?

**Jinja2** is a templating engine for Python. It takes a template (text with placeholders) and data (variables), and produces the final output.

```
Jinja2 = Template + Data → Output

It's named after the Japanese word for "temple" (神社).
The "2" is because it's version 2 of the original Jinja library.
```

### Why Jinja2 Specifically?

There are many templating engines:

```
Templating Engine    Language      Used By
─────────────────    ────────      ────────
Jinja2               Python        Ansible, Flask, Salt, Airflow
Go Templates         Go            Helm, Hugo, Docker
ERB                  Ruby          Chef, Puppet
Mustache             Language-agnostic   Various
Handlebars           JavaScript    Various
Mako                 Python        SQLAlchemy, Pylons
```

Jinja2 is the most important one for DevOps because **Ansible uses it** — and Ansible is one of the most popular configuration management tools.

## 5. Jinja2 Basics — The Three Building Blocks

Everything in Jinja2 uses one of three delimiters:

```
Delimiter          Purpose                  Example
──────────         ───────                  ───────
{{ ... }}           Expressions (output)     {{ username }}
{% ... %}           Statements (logic)       {% if enabled %}
{# ... #}           Comments                 {# This is a comment #}
```

### Quick Example

```jinja2
{# This is a Jinja2 template for an Nginx virtual host #}
{# Variables: server_name, port, upstream_servers #}

server {
    listen {{ port }};
    server_name {{ server_name }};

    {% if ssl_enabled %}
    ssl_certificate /etc/ssl/{{ server_name }}.crt;
    ssl_certificate_key /etc/ssl/{{ server_name }}.key;
    {% endif %}

    location / {
        {% for server in upstream_servers %}
        proxy_pass http://{{ server }};
        {% endfor %}
    }
}
```

Let's break each building block down in detail.

## 6. Variables — Inserting Dynamic Values

Variables are the most basic feature. Use `{{ }}` to output a variable's value.

### Simple Variables

```jinja2
Hello, {{ name }}!
Your server is {{ server_ip }} on port {{ server_port }}.
```

With data `{ name: "Ritesh", server_ip: "10.0.1.5", server_port: 8080 }`, this produces:

```
Hello, Ritesh!
Your server is 10.0.1.5 on port 8080.
```

### Accessing Nested Data

```jinja2
{# Data: { server: { name: "web-1", network: { ip: "10.0.1.5", subnet: "10.0.1.0/24" } } } #}

Server: {{ server.name }}
IP: {{ server.network.ip }}
Subnet: {{ server.network.subnet }}

{# You can also use bracket notation (useful when keys have special characters) #}
Server: {{ server['name'] }}
IP: {{ server['network']['ip'] }}
```

### Accessing List Items

```jinja2
{# Data: { ports: [80, 443, 8080] } #}

First port: {{ ports[0] }}     {# → 80 #}
Last port: {{ ports[-1] }}     {# → 8080 #}
All ports: {{ ports }}         {# → [80, 443, 8080] #}
```

### Default Values

```jinja2
{# If the variable doesn't exist, provide a default #}
Port: {{ port | default(8080) }}
Log level: {{ log_level | default("info") }}
Environment: {{ env | default("development") }}

{# default() also works when the value exists but is empty or false #}
{# Use default(value, true) to replace empty/false values too #}
Enabled: {{ feature_flag | default(true, true) }}
```

### Undefined Variables

```jinja2
{# If a variable is undefined and you don't provide a default,
   Jinja2 will throw an UndefinedError (in strict mode) 
   or render nothing (in lenient mode) #}

{# In Ansible, undefined variables cause the play to fail.
   Always use default() if a variable might not exist. #}
```

## 7. Filters — Transforming Data

Filters modify variables. They're applied using the pipe `|` character (same idea as Unix pipes!).

```
{{ variable | filter_name }}
{{ variable | filter1 | filter2 | filter3 }}    ← You can chain filters!
```

### Common String Filters

```jinja2
{{ "hello world" | upper }}          → HELLO WORLD
{{ "HELLO WORLD" | lower }}          → hello world
{{ "hello world" | capitalize }}     → Hello world
{{ "hello world" | title }}          → Hello World
{{ "  hello  " | trim }}             → hello
{{ "hello" | replace("hello", "hi") }} → hi
{{ "hello world" | truncate(5) }}    → he...
{{ "hello" | length }}               → 5
{{ "hello" | reverse }}              → olleh
{{ "hello" | center(20) }}           →        hello        
```

### Number Filters

```jinja2
{{ 3.14159 | round }}                → 3.0
{{ 3.14159 | round(2) }}             → 3.14
{{ 42 | abs }}                       → 42
{{ -42 | abs }}                      → 42
{{ 1000000 | filesizeformat }}       → 976.6 KiB (Ansible)
```

### List Filters

```jinja2
{# Data: { servers: ["web-1", "web-2", "db-1", "web-3"] } #}

{{ servers | length }}                        → 4
{{ servers | first }}                         → web-1
{{ servers | last }}                          → web-3
{{ servers | sort }}                          → ['db-1', 'web-1', 'web-2', 'web-3']
{{ servers | reverse | list }}                → ['web-3', 'db-1', 'web-2', 'web-1']
{{ servers | join(", ") }}                    → web-1, web-2, db-1, web-3
{{ servers | unique | list }}                 → (removes duplicates)
{{ [1, 2, 3, 4, 5] | sum }}                  → 15
{{ [1, 2, 3, 4, 5] | min }}                  → 1
{{ [1, 2, 3, 4, 5] | max }}                  → 5
{{ servers | select("match", "web-*") | list }} → ['web-1', 'web-2', 'web-3']
{{ servers | reject("match", "web-*") | list }} → ['db-1']
```

### Dictionary Filters

```jinja2
{# Data: { config: { host: "10.0.1.5", port: 8080, ssl: true } } #}

{{ config | dictsort }}              → Sorted key-value pairs
{{ config.keys() | list }}           → ['host', 'port', 'ssl']
{{ config.values() | list }}         → ['10.0.1.5', 8080, true]
{{ config.items() | list }}          → [('host', '10.0.1.5'), ('port', 8080), ('ssl', true)]
```

## 9. Conditionals — If/Else Logic

Use `{% if %}`, `{% elif %}`, and `{% else %}` for conditional rendering.

### Basic If/Else

```jinja2
{% if environment == "production" %}
server {
    listen 443 ssl;
    ssl_certificate /etc/ssl/prod.crt;
}
{% elif environment == "staging" %}
server {
    listen 443 ssl;
    ssl_certificate /etc/ssl/staging.crt;
}
{% else %}
server {
    listen 80;
    # No SSL in development
}
{% endif %}
```

### Comparison Operators

```jinja2
{# Equality #}
{% if port == 80 %}...{% endif %}
{% if name != "admin" %}...{% endif %}

{# Comparison #}
{% if replicas > 1 %}...{% endif %}
{% if memory < 512 %}...{% endif %}
{% if cpu >= 2 %}...{% endif %}

{# Boolean checks #}
{% if ssl_enabled %}...{% endif %}               {# True if ssl_enabled is truthy #}
{% if not debug_mode %}...{% endif %}            {# True if debug_mode is falsy #}

{# Membership #}
{% if "admin" in roles %}...{% endif %}          {# Check if value is in a list #}
{% if server not in blacklist %}...{% endif %}

{# Defined check #}
{% if custom_port is defined %}
    listen {{ custom_port }};
{% else %}
    listen 80;
{% endif %}

{# Undefined check #}
{% if backup_path is undefined %}
    # No backup path configured!
{% endif %}

{# Combined conditions #}
{% if environment == "production" and replicas >= 3 %}
    # Production with enough replicas
{% endif %}

{% if port == 80 or port == 443 %}
    # Standard web port
{% endif %}
```

### Testing Values

```jinja2
{# Jinja2 has built-in "tests" using the "is" keyword #}

{% if port is defined %}...{% endif %}
{% if name is undefined %}...{% endif %}
{% if value is none %}...{% endif %}
{% if count is even %}...{% endif %}
{% if count is odd %}...{% endif %}
{% if name is string %}...{% endif %}
{% if port is number %}...{% endif %}
{% if items is iterable %}...{% endif %}
{% if mapping is mapping %}...{% endif %}   {# Is it a dictionary? #}

{# Ansible adds more tests #}
{% if path is file %}...{% endif %}
{% if path is directory %}...{% endif %}
{% if result is succeeded %}...{% endif %}
{% if result is failed %}...{% endif %}
{% if hostname is match("web-\d+") %}...{% endif %}   {# Regex match #}
{% if hostname is search("web") %}...{% endif %}      {# Regex search #}
```

### Inline If (Ternary)

```jinja2
{# Short one-line conditional #}
{{ "production" if environment == "prod" else "development" }}

{# Used in configs #}
log_level: {{ "warn" if environment == "production" else "debug" }}
replicas: {{ 5 if high_availability else 1 }}
protocol: {{ "https" if ssl_enabled else "http" }}
```

## 10. Loops — Repeating Blocks

Use `{% for %}` to iterate over lists and dictionaries.

### Looping Over a List

```jinja2
{# Data: { servers: ["10.0.1.1", "10.0.1.2", "10.0.1.3"] } #}

upstream backend {
    {% for server in servers %}
    server {{ server }}:8080;
    {% endfor %}
}
```

**Output:**

```
upstream backend {
    server 10.0.1.1:8080;
    server 10.0.1.2:8080;
    server 10.0.1.3:8080;
}
```

### Looping Over a List of Dictionaries

```jinja2
{# Data: 
   containers:
     - name: nginx
       image: nginx:1.25
       port: 80
     - name: redis
       image: redis:7
       port: 6379
#}

{% for container in containers %}
Container: {{ container.name }}
  Image: {{ container.image }}
  Port: {{ container.port }}

{% endfor %}
```

### Looping Over a Dictionary

```jinja2
{# Data: { env_vars: { DB_HOST: "localhost", DB_PORT: "5432", DEBUG: "true" } } #}

{% for key, value in env_vars.items() %}
export {{ key }}="{{ value }}"
{% endfor %}
```

**Output:**

```
export DB_HOST="localhost"
export DB_PORT="5432"
export DEBUG="true"
```

### Loop with Conditions

```jinja2
{# Only include production servers #}
{% for server in servers if server.environment == "production" %}
server {{ server.ip }}:{{ server.port }};
{% endfor %}

{# Empty loop handling #}
{% for user in admins %}
{{ user.name }}
{% else %}
No admin users found!
{% endfor %}
```

### Nested Loops

```jinja2
{# Data:
   datacenters:
     - name: us-east-1
       servers: ["10.0.1.1", "10.0.1.2"]
     - name: eu-west-1
       servers: ["10.1.1.1", "10.1.1.2"]
#}

{% for dc in datacenters %}
# Datacenter: {{ dc.name }}
{% for server in dc.servers %}
  server {{ server }}:8080;
{% endfor %}

{% endfor %}
```

## 11. Loop Variables — Special Loop Helpers

Inside a `{% for %}` loop, Jinja2 gives you special variables:

```
Variable            Description                    Example Value
────────            ───────────                    ─────────────
loop.index          Current iteration (1-based)    1, 2, 3, ...
loop.index0         Current iteration (0-based)    0, 1, 2, ...
loop.revindex       Reverse index (1-based)        3, 2, 1
loop.revindex0      Reverse index (0-based)        2, 1, 0
loop.first          True if first iteration        true/false
loop.last           True if last iteration         true/false
loop.length         Total number of items          3
loop.cycle(a,b)     Cycle through values           a, b, a, b, ...
loop.previtem       Previous item                  (the previous value)
loop.nextitem       Next item                      (the next value)
```

### Practical Examples

```jinja2
{# Add commas between items, but NOT after the last one #}
upstream backend {
    {% for server in servers %}
    server {{ server }}{% if not loop.last %},{% endif %}

    {% endfor %}
}

{# Number each item #}
{% for user in users %}
{{ loop.index }}. {{ user.name }} ({{ user.role }})
{% endfor %}

{# Alternating row colors (for HTML) #}
{% for item in items %}
<tr class="{{ loop.cycle('odd', 'even') }}">
    <td>{{ item }}</td>
</tr>
{% endfor %}

{# Show separator between items #}
{% for server in servers %}
{{ server.name }}: {{ server.ip }}
{% if not loop.last %}
---
{% endif %}
{% endfor %}
```

## 12. Whitespace Control

Jinja2 templates often produce extra blank lines and spaces. You can control this.

### The Problem

```jinja2
{% for server in servers %}
server {{ server }};
{% endfor %}
```

Output (notice the blank lines):

```

server 10.0.1.1;

server 10.0.1.2;

server 10.0.1.3;

```

### The Solution — Trim Marks

Add a `-` to the inside of the delimiter to strip whitespace:

```jinja2
{# Strip whitespace BEFORE the tag #}
{%- for server in servers %}

{# Strip whitespace AFTER the tag #}
{% for server in servers -%}

{# Strip whitespace on BOTH sides #}
{%- for server in servers -%}
```

### Fixed Example

```jinja2
{%- for server in servers %}
server {{ server }};
{%- endfor %}
```

Output (clean!):

```
server 10.0.1.1;
server 10.0.1.2;
server 10.0.1.3;
```

### Common Whitespace Pattern

```jinja2
{# Generate a comma-separated list without trailing comma #}
servers: [
  {%- for s in servers %}
  "{{ s }}"{% if not loop.last %},{% endif %}
  {%- endfor %}
]
```

## 13. Template Inheritance — Reusable Base Templates

Template inheritance lets you create a "base" template and extend it. This is huge for avoiding repetition.

### Base Template

```jinja2
{# File: base_vhost.conf.j2 #}

server {
    listen {{ port | default(80) }};
    server_name {{ server_name }};

    access_log /var/log/nginx/{{ server_name }}_access.log;
    error_log /var/log/nginx/{{ server_name }}_error.log;

    {% block ssl %}
    {# Child templates can override this block #}
    {% endblock %}

    {% block location %}
    location / {
        return 200 'Hello from {{ server_name }}';
    }
    {% endblock %}

    {% block extra %}
    {# Additional directives can be added by child templates #}
    {% endblock %}
}
```

### Child Template — Extends the Base

```jinja2
{# File: app_vhost.conf.j2 #}

{% extends "base_vhost.conf.j2" %}

{% block ssl %}
    ssl_certificate /etc/ssl/{{ server_name }}.crt;
    ssl_certificate_key /etc/ssl/{{ server_name }}.key;
{% endblock %}

{% block location %}
    location / {
        proxy_pass http://{{ upstream_address }};
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }

    location /static/ {
        alias /var/www/{{ server_name }}/static/;
        expires 30d;
    }
{% endblock %}
```

### How Inheritance Works

```
1. Jinja2 loads the child template
2. It sees {% extends "base_vhost.conf.j2" %}
3. It loads the base template
4. For each {% block %} in the base:
   - If the child defines the same block → use the child's version
   - If the child doesn't define it → use the base's version
5. The result is the base template with child's blocks filled in
```

### Using `{{ super() }}` — Extending a Block

```jinja2
{# Instead of REPLACING the base block, you can ADD to it #}

{% block location %}
    {{ super() }}    {# This includes the base template's location block #}
    
    {# And then add more #}
    location /admin/ {
        proxy_pass http://admin-backend:9090;
    }
{% endblock %}
```

## 14. Macros — Reusable Template Functions

Macros are like functions in programming. You define them once and call them multiple times.

### Defining and Using a Macro

```jinja2
{# Define a macro — like a function #}
{% macro nginx_upstream(name, servers, port=8080) %}
upstream {{ name }} {
    {% for server in servers %}
    server {{ server }}:{{ port }};
    {% endfor %}
}
{% endmacro %}

{# Call the macro — like calling a function #}
{{ nginx_upstream("web_backend", ["10.0.1.1", "10.0.1.2", "10.0.1.3"]) }}

{{ nginx_upstream("api_backend", ["10.0.2.1", "10.0.2.2"], port=3000) }}
```

**Output:**

```
upstream web_backend {
    server 10.0.1.1:8080;
    server 10.0.1.2:8080;
    server 10.0.1.3:8080;
}

upstream api_backend {
    server 10.0.2.1:3000;
    server 10.0.2.2:3000;
}
```

### Macro for Environment Variables

```jinja2
{% macro env_var(name, value, comment="") %}
{%- if comment %}
# {{ comment }}
{%- endif %}
- name: {{ name }}
  value: "{{ value }}"
{% endmacro %}

{# Use it in a Kubernetes manifest #}
spec:
  containers:
    - name: app
      env:
        {{ env_var("DB_HOST", database.host, "Database hostname") }}
        {{ env_var("DB_PORT", database.port, "Database port") }}
        {{ env_var("LOG_LEVEL", "info") }}
```

## 15. Include & Import — Breaking Templates into Pieces

### Include — Insert Another Template's Content

```jinja2
{# File: main.conf.j2 #}

# Main configuration
{% include 'header.j2' %}

server {
    {% include 'ssl_config.j2' %}
    {% include 'proxy_config.j2' %}
}

{% include 'footer.j2' %}
```


## 19. Jinja2 in Helm Charts (Go Templates Comparison)

Helm uses **Go templates**, not Jinja2. But the concepts are very similar. Here's a side-by-side comparison so you can translate between them:

```
Concept              Jinja2                              Go Templates (Helm)
───────              ──────                              ───────────────────
Variable             {{ name }}                          {{ .Values.name }}
Comment              {# comment #}                      {{/* comment */}}
If                   {% if x %}                          {{ if .Values.x }}
Else                 {% else %}                          {{ else }}
End If               {% endif %}                         {{ end }}
For loop             {% for x in list %}                 {{ range .Values.list }}
End loop             {% endfor %}                        {{ end }}
Default              {{ x | default("val") }}            {{ default "val" .Values.x }}
Upper                {{ x | upper }}                     {{ upper .Values.x }}
Include              {% include "file.j2" %}             {{ include "chart.helper" . }}
Pipe filters         {{ x | lower | trim }}              {{ .Values.x | lower | trim }}
```

### Same Template in Both

**Jinja2:**
```jinja2
{% if ssl_enabled %}
port: 443
{% else %}
port: 80
{% endif %}

{% for host in allowed_hosts %}
- {{ host }}
{% endfor %}
```

**Helm (Go Templates):**
```
{{ if .Values.sslEnabled }}
port: 443
{{ else }}
port: 80
{{ end }}

{{ range .Values.allowedHosts }}
- {{ . }}
{{ end }}
```

**Key Differences:**
- Go templates use `{{ end }}` instead of `{% endif %}` / `{% endfor %}`
- Go templates access values through `.Values.` prefix
- In Go's `range`, the current item is `.` (dot), not a named variable
- Filter syntax is slightly different

## 22. Custom Filters

You can create your own filters in Python for use in templates.

### Creating Custom Filters

```python
from jinja2 import Environment, FileSystemLoader

def to_megabytes(value):
    """Convert a value like '2Gi' or '512Mi' to megabytes."""
    value = str(value)
    if value.endswith("Gi"):
        return int(value[:-2]) * 1024
    elif value.endswith("Mi"):
        return int(value[:-2])
    elif value.endswith("Ki"):
        return int(value[:-2]) // 1024
    return int(value)

def mask_secret(value, show=4):
    """Mask a secret, showing only the last N characters."""
    if len(str(value)) <= show:
        return str(value)
    return "*" * (len(str(value)) - show) + str(value)[-show:]

def to_env_name(value):
    """Convert a string to an environment variable name."""
    return str(value).upper().replace("-", "_").replace(" ", "_")

# Register the filters
env = Environment(loader=FileSystemLoader("templates/"))
env.filters["to_megabytes"] = to_megabytes
env.filters["mask_secret"] = mask_secret
env.filters["to_env_name"] = to_env_name

# Now use them in templates:
template = env.from_string("""
Memory limit: {{ "2Gi" | to_megabytes }}Mi
API Key: {{ "sk-abc123def456" | mask_secret }}
Env name: {{ "my-app-name" | to_env_name }}
""")

print(template.render())
# Output:
# Memory limit: 2048Mi
# API Key: ********f456
# Env name: MY_APP_NAME
```

### Custom Filters in Ansible

Ansible has its own way of adding custom filters:

```python
# File: filter_plugins/custom_filters.py
# Place this in your Ansible project's filter_plugins/ directory

class FilterModule(object):
    def filters(self):
        return {
            'to_megabytes': self.to_megabytes,
            'env_name': self.env_name,
        }

    def to_megabytes(self, value):
        value = str(value)
        if value.endswith("Gi"):
            return int(value[:-2]) * 1024
        elif value.endswith("Mi"):
            return int(value[:-2])
        return int(value)

    def env_name(self, value):
        return str(value).upper().replace("-", "_")
```

## 23. Security Considerations

### Autoescaping — Preventing Injection

```python
from jinja2 import Environment, FileSystemLoader, select_autoescape

# For HTML templates, ALWAYS enable autoescape
env = Environment(
    loader=FileSystemLoader("templates/"),
    autoescape=select_autoescape(["html", "xml"]),  # Auto-escape HTML/XML files
)

# This prevents XSS (Cross-Site Scripting) attacks
# Without autoescape:  {{ user_input }}  → <script>alert('hacked')</script>
# With autoescape:     {{ user_input }}  → &lt;script&gt;alert('hacked')&lt;/script&gt;
```

### The `|safe` Filter — Use with Caution!

```jinja2
{# This marks content as "safe" and skips escaping #}
{# Only use when you TRUST the content #}
{{ trusted_html | safe }}

{# NEVER do this with user input! #}
{{ user_input | safe }}    ← DANGEROUS! Could allow XSS
```

### Sandboxed Environment

```python
from jinja2.sandbox import SandboxedEnvironment

# Use this when rendering untrusted templates
env = SandboxedEnvironment()
# This prevents templates from accessing dangerous functions
# like os.system(), open(), eval(), etc.
```

### Best Practices for Security

```
1. ALWAYS use autoescape for HTML output
2. NEVER use |safe with user-provided data
3. Use SandboxedEnvironment for untrusted templates
4. Don't pass sensitive functions (os, subprocess) to template context
5. Validate all template inputs before rendering
6. Don't render user-provided template strings (template injection!)
```

## 24. Common Mistakes and Debugging

### Mistake 1: Using Jinja2 Syntax in Non-Template Context

```yaml
# WRONG — This is a plain YAML file, NOT a Jinja2 template
# Jinja2 syntax here will be treated as literal text
apiVersion: v1
kind: ConfigMap
data:
  value: {{ my_var }}    # This WON'T be replaced — it's not a template!

# RIGHT — Use the Ansible template module to process .j2 files
# Or use Helm for Kubernetes manifests
```

### Mistake 2: Indentation in YAML Templates

```jinja2
{# WRONG — Jinja2 indentation breaks YAML structure #}
spec:
  containers:
{% for container in containers %}
    - name: {{ container.name }}
{% endfor %}

{# RIGHT — Match the YAML indentation #}
spec:
  containers:
    {% for container in containers %}
    - name: {{ container.name }}
      image: {{ container.image }}
    {% endfor %}

{# ALSO RIGHT — Use indent filter #}
spec:
  containers:
{{ containers_yaml | indent(4, first=true) }}
```

### Mistake 3: Forgetting to Quote Strings in YAML

```jinja2
{# WRONG — If app_port is 80, YAML sees a number, not a string #}
port: {{ app_port }}

{# RIGHT — Quote if you need a string #}
port: "{{ app_port }}"

{# For booleans: #}
debug: {{ debug_mode }}        {# Renders as: debug: True (Python boolean) #}
debug: {{ debug_mode | lower }}  {# Renders as: debug: true (YAML boolean) #}
```

### Mistake 4: Variable Not Defined

```jinja2
{# ERROR: 'custom_port' is undefined #}
port: {{ custom_port }}

{# FIX: Use default filter #}
port: {{ custom_port | default(8080) }}

{# FIX: Check if defined first #}
{% if custom_port is defined %}
port: {{ custom_port }}
{% endif %}
```

### Debugging Tips

```bash
# 1. In Ansible — Increase verbosity
ansible-playbook playbook.yml -vvv

# 2. In Ansible — Use debug module to print variables
- debug:
    var: my_variable
- debug:
    msg: "{{ my_complex_expression }}"

# 3. In Python — Catch template errors
from jinja2 import Template, UndefinedError, TemplateSyntaxError

try:

except UndefinedError as e:
    print(f"Variable error: {e}")

    print(f"Syntax error on line {e.lineno}: {e}")
```
