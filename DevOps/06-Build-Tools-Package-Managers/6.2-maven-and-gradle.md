## 1. Why Do We Need Build Tools?

Imagine you're building a Java app with 50 source files and 30 external libraries. Without a build tool, you'd have to:

1. Download each library manually from the internet
2. Put them in the right folder
3. Compile every `.java` file with the right classpath
4. Run tests manually
5. Package everything into a JAR with the right manifest
6. Do this EVERY SINGLE TIME you make a change

That's insanity. **Build tools automate all of this.**

### What Build Tools Do

| Task | Manual Way | With Build Tool |
|------|-----------|----------------|
| **Download libraries** | Go to website, download ZIP, extract | One line in config file |
| **Compile code** | `javac -cp lib1.jar:lib2.jar:... *.java` | `mvn compile` |
| **Run tests** | Find test runner, configure, run manually | `mvn test` |
| **Create JAR** | `jar cf myapp.jar ...` with correct manifest | `mvn package` |
| **Update a library** | Download new version, replace old one | Change version number |
| **Build 50-file project** | Write a huge shell script | One command |

## 2. What is Maven?

**Maven** (pronounced "may-vin") is the most widely used Java build tool. Created by Apache in 2004, it's been the **default choice** for Java projects for over 20 years.

### Key Concepts

> Think of Maven like a **factory assembly line**. You give it raw materials (source code + dependencies), and it follows a **standard process** (lifecycle) to produce a finished product (JAR/WAR file).

- **Convention over Configuration** — Maven has a standard project structure. If you follow it, you barely need any configuration.
- **POM (Project Object Model)** — The `pom.xml` file describes your project: name, version, dependencies, plugins.
- **Dependency Management** — Maven downloads libraries from the internet automatically.
- **Lifecycle** — A fixed sequence of build phases (compile → test → package → install → deploy).
- **Plugins** — Everything Maven does is through plugins. Compilation? A plugin. Testing? A plugin. Packaging? A plugin.

## 4. Maven Project Structure

Maven follows a **standard directory layout**. If you follow it, Maven knows where everything is without you telling it.

```
my-java-app/
├── pom.xml                          ← The build configuration file
├── src/
│   ├── main/
│   │   ├── java/                    ← Your application source code
│   │   │   └── com/myapp/
│   │   │       ├── App.java
│   │   │       ├── UserService.java
│   │   │       └── UserController.java
│   │   └── resources/               ← Configuration files, templates
│   │       ├── application.yml
│   │       └── logback.xml
│   └── test/
│       ├── java/                    ← Your test code
│       │   └── com/myapp/
│       │       ├── AppTest.java
│       │       └── UserServiceTest.java
│       └── resources/               ← Test-specific configs
│           └── test-application.yml
└── target/                          ← Build output (generated — don't commit to Git!)
    ├── classes/                     ← Compiled .class files
    ├── test-classes/                ← Compiled test classes
    ├── my-java-app-1.0.0.jar       ← The final artifact!
    └── surefire-reports/            ← Test reports
```

### Key Directories

| Directory | Purpose |
|-----------|---------|
| `src/main/java` | Application source code |
| `src/main/resources` | Config files (application.yml, etc.) |
| `src/test/java` | Test source code |
| `src/test/resources` | Test config files |
| `target/` | Build output — generated by Maven, **add to .gitignore** |

### Create a New Maven Project (Quickstart)

```bash
# Generate a new project from a template (archetype)
mvn archetype:generate \
  -DgroupId=com.myapp \
  -DartifactId=my-java-app \
  -DarchetypeArtifactId=maven-archetype-quickstart \
  -DarchetypeVersion=1.4 \
  -DinteractiveMode=false

cd my-java-app
```

This creates the standard structure automatically!

## 5. The pom.xml — Maven's Heart

The `pom.xml` (Project Object Model) is the **single most important file** in a Maven project. It tells Maven everything about your project.

### Minimal pom.xml

```xml
<?xml version="1.0" encoding="UTF-8"?>
<project xmlns="http://maven.apache.org/POM/4.0.0"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="http://maven.apache.org/POM/4.0.0
         http://maven.apache.org/xsd/maven-4.0.0.xsd">

    <!-- POM model version (always 4.0.0) -->
    <modelVersion>4.0.0</modelVersion>

    <!-- Project coordinates — uniquely identify your project -->
    <groupId>com.myapp</groupId>          <!-- Like a namespace/org name -->
    <artifactId>my-java-app</artifactId>   <!-- Project name -->
    <version>1.0.0</version>               <!-- Project version -->
    <packaging>jar</packaging>             <!-- Output type: jar, war, pom -->

    <!-- Project metadata -->
    <name>My Java App</name>
    <description>A demo application</description>

    <!-- Java version -->
    <properties>
        <maven.compiler.source>17</maven.compiler.source>
        <maven.compiler.target>17</maven.compiler.target>
        <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
    </properties>

    <!-- Dependencies (external libraries) -->
    <dependencies>
        <!-- JUnit 5 for testing -->
        <dependency>
            <groupId>org.junit.jupiter</groupId>
            <artifactId>junit-jupiter</artifactId>
            <version>5.10.1</version>
            <scope>test</scope>
        </dependency>
    </dependencies>

</project>
```

### Understanding GAV Coordinates

Every Maven artifact is uniquely identified by three values: **G**roup, **A**rtifact, **V**ersion (GAV):

```
groupId:artifactId:version
com.myapp:my-java-app:1.0.0
```

Think of it like a **postal address**:

| Coordinate | Analogy | Example |
|-----------|---------|---------|
| `groupId` | City/Organization | `com.google.guava`, `org.springframework` |
| `artifactId` | Street/Project name | `guava`, `spring-boot-starter-web` |
| `version` | House number | `32.1.2`, `3.2.0` |

### A Real-World pom.xml (Spring Boot App)

```xml
<?xml version="1.0" encoding="UTF-8"?>
<project xmlns="http://maven.apache.org/POM/4.0.0"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="http://maven.apache.org/POM/4.0.0
         http://maven.apache.org/xsd/maven-4.0.0.xsd">
    <modelVersion>4.0.0</modelVersion>

    <!-- Inherit from Spring Boot parent (provides sensible defaults) -->
    <parent>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-parent</artifactId>
        <version>3.2.0</version>
    </parent>

    <groupId>com.mycompany</groupId>
    <artifactId>user-service</artifactId>
    <version>2.1.0</version>

    <properties>
        <java.version>17</java.version>
    </properties>

    <dependencies>
        <!-- Spring Boot Web (REST APIs) -->
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-web</artifactId>
        </dependency>

        <!-- Spring Boot Data JPA (Database) -->
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-data-jpa</artifactId>
        </dependency>

        <!-- PostgreSQL Driver -->
        <dependency>
            <groupId>org.postgresql</groupId>
            <artifactId>postgresql</artifactId>
            <scope>runtime</scope>
        </dependency>

        <!-- Testing -->
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-test</artifactId>
            <scope>test</scope>
        </dependency>
    </dependencies>

    <build>
        <plugins>
            <!-- Spring Boot Maven Plugin (creates executable JAR) -->
            <plugin>
                <groupId>org.springframework.boot</groupId>
                <artifactId>spring-boot-maven-plugin</artifactId>
            </plugin>
        </plugins>
    </build>
</project>
```

## 6. Maven Build Lifecycle

Maven has a **fixed sequence of phases** called the **build lifecycle**. Each phase runs all previous phases automatically.

### Default Lifecycle (the main one)

```
validate  →  compile  →  test  →  package  →  verify  →  install  →  deploy
   1           2          3         4           5          6           7
```

| Phase | What It Does |
|-------|-------------|
| `validate` | Check if the project structure is correct |
| `compile` | Compile `.java` files → `.class` files |
| `test` | Run unit tests (using JUnit, TestNG, etc.) |
| `package` | Create the artifact (JAR, WAR) |
| `verify` | Run integration tests and quality checks |
| `install` | Copy the artifact to your local Maven repository (~/.m2/) |
| `deploy` | Upload the artifact to a remote repository (Nexus, Artifactory) |

### The Key Insight: Phases Run in Order

When you run `mvn package`, Maven automatically runs:

```
validate → compile → test → package
```

When you run `mvn install`, Maven runs:

```
validate → compile → test → package → verify → install
```

You **don't** have to run each phase separately. Just tell Maven the **last phase** you want, and it runs everything before it.

### Other Lifecycles

| Lifecycle | Phases | Purpose |
|-----------|--------|---------|
| **default** | validate → compile → test → package → verify → install → deploy | Main build |
| **clean** | pre-clean → clean → post-clean | Delete previous build output |
| **site** | pre-site → site → post-site → site-deploy | Generate documentation |

## 7. Maven Commands — The Ones You'll Use Daily

### Essential Commands

```bash
# Clean — Delete the target/ directory (previous build output)
mvn clean

# Compile — Compile source code
mvn compile

# Test — Run unit tests
mvn test

# Package — Create JAR/WAR file
mvn package

# Clean + Package — Most common combination!
mvn clean package

# Install — Build and put in local repository
mvn clean install

# Skip tests (when you just want to build fast)
mvn clean package -DskipTests

# Run a Spring Boot app
mvn spring-boot:run
```

### Useful Flags

```bash
# Quiet mode (less output)
mvn clean package -q

# Debug mode (very verbose — useful for troubleshooting)
mvn clean package -X

# Offline mode (don't download anything)
mvn clean package -o

# Parallel builds (faster on multi-core machines)
mvn clean package -T 4        # 4 threads
mvn clean package -T 1C       # 1 thread per CPU core

# Show effective POM (what Maven actually uses after inheritance)
mvn help:effective-pom

# Show dependency tree
mvn dependency:tree

# Check for dependency updates
mvn versions:display-dependency-updates
```

### Understanding the Output


## 8. Maven Dependencies

### Adding a Dependency

To use a library, add it to the `<dependencies>` section of `pom.xml`:

```xml
<dependencies>
    <!-- Google's Guava library -->
    <dependency>
        <groupId>com.google.guava</groupId>
        <artifactId>guava</artifactId>
        <version>32.1.2-jre</version>
    </dependency>
</dependencies>
```

When you run `mvn compile`, Maven:
1. Checks if Guava 32.1.2 is already in your local cache (`~/.m2/repository/`)
2. If not, downloads it from **Maven Central** (the default public repository)
3. Adds it to the **classpath** so your code can use it

### Dependency Scopes

The `<scope>` tag tells Maven **when** a dependency is needed:

| Scope | Available At | Included in JAR? | Use Case |
|-------|-------------|-------------------|----------|
| `compile` (default) | Compile + Test + Runtime | Yes | Libraries your code directly uses |
| `provided` | Compile + Test | **No** | Server provides it (e.g., Servlet API in Tomcat) |
| `runtime` | Test + Runtime | Yes | Needed at runtime but not compile (e.g., JDBC driver) |
| `test` | Test only | **No** | Testing libraries (JUnit, Mockito) |
| `system` | Compile + Test | No | Explicit path to a JAR (avoid this!) |

```xml
<dependency>
    <groupId>org.junit.jupiter</groupId>
    <artifactId>junit-jupiter</artifactId>
    <version>5.10.1</version>
    <scope>test</scope>     <!-- Only for testing — NOT included in the final JAR -->
</dependency>

<dependency>
    <groupId>org.postgresql</groupId>
    <artifactId>postgresql</artifactId>
    <version>42.7.1</version>
    <scope>runtime</scope>  <!-- Needed when app runs, not when compiling -->
</dependency>
```

### Dependency Tree

See what dependencies your project has (including transitive ones):

```bash
mvn dependency:tree

# Output:
# com.myapp:my-app:jar:1.0.0
# ├── org.springframework.boot:spring-boot-starter-web:jar:3.2.0
# │   ├── org.springframework.boot:spring-boot-starter:jar:3.2.0
# │   │   ├── org.springframework.boot:spring-boot:jar:3.2.0
# │   │   ├── org.springframework.boot:spring-boot-autoconfigure:jar:3.2.0
# │   │   └── org.springframework:spring-core:jar:6.1.1
# │   ├── org.springframework:spring-web:jar:6.1.1
# │   └── org.apache.tomcat.embed:tomcat-embed-core:jar:10.1.16
# └── org.junit.jupiter:junit-jupiter:jar:5.10.1 (test)
```

### Dependency Management (Parent POM)

In multi-module projects, use `<dependencyManagement>` to set versions in one place:

```xml
<!-- Parent pom.xml -->
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>com.google.guava</groupId>
            <artifactId>guava</artifactId>
            <version>32.1.2-jre</version>  <!-- Version defined here -->
        </dependency>
    </dependencies>
</dependencyManagement>

<!-- Child pom.xml — no need to specify version! -->
<dependencies>
    <dependency>
        <groupId>com.google.guava</groupId>
        <artifactId>guava</artifactId>
        <!-- Version inherited from parent -->
    </dependency>
</dependencies>
```

### Excluding a Transitive Dependency

Sometimes a transitive dependency causes conflicts. You can exclude it:

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
    <exclusions>
        <exclusion>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-tomcat</artifactId>
        </exclusion>
    </exclusions>
</dependency>

<!-- Use Jetty instead of Tomcat -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-jetty</artifactId>
</dependency>
```

## 9. Maven Repositories

### What is a Maven Repository?

A repository is a **storage location** for Maven artifacts (JARs, POMs, etc.).

### Types of Repositories

```
Your Code → mvn install → Local Repo (~/.m2/repository/)
                              ↓ (if not found locally)
                         Remote Repo (Maven Central / Nexus / Artifactory)
                              ↓ (downloads to)
                         Local Repo (~/.m2/repository/)
```

| Type | Location | Purpose |
|------|----------|---------|
| **Local** | `~/.m2/repository/` | Cache on your machine |
| **Central** | `https://repo.maven.apache.org/maven2` | Public default repository |
| **Remote/Private** | Your company's Nexus/Artifactory server | Private artifacts + proxy cache |

### Configuring a Custom Repository

```xml
<!-- In pom.xml -->
<repositories>
    <repository>
        <id>company-nexus</id>
        <name>Company Nexus Repository</name>
        <url>https://nexus.mycompany.com/repository/maven-public/</url>
    </repository>
</repositories>
```

### Local Repository Cache

```bash
# Your local repository location
ls ~/.m2/repository/

# Clear the local cache (force re-download everything)
rm -rf ~/.m2/repository/

# Clear cache for a specific dependency
rm -rf ~/.m2/repository/com/google/guava/
```

### Maven Settings (settings.xml)

Global Maven configuration lives in `~/.m2/settings.xml`:

```xml
<settings>
    <!-- Repository credentials -->
    <servers>
        <server>
            <id>company-nexus</id>
            <username>deploy-user</username>
            <password>${env.NEXUS_PASSWORD}</password>
        </server>
    </servers>

    <!-- Mirror: route ALL requests through company Nexus -->
    <mirrors>
        <mirror>
            <id>company-mirror</id>
            <mirrorOf>*</mirrorOf>
            <url>https://nexus.mycompany.com/repository/maven-public/</url>
        </mirror>
    </mirrors>
</settings>
```

## 10. Maven Plugins

**Everything Maven does is through plugins.** Compilation, testing, packaging — all plugins.

### Default Plugins (you get these for free)

| Plugin | Phase | What It Does |
|--------|-------|-------------|
| `maven-compiler-plugin` | compile | Compiles Java source code |
| `maven-surefire-plugin` | test | Runs unit tests |
| `maven-jar-plugin` | package | Creates JAR files |
| `maven-war-plugin` | package | Creates WAR files |
| `maven-install-plugin` | install | Installs to local repo |
| `maven-deploy-plugin` | deploy | Uploads to remote repo |

### Configuring a Plugin

```xml
<build>
    <plugins>
        <!-- Configure the compiler plugin -->
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-compiler-plugin</artifactId>
            <version>3.12.1</version>
            <configuration>
                <source>17</source>
                <target>17</target>
            </configuration>
        </plugin>

        <!-- Create an executable JAR with dependencies (fat JAR) -->
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-shade-plugin</artifactId>
            <version>3.5.1</version>
            <executions>
                <execution>
                    <phase>package</phase>
                    <goals>
                        <goal>shade</goal>
                    </goals>
                    <configuration>
                        <transformers>
                            <transformer implementation="org.apache.maven.plugins.shade.resource.ManifestResourceTransformer">
                                <mainClass>com.myapp.App</mainClass>
                            </transformer>
                        </transformers>
                    </configuration>
                </execution>
            </executions>
        </plugin>
    </plugins>
</build>
```

### Useful Plugins for DevOps

| Plugin | Command | Purpose |
|--------|---------|---------|
| `maven-shade-plugin` | `mvn package` | Create fat JAR (all deps included) |
| `spring-boot-maven-plugin` | `mvn spring-boot:run` | Run Spring Boot app |
| `dockerfile-maven-plugin` | `mvn dockerfile:build` | Build Docker image |
| `jacoco-maven-plugin` | `mvn jacoco:report` | Code coverage reports |
| `maven-release-plugin` | `mvn release:prepare` | Automate releases |
| `versions-maven-plugin` | `mvn versions:display-dependency-updates` | Find outdated deps |

## 11. Maven Profiles — Environment-Specific Builds

Profiles let you customize the build for different environments (dev, staging, prod).

```xml
<profiles>
    <!-- Development profile -->
    <profile>
        <id>dev</id>
        <activation>
            <activeByDefault>true</activeByDefault>
        </activation>
        <properties>
            <db.url>jdbc:postgresql://localhost:5432/devdb</db.url>
            <log.level>DEBUG</log.level>
        </properties>
    </profile>

    <!-- Production profile -->
    <profile>
        <id>prod</id>
        <properties>
            <db.url>jdbc:postgresql://prod-db.mycompany.com:5432/proddb</db.url>
            <log.level>WARN</log.level>
        </properties>
    </profile>
</profiles>
```

```bash
# Build with dev profile (default)
mvn clean package

# Build with production profile
mvn clean package -Pprod

# Activate multiple profiles
mvn clean package -Pprod,docker
```

## 12. Maven Wrapper (mvnw)

The **Maven Wrapper** ensures everyone uses the **same Maven version** — no more "it works with my Maven version" issues.

### Setting It Up

```bash
# Add Maven Wrapper to your project
mvn wrapper:wrapper -Dmaven=3.9.6

# This creates:
# .mvn/wrapper/maven-wrapper.properties   ← Maven version config
# mvnw                                     ← Unix wrapper script
# mvnw.cmd                                 ← Windows wrapper script
```

### Using It

```bash
# Instead of 'mvn', use './mvnw'
./mvnw clean package

# Windows
mvnw.cmd clean package
```

### Why It Matters for DevOps

- **CI/CD:** No need to install Maven on the build server — the wrapper handles it
- **Consistency:** Everyone (developers, CI/CD) uses the exact same Maven version
- **Commit it to Git:** Always commit `.mvn/` and `mvnw` to the repository

```yaml
# GitHub Actions — no need for setup-maven step!
steps:
  - uses: actions/checkout@v4
  - uses: actions/setup-java@v4
    with:
      java-version: '17'
      distribution: 'temurin'
  - run: ./mvnw clean package  # Uses the wrapper!
```

## 13. Multi-Module Maven Projects

Large projects are split into **modules** — each module is a sub-project with its own `pom.xml`.

### Structure

```
my-platform/
├── pom.xml                    ← Parent POM
├── user-service/
│   ├── pom.xml                ← Module POM
│   └── src/
├── order-service/
│   ├── pom.xml
│   └── src/
├── common-lib/
│   ├── pom.xml
│   └── src/
└── api-gateway/
    ├── pom.xml
    └── src/
```

### Parent POM

```xml
<!-- my-platform/pom.xml -->
<project>
    <modelVersion>4.0.0</modelVersion>
    <groupId>com.myplatform</groupId>
    <artifactId>my-platform</artifactId>
    <version>1.0.0</version>
    <packaging>pom</packaging>      <!-- "pom" = this is a parent, not a JAR -->

    <modules>
        <module>common-lib</module>
        <module>user-service</module>
        <module>order-service</module>
        <module>api-gateway</module>
    </modules>

    <dependencyManagement>
        <dependencies>
            <!-- All modules use the same Spring Boot version -->
            <dependency>
                <groupId>org.springframework.boot</groupId>
                <artifactId>spring-boot-dependencies</artifactId>
                <version>3.2.0</version>
                <type>pom</type>
                <scope>import</scope>
            </dependency>
        </dependencies>
    </dependencyManagement>
</project>
```

### Build All Modules

```bash
# Build everything
cd my-platform
mvn clean package

# Build a specific module (and its dependencies)
mvn clean package -pl user-service -am
# -pl = project list (which module)
# -am = also make (build dependencies of that module too)
```

## 14. What is Gradle?

**Gradle** is the modern alternative to Maven. It was created in 2007 and has been gaining popularity, especially in the Android and Spring Boot ecosystems.

### Key Differences from Maven

| Feature | Maven | Gradle |
|---------|-------|--------|
| **Config format** | XML (`pom.xml`) | Groovy/Kotlin DSL (`build.gradle`) |
| **Config style** | Declarative (rigid structure) | Declarative + Imperative (flexible scripts) |
| **Performance** | Builds everything every time | Incremental + cached builds |
| **Daemon** | No (starts fresh each time) | Yes (background process = faster starts) |
| **Android** | Rarely used | **The** build tool for Android |
| **Learning curve** | Easier to start | More powerful but steeper curve |

### Why Gradle?

> If Maven is a **factory assembly line** (fixed steps), Gradle is a **workshop with power tools** (you build what you want, how you want).

```
Maven:   "Follow MY rules. Here's your XML. Don't get creative."
Gradle:  "Here are powerful tools. Build however you want. Use code!"
```

## 17. The build.gradle — Gradle's Heart

### Groovy DSL (Most Common)

```groovy
// build.gradle

// Apply plugins
plugins {
    id 'java'                              // Java compilation support
    id 'application'                       // Makes it runnable
    id 'org.springframework.boot' version '3.2.0'   // Spring Boot
    id 'io.spring.dependency-management' version '1.1.4'
}

// Project info
group = 'com.myapp'
version = '1.0.0'

// Java version
java {
    sourceCompatibility = JavaVersion.VERSION_17
    targetCompatibility = JavaVersion.VERSION_17
}

// Where to download dependencies from
repositories {
    mavenCentral()            // Public Maven Central
    // maven { url 'https://nexus.mycompany.com/repository/maven-public/' }
}

// Dependencies
dependencies {
    // Application dependencies
    implementation 'org.springframework.boot:spring-boot-starter-web'
    implementation 'org.springframework.boot:spring-boot-starter-data-jpa'

    // Runtime only
    runtimeOnly 'org.postgresql:postgresql'

    // Test dependencies
    testImplementation 'org.springframework.boot:spring-boot-starter-test'
}

// Configure test runner
tasks.named('test') {
    useJUnitPlatform()
}

// Main class for 'application' plugin
application {
    mainClass = 'com.myapp.App'
}
```

### Kotlin DSL (Newer, Type-Safe)

```kotlin
// build.gradle.kts

plugins {
    java
    application
    id("org.springframework.boot") version "3.2.0"
    id("io.spring.dependency-management") version "1.1.4"
}

group = "com.myapp"
version = "1.0.0"

java {
    sourceCompatibility = JavaVersion.VERSION_17
}

repositories {
    mavenCentral()
}

dependencies {
    implementation("org.springframework.boot:spring-boot-starter-web")
    implementation("org.springframework.boot:spring-boot-starter-data-jpa")
    runtimeOnly("org.postgresql:postgresql")
    testImplementation("org.springframework.boot:spring-boot-starter-test")
}

tasks.test {
    useJUnitPlatform()
}
```

### Maven XML vs Gradle Groovy — Side by Side

**Adding a dependency:**

```xml
<!-- Maven (verbose XML) -->
<dependency>
    <groupId>com.google.guava</groupId>
    <artifactId>guava</artifactId>
    <version>32.1.2-jre</version>
</dependency>
```

```groovy
// Gradle (one clean line!)
implementation 'com.google.guava:guava:32.1.2-jre'
```

Much cleaner, right?

## 18. Gradle Build Lifecycle

Gradle's lifecycle is different from Maven's. It has three phases:

```
1. INITIALIZATION  →  2. CONFIGURATION  →  3. EXECUTION
```

| Phase | What Happens |
|-------|-------------|
| **Initialization** | Reads `settings.gradle`, determines which projects are in the build |
| **Configuration** | Reads `build.gradle` for ALL projects, builds the task graph |
| **Execution** | Runs only the requested tasks (and their dependencies) |

### Tasks (Not Phases!)

Unlike Maven's phases, Gradle uses **tasks**. Tasks have dependencies on each other:

```
compileJava → processResources → classes → jar → assemble
                                         → test → check → build
```

```bash
# See all available tasks
./gradlew tasks

# See task dependencies
./gradlew build --dry-run
```

## 19. Gradle Commands — The Ones You'll Use Daily

```bash
# Build (compile + test + jar)
./gradlew build

# Clean (delete build/ directory)

./gradlew build --parallel
```

### Gradle Daemon

Gradle uses a **background daemon** process for faster builds:

```bash
# Check daemon status

# Stop all daemons
./gradlew --stop

# Build without daemon
./gradlew build --no-daemon
```

The daemon keeps the JVM warm, so subsequent builds start **much faster** than Maven.

## 20. Gradle Dependencies

### Dependency Configurations (Scopes)

| Gradle Configuration | Maven Scope | Meaning |
|---------------------|-------------|---------|
| `implementation` | `compile` | Needed to compile and run, but NOT exposed to consumers |
| `api` | `compile` | Needed to compile and run, AND exposed to consumers |
| `runtimeOnly` | `runtime` | Only needed at runtime |
| `compileOnly` | `provided` | Only needed at compile time |
| `testImplementation` | `test` | Only for testing |
| `testRuntimeOnly` | `test` (runtime) | Test runtime only |
| `annotationProcessor` | — | Annotation processing (Lombok, MapStruct) |

```groovy
dependencies {
    // Your code uses this library directly
    implementation 'com.google.guava:guava:32.1.2-jre'

    // Only needed when the app runs (not for compilation)
    runtimeOnly 'org.postgresql:postgresql:42.7.1'

    // Only for testing
    testImplementation 'org.junit.jupiter:junit-jupiter:5.10.1'

    // Annotation processor (generates code at compile time)
    annotationProcessor 'org.projectlombok:lombok:1.18.30'
    compileOnly 'org.projectlombok:lombok:1.18.30'
}
```

### `implementation` vs `api`

This is a common source of confusion. Here's the simple explanation:

```
Module A → depends on → Module B → depends on → Guava

If Module B declares Guava as:
  implementation → Module A CANNOT see/use Guava
  api            → Module A CAN see/use Guava
```

**Rule of thumb:** Use `implementation` by default. Only use `api` when the dependency is part of your public API.

### Version Catalogs (Modern Approach)

Gradle 7+ supports **version catalogs** — one central place for all dependency versions:

```toml
# gradle/libs.versions.toml
[versions]
spring-boot = "3.2.0"
junit = "5.10.1"
guava = "32.1.2-jre"

[libraries]
spring-boot-web = { module = "org.springframework.boot:spring-boot-starter-web", version.ref = "spring-boot" }
spring-boot-test = { module = "org.springframework.boot:spring-boot-starter-test", version.ref = "spring-boot" }
junit = { module = "org.junit.jupiter:junit-jupiter", version.ref = "junit" }
guava = { module = "com.google.guava:guava", version.ref = "guava" }

[plugins]
spring-boot = { id = "org.springframework.boot", version.ref = "spring-boot" }
```

```groovy
// build.gradle — reference from catalog
dependencies {
    implementation libs.spring.boot.web
    implementation libs.guava
    testImplementation libs.junit
}
```

## 21. Gradle Wrapper (gradlew)

Just like Maven Wrapper, the **Gradle Wrapper** ensures consistent Gradle versions.

### Generate the Wrapper

```bash
gradle wrapper --gradle-version 8.5
```

### Files Created

```
my-project/
├── gradlew                    ← Unix wrapper script (chmod +x)
├── gradlew.bat                ← Windows wrapper script
└── gradle/
    └── wrapper/
        ├── gradle-wrapper.jar            ← The wrapper bootstrapper
        └── gradle-wrapper.properties     ← Gradle version config
```

### gradle-wrapper.properties

```properties
distributionUrl=https\://services.gradle.org/distributions/gradle-8.5-bin.zip
distributionBase=GRADLE_USER_HOME
distributionPath=wrapper/dists
zipStoreBase=GRADLE_USER_HOME
zipStorePath=wrapper/dists
```

### Usage

```bash
# Always use ./gradlew instead of gradle
./gradlew build

# Windows
gradlew.bat build

# Update the wrapper version
./gradlew wrapper --gradle-version 8.6
```

> **Always commit the wrapper to Git!** It's the easiest way to ensure everyone uses the same Gradle version.

## 22. Gradle vs Maven — Head-to-Head Comparison

| Feature | Maven | Gradle |
|---------|-------|--------|
| **Config format** | XML (`pom.xml`) | Groovy/Kotlin (`build.gradle`) |
| **Config verbosity** | Very verbose | Concise |
| **Learning curve** | Easier to start | More powerful, steeper |
| **Performance** | Slower (no caching) | **Faster** (incremental, cached, daemon) |
| **Flexibility** | Rigid (convention-based) | Very flexible (scripting) |
| **IDE support** | Excellent | Excellent |
| **Android** | Not used | **The** standard |
| **Spring Boot** | Both supported | Both supported |
| **Plugin ecosystem** | Very mature, huge | Growing, sometimes ports Maven plugins |
| **Multi-module** | Good | Better (faster, more flexible) |
| **Adoption** | Still #1 in enterprise | Growing fast, #1 in Android/new projects |
| **Wrapper** | `mvnw` | `gradlew` |

### Build Speed Comparison

```
First Build (clean):
  Maven:  45 seconds
  Gradle: 40 seconds

Second Build (no changes):
  Maven:  30 seconds (rebuilds everything!)
  Gradle:  2 seconds (UP-TO-DATE — nothing to do!)

Build After Small Change:
  Maven:  30 seconds (recompiles everything)
  Gradle:  5 seconds (recompiles only changed files)
```

Gradle's **incremental builds** and **build cache** make a HUGE difference in CI/CD pipeline speed.

### Which One Should You Use?

| Situation | Recommendation |
|-----------|---------------|
| New Java project | Either works — Gradle is slightly more modern |
| Existing project | Use whatever it already uses |
| Android app | Gradle (only option) |
| Enterprise / legacy | Probably Maven |
| Need max build speed | Gradle |
| Simple project, simple needs | Maven (less to learn) |
| Complex build logic | Gradle (scripting capabilities) |

## 23. CI/CD Pipeline Examples

### Maven in GitHub Actions

```yaml
name: Java CI with Maven

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Set up JDK 17
        uses: actions/setup-java@v4
        with:
          java-version: '17'
          distribution: 'temurin'
          cache: 'maven'         # Cache Maven dependencies!

      - name: Build with Maven
        run: ./mvnw clean verify

      - name: Upload artifact
        uses: actions/upload-artifact@v4
        with:
          name: app-jar
          path: target/*.jar
```

### Gradle in GitHub Actions

```yaml
name: Java CI with Gradle

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Set up JDK 17
        uses: actions/setup-java@v4
        with:
          java-version: '17'
          distribution: 'temurin'

      - name: Setup Gradle
        uses: gradle/actions/setup-gradle@v3   # Handles caching automatically!

      - name: Build with Gradle
        run: ./gradlew build

      - name: Upload artifact
        uses: actions/upload-artifact@v4
        with:
          name: app-jar
          path: build/libs/*.jar
```

### Maven in Jenkins (Jenkinsfile)

```groovy
pipeline {
    agent any

    tools {
        maven 'Maven-3.9'
        jdk 'JDK-17'
    }

    stages {
        stage('Build') {
            steps {
                sh './mvnw clean package -DskipTests'
            }
        }

        stage('Test') {
            steps {
                sh './mvnw test'
            }
            post {
                always {
                    junit 'target/surefire-reports/*.xml'
                }
            }
        }

        stage('Docker Build') {
            steps {
                sh 'docker build -t myapp:${BUILD_NUMBER} .'
            }
        }
    }
}
```

### Docker Multi-Stage Build (Maven)

```dockerfile
# Stage 1: Build with Maven
FROM maven:3.9-eclipse-temurin-17 AS builder
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline          # Download deps (cached layer!)
COPY src ./src
RUN mvn clean package -DskipTests

# Stage 2: Run
FROM eclipse-temurin:17-jre-alpine
WORKDIR /app
COPY --from=builder /app/target/*.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]
```

### Docker Multi-Stage Build (Gradle)

```dockerfile
# Stage 1: Build with Gradle
FROM gradle:8.5-jdk17 AS builder
WORKDIR /app
COPY build.gradle settings.gradle ./
COPY gradle ./gradle
RUN gradle dependencies --no-daemon     # Download deps (cached layer!)
COPY src ./src
RUN gradle build -x test --no-daemon

# Stage 2: Run
FROM eclipse-temurin:17-jre-alpine
WORKDIR /app
COPY --from=builder /app/build/libs/*.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]
```

## 24. Troubleshooting Common Issues

### Issue 1: "Could not resolve dependencies"

```
ERROR: Could not resolve com.example:my-lib:1.0.0
```

**Fix Checklist:**
```bash
# 1. Check if the dependency exists
# Search on https://search.maven.org/

# 2. Check your repository configuration
mvn help:effective-settings    # Maven
./gradlew properties           # Gradle

# 3. Clear local cache and retry
rm -rf ~/.m2/repository/com/example/my-lib/
mvn clean install

# 4. Check network/proxy
mvn clean install -X | grep -i proxy
```

### Issue 2: "Compilation failure"

```
ERROR: java: error: release version 17 not supported
```

**Fix:** Your JDK version doesn't match the project configuration:

```bash
# Check your Java version
java -version
javac -version

# Make sure it matches pom.xml/build.gradle
# Maven: <maven.compiler.source>17</maven.compiler.source>
# Gradle: sourceCompatibility = JavaVersion.VERSION_17
```

### Issue 3: "Tests failing in CI but passing locally"

| Possible Cause | Fix |
|----------------|-----|
| Different Java version | Specify exact version in CI |
| Different OS | Check for OS-specific file paths |
| Database not available | Use embedded database for tests (H2) |
| Timezone differences | Set timezone in CI: `TZ=UTC` |
| Random test order | Make tests independent (no shared state) |

### Issue 4: "Build is too slow"

```bash
# Maven — parallel builds
mvn clean package -T 1C

# Gradle — parallel + cached + daemon
./gradlew build --parallel --build-cache

# Both — skip tests when just building
mvn clean package -DskipTests
./gradlew build -x test

# Docker — fix layer caching (copy deps first, then source)
```

### Issue 5: "Out of memory"

```bash
# Maven
export MAVEN_OPTS="-Xmx2g -XX:MaxMetaspaceSize=512m"

# Gradle (in gradle.properties)
org.gradle.jvmargs=-Xmx2g -XX:MaxMetaspaceSize=512m

# CI/CD — increase runner memory
```

## 25. Summary

### Maven vs Gradle at a Glance

```
Maven:
  Config:     pom.xml (XML)
  Build:      mvn clean package
  Wrapper:    ./mvnw
  Deps cache: ~/.m2/repository/
  Output:     target/

Gradle:
  Config:     build.gradle (Groovy/Kotlin)
  Build:      ./gradlew build
  Wrapper:    ./gradlew
  Deps cache: ~/.gradle/caches/
  Output:     build/
```

### Commands You'll Use Most

| Task | Maven | Gradle |
|------|-------|--------|
| Clean + Build | `mvn clean package` | `./gradlew clean build` |
| Skip tests | `mvn package -DskipTests` | `./gradlew build -x test` |
| Run tests | `mvn test` | `./gradlew test` |
| Run app | `mvn spring-boot:run` | `./gradlew bootRun` |
| Dependency tree | `mvn dependency:tree` | `./gradlew dependencies` |
| Refresh deps | `mvn -U clean install` | `./gradlew --refresh-dependencies` |

### Key Takeaways

1. **Maven = Convention over Configuration** — rigid but predictable. Uses XML.
2. **Gradle = Flexibility + Performance** — more powerful, uses Groovy/Kotlin DSL.
3. **Always use the wrapper** (`./mvnw` or `./gradlew`) — ensures consistent versions.
4. **Understand the lifecycle** — know what `mvn package` or `./gradlew build` actually does under the hood.
5. **Cache dependencies in CI/CD** — saves minutes on every build.
6. **Use multi-stage Docker builds** — copy dependency files first for layer caching.
7. **`mvn dependency:tree`** and **`./gradlew dependencies`** are your debugging best friends.
8. **Both tools are production-ready** — pick what your project already uses, or Gradle for new projects that need speed.
