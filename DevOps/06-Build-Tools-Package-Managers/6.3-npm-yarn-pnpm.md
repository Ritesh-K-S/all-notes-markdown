## 1. What is a Package Manager?

A **package manager** is a tool that automates the process of **installing, updating, configuring, and removing** software packages (libraries).

### Analogy

> Imagine you're cooking dinner. You need ingredients (packages/libraries). Instead of:
> - Going to the farm for tomatoes 🍅
> - Going to the bakery for bread 🍞
> - Going to the dairy for cheese 🧀
>
> You just open a **food delivery app** (package manager), search for what you need, and it brings everything to your door in the right quantities.
>
> That's what npm/yarn/pnpm do — but for code libraries.

### What They Handle

| Task | Manual Way | With Package Manager |
|------|-----------|---------------------|
| Install a library | Download ZIP, extract, put in right folder | `npm install express` |
| Track what you installed | Write it down somewhere... | `package.json` auto-updated |
| Install exact same versions | Good luck! | `package-lock.json` |
| Update a library | Download new version, replace old | `npm update express` |
| Remove a library | Delete files, hope you got everything | `npm uninstall express` |
| Handle sub-dependencies | Manually install each one | Automatic! |
| Security scan | Manually check CVE databases | `npm audit` |

## 3. npm — Node Package Manager

**npm** is the **default** package manager that comes bundled with Node.js. When you install Node.js, you get npm for free.

### Installing Node.js & npm

```bash
# Check if Node.js and npm are installed
node -v     # v20.x.x
npm -v      # 10.x.x

# Install Node.js (which includes npm)
# Ubuntu/Debian
curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
sudo apt install -y nodejs

# macOS
brew install node

# Windows
winget install OpenJS.NodeJS.LTS

# Using nvm (Node Version Manager) — RECOMMENDED
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.7/install.sh | bash
nvm install 20
nvm use 20
```

### Why nvm is Recommended

```bash
# Install multiple Node.js versions
nvm install 18
nvm install 20
nvm install 22

# Switch between versions
nvm use 18    # Switch to Node 18
nvm use 20    # Switch to Node 20

# Set default
nvm alias default 20

# List installed versions
nvm ls

# Use version specified in .nvmrc file
echo "20" > .nvmrc
nvm use    # Reads .nvmrc
```

This is crucial because different projects may require different Node.js versions.

## 4. package.json — The Heart of Every Node.js Project

Every Node.js project has a `package.json` file. It's like Maven's `pom.xml` or Gradle's `build.gradle`.

### Creating a New Project

```bash
# Interactive creation (asks questions)
npm init

# Quick creation (accepts all defaults)
npm init -y
```

### Anatomy of package.json

```json
{
  "name": "my-web-app",
  "version": "1.0.0",
  "description": "A demo web application",
  "main": "src/index.js",
  "scripts": {
    "start": "node src/index.js",
    "dev": "nodemon src/index.js",
    "build": "webpack --mode production",
    "test": "jest",
    "test:watch": "jest --watch",
    "lint": "eslint src/",
    "lint:fix": "eslint src/ --fix",
    "format": "prettier --write ."
  },
  "keywords": ["web", "api", "demo"],
  "author": "DevOps Engineer",
  "license": "MIT",

  "dependencies": {
    "express": "^4.18.2",
    "cors": "^2.8.5",
    "dotenv": "^16.3.1",
    "mongoose": "^8.0.0"
  },

  "devDependencies": {
    "jest": "^29.7.0",
    "eslint": "^8.56.0",
    "nodemon": "^3.0.2",
    "webpack": "^5.89.0",
    "webpack-cli": "^5.1.4"
  },

  "engines": {
    "node": ">=18.0.0",
    "npm": ">=9.0.0"
  }
}
```

Let's break it down:

| Field | Purpose | Example |
|-------|---------|---------|
| `name` | Package name (must be unique on npm if publishing) | `"my-web-app"` |
| `version` | Current version (SemVer) | `"1.0.0"` |
| `main` | Entry point of the application | `"src/index.js"` |
| `scripts` | Custom commands you can run with `npm run <name>` | `"start": "node src/index.js"` |
| `dependencies` | Libraries needed to RUN the app | `express`, `mongoose` |
| `devDependencies` | Libraries needed only for DEVELOPMENT | `jest`, `eslint`, `webpack` |
| `engines` | Required Node.js/npm versions | `"node": ">=18.0.0"` |

### dependencies vs devDependencies

This is a **critical** distinction for DevOps:

| Type | When Needed | Installed in Production? | Examples |
|------|------------|-------------------------|----------|
| `dependencies` | At runtime | **Yes** | express, axios, mongoose |
| `devDependencies` | Only during development/build | **No** (ideally) | jest, eslint, webpack, typescript |

```bash
# Install a production dependency
npm install express
# Added to "dependencies" in package.json

# Install a development dependency
npm install --save-dev jest
# Added to "devDependencies" in package.json

# Install only production dependencies (for deployment!)
npm install --production
# OR
npm ci --omit=dev
```

## 5. Installing Packages

### Basic Installation Commands

```bash
# Install ALL dependencies listed in package.json
npm install
# Short form:
npm i

# Install a specific package (adds to dependencies)
npm install express

# Install a specific version
npm install express@4.18.2

# Install as dev dependency
npm install --save-dev jest
# Short form:
npm i -D jest

# Install globally (system-wide — use sparingly!)
npm install -g nodemon

# Uninstall a package
npm uninstall express
# Short form:
npm rm express

# Update all packages to latest allowed versions
npm update

# Update a specific package
npm update express

# See outdated packages
npm outdated
```

### What Happens When You Run `npm install`?

```
1. npm reads package.json
2. Checks package-lock.json for exact versions
3. Checks if packages exist in node_modules/
4. Downloads missing packages from the npm registry
5. Resolves the dependency tree
6. Installs packages into node_modules/
7. Updates package-lock.json
```

### `npm install` vs `npm ci`

This is **very important** for CI/CD:

| Command | Behavior | Use Case |
|---------|----------|----------|
| `npm install` | Reads `package.json`, may update lock file | **Development** (flexible) |
| `npm ci` | Reads `package-lock.json` strictly, deletes `node_modules` first | **CI/CD** (strict, reproducible) |

```bash
# In development (on your laptop)
npm install

# In CI/CD pipeline (always use ci!)
npm ci
```

**Why `npm ci` for CI/CD?**

1. It's **faster** (skips dependency resolution)
2. It's **deterministic** (exact versions from lock file)
3. It **fails** if `package-lock.json` is out of sync with `package.json`
4. It **always starts clean** (deletes `node_modules` first)

## 6. package-lock.json — Why It Matters

The `package-lock.json` file records the **exact version** of every dependency (and every transitive dependency) that was installed.

### Why Do We Need It?

```json
// package.json says:
"express": "^4.18.0"

// This means "any version >= 4.18.0 and < 5.0.0"
// Without a lock file:
//   Developer A installs → gets 4.18.0
//   Developer B installs → gets 4.18.2 (newer)
//   CI/CD installs       → gets 4.19.0 (even newer!)

// With package-lock.json:
//   Everyone gets exactly 4.18.0 ✅
```

### Rules for Lock Files

| Rule | Why |
|------|-----|
| **Always commit to Git** | Ensures everyone gets the same versions |
| **Never edit manually** | Let npm manage it |
| **Use `npm ci` in CI/CD** | Reads the lock file strictly |
| **Regenerate if corrupted** | Delete `node_modules` + `package-lock.json`, then `npm install` |

```bash
# If something is messed up, do a fresh install
rm -rf node_modules package-lock.json
npm install
# This regenerates a clean lock file
```

## 7. npm Scripts — Your Build System

npm scripts are **custom commands** defined in `package.json`. They're the primary way to build, test, and run Node.js projects.

### Defining Scripts

```json
{
  "scripts": {
    "start": "node src/index.js",
    "dev": "nodemon src/index.js",
    "build": "webpack --mode production",
    "build:css": "tailwindcss -i ./src/input.css -o ./dist/output.css",
    "test": "jest",
    "test:coverage": "jest --coverage",
    "lint": "eslint src/",
    "lint:fix": "eslint src/ --fix",
    "format": "prettier --write .",
    "docker:build": "docker build -t myapp .",
    "docker:run": "docker run -p 3000:3000 myapp",
    "clean": "rm -rf dist node_modules",
    "prestart": "npm run build",
    "poststart": "echo 'App started successfully!'"
  }
}
```

### Running Scripts

```bash
# Special scripts (no "run" needed)
npm start       # Runs "start" script
npm test        # Runs "test" script
npm stop        # Runs "stop" script
npm restart     # Runs "stop" then "restart" then "start"

# Custom scripts (need "run")
npm run build
npm run dev
npm run lint
npm run lint:fix
npm run test:coverage

# Run with arguments
npm test -- --verbose
npm run build -- --watch
```

### Pre & Post Scripts

npm automatically runs `pre` and `post` scripts:

```json
{
  "scripts": {
    "prebuild": "echo 'About to build...'",
    "build": "webpack --mode production",
    "postbuild": "echo 'Build complete!'",
    
    "pretest": "npm run lint",
    "test": "jest",
    "posttest": "echo 'All tests passed!'"
  }
}
```

```bash
npm run build
# Runs: prebuild → build → postbuild

npm test
# Runs: pretest (lint) → test (jest) → posttest
```

### Running Multiple Scripts

```bash
# Run scripts sequentially (one after another)
npm run lint && npm run test && npm run build

# Run scripts in parallel (at the same time)
# Using the concurrently package
npm install --save-dev concurrently
```

```json
{
  "scripts": {
    "dev": "concurrently \"npm run server\" \"npm run client\"",
    "server": "nodemon src/server.js",
    "client": "cd client && npm start"
  }
}
```

## 8. Semantic Versioning & Version Ranges

### SemVer Recap

```
MAJOR.MINOR.PATCH
  1  .  2  .  3

MAJOR = Breaking changes (your code might need updates)
MINOR = New features (backward compatible)
PATCH = Bug fixes (backward compatible)
```

### Version Ranges in package.json

| Prefix | Meaning | Example | Matches |
|--------|---------|---------|---------|
| (none) | Exact version | `"1.2.3"` | Only 1.2.3 |
| `^` | Compatible with version (default) | `"^1.2.3"` | ≥1.2.3, <2.0.0 |
| `~` | Approximately equivalent | `"~1.2.3"` | ≥1.2.3, <1.3.0 |
| `>` | Greater than | `">1.2.3"` | Any version > 1.2.3 |
| `>=` | Greater or equal | `">=1.2.3"` | Any version ≥ 1.2.3 |
| `<` | Less than | `"<2.0.0"` | Any version < 2.0.0 |
| `*` | Any version | `"*"` | Anything (dangerous!) |
| `1.x` | Any minor/patch in major 1 | `"1.x"` | ≥1.0.0, <2.0.0 |

### The Most Important: `^` (caret)

The `^` is the **default** when you `npm install`. It allows updates within the same **major** version:

```json
"express": "^4.18.0"
// Allows: 4.18.0, 4.18.1, 4.19.0, 4.99.99
// Blocks: 5.0.0 (major change = potential breaking!)
```

This is a good default because it allows bug fixes and new features while protecting against breaking changes.

### The Tilde `~`

The `~` is more restrictive — only allows **patch** updates:

```json
"express": "~4.18.0"
// Allows: 4.18.0, 4.18.1, 4.18.99
// Blocks: 4.19.0 (minor change blocked)
```

Use `~` when you want to be extra careful.

## 9. node_modules — The Heaviest Object in the Universe

```
     🌍 Earth (5.97 × 10²⁴ kg)
     ☀️ The Sun (1.99 × 10³⁰ kg)
     🕳️ Black Hole (varies)
     📁 node_modules (∞)
```

That's a joke in the developer community, but there's truth to it. A simple React project can have **1,000+ packages** in `node_modules`.

### Why Is It So Big?

```bash
# Create a simple React app
npx create-react-app my-app
cd my-app

# Check node_modules size
du -sh node_modules/
# 300MB+ for a "simple" app!

# Count packages
ls node_modules/ | wc -l
# 900+ packages!
```

Each package has its own dependencies, which have their own dependencies... it's turtles all the way down.

### Rules for node_modules

| Rule | Why |
|------|-----|
| **NEVER commit to Git** | It's too big and regeneratable |
| **Add to .gitignore** | `node_modules/` should be in every `.gitignore` |
| **Regenerate from lock file** | `npm ci` recreates it exactly |
| **Don't edit files inside it** | Changes will be lost on next install |

### .gitignore for Node.js Projects

```gitignore
# Dependencies
node_modules/

# Build output
dist/
build/
.next/

# Environment variables (NEVER commit secrets!)
.env
.env.local
.env.production

# Logs
logs/
*.log
npm-debug.log*

# OS files
.DS_Store
Thumbs.db

# IDE
.vscode/
.idea/
```

npm has a built-in security scanner that checks your dependencies against known vulnerability databases.

```bash
# Auto-fix vulnerabilities
npm audit fix

# Force-fix (may include breaking changes)

```

### In CI/CD

```yaml
# GitHub Actions — fail the build on high vulnerabilities
- name: Security audit
  run: npm audit --audit-level=high
```

## 11. npx — Run Without Installing

`npx` lets you run npm packages **without permanently installing them**. It comes bundled with npm.

```bash
# Without npx — install globally, then run
npm install -g create-react-app
create-react-app my-app

# With npx — run directly (downloads temporarily, runs, cleans up)
npx create-react-app my-app

# More examples
npx eslint .                    # Run ESLint without global install
npx prettier --write .          # Format code
npx http-server ./dist          # Quick static file server
npx kill-port 3000              # Kill process on port 3000
npx cowsay "Hello DevOps!"     # Fun! 🐄
```

### Why npx?

1. **No global pollution** — packages are temporary
2. **Always use the latest version** — downloads fresh each time
3. **Great for one-time commands** — scaffolding, generators
4. **Run project-local tools** — uses `node_modules/.bin/` automatically

## 12. .npmrc — Configuration File

The `.npmrc` file configures npm behavior. It can exist at multiple levels.

### Levels (in order of priority)

| Level | Location | Scope |
|-------|----------|-------|
| **Project** | `my-project/.npmrc` | This project only |
| **User** | `~/.npmrc` | All projects for this user |
| **Global** | `/etc/npmrc` | All users on this machine |

### Common .npmrc Settings

```ini
# Use a private registry (company Nexus/Artifactory)
registry=https://nexus.mycompany.com/repository/npm-group/

# Scoped registry (packages with @mycompany/ prefix use private registry)
@mycompany:registry=https://nexus.mycompany.com/repository/npm-private/

# Authentication token
//nexus.mycompany.com/repository/npm-private/:_authToken=${NPM_TOKEN}

# Save exact versions (no ^ prefix)
save-exact=true

# Automatically run audit on install
audit=true

# Set Node.js engine strictness
engine-strict=true

# Don't generate package-lock.json (rare, for libraries)
# package-lock=false
```

### Environment Variable for Auth (CI/CD)

```bash
# Set token as environment variable
export NPM_TOKEN=your-token-here

# .npmrc reads it with ${NPM_TOKEN}
```

```yaml
# GitHub Actions
- name: Install dependencies
  run: npm ci
  env:
    NPM_TOKEN: ${{ secrets.NPM_TOKEN }}
```

## 13. Yarn — Facebook's Alternative

**Yarn** was created by Facebook in 2016 to fix npm's problems at the time — slow installs, inconsistent dependency resolution, and lack of a lock file. (npm has since caught up on many of these issues.)

### Installing Yarn

```bash
# Via npm (ironic, but it works!)
npm install -g yarn

# Via Corepack (recommended — built into Node.js 16+)
corepack enable
corepack prepare yarn@stable --activate

# macOS
brew install yarn

# Verify
yarn -v     # 1.22.x (Classic) or 4.x (Berry)
```

### Yarn Classic (v1) Commands

```bash
# Initialize a project
yarn init
yarn init -y

# Install ALL dependencies
yarn install
# OR just:
yarn

# Add a package
yarn add express
yarn add --dev jest       # Dev dependency

# Remove a package
yarn remove express

# Update packages
yarn upgrade

# Update a specific package
yarn upgrade express

# Run scripts
yarn start                # No "run" needed for custom scripts too!
yarn build
yarn test

# Check outdated packages
yarn outdated

# See dependency tree
yarn list

# Global install
yarn global add nodemon

# Clean cache
yarn cache clean
```

### Yarn Lock File

Yarn uses `yarn.lock` instead of `package-lock.json`:

```bash
# yarn.lock is auto-generated — always commit it to Git!
# It serves the same purpose as package-lock.json
```

### Yarn vs npm Commands — Translation Table

| npm | Yarn | Purpose |
|-----|------|---------|
| `npm install` | `yarn` or `yarn install` | Install all deps |
| `npm ci` | `yarn install --frozen-lockfile` | CI install (strict) |
| `npm install express` | `yarn add express` | Add dependency |
| `npm install -D jest` | `yarn add --dev jest` | Add dev dependency |
| `npm uninstall express` | `yarn remove express` | Remove dependency |
| `npm update` | `yarn upgrade` | Update all deps |
| `npm run build` | `yarn build` | Run script |
| `npm test` | `yarn test` | Run tests |
| `npm audit` | `yarn audit` | Security scan |
| `npx create-react-app` | `yarn create react-app` | Run package |
| `npm cache clean --force` | `yarn cache clean` | Clear cache |
| `npm ls` | `yarn list` | Dependency tree |

## 14. Yarn Berry (v2+) — Modern Yarn

Yarn v2+ (codenamed "Berry") is a **complete rewrite** with some radical changes.

### Key Innovation: Plug'n'Play (PnP)

Instead of a giant `node_modules` folder, Yarn Berry stores dependencies as **compressed ZIP files** and uses a `.pnp.cjs` file to tell Node.js where to find them.

```
Traditional (npm/yarn v1):
  node_modules/
  ├── express/          ← 50+ files
  ├── lodash/           ← 600+ files
  ├── react/            ← 30+ files
  └── ... (thousands of folders and files)

Yarn Berry PnP:
  .yarn/cache/
  ├── express-npm-4.18.2-hash.zip      ← One file!
  ├── lodash-npm-4.17.21-hash.zip      ← One file!
  └── react-npm-18.2.0-hash.zip        ← One file!
  .pnp.cjs                              ← Resolution map
```

### Benefits

| Benefit | Explanation |
|---------|-------------|
| **Faster installs** | No need to extract thousands of files |
| **Less disk space** | ZIP files are smaller |
| **No phantom dependencies** | Can't accidentally import undeclared packages |
| **Zero-installs** | Commit `.yarn/cache/` to Git → no install step in CI! |

### Setting Up Yarn Berry

```bash
# Enable Corepack
corepack enable

# Create a new project with Yarn Berry
yarn init -2

# Or migrate an existing project
yarn set version berry
yarn install
```

### Zero-Installs

With Yarn Berry, you can commit the **entire cache** to Git, so CI/CD doesn't need to run `yarn install` at all:

```yaml
# .gitattributes
.yarn/cache/** binary         # Treat as binary (no diff)
.yarn/releases/** binary

# .yarnrc.yml
enableGlobalCache: false      # Store cache in project
```

```yaml
# CI/CD — no install step needed!
steps:
  - uses: actions/checkout@v4
  - run: yarn build    # Dependencies are already in the repo!
```

## 15. pnpm — The Performance Champion

**pnpm** (Performant npm) takes a different approach to dependency storage. Instead of copying packages into each project's `node_modules`, it uses a **global store** with **hard links**.

### How pnpm Is Different

```
npm/yarn (traditional):
  project-a/node_modules/express/  ← Full copy (50MB)
  project-b/node_modules/express/  ← Another full copy (50MB)
  project-c/node_modules/express/  ← Yet another copy (50MB)
  Total: 150MB for ONE package across 3 projects

pnpm:
  ~/.pnpm-store/express@4.18.2/   ← Single copy (50MB)
  project-a/node_modules/express  → hard link to store
  project-b/node_modules/express  → hard link to store
  project-c/node_modules/express  → hard link to store
  Total: 50MB for ONE package across 3 projects
```

### Installing pnpm

```bash
# Via npm
npm install -g pnpm

# Via Corepack (Node.js 16+)
corepack enable
corepack prepare pnpm@latest --activate

# macOS

iwr https://get.pnpm.io/install.ps1 -useb | iex

```


### pnpm Lock File

pnpm uses `pnpm-lock.yaml`:

```yaml
# pnpm-lock.yaml (auto-generated — commit to Git!)
lockfileVersion: '6.0'
dependencies:
  express:
    specifier: ^4.18.2
    version: 4.18.2
packages:
  /express@4.18.2:
    resolution: {integrity: sha512-xxx...}
    dependencies:
      accepts: 1.3.8
      # ... more transitive deps
```

### Strict node_modules Structure

pnpm creates a **strict** `node_modules` structure that prevents phantom dependencies:

```
# npm/yarn — flat structure (can accidentally use undeclared deps)
node_modules/
├── express/
├── accepts/         ← You can import this even though YOU didn't declare it!
├── body-parser/     ← Same — "phantom dependency"
└── ...

# pnpm — strict structure (only your declared deps are accessible)
node_modules/
├── .pnpm/                    ← All packages live here
│   ├── express@4.18.2/
│   │   └── node_modules/
│   │       ├── express/      ← Actual code
│   │       ├── accepts/      ← express's dependency (not yours!)
│   │       └── body-parser/
│   └── ...
├── express -> .pnpm/express@4.18.2/node_modules/express   ← Symlink!
└── (only YOUR declared dependencies appear here)
```

If you try to import a package you didn't declare in `package.json`, pnpm will throw an error. This catches bugs early!

## 16. npm vs yarn vs pnpm — Head-to-Head

### Feature Comparison

| Feature | npm | Yarn Classic (v1) | Yarn Berry (v2+) | pnpm |
|---------|-----|-------------------|-------------------|------|
| **Bundled with Node** | ✅ Yes | ❌ No | ❌ No | ❌ No |
| **Lock file** | `package-lock.json` | `yarn.lock` | `yarn.lock` | `pnpm-lock.yaml` |
| **Speed** | Good | Good | Better | **Best** |
| **Disk space** | High | High | Low (PnP) | **Lowest** |
| **Strict deps** | ❌ Hoisted (loose) | ❌ Hoisted | ✅ PnP strict | ✅ Strict |
| **Workspaces** | ✅ v7+ | ✅ | ✅ | ✅ |
| **Monorepo support** | Basic | Basic | Good | **Best** |
| **Zero-installs** | ❌ | ❌ | ✅ | ❌ |
| **Parallel installs** | ✅ | ✅ | ✅ | ✅ |
| **CI install cmd** | `npm ci` | `yarn --frozen-lockfile` | `yarn --immutable` | `pnpm install --frozen-lockfile` |
| **Offline support** | Basic | Good | Excellent | Good |
| **Plugin system** | ❌ | ❌ | ✅ | ❌ |

### Speed Comparison (Rough Estimates)

```
Fresh install (no cache):
  npm:   45 seconds
  yarn:  35 seconds
  pnpm:  25 seconds  ← Winner!

Cached install (with lock file):
  npm:   20 seconds
  yarn:  15 seconds
  pnpm:   8 seconds  ← Winner!

CI install (npm ci / frozen-lockfile):
  npm:   25 seconds
  yarn:  18 seconds
  pnpm:  10 seconds  ← Winner!
```

### Which One Should You Use?

| Situation | Recommendation |
|-----------|---------------|
| **Default / simple projects** | **npm** — comes with Node, zero setup |
| **Existing project** | Use whatever it already uses (check lock file!) |
| **Performance matters** | **pnpm** — fastest, least disk space |
| **Monorepo** | **pnpm** or **Yarn Berry** |
| **Zero-install CI** | **Yarn Berry** (PnP) |
| **Strict dependency checking** | **pnpm** |
| **Team familiarity** | Use what the team knows |

### How to Tell Which Package Manager a Project Uses

Look at the lock file:

| Lock File | Package Manager |
|-----------|----------------|
| `package-lock.json` | npm |
| `yarn.lock` | Yarn |
| `pnpm-lock.yaml` | pnpm |

## 17. Workspaces & Monorepos

**Workspaces** let you manage **multiple packages** in a **single repository** (monorepo).

### Structure

```
my-monorepo/
├── package.json            ← Root package.json (workspace config)
├── packages/
│   ├── shared/
│   │   ├── package.json    ← @myapp/shared
│   │   └── src/
│   ├── frontend/
│   │   ├── package.json    ← @myapp/frontend
│   │   └── src/
│   ├── backend/
│   │   ├── package.json    ← @myapp/backend
│   │   └── src/
│   └── cli/
│       ├── package.json    ← @myapp/cli
│       └── src/
└── (lock file)
```

### npm Workspaces

```json
// Root package.json
{
  "name": "my-monorepo",
  "private": true,
  "workspaces": [
    "packages/*"
  ]
}
```

```bash
# Install all workspace dependencies
npm install

# Run a script in a specific workspace
npm run build -w packages/frontend
npm test -w packages/backend

# Run a script in ALL workspaces
npm run build --workspaces

# Add a dependency to a specific workspace
npm install express -w packages/backend

# Link workspaces to each other
# In packages/backend/package.json:
# "dependencies": { "@myapp/shared": "*" }
# npm automatically links to the local workspace!
```

### pnpm Workspaces

```yaml
# pnpm-workspace.yaml (in root)
packages:
  - 'packages/*'
```

```bash
# Install all
pnpm install

# Run in specific workspace
pnpm --filter @myapp/frontend build
pnpm --filter @myapp/backend test

# Run in all workspaces
pnpm -r build           # -r = recursive

# Add dependency to a workspace
pnpm --filter @myapp/backend add express

# Run in parallel
pnpm -r --parallel build
```

### Yarn Workspaces

```json
// Root package.json
{
  "private": true,
  "workspaces": [
    "packages/*"
  ]
}
```

```bash
yarn workspace @myapp/frontend build
yarn workspace @myapp/backend test
yarn workspaces run build    # All workspaces
```

## 18. CI/CD Best Practices

### 1. Always Use Lock Files

```yaml
# GitHub Actions — npm
- run: npm ci    # NOT npm install

# GitHub Actions — yarn
- run: yarn install --frozen-lockfile    # v1
- run: yarn install --immutable          # Berry

# GitHub Actions — pnpm
- run: pnpm install --frozen-lockfile
```

### 2. Cache Dependencies

```yaml
# npm caching
- name: Cache node modules
  uses: actions/cache@v4
  with:
    path: ~/.npm
    key: ${{ runner.os }}-npm-${{ hashFiles('**/package-lock.json') }}
    restore-keys: ${{ runner.os }}-npm-

- run: npm ci
```

```yaml
# pnpm caching
- uses: pnpm/action-setup@v2
  with:
    version: 8

- name: Get pnpm store directory
  id: pnpm-cache
  run: echo "STORE_PATH=$(pnpm store path)" >> $GITHUB_OUTPUT

- uses: actions/cache@v4
  with:
    path: ${{ steps.pnpm-cache.outputs.STORE_PATH }}
    key: ${{ runner.os }}-pnpm-${{ hashFiles('**/pnpm-lock.yaml') }}

- run: pnpm install --frozen-lockfile
```

### 3. Install Only Production Dependencies for Deployment

```bash
# After building, install only what's needed to RUN
npm ci --omit=dev
pnpm install --prod
yarn install --production
```

### 4. Complete CI/CD Pipeline Example

```yaml
name: Node.js CI/CD

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  build-and-test:
    runs-on: ubuntu-latest

    strategy:
      matrix:
        node-version: [18, 20, 22]    # Test on multiple Node versions

    steps:
      - uses: actions/checkout@v4

      - name: Use Node.js ${{ matrix.node-version }}
        uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
          cache: 'npm'    # Auto-caches ~/.npm

      - name: Install dependencies
        run: npm ci

      - name: Lint
        run: npm run lint

      - name: Test
        run: npm test

      - name: Build
        run: npm run build

      - name: Security audit
        run: npm audit --audit-level=high

  docker:
    needs: build-and-test
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'

    steps:
      - uses: actions/checkout@v4

      - name: Build Docker image
        run: docker build -t myapp:${{ github.sha }} .

      - name: Push to registry
        run: |
          echo "${{ secrets.DOCKER_PASSWORD }}" | docker login -u "${{ secrets.DOCKER_USERNAME }}" --password-stdin
          docker tag myapp:${{ github.sha }} myregistry.io/myapp:${{ github.sha }}
          docker push myregistry.io/myapp:${{ github.sha }}
```

## 19. Docker Best Practices for Node.js

### Good Dockerfile

```dockerfile
# Use specific version (not 'latest'!)
FROM node:20-alpine AS builder

# Create non-root user
RUN addgroup -S appgroup && adduser -S appuser -G appgroup

WORKDIR /app

# Copy dependency files FIRST (layer caching!)
COPY package.json package-lock.json ./

# Install ALL dependencies (including devDependencies for building)
RUN npm ci

# Copy source code
COPY . .

# Build the application
RUN npm run build

# --- Production Stage ---
FROM node:20-alpine

RUN addgroup -S appgroup && adduser -S appuser -G appgroup

WORKDIR /app

# Copy dependency files
COPY package.json package-lock.json ./

# Install ONLY production dependencies
RUN npm ci --omit=dev

# Copy built files from builder stage
COPY --from=builder /app/dist ./dist

# Switch to non-root user
USER appuser

EXPOSE 3000

# Use node directly (not npm start — better signal handling)
CMD ["node", "dist/index.js"]
```

### .dockerignore

```dockerignore
node_modules
npm-debug.log
.git
.gitignore
.env
.env.*
dist
build
coverage
.nyc_output
*.md
.vscode
.idea
Dockerfile
docker-compose*.yml
```

### Common Mistakes

```dockerfile
# ❌ BAD — copies node_modules from host (might be different OS!)
COPY . .
# Then uses the host's node_modules inside Linux container = crashes!

# ✅ GOOD — always install fresh inside the container
COPY package.json package-lock.json ./
RUN npm ci
COPY . .

# ❌ BAD — using npm start (extra process, bad signal handling)
CMD ["npm", "start"]

# ✅ GOOD — run node directly
CMD ["node", "src/index.js"]

# ❌ BAD — running as root
USER root

# ✅ GOOD — running as non-root user
USER appuser
```

## 21. Troubleshooting Common Issues

### Issue 1: "Module not found" / "Cannot find module"

```
Error: Cannot find module 'express'
```

```bash
# Fix: Install the missing dependency
npm install express

# If already in package.json, delete and reinstall
rm -rf node_modules package-lock.json
npm install

# Check if the package is actually installed
npm ls express
```

### Issue 2: "ERESOLVE unable to resolve dependency tree"

```
npm ERR! ERESOLVE unable to resolve dependency tree
npm ERR! peer dependency conflict
```

```bash
# Option 1: Use --legacy-peer-deps
npm install --legacy-peer-deps

# Option 2: Use --force
npm install --force

# Option 3: Fix the actual conflict
npm ls conflicting-package    # See the tree
# Then update the conflicting packages
```

### Issue 3: "Permission denied" (EACCES)

```bash
# NEVER use sudo with npm!

# Fix: Use nvm (recommended)
nvm install 20

# Fix: Change npm's default directory
mkdir ~/.npm-global
npm config set prefix '~/.npm-global'
export PATH=~/.npm-global/bin:$PATH
```

### Issue 4: "ENOSPC: no space left on device"

```bash
# Clear npm cache
npm cache clean --force

# Clear pnpm store
pnpm store prune

# Clear yarn cache
yarn cache clean

# Check disk space
df -h
```

### Issue 5: Lock File Conflicts After Merge

```bash
# Don't try to manually merge lock files!

# Accept either version, then regenerate:
git checkout --theirs package-lock.json    # Or --ours
npm install                                 # Regenerates lock file
git add package-lock.json
git commit -m "Regenerate lock file after merge"
```

### Issue 6: Build Fails in CI but Works Locally

| Cause | Fix |
|-------|-----|
| Different Node version | Use `.nvmrc` + `setup-node` with exact version |
| `npm install` vs `npm ci` | Always use `npm ci` in CI |
| Missing env variables | Set them in CI secrets |
| OS differences | Use Docker for consistent builds |
| Phantom dependency | Switch to pnpm (strict mode) |
| Stale cache | Clear CI cache and rebuild |

```yaml
# Pin exact Node.js version in CI
- uses: actions/setup-node@v4
  with:
    node-version-file: '.nvmrc'    # Reads version from .nvmrc
    cache: 'npm'
```

## 22. Summary

### Quick Reference — Package Manager Commands

| Task | npm | yarn | pnpm |
|------|-----|------|------|
| **Install all** | `npm install` | `yarn` | `pnpm install` |
| **CI install** | `npm ci` | `yarn --frozen-lockfile` | `pnpm install --frozen-lockfile` |
| **Add package** | `npm install pkg` | `yarn add pkg` | `pnpm add pkg` |
| **Add dev dep** | `npm i -D pkg` | `yarn add -D pkg` | `pnpm add -D pkg` |
| **Remove** | `npm uninstall pkg` | `yarn remove pkg` | `pnpm remove pkg` |
| **Update all** | `npm update` | `yarn upgrade` | `pnpm update` |
| **Run script** | `npm run build` | `yarn build` | `pnpm build` |
| **Audit** | `npm audit` | `yarn audit` | `pnpm audit` |
| **Outdated** | `npm outdated` | `yarn outdated` | `pnpm outdated` |
| **Dep tree** | `npm ls` | `yarn list` | `pnpm list` |
| **Lock file** | `package-lock.json` | `yarn.lock` | `pnpm-lock.yaml` |
| **Cache clean** | `npm cache clean --force` | `yarn cache clean` | `pnpm store prune` |

### Key Takeaways

1. **All three use the same npm registry** — they're just different tools to download and manage packages.
2. **Always use lock files** — commit them to Git, use `ci`/`--frozen-lockfile` in CI/CD.
3. **`npm ci` > `npm install`** in CI/CD — faster, deterministic, strict.
4. **Never commit `node_modules/`** — it's regeneratable from the lock file.
5. **Use `devDependencies` correctly** — development tools don't belong in production.
6. **pnpm is the fastest** and uses the least disk space — great for monorepos.
7. **Yarn Berry (PnP) is innovative** — eliminates `node_modules` entirely.
8. **npm is the safe default** — comes with Node.js, everyone knows it.
9. **Cache dependencies in CI/CD** — saves minutes on every build.
10. **Docker: copy `package.json` first, then source code** — layer caching saves rebuild time.
