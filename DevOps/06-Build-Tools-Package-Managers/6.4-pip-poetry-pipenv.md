## 1. Why Do We Need Python Package Managers?

Imagine you're building a house. You don't make your own bricks, nails, and glass from scratch — you **buy them** from a store. Similarly, when you write Python code, you don't write everything from scratch. You use **packages** (libraries) that other people have already written.

A **package manager** is like that store — it helps you:

- **Find** packages (searching a catalog)
- **Install** packages (downloading and setting them up)
- **Update** packages (getting newer versions)
- **Remove** packages (uninstalling what you don't need)
- **Track** which packages your project needs (so others can recreate the same setup)

### The Problem Without a Package Manager

```
You: "My code works on my machine!"
Your teammate: "It doesn't work on mine..."
You: "What version of requests do you have?"
Your teammate: "Ummm... I don't know?"
```

This is the classic **"works on my machine"** problem. Package managers solve this by keeping a **record of every package and its exact version** your project needs.

## 2. pip — The Default Python Package Manager

### What is pip?

**pip** stands for **"pip Installs Packages"** (yes, it's a recursive acronym!). It comes **pre-installed** with Python (version 3.4+). Think of it as the **App Store for Python**.

pip downloads packages from **PyPI** (Python Package Index) — a massive online repository with over 500,000+ packages.

### Check if pip is Installed

```bash
# Check pip version
pip --version

# Output example:
# pip 23.3.1 from /usr/lib/python3/dist-packages/pip (python 3.11)
```

```bash
# If you have both Python 2 and Python 3
pip3 --version
```

### How pip Works — The Big Picture

```
You run: pip install requests
         │
         ▼
pip contacts PyPI (pypi.org)
         │
         ▼
Downloads the "requests" package
         │
         ▼
Installs it into your Python's site-packages directory
         │
         ▼
Now you can do: import requests
```

## 3. Virtual Environments — The Foundation

Before we go deep into pip, we **must** understand virtual environments. This is one of the most important concepts.

### The Problem

Imagine two projects on the same machine:

```
Project A needs: Django 3.2
Project B needs: Django 4.2
```

If you install Django globally, you can only have **one version**. One of your projects will break. 😢

### The Solution — Virtual Environments

A **virtual environment** is like a **separate room** for each project. Each room has its own set of packages, completely isolated from other rooms.

```
Your Machine
├── Project A's Virtual Env
│   └── Django 3.2
│   └── requests 2.28
│
├── Project B's Virtual Env
│   └── Django 4.2
│   └── requests 2.31
│
└── Global Python (base)
    └── pip (always available)
```

### Creating and Using Virtual Environments

```bash
# Step 1: Create a virtual environment named "venv"
python3 -m venv venv
# What this does:
# - python3 -m venv  → Run the venv module
# - venv             → Name of the folder to create (you can name it anything)
```

```bash
# Step 2: Activate the virtual environment

# On Linux/macOS:
source venv/bin/activate

# On Windows (Command Prompt):
venv\Scripts\activate

# On Windows (PowerShell):
venv\Scripts\Activate.ps1
```

```bash
# You'll see (venv) at the start of your terminal prompt:
(venv) user@machine:~/myproject$

# This means you're inside the virtual environment!
```

```bash
# Step 3: Install packages (they go into THIS virtual environment only)
pip install django==4.2

# Step 4: When you're done, deactivate
deactivate
```

### Why This Matters for DevOps

As a DevOps engineer, you'll:
- Build Docker images that need virtual environments
- Set up CI/CD pipelines that install Python dependencies
- Manage deployment scripts that require specific package versions

## 4. pip Deep Dive — Everything You Need

### Installing Packages

```bash
# Install the latest version
pip install requests
# What happens: Downloads the newest version of "requests" from PyPI

# Install a specific version
pip install requests==2.31.0
# What happens: Downloads exactly version 2.31.0

# Install minimum version
pip install requests>=2.28.0
# What happens: Downloads version 2.28.0 or any newer version

# Install a version range
pip install "requests>=2.28.0,<3.0.0"
# What happens: Downloads any version from 2.28.0 up to (but not including) 3.0.0

# Install multiple packages at once
pip install requests flask django
# What happens: Downloads and installs all three packages
```

### Upgrading Packages

```bash
# Upgrade a package to the latest version
pip install --upgrade requests
# Short form:
pip install -U requests

# Upgrade pip itself (do this regularly!)
pip install --upgrade pip
```

### Uninstalling Packages

```bash
# Remove a package
pip uninstall requests
# It will ask for confirmation: Proceed (Y/n)?

# Remove without confirmation
pip uninstall -y requests
```

### Listing Installed Packages

```bash
# See all installed packages
pip list

# Output:
# Package    Version
# ---------- -------
# requests   2.31.0
# flask      3.0.0
# pip        23.3.1

# See outdated packages
pip list --outdated

# Output:
# Package    Version   Latest    Type
# ---------- --------- --------- -----
# requests   2.28.0    2.31.0    wheel
```

### Getting Package Info

```bash
# Show detailed info about a package
pip show requests

# Output:
# Name: requests
# Version: 2.31.0
# Summary: Python HTTP for Humans.
# Home-page: https://requests.readthedocs.io
# Author: Kenneth Reitz
# License: Apache 2.0
# Location: /home/user/venv/lib/python3.11/site-packages
# Requires: certifi, charset-normalizer, idna, urllib3
# Required-by: docker, kubernetes
```

### The requirements.txt File — The Heart of pip

This is the **most important file** for pip-based projects. It's a simple text file that lists all packages your project needs.

#### Creating requirements.txt

```bash
# Method 1: Freeze current environment (captures EVERYTHING installed)
pip freeze > requirements.txt

# This creates a file like:
# certifi==2023.7.22
# charset-normalizer==3.3.2
# idna==3.6
# requests==2.31.0
# urllib3==2.1.0
```

```bash
# Method 2: Write it manually (recommended for clean projects)
cat > requirements.txt << 'EOF'
# Web framework
flask==3.0.0

# HTTP library
requests>=2.28.0,<3.0.0

# Database
psycopg2-binary==2.9.9

# Environment variables
python-dotenv==1.0.0
EOF
```

#### Installing from requirements.txt

```bash
# Install all packages listed in requirements.txt
pip install -r requirements.txt
# What happens:
# - pip reads each line of the file
# - Downloads and installs each package with the specified version
# - This is how you recreate the same environment on any machine!
```

#### Why This Matters in DevOps

In a **Dockerfile**, you'll see this pattern everywhere:

```dockerfile
FROM python:3.11-slim

WORKDIR /app

# Copy requirements first (for Docker layer caching)
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Then copy the rest of the code
COPY . .

CMD ["python", "app.py"]
```

In **CI/CD pipelines**:

```yaml
# .github/workflows/test.yml
steps:
  - uses: actions/checkout@v4
  - uses: actions/setup-python@v5
    with:
      python-version: '3.11'
  - run: pip install -r requirements.txt
  - run: pytest
```

### The Problem with pip + requirements.txt

While pip is great, it has some limitations:

| Problem | Description |
|---------|-------------|
| **No lock file by default** | `pip freeze` captures everything, including sub-dependencies. Hard to tell what YOU installed vs. what was pulled in automatically |
| **No dependency resolution** | pip can sometimes install conflicting versions |
| **Manual virtual env management** | You have to create and activate venvs yourself |
| **No separation of dev/prod** | All dependencies go in one file. Your testing tools shouldn't go to production! |

This is where **Pipenv** and **Poetry** come in to save the day! 🎉

## 5. Pipenv — pip + virtualenv Combined

### What is Pipenv?

**Pipenv** = **pip** + **virtualenv** rolled into one tool. It was created to solve the problems we just discussed. Think of it as pip with superpowers.

Pipenv gives you:
- **Automatic virtual environment management** (no more `python -m venv`)
- **Pipfile** instead of `requirements.txt` (cleaner, more readable)
- **Pipfile.lock** for deterministic builds (exact same versions everywhere)
- **Separation of dev and production dependencies**

### Installing Pipenv

```bash
# Install Pipenv globally
pip install pipenv

# Verify installation
pipenv --version
# Output: pipenv, version 2023.12.1
```

### Starting a New Project with Pipenv

```bash
# Create a project directory
mkdir my-project && cd my-project

# Install a package (Pipenv creates the virtual env automatically!)
pipenv install requests
```

**What just happened:**
1. Pipenv checked if a virtual environment exists → No? Created one automatically!
2. Created a `Pipfile` in your project
3. Installed `requests` in the virtual environment
4. Created a `Pipfile.lock` with exact versions

### Understanding the Pipfile

```toml
# Pipfile — this is like a smarter requirements.txt

[[source]]
url = "https://pypi.org/simple"       # Where to download packages from
verify_ssl = true                      # Verify SSL certificates
name = "pypi"                          # Name of the source

[packages]                             # PRODUCTION dependencies
requests = "*"                         # * means "any version"
flask = "==3.0.0"                      # Exact version
psycopg2-binary = ">=2.9"             # Minimum version

[dev-packages]                         # DEVELOPMENT-ONLY dependencies
pytest = "*"                           # Testing framework
black = "*"                            # Code formatter
flake8 = "*"                           # Linter

[requires]
python_version = "3.11"                # Python version required
```

**See the magic?** Dev and production dependencies are **separated**! Your testing tools won't end up in your production container.

### Understanding Pipfile.lock

The `Pipfile.lock` is an auto-generated file (you don't edit it manually). It contains:

```json
{
    "default": {
        "requests": {
            "version": "==2.31.0",
            "hashes": [
                "sha256:942c5a758f98d790eaed1a29cb6eefc7f0edf3..."
            ]
        },
        "urllib3": {
            "version": "==2.1.0",
            "hashes": [
                "sha256:55901e917a5896a349ff771be919f8bd..."
            ]
        }
    }
}
```

**Why is this important?**
- **Exact versions** — Every machine gets the SAME versions
- **Hash verification** — The hashes ensure the downloaded package hasn't been tampered with (security!)
- **Deterministic builds** — "Works on my machine" is no longer a problem

### Common Pipenv Commands

```bash
# Install a production dependency
pipenv install flask
# This adds flask to [packages] in Pipfile

# Install a development dependency
pipenv install --dev pytest
# This adds pytest to [dev-packages] in Pipfile

# Install ALL dependencies from Pipfile
pipenv install
# Installs everything in [packages]

# Install ALL dependencies INCLUDING dev
pipenv install --dev
# Installs everything in [packages] AND [dev-packages]

# Install from Pipfile.lock (for deployment — exact versions)
pipenv install --deploy
# This FAILS if Pipfile.lock is out of date (safety check!)

# Uninstall a package
pipenv uninstall requests

# Activate the virtual environment shell
pipenv shell
# Now you're inside the venv — your prompt changes

# Exit the virtual environment
exit

# Run a command inside the virtual env WITHOUT activating it
pipenv run python app.py
pipenv run pytest

# See where the virtual environment lives
pipenv --venv
# Output: /home/user/.local/share/virtualenvs/my-project-abc123

# See the dependency graph
pipenv graph
# Output:
# flask==3.0.0
#   - blinker [required: >=1.6.2, installed: 1.7.0]
#   - click [required: >=8.1.3, installed: 8.1.7]
#   - itsdangerous [required: >=2.1.2, installed: 2.1.2]
#   - Jinja2 [required: >=3.1.2, installed: 3.1.2]
#   - Werkzeug [required: >=3.0.0, installed: 3.0.1]

# Generate a requirements.txt from Pipfile.lock (for compatibility)
pipenv requirements > requirements.txt

# Check for security vulnerabilities in your dependencies
pipenv check
```

### Using Pipenv in Docker

```dockerfile
FROM python:3.11-slim

# Install Pipenv
RUN pip install pipenv

WORKDIR /app

# Copy Pipfile and Pipfile.lock
COPY Pipfile Pipfile.lock ./

# Install dependencies (system-wide inside the container, no need for venv)
RUN pipenv install --system --deploy
# --system   → Install into system Python (skip creating a venv inside Docker)
# --deploy   → Fail if Pipfile.lock is outdated

COPY . .

CMD ["python", "app.py"]
```

## 6. Poetry — The Modern Python Packaging Tool

### What is Poetry?

**Poetry** is the newest and most powerful Python package manager. It does everything Pipenv does, **plus** it can build and publish packages to PyPI.

Think of it this way:
- **pip** = Install packages
- **Pipenv** = Install packages + manage virtual environments
- **Poetry** = Install packages + manage virtual environments + build & publish your own packages

### Installing Poetry

```bash
# Recommended: Install via the official installer (Linux/macOS/WSL)
curl -sSL https://install.python-poetry.org | python3 -

# On Windows (PowerShell):
(Invoke-WebRequest -Uri https://install.python-poetry.org -UseBasicParsing).Content | python -

# Verify installation
poetry --version
# Output: Poetry (version 1.7.1)
```

```bash
# Add Poetry to your PATH (if not already)
# Linux/macOS:
export PATH="$HOME/.local/bin:$PATH"

# Add this to your ~/.bashrc or ~/.zshrc to make it permanent
```

### Starting a New Project with Poetry

```bash
# Option 1: Create a brand new project
poetry new my-awesome-project

# This creates:
# my-awesome-project/
# ├── pyproject.toml          ← The main configuration file
# ├── README.md
# ├── my_awesome_project/     ← Your source code
# │   └── __init__.py
# └── tests/                  ← Test directory
#     └── __init__.py
```

```bash
# Option 2: Initialize Poetry in an existing project
cd my-existing-project
poetry init
# This walks you through an interactive wizard to create pyproject.toml
```

### Understanding pyproject.toml

This is the **single configuration file** for your entire project. It replaces `setup.py`, `setup.cfg`, `requirements.txt`, and `Pipfile` — all in one!

```toml
[tool.poetry]
name = "my-awesome-project"
version = "0.1.0"
description = "A cool DevOps automation tool"
authors = ["Your Name <you@example.com>"]
readme = "README.md"
license = "MIT"

[tool.poetry.dependencies]
python = "^3.11"          # Requires Python 3.11 or higher (but < 4.0)
requests = "^2.31.0"      # 2.31.0 or higher (but < 3.0.0)
flask = "^3.0.0"          # 3.0.0 or higher (but < 4.0.0)
pydantic = ">=2.0,<3.0"   # Between 2.0 and 3.0

[tool.poetry.group.dev.dependencies]
pytest = "^7.4.0"         # Development-only dependency
black = "^23.0"           # Code formatter
mypy = "^1.7"             # Type checker
ruff = "^0.1.0"           # Linter

[build-system]
requires = ["poetry-core"]
build-backend = "poetry.core.masonry.api"
```

### Understanding the Caret (^) Version Syntax

Poetry uses **caret (^)** syntax by default. Let's decode it:

```
^2.31.0 means >=2.31.0 and <3.0.0
│
├── ✅ 2.31.0 (exact match)
├── ✅ 2.31.1 (patch update)
├── ✅ 2.32.0 (minor update)
└── ❌ 3.0.0  (major update — might break things!)

^0.9.0 means >=0.9.0 and <0.10.0
│
└── For versions starting with 0, it's more conservative
    because 0.x versions are considered unstable

^3.0 means >=3.0.0 and <4.0.0
```

### The poetry.lock File

Just like Pipenv's `Pipfile.lock`, Poetry creates a `poetry.lock` file that pins **exact versions** of everything.

```bash
# Always commit poetry.lock to version control!
git add pyproject.toml poetry.lock
git commit -m "Add project dependencies"
```

**Rule of thumb:**
- **Applications** (web apps, scripts, services) → Commit `poetry.lock` ✅
- **Libraries** (packages others will install) → Don't commit `poetry.lock` ❌

### Common Poetry Commands

```bash
# ─── Managing Dependencies ───

# Add a production dependency
poetry add requests
# What happens: Adds to [tool.poetry.dependencies], installs it, updates lock file

# Add a specific version
poetry add requests@2.31.0
# or
poetry add "requests==2.31.0"

# Add a dev dependency
poetry add --group dev pytest
# What happens: Adds to [tool.poetry.group.dev.dependencies]

# Remove a dependency
poetry remove requests

# Update all dependencies (within version constraints)
poetry update

# Update a specific package
poetry update requests

# Show all installed packages
poetry show

# Show as a tree (dependency graph)
poetry show --tree
# Output:
# flask 3.0.0
# ├── blinker >=1.6.2
# ├── click >=8.1.3
# ├── itsdangerous >=2.1.2
# ├── jinja2 >=3.1.2
# │   └── markupsafe >=2.0
# └── werkzeug >=3.0.0
#     └── markupsafe >=2.1.1

# Show outdated packages
poetry show --outdated
```

```bash
# ─── Virtual Environment Management ───

# Install all dependencies (from poetry.lock)
poetry install

# Install without dev dependencies (for production)
poetry install --without dev

# Activate the virtual environment
poetry shell

# Run a command inside the virtual environment
poetry run python app.py
poetry run pytest
poetry run flask run

# See virtual environment info
poetry env info

# List all virtual environments for this project
poetry env list

# Delete a virtual environment
poetry env remove python3.11
```

```bash
# ─── Building & Publishing ───

# Build your package (creates .tar.gz and .whl files)
poetry build
# Output:
# Building my-awesome-project (0.1.0)
#   - Building sdist
#   - Built my_awesome_project-0.1.0.tar.gz
#   - Building wheel
#   - Built my_awesome_project-0.1.0-py3-none-any.whl

# Publish to PyPI
poetry publish

# Build and publish in one step
poetry publish --build
```

### Using Poetry in Docker

```dockerfile
FROM python:3.11-slim

# Install Poetry
RUN pip install poetry

WORKDIR /app

# Copy dependency files
COPY pyproject.toml poetry.lock ./

# Configure Poetry:
# - Don't create a virtual environment inside Docker (unnecessary)
# - Install only production dependencies
RUN poetry config virtualenvs.create false \
    && poetry install --without dev --no-interaction --no-ansi

COPY . .

CMD ["python", "app.py"]
```

**Optimized version with multi-stage build:**

```dockerfile
# Stage 1: Install dependencies
FROM python:3.11-slim as builder

RUN pip install poetry

WORKDIR /app
COPY pyproject.toml poetry.lock ./

# Export to requirements.txt (avoid installing Poetry in final image)
RUN poetry export -f requirements.txt --output requirements.txt --without-hashes

# Stage 2: Final slim image
FROM python:3.11-slim

WORKDIR /app
COPY --from=builder /app/requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

CMD ["python", "app.py"]
```

### Using Poetry in CI/CD (GitHub Actions)

```yaml
# .github/workflows/test.yml
name: Tests

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'

      - name: Install Poetry
        run: |
          curl -sSL https://install.python-poetry.org | python3 -
          echo "$HOME/.local/bin" >> $GITHUB_PATH

      - name: Cache Poetry dependencies
        uses: actions/cache@v4
        with:
          path: ~/.cache/pypoetry
          key: ${{ runner.os }}-poetry-${{ hashFiles('**/poetry.lock') }}

      - name: Install dependencies
        run: poetry install

      - name: Run tests
        run: poetry run pytest

      - name: Run linter
        run: poetry run ruff check .
```

## 7. pip vs Pipenv vs Poetry — Comparison

### Feature Comparison Table

| Feature | pip | Pipenv | Poetry |
|---------|-----|--------|--------|
| **Install packages** | ✅ | ✅ | ✅ |
| **Virtual env management** | ❌ (manual) | ✅ (automatic) | ✅ (automatic) |
| **Lock file** | ❌ | ✅ (Pipfile.lock) | ✅ (poetry.lock) |
| **Dev/Prod separation** | ❌ | ✅ | ✅ |
| **Dependency resolution** | Basic | Better | Best |
| **Build & publish packages** | ❌ | ❌ | ✅ |
| **Config file** | requirements.txt | Pipfile | pyproject.toml |
| **Speed** | Fast | Slow | Fast |
| **PEP 621 compliant** | N/A | ❌ | ✅ |
| **Actively maintained** | ✅ | ⚠️ (slower) | ✅ (very active) |
| **Industry adoption** | Universal | Declining | Growing fast |

### When to Use What?

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  Simple scripts, quick prototypes → pip + venv          │
│                                                         │
│  Medium projects, team collaboration → Poetry           │
│                                                         │
│  Legacy projects already using Pipenv → Keep Pipenv     │
│                                                         │
│  Building Python libraries/packages → Poetry            │
│                                                         │
│  Docker containers (minimal) → pip + requirements.txt   │
│                                                         │
│  New projects in 2024+ → Poetry (recommended)           │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 8. Best Practices for DevOps Engineers

### 1. Always Use Virtual Environments

```bash
# NEVER install packages globally (except pip, pipenv, poetry themselves)
# BAD:
sudo pip install flask    # ❌ Installs globally, can break system Python

# GOOD:
python -m venv venv       # ✅ Create isolated environment first
source venv/bin/activate
pip install flask
```

### 2. Pin Your Versions in Production

```bash
# BAD (in production):
requests         # ❌ Any version — could break tomorrow

# GOOD (in production):
requests==2.31.0  # ✅ Exact version — predictable and reproducible
```

### 3. Always Commit Lock Files

```bash
# For pip projects:
git add requirements.txt
# Make sure it has pinned versions (pip freeze output)

# For Pipenv projects:
git add Pipfile Pipfile.lock

# For Poetry projects:
git add pyproject.toml poetry.lock
```

### 4. Separate Dev and Prod Dependencies

```bash
# pip approach (two files):
# requirements.txt       → Production dependencies
# requirements-dev.txt   → Dev dependencies
#   Contains: -r requirements.txt  (includes prod deps too)
#             pytest==7.4.0
#             black==23.0.0

# Poetry approach (built-in):
poetry add flask              # Production
poetry add --group dev pytest  # Development only
```

### 5. Use --no-cache-dir in Docker

```dockerfile
# Saves space in your Docker image
RUN pip install --no-cache-dir -r requirements.txt
```

### 6. Scan for Vulnerabilities

```bash
# Using pip-audit
pip install pip-audit
pip-audit

# Using safety
pip install safety
safety check

# Using Pipenv
pipenv check

# Using Poetry (with plugin)
poetry add --group dev safety
poetry run safety check
```

## 9. Real-World Scenarios

### Scenario 1: Setting Up a Python Project for a Team

```bash
# Using Poetry (recommended)
mkdir devops-dashboard && cd devops-dashboard

# Initialize the project
poetry init --name devops-dashboard \
            --description "Internal DevOps Dashboard" \
            --author "DevOps Team" \
            --python "^3.11" \
            --no-interaction

# Add dependencies
poetry add flask requests pyyaml python-dotenv
poetry add --group dev pytest black ruff mypy

# Share with team
git init
git add pyproject.toml poetry.lock
git commit -m "Initial project setup"

# Teammate clones and runs:
git clone <repo-url>
cd devops-dashboard
poetry install        # Exact same versions as you!
poetry run python app.py
```

### Scenario 2: Converting requirements.txt to Poetry

```bash
# You have an old project with requirements.txt
# Step 1: Initialize Poetry
poetry init --no-interaction

# Step 2: Add each dependency from requirements.txt
# Manually:
poetry add flask==3.0.0 requests==2.31.0

# Or use a helper tool:
# Install dephell or use cat + xargs:
cat requirements.txt | grep -v "^#" | xargs poetry add
```

### Scenario 3: CI/CD Pipeline with Caching


## 10. Cheat Sheet

### pip Quick Reference

```bash
pip install <package>              # Install
pip install <package>==1.0.0       # Install specific version
pip install -r requirements.txt    # Install from file
pip install --upgrade <package>    # Upgrade
pip uninstall <package>            # Remove
pip list                           # List installed
pip list --outdated                # List outdated
pip freeze > requirements.txt     # Export to file
pip show <package>                 # Package details
pip search <query>                 # Search (deprecated, use pypi.org)
```

### Pipenv Quick Reference

```bash
pipenv install                     # Install all from Pipfile
pipenv install <package>           # Add + install
pipenv install --dev <package>     # Add as dev dependency
pipenv uninstall <package>         # Remove
pipenv shell                       # Activate venv
pipenv run <command>               # Run in venv
pipenv graph                       # Dependency tree
pipenv check                       # Security check
pipenv lock                        # Generate lock file
pipenv requirements                # Export as requirements.txt
```

### Poetry Quick Reference

```bash
poetry new <project>               # New project
poetry init                        # Init in existing project
poetry add <package>               # Add dependency
poetry add --group dev <package>   # Add dev dependency
poetry remove <package>            # Remove
poetry install                     # Install all
poetry install --without dev       # Install prod only
poetry update                      # Update packages
poetry show --tree                 # Dependency tree
poetry shell                       # Activate venv
poetry run <command>               # Run in venv
poetry build                       # Build package
poetry publish                     # Publish to PyPI
poetry export -f requirements.txt  # Export as requirements.txt
```

## Summary

| Tool | Best For | Config File | Lock File |
|------|----------|-------------|-----------|
| **pip** | Simple projects, Docker, universal compatibility | requirements.txt | ❌ |
| **Pipenv** | Legacy projects that already use it | Pipfile | Pipfile.lock |
| **Poetry** | New projects, teams, building libraries | pyproject.toml | poetry.lock |

**As a DevOps engineer:**
- You'll encounter **all three** in the wild
- **pip** is everywhere — learn it deeply
- **Poetry** is the future — use it for new projects
- **Pipenv** is fading but still exists in many codebases
- **Always** use virtual environments, pin versions, and commit lock files
