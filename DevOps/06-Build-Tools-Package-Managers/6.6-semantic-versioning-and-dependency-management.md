## 1. Why Do We Need Versioning?

Imagine this scenario:

```
Monday:    You install library v??? → Your app works perfectly ✅
Wednesday: Library author pushes an update
Thursday:  Your colleague installs the library → Gets the new version → App breaks 💥

You: "But it works on my machine!"
Colleague: "Well, it doesn't on mine!"
```

Without versioning, you have **no way** to:
- Know which version of a library you're using
- Communicate what changed between releases
- Guarantee your app will work the same everywhere
- Roll back to a working version

**Versioning gives every release a unique identity**, like a name tag. And **Semantic Versioning** gives that name tag a **meaning**.

## 2. What is Semantic Versioning (SemVer)?

**Semantic Versioning** (or **SemVer**) is a versioning system where the version number itself **tells you what kind of changes were made**. It was created by Tom Preston-Werner (co-founder of GitHub).

The format is:

```
MAJOR.MINOR.PATCH

Example: 2.4.1
         │ │ │
         │ │ └── PATCH = 1 (bug fixes)
         │ └──── MINOR = 4 (new features)
         └────── MAJOR = 2 (breaking changes)
```

Think of it like this:

```
MAJOR = "I changed the rules. Your old code might break."
MINOR = "I added new stuff. Your old code still works."
PATCH = "I fixed bugs. Nothing else changed."
```

### The Official Rules

From [semver.org](https://semver.org):

1. **PATCH** version increments when you make **backward-compatible bug fixes**
2. **MINOR** version increments when you add **backward-compatible new functionality**
3. **MAJOR** version increments when you make **backward-incompatible (breaking) changes**

## 3. Breaking Down MAJOR.MINOR.PATCH

### PATCH Version (x.y.**Z**)

**When to bump:** You fixed a bug. Nothing else changed.

```
1.0.0 → 1.0.1  (fixed a typo in error message)
1.0.1 → 1.0.2  (fixed a crash when input is empty)
2.3.5 → 2.3.6  (fixed a security vulnerability)
```

**Real-world example:**

```python
# Version 1.0.0
def calculate_tax(amount):
    return amount * 0.1    # Bug: should be 0.10, but uses wrong formula for negatives

# Version 1.0.1 (PATCH — bug fix)
def calculate_tax(amount):
    if amount < 0:
        return 0           # Fixed: handle negative amounts
    return amount * 0.1
```

**For users:** Safe to update. Your code won't break.

---

### MINOR Version (x.**Y**.z)

**When to bump:** You added new features, but everything that worked before still works.

```
1.0.2 → 1.1.0  (added a new function)
2.3.6 → 2.4.0  (added a new optional parameter)
3.0.0 → 3.1.0  (added support for a new file format)
```

**Notice:** When MINOR goes up, PATCH resets to 0.

**Real-world example:**

```python
# Version 1.0.0
def send_email(to, subject, body):
    """Send an email."""
    # ... sends email

# Version 1.1.0 (MINOR — new feature, old code still works)
def send_email(to, subject, body, cc=None, bcc=None):
    """Send an email. Now supports CC and BCC!"""
    # ... sends email with optional CC/BCC
    # Old code: send_email("a@b.com", "Hi", "Hello") → Still works! ✅
    # New code: send_email("a@b.com", "Hi", "Hello", cc="c@b.com") → New feature! 🎉
```

**For users:** Safe to update. New stuff available, old stuff still works.

---

### MAJOR Version (x.y.z → **X**.0.0)

**When to bump:** You made changes that will **break** existing code.

```
1.4.2 → 2.0.0  (renamed a function)
2.3.1 → 3.0.0  (removed a feature)
3.1.0 → 4.0.0  (changed how configuration works)
```

**Notice:** When MAJOR goes up, both MINOR and PATCH reset to 0.

**Real-world example:**

```python
# Version 1.5.3
def get_user(user_id):
    """Returns a dictionary."""
    return {"id": user_id, "name": "Alice"}

# Version 2.0.0 (MAJOR — breaking change!)
def get_user(user_id):
    """Now returns a User object instead of a dictionary."""
    return User(id=user_id, name="Alice")

# Old code that worked with v1.x:
user = get_user(1)
print(user["name"])     # ❌ BREAKS in v2.0.0! (User object doesn't support ["name"])

# New code for v2.x:
user = get_user(1)
print(user.name)        # ✅ Works with v2.0.0
```

**For users:** Read the changelog before updating! Your code might need changes.

### Visual Summary

```
Version: 3.7.2
         │ │ │
         │ │ └── 2 bug fixes since 3.7.0
         │ └──── 7 feature additions since 3.0.0
         └────── 3 sets of breaking changes since 1.0.0

Version History Flow:
1.0.0 → 1.0.1 → 1.0.2 → 1.1.0 → 1.1.1 → 1.2.0 → 2.0.0 → 2.0.1 → 2.1.0
  │       │       │       │       │       │       │       │       │
 init   patch   patch   minor   patch   minor   MAJOR   patch   minor
```

## 4. Pre-release Versions & Build Metadata

### Pre-release Versions

Sometimes you want to release a version for testing **before** the official release. SemVer supports this with a **hyphen and a label**:

```
1.0.0-alpha       → Very early, unstable, might not even be feature-complete
1.0.0-alpha.1     → First alpha revision
1.0.0-alpha.2     → Second alpha revision
1.0.0-beta        → Feature-complete but may have bugs
1.0.0-beta.1      → First beta revision
1.0.0-rc.1        → Release Candidate 1 — "we think it's ready"
1.0.0-rc.2        → Release Candidate 2 — "okay, NOW it's ready"
1.0.0             → Official release! 🎉
```

**The typical release pipeline:**

```
alpha → beta → rc (release candidate) → stable

1.0.0-alpha.1   "Here's a sneak peek, expect bugs"
      ↓
1.0.0-beta.1    "Feature complete, please test"
      ↓
1.0.0-rc.1      "We think this is the final version"
      ↓
1.0.0           "Official release!"
```

### Version Precedence (Sorting)

Pre-release versions have **lower precedence** than the release version:

```
1.0.0-alpha < 1.0.0-alpha.1 < 1.0.0-beta < 1.0.0-beta.2 < 1.0.0-rc.1 < 1.0.0
```

This means if you ask for "version 1.0.0 or higher", pre-releases are **excluded** by default (which is what you want).

### Build Metadata

Build metadata is appended with a `+` sign and is **ignored** when comparing versions:

```
1.0.0+build.123             → Build number 123
1.0.0+20240101               → Built on January 1, 2024
1.0.0-beta+exp.sha.5114f85  → Beta with git commit hash

# These are ALL considered equal:
1.0.0+build.1
1.0.0+build.2
1.0.0+build.999
# (Build metadata does NOT affect version precedence)
```

## 5. Version Ranges & Constraints

When you install a dependency, you usually don't want **exactly** one version — you want a **range** of acceptable versions. Different tools use different syntax:

### Exact Version

```
requests==2.31.0       # pip — exactly 2.31.0, nothing else
"requests": "2.31.0"   # npm — exactly 2.31.0
```

### Comparison Operators

```bash
# pip / Python
>=2.0.0          # 2.0.0 or higher
<=3.0.0          # 3.0.0 or lower
>2.0.0           # Higher than 2.0.0 (not including 2.0.0)
!=2.5.0          # Anything except 2.5.0
>=2.0.0,<3.0.0   # Between 2.0.0 (inclusive) and 3.0.0 (exclusive)
```

### Tilde (~) — Patch-Level Changes Only

The tilde says: "Allow PATCH updates, but nothing higher."

```
~2.3.1

Means: >=2.3.1 and <2.4.0

✅ 2.3.1
✅ 2.3.2
✅ 2.3.9
❌ 2.4.0    (too high — MINOR changed)
❌ 3.0.0    (way too high — MAJOR changed)
```

Think of `~` as: **"About this version"** — allow tiny bug fixes only.

### Caret (^) — Minor-Level Changes

The caret says: "Allow MINOR and PATCH updates, but not MAJOR."

```
^2.3.1

Means: >=2.3.1 and <3.0.0

✅ 2.3.1
✅ 2.3.9
✅ 2.4.0
✅ 2.99.99
❌ 3.0.0    (MAJOR changed — could break things)
```

Think of `^` as: **"Compatible with this version"** — allow new features and bug fixes, but not breaking changes.

### Special Case — Versions Starting with 0

For 0.x versions (unstable/pre-1.0 software), the rules are stricter:

```
^0.2.3

Means: >=0.2.3 and <0.3.0  (NOT <1.0.0!)

Why? Because 0.x software is considered unstable.
Even a MINOR bump (0.2 → 0.3) could contain breaking changes.
```

### Wildcard (*)

```
2.3.*    →  Any patch version of 2.3 (same as ~2.3.0)
2.*      →  Any minor/patch version of 2.x (same as ^2.0.0)
*        →  Any version (dangerous in production!)
```

### Comparison Table

| Syntax | Meaning | Example Range |
|--------|---------|---------------|
| `2.3.1` | Exactly 2.3.1 | 2.3.1 only |
| `~2.3.1` | Patch updates only | >=2.3.1, <2.4.0 |
| `^2.3.1` | Minor + patch updates | >=2.3.1, <3.0.0 |
| `>=2.3.1` | This version or higher | 2.3.1, 2.4.0, 3.0.0, 99.0.0... |
| `>=2.0,<3.0` | Range | 2.0.0 through 2.99.99 |
| `2.*` | Any 2.x version | 2.0.0 through 2.99.99 |
| `*` | Any version | Everything |

## 6. What is Dependency Management?

**Dependency management** is the process of handling all the external packages (libraries, tools, frameworks) your project relies on.

### Why It Matters

Your project doesn't exist in isolation. It depends on **other people's code**:

```
Your App
├── flask (web framework)
│   ├── werkzeug (HTTP toolkit)
│   │   └── markupsafe
│   ├── jinja2 (templates)
│   │   └── markupsafe
│   ├── click (CLI)
│   ├── blinker (signals)
│   └── itsdangerous (crypto)
├── requests (HTTP client)
│   ├── urllib3
│   ├── certifi
│   ├── charset-normalizer
│   └── idna
└── sqlalchemy (database)
    └── ...
```

You installed **3 packages**, but you actually depend on **15+**! Those extra packages are called **transitive dependencies** (or indirect dependencies).

### Direct vs Transitive Dependencies

```
DIRECT Dependencies        TRANSITIVE Dependencies
(you explicitly install)   (pulled in automatically)

┌─────────────┐
│ Your App    │
├─────────────┤
│ flask       │──────→ werkzeug, jinja2, click, blinker, itsdangerous
│ requests    │──────→ urllib3, certifi, charset-normalizer, idna
│ sqlalchemy  │──────→ greenlet
└─────────────┘

You manage 3 packages.
Your package manager manages 12+.
```

```bash
# npm

# Poetry
poetry show --tree

# Go
go mod graph

# Maven
mvn dependency:tree
```

### Diamond Dependency Problem

This is a classic issue in dependency management:

```
        Your App
       /        \
    Lib A      Lib B
       \        /
      Lib C v1.0   ← Lib A wants v1.0
      Lib C v2.0   ← Lib B wants v2.0
        
        🤔 Which version of Lib C should we install?
```

This is called the **diamond dependency problem**. Different package managers handle it differently:

| Package Manager | Strategy |
|----------------|----------|
| **pip** | Installs whichever version it encounters last (can cause issues) |
| **npm** | Installs BOTH versions in nested node_modules |
| **Poetry** | Uses a SAT solver to find a compatible version |
| **Go modules** | Uses Minimal Version Selection (MVS) |
| **Maven** | Uses "nearest definition" wins |

## 8. Lock Files — The Secret to Reproducible Builds

### What is a Lock File?

A **lock file** captures the **exact version** of every dependency (direct AND transitive) at a specific point in time.

```
Without lock file:                With lock file:
┌─────────────────────┐          ┌─────────────────────┐
│ pyproject.toml      │          │ poetry.lock          │
│                     │          │                      │
│ flask = "^3.0.0"    │          │ flask==3.0.0         │
│ (could be 3.0.0,    │          │ werkzeug==3.0.1      │
│  3.0.1, 3.1.0...)   │          │ jinja2==3.1.2        │
│                     │          │ markupsafe==2.1.3    │
│ Result: DIFFERENT   │          │ click==8.1.7         │
│ versions on each    │          │ blinker==1.7.0       │
│ machine ❌          │          │ itsdangerous==2.1.2  │
└─────────────────────┘          │                      │
                                 │ Result: IDENTICAL    │
                                 │ versions everywhere ✅│
                                 └─────────────────────┘
```

### Lock Files Across Languages

| Language | Manifest File | Lock File |
|----------|--------------|-----------|
| Python (pip) | requirements.txt | ❌ (pin versions manually) |
| Python (Pipenv) | Pipfile | Pipfile.lock |
| Python (Poetry) | pyproject.toml | poetry.lock |
| JavaScript (npm) | package.json | package-lock.json |
| JavaScript (yarn) | package.json | yarn.lock |
| JavaScript (pnpm) | package.json | pnpm-lock.yaml |
| Go | go.mod | go.sum |
| Rust | Cargo.toml | Cargo.lock |
| Ruby | Gemfile | Gemfile.lock |
| Java (Gradle) | build.gradle | gradle.lockfile |
| PHP | composer.json | composer.lock |

### Should You Commit Lock Files?

```
┌─────────────────────────────────────────────────────┐
│                                                     │
│  Applications (web apps, APIs, services, scripts):  │
│  ✅ YES — Always commit the lock file               │
│  Why: Everyone gets the exact same versions          │
│                                                     │
│  Libraries (packages others install):               │
│  ❌ NO — Don't commit the lock file                  │
│  Why: Users should resolve versions themselves       │
│  (But do commit pyproject.toml / package.json!)     │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### How Lock Files Work in Practice

```bash
# Developer 1 creates the project:
poetry add flask requests
# This creates:
# - pyproject.toml (version RANGES like ^3.0.0)
# - poetry.lock    (EXACT versions like 3.0.0)

git add pyproject.toml poetry.lock
git commit -m "Add dependencies"
git push

# Developer 2 clones the project:
git clone <repo>
cd <repo>
poetry install
# Poetry reads poetry.lock → installs EXACT same versions as Developer 1
# 100% identical environment! ✅

# Months later, Developer 3 joins:
git clone <repo>
poetry install
# STILL gets the exact same versions from poetry.lock! ✅
# Even though newer versions exist on PyPI
```

### Updating Lock Files

```bash
# When you intentionally want to update:
poetry update                    # Update ALL packages (within range constraints)
poetry update requests           # Update only "requests"

# For npm:
npm update                       # Update within ranges
npm install package@latest       # Update specific package

# For pip:
pip install --upgrade -r requirements.txt   # Update and re-freeze
pip freeze > requirements.txt               # Overwrite with new versions
```

## 9. Dependency Hell — And How to Escape It

**Dependency hell** is when managing dependencies becomes a nightmare. Here are the most common forms:

### 1. Version Conflict

```
Package A requires: library >= 2.0, < 3.0
Package B requires: library >= 3.0

→ IMPOSSIBLE to satisfy both! 💥
```

**Solution:**
- Check if newer versions of Package A or B are compatible
- Look for alternative packages
- Use a tool with better dependency resolution (Poetry > pip)

### 2. Outdated Dependencies

```
Your app uses a package that was last updated in 2019.
That package depends on another package that's now deprecated.
That deprecated package has a known security vulnerability.

→ You're stuck with a vulnerable transitive dependency! 💀
```

**Solution:**
- Regularly update dependencies (`poetry update`, `npm update`)
- Use tools like Dependabot or Renovate for automated updates
- Monitor security advisories

### 3. Too Many Dependencies

```
You install 5 packages.
They bring in 200 transitive dependencies.
One of those 200 gets compromised.

→ Supply chain attack! 🎯
```

**Real example:** The `event-stream` npm package was compromised in 2018, affecting thousands of projects.

**Solution:**
- Audit your dependency tree regularly
- Use lock files with hash verification
- Prefer well-maintained packages with few dependencies
- Use `pip-audit`, `npm audit`, or `snyk` for security scanning

### 4. "Works on My Machine" Syndrome

```
Dev machine:   Python 3.11, requests 2.31.0, flask 3.0.0
CI server:     Python 3.9,  requests 2.28.0, flask 2.3.0
Production:    Python 3.10, requests ???,     flask ???

→ Three different environments, three different behaviors! 🤯
```

**Solution:**
- Use lock files (always!)
- Use Docker for consistent environments
- Pin Python/Node/Go versions in CI/CD
- Use `pyenv`, `nvm`, or `asdf` for version management

## 10. Dependency Management Across Languages

### Python

```bash
# ─── pip (basic) ───
pip install flask==3.0.0
pip freeze > requirements.txt          # Export
pip install -r requirements.txt        # Import

# ─── Poetry (recommended) ───
poetry add flask                       # Add dependency

```

### JavaScript (Node.js)

```bash
# ─── npm ───
npm init -y                            # Create package.json

npm ci                                 # Install from lock file (CI-friendly)
npm audit                              # Security check

# ─── yarn ───
yarn add express                       # Add dependency
yarn add --dev jest                    # Add dev dependency
yarn install --frozen-lockfile         # Install from lock file

# ─── pnpm ───
pnpm add express                       # Add dependency
pnpm install --frozen-lockfile         # Install from lock file
```

```json
// package.json example
{
  "name": "my-api",

  },
  "devDependencies": {
    "jest": "^29.0.0",

```

## 11. Security & Vulnerability Management

### Why Security Matters in Dependencies

Your app is only as secure as its **weakest dependency**. In 2021, the Log4Shell vulnerability (CVE-2021-44228) affected millions of Java applications through the log4j library.

```
Your App
└── Some Framework
    └── Some Utility
        └── log4j 2.14.1  ← VULNERABLE! Remote Code Execution! 💀
```

You might not even know you're using log4j — it's three levels deep!

### Security Scanning Tools

```bash
# ─── Python ───
# pip-audit (official)
pip install pip-audit
pip-audit
# Output:
# Name       Version  ID                   Fix Versions
# ---------- -------- -------------------- ---------------
# urllib3    1.26.5   PYSEC-2023-212       1.26.18, 2.0.7

# safety
pip install safety
safety check

# Poetry
poetry add --group dev pip-audit
poetry run pip-audit
```

```bash
# ─── JavaScript ───
npm audit
npm audit fix         # Auto-fix vulnerabilities (if possible)
npm audit --json      # JSON output for CI/CD parsing

yarn audit
```

```bash
# ─── Go ───
go install golang.org/x/vuln/cmd/govulncheck@latest
govulncheck ./...
```

```bash
# ─── Universal tools ───
# Snyk (free tier available)
snyk test              # Test current project
snyk monitor           # Monitor for new vulnerabilities

# Trivy (by Aqua Security — great for containers too!)
trivy fs .             # Scan filesystem
trivy image myapp:1.0  # Scan Docker image
```

### Automated Dependency Updates

Instead of manually checking for updates, use **bots** that create pull requests automatically:

#### GitHub Dependabot

```yaml
# .github/dependabot.yml
version: 2
updates:
  # Python dependencies
  - package-ecosystem: "pip"
    directory: "/"
    schedule:
      interval: "weekly"
    open-pull-requests-limit: 10

  # Docker base images
  - package-ecosystem: "docker"
    directory: "/"
    schedule:
      interval: "weekly"

  # GitHub Actions
  - package-ecosystem: "github-actions"
    directory: "/"
    schedule:
      interval: "weekly"

  # npm dependencies
  - package-ecosystem: "npm"
    directory: "/frontend"
    schedule:
      interval: "daily"
    # Group minor and patch updates into one PR
    groups:
      minor-and-patch:
        update-types:
          - "minor"
          - "patch"
```

#### Renovate Bot

```json
// renovate.json
{
  "$schema": "https://docs.renovatebot.com/renovate-schema.json",
  "extends": [
    "config:base",
    ":automergeMinor",
    ":automergeDigest"
  ],
  "packageRules": [
    {
      "matchUpdateTypes": ["minor", "patch"],
      "automerge": true
    },
    {
      "matchUpdateTypes": ["major"],
      "automerge": false,
      "labels": ["major-update", "needs-review"]
    }
  ]
}
```

**What these do:**
- Automatically create PRs when new versions are available
- Run your CI/CD tests against the updated versions
- Auto-merge safe updates (minor/patch) if tests pass
- Flag major updates for manual review

## 12. Versioning in DevOps Workflows

### Docker Image Versioning

```bash
# ❌ BAD — Using "latest" tag in production
docker build -t myapp:latest .
# Problem: "latest" changes every build. Which version is running?

# ✅ GOOD — Using semantic versions
docker build -t myapp:1.2.3 .
docker build -t myapp:1.2.3-rc.1 .  # Release candidate

# ✅ BEST — Multiple tags for flexibility
docker build -t myapp:1.2.3 \
             -t myapp:1.2 \
             -t myapp:1 \
             -t myapp:latest .
# Users can choose their precision:
# myapp:1.2.3  → Exact version (production)
# myapp:1.2    → Latest patch of 1.2.x
# myapp:1      → Latest minor of 1.x
# myapp:latest → Latest release (development only)
```

### Git Tags for Releases

```bash
# Create a version tag
git tag v1.0.0
git tag -a v1.0.0 -m "Release version 1.0.0"  # Annotated tag (recommended)

# Push tags to remote
git push origin v1.0.0
git push origin --tags      # Push ALL tags

# List tags
git tag -l "v1.*"           # List all v1.x tags

# Delete a tag
git tag -d v1.0.0-beta      # Delete locally
git push origin :v1.0.0-beta  # Delete from remote
```

### CI/CD Pipeline with Versioning

```yaml
# GitHub Actions — Build and tag on release
name: Release

on:
  push:
    tags:
      - 'v*'     # Trigger on version tags (v1.0.0, v2.1.3, etc.)

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Extract version from tag
        id: version
        run: echo "VERSION=${GITHUB_REF#refs/tags/v}" >> $GITHUB_OUTPUT
        # If tag is "v1.2.3", VERSION becomes "1.2.3"

      - name: Build Docker image
        run: |
          docker build \
            -t myregistry/myapp:${{ steps.version.outputs.VERSION }} \
            -t myregistry/myapp:latest \
            .

      - name: Push Docker image
        run: |
          docker push myregistry/myapp:${{ steps.version.outputs.VERSION }}
          docker push myregistry/myapp:latest

      - name: Create GitHub Release
        uses: softprops/action-gh-release@v1
        with:
          generate_release_notes: true
```

### Helm Chart Versioning

```yaml
# Chart.yaml
```

## 13. Automating Version Management

### Conventional Commits

**Conventional Commits** is a specification for commit messages that tools can parse to **automatically determine** the next version number.

```
Format: <type>(<scope>): <description>

Types:
  fix:      → Bumps PATCH (1.0.0 → 1.0.1)
  feat:     → Bumps MINOR (1.0.0 → 1.1.0)
  feat!:    → Bumps MAJOR (1.0.0 → 2.0.0)  (! = breaking change)
  
  BREAKING CHANGE: in footer → Also bumps MAJOR
```

**Examples:**

```bash
# These commit messages are machine-readable:

git commit -m "fix: resolve null pointer in user login"
# → Next version: PATCH bump (1.2.3 → 1.2.4)

git commit -m "feat: add OAuth2 authentication support"
# → Next version: MINOR bump (1.2.3 → 1.3.0)

git commit -m "feat!: redesign configuration file format"
# → Next version: MAJOR bump (1.2.3 → 2.0.0)

git commit -m "fix(api): handle timeout errors gracefully"
# → PATCH bump, scoped to "api" component

git commit -m "docs: update README installation steps"
# → No version bump (just documentation)

git commit -m "chore: update CI pipeline configuration"
# → No version bump (internal maintenance)
```

### Semantic Release (Automated Versioning)

**semantic-release** is a tool that:
1. Reads your commit messages
2. Determines the next version number
3. Creates a git tag
4. Generates a changelog
5. Publishes the release

```yaml
# .github/workflows/release.yml
name: Release

on:
  push:
    branches: [main]

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # Need full history for changelog

      - uses: actions/setup-node@v4
        with:
          node-version: 20

      - name: Install semantic-release
        run: npm install -g semantic-release @semantic-release/changelog @semantic-release/git

      - name: Release
        run: npx semantic-release
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

```json
// .releaserc.json
{
  "branches": ["main"],
  "plugins": [
    "@semantic-release/commit-analyzer",
    "@semantic-release/release-notes-generator",
    "@semantic-release/changelog",
    "@semantic-release/github",
    "@semantic-release/git"
  ]
}
```

**The flow:**

```
Developer pushes to main:
  commit: "fix: handle edge case in parser"
  commit: "feat: add CSV export feature"

semantic-release runs:
  1. Analyzes commits since last release
  2. Finds: 1 fix + 1 feat = MINOR bump
  3. Current version: 1.3.2
  4. New version: 1.4.0
  5. Creates tag: v1.4.0
  6. Generates changelog
  7. Creates GitHub release with notes
  
All automatically! 🎉
```

### Python-Specific: python-semantic-release

```bash
# Install
pip install python-semantic-release

# Configuration in pyproject.toml:
```

```toml
[tool.semantic_release]
version_toml = ["pyproject.toml:tool.poetry.version"]
branch = "main"
build_command = "poetry build"
commit_message = "chore(release): {version}"
```

```bash

```

## 14. Best Practices

### 1. Always Use Semantic Versioning

```
✅ v1.2.3
✅ v0.1.0-alpha
✅ v2.0.0-rc.1

❌ v1.2.3.4          (too many numbers)
❌ build-2024-01-15  (not SemVer)
❌ v1.2              (missing PATCH)
```

### 2. Start at 0.1.0 for New Projects

```
0.1.0 → "This is new, expect changes, no promises"
0.x.x → "Still developing, API might change"
1.0.0 → "This is stable! We promise backward compatibility"
```

### 3. Never Modify a Released Version

```
❌ WRONG:  Release 1.2.3, find a bug, fix it, publish as 1.2.3 again
✅ RIGHT:  Release 1.2.3, find a bug, fix it, publish as 1.2.4
```

Once a version is published, it's **immutable**. Always increment.

### 4. Pin Versions in Production

```bash
# Development (flexible):
flask^=3.0.0        # "I want new features"

# Production (strict):
flask==3.0.2        # "I want this EXACT version, tested and verified"
```

### 5. Use Lock Files

```bash
# Always commit lock files for applications
git add poetry.lock          # Python
git add package-lock.json    # Node.js
git add go.sum               # Go
git add Cargo.lock            # Rust
```

### 6. Update Dependencies Regularly

```bash
# Don't wait 2 years to update — the diff will be huge and painful
# Instead, update weekly/monthly:

poetry update          # Update within constraints
npm update             # Update within constraints
go get -u ./...        # Update Go dependencies

# Use Dependabot or Renovate for automation
```

### 7. Audit Dependencies for Security

```bash
# Make this part of your CI/CD pipeline:
pip-audit              # Python
npm audit              # Node.js
govulncheck ./...      # Go
trivy fs .             # Universal
```

### 8. Use Conventional Commits

```bash
# Structure your commit messages for automated versioning:
git commit -m "feat: add user registration endpoint"
git commit -m "fix: prevent SQL injection in search query"
git commit -m "docs: update API documentation"
git commit -m "feat!: migrate from REST to GraphQL"
```

## 15. Cheat Sheet

### Semantic Version Format

```
MAJOR.MINOR.PATCH[-pre-release][+build-metadata]

Examples:
  1.0.0
  1.2.3
  2.0.0-alpha.1
  1.0.0-beta+build.123
  3.1.4-rc.2+20240101
```

### Version Bump Rules

```
Bug fix (backward compatible)        → Bump PATCH  (1.0.0 → 1.0.1)
New feature (backward compatible)    → Bump MINOR  (1.0.0 → 1.1.0)
Breaking change                      → Bump MAJOR  (1.0.0 → 2.0.0)
```

### Version Ranges Quick Reference

```
==1.2.3     Exactly 1.2.3
>=1.2.3     1.2.3 or higher
~1.2.3      >=1.2.3, <1.3.0  (patch updates only)
^1.2.3      >=1.2.3, <2.0.0  (minor + patch updates)
1.2.*       >=1.2.0, <1.3.0  (any patch of 1.2)
*           Any version
```

### Conventional Commit Types

```
fix:        Bug fix              → PATCH bump
feat:       New feature          → MINOR bump
feat!:      Breaking feature     → MAJOR bump
docs:       Documentation        → No bump
style:      Formatting           → No bump
refactor:   Code restructuring   → No bump
test:       Tests                → No bump
chore:      Maintenance          → No bump
perf:       Performance          → No bump (or PATCH)
ci:         CI/CD changes        → No bump
build:      Build system         → No bump
```

### Lock File Commands

```bash
# Python (Poetry)
poetry lock                    # Generate/update lock file
poetry install                 # Install from lock file

# Node.js (npm)
npm ci                         # Install from lock file (CI)
npm install                    # Install and update lock file

# Go
go mod tidy                    # Clean up and update go.sum
go mod verify                  # Verify dependencies

# Rust
cargo update                   # Update lock file
cargo install --locked         # Install from lock file
```

### Security Scanning

```bash
# Python
pip-audit                      # Scan pip packages
safety check                   # Scan for vulnerabilities

# Node.js
npm audit                      # Built-in audit
npm audit fix                  # Auto-fix

# Go
govulncheck ./...             # Official vuln checker

# Universal
trivy fs .                     # Filesystem scan
snyk test                      # Snyk scan
```

## Summary

| Concept | What It Means |
|---------|--------------|
| **Semantic Versioning** | MAJOR.MINOR.PATCH — version numbers that tell you what changed |
| **MAJOR** | Breaking changes — your old code might not work |
| **MINOR** | New features — backward compatible |
| **PATCH** | Bug fixes — backward compatible |
| **Lock file** | Records exact versions for reproducible builds |
| **Dependency tree** | Full hierarchy of direct + transitive dependencies |
| **Dependency hell** | When dependencies conflict and nothing works |
| **Conventional Commits** | Structured commit messages for automated versioning |
| **semantic-release** | Automates version bumps, changelogs, and releases |
| **Dependabot/Renovate** | Bots that automatically create update PRs |

**As a DevOps engineer, versioning touches everything you do:**
- Docker images need proper tags
- Helm charts have versions
- Terraform modules are versioned
- CI/CD pipelines create releases
- Dependencies must be managed and secured

**The golden rule:** Be explicit about versions. If you don't control the version, the version controls you.
