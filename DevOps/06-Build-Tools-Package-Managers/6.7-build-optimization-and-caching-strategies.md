## 1. Why Do Builds Get Slow?

Before we optimize, let's understand **why builds become painfully slow** in the first place. Here are the usual suspects:

| Problem | What Happens |
|---------|-------------|
| **Re-downloading dependencies** every time | npm install / pip install runs from scratch on every build |
| **No caching** | Build tools redo everything even if nothing changed |
| **Monolithic builds** | The entire app is rebuilt even when you change one tiny file |
| **Large base images** | Docker pulls 1 GB+ images every time |
| **Serial (sequential) execution** | Tasks that could run in parallel are waiting in line |
| **Unnecessary files in build context** | Sending gigabytes of irrelevant files to the builder |
| **No .dockerignore / .gitignore** | node_modules, logs, temp files all get included |

> **The golden rule:** Don't redo work that's already been done. That's the entire philosophy of build optimization.

## 2. What is Build Optimization?

**Build optimization** means making the process of turning your source code into a deployable artifact as **fast, efficient, and resource-friendly** as possible.

Think of it like cooking:
- Instead of buying fresh groceries every single time, you keep staples stocked (caching).
- Instead of making everything from scratch, you prep ingredients in advance (incremental builds).
- Instead of cooking dishes one at a time, you use multiple burners (parallel builds).

### The Goals of Build Optimization

```
┌─────────────────────────────────────────────┐
│           Build Optimization Goals           │
├─────────────────────────────────────────────┤
│  ✅ Reduce build time (minutes → seconds)    │
│  ✅ Save compute resources (CPU, memory)     │
│  ✅ Lower CI/CD costs (fewer build minutes)  │
│  ✅ Faster developer feedback loops          │
│  ✅ More frequent, confident deployments     │
└─────────────────────────────────────────────┘
```

## 3. What is Build Caching?

**Build caching** is the practice of **storing the results of expensive operations** so that the next time the same operation is needed, you can just reuse the stored result instead of recalculating it.

### Analogy

Imagine you solve a complex math problem and write down the answer. Next time someone asks the same question, you just read the answer instead of solving it again. That's caching!

### How Caching Works (General Flow)

```
Build Step Starts
      │
      ▼
 Has anything changed    ──── NO ────→  Use cached result ✅
 since last build?                       (skip the work)
      │
     YES
      │
      ▼
 Run the step from scratch
      │
      ▼
 Store result in cache
 for next time
```

### Cache Key Concept

Every cache needs a **key** — a unique identifier that tells the build system **when to reuse and when to rebuild**.

Common cache keys include:
- **File hash** — A checksum (SHA256) of the input files. If the file hasn't changed, the hash is the same → use cache.
- **Lock file hash** — For dependencies, the lock file (`package-lock.json`, `Pipfile.lock`, `go.sum`) acts as the key.
- **Branch name** — Some CI systems cache per-branch.
- **Timestamp** — Rarely used alone (too unreliable), but sometimes combined with other keys.


## 6. Dependency Caching

This is the **single most impactful optimization** for most projects. Why? Because downloading and installing dependencies (libraries, packages) often takes the **majority** of build time.

### The Problem

Without caching, every build does this:

```
Step 1: Delete everything from last build
Step 2: Download ALL packages from internet (again!)
Step 3: Install ALL packages (again!)
Step 4: Build the actual app
```

That's like going to the grocery store to buy everything from scratch every time you cook dinner — even though you already have salt, oil, and spices at home.

### The Solution: Cache the Dependencies

```
Step 1: Check — has the dependency file changed?
          (package-lock.json / requirements.txt / pom.xml)
Step 2: If NO → restore cached node_modules / .venv / .m2
Step 3: If YES → install fresh, then update the cache
Step 4: Build the actual app
```

### Example: npm Dependency Caching in GitHub Actions

```yaml
# .github/workflows/build.yml
name: Build App

on: push

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      # Cache node_modules based on package-lock.json
      - name: Cache Node.js dependencies
        uses: actions/cache@v4
        with:
          path: ~/.npm                              # What to cache
          key: npm-${{ hashFiles('**/package-lock.json') }}  # Cache key
          restore-keys: |                            # Fallback keys
            npm-

      - name: Install dependencies
        run: npm ci    # Uses cache if available!

      - name: Build
        run: npm run build
```

**How this works line by line:**

1. `path: ~/.npm` — This tells GitHub Actions **where** npm stores its cached packages. We're saving this folder.
2. `key: npm-${{ hashFiles('**/package-lock.json') }}` — This creates a **unique cache key** by hashing the lock file. If `package-lock.json` hasn't changed, the hash is the same → **cache hit!**
3. `restore-keys: npm-` — If there's no exact match, try to find any cache that starts with `npm-`. This gives a **partial cache hit**, which is still faster than nothing.
4. `npm ci` — This is the **clean install** command. It installs exactly what's in `package-lock.json`. If the cache is available, npm skips downloading what's already cached.

### Example: pip Dependency Caching in GitHub Actions

```yaml
- name: Cache pip packages
  uses: actions/cache@v4
  with:
    path: ~/.cache/pip
    key: pip-${{ hashFiles('**/requirements.txt') }}
    restore-keys: |
      pip-

- name: Install dependencies
  run: pip install -r requirements.txt
```

### Example: Maven Dependency Caching in GitHub Actions

```yaml
- name: Cache Maven packages
  uses: actions/cache@v4
  with:
    path: ~/.m2/repository
    key: maven-${{ hashFiles('**/pom.xml') }}
    restore-keys: |
      maven-

- name: Build with Maven
  run: mvn clean package -DskipTests
```

### Example: Gradle Dependency Caching

```yaml
- name: Cache Gradle packages
  uses: actions/cache@v4
  with:
    path: |
      ~/.gradle/caches
      ~/.gradle/wrapper
    key: gradle-${{ hashFiles('**/*.gradle*', '**/gradle-wrapper.properties') }}
    restore-keys: |
      gradle-

- name: Build with Gradle
  run: ./gradlew build
```

### Cache Locations by Tool

| Tool | Cache Directory | Lock File for Key |
|------|----------------|-------------------|
| npm | `~/.npm` | `package-lock.json` |
| yarn | `~/.cache/yarn` | `yarn.lock` |
| pnpm | `~/.pnpm-store` | `pnpm-lock.yaml` |
| pip | `~/.cache/pip` | `requirements.txt` or `Pipfile.lock` |
| Maven | `~/.m2/repository` | `pom.xml` |
| Gradle | `~/.gradle/caches` | `*.gradle` files |
| Go | `~/go/pkg/mod` | `go.sum` |
| Rust/Cargo | `~/.cargo` | `Cargo.lock` |

## 7. Layer Caching (Docker Context)

Docker builds images in **layers** — each instruction in a Dockerfile creates a layer. Docker is smart: if a layer hasn't changed, it reuses the cached layer instead of rebuilding it.

### How Docker Layer Caching Works

```dockerfile
FROM node:18-alpine        # Layer 1 — Base image (rarely changes)
WORKDIR /app               # Layer 2 — Set directory (never changes)
COPY package*.json ./      # Layer 3 — Copy dependency files
RUN npm ci                 # Layer 4 — Install dependencies
COPY . .                   # Layer 5 — Copy source code (changes often!)
RUN npm run build          # Layer 6 — Build the app
```

**Key insight:** Docker caches layers **from top to bottom**. The moment it finds a layer that changed, it **invalidates all layers below it**.

```
Layer 1: FROM node:18        → Cached ✅  (base image didn't change)
Layer 2: WORKDIR /app        → Cached ✅  (instruction didn't change)
Layer 3: COPY package*.json  → Cached ✅  (package.json didn't change)
Layer 4: RUN npm ci          → Cached ✅  (dependencies didn't change)
Layer 5: COPY . .            → MISS ❌    (you changed a source file!)
Layer 6: RUN npm run build   → MISS ❌    (must rebuild — layer above changed)
```

### The Golden Rule for Dockerfiles

> **Put things that change LEAST at the TOP, and things that change MOST at the BOTTOM.**

### Bad Example (No Layer Caching Benefit)

```dockerfile
FROM node:18
WORKDIR /app
COPY . .           # ❌ Copies EVERYTHING first — any file change busts the cache
RUN npm ci         # ❌ Reinstalls dependencies every time a source file changes
RUN npm run build
```

Here, even if you only changed a comment in one file, Docker will re-install all npm packages. Terrible!

### Good Example (Layer Caching Optimized)

```dockerfile
FROM node:18-alpine
WORKDIR /app
COPY package.json package-lock.json ./    # ✅ Copy dependency files first
RUN npm ci                                 # ✅ Cached until dependencies change
COPY . .                                   # ✅ Source code changes don't affect above
RUN npm run build
```

Now, if you only change source code, layers 1-4 stay cached. Only layers 5-6 rebuild. **This can save minutes on every build!**

### .dockerignore — Reduce Build Context

Every time Docker builds, it sends the **entire build context** (the directory) to the Docker daemon. If your project has `node_modules/`, `.git/`, or large files, this slows things down massively.

```dockerignore
# .dockerignore
node_modules
.git
.gitignore
*.md
.env
.env.*
dist
coverage
.vscode
.idea
__pycache__
*.pyc
```

**Impact:** Sending a 500 MB build context vs a 5 MB context — the difference is enormous!

### Using BuildKit Cache Mounts

Docker BuildKit (the modern builder) supports **cache mounts** — these let you cache package manager directories directly in the build process:

```dockerfile
# syntax=docker/dockerfile:1
FROM node:18-alpine
WORKDIR /app

COPY package.json package-lock.json ./

# Mount npm cache as a separate cache layer
RUN --mount=type=cache,target=/root/.npm \
    npm ci

COPY . .
RUN npm run build
```

```dockerfile
# Python example with pip cache mount
FROM python:3.12-slim
WORKDIR /app

COPY requirements.txt .

RUN --mount=type=cache,target=/root/.cache/pip \
    pip install -r requirements.txt

COPY . .
```

**What `--mount=type=cache` does:**
- Creates a **persistent cache** that survives across builds.
- The `target` is the directory inside the container where the cache lives.
- This cache is **not included in the final image** — it's purely for speeding up builds.
- Works like a sticky note pad that stays on your desk between work sessions.

### Enable BuildKit

```bash
# Set environment variable to enable BuildKit
export DOCKER_BUILDKIT=1

# Or use Docker Buildx explicitly
docker buildx build -t myapp .
```

## 8. Incremental Builds

**Incremental builds** means only rebuilding the parts that actually changed, instead of rebuilding everything from scratch.

### How Incremental Builds Work

```
Full Rebuild:
  Compile File A  →  Compile File B  →  Compile File C  →  Link All
  (even if only File B changed, you redo everything)

Incremental Build:
  File A unchanged  →  Skip ✅
  File B changed    →  Recompile ✅
  File C unchanged  →  Skip ✅
  Link only changed parts  →  Done!
```

### Incremental Builds by Tool

#### Make (C/C++)

`make` is inherently incremental. It checks file timestamps to decide what to rebuild.

```makefile
# Makefile
CC = gcc
CFLAGS = -Wall -O2

# Object files
OBJS = main.o utils.o network.o

# Final binary depends on all objects
app: $(OBJS)
	$(CC) -o app $(OBJS)

# Each .o depends on its .c file
# make only recompiles .o files whose .c file changed
%.o: %.c
	$(CC) $(CFLAGS) -c $< -o $@

clean:
	rm -f app $(OBJS)
```

**How it works:**
- `make` checks: Is `main.c` newer than `main.o`?
  - **YES** → Recompile `main.o`
  - **NO** → Skip it
- This simple timestamp comparison gives you incremental builds for free!

#### Gradle (Java/Kotlin)

Gradle has built-in incremental compilation:

```groovy
// build.gradle
tasks.withType(JavaCompile) {
    options.incremental = true  // Enable incremental compilation
}
```

```bash
# First build — compiles everything
./gradlew build
# Build time: 45 seconds

# Change one file, build again — only recompiles what changed
./gradlew build
# Build time: 3 seconds  ← WOW!
```

#### Webpack (JavaScript)

```javascript
// webpack.config.js
module.exports = {
  // Enable caching for faster rebuilds
  cache: {
    type: 'filesystem',          // Persist cache to disk
    buildDependencies: {
      config: [__filename],      // Invalidate cache when config changes
    },
  },
  
  // Watch mode — rebuilds only what changes
  watch: true,
  watchOptions: {
    ignored: /node_modules/,     // Don't watch node_modules
    aggregateTimeout: 300,       // Wait 300ms after last change before rebuilding
  },
};
```

**What each option does:**
- `cache.type: 'filesystem'` — Saves the compilation cache to disk. Without this, the cache only lives in memory and is lost when webpack exits.
- `buildDependencies.config` — If you change your webpack config, the cache is invalidated (because the build rules changed).
- `watch: true` — webpack stays running and watches for file changes. When something changes, it only rebuilds the affected modules.

#### TypeScript (tsc)

```json
// tsconfig.json
{
  "compilerOptions": {
    "incremental": true,           // Enable incremental compilation
    "tsBuildInfoFile": "./.tsbuildinfo"  // Where to store build info
  }
}
```

```bash
# First compile
tsc
# Creates .tsbuildinfo file with build state

# Second compile (after small change)
tsc    # Only recompiles changed files — MUCH faster!
```

The `.tsbuildinfo` file stores information about what was compiled and their hashes, so TypeScript can skip files that haven't changed.

## 9. Parallel Builds

**Parallel builds** run independent tasks at the same time instead of one after another.

### Analogy

Think of a restaurant kitchen:
- **Sequential:** One chef makes appetizer → then entrée → then dessert → takes 3 hours
- **Parallel:** Three chefs work on appetizer, entrée, and dessert at the same time → takes 1 hour

### Parallel Builds by Tool

#### Make

```bash
# Run make with 4 parallel jobs
make -j4

# Use all available CPU cores
make -j$(nproc)
```

```makefile
# Makefile that benefits from parallelism
# These three targets are independent — they can build at the same time
frontend: 
	cd frontend && npm run build

backend:
	cd backend && mvn package

docs:
	cd docs && mkdocs build

# 'all' depends on all three — make runs them in parallel with -j
all: frontend backend docs
```

#### Gradle

```bash
# Enable parallel execution
./gradlew build --parallel

# Or set it permanently in gradle.properties
```

```properties
# gradle.properties
org.gradle.parallel=true
org.gradle.workers.max=4
org.gradle.caching=true
```

#### npm / Node.js

```bash
# Run multiple npm scripts in parallel using concurrently
npx concurrently "npm:build:css" "npm:build:js" "npm:build:images"
```

```json
// package.json
{
  "scripts": {
    "build:css": "postcss src/css -o dist/styles.css",
    "build:js": "webpack --mode production",
    "build:images": "imagemin src/images/* --out-dir=dist/images",
    "build": "concurrently \"npm:build:css\" \"npm:build:js\" \"npm:build:images\""
  }
}
```

#### CI/CD Pipeline Parallelism (GitHub Actions)

```yaml
# Run independent jobs in parallel
name: Build and Test

on: push

jobs:
  # These three jobs run IN PARALLEL by default!
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run lint

  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test

  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run build

  # This job runs AFTER all three above finish
  deploy:
    runs-on: ubuntu-latest
    needs: [lint, test, build]    # Wait for all three
    steps:
      - run: echo "Deploying..."
```

**How it works:**
- `lint`, `test`, and `build` are independent → GitHub Actions runs them **at the same time**.
- `deploy` has `needs: [lint, test, build]` → it waits until all three succeed before starting.
- This can easily cut your pipeline time by 50-70%!

#### GitLab CI Parallelism

```yaml
# .gitlab-ci.yml
stages:
  - test
  - build
  - deploy

# Jobs in the same stage run in parallel!
unit-tests:
  stage: test
  script: npm test

integration-tests:
  stage: test
  script: npm run test:integration

lint:
  stage: test
  script: npm run lint

# This runs after ALL test stage jobs complete
build-app:
  stage: build
  script: npm run build
```

## 10. Build Caching in CI/CD Pipelines

CI/CD pipelines are where caching has the **biggest impact** because every pipeline run starts with a fresh, clean environment. Without caching, EVERY run downloads EVERYTHING from scratch.

### GitHub Actions — Full Caching Example

```yaml
name: Optimized Build

on: push

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      # ---- Step 1: Cache Node.js modules ----
      - name: Cache node_modules
        id: cache-node
        uses: actions/cache@v4
        with:
          path: node_modules
          key: node-${{ runner.os }}-${{ hashFiles('package-lock.json') }}

      # ---- Step 2: Install only if cache missed ----
      - name: Install dependencies
        if: steps.cache-node.outputs.cache-hit != 'true'
        run: npm ci

      # ---- Step 3: Cache build output ----
      - name: Cache build output
        uses: actions/cache@v4
        with:
          path: dist
          key: build-${{ runner.os }}-${{ hashFiles('src/**') }}

      # ---- Step 4: Build only if source changed ----
      - name: Build
        if: steps.cache-build.outputs.cache-hit != 'true'
        run: npm run build

      # ---- Step 5: Run tests ----
      - name: Test
        run: npm test
```

**What's happening here:**
1. **Cache `node_modules`** using `package-lock.json` as the key. If dependencies haven't changed → skip `npm ci` entirely.
2. **Conditional install** — The `if` statement checks if the cache was hit. Only runs `npm ci` when the cache misses.
3. **Cache build output** using a hash of source files. If no source files changed → skip the build entirely!
4. **Tests always run** — We don't cache test results because we always want to verify.

### GitLab CI — Full Caching Example

```yaml
# .gitlab-ci.yml
variables:
  npm_config_cache: "$CI_PROJECT_DIR/.npm-cache"

cache:
  key:
    files:
      - package-lock.json   # Cache key based on lock file
  paths:
    - .npm-cache/            # What to cache
    - node_modules/

stages:
  - install
  - build
  - test

install:
  stage: install
  script:
    - npm ci --cache .npm-cache
  artifacts:
    paths:
      - node_modules/
    expire_in: 1 hour

build:
  stage: build
  script:
    - npm run build
  artifacts:
    paths:
      - dist/
    expire_in: 1 day

test:
  stage: test
  script:
    - npm test
```

### Jenkins Pipeline — Caching Example

```groovy
// Jenkinsfile
pipeline {
    agent any

    stages {
        stage('Cache & Install') {
            steps {
                // Jenkins doesn't have built-in caching like GitHub Actions
                // Use stash/unstash or a shared volume
                script {
                    // Check if node_modules exists from a previous build
                    def lockHash = sh(
                        script: 'md5sum package-lock.json | cut -d" " -f1',
                        returnStdout: true
                    ).trim()
                    
                    def cacheDir = "/tmp/npm-cache/${lockHash}"
                    
                    if (fileExists(cacheDir)) {
                        sh "cp -r ${cacheDir}/node_modules ."
                        echo "Cache hit! Restored node_modules"
                    } else {
                        sh 'npm ci'
                        sh "mkdir -p ${cacheDir} && cp -r node_modules ${cacheDir}/"
                        echo "Cache miss. Installed fresh and cached."
                    }
                }
            }
        }

        stage('Build') {
            steps {
                sh 'npm run build'
            }
        }

        stage('Test') {
            steps {
                sh 'npm test'
            }
        }
    }
}
```

## 11. Build Optimization by Tool / Ecosystem

### Node.js / JavaScript Optimization

```bash
# Use npm ci instead of npm install (faster, deterministic)
npm ci

# Use pnpm — it's MUCH faster due to content-addressable storage
pnpm install     # Symlinks packages instead of copying — saves disk and time!

# Yarn with Plug'n'Play (PnP) — zero installs
yarn set version berry
yarn install     # With PnP, node_modules isn't even created!
```

**Why `npm ci` is better than `npm install` for CI:**

| Feature | `npm install` | `npm ci` |
|---------|--------------|----------|
| Reads from | `package.json` | `package-lock.json` |
| Can modify lock file | Yes | No |
| Removes `node_modules` first | No | Yes |
| Deterministic | Not always | Always |
| Speed in CI | Slower | Faster |

### Python Optimization

```bash
# Use pip with cache
pip install --cache-dir .pip-cache -r requirements.txt

# Use pip-tools for deterministic builds
pip-compile requirements.in         # Generate locked requirements.txt
pip-sync requirements.txt           # Install exactly what's specified

# Use uv — a blazing fast Python package installer (written in Rust)
uv pip install -r requirements.txt  # 10-100x faster than pip!
```

### Java / Maven Optimization

```xml
<!-- pom.xml — Skip tests during build for faster feedback -->
<!-- Use: mvn package -DskipTests -->

<!-- Enable parallel module builds -->
<!-- Use: mvn -T 4 package (4 threads) -->
<!-- Use: mvn -T 1C package (1 thread per CPU core) -->
```

```bash
# Maven with parallel threads
mvn -T 1C clean package    # 1 thread per CPU core

# Skip tests when you just need to build
mvn package -DskipTests

# Offline mode — use only cached dependencies
mvn -o package    # Doesn't check for updates online
```

### Go Optimization

```bash
# Go has built-in build caching — it's automatic!
go build ./...       # First build — full compilation
go build ./...       # Second build — uses cache (almost instant if nothing changed)

# See cache stats
go env GOCACHE       # Shows cache directory

# The Go build cache caches individual package compilations
# It checks file modification times to decide what to rebuild
```

## 12. Artifact Caching & Reuse

Sometimes you don't need to rebuild at all — you can **reuse artifacts** from a previous build.

### What is an Artifact?

An artifact is the **output of a build** — the thing you deploy:
- A `.jar` file (Java)
- A `dist/` folder (JavaScript)
- A Docker image
- A binary executable (Go, Rust)

### Strategy: Don't Rebuild if Source Hasn't Changed

```yaml
# GitHub Actions — Upload and reuse artifacts
name: Build and Deploy

on: push

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run build
      
      # Save the build output as an artifact
      - uses: actions/upload-artifact@v4
        with:
          name: dist
          path: dist/
          retention-days: 5

  deploy:
    needs: build
    runs-on: ubuntu-latest
    steps:
      # Download the artifact from the build job
      - uses: actions/download-artifact@v4
        with:
          name: dist
          path: dist/
      
      - run: echo "Deploying from dist/"
```

**Why this matters:** The `deploy` job doesn't need to rebuild the app. It just downloads the pre-built artifact. This keeps the deploy job fast and ensures you deploy exactly what you tested.

## 13. Remote / Distributed Build Caches

For larger teams, a **remote build cache** lets everyone share cached build results.

### The Idea

```
Developer A builds module-auth → stores result in remote cache
Developer B needs module-auth  → downloads from remote cache (instant!)
CI Server builds module-auth   → also downloads from cache (instant!)
```

Everyone benefits from each other's builds!

### Tools for Remote Build Caching

| Tool | Ecosystem | Description |
|------|-----------|-------------|
| **Nx Cloud** | JavaScript/TypeScript | Remote caching + distributed task execution for Nx monorepos |
| **Turborepo** | JavaScript/TypeScript | Remote caching for monorepo builds |
| **Gradle Build Cache** | Java/Kotlin | Native remote cache support |
| **Bazel Remote Cache** | Polyglot | Google's build tool with powerful remote caching |
| **ccache** | C/C++ | Compiler cache for GCC/Clang |
| **sccache** | C/C++/Rust | Shared compiler cache (supports S3, GCS, Azure) |

### Example: Turborepo Remote Caching

```json
// turbo.json
{
  "pipeline": {
    "build": {
      "outputs": ["dist/**", ".next/**"],
      "dependsOn": ["^build"]
    },
    "test": {
      "dependsOn": ["build"],
      "outputs": []
    },
    "lint": {
      "outputs": []
    }
  }
}
```

```bash
# Enable remote caching
npx turbo login            # Authenticate with Vercel
npx turbo link             # Link your repo to remote cache

# Now builds are cached remotely
npx turbo run build        # First run — caches results remotely
# Another developer runs the same build:
npx turbo run build        # Cache HIT — downloads result instantly!
```

### Example: Gradle Remote Build Cache

```groovy
// settings.gradle
buildCache {
    local {
        enabled = true                      // Local cache (on-disk)
    }
    remote(HttpBuildCache) {
        enabled = true
        url = 'https://cache.mycompany.com/cache/'
        push = true                          // Allow pushing to remote cache
        credentials {
            username = System.getenv('CACHE_USER')
            password = System.getenv('CACHE_PASS')
        }
    }
}
```

```bash
# Build with remote cache
./gradlew build --build-cache
```

## 14. Anti-Patterns — Common Mistakes

Learn from others' mistakes! Here are the most common build optimization anti-patterns:

### ❌ Anti-Pattern 1: Caching Too Aggressively

```yaml
# BAD — Caching test results
- name: Cache test results
  uses: actions/cache@v4
  with:
    path: test-results/
    key: tests-${{ hashFiles('src/**') }}

# Why it's bad: Tests should ALWAYS run. 
# A cached test pass doesn't mean the code works now.
```

### ❌ Anti-Pattern 2: Wrong Cache Key

```yaml
# BAD — Using a static key
- uses: actions/cache@v4
  with:
    path: node_modules
    key: my-cache        # ❌ Never changes! You'll use stale dependencies forever!

# GOOD — Key based on lock file hash
- uses: actions/cache@v4
  with:
    path: node_modules
    key: node-${{ hashFiles('package-lock.json') }}  # ✅ Changes when deps change
```

### ❌ Anti-Pattern 3: Not Using .dockerignore

```
Without .dockerignore:
  Build context: 800 MB (includes node_modules, .git, everything)
  Upload time: 45 seconds

With .dockerignore:
  Build context: 2 MB (only source code)
  Upload time: 0.5 seconds
```

### ❌ Anti-Pattern 4: Installing Dev Dependencies in Production

```dockerfile
# BAD
RUN npm install   # Installs ALL dependencies including dev ones

# GOOD
RUN npm ci --only=production   # Only production dependencies
# or for newer npm:
RUN npm ci --omit=dev
```

### ❌ Anti-Pattern 5: Not Ordering Dockerfile Layers Correctly

```dockerfile
# BAD — Copying everything first busts dependency cache
COPY . .
RUN npm ci

# GOOD — Copy dependency files first, then source code
COPY package*.json ./
RUN npm ci
COPY . .
```

## 15. Real-World Scenarios

### Scenario 1: "Our CI builds take 20 minutes"

**Before:**
```yaml
steps:
  - checkout
  - run: npm install          # 8 minutes
  - run: npm run build        # 5 minutes
  - run: npm test             # 4 minutes
  - run: docker build .       # 3 minutes
# Total: 20 minutes
```

**After optimization:**
```yaml
steps:
  - checkout
  
  # Dependency caching
  - cache: node_modules (key: package-lock.json hash)
  - run: npm ci                # 30 seconds (cache hit!)
  
  # Parallel builds
  - parallel:
    - run: npm run build       # 5 minutes ─┐
    - run: npm test            # 4 minutes ─┤ Run at same time!
  # Parallel time: ~5 minutes (not 9!)       │
  
  # Docker with layer caching + BuildKit
  - run: docker buildx build --cache-from=type=gha --cache-to=type=gha .
  # Docker build: 45 seconds (layer cache hit!)

# Total: ~6.5 minutes (from 20 minutes — 67% faster!)
```

### Scenario 2: "Every developer waits 10 minutes to build locally"

**Solution: Remote build cache with Turborepo/Nx**

```bash
# Developer A builds feature-x at 9:00 AM
turbo run build    # Takes 10 minutes, caches result remotely

# Developer B pulls feature-x at 9:30 AM
turbo run build    # Cache HIT! Takes 15 seconds!

# CI runs at 10:00 AM
turbo run build    # Cache HIT! Takes 15 seconds!
```

## 16. Summary & Cheat Sheet

### Quick Reference — What to Cache Where

| What | Cache Location | Cache Key | When to Invalidate |
|------|---------------|-----------|---------------------|
| npm packages | `~/.npm` or `node_modules` | `package-lock.json` hash | When dependencies change |
| pip packages | `~/.cache/pip` | `requirements.txt` hash | When dependencies change |
| Maven packages | `~/.m2/repository` | `pom.xml` hash | When dependencies change |
| Docker layers | Docker daemon | Layer content hash | When Dockerfile/source changes |
| Build output | `dist/`, `build/` | Source code hash | When source changes |
| Compiled objects | Tool-specific | Source file timestamps | When source changes |

### The Build Optimization Checklist

```
□ Are you caching dependencies?
□ Is your Dockerfile ordered correctly (least → most changing)?
□ Do you have a .dockerignore file?
□ Are you using parallel builds where possible?
□ Is incremental compilation enabled?
□ Are CI/CD jobs running in parallel when independent?
□ Are you using npm ci instead of npm install in CI?
□ Are you using BuildKit for Docker builds?
□ Have you considered a remote build cache for your team?
□ Are you avoiding caching things that shouldn't be cached (like test results)?
```

### Impact Matrix

```
HIGH IMPACT, LOW EFFORT:
  ✅ Dependency caching
  ✅ .dockerignore
  ✅ Correct Dockerfile layer order
  ✅ npm ci instead of npm install

HIGH IMPACT, MEDIUM EFFORT:
  ✅ CI/CD job parallelism
  ✅ BuildKit cache mounts
  ✅ Incremental compilation

HIGH IMPACT, HIGH EFFORT:
  ✅ Remote / distributed build cache
  ✅ Monorepo build orchestration (Nx, Turborepo, Bazel)
```

> **Key Takeaway:** The fastest build is the one that doesn't have to run. Cache everything that's expensive and hasn't changed. Parallelize everything that's independent. And always measure before and after — you can't improve what you don't measure! 🚀
