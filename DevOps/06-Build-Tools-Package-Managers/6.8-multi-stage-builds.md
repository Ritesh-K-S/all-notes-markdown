## 1. The Problem: Fat Docker Images

Let's start with **why** multi-stage builds exist. Look at this common scenario:

### Single-Stage Build (The Old Way)

```dockerfile
FROM node:18
WORKDIR /app
COPY . .
RUN npm ci
RUN npm run build
CMD ["node", "dist/server.js"]
```

Seems fine, right? But look at what ends up in your final image:

```
Final Image Contains:
├── Node.js runtime (~300 MB)
├── npm (~50 MB)
├── node_modules/ (~400 MB including devDependencies!)
│   ├── webpack (only needed for building)
│   ├── typescript (only needed for building)
│   ├── eslint (only needed for development)
│   ├── jest (only needed for testing)
│   └── ... hundreds of dev packages
├── src/ (source files — not needed at runtime!)
├── tsconfig.json (not needed at runtime!)
├── package.json
└── dist/ (this is all we actually need!)

Total image size: ~900 MB 😱
```

You're shipping a **moving truck** when all you need is a **backpack**.

### Why Fat Images Are Bad

| Problem | Impact |
|---------|--------|
| **Slow deployments** | Pulling 900 MB vs 50 MB — massive difference in deploy time |
| **Wasted storage** | Each version takes 900 MB in your registry |
| **Larger attack surface** | More tools = more potential vulnerabilities |
| **Slow CI/CD** | Bigger images take longer to build, push, and pull |
| **Higher costs** | More bandwidth, more storage, more compute |

## 2. What Are Multi-Stage Builds?

**Multi-stage builds** let you use multiple `FROM` statements in a single Dockerfile. Each `FROM` starts a new **stage**. You can **copy artifacts** from one stage to another, leaving behind everything you don't need.

### The Analogy

Think of it like a factory:

```
🏭 Stage 1: THE FACTORY (Build Stage)
   - Has all the heavy machinery (compilers, build tools)
   - Takes raw materials (source code)
   - Produces the finished product (compiled app)
   - Is big, messy, and full of tools

📦 Stage 2: THE SHIPPING BOX (Production Stage)
   - Only contains the finished product
   - Small, clean, minimal
   - Ready to be shipped to the customer (deployed)
```

You **don't ship the entire factory** to the customer. You only ship the product. That's multi-stage builds!

### Visual Diagram

```
┌──────────────────────────────────────────────────────┐
│  Dockerfile                                          │
│                                                      │
│  ┌──────────────────────────────────────────────┐    │
│  │  Stage 1: BUILD                               │    │
│  │  FROM node:18 AS builder                      │    │
│  │  ├── Install ALL dependencies                 │    │
│  │  ├── Copy ALL source code                     │    │
│  │  ├── Run build command                        │    │
│  │  └── Output: dist/ folder                     │    │
│  │  Size: ~900 MB                                │    │
│  └─────────────────────┬────────────────────────┘    │
│                        │                              │
│                   COPY dist/                          │
│                   (only the output!)                  │
│                        │                              │
│  ┌─────────────────────▼────────────────────────┐    │
│  │  Stage 2: PRODUCTION                          │    │
│  │  FROM node:18-alpine                          │    │
│  │  ├── Install ONLY production dependencies     │    │
│  │  ├── Copy dist/ from Stage 1                  │    │
│  │  └── Run the app                              │    │
│  │  Size: ~80 MB  ✅                             │    │
│  └──────────────────────────────────────────────┘    │
│                                                      │
│  Final image = Stage 2 ONLY                          │
└──────────────────────────────────────────────────────┘
```

## 3. Your First Multi-Stage Build

Let's start with the simplest possible example:

```dockerfile
# ============================================
# Stage 1: Build
# ============================================
FROM node:18 AS builder
# "AS builder" gives this stage a name we can reference later

WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build
# At this point, we have our compiled app in /app/dist/

# ============================================
# Stage 2: Production
# ============================================
FROM node:18-alpine
# This starts a COMPLETELY NEW image — nothing from Stage 1 carries over

WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev    
# Only install production dependencies (no dev tools!)

# Copy ONLY the build output from Stage 1
COPY --from=builder /app/dist ./dist

# Run the app
CMD ["node", "dist/server.js"]
```

### What `COPY --from=builder` Does

This is the **magic line**. Let's break it down:

```
COPY --from=builder /app/dist ./dist
       │               │        │
       │               │        └── Where to put it in the NEW stage
       │               └── What to copy FROM the builder stage
       └── Which stage to copy from (the name we gave with AS)
```

- `--from=builder` — Reach back into Stage 1 (named "builder") and grab files from it.
- `/app/dist` — The path inside Stage 1 where our compiled app lives.
- `./dist` — Where to put it in the current (production) stage.

### Size Comparison

```
Single-stage image:  ~900 MB
Multi-stage image:   ~80 MB   ← 91% smaller! 🎉
```

## 4. How Multi-Stage Builds Work — Step by Step

Let's trace exactly what Docker does when it encounters a multi-stage Dockerfile:

```
Step 1: Docker reads the entire Dockerfile

Step 2: It sees the first FROM → creates Stage 1
   - Pulls node:18 image
   - Runs all instructions in Stage 1
   - Stores the result as a temporary image (call it "temp-1")

Step 3: It sees the second FROM → creates Stage 2
   - Pulls node:18-alpine image (fresh start!)
   - Runs instructions in Stage 2
   - When it hits COPY --from=builder:
     - Reaches into temp-1
     - Copies the specified files
     - Pastes them into the current stage

Step 4: Final image = Stage 2 ONLY
   - temp-1 is discarded (not part of the final image!)
   - Only Stage 2 layers become the final image
   - All the build tools, source code, dev dependencies from Stage 1? GONE.
```

## 5. Multi-Stage Build for Node.js

### Express.js / Backend API

```dockerfile
# ============================================
# Stage 1: Build the TypeScript application
# ============================================
FROM node:20-alpine AS builder

WORKDIR /app

# Copy dependency files first (for better caching)
COPY package.json package-lock.json ./

# Install ALL dependencies (including devDependencies like TypeScript)
RUN npm ci

# Copy source code
COPY tsconfig.json ./
COPY src/ ./src/

# Compile TypeScript to JavaScript
RUN npm run build
# This creates the dist/ folder with compiled .js files

# ============================================
# Stage 2: Production image
# ============================================
FROM node:20-alpine

# Create a non-root user for security
RUN addgroup -S appgroup && adduser -S appuser -G appgroup

WORKDIR /app

# Copy dependency files
COPY package.json package-lock.json ./

# Install ONLY production dependencies
RUN npm ci --omit=dev

# Copy compiled code from builder stage
COPY --from=builder /app/dist ./dist

# Switch to non-root user
USER appuser

# Expose port and start
EXPOSE 3000
CMD ["node", "dist/server.js"]
```

**What's happening and why:**

1. **Stage 1 (`builder`)** — We install TypeScript, all dev tools, compile the code. This stage is ~500 MB.
2. **Stage 2 (production)** — We start fresh with a clean Alpine image. We install only production dependencies. We copy only the compiled JavaScript. No TypeScript compiler, no dev tools, no source `.ts` files.
3. **Non-root user** — For security! Never run containers as root in production.
4. **Result:** A tiny, secure image with only what's needed to run the app.

```
Builder stage:     ~500 MB (has TypeScript, webpack, all dev tools)
Production image:  ~80 MB  (just Node.js runtime + compiled code + prod deps)
```

## 6. Multi-Stage Build for Python

### Flask / FastAPI Application

```dockerfile
# ============================================
# Stage 1: Build dependencies
# ============================================
FROM python:3.12-slim AS builder

WORKDIR /app

# Install build tools needed for some Python packages
RUN apt-get update && apt-get install -y --no-install-recommends \
    gcc \
    libpq-dev \
    && rm -rf /var/lib/apt/lists/*

# Create a virtual environment
RUN python -m venv /opt/venv

# Make sure we use the venv
ENV PATH="/opt/venv/bin:$PATH"

# Install dependencies
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# ============================================
# Stage 2: Production image
# ============================================
FROM python:3.12-slim

# Install only runtime libraries (not build tools!)
RUN apt-get update && apt-get install -y --no-install-recommends \
    libpq5 \
    && rm -rf /var/lib/apt/lists/*

# Create non-root user
RUN useradd --create-home appuser

# Copy the virtual environment from the builder
COPY --from=builder /opt/venv /opt/venv

# Make sure we use the venv
ENV PATH="/opt/venv/bin:$PATH"

WORKDIR /app

# Copy application code
COPY . .

# Switch to non-root user
USER appuser

EXPOSE 8000
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
```

**Why we copy the entire `/opt/venv`:**

Python's `pip` installs packages into a **virtual environment**. By copying the whole venv, we get all installed packages without needing `pip` or `gcc` in the production image.

```
Key insight:
  Stage 1 has: gcc, libpq-dev (build tools — needed to COMPILE packages)
  Stage 2 has: libpq5 (runtime library — needed to USE packages)
  
  gcc is ~100 MB. We don't need it at runtime. Multi-stage saves us that!
```

**Another insight — `libpq-dev` vs `libpq5`:**
- `libpq-dev` = development version (includes header files for compiling)
- `libpq5` = runtime version (just the library for connecting to PostgreSQL)
- We only need the runtime version in production!

## 7. Multi-Stage Build for Go

Go is where multi-stage builds **really shine** because Go compiles to a **single static binary** — meaning the production image doesn't even need Go installed!

```dockerfile
# ============================================
# Stage 1: Build the Go binary
# ============================================
FROM golang:1.22-alpine AS builder

WORKDIR /app

# Copy dependency files first (for caching)
COPY go.mod go.sum ./
RUN go mod download

# Copy source code
COPY . .

# Build a static binary
# CGO_ENABLED=0 → don't link against C libraries (fully static)
# -ldflags="-s -w" → strip debug info (smaller binary)
RUN CGO_ENABLED=0 GOOS=linux go build \
    -ldflags="-s -w" \
    -o /app/server \
    ./cmd/server/

# ============================================
# Stage 2: Minimal production image
# ============================================
FROM scratch
# "scratch" is the EMPTY image — literally nothing!
# No OS, no shell, no tools — just the Linux kernel interface

# Copy SSL certificates (needed for HTTPS calls)
COPY --from=builder /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/

# Copy the compiled binary
COPY --from=builder /app/server /server

# Expose port
EXPOSE 8080

# Run the binary
ENTRYPOINT ["/server"]
```

**Mind-blowing results:**

```
Go build stage:      ~800 MB (Go compiler, tools, source code, dependencies)
Production image:    ~8 MB   (just the compiled binary!) 🤯

That's a 99% reduction in image size!
```

**Line-by-line explanation:**

1. `FROM golang:1.22-alpine AS builder` — Use the Go compiler image. Alpine variant is smaller.

2. `COPY go.mod go.sum ./` + `RUN go mod download` — Download dependencies first (caching layer!). If dependencies don't change, this layer is cached.

3. `CGO_ENABLED=0` — Disable CGo (C bindings). This creates a **fully static binary** that doesn't depend on any system libraries. This is what allows us to use `scratch` as the base.

4. `GOOS=linux` — Target Linux (since containers run Linux).

5. `-ldflags="-s -w"` — Strip debug symbols (`-s`) and DWARF info (`-w`). This reduces binary size by 20-30%.

6. `FROM scratch` — The **empty image**. Literally nothing. No OS, no shell, no package manager. Just a blank canvas. This is the most secure and smallest possible base.

7. `COPY --from=builder /etc/ssl/certs/...` — If your Go app makes HTTPS requests, it needs SSL certificates. We copy them from the builder stage.

### Alternative: Using `distroless` Instead of `scratch`

`scratch` is extreme — no shell means you can't exec into the container for debugging. A good middle ground is Google's **distroless** images:

```dockerfile
FROM gcr.io/distroless/static-debian12

COPY --from=builder /app/server /server

EXPOSE 8080
ENTRYPOINT ["/server"]
```

**Distroless images:**
- Include essential things like SSL certs and timezone data
- No shell, no package manager (minimal attack surface)
- Slightly larger than `scratch` (~2 MB) but more practical
- Google's recommended approach for production containers

## 8. Multi-Stage Build for Java (Maven)

```dockerfile
# ============================================
# Stage 1: Build with Maven
# ============================================
FROM maven:3.9-eclipse-temurin-21 AS builder

WORKDIR /app

# Copy dependency files first (caching!)
COPY pom.xml .

# Download dependencies (cached if pom.xml doesn't change)
RUN mvn dependency:go-offline -B

# Copy source code
COPY src/ ./src/

# Build the JAR (skip tests for faster builds; run tests separately)
RUN mvn package -DskipTests -B

# ============================================
# Stage 2: Production with JRE only
# ============================================
FROM eclipse-temurin:21-jre-alpine
# Notice: JRE not JDK!
# JDK (~400 MB) has compiler + tools we don't need at runtime
# JRE (~100 MB) has just the runtime

# Create non-root user
RUN addgroup -S appgroup && adduser -S appuser -G appgroup

WORKDIR /app

# Copy the built JAR from Stage 1
COPY --from=builder /app/target/*.jar app.jar

# Switch to non-root user
USER appuser

EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]
```

**Key decision: JDK vs JRE**

```
JDK (Java Development Kit): ~400 MB
  ├── Java compiler (javac)
  ├── Build tools
  ├── Debugger
  └── JRE (runtime)

JRE (Java Runtime Environment): ~100 MB
  └── Just what's needed to RUN Java apps

In production, we only need JRE! 
→ Stage 1 uses JDK (to compile)
→ Stage 2 uses JRE (to run)
```

**Size comparison:**

```
Single-stage (with JDK + Maven):  ~800 MB
Multi-stage (with JRE only):      ~150 MB  ← 81% smaller!
```

### With Custom JRE (jlink)

For even smaller images, Java 9+ lets you create a **custom JRE** with only the modules your app needs:

```dockerfile
# Stage 1: Build
FROM maven:3.9-eclipse-temurin-21 AS builder
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline -B
COPY src/ ./src/
RUN mvn package -DskipTests -B

# Stage 2: Create custom JRE
FROM eclipse-temurin:21 AS jre-builder
COPY --from=builder /app/target/*.jar /app/app.jar

# Find which Java modules the app needs
RUN jar xf /app/app.jar && \
    jdeps --ignore-missing-deps --multi-release 21 \
    --print-module-deps /app/app.jar > /app/deps.txt

# Create a minimal custom JRE with only needed modules
RUN jlink \
    --add-modules $(cat /app/deps.txt) \
    --strip-debug \
    --compress=2 \
    --no-header-files \
    --no-man-pages \
    --output /custom-jre

# Stage 3: Production with custom JRE
FROM debian:bookworm-slim
COPY --from=jre-builder /custom-jre /opt/java
COPY --from=builder /app/target/*.jar /app/app.jar

ENV PATH="/opt/java/bin:$PATH"
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "/app/app.jar"]
```

This is a **3-stage build** — yes, you can have as many stages as you want!

```
Standard JRE image:   ~150 MB
Custom JRE image:      ~70 MB  ← Even smaller!
```

## 9. Multi-Stage Build for React / Frontend

Frontend apps are a **perfect** use case for multi-stage builds because:
- You need Node.js to **build** the app (compile JSX, bundle, minify)
- You only need a **web server** (like Nginx) to **serve** the static files

```dockerfile
# ============================================
# Stage 1: Build the React app
# ============================================
FROM node:20-alpine AS builder

WORKDIR /app

# Install dependencies (cached layer)
COPY package.json package-lock.json ./
RUN npm ci

# Copy source code and build
COPY . .
RUN npm run build
# This creates /app/dist/ (or /app/build/) with static HTML/CSS/JS files

# ============================================
# Stage 2: Serve with Nginx
# ============================================
FROM nginx:alpine

# Remove default Nginx content
RUN rm -rf /usr/share/nginx/html/*

# Copy built static files from Stage 1
COPY --from=builder /app/dist /usr/share/nginx/html

# Copy custom Nginx config (for SPA routing)
COPY nginx.conf /etc/nginx/conf.d/default.conf

EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```

**The corresponding Nginx config for a Single Page App (SPA):**

```nginx
# nginx.conf
server {
    listen 80;
    server_name _;

    root /usr/share/nginx/html;
    index index.html;

    # For SPA: send all routes to index.html
    location / {
        try_files $uri $uri/ /index.html;
    }

    # Cache static assets aggressively
    location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff|woff2)$ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }

    # Security headers
    add_header X-Frame-Options "SAMEORIGIN" always;
    add_header X-Content-Type-Options "nosniff" always;
}
```

**Size comparison:**

```
Single-stage (Node.js):    ~400 MB (includes Node.js, npm, node_modules)
Multi-stage (Nginx):        ~25 MB (just Nginx + static files) 🎉

And Nginx serves files WAY faster than a Node.js dev server!
```

## 10. Named Stages

We've been using named stages already (with `AS builder`), but let's formalize this:

```dockerfile
# You can name stages anything
FROM node:20 AS deps
# ... install dependencies

FROM node:20 AS builder
# ... build the app

FROM node:20 AS tester
# ... run tests

FROM node:20-alpine AS production
# ... final image
```

### Referencing Stages by Name vs Number

```dockerfile
# By name (preferred — more readable!)
COPY --from=builder /app/dist ./dist

# By number (0-indexed — works but less readable)
COPY --from=0 /app/dist ./dist    # Copies from the FIRST stage
COPY --from=1 /app/dist ./dist    # Copies from the SECOND stage
```

**Always use names!** Numbers break if you reorder stages.

## 11. Copying from External Images

You're not limited to copying from stages in your Dockerfile. You can copy from **any existing image**!

```dockerfile
FROM ubuntu:22.04

# Copy the kubectl binary from the official Kubernetes image
COPY --from=bitnami/kubectl:latest /opt/bitnami/kubectl/bin/kubectl /usr/local/bin/kubectl

# Copy the AWS CLI from the official AWS image
COPY --from=amazon/aws-cli:latest /usr/local/aws-cli /usr/local/aws-cli

# Copy a specific tool from another image
COPY --from=hashicorp/terraform:latest /bin/terraform /usr/local/bin/terraform
```

**Why this is useful:**
- No need to install tools manually (no `apt-get`, `curl`, etc.)
- You get the exact binary from the official image
- Cleaner, more reproducible builds

## 12. Building Specific Stages

You don't always need to build the entire Dockerfile. You can **stop at a specific stage**:

```bash
# Build ONLY the builder stage (useful for debugging)
docker build --target builder -t myapp:build-stage .

# Build ONLY up to the test stage
docker build --target tester -t myapp:test .

# Build the full production image (default — builds to the LAST stage)
docker build -t myapp:latest .
```

### Use Case: Different Images for Different Environments

```dockerfile
FROM node:20 AS base
WORKDIR /app
COPY package*.json ./
RUN npm ci

FROM base AS development
# Dev stage has hot reload, debug tools
COPY . .
CMD ["npm", "run", "dev"]

FROM base AS test
# Test stage runs tests
COPY . .
CMD ["npm", "test"]

FROM base AS builder
COPY . .
RUN npm run build

FROM node:20-alpine AS production
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
CMD ["node", "dist/server.js"]
```

```bash
# Build for development
docker build --target development -t myapp:dev .

# Build for testing
docker build --target test -t myapp:test .

# Build for production
docker build --target production -t myapp:prod .
# OR just:
docker build -t myapp:prod .    # Last stage is default
```

This is a great pattern! **One Dockerfile, multiple purposes.**

## 13. Multi-Stage + Build Arguments

Use `ARG` to make your multi-stage builds flexible:

```dockerfile
# Global ARG (available to all stages)
ARG NODE_VERSION=20

# ============================================
# Stage 1: Build
# ============================================
FROM node:${NODE_VERSION}-alpine AS builder
# ARG values from before FROM are available here

ARG BUILD_ENV=production
# This ARG is scoped to THIS stage

WORKDIR /app
COPY package*.json ./
RUN npm ci

COPY . .

# Use the build argument to control the build
RUN if [ "$BUILD_ENV" = "production" ]; then \
      npm run build:prod; \
    else \
      npm run build:dev; \
    fi

# ============================================
# Stage 2: Production
# ============================================
FROM node:${NODE_VERSION}-alpine

WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules

CMD ["node", "dist/server.js"]
```

```bash
# Build with defaults
docker build -t myapp .

# Build with specific Node version
docker build --build-arg NODE_VERSION=18 -t myapp .

# Build for staging
docker build --build-arg BUILD_ENV=staging -t myapp:staging .
```

**Important note about ARG and FROM:**
- `ARG` declared **before** the first `FROM` is available in `FROM` lines.
- `ARG` declared **inside** a stage is available only within that stage.
- To use a global `ARG` inside a stage, you need to redeclare it:

```dockerfile
ARG NODE_VERSION=20
FROM node:${NODE_VERSION} AS builder

# To use NODE_VERSION inside the stage, redeclare it:
ARG NODE_VERSION
RUN echo "Building with Node $NODE_VERSION"
```

## 14. Multi-Stage + Caching (Combining with 6.7)

This is where the magic of combining multi-stage builds with caching strategies from [6.7](6.7-build-optimization-and-caching-strategies.md) creates blazing fast builds.

### BuildKit Cache Mounts in Multi-Stage Builds

```dockerfile
# syntax=docker/dockerfile:1
FROM node:20-alpine AS builder

WORKDIR /app

COPY package.json package-lock.json ./

# Cache npm packages across builds!
RUN --mount=type=cache,target=/root/.npm \
    npm ci

COPY . .
RUN npm run build

# ============================================
FROM node:20-alpine

WORKDIR /app

COPY package.json package-lock.json ./

# Separate cache mount for production deps
RUN --mount=type=cache,target=/root/.npm \
    npm ci --omit=dev

COPY --from=builder /app/dist ./dist

CMD ["node", "dist/server.js"]
```

### GitHub Actions with Docker Layer Caching

```yaml
name: Build and Push

on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      # Set up Docker Buildx (required for advanced caching)
      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      # Login to Docker registry
      - name: Login to Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKER_USERNAME }}
          password: ${{ secrets.DOCKER_TOKEN }}

      # Build and push with GitHub Actions cache
      - name: Build and Push
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: myuser/myapp:latest
          # Cache Docker layers in GitHub Actions cache
          cache-from: type=gha
          cache-to: type=gha,mode=max
```

**What `cache-from` and `cache-to` do:**
- `cache-from: type=gha` — **Pull** cached Docker layers from GitHub Actions cache.
- `cache-to: type=gha,mode=max` — **Push** all layers to cache (not just the final stage).
- `mode=max` — Cache **all layers from all stages**, not just the final image. This means your builder stage layers are cached too!

### Registry-Based Caching

Instead of CI-specific cache, you can cache layers in your Docker registry:

```yaml
- name: Build and Push
  uses: docker/build-push-action@v5
  with:
    context: .
    push: true
    tags: myuser/myapp:latest
    # Use the registry itself as the cache!
    cache-from: type=registry,ref=myuser/myapp:cache
    cache-to: type=registry,ref=myuser/myapp:cache,mode=max
```

This is useful when multiple CI systems need to share the same cache.

## 15. Security Benefits

Multi-stage builds are a **security win** because they follow the principle of **least privilege** — only include what's necessary.

### Reduced Attack Surface

```
Single-stage image includes:
  ❌ gcc, make (compilers — could be used to compile exploits)
  ❌ wget, curl (download tools — could fetch malware)
  ❌ npm, pip (package managers — could install malicious packages)
  ❌ Source code (reveals your code structure)
  ❌ .env files, configs (might contain secrets!)
  ❌ Build logs, temp files

Multi-stage production image includes:
  ✅ Runtime only (Node.js, JRE, or just a binary)
  ✅ Compiled app
  ✅ Production dependencies only
  ✅ Nothing else
```

### No Leaked Secrets

```dockerfile
# Stage 1: Build (has secrets for private packages)
FROM node:20 AS builder

# This secret is used during build but NOT included in the final image!
ARG NPM_TOKEN

# Use the token to install private packages
RUN echo "//registry.npmjs.org/:_authToken=${NPM_TOKEN}" > .npmrc
RUN npm ci
RUN rm .npmrc    # Clean up just in case

COPY . .
RUN npm run build

# Stage 2: Production (no secrets here!)
FROM node:20-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
# The NPM_TOKEN and .npmrc are NOT in this image!
CMD ["node", "dist/server.js"]
```

**Even better — use BuildKit secrets:**

```dockerfile
# syntax=docker/dockerfile:1
FROM node:20 AS builder

WORKDIR /app

# Mount the secret file — it's NEVER stored in any layer!
RUN --mount=type=secret,id=npmrc,target=/root/.npmrc \
    npm ci

COPY . .
RUN npm run build

FROM node:20-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
CMD ["node", "dist/server.js"]
```

```bash
# Build with secret
docker build --secret id=npmrc,src=$HOME/.npmrc -t myapp .
```

**Why `--mount=type=secret` is better than `ARG`:**
- ARG values can leak in `docker history`
- Mounted secrets are **never stored in any layer** — they only exist during the build step

### Vulnerability Scanning Comparison

```
Single-stage image:
  Base OS packages: 847
  Known vulnerabilities: 142 (23 critical!)

Multi-stage (alpine-based) image:
  Base OS packages: 14
  Known vulnerabilities: 2 (0 critical) ✅
```

## 16. Common Patterns & Best Practices

### Pattern 1: Dependencies → Build → Runtime (The Standard Pattern)

```dockerfile
# Stage 1: Dependencies
FROM node:20-alpine AS deps
WORKDIR /app
COPY package*.json ./
RUN npm ci

# Stage 2: Build
FROM deps AS builder
COPY . .
RUN npm run build

# Stage 3: Runtime
FROM node:20-alpine AS runtime
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY --from=builder /app/dist ./dist
CMD ["node", "dist/server.js"]
```

**Why separate deps and build?** Better caching! If only source code changes, the deps stage stays fully cached.

### Pattern 2: Testing in Build Pipeline

```dockerfile
FROM node:20-alpine AS deps
WORKDIR /app
COPY package*.json ./
RUN npm ci

FROM deps AS builder
COPY . .
RUN npm run build

# Test stage — runs tests as part of the build
FROM deps AS tester
COPY . .
RUN npm test
# If tests fail, the entire build fails! ✅

# Production stage — only reached if tests pass
FROM node:20-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
CMD ["node", "dist/server.js"]
```

**Wait, does the test stage run automatically?**

No! Docker only builds stages that are **needed for the final image**. Since the final stage doesn't `COPY --from=tester`, the test stage is skipped by default.

To run it, use `--target`:
```bash
docker build --target tester -t myapp:test .
```

Or better, run tests in CI separately.

### Pattern 3: Healthcheck in the Image

```dockerfile
FROM node:20-alpine AS builder
WORKDIR /app
COPY . .
RUN npm ci && npm run build

FROM node:20-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules

# Add a healthcheck
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
  CMD wget --no-verbose --tries=1 --spider http://localhost:3000/health || exit 1

EXPOSE 3000
CMD ["node", "dist/server.js"]
```

### Best Practices Checklist

```
✅ Always name your stages with AS
✅ Use alpine or slim base images in production stage
✅ Copy dependency files BEFORE source code (caching!)
✅ Use --omit=dev for production dependencies
✅ Create and use non-root users
✅ Use .dockerignore to reduce build context
✅ Copy only what's needed with COPY --from
✅ Use specific image tags (node:20-alpine, not node:latest)
✅ Use BuildKit secret mounts instead of ARG for secrets
✅ Add HEALTHCHECK for production readiness
```

## 17. Anti-Patterns — What NOT to Do

### ❌ Anti-Pattern 1: Copying Everything from Builder

```dockerfile
# BAD — copies the entire builder filesystem!
COPY --from=builder / /

# GOOD — copy only what you need
COPY --from=builder /app/dist ./dist
```

### ❌ Anti-Pattern 2: Using the Same Heavy Base for All Stages

```dockerfile
# BAD — using ubuntu (huge!) for production
FROM ubuntu:22.04 AS builder
# ... build ...
FROM ubuntu:22.04
# Still 80 MB just for the base OS!

# GOOD — use Alpine or distroless for production
FROM ubuntu:22.04 AS builder
# ... build ...
FROM alpine:3.19
# Base is only 5 MB!
```

### ❌ Anti-Pattern 3: Putting Secrets in ARG

```dockerfile
# BAD — ARG values are visible in docker history!
ARG DB_PASSWORD
RUN echo "Connecting to DB with $DB_PASSWORD"

# GOOD — use BuildKit secret mounts
RUN --mount=type=secret,id=db_password \
    cat /run/secrets/db_password | connect-to-db
```

### ❌ Anti-Pattern 4: Not Using .dockerignore

```
Without .dockerignore, your build context includes:
  node_modules/    (400 MB) — will be installed fresh anyway!
  .git/            (100 MB) — not needed at all!
  *.md             (docs)   — not needed at runtime!
  .env             (secrets) — DANGER! Don't include!
```

### ❌ Anti-Pattern 5: Too Many Stages Without Purpose

```dockerfile
# BAD — unnecessary stages that add complexity
FROM node:20 AS stage1
RUN echo "hi"

FROM node:20 AS stage2
COPY --from=stage1 /tmp/hi .

FROM node:20 AS stage3
COPY --from=stage2 /tmp/hi .

# ...this is just unnecessary complexity!
```

Keep it simple. 2-3 stages is usually the sweet spot.

## 18. Real-World Example: Full Production Pipeline

Here's a complete, production-ready multi-stage Dockerfile that combines everything we've learned:

```dockerfile
# syntax=docker/dockerfile:1

# ============================================
# Global build arguments
# ============================================
ARG NODE_VERSION=20
ARG ALPINE_VERSION=3.19

# ============================================
# Stage 1: Dependencies
# ============================================
FROM node:${NODE_VERSION}-alpine${ALPINE_VERSION} AS deps

WORKDIR /app

# Copy dependency files
COPY package.json package-lock.json ./

# Install all dependencies with cache mount
RUN --mount=type=cache,target=/root/.npm \
    npm ci

# ============================================
# Stage 2: Build
# ============================================
FROM deps AS builder

# Copy source code
COPY tsconfig.json ./
COPY src/ ./src/
COPY public/ ./public/

# Build the application
ARG BUILD_ENV=production
RUN npm run build

# ============================================
# Stage 3: Production dependencies only
# ============================================
FROM node:${NODE_VERSION}-alpine${ALPINE_VERSION} AS prod-deps

WORKDIR /app
COPY package.json package-lock.json ./

RUN --mount=type=cache,target=/root/.npm \
    npm ci --omit=dev

# ============================================
# Stage 4: Production image
# ============================================
FROM node:${NODE_VERSION}-alpine${ALPINE_VERSION} AS production

# Install security updates
RUN apk update && apk upgrade --no-cache

# Create non-root user
RUN addgroup -S appgroup && adduser -S appuser -G appgroup

WORKDIR /app

# Copy production dependencies
COPY --from=prod-deps --chown=appuser:appgroup /app/node_modules ./node_modules

# Copy built application
COPY --from=builder --chown=appuser:appgroup /app/dist ./dist

# Copy package.json (for version info, etc.)
COPY --chown=appuser:appgroup package.json ./

# Switch to non-root user
USER appuser

# Healthcheck
HEALTHCHECK --interval=30s --timeout=3s --start-period=10s --retries=3 \
    CMD wget --no-verbose --tries=1 --spider http://localhost:3000/health || exit 1

# Set environment
ENV NODE_ENV=production
ENV PORT=3000

EXPOSE 3000

CMD ["node", "dist/server.js"]
```

**What makes this production-ready:**

1. **4 stages** — deps, builder, prod-deps, production (clean separation)
2. **Cache mounts** — npm packages cached across builds
3. **Non-root user** — Security best practice
4. **`--chown`** — Files owned by the app user, not root
5. **Health check** — Container orchestrator knows when the app is ready
6. **Security updates** — `apk upgrade` patches known vulnerabilities
7. **`NODE_ENV=production`** — Node.js runs in production mode (faster, less logging)
8. **Specific versions** — No `latest` tags, everything is pinned

**Build it:**

```bash
# Enable BuildKit
export DOCKER_BUILDKIT=1

# Build for production
docker build -t myapp:v1.0.0 .

# Check the image size
docker images myapp
# REPOSITORY   TAG       SIZE
# myapp        v1.0.0    85 MB  ← Nice and small!

# Scan for vulnerabilities
docker scout cves myapp:v1.0.0
```

## 19. Summary & Cheat Sheet

### Multi-Stage Build in One Sentence

> Use one stage to **build** your app with all the heavy tools, then copy only the **result** into a tiny production image.

### Quick Reference

| Language | Build Base | Production Base | Typical Size Reduction |
|----------|-----------|----------------|----------------------|
| Node.js | `node:20` | `node:20-alpine` | 900 MB → 80 MB |
| Python | `python:3.12` | `python:3.12-slim` | 1.2 GB → 150 MB |
| Go | `golang:1.22` | `scratch` or `distroless` | 800 MB → 8 MB |
| Java | `maven:3.9-jdk-21` | `eclipse-temurin:21-jre-alpine` | 800 MB → 150 MB |
| React | `node:20` | `nginx:alpine` | 400 MB → 25 MB |
| Rust | `rust:1.75` | `scratch` or `distroless` | 1.5 GB → 10 MB |

### Key Commands

```bash
# Build the full image (builds to the LAST stage)
docker build -t myapp .

# Build a specific stage
docker build --target builder -t myapp:build .

# Build with BuildKit enabled
DOCKER_BUILDKIT=1 docker build -t myapp .

# Build with build arguments
docker build --build-arg NODE_VERSION=18 -t myapp .

# Build with secrets (BuildKit)
docker build --secret id=npmrc,src=$HOME/.npmrc -t myapp .

# Check image size
docker images myapp

# Inspect image layers
docker history myapp:latest

# Scan for vulnerabilities
docker scout cves myapp:latest
```

### Syntax Cheat Sheet

```dockerfile
# Name a stage
FROM image:tag AS stage_name

# Copy from a named stage
COPY --from=stage_name /source/path /dest/path

# Copy from stage by number (0-indexed)
COPY --from=0 /source/path /dest/path

# Copy from an external image
COPY --from=nginx:alpine /etc/nginx/nginx.conf /etc/nginx/

# Change ownership while copying
COPY --from=builder --chown=user:group /app/dist ./dist

# BuildKit cache mount
RUN --mount=type=cache,target=/root/.npm npm ci

# BuildKit secret mount
RUN --mount=type=secret,id=mysecret cat /run/secrets/mysecret
```
