> **One Line Answer:** A relational database stores data in **tables** (rows and columns), and you use **SQL** (Structured Query Language) to talk to it. Think of it like a super-powered Excel spreadsheet that can handle millions of rows and connect tables together.

## рҹӨ” Why Should a DevOps Engineer Care About Databases?

You might be thinking вҖ” "I'm not a developer, why do I need to know databases?"

Here's the reality:

| Situation | Why You Need Database Knowledge |
|-----------|-------------------------------|
| App is slow | Maybe it's a slow database query, not the server |
| Deploying an app | You need to set up the database, run migrations, configure connections |
| Disaster recovery | You need to backup and restore databases |
| Scaling | You need to decide: scale up the database? Add replicas? Add caching? |
| Debugging | Logs say "database connection refused" вҖ” now what? |
| Kubernetes | You'll run databases in containers or connect to managed ones |
| CI/CD | Your pipeline might need to run database migrations automatically |

**Bottom line:** Every application uses a database. If you can't manage databases, you can't manage applications.

## рҹҸ—пёҸ What is a Relational Database?

A **relational database** organizes data into **tables** (also called relations). Each table has:

- **Columns** (also called fields) вҖ” define WHAT kind of data is stored
- **Rows** (also called records) вҖ” contain the ACTUAL data

### Example вҖ” A `users` Table:

| id | name | email | age | city |
|----|------|-------|-----|------|
| 1 | Ritesh | ritesh@email.com | 25 | Delhi |
| 2 | Priya | priya@email.com | 28 | Mumbai |
| 3 | Rahul | rahul@email.com | 22 | Bangalore |

Here:
- **Columns:** id, name, email, age, city
- **Rows:** Each person is one row
- **id** is the **Primary Key** вҖ” a unique identifier for each row (no two rows can have the same id)

### Why "Relational"?

Because tables can be **related** to each other!

## рҹ”‘ Key Concepts in Relational Databases

### 1. Primary Key

A column (or set of columns) that **uniquely identifies** each row.

```sql
-- 'id' is the primary key here
CREATE TABLE users (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    email VARCHAR(100)
);
```

Rules:
- Every table should have a primary key
- It must be **unique** вҖ” no two rows can have the same value
- It cannot be **NULL** (empty)

### 2. Foreign Key

A column that **references** a primary key in another table. It creates a link between tables.

```sql
CREATE TABLE orders (
    order_id INT PRIMARY KEY,
    user_id INT,
    item VARCHAR(100),
    price DECIMAL(10,2),
    FOREIGN KEY (user_id) REFERENCES users(id)
);
```

This means: "The `user_id` in orders must match an existing `id` in the users table."

### 3. Schema

The **blueprint** or **structure** of your database вҖ” what tables exist, what columns they have, what data types, what constraints.

```
Schema = Table definitions + Column types + Relationships + Constraints
```

### 4. ACID Properties

Relational databases guarantee **ACID** properties for transactions:

| Property | Meaning | Simple Explanation |
|----------|---------|-------------------|
| **A**tomicity | All or nothing | If a transaction has 5 steps and step 3 fails, ALL 5 steps are rolled back |
| **C**onsistency | Data stays valid | Database moves from one valid state to another valid state |
| **I**solation | Transactions don't interfere | Two people buying the last item won't both get it |
| **D**urability | Changes are permanent | Once committed, data survives even if the server crashes |

**Why ACID matters:** Imagine a bank transfer вҖ” you take вӮ№1000 from Account A and add вӮ№1000 to Account B. What if the system crashes after taking from A but before adding to B? ACID prevents this вҖ” either BOTH happen, or NEITHER happens.

## рҹҗ¬ MySQL vs рҹҗҳ PostgreSQL вҖ” The Two Giants

These are the two most popular open-source relational databases. Let's compare:

| Feature | MySQL | PostgreSQL |
|---------|-------|------------|
| **First Released** | 1995 | 1996 |
| **License** | GPL (owned by Oracle) | PostgreSQL License (fully open) |
| **Motto** | "The world's most popular open-source database" | "The world's most advanced open-source database" |
| **Speed** | Faster for simple read-heavy workloads | Faster for complex queries and write-heavy workloads |
| **ACID Compliance** | Yes (with InnoDB engine) | Yes (by default) |
| **JSON Support** | Basic | Excellent (JSONB вҖ” binary JSON, super fast) |
| **Replication** | Master-Slave, Group Replication | Streaming, Logical, Multi-Master |
| **Extensions** | Limited | Very extensible (PostGIS, pg_trgm, TimescaleDB, etc.) |
| **Used By** | Facebook, Twitter, Uber, Airbnb | Apple, Instagram, Spotify, Reddit, NASA |
| **Best For** | Web apps, simple CRUD apps, read-heavy apps | Complex apps, analytics, GIS, time-series data |

### Which One Should You Learn?

**Both!** But if you're starting from scratch:
- **MySQL** вҖ” easier to start with, widely used, great for web apps
- **PostgreSQL** вҖ” more powerful, better for complex use cases, growing rapidly

For DevOps purposes, you'll encounter both regularly. The SQL syntax is 90% the same.

## рҹӣ пёҸ Installing MySQL and PostgreSQL

### Installing MySQL on Ubuntu/Debian

```bash
# Update package list
sudo apt update

# Install MySQL server
sudo apt install mysql-server -y

# Start MySQL service
sudo systemctl start mysql

# Enable MySQL to start on boot
sudo systemctl enable mysql

# Check if MySQL is running
sudo systemctl status mysql

# Secure the installation (set root password, remove test databases, etc.)
sudo mysql_secure_installation
```

### Installing PostgreSQL on Ubuntu/Debian

```bash
# Update package list
sudo apt update

# Install PostgreSQL
sudo apt install postgresql postgresql-contrib -y

# Start PostgreSQL service
sudo systemctl start postgresql

# Enable PostgreSQL to start on boot
sudo systemctl enable postgresql

# Check if PostgreSQL is running
sudo systemctl status postgresql
```

### Installing MySQL on CentOS/RHEL

```bash
# Install MySQL repository
sudo yum install https://dev.mysql.com/get/mysql80-community-release-el7-5.noarch.rpm -y

# Install MySQL server
sudo yum install mysql-server -y

# Start and enable
sudo systemctl start mysqld
sudo systemctl enable mysqld

# Get temporary root password (MySQL generates one during install)
sudo grep 'temporary password' /var/log/mysqld.log
```

### Installing PostgreSQL on CentOS/RHEL

```bash
# Install PostgreSQL
sudo yum install postgresql-server postgresql-contrib -y

# Initialize the database
sudo postgresql-setup initdb

# Start and enable
sudo systemctl start postgresql
sudo systemctl enable postgresql
```

### Connecting to PostgreSQL

```bash
# PostgreSQL creates a system user called 'postgres'
# Switch to that user first
sudo -u postgres psql

# You'll see the PostgreSQL prompt:
# postgres=#
```

### Connection String Format (for applications)

Applications connect to databases using a **connection string** (also called a **DSN** вҖ” Data Source Name):

```
MySQL:      mysql://username:password@hostname:3306/database_name
PostgreSQL: postgresql://username:password@hostname:5432/database_name
```

| Part | Meaning |
|------|---------|
| `username` | Database user |
| `password` | User's password |
| `hostname` | Server IP or hostname (e.g., localhost, 10.0.0.5) |
| `3306` / `5432` | Default port for MySQL / PostgreSQL |
| `database_name` | Which database to connect to |

**DevOps tip:** You'll often configure these as environment variables in your app deployments:

```bash
export DB_HOST=10.0.0.5
export DB_PORT=5432
export DB_USER=myapp
export DB_PASSWORD=secret123
export DB_NAME=production_db
```

## рҹ“қ SQL вҖ” The Language of Relational Databases

SQL (Structured Query Language) is the language you use to talk to relational databases. It's the same basic language for MySQL, PostgreSQL, and others (with some minor differences).

SQL has different categories of commands:

| Category | Full Form | What It Does | Commands |
|----------|-----------|-------------|----------|
| **DDL** | Data Definition Language | Create/modify structure | CREATE, ALTER, DROP, TRUNCATE |
| **DML** | Data Manipulation Language | Insert/update/delete data | INSERT, UPDATE, DELETE |
| **DQL** | Data Query Language | Read/retrieve data | SELECT |
| **DCL** | Data Control Language | Permissions | GRANT, REVOKE |
| **TCL** | Transaction Control Language | Transaction management | COMMIT, ROLLBACK, SAVEPOINT |

### Creating a Database

```sql
-- Create a new database
CREATE DATABASE myapp_db;

SHOW DATABASES;

\l

-- MySQL:
USE myapp_db;

\c myapp_db
```

### Creating Tables

```sql
-- Create a users table
CREATE TABLE users (
    id          INT PRIMARY KEY AUTO_INCREMENT,   -- MySQL uses AUTO_INCREMENT
    name        VARCHAR(100) NOT NULL,            -- Variable-length string, max 100 chars, cannot be empty
    email       VARCHAR(255) UNIQUE NOT NULL,     -- Must be unique and not empty
    age         INT,                              -- Integer, can be NULL
    salary      DECIMAL(10, 2),                   -- Decimal number with 10 digits total, 2 after decimal
    is_active   BOOLEAN DEFAULT TRUE,             -- Boolean with default value
    created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP -- Auto-set to current time
);
```

**PostgreSQL version** (slightly different syntax):

```sql
CREATE TABLE users (
    id          SERIAL PRIMARY KEY,               -- PostgreSQL uses SERIAL for auto-increment
    name        VARCHAR(100) NOT NULL,
    email       VARCHAR(255) UNIQUE NOT NULL,
    age         INT,
    salary      DECIMAL(10, 2),
    is_active   BOOLEAN DEFAULT TRUE,
    created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

#### Common Data Types

| Data Type | MySQL | PostgreSQL | Description |
|-----------|-------|------------|-------------|
| Integer | INT, BIGINT | INT, BIGINT | Whole numbers |
| Auto-increment | AUTO_INCREMENT | SERIAL, BIGSERIAL | Auto-generating IDs |
| Small text | VARCHAR(n) | VARCHAR(n) | Variable-length string |
| Long text | TEXT | TEXT | Unlimited text |
| Decimal | DECIMAL(p,s) | DECIMAL(p,s) / NUMERIC(p,s) | Exact decimal numbers |
| Boolean | BOOLEAN (TINYINT) | BOOLEAN | True/False |
| Date | DATE | DATE | Date only (2024-01-15) |
| Timestamp | TIMESTAMP | TIMESTAMP | Date + Time |
| JSON | JSON | JSON / JSONB | JSON data |
| UUID | CHAR(36) | UUID | Universally unique ID |

### Modifying Tables (ALTER TABLE)

```sql
-- Add a new column
ALTER TABLE users ADD COLUMN phone VARCHAR(15);

-- Change column data type
-- MySQL:
ALTER TABLE users MODIFY COLUMN phone VARCHAR(20);
-- PostgreSQL:
ALTER TABLE users ALTER COLUMN phone TYPE VARCHAR(20);

-- Rename a column
-- MySQL:
ALTER TABLE users CHANGE COLUMN phone mobile VARCHAR(20);
-- PostgreSQL:
ALTER TABLE users RENAME COLUMN phone TO mobile;

-- Drop (delete) a column
ALTER TABLE users DROP COLUMN mobile;

-- Add a constraint
ALTER TABLE users ADD CONSTRAINT unique_email UNIQUE (email);
```

### Dropping (Deleting) Tables

```sql
-- Delete a table and all its data (CAREFUL!)
DROP TABLE users;

-- Delete only if it exists (avoids error)
DROP TABLE IF EXISTS users;

-- Delete all data but keep the table structure
TRUNCATE TABLE users;
```

**вҡ пёҸ Warning:** `DROP TABLE` and `TRUNCATE` are **irreversible**! Always take a backup first.

## вңҸпёҸ DML вҖ” Inserting, Updating, and Deleting Data

### INSERT вҖ” Adding Data

```sql
-- Insert a single row
INSERT INTO users (name, email, age, salary) 
VALUES ('Ritesh', 'ritesh@email.com', 25, 75000.00);

-- Insert multiple rows at once
INSERT INTO users (name, email, age, salary) VALUES 
    ('Priya', 'priya@email.com', 28, 85000.00),
    ('Rahul', 'rahul@email.com', 22, 55000.00),
    ('Sneha', 'sneha@email.com', 30, 95000.00),
    ('Amit', 'amit@email.com', 35, 120000.00);
```

### UPDATE вҖ” Modifying Existing Data

```sql
-- Update a specific user's salary
UPDATE users SET salary = 80000.00 WHERE id = 1;

-- Update multiple columns
UPDATE users SET salary = 90000.00, age = 26 WHERE name = 'Ritesh';

-- Update all rows (DANGEROUS вҖ” no WHERE clause!)
UPDATE users SET is_active = TRUE;
```

**вҡ пёҸ Golden Rule:** ALWAYS use a `WHERE` clause with `UPDATE` and `DELETE`. Without it, you'll change/delete ALL rows!

### DELETE вҖ” Removing Data

```sql
-- Delete a specific user
DELETE FROM users WHERE id = 3;

-- Delete all inactive users
DELETE FROM users WHERE is_active = FALSE;

-- Delete ALL rows (but keep the table)
DELETE FROM users;
-- Better to use TRUNCATE for this:
TRUNCATE TABLE users;
```

## рҹ”Қ DQL вҖ” Querying Data (SELECT)

This is the most important part вҖ” **reading data**. You'll use SELECT more than anything else.

### Basic SELECT

```sql
-- Select all columns and all rows
SELECT * FROM users;

-- Select specific columns
SELECT name, email, salary FROM users;

-- Select with an alias (rename column in output)
SELECT name AS "Full Name", salary AS "Monthly Pay" FROM users;
```

### WHERE вҖ” Filtering Rows

```sql
-- Exact match
SELECT * FROM users WHERE name = 'Ritesh';

-- Comparison operators
SELECT * FROM users WHERE age > 25;
SELECT * FROM users WHERE salary >= 80000;
SELECT * FROM users WHERE age != 30;    -- Not equal
SELECT * FROM users WHERE age <> 30;    -- Also not equal

-- Multiple conditions
SELECT * FROM users WHERE age > 25 AND salary > 80000;
SELECT * FROM users WHERE city = 'Delhi' OR city = 'Mumbai';

-- IN вҖ” match any value in a list
SELECT * FROM users WHERE city IN ('Delhi', 'Mumbai', 'Bangalore');

-- BETWEEN вҖ” range
SELECT * FROM users WHERE age BETWEEN 25 AND 35;

-- LIKE вҖ” pattern matching
SELECT * FROM users WHERE name LIKE 'R%';      -- Starts with R
SELECT * FROM users WHERE email LIKE '%@gmail%'; -- Contains @gmail
SELECT * FROM users WHERE name LIKE '_i%';      -- Second letter is 'i'

-- IS NULL / IS NOT NULL
SELECT * FROM users WHERE phone IS NULL;
SELECT * FROM users WHERE phone IS NOT NULL;
```

### ORDER BY вҖ” Sorting Results

```sql
-- Sort by salary (ascending вҖ” lowest first)
SELECT * FROM users ORDER BY salary ASC;

-- Sort by salary (descending вҖ” highest first)
SELECT * FROM users ORDER BY salary DESC;

-- Sort by multiple columns
SELECT * FROM users ORDER BY city ASC, salary DESC;
```

### LIMIT and OFFSET вҖ” Pagination

```sql
-- Get only the first 5 rows
SELECT * FROM users LIMIT 5;

-- Skip first 10 rows, then get next 5 (for pagination)
SELECT * FROM users LIMIT 5 OFFSET 10;

-- MySQL shorthand
SELECT * FROM users LIMIT 10, 5;  -- Same as LIMIT 5 OFFSET 10
```

### Aggregate Functions

```sql
-- COUNT вҖ” how many rows?
SELECT COUNT(*) FROM users;                     -- Total users
SELECT COUNT(*) FROM users WHERE is_active = TRUE;  -- Active users

-- SUM вҖ” total
SELECT SUM(salary) FROM users;

-- AVG вҖ” average
SELECT AVG(salary) FROM users;
SELECT AVG(age) FROM users WHERE city = 'Delhi';

-- MIN and MAX
SELECT MIN(salary) FROM users;
SELECT MAX(salary) FROM users;
```

### GROUP BY вҖ” Grouping Results

```sql
-- Count users per city
SELECT city, COUNT(*) AS user_count 
FROM users 
GROUP BY city;

-- Average salary per city
SELECT city, AVG(salary) AS avg_salary 
FROM users 
GROUP BY city;

-- HAVING вҖ” filter groups (like WHERE but for groups)
SELECT city, COUNT(*) AS user_count 
FROM users 
GROUP BY city 
HAVING COUNT(*) > 5;    -- Only cities with more than 5 users
```

**Remember:** `WHERE` filters rows BEFORE grouping. `HAVING` filters groups AFTER grouping.

### JOINs вҖ” Combining Tables

This is where relational databases shine! JOINs let you combine data from multiple tables.

Let's say we have two tables:

```sql

CREATE TABLE users (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    email VARCHAR(255)
);

-- orders table
CREATE TABLE orders (
    order_id INT PRIMARY KEY,
    user_id INT,
    product VARCHAR(100),
    amount DECIMAL(10,2),
    FOREIGN KEY (user_id) REFERENCES users(id)
);
```

#### INNER JOIN вҖ” Only matching rows from both tables

```sql
SELECT users.name, orders.product, orders.amount
FROM users
INNER JOIN orders ON users.id = orders.user_id;
```

## рҹ“Ҡ Indexing вҖ” Making Queries Fast

### What is an Index?

An index is like the **index at the back of a book**. Instead of reading every page to find a topic, you look at the index, find the page number, and jump directly there.

Without an index, the database does a **Full Table Scan** вҖ” it reads EVERY row to find what you're looking for. With millions of rows, this is **painfully slow**.

```
Without Index (Full Table Scan):
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ Read row 1... nope           в”Ӯ
в”Ӯ Read row 2... nope           в”Ӯ
в”Ӯ Read row 3... nope           в”Ӯ
в”Ӯ ...                          в”Ӯ
в”Ӯ Read row 999,999... nope     в”Ӯ
в”Ӯ Read row 1,000,000... FOUND! в”Ӯ вҶҗ Had to read ALL 1 million rows!
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ

With Index:
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ Check index...   в”Ӯ
в”Ӯ Jump to row      в”Ӯ
в”Ӯ 1,000,000...     в”Ӯ
в”Ӯ FOUND!           в”Ӯ вҶҗ Jumped directly! Milliseconds!
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

### How Indexes Work (Simplified)

Databases typically use a **B-Tree** (balanced tree) structure for indexes:

```
                    [M]
                   /   \
              [D, H]    [R, W]
             / | \      / | \
           [A-C][E-G][I-L][N-Q][S-V][X-Z]
           
Looking for "Ritesh":
1. Start at root [M] вҶ’ R > M вҶ’ go right
2. At [R, W] вҶ’ R = R вҶ’ go to R's subtree
3. At [N-Q] вҶ’ not here вҶ’ check [S-V] вҶ’ found!

Only 3 steps instead of scanning millions of rows!
```

### Creating Indexes

```sql
-- Create an index on the email column
CREATE INDEX idx_users_email ON users(email);

-- Create a unique index (no duplicate values allowed)
CREATE UNIQUE INDEX idx_users_email ON users(email);

-- Create a composite index (multiple columns)
CREATE INDEX idx_users_city_age ON users(city, age);

-- PostgreSQL: Create an index concurrently (doesn't lock the table)
CREATE INDEX CONCURRENTLY idx_users_email ON users(email);
```

### When to Create Indexes

| вң… Create Index When | вқҢ Don't Create Index When |
|---------------------|--------------------------|
| Column used in WHERE clauses frequently | Table has very few rows (< 1000) |
| Column used in JOIN conditions | Column has very few unique values (e.g., gender) |
| Column used in ORDER BY | Table is written to much more than read |
| Column used in GROUP BY | You're indexing every column (too many indexes slow writes) |

### Viewing Existing Indexes

```sql
-- MySQL: Show indexes for a table
SHOW INDEX FROM users;

-- PostgreSQL: List indexes
\di
-- Or:
SELECT indexname, indexdef FROM pg_indexes WHERE tablename = 'users';
```

### Analyzing Query Performance (EXPLAIN)

The `EXPLAIN` command shows you HOW the database executes a query вҖ” whether it's using an index or doing a full scan.

```sql
-- MySQL
EXPLAIN SELECT * FROM users WHERE email = 'ritesh@email.com';

-- PostgreSQL (more detailed)
EXPLAIN ANALYZE SELECT * FROM users WHERE email = 'ritesh@email.com';
```

**What to look for:**

| Output | Meaning | Good or Bad? |
|--------|---------|-------------|
| `Seq Scan` / `ALL` | Full table scan вҖ” reading every row | вқҢ Bad (for large tables) |
| `Index Scan` / `ref` | Using an index | вң… Good |
| `Index Only Scan` | Answering from index alone (no table access) | вң…вң… Best |
| `Bitmap Index Scan` | Using index + some table access | вң… Good |

## рҹ‘Ө User Management and Permissions

As a DevOps engineer, you'll need to create database users and control who can do what.

### MySQL User Management

```sql
-- Create a new user
CREATE USER 'myapp'@'localhost' IDENTIFIED BY 'StrongP@ssw0rd!';

-- Create a user that can connect from any host
CREATE USER 'myapp'@'%' IDENTIFIED BY 'StrongP@ssw0rd!';

-- Grant permissions
-- Give full access to a specific database
GRANT ALL PRIVILEGES ON myapp_db.* TO 'myapp'@'%';

-- Give read-only access
GRANT SELECT ON myapp_db.* TO 'readonly_user'@'%';

-- Give specific permissions
GRANT SELECT, INSERT, UPDATE ON myapp_db.* TO 'myapp'@'%';

-- Apply the changes
FLUSH PRIVILEGES;

-- Show grants for a user
SHOW GRANTS FOR 'myapp'@'%';

-- Revoke permissions
REVOKE ALL PRIVILEGES ON myapp_db.* FROM 'myapp'@'%';

-- Delete a user
DROP USER 'myapp'@'%';
```

### PostgreSQL User Management

```sql
-- Create a new user (called "role" in PostgreSQL)
CREATE USER myapp WITH PASSWORD 'StrongP@ssw0rd!';

-- Create a database and assign owner
CREATE DATABASE myapp_db OWNER myapp;

-- Grant permissions
GRANT ALL PRIVILEGES ON DATABASE myapp_db TO myapp;

-- Grant read-only access
GRANT CONNECT ON DATABASE myapp_db TO readonly_user;
GRANT USAGE ON SCHEMA public TO readonly_user;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO readonly_user;

-- Make future tables also readable
ALTER DEFAULT PRIVILEGES IN SCHEMA public 
GRANT SELECT ON TABLES TO readonly_user;

-- List all users/roles
\du

-- Revoke permissions
REVOKE ALL PRIVILEGES ON DATABASE myapp_db FROM myapp;

-- Delete a user
DROP USER myapp;
```

## вҡҷпёҸ Configuration Files вҖ” Where DevOps Meets Databases

### MySQL Configuration

```bash
# Main config file location
/etc/mysql/mysql.conf.d/mysqld.cnf    # Ubuntu/Debian
/etc/my.cnf                            # CentOS/RHEL
```

Key settings you'll tune:

```ini
[mysqld]
# Networking
bind-address = 0.0.0.0          # Listen on all interfaces (default: 127.0.0.1)
port = 3306                      # Default port

# Performance
max_connections = 200            # Max simultaneous connections
innodb_buffer_pool_size = 1G     # RAM allocated for caching data (set to 70-80% of RAM for dedicated DB)
query_cache_size = 64M           # Cache for repeated queries

# Logging
slow_query_log = 1               # Enable slow query logging
slow_query_log_file = /var/log/mysql/slow.log
long_query_time = 2              # Log queries taking more than 2 seconds

# Data directory
datadir = /var/lib/mysql
```

### PostgreSQL Configuration

```bash
# Main config files
/etc/postgresql/16/main/postgresql.conf    # Main config
/etc/postgresql/16/main/pg_hba.conf        # Client authentication
```

Key settings in `postgresql.conf`:

```ini
# Networking
listen_addresses = '*'           # Listen on all interfaces (default: 'localhost')
port = 5432                      # Default port

# Performance
max_connections = 200            # Max simultaneous connections
shared_buffers = 256MB           # RAM for caching (set to 25% of total RAM)
effective_cache_size = 768MB     # Total RAM available for caching (set to 75% of total RAM)
work_mem = 4MB                   # RAM per query operation (sorts, joins)

# Logging
logging_collector = on
log_directory = 'pg_log'
log_min_duration_statement = 2000   # Log queries taking more than 2000ms (2 seconds)
```

Key settings in `pg_hba.conf` (controls WHO can connect):

```
# TYPE  DATABASE    USER        ADDRESS         METHOD
local   all         all                         peer        # Local socket connections
host    all         all         127.0.0.1/32    md5         # Localhost with password
host    all         all         10.0.0.0/24     md5         # Allow from 10.0.0.x network
host    myapp_db    myapp       0.0.0.0/0       md5         # myapp user from anywhere
```

**After changing config, restart the service:**

```bash
# MySQL
sudo systemctl restart mysql

# PostgreSQL
sudo systemctl restart postgresql
# Or just reload (doesn't disconnect users):
sudo systemctl reload postgresql
```

## рҹ“Ҡ Important Database Commands for DevOps

### Check Database Size

```sql
-- MySQL: Size of all databases
SELECT table_schema AS "Database", 
       ROUND(SUM(data_length + index_length) / 1024 / 1024, 2) AS "Size (MB)"
FROM information_schema.tables 
GROUP BY table_schema;

-- PostgreSQL: Size of current database
SELECT pg_size_pretty(pg_database_size(current_database()));

-- PostgreSQL: Size of all databases
SELECT datname, pg_size_pretty(pg_database_size(datname)) 
FROM pg_database ORDER BY pg_database_size(datname) DESC;
```

### Check Active Connections

```sql
-- MySQL
SHOW PROCESSLIST;

-- PostgreSQL
SELECT pid, usename, client_addr, state, query 
FROM pg_stat_activity 
WHERE state = 'active';
```

### Kill a Long-Running Query

```sql
-- MySQL
KILL <process_id>;

-- PostgreSQL
SELECT pg_terminate_backend(<pid>);
```

### Check Table Sizes

```sql
-- MySQL
SELECT table_name, 
       ROUND((data_length + index_length) / 1024 / 1024, 2) AS "Size (MB)"
FROM information_schema.tables 
WHERE table_schema = 'myapp_db'
ORDER BY (data_length + index_length) DESC;

-- PostgreSQL
SELECT relname AS "Table", 
       pg_size_pretty(pg_total_relation_size(relid)) AS "Total Size"
FROM pg_catalog.pg_statio_user_tables 
ORDER BY pg_total_relation_size(relid) DESC;
```

## рҹҗі Running MySQL and PostgreSQL in Docker

As a DevOps engineer, you'll often spin up databases in containers for development or testing:

### MySQL in Docker

```bash
# Run MySQL container
docker run -d \
  --name mysql-server \
  -e MYSQL_ROOT_PASSWORD=rootpassword \
  -e MYSQL_DATABASE=myapp_db \
  -e MYSQL_USER=myapp \
  -e MYSQL_PASSWORD=apppassword \
  -p 3306:3306 \
  -v mysql_data:/var/lib/mysql \
  mysql:8.0

# Connect to MySQL in the container
docker exec -it mysql-server mysql -u root -p
```

### PostgreSQL in Docker

```bash
# Run PostgreSQL container
docker run -d \
  --name postgres-server \
  -e POSTGRES_PASSWORD=rootpassword \
  -e POSTGRES_DB=myapp_db \
  -e POSTGRES_USER=myapp \
  -p 5432:5432 \
  -v postgres_data:/var/lib/postgresql/data \
  postgres:16

# Connect to PostgreSQL in the container
docker exec -it postgres-server psql -U myapp -d myapp_db
```

### Docker Compose (Both Together!)

```yaml
# docker-compose.yml
version: '3.8'

services:
  mysql:
    image: mysql:8.0
    container_name: mysql-db
    environment:
      MYSQL_ROOT_PASSWORD: rootpassword
      MYSQL_DATABASE: myapp_db
      MYSQL_USER: myapp
      MYSQL_PASSWORD: apppassword
    ports:
      - "3306:3306"
    volumes:
      - mysql_data:/var/lib/mysql
    restart: unless-stopped

  postgres:
    image: postgres:16
    container_name: postgres-db
    environment:
      POSTGRES_PASSWORD: rootpassword
      POSTGRES_DB: myapp_db
      POSTGRES_USER: myapp
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data
    restart: unless-stopped

volumes:
  mysql_data:
  postgres_data:
```

```bash
# Start both
docker-compose up -d

# Stop both
docker-compose down

# Stop and remove data volumes too
docker-compose down -v
```

## рҹ§  Quick Reference вҖ” SQL Cheat Sheet

```sql
-- DATABASE OPERATIONS
CREATE DATABASE mydb;
DROP DATABASE mydb;
USE mydb;                          -- MySQL
\c mydb                            -- PostgreSQL

-- TABLE OPERATIONS
CREATE TABLE t (id INT PRIMARY KEY, name VARCHAR(100));
ALTER TABLE t ADD COLUMN age INT;
DROP TABLE t;
TRUNCATE TABLE t;

-- CRUD OPERATIONS
INSERT INTO t (name, age) VALUES ('John', 25);
SELECT * FROM t WHERE age > 20 ORDER BY name LIMIT 10;
UPDATE t SET age = 26 WHERE name = 'John';
DELETE FROM t WHERE id = 1;

-- AGGREGATES
SELECT COUNT(*), AVG(age), MIN(age), MAX(age), SUM(age) FROM t;
SELECT city, COUNT(*) FROM t GROUP BY city HAVING COUNT(*) > 5;

-- JOINS
SELECT * FROM a INNER JOIN b ON a.id = b.a_id;
SELECT * FROM a LEFT JOIN b ON a.id = b.a_id;

-- INDEXES
CREATE INDEX idx_name ON t(name);
DROP INDEX idx_name;               -- PostgreSQL
DROP INDEX idx_name ON t;          -- MySQL

-- USERS
CREATE USER 'user'@'%' IDENTIFIED BY 'pass';    -- MySQL
CREATE USER user WITH PASSWORD 'pass';            -- PostgreSQL
GRANT ALL ON db.* TO 'user'@'%';                 -- MySQL
GRANT ALL ON DATABASE db TO user;                 -- PostgreSQL
```
