# 7.2 — NoSQL Databases — MongoDB, Redis, DynamoDB, Cassandra

> **One Line Answer:** NoSQL databases are **non-relational** databases that store data in formats other than tables — like documents (JSON), key-value pairs, wide columns, or graphs. They're designed for **flexibility, scalability, and speed** when traditional SQL databases aren't the best fit.

---

## 🤔 Why Do We Need NoSQL? What's Wrong With SQL?

Nothing is "wrong" with SQL! Relational databases are amazing. But they have limitations in certain situations:

| Scenario | Problem with SQL | NoSQL Solution |
|----------|-----------------|----------------|
| Social media posts (millions per second) | Single SQL server can't handle the write volume | Distributed databases that spread across many servers |
| User profiles with varying fields | Every row must have the same columns in SQL | Documents can have different fields per record |
| Shopping cart / session data | Need super-fast reads/writes (milliseconds) | In-memory databases like Redis |
| IoT data (billions of sensor readings) | SQL struggles with massive time-series data | Wide-column stores like Cassandra |
| Real-time recommendations | Complex relationships between users/products | Graph databases |

### The Simple Analogy

```
SQL Database   = A filing cabinet with fixed folders
                 Every folder has the SAME structure
                 Great for organized, structured data
                 
NoSQL Database = A box where you can throw in anything
                 Documents, sticky notes, photos, whatever
                 Flexible, but you need to know what's inside
```

---

## 🗂️ Types of NoSQL Databases

There are **4 main types** of NoSQL databases. Each one solves a different problem:

```
NoSQL Database Types
├── 📄 Document Stores      → MongoDB, CouchDB
│    └── Store data as JSON-like documents
│
├── 🔑 Key-Value Stores     → Redis, Memcached, DynamoDB
│    └── Store data as key → value pairs (like a dictionary)
│
├── 📊 Wide-Column Stores   → Cassandra, HBase, ScyllaDB
│    └── Store data in rows with dynamic columns
│
└── 🕸️ Graph Databases      → Neo4j, Amazon Neptune
     └── Store data as nodes and relationships
```

| Type | Think of it as... | Best For |
|------|------------------|----------|
| Document | A folder full of JSON files | User profiles, product catalogs, content management |
| Key-Value | A giant dictionary/hashmap | Caching, sessions, shopping carts, leaderboards |
| Wide-Column | A spreadsheet where each row can have different columns | IoT data, time-series, logging, analytics |
| Graph | A mind map / social network diagram | Social networks, recommendation engines, fraud detection |

---

## 📝 SQL vs NoSQL — The Full Comparison

| Feature | SQL (Relational) | NoSQL |
|---------|-----------------|-------|
| **Data Model** | Tables with rows and columns | Documents, key-value, columns, graphs |
| **Schema** | Fixed schema (define structure first) | Dynamic schema (structure can change anytime) |
| **Scaling** | Vertical (bigger server) | Horizontal (add more servers) |
| **Query Language** | SQL (standardized) | Database-specific (no universal language) |
| **Relationships** | JOINs across tables | Typically embedded/nested data |
| **ACID** | Strong ACID | Eventually consistent (some support ACID) |
| **Best For** | Complex queries, transactions, structured data | Large-scale, flexible, high-speed workloads |
| **Examples** | MySQL, PostgreSQL, Oracle, SQL Server | MongoDB, Redis, Cassandra, DynamoDB |

### When to Use What?

```
Use SQL When:                           Use NoSQL When:
✅ Data is highly structured            ✅ Data structure changes frequently
✅ You need complex JOINs               ✅ You need massive horizontal scaling
✅ ACID transactions are critical       ✅ You need super-fast read/write speed
✅ Data integrity is paramount          ✅ You're dealing with huge volumes of data
✅ Banking, e-commerce checkout         ✅ Real-time apps, IoT, social media
```

**Real-world truth:** Most modern applications use BOTH! This is called **polyglot persistence**:

```
Example: An e-commerce platform
├── PostgreSQL    → User accounts, orders, payments (needs ACID)
├── MongoDB       → Product catalog (flexible schemas)
├── Redis         → Shopping cart, sessions, caching (needs speed)
├── Elasticsearch → Product search (full-text search)
└── Cassandra     → Click-stream analytics (massive writes)
```

## 📄 MongoDB — The Document Database

### What is MongoDB?

MongoDB is the most popular **document database**. It stores data as **JSON-like documents** (technically BSON — Binary JSON).

### How Does It Work?

Instead of tables, rows, and columns, MongoDB uses:

| SQL Term | MongoDB Term | Description |
|----------|-------------|-------------|
| Database | Database | Same concept — a container for collections |
| Table | Collection | A group of documents (like a folder) |
| Row | Document | A single JSON record |
| Column | Field | A key in the JSON document |
| Primary Key | `_id` | Automatically generated unique identifier |
| JOIN | Embedding / `$lookup` | Nest related data inside the document or link them |

### Example: SQL vs MongoDB

**SQL (MySQL/PostgreSQL):**

```sql
-- Users table
CREATE TABLE users (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    email VARCHAR(255),
    age INT
);

INSERT INTO users VALUES (1, 'Ritesh', 'ritesh@email.com', 25);
```

**MongoDB:**

```javascript
// users collection
{
    "_id": ObjectId("64a7b8c9d1e2f3a4b5c6d7e8"),
    "name": "Ritesh",
    "email": "ritesh@email.com",
    "age": 25,
    "address": {                    // ← Nested object! Can't do this easily in SQL
        "street": "123 Main St",
        "city": "Delhi",
        "pin": "110001"
    },
    "hobbies": ["coding", "reading", "gaming"],   // ← Array! Also hard in SQL
    "orders": [                     // ← Related data embedded right here!
        {"item": "Laptop", "price": 50000},
        {"item": "Mouse", "price": 500}
    ]
}
```

**See the difference?** In SQL, you'd need 3 separate tables (users, addresses, orders) and JOINs. In MongoDB, everything lives in ONE document. This is called **embedding** or **denormalization**.

### MongoDB CRUD Operations

#### Create (Insert)

```javascript
// Switch to a database (creates it if it doesn't exist)
use myapp_db

// Insert one document
db.users.insertOne({
    name: "Ritesh",
    email: "ritesh@email.com",
    age: 25,
    city: "Delhi",
    skills: ["Python", "Docker", "Kubernetes"]
})

// Insert multiple documents
db.users.insertMany([
    {
        name: "Priya",
        email: "priya@email.com",
        age: 28,
        city: "Mumbai",
        skills: ["Java", "AWS", "Terraform"]
    },
    {
        name: "Rahul",
        email: "rahul@email.com",
        age: 22,
        city: "Bangalore",
        skills: ["Go", "Docker", "Linux"]
    }
])
```

#### Read (Find/Query)

```javascript
// Find all documents in the collection
db.users.find()

// Pretty print (formatted output)
db.users.find().pretty()

// Find with a filter (like SQL WHERE)
db.users.find({ city: "Delhi" })

// Find one document
db.users.findOne({ name: "Ritesh" })

// Comparison operators
db.users.find({ age: { $gt: 25 } })        // age > 25
db.users.find({ age: { $gte: 25 } })       // age >= 25
db.users.find({ age: { $lt: 30 } })        // age < 30
db.users.find({ age: { $lte: 30 } })       // age <= 30
db.users.find({ age: { $ne: 25 } })        // age != 25

// Multiple conditions (AND)
db.users.find({ city: "Delhi", age: { $gt: 20 } })

// OR condition
db.users.find({ $or: [{ city: "Delhi" }, { city: "Mumbai" }] })

// IN — match any value in array
db.users.find({ city: { $in: ["Delhi", "Mumbai"] } })

// Check if a field exists
db.users.find({ phone: { $exists: true } })

// Search inside arrays
db.users.find({ skills: "Docker" })         // Has "Docker" in skills array

// Select specific fields (projection) — like SELECT name, email
db.users.find({}, { name: 1, email: 1, _id: 0 })

// Sort results
db.users.find().sort({ age: 1 })            // Ascending
db.users.find().sort({ age: -1 })           // Descending

// Limit and skip (pagination)
db.users.find().limit(5)                     // First 5
db.users.find().skip(10).limit(5)           // Page 3 (skip 10, get 5)

// Count documents
db.users.countDocuments({ city: "Delhi" })
```

#### Update

```javascript
// Update one document
db.users.updateOne(
    { name: "Ritesh" },                     // Filter: which document
    { $set: { age: 26, city: "Noida" } }   // Update: what to change
)

// Update multiple documents
db.users.updateMany(
    { city: "Delhi" },                      // Filter
    { $set: { region: "North India" } }     // Update
)

// Add to an array
db.users.updateOne(
    { name: "Ritesh" },
    { $push: { skills: "Ansible" } }        // Add "Ansible" to skills array
)

// Remove from an array
db.users.updateOne(
    { name: "Ritesh" },
    { $pull: { skills: "Python" } }         // Remove "Python" from skills
)

// Increment a number
db.users.updateOne(
    { name: "Ritesh" },
    { $inc: { age: 1 } }                    // Increase age by 1
)

// Replace entire document (keep the _id)
db.users.replaceOne(
    { name: "Ritesh" },
    { name: "Ritesh Singh", email: "ritesh@new.com", age: 26 }
)
```

#### Delete

```javascript
// Delete one document
db.users.deleteOne({ name: "Rahul" })

// Delete multiple documents
db.users.deleteMany({ age: { $lt: 20 } })

// Delete all documents in a collection
db.users.deleteMany({})

// Drop (delete) an entire collection
db.users.drop()

// Drop (delete) a database
db.dropDatabase()
```

### MongoDB Indexes

```javascript
// Create an index

// Create a unique index
db.users.createIndex({ email: 1 }, { unique: true })

// Compound index

```

## 🔴 Redis — The In-Memory Key-Value Store

### What is Redis?

Redis stands for **RE**mote **DI**ctionary **S**erver. It's an **in-memory** key-value store that's incredibly fast because it keeps ALL data in RAM.

```
How fast is Redis?
┌─────────────────────────────────────────┐
│ Traditional DB (MySQL):  1-10 ms        │
│ Redis:                   0.1-0.5 ms     │  ← 10-100x faster!
└─────────────────────────────────────────┘

Why? Because RAM is 100,000x faster than disk.
```

### What is Redis Used For?

Redis is NOT a replacement for your main database. It's a **complement** — used alongside MySQL/PostgreSQL for specific use cases:

| Use Case | How Redis Helps | Example |
|----------|----------------|---------|
| **Caching** | Store frequently accessed data in RAM | Cache database query results, API responses |
| **Session Storage** | Store user sessions | Login sessions for web apps |
| **Rate Limiting** | Track request counts | "Max 100 API calls per minute" |
| **Queues** | Task/job queues | Background email sending, image processing |
| **Real-time Data** | Pub/Sub messaging | Chat apps, live notifications |
| **Leaderboards** | Sorted sets | Gaming leaderboards, trending topics |
| **Counters** | Atomic increment/decrement | Page views, likes, stock counts |

### How Redis Stores Data

Redis stores data as **key-value pairs**. Think of it like a Python dictionary or JavaScript object:

```
Key            →  Value
─────────────────────────────
"user:1:name"  →  "Ritesh"
"user:1:email" →  "ritesh@email.com"
"page:home:views" → "15234"
"session:abc123"  → "{user_id: 1, role: admin}"
```

But Redis supports more than just simple strings:

| Data Type | What It Is | Example |
|-----------|-----------|---------|
| **String** | Simple text or number | `"hello"`, `"42"`, `"true"` |
| **List** | Ordered list of strings | `["task1", "task2", "task3"]` |
| **Set** | Unordered collection of unique strings | `{"apple", "banana", "cherry"}` |
| **Sorted Set** | Set where each element has a score | `{("player1", 100), ("player2", 85)}` |
| **Hash** | A mini key-value store (like a nested object) | `{name: "Ritesh", age: 25}` |
| **Stream** | Append-only log of events | Event streaming, activity logs |

### Redis Commands — By Data Type

#### Strings

```redis
# SET a key-value pair
SET name "Ritesh"

# GET a value by key
GET name
# → "Ritesh"

# SET with expiration (TTL — Time To Live)
SET session:abc123 "user_data_here" EX 3600    # Expires in 3600 seconds (1 hour)
SETEX session:abc123 3600 "user_data_here"     # Same thing, different syntax

# Check remaining TTL
TTL session:abc123
# → 3598 (seconds remaining)

# SET only if key doesn't exist (useful for locks)
SETNX lock:order:123 "locked"

# Increment / Decrement (great for counters)
SET page_views 0
INCR page_views          # → 1
INCR page_views          # → 2
INCRBY page_views 10     # → 12
DECR page_views          # → 11

# Multiple SET and GET
MSET user:1:name "Ritesh" user:1:age "25" user:1:city "Delhi"
MGET user:1:name user:1:age user:1:city
# → 1) "Ritesh"
# → 2) "25"
# → 3) "Delhi"
```

#### Hashes (like Objects/Dictionaries)

```redis
# Set hash fields
HSET user:1 name "Ritesh" email "ritesh@email.com" age "25"

# Get one field
HGET user:1 name
# → "Ritesh"

# Get all fields and values
HGETALL user:1
# → 1) "name"
# → 2) "Ritesh"
# → 3) "email"
# → 4) "ritesh@email.com"
# → 5) "age"
# → 6) "25"

# Check if a field exists
HEXISTS user:1 email
# → 1 (true)

# Delete a field
HDEL user:1 age

# Increment a hash field value
HINCRBY user:1 age 1
```

#### Lists (Ordered, allows duplicates)

```redis
# Push to the left (beginning) of a list
LPUSH tasks "Send email" "Process payment" "Generate report"

# Push to the right (end) of a list
RPUSH tasks "Clean up temp files"

# Get all elements (0 to -1 means start to end)
LRANGE tasks 0 -1
# → 1) "Generate report"
# → 2) "Process payment"
# → 3) "Send email"
# → 4) "Clean up temp files"

# Pop from the left (remove and return first element)
LPOP tasks
# → "Generate report"

# Pop from the right (remove and return last element)
RPOP tasks
# → "Clean up temp files"

# Get list length
LLEN tasks
```

**Use case:** Task queues! Push tasks to one end, pop from the other.

#### Sets (Unique, unordered)

```redis
# Add members to a set
SADD online_users "Ritesh" "Priya" "Rahul"
SADD online_users "Ritesh"    # No effect — already exists!

# Get all members
SMEMBERS online_users
# → 1) "Rahul"
# → 2) "Priya"
# → 3) "Ritesh"

# Check if member exists
SISMEMBER online_users "Ritesh"
# → 1 (true)

# Remove a member
SREM online_users "Rahul"

# Set operations
SADD team_a "Ritesh" "Priya" "Amit"
SADD team_b "Priya" "Rahul" "Sneha"

SINTER team_a team_b       # Intersection → "Priya" (in both)
SUNION team_a team_b       # Union → all unique members
SDIFF team_a team_b        # Difference → "Ritesh", "Amit" (in A but not B)

# Count members
SCARD online_users
```

#### Sorted Sets (Unique, ordered by score)

```redis
# Add members with scores
ZADD leaderboard 100 "Ritesh" 85 "Priya" 92 "Rahul" 78 "Amit"

# Get top 3 (highest scores)
ZREVRANGE leaderboard 0 2 WITHSCORES
# → 1) "Ritesh"   2) "100"
# → 3) "Rahul"    4) "92"
# → 5) "Priya"    6) "85"

# Get rank of a member (0-based, highest first)
ZREVRANK leaderboard "Rahul"
# → 1 (second place)

# Get score of a member
ZSCORE leaderboard "Ritesh"
# → "100"

# Increment score
ZINCRBY leaderboard 15 "Priya"    # Priya now has 100
```

**Use case:** Gaming leaderboards, trending articles, priority queues.

### Key Management Commands

```redis
# Check if a key exists
EXISTS user:1
# → 1 (exists) or 0 (doesn't exist)

# Delete a key
DEL user:1

# Find keys matching a pattern (CAREFUL in production!)
KEYS user:*       # All keys starting with "user:"
KEYS *            # ALL keys (DON'T do this in production!)

# Better alternative for production — SCAN (non-blocking)
SCAN 0 MATCH user:* COUNT 100

# Set expiration on an existing key
EXPIRE user:1 3600          # Expire in 3600 seconds
EXPIREAT user:1 1735689600  # Expire at specific Unix timestamp

# Remove expiration (make key permanent)
PERSIST user:1

# Get the data type of a key
TYPE user:1

# Get total number of keys
DBSIZE

# Delete ALL keys (DANGEROUS!)
FLUSHDB       # Current database
FLUSHALL      # ALL databases
```

### Redis Configuration

```bash
# Config file location
/etc/redis/redis.conf           # Ubuntu/Debian
/etc/redis.conf                 # CentOS/RHEL
```

Key settings:

```ini
# Networking
bind 0.0.0.0                  # Listen on all interfaces (default: 127.0.0.1)
port 6379                      # Default port
protected-mode yes             # Require authentication

# Authentication
requirepass YourStrongPassword123!

# Memory
maxmemory 256mb                # Maximum memory Redis can use
maxmemory-policy allkeys-lru   # When memory is full, evict least recently used keys

# Persistence (saving data to disk)
save 900 1                     # Save if at least 1 key changed in 900 seconds
save 300 10                    # Save if at least 10 keys changed in 300 seconds
save 60 10000                  # Save if at least 10000 keys changed in 60 seconds

# Append Only File (more durable than RDB)
appendonly yes
appendfsync everysec           # Sync to disk every second
```

### Redis Eviction Policies

When Redis runs out of memory, it needs a strategy for what to remove:

| Policy | Behavior |
|--------|----------|
| `noeviction` | Return error when memory is full (default) |
| `allkeys-lru` | Remove least recently used keys from ALL keys |
| `volatile-lru` | Remove least recently used keys that have an expiration set |
| `allkeys-random` | Remove random keys |
| `volatile-ttl` | Remove keys with the shortest TTL first |
| `allkeys-lfu` | Remove least frequently used keys |

**Most common choice:** `allkeys-lru` — automatically removes data that hasn't been accessed recently.

## ⚡ Amazon DynamoDB — The Serverless NoSQL

### What is DynamoDB?

DynamoDB is Amazon's **fully managed**, **serverless** NoSQL database. You don't manage any servers — AWS handles everything (scaling, replication, backups, patching).

```
Traditional Database:                   DynamoDB:
┌─────────────────────────┐            ┌─────────────────────────┐
│ You manage:             │            │ AWS manages:            │
│ ✗ Server setup          │            │ ✓ Server setup          │
│ ✗ OS patching           │            │ ✓ OS patching           │
│ ✗ Scaling               │            │ ✓ Auto-scaling          │
│ ✗ Replication           │            │ ✓ Multi-AZ replication  │
│ ✗ Backups               │            │ ✓ Continuous backups    │
│ ✗ Security patches      │            │ ✓ Security patches      │
│ ✗ Monitoring            │            │ ✓ Built-in monitoring   │
└─────────────────────────┘            └─────────────────────────┘
You: Just read and write data!
```

### DynamoDB Key Concepts

| Concept | Description | SQL Equivalent |
|---------|-------------|---------------|
| **Table** | Collection of items | Table |
| **Item** | A single record | Row |
| **Attribute** | A field in an item | Column |
| **Partition Key** | Primary key for distributing data | Primary Key |
| **Sort Key** | Optional second part of primary key | Composite Key |
| **GSI** | Global Secondary Index | Secondary Index |
| **LSI** | Local Secondary Index | Unique Index |

### DynamoDB Data Model

```
Table: Users
┌─────────────────┬──────────────────┬─────────────┬──────────┐
│ user_id (PK)    │ name             │ email       │ age      │
├─────────────────┼──────────────────┼─────────────┼──────────┤
│ "u001"          │ "Ritesh"         │ "r@e.com"   │ 25       │
│ "u002"          │ "Priya"          │ "p@e.com"   │ 28       │
│ "u003"          │ "Rahul"          │              │ 22       │ ← No email! That's OK
│ "u004"          │ "Sneha"          │ "s@e.com"   │          │ ← No age! Also OK
└─────────────────┴──────────────────┴─────────────┴──────────┘

PK = Partition Key (required, used for data distribution)
```

**Notice:** Items (rows) can have different attributes (columns). Rahul has no email, Sneha has no age. That's perfectly fine in DynamoDB — it's **schemaless**.

### DynamoDB with Partition Key + Sort Key

```
Table: Orders
┌─────────────────┬──────────────────┬─────────────┬──────────┐
│ user_id (PK)    │ order_date (SK)  │ item        │ amount   │
├─────────────────┼──────────────────┼─────────────┼──────────┤
│ "u001"          │ "2024-01-15"     │ "Laptop"    │ 50000    │
│ "u001"          │ "2024-02-20"     │ "Mouse"     │ 500      │
│ "u001"          │ "2024-03-10"     │ "Keyboard"  │ 1500     │
│ "u002"          │ "2024-01-20"     │ "Phone"     │ 30000    │
│ "u002"          │ "2024-02-28"     │ "Case"      │ 300      │
└─────────────────┴──────────────────┴─────────────┴──────────┘

PK = user_id (which user?)
SK = order_date (which order? sorted by date)
Together they form the primary key (PK + SK must be unique)
```

You can now efficiently query: *"Give me all orders for user u001 between January and March 2024"*

### Using DynamoDB with AWS CLI

```bash
# Create a table
aws dynamodb create-table \
    --table-name Users \
    --attribute-definitions \
        AttributeName=user_id,AttributeType=S \
    --key-schema \
        AttributeName=user_id,KeyType=HASH \
    --billing-mode PAY_PER_REQUEST

# Put an item
aws dynamodb put-item \
    --table-name Users \
    --item '{
        "user_id": {"S": "u001"},
        "name": {"S": "Ritesh"},
        "email": {"S": "ritesh@email.com"},
        "age": {"N": "25"}
    }'

# Get an item
aws dynamodb get-item \
    --table-name Users \
    --key '{"user_id": {"S": "u001"}}'

# Query items (requires partition key)
aws dynamodb query \
    --table-name Orders \
    --key-condition-expression "user_id = :uid" \
    --expression-attribute-values '{":uid": {"S": "u001"}}'

# Scan all items (reads entire table — expensive!)
aws dynamodb scan --table-name Users

# Delete an item
aws dynamodb delete-item \
    --table-name Users \
    --key '{"user_id": {"S": "u001"}}'

# Delete a table
aws dynamodb delete-table --table-name Users
```

### Using DynamoDB with Python (boto3)

```python
import boto3

# Create DynamoDB resource
dynamodb = boto3.resource('dynamodb', region_name='ap-south-1')

# Reference a table

    }

    Key={'user_id': 'u001'},

```

## 🏛️ Apache Cassandra — The Wide-Column Store

### What is Cassandra?

Cassandra is a **distributed**, **wide-column** NoSQL database designed for **massive scale**. It was originally built at Facebook to power their inbox search.

Key features:
- **No single point of failure** — every node is equal (no master-slave)
- **Linear scalability** — add more nodes = handle more data
- **Multi-datacenter replication** — built-in
- **Write-optimized** — handles massive write workloads

```
Traditional Master-Slave:            Cassandra (Peer-to-Peer):

    ┌──────────┐                    ┌────┐   ┌────┐   ┌────┐
    │  Master  │                    │Node│───│Node│───│Node│
    └────┬─────┘                    │ 1  │   │ 2  │   │ 3  │
    ┌────┴─────┐                    └──┬─┘   └──┬─┘   └──┬─┘
    │          │                       │        │        │
┌───┴──┐  ┌───┴──┐                 ┌──┴─┐   ┌──┴─┐   ┌──┴─┐
│Slave │  │Slave │                 │Node│───│Node│───│Node│
│  1   │  │  2   │                 │ 4  │   │ 5  │   │ 6  │
└──────┘  └──────┘                 └────┘   └────┘   └────┘

If Master dies → problem!          ANY node can die → no problem!
```

### Cassandra Key Concepts

| Concept | Description |
|---------|-------------|
| **Keyspace** | Like a database — container for tables |
| **Table** | Collection of rows (similar to SQL table) |
| **Partition Key** | Determines which node stores the data |
| **Clustering Key** | Determines the sort order within a partition |
| **Replication Factor** | How many copies of data exist across nodes |
| **Consistency Level** | How many nodes must agree before returning data |

### Cassandra vs SQL Terminology

| SQL | Cassandra |
|-----|-----------|
| Database | Keyspace |
| Table | Table (same!) |
| Row | Row / Partition |
| Primary Key | Partition Key + Clustering Key |
| Index | Secondary Index / Materialized View |

### Using CQL (Cassandra Query Language)

Cassandra uses **CQL** which looks very similar to SQL!

```bash
# Connect to Cassandra
cqlsh
```

```sql
-- Create a keyspace (like a database)
CREATE KEYSPACE myapp 
WITH replication = {
    'class': 'SimpleStrategy',        -- For single datacenter
    'replication_factor': 3           -- 3 copies of data
};

-- Use the keyspace
USE myapp;

-- Create a table
CREATE TABLE users (
    user_id UUID PRIMARY KEY,
    name TEXT,
    email TEXT,
    age INT,
    created_at TIMESTAMP
);

-- Create a table with composite primary key
CREATE TABLE orders (
    user_id UUID,                     -- Partition Key: data for same user on same node
    order_date TIMESTAMP,             -- Clustering Key: orders sorted by date
    order_id UUID,
    product TEXT,
    amount DECIMAL,
    PRIMARY KEY (user_id, order_date) -- (Partition Key, Clustering Key)
) WITH CLUSTERING ORDER BY (order_date DESC);  -- Newest orders first

-- Insert data
INSERT INTO users (user_id, name, email, age, created_at)
VALUES (uuid(), 'Ritesh', 'ritesh@email.com', 25, toTimestamp(now()));

-- Query data
SELECT * FROM users WHERE user_id = <some-uuid>;

-- Query orders for a user (efficient — stays on one partition)
SELECT * FROM orders WHERE user_id = <some-uuid>;

-- Query orders for a user in a date range
SELECT * FROM orders 
WHERE user_id = <some-uuid> 
AND order_date >= '2024-01-01' 
AND order_date <= '2024-12-31';

-- Update data
UPDATE users SET age = 26 WHERE user_id = <some-uuid>;

-- Delete data
DELETE FROM users WHERE user_id = <some-uuid>;
```

### Cassandra Consistency Levels

```
Replication Factor = 3 (3 copies of data on 3 different nodes)

Write with QUORUM:                 Read with QUORUM:
Write to at least 2 of 3 nodes    Read from at least 2 of 3 nodes

  Node 1: ✅ Written               Node 1: ✅ Read "Ritesh"
  Node 2: ✅ Written               Node 2: ✅ Read "Ritesh"    ← 2 agree = return result
  Node 3: ⏳ Will replicate later  Node 3: ⏳ Not read

QUORUM = (Replication Factor / 2) + 1 = (3/2) + 1 = 2
```

| Consistency Level | Meaning | Speed | Accuracy |
|------------------|---------|-------|----------|
| ONE | Write/Read from 1 node | Fastest | Might be stale |
| QUORUM | Majority of nodes must agree | Balanced | Usually consistent |
| ALL | ALL nodes must agree | Slowest | Always consistent |
| LOCAL_QUORUM | Majority in local datacenter | Fast + consistent locally | Best for multi-DC |

## 🧠 The CAP Theorem — Why NoSQL Makes Trade-offs

Every distributed database must choose between three guarantees (and can only have two):

```
             Consistency
                 /\
                /  \
               /    \
              / CP   \
             / (Can't \
            /  have A) \
           /____________\
          /              \
    Partition ─────── Availability
    Tolerance    AP
              (Can't have C)
```

| Property | Meaning | Simple Explanation |
|----------|---------|-------------------|
| **C**onsistency | Every read gets the latest write | All nodes show the same data |
| **A**vailability | Every request gets a response | System always responds, even during failures |
| **P**artition Tolerance | System works even if nodes can't communicate | Network failures don't bring the system down |

**The trade-off:**

| Database | CAP Choice | Meaning |
|----------|-----------|---------|
| MySQL / PostgreSQL | CA | Consistent + Available, but doesn't handle network partitions well |
| MongoDB | CP | Consistent + Partition Tolerant, but may be unavailable during failures |
| Cassandra / DynamoDB | AP | Available + Partition Tolerant, but data might be temporarily inconsistent |
| Redis | CP | Consistent + Partition Tolerant (with Redis Cluster) |

**In the real world:** Partition tolerance is non-negotiable (networks WILL fail), so the real choice is between **Consistency** and **Availability**.
