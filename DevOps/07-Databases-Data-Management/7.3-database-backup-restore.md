> **One Line Answer:** Database backup means making a **copy of your data** so you can recover it if something goes wrong — accidental deletion, hardware failure, ransomware, or any disaster. A backup without a tested restore plan is just a file taking up space.

## 🤔 Why Are Backups So Important?

Let me tell you a scary truth:

```
Things that WILL happen to your database at some point:
├── 💀 Someone accidentally runs DELETE without WHERE clause
├── 🔥 Server hardware fails (disks die, RAM corrupts)
├── 🦠 Ransomware encrypts your data
├── 🌊 Natural disaster takes out your data center
├── 🐛 A bad deployment corrupts data
├── 👤 A disgruntled employee deletes everything
└── 🤦 You run DROP TABLE instead of DROP INDEX
```

**The question isn't IF you'll need a backup. It's WHEN.**

### Real-World Horror Stories

| Company | What Happened | Cost |
|---------|--------------|------|
| GitLab (2017) | Engineer accidentally deleted production database | 6 hours of data lost, major outage |
| Pixar (1998) | Someone ran `rm -rf *` on Toy Story 2 files | Nearly lost the entire movie (saved by a home backup!) |
| MySpace (2019) | Server migration lost 12 years of user uploads | 50 million songs, photos, videos — gone forever |

**Lesson:** If these big companies can lose data, so can you. Backups are NOT optional.

## 📋 Backup Terminology — Know These Terms

| Term | Meaning | Simple Explanation |
|------|---------|-------------------|
| **Full Backup** | Copy of ALL data | Like photocopying every page of a book |
| **Incremental Backup** | Only data changed since LAST backup | Like noting only the pages you edited today |
| **Differential Backup** | Data changed since last FULL backup | Like noting all pages edited since the last full copy |
| **Logical Backup** | SQL dump (human-readable) | Export data as SQL statements (INSERT, CREATE) |
| **Physical Backup** | Raw data files copy | Copy the actual database files from disk |
| **Hot Backup** | Backup while database is running | No downtime — users don't notice |
| **Cold Backup** | Backup while database is stopped | Requires downtime — but simpler and safer |
| **Warm Backup** | Database is in read-only mode during backup | No writes allowed, but reads work |
| **RPO** | Recovery Point Objective | How much data can you afford to LOSE? (e.g., last 1 hour) |
| **RTO** | Recovery Time Objective | How fast must you RECOVER? (e.g., within 30 minutes) |
| **Point-in-Time Recovery (PITR)** | Restore to any specific moment | "Restore to exactly 2:47 PM yesterday" |
| **WAL / Binary Log** | Transaction log | Records every change — allows replaying/undoing changes |

### RPO vs RTO — The Two Most Important Numbers

```
                    Disaster
                    Happens
                       │
  ─────────────────────┼─────────────────────────────>  Time
                       │
  ◄────── RPO ────────►│◄────── RTO ──────────────►
  (Data you might lose) │ (Time to get back online)
                       │
  Last backup ─────────┘└─────── Recovery complete
```

**Example:**
- RPO = 1 hour → You take backups every hour. If disaster strikes, you lose at most 1 hour of data.
- RTO = 30 minutes → Your team must restore the database within 30 minutes.

**Business decides RPO and RTO, DevOps implements them.**

## 🗂️ Types of Backups — Explained

### 1. Full Backup

```
Day 1: Full Backup (100 GB) ─────── [████████████████████]

Pros: Complete copy, simplest to restore
Cons: Takes the most time and storage
```

### 2. Incremental Backup

```
Day 1: Full Backup (100 GB) ────── [████████████████████]
Day 2: Changes since Day 1 (2 GB) ── [██]
Day 3: Changes since Day 2 (3 GB) ── [███]
Day 4: Changes since Day 3 (1 GB) ── [█]
Day 5: Changes since Day 4 (4 GB) ── [████]

To restore Day 5: Need Full + Day 2 + Day 3 + Day 4 + Day 5
Pros: Fast backups, less storage
Cons: Slower restore (need to chain all backups)
```

### 3. Differential Backup

```
Day 1: Full Backup (100 GB) ────── [████████████████████]
Day 2: Changes since Full (2 GB) ── [██]
Day 3: Changes since Full (5 GB) ── [█████]
Day 4: Changes since Full (6 GB) ── [██████]
Day 5: Changes since Full (10 GB) ─ [██████████]

To restore Day 5: Need only Full + Day 5
Pros: Faster restore than incremental (only 2 files needed)
Cons: Backups grow larger each day
```

### Which One to Choose?

| Strategy | Backup Speed | Restore Speed | Storage Used | Best For |
|----------|-------------|---------------|-------------|----------|
| Full Only | Slowest | Fastest | Most | Small databases (< 10 GB) |
| Full + Incremental | Fastest | Slowest | Least | Large databases, frequent backups |
| Full + Differential | Medium | Medium | Medium | Balanced approach |

**Common Strategy:**

```
Weekly Full + Daily Incremental + WAL/Binary Log for PITR

Monday:    FULL backup
Tuesday:   Incremental
Wednesday: Incremental
Thursday:  Incremental
Friday:    Incremental
Saturday:  Incremental
Sunday:    Incremental
Monday:    FULL backup (new cycle)

+ Continuous WAL/Binary log shipping for point-in-time recovery
```

## 🐬 MySQL Backup & Restore

### Method 1: mysqldump (Logical Backup)

`mysqldump` is the most common MySQL backup tool. It creates a `.sql` file containing all the CREATE TABLE and INSERT statements needed to recreate your database.

#### Taking Backups

```bash
# Backup a single database
mysqldump -u root -p myapp_db > /backups/myapp_db_$(date +%Y%m%d_%H%M%S).sql

# What's inside the backup file?
# CREATE TABLE users (...)
# INSERT INTO users VALUES (1, 'Ritesh', 'ritesh@email.com', 25);
# INSERT INTO users VALUES (2, 'Priya', 'priya@email.com', 28);
# ... (every row as an INSERT statement)

# Backup a single database with compression (saves disk space)
mysqldump -u root -p myapp_db | gzip > /backups/myapp_db_$(date +%Y%m%d).sql.gz

# Backup specific tables only
mysqldump -u root -p myapp_db users orders > /backups/myapp_tables.sql

# Backup ALL databases
mysqldump -u root -p --all-databases > /backups/all_databases.sql

# Backup with routines (stored procedures, functions) and events
mysqldump -u root -p --routines --events myapp_db > /backups/myapp_full.sql

# Backup structure only (no data — just CREATE TABLE statements)
mysqldump -u root -p --no-data myapp_db > /backups/myapp_schema.sql

# Backup data only (no CREATE TABLE — just INSERT statements)
mysqldump -u root -p --no-create-info myapp_db > /backups/myapp_data.sql

# Consistent backup (locks tables during dump — use for InnoDB)
mysqldump -u root -p --single-transaction --routines --triggers myapp_db > /backups/myapp_consistent.sql
```

**Important flags explained:**

| Flag | What It Does |
|------|-------------|
| `--single-transaction` | Takes a consistent snapshot without locking tables (InnoDB only) |
| `--routines` | Include stored procedures and functions |
| `--triggers` | Include triggers (enabled by default) |
| `--events` | Include scheduled events |
| `--lock-tables` | Lock all tables before dump (MyISAM) |
| `--flush-logs` | Flush binary logs before dump (useful for PITR) |
| `--master-data=2` | Include binary log position (for replication setup) |

#### Restoring from mysqldump

```bash
# Restore a database
mysql -u root -p myapp_db < /backups/myapp_db_20240115.sql

# Restore from compressed backup
gunzip < /backups/myapp_db_20240115.sql.gz | mysql -u root -p myapp_db

# Restore all databases
mysql -u root -p < /backups/all_databases.sql

# Restore to a DIFFERENT database
mysql -u root -p new_database < /backups/myapp_db_20240115.sql

# Check before restore — what's in this backup?
head -50 /backups/myapp_db_20240115.sql    # See first 50 lines
grep "^CREATE TABLE" /backups/myapp_db_20240115.sql    # List all tables
```

### Method 2: MySQL Binary Log (for Point-in-Time Recovery)

The binary log records every change made to the database. Combined with a full backup, you can restore to any point in time.

#### Enable Binary Logging

```ini
# In /etc/mysql/mysql.conf.d/mysqld.cnf
[mysqld]
server-id = 1
log_bin = /var/log/mysql/mysql-bin
binlog_format = ROW
expire_logs_days = 14          # Keep binary logs for 14 days
max_binlog_size = 100M         # Max size per binary log file
```

```bash
# Restart MySQL after config change
sudo systemctl restart mysql
```

#### Point-in-Time Recovery Steps

```bash
# Step 1: Identify the binary log files
mysql -u root -p -e "SHOW BINARY LOGS;"

# Step 2: Take a full backup with binary log position
mysqldump -u root -p --single-transaction --flush-logs --master-data=2 myapp_db > /backups/full_backup.sql

# Step 3: Find the binary log position in the backup
head -30 /backups/full_backup.sql | grep "CHANGE MASTER"
# Output: -- CHANGE MASTER TO MASTER_LOG_FILE='mysql-bin.000005', MASTER_LOG_POS=154;

# Now let's say disaster happens at 2:30 PM.
# We want to restore to 2:25 PM (just before the disaster)

# Step 4: Restore the full backup
mysql -u root -p myapp_db < /backups/full_backup.sql

# Step 5: Apply binary logs up to 2:25 PM
mysqlbinlog --stop-datetime="2024-01-15 14:25:00" \
    /var/log/mysql/mysql-bin.000005 \
    /var/log/mysql/mysql-bin.000006 | mysql -u root -p myapp_db
```

### Method 3: Physical Backup with Percona XtraBackup

For large databases (hundreds of GBs), `mysqldump` is too slow. **Percona XtraBackup** takes physical backups — it copies the actual data files.

```bash
# Install Percona XtraBackup
sudo apt install percona-xtrabackup-80 -y

# Take a full backup (HOT — no downtime!)
xtrabackup --backup --target-dir=/backups/full_backup --user=root --password=secret

# Prepare the backup (makes it consistent)
xtrabackup --prepare --target-dir=/backups/full_backup

# Restore (must stop MySQL first)
sudo systemctl stop mysql
sudo rm -rf /var/lib/mysql/*                          # Clear data directory
xtrabackup --copy-back --target-dir=/backups/full_backup
sudo chown -R mysql:mysql /var/lib/mysql              # Fix permissions
sudo systemctl start mysql

# Take an incremental backup
xtrabackup --backup --target-dir=/backups/inc1 \
    --incremental-basedir=/backups/full_backup \
    --user=root --password=secret
```

| Method | Speed | Downtime | Size | PITR | Best For |
|--------|-------|----------|------|------|----------|
| mysqldump | Slow | No (with --single-transaction) | Small (SQL text) | With binary logs | Small to medium DBs (< 50 GB) |
| XtraBackup | Fast | No | Large (raw files) | Yes | Large DBs (> 50 GB) |
| Binary Logs | N/A | No | Small | Yes | Continuous protection |

## 🐘 PostgreSQL Backup & Restore

### Method 1: pg_dump (Logical Backup)

`pg_dump` is PostgreSQL's equivalent of `mysqldump`.

#### Taking Backups

```bash
# Backup a single database (as SQL)
pg_dump -U postgres myapp_db > /backups/myapp_db_$(date +%Y%m%d_%H%M%S).sql

# Backup in custom format (compressed, most flexible)
pg_dump -U postgres -Fc myapp_db > /backups/myapp_db.dump

# Backup in directory format (parallel backup — faster for large DBs)
pg_dump -U postgres -Fd -j 4 myapp_db -f /backups/myapp_db_dir/
# -j 4 = use 4 parallel jobs

# Backup with compression
pg_dump -U postgres myapp_db | gzip > /backups/myapp_db.sql.gz

# Backup specific tables
pg_dump -U postgres -t users -t orders myapp_db > /backups/tables.sql

# Backup schema only
pg_dump -U postgres --schema-only myapp_db > /backups/schema.sql

# Backup data only
pg_dump -U postgres --data-only myapp_db > /backups/data.sql

# Backup ALL databases (use pg_dumpall)
pg_dumpall -U postgres > /backups/all_databases.sql

# Backup only global objects (roles, tablespaces)
pg_dumpall -U postgres --globals-only > /backups/globals.sql
```

**pg_dump format options:**

| Format | Flag | Extension | Compressed? | Parallel Restore? | Selective Restore? |
|--------|------|-----------|-------------|-------------------|-------------------|
| Plain SQL | `-Fp` (default) | `.sql` | No | No | No |
| Custom | `-Fc` | `.dump` | Yes | Yes | Yes (specific tables) |
| Directory | `-Fd` | folder | Yes | Yes (backup too!) | Yes |
| Tar | `-Ft` | `.tar` | No | No | Yes |

**Recommendation:** Always use **Custom format** (`-Fc`) for production backups. It's compressed, supports parallel restore, and lets you restore specific tables.

#### Restoring from pg_dump

```bash
# Restore from plain SQL
psql -U postgres myapp_db < /backups/myapp_db.sql

# Restore from custom format (.dump)
pg_restore -U postgres -d myapp_db /backups/myapp_db.dump

# Restore from custom format — create database first
pg_restore -U postgres -C -d postgres /backups/myapp_db.dump
# -C = create the database during restore

# Restore specific tables only
pg_restore -U postgres -d myapp_db -t users -t orders /backups/myapp_db.dump

# Parallel restore (MUCH faster for large databases)
pg_restore -U postgres -d myapp_db -j 4 /backups/myapp_db.dump
# -j 4 = use 4 parallel jobs

# Restore schema only
pg_restore -U postgres -d myapp_db --schema-only /backups/myapp_db.dump

# Restore data only
pg_restore -U postgres -d myapp_db --data-only /backups/myapp_db.dump

# Restore from compressed SQL
gunzip < /backups/myapp_db.sql.gz | psql -U postgres myapp_db

# Clean restore (drop existing objects first, then recreate)
pg_restore -U postgres -d myapp_db --clean /backups/myapp_db.dump
```

### Method 2: PostgreSQL WAL Archiving (for Point-in-Time Recovery)

PostgreSQL's Write-Ahead Log (WAL) is equivalent to MySQL's binary log. It records every change.

#### Enable WAL Archiving

```ini
# In postgresql.conf
wal_level = replica                     # Enable WAL archiving
archive_mode = on                       # Turn on archiving
archive_command = 'cp %p /backups/wal_archive/%f'   # Command to archive WAL files
max_wal_senders = 3                     # Allow streaming replication connections
```

```bash
# Create the archive directory
sudo mkdir -p /backups/wal_archive
sudo chown postgres:postgres /backups/wal_archive

# Restart PostgreSQL
sudo systemctl restart postgresql
```

#### Point-in-Time Recovery Steps

```bash
# Step 1: Take a base backup
pg_basebackup -U postgres -D /backups/base_backup -Ft -z -Xs -P
# -Ft = tar format
# -z  = compress
# -Xs = stream WAL during backup
# -P  = show progress

# Step 2: Wait... disaster happens at 3:00 PM
# We want to restore to 2:55 PM

# Step 3: Stop PostgreSQL
sudo systemctl stop postgresql

# Step 4: Clear the data directory (CAREFUL!)
sudo rm -rf /var/lib/postgresql/16/main/*

# Step 5: Extract the base backup
sudo tar xzf /backups/base_backup/base.tar.gz -C /var/lib/postgresql/16/main/

# Step 6: Create recovery configuration
# PostgreSQL 12+: use postgresql.conf + recovery signal file
sudo touch /var/lib/postgresql/16/main/recovery.signal

# Add to postgresql.conf:
echo "restore_command = 'cp /backups/wal_archive/%f %p'" | \
    sudo tee -a /var/lib/postgresql/16/main/postgresql.auto.conf

echo "recovery_target_time = '2024-01-15 14:55:00'" | \
    sudo tee -a /var/lib/postgresql/16/main/postgresql.auto.conf

# Step 7: Fix permissions and start
sudo chown -R postgres:postgres /var/lib/postgresql/16/main
sudo systemctl start postgresql

# Step 8: PostgreSQL will replay WAL up to 2:55 PM, then pause
# Verify the data is correct, then:
psql -U postgres -c "SELECT pg_wal_replay_resume();"
```

### Method 3: pg_basebackup (Physical Backup)

```bash
# Take a physical backup (streaming)
pg_basebackup -U postgres \
    -D /backups/physical_backup \
    -Ft \          # Tar format
    -z \           # Compress
    -Xs \          # Stream WAL
    -P \           # Show progress
    -R             # Create standby.signal (for setting up a replica)

# This creates:
# /backups/physical_backup/base.tar.gz      — Data files
# /backups/physical_backup/pg_wal.tar.gz    — WAL files
```

## 📄 MongoDB Backup & Restore

### Method 1: mongodump / mongorestore

```bash
# Backup ALL databases
mongodump --uri="mongodb://admin:password@localhost:27017" \
    --out=/backups/mongo_$(date +%Y%m%d)

# Backup a specific database
mongodump --uri="mongodb://admin:password@localhost:27017" \
    --db=myapp_db \
    --out=/backups/mongo_myapp

# Backup a specific collection
mongodump --uri="mongodb://admin:password@localhost:27017" \
    --db=myapp_db \
    --collection=users \
    --out=/backups/mongo_users

# Backup with compression
mongodump --uri="mongodb://admin:password@localhost:27017" \
    --db=myapp_db \
    --gzip \
    --out=/backups/mongo_compressed

# Backup to a single archive file
mongodump --uri="mongodb://admin:password@localhost:27017" \
    --db=myapp_db \
    --archive=/backups/myapp_db.archive

# Backup with oplog (for point-in-time recovery with replica sets)
mongodump --uri="mongodb://admin:password@localhost:27017" \
    --oplog \
    --out=/backups/mongo_pitr
```

### Restoring MongoDB

```bash
# Restore ALL databases
mongorestore --uri="mongodb://admin:password@localhost:27017" \
    /backups/mongo_20240115/

# Restore a specific database
mongorestore --uri="mongodb://admin:password@localhost:27017" \
    --db=myapp_db \
    /backups/mongo_myapp/myapp_db/

# Restore a specific collection
mongorestore --uri="mongodb://admin:password@localhost:27017" \
    --db=myapp_db \
    --collection=users \
    /backups/mongo_users/myapp_db/users.bson

# Restore from compressed backup
mongorestore --uri="mongodb://admin:password@localhost:27017" \
    --gzip \
    /backups/mongo_compressed/

# Restore from archive
mongorestore --uri="mongodb://admin:password@localhost:27017" \
    --archive=/backups/myapp_db.archive

# Restore and DROP existing collections first
mongorestore --uri="mongodb://admin:password@localhost:27017" \
    --drop \
    /backups/mongo_20240115/

# Restore with oplog replay (point-in-time)
mongorestore --uri="mongodb://admin:password@localhost:27017" \
    --oplogReplay \
    /backups/mongo_pitr/
```

## 🔴 Redis Backup & Restore

### Method 1: RDB Snapshots (Default)

Redis automatically saves snapshots of your data to disk (called RDB — Redis Database).

```ini
# In redis.conf — these are the default save rules
save 900 1        # Save if at least 1 key changed in 900 seconds (15 min)
save 300 10       # Save if at least 10 keys changed in 300 seconds (5 min)
save 60 10000     # Save if at least 10000 keys changed in 60 seconds (1 min)

# The snapshot file
dbfilename dump.rdb
dir /var/lib/redis/
```

```bash
# Manually trigger a snapshot
redis-cli BGSAVE
# This runs in the background — doesn't block Redis

# Synchronous save (blocks Redis — don't use in production)
redis-cli SAVE

# Check when last save happened
redis-cli LASTSAVE

# Find the RDB file
redis-cli CONFIG GET dir
redis-cli CONFIG GET dbfilename
# → /var/lib/redis/dump.rdb
```

#### Backup RDB File

```bash
# Simply copy the RDB file
cp /var/lib/redis/dump.rdb /backups/redis_$(date +%Y%m%d_%H%M%S).rdb
```

#### Restore from RDB

```bash
# Step 1: Stop Redis
sudo systemctl stop redis-server

# Step 2: Replace the RDB file
cp /backups/redis_20240115.rdb /var/lib/redis/dump.rdb

# Step 3: Fix permissions
sudo chown redis:redis /var/lib/redis/dump.rdb

# Step 4: Start Redis (it automatically loads the RDB file)
sudo systemctl start redis-server
```

### Method 2: AOF (Append Only File) — More Durable

AOF logs every write command. It's more durable than RDB because you lose at most 1 second of data.

```ini
# In redis.conf
appendonly yes                     # Enable AOF
appendfilename "appendonly.aof"    # AOF file name
appendfsync everysec               # Sync to disk every second (recommended)
# Options: always (safest, slowest), everysec (balanced), no (fastest, least safe)
```

```bash
# Rewrite AOF (compress it — removes redundant commands)
redis-cli BGREWRITEAOF
```

**Best practice:** Use BOTH RDB and AOF together:

```
RDB:  Fast recovery (load the snapshot), but might lose a few minutes of data
AOF:  Slower recovery (replay all commands), but loses at most 1 second
Both: Start with RDB for speed, then replay AOF for the last few missing seconds
```

## 🤖 Automating Backups with Cron Jobs

Manual backups are useless — you'll forget. **Automate everything!**

### MySQL Automated Backup Script

```bash
#!/bin/bash
# /scripts/backup_mysql.sh

# ---- Configuration ----
DB_USER="backup_user"
DB_PASS="backup_password"
DB_NAME="myapp_db"
BACKUP_DIR="/backups/mysql"
RETENTION_DAYS=30
DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="${BACKUP_DIR}/${DB_NAME}_${DATE}.sql.gz"

# ---- Create backup directory if it doesn't exist ----
mkdir -p "${BACKUP_DIR}"

# ---- Take the backup ----
echo "[$(date)] Starting MySQL backup of ${DB_NAME}..."

mysqldump -u "${DB_USER}" -p"${DB_PASS}" \
    --single-transaction \
    --routines \
    --triggers \
    --events \
    "${DB_NAME}" | gzip > "${BACKUP_FILE}"

# ---- Check if backup was successful ----
if [ $? -eq 0 ]; then
    FILESIZE=$(du -h "${BACKUP_FILE}" | cut -f1)
    echo "[$(date)] Backup successful! File: ${BACKUP_FILE} (${FILESIZE})"
else
    echo "[$(date)] ERROR: Backup failed!" >&2
    exit 1
fi

# ---- Delete old backups (older than RETENTION_DAYS) ----
echo "[$(date)] Cleaning up backups older than ${RETENTION_DAYS} days..."
find "${BACKUP_DIR}" -name "*.sql.gz" -mtime +${RETENTION_DAYS} -delete

echo "[$(date)] Backup process completed."
```

### MongoDB Automated Backup Script

```bash
#!/bin/bash
# /scripts/backup_mongo.sh

# ---- Configuration ----
MONGO_URI="mongodb://admin:password@localhost:27017"
DB_NAME="myapp_db"
BACKUP_DIR="/backups/mongodb"
RETENTION_DAYS=30
DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_PATH="${BACKUP_DIR}/${DB_NAME}_${DATE}"

# ---- Create backup directory ----
mkdir -p "${BACKUP_DIR}"

# ---- Take the backup ----
echo "[$(date)] Starting MongoDB backup of ${DB_NAME}..."

mongodump --uri="${MONGO_URI}" \
    --db="${DB_NAME}" \
    --gzip \
    --out="${BACKUP_PATH}"

if [ $? -eq 0 ]; then
    DIRSIZE=$(du -sh "${BACKUP_PATH}" | cut -f1)
    echo "[$(date)] Backup successful! Directory: ${BACKUP_PATH} (${DIRSIZE})"
else
    echo "[$(date)] ERROR: Backup failed!" >&2
    exit 1
fi

# ---- Cleanup old backups ----
find "${BACKUP_DIR}" -maxdepth 1 -type d -mtime +${RETENTION_DAYS} -exec rm -rf {} +

echo "[$(date)] Backup process completed."
```

## ☁️ Offsite Backup — The 3-2-1 Rule

The **3-2-1 backup rule** is the gold standard:

```
3 — Keep at least 3 copies of your data
2 — Store them on 2 different types of media
1 — Keep 1 copy offsite (different physical location)

Example:
Copy 1: Live database (on the server)
Copy 2: Local backup (on a different disk/NAS)
Copy 3: Cloud storage (S3, GCS, Azure Blob)
```

### Sending Backups to AWS S3

```bash
# Install AWS CLI (if not already installed)
sudo apt install awscli -y

# Configure AWS credentials
aws configure
# Enter your AWS Access Key ID, Secret Key, Region

# Upload backup to S3
aws s3 cp /backups/myapp_db_20240115.sql.gz \
    s3://my-database-backups/mysql/myapp_db_20240115.sql.gz

# Sync entire backup directory to S3
aws s3 sync /backups/mysql/ s3://my-database-backups/mysql/

# Set up lifecycle policy (auto-delete old backups from S3)
# Create lifecycle.json:
cat > /tmp/lifecycle.json << 'EOF'
{
    "Rules": [
        {
            "ID": "DeleteOldBackups",
            "Status": "Enabled",
            "Filter": {"Prefix": ""},
            "Expiration": {"Days": 90},
            "Transitions": [
                {
                    "Days": 30,
                    "StorageClass": "STANDARD_IA"
                },
                {
                    "Days": 60,
                    "StorageClass": "GLACIER"
                }
            ]
        }
    ]
}
EOF

aws s3api put-bucket-lifecycle-configuration \
    --bucket my-database-backups \
    --lifecycle-configuration file:///tmp/lifecycle.json
```

**This lifecycle policy:**
- Days 0-30: Store in S3 Standard (frequent access)
- Days 30-60: Move to S3 Infrequent Access (cheaper)
- Days 60-90: Move to Glacier (very cheap, slow retrieval)
- After 90 days: Delete

### Adding S3 Upload to Backup Scripts

```bash
# Add this to the end of your backup script:

# ---- Upload to S3 ----
echo "[$(date)] Uploading to S3..."
aws s3 cp "${BACKUP_FILE}" "s3://my-database-backups/mysql/" --storage-class STANDARD_IA

if [ $? -eq 0 ]; then
    echo "[$(date)] S3 upload successful!"
else
    echo "[$(date)] WARNING: S3 upload failed! Local backup exists at ${BACKUP_FILE}" >&2
fi
```

## 🧪 Testing Your Backups — The Most Important Step!

> **"A backup that hasn't been tested is not a backup — it's a hope."**

### Why Test Backups?

| Reason | What Can Go Wrong |
|--------|-------------------|
| Corrupted backup file | The `.sql` file might be incomplete or damaged |
| Missing tables | Maybe your dump didn't include all tables |
| Version mismatch | Backup taken on MySQL 8.0, trying to restore on MySQL 5.7 |
| Permission issues | Backup doesn't include user grants |
| No one knows how | The person who set up backups left the company |

### How to Test Backups

```bash
# ---- Step 1: Create a test database ----
mysql -u root -p -e "CREATE DATABASE backup_test;"

# ---- Step 2: Restore the backup to the test database ----
mysql -u root -p backup_test < /backups/myapp_db_20240115.sql

# ---- Step 3: Verify the data ----
mysql -u root -p backup_test -e "
    SELECT COUNT(*) AS total_users FROM users;
    SELECT COUNT(*) AS total_orders FROM orders;
    SHOW TABLES;
"

# ---- Step 4: Compare with production ----
# Count should match production
mysql -u root -p myapp_db -e "SELECT COUNT(*) FROM users;"
mysql -u root -p backup_test -e "SELECT COUNT(*) FROM users;"

# ---- Step 5: Clean up ----
mysql -u root -p -e "DROP DATABASE backup_test;"
```

### Automated Backup Verification Script

```bash
#!/bin/bash
# /scripts/verify_backup.sh

BACKUP_FILE=$1
DB_USER="root"
TEST_DB="backup_verification_$(date +%s)"

echo "[$(date)] Starting backup verification..."

# Create test database
mysql -u "${DB_USER}" -p -e "CREATE DATABASE ${TEST_DB};"

# Restore
mysql -u "${DB_USER}" -p "${TEST_DB}" < "${BACKUP_FILE}"

if [ $? -ne 0 ]; then
    echo "FAIL: Restore failed!"
    mysql -u "${DB_USER}" -p -e "DROP DATABASE IF EXISTS ${TEST_DB};"
    exit 1
fi

# Verify tables exist
TABLE_COUNT=$(mysql -u "${DB_USER}" -p -N -e "SELECT COUNT(*) FROM information_schema.tables WHERE table_schema='${TEST_DB}';")

if [ "${TABLE_COUNT}" -gt 0 ]; then
    echo "PASS: ${TABLE_COUNT} tables restored successfully"
else
    echo "FAIL: No tables found in backup!"
    mysql -u "${DB_USER}" -p -e "DROP DATABASE IF EXISTS ${TEST_DB};"
    exit 1
fi

# Cleanup
mysql -u "${DB_USER}" -p -e "DROP DATABASE ${TEST_DB};"

echo "[$(date)] Backup verification completed successfully!"
```

## 📋 Backup Strategy Cheat Sheet

### For Small Apps (< 10 GB database)

```
Strategy:
├── Daily mysqldump/pg_dump (compressed)
├── Upload to S3 nightly
├── Keep 30 days of local backups
├── Keep 90 days on S3
├── Test restore monthly
└── RPO: 24 hours | RTO: 1 hour
```

### For Medium Apps (10-100 GB database)

```
Strategy:
├── Weekly full backup (XtraBackup/pg_basebackup)
├── Daily incremental backups
├── Continuous binary log / WAL archiving
├── Upload to S3 with lifecycle policy
├── Keep 7 days locally, 90 days on S3
├── Point-in-Time Recovery enabled
├── Test restore weekly
└── RPO: 1 hour | RTO: 30 minutes
```

### For Large/Critical Apps (100+ GB database)

```
Strategy:
├── Use managed database service (RDS, Cloud SQL)
├── Automated daily snapshots with 35-day retention
├── Continuous replication to standby in another AZ/region
├── Point-in-Time Recovery enabled
├── Cross-region snapshot copies
├── Automated restore testing (weekly)
├── Runbook documented and tested quarterly
└── RPO: 5 minutes | RTO: 15 minutes
```

## 🎯 Key Takeaways

| Concept | Remember This |
|---------|--------------|
| RPO & RTO | RPO = how much data you can lose. RTO = how fast you recover |
| 3-2-1 Rule | 3 copies, 2 media types, 1 offsite |
| Full vs Incremental | Full = complete copy. Incremental = only changes |
| Logical vs Physical | Logical = SQL dump (portable). Physical = raw files (fast) |
| mysqldump / pg_dump | Standard backup tools for MySQL/PostgreSQL |
| Binary Log / WAL | Enable these for point-in-time recovery |
| Test Your Backups | An untested backup is not a backup |
| Automate Everything | Use cron jobs + scripts + monitoring |
| Cloud Managed | AWS RDS, Atlas, etc. handle backups for you |
| Never Skip Backups | It's not IF you'll need one, it's WHEN |
