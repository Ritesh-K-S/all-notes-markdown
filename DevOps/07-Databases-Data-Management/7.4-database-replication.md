## 1. What is Database Replication?

Imagine you have a notebook where you write all your important notes. Now imagine — what if that notebook gets lost or damaged? All your data is gone! 😱

**Database replication** is the process of **copying data from one database server to one or more other database servers** — so you always have backup copies of your data, and your application can keep running even if one server goes down.

```
Simple Analogy:
─────────────────────────────
You write in Notebook A (Master)
    ↓ (automatic copy)
Notebook B (Slave) gets the same content
Notebook C (Slave) gets the same content

If Notebook A is lost → you still have B and C!
```

In technical terms:

> **Replication** = Keeping multiple copies of the same data on different database servers, automatically synchronized.

## 2. Why Do We Need Replication?

Here are the real-world problems that replication solves:

### Problem 1: Single Point of Failure

```
Without Replication:
─────────────────────────────
[App] → [Single Database Server]
             ↓
        Server dies 💀
             ↓
     Entire app is DOWN ❌
```

### Problem 2: Too Many Read Requests

```
Without Replication:
─────────────────────────────
1000 users reading data  ─→  [One Database] ← Overloaded! Slow! 🐢
50 users writing data    ─→  
```

### Problem 3: Data Loss

If the single server's hard drive fails, your data could be gone forever.

### How Replication Solves These:

| Problem | How Replication Helps |
|---------|----------------------|
| **High Availability** | If one server dies, another takes over instantly |
| **Read Scaling** | Spread read queries across multiple servers |
| **Disaster Recovery** | Data exists on multiple machines, even in different data centers |
| **Geographic Distribution** | Put replicas closer to users for faster response times |
| **Backup Without Downtime** | Take backups from a replica, not the busy master |

## 3. Key Terminology

Before we dive in, let's get the vocabulary straight:

| Term | Meaning |
|------|---------|
| **Master (Primary)** | The main database server that handles **writes** (INSERT, UPDATE, DELETE) |
| **Slave (Replica)** | A copy of the master that receives changes and usually handles **reads** (SELECT) |
| **Replication** | The process of copying data changes from master to slave(s) |
| **Binary Log (binlog)** | MySQL's transaction log — records every change made to the database |
| **WAL (Write-Ahead Log)** | PostgreSQL's transaction log — same idea as binlog |
| **Replication Lag** | The delay between when a change happens on the master and when the slave gets it |
| **Failover** | Promoting a slave to become the new master when the master fails |
| **Switchover** | Planned, graceful swap of master and slave roles |
| **Topology** | The arrangement/pattern of how servers are connected for replication |

> **Note:** The terms "Master-Slave" are being replaced by "Primary-Replica" in many systems for more inclusive language. MySQL 8.0+ uses "Source-Replica", PostgreSQL uses "Primary-Standby". We'll use both terms so you understand all documentation you encounter.

## 4. Types of Replication

There are several ways to categorize replication:

### By Direction of Data Flow

```
1. Master-Slave (One-way)
   ────────────────────────
   [Master] ──writes──→ [Slave 1]
                    ──→ [Slave 2]
   Only Master accepts writes.
   Slaves are read-only copies.

2. Master-Master (Two-way / Multi-directional)
   ────────────────────────
   [Master 1] ←──writes──→ [Master 2]
   Both accept writes.
   Both replicate to each other.
```

### By Timing

| Type | How It Works | Trade-off |
|------|-------------|-----------|
| **Synchronous** | Master waits for slave to confirm it received the data before committing | Slower writes, but guaranteed consistency |
| **Asynchronous** | Master commits immediately, slave catches up later | Faster writes, but possible data lag |
| **Semi-synchronous** | Master waits for at least ONE slave to acknowledge | Balance of speed and safety |

### By What Gets Replicated

| Type | What's Copied |
|------|---------------|
| **Statement-based** | The actual SQL statements (INSERT INTO users VALUES...) |
| **Row-based** | The actual row data that changed |
| **Mixed** | Combination — uses statement-based when possible, row-based when needed |

## 5. Master-Slave Replication (Primary-Replica)

This is the **most common** type of replication. Let's understand it deeply.

### The Concept

```
                    ┌─────────────┐
                    │   Master    │
                    │  (Primary)  │
                    │             │
                    │ Reads ✅    │
                    │ Writes ✅   │
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
              ▼            ▼            ▼
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │  Slave 1 │ │  Slave 2 │ │  Slave 3 │
        │ (Replica)│ │ (Replica)│ │ (Replica)│
        │          │ │          │ │          │
        │ Reads ✅ │ │ Reads ✅ │ │ Reads ✅ │
        │ Writes ❌│ │ Writes ❌│ │ Writes ❌│
        └──────────┘ └──────────┘ └──────────┘
```

### Rules:

1. **All writes** (INSERT, UPDATE, DELETE) go to the **Master only**
2. **Reads** (SELECT) can go to **any server** — Master or Slaves
3. The Master **automatically pushes changes** to all Slaves
4. Slaves are **read-only** — they cannot accept write operations
5. You can have **1 Master and many Slaves** (1 to N relationship)

### How the Application Uses It

```
Application Layer:
──────────────────────

Write Request (INSERT new user)
    └──→ Send to Master

Read Request (SELECT user details)
    └──→ Send to any Slave (via load balancer)
```

This is called **Read-Write Splitting** — a very common pattern.

### Advantages of Master-Slave

| Advantage | Explanation |
|-----------|-------------|
| **Simple** | Easy to understand and set up |
| **Read scaling** | Add more slaves to handle more read traffic |
| **No write conflicts** | Only one server accepts writes, so no conflicts |
| **Backup friendly** | Take backups from slaves without impacting master |
| **Analytics** | Run heavy reports on a slave without slowing down the master |

### Disadvantages of Master-Slave

| Disadvantage | Explanation |
|--------------|-------------|
| **Single point of failure for writes** | If master dies, no writes until a slave is promoted |
| **Replication lag** | Slaves might be slightly behind the master |
| **Write bottleneck** | All writes hit one server — can't scale writes |
| **Failover complexity** | Promoting a slave to master requires careful handling |

## 6. How Master-Slave Replication Works — Step by Step

Let's trace exactly what happens when you run an INSERT on the master:

### MySQL Replication Flow

```
Step 1: Client sends INSERT to Master
─────────────────────────────────────
Client → INSERT INTO users (name) VALUES ('Ritesh') → Master

Step 2: Master executes the query
─────────────────────────────────
Master executes the INSERT and writes the data to its own tables.

Step 3: Master writes to Binary Log (binlog)
─────────────────────────────────────────────
The change is recorded in the master's binary log file.
Binary log = a sequential record of all changes.

Step 4: Slave's IO Thread fetches the binlog
─────────────────────────────────────────────
Each slave has an IO thread that connects to the master
and continuously reads new entries from the master's binlog.

Step 5: Slave writes to its Relay Log
──────────────────────────────────────
The slave stores the fetched binlog events in its own
local relay log file.

Step 6: Slave's SQL Thread replays the changes
───────────────────────────────────────────────
The slave's SQL thread reads the relay log and executes
the same changes on the slave's own database.

Step 7: Slave is now in sync with Master ✅
```

### The Visual Flow:

```
[Client]
   │
   ▼ (INSERT)
[Master]
   │
   ├─→ Execute query on Master's database
   │
   ├─→ Write to Binary Log (binlog)
   │
   ▼
[Slave's IO Thread] ──reads──→ [Master's binlog]
   │
   ▼
[Slave's Relay Log]
   │
   ▼
[Slave's SQL Thread] ──executes──→ [Slave's Database]
```

### PostgreSQL Replication Flow

PostgreSQL uses a slightly different mechanism:

```
Step 1: Client sends INSERT to Primary
Step 2: Primary executes and writes to WAL (Write-Ahead Log)
Step 3: WAL Sender process on Primary streams WAL to Standby
Step 4: WAL Receiver process on Standby receives WAL data
Step 5: Standby replays the WAL entries
Step 6: Standby is now in sync ✅
```

## 7. Setting Up MySQL Master-Slave Replication — Hands-On

Let's do this for real! We'll set up two MySQL servers — one master, one slave.

### Prerequisites

- Two MySQL servers (can be on different machines, VMs, or Docker containers)
- MySQL 8.0+ installed on both
- Network connectivity between them

### Step 1: Configure the Master

Edit the MySQL configuration file on the master:

```bash
# On the Master server
sudo nano /etc/mysql/mysql.conf.d/mysqld.cnf
```

Add or modify these settings:

```ini
[mysqld]
# Unique ID for this server (must be different on each server)
server-id = 1

# Enable binary logging — this is REQUIRED for replication
log_bin = /var/log/mysql/mysql-bin.log

# Which database(s) to replicate (optional — omit to replicate ALL)
# binlog_do_db = myapp_db

# Recommended: Use ROW-based replication (more reliable than STATEMENT)
binlog_format = ROW

# How long to keep binary logs (in seconds) — 7 days
binlog_expire_logs_seconds = 604800

# Sync binlog to disk after every transaction (safer but slightly slower)
sync_binlog = 1
```

**What each setting means:**

```
server-id = 1
→ Every server in replication MUST have a unique ID.
  Master = 1, Slave = 2, etc.

log_bin = /var/log/mysql/mysql-bin.log
→ Turns ON binary logging. Without this, replication is impossible.
  The binlog records every change made to the database.

binlog_format = ROW
→ Records the actual row changes, not the SQL statements.
  More reliable — avoids issues with non-deterministic functions.

sync_binlog = 1
→ Flushes the binlog to disk after every transaction commit.
  Prevents data loss if the server crashes.
```

Restart MySQL on the master:

```bash
sudo systemctl restart mysql
```

### Step 2: Create a Replication User on the Master

The slave needs a user account to connect to the master and read the binlog:

```sql
-- Connect to MySQL on the Master
mysql -u root -p

-- Create a user for replication
-- This user will be used by the slave to connect
CREATE USER 'repl_user'@'%' IDENTIFIED BY 'StrongPassword123!';

-- Grant REPLICATION SLAVE privilege
-- This is a special privilege that allows reading the binlog
GRANT REPLICATION SLAVE ON *.* TO 'repl_user'@'%';

-- Apply the privilege changes
FLUSH PRIVILEGES;
```

**Why `'repl_user'@'%'`?**

```
'repl_user'@'%'
       │       │
       │       └── '%' means this user can connect from ANY host
       └────────── The username
       
In production, replace '%' with the slave's specific IP:
'repl_user'@'192.168.1.20'
```

### Step 3: Get the Master's Binary Log Position

This is critical — the slave needs to know WHERE to start reading from:

```sql
-- Lock tables to get a consistent snapshot
FLUSH TABLES WITH READ LOCK;

-- Get the current binlog file and position
SHOW MASTER STATUS;
```

You'll see output like:

```
+------------------+----------+--------------+------------------+
| File             | Position | Binlog_Do_DB | Binlog_Ignore_DB |
+------------------+----------+--------------+------------------+
| mysql-bin.000003 |      857 |              |                  |
+------------------+----------+--------------+------------------+
```

**Write down these values! You'll need them:**

```
File:     mysql-bin.000003    ← The current binlog file
Position: 857                 ← The current position in that file
```

### Step 4: Export the Master's Data (if the master already has data)

```bash
# In another terminal — DON'T unlock the tables yet!
mysqldump -u root -p --all-databases --master-data > master_dump.sql

# Copy this dump to the slave server
scp master_dump.sql user@slave-server:/tmp/
```

Now unlock the tables on the master:

```sql
-- Back in the MySQL prompt on the master
UNLOCK TABLES;
```

### Step 5: Configure the Slave

Edit the MySQL configuration file on the slave:

```bash
# On the Slave server
sudo nano /etc/mysql/mysql.conf.d/mysqld.cnf
```

```ini
[mysqld]
# Unique server ID — MUST be different from master
server-id = 2

# Enable relay log
relay_log = /var/log/mysql/mysql-relay-bin.log

# Make this slave read-only (safety measure)
read_only = ON

# Optional: Also enable binlog on slave (needed if this slave
# will ever become a master, or for cascading replication)
log_bin = /var/log/mysql/mysql-bin.log
```

Restart MySQL on the slave:

```bash
sudo systemctl restart mysql
```

### Step 6: Import the Master's Data on the Slave

```bash
# On the Slave server
mysql -u root -p < /tmp/master_dump.sql
```

### Step 7: Configure the Slave to Connect to the Master

```sql
-- Connect to MySQL on the Slave
mysql -u root -p

-- Tell the slave where the master is and where to start reading
CHANGE REPLICATION SOURCE TO
    SOURCE_HOST = '192.168.1.10',        -- Master's IP address
    SOURCE_USER = 'repl_user',            -- Replication user we created
    SOURCE_PASSWORD = 'StrongPassword123!', -- Password for that user
    SOURCE_LOG_FILE = 'mysql-bin.000003',  -- From SHOW MASTER STATUS
    SOURCE_LOG_POS = 857;                  -- From SHOW MASTER STATUS
```

**What each parameter means:**

```
SOURCE_HOST      → IP address or hostname of the master server
SOURCE_USER      → The replication user we created in Step 2
SOURCE_PASSWORD  → Password for the replication user
SOURCE_LOG_FILE  → The binlog file name from SHOW MASTER STATUS
SOURCE_LOG_POS   → The position in the binlog file to start reading from
```

> **Note:** In MySQL versions before 8.0.23, the syntax uses `CHANGE MASTER TO` with `MASTER_HOST`, `MASTER_USER`, etc. The new syntax is more inclusive.

### Step 8: Start Replication

```sql
-- Start the slave
START REPLICA;

-- Check if replication is working
SHOW REPLICA STATUS\G
```

### Step 9: Verify — The Most Important Step!

Look for these lines in the output of `SHOW REPLICA STATUS\G`:

```
Replica_IO_Running: Yes      ← The IO thread is running (fetching binlog)
Replica_SQL_Running: Yes     ← The SQL thread is running (applying changes)
Seconds_Behind_Source: 0     ← No lag — slave is caught up!
Last_Error:                  ← Empty = No errors 🎉
```

**If both say "Yes" and there are no errors — congratulations! Replication is working!**

### Step 10: Test It!

```sql
-- On the MASTER, create a test database and table:
CREATE DATABASE test_replication;
USE test_replication;
CREATE TABLE hello (id INT AUTO_INCREMENT PRIMARY KEY, message VARCHAR(100));
INSERT INTO hello (message) VALUES ('Hello from Master!');

-- Now go to the SLAVE and check:
USE test_replication;
SELECT * FROM hello;
```

Expected output on the slave:

```
+----+--------------------+
| id | message            |
+----+--------------------+
|  1 | Hello from Master! |
+----+--------------------+
```

If you see that data on the slave — **replication is working perfectly!** 🎉

## 8. Setting Up PostgreSQL Streaming Replication — Hands-On

PostgreSQL handles replication differently from MySQL. Let's set it up.

### Step 1: Configure the Primary (Master)

Edit `postgresql.conf` on the primary:

```bash
sudo nano /etc/postgresql/16/main/postgresql.conf
```

```ini
# Listen on all interfaces (not just localhost)
listen_addresses = '*'

# WAL (Write-Ahead Log) settings for replication
wal_level = replica          # Minimum level needed for replication
max_wal_senders = 5          # Max number of standby servers that can connect
wal_keep_size = 1024         # Keep 1GB of WAL files for standbys to catch up

# Optional but recommended:
hot_standby = on             # Allow standbys to serve read queries
```

**What these settings mean:**

```
wal_level = replica
→ Tells PostgreSQL to write enough detail in the WAL 
  for replication to work. Default is 'replica' in PG 10+.

max_wal_senders = 5
→ How many standby servers can connect simultaneously 
  to stream WAL data. Set this to at least the number 
  of standbys you plan to have + 1 or 2 extra.

wal_keep_size = 1024
→ Keep 1GB worth of WAL files on the primary.
  If a standby disconnects and reconnects later, 
  it needs these WAL files to catch up.
```

### Step 2: Create a Replication User

```sql
-- Connect to PostgreSQL on the Primary
sudo -u postgres psql

-- Create a user with REPLICATION privilege
CREATE USER repl_user WITH REPLICATION ENCRYPTED PASSWORD 'StrongPassword123!';
```

### Step 3: Allow the Standby to Connect

Edit `pg_hba.conf` (PostgreSQL's access control file):

```bash
sudo nano /etc/postgresql/16/main/pg_hba.conf
```

Add this line:

```
# Allow replication connections from the standby server
# TYPE  DATABASE        USER         ADDRESS            METHOD
host    replication     repl_user    192.168.1.20/32    md5
```

**What this means:**

```
host          → Allow TCP/IP connections
replication   → This is a replication connection (not a regular database)
repl_user     → Only this user can make replication connections
192.168.1.20  → Only from this IP address (the standby server)
md5           → Use password authentication
```

Restart PostgreSQL:

```bash
sudo systemctl restart postgresql
```

### Step 4: Set Up the Standby (Slave)

On the standby server, stop PostgreSQL and wipe the data directory:

```bash
# Stop PostgreSQL on the standby
sudo systemctl stop postgresql

# Remove the existing data directory
sudo rm -rf /var/lib/postgresql/16/main/*
```

Use `pg_basebackup` to copy the primary's data:

```bash
# This copies the entire database from primary to standby
sudo -u postgres pg_basebackup \
    -h 192.168.1.10 \          # Primary's IP address
    -U repl_user \              # Replication user
    -D /var/lib/postgresql/16/main \  # Where to put the data
    -Fp \                       # Plain format
    -Xs \                       # Stream WAL during backup
    -P \                        # Show progress
    -R                          # Create standby.signal and configure recovery
```

**The `-R` flag is magical** — it automatically:
1. Creates a `standby.signal` file (tells PostgreSQL "I'm a standby")
2. Adds replication settings to `postgresql.auto.conf`

### Step 5: Start the Standby

```bash
sudo systemctl start postgresql
```

### Step 6: Verify Replication

On the **Primary**, check connected standbys:

```sql
-- On the Primary
SELECT client_addr, state, sent_lsn, write_lsn, flush_lsn, replay_lsn
FROM pg_stat_replication;
```

Expected output:

```
 client_addr   |   state   |  sent_lsn   | write_lsn   | flush_lsn   | replay_lsn
---------------+-----------+-------------+-------------+-------------+------------
 192.168.1.20  | streaming | 0/3000148   | 0/3000148   | 0/3000148   | 0/3000148
```

**`state = streaming`** means the standby is actively receiving WAL data in real-time. 

### Step 7: Test It!

```sql
-- On the PRIMARY, create test data:
CREATE DATABASE test_repl;
\c test_repl
CREATE TABLE greetings (id SERIAL PRIMARY KEY, msg TEXT);
INSERT INTO greetings (msg) VALUES ('Hello from Primary!');

-- On the STANDBY, check:
\c test_repl
SELECT * FROM greetings;
-- You should see: Hello from Primary!
```

## 9. Master-Master Replication (Multi-Master / Active-Active)

### The Concept

In Master-Master replication, **both servers can accept writes**, and they replicate changes to each other.

```
        ┌─────────────┐          ┌─────────────┐
        │   Master 1  │ ◄──────► │   Master 2  │
        │             │          │             │
        │ Reads ✅    │          │ Reads ✅    │
        │ Writes ✅   │          │ Writes ✅   │
        └─────────────┘          └─────────────┘
              ▲                        ▲
              │                        │
         Some users               Some users
        write here               write here
```

### How It Differs from Master-Slave

```
Master-Slave:
─────────────
Writes → [Master] ──replicates──→ [Slave]
                                    (read-only)

Master-Master:
─────────────
Writes → [Master 1] ◄──replicates──► [Master 2] ← Writes
          (read/write)                 (read/write)
```

### Advantages

| Advantage | Explanation |
|-----------|-------------|
| **No single point of failure for writes** | If one master dies, the other keeps accepting writes |
| **Write scaling (limited)** | Writes can be distributed across both masters |
| **Geographic distribution** | Put one master in US, another in Europe |
| **High availability** | Zero downtime for failover — the other master is already active |

### Disadvantages (and they're serious!)

| Disadvantage | Explanation |
|--------------|-------------|
| **Write conflicts** | What if both masters update the same row at the same time? |
| **Data inconsistency risk** | Conflicts can cause data to diverge between masters |
| **Complex setup** | Much harder to configure and maintain |
| **Auto-increment conflicts** | Both servers might generate the same ID |
| **Not suitable for all workloads** | Only works well when conflicts are rare |

## 10. How Master-Master Replication Works

Master-Master is essentially **two Master-Slave setups in opposite directions**:

```
Direction 1: Master 1 → Master 2
─────────────────────────────────
Master 1 writes to its binlog
    ↓
Master 2's IO thread reads Master 1's binlog
    ↓
Master 2 applies the changes

Direction 2: Master 2 → Master 1
─────────────────────────────────
Master 2 writes to its binlog
    ↓
Master 1's IO thread reads Master 2's binlog
    ↓
Master 1 applies the changes
```

### The Auto-Increment Conflict Problem

This is the #1 issue people hit:

```
Without proper config:
──────────────────────
Master 1: INSERT INTO users → id = 1
Master 2: INSERT INTO users → id = 1  ← CONFLICT! Same ID!

With proper config:
──────────────────
Master 1: Uses ODD IDs  → 1, 3, 5, 7, 9...
Master 2: Uses EVEN IDs → 2, 4, 6, 8, 10...

No conflicts! ✅
```

This is solved with two settings:
- `auto_increment_increment` = 2 (step by 2)
- `auto_increment_offset` = 1 (on Master 1) or 2 (on Master 2)

## 11. Setting Up MySQL Master-Master Replication — Hands-On

We'll set up two MySQL servers that replicate to each other.

### Step 1: Configure Master 1

```bash
sudo nano /etc/mysql/mysql.conf.d/mysqld.cnf
```

```ini
[mysqld]
server-id = 1
log_bin = /var/log/mysql/mysql-bin.log
binlog_format = ROW

# Auto-increment settings to avoid ID conflicts
auto_increment_increment = 2    # Step by 2
auto_increment_offset = 1       # Start with ODD numbers (1, 3, 5...)

# Prevent infinite replication loops
log_slave_updates = ON          # Log changes received via replication
                                # (needed so they get sent to the other master)
```

### Step 2: Configure Master 2

```bash
sudo nano /etc/mysql/mysql.conf.d/mysqld.cnf
```

```ini
[mysqld]
server-id = 2
log_bin = /var/log/mysql/mysql-bin.log
binlog_format = ROW

# Auto-increment settings to avoid ID conflicts
auto_increment_increment = 2    # Step by 2
auto_increment_offset = 2       # Start with EVEN numbers (2, 4, 6...)

# Prevent infinite replication loops
log_slave_updates = ON
```

### Step 3: Create Replication Users on BOTH Servers

On **Master 1**:

```sql
CREATE USER 'repl_user'@'%' IDENTIFIED BY 'StrongPass1!';
GRANT REPLICATION SLAVE ON *.* TO 'repl_user'@'%';
FLUSH PRIVILEGES;
SHOW MASTER STATUS;  -- Note the File and Position
```

On **Master 2**:

```sql
CREATE USER 'repl_user'@'%' IDENTIFIED BY 'StrongPass2!';
GRANT REPLICATION SLAVE ON *.* TO 'repl_user'@'%';
FLUSH PRIVILEGES;
SHOW MASTER STATUS;  -- Note the File and Position
```

### Step 4: Set Up Replication in Both Directions

On **Master 1** — point to Master 2:


On **Master 2** — point to Master 1:

```sql
CHANGE REPLICATION SOURCE TO
    SOURCE_HOST = '192.168.1.10',         -- Master 1's IP
    SOURCE_USER = 'repl_user',
    SOURCE_PASSWORD = 'StrongPass1!',
    SOURCE_LOG_FILE = 'mysql-bin.000001',  -- From Master 1's SHOW MASTER STATUS
    SOURCE_LOG_POS = 154;                  -- From Master 1's SHOW MASTER STATUS

START REPLICA;
```

### Step 5: Verify on Both Servers

```sql
-- On both servers, check:

-- Look for:
-- Replica_IO_Running: Yes
-- Replica_SQL_Running: Yes
```

## 12. Synchronous vs Asynchronous Replication

This is one of the most important concepts to understand:

### Asynchronous Replication (Default in most databases)

```
Timeline:
─────────────────────────────────────────────────────────
Client → WRITE → Master → Commits → Returns "SUCCESS" to client
                   │
                   └──→ (Meanwhile, in background) → Slave receives the data
                        (might be milliseconds or seconds later)
```

**The master does NOT wait for the slave.** It commits the transaction and tells the client "done!" immediately. The slave gets the data a bit later.

```
Pros:
  ✅ Fast writes (no waiting)
  ✅ Master is not affected if a slave is slow or offline

Cons:
  ❌ If master crashes RIGHT AFTER committing but BEFORE slave receives it
     → That transaction is LOST on the slave
  ❌ Slave might serve stale (outdated) data
```

### Synchronous Replication

```
Timeline:
─────────────────────────────────────────────────────────
Client → WRITE → Master → Sends to Slave → Slave confirms 
                                                 → Master commits 
                                                 → Returns "SUCCESS" to client
```

**The master WAITS for the slave** to confirm it received (and sometimes applied) the data before committing.

```
Pros:
  ✅ Zero data loss — slave always has the latest data
  ✅ Perfect for critical financial or medical systems

Cons:
  ❌ Slower writes (must wait for network round-trip)
  ❌ If slave is slow or offline → master is stuck waiting!
  ❌ Higher latency, especially for geographically distant slaves
```

### Visual Comparison

```
Asynchronous:
Client ──→ Master ──→ "Done!" ──→ Client happy
                  └──→ Slave (catches up later)

Synchronous:
Client ──→ Master ──→ waits... ──→ Slave says "Got it!" ──→ Master commits ──→ "Done!"
```

## 13. Semi-Synchronous Replication

This is the **Goldilocks option** — not too slow, not too risky.

```
How it works:
─────────────────────────────────────────────────────────
Client → WRITE → Master → Sends to Slave(s)
                             → At least ONE slave confirms receipt
                             → Master commits
                             → Returns "SUCCESS"

Note: The slave confirms it RECEIVED the data,
      not that it APPLIED (replayed) it.
```

### Setting Up Semi-Synchronous in MySQL

```sql
-- On the Master:
INSTALL PLUGIN rpl_semi_sync_master SONAME 'semisync_master.so';
SET GLOBAL rpl_semi_sync_master_enabled = 1;
SET GLOBAL rpl_semi_sync_master_timeout = 10000;  -- 10 seconds timeout

-- On the Slave:
INSTALL PLUGIN rpl_semi_sync_slave SONAME 'semisync_slave.so';
SET GLOBAL rpl_semi_sync_slave_enabled = 1;
STOP REPLICA;
START REPLICA;
```

**What happens if the slave doesn't respond?**

If no slave confirms within the timeout period, the master **falls back to asynchronous** replication. This prevents the master from hanging forever.

```
Semi-Sync Behavior:
────────────────────
Slave responds within timeout → Semi-synchronous (safe) ✅
Slave doesn't respond in time → Falls back to async (fast) ⚠️
```

## 14. Replication Lag — The Silent Killer

### What is Replication Lag?

Replication lag is the **delay** between when data is written on the master and when it appears on the slave.

```
Master:  INSERT at 10:00:00.000
Slave:   Receives INSERT at 10:00:00.500
Lag:     500 milliseconds

Problem scenario:
─────────────────
1. User updates their profile (goes to Master)
2. User immediately views their profile (goes to Slave)
3. Slave hasn't received the update yet!
4. User sees OLD data → "My update didn't save!" 😤
```

### Causes of Replication Lag

| Cause | Explanation |
|-------|-------------|
| **Heavy writes on master** | Slave can't keep up with the volume of changes |
| **Slow slave hardware** | Slave has weaker CPU, less memory, or slower disk |
| **Network latency** | Master and slave are far apart (different data centers) |
| **Long-running queries on slave** | Heavy SELECT queries slow down the SQL thread |
| **Large transactions** | A huge UPDATE affecting millions of rows takes time to replay |
| **Single-threaded replication** | MySQL traditionally replays changes with one thread |

### How to Monitor Replication Lag

**MySQL:**

```sql
SHOW REPLICA STATUS\G

-- Look for:
-- Seconds_Behind_Source: 0    ← 0 means caught up!
-- Seconds_Behind_Source: 120  ← 2 minutes behind! 😱
```

**PostgreSQL:**

```sql
-- On the Primary:
SELECT client_addr,
       state,
       pg_wal_lsn_diff(pg_current_wal_lsn(), replay_lsn) AS lag_bytes
FROM pg_stat_replication;

-- On the Standby:
SELECT now() - pg_last_xact_replay_timestamp() AS replication_lag;
```

### How to Reduce Replication Lag

```
1. Use faster hardware on slaves (SSDs, more RAM)
2. Enable parallel replication (MySQL 5.7+):
   SET GLOBAL replica_parallel_workers = 4;

3. Reduce heavy queries on slaves
4. Use semi-synchronous replication
5. Monitor and alert on lag > threshold
```

## 15. Conflict Resolution in Multi-Master

When two masters accept writes simultaneously, conflicts can happen:

### Types of Conflicts

```
1. INSERT Conflict (same primary key)
   Master 1: INSERT INTO users (id, name) VALUES (5, 'Alice')
   Master 2: INSERT INTO users (id, name) VALUES (5, 'Bob')
   → Which one wins?

2. UPDATE Conflict (same row)
   Master 1: UPDATE users SET email = 'alice@new.com' WHERE id = 1
   Master 2: UPDATE users SET email = 'alice@other.com' WHERE id = 1
   → Which email should it be?

3. DELETE vs UPDATE Conflict
   Master 1: DELETE FROM users WHERE id = 1
   Master 2: UPDATE users SET name = 'New Name' WHERE id = 1
   → Delete it or update it?
```

### Conflict Resolution Strategies

| Strategy | How It Works |
|----------|-------------|
| **Last Write Wins (LWW)** | The most recent timestamp wins — simple but can lose data |
| **Server Priority** | One server always wins — e.g., Master 1 overrides Master 2 |
| **Application-Level** | The application detects conflicts and asks the user to resolve |
| **Avoid Conflicts** | Partition data so each master writes to different tables/rows |
| **CRDTs** | Conflict-free Replicated Data Types — mathematically guaranteed conflict resolution |

### Best Practice: Avoid Conflicts!

The best conflict resolution is **no conflicts at all**. Design your system so that:

```
Strategy 1: Geographic Partitioning
────────────────────────────────────
US users → always write to Master US
EU users → always write to Master EU

Strategy 2: Table Partitioning
──────────────────────────────
Orders  → always write to Master 1
Products → always write to Master 2

Strategy 3: Use Master-Master for HA only
─────────────────────────────────────────
Both masters exist, but only ONE accepts writes at a time.
The other is a hot standby for instant failover.
(This is called Active-Passive, not true Active-Active)
```

## 16. Failover and Promotion

### What is Failover?

When the master dies, you need to **promote a slave to become the new master**. This is called failover.

```
Before Failover:
────────────────
[Master] ← dies 💀
   ├──→ [Slave 1]
   └──→ [Slave 2]

After Failover:
───────────────
[Slave 1] ← promoted to NEW Master! 🎉
   └──→ [Slave 2] ← now replicates from Slave 1
```

### Manual Failover Steps (MySQL)

```sql
-- Step 1: On the slave you want to promote, stop replication
STOP REPLICA;
RESET REPLICA ALL;

-- Step 2: Make it writable
SET GLOBAL read_only = OFF;

-- Step 3: Point other slaves to the new master
-- On Slave 2:
STOP REPLICA;
CHANGE REPLICATION SOURCE TO
    SOURCE_HOST = 'new_master_ip',
    SOURCE_USER = 'repl_user',
    SOURCE_PASSWORD = 'password',
    SOURCE_LOG_FILE = '...',
    SOURCE_LOG_POS = ...;
START REPLICA;

-- Step 4: Update application configuration to point to new master
```

### Automated Failover Tools

In production, you should **never do manual failover**. Use automated tools:

| Tool | Database | What It Does |
|------|----------|-------------|
| **MHA (Master High Availability)** | MySQL | Monitors master, auto-promotes fastest slave |
| **Orchestrator** | MySQL | Auto-failover + topology management + web UI |
| **ProxySQL** | MySQL | SQL-aware proxy with automatic failover |
| **Patroni** | PostgreSQL | Auto-failover + leader election using etcd/ZooKeeper |
| **repmgr** | PostgreSQL | Replication management + auto-failover |
| **pg_auto_failover** | PostgreSQL | Microsoft's automated HA solution |

### Patroni Example (PostgreSQL)

Patroni is the gold standard for PostgreSQL HA:

```yaml
# /etc/patroni/patroni.yml
scope: my-postgres-cluster
name: node1

restapi:
  listen: 0.0.0.0:8008
  connect_address: 192.168.1.10:8008

etcd:
  hosts: 192.168.1.100:2379

bootstrap:
  dcs:
    ttl: 30
    loop_wait: 10
    retry_timeout: 10
    maximum_lag_on_failover: 1048576  # 1MB max lag for failover
    postgresql:
      use_pg_rewind: true
      parameters:
        max_connections: 100
        max_wal_senders: 5
        wal_level: replica
        hot_standby: "on"

  initdb:
    - encoding: UTF8
    - data-checksums

postgresql:
  listen: 0.0.0.0:5432
  connect_address: 192.168.1.10:5432
  data_dir: /var/lib/postgresql/16/main
  authentication:
    replication:
      username: repl_user
      password: StrongPassword123!
    superuser:
      username: postgres
      password: PostgresPass123!
```

**How Patroni works:**

```
                    ┌──────────┐
                    │   etcd   │ ← Distributed consensus store
                    │          │   (who is the leader?)
                    └────┬─────┘
                         │
           ┌─────────────┼─────────────┐
           │             │             │
    ┌──────┴──────┐ ┌────┴────┐ ┌─────┴─────┐
    │  Patroni 1  │ │Patroni 2│ │ Patroni 3 │
    │  (Leader)   │ │(Replica)│ │ (Replica) │
    │  PostgreSQL │ │PostgreSQL│ │PostgreSQL │
    └─────────────┘ └─────────┘ └───────────┘

If Leader dies:
→ etcd detects missing heartbeat
→ Patroni holds an election
→ Most up-to-date replica becomes new leader
→ Other replicas point to new leader
→ All automatic! 🎉
```

## 17. Replication in NoSQL Databases

It's not just SQL databases that do replication! Let's look at a few popular NoSQL databases:

### MongoDB Replica Sets

MongoDB has **built-in replication** called Replica Sets:

```
          ┌──────────────┐
          │   Primary    │ ← Accepts reads and writes
          └──────┬───────┘
                 │
        ┌────────┼────────┐
        │                 │
┌───────┴───────┐ ┌───────┴───────┐
│  Secondary 1  │ │  Secondary 2  │  ← Read-only replicas
└───────────────┘ └───────────────┘

If Primary dies → automatic election → one Secondary becomes Primary
```

Setting up a MongoDB Replica Set:

```javascript
// Connect to MongoDB
mongosh

// Initiate a replica set
rs.initiate({
  _id: "myReplicaSet",
  members: [
    { _id: 0, host: "mongo1:27017" },   // This will be Primary initially
    { _id: 1, host: "mongo2:27017" },   // Secondary
    { _id: 2, host: "mongo3:27017" }    // Secondary
  ]
})

// Check status
rs.status()
```

### Redis Replication

Redis uses a simple master-slave model:

```bash
# On the Replica, edit redis.conf:
replicaof 192.168.1.10 6379     # Point to the master
masterauth YourMasterPassword    # Master's password (if set)

# Or do it at runtime:
redis-cli
> REPLICAOF 192.168.1.10 6379
```

Redis also has **Redis Sentinel** for automated failover and **Redis Cluster** for sharding + replication.

### Cassandra Replication

Cassandra is special — it's **masterless** (peer-to-peer). Every node can accept reads and writes:

```
   ┌─────┐     ┌─────┐     ┌─────┐
   │Node1│ ←─► │Node2│ ←─► │Node3│
   └──┬──┘     └──┬──┘     └──┬──┘
      │            │            │
      └────────────┼────────────┘
              (Ring topology)

All nodes are equal. No master, no slave.
Replication factor = 3 means every piece of data
is stored on 3 different nodes.
```

```sql
-- Cassandra: Set replication when creating a keyspace
CREATE KEYSPACE my_app
WITH replication = {
    'class': 'NetworkTopologyStrategy',
    'datacenter1': 3,   -- 3 replicas in datacenter1
    'datacenter2': 2    -- 2 replicas in datacenter2
};
```

## 18. Master-Slave vs Master-Master — Comparison Table

| Feature | Master-Slave | Master-Master |
|---------|-------------|---------------|
| **Write nodes** | 1 (Master only) | Multiple (all masters) |
| **Read scaling** | ✅ Excellent | ✅ Excellent |
| **Write scaling** | ❌ No | ⚠️ Limited |
| **Complexity** | Simple | Complex |
| **Conflict risk** | None | High |
| **Failover** | Manual or semi-automatic | Instant (other master takes over) |
| **Data consistency** | Strong | Eventual (with possible conflicts) |
| **Best for** | Read-heavy workloads | Write HA requirements |
| **Setup difficulty** | Easy | Hard |
| **Common in production** | Very common | Less common (usually Active-Passive) |
| **Examples** | Most web apps | Active-Passive for HA, geo-distributed apps |

### When to Use What

```
Use Master-Slave when:
  ✅ You have a read-heavy workload (90% reads, 10% writes)
  ✅ You need simple, reliable replication
  ✅ You can tolerate a brief write outage during failover
  ✅ You want to run analytics/reports on a replica

Use Master-Master when:
  ✅ You need zero-downtime write availability
  ✅ You have geographically distributed users
  ✅ You can design your schema to avoid conflicts
  ⚠️ But consider using Active-Passive instead of true Active-Active
```

## 19. Replication Topologies

Beyond simple one-to-one setups, here are common replication patterns:

### Star Topology (One Master, Multiple Slaves)

```
              [Master]
             /   |   \
            /    |    \
      [Slave1] [Slave2] [Slave3]

Most common. Simple and effective.
```

### Chain/Cascade Topology

```
[Master] → [Slave 1] → [Slave 2] → [Slave 3]

Each slave replicates from the one before it.
Reduces load on master but increases total lag.
```

### Tree Topology

```
              [Master]
             /        \
        [Slave 1]   [Slave 2]
        /      \        |
   [Slave 3] [Slave 4] [Slave 5]

Intermediate slaves relay to further slaves.
Good for large-scale deployments.
```

### Ring Topology (Multi-Master)

```
[Master 1] → [Master 2] → [Master 3]
     ↑                         │
     └─────────────────────────┘

Each node replicates to the next in a circle.
Used in some Galera/Group Replication setups.
```

## 20. Monitoring Replication Health

**Always monitor replication!** Here are the key things to watch:

### MySQL Monitoring Queries

```sql
-- Check replication status
SHOW REPLICA STATUS\G

-- Key metrics to monitor:
-- 1. Replica_IO_Running: Must be 'Yes'
-- 2. Replica_SQL_Running: Must be 'Yes'
-- 3. Seconds_Behind_Source: Should be 0 or very low
-- 4. Last_Error: Should be empty
-- 5. Retrieved_Gtid_Set vs Executed_Gtid_Set: Should be close

-- Quick health check:
SELECT 
    CHANNEL_NAME,
    SERVICE_STATE AS io_thread,
    LAST_ERROR_MESSAGE
FROM performance_schema.replication_connection_status;
```

### PostgreSQL Monitoring Queries

```sql
-- On the Primary — check connected standbys:
SELECT 
    client_addr,
    state,
    sync_state,
    pg_wal_lsn_diff(pg_current_wal_lsn(), replay_lsn) AS lag_bytes,
    pg_wal_lsn_diff(pg_current_wal_lsn(), replay_lsn) / 1024 / 1024 AS lag_mb
FROM pg_stat_replication;

-- On the Standby — check lag:
SELECT 
    now() - pg_last_xact_replay_timestamp() AS replication_lag,
    pg_is_in_recovery() AS is_standby;
```

## 21. Common Replication Problems and Fixes

### Problem 1: Replication Stopped — Duplicate Key Error

```
Error: Could not execute Write_rows event: Duplicate entry '12345' for key 'PRIMARY'
```

**Fix:**

```sql
-- Option 1: Skip the problematic transaction (use cautiously!)
STOP REPLICA;
SET GLOBAL sql_replica_skip_counter = 1;
START REPLICA;

-- Option 2: Delete the conflicting row on the slave, then restart
-- Find the conflicting row, delete it, then:
START REPLICA;
```

### Problem 2: Slave IO Thread Not Running

```
Replica_IO_Running: No
Last_IO_Error: error connecting to master...
```

**Common causes:**
- Wrong master IP, username, or password
- Firewall blocking port 3306
- Master's binlog has been purged

**Fix:**

```bash
# Check connectivity
mysql -h master_ip -u repl_user -p

# Check firewall
telnet master_ip 3306
```

### Problem 3: Slave SQL Thread Not Running

```
Replica_SQL_Running: No
Last_SQL_Error: Error executing row event: 'Table doesn't exist'
```

**Fix:** The slave is missing a table that exists on the master. Re-sync the slave:

```bash
# Take a fresh dump from master and re-initialize the slave
mysqldump -u root -p --all-databases --master-data=2 > fresh_dump.sql
```

### Problem 4: High Replication Lag

```sql
-- Enable parallel replication (MySQL 5.7+)
SET GLOBAL replica_parallel_workers = 4;
SET GLOBAL replica_parallel_type = 'LOGICAL_CLOCK';

-- Increase InnoDB buffer pool on slave
SET GLOBAL innodb_buffer_pool_size = 4294967296;  -- 4GB
```

## 22. Best Practices

### Design Best Practices

```
1. Always use GTID-based replication (Global Transaction Identifiers)
   → Makes failover and re-pointing slaves much easier

2. Use ROW-based binlog format
   → More reliable than STATEMENT-based

3. Make slaves read-only
   → SET GLOBAL read_only = ON;
   → SET GLOBAL super_read_only = ON;

4. Monitor replication lag continuously
   → Set up alerts for lag > 30 seconds

5. Use automated failover tools (Orchestrator, Patroni)
   → Never rely on manual failover in production

6. Test failover regularly
   → Chaos engineering — kill the master intentionally
   → Verify failover works as expected
```

### Security Best Practices

```
1. Use SSL/TLS for replication traffic
   CHANGE REPLICATION SOURCE TO ... SOURCE_SSL = 1;

2. Restrict replication user to specific IPs
   CREATE USER 'repl'@'192.168.1.20' ...

3. Use strong passwords for replication users

4. Don't grant unnecessary privileges
   → REPLICATION SLAVE is the ONLY privilege needed
```

### Operational Best Practices

```
1. Keep master and slave MySQL versions compatible
2. Use the same storage engine (InnoDB) on master and slave
3. Ensure slaves have enough disk space
4. Regularly verify data consistency between master and slave
   → Use pt-table-checksum from Percona Toolkit
5. Keep binary logs for at least 7 days
6. Document your replication topology
```
