## 1. The Problem — Why Database Migrations Exist

Imagine this nightmare scenario:

```
Monday:
────────
Developer A: "I added a new column 'email' to the users table on my laptop."

Tuesday:
────────
Developer B: "Wait, where's the 'email' column? I don't have it. My app is crashing."

Wednesday:
──────────
Ops team: "We deployed to production. The app is broken because the 
           'email' column doesn't exist in the production database!"

Thursday:
─────────
Manager: "Who changed the database? What was the schema 3 weeks ago? 
          Can we roll back? NOBODY KNOWS! 😱"
```

### The Core Problems

```
1. No Version Control for Databases
   → Code is in Git, but database changes are... where? 
     In someone's head? In a random SQL file? Nowhere?

2. No Consistency Across Environments
   → Dev database ≠ Staging database ≠ Production database
   → "It works on my machine!" 🤷

3. No Audit Trail
   → Who changed what? When? Why? Nobody knows.

4. Risky Deployments
   → Applying SQL scripts manually is error-prone.
   → "Did we already run this script on production?"

5. No Easy Rollback
   → Something went wrong? Good luck reverting manually!
```

**Database migrations solve ALL of these problems.**

## 2. What is a Database Migration?

A **database migration** is a **version-controlled, incremental change** to your database schema (or data). 

Think of it like Git commits, but for your database:

```
Git (for code):                    Migrations (for databases):
──────────────                     ──────────────────────────
Commit 1: Initial code             V1: CREATE TABLE users
Commit 2: Add login feature        V2: ADD COLUMN email to users
Commit 3: Fix bug in login         V3: CREATE TABLE orders
Commit 4: Add profile page         V4: ADD INDEX on orders.user_id

Each commit is:                    Each migration is:
  - Versioned                        - Versioned (V1, V2, V3...)
  - Ordered                          - Ordered (runs in sequence)
  - Trackable                        - Trackable (tool records what's run)
  - Reversible (git revert)          - Reversible (rollback/undo)
```

### What a Migration File Looks Like

```sql
-- V2__add_email_to_users.sql

ALTER TABLE users ADD COLUMN email VARCHAR(255);
CREATE INDEX idx_users_email ON users(email);
```

That's it! A simple SQL file with a version number. The migration tool handles the rest.

## 3. Key Concepts and Terminology

| Term | Meaning |
|------|---------|
| **Migration** | A single change to the database (one SQL file = one migration) |
| **Schema** | The structure of your database (tables, columns, indexes, constraints) |
| **Version** | A number or identifier for each migration (V1, V2, V3...) |
| **Changelog** | The ordered list of all migrations (Liquibase terminology) |
| **ChangeSet** | A single unit of change in Liquibase (like one migration) |
| **Baseline** | The starting point — takes an existing database and marks it as "V1" |
| **Migrate / Update** | Apply pending migrations to bring the database up to date |
| **Rollback / Undo** | Reverse a migration, going back to a previous version |
| **History Table** | A table the tool creates in YOUR database to track which migrations have been applied |
| **Checksum** | A hash of each migration file — detects if someone modified an already-applied migration |
| **Pending** | A migration that exists as a file but hasn't been applied to the database yet |

## 4. How Database Migrations Work — The Big Picture

Every migration tool follows the same basic flow:

```
Step 1: Developer writes a migration file
──────────────────────────────────────────
V3__create_orders_table.sql

Step 2: Developer commits it to Git
────────────────────────────────────
git add V3__create_orders_table.sql
git commit -m "Add orders table"

Step 3: Migration tool checks the database
──────────────────────────────────────────
Tool: "Let me check the history table..."
Tool: "V1 ✅ applied, V2 ✅ applied, V3 ❌ not yet"
Tool: "I need to run V3!"

Step 4: Migration tool executes V3
──────────────────────────────────
Tool executes: CREATE TABLE orders (...)
Tool records: "V3 applied at 2025-01-15 14:30:00" in the history table

Step 5: Database is now up to date ✅
```

### The History Table

This is the **magic** of migration tools. They create a special table in your database:

```
Flyway's history table: flyway_schema_history
Liquibase's history table: databasechangelog

Example (Flyway):
┌─────────┬─────────────────────────────┬────────┬─────────────────────┐
│ version │ description                 │ success│ installed_on        │
├─────────┼─────────────────────────────┼────────┼─────────────────────┤
│ 1       │ create users table          │ true   │ 2025-01-10 09:00:00 │
│ 2       │ add email to users          │ true   │ 2025-01-12 11:30:00 │
│ 3       │ create orders table         │ true   │ 2025-01-15 14:30:00 │
└─────────┴─────────────────────────────┴────────┴─────────────────────┘
```

The tool looks at this table to know:
- Which migrations have already been applied
- Which migrations are pending
- Whether any applied migrations have been tampered with (checksum)

## 5. Flyway — Overview and Philosophy

**Flyway** is one of the most popular database migration tools. Created by Redgate, it's known for being **simple and convention-driven**.

### Flyway's Philosophy

```
"Simple, focused, and powerful."

Flyway believes in:
  1. SQL-first — write migrations as plain SQL files
  2. Convention over configuration — follow naming conventions, and it just works
  3. Simplicity — minimal config needed to get started
  4. No XML/YAML required — just SQL files with proper names
```

### Key Features

| Feature | Description |
|---------|-------------|
| **SQL-based migrations** | Write plain SQL — no proprietary format needed |
| **Java-based migrations** | For complex migrations, write Java code |
| **Versioned migrations** | Numbered, sequential, run once |
| **Repeatable migrations** | Run every time they change (good for views, stored procs) |
| **Baseline** | Adopt Flyway on an existing database |
| **Validate** | Check that applied migrations haven't been modified |
| **Undo** | Roll back migrations (paid Teams/Enterprise feature) |
| **Checksum verification** | Detects if migration files were changed after being applied |

### Supported Databases

```
PostgreSQL, MySQL, MariaDB, SQL Server, Oracle,
SQLite, H2, CockroachDB, Amazon RDS, Azure SQL,
Google Cloud SQL, Snowflake, and 20+ more.
```

## 7. Flyway — Writing Your First Migration

### The Naming Convention (Critical!)

Flyway uses a strict **naming convention** for migration files:

```
V{version}__{description}.sql
│  │          │             │
│  │          │             └── Must end with .sql
│  │          └──────────────── Description (use underscores for spaces)
│  └─────────────────────────── Version number (1, 2, 3 or 1.1, 1.2, etc.)
└────────────────────────────── V = Versioned migration (capital V!)
     ^^
     Two underscores! (not one, not three — exactly TWO)
```

### Examples of Valid Migration Names

```
V1__create_users_table.sql
V2__add_email_to_users.sql
V3__create_orders_table.sql
V4__add_index_on_orders.sql
V1.1__add_phone_to_users.sql
V20250115.1__create_products.sql
```

### Examples of INVALID Names (common mistakes)

```
❌ v1__create_users.sql        → Must be capital V
❌ V1_create_users.sql         → Must be TWO underscores
❌ V1___create_users.sql       → Must be exactly TWO underscores
❌ create_users.sql            → Missing version prefix
❌ V1__create users.sql        → No spaces in filename!
```

### Writing Migration Files

**V1__create_users_table.sql**

```sql
-- Create the users table
-- This is our first migration!

CREATE TABLE users (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,
    username    VARCHAR(50)  NOT NULL UNIQUE,
    email       VARCHAR(255) NOT NULL UNIQUE,
    password    VARCHAR(255) NOT NULL,
    created_at  TIMESTAMP    DEFAULT CURRENT_TIMESTAMP,
    updated_at  TIMESTAMP    DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- Add a comment explaining why this table exists
-- (comments are fine in migration files!)
```

**V2__create_orders_table.sql**

```sql
-- Create orders table with foreign key to users

CREATE TABLE orders (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id     BIGINT       NOT NULL,
    total       DECIMAL(10,2) NOT NULL DEFAULT 0.00,
    status      ENUM('pending', 'processing', 'shipped', 'delivered', 'cancelled') 
                DEFAULT 'pending',
    created_at  TIMESTAMP    DEFAULT CURRENT_TIMESTAMP,
    
    -- Foreign key constraint
    CONSTRAINT fk_orders_user 
        FOREIGN KEY (user_id) REFERENCES users(id) 
        ON DELETE CASCADE
);

-- Index for faster lookups by user
CREATE INDEX idx_orders_user_id ON orders(user_id);

-- Index for filtering by status
CREATE INDEX idx_orders_status ON orders(status);
```

**V3__add_phone_to_users.sql**

```sql
-- Add phone number column to users table
-- This is needed for the new 2FA feature

ALTER TABLE users 
    ADD COLUMN phone VARCHAR(20) DEFAULT NULL 
    AFTER email;
```

**V4__create_products_and_order_items.sql**

```sql
-- Create products table and order_items junction table

CREATE TABLE products (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,
    name        VARCHAR(200) NOT NULL,
    description TEXT,
    price       DECIMAL(10,2) NOT NULL,
    stock       INT          NOT NULL DEFAULT 0,
    created_at  TIMESTAMP    DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE order_items (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,
    order_id    BIGINT       NOT NULL,
    product_id  BIGINT       NOT NULL,
    quantity    INT          NOT NULL DEFAULT 1,
    price       DECIMAL(10,2) NOT NULL,
    
    CONSTRAINT fk_items_order 
        FOREIGN KEY (order_id) REFERENCES orders(id) ON DELETE CASCADE,
    CONSTRAINT fk_items_product 
        FOREIGN KEY (product_id) REFERENCES products(id) ON DELETE RESTRICT
);
```

### File Organization

```
project/
├── src/
│   └── main/
│       └── resources/
│           └── db/
│               └── migration/          ← Flyway looks here by default
│                   ├── V1__create_users_table.sql
```

### What Happens When You Add a New Migration?

```bash
# Developer adds V5__add_roles_table.sql
# Then runs migrate again:

flyway migrate

# Output:
# Current version of schema `mydb`: 4
# Migrating schema `mydb` to version "5 - add roles table"
# Successfully applied 1 migration (execution time 00:00.089s)
```

**Flyway only runs NEW (pending) migrations.** It skips V1-V4 because they're already in the history table.

## 9. Flyway — Versioned vs Repeatable Migrations

### Versioned Migrations (V prefix)

These run **exactly once**, in order:

```
V1__create_users.sql      → Runs once, recorded as version 1
V2__create_orders.sql     → Runs once, recorded as version 2
V3__add_columns.sql       → Runs once, recorded as version 3
```

**Use for:** Creating tables, altering columns, adding indexes — structural changes.

### Repeatable Migrations (R prefix)

These run **every time their content changes**:

```
R__create_views.sql        → Runs again whenever the file is modified
R__update_stored_procs.sql → Runs again whenever the file is modified
```

**Naming convention:**

```
R__{description}.sql
│     │
│     └── Description (no version number!)
└──────── R = Repeatable
```

**Use for:** Views, stored procedures, functions — things you want to CREATE OR REPLACE.

**Example — R__create_user_stats_view.sql:**

```sql
-- This is a repeatable migration
-- It runs every time the file content changes
-- Perfect for views that need to be recreated

CREATE OR REPLACE VIEW user_stats AS
SELECT 
    u.id,
    u.username,
    COUNT(o.id) AS total_orders,
    COALESCE(SUM(o.total), 0) AS total_spent,
    MAX(o.created_at) AS last_order_date
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id, u.username;
```

### Execution Order

```
Flyway always runs migrations in this order:
1. All VERSIONED migrations first (V1, V2, V3... in version order)
2. Then all REPEATABLE migrations (in alphabetical order by description)
```

## 10. Flyway — Undo Migrations (Rollbacks)

> **Note:** Undo migrations are a **paid feature** (Flyway Teams/Enterprise). The community edition doesn't support them. We'll cover both the paid approach and free workarounds.

### Paid Feature: Undo Migrations (U prefix)

For each versioned migration, you write a corresponding undo file:

```
sql/
├── V1__create_users.sql       → Forward migration
├── U1__create_users.sql       → Undo (rollback) migration
├── V2__create_orders.sql      → Forward migration
├── U2__create_orders.sql      → Undo (rollback) migration
```

**V1__create_users.sql:**

```sql
CREATE TABLE users (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(50) NOT NULL
);
```

**U1__create_users.sql (undo):**

```sql
DROP TABLE users;
```

Run the undo:

```bash
flyway undo   # Rolls back the last applied migration
```

### Free Alternative: Manual Rollback Migrations

In the community edition, just write a **new forward migration** that reverses the change:

```
V1__create_users.sql          → Creates the table
V2__add_email_to_users.sql    → Adds email column
V3__remove_email_rollback.sql → Removes email column (rollback as a new migration)
```

**V3__remove_email_rollback.sql:**

```sql
-- Rollback: Remove the email column added in V2
-- Reason: Feature was cancelled

ALTER TABLE users DROP COLUMN email;
```

This is **actually the recommended approach** by many teams because:
1. It's version-controlled
2. It's trackable in the history
3. It works with the free version
4. It's honest — the history shows what happened

## 11. Flyway — Configuration Deep Dive

### Configuration File: flyway.conf

```properties
# flyway.conf — Flyway configuration file

# ── Database Connection ──────────────────────────
flyway.url=jdbc:mysql://localhost:3306/mydb
flyway.user=flyway_user
flyway.password=SecurePass123!

# ── Migration File Locations ─────────────────────
# Where to find migration files (default: filesystem:sql)
flyway.locations=filesystem:sql,classpath:db/migration

# ── Schema ───────────────────────────────────────
# Which schema(s) to manage
flyway.schemas=mydb

# ── Table ────────────────────────────────────────
# Name of the history table (default: flyway_schema_history)
flyway.table=flyway_schema_history

# ── Baseline ─────────────────────────────────────
# For existing databases — mark current state as version 1
flyway.baselineOnMigrate=false
flyway.baselineVersion=1

# ── Validation ───────────────────────────────────
# What to do if a migration checksum doesn't match
flyway.validateOnMigrate=true

# ── Encoding ─────────────────────────────────────
flyway.encoding=UTF-8

# ── Clean ────────────────────────────────────────
# DANGER: 'clean' drops ALL objects in the schema!
# Disabled by default for safety
flyway.cleanDisabled=true

# ── Placeholders ─────────────────────────────────
# Use ${placeholder} in your SQL files
flyway.placeholders.schema_name=mydb
flyway.placeholders.admin_email=admin@example.com

# ── Output ───────────────────────────────────────
flyway.outputQueryResults=true
```

### Using Environment Variables

Flyway reads environment variables prefixed with `FLYWAY_`:

```bash
# Instead of putting credentials in a config file:
export FLYWAY_URL="jdbc:mysql://localhost:3306/mydb"
export FLYWAY_USER="flyway_user"
export FLYWAY_PASSWORD="SecurePass123!"

# Then just run:
flyway migrate
```

**This is the recommended approach for CI/CD** — never put passwords in config files!

### Placeholders

You can use variables in your migration files:

**flyway.conf:**

```properties
flyway.placeholders.table_prefix=app_
flyway.placeholders.default_role=viewer
```

**V1__create_users.sql:**

```sql
CREATE TABLE ${table_prefix}users (
    id BIGINT PRIMARY KEY,
    username VARCHAR(50),
    role VARCHAR(20) DEFAULT '${default_role}'
);
```

**Result:** Creates a table named `app_users` with default role `viewer`.

## 12. Flyway — CLI Commands Reference

| Command | What It Does |
|---------|-------------|
| `flyway migrate` | Apply all pending migrations |
| `flyway info` | Show the status of all migrations (applied + pending) |
| `flyway validate` | Check that applied migrations haven't been modified |
| `flyway baseline` | Mark an existing database as version N (for adopting Flyway on an existing DB) |
| `flyway repair` | Fix the history table (remove failed entries, realign checksums) |
| `flyway clean` | **⚠️ DANGER** — drops ALL objects in the schema (tables, views, everything!) |
| `flyway undo` | Roll back the last migration (Teams/Enterprise only) |

### Usage Examples

```bash
# Apply migrations
flyway -configFiles=flyway.conf migrate

# Check what's pending
flyway -configFiles=flyway.conf info

# Validate integrity of applied migrations
flyway -configFiles=flyway.conf validate

# Adopt Flyway on an existing database
# "My database is already at version 3, start tracking from here"
flyway -configFiles=flyway.conf -baselineVersion=3 baseline

# Fix corrupted history (e.g., failed migration left a mess)
flyway -configFiles=flyway.conf repair

# ⚠️ NUCLEAR OPTION — destroy everything (NEVER in production!)
flyway -configFiles=flyway.conf clean
```

## 13. Flyway in CI/CD Pipelines

### GitHub Actions Example

```yaml
# .github/workflows/db-migrate.yml
name: Database Migration

on:
  push:
    branches: [main]
    paths:
      - 'db/migration/**'  # Only run when migration files change

jobs:
  migrate:
    runs-on: ubuntu-latest
    
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Run Flyway migrations
        uses: flyway/flyway-action@master
        with:
          url: ${{ secrets.DB_URL }}
          user: ${{ secrets.DB_USER }}
          password: ${{ secrets.DB_PASSWORD }}
          locations: filesystem:db/migration
          command: migrate

      - name: Show migration info
        uses: flyway/flyway-action@master
        with:
          url: ${{ secrets.DB_URL }}
          user: ${{ secrets.DB_USER }}
          password: ${{ secrets.DB_PASSWORD }}
          command: info
```

### GitLab CI Example

```yaml
# .gitlab-ci.yml
stages:
  - migrate

db-migrate:
  stage: migrate
  image: flyway/flyway:latest
  variables:
    FLYWAY_URL: ${DB_URL}
    FLYWAY_USER: ${DB_USER}
    FLYWAY_PASSWORD: ${DB_PASSWORD}
    FLYWAY_LOCATIONS: filesystem:db/migration
  script:
    - flyway info
    - flyway migrate
    - flyway info
  only:
    changes:
      - db/migration/**
```

## 14. Liquibase — Overview and Philosophy

**Liquibase** is the other major database migration tool. It's more **feature-rich** and **flexible** than Flyway, but also more complex.

### Liquibase's Philosophy

```
"Database changes done right."

Liquibase believes in:
  1. Database-agnostic — write once, run on any database
  2. Multiple formats — YAML, XML, JSON, or raw SQL
  3. Rich rollback support — built-in rollback for all change types
  4. Fine-grained control — contexts, labels, preconditions
  5. Enterprise-ready — extensive tracking and auditing
```

### How Liquibase Differs from Flyway

```
Flyway:   "Here's a SQL file. Run it."          → Simple
Liquibase: "Here's a YAML/XML description of    → Flexible
            what you want to change. I'll 
            generate the right SQL for your 
            specific database."
```

### Key Features

| Feature | Description |
|---------|-------------|
| **Multiple formats** | Write migrations in YAML, XML, JSON, or SQL |
| **Database-agnostic** | Same changelog works on MySQL, PostgreSQL, Oracle, etc. |
| **Auto-rollback** | Many change types have automatic rollback generated |
| **Contexts** | Apply different migrations to different environments |
| **Labels** | Tag migrations for selective execution |
| **Preconditions** | Only run a migration if certain conditions are met |
| **Diff** | Compare two databases and generate a changelog |
| **Snapshots** | Take a snapshot of your database schema |

## 16. Liquibase — Changelogs and ChangeSets

### The Structure

Liquibase organizes migrations differently from Flyway:

```
Flyway:
  Individual files → V1__.sql, V2__.sql, V3__.sql

Liquibase:
  Master Changelog → includes → individual changelog files
```

### Master Changelog

The master changelog is the **entry point** — it includes all your individual changelogs:

```yaml
# db/changelog/db.changelog-master.yaml
databaseChangeLog:
  - include:
      file: db/changelog/changes/001-create-users.yaml
  - include:
      file: db/changelog/changes/002-create-orders.yaml
  - include:
      file: db/changelog/changes/003-add-phone-to-users.yaml
```

### ChangeSets

A **ChangeSet** is a single unit of change. Each ChangeSet has:
- An **id** (unique identifier)
- An **author** (who wrote it)
- One or more **changes** (what to do)

```yaml
# A single ChangeSet
- changeSet:
    id: 1                    # Unique ID
    author: ritesh           # Who wrote this
    changes:                 # What to change
      - createTable:
          tableName: users
          columns:
            - column:
                name: id
                type: BIGINT
                autoIncrement: true
                constraints:
                  primaryKey: true
```

### Rules for ChangeSets

```
1. Each ChangeSet is identified by: id + author + filename
   → This combination must be UNIQUE

2. Once a ChangeSet is applied, NEVER modify it
   → Liquibase tracks a checksum — if the file changes, it errors out

3. One ChangeSet = one logical change
   → Don't put 10 unrelated changes in one ChangeSet

4. ChangeSets are applied in the order they appear in the changelog
```

## 17. Liquibase — Writing Migrations in YAML

YAML is the **most popular format** for Liquibase. Let's write some real migrations.

### File Structure

```
project/
├── db/
│   └── changelog/
│       ├── db.changelog-master.yaml     ← Master changelog
│       └── changes/
│           ├── 001-create-users.yaml
│           ├── 002-create-orders.yaml
│           ├── 003-add-phone-to-users.yaml
│           └── 004-create-products.yaml
```

### Example 1: Create a Table

**001-create-users.yaml:**

```yaml
databaseChangeLog:
  - changeSet:
      id: 1
      author: ritesh
      comment: "Create the users table for user management"
      changes:
        - createTable:
            tableName: users
            columns:
              - column:
                  name: id
                  type: BIGINT
                  autoIncrement: true
                  constraints:
                    primaryKey: true
                    nullable: false
              - column:
                  name: username
                  type: VARCHAR(50)
                  constraints:
                    nullable: false
                    unique: true
              - column:
                  name: email
                  type: VARCHAR(255)
                  constraints:
                    nullable: false
                    unique: true
              - column:
                  name: password
                  type: VARCHAR(255)
                  constraints:
                    nullable: false
              - column:
                  name: created_at
                  type: TIMESTAMP
                  defaultValueComputed: CURRENT_TIMESTAMP
              - column:
                  name: updated_at
                  type: TIMESTAMP
                  defaultValueComputed: CURRENT_TIMESTAMP
```

**Why is this cool?** This YAML will generate the correct SQL for **any database**:
- MySQL: `BIGINT AUTO_INCREMENT`
- PostgreSQL: `BIGSERIAL`
- Oracle: Uses a sequence
- SQL Server: `BIGINT IDENTITY`

### Example 2: Create a Table with Foreign Key

**002-create-orders.yaml:**

```yaml
databaseChangeLog:
  - changeSet:
      id: 2
      author: ritesh
      comment: "Create orders table linked to users"
      changes:
        - createTable:
            tableName: orders
            columns:
              - column:
                  name: id
                  type: BIGINT
                  autoIncrement: true
                  constraints:
                    primaryKey: true
              - column:
                  name: user_id
                  type: BIGINT
                  constraints:
                    nullable: false
              - column:
                  name: total
                  type: DECIMAL(10,2)
                  defaultValueNumeric: 0
              - column:
                  name: status
                  type: VARCHAR(20)
                  defaultValue: "pending"
              - column:
                  name: created_at
                  type: TIMESTAMP
                  defaultValueComputed: CURRENT_TIMESTAMP

        # Add foreign key constraint
        - addForeignKeyConstraint:
            baseTableName: orders
            baseColumnNames: user_id
            referencedTableName: users
            referencedColumnNames: id
            constraintName: fk_orders_user
            onDelete: CASCADE

        # Add index for performance
        - createIndex:
            tableName: orders
            indexName: idx_orders_user_id
            columns:
              - column:
                  name: user_id

        - createIndex:
            tableName: orders
            indexName: idx_orders_status
            columns:
              - column:
                  name: status
```

### Example 3: Add a Column

**003-add-phone-to-users.yaml:**

```yaml
databaseChangeLog:
  - changeSet:
      id: 3
      author: ritesh
      comment: "Add phone number column for 2FA feature"
      changes:
        - addColumn:
            tableName: users
            columns:
              - column:
                  name: phone
                  type: VARCHAR(20)

      # Define how to roll this back

```

## 18. Liquibase — Writing Migrations in SQL

If you prefer plain SQL (many people do!), Liquibase supports it:

### SQL Changelog Format

**005-create-audit-log.sql:**

```sql
-- liquibase formatted sql

-- changeset ritesh:5
-- comment: Create audit log table for tracking changes
CREATE TABLE audit_log (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,
    table_name  VARCHAR(100) NOT NULL,
    record_id   BIGINT NOT NULL,
    action      ENUM('INSERT', 'UPDATE', 'DELETE') NOT NULL,
    old_data    JSON,
    new_data    JSON,
    changed_by  VARCHAR(50),
    changed_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_audit_table ON audit_log(table_name);
CREATE INDEX idx_audit_record ON audit_log(record_id);

-- rollback DROP TABLE audit_log;

-- changeset ritesh:6
-- comment: Add is_active column to users
ALTER TABLE users ADD COLUMN is_active BOOLEAN DEFAULT TRUE;

-- rollback ALTER TABLE users DROP COLUMN is_active;
```

**Key things to notice:**

```
-- liquibase formatted sql        ← REQUIRED at the very top of the file!
-- changeset author:id            ← Marks the start of a changeset
-- comment: description           ← Optional description
-- rollback SQL_STATEMENT         ← How to undo this changeset
```

### Including SQL Files in the Master Changelog

```yaml
# db.changelog-master.yaml
databaseChangeLog:
  - include:
      file: db/changelog/changes/001-create-users.yaml
  - include:
      file: db/changelog/changes/005-create-audit-log.sql    # SQL file!
```

## 21. Liquibase — Rollbacks

**Liquibase has excellent rollback support** — much better than Flyway's free tier.

### Automatic Rollbacks

Many Liquibase change types have **automatic rollback** — you don't need to write anything:

```yaml
# These changes auto-generate rollbacks:
- createTable      → Rollback: DROP TABLE
- addColumn        → Rollback: DROP COLUMN  
- createIndex      → Rollback: DROP INDEX
- addForeignKey    → Rollback: DROP FOREIGN KEY
- createSequence   → Rollback: DROP SEQUENCE
- createView       → Rollback: DROP VIEW
- renameTable      → Rollback: RENAME BACK
- renameColumn     → Rollback: RENAME BACK
```

### Manual Rollbacks

For changes that can't be auto-rolled-back (or when you need custom logic):

```yaml
- changeSet:
    id: 10
    author: ritesh
    changes:
      - sql:
          sql: "UPDATE users SET status = 'active' WHERE status IS NULL"
    rollback:
      - sql:
          sql: "UPDATE users SET status = NULL WHERE status = 'active'"
```

### Rolling Back

```bash
# Rollback the last N changesets
liquibase rollback-count 1     # Undo last 1 changeset
liquibase rollback-count 3     # Undo last 3 changesets

# Rollback to a specific tag
liquibase rollback --tag=v1.0

# Rollback to a specific date
liquibase rollback-to-date 2025-01-10

# Preview rollback SQL without executing
liquibase rollback-count-sql 1
```

### Using Tags

Tags let you mark specific points to roll back to:

```bash
# Apply migrations
liquibase update

# Tag the current state as "v1.0"
liquibase tag --tag=v1.0

# Apply more migrations
liquibase update

# Oh no, something went wrong! Roll back to v1.0
liquibase rollback --tag=v1.0
```

## 22. Liquibase — Contexts and Labels

### Contexts

Contexts let you apply **different migrations to different environments**:

```yaml
- changeSet:
    id: 100
    author: ritesh
    context: dev          # Only runs in 'dev' context
    changes:
      - insert:
          tableName: users
          columns:
            - column:
                name: username
                value: testuser
            - column:
                name: email
                value: test@example.com

- changeSet:
    id: 101
    author: ritesh
    context: prod         # Only runs in 'prod' context
    changes:
      - insert:
          tableName: config
          columns:
            - column:
                name: key
                value: cache_ttl
            - column:
                name: value
                value: "3600"
```

```bash
# Run with a specific context
liquibase --contexts=dev update       # Runs dev-only changesets
liquibase --contexts=prod update      # Runs prod-only changesets
liquibase --contexts=dev,test update  # Runs dev AND test changesets
```

### Labels

Labels are like tags for selective execution:

```yaml
- changeSet:
    id: 200
    author: ritesh
    labels: "feature-auth,sprint-12"
    changes:
      - createTable:
          tableName: sessions
          # ...
```

```bash
# Only run changesets with specific labels
liquibase --label-filter="feature-auth" update
liquibase --label-filter="sprint-12" update
```

### Contexts vs Labels

```
Contexts: "WHERE does this run?"  → dev, staging, prod
Labels:   "WHAT feature is this?" → feature-auth, sprint-12, hotfix-101
```

## 25. Flyway vs Liquibase — Detailed Comparison

| Feature | Flyway | Liquibase |
|---------|--------|-----------|
| **Migration format** | SQL (primary), Java | YAML, XML, JSON, SQL |
| **Database-agnostic** | No (write DB-specific SQL) | Yes (YAML/XML generates correct SQL) |
| **Learning curve** | Easy | Moderate |

| **Commercial company** | Redgate | Liquibase Inc. |
| **License** | Apache 2.0 (Community) | Apache 2.0 (Community) |

### When to Choose Flyway

```
✅ You prefer plain SQL migrations
✅ Your project uses only one database type
✅ You want the simplest possible setup
✅ Your team is comfortable with SQL
✅ You're working on a smaller project
```

### When to Choose Liquibase

```
✅ You need to support multiple database types
✅ You need built-in rollback support
✅ You want contexts/labels for environment-specific migrations
✅ You need diff/compare capabilities
✅ You're working on a large enterprise project
✅ You need preconditions (only run if table doesn't exist, etc.)
```

## 27. Database Migration Best Practices

### 1. Never Modify Applied Migrations

```
❌ WRONG: Editing V2__create_orders.sql after it's been applied
   → The checksum changes → tool throws an error
   → Other environments have the OLD version

✅ RIGHT: Create a NEW migration (V5__fix_orders_column.sql)
   → The fix is tracked, versioned, and applied everywhere
```

### 2. Make Migrations Backward-Compatible

```
❌ DANGEROUS: Rename a column that the app is still using
   ALTER TABLE users RENAME COLUMN name TO full_name;
   → App crashes because it still queries "name"!

✅ SAFE: Do it in stages:
   V1: ADD the new column
   V2: Copy data to new column (app can read from either)
   V3: Update app to use new column
   V4: Drop old column (after app is fully updated)
```

### 3. Keep Migrations Small

```
❌ One giant migration with 50 changes
   → Hard to debug if it fails
   → Hard to roll back partially

✅ Small, focused migrations
   → V1: Create users table
   → V2: Create orders table
   → V3: Add index on orders
   → Easy to find where things went wrong
```

### 4. Test Migrations Before Production

```bash
# Always preview the SQL first:
flyway migrate -dryRunOutput=dryrun.sql
liquibase update-sql > preview.sql

# Review the preview, then apply
```

### 5. Use Transactions

```yaml
# Liquibase — wrap changeset in a transaction (default)
- changeSet:
    id: 1
    author: ritesh
    runInTransaction: true    # This is the default
    changes:
      - createTable: ...
```

### 6. Version Control Everything

```
project/
├── .git/                           ← Code and migrations in same repo!
├── src/
│   └── ...                         ← Application code
├── db/
│   └── migration/                  ← Migration files (versioned with code)
│       ├── V1__initial_schema.sql
│       └── V2__add_orders.sql
└── flyway.conf
```

### 7. Use Environment Variables for Credentials

```bash
# NEVER put passwords in config files that go into Git!
# ❌ flyway.password=MySecretPassword

# ✅ Use environment variables:
export FLYWAY_PASSWORD="${DB_PASSWORD}"

# ✅ Or use CI/CD secrets:
flyway -password=${{ secrets.DB_PASSWORD }} migrate
```

## 28. Common Pitfalls and How to Avoid Them

### Pitfall 1: "Checksum mismatch" Error

```
ERROR: Migration checksum mismatch for migration version 3
Applied: 1234567890
Resolved: 9876543210
```

**Cause:** Someone modified a migration file AFTER it was applied.

**Fix:**

```bash
# Flyway:
flyway repair   # Recalculates checksums

# Liquibase:
liquibase clear-checksums
```

**Prevention:** Never edit applied migrations. Create a new one instead.

### Pitfall 2: Migration Works on Dev, Fails on Production

**Cause:** Database-specific SQL syntax.

```sql
-- ❌ This works on MySQL but NOT on PostgreSQL:
ALTER TABLE users MODIFY COLUMN email VARCHAR(500);

-- ✅ PostgreSQL syntax:
ALTER TABLE users ALTER COLUMN email TYPE VARCHAR(500);
```

**Fix:** Use Liquibase's database-agnostic YAML format, or test migrations against all target databases.

### Pitfall 3: Long-Running Migrations Lock the Database

```sql
-- ❌ This locks the entire table for minutes/hours on a large table:
ALTER TABLE huge_table ADD COLUMN new_col VARCHAR(100) DEFAULT 'value';
```

**Fix:** Use techniques like:

```sql
-- ✅ Add column without default first (instant in PostgreSQL 11+):
ALTER TABLE huge_table ADD COLUMN new_col VARCHAR(100);

-- ✅ Then backfill in batches:
UPDATE huge_table SET new_col = 'value' WHERE id BETWEEN 1 AND 10000;
UPDATE huge_table SET new_col = 'value' WHERE id BETWEEN 10001 AND 20000;
-- ... and so on
```

### Pitfall 4: Forgetting to Run Migrations

**Fix:** Integrate migrations into your deployment pipeline — make them run automatically:

```
Deploy Pipeline:
1. Run database migrations     ← FIRST
2. Deploy new application code ← SECOND
3. Health check
```

## 29. Real-World Migration Workflow

Here's how a professional team uses migrations day-to-day:

```
Developer's Workflow:
═══════════════════════════════════════════════════════

1. Pull latest code from Git
   $ git pull origin main

2. Run migrations on local database
   $ flyway migrate

3. Develop new feature that needs a DB change

4. Create a new migration file
   $ touch sql/V15__add_user_preferences_table.sql
   $ vim sql/V15__add_user_preferences_table.sql

5. Test migration locally
   $ flyway migrate
   $ flyway info    # Verify it applied

6. Write application code that uses the new table

7. Commit everything together
   $ git add sql/V15__add_user_preferences_table.sql
   $ git add src/preferences.py
   $ git commit -m "Add user preferences feature"
   $ git push

8. CI/CD pipeline runs:
   a. Run migrations on staging database
   b. Deploy app to staging
   c. Run tests
   d. If tests pass → run migrations on production
   e. Deploy app to production

═══════════════════════════════════════════════════════
```

## 30. Summary

```
What We Learned:
═══════════════════════════════════════════════════════════════

1. Database migrations = version control for your database
   → Track, apply, and rollback schema changes systematically

2. Flyway:
   → Simple, SQL-first, convention-driven
   → Files: V1__description.sql, V2__description.sql
   → Commands: migrate, info, validate, baseline, repair
   → Best for: Simple projects, SQL-loving teams

3. Liquibase:
   → Flexible, multi-format (YAML/XML/JSON/SQL)
   → Database-agnostic (generates correct SQL per DB)
   → Built-in rollback, contexts, labels, diff
   → Commands: update, status, rollback, diff, snapshot
   → Best for: Complex/enterprise projects, multi-DB environments

4. Both tools:
   → Track applied migrations in a history table
   → Run pending migrations automatically
   → Detect tampering via checksums
   → Integrate with CI/CD pipelines

5. Best practices:
   → Never modify applied migrations
   → Keep migrations small and focused
   → Make changes backward-compatible
   → Test before production
   → Automate in CI/CD
   → Use environment variables for credentials

═══════════════════════════════════════════════════════════════
```
