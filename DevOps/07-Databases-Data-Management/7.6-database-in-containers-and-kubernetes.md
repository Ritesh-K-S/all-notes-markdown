## 1. Why Running Databases in Containers is Controversial

This is one of the most debated topics in DevOps. Let's understand both sides:

### The "Don't Do It" Camp Says:

```
"Databases are not like web apps!"

Web apps:                          Databases:
──────────                         ──────────
Stateless ✅                       Stateful ❌
Kill and restart freely ✅         Data must survive restarts ❌
Scale horizontally easily ✅       Scaling is complex ❌
Disposable ✅                      Data is precious ❌
```

Their argument:

```
1. Container dies → Your data might be gone!
2. Container restarts → Different IP, different hostname!
3. Container moves to another node → Where's the disk?
4. Kubernetes schedules pods → It doesn't understand databases!
```

### The "It's Fine" Camp Says:

```
"With the right tools (StatefulSets, PVs, Operators), 
databases in containers work great!"

1. Persistent Volumes → Data survives container restarts ✅
2. StatefulSets → Stable network identities ✅
3. Operators → Automated database management ✅
4. Major companies (GitLab, Spotify, Bloomberg) do it in production ✅
```

### The Reality (2025)

```
Development/Testing:    → Containerized databases are THE standard ✅
Staging:                → Very common and works well ✅
Production (simple):    → Works great with proper setup ✅
Production (critical):  → Use Operators or managed services (RDS, Cloud SQL) ✅
                         → Don't DIY unless you really know what you're doing ⚠️
```

## 2. The Core Challenge — State

To understand why databases are special, you need to understand the difference between **stateless** and **stateful** applications:

### Stateless Applications (Easy to Containerize)

```
[Web App Container]
     │
     ├── No local data to worry about
     ├── Can be killed and restarted anywhere
     ├── All instances are identical
     └── Scaling = just add more containers

Example: A Node.js API that reads from a database
         → The API itself stores nothing
         → Kill it, restart it, who cares?
```

### Stateful Applications (Tricky to Containerize)

```
[Database Container]
     │
     ├── Data files on disk (MUST persist!)
     ├── Specific identity (server-id, hostname)
     ├── Unique configuration per instance
     ├── Ordering matters (master before slave)
     └── Data loss = disaster

Example: A MySQL database with 500GB of customer data
         → You CANNOT just kill and restart it
         → You CANNOT lose the data files
         → Each replica has a unique identity
```

### What Databases Need That Regular Apps Don't

| Need | Why |
|------|-----|
| **Persistent storage** | Data must survive container restarts/deletions |
| **Stable network identity** | Other containers need to find the database at a known address |
| **Ordered startup/shutdown** | Master must start before replicas |
| **Unique configuration** | Each replica might have different server-id, role, etc. |
| **Careful scaling** | Can't just "add 3 more" — need replication setup |
| **Graceful shutdown** | Must flush data to disk before stopping |

## 3. Running Databases in Docker — The Basics

Let's start simple — running a single database in Docker.

### PostgreSQL in Docker

```bash
# Run PostgreSQL
docker run -d \
    --name my-postgres \
    -e POSTGRES_USER=myuser \
    -e POSTGRES_PASSWORD=mypassword \
    -e POSTGRES_DB=myapp \
    -p 5432:5432 \
    postgres:16
```

**What each flag means:**

```
-d                          → Run in background (detached)
--name my-postgres          → Give the container a name
-e POSTGRES_USER=myuser     → Set the database username
-e POSTGRES_PASSWORD=...    → Set the password
-e POSTGRES_DB=myapp        → Create a database named 'myapp'
-p 5432:5432                → Map host port 5432 to container port 5432
postgres:16                 → Use PostgreSQL version 16 image
```

**Connect to it:**

```bash
# From the host
psql -h localhost -U myuser -d myapp

# From inside the container
docker exec -it my-postgres psql -U myuser -d myapp
```

### MySQL in Docker

```bash
docker run -d \
    --name my-mysql \
    -e MYSQL_ROOT_PASSWORD=rootpass \
    -e MYSQL_DATABASE=myapp \
    -e MYSQL_USER=myuser \
    -e MYSQL_PASSWORD=mypassword \
    -p 3306:3306 \
    mysql:8.0
```

### MongoDB in Docker

```bash
docker run -d \
    --name my-mongo \
    -e MONGO_INITDB_ROOT_USERNAME=admin \
    -e MONGO_INITDB_ROOT_PASSWORD=adminpass \
    -p 27017:27017 \
    mongo:7
```

### Redis in Docker

```bash
docker run -d \
    --name my-redis \
    -p 6379:6379 \
    redis:7 \
    redis-server --requirepass myredispassword
```

### The Problem — Data Loss!

```bash
# Create some data
docker exec -it my-postgres psql -U myuser -d myapp -c \
    "CREATE TABLE test (id SERIAL, msg TEXT); INSERT INTO test (msg) VALUES ('Hello!');"

# Delete the container
docker rm -f my-postgres

# Start a new container
docker run -d --name my-postgres -e POSTGRES_PASSWORD=mypassword postgres:16

# Check for data...
docker exec -it my-postgres psql -U postgres -c "SELECT * FROM test;"
# ERROR: relation "test" does not exist
# 😱 DATA IS GONE!
```

**Why?** Because container storage is **ephemeral** — when the container is deleted, everything inside it is deleted too. This is where **volumes** come in.

## 4. Docker Volumes — Persisting Database Data

Docker Volumes store data **outside the container**, so it survives container deletion.

### Named Volumes (Recommended)

```bash
# Create and run with a named volume
docker run -d \
    --name my-postgres \
    -e POSTGRES_USER=myuser \
    -e POSTGRES_PASSWORD=mypassword \
    -e POSTGRES_DB=myapp \
    -p 5432:5432 \
    -v postgres_data:/var/lib/postgresql/data \
    postgres:16
```

**What `-v postgres_data:/var/lib/postgresql/data` means:**

```
postgres_data                    → Name of the Docker volume (created automatically)
:                                → Separator
/var/lib/postgresql/data         → Path INSIDE the container where PostgreSQL stores data

Docker maps:
  Container: /var/lib/postgresql/data → Volume: postgres_data (on host disk)
```

Now let's test:

```bash
# Add data
docker exec -it my-postgres psql -U myuser -d myapp -c \
    "CREATE TABLE test (id SERIAL, msg TEXT); INSERT INTO test (msg) VALUES ('I survive!');"

# Delete the container
docker rm -f my-postgres

# Create a NEW container with the SAME volume
docker run -d \
    --name my-postgres \
    -e POSTGRES_USER=myuser \
    -e POSTGRES_PASSWORD=mypassword \
    -e POSTGRES_DB=myapp \
    -p 5432:5432 \
    -v postgres_data:/var/lib/postgresql/data \
    postgres:16

# Check data
docker exec -it my-postgres psql -U myuser -d myapp -c "SELECT * FROM test;"
# id | msg
# ---+-------------
#  1 | I survive!
# 🎉 DATA SURVIVED!
```

### Bind Mounts (Map to Host Directory)

```bash
# Store data in a specific host directory
docker run -d \
    --name my-postgres \
    -e POSTGRES_PASSWORD=mypassword \
    -p 5432:5432 \
    -v /home/user/pg_data:/var/lib/postgresql/data \
    postgres:16
```

### Volume Commands

```bash
# List all volumes
docker volume ls

# Inspect a volume (see where it's stored on the host)
docker volume inspect postgres_data

# Remove a volume (CAREFUL — deletes the data!)
docker volume rm postgres_data

# Remove ALL unused volumes
docker volume prune
```

### Volume Locations by Database

| Database | Data Directory Inside Container |
|----------|---------------------------------|
| **PostgreSQL** | `/var/lib/postgresql/data` |
| **MySQL** | `/var/lib/mysql` |
| **MongoDB** | `/data/db` |
| **Redis** | `/data` |
| **Cassandra** | `/var/lib/cassandra` |
| **Elasticsearch** | `/usr/share/elasticsearch/data` |

## 5. Docker Compose for Multi-Container Database Setups

Docker Compose makes it easy to run databases alongside your application:

### Simple App + Database Setup

```yaml
# docker-compose.yml
version: '3.8'

services:
  # PostgreSQL Database
  db:
    image: postgres:16
    container_name: myapp-db
    restart: unless-stopped
    environment:
      POSTGRES_USER: myuser
      POSTGRES_PASSWORD: mypassword
      POSTGRES_DB: myapp
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data
      - ./db/init:/docker-entrypoint-initdb.d  # Initialization scripts
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U myuser -d myapp"]
      interval: 10s
      timeout: 5s
      retries: 5

  # Your Application
  app:
    build: .
    ports:
      - "3000:3000"
    environment:
      DATABASE_URL: postgres://myuser:mypassword@db:5432/myapp
    depends_on:
      db:
        condition: service_healthy

volumes:
  postgres_data:
```

**Key things to notice:**

```
restart: unless-stopped
→ Auto-restart the database if it crashes

healthcheck:
→ Docker checks if PostgreSQL is ready before starting the app

depends_on: + condition: service_healthy
→ The app waits for the DB to be HEALTHY, not just started

./db/init:/docker-entrypoint-initdb.d
→ Any .sql files in db/init/ will run when the DB is first created
   (Great for initial schema, seed data, etc.)
```

### MySQL Replication with Docker Compose

```yaml
# docker-compose.yml — MySQL Master-Slave with Docker
version: '3.8'

services:
  mysql-master:
    image: mysql:8.0
    container_name: mysql-master
    environment:
      MYSQL_ROOT_PASSWORD: masterpass
      MYSQL_DATABASE: myapp
    ports:
      - "3306:3306"
    volumes:
      - master_data:/var/lib/mysql
      - ./master.cnf:/etc/mysql/conf.d/master.cnf
    command: --server-id=1 --log-bin=mysql-bin --binlog-format=ROW

  mysql-slave:
    image: mysql:8.0
    container_name: mysql-slave
    environment:
      MYSQL_ROOT_PASSWORD: slavepass
    ports:
      - "3307:3306"
    volumes:
      - slave_data:/var/lib/mysql
    command: --server-id=2 --relay-log=relay-bin --read-only=ON
    depends_on:
      - mysql-master

volumes:
  master_data:
  slave_data:
```

### Redis + PostgreSQL Stack

```yaml
version: '3.8'

services:
  postgres:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: postgres
    volumes:
      - pg_data:/var/lib/postgresql/data
    ports:
      - "5432:5432"

  redis:
    image: redis:7
    command: redis-server --appendonly yes --requirepass redispass
    volumes:
      - redis_data:/data
    ports:
      - "6379:6379"

  adminer:
    image: adminer
    ports:
      - "8080:8080"  # Web UI for managing databases

volumes:
  pg_data:
  redis_data:
```

## 6. When to Use Containerized Databases

### Perfect Use Cases

```
✅ Local development          → "docker-compose up" and you have a database
✅ CI/CD testing              → Spin up a clean DB for each test run
✅ Staging environments       → Mirror production setup
✅ Demos and POCs             → Quick, disposable environments
✅ Microservice development   → Each service gets its own DB
✅ Learning and training      → Try different DB versions easily
```

### Be Careful With

```
⚠️ Production databases with critical data
   → Use managed services (RDS, Cloud SQL) OR
   → Use Kubernetes Operators with proper setup

⚠️ High-performance databases
   → Container overhead might matter for extreme workloads
   → Network storage might add latency

⚠️ Very large databases (multi-TB)
   → Storage management becomes complex
   → Backup/restore takes significant time
```

### Don't Use For

```
❌ Quick throwaway containers without volumes
   → You WILL lose data

❌ Databases without proper backup strategy
   → Volumes can be accidentally deleted

❌ Without monitoring and alerting
   → Silent failures are the worst kind
```

## 7. Kubernetes Storage Concepts — PV, PVC, StorageClass

Before we run databases on Kubernetes, we need to understand how Kubernetes handles storage. This is **critical**.

### The Problem

```
Normal Pod (stateless):
──────────────────────
Pod starts → uses ephemeral storage → Pod dies → storage is gone ✅ (that's fine!)

Database Pod (stateful):
───────────────────────
Pod starts → uses ephemeral storage → Pod dies → storage is gone ❌ (DISASTER!)
```

Kubernetes solves this with three concepts:

### PersistentVolume (PV) — The Actual Disk

A **PV** represents a piece of actual storage — a disk in the cloud, an NFS share, or a local SSD.

```yaml
# A PersistentVolume — represents a real disk
apiVersion: v1
kind: PersistentVolume
metadata:
  name: postgres-pv
spec:
  capacity:
    storage: 50Gi                # 50 GB disk
  accessModes:
    - ReadWriteOnce              # Only one pod can write at a time
  persistentVolumeReclaimPolicy: Retain  # Keep data when PVC is deleted
  storageClassName: standard
  hostPath:                      # For local testing only!
    path: /mnt/data/postgres
```

**Think of PV as:** "Hey Kubernetes, here's a 50GB disk you can use."

### PersistentVolumeClaim (PVC) — The Request for Storage

A **PVC** is a **request** for storage by a pod. It's like saying "I need a 20GB disk."

```yaml
# A PersistentVolumeClaim — "I need storage!"
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: postgres-pvc
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 20Gi             # I need 20 GB
  storageClassName: standard
```

**Think of PVC as:** "Kubernetes, please give me a 20GB disk from available storage."

### StorageClass — The Type of Storage

A **StorageClass** defines what **type** of storage to provision (SSD, HDD, network storage, etc.):

```yaml
# StorageClass — defines the type of storage
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-ssd
provisioner: kubernetes.io/aws-ebs   # AWS EBS volumes
parameters:
  type: gp3                          # General Purpose SSD
  iopsPerGB: "10"
reclaimPolicy: Retain                # Keep disk after PVC is deleted
allowVolumeExpansion: true           # Allow resizing
volumeBindingMode: WaitForFirstConsumer
```

### How They Work Together

```
           Request                    Bind                     Mount
[Pod] ──→ [PVC] ──────────────→ [PV] ──────────────→ [Actual Disk]
          "I need 20GB"      "Here's a 50GB disk"    "AWS EBS gp3 SSD"
                                                     
           ┌─────────────────────────────────────┐
           │                                     │
           │  StorageClass "fast-ssd"             │
           │  → Auto-creates the PV and disk     │
           │  → Dynamic provisioning!            │
           │                                     │
           └─────────────────────────────────────┘
```

### Dynamic vs Static Provisioning

```
Static Provisioning:
────────────────────
Admin manually creates PV → PVC binds to it
(Admin pre-provisions the disk)

Dynamic Provisioning (recommended):
────────────────────────────────────
PVC requests storage → StorageClass auto-creates PV and disk
(No admin intervention needed!)
```

**In the cloud (AWS, GCP, Azure), dynamic provisioning is the norm.** You just create a PVC, and the cloud provider automatically creates a disk.

## 8. Deployments vs StatefulSets — The Critical Difference

This is the **most important concept** for running databases on Kubernetes.

### Deployments (for Stateless Apps)

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
spec:
  replicas: 3
  # ...
```

**What Deployments do:**

```
Pod names:        web-app-7d8f9b5c4f-abc12    (random!)
                  web-app-7d8f9b5c4f-xyz98    (random!)
                  web-app-7d8f9b5c4f-mno45    (random!)

Storage:          Shared or none (all pods are identical)

Scaling up:       Create new pod with random name (instant, any order)
Scaling down:     Kill any pod (doesn't matter which)

Restart:          New random name, might go to a different node
                  Doesn't matter — app is stateless
```

### StatefulSets (for Stateful Apps like Databases)

```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres
spec:
  replicas: 3
  # ...
```

**What StatefulSets do:**

```
Pod names:        postgres-0    (predictable! always 0, 1, 2...)
                  postgres-1
                  postgres-2

Storage:          EACH pod gets its OWN persistent volume
                  postgres-0 → PVC: data-postgres-0
                  postgres-1 → PVC: data-postgres-1
                  postgres-2 → PVC: data-postgres-2

Scaling up:       Created in ORDER (0, then 1, then 2)
                  Each waits for the previous to be ready

Scaling down:     Deleted in REVERSE order (2, then 1, then 0)

Restart:          SAME name, reattaches to the SAME volume
                  postgres-0 always gets data-postgres-0
```

### Side-by-Side Comparison

| Feature | Deployment | StatefulSet |
|---------|-----------|-------------|
| **Pod names** | Random (web-abc12) | Ordered (db-0, db-1, db-2) |
| **Storage** | Shared or ephemeral | Each pod gets unique PVC |
| **Network identity** | Random IP each restart | Stable DNS name per pod |
| **Start order** | All at once (parallel) | One by one (sequential) |
| **Delete order** | Any order | Reverse order |
| **Pod replacement** | New random name | Same name, same volume |
| **Use case** | Web apps, APIs | Databases, message queues |

### Why This Matters for Databases

```
Database replication requires:
──────────────────────────────
1. Master = postgres-0  (always pod 0!)
2. Slave 1 = postgres-1
3. Slave 2 = postgres-2

If postgres-0 restarts:
  → It gets the SAME name (postgres-0)
  → It reconnects to the SAME disk (data-postgres-0)
  → Other pods still find it at postgres-0.postgres.default.svc.cluster.local
  → Replication continues uninterrupted!

With a Deployment, this would be IMPOSSIBLE:
  → Random name, random volume, chaos! 💥
```

## 9. StatefulSets Deep Dive

Let's understand every part of a StatefulSet:

### Full StatefulSet YAML Explained

```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres
  namespace: database
spec:
  # ── Service Name (REQUIRED) ─────────────────────────
  # Must match a Headless Service for DNS to work
  serviceName: postgres-headless
  
  # ── Replicas ────────────────────────────────────────
  replicas: 3
  
  # ── Update Strategy ─────────────────────────────────
  # RollingUpdate: Update one pod at a time, in reverse order
  updateStrategy:
    type: RollingUpdate
    rollingUpdate:
      partition: 0    # Update all pods (set higher to do canary updates)
  
  # ── Pod Management Policy ───────────────────────────
  # OrderedReady: Start pods in order, wait for each to be Ready
  # Parallel: Start all pods at once (useful for some databases)
  podManagementPolicy: OrderedReady
  
  # ── Selector ────────────────────────────────────────
  selector:
    matchLabels:
      app: postgres
  
  # ── Pod Template ────────────────────────────────────
  template:
    metadata:
      labels:
        app: postgres
    spec:
      terminationGracePeriodSeconds: 30   # Give DB 30s to shutdown gracefully
      containers:
        - name: postgres
          image: postgres:16
          ports:
            - containerPort: 5432
              name: postgres
          env:
            - name: POSTGRES_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: postgres-secret
                  key: password
            - name: PGDATA
              value: /var/lib/postgresql/data/pgdata
          volumeMounts:
            - name: data
              mountPath: /var/lib/postgresql/data
          resources:
            requests:
              cpu: "500m"
              memory: "1Gi"
            limits:
              cpu: "2"
              memory: "4Gi"
          # ── Probes ──────────────────────────────────
          readinessProbe:
            exec:
              command:
                - pg_isready
                - -U
                - postgres
            initialDelaySeconds: 5
            periodSeconds: 10
          livenessProbe:
            exec:
              command:
                - pg_isready
                - -U
                - postgres
            initialDelaySeconds: 30
            periodSeconds: 10
  
  # ── Volume Claim Templates ─────────────────────────
  # This is the MAGIC — each pod gets its OWN PVC automatically!
  volumeClaimTemplates:
    - metadata:
        name: data
      spec:
        accessModes:
          - ReadWriteOnce
        storageClassName: standard
        resources:
          requests:
            storage: 50Gi
```

### What `volumeClaimTemplates` Does

This is the **key feature** of StatefulSets:

```
When you create a StatefulSet with 3 replicas and a volumeClaimTemplate named "data":

Kubernetes automatically creates:
  PVC: data-postgres-0 → 50Gi disk → mounted to postgres-0
  PVC: data-postgres-1 → 50Gi disk → mounted to postgres-1
  PVC: data-postgres-2 → 50Gi disk → mounted to postgres-2

Each pod gets its OWN dedicated disk!
If postgres-1 is deleted and recreated → it reconnects to data-postgres-1
```

### StatefulSet Lifecycle

```
Creating (scaling up):
──────────────────────
1. Create PVC data-postgres-0
2. Start postgres-0
3. Wait until postgres-0 is Ready ✅
4. Create PVC data-postgres-1
5. Start postgres-1
6. Wait until postgres-1 is Ready ✅
7. Create PVC data-postgres-2
8. Start postgres-2
9. Wait until postgres-2 is Ready ✅

Deleting (scaling down):
────────────────────────
1. Delete postgres-2 (last pod first)
2. Wait until terminated
3. Delete postgres-1
4. Wait until terminated
5. Delete postgres-0

⚠️ PVCs are NOT deleted when the StatefulSet is deleted!
   → This protects your data from accidental deletion
   → You must manually delete PVCs to free the storage
```

## 10. Running PostgreSQL on Kubernetes with StatefulSets

Let's deploy a complete PostgreSQL setup on Kubernetes.

### Step 1: Create a Namespace

```yaml
# namespace.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: database
```

### Step 2: Create a Secret for the Password

```bash
# Create a secret (command line)
kubectl create secret generic postgres-secret \
    --namespace=database \
    --from-literal=password='SuperSecure123!' \
    --from-literal=replication-password='ReplPass123!'
```

Or with YAML:

```yaml
# secret.yaml
apiVersion: v1
kind: Secret
metadata:
  name: postgres-secret
  namespace: database
type: Opaque
stringData:
  password: "SuperSecure123!"
  replication-password: "ReplPass123!"
```

### Step 3: Create a ConfigMap for PostgreSQL Configuration

```yaml
# configmap.yaml
apiVersion: v1

```

```
  → postgres-1.postgres-headless.database.svc.cluster.local → Pod 1's IP
  → postgres-2.postgres-headless.database.svc.cluster.local → Pod 2's IP
  → ✅ Perfect for databases — each pod has its own DNS name
```

## 12. Headless Services — Why StatefulSets Need Them

Let's understand headless services more deeply, because they're essential:

### Regular Service vs Headless Service

```
Regular Service (clusterIP: <auto>):
────────────────────────────────────
DNS lookup: postgres.database.svc.cluster.local
Returns:    10.96.45.123  (the Service's virtual IP)
Traffic is load-balanced across all pods.

Headless Service (clusterIP: None):
───────────────────────────────────
DNS lookup: postgres-headless.database.svc.cluster.local
Returns:    10.244.1.5, 10.244.2.8, 10.244.3.12  (ALL pod IPs!)

Individual pod DNS:
  postgres-0.postgres-headless.database.svc.cluster.local → 10.244.1.5
  postgres-1.postgres-headless.database.svc.cluster.local → 10.244.2.8
  postgres-2.postgres-headless.database.svc.cluster.local → 10.244.3.12
```

### DNS Pattern

```
{pod-name}.{headless-service-name}.{namespace}.svc.cluster.local

Examples:
  postgres-0.postgres-headless.database.svc.cluster.local
  mysql-0.mysql-headless.database.svc.cluster.local
  mongo-2.mongo-headless.database.svc.cluster.local
```

### Why Databases Need This

```
Scenario: MySQL replication on Kubernetes
─────────────────────────────────────────

mysql-0 = Master
mysql-1 = Slave
mysql-2 = Slave

Slave 1 needs to connect to the Master:
  CHANGE REPLICATION SOURCE TO
    SOURCE_HOST = 'mysql-0.mysql-headless.database.svc.cluster.local'

This ALWAYS resolves to mysql-0, no matter how many times it restarts!
Without headless service, you'd have a random IP that changes on restart.
```

## 14. Database Backup and Restore in Kubernetes

### Strategy 1: CronJob for Regular Backups

```yaml
# backup-cronjob.yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: postgres-backup
  namespace: database
spec:
  schedule: "0 2 * * *"           # Every day at 2:00 AM
  concurrencyPolicy: Forbid       # Don't run if previous backup is still running
  successfulJobsHistoryLimit: 7   # Keep last 7 successful backups
  failedJobsHistoryLimit: 3       # Keep last 3 failed backups
  jobTemplate:
    spec:
      template:
        spec:
          containers:
            - name: backup
              image: postgres:16
              command:
                - /bin/bash
                - -c
                - |
                  TIMESTAMP=$(date +%Y%m%d_%H%M%S)
                  BACKUP_FILE="/backups/postgres_${TIMESTAMP}.sql.gz"
                  
                  echo "Starting backup at $(date)"
                  
                  pg_dumpall \
                    -h postgres-0.postgres-headless.database.svc.cluster.local \
                    -U postgres | gzip > "${BACKUP_FILE}"
                  
                  if [ $? -eq 0 ]; then
                    echo "Backup successful: ${BACKUP_FILE}"
                    echo "Size: $(du -sh ${BACKUP_FILE} | cut -f1)"
                    
                    # Delete backups older than 30 days
                    find /backups -name "*.sql.gz" -mtime +30 -delete
                    echo "Old backups cleaned up"
                  else
                    echo "Backup FAILED!"
                    exit 1
                  fi
              env:
                - name: PGPASSWORD
                  valueFrom:
                    secretKeyRef:
                      name: postgres-secret
                      key: password
              volumeMounts:
                - name: backup-storage
                  mountPath: /backups
          restartPolicy: OnFailure
          volumes:
            - name: backup-storage
              persistentVolumeClaim:
                claimName: backup-pvc
```

### Strategy 2: Backup to Cloud Storage (S3)

```yaml
# backup-to-s3.yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: postgres-backup-s3
  namespace: database
spec:
  schedule: "0 */6 * * *"         # Every 6 hours
  jobTemplate:
    spec:
      template:
        spec:
          containers:
            - name: backup
              image: postgres:16
              command:
                - /bin/bash
                - -c
                - |
                  apt-get update && apt-get install -y awscli
                  TIMESTAMP=$(date +%Y%m%d_%H%M%S)
                  
                  pg_dumpall -h postgres-headless -U postgres | \
                    gzip | \
                    aws s3 cp - "s3://my-backups/postgres/${TIMESTAMP}.sql.gz"
                  
                  echo "Backup uploaded to S3: ${TIMESTAMP}.sql.gz"
              env:
                - name: PGPASSWORD
                  valueFrom:
                    secretKeyRef:
                      name: postgres-secret
                      key: password
                - name: AWS_ACCESS_KEY_ID
                  valueFrom:
                    secretKeyRef:
                      name: aws-credentials
                      key: access-key
                - name: AWS_SECRET_ACCESS_KEY
                  valueFrom:
                    secretKeyRef:
                      name: aws-credentials
                      key: secret-key
                - name: AWS_DEFAULT_REGION
                  value: us-east-1
          restartPolicy: OnFailure
```

### Restore from Backup


## 16. How Operators Work — The Operator Pattern

### The Architecture

```
                    ┌──────────────────────────┐
                    │    Kubernetes API Server  │
                    └────────────┬─────────────┘
                                 │
              ┌──────────────────┼──────────────────┐
              │                  │                   │
              ▼                  ▼                   ▼
     ┌────────────────┐  ┌──────────────┐  ┌────────────────┐
     │   Operator Pod │  │  Your App    │  │  Database Pods │
     │                │  │  Pods        │  │  (managed by   │
     │  Watches CRDs  │  │              │  │   Operator)    │
     │  Reconciles    │  │              │  │                │
     │  state         │  │              │  │  postgres-0    │
     └────────────────┘  └──────────────┘  │  postgres-1    │
              │                            │  postgres-2    │
              │                            └────────────────┘
              │
              ▼
     The Operator constantly checks:
     "Is the ACTUAL state == DESIRED state?"
     If not → take action to fix it!
```

### Custom Resource Definitions (CRDs)

Operators extend Kubernetes with **Custom Resource Definitions** — new types of Kubernetes objects.

```
Standard Kubernetes objects:
  Pod, Deployment, Service, ConfigMap, Secret...

With a PostgreSQL Operator, you also get:
  PostgresCluster, Backup, ScheduledBackup...

These are CUSTOM objects that the Operator understands!
```

### The Reconciliation Loop

```
Operator's infinite loop:
─────────────────────────
1. Watch: "Did anything change? (new CRD, pod died, config changed?)"
2. Compare: "What is the DESIRED state (from CRD)?"
            "What is the ACTUAL state (in the cluster)?"
3. Reconcile: If they differ → take action:
              - Create missing pods
              - Fix broken replication
              - Promote a replica after master failure
              - Scale up/down
              - Run backup
4. Go to step 1

This is the "Control Loop" or "Reconciliation Loop"
```

### Example: What Happens When a Master Dies

```
Without Operator:
─────────────────
1. Master pod dies (crash, node failure, etc.)
2. Alert fires → pages the on-call engineer
3. Engineer wakes up at 3 AM 😴
4. Engineer connects to Kubernetes
5. Engineer checks which replica is most up-to-date
6. Engineer promotes that replica to master
7. Engineer reconfigures other replicas to follow new master
8. Engineer verifies everything works
9. Time elapsed: 15-60 minutes ⏰

With Operator:
──────────────
1. Master pod dies
2. Operator detects it within seconds
3. Operator checks which replica is most up-to-date
4. Operator promotes that replica to master
5. Operator reconfigures other replicas
6. Operator verifies replication is working
7. Time elapsed: 10-30 seconds ⚡
8. On-call engineer gets a notification: "Failover completed successfully" 😎
```

With the Operator installed, creating a full PostgreSQL cluster is **this simple**:

```yaml
# postgres-cluster.yaml
apiVersion: postgresql.cnpg.io/v1
kind: Cluster
metadata:
  name: my-postgres
  namespace: database
spec:
  # Number of instances (1 primary + 2 replicas)
  instances: 3
  
  # PostgreSQL version
  imageName: ghcr.io/cloudnative-pg/postgresql:16

  storage:
    size: 50Gi
    storageClass: standard

      max_connections: "200"
      shared_buffers: "256MB"
      effective_cache_size: "768MB"
      log_statement: "ddl"
    pg_hba:
      - host all all 0.0.0.0/0 md5
  
  # Superuser secret
  superuserSecret:
    name: postgres-superuser
  
  # Bootstrap — create initial database and user
  bootstrap:
    initdb:
      database: myapp
      owner: myappuser
      secret:
        name: myapp-credentials
  
  # Backup configuration (to S3)
  backup:
    barmanObjectStore:
      destinationPath: s3://my-backups/postgres/
      s3Credentials:
        accessKeyId:
          name: aws-creds
          key: ACCESS_KEY_ID
        secretAccessKey:
          name: aws-creds
          key: ACCESS_SECRET_KEY
      wal:
        compression: gzip
  
  # Resource limits
  resources:
    requests:
      cpu: "500m"
      memory: "1Gi"
    limits:
      cpu: "2"
      memory: "4Gi"
  
  # Monitoring
  monitoring:
    enablePodMonitor: true
```

## 22. Operator vs Manual StatefulSet — Comparison

| Feature | Manual StatefulSet | Operator |
|---------|-------------------|----------|
| **Setup complexity** | High (write all YAMLs yourself) | Low (one CRD YAML) |
| **Replication setup** | Manual configuration | Automatic |
| **Failover** | Manual or separate tool | Automatic (seconds) |
| **Backup** | Manual CronJobs | Built-in + scheduled |
| **Scaling** | Manual + reconfigure | `kubectl scale` or edit CRD |
| **Upgrades** | Manual, risky | Rolling upgrade built-in |
| **Monitoring** | Set up yourself | Often built-in |
| **Recovery** | Manual intervention | Self-healing |
| **Expertise needed** | Deep DB + K8s knowledge | Basic K8s knowledge |
| **Customization** | Full control | Limited to CRD options |

### The Verdict

```
For Development / Testing:
  → Manual StatefulSet or Docker Compose is fine ✅

For Staging:
  → Operator is ideal ✅

For Production:
  → Operator is strongly recommended ✅
  → Or use a managed service (RDS, Cloud SQL) ✅
  → Manual StatefulSet only if you have dedicated DB+K8s experts ⚠️
```

## 23. Production Considerations

### Resource Planning

```yaml
# Production-grade resource settings
resources:
  requests:
    cpu: "2"           # Guaranteed CPU
    memory: "8Gi"      # Guaranteed memory
  limits:
    cpu: "4"           # Max CPU
    memory: "16Gi"     # Max memory (OOMKilled if exceeded!)

# IMPORTANT: For databases, set requests = limits
# This ensures the database always has dedicated resources
# and prevents Kubernetes from throttling it
```

### Pod Disruption Budgets (PDB)

Prevent Kubernetes from killing too many database pods at once:

```yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: postgres-pdb
  namespace: database
spec:
  maxUnavailable: 1        # At most 1 pod can be down at a time
  selector:
    matchLabels:
      app: postgres
```

### Pod Anti-Affinity

Ensure database pods run on **different nodes** (so a node failure doesn't kill all replicas):

```yaml
spec:
  template:
    spec:
      affinity:
        podAntiAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
            - labelSelector:
                matchExpressions:
                  - key: app
                    operator: In
                    values:
                      - postgres
              topologyKey: kubernetes.io/hostname
```

**What this means:**

```
"Never put two postgres pods on the same node"

Node 1: postgres-0 ✅
Node 2: postgres-1 ✅
Node 3: postgres-2 ✅

If Node 1 dies → only postgres-0 is affected
→ postgres-1 and postgres-2 are still running on other nodes
→ Failover to postgres-1, zero data loss!
```

### Network Policies

Restrict who can access your database:

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: postgres-network-policy
  namespace: database
spec:
  podSelector:
    matchLabels:
      app: postgres
  policyTypes:
    - Ingress
  ingress:
    - from:
        # Only allow traffic from the app namespace
        - namespaceSelector:
            matchLabels:
              name: myapp
        # And from the monitoring namespace
        - namespaceSelector:
            matchLabels:
              name: monitoring
      ports:
        - port: 5432
          protocol: TCP
```

### Storage Best Practices

```yaml
# Use SSD storage for databases!
storageClassName: fast-ssd    # Not the default "standard" (usually HDD)

# Allow volume expansion (grow disk without downtime)
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-ssd
provisioner: kubernetes.io/aws-ebs
parameters:
  type: gp3
  iopsPerGB: "50"
allowVolumeExpansion: true     # ← This allows resizing PVCs!
reclaimPolicy: Retain          # ← Don't delete data when PVC is deleted!
```

## 24. Anti-Patterns — What NOT to Do

### Anti-Pattern 1: Database Without Persistent Volume

```yaml
# ❌ NEVER do this in production!
spec:
  containers:
    - name: postgres
      image: postgres:16
      # No volumeMounts!
      # Data lives inside the container
      # Container dies → DATA IS GONE! 💀
```

### Anti-Pattern 2: Using Deployment Instead of StatefulSet

```yaml
# ❌ WRONG for databases
apiVersion: apps/v1
kind: Deployment          # ← Deployment can't guarantee stable identities!
metadata:
  name: postgres
```

```yaml
# ✅ CORRECT
apiVersion: apps/v1
kind: StatefulSet         # ← StatefulSet provides stable identities + storage
metadata:
  name: postgres
```

### Anti-Pattern 3: No Resource Limits

```yaml
# ❌ No limits — database could consume all node resources
spec:
  containers:
    - name: postgres
      # No resources defined!
```

```yaml
# ✅ Always set resource requests AND limits
spec:
  containers:
    - name: postgres
      resources:
        requests:
          cpu: "1"
          memory: "4Gi"
        limits:
          cpu: "2"
          memory: "8Gi"
```

### Anti-Pattern 4: Database Passwords in Plain YAML

```yaml
# ❌ Password visible in Git!
env:
  - name: POSTGRES_PASSWORD
    value: "MySecretPassword123"     # ← NEVER do this!
```

```yaml
# ✅ Use Kubernetes Secrets
env:
  - name: POSTGRES_PASSWORD
    valueFrom:
      secretKeyRef:
        name: postgres-secret
        key: password
```

### Anti-Pattern 5: No Backup Strategy

```
❌ "We have replication, we don't need backups"
   → Replication is NOT backup!
   → If you accidentally DROP TABLE, the DROP replicates to all replicas!
   → You need point-in-time recovery (PITR) from backups

✅ Always have:
   → Regular backups (daily full + continuous WAL/binlog)
   → Backup stored OUTSIDE the cluster (S3, GCS, etc.)
   → Regular backup restoration tests
   → Documented recovery procedures
```

### Anti-Pattern 6: All Database Pods on the Same Node

```
❌ Without anti-affinity:
   Node 1: postgres-0, postgres-1, postgres-2
   Node 1 dies → ALL databases are down!

✅ With anti-affinity:
   Node 1: postgres-0
   Node 2: postgres-1
   Node 3: postgres-2
   Node 1 dies → Only postgres-0 affected, automatic failover
```

## 26. Summary

```
What We Learned:
═══════════════════════════════════════════════════════════════

1. Databases in Containers:
   → Controversial but works well with proper setup
   → Docker volumes are essential for data persistence
   → Great for dev/test, careful in production

2. Docker:
   → Use named volumes: -v pgdata:/var/lib/postgresql/data
   → Docker Compose for multi-container setups
   → Always define healthchecks

3. Kubernetes Storage:
   → PersistentVolume (PV) = the actual disk
   → PersistentVolumeClaim (PVC) = request for storage
   → StorageClass = type of storage (SSD, HDD, etc.)
   → Dynamic provisioning = cloud auto-creates disks

4. StatefulSets (not Deployments!) for databases:
   → Stable pod names (db-0, db-1, db-2)
   → Each pod gets its own PVC
   → Ordered creation and deletion
   → Stable network identity via Headless Services

5. Headless Services:
   → clusterIP: None
   → Individual DNS per pod
   → Essential for database replication

6. Kubernetes Operators — the game-changer:
   → Encode DBA knowledge into software
   → Automatic failover, replication, backups, scaling
   → CloudNativePG (PostgreSQL), Percona (MySQL), 
     MongoDB Operator, Redis Operator

7. Production essentials:
   → Pod Anti-Affinity (pods on different nodes)
   → Pod Disruption Budgets (limit simultaneous downtime)
   → Network Policies (restrict access)
   → Resource requests and limits
   → SSD storage with Retain policy
   → Regular backups to external storage
   → Monitoring and alerting

8. Anti-patterns to avoid:
   → No persistent volume → data loss
   → Deployment instead of StatefulSet
   → No resource limits
   → Passwords in plain YAML
   → No backup strategy
   → All pods on the same node

═══════════════════════════════════════════════════════════════
```
