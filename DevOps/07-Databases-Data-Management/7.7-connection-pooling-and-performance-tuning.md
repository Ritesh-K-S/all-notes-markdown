# 7.7 — Connection Pooling & Database Performance Tuning

> **Level:** Advanced
> **Chapter:** 07 — Databases & Data Management

---

## The Problem — Why Do We Even Need Connection Pooling?

Let's start with a simple story.

Imagine you have a web application that gets **1,000 requests per second**. Every single request needs to talk to the database — fetch some data, update a row, insert something, etc.

Now, here's the thing: **opening a database connection is expensive**. It's not like flipping a light switch. Every time your app wants to talk to the database, this is what happens behind the scenes:

1. Your app sends a TCP connection request to the database server.
2. A TCP 3-way handshake happens (SYN → SYN-ACK → ACK).
3. If it's SSL/TLS, there's a TLS handshake too (even more round trips).
4. The database authenticates your credentials (username, password, permissions).
5. The database allocates memory for this session (buffers, caches, process/thread).
6. Finally — your query runs.
7. After the query is done, the connection is torn down (TCP FIN handshake).

All of this takes **20–100ms or more** just to set up the connection — before your actual query even runs!

Now multiply that by 1,000 requests per second. You're spending **most of your time just opening and closing connections** instead of actually doing useful work. That's insanely wasteful.

**This is the problem connection pooling solves.**

---

## What is Connection Pooling?

> **Connection pooling** means keeping a set of **pre-opened, reusable database connections** ready to go, so your application can just "borrow" one when it needs to run a query and "return" it when it's done — without the cost of opening/closing connections every time.

Think of it like a **car rental service**:

- **Without pooling:** Every time you need a car, you go to a factory, build a car from scratch, drive it, and then scrap it when you're done. Insanely wasteful.
- **With pooling:** There's a parking lot with 20 cars ready. You pick one up, drive it, bring it back. The next person takes the same car. No one builds or scraps anything.

### How It Works — Step by Step

```
┌─────────────────────────────────────────────────┐
│                  APPLICATION                     │
│                                                  │
│   Request 1 ──┐                                  │
│   Request 2 ──┤                                  │
│   Request 3 ──┤    ┌─────────────────────┐       │
│   Request 4 ──┼───►│  CONNECTION POOL     │       │
│   Request 5 ──┤    │                     │       │
│   ...        ─┤    │  Conn 1 (in use)    │       │
│               │    │  Conn 2 (in use)    │       │
│               │    │  Conn 3 (idle)      │──────►│ DATABASE
│               │    │  Conn 4 (idle)      │       │
│               │    │  Conn 5 (in use)    │       │
│               │    └─────────────────────┘       │
└─────────────────────────────────────────────────┘
```

1. When your app starts, the pool creates a **minimum number of connections** (say 5) and keeps them open.
2. When a request needs the database, it **borrows** a connection from the pool.
3. The request runs its query using that borrowed connection.
4. When the query is done, the connection goes **back to the pool** (not closed!).
5. The next request can now **reuse** that same connection.
6. If all connections are busy and a new request comes in, the pool can **create more** (up to a maximum limit).
7. If no connections are available and the max is reached, the request **waits in a queue** until one frees up.

---

## Key Connection Pool Parameters

Every connection pool has these important settings. Understanding them is critical for performance tuning.

| Parameter | What It Does | Typical Default |
|---|---|---|
| **Min Pool Size** (`minIdle`) | Minimum number of connections kept alive even when idle | 5–10 |
| **Max Pool Size** (`maxPoolSize`) | Maximum connections the pool will ever create | 10–50 |
| **Connection Timeout** | How long a request waits for a connection before throwing an error | 30 seconds |
| **Idle Timeout** | How long an idle connection sits unused before being closed | 10–30 minutes |
| **Max Lifetime** | Maximum time a connection can exist before being recycled (even if active) | 30–60 minutes |
| **Validation Query** | A simple query (like `SELECT 1`) to check if a connection is still alive | `SELECT 1` |
| **Leak Detection** | Time threshold to detect if a connection was borrowed and never returned | 60 seconds |

## Connection Pooling in Different Languages — With Code

### 1. Java — HikariCP (The Gold Standard)

HikariCP is the **fastest and most popular** Java connection pool. Spring Boot uses it by default.

**Maven Dependency:**
```xml
<dependency>
    <groupId>com.zaxxer</groupId>
    <artifactId>HikariCP</artifactId>
    <version>5.1.0</version>
</dependency>
```

**Configuration (application.yml in Spring Boot):**
```yaml
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/mydb
    username: myuser
    password: mypassword
    hikari:
      minimum-idle: 5              # Min connections to keep idle
      maximum-pool-size: 20        # Max connections in the pool
      idle-timeout: 300000         # 5 minutes — close idle connections after this
      max-lifetime: 1800000        # 30 minutes — recycle connections after this
      connection-timeout: 30000    # 30 seconds — wait time for a connection
      leak-detection-threshold: 60000  # 60 seconds — warn if connection held too long
      pool-name: MyAppPool
```

**What each setting does:**

- `minimum-idle: 5` → Even if no one is using the DB, keep 5 connections warmed up and ready.
- `maximum-pool-size: 20` → Never create more than 20 connections. If all 20 are busy, the 21st request waits.
- `idle-timeout: 300000` → If a connection has been sitting idle for 5 minutes, close it (unless we're at `minimum-idle`).
- `max-lifetime: 1800000` → Even if a connection is healthy, recycle it after 30 minutes. This prevents issues with firewalls, load balancers, or DB servers that silently kill old connections.
- `connection-timeout: 30000` → If a thread waits more than 30 seconds for a connection, throw a `SQLTransientConnectionException`.
- `leak-detection-threshold: 60000` → If code borrows a connection and doesn't return it within 60 seconds, log a warning with a stack trace so you can find the leak.

**Using it programmatically (without Spring Boot):**
```java
import com.zaxxer.hikari.HikariConfig;
import com.zaxxer.hikari.HikariDataSource;

import java.sql.Connection;
import java.sql.ResultSet;
import java.sql.Statement;

public class PoolExample {
    public static void main(String[] args) throws Exception {
        // Configure the pool
        HikariConfig config = new HikariConfig();
        config.setJdbcUrl("jdbc:postgresql://localhost:5432/mydb");
        config.setUsername("myuser");
        config.setPassword("mypassword");
        config.setMaximumPoolSize(20);
        config.setMinimumIdle(5);
        config.setIdleTimeout(300000);
        config.setMaxLifetime(1800000);
        config.setConnectionTimeout(30000);

        // Create the pool (connections are pre-opened here)
        HikariDataSource dataSource = new HikariDataSource(config);

        // Borrow a connection, use it, and return it
        try (Connection conn = dataSource.getConnection();       // Borrow from pool
             Statement stmt = conn.createStatement();
             ResultSet rs = stmt.executeQuery("SELECT * FROM users LIMIT 10")) {

            while (rs.next()) {
                System.out.println(rs.getString("name"));
            }
        }
        // Connection is automatically returned to pool when try-with-resources closes it
        // It's NOT actually closed — just returned to the pool!

        // Shutdown the pool when your app stops
        dataSource.close();
    }
}
```

**Key thing to notice:** When you call `conn.close()` (or the try-with-resources block ends), the connection is **NOT actually closed**. The pool intercepts the `close()` call and **returns the connection to the pool** instead. Clever, right?

### 2. Python — SQLAlchemy with Connection Pooling

SQLAlchemy has **built-in connection pooling** — you don't need to install anything extra.

```python
from sqlalchemy import create_engine, text

# Create an engine with pooling configured
engine = create_engine(
    "postgresql://myuser:mypassword@localhost:5432/mydb",
    pool_size=10,          # Max connections to keep in the pool
    max_overflow=20,       # Extra connections allowed above pool_size (temporary)
    pool_timeout=30,       # Seconds to wait for a connection before error
    pool_recycle=1800,     # Recycle connections after 30 minutes
    pool_pre_ping=True,    # Check if connection is alive before using it
    echo=False             # Set True to see SQL logs
)

# Use a connection from the pool
with engine.connect() as conn:
    result = conn.execute(text("SELECT * FROM users LIMIT 10"))
    for row in result:
        print(row)
# Connection is returned to pool when `with` block ends

# Check pool status
print(f"Pool size: {engine.pool.size()}")
print(f"Checked out connections: {engine.pool.checkedout()}")
print(f"Overflow: {engine.pool.overflow()}")

# When your app shuts down
engine.dispose()
```

**What each parameter means:**

- `pool_size=10` → Keep up to 10 connections in the pool at all times.
- `max_overflow=20` → If all 10 are busy, allow up to 20 *extra* temporary connections (so total max = 30). These overflow connections are closed as soon as they're returned.
- `pool_timeout=30` → If the pool is full and no connections are available, wait 30 seconds. If still nothing, raise a `TimeoutError`.
- `pool_recycle=1800` → Recycle connections every 30 minutes. This is essential for MySQL, which has a default `wait_timeout` of 8 hours — but cloud load balancers and firewalls often kill idle connections sooner.
- `pool_pre_ping=True` → Before handing a connection to your code, the pool sends a lightweight "ping" (like `SELECT 1`) to make sure the connection is still alive. If it's dead, the pool silently discards it and gives you a fresh one.

### 3. Node.js — pg-pool (PostgreSQL)

```javascript
const { Pool } = require('pg');

// Create a connection pool
const pool = new Pool({
    host: 'localhost',
    port: 5432,
    database: 'mydb',
    user: 'myuser',
    password: 'mypassword',
    max: 20,                    // Maximum connections in the pool
    idleTimeoutMillis: 300000,  // Close idle connections after 5 minutes
    connectionTimeoutMillis: 30000, // Wait 30s for connection before error
});

// Use a connection from the pool
async function getUsers() {
    const client = await pool.connect();   // Borrow a connection
    try {
        const result = await client.query('SELECT * FROM users LIMIT 10');
        console.log(result.rows);
    } finally {
        client.release();                  // Return connection to pool
    }
}

// Or use the simpler shortcut (pool.query automatically borrows and returns)
async function getUsersSimple() {
    const result = await pool.query('SELECT * FROM users LIMIT 10');
    console.log(result.rows);
}

// Pool events
pool.on('connect', (client) => {
    console.log('New connection created in pool');
});

pool.on('error', (err, client) => {
    console.error('Unexpected error on idle client', err);
});

// Check pool status
console.log(`Total: ${pool.totalCount}`);
console.log(`Idle: ${pool.idleCount}`);
console.log(`Waiting: ${pool.waitingCount}`);

// Shutdown
pool.end();
```

### 4. Go — database/sql (Built-in Pooling)

Go's standard `database/sql` package has **built-in connection pooling**. You don't need any external library.

```go
package main

import (
    "database/sql"
    "fmt"
    "time"

    _ "github.com/lib/pq" // PostgreSQL driver
)

func main() {
    // Open creates a pool (it doesn't open a connection yet)
    db, err := sql.Open("postgres", "postgres://myuser:mypassword@localhost:5432/mydb?sslmode=disable")
    if err != nil {
        panic(err)
    }
    defer db.Close()

    // Configure the pool
    db.SetMaxOpenConns(25)              // Max open connections to DB
    db.SetMaxIdleConns(10)              // Max idle connections in the pool
    db.SetConnMaxLifetime(30 * time.Minute) // Recycle connections after 30 min
    db.SetConnMaxIdleTime(5 * time.Minute)  // Close idle connections after 5 min

    // Verify the connection works
    if err := db.Ping(); err != nil {
        panic(err)
    }
    fmt.Println("Connected to database!")

    // Use the pool — it automatically borrows and returns connections
    rows, err := db.Query("SELECT name FROM users LIMIT 10")
    if err != nil {
        panic(err)
    }
    defer rows.Close()

    for rows.Next() {
        var name string
        rows.Scan(&name)
        fmt.Println(name)
    }

    // Check pool stats
    stats := db.Stats()
    fmt.Printf("Open connections: %d\n", stats.OpenConnections)
    fmt.Printf("In use: %d\n", stats.InUse)
    fmt.Printf("Idle: %d\n", stats.Idle)
    fmt.Printf("Wait count: %d\n", stats.WaitCount)
    fmt.Printf("Wait duration: %v\n", stats.WaitDuration)
}
```

## External Connection Poolers — PgBouncer & ProxySQL

Sometimes, connection pooling inside your application isn't enough. When you have **hundreds of app instances** (like in Kubernetes), each running its own pool, the database gets overwhelmed by sheer connection count.

Example: 100 pods × 20 connections/pool = **2,000 connections** to the DB. Most databases can't handle that.

This is where **external connection poolers** come in. They sit **between your app and the database** and manage connections for ALL your app instances.

### PgBouncer (for PostgreSQL)

PgBouncer is a **lightweight connection pooler** that sits in front of PostgreSQL.

```
┌──────────┐   ┌──────────┐   ┌──────────┐
│  App 1   │   │  App 2   │   │  App 3   │
│(20 conns)│   │(20 conns)│   │(20 conns)│
└────┬─────┘   └────┬─────┘   └────┬─────┘
     │              │              │
     └──────────────┼──────────────┘
                    │
                    ▼
           ┌────────────────┐
           │   PgBouncer    │  ← Accepts 60 connections from apps
           │   (pooler)     │  ← But only keeps 10 connections to PostgreSQL!
           └───────┬────────┘
                   │
                   ▼
           ┌────────────────┐
           │  PostgreSQL    │  ← Only sees 10 connections. Happy and healthy.
           └────────────────┘
```

**PgBouncer Configuration (`pgbouncer.ini`):**

```ini
[databases]
; Map database names to actual PostgreSQL servers
mydb = host=127.0.0.1 port=5432 dbname=mydb

[pgbouncer]
; Listen on this address and port
listen_addr = 0.0.0.0
listen_port = 6432

; Authentication
auth_type = md5
auth_file = /etc/pgbouncer/userlist.txt

; Pool Mode (VERY IMPORTANT — read below)
pool_mode = transaction

; Pool sizing
default_pool_size = 20       ; Connections per user/database pair
min_pool_size = 5            ; Minimum connections to keep
max_client_conn = 1000       ; Max client connections PgBouncer will accept
max_db_connections = 50      ; Max connections PgBouncer opens to the actual DB

; Timeouts
server_idle_timeout = 600    ; Close server connection after 10 min idle
client_idle_timeout = 0      ; 0 = never disconnect idle clients
query_timeout = 0            ; 0 = no query timeout

; Logging
log_connections = 1
log_disconnections = 1
log_pooler_errors = 1
```

**Pool Modes Explained:**

| Mode | How It Works | Best For |
|---|---|---|
| **Session** | Client gets a dedicated server connection for the entire session. Connection returned only when client disconnects. | Legacy apps that use session-level features (LISTEN/NOTIFY, prepared statements) |
| **Transaction** | Client gets a server connection only for the duration of a transaction. Between transactions, the connection is returned to the pool. **Most common mode.** | Modern web apps (stateless requests) |
| **Statement** | Client gets a connection for a single statement. Multi-statement transactions are NOT allowed. | Simple read-heavy workloads |

**Transaction mode** is the sweet spot for most applications. It gives you the **maximum connection multiplexing** while still supporting transactions.

**Running PgBouncer in Docker:**

```bash
docker run -d \
  --name pgbouncer \
  -p 6432:6432 \
  -e DATABASE_URL="postgres://myuser:mypassword@host.docker.internal:5432/mydb" \
  -e POOL_MODE=transaction \
  -e DEFAULT_POOL_SIZE=20 \
  -e MAX_CLIENT_CONN=500 \
  edoburu/pgbouncer:latest
```

Now your app connects to `localhost:6432` instead of `localhost:5432`. PgBouncer handles the rest.

### ProxySQL (for MySQL)

ProxySQL does for MySQL what PgBouncer does for PostgreSQL — but with even more features like query routing, read/write splitting, and query caching.

```bash
# Run ProxySQL in Docker
docker run -d \
  --name proxysql \
  -p 6033:6033 \      # App connects here
  -p 6032:6032 \      # Admin interface
  proxysql/proxysql:latest
```

**Basic ProxySQL configuration (via admin interface):**

```sql
-- Connect to ProxySQL admin
mysql -u admin -padmin -h 127.0.0.1 -P 6032

-- Add your MySQL backend servers
INSERT INTO mysql_servers (hostgroup_id, hostname, port, max_connections)
VALUES (1, '10.0.0.1', 3306, 100);   -- Writer (primary)

INSERT INTO mysql_servers (hostgroup_id, hostname, port, max_connections)
VALUES (2, '10.0.0.2', 3306, 100);   -- Reader (replica)

-- Add a user
INSERT INTO mysql_users (username, password, default_hostgroup)
VALUES ('myuser', 'mypassword', 1);

-- Set connection pool settings
UPDATE mysql_servers SET max_connections=100, max_replication_lag=5;

-- Route SELECT queries to readers, everything else to writers
INSERT INTO mysql_query_rules (rule_id, match_pattern, destination_hostgroup)
VALUES (1, '^SELECT.*', 2);   -- SELECTs go to reader hostgroup

INSERT INTO mysql_query_rules (rule_id, match_pattern, destination_hostgroup)
VALUES (2, '.*', 1);          -- Everything else goes to writer hostgroup

-- Apply changes
LOAD MYSQL SERVERS TO RUNTIME;
LOAD MYSQL USERS TO RUNTIME;
LOAD MYSQL QUERY RULES TO RUNTIME;

-- Save to disk (persist across restarts)
SAVE MYSQL SERVERS TO DISK;
SAVE MYSQL USERS TO DISK;
SAVE MYSQL QUERY RULES TO DISK;
```

## Database Performance Tuning — The Complete Guide

Now let's talk about making your database **fast**. Connection pooling is just one piece. Let's cover everything.

---

### 1. Query Optimization — The Biggest Win

> **90% of database performance problems are caused by bad queries, not bad hardware.**

#### Use EXPLAIN / EXPLAIN ANALYZE

Before optimizing anything, you need to **understand what the database is actually doing** when it runs your query.

```sql
-- PostgreSQL
EXPLAIN ANALYZE SELECT * FROM orders WHERE customer_id = 42;
```

**Sample output:**
```
Seq Scan on orders  (cost=0.00..1250.00 rows=50 width=120) (actual time=0.015..12.450 rows=47 loops=1)
  Filter: (customer_id = 42)
  Rows Removed by Filter: 49953
Planning Time: 0.085 ms
Execution Time: 12.490 ms
```

**Reading this output:**

- **Seq Scan** = The database is scanning the **entire table** row by row. This is BAD for large tables. It means there's no index on `customer_id`.
- **cost=0.00..1250.00** = Estimated cost (arbitrary units). Higher = slower.
- **rows=50** = Database estimated it would find 50 rows.
- **actual rows=47** = It actually found 47 rows.
- **Rows Removed by Filter: 49953** = It had to look at ~50,000 rows to find just 47. Incredibly wasteful.
- **Execution Time: 12.490 ms** = How long it took.

**Now let's add an index and see the difference:**

```sql
CREATE INDEX idx_orders_customer_id ON orders(customer_id);

EXPLAIN ANALYZE SELECT * FROM orders WHERE customer_id = 42;
```

**New output:**
```
Index Scan using idx_orders_customer_id on orders  (cost=0.29..12.50 rows=50 width=120) (actual time=0.025..0.180 rows=47 loops=1)
  Index Cond: (customer_id = 42)
Planning Time: 0.095 ms
Execution Time: 0.210 ms
```

**Look at that:**

- **Index Scan** instead of Seq Scan — it goes directly to the matching rows.
- **cost dropped** from 1250 to 12.50.
- **Execution Time** went from **12.490 ms → 0.210 ms** — a **59x improvement** just by adding an index!

#### Common Query Anti-Patterns (And How to Fix Them)

**❌ Anti-pattern 1: SELECT ***

```sql
-- BAD — fetches all columns, even if you only need 2
SELECT * FROM users WHERE id = 42;

-- GOOD — only fetch what you need
SELECT name, email FROM users WHERE id = 42;
```

Why it matters: If your table has 50 columns including large TEXT or BLOB fields, `SELECT *` pulls everything. More data = more I/O = slower.

---

**❌ Anti-pattern 2: Missing WHERE clause on large tables**

```sql
-- BAD — scans 10 million rows
SELECT COUNT(*) FROM logs;

-- BETTER — use approximate count if exact number isn't critical (PostgreSQL)
SELECT reltuples AS approximate_count FROM pg_class WHERE relname = 'logs';
```

---

**❌ Anti-pattern 3: Using functions on indexed columns**

```sql
-- BAD — the index on created_at is USELESS because you wrapped it in a function
SELECT * FROM orders WHERE YEAR(created_at) = 2024;

-- GOOD — rewrite to use the raw column so the index can be used
SELECT * FROM orders WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01';
```

When you apply a function to a column (`YEAR()`, `LOWER()`, `UPPER()`, etc.), the database can't use the index on that column. It has to evaluate the function on **every single row**.

---

**❌ Anti-pattern 4: N+1 Query Problem**

```python
# BAD — 1 query to get users + N queries to get orders (if 100 users, that's 101 queries!)
users = db.query("SELECT * FROM users LIMIT 100")
for user in users:
    orders = db.query(f"SELECT * FROM orders WHERE user_id = {user.id}")

# GOOD — use a JOIN (just 1 query!)
results = db.query("""
    SELECT u.name, o.order_id, o.amount
    FROM users u
    JOIN orders o ON u.id = o.user_id
    LIMIT 100
""")
```

---

**❌ Anti-pattern 5: Not using LIMIT**

```sql
-- BAD — returns all million rows to your app, which discards most of them
SELECT * FROM logs WHERE level = 'ERROR';

-- GOOD — paginate
SELECT * FROM logs WHERE level = 'ERROR' ORDER BY created_at DESC LIMIT 50 OFFSET 0;
```

### 2. Indexing Strategies

Indexes are **THE most important** performance tool. Here's a deep dive.

#### Types of Indexes

| Index Type | What It Is | Best For | Example |
|---|---|---|---|
| **B-tree** (default) | Balanced tree structure. The default in PostgreSQL, MySQL, etc. | Equality (`=`) and range queries (`<`, `>`, `BETWEEN`) | `CREATE INDEX idx ON users(age);` |
| **Hash** | Hash table. O(1) lookups. | Only equality (`=`) queries. Can't do ranges. | `CREATE INDEX idx ON users USING hash(email);` |
| **GIN** (Generalized Inverted Index) | Maps values to rows. Good for multi-valued data. | Full-text search, JSONB, arrays | `CREATE INDEX idx ON articles USING gin(to_tsvector('english', body));` |
| **GiST** (Generalized Search Tree) | Tree structure for complex data types. | Geospatial queries (PostGIS), range types, nearest-neighbor | `CREATE INDEX idx ON locations USING gist(coordinates);` |
| **BRIN** (Block Range Index) | Stores min/max values per block of rows. Very small index. | Time-series data, naturally ordered columns | `CREATE INDEX idx ON logs USING brin(created_at);` |

#### Composite (Multi-Column) Indexes

```sql
-- If you frequently query by both city AND age:
CREATE INDEX idx_users_city_age ON users(city, age);

-- This index works for:
SELECT * FROM users WHERE city = 'Mumbai';                  -- ✅ Uses index (leftmost column)
SELECT * FROM users WHERE city = 'Mumbai' AND age > 25;     -- ✅ Uses index (both columns)
SELECT * FROM users WHERE age > 25;                          -- ❌ Does NOT use index (age isn't the leftmost)
```

**Rule of thumb (leftmost prefix rule):** A composite index on `(A, B, C)` can be used for queries filtering on:
- `A` alone ✅
- `A` and `B` ✅
- `A`, `B`, and `C` ✅
- `B` alone ❌
- `C` alone ❌
- `B` and `C` ❌

**Column order matters.** Put the most selective column (the one that narrows down results the most) first.

#### Covering Indexes (Index-Only Scans)

A **covering index** includes all the columns your query needs, so the database can answer the query **entirely from the index** without touching the table at all.

```sql
-- Your query:
SELECT name, email FROM users WHERE city = 'Mumbai';

-- A covering index for this query:
CREATE INDEX idx_covering ON users(city) INCLUDE (name, email);

-- Now PostgreSQL can do an "Index Only Scan" — it never reads the table!
```

#### Partial Indexes

Only index the rows you actually query. Saves space and makes the index faster.

```sql
-- If you only ever query active users:
CREATE INDEX idx_active_users ON users(email) WHERE is_active = true;

-- This index is MUCH smaller than indexing all users
-- And queries with WHERE is_active = true will be faster
```

#### When NOT to Index

- **Small tables** (< 1,000 rows) — A sequential scan is actually faster than an index scan for tiny tables.
- **Columns with very low cardinality** — e.g., a `gender` column with only 2 values. The index won't help much.
- **Tables with heavy writes** — Every INSERT, UPDATE, and DELETE has to update ALL indexes on that table. Too many indexes slow down writes.

### 3. PostgreSQL Performance Tuning (postgresql.conf)

These are the most impactful settings to tune in PostgreSQL.

```ini
# ==========================================
# MEMORY SETTINGS
# ==========================================

# Shared memory used for caching data. Set to 25% of total RAM.
# Example: Server has 16 GB RAM → set to 4 GB
shared_buffers = 4GB

# Memory for each query's sort, hash, and join operations.
# Set to 256MB–1GB for OLAP, 64MB–256MB for OLTP.
work_mem = 256MB

# Memory for maintenance operations (VACUUM, CREATE INDEX, ALTER TABLE).
# Set to 1–2 GB.
maintenance_work_mem = 1GB

# Total memory available for caching. Set to 75% of total RAM.
# PostgreSQL uses this to estimate how much data is cached by the OS.
effective_cache_size = 12GB

# ==========================================
# WAL (Write-Ahead Log) SETTINGS
# ==========================================

# Size of WAL buffers. Set to 64MB for write-heavy workloads.
wal_buffers = 64MB

# Minimum WAL size before checkpoint is triggered.
min_wal_size = 1GB

# Maximum WAL size.
max_wal_size = 4GB

# How aggressively to flush dirty pages during checkpoint.
# 0.9 = spread the I/O over 90% of the checkpoint interval (less I/O spikes).
checkpoint_completion_target = 0.9

# ==========================================
# QUERY PLANNER SETTINGS
# ==========================================

# Cost of a random page read (from disk). Lower this if you have SSDs.
# Default is 4.0 (for HDDs). Set to 1.1–1.5 for SSDs.
random_page_cost = 1.1

# Cost of a sequential page read. Usually keep at 1.0.
seq_page_cost = 1.0

# How many parallel workers a single query can use.
max_parallel_workers_per_gather = 4

# Total parallel workers available for the entire server.
max_parallel_workers = 8

# ==========================================
# CONNECTION SETTINGS
# ==========================================

# Maximum number of connections. Keep this LOW if using PgBouncer.
# Formula: max_connections = (CPU cores × 2) + effective_spindle_count
# For most systems: 100–200

# ==========================================

# ==========================================

log_lock_waits = on
```

### 5. VACUUM and ANALYZE (PostgreSQL Specific)

PostgreSQL uses **MVCC** (Multi-Version Concurrency Control). When you UPDATE or DELETE a row, the old version of the row **isn't actually deleted** — it's just marked as "dead." These dead rows ("dead tuples") pile up over time and make your table bloated and slow.

**VACUUM** cleans up these dead tuples.

```sql
-- Manual vacuum (rarely needed — autovacuum handles this)
VACUUM VERBOSE my_table;

-- VACUUM FULL — rewrites the entire table (reclaims disk space, but locks the table!)
VACUUM FULL my_table;

-- ANALYZE — updates table statistics so the query planner makes better decisions
ANALYZE my_table;

-- Do both
VACUUM ANALYZE my_table;
```

**Autovacuum tuning (postgresql.conf):**

```ini
# Enable autovacuum (should always be on!)
autovacuum = on

# Run autovacuum every 1 minute
autovacuum_naptime = 60

# Trigger vacuum when 20% of rows + 50 rows are dead
autovacuum_vacuum_scale_factor = 0.2
autovacuum_vacuum_threshold = 50

# Trigger analyze when 10% of rows + 50 rows have changed
autovacuum_analyze_scale_factor = 0.1
autovacuum_analyze_threshold = 50

# For large tables (millions of rows), lower the scale factor:
-- ALTER TABLE huge_table SET (autovacuum_vacuum_scale_factor = 0.01);
-- This triggers vacuum after just 1% of rows become dead
```

### 6. Caching Strategies

Don't hit the database for every request. Use caching.

```
┌──────────┐        ┌──────────┐        ┌──────────┐
│   App    │───────►│  Cache   │───────►│ Database │
│          │◄───────│  (Redis) │◄───────│          │
└──────────┘  Hit?  └──────────┘  Miss  └──────────┘
               │                          │
               │ Yes → return cached      │ Fetch from DB
               │       data immediately   │ Store in cache
               │                          │ Return to app
```

**Common patterns:**

#### Cache-Aside (Lazy Loading)
```python
import redis
import json

r = redis.Redis(host='localhost', port=6379, db=0)

def get_user(user_id):
    # 1. Check cache first
    cache_key = f"user:{user_id}"
    cached = r.get(cache_key)

    if cached:
        print("Cache HIT")
        return json.loads(cached)

    # 2. Cache miss — query database
    print("Cache MISS — querying DB")
    user = db.query(f"SELECT * FROM users WHERE id = {user_id}")

    # 3. Store in cache for next time (expire after 1 hour)
    r.setex(cache_key, 3600, json.dumps(user))

    return user

def update_user(user_id, data):
    # 1. Update database
    db.execute(f"UPDATE users SET ... WHERE id = {user_id}")

    # 2. Invalidate cache (so next read fetches fresh data)
    r.delete(f"user:{user_id}")
```

#### Write-Through
```python
def update_user(user_id, data):
    # 1. Update database
    db.execute(f"UPDATE users SET ... WHERE id = {user_id}")

    # 2. Update cache immediately (not invalidate — update!)
    r.setex(f"user:{user_id}", 3600, json.dumps(data))
```

### 7. Monitoring Your Database

You can't tune what you can't measure. Here are the key metrics to watch:

| Metric | What It Tells You | Danger Zone |
|---|---|---|
| **Active connections** | How many queries are running right now | > 80% of max_connections |
| **Query latency (p95, p99)** | How long queries take | > 500ms for OLTP |
| **Cache hit ratio** | % of data served from memory vs disk | < 99% for PostgreSQL (shared_buffers) |
| **Dead tuples** (PostgreSQL) | How much bloat is in your tables | Growing continuously = autovacuum isn't keeping up |
| **Replication lag** | How far behind replicas are | > 1 second |
| **Lock waits** | Queries waiting for locks | Any long-held locks |
| **Disk I/O** | Read/write throughput and IOPS | Sustained high I/O = need more RAM for caching |
| **Slow query count** | Number of queries exceeding your threshold | Any upward trend |

**PostgreSQL monitoring queries:**

```sql
-- Check cache hit ratio (should be > 99%)
SELECT
    sum(heap_blks_hit) / (sum(heap_blks_hit) + sum(heap_blks_read)) AS cache_hit_ratio
FROM pg_statio_user_tables;

-- Find slow queries (if pg_stat_statements extension is enabled)
SELECT
    query,
    calls,
    total_exec_time / calls AS avg_time_ms,
    rows / calls AS avg_rows
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 10;

-- Check table bloat (dead tuples)
SELECT
    schemaname, relname,
    n_live_tup,
    n_dead_tup,
    ROUND(n_dead_tup * 100.0 / NULLIF(n_live_tup, 0), 2) AS dead_pct,
    last_vacuum,
    last_autovacuum
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC
LIMIT 10;

-- Check active connections
SELECT
    state,
    COUNT(*)
FROM pg_stat_activity
GROUP BY state;

-- Check long-running queries
SELECT
    pid,
    now() - pg_stat_activity.query_start AS duration,
    query,
    state
FROM pg_stat_activity
WHERE state != 'idle'
  AND (now() - pg_stat_activity.query_start) > interval '5 minutes'
ORDER BY duration DESC;

-- Check locks
SELECT
    blocked.pid AS blocked_pid,
    blocked.query AS blocked_query,
    blocking.pid AS blocking_pid,
    blocking.query AS blocking_query
FROM pg_stat_activity blocked
JOIN pg_locks bl ON bl.pid = blocked.pid
JOIN pg_locks kl ON kl.locktype = bl.locktype
    AND kl.database IS NOT DISTINCT FROM bl.database
    AND kl.relation IS NOT DISTINCT FROM bl.relation
    AND kl.page IS NOT DISTINCT FROM bl.page
    AND kl.tuple IS NOT DISTINCT FROM bl.tuple
    AND kl.pid != bl.pid
JOIN pg_stat_activity blocking ON blocking.pid = kl.pid
WHERE NOT bl.granted;
```

## The Golden Formula for Pool Sizing

This is the famous **PostgreSQL wiki formula** for optimal pool size:

$$
\text{Pool Size} = (C \times 2) + D
$$

Where:
- $C$ = Number of CPU cores on the database server
- $D$ = Number of spinning disk drives (set to 1 for SSDs)

**Example:**
- 8-core CPU, SSD storage → Pool Size = (8 × 2) + 1 = **17**

That's it! More connections ≠ better performance. In fact, **too many connections make things worse** because of context switching overhead.

> **Counterintuitive truth:** A pool size of 20 will almost always outperform a pool size of 200 on the same hardware.

## Quick Reference — Connection Pooling Checklist

| ✅ | Best Practice |
|---|---|
| ✅ | Always use connection pooling — never open/close connections per request |
| ✅ | Set `max pool size` based on the CPU formula, not arbitrarily |
| ✅ | Use `pool_pre_ping` or validation queries to handle stale connections |
| ✅ | Set `max_lifetime` to avoid issues with firewalls/load balancers killing idle connections |
| ✅ | Enable leak detection to catch unreturned connections |
| ✅ | Use an external pooler (PgBouncer/ProxySQL) in microservices/Kubernetes |
| ✅ | Monitor pool metrics — checked out, idle, wait count, wait time |
| ✅ | Use `EXPLAIN ANALYZE` before and after optimization |
| ✅ | Add indexes based on actual query patterns, not guesses |
| ✅ | Keep `max_connections` low and use pooling to multiplex |
| ✅ | Tune memory settings (shared_buffers, work_mem, innodb_buffer_pool_size) |
| ✅ | Set up slow query logging from day one |
| ✅ | Cache frequently-read, rarely-changed data in Redis |

## Summary

| Concept | Key Takeaway |
|---|---|
| **Connection Pooling** | Reuse connections instead of creating/destroying them. Saves 20–100ms per request. |
| **Pool Sizing** | Use the formula: (CPU cores × 2) + 1. More connections ≠ better. |
| **External Poolers** | Use PgBouncer or ProxySQL when you have many app instances. |
| **Query Optimization** | Use EXPLAIN ANALYZE. Avoid SELECT *, avoid N+1, use proper indexes. |
| **Indexing** | B-tree for most cases. Use composite, covering, and partial indexes strategically. |
| **Memory Tuning** | shared_buffers = 25% RAM, effective_cache_size = 75% RAM, innodb_buffer_pool = 70-80% RAM. |
| **VACUUM** | Keep autovacuum enabled and tuned for your workload. |
| **Caching** | Use Redis. Cache-aside pattern is the most common. |
| **Monitoring** | Track cache hit ratio, active connections, slow queries, dead tuples, replication lag. |
