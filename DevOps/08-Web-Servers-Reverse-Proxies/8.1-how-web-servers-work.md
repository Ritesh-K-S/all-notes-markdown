# 8.1 — How Web Servers Work — Request/Response Lifecycle

> **Level:** Beginner  
> **Chapter:** 08 — Web Servers & Reverse Proxies  
> **Goal:** Understand what a web server actually does, how it receives requests and sends back responses, and the complete journey of a web request from your browser to the server and back.

---

## 📖 What is a Web Server?

A **web server** is simply a piece of software (sometimes also the physical/virtual machine it runs on) that **listens for incoming requests** from clients (like your browser) and **sends back responses** (like a web page, an image, or JSON data).

Think of it like a **restaurant waiter**:
- You (the client/browser) sit at a table and place an order (send a request).
- The waiter (web server) takes your order to the kitchen (backend application, database, files, etc.).
- The kitchen prepares your food (processes the request).
- The waiter brings the food back to your table (sends the response).

That's it. At its core, a web server is a **request handler**.

## 🧱 Two Types of Content a Web Server Can Serve

### 1. Static Content
Files that don't change — they are sent to the client exactly as they are stored on disk.

**Examples:**
- HTML files (`index.html`)
- CSS files (`style.css`)
- JavaScript files (`app.js`)
- Images (`logo.png`, `photo.jpg`)
- PDF documents
- Videos

The web server simply **reads the file from disk** and **sends it to the client**. No processing needed.

### 2. Dynamic Content
Content that is **generated on-the-fly** based on the request. The web server hands the request to a backend application (written in Python, Node.js, Java, Go, PHP, etc.) which processes it and returns the result.

**Examples:**
- A user's profile page (different data for each user)
- Search results (depends on the search query)
- API responses (JSON data from a database)
- Dashboard with real-time statistics

## 🔄 The Complete Request/Response Lifecycle

Let's trace what happens when you type `https://www.example.com/about` in your browser and hit Enter. This is the **full journey**:

### Step 1: URL Parsing (Browser)

Your browser breaks down the URL:

```
https://www.example.com/about
  │        │              │
  │        │              └── Path: /about
  │        └── Domain: www.example.com
  └── Protocol: HTTPS (port 443)
```

The browser now knows:
- **Protocol:** HTTPS (secure HTTP)
- **Domain:** www.example.com
- **Port:** 443 (default for HTTPS)
- **Path:** /about

---

### Step 2: DNS Resolution (Finding the Server's IP Address)

Your browser doesn't know where `www.example.com` physically lives. It needs an **IP address** to connect. So it performs a **DNS lookup**:

1. Browser checks its **local cache** → "Do I already know this IP?"
2. If not, it asks the **Operating System's DNS cache**.
3. If not there, it asks the **DNS resolver** (usually your ISP's DNS server or something like `8.8.8.8` Google DNS).
4. The resolver queries **root servers → TLD servers → authoritative name servers** until it finds the answer.

**Result:** `www.example.com` → `93.184.216.34`

Now the browser knows exactly which server to talk to!

---

### Step 3: TCP Connection (Three-Way Handshake)

Before any data can be sent, the browser and the server need to **establish a connection** using **TCP** (Transmission Control Protocol).

This happens via the **three-way handshake**:

```
Browser                          Server (93.184.216.34:443)
   │                                  │
   │──── SYN ────────────────────────►│   (1) "Hey, I want to connect"
   │                                  │
   │◄─── SYN-ACK ────────────────────│   (2) "Got it, I'm ready too"
   │                                  │
   │──── ACK ────────────────────────►│   (3) "Great, let's go!"
   │                                  │
   ══════ Connection Established ══════
```

- **SYN** — Browser says "I want to start a conversation"
- **SYN-ACK** — Server says "I heard you, and I'm also ready"
- **ACK** — Browser says "Awesome, let's talk"

Now we have a **reliable connection** between the browser and the server.

---

### Step 4: TLS/SSL Handshake (For HTTPS Only)

Since we're using HTTPS, there's an additional step to **encrypt the connection**. The **TLS handshake** happens right after the TCP handshake:

```
Browser                              Server
   │                                    │
   │── ClientHello ────────────────────►│  (Supported TLS versions, ciphers)
   │                                    │
   │◄── ServerHello + Certificate ──────│  (Server's SSL certificate + public key)
   │                                    │
   │── Key Exchange ───────────────────►│  (Browser generates a session key)
   │                                    │
   │◄── Finished ──────────────────────│
   │                                    │
   ═══════ Encrypted Connection ═══════
```

**In simple words:**
1. Browser says "Here's what encryption methods I support."
2. Server picks one and sends its **SSL certificate** (proves it's really `example.com`).
3. Browser verifies the certificate, and both sides agree on a **secret encryption key**.
4. From now on, all communication is **encrypted** — no one in between can read it.

---

### Step 5: HTTP Request (Browser → Server)

Now the browser sends the actual **HTTP request** through the encrypted connection:

```http
GET /about HTTP/1.1
Host: www.example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept: text/html,application/xhtml+xml
Accept-Language: en-US,en;q=0.9
Accept-Encoding: gzip, deflate, br
Connection: keep-alive
```

Let's break this down:

| Part | Meaning |
|------|---------|
| `GET` | HTTP method — "I want to **read/get** something" |
| `/about` | The path — which resource (page) I want |
| `HTTP/1.1` | The version of HTTP we're using |
| `Host: www.example.com` | Which website (important because one server can host many sites) |
| `User-Agent` | Info about the browser making the request |
| `Accept` | What types of content the browser can handle |
| `Accept-Encoding` | Compression methods the browser supports |
| `Connection: keep-alive` | Keep the connection open for more requests |

---

### Step 6: Web Server Receives & Processes the Request

The web server (e.g., Nginx, Apache) receives the request and decides what to do:

```
                    ┌──────────────────────────┐
                    │       Web Server          │
                    │      (e.g., Nginx)        │
                    └────────────┬─────────────┘
                                 │
                    ┌────────────▼─────────────┐
                    │   Check the request:      │
                    │   Path = /about           │
                    └────────────┬─────────────┘
                                 │
                 ┌───────────────┼───────────────┐
                 ▼                               ▼
        ┌────────────────┐              ┌────────────────┐
        │ Static File?   │              │ Dynamic App?   │
        │ /about.html    │              │ Pass to backend│
        └───────┬────────┘              └───────┬────────┘
                │                               │
                ▼                               ▼
        Read file from                  Forward to app server
        disk and serve it               (Node.js, Python, etc.)
                                        App processes & returns
                                        the response
```

**If it's a static file:**
The web server reads `/about.html` from the file system and prepares the response.

**If it's a dynamic request:**
The web server **forwards (proxies)** the request to a backend application:
- A Node.js app running on port 3000
- A Python Flask/Django app running on port 8000
- A Java Spring Boot app running on port 8080
- etc.

The backend app processes the request (maybe queries a database), and sends the result back to the web server.

---

### Step 7: HTTP Response (Server → Browser)

The web server sends back an **HTTP response**:

```http
HTTP/1.1 200 OK
Content-Type: text/html; charset=UTF-8
Content-Length: 5432
Content-Encoding: gzip
Cache-Control: max-age=3600
Date: Thu, 29 May 2026 08:00:00 GMT
Server: nginx/1.25.3

<!DOCTYPE html>
<html>
<head>
    <title>About Us — Example</title>
</head>
<body>
    <h1>About Us</h1>
    <p>Welcome to our about page!</p>
</body>
</html>
```

Let's break this down:

| Part | Meaning |
|------|---------|
| `HTTP/1.1 200 OK` | Status line — version + **status code** (200 = success!) |
| `Content-Type: text/html` | "I'm sending you an HTML page" |
| `Content-Length: 5432` | "The body is 5432 bytes" |
| `Content-Encoding: gzip` | "The body is compressed with gzip" |
| `Cache-Control: max-age=3600` | "You can cache this for 1 hour" |
| `Server: nginx/1.25.3` | "I'm running Nginx" |
| `<html>...` | The actual HTML content (the **response body**) |

---

### Step 8: Browser Renders the Page

The browser receives the response and:

1. **Parses the HTML** — builds the DOM (Document Object Model) tree.
2. **Fetches additional resources** — the HTML might reference CSS files, JavaScript files, images, fonts, etc. Each of these triggers **additional HTTP requests**.
3. **Applies CSS** — styles the elements.
4. **Executes JavaScript** — makes the page interactive.
5. **Paints the page** — renders everything on your screen.

You see the "About Us" page! 🎉

## 🔢 HTTP Status Codes You Must Know

Web servers respond with **status codes** that tell the client what happened:

### 1xx — Informational
| Code | Meaning |
|------|---------|
| 100 | Continue — "Keep going, send the rest of the request" |
| 101 | Switching Protocols — "Sure, let's switch to WebSocket" |

### 2xx — Success ✅
| Code | Meaning |
|------|---------|
| 200 | OK — "Everything went well, here's your data" |
| 201 | Created — "Resource was successfully created" |
| 204 | No Content — "Success, but there's nothing to send back" |

### 3xx — Redirection ↪️
| Code | Meaning |
|------|---------|
| 301 | Moved Permanently — "This page has moved forever, go to new URL" |
| 302 | Found (Temporary Redirect) — "Temporarily go to this other URL" |
| 304 | Not Modified — "Use your cached version, nothing changed" |

### 4xx — Client Error ❌
| Code | Meaning |
|------|---------|
| 400 | Bad Request — "Your request doesn't make sense" |
| 401 | Unauthorized — "You need to log in first" |
| 403 | Forbidden — "You're logged in but don't have permission" |
| 404 | Not Found — "This page/resource doesn't exist" |
| 405 | Method Not Allowed — "You can't use this HTTP method here" |
| 408 | Request Timeout — "You took too long to send the request" |
| 429 | Too Many Requests — "Slow down! You're being rate-limited" |

### 5xx — Server Error 💥
| Code | Meaning |
|------|---------|
| 500 | Internal Server Error — "Something broke on the server" |
| 502 | Bad Gateway — "The web server got a bad response from the backend" |
| 503 | Service Unavailable — "Server is overloaded or down for maintenance" |
| 504 | Gateway Timeout — "The backend took too long to respond" |

## 🔧 HTTP Methods (Verbs)

When the client sends a request, it specifies an **HTTP method** that tells the server **what action** to perform:

| Method | Purpose | Example |
|--------|---------|---------|
| `GET` | **Read/retrieve** data | Get a web page, fetch API data |
| `POST` | **Create** a new resource | Submit a form, create a new user |
| `PUT` | **Update/replace** an entire resource | Update a user's entire profile |
| `PATCH` | **Partially update** a resource | Change just the email address |
| `DELETE` | **Remove** a resource | Delete a user account |
| `HEAD` | Same as GET but **no body** — just headers | Check if a page exists |
| `OPTIONS` | Ask what methods are **allowed** | CORS preflight requests |

## 🏗️ How a Web Server Handles Multiple Requests

A busy website might receive **thousands of requests per second**. How does a web server handle all of them? There are different models:

### 1. Process-Per-Request (Old Apache Model)

```
Request 1 → Process 1 (handles the request, uses lots of memory)
Request 2 → Process 2 (handles the request, uses lots of memory)
Request 3 → Process 3 (handles the request, uses lots of memory)
...
```

- Each request gets its own **process** (a separate running copy of the server).
- **Problem:** Each process uses a lot of memory. Handling 10,000 connections = 10,000 processes = 💥 (server runs out of memory).
- This is the **old Apache model** (`prefork`).

### 2. Thread-Per-Request (Apache Worker Model)

```
Request 1 → Thread 1 (lighter than a process)
Request 2 → Thread 2 
Request 3 → Thread 3
...
```

- Each request gets a **thread** (lighter than a process, shares memory).
- Better than processes, but still limited — thousands of threads can still overwhelm the server.

### 3. Event-Driven / Asynchronous (Nginx Model) ⭐

```
          ┌────────────────────────────────┐
          │     Single Worker Process       │
          │                                │
          │  Event Loop:                   │
          │    → Handle Request 1          │
          │    → Handle Request 2          │
          │    → Handle Request 3          │
          │    → Handle Request 4          │
          │    → ... (thousands more)      │
          └────────────────────────────────┘
```

- A **single worker process** handles **thousands of connections** using an **event loop**.
- Instead of waiting for a disk read or network response, it moves on to the next request and comes back when the data is ready.
- This is why **Nginx can handle 10,000+ concurrent connections** with very little memory.
- This is the model that made Nginx so popular!

## 🔌 Ports — Where Web Servers Listen

A web server **listens on a specific port** for incoming connections:

| Port | Protocol | Description |
|------|----------|-------------|
| **80** | HTTP | Default for unencrypted web traffic |
| **443** | HTTPS | Default for encrypted (SSL/TLS) web traffic |
| **8080** | HTTP (alt) | Common alternative port for development or proxied traffic |
| **8443** | HTTPS (alt) | Common alternative HTTPS port |

When you visit `http://example.com`, the browser automatically connects to port **80**.  
When you visit `https://example.com`, the browser automatically connects to port **443**.

You can run a web server on **any port**, but these are the standard ones.

## 🏠 Virtual Hosts — One Server, Multiple Websites

A single web server can host **multiple websites** on the same IP address using **virtual hosts** (also called **server blocks** in Nginx).

How? The browser sends the `Host` header in the HTTP request:

```http
GET / HTTP/1.1
Host: www.siteA.com       ← This tells the server which site you want
```

```http
GET / HTTP/1.1
Host: www.siteB.com       ← Different Host header = different site
```

The web server reads the `Host` header and routes the request to the correct website's files:

```
                   ┌───────────────────────┐
                   │      Web Server        │
                   │    (one IP address)    │
                   └───────────┬───────────┘
                               │
              ┌────────────────┼────────────────┐
              ▼                                 ▼
     Host: siteA.com                   Host: siteB.com
     Serve files from:                 Serve files from:
     /var/www/siteA/                   /var/www/siteB/
```

This is **extremely common** in production — a single server often hosts dozens of websites.

## 🔒 HTTP vs HTTPS — Why HTTPS Matters

| Feature | HTTP | HTTPS |
|---------|------|-------|
| Encryption | ❌ None — data sent in plain text | ✅ Encrypted with TLS/SSL |
| Port | 80 | 443 |
| Security | Anyone can read the data in transit | Only sender & receiver can read it |
| SEO | Google penalizes HTTP sites | Google prefers HTTPS sites |
| Trust | Browser shows "Not Secure" warning | Browser shows 🔒 padlock icon |

**Bottom line:** Always use HTTPS. There's no reason to use plain HTTP in production anymore.

## 📝 Key Terminology Quick Reference

| Term | Simple Definition |
|------|-------------------|
| **Client** | The thing making the request (usually a browser) |
| **Server** | The machine/software receiving and responding to requests |
| **Request** | A message from the client to the server asking for something |
| **Response** | A message from the server back to the client with the answer |
| **HTTP** | The protocol (language) that clients and servers use to communicate |
| **HTTPS** | HTTP + encryption (secure version) |
| **Port** | A numbered "door" on the server where it listens for connections |
| **DNS** | The system that translates domain names to IP addresses |
| **Static Content** | Files served as-is from disk (HTML, CSS, images) |
| **Dynamic Content** | Content generated on-the-fly by a backend application |
| **Virtual Host** | A configuration that allows one server to host multiple websites |
| **Reverse Proxy** | A server that sits in front of backend apps and forwards requests to them |
| **TLS/SSL** | The encryption protocols that make HTTPS secure |
| **Status Code** | A number in the response that tells the client what happened (200, 404, 500, etc.) |

## ❓ Common Interview Questions

1. **What is a web server?**
   → Software that listens for HTTP requests and sends back HTTP responses.

2. **What's the difference between static and dynamic content?**
   → Static content is served as-is from files. Dynamic content is generated by backend applications per-request.

3. **Explain the HTTP request/response lifecycle.**
   → DNS lookup → TCP handshake → TLS handshake (HTTPS) → HTTP request → Server processing → HTTP response → Browser rendering.

4. **Why is Nginx faster than traditional Apache?**
   → Nginx uses an event-driven, non-blocking architecture that handles thousands of concurrent connections with minimal memory, while Apache traditionally spawns a new process/thread per connection.

5. **What's the difference between HTTP and HTTPS?**
   → HTTPS is HTTP with TLS/SSL encryption. It ensures data privacy, integrity, and authentication.

6. **What is a virtual host?**
   → A configuration that allows a single web server to host multiple websites on one IP address by checking the `Host` header of each request.
