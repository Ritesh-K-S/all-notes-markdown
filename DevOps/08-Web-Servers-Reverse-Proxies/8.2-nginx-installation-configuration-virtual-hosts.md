## 📖 What is Nginx?

**Nginx** (pronounced "Engine-X") is the **most popular web server** in the world. It was created in 2004 by **Igor Sysoev** to solve the **C10K problem** — how to handle 10,000 concurrent connections on a single server.

Today, Nginx powers a huge percentage of the internet including sites like Netflix, Airbnb, Dropbox, WordPress.com, and more.

### Why is Nginx So Popular?

| Feature | Description |
|---------|-------------|
| **Blazing Fast** | Event-driven architecture handles thousands of connections with tiny memory footprint |
| **Lightweight** | Uses way less RAM than Apache |
| **Reverse Proxy** | Can sit in front of backend apps and forward requests |
| **Load Balancer** | Can distribute traffic across multiple backend servers |
| **Static Content** | Serves static files (HTML, CSS, JS, images) incredibly fast |
| **SSL/TLS** | Handles HTTPS encryption efficiently |
| **Highly Configurable** | Very flexible configuration system |
| **Widely Adopted** | Massive community, tons of tutorials, well-documented |

## 📁 Nginx File Structure — Where Everything Lives

Understanding the file structure is **crucial** before you start configuring anything. Here's the layout:

```
/etc/nginx/                        ← Main Nginx configuration directory
├── nginx.conf                     ← THE main config file (the boss)
├── conf.d/                        ← Additional config files (loaded by nginx.conf)
│   └── default.conf               ← Default server configuration
├── sites-available/               ← All site configs (available but not necessarily active)
│   └── default                    ← Default site config
├── sites-enabled/                 ← Active site configs (symlinks to sites-available)
│   └── default → ../sites-available/default
├── mime.types                     ← Maps file extensions to MIME types
├── modules-available/             ← Available Nginx modules
├── modules-enabled/               ← Active Nginx modules
└── snippets/                      ← Reusable config snippets

/var/www/                          ← Default web root (where website files live)
└── html/
    └── index.nginx-debian.html    ← The default welcome page

/var/log/nginx/                    ← Nginx log files
├── access.log                     ← Every request that hits the server
└── error.log                      ← Errors and warnings

/run/nginx.pid                     ← Process ID of running Nginx
```

### Key Files Explained

| File/Directory | Purpose |
|----------------|---------|
| `/etc/nginx/nginx.conf` | The **main** configuration file. Everything starts here. |
| `/etc/nginx/sites-available/` | Store all your site configs here (one file per site). |
| `/etc/nginx/sites-enabled/` | **Symlinks** to configs in `sites-available/` — only these are active. |
| `/etc/nginx/conf.d/` | Alternative to `sites-available/` — some distros use this instead. |
| `/var/www/html/` | Default directory where website files are served from. |
| `/var/log/nginx/access.log` | Logs every request (who visited, what they requested, status code). |
| `/var/log/nginx/error.log` | Logs errors (misconfigurations, crashes, failed requests). |

> **Note:** Ubuntu/Debian uses `sites-available/` and `sites-enabled/`. CentOS/RHEL typically uses `conf.d/`. Both work the same way.

## 📄 Understanding nginx.conf — The Main Config File

Let's look at the main config file and understand every part:

```bash
sudo cat /etc/nginx/nginx.conf
```

Here's a typical `nginx.conf` with detailed comments:

```nginx
# ─── Global Context ──────────────────────────────────────────
# These settings apply to the entire Nginx process.

user www-data;                    # Which user Nginx runs as
worker_processes auto;            # Number of worker processes (auto = one per CPU core)
pid /run/nginx.pid;               # Where to store the process ID
error_log /var/log/nginx/error.log;  # Global error log file

# ─── Events Context ──────────────────────────────────────────
# Controls how Nginx handles connections at a low level.

events {
    worker_connections 1024;      # Max connections per worker process
                                  # Total max connections = worker_processes × worker_connections
                                  # Example: 4 cores × 1024 = 4096 simultaneous connections
    
    # multi_accept on;            # Accept multiple connections at once (optional)
    # use epoll;                  # Use the epoll event model on Linux (auto-detected)
}

# ─── HTTP Context ─────────────────────────────────────────────
# All HTTP-related configuration goes here.

http {
    # ── Basic Settings ──
    sendfile on;                  # Efficient file transfer (bypass user-space)
    tcp_nopush on;                # Optimize the amount of data sent at once
    tcp_nodelay on;               # Send data immediately (don't wait to batch)
    keepalive_timeout 65;         # Keep connections alive for 65 seconds
    types_hash_max_size 2048;     # Size of the MIME types hash table

    include /etc/nginx/mime.types;    # Load MIME type mappings
    default_type application/octet-stream;  # Default type if not matched

    # ── Logging ──
    access_log /var/log/nginx/access.log;   # Log all requests
    error_log /var/log/nginx/error.log;     # Log errors

    # ── Gzip Compression ──
    gzip on;                      # Enable gzip compression
    gzip_types text/plain text/css application/json application/javascript;

    # ── Include Site Configs ──
    include /etc/nginx/conf.d/*.conf;           # Load configs from conf.d/
    include /etc/nginx/sites-enabled/*;         # Load active site configs
}
```

### The Config Hierarchy (Contexts)

Nginx configuration is organized in **nested blocks (contexts)**:

```
Main (Global)          ← Top-level settings (user, workers, pid)
├── events { }         ← Connection handling settings
└── http { }           ← All HTTP settings
    └── server { }     ← Virtual host / site config
        └── location { }  ← URL path matching rules
```

Each context **inherits** settings from its parent and can **override** them.

## 🏗️ Your First Nginx Configuration — Serving a Static Website

Let's create a real website and serve it with Nginx!

### Step 1: Create the Website Files

```bash
# Create a directory for your website
sudo mkdir -p /var/www/mysite

# Create an HTML file
sudo nano /var/www/mysite/index.html
```

Add this HTML:

```html
<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>My First Nginx Site</title>
    <style>
        body {
            font-family: Arial, sans-serif;
            max-width: 800px;
            margin: 50px auto;
            padding: 20px;
            background-color: #f5f5f5;
        }
        h1 { color: #2d5016; }
        p { line-height: 1.6; }
    </style>
</head>
<body>
    <h1>Hello from Nginx!</h1>
    <p>If you see this page, your Nginx configuration is working correctly.</p>
    <p>Server Time: <!-- This is a static page, no dynamic time --></p>
</body>
</html>
```

### Step 2: Set Correct Ownership and Permissions

```bash
# Make sure Nginx can read the files
sudo chown -R www-data:www-data /var/www/mysite
sudo chmod -R 755 /var/www/mysite
```

### Step 3: Create the Nginx Server Block (Site Config)

```bash
sudo nano /etc/nginx/sites-available/mysite
```

Add this configuration:

```nginx
server {
    listen 80;                          # Listen on port 80 (HTTP)
    server_name mysite.local;           # Domain name for this site
    root /var/www/mysite;               # Where the website files are
    index index.html;                   # Default file to serve

    # Serve static files
    location / {
        try_files $uri $uri/ =404;      # Try the file, then directory, then 404
    }

    # Custom error page
    error_page 404 /404.html;

    # Logs for this specific site
    access_log /var/log/nginx/mysite.access.log;
    error_log /var/log/nginx/mysite.error.log;
}
```

Let's break down every directive:

| Directive | What It Does |
|-----------|-------------|
| `listen 80` | Listen for connections on port 80 (HTTP) |
| `server_name mysite.local` | This block handles requests for `mysite.local` |
| `root /var/www/mysite` | The base directory for this website's files |
| `index index.html` | If someone visits `/`, serve `index.html` |
| `try_files $uri $uri/ =404` | Try the requested file → try as a directory → return 404 if nothing found |
| `access_log` | Where to log requests for this site |
| `error_log` | Where to log errors for this site |

### Step 4: Enable the Site

```bash
# Create a symbolic link from sites-available to sites-enabled
sudo ln -s /etc/nginx/sites-available/mysite /etc/nginx/sites-enabled/

# (Optional) Remove the default site if you want
# sudo rm /etc/nginx/sites-enabled/default
```

### Step 5: Test the Configuration

**Always test before reloading!** A bad config can bring down Nginx.

```bash
sudo nginx -t
```

Expected output:
```
nginx: the configuration file /etc/nginx/nginx.conf syntax is ok
nginx: configuration file /etc/nginx/nginx.conf test is successful
```

### Step 6: Reload Nginx

```bash
# Reload applies new config WITHOUT dropping existing connections
sudo systemctl reload nginx
```

> **reload vs restart:**
> - `reload` = Gracefully apply new config (no downtime)
> - `restart` = Stop and start Nginx (brief downtime)
> - Always prefer `reload` in production!

### Step 7: Test It

```bash
# Add a local DNS entry (since mysite.local doesn't exist in real DNS)
echo "127.0.0.1 mysite.local" | sudo tee -a /etc/hosts

# Now test it
curl http://mysite.local
```

You should see your HTML page! 🎉

## 🔑 Essential Nginx Directives — Detailed Explanation

### The `server` Block

The `server` block defines a **virtual host** — a website or application:

```nginx
server {
    listen 80;                    # Which port to listen on
    listen [::]:80;               # Also listen on IPv6
    server_name example.com www.example.com;  # Domain names
    root /var/www/example;        # Document root
    index index.html index.htm;   # Default files to serve
}
```

### The `location` Block

The `location` block tells Nginx **what to do** for specific URL paths:

```nginx
# Exact match
location = /about {
    # Only matches /about exactly (not /about/ or /about/team)
    return 200 "This is the about page";
}

# Prefix match (most common)
location / {
    # Matches everything starting with /
    try_files $uri $uri/ =404;
}

# Regex match (case-sensitive)
location ~ \.php$ {
    # Matches any URL ending in .php
    # Example: /index.php, /api/users.php
}

# Regex match (case-insensitive)
location ~* \.(jpg|jpeg|png|gif|ico)$ {
    # Matches image files regardless of case
    # Example: /images/Logo.PNG, /photos/cat.jpg
    expires 30d;    # Cache images for 30 days
}

# Prefix match (prioritized over regex)
location ^~ /static/ {
    # If the URL starts with /static/, use this block
    # and DON'T check regex locations
    root /var/www/assets;
}
```

### Location Matching Priority

Nginx picks the location block in this order:

```
1.  =   (Exact match)              ← Highest priority
2.  ^~  (Prefix match, skip regex) ← Second highest
3.  ~   (Regex, case-sensitive)    ← Third
4.  ~*  (Regex, case-insensitive)  ← Fourth
5.  /   (Prefix match, normal)     ← Lowest priority (fallback)
```

**Example:**
For a request to `/images/cat.jpg`:

```nginx
location = /images/cat.jpg { }    # ← Wins if it exists (exact match)
location ^~ /images/ { }          # ← Wins next (prefix, skip regex)
location ~* \.(jpg|png)$ { }      # ← Wins next (regex)
location /images/ { }             # ← Lowest priority
```

## 🌐 Virtual Hosts — Hosting Multiple Websites on One Server

One of the most powerful features of Nginx is hosting **multiple websites** on a single server. Each website gets its own **server block**.

### Example: Hosting Two Websites

#### Site 1: blog.example.com

```bash
# Create directory and files
sudo mkdir -p /var/www/blog
echo "<h1>Welcome to My Blog!</h1>" | sudo tee /var/www/blog/index.html

# Create the config
sudo nano /etc/nginx/sites-available/blog
```

```nginx
server {
    listen 80;
    server_name blog.example.com;
    root /var/www/blog;
    index index.html;

    location / {
        try_files $uri $uri/ =404;
    }

    access_log /var/log/nginx/blog.access.log;
    error_log /var/log/nginx/blog.error.log;
}
```

#### Site 2: shop.example.com

```bash
# Create directory and files
sudo mkdir -p /var/www/shop
echo "<h1>Welcome to My Shop!</h1>" | sudo tee /var/www/shop/index.html

# Create the config
sudo nano /etc/nginx/sites-available/shop
```

```nginx
server {
    listen 80;
    server_name shop.example.com;
    root /var/www/shop;
    index index.html;

    location / {
        try_files $uri $uri/ =404;
    }

    access_log /var/log/nginx/shop.access.log;
    error_log /var/log/nginx/shop.error.log;
}
```

#### Enable Both Sites

```bash
# Enable the sites
sudo ln -s /etc/nginx/sites-available/blog /etc/nginx/sites-enabled/
sudo ln -s /etc/nginx/sites-available/shop /etc/nginx/sites-enabled/

# Test the configuration
sudo nginx -t

# Reload Nginx
sudo systemctl reload nginx
```

Now:
- Requests to `blog.example.com` → served from `/var/www/blog/`
- Requests to `shop.example.com` → served from `/var/www/shop/`

Both running on the **same server, same IP, same port 80**!

### How Does It Work?

```
Browser sends: GET / HTTP/1.1
               Host: blog.example.com     ← Nginx reads this header
                                            and matches it to the right
                                            server block

Browser sends: GET / HTTP/1.1
               Host: shop.example.com     ← Different Host = different site
```

### Default Server (Catch-All)

What happens if someone accesses your server by IP address or a domain you haven't configured? The **default server** handles it:

```nginx
server {
    listen 80 default_server;         # This is the default/fallback server
    server_name _;                    # _ matches nothing (catch-all)

    # Option 1: Show a custom page
    root /var/www/default;
    index index.html;

    # Option 2: Return 444 (Nginx drops the connection)
    # return 444;

    # Option 3: Redirect to your main site
    # return 301 https://www.example.com$request_uri;
}
```

## 📊 Serving Static Files Efficiently

Nginx is **incredibly fast** at serving static files. Here are best practices:

### Basic Static File Serving

```nginx
server {
    listen 80;
    server_name static.example.com;
    root /var/www/static;

    # Serve static files with caching headers
    location ~* \.(css|js)$ {
        expires 7d;                    # Cache CSS/JS for 7 days
        add_header Cache-Control "public, immutable";
    }

    location ~* \.(jpg|jpeg|png|gif|ico|svg|webp)$ {
        expires 30d;                   # Cache images for 30 days
        add_header Cache-Control "public, immutable";
    }

    location ~* \.(woff|woff2|ttf|eot)$ {
        expires 365d;                  # Cache fonts for 1 year
        add_header Cache-Control "public, immutable";
    }

    # Enable gzip compression
    gzip on;
    gzip_types text/plain text/css application/json application/javascript text/xml;
    gzip_min_length 1000;             # Don't compress tiny files
}
```

### What `try_files` Does

`try_files` is one of the most used directives. It tries multiple options in order:

```nginx
location / {
    try_files $uri $uri/ /index.html;
}
```

What this means:
1. `$uri` — Try the exact file requested (e.g., `/about.html`)
2. `$uri/` — Try as a directory (e.g., `/about/index.html`)
3. `/index.html` — If nothing found, serve `/index.html` (great for SPAs like React/Vue)

For a request to `/about`:
```
Step 1: Does /var/www/mysite/about exist as a file? → No
Step 2: Does /var/www/mysite/about/ exist as a directory? → No  
Step 3: Serve /var/www/mysite/index.html → Yes! (SPA handles routing)
```

## 🧩 Nginx Variables You Should Know

Nginx has built-in variables you can use in your config:

| Variable | Value |
|----------|-------|
| `$host` | The domain name from the request (e.g., `example.com`) |
| `$uri` | The current URI path (e.g., `/about`) |
| `$request_uri` | The full original URI with query string (e.g., `/search?q=nginx`) |
| `$args` | The query string (e.g., `q=nginx&page=2`) |
| `$remote_addr` | The client's IP address |
| `$request_method` | The HTTP method (GET, POST, etc.) |
| `$scheme` | `http` or `https` |
| `$server_port` | The port the request came in on |
| `$status` | The HTTP status code of the response |
| `$request_time` | How long the request took to process (in seconds) |

### Using Variables in Config

```nginx
# Custom log format with useful variables
log_format custom '$remote_addr - $host [$time_local] '
                  '"$request" $status $body_bytes_sent '
                  '"$http_referer" "$http_user_agent" '
                  '$request_time';

access_log /var/log/nginx/access.log custom;
```

## 🔀 Redirects and Rewrites

### Redirect HTTP to HTTPS

```nginx
server {
    listen 80;
    server_name example.com www.example.com;
    
    # 301 = permanent redirect
    return 301 https://$host$request_uri;
}
```

### Redirect www to non-www

```nginx
server {
    listen 80;
    server_name www.example.com;
    return 301 https://example.com$request_uri;
}
```

### Redirect a Specific Path

```nginx
location = /old-page {
    return 301 /new-page;
}
```

### Rewrite URLs (Internal)

```nginx
# Internally rewrite /user/john to /profile?name=john
# The user's browser still shows /user/john
rewrite ^/user/(.+)$ /profile?name=$1 last;
```

## 🛡️ Basic Security Headers

Add these headers to improve security:

```nginx
server {
    # ... your other config ...

    # Prevent clickjacking
    add_header X-Frame-Options "SAMEORIGIN" always;

    # Prevent MIME-type sniffing
    add_header X-Content-Type-Options "nosniff" always;

    # Enable XSS protection
    add_header X-XSS-Protection "1; mode=block" always;

    # Referrer policy
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;

    # Hide Nginx version (don't reveal server info)
    server_tokens off;
}
```

## 🐛 Common Mistakes & Troubleshooting

### Mistake 1: Forgetting to test config before reloading

```bash
# ALWAYS do this before reload:
sudo nginx -t

# If you see errors, DO NOT reload. Fix the config first.
```

### Mistake 2: Forgetting the semicolon

```nginx
# WRONG — missing semicolon
server_name example.com    # ← Will cause "unexpected }" error

# CORRECT
server_name example.com;   # ← Semicolons are required after every directive
```

### Mistake 3: Permission denied errors

```bash
# Check file permissions
ls -la /var/www/mysite/

# Fix ownership
sudo chown -R www-data:www-data /var/www/mysite/

# Fix permissions
sudo chmod -R 755 /var/www/mysite/
```

### Mistake 4: Site not loading after creating config

```bash
# Did you create the symlink?
ls -la /etc/nginx/sites-enabled/

# If your site isn't listed, create the symlink:
sudo ln -s /etc/nginx/sites-available/mysite /etc/nginx/sites-enabled/

# Then reload:
sudo nginx -t && sudo systemctl reload nginx
```

### Mistake 5: "Address already in use" error

```bash
# Check what's using port 80
sudo ss -tlnp | grep :80

# If Apache is running, stop it:
sudo systemctl stop apache2
sudo systemctl disable apache2
```

## 📋 Complete Example — Production-Ready Static Site Config

Here's a full, production-ready config for a static website:

```nginx
server {
    listen 80;
    listen [::]:80;
    server_name example.com www.example.com;

    # Redirect HTTP to HTTPS
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl http2;
    listen [::]:443 ssl http2;
    server_name example.com www.example.com;

    # SSL Certificate (we'll cover SSL setup in 8.3)
    ssl_certificate /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

    # Document root
    root /var/www/example;
    index index.html;

    # Security headers
    add_header X-Frame-Options "SAMEORIGIN" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-XSS-Protection "1; mode=block" always;
    server_tokens off;

    # Gzip compression
    gzip on;
    gzip_types text/plain text/css application/json application/javascript text/xml;
    gzip_min_length 1000;

    # Main location
    location / {
        try_files $uri $uri/ =404;
    }

    # Cache static assets
    location ~* \.(css|js|jpg|jpeg|png|gif|ico|svg|woff|woff2)$ {
        expires 30d;
        add_header Cache-Control "public, immutable";
    }

    # Deny access to hidden files (e.g., .git, .env)
    location ~ /\. {
        deny all;
        access_log off;
        log_not_found off;
    }

    # Custom 404 page
    error_page 404 /404.html;
    location = /404.html {
        internal;
    }

    # Logs
    access_log /var/log/nginx/example.access.log;
    error_log /var/log/nginx/example.error.log;
}
```

## ❓ Common Interview Questions

1. **What is the difference between `sites-available` and `sites-enabled`?**
   → `sites-available` stores all site configs. `sites-enabled` contains symlinks to configs that are currently active. This lets you enable/disable sites without deleting configs.

2. **How do you test an Nginx config without restarting?**
   → Run `sudo nginx -t` to validate the config syntax.

3. **What's the difference between `reload` and `restart`?**
   → `reload` gracefully applies new config without dropping existing connections. `restart` stops and starts Nginx, causing a brief downtime.

4. **How does Nginx handle multiple virtual hosts on the same port?**
   → It reads the `Host` header from the HTTP request and matches it against the `server_name` directive in each server block.

5. **What does `try_files $uri $uri/ =404` mean?**
   → Try the exact file, then try as a directory, then return a 404 error if nothing is found.

6. **How do you host multiple websites on one Nginx server?**
   → Create separate server blocks (virtual hosts) with different `server_name` values, each pointing to its own `root` directory.
