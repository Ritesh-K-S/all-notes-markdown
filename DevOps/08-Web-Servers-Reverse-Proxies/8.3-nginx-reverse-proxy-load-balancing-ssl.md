# 8.3 — Nginx — Reverse Proxy, Load Balancing & SSL Termination

> **Level:** Intermediate  
> **Chapter:** 08 — Web Servers & Reverse Proxies  
> **Goal:** Master Nginx as a reverse proxy to sit in front of backend applications, distribute traffic across multiple servers with load balancing, and handle SSL/TLS termination — the three most critical Nginx skills in DevOps.

---

## 📖 What is a Reverse Proxy?

A **reverse proxy** is a server that sits **between the client and your backend application**. Instead of the client talking directly to your app, the client talks to the reverse proxy, and the reverse proxy forwards the request to your app.

### The Analogy

Think of a **receptionist at a big company**:

- You (the client) walk into the building and tell the receptionist (Nginx) who you want to meet.
- The receptionist doesn't let you wander around the building (backend servers) yourself.
- Instead, the receptionist contacts the right person (backend app) on your behalf.
- The person sends their reply through the receptionist back to you.

You **never directly interact** with the person inside — you only talk to the receptionist.

---

## 🔄 Forward Proxy vs Reverse Proxy

People often confuse these, so let's clear it up:

### Forward Proxy (Client-Side)

```
Client ──► Forward Proxy ──► Internet ──► Server
                 │
          "I sit in front of            
           the CLIENT and              
           hide the client's           
           identity"                   
```

- Sits in front of **clients** (users).
- Clients use it to access the internet anonymously or bypass restrictions.
- The **server doesn't know** who the real client is.
- Example: VPN, corporate proxy, Tor.

### Reverse Proxy (Server-Side)

```
Client ──► Internet ──► Reverse Proxy ──► Backend Server(s)
                              │
                       "I sit in front of
                        the SERVER and
                        hide the server's
                        identity"
```

- Sits in front of **backend servers**.
- Clients don't know the real backend server.
- The **client only sees** the reverse proxy.
- Example: Nginx, HAProxy, Traefik.

---

## 🤔 Why Use a Reverse Proxy?

| Benefit | Explanation |
|---------|-------------|
| **Security** | Hides backend servers from direct internet exposure. Attackers can't directly attack your app. |
| **SSL Termination** | Handles HTTPS encryption so your backend apps don't have to. |
| **Load Balancing** | Distributes traffic across multiple backend servers. |
| **Caching** | Caches responses to reduce load on backend servers. |
| **Compression** | Compresses responses (gzip) before sending to clients. |
| **Rate Limiting** | Controls how many requests a client can make (prevents abuse). |
| **Single Entry Point** | One domain can route to many different backend services. |
| **Static Content** | Nginx serves static files directly (fast!), only forwarding dynamic requests to the backend. |

## 🏗️ Setting Up Nginx as a Reverse Proxy

### The Scenario

You have a **Node.js application** running on `http://localhost:3000` and you want users to access it through `http://myapp.com` (port 80) via Nginx.

```
User's Browser                 Nginx (port 80)              Node.js App (port 3000)
     │                              │                              │
     │── GET http://myapp.com ─────►│                              │
     │                              │── GET http://localhost:3000 ─►│
     │                              │                              │
     │                              │◄── Response ─────────────────│
     │◄── Response ────────────────│                              │
```

### Step 1: Make Sure Your Backend App is Running

```bash
# Example: a simple Node.js app
# (your app can be Python, Java, Go, Ruby — anything)

# Start your app
node app.js    # Running on port 3000
```

### Step 2: Create the Nginx Reverse Proxy Config

```bash
sudo nano /etc/nginx/sites-available/myapp
```

```nginx
server {
    listen 80;
    server_name myapp.com www.myapp.com;

    location / {
        # Forward all requests to the Node.js app
        proxy_pass http://localhost:3000;

        # Pass important headers to the backend
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

### Step 3: Enable, Test & Reload

```bash
sudo ln -s /etc/nginx/sites-available/myapp /etc/nginx/sites-enabled/
sudo nginx -t
sudo systemctl reload nginx
```

That's it! Users visit `http://myapp.com` and Nginx forwards the request to your Node.js app on port 3000.

## 🔍 Understanding proxy_pass and Proxy Headers — In Detail

### `proxy_pass` — The Core Directive

`proxy_pass` tells Nginx **where to forward** the request:

```nginx
# Forward to a local app
proxy_pass http://localhost:3000;

# Forward to a specific IP
proxy_pass http://192.168.1.10:8080;

# Forward to a Unix socket
proxy_pass http://unix:/var/run/myapp.sock;
```

### Important: Trailing Slash Matters!

This is a very common source of bugs:

```nginx
# Without trailing slash — path is APPENDED
location /api/ {
    proxy_pass http://localhost:3000;
}
# Request: /api/users → Backend receives: /api/users

# With trailing slash — path is REPLACED
location /api/ {
    proxy_pass http://localhost:3000/;
}
# Request: /api/users → Backend receives: /users
```

**Rule of thumb:**
- **No trailing slash** on `proxy_pass` → the full original path is kept.
- **Trailing slash** on `proxy_pass` → the matched `location` prefix is stripped.

### Proxy Headers — Why They Matter

When Nginx forwards a request, the backend sees the request as coming from **Nginx** (127.0.0.1), not from the real user. Proxy headers fix this:

```nginx
# Tell the backend the original domain the user accessed
proxy_set_header Host $host;

# Tell the backend the real client IP address
proxy_set_header X-Real-IP $remote_addr;

# Chain of proxy IPs (if there are multiple proxies)
proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;

# Tell the backend whether the original request was HTTP or HTTPS
proxy_set_header X-Forwarded-Proto $scheme;
```

**Without these headers:**
```
Backend sees:
  - Client IP: 127.0.0.1 (Nginx's IP, not the real user)
  - Host: localhost:3000 (not the real domain)
  - Protocol: http (even if the user used HTTPS)
```

**With these headers:**
```
Backend sees:
  - Client IP: 203.0.113.50 (the real user's IP)
  - Host: myapp.com (the real domain)
  - Protocol: https (the real protocol)
```

## 🔄 Proxying Multiple Services (Path-Based Routing)

A very common pattern — one domain routing to **different backend services** based on the URL path:

```nginx
server {
    listen 80;
    server_name myapp.com;

    # Main website (React/Vue SPA)
    location / {
        proxy_pass http://localhost:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # API service (Node.js/Python/Go)
    location /api/ {
        proxy_pass http://localhost:4000/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # Admin dashboard (separate app)
    location /admin/ {
        proxy_pass http://localhost:5000/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # Static files served directly by Nginx (no proxy needed)
    location /static/ {
        alias /var/www/myapp/static/;
        expires 30d;
    }
}
```

How the routing works:

```
myapp.com/                 → Frontend app (port 3000)
myapp.com/api/users        → API service (port 4000) → receives /users
myapp.com/admin/dashboard  → Admin app (port 5000) → receives /dashboard
myapp.com/static/style.css → Served directly by Nginx from disk
```

This is basically how **microservices** are exposed through a single domain!

## ⚖️ Load Balancing with Nginx

### What is Load Balancing?

**Load balancing** is distributing incoming traffic across **multiple backend servers** so no single server gets overwhelmed.

```
                                    ┌─────────────────┐
                               ┌───►│  Backend 1      │
                               │    │  (port 3001)    │
┌────────┐    ┌──────────┐     │    └─────────────────┘
│ Client  ├───►│  Nginx   ├─────┤
│         │    │  (LB)    │     │    ┌─────────────────┐
└────────┘    └──────────┘     ├───►│  Backend 2      │
                               │    │  (port 3002)    │
                               │    └─────────────────┘
                               │
                               │    ┌─────────────────┐
                               └───►│  Backend 3      │
                                    │  (port 3003)    │
                                    └─────────────────┘
```

### Why Load Balance?

| Benefit | Explanation |
|---------|-------------|
| **High Availability** | If one server dies, traffic goes to the others |
| **Scalability** | Add more servers to handle more traffic |
| **Performance** | Spread the load so no single server is overworked |
| **Zero Downtime Deploys** | Update one server at a time while others handle traffic |

---

### Setting Up Load Balancing

#### Step 1: Define an Upstream Block

The `upstream` block defines a **group of backend servers**:

```nginx
# Define the group of backend servers
upstream my_backend {
    server 127.0.0.1:3001;    # Backend server 1
    server 127.0.0.1:3002;    # Backend server 2
    server 127.0.0.1:3003;    # Backend server 3
}

server {
    listen 80;
    server_name myapp.com;

    location / {
        # Forward requests to the upstream group
        proxy_pass http://my_backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

That's it! Nginx will now distribute requests across all three backends.

---

### Load Balancing Algorithms

Nginx supports several algorithms to decide **which backend** gets the next request:

#### 1. Round Robin (Default)

Requests are distributed **evenly in order**: Server 1 → Server 2 → Server 3 → Server 1 → ...

```nginx
upstream my_backend {
    # Round Robin is the default — no directive needed
    server 127.0.0.1:3001;
    server 127.0.0.1:3002;
    server 127.0.0.1:3003;
}
```

```
Request 1 → Server 1
Request 2 → Server 2
Request 3 → Server 3
Request 4 → Server 1
Request 5 → Server 2
...
```

**Best for:** Servers with similar hardware and load.

#### 2. Least Connections

Each request goes to the server with the **fewest active connections**:

```nginx
upstream my_backend {
    least_conn;                    # ← Use least connections algorithm
    server 127.0.0.1:3001;
    server 127.0.0.1:3002;
    server 127.0.0.1:3003;
}
```

```
Server 1: 5 active connections
Server 2: 2 active connections   ← Next request goes here
Server 3: 8 active connections
```

**Best for:** When requests have **varying processing times** (some are quick, some are slow).

#### 3. IP Hash

Requests from the **same client IP** always go to the **same backend server**:

```nginx
upstream my_backend {
    ip_hash;                       # ← Use IP hash algorithm
    server 127.0.0.1:3001;
    server 127.0.0.1:3002;
    server 127.0.0.1:3003;
}
```

```
Client 203.0.113.50 → Always goes to Server 2
Client 198.51.100.10 → Always goes to Server 1
Client 192.0.2.75 → Always goes to Server 3
```

**Best for:** Applications that need **session persistence** (sticky sessions) — the user always hits the same backend so their session data is there.

#### 4. Weighted Round Robin

Give **more powerful servers** a higher weight so they get **more requests**:

```nginx
upstream my_backend {
    server 127.0.0.1:3001 weight=5;   # Gets 5x more requests
    server 127.0.0.1:3002 weight=3;   # Gets 3x more requests
    server 127.0.0.1:3003 weight=1;   # Gets 1x requests (baseline)
}
```

```
Out of every 9 requests:
  Server 1 gets: 5 requests (big powerful server)
  Server 2 gets: 3 requests (medium server)
  Server 3 gets: 1 request  (small server)
```

**Best for:** When backend servers have **different hardware** (some are bigger than others).

#### 5. Random with Two Choices

Randomly picks **two servers** and sends the request to the one with **fewer connections**:

```nginx
upstream my_backend {
    random two least_conn;
    server 127.0.0.1:3001;
    server 127.0.0.1:3002;
    server 127.0.0.1:3003;
}
```

**Best for:** Large-scale deployments where full state tracking is expensive.

### Health Checks — Don't Send Traffic to Dead Servers

What if one of your backend servers crashes? You don't want Nginx to keep sending requests to a dead server.

#### Passive Health Checks (Built-in, Free)

Nginx automatically detects when a backend fails:

```nginx
upstream my_backend {
    server 127.0.0.1:3001 max_fails=3 fail_timeout=30s;
    server 127.0.0.1:3002 max_fails=3 fail_timeout=30s;
    server 127.0.0.1:3003 max_fails=3 fail_timeout=30s;
}
```

What this means:
- `max_fails=3` — If a server fails **3 times**...
- `fail_timeout=30s` — ...mark it as **down for 30 seconds**. After 30 seconds, try again.

```
Server 1: Responds normally ✅
Server 2: Failed 3 times → marked as DOWN for 30 seconds ❌
Server 3: Responds normally ✅

→ Traffic only goes to Server 1 and Server 3 for the next 30 seconds.
→ After 30 seconds, Nginx tries Server 2 again.
```

#### Backup Servers

You can mark a server as a **backup** — it only receives traffic when **all primary servers are down**:

```nginx
upstream my_backend {
    server 127.0.0.1:3001;                # Primary
    server 127.0.0.1:3002;                # Primary
    server 127.0.0.1:3003 backup;         # Backup — only used when 3001 & 3002 are both down
}
```

#### Mark a Server as Down (For Maintenance)

```nginx
upstream my_backend {
    server 127.0.0.1:3001;
    server 127.0.0.1:3002 down;           # Temporarily removed from rotation
    server 127.0.0.1:3003;
}
```

## 🔒 SSL/TLS Termination

### What is SSL Termination?

**SSL termination** means Nginx handles the **HTTPS encryption/decryption** so your backend apps don't have to.

```
Client ──── HTTPS (encrypted) ────► Nginx ──── HTTP (plain) ────► Backend App
                                      │
                              SSL terminates here
                              Nginx decrypts the
                              request and forwards
                              it as plain HTTP
```

### Why Terminate SSL at Nginx?

| Reason | Explanation |
|--------|-------------|
| **Simplicity** | Backend apps don't need to deal with certificates |
| **Performance** | Nginx is optimized for SSL. Your app CPU stays focused on business logic |
| **Central Management** | One place to manage all SSL certificates |
| **Easy Renewals** | Renew certs once at Nginx, not on every backend server |

---

### Setting Up SSL with Let's Encrypt (Free Certificates!)

**Let's Encrypt** provides free SSL certificates. **Certbot** is the tool that automates the process.

#### Step 1: Install Certbot

```bash
# Ubuntu/Debian

# CentOS/RHEL
sudo yum install certbot python3-certbot-nginx -y
```

#### Step 2: Get a Certificate

```bash

```

Certbot will:
1. Verify you own the domain (by placing a challenge file).
2. Obtain the SSL certificate.
3. Automatically modify your Nginx config to use HTTPS.
4. Set up HTTP → HTTPS redirect.

#### Step 3: Auto-Renewal

Let's Encrypt certificates expire every 90 days. Certbot sets up auto-renewal:

```bash

# Certbot auto-renewal runs via a systemd timer or cron job
sudo systemctl status certbot.timer
```

### Manual SSL Configuration

If you already have SSL certificates (from a certificate authority or self-signed), here's how to configure them manually:

```nginx
server {
    listen 80;
    server_name myapp.com www.myapp.com;
    
    # Redirect all HTTP to HTTPS
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl http2;
    listen [::]:443 ssl http2;
    server_name myapp.com www.myapp.com;

    # ─── SSL Certificate Files ───
    ssl_certificate /etc/letsencrypt/live/myapp.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/myapp.com/privkey.pem;

    # ─── SSL Settings (Modern & Secure) ───
    ssl_protocols TLSv1.2 TLSv1.3;                    # Only modern TLS versions
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384;
    ssl_prefer_server_ciphers off;                      # Let the client choose the cipher

    # ─── SSL Performance ───
    ssl_session_cache shared:SSL:10m;                   # Cache SSL sessions (10MB ≈ 40,000 sessions)
    ssl_session_timeout 1d;                             # Sessions valid for 1 day
    ssl_session_tickets off;                            # Disable session tickets (better security)

    # ─── HSTS (HTTP Strict Transport Security) ───
    add_header Strict-Transport-Security "max-age=63072000; includeSubDomains; preload" always;
    # This tells browsers: "ALWAYS use HTTPS for this site for the next 2 years"

    # ─── OCSP Stapling (Certificate Validation Optimization) ───
    ssl_stapling on;
    ssl_stapling_verify on;
    resolver 8.8.8.8 8.8.4.4 valid=300s;
    resolver_timeout 5s;

    # ─── Reverse Proxy to Backend ───
    location / {
        proxy_pass http://localhost:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

Let's break down the important SSL directives:

| Directive | Purpose |
|-----------|---------|
| `ssl_certificate` | Path to the full certificate chain (your cert + intermediate certs) |
| `ssl_certificate_key` | Path to the private key |
| `ssl_protocols TLSv1.2 TLSv1.3` | Only allow TLS 1.2 and 1.3 (disable old, insecure versions) |
| `ssl_ciphers` | Which encryption algorithms to use (strong ones only) |
| `ssl_session_cache` | Cache SSL handshakes for faster reconnections |
| `ssl_stapling` | Speed up certificate verification |
| `Strict-Transport-Security` | Force browsers to always use HTTPS |

```bash
# Generate a self-signed certificate (NOT for production!)
```

Then use it in your config:

```nginx

```

> **Warning:** Browsers will show a security warning for self-signed certs. This is expected. Only use self-signed certs for development/testing.

## 🌐 WebSocket Proxying

Modern apps often use **WebSockets** for real-time communication (chat, live updates, notifications). Nginx needs special config to handle WebSocket connections:

```nginx
server {
    listen 80;
    server_name myapp.com;

    # Regular HTTP requests
    location / {
        proxy_pass http://localhost:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # WebSocket connections
    location /ws/ {
        proxy_pass http://localhost:3000;
        proxy_http_version 1.1;                  # WebSocket requires HTTP/1.1
        proxy_set_header Upgrade $http_upgrade;   # Required for WebSocket
        proxy_set_header Connection "upgrade";    # Required for WebSocket
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;

        # WebSocket-specific timeouts
        proxy_read_timeout 86400s;                # Keep connection alive for 24 hours
        proxy_send_timeout 86400s;
    }
}
```

The key lines for WebSocket support:
```nginx
proxy_http_version 1.1;                  # Must be HTTP/1.1
proxy_set_header Upgrade $http_upgrade;   # "Upgrade" the connection to WebSocket
proxy_set_header Connection "upgrade";    # Confirm the upgrade
```

## ⏱️ Timeouts & Buffering — Fine-Tuning the Proxy

### Timeout Settings

```nginx
location / {
    proxy_pass http://localhost:3000;

    # How long to wait for the backend to CONNECT
    proxy_connect_timeout 5s;         # Default: 60s. If backend doesn't connect in 5s, fail.

    # How long to wait for the backend to SEND data
    proxy_send_timeout 10s;           # Default: 60s. Time to transmit the request to backend.

    # How long to wait for the backend to RESPOND
    proxy_read_timeout 30s;           # Default: 60s. Time to wait for backend's response.
                                      # Increase for slow APIs or file uploads.
}
```

### Buffering Settings

By default, Nginx **buffers** the backend's response before sending it to the client. This is usually good — but sometimes you want to disable it:

```nginx
location / {
    proxy_pass http://localhost:3000;

    # ── Buffering ON (default, good for most cases) ──
    proxy_buffering on;
    proxy_buffer_size 4k;              # Buffer for response headers
    proxy_buffers 8 4k;                # 8 buffers of 4KB each for response body
    proxy_busy_buffers_size 8k;        # Max size of buffers being sent to client

    # ── Buffering OFF (for streaming, Server-Sent Events, etc.) ──
    # proxy_buffering off;             # Send data to client immediately
}
```

**When to disable buffering:**
- Server-Sent Events (SSE)
- Long-polling
- Streaming responses
- File downloads (large files)

## 📊 Complete Production Example — The Full Package

Here's a **real-world, production-grade** Nginx config that combines everything we've learned:

```nginx
# ─── Upstream: Define Backend Servers ───
upstream app_servers {
    least_conn;                                   # Use least connections algorithm
    server 10.0.1.10:3000 weight=3 max_fails=3 fail_timeout=30s;   # Primary (bigger server)
    server 10.0.1.11:3000 weight=2 max_fails=3 fail_timeout=30s;   # Secondary
    server 10.0.1.12:3000 weight=1 max_fails=3 fail_timeout=30s;   # Tertiary
    server 10.0.1.20:3000 backup;                 # Backup server (used only when others are down)
}

# ─── HTTP → HTTPS Redirect ───
server {
    listen 80;
    listen [::]:80;
    server_name myapp.com www.myapp.com;
    return 301 https://$host$request_uri;
}

# ─── Main HTTPS Server ───
server {
    listen 443 ssl http2;
    listen [::]:443 ssl http2;
    server_name myapp.com www.myapp.com;

    # ── SSL Configuration ──
    ssl_certificate /etc/letsencrypt/live/myapp.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/myapp.com/privkey.pem;
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384;
    ssl_prefer_server_ciphers off;
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 1d;
    ssl_session_tickets off;

    # OCSP Stapling
    ssl_stapling on;
    ssl_stapling_verify on;
    resolver 8.8.8.8 8.8.4.4 valid=300s;

    # ── Security Headers ──
    add_header Strict-Transport-Security "max-age=63072000; includeSubDomains; preload" always;
    add_header X-Frame-Options "SAMEORIGIN" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-XSS-Protection "1; mode=block" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;
    server_tokens off;

    # ── Gzip Compression ──
    gzip on;
    gzip_types text/plain text/css application/json application/javascript text/xml application/xml;
    gzip_min_length 1000;
    gzip_comp_level 5;

    # ── Client Body Size (for file uploads) ──
    client_max_body_size 50M;          # Allow uploads up to 50MB

    # ── Static Files (served directly by Nginx) ──
    location /static/ {
        alias /var/www/myapp/static/;
        expires 30d;
        add_header Cache-Control "public, immutable";
        access_log off;                # Don't log static file requests
    }

    # ── API Requests → Backend ──
    location /api/ {
        proxy_pass http://app_servers;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_connect_timeout 5s;
        proxy_read_timeout 30s;
    }

    # ── WebSocket Support ──
    location /ws/ {
        proxy_pass http://app_servers;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_read_timeout 86400s;
    }

    # ── Frontend App (SPA) ──
    location / {
        proxy_pass http://app_servers;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # ── Deny Access to Hidden Files ──
    location ~ /\. {
        deny all;
        access_log off;
        log_not_found off;
    }

    # ── Health Check Endpoint ──
    location /health {
        access_log off;
        return 200 "OK";
        add_header Content-Type text/plain;
    }

    # ── Logs ──
    access_log /var/log/nginx/myapp.access.log;
    error_log /var/log/nginx/myapp.error.log;
}
```

## ❓ Common Interview Questions

1. **What is a reverse proxy and why do we use it?**
   → A reverse proxy sits between clients and backend servers. It provides security (hides backends), load balancing, SSL termination, caching, and a single entry point for multiple services.

2. **What's the difference between a forward proxy and a reverse proxy?**
   → A forward proxy sits in front of clients (hides the client's identity). A reverse proxy sits in front of servers (hides the server's identity).

3. **How does Nginx load balancing work?**
   → You define an `upstream` block with a list of backend servers. Nginx distributes requests using algorithms like round robin, least connections, or IP hash.

4. **What is SSL termination?**
   → SSL termination is when the reverse proxy (Nginx) handles the HTTPS encryption/decryption, and forwards plain HTTP to the backend. This simplifies the backend and centralizes certificate management.

5. **What happens if a backend server goes down?**
   → Nginx's passive health checks detect the failure (based on `max_fails` and `fail_timeout`). The server is temporarily removed from the pool. Nginx retries after the timeout period.

6. **Why do you need proxy headers like `X-Real-IP` and `X-Forwarded-For`?**
   → Without them, the backend sees all requests as coming from the Nginx server (127.0.0.1) instead of the real client. These headers pass the original client information to the backend.

7. **What's the difference between `proxy_pass http://backend` and `proxy_pass http://backend/`?**
   → Without trailing slash, the full request URI is appended. With trailing slash, the location prefix is stripped and the remaining path is forwarded.

8. **How do you achieve zero-downtime deployments with Nginx?**
   → Use load balancing. Update one backend at a time (rolling deploy). Mark the server being updated as `down`, deploy new code, bring it back, then move to the next server.
