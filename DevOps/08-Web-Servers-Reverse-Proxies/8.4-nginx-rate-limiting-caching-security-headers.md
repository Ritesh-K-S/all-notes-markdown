## 1. Why Do We Need Rate Limiting, Caching & Security Headers?

Imagine you've got your Nginx server running beautifully, serving your app to users. But then...

- **Problem 1:** A bot hammers your login page with 10,000 requests per second → Your server crashes 💥
- **Problem 2:** Every user request hits your backend (Node.js, Python, etc.) even for the same page → Slow response times 🐢
- **Problem 3:** A hacker injects a malicious script into your page → Your users' data gets stolen 🕵️

These three features solve exactly these three problems:

| Feature | Problem It Solves | Simple Analogy |
|---------|------------------|----------------|
| **Rate Limiting** | Too many requests from one source | A bouncer at a club — "Only 10 people per minute can enter" |
| **Caching** | Backend doing the same work repeatedly | A photocopy machine — "Why re-write the same letter? Just photocopy it!" |
| **Security Headers** | Browser-based attacks (XSS, clickjacking, etc.) | A rulebook for the browser — "Here's what you're allowed to do and NOT do" |

## 2. Part 1 — Rate Limiting

### What is Rate Limiting?

Rate limiting is like putting a speed limit on a highway. You're saying:

> "Hey, each user (or IP address) can only make X number of requests in Y amount of time."

If they go over the limit, Nginx will either:
- **Reject** the extra requests (return `503` error)
- **Queue** them (make them wait)
- **Delay** them (slow them down)

### Real-World Use Cases

- Protect login pages from brute-force attacks
- Prevent API abuse
- Stop web scraping bots
- Protect against DDoS (to some extent)
- Prevent resource exhaustion

---

### How Nginx Rate Limiting Works (Leaky Bucket Algorithm)

Nginx uses the **Leaky Bucket Algorithm**. Here's how it works in plain English:

Think of a bucket with a small hole at the bottom:

```
        ┌──────────────────┐
        │  Incoming        │  ← Requests pour in from the top
        │  Requests        │
        │  ████████████    │
        │  ████████████    │  ← Bucket fills up
        │  ████████████    │
        └──────┬───────────┘
               │
               ▼              ← Requests leak out at a fixed rate
         (processed)
```

- **Requests come in** (water pours into the bucket)
- **Requests are processed at a fixed rate** (water leaks out at a steady pace)
- **If the bucket is full**, new requests are rejected (water overflows)
- The `burst` parameter = how big the bucket is (how many extra requests can wait)

---

### Basic Rate Limiting Setup

Rate limiting in Nginx uses two directives:

1. **`limit_req_zone`** — Defines the zone (placed in `http` block)
2. **`limit_req`** — Applies the zone to a location (placed in `server` or `location` block)

```nginx
# Step 1: Define the rate limit zone (in the http block)
http {
    # Syntax: limit_req_zone <key> zone=<name>:<size> rate=<rate>;
    limit_req_zone $binary_remote_addr zone=mylimit:10m rate=10r/s;

    server {
        listen 80;
        server_name example.com;

        location / {
            # Step 2: Apply the rate limit
            limit_req zone=mylimit;

            proxy_pass http://backend;
        }
    }
}
```

**Let's break this down piece by piece:**

| Part | Meaning |
|------|---------|
| `$binary_remote_addr` | The key — each unique IP gets its own counter. Uses binary format (saves memory — 4 bytes for IPv4 vs ~15 bytes for text) |
| `zone=mylimit:10m` | Creates a shared memory zone called "mylimit" with 10 megabytes of storage. 10MB can store ~160,000 IP addresses |
| `rate=10r/s` | Allow 10 requests per second per IP |
| `limit_req zone=mylimit` | Apply this rate limit to the location |

**What happens when someone exceeds the limit?**
- Nginx returns **503 Service Temporarily Unavailable**
- The excess requests are immediately rejected (no queuing)

> **Rate Format:** You can use `r/s` (requests per second) or `r/m` (requests per minute).
> - `rate=1r/s` = 1 request per second
> - `rate=30r/m` = 30 requests per minute (= 1 every 2 seconds)

---

### Burst — Allowing Short Spikes

The basic rate limit is very strict. If you set `rate=10r/s`, Nginx actually spaces requests evenly — one every 100ms. If two requests come within 50ms, the second one is rejected!

That's often too strict. Real users don't send requests at perfectly even intervals. That's where **burst** comes in:

```nginx
location /api/ {
    limit_req zone=mylimit burst=20;
    proxy_pass http://backend;
}
```

**What does `burst=20` do?**

- It creates a **queue** that can hold up to 20 extra requests
- If requests come in faster than the rate, they're queued (not rejected)
- They're processed one at a time at the defined rate
- If the queue is full (more than 20 waiting), then excess requests ARE rejected

**Think of it like this:**
- Without burst: "Only 1 person per 100ms. Everyone else is turned away."
- With burst=20: "Only 1 person per 100ms, but up to 20 people can wait in line."

**The problem:** Those 20 queued requests are delayed. The 20th request in the burst might wait 2 seconds before being processed. That feels slow to the user.

---

### The `nodelay` Option

To fix the delay problem, add `nodelay`:

```nginx
location /api/ {
    limit_req zone=mylimit burst=20 nodelay;
    proxy_pass http://backend;
}
```

**What changes with `nodelay`?**

- The 20 burst requests are processed **immediately** (no waiting in line)
- But they still "use up" burst slots
- The burst slots refill at the defined rate (10 per second)
- Once all burst slots are used AND new requests come in, they're rejected

**Think of it like this:**
- Without `nodelay`: "20 people can wait in line, but they must enter one at a time."
- With `nodelay`: "20 people can rush in all at once, but then the door closes until slots free up."

**There's also `delay=`:**

```nginx
location /api/ {
    limit_req zone=mylimit burst=20 delay=8;
    proxy_pass http://backend;
}
```

This means:
- First 8 requests in the burst → processed immediately (no delay)
- Requests 9-20 → queued and processed at the steady rate
- Beyond 20 → rejected

This is the **sweet spot** for most production setups — it gives users a fast initial experience while still controlling the flow.

---

### Rate Limiting by Different Keys

You're not limited to rate limiting by IP address. You can use any Nginx variable:

```nginx
# Rate limit by IP (most common)
limit_req_zone $binary_remote_addr zone=per_ip:10m rate=10r/s;

# Rate limit by server name (per virtual host)
limit_req_zone $server_name zone=per_server:10m rate=100r/s;

# Rate limit by URI (per page)
limit_req_zone $uri zone=per_uri:10m rate=50r/s;

# Rate limit by a custom header (e.g., API key)
limit_req_zone $http_x_api_key zone=per_api_key:10m rate=20r/s;

# Rate limit by combination (IP + URI)
limit_req_zone $binary_remote_addr$uri zone=per_ip_uri:10m rate=5r/s;
```

**When to use which:**

| Key | Use Case |
|-----|----------|
| `$binary_remote_addr` | General protection — limit per user/IP |
| `$server_name` | Overall server protection |
| `$uri` | Protect specific heavy endpoints |
| `$http_x_api_key` | API rate limiting per customer |
| Combined | Very specific protection |

---

### Multiple Rate Limiting Zones

You can apply **multiple rate limits** at the same time:

```nginx
http {
    # Global rate limit: 100 requests/second per IP
    limit_req_zone $binary_remote_addr zone=global:10m rate=100r/s;
    
    # Login rate limit: 5 requests/minute per IP (strict!)
    limit_req_zone $binary_remote_addr zone=login:10m rate=5r/m;

    server {
        # Apply global rate limit to everything
        limit_req zone=global burst=50 nodelay;

        location /login {
            # Apply BOTH global AND login rate limits
            limit_req zone=global burst=50 nodelay;
            limit_req zone=login burst=3 nodelay;
            proxy_pass http://backend;
        }

        location / {
            proxy_pass http://backend;
        }
    }
}
```

When multiple zones are applied, a request must pass **ALL** of them. If any one zone rejects the request, it's rejected.

---

### Whitelisting IPs from Rate Limiting

Sometimes you want to exclude certain IPs (like your own office, monitoring systems, or trusted partners):

```nginx
http {
    # Create a map: if the IP is trusted, set $limit to empty string
    # If not trusted, set $limit to the client's IP
    geo $limit {
        default         $binary_remote_addr;
        10.0.0.0/8      "";       # Internal network — no rate limit
        192.168.1.0/24   "";      # Office network — no rate limit
        203.0.113.50     "";      # Monitoring server — no rate limit
    }

    # Use $limit as the key — empty string means "don't track"
    limit_req_zone $limit zone=mylimit:10m rate=10r/s;

    server {
        location / {
            limit_req zone=mylimit burst=20 nodelay;
            proxy_pass http://backend;
        }
    }
}
```

**How this works:**
- For whitelisted IPs, `$limit` is an empty string `""`
- Nginx groups all empty-string keys together into one counter — so they effectively share one bucket (which never fills up because there are so few)
- For everyone else, `$limit` is their IP address, so they get individual rate limits

### Custom Error Page for Rate Limited Requests

By default, rate-limited users get a generic `503` error. Let's make it friendlier:

```nginx
http {
    limit_req_zone $binary_remote_addr zone=mylimit:10m rate=10r/s;

    # Change the error code for rate limiting from 503 to 429
    limit_req_status 429;

    server {
        # Custom error page for 429
        error_page 429 /rate_limit.html;

        location = /rate_limit.html {
            root /usr/share/nginx/html;
            internal;  # Only accessible via internal redirect
        }

        location /api/ {
            limit_req zone=mylimit burst=10 nodelay;
            proxy_pass http://backend;
        }
    }
}
```

**Why 429 instead of 503?**
- `429 Too Many Requests` is the proper HTTP status code for rate limiting
- `503 Service Temporarily Unavailable` is more generic (could mean server is down)
- Using `429` helps clients understand they need to slow down

---

### Connection Limiting vs Rate Limiting

Nginx also has **connection limiting** — which limits how many *simultaneous connections* an IP can have (different from rate limiting which counts requests over time):

```nginx
http {
    # Limit concurrent connections per IP
    limit_conn_zone $binary_remote_addr zone=conn_limit:10m;

    server {
        # Each IP can have max 10 simultaneous connections
        limit_conn conn_limit 10;

        # Rate limit (requests over time)
        limit_req zone=mylimit burst=20 nodelay;

        location / {
            proxy_pass http://backend;
        }
    }
}
```

| Feature | Rate Limiting (`limit_req`) | Connection Limiting (`limit_conn`) |
|---------|---------------------------|-----------------------------------|
| What it limits | Requests per time period | Simultaneous connections |
| Use case | API protection, login brute-force | Download throttling, resource protection |
| Analogy | "You can enter 10 times per minute" | "Only 5 of you can be inside at once" |

## 3. Part 2 — Caching

### What is Caching?

Caching means **storing a copy of a response** so you don't have to generate it again next time someone asks for the same thing.

**Without caching:**
```
User A → Nginx → Backend → Database → Backend → Nginx → User A  (500ms)
User B → Nginx → Backend → Database → Backend → Nginx → User B  (500ms)
User C → Nginx → Backend → Database → Backend → Nginx → User C  (500ms)
                                                          Total: 1500ms
```

**With caching:**
```
User A → Nginx → Backend → Database → Backend → Nginx → User A  (500ms)
                                                  ↓
                                           [Cache stored]

User B → Nginx → [Cache hit!] → User B                          (5ms)
User C → Nginx → [Cache hit!] → User C                          (5ms)
                                                          Total: 510ms
```

That's a **66% reduction** in total backend load and nearly **100x faster** for cached responses!

---

### Why Use Nginx for Caching?

You might wonder: "Why cache at the Nginx level? Can't the backend do caching too?"

Yes, backends can cache too (Redis, Memcached, etc.). But Nginx caching has special advantages:

| Advantage | Explanation |
|-----------|-------------|
| **Sits in front** | Catches requests before they even reach the backend |
| **Incredibly fast** | Serves from disk/memory — no application code runs |
| **Protects backend** | Backend might be down, but Nginx can still serve stale cache |
| **Language-agnostic** | Works no matter what your backend is (Node, Python, Go, PHP...) |
| **Simple to set up** | Just a few lines in the config |

---

### Types of Caching in Nginx

| Type | What It Caches | Use Case |
|------|---------------|----------|
| **Proxy Cache** | Responses from upstream (backend) servers | APIs, dynamic pages |
| **FastCGI Cache** | Responses from FastCGI (PHP-FPM, etc.) | WordPress, Laravel |
| **Static File Cache** | Static assets (CSS, JS, images) | All websites |
| **Browser Cache** | Instructs browsers to cache locally | All websites |

### Proxy Cache Setup (Full Guide)

This is the most common and powerful type of caching. Let's build it step by step:

```nginx
http {
    # Step 1: Define the cache path
    proxy_cache_path /var/cache/nginx    # Where to store cached files on disk
        levels=1:2                       # Directory structure (e.g., /var/cache/nginx/a/1b/...)
        keys_zone=my_cache:10m           # Shared memory zone: name=my_cache, size=10MB (stores cache keys)
        max_size=1g                      # Maximum disk space for cached files
        inactive=60m                     # Remove cached items not accessed for 60 minutes
        use_temp_path=off;               # Write directly to cache dir (faster)

    server {
        listen 80;
        server_name example.com;

        location / {
            # Step 2: Enable caching
            proxy_cache my_cache;

            # Step 3: Define what makes a unique cache entry
            proxy_cache_key "$scheme$request_method$host$request_uri";

            # Step 4: When to cache (cache 200/301/302 responses for 10 minutes)
            proxy_cache_valid 200 301 302 10m;
            proxy_cache_valid 404 1m;        # Cache 404s for 1 minute
            proxy_cache_valid any 5m;        # Cache everything else for 5 minutes

            # Step 5: Minimum number of requests before caching
            proxy_cache_min_uses 2;          # Only cache after 2 identical requests

            # Step 6: Add a header to see if response was cached
            add_header X-Cache-Status $upstream_cache_status;

            proxy_pass http://backend;
        }
    }
}
```

**Let's break down every part:**

**`proxy_cache_path` parameters:**

| Parameter | What It Does | Example Value |
|-----------|-------------|---------------|
| Path | Physical directory on disk | `/var/cache/nginx` |
| `levels=1:2` | Creates subdirectory structure. `1:2` means first 1 char, then 2 chars of the hash. Prevents too many files in one directory | `/a/1b/abc123...` |
| `keys_zone=name:size` | Shared memory zone for cache keys. 1MB ≈ 8,000 keys | `my_cache:10m` |
| `max_size` | Maximum total disk usage for cache | `1g` (1 gigabyte) |
| `inactive` | Remove items not accessed for this long | `60m` (60 minutes) |
| `use_temp_path=off` | Write directly to cache (no temp file copy) | Recommended for performance |

**`X-Cache-Status` header values:**

| Value | Meaning |
|-------|---------|
| `MISS` | Not in cache — fetched from backend |
| `HIT` | Served from cache — backend not contacted |
| `EXPIRED` | Was in cache but expired — fetched from backend again |
| `STALE` | Expired but served anyway (because backend is down) |
| `BYPASS` | Cache was intentionally skipped |
| `REVALIDATED` | Cached entry revalidated with backend |
| `UPDATING` | Stale content served while cache is being updated |

### Cache Key Customization

The **cache key** determines what makes a unique cached response. The default key is:

```nginx
proxy_cache_key "$scheme$request_method$host$request_uri";
```

This means `http://example.com/page1` and `https://example.com/page1` are cached separately.

**Custom examples:**

```nginx
# Include query parameters (default behavior via $request_uri)
# /api/users?page=1 and /api/users?page=2 are cached separately
proxy_cache_key "$scheme$request_method$host$request_uri";

# Ignore query parameters (cache /api/data?v=1 and /api/data?v=2 as the same)
proxy_cache_key "$scheme$request_method$host$uri";

# Include a specific header (e.g., cache per language)
proxy_cache_key "$scheme$request_method$host$request_uri$http_accept_language";

# Include cookies (careful — can kill cache hit rate)
proxy_cache_key "$scheme$host$request_uri$cookie_session";
```

> ⚠️ **Warning:** The more variables in the cache key, the fewer cache hits you'll get. Keep it as simple as possible.

### Cache Bypass & Cache Purge

Sometimes you need to **skip the cache** or **force-refresh** it:

**Cache Bypass (skip cache for certain requests):**

```nginx
location / {
    proxy_cache my_cache;

    # Don't use cache if user sends a Cache-Control: no-cache header
    proxy_cache_bypass $http_cache_control;

    # Don't use cache for requests with a "nocache" cookie
    proxy_cache_bypass $cookie_nocache;

    # Don't cache responses that set cookies (usually means dynamic/user-specific)
    proxy_no_cache $http_set_cookie;

    # Custom bypass: skip cache for logged-in users
    set $bypass 0;
    if ($http_cookie ~* "session_id") {
        set $bypass 1;
    }
    proxy_cache_bypass $bypass;
    proxy_no_cache $bypass;

    proxy_pass http://backend;
}
```

| Directive | What It Does |
|-----------|-------------|
| `proxy_cache_bypass` | Skip the cache and go to backend, but still cache the response |
| `proxy_no_cache` | Don't store the response in cache |

**Cache Purge (delete specific cached items):**

Nginx open-source doesn't have a built-in purge module. But you can work around it:

```nginx
# Method 1: Map-based purge (simple approach)
map $request_method $purge_method {
    PURGE   1;
    default 0;
}

server {
    location / {
        proxy_cache my_cache;
        proxy_cache_bypass $purge_method;
        proxy_no_cache $purge_method;
        proxy_pass http://backend;
    }
}

# Then send: curl -X PURGE http://example.com/cached-page
# This bypasses cache for that URL and refreshes it
```

```bash
# Method 2: Delete cache files directly
# Find cache files:
grep -r "KEY: GETexample.com/page1" /var/cache/nginx/
# Then delete the matching file

# Method 3: Nuclear option — clear all cache
rm -rf /var/cache/nginx/*
# Don't forget to reload: nginx -s reload
```

### Stale Cache — Serving Old Content When Backend is Down

This is one of the **most powerful features** of Nginx caching. When your backend crashes, Nginx can still serve the last known good response:

```nginx
location / {
    proxy_cache my_cache;
    proxy_cache_valid 200 10m;

    # Serve stale (expired) cache when:
    proxy_cache_use_stale error          # Backend returned an error
                          timeout        # Backend timed out
                          invalid_header # Backend sent garbage headers
                          updating       # Cache is being refreshed
                          http_500       # Backend returned 500
                          http_502       # Backend returned 502
                          http_503       # Backend returned 503
                          http_504;      # Backend returned 504

    # How long stale cache can be served
    proxy_cache_background_update on;    # Update cache in background while serving stale

    # Lock: only one request updates the cache at a time
    proxy_cache_lock on;                 # Prevents "cache stampede"
    proxy_cache_lock_timeout 5s;         # If lock held too long, pass through

    proxy_pass http://backend;
}
```

**Why is this amazing?**

Without stale cache:
```
Backend crashes → All users see 502 Bad Gateway → Panic! 🔥
```

With stale cache:
```
Backend crashes → Users see the last cached version → No one notices → You have time to fix it calmly ☕
```

### FastCGI Cache (for PHP/Python apps)

If you're running PHP-FPM (WordPress, Laravel) or any FastCGI application:

```nginx
http {
    # Define FastCGI cache
    fastcgi_cache_path /var/cache/nginx/fastcgi
        levels=1:2
        keys_zone=fcgi_cache:10m
        max_size=512m
        inactive=30m;

    server {
        location ~ \.php$ {
            fastcgi_cache fcgi_cache;
            fastcgi_cache_key "$scheme$request_method$host$request_uri";
            fastcgi_cache_valid 200 10m;
            fastcgi_cache_valid 301 302 5m;
            fastcgi_cache_valid 404 1m;

            # Don't cache for logged-in users (WordPress example)
            fastcgi_cache_bypass $cookie_wordpress_logged_in;
            fastcgi_no_cache $cookie_wordpress_logged_in;

            add_header X-FastCGI-Cache $upstream_cache_status;

            fastcgi_pass unix:/run/php/php8.2-fpm.sock;
            include fastcgi_params;
        }
    }
}
```

### Static File Caching with `expires`

For static assets (CSS, JS, images, fonts), you want the **browser** to cache them:

```nginx
server {
    # Images — cache for 30 days
    location ~* \.(jpg|jpeg|png|gif|ico|svg|webp)$ {
        expires 30d;
        add_header Cache-Control "public, immutable";
        access_log off;  # Don't log static file access
    }

    # CSS and JavaScript — cache for 7 days
    location ~* \.(css|js)$ {
        expires 7d;
        add_header Cache-Control "public";
    }

    # Fonts — cache for 1 year
    location ~* \.(woff|woff2|ttf|eot|otf)$ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }

    # HTML — don't cache (always fresh)
    location ~* \.html$ {
        expires -1;  # -1 means no cache
        add_header Cache-Control "no-store, no-cache, must-revalidate";
    }
}
```

**What `expires` does:**

| Value | Meaning | Cache-Control Header Sent |
|-------|---------|--------------------------|
| `30d` | 30 days | `max-age=2592000` |
| `7d` | 7 days | `max-age=604800` |
| `1y` | 1 year | `max-age=31536000` |
| `1h` | 1 hour | `max-age=3600` |
| `-1` | Don't cache | `no-cache` |
| `epoch` | Already expired | `max-age=0` |
| `max` | Maximum (10 years) | `max-age=315360000` |

> 💡 **Tip:** Use `immutable` for files that never change (like versioned assets: `style.v3.css`). This tells the browser to never even check for updates.

### Micro-Caching — The 1-Second Cache Trick

This is a brilliant trick for high-traffic dynamic sites. Cache everything for just **1 second**:

```nginx
http {
    proxy_cache_path /var/cache/nginx/micro
        keys_zone=micro_cache:5m
        max_size=500m
        inactive=10m;

    server {
        location / {
            proxy_cache micro_cache;
            proxy_cache_valid 200 1s;       # Cache for just 1 second!
            proxy_cache_lock on;            # Only 1 request updates cache
            proxy_cache_use_stale updating; # Serve stale while updating

            add_header X-Cache-Status $upstream_cache_status;
            proxy_pass http://backend;
        }
    }
}
```

**Why is this genius?**

If you get 10,000 requests per second:
- **Without micro-caching:** 10,000 requests hit the backend
- **With 1-second cache:** Only **1 request** hits the backend, the other 9,999 are served from cache

Your content is still "fresh" (never more than 1 second old), but your backend load drops by **99.99%**!

---

### Monitoring Cache Performance

Check how well your cache is performing:

```bash

# Look for: X-Cache-Status: HIT or MISS

# Check cache directory size

# Watch cache files in real time
```

## 4. Part 3 — Security Headers

### What are Security Headers?

Security headers are **HTTP response headers** that tell the browser how to behave when handling your site's content. They're like security instructions that your server sends to the browser.

```
Server: "Hey browser, here's the web page. Also, here are some rules:
         - Don't let anyone embed this page in a frame
         - Only load scripts from my domain
         - Always use HTTPS
         - Don't sniff the content type"

Browser: "Got it! I'll enforce those rules." ✅
```

---

### Why Security Headers Matter

Without security headers, your users are vulnerable to:

| Attack | What Happens | Which Header Prevents It |
|--------|-------------|-------------------------|
| **Clickjacking** | Your site is embedded in a hidden iframe, users click things without knowing | `X-Frame-Options` |
| **XSS (Cross-Site Scripting)** | Attackers inject malicious JavaScript into your pages | `Content-Security-Policy` |
| **MIME Sniffing** | Browser guesses wrong file type, executes a "text" file as JavaScript | `X-Content-Type-Options` |
| **Protocol Downgrade** | User is tricked into using HTTP instead of HTTPS | `Strict-Transport-Security` |
| **Data Leaking via Referrer** | Your full URL (including sensitive query params) is sent to other sites | `Referrer-Policy` |
| **Feature Abuse** | Embedded content accesses camera, microphone, etc. | `Permissions-Policy` |

---

### Essential Security Headers (Complete List)

Let's go through every important security header one by one:

#### 1. X-Frame-Options

**What it does:** Controls whether your site can be embedded in an `<iframe>`, `<frame>`, or `<object>`.

**Why it matters:** Prevents **clickjacking** — where an attacker puts your site in a transparent iframe and tricks users into clicking things.

```nginx
# Options:
add_header X-Frame-Options "DENY" always;              # No one can frame this page
add_header X-Frame-Options "SAMEORIGIN" always;         # Only my own site can frame it
```

| Value | Meaning |
|-------|---------|
| `DENY` | No one can embed this page. Safest option |
| `SAMEORIGIN` | Only pages from the same domain can embed it |

> 📝 **Note:** `X-Frame-Options` is being replaced by `Content-Security-Policy: frame-ancestors`, but you should still include it for older browser support.

---

#### 2. X-Content-Type-Options

**What it does:** Tells the browser to **not try to guess** (sniff) the MIME type of a file. Trust the `Content-Type` header instead.

**Why it matters:** Without this, a browser might treat a text file as JavaScript and execute it.

```nginx
add_header X-Content-Type-Options "nosniff" always;
```

There's only one value: `nosniff`. Always use it.

---

#### 3. X-XSS-Protection

**What it does:** Activates the browser's built-in XSS filter.

```nginx
add_header X-XSS-Protection "1; mode=block" always;
```

| Value | Meaning |
|-------|---------|
| `0` | Disable XSS filter |
| `1` | Enable XSS filter (sanitize the page) |
| `1; mode=block` | Enable and block the entire page if XSS is detected |

> 📝 **Note:** This header is deprecated in modern browsers (Chrome removed it). Use `Content-Security-Policy` instead. Still good to include for older browsers.

---

#### 4. Strict-Transport-Security (HSTS)

**What it does:** Forces browsers to **always use HTTPS** for your site. Once a browser sees this header, it will never try HTTP again (until the max-age expires).

```nginx
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always;
```

| Parameter | Meaning |
|-----------|---------|
| `max-age=31536000` | Remember this rule for 1 year (in seconds) |
| `includeSubDomains` | Apply to all subdomains too |
| `preload` | Allow inclusion in browser's built-in preload list |

**How it works:**
1. User visits `http://example.com`
2. Server redirects to `https://example.com` and sends HSTS header
3. Browser remembers: "Always use HTTPS for example.com"
4. Next time user types `http://example.com`, browser **automatically** changes it to `https://` — the HTTP request is never even sent!

> ⚠️ **Warning:** Once you enable HSTS with a long max-age, you MUST keep HTTPS working. If your SSL certificate expires, users literally cannot access your site (even over HTTP).

---

#### 5. Referrer-Policy

**What it does:** Controls what information is sent in the `Referer` header when users click links to other sites.

```nginx
add_header Referrer-Policy "strict-origin-when-cross-origin" always;
```

| Value | What's Sent to Other Sites |
|-------|---------------------------|
| `no-referrer` | Nothing |
| `origin` | Just the domain (e.g., `https://example.com`) |
| `strict-origin` | Domain only, and only over HTTPS |
| `strict-origin-when-cross-origin` | Full URL for same-site, domain only for other sites, nothing over HTTP |
| `no-referrer-when-downgrade` | Full URL, but nothing when going from HTTPS to HTTP |

**Recommended:** `strict-origin-when-cross-origin` — It's the best balance of privacy and functionality.

---

#### 6. Permissions-Policy (formerly Feature-Policy)

**What it does:** Controls which browser features and APIs your site can use (camera, microphone, geolocation, etc.)

```nginx
add_header Permissions-Policy "camera=(), microphone=(), geolocation=(), payment=()" always;
```

| Syntax | Meaning |
|--------|---------|
| `camera=()` | Disable camera access completely |
| `camera=(self)` | Allow only for your own site |
| `camera=(self "https://trusted.com")` | Allow for your site and trusted.com |
| `camera=*` | Allow for everyone (not recommended) |

Common features to control:
```nginx
add_header Permissions-Policy "
    camera=(),
    microphone=(),
    geolocation=(),
    payment=(),
    usb=(),
    magnetometer=(),
    gyroscope=(),
    accelerometer=(),
    autoplay=(self),
    fullscreen=(self)
" always;
```

---

#### 7. Content-Security-Policy (CSP)

This is the **most powerful** (and most complex) security header. It gets its own deep dive section below.

---

### Complete Security Headers Configuration

Here's a production-ready security headers block you can copy-paste:

```nginx
server {
    listen 443 ssl http2;
    server_name example.com;

    # ==========================================
    # SECURITY HEADERS
    # ==========================================

    # Prevent clickjacking
    add_header X-Frame-Options "SAMEORIGIN" always;

    # Prevent MIME type sniffing
    add_header X-Content-Type-Options "nosniff" always;

    # XSS Protection (legacy browsers)
    add_header X-XSS-Protection "1; mode=block" always;

    # Force HTTPS for 1 year
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always;

    # Control referrer information
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;

    # Restrict browser features
    add_header Permissions-Policy "camera=(), microphone=(), geolocation=(), payment=()" always;

    # Content Security Policy (see deep dive below for customization)
    add_header Content-Security-Policy "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data:; font-src 'self'; connect-src 'self'; frame-ancestors 'self'; base-uri 'self'; form-action 'self'" always;

    # Prevent information leakage
    server_tokens off;            # Hide Nginx version
    more_clear_headers Server;    # Remove Server header entirely (requires headers-more module)

    # ==========================================

    location / {
        proxy_pass http://backend;
    }
}
```

> 💡 **The `always` keyword:** Adding `always` at the end ensures the header is sent even for error responses (4xx, 5xx). Without it, headers are only sent for successful responses.

---

### Content Security Policy (CSP) Deep Dive

CSP is like a whitelist for your webpage. It tells the browser exactly what resources are allowed to load and from where.

**The core idea:**
```
"Dear browser, only load JavaScript from MY domain. If you see a <script> tag
 trying to load from evil-hacker.com, BLOCK IT."
```

**Basic CSP structure:**

```nginx
add_header Content-Security-Policy "
    default-src 'self';
    script-src 'self' https://cdn.example.com;
    style-src 'self' 'unsafe-inline';
    img-src 'self' data: https:;
    font-src 'self' https://fonts.gstatic.com;
    connect-src 'self' https://api.example.com;
    frame-ancestors 'self';
    base-uri 'self';
    form-action 'self'
" always;
```

**CSP directives explained:**

| Directive | Controls | Example |
|-----------|----------|---------|
| `default-src` | Fallback for everything not specified | `'self'` |
| `script-src` | JavaScript sources | `'self' https://cdn.example.com` |
| `style-src` | CSS sources | `'self' 'unsafe-inline'` |
| `img-src` | Image sources | `'self' data: https:` |
| `font-src` | Font sources | `'self' https://fonts.gstatic.com` |
| `connect-src` | XHR, WebSocket, fetch() | `'self' https://api.example.com` |
| `media-src` | Audio/video sources | `'self'` |
| `frame-src` | Allowed iframe sources | `'self' https://youtube.com` |
| `frame-ancestors` | Who can embed THIS page | `'self'` (replaces X-Frame-Options) |
| `base-uri` | Allowed `<base>` tag URLs | `'self'` |
| `form-action` | Where forms can submit to | `'self'` |
| `object-src` | Flash, Java applets (legacy) | `'none'` |

**CSP source values:**

| Value | Meaning |
|-------|---------|
| `'self'` | Same origin (your own domain) |
| `'none'` | Block everything |
| `'unsafe-inline'` | Allow inline `<script>` and `<style>` tags (avoid if possible!) |
| `'unsafe-eval'` | Allow `eval()` in JavaScript (avoid!) |
| `https:` | Any HTTPS URL |
| `data:` | Allow `data:` URIs (used for inline images) |
| `https://example.com` | Specific domain |
| `*.example.com` | Any subdomain of example.com |
| `'nonce-abc123'` | Allow specific inline script with matching nonce |
| `'sha256-...'` | Allow script with matching hash |

**Report-Only Mode (test without breaking things):**

When first implementing CSP, use **report-only** mode to see what would break:

```nginx
# This WON'T block anything — just reports violations
add_header Content-Security-Policy-Report-Only "
    default-src 'self';
    script-src 'self';
    report-uri /csp-report;
" always;
```

Then check the reports and adjust your policy before enforcing it.

**Common CSP configurations:**

```nginx
# Strict (for simple apps with no external dependencies)
add_header Content-Security-Policy "default-src 'self'; object-src 'none'; base-uri 'self'" always;

# For apps using Google Fonts + CDN
add_header Content-Security-Policy "
    default-src 'self';
    script-src 'self' https://cdn.jsdelivr.net;
    style-src 'self' https://fonts.googleapis.com 'unsafe-inline';
    font-src 'self' https://fonts.gstatic.com;
    img-src 'self' data: https:;
    object-src 'none'
" always;

# For apps using Google Analytics + YouTube
add_header Content-Security-Policy "
    default-src 'self';
    script-src 'self' https://www.googletagmanager.com https://www.google-analytics.com;
    img-src 'self' https://www.google-analytics.com;
    frame-src https://www.youtube.com;
    connect-src 'self' https://www.google-analytics.com;
    object-src 'none'
" always;
```

## 5. Putting It All Together — Production-Ready Config

Here's a complete, production-ready Nginx configuration with rate limiting, caching, AND security headers:

```nginx
# /etc/nginx/nginx.conf

user nginx;
worker_processes auto;
error_log /var/log/nginx/error.log warn;
pid /var/run/nginx.pid;

events {
    worker_connections 4096;
}

http {
    include /etc/nginx/mime.types;
    default_type application/octet-stream;

    # ==========================================
    # LOGGING (with cache status)
    # ==========================================
    log_format main '$remote_addr - $remote_user [$time_local] '
                    '"$request" $status $body_bytes_sent '
                    '"$http_referer" "$http_user_agent" '
                    'cache=$upstream_cache_status rt=$request_time';

    access_log /var/log/nginx/access.log main;

    # ==========================================
    # PERFORMANCE
    # ==========================================
    sendfile on;
    tcp_nopush on;
    tcp_nodelay on;
    keepalive_timeout 65;
    gzip on;
    gzip_types text/plain text/css application/json application/javascript text/xml;
    gzip_min_length 256;

    # ==========================================
    # RATE LIMITING ZONES
    # ==========================================
    
    # General rate limit: 50 req/sec per IP
    limit_req_zone $binary_remote_addr zone=general:10m rate=50r/s;
    
    # API rate limit: 20 req/sec per IP
    limit_req_zone $binary_remote_addr zone=api:10m rate=20r/s;
    
    # Login rate limit: 5 req/min per IP (strict!)
    limit_req_zone $binary_remote_addr zone=login:10m rate=5r/m;

    # Connection limit
    limit_conn_zone $binary_remote_addr zone=conn:10m;

    # Return 429 instead of 503 for rate limiting
    limit_req_status 429;
    limit_conn_status 429;

    # ==========================================
    # CACHING
    # ==========================================
    proxy_cache_path /var/cache/nginx/proxy
        levels=1:2
        keys_zone=proxy_cache:20m
        max_size=2g
        inactive=60m
        use_temp_path=off;

    # ==========================================
    # UPSTREAM
    # ==========================================
    upstream backend {
        least_conn;
        server 127.0.0.1:3000;
        server 127.0.0.1:3001;
        keepalive 32;
    }

    # ==========================================
    # REDIRECT HTTP → HTTPS
    # ==========================================
    server {
        listen 80;
        server_name example.com www.example.com;
        return 301 https://$host$request_uri;
    }

    # ==========================================
    # MAIN HTTPS SERVER
    # ==========================================
    server {
        listen 443 ssl http2;
        server_name example.com www.example.com;

        # SSL Configuration
        ssl_certificate /etc/ssl/certs/example.com.crt;
        ssl_certificate_key /etc/ssl/private/example.com.key;
        ssl_protocols TLSv1.2 TLSv1.3;
        ssl_ciphers HIGH:!aNULL:!MD5;
        ssl_prefer_server_ciphers on;
        ssl_session_cache shared:SSL:10m;

        # Hide Nginx version
        server_tokens off;

        # ======================================
        # SECURITY HEADERS (applied globally)
        # ======================================
        add_header X-Frame-Options "SAMEORIGIN" always;
        add_header X-Content-Type-Options "nosniff" always;
        add_header X-XSS-Protection "1; mode=block" always;
        add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always;
        add_header Referrer-Policy "strict-origin-when-cross-origin" always;
        add_header Permissions-Policy "camera=(), microphone=(), geolocation=()" always;
        add_header Content-Security-Policy "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; font-src 'self'; connect-src 'self'; frame-ancestors 'self'; base-uri 'self'; form-action 'self'" always;

        # Connection limit
        limit_conn conn 50;

        # ======================================
        # STATIC FILES (browser-cached)
        # ======================================
        location ~* \.(jpg|jpeg|png|gif|ico|svg|webp|woff|woff2|ttf|eot)$ {
            root /var/www/example.com/public;
            expires 30d;
            add_header Cache-Control "public, immutable";
            access_log off;
        }

        location ~* \.(css|js)$ {
            root /var/www/example.com/public;
            expires 7d;
            add_header Cache-Control "public";
        }

        # ======================================
        # LOGIN (strict rate limiting)
        # ======================================
        location /login {
            limit_req zone=login burst=3 nodelay;
            limit_req zone=general burst=20 nodelay;
            proxy_pass http://backend;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
        }

        # ======================================
        # API (moderate rate limiting + caching)
        # ======================================
        location /api/ {
            limit_req zone=api burst=30 delay=10;

            # Cache GET requests
            proxy_cache proxy_cache;
            proxy_cache_key "$request_method$host$request_uri";
            proxy_cache_valid 200 1m;   # Micro-cache for 1 minute
            proxy_cache_valid 404 30s;
            proxy_cache_methods GET HEAD;
            proxy_cache_use_stale error timeout http_500 http_502 http_503 http_504;
            proxy_cache_lock on;
            proxy_cache_background_update on;

            # Don't cache POST/PUT/DELETE
            set $no_cache 0;
            if ($request_method != GET) {
                set $no_cache 1;
            }
            proxy_no_cache $no_cache;
            proxy_cache_bypass $no_cache;

            add_header X-Cache-Status $upstream_cache_status always;

            proxy_pass http://backend;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
        }

        # ======================================
        # EVERYTHING ELSE
        # ======================================
        location / {
            limit_req zone=general burst=50 nodelay;

            proxy_cache proxy_cache;
            proxy_cache_valid 200 5m;
            proxy_cache_use_stale error timeout http_500 http_502 http_503;
            proxy_cache_background_update on;

            add_header X-Cache-Status $upstream_cache_status always;

            proxy_pass http://backend;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
        }

        # ======================================
        # CUSTOM ERROR PAGES
        # ======================================
        error_page 429 /429.html;
        location = /429.html {
            root /usr/share/nginx/html;
            internal;
        }

        error_page 500 502 503 504 /50x.html;
        location = /50x.html {
            root /usr/share/nginx/html;
            internal;
        }
    }
}
```

## 6. Common Mistakes & Troubleshooting

### Rate Limiting Issues

| Problem | Cause | Fix |
|---------|-------|-----|
| All users getting rate limited | Rate limiting by wrong key | Make sure you're using `$binary_remote_addr`, not `$remote_addr` when behind a load balancer, use `$http_x_forwarded_for` |
| Rate limit not working | Zone not defined in `http` block | `limit_req_zone` MUST be in the `http` block, not `server` or `location` |
| Getting 503 instead of 429 | Missing `limit_req_status` | Add `limit_req_status 429;` in `http` block |
| Legitimate users affected | Rate too strict | Increase `burst` and use `nodelay` or `delay=` |

### Caching Issues

| Problem | Cause | Fix |
|---------|-------|-----|
| Nothing is being cached | Backend sends `Cache-Control: no-cache` or `Set-Cookie` | Use `proxy_ignore_headers Cache-Control Set-Cookie;` |
| Cache not updating | `inactive` period too long | Reduce `inactive` value or use `proxy_cache_valid` |
| Users see old content | Cache TTL too long | Reduce `proxy_cache_valid` or implement cache purging |
| Cache directory fills up | `max_size` too large or `inactive` too long | Adjust both values |
| Cache not created | Directory permissions wrong | `chown nginx:nginx /var/cache/nginx` |

### Security Headers Issues

| Problem | Cause | Fix |
|---------|-------|-----|
| Headers not showing | Using `add_header` in a `location` block that already has `add_header` | Nginx `add_header` is NOT inherited if any `add_header` is used in the child block. Repeat all headers. |
| CSP breaking the site | Policy too strict | Use `Content-Security-Policy-Report-Only` first |
| HSTS locking out users | SSL certificate expired | Keep certificates valid! Consider a shorter `max-age` initially |
| Security headers not on errors | Missing `always` keyword | Add `always` to all `add_header` directives |

> ⚠️ **Critical Nginx Gotcha:** The `add_header` directive is NOT inherited from parent blocks if ANY `add_header` is used in a child block. If you add a header in a `location` block, you must repeat ALL headers from the `server` block too!

```nginx
# ❌ BAD — security headers from server block are LOST in this location
server {
    add_header X-Frame-Options "SAMEORIGIN" always;
    add_header X-Content-Type-Options "nosniff" always;

    location /api/ {
        add_header X-Cache-Status $upstream_cache_status;  # This kills the inherited headers!
        proxy_pass http://backend;
    }
}

# ✅ GOOD — repeat all headers, or use the 'include' trick
server {
    include /etc/nginx/security-headers.conf;  # Put all security headers in a file

    location /api/ {
        include /etc/nginx/security-headers.conf;  # Include it here too
        add_header X-Cache-Status $upstream_cache_status always;
        proxy_pass http://backend;
    }
}
```

```nginx
# Define zone (http block)
limit_req_zone $binary_remote_addr zone=NAME:SIZE rate=RATE;

limit_req_status 429;

limit_conn_zone $binary_remote_addr zone=NAME:SIZE;
limit_conn NAME N;
```

```nginx
# Define cache (http block)
proxy_cache_path PATH levels=1:2 keys_zone=NAME:SIZE max_size=SIZE inactive=TIME use_temp_path=off;
```
