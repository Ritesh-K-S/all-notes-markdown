## 1. What is HAProxy?

**HAProxy** (High Availability Proxy) is a **free, open-source, high-performance TCP/HTTP load balancer and proxy server**. It's considered the gold standard for load balancing in the industry.

**Think of HAProxy like this:** If your backend servers are restaurants in a food court, HAProxy is the person at the entrance directing customers to the restaurant with the shortest line. It makes sure no single restaurant gets overwhelmed while others sit empty.

**Key facts:**
- Written in **C** — incredibly fast and efficient
- Can handle **millions of connections** simultaneously
- Used by **GitHub, Stack Overflow, Reddit, Airbnb, Instagram, Twitter/X**
- Supports both **Layer 4 (TCP)** and **Layer 7 (HTTP)** load balancing
- First release in **2000** — battle-tested for 25+ years
- Very low memory footprint (~15MB for simple setups)
- Single-threaded by design (but supports multi-threading since v1.8)

### What Can HAProxy Do?

| Capability | Description |
|-----------|-------------|
| **Load Balancing** | Distribute traffic across multiple backend servers |
| **Health Checking** | Automatically detect and remove unhealthy servers |
| **SSL Termination** | Handle HTTPS and forward plain HTTP to backends |
| **Rate Limiting** | Protect backends from abuse using stick tables |
| **Content-Based Routing** | Route based on URL, headers, cookies, etc. |
| **Session Persistence** | Ensure a user always goes to the same backend (sticky sessions) |
| **TCP Proxying** | Load balance databases, mail servers, any TCP service |
| **HTTP/2 & WebSocket** | Full support for modern protocols |
| **Connection Queuing** | Queue connections when backends are at capacity |

## 2. HAProxy vs Nginx vs Other Load Balancers

| Feature | HAProxy | Nginx | AWS ALB/NLB | Traefik |
|---------|---------|-------|-------------|---------|
| **Primary purpose** | Load balancing | Web server + proxy | Cloud LB | Cloud-native proxy |
| **L4 (TCP) LB** | ✅ Excellent | ✅ Good | ✅ NLB | ✅ Good |
| **L7 (HTTP) LB** | ✅ Excellent | ✅ Excellent | ✅ ALB | ✅ Good |
| **Static files** | ❌ Not designed for this | ✅ Excellent | ❌ N/A | ❌ Not designed for this |
| **Health checks** | ✅ Very advanced | ✅ Basic (advanced in Plus) | ✅ Good | ✅ Good |
| **Stats dashboard** | ✅ Built-in | ❌ Only in Plus | ✅ AWS Console | ✅ Built-in |
| **Configuration** | Config file | Config file | API/Console | Labels/Files/API |
| **Performance** | ✅ Extremely high | ✅ Very high | ✅ Auto-scales | ✅ Good |
| **SSL termination** | ✅ Yes | ✅ Yes | ✅ Yes | ✅ Auto (Let's Encrypt) |
| **Stick tables** | ✅ Unique feature! | ❌ No | ❌ No | ❌ No |
| **Cost** | Free | Free (Plus = paid) | Pay per use | Free |

**When to use HAProxy:**
- You need **serious load balancing** (high traffic, complex routing)
- You need **TCP-level** load balancing (databases, message queues)
- You need **advanced health checks** (custom scripts, intervals)
- You want a **dedicated load balancer** that does one thing extremely well
- You need **stick tables** for rate limiting and session tracking

## 3. HAProxy Architecture

### How HAProxy Processes Connections

```
                     Internet
                        │
                        ▼
            ┌──────────────────────┐
            │      FRONTEND        │    ← Listens on a port (e.g., 80, 443)
            │  (Accepts connections)│    ← Applies ACLs (routing rules)
            │  (SSL termination)   │    ← Chooses which backend to use
            └──────────┬───────────┘
                       │
          ┌────────────┼────────────┐
          │            │            │
          ▼            ▼            ▼
    ┌──────────┐ ┌──────────┐ ┌──────────┐
    │ BACKEND  │ │ BACKEND  │ │ BACKEND  │
    │ (api)    │ │ (web)    │ │ (static) │
    └────┬─────┘ └────┬─────┘ └────┬─────┘
         │            │            │
    ┌────┼────┐  ┌────┼────┐  ┌────┼────┐
    ▼    ▼    ▼  ▼    ▼    ▼  ▼    ▼    ▼
  [S1] [S2] [S3] [S4] [S5] [S6] [S7] [S8] [S9]
  
  S = Backend Server
```

**The flow:**
1. **Client** connects to HAProxy
2. **Frontend** accepts the connection, applies rules (ACLs) to decide routing
3. **Backend** receives the connection, picks a server using the load balancing algorithm
4. **Server** processes the request, sends response back through HAProxy

### Layer 4 vs Layer 7 Load Balancing

| Aspect | Layer 4 (TCP) | Layer 7 (HTTP) |
|--------|--------------|----------------|
| **Works at** | Transport layer (TCP/UDP) | Application layer (HTTP) |
| **Sees** | IP addresses, ports, raw TCP | URLs, headers, cookies, body |
| **Can route by** | IP, port | URL path, hostname, headers, cookies |
| **Speed** | Faster (less processing) | Slightly slower (must parse HTTP) |
| **Use case** | Databases, mail, generic TCP | Web apps, APIs, microservices |
| **Mode in HAProxy** | `mode tcp` | `mode http` |

**Examples:**

```
Layer 4 (TCP mode):
"Hey, a TCP connection came in on port 3306. I'll send it to one of the MySQL servers."
→ HAProxy doesn't even look at the data inside the packets.

Layer 7 (HTTP mode):
"Hey, an HTTP request came in for /api/users. Let me check the URL, cookies, and headers,
 then send it to the API backend. Oh, and /images/* goes to the static backend."
→ HAProxy fully understands HTTP and makes smart routing decisions.
```

## 5. Configuration File Structure

HAProxy's configuration file (`/etc/haproxy/haproxy.cfg`) has **four main sections**:

```
┌─────────────────────────────────────────────────┐
│                   GLOBAL                         │
│  Process-wide settings (logging, threads, etc.)  │
├─────────────────────────────────────────────────┤
│                  DEFAULTS                        │
│  Default settings inherited by all frontends     │
│  and backends (timeouts, mode, etc.)             │
├─────────────────────────────────────────────────┤
│                  FRONTEND                        │
│  Where connections come in (bind, ACLs, routing) │
│  You can have multiple frontends                 │
├─────────────────────────────────────────────────┤
│                  BACKEND                         │
│  Where connections go (servers, health checks)   │
│  You can have multiple backends                  │
├─────────────────────────────────────────────────┤
│               LISTEN (Optional)                  │
│  Combined frontend + backend in one block        │
└─────────────────────────────────────────────────┘
```

Here's a minimal working configuration to get started:

```haproxy
# /etc/haproxy/haproxy.cfg

global
    log /dev/log local0
    maxconn 4096
    user haproxy
    group haproxy
    daemon

defaults
    log     global
    mode    http
    option  httplog
    option  dontlognull
    timeout connect 5s
    timeout client  30s
    timeout server  30s

frontend http_front
    bind *:80
    default_backend web_servers

backend web_servers
    balance roundrobin
    server web1 192.168.1.10:80 check
    server web2 192.168.1.11:80 check
    server web3 192.168.1.12:80 check
```

That's it! This config:
- Listens on port 80
- Load balances across 3 web servers using round-robin
- Health checks each server

Let's now go deep into each section.

## 6. The Global Section

The `global` section defines process-wide settings. It runs once when HAProxy starts.

```haproxy
global
    # ==========================================
    # LOGGING
    # ==========================================
    log /dev/log local0            # Send logs to syslog, facility local0
    log /dev/log local1 notice     # Additional log target with minimum level "notice"

    # ==========================================
    # PROCESS MANAGEMENT
    # ==========================================
    daemon                          # Run as a background daemon
    user haproxy                    # Run as this user (never root!)
    group haproxy                   # Run as this group
    chroot /var/lib/haproxy         # Jail HAProxy to this directory (security)
    pidfile /var/run/haproxy.pid    # PID file location

    # ==========================================
    # PERFORMANCE
    # ==========================================
    maxconn 50000                   # Maximum concurrent connections (total)
    nbthread 4                      # Number of threads (usually = CPU cores)
    cpu-map auto:1/1-4 0-3         # Pin threads to CPU cores

    # ==========================================
    # SSL SETTINGS
    # ==========================================
    ssl-default-bind-ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256
    ssl-default-bind-ciphersuites TLS_AES_128_GCM_SHA256:TLS_AES_256_GCM_SHA384
    ssl-default-bind-options ssl-min-ver TLSv1.2 no-tls-tickets
    tune.ssl.default-dh-param 2048  # DH parameter size for SSL

    # ==========================================
    # STATS SOCKET (for runtime management)
    # ==========================================
    stats socket /var/run/haproxy/admin.sock mode 660 level admin
    stats timeout 30s
```

**Key settings explained:**

| Setting | What It Does | Typical Value |
|---------|-------------|---------------|
| `maxconn` | Total connections HAProxy can handle | 50000-100000 |
| `nbthread` | Number of threads | Number of CPU cores |
| `chroot` | Restricts HAProxy to a directory (can't access anything outside) | `/var/lib/haproxy` |
| `stats socket` | Creates a Unix socket for runtime commands (drain servers, check stats) | Required for operations |

## 7. The Defaults Section

The `defaults` section sets default values that apply to ALL frontends and backends (unless overridden).

```haproxy
defaults
    # ==========================================
    # LOGGING
    # ==========================================
    log global                      # Use the log settings from 'global'
    option httplog                  # Log full HTTP request details
    option dontlognull              # Don't log empty connections (health checks from LB)

    # ==========================================
    # MODE
    # ==========================================
    mode http                       # Default mode: HTTP (Layer 7)
                                    # Use 'tcp' for Layer 4

    # ==========================================
    # TIMEOUTS (CRITICAL! — always set these)
    # ==========================================
    timeout connect 5s              # Time to wait for backend connection
    timeout client  30s             # Max time for client to send data
    timeout server  30s             # Max time for backend to respond
    timeout http-request 10s        # Max time for complete HTTP request
    timeout http-keep-alive 10s     # Max idle time for keep-alive connections
    timeout queue  30s              # Max time a request waits in queue
    timeout check  5s               # Max time for health check response
    timeout tunnel 1h               # Timeout for WebSocket/tunnel connections

    # ==========================================
    # OPTIONS
    # ==========================================
    option http-server-close        # Close connection to server after response
    option forwardfor               # Add X-Forwarded-For header
    option redispatch               # Retry on a different server if one fails
    retries 3                       # Number of retries before giving up

    # ==========================================
    # ERROR PAGES
    # ==========================================
    errorfile 400 /etc/haproxy/errors/400.http
    errorfile 403 /etc/haproxy/errors/403.http
    errorfile 408 /etc/haproxy/errors/408.http
    errorfile 500 /etc/haproxy/errors/500.http
    errorfile 502 /etc/haproxy/errors/502.http
    errorfile 503 /etc/haproxy/errors/503.http
    errorfile 504 /etc/haproxy/errors/504.http
```

### Understanding Timeouts (This is CRITICAL!)

Timeouts are the **#1 source of HAProxy issues**. Let's visualize them:

```
Client                    HAProxy                   Backend Server
  │                         │                           │
  │── TCP connect ──────────│                           │
  │                         │── TCP connect ─────────── │
  │                         │   ◄── timeout connect ──► │
  │                         │                           │
  │── HTTP Request ────────►│                           │
  │   ◄── timeout client ──►│                           │
  │                         │── Forward Request ──────► │
  │                         │   ◄── timeout server ──►  │
  │                         │◄── Response ──────────── │
  │◄── Response ────────────│                           │
  │                         │                           │
  │   (idle — keep alive)   │                           │
  │  ◄─ timeout http-keep-alive ─►                     │
```

| Timeout | What Happens If Exceeded | Typical Value |
|---------|-------------------------|---------------|
| `connect` | HAProxy can't reach backend → tries next server | 5s |
| `client` | Client is too slow sending data → connection dropped | 30s |
| `server` | Backend is too slow responding → 504 returned | 30s |
| `http-request` | Client takes too long to send complete HTTP request → 408 | 10s |
| `http-keep-alive` | Idle keep-alive connection closed | 10s |
| `queue` | Request waited too long in queue → 503 | 30s |

> ⚠️ **Critical Rule:** `timeout server` must be greater than your backend's longest response time. If your API can take 60 seconds for heavy queries, set `timeout server 65s`.

## 8. Frontend — Where Traffic Comes In

The frontend defines **how HAProxy receives connections** from clients.

```haproxy
frontend http_front
    # ==========================================
    # BIND — What port/address to listen on
    # ==========================================
    bind *:80                      # Listen on all interfaces, port 80
    bind *:443 ssl crt /etc/ssl/certs/combined.pem  # HTTPS

    # ==========================================
    # MODE
    # ==========================================
    mode http                       # HTTP mode (Layer 7)

    # ==========================================
    # DEFAULT BACKEND (if no ACL matches)
    # ==========================================
    default_backend web_servers

    # ==========================================
    # ACLs — Routing Rules
    # ==========================================
    # Route based on URL path
    acl is_api path_beg /api/
    acl is_static path_end .css .js .png .jpg .gif .ico

    # Route based on hostname
    acl is_blog hdr(host) -i blog.example.com
    acl is_api_host hdr(host) -i api.example.com

    # Apply routing
    use_backend api_servers if is_api
    use_backend api_servers if is_api_host
    use_backend static_servers if is_static
    use_backend blog_servers if is_blog

    # ==========================================
    # HTTP HEADERS
    # ==========================================
    # Add security headers
    http-response set-header X-Frame-Options SAMEORIGIN
    http-response set-header X-Content-Type-Options nosniff
    http-response set-header X-XSS-Protection "1; mode=block"
    http-response set-header Strict-Transport-Security "max-age=31536000"

    # Remove server information header
    http-response del-header Server

    # ==========================================
    # REDIRECT HTTP → HTTPS
    # ==========================================
    http-request redirect scheme https unless { ssl_fc }
```

**Key frontend directives:**

| Directive | What It Does | Example |
|-----------|-------------|---------|
| `bind` | Address and port to listen on | `*:80`, `*:443 ssl crt /path/cert.pem` |
| `mode` | Operating mode | `http` or `tcp` |
| `default_backend` | Where to send traffic if no ACL matches | `web_servers` |
| `acl` | Define a condition | `acl is_api path_beg /api/` |
| `use_backend` | Route to backend if ACL matches | `use_backend api if is_api` |
| `http-request` | Manipulate the request | Redirect, add headers, deny |
| `http-response` | Manipulate the response | Set headers, remove headers |

## 9. Backend — Where Traffic Goes

The backend defines the **pool of servers** that handle requests and how to distribute traffic among them.

```haproxy
backend web_servers
    # ==========================================
    # MODE & BALANCE
    # ==========================================
    mode http
    balance roundrobin              # Load balancing algorithm

    # ==========================================
    # STICKY SESSIONS (optional)
    # ==========================================
    cookie SERVERID insert indirect nocache

    # ==========================================
    # HTTP OPTIONS
    # ==========================================
    option httpchk GET /health      # Health check: send GET /health
    http-check expect status 200    # Expect 200 OK response

    # Forward original client IP
    http-request set-header X-Forwarded-For %[src]
    http-request set-header X-Forwarded-Proto %[ssl_fc,iif(https,http)]
    http-request set-header X-Real-IP %[src]

    # ==========================================
    # SERVERS
    # ==========================================
    # Syntax: server <name> <address>:<port> [options]
    server web1 192.168.1.10:8080 check cookie web1 weight 100
    server web2 192.168.1.11:8080 check cookie web2 weight 100
    server web3 192.168.1.12:8080 check cookie web3 weight 50    # Gets half the traffic
    server web4 192.168.1.13:8080 check cookie web4 backup       # Only used when others are down
```

**Server options explained:**

| Option | What It Does | Example |
|--------|-------------|---------|
| `check` | Enable health checking | `server web1 ... check` |
| `weight` | Relative weight for load balancing (default: 1) | `weight 100` |
| `cookie` | Set a cookie value for sticky sessions | `cookie web1` |
| `backup` | Only use when all other servers are down | `backup` |
| `maxconn` | Maximum connections to this server | `maxconn 100` |
| `inter` | Health check interval | `inter 3s` |
| `fall` | Consecutive failed checks before marking down | `fall 3` |
| `rise` | Consecutive passed checks before marking up | `rise 2` |
| `disabled` | Start in disabled state (maintenance) | `disabled` |
| `resolve-prefer ipv4` | Prefer IPv4 for DNS resolution | `resolve-prefer ipv4` |

### Full Server Line Examples

```haproxy
# Basic server with health check
server web1 192.168.1.10:8080 check

# Server with all options
server web1 192.168.1.10:8080 check inter 5s fall 3 rise 2 weight 100 maxconn 200 cookie web1

# Backup server (failover only)
server backup1 192.168.1.20:8080 check backup

# Server using HTTPS to communicate with backend
server web1 192.168.1.10:443 check ssl verify none

# Server using a Unix socket
server local1 /var/run/app.sock check

# Server with custom port for health checks
server web1 192.168.1.10:8080 check port 8081
```

## 10. Listen — Frontend + Backend in One

The `listen` section combines a frontend and backend in a single block. It's convenient for simple setups:

```haproxy
# Instead of separate frontend and backend:
listen web_app
    bind *:80
    mode http
    balance roundrobin
    option httpchk GET /health
    http-check expect status 200
    server web1 192.168.1.10:8080 check
    server web2 192.168.1.11:8080 check
    server web3 192.168.1.12:8080 check
```

This is equivalent to:
```haproxy
frontend web_app_front
    bind *:80
    default_backend web_app_back

backend web_app_back
    mode http
    balance roundrobin
    option httpchk GET /health
    http-check expect status 200
    server web1 192.168.1.10:8080 check
    server web2 192.168.1.11:8080 check
    server web3 192.168.1.12:8080 check
```

**When to use `listen`:**
- Simple setups with one frontend → one backend
- Stats dashboard
- TCP proxying (databases, Redis, etc.)

**When to use separate `frontend`/`backend`:**
- Multiple backends with ACL-based routing
- Complex routing rules
- Sharing backends across frontends

## 11. Load Balancing Algorithms — Deep Dive

This is the heart of HAProxy. The algorithm determines **which backend server** gets each request.

### 1. Round Robin (`roundrobin`)

```haproxy
backend web_servers
    balance roundrobin
    server web1 192.168.1.10:8080 check weight 1
    server web2 192.168.1.11:8080 check weight 1
    server web3 192.168.1.12:8080 check weight 1
```

**How it works:** Requests are distributed evenly, one by one, in order.

```
Request 1 → web1
Request 2 → web2
Request 3 → web3
Request 4 → web1
Request 5 → web2
...
```

**With weights:**
```haproxy
server web1 192.168.1.10:8080 check weight 3   # Gets 3x traffic
server web2 192.168.1.11:8080 check weight 1   # Gets 1x traffic
```
```
Request 1 → web1
Request 2 → web1
Request 3 → web1
Request 4 → web2
Request 5 → web1
...
```

**Best for:** General use, when all servers are similar. Default and most common.

---

### 2. Least Connections (`leastconn`)

```haproxy
backend web_servers
    balance leastconn
    server web1 192.168.1.10:8080 check
    server web2 192.168.1.11:8080 check
    server web3 192.168.1.12:8080 check
```

**How it works:** Sends each new request to the server with the **fewest active connections**.

```
web1: 5 active connections
web2: 2 active connections  ← New request goes here!
web3: 8 active connections
```

**Best for:** When requests have **varying processing times** (some fast, some slow). Long-lived connections like WebSockets.

---

### 3. Source IP Hash (`source`)

```haproxy
backend web_servers
    balance source
    hash-type consistent    # Use consistent hashing (better for dynamic servers)
    server web1 192.168.1.10:8080 check
    server web2 192.168.1.11:8080 check
    server web3 192.168.1.12:8080 check
```

**How it works:** Hashes the client's source IP address to always send them to the **same backend server**.

```
Client 1.2.3.4 → hash(1.2.3.4) = web2  (always web2!)
Client 5.6.7.8 → hash(5.6.7.8) = web1  (always web1!)
Client 9.0.1.2 → hash(9.0.1.2) = web3  (always web3!)
```

**Best for:** Simple session persistence when you can't use cookies. **Caution:** Doesn't work well behind another proxy/NAT (all users appear as one IP).

---

### 4. URI Hash (`uri`)

```haproxy
backend cache_servers
    balance uri
    hash-type consistent
    server cache1 192.168.1.10:8080 check
    server cache2 192.168.1.11:8080 check
    server cache3 192.168.1.12:8080 check
```

**How it works:** Hashes the request URI so the **same URL always goes to the same server**.

```
/api/users     → hash("/api/users") = cache1     (always cache1!)
/api/products  → hash("/api/products") = cache2   (always cache2!)
/api/orders    → hash("/api/orders") = cache1     (always cache1!)
```

**Best for:** Caching proxies — maximizes cache hit rate because the same content is always on the same server.

---

### 5. Header Hash (`hdr`)

```haproxy
backend api_servers
    balance hdr(X-API-Key)
    hash-type consistent
    server api1 192.168.1.10:8080 check
    server api2 192.168.1.11:8080 check
```

**How it works:** Hashes a specific HTTP header value.

**Best for:** Routing API customers to consistent servers based on their API key.

---

### 6. Random (`random`)

```haproxy
backend web_servers
    balance random
    server web1 192.168.1.10:8080 check weight 1
    server web2 192.168.1.11:8080 check weight 2
```

**How it works:** Picks a random server (respects weights).

**Best for:** Large clusters where randomness provides good distribution.

---

### 7. First Available (`first`)

```haproxy
backend web_servers
    balance first
    server web1 192.168.1.10:8080 check maxconn 100
    server web2 192.168.1.11:8080 check maxconn 100
    server web3 192.168.1.12:8080 check maxconn 100
```

**How it works:** Always sends to the first server that has available connection slots. Fills up servers one at a time.

**Best for:** Cost optimization — you can scale down idle servers if using cloud.

---

### Algorithm Comparison

| Algorithm | Persistence | Use Case | Overhead |
|-----------|------------|----------|----------|
| `roundrobin` | ❌ No | General purpose | Very low |
| `leastconn` | ❌ No | Variable request duration | Low |
| `source` | ✅ By IP | Simple stickiness | Low |
| `uri` | ✅ By URL | Caching proxies | Low |
| `hdr(name)` | ✅ By header | API key routing | Low |
| `random` | ❌ No | Large clusters | Very low |
| `first` | ❌ No | Cost optimization | Very low |

## 12. Health Checks — Keeping Backends Healthy

Health checks are how HAProxy knows if a backend server is alive and can handle requests.

### TCP Health Check (Layer 4)

The simplest check — just verifies the port is open:

```haproxy
backend web_servers
    # Just adding "check" does a TCP connect check
    server web1 192.168.1.10:8080 check
    server web2 192.168.1.11:8080 check inter 3s fall 3 rise 2
```

| Option | What It Does | Default |
|--------|-------------|---------|
| `inter` | Time between checks | 2s |
| `fall` | Failed checks before marking DOWN | 3 |
| `rise` | Successful checks before marking UP | 2 |
| `timeout check` | Max time to wait for check response | depends on `inter` |

**How it works:**
```
HAProxy → "Can I connect to 192.168.1.10:8080?"
Server  → "Yes, TCP connection accepted" → HEALTHY ✅
Server  → [no response] → Count failure. 3 failures = DOWN ❌
```

### HTTP Health Check (Layer 7)

More thorough — actually sends an HTTP request:

```haproxy
backend web_servers
    option httpchk GET /health        # Send GET /health
    http-check expect status 200      # Expect 200 OK

    server web1 192.168.1.10:8080 check
    server web2 192.168.1.11:8080 check
```

**Advanced HTTP health checks:**

```haproxy
backend api_servers
    # Simple: check a URL
    option httpchk GET /health

    # With custom headers
    option httpchk
    http-check send meth GET uri /health hdr Host api.example.com hdr User-Agent HAProxy-HealthCheck

    # Expect specific response
    http-check expect status 200                       # Expect 200
    http-check expect string "status":"ok"             # Expect body contains this string
    http-check expect rstatus ^2                       # Expect any 2xx status
    http-check expect ! string "maintenance"           # Body must NOT contain "maintenance"

    server api1 192.168.1.10:8080 check inter 5s fall 3 rise 2
    server api2 192.168.1.11:8080 check inter 5s fall 3 rise 2
```

### Multi-Step Health Checks

HAProxy 2.2+ supports **multi-step health checks** — a sequence of checks:

```haproxy
backend database_servers
    option httpchk
    http-check connect                              # Step 1: TCP connect
    http-check send meth GET uri /health           # Step 2: Send request
    http-check expect status 200                    # Step 3: Check response
    http-check send meth GET uri /db-check         # Step 4: Check DB connection
    http-check expect string "database: connected"  # Step 5: Verify DB
    
    server db1 192.168.1.10:8080 check
```

### Different Port for Health Checks

Sometimes your app has a separate health endpoint on a different port:

```haproxy
backend web_servers
    option httpchk GET /health
    
    # App runs on 8080, but health endpoint is on 8081
    server web1 192.168.1.10:8080 check port 8081
```

### Server States

```
                  ┌─────┐
           ┌─────│ UP   │◄────────┐
           │     └──┬───┘         │
           │        │             │ rise (2 successful checks)
           │  fall  │             │
           │ (3 failures)        │
           │        │       ┌────┴──────┐
           │        └──────►│   DOWN    │
           │                └────┬──────┘
           │                     │
     ┌─────▼─────┐              │
     │  DRAIN    │              │ (admin action)
     │(finishing) │              │
     └───────────┘         ┌────▼──────┐
                           │   MAINT   │
                           │(admin set) │
                           └───────────┘
```

| State | Meaning |
|-------|---------|
| **UP** | Server is healthy and accepting connections |
| **DOWN** | Server failed health checks |
| **DRAIN** | Server is draining — no new connections, existing ones finish |
| **MAINT** | Manually set to maintenance mode |
| **NOLB** | Server is up but excluded from load balancing |

**Change server state at runtime:**

```bash
# Using the stats socket
echo "set server web_servers/web1 state drain" | sudo socat stdio /var/run/haproxy/admin.sock
echo "set server web_servers/web1 state maint" | sudo socat stdio /var/run/haproxy/admin.sock
echo "set server web_servers/web1 state ready" | sudo socat stdio /var/run/haproxy/admin.sock

# Disable a server (stop sending new connections)
echo "disable server web_servers/web1" | sudo socat stdio /var/run/haproxy/admin.sock

# Enable it back
echo "enable server web_servers/web1" | sudo socat stdio /var/run/haproxy/admin.sock
```

## 13. ACLs — Smart Routing Decisions

ACLs (Access Control Lists) are HAProxy's way of making **intelligent routing decisions** based on request properties.

### ACL Syntax

```haproxy
# Syntax: acl <name> <criterion> [flags] <value>
acl is_api path_beg /api/
```

### Common ACL Criteria

| Criterion | Matches On | Example |
|-----------|-----------|---------|
| `path` | Exact URL path | `acl is_root path /` |
| `path_beg` | URL starts with | `acl is_api path_beg /api/` |
| `path_end` | URL ends with | `acl is_css path_end .css` |
| `path_reg` | Regex on URL path | `acl is_v2 path_reg ^/api/v2/` |
| `hdr(name)` | HTTP header value | `acl is_blog hdr(host) -i blog.example.com` |
| `hdr_beg(name)` | Header starts with | `acl is_mobile hdr_beg(User-Agent) -i mobile` |
| `src` | Source IP | `acl is_office src 10.0.0.0/8` |
| `method` | HTTP method | `acl is_post method POST` |
| `ssl_fc` | Is HTTPS? | `acl is_https ssl_fc` |
| `req.cook(name)` | Cookie value | `acl has_session req.cook(session_id) -m found` |
| `url_param(name)` | URL query parameter | `acl is_debug url_param(debug) -m str true` |

### ACL Flags

| Flag | Meaning | Example |
|------|---------|---------|
| `-i` | Case insensitive | `hdr(host) -i Example.COM` |
| `-m str` | Exact string match | `-m str admin` |
| `-m beg` | Begins with | `-m beg /api` |
| `-m end` | Ends with | `-m end .html` |
| `-m reg` | Regular expression | `-m reg ^/api/v[0-9]+` |
| `-m found` | Value exists | `-m found` |

### Practical ACL Examples

```haproxy
frontend http_front
    bind *:80
    bind *:443 ssl crt /etc/ssl/certs/combined.pem

    # ==========================================
    # DEFINE ACLs
    # ==========================================
    
    # Route by path
    acl is_api path_beg /api/
    acl is_static path_end .css .js .png .jpg .gif .svg .ico .woff .woff2
    acl is_websocket hdr(Upgrade) -i websocket
    
    # Route by hostname
    acl is_blog hdr(host) -i blog.example.com
    acl is_api_host hdr(host) -i api.example.com
    acl is_admin hdr(host) -i admin.example.com
    
    # Security rules
    acl is_office src 10.0.0.0/8 192.168.1.0/24
    acl is_blocked_country src -f /etc/haproxy/blocked_ips.lst   # IPs from a file
    acl is_bad_bot hdr_sub(User-Agent) -i bot crawl spider
    
    # HTTP method rules
    acl is_options method OPTIONS
    
    # HTTPS check
    acl is_https ssl_fc

    # ==========================================
    # APPLY RULES (order matters!)
    # ==========================================
    
    # Force HTTPS
    http-request redirect scheme https unless is_https
    
    # Block bad bots
    http-request deny if is_bad_bot
    
    # Block countries
    http-request deny if is_blocked_country
    
    # Allow CORS preflight
    http-request allow if is_options
    
    # Admin area — office only
    http-request deny if is_admin !is_office
    
    # Route to backends
    use_backend websocket_servers if is_websocket
    use_backend static_servers if is_static
    use_backend api_servers if is_api OR is_api_host
    use_backend blog_servers if is_blog
    use_backend admin_servers if is_admin
    
    # Default
    default_backend web_servers
```

### Combining ACLs

```haproxy
# AND (both must be true) — just list them
use_backend secure_api if is_api is_https

# OR — use the OR keyword
use_backend api_servers if is_api OR is_api_host

# NOT — use !
http-request deny if !is_office

# Complex combinations
# "If it's an API request AND it's HTTPS AND NOT from a blocked country"
use_backend api_servers if is_api is_https !is_blocked_country
```

## 14. SSL/TLS Termination

HAProxy can terminate SSL (handle HTTPS), decrypt the traffic, and forward plain HTTP to your backends.

### Basic SSL Setup

```haproxy
frontend https_front
    # Bind with SSL certificate
    bind *:443 ssl crt /etc/haproxy/certs/example.com.pem

    # Redirect HTTP to HTTPS
    bind *:80
    http-request redirect scheme https unless { ssl_fc }

    default_backend web_servers
```

### Preparing the Certificate File

HAProxy needs the **certificate and private key in a SINGLE file**:

```bash
# Combine certificate + private key
cat /etc/ssl/certs/example.com.crt /etc/ssl/private/example.com.key > /etc/haproxy/certs/example.com.pem

# If you have an intermediate certificate (chain):
cat /etc/ssl/certs/example.com.crt /etc/ssl/certs/chain.crt /etc/ssl/private/example.com.key > /etc/haproxy/certs/example.com.pem

# Set proper permissions
chmod 600 /etc/haproxy/certs/example.com.pem
chown haproxy:haproxy /etc/haproxy/certs/example.com.pem
```

### Multiple Certificates (SNI)

Serve different certificates for different domains:

```haproxy
frontend https_front
    # HAProxy automatically selects the right certificate based on SNI
    bind *:443 ssl crt /etc/haproxy/certs/    # Point to a DIRECTORY — HAProxy reads all .pem files!

    # Or specify multiple certificates explicitly:
    bind *:443 ssl crt /etc/haproxy/certs/example.com.pem crt /etc/haproxy/certs/blog.com.pem

    default_backend web_servers
```

### SSL Hardening

```haproxy
global
    # Minimum TLS version
    ssl-default-bind-options ssl-min-ver TLSv1.2 no-tls-tickets

    # Strong cipher suites
    ssl-default-bind-ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384

    # TLS 1.3 cipher suites
    ssl-default-bind-ciphersuites TLS_AES_128_GCM_SHA256:TLS_AES_256_GCM_SHA384:TLS_CHACHA20_POLY1305_SHA256

    # DH parameter size
    tune.ssl.default-dh-param 2048

frontend https_front
    bind *:443 ssl crt /etc/haproxy/certs/example.com.pem alpn h2,http/1.1

    # HSTS
    http-response set-header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload"
```

### SSL Passthrough (Don't Decrypt — Just Forward)

Sometimes you want HAProxy to just forward encrypted traffic without decrypting it (backend handles SSL):

```haproxy
frontend tcp_front
    bind *:443
    mode tcp                        # TCP mode — no HTTP processing!
    
    # Use SNI to route (since we can't see HTTP headers)
    tcp-request inspect-delay 5s
    tcp-request content accept if { req_ssl_hello_type 1 }
    
    acl is_api req_ssl_sni -i api.example.com
    acl is_web req_ssl_sni -i www.example.com
    
    use_backend api_tcp if is_api
    default_backend web_tcp

backend api_tcp
    mode tcp
    server api1 192.168.1.10:443 check

backend web_tcp
    mode tcp
    server web1 192.168.1.20:443 check
```

## 15. Stick Tables & Session Persistence

### What are Stick Tables?

Stick tables are HAProxy's **in-memory key-value stores**. They're used for:
- **Session persistence** (sticky sessions — send user to same server)
- **Rate limiting** (count requests per IP)
- **Tracking** (connection counts, error rates)
- **Abuse detection** (identify bad actors)

Think of them as a small, fast database that HAProxy keeps in memory.

### Session Persistence with Cookies

The most reliable method for sticky sessions:

```haproxy
backend web_servers
    balance roundrobin
    
    # Insert a cookie named SERVERID
    cookie SERVERID insert indirect nocache
    
    # Each server gets a unique cookie value
    server web1 192.168.1.10:8080 check cookie s1
    server web2 192.168.1.11:8080 check cookie s2
    server web3 192.168.1.12:8080 check cookie s3
```

**How it works:**
1. User's first request → HAProxy picks a server (e.g., web2) using round-robin
2. HAProxy adds `Set-Cookie: SERVERID=s2` to the response
3. User's next request includes `Cookie: SERVERID=s2`
4. HAProxy reads the cookie and sends the request directly to web2
5. The user always goes to web2 until the cookie expires or the server goes down

**Cookie options:**

| Option | Meaning |
|--------|---------|
| `insert` | HAProxy adds the cookie (not the server) |
| `indirect` | Remove the cookie before forwarding to the server |
| `nocache` | Tell caches not to cache this response (has Set-Cookie) |
| `prefix` | Prefix existing cookie instead of adding a new one |
| `httponly` | Set HttpOnly flag on the cookie |
| `secure` | Set Secure flag on the cookie |

### Session Persistence with Stick Tables

```haproxy
backend web_servers
    balance roundrobin
    
    # Create a stick table: track source IPs for 30 minutes
    stick-table type ip size 200k expire 30m
    
    # Stick on source IP
    stick on src
    
    server web1 192.168.1.10:8080 check
    server web2 192.168.1.11:8080 check
    server web3 192.168.1.12:8080 check
```

**How it works:**
1. User with IP `1.2.3.4` connects
2. HAProxy picks web2 using round-robin
3. HAProxy stores `1.2.3.4 → web2` in the stick table
4. Next time `1.2.3.4` connects, HAProxy looks up the table and sends to web2

### Stick Table Types

| Type | Key Size | Use Case |
|------|---------|----------|
| `ip` | 4 bytes | Track by IPv4 address |
| `ipv6` | 16 bytes | Track by IPv6 address |
| `integer` | 4 bytes | Track by a number |
| `string len N` | N bytes | Track by a string (URL, cookie, etc.) |
| `binary len N` | N bytes | Track by binary data |

## 16. Rate Limiting with Stick Tables

This is one of HAProxy's **killer features** — sophisticated rate limiting without any external tools.

### Basic Rate Limiting (by requests per second)

```haproxy
frontend http_front
    bind *:80
    
    # Define a stick table to track request rates per IP
    stick-table type ip size 100k expire 30s store http_req_rate(10s)
    
    # Track each request in the table
    http-request track-sc0 src
    
    # Deny if more than 100 requests in 10 seconds
    http-request deny deny_status 429 if { sc_http_req_rate(0) gt 100 }
    
    default_backend web_servers
```

**Breaking this down:**

| Part | Meaning |
|------|---------|
| `type ip` | Key is client IP address |
| `size 100k` | Store up to 100,000 IPs |
| `expire 30s` | Remove entries not seen for 30 seconds |
| `store http_req_rate(10s)` | Track the HTTP request rate over 10 seconds |
| `track-sc0 src` | Use sticky counter 0, keyed on source IP |
| `sc_http_req_rate(0) gt 100` | If counter 0's request rate > 100 |
| `deny deny_status 429` | Return 429 Too Many Requests |

### Multi-Level Rate Limiting

```haproxy
frontend http_front
    bind *:80
    
    # Table 1: Track request rate (requests per second)
    stick-table type ip size 100k expire 30s store http_req_rate(10s),conn_cur,conn_rate(10s)
    
    # Track
    http-request track-sc0 src
    
    # Level 1: More than 200 requests in 10s → block completely
    http-request deny deny_status 429 if { sc_http_req_rate(0) gt 200 }
    
    # Level 2: More than 100 requests in 10s → add a delay (tarpit)
    http-request tarpit deny_status 429 if { sc_http_req_rate(0) gt 100 }
    
    # Level 3: More than 20 concurrent connections → block
    http-request deny deny_status 429 if { sc_conn_cur(0) gt 20 }
    
    default_backend web_servers
```

### Rate Limiting Specific Endpoints

```haproxy
frontend http_front
    bind *:80
    
    # General rate limit table
    stick-table type ip size 100k expire 30s store http_req_rate(10s)
    
    # Login-specific rate limit table (stricter)
    # Note: using a backend's stick table since frontends can only have one
    
    http-request track-sc0 src
    
    # Strict limit for login endpoint
    acl is_login path_beg /login /auth
    http-request deny deny_status 429 if is_login { sc_http_req_rate(0) gt 10 }
    
    # Normal limit for everything else
    http-request deny deny_status 429 if { sc_http_req_rate(0) gt 100 }
    
    default_backend web_servers

# You can also use a backend's stick table for separate tracking
backend rate_limit_login
    stick-table type ip size 50k expire 60s store http_req_rate(60s)

frontend http_front_v2
    bind *:80
    
    # Track in frontend's table
    http-request track-sc0 src
    
    # Track login attempts in the backend's table
    acl is_login path_beg /login
    http-request track-sc1 src table rate_limit_login if is_login
    
    # 5 login attempts per minute max
    http-request deny deny_status 429 if is_login { sc_http_req_rate(1,rate_limit_login) gt 5 }
    
    # 100 requests per 10 seconds for general traffic
    http-request deny deny_status 429 if { sc_http_req_rate(0) gt 100 }
    
    default_backend web_servers
```

### Stick Table Data You Can Track

| Counter | What It Tracks |
|---------|---------------|
| `http_req_rate(period)` | HTTP request rate over period |
| `http_req_cnt` | Total HTTP request count |
| `http_err_rate(period)` | HTTP error rate (4xx/5xx) |
| `http_err_cnt` | Total HTTP error count |
| `conn_cur` | Current concurrent connections |
| `conn_rate(period)` | Connection rate over period |
| `conn_cnt` | Total connection count |
| `sess_rate(period)` | Session rate over period |
| `bytes_in_rate(period)` | Incoming bytes rate |
| `bytes_out_rate(period)` | Outgoing bytes rate |
| `gpc0` | General purpose counter 0 |
| `gpc0_rate(period)` | Rate of gpc0 increments |

### Viewing Stick Table Contents

```bash
# View all entries in a stick table
echo "show table http_front" | sudo socat stdio /var/run/haproxy/admin.sock

# Output:
# table: http_front, type: ip, size:102400, used:3
# 0x55a1234: key=192.168.1.100 use=1 exp=25432 http_req_rate(10000)=15 conn_cur=2
# 0x55a5678: key=10.0.0.50 use=1 exp=28000 http_req_rate(10000)=3 conn_cur=1

# Clear a specific entry
echo "clear table http_front key 192.168.1.100" | sudo socat stdio /var/run/haproxy/admin.sock

# Clear all entries
echo "clear table http_front" | sudo socat stdio /var/run/haproxy/admin.sock
```

```
local0.* /var/log/haproxy/haproxy.log

& stop
```

```bash
# Create log directory
sudo mkdir -p /var/log/haproxy

# Restart rsyslog
sudo systemctl restart rsyslog

# Restart HAProxy
sudo systemctl restart haproxy
```

### HAProxy Log Configuration

```haproxy
global
    log /dev/log local0
    log /dev/log local1 notice

defaults
    log global
    option httplog              # Detailed HTTP logging
    option dontlognull          # Skip logging empty connections

    # Custom log format
    log-format "%ci:%cp [%tr] %ft %b/%s %TR/%Tw/%Tc/%Tr/%Ta %ST %B %CC %CS %tsc %ac/%fc/%bc/%sc/%rc %sq/%bq %hr %hs %{+Q}r"
```

### Understanding HAProxy Log Fields

A typical HAProxy log line:
```

```

| Field | Value | Meaning |
|-------|-------|---------|
| Client IP:Port | `192.168.1.100:54321` | Who connected |
| Timestamp | `[29/May/2026:14:30:00.123]` | When |
| Frontend | `http_front` | Which frontend received it |
| Backend/Server | `web_servers/web1` | Which backend and server handled it |
| Timers `Tq/Tw/Tc/Tr/Ta` | `0/0/1/5/6` | Request/Queue/Connect/Response/Total time (ms) |
| Status code | `200` | HTTP response code |
| Bytes | `1234` | Response size |
| Termination state | `----` | How the connection ended |
| Connections | `100/50/25/10/0` | active/frontend/backend/server/retries |
| Queue | `0/0` | Server queue/Backend queue |
| Request | `GET /api/users HTTP/1.1` | The actual request |

**Timer fields are the most important for debugging:**

| Timer | Full Name | What It Measures |
|-------|-----------|-----------------|
| `TR` | Total Request | Time for client to send the full request |
| `Tw` | Wait in Queue | Time the request waited in queue |
| `Tc` | Connect | Time to establish TCP connection to backend |
| `Tr` | Response | Time for backend to send response headers |
| `Ta` | Total Active | Total time from accept to close |

## 19. High Availability — HAProxy with Keepalived

A single HAProxy is a **single point of failure**. If it goes down, everything goes down. The solution: **two HAProxy instances with Keepalived** for failover.

```
                     Internet
                        │
                   Virtual IP
                  192.168.1.100
                        │
          ┌─────────────┼─────────────┐
          │                           │
    ┌─────▼──────┐            ┌──────▼─────┐
    │ HAProxy 1  │            │ HAProxy 2  │
    │  MASTER    │◄──VRRP───►│  BACKUP    │
    │ 192.168.1.1│            │ 192.168.1.2│
    └─────┬──────┘            └──────┬─────┘
          │                          │
          └──────────┬───────────────┘
                     │
              ┌──────┼──────┐
              ▼      ▼      ▼
            [Web1] [Web2] [Web3]
```

**How it works:**
1. Both HAProxy servers run **Keepalived**
2. They share a **Virtual IP (VIP)** — `192.168.1.100`
3. The MASTER holds the VIP and serves traffic
4. If MASTER dies, BACKUP takes over the VIP in seconds
5. Clients always connect to the VIP — they never notice the failover

### Install and Configure Keepalived

```bash
sudo apt install keepalived -y
```

**On MASTER (HAProxy 1):**

```conf
# /etc/keepalived/keepalived.conf

vrrp_script check_haproxy {
    script "/usr/bin/killall -0 haproxy"     # Check if HAProxy is running
    interval 2                                # Check every 2 seconds
    weight 2                                  # Add 2 to priority if healthy
}

vrrp_instance VI_1 {
    state MASTER                     # This is the master
    interface eth0                   # Network interface
    virtual_router_id 51             # Must be same on both servers
    priority 101                     # Higher priority = master (100+weight=102 when healthy)

    authentication {
        auth_type PASS
        auth_pass mysecret           # Must match on both servers
    }

    virtual_ipaddress {
        192.168.1.100/24             # The Virtual IP
    }

    track_script {
        check_haproxy
    }
}
```

**On BACKUP (HAProxy 2):**

```conf
# /etc/keepalived/keepalived.conf

vrrp_script check_haproxy {
    script "/usr/bin/killall -0 haproxy"
    interval 2
    weight 2
}

vrrp_instance VI_1 {
    state BACKUP                     # This is the backup
    interface eth0
    virtual_router_id 51             # Same as master!
    priority 100                     # Lower priority

    authentication {
        auth_type PASS
        auth_pass mysecret           # Same as master!
    }

    virtual_ipaddress {
        192.168.1.100/24             # Same VIP
    }

    track_script {
        check_haproxy
    }
}
```

```bash
# Start Keepalived on both servers
sudo systemctl start keepalived
sudo systemctl enable keepalived

# Verify VIP is on master
ip addr show eth0 | grep 192.168.1.100

# Test failover: stop HAProxy on master
sudo systemctl stop haproxy
# VIP should move to backup within ~2 seconds!

# Verify on backup:
ip addr show eth0 | grep 192.168.1.100
```

## 20. Production-Ready Configuration

Here's a complete, production-grade HAProxy configuration:

```haproxy
# /etc/haproxy/haproxy.cfg
# Production-Ready HAProxy Configuration

# ==========================================
# GLOBAL
# ==========================================
global
    log /dev/log local0
    log /dev/log local1 notice

    chroot /var/lib/haproxy
    user haproxy
    group haproxy
    daemon

    # Performance
    maxconn 100000
    nbthread 4

    # SSL
    ssl-default-bind-ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384
    ssl-default-bind-ciphersuites TLS_AES_128_GCM_SHA256:TLS_AES_256_GCM_SHA384
    ssl-default-bind-options ssl-min-ver TLSv1.2 no-tls-tickets
    tune.ssl.default-dh-param 2048

    # Runtime management
    stats socket /var/run/haproxy/admin.sock mode 660 level admin expose-fd listeners
    stats timeout 30s

# ==========================================
# DEFAULTS
# ==========================================
defaults
    log global
    mode http
    option httplog
    option dontlognull
    option http-server-close
    option forwardfor except 127.0.0.0/8
    option redispatch
    retries 3

    # Timeouts
    timeout connect     5s
    timeout client      30s
    timeout server      30s
    timeout http-request 10s
    timeout http-keep-alive 10s
    timeout queue       30s
    timeout check       5s
    timeout tunnel      1h

    # Error pages
    errorfile 400 /etc/haproxy/errors/400.http
    errorfile 403 /etc/haproxy/errors/403.http
    errorfile 408 /etc/haproxy/errors/408.http
    errorfile 500 /etc/haproxy/errors/500.http
    errorfile 502 /etc/haproxy/errors/502.http
    errorfile 503 /etc/haproxy/errors/503.http
    errorfile 504 /etc/haproxy/errors/504.http

# ==========================================
# STATS DASHBOARD
# ==========================================
listen stats
    bind 127.0.0.1:8404
    mode http
    stats enable
    stats uri /stats
    stats refresh 10s
    stats show-legends
    stats show-node
    stats auth admin:YourSecurePassword123

# ==========================================
# FRONTEND — HTTP (redirect to HTTPS)
# ==========================================
frontend http_front
    bind *:80

    # Rate limiting table
    stick-table type ip size 100k expire 30s store http_req_rate(10s),conn_cur

    # Track requests
    http-request track-sc0 src

    # Block abusive IPs (more than 200 req/10s)
    http-request deny deny_status 429 if { sc_http_req_rate(0) gt 200 }

    # Block too many concurrent connections (more than 50)
    http-request deny deny_status 429 if { sc_conn_cur(0) gt 50 }

    # Redirect to HTTPS
    http-request redirect scheme https code 301

# ==========================================
# FRONTEND — HTTPS (main entry point)
# ==========================================
frontend https_front
    bind *:443 ssl crt /etc/haproxy/certs/ alpn h2,http/1.1

    # Rate limiting (separate table for HTTPS)
    stick-table type ip size 100k expire 30s store http_req_rate(10s),conn_cur
    http-request track-sc0 src

    # Block abusive IPs
    http-request deny deny_status 429 if { sc_http_req_rate(0) gt 200 }

    # ==========================================
    # SECURITY HEADERS
    # ==========================================
    http-response set-header X-Frame-Options SAMEORIGIN
    http-response set-header X-Content-Type-Options nosniff
    http-response set-header X-XSS-Protection "1; mode=block"
    http-response set-header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload"
    http-response set-header Referrer-Policy strict-origin-when-cross-origin
    http-response del-header Server

    # ==========================================
    # ACLs
    # ==========================================
    acl is_api path_beg /api/
    acl is_static path_end .css .js .png .jpg .jpeg .gif .svg .ico .woff .woff2
    acl is_websocket hdr(Upgrade) -i websocket
    acl is_health path /health /ready /live

    # ==========================================
    # ROUTING
    # ==========================================
    use_backend api_servers if is_api
    use_backend static_servers if is_static
    use_backend websocket_servers if is_websocket
    use_backend health_servers if is_health
    default_backend web_servers

# ==========================================
# BACKEND — Web Application
# ==========================================
backend web_servers
    balance leastconn
    cookie SERVERID insert indirect nocache httponly secure

    option httpchk GET /health
    http-check expect status 200

    http-request set-header X-Forwarded-For %[src]
    http-request set-header X-Forwarded-Proto https
    http-request set-header X-Real-IP %[src]

    server web1 192.168.1.10:8080 check inter 5s fall 3 rise 2 cookie s1 maxconn 200
    server web2 192.168.1.11:8080 check inter 5s fall 3 rise 2 cookie s2 maxconn 200
    server web3 192.168.1.12:8080 check inter 5s fall 3 rise 2 cookie s3 maxconn 200

# ==========================================
# BACKEND — API Servers
# ==========================================
backend api_servers
    balance leastconn

    option httpchk GET /api/health
    http-check expect status 200

    # Rate limit login endpoint (5 per minute per IP)
    stick-table type ip size 50k expire 60s store http_req_rate(60s)
    acl is_login path_beg /api/login /api/auth
    http-request track-sc1 src if is_login
    http-request deny deny_status 429 if is_login { sc_http_req_rate(1) gt 5 }

    http-request set-header X-Forwarded-For %[src]
    http-request set-header X-Forwarded-Proto https
    http-request set-header X-Real-IP %[src]

    server api1 192.168.1.20:3000 check inter 5s fall 3 rise 2 maxconn 300
    server api2 192.168.1.21:3000 check inter 5s fall 3 rise 2 maxconn 300

# ==========================================
# BACKEND — Static Content
# ==========================================
backend static_servers
    balance roundrobin

    http-response set-header Cache-Control "public, max-age=2592000, immutable"

    server static1 192.168.1.30:80 check
    server static2 192.168.1.31:80 check

# ==========================================
# BACKEND — WebSocket
# ==========================================
backend websocket_servers
    balance source

    timeout tunnel 1h
    timeout server 1h

    server ws1 192.168.1.40:8080 check
    server ws2 192.168.1.41:8080 check

# ==========================================
# BACKEND — Health Check (internal)
# ==========================================
backend health_servers
    mode http
    http-request return status 200 content-type "text/plain" string "OK"
```

### Troubleshooting Guide

| Problem | Likely Cause | Solution |
|---------|-------------|----------|
| `503 Service Unavailable` | All backends are DOWN | Check backend health: `echo "show servers state" \| socat ...` |
| `502 Bad Gateway` | Backend is returning errors | Check backend logs, verify backend is running |
| `408 Request Timeout` | Client too slow sending request | Increase `timeout http-request` |
| `504 Gateway Timeout` | Backend too slow responding | Increase `timeout server` |
| Connection refused | HAProxy not binding correctly | Check `bind` address/port, check firewall |
| Config won't load | Syntax error | Run `haproxy -c -f /etc/haproxy/haproxy.cfg` |
| Health checks failing | Wrong health check endpoint | Verify `option httpchk` path and expected response |
| High connection queue | Too many requests, not enough `maxconn` | Increase `maxconn` on servers and global |
| SSL errors | Certificate format/path wrong | Ensure PEM has cert+key+chain, check permissions |
| Sticky sessions not working | Cookie configuration issue | Check `cookie` directive in backend and server lines |
| "No server is available" in logs | All servers down or at `maxconn` | Add servers, increase `maxconn`, or add `option allbackups` |

### Performance Debugging

```bash
# Check current connections and limits
echo "show info" | sudo socat stdio /var/run/haproxy/admin.sock | grep -i "conn\|sess\|max"

# Key metrics to watch:
# CurrConns: Current active connections
# MaxConn: Maximum allowed (from global maxconn)
# ConnRate: Connections per second
# SessRate: Sessions per second
# MaxSessRate: Peak session rate ever

# Check per-backend stats
echo "show stat" | sudo socat stdio /var/run/haproxy/admin.sock | cut -d',' -f1,2,5,8,33,34,36 | column -s',' -t

# Check for queued requests
echo "show stat" | sudo socat stdio /var/run/haproxy/admin.sock | grep -v "^#" | awk -F',' '{if($3>0) print $1"/"$2": qcur="$3" scur="$5}'
```

### Zero-Downtime Reload

HAProxy supports **hitless reloads** — update the config without dropping a single connection:

```bash
# Method 1: systemctl reload (uses -sf flag internally)
sudo systemctl reload haproxy

# Method 2: Manual hitless reload
haproxy -f /etc/haproxy/haproxy.cfg -sf $(pidof haproxy)
# -sf = "soft finish" — old process finishes current connections, new process takes over

# Method 3: Using the stats socket (HAProxy 2.0+)
echo "show proc" | sudo socat stdio /var/run/haproxy/admin.sock
```
