## What is Traefik?

Imagine you have a huge office building with dozens of companies inside. Every company has a different name on their door. When a visitor arrives at the lobby, the **receptionist** reads the visitor's pass, checks which company they want to visit, and directs them to the correct floor and room — **automatically**.

**Traefik is that receptionist — but for your applications and microservices.**

Traefik (pronounced "traffic") is an **open-source, cloud-native reverse proxy and load balancer** designed to make deploying and routing traffic to microservices **ridiculously easy**.

### One-Line Definition

> Traefik is a modern HTTP reverse proxy and load balancer that **automatically discovers** your services and configures itself — no manual config file changes needed.

---

## Why Traefik? What Problem Does It Solve?

Let's think about the **traditional way** with Nginx or HAProxy:

1. You deploy a new microservice
2. You SSH into your server
3. You manually edit the Nginx/HAProxy config file
4. You add a new `server` block or `backend` entry
5. You reload the config
6. Repeat every time you add, remove, or change a service

**This works fine for 2-3 services. But what about 50? 100?** In the world of containers and Kubernetes, services come and go every minute. You can't keep editing config files manually.

### Traefik's Superpower: Auto-Discovery

Traefik **watches your infrastructure** (Docker, Kubernetes, etc.) and **automatically detects** when services appear, disappear, or change. It updates its routing rules in **real-time** — with **zero manual intervention**.

| Traditional (Nginx/HAProxy) | Traefik |
|-----------------------------|---------|
| Manually edit config files | Automatic service discovery |
| Restart/reload after changes | Real-time updates, no restarts |
| Static configuration | Dynamic configuration |
| Config becomes complex at scale | Stays simple at any scale |
| SSL certificates — manual setup | Auto SSL via Let's Encrypt built-in |
| Dashboard — separate tool needed | Built-in web dashboard |

## Core Concepts — The Traefik Mental Model

Before we jump into code, let's understand how Traefik thinks. Traefik has a very clean architecture built around a few key concepts:

```
                    INTERNET
                       │
                       ▼
              ┌────────────────┐
              │  ENTRYPOINTS   │  ◄── "Where traffic enters" (ports)
              │  :80   :443    │
              └───────┬────────┘
                      │
                      ▼
              ┌────────────────┐
              │    ROUTERS     │  ◄── "Where should this request go?" (rules)
              │  Host, Path,   │
              │  Headers match │
              └───────┬────────┘
                      │
                      ▼
              ┌────────────────┐
              │  MIDDLEWARES   │  ◄── "Do something to the request first" (optional)
              │  Auth, Rate    │
              │  Limit, Strip  │
              └───────┬────────┘
                      │
                      ▼
              ┌────────────────┐
              │   SERVICES     │  ◄── "The actual backend application"
              │  app:8080      │
              │  api:3000      │
              └────────────────┘
```

### 1. Entrypoints

An **Entrypoint** is the network entry point into Traefik — basically the **port** that Traefik listens on.

- Port `80` for HTTP traffic
- Port `443` for HTTPS traffic
- You can define any custom port

Think of it as the **front door** of the building.

### 2. Routers

A **Router** is the brain that decides **where a request should go**. It looks at the incoming request and matches it against rules:

- "If the domain is `api.example.com`, send it to the API service"
- "If the path starts with `/blog`, send it to the blog service"
- "If the header says `X-Custom: v2`, send it to the v2 service"

Think of it as the **signboard at the lobby** — it tells you which floor to go to.

### 3. Middlewares

**Middlewares** sit between the router and the service. They can **modify the request or response** before it reaches the backend. Examples:

- Add authentication (BasicAuth, OAuth)
- Add rate limiting
- Strip a path prefix (`/api/users` → `/users`)
- Add security headers
- Redirect HTTP to HTTPS
- Compress responses

Think of it as the **security checkpoint** — before you enter the office, you show your ID, get a badge, go through screening.

### 4. Services

A **Service** is the actual backend application that handles the request. Traefik load balances across multiple instances of a service.

Think of it as the **actual office** where the meeting happens.

## How Traefik Discovers Services — Providers

Traefik uses the concept of **Providers** to discover services. A provider is a source of configuration — it tells Traefik what services exist and how to route to them.

| Provider | What It Watches |
|----------|----------------|
| **Docker** | Docker containers with labels |
| **Kubernetes** | Kubernetes Ingress resources and IngressRoute CRDs |
| **Docker Swarm** | Swarm services |
| **Consul Catalog** | Services registered in Consul |
| **Etcd** | Key-value store |
| **File** | Static YAML/TOML config files |
| **Marathon / Mesos** | Marathon apps |
| **Rancher** | Rancher services |

The most common providers are **Docker** and **Kubernetes**.

## Traefik Configuration — Two Layers

Traefik has **two types of configuration**:

### 1. Static Configuration

This is the **startup configuration** — it defines things that don't change often:
- Entrypoints (what ports to listen on)
- Providers (where to discover services)
- Logging settings
- API/Dashboard settings

This is set **once** and requires a restart to change.

**Where you can define it:**
- `traefik.yml` or `traefik.toml` file
- Command-line arguments
- Environment variables

### 2. Dynamic Configuration

This is the **runtime configuration** — it defines things that change frequently:
- Routers (routing rules)
- Middlewares (request processing)
- Services (backends)
- TLS certificates

This is updated **automatically** by providers — no restart needed!

## Hands-On: Traefik with Docker — Complete Example

Let's build a real setup step by step. We'll run Traefik as a reverse proxy in front of multiple services using Docker Compose.

### Step 1: Create the Project Structure

```
traefik-demo/
├── docker-compose.yml
└── traefik/
    └── traefik.yml
```

### Step 2: Create the Static Configuration — `traefik/traefik.yml`

```yaml
# traefik/traefik.yml

# ===========================================
# STATIC CONFIGURATION
# ===========================================

# --- API & Dashboard ---
api:
  dashboard: true          # Enable the web dashboard
  insecure: true           # Allow access without HTTPS (dev only!)
  # In production, NEVER set insecure: true
  # Use proper authentication instead

# --- Entrypoints ---
entryPoints:
  web:                     # Name for HTTP entrypoint
    address: ":80"         # Listen on port 80
  websecure:               # Name for HTTPS entrypoint
    address: ":443"        # Listen on port 443

# --- Providers ---
providers:
  docker:
    endpoint: "unix:///var/run/docker.sock"   # Connect to Docker socket
    exposedByDefault: false                    # Don't expose containers unless explicitly told
    # This is important! Without this, every container would be exposed
    watch: true                                # Watch for container changes in real-time

# --- Logging ---
log:
  level: INFO              # DEBUG, INFO, WARN, ERROR
  # level: DEBUG           # Use this during development for more details

accessLog: {}              # Enable access logging (like Nginx access.log)
```

**Let's break down each section:**

| Section | What It Does |
|---------|-------------|
| `api.dashboard: true` | Turns on the beautiful web dashboard where you can see all routes |
| `api.insecure: true` | Allows accessing the dashboard on port 8080 without authentication. **Never do this in production!** |
| `entryPoints.web` | Creates an entrypoint named "web" listening on port 80 (HTTP) |
| `entryPoints.websecure` | Creates an entrypoint named "websecure" listening on port 443 (HTTPS) |
| `providers.docker` | Tells Traefik to watch Docker for services |
| `exposedByDefault: false` | Only expose containers that have `traefik.enable=true` label |
| `watch: true` | Automatically detect when containers start/stop |

### Step 3: Create the Docker Compose File — `docker-compose.yml`

```yaml
# docker-compose.yml

services:
  # ================================
  # TRAEFIK - The Reverse Proxy
  # ================================
  traefik:
    image: traefik:v3.1
    container_name: traefik
    restart: unless-stopped
    ports:
      - "80:80"        # HTTP
      - "443:443"      # HTTPS
      - "8080:8080"    # Dashboard (dev only!)
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock:ro   # Read-only access to Docker socket
      - ./traefik/traefik.yml:/etc/traefik/traefik.yml  # Static config
    networks:
      - web

  # ================================
  # SERVICE 1 - A Simple Web App
  # ================================
  whoami-app:
    image: traefik/whoami        # Simple HTTP service that returns request info
    container_name: whoami-app
    restart: unless-stopped
    labels:
      - "traefik.enable=true"                                             # Tell Traefik to route to this container

      # --- Router ---
      - "traefik.http.routers.whoami.rule=Host(`whoami.localhost`)"        # Route based on hostname
      - "traefik.http.routers.whoami.entrypoints=web"                     # Listen on HTTP port

      # --- Service ---
      - "traefik.http.services.whoami.loadbalancer.server.port=80"        # The container's internal port
    networks:
      - web

  # ================================
  # SERVICE 2 - A Blog App
  # ================================
  blog:
    image: traefik/whoami        # Using whoami as a stand-in for a blog app
    container_name: blog
    restart: unless-stopped
    labels:
      - "traefik.enable=true"
      - "traefik.http.routers.blog.rule=Host(`blog.localhost`)"
      - "traefik.http.routers.blog.entrypoints=web"
      - "traefik.http.services.blog.loadbalancer.server.port=80"
    networks:
      - web

  # ================================
  # SERVICE 3 - An API Service
  # ================================
  api:
    image: traefik/whoami
    container_name: api
    restart: unless-stopped
    labels:
      - "traefik.enable=true"
      - "traefik.http.routers.api.rule=Host(`api.localhost`)"
      - "traefik.http.routers.api.entrypoints=web"
      - "traefik.http.services.api.loadbalancer.server.port=80"
    networks:
      - web

networks:
  web:
    name: web
```

### Understanding the Docker Labels — This is Where the Magic Happens!

Traefik reads **Docker labels** on your containers to automatically configure routing. No separate config file needed! Let's decode each label:

```yaml
# "Hey Traefik, please handle routing for this container"
- "traefik.enable=true"

# "Create a router named 'whoami' that matches requests where
#  the Host header equals 'whoami.localhost'"
- "traefik.http.routers.whoami.rule=Host(`whoami.localhost`)"

# "This router should listen on the 'web' entrypoint (port 80)"
- "traefik.http.routers.whoami.entrypoints=web"

# "The actual service inside this container runs on port 80"
- "traefik.http.services.whoami.loadbalancer.server.port=80"
```

The label format follows this pattern:

```
traefik.http.routers.<router-name>.<property>=<value>
traefik.http.services.<service-name>.<property>=<value>
traefik.http.middlewares.<middleware-name>.<property>=<value>
```

## Routing Rules — How Traefik Decides Where to Send Traffic

Traefik routers use **rules** to match incoming requests. Here are all the available matchers:

### Host-Based Routing

```yaml
# Route based on the domain name
- "traefik.http.routers.myapp.rule=Host(`myapp.example.com`)"

# Multiple hosts
- "traefik.http.routers.myapp.rule=Host(`myapp.example.com`) || Host(`www.myapp.example.com`)"
```

### Path-Based Routing

```yaml
# Route based on URL path
- "traefik.http.routers.api.rule=PathPrefix(`/api`)"

# Exact path match
- "traefik.http.routers.health.rule=Path(`/health`)"
```

### Combined Rules

```yaml
# Host AND Path
- "traefik.http.routers.api.rule=Host(`example.com`) && PathPrefix(`/api`)"

# Host AND specific headers
- "traefik.http.routers.v2.rule=Host(`example.com`) && Headers(`X-Version`, `v2`)"
```

### All Available Rule Matchers

| Matcher | Example | What It Matches |
|---------|---------|----------------|
| `Host()` | `Host(\`example.com\`)` | Request domain name |
| `HostRegexp()` | `HostRegexp(\`{subdomain:.+}.example.com\`)` | Domain with regex |
| `Path()` | `Path(\`/exact/path\`)` | Exact URL path |
| `PathPrefix()` | `PathPrefix(\`/api\`)` | URL path starts with |
| `Method()` | `Method(\`GET\`, \`POST\`)` | HTTP method |
| `Headers()` | `Headers(\`X-Custom\`, \`value\`)` | Request headers |
| `Query()` | `Query(\`token=abc\`)` | Query parameters |
| `ClientIP()` | `ClientIP(\`192.168.1.0/24\`)` | Source IP address |

You can combine these with `&&` (AND) and `||` (OR).

## Middlewares — Processing Requests Before They Hit Your App

Middlewares are one of Traefik's most powerful features. They let you **modify requests and responses** without changing your application code.

### Redirect HTTP to HTTPS

```yaml
labels:
  # Create a middleware that redirects HTTP → HTTPS
  - "traefik.http.middlewares.redirect-to-https.redirectscheme.scheme=https"
  - "traefik.http.middlewares.redirect-to-https.redirectscheme.permanent=true"

  # Apply the middleware to a router
  - "traefik.http.routers.myapp-http.middlewares=redirect-to-https"
```

**What's happening here?**
- We create a middleware named `redirect-to-https`
- It uses the `redirectscheme` type
- It redirects to HTTPS with a 301 (permanent) redirect
- We attach this middleware to the HTTP router

### Basic Authentication

```yaml
labels:
  # Create a BasicAuth middleware
  # Password is generated using: htpasswd -nb admin secretpassword
  - "traefik.http.middlewares.auth.basicauth.users=admin:$$apr1$$xyz$$hashedpassword"

  # Apply it to a router
  - "traefik.http.routers.dashboard.middlewares=auth"
```

**Note:** In Docker Compose labels, you need to escape `$` with `$$`.

To generate the password hash:
```bash
# Install htpasswd (part of apache2-utils)
sudo apt-get install apache2-utils

# Generate the hash
htpasswd -nb admin yourpassword
# Output: admin:$apr1$ruca84Hq$mbjdMZBAG.KWn7vfN/SNK/
```

### Rate Limiting

```yaml
labels:
  # Allow max 100 requests per second, with a burst of 50
  - "traefik.http.middlewares.rate-limit.ratelimit.average=100"
  - "traefik.http.middlewares.rate-limit.ratelimit.burst=50"
  - "traefik.http.middlewares.rate-limit.ratelimit.period=1s"

  # Apply it
  - "traefik.http.routers.api.middlewares=rate-limit"
```

### Strip Path Prefix

Sometimes your backend app doesn't know about the path prefix. For example, requests come to `/api/users` but your app expects just `/users`.

```yaml
labels:
  # Strip the /api prefix before sending to backend
  - "traefik.http.middlewares.strip-api.stripprefix.prefixes=/api"

  # Apply it
  - "traefik.http.routers.api.middlewares=strip-api"
```

**What happens:**
```
Client request:  GET /api/users  →  Traefik strips /api  →  Backend receives: GET /users
```

### Adding Security Headers

```yaml
labels:
  - "traefik.http.middlewares.security-headers.headers.framedeny=true"
  - "traefik.http.middlewares.security-headers.headers.browserxssfilter=true"
  - "traefik.http.middlewares.security-headers.headers.contenttypenosniff=true"
  - "traefik.http.middlewares.security-headers.headers.stsincludessubdomains=true"
  - "traefik.http.middlewares.security-headers.headers.stsseconds=31536000"

  - "traefik.http.routers.myapp.middlewares=security-headers"
```

### Chaining Multiple Middlewares

You can apply multiple middlewares to a single router:

```yaml
labels:
  - "traefik.http.routers.myapp.middlewares=redirect-to-https,rate-limit,security-headers"
```

They execute in the order listed: redirect → rate limit → security headers → backend.

### All Available Middleware Types

| Middleware | Purpose |
|-----------|---------|
| `addprefix` | Adds a path prefix |
| `basicauth` | Basic HTTP authentication |
| `buffering` | Buffers request/response |
| `chain` | Combines multiple middlewares |
| `circuitbreaker` | Stops sending traffic to failing services |
| `compress` | Compresses responses (gzip, brotli) |
| `digestauth` | Digest HTTP authentication |
| `errors` | Custom error pages |
| `forwardauth` | Delegates auth to an external service |
| `headers` | Adds/modifies HTTP headers |
| `ipallowlist` | Only allow specific IPs |
| `ratelimit` | Rate limiting |
| `redirectregex` | Redirect using regex |
| `redirectscheme` | Redirect HTTP → HTTPS |
| `replacepath` | Replaces the request path |
| `replacepathregex` | Replaces path using regex |
| `retry` | Retries failed requests |
| `stripprefix` | Strips a path prefix |
| `stripprefixregex` | Strips path prefix using regex |

## HTTPS with Let's Encrypt — Automatic SSL Certificates

One of Traefik's **killer features** is automatic SSL certificate management with Let's Encrypt. No more manual certificate renewals!

### How It Works

1. Traefik detects a new service that needs HTTPS
2. It automatically requests a certificate from Let's Encrypt
3. Let's Encrypt verifies you own the domain (via HTTP or DNS challenge)
4. Traefik receives the certificate and starts serving HTTPS
5. Traefik automatically renews the certificate before it expires

### Setting Up Let's Encrypt

Update your `traefik.yml`:

```yaml
# traefik/traefik.yml

api:
  dashboard: true

entryPoints:
  web:
    address: ":80"
    http:
      redirections:
        entryPoint:
          to: websecure         # Auto-redirect all HTTP to HTTPS
          scheme: https
  websecure:
    address: ":443"

providers:
  docker:
    endpoint: "unix:///var/run/docker.sock"
    exposedByDefault: false

# --- Let's Encrypt Certificate Resolver ---
certificatesResolvers:
  letsencrypt:                                     # Name of the resolver
    acme:
      email: your-email@example.com                # Your email for Let's Encrypt notifications
      storage: /letsencrypt/acme.json              # Where to store certificates
      httpChallenge:
        entryPoint: web                            # Use HTTP challenge on port 80
      # For wildcard certificates, use DNS challenge instead:
      # dnsChallenge:
      #   provider: cloudflare
      #   resolvers:
      #     - "1.1.1.1:53"
      #     - "8.8.8.8:53"

log:
  level: INFO
```

**What each part does:**

| Config | Purpose |
|--------|---------|
| `certificatesResolvers.letsencrypt` | Creates a certificate resolver named "letsencrypt" |
| `acme.email` | Your email — Let's Encrypt sends renewal notifications here |
| `acme.storage` | File path where certificates are stored (must be persistent!) |
| `httpChallenge.entryPoint: web` | Uses the HTTP-01 challenge — Let's Encrypt verifies domain ownership by making an HTTP request to port 80 |
| `entryPoints.web.http.redirections` | Automatically redirects all HTTP traffic to HTTPS |

Update your Docker Compose:

```yaml
services:
  traefik:
    image: traefik:v3.1
    container_name: traefik
    restart: unless-stopped
    ports:
      - "80:80"
      - "443:443"
      - "8080:8080"
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock:ro
      - ./traefik/traefik.yml:/etc/traefik/traefik.yml
      - ./letsencrypt:/letsencrypt       # Persist certificates!
    networks:
      - web

  myapp:
    image: traefik/whoami
    container_name: myapp
    labels:
      - "traefik.enable=true"

      # HTTP Router — just for redirect
      - "traefik.http.routers.myapp-http.rule=Host(`myapp.example.com`)"
      - "traefik.http.routers.myapp-http.entrypoints=web"

      # HTTPS Router
      - "traefik.http.routers.myapp.rule=Host(`myapp.example.com`)"
      - "traefik.http.routers.myapp.entrypoints=websecure"
      - "traefik.http.routers.myapp.tls=true"                              # Enable TLS
      - "traefik.http.routers.myapp.tls.certresolver=letsencrypt"           # Use our Let's Encrypt resolver

      - "traefik.http.services.myapp.loadbalancer.server.port=80"
    networks:
      - web

networks:
  web:
    name: web
```

That's it! Traefik will automatically get an SSL certificate for `myapp.example.com` and serve it over HTTPS. When the certificate is about to expire, Traefik renews it automatically.

## Load Balancing with Traefik

Traefik automatically load balances when you have multiple instances of the same service.

### Scaling Services with Docker Compose

```yaml
services:
  traefik:
    image: traefik:v3.1
    # ... (same as before)

  webapp:
    image: traefik/whoami
    # No container_name when scaling — Docker assigns unique names
    labels:
      - "traefik.enable=true"
      - "traefik.http.routers.webapp.rule=Host(`webapp.localhost`)"
      - "traefik.http.routers.webapp.entrypoints=web"
      - "traefik.http.services.webapp.loadbalancer.server.port=80"
    networks:
      - web
    deploy:
      replicas: 3            # Run 3 instances of this service
```

```bash
# Start with 3 replicas
docker compose up -d --scale webapp=3

# Test — each request should show a different hostname
curl -H "Host: webapp.localhost" http://localhost
curl -H "Host: webapp.localhost" http://localhost
curl -H "Host: webapp.localhost" http://localhost
# Each response will show a different container ID!
```

Traefik automatically detects all 3 instances and distributes traffic across them using **Round Robin** by default.

### Weighted Load Balancing (Canary Deployments)

You can send a percentage of traffic to different service versions:

```yaml
services:
  app-v1:
    image: myapp:v1
    labels:
      - "traefik.enable=true"
      - "traefik.http.routers.app.rule=Host(`app.localhost`)"
      - "traefik.http.services.app.loadbalancer.server.port=80"
      - "traefik.http.services.app.loadbalancer.weight=90"     # 90% of traffic
    networks:
      - web

  app-v2:
    image: myapp:v2
    labels:
      - "traefik.enable=true"
      - "traefik.http.services.app.loadbalancer.server.port=80"
      - "traefik.http.services.app.loadbalancer.weight=10"     # 10% of traffic
    networks:
      - web
```

This sends 90% of traffic to v1 and 10% to v2 — perfect for **canary deployments** where you test a new version with a small percentage of users first.

### Health Checks

Traefik can check if your backends are healthy:

```yaml
labels:
  - "traefik.http.services.myapp.loadbalancer.healthcheck.path=/health"
  - "traefik.http.services.myapp.loadbalancer.healthcheck.interval=10s"
  - "traefik.http.services.myapp.loadbalancer.healthcheck.timeout=3s"
```

If a backend fails the health check, Traefik stops sending traffic to it until it recovers.

## Traefik with Kubernetes — IngressRoute

Traefik is a **popular Ingress Controller** for Kubernetes. It can replace the default Nginx Ingress Controller.

### Install Traefik in Kubernetes via Helm

```bash
# Add the Traefik Helm repo
helm repo add traefik https://traefik.github.io/charts
helm repo update

# Install
helm install traefik traefik/traefik \
  --namespace traefik \
  --create-namespace \
  --set dashboard.enabled=true
```

### Using Kubernetes Standard Ingress

Traefik supports standard Kubernetes `Ingress` resources:

```yaml
# ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-ingress
  annotations:
    traefik.ingress.kubernetes.io/router.entrypoints: websecure
    traefik.ingress.kubernetes.io/router.tls: "true"
spec:
  rules:
    - host: myapp.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: myapp-service
                port:
                  number: 80
```

### Using Traefik's Custom CRD — IngressRoute (Recommended)

Traefik also provides its own **Custom Resource Definitions (CRDs)** which are more powerful than standard Ingress:

```yaml
# ingressroute.yaml
apiVersion: traefik.io/v1alpha1
kind: IngressRoute
metadata:
  name: myapp-ingressroute
  namespace: default
spec:
  entryPoints:
    - websecure
  routes:
    - match: Host(`myapp.example.com`)
      kind: Rule
      services:
        - name: myapp-service
          port: 80
      middlewares:
        - name: rate-limit
        - name: security-headers
  tls:
    certResolver: letsencrypt
```

### Middleware CRD

```yaml
# middleware.yaml
apiVersion: traefik.io/v1alpha1
kind: Middleware
metadata:
  name: rate-limit
  namespace: default
spec:
  rateLimit:
    average: 100
    burst: 50
---
apiVersion: traefik.io/v1alpha1
kind: Middleware
metadata:
  name: security-headers
  namespace: default
spec:
  headers:
    frameDeny: true
    browserXssFilter: true
    contentTypeNosniff: true
    stsIncludeSubdomains: true
    stsSeconds: 31536000
```

### Why IngressRoute Over Standard Ingress?

| Feature | Standard Ingress | Traefik IngressRoute |
|---------|-----------------|---------------------|
| Middleware support | Limited (annotations) | Full native support |
| TCP/UDP routing | Not supported | Supported |
| Weighted routing | Not supported | Supported |
| TLS passthrough | Limited | Full support |
| Multiple matchers | Limited | Host, Path, Headers, Query, etc. |

## TCP and UDP Routing

Traefik isn't limited to HTTP. It can route raw **TCP and UDP** traffic too.

### TCP Example — Routing to a Database

```yaml
# traefik.yml
entryPoints:

    address: ":3306"

# docker-compose.yml
services:

    image: mysql:8.0

      - "traefik.enable=true"
      - "traefik.tcp.routers.mysql.rule=HostSNI(`*`)"
      - "traefik.tcp.routers.mysql.entrypoints=mysql"
      - "traefik.tcp.services.mysql.loadbalancer.server.port=3306"
```

```yaml
# traefik.yml
entryPoints:

```

## File-Based Dynamic Configuration

Sometimes you want to define routing rules in files instead of Docker labels (e.g., for non-Docker services).

### Static Config — Enable the File Provider

```yaml
# traefik.yml
providers:
  docker:
    endpoint: "unix:///var/run/docker.sock"
    exposedByDefault: false
  file:
    directory: /etc/traefik/dynamic/    # Watch this directory for config files
    watch: true                          # Auto-reload when files change
```

### Dynamic Config File

```yaml
# /etc/traefik/dynamic/services.yml

http:
  routers:
    legacy-app:
      rule: "Host(`legacy.example.com`)"
      entryPoints:
        - websecure
      service: legacy-app
      tls:
        certResolver: letsencrypt

  services:
    legacy-app:
      loadBalancer:
        servers:
          - url: "http://192.168.1.50:8080"    # Route to a non-Docker server
          - url: "http://192.168.1.51:8080"    # Second instance for load balancing
        healthCheck:
          path: /health
          interval: 10s
```

This is useful for routing to **external services** that aren't running in Docker.

## Monitoring & Observability

### Prometheus Metrics

Traefik exposes Prometheus-compatible metrics out of the box:

```yaml
# traefik.yml
metrics:

    addEntryPointsLabels: true
    addServicesLabels: true

      - 0.3
      - 1.2
```

## Production Best Practices

### 1. Never Expose the Dashboard Without Authentication

```yaml
# traefik.yml
api:
  dashboard: true
  # Remove insecure: true !

# Use a middleware for authentication
# In docker-compose:
labels:
  - "traefik.http.routers.dashboard.rule=Host(`traefik.example.com`)"
  - "traefik.http.routers.dashboard.service=api@internal"
  - "traefik.http.routers.dashboard.middlewares=auth"
  - "traefik.http.middlewares.auth.basicauth.users=admin:$$apr1$$xyz$$hashedpass"
```

### 2. Don't Mount Docker Socket Without Read-Only

```yaml
volumes:
  - /var/run/docker.sock:/var/run/docker.sock:ro    # Always :ro (read-only)
```

The Docker socket gives **full control over Docker**. Always mount it read-only. For even better security, consider using a **Docker Socket Proxy** like `tecnativa/docker-socket-proxy`.

### 3. Use Access Logs and Metrics

Always enable access logs and Prometheus metrics in production for debugging and monitoring.

### 4. Persist Let's Encrypt Certificates

```yaml
volumes:
  - ./letsencrypt:/letsencrypt    # Store certificates on a persistent volume
```

If you lose the `acme.json` file, Traefik will need to request new certificates. Let's Encrypt has rate limits, so you could get temporarily blocked.

### 5. Set Resource Limits

```yaml
services:
  traefik:
    deploy:
      resources:
        limits:
          cpus: "1.0"
          memory: 256M
        reservations:
          cpus: "0.25"
          memory: 128M
```

## Traefik vs Nginx vs HAProxy — Quick Comparison

| Feature | Traefik | Nginx | HAProxy |
|---------|---------|-------|---------|
| Auto service discovery | Yes (built-in) | No | No |
| Dynamic config | Yes (real-time) | Requires reload | Requires reload |
| Let's Encrypt built-in | Yes | No (use Certbot) | No |
| Dashboard | Yes (built-in) | No (3rd party) | Yes (stats page) |
| Docker native | Yes | No | No |
| Kubernetes Ingress | Yes (CRD) | Yes | Yes |
| TCP/UDP routing | Yes | Yes | Yes (TCP only) |
| Performance | Good | Excellent | Excellent |
| Middlewares | Rich built-in | Via config | Via config |
| Configuration | YAML/TOML/Labels | Custom syntax | Custom syntax |
| Learning curve | Moderate | Moderate | Steep |
| Best for | Microservices, Docker, K8s | General purpose, high traffic | Advanced L4/L7 LB |

## Key Takeaways

| Point | Detail |
|-------|--------|
| **What is Traefik** | A cloud-native reverse proxy with automatic service discovery |
| **Best for** | Docker, Kubernetes, and microservice architectures |
| **Killer features** | Auto-discovery, dynamic config, built-in Let's Encrypt, dashboard |
| **Configuration** | Docker labels, Kubernetes CRDs, YAML files — no manual reload |
| **Core concepts** | Entrypoints → Routers → Middlewares → Services |
| **Middlewares** | Auth, rate limiting, headers, redirects, circuit breakers — all built-in |
| **vs Nginx/HAProxy** | More automation, less performance; best for dynamic environments |
| **Production tips** | Secure the dashboard, persist certs, use read-only Docker socket, enable metrics |
