## What is Caddy?

Imagine you just moved into a new house and you need to set up electricity. With most providers, you'd need to fill out a bunch of forms, call someone, schedule an appointment, and configure your breaker box. But what if there was a provider that just... **turned the lights on the moment you walked in**? No forms, no calls, no configuration. It just works.

**That's Caddy.**

Caddy is a **modern, open-source web server** written in Go that focuses on making web serving as **simple and secure as possible**. Its biggest claim to fame? **Automatic HTTPS by default** — with zero configuration.

### One-Line Definition

> Caddy is a web server that **automatically provisions and renews HTTPS certificates** for your sites, serves static files, acts as a reverse proxy, and does it all with the **simplest configuration file** you've ever seen.

---

## Why Caddy? What Makes It Special?

Let's be real — we already have Nginx, Apache, HAProxy, and Traefik. So why would anyone use Caddy?

### The Problem Caddy Solves

Setting up HTTPS with other web servers typically looks like this:

```
1. Buy/generate an SSL certificate (or use Let's Encrypt + Certbot)
2. Configure your web server to use the certificate files
3. Set up certificate renewal (cron job or systemd timer)
4. Configure TLS settings (cipher suites, protocols, HSTS)
5. Set up HTTP → HTTPS redirect
6. Hope nothing breaks on renewal day
```

With Caddy, it looks like this:

```
1. Tell Caddy your domain name
2. ...that's it. Caddy does everything else automatically.
```

| Feature | Caddy | Nginx | Apache | Traefik |
|---------|-------|-------|--------|---------|

```bash
# Install required packages
sudo apt install -y debian-keyring debian-archive-keyring apt-transport-https curl
```

## The Caddyfile — The Simplest Web Server Config

The **Caddyfile** is Caddy's configuration format. If you've ever struggled with Nginx or Apache config syntax, the Caddyfile will feel like a breath of fresh air.

### The Simplest Possible Caddyfile

```caddyfile
localhost

respond "Hello, World!"
```

That's it. That's a complete web server config. Run it with:

```bash
caddy run --config Caddyfile
```

Visit `https://localhost` and you'll see "Hello, World!" — served over **HTTPS** with an automatically generated self-signed certificate (since it's localhost).

### Serving a Static Website

```caddyfile
example.com

root * /var/www/mysite
file_server
```

**What happens when you run this:**

1. Caddy starts listening on ports 80 and 443
2. It automatically requests an SSL certificate from Let's Encrypt for `example.com`
3. It redirects all HTTP traffic to HTTPS
4. It serves files from `/var/www/mysite`
5. HTTP/2 is enabled automatically
6. When the cert is about to expire, Caddy renews it automatically

**That's 3 lines of config for what would take 30+ lines in Nginx.**

## Caddyfile Syntax — Complete Guide

### Basic Structure

```caddyfile
# Site address
example.com {
    # Directives go inside the curly braces
    directive1
    directive2
}

# Another site
api.example.com {
    directive1
}
```

If you only have one site, you can skip the curly braces:

```caddyfile
example.com
directive1
directive2
```

### Comments

```caddyfile
# This is a comment
example.com {
    # Comments work anywhere
    file_server  # Even inline!
}
```

### Multiple Site Blocks

```caddyfile
# Site 1
example.com {
    root * /var/www/main
    file_server
}

# Site 2
blog.example.com {
    root * /var/www/blog
    file_server
}

# Site 3
api.example.com {
    reverse_proxy localhost:3000
}
```

Caddy automatically gets separate SSL certificates for each domain!

### Wildcard and Multiple Addresses

```caddyfile
# Multiple domains for the same site
example.com, www.example.com {
    root * /var/www/main
    file_server
}

# All subdomains (wildcard) — requires DNS challenge
*.example.com {
    tls {
        dns cloudflare {env.CF_API_TOKEN}
    }
    reverse_proxy localhost:8080
}
```

## Caddy as a Reverse Proxy — The Most Common Use Case

This is what most DevOps engineers use Caddy for — sitting in front of backend services and handling HTTPS.

### Basic Reverse Proxy

```caddyfile
example.com {
    reverse_proxy localhost:3000
}
```

**That's it.** One line. Caddy will:
- Get an SSL certificate for `example.com`
- Listen on ports 80 and 443
- Redirect HTTP → HTTPS
- Forward all HTTPS requests to your app running on port 3000
- Add proper proxy headers (`X-Forwarded-For`, `X-Forwarded-Proto`, etc.)

### Reverse Proxy to Multiple Backends (Load Balancing)

```caddyfile
example.com {
    reverse_proxy localhost:3001 localhost:3002 localhost:3003
}
```

Caddy load balances across all three backends using **Round Robin** by default.

### Load Balancing Policies

```caddyfile
example.com {
    reverse_proxy localhost:3001 localhost:3002 localhost:3003 {
        lb_policy round_robin       # Default
        # lb_policy least_conn      # Send to least busy server
        # lb_policy random          # Random selection
        # lb_policy first           # Always first available
        # lb_policy ip_hash         # Same client IP → same server
        # lb_policy uri_hash        # Same URI → same server
        # lb_policy header X-Custom # Hash based on a header
    }
}
```

### Health Checks

```caddyfile
example.com {
    reverse_proxy localhost:3001 localhost:3002 {
        health_uri /health          # Path to check
        health_interval 10s         # How often to check
        health_timeout 5s           # Timeout for health check
        health_status 200           # Expected status code
        health_body "ok"            # Expected response body (optional)
    }
}
```

If a backend fails health checks, Caddy stops sending traffic to it until it recovers.

### Path-Based Routing

```caddyfile
example.com {
    # Requests to /api/* go to the API backend
    reverse_proxy /api/* localhost:3000

    # Requests to /admin/* go to the admin backend
    reverse_proxy /admin/* localhost:4000

    # Everything else — serve static files
    root * /var/www/frontend
    file_server
}
```

### Header-Based Routing

```caddyfile
example.com {
    @api {
        header X-Service api
    }
    reverse_proxy @api localhost:3000

    @web {
        header X-Service web
    }
    reverse_proxy @web localhost:4000
}
```

### Adding Headers to Proxied Requests

```caddyfile
example.com {
    reverse_proxy localhost:3000 {
        header_up X-Real-IP {remote_host}
        header_up X-Forwarded-Port {server_port}

        # Modify response headers from the backend
        header_down -Server               # Remove the Server header
        header_down X-Powered-By ""       # Remove X-Powered-By
    }
}
```

**Explanation:**
- `header_up` = headers sent TO the backend (upstream)
- `header_down` = headers sent back TO the client (downstream)
- `-HeaderName` = remove the header
- `HeaderName ""` = set header to empty (effectively removes it)

## Automatic HTTPS — How It Works Under the Hood

This is Caddy's crown jewel. Let's understand exactly what happens.

### The Full Process

```
1. You write: example.com { ... } in your Caddyfile
2. Caddy sees a public domain name
3. Caddy checks if it already has a valid certificate for this domain
4. If not, Caddy contacts Let's Encrypt (or ZeroSSL)
5. Let's Encrypt issues an ACME challenge:
   - HTTP-01: "Put this file at http://example.com/.well-known/acme-challenge/xyz"
   - Caddy automatically handles this challenge
6. Let's Encrypt verifies and issues the certificate
7. Caddy installs the certificate and starts serving HTTPS
8. Caddy sets up HTTP → HTTPS redirect automatically
9. Before the certificate expires (usually 90 days), Caddy automatically renews it
10. All of this happens with ZERO configuration from you
```

### Different Scenarios — What Caddy Does Automatically

| Scenario | What Caddy Does |
|----------|----------------|
| `example.com` | Gets real Let's Encrypt certificate, enables HTTPS |
| `localhost` | Generates a self-signed certificate for local development |
| `:8080` (just a port) | No HTTPS — serves plain HTTP |
| `http://example.com` | You explicitly asked for HTTP, so no HTTPS |
| `192.168.1.10` | IP address — generates self-signed certificate |

### Customizing TLS

Most of the time, Caddy's defaults are perfect. But you can customize if needed:

```caddyfile
example.com {
    tls your-email@example.com    # Email for Let's Encrypt notifications

    reverse_proxy localhost:3000
}
```

### Using a Custom Certificate (Not Let's Encrypt)

```caddyfile
example.com {
    tls /path/to/cert.pem /path/to/key.pem

    reverse_proxy localhost:3000
}
```

### DNS Challenge for Wildcard Certificates

Wildcard certificates (`*.example.com`) can't use the HTTP challenge — they need the **DNS challenge**. Caddy supports many DNS providers:

```caddyfile
*.example.com {
    tls {
        dns cloudflare {env.CF_API_TOKEN}
    }

    @blog host blog.example.com
    reverse_proxy @blog localhost:4000

    @api host api.example.com
    reverse_proxy @api localhost:3000
}
```

**Note:** You'll need to build Caddy with the DNS plugin using `xcaddy`:

```bash
xcaddy build --with github.com/caddy-dns/cloudflare
```

### On-Demand TLS — Certificates on the Fly

This is amazing for multi-tenant platforms where you don't know all domains ahead of time:

```caddyfile
{
    on_demand_tls {
        ask http://localhost:5555/check    # Caddy asks this endpoint if it should get a cert
    }
}

https:// {
    tls {
        on_demand
    }
    reverse_proxy localhost:8080
}
```

When a request comes in for `customer1.example.com`, Caddy:
1. Asks your `/check` endpoint if this domain is allowed
2. If yes, gets a certificate from Let's Encrypt in real-time
3. Serves the request over HTTPS

## Matchers — Caddy's Way of Saying "If This, Then That"

Matchers let you apply directives to specific requests. They're like Nginx's `location` blocks but way more flexible.

### Named Matchers

```caddyfile
example.com {
    # Define a matcher named "api"
    @api {
        path /api/*
        method GET POST
    }

    # Use it
    reverse_proxy @api localhost:3000

    # Another matcher
    @static {
        path *.css *.js *.png *.jpg *.gif *.svg *.woff2
    }

    # Serve static files with caching headers
    header @static Cache-Control "public, max-age=31536000"
    file_server
}
```

### Inline Matchers

For simple, one-condition matchers, you can use them inline:

```caddyfile
example.com {
    # Path matcher (inline)
    reverse_proxy /api/* localhost:3000

    # Everything else
    file_server
}
```

### All Available Matchers

| Matcher | Example | What It Matches |
|---------|---------|----------------|
| `path` | `path /api/*` | URL path pattern |
| `path_regexp` | `path_regexp \.php$` | Path with regex |
| `host` | `host api.example.com` | Request hostname |
| `method` | `method GET POST` | HTTP method |
| `header` | `header X-Custom value` | Request header |
| `header_regexp` | `header_regexp Cookie session=(.*)` | Header with regex |
| `protocol` | `protocol https` | HTTP or HTTPS |
| `query` | `query token=abc` | Query parameters |
| `remote_ip` | `remote_ip 192.168.1.0/24` | Client IP |
| `expression` | `expression {method} == "POST"` | CEL expression |
| `file` | `file {path}.html {path}/index.html` | File exists on disk |
| `not` | `not path /api/*` | Negation |

### Combining Matchers

```caddyfile
example.com {
    @secure-api {
        path /api/*
        method POST PUT DELETE
        not remote_ip 10.0.0.0/8     # Block internal IPs? No — allow only specific
    }

    # Apply rate limiting and auth to this specific matcher
    rate_limit @secure-api {
        zone api_zone {
            key {remote_host}
            events 10
            window 1m
        }
    }

    reverse_proxy @secure-api localhost:3000
}
```

## Static File Server — Serving Websites

### Basic Static Site

```caddyfile
example.com {
    root * /var/www/mysite
    file_server
}
```

### With Directory Browsing

```caddyfile
example.com {
    root * /var/www/files
    file_server browse        # Shows directory listing
}
```

### Single Page Application (SPA) — React, Vue, Angular

SPAs need all routes to serve `index.html` (so the frontend router handles paths):

```caddyfile
example.com {
    root * /var/www/spa
    try_files {path} /index.html    # If file doesn't exist, serve index.html
    file_server
}
```

**What `try_files` does:**
```
Request: /about
  → Does /var/www/spa/about exist? No
  → Serve /var/www/spa/index.html instead
  → React/Vue/Angular frontend router handles /about

Request: /static/main.css
  → Does /var/www/spa/static/main.css exist? Yes
  → Serve the CSS file directly
```

### Compression

```caddyfile
example.com {
    root * /var/www/mysite
    encode gzip zstd           # Enable gzip and zstd compression
    file_server
}
```

### Caching Headers

```caddyfile
example.com {
    root * /var/www/mysite

    # Cache static assets for 1 year
    @static {
        path *.css *.js *.png *.jpg *.gif *.svg *.woff2 *.ico
    }
    header @static Cache-Control "public, max-age=31536000, immutable"

    # Don't cache HTML files
    @html {
        path *.html
    }
    header @html Cache-Control "no-cache, no-store, must-revalidate"

    encode gzip
    file_server
}
```

## Caddy with Docker

### Basic Docker Setup

```yaml
# docker-compose.yml
services:
  caddy:
    image: caddy:2
    container_name: caddy
    restart: unless-stopped
    ports:
      - "80:80"
      - "443:443"
      - "443:443/udp"     # HTTP/3 support!
    volumes:
      - ./Caddyfile:/etc/caddy/Caddyfile     # Your config
      - ./site:/srv                            # Your website files
      - caddy_data:/data                       # Certificate storage
      - caddy_config:/config                   # Caddy config storage

volumes:
  caddy_data:        # IMPORTANT: This persists your SSL certificates
  caddy_config:
```

### Reverse Proxy with Docker Compose

```yaml
# docker-compose.yml
services:
  caddy:
    image: caddy:2
    container_name: caddy
    restart: unless-stopped
    ports:
      - "80:80"
      - "443:443"
      - "443:443/udp"
    volumes:
      - ./Caddyfile:/etc/caddy/Caddyfile
      - caddy_data:/data
      - caddy_config:/config
    networks:
      - web

  frontend:
    image: node:20-alpine
    container_name: frontend
    working_dir: /app
    volumes:
      - ./frontend:/app
    command: npm start
    networks:
      - web

  backend:
    image: node:20-alpine
    container_name: backend
    working_dir: /app
    volumes:
      - ./backend:/app
    command: npm start
    networks:
      - web

  postgres:
    image: postgres:16
    container_name: postgres
    environment:
      POSTGRES_DB: myapp
      POSTGRES_USER: user
      POSTGRES_PASSWORD_FILE: /run/secrets/db_password
    volumes:
      - pg_data:/var/lib/postgresql/data
    networks:
      - web

networks:
  web:
    name: web

volumes:
  caddy_data:
  caddy_config:
  pg_data:
```

```caddyfile
# Caddyfile
example.com {
    reverse_proxy frontend:3000
}

api.example.com {
    reverse_proxy backend:8080
}
```

That's it! Caddy handles HTTPS for both domains, reverse proxies to the correct containers, and manages everything automatically.

## Security Headers & Hardening

### Adding Security Headers

```caddyfile
example.com {
    header {
        # Prevent clickjacking
        X-Frame-Options "DENY"

        # Prevent MIME type sniffing
        X-Content-Type-Options "nosniff"

        # Enable XSS protection
        X-XSS-Protection "1; mode=block"

        # Referrer policy
        Referrer-Policy "strict-origin-when-cross-origin"

        # Content Security Policy
        Content-Security-Policy "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'"

        # Strict Transport Security (HSTS)
        Strict-Transport-Security "max-age=31536000; includeSubDomains; preload"

        # Permissions Policy
        Permissions-Policy "camera=(), microphone=(), geolocation=()"

        # Remove server identification headers
        -Server                    # Remove the Server header
        -X-Powered-By              # Remove X-Powered-By if set by backend
    }

    reverse_proxy localhost:3000
}
```

### Basic Authentication

```caddyfile
example.com {
    basicauth /admin/* {
        admin $2a$14$Zkx19XLiW6VYouLCR5NmveL8oN5FvYhza4TAdq7Lo/9Jmx5VPOYm6
        # Generate hash: caddy hash-password
    }

    reverse_proxy localhost:3000
}
```

To generate the password hash:

```bash
# Caddy has a built-in password hashing command
caddy hash-password
# Enter your password when prompted
# It outputs a bcrypt hash you can paste into the Caddyfile
```

### IP Allow/Deny Lists

```caddyfile
example.com {
    @blocked {
        remote_ip 192.168.1.100
        remote_ip 10.0.0.0/8
    }
    respond @blocked "Access Denied" 403

    # Or allow only specific IPs
    @allowed {
        remote_ip 203.0.113.0/24
    }
    reverse_proxy @allowed localhost:3000
}
```

## Rate Limiting

Caddy supports rate limiting through the `rate_limit` directive (available as a plugin):

```caddyfile
{
    order rate_limit before basicauth
}

example.com {
    rate_limit {
        zone static_zone {
            key {remote_host}
            events 100
            window 1m
        }
    }

    reverse_proxy localhost:3000
}
```

This limits each IP to 100 requests per minute.

## The JSON Config API — Dynamic Configuration

Besides the Caddyfile, Caddy has a powerful **REST API** for managing its configuration at runtime.

### View Current Config

```bash
curl http://localhost:2019/config/ | jq .
```

### Add a New Site via API

```bash
curl -X POST http://localhost:2019/config/apps/http/servers/srv0/routes \
  -H "Content-Type: application/json" \
  -d '{
    "match": [{"host": ["newsite.example.com"]}],
    "handle": [{
      "handler": "reverse_proxy",
      "upstreams": [{"dial": "localhost:5000"}]
    }]
  }'
```

### Reload Config Without Downtime

```bash
# Reload the Caddyfile
caddy reload --config /etc/caddy/Caddyfile

# Or adapt and load
caddy adapt --config /etc/caddy/Caddyfile | curl -X POST http://localhost:2019/load \
  -H "Content-Type: application/json" \
  -d @-
```

The API endpoint `localhost:2019` is Caddy's admin API. It's only accessible locally by default — not exposed to the internet.

## Running Caddy as a System Service

When you install Caddy via a package manager, it sets up a systemd service automatically. But if you installed manually:

### Create a Systemd Service

```bash
# Create a caddy user
sudo groupadd --system caddy
sudo useradd --system \
  --gid caddy \
  --create-home \
  --home-dir /var/lib/caddy \
  --shell /usr/sbin/nologin \
  caddy
```

```ini
# /etc/systemd/system/caddy.service
[Unit]
Description=Caddy Web Server
Documentation=https://caddyserver.com/docs/
After=network.target network-online.target
Requires=network-online.target

[Service]
Type=notify
User=caddy
Group=caddy
ExecStart=/usr/local/bin/caddy run --environ --config /etc/caddy/Caddyfile
ExecReload=/usr/local/bin/caddy reload --config /etc/caddy/Caddyfile --force
TimeoutStopSec=5s
LimitNOFILE=1048576
LimitNPROC=512
PrivateTmp=true
ProtectSystem=full
AmbientCapabilities=CAP_NET_BIND_SERVICE

[Install]
WantedBy=multi-user.target
```

```bash
# Enable and start the service
sudo systemctl daemon-reload
sudo systemctl enable caddy
sudo systemctl start caddy

# Check status
sudo systemctl status caddy

# View logs
sudo journalctl -u caddy -f
```

## Complete Real-World Examples

### Example 1: WordPress + Caddy

```caddyfile
example.com {
    root * /var/www/wordpress
    php_fastcgi unix//run/php/php8.2-fpm.sock
    file_server
    encode gzip

    # Block access to sensitive files
    @blocked {
        path /wp-config.php
        path /.htaccess
        path /readme.html
        path /license.txt
    }
    respond @blocked 404
}
```

### Example 2: Full-Stack App (React + Node.js API + WebSocket)

```caddyfile
example.com {
    # API requests → Node.js backend
    reverse_proxy /api/* backend:3000

    # WebSocket connections → backend
    @websocket {
        header Connection *Upgrade*
        header Upgrade websocket
    }
    reverse_proxy @websocket backend:3000

    # Everything else → React frontend
    root * /var/www/frontend/build
    try_files {path} /index.html
    file_server

    # Compression
```

## Caddy vs Nginx — Side-by-Side Config Comparison

Let's see how the same setup looks in both:

### Task: Reverse proxy with HTTPS, HTTP redirect, security headers, and gzip

**Nginx (30+ lines):**

```nginx
server {
    listen 80;
    server_name example.com;
    return 301 https://$server_name$request_uri;
}

server {
    listen 443 ssl http2;
    server_name example.com;

    ssl_certificate /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;

    add_header X-Frame-Options "DENY";
    add_header X-Content-Type-Options "nosniff";
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;

    gzip on;
    gzip_types text/plain application/json text/css application/javascript;

    location / {
        proxy_pass http://localhost:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

Plus you need to set up Certbot separately for SSL certificate management.

**Caddy (12 lines):**

```caddyfile
example.com {
    reverse_proxy localhost:3000

    encode gzip

    header {
        X-Frame-Options "DENY"
        X-Content-Type-Options "nosniff"
        Strict-Transport-Security "max-age=31536000; includeSubDomains"
    }
}
```

No SSL setup. No HTTP redirect config. No Certbot. **Caddy handles all of it automatically.**

## Production Best Practices

### 1. Always Persist the `/data` Volume

```yaml
volumes:
  caddy_data:    # This stores your SSL certificates!
```

If you lose this volume, Caddy will need to request new certificates. Let's Encrypt has rate limits (50 certificates per domain per week), so losing certificates can temporarily lock you out.

### 2. Use a Global Email

```caddyfile
{
    email admin@example.com
}
```

Let's Encrypt uses this to notify you about certificate issues.

### 3. Test with Staging First

```caddyfile
{
    acme_ca https://acme-staging-v02.api.letsencrypt.org/directory
}
```

The staging environment has no rate limits. Test your setup here first, then switch to production.

### 4. Enable Compression

```caddyfile
encode gzip zstd
```

This reduces bandwidth and improves page load times.

### 5. Set Proper Timeouts

```caddyfile
example.com {
    reverse_proxy localhost:3000 {
        transport http {
            read_timeout 30s
            write_timeout 30s
            dial_timeout 10s
        }
    }
}
```

### 6. Don't Expose the Admin API

The admin API (`localhost:2019`) should never be accessible from the internet. It's bound to localhost by default — don't change this in production.
