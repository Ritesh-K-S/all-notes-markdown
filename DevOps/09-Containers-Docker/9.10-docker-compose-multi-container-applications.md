# 9.10 — Docker Compose — Multi-Container Applications (v2)

> **One Line Answer:** Docker Compose is a tool that lets you **define and run multiple Docker containers together** using a single YAML file — so you can spin up your entire application stack (app + database + cache + queue) with just **one command**: `docker compose up`.

---

## 🤔 Why Do We Need Docker Compose?

Let's say you're building a web application. Your app doesn't live alone — it needs friends:

```
Your Application Stack:
┌─────────────────────────────────────────────────┐
│  🌐 Frontend (React/Nginx)     — Port 3000      │
│  ⚙️  Backend (Node.js/Python)   — Port 8080      │
│  🗄️ Database (PostgreSQL)       — Port 5432      │
│  ⚡ Cache (Redis)               — Port 6379      │
│  📨 Message Queue (RabbitMQ)    — Port 5672      │
└─────────────────────────────────────────────────┘
```

Now, without Docker Compose, you'd have to run each container **manually**:

```bash
# Without Docker Compose — the painful way 😫

# Step 1: Create a network
docker network create myapp-network

# Step 2: Start the database
docker run -d --name postgres --network myapp-network \
  -e POSTGRES_PASSWORD=secret \
  -v pgdata:/var/lib/postgresql/data \
  postgres:16

# Step 3: Start Redis
docker run -d --name redis --network myapp-network \
  redis:7-alpine

# Step 4: Start the backend
docker run -d --name backend --network myapp-network \
  -e DATABASE_URL=postgresql://postgres:secret@postgres:5432/mydb \
  -e REDIS_URL=redis://redis:6379 \
  -p 8080:8080 \
  myapp-backend:latest

# Step 5: Start the frontend
docker run -d --name frontend --network myapp-network \
  -p 3000:80 \
  myapp-frontend:latest
```

That's **4 separate commands**, and you need to remember:
- The order to start them in
- The network they share
- All environment variables
- All volume mounts
- All port mappings

And when you want to stop everything? You need to stop and remove each container **one by one**.

**Docker Compose solves all of this.** You put everything in ONE file, and run ONE command. Done. ✅

---

## ✅ What is Docker Compose?

Docker Compose is a tool that:

| What it does | How |
|---|---|
| **Defines** your multi-container app | Using a YAML file called `docker-compose.yml` (or `compose.yml`) |
| **Runs** all containers together | With a single command: `docker compose up` |
| **Manages** the lifecycle | Start, stop, rebuild, view logs — all containers at once |
| **Creates networking** automatically | All services can talk to each other by name |
| **Handles volumes** | Persistent data is defined in the same file |

### The Simple Formula:

```
Docker Compose = Multi-Container Definition (YAML) + Single Command Lifecycle Management
```

## 📌 Compose V1 vs V2 — What Changed?

You might see two versions of Compose mentioned online. Here's the difference:

| Feature | Compose V1 (Legacy) | Compose V2 (Current ✅) |
|---|---|---|
| **Command** | `docker-compose` (with hyphen) | `docker compose` (space, no hyphen) |
| **Written in** | Python | Go (integrated into Docker CLI) |
| **Installation** | Separate install | Comes built-in with Docker Desktop & Docker Engine |
| **Performance** | Slower | Faster |
| **File format** | Required `version: "3"` at top | `version` key is **optional** (ignored) |
| **Status** | ❌ Deprecated (EOL June 2023) | ✅ Active, recommended |

> **Bottom line:** Always use `docker compose` (V2). If you see `docker-compose` in old tutorials, just replace the hyphen with a space.

## 📁 The Compose File — `compose.yml`

The heart of Docker Compose is the **Compose file**. It's a YAML file that defines everything about your multi-container application.

### File Naming (What Docker Compose Looks For):

Docker Compose looks for files in this order:

```
1. compose.yml          ← Preferred (new standard)
2. compose.yaml
3. docker-compose.yml   ← Still works (legacy)
4. docker-compose.yaml
```

> **Tip:** Use `compose.yml` for new projects. It's the modern convention.

### Basic Structure:

```yaml
# compose.yml — The skeleton

services:        # 🟢 Required — Define your containers here
  web:
    image: nginx
  db:
    image: postgres

volumes:         # 🟡 Optional — Define named volumes
  dbdata:

networks:        # 🟡 Optional — Define custom networks
  frontend:
  backend:
```

The three **top-level keys** are:

| Key | Purpose | Required? |
|---|---|---|
| `services` | Define each container (what to run) | ✅ Yes |
| `volumes` | Define named volumes for persistent data | Optional |
| `networks` | Define custom networks | Optional (auto-created) |

## 🚀 Your First Docker Compose File

Let's start simple — a web server and a database:

```yaml
# compose.yml

services:
  web:
    image: nginx:alpine
    ports:
      - "8080:80"    # Host port 8080 → Container port 80

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: myuser
      POSTGRES_PASSWORD: mypassword
      POSTGRES_DB: mydb
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  pgdata:    # This creates a named volume called "pgdata"
```

### Let's break down every line:

```yaml
services:                    # Start defining our containers
  web:                       # Service name — you pick this (used for DNS too!)
    image: nginx:alpine      # Which Docker image to use
    ports:
      - "8080:80"            # Map port 8080 on your machine to port 80 in container
                             # Format: "HOST_PORT:CONTAINER_PORT"

  db:                        # Another service
    image: postgres:16-alpine
    environment:             # Environment variables (like -e in docker run)
      POSTGRES_USER: myuser
      POSTGRES_PASSWORD: mypassword
      POSTGRES_DB: mydb
    volumes:
      - pgdata:/var/lib/postgresql/data    # Mount named volume for data persistence
                                           # Format: VOLUME_NAME:CONTAINER_PATH

volumes:
  pgdata:                    # Declare the named volume
                             # Docker will manage its storage location
```

### Run It:

```bash
# Start everything (in foreground — you see logs)
docker compose up

# Start everything (in background — detached mode)
docker compose up -d

# Check running services
docker compose ps

# View logs
docker compose logs

# Stop everything
docker compose down
```

**That's it!** Two containers, one network (auto-created), persistent storage — all from one file. 🎉

## 📖 Complete Service Configuration Reference

Here's everything you can define for a service, with explanations:

### 1. `image` — Which Image to Use

```yaml
services:
  web:
    image: nginx:alpine                    # From Docker Hub
    # OR
    image: myregistry.com/myapp:v1.2.3     # From a private registry
    # OR
    image: ghcr.io/username/myapp:latest   # From GitHub Container Registry
```

### 2. `build` — Build from Dockerfile

Instead of pulling an image, you can **build** one:

```yaml
services:
  backend:
    build: ./backend                # Path to directory containing Dockerfile
    # This is shorthand for:
    build:
      context: ./backend            # Build context (where files are)
      dockerfile: Dockerfile        # Dockerfile name (default: Dockerfile)
```

More detailed build options:

```yaml
services:
  backend:
    build:
      context: ./backend                # Build context directory
      dockerfile: Dockerfile.prod       # Custom Dockerfile name
      args:                             # Build arguments (ARG in Dockerfile)
        NODE_ENV: production
        APP_VERSION: "1.2.3"
      target: production                # Multi-stage build target
      cache_from:                       # Use these images as cache source
        - myapp:latest
    image: myapp:latest                 # Tag the built image with this name
```

**What each option means:**

| Option | Purpose |
|---|---|
| `context` | The directory Docker uses as the build context (files available during build) |
| `dockerfile` | Name of the Dockerfile (useful if you have multiple, like `Dockerfile.dev`) |
| `args` | Build-time variables (passed to `ARG` instructions in Dockerfile) |
| `target` | Which stage to build up to in a multi-stage Dockerfile |
| `cache_from` | Images to use as layer cache (speeds up builds) |

### 3. `ports` — Port Mapping

```yaml
services:
  web:
    ports:
      - "8080:80"              # HOST:CONTAINER
      - "443:443"              # Map HTTPS too
      - "8080:80/udp"          # UDP protocol
      - "127.0.0.1:8080:80"    # Only accessible from localhost (more secure!)
```

> **Security Tip:** Use `127.0.0.1:PORT:PORT` for services that shouldn't be exposed to the network (like databases).

### 4. `environment` — Environment Variables

Two ways to set them:

```yaml
services:
  backend:
    environment:
      # Way 1: Key-Value map (recommended)
      DATABASE_URL: postgresql://user:pass@db:5432/mydb
      REDIS_URL: redis://redis:6379
      NODE_ENV: production

    # OR
    environment:
      # Way 2: List format
      - DATABASE_URL=postgresql://user:pass@db:5432/mydb
      - REDIS_URL=redis://redis:6379
      - NODE_ENV=production
```

### 5. `env_file` — Load Variables from a File

Instead of listing every variable, load them from a `.env` file:

```yaml
services:
  backend:
    env_file:
      - .env              # Load from .env file
      - .env.production   # Load another file (later values override earlier ones)
```

The `.env` file:
```bash
# .env
DATABASE_URL=postgresql://user:pass@db:5432/mydb
REDIS_URL=redis://redis:6379
NODE_ENV=production
SECRET_KEY=my-super-secret-key
```

> **Security Tip:** Add `.env` to your `.gitignore`! Never commit secrets to Git.

### 6. `volumes` — Mount Volumes

```yaml
services:
  db:
    volumes:
      # Named volume — Docker manages the storage
      - pgdata:/var/lib/postgresql/data

      # Bind mount — Maps a host directory to container
      - ./init-scripts:/docker-entrypoint-initdb.d

      # Read-only bind mount — Container can't write to it
      - ./config/nginx.conf:/etc/nginx/nginx.conf:ro

volumes:
  pgdata:    # Declare named volumes here
```

| Type | Syntax | Use Case |
|---|---|---|
| Named Volume | `volumename:/path` | Persistent data (databases, uploads) |
| Bind Mount | `./host/path:/container/path` | Development (live code reload) |
| Read-Only | `./path:/path:ro` | Config files that shouldn't be modified |

### 7. `networks` — Custom Networking

```yaml
services:
  frontend:
    networks:
      - frontend-net        # Only on frontend network

  backend:
    networks:
      - frontend-net        # Can talk to frontend
      - backend-net         # Can also talk to database

  db:
    networks:
      - backend-net         # Only on backend network (frontend CAN'T reach it!)

networks:
  frontend-net:
  backend-net:
```

This creates **network isolation** — the frontend can talk to the backend, but it **cannot** directly access the database. Security win! 🔒

### 8. `restart` — Restart Policy

```yaml
services:
  web:
    restart: always          # Always restart (even after host reboot)

  db:
    restart: unless-stopped  # Restart unless manually stopped

  worker:
    restart: on-failure      # Only restart if container exits with error
    # restart: "no"          # Never restart (default)
```

| Policy | Behavior |
|---|---|
| `"no"` | Never restart (default) |
| `always` | Always restart, even after reboot |
| `on-failure` | Restart only if exit code ≠ 0 |
| `unless-stopped` | Like `always`, but respects manual stop |

### 9. `command` and `entrypoint` — Override Commands

```yaml
services:
  backend:
    image: node:20-alpine
    command: npm run dev          # Override CMD from Dockerfile
    # OR for multiple arguments:
    command: ["npm", "run", "dev"]

  worker:
    image: python:3.12
    entrypoint: ["python"]        # Override ENTRYPOINT
    command: ["worker.py", "--verbose"]
```

### 10. `depends_on` — Service Dependencies

```yaml
services:
  backend:
    depends_on:
      - db        # Start db before backend
      - redis     # Start redis before backend

  db:
    image: postgres:16

  redis:
    image: redis:7
```

> ⚠️ **Important:** `depends_on` only waits for the container to **start**, NOT for the service inside to be **ready**. PostgreSQL might start the container but still be initializing. We'll cover `healthchecks` in the next chapter (9.11) to solve this!

## 🌐 Networking in Docker Compose

One of the best features of Compose is **automatic networking**.

### How It Works:

When you run `docker compose up`, Compose automatically:

1. Creates a **network** for your project
2. Connects **all services** to this network
3. Each service is accessible by its **service name** as a hostname

```yaml
# compose.yml
services:
  backend:
    image: myapp-backend
    environment:
      DATABASE_HOST: db          # ← Use the service name as hostname!
      REDIS_HOST: redis          # ← Same here!

  db:
    image: postgres:16

  redis:
    image: redis:7
```

```
Inside the Compose network:
┌──────────────────────────────────────────┐
│  Network: myproject_default              │
│                                          │
│  backend ←──→ db      (via hostname "db")│
│     ↕                                    │
│  backend ←──→ redis   (via hostname "redis")│
│                                          │
│  db ✘──→ redis  (they CAN talk, same net)│
└──────────────────────────────────────────┘
```

**The network name** is `<project-name>_default`. The project name defaults to the **directory name**.

### DNS Resolution Example:

Inside the `backend` container:
```bash
# You can ping other services by name!
ping db          # Resolves to the db container's IP
ping redis       # Resolves to the redis container's IP

# Your app connects using service names
# postgresql://user:pass@db:5432/mydb     ← "db" resolves automatically
# redis://redis:6379                      ← "redis" resolves automatically
```

## 🔧 Essential Docker Compose CLI Commands

Here's your complete command reference:

### Lifecycle Commands:

```bash
# Start all services (build if needed)
docker compose up

# Start in detached (background) mode
docker compose up -d

# Start and force rebuild images
docker compose up -d --build

# Start only specific services
docker compose up -d backend db

# Stop all services (containers keep existing)
docker compose stop

# Stop and REMOVE containers, networks
docker compose down

# Stop and remove EVERYTHING (including volumes — ⚠️ DATA LOSS!)
docker compose down -v

# Restart all services
docker compose restart

# Restart specific service
docker compose restart backend
```

### Monitoring Commands:

```bash
# List running services
docker compose ps

# List ALL services (including stopped)
docker compose ps -a

# View logs (all services)
docker compose logs

# View logs (specific service)
docker compose logs backend

# Follow logs in real-time (like tail -f)
docker compose logs -f

# Follow logs for specific service with timestamps
docker compose logs -f --timestamps backend

# Show last 100 lines
docker compose logs --tail 100 backend
```

### Exec & Run Commands:

```bash
# Execute command in a RUNNING container
docker compose exec backend sh
docker compose exec db psql -U myuser -d mydb

# Run a ONE-TIME command in a NEW container
docker compose run backend npm test
docker compose run --rm backend npm run migrate
# --rm removes the container after it exits

# View resource usage
docker compose top
```

### Build Commands:

```bash
# Build all images
docker compose build

# Build specific service
docker compose build backend

# Build with no cache
docker compose build --no-cache

# Pull latest images (for services using image:)
docker compose pull
```

### Scaling (Running Multiple Instances):

```bash
# Scale a service to multiple instances
docker compose up -d --scale worker=3

# This creates: worker-1, worker-2, worker-3
# ⚠️ Don't use fixed host ports with scaling (port conflicts!)
```

## 🏗️ Real-World Example — Full Stack Application

Let's build a complete full-stack application with Docker Compose:

```yaml
# compose.yml — Full Stack Application

services:
  # ────────────── Frontend ──────────────
  frontend:
    build:
      context: ./frontend
      dockerfile: Dockerfile
    ports:
      - "3000:80"
    depends_on:
      - backend
    restart: unless-stopped

  # ────────────── Backend API ──────────────
  backend:
    build:
      context: ./backend
      dockerfile: Dockerfile
    ports:
      - "8080:8080"
    environment:
      DATABASE_URL: postgresql://appuser:apppass@db:5432/appdb
      REDIS_URL: redis://redis:6379
      JWT_SECRET: ${JWT_SECRET}          # From .env file or host environment
      NODE_ENV: production
    depends_on:
      - db
      - redis
    restart: unless-stopped
    volumes:
      - uploads:/app/uploads             # Persistent file uploads

  # ────────────── Database ──────────────
  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: appuser
      POSTGRES_PASSWORD: apppass
      POSTGRES_DB: appdb
    volumes:
      - pgdata:/var/lib/postgresql/data  # Persistent database storage
      - ./db/init.sql:/docker-entrypoint-initdb.d/init.sql  # Auto-run on first start
    ports:
      - "127.0.0.1:5432:5432"           # Only accessible from localhost
    restart: unless-stopped

  # ────────────── Cache ──────────────
  redis:
    image: redis:7-alpine
    command: redis-server --appendonly yes   # Enable persistence
    volumes:
      - redisdata:/data
    restart: unless-stopped

  # ────────────── Background Worker ──────────────
  worker:
    build:
      context: ./backend
      dockerfile: Dockerfile
    command: npm run worker               # Override default command
    environment:
      DATABASE_URL: postgresql://appuser:apppass@db:5432/appdb
      REDIS_URL: redis://redis:6379
    depends_on:
      - db
      - redis
    restart: unless-stopped

# ────────────── Volumes ──────────────
volumes:
  pgdata:
  redisdata:
  uploads:
```

### What's happening here:

```
┌──────────────────────────────────────────────────────┐
│                    YOUR APPLICATION                   │
│                                                      │
│  ┌──────────┐     ┌──────────┐     ┌──────────┐     │
│  │ Frontend │────▶│ Backend  │────▶│ Database │     │
│  │ (React)  │     │ (Node.js)│     │(Postgres)│     │
│  │ :3000    │     │ :8080    │     │ :5432    │     │
│  └──────────┘     └────┬─────┘     └──────────┘     │
│                        │                              │
│                   ┌────▼─────┐     ┌──────────┐     │
│                   │  Redis   │     │  Worker  │     │
│                   │ (Cache)  │◀────│(Background│     │
│                   │ :6379    │     │  Jobs)   │     │
│                   └──────────┘     └──────────┘     │
└──────────────────────────────────────────────────────┘
```

## 🛠️ Development vs Production Compose Files

You'll often want different configurations for development and production.

### Development — `compose.yml`:

```yaml
# compose.yml — Development configuration

services:
  backend:
    build:
      context: ./backend
    ports:
      - "8080:8080"
      - "9229:9229"                # Node.js debugger port
    environment:
      NODE_ENV: development
    volumes:
      - ./backend/src:/app/src     # 🔥 Live code reload! Changes reflect instantly
    command: npm run dev            # Use development server with hot reload

  frontend:
    build:
      context: ./frontend
    ports:
      - "3000:3000"
    volumes:
      - ./frontend/src:/app/src    # 🔥 Live code reload for frontend too
    command: npm start

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: devuser
      POSTGRES_PASSWORD: devpass
      POSTGRES_DB: devdb
    ports:
      - "5432:5432"                # Expose DB port for local tools (pgAdmin, DBeaver)
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  pgdata:
```

### Production — `compose.prod.yml`:

```yaml
# compose.prod.yml — Production overrides

services:
  backend:
    build:
      context: ./backend
      dockerfile: Dockerfile.prod    # Optimized production Dockerfile
    ports:
      - "8080:8080"
    environment:
      NODE_ENV: production
    restart: always
    # No volume mounts — use the built code inside the image

  frontend:
    build:
      context: ./frontend
      dockerfile: Dockerfile.prod
    ports:
      - "80:80"
    restart: always

  db:
    environment:
      POSTGRES_USER: ${DB_USER}          # From .env — don't hardcode in production!
      POSTGRES_PASSWORD: ${DB_PASSWORD}
      POSTGRES_DB: ${DB_NAME}
    ports:
      - "127.0.0.1:5432:5432"           # Locked to localhost only
    restart: always
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  pgdata:
```

### Running with Multiple Files:

```bash
# Development (default compose.yml)
docker compose up -d

# Production (merge base + production overrides)
docker compose -f compose.yml -f compose.prod.yml up -d

# You can also use COMPOSE_FILE environment variable
export COMPOSE_FILE=compose.yml:compose.prod.yml
docker compose up -d
```

When you use multiple `-f` flags, the **later files override the earlier ones**. So `compose.prod.yml` overrides the development settings from `compose.yml`.

## 🔄 Variable Substitution & `.env` File

Compose supports **variable substitution** — you can use `${VARIABLE}` in your compose file:

### The `.env` File:

```bash
# .env (in the same directory as compose.yml)

# Database
DB_USER=myuser
DB_PASSWORD=supersecretpassword
DB_NAME=production_db
DB_PORT=5432

# Application
APP_PORT=8080
NODE_ENV=production

# Versions
POSTGRES_VERSION=16-alpine
REDIS_VERSION=7-alpine
```

### Using Variables in Compose:

```yaml
# compose.yml

services:
  backend:
    ports:
      - "${APP_PORT:-8080}:8080"    # Use APP_PORT, default to 8080 if not set
    environment:
      NODE_ENV: ${NODE_ENV}

  db:
    image: postgres:${POSTGRES_VERSION}
    environment:
      POSTGRES_USER: ${DB_USER}
      POSTGRES_PASSWORD: ${DB_PASSWORD}
      POSTGRES_DB: ${DB_NAME}
    ports:
      - "127.0.0.1:${DB_PORT}:5432"

  redis:
    image: redis:${REDIS_VERSION}
```

### Variable Substitution Syntax:

| Syntax | Meaning |
|---|---|
| `${VAR}` | Use the value of `VAR` |
| `${VAR:-default}` | Use `VAR`, or `default` if `VAR` is not set or empty |
| `${VAR-default}` | Use `VAR`, or `default` if `VAR` is not set (empty is OK) |
| `${VAR:?error message}` | Use `VAR`, or show error and stop if not set or empty |
| `${VAR?error message}` | Use `VAR`, or show error and stop if not set |

### Check Your Configuration:

```bash
# See the final resolved compose file (with variables replaced)
docker compose config

# This is super useful for debugging variable substitution issues!
```

## 📋 Project Name & Resource Naming

Compose uses a **project name** to namespace all resources (containers, networks, volumes).

### Default Naming:

```
Project name = Directory name

If your compose.yml is in /home/user/myapp/compose.yml
→ Project name = "myapp"

Container names:    myapp-backend-1, myapp-db-1
Network name:       myapp_default
Volume names:       myapp_pgdata
```

### Override Project Name:

```bash
# Using -p flag
docker compose -p my-custom-name up -d

# Using environment variable
export COMPOSE_PROJECT_NAME=my-custom-name
docker compose up -d
```

### Set in Compose File (top-level):

```yaml
name: my-awesome-project

services:
  web:
    image: nginx
```

## ✅ Common Patterns & Tips

### Pattern 1: Wait for Database Before Starting App

```yaml
services:
  backend:
    depends_on:
      - db
    # Simple but not perfect — db container starts but might not be ready
    # See chapter 9.11 for healthcheck-based waiting!
```

### Pattern 2: Share Code in Development

```yaml
services:
  backend:
    volumes:
      - ./src:/app/src             # Your source code — live reload
      - /app/node_modules          # Anonymous volume — don't overwrite node_modules!
```

That second line (`/app/node_modules`) is important! Without it, your host's `./src` mount would **overwrite** the `node_modules` folder inside the container. The anonymous volume tells Docker: "Keep the container's `node_modules` as-is."

### Pattern 3: Run Database Migrations

```bash
# Run a one-off migration command
docker compose run --rm backend npm run migrate

# Or use a dedicated service
```

```yaml
services:
  migrate:
    build: ./backend
    command: npm run migrate
    depends_on:
      - db
    profiles:
      - tools    # Only runs when explicitly called (see ch 9.11)
```

### Pattern 4: Container Names

```yaml
services:
  db:
    image: postgres:16
    container_name: my-postgres    # Fixed name (useful for external tools)
    # ⚠️ Don't use container_name with --scale (each needs a unique name)
```

## 🔍 Debugging Docker Compose

When things go wrong, here's your debugging toolkit:

```bash
# 1. Check if services are running
docker compose ps

# 2. Check logs for errors
docker compose logs backend
docker compose logs --tail 50 backend

# 3. See the resolved config (catches YAML/variable errors)
docker compose config

# 4. Get inside a container
docker compose exec backend sh

# 5. Check environment variables inside container
docker compose exec backend env

# 6. Check network connectivity
docker compose exec backend ping db

# 7. View detailed container info
docker compose exec backend cat /etc/hosts

# 8. Rebuild from scratch (nuclear option)
docker compose down -v          # Remove everything including volumes
docker compose build --no-cache # Rebuild without cache
docker compose up -d            # Start fresh
```

## 🎯 Summary

| Concept | Key Takeaway |
|---|---|
| Docker Compose | Define & run multi-container apps with a YAML file |
| Compose V2 | Use `docker compose` (no hyphen) — V1 is deprecated |
| Compose File | `compose.yml` with `services`, `volumes`, `networks` |
| Networking | Automatic — services reach each other by name |
| Variables | Use `${VAR}` syntax + `.env` file |
| Dev vs Prod | Use multiple compose files with `-f` flag |
| Lifecycle | `up -d` to start, `down` to stop, `logs` to debug |
