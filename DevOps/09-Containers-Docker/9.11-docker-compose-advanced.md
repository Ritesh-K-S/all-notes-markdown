## 🤔 Why Do We Need Advanced Compose Features?

In the previous chapter (9.10), we built multi-container apps. But in real-world scenarios, you hit problems:

```
Problem 1: "My app starts before the database is ready → crash!"
Problem 2: "I want monitoring tools in dev but NOT in production."
Problem 3: "I have 5 services with almost identical configs — so much copy-paste!"
Problem 4: "I need one-off tasks (migrations, seeds) that don't run by default."
```

Advanced Compose features solve all of these. Let's dive in! 🏊

## 1️⃣ `depends_on` — Smart Service Dependencies

### The Problem with Basic `depends_on`

In chapter 9.10, we saw the basic `depends_on`:

```yaml
services:
  backend:
    depends_on:
      - db          # Start db BEFORE backend
  db:
    image: postgres:16
```

But here's the catch — this only ensures the **container starts**, NOT that the **service inside is ready**.

```
Timeline — What actually happens:
─────────────────────────────────────────────────────
Time 0s:  docker compose up
Time 1s:  PostgreSQL container STARTS           ← depends_on says "OK, db started!"
Time 2s:  Backend container STARTS              ← App tries to connect to DB...
Time 3s:  PostgreSQL is still initializing...   ← 💥 "Connection refused!"
Time 5s:  PostgreSQL is finally ready           ← Too late, backend already crashed
─────────────────────────────────────────────────────
```

### The Solution: `depends_on` + `condition` + `healthcheck`

You can tell Compose to wait until a service is actually **healthy** — not just started:

```yaml
services:
  backend:
    build: ./backend
    depends_on:
      db:
        condition: service_healthy      # ← Wait until db is HEALTHY!
      redis:
        condition: service_healthy      # ← Wait until redis is HEALTHY too!
    environment:
      DATABASE_URL: postgresql://user:pass@db:5432/mydb

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: mydb
    healthcheck:                         # ← Define how to check health
      test: ["CMD-SHELL", "pg_isready -U user -d mydb"]
      interval: 5s                       # Check every 5 seconds
      timeout: 5s                        # Each check times out after 5 seconds
      retries: 5                         # Fail after 5 unsuccessful checks
      start_period: 10s                  # Give the DB 10s to initialize before checking

  redis:
    image: redis:7-alpine
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 3s
      retries: 3
```

### Now the timeline looks like this:

```
Time 0s:  docker compose up
Time 1s:  PostgreSQL container STARTS
Time 1s:  Redis container STARTS
Time 2s:  Healthcheck: pg_isready → "not ready yet" (within start_period, OK)
Time 5s:  Healthcheck: redis-cli ping → "PONG" ✅ Redis is healthy!
Time 11s: Healthcheck: pg_isready → "accepting connections" ✅ DB is healthy!
Time 12s: Backend container STARTS ← Both dependencies are healthy! 🎉
```

**No more crashes!** 🎉

## 🏥 Healthchecks — Deep Dive

Healthchecks tell Docker how to determine if a service is actually working, not just running.

### Healthcheck Syntax:

```yaml
healthcheck:
  test: <command>           # The command to run
  interval: 30s             # How often to check (default: 30s)
  timeout: 30s              # Max time per check (default: 30s)
  retries: 3                # How many failures before "unhealthy" (default: 3)
  start_period: 0s          # Grace period — failures during this time don't count
  start_interval: 5s        # Check interval during start_period (Compose v2.20+)
```

### The `test` Command — Three Formats:

```yaml
# Format 1: CMD-SHELL — Run through shell (supports pipes, variables)
healthcheck:
  test: ["CMD-SHELL", "curl -f http://localhost:8080/health || exit 1"]

# Format 2: CMD — Run directly (no shell — more efficient)
healthcheck:
  test: ["CMD", "pg_isready", "-U", "postgres"]

# Format 3: Shell string (shorthand for CMD-SHELL)
healthcheck:
  test: curl -f http://localhost:8080/health || exit 1
```

> **Rule of thumb:** The test command should exit with `0` for healthy, `1` for unhealthy.

### Healthcheck Examples for Common Services:

```yaml
# ──────── PostgreSQL ────────
db:
  image: postgres:16-alpine
  healthcheck:
    test: ["CMD-SHELL", "pg_isready -U ${POSTGRES_USER:-postgres} -d ${POSTGRES_DB:-postgres}"]
    interval: 5s
    timeout: 5s
    retries: 5
    start_period: 15s

# ──────── MySQL / MariaDB ────────
mysql:
  image: mysql:8
  healthcheck:
    test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
    interval: 10s
    timeout: 5s
    retries: 5
    start_period: 30s

# ──────── Redis ────────
redis:
  image: redis:7-alpine
  healthcheck:
    test: ["CMD", "redis-cli", "ping"]
    interval: 5s
    timeout: 3s
    retries: 3

# ──────── MongoDB ────────
mongo:
  image: mongo:7
  healthcheck:
    test: ["CMD", "mongosh", "--eval", "db.adminCommand('ping')"]
    interval: 10s
    timeout: 5s
    retries: 5
    start_period: 20s

# ──────── Elasticsearch ────────
elasticsearch:
  image: elasticsearch:8.12.0
  healthcheck:
    test: ["CMD-SHELL", "curl -fsSL http://localhost:9200/_cluster/health || exit 1"]
    interval: 15s
    timeout: 10s
    retries: 5
    start_period: 60s

# ──────── RabbitMQ ────────
rabbitmq:
  image: rabbitmq:3-management
  healthcheck:
    test: ["CMD", "rabbitmq-diagnostics", "-q", "ping"]
    interval: 10s
    timeout: 5s
    retries: 5
    start_period: 30s

# ──────── Nginx / Web Apps ────────
web:
  image: nginx:alpine
  healthcheck:
    test: ["CMD-SHELL", "curl -f http://localhost/ || exit 1"]
    interval: 10s
    timeout: 5s
    retries: 3

# ──────── Custom API ────────
api:
  build: ./api
  healthcheck:
    test: ["CMD-SHELL", "curl -f http://localhost:8080/health || exit 1"]
    interval: 15s
    timeout: 5s
    retries: 3
    start_period: 20s
```

### Health Status States:

```
Container Health States:
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  starting   │───▶│   healthy   │───▶│  unhealthy  │
│             │    │             │    │             │
│ (within     │    │ (test       │    │ (test       │
│  start_     │    │  passes)    │    │  failed     │
│  period)    │    │             │    │  retries    │
│             │    │             │    │  times)     │
└─────────────┘    └─────────────┘    └─────────────┘
```

### Check Container Health:

```bash
# See health status in docker compose ps
docker compose ps
# NAME        STATUS              PORTS
# db-1        Up 5m (healthy)     5432/tcp
# backend-1   Up 4m (healthy)     8080/tcp

# Detailed health info
docker inspect --format='{{json .State.Health}}' myapp-db-1

# Watch health in real-time
docker compose events
```

### Disabling a Healthcheck from Image:

Some images come with built-in healthchecks. You can disable them:

```yaml
services:
  myservice:
    image: some-image-with-healthcheck
    healthcheck:
      disable: true    # ← Turn off healthcheck from the image
```

## 🔗 `depends_on` — All Conditions

The `condition` field in `depends_on` supports three values:

```yaml
services:
  app:
    depends_on:
      db:
        condition: service_started       # Default — just wait for container to start
      redis:
        condition: service_healthy       # Wait for healthcheck to pass ✅
      migrations:
        condition: service_completed_successfully  # Wait for it to finish (exit 0)
```

| Condition | Wait For | Use Case |
|---|---|---|
| `service_started` | Container starts | Default, basic ordering |
| `service_healthy` | Healthcheck passes ✅ | Databases, caches, any service that takes time to initialize |
| `service_completed_successfully` | Container exits with code 0 | Migrations, seeders, one-time setup tasks |

### Example — Wait for Migrations Before Starting App:

```yaml
services:
  # Step 1: Database must be healthy
  db:
    image: postgres:16
    healthcheck:
      test: ["CMD-SHELL", "pg_isready"]
      interval: 5s
      timeout: 5s
      retries: 5

  # Step 2: Run migrations (must complete successfully)
  migrate:
    build: ./backend
    command: npm run migrate
    depends_on:
      db:
        condition: service_healthy       # DB must be ready first
    restart: "no"                        # Don't restart — just run once and exit

  # Step 3: Start the app (after migrations are done)
  backend:
    build: ./backend
    command: npm start
    depends_on:
      db:
        condition: service_healthy
      migrate:
        condition: service_completed_successfully   # Migrations must finish!
```

```
Execution flow:
┌──────┐         ┌─────────┐         ┌─────────┐
│  DB  │ healthy │ Migrate │ exit 0  │ Backend │
│start │────────▶│  run    │────────▶│  start  │
│      │         │         │         │         │
└──────┘         └─────────┘         └─────────┘
```

**This is a production-grade pattern!** Your app only starts when the database is ready AND migrations have been applied. 🎯

## 2️⃣ Profiles — Optional Services

### The Problem:

You have services that should only run in certain scenarios:

- Debug tools → only in development
- Monitoring stack → only in staging/production
- Database admin tools → only when you need them
- Seed/migration tasks → only when explicitly called

Without profiles, you'd need separate compose files for each combination. Profiles make it elegant.

### How Profiles Work:

```yaml
services:
  # ──── Always runs (no profile assigned) ────
  backend:
    build: ./backend
    ports:
      - "8080:8080"

  db:
    image: postgres:16

  # ──── Only runs in "debug" profile ────
  pgadmin:
    image: dpage/pgadmin4
    ports:
      - "5050:80"
    environment:
      PGADMIN_DEFAULT_EMAIL: admin@admin.com
      PGADMIN_DEFAULT_PASSWORD: admin
    profiles:
      - debug                    # ← Only starts with "debug" profile

  # ──── Only runs in "monitoring" profile ────
  prometheus:
    image: prom/prometheus
    ports:
      - "9090:9090"
    profiles:
      - monitoring               # ← Only starts with "monitoring" profile

  grafana:
    image: grafana/grafana
    ports:
      - "3000:3000"
    profiles:
      - monitoring               # ← Same profile as Prometheus

  # ──── Only runs in "tools" profile ────
  migrate:
    build: ./backend
    command: npm run migrate
    depends_on:
      db:
        condition: service_healthy
    profiles:
      - tools
    restart: "no"

  seed:
    build: ./backend
    command: npm run seed
    depends_on:
      db:
        condition: service_healthy
    profiles:
      - tools
    restart: "no"
```

### Using Profiles:

```bash
# Start only services WITHOUT profiles (backend, db)
docker compose up -d

# Start with the "debug" profile (backend, db, pgadmin)
docker compose --profile debug up -d

# Start with the "monitoring" profile (backend, db, prometheus, grafana)
docker compose --profile monitoring up -d

# Start with MULTIPLE profiles
docker compose --profile debug --profile monitoring up -d

# Run a one-off tool
docker compose run --rm migrate         # ← Profiles are auto-activated for run!

# Using environment variable
export COMPOSE_PROFILES=debug,monitoring
docker compose up -d
```

### Profile Rules:

| Rule | Description |
|---|---|
| No profile → always starts | Services without `profiles:` always run |
| Profile assigned → opt-in only | Only starts when that profile is activated |
| `docker compose run` → auto-activates | Running a specific service activates its profile |
| Multiple profiles allowed | A service can belong to multiple profiles |

### Visual Summary:

```
docker compose up -d
├── backend          ✅ (no profile — always runs)
├── db               ✅ (no profile — always runs)
├── pgadmin          ❌ (profile: debug — NOT activated)
├── prometheus       ❌ (profile: monitoring — NOT activated)
├── grafana          ❌ (profile: monitoring — NOT activated)
├── migrate          ❌ (profile: tools — NOT activated)
└── seed             ❌ (profile: tools — NOT activated)

docker compose --profile debug up -d
├── backend          ✅
├── db               ✅
├── pgadmin          ✅ (profile: debug — ACTIVATED)
├── prometheus       ❌
├── grafana          ❌
├── migrate          ❌
└── seed             ❌

docker compose --profile debug --profile monitoring up -d
├── backend          ✅
├── db               ✅
├── pgadmin          ✅ (debug — ACTIVATED)
├── prometheus       ✅ (monitoring — ACTIVATED)
├── grafana          ✅ (monitoring — ACTIVATED)
├── migrate          ❌
└── seed             ❌
```

## 3️⃣ `extends` — Reuse Service Configurations

### The Problem:

You have multiple services that share almost identical configurations:

```yaml
# Without extends — so much copy-paste! 😫
services:
  worker-email:
    build: ./backend
    environment:
      DATABASE_URL: postgresql://user:pass@db:5432/mydb
      REDIS_URL: redis://redis:6379
      LOG_LEVEL: info
    depends_on:
      - db
      - redis
    restart: unless-stopped
    command: npm run worker:email

  worker-payment:
    build: ./backend                       # Same!
    environment:
      DATABASE_URL: postgresql://user:pass@db:5432/mydb   # Same!
      REDIS_URL: redis://redis:6379                       # Same!
      LOG_LEVEL: info                                     # Same!
    depends_on:                            # Same!
      - db
      - redis
    restart: unless-stopped                # Same!
    command: npm run worker:payment        # Only this is different!

  worker-notification:
    build: ./backend                       # Same again!
    # ... you get the point 😫
```

### Solution: `extends` — Define Once, Reuse Everywhere

```yaml
# compose.yml — Using extends

services:
  # ──── Base service (template) ────
  worker-base:
    build: ./backend
    environment:
      DATABASE_URL: postgresql://user:pass@db:5432/mydb
      REDIS_URL: redis://redis:6379
      LOG_LEVEL: info
    depends_on:
      - db
      - redis
    restart: unless-stopped
    profiles:
      - do-not-start           # Don't start this — it's just a template!

  # ──── Actual services that extend the base ────
  worker-email:
    extends:
      service: worker-base     # ← Inherit everything from worker-base
    command: npm run worker:email

  worker-payment:
    extends:
      service: worker-base
    command: npm run worker:payment

  worker-notification:
    extends:
      service: worker-base
    command: npm run worker:notification
    environment:
      LOG_LEVEL: debug         # ← Override one specific value
```

**So much cleaner!** Each worker inherits the entire base config and only specifies what's different.

### Extending from a Different File:

You can also extend from services defined in another file:

```yaml
# common-services.yml — Shared service definitions

services:
  web-base:
    build:
      context: .
    environment:
      LOG_FORMAT: json
      LOG_LEVEL: info
    restart: unless-stopped
    deploy:
      resources:
        limits:
          memory: 512M
```

```yaml
# compose.yml — Reference the external file

services:
  frontend:
    extends:
      file: common-services.yml    # ← From this file
      service: web-base            # ← This service
    ports:
      - "3000:80"
    command: nginx -g "daemon off;"

  api:
    extends:
      file: common-services.yml
      service: web-base
    ports:
      - "8080:8080"
    command: npm start
    environment:
      DATABASE_URL: postgresql://user:pass@db:5432/mydb
```

### `extends` Rules:

| Rule | Detail |
|---|---|
| **Inherits** | All configuration from the parent |
| **Overrides** | Child values override parent values |
| **Merges** | Maps (like `environment`) are merged; child values win |
| **Replaces** | Lists (like `ports`) from child replace parent lists |
| **Cannot extend** | `links`, `volumes_from`, `depends_on` are NOT inherited |
| **No circular extends** | Service A cannot extend B which extends A |

## 4️⃣ YAML Anchors — Another Way to Avoid Repetition

YAML itself has a feature called **anchors** (`&`) and **aliases** (`*`) that let you reuse blocks:

```yaml
# compose.yml — Using YAML anchors

# Define an anchor (template)
x-common-env: &common-env
  DATABASE_URL: postgresql://user:pass@db:5432/mydb
  REDIS_URL: redis://redis:6379
  LOG_LEVEL: info
  NODE_ENV: production

x-common-restart: &common-restart
  restart: unless-stopped

x-common-healthcheck: &default-healthcheck
  interval: 10s
  timeout: 5s
  retries: 3

services:
  backend:
    build: ./backend
    <<: *common-restart                    # ← Merge restart config
    environment:
      <<: *common-env                      # ← Merge all common env vars
      PORT: "8080"                         # ← Add extra env var
    healthcheck:
      test: ["CMD-SHELL", "curl -f http://localhost:8080/health"]
      <<: *default-healthcheck             # ← Merge healthcheck timing

  worker-email:
    build: ./backend
    <<: *common-restart
    environment:
      <<: *common-env
      WORKER_TYPE: email
    command: npm run worker:email

  worker-payment:
    build: ./backend
    <<: *common-restart
    environment:
      <<: *common-env
      WORKER_TYPE: payment
    command: npm run worker:payment
```

### How Anchors & Aliases Work:

```yaml
# Step 1: Define an anchor with &name
x-my-template: &my-template      # &my-template creates the anchor
  key1: value1
  key2: value2

# Step 2: Use the anchor with *name
services:
  myservice:
    <<: *my-template              # *my-template references the anchor
                                  # << is the YAML merge key
    key3: value3                  # Add more keys
    key2: override-value          # Override a key from the anchor
```

| Symbol | Purpose | Example |
|---|---|---|
| `&name` | Create an anchor (save a block) | `&common-env` |
| `*name` | Use an alias (reference the block) | `*common-env` |
| `<<:` | Merge key (merge the block into current mapping) | `<<: *common-env` |
| `x-` prefix | Extension field (ignored by Compose, used for anchors) | `x-common-env:` |

> **The `x-` prefix** is a Compose convention. Any top-level key starting with `x-` is treated as an extension and **ignored** by Compose. It's the perfect place for anchor definitions.

### `extends` vs YAML Anchors:

| Feature | `extends` | YAML Anchors |
|---|---|---|
| Scope | Compose-specific | Standard YAML |
| Can reference other files | ✅ Yes | ❌ No |
| Handles `depends_on` | ❌ Not inherited | Works (it's just YAML) |
| Merging behavior | Compose-aware (smart merge) | Simple YAML merge |
| Use when | Extending full service configs | Reusing small config blocks |

## 5️⃣ `deploy` — Resource Limits & Placement

Control how many resources each container can use:

```yaml
services:
  backend:
    build: ./backend
    deploy:
      resources:
        limits:
          cpus: "1.0"           # Max 1 CPU core
          memory: 512M          # Max 512 MB RAM
        reservations:
          cpus: "0.25"          # Reserve 0.25 CPU cores
          memory: 128M          # Reserve 128 MB RAM
      replicas: 3               # Run 3 instances of this service
```

| Setting | Meaning |
|---|---|
| `limits.cpus` | Maximum CPU cores the container can use |
| `limits.memory` | Maximum RAM (e.g., `256M`, `1G`) |
| `reservations.cpus` | Guaranteed minimum CPU |
| `reservations.memory` | Guaranteed minimum RAM |
| `replicas` | Number of container instances |

> ⚠️ **Note:** `deploy.replicas` is supported in Compose V2, but some `deploy` options are only fully implemented in Docker Swarm mode.

### Restart Policy (Under deploy):

```yaml
services:
  web:
    deploy:
      restart_policy:
        condition: on-failure
        delay: 5s             # Wait 5s between restart attempts
        max_attempts: 3       # Give up after 3 attempts
        window: 120s          # Consider it stable after 120s
```

## 6️⃣ `watch` — Auto-Rebuild on File Changes (Compose Watch)

This is a newer feature (Compose v2.22+) that watches your source files and automatically rebuilds/restarts when things change:

```yaml
services:
  backend:
    build: ./backend
    develop:
      watch:
        # Sync files without rebuilding (for source code)
        - action: sync
          path: ./backend/src
          target: /app/src

        # Sync + restart container (for config changes)
        - action: sync+restart
          path: ./backend/config
          target: /app/config

        # Full rebuild (for dependency changes)
        - action: rebuild
          path: ./backend/package.json
```

### Watch Actions:

| Action | What Happens | Use For |
|---|---|---|
| `sync` | Files are synced to the container (no restart) | Source code with hot reload |
| `sync+restart` | Files synced, then container restarts | Config files |
| `rebuild` | Entire image is rebuilt and container recreated | Dependency files (`package.json`, `requirements.txt`) |

### Using Watch:

```bash
# Start services with file watching
docker compose watch

# Or start in background with watch
docker compose up -d
docker compose watch    # In another terminal
```

## 7️⃣ `configs` and `secrets` — Managing Configuration & Secrets

### Configs — External Configuration Files:

```yaml
services:
  web:
    image: nginx:alpine
    configs:
      - source: nginx_conf           # Reference the config defined below
        target: /etc/nginx/nginx.conf # Where to mount it in the container
        mode: 0444                    # Read-only permissions

configs:
  nginx_conf:
    file: ./nginx/nginx.conf          # Source file on host
```

### Secrets — Sensitive Data:

```yaml
services:
  db:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD_FILE: /run/secrets/db_password    # Read password from file
    secrets:
      - db_password                   # Mount this secret

  backend:
    build: ./backend
    secrets:
      - db_password
      - api_key

secrets:
  db_password:
    file: ./secrets/db_password.txt   # File containing the secret
  api_key:
    environment: MY_API_KEY           # Read from host environment variable
```

**How secrets work in the container:**

```bash
# Secrets are mounted as files in /run/secrets/
docker compose exec backend cat /run/secrets/db_password
# Output: mysupersecretpassword

docker compose exec backend cat /run/secrets/api_key
# Output: sk-abc123xyz
```

> **Why use secrets instead of environment variables?**
> - Environment variables can leak into logs, error reports, and child processes
> - Secrets are mounted as files — more secure
> - Many images support `_FILE` variants (e.g., `POSTGRES_PASSWORD_FILE`)

## 8️⃣ `logging` — Configure Container Logging

Control how containers log their output:

```yaml
services:
  backend:
    build: ./backend
    logging:
      driver: json-file              # Default Docker log driver
      options:
        max-size: "10m"              # Max 10 MB per log file
        max-file: "3"                # Keep 3 rotated log files
        compress: "true"             # Compress rotated files

  # Or use no logging for noisy services
  noisy-service:
    image: some-noisy-image
    logging:
      driver: "none"                 # Disable logging completely
```

### Common Log Drivers:

| Driver | Description |
|---|---|
| `json-file` | Default — JSON formatted, stored on disk |
| `syslog` | Send to syslog |
| `fluentd` | Send to Fluentd |
| `awslogs` | Send to AWS CloudWatch |
| `none` | Disable logging |

## 9️⃣ `init` — Proper Signal Handling

If your container's main process doesn't handle signals properly (like `SIGTERM`), use `init`:

```yaml
services:
  backend:
    build: ./backend
    init: true    # ← Adds a tiny init process (tini) as PID 1
```

**What this does:**
- Runs `tini` as PID 1 (instead of your app)
- `tini` properly forwards signals to your app
- Reaps zombie processes
- Ensures clean shutdown when you run `docker compose down`

> **When to use:** Always a good idea for containers running Node.js, Python, or other interpreted languages that might not handle signals correctly.

## 🔟 `extra_hosts` — Custom DNS Entries

Add custom entries to the container's `/etc/hosts`:

```yaml
services:
  backend:
    build: ./backend
    extra_hosts:
      - "host.docker.internal:host-gateway"    # Access host machine
      - "api.local:192.168.1.100"              # Custom DNS mapping
```

`host-gateway` is a special value that resolves to the host machine's IP — useful when your container needs to access services running on the host.

Let's put it all together — a production-ready compose file using all the advanced features:

```yaml
# compose.yml — Production-Ready Full Stack App

# ──── Reusable Blocks ────
x-common-env: &common-env
  DATABASE_URL: postgresql://${DB_USER}:${DB_PASSWORD}@db:5432/${DB_NAME}
  REDIS_URL: redis://redis:6379
  LOG_LEVEL: ${LOG_LEVEL:-info}
  NODE_ENV: ${NODE_ENV:-production}

x-healthcheck-defaults: &healthcheck-defaults
  interval: 10s
  timeout: 5s
  retries: 5

# ──── Services ────
services:

  # ===== CORE SERVICES (always run) =====

  backend:
    build:
      context: ./backend
      dockerfile: Dockerfile
    ports:
      - "${APP_PORT:-8080}:8080"
    environment:
      <<: *common-env
      PORT: "8080"
    depends_on:
      db:
        condition: service_healthy
      redis:
        condition: service_healthy
      migrate:
        condition: service_completed_successfully
    healthcheck:
      test: ["CMD-SHELL", "curl -f http://localhost:8080/health || exit 1"]
      <<: *healthcheck-defaults
      start_period: 15s
    restart: unless-stopped
    init: true
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "5"
    deploy:
      resources:
        limits:
          memory: 512M
          cpus: "1.0"

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: ${DB_USER}
      POSTGRES_PASSWORD: ${DB_PASSWORD}
      POSTGRES_DB: ${DB_NAME}
    volumes:
      - pgdata:/var/lib/postgresql/data
      - ./db/init.sql:/docker-entrypoint-initdb.d/init.sql:ro
    ports:
      - "127.0.0.1:5432:5432"
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${DB_USER} -d ${DB_NAME}"]
      <<: *healthcheck-defaults
      start_period: 20s
    restart: unless-stopped
    deploy:
      resources:
        limits:
          memory: 1G

  redis:
    image: redis:7-alpine
    command: redis-server --appendonly yes --maxmemory 256mb --maxmemory-policy allkeys-lru
    volumes:
      - redisdata:/data
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      <<: *healthcheck-defaults
    restart: unless-stopped

  # ===== WORKER SERVICES =====

  worker-email:
    build: ./backend
    command: npm run worker:email
    environment:
      <<: *common-env
      WORKER_TYPE: email
    depends_on:
      db:
        condition: service_healthy
      redis:
        condition: service_healthy
    restart: unless-stopped
    init: true

  worker-jobs:
    build: ./backend
    command: npm run worker:jobs
    environment:
      <<: *common-env
      WORKER_TYPE: jobs
    depends_on:
      db:
        condition: service_healthy
      redis:
        condition: service_healthy
    restart: unless-stopped
    init: true

  # ===== ONE-TIME TASKS (profiles: tools) =====

  migrate:
    build: ./backend
    command: npm run migrate
    environment:
      <<: *common-env
    depends_on:
      db:
        condition: service_healthy
    profiles:
      - tools
    restart: "no"

  seed:
    build: ./backend
    command: npm run seed
    environment:
      <<: *common-env
    depends_on:
      db:
        condition: service_healthy
      migrate:
        condition: service_completed_successfully
    profiles:
      - tools
    restart: "no"

  # ===== DEBUG TOOLS (profiles: debug) =====

  pgadmin:
    image: dpage/pgadmin4:latest
    environment:
      PGADMIN_DEFAULT_EMAIL: admin@admin.com
      PGADMIN_DEFAULT_PASSWORD: admin
    ports:
      - "5050:80"
    depends_on:
      db:
        condition: service_healthy
    profiles:
      - debug

  redis-commander:
    image: rediscommander/redis-commander:latest

      REDIS_HOSTS: local:redis:6379
    ports:
      - "8081:8081"

  # ===== MONITORING (profiles: monitoring) =====

  prometheus:
    image: prom/prometheus:latest
    volumes:
      - ./monitoring/prometheus.yml:/etc/prometheus/prometheus.yml:ro

    ports:
      - "9090:9090"
    profiles:
      - monitoring

  grafana:
    image: grafana/grafana:latest
    volumes:
      - grafana-data:/var/lib/grafana
    ports:
      - "3000:3000"
    depends_on:
      - prometheus
    profiles:
      - monitoring

# ──── Volumes ────
volumes:
  pgdata:
  redisdata:
  prometheus-data:
  grafana-data:
```

## 🎯 Summary

| Feature | What It Does | When to Use |
|---|---|---|

| Healthchecks | Define how to check if service is healthy | Every critical service |

| `extends` | Inherit service configuration | Multiple similar services |
| YAML Anchors | Reuse YAML blocks | Shared env vars, settings |
| `deploy` | Resource limits and replicas | Production resource control |
| `watch` | Auto-rebuild on changes | Development workflow |
| `configs`/`secrets` | Manage configs and sensitive data | Production secrets handling |
| `logging` | Control log behavior | Production log management |
| `init` | Proper signal handling | Interpreted language apps |
