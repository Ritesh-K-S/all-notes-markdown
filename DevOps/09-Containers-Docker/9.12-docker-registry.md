# 9.12 — Docker Registry — Docker Hub, Private Registry, Harbor, ECR, ACR, GCR

> **One Line Answer:** A Docker Registry is a **storage and distribution system for Docker images** — think of it as a "warehouse" where you store your container images so that any machine can pull and run them. Docker Hub is the default public one, but in production you'll often use **private registries** like Harbor, ECR, ACR, or GCR.

## 🤔 Why Do We Need a Registry?

Let's think about it with a simple story:

```
You build a Docker image on your laptop:
  docker build -t myapp:v1.0 .

Now you want to run it on:
  ✅ Your laptop          → Easy, it's already here
  ❌ Your colleague's laptop → How do they get it?
  ❌ A staging server      → How does the server get it?
  ❌ 50 production servers → How do ALL of them get it?

You can't just copy the image file to 50 servers manually!
```

**Solution: Push the image to a Registry. Everyone pulls from there.**

```
                          ┌─────────────┐
  docker push ──────────▶ │  REGISTRY   │ ◀────────── docker push
  (your laptop)           │             │             (CI/CD pipeline)
                          │  Stores all │
                          │  your images│
                          └──────┬──────┘
                                 │
              ┌──────────────────┼──────────────────┐
              │                  │                  │
              ▼                  ▼                  ▼
        docker pull        docker pull        docker pull
        (colleague)        (staging)          (50 prod servers)
```

## 📖 How Docker Registries Work

### Key Concepts:

| Term | What It Means | Example |
|---|---|---|
| **Registry** | The server that stores images | Docker Hub, ECR, Harbor |
| **Repository** | A collection of related images (like a Git repo) | `mycompany/backend` |
| **Tag** | A version label for an image | `v1.0`, `latest`, `dev` |
| **Image Reference** | The full address of an image | `registry.example.com/mycompany/backend:v1.0` |
| **Manifest** | Metadata describing an image (layers, architecture, etc.) | JSON document |
| **Blob** | The actual image layer data | Binary data |

### Image Reference — The Full Address:

Every Docker image has a full address (reference) made of these parts:

```
registry.example.com / mycompany / backend : v1.0
────────────────────   ─────────   ───────   ────
     Registry            Namespace  Repo      Tag
     (hostname)          (org/user) (name)    (version)
```

When you just write `nginx:alpine`, Docker expands it to:

```
nginx:alpine
  → docker.io/library/nginx:alpine
    ─────────  ───────  ─────  ──────
    Registry   Namespace Repo   Tag
    (Docker Hub) (official) (nginx) (alpine)
```

### The Push/Pull Flow:

```
Building & Pushing:
┌──────────┐    docker build     ┌──────────────┐    docker push     ┌──────────┐
│          │ ──────────────────▶ │  Local Image  │ ──────────────────▶│ Registry │
│  Source  │                     │   Store       │                     │          │
│  Code    │                     │  (your machine│                     │ (remote) │
│  +       │                     │   only)       │                     │          │
│Dockerfile│                     └──────────────┘                     └──────────┘
└──────────┘

Pulling & Running:
┌──────────┐    docker pull      ┌──────────────┐    docker run      ┌──────────┐
│ Registry │ ──────────────────▶ │  Local Image  │ ──────────────────▶│ Running  │
│          │                     │   Store       │                     │Container │
│ (remote) │                     │  (any machine)│                     │          │
└──────────┘                     └──────────────┘                     └──────────┘
```

## 1️⃣ Docker Hub — The Default Public Registry

Docker Hub is the **default registry** for Docker. When you type `docker pull nginx`, it goes to Docker Hub.

### What Docker Hub Offers:

| Feature | Free | Pro ($5/mo) | Team ($7/user/mo) |
|---|---|---|---|
| Public Repos | Unlimited | Unlimited | Unlimited |
| Private Repos | 1 | Unlimited | Unlimited |
| Pulls (per 6h) | 100 (anonymous), 200 (authenticated) | Unlimited | Unlimited |
| Automated Builds | ❌ | ✅ | ✅ |
| Vulnerability Scanning | Limited | ✅ | ✅ |
| Team Management | ❌ | ❌ | ✅ |

> **Note:** Docker Hub rate limits unauthenticated pulls to 100 per 6 hours per IP. This can bite you in CI/CD! Always authenticate.

### Using Docker Hub:

#### Step 1: Create an Account
Go to [hub.docker.com](https://hub.docker.com) and sign up.

#### Step 2: Login from CLI

```bash
# Login to Docker Hub
docker login

# It will prompt:
# Username: yourusername
# Password: your-password-or-access-token

# You'll see: Login Succeeded
```

> **Security Tip:** Use **Access Tokens** instead of your password! Go to Docker Hub → Account Settings → Security → New Access Token.

#### Step 3: Tag Your Image

```bash
# Docker Hub image format: username/repository:tag

# Tag an existing image
docker tag myapp:v1.0 yourusername/myapp:v1.0
docker tag myapp:v1.0 yourusername/myapp:latest

# Or build with the correct tag directly
docker build -t yourusername/myapp:v1.0 .
```

#### Step 4: Push to Docker Hub

```bash
# Push a specific tag
docker push yourusername/myapp:v1.0

# Push all tags for a repository
docker push yourusername/myapp --all-tags
```

#### Step 5: Pull from Docker Hub (on another machine)

```bash
# Anyone can pull public images
docker pull yourusername/myapp:v1.0

# Or just run it (auto-pulls if not local)
docker run yourusername/myapp:v1.0
```

### Docker Hub — Organizations & Teams:

```
Organization: mycompany
├── Team: backend-devs
│   ├── User: alice     (can push to backend repos)
│   └── User: bob
├── Team: frontend-devs
│   ├── User: charlie   (can push to frontend repos)
│   └── User: dave
└── Repositories:
    ├── mycompany/backend       ← backend-devs have push access
    ├── mycompany/frontend      ← frontend-devs have push access
    └── mycompany/shared-lib    ← both teams can pull
```

### Docker Hub Automated Builds (Webhook):

You can connect Docker Hub to GitHub/GitLab so that every push to your repo automatically builds a new image:

```
GitHub Push → Docker Hub Webhook → Auto-Build Image → Image Available
```

> **Note:** Most teams now use CI/CD pipelines (GitHub Actions, GitLab CI, Jenkins) for building images instead of Docker Hub's built-in automated builds — more flexibility.

## 2️⃣ Docker's Official Registry — Self-Hosted

Docker provides an open-source registry you can run yourself. It's called **Distribution** (formerly `docker/registry`).

### Why Self-Host?

| Reason | Details |
|---|---|
| **Privacy** | Keep images internal — not on a third-party service |
| **Speed** | Images don't travel over the internet (fast pulls on LAN) |
| **No Rate Limits** | No pull limits whatsoever |
| **Compliance** | Some regulations require data to stay in your infrastructure |
| **Cost** | Free (just need a server) |

### Run a Basic Registry:

```bash
# Start a registry on port 5000
docker run -d \
  --name registry \
  -p 5000:5000 \
  -v registry-data:/var/lib/registry \
  registry:2
```

That's it! You now have a private registry running on `localhost:5000`.

### Push/Pull from Your Registry:

```bash
# Tag image for your registry
docker tag myapp:v1.0 localhost:5000/myapp:v1.0

# Push to your registry
docker push localhost:5000/myapp:v1.0

# Pull from your registry (on another machine, replace localhost with server IP)
docker pull your-server-ip:5000/myapp:v1.0
```

### Add TLS/HTTPS (Production Required!):

For production, you MUST use HTTPS. Without it, Docker will refuse to push/pull (unless you configure "insecure registries").

```bash
# Generate self-signed certificate (for testing)
mkdir -p certs
openssl req -newkey rsa:4096 -nodes -sha256 \
  -keyout certs/domain.key \
  -x509 -days 365 \
  -subj "/CN=registry.mycompany.com" \
  -out certs/domain.crt

# Run registry with TLS
docker run -d \
  --name registry \
  -p 443:443 \
  -v registry-data:/var/lib/registry \
  -v $(pwd)/certs:/certs \
  -e REGISTRY_HTTP_ADDR=0.0.0.0:443 \
  -e REGISTRY_HTTP_TLS_CERTIFICATE=/certs/domain.crt \
  -e REGISTRY_HTTP_TLS_KEY=/certs/domain.key \
  registry:2
```

### Add Authentication:

```bash
# Create password file using htpasswd
mkdir auth
docker run --rm --entrypoint htpasswd \
  httpd:2 -Bbn myuser mypassword > auth/htpasswd

# Run registry with auth
docker run -d \
  --name registry \
  -p 5000:5000 \
  -v registry-data:/var/lib/registry \
  -v $(pwd)/auth:/auth \
  -v $(pwd)/certs:/certs \
  -e REGISTRY_AUTH=htpasswd \
  -e REGISTRY_AUTH_HTPASSWD_REALM="My Registry" \
  -e REGISTRY_AUTH_HTPASSWD_PATH=/auth/htpasswd \
  -e REGISTRY_HTTP_TLS_CERTIFICATE=/certs/domain.crt \
  -e REGISTRY_HTTP_TLS_KEY=/certs/domain.key \
  registry:2
```

### Self-Hosted Registry with Docker Compose:

```yaml
# compose.yml — Private Docker Registry

services:
  registry:
    image: registry:2
    ports:
      - "5000:5000"
    volumes:
      - registry-data:/var/lib/registry
      - ./certs:/certs:ro
      - ./auth:/auth:ro
    environment:
      REGISTRY_AUTH: htpasswd
      REGISTRY_AUTH_HTPASSWD_REALM: "My Docker Registry"
      REGISTRY_AUTH_HTPASSWD_PATH: /auth/htpasswd
      REGISTRY_HTTP_TLS_CERTIFICATE: /certs/domain.crt
      REGISTRY_HTTP_TLS_KEY: /certs/domain.key
      REGISTRY_STORAGE_DELETE_ENABLED: "true"    # Allow image deletion
    restart: unless-stopped

  # Optional: Web UI for browsing images
  registry-ui:
    image: joxit/docker-registry-ui:latest
    ports:
      - "8080:80"
    environment:
      REGISTRY_TITLE: "My Docker Registry"
      REGISTRY_URL: https://registry:5000
      SINGLE_REGISTRY: "true"
    depends_on:
      - registry

volumes:
  registry-data:
```

### Registry API — Useful Endpoints:

The registry has a REST API you can query:

```bash
# Check if registry is up
curl https://registry.mycompany.com/v2/

# List all repositories (images)
curl https://registry.mycompany.com/v2/_catalog
# {"repositories": ["myapp", "frontend", "backend"]}

# List tags for a repository
curl https://registry.mycompany.com/v2/myapp/tags/list
# {"name": "myapp", "tags": ["v1.0", "v1.1", "latest"]}

# With authentication
curl -u myuser:mypassword https://registry.mycompany.com/v2/_catalog
```

## 3️⃣ Harbor — Enterprise-Grade Registry

Harbor is an **open-source enterprise container registry** built by VMware. Think of it as Docker's official registry on steroids.

### Why Harbor?

| Feature | Docker Registry | Harbor |
|---|---|---|
| Image Storage | ✅ | ✅ |
| Web UI | ❌ (need add-on) | ✅ Beautiful UI |
| Vulnerability Scanning | ❌ | ✅ (Trivy built-in) |
| RBAC (Role-Based Access) | ❌ (basic auth only) | ✅ Full RBAC |
| Image Signing | ❌ | ✅ (Cosign/Notary) |
| Replication | ❌ | ✅ (between registries) |
| Garbage Collection | Manual | ✅ Automated |
| LDAP/AD Integration | ❌ | ✅ |
| Audit Logs | ❌ | ✅ |
| Quotas | ❌ | ✅ (per project) |
| Robot Accounts | ❌ | ✅ (for CI/CD) |
| OCI Artifact Support | Basic | ✅ (Helm charts, etc.) |

### Install Harbor:

```bash
# Download Harbor installer
wget https://github.com/goharbor/harbor/releases/download/v2.10.0/harbor-offline-installer-v2.10.0.tgz

# Extract
tar xzvf harbor-offline-installer-v2.10.0.tgz
cd harbor

# Copy and edit the config
cp harbor.yml.tmpl harbor.yml
```

Edit `harbor.yml`:
```yaml
# harbor.yml — Key settings

hostname: registry.mycompany.com     # Your domain

# HTTPS configuration
https:
  port: 443
  certificate: /path/to/cert.pem
  private_key: /path/to/key.pem

# Initial admin password (change after first login!)
harbor_admin_password: Harbor12345

# Database
database:
  password: root123
  max_idle_conns: 50
  max_open_conns: 1000

# Storage
data_volume: /data                   # Where images are stored
```

```bash
# Run the installer
sudo ./install.sh

# With vulnerability scanning (Trivy)
sudo ./install.sh --with-trivy
```

### Using Harbor:

```bash
# Login to Harbor
docker login registry.mycompany.com
# Username: admin
# Password: Harbor12345

# Tag image for Harbor
# Format: harbor-host/project-name/image-name:tag
docker tag myapp:v1.0 registry.mycompany.com/myproject/myapp:v1.0

# Push to Harbor
docker push registry.mycompany.com/myproject/myapp:v1.0

# Pull from Harbor
docker pull registry.mycompany.com/myproject/myapp:v1.0
```

### Harbor Projects & RBAC:

```
Harbor
├── Project: backend-services
│   ├── Members:
│   │   ├── alice (Project Admin)     → Push, Pull, Delete, Manage
│   │   ├── bob (Developer)          → Push, Pull
│   │   └── ci-robot (Robot Account) → Push, Pull (for CI/CD)
│   ├── Images:
│   │   ├── backend-services/api:v2.1
│   │   ├── backend-services/worker:v2.1
│   │   └── backend-services/migration:v2.1
│   └── Settings:
│       ├── Vulnerability Scanning: Auto-scan on push ✅
│       ├── Storage Quota: 50 GB
│       └── Image Signing: Required ✅
│
├── Project: frontend
│   ├── Members: ...
│   └── Images:
│       └── frontend/webapp:v3.0
│
└── Project: shared-libs (Public)
    └── Images:
        ├── shared-libs/base-node:18
        └── shared-libs/base-python:3.12
```

### Harbor Robot Accounts (for CI/CD):

Instead of using personal credentials in CI/CD pipelines, create robot accounts:

```
Harbor UI → Project → Robot Accounts → New Robot Account
  Name: ci-pipeline
  Permissions: Push, Pull
  Expiration: 365 days
  → Generates: robot$ci-pipeline + secret token
```

```bash
# Use in CI/CD
docker login registry.mycompany.com \
  -u 'robot$ci-pipeline' \
  -p 'eyJhbGciOiJSUzI1NiIs...'
```

### Harbor Replication:

Harbor can replicate images between registries:

```
┌─────────────────┐          Replication          ┌─────────────────┐
│  Harbor (US)    │ ◀────────────────────────────▶ │  Harbor (EU)    │
│  Primary        │                                │  Replica        │
│                 │     Push to US → Auto-sync    │                 │
│  myapp:v1.0  ──│──────────────────────────────▶│── myapp:v1.0   │
└─────────────────┘                                └─────────────────┘

Also supports replication to/from:
  - Docker Hub
  - AWS ECR
  - Google GCR / Artifact Registry
  - Azure ACR
  - Other Harbor instances
```

## 4️⃣ AWS ECR — Elastic Container Registry

ECR is Amazon's managed Docker registry service. It's deeply integrated with AWS services.

### Key Features:

| Feature | Details |
|---|---|
| Fully Managed | AWS handles scaling, availability, security |
| IAM Integration | Use AWS IAM for access control (no separate users) |
| Encryption | Images encrypted at rest (AWS KMS) |
| Vulnerability Scanning | Built-in (Amazon Inspector) |
| Lifecycle Policies | Auto-delete old/untagged images |
| Cross-Region Replication | Replicate images across AWS regions |
| Pricing | $0.10/GB/month storage + $0.09/GB data transfer |

### Types of ECR:

| Type | Scope | Use Case |
|---|---|---|
| **ECR Private** | Your AWS account only | Production images |
| **ECR Public** | Anyone can pull | Open source projects (like Docker Hub) |

### Setting Up ECR:

#### Using AWS CLI:

```bash
# Create a private repository
aws ecr create-repository \
  --repository-name myapp/backend \
  --region us-east-1 \
  --image-scanning-configuration scanOnPush=true \
  --encryption-configuration encryptionType=AES256

# Output:
# {
#   "repository": {
#     "repositoryUri": "123456789012.dkr.ecr.us-east-1.amazonaws.com/myapp/backend"
#   }
# }
```

#### Login to ECR:

```bash
# Get login token and pipe to docker login
aws ecr get-login-password --region us-east-1 | \
  docker login --username AWS --password-stdin \
  123456789012.dkr.ecr.us-east-1.amazonaws.com

# Login Succeeded!
```

> **Important:** ECR login tokens expire after **12 hours**. You need to re-authenticate in your CI/CD pipelines.

#### Tag & Push:

```bash
# ECR image format:
# ACCOUNT_ID.dkr.ecr.REGION.amazonaws.com/REPOSITORY:TAG

# Tag
docker tag myapp:v1.0 \
  123456789012.dkr.ecr.us-east-1.amazonaws.com/myapp/backend:v1.0

# Push
docker push \
  123456789012.dkr.ecr.us-east-1.amazonaws.com/myapp/backend:v1.0
```

#### Pull:

```bash
docker pull \
  123456789012.dkr.ecr.us-east-1.amazonaws.com/myapp/backend:v1.0
```

### ECR Lifecycle Policies:

Automatically clean up old images to save storage costs:

```json
{
  "rules": [
    {
      "rulePriority": 1,
      "description": "Delete untagged images older than 7 days",
      "selection": {
        "tagStatus": "untagged",
        "countType": "sinceImagePushed",
        "countUnit": "days",
        "countNumber": 7
      },
      "action": {
        "type": "expire"
      }
    },
    {
      "rulePriority": 2,
      "description": "Keep only the last 10 tagged images",
      "selection": {
        "tagStatus": "tagged",
        "tagPatternList": ["v*"],
        "countType": "imageCountMoreThan",
        "countNumber": 10
      },
      "action": {
        "type": "expire"
      }
    }
  ]
}
```

```bash
# Apply the lifecycle policy
aws ecr put-lifecycle-policy \
  --repository-name myapp/backend \
  --lifecycle-policy-text file://lifecycle-policy.json
```

### ECR in CI/CD (GitHub Actions Example):

```yaml
# .github/workflows/build.yml
name: Build and Push to ECR

on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: us-east-1

      - name: Login to Amazon ECR
        id: login-ecr
        uses: aws-actions/amazon-ecr-login@v2

      - name: Build, tag, and push image
        env:
          ECR_REGISTRY: ${{ steps.login-ecr.outputs.registry }}
          IMAGE_TAG: ${{ github.sha }}
        run: |
          docker build -t $ECR_REGISTRY/myapp/backend:$IMAGE_TAG .
          docker build -t $ECR_REGISTRY/myapp/backend:latest .
          docker push $ECR_REGISTRY/myapp/backend:$IMAGE_TAG
          docker push $ECR_REGISTRY/myapp/backend:latest
```

## 5️⃣ Azure ACR — Azure Container Registry

ACR is Microsoft Azure's managed container registry.

### Key Features:

| Feature | Details |
|---|---|
| Fully Managed | Azure handles everything |
| Azure AD Integration | Use Azure Active Directory for auth |
| Geo-Replication | Replicate to multiple Azure regions |
| ACR Tasks | Build images in the cloud (no local Docker needed!) |
| Vulnerability Scanning | Microsoft Defender integration |
| Content Trust | Image signing with Notary |
| Private Endpoints | VNet integration |

### Tiers:

| Tier | Storage | Price/month | Best For |
|---|---|---|---|
| Basic | 10 GB | ~$5 | Dev/testing |
| Standard | 100 GB | ~$20 | Small production |
| Premium | 500 GB | ~$50 | Geo-replication, enterprise |

### Setting Up ACR:

```bash
# Create an ACR registry
az acr create \
  --resource-group myResourceGroup \
  --name mycompanyregistry \
  --sku Standard \
  --location eastus

# Registry URL: mycompanyregistry.azurecr.io
```

### Login to ACR:

```bash
# Method 1: Azure CLI (easiest)
az acr login --name mycompanyregistry

# Method 2: Docker login with service principal
docker login mycompanyregistry.azurecr.io \
  --username <service-principal-id> \
  --password <service-principal-password>

# Method 3: Admin user (not recommended for production)
az acr update --name mycompanyregistry --admin-enabled true
az acr credential show --name mycompanyregistry
# Then use the username/password with docker login
```

### Tag, Push, Pull:

```bash
# ACR image format: registryname.azurecr.io/repository:tag

# Tag
docker tag myapp:v1.0 mycompanyregistry.azurecr.io/myapp/backend:v1.0

# Push
docker push mycompanyregistry.azurecr.io/myapp/backend:v1.0

# Pull
docker pull mycompanyregistry.azurecr.io/myapp/backend:v1.0

# List repositories
az acr repository list --name mycompanyregistry

# List tags
az acr repository show-tags --name mycompanyregistry --repository myapp/backend
```

### ACR Tasks — Build in the Cloud:

This is a unique ACR feature — build images **in Azure** without needing Docker locally:

```bash
# Quick build — send Dockerfile + context to Azure, build there
az acr build \
  --registry mycompanyregistry \
  --image myapp/backend:v1.0 \
  ./backend

# This uploads your source code to Azure, builds the image there,
# and pushes it to your ACR — all in one command!
```

### ACR in CI/CD (GitHub Actions):

```yaml
# .github/workflows/build.yml
name: Build and Push to ACR

on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Login to ACR
        uses: azure/docker-login@v2
        with:
          login-server: mycompanyregistry.azurecr.io
          username: ${{ secrets.ACR_USERNAME }}
          password: ${{ secrets.ACR_PASSWORD }}

      - name: Build and push
        run: |
          docker build -t mycompanyregistry.azurecr.io/myapp/backend:${{ github.sha }} .
          docker push mycompanyregistry.azurecr.io/myapp/backend:${{ github.sha }}
```

## 6️⃣ Google GCR & Artifact Registry

Google Cloud has two container registries:

| Service | Status | Notes |
|---|---|---|
| **GCR** (Container Registry) | ⚠️ Deprecated | Uses Google Cloud Storage buckets |
| **Artifact Registry** | ✅ Recommended | Supports containers + other artifacts (npm, Maven, Python) |

### Key Features (Artifact Registry):

| Feature | Details |
|---|---|
| Multi-Format | Docker, Maven, npm, Python, Go, OS packages |
| IAM Integration | Google Cloud IAM for access control |
| Vulnerability Scanning | Automated with Container Analysis |
| VPC Service Controls | Network-level security |
| Regional/Multi-Region | Choose where images are stored |
| Cleanup Policies | Auto-delete old images |

### Setting Up Artifact Registry:

```bash
# Enable the API
gcloud services enable artifactregistry.googleapis.com

# Create a Docker repository
gcloud artifacts repositories create myrepo \
  --repository-format=docker \
  --location=us-central1 \
  --description="My Docker images"

# Repository URL: us-central1-docker.pkg.dev/my-project-id/myrepo
```

### Authentication:

```bash
# Configure Docker to use gcloud for authentication
gcloud auth configure-docker us-central1-docker.pkg.dev

# This adds the registry to Docker's credential helpers
# Now docker push/pull will automatically authenticate
```

### Tag, Push, Pull:

```bash
# Artifact Registry image format:
# LOCATION-docker.pkg.dev/PROJECT_ID/REPOSITORY/IMAGE:TAG

# Tag
```

## 🏷️ Image Tagging Strategies

How you tag your images matters for production reliability:

### Common Tagging Strategies:

```bash
# 1. Semantic Versioning (recommended for releases)
myapp:1.0.0
myapp:1.0.1
myapp:1.1.0
myapp:2.0.0

# 2. Git SHA (recommended for CI/CD — unique & traceable)
myapp:a1b2c3d
myapp:abc123def456

# 3. Branch + Build Number
myapp:main-42
myapp:develop-108

# 4. Date-based
myapp:2025-01-15
myapp:20250115-143022

# 5. Combined (best practice!)
myapp:1.2.3                    # Semantic version
myapp:1.2.3-a1b2c3d            # Version + Git SHA
myapp:latest                    # Always points to newest
```

### The `latest` Tag — A Common Trap:

```
⚠️ WARNING: "latest" does NOT mean "most recent"!

"latest" is just a tag name. It's the DEFAULT tag when no tag is specified.

docker build -t myapp .         → myapp:latest  (tagged as latest by default)
docker build -t myapp:v2.0 .    → myapp:v2.0    (latest is NOT updated!)

Now "latest" points to the OLD image, not v2.0!
```

**Best Practice:** Don't rely on `latest` in production. Always use specific version tags.

### Multi-Tag Pushing:

```bash
# Tag with multiple tags, then push all
docker tag myapp:v1.0 myregistry.com/myapp:v1.0
docker tag myapp:v1.0 myregistry.com/myapp:v1
docker tag myapp:v1.0 myregistry.com/myapp:latest

docker push myregistry.com/myapp:v1.0
docker push myregistry.com/myapp:v1
docker push myregistry.com/myapp:latest
# All three tags point to the same image (same layers, no extra storage!)
```

## 🔒 Registry Security Best Practices

### 1. Always Use HTTPS

```bash
# ❌ BAD — insecure registry
docker push http://registry.example.com:5000/myapp:v1.0

# ✅ GOOD — TLS encrypted
docker push https://registry.example.com/myapp:v1.0
```

### 2. Use Service Accounts / Robot Accounts for CI/CD

```bash
# ❌ BAD — using personal credentials in CI/CD
docker login -u alice -p alices-password registry.example.com

# ✅ GOOD — using dedicated service account
docker login -u 'robot$ci-pipeline' -p $CI_ROBOT_TOKEN registry.example.com
```

### 3. Scan Images for Vulnerabilities

```bash
# Using Trivy (open source scanner)
trivy image myapp:v1.0

# Using Docker Scout (built into Docker)
docker scout cves myapp:v1.0

# Using Grype (Anchore)
grype myapp:v1.0
```

### 4. Sign Your Images

```bash
# Using Cosign (Sigstore)
# Generate a key pair
cosign generate-key-pair

# Sign an image
cosign sign --key cosign.key myregistry.com/myapp:v1.0

# Verify a signature
cosign verify --key cosign.pub myregistry.com/myapp:v1.0
```

### 5. Use Immutable Tags

Once pushed, a tag can't be overwritten. This prevents someone from replacing `v1.0` with a different (potentially malicious) image.

```bash
# ECR — Enable immutable tags
aws ecr put-image-tag-mutability \
  --repository-name myapp \
  --image-tag-mutability IMMUTABLE

# Harbor — Enable tag immutability in project settings
# ACR — Use tag locking
```

### 6. Clean Up Unused Images

```bash
# List dangling (untagged) images locally
docker image prune

# Delete old images from registry (varies by registry)
# ECR: Lifecycle policies
# ACR: Retention policies
# Harbor: Garbage collection + tag retention
```

## 🎯 Summary

| Topic | Key Takeaway |
|---|---|
| **What is a Registry** | Storage and distribution system for Docker images |
| **Docker Hub** | Default public registry — good for open source, has rate limits |
| **Self-Hosted Registry** | Free, private, fast on LAN — but YOU manage it |
| **Harbor** | Enterprise self-hosted — UI, scanning, RBAC, replication |
| **AWS ECR** | Managed registry for AWS — IAM integration, lifecycle policies |
| **Azure ACR** | Managed registry for Azure — ACR Tasks can build in cloud |
| **Google Artifact Registry** | Managed for GCP — supports Docker + npm/Maven/Python |
| **GHCR** | Free for public repos, perfect with GitHub Actions |
| **Tagging** | Use semantic versions + git SHA; don't rely on `latest` |
| **Security** | HTTPS, service accounts, scan images, sign images, immutable tags |

### Choosing the Right Registry:

```
Are you using a cloud provider?
├── AWS    → ECR
├── Azure  → ACR
├── GCP    → Artifact Registry
│
Do you need enterprise features (scanning, RBAC)?
├── Yes, self-hosted → Harbor
├── Yes, managed     → Cloud provider registry
│
Are you open source / small team?
├── Public images → Docker Hub or GHCR
├── Private + GitHub → GHCR
├── Private + GitLab → GitLab Registry
│
Want full control?
└── Self-hosted Docker Registry
```
