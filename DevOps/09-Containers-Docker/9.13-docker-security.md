## 1. Why Docker Security Matters

Let's start with the big picture: **containers are NOT virtual machines**. They share the host kernel. That means if a container gets compromised and has too many privileges, the attacker could potentially escape the container and take over the entire host machine.

Here's a scary but real scenario:

```
What happens when security is ignored:

1. Developer pulls random image from Docker Hub
2. Image has malware baked into it
3. Container runs as root (default!)
4. Container has access to host filesystem via bind mount
5. Attacker escapes container → owns the host
6. 💀 Game over
```

### Real-World Incidents

| Incident | What Happened |
|----------|---------------|
| **Crypto-mining images on Docker Hub** | Malicious images mined cryptocurrency on unsuspecting users' machines |
| **Tesla Kubernetes hack (2018)** | Unsecured Docker/K8s dashboard → attackers ran crypto miners on Tesla's cloud |
| **Codecov supply chain attack** | Compromised Docker image in CI pipeline leaked secrets |

> **Bottom line:** Docker security is not optional. It's something you MUST think about from day one.

## 3. Running Containers as Non-Root Users

### The Problem

By default, when you run a Docker container, the process inside runs as **root** (UID 0). This is dangerous because:

- If the attacker escapes the container, they're root on the host
- Root inside the container can modify system files
- Root can install packages, change configurations, etc.

```bash
# Prove it — run a container and check who you are
docker run --rm alpine whoami
# Output: root

docker run --rm alpine id
# Output: uid=0(root) gid=0(root) groups=0(root)
```

See? Root by default. Not great.

### The Fix — Create a Non-Root User in Your Dockerfile

```dockerfile
# ❌ BAD — Running as root (default)
FROM node:20
WORKDIR /app
COPY . .
RUN npm install
CMD ["node", "server.js"]

# ✅ GOOD — Running as a non-root user
FROM node:20
WORKDIR /app
COPY . .
RUN npm install

# Create a non-root user
RUN groupadd -r appgroup && useradd -r -g appgroup -d /app -s /sbin/nologin appuser

# Change ownership of app files to the new user
RUN chown -R appuser:appgroup /app

# Switch to the non-root user
USER appuser

CMD ["node", "server.js"]
```

**What each line does:**

| Line | Explanation |
|------|-------------|
| `groupadd -r appgroup` | Creates a system group called `appgroup` (the `-r` flag means "system group") |
| `useradd -r -g appgroup -d /app -s /sbin/nologin appuser` | Creates a system user `appuser`, assigns it to `appgroup`, sets home dir to `/app`, and prevents shell login |
| `chown -R appuser:appgroup /app` | Makes the `appuser` the owner of all files in `/app` (so the app can read/write its own files) |
| `USER appuser` | All commands after this line (including `CMD`) will run as `appuser` instead of root |

### Running as Non-Root at Runtime (Without Modifying Dockerfile)

Sometimes you can't modify the Dockerfile (e.g., you're using a third-party image). You can still force a non-root user:

```bash
# Run as a specific user ID
docker run --rm --user 1000:1000 nginx id
# Output: uid=1000 gid=1000 groups=1000

# Run as "nobody" user
docker run --rm --user nobody nginx whoami
# Output: nobody
```

### Verifying Your Container Doesn't Run as Root

```bash
# Check running containers' user
docker inspect --format='{{.Config.User}}' <container_id>

# Or from inside the container
docker exec <container_id> whoami
```

> **Rule of Thumb:** If your app doesn't need root, don't give it root. Period.

## 4. Rootless Docker — Running the Docker Daemon Without Root

### What's Different from "Non-Root User in Container"?

There are **two different things** people confuse:

```
1. Non-root USER inside container (Section 3 above)
   → The PROCESS in the container runs as non-root
   → But the Docker DAEMON still runs as root on the host

2. Rootless Docker (This section)
   → The Docker DAEMON ITSELF runs as a non-root user
   → Much stronger security — even the daemon can't mess with the host
```

### Why Rootless Docker?

The Docker daemon normally runs as root. That means:
- Anyone who can talk to the Docker daemon effectively has root on the host
- Mounting `/var/run/docker.sock` gives root-level access
- A bug in the daemon = root-level compromise

**Rootless Docker** runs the entire Docker engine under a regular user, using **user namespaces** to map UIDs.

### Setting Up Rootless Docker

```bash
# Step 1: Install prerequisites
sudo apt-get install -y uidmap dbus-user-session

# Step 2: Install rootless Docker
# (You must run this as a non-root user, NOT with sudo)
dockerd-rootless-setuptool.sh install

# Step 3: Set environment variables (add to ~/.bashrc)
export PATH=/home/$USER/bin:$PATH
export DOCKER_HOST=unix:///run/user/$(id -u)/docker.sock

# Step 4: Verify
docker info | grep -i "root"
# Should show: rootless: true
```

### How Rootless Docker Works Under the Hood

```
Normal Docker:                          Rootless Docker:
──────────────                          ────────────────
                                        
Host (root)                             Host (non-root user "alice")
  │                                       │
  ├── dockerd (PID 1, root)              ├── dockerd (PID 1234, alice)
  │     │                                │     │
  │     ├── containerd (root)            │     ├── containerd (alice)
  │     │     │                          │     │     │
  │     │     ├── Container A (root)     │     │     ├── Container A
  │     │     └── Container B (root)     │     │     │   (mapped UID)
  │     │                                │     │     └── Container B
  │     └── docker.sock                  │     │         (mapped UID)
  │         (/var/run/docker.sock)       │     └── docker.sock
  │         (root-only access)           │         (/run/user/1000/docker.sock)
  │                                       │         (alice-only access)
```

### Limitations of Rootless Docker

| Feature | Works in Rootless? | Notes |
|---------|-------------------|-------|
| Building images | ✅ Yes | Works normally |
| Running containers | ✅ Yes | Works normally |
| Port mapping (>1024) | ✅ Yes | No issues |
| Port mapping (<1024) | ⚠️ Limited | Needs `setcap` or `slirp4netns` |
| Overlay2 storage | ✅ Yes | On supported kernels |
| `--privileged` | ❌ No | Defeats the purpose |
| `--net=host` | ⚠️ Limited | Limited functionality |
| Docker Compose | ✅ Yes | Works normally |
| AppArmor / SELinux | ⚠️ Limited | Some restrictions |

## 5. Docker Content Trust (DCT) — Image Signing & Verification

### The Problem

When you `docker pull nginx`, how do you know that image is really from the official Nginx team? What if someone performed a man-in-the-middle attack and swapped the image?

### The Solution — Docker Content Trust

DCT uses **digital signatures** to verify that images haven't been tampered with.

```bash
# Enable Docker Content Trust
export DOCKER_CONTENT_TRUST=1

# Now when you pull, Docker REQUIRES a valid signature
docker pull nginx:latest
# ✅ If signed: pulls successfully
# ❌ If unsigned: REFUSES to pull

# When you push, Docker will sign the image automatically
docker push myrepo/myapp:v1.0
# Docker will prompt you to create signing keys the first time
```

### How It Works

```
Image Publisher:                    Image Consumer:
────────────────                    ────────────────

1. Build image                      1. docker pull myimage:v1
2. docker push myimage:v1           2. Docker contacts Notary server
3. Docker signs with private key    3. Downloads signature
4. Signature → Notary server        4. Verifies signature with public key
5. Image → Docker Registry          5. If valid → pulls image
                                    6. If invalid → REFUSES to pull
```

### Managing DCT Keys

```bash
# List your signing keys
docker trust key list

# Generate a new key pair
docker trust key generate my-signer

# Add a signer to a repository
docker trust signer add --key my-signer.pub my-signer myrepo/myapp

# Inspect trust data for an image
docker trust inspect --pretty myrepo/myapp

# Revoke trust (if a key is compromised)
docker trust revoke myrepo/myapp:v1.0
```

## 6. Image Scanning — Finding Vulnerabilities Before They Hit Production

### What is Image Scanning?

Image scanning analyzes every layer of your Docker image to find:
- Known vulnerabilities (CVEs) in OS packages
- Known vulnerabilities in application dependencies
- Misconfigurations
- Hardcoded secrets (API keys, passwords)
- Compliance violations

### When to Scan?

```
Developer's Machine          CI/CD Pipeline          Registry          Runtime
      │                           │                     │                │
      ▼                           ▼                     ▼                ▼
   Build Image ──────────► Push to Pipeline ────► Push to Registry ──► Deploy
      │                           │                     │                │
   ┌──▼──┐                  ┌────▼────┐           ┌───▼───┐       ┌───▼───┐
   │SCAN │                  │  SCAN   │           │ SCAN  │       │ SCAN  │
   │here │                  │  here   │           │ here  │       │ here  │
   └─────┘                  └─────────┘           └───────┘       └───────┘
   
   (Shift Left)            (Automated Gate)      (Admission)     (Continuous)
```

**Best practice:** Scan at EVERY stage. The earlier you catch it, the cheaper it is to fix.

### Tool 1: Docker Scout (Built-in)

Docker Scout is Docker's native scanning tool (successor to `docker scan`).

```bash
# Scan an image with Docker Scout
docker scout cves nginx:latest

# Get a quick overview
docker scout quickview nginx:latest

# Compare two images
docker scout compare nginx:1.24 nginx:1.25

# Get recommendations
docker scout recommendations nginx:latest
```

**Example output:**


### Tool 2: Trivy (Very Popular, Open Source)

Trivy by Aqua Security is one of the most popular scanners. It's fast, comprehensive, and easy to use.

```bash
# Install Trivy
# On Ubuntu/Debian
sudo apt-get install -y trivy

# On Mac
brew install trivy

# Scan an image
trivy image nginx:latest

# Scan with severity filter (only show HIGH and CRITICAL)
trivy image --severity HIGH,CRITICAL nginx:latest

# Scan and fail if critical vulnerabilities found (great for CI/CD)
trivy image --exit-code 1 --severity CRITICAL myapp:v1.0

# Scan a local Dockerfile
trivy config Dockerfile

# Scan a filesystem
trivy fs /path/to/your/project

# Output as JSON (for automated processing)
trivy image --format json --output results.json nginx:latest

# Scan and ignore unfixed vulnerabilities
trivy image --ignore-unfixed nginx:latest
```

### Integrating Scanning into CI/CD (GitHub Actions Example)

```yaml
# .github/workflows/docker-security.yml
name: Docker Security Scan

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  scan:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Build Docker image
        run: docker build -t myapp:${{ github.sha }} .

      - name: Run Trivy vulnerability scanner
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: 'myapp:${{ github.sha }}'
          format: 'table'
          exit-code: '1'                    # Fail the pipeline if vulns found
          severity: 'CRITICAL,HIGH'         # Only fail on high/critical
          ignore-unfixed: true              # Ignore vulns with no fix available
```

## 7. Distroless Images — The Ultimate Minimal Container

### What are Distroless Images?

Distroless images are Docker images that contain **only your application and its runtime dependencies**. They do NOT contain:
- Package managers (no `apt`, `yum`, `apk`)
- Shells (no `bash`, `sh`)
- Common utilities (no `ls`, `cat`, `curl`, `wget`)
- Anything else that isn't strictly needed to run your app

They're made by Google and available at `gcr.io/distroless/`.

### Why Distroless?

```
Traditional Image (e.g., ubuntu:22.04):
┌────────────────────────────────────────────┐
│  Full OS: bash, apt, curl, wget, vim,      │  ← 77 MB
│  openssl, coreutils, grep, sed, awk...     │
│  ┌──────────────────────────────────────┐  │
│  │  Runtime (Node.js, Python, Java...)  │  │  ← 100+ MB
│  │  ┌────────────────────────────────┐  │  │
│  │  │  Your Application              │  │  │  ← 10 MB
│  │  └────────────────────────────────┘  │  │
│  └──────────────────────────────────────┘  │
│  Total: ~200+ MB                           │
│  Attack surface: HUGE (hundreds of pkgs)   │
└────────────────────────────────────────────┘

Distroless Image:
┌────────────────────────────────────────────┐
│  Minimal runtime libraries (glibc, etc.)   │  ← 2 MB
│  ┌──────────────────────────────────────┐  │
│  │  Runtime (Node.js, Python, Java...)  │  │  ← 50+ MB
│  │  ┌────────────────────────────────┐  │  │
│  │  │  Your Application              │  │  │  ← 10 MB
│  │  └────────────────────────────────┘  │  │
│  └──────────────────────────────────────┘  │
│  Total: ~65 MB                             │
│  Attack surface: TINY (almost nothing)     │
└────────────────────────────────────────────┘
```

### Security Benefit

If an attacker somehow gets into your container, they can't do much because:
- No shell → can't run commands
- No package manager → can't install tools
- No curl/wget → can't download exploits
- No ls/cat → can't explore the filesystem

It's like breaking into a house that has nothing in it. You got in, but there's nothing to steal or use.

### Available Distroless Images

| Image | Use Case | Base Size |
|-------|----------|-----------|
| `gcr.io/distroless/static-debian12` | Statically compiled binaries (Go, Rust) | ~2 MB |
| `gcr.io/distroless/base-debian12` | Dynamically linked binaries | ~20 MB |
| `gcr.io/distroless/cc-debian12` | C/C++ applications | ~22 MB |
| `gcr.io/distroless/python3-debian12` | Python applications | ~52 MB |
| `gcr.io/distroless/java21-debian12` | Java applications | ~90 MB |
| `gcr.io/distroless/nodejs20-debian12` | Node.js applications | ~60 MB |

### Example 1: Go Application with Distroless

```dockerfile
# ==========================================
# Stage 1: Build the Go binary
# ==========================================
FROM golang:1.22-alpine AS builder

WORKDIR /app

# Copy dependency files first (for layer caching)
COPY go.mod go.sum ./
RUN go mod download

# Copy source code
COPY . .

# Build a statically linked binary
# CGO_ENABLED=0 → no C library dependencies
# -ldflags="-s -w" → strip debug info (smaller binary)
RUN CGO_ENABLED=0 GOOS=linux go build -ldflags="-s -w" -o /app/server .

# ==========================================
# Stage 2: Run with distroless
# ==========================================
FROM gcr.io/distroless/static-debian12

# Copy ONLY the binary from the builder stage
COPY --from=builder /app/server /server

# Run as non-root (distroless supports this!)
USER nonroot:nonroot

# Run the binary
ENTRYPOINT ["/server"]
```

**What's happening here:**
1. **Stage 1 (builder):** We use a full Go image to compile our code
2. `CGO_ENABLED=0` makes the binary fully self-contained (no dependency on libc)
3. `-ldflags="-s -w"` strips debug symbols to make the binary smaller
4. **Stage 2:** We use the tiniest distroless image (`static`) since our Go binary has zero external dependencies
5. We copy only the compiled binary — nothing else
6. `USER nonroot:nonroot` runs as non-root (distroless images include a `nonroot` user)

**Result:** Final image is roughly **7-8 MB** compared to 300+ MB with a full Go image.

### Example 2: Node.js Application with Distroless

```dockerfile
# Stage 1: Build
FROM node:20-alpine AS builder

WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY . .

# Stage 2: Run with distroless
FROM gcr.io/distroless/nodejs20-debian12

WORKDIR /app
COPY --from=builder /app /app

USER nonroot:nonroot

CMD ["server.js"]
```

> **Note:** With distroless Node.js, you can't use `npm start` because there's no shell. You must directly specify the JS file to run.

### Example 3: Java Application with Distroless

```dockerfile
# Stage 1: Build with Maven
FROM maven:3.9-eclipse-temurin-21 AS builder

WORKDIR /app
COPY pom.xml .
RUN mvn dependency:resolve
COPY src ./src
RUN mvn package -DskipTests

# Stage 2: Run with distroless
FROM gcr.io/distroless/java21-debian12

COPY --from=builder /app/target/myapp.jar /app/myapp.jar

USER nonroot:nonroot

ENTRYPOINT ["java", "-jar", "/app/myapp.jar"]
```

### Debugging Distroless Containers

Since distroless has no shell, you can't `docker exec -it container sh`. But you have options:

```bash
# Option 1: Use the :debug tag (includes a busybox shell)
# ONLY use this in development, never in production!
FROM gcr.io/distroless/static-debian12:debug
# Now you CAN shell in:
docker exec -it <container> /busybox/sh

# Option 2: Use ephemeral debug containers (Kubernetes)
kubectl debug -it <pod-name> --image=busybox --target=<container-name>

# Option 3: Use docker cp to copy files out
docker cp <container>:/app/logs/error.log ./error.log

# Option 4: Check logs
docker logs <container>
```

```
│ distroless/static   │ ~2 MB   │ ❌ none   │ ❌ none      │
│ scratch             │ 0 MB    │ ❌ none   │ ❌ none      │
└─────────────────────┴─────────┴───────────┴──────────────┘
```

### Alpine Linux

Alpine is a popular choice because it's tiny (~7 MB) but still has a shell and package manager.

```dockerfile
FROM alpine:3.19

# Alpine uses apk instead of apt
RUN apk add --no-cache python3 py3-pip

COPY . /app
WORKDIR /app

RUN pip install --no-cache-dir -r requirements.txt

USER nobody

CMD ["python3", "app.py"]
```

**Gotcha with Alpine:** Alpine uses **musl libc** instead of **glibc**. Some applications (especially those with compiled C extensions) might have compatibility issues. If you run into weird errors, try `-slim` variants instead.

### Scratch — The Empty Image

`scratch` is literally an **empty image**. No OS, no files, nothing. It's perfect for statically compiled binaries.

```dockerfile
# Build
FROM golang:1.22 AS builder
WORKDIR /app
COPY . .
RUN CGO_ENABLED=0 go build -o /app/myapp .

# Run from scratch (literally nothing)
FROM scratch
COPY --from=builder /app/myapp /myapp
ENTRYPOINT ["/myapp"]
```

### When to Use What?

| Scenario | Recommended Base |
|----------|-----------------|
| Go / Rust (static binary) | `scratch` or `distroless/static` |
| Java application | `distroless/java` or `eclipse-temurin:xx-jre-alpine` |
| Node.js application | `distroless/nodejs` or `node:xx-alpine` |
| Python application | `distroless/python3` or `python:xx-slim` |
| Need to debug in container | `alpine` or `*-slim` variants |
| Maximum security | `distroless` or `scratch` |

## 9. Docker Security Best Practices for Dockerfiles

Here's a collection of practical tips for writing secure Dockerfiles:

### 1. Pin Image Versions — NEVER Use `latest`

```dockerfile
# ❌ BAD — "latest" can change anytime. You don't know what you're getting.
FROM node:latest

# ⚠️ BETTER — pinned to major.minor
FROM node:20-alpine

# ✅ BEST — pinned to exact digest (immutable)
FROM node:20-alpine@sha256:abc123def456...
```

### 2. Never Hardcode Secrets

```dockerfile
# ❌ TERRIBLE — secrets visible in image layers!
ENV API_KEY=sk-12345678
ENV DB_PASSWORD=supersecret

# ✅ GOOD — pass at runtime
# docker run -e API_KEY=$API_KEY myapp
# Or use Docker secrets (Swarm) / Kubernetes secrets
```

### 3. Use Multi-Stage Builds to Remove Build Tools

```dockerfile
# ❌ BAD — build tools, source code, and caches are in the final image
FROM node:20
WORKDIR /app
COPY . .
RUN npm install
RUN npm run build
CMD ["node", "dist/server.js"]
# Image contains: npm, node_modules (dev deps), source code, build cache

# ✅ GOOD — multi-stage: build tools stay in the builder stage
FROM node:20 AS builder
WORKDIR /app
COPY . .
RUN npm ci
RUN npm run build

FROM node:20-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/package.json ./
USER node
CMD ["node", "dist/server.js"]
# Image contains: ONLY what's needed to run
```

### 4. Use COPY Instead of ADD

```dockerfile
# ❌ ADD can auto-extract archives and fetch URLs — unexpected behavior
ADD https://example.com/app.tar.gz /app/

# ✅ COPY is simple and predictable
COPY app.tar.gz /app/
RUN tar -xzf /app/app.tar.gz -C /app/ && rm /app/app.tar.gz
```

### 5. Don't Install Unnecessary Packages

```dockerfile
# ❌ BAD — why is vim, curl, and wget in a production container?
RUN apt-get update && apt-get install -y \
    vim \
    curl \
    wget \
    net-tools \
    htop

# ✅ GOOD — install ONLY what's needed, clean up cache
RUN apt-get update && apt-get install -y --no-install-recommends \
    ca-certificates \
    && rm -rf /var/lib/apt/lists/*
```

### 6. Use .dockerignore

```dockerignore
# .dockerignore — prevent these from being copied into the image

.git
.gitignore
.env
.env.*
node_modules
npm-debug.log
Dockerfile
docker-compose.yml
README.md
.vscode
.idea
*.md
tests/
coverage/
```

### 7. Set HEALTHCHECK

```dockerfile
HEALTHCHECK --interval=30s --timeout=3s --start-period=10s --retries=3 \
  CMD curl -f http://localhost:8080/health || exit 1
```

## 10. Read-Only File Systems

One powerful security measure is making the container's filesystem read-only. This prevents attackers from writing malicious scripts or modifying binaries.

```bash
# Run container with read-only filesystem
docker run --read-only nginx

# But what if the app needs to write temp files?
# Use tmpfs mounts for writable directories
docker run --read-only \
  --tmpfs /tmp \
  --tmpfs /var/cache/nginx \
  --tmpfs /var/run \
  nginx
```

In Docker Compose:

```yaml
services:
  web:
    image: nginx
    read_only: true
    tmpfs:
      - /tmp
      - /var/cache/nginx
      - /var/run
```

**What `--tmpfs` does:** Creates an in-memory filesystem (stored in RAM, not on disk). When the container stops, everything in tmpfs is gone. This is perfect for temp files, caches, and PID files.

## 11. Dropping Linux Capabilities

### What Are Linux Capabilities?

Linux capabilities break down the big "root" privilege into smaller, individual permissions. Instead of giving a process ALL root powers, you can give it only specific ones.

```
Full root = ALL capabilities (dangerous)

Capabilities let you say:
  ✅ "You CAN bind to port 80"    (CAP_NET_BIND_SERVICE)
  ❌ "You CANNOT load kernel modules"  (CAP_SYS_MODULE)
  ❌ "You CANNOT change file ownership"  (CAP_CHOWN)
  ❌ "You CANNOT reboot the system"  (CAP_SYS_BOOT)
```

### Docker's Default Capabilities

Docker already drops many capabilities by default, but it still keeps some that most containers don't need:

```
Capabilities Docker KEEPS by default:
────────────────────────────────────
CAP_CHOWN           — Change file ownership
CAP_DAC_OVERRIDE    — Bypass file read/write permissions
CAP_FSETID          — Set file set-group-ID
CAP_FOWNER          — Bypass file ownership checks
CAP_MKNOD           — Create special files
CAP_NET_RAW         — Use RAW/PACKET sockets (ping, etc.)
CAP_SETGID          — Change group ID
CAP_SETUID          — Change user ID
CAP_SETFCAP         — Set file capabilities
CAP_SETPCAP         — Modify process capabilities
CAP_NET_BIND_SERVICE — Bind to ports below 1024
CAP_SYS_CHROOT      — Use chroot
CAP_KILL            — Send signals to processes
CAP_AUDIT_WRITE     — Write to kernel audit log
```

### Dropping All Capabilities and Adding Only What's Needed

```bash
# Drop ALL capabilities, then add only what's needed
docker run --cap-drop=ALL --cap-add=NET_BIND_SERVICE nginx

# Drop specific capabilities
docker run --cap-drop=NET_RAW --cap-drop=SYS_CHROOT myapp

# Check capabilities of a running container
docker inspect --format='{{.HostConfig.CapAdd}}' <container>
docker inspect --format='{{.HostConfig.CapDrop}}' <container>
```

In Docker Compose:

```yaml
services:
  web:
    image: nginx
    cap_drop:
      - ALL
    cap_add:
      - NET_BIND_SERVICE
      - CHOWN
      - SETGID
      - SETUID
```

### The `--privileged` Flag — NEVER Use It in Production

```bash
# ❌ NEVER DO THIS IN PRODUCTION
docker run --privileged myapp

# --privileged gives the container ALL capabilities + access to ALL devices
# It's essentially the same as running on the host with root
# Only use for very specific development/debugging scenarios
```

## 12. Seccomp Profiles — Restricting System Calls

### What is Seccomp?

**Seccomp** (Secure Computing Mode) limits which **system calls** a container can make to the kernel. Since containers share the host kernel, limiting syscalls reduces the attack surface.

Think of it as a whitelist/blocklist for what the container is allowed to ask the kernel to do.

```
Without seccomp:
Container → can call any of ~400+ syscalls → Kernel

With seccomp:
Container → can only call ~50 allowed syscalls → Kernel
           → blocked syscalls return an error
```

### Docker's Default Seccomp Profile

Docker ships with a default seccomp profile that blocks about 44 dangerous syscalls. This is enabled automatically:

```bash
# Check if seccomp is active
docker info | grep -i seccomp

# Run with default profile (this is automatic)
docker run --security-opt seccomp=default nginx

# Disable seccomp (DON'T do this in production)
docker run --security-opt seccomp=unconfined nginx
```

### Syscalls Blocked by Default

Some of the dangerous syscalls Docker blocks:

| Syscall | What It Does | Why It's Blocked |
|---------|-------------|-----------------|
| `reboot` | Reboots the host | Container shouldn't reboot the host |
| `mount` | Mounts filesystems | Could mount host filesystems |
| `kexec_load` | Loads a new kernel | Could replace the kernel |
| `open_by_handle_at` | Opens files by handle | Could escape container |
| `init_module` | Loads kernel modules | Could modify the kernel |
| `delete_module` | Removes kernel modules | Could remove security modules |
| `clock_settime` | Changes system clock | Could mess up host time |

### Creating a Custom Seccomp Profile

```json
{
  "defaultAction": "SCMP_ACT_ERRNO",
  "architectures": [
    "SCMP_ARCH_X86_64",
    "SCMP_ARCH_X86",
    "SCMP_ARCH_AARCH64"
  ],
  "syscalls": [
    {
      "names": [
        "accept", "accept4", "access", "arch_prctl",
        "bind", "brk", "close", "connect",
        "dup", "dup2", "dup3", "epoll_create",
        "epoll_create1", "epoll_ctl", "epoll_wait",
        "execve", "exit", "exit_group",
        "fcntl", "fstat", "futex",
        "getdents64", "getpid", "getuid",
        "listen", "lseek", "mmap", "mprotect",
        "munmap", "open", "openat",
        "read", "recvfrom", "recvmsg",
        "rt_sigaction", "rt_sigprocmask",
        "sendmsg", "sendto", "setsockopt",
        "socket", "stat", "write", "writev"
      ],
      "action": "SCMP_ACT_ALLOW"
    }
  ]
}
```

```bash
# Run with custom seccomp profile
docker run --security-opt seccomp=custom-profile.json myapp
```

**Explanation of the profile:**
- `"defaultAction": "SCMP_ACT_ERRNO"` — Block everything by default (return an error)
- `"syscalls"` — List of allowed syscalls
- `"action": "SCMP_ACT_ALLOW"` — Allow these specific syscalls
- Only the syscalls in the list will work; everything else is denied

### Generating a Seccomp Profile Using OCI Seccomp BPF Hook

You can auto-generate a profile by recording what syscalls your app actually uses:

```bash
# Install the tool
sudo apt-get install -y oci-seccomp-bpf-hook

# Run your container and record syscalls
docker run --annotation io.containers.trace-syscall="of:/tmp/myapp-seccomp.json" myapp

# Now use the generated profile
docker run --security-opt seccomp=/tmp/myapp-seccomp.json myapp
```

## 14. Docker Secrets Management

### The Problem — Secrets in Environment Variables

```bash
# ❌ BAD — secrets visible in process listing and docker inspect
docker run -e DB_PASSWORD=supersecret myapp

# Anyone can see it:
docker inspect <container> | grep DB_PASSWORD
# Output: "DB_PASSWORD=supersecret"   ← exposed!
```

### Docker Secrets (Swarm Mode)

Docker secrets are encrypted at rest and only exposed to containers that need them.

```bash
# Initialize Swarm (secrets require Swarm mode)
docker swarm init

# Create a secret from a string
echo "supersecretpassword" | docker secret create db_password -

# Create a secret from a file
docker secret create tls_cert ./server.crt

# List secrets
docker secret ls

# Use secret in a service
docker service create \
  --name myapp \
  --secret db_password \
  myapp:latest

# Inside the container, the secret is available as a FILE at:
# /run/secrets/db_password
```

### Using Secrets in Docker Compose

```yaml
# docker-compose.yml
version: "3.9"

services:
  db:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD_FILE: /run/secrets/db_password
    secrets:
      - db_password

  app:
    image: myapp:latest
    secrets:
      - db_password
      - api_key

secrets:
  db_password:
    file: ./secrets/db_password.txt
  api_key:
    external: true     # Must be created beforehand via docker secret create
```

### Reading Secrets in Your Application

```python
# Python example — reading Docker secrets
import os

def get_secret(secret_name):
    """Read a Docker secret from /run/secrets/"""
    secret_path = f"/run/secrets/{secret_name}"
    try:
        with open(secret_path, 'r') as f:
            return f.read().strip()
    except FileNotFoundError:
        # Fallback to environment variable (for local development)
        return os.environ.get(secret_name.upper())

# Usage
db_password = get_secret("db_password")
api_key = get_secret("api_key")
```

### BuildKit Secrets (For Build-Time Secrets)

Sometimes you need secrets during the build (e.g., private npm registry, private Git repos):

```dockerfile
# syntax=docker/dockerfile:1

FROM node:20-alpine

WORKDIR /app
COPY package*.json ./

# Mount the secret during build — it's NEVER stored in any image layer
RUN --mount=type=secret,id=npm_token \
    NPM_TOKEN=$(cat /run/secrets/npm_token) \
    npm ci --registry=https://registry.npmjs.org/

COPY . .

USER node
CMD ["node", "server.js"]
```

```bash
# Build with the secret
DOCKER_BUILDKIT=1 docker build \
  --secret id=npm_token,src=./.npm_token \
  -t myapp:latest .
```

**Why this is safe:** The `--mount=type=secret` makes the secret available ONLY during that specific `RUN` instruction. It never becomes part of any image layer, so even if someone pulls your image, they can't find the secret.

## 15. Network Security — Isolating Containers

### Default Bridge Network — The Problem

By default, all containers on the same bridge network can talk to each other. This is bad for security.

```bash
# Prove it — start two containers
docker run -d --name app1 nginx
docker run -d --name app2 alpine sleep 3600

# app2 can reach app1
docker exec app2 wget -qO- http://app1
# ← This works! No restrictions!
```

### Custom Networks — Better Isolation

```bash
# Create isolated networks
docker network create frontend-net
docker network create backend-net

# Put containers in specific networks
docker run -d --name web --network frontend-net nginx
docker run -d --name api --network frontend-net --network backend-net myapi
docker run -d --name db --network backend-net postgres

# Result:
# web → can talk to api (both on frontend-net)
# api → can talk to web AND db (on both networks)
# web → CANNOT talk to db (different networks, no overlap)
# db → CANNOT talk to web (different networks, no overlap)
```

```
┌──────────────────┐   ┌──────────────────┐
│  frontend-net    │   │   backend-net    │
│                  │   │                  │
│  ┌─────┐         │   │         ┌─────┐ │
│  │ web │ ◄──────►│───│◄──────► │ db  │ │
│  └─────┘         │   │         └─────┘ │
│         ┌─────┐  │   │  ┌─────┐        │
│         │ api │◄─┼───┼─►│ api │        │
│         └─────┘  │   │  └─────┘        │
└──────────────────┘   └──────────────────┘
     web ↔ api              api ↔ db
     web ✗ db               web ✗ db
```

### Disable Inter-Container Communication

```bash
# Create a network where containers can't talk to each other
docker network create --internal isolated-net

# Or disable ICC (inter-container communication)
docker network create -o com.docker.network.bridge.enable_icc=false my-net
```

### Don't Expose Ports Unnecessarily

```bash
# ❌ BAD — exposes port to the ENTIRE world (0.0.0.0)
docker run -p 5432:5432 postgres

# ✅ GOOD — bind to localhost only
docker run -p 127.0.0.1:5432:5432 postgres

# ✅ GOOD — use internal networks, don't expose at all
# Let other containers connect via Docker network, not published ports
```

## 16. Resource Limits — Preventing DoS

Without resource limits, a single container can eat ALL the CPU and memory on your host, causing a denial-of-service for other containers and the host itself.

```bash
# Limit memory to 512MB
docker run --memory=512m nginx

# Limit memory AND swap
docker run --memory=512m --memory-swap=1g nginx

# Limit CPU to 1.5 cores
docker run --cpus=1.5 nginx

# Limit CPU shares (relative weight — default is 1024)
docker run --cpu-shares=512 nginx

# Limit the number of processes (prevent fork bombs)
docker run --pids-limit=100 nginx

# Combine all limits
docker run \
  --memory=512m \
  --memory-swap=512m \
  --cpus=1.0 \
  --pids-limit=50 \
  --read-only \
  nginx
```

In Docker Compose:

```yaml
services:
  web:
    image: nginx
    deploy:
      resources:
        limits:
          cpus: '1.0'
          memory: 512M
        reservations:
          cpus: '0.5'
          memory: 256M
    pids_limit: 100
```

## 17. Docker Bench Security — Automated Auditing

Docker Bench for Security is an official script that checks your Docker installation against the **CIS Docker Benchmark** (industry-standard security guidelines).

```bash
# Run Docker Bench for Security
docker run --rm --net host --pid host --userns host --cap-add audit_control \
  -e DOCKER_CONTENT_TRUST=$DOCKER_CONTENT_TRUST \
  -v /etc:/etc:ro \
  -v /usr/bin/containerd:/usr/bin/containerd:ro \
  -v /usr/bin/runc:/usr/bin/runc:ro \
  -v /usr/lib/systemd:/usr/lib/systemd:ro \
  -v /var/lib:/var/lib:ro \
  -v /var/run/docker.sock:/var/run/docker.sock:ro \
  docker/docker-bench-security
```

**Example output:**

```
[INFO] 1 - Host Configuration
[PASS] 1.1  - Ensure a separate partition for containers has been created
[WARN] 1.2  - Ensure only trusted users are allowed to control Docker daemon
[PASS] 1.3  - Ensure auditing is configured for Docker daemon

[INFO] 4 - Container Images and Build File
[WARN] 4.1  - Ensure that a user for the container has been created
[PASS] 4.2  - Ensure that containers use only trusted base images
[WARN] 4.5  - Ensure Content trust for Docker is Enabled

[INFO] 5 - Container Runtime
[WARN] 5.1  - Ensure that AppArmor profile is set appropriately
[WARN] 5.2  - Ensure that SELinux security options are set
[PASS] 5.3  - Ensure that Linux Kernel Capabilities are restricted
[WARN] 5.4  - Ensure that privileged containers are not used
```

Each item is either `PASS`, `WARN`, `NOTE`, or `INFO`. Fix the `WARN` items to improve your security posture.

## 18. Supply Chain Security — SBOMs and Provenance

### What is an SBOM?

An **SBOM** (Software Bill of Materials) is a complete list of all components inside your container image — every package, library, and dependency with exact versions.

Think of it like an ingredients list on food packaging. You want to know exactly what's in your container.

### Generating SBOMs

```bash
# Using Docker Scout
docker scout sbom nginx:latest

# Using Syft (by Anchore — popular open-source tool)
# Install Syft
curl -sSfL https://raw.githubusercontent.com/anchore/syft/main/install.sh | sh -s

# Generate SBOM in SPDX format
syft nginx:latest -o spdx-json > sbom.json

# Generate SBOM in CycloneDX format
syft nginx:latest -o cyclonedx-json > sbom.json

# Using Trivy
trivy image --format spdx-json nginx:latest > sbom.json
```

### Image Provenance with Cosign

**Cosign** (by Sigstore) lets you sign container images with keyless signing using OIDC identity.

```bash
# Install Cosign
go install github.com/sigstore/cosign/v2/cmd/cosign@latest

# Sign an image (keyless — uses your GitHub/Google/Microsoft identity)
cosign sign myregistry/myapp:v1.0

# Verify a signed image
cosign verify myregistry/myapp:v1.0

# Sign with a key
cosign generate-key-pair
cosign sign --key cosign.key myregistry/myapp:v1.0
cosign verify --key cosign.pub myregistry/myapp:v1.0
```

## 19. Real-World Security Checklist

Here's a comprehensive checklist you can use for every Docker project:

### Dockerfile Security

- [ ] Use specific image tags (never `latest`)
- [ ] Use multi-stage builds
- [ ] Run as non-root user (`USER` instruction)
- [ ] Don't store secrets in images (no `ENV` with secrets)
- [ ] Use `COPY` instead of `ADD`
- [ ] Use `.dockerignore`
- [ ] Remove unnecessary packages
- [ ] Set `HEALTHCHECK`

### Runtime Security

- [ ] Run with `--read-only` filesystem
- [ ] Drop all capabilities, add only needed ones (`--cap-drop=ALL`)
- [ ] Set resource limits (`--memory`, `--cpus`, `--pids-limit`)
- [ ] Don't use `--privileged`
- [ ] Don't mount Docker socket into containers
- [ ] Bind ports to localhost when possible (`127.0.0.1:`)
- [ ] Use custom networks for isolation

### Image Security

- [ ] Scan images regularly (Trivy, Scout, Grype)
- [ ] Use distroless or minimal base images
- [ ] Enable Docker Content Trust
- [ ] Generate and store SBOMs
- [ ] Sign images with Cosign
- [ ] Pin base images by digest

### Host Security

- [ ] Keep Docker updated
- [ ] Consider rootless Docker
- [ ] Run Docker Bench for Security
- [ ] Enable seccomp profiles
- [ ] Use AppArmor/SELinux
- [ ] Restrict Docker daemon access

## 20. Summary

| Topic | Key Takeaway |
|-------|-------------|
| **Non-root user** | Always use `USER` in Dockerfile; never run as root |
| **Rootless Docker** | Run the daemon itself as non-root for maximum security |
| **Content Trust** | Sign and verify images to prevent tampering |
| **Image Scanning** | Scan at every stage: build, CI/CD, registry, runtime |
| **Distroless** | No shell, no package manager = tiny attack surface |
| **Minimal Images** | Use Alpine, slim, scratch, or distroless |
| **Capabilities** | Drop ALL, add only what's needed |
| **Seccomp** | Restrict syscalls to only what the app uses |
| **Secrets** | Use Docker secrets or BuildKit mounts, never env vars for sensitive data |
| **Networking** | Use custom networks; isolate frontend from backend |
| **Resource Limits** | Always set memory, CPU, and PID limits |
| **Docker Bench** | Run it regularly to audit your setup |
| **SBOMs** | Know what's inside your images |

> **Remember:** Security is not a one-time thing. It's a continuous process. Scan regularly, update base images, and follow the principle of least privilege — give your containers only what they need, nothing more.
