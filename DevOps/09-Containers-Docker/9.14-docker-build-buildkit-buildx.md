## 1. The Evolution of Docker Builds

Docker's build system has evolved significantly:

```
Timeline of Docker Builds:
──────────────────────────

2013 ── docker build (Legacy Builder)
        │ • Sequential execution
        │ • No parallel stages
        │ • Basic caching
        │ • No secrets support
        │ • Single platform only
        │
2017 ── BuildKit (Experimental)
        │ • Parallel stage execution
        │ • Better caching
        │ • Secret/SSH mounts
        │ • More efficient
        │
2019 ── Buildx (Plugin)
        │ • Multi-platform builds
        │ • Multiple builder instances
        │ • Remote builders
        │ • Bake (build definitions)
        │
2023 ── BuildKit is the DEFAULT
        │ • BuildKit is now used by default
        │ • Legacy builder is deprecated
        │ • Buildx comes bundled with Docker Desktop
        │
Today ── BuildKit + Buildx = Standard
```

> **Bottom line:** If you're still using the legacy builder, you're missing out on speed, security, and multi-platform support. BuildKit + Buildx is the present and future.

## 2. What is BuildKit?

**BuildKit** is Docker's modern build engine. It replaced the old "legacy builder" and brings massive improvements.

Think of it this way:

```
Legacy Builder = Old car with manual transmission, no GPS, basic engine
BuildKit       = Modern car with automatic, GPS, turbo engine, better fuel efficiency
```

### Key Improvements Over Legacy Builder

| Feature | Legacy Builder | BuildKit |
|---------|---------------|----------|
| **Build execution** | Sequential (one step at a time) | Parallel (independent steps run together) |
| **Layer caching** | Basic (invalidates entire chain) | Smart (fine-grained, content-based) |
| **Secrets handling** | ❌ Not supported | ✅ Secret mounts |
| **SSH forwarding** | ❌ Not supported | ✅ SSH mounts |
| **Build output** | Only Docker images | Images, tarballs, local files, OCI |
| **Garbage collection** | Manual | Automatic |
| **Progress output** | Verbose text | Clean, colored, real-time progress |
| **Multi-platform** | ❌ Not supported | ✅ (via Buildx) |
| **Remote cache** | ❌ Not supported | ✅ Registry-based cache |

## 3. Enabling BuildKit

### Check If BuildKit Is Enabled

```bash
# Check your Docker version
docker version
# BuildKit is default in Docker Engine 23.0+ and Docker Desktop 4.x+

# Check if BuildKit is active
docker info | grep -i buildkit
# Output should show: buildkit: true
```

### How to Enable BuildKit

```bash
# Method 1: Environment variable (temporary, for current session)
export DOCKER_BUILDKIT=1
docker build -t myapp .

# Method 2: Inline (per-command)
DOCKER_BUILDKIT=1 docker build -t myapp .

# Method 3: Docker daemon config (permanent)
# Edit /etc/docker/daemon.json
{
  "features": {
    "buildkit": true
  }
}
# Then restart Docker
sudo systemctl restart docker

# Method 4: Use buildx (automatically uses BuildKit)
docker buildx build -t myapp .
```

## 5. Parallel Build Stages

This is one of BuildKit's biggest wins. In a multi-stage build, BuildKit figures out which stages are independent and runs them at the same time.

### Example — Legacy Builder (Sequential)

```dockerfile
# This Dockerfile has 3 independent stages

# Stage 1: Build frontend
FROM node:20 AS frontend
WORKDIR /frontend
COPY frontend/ .
RUN npm install && npm run build

# Stage 2: Build backend
FROM golang:1.22 AS backend
WORKDIR /backend
COPY backend/ .
RUN go build -o server .

# Stage 3: Build docs
FROM python:3.12 AS docs
WORKDIR /docs
COPY docs/ .
RUN pip install mkdocs && mkdocs build

# Final stage: Combine everything
FROM nginx:alpine
COPY --from=frontend /frontend/dist /usr/share/nginx/html
COPY --from=backend /backend/server /usr/local/bin/server
COPY --from=docs /docs/site /usr/share/nginx/html/docs
```

```
Legacy Builder execution:
─────────────────────────
[frontend]  ████████████████ (60s)
                              [backend]  ████████████ (45s)
                                                      [docs]  ████████ (30s)
                                                                       [final]  ██ (5s)
Total: 140 seconds (sequential!)


BuildKit execution:
───────────────────
[frontend]  ████████████████ (60s)
[backend]   ████████████ (45s)          ← runs in PARALLEL!
[docs]      ████████ (30s)              ← runs in PARALLEL!
                              [final]  ██ (5s)
Total: 65 seconds (parallel!)
```

BuildKit automatically detects that `frontend`, `backend`, and `docs` are independent (none depends on another) and runs them simultaneously. **More than 2x speedup** for free — no code changes needed!

## 6. Build Cache — Speed Up Your Builds

### How Layer Caching Works

Every instruction in a Dockerfile creates a layer. If nothing changes, Docker reuses the cached layer.

```dockerfile
FROM node:20-alpine
WORKDIR /app

# These layers are cached if unchanged
COPY package.json package-lock.json ./
RUN npm ci

# This layer invalidates when source code changes
COPY . .
RUN npm run build

CMD ["node", "dist/server.js"]
```

```
First build:
  FROM node:20-alpine           ← Downloaded from registry
  WORKDIR /app                  ← Cached
  COPY package*.json ./         ← New layer created
  RUN npm ci                    ← New layer created (slow, installs deps)
  COPY . .                      ← New layer created
  RUN npm run build             ← New layer created

Second build (only source code changed):
  FROM node:20-alpine           ← CACHED ✅
  WORKDIR /app                  ← CACHED ✅
  COPY package*.json ./         ← CACHED ✅ (package.json didn't change)
  RUN npm ci                    ← CACHED ✅ (skipped! saved 30+ seconds)
  COPY . .                      ← REBUILT ❌ (source code changed)
  RUN npm run build             ← REBUILT ❌ (previous layer changed)
```

### Inline Cache — Store Cache Metadata in the Image

```bash
# Build with inline cache (stores cache info in the image itself)
docker buildx build \
  --cache-to type=inline \
  --push \
  -t myregistry/myapp:latest .

# On another machine, use that cache
docker buildx build \
  --cache-from type=registry,ref=myregistry/myapp:latest \
  -t myregistry/myapp:v2.0 .
```

### Registry Cache — Dedicated Cache Image

```bash
# Push cache to a separate image in the registry
docker buildx build \
  --cache-to type=registry,ref=myregistry/myapp:cache \
  --cache-from type=registry,ref=myregistry/myapp:cache \
  --push \
  -t myregistry/myapp:latest .
```

This is extremely useful in CI/CD where each build runs on a fresh machine with no local cache.

### Local Cache — File-Based Cache

```bash
# Export cache to local directory
docker buildx build \
  --cache-to type=local,dest=/tmp/docker-cache \
  -t myapp:latest .

# Use local cache on next build
docker buildx build \
  --cache-from type=local,src=/tmp/docker-cache \
  -t myapp:latest .
```

### GitHub Actions Cache

```bash
# Use GitHub Actions cache backend
docker buildx build \
  --cache-to type=gha \
  --cache-from type=gha \
  -t myapp:latest .
```

## 7. Cache Mounts — Persistent Caches Across Builds

Cache mounts let you persist package manager caches (npm, pip, apt, go, maven) across builds. This is different from layer caching — it persists the actual download cache.

### Without Cache Mount (Slow)

```dockerfile
# Every build re-downloads ALL npm packages from scratch
RUN npm ci
# Even if only one package changed, everything is re-downloaded
```

### With Cache Mount (Fast)

```dockerfile
# syntax=docker/dockerfile:1

FROM node:20-alpine
WORKDIR /app
COPY package*.json ./

# Mount npm cache — persists across builds!
RUN --mount=type=cache,target=/root/.npm \
    npm ci

COPY . .
RUN npm run build
CMD ["node", "dist/server.js"]
```

**What `--mount=type=cache` does:**
- Creates a persistent cache directory at the specified `target` path
- This directory survives across builds (it's NOT part of the image layer)
- On the next build, the cached packages are already there
- Only new/changed packages need to be downloaded

### Cache Mounts for Different Package Managers

```dockerfile
# syntax=docker/dockerfile:1

# ── npm / Node.js ──
RUN --mount=type=cache,target=/root/.npm \
    npm ci

# ── pip / Python ──
RUN --mount=type=cache,target=/root/.cache/pip \
    pip install -r requirements.txt

# ── apt / Debian ──
RUN --mount=type=cache,target=/var/cache/apt \
    --mount=type=cache,target=/var/lib/apt/lists \
    apt-get update && apt-get install -y curl

# ── Go modules ──
RUN --mount=type=cache,target=/go/pkg/mod \
    --mount=type=cache,target=/root/.cache/go-build \
    go build -o /app/server .

# ── Maven / Java ──
RUN --mount=type=cache,target=/root/.m2/repository \
    mvn package -DskipTests

# ── Gradle / Java ──
RUN --mount=type=cache,target=/root/.gradle/caches \
    gradle build --no-daemon

# ── Rust / Cargo ──
RUN --mount=type=cache,target=/usr/local/cargo/registry \
    --mount=type=cache,target=/app/target \
    cargo build --release
```

### Cache Mount Options

```dockerfile
# Full syntax with all options
RUN --mount=type=cache,\
    target=/path/in/container,\     # Where to mount the cache
    id=my-cache,\                   # Unique cache ID (for sharing between stages)
    sharing=shared,\                # shared|private|locked
    mode=0755,\                     # Permissions
    uid=1000,\                      # Owner user ID
    gid=1000 \                      # Owner group ID
    npm ci
```

| Option | Values | Explanation |
|--------|--------|-------------|
| `target` | Path | Where the cache appears inside the container |
| `id` | String | Unique identifier; defaults to `target` path. Use to share cache between stages |
| `sharing` | `shared` / `private` / `locked` | `shared` = multiple builds can read/write simultaneously; `locked` = only one at a time |
| `mode` | Octal | File permissions for the cache mount |

## 8. Secret Mounts — Safe Build-Time Secrets

We covered this briefly in 9.13, but let's go deeper.

### The Problem

```dockerfile
# ❌ TERRIBLE — secret is stored in image layer history FOREVER
FROM node:20
ARG NPM_TOKEN
RUN echo "//registry.npmjs.org/:_authToken=${NPM_TOKEN}" > .npmrc
RUN npm ci
RUN rm .npmrc   # ← Deleting doesn't help! It's still in a previous layer!
```

Even if you delete the secret in a later step, it's still in the image history:

```bash
docker history myapp
# Shows ALL layers, including the one with the secret
```

### The Fix — BuildKit Secret Mounts

```dockerfile
# syntax=docker/dockerfile:1

FROM node:20-alpine
WORKDIR /app
COPY package*.json ./

# The secret is mounted as a file, used during this RUN, then GONE
# It's NEVER part of any image layer
RUN --mount=type=secret,id=npmrc,target=/root/.npmrc \
    npm ci

COPY . .
RUN npm run build

USER node
CMD ["node", "dist/server.js"]
```

```bash
# Build with the secret
docker buildx build \
  --secret id=npmrc,src=$HOME/.npmrc \
  -t myapp:latest .
```

### Multiple Secrets

```dockerfile
# syntax=docker/dockerfile:1

FROM python:3.12-slim

RUN --mount=type=secret,id=pip_conf,target=/etc/pip.conf \
    --mount=type=secret,id=aws_creds,target=/root/.aws/credentials \
    pip install -r requirements.txt

# Both secrets are available ONLY during this RUN instruction
# Neither appears in any image layer
```

```bash
docker buildx build \
  --secret id=pip_conf,src=./pip.conf \
  --secret id=aws_creds,src=$HOME/.aws/credentials \
  -t myapp .
```

### Secrets from Environment Variables

```bash
# Pass a secret from an env var (not a file)
export MY_SECRET="super-secret-value"

docker buildx build \
  --secret id=my_secret,env=MY_SECRET \
  -t myapp .
```

```dockerfile
# Access it in the Dockerfile
RUN --mount=type=secret,id=my_secret \
    export TOKEN=$(cat /run/secrets/my_secret) && \
    curl -H "Authorization: Bearer $TOKEN" https://api.example.com/download
```

## 9. SSH Mounts — Accessing Private Repos During Build

### The Problem

You need to clone a private Git repo during the build. How do you get the SSH key into the build securely?

```dockerfile
# ❌ BAD — copies your private key into the image!
COPY id_rsa /root/.ssh/id_rsa
RUN git clone git@github.com:myorg/private-repo.git
RUN rm /root/.ssh/id_rsa    # Still in layer history!
```

### The Fix — SSH Mount

```dockerfile
# syntax=docker/dockerfile:1

FROM python:3.12-slim

# Install git
RUN apt-get update && apt-get install -y --no-install-recommends git openssh-client \
    && rm -rf /var/lib/apt/lists/*

# Add GitHub's host key to known_hosts (prevents "unknown host" prompt)
RUN mkdir -p /root/.ssh && ssh-keyscan github.com >> /root/.ssh/known_hosts

# Clone using the mounted SSH key — key is NEVER stored in the image
RUN --mount=type=ssh \
    git clone git@github.com:myorg/private-repo.git /app/private-repo

WORKDIR /app
```

```bash
# Build with SSH forwarding

# Step 1: Make sure your SSH agent has the key loaded
eval $(ssh-agent)
ssh-add ~/.ssh/id_rsa

# Step 2: Build with --ssh flag
docker buildx build --ssh default -t myapp .
```

**How it works:**
1. Your SSH agent (running on the host) holds your private key
2. `--ssh default` tells Docker to forward the SSH agent to the build
3. `--mount=type=ssh` in the Dockerfile makes the SSH agent available during that `RUN`
4. The private key NEVER touches the image — it stays on your machine
5. After the `RUN` completes, the SSH agent connection is closed

### Named SSH Sources

```bash
# If you have multiple SSH keys for different purposes
docker buildx build \
  --ssh github=$HOME/.ssh/github_key \
  --ssh gitlab=$HOME/.ssh/gitlab_key \
  -t myapp .
```

```dockerfile
# Use specific SSH source
RUN --mount=type=ssh,id=github \
    git clone git@github.com:myorg/repo1.git

RUN --mount=type=ssh,id=gitlab \
    git clone git@gitlab.com:myorg/repo2.git
```

## 10. BuildKit Frontend Directives

The `# syntax=` line at the top of a Dockerfile tells BuildKit which parser to use:

```dockerfile
# syntax=docker/dockerfile:1
```

### Available Directives

```dockerfile
# Use the latest stable Dockerfile syntax
# syntax=docker/dockerfile:1

# Use a specific version
# syntax=docker/dockerfile:1.6

# Use experimental features
# syntax=docker/dockerfile:labs

# Use a completely custom frontend (advanced)
# syntax=my-custom-frontend:latest
```

### Why This Matters

Different syntax versions support different features:

| Feature | Minimum Syntax Version |
|---------|----------------------|
| `RUN --mount=type=cache` | `docker/dockerfile:1.2` |
| `RUN --mount=type=secret` | `docker/dockerfile:1.2` |
| `RUN --mount=type=ssh` | `docker/dockerfile:1.2` |
| `COPY --link` | `docker/dockerfile:1.4` |
| `ADD --checksum` | `docker/dockerfile:1.6` |
| `RUN --mount=type=bind` | `docker/dockerfile:1.2` |
| Here-documents | `docker/dockerfile:labs` → `1.4` |

### Here-Documents (Multi-Line Scripts)

```dockerfile
# syntax=docker/dockerfile:1

# Instead of ugly one-liners with &&
# You can write clean multi-line scripts:

RUN <<EOF
apt-get update
apt-get install -y --no-install-recommends \
    curl \
    git \
    vim
rm -rf /var/lib/apt/lists/*
EOF

# You can even use different interpreters:
RUN <<'PYTHON'
#!/usr/bin/env python3
import json
config = {"debug": False, "port": 8080}
with open("/app/config.json", "w") as f:
    json.dump(config, f)
PYTHON

# Create files inline:
COPY <<EOF /etc/nginx/nginx.conf
server {
    listen 80;
    server_name localhost;
    location / {
        proxy_pass http://backend:3000;
    }
}
EOF
```

### COPY --link (Efficient Layer Reuse)

```dockerfile
# syntax=docker/dockerfile:1

# Normal COPY creates a layer that depends on ALL previous layers
COPY file.txt /app/              # ← depends on parent layers

# COPY --link creates an independent layer (better cache reuse)
COPY --link file.txt /app/       # ← independent! can be reused even if
                                 #   parent layers change
```

**When to use `--link`:** When copying files in the final stage of a multi-stage build. The copied files don't depend on previous layers, so `--link` allows better cache reuse.

## 11. What is Buildx?

**Buildx** is a Docker CLI plugin that extends the `docker build` command with full BuildKit features plus multi-platform support.

```
Regular docker build:                    docker buildx build:
─────────────────────                    ─────────────────────

• Uses BuildKit (default now)            • Uses BuildKit
• Single platform only                   • Multi-platform builds ✅
• Single builder                         • Multiple builder instances ✅
• Output: Docker image                   • Multiple output formats ✅
• No remote builders                     • Remote builders ✅
• No bake files                          • Bake support ✅
```

### Is Buildx Installed?

```bash
# Check if buildx is available
docker buildx version
# Output: github.com/docker/buildx v0.12.0 ...

# If not installed (older Docker), install it:
# Docker Desktop: comes bundled
# Linux: 
mkdir -p ~/.docker/cli-plugins
curl -SL https://github.com/docker/buildx/releases/latest/download/buildx-v0.12.0.linux-amd64 \
  -o ~/.docker/cli-plugins/docker-buildx
chmod +x ~/.docker/cli-plugins/docker-buildx
```

## 12. Setting Up Buildx

### Creating a Builder Instance

By default, Docker uses a "default" builder. For multi-platform builds and advanced features, you need to create a new builder:

```bash
# List existing builders
docker buildx ls
# Output:
# NAME/NODE       DRIVER/ENDPOINT  STATUS   BUILDKIT  PLATFORMS
# default *       docker                             
#   default       default          running  v0.12.0   linux/amd64
# desktop-linux   docker                              
#   desktop-linux desktop-linux    running  v0.12.0   linux/amd64, linux/arm64

# Create a new builder with docker-container driver
docker buildx create --name mybuilder --driver docker-container --bootstrap

# Switch to the new builder
docker buildx use mybuilder

# Verify
docker buildx ls
# The new builder should now have a * next to it

# Inspect the builder (shows supported platforms)
docker buildx inspect mybuilder
```

### Builder Drivers

| Driver | Description | Multi-platform | Use Case |
|--------|-------------|---------------|----------|
| `docker` | Uses the Docker daemon directly | ❌ Single platform | Simple builds, default |
| `docker-container` | Runs BuildKit in a container | ✅ Yes (via QEMU) | Multi-platform builds |
| `kubernetes` | Runs BuildKit pods in K8s | ✅ Yes | CI/CD in Kubernetes |
| `remote` | Connects to remote BuildKit | ✅ Yes | Distributed builds |

```bash
# Create builder with docker-container driver (most common for multi-platform)
docker buildx create --name multi-builder --driver docker-container

# Create builder with kubernetes driver
docker buildx create --name k8s-builder --driver kubernetes \
  --driver-opt namespace=buildkit

# Create builder with remote driver
docker buildx create --name remote-builder --driver remote \
  tcp://buildkit-host:1234
```

## 13. Buildx Builders — Understanding Builder Instances

### What is a Builder Instance?

A builder instance is a separate BuildKit daemon that handles your builds. You can have multiple builders for different purposes:

```
┌─────────────────────────────────────────────────────┐
│                  Docker Host                        │
│                                                      │
│  ┌──────────────┐  ┌──────────────┐  ┌────────────┐ │
│  │   default     │  │ mybuilder    │  │ ci-builder │ │
│  │   builder     │  │              │  │            │ │
│  │              │  │  BuildKit in  │  │  BuildKit  │ │
│  │  docker      │  │  container   │  │  in K8s    │ │
│  │  driver      │  │              │  │            │ │
│  │              │  │  Supports:   │  │  Supports: │ │
│  │  amd64 only  │  │  amd64       │  │  amd64     │ │
│  │              │  │  arm64       │  │  arm64     │ │
│  │              │  │  arm/v7      │  │  arm/v7    │ │
│  └──────────────┘  └──────────────┘  └────────────┘ │
└─────────────────────────────────────────────────────┘
```

### Managing Builders

```bash
# Create a builder
docker buildx create --name mybuilder --driver docker-container

# Switch to a builder
docker buildx use mybuilder

# Start/bootstrap a builder
docker buildx inspect --bootstrap mybuilder

# Stop a builder
docker buildx stop mybuilder

# Remove a builder
docker buildx rm mybuilder

# Use a specific builder for one build (without switching)
docker buildx build --builder mybuilder -t myapp .
```

### Multi-Node Builders

You can create a builder that spans multiple machines (nodes), each handling different platforms natively:

```bash
# Create a builder with an AMD64 node
docker buildx create --name cross-builder --node amd64-node \
  --platform linux/amd64 \
  ssh://user@amd64-machine

# Add an ARM64 node to the same builder
docker buildx create --name cross-builder --append --node arm64-node \
  --platform linux/arm64 \
  ssh://user@arm64-machine

# Now builds for each platform run on native hardware
# No emulation needed = MUCH faster!
```

## 14. Multi-Platform Builds — Why?

### The Problem

Different computers use different CPU architectures:

```
Architecture Landscape:
───────────────────────

AMD64 (x86_64):
  • Most desktop PCs and laptops
  • Most traditional cloud servers (AWS EC2, Azure VMs)
  • Intel and AMD processors

ARM64 (aarch64):
  • Apple Silicon (M1, M2, M3, M4)
  • AWS Graviton instances (cheaper + faster)
  • Raspberry Pi 4/5
  • Most smartphones
  • Edge/IoT devices

ARM v7 (armhf):
  • Older Raspberry Pi models
  • Some IoT devices
  • Some Android devices

Others:
  • s390x (IBM mainframes)
  • ppc64le (IBM POWER)
  • riscv64 (emerging architecture)
```

If you build a Docker image on your AMD64 laptop, it **won't run** on an ARM64 server (like AWS Graviton or a Raspberry Pi). You need separate images for each platform.

### Without Multi-Platform Builds (The Hard Way)

```bash
# Build on an AMD64 machine
docker build -t myapp:amd64 .
docker push myapp:amd64

# SSH into an ARM64 machine
ssh arm-server
docker build -t myapp:arm64 .
docker push myapp:arm64

# Users need to pull the RIGHT image for their platform
# Error-prone and tedious!
```

### With Multi-Platform Builds (The Easy Way)

```bash
# Build for BOTH platforms from a single machine!
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  -t myapp:latest \
  --push .

# Users just pull "myapp:latest" and Docker automatically
# gives them the right image for their platform. Magic! ✨
```

## 15. Building Multi-Platform Images

### Step-by-Step Guide

```bash
# Step 1: Create a builder that supports multi-platform
docker buildx create --name multiplatform --driver docker-container --bootstrap
docker buildx use multiplatform

# Step 2: Check what platforms are supported
docker buildx inspect multiplatform
# Platforms: linux/amd64, linux/arm64, linux/arm/v7, linux/arm/v6, linux/386, ...

# Step 3: Build for multiple platforms
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  -t myregistry/myapp:v1.0 \
  --push \
  .
```

### Important: --push vs --load

```bash
# --push: pushes to registry (REQUIRED for multi-platform)
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  -t myregistry/myapp:v1.0 \
  --push .

# --load: loads into local Docker (ONLY works for single platform)
docker buildx build \
  --platform linux/amd64 \
  -t myapp:v1.0 \
  --load .

# ❌ This FAILS — can't load multi-platform image locally
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  -t myapp:v1.0 \
  --load .
# Error: docker exporter does not support multi-platform
```

**Why?** Your local Docker daemon can only hold images for your current platform. Multi-platform images are a registry concept (manifest lists), so they need to be pushed.

### Building for a Single Different Platform

```bash
# Build an ARM64 image on your AMD64 machine
docker buildx build \
  --platform linux/arm64 \
  -t myapp:arm64 \
  --load .

# Now run it (uses QEMU emulation automatically)
docker run --rm myapp:arm64
```

## 16. How Multi-Platform Builds Work Under the Hood

### QEMU Emulation

When you build for a different architecture, BuildKit uses **QEMU** (an emulator) to run the foreign architecture's instructions on your machine.

```
Your AMD64 Machine:
───────────────────

Building for linux/amd64:
  ┌────────────┐
  │  BuildKit  │ → Runs instructions natively → FAST ⚡
  └────────────┘

Building for linux/arm64:
  ┌────────────┐      ┌──────┐
  │  BuildKit  │ ───► │ QEMU │ → Translates ARM → x86 → SLOW 🐌
  └────────────┘      └──────┘
  (Each ARM instruction is emulated on your x86 CPU)
```

### Setting Up QEMU

```bash
# Docker Desktop: QEMU is built-in, nothing to do!

# Linux: Install QEMU user-static
docker run --privileged --rm tonistiigi/binfmt --install all

# Verify supported platforms
docker buildx ls
# Should show: linux/amd64, linux/arm64, linux/arm/v7, ...

# Or check directly
ls /proc/sys/fs/binfmt_misc/
# Should show: qemu-aarch64, qemu-arm, etc.
```

### Manifest Lists — The Multi-Platform Magic

When you push a multi-platform image, Docker creates a **manifest list** (also called an image index). This is a pointer that says "here's the right image for each platform."

```
myregistry/myapp:v1.0  (Manifest List)
├── linux/amd64  → sha256:aaa111... (AMD64 image)
├── linux/arm64  → sha256:bbb222... (ARM64 image)
└── linux/arm/v7 → sha256:ccc333... (ARMv7 image)

When you run:  docker pull myregistry/myapp:v1.0
  └── Docker checks your platform
  └── AMD64 machine → pulls sha256:aaa111...
  └── ARM64 machine → pulls sha256:bbb222...
  └── Raspberry Pi  → pulls sha256:ccc333...
```

### Inspecting Manifest Lists

```bash
# See the manifest list for an image
docker buildx imagetools inspect myregistry/myapp:v1.0

# Example output:
# Name:      myregistry/myapp:v1.0
# MediaType: application/vnd.oci.image.index.v1+json
# Digest:    sha256:xyz789...
#
# Manifests:
#   Name:      myregistry/myapp:v1.0@sha256:aaa111...
#   MediaType: application/vnd.oci.image.manifest.v1+json
#   Platform:  linux/amd64
#
#   Name:      myregistry/myapp:v1.0@sha256:bbb222...
#   MediaType: application/vnd.oci.image.manifest.v1+json
#   Platform:  linux/arm64

# Inspect a specific platform
docker buildx imagetools inspect --raw myregistry/myapp:v1.0 | jq
```

## 17. Multi-Platform Dockerfile Tips

### Using Platform-Specific Build Args

BuildKit automatically provides platform variables you can use:

```dockerfile
# syntax=docker/dockerfile:1

FROM --platform=$BUILDPLATFORM golang:1.22 AS builder

# These are set automatically by BuildKit:
ARG TARGETPLATFORM    # e.g., linux/arm64
ARG TARGETOS          # e.g., linux
ARG TARGETARCH        # e.g., arm64
ARG TARGETVARIANT     # e.g., v7 (for ARM)
ARG BUILDPLATFORM     # e.g., linux/amd64 (your machine)
ARG BUILDOS           # e.g., linux
ARG BUILDARCH         # e.g., amd64

WORKDIR /app
COPY . .

# Cross-compile for the target platform
# This runs on your native platform (fast!) but compiles for the target
RUN GOOS=${TARGETOS} GOARCH=${TARGETARCH} go build -o /app/server .

# Final stage: use the target platform image
FROM --platform=$TARGETPLATFORM alpine:3.19

COPY --from=builder /app/server /server
ENTRYPOINT ["/server"]
```

**Key insight:** Notice `FROM --platform=$BUILDPLATFORM` in the builder stage. This means the build step runs on YOUR machine's architecture (native, fast), but cross-compiles for the target. Without this, it would use QEMU (slow).

### Available Platform Variables

| Variable | Example | Description |
|----------|---------|-------------|
| `BUILDPLATFORM` | `linux/amd64` | Platform of the build machine |
| `BUILDOS` | `linux` | OS of the build machine |
| `BUILDARCH` | `amd64` | Architecture of the build machine |
| `TARGETPLATFORM` | `linux/arm64` | Target platform being built |
| `TARGETOS` | `linux` | Target OS |
| `TARGETARCH` | `arm64` | Target architecture |
| `TARGETVARIANT` | `v7` | Target variant (e.g., ARM v7) |

### Handling Platform-Specific Dependencies

```dockerfile
# syntax=docker/dockerfile:1

FROM --platform=$TARGETPLATFORM ubuntu:22.04

ARG TARGETARCH

# Install platform-specific packages
RUN apt-get update && \
    if [ "$TARGETARCH" = "amd64" ]; then \
        apt-get install -y lib-amd64-specific; \
    elif [ "$TARGETARCH" = "arm64" ]; then \
        apt-get install -y lib-arm64-specific; \
    fi

# Download platform-specific binary
ARG TARGETOS
ARG TARGETARCH
RUN curl -L "https://example.com/tool-${TARGETOS}-${TARGETARCH}" \
    -o /usr/local/bin/tool && \
    chmod +x /usr/local/bin/tool
```

## 19. Build Outputs — Local, Registry, OCI, tar

Buildx can output images in different formats, not just to the Docker daemon:

```bash
# Output 1: Push to registry (default for multi-platform)
docker buildx build --output type=registry,name=myregistry/myapp:v1 .
# Same as: docker buildx build --push -t myregistry/myapp:v1 .

# Output 2: Load into local Docker daemon
docker buildx build --output type=docker,name=myapp:v1 .
# Same as: docker buildx build --load -t myapp:v1 .

# Output 3: Save as OCI tar archive
docker buildx build --output type=oci,dest=./myapp.tar .
# Creates an OCI-compliant tar file

# Output 4: Save as Docker tar archive
docker buildx build --output type=docker,dest=./myapp-docker.tar .
# Same as docker save

# Output 5: Export to local filesystem (just the files, no image)
docker buildx build --output type=local,dest=./output .
# Extracts the image's filesystem to ./output directory
# Useful for extracting build artifacts without creating an image
```

### Practical Use Case — Extract Build Artifacts

```dockerfile
# Build stage produces artifacts
FROM node:20 AS builder
WORKDIR /app
COPY . .
RUN npm ci && npm run build

# No final stage needed! Just extract the artifacts.
FROM scratch
COPY --from=builder /app/dist /dist
```

```bash
# Extract only the built files to your local machine
docker buildx build --output type=local,dest=./build-output .
ls ./build-output/dist/
# → Your built files are here, no Docker image created!
```

## 21. Bake — Declarative Build Definitions

**Bake** is like Docker Compose but for builds. Instead of long `docker buildx build` commands, you define everything in a file.

### docker-bake.hcl (HCL Format)

```hcl
# docker-bake.hcl

# Variables
variable "TAG" {
  default = "latest"
}

variable "REGISTRY" {
  default = "myregistry"
}

# Groups — build multiple targets at once
group "default" {
  targets = ["app", "worker", "nginx"]
}

group "ci" {
  targets = ["app", "worker"]
}

# Target: App
target "app" {
  dockerfile = "Dockerfile"
  context    = "./app"
  tags       = ["${REGISTRY}/app:${TAG}"]
  platforms  = ["linux/amd64", "linux/arm64"]
  cache-from = ["type=registry,ref=${REGISTRY}/app:cache"]
  cache-to   = ["type=registry,ref=${REGISTRY}/app:cache,mode=max"]
}

# Target: Worker
target "worker" {
  dockerfile = "Dockerfile.worker"
  context    = "./worker"
  tags       = ["${REGISTRY}/worker:${TAG}"]
  platforms  = ["linux/amd64", "linux/arm64"]
  args = {
    NODE_ENV = "production"
  }
}

# Target: Nginx
target "nginx" {
  dockerfile = "nginx/Dockerfile"
  context    = "./nginx"
  tags       = ["${REGISTRY}/nginx:${TAG}"]
  platforms  = ["linux/amd64"]
}
```

### Using Bake

```bash
# Build all targets in the default group
docker buildx bake

# Build a specific target
docker buildx bake app

# Build a group
docker buildx bake ci

# Override variables
docker buildx bake --set TAG=v2.0

# Push all images
docker buildx bake --push

# Dry run — see what would be built
docker buildx bake --print
```

### docker-bake.json (JSON Format)

```json
{
  "group": {
    "default": {
      "targets": ["app", "worker"]
    }
  },
  "target": {
    "app": {
      "dockerfile": "Dockerfile",
      "context": "./app",
      "tags": ["myregistry/app:latest"],
      "platforms": ["linux/amd64", "linux/arm64"]
    },
    "worker": {
      "dockerfile": "Dockerfile.worker",
      "context": "./worker",
      "tags": ["myregistry/worker:latest"]
    }
  }
}
```

### Inheritance in Bake

```hcl
# Common settings as a base target
target "_common" {
  platforms = ["linux/amd64", "linux/arm64"]
  cache-from = ["type=gha"]
  cache-to   = ["type=gha,mode=max"]
}

# Inherit from _common
target "app" {
  inherits = ["_common"]
  dockerfile = "Dockerfile"
  tags = ["myregistry/app:latest"]
}

target "worker" {
  inherits = ["_common"]
  dockerfile = "Dockerfile.worker"
  tags = ["myregistry/worker:latest"]
}
```

## 22. BuildKit Debugging and Troubleshooting

### Build with Verbose Output

```bash
# Show detailed output (don't collapse finished steps)
docker buildx build --progress=plain -t myapp .

# Progress output options:
# --progress=auto     → Default, auto-detect terminal
# --progress=plain    → Full output, no collapsing (best for CI)
# --progress=tty      → Fancy TTY output with collapsing
# --progress=rawjson  → JSON output (for tooling)
```

### Debug a Failing Build

```bash
# Method 1: Use --progress=plain to see full output
docker buildx build --progress=plain -t myapp . 2>&1 | tee build.log

# Method 2: Use --no-cache to force fresh build
docker buildx build --no-cache -t myapp .

# Method 3: Build up to a specific stage
docker buildx build --target builder -t myapp:builder .
# Then inspect:
docker run --rm -it myapp:builder sh

# Method 4: Add debug RUN instructions
# In your Dockerfile:
RUN ls -la /app/           # Check files
RUN cat /app/package.json  # Check content
RUN whoami                 # Check user
RUN env                    # Check environment
```

### Common BuildKit Errors and Fixes

| Error | Cause | Fix |
|-------|-------|-----|
| `failed to solve: rpc error` | BuildKit daemon issue | `docker buildx rm mybuilder && docker buildx create --name mybuilder` |
| `no match for platform` | Image doesn't support target platform | Check if base image supports the platform |
| `cache export failed` | Registry auth issue | Check `docker login` |
| `executor failed running...` | Command failed during build | Check the failing RUN instruction |
| `context canceled` | Build timeout or interruption | Increase timeout, check resources |

### Cleaning Up BuildKit Cache

```bash
# See BuildKit cache usage
docker buildx du

# Prune BuildKit cache
docker buildx prune

# Prune ALL cache (aggressive)
docker buildx prune --all --force

# Prune cache older than 24 hours
docker buildx prune --filter until=24h
```

## 23. Performance Optimization Tips

### 1. Order Dockerfile Instructions by Change Frequency

```dockerfile
# ✅ Put rarely-changing stuff first, frequently-changing stuff last

FROM node:20-alpine          # Rarely changes
WORKDIR /app                 # Never changes

COPY package*.json ./        # Changes when dependencies change
RUN npm ci                   # Re-runs only when package.json changes

COPY . .                     # Changes on every code change
RUN npm run build            # Re-runs on every code change
```

### 2. Use Cache Mounts for Package Managers

```dockerfile
# 40-60% faster rebuilds!
RUN --mount=type=cache,target=/root/.npm npm ci
```

### 3. Use --platform=$BUILDPLATFORM for Cross-Compilation

```dockerfile
# ✅ Build natively, cross-compile for target
FROM --platform=$BUILDPLATFORM golang:1.22 AS builder
ARG TARGETOS TARGETARCH
RUN GOOS=$TARGETOS GOARCH=$TARGETARCH go build -o /app .

# ❌ Without this, QEMU emulates the entire build = 10x slower
```

### 4. Use Multi-Stage Builds to Parallelize

```dockerfile
# Independent stages run in parallel automatically
FROM node:20 AS frontend
RUN npm run build

FROM golang:1.22 AS backend
RUN go build

FROM alpine
COPY --from=frontend /dist ./
COPY --from=backend /app ./
```

### 5. Use COPY --link for Better Cache Reuse

```dockerfile
# Independent of parent layers → better cache hit rate
COPY --link --from=builder /app/binary /usr/local/bin/
```

### Build Speed Comparison

```
Scenario                                    Time
────────────────────────────────────────────────────
First build (no cache)                      120s
Rebuild (code change, WITH layer cache)      15s
Rebuild (WITH cache mounts)                   8s
Cross-platform (WITH QEMU emulation)        300s
Cross-platform (WITH cross-compilation)      25s
Parallel multi-stage (3 stages)              45s
Sequential multi-stage (3 stages)           120s
```
