## 1. Why Logging & Debugging Matter in Docker

In the container world, you can't just SSH into a server and tail a log file like the old days. Containers are:

- **Ephemeral** — they can be created, destroyed, and recreated at any time
- **Isolated** — you can't just look at files on the host
- **Distributed** — your app might run across 50 containers

This means you need a proper logging strategy and solid debugging skills:

```
Without proper logging:                  With proper logging:
──────────────────────                   ─────────────────────

"The app is broken."                     "Container X threw OOM error at
"I don't know why."                       14:32:05 UTC. Memory usage spiked
"Let me SSH in... wait,                   from 256MB to 512MB limit. Root
 there's no SSH."                         cause: memory leak in endpoint
"Which container was it?"                 /api/users. Stack trace: ..."
"The logs are gone because
 the container restarted."               Clear. Actionable. Fixable.
```

## 2. How Docker Logging Works — The Basics

Docker captures everything your application writes to **STDOUT** (standard output) and **STDERR** (standard error), and sends it to a **log driver**.

```
┌────────────────────────────────────────────────────┐
│                     Container                       │
│                                                     │
│  ┌─────────────┐                                   │
│  │ Application  │                                   │
│  │              │──── print("Hello") ──► STDOUT ────┤
│  │              │──── console.error() ─► STDERR ────┤
│  └─────────────┘                                   │
│                                                     │
└──────────────────────┬─────────────────────────────┘
                       │
                       ▼
            ┌─────────────────────┐
            │    Docker Daemon    │
            │    (dockerd)        │
            │                     │
            │  ┌──────────────┐   │
            │  │  Log Driver   │  │
            │  └──────┬───────┘  │
            └─────────┤──────────┘
                      │
         ┌────────────┼────────────────┐
         ▼            ▼                ▼
    ┌─────────┐  ┌─────────┐    ┌──────────┐
    │ json-file│  │ syslog  │    │ fluentd  │
    │ (local)  │  │         │    │          │
    └─────────┘  └─────────┘    └──────────┘
```

**Key concept:** Docker doesn't read log files from your container's filesystem. It only captures what goes to STDOUT and STDERR. So your application must log to STDOUT/STDERR, not to files inside the container.

## 3. docker logs — Your First Debugging Tool

The `docker logs` command shows the logs captured from a container's STDOUT and STDERR.

```bash
# Basic usage — show all logs
docker logs <container_name_or_id>

# Follow logs in real-time (like tail -f)
docker logs -f mycontainer

# Show last 100 lines
docker logs --tail 100 mycontainer

# Show logs with timestamps
docker logs -t mycontainer

# Show logs since a specific time
docker logs --since 2024-01-15T10:00:00 mycontainer

# Show logs in the last 30 minutes
docker logs --since 30m mycontainer

# Show logs until a specific time
docker logs --until 2024-01-15T11:00:00 mycontainer

# Combine: last 50 lines with timestamps, follow
docker logs --tail 50 -t -f mycontainer

# Show only STDERR
docker logs mycontainer 2>&1 1>/dev/null

# Show only STDOUT
docker logs mycontainer 2>/dev/null

# Search logs with grep
docker logs mycontainer 2>&1 | grep "ERROR"
docker logs mycontainer 2>&1 | grep -i "exception"
```

### Important Limitations of `docker logs`

| Limitation | Details |
|-----------|---------|
| **Only works with some log drivers** | `json-file`, `local`, `journald` support `docker logs`. Others (syslog, fluentd, etc.) do NOT. |
| **Not for production monitoring** | Use a centralized logging system (EFK, Loki, CloudWatch) |
| **Can use lots of disk** | Without log rotation, logs can fill your disk |
| **Logs are lost when container is removed** | `docker rm` = logs gone (unless using external driver) |

## 4. Understanding STDOUT and STDERR

### Why STDOUT and STDERR?

Docker follows the **12-Factor App** methodology, which says:

> **"Treat logs as event streams. A twelve-factor app never concerns itself with routing or storage of its output stream."**

In plain English: your app should just print to STDOUT/STDERR and let the platform (Docker) handle where the logs go.

### Configuring Your App to Log to STDOUT

```python
# Python — configure logging to stdout
import logging
import sys

logging.basicConfig(
    level=logging.INFO,
    format='%(asctime)s [%(levelname)s] %(name)s: %(message)s',
    handlers=[
        logging.StreamHandler(sys.stdout)   # ← Logs to STDOUT
    ]
)

logger = logging.getLogger(__name__)
logger.info("Application started")          # → goes to STDOUT → docker logs
logger.error("Something went wrong")        # → goes to STDERR → docker logs
```

```javascript
// Node.js — console.log goes to STDOUT, console.error goes to STDERR
console.log("Info message");      // STDOUT
console.error("Error message");   // STDERR

// For structured logging, use a library like winston or pino:
const pino = require('pino');
const logger = pino({ level: 'info' });   // Logs to STDOUT as JSON
logger.info({ userId: 123 }, 'User logged in');
```

```nginx
# Nginx — redirect logs to stdout/stderr in Dockerfile
# This is already done in the official nginx image, but here's how:

# In nginx.conf or Dockerfile:
RUN ln -sf /dev/stdout /var/log/nginx/access.log \
    && ln -sf /dev/stderr /var/log/nginx/error.log
```

### What If My App Writes to Files?

Some apps (especially legacy ones) write logs to files. You have options:

```dockerfile
# Option 1: Symlink log files to stdout/stderr
RUN ln -sf /dev/stdout /var/log/app/access.log \
    && ln -sf /dev/stderr /var/log/app/error.log

# Option 2: Use tail to stream file to stdout
CMD ["sh", "-c", "myapp & tail -f /var/log/app/app.log"]

# Option 3: Use a sidecar container (Kubernetes pattern)
# One container writes logs to a shared volume
# Another container reads and forwards them
```

## 5. Docker Log Drivers — Overview

A **log driver** determines WHERE Docker sends your container's logs. You can configure it globally (for all containers) or per-container.

### Available Log Drivers

| Driver | Where Logs Go | `docker logs` Works? | Use Case |
|--------|--------------|---------------------|----------|
| `json-file` | JSON files on host | ✅ Yes | Default, development |
| `local` | Optimized local files | ✅ Yes | Better default for production |
| `syslog` | Syslog daemon | ❌ No | Linux syslog integration |
| `journald` | systemd journal | ✅ Yes | Systemd-based systems |
| `fluentd` | Fluentd collector | ❌ No | Centralized logging (EFK) |
| `awslogs` | AWS CloudWatch | ❌ No | AWS environments |
| `splunk` | Splunk HEC | ❌ No | Splunk users |
| `gelf` | Graylog (GELF) | ❌ No | Graylog/ELK users |
| `gcplogs` | Google Cloud Logging | ❌ No | GCP environments |
| `none` | Nowhere (disabled) | ❌ No | Performance-critical, no logs needed |

### Setting Log Driver — Per Container

```bash
# Use syslog driver for this specific container
docker run --log-driver=syslog --log-opt syslog-address=udp://logserver:514 nginx

# Use json-file with specific options
docker run --log-driver=json-file \
  --log-opt max-size=10m \
  --log-opt max-file=3 \
  nginx
```

### Setting Log Driver — Globally (Default for All Containers)

```json
// /etc/docker/daemon.json
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3",
    "labels": "production_status",
    "env": "os,customer"
  }
}
```

```bash
# After editing daemon.json, restart Docker
sudo systemctl restart docker

# Verify
docker info --format '{{.LoggingDriver}}'
# Output: json-file
```

### Checking a Container's Log Driver

```bash
# Check what log driver a container is using
docker inspect --format='{{.HostConfig.LogConfig.Type}}' <container>
# Output: json-file
```

## 6. json-file Log Driver (Default)

This is Docker's default log driver. It stores logs as JSON files on the host.

### Where Are the Log Files?

```bash
# Logs are stored here (Linux):
/var/lib/docker/containers/<container-id>/<container-id>-json.log

# Find the log file for a specific container
docker inspect --format='{{.LogPath}}' <container>
# Output: /var/lib/docker/containers/abc123.../abc123...-json.log
```

### What the Log File Looks Like

```json
{"log":"2024-01-15T10:30:00.123456789Z Starting application...\n","stream":"stdout","time":"2024-01-15T10:30:00.123456789Z"}
{"log":"2024-01-15T10:30:01.234567890Z Listening on port 8080\n","stream":"stdout","time":"2024-01-15T10:30:01.23456789Z"}
{"log":"2024-01-15T10:30:05.345678901Z ERROR: Connection refused to database\n","stream":"stderr","time":"2024-01-15T10:30:05.345678901Z"}
```

Each line is a JSON object with:
- `log` — The actual log message
- `stream` — Either `stdout` or `stderr`
- `time` — When the log was captured

### Configuration Options

```bash
docker run \
  --log-driver=json-file \
  --log-opt max-size=10m \      # Max size per log file (before rotation)
  --log-opt max-file=5 \        # Keep max 5 rotated log files
  --log-opt compress=true \     # Compress rotated files with gzip
  --log-opt labels=app,env \    # Add container labels to log metadata
  --log-opt tag="{{.Name}}" \   # Custom tag in log output
  nginx
```

| Option | Default | Description |
|--------|---------|-------------|
| `max-size` | unlimited | Max log file size before rotation (e.g., `10m`, `1g`) |
| `max-file` | 1 | Number of rotated log files to keep |
| `compress` | disabled | Compress rotated files |
| `labels` | none | Include specified container labels in log metadata |
| `env` | none | Include specified env vars in log metadata |
| `tag` | Container ID | Custom tag template |

### ⚠️ The Biggest Gotcha with json-file

By default, there is **NO log rotation** and **NO size limit**! This means logs can grow forever and fill your entire disk.

```
Day 1:   Container log file = 10 MB       ← Fine
Day 30:  Container log file = 300 MB      ← Getting big
Day 90:  Container log file = 1 GB        ← Uh oh
Day 365: Container log file = 4 GB        ← DISK FULL! 💀
```

**ALWAYS set `max-size` and `max-file`**. See Section 15 for details.

## 7. local Log Driver

The `local` log driver is an improved alternative to `json-file`. It uses a more efficient format and has built-in rotation.

```json
// /etc/docker/daemon.json
{
  "log-driver": "local",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3",
    "compress": "true"
  }
}
```

### Advantages Over json-file

| Feature | json-file | local |
|---------|----------|-------|
| Format | JSON (text) | Protocol buffers (binary) |
| Disk usage | Higher | ~Lower (compressed) |
| Performance | Good | Better |
| `docker logs` works | ✅ Yes | ✅ Yes |
| Default rotation | ❌ No | ✅ Yes (100MB, 5 files) |
| Human-readable files | ✅ Yes | ❌ No (binary format) |

### When to Use `local` Instead of `json-file`

- When you want built-in rotation without extra config
- When disk space is a concern
- When you don't need to parse log files directly (only use `docker logs`)

## 8. syslog Log Driver

Sends logs to a syslog server (like rsyslog or syslog-ng).

```bash
# Send logs to local syslog
docker run --log-driver=syslog nginx

# Send logs to remote syslog server
docker run \
  --log-driver=syslog \
  --log-opt syslog-address=udp://logserver:514 \
  --log-opt syslog-facility=daemon \
  --log-opt tag="myapp" \
  nginx

# Using TCP
docker run \
  --log-driver=syslog \
  --log-opt syslog-address=tcp://logserver:514 \
  --log-opt syslog-tls-cert=/certs/cert.pem \
  --log-opt syslog-tls-key=/certs/key.pem \
  nginx
```

### Configuration Options

| Option | Description | Example |
|--------|-------------|---------|
| `syslog-address` | Address of syslog server | `udp://loghost:514` |
| `syslog-facility` | Syslog facility | `daemon`, `local0`-`local7` |
| `tag` | Tag for log messages | `myapp`, `{{.Name}}` |
| `syslog-tls-cert` | TLS certificate path | `/certs/cert.pem` |
| `syslog-format` | Syslog format | `rfc3164`, `rfc5424` |

> **Note:** `docker logs` does NOT work with the syslog driver. You need to query your syslog server directly.

## 9. journald Log Driver

Sends logs to systemd's journal. Great for systems already using systemd.

```bash
# Use journald driver
docker run --log-driver=journald --name myapp nginx

# View logs using journalctl
journalctl CONTAINER_NAME=myapp

# Follow logs
journalctl -f CONTAINER_NAME=myapp

# Filter by time
journalctl CONTAINER_NAME=myapp --since "2024-01-15 10:00" --until "2024-01-15 11:00"

# docker logs also works with journald!
docker logs myapp
```

**Best of both worlds:** journald is one of the few non-default drivers where `docker logs` still works!

## 10. fluentd Log Driver

Sends logs to Fluentd, which is a popular log collector/aggregator.

```bash
# Run Fluentd first
docker run -d \
  --name fluentd \
  -p 24224:24224 \
  fluent/fluentd:v1.16

# Then run your app with fluentd log driver
docker run \
  --log-driver=fluentd \
  --log-opt fluentd-address=localhost:24224 \
  --log-opt tag="docker.{{.Name}}" \
  --log-opt fluentd-async=true \
  nginx
```

### Docker Compose with Fluentd

```yaml
services:
  app:
    image: myapp
    logging:
      driver: fluentd
      options:
        fluentd-address: "localhost:24224"
        tag: "docker.{{.Name}}"
        fluentd-async: "true"      # Don't block the app if Fluentd is down

  fluentd:
    image: fluent/fluentd:v1.16
    ports:
      - "24224:24224"
    volumes:
      - ./fluentd/conf:/fluentd/etc
```

| Option | Description |
|--------|-------------|
| `fluentd-address` | Where Fluentd is listening |
| `tag` | Tag for routing logs in Fluentd |
| `fluentd-async` | Send logs asynchronously (recommended!) |
| `fluentd-buffer-limit` | Buffer size before dropping logs |
| `fluentd-retry-wait` | Wait time between retries |

> **Important:** Set `fluentd-async=true`! Otherwise, if Fluentd goes down, your containers will hang waiting to send logs.

## 15. Log Rotation — Preventing Disk Full

This is **critical for production**. Without log rotation, Docker will fill your entire disk.

### Configure Log Rotation Globally

```json
// /etc/docker/daemon.json
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "50m",
    "max-file": "5"
  }
}
```

This means:
- Each log file can grow up to **50 MB**
- Docker keeps the last **5** rotated files
- Maximum disk usage per container: **250 MB** (50 MB × 5 files)
- When the 6th file would be created, the oldest one is deleted

```bash
# Restart Docker to apply
sudo systemctl restart docker
```

### Configure Log Rotation Per Container

```bash
docker run \
  --log-opt max-size=10m \
  --log-opt max-file=3 \
  nginx
```

### Configure Log Rotation in Docker Compose

```yaml
services:
  web:
    image: nginx
    logging:
      driver: json-file
      options:
        max-size: "50m"
        max-file: "5"

  app:
    image: myapp
    logging:
      driver: json-file
      options:
        max-size: "100m"
        max-file: "10"
```

### Check Current Log File Sizes

```bash
# Check log file sizes for ALL containers
sudo du -sh /var/lib/docker/containers/*/*-json.log

# Check a specific container's log
sudo du -sh $(docker inspect --format='{{.LogPath}}' mycontainer)
```

### Emergency: Disk Full Because of Logs

```bash
# Quick fix — truncate the log file (don't delete it!)
sudo truncate -s 0 /var/lib/docker/containers/<container-id>/<container-id>-json.log

# Better fix — configure rotation and restart the container
docker update --log-opt max-size=50m --log-opt max-file=5 <container>
```

## 16. Logging Best Practices for Applications

### 1. Use Structured Logging (JSON)

```
❌ Unstructured log:
2024-01-15 10:30:00 User john logged in from 192.168.1.5

✅ Structured log (JSON):
{"time":"2024-01-15T10:30:00Z","level":"info","msg":"User logged in","user":"john","ip":"192.168.1.5","module":"auth"}
```

Structured logs are easier to search, filter, and analyze in tools like Elasticsearch, Loki, or CloudWatch.

### 2. Use Proper Log Levels

```
FATAL  — App is about to crash. Call the fire department.
ERROR  — Something broke. Needs attention but app continues.
WARN   — Something unexpected but not broken. Keep an eye on it.
INFO   — Normal operations. "User logged in", "Order placed".
DEBUG  — Detailed info for developers. Too verbose for production.
TRACE  — Super detailed. Only for debugging specific issues.
```

### 3. Include Context in Logs

```json
// ❌ Bad — useless
{"msg": "Error occurred"}

// ✅ Good — actionable
{
  "msg": "Database connection failed",
  "error": "connection refused",
  "host": "db-primary.internal",
  "port": 5432,
  "retry_count": 3,
  "request_id": "req-abc-123"
}
```

### 4. Don't Log Sensitive Data

```python
# ❌ NEVER log these
logger.info(f"User login: password={password}")
logger.info(f"API call with token: {api_key}")
logger.info(f"Credit card: {card_number}")

# ✅ Mask or omit sensitive data
logger.info(f"User login: user={username}")
logger.info(f"API call: token=***{api_key[-4:]}")
```

### 5. Use Request IDs for Tracing

```python
import uuid

request_id = str(uuid.uuid4())
logger.info(f"Processing order", extra={
    "request_id": request_id,
    "order_id": 12345,
    "user_id": 678
})
# Now you can trace a single request across multiple containers
```

## 17. Debugging: Container Won't Start

This is the most common issue. Let's go through the debugging steps:

### Step 1: Check Container Status

```bash
# List all containers (including stopped ones)
docker ps -a

# Output:
# CONTAINER ID  IMAGE     STATUS                        NAMES
# abc123        myapp     Exited (1) 2 minutes ago     myapp
# def456        myapp     Created                       myapp2
```

**Status meanings:**

| Status | Meaning |
|--------|---------|
| `Created` | Container was created but never started |
| `Running` | Container is running normally |
| `Exited (0)` | Container finished successfully |
| `Exited (1)` | Container crashed (error code 1) |
| `Exited (137)` | Container killed (OOM or `docker kill`) |
| `Exited (139)` | Segmentation fault |
| `Exited (143)` | Graceful termination (SIGTERM) |
| `Restarting` | Container is in a restart loop |

### Step 2: Check Logs

```bash
# Even if the container exited, logs are still available
docker logs myapp
docker logs --tail 50 myapp
```

### Step 3: Check Container Details

```bash
# Full details about the container
docker inspect myapp

# Check the error specifically
docker inspect myapp --format='{{.State.Error}}'
docker inspect myapp --format='{{.State.ExitCode}}'
docker inspect myapp --format='{{.State.OOMKilled}}'
```

### Step 4: Try Running Interactively

```bash
# Override CMD and get a shell instead
docker run -it --rm myapp sh
# or
docker run -it --rm myapp bash

# Now you can poke around inside the container
ls -la /app/
cat /app/config.json
env                     # Check environment variables
whoami                  # Check user
```

### Step 5: Check the Entrypoint

```bash
# See what the container is trying to run
docker inspect myapp --format='{{json .Config.Entrypoint}}'
docker inspect myapp --format='{{json .Config.Cmd}}'

# Common issue: entrypoint script doesn't have execute permission
docker run -it --rm --entrypoint sh myapp
chmod +x /entrypoint.sh     # Fix it
```

### Common "Won't Start" Issues & Fixes

| Problem | Symptoms | Fix |
|---------|----------|-----|
| **Port already in use** | "bind: address already in use" | Use a different host port: `-p 8081:80` |
| **Missing env vars** | App crashes immediately | Set required env vars: `-e VAR=value` |
| **Permission denied** | "Permission denied" in logs | Fix file permissions or run as correct user |
| **Missing files** | "No such file" in logs | Check COPY/ADD in Dockerfile, check volumes |
| **Wrong CMD/ENTRYPOINT** | "exec format error" | Check shebang line, file encoding, architecture |
| **OOM Killed** | Exit code 137, OOMKilled=true | Increase memory limit: `--memory=1g` |
| **Wrong image arch** | "exec format error" | Use correct platform image |

## 22. docker stats — Real-Time Resource Monitoring

```bash
# Show live stats for all running containers
docker stats

# Output:
# CONTAINER ID  NAME    CPU %  MEM USAGE / LIMIT    MEM %  NET I/O          BLOCK I/O
# abc123        web     2.5%   128MiB / 512MiB      25%    1.5kB / 2.1kB    0B / 0B
# def456        db      15.3%  384MiB / 1GiB        37.5%  5.2MB / 3.1MB    25MB / 10MB
# ghi789        cache   0.1%   64MiB / 256MiB       25%    800B / 500B      0B / 0B

# Stats for specific containers
docker stats web db cache

# One-shot (no live update)
docker stats --no-stream

# Custom format
docker stats --format "table {{.Name}}\t{{.CPUPerc}}\t{{.MemUsage}}\t{{.MemPerc}}"

# JSON output
docker stats --no-stream --format '{{json .}}' | jq
```

### Understanding the Stats

| Column | What It Means | Warning Signs |
|--------|--------------|---------------|
| **CPU %** | CPU usage relative to all host CPUs | >80% sustained = investigate |
| **MEM USAGE** | Current memory usage | Approaching LIMIT = OOM risk |
| **MEM %** | Percentage of memory limit used | >90% = danger zone |
| **NET I/O** | Network traffic in/out | Unexpected high traffic = leak/attack |
| **BLOCK I/O** | Disk read/write | High writes = check logging/temp files |
| **PIDs** | Number of processes | Growing = possible fork bomb |

### How `docker stats` actually works

`docker stats` reads metrics that the Linux kernel already tracks. It is not inventing values — it is displaying cgroup and kernel counters for the container.

`docker stats` uses data from:
- `/sys/fs/cgroup/` — container CPU, memory, block I/O, and PID accounting
- network interfaces and veth pairs — bytes in/out for the container
- container process state and kernel counters

Example commands to inspect the underlying data:

```bash
# Find the main PID of a container
docker inspect --format='{{.State.Pid}}' mycontainer

# Show the cgroup paths for that PID
cat /proc/$(docker inspect --format='{{.State.Pid}}' mycontainer)/cgroup

# Read current memory usage from cgroup
cat /sys/fs/cgroup/memory/docker/<container-id>/memory.current
```

This is the right answer to "how much resources are consumed by any Docker container right now".

For production monitoring, teams usually use Prometheus, cAdvisor, Grafana, or cloud provider metrics, which also read the same kernel/cgroup data.

## 30. Centralized Logging with EFK Stack

For production, you want centralized logging. The **EFK** stack (Elasticsearch + Fluentd + Kibana) is a popular choice.

```
┌──────────┐    ┌──────────┐    ┌──────────┐
│ Container │    │ Container │    │ Container │
│     A     │    │     B     │    │     C     │
└─────┬─────┘    └─────┬─────┘    └─────┬─────┘
      │                │                │
      └────────────────┼────────────────┘
                       │
                       ▼
              ┌────────────────┐
              │    Fluentd     │  ← Collects & routes logs
              │  (Log Router)  │
              └───────┬────────┘
                      │
                      ▼
              ┌────────────────┐
              │ Elasticsearch  │  ← Stores & indexes logs
              │  (Log Store)   │
              └───────┬────────┘
                      │
                      ▼
              ┌────────────────┐
              │    Kibana      │  ← Visualizes & searches logs
              │  (Dashboard)   │
              └────────────────┘
```

### Docker Compose for EFK Stack

```yaml
# docker-compose.yml
services:
  elasticsearch:
    image: docker.elastic.co/elasticsearch/elasticsearch:8.12.0
    environment:
      - discovery.type=single-node
      - ES_JAVA_OPTS=-Xms512m -Xmx512m
      - xpack.security.enabled=false
    ports:
      - "9200:9200"
    volumes:
      - es-data:/usr/share/elasticsearch/data
    logging:
      driver: json-file
      options:
        max-size: "50m"
        max-file: "3"

  kibana:
    image: docker.elastic.co/kibana/kibana:8.12.0
    ports:
      - "5601:5601"
    environment:
      - ELASTICSEARCH_HOSTS=http://elasticsearch:9200
    depends_on:
      - elasticsearch

  fluentd:
    build: ./fluentd
    ports:
      - "24224:24224"
      - "24224:24224/udp"
    depends_on:
      - elasticsearch

  # Your application — uses fluentd log driver
  app:
    image: myapp:latest
    logging:
      driver: fluentd
      options:
        fluentd-address: "localhost:24224"
        tag: "app.{{.Name}}"
        fluentd-async: "true"

volumes:
  es-data:
```

### Alternative: Grafana Loki (Lighter Weight)

```yaml
services:
  loki:
    image: grafana/loki:2.9.0
    ports:
      - "3100:3100"
    volumes:
      - loki-data:/loki

  promtail:
    image: grafana/promtail:2.9.0
    volumes:
      - /var/log:/var/log
      - /var/lib/docker/containers:/var/lib/docker/containers:ro
    command: -config.file=/etc/promtail/config.yml

  grafana:
    image: grafana/grafana:10.2.0
    ports:
      - "3000:3000"
    environment:
      - GF_AUTH_ANONYMOUS_ENABLED=true
      - GF_AUTH_ANONYMOUS_ORG_ROLE=Admin

volumes:
  loki-data:
```

## 31. Real-World Debugging Scenarios

### Scenario 1: "My Container Keeps Restarting"

```bash
# Step 1: Check status
docker ps -a
# STATUS: Restarting (1) 3 seconds ago

# Step 2: Check logs
docker logs --tail 20 myapp
# Error: EADDRINUSE: address already in use :::3000

# Step 3: Diagnose — another process is using port 3000
docker ps --filter "publish=3000"
# Found it! Another container is already using port 3000

# Step 4: Fix
docker stop other-container
docker start myapp
# ✅ Fixed!
```

### Scenario 2: "The App Is Super Slow"

```bash
# Step 1: Check resource usage
docker stats myapp
# CPU: 95%, MEM: 490MiB / 512MiB (96%)

# Step 2: The container is hitting its memory limit!
# Check if it was OOM killed recently
docker inspect myapp --format='{{.State.OOMKilled}}'
# true

# Step 3: Fix — increase memory limit
docker update --memory=1g --memory-swap=2g myapp
# Or recreate with higher limits
docker run --memory=1g --cpus=2 myapp
```

### Scenario 3: "Container Can't Connect to Database"

```bash
# Step 1: Check if containers are on the same network
docker network inspect my-network | jq '.[0].Containers'

# Step 2: App is on "bridge", DB is on "my-network" — different networks!
docker network connect my-network myapp

# Step 3: Verify connectivity
docker exec myapp ping db
# PING db (172.18.0.3): 56 data bytes, 64 bytes from 172.18.0.3
# ✅ Connected!
```

### Scenario 4: "My App Can't Write to Disk"

```bash
# Step 1: Check if filesystem is read-only
docker inspect myapp --format='{{.HostConfig.ReadonlyRootfs}}'
# true — read-only filesystem!

# Step 2: Check the error
docker logs myapp
# EROFS: read-only file system, open '/app/data/cache.json'

# Step 3: Fix — mount a writable volume for the data directory
docker run --read-only \
  --tmpfs /tmp \
  -v app-data:/app/data \
  myapp
```

## 32. Debugging Cheat Sheet

### Quick Reference

```bash
# ── Container Status ──
docker ps -a                                  # List all containers
docker inspect <container>                    # Full details (JSON)
docker inspect --format='{{.State.Status}}' c # Just the status

# ── Logs ──
docker logs <container>                       # All logs
docker logs -f <container>                    # Follow (live)
docker logs --tail 100 <container>            # Last 100 lines
docker logs --since 30m <container>           # Last 30 minutes
docker logs <container> 2>&1 | grep ERROR     # Search for errors

# ── Get Inside ──
docker exec -it <container> sh                # Get a shell
docker exec -it <container> bash              # Bash shell
docker exec <container> env                   # See env vars
docker exec <container> cat /etc/hosts        # See hosts file

# ── Resources ──
docker stats                                  # Live resource usage
docker stats --no-stream                      # Snapshot
docker top <container>                        # Processes

# ── Networking ──
docker port <container>                       # Port mappings
docker network ls                             # List networks
docker network inspect <network>              # Network details
docker exec <c> ping <other-container>        # Test connectivity

# ── Files ──
docker diff <container>                       # Changed files
docker cp <container>:/path ./local-path      # Copy files out
docker cp ./local-file <container>:/path      # Copy files in

# ── Cleanup ──
docker system df                              # Disk usage
docker system prune -a                        # Clean everything

# ── Events ──
docker events                                 # Live events
docker events --filter event=die              # Only crashes
docker events --filter event=oom              # Only OOM events
```

### Exit Code Reference

| Exit Code | Meaning | Common Cause |
|-----------|---------|-------------|
| `0` | Success | Normal exit |
| `1` | General error | Application error |
| `2` | Misuse of shell command | Bad script syntax |
| `126` | Command not executable | Permission issue |
| `127` | Command not found | Wrong CMD/ENTRYPOINT |
| `128+n` | Fatal signal n | Container received signal |
| `137` | SIGKILL (128+9) | OOM killed or `docker kill` |
| `139` | SIGSEGV (128+11) | Segmentation fault |
| `143` | SIGTERM (128+15) | Graceful stop (`docker stop`) |
| `255` | Exit status out of range | Error in shell script |

## 33. Summary

| Topic | Key Takeaway |
|-------|-------------|
| **How logging works** | Docker captures STDOUT/STDERR and routes to log driver |
| **docker logs** | First tool for debugging; works with json-file, local, journald |
| **Log drivers** | json-file (default), local, syslog, fluentd, awslogs, etc. |
| **Log rotation** | ALWAYS configure `max-size` and `max-file` — prevent disk full! |
| **Structured logging** | Use JSON format for easier searching/filtering |
| **Container won't start** | Check `docker logs`, `docker inspect`, try interactive shell |
| **docker inspect** | Get ALL details about a container/image/network/volume |
| **docker exec** | Run commands inside running containers |
| **docker stats** | Real-time CPU, memory, network, disk monitoring |
| **docker events** | Watch Docker activity (starts, stops, crashes, OOM) |
| **docker diff** | See what files changed inside a container |
| **Network debugging** | Use nicolaka/netshoot, check networks, DNS, ports |
| **Centralized logging** | EFK stack or Grafana Loki for production |
| **Exit codes** | 137 = OOM/killed, 143 = graceful stop, 1 = app error |

> **Golden Rule of Docker Logging:** Log to STDOUT, configure rotation, centralize for production, and ALWAYS have a way to find out what went wrong when things break. Because they will break.
