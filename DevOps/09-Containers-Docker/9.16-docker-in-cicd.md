## 1. Why Docker in CI/CD?

In modern software development, most CI/CD pipelines need to do one or more of these things:

- **Build Docker images** from your application code
- **Run tests inside containers** to get a clean, reproducible environment
- **Push Docker images** to a container registry (Docker Hub, ECR, GCR, ACR, etc.)
- **Deploy containers** to staging or production

The challenge? **Most CI/CD runners are themselves running inside containers!** So you end up needing to run Docker *inside* Docker — and that's where things get interesting.

Think of it like this: imagine you're inside a box (a container), and someone asks you to build another box (another container). How do you do that when you're already boxed in?

There are several approaches, and each has its own pros and cons. Let's go through all of them.

---

## 2. The Core Problem — Building Images Inside Containers

Here's the situation:

```
┌──────────────────────────────────────────────┐
│              Host Machine                     │
│                                               │
│   Docker Daemon running here                  │
│                                               │
│   ┌─────────────────────────────────────┐     │
│   │        CI/CD Runner Container        │    │
│   │                                      │    │
│   │   Your pipeline says:                │    │
│   │   "docker build -t myapp:v1 ."       │    │
│   │                                      │    │
│   │   But wait... there's no Docker      │    │
│   │   daemon inside THIS container!      │    │
│   │                                      │    │
│   └─────────────────────────────────────┘     │
└──────────────────────────────────────────────┘
```

The Docker CLI is just a client — it needs to talk to a **Docker daemon** to actually build images, create containers, etc. Inside a CI/CD runner container, there's typically no daemon running.

So how do we solve this? Let's look at the approaches.

---

## 3. Approach 1 — Docker Socket Mounting (DooD)

### What Is It?

**Docker Socket Mounting** (sometimes called **Docker-outside-of-Docker** or **DooD**) is the simplest approach. You mount the host machine's Docker socket (`/var/run/docker.sock`) into the CI/CD runner container.

This means the Docker CLI *inside* the container talks to the Docker daemon *on the host*. You're not running Docker inside Docker — you're using the **host's Docker** from inside a container.

### How It Works

```
┌──────────────────────────────────────────────┐
│              Host Machine                     │
│                                               │
│   Docker Daemon ◄────── /var/run/docker.sock  │
│        ▲                       │              │
│        │                       │ (mounted)    │
│   ┌────┼───────────────────────┼────────┐     │
│   │    │   CI/CD Runner        │        │     │
│   │    │                       ▼        │     │
│   │  Docker CLI ──► /var/run/docker.sock│     │
│   │                                     │     │
│   │  "docker build" uses HOST's daemon  │     │
│   └─────────────────────────────────────┘     │
└──────────────────────────────────────────────┘
```

### How to Do It

You simply mount the Docker socket when starting the CI/CD runner container:

```bash
docker run -v /var/run/docker.sock:/var/run/docker.sock \
           -v $(which docker):/usr/bin/docker \
           my-ci-runner
```

**Let's break this down:**

| Part | What It Does |
|------|-------------|
| `-v /var/run/docker.sock:/var/run/docker.sock` | Mounts the host's Docker socket into the container — this is how the container talks to the host's Docker daemon |
| `-v $(which docker):/usr/bin/docker` | Mounts the Docker binary so the container has the `docker` command available |
| `my-ci-runner` | Your CI/CD runner image |

### Example — Docker Socket Mounting in Docker Compose

```yaml
version: "3.8"
services:
  ci-runner:
    image: my-ci-runner:latest
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock
    # Now this container can run docker commands
    # that execute on the HOST's Docker daemon
```

### Pros

| Advantage | Why It Matters |
|-----------|---------------|
| **Simple setup** | Just mount one file — done! |
| **Fast** | No overhead — uses the existing daemon |
| **Shared layer cache** | The host's Docker cache is available, so builds are faster |
| **Low resource usage** | No extra daemon running |

### Cons & Security Risks

| Disadvantage | Why It's a Problem |
|-------------|-------------------|
| **MAJOR SECURITY RISK** | Anyone with access to the Docker socket has **root-level access to the host**. A malicious build could escape the container. |
| **No isolation** | Containers created by the CI job run on the host, alongside other containers |
| **Side effects** | A CI job could accidentally interfere with other containers on the host |
| **File system context issues** | When you `COPY` files in a Dockerfile, the paths are relative to the host, not the runner container. This can cause confusion. |

### ⚠️ Security Warning

Mounting the Docker socket is like giving someone the keys to your house. Here's why:

```bash
# Someone inside the container could do THIS:
docker run -v /:/host-root -it ubuntu bash

# Now they have access to the ENTIRE host filesystem!
# This is essentially root access to the host machine.
```

**Use this approach only when you trust the code running in your pipeline.**

## 4. Approach 2 — Docker-in-Docker (DinD)

### What Is It?

**Docker-in-Docker (DinD)** means running a **full Docker daemon inside a container**. The CI/CD runner gets its own isolated Docker daemon, completely separate from the host's.

### How It Works

```
┌──────────────────────────────────────────────┐
│              Host Machine                     │
│                                               │
│   Host Docker Daemon                          │
│                                               │
│   ┌─────────────────────────────────────┐     │
│   │  CI/CD Runner Container              │    │
│   │                                      │    │
│   │   ┌──────────────────────────────┐   │    │
│   │   │  Nested Docker Daemon        │   │    │
│   │   │  (its own daemon, own cache) │   │    │
│   │   └──────────────────────────────┘   │    │
│   │                                      │    │
│   │  Docker CLI ──► talks to INNER daemon│    │
│   │  "docker build" runs INSIDE here     │    │
│   └─────────────────────────────────────┘     │
└──────────────────────────────────────────────┘
```

### How to Do It

Docker provides an official DinD image: `docker:dind`

```bash
# Start a DinD container
docker run --privileged --name dind-daemon \
           -d docker:dind

# Start your CI runner that connects to the DinD daemon
docker run --link dind-daemon:docker \
           -e DOCKER_HOST=tcp://docker:2376 \
           -e DOCKER_TLS_CERTDIR=/certs \
           docker:latest sh -c "docker info"
```

**What does each part mean?**

| Part | Explanation |
|------|------------|
| `--privileged` | Gives the container full access to host devices. Required for the inner Docker daemon to work (it needs to manage cgroups, namespaces, etc.) |
| `docker:dind` | Official Docker image that includes a full Docker daemon |
| `--link dind-daemon:docker` | Links the runner container to the DinD container so it can reach the daemon |
| `DOCKER_HOST=tcp://docker:2376` | Tells the Docker CLI where to find the daemon (over TCP instead of a socket) |
| `DOCKER_TLS_CERTDIR=/certs` | Enables TLS for secure communication between the CLI and daemon |

### Example — DinD with Docker Compose

```yaml
version: "3.8"
services:
  # This runs the Docker daemon
  dind:
    image: docker:dind
    privileged: true
    environment:
      DOCKER_TLS_CERTDIR: /certs
    volumes:
      - docker-certs-ca:/certs/ca
      - docker-certs-client:/certs/client
    networks:
      - ci-network

  # This is your CI runner that uses the DinD daemon
  runner:
    image: docker:latest
    environment:
      DOCKER_HOST: tcp://dind:2376
      DOCKER_TLS_VERIFY: "1"
      DOCKER_CERT_PATH: /certs/client
    volumes:
      - docker-certs-client:/certs/client:ro
    networks:
      - ci-network
    depends_on:
      - dind

volumes:
  docker-certs-ca:
  docker-certs-client:

networks:
  ci-network:
```

### Pros

| Advantage | Why It Matters |
|-----------|---------------|
| **Full isolation** | Each CI job gets its own Docker daemon. Jobs can't interfere with each other or the host. |
| **Clean environment** | Every pipeline run starts fresh — no leftover containers or images from previous runs |
| **No host access** | The inner Docker daemon can't access the host's Docker resources |

### Cons

| Disadvantage | Why It's a Problem |
|-------------|-------------------|
| **Requires `--privileged`** | This is a security risk — privileged containers have almost full host access |
| **No layer cache** | Each DinD container starts with an empty cache, so builds are slower |
| **Storage driver issues** | Running a storage driver (like overlay2) inside a container can cause filesystem issues |
| **More resource usage** | Running a full Docker daemon takes extra CPU and memory |
| **Slower** | Starting a fresh daemon + no cache = slower builds |

### Solving the Cache Problem

You can use Docker's `--cache-from` flag or BuildKit cache mounts to mitigate the cache issue:

```bash
# Pull the previous image to use as cache
docker pull myapp:latest || true

# Build using the pulled image as cache source
docker build --cache-from myapp:latest -t myapp:v2 .
```

Or use BuildKit with registry-based cache:

```bash
docker buildx build \
  --cache-from type=registry,ref=myregistry.com/myapp:cache \
  --cache-to type=registry,ref=myregistry.com/myapp:cache \
  -t myapp:v2 .
```

## 5. Approach 3 — Using Kaniko (Rootless & Daemonless)

### What Is It?

**Kaniko** is a tool from Google that builds Docker images **without needing a Docker daemon at all**. It runs entirely in userspace and doesn't need privileged access.

This makes it the **safest approach** for building images in CI/CD, especially in multi-tenant environments like Kubernetes.

### How It Works

```
┌──────────────────────────────────────────────┐
│              Host Machine / K8s Cluster        │
│                                               │
│   ┌─────────────────────────────────────┐     │
│   │  Kaniko Container                    │    │
│   │                                      │    │
│   │  - Reads your Dockerfile             │    │
│   │  - Executes each command             │    │
│   │  - Snapshots the filesystem          │    │
│   │  - Creates layers from snapshots     │    │
│   │  - Pushes final image to registry    │    │
│   │                                      │    │
│   │  NO Docker daemon needed!            │    │
│   │  NO privileged access needed!        │    │
│   └─────────────────────────────────────┘     │
└──────────────────────────────────────────────┘
```

### How to Use Kaniko

```bash
docker run \
  -v $(pwd):/workspace \
  -v ~/.docker/config.json:/kaniko/.docker/config.json:ro \
  gcr.io/kaniko-project/executor:latest \
  --dockerfile /workspace/Dockerfile \
  --destination myregistry.com/myapp:v1 \
  --context /workspace
```

**Breaking it down:**

| Part | What It Does |
|------|-------------|
| `-v $(pwd):/workspace` | Mounts your source code into the Kaniko container |
| `-v ~/.docker/config.json:...` | Provides registry credentials so Kaniko can push the image |
| `gcr.io/kaniko-project/executor` | The Kaniko executor image |
| `--dockerfile` | Path to your Dockerfile |
| `--destination` | Where to push the built image |
| `--context` | The build context (where your code lives) |

### Kaniko in Kubernetes (GitLab CI Example)

```yaml
# .gitlab-ci.yml
build:
  stage: build
  image:
    name: gcr.io/kaniko-project/executor:debug
    entrypoint: [""]
  script:
    - mkdir -p /kaniko/.docker
    # Create Docker config with registry credentials
    - echo "{\"auths\":{\"$CI_REGISTRY\":{\"auth\":\"$(echo -n ${CI_REGISTRY_USER}:${CI_REGISTRY_PASSWORD} | base64)\"}}}" > /kaniko/.docker/config.json
    # Build and push
    - /kaniko/executor
      --context "${CI_PROJECT_DIR}"
      --dockerfile "${CI_PROJECT_DIR}/Dockerfile"
      --destination "${CI_REGISTRY_IMAGE}:${CI_COMMIT_TAG}"
```

### Pros & Cons

| Pros | Cons |
|------|------|
| No Docker daemon needed | Slower than native Docker builds |
| No privileged access | Limited caching support |
| Safe for multi-tenant clusters | Can't run `docker` commands (only builds images) |
| Works great in Kubernetes | Some Dockerfile instructions may behave differently |

## 6. Approach 4 — Using Buildah (Daemonless)

### What Is It?

**Buildah** is a tool by Red Hat for building OCI-compliant container images without a daemon. Unlike Kaniko, Buildah can also manage images locally (not just push to a registry).

### Basic Usage

```bash
# Build an image (using Dockerfile)
buildah bud -t myapp:v1 .

# Build an image (interactively, without Dockerfile)
container=$(buildah from ubuntu:22.04)
buildah run $container -- apt-get update
buildah run $container -- apt-get install -y nginx
buildah commit $container myapp:v1

# Push to registry
buildah push myapp:v1 docker://myregistry.com/myapp:v1
```

### Buildah in CI/CD

```yaml
# .gitlab-ci.yml
build:
  stage: build
  image: quay.io/buildah/stable
  variables:
    STORAGE_DRIVER: vfs  # Use vfs driver (works without privileges)
  script:
    - buildah login -u "$CI_REGISTRY_USER" -p "$CI_REGISTRY_PASSWORD" "$CI_REGISTRY"
    - buildah bud -t "$CI_REGISTRY_IMAGE:$CI_COMMIT_TAG" .
    - buildah push "$CI_REGISTRY_IMAGE:$CI_COMMIT_TAG"
```

## 7. Comparison of All Approaches

| Feature | Socket Mounting | DinD | Kaniko | Buildah |
|---------|:-:|:-:|:-:|:-:|
| **Needs Docker daemon?** | Yes (host's) | Yes (its own) | No | No |
| **Needs `--privileged`?** | No | Yes | No | No (with vfs) |
| **Security** | ⚠️ Low | ⚠️ Medium | ✅ High | ✅ High |
| **Build speed** | 🚀 Fast | 🐢 Slow | 🐢 Slow | 🏃 Medium |
| **Layer cache** | ✅ Host cache | ❌ Fresh each time | ⚠️ Registry cache | ⚠️ Limited |
| **Isolation** | ❌ None | ✅ Full | ✅ Full | ✅ Full |
| **K8s friendly?** | ⚠️ Not ideal | ⚠️ Needs privileged pod | ✅ Yes | ✅ Yes |
| **Best for** | Local CI, trusted envs | Jenkins, GitLab runners | Kubernetes CI/CD | Kubernetes CI/CD |

### Quick Decision Guide

```
Do you need to run Docker commands (not just build)?
├── YES → Socket Mounting or DinD
│   ├── Need isolation? → DinD
│   └── Need speed? → Socket Mounting
└── NO (just building images)
    ├── Running in Kubernetes? → Kaniko
    └── Need flexibility? → Buildah
```

## 8. Docker in GitHub Actions

GitHub Actions runners come with Docker **pre-installed**, so you don't need DinD or socket mounting. The runner is a full VM, not a container.

### Building and Pushing an Image

```yaml
# .github/workflows/docker-build.yml
name: Build and Push Docker Image

on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      # Step 1: Checkout the code
      - name: Checkout code
        uses: actions/checkout@v4

      # Step 2: Log in to Docker Hub
      - name: Log in to Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKERHUB_USERNAME }}
          password: ${{ secrets.DOCKERHUB_TOKEN }}

      # Step 3: Set up Docker Buildx (for advanced builds)
      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      # Step 4: Build and push the image
      - name: Build and Push
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: |
            myuser/myapp:latest
            myuser/myapp:${{ github.sha }}
          cache-from: type=gha  # GitHub Actions cache
          cache-to: type=gha,mode=max
```

**What each step does:**

| Step | Purpose |
|------|---------|
| `actions/checkout@v4` | Pulls your source code into the runner |
| `docker/login-action@v3` | Authenticates with Docker Hub (or any registry) using secrets |
| `docker/setup-buildx-action@v3` | Sets up BuildKit for advanced features like multi-platform builds and caching |
| `docker/build-push-action@v5` | Builds the Docker image and pushes it to the registry |
| `cache-from/cache-to: type=gha` | Uses GitHub Actions' built-in cache for Docker layers — makes subsequent builds much faster |

### Multi-Platform Build (ARM + AMD64)

```yaml
      - name: Build and Push Multi-Platform
        uses: docker/build-push-action@v5
        with:
          context: .
          platforms: linux/amd64,linux/arm64
          push: true
          tags: myuser/myapp:latest
```

## 9. Docker in GitLab CI/CD

### Method 1 — Using DinD Service

```yaml
# .gitlab-ci.yml
image: docker:24.0

services:
  - docker:24.0-dind  # This runs the DinD daemon as a "service"

variables:
  DOCKER_HOST: tcp://docker:2376
  DOCKER_TLS_CERTDIR: "/certs"
  DOCKER_DRIVER: overlay2

stages:
  - build
  - test
  - push

build:
  stage: build
  script:
    - docker build -t $CI_REGISTRY_IMAGE:$CI_COMMIT_SHA .

test:
  stage: test
  script:
    - docker run --rm $CI_REGISTRY_IMAGE:$CI_COMMIT_SHA npm test

push:
  stage: push
  script:
    - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $CI_REGISTRY
    - docker push $CI_REGISTRY_IMAGE:$CI_COMMIT_SHA
    - docker tag $CI_REGISTRY_IMAGE:$CI_COMMIT_SHA $CI_REGISTRY_IMAGE:latest
    - docker push $CI_REGISTRY_IMAGE:latest
  only:
    - main
```

**What's happening here:**

- `services: - docker:24.0-dind` — GitLab automatically runs a DinD container alongside your job. This is the Docker daemon.
- `DOCKER_HOST: tcp://docker:2376` — Tells the Docker CLI to connect to the DinD service.
- `$CI_REGISTRY_IMAGE` — GitLab's built-in variable that points to the project's container registry.

### Method 2 — Using Kaniko

```yaml
# .gitlab-ci.yml
build:
  stage: build
  image:
    name: gcr.io/kaniko-project/executor:v1.9.0-debug
    entrypoint: [""]
  script:
    - /kaniko/executor
      --context "${CI_PROJECT_DIR}"
      --dockerfile "${CI_PROJECT_DIR}/Dockerfile"
      --destination "${CI_REGISTRY_IMAGE}:${CI_COMMIT_SHA}"
      --destination "${CI_REGISTRY_IMAGE}:latest"
```

This is cleaner, faster (in K8s environments), and **doesn't need privileged mode**.

## 10. Docker in Jenkins

### Method 1 — Docker Pipeline Plugin (Socket Mounting)

Jenkins has a **Docker Pipeline plugin** that makes running Docker commands in pipelines easy.

```groovy
// Jenkinsfile (Declarative Pipeline)
pipeline {
    agent any

    environment {
        REGISTRY = 'myregistry.com'
        IMAGE = "${REGISTRY}/myapp"
        TAG = "${env.BUILD_NUMBER}"
    }

    stages {
        stage('Build') {
            steps {
                script {
                    // Build the Docker image
                    def app = docker.build("${IMAGE}:${TAG}")
                }
            }
        }

        stage('Test') {
            steps {
                script {
                    // Run tests inside the built image
                    docker.image("${IMAGE}:${TAG}").inside {
                        sh 'npm test'
                    }
                }
            }
        }

        stage('Push') {
            steps {
                script {
                    docker.withRegistry("https://${REGISTRY}", 'registry-credentials') {
                        def app = docker.image("${IMAGE}:${TAG}")
                        app.push()
                        app.push('latest')
                    }
                }
            }
        }
    }

    post {
        always {
            // Clean up
            sh "docker rmi ${IMAGE}:${TAG} || true"
        }
    }
}
```

**What each part does:**

| Part | Purpose |
|------|---------|
| `docker.build(...)` | Builds a Docker image from the Jenkinsfile's workspace |
| `docker.image(...).inside { }` | Runs commands *inside* a container created from the image |
| `docker.withRegistry(...)` | Authenticates with a Docker registry using Jenkins credentials |
| `app.push()` | Pushes the image to the registry |

### Method 2 — Jenkins Agent Running in Docker with DinD

```groovy
pipeline {
    agent {
        docker {
            image 'docker:24.0'
            args '-v /var/run/docker.sock:/var/run/docker.sock'
        }
    }
    stages {
        stage('Build') {
            steps {
                sh 'docker build -t myapp:${BUILD_NUMBER} .'
            }
        }
    }
}
```

## 11. Best Practices for Docker in CI/CD

### 1. Use Multi-Stage Builds

Multi-stage builds keep your final images small and your CI pipeline clean:

```dockerfile
# Stage 1: Build
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Stage 2: Production image (small!)
FROM node:20-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
CMD ["node", "dist/index.js"]
```

### 2. Tag Images Properly

Never rely on `:latest` alone. Use **meaningful, unique tags**:

```bash
# Good: Use commit SHA, build number, or semantic version
docker build -t myapp:${COMMIT_SHA} .
docker build -t myapp:v1.2.3 .
docker build -t myapp:build-${BUILD_NUMBER} .

# Bad: Only using latest
docker build -t myapp:latest .  # Which version is this?!
```

### 3. Leverage Layer Caching

```yaml
# GitHub Actions example with caching
- name: Build with cache
  uses: docker/build-push-action@v5
  with:
    context: .
    cache-from: type=gha
    cache-to: type=gha,mode=max
```

### 4. Scan Images for Vulnerabilities

Always scan your images before pushing to production:

```yaml
# In your CI pipeline
- name: Scan image
  uses: aquasecurity/trivy-action@master
  with:
    image-ref: myapp:${{ github.sha }}
    format: 'table'
    exit-code: '1'           # Fail the build if vulnerabilities found
    severity: 'CRITICAL,HIGH'
```

### 5. Use .dockerignore

Always include a `.dockerignore` to keep your build context small and your secrets safe:

```
# .dockerignore
node_modules
.git
.env
*.md
docker-compose*.yml
.github
tests
```

### 6. Pin Base Image Versions

```dockerfile
# Good: Pinned version
FROM node:20.11.0-alpine3.19

# Okay: Minor version pinned
FROM node:20-alpine

# Bad: Using latest (builds are not reproducible!)
FROM node:latest
```

### 7. Use Secrets Properly

Never put secrets in your Dockerfile or build args:

```dockerfile
# BAD — Secret is stored in image layer!
ARG DB_PASSWORD
RUN echo "password=$DB_PASSWORD" > /app/config

# GOOD — Use BuildKit secrets (not stored in layers)
RUN --mount=type=secret,id=db_password \
    cat /run/secrets/db_password > /app/config
```

Build with:
```bash
docker build --secret id=db_password,src=./db_password.txt -t myapp .
```

## 12. Common Pitfalls & Troubleshooting

### Problem 1: "Cannot connect to the Docker daemon"

```
Cannot connect to the Docker daemon at unix:///var/run/docker.sock. Is the docker daemon running?
```

**Cause:** The Docker CLI can't find a Docker daemon.

**Solutions:**
- If using DinD: Make sure `DOCKER_HOST` is set correctly (`tcp://docker:2376`)
- If using socket mounting: Check the socket is mounted (`-v /var/run/docker.sock:/var/run/docker.sock`)
- Check permissions: The user running Docker might not have access to the socket

```bash
# Fix permission issue
sudo chmod 666 /var/run/docker.sock
# OR add user to docker group
sudo usermod -aG docker $USER
```

### Problem 2: "No space left on device" in CI

**Cause:** Docker images and layers fill up the disk over time.

**Solution:** Add cleanup steps to your pipeline:

```yaml
# Clean up at the start of each build
- name: Clean Docker
  run: |
    docker system prune -af
    docker volume prune -f
```

### Problem 3: Slow builds due to no cache

**Solution:** Use registry-based caching:

```bash
# Pull previous image for cache
docker pull myregistry.com/myapp:latest || true

# Build with cache
docker build \
  --cache-from myregistry.com/myapp:latest \
  -t myregistry.com/myapp:${VERSION} \
  .
```

### Problem 4: Build context too large

```
Sending build context to Docker daemon  2.5GB
```

**Solution:** Add a `.dockerignore` file (see best practices above) and keep your build context minimal.

## 13. Summary

| Concept | Key Takeaway |
|---------|-------------|
| **Docker Socket Mounting** | Fast & simple, but a security risk — the container gets full host Docker access |
| **Docker-in-Docker (DinD)** | Full isolation with its own daemon, but needs `--privileged` and has no cache |
| **Kaniko** | Daemonless, rootless image builder — safest for Kubernetes CI/CD |
| **Buildah** | Red Hat's daemonless alternative — flexible, works without privileges |
| **GitHub Actions** | Docker pre-installed, use `docker/build-push-action` for best experience |
| **GitLab CI** | Use DinD service or Kaniko depending on your security requirements |
| **Jenkins** | Docker Pipeline plugin + socket mounting is the most common setup |
| **Best Practices** | Multi-stage builds, proper tagging, caching, image scanning, and secrets management |

> **Rule of Thumb:** If you're in Kubernetes → use **Kaniko**. If you're on dedicated runners and need speed → use **socket mounting**. If you need isolation → use **DinD**. Always scan your images and never store secrets in layers!
