## 1. What is a Container Runtime?

A **container runtime** is the software that actually **creates and runs containers**. When you type `docker run nginx`, there's a whole chain of software involved in making that container come to life. The "runtime" is the piece that does the heavy lifting.

But here's the thing — the term "container runtime" is a bit confusing because it can mean different things depending on who you ask:

| Layer | What It Does | Examples |
|-------|-------------|---------|
| **High-level runtime** | Manages the container lifecycle — pulls images, manages storage, creates network namespaces, then hands off to a low-level runtime | containerd, CRI-O |
| **Low-level runtime** | Actually creates the container — sets up Linux namespaces, cgroups, mounts filesystems, and starts the process | runc, crun, gVisor, Kata |

Think of it like ordering food at a restaurant:
- **High-level runtime** = The waiter (takes your order, manages the workflow, serves the food)
- **Low-level runtime** = The chef (actually makes the food)

## 3. OCI — Open Container Initiative

Before we dive into runtimes, you need to know about the **OCI** (Open Container Initiative). It's a set of **open standards** that define:

| Standard | What It Defines |
|----------|----------------|
| **OCI Image Spec** | How container images are built and structured (layers, manifests, configs) |
| **OCI Runtime Spec** | How a container should be created and run (filesystem layout, lifecycle) |
| **OCI Distribution Spec** | How images are pushed/pulled from registries |

### Why Does OCI Matter?

Before OCI, Docker had its own proprietary way of doing things. If you built an image with Docker, it might not work with other tools. OCI standardized everything so that:

- Images built with Docker work with Podman, Buildah, containerd, etc.
- Any OCI-compliant runtime can run any OCI-compliant image
- You're not locked into a single vendor

Think of OCI like **USB standards** — any USB device works with any USB port because they all follow the same standard. Similarly, any OCI image works with any OCI runtime.

### OCI Runtime Spec — config.json

When a high-level runtime (like containerd) asks a low-level runtime (like runc) to create a container, it gives it a `config.json` file. This file describes everything about the container:

```json
{
    "ociVersion": "1.0.2",
    "process": {
        "terminal": true,
        "user": { "uid": 0, "gid": 0 },
        "args": ["sh"],
        "env": [
            "PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin",
            "TERM=xterm"
        ],
        "cwd": "/"
    },
    "root": {
        "path": "rootfs",
        "readonly": false
    },
    "hostname": "my-container",
    "linux": {
        "namespaces": [
            { "type": "pid" },
            { "type": "network" },
            { "type": "ipc" },
            { "type": "uts" },
            { "type": "mount" }
        ],
        "resources": {
            "memory": { "limit": 268435456 },
            "cpu": { "shares": 1024 }
        }
    }
}
```

**What this config says:**

| Section | Meaning |
|---------|---------|
| `process.args: ["sh"]` | Run the `sh` shell as the main process |
| `process.user.uid: 0` | Run as root (UID 0) |
| `root.path: "rootfs"` | The container's root filesystem is in the `rootfs` directory |
| `linux.namespaces` | Create separate PID, network, IPC, UTS, and mount namespaces |
| `linux.resources.memory.limit` | Limit memory to ~256 MB |
| `linux.resources.cpu.shares` | CPU share weight of 1024 |

## 4. Low-Level Runtimes

### 4.1 runc

**runc** is the **reference implementation** of the OCI runtime specification. It was originally extracted from Docker and donated to the OCI. Almost every container you've ever run used runc under the hood.

#### What runc Does

runc is a simple CLI tool that creates and runs containers according to the OCI spec. That's it. It doesn't:
- Pull images
- Manage storage
- Handle networking at a high level
- Provide an API

It just takes a bundle (filesystem + config.json) and creates a container.

#### Using runc Directly

Most people never use runc directly, but let's see how it works to understand what's happening behind the scenes:

```bash
# Step 1: Create a directory for your container bundle
mkdir -p mycontainer/rootfs

# Step 2: Export an image's filesystem into the rootfs directory
docker export $(docker create busybox) | tar -C mycontainer/rootfs -xf -

# Step 3: Generate the OCI config.json
cd mycontainer
runc spec

# This creates a config.json with default settings

# Step 4: Run the container!
sudo runc run my-container-id

# You're now inside a container created directly by runc!
# Type 'exit' to leave
```

**What just happened:**

1. We created a root filesystem by extracting a BusyBox image
2. `runc spec` generated a standard OCI config.json
3. `runc run` read the config.json, set up namespaces, cgroups, and filesystem isolation, then started the process

#### runc Lifecycle Commands

```bash
# Create a container (but don't start it)
sudo runc create my-container

# Start a created container
sudo runc start my-container

# List running containers
sudo runc list

# Get container state
sudo runc state my-container

# Execute a command in a running container
sudo runc exec my-container ls /

# Pause a container (freeze all processes)
sudo runc pause my-container

# Resume a paused container
sudo runc resume my-container

# Send a signal to the container's init process
sudo runc kill my-container SIGTERM

# Delete a stopped container
sudo runc delete my-container
```

### 4.2 crun

**crun** is an alternative OCI runtime written in **C** (runc is written in Go). It's **faster and lighter** than runc.

| Feature | runc | crun |
|---------|------|------|
| Language | Go | C |
| Startup time | ~50-100ms | ~20-50ms |
| Memory usage | Higher | Lower |
| OCI compliant | Yes | Yes |
| Default in | Docker, containerd | Podman (RHEL 9+) |

```bash
# Check if crun is installed
crun --version

# Use crun instead of runc
sudo crun run my-container
```

**crun** is the default runtime in newer versions of **Podman** and **RHEL/Fedora**.

## 5. High-Level Runtimes

### 5.1 containerd

**containerd** (pronounced "container-dee") is the most widely used high-level container runtime. It was originally a component of Docker, then extracted into its own project and donated to the CNCF (Cloud Native Computing Foundation).

#### What containerd Does

containerd manages the **complete container lifecycle**:

```
┌───────────────────────────────────────────┐
│              containerd                    │
│                                           │
│  ┌─────────────────────────────────────┐  │
│  │  Image Management                   │  │
│  │  - Pull/push images from registries │  │
│  │  - Store and manage image layers    │  │
│  │  - Image format conversion          │  │
│  └─────────────────────────────────────┘  │
│                                           │
│  ┌─────────────────────────────────────┐  │
│  │  Container Management               │  │
│  │  - Create/start/stop containers     │  │
│  │  - Manage container lifecycle       │  │
│  │  - Attach to running containers     │  │
│  └─────────────────────────────────────┘  │
│                                           │
│  ┌─────────────────────────────────────┐  │
│  │  Storage (Snapshotter)              │  │
│  │  - Manage filesystem snapshots      │  │
│  │  - overlay2, btrfs, zfs, etc.       │  │
│  └─────────────────────────────────────┘  │
│                                           │
│  ┌─────────────────────────────────────┐  │
│  │  Networking (via CNI plugins)       │  │
│  │  - Set up container networking      │  │
│  └─────────────────────────────────────┘  │
│                                           │
│  ┌─────────────────────────────────────┐  │
│  │  Calls low-level runtime (runc)     │  │
│  │  to actually create containers      │  │
│  └─────────────────────────────────────┘  │
└───────────────────────────────────────────┘
```

#### containerd Architecture

```
                    gRPC API
                       │
                       ▼
              ┌─────────────────┐
              │   containerd     │
              │   daemon         │
              └────────┬────────┘
                       │
        ┌──────────────┼──────────────┐
        │              │              │
   ┌────▼────┐   ┌────▼────┐   ┌────▼────┐
   │ Content  │   │Snapshot │   │  Task    │
   │ Store    │   │ Manager │   │ Manager  │
   │(images)  │   │(layers) │   │(running) │
   └─────────┘   └─────────┘   └────┬─────┘
                                     │
                               ┌─────▼─────┐
                               │   shim     │
                               │(containerd-│
                               │ shim-runc) │
                               └─────┬─────┘
                                     │
                               ┌─────▼─────┐
                               │   runc     │
                               │ (creates   │
                               │ container) │
                               └───────────┘
```

**What's a "shim"?**

The **containerd-shim** is a small process that sits between containerd and runc. It exists so that:
- containerd can be restarted without killing running containers
- Container stdout/stderr can be captured even if containerd is down
- The container's exit status is kept after it stops

#### Using containerd Directly (ctr & nerdctl)

containerd comes with a basic CLI tool called `ctr`, but there's also a Docker-compatible CLI called **nerdctl**:

```bash
# Using ctr (containerd's built-in CLI — basic)
# Pull an image
sudo ctr images pull docker.io/library/nginx:alpine

# List images
sudo ctr images ls

# Run a container
sudo ctr run --rm docker.io/library/nginx:alpine my-nginx

# List running containers
sudo ctr containers ls

# List tasks (running processes)
sudo ctr tasks ls
```

```bash
# Using nerdctl (Docker-compatible CLI for containerd)
# Install nerdctl: https://github.com/containerd/nerdctl

# These commands look exactly like Docker!
nerdctl run -d --name web -p 8080:80 nginx:alpine
nerdctl ps
nerdctl logs web
nerdctl stop web
nerdctl rm web

# Even compose works!
nerdctl compose up -d
```

#### containerd Configuration

containerd is configured via `/etc/containerd/config.toml`:

```toml
# /etc/containerd/config.toml

version = 2

[plugins]
  [plugins."io.containerd.grpc.v1.cri"]
    # Sandbox (pause) image used by Kubernetes
    sandbox_image = "registry.k8s.io/pause:3.9"

    [plugins."io.containerd.grpc.v1.cri".containerd]
      # Default runtime
      default_runtime_name = "runc"

      [plugins."io.containerd.grpc.v1.cri".containerd.runtimes]
        [plugins."io.containerd.grpc.v1.cri".containerd.runtimes.runc]
          runtime_type = "io.containerd.runc.v2"

          [plugins."io.containerd.grpc.v1.cri".containerd.runtimes.runc.options]
            SystemdCgroup = true
```

**What these settings mean:**

| Setting | Purpose |
|---------|---------|
| `sandbox_image` | The "pause" container image Kubernetes uses for each pod |
| `default_runtime_name` | Which OCI runtime to use by default (runc) |
| `SystemdCgroup = true` | Use systemd for cgroup management (recommended for Kubernetes) |

### 5.2 CRI-O

**CRI-O** (pronounced "cry-oh") is a **lightweight container runtime specifically built for Kubernetes**. While containerd is a general-purpose runtime that Docker and Kubernetes both use, CRI-O is laser-focused on Kubernetes.

#### CRI-O's Philosophy

> "CRI-O: The OCI-based implementation of the Kubernetes CRI (Container Runtime Interface)"

The name says it all:
- **CRI** = Container Runtime Interface (Kubernetes' standard interface)
- **O** = OCI (Open Container Initiative)

CRI-O implements the CRI so Kubernetes can use any OCI-compliant runtime.

#### What CRI-O Does (and Doesn't Do)

| Does | Doesn't |
|------|---------|
| Pull OCI images from registries | Provide a general-purpose CLI (no `docker run` equivalent) |
| Manage container storage | Support Docker Compose |
| Set up container networking (via CNI) | Aim to be a general-purpose runtime |
| Create containers via runc (or other OCI runtime) | Provide image build capabilities |
| Serve the CRI gRPC API for kubelet | Work outside of Kubernetes |

#### CRI-O Architecture

```
┌──────────────────────────────────────────┐
│  Kubernetes                               │
│  ┌──────────┐                            │
│  │ kubelet  │                            │
│  └────┬─────┘                            │
│       │  CRI (gRPC)                      │
│       ▼                                  │
│  ┌──────────────────────────────┐        │
│  │        CRI-O                  │        │
│  │                               │        │
│  │  ┌────────────────────────┐   │        │
│  │  │ Image Management       │   │        │
│  │  │ (pull from registries) │   │        │
│  │  └────────────────────────┘   │        │
│  │                               │        │
│  │  ┌────────────────────────┐   │        │
│  │  │ Storage Management     │   │        │
│  │  │ (container layers)     │   │        │
│  │  └────────────────────────┘   │        │
│  │                               │        │
│  │  ┌────────────────────────┐   │        │
│  │  │ Networking (CNI)       │   │        │
│  │  │ (pod networking)       │   │        │
│  │  └────────────────────────┘   │        │
│  │                               │        │
│  │  ┌────────────────────────┐   │        │
│  │  │ Container Runtime      │   │        │
│  │  │ (calls runc/crun)      │   │        │
│  │  └────────────────────────┘   │        │
│  └──────────────────────────────┘        │
└──────────────────────────────────────────┘
```

#### CRI-O vs containerd

| Aspect | containerd | CRI-O |
|--------|-----------|-------|
| **Scope** | General-purpose | Kubernetes-only |
| **Used by** | Docker, Kubernetes, Podman (indirectly) | Kubernetes only |
| **CLI tools** | ctr, nerdctl, crictl | crictl only |
| **Image building** | No (but nerdctl + BuildKit can) | No |
| **Docker compatibility** | Partial (via nerdctl) | None |
| **Kubernetes versions** | Matches K8s versions loosely | Matches K8s versions exactly (1:1) |
| **Default in** | Docker Desktop, EKS, GKE, AKS | OpenShift, some RHEL setups |
| **Size / Footprint** | Medium | Smaller |
| **Stability model** | Independent releases | Locked to Kubernetes releases |

## 6. How Docker Uses These Runtimes

When you run `docker run nginx`, here's what actually happens:

```
You type: docker run nginx
         │
         ▼
┌─────────────────┐
│   Docker CLI     │  ← Parses your command
│   (docker)       │
└────────┬────────┘
         │ REST API call
         ▼
┌─────────────────┐
│   Docker Daemon  │  ← dockerd: manages everything Docker-specific
│   (dockerd)      │     (volumes, networks, images, build, etc.)
└────────┬────────┘
         │ gRPC call
         ▼
┌─────────────────┐
│   containerd     │  ← Actually manages the container lifecycle
│                  │     (pulls image, creates snapshot, calls shim)
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  containerd-shim │  ← Keeps container alive even if containerd restarts
│                  │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│     runc         │  ← Sets up namespaces, cgroups, starts the process
│                  │     Then exits! (shim takes over)
└─────────────────┘
         │
         ▼
┌─────────────────┐
│   Your Container │  ← nginx is running!
│   (nginx)        │
└─────────────────┘
```

**Key insight:** Docker is actually just a friendly wrapper around containerd, which is a wrapper around runc. When Docker starts a container:

1. **Docker CLI** sends the request to **dockerd** (the Docker daemon)
2. **dockerd** talks to **containerd** via gRPC
3. **containerd** pulls the image, prepares the filesystem, creates a **shim**
4. The **shim** calls **runc** to create the container
5. **runc** sets up Linux namespaces and cgroups, starts the process, and **exits**
6. The **shim** becomes the parent of the container process

## 7. How Kubernetes Uses These Runtimes

Kubernetes doesn't use Docker anymore (since v1.24). It talks directly to containerd or CRI-O via the **CRI (Container Runtime Interface)**.

### Before Kubernetes 1.24 (Docker was supported)

```
kubelet → dockershim → dockerd → containerd → runc
          (removed!)
```

The problem? Kubernetes had to maintain a special adapter called **dockershim** to talk to Docker. This was unnecessary overhead since Docker itself just called containerd.

### After Kubernetes 1.24 (Docker removed, containerd/CRI-O direct)

```
Option A:  kubelet → CRI → containerd → runc
Option B:  kubelet → CRI → CRI-O → runc
```

Much simpler! Kubernetes talks directly to the container runtime.

> **Important:** This change did NOT break Docker images! Docker images are OCI-compliant, so they work with any OCI runtime. You just can't use the `docker` CLI on Kubernetes nodes to manage Kubernetes containers anymore.

### Using crictl (CRI Client)

On Kubernetes nodes, you use `crictl` to interact with the container runtime:

```bash
# List pods
crictl pods

# List containers
crictl ps

# List images
crictl images

# Pull an image
crictl pull nginx:alpine

# Get container logs
crictl logs <container-id>

# Execute a command in a container
crictl exec -it <container-id> sh

# Get runtime info
crictl info
```

## 8. Sandboxed / Hardened Runtimes

Regular containers (runc) share the **host's Linux kernel**. This means a kernel vulnerability could allow a container to escape and compromise the host. For high-security environments, there are **sandboxed runtimes** that add extra isolation.

```
Regular Container (runc)          Sandboxed Container (gVisor/Kata)
┌──────────────────────┐         ┌──────────────────────────────┐
│  Application         │         │  Application                 │
│  ┌───────────────┐   │         │  ┌───────────────┐           │
│  │ Container      │   │         │  │ Container      │          │
│  │ Process(es)    │   │         │  │ Process(es)    │          │
│  └───────┬───────┘   │         │  └───────┬───────┘           │
│          │            │         │          │                    │
│  Shares host kernel!  │         │  ┌───────▼───────┐           │
│          │            │         │  │ Sandbox Layer  │           │
│          ▼            │         │  │ (gVisor kernel │           │
│  ┌───────────────┐   │         │  │  or micro-VM)  │           │
│  │  Host Kernel   │   │         │  └───────┬───────┘           │
│  └───────────────┘   │         │          │                    │
└──────────────────────┘         │  ┌───────▼───────┐           │
                                  │  │  Host Kernel   │          │
                                  │  └───────────────┘           │
                                  └──────────────────────────────┘
```

### 8.1 gVisor (runsc)

**gVisor** is Google's open-source sandboxed container runtime. It provides an additional layer of isolation by implementing a **user-space kernel** that intercepts and handles system calls from the container.

#### How gVisor Works

Instead of letting container processes talk directly to the host kernel, gVisor puts a **Sentry** (a user-space kernel written in Go) between the container and the host:

```
┌─────────────────────────────────────────┐
│  Container                               │
│  ┌───────────────────────────────────┐   │
│  │  Application                       │  │
│  │  (nginx, python, etc.)             │  │
│  └───────────┬───────────────────────┘   │
│              │ System calls (open, read, │
│              │ write, socket, etc.)      │
│              ▼                           │
│  ┌───────────────────────────────────┐   │
│  │  Sentry (user-space kernel)        │  │
│  │                                    │  │
│  │  - Intercepts ALL system calls     │  │
│  │  - Implements ~200+ Linux syscalls │  │
│  │  - Written in Go (memory-safe)     │  │
│  │  - Runs in user space, not kernel  │  │
│  └───────────┬───────────────────────┘   │
│              │ Limited, filtered         │
│              │ host syscalls             │
│              ▼                           │
│  ┌───────────────────────────────────┐   │
│  │  Gofer (file system proxy)         │  │
│  │  - Handles file I/O               │  │
│  │  - Separate process for isolation  │  │
│  └───────────────────────────────────┘   │
└───────────────────┬─────────────────────┘
                    │
            ┌───────▼───────┐
            │  Host Kernel   │
            │  (minimal      │
            │   exposure)    │
            └───────────────┘
```

#### Key Components

| Component | Role |
|-----------|------|
| **Sentry** | The user-space kernel. Intercepts system calls from the container and implements them. Only makes a minimal set of host syscalls. |
| **Gofer** | A file system proxy. Handles all file operations for the container, running as a separate process. |
| **runsc** | The OCI runtime binary (like runc, but runs containers in gVisor's sandbox). |

#### Using gVisor

```bash
# Install gVisor (runsc)
# Download from https://gvisor.dev/docs/user_guide/install/

# Configure Docker to use gVisor
# Add to /etc/docker/daemon.json:
{
    "runtimes": {
        "runsc": {
            "path": "/usr/local/bin/runsc"
        }
    }
}

# Restart Docker
sudo systemctl restart docker

# Run a container with gVisor
docker run --runtime=runsc -d --name secure-nginx nginx

# Verify it's using gVisor
docker inspect secure-nginx | grep -i runtime
# "Runtime": "runsc"

# Inside the container, you can see it's NOT the real kernel
docker exec secure-nginx dmesg
# You'll see gVisor's kernel messages, not the host's
docker exec secure-nginx uname -a
# Shows gVisor's kernel version
```

#### gVisor Pros & Cons

| Pros | Cons |
|------|------|
| Strong isolation — container never touches host kernel directly | Performance overhead (10-30% for I/O-heavy workloads) |
| Lightweight — no VM overhead | Not all syscalls are implemented (~200 of ~400+) |
| Compatible with existing container images | Some applications may not work (special syscalls) |
| Fast startup (no VM boot) | Debugging can be harder |
| Written in Go (memory-safe language) | Networking can be slower |
| Used at scale by Google (GKE Sandbox) | |

#### When to Use gVisor

- Running **untrusted code** (CI/CD builds, user-submitted code)
- **Multi-tenant environments** where containers from different users run on the same host
- **Compliance requirements** that mandate additional isolation
- Google GKE's "Sandbox" feature uses gVisor under the hood

### 8.2 Kata Containers

**Kata Containers** takes a completely different approach — it runs each container inside a **lightweight virtual machine (micro-VM)**. This gives you VM-level isolation with container-like speed.

#### How Kata Containers Works

```
┌──────────────────────────────────────────────┐
│  Host Machine                                 │
│                                               │
│  ┌─────────────────────────────────────┐      │
│  │        Lightweight VM (micro-VM)     │     │
│  │  ┌──────────────────────────────┐   │      │
│  │  │  Guest Kernel (stripped down) │   │     │
│  │  │  (minimal Linux kernel)       │   │     │
│  │  └──────────┬───────────────────┘   │      │
│  │             │                        │     │
│  │  ┌──────────▼───────────────────┐   │      │
│  │  │  Container Process            │   │     │
│  │  │  (nginx, python, etc.)        │   │     │
│  │  └──────────────────────────────┘   │      │
│  │                                     │      │
│  │  Hardware-level isolation           │      │
│  │  (separate kernel, memory, devices) │      │
│  └──────────────────┬──────────────────┘      │
│                     │                          │
│           ┌─────────▼──────────┐               │
│           │  Hypervisor        │               │
│           │  (QEMU/Cloud       │               │
│           │   Hypervisor/      │               │
│           │   Firecracker)     │               │
│           └────────────────────┘               │
│                                               │
│  Host Kernel (KVM)                            │
└──────────────────────────────────────────────┘
```

#### Key Concepts

| Component | What It Is |
|-----------|-----------|
| **Micro-VM** | A tiny, fast-booting virtual machine. Boots in ~100ms instead of the usual 30+ seconds for a regular VM. |
| **Guest Kernel** | A minimal Linux kernel running inside the VM. The container processes use THIS kernel, not the host's. |
| **Hypervisor** | The software that creates VMs. Kata supports QEMU, Cloud Hypervisor, and Firecracker. |
| **kata-runtime** | The OCI runtime binary (replaces runc). Creates a micro-VM for each container/pod. |

#### Using Kata Containers

```bash
# Install Kata Containers
# Follow: https://katacontainers.io/docs/

# Configure Docker to use Kata
# Add to /etc/docker/daemon.json:
{
    "runtimes": {
        "kata": {
            "path": "/usr/bin/kata-runtime"
        }
    }
}

# Restart Docker
sudo systemctl restart docker

# Run a container with Kata
docker run --runtime=kata -d --name secure-web nginx

# Verify
docker inspect secure-web | grep -i runtime
# "Runtime": "kata"

# Inside the container, you'll see a DIFFERENT kernel
docker exec secure-web uname -r
# Shows the guest kernel version (different from host!)
```

#### Kata with Firecracker

**Firecracker** is Amazon's micro-VM technology (powers AWS Lambda and Fargate). It's the fastest hypervisor for Kata:

```bash
# Kata can use Firecracker as the hypervisor
# Configured in /etc/kata-containers/configuration.toml

[hypervisor.firecracker]
path = "/usr/bin/firecracker"
kernel = "/usr/share/kata-containers/vmlinux.container"
```

#### Kata Containers Pros & Cons

| Pros | Cons |
|------|------|
| **VM-level isolation** — truly separate kernel | Slower startup (~100-500ms vs ~50ms for runc) |
| **Full syscall compatibility** (real kernel) | Higher memory overhead (~20-40 MB per container) |
| Protects against kernel exploits | Requires hardware virtualization (KVM) |
| Works with ALL applications (no syscall limitations) | More complex setup |
| Great for compliance (PCI-DSS, HIPAA) | Not available on all cloud providers |

#### When to Use Kata Containers

- **Strict security requirements** (financial, healthcare, government)
- Running **untrusted workloads** that need full Linux compatibility
- When gVisor's syscall limitations are a problem
- **Compliance mandates** require VM-level isolation
- Running containers from **unknown sources**

## 9. Runtime Comparison Table

| Feature | runc | crun | gVisor | Kata Containers |
|---------|:----:|:----:|:------:|:---------------:|
| **Type** | Standard OCI | Standard OCI | Sandboxed | VM-based |
| **Isolation** | Namespaces + cgroups | Namespaces + cgroups | User-space kernel | Micro-VM |
| **Security** | Standard | Standard | High | Very High |
| **Startup time** | ~50ms | ~20ms | ~100ms | ~200-500ms |
| **Memory overhead** | Minimal | Minimal | Low | ~20-40 MB |
| **Performance** | Native | Native | 10-30% overhead | 5-15% overhead |
| **Syscall compatibility** | Full | Full | Partial (~200) | Full |
| **Needs HW virtualization** | No | No | No | Yes (KVM) |
| **Language** | Go | C | Go | Go + Rust |
| **Default in** | Docker, containerd | Podman (RHEL 9+) | GKE Sandbox | OpenStack Kata |
| **Best for** | General use | Performance | Multi-tenant, untrusted | High-security, compliance |

### Decision Tree

```
What's your security requirement?
│
├── Standard (trusted code, single-tenant)
│   └── Use runc (or crun for slightly better performance)
│
├── Enhanced (multi-tenant, running untrusted code)
│   └── Can your app handle limited syscalls?
│       ├── YES → gVisor (lighter, faster)
│       └── NO → Kata Containers (full compatibility)
│
└── Maximum (compliance, financial, healthcare)
    └── Kata Containers (VM-level isolation)
```

## 10. CRI — Container Runtime Interface

The **CRI (Container Runtime Interface)** is a Kubernetes API that defines how the kubelet communicates with container runtimes. It's a **plugin interface** — any runtime that implements CRI can be used with Kubernetes.

### How CRI Works

```
┌─────────────────────────────────────────────────┐
│  Kubernetes Node                                 │
│                                                  │
│  ┌──────────┐                                    │
│  │ kubelet   │                                   │
│  └─────┬────┘                                    │
│        │ CRI gRPC API                            │
│        │                                         │
│        │  Two services:                          │
│        │  1. RuntimeService (container lifecycle)│
│        │  2. ImageService (image management)     │
│        │                                         │
│        ▼                                         │
│  ┌──────────────────────┐                        │
│  │  CRI Implementation   │                       │
│  │  (containerd / CRI-O) │                       │
│  └──────────────────────┘                        │
└─────────────────────────────────────────────────┘
```

### CRI API — Key Operations

| Service | Operations | What They Do |
|---------|-----------|-------------|
| **RuntimeService** | `RunPodSandbox` | Create the pod's network namespace |
| | `StopPodSandbox` | Stop the pod sandbox |
| | `RemovePodSandbox` | Remove the pod sandbox |
| | `CreateContainer` | Create a container in a pod |
| | `StartContainer` | Start a created container |
| | `StopContainer` | Stop a running container |
| | `RemoveContainer` | Remove a stopped container |
| | `ListContainers` | List all containers |
| | `ExecSync` | Execute a command in a container |
| **ImageService** | `PullImage` | Pull an image from a registry |
| | `ListImages` | List all images |
| | `RemoveImage` | Remove an image |
| | `ImageStatus` | Get image info |

## 11. Switching Container Runtimes in Kubernetes

### Switching to containerd

Most Kubernetes distributions already use containerd. If you need to configure it:

```bash
# 1. Install containerd
sudo apt-get update
sudo apt-get install -y containerd

# 2. Generate default config
sudo mkdir -p /etc/containerd
containerd config default | sudo tee /etc/containerd/config.toml

# 3. Enable SystemdCgroup (important for Kubernetes!)
sudo sed -i 's/SystemdCgroup = false/SystemdCgroup = true/' /etc/containerd/config.toml

# 4. Restart containerd
sudo systemctl restart containerd

# 5. Configure kubelet to use containerd
# Edit /var/lib/kubelet/kubeadm-flags.env or kubelet config:
# --container-runtime-endpoint=unix:///run/containerd/containerd.sock

# 6. Restart kubelet
sudo systemctl restart kubelet
```

### Switching to CRI-O

```bash
# 1. Install CRI-O (example for Ubuntu)
# Add the CRI-O repository and install
sudo apt-get update
sudo apt-get install -y cri-o cri-o-runc

# 2. Start CRI-O
sudo systemctl enable crio
sudo systemctl start crio

# 3. Configure kubelet
# --container-runtime-endpoint=unix:///var/run/crio/crio.sock

# 4. Restart kubelet
sudo systemctl restart kubelet
```

### Verifying the Runtime

```bash
# Check which runtime Kubernetes nodes are using
kubectl get nodes -o wide

# Output:
# NAME     STATUS   VERSION   CONTAINER-RUNTIME
# node1    Ready    v1.28.0   containerd://1.7.2
# node2    Ready    v1.28.0   cri-o://1.28.0

# Using crictl
crictl info | grep -i runtime
```

## 12. RuntimeClass in Kubernetes

**RuntimeClass** is a Kubernetes resource that lets you run **different pods with different runtimes** on the same cluster. For example, run most pods with runc but security-sensitive pods with gVisor or Kata.

### Setting Up RuntimeClass

**Step 1: Configure containerd to support multiple runtimes**

```toml
# /etc/containerd/config.toml

# Default runtime: runc
[plugins."io.containerd.grpc.v1.cri".containerd.runtimes.runc]
  runtime_type = "io.containerd.runc.v2"

# gVisor runtime
[plugins."io.containerd.grpc.v1.cri".containerd.runtimes.runsc]
  runtime_type = "io.containerd.runsc.v1"

# Kata Containers runtime
[plugins."io.containerd.grpc.v1.cri".containerd.runtimes.kata]
  runtime_type = "io.containerd.kata.v2"
```

**Step 2: Create RuntimeClass resources in Kubernetes**

```yaml
# runtimeclass-gvisor.yaml
apiVersion: node.k8s.io/v1
kind: RuntimeClass
metadata:
  name: gvisor
handler: runsc  # Matches the name in containerd config
---
# runtimeclass-kata.yaml
apiVersion: node.k8s.io/v1
kind: RuntimeClass
metadata:
  name: kata
handler: kata  # Matches the name in containerd config
```

```bash
kubectl apply -f runtimeclass-gvisor.yaml
kubectl apply -f runtimeclass-kata.yaml
```

**Step 3: Use RuntimeClass in your Pods**

```yaml
# pod-with-gvisor.yaml
apiVersion: v1
kind: Pod
metadata:
  name: secure-pod
spec:
  runtimeClassName: gvisor  # Use gVisor for this pod!
  containers:
    - name: app
      image: nginx:alpine
---
# pod-with-kata.yaml
apiVersion: v1
kind: Pod
metadata:
  name: ultra-secure-pod
spec:
  runtimeClassName: kata  # Use Kata for this pod!
  containers:
    - name: app
      image: nginx:alpine
---
# Regular pod (uses default runc)
apiVersion: v1
kind: Pod
metadata:
  name: normal-pod
spec:
  # No runtimeClassName — uses default (runc)
  containers:
    - name: app
      image: nginx:alpine
```

### Real-World Example: Mixed Runtimes

```yaml
# Trusted internal service → runc (fast, no overhead)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: internal-api
spec:
  replicas: 3
  template:
    spec:
      # No runtimeClassName → default runc
      containers:
        - name: api
          image: company/internal-api:v1
---
# Running user-submitted code → gVisor (sandboxed)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: code-runner
spec:
  replicas: 5
  template:
    spec:
      runtimeClassName: gvisor  # Sandboxed!
      containers:
        - name: runner
          image: company/code-runner:v1
---
# Financial transaction service → Kata (VM-level isolation)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: payment-service
spec:
  replicas: 2
  template:
    spec:
      runtimeClassName: kata  # Maximum isolation!
      containers:
        - name: payment
          image: company/payment:v1
```

## 13. How to Check Which Runtime You're Using

### On a Docker Host

```bash
# Check Docker's default runtime
docker info | grep -i runtime
# Default Runtime: runc
# Runtimes: io.containerd.runc.v2 runc

# Check which runtime a specific container is using
docker inspect <container-id> | grep -i runtime
# "Runtime": "runc"

# Check containerd version
containerd --version
# containerd containerd.io 1.7.2 ...
```

### On a Kubernetes Node

```bash
# Check which runtime the node is using
kubectl get nodes -o wide
# Look at the CONTAINER-RUNTIME column

# Detailed node info
kubectl describe node <node-name> | grep -i runtime
# Container Runtime Version:  containerd://1.7.2

# Using crictl
crictl version
# RuntimeName:  containerd
# RuntimeVersion:  1.7.2

# Check runtime for a specific pod
kubectl get pod <pod-name> -o jsonpath='{.spec.runtimeClassName}'
```

## 14. Summary

| Concept | Key Takeaway |
|---------|-------------|
| **Container Runtime** | The software that creates and runs containers. Two layers: high-level (containerd/CRI-O) and low-level (runc/crun/gVisor/Kata) |
| **OCI** | Open standards for container images, runtimes, and distribution. Ensures interoperability. |
| **runc** | The default, reference OCI runtime. Uses Linux namespaces + cgroups. Fast, lightweight, standard security. |
| **crun** | A faster, lighter alternative to runc, written in C. Default in newer Podman. |
| **containerd** | The most popular high-level runtime. Used by Docker and Kubernetes. Manages images, storage, and the full container lifecycle. |
| **CRI-O** | A Kubernetes-only high-level runtime. Lighter than containerd, laser-focused on the CRI API. Default in OpenShift. |
| **gVisor** | Google's sandboxed runtime. Implements a user-space kernel (Sentry) for enhanced isolation. Great for multi-tenant and untrusted workloads. |
| **Kata Containers** | VM-based runtime. Runs each container in a micro-VM for hardware-level isolation. Best for maximum security/compliance. |
| **CRI** | Kubernetes' Container Runtime Interface — the standard API that kubelet uses to talk to container runtimes. |
| **RuntimeClass** | A Kubernetes resource that lets you run different pods with different runtimes (runc for normal, gVisor for untrusted, Kata for high-security). |

### The Big Picture — One Last Time

```
You (developer)
    ↓
Docker CLI / kubectl
    ↓
Docker Daemon / kubelet
    ↓
High-level runtime (containerd / CRI-O)
  • Pulls images
  • Manages storage
  • Sets up networking
    ↓
Low-level runtime (runc / crun / gVisor / Kata)
  • Creates namespaces & cgroups (runc)
  • OR intercepts syscalls in user-space (gVisor)
  • OR creates a micro-VM (Kata)
    ↓
Your container is running! 🎉
```

> **Remember:** No matter which runtime you use, if it's OCI-compliant, your Docker images will work. The runtime is like the engine in a car — you can swap engines, but the car still drives the same roads (OCI images).
