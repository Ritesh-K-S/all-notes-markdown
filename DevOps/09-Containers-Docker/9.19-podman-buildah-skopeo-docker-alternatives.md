## 1. Why Look Beyond Docker?

Docker is amazing — it literally started the container revolution. But over time, some concerns have come up:

| Concern | Explanation |
|---------|-------------|
| **Daemon dependency** | Docker runs a long-lived daemon (`dockerd`) that has root privileges. If the daemon crashes, ALL your containers go down. |
| **Root access requirement** | By default, Docker needs root. That's a security concern, especially in shared or production environments. |
| **Monolithic architecture** | Docker bundles everything (build, run, push, pull) in one tool. In the Unix philosophy, each tool should do one thing well. |
| **Licensing changes** | Docker Desktop introduced commercial licensing for larger organizations (as of 2022+). |
| **Kubernetes moved on** | Kubernetes deprecated the Docker shim (dockershim) in v1.24. It now uses `containerd` or `CRI-O` directly — not Docker. |

So the Red Hat team created three separate tools:

```
┌─────────────────────────────────────────────────┐
│           The "Docker Alternative" Trio          │
│                                                  │
│   🔧 Podman   → Run containers (like docker)    │
│   🏗️ Buildah  → Build images (like docker build)│
│   📋 Skopeo   → Copy/inspect images             │
│                                                  │
│   All are daemonless, rootless, OCI-compliant    │
└─────────────────────────────────────────────────┘
```

> **Key Point:** You don't *have* to replace Docker. These are *alternatives* that solve specific pain points. Many teams use Docker in development and Podman in production (or vice versa).

## 2. Podman — The Daemonless Container Engine

### What is Podman?

**Podman** (short for **Pod Manager**) is a container engine that lets you run, manage, and work with containers — just like Docker — but **without a daemon** and **without requiring root**.

Think of Podman as a drop-in replacement for Docker. In fact, the CLI is so similar that you can literally do this:

```bash
alias docker=podman
```

And most of your Docker commands will just work!

### Podman vs Docker — Key Differences

| Feature | Docker | Podman |
|---------|--------|--------|
| **Architecture** | Client-Server (daemon) | Daemonless (fork-exec model) |
| **Root required?** | Yes (by default) | No (rootless by default!) |
| **Container process** | Child of `dockerd` daemon | Child of the user's shell |
| **Pod support** | No native pods | Yes! First-class pod support |
| **Docker Compose** | Built-in | Supported via `podman-compose` or Docker Compose v2 |
| **Kubernetes YAML** | Not supported | Can generate & play K8s YAML |
| **Socket** | `/var/run/docker.sock` | `/run/user/$UID/podman/podman.sock` (per-user) |
| **systemd integration** | Limited | Excellent (generate systemd units) |
| **OCI compliant** | Yes | Yes |
| **Image format** | Docker & OCI | Docker & OCI |
| **Default runtime** | runc/containerd | crun (faster, written in C) |

#### How Docker Works (Client-Server with Daemon):

```
┌──────────┐       ┌──────────────────┐       ┌───────────┐
│  Docker   │──────▶│   Docker Daemon   │──────▶│ Container │
│  CLI      │ REST  │   (dockerd)       │ fork  │ (runc)    │
│           │  API  │   🔴 runs as root │       │           │
└──────────┘       └──────────────────┘       └───────────┘
                         │
                         ▼
                   ALL containers are
                   children of dockerd
                   (daemon dies = all die)
```

#### How Podman Works (Daemonless, Fork-Exec):

```
┌──────────┐       ┌───────────┐
│  Podman   │─────▶│ Container │
│  CLI      │ fork │ (crun)    │
│  (no      │ exec │           │
│   daemon!)│      │ 🟢 child  │
└──────────┘      │  of user  │
                   └───────────┘
                   
   Each container is independently
   managed — no single point of failure!
```

### Podman CLI — It's Almost the Same as Docker!

Here's the beautiful part — if you know Docker, you already know Podman:

#### Running Containers:

```bash
# Pull an image
podman pull nginx:latest

# Run a container (same as docker run!)
podman run -d --name my-nginx -p 8080:80 nginx:latest

# List running containers
podman ps

# List ALL containers (including stopped)
podman ps -a

# View logs
podman logs my-nginx

# Execute a command inside a running container
podman exec -it my-nginx /bin/bash

# Stop a container
podman stop my-nginx

# Remove a container
podman rm my-nginx

# Remove all stopped containers
podman rm -a
```

#### Working with Images:

```bash
# List images

# Pull an image
podman pull alpine:3.19

# Push an image
podman push myregistry.com/alpine:v1

# Remove an image
podman rmi alpine:3.19

# Build an image from a Dockerfile (yes, Podman reads Dockerfiles!)
podman build -t myapp:v1 .

# Inspect an image
podman inspect nginx:latest

# View image history (layers)
podman history nginx:latest
```

#### Volumes:

```bash
# Create a named volume

podman volume ls

# Use a volume
podman run -d -v mydata:/data alpine

# Remove a volume
podman volume rm mydata
```

#### Networks:

```bash
# Create a network
podman network create mynet

# List networks
podman network ls

podman run -d --network mynet --name app1 nginx

podman network inspect mynet

# Remove a network

```

### Rootless Containers with Podman

This is one of Podman's superpowers. Running containers **without root** means:
- No privilege escalation risk
- Each user has their own container space
- Containers run as the user's UID, not root
- If a container is compromised, the attacker only has the user's privileges — NOT root!

```bash
# Check if running rootless
podman info | grep rootless
# rootless: true  ✅

# Run a container as a regular user (no sudo needed!)
podman run -d --name web -p 8080:80 nginx

# Check who owns the container process
ps aux | grep nginx
# youruser  12345 ... nginx   ← runs as YOUR user, not root!
```

#### How Rootless Works Under the Hood:

```
┌─────────────────────────────────────────────┐
│              User Namespace Mapping          │
│                                              │
│  Host User: ritesh (UID 1000)               │
│                                              │
│  Inside Container:                           │
│    root (UID 0)  →  maps to UID 100000      │
│    user1 (UID 1) →  maps to UID 100001      │
│    ...                                       │
│                                              │
│  The "root" inside the container is NOT      │
│  actually root on the host! It's a regular   │
│  unprivileged user. 🔒                       │
└─────────────────────────────────────────────┘
```

The UID mapping is configured in:

```bash
# Check subuid mapping
cat /etc/subuid
# ritesh:100000:65536
# This means user "ritesh" can use UIDs 100000 through 165535

# Check subgid mapping
cat /etc/subgid
# ritesh:100000:65536
```

### Pods in Podman — The Kubernetes Connection

Unlike Docker, Podman has **native pod support**. A pod is a group of containers that share the same network namespace — exactly like Kubernetes pods!

```bash
# Create a pod
podman pod create --name my-pod -p 8080:80

# Add containers to the pod
podman run -d --pod my-pod --name web nginx
podman run -d --pod my-pod --name sidecar busybox sleep 3600

# List pods
podman pod list

# Inspect a pod
podman pod inspect my-pod

# The containers in the pod share localhost!
# From the sidecar, you can access nginx at localhost:80
podman exec sidecar wget -qO- http://localhost:80

# Stop a pod (stops all containers in it)
podman pod stop my-pod

# Remove a pod
podman pod rm my-pod

# Remove pod and all its containers
podman pod rm -f my-pod
```

#### Pod Architecture:

```
┌──────────────── Pod: my-pod ────────────────┐
│                                              │
│  ┌──────────┐  ┌──────────┐  ┌───────────┐ │
│  │  Infra    │  │  nginx   │  │  sidecar  │ │
│  │Container  │  │  (web)   │  │ (busybox) │ │
│  │ (pause)   │  │          │  │           │ │
│  └──────────┘  └──────────┘  └───────────┘ │
│                                              │
│  ← Shared Network Namespace (localhost) →    │
│  ← Shared Port Mappings →                   │
│  ← Shared IPC Namespace →                   │
└──────────────────────────────────────────────┘
```

> **Note:** The "infra" container (also called "pause" container) holds the network namespace. It does nothing but sleep — just like in Kubernetes!

### Podman Compose

Podman supports Docker Compose files! You have two options:

#### Option 1: `podman-compose` (Python-based)

```bash
# Install podman-compose
pip3 install podman-compose

# Use it just like docker-compose
podman-compose up -d
podman-compose ps
podman-compose down
```

#### Option 2: Docker Compose v2 with Podman socket

```bash
# Start the Podman socket (emulates Docker socket)
systemctl --user start podman.socket

# Set the Docker host to use Podman's socket
export DOCKER_HOST=unix:///run/user/$UID/podman/podman.sock

# Now docker-compose just works!
docker compose up -d
```

#### Example `docker-compose.yml` (works with both Podman and Docker):

```yaml
version: "3.8"

services:
  web:
    image: nginx:alpine
    ports:
      - "8080:80"
    volumes:
      - ./html:/usr/share/nginx/html
    depends_on:
      - api

  api:
    image: python:3.11-slim
    working_dir: /app
    command: python -m http.server 5000
    ports:
      - "5000:5000"
    volumes:
      - ./app:/app

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_PASSWORD: mysecretpassword
      POSTGRES_DB: myapp
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  pgdata:
```

```bash
# Run with podman-compose
podman-compose up -d

# Check running containers
podman-compose ps

# Tear it down
podman-compose down
```

### Generating Kubernetes YAML from Podman

This is a **killer feature** of Podman. You can take your running pods/containers and generate Kubernetes-ready YAML!

```bash
# Create a pod with containers
podman pod create --name webapp -p 8080:80
podman run -d --pod webapp --name frontend nginx:alpine
podman run -d --pod webapp --name backend python:3.11-slim python -m http.server 5000

# Generate Kubernetes YAML!
podman generate kube webapp > webapp.yaml
```

The generated `webapp.yaml` will look something like:

```yaml
apiVersion: v1
kind: Pod
metadata:
  labels:
    app: webapp
  name: webapp
spec:
  containers:
    - name: frontend
      image: docker.io/library/nginx:alpine
      ports:
        - containerPort: 80
          hostPort: 8080
    - name: backend
      image: docker.io/library/python:3.11-slim
      command:
        - python
        - -m
        - http.server
        - "5000"
```

You can also **play** (apply) Kubernetes YAML in Podman:

```bash
# Run containers from Kubernetes YAML (no Kubernetes cluster needed!)
podman play kube webapp.yaml

# Tear it down
podman play kube --down webapp.yaml
```

> **Why this is awesome:** You develop locally with Podman, generate K8s YAML, and deploy to Kubernetes. No rewriting config files!

### Podman with systemd — Running Containers as Services

Podman integrates beautifully with `systemd`. You can generate systemd service files for your containers!

```bash
# Run a container
podman run -d --name my-web -p 8080:80 nginx:alpine

# Generate a systemd service file
podman generate systemd --name my-web --new --files
# Creates: container-my-web.service

# Move it to the systemd user directory
mkdir -p ~/.config/systemd/user/
mv container-my-web.service ~/.config/systemd/user/

# Reload systemd
systemctl --user daemon-reload

# Enable and start the service
systemctl --user enable --now container-my-web.service

# Check status
systemctl --user status container-my-web.service

# The container will now auto-start on login!
```

#### Quadlet — The Modern Way (Podman 4.4+):

Quadlet is the newer, cleaner way to run containers as systemd services:

```ini
# ~/.config/containers/systemd/my-web.container
[Container]
Image=nginx:alpine
PublishPort=8080:80
Volume=./html:/usr/share/nginx/html

[Service]
Restart=always

[Install]
WantedBy=default.target
```

```bash
# Reload systemd to pick up the Quadlet file
systemctl --user daemon-reload

# Start the service
systemctl --user start my-web

# Enable it at boot
systemctl --user enable my-web
```

## 3. Buildah — Build OCI Images Without a Daemon

### What is Buildah?

**Buildah** is a tool for **building OCI-compliant container images**. Think of it as `docker build` — but without needing a daemon, and with way more control over the build process.

The name "Buildah" comes from the way people in Boston say "builder" (Build-ah). 😄

### Why Buildah Over `docker build`?

| Feature | `docker build` | Buildah |
|---------|----------------|---------|
| **Daemon needed?** | Yes | No |
| **Root needed?** | Yes (default) | No (rootless) |
| **Dockerfile support** | Yes | Yes |
| **Build without Dockerfile** | No | Yes! (scripted builds) |
| **Fine-grained control** | Limited | Full control over each layer |
| **Commit layers manually** | No | Yes |
| **OCI image format** | Yes | Yes (default) |
| **Smaller images** | Depends | Easier to create minimal images |

### Installing Buildah

```bash
# Ubuntu/Debian
sudo apt install -y buildah

# Fedora
sudo dnf install -y buildah

# RHEL/CentOS 8+
sudo yum install -y buildah
```

```bash
# Verify installation
buildah --version
# buildah version 1.33.2
```

### Building Images with Buildah — Two Ways

Buildah gives you **two approaches** to build images:

```
┌──────────────────────────────────────────────┐
│         Two Ways to Build with Buildah        │
│                                               │
│  1. Dockerfile-based  → buildah bud           │
│     (same as docker build, uses Dockerfile)   │
│                                               │
│  2. Scripted/CLI-based → buildah from/run/..  │
│     (no Dockerfile needed! Full control!)     │
│                                               │
└──────────────────────────────────────────────┘
```

### Buildah CLI Commands

Here are the core Buildah commands:

| Command | What it does |
|---------|-------------|
| `buildah from` | Create a new working container from a base image |
| `buildah run` | Run a command inside the working container |
| `buildah copy` | Copy files into the container |
| `buildah config` | Set image configuration (CMD, ENTRYPOINT, ENV, etc.) |
| `buildah commit` | Save the working container as a new image |
| `buildah bud` | Build Using Dockerfile (like `docker build`) |
| `buildah images` | List images |
| `buildah containers` | List working containers |
| `buildah rm` | Remove a working container |
| `buildah rmi` | Remove an image |
| `buildah push` | Push an image to a registry |
| `buildah mount` | Mount a container's root filesystem |
| `buildah unmount` | Unmount a container's root filesystem |

### Scripted Builds with Buildah (No Dockerfile Needed!)

This is Buildah's **superpower**. You can build images step by step using shell commands:

```bash
#!/bin/bash
# build-app.sh — Build a Node.js app image without a Dockerfile!

# Step 1: Start a new container from a base image
container=$(buildah from node:20-alpine)

# Step 2: Set the working directory
buildah config --workingdir /app $container

# Step 3: Copy package files first (for layer caching)
buildah copy $container package.json package-lock.json ./

# Step 4: Install dependencies
buildah run $container -- npm ci --production

# Step 5: Copy the rest of the application code
buildah copy $container . .

# Step 6: Set environment variables
buildah config --env NODE_ENV=production $container
buildah config --env PORT=3000 $container

# Step 7: Expose port
buildah config --port 3000 $container

# Step 8: Set the startup command
buildah config --cmd "node server.js" $container

# Step 9: Set labels
buildah config --label maintainer="ritesh@example.com" $container
buildah config --label version="1.0.0" $container

# Step 10: Commit the container to an image
buildah commit $container myapp:v1

# Step 11: Clean up the working container
buildah rm $container

echo "✅ Image built: myapp:v1"
```

```bash
# Run the build script
chmod +x build-app.sh
./build-app.sh

# Verify the image was created
buildah images
# REPOSITORY    TAG   IMAGE ID       CREATED         SIZE
# myapp         v1    a1b2c3d4e5f6   5 seconds ago   120 MB
```

#### Why is this better than a Dockerfile in some cases?

1. **Full scripting power** — Use if/else, loops, variables, functions
2. **Conditional builds** — Include debug tools only in dev builds
3. **Dynamic content** — Fetch configs from APIs during build
4. **Mount host filesystem** — Build faster with host tools

Example with conditional logic:

```bash
#!/bin/bash
# Conditional build — add debug tools only in development

container=$(buildah from python:3.11-slim)

buildah copy $container requirements.txt /app/requirements.txt
buildah run $container -- pip install -r /app/requirements.txt
buildah copy $container . /app

# Conditional: Add debug tools only for dev builds
if [ "$BUILD_ENV" = "development" ]; then
    buildah run $container -- pip install debugpy ipdb
    buildah config --env DEBUG=true $container
    echo "🐛 Debug tools installed"
fi

buildah config --workingdir /app $container
buildah config --cmd "python app.py" $container
buildah commit $container myapp:latest
buildah rm $container
```

```bash
# Build for development (with debug tools)
BUILD_ENV=development ./build.sh

# Build for production (no debug tools)
BUILD_ENV=production ./build.sh
```

#### Mount the Container Filesystem:

```bash
# Create a container
container=$(buildah from alpine)

# Mount its filesystem to the host
mountpoint=$(buildah mount $container)
echo $mountpoint
# /home/user/.local/share/containers/storage/overlay/xxxx/merged

# Now you can use HOST tools to modify the container!
# No need to install tools inside the container
echo "Hello from Buildah!" > $mountpoint/greeting.txt

# Or use host's package manager to install things
# (This is MUCH faster than buildah run)
dnf install --installroot $mountpoint -y python3

# Unmount when done
buildah unmount $container

# Commit the changes
buildah commit $container my-custom-image:v1
```

### Building from a Dockerfile with Buildah

If you already have Dockerfiles, Buildah can use them directly:

```bash
# Build Using Dockerfile (bud = Build Using Dockerfile)
buildah bud -t myapp:v1 .

# Specify a custom Dockerfile name
buildah bud -t myapp:v1 -f Dockerfile.prod .

# Build with build arguments
buildah bud --build-arg VERSION=1.0 -t myapp:v1 .

# Build without cache
buildah bud --no-cache -t myapp:v1 .

# Multi-stage build — same as Docker
buildah bud -t myapp:v1 --target production .
```

Example Dockerfile (works with both `docker build` and `buildah bud`):

```dockerfile
# Dockerfile
FROM golang:1.22-alpine AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 go build -o /app/server .

FROM alpine:3.19
RUN apk --no-cache add ca-certificates
COPY --from=builder /app/server /usr/local/bin/server
EXPOSE 8080
CMD ["server"]
```

```bash
# Build with Buildah
buildah bud -t my-go-app:v1 .

# Push to a registry
buildah push my-go-app:v1 docker://docker.io/myuser/my-go-app:v1
```

## 4. Skopeo — Inspect & Copy Container Images

### What is Skopeo?

**Skopeo** is a command-line tool for working with container images in **registries** — without needing to pull (download) the entire image. It can:

- **Inspect** image metadata without downloading
- **Copy** images between different registries
- **Delete** images from registries
- **Sign** and verify images
- **Sync** repositories

Think of Skopeo as a "Swiss Army knife" for container registries.

```
┌──────────────────────────────────────────────┐
│                 Skopeo Can...                 │
│                                               │
│  📋 Inspect images remotely (no pull needed)  │
│  📦 Copy images between registries            │
│  🔄 Convert between image formats             │
│  🗑️ Delete images from registries             │
│  ✍️ Sign images for verification              │
│  📂 Save images to local directories/tarballs│
└──────────────────────────────────────────────┘
```

### Installing Skopeo

```bash
# Ubuntu/Debian
sudo apt install -y skopeo

# Fedora
sudo dnf install -y skopeo

# RHEL/CentOS 8+
sudo yum install -y skopeo

# macOS
brew install skopeo
```

```bash
# Verify
skopeo --version
```

### Skopeo Commands & Use Cases

#### Understanding Image Transports

Skopeo uses **transports** to specify where images are. Here are the most common ones:

| Transport | Description | Example |
|-----------|-------------|---------|
| `docker://` | Docker registry (Docker Hub, ECR, GCR, etc.) | `docker://docker.io/library/nginx:latest` |
| `containers-storage:` | Local Podman/Buildah storage | `containers-storage:localhost/myapp:v1` |
| `dir:` | Local directory | `dir:/tmp/nginx-image` |
| `docker-archive:` | Docker save/load tarball | `docker-archive:/tmp/nginx.tar` |
| `oci:` | OCI layout directory | `oci:/tmp/nginx-oci:latest` |
| `docker-daemon:` | Local Docker daemon | `docker-daemon:nginx:latest` |

### Copying Images Between Registries

This is Skopeo's most powerful feature — copy images directly between registries without pulling to your local machine:

```bash
# Copy from Docker Hub to a private registry
skopeo copy \
  docker://docker.io/library/nginx:latest \
  docker://myregistry.company.com/nginx:latest

# Copy from one registry to another
skopeo copy \
  docker://gcr.io/my-project/myapp:v1 \
  docker://123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:v1

# Copy with authentication
skopeo copy \
  --src-creds user:password \
  --dest-creds awsuser:awspassword \
  docker://docker.io/myuser/myapp:v1 \
  docker://my-ecr-registry/myapp:v1
```

#### Why is this better than `docker pull` + `docker tag` + `docker push`?

```
Traditional Way (Docker):
  Registry A → [Download to local] → [Re-upload] → Registry B
  ⏱️ Slow: Downloads entire image, then uploads again

Skopeo Way:
  Registry A → Registry B  (direct server-to-server)
  ⚡ Fast: Copies directly, no local download needed!
  💾 Saves: No local disk space required
```

#### More Copy Examples:

```bash
# Save an image to a local directory (no Docker needed)
skopeo copy docker://alpine:latest dir:/tmp/alpine-image

# Save as a Docker tarball
skopeo copy docker://alpine:latest docker-archive:/tmp/alpine.tar

# Save as OCI layout
skopeo copy docker://alpine:latest oci:/tmp/alpine-oci:latest

# Load from tarball to a registry
skopeo copy docker-archive:/tmp/alpine.tar docker://myregistry.com/alpine:latest

# Copy from local Podman storage to a registry
skopeo copy containers-storage:localhost/myapp:v1 docker://myregistry.com/myapp:v1

# Copy with multi-arch support (manifest list)
skopeo copy --all docker://nginx:latest docker://myregistry.com/nginx:latest
```

### Real-World Workflow Example:

```bash
#!/bin/bash

# Step 1: Build with Buildah

buildah bud -t myapp:v1 .

# Step 2: Test with Podman

podman run --rm myapp:v1 npm test

# Step 3: Run locally with Podman to verify

  containers-storage:localhost/myapp:v1 \
  docker://myregistry.com/myapp:v1

```

## 8. Common Interview Questions

### Q1: What is the main architectural difference between Docker and Podman?
**A:** Docker uses a client-server architecture with a long-lived daemon (`dockerd`) running as root. Podman uses a daemonless, fork-exec model where each container is a direct child process of the user's shell. This means no single point of failure and native rootless support.

### Q2: Can I use my existing Dockerfiles with Podman and Buildah?
**A:** Yes! Both Podman (`podman build`) and Buildah (`buildah bud`) can read and build from standard Dockerfiles without any changes.

### Q3: What's the advantage of Buildah's scripted builds over Dockerfiles?
**A:** Scripted builds give you full programming power — conditional logic, loops, variables, host tool access, and dynamic builds. You can also mount the container filesystem and use host tools directly, which is much faster than installing tools inside the container.

### Q4: How does Skopeo copy images between registries without pulling?
**A:** Skopeo communicates directly with registry APIs. It reads image layers from the source registry and writes them to the destination registry — the data flows server-to-server and never touches your local disk. This is faster and requires no local storage.

### Q5: Why does Podman have native pod support?
**A:** Podman was designed with Kubernetes compatibility in mind. Pods in Podman share network namespaces, just like Kubernetes pods. You can even generate Kubernetes YAML from Podman pods using `podman generate kube`, making the transition from local development to Kubernetes smooth.

### Q6: Can Podman replace Docker in a CI/CD pipeline?
**A:** Absolutely. Podman, Buildah, and Skopeo together cover all CI/CD container needs. Buildah builds images without needing a privileged daemon (solving Docker-in-Docker security issues), and Skopeo pushes images without needing a local daemon.

### Q7: What is a Quadlet in Podman?
**A:** Quadlet is a systemd integration feature (Podman 4.4+) that lets you define containers as systemd units using simple `.container` files. The system reads these files and auto-generates proper systemd services, making it easy to run containers as managed system services.

## 9. Summary

| Tool | Purpose | Key Superpower |
|------|---------|----------------|
| **Podman** | Run & manage containers | Daemonless + rootless + pod support |
| **Buildah** | Build container images | Scripted builds without Dockerfile |
| **Skopeo** | Inspect & copy images | Registry-to-registry copy without local pull |

### When to Use What:

- **Use Docker** when: You're in development, using Docker Desktop, team is comfortable with it
- **Use Podman** when: You need rootless containers, systemd integration, pod support, or Kubernetes YAML generation
- **Use Buildah** when: You need fine-grained control over builds, conditional build logic, or daemonless builds in CI/CD
- **Use Skopeo** when: You need to inspect images remotely, copy between registries, or work in air-gapped environments

### Key Takeaway:

> You don't have to choose one over the other. Docker and the Podman/Buildah/Skopeo trio can coexist. Use whatever makes your workflow simpler. The container images they produce are all OCI-compliant and interchangeable! 🐳
