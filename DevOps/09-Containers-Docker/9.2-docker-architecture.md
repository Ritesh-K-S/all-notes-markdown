# 9.2 — Docker Architecture — Daemon, Client, Registry, Images, Containers

> **One Line Answer:** Docker uses a **client-server architecture** where the **Docker Client** (CLI) sends commands to the **Docker Daemon** (server), which does the heavy lifting — building images, running containers, and talking to **Registries** to push/pull images.

---

## 🤔 Before We Dive In — Why Architecture Matters

You might be thinking: *"I just want to run `docker run` and get stuff working. Why do I need to know the architecture?"*

Great question. Here's why:

1. **Troubleshooting** — When something breaks (and it will), knowing the architecture helps you figure out WHERE the problem is. Is it the daemon? The client? The registry? Networking?

2. **Security** — Understanding the daemon runs as root helps you understand the security implications

3. **Interviews** — "Explain Docker's architecture" is a **guaranteed** interview question

4. **Advanced Usage** — Remote Docker daemons, Docker-in-Docker, rootless Docker — none of these make sense without understanding the architecture

So let's break it all down, piece by piece.

---

## 🏗️ Docker Architecture — The Big Picture

Docker follows a **client-server** model. Here's the complete picture:

```

                          │   DOCKER REGISTRY  │
                          │   (Docker Hub,     │
                          │    ECR, GCR, etc.) │

                          └─────────▲──────────┘

┌──────────────┐   REST API   ┌─────┴──────────────────────────────┐
│ DOCKER CLIENT│ ──────────── │         DOCKER DAEMON              │
│  (docker CLI)│   (over      │         (dockerd)                  │

│ docker run   │   or TCP)    │  ┌───────────┐  ┌───────────────┐  │
```

There are **three main components**:

| Component | What It Is |
|-----------|------------|
| **Docker Client** | The CLI tool you interact with (`docker` command) |
| **Docker Daemon** | The background service that does all the work (`dockerd`) |
| **Docker Registry** | Remote storage for container images (Docker Hub, etc.) |

And **four objects** that Docker manages:

| Object | What It Is |
|--------|------------|
| **Images** | Read-only templates to create containers |
| **Containers** | Running instances of images |
| **Volumes** | Persistent data storage |
| **Networks** | Communication channels between containers |

## 1️⃣ Docker Client (`docker`)

The Docker Client is what **you** interact with. It's the command-line tool that you type commands into.

### What Does the Client Do?

The client is basically a **messenger**. It doesn't do any real work itself. It:

1. Takes your command (e.g., `docker run nginx`)
2. Converts it into a **REST API call**
3. Sends it to the **Docker Daemon**
4. Receives the response and displays it to you

```
You type:  docker run -d -p 80:80 nginx

What actually happens:

  ┌──────────────┐                    ┌──────────────┐
  │ Docker Client│ ── REST API ────── │ Docker Daemon│
  │              │    POST /v1.41/    │              │
  │ "Please run  │    containers/    │ "OK, I'll    │
  │  nginx on    │    create         │  pull nginx, │
  │  port 80"    │                    │  create the  │
  │              │ ◀── Response ───── │  container,  │
  │ "Container   │    201 Created     │  start it"   │
  │  ID: abc123" │                    │              │
  └──────────────┘                    └──────────────┘
```

### How Does the Client Talk to the Daemon?

By default, they communicate over a **Unix socket**:

```bash
# The default socket path on Linux:
/var/run/docker.sock

# The client connects to this socket to send commands
```

But the client can also talk to a **remote daemon** over TCP:

```bash
# Connect to a remote Docker daemon
export DOCKER_HOST=tcp://192.168.1.100:2376

# Now all docker commands go to that remote machine!
docker ps   # Shows containers on the REMOTE machine
```

This is how you can manage Docker on a remote server from your local machine.

### Client Configuration:

The client stores its config in `~/.docker/config.json`:

```json
{
  "auths": {
    "https://index.docker.io/v1/": {
      "auth": "base64-encoded-credentials"
    }
  },
  "credsStore": "desktop",
  "currentContext": "default"
}
```

This file stores:
- **Registry credentials** (for `docker login`)
- **Docker contexts** (for switching between local/remote daemons)
- **Plugin settings**

### Docker Contexts (Switching Between Daemons):

```bash
# List available contexts
docker context ls

# Create a new context for a remote server
docker context create my-server \
  --docker "host=tcp://192.168.1.100:2376"

# Switch to that context
docker context use my-server

# Now all commands go to the remote server
docker ps  # Shows remote containers
```

## 2️⃣ Docker Daemon (`dockerd`)

The Docker Daemon is the **brain** of Docker. It's a background process (service) that runs on the host machine and does ALL the heavy lifting.

### What Does the Daemon Do?

| Task | Description |
|------|-------------|
| **Manages Images** | Builds, pulls, pushes, stores, and deletes images |
| **Manages Containers** | Creates, starts, stops, restarts, kills, and removes containers |
| **Manages Volumes** | Creates and manages persistent storage |
| **Manages Networks** | Creates and manages container networks |
| **Listens for API calls** | Receives and processes requests from the Docker Client |
| **Talks to Registries** | Pulls images from and pushes images to registries |

### How the Daemon Runs:

```bash
# On most Linux systems, dockerd runs as a systemd service
sudo systemctl status docker

# Output:
# ● docker.service - Docker Application Container Engine
#    Loaded: loaded
#    Active: active (running)
#    Main PID: 1234 (dockerd)

# You can also run it manually (for debugging):
sudo dockerd --debug
```

### Daemon Configuration:

The daemon is configured via `/etc/docker/daemon.json`:

```json
{
  "storage-driver": "overlay2",
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3"
  },
  "default-address-pools": [
    {
      "base": "172.17.0.0/16",
      "size": 24
    }
  ],
  "dns": ["8.8.8.8", "8.8.4.4"],
  "insecure-registries": [],
  "registry-mirrors": ["https://mirror.example.com"],
  "live-restore": true,
  "debug": false
}
```

Let's understand the important settings:

| Setting | What It Does |
|---------|--------------|
| `storage-driver` | How images and container layers are stored on disk (overlay2 is the default and best for most cases) |
| `log-driver` | Default logging driver for containers (json-file, syslog, fluentd, etc.) |
| `log-opts` | Log rotation settings (max-size, max-file) — **very important in production!** |
| `dns` | Default DNS servers for containers |
| `insecure-registries` | Allow pulling from HTTP (non-HTTPS) registries (not recommended for production) |
| `registry-mirrors` | Mirror registries for faster image pulls |
| `live-restore` | Keeps containers running even if the daemon is restarted |
| `debug` | Enable debug-level logging |

### The Daemon Runs as Root ⚠️

By default, the Docker daemon runs as the **root** user. This is important to understand:

```
dockerd (runs as root)
  └── Has full access to the host system
  └── Can mount any host directory
  └── Can access any network interface
  └── Can create/modify/delete any file
```

**Security Implication:** Anyone who can talk to the Docker daemon effectively has **root access** to the host machine. This is why:

- The Docker socket (`/var/run/docker.sock`) is owned by `root:docker`
- Only users in the `docker` group can use Docker without `sudo`
- Adding a user to the `docker` group is essentially giving them root access
- In production, consider **Rootless Docker** (covered in 9.13)

```bash
# Adding a user to the docker group
sudo usermod -aG docker $USER

# ⚠️ This gives the user root-equivalent access!
# They can now mount host directories, access host networking, etc.
```

## 3️⃣ Docker Daemon — Under the Hood (containerd + runc)

Here's something most beginners don't know: **Docker doesn't actually run containers directly**. It uses **containerd** and **runc** under the hood.

### The Container Runtime Stack:

```
┌──────────────────────────────────────────────────────┐
│                    YOU (User)                         │
│                      │                                │
│                 docker run nginx                      │
│                      ▼                                │
│            ┌─────────────────┐                        │
│            │  Docker Client  │   (CLI - docker)       │
│            │                 │                        │
│            └────────┬────────┘                        │
│                     │ REST API                        │
│                     ▼                                 │
│            ┌─────────────────┐                        │
│            │  Docker Daemon  │   (dockerd)            │
│            │                 │   - Image management   │
│            │                 │   - Build               │
│            │                 │   - Network             │
│            │                 │   - Volume              │
│            └────────┬────────┘                        │
│                     │ gRPC                            │
│                     ▼                                 │
│            ┌─────────────────┐                        │
│            │   containerd    │   (High-level runtime)  │
│            │                 │   - Image pull/push     │
│            │                 │   - Container lifecycle  │
│            │                 │   - Storage              │
│            │                 │   - Networking setup     │
│            └────────┬────────┘                        │
│                     │ OCI spec                        │
│                     ▼                                 │
│            ┌─────────────────┐                        │
│            │      runc       │   (Low-level OCI runtime)│
│            │                 │   - Creates namespaces   │
│            │                 │   - Sets up cgroups      │
│            │                 │   - Actually runs the    │
│            │                 │     container process    │
│            └────────┬────────┘                        │
│                     │                                 │
│                     ▼                                 │
│            ┌─────────────────┐                        │
│            │  Linux Kernel   │   (namespaces + cgroups) │
│            └─────────────────┘                        │
└──────────────────────────────────────────────────────┘
```

### Let's Understand Each Layer:

| Component | Role | Analogy |
|-----------|------|---------|
| **Docker CLI** | Takes your commands, sends to daemon | You (the person giving orders) |
| **Docker Daemon (dockerd)** | Orchestrates everything, manages high-level objects | The manager who coordinates |
| **containerd** | Manages container lifecycle, pulls images | The supervisor on the factory floor |
| **runc** | Actually creates and runs the container | The worker who builds the thing |
| **Linux Kernel** | Provides isolation (namespaces) and limits (cgroups) | The factory building itself |

### Why the Separation?

You might wonder: *"Why so many layers? Why doesn't Docker just run containers directly?"*

**Historical reason:** Docker originally was one big monolith. Over time, it was broken into smaller, modular components:

```
2013 — Docker (one big binary)
  └── Did everything: CLI, daemon, runtime, image management

2015 — Docker donated runc to OCI
  └── Low-level runtime became a separate project

2016 — Docker donated containerd to CNCF
  └── High-level runtime became a separate project

Now — Docker is modular:
  └── Docker CLI → talks to dockerd
  └── dockerd → talks to containerd
  └── containerd → talks to runc
  └── runc → talks to the kernel
```

**Why this matters:**
- **Kubernetes** dropped Docker and uses **containerd** directly (since Kubernetes 1.24)
- You can use **containerd** without Docker at all
- You can swap **runc** for other OCI runtimes (gVisor, Kata Containers)

```
Docker path:     docker CLI → dockerd → containerd → runc → kernel
Kubernetes path:            kubelet → containerd → runc → kernel
                            (No Docker needed!)
```

## 4️⃣ Container Internals — Namespaces, Cgroups, Storage

A container is not a magic object. It is a normal Linux process with extra isolation and resource controls.

### Container = Process + Namespaces + Cgroups + Filesystem

- **Container** = one or more processes running on the host kernel
- **Namespaces** = separate visibility; a filtered view of kernel resources
- **Cgroups** = resource limits and accounting (CPU, memory, I/O)
- **Filesystem** = the image and writable layer mounted by the runtime

### Namespaces are about what a process can see

Namespaces are not separate kernels or separate hardware. They are the Linux kernel giving a process a limited view.

- PID namespace → each container sees its own PID numbering
- NET namespace → each container sees its own network devices and routes
- MNT namespace → each container sees its own filesystem mounts
- UTS, IPC, USER namespaces → isolate hostnames, IPC, and user IDs

**Example:**
- Container A sees `ps aux` and only its own processes
- Container B sees only its own processes
- Host sees both containers and all host processes

### Cgroups are about how much a process can use

Cgroups are different from namespaces.

- `namespace` = visibility isolation
- `cgroup` = resource isolation

Example:
- Container A can be limited to `1GB RAM`
- Container B can be limited to `2 CPUs`
- If one container leaks memory, cgroups prevent it from crashing the host

### Where Docker objects actually exist

Docker objects are physical. They exist on disk and in kernel state.

- `/var/lib/docker/containers/<id>/` → container metadata, config, logs
- `/var/lib/docker/overlay2/` → image and writable filesystem layers
- `/var/lib/docker/volumes/` → persistent volume data
- `/var/run/docker.sock` → Docker daemon control socket
- `/proc/<pid>/ns/` → namespace references for each process
- `/sys/fs/cgroup/` → cgroup control files and metrics
- `docker0` and `veth*` → Linux network interfaces for containers

### The real internal flow

When you run `docker run nginx`:
1. Docker CLI sends the request to `dockerd`
2. `dockerd` talks to `containerd`
3. `containerd` calls `runc`
4. `runc` uses Linux syscalls like `clone()`, `setns()`, `unshare()`, and `mount()` to create the container
5. The Linux kernel creates namespaces, attaches the process, and enforces cgroup limits

### Mental model

```
Container = normal Linux process
          + isolated view (namespaces)
          + resource limits (cgroups)
          + image filesystem
```

This is the foundation of Docker and Kubernetes. Everything else is management tooling.

## 5️⃣ Docker Registry

A Docker Registry is a **storage and distribution system** for container images. Think of it as **GitHub but for container images** instead of code.

### How Registries Work:

```
┌───────────────────────────────────────────────┐
│                 DOCKER REGISTRY                │
│                                               │
│  Repository: nginx                            │
│  ┌────────────────────────────────────────┐   │
│  │  Tags:                                 │   │
│  │    nginx:latest    → Image Manifest    │   │
│  │    nginx:1.25      → Image Manifest    │   │
│  │    nginx:1.25.3    → Image Manifest    │   │
│  │    nginx:alpine    → Image Manifest    │   │
│  └────────────────────────────────────────┘   │
│                                               │
│  Repository: ubuntu                           │
│  ┌────────────────────────────────────────┐   │
│  │  Tags:                                 │   │
│  │    ubuntu:22.04    → Image Manifest    │   │
│  │    ubuntu:latest   → Image Manifest    │   │
│  │    ubuntu:20.04    → Image Manifest    │   │
│  └────────────────────────────────────────┘   │
│                                               │
└───────────────────────────────────────────────┘
```

### Key Concepts:

| Term | Meaning | Example |
|------|---------|---------|
| **Registry** | The server hosting images | `docker.io` (Docker Hub), `ghcr.io` (GitHub) |
| **Repository** | A collection of related images | `nginx`, `ubuntu`, `mycompany/myapp` |
| **Tag** | A specific version of an image | `nginx:1.25`, `nginx:latest`, `nginx:alpine` |
| **Digest** | A unique SHA256 hash of an image | `nginx@sha256:abc123...` (immutable, can't be changed) |

### Image Naming Convention:

```
Full image name format:
  [registry/][namespace/]repository[:tag|@digest]

Examples:
  nginx                              → docker.io/library/nginx:latest
  nginx:1.25                         → docker.io/library/nginx:1.25
  mycompany/myapp:v2                 → docker.io/mycompany/myapp:v2
  ghcr.io/myorg/myapp:latest        → ghcr.io/myorg/myapp:latest
  123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:v1
                                     → Full AWS ECR path
```

When you just type `nginx`, Docker assumes:
- **Registry:** `docker.io` (Docker Hub)
- **Namespace:** `library` (official images)
- **Tag:** `latest`

### Types of Registries:

| Registry | Type | Description |
|----------|------|-------------|
| **Docker Hub** (`docker.io`) | Public | Default registry. Largest. Official images. Free for public repos |
| **GitHub Container Registry** (`ghcr.io`) | Public/Private | Integrated with GitHub. Free for public repos |
| **Amazon ECR** | Private | AWS-managed registry. Integrates with ECS/EKS |
| **Google GCR / Artifact Registry** | Private | GCP-managed registry. Integrates with GKE |
| **Azure ACR** | Private | Azure-managed registry. Integrates with AKS |
| **Harbor** | Self-hosted | Open-source, enterprise-grade, feature-rich |
| **Quay** (`quay.io`) | Public/Private | Red Hat managed. Used for OpenShift |
| **Docker Registry (official)** | Self-hosted | Basic, open-source registry you can run yourself |

### Push and Pull Flow:

```bash
# PULL: Download an image from a registry
docker pull nginx:1.25

# What happens internally:
# 1. Docker client tells daemon: "Pull nginx:1.25"
# 2. Daemon contacts Docker Hub API
# 3. Docker Hub returns the image manifest (list of layers)
# 4. Daemon downloads each layer it doesn't already have
# 5. Layers are stored locally in /var/lib/docker/
# 6. Image is now available locally

# PUSH: Upload an image to a registry
docker push mycompany/myapp:v2

# What happens internally:
# 1. Docker client tells daemon: "Push mycompany/myapp:v2"
# 2. Daemon checks which layers the registry already has
# 3. Only uploads NEW layers (saves bandwidth!)
# 4. Uploads the image manifest
# 5. Image is now available in the registry
```

### Docker Hub — The Default Registry:

```
Docker Hub (hub.docker.com)
├── Official Images (maintained by Docker + upstream projects)
│   ├── nginx
│   ├── ubuntu
│   ├── python
│   ├── node
│   ├── postgres
│   └── redis
├── Verified Publisher Images (companies)
│   ├── bitnami/nginx
│   ├── grafana/grafana
│   └── hashicorp/terraform
└── Community Images (anyone can push)
    ├── someuser/myapp
    └── anotheruser/tool
```

**Important:** Always prefer **Official Images** or **Verified Publisher Images**. Community images can contain anything — even malware!

## 5️⃣ Docker Images — The Template

A Docker Image is a **read-only template** that contains everything needed to run an application. Think of it as a **snapshot** or a **blueprint**.

### What's Inside an Image?

```
┌──────────────────────────────────┐
│          DOCKER IMAGE            │
│  ┌────────────────────────────┐  │
│  │  Application Code          │  │
│  │  (your app, scripts, etc.) │  │
│  ├────────────────────────────┤  │
│  │  Runtime                   │  │
│  │  (Python, Node.js, Java)   │  │
│  ├────────────────────────────┤  │
│  │  System Libraries          │  │
│  │  (libc, openssl, etc.)     │  │
│  ├────────────────────────────┤  │
│  │  System Tools              │  │
│  │  (bash, curl, etc.)        │  │
│  ├────────────────────────────┤  │
│  │  OS Base Layer             │  │
│  │  (Ubuntu, Alpine, Debian)  │  │
│  ├────────────────────────────┤  │
│  │  Metadata                  │  │
│  │  (ENV vars, CMD, EXPOSE)   │  │
│  └────────────────────────────┘  │
└──────────────────────────────────┘
```

### Images are Made of Layers:

Every instruction in a Dockerfile creates a **layer**. Layers are stacked on top of each other.

```dockerfile
FROM ubuntu:22.04          # Layer 1: Base OS
RUN apt-get update         # Layer 2: Updated package list
RUN apt-get install -y python3  # Layer 3: Python installed
COPY app.py /app/          # Layer 4: Your code
CMD ["python3", "/app/app.py"]  # Metadata (no new layer)
```

```
Image Layers (read from bottom to top):
┌────────────────────────────┐
│ Layer 4: COPY app.py       │  (smallest — just your code)
├────────────────────────────┤
│ Layer 3: Install Python    │  (medium — Python packages)
├────────────────────────────┤
│ Layer 2: apt-get update    │  (small — package lists)
├────────────────────────────┤
│ Layer 1: Ubuntu 22.04      │  (largest — base OS)
└────────────────────────────┘
```

### Why Layers Matter:

1. **Caching** — If you change only your app code, Docker only rebuilds Layer 4. Layers 1-3 are cached. This makes builds **fast**.

2. **Sharing** — If 10 containers all use `ubuntu:22.04` as their base, that layer exists only **once on disk**. Huge space savings!

3. **Efficiency** — When pushing/pulling images, only layers that don't exist at the destination are transferred.

### Common Image Commands:

```bash
# List local images
docker images
docker image ls

# Pull an image
docker pull nginx:1.25

# Remove an image
docker rmi nginx:1.25
docker image rm nginx:1.25

# Inspect image details
docker image inspect nginx:1.25

# See image layers and sizes
docker history nginx:1.25

# Search for images on Docker Hub
docker search nginx

# Tag an image (create an alias)
docker tag nginx:1.25 myregistry.com/nginx:1.25

# Build an image from a Dockerfile
docker build -t myapp:v1 .
```

### Image ID vs Tag vs Digest:

```
┌──────────────────────────────────────────────────────┐
│  Image Reference Types                                │
│                                                       │
│  TAG:    nginx:1.25                                   │
│          Human-readable. Mutable (can point to        │
│          different images over time).                  │
│          ⚠️ "latest" is NOT always the latest!        │
│                                                       │
│  ID:     sha256:a1b2c3d4e5f6...                      │
│          Short form: a1b2c3d4e5f6                     │
│          Unique identifier of the image locally.      │
│                                                       │
│  DIGEST: nginx@sha256:abc123def456...                │
│          Content-addressable. Immutable. ALWAYS       │
│          points to the exact same image.              │
│          Best for production! 🏆                      │
└──────────────────────────────────────────────────────┘
```

## 6️⃣ Docker Containers — The Running Instance

A container is a **running instance of an image**. If an image is a class, a container is an object. If an image is a recipe, a container is the dish.

### Container = Image + Writable Layer

When you create a container from an image, Docker adds a thin **writable layer** on top:

```
┌─────────────────────────────────┐
│    Writable Layer (Container)   │  ← New files, modified files go here
├─────────────────────────────────┤
│    Image Layer 4 (Read-only)    │
├─────────────────────────────────┤
│    Image Layer 3 (Read-only)    │
├─────────────────────────────────┤
│    Image Layer 2 (Read-only)    │
├─────────────────────────────────┤
│    Image Layer 1 (Read-only)    │
└─────────────────────────────────┘
```

**Important:** When a container modifies a file from a read-only image layer, it uses **Copy-on-Write (CoW)**:
1. The file is copied from the image layer to the writable layer
2. The modification is made on the copy
3. The original image layer is untouched

This means:
- **Multiple containers can share the same image layers** (read-only)
- **Each container has its own writable layer** (isolated changes)
- **When a container is deleted, its writable layer is deleted** (data is lost unless you use volumes!)

### Container Lifecycle:

```
Created ──── Started (Running) ──── Paused ──── Unpaused (Running)
   │              │                                    │
   │              └──── Stopped (Exited) ──────────────┘
   │                         │
   └─────────────────────────┘
                             │
                        Removed (Deleted)
```

| State | Description |
|-------|-------------|
| **Created** | Container exists but hasn't started yet |
| **Running** | Container's main process is actively executing |
| **Paused** | Container's processes are frozen (using cgroups freezer) |
| **Stopped/Exited** | Container's main process has stopped. Container still exists (and can be restarted) |
| **Removed** | Container is deleted. Writable layer is gone. Cannot be recovered |

### Common Container Commands:

```bash
# Create and start a container
docker run -d --name web -p 80:80 nginx
# -d: detached (run in background)
# --name: give it a friendly name
# -p: port mapping (host:container)

# List running containers
docker ps

# List ALL containers (including stopped)
docker ps -a

# Stop a container (sends SIGTERM, then SIGKILL after 10s)
docker stop web

# Start a stopped container
docker start web

# Restart a container
docker restart web

# Pause a container (freeze processes)
docker pause web

# Unpause a container
docker unpause web

# Remove a container (must be stopped first)
docker rm web

# Force remove a running container
docker rm -f web

# View container logs
docker logs web
docker logs -f web  # Follow (like tail -f)

# Execute a command inside a running container
docker exec -it web bash
# -i: interactive (keep STDIN open)
# -t: allocate a pseudo-TTY
# This gives you a shell inside the container!

# Inspect container details (JSON output)
docker inspect web

# View container resource usage
docker stats
docker stats web

# Copy files between host and container
docker cp ./index.html web:/usr/share/nginx/html/
docker cp web:/var/log/nginx/access.log ./
```

## 7️⃣ Docker Volumes and Networks (Brief Overview)

We'll cover these in detail in sections 9.8 and 9.9, but here's a quick overview since they're part of the architecture.

### Volumes — Persistent Data

By default, data inside a container is **lost when the container is deleted** (remember, the writable layer gets deleted). Volumes solve this:

```
┌──────────────────────────────┐
│         Container            │
│  ┌────────────────────────┐  │
│  │  /app/data  ──────────────────── Volume (on host)
│  │                        │  │     /var/lib/docker/volumes/
│  │  Data persists even if │  │     my-volume/_data/
│  │  container is deleted! │  │
│  └────────────────────────┘  │
└──────────────────────────────┘
```

```bash
# Create a volume
docker volume create my-data

# Use a volume when running a container
docker run -d -v my-data:/app/data nginx
```

### Networks — Container Communication

Docker creates virtual networks so containers can talk to each other:

```
┌──────────────────────────────────────────────┐
│           Docker Network (bridge)             │
│                                              │
│  ┌─────────────┐        ┌─────────────┐     │
│  │  Container A │◀──────▶│  Container B │     │
│  │  172.17.0.2  │        │  172.17.0.3  │     │
│  │  (web)       │        │  (database)  │     │
│  └─────────────┘        └─────────────┘     │
│                                              │
└──────────────────────────────────────────────┘
```

```bash
# Create a network
docker network create my-network

# Run containers on the same network
docker run -d --name web --network my-network nginx
docker run -d --name db --network my-network postgres

# Containers can reach each other by name!
# From inside 'web': curl http://db:5432
```

## 8️⃣ Putting It All Together — What Happens When You Run `docker run`?

Let's trace the **complete journey** of `docker run -d -p 80:80 nginx`:

```
Step 1: You type the command
  └── docker run -d -p 80:80 nginx

Step 2: Docker Client parses the command
  └── Converts it to a REST API call
  └── Sends to Docker Daemon via /var/run/docker.sock

Step 3: Docker Daemon receives the request
  └── Checks if the image "nginx:latest" exists locally
  └── If NOT → goes to Step 4
  └── If YES → goes to Step 5

Step 4: Daemon pulls the image from Docker Hub
  └── Contacts registry.docker.io
  └── Downloads the image manifest
  └── Downloads each layer (only layers not already cached)
  └── Stores layers in /var/lib/docker/overlay2/
  └── Image is now available locally

Step 5: Daemon tells containerd to create a container
  └── containerd creates a container bundle (config + rootfs)
  └── Container gets a unique ID (e.g., abc123def456)

Step 6: containerd tells runc to start the container
  └── runc creates Linux namespaces (PID, NET, MNT, UTS, IPC)
  └── runc sets up cgroups (resource limits)
  └── runc sets up the container's root filesystem (union mount of image layers)
  └── runc starts the container's main process (nginx, in this case)
  └── runc exits — the container process is now managed by containerd

Step 7: Daemon sets up networking
  └── Creates a virtual ethernet pair (veth)
  └── Connects one end to the container, one to the docker0 bridge
  └── Assigns the container an IP address (e.g., 172.17.0.2)
  └── Sets up port mapping: host:80 → container:80 (using iptables)

Step 8: Container is running!
  └── nginx is serving requests on port 80
  └── Daemon returns the container ID to the client
  └── You see: "abc123def456789..."
```

Here's the same thing in a simpler visual:

```
docker run nginx
       │
       ▼
  Docker Client  ──(REST API)──▶  Docker Daemon
                                       │
                                       ├── Image exists locally? 
                                       │   NO → Pull from Docker Hub
                                       │   YES → Continue
                                       │
                                       ▼
                                  containerd
                                       │
                                       ▼
                                     runc
                                       │
                                       ├── Create namespaces
                                       ├── Set up cgroups
                                       ├── Mount filesystem
                                       ├── Start process
                                       │
                                       ▼
                                 🏃 Container Running!
                                   (nginx serving traffic)
```

## 9️⃣ Docker Storage — Where Everything Lives

On a Linux host, Docker stores everything under `/var/lib/docker/`:

```
/var/lib/docker/
├── overlay2/          # Image and container layers (default storage driver)
│   ├── abc123.../     # Layer 1
│   ├── def456.../     # Layer 2
│   └── ...
├── containers/        # Container-specific data
│   ├── <container-id>/
│   │   ├── config.v2.json   # Container configuration
│   │   ├── hostconfig.json  # Host-specific config
│   │   └── <container-id>-json.log  # Container logs!
│   └── ...
├── image/             # Image metadata
│   └── overlay2/
│       ├── imagedb/   # Image database
│       └── layerdb/   # Layer database
├── volumes/           # Named volumes
│   ├── my-volume/
│   │   └── _data/     # Actual volume data
│   └── ...
├── network/           # Network configuration
├── plugins/           # Docker plugins
├── tmp/               # Temporary files
└── buildkit/          # BuildKit cache and data
```

### Storage Drivers:

Docker uses a **storage driver** to manage image layers and the writable container layer:

| Driver | Status | When to Use |
|--------|--------|-------------|
| **overlay2** | ✅ Recommended | Default on modern Linux. Best performance. Use this. |
| **btrfs** | Supported | If your host uses btrfs filesystem |
| **zfs** | Supported | If your host uses ZFS filesystem |
| **devicemapper** | Deprecated | Old. Don't use. |
| **aufs** | Deprecated | Old. Used in early Docker. Don't use. |
| **vfs** | Testing only | No copy-on-write. Very slow. For testing only. |

## 🔑 Key Takeaways

```
✅ Docker uses a client-server architecture
✅ Docker Client (CLI) → sends REST API calls → Docker Daemon (dockerd)
✅ Docker Daemon manages images, containers, volumes, and networks
✅ Under the hood: dockerd → containerd → runc → Linux kernel
✅ Docker Registry stores and distributes container images
✅ Images are read-only templates made of layers
✅ Containers are running instances of images with a writable layer
✅ Container data is ephemeral — use volumes for persistence
✅ The daemon runs as root — understand the security implications
✅ OCI standards ensure interoperability between different container tools
✅ Kubernetes uses containerd directly (doesn't need Docker!)
```

## 🎯 Interview Questions You Should Be Able to Answer

1. **Explain Docker's architecture.**
   → Client-server model. Docker CLI sends REST API calls to dockerd (daemon). The daemon manages images, containers, volumes, and networks. Under the hood, it uses containerd and runc.

2. **What's the difference between Docker daemon, containerd, and runc?**
   → dockerd is the high-level manager. containerd manages container lifecycle and image operations. runc is the low-level OCI runtime that actually creates namespaces/cgroups and starts the container process.

3. **What happens when you type `docker run nginx`?**
   → Client sends API call to daemon → daemon checks for image locally → pulls if needed → containerd creates container → runc sets up namespaces/cgroups/filesystem → starts the process → networking is configured → container is running.

4. **What is a Docker Registry?**
   → A storage and distribution system for container images. Docker Hub is the default public registry. Private options include ECR, GCR, ACR, and Harbor.

5. **Why does the Docker daemon run as root?**
   → Because it needs to create namespaces, manage networking (iptables), mount filesystems, and manage cgroups — all of which require root privileges. This is why Docker socket access equals root access.

6. **What is the Docker socket and why is it important?**
   → `/var/run/docker.sock` is the Unix socket the daemon listens on. Anyone with access to this socket can control Docker and effectively has root access to the host.

7. **Can Kubernetes use Docker?**
   → Kubernetes deprecated the Docker shim in v1.24. It now uses containerd or CRI-O directly. Docker images still work because they're OCI-compliant.
