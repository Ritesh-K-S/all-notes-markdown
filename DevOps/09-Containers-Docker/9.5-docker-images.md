# 9.5 — Docker Images — Pulling, Building, Tagging, Pushing

> **One Line Answer:** A Docker image is a **read-only template (blueprint)** that contains everything needed to run an application — the OS, dependencies, code, and configuration — and you can **pull** images from registries, **build** your own images, **tag** them with meaningful names/versions, and **push** them to registries for others to use.

---

## 🤔 The Big Picture — What Is a Docker Image?

Let's use a simple analogy. Think of Docker images like a **recipe card for a dish**:

- The **recipe** (image) has all the instructions and ingredients listed
- You can **cook** (create a container) from that recipe as many times as you want
- Each time you cook, you get a **separate dish** (container)
- The recipe itself never changes — it's **read-only**
- You can **share** the recipe with others (push to a registry)
- Others can **get** your recipe (pull from a registry)

```
┌─────────────────────────────────────────────────────────┐
│                    Docker Image                          │
│                  (Read-Only Blueprint)                    │
│                                                          │
│  ┌────────────────────────────────────────────────────┐  │
│  │ Layer 5: Your App Code (COPY . /app)               │  │
│  ├────────────────────────────────────────────────────┤  │
│  │ Layer 4: Install Dependencies (RUN npm install)    │  │
│  ├────────────────────────────────────────────────────┤  │
│  │ Layer 3: Set Working Dir (WORKDIR /app)            │  │
│  ├────────────────────────────────────────────────────┤  │
│  │ Layer 2: Install Node.js (RUN apt install nodejs)  │  │
│  ├────────────────────────────────────────────────────┤  │
│  │ Layer 1: Base OS — Ubuntu (FROM ubuntu:24.04)      │  │
│  └────────────────────────────────────────────────────┘  │
│                                                          │
│  → Each layer is CACHED and REUSABLE                     │
│  → Layers are STACKED on top of each other               │
│  → The whole thing is READ-ONLY                          │
└─────────────────────────────────────────────────────────┘
          │                    │                    │
          ▼                    ▼                    ▼
    ┌──────────┐        ┌──────────┐        ┌──────────┐
    │Container1│        │Container2│        │Container3│
    │(Writable)│        │(Writable)│        │(Writable)│
    └──────────┘        └──────────┘        └──────────┘

Each container adds its OWN thin writable layer on top of the image.
```

### Key Things to Remember

1. **Images are made of LAYERS** — each instruction in a Dockerfile creates a new layer
2. **Layers are CACHED** — if a layer hasn't changed, Docker reuses it (super fast!)
3. **Images are READ-ONLY** — when a container runs, it adds a thin writable layer on top
4. **Images are SHARED** — multiple containers can share the same image layers

## 1️⃣ Understanding Image Names — The Naming Convention

Before we start pulling and pushing, let's understand how Docker image names work:

```
[registry/][username/]repository[:tag][@digest]
```

Let's break it down:

```
docker.io/library/nginx:1.27.0
├────────┤ ├─────┤ ├───┤ ├───┤
│         │       │      │
│         │       │      └── Tag (version)
│         │       └── Repository (image name)
│         └── Username/Organization (optional for official images)
└── Registry (default: docker.io = Docker Hub)
```

### Examples

| Full Name | Registry | Username | Repository | Tag |
|-----------|----------|----------|------------|-----|
| `nginx` | docker.io (default) | library (default) | nginx | latest (default) |
| `nginx:1.27` | docker.io | library | nginx | 1.27 |
| `myuser/myapp:v2.0` | docker.io | myuser | myapp | v2.0 |
| `ghcr.io/myorg/myapp:latest` | ghcr.io (GitHub) | myorg | myapp | latest |
| `123456.dkr.ecr.us-east-1.amazonaws.com/myapp:prod` | AWS ECR | — | myapp | prod |

### The `latest` Tag — A Common Gotcha!

```bash
# These two are EXACTLY the same
docker pull nginx
docker pull nginx:latest
```

⚠️ **IMPORTANT:** `latest` doesn't mean "the newest version"! It's just a tag name. It's the **default tag** when no tag is specified. In practice, many teams update `latest` to point to the newest version, but that's just a convention — not guaranteed.

**Best practice:** Always use specific tags in production!

```bash
# Bad — you don't know which version you'll get
docker pull nginx:latest

# Good — you know exactly what you're getting
docker pull nginx:1.27.0
```

## 2️⃣ Pulling Images — `docker pull`

### What It Does

`docker pull` **downloads an image** from a registry (default: Docker Hub) to your local machine.

### Basic Syntax

```bash
docker pull [OPTIONS] NAME[:TAG|@DIGEST]
```

### Pulling Images from Docker Hub

```bash
# Pull the latest Nginx image
docker pull nginx

# Pull a specific version
docker pull nginx:1.27.0

# Pull a lightweight Alpine-based version
docker pull nginx:alpine

# Pull Ubuntu 24.04
docker pull ubuntu:24.04

# Pull a specific Python version
docker pull python:3.12-slim
```

**Output:**
```
1.27.0: Pulling from library/nginx
a2abf6c4d29d: Pull complete        ← Layer 1 downloaded
a9edb18cadd1: Pull complete        ← Layer 2 downloaded
589b7251471a: Pull complete        ← Layer 3 downloaded
186b1aaa4aa6: Pull complete        ← Layer 4 downloaded
b4df32aa5a72: Pull complete        ← Layer 5 downloaded
a0bcbecc962e: Pull complete        ← Layer 6 downloaded
Digest: sha256:abc123def456...     ← Unique hash of this exact image
Status: Downloaded newer image for nginx:1.27.0
docker.io/library/nginx:1.27.0
```

### Understanding What Happens When You Pull

```
Your Machine                          Docker Hub (Registry)
┌──────────────┐                     ┌──────────────────────┐
│              │  docker pull nginx   │                      │
│  Docker      │ ──────────────────→ │   nginx:1.27.0       │
│  Daemon      │                     │   ┌──────────┐       │
│              │ ←────────────────── │   │  Layer 6  │       │
│  Downloads   │   Layer by layer    │   │  Layer 5  │       │
│  and stores  │                     │   │  Layer 4  │       │
│  locally     │                     │   │  Layer 3  │       │
│              │                     │   │  Layer 2  │       │
│              │                     │   │  Layer 1  │       │
│              │                     │   └──────────┘       │
└──────────────┘                     └──────────────────────┘
```

### Pulling from Other Registries

Docker Hub isn't the only registry! You can pull from any registry:

```bash
# Pull from GitHub Container Registry
docker pull ghcr.io/my-org/my-app:v1.0

# Pull from AWS ECR
docker pull 123456789.dkr.ecr.us-east-1.amazonaws.com/my-app:latest

# Pull from Google Container Registry
docker pull gcr.io/my-project/my-app:v2.0

# Pull from Azure Container Registry
docker pull myregistry.azurecr.io/my-app:latest

# Pull from a self-hosted private registry
docker pull registry.mycompany.com/my-app:v1.0
```

### Pull All Tags for an Image

```bash
# Pull ALL available tags of an image (usually not recommended — uses lots of space)
docker pull -a nginx
```

### Check If an Image Exists Without Pulling

```bash
# Just check the manifest (metadata) without downloading
docker manifest inspect nginx:1.27.0
```

### Smart Layer Sharing

Docker is really clever about layers. If two images share layers, Docker only downloads each layer **once**:

```bash
# Pull nginx (downloads layers A, B, C, D, E, F)
docker pull nginx:1.27.0

# Pull nginx:alpine (layers A and B are shared → only downloads new layers)
docker pull nginx:alpine
```

```
nginx:1.27.0                    nginx:alpine
┌────────────┐                 ┌────────────┐
│   Layer F  │                 │   Layer X  │  ← New, download needed
│   Layer E  │                 │   Layer W  │  ← New, download needed
│   Layer D  │                 │   Layer B  │  ← Already have it! Skip!
│   Layer C  │                 │   Layer A  │  ← Already have it! Skip!
│   Layer B  │                 └────────────┘
│   Layer A  │
└────────────┘
```

## 3️⃣ Building Images — `docker build`

### What It Does

`docker build` creates a **new image** from a `Dockerfile` (a text file with build instructions). This is how you package YOUR application into a Docker image.

> We'll cover Dockerfiles in detail in [section 9.6](9.6-dockerfile.md). Here we'll focus on the `docker build` command itself.

### Basic Syntax

```bash
docker build [OPTIONS] PATH
```

The `PATH` is the **build context** — usually `.` (current directory), which tells Docker where to find the Dockerfile and any files it needs.

### Your First Build

Let's create a simple example. First, create a project directory:

```bash
mkdir my-app
cd my-app
```

Create a simple `index.html`:

```html
<!DOCTYPE html>
<html>
<head><title>My App</title></head>
<body><h1>Hello from Docker!</h1></body>
</html>
```

Create a `Dockerfile`:

```dockerfile
FROM nginx:alpine
COPY index.html /usr/share/nginx/html/index.html
```

Now build the image:

```bash
docker build -t my-app:v1.0 .
```

Let's break this down:
- `docker build` → the build command
- `-t my-app:v1.0` → **tag** the image as `my-app` with version `v1.0`
- `.` → build context is the current directory

**Output:**
```
[+] Building 2.5s (7/7) FINISHED
 => [internal] load build definition from Dockerfile             0.0s
 => [internal] load .dockerignore                                0.0s
 => [internal] load metadata for docker.io/library/nginx:alpine  1.2s
 => [1/2] FROM docker.io/library/nginx:alpine@sha256:abc123...  0.0s
 => [internal] load build context                                0.0s
 => [2/2] COPY index.html /usr/share/nginx/html/index.html      0.1s
 => exporting to image                                           0.1s
 => => naming to docker.io/library/my-app:v1.0                  0.0s
```

Your image is built! Let's run it:

```bash
docker run -d --name my-website -p 8080:80 my-app:v1.0
```

Visit `http://localhost:8080` and you'll see "Hello from Docker!"

### Important `docker build` Options

```bash
# Specify a different Dockerfile name
docker build -t my-app:v1.0 -f Dockerfile.prod .

# Build with no cache (forces rebuild of all layers)
docker build --no-cache -t my-app:v1.0 .

# Pass build arguments
docker build --build-arg NODE_ENV=production -t my-app:v1.0 .

# Set the target stage in a multi-stage build
docker build --target builder -t my-app:builder .

# Build for a specific platform
docker build --platform linux/amd64 -t my-app:v1.0 .
```

### Understanding the Build Context

The `.` at the end of `docker build` is the **build context**. Docker sends this entire directory to the Docker daemon.

```bash
# Current directory as build context
docker build -t my-app .

# A specific directory as build context
docker build -t my-app ./path/to/project

# Using a URL (Git repo) as build context
docker build -t my-app https://github.com/user/repo.git
```

⚠️ **Important:** Don't put unnecessary files in your build context! Docker sends the **entire directory** to the daemon. Use a `.dockerignore` file to exclude files:

```
# .dockerignore
node_modules
.git
*.md
.env
tests/
```

### Viewing the Build Cache

```bash
# See how much space the build cache is using
docker builder prune

# Check build cache usage
docker system df
```

### A More Complete Build Example

```bash
# A real-world Node.js application build
docker build \
  -t my-company/api-server:v2.1.0 \
  -t my-company/api-server:latest \
  --build-arg NODE_ENV=production \
  --build-arg BUILD_DATE=$(date -u +'%Y-%m-%dT%H:%M:%SZ') \
  --no-cache \
  .
```

This:
- Tags the image with both a specific version AND `latest`
- Passes build arguments
- Rebuilds everything from scratch (no cache)

## 4️⃣ Tagging Images — `docker tag`

### What It Does

`docker tag` creates a **new name (alias)** for an existing image. It doesn't copy the image — it just adds another "pointer" to the same image data.

Think of it like giving someone a **nickname** — they don't become a different person, they just have another name.

### Basic Syntax

```bash
docker tag SOURCE_IMAGE[:TAG] TARGET_IMAGE[:TAG]
```

### Examples

```bash
# Give an image a new version tag
docker tag my-app:v1.0 my-app:v1

# Also tag it as latest
docker tag my-app:v1.0 my-app:latest

# Tag for pushing to Docker Hub
docker tag my-app:v1.0 myusername/my-app:v1.0

# Tag for pushing to a private registry
docker tag my-app:v1.0 registry.mycompany.com/my-app:v1.0

# Tag for pushing to AWS ECR
docker tag my-app:v1.0 123456789.dkr.ecr.us-east-1.amazonaws.com/my-app:v1.0

# Tag for pushing to GitHub Container Registry
docker tag my-app:v1.0 ghcr.io/my-org/my-app:v1.0
```

### How Tagging Works Visually

```
BEFORE tagging:
┌──────────────────────────┐
│ Image ID: abc123def456   │
│ Tags:                    │
│   - my-app:v1.0          │
└──────────────────────────┘

$ docker tag my-app:v1.0 my-app:latest
$ docker tag my-app:v1.0 myusername/my-app:v1.0

AFTER tagging:
┌──────────────────────────────────────┐
│ Image ID: abc123def456               │  ← SAME image!
│ Tags:                                │
│   - my-app:v1.0                      │  ← Original tag
│   - my-app:latest                    │  ← New alias
│   - myusername/my-app:v1.0           │  ← Ready for Docker Hub
└──────────────────────────────────────┘
```

You can verify this:

```bash
docker images
```

**Output:**
```
REPOSITORY            TAG       IMAGE ID       CREATED       SIZE
my-app                v1.0      abc123def456   5 min ago     43MB
my-app                latest    abc123def456   5 min ago     43MB  ← Same ID!
myusername/my-app     v1.0      abc123def456   5 min ago     43MB  ← Same ID!
```

All three tags point to the **exact same image** (same IMAGE ID).

### Tagging Strategies (Best Practices)

#### Semantic Versioning Tags

```bash
# When you release version 2.1.3:
docker tag my-app:build-123 my-app:2.1.3    # Exact version
docker tag my-app:build-123 my-app:2.1      # Minor version
docker tag my-app:build-123 my-app:2        # Major version
docker tag my-app:build-123 my-app:latest   # Latest
```

This way users can choose their precision:
- `my-app:2.1.3` → "I want exactly this version"
- `my-app:2.1` → "Give me the latest patch of 2.1"
- `my-app:2` → "Give me the latest 2.x"
- `my-app:latest` → "Give me the newest"

#### Environment Tags

```bash
docker tag my-app:build-123 my-app:dev
docker tag my-app:build-123 my-app:staging
docker tag my-app:build-123 my-app:production
```

#### Git Commit Tags

```bash
# Tag with the Git commit hash
docker tag my-app:build-123 my-app:$(git rev-parse --short HEAD)
# Result: my-app:a1b2c3d
```

## 5️⃣ Pushing Images — `docker push`

### What It Does

`docker push` **uploads your image** to a registry (like Docker Hub, ECR, GCR, etc.) so that others can pull and use it.

### Basic Syntax

```bash
docker push NAME[:TAG]
```

### Step-by-Step: Push to Docker Hub

#### Step 1: Create a Docker Hub Account

Go to [hub.docker.com](https://hub.docker.com) and create a free account (e.g., username: `myusername`).

#### Step 2: Login from the CLI

```bash
docker login
```

**Output:**
```
Login with your Docker ID to push and pull images from Docker Hub.
Username: myusername
Password: ********
Login Succeeded
```

#### Step 3: Tag Your Image with Your Username

```bash
# Your image must be tagged as: username/repository:tag
docker tag my-app:v1.0 myusername/my-app:v1.0
```

#### Step 4: Push!

```bash
docker push myusername/my-app:v1.0
```

**Output:**
```
The push refers to repository [docker.io/myusername/my-app]
5f70bf18a086: Pushed         ← Layer 1 uploaded
a1b2c3d4e5f6: Pushed         ← Layer 2 uploaded
b2c3d4e5f6g7: Mounted        ← Layer already exists in registry, reused!
v1.0: digest: sha256:abc123... size: 1234
```

Notice `Mounted` — Docker is smart! If a layer already exists in the registry (from a base image like `nginx:alpine`), it doesn't upload it again.

### Push Multiple Tags

```bash
# Tag and push multiple versions
docker tag my-app:v1.0 myusername/my-app:v1.0
docker tag my-app:v1.0 myusername/my-app:latest

docker push myusername/my-app:v1.0
docker push myusername/my-app:latest

# Or push all tags at once
docker push --all-tags myusername/my-app
```

### Push to Other Registries

#### GitHub Container Registry (ghcr.io)

```bash
# Login
echo $GITHUB_TOKEN | docker login ghcr.io -u USERNAME --password-stdin

# Tag
docker tag my-app:v1.0 ghcr.io/my-org/my-app:v1.0

# Push
docker push ghcr.io/my-org/my-app:v1.0
```

#### AWS ECR (Elastic Container Registry)

```bash
# Login (using AWS CLI)
aws ecr get-login-password --region us-east-1 | \
  docker login --username AWS --password-stdin 123456789.dkr.ecr.us-east-1.amazonaws.com

# Tag
docker tag my-app:v1.0 123456789.dkr.ecr.us-east-1.amazonaws.com/my-app:v1.0

# Push
docker push 123456789.dkr.ecr.us-east-1.amazonaws.com/my-app:v1.0
```

#### Google Container Registry (GCR) / Artifact Registry

```bash
# Login
gcloud auth configure-docker

# Tag
docker tag my-app:v1.0 gcr.io/my-project/my-app:v1.0

# Push
docker push gcr.io/my-project/my-app:v1.0
```

#### Azure Container Registry (ACR)

```bash
# Login
az acr login --name myregistry

# Tag
docker tag my-app:v1.0 myregistry.azurecr.io/my-app:v1.0

# Push
docker push myregistry.azurecr.io/my-app:v1.0
```

```

│                 │         │ user/my-app:v1   │         │                      │
└─────────────────┘         └──────────────────┘         └──────────┬───────────┘
                                                                    │

                                                         ┌──────────────────────┐
                                                         │   Docker Registry    │
                                                         │   (Docker Hub/ECR/   │
                                                         │    GCR/ACR/etc.)     │
                                                         │                      │
                                                         │  Others can now:     │
                                                         │  docker pull         │
                                                         │  user/my-app:v1     │

```

Each row is a **layer**. You can see:
- The base `debian` layer (97.1MB)
- The Nginx installation layer (62.1MB)
- Several small COPY layers
- Configuration layers (0B — they only add metadata, not files)

### How Layer Caching Works

```dockerfile
FROM node:20-alpine           # Layer 1 — cached after first build
WORKDIR /app                  # Layer 2 — cached
COPY package.json .           # Layer 3 — cached IF package.json didn't change
RUN npm install               # Layer 4 — cached IF Layer 3 didn't change
COPY . .                      # Layer 5 — CHANGES every time you edit code
CMD ["node", "server.js"]     # Layer 6 — cached
```

```
First build:
  Layer 1: MISS → Download node:20-alpine          (slow)
  Layer 2: MISS → Create /app directory             (fast)
  Layer 3: MISS → Copy package.json                 (fast)
  Layer 4: MISS → Run npm install                   (slow!)
  Layer 5: MISS → Copy source code                  (fast)
  Layer 6: MISS → Set CMD                           (fast)

Second build (only code changed, not package.json):
  Layer 1: HIT  → Use cached layer                  ✓ instant
  Layer 2: HIT  → Use cached layer                  ✓ instant
  Layer 3: HIT  → package.json didn't change        ✓ instant
  Layer 4: HIT  → npm install cached!               ✓ instant (saved minutes!)
  Layer 5: MISS → Source code changed, rebuild       (fast)
  Layer 6: MISS → Must rebuild (after a MISS)        (fast)
```

This is why the **order of instructions in a Dockerfile matters** — put things that change less frequently (like installing dependencies) before things that change often (like copying source code).

## 🏷️ Image Digests — The Immutable Identifier

Tags can change (someone can push a new image with the same tag), but **digests never change**. A digest is a SHA256 hash of the image content.

```bash
# Pull by digest (guaranteed to get the exact same image every time)
docker pull nginx@sha256:abc123def456789...

# See the digest of a local image
docker inspect --format '{{.RepoDigests}}' nginx:latest
# Output: [nginx@sha256:abc123def456...]
```

### When to Use Digests

- **Production deployments** — when you need 100% reproducibility
- **Security-critical environments** — to ensure the exact image you tested is what gets deployed
- **CI/CD pipelines** — to pin exact versions

```bash
# In a Dockerfile, you can use digests too:
FROM nginx@sha256:abc123def456789...
```

## 🧹 Image Management Commands

### Listing Images

```bash
# List all images
docker images

# List images with a specific name
docker images nginx

# List only image IDs
docker images -q

# List images with their digests
docker images --digests
```

### Removing Images

```bash
# Remove a specific image
docker rmi nginx:1.27.0

# Remove an image by ID
docker rmi a1b2c3d4e5f6

# Force remove (even if containers are using it)
docker rmi -f nginx:latest

# Remove all dangling images (untagged)
docker image prune

# Remove ALL unused images
docker image prune -a

# Remove images older than 24 hours
docker image prune -a --filter "until=24h"
```

### Saving and Loading Images (Offline Transfer)

Sometimes you need to transfer images without a registry (e.g., air-gapped environments):

```bash
# Save an image to a tar file
docker save -o my-app-v1.tar my-app:v1.0
# This creates a file: my-app-v1.tar

# Transfer the file (USB, SCP, etc.)
scp my-app-v1.tar user@remote-server:/tmp/

# Load the image on the other machine
docker load -i /tmp/my-app-v1.tar
```

### Import/Export Containers (Not Images)

```bash
# Export a container's filesystem to a tar
docker export my-container > my-container.tar

# Import a tar as a new image
docker import my-container.tar my-imported-image:v1
```

**Difference:** `save/load` preserves image layers and metadata. `export/import` flattens everything into a single layer.

## 🔗 Complete Real-World Workflow

Here's how everything fits together in a typical DevOps workflow:

```bash
# 1. Write your application code
vim app.py

# 2. Write a Dockerfile
vim Dockerfile

# 3. Build the image
docker build -t my-app:v1.0 .

# 4. Test it locally
docker run -d --name test-app -p 3000:3000 my-app:v1.0
curl http://localhost:3000        # Test it works
docker logs test-app              # Check for errors

# 5. Everything works! Tag it for your registry
docker tag my-app:v1.0 myusername/my-app:v1.0
docker tag my-app:v1.0 myusername/my-app:latest

# 6. Login to the registry
docker login

# 7. Push to the registry
docker push myusername/my-app:v1.0
docker push myusername/my-app:latest

# 8. On the production server (or in Kubernetes)
docker pull myusername/my-app:v1.0
docker run -d --name production-app -p 80:3000 myusername/my-app:v1.0

# 9. Clean up locally
docker stop test-app
docker rm test-app
docker image prune
```

## 💡 Pro Tips & Best Practices

### 1. Always Use Specific Tags

```bash
# Bad — you never know what version you'll get
FROM python:latest

# Good — pinned to a specific version
FROM python:3.12.1-slim-bookworm
```

### 2. Use Slim / Alpine Variants

```bash
# Full image — 1.1 GB
docker pull python:3.12

# Slim variant — 150 MB
docker pull python:3.12-slim

# Alpine variant — 55 MB
docker pull python:3.12-alpine
```

### 3. Check Image Vulnerabilities

```bash
# Docker Scout (built into Docker Desktop)
docker scout cves my-app:v1.0

# Trivy (popular open-source scanner)
trivy image my-app:v1.0
```

### 4. Use Multi-Platform Images

```bash
# Build for multiple architectures
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  -t myuser/my-app:v1.0 \
  --push \
  .
```

### 5. Use `.dockerignore` to Keep Images Small

```
# .dockerignore
node_modules
.git
.env
*.md
tests/
coverage/
.vscode/
```

## 📝 Summary

| Concept | Command | What It Does |
|---------|---------|-------------|
| **Pull** | `docker pull nginx:1.27` | Downloads an image from a registry |
| **Build** | `docker build -t my-app:v1 .` | Creates an image from a Dockerfile |
| **Tag** | `docker tag my-app:v1 user/my-app:v1` | Adds a new name/alias to an image |
| **Push** | `docker push user/my-app:v1` | Uploads an image to a registry |
| **List** | `docker images` | Shows all local images |
| **Remove** | `docker rmi nginx:1.27` | Deletes a local image |
| **History** | `docker history nginx` | Shows image layers |
| **Inspect** | `docker inspect nginx` | Shows detailed image info |
| **Save** | `docker save -o file.tar my-app:v1` | Exports image to a tar file |
| **Load** | `docker load -i file.tar` | Imports image from a tar file |

**The Image Lifecycle:** `build` → `tag` → `push` → (on another machine) → `pull` → `run`
