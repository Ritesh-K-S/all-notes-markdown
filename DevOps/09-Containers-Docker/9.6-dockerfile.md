# 9.6 — Dockerfile — FROM, RUN, COPY, ADD, CMD, ENTRYPOINT, WORKDIR, ENV, ARG, EXPOSE

## 🤔 The Big Picture — What Is a Dockerfile?

Imagine you're writing a **recipe** for someone who has never cooked before. You need to be very specific:

1. Start with a clean kitchen (base image)
2. Get these ingredients (install dependencies)
3. Put the food here (copy files)
4. Set the oven to 350°F (environment variables)
5. Cook for 30 minutes (run commands)
6. Serve on this plate (expose port)
7. This is how you eat it (startup command)

A Dockerfile is exactly this — a recipe for building a Docker image.

```dockerfile
# This is a Dockerfile — each line is an instruction
FROM python:3.12-slim          # 1. Start with Python installed
WORKDIR /app                   # 2. Create and go to /app directory
COPY requirements.txt .        # 3. Copy the requirements file
RUN pip install -r requirements.txt  # 4. Install dependencies
COPY . .                       # 5. Copy all application code
EXPOSE 5000                    # 6. Document that the app uses port 5000
CMD ["python", "app.py"]       # 7. Start the application
```

```
Dockerfile                                     Docker Image
┌──────────────────────────┐                   ┌──────────────────────────┐
│ FROM python:3.12-slim    │ ─── builds ──→    │ Layer 1: Python 3.12     │
│ WORKDIR /app             │                   │ Layer 2: /app directory  │
│ COPY requirements.txt .  │                   │ Layer 3: requirements    │
│ RUN pip install ...      │                   │ Layer 4: dependencies    │
│ COPY . .                 │                   │ Layer 5: app code        │
│ EXPOSE 5000              │                   │ Metadata: port 5000     │
│ CMD ["python", "app.py"] │                   │ Metadata: start command  │
└──────────────────────────┘                   └──────────────────────────┘
                                                         │
                                                         ▼
                                                   ┌──────────┐
                                                   │Container │
                                                   │ Running! │
                                                   └──────────┘
```

## 📋 Quick Reference — All Dockerfile Instructions

| Instruction | What It Does | Example |
|-------------|-------------|---------|
| `FROM` | Sets the base image | `FROM ubuntu:24.04` |
| `RUN` | Executes a command during build | `RUN apt-get install -y nginx` |
| `COPY` | Copies files from host to image | `COPY app.py /app/` |
| `ADD` | Like COPY, but with extra features | `ADD archive.tar.gz /app/` |
| `CMD` | Default command when container starts | `CMD ["python", "app.py"]` |
| `ENTRYPOINT` | Fixed command that always runs | `ENTRYPOINT ["python"]` |
| `WORKDIR` | Sets the working directory | `WORKDIR /app` |
| `ENV` | Sets environment variables | `ENV NODE_ENV=production` |
| `ARG` | Build-time variables | `ARG VERSION=1.0` |
| `EXPOSE` | Documents which port the app uses | `EXPOSE 3000` |

Now let's learn each one in detail!

## 1️⃣ `FROM` — The Starting Point

### What It Does

`FROM` is always the **FIRST instruction** in a Dockerfile (except for `ARG` which can come before it). It tells Docker which **base image** to start from.

Think of it like: "I want to start with a computer that already has _____ installed."

### Syntax

```dockerfile
FROM image[:tag] [AS name]
```

### Examples

```dockerfile
# Start with Ubuntu
FROM ubuntu:24.04

# Start with a slim Python environment
FROM python:3.12-slim

# Start with Node.js
FROM node:20-alpine

# Start with just a basic Alpine Linux (only 5MB!)
FROM alpine:3.19

# Start from scratch (completely empty — for advanced users)
FROM scratch
```

### Choosing the Right Base Image

```
┌─────────────────────────────────────────────────────────────────┐
│                    Base Image Decision Tree                      │
│                                                                  │
│  What language does your app use?                                │
│  ├── Python → python:3.12-slim (or python:3.12-alpine)          │
│  ├── Node.js → node:20-alpine                                   │
│  ├── Java → eclipse-temurin:21-jre-alpine                       │
│  ├── Go → golang:1.22-alpine (build) → scratch/alpine (run)     │
│  ├── .NET → mcr.microsoft.com/dotnet/aspnet:8.0                 │
│  └── Static files → nginx:alpine                                │
│                                                                  │
│  Want the smallest possible image?                               │
│  └── Use Alpine variants or Distroless images                    │
│                                                                  │
│  Need debugging tools in production?                             │
│  └── Use slim variants (not alpine — Alpine uses musl libc)      │
└─────────────────────────────────────────────────────────────────┘
```

### Image Size Comparison

```dockerfile
FROM python:3.12           # ~1.1 GB — Full Debian + Python + tools
FROM python:3.12-slim      # ~150 MB — Minimal Debian + Python
FROM python:3.12-alpine    # ~55 MB  — Alpine Linux + Python
```

**Rule of Thumb:**
- **Development:** Use `slim` (has debugging tools)
- **Production:** Use `alpine` or `slim` (smallest possible)
- **Avoid:** Using the full image (too large)

### Multi-Stage Builds with `AS`

You can use `FROM` multiple times in a Dockerfile. This is called **multi-stage builds**:

```dockerfile
# Stage 1: Build the application
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npm run build

# Stage 2: Run the application (much smaller!)
FROM nginx:alpine AS production
COPY --from=builder /app/dist /usr/share/nginx/html
```

The final image only contains the `nginx:alpine` base + your built files — not Node.js, not `node_modules`, nothing else! This makes the final image **tiny**.

## 2️⃣ `RUN` — Execute Commands During Build

### What It Does

`RUN` executes a command **during the image build process**. It's used to install packages, create directories, download files — anything you need to set up the environment.

Each `RUN` instruction creates a **new layer** in the image.

### Syntax — Two Forms

```dockerfile
# Shell form (runs in /bin/sh -c)
RUN command

# Exec form (runs directly, no shell)
RUN ["executable", "param1", "param2"]
```

### Examples

```dockerfile
# Install packages on Ubuntu/Debian
RUN apt-get update && apt-get install -y \
    curl \
    wget \
    vim \
    git

# Install packages on Alpine
RUN apk add --no-cache curl wget git

# Install Python dependencies
RUN pip install --no-cache-dir flask gunicorn requests

# Create a directory
RUN mkdir -p /app/logs

# Download a file
RUN curl -fsSL https://example.com/setup.sh | bash

# Change permissions
RUN chmod +x /app/entrypoint.sh

# Create a non-root user
RUN groupadd -r appuser && useradd -r -g appuser appuser
```

### ⚠️ CRITICAL Best Practice: Combine `RUN` Commands!

Every `RUN` creates a new layer. Too many layers = bigger image. Combine related commands with `&&`:

```dockerfile
# BAD — Creates 3 layers (and intermediate apt cache is stored in Layer 1!)
RUN apt-get update
RUN apt-get install -y curl wget
RUN rm -rf /var/lib/apt/lists/*

# GOOD — Creates 1 layer (and cleanup happens in the same layer)
RUN apt-get update && \
    apt-get install -y --no-install-recommends \
        curl \
        wget && \
    rm -rf /var/lib/apt/lists/*
```

**Why does this matter?**

```
BAD (3 layers):
┌────────────────────────────────────────────┐
│ Layer 3: rm -rf /var/lib/apt/lists/*       │ 0 MB    (files removed)
│ Layer 2: apt-get install curl wget         │ 50 MB   
│ Layer 1: apt-get update                    │ 30 MB   (apt cache still in THIS layer!)
└────────────────────────────────────────────┘
Total: 80 MB ← The cache from Layer 1 is STILL there!

GOOD (1 layer):
┌────────────────────────────────────────────┐
│ Layer 1: update + install + cleanup        │ 50 MB   (cache was cleaned in same layer)
└────────────────────────────────────────────┘
Total: 50 MB ← 30 MB saved!
```

In Docker, once data is written to a layer, **removing it in a later layer doesn't reduce the image size** — the data still exists in the earlier layer. That's why you must install AND clean up in the **same** `RUN` command.

### Real-World `RUN` Examples

#### Node.js Application

```dockerfile
# Install dependencies (production only)
RUN npm ci --only=production && \
    npm cache clean --force
```

#### Python Application

```dockerfile
# Install dependencies
RUN pip install --no-cache-dir --upgrade pip && \
    pip install --no-cache-dir -r requirements.txt
```

#### Java Application

```dockerfile
# Build the application with Maven
RUN mvn clean package -DskipTests && \
    mv target/app.jar /app/app.jar && \
    rm -rf target/ ~/.m2
```

## 3️⃣ `COPY` — Copy Files from Host to Image

### What It Does

`COPY` copies files and directories from your **build context** (your local machine) into the Docker image.

### Syntax

```dockerfile
COPY [--chown=user:group] source destination
```

### Examples

```dockerfile
# Copy a single file
COPY app.py /app/app.py

# Copy a file to the WORKDIR (current directory in the image)
COPY app.py .

# Copy everything from the build context to /app/
COPY . /app/

# Copy with specific ownership
COPY --chown=node:node package.json /app/

# Copy multiple specific files
COPY package.json package-lock.json ./

# Copy an entire directory
COPY src/ /app/src/
```

### How COPY Works Visually

```
Your Project (Build Context)          Docker Image
┌──────────────────────┐             ┌──────────────────────┐
│ my-project/          │             │ /app/                │
│ ├── app.py           │ ── COPY ─→ │ ├── app.py           │
│ ├── requirements.txt │             │ ├── requirements.txt │
│ ├── templates/       │             │ ├── templates/       │
│ │   └── index.html   │             │ │   └── index.html   │
│ ├── static/          │             │ ├── static/          │
│ │   └── style.css    │             │ │   └── style.css    │
│ ├── Dockerfile       │  (excluded) │ └── ...              │
│ └── .git/            │  (excluded) │                      │
└──────────────────────┘             └──────────────────────┘

Note: Dockerfile and .git are excluded because they should be
in your .dockerignore file!
```

### The COPY Order Trick for Better Caching

This is one of the **most important Dockerfile optimization techniques**:

```dockerfile
# BAD — Any file change invalidates the npm install cache
COPY . /app/
RUN npm install

# GOOD — Only re-run npm install when package files change
COPY package.json package-lock.json ./    # ← Changes rarely
RUN npm install                           # ← Cached when package files don't change!
COPY . .                                  # ← Changes often (your source code)
```

Why? Because Docker caches layers. If `package.json` hasn't changed, Docker reuses the cached `npm install` layer — **saving minutes on every build!**

## 4️⃣ `ADD` — Copy Files with Superpowers

### What It Does

`ADD` is similar to `COPY` but with two extra features:
1. It can **auto-extract tar archives** (`.tar`, `.tar.gz`, `.tar.bz2`, etc.)
2. It can **download files from URLs**

### Syntax

```dockerfile
ADD [--chown=user:group] source destination
```

### Examples

```dockerfile
# Auto-extract a tar archive into the image
ADD app.tar.gz /app/
# This extracts the contents of app.tar.gz into /app/

# Download a file from a URL (not recommended — see below)
ADD https://example.com/big-file.tar.gz /tmp/
```

### `COPY` vs `ADD` — When to Use What

| Feature | `COPY` | `ADD` |
|---------|--------|-------|
| Copy local files | ✅ | ✅ |
| Auto-extract archives | ❌ | ✅ |
| Download from URLs | ❌ | ✅ |
| **Recommended?** | ✅ Yes — default choice | ⚠️ Only when you need tar extraction |

```dockerfile
# USE COPY (recommended for most cases)
COPY app.py /app/
COPY requirements.txt /app/

# USE ADD (only when you need tar extraction)
ADD app.tar.gz /app/

# DON'T use ADD for downloading — use RUN + curl instead
# Bad:
ADD https://example.com/file.tar.gz /tmp/

# Good:
RUN curl -fsSL https://example.com/file.tar.gz | tar -xz -C /app/
```

**Why is `curl` better than `ADD` for downloads?**
- With `ADD`, the downloaded file becomes a permanent layer
- With `RUN curl`, you can download, extract, AND clean up in a single layer

### The Docker Best Practice Rule

> **Use `COPY` unless you specifically need `ADD`'s tar auto-extraction feature.**

## 5️⃣ `CMD` — The Default Startup Command

### What It Does

`CMD` sets the **default command** that runs when a container starts. It's the "what happens when someone types `docker run my-image`."

**Key point:** `CMD` can be **overridden** by the user when they run the container.

### Syntax — Three Forms

```dockerfile
# Exec form (RECOMMENDED)
CMD ["executable", "param1", "param2"]

# Shell form
CMD command param1 param2

# As default parameters for ENTRYPOINT
CMD ["param1", "param2"]
```

### Examples

```dockerfile
# Run a Python app
CMD ["python", "app.py"]

# Run a Node.js app
CMD ["node", "server.js"]

# Start Nginx
CMD ["nginx", "-g", "daemon off;"]

# Run a shell command
CMD ["sh", "-c", "echo Hello && exec python app.py"]
```

### CMD Can Be Overridden!

```dockerfile
# Dockerfile
FROM ubuntu:24.04
CMD ["echo", "Hello from the image!"]
```

```bash
# Run with default CMD
docker run my-image
# Output: Hello from the image!

# Override CMD at runtime
docker run my-image echo "I overrode the CMD!"
# Output: I overrode the CMD!

# Override with a shell
docker run -it my-image bash
# Now you're in a bash shell — CMD was completely replaced
```

### Exec Form vs Shell Form

```dockerfile
# Exec form — runs directly (RECOMMENDED)
CMD ["python", "app.py"]
# The process IS PID 1 → receives signals correctly (SIGTERM, etc.)

# Shell form — runs through /bin/sh -c
CMD python app.py
# Actually runs as: /bin/sh -c "python app.py"
# sh is PID 1, python is a child → signals may not reach your app!
```

**Why does PID 1 matter?**

```
Exec form:                           Shell form:
┌────────────────────┐               ┌────────────────────┐
│ Container          │               │ Container          │
│                    │               │                    │
│ PID 1: python app  │ ← gets       │ PID 1: /bin/sh     │ ← gets signal
│                    │   SIGTERM     │ └── python app     │ ← may NOT get signal!
└────────────────────┘               └────────────────────┘

When Docker sends SIGTERM (docker stop), you want YOUR app to receive it
so it can shut down gracefully (close connections, save state, etc.)
```

**Rule: Always use exec form `CMD ["python", "app.py"]` — not shell form `CMD python app.py`**

### Only ONE CMD!

If you have multiple `CMD` instructions, only the **last one** takes effect:

```dockerfile
CMD ["echo", "first"]
CMD ["echo", "second"]
CMD ["echo", "third"]    # Only THIS one runs!
```

## 6️⃣ `ENTRYPOINT` — The Fixed Command

### What It Does

`ENTRYPOINT` sets a command that **always runs** when the container starts. Unlike `CMD`, it **cannot be easily overridden** by the user.

### Syntax

```dockerfile
# Exec form (RECOMMENDED)
ENTRYPOINT ["executable", "param1"]

# Shell form
ENTRYPOINT command param1
```

### Basic Example

```dockerfile
FROM ubuntu:24.04
ENTRYPOINT ["echo", "Hello"]
```

```bash
# Run with no arguments
docker run my-image
# Output: Hello

# Run with arguments — they get APPENDED to ENTRYPOINT
docker run my-image World
# Output: Hello World

# Run with different arguments
docker run my-image Docker
# Output: Hello Docker
```

See the difference? With `ENTRYPOINT`, extra arguments are **appended**, not replacing.

### CMD vs ENTRYPOINT — The Key Difference

| | `CMD` | `ENTRYPOINT` |
|---|---|---|
| **Purpose** | Default command (can be overridden) | Fixed command (always runs) |
| **Override behavior** | Completely replaced | Arguments are appended |
| **When to use** | When users might want to run a different command | When the container IS the command |

### The Magic Combo: `ENTRYPOINT` + `CMD`

This is the **most powerful pattern** — use `ENTRYPOINT` for the fixed command and `CMD` for default arguments:

```dockerfile
FROM python:3.12-slim

# ENTRYPOINT = the program to run (fixed)
ENTRYPOINT ["python"]

# CMD = default arguments (can be overridden)
CMD ["app.py"]
```

```bash
# Run with default CMD → python app.py
docker run my-image

# Override CMD → python test.py
docker run my-image test.py

# Override CMD → python -c "print('hello')"
docker run my-image -c "print('hello')"
```

### Real-World Examples

#### A CLI Tool

```dockerfile
FROM alpine:3.19
RUN apk add --no-cache curl
ENTRYPOINT ["curl"]
CMD ["--help"]
```

```bash
docker run my-curl                          # curl --help
docker run my-curl https://example.com      # curl https://example.com
docker run my-curl -I https://example.com   # curl -I https://example.com
```

The container **IS** curl. The user just passes URLs and flags.

#### A Web Server with Custom Entrypoint Script

```dockerfile
FROM python:3.12-slim
WORKDIR /app
COPY . .
RUN pip install -r requirements.txt
COPY entrypoint.sh /entrypoint.sh
RUN chmod +x /entrypoint.sh
ENTRYPOINT ["/entrypoint.sh"]
CMD ["gunicorn", "app:app", "--bind", "0.0.0.0:5000"]
```

The `entrypoint.sh` can do setup tasks before running the main command:

```bash
#!/bin/bash
# entrypoint.sh

echo "Running database migrations..."
python manage.py migrate

echo "Collecting static files..."
python manage.py collectstatic --noinput

echo "Starting application..."
exec "$@"    # ← This runs whatever CMD is (gunicorn in this case)
```

**The `exec "$@"` trick:** `$@` contains all the arguments passed to the script (which is the `CMD`). `exec` replaces the shell process with the actual command, making it PID 1.

### Overriding ENTRYPOINT (When You Really Need To)

```bash
# Override ENTRYPOINT with --entrypoint flag
docker run --entrypoint bash my-image
docker run --entrypoint /bin/sh my-image
```

### Decision Table: When to Use What

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│  SCENARIO                        USE                             │
│  ─────────────────────────────────────────────────────────────   │
│  Simple app, standard startup    CMD ["python", "app.py"]        │
│                                                                  │
│  Container IS the tool           ENTRYPOINT ["curl"]             │
│  (curl, aws-cli, terraform)      CMD ["--help"]                  │
│                                                                  │
│  App needs setup before start    ENTRYPOINT ["./entrypoint.sh"]  │
│  (migrations, config)            CMD ["gunicorn", "app:app"]     │
│                                                                  │
│  User should be able to run      CMD ["python", "app.py"]        │
│  different commands easily                                       │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### Complete Interaction Table

| Dockerfile | `docker run my-image` | `docker run my-image world` |
|---|---|---|
| No CMD, no ENTRYPOINT | Error | Error |
| `CMD ["echo", "hello"]` | `echo hello` | `echo world` (CMD overridden) |
| `ENTRYPOINT ["echo", "hello"]` | `echo hello` | `echo hello world` (appended) |
| `ENTRYPOINT ["echo"]` + `CMD ["hello"]` | `echo hello` | `echo world` (CMD replaced) |

## 7️⃣ `WORKDIR` — Set the Working Directory

### What It Does

`WORKDIR` sets the **working directory** for all subsequent instructions (`RUN`, `CMD`, `ENTRYPOINT`, `COPY`, `ADD`). If the directory doesn't exist, Docker creates it automatically.

It's like `cd` — but permanent for the rest of the Dockerfile.

### Syntax

```dockerfile
WORKDIR /path/to/directory
```

### Examples

```dockerfile
# Set working directory to /app
WORKDIR /app

# Now all subsequent commands run from /app
COPY package.json .          # Copies to /app/package.json
RUN npm install              # Runs in /app
COPY . .                     # Copies everything to /app/
CMD ["node", "server.js"]    # Runs from /app → node /app/server.js
```

### Why Not Use `RUN cd`?

```dockerfile
# BAD — cd only works within that single RUN command!
RUN cd /app
RUN npm install    # ← This runs in / not in /app! cd was lost!

# GOOD — WORKDIR persists for all following instructions
WORKDIR /app
RUN npm install    # ← This runs in /app ✓
```

### Multiple WORKDIRs

You can use `WORKDIR` multiple times. Relative paths stack on top of each other:

```dockerfile
WORKDIR /app          # Now in /app
WORKDIR src           # Now in /app/src
WORKDIR utils         # Now in /app/src/utils

# Same as:
WORKDIR /app/src/utils
```

### Complete Example

```dockerfile
FROM node:20-alpine

# Create and switch to app directory
WORKDIR /app

# Copy dependency files first (for caching)
COPY package.json package-lock.json ./

# Install dependencies (runs in /app)
RUN npm ci --only=production

# Copy source code (into /app)
COPY . .

# Create logs directory
RUN mkdir -p logs

# The app starts from /app
CMD ["node", "server.js"]
```

## 8️⃣ `ENV` — Set Environment Variables

### What It Does

`ENV` sets **environment variables** that are available:
1. During the build process (in subsequent `RUN` instructions)
2. At runtime (when the container is running)

### Syntax

```dockerfile
# Single variable
ENV KEY=value

# Multiple variables (one per line)
ENV KEY1=value1
ENV KEY2=value2

# Old syntax (still works but not recommended)
ENV KEY value
```

### Examples

```dockerfile
# Set the application environment
ENV NODE_ENV=production

# Set application configuration
ENV APP_PORT=3000
ENV APP_HOST=0.0.0.0
ENV DB_HOST=localhost
ENV DB_PORT=5432

# Set PATH (common for adding custom binaries)
ENV PATH="/app/bin:${PATH}"

# Set timezone
ENV TZ=America/New_York

# Set language/locale
ENV LANG=C.UTF-8
```

### How ENV Works — Build Time AND Run Time

```dockerfile
FROM node:20-alpine

# This variable is available during build AND when container runs
ENV NODE_ENV=production

# RUN uses it during build
RUN echo "Building in $NODE_ENV mode"    # Prints: Building in production mode

# When container runs, the variable is still there
CMD ["sh", "-c", "echo Running in $NODE_ENV mode"]
# Prints: Running in production mode
```

### Overriding ENV at Runtime

```bash
# ENV is set in Dockerfile, but you can override it when running
docker run -e NODE_ENV=development my-app
# Now NODE_ENV=development inside the container
```

### Using ENV in Other Instructions

```dockerfile
ENV APP_HOME=/opt/myapp
WORKDIR ${APP_HOME}          # WORKDIR /opt/myapp
COPY . ${APP_HOME}           # COPY to /opt/myapp
RUN echo "App is in ${APP_HOME}"
```

### A Real-World Example

```dockerfile
FROM python:3.12-slim

# Python behavior settings
ENV PYTHONDONTWRITEBYTECODE=1       # Don't create .pyc files
ENV PYTHONUNBUFFERED=1              # Don't buffer stdout/stderr

# Application configuration
ENV APP_ENV=production
ENV APP_PORT=5000
ENV LOG_LEVEL=info
ENV WORKERS=4

WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .

EXPOSE ${APP_PORT}
CMD ["gunicorn", "app:app", "--bind", "0.0.0.0:5000", "--workers", "4"]
```

### ⚠️ Security Warning!

**Never put secrets (passwords, API keys, tokens) in ENV instructions!** They become part of the image and can be seen by anyone with access to the image:

```dockerfile
# NEVER DO THIS!
ENV DB_PASSWORD=supersecretpassword123
ENV API_KEY=sk-1234567890

# Instead, pass secrets at runtime:
# docker run -e DB_PASSWORD=secret my-app
# Or use Docker secrets, Vault, etc.
```

## 9️⃣ `ARG` — Build-Time Variables

### What It Does

`ARG` defines variables that are only available **during the build process**. Unlike `ENV`, they are NOT available when the container runs.

### Syntax

```dockerfile
# Declare with a default value
ARG VARIABLE_NAME=default_value

# Declare without a default (must be provided at build time)
ARG VARIABLE_NAME
```

### Examples

```dockerfile
# Define a build argument with a default
ARG NODE_VERSION=20
FROM node:${NODE_VERSION}-alpine

ARG APP_VERSION=1.0.0
RUN echo "Building version $APP_VERSION"

LABEL version=$APP_VERSION
```

### Passing ARG Values at Build Time

```bash
# Use the default value
docker build -t my-app .

# Override the ARG value
docker build --build-arg NODE_VERSION=18 -t my-app .
docker build --build-arg APP_VERSION=2.0.0 -t my-app .

# Multiple build args
docker build \
  --build-arg NODE_VERSION=20 \
  --build-arg APP_VERSION=2.0.0 \
  -t my-app .
```

### ARG vs ENV — The Difference

```
┌─────────────────────────────────────────────────────────────┐
│                                                              │
│  ARG                              ENV                        │
│  ───                              ───                        │
│  Available during BUILD only      Available during BUILD     │
│                                   AND at RUNTIME             │
│                                                              │
│  Set with --build-arg             Set in Dockerfile          │
│                                   Override with -e at run    │
│                                                              │
│  NOT in final image               IS in final image          │
│  (not visible in running          (visible via docker        │
│   container)                       inspect, env command)     │
│                                                              │
│  Use for: build-time choices      Use for: runtime config    │
│  (versions, flags)                (ports, hosts, modes)      │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### Converting ARG to ENV

Sometimes you want a build-time variable to also be available at runtime:

```dockerfile
# ARG is only available during build
ARG APP_VERSION=1.0.0

# Convert to ENV so it's available at runtime too
ENV APP_VERSION=${APP_VERSION}

# Now $APP_VERSION works both during build and when the container runs
RUN echo "Building version $APP_VERSION"
CMD ["sh", "-c", "echo Running version $APP_VERSION"]
```

### ARG Before FROM (Special Case)

You can use `ARG` before `FROM` to parameterize the base image:

```dockerfile
ARG PYTHON_VERSION=3.12
FROM python:${PYTHON_VERSION}-slim

# NOTE: ARGs declared before FROM are NOT available after FROM
# You need to re-declare them:
ARG PYTHON_VERSION
RUN echo "Using Python $PYTHON_VERSION"
```

### Practical Example: Configurable Build

```dockerfile
# Configurable base image version
ARG NODE_VERSION=20
FROM node:${NODE_VERSION}-alpine

# Configurable app metadata
ARG BUILD_DATE
ARG GIT_COMMIT
ARG APP_VERSION=latest

# Make version available at runtime
ENV APP_VERSION=${APP_VERSION}

LABEL build-date=${BUILD_DATE}
LABEL git-commit=${GIT_COMMIT}
LABEL version=${APP_VERSION}

WORKDIR /app
COPY . .
RUN npm ci --only=production

CMD ["node", "server.js"]
```

Build command:

```bash
docker build \
  --build-arg BUILD_DATE=$(date -u +'%Y-%m-%dT%H:%M:%SZ') \
  --build-arg GIT_COMMIT=$(git rev-parse --short HEAD) \
  --build-arg APP_VERSION=2.1.0 \
  -t my-app:2.1.0 .
```

## 🔟 `EXPOSE` — Document the Port

### What It Does

`EXPOSE` **documents** which port(s) the application inside the container listens on. It's essentially a note for developers and tools — it does NOT actually publish the port.

### Syntax

```dockerfile
EXPOSE port [port/protocol...]
```

### Examples

```dockerfile
# Expose HTTP port
EXPOSE 80

# Expose HTTPS port
EXPOSE 443

# Expose a custom app port
EXPOSE 3000

# Expose multiple ports
EXPOSE 80 443

# Expose UDP port (default is TCP)
EXPOSE 53/udp

# Expose both TCP and UDP
EXPOSE 53/tcp
EXPOSE 53/udp
```

### ⚠️ The Common Misconception

**`EXPOSE` does NOT publish the port to the host!** It's just documentation.

```dockerfile
# This alone does NOT make port 3000 accessible from outside
EXPOSE 3000
```

To actually make the port accessible, you MUST use `-p` when running:

```bash
# This publishes the port
docker run -p 3000:3000 my-app

# Or publish all EXPOSE'd ports to random host ports
docker run -P my-app
```

```
EXPOSE in Dockerfile = "This container uses port 3000" (documentation)
-p 3000:3000 in docker run = "Map host port 3000 to container port 3000" (action)
```

### So Why Bother with EXPOSE?

1. **Documentation** — tells other developers which port the app uses
2. **`docker run -P`** — automatically publishes all `EXPOSE`'d ports to random host ports
3. **Docker Compose** — uses `EXPOSE` for inter-container communication
4. **Tools & Orchestrators** — Kubernetes, Docker Swarm use this info

```bash
# -P (capital P) publishes all EXPOSE'd ports to random host ports
docker run -d -P my-app

# Check which random ports were assigned
docker ps
# Output: 0.0.0.0:32768->3000/tcp
```

## 📦 Putting It All Together — Complete Dockerfile Examples

### Example 1: Simple Python Flask Application

```dockerfile
# Use a slim Python base image
FROM python:3.12-slim

# Set environment variables
ENV PYTHONDONTWRITEBYTECODE=1
ENV PYTHONUNBUFFERED=1

# Set working directory
WORKDIR /app

# Copy requirements first (for caching)
COPY requirements.txt .

# Install dependencies
RUN pip install --no-cache-dir -r requirements.txt

# Copy application code
COPY . .

# Document the port

# Start the application
CMD ["python", "app.py"]
```

Build and run:
```bash
docker build -t flask-app:v1 .
docker run -d --name my-flask -p 5000:5000 flask-app:v1
```

### Example 2: Node.js Express Application

```dockerfile
# Use Alpine Node.js for smaller image
FROM node:20-alpine

# Create app directory
WORKDIR /app

# Copy package files first (caching!)
COPY package.json package-lock.json ./

# Install only production dependencies
RUN npm ci --only=production && \
    npm cache clean --force

# Copy source code
COPY . .

# Set environment
ENV NODE_ENV=production
ENV PORT=3000

# Document the port
EXPOSE 3000

# Start the app
CMD ["node", "server.js"]
```

### Example 3: Multi-Stage Build — React + Nginx

```dockerfile
# ===== Stage 1: Build the React app =====
FROM node:20-alpine AS builder

WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci
COPY . .
RUN npm run build

# ===== Stage 2: Serve with Nginx =====
FROM nginx:alpine

# Copy built files from the builder stage
COPY --from=builder /app/build /usr/share/nginx/html

# Copy custom Nginx config (optional)
COPY nginx.conf /etc/nginx/conf.d/default.conf

EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```

**What this achieves:**
- Stage 1: Uses Node.js to build the React app (~400 MB)
- Stage 2: Only copies the built files into Nginx (~25 MB)
- Final image is tiny — just Nginx + your static files!

### Example 4: Go Application (Ultra-Small Image)

```dockerfile
# ===== Stage 1: Build =====
FROM golang:1.22-alpine AS builder

WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -o /app/server .

# ===== Stage 2: Run (from scratch — ~0 MB base!) =====
FROM scratch

COPY --from=builder /app/server /server

EXPOSE 8080
ENTRYPOINT ["/server"]
```

This produces an image that's just a few MB — just the compiled Go binary!

### Example 5: Java Spring Boot Application

```dockerfile
# ===== Stage 1: Build with Maven =====
FROM maven:3.9-eclipse-temurin-21 AS builder

WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline
COPY src ./src
RUN mvn package -DskipTests

# ===== Stage 2: Run =====
FROM eclipse-temurin:21-jre-alpine

WORKDIR /app
COPY --from=builder /app/target/*.jar app.jar

ENV JAVA_OPTS="-Xmx512m -Xms256m"
EXPOSE 8080

ENTRYPOINT ["sh", "-c", "java $JAVA_OPTS -jar app.jar"]
```

## 📚 Bonus Instructions (Good to Know)

### `LABEL` — Add Metadata

```dockerfile
LABEL maintainer="ritesh@example.com"
LABEL version="1.0"
LABEL description="My awesome application"
LABEL org.opencontainers.image.source="https://github.com/user/repo"
```

### `USER` — Run as Non-Root

```dockerfile
# Create a non-root user
RUN addgroup -S appgroup && adduser -S appuser -G appgroup

# Switch to that user
USER appuser

# All subsequent commands run as appuser (not root!)
CMD ["python", "app.py"]
```

**Always run as non-root in production** for better security.

`USER` sets the default runtime user for the container. It changes the user for subsequent `RUN`, `CMD`, and `ENTRYPOINT` instructions, but it does not prevent a Docker operator from later attaching to the container or using `docker exec -u root` if they have control of the host.

### `VOLUME` — Declare Mount Points

```dockerfile
# Declare that /data should be a volume
VOLUME ["/data"]

# The data in /data will persist even after the container is deleted
```

`VOLUME` marks a directory inside the image as a mount point for persistent storage. When the container starts, Docker will mount a volume at that path:
- If you specify `-v my-data:/data` or `--mount type=volume,source=my-data,target=/data`, Docker uses the named volume you provided.
- If you do not provide a source volume, Docker creates an anonymous volume automatically.

This is useful to keep large or mutable data out of the image layer and make the path persistent across container restarts and even container removal. However, `VOLUME` does not choose a host path for you — it only declares that `/data` should be backed by a volume.

> Note: `docker rm` does not remove attached volumes unless you pass `-v`. Anonymous volumes can remain orphaned if you delete the container without removing the volume.

### `HEALTHCHECK` — Container Health Monitoring

```dockerfile
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
  CMD curl -f http://localhost:3000/health || exit 1
```

This tells Docker to check the container's health every 30 seconds.

### `SHELL` — Change Default Shell

```dockerfile
# Change the default shell (useful on Windows containers)
SHELL ["/bin/bash", "-c"]

# Now RUN commands use bash instead of sh
RUN echo "Using bash now!"
```

### `STOPSIGNAL` — Custom Stop Signal

```dockerfile
# Use SIGQUIT instead of SIGTERM to stop the container
STOPSIGNAL SIGQUIT
```

## 💡 Dockerfile Best Practices Summary

### 1. Use Specific Base Image Tags
```dockerfile
# Bad
FROM python:latest

# Good
FROM python:3.12-slim
```

### 2. Combine RUN Commands
```dockerfile
# Bad (3 layers)
RUN apt-get update
RUN apt-get install -y curl
RUN rm -rf /var/lib/apt/lists/*

# Good (1 layer)
RUN apt-get update && \
    apt-get install -y --no-install-recommends curl && \
    rm -rf /var/lib/apt/lists/*
```

### 3. Order Instructions by Change Frequency
```dockerfile
# Things that change LESS → put FIRST (better caching)
FROM node:20-alpine
WORKDIR /app
COPY package.json package-lock.json ./     # Changes rarely
RUN npm install                            # Cached if packages didn't change
COPY . .                                   # Changes often — put LAST
```

### 4. Use `.dockerignore`
```
node_modules
.git
.env
*.md
Dockerfile
docker-compose.yml
tests/
coverage/
.vscode/
```

### 5. Run as Non-Root
```dockerfile
RUN addgroup -S app && adduser -S app -G app
USER app
```

### 6. Use Multi-Stage Builds
Keep your production images lean by building in one stage and copying artifacts to a minimal stage.

### 7. Use Exec Form for CMD/ENTRYPOINT
```dockerfile
# Good — process is PID 1, receives signals properly
CMD ["python", "app.py"]

# Bad — wraps in /bin/sh, signal handling issues
CMD python app.py
```

## 📝 Summary

| Instruction | Purpose | Available At | Key Point |
|-------------|---------|-------------|-----------|
| `FROM` | Base image | — | Always first, use specific tags |
| `RUN` | Execute build commands | Build time | Combine with `&&`, clean up in same layer |
| `COPY` | Copy files from host | Build time | Preferred over ADD |
| `ADD` | Copy + extract archives | Build time | Only use for tar extraction |
| `CMD` | Default startup command | Runtime | Can be overridden, use exec form |
| `ENTRYPOINT` | Fixed startup command | Runtime | Arguments get appended |
| `WORKDIR` | Set working directory | Both | Creates dir if not exists |
| `ENV` | Set environment variables | Both | Don't store secrets! |
| `ARG` | Build-time variables | Build time only | Use `--build-arg` to pass values |
| `EXPOSE` | Document port | Documentation | Does NOT publish — use `-p` for that |

**The most common Dockerfile pattern:**
```dockerfile
FROM base-image:tag
WORKDIR /app
COPY dependencies-file .
RUN install-dependencies
COPY . .
ENV CONFIG=value
EXPOSE port
CMD ["start-command"]
```
