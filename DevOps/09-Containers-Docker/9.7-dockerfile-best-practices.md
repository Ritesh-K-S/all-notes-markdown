# 9.7 — Dockerfile Best Practices — Layer Caching, Multi-Stage Builds, .dockerignore

> **One Line Answer:** Writing a Dockerfile is easy, but writing a **good** Dockerfile is an art — you need to understand **layer caching** (so builds are fast), **multi-stage builds** (so images are small), and **.dockerignore** (so unnecessary files don't bloat your image) — mastering these three things will make you 10x better at Docker.

---

## 🤔 Why Do Best Practices Matter?

You can write a Dockerfile that "works" in 5 minutes. But a badly written Dockerfile will:

| Problem | Impact |
|---------|--------|
| Huge image size (2GB+) | Slow to push, pull, and deploy |
| Slow builds (10+ minutes) | Wasted developer time, slow CI/CD pipelines |
| Security vulnerabilities | More packages = more attack surface |
| Unnecessary files in image | `.git/`, `node_modules/`, secrets accidentally leaked |
| Cache busted on every build | Rebuilds EVERYTHING even for tiny code changes |

A well-written Dockerfile:
- Builds in **seconds** (thanks to layer caching)
- Produces images that are **10-100MB** instead of 1-2GB
- Contains **only what's needed** to run the app
- Is **secure** with minimal attack surface

# 📦 Part 1: Understanding Docker Layers & Caching

---

## How Docker Layers Work

This is the **foundation** of everything. If you understand layers, everything else clicks.

Every instruction in a Dockerfile creates a **layer**. Think of layers like transparent sheets stacked on top of each other — each one adds something new.

```dockerfile
FROM ubuntu:22.04        # Layer 1 — Base OS
RUN apt-get update       # Layer 2 — Update package list  
RUN apt-get install -y curl  # Layer 3 — Install curl
COPY app.py /app/        # Layer 4 — Copy your app
CMD ["python3", "app.py"] # Layer 5 — Run command (metadata, not a real layer)
```

```
Visual: How layers stack up

┌──────────────────────────────┐
│ Layer 5: CMD (metadata only) │  ← Doesn't create a filesystem layer
├──────────────────────────────┤
│ Layer 4: COPY app.py         │  ← Your application code
├──────────────────────────────┤
│ Layer 3: apt install curl    │  ← curl binary added
├──────────────────────────────┤
│ Layer 2: apt-get update      │  ← Package lists updated
├──────────────────────────────┤
│ Layer 1: ubuntu:22.04 base   │  ← Full Ubuntu filesystem
└──────────────────────────────┘
```

### Key Facts About Layers

| Fact | Explanation |
|------|-------------|
| Each `RUN`, `COPY`, `ADD` creates a new layer | `CMD`, `ENTRYPOINT`, `ENV`, `EXPOSE` only add metadata |
| Layers are **read-only** | Once created, a layer never changes |
| Layers are **cached** | Docker reuses them if nothing changed |
| Layers are **shared** | If 5 images use `ubuntu:22.04`, they all share the same base layer on disk |
| Final running container adds a **writable layer** on top | This is where runtime changes happen |

---

## How Docker Layer Caching Works

This is the **most important concept** for fast builds. Here's how Docker decides whether to use cache or rebuild a layer:

```
Docker Build Process (for EACH instruction):

1. Look at the instruction
2. Check: "Have I built this exact instruction before?"
3. Check: "Has anything ABOVE this layer changed?"

If YES to both → USE CACHE ✅ (instant!)
If NO to either → REBUILD this layer AND all layers below it ❌
```

### The Golden Rule of Caching

> **Once a layer's cache is invalidated, ALL subsequent layers are also invalidated.**

This is huge. Let me show you why:

### ❌ Bad Dockerfile — Cache Busted Every Time

```dockerfile
FROM node:18-alpine
WORKDIR /app
COPY . .                          # Layer 1: Copies EVERYTHING
RUN npm install                   # Layer 2: Installs dependencies
CMD ["node", "server.js"]
```

**What happens when you change ONE line of code in `server.js`?**

```
COPY . .          → Cache BUSTED ❌ (files changed!)
RUN npm install   → Cache BUSTED ❌ (previous layer changed, so this rebuilds too!)

Result: npm install runs EVERY TIME you change ANY file!
        Even if package.json hasn't changed!
        This could take 30-60 seconds each time! 😤
```

### ✅ Good Dockerfile — Smart Cache Usage

```dockerfile
FROM node:18-alpine
WORKDIR /app
COPY package.json package-lock.json ./   # Layer 1: Only dependency files
RUN npm install                           # Layer 2: Install dependencies
COPY . .                                  # Layer 3: Copy everything else
CMD ["node", "server.js"]
```

**What happens when you change ONE line of code in `server.js`?**

```
COPY package.json ...  → Cache HIT ✅ (package.json didn't change)
RUN npm install        → Cache HIT ✅ (previous layer cached, so this is cached too!)
COPY . .               → Cache BUSTED ❌ (source code changed)

Result: npm install is CACHED! Only the final COPY runs.
        Build takes 1-2 seconds instead of 30-60 seconds! 🎉
```

### Visual: Cache Flow

```
❌ BAD ORDER:                          ✅ GOOD ORDER:

COPY . . ──────────── BUSTED          COPY package*.json ── CACHED
    │                                       │
    ▼                                       ▼
RUN npm install ───── BUSTED          RUN npm install ───── CACHED
    │                                       │
    ▼                                       ▼
Result: Full rebuild                  COPY . . ──────────── BUSTED (only this!)
        every time                          │
                                            ▼
                                      Result: Only code copy
                                              happens. FAST! ⚡
```

---

## Cache Rules for Different Instructions

| Instruction | When Cache is Invalidated |
|-------------|--------------------------|
| `RUN` | If the command string changes (even a space!) OR any previous layer changed |
| `COPY` | If any file being copied has changed (checks file checksums) |
| `ADD` | Same as COPY, plus re-downloads URLs every time (avoid ADD for URLs!) |
| `FROM` | If the base image tag resolves to a different digest |
| `ARG` / `ENV` | If the value changes |

---

## Practical Cache Optimization Tips

### Tip 1: Put Rarely-Changing Things FIRST

```dockerfile
# ✅ GOOD: Things that change rarely → at the top
FROM python:3.11-slim

# System dependencies (rarely change)
RUN apt-get update && apt-get install -y \
    gcc \
    libpq-dev \
    && rm -rf /var/lib/apt/lists/*

# Python dependencies (change sometimes)
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Application code (changes frequently)
COPY . .

CMD ["python", "app.py"]
```

### Tip 2: Combine RUN Commands to Reduce Layers

```dockerfile
# ❌ BAD: 3 separate layers
RUN apt-get update
RUN apt-get install -y curl wget git
RUN rm -rf /var/lib/apt/lists/*

# ✅ GOOD: 1 layer with cleanup in same layer
RUN apt-get update && apt-get install -y \
    curl \
    wget \
    git \
    && rm -rf /var/lib/apt/lists/*
```

**Why combine?** Two reasons:
1. Fewer layers = smaller image
2. The `rm` cleanup in a separate `RUN` doesn't actually save space! The files still exist in the previous layer. You MUST delete them in the **same** `RUN` command.

### Tip 3: Use `--no-cache-dir` for pip

```dockerfile
# ❌ Leaves pip cache in the image (wastes space)
RUN pip install -r requirements.txt

# ✅ No cache stored in the image
RUN pip install --no-cache-dir -r requirements.txt
```

### Tip 4: Use `--mount=type=cache` for Persistent Build Caches (BuildKit)

```dockerfile
# syntax=docker/dockerfile:1
FROM node:18-alpine
WORKDIR /app
COPY package*.json ./

# npm cache is persisted between builds!
RUN --mount=type=cache,target=/root/.npm \
    npm install

COPY . .
CMD ["node", "server.js"]
```

This keeps npm's download cache across builds, so even if `package.json` changes, packages that haven't changed are not re-downloaded.

# 🏗️ Part 2: Multi-Stage Builds

---

## The Problem Multi-Stage Builds Solve

When you build an application, you typically need **two different sets of tools**:

1. **Build tools** — compilers, build frameworks, dev dependencies (only needed during build)
2. **Runtime** — just the compiled binary/app and runtime dependencies (needed to run)

Without multi-stage builds, your final image contains EVERYTHING:

```
Single-Stage Build (❌):

┌─────────────────────────────────────────┐
│            Final Docker Image            │
│                                          │
│  ✅ Your compiled app (5 MB)            │
│  ❌ Go compiler (500 MB)                │
│  ❌ Source code (10 MB)                 │
│  ❌ Build cache (200 MB)                │
│  ❌ Dev dependencies (300 MB)           │
│                                          │
│  Total: ~1 GB 😱                        │
└─────────────────────────────────────────┘
```

With multi-stage builds, you **build** in one stage and **copy only what you need** to the final stage:

```
Multi-Stage Build (✅):

Stage 1: "builder"                    Stage 2: "final"
┌────────────────────────┐           ┌────────────────────────┐
│ Go compiler            │           │ Alpine Linux (5 MB)    │
│ Source code             │    COPY   │ Your compiled app      │
│ Build dependencies      │ ───────→ │   (5 MB)               │
│ Compiled binary ✅      │  (only   │                        │
│                         │  binary) │ Total: ~10 MB 🎉       │
│ Total: ~1 GB           │           └────────────────────────┘
└────────────────────────┘
  (This stage is thrown away!)
```

---

## Multi-Stage Build Syntax

The key idea: You can have **multiple `FROM`** instructions in a single Dockerfile. Each `FROM` starts a new **stage**.

### Example 1: Go Application

```dockerfile
# ==================== STAGE 1: Build ====================
FROM golang:1.21-alpine AS builder

WORKDIR /app

# Copy dependency files first (for caching)
COPY go.mod go.sum ./
RUN go mod download

# Copy source code and build
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -o /app/myapp .

# ==================== STAGE 2: Run ====================
FROM alpine:3.18

# Add a non-root user for security
RUN adduser -D -g '' appuser

WORKDIR /app

# Copy ONLY the compiled binary from the builder stage
COPY --from=builder /app/myapp .

# Switch to non-root user
USER appuser

# Run the app
ENTRYPOINT ["./myapp"]
```

**What's happening here:**

| Stage | What It Does | What's Included in Final Image? |
|-------|-------------|-------------------------------|
| Stage 1 (`builder`) | Downloads Go, compiles the app | ❌ Thrown away |
| Stage 2 (final) | Starts fresh from tiny `alpine`, copies only the binary | ✅ This is the final image |

**Result:**
- Without multi-stage: **~1 GB** image
- With multi-stage: **~10-15 MB** image

That's a **99% reduction** in image size!

### Example 2: Node.js Application

```dockerfile
# ==================== STAGE 1: Build ====================
FROM node:18-alpine AS builder

WORKDIR /app

COPY package*.json ./
RUN npm ci

COPY . .
RUN npm run build    # Creates a /app/dist folder

# ==================== STAGE 2: Run ====================
FROM node:18-alpine

WORKDIR /app

# Only copy production dependencies
COPY package*.json ./
RUN npm ci --only=production

# Copy built files from builder stage
COPY --from=builder /app/dist ./dist

USER node

CMD ["node", "dist/server.js"]
```

### Example 3: React/Frontend Application

```dockerfile
# ==================== STAGE 1: Build ====================
FROM node:18-alpine AS builder

WORKDIR /app

COPY package*.json ./
RUN npm ci

COPY . .
RUN npm run build    # Creates /app/build with static files

# ==================== STAGE 2: Serve with Nginx ====================
FROM nginx:alpine

# Copy the built static files to Nginx's serve directory
COPY --from=builder /app/build /usr/share/nginx/html

# Copy custom Nginx config (optional)
COPY nginx.conf /etc/nginx/conf.d/default.conf

EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```

**Result:** Final image is just Nginx + static HTML/CSS/JS files (~25 MB)

### Example 4: Python Application

```dockerfile
# ==================== STAGE 1: Build dependencies ====================
FROM python:3.11-slim AS builder

WORKDIR /app

# Install build tools needed for compiling Python packages
RUN apt-get update && apt-get install -y \
    gcc \
    libpq-dev \
    && rm -rf /var/lib/apt/lists/*

COPY requirements.txt .
RUN pip install --no-cache-dir --prefix=/install -r requirements.txt

# ==================== STAGE 2: Runtime ====================
FROM python:3.11-slim

WORKDIR /app

# Copy only the installed packages from builder
COPY --from=builder /install /usr/local

# Only runtime library needed (no gcc!)
RUN apt-get update && apt-get install -y \
    libpq5 \
    && rm -rf /var/lib/apt/lists/*

COPY . .

USER nobody

CMD ["python", "app.py"]
```

---

## Using Named Stages

You can name stages with `AS` and reference them:

```dockerfile
FROM golang:1.21 AS builder
# ... build stuff ...

FROM alpine:3.18 AS production
COPY --from=builder /app/binary .

# You can also copy from a completely external image!
COPY --from=nginx:alpine /etc/nginx/nginx.conf /etc/nginx/
```

### Building a Specific Stage

You can build just one stage (useful for debugging):

```bash
# Build only the builder stage
docker build --target builder -t myapp:builder .

# Build the full image (default — builds all stages)
docker build -t myapp:latest .
```

---

## Multi-Stage Build Patterns

### Pattern 1: Builder Pattern (Most Common)

```
Stage 1: Build → Stage 2: Run
(Heavy tools)     (Minimal runtime)
```

### Pattern 2: Testing in Stages

```dockerfile
FROM node:18-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Testing stage
FROM builder AS tester
RUN npm test

# Production stage (only reached if tests pass in CI)
FROM node:18-alpine AS production
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY --from=builder /app/dist ./dist
CMD ["node", "dist/server.js"]
```

### Pattern 3: Development vs Production

```dockerfile
FROM node:18-alpine AS base
WORKDIR /app
COPY package*.json ./

# Development stage — includes dev dependencies and hot reload
FROM base AS development
RUN npm install
COPY . .
CMD ["npm", "run", "dev"]

# Production stage — minimal and optimized
FROM base AS production
RUN npm ci --only=production
COPY . .
RUN npm run build
CMD ["node", "dist/server.js"]
```

Build the one you need:
```bash
docker build --target development -t myapp:dev .
docker build --target production -t myapp:prod .
```

# 📄 Part 3: .dockerignore — Keep Your Image Clean

---

## What is .dockerignore?

When you run `docker build`, the very first thing Docker does is send the **build context** (all files in the directory) to the Docker daemon:

```
$ docker build -t myapp .
Sending build context to Docker daemon  256MB    ← 😱 This is huge!
```

That `256MB` means Docker is sending EVERYTHING in your project directory — including `node_modules/`, `.git/`, test files, docs, secrets, etc.

`.dockerignore` works exactly like `.gitignore` — it tells Docker which files to **exclude** from the build context.

---

## Why is .dockerignore Important?

| Without .dockerignore | With .dockerignore |
|----------------------|-------------------|
| Build context: 256MB | Build context: 2MB |
| Slow build start | Fast build start |
| `node_modules/` copied into image | Only needed files included |
| `.git/` history in image | Clean image |
| `.env` secrets in image 🚨 | Secrets excluded |
| Build cache invalidated by irrelevant file changes | Cache only busted by real changes |

---

## How to Create a .dockerignore File

Create a file named `.dockerignore` in the **root of your project** (same directory as your `Dockerfile`):

### Example .dockerignore for a Node.js Project

---

## .dockerignore Syntax Rules

| Pattern | What It Matches |
|---------|----------------|
| `node_modules` | `node_modules` directory anywhere |
| `*.log` | All files ending in `.log` |
| `**/*.test.js` | All `.test.js` files in any subdirectory |
| `!important.log` | Exception — DON'T ignore `important.log` |
| `temp?` | `temp` followed by any single character (`temp1`, `tempa`) |
| `#` | Comment line (ignored) |

### Using Exceptions (!) — Include Something Back

```dockerignore
# Ignore everything
*

# But include these
!src/
!package.json
!package-lock.json
!tsconfig.json

# This pattern is great for whitelisting —
# instead of blacklisting bad files, you whitelist only what you need!
```

This is the **most secure approach** — start by ignoring everything, then whitelist only what's needed.

## Verifying Your .dockerignore Works

You can check what files are being sent as build context:

```bash
# See the size of the build context
docker build -t test . 2>&1 | head -5
# Output: Sending build context to Docker daemon  2.048kB  ← Nice and small!

# If you want to see exactly which files Docker sees:
# (Create a temporary Dockerfile)
echo "FROM busybox" > Dockerfile.test
echo "COPY . /context" >> Dockerfile.test
echo "RUN find /context -type f" >> Dockerfile.test
docker build -f Dockerfile.test -t context-check . 2>&1 | grep "context/"
```

# 🛡️ Part 4: General Dockerfile Best Practices

---

## 1. Choose the Right Base Image

```dockerfile
# ❌ BAD: Full OS image (900 MB)
FROM ubuntu:22.04

# ❌ OKAY but big: Full language image (350 MB)
FROM python:3.11

# ✅ GOOD: Slim variant (150 MB)
FROM python:3.11-slim

# ✅ BEST (if your app supports it): Alpine-based (50 MB)
FROM python:3.11-alpine

# ✅ BEST for Go/compiled languages: Distroless (< 20 MB)
FROM gcr.io/distroless/static-debian12
```

### Base Image Comparison

```
┌────────────────────────────────────────────────────────────────┐
│              Base Image Sizes (approximate)                     │
│                                                                 │
│  ubuntu:22.04        ████████████████████████████████  ~75 MB  │
│  python:3.11         ████████████████████████████████████ ~900 MB│
│  python:3.11-slim    ███████████████                    ~150 MB │
│  python:3.11-alpine  ██████                             ~50 MB  │
│  alpine:3.18         ███                                ~7 MB   │
│  distroless          ██                                 ~3 MB   │
│  scratch             (empty)                            0 MB    │
└────────────────────────────────────────────────────────────────┘
```

### When to Use What

| Base Image | Use When |
|-----------|---------|
| `ubuntu` / `debian` | You need full OS tools, or packages require glibc |
| `*-slim` | Good default — most packages work, much smaller |
| `*-alpine` | Size matters, your app works with musl libc (test first!) |
| `distroless` | Production Go/Java/Python apps, maximum security |
| `scratch` | Statically compiled Go binaries, ultimate minimal |

---

## 2. Pin Your Base Image Versions

```dockerfile
# ❌ BAD: "latest" can change and break your build
FROM node:latest
FROM python

# ❌ RISKY: Major version can include breaking changes
FROM node:18

# ✅ GOOD: Pinned to minor version
FROM node:18.19-alpine

# ✅ BEST: Pinned to exact digest (for reproducible builds)
FROM node:18.19-alpine@sha256:abc123...
```

---

## 3. Use a Non-Root User

By default, containers run as `root`. This is a security risk.

```dockerfile
# ✅ Create and use a non-root user
FROM node:18-alpine

# Create a group and user
RUN addgroup -S appgroup && adduser -S appuser -G appgroup

WORKDIR /app
COPY --chown=appuser:appgroup . .

# Switch to non-root user
USER appuser

CMD ["node", "server.js"]
```

---

## 4. Use COPY Instead of ADD

```dockerfile
# ❌ ADD has magic behaviors (auto-extracts tars, downloads URLs)
ADD https://example.com/file.tar.gz /app/
ADD archive.tar.gz /app/

# ✅ COPY is simple and predictable
COPY . .

# Only use ADD if you specifically WANT auto-extraction:
ADD archive.tar.gz /app/   # This auto-extracts the tar — sometimes useful
```

---

## 5. Set Proper Labels (Metadata)

```dockerfile
FROM python:3.11-slim

LABEL maintainer="your-name@email.com"
LABEL version="1.0.0"
LABEL description="My awesome Python API"
LABEL org.opencontainers.image.source="https://github.com/you/repo"

# Using OCI standard labels
LABEL org.opencontainers.image.title="My App"
LABEL org.opencontainers.image.version="1.0.0"
LABEL org.opencontainers.image.vendor="My Company"
```

---

## 6. Health Checks

```dockerfile
FROM node:18-alpine

WORKDIR /app
COPY . .
RUN npm ci --only=production

# Add a health check
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
    CMD curl -f http://localhost:3000/health || exit 1

CMD ["node", "server.js"]
```

What this does:
- Every **30 seconds**, Docker runs `curl http://localhost:3000/health`
- If it fails **3 times in a row**, the container is marked as **unhealthy**
- Docker Swarm and Compose can automatically restart unhealthy containers

---

## 7. Don't Store Secrets in Images

```dockerfile
# ❌ NEVER DO THIS — secrets are stored in image layers permanently!
ENV API_KEY=sk-12345678
COPY .env /app/.env
RUN echo "password123" > /app/secret.txt

# ✅ Pass secrets at RUNTIME via environment variables
CMD ["node", "server.js"]
# Then: docker run -e API_KEY=sk-12345 myapp

# ✅ Or use Docker secrets (Swarm) or Kubernetes secrets
# ✅ Or use BuildKit secret mounts for build-time secrets:
RUN --mount=type=secret,id=my_secret cat /run/secrets/my_secret
```

---

## 8. Use `.env` Files with `--env-file` at Runtime

```bash
# Create an .env file (NOT copied into the image)
echo "DB_HOST=localhost" > .env
echo "DB_PASSWORD=secret123" >> .env

# Pass it at runtime
docker run --env-file .env myapp
```

# 🧪 Complete Real-World Example — Putting It All Together

Here's a production-ready Dockerfile for a Node.js API:

```dockerfile
# ==================== STAGE 1: Build ====================
FROM node:18.19-alpine AS builder

# Set working directory
WORKDIR /app

# Copy dependency files first (cache optimization)
COPY package.json package-lock.json ./

# Install ALL dependencies (including devDependencies for building)
RUN npm ci

# Copy source code
COPY src/ ./src/
COPY tsconfig.json ./

# Build the TypeScript project
RUN npm run build

# ==================== STAGE 2: Production ====================
FROM node:18.19-alpine

# Add labels
LABEL maintainer="team@company.com"
LABEL version="1.0.0"

# Create non-root user
RUN addgroup -S app && adduser -S app -G app

# Set working directory
WORKDIR /app

# Copy only production dependencies
COPY package.json package-lock.json ./
RUN npm ci --only=production && npm cache clean --force

# Copy built files from builder stage
COPY --from=builder /app/dist ./dist

# Change ownership
RUN chown -R app:app /app

# Switch to non-root user
USER app

# Expose port
EXPOSE 3000

# Health check
HEALTHCHECK --interval=30s --timeout=3s --retries=3 \
    CMD wget --no-verbose --tries=1 --spider http://localhost:3000/health || exit 1

# Start the app
CMD ["node", "dist/server.js"]
```

## 📊 Quick Reference — Best Practices Checklist

```
✅ Dockerfile Best Practices Checklist:

□ Use specific base image tags (not "latest")
□ Use slim/alpine base images when possible
□ Order instructions: rarely-changing → frequently-changing
□ Copy dependency files BEFORE source code
□ Combine RUN commands with && and clean up in same layer
□ Use multi-stage builds (build stage → runtime stage)
□ Create and use a non-root USER
□ Add HEALTHCHECK instruction
□ Use COPY instead of ADD
□ Add meaningful LABELs
□ NEVER put secrets in Dockerfile or image layers
□ Create a comprehensive .dockerignore
□ Use --no-cache-dir for pip, npm cache clean, etc.
□ Pin base image versions
□ Use BuildKit for advanced features (--mount=type=cache)
```

## 🔑 Key Takeaways

| Concept | Remember This |
|---------|--------------|
| **Layer Caching** | Put things that change RARELY at the top, things that change OFTEN at the bottom |
| **Multi-Stage Builds** | Build in one stage, run in another — copy only what you need |
| **.dockerignore** | Like `.gitignore` for Docker — exclude `node_modules`, `.git`, `.env`, etc. |
| **Base Images** | Use `slim` or `alpine` variants — avoid full OS images |
| **Security** | Non-root user, no secrets in images, health checks |
| **RUN Commands** | Combine with `&&` and clean up in the same layer |
