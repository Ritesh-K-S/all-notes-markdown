# 9.8 — Docker Volumes — Named Volumes, Bind Mounts, tmpfs

> **One Line Answer:** Docker containers are **ephemeral** — when a container is deleted, all data inside it is gone forever. **Docker Volumes** solve this by providing persistent storage that survives container restarts and deletions — you can use **named volumes** (managed by Docker), **bind mounts** (map a host folder into the container), or **tmpfs** (temporary in-memory storage that disappears on stop).

---

## 🤔 Why Do We Need Volumes?

Let's understand the problem first.

### The Problem: Containers Are Ephemeral

```
Container Lifecycle WITHOUT Volumes:

1. Start a container
   ┌─────────────────────┐
   │  MySQL Container     │
   │  /var/lib/mysql/     │
   │  (empty database)    │
   └─────────────────────┘

2. App writes data to the database
   ┌─────────────────────┐
   │  MySQL Container     │
   │  /var/lib/mysql/     │
   │  📊 users table      │
   │  📊 orders table     │
   │  📊 products table   │
   └─────────────────────┘

3. Container crashes or is deleted
   ┌─────────────────────┐
   │  💥 DELETED!         │
   │  ALL DATA IS GONE!   │
   │  users table? GONE   │
   │  orders table? GONE  │
   │  products table? GONE│
   └─────────────────────┘

   😱 Your production data just vanished!
```

### The Solution: Volumes

```
Container Lifecycle WITH Volumes:

1. Start a container with a volume
   ┌─────────────────────┐      ┌──────────────────┐
   │  MySQL Container     │ ───→ │  Docker Volume    │
   │  /var/lib/mysql/ ────│──────│  (on host disk)   │
   └─────────────────────┘      └──────────────────┘

2. App writes data — stored on the VOLUME (host disk)
   ┌─────────────────────┐      ┌──────────────────┐
   │  MySQL Container     │ ───→ │  Docker Volume    │
   │  /var/lib/mysql/ ────│──────│  📊 users table   │
   │                      │      │  📊 orders table  │
   └─────────────────────┘      │  📊 products table│
                                 └──────────────────┘

3. Container crashes or is deleted
   ┌─────────────────────┐      ┌──────────────────┐
   │  💥 Container gone   │      │  Docker Volume    │
   │                      │      │  📊 users table   │ ← STILL HERE! ✅
   └─────────────────────┘      │  📊 orders table  │ ← STILL HERE! ✅
                                 │  📊 products table│ ← STILL HERE! ✅
                                 └──────────────────┘

4. Start a NEW container and attach the SAME volume
   ┌─────────────────────┐      ┌──────────────────┐
   │  New MySQL Container │ ───→ │  Docker Volume    │
   │  /var/lib/mysql/ ────│──────│  📊 users table   │ ← All data is back!
   │                      │      │  📊 orders table  │
   └─────────────────────┘      │  📊 products table│
                                 └──────────────────┘
   🎉 Data survived the container death!
```

## 🗂️ Three Types of Docker Storage

Docker provides three ways to persist or manage data:

```
┌─────────────────────────────────────────────────────────────────┐
│                    Docker Storage Types                          │
│                                                                  │
│  ┌─────────────────┐  ┌─────────────────┐  ┌────────────────┐  │
│  │  Named Volumes   │  │  Bind Mounts     │  │  tmpfs Mounts  │  │
│  │                  │  │                  │  │                │  │
│  │  Managed by      │  │  YOU choose the  │  │  Stored in     │  │
│  │  Docker          │  │  host path       │  │  RAM only      │  │
│  │                  │  │                  │  │                │  │
│  │  Stored in:      │  │  Stored in:      │  │  Stored in:    │  │
│  │  /var/lib/docker │  │  Anywhere on     │  │  Host memory   │  │
│  │  /volumes/       │  │  host filesystem │  │  (never disk)  │  │
│  │                  │  │                  │  │                │  │
│  │  Best for:       │  │  Best for:       │  │  Best for:     │  │
│  │  Databases,      │  │  Dev: share code │  │  Secrets,      │  │
│  │  persistent data │  │  with container  │  │  temp data     │  │
│  └─────────────────┘  └─────────────────┘  └────────────────┘  │
│                                                                  │
│        ⭐ RECOMMENDED        💻 DEVELOPMENT           🔒 SPECIAL │
└─────────────────────────────────────────────────────────────────┘
```

### Quick Comparison

| Feature | Named Volumes | Bind Mounts | tmpfs |
|---------|--------------|-------------|-------|
| **Managed by** | Docker | You | OS (memory) |
| **Location on host** | `/var/lib/docker/volumes/` | Anywhere you choose | RAM |
| **Survives container restart** | ✅ Yes | ✅ Yes | ❌ No |
| **Survives container removal** | ✅ Yes | ✅ Yes | ❌ No |
| **Shared between containers** | ✅ Yes | ✅ Yes | ❌ No |
| **Pre-populated with image data** | ✅ Yes | ❌ No (overwrites) | ❌ No |
| **Works on all OS** | ✅ Yes | ⚠️ Path issues Windows/Mac | ✅ Linux only |
| **Best for** | Production data | Development, configs | Sensitive temp data |

# 📦 Part 1: Named Volumes (Docker-Managed)

---

## What Are Named Volumes?

Named volumes are the **recommended way** to persist data in Docker. Docker manages everything — you just give it a name.

Think of it like a **USB drive** that Docker manages for you:
- You plug it into a container (mount it)
- Data is stored on it
- You can unplug it and plug it into a different container
- Docker handles where it's stored on disk

---

## Creating and Using Named Volumes

### Method 1: Create Volume First, Then Use It

```bash
# Create a named volume
docker volume create my-data

# Use it in a container
docker run -d \
    --name my-db \
    -v my-data:/var/lib/mysql \
    mysql:8.0
```

### Method 2: Create On-the-Fly (Docker Creates It Automatically)

```bash
# If "my-data" doesn't exist, Docker creates it automatically
docker run -d \
    --name my-db \
    -v my-data:/var/lib/mysql \
    mysql:8.0
```

### Using the `--mount` Syntax (More Explicit and Recommended)

```bash
docker run -d \
    --name my-db \
    --mount type=volume,source=my-data,target=/var/lib/mysql \
    mysql:8.0
```

**`-v` vs `--mount`:** Both do the same thing. `--mount` is more verbose but clearer. Docker's official docs recommend `--mount` for clarity.

| Part | `-v` Syntax | `--mount` Syntax |
|------|------------|----------------|
| Format | `name:container_path` | `type=volume,source=name,target=path` |
| Create if missing | ✅ Auto-creates | ✅ Auto-creates |
| Readability | Shorter | More explicit |

> Tip: A Dockerfile `VOLUME ["/data"]` declares that `/data` should be backed by a volume. If you start the container without specifying a volume source, Docker will still create an anonymous volume for that path. For production, prefer explicit named volumes so the data location is clear.

---

## Managing Named Volumes

```bash
# List all volumes
docker volume ls

# Output:
# DRIVER    VOLUME NAME
# local     my-data
# local     postgres-data
# local     redis-data

# Inspect a volume (see where it's stored)
docker volume inspect my-data

# Output:
# [
#     {
#         "CreatedAt": "2024-01-15T10:30:00Z",
#         "Driver": "local",
#         "Labels": {},
#         "Mountpoint": "/var/lib/docker/volumes/my-data/_data",
#         "Name": "my-data",
#         "Options": {},
#         "Scope": "local"
#     }
# ]

# Remove a volume (only if no container is using it)
docker volume rm my-data

# Remove ALL unused volumes (be careful!)
docker volume prune

# Remove ALL unused volumes without confirmation prompt
docker volume prune -f
```

## Real-World Example: MySQL with Named Volume

```bash
# Step 1: Run MySQL with a named volume
docker run -d \
    --name mysql-server \
    -e MYSQL_ROOT_PASSWORD=mypassword \
    -e MYSQL_DATABASE=myapp \
    -v mysql-data:/var/lib/mysql \
    -p 3306:3306 \
    mysql:8.0

# Step 2: Connect and create some data
docker exec -it mysql-server mysql -u root -pmypassword -e \
    "USE myapp; CREATE TABLE users (id INT, name VARCHAR(50)); INSERT INTO users VALUES (1, 'Alice'), (2, 'Bob');"

# Step 3: Verify the data exists
docker exec -it mysql-server mysql -u root -pmypassword -e \
    "USE myapp; SELECT * FROM users;"
# Output:
# +------+-------+
# | id   | name  |
# +------+-------+
# |    1 | Alice |
# |    2 | Bob   |
# +------+-------+

# Step 4: DELETE the container (simulating a crash)
docker rm -f mysql-server

# Step 5: Create a BRAND NEW container with the SAME volume
docker run -d \
    --name mysql-server-new \
    -e MYSQL_ROOT_PASSWORD=mypassword \
    -v mysql-data:/var/lib/mysql \
    -p 3306:3306 \
    mysql:8.0

# Step 6: Check — is the data still there?
docker exec -it mysql-server-new mysql -u root -pmypassword -e \
    "USE myapp; SELECT * FROM users;"
# Output:
# +------+-------+
# | id   | name  |
# +------+-------+
# |    1 | Alice |  ← Still here! 🎉
# |    2 | Bob   |  ← Still here! 🎉
# +------+-------+
```

**The data survived!** Because it was stored in the `mysql-data` volume, not inside the container.

## Sharing a Volume Between Multiple Containers

```bash
# Container 1: Writer — writes data to the volume
docker run -d \
    --name writer \
    -v shared-data:/data \
    alpine sh -c "while true; do date >> /data/log.txt; sleep 5; done"

# Container 2: Reader — reads data from the same volume
docker run -it \
    --name reader \
    -v shared-data:/data \
    alpine cat /data/log.txt

# Both containers see the same /data directory!
```

```
┌─────────────────┐          ┌─────────────────┐
│  Container 1    │          │  Container 2    │
│  (Writer)       │          │  (Reader)       │
│  /data/ ────────│───┐  ┌───│──── /data/      │
└─────────────────┘   │  │   └─────────────────┘
                      │  │
                      ▼  ▼
                 ┌──────────────┐
                 │ shared-data  │
                 │ volume       │
                 │ (on host)    │
                 └──────────────┘
```

---

## Read-Only Volumes

Sometimes you want a container to **read** data but NOT **write** to it:

```bash
# Mount volume as read-only
docker run -d \
    --name web \
    -v config-data:/etc/nginx/conf.d:ro \
    nginx

# Using --mount syntax
docker run -d \
    --name web \
    --mount type=volume,source=config-data,target=/etc/nginx/conf.d,readonly \
    nginx
```

The `:ro` at the end means **read-only**. The container can see the files but cannot modify them.

# 💻 Part 2: Bind Mounts — Your Folder → Container's Folder

---

## What Are Bind Mounts?

Bind mounts **link a specific directory on your host machine** directly into the container. Unlike named volumes where Docker picks the location, with bind mounts **you choose exactly** which host folder to use.

Think of it as creating a **portal** between your host and the container:

```
Your Host Machine:                    Container:
                                      
/home/you/myproject/                  /app/
├── src/                              ├── src/
│   ├── index.js          ═══════>   │   ├── index.js
│   └── utils.js          ═══════>   │   └── utils.js
├── package.json           ═══════>   ├── package.json
└── README.md              ═══════>   └── README.md

    Changes on EITHER side are instantly reflected on BOTH sides!
```

---

## Why Use Bind Mounts?

The **#1 use case** is **local development**. You want to:
1. Edit code on your host machine (with your favorite editor/IDE)
2. Have those changes **instantly** reflected inside the running container
3. No need to rebuild the image every time you change a line of code!

```
Development Workflow WITHOUT Bind Mounts:
1. Edit code on host
2. docker build -t myapp .        ← Wait 30 seconds
3. docker rm -f mycontainer
4. docker run -d myapp            ← Wait for startup
5. Test the change
6. Repeat steps 1-5 for every change 😤

Development Workflow WITH Bind Mounts:
1. Edit code on host
2. Change is INSTANTLY reflected in container
3. Test the change
4. Done! 🎉 (No rebuild needed!)
```

---

## Using Bind Mounts

### Method 1: `-v` Syntax (Short Form)

```bash
# Format: -v /host/path:/container/path
docker run -d \
    --name dev-server \
    -v /home/you/myproject:/app \
    -p 3000:3000 \
    node:18-alpine \
    sh -c "cd /app && npm install && npm run dev"
```

**Important:** For bind mounts, the host path **must be an absolute path** (starts with `/` on Linux/Mac or `C:\` on Windows).

### Method 2: `--mount` Syntax (Recommended)

```bash
docker run -d \
    --name dev-server \
    --mount type=bind,source=/home/you/myproject,target=/app \
    -p 3000:3000 \
    node:18-alpine \
    sh -c "cd /app && npm install && npm run dev"
```

### Using `$(pwd)` for Current Directory

```bash
# Use $(pwd) to get the current directory automatically
docker run -d \
    --name dev-server \
    -v $(pwd):/app \
    -p 3000:3000 \
    node:18-alpine \
    sh -c "cd /app && npm install && npm run dev"

# On Windows PowerShell, use ${PWD}
docker run -d \
    --name dev-server \
    -v ${PWD}:/app \
    -p 3000:3000 \
    node:18-alpine \
    sh -c "cd /app && npm install && npm run dev"
```

---

## Key Differences: Bind Mounts vs Named Volumes

### ⚠️ Important: Bind Mounts OVERWRITE Container Contents

```bash
# Named Volume — if /app in the image has files, they're PRESERVED in the volume
docker run -v my-vol:/app myimage
# /app in container = original files from image + volume data

# Bind Mount — host directory OVERWRITES whatever was in /app
docker run -v /my/empty/folder:/app myimage
# /app in container = contents of /my/empty/folder (image files GONE!)
```

This is actually useful for development (your source code replaces the image code), but be aware of it.

---

## Real-World Example: Node.js Development with Bind Mount

```bash
# Project structure on your host:
# /home/you/myapp/
# ├── src/
# │   └── server.js
# ├── package.json
# └── package-lock.json

# Run with bind mount for live development
docker run -d \
    --name dev \
    -v $(pwd):/app \
    -v /app/node_modules \
    -p 3000:3000 \
    -w /app \
    node:18-alpine \
    sh -c "npm install && npm run dev"
```

**Wait — what's that `-v /app/node_modules` line?**

This is the **anonymous volume trick**. Here's why it's needed:

```
Problem:
1. Your host folder doesn't have node_modules/
2. Bind mount maps your host folder → /app in container
3. When npm install runs, it creates node_modules/ in /app
4. But wait — /app is the bind mount — so node_modules appears
   on your HOST too (with Linux binaries that won't work on your Mac/Windows!)

Solution: Create an anonymous volume for node_modules
-v /app/node_modules    ← This creates a separate volume for node_modules
                          so it stays INSIDE the container only

Result:
┌──────────────────────┐
│  Container /app/     │
│  ├── src/    ←──────── Bind mount from host
│  ├── package.json ←── Bind mount from host
│  └── node_modules/ ── Anonymous volume (container only, not on host)
└──────────────────────┘
```

---

## Real-World Example: Mounting Config Files

```bash
# Mount a single file (not a directory)
docker run -d \
    --name web \
    -v $(pwd)/my-nginx.conf:/etc/nginx/conf.d/default.conf:ro \
    -p 80:80 \
    nginx

# This replaces just the default.conf file while keeping
# everything else in /etc/nginx/ intact
```

---

## Bind Mount Security: Read-Only

```bash
# Make the bind mount read-only so the container can't modify your files
docker run -d \
    --name web \
    -v $(pwd)/config:/app/config:ro \
    myapp

# Container can READ /app/config but CANNOT WRITE to it
```

# 🧠 Part 3: tmpfs Mounts — Temporary In-Memory Storage

---

## What is tmpfs?

`tmpfs` is storage that exists **only in the host's memory (RAM)**. It's never written to disk.

```
Named Volume:     Data saved on DISK     → Persists after container stops
Bind Mount:       Data saved on DISK     → Persists after container stops  
tmpfs Mount:      Data saved in RAM      → GONE when container stops ❌
```

---

## When to Use tmpfs?

| Use Case | Why tmpfs? |
|----------|-----------|
| Sensitive data (passwords, tokens) | Never written to disk = harder to leak |
| Temporary processing files | Fast I/O, auto-cleanup on stop |
| Session data | Don't need persistence |
| Scratch space for builds | Speed over persistence |

---

## Using tmpfs Mounts

```bash
# Using --tmpfs flag (simple)
docker run -d \
    --name myapp \
    --tmpfs /tmp \
    myimage

# Using --mount flag (more control)
docker run -d \
    --name myapp \
    --mount type=tmpfs,target=/tmp,tmpfs-size=100m \
    myimage

# With size limit and permissions
docker run -d \
    --name myapp \
    --mount type=tmpfs,target=/app/temp,tmpfs-size=64m,tmpfs-mode=1770 \
    myimage
```

### tmpfs Options

| Option | Description | Example |
|--------|-------------|---------|
| `tmpfs-size` | Maximum size in bytes | `tmpfs-size=100m` (100 MB) |
| `tmpfs-mode` | File permissions (octal) | `tmpfs-mode=1770` |

---

## Real-World Example: Secure Temp Storage

```bash
# Application that processes sensitive files temporarily
docker run -d \
    --name processor \
    -v input-data:/app/input:ro \
    --mount type=tmpfs,target=/app/temp,tmpfs-size=256m \
    -v output-data:/app/output \
    myprocessor

# Flow:
# 1. Read encrypted files from /app/input (named volume, read-only)
# 2. Decrypt and process in /app/temp (tmpfs — never hits disk!)
# 3. Write results to /app/output (named volume, persistent)
# 4. When container stops, /app/temp contents vanish from RAM
```

---

## ⚠️ tmpfs Limitations

| Limitation | Details |
|-----------|---------|
| Linux only | Not available on Docker Desktop for Windows/Mac |
| Not shared | Cannot be shared between containers |
| Not persistent | Data is lost when container stops |
| Uses host RAM | Large tmpfs mounts reduce available memory |

# 📋 Part 4: Volumes in Docker Compose

---

Volumes are most commonly used in Docker Compose files. Here's how:

## Named Volumes in Compose

```yaml
# docker-compose.yml
version: "3.8"

services:
  db:
    image: postgres:15
    volumes:
      - postgres-data:/var/lib/postgresql/data  # Named volume
    environment:
      POSTGRES_PASSWORD: secret

  app:
    image: myapp:latest
    volumes:
      - app-uploads:/app/uploads  # Named volume for user uploads
    depends_on:
      - db

# Declare named volumes at the bottom
volumes:
  postgres-data:     # Docker manages this volume
  app-uploads:       # Docker manages this volume
```

## Bind Mounts in Compose (for Development)

```yaml
# docker-compose.dev.yml
version: "3.8"

services:
  app:
    build: .
    volumes:
      - ./src:/app/src          # Bind mount: host's ./src → container's /app/src
      - ./config:/app/config:ro # Bind mount: read-only config
      - /app/node_modules       # Anonymous volume: keep node_modules in container
    ports:
      - "3000:3000"
    command: npm run dev
```

## tmpfs in Compose

```yaml
# docker-compose.yml
version: "3.8"

services:
  app:
    image: myapp:latest
    tmpfs:
      - /tmp
      - /app/temp:size=100m
```

## All Three Types Together

```yaml
# docker-compose.yml — Production
version: "3.8"

services:
  web:
    image: nginx:alpine
    volumes:
      - static-files:/usr/share/nginx/html:ro   # Named volume (read-only)
      - ./nginx.conf:/etc/nginx/nginx.conf:ro    # Bind mount (config file)
    tmpfs:
      - /var/cache/nginx                          # tmpfs for Nginx cache
    ports:
      - "80:80"

  api:
    image: myapi:latest
    volumes:
      - uploads:/app/uploads                      # Named volume for uploads
    tmpfs:
      - /tmp:size=64m                             # tmpfs for temp processing

  db:
    image: postgres:15
    volumes:
      - db-data:/var/lib/postgresql/data          # Named volume (critical data!)
    environment:
      POSTGRES_PASSWORD: ${DB_PASSWORD}

volumes:
  static-files:
  uploads:
  db-data:
```

# 🛠️ Part 5: Volume Backup and Restore

---

Named volumes store data on the host, but how do you back them up?

## Backup a Volume

```bash
# Strategy: Run a temporary container that mounts the volume AND a bind mount,
# then tar the volume contents to the bind mount

docker run --rm \
    -v my-data:/source:ro \
    -v $(pwd):/backup \
    alpine tar czf /backup/my-data-backup.tar.gz -C /source .

# Explanation:
# --rm                  → Remove container after it finishes
# -v my-data:/source:ro → Mount the volume we want to backup (read-only)
# -v $(pwd):/backup     → Bind mount current directory for the backup file
# tar czf ...           → Create compressed archive of volume contents
```

```
What's happening:

┌────────────────────────────────┐
│  Temporary Alpine Container    │
│                                │
│  /source/ ← Volume "my-data"  │──── Read data from here
│  /backup/ ← Your host's $(pwd)│──── Write backup file here
│                                │
│  tar czf /backup/backup.tar.gz │
│      -C /source .              │
└────────────────────────────────┘

Result: my-data-backup.tar.gz appears in your current directory!
```

## Restore a Volume

```bash
# Create a new volume (or use the existing one)
docker volume create my-data-restored

# Restore from backup
docker run --rm \
    -v my-data-restored:/target \
    -v $(pwd):/backup:ro \
    alpine tar xzf /backup/my-data-backup.tar.gz -C /target

# Now "my-data-restored" has all the data from the backup!
```

---

## Copy Data Between Volumes

```bash
# Copy data from volume-a to volume-b
docker run --rm \
    -v volume-a:/source:ro \
    -v volume-b:/target \
    alpine sh -c "cp -a /source/. /target/"
```

# 🧪 Part 6: Practical Scenarios

---

## Scenario 1: WordPress with Persistent Database

```yaml
# docker-compose.yml
version: "3.8"

services:
  wordpress:
    image: wordpress:latest
    ports:
      - "8080:80"
    volumes:
      - wp-content:/var/www/html/wp-content   # Persist themes, plugins, uploads
    environment:
      WORDPRESS_DB_HOST: db
      WORDPRESS_DB_PASSWORD: secret
    depends_on:
      - db

  db:
    image: mysql:8.0
    volumes:
      - db-data:/var/lib/mysql                 # Persist database files
    environment:
      MYSQL_ROOT_PASSWORD: secret
      MYSQL_DATABASE: wordpress

volumes:
  wp-content:    # WordPress uploads, themes, plugins
  db-data:       # MySQL database files
```

## Scenario 2: Development Environment with Live Reload

```yaml
# docker-compose.dev.yml
version: "3.8"

services:
  frontend:
    build:
      context: ./frontend
      dockerfile: Dockerfile.dev
    volumes:
      - ./frontend/src:/app/src          # Live code reload
      - /app/node_modules                # Keep node_modules in container
    ports:
      - "3000:3000"
    environment:
      - CHOKIDAR_USEPOLLING=true         # Enable hot reload in Docker

  backend:
    build:
      context: ./backend
      dockerfile: Dockerfile.dev
    volumes:
      - ./backend/src:/app/src           # Live code reload
      - /app/node_modules
    ports:
      - "5000:5000"

  db:
    image: postgres:15
    volumes:
      - dev-db:/var/lib/postgresql/data  # Persist DB even in development
    environment:
      POSTGRES_PASSWORD: devpassword

volumes:
  dev-db:
```

## Scenario 3: Log Aggregation

```bash
# Multiple containers write logs to a shared volume
# A log shipper container reads and forwards them

docker volume create app-logs

# App container writes logs
docker run -d --name app \
    -v app-logs:/var/log/app \
    myapp

# Log shipper reads logs and sends to ELK/Splunk
docker run -d --name log-shipper \
    -v app-logs:/var/log/app:ro \
    log-shipper-image
```

# 🗑️ Part 7: Cleaning Up Volumes

---

Volumes can accumulate over time. Here's how to manage them:

```bash
# See all volumes
docker volume ls

# See volumes that are NOT attached to any container
docker volume ls -f dangling=true

# Remove a specific volume
docker volume rm my-data

# Remove ALL unused volumes (not attached to any container)
docker volume prune

# Nuclear option: remove ALL unused Docker resources (images, containers, volumes, networks)
docker system prune --volumes
# ⚠️ WARNING: This removes ALL stopped containers, unused networks,
#             dangling images, AND unused volumes!
```

## 🔑 Key Takeaways

| Concept | Remember This |
|---------|--------------|
| **Containers are ephemeral** | Data inside a container is lost when the container is removed |
| **Named Volumes** | Docker manages them. Best for databases & persistent production data |
| **Bind Mounts** | You choose the host path. Best for development (live code reload) |
| **tmpfs** | In-memory only. Best for sensitive temporary data. Linux only |
| **Volume in Compose** | Declare volumes at the bottom of `docker-compose.yml` |
| **Backup** | Use a temp container to tar the volume contents |
| **Read-Only** | Use `:ro` to prevent containers from writing |
| **Anonymous Volume Trick** | `-v /app/node_modules` to keep node_modules inside container only |
