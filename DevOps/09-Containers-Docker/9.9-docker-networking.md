# 9.9 — Docker Networking — Bridge, Host, None, Overlay, Macvlan

> **One Line Answer:** Docker networking is how containers talk to each other and to the outside world — Docker provides five network drivers: **bridge** (default, containers on same host talk to each other), **host** (container shares the host's network directly), **none** (no networking at all), **overlay** (containers across multiple hosts talk to each other), and **macvlan** (container gets its own real IP address on the physical network).

---

## 🤔 Why Do You Need to Understand Docker Networking?

Every application needs to communicate — with other services, databases, the internet, or users. When you put apps in containers, you need to understand HOW they communicate.


# 🌉 Part 1: Bridge Network (Default)

---

## What is the Bridge Network?

The bridge network is Docker's **default** network driver. When you start a container without specifying a network, it connects to the **default bridge network**.

Think of a bridge network like a **private room with a door**:
- Containers inside the room can talk to each other
- To reach the outside world, they go through the door (the bridge)
- Outside traffic can only enter if you explicitly open a port (port mapping)

```
┌──────────────────────────────────────────────────────────────┐
│                        Host Machine                           │
│                                                               │
│  ┌─────────────────────────────────────────────────────────┐  │
│  │              Bridge Network (docker0)                    │  │
│  │              Subnet: 172.17.0.0/16                       │  │
│  │                                                          │  │
│  │  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐   │  │
│  │  │ Container A  │  │ Container B  │  │ Container C  │   │  │
│  │  │ 172.17.0.2   │  │ 172.17.0.3   │  │ 172.17.0.4   │   │  │
│  │  │              │  │              │  │              │   │  │
│  │  │ -p 8080:80   │  │ (no port map)│  │ -p 5432:5432 │   │  │
│  │  └──────────────┘  └──────────────┘  └──────────────┘   │  │
│  │              │                               │           │  │
│  └──────────────┼───────────────────────────────┼───────────┘  │
│                 │                               │              │
│              Port 8080                       Port 5432         │
│                 │                               │              │
└─────────────────┼───────────────────────────────┼──────────────┘
                  │                               │
              Internet can reach              Internet can reach
              Container A on :8080            Container C on :5432
              Container B is NOT reachable from outside
```

---

## Default Bridge vs User-Defined Bridge

Docker has TWO types of bridge networks:

### 1. Default Bridge Network (`docker0`)

This is automatically created by Docker. It has some limitations:

```bash
# Containers on the default bridge
docker run -d --name web nginx
docker run -d --name api myapi

# These containers CAN talk to each other via IP address:
docker exec web ping 172.17.0.3       # Works ✅

# But they CANNOT talk to each other by NAME:
docker exec web ping api              # FAILS ❌
# ping: bad address 'api'
```

### 2. User-Defined Bridge Network (RECOMMENDED)

```bash
# Create a custom bridge network
docker network create my-app-network

# Run containers on this network
docker run -d --name web --network my-app-network nginx
docker run -d --name api --network my-app-network myapi

# Now containers CAN talk to each other by NAME!
docker exec web ping api              # Works! ✅
docker exec api ping web              # Works! ✅
```

**This is huge!** User-defined bridge networks have **automatic DNS resolution**. Containers can find each other by name. This is the foundation of how Docker Compose works.

---

## Default Bridge vs User-Defined Bridge — Full Comparison

| Feature | Default Bridge | User-Defined Bridge |
|---------|---------------|-------------------|
| DNS resolution (by name) | ❌ No | ✅ Yes |
| Automatic connection | ✅ All containers auto-join | ❌ Must specify `--network` |
| Isolation | ❌ All containers see each other | ✅ Only containers on the same network |
| Connect/Disconnect live | ❌ Must stop container | ✅ Can connect/disconnect while running |
| Environment variable sharing | ✅ Via `--link` (deprecated) | ❌ Not needed (use DNS) |

---

## User-Defined Bridge — Full Example

Let's build a real application with a frontend, backend, and database:

```bash
# Step 1: Create a network
docker network create app-network

# Step 2: Run a PostgreSQL database
docker run -d \
    --name db \
    --network app-network \
    -e POSTGRES_PASSWORD=secret \
    -e POSTGRES_DB=myapp \
    -v db-data:/var/lib/postgresql/data \
    postgres:15

# Step 3: Run a backend API
docker run -d \
    --name api \
    --network app-network \
    -e DATABASE_URL=postgresql://postgres:secret@db:5432/myapp \
    -p 5000:5000 \
    myapi:latest

# Step 4: Run a frontend
docker run -d \
    --name web \
    --network app-network \
    -e API_URL=http://api:5000 \
    -p 80:80 \
    myfrontend:latest
```

**Notice the magic in Step 3:** `DATABASE_URL=...@db:5432/...` — the hostname `db` is automatically resolved to the database container's IP address because they're on the same user-defined bridge network!

```
┌────────────────────────────────────────────────────────┐
│                   app-network                           │
│                                                         │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐          │
│  │   web    │───→│   api    │───→│    db    │          │
│  │ :80→80   │    │ :5000    │    │ :5432    │          │
│  │          │    │          │    │          │          │
│  │ Can reach│    │ Can reach│    │ No port  │          │
│  │ api by   │    │ db by    │    │ mapping  │          │
│  │ name     │    │ name     │    │ = safe!  │          │
│  └──────────┘    └──────────┘    └──────────┘          │
│       │                                                 │
└───────┼─────────────────────────────────────────────────┘
        │
     Port 80 exposed to outside world
     (only the frontend is publicly accessible)
```

**Security bonus:** The database has NO port mapping (`-p`), so it's **only accessible from within the Docker network**. No one from outside can connect directly to it!

## Port Mapping Deep Dive

Port mapping (`-p`) is how you make containers accessible from outside:

```bash
# Format: -p HOST_PORT:CONTAINER_PORT
docker run -d -p 8080:80 nginx
#             ↑         ↑
#             │         └── Nginx listens on port 80 INSIDE the container
#             └──────────── Accessible on port 8080 on the HOST

# Multiple port mappings
docker run -d -p 80:80 -p 443:443 nginx

# Bind to a specific host interface
docker run -d -p 127.0.0.1:8080:80 nginx
#             └── Only accessible from localhost, not from other machines

# Random host port (Docker picks an available port)
docker run -d -p 80 nginx
# Check which port was assigned:
docker port <container_id>
# 80/tcp -> 0.0.0.0:49153

# Map UDP ports
docker run -d -p 53:53/udp dns-server
```

## Isolating Services with Multiple Networks

You can create multiple networks for isolation:

```bash
# Create separate networks
docker network create frontend-net
docker network create backend-net

# Frontend can ONLY reach the API
docker run -d --name web --network frontend-net -p 80:80 nginx

# API is on BOTH networks — it bridges frontend and backend
docker run -d --name api --network frontend-net myapi
docker network connect backend-net api    # Add to backend network too

# Database is ONLY on the backend network — frontend can't reach it!
docker run -d --name db --network backend-net postgres:15
```

```
┌─────────────────────────┐    ┌─────────────────────────┐
│     frontend-net         │    │     backend-net          │
│                          │    │                          │
│  ┌───────┐  ┌───────┐   │    │   ┌───────┐  ┌───────┐  │
│  │  web  │──│  api  │───│────│───│  api  │──│  db   │  │
│  └───────┘  └───────┘   │    │   └───────┘  └───────┘  │
│                          │    │                          │
│  web CAN reach api ✅    │    │   api CAN reach db ✅    │
│  web CANNOT reach db ❌  │    │   db CANNOT reach web ❌ │
└─────────────────────────┘    └─────────────────────────┘

api is on BOTH networks — it acts as the bridge between frontend and backend
```

This is a very common security pattern!

# 🖥️ Part 2: Host Network

---

## What is the Host Network?

When you use the **host** network, the container **shares the host's network stack directly**. There's no network isolation at all — the container uses the host's IP address and ports directly.

```
Bridge Network (default):                Host Network:

┌──────────────────────┐                ┌──────────────────────┐
│  Host: 192.168.1.100 │                │  Host: 192.168.1.100 │
│                      │                │                      │
│  ┌────────────────┐  │                │  Container directly  │
│  │  Container     │  │                │  uses host's network │
│  │  172.17.0.2:80 │  │                │                      │
│  │  (own IP!)     │  │                │  Container port 80 = │
│  └────────────────┘  │                │  Host port 80        │
│         │            │                │                      │
│  -p 8080:80 mapping  │                │  No port mapping     │
│         │            │                │  needed!             │
│  Host:8080 → Cont:80 │                │                      │
└──────────────────────┘                └──────────────────────┘

User: http://192.168.1.100:8080        User: http://192.168.1.100:80
```

---

## Using Host Network

```bash
# Run with host network
docker run -d \
    --name web \
    --network host \
    nginx

# Now Nginx is directly accessible on the HOST's port 80
# No need for -p port mapping!
curl http://localhost:80    # Works directly!
```

---

## When to Use Host Network

| Use When | Why |
|----------|-----|
| **Maximum network performance** | No NAT overhead, no bridge processing |
| **Container needs to bind to many ports** | No need to map each port individually |
| **Monitoring tools** | Tools like Prometheus node-exporter that need to see host's network |
| **Network debugging** | Container can see all host network interfaces |

## When NOT to Use Host Network

| Avoid When | Why |
|-----------|-----|
| **Running multiple containers** | Port conflicts! Two containers can't both use port 80 |
| **Security matters** | No network isolation — container sees everything on the host |
| **Running on Mac/Windows** | Host networking only works on Linux (Docker Desktop limitation) |
| **Multi-tenant environments** | Containers shouldn't see each other's traffic |

---

## Host Network Example: Prometheus Node Exporter

```bash
# Node Exporter needs access to host's /proc and /sys
# Host network is perfect for this
docker run -d \
    --name node-exporter \
    --network host \
    --pid host \
    -v /proc:/host/proc:ro \
    -v /sys:/host/sys:ro \
    prom/node-exporter \
    --path.procfs=/host/proc \
    --path.sysfs=/host/sys

# Accessible at http://localhost:9100/metrics
```

# 🚫 Part 3: None Network

---

## What is the None Network?

The **none** network driver gives the container **absolutely NO networking**. The container is completely disconnected from everything — no network interface, no internet, no communication with other containers.

```
None Network:

┌──────────────────────────────┐
│  Host Machine                 │
│                               │
│  ┌─────────────────────────┐  │
│  │  Container               │  │
│  │                          │  │
│  │  lo (loopback) only      │  │
│  │  NO eth0                 │  │
│  │  NO internet             │  │
│  │  NO other containers     │  │
│  │                          │  │
│  │  🔒 Completely isolated  │  │
│  └─────────────────────────┘  │
│                               │
└──────────────────────────────┘
```

---

## Using None Network

```bash
# Run a container with no networking
docker run -d \
    --name isolated-task \
    --network none \
    alpine sleep 3600

# Verify — no network interfaces (except loopback)
docker exec isolated-task ip addr
# 1: lo: <LOOPBACK,UP,LOWER_UP>
#     inet 127.0.0.1/8
# That's it! No eth0, no external connectivity.

# Try to ping anything — fails!
docker exec isolated-task ping google.com
# ping: bad address 'google.com'   ← No DNS, no network!
```

---

## When to Use None Network

| Use Case | Why |
|----------|-----|
| **Data processing that doesn't need network** | Security — no accidental data leaks |
| **Running untrusted code** | Sandboxing — code can't call home |
| **Batch jobs / computations** | Process data from a volume, no network needed |
| **Generating random numbers / crypto keys** | Extra security layer |
| **Compliance requirements** | Some regulations require network isolation |

### Example: Process Sensitive Data Without Network

```bash
# This container processes files but has NO way to send data anywhere
docker run --rm \
    --network none \
    -v sensitive-data:/input:ro \
    -v results:/output \
    data-processor
```

Even if the container is compromised, it **cannot** communicate with the outside world. The data can only leave through the mounted volume.

# 🌐 Part 4: Overlay Network

---

## What is the Overlay Network?

The overlay network enables **containers on DIFFERENT Docker hosts** to communicate with each other as if they were on the same network. This is used in **Docker Swarm** and is the foundation for multi-host container orchestration.

```
Without Overlay (containers on different hosts can't talk):

Host 1 (192.168.1.10)          Host 2 (192.168.1.20)
┌──────────────────┐           ┌──────────────────┐
│ ┌──────────────┐ │           │ ┌──────────────┐ │
│ │ Container A  │ │     ❌    │ │ Container B  │ │
│ │ 172.17.0.2   │ │ ────────→ │ │ 172.17.0.2   │ │
│ │ (frontend)   │ │  Can't    │ │ (backend)    │ │
│ └──────────────┘ │  reach!   │ └──────────────┘ │
└──────────────────┘           └──────────────────┘


With Overlay (containers across hosts can talk):

Host 1 (192.168.1.10)          Host 2 (192.168.1.20)
┌──────────────────┐           ┌──────────────────┐
│ ┌──────────────┐ │           │ ┌──────────────┐ │
│ │ Container A  │ │     ✅    │ │ Container B  │ │
│ │ 10.0.1.2     │ │ ════════> │ │ 10.0.1.3     │ │
│ │ (frontend)   │ │  Overlay  │ │ (backend)    │ │
│ └──────────────┘ │  Network  │ └──────────────┘ │
│                  │  (VXLAN)  │                  │
└──────────────────┘           └──────────────────┘
         │                              │
         └──────── Physical Network ────┘
                   (192.168.1.0/24)
```

---

## How Overlay Networks Work (Under the Hood)

Overlay networks use a technology called **VXLAN** (Virtual Extensible LAN) to encapsulate container traffic inside regular network packets:

```
Container A sends a packet to Container B:

1. Container A creates packet:
   ┌──────────────────────────────────┐
   │ Src: 10.0.1.2  Dst: 10.0.1.3   │  ← Overlay network IPs
   │ Payload: "Hello Container B!"    │
   └──────────────────────────────────┘

2. Docker wraps it in a VXLAN header + regular IP packet:
   ┌──────────────────────────────────────────────────────┐
   │ Outer Src: 192.168.1.10  Outer Dst: 192.168.1.20   │  ← Physical IPs
   │ ┌────────────────────────────────────────────────┐   │
   │ │ VXLAN Header                                    │  │
   │ │ ┌──────────────────────────────────────────┐    │  │
   │ │ │ Src: 10.0.1.2  Dst: 10.0.1.3            │    │  │ ← Original packet
   │ │ │ Payload: "Hello Container B!"             │    │  │
   │ │ └──────────────────────────────────────────┘    │  │
   │ └────────────────────────────────────────────────┘   │
   └──────────────────────────────────────────────────────┘

3. Sent over the physical network to Host 2
4. Host 2 unwraps the VXLAN → delivers to Container B
```

---

## Creating Overlay Networks

Overlay networks require **Docker Swarm** to be initialized:

```bash
# Step 1: Initialize Docker Swarm on the manager node
docker swarm init --advertise-addr 192.168.1.10

# Output includes a join token:
# docker swarm join --token SWMTKN-1-xxx 192.168.1.10:2377

# Step 2: Join worker nodes
# (Run this on the other machines)
docker swarm join --token SWMTKN-1-xxx 192.168.1.10:2377

# Step 3: Create an overlay network
docker network create \
    --driver overlay \
    --subnet 10.0.1.0/24 \
    my-overlay-network

# Step 4: Deploy services on the overlay network
docker service create \
    --name web \
    --network my-overlay-network \
    --replicas 3 \
    -p 80:80 \
    nginx

docker service create \
    --name api \
    --network my-overlay-network \
    --replicas 2 \
    myapi:latest
```

### Attachable Overlay Networks

By default, overlay networks only work with Swarm services. To use them with standalone containers too:

```bash
# Create an attachable overlay network
docker network create \
    --driver overlay \
    --attachable \
    my-attachable-overlay

# Now standalone containers can join this network
docker run -d --name debug-tool --network my-attachable-overlay alpine sleep 3600
```

---

## When to Use Overlay Network

| Use Case | Why |
|----------|-----|
| **Docker Swarm clusters** | Services across multiple nodes need to communicate |
| **Multi-host applications** | Frontend on Node 1, Backend on Node 2, DB on Node 3 |
| **Service discovery** | Built-in DNS for service names |
| **Load balancing** | Swarm routes traffic to healthy containers |

---

## Overlay Network — Real Example

```bash
# Create a production overlay network
docker network create --driver overlay production

# Deploy a full stack across the cluster
docker service create \
    --name frontend \
    --network production \
    --replicas 3 \
    --publish 80:80 \
    myfrontend:latest

docker service create \
    --name backend \
    --network production \
    --replicas 2 \
    --env DATABASE_URL=postgresql://user:pass@db:5432/myapp \
    mybackend:latest

docker service create \
    --name db \
    --network production \
    --replicas 1 \
    --mount type=volume,source=db-data,target=/var/lib/postgresql/data \
    postgres:15

# All three services can reach each other by name
# across ANY node in the Swarm cluster!
```

# 📍 Part 5: Macvlan Network

---

## What is Macvlan?

Macvlan gives each container its **own MAC address** and **its own IP address on the physical network**. The container appears as a **real physical device** on your LAN.

```
Bridge Network:                         Macvlan Network:

Physical Network: 192.168.1.0/24       Physical Network: 192.168.1.0/24

┌──────────────────────┐                ┌──────────────────────┐
│  Host: 192.168.1.100 │                │  Host: 192.168.1.100 │
│  ┌────────────────┐  │                │                      │
│  │  Container     │  │                │  Container appears   │
│  │  172.17.0.2    │  │                │  as a REAL device:   │
│  │  (private IP)  │  │                │  192.168.1.101       │
│  │  NOT on LAN    │  │                │  (ON the LAN!)       │
│  └────────────────┘  │                │                      │
│  Need port mapping   │                │  No port mapping     │
│  to reach container  │                │  needed — it has a   │
│                      │                │  real LAN IP!        │
└──────────────────────┘                └──────────────────────┘

Other LAN devices CAN'T                 Other LAN devices CAN
directly reach the container            directly reach the container
(must go through host's port)           at 192.168.1.101
```

---

## Creating a Macvlan Network

```bash
# Create a macvlan network
docker network create \
    --driver macvlan \
    --subnet 192.168.1.0/24 \
    --gateway 192.168.1.1 \
    --ip-range 192.168.1.200/29 \
    -o parent=eth0 \
    my-macvlan

# Explanation:
# --subnet        → Your physical network's subnet
# --gateway       → Your router's IP
# --ip-range      → Range of IPs Docker can assign to containers
#                    (192.168.1.200-207 in this case)
# -o parent=eth0  → The host's physical network interface
```

```bash
# Run a container on the macvlan network
docker run -d \
    --name my-server \
    --network my-macvlan \
    --ip 192.168.1.201 \
    nginx

# This container is now accessible at 192.168.1.201
# from ANY device on your LAN!
# No port mapping needed.
```

---

## When to Use Macvlan

| Use Case | Why |
|----------|-----|
| **Legacy applications** | Apps that expect to have a real network interface with a unique IP |
| **Applications requiring direct LAN access** | DHCP servers, network monitoring tools |
| **Replacing physical servers with containers** | Container gets the same IP the old server had |
| **Network appliances** | Firewalls, load balancers running in containers |

## Macvlan Limitations

| Limitation | Details |
|-----------|---------|
| **Host can't reach container** | The host CANNOT communicate with its own macvlan containers (by default) |
| **Promiscuous mode** | The host NIC must support promiscuous mode |
| **Cloud incompatibility** | Most cloud providers (AWS, GCP, Azure) don't allow macvlan |
| **IP management** | You must manually manage IPs to avoid conflicts on your LAN |

### Solving the "Host Can't Reach Container" Problem

```bash
# Create a macvlan sub-interface on the host
ip link add macvlan-shim link eth0 type macvlan mode bridge
ip addr add 192.168.1.199/32 dev macvlan-shim
ip link set macvlan-shim up

# Add a route to the container's IP range via the shim
ip route add 192.168.1.200/29 dev macvlan-shim

# Now the host can reach its own macvlan containers!
```

# 🐳 Part 6: Docker Compose Networking

---

Docker Compose handles networking automatically and this is how most people use Docker networking in practice.

## Default Compose Networking

When you run `docker compose up`, Docker Compose automatically:
1. Creates a **user-defined bridge network** named `<project>_default`
2. Connects ALL services to this network
3. Enables **DNS resolution** by service name

```yaml
# docker-compose.yml
version: "3.8"

services:
  web:
    image: nginx
    ports:
      - "80:80"

  api:
    image: myapi
    ports:
      - "5000:5000"

  db:
    image: postgres:15
    environment:
      POSTGRES_PASSWORD: secret
```

```bash
docker compose up -d

# Docker automatically creates: myproject_default network
# All three services can reach each other by name:
# - web can reach api at http://api:5000
# - api can reach db at postgresql://db:5432
# - web can reach db at postgresql://db:5432
```

---

## Custom Networks in Compose

```yaml
# docker-compose.yml
version: "3.8"

services:
  web:
    image: nginx
    ports:
      - "80:80"
    networks:
      - frontend

  api:
    image: myapi
    networks:
      - frontend    # Can be reached by web
      - backend     # Can reach db

  db:
    image: postgres:15
    environment:
      POSTGRES_PASSWORD: secret
    networks:
      - backend     # Only reachable by api, NOT by web!

  redis:
    image: redis:7
    networks:
      - backend     # Only reachable by api

networks:
  frontend:          # Web-facing services
  backend:           # Internal services only
```

## Custom Network Configuration

```yaml
# docker-compose.yml
version: "3.8"

services:
  web:
    image: nginx
    networks:
      app-net:
        ipv4_address: 172.28.0.10    # Static IP (optional)

  api:
    image: myapi
    networks:
      app-net:
        aliases:
          - backend-service           # Additional DNS name
          - api-server

networks:
  app-net:
    driver: bridge
    ipam:
      config:
        - subnet: 172.28.0.0/16      # Custom subnet
          gateway: 172.28.0.1
    driver_opts:
      com.docker.network.bridge.name: my-custom-bridge
```

## Using External (Pre-Existing) Networks

```yaml
# docker-compose.yml
version: "3.8"

services:
  web:
    image: nginx
    networks:
      - existing-network

networks:
  existing-network:
    external: true    # Don't create this network, it must already exist
    name: my-pre-existing-network
```

```bash
# Create the network first
docker network create my-pre-existing-network

# Then run compose
docker compose up -d
```

# 🔍 Part 7: Docker DNS & Service Discovery

---

## How DNS Works in Docker

On user-defined networks, Docker runs an **embedded DNS server** at `127.0.0.11` that resolves container names to IP addresses.

```bash
# Create a network
docker network create mynet

# Run two containers
docker run -d --name web --network mynet nginx
docker run -d --name api --network mynet myapi

# Check DNS resolution from inside a container
docker exec web nslookup api
# Server:    127.0.0.11       ← Docker's embedded DNS server
# Address:   127.0.0.11:53
#
# Name:      api
# Address:   172.20.0.3       ← Container 'api' resolves to this IP

# You can also use dig
docker exec web dig api
# ;; ANSWER SECTION:
# api.    600    IN    A    172.20.0.3
```

---

## Network Aliases

You can give containers additional DNS names:

```bash
# Run with an alias
docker run -d \
    --name my-postgres-container \
    --network mynet \
    --network-alias db \
    --network-alias database \
    postgres:15

# Now this container can be reached by THREE names:
# - my-postgres-container (container name)
# - db                    (alias)
# - database              (alias)
```

This is useful when you want **multiple containers to share a name** for load balancing:

```bash
# Run 3 web servers with the same alias
docker run -d --name web1 --network mynet --network-alias web nginx
docker run -d --name web2 --network mynet --network-alias web nginx
docker run -d --name web3 --network mynet --network-alias web nginx

# DNS resolution for "web" returns ALL three IPs (round-robin)
docker exec some-client nslookup web
# Name:      web
# Address:   172.20.0.5
# Address:   172.20.0.6
# Address:   172.20.0.7
```

# 🛠️ Part 8: Troubleshooting Docker Networking

---

## Common Debugging Commands

```bash
# 1. Inspect a network — see all connected containers and their IPs
docker network inspect bridge

# 2. Check a container's network configuration
docker inspect --format='{{json .NetworkSettings.Networks}}' my-container | python -m json.tool

# 3. See which ports are mapped
docker port my-container
# 80/tcp -> 0.0.0.0:8080

# 4. Test connectivity from inside a container
docker exec -it my-container ping other-container
docker exec -it my-container curl http://api:5000/health
docker exec -it my-container nslookup db

# 5. Use a debug container with network tools
docker run -it --rm --network my-network nicolaka/netshoot
# This image has: curl, dig, nslookup, ping, traceroute, tcpdump, iperf, etc.

# 6. Check iptables rules (on the host)
sudo iptables -t nat -L -n    # See Docker's NAT rules
sudo iptables -L -n            # See Docker's filter rules
```

---

## Common Network Problems and Solutions

### Problem 1: Container Can't Reach Another Container by Name

```
Symptom: ping api → "bad address 'api'"
```

| Possible Cause | Solution |
|---------------|---------|
| Using default bridge network | Switch to a user-defined bridge network |
| Containers on different networks | Put them on the same network |
| Typo in container name | Check with `docker ps` |

```bash
# Fix: Create a user-defined network
docker network create mynet
docker run -d --name api --network mynet myapi
docker run -d --name web --network mynet nginx
# Now: docker exec web ping api → works!
```

### Problem 2: Port Already in Use

```
Symptom: "Bind for 0.0.0.0:80 failed: port is already allocated"
```

```bash
# Find what's using the port
sudo lsof -i :80
# or
sudo ss -tlnp | grep :80

# Solutions:
# 1. Stop the other process
# 2. Use a different host port: -p 8080:80 instead of -p 80:80
```

### Problem 3: Container Can't Reach the Internet

```bash
# Check DNS from inside the container
docker exec my-container cat /etc/resolv.conf

# Check if the container can reach the gateway
docker exec my-container ping 172.17.0.1

# Fix: Add DNS servers manually
docker run -d --dns 8.8.8.8 --dns 8.8.4.4 myapp
```

### Problem 4: Container Can't Be Reached from Outside

```bash
# Verify port mapping
docker port my-container

# Check firewall rules
sudo iptables -L -n | grep -i docker
sudo ufw status

# Check if Docker is listening on the port
sudo ss -tlnp | grep docker
```

## 🔑 Key Takeaways

| Concept | Remember This |
|---------|--------------|
| **Bridge (Default)** | Containers on same host talk to each other. Use **user-defined** bridge for DNS by name |
| **Host** | No isolation. Container uses host's network directly. Best for performance. Linux only |
| **None** | Total isolation. No networking at all. Use for security-sensitive offline tasks |
| **Overlay** | Containers across multiple hosts. Requires Docker Swarm. Uses VXLAN tunneling |
| **Macvlan** | Container gets a real LAN IP. Looks like a physical device. For legacy apps |
| **User-defined > Default** | Always create your own bridge network — you get DNS resolution by container name |
| **Port Mapping** | `-p HOST:CONTAINER` — only way outside world reaches bridge-networked containers |
| **Network Isolation** | Use multiple networks to control which services can talk to each other |
| **Docker Compose** | Auto-creates a user-defined bridge network. Services can reach each other by name |
