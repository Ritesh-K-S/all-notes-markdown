# 10.10 — DaemonSets — Running Pods on Every Node

> **One Line Answer:** A DaemonSet makes sure that a **copy of a specific Pod runs on every single node** (or a selected set of nodes) in your Kubernetes cluster — automatically.

---

## 🤔 What Problem Does a DaemonSet Solve?

Imagine you have a Kubernetes cluster with **10 worker nodes**. You want to run a **log collector** (like Fluentd) on **every single node** so it can collect logs from all the applications running on that node.

Now, you could manually create 10 individual Pods — one for each node. But what happens when:
- A **new node** is added to the cluster? → You'd have to manually deploy a Pod on it.
- A **node is removed**? → You'd need to clean up.
- A **Pod crashes** on one node? → You'd have to restart it yourself.

That's a nightmare to manage. 😩

**This is exactly what a DaemonSet does for you — automatically.**

| Without DaemonSet | With DaemonSet |
|-------------------|----------------|
| You manually deploy Pods on each node | Kubernetes does it automatically |
| New node added? You must remember to deploy | New node gets a Pod automatically |
| Node removed? Orphan cleanup is your job | Pod is removed automatically |
| Pod crashes? You restart manually | Kubernetes restarts it automatically |
| Scale from 10 to 50 nodes? Painful | Seamless — 40 new Pods auto-created |

---

## 📦 What Exactly is a DaemonSet?

A **DaemonSet** is a Kubernetes workload resource that ensures:

> **Exactly one copy of a Pod runs on every node** (or on a selected subset of nodes) in the cluster.

Think of it like a **"one Pod per node" rule** that Kubernetes enforces for you.

### Real-World Analogy

Think of a DaemonSet like **security cameras in a building**:
- Every floor (node) MUST have a security camera (Pod).
- When a new floor is built (new node added), a camera is automatically installed.
- When a floor is demolished (node removed), the camera is removed.
- If a camera breaks, it's replaced immediately.

You don't manually go floor-to-floor — the building management system handles it.

**That's a DaemonSet!**

---

## 🔧 Common Use Cases for DaemonSets

DaemonSets are used when you need **something running on every node**. Here are the most common use cases:

| Use Case | Example Tool | Why? |
|----------|-------------|------|
| **Log Collection** | Fluentd, Filebeat, Fluent Bit | Collect logs from all Pods on each node |
| **Monitoring / Metrics** | Prometheus Node Exporter, Datadog Agent | Gather CPU, memory, disk metrics from each node |
| **Networking** | Calico, Cilium, Weave Net, kube-proxy | Manage network rules on every node |
| **Storage** | GlusterFS, Ceph | Provide distributed storage from each node |
| **Security** | Falco, Tetragon | Monitor system calls for security threats on every node |
| **Node Management** | Node Problem Detector | Detect hardware/software issues on each node |

### 💡 The Key Insight

All of these tools have one thing in common: **they need to run on EVERY node** to do their job properly. A log collector only on 3 out of 10 nodes means you're missing logs from 7 nodes!

## 📝 DaemonSet YAML — Your First DaemonSet

Let's create a DaemonSet that runs an **Nginx Pod on every node** in the cluster (for learning purposes):

```yaml
# daemonset-example.yaml
apiVersion: apps/v1          # API version for DaemonSet
kind: DaemonSet              # We're creating a DaemonSet (not Deployment, not StatefulSet)
metadata:
  name: nginx-daemonset      # Name of this DaemonSet
  labels:
    app: nginx-daemon         # Labels for organizing
spec:
  selector:
    matchLabels:
      app: nginx-daemon       # Must match the Pod template labels below
  template:                   # Pod template — defines what Pod to run
    metadata:
      labels:
        app: nginx-daemon     # Labels on the Pods created by this DaemonSet
    spec:
      containers:
      - name: nginx
        image: nginx:1.25     # Container image to use
        ports:
        - containerPort: 80   # Port the container listens on
        resources:
          limits:
            memory: "128Mi"
            cpu: "250m"
          requests:
            memory: "64Mi"
            cpu: "100m"
```

### Let's Break This Down Line by Line

| Field | What It Does |
|-------|-------------|
| `apiVersion: apps/v1` | DaemonSets are part of the `apps` API group |
| `kind: DaemonSet` | Tells Kubernetes "this is a DaemonSet" |
| `metadata.name` | The name of your DaemonSet |
| `spec.selector.matchLabels` | How the DaemonSet finds which Pods belong to it |
| `spec.template` | The Pod blueprint — what container to run |
| `spec.template.metadata.labels` | Labels on the Pods (MUST match `selector.matchLabels`) |
| `spec.template.spec.containers` | The actual containers to run in each Pod |
| `resources.requests` | Minimum resources the Pod needs |
| `resources.limits` | Maximum resources the Pod can use |

### Apply and Verify

```bash
# Create the DaemonSet
kubectl apply -f daemonset-example.yaml

# Check the DaemonSet
kubectl get daemonset
# or shorthand
kubectl get ds
```

**Output:**

```
NAME               DESIRED   CURRENT   READY   UP-TO-DATE   AVAILABLE   NODE SELECTOR   AGE
nginx-daemonset    3         3         3       3            3           <none>           30s
```

Let's understand each column:

| Column | Meaning |
|--------|---------|
| `DESIRED` | How many nodes should have this Pod (= total number of eligible nodes) |
| `CURRENT` | How many Pods currently exist |
| `READY` | How many Pods are Ready (passing health checks) |
| `UP-TO-DATE` | How many Pods have the latest Pod template |
| `AVAILABLE` | How many Pods are available to serve |
| `NODE SELECTOR` | If you're targeting specific nodes (we'll cover this below) |

```bash
# See the Pods created by the DaemonSet
kubectl get pods -o wide
```

**Output:**

```
NAME                    READY   STATUS    RESTARTS   AGE   IP           NODE
nginx-daemonset-abc12   1/1     Running   0          30s   10.244.1.5   worker-1
nginx-daemonset-def34   1/1     Running   0          30s   10.244.2.8   worker-2
nginx-daemonset-ghi56   1/1     Running   0          30s   10.244.3.3   worker-3
```

Notice: **One Pod per node**, each with a unique name (auto-generated by adding a random suffix).

## 🔍 DaemonSet vs Deployment — What's the Difference?

This is a very common question. Let's clear it up:

| Feature | Deployment | DaemonSet |
|---------|-----------|-----------|
| **How many Pods?** | You specify `replicas: N` | One Pod per node (automatic) |
| **Where do Pods run?** | Scheduler decides (could be any node) | Exactly one on each node |
| **Scaling** | You change `replicas` count | Automatic — based on number of nodes |
| **Use Case** | Application workloads (web apps, APIs) | Node-level agents (logging, monitoring) |
| **New node added** | Nothing happens (unless HPA triggers) | New Pod automatically created |
| **Pod scheduling** | kube-scheduler places Pods | DaemonSet controller places Pods |

### Visual Comparison

```
DEPLOYMENT (replicas: 3):
┌──────────┐  ┌──────────┐  ┌──────────┐
│  Node 1  │  │  Node 2  │  │  Node 3  │
│ [Pod][Pod]│  │  [Pod]   │  │          │
│           │  │          │  │          │
└──────────┘  └──────────┘  └──────────┘
→ 3 Pods total, but scheduler decides where. Node 3 might have ZERO Pods.

DAEMONSET:
┌──────────┐  ┌──────────┐  ┌──────────┐
│  Node 1  │  │  Node 2  │  │  Node 3  │
│  [Pod]   │  │  [Pod]   │  │  [Pod]   │
│           │  │          │  │          │
└──────────┘  └──────────┘  └──────────┘
→ Exactly ONE Pod per node. Always.
```

## 🎯 Running DaemonSet on Specific Nodes Only

By default, a DaemonSet runs a Pod on **every node**. But sometimes you only want it on **certain nodes**. For example:
- Run a GPU monitoring agent only on nodes with GPUs.
- Run a special logging agent only on nodes labeled `environment=production`.

### Method 1: Using `nodeSelector`

First, label your nodes:

```bash
# Add a label to a node
kubectl label nodes worker-1 disk=ssd
kubectl label nodes worker-2 disk=ssd
# worker-3 does NOT have this label
```

Then use `nodeSelector` in your DaemonSet:

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: ssd-monitor
spec:
  selector:
    matchLabels:
      app: ssd-monitor
  template:
    metadata:
      labels:
        app: ssd-monitor
    spec:
      nodeSelector:
        disk: ssd              # Only run on nodes with label disk=ssd
      containers:
      - name: ssd-monitor
        image: busybox
        command: ["sh", "-c", "echo Monitoring SSD on $(hostname); sleep 3600"]
```

**Result:** Pods will only run on `worker-1` and `worker-2` (the nodes with `disk=ssd` label), NOT on `worker-3`.

### Method 2: Using `affinity` (More Powerful)

Node affinity gives you more flexible rules:

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: gpu-monitor
spec:
  selector:
    matchLabels:
      app: gpu-monitor
  template:
    metadata:
      labels:
        app: gpu-monitor
    spec:
      affinity:
        nodeAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
            nodeSelectorTerms:
            - matchExpressions:
              - key: gpu-type
                operator: In
                values:
                - nvidia
                - amd
      containers:
      - name: gpu-monitor
        image: gpu-monitor:latest
        resources:
          limits:
            memory: "256Mi"
```

This says: "Only schedule on nodes where `gpu-type` label is either `nvidia` or `amd`."

### Method 3: Using Tolerations (Run on Control Plane Nodes)

By default, **DaemonSet Pods do NOT run on control plane (master) nodes** because those nodes have a **taint** that repels regular Pods. But sometimes you need a DaemonSet on every node INCLUDING the control plane (e.g., for networking or monitoring).

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: monitoring-agent
spec:
  selector:
    matchLabels:
      app: monitoring-agent
  template:
    metadata:
      labels:
        app: monitoring-agent
    spec:
      tolerations:
      # This toleration allows the Pod to run on control plane nodes
      - key: node-role.kubernetes.io/control-plane
        operator: Exists
        effect: NoSchedule
      # Older clusters may use this taint name
      - key: node-role.kubernetes.io/master
        operator: Exists
        effect: NoSchedule
      containers:
      - name: monitor
        image: prom/node-exporter:latest
        ports:
        - containerPort: 9100
```

Now the monitoring agent runs on **ALL nodes** — workers AND control plane.

## 🔄 DaemonSet Update Strategies

When you update a DaemonSet (e.g., change the container image), how should Kubernetes roll out the change?

There are **two update strategies**:

### 1. `RollingUpdate` (Default)

Old Pods are replaced with new Pods one node at a time.

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: fluentd
spec:
  updateStrategy:
    type: RollingUpdate        # Default — updates one by one
    rollingUpdate:
      maxUnavailable: 1        # Maximum Pods that can be unavailable during update
      maxSurge: 0              # (K8s 1.22+) Max extra Pods during update
  selector:
    matchLabels:
      app: fluentd
  template:
    metadata:
      labels:
        app: fluentd
    spec:
      containers:
      - name: fluentd
        image: fluent/fluentd:v1.16    # Change this to trigger a rolling update
```

**How it works:**
1. Kubernetes kills the old Pod on Node 1.
2. Starts a new Pod with the updated image on Node 1.
3. Waits for it to be Ready.
4. Moves on to Node 2.
5. Repeats until all nodes have the updated Pod.

| `maxUnavailable` Setting | Meaning |
|--------------------------|---------|
| `1` (default) | Update one node at a time |
| `2` | Update two nodes simultaneously |
| `"25%"` | Update 25% of nodes at once |
| `"100%"` | Update all nodes at once (similar to OnDelete but automated) |

### 2. `OnDelete`

Pods are NOT automatically updated. You must **manually delete** the old Pod, and only then does the DaemonSet create a new one with the updated spec.

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: critical-agent
spec:
  updateStrategy:
    type: OnDelete              # Only update when you manually delete old Pods
  selector:
    matchLabels:
      app: critical-agent
  template:
    metadata:
      labels:
        app: critical-agent
    spec:
      containers:
      - name: agent
        image: critical-agent:v2
```

**When to use `OnDelete`?**
- When the agent is so critical that you want **full manual control** over when each node gets updated.
- When you want to do a node-by-node canary update manually.

```bash
# With OnDelete strategy, update process is manual:
# 1. Update the DaemonSet spec (change image, etc.)
kubectl apply -f critical-agent-daemonset.yaml

# 2. Nothing happens yet! Old Pods keep running.

# 3. Manually delete the Pod on a specific node to trigger the update:
kubectl delete pod critical-agent-abc12    # on worker-1

# 4. DaemonSet creates a new Pod on worker-1 with updated image.

# 5. Verify it works, then move to the next node:
kubectl delete pod critical-agent-def34    # on worker-2
```

## 🏗️ Real-World DaemonSet Example: Fluentd Log Collector

This is probably the most common DaemonSet you'll see in production:

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: fluentd
  namespace: kube-system          # System-level DaemonSets often go in kube-system
  labels:
    app: fluentd-logging
spec:
  selector:
    matchLabels:
      app: fluentd-logging
  updateStrategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1
  template:
    metadata:
      labels:
        app: fluentd-logging
    spec:
      tolerations:
      - key: node-role.kubernetes.io/control-plane
        operator: Exists
        effect: NoSchedule
      containers:
      - name: fluentd
        image: fluent/fluentd-kubernetes-daemonset:v1.16-debian-elasticsearch8-1
        env:
        - name: FLUENT_ELASTICSEARCH_HOST
          value: "elasticsearch.logging.svc.cluster.local"
        - name: FLUENT_ELASTICSEARCH_PORT
          value: "9200"
        resources:
          limits:
            memory: "512Mi"
            cpu: "500m"
          requests:
            memory: "200Mi"
            cpu: "100m"
        volumeMounts:
        - name: varlog
          mountPath: /var/log             # Mount host's /var/log directory
        - name: containers-log
          mountPath: /var/lib/docker/containers   # Mount container logs
          readOnly: true
      terminationGracePeriodSeconds: 30
      volumes:
      - name: varlog
        hostPath:
          path: /var/log                  # Host directory for system logs
      - name: containers-log
        hostPath:
          path: /var/lib/docker/containers  # Host directory for container logs
```

### What's Happening Here?

| Part | Explanation |
|------|-------------|
| `namespace: kube-system` | Running in the system namespace (standard for infrastructure tools) |
| `tolerations` | Allows the Pod to run even on control plane nodes |
| `env` | Passes Elasticsearch connection info as environment variables |
| `volumeMounts` / `volumes` | Mounts the **host's log directories** into the container so Fluentd can read logs |
| `hostPath` | A volume type that mounts a directory from the node's filesystem |
| `readOnly: true` | Container can read the logs but not modify them |
| `terminationGracePeriodSeconds` | Gives Fluentd 30 seconds to flush logs before shutting down |

### Why `hostPath` Volumes?

Fluentd needs to access **the actual log files on the node**, not inside the container's filesystem. By mounting `/var/log` from the host into the container, Fluentd can read all system and container logs from that specific node.

```
┌─────────── Node (worker-1) ───────────┐
│                                        │
│  /var/log/  ←── Host log files         │
│      │                                 │
│      │  (mounted into)                 │
│      ▼                                 │
│  ┌──────────────────────┐              │
│  │  Fluentd Container   │              │
│  │  /var/log/ (mounted) │──→ Sends to  │
│  │  Reads logs here     │  Elasticsearch│
│  └──────────────────────┘              │
│                                        │
└────────────────────────────────────────┘
```

```bash
kubectl rollout history ds/fluentd

kubectl rollout undo ds/fluentd

kubectl rollout undo ds/fluentd --to-revision=2

kubectl delete ds nginx-daemonset

# Update the image of a DaemonSet (triggers rolling update)
kubectl set image ds/fluentd fluentd=fluent/fluentd:v1.17

# Edit a DaemonSet directly (opens in your editor)
kubectl edit ds fluentd
```

## ⚠️ Important Things to Know About DaemonSets

### 1. DaemonSet Pods Bypass the Scheduler (Partially)

Before Kubernetes 1.12, DaemonSet Pods were created directly by the DaemonSet controller and **bypassed the kube-scheduler entirely**. Since Kubernetes 1.12+, DaemonSet Pods are scheduled by the **default scheduler** by setting `nodeAffinity` automatically, which is more consistent and predictable.

### 2. DaemonSet Pods Are Not Evicted Easily

DaemonSet Pods have a special behavior during **node drain** operations:

```bash
# When you drain a node for maintenance:
kubectl drain worker-1 --ignore-daemonsets
```

The `--ignore-daemonsets` flag is needed because **DaemonSet Pods are not evicted by drain** by default. They're considered essential to the node.

### 3. Resource Awareness

DaemonSet Pods consume resources on **every node**. If your DaemonSet Pod requests 500Mi memory and you have 100 nodes, that's **50 GB of cluster memory** just for that DaemonSet! Always set resource limits carefully.

### 4. Priority and Preemption

You can set a `priorityClassName` on DaemonSet Pods to ensure they get scheduled even when nodes are under resource pressure:

```yaml
spec:
  template:
    spec:
      priorityClassName: system-node-critical    # High priority
      containers:
      - name: critical-agent
        image: critical-agent:latest
```

## 🔑 Quick Summary

| Concept | Key Point |
|---------|-----------|
| **What** | A DaemonSet ensures one Pod runs on every (or selected) node |
| **Why** | For node-level tasks: logging, monitoring, networking, security |
| **How many Pods** | Equal to the number of eligible nodes |
| **New node** | Pod is automatically added |
| **Removed node** | Pod is automatically cleaned up |
| **Default update** | RollingUpdate — one node at a time |
| **Target nodes** | Use `nodeSelector`, `affinity`, or `tolerations` |
| **Not for** | Application workloads (use Deployments for that) |

## 🧠 Quick Comparison Table

| Resource | Purpose | Replicas | Scheduling |
|----------|---------|----------|------------|
| **Deployment** | Stateless apps | You decide (`replicas: N`) | Scheduler decides where |
| **StatefulSet** | Stateful apps (databases) | You decide (`replicas: N`) | Ordered, stable identity |
| **DaemonSet** | One per node agents | = number of nodes | One per node (automatic) |
| **ReplicaSet** | Low-level (used by Deployment) | You decide | Scheduler decides |

## ❓ Interview Questions

1. **What is a DaemonSet and when would you use one?**
   → A DaemonSet ensures one Pod runs on every node. Used for node-level agents like log collectors, monitoring agents, and networking plugins.

2. **How is a DaemonSet different from a Deployment?**
   → A Deployment runs a specific number of replicas wherever the scheduler decides. A DaemonSet runs exactly one Pod per node.

3. **How do you run a DaemonSet on control plane nodes?**
   → Add tolerations for the control plane taint (`node-role.kubernetes.io/control-plane:NoSchedule`).

4. **What happens when you add a new node to a cluster that has a DaemonSet?**
   → The DaemonSet controller automatically creates a Pod on the new node.

5. **What are the two update strategies for DaemonSets?**
   → `RollingUpdate` (default — updates one node at a time) and `OnDelete` (manual — Pod is updated only when you delete the old one).
