# 10.11 вҖ” Jobs & CronJobs вҖ” Batch Processing

> **One Line Answer:** A **Job** runs a Pod that performs a task and then **exits** (it doesn't run forever), and a **CronJob** is a Job that runs **on a schedule** (like a cron task in Linux).

---

## рҹӨ” The Problem вҖ” Not Everything Runs Forever

So far, we've learned about Deployments, StatefulSets, and DaemonSets. All of these are designed for **long-running workloads** вҖ” they run your Pods and keep them alive forever (or at least try to restart them if they crash).

But not every task is a "run forever" kind of task. Some tasks are **one-and-done**:

- Run a database migration script вҶ’ it finishes вҶ’ done вң…
- Generate a report from your database вҶ’ save the file вҶ’ done вң…
- Send out a batch of emails вҶ’ all sent вҶ’ done вң…
- Process a bunch of images вҶ’ resize them вҶ’ done вң…
- Take a database backup every night at 2 AM вҶ’ done вң…
- Clean up old log files every week вҶ’ done вң…

If you put these in a **Deployment**, Kubernetes would keep restarting them every time they finish, because Deployments think: *"Oh no, the Pod exited! Something went wrong! Let me restart it!"*

That's where **Jobs** and **CronJobs** come in.

| Resource | Behavior | Example |
|----------|----------|---------|
| **Deployment** | Runs forever, restarts if exits | Web server, API |
| **Job** | Runs once, exits when done | Database migration |
| **CronJob** | Runs on a schedule, exits each time | Nightly backup |

## рҹ“Ұ Part 1: Jobs вҖ” Run Once and Exit

### What is a Job?

A Job creates one or more Pods and ensures that a specified number of them **successfully complete** (exit with code 0). Once the task is done, the Job is considered complete.

Key properties:
- The Pod **runs to completion** (it's not restarted when it finishes successfully).
- If the Pod **fails** (exits with non-zero code), the Job **retries** it (by default).
- Once the Job succeeds, the Pods are **kept around** (in `Completed` status) so you can check their logs.

### Your First Job вҖ” Simple Example

```yaml
# job-example.yaml
apiVersion: batch/v1              # Jobs are in the "batch" API group
kind: Job
metadata:
  name: hello-job
spec:
  template:                       # Pod template (same as in Deployments)
    metadata:
      labels:
        app: hello-job
    spec:
      containers:
      - name: hello
        image: busybox
        command: ["sh", "-c", "echo 'Hello from Kubernetes Job!'; echo 'Task completed successfully!'; sleep 5"]
      restartPolicy: Never        # IMPORTANT: Must be "Never" or "OnFailure" for Jobs
  backoffLimit: 4                 # Retry up to 4 times if the Pod fails
```

### Breaking It Down

| Field | What It Does |
|-------|-------------|
| `apiVersion: batch/v1` | Jobs are in the `batch` API group (not `apps` like Deployments) |
| `kind: Job` | We're creating a Job |
| `spec.template` | The Pod specification вҖ” what container to run |
| `restartPolicy: Never` | Don't restart the container if it exits. **Required for Jobs** вҖ” you can't use `Always` (the default for Deployments) |
| `backoffLimit: 4` | If the Pod fails, retry up to 4 times before giving up |

> **вҡ пёҸ Important:** Jobs require `restartPolicy` to be either `Never` or `OnFailure`. You CANNOT use `Always` (which is the default for Deployments). This is because a Job is meant to **finish**, not run forever.

### Run It and See What Happens

```bash
# Create the Job
kubectl apply -f job-example.yaml

# Watch the Job
kubectl get jobs
```

**Output:**


Wait a few seconds...


|--------|---------|
| `COMPLETIONS` | `1/1` means 1 out of 1 required completions are done |



### `restartPolicy` вҖ” Never vs OnFailure

| `restartPolicy` | Behavior When Pod Fails | Behavior When Pod Succeeds |
|-----------------|------------------------|---------------------------|
| `Never` | Job creates a **new Pod** to retry | Pod stays in `Completed` status |
| `OnFailure` | **Same Pod** is restarted in-place | Pod stays in `Completed` status |

With `Never`, if a Pod fails 4 times (matching `backoffLimit`), you'll see 4 separate failed Pods:

```
NAME                READY   STATUS   RESTARTS   AGE
hello-job-abc12     0/1     Error    0          60s
hello-job-def34     0/1     Error    0          45s
hello-job-ghi56     0/1     Error    0          30s
hello-job-jkl78     0/1     Error    0          15s
```

With `OnFailure`, the same Pod restarts inside itself, so you see just one Pod with multiple restarts:

```
NAME              READY   STATUS             RESTARTS   AGE
hello-job-abc12   0/1     CrashLoopBackOff   4          60s
```

**Which one to use?**
- Use `Never` if you want to see logs from **each failed attempt** (each is a separate Pod).
- Use `OnFailure` if you want to keep things cleaner (one Pod, retried in place).

### Parallel Jobs вҖ” Running Multiple Pods at Once

Sometimes your task can be split into parallel chunks. Jobs support this with two settings:

| Field | Meaning | Default |
|-------|---------|---------|
| `spec.completions` | Total number of Pods that must succeed | 1 |
| `spec.parallelism` | How many Pods can run at the same time | 1 |

#### Example: Process 5 Tasks, 2 at a Time

```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: parallel-job
spec:
  completions: 5        # We need 5 Pods to complete successfully
  parallelism: 2        # Run 2 Pods at a time
  template:
    spec:
      containers:
      - name: worker
        image: busybox
        command: ["sh", "-c", "echo Processing task on $(hostname); sleep 10; echo Done"]
      restartPolicy: Never
  backoffLimit: 6
```

**What happens:**

```
Time 0s:    Pod-1 starts, Pod-2 starts        (2 running)
Time 10s:   Pod-1 done, Pod-2 done вҶ’ Pod-3 starts, Pod-4 starts  (2 running)
Time 20s:   Pod-3 done, Pod-4 done вҶ’ Pod-5 starts               (1 running)
Time 30s:   Pod-5 done вҶ’ Job Complete!         (0 running)
```

```bash
kubectl get jobs parallel-job --watch
```


### Job Types Summary

| Type | `completions` | `parallelism` | Use Case |
|------|--------------|---------------|----------|
| **Single Pod** | 1 (default) | 1 (default) | Simple one-time task |
| **Fixed Completion Count** | N | M (M вүӨ N) | Process N items, M at a time |
| **Work Queue** | not set | N | Pods pull work from a queue, exit when queue is empty |

### Automatic Cleanup вҖ” `ttlSecondsAfterFinished`

Completed Job Pods stick around forever by default (so you can check their logs). But in production, they pile up! Use `ttlSecondsAfterFinished` to auto-delete:

```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: cleanup-example
spec:
  ttlSecondsAfterFinished: 300     # Delete Job + Pods 300 seconds (5 min) after completion
  template:
    spec:
      containers:
      - name: task
        image: busybox
        command: ["echo", "Quick task done"]
      restartPolicy: Never
```

After the Job completes, Kubernetes will automatically delete it (and its Pods) after 5 minutes.

| Value | Behavior |
|-------|----------|
| `0` | Delete immediately after Job finishes |
| `300` | Delete 5 minutes after finishing |
| Not set | Job is never automatically deleted |

---

### `activeDeadlineSeconds` вҖ” Job Timeout

What if a Job hangs and never finishes? Set a deadline:

```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: timeout-job
spec:
  activeDeadlineSeconds: 120        # Kill the Job if it runs longer than 120 seconds
  backoffLimit: 3
  template:
    spec:
      containers:
      - name: long-task
        image: busybox
        command: ["sh", "-c", "echo 'Starting...'; sleep 3600"]   # Pretends to hang
      restartPolicy: Never
```

After 120 seconds, Kubernetes terminates the Job and marks it as **Failed**, regardless of how many retries are left.

## вҸ° Part 2: CronJobs вҖ” Scheduled Jobs

### What is a CronJob?

A CronJob is simply a **Job that runs on a schedule**. It's the Kubernetes equivalent of Linux's `crontab`.

Think of it this way:
- **Job** = "Do this task NOW"
- **CronJob** = "Do this task every day at 3 AM" or "every 5 minutes" or "every Monday"

A CronJob **creates Jobs** at the scheduled times. Each Job then creates Pods, which do the actual work.

```
CronJob
  в””в”Җв”Җ Creates a new Job at each scheduled time
        в””в”Җв”Җ Job creates Pod(s)
              в””в”Җв”Җ Pod does the work and exits
```

---

### Cron Schedule Syntax

If you've used Linux cron, this is the same. If not, here's the complete breakdown:

```
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ minute (0 - 59)
в”Ӯ в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ hour (0 - 23)
в”Ӯ в”Ӯ в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ day of month (1 - 31)
в”Ӯ в”Ӯ в”Ӯ в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ month (1 - 12)
в”Ӯ в”Ӯ в”Ӯ в”Ӯ в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ day of week (0 - 6, Sun = 0)
в”Ӯ в”Ӯ в”Ӯ в”Ӯ в”Ӯ
* * * * *
```

### Common Schedule Examples

| Schedule | Meaning |
|----------|---------|
| `"*/5 * * * *"` | Every 5 minutes |
| `"0 * * * *"` | Every hour (at minute 0) |
| `"0 2 * * *"` | Every day at 2:00 AM |
| `"0 0 * * 0"` | Every Sunday at midnight |
| `"0 3 1 * *"` | First day of every month at 3:00 AM |
| `"30 8 * * 1-5"` | Monday to Friday at 8:30 AM |
| `"0 0 1 1 *"` | January 1st at midnight (once a year) |
| `"*/15 9-17 * * 1-5"` | Every 15 min, 9 AMвҖ“5 PM, MonвҖ“Fri |

### Special Characters

| Character | Meaning | Example |
|-----------|---------|---------|
| `*` | Every value | `* * * * *` = every minute |
| `,` | Multiple values | `1,15 * * * *` = minute 1 and 15 |
| `-` | Range | `1-5` = 1 through 5 |
| `/` | Step | `*/10` = every 10th value |

### Your First CronJob

```yaml
# cronjob-example.yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: hello-cronjob
spec:
  schedule: "*/5 * * * *"          # Run every 5 minutes
  jobTemplate:                     # Template for the Job that will be created
    spec:
      template:                    # Template for the Pod inside the Job
        spec:
          containers:
          - name: hello
            image: busybox
            command: ["sh", "-c", "echo 'CronJob ran at:' $(date); echo 'Hello from scheduled task!'"]
          restartPolicy: OnFailure
      backoffLimit: 3
```

### Breaking It Down

Notice the structure вҖ” it's **nested**:

```
CronJob spec
  в””в”Җв”Җ schedule (when to run)
  в””в”Җв”Җ jobTemplate (template for the Job)
        в””в”Җв”Җ spec (Job spec)
              в””в”Җв”Җ template (Pod template)
                    в””в”Җв”Җ spec (Pod spec)
                          в””в”Җв”Җ containers (what to run)
```

| Field | What It Does |
|-------|-------------|
| `spec.schedule` | The cron expression вҖ” when to create Jobs |
| `spec.jobTemplate` | The entire Job definition (same as a standalone Job) |
| `spec.jobTemplate.spec.template` | The Pod template (same as always) |

### Run It

```bash
# Create the CronJob
kubectl apply -f cronjob-example.yaml

# Check the CronJob
kubectl get cronjobs
# or shorthand
kubectl get cj
```

```

```

| Column | Meaning |
|--------|---------|
| `SCHEDULE` | The cron expression |
| `SUSPEND` | Is the CronJob paused? |
| `ACTIVE` | Number of currently running Jobs |
| `LAST SCHEDULE` | When the last Job was triggered |

### CronJob Configuration Options

#### `concurrencyPolicy` вҖ” What If the Previous Job Is Still Running?

What if a scheduled Job takes longer than the interval? E.g., your Job runs every 5 minutes but sometimes takes 7 minutes.

```yaml
spec:
  schedule: "*/5 * * * *"
  concurrencyPolicy: Forbid       # Options: Allow, Forbid, Replace
```

| Policy | Behavior |
|--------|----------|
| `Allow` (default) | Multiple Jobs can run at the same time (concurrent) |
| `Forbid` | Skip the new Job if the previous one is still running |
| `Replace` | Kill the old running Job and start a new one |

**Which to use?**
- `Allow` вҖ” when multiple instances can safely run together (e.g., sending different batches of emails)
- `Forbid` вҖ” when running two at once would cause problems (e.g., database backup вҖ” two backups at once could corrupt data)
- `Replace` вҖ” when the latest run is more important than the previous one (e.g., cache refresh)

#### `startingDeadlineSeconds` вҖ” Missed Schedule Grace Period

If Kubernetes misses a scheduled time (e.g., the control plane was down), how long should it still try to create the Job?

```yaml
spec:
  schedule: "0 2 * * *"
  startingDeadlineSeconds: 200     # If missed by more than 200 seconds, skip it
```

If more than 100 schedules are missed (and `startingDeadlineSeconds` is not set), the CronJob stops running entirely. Always set this in production!

#### `successfulJobsHistoryLimit` & `failedJobsHistoryLimit`

How many completed/failed Jobs to keep:

```yaml
spec:
  successfulJobsHistoryLimit: 3    # Keep last 3 successful Jobs (default: 3)
  failedJobsHistoryLimit: 1        # Keep last 1 failed Job (default: 1)
```

Older Jobs (and their Pods) are automatically deleted.

#### `suspend` вҖ” Pause a CronJob

```yaml
spec:
  suspend: true                    # Pause вҖ” no new Jobs will be created
```

Or via kubectl:

```bash
# Pause a CronJob
kubectl patch cronjob hello-cronjob -p '{"spec":{"suspend":true}}'

# Resume it
kubectl patch cronjob hello-cronjob -p '{"spec":{"suspend":false}}'
```

### Real-World CronJob Example: Database Backup

This is probably the most classic CronJob use case:

```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: postgres-backup
  namespace: database
spec:
  schedule: "0 2 * * *"             # Every day at 2:00 AM
  concurrencyPolicy: Forbid          # Don't overlap backups
  startingDeadlineSeconds: 600       # 10-minute grace period
  successfulJobsHistoryLimit: 7      # Keep last 7 days of backups
  failedJobsHistoryLimit: 3          # Keep last 3 failures for debugging
  jobTemplate:
    spec:
      activeDeadlineSeconds: 3600    # Timeout after 1 hour
      backoffLimit: 2                # Retry up to 2 times
      template:
        spec:
          containers:
          - name: backup
            image: postgres:16
            command:
            - /bin/bash
            - -c
            - |
              set -e
              TIMESTAMP=$(date +%Y%m%d_%H%M%S)
              BACKUP_FILE="/backups/db_backup_${TIMESTAMP}.sql.gz"
              echo "Starting backup at $(date)"
              pg_dump -h $POSTGRES_HOST -U $POSTGRES_USER -d $POSTGRES_DB | gzip > $BACKUP_FILE
              echo "Backup completed: $BACKUP_FILE"
              echo "Backup size: $(du -h $BACKUP_FILE | cut -f1)"
            env:
            - name: POSTGRES_HOST
              value: "postgres-service.database.svc.cluster.local"
            - name: POSTGRES_USER
              valueFrom:
                secretKeyRef:
                  name: postgres-credentials
                  key: username
            - name: POSTGRES_DB
              value: "myapp_production"
            - name: PGPASSWORD
              valueFrom:
                secretKeyRef:
                  name: postgres-credentials
                  key: password
            volumeMounts:
            - name: backup-storage
              mountPath: /backups
            resources:
              requests:
                memory: "256Mi"
                cpu: "250m"
              limits:
                memory: "512Mi"
                cpu: "500m"
          restartPolicy: OnFailure
          volumes:
          - name: backup-storage
            persistentVolumeClaim:
              claimName: backup-pvc
```

### What's Happening Here?

| Part | Explanation |
|------|-------------|
| `schedule: "0 2 * * *"` | Runs every night at 2 AM |
| `concurrencyPolicy: Forbid` | If previous backup is still running, skip this one |
| `startingDeadlineSeconds: 600` | If the schedule is missed, still run if within 10 minutes |
| `activeDeadlineSeconds: 3600` | Kill the Job if it runs longer than 1 hour |
| `pg_dump ... \| gzip` | Creates a compressed PostgreSQL backup |
| `secretKeyRef` | Pulls DB credentials from a Kubernetes Secret (secure!) |
| `PersistentVolumeClaim` | Stores the backup file on persistent storage |

### Manually Trigger a CronJob

Sometimes you want to run a CronJob immediately without waiting for the schedule:

```bash
# Create a Job from an existing CronJob (runs it right now)
kubectl create job --from=cronjob/postgres-backup manual-backup-run
```

This creates a one-time Job using the CronJob's template. Super useful for testing!

| Concept | Job | CronJob |
|---------|-----|---------|
| **Purpose** | Run a task once | Run a task on a schedule |

## вқ“ Interview Questions

1. **What is the difference between a Job and a CronJob?**
   вҶ’ A Job runs a task once and exits. A CronJob creates Jobs on a schedule (like cron in Linux).

2. **Why can't you use `restartPolicy: Always` in a Job?**
   вҶ’ Because a Job is meant to complete and exit. `Always` would keep restarting the container forever, defeating the purpose.

3. **What does `backoffLimit` do?**
   вҶ’ It controls how many times Kubernetes retries a failed Job before marking it as permanently failed.

4. **What is `concurrencyPolicy` in a CronJob?**
   вҶ’ It controls what happens when a new scheduled Job is due but the previous one is still running. Options: `Allow` (run both), `Forbid` (skip new), `Replace` (kill old, start new).

5. **How do you manually trigger a CronJob?**
   вҶ’ `kubectl create job my-job --from=cronjob/my-cronjob`

6. **What's the difference between `restartPolicy: Never` and `OnFailure` in a Job?**
   вҶ’ `Never` creates a new Pod for each retry. `OnFailure` restarts the same Pod in-place.

7. **How do you prevent old completed Jobs from piling up?**
   вҶ’ Use `ttlSecondsAfterFinished` on Jobs, or `successfulJobsHistoryLimit` / `failedJobsHistoryLimit` on CronJobs.
