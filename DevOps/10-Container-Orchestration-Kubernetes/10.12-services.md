## 🤔 The Problem — Why Do We Need Services?

Let's say you have a **Deployment** with 3 replicas of a web application:

```bash
kubectl get pods -o wide
```

```
NAME                        IP            NODE
webapp-abc12                10.244.1.15   worker-1
webapp-def34                10.244.2.22   worker-2
webapp-ghi56                10.244.3.8    worker-3
```

Each Pod has its own IP address. But here's the problem:

### Problem 1: Pod IPs Are NOT Permanent

Pods are **ephemeral** — they can be killed and recreated at any time (crashes, scaling, rolling updates, node failures). When a Pod is recreated, it gets a **brand new IP address**.

```
Before restart:  webapp-abc12  → IP: 10.244.1.15
After restart:   webapp-xyz99  → IP: 10.244.1.47  ← Different IP!
```

So if another Pod (like your API server) was talking to `10.244.1.15`, it would suddenly lose connection. 💥

### Problem 2: Which Pod Should Receive Traffic?

You have 3 replicas. When someone sends a request, **which of the 3 Pods should handle it?** You need something to distribute traffic evenly among all healthy Pods — a load balancer.

### Problem 3: How Does the Outside World Reach Your Pods?

Pod IPs are **internal** to the cluster. People on the internet can't reach `10.244.1.15` — it's a private cluster IP. You need a way to expose your application to the outside world.

**Services solve ALL three of these problems:**

| Problem | Service Solution |
|---------|-----------------|
| Pod IPs change | Service provides a **stable, permanent IP** (and DNS name) |
| Load balancing | Service distributes traffic across all matching Pods |
| External access | Certain Service types expose Pods to the outside world |

## 📦 What Exactly is a Service?

A **Service** is a Kubernetes resource that:

1. **Groups a set of Pods** together (using label selectors)
2. Gives them a **single stable IP address** (called the ClusterIP)
3. Gives them a **DNS name** (e.g., `my-service.my-namespace.svc.cluster.local`)
4. **Load-balances** traffic across all the Pods in the group
5. **Tracks Pod changes** automatically — if Pods are added/removed/restarted, the Service updates itself

### Real-World Analogy

Think of a Service like a **customer service phone number** for a company:

- The company has **50 support agents** (Pods) sitting at their desks.
- Agents come and go (some quit, new ones are hired, some are on break).
- But the **phone number never changes** — customers always call the same number.
- A **phone system** (Service) routes the call to an available agent.
- The customer doesn't know (or care) which specific agent picks up.

That phone number = **Service**. The agents = **Pods**.

## 🏗️ How Services Work — Under the Hood

Here's the flow when a request reaches a Service:

```
┌──────────────────────────────────────────────────────────┐
│                    Kubernetes Cluster                     │
│                                                          │
│   Client Pod                                             │
│   (e.g., API server)                                     │
│       │                                                  │
│       │ request to "webapp-service:80"                    │
│       ▼                                                  │
│   ┌──────────────────┐                                   │
│   │    Service        │  ← Stable IP: 10.96.45.123       │
│   │  "webapp-service" │  ← DNS: webapp-service.default   │
│   └────────┬─────────┘                                   │
│            │                                             │
│            │  kube-proxy routes traffic (via iptables     │
│            │  or IPVS rules on each node)                │
│            │                                             │
│     ┌──────┼──────┐                                      │
│     ▼      ▼      ▼                                      │
│  ┌──────┐ ┌──────┐ ┌──────┐                              │
│  │Pod 1 │ │Pod 2 │ │Pod 3 │  ← Actual application Pods  │
│  │.1.15 │ │.2.22 │ │.3.8  │  ← IPs can change           │
│  └──────┘ └──────┘ └──────┘                              │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### The Key Components

| Component | Role |
|-----------|------|
| **Service** | Provides the stable IP and DNS name |
| **Endpoints** | A list of Pod IPs that currently match the Service's selector. Automatically updated! |
| **kube-proxy** | Runs on every node. Programs network rules (iptables/IPVS) to route Service traffic to the right Pods |
| **CoreDNS** | Kubernetes' internal DNS server. Resolves Service names to their ClusterIP |

### How Endpoints Work

When you create a Service with a label selector, Kubernetes automatically creates an **Endpoints** object that tracks all matching Pod IPs:

```bash
# See the Endpoints for a Service
kubectl get endpoints webapp-service
```

```
NAME              ENDPOINTS                                    AGE
webapp-service    10.244.1.15:80,10.244.2.22:80,10.244.3.8:80  5m
```

When a Pod is deleted or a new one is created, the Endpoints list is **automatically updated**. You never have to manage this manually.

## 🔵 Service Type 1: ClusterIP (Default)

### What Is It?

ClusterIP is the **default** Service type. It gives your Pods a **stable internal IP address** that is only reachable from **inside the cluster**.

Think of it as: "Pods within the cluster can talk to each other using this address."

### When to Use It?

- Your frontend Pod needs to talk to your backend Pod
- Your API needs to talk to a database
- Any **internal communication** between microservices

### Example

```yaml
# service-clusterip.yaml
apiVersion: v1
kind: Service
metadata:
  name: backend-service
spec:
  type: ClusterIP              # Default — can be omitted
  selector:
    app: backend               # Route traffic to Pods with label app=backend
  ports:
  - protocol: TCP
    port: 80                   # Port the Service listens on
    targetPort: 8080           # Port the container is listening on
```

### Breaking It Down

| Field | What It Does |
|-------|-------------|
| `type: ClusterIP` | Internal-only Service (default, so you can omit this line) |
| `selector: app: backend` | Find all Pods with the label `app: backend` and route traffic to them |
| `port: 80` | The port that **other Pods use** to reach this Service |
| `targetPort: 8080` | The port that the **container is actually listening on** |

### `port` vs `targetPort` — The Most Common Confusion

This trips up almost everyone. Let me make it crystal clear:

```
Client Pod → Service:80 → Container:8080

                port: 80           targetPort: 8080
Client     ──────────────►  Service  ──────────────►  Container
           "I call port 80"          "I forward to 8080"
```

| Field | Who Uses It | Example |
|-------|-------------|---------|
| `port` | The caller (other Pods, users) | "Call me on port 80" |
| `targetPort` | The destination container | "My app listens on 8080" |

**Why different?** Because you might want the Service to be on port 80 (standard HTTP), but your app runs on port 8080 (common for Java/Node apps). The Service bridges the gap.

If `port` and `targetPort` are the same, you can omit `targetPort`:

```yaml
ports:
- port: 80             # Both port and targetPort will be 80
```

### Apply and Use

```bash
# Create the Service
kubectl apply -f service-clusterip.yaml

# Check it
kubectl get services
# or shorthand
kubectl get svc
```

**Output:**

```
NAME              TYPE        CLUSTER-IP     EXTERNAL-IP   PORT(S)   AGE
backend-service   ClusterIP   10.96.45.123   <none>        80/TCP    10s
kubernetes        ClusterIP   10.96.0.1      <none>        443/TCP   5d
```

| Column | Meaning |
|--------|---------|
| `CLUSTER-IP` | The stable internal IP assigned to this Service |
| `EXTERNAL-IP` | `<none>` — ClusterIP is internal only |
| `PORT(S)` | Port the Service listens on |

### Accessing the ClusterIP Service

From **inside** the cluster (another Pod), you can reach it via:

```bash
# By ClusterIP
curl http://10.96.45.123:80

# By Service name (same namespace)
curl http://backend-service:80

# By fully qualified DNS name (from any namespace)
curl http://backend-service.default.svc.cluster.local:80
```

### DNS Format

Kubernetes DNS follows this pattern:

```
<service-name>.<namespace>.svc.cluster.local
```

| Part | Example | Meaning |
|------|---------|---------|
| `service-name` | `backend-service` | Name of your Service |
| `namespace` | `default` | Namespace the Service is in |
| `svc` | `svc` | Always "svc" (means Service) |
| `cluster.local` | `cluster.local` | Default cluster domain |

**Shortcuts:**
- Same namespace: just use `backend-service`
- Different namespace: use `backend-service.other-namespace`
- Full FQDN: `backend-service.other-namespace.svc.cluster.local`

## 🟢 Service Type 2: NodePort

### What Is It?

NodePort exposes the Service on a **static port on every node's IP address**. This makes the Service accessible from **outside the cluster** by hitting any node's IP at that port.

Think of it as: "Open a port on every machine in the cluster and forward traffic from there to my Pods."

### When to Use It?

- Quick development or testing
- When you don't have a cloud load balancer
- Simple external access scenarios
- Learning environments (Minikube, Kind)

### How It Works

```
External User
      │
      │  http://<any-node-ip>:30080
      ▼
┌──────────┐  ┌──────────┐  ┌──────────┐
│  Node 1  │  │  Node 2  │  │  Node 3  │
│ :30080   │  │ :30080   │  │ :30080   │
│    │     │  │    │     │  │    │     │
│    ▼     │  │    ▼     │  │    ▼     │
│ ┌──────┐ │  │ ┌──────┐ │  │ ┌──────┐ │
│ │ kube │ │  │ │ kube │ │  │ │ kube │ │
│ │proxy │ │  │ │proxy │ │  │ │proxy │ │
│ └──┬───┘ │  │ └──┬───┘ │  │ └──┬───┘ │
└────┼─────┘  └────┼─────┘  └────┼─────┘
     │              │              │
     └──────────┬───┘──────────────┘
                │
         Service (ClusterIP)
                │
     ┌──────────┼──────────┐
     ▼          ▼          ▼
  ┌──────┐  ┌──────┐  ┌──────┐
  │Pod 1 │  │Pod 2 │  │Pod 3 │
  └──────┘  └──────┘  └──────┘
```

**Important:** The traffic can hit **ANY node** (even a node that doesn't have the Pod running on it). kube-proxy on that node will forward the traffic to the right Pod on the right node.

### Example

```yaml
# service-nodeport.yaml
apiVersion: v1
kind: Service
metadata:
  name: webapp-nodeport
spec:
  type: NodePort
  selector:
    app: webapp
  ports:
  - protocol: TCP
    port: 80               # Internal Service port (inside cluster)
    targetPort: 8080       # Container port
    nodePort: 30080        # Port opened on ALL nodes (range: 30000-32767)
```

### Understanding the THREE Ports

This is the trickiest part about NodePort. There are **three** ports:

```
External User ──► Node:30080 ──► Service:80 ──► Container:8080
                  (nodePort)     (port)         (targetPort)
```

| Port | What It Is | Who Uses It | Range |
|------|-----------|-------------|-------|
| `nodePort` | Port on the **node's** actual IP | External users | 30000–32767 |
| `port` | Port on the **Service's** ClusterIP | Internal Pods | Any |
| `targetPort` | Port the **container** listens on | The application | Any |

If you don't specify `nodePort`, Kubernetes auto-assigns one from the 30000–32767 range.

### Apply and Use

```bash
kubectl apply -f service-nodeport.yaml
kubectl get svc webapp-nodeport
```

**Output:**

```
NAME              TYPE       CLUSTER-IP     EXTERNAL-IP   PORT(S)        AGE
webapp-nodeport   NodePort   10.96.88.77    <none>        80:30080/TCP   10s
```

`80:30080` means: Service port 80 is mapped to NodePort 30080.

### Accessing It

```bash
# From outside the cluster — use any node's IP
curl http://<node-1-ip>:30080
curl http://<node-2-ip>:30080    # Same result — traffic goes to the same Pods
curl http://<node-3-ip>:30080

# From inside the cluster — still works via ClusterIP
curl http://webapp-nodeport:80
curl http://10.96.88.77:80
```

### ⚠️ NodePort Limitations

| Limitation | Details |
|-----------|---------|
| **Port range** | Only 30000–32767 (non-standard ports, ugly URLs) |
| **Security** | Every node exposes the port — larger attack surface |
| **No real load balancing** | No single entry point — you need to know which node IP to hit |
| **Not production-ready** | For production, use LoadBalancer or Ingress instead |

## 🟣 Service Type 3: LoadBalancer

### What Is It?

LoadBalancer provisions an **actual cloud load balancer** (from AWS, GCP, Azure, etc.) that gets a **real external IP address** and forwards traffic to your Service.

This is what you typically use in **production on cloud platforms**.

### When to Use It?

- Production workloads on cloud (EKS, GKE, AKS)
- When you need a **real external IP** or DNS endpoint
- When you need cloud-native load balancing features

### How It Works

```
Internet Users
      │
      │  http://a1b2c3.elb.amazonaws.com  (or external IP)
      ▼
┌─────────────────┐
│  Cloud Load     │  ← Provisioned automatically by cloud provider
│  Balancer       │  ← Has a real external IP / DNS
│  (AWS ELB,      │
│   GCP LB, etc.) │
└────────┬────────┘
         │
         ▼
    ┌─────────┐  ┌─────────┐  ┌─────────┐
    │ Node 1  │  │ Node 2  │  │ Node 3  │
    │ :30080  │  │ :30080  │  │ :30080  │
    └────┬────┘  └────┬────┘  └────┬────┘
         │            │            │
         └─────┬──────┘────────────┘
               │
          Service (ClusterIP)
               │
        ┌──────┼──────┐
        ▼      ▼      ▼
     Pod 1   Pod 2   Pod 3
```

A LoadBalancer Service is essentially **NodePort + ClusterIP + an external load balancer** — all in one. It includes all the features of the other types.

### Example

```yaml
# service-loadbalancer.yaml
apiVersion: v1
kind: Service
metadata:
  name: webapp-lb
spec:
  type: LoadBalancer
  selector:
    app: webapp
  ports:
  - protocol: TCP
    port: 80               # Service port
    targetPort: 8080       # Container port
```

### Apply and Use

```bash
kubectl apply -f service-loadbalancer.yaml
kubectl get svc webapp-lb
```

**Output (cloud environment):**

```
NAME        TYPE           CLUSTER-IP     EXTERNAL-IP      PORT(S)        AGE
webapp-lb   LoadBalancer   10.96.55.44    34.120.15.88     80:31234/TCP   60s
```

| Column | Value | Meaning |
|--------|-------|---------|
| `EXTERNAL-IP` | `34.120.15.88` | Real external IP assigned by cloud provider |
| `PORT(S)` | `80:31234` | Service port 80 → auto-assigned NodePort 31234 |

> **Note:** On Minikube or local clusters, `EXTERNAL-IP` will show `<pending>` forever because there's no cloud provider to create a real load balancer. Use `minikube tunnel` or MetalLB for local testing.

### Accessing It

```bash
# From the internet — use the external IP
curl http://34.120.15.88

# From inside the cluster — ClusterIP still works
curl http://webapp-lb:80
```

### Cloud Provider Annotations

Each cloud provider supports additional configuration via **annotations**:

#### AWS (EKS)

```yaml
metadata:
  annotations:
    service.beta.kubernetes.io/aws-load-balancer-type: "nlb"         # Network LB (Layer 4)
    service.beta.kubernetes.io/aws-load-balancer-scheme: "internet-facing"
    service.beta.kubernetes.io/aws-load-balancer-ssl-cert: "arn:aws:acm:..."
```

#### GCP (GKE)

```yaml
metadata:
  annotations:
    cloud.google.com/load-balancer-type: "Internal"   # Internal LB only
```

#### Azure (AKS)

```yaml
metadata:
  annotations:
    service.beta.kubernetes.io/azure-load-balancer-internal: "true"
```

### ⚠️ LoadBalancer Limitations

| Limitation | Details |
|-----------|---------|
| **Cost** | Each LoadBalancer Service creates a **separate cloud LB** — this gets expensive fast! |
| **One per service** | 10 services = 10 cloud load balancers = 10x the cost 💸 |
| **Layer 4 only** | Basic load balancers are L4 (TCP) — no path-based routing |
| **Better alternative** | Use **Ingress** (covered in 10.13) for HTTP/HTTPS with a single load balancer |

## 🟠 Service Type 4: ExternalName

### What Is It?

ExternalName is a special Service type that doesn't route to Pods at all. Instead, it creates a **DNS alias** (CNAME record) that points to an **external DNS name**.

Think of it as: "When someone inside the cluster asks for `my-service`, redirect them to `some.external.database.com`."

### When to Use It?

- You want to access an **external service** (like an external database, API, or SaaS) using a Kubernetes-style DNS name.
- You want to **decouple** your app from the exact external endpoint (so you can change it later without modifying your app).
- During **migrations** — point to an external DB now, switch to an in-cluster DB later by just changing the Service definition.

### Example

```yaml
# service-externalname.yaml
apiVersion: v1
kind: Service
metadata:
  name: external-db
  namespace: production
spec:
  type: ExternalName
  externalName: my-database.abc123.us-east-1.rds.amazonaws.com
```

### How It Works

```
┌──────────── Kubernetes Cluster ────────────┐
│                                            │
│  Your App Pod                              │
│       │                                    │
│       │ connects to "external-db:5432"     │
│       ▼                                    │
│   CoreDNS resolves "external-db" to        │
│   "my-database.abc123.rds.amazonaws.com"   │
│       │                                    │
└───────┼────────────────────────────────────┘
        │  (DNS CNAME redirect)
        ▼
┌────────────────────────────────┐
│  AWS RDS PostgreSQL            │
│  my-database.abc123.rds.aws... │
│  (External to the cluster)     │
└────────────────────────────────┘
```

### Inside Your Application

Your app code would connect to:

```python
# In your app, just use the Service name
db_host = "external-db"     # Kubernetes Service name
db_port = 5432
connection = connect(host=db_host, port=db_port)
```

CoreDNS translates `external-db` → `my-database.abc123.us-east-1.rds.amazonaws.com` automatically via a CNAME record.

### Key Points

| Feature | Details |
|---------|---------|
| **No selector** | ExternalName Services have NO `selector` — they don't target Pods |
| **No ClusterIP** | No IP is assigned — it's pure DNS redirection |
| **No proxying** | Traffic is NOT proxied through kube-proxy — it's a direct DNS redirect |
| **CNAME only** | Works by returning a CNAME record, so it only works with hostnames (not IP addresses) |

### ⚠️ ExternalName Limitations

- Can only point to a **hostname** (not an IP address). If your external service only has an IP, use a Service with `Endpoints` instead (covered below).
- CNAME resolution adds a tiny bit of latency (one extra DNS lookup).
- Some apps may not handle CNAME redirects properly.

### Accessing External Service by IP (Without ExternalName)

If you need to access an external service by **IP address**, create a Service without a selector and manually define Endpoints:

```yaml
# Service without selector
apiVersion: v1
kind: Service
metadata:
  name: external-legacy-api
spec:
  ports:
  - port: 443
    targetPort: 443
---
# Manually define the Endpoints
apiVersion: v1
kind: Endpoints
metadata:
  name: external-legacy-api    # Must match the Service name!
subsets:
- addresses:
  - ip: 203.0.113.50           # External IP address
  - ip: 203.0.113.51           # Another external IP (for redundancy)
  ports:
  - port: 443
```

Now Pods in the cluster can reach `external-legacy-api:443`, and traffic goes to `203.0.113.50` or `203.0.113.51`.

## 📊 Service Types Comparison — The Complete Picture

| Feature | ClusterIP | NodePort | LoadBalancer | ExternalName |
|---------|-----------|----------|--------------|--------------|
| **Access from** | Inside cluster only | Inside + Outside | Inside + Outside (via LB) | DNS redirect |
| **External IP** | ❌ None | ❌ None (uses Node IPs) | ✅ Real external IP | ❌ None (DNS only) |
| **Port** | Service port | Node port (30000-32767) | Standard ports (80, 443) | N/A |
| **Use case** | Internal microservices | Dev/testing | Production cloud | External services |
| **Cost** | Free | Free | $$ (cloud LB cost) | Free |
| **Load balancing** | Internal (kube-proxy) | Internal (kube-proxy) | Cloud LB + kube-proxy | None |
| **Layer** | L4 | L4 | L4 (L7 with Ingress) | DNS |
| **Includes ClusterIP?** | ✅ Yes (it IS ClusterIP) | ✅ Yes | ✅ Yes | ❌ No |
| **Includes NodePort?** | ❌ No | ✅ Yes (it IS NodePort) | ✅ Yes | ❌ No |

### Visual Stack

```
LoadBalancer  ═══╗
                 ║ includes
NodePort     ════╬══╗
                 ║  ║ includes
ClusterIP    ════╬══╝
                 ║
ExternalName ════╝ (separate — DNS only, no IP)
```

**LoadBalancer** is the most complete — it includes ClusterIP + NodePort + external LB.
**NodePort** includes ClusterIP.
**ClusterIP** is the base.
**ExternalName** is completely different — it's just DNS.

## 🏗️ Multi-Port Services

Sometimes your application listens on **multiple ports** (e.g., HTTP on 80 and metrics on 9090):

```yaml
apiVersion: v1
kind: Service
metadata:
  name: multi-port-service
spec:
  selector:
    app: webapp
  ports:
  - name: http                # Name is REQUIRED when you have multiple ports
    protocol: TCP
    port: 80
    targetPort: 8080
  - name: metrics             # Each port must have a unique name
    protocol: TCP
    port: 9090
    targetPort: 9090
```

> **Rule:** When a Service has multiple ports, **every port must have a `name`**.

## 🔍 Headless Services — When You Don't Want a ClusterIP

Sometimes you don't want Kubernetes to provide load balancing or a single ClusterIP. Instead, you want **direct access to individual Pod IPs**. This is called a **Headless Service**.

### When Would You Want This?

- **StatefulSets** — when each Pod needs to be individually addressable (database replicas, Kafka brokers)
- **Client-side load balancing** — your app wants to know all Pod IPs and balance traffic itself
- **Service discovery** — you want DNS to return all Pod IPs instead of a single ClusterIP

### How to Create a Headless Service

Just set `clusterIP: None`:

```yaml
apiVersion: v1
kind: Service
metadata:
  name: headless-service
spec:
  clusterIP: None              # This makes it headless!
  selector:
    app: database
  ports:
  - port: 5432
    targetPort: 5432
```

### What Changes with Headless?

| Feature | Regular Service | Headless Service |
|---------|----------------|-----------------|
| ClusterIP | Assigned (e.g., `10.96.x.x`) | `None` |
| DNS returns | Single ClusterIP | **All Pod IPs** (A records) |
| Load balancing | kube-proxy handles it | No built-in LB — client decides |
| Use with StatefulSet | Possible but not ideal | Recommended |

### DNS Behavior

```bash
# Regular Service DNS lookup:
nslookup backend-service
→ 10.96.45.123  (single ClusterIP)

# Headless Service DNS lookup:
nslookup headless-service
→ 10.244.1.15   (Pod 1)
→ 10.244.2.22   (Pod 2)
→ 10.244.3.8    (Pod 3)
```

With StatefulSets, each Pod even gets its own DNS entry:

```
database-0.headless-service.default.svc.cluster.local → 10.244.1.15
database-1.headless-service.default.svc.cluster.local → 10.244.2.22
database-2.headless-service.default.svc.cluster.local → 10.244.3.8
```

## 🏗️ Complete Real-World Example — Putting It All Together

Let's create a full setup with a Deployment and multiple Service types:

### Step 1: The Deployment

```yaml
# webapp-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
```

## 🔧 `sessionAffinity` — Sticky Sessions

By default, Services distribute traffic randomly. If you want the same client to always go to the **same Pod** (sticky sessions):

```yaml
apiVersion: v1
kind: Service
metadata:
  name: sticky-service
spec:
  selector:
    app: webapp
  sessionAffinity: ClientIP          # Stick to the same Pod per client IP
  sessionAffinityConfig:
    clientIP:
      timeoutSeconds: 1800           # Sticky for 30 minutes
  ports:
  - port: 80
    targetPort: 80
```

| `sessionAffinity` | Behavior |
|-------------------|----------|
| `None` (default) | Random Pod selection on each request |
| `ClientIP` | Same client IP always goes to the same Pod |

## 🔑 Quick Summary

| Service Type | Access | Use Case | Key Feature |
|-------------|--------|----------|-------------|
| **ClusterIP** | Internal only | Microservice communication | Stable internal IP + DNS |
| **NodePort** | Internal + External (via node IP) | Dev/testing | Opens port 30000-32767 on all nodes |
| **LoadBalancer** | Internal + External (via cloud LB) | Production on cloud | Real external IP from cloud provider |
| **ExternalName** | DNS redirect to external | Accessing external services | CNAME record, no proxying |
| **Headless** | Direct Pod IPs | StatefulSets, service discovery | `clusterIP: None`, DNS returns all Pod IPs |

## ❓ Interview Questions

1. **What is a Kubernetes Service and why do we need it?**
   → A Service provides a stable IP and DNS name to a group of Pods. We need it because Pod IPs are ephemeral and change on restarts.

2. **What are the four types of Services in Kubernetes?**
   → ClusterIP (internal), NodePort (node-level port), LoadBalancer (cloud LB with external IP), and ExternalName (DNS alias to external service).

3. **What is the difference between `port`, `targetPort`, and `nodePort`?**
   → `port` is what clients use to reach the Service. `targetPort` is the port the container listens on. `nodePort` is the port opened on every node (only for NodePort/LoadBalancer).

4. **What is a Headless Service and when would you use it?**
   → A Service with `clusterIP: None`. DNS returns individual Pod IPs instead of a single ClusterIP. Used with StatefulSets for individual Pod addressing.

5. **How does kube-proxy route traffic to Pods?**
   → kube-proxy runs on each node and programs iptables or IPVS rules that intercept traffic to Service ClusterIPs and forward it to one of the backend Pod IPs.

6. **Why shouldn't you use LoadBalancer type for every Service in production?**
   → Each LoadBalancer Service provisions a separate cloud load balancer, which is expensive. Use Ingress instead to share one load balancer across multiple Services.

7. **How does a Service know which Pods to route traffic to?**
   → Through label selectors. The Service's `selector` field matches against Pod labels. Matching Pods are added to the Service's Endpoints.

8. **What is the DNS format for accessing a Service in Kubernetes?**
   → `<service-name>.<namespace>.svc.cluster.local`
