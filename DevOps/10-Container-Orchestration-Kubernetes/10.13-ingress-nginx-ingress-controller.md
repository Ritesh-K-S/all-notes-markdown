# 10.13 — Ingress — Nginx Ingress Controller, Path-Based & Host-Based Routing

> **One Line Answer:** An **Ingress** is a Kubernetes resource that manages **external HTTP/HTTPS access** to your services — it acts like a smart front door that routes traffic to different services based on the **URL path** or **hostname**, so you don't need to expose every service individually.

---

## 🤔 Why Do We Need Ingress?

Let's start with the problem.

You've got multiple services running inside your Kubernetes cluster:

```
Your Cluster:
┌──────────────────────────────────────────────────┐
│  🌐 frontend-service    (React app)              │
│  ⚙️  api-service          (REST API)              │
│  📊 dashboard-service   (Admin Dashboard)         │
│  🔐 auth-service         (Authentication)         │
└──────────────────────────────────────────────────┘
```

Now, how do users from the **outside world** reach these services?

### Option 1: Use NodePort Services (❌ Not Great)

```yaml
# Each service gets its own port on the node
frontend-service  → NodePort 30001
api-service       → NodePort 30002
dashboard-service → NodePort 30003
auth-service      → NodePort 30004
```

Problems:
- Users have to remember port numbers: `http://mysite.com:30001`, `http://mysite.com:30002`
- Ports are limited (30000-32767 range)
- No SSL/TLS termination
- No path-based routing
- Looks super unprofessional

### Option 2: Use LoadBalancer Services (💸 Expensive)

```yaml
# Each service gets its own cloud load balancer
frontend-service  → LoadBalancer ($$$)
api-service       → LoadBalancer ($$$)
dashboard-service → LoadBalancer ($$$)
auth-service      → LoadBalancer ($$$)
```

Problems:
- Each LoadBalancer costs money (AWS ELB = ~$18/month each)
- 4 services = 4 load balancers = 💸💸💸💸
- Each gets a different IP/DNS — hard to manage
- Still no path-based routing

### Option 3: Use Ingress (✅ The Right Way!)

```
                    ONE Entry Point
                         │
                    ┌─────────────┐
                    │   Ingress   │ ← ONE Load Balancer
                    │  Controller │    ONE IP Address
                    └──────┬──────┘    ONE SSL Certificate
                           │
            ┌──────────────┼──────────────┐
            │              │              │
     /app/*          /api/*         /dashboard/*
            │              │              │
    ┌───────┴──┐   ┌───────┴──┐   ┌───────┴──────┐
    │ frontend │   │   api    │   │  dashboard   │
    │ service  │   │ service  │   │   service    │
    └──────────┘   └──────────┘   └──────────────┘
```

**One Ingress** = One Load Balancer = One IP = Smart Routing = 💰 Saved!

## ✅ What Exactly is an Ingress?

An Ingress is a **Kubernetes API object** (just like a Pod or Service). It defines rules for routing HTTP/HTTPS traffic from **outside** the cluster to **services inside** the cluster.

Think of it like this:

| Real World Analogy | Kubernetes Equivalent |
|---|---|
| A hotel reception desk | Ingress |
| The receptionist who directs guests | Ingress Controller |
| Different rooms (101, 102, 103) | Different Services |
| Guest says "I'm here for room 101" | URL path or hostname |

**Key Point:** An Ingress resource by itself does **nothing**. It's just a set of rules written in YAML. You need an **Ingress Controller** (the actual software) to read those rules and make routing happen.

---

## 🔧 Ingress vs Ingress Controller — What's the Difference?

This confuses a lot of people, so let's be very clear:

| Concept | What It Is | Analogy |
|---|---|---|
| **Ingress** (Resource) | A YAML file with routing rules | A recipe |
| **Ingress Controller** | The actual software that implements those rules | The chef who cooks from the recipe |

```
Ingress Resource (YAML rules)
        │
        │  "reads"
        ▼
Ingress Controller (running software like Nginx, Traefik, etc.)
        │
        │  "routes traffic"
        ▼
Your Backend Services
```

Without an Ingress Controller running in your cluster, creating an Ingress resource is like writing a recipe with no chef in the kitchen — nothing happens!

## 🚀 Setting Up the Nginx Ingress Controller

The **Nginx Ingress Controller** is the most popular Ingress Controller. It uses Nginx (the web server) under the hood to handle routing.

### Step 1: Install the Nginx Ingress Controller

There are two versions — make sure you use the right one:

| Version | Maintained By | Repo |
|---|---|---|
| **kubernetes/ingress-nginx** | Kubernetes community | The standard one ✅ |
| **nginxinc/kubernetes-ingress** | Nginx Inc (F5) | Commercial version |

We'll use the **community version** (the most common one).

#### Using Helm (Recommended):

```bash
# Add the ingress-nginx Helm repo
helm repo add ingress-nginx https://kubernetes.github.io/ingress-nginx
helm repo update

# Install the Nginx Ingress Controller
helm install ingress-nginx ingress-nginx/ingress-nginx \
  --namespace ingress-nginx \
  --create-namespace
```

#### Using kubectl (Raw Manifests):

```bash
kubectl apply -f https://raw.githubusercontent.com/kubernetes/ingress-nginx/controller-v1.10.0/deploy/static/provider/cloud/deploy.yaml
```

### Step 2: Verify Installation

```bash
# Check if the controller pod is running
kubectl get pods -n ingress-nginx

# Expected output:
# NAME                                        READY   STATUS    RESTARTS   AGE
# ingress-nginx-controller-xxxxxxxxx-xxxxx    1/1     Running   0          2m

# Check the service (this will have the external IP)
kubectl get svc -n ingress-nginx

# Expected output:
# NAME                       TYPE           CLUSTER-IP     EXTERNAL-IP      PORT(S)
# ingress-nginx-controller   LoadBalancer   10.96.xxx.xx   203.0.113.100    80:30080/TCP,443:30443/TCP
```

The `EXTERNAL-IP` is what you'll point your DNS records to. If you're on Minikube, run:

```bash
minikube addons enable ingress
```

## 📝 Your First Ingress — A Simple Example

Let's say you have a simple web app running as a service:

### Step 1: Create a Deployment and Service First

```yaml
# app-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: hello-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: hello
  template:
    metadata:
      labels:
        app: hello
    spec:
      containers:
      - name: hello
        image: hashicorp/http-echo
        args:
        - "-text=Hello from the app!"
        ports:
        - containerPort: 5678
---
apiVersion: v1
kind: Service
metadata:
  name: hello-service
spec:
  selector:
    app: hello
  ports:
  - port: 80
    targetPort: 5678
  type: ClusterIP   # <-- Notice: ClusterIP, NOT LoadBalancer!
```

```bash
kubectl apply -f app-deployment.yaml
```

### Step 2: Create the Ingress Resource

```yaml
# hello-ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: hello-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx          # <-- Tells K8s which controller to use
  rules:
  - host: hello.example.com       # <-- The hostname
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: hello-service    # <-- Route to this service
            port:
              number: 80           # <-- On this port
```

```bash
kubectl apply -f hello-ingress.yaml
```

### Step 3: Test It!

```bash
# Check the ingress
kubectl get ingress

# NAME            CLASS   HOSTS               ADDRESS         PORTS   AGE
# hello-ingress   nginx   hello.example.com   203.0.113.100   80      1m

# Test with curl (use the external IP)
curl -H "Host: hello.example.com" http://203.0.113.100
# Output: Hello from the app!
```

## 🔀 Path-Based Routing — Route by URL Path

This is one of the most common use cases. Different URL paths go to different services.

**The Idea:**
```
https://mysite.com/           → frontend-service
https://mysite.com/api/       → api-service
https://mysite.com/dashboard/ → dashboard-service
```

### Full Example:

First, let's create the backend services:

```yaml
# services.yaml
# ---- Frontend ----
apiVersion: apps/v1
kind: Deployment
metadata:
  name: frontend
spec:
  replicas: 2
  selector:
    matchLabels:
      app: frontend
  template:
    metadata:
      labels:
        app: frontend
    spec:
      containers:
      - name: frontend
        image: hashicorp/http-echo
        args: ["-text=🌐 Hello from Frontend!"]
        ports:
        - containerPort: 5678
---
apiVersion: v1
kind: Service
metadata:
  name: frontend-service
spec:
  selector:
    app: frontend
  ports:
  - port: 80
    targetPort: 5678

---
# ---- API ----
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api
spec:
  replicas: 3
  selector:
    matchLabels:
      app: api
  template:
    metadata:
      labels:
        app: api
    spec:
      containers:
      - name: api
        image: hashicorp/http-echo
        args: ["-text=⚙️ Hello from API!"]
        ports:
        - containerPort: 5678
---
apiVersion: v1
kind: Service
metadata:
  name: api-service
spec:
  selector:
    app: api
  ports:
  - port: 80
    targetPort: 5678

---
# ---- Dashboard ----
apiVersion: apps/v1
kind: Deployment
metadata:
  name: dashboard
spec:
  replicas: 2
  selector:
    matchLabels:
      app: dashboard
  template:
    metadata:
      labels:
        app: dashboard
    spec:
      containers:
      - name: dashboard
        image: hashicorp/http-echo
        args: ["-text=📊 Hello from Dashboard!"]
        ports:
        - containerPort: 5678
---
apiVersion: v1
kind: Service
metadata:
  name: dashboard-service
spec:
  selector:
    app: dashboard
  ports:
  - port: 80
    targetPort: 5678
```

Now, the Ingress with **path-based routing**:

```yaml
# path-based-ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: path-based-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  rules:
  - host: mysite.com
    http:
      paths:
      # Route 1: / → frontend
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend-service
            port:
              number: 80

      # Route 2: /api → api-service
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: api-service
            port:
              number: 80

      # Route 3: /dashboard → dashboard-service
      - path: /dashboard
        pathType: Prefix
        backend:
          service:
            name: dashboard-service
            port:
              number: 80
```

```bash
kubectl apply -f path-based-ingress.yaml
```

### How It Works:

```
User visits mysite.com/api/users
        │
        ▼
  Ingress Controller checks the path
        │
        ├── Path starts with /api?       → YES → Send to api-service
        ├── Path starts with /dashboard? → NO
        └── Path starts with /?          → (fallback)
```

### Understanding `pathType`:

| pathType | Behavior | Example |
|---|---|---|
| `Prefix` | Matches the URL prefix | `/api` matches `/api`, `/api/users`, `/api/v2/data` |
| `Exact` | Matches the exact path only | `/api` matches ONLY `/api`, NOT `/api/users` |
| `ImplementationSpecific` | Depends on the Ingress Controller | Varies |

**Pro Tip:** `Prefix` is what you'll use 99% of the time.

## 🏠 Host-Based Routing — Route by Hostname (Virtual Hosting)

Instead of different paths, you use different **hostnames** (subdomains or domains) to route traffic.

**The Idea:**
```
https://app.mysite.com       → frontend-service
https://api.mysite.com       → api-service
https://dashboard.mysite.com → dashboard-service
```

### Full Example:

```yaml
# host-based-ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: host-based-ingress
spec:
  ingressClassName: nginx
  rules:
  # Rule 1: app.mysite.com → frontend
  - host: app.mysite.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend-service
            port:
              number: 80

  # Rule 2: api.mysite.com → api-service
  - host: api.mysite.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: api-service
            port:
              number: 80

  # Rule 3: dashboard.mysite.com → dashboard-service
  - host: dashboard.mysite.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: dashboard-service
            port:
              number: 80
```

### How It Works:

```
User visits api.mysite.com/users
        │
        ▼
  DNS resolves api.mysite.com → Ingress Controller's IP
        │
        ▼
  Ingress Controller checks the Host header
        │
        ├── Host = app.mysite.com?       → NO
        ├── Host = api.mysite.com?       → YES → Send to api-service ✅
        └── Host = dashboard.mysite.com? → NO
```

### DNS Setup Required:

For host-based routing to work, you need to set up DNS records pointing all hostnames to the Ingress Controller's external IP:

```
app.mysite.com       → A record → 203.0.113.100 (Ingress Controller IP)
api.mysite.com       → A record → 203.0.113.100
dashboard.mysite.com → A record → 203.0.113.100
```

Or use a **wildcard DNS record**:

```
*.mysite.com → A record → 203.0.113.100
```

## 🔀 Combining Path-Based AND Host-Based Routing

You can absolutely mix both! This is very common in real-world setups:

```yaml
# combined-ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: combined-ingress
spec:
  ingressClassName: nginx
  rules:
  # Main site
  - host: mysite.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend-service
            port:
              number: 80
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: api-service
            port:
              number: 80

  # Admin subdomain with its own paths
  - host: admin.mysite.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: dashboard-service
            port:
              number: 80
      - path: /auth
        pathType: Prefix
        backend:
          service:
            name: auth-service
            port:
              number: 80
```

## 🔒 TLS/SSL Termination — Adding HTTPS

Ingress can handle SSL/TLS certificates so your backend services don't have to. This is called **SSL termination**.

### Step 1: Create a TLS Secret

```bash
# If you have your own certificate:
kubectl create secret tls mysite-tls \
  --cert=path/to/tls.crt \
  --key=path/to/tls.key
```

Or in YAML:

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: mysite-tls
type: kubernetes.io/tls
data:
  tls.crt: <base64-encoded-cert>
  tls.key: <base64-encoded-key>
```

### Step 2: Add TLS to the Ingress

```yaml
# tls-ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: tls-ingress
  annotations:
    nginx.ingress.kubernetes.io/ssl-redirect: "true"  # Force HTTPS
spec:
  ingressClassName: nginx
  tls:                          # <-- TLS configuration
  - hosts:
    - mysite.com
    - api.mysite.com
    secretName: mysite-tls      # <-- Reference to the TLS secret
  rules:
  - host: mysite.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend-service
            port:
              number: 80
  - host: api.mysite.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: api-service
            port:
              number: 80
```

### Automatic TLS with cert-manager (The Best Way!)

Instead of manually creating certificates, use **cert-manager** to automatically get free certificates from **Let's Encrypt**:

```bash
# Install cert-manager
kubectl apply -f https://github.com/cert-manager/cert-manager/releases/download/v1.14.0/cert-manager.yaml
```

```yaml
# Create a ClusterIssuer for Let's Encrypt
apiVersion: cert-manager.io/v1
kind: ClusterIssuer
metadata:
  name: letsencrypt-prod
spec:
  acme:
    server: https://acme-v02.api.letsencrypt.org/directory
    email: your-email@example.com
    privateKeySecretRef:
      name: letsencrypt-prod-key
    solvers:
    - http01:
        ingress:
          class: nginx
```

Now just add an annotation to your Ingress:

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: auto-tls-ingress
  annotations:
    cert-manager.io/cluster-issuer: "letsencrypt-prod"  # <-- Auto TLS!
spec:
  ingressClassName: nginx
  tls:
  - hosts:
    - mysite.com
    secretName: mysite-auto-tls   # cert-manager creates this automatically!
  rules:
  - host: mysite.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend-service
            port:
              number: 80
```

cert-manager will automatically:
1. Request a certificate from Let's Encrypt
2. Complete the HTTP-01 challenge
3. Store the certificate in the secret `mysite-auto-tls`
4. Renew it before it expires

## ⚙️ Important Nginx Ingress Annotations

Annotations are how you configure the Nginx Ingress Controller's behavior. Here are the most commonly used ones:

### Rewrite Rules

```yaml
metadata:
  annotations:
    # Rewrite the URL path before forwarding to the backend
    nginx.ingress.kubernetes.io/rewrite-target: /
    # Example: /api/users → / (strips /api prefix)
```

A more advanced rewrite with regex capture groups:

```yaml
metadata:
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /$2
    nginx.ingress.kubernetes.io/use-regex: "true"
spec:
  rules:
  - host: mysite.com
    http:
      paths:
      - path: /api(/|$)(.*)    # Capture everything after /api
        pathType: ImplementationSpecific
        backend:
          service:
            name: api-service
            port:
              number: 80
```

This means:
- `/api/users` → backend receives `/users`
- `/api/products/123` → backend receives `/products/123`

### Rate Limiting

```yaml
metadata:
  annotations:
    nginx.ingress.kubernetes.io/limit-rps: "10"        # 10 requests per second
    nginx.ingress.kubernetes.io/limit-connections: "5"  # 5 concurrent connections
```

### Timeouts

```yaml
metadata:
  annotations:
    nginx.ingress.kubernetes.io/proxy-connect-timeout: "60"
    nginx.ingress.kubernetes.io/proxy-read-timeout: "120"
    nginx.ingress.kubernetes.io/proxy-send-timeout: "120"
```

### CORS (Cross-Origin Resource Sharing)

```yaml
metadata:
  annotations:
    nginx.ingress.kubernetes.io/enable-cors: "true"
    nginx.ingress.kubernetes.io/cors-allow-origin: "https://mysite.com"
    nginx.ingress.kubernetes.io/cors-allow-methods: "GET, POST, PUT, DELETE"
```

### Authentication

```yaml
metadata:
  annotations:
    # Basic Auth
    nginx.ingress.kubernetes.io/auth-type: basic
    nginx.ingress.kubernetes.io/auth-secret: basic-auth-secret
    nginx.ingress.kubernetes.io/auth-realm: "Authentication Required"
```

### Request Size Limit

```yaml
metadata:
  annotations:
    nginx.ingress.kubernetes.io/proxy-body-size: "50m"   # Allow 50MB uploads
```

### Custom Headers

```yaml
metadata:
  annotations:
    nginx.ingress.kubernetes.io/configuration-snippet: |
      more_set_headers "X-Custom-Header: my-value";
      more_set_headers "X-Frame-Options: DENY";
```

## 🎯 Default Backend — Handling Unmatched Routes

What happens when someone hits a URL that doesn't match any rule? You can set a **default backend**:

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: ingress-with-default
spec:
  ingressClassName: nginx
  defaultBackend:                  # <-- Catch-all
    service:
      name: default-404-service
      port:
        number: 80
  rules:
  - host: mysite.com
    http:
      paths:
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: api-service
            port:
              number: 80
```

Now:
- `mysite.com/api/*` → goes to `api-service`
- `mysite.com/anything-else` → goes to `default-404-service`

```bash
# Check if the Ingress Controller is actually serving your rules
kubectl exec -n ingress-nginx ingress-nginx-controller-xxxxxxx-xxxxx -- cat /etc/nginx/nginx.conf

# Test from inside the cluster
kubectl run test-pod --rm -it --image=curlimages/curl -- sh

```

### Common Issues & Fixes

| Problem | Possible Cause | Fix |
|---|---|---|
| Ingress has no ADDRESS | No Ingress Controller installed | Install nginx-ingress |
| 404 errors | Wrong service name or port | Check `kubectl describe ingress` |
| 503 errors | Backend service has no healthy pods | Check `kubectl get pods` |
| SSL errors | Wrong secret name or expired cert | Check TLS secret |
| `ingressClassName` not working | Old K8s version or annotation needed | Use annotation: `kubernetes.io/ingress.class: nginx` |

## 📊 Complete Ingress Flow — How Everything Connects

Let's trace a request from start to finish:

```
1. User types: https://api.mysite.com/users

2. DNS Resolution:
   api.mysite.com → 203.0.113.100 (Ingress Controller's LoadBalancer IP)

3. Request reaches the Ingress Controller (Nginx pod):
   - Checks the Host header: "api.mysite.com"
   - Checks the path: "/users"
   - Matches rule: host=api.mysite.com, path=/ (Prefix)
   - TLS termination happens here (HTTPS → HTTP)

4. Ingress Controller forwards to the Service:
   → api-service:80

5. Service routes to a healthy Pod:
   → api-pod-abc123:8080 (via kube-proxy/iptables/IPVS)

6. Pod processes the request and returns response

7. Response flows back:
   Pod → Service → Ingress Controller → User
```

Visual:

```
┌──────┐    HTTPS     ┌───────────┐    HTTP    ┌─────────────┐         ┌─────┐
│ User │ ───────────► │  Ingress  │ ─────────► │   Service   │ ──────► │ Pod │
│      │ ◄─────────── │Controller │ ◄───────── │(ClusterIP)  │ ◄────── │     │
└──────┘   Response   └───────────┘            └─────────────┘         └─────┘
               ▲              │
               │              │
          TLS Termination     │
          (HTTPS → HTTP)      │
                              │
                    Reads Ingress rules
                    from Kubernetes API
```

## 🧠 Quick Recap — Cheat Sheet

| Concept | Description |
|---|---|
| **Ingress** | K8s resource defining HTTP routing rules |
| **Ingress Controller** | Software (like Nginx) that implements those rules |
| **Path-Based Routing** | `/api` → service-a, `/web` → service-b |
| **Host-Based Routing** | `api.site.com` → service-a, `web.site.com` → service-b |
| **TLS Termination** | Ingress handles HTTPS; backends get plain HTTP |
| **Default Backend** | Catch-all for unmatched routes |
| **ingressClassName** | Specifies which controller handles the Ingress |
| **Annotations** | Fine-tune behavior (rewrite, rate-limit, CORS, etc.) |

### Essential kubectl Commands:

```bash
# Create/update an ingress
kubectl apply -f my-ingress.yaml
```

## 🎯 Real-World Best Practices

1. **Always use `ingressClassName`** — Don't rely on the default class; be explicit.
2. **Use cert-manager for TLS** — Don't manage certificates manually.
3. **Set rate limits** — Protect your services from abuse.
4. **Use health checks** — Nginx will stop routing to unhealthy backends automatically.
5. **Monitor Ingress Controller metrics** — It exposes Prometheus metrics on port 10254.
6. **Separate Ingress resources per team/namespace** — Don't put everything in one giant Ingress.
7. **Use `rewrite-target` carefully** — Misconfigured rewrites are the #1 cause of 404 errors.
8. **Always set `proxy-body-size`** — The default (1MB) is too small for file uploads.
