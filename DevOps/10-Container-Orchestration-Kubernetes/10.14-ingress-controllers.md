## 🤔 Wait, Why Are There So Many Ingress Controllers?

Great question! Kubernetes designed the Ingress system to be **pluggable**. The Ingress resource is just a standard API — the *how* of routing is left to the controller.

Think of it this way:

```
Ingress Resource = Universal Remote Control (standard buttons)
Ingress Controller = The TV (different brands handle the signals differently)

Same remote, different TVs:
┌──────────────────────┐
│  Ingress Resource    │ ←── Same YAML for all controllers
│  (The Standard)      │
└──────────┬───────────┘
           │
     ┌─────┼──────┬──────────┬──────────┐
     │     │      │          │          │
   Nginx  Traefik HAProxy  AWS ALB   Istio
     │     │      │          │          │
   (Each implements routing differently)
```

Different controllers have different:
- **Features** (auto-discovery, dashboard, canary deploys)
- **Performance** characteristics
- **Configuration** methods
- **Cloud integrations**
- **Learning curves**

## 1️⃣ Nginx Ingress Controller — The Industry Standard

### What Is It?

The **most widely used** Ingress Controller. It runs Nginx (the battle-tested web server/reverse proxy) inside a Kubernetes pod, and dynamically updates its config based on your Ingress rules.

```
┌─────────────────────────────────────────────┐
│         Nginx Ingress Controller            │
│                                             │
│  ┌─────────────────────────────────────┐    │
│  │         Nginx Process               │    │
│  │  (auto-configured from Ingress      │    │
│  │   rules via Lua + Go controller)    │    │
│  └─────────────────────────────────────┘    │
│                                             │
│  Watches: Ingress, Service, Endpoint,       │
│           Secret resources via K8s API      │
└─────────────────────────────────────────────┘
```

### Installation

```bash
# Using Helm (recommended)
helm repo add ingress-nginx https://kubernetes.github.io/ingress-nginx
helm repo update

helm install ingress-nginx ingress-nginx/ingress-nginx \
  --namespace ingress-nginx \
  --create-namespace \
  --set controller.replicaCount=2 \
  --set controller.metrics.enabled=true
```

### Sample Ingress with Nginx

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-nginx-ingress
  annotations:
    # Nginx-specific annotations
    nginx.ingress.kubernetes.io/rewrite-target: /
    nginx.ingress.kubernetes.io/ssl-redirect: "true"
    nginx.ingress.kubernetes.io/proxy-body-size: "50m"
    nginx.ingress.kubernetes.io/rate-limit: "10"
spec:
  ingressClassName: nginx
  tls:
  - hosts:
    - myapp.com
    secretName: myapp-tls
  rules:
  - host: myapp.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend
            port:
              number: 80
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: backend-api
            port:
              number: 8080
```

### Canary Deployments with Nginx

One of the coolest features — you can do canary (gradual) deployments using annotations:

```yaml
# Main ingress (receives 90% traffic)
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: main-ingress
spec:
  ingressClassName: nginx
  rules:
  - host: myapp.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: app-v1        # <-- Old version
            port:
              number: 80

---
# Canary ingress (receives 10% traffic)
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: canary-ingress
  annotations:
    nginx.ingress.kubernetes.io/canary: "true"
    nginx.ingress.kubernetes.io/canary-weight: "10"   # 10% of traffic
spec:
  ingressClassName: nginx
  rules:
  - host: myapp.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: app-v2        # <-- New version
            port:
              number: 80
```

This sends **10% of all traffic** to the new version. You can also route canary traffic based on headers or cookies:

```yaml
annotations:
  nginx.ingress.kubernetes.io/canary: "true"
  nginx.ingress.kubernetes.io/canary-by-header: "X-Canary"
  # Requests with header "X-Canary: always" go to the canary
```

### Custom Nginx Configuration

You can pass custom Nginx config using a ConfigMap:

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: ingress-nginx-controller
  namespace: ingress-nginx
data:
  # Global settings
  proxy-connect-timeout: "10"
  proxy-read-timeout: "120"
  proxy-send-timeout: "120"
  use-forwarded-headers: "true"
  max-worker-connections: "65536"
  worker-processes: "auto"
  
  # Security headers
  hide-headers: "X-Powered-By,Server"
  
  # Enable gzip compression
  use-gzip: "true"
  gzip-types: "application/json application/javascript text/css text/plain"
```

### Pros & Cons

| ✅ Pros | ❌ Cons |
|---|---|
| Most battle-tested and stable | Configuration via annotations can get messy |
| Huge community, tons of docs | No built-in dashboard |
| Excellent performance | No auto service discovery |
| Feature-rich (canary, auth, CORS) | No built-in Let's Encrypt (need cert-manager) |
| Works everywhere (any cloud, bare metal) | Nginx config reloads on changes (brief disruption possible) |

## 2️⃣ Traefik — The Cloud-Native Champion

### What Is It?

Traefik (pronounced "traffic") is a modern, **cloud-native** reverse proxy and Ingress Controller that was built from the ground up for microservices and containers. Its killer feature? **Automatic service discovery** — it watches your Kubernetes API and automatically configures routing without you writing a single annotation.

```
┌─────────────────────────────────────────────────┐
│              Traefik Ingress Controller          │
│                                                 │
│  ┌──────────────┐    ┌──────────────────────┐   │
│  │  Providers   │    │   Built-in Features  │   │
│  │ (K8s, Docker,│    │ - Auto TLS           │   │
│  │  File, etc.) │    │ - Dashboard          │   │
│  └──────────────┘    │ - Metrics            │   │
│                      │ - Middlewares        │   │
│  Auto-discovers      │ - Rate Limiting      │   │
│  services!           └──────────────────────┘   │
└─────────────────────────────────────────────────┘
```

### Installation

```bash
# Using Helm
helm repo add traefik https://traefik.github.io/charts
helm repo update

helm install traefik traefik/traefik \
  --namespace traefik \
  --create-namespace \
  --set dashboard.enabled=true \
  --set ports.websecure.tls.enabled=true
```

### Using Standard Kubernetes Ingress

Traefik supports standard Kubernetes Ingress resources:

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-traefik-ingress
  annotations:
    traefik.ingress.kubernetes.io/router.entrypoints: websecure
    traefik.ingress.kubernetes.io/router.tls: "true"
spec:
  ingressClassName: traefik
  rules:
  - host: myapp.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend
            port:
              number: 80
```

### Using Traefik's Custom CRDs (The Traefik Way)

Traefik has its own Custom Resource Definitions (CRDs) which are **much more powerful** than standard Ingress:

```yaml
# IngressRoute — Traefik's custom Ingress resource
apiVersion: traefik.io/v1alpha1
kind: IngressRoute
metadata:
  name: my-ingressroute
spec:
  entryPoints:
  - websecure
  routes:
  - match: Host(`myapp.com`) && PathPrefix(`/api`)
    kind: Rule
    services:
    - name: api-service
      port: 80
    middlewares:
    - name: rate-limit
    - name: add-prefix

  - match: Host(`myapp.com`)
    kind: Rule
    services:
    - name: frontend
      port: 80
  tls:
    certResolver: letsencrypt     # <-- Built-in Let's Encrypt!
```

### Traefik Middlewares (Like Plugins!)

Middlewares are reusable building blocks that modify requests/responses:

```yaml
# Rate limiting middleware
apiVersion: traefik.io/v1alpha1
kind: Middleware
metadata:
  name: rate-limit
spec:
  rateLimit:
    average: 100
    burst: 50

---
# Strip path prefix middleware
apiVersion: traefik.io/v1alpha1
kind: Middleware
metadata:
  name: strip-api-prefix
spec:
  stripPrefix:
    prefixes:
    - /api

---
# Basic auth middleware
apiVersion: traefik.io/v1alpha1
kind: Middleware
metadata:
  name: basic-auth
spec:
  basicAuth:
    secret: auth-secret

---
# Redirect HTTP to HTTPS
apiVersion: traefik.io/v1alpha1
kind: Middleware
metadata:
  name: redirect-https
spec:
  redirectScheme:
    scheme: https
    permanent: true

---
# Headers middleware (security headers)
apiVersion: traefik.io/v1alpha1
kind: Middleware
metadata:
  name: security-headers
spec:
  headers:
    frameDeny: true
    contentTypeNosniff: true
    browserXssFilter: true
    stsSeconds: 31536000
    stsIncludeSubdomains: true
```

### Built-in Let's Encrypt!

This is Traefik's superpower — no need for cert-manager:

```yaml
# Traefik Helm values
additionalArguments:
- "--certificatesresolvers.letsencrypt.acme.email=your@email.com"
- "--certificatesresolvers.letsencrypt.acme.storage=/data/acme.json"
- "--certificatesresolvers.letsencrypt.acme.httpchallenge.entrypoint=web"
```

Then in your IngressRoute:

```yaml
spec:
  tls:
    certResolver: letsencrypt   # That's it! Auto TLS! 🎉
```

### Traefik Dashboard

Traefik comes with a beautiful built-in web dashboard:

```bash
# Port-forward to access the dashboard
kubectl port-forward -n traefik svc/traefik 9000:9000

# Open: http://localhost:9000/dashboard/
```

The dashboard shows:
- All routers (routing rules)
- All services
- All middlewares
- Health status
- Metrics

### Weighted Round Robin (Traffic Splitting)

```yaml
apiVersion: traefik.io/v1alpha1
kind: TraefikService
metadata:
  name: weighted-service
spec:
  weighted:
    services:
    - name: app-v1
      port: 80
      weight: 90      # 90% of traffic
    - name: app-v2
      port: 80
      weight: 10      # 10% of traffic (canary)
```

### Pros & Cons

| ✅ Pros | ❌ Cons |
|---|---|
| Auto service discovery | Less mature than Nginx |
| Built-in Let's Encrypt | CRDs add complexity (learning curve for CRDs) |
| Beautiful dashboard | Community smaller than Nginx's |
| Middlewares are powerful & reusable | Can be overkill for simple setups |
| Hot reload (no config reload downtime) | Premium features need Traefik Enterprise |
| Native Docker & K8s support | Performance slightly below Nginx for raw throughput |

## 3️⃣ HAProxy Ingress Controller — The Performance Beast

### What Is It?

HAProxy (High Availability Proxy) is one of the **oldest and most reliable** load balancers. It's famous for extremely high performance and is used by companies like GitHub, Reddit, and Stack Overflow. The HAProxy Ingress Controller brings this power to Kubernetes.

```
┌──────────────────────────────────────────────────┐
│          HAProxy Ingress Controller               │
│                                                  │
│  ┌──────────────────────────────────────────┐    │
│  │            HAProxy Process               │    │
│  │  - C-based, extremely fast               │    │
│  │  - Handles millions of connections       │    │
│  │  - Advanced health checking              │    │
│  │  - TCP/HTTP load balancing               │    │
│  └──────────────────────────────────────────┘    │
│                                                  │
│  Known for: Raw performance & reliability        │
└──────────────────────────────────────────────────┘
```

### Installation

There are **two HAProxy Ingress Controllers** (confusing, I know):

| Version | Maintained By |
|---|---|
| `haproxytech/kubernetes-ingress` | HAProxy Technologies (the company) |
| `jcmoraisjr/haproxy-ingress` | Community |

We'll use the HAProxy Technologies version:

```bash
# Using Helm
helm repo add haproxytech https://haproxytech.github.io/helm-charts
helm repo update

helm install haproxy-ingress haproxytech/kubernetes-ingress \
  --namespace haproxy-ingress \
  --create-namespace \
  --set controller.replicaCount=2 \
  --set controller.ingressClass=haproxy
```

### Sample Ingress with HAProxy

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-haproxy-ingress
  annotations:
    # HAProxy-specific annotations
    haproxy.org/rate-limit: "ON"
    haproxy.org/rate-limit-requests: "100"
    haproxy.org/rate-limit-period: "1m"
    haproxy.org/timeout-server: "60s"
    haproxy.org/ssl-redirect: "true"
    haproxy.org/load-balance: "leastconn"   # Load balancing algorithm
spec:
  ingressClassName: haproxy
  tls:
  - hosts:
    - myapp.com
    secretName: myapp-tls
  rules:
  - host: myapp.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend
            port:
              number: 80
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: api-service
            port:
              number: 8080
```

### TCP Load Balancing (HAProxy's Strength)

HAProxy excels at TCP load balancing — perfect for databases, gRPC, and other non-HTTP protocols:

```yaml
# TCP service using ConfigMap
apiVersion: v1
kind: ConfigMap
metadata:
  name: tcp-services
  namespace: haproxy-ingress
data:
  "5432": "default/postgres-service:5432"      # PostgreSQL
  "6379": "default/redis-service:6379"          # Redis
  "27017": "default/mongodb-service:27017"      # MongoDB
```

### Advanced Health Checking

```yaml
metadata:
  annotations:
    haproxy.org/check: "true"
    haproxy.org/check-interval: "5s"
    haproxy.org/check-http: "/healthz"
    haproxy.org/backend-config-snippet: |
      option httpchk GET /healthz
      http-check expect status 200
```

### Custom HAProxy Configuration

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: haproxy-configmap
  namespace: haproxy-ingress
data:
  # Global settings
  maxconn: "65536"
  timeout-connect: "5s"
  timeout-client: "30s"
  timeout-server: "30s"
  
  # Logging
  syslog-server: "address:stdout, format:raw, facility:daemon"
  
  # SSL
  ssl-options: "no-sslv3 no-tlsv10 no-tlsv11"
  ssl-ciphers: "ECDHE+AESGCM:DHE+AESGCM"
```

### Pros & Cons

| ✅ Pros | ❌ Cons |
|---|---|
| Blazing fast performance | Smaller Kubernetes community |
| Excellent TCP/UDP support | Less documentation for K8s specifically |
| Advanced health checking | Fewer K8s-specific features |
| Battle-tested (used by GitHub, Reddit) | Two competing versions (confusing) |
| Great for high-traffic scenarios | No built-in Let's Encrypt |
| Detailed stats page | Steeper learning curve |

## 4️⃣ AWS ALB Ingress Controller (AWS Load Balancer Controller)

### What Is It?

If you're running Kubernetes on **AWS (EKS)**, this is the native option. Instead of running a proxy pod in your cluster, it creates and manages actual **AWS Application Load Balancers** (ALBs) for your Ingress resources.

```
Traditional Ingress Controller:
┌──────────────────────────────────┐
│  Kubernetes Cluster              │
│                                  │
│  ┌────────────────────────────┐  │
│  │  Ingress Controller Pod    │  │  ← Runs INSIDE the cluster
│  │  (Nginx/Traefik/HAProxy)   │  │
│  └────────────────────────────┘  │
│                                  │
└──────────────────────────────────┘

AWS ALB Controller:
┌──────────────────────────────────┐     ┌──────────────────────┐
│  Kubernetes Cluster              │     │   AWS Cloud          │
│                                  │     │                      │
│  ┌────────────────────────────┐  │     │  ┌────────────────┐  │
│  │  ALB Controller Pod        │──┼────►│  │  AWS ALB       │  │
│  │  (just manages AWS ALBs)   │  │     │  │  (actual LB)   │  │
│  └────────────────────────────┘  │     │  └────────────────┘  │
│                                  │     │                      │
└──────────────────────────────────┘     └──────────────────────┘
     Controller runs in cluster           Load balancer runs in AWS
```

The big difference: the actual load balancing happens **outside** the cluster, on AWS infrastructure.

### Installation (on EKS)

#### Step 1: Create an IAM Policy

```bash
# Download the IAM policy
curl -o iam-policy.json https://raw.githubusercontent.com/kubernetes-sigs/aws-load-balancer-controller/v2.7.1/docs/install/iam_policy.json

# Create the IAM policy
aws iam create-policy \
  --policy-name AWSLoadBalancerControllerIAMPolicy \
  --policy-document file://iam-policy.json
```

#### Step 2: Create an IAM Role for Service Account

```bash
eksctl create iamserviceaccount \
  --cluster=my-cluster \
  --namespace=kube-system \
  --name=aws-load-balancer-controller \
  --attach-policy-arn=arn:aws:iam::YOUR_ACCOUNT_ID:policy/AWSLoadBalancerControllerIAMPolicy \
  --approve
```

#### Step 3: Install the Controller

```bash
helm repo add eks https://aws.github.io/eks-charts
helm repo update

helm install aws-load-balancer-controller eks/aws-load-balancer-controller \
  --namespace kube-system \
  --set clusterName=my-cluster \
  --set serviceAccount.create=false \
  --set serviceAccount.name=aws-load-balancer-controller
```

### Sample Ingress with AWS ALB

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-alb-ingress
  annotations:
    # ALB-specific annotations
    alb.ingress.kubernetes.io/scheme: internet-facing       # or "internal"
    alb.ingress.kubernetes.io/target-type: ip               # or "instance"
    alb.ingress.kubernetes.io/listen-ports: '[{"HTTPS":443}]'
    alb.ingress.kubernetes.io/certificate-arn: arn:aws:acm:us-east-1:123456789:certificate/abc-123
    alb.ingress.kubernetes.io/ssl-redirect: "443"
    alb.ingress.kubernetes.io/healthcheck-path: /healthz
    alb.ingress.kubernetes.io/healthcheck-interval-seconds: "15"
    alb.ingress.kubernetes.io/success-codes: "200,301"
    alb.ingress.kubernetes.io/group.name: my-app-group      # Share ALB across Ingresses!
spec:
  ingressClassName: alb
  rules:
  - host: myapp.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend
            port:
              number: 80
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: api-service
            port:
              number: 8080
```

### Key ALB Annotations Explained

```yaml
annotations:
  # Where the ALB is accessible from
  alb.ingress.kubernetes.io/scheme: internet-facing  # Public
  # alb.ingress.kubernetes.io/scheme: internal       # Private (VPC only)

  # How ALB routes to pods
  alb.ingress.kubernetes.io/target-type: ip          # Routes directly to pod IPs (faster)
  # alb.ingress.kubernetes.io/target-type: instance  # Routes to NodePort

  # SSL Certificate from AWS Certificate Manager (ACM)
  alb.ingress.kubernetes.io/certificate-arn: arn:aws:acm:...

  # WAF (Web Application Firewall) integration
  alb.ingress.kubernetes.io/waf-acl-id: "your-waf-acl-id"

  # Cognito authentication (login with AWS Cognito!)
  alb.ingress.kubernetes.io/auth-type: cognito
  alb.ingress.kubernetes.io/auth-idp-cognito: '{"userPoolARN":"...","userPoolClientID":"...","userPoolDomain":"..."}'

  # Sticky sessions
  alb.ingress.kubernetes.io/target-group-attributes: stickiness.enabled=true,stickiness.lb_cookie.duration_seconds=3600

  # Sharing a single ALB across multiple Ingress resources (saves cost!)
  alb.ingress.kubernetes.io/group.name: shared-alb
  alb.ingress.kubernetes.io/group.order: "10"
```

### ALB Ingress Group — Share One ALB (Save Money!)

By default, each Ingress creates a **separate ALB** ($$$). Use `group.name` to share:

```yaml
# Ingress 1 — Frontend
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: frontend-ingress
  annotations:
    alb.ingress.kubernetes.io/group.name: my-shared-alb   # Same group!
    alb.ingress.kubernetes.io/group.order: "1"
spec:
  ingressClassName: alb
  rules:
  - host: myapp.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend
            port:
              number: 80

---
# Ingress 2 — API (shares the same ALB!)
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: api-ingress
  annotations:
    alb.ingress.kubernetes.io/group.name: my-shared-alb   # Same group!
    alb.ingress.kubernetes.io/group.order: "2"
spec:
  ingressClassName: alb
  rules:
  - host: api.myapp.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: api-service
            port:
              number: 8080
```

Both Ingresses share **ONE ALB** — saving you ~$18/month!

### Pros & Cons

| ✅ Pros | ❌ Cons |
|---|---|
| Native AWS integration | AWS-only (not portable) |
| Uses AWS ACM for free SSL certificates | More complex IAM setup |
| AWS WAF integration for security | No TCP/UDP support (HTTP/HTTPS only) |
| Highly available (AWS managed) | Slower to provision (2-3 minutes for new ALB) |
| No proxy pod resource usage in cluster | Costs money per ALB ($18/mo + traffic) |
| Cognito authentication built-in | Harder to debug (AWS console + K8s) |

## 🏆 How to Choose — Decision Matrix

### By Use Case:

```
Are you on AWS?
├── YES → Do you want deep AWS integration (WAF, Cognito, ACM)?
│         ├── YES → AWS ALB Controller ✅
│         └── NO  → Nginx or Traefik (more portable)
│
└── NO → Do you need auto service discovery & built-in TLS?
          ├── YES → Traefik ✅
          └── NO  → Do you need maximum raw performance?
                    ├── YES → HAProxy ✅
                    └── NO  → Nginx ✅ (safe default choice)
```

### By Team Size & Experience:

| Team | Recommendation | Why |
|---|---|---|
| Small startup | Traefik | Easy setup, auto-TLS, dashboard |
| Medium company | Nginx | Most docs, community support |
| Enterprise (AWS) | AWS ALB | Native integration, compliance |
| High-traffic platform | HAProxy | Raw performance |
| Multi-cloud | Nginx or Traefik | Cloud-agnostic |

### By Environment:

| Environment | Best Choice |
|---|---|
| Local (Minikube, Kind) | Nginx (simplest setup) |
| On-premises / Bare Metal | Nginx or HAProxy |
| AWS EKS | AWS ALB Controller |
| GKE | Nginx or GCE Ingress (native) |
| AKS (Azure) | Nginx or Azure Application Gateway |
| Microservices (lots of services) | Traefik |

## 🔄 Running Multiple Ingress Controllers

Yes, you can run **multiple** Ingress Controllers in the same cluster! This is actually common.

Each Ingress resource specifies which controller should handle it using `ingressClassName`:

```yaml
# This Ingress is handled by Nginx
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: internal-ingress
spec:
  ingressClassName: nginx         # <-- Nginx handles this
  rules:
  - host: internal.myapp.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: internal-service
            port:
              number: 80

---
# This Ingress is handled by AWS ALB
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: public-ingress
spec:
  ingressClassName: alb           # <-- ALB handles this
  rules:
  - host: public.myapp.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: public-service
            port:
              number: 80
```

### Check Available IngressClasses

```bash
kubectl get ingressclass

# NAME      CONTROLLER                            PARAMETERS   AGE
# nginx     k8s.io/ingress-nginx                  <none>       30d
# alb       ingress.k8s.aws/alb                   <none>       30d
# traefik   traefik.io/ingress-controller          <none>       30d
```

## 🧠 Quick Summary

| Controller | Best For | One-Line Summary |
|---|---|---|
| **Nginx** | General purpose, most projects | The safe, battle-tested default |
| **Traefik** | Cloud-native, auto-everything | The modern, developer-friendly option |
| **HAProxy** | High-performance, TCP workloads | The raw performance champion |
| **AWS ALB** | AWS-native applications | Deep AWS integration, managed infra |

**If in doubt, start with Nginx.** It works everywhere, has the most documentation, and is what most tutorials and examples use. You can always switch later.
