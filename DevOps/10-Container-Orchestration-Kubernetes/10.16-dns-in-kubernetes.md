# 10.16 — DNS in Kubernetes — CoreDNS & Service Discovery

> **One Line Answer:** DNS in Kubernetes is the magic that lets Pods find and talk to Services by name (like `my-service` or `my-service.my-namespace.svc.cluster.local`) instead of remembering IP addresses — it's powered by **CoreDNS**, a lightweight DNS server that runs inside your cluster and automatically creates DNS records every time you create a Service.

## 🤔 Why Do You Need DNS in Kubernetes?

Imagine you have 50 microservices running in your cluster. Each one needs to talk to other services. But here's the problem:

```
Without DNS:
┌──────────────────────────────────────────────────────────┐
│  Pod A (Frontend) wants to talk to Pod B (Backend API)   │
│                                                          │
│  Frontend code:                                          │
│    fetch("http://10.96.45.123:8080/api/users")          │
│                                                          │
│  But wait... what if the Backend Pod dies and restarts?  │
│  It gets a NEW IP address! 😱                            │
│                                                          │
│  Now Frontend is calling a dead IP address!              │
│  Your app is BROKEN! 💥                                  │
└──────────────────────────────────────────────────────────┘

With DNS:
┌──────────────────────────────────────────────────────────┐
│  Pod A (Frontend) wants to talk to Pod B (Backend API)   │
│                                                          │
│  Frontend code:                                          │
│    fetch("http://backend-api:8080/api/users")           │
│                                                          │
│  Backend Pod dies and restarts with a new IP?            │
│  No problem! DNS automatically resolves "backend-api"    │
│  to the NEW IP address! ✅                               │
│                                                          │
│  Your app keeps running! 🎉                              │
└──────────────────────────────────────────────────────────┘
```

**Key reasons you need DNS in Kubernetes:**

1. **Pods are ephemeral** — they get new IP addresses when they restart
2. **Services provide stable endpoints** — but you still need a way to find them by name
3. **No one wants to hardcode IP addresses** — that's fragile and unmaintainable
4. **Microservices need to discover each other** — DNS makes this automatic

## 📦 What is CoreDNS?

CoreDNS is the **default DNS server** in Kubernetes (since Kubernetes v1.13). Before CoreDNS, Kubernetes used **kube-dns**, but CoreDNS replaced it because it's:

- **Faster** — built in Go, very lightweight
- **Plugin-based** — modular architecture, easy to extend
- **Flexible** — you can customize it with a simple config file called `Corefile`
- **More reliable** — single binary, fewer moving parts

```
┌────────────────────────────────────────────────────────────────┐
│                       CoreDNS Architecture                      │
│                                                                 │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │                      Corefile                             │  │
│  │  (Configuration file that defines how DNS queries         │  │
│  │   are handled — which plugins to use, which zones         │  │
│  │   to serve, where to forward unknown queries)             │  │
│  └──────────────────────────────────────────────────────────┘  │
│                            │                                    │
│                            ▼                                    │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │                    Plugin Chain                            │  │
│  │                                                           │  │
│  │  ┌───────────┐ ┌───────────┐ ┌──────────┐ ┌──────────┐  │  │
│  │  │kubernetes │ │  forward  │ │  cache   │ │   log    │  │  │
│  │  │ plugin    │ │  plugin   │ │  plugin  │ │  plugin  │  │  │
│  │  │           │ │           │ │          │ │          │  │  │
│  │  │ Resolves  │ │ Forwards  │ │ Caches   │ │ Logs    │  │  │
│  │  │ k8s       │ │ to        │ │ DNS      │ │ DNS     │  │  │
│  │  │ service   │ │ upstream  │ │ results  │ │ queries │  │  │
│  │  │ names     │ │ DNS       │ │          │ │         │  │  │
│  │  └───────────┘ └───────────┘ └──────────┘ └──────────┘  │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                 │
│  Runs as a Deployment in the kube-system namespace              │
│  Usually 2 replicas for high availability                       │
└────────────────────────────────────────────────────────────────┘
```

### Where Does CoreDNS Live?

CoreDNS runs as a **Deployment** in the `kube-system` namespace:

```bash
# See CoreDNS pods
kubectl get pods -n kube-system -l k8s-app=kube-dns

# Output:
# NAME                       READY   STATUS    RESTARTS   AGE
# coredns-5d78c9869d-abc12   1/1     Running   0          24h
# coredns-5d78c9869d-def34   1/1     Running   0          24h

# See CoreDNS service
kubectl get svc -n kube-system kube-dns

# Output:
# NAME       TYPE        CLUSTER-IP   EXTERNAL-IP   PORT(S)                  AGE
# kube-dns   ClusterIP   10.96.0.10   <none>        53/UDP,53/TCP,9153/TCP   24h
```

> **Fun fact:** The Service is still called `kube-dns` (for backward compatibility), even though it's actually running CoreDNS!

## 🔍 How Pods Know to Use CoreDNS — The `/etc/resolv.conf` File

When Kubernetes creates a Pod, it automatically configures the Pod's DNS settings by writing a `/etc/resolv.conf` file inside the Pod:

```bash
# Exec into any Pod and check:
kubectl exec -it my-pod -- cat /etc/resolv.conf

# Output:
# nameserver 10.96.0.10
# search default.svc.cluster.local svc.cluster.local cluster.local
# options ndots:5
```

Let's break this down:

```
┌─────────────────────────────────────────────────────────────────────────┐
│  /etc/resolv.conf — explained                                           │
│                                                                         │
│  nameserver 10.96.0.10                                                  │
│  ├── This is the ClusterIP of the kube-dns Service                     │
│  └── ALL DNS queries from this Pod go to CoreDNS at this IP            │
│                                                                         │
│  search default.svc.cluster.local svc.cluster.local cluster.local      │
│  ├── These are "search domains"                                        │
│  ├── When you type just "backend-api", Kubernetes tries:               │
│  │   1. backend-api.default.svc.cluster.local    ← tries this first   │
│  │   2. backend-api.svc.cluster.local            ← then this          │
│  │   3. backend-api.cluster.local                ← then this          │
│  │   4. backend-api                              ← finally bare name  │
│  └── This is why short names like "backend-api" work!                  │
│                                                                         │
│  options ndots:5                                                        │
│  ├── If a name has FEWER than 5 dots, try search domains first         │
│  ├── "backend-api" has 0 dots (< 5) → tries search domains first      │
│  └── "google.com" has 1 dot (< 5) → also tries search domains first   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## 🏷️ Kubernetes DNS Naming Convention — The FQDN

Every Service in Kubernetes gets a **Fully Qualified Domain Name (FQDN)** following this pattern:

```
<service-name>.<namespace>.svc.<cluster-domain>
```

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│  Example: backend-api.production.svc.cluster.local              │
│                                                                  │
│  ┌──────────────┐ ┌────────────┐ ┌─────┐ ┌───────────────┐     │
│  │ backend-api  │.│ production │.│ svc │.│ cluster.local │     │
│  └──────┬───────┘ └──────┬─────┘ └──┬──┘ └───────┬───────┘     │
│         │                │          │             │              │
│    Service Name    Namespace    "service"    Cluster Domain     │
│    (what you       (where it   (fixed       (usually            │
│     created)        lives)     string)      "cluster.local")    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### Different Ways to Reach a Service

```yaml
# From a Pod in the SAME namespace ("default"):
# You can use just the service name!
curl http://backend-api:8080

# From a Pod in a DIFFERENT namespace:
# You need to add the namespace
curl http://backend-api.production:8080

# The full FQDN (works from anywhere):
curl http://backend-api.production.svc.cluster.local:8080
```

```
┌──────────────────────────────────────────────────────────────────────┐
│  When to Use What?                                                    │
│                                                                       │
│  ┌─────────────────────────────────────────────────────────────────┐ │
│  │ Same Namespace?                                                  │ │
│  │ → Just use: "backend-api"                                       │ │
│  │   (search domains auto-expand it)                                │ │
│  ├─────────────────────────────────────────────────────────────────┤ │
│  │ Different Namespace?                                             │ │
│  │ → Use: "backend-api.production"                                 │ │
│  │   (search domains expand "svc.cluster.local" part)              │ │
│  ├─────────────────────────────────────────────────────────────────┤ │
│  │ Want to be super explicit / avoid ambiguity?                     │ │
│  │ → Use the full FQDN: "backend-api.production.svc.cluster.local"│ │
│  │   (no search domain expansion needed — trailing dot optional)   │ │
│  └─────────────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────────────┘
```

## 🔬 DNS Records Kubernetes Creates — A, SRV, and More

### 1. A Records (for Services)

When you create a **ClusterIP Service**, Kubernetes creates an **A record**:

```
my-service.my-namespace.svc.cluster.local → <ClusterIP>
```

```yaml
# Create a simple Service
apiVersion: v1
kind: Service
metadata:
  name: my-web-app
  namespace: production
spec:
  selector:
    app: web
  ports:
    - port: 80          # Service port
      targetPort: 8080  # Container port
  type: ClusterIP
```

```bash
# The DNS record created:
# my-web-app.production.svc.cluster.local → 10.96.123.45 (ClusterIP)

# Test it from inside a Pod:
kubectl run dns-test --image=busybox --rm -it -- nslookup my-web-app.production.svc.cluster.local

# Output:
# Server:    10.96.0.10
# Address:   10.96.0.10:53
#
# Name:      my-web-app.production.svc.cluster.local
# Address:   10.96.123.45
```

### 2. Headless Services (No ClusterIP) — A Records for Individual Pods

A **headless service** (with `clusterIP: None`) doesn't get a single IP. Instead, DNS returns the IP addresses of **all the Pods** behind it:

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-db
  namespace: production
spec:
  clusterIP: None        # ← This makes it headless!
  selector:
    app: database
  ports:
    - port: 5432
```

```bash
# DNS returns MULTIPLE A records (one per Pod):
kubectl run dns-test --image=busybox --rm -it -- nslookup my-db.production.svc.cluster.local

# Output:
# Server:    10.96.0.10
# Address:   10.96.0.10:53
#
# Name:      my-db.production.svc.cluster.local
# Address:   10.244.1.15     ← Pod 1's IP
# Address:   10.244.2.23     ← Pod 2's IP
# Address:   10.244.3.8      ← Pod 3's IP
```

```
┌──────────────────────────────────────────────────────────────────┐
│  Normal Service vs Headless Service — DNS Behavior               │
│                                                                   │
│  Normal (ClusterIP) Service:                                      │
│  ┌──────────┐    DNS     ┌──────────────┐                        │
│  │ my-svc   │ ────────→  │ 10.96.123.45 │  ← Single ClusterIP   │
│  └──────────┘            └──────┬───────┘                        │
│                                 │ kube-proxy routes to           │
│                          ┌──────┴──────┐                         │
│                          │ Pod1 | Pod2 │                         │
│                          └─────────────┘                         │
│                                                                   │
│  Headless Service (clusterIP: None):                              │
│  ┌──────────┐    DNS     ┌──────────────┐                        │
│  │ my-svc   │ ────────→  │ 10.244.1.15  │  ← Pod 1 IP           │
│  └──────────┘            │ 10.244.2.23  │  ← Pod 2 IP           │
│                          │ 10.244.3.8   │  ← Pod 3 IP           │
│                          └──────────────┘                        │
│  Client talks DIRECTLY to Pods (no kube-proxy in between)        │
│  Used for: StatefulSets, databases, direct Pod communication     │
└──────────────────────────────────────────────────────────────────┘
```

### 3. SRV Records

SRV records include **port information** along with the hostname. They're created for **named ports**:

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-web
spec:
  ports:
    - name: http          # ← Named port!
      port: 80
      targetPort: 8080
    - name: https         # ← Named port!
      port: 443
      targetPort: 8443
  selector:
    app: web
```

```bash
# SRV record format:
# _<port-name>._<protocol>.<service>.<namespace>.svc.cluster.local

# Query SRV records:
kubectl run dns-test --image=busybox --rm -it -- nslookup -type=SRV _http._tcp.my-web.default.svc.cluster.local

# Output:
# _http._tcp.my-web.default.svc.cluster.local
#   SRV 0 100 80 my-web.default.svc.cluster.local
```

### 4. Pod DNS Records

Pods can also get their own DNS records! The format is:

```
<pod-ip-with-dashes>.<namespace>.pod.cluster.local
```

```bash
# If Pod IP is 10.244.1.15:
# DNS name: 10-244-1-15.default.pod.cluster.local
#           ↑ dots replaced with dashes!

# Test it:
kubectl run dns-test --image=busybox --rm -it -- nslookup 10-244-1-15.default.pod.cluster.local
```

### 5. StatefulSet Pod DNS Records

StatefulSets are special — each Pod gets a **predictable, stable DNS name**:

```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: mysql
spec:
  serviceName: mysql-headless   # ← Must point to a headless Service!
  replicas: 3
  selector:
    matchLabels:
      app: mysql
  template:
    metadata:
      labels:
        app: mysql
    spec:
      containers:
        - name: mysql
          image: mysql:8.0
          ports:
            - containerPort: 3306
---
apiVersion: v1
kind: Service
metadata:
  name: mysql-headless
spec:
  clusterIP: None
  selector:
    app: mysql
  ports:
    - port: 3306
```

```
┌──────────────────────────────────────────────────────────────────────────┐
│  StatefulSet Pods get STABLE DNS names:                                   │
│                                                                          │
│  Pod Name        DNS Name (FQDN)                                         │
│  ─────────       ──────────────────────────────────────────────          │
│  mysql-0         mysql-0.mysql-headless.default.svc.cluster.local        │
│  mysql-1         mysql-1.mysql-headless.default.svc.cluster.local        │
│  mysql-2         mysql-2.mysql-headless.default.svc.cluster.local        │
│                                                                          │
│  Format: <pod-name>.<headless-service>.<namespace>.svc.cluster.local    │
│                                                                          │
│  Even if mysql-1 dies and restarts:                                      │
│  → It gets the SAME name: mysql-1                                        │
│  → The DNS record updates to point to its new IP                         │
│  → Other Pods can always reach it at the same DNS name! 🎉              │
└──────────────────────────────────────────────────────────────────────────┘
```

## ⚙️ CoreDNS Configuration — The Corefile

CoreDNS is configured through a **ConfigMap** in the `kube-system` namespace:

```bash
# View the CoreDNS ConfigMap
kubectl get configmap coredns -n kube-system -o yaml
```

Here's the default Corefile:

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: coredns
  namespace: kube-system
data:
  Corefile: |
    .:53 {
        errors
        health {
            lameduck 5s
        }
        ready
        kubernetes cluster.local in-addr.arpa ip6.arpa {
            pods insecure
            fallthrough in-addr.arpa ip6.arpa
            ttl 30
        }
        prometheus :9153
        forward . /etc/resolv.conf {
            max_concurrent 1000
        }
        cache 30
        loop
        reload
        loadbalance
    }
```

Let's break down every single line:

```
┌──────────────────────────────────────────────────────────────────────────┐
│  Corefile Explained Line by Line                                          │
│                                                                          │
│  .:53 {                                                                  │
│  │  Listen on port 53 (standard DNS port) for ALL domains (".")          │
│  │                                                                       │
│  │  errors                                                               │
│  │  │  Log errors to stdout                                              │
│  │                                                                       │
│  │  health { lameduck 5s }                                               │
│  │  │  Health check endpoint at :8080/health                             │
│  │  │  Waits 5s before shutting down (graceful shutdown)                  │
│  │                                                                       │
│  │  ready                                                                │
│  │  │  Readiness endpoint at :8181/ready                                 │
│  │                                                                       │
│  │  kubernetes cluster.local in-addr.arpa ip6.arpa {                     │
│  │  │  THE MAIN PLUGIN! Handles k8s service/pod DNS resolution           │
│  │  │  Serves the "cluster.local" zone                                   │
│  │  │  pods insecure — creates Pod DNS records (without verification)    │
│  │  │  fallthrough — if not found in k8s, try next plugin                │
│  │  │  ttl 30 — cache DNS responses for 30 seconds                      │
│  │  }                                                                    │
│  │                                                                       │
│  │  prometheus :9153                                                     │
│  │  │  Expose metrics on port 9153 for Prometheus                        │
│  │                                                                       │
│  │  forward . /etc/resolv.conf { max_concurrent 1000 }                   │
│  │  │  Forward non-k8s DNS queries (like google.com) to the              │
│  │  │  upstream DNS servers (from the node's /etc/resolv.conf)           │
│  │  │  max_concurrent — max 1000 concurrent connections                  │
│  │                                                                       │
│  │  cache 30                                                             │
│  │  │  Cache ALL DNS responses for 30 seconds                            │
│  │                                                                       │
│  │  loop                                                                 │
│  │  │  Detect and stop DNS forwarding loops                              │
│  │                                                                       │
│  │  reload                                                               │
│  │  │  Auto-reload Corefile when ConfigMap changes (no restart needed!)  │
│  │                                                                       │
│  │  loadbalance                                                          │
│  │  │  Round-robin DNS responses                                         │
│  }                                                                       │
└──────────────────────────────────────────────────────────────────────────┘
```

## 🛠️ Customizing CoreDNS — Common Use Cases

### Use Case 1: Forward Specific Domains to Custom DNS

Maybe your company has an internal DNS server for `*.mycompany.internal`:

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: coredns
  namespace: kube-system
data:
  Corefile: |
    .:53 {
        errors
        health { lameduck 5s }
        ready
        kubernetes cluster.local in-addr.arpa ip6.arpa {
            pods insecure
            fallthrough in-addr.arpa ip6.arpa
            ttl 30
        }
        prometheus :9153
        forward . /etc/resolv.conf { max_concurrent 1000 }
        cache 30
        loop
        reload
        loadbalance
    }
    mycompany.internal:53 {
        errors
        cache 30
        forward . 10.0.0.53 10.0.0.54
    }
```

```
# What this does:
# ─────────────────────────────────────────────────────
# Query for "app.mycompany.internal"
#   → Forwarded to 10.0.0.53 or 10.0.0.54 (your internal DNS)
#
# Query for "my-service.default.svc.cluster.local"
#   → Resolved by the kubernetes plugin (CoreDNS)
#
# Query for "google.com"
#   → Forwarded to upstream DNS (from /etc/resolv.conf)
```

### Use Case 2: Add Custom DNS Entries (Static Records)

Sometimes you want to add hardcoded DNS entries:

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: coredns-custom     # ← Some clusters use a separate ConfigMap
  namespace: kube-system
data:
  custom.server: |
    legacy-app.example.com:53 {
        hosts {
            192.168.1.100 legacy-app.example.com
            fallthrough
        }
    }
```

### Use Case 3: Enable DNS Query Logging (for Debugging)

```yaml
# Add the "log" plugin to the Corefile
data:
  Corefile: |
    .:53 {
        log        # ← Add this line to see ALL DNS queries!
        errors
        health { lameduck 5s }
        ready
        kubernetes cluster.local in-addr.arpa ip6.arpa {
            pods insecure
            fallthrough in-addr.arpa ip6.arpa
            ttl 30
        }
        prometheus :9153
        forward . /etc/resolv.conf { max_concurrent 1000 }
        cache 30
        loop
        reload
        loadbalance
    }
```

```bash
# After adding the log plugin, check CoreDNS logs:
kubectl logs -n kube-system -l k8s-app=kube-dns -f

# You'll see every DNS query:
# [INFO] 10.244.1.15:54321 - 12345 "A IN backend-api.default.svc.cluster.local. udp 50 false 512" NOERROR qr,aa 106 0.000321s
```

> ⚠️ **Warning:** Enabling DNS logging in production generates A LOT of logs. Use it only for debugging, then remove it.

## 🔧 DNS Policies — Controlling Pod DNS Behavior

Kubernetes lets you control how Pods handle DNS with the `dnsPolicy` field:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-pod
spec:
  dnsPolicy: ClusterFirst     # ← This is the default
  containers:
    - name: app
      image: nginx
```

### The Four DNS Policies

```
┌─────────────────────────────────────────────────────────────────────────┐
│  DNS Policy          What It Does                                       │
│  ──────────────      ──────────────────────────────────────────────     │
│                                                                         │
│  ClusterFirst        DEFAULT. Uses CoreDNS for cluster names.           │
│  (default)           Falls back to upstream DNS for external names.     │
│                      /etc/resolv.conf → nameserver 10.96.0.10           │
│                                                                         │
│  Default             Uses the NODE's DNS settings (not CoreDNS!).       │
│                      Pod gets the same /etc/resolv.conf as the node.    │
│                      ⚠️ Can't resolve k8s service names!               │
│                                                                         │
│  None                Blank DNS config. You MUST provide your own        │
│                      DNS settings using dnsConfig (see below).          │
│                                                                         │
│  ClusterFirstWithHostNet  Same as ClusterFirst, but for Pods using     │
│                           hostNetwork: true. Without this, hostNetwork │
│                           Pods would use the node's DNS by default.     │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### Custom DNS Configuration with `dnsConfig`

You can fine-tune DNS settings for individual Pods:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: custom-dns-pod
spec:
  dnsPolicy: None            # ← Start with a blank slate
  dnsConfig:
    nameservers:
      - 8.8.8.8              # Google DNS
      - 10.96.0.10           # CoreDNS (so we can still resolve k8s services)
    searches:
      - default.svc.cluster.local
      - svc.cluster.local
      - cluster.local
      - mycompany.internal    # ← Add your company domain!
    options:
      - name: ndots
        value: "5"
      - name: timeout
        value: "2"           # DNS timeout of 2 seconds
      - name: attempts
        value: "3"           # Retry 3 times
  containers:
    - name: app
      image: nginx
```

## 🔍 Service Discovery — Finding Services Automatically

Service Discovery in Kubernetes has two mechanisms:

### Method 1: DNS-Based Service Discovery (Recommended!)

This is what we've been discussing — using DNS names to find Services:

```yaml
# Your application code can just use service names:
# Python example
import requests

# Same namespace — just the service name
response = requests.get("http://backend-api:8080/api/users")

# Different namespace — include the namespace
response = requests.get("http://backend-api.production:8080/api/users")
```

### Method 2: Environment Variable-Based Service Discovery

Kubernetes also injects environment variables for each Service into every Pod:

```bash
# Exec into any Pod:
kubectl exec -it my-pod -- env | grep BACKEND

# Output (for a service called "backend-api" with ClusterIP 10.96.45.123):
# BACKEND_API_SERVICE_HOST=10.96.45.123
# BACKEND_API_SERVICE_PORT=8080
# BACKEND_API_PORT=tcp://10.96.45.123:8080
# BACKEND_API_PORT_8080_TCP=tcp://10.96.45.123:8080
# BACKEND_API_PORT_8080_TCP_PROTO=tcp
# BACKEND_API_PORT_8080_TCP_PORT=8080
# BACKEND_API_PORT_8080_TCP_ADDR=10.96.45.123
```

```
┌──────────────────────────────────────────────────────────────────────┐
│  DNS vs Environment Variables — Which to Use?                        │
│                                                                       │
│  DNS (Recommended ✅)              Environment Variables (Legacy ⚠️) │
│  ─────────────────────             ─────────────────────────────     │
│  ✅ Works across namespaces        ❌ Only for same namespace        │
│  ✅ Dynamic (updates in real-time) ❌ Set at Pod creation only       │
│  ✅ No ordering dependency         ❌ Service must exist BEFORE Pod  │
│  ✅ Standard DNS protocol          ❌ Kubernetes-specific            │
│  ✅ Supports all record types      ❌ Only host:port info            │
│                                                                       │
│  Verdict: Always prefer DNS! Environment variables are legacy.       │
└──────────────────────────────────────────────────────────────────────┘
```

## 🧪 Hands-On: Debugging DNS in Kubernetes

### Step 1: Deploy a DNS Testing Pod

```yaml
# dns-debug-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: dns-debug
  namespace: default
spec:
  containers:
    - name: dnsutils
      image: registry.k8s.io/e2e-test-images/jessie-dnsutils:1.3
      command:
        - sleep
        - "infinity"
```

```bash
kubectl apply -f dns-debug-pod.yaml
kubectl exec -it dns-debug -- bash
```

### Step 2: Test DNS Resolution

```bash
# Inside the dns-debug pod:

# 1. Check /etc/resolv.conf
cat /etc/resolv.conf
# nameserver 10.96.0.10
# search default.svc.cluster.local svc.cluster.local cluster.local
# options ndots:5

# 2. Resolve a Kubernetes service
nslookup kubernetes.default.svc.cluster.local
# Server:     10.96.0.10
# Address:    10.96.0.10#53
#
# Name:       kubernetes.default.svc.cluster.local
# Address:    10.96.0.1

# 3. Resolve using short name (search domains kick in)
nslookup kubernetes
# Same result! Because search domains expand it.

# 4. Resolve an external domain
nslookup google.com
# Server:     10.96.0.10
# Address:    10.96.0.10#53
#
# Non-authoritative answer:
# Name:       google.com
# Address:    142.250.80.46

# 5. Use dig for more detailed output
dig backend-api.default.svc.cluster.local

# 6. Check SRV records
dig SRV _http._tcp.my-web.default.svc.cluster.local
```

### Step 3: Common DNS Debugging Commands

```bash
# Check if CoreDNS pods are running
kubectl get pods -n kube-system -l k8s-app=kube-dns

# Check CoreDNS logs for errors
kubectl logs -n kube-system -l k8s-app=kube-dns

# Check if CoreDNS service exists
kubectl get svc kube-dns -n kube-system

# Check CoreDNS endpoints
kubectl get endpoints kube-dns -n kube-system

# Check the ConfigMap
kubectl describe configmap coredns -n kube-system

# Restart CoreDNS (if you changed the ConfigMap)
kubectl rollout restart deployment coredns -n kube-system
```

## ⚠️ Common DNS Issues & Troubleshooting

### Issue 1: "Could not resolve host" Error

```bash
# Symptom: Pod can't resolve any DNS names
kubectl exec -it my-pod -- nslookup google.com
# ;; connection timed out; no servers could be reached
```

**Fix checklist:**

```bash
# 1. Is CoreDNS running?
kubectl get pods -n kube-system -l k8s-app=kube-dns
# If pods are in CrashLoopBackOff → check logs

# 2. Is the kube-dns Service there?
kubectl get svc kube-dns -n kube-system
# Must have a ClusterIP

# 3. Can the Pod reach CoreDNS?
kubectl exec -it my-pod -- nslookup kubernetes.default 10.96.0.10
# If this fails → network issue between Pod and CoreDNS

# 4. Check firewall / network policies
# Make sure port 53 (UDP & TCP) is not blocked
```

### Issue 2: DNS Resolution is Slow

```
┌──────────────────────────────────────────────────────────────────────┐
│  The ndots Problem — Why External DNS Lookups Can Be Slow            │
│                                                                       │
│  With ndots:5, looking up "api.example.com" (2 dots, less than 5):  │
│                                                                       │
│  1. api.example.com.default.svc.cluster.local    → NOT FOUND (wait) │
│  2. api.example.com.svc.cluster.local            → NOT FOUND (wait) │
│  3. api.example.com.cluster.local                → NOT FOUND (wait) │
│  4. api.example.com                              → FOUND! ✅         │
│                                                                       │
│  That's 3 WASTED DNS queries before finding the answer!              │
│  At ~5ms each = 15ms of unnecessary latency 😫                       │
│                                                                       │
│  Fix Option 1: Use FQDN with trailing dot                            │
│    curl http://api.example.com.                                      │
│    (trailing dot = "this is already fully qualified, don't search")   │
│                                                                       │
│  Fix Option 2: Lower ndots for your Pod                              │
│    dnsConfig:                                                         │
│      options:                                                         │
│        - name: ndots                                                  │
│          value: "2"                                                   │
│                                                                       │
│  Fix Option 3: Use NodeLocal DNSCache (see below)                    │
└──────────────────────────────────────────────────────────────────────┘
```

### Issue 3: DNS Works but Service Name Doesn't Resolve

```bash
# Symptom:
nslookup my-service.default.svc.cluster.local
# ** server can't find my-service.default.svc.cluster.local: NXDOMAIN

# Common causes:
# 1. Service doesn't exist in that namespace
kubectl get svc my-service -n default

# 2. Wrong namespace
kubectl get svc my-service --all-namespaces

# 3. Service has no endpoints (no Pods matching the selector)
kubectl get endpoints my-service -n default
# If ENDPOINTS is <none>, no Pods match the selector!
kubectl describe svc my-service -n default
# Check the "Selector" — does it match your Pod labels?
```

## 🚀 NodeLocal DNSCache — Speeding Up DNS

NodeLocal DNSCache runs a DNS cache on every node, reducing latency and load on CoreDNS:

```
Without NodeLocal DNSCache:
┌────────┐     DNS query     ┌──────────┐
│  Pod   │ ──────────────→   │ CoreDNS  │   (every query goes to CoreDNS)
│        │                   │ Pod      │   (across the network)
└────────┘                   └──────────┘

With NodeLocal DNSCache:
┌────────┐     DNS query     ┌────────────────┐     cache miss     ┌──────────┐
│  Pod   │ ──────────────→   │ NodeLocal DNS  │ ──────────────→    │ CoreDNS  │
│        │  (stays on node!) │ Cache (on same │    (only if not   │ Pod      │
└────────┘                   │ node as Pod)   │     cached)       └──────────┘
                             └────────────────┘

Benefits:
✅ Most queries answered from local cache (super fast!)
✅ Reduces load on CoreDNS
✅ Reduces network traffic
✅ Avoids conntrack issues (uses TCP instead of UDP for upstream)
```

To enable NodeLocal DNSCache:

```bash
# Deploy the NodeLocal DNSCache DaemonSet
# (usually done via a YAML manifest from the Kubernetes repo)
kubectl apply -f https://raw.githubusercontent.com/kubernetes/kubernetes/master/cluster/addons/dns/nodelocaldns/nodelocaldns.yaml
```

## 📋 ExternalName Services — DNS Aliasing

ExternalName is a special Service type that creates a **CNAME** DNS record instead of routing traffic:

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-database
  namespace: default
spec:
  type: ExternalName
  externalName: db-prod.abc123.us-east-1.rds.amazonaws.com
```

```
┌──────────────────────────────────────────────────────────────────────┐
│  What this does:                                                      │
│                                                                       │
│  Your app calls: http://my-database:5432                             │
│                                                                       │
│  CoreDNS resolves "my-database" as a CNAME record:                   │
│  my-database.default.svc.cluster.local                               │
│    → CNAME → db-prod.abc123.us-east-1.rds.amazonaws.com             │
│                                                                       │
│  Use case:                                                            │
│  - Point to an external database (RDS, Cloud SQL, etc.)              │
│  - Migrate from in-cluster to external DB without changing app code  │
│  - Create an alias for external services                              │
│                                                                       │
│  ⚠️ Note: ExternalName does NOT proxy traffic — it only creates      │
│  a CNAME record. No port mapping, no load balancing.                 │
└──────────────────────────────────────────────────────────────────────┘
```

## 🧠 Quick Summary Table

| Concept | What It Does |
|---------|-------------|
| **CoreDNS** | The DNS server running inside your cluster (kube-system namespace) |
| **Corefile** | CoreDNS config file (stored as a ConfigMap) |
| **Service A Record** | `<svc>.<ns>.svc.cluster.local` → ClusterIP |
| **Headless Service** | `clusterIP: None` → DNS returns individual Pod IPs |
| **StatefulSet DNS** | `<pod-name>.<headless-svc>.<ns>.svc.cluster.local` → stable Pod DNS |
| **Pod A Record** | `<pod-ip-dashed>.<ns>.pod.cluster.local` → Pod IP |
| **SRV Record** | `_<port>._<proto>.<svc>.<ns>.svc.cluster.local` → port + host |
| **ExternalName** | CNAME record to external service |
| **dnsPolicy** | Controls which DNS server a Pod uses |
| **ndots** | Number of dots before treating a name as FQDN |
| **NodeLocal DNSCache** | Per-node DNS cache for better performance |
| **Search domains** | Auto-expansion of short names to FQDNs |

---

## 🎯 Key Takeaways

1. **CoreDNS is the brain** — it runs in `kube-system` and resolves all cluster DNS queries
2. **Every Service gets a DNS name** — `<service>.<namespace>.svc.cluster.local`
3. **Short names work in the same namespace** — just use `my-service` instead of the full FQDN
4. **Headless Services** return individual Pod IPs (great for StatefulSets and databases)
5. **Always prefer DNS over environment variables** for service discovery
6. **The `ndots:5` setting** can cause slow external DNS — use FQDN with trailing dot or lower ndots
7. **Debug DNS** with `nslookup`, `dig`, and the `dnsutils` Pod
8. **NodeLocal DNSCache** dramatically improves DNS performance in large clusters
