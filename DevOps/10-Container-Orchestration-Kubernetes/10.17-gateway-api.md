# 10.17 — Gateway API — Next-Gen Ingress

> **One Line Answer:** The Gateway API is the **next evolution of Kubernetes Ingress** — it's a more powerful, expressive, and role-oriented way to manage how external traffic enters your cluster, supporting advanced features like traffic splitting, header-based routing, and cross-namespace routing that the old Ingress resource simply can't do.

## 🤔 Why Do We Need Gateway API? What's Wrong with Ingress?

Let's be honest — Kubernetes Ingress works, but it has some serious problems:

```
┌──────────────────────────────────────────────────────────────────────────┐
│                    Problems with Traditional Ingress                      │
│                                                                          │
│  1. TOO SIMPLE                                                           │
│     Ingress only supports basic path-based and host-based routing.       │
│     Want to route based on headers? Query params? → Use annotations! 😩  │
│                                                                          │
│  2. ANNOTATION HELL                                                      │
│     Every Ingress Controller has its OWN custom annotations:             │
│                                                                          │
│     nginx.ingress.kubernetes.io/rewrite-target: /                       │
│     nginx.ingress.kubernetes.io/ssl-redirect: "true"                    │
│     nginx.ingress.kubernetes.io/rate-limiting: ...                      │
│     traefik.ingress.kubernetes.io/router.middlewares: ...               │
│                                                                          │
│     These are NOT portable! Switch from Nginx to Traefik? Rewrite       │
│     ALL your annotations! 🤯                                            │
│                                                                          │
│  3. ONE RESOURCE DOES EVERYTHING                                         │
│     With Ingress, a single YAML file controls:                           │
│     - Infrastructure (which load balancer to use)                        │
│     - Routing rules (where to send traffic)                              │
│     - TLS settings                                                       │
│     One person manages it all — no separation of concerns.               │
│                                                                          │
│  4. NO CROSS-NAMESPACE SUPPORT                                           │
│     Ingress can only route to Services in the SAME namespace.            │
│     Want to route from "frontend" namespace to "backend" namespace?     │
│     Not natively supported! 😤                                           │
│                                                                          │
│  5. LIMITED PROTOCOL SUPPORT                                             │
│     Ingress was designed for HTTP/HTTPS only.                            │
│     Need TCP? UDP? gRPC? → Hacks and workarounds.                       │
│                                                                          │
└──────────────────────────────────────────────────────────────────────────┘
```

**Gateway API solves ALL of these problems.** Let's see how!

## 🏗️ The Big Picture — Gateway API Architecture

The Gateway API splits traffic management into **three resources** managed by **three different personas**:

```
┌──────────────────────────────────────────────────────────────────────────┐
│                      Gateway API — Role Separation                       │
│                                                                          │
│  ┌──────────────────────────────────────────────────────────────────┐   │
│  │  INFRASTRUCTURE PROVIDER (Cluster Operator / Platform Team)      │   │
│  │                                                                   │   │
│  │  Creates: GatewayClass                                           │   │
│  │  "Which load balancer / gateway implementation to use?"           │   │
│  │                                                                   │   │
│  │  Example: "We use AWS ALB" or "We use Envoy" or "We use Nginx"  │   │
│  └──────────────────────────────┬───────────────────────────────────┘   │
│                                 │                                        │
│                                 ▼                                        │
│  ┌──────────────────────────────────────────────────────────────────┐   │
│  │  CLUSTER OPERATOR (Platform / Networking Team)                   │   │
│  │                                                                   │   │
│  │  Creates: Gateway                                                │   │
│  │  "Set up a load balancer with these listeners (ports, protocols, │   │
│  │   TLS certs, hostnames)"                                         │   │
│  │                                                                   │   │
│  │  Example: "Listen on port 443 for *.myapp.com with this TLS     │   │
│  │   certificate"                                                    │   │
│  └──────────────────────────────┬───────────────────────────────────┘   │
│                                 │                                        │
│                                 ▼                                        │
│  ┌──────────────────────────────────────────────────────────────────┐   │
│  │  APPLICATION DEVELOPER (Dev Teams)                               │   │
│  │                                                                   │   │
│  │  Creates: HTTPRoute / GRPCRoute / TCPRoute / TLSRoute           │   │
│  │  "Route traffic matching these rules to my Service"              │   │
│  │                                                                   │   │
│  │  Example: "Send /api/* traffic to backend-svc,                   │   │
│  │   /web/* to frontend-svc"                                        │   │
│  └──────────────────────────────────────────────────────────────────┘   │
│                                                                          │
└──────────────────────────────────────────────────────────────────────────┘
```

Compare this to the old Ingress:

```
Old Ingress (everything in one resource):
┌────────────────────────────────────────┐
│  apiVersion: networking.k8s.io/v1      │
│  kind: Ingress                         │
│  metadata:                             │
│    annotations:                        │  ← Infrastructure + Routing
│      nginx.ingress.../ssl: "true"      │     all mixed together!
│  spec:                                 │
│    tls: ...                            │  ← TLS config
│    rules:                              │
│      - host: myapp.com                 │  ← Routing rules
│        http:                           │
│          paths: ...                    │
└────────────────────────────────────────┘

New Gateway API (separated into roles):
┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│ GatewayClass │   │   Gateway    │   │  HTTPRoute   │
│              │   │              │   │              │
│ Infra team   │ → │ Network team │ → │ Dev team     │
│ sets up      │   │ configures   │   │ defines      │
│ provider     │   │ listeners    │   │ routing      │
└──────────────┘   └──────────────┘   └──────────────┘
```

## 📦 Gateway API Resources — Deep Dive

### 1. GatewayClass — "Which Implementation?"

A GatewayClass defines which **controller** (implementation) will handle your Gateways. Think of it like a Kubernetes `StorageClass` but for networking:

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: GatewayClass
metadata:
  name: my-gateway-class
spec:
  controllerName: gateway.envoyproxy.io/gatewayclass-controller
  # This tells Kubernetes: "Use Envoy to handle Gateways of this class"
```

```
┌──────────────────────────────────────────────────────────────────────┐
│  Common GatewayClass Controllers:                                     │
│                                                                       │
│  Provider          controllerName                                    │
│  ────────          ──────────────────────────────────────────        │
│  Envoy Gateway     gateway.envoyproxy.io/gatewayclass-controller    │
│  Istio             istio.io/gateway-controller                       │
│  Nginx Gateway     gateway.nginx.org/nginx-gateway-controller       │
│  Traefik           traefik.io/gateway-controller                    │
│  Cilium            io.cilium/gateway-controller                      │
│  AWS ALB           eks.amazonaws.com/alb-gateway-controller          │
│  Kong              konghq.com/kic-gateway-controller                │
│  HAProxy           haproxy.org/gateway-controller                    │
│                                                                       │
│  You pick one based on your infrastructure!                           │
└──────────────────────────────────────────────────────────────────────┘
```

```bash
# List available GatewayClasses in your cluster
kubectl get gatewayclasses

# Output:
# NAME               CONTROLLER                                       ACCEPTED   AGE
# my-gateway-class   gateway.envoyproxy.io/gatewayclass-controller    True       24h
```

> **Who creates this?** The infrastructure / platform team. Usually created once and shared across the whole cluster.

### 2. Gateway — "Set Up the Load Balancer"

A Gateway creates the actual load balancer / entry point and defines **listeners** (what ports and protocols to accept traffic on):

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: my-gateway
  namespace: infra
spec:
  gatewayClassName: my-gateway-class     # ← Which GatewayClass to use
  listeners:
    # Listener 1: HTTP on port 80
    - name: http
      protocol: HTTP
      port: 80
      allowedRoutes:
        namespaces:
          from: All          # Allow routes from ALL namespaces

    # Listener 2: HTTPS on port 443
    - name: https
      protocol: HTTPS
      port: 443
      hostname: "*.myapp.com"    # ← Only accept traffic for *.myapp.com
      tls:
        mode: Terminate          # ← Terminate TLS at the Gateway
        certificateRefs:
          - name: myapp-tls-cert
            kind: Secret
      allowedRoutes:
        namespaces:
          from: Selector
          selector:
            matchLabels:
              shared-gateway: "true"   # Only namespaces with this label
```

Let's break this down:

```
┌──────────────────────────────────────────────────────────────────────────┐
│  Gateway Spec Explained                                                   │
│                                                                          │
│  gatewayClassName: my-gateway-class                                      │
│  │  Which implementation to use (Envoy, Nginx, Istio, etc.)             │
│                                                                          │
│  listeners:                                                              │
│  │                                                                       │
│  ├── name: http                                                          │
│  │   protocol: HTTP                                                      │
│  │   port: 80                                                            │
│  │   allowedRoutes:                                                      │
│  │     namespaces:                                                       │
│  │       from: All         ← Any namespace can attach routes here        │
│  │                                                                       │
│  ├── name: https                                                         │
│  │   protocol: HTTPS                                                     │
│  │   port: 443                                                           │
│  │   hostname: "*.myapp.com"  ← Only accept traffic for this hostname   │
│  │   tls:                                                                │
│  │     mode: Terminate      ← TLS is terminated at the gateway          │
│  │     certificateRefs:     ← Reference to a TLS Secret                 │
│  │       - name: myapp-tls-cert                                         │
│  │   allowedRoutes:                                                      │
│  │     namespaces:                                                       │
│  │       from: Selector     ← Only namespaces with matching labels      │
│  │       selector:                                                       │
│  │         matchLabels:                                                  │
│  │           shared-gateway: "true"                                      │
│                                                                          │
└──────────────────────────────────────────────────────────────────────────┘
```

```bash
# Check the Gateway status
kubectl get gateway my-gateway -n infra

# Output:
# NAME         CLASS              ADDRESS        PROGRAMMED   AGE
# my-gateway   my-gateway-class   203.0.113.10   True         1h

# The "ADDRESS" is your external IP/hostname — point your DNS here!
```

> **Who creates this?** The cluster/network operator team. They manage the load balancers, TLS certs, and access policies.

### 3. HTTPRoute — "Where Does Traffic Go?"

HTTPRoute is where **developers** define their routing rules. This is the most commonly used route type:

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: my-app-routes
  namespace: my-app          # ← Developer's namespace!
spec:
  parentRefs:
    - name: my-gateway       # ← Attach to this Gateway
      namespace: infra       # ← The Gateway lives in "infra" namespace
      sectionName: https     # ← Specifically the "https" listener

  hostnames:
    - "api.myapp.com"        # ← Route traffic for this hostname

  rules:
    # Rule 1: /api/v1/* → backend-v1 service
    - matches:
        - path:
            type: PathPrefix
            value: /api/v1
      backendRefs:
        - name: backend-v1
          port: 8080

    # Rule 2: /api/v2/* → backend-v2 service
    - matches:
        - path:
            type: PathPrefix
            value: /api/v2
      backendRefs:
        - name: backend-v2
          port: 8080

    # Rule 3: Everything else → frontend service
    - matches:
        - path:
            type: PathPrefix
            value: /
      backendRefs:
        - name: frontend
          port: 3000
```

```
┌──────────────────────────────────────────────────────────────────────────┐
│  How the Traffic Flows:                                                   │
│                                                                          │
│  User → https://api.myapp.com/api/v1/users                              │
│                                                                          │
│  1. DNS resolves api.myapp.com → 203.0.113.10 (Gateway's IP)           │
│                                                                          │
│  2. Gateway receives the request on port 443 (HTTPS listener)           │
│     → Terminates TLS                                                     │
│     → Checks hostname: "api.myapp.com" ✅ matches                       │
│                                                                          │
│  3. Gateway looks at HTTPRoutes attached to it                           │
│     → Finds "my-app-routes" which handles "api.myapp.com"              │
│                                                                          │
│  4. HTTPRoute matches the path:                                          │
│     /api/v1/users → matches PathPrefix "/api/v1" → Rule 1              │
│                                                                          │
│  5. Traffic is sent to:                                                   │
│     backend-v1:8080 in the "my-app" namespace                           │
│                                                                          │
└──────────────────────────────────────────────────────────────────────────┘
```

> **Who creates this?** The application developers! They only need to know the Gateway name — they don't need to worry about the load balancer, TLS certs, or infrastructure.

## 🔥 Advanced Routing Features

This is where Gateway API really shines compared to old Ingress!

### Feature 1: Header-Based Routing

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: header-routing
spec:
  parentRefs:
    - name: my-gateway
      namespace: infra
  hostnames:
    - "api.myapp.com"
  rules:
    # Route requests with "X-Version: beta" header to beta service
    - matches:
        - headers:
            - name: X-Version
              value: beta
      backendRefs:
        - name: backend-beta
          port: 8080

    # All other requests go to stable service
    - matches:
        - path:
            type: PathPrefix
            value: /
      backendRefs:
        - name: backend-stable
          port: 8080
```

```
# Try it:
# curl -H "X-Version: beta" https://api.myapp.com/users    → goes to backend-beta
# curl https://api.myapp.com/users                          → goes to backend-stable
```

### Feature 2: Traffic Splitting (Canary Deployments!)

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: canary-route
spec:
  parentRefs:
    - name: my-gateway
      namespace: infra
  hostnames:
    - "api.myapp.com"
  rules:
    - matches:
        - path:
            type: PathPrefix
            value: /
      backendRefs:
        # 90% of traffic → stable version
        - name: backend-v1
          port: 8080
          weight: 90

        # 10% of traffic → new version (canary!)
        - name: backend-v2
          port: 8080
          weight: 10
```

```
┌──────────────────────────────────────────────────────────────────────────┐
│  Traffic Splitting Visualization:                                         │
│                                                                          │
│  100 requests to api.myapp.com                                           │
│                                                                          │
│  ████████████████████████████████████████████████████ 90 → backend-v1   │
│  ██████                                              10 → backend-v2   │
│                                                                          │
│  This is PERFECT for canary deployments!                                 │
│  Slowly increase v2's weight: 10% → 25% → 50% → 100%                   │
│  If v2 has problems, roll back to 100% v1 instantly!                     │
└──────────────────────────────────────────────────────────────────────────┘
```

### Feature 3: Query Parameter Matching

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: query-routing
spec:
  parentRefs:
    - name: my-gateway
      namespace: infra
  rules:
    # Route requests with ?debug=true to debug service
    - matches:
        - queryParams:
            - name: debug
              value: "true"
      backendRefs:
        - name: debug-service
          port: 8080

    - backendRefs:
        - name: production-service
          port: 8080
```

### Feature 4: HTTP Method Matching

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: method-routing
spec:
  parentRefs:
    - name: my-gateway
      namespace: infra
  rules:
    # GET requests → read-only service (can be cached)
    - matches:
        - method: GET
          path:
            type: PathPrefix
            value: /api
      backendRefs:
        - name: read-service
          port: 8080

    # POST/PUT/DELETE → write service
    - matches:
        - method: POST
          path:
            type: PathPrefix
            value: /api
        - method: PUT
          path:
            type: PathPrefix
            value: /api
        - method: DELETE
          path:
            type: PathPrefix
            value: /api
      backendRefs:
        - name: write-service
          port: 8080
```

### Feature 5: Request Header Modification

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: header-modify
spec:
  parentRefs:
    - name: my-gateway
      namespace: infra
  rules:
    - matches:
        - path:
            type: PathPrefix
            value: /api
      filters:
        # Add a header
        - type: RequestHeaderModifier
          requestHeaderModifier:
            add:
              - name: X-Request-Source
                value: gateway
            set:
              - name: X-Forwarded-Proto
                value: https
            remove:
              - X-Internal-Only
      backendRefs:
        - name: backend
          port: 8080
```

### Feature 6: URL Rewriting

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: url-rewrite
spec:
  parentRefs:
    - name: my-gateway
      namespace: infra
  rules:
    # Rewrite /store/products → /api/v2/products
    - matches:
        - path:
            type: PathPrefix
            value: /store
      filters:
        - type: URLRewrite
          urlRewrite:
            path:
              type: ReplacePrefixMatch
              replacePrefixMatch: /api/v2
      backendRefs:
        - name: store-backend
          port: 8080
```

### Feature 7: Request Redirect

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: redirect
spec:
  parentRefs:
    - name: my-gateway
      namespace: infra
      sectionName: http       # ← Attach to HTTP listener (port 80)
  rules:
    # Redirect ALL HTTP traffic to HTTPS
    - filters:
        - type: RequestRedirect
          requestRedirect:
            scheme: https
            statusCode: 301
```

## 🌐 Cross-Namespace Routing

One of the BEST features of Gateway API — routing across namespaces!

```yaml
# Gateway in "infra" namespace (managed by platform team)
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: shared-gateway
  namespace: infra
spec:
  gatewayClassName: my-gateway-class
  listeners:
    - name: https
      protocol: HTTPS
      port: 443
      hostname: "*.myapp.com"
      tls:
        mode: Terminate
        certificateRefs:
          - name: wildcard-cert
      allowedRoutes:
        namespaces:
          from: All           # ← Allow ANY namespace to attach routes!
---
# HTTPRoute in "team-a" namespace (managed by Team A)
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: team-a-routes
  namespace: team-a           # ← Team A's namespace
spec:
  parentRefs:
    - name: shared-gateway
      namespace: infra        # ← Reference the shared Gateway
  hostnames:
    - "team-a.myapp.com"
  rules:
    - backendRefs:
        - name: team-a-service
          port: 8080
---
# HTTPRoute in "team-b" namespace (managed by Team B)
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: team-b-routes
  namespace: team-b           # ← Team B's namespace
spec:
  parentRefs:
    - name: shared-gateway
      namespace: infra        # ← Same shared Gateway!
  hostnames:
    - "team-b.myapp.com"
  rules:
    - backendRefs:
        - name: team-b-service
          port: 8080
```

```
┌──────────────────────────────────────────────────────────────────────────┐
│  Cross-Namespace Routing — How It Works:                                  │
│                                                                          │
│  Namespace: infra                                                        │
│  ┌─────────────────────────────────────────────┐                        │
│  │  Gateway: shared-gateway                     │                        │
│  │  Listener: port 443, *.myapp.com             │                        │
│  │  allowedRoutes: All namespaces               │                        │
│  └──────────────────┬──────────────┬────────────┘                        │
│                     │              │                                      │
│            ┌────────┘              └────────┐                            │
│            ▼                                ▼                            │
│  Namespace: team-a                Namespace: team-b                      │
│  ┌────────────────────┐          ┌────────────────────┐                  │
│  │ HTTPRoute:          │          │ HTTPRoute:          │                  │
│  │ team-a.myapp.com    │          │ team-b.myapp.com    │                  │
│  │   → team-a-service  │          │   → team-b-service  │                  │
│  └────────────────────┘          └────────────────────┘                  │
│                                                                          │
│  Each team manages their OWN routes in their OWN namespace!              │
│  Platform team manages the shared Gateway.                               │
│  Perfect separation of concerns! 🎉                                     │
└──────────────────────────────────────────────────────────────────────────┘
```

## 📡 Other Route Types — Beyond HTTP

Gateway API isn't limited to HTTP! It supports multiple protocols:

### GRPCRoute — For gRPC Services

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: GRPCRoute
metadata:
  name: grpc-route
spec:
  parentRefs:
    - name: my-gateway
      namespace: infra
  hostnames:
    - "grpc.myapp.com"
  rules:
    - matches:
        - method:
            service: myapp.UserService    # gRPC service name
            method: GetUser               # gRPC method name
      backendRefs:
        - name: user-grpc-service
          port: 50051
```

### TCPRoute — For Raw TCP Traffic

```yaml
apiVersion: gateway.networking.k8s.io/v1alpha2
kind: TCPRoute
metadata:
  name: tcp-route
spec:
  parentRefs:
    - name: my-gateway
      namespace: infra
      sectionName: tcp-listener
  rules:
    - backendRefs:
        - name: database-service
          port: 5432
```

### TLSRoute — For TLS Passthrough

```yaml
apiVersion: gateway.networking.k8s.io/v1alpha2
kind: TLSRoute
metadata:
  name: tls-passthrough
spec:
  parentRefs:
    - name: my-gateway
      namespace: infra
  hostnames:
    - "secure.myapp.com"
  rules:
    - backendRefs:
        - name: tls-backend
          port: 8443
```

## 🛠️ Installing Gateway API — Hands-On Setup

### Step 1: Install the Gateway API CRDs

The Gateway API resources don't come pre-installed. You need to install the CRDs first:

```bash
# Install the standard channel (stable APIs)
kubectl apply -f https://github.com/kubernetes-sigs/gateway-api/releases/download/v1.2.0/standard-install.yaml

# Or install the experimental channel (includes TCPRoute, UDPRoute, TLSRoute, etc.)
kubectl apply -f https://github.com/kubernetes-sigs/gateway-api/releases/download/v1.2.0/experimental-install.yaml

# Verify the CRDs are installed
kubectl get crds | grep gateway

# Output:
# gatewayclasses.gateway.networking.k8s.io     2024-01-15T10:00:00Z
# gateways.gateway.networking.k8s.io           2024-01-15T10:00:00Z
# httproutes.gateway.networking.k8s.io         2024-01-15T10:00:00Z
# grpcroutes.gateway.networking.k8s.io         2024-01-15T10:00:00Z
# referencegrants.gateway.networking.k8s.io    2024-01-15T10:00:00Z
```

### Step 2: Install a Gateway Controller

You need a controller that implements the Gateway API. Let's use **Envoy Gateway** as an example:

```bash
# Install Envoy Gateway using Helm
helm install eg oci://docker.io/envoyproxy/gateway-helm \
  --version v1.2.0 \
  -n envoy-gateway-system \
  --create-namespace

# Verify it's running
kubectl get pods -n envoy-gateway-system

# Output:
# NAME                                        READY   STATUS    RESTARTS   AGE
# envoy-gateway-5d89f7c6b4-xxxxx              1/1     Running   0          60s

# Check the GatewayClass created by Envoy Gateway
kubectl get gatewayclass

# Output:
# NAME    CONTROLLER                                      ACCEPTED   AGE
# eg      gateway.envoyproxy.io/gatewayclass-controller    True       60s
```

### Step 3: Create a Gateway

```yaml
# my-gateway.yaml
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: my-gateway
  namespace: default
spec:
  gatewayClassName: eg        # ← Use the Envoy Gateway class
  listeners:
    - name: http
      protocol: HTTP
      port: 80
      allowedRoutes:
        namespaces:
          from: Same          # ← Only routes from same namespace
```

```bash
kubectl apply -f my-gateway.yaml

# Check the Gateway
kubectl get gateway my-gateway

# Output:
# NAME         CLASS   ADDRESS          PROGRAMMED   AGE
# my-gateway   eg      172.18.255.200   True         30s
```

### Step 4: Deploy a Sample App and Create an HTTPRoute

```yaml
# sample-app.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: hello-app
spec:
  replicas: 2
  selector:
    matchLabels:
      app: hello
  template:
    metadata:
      labels:
        app: hello
    spec:
      containers:
        - name: hello
          image: hashicorp/http-echo
          args:
            - "-text=Hello from Gateway API! 🎉"
          ports:
            - containerPort: 5678
---
apiVersion: v1
kind: Service
metadata:
  name: hello-service
spec:
  selector:
    app: hello
  ports:
    - port: 8080
      targetPort: 5678
---
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: hello-route
spec:
  parentRefs:
    - name: my-gateway
  rules:
    - matches:
        - path:
            type: PathPrefix
            value: /hello
      backendRefs:
        - name: hello-service
          port: 8080
```

```bash
kubectl apply -f sample-app.yaml

# Get the Gateway's external IP
GATEWAY_IP=$(kubectl get gateway my-gateway -o jsonpath='{.status.addresses[0].value}')

# Test the route
curl http://$GATEWAY_IP/hello
# Hello from Gateway API! 🎉
```

## 🔒 ReferenceGrant — Secure Cross-Namespace References

When an HTTPRoute in one namespace wants to route traffic to a Service in another namespace, you need a **ReferenceGrant**:

```yaml
# This ReferenceGrant lives in the "backend" namespace
# It says: "HTTPRoutes from the 'frontend' namespace are allowed
#           to reference Services in MY namespace"
apiVersion: gateway.networking.k8s.io/v1beta1
kind: ReferenceGrant
metadata:
  name: allow-frontend-to-backend
  namespace: backend           # ← This is in the TARGET namespace
spec:
  from:
    - group: gateway.networking.k8s.io
      kind: HTTPRoute
      namespace: frontend      # ← Allow references FROM this namespace
  to:
    - group: ""
      kind: Service            # ← Allow references TO Services
```

```
┌──────────────────────────────────────────────────────────────────────────┐
│  ReferenceGrant — Think of it like a permission slip:                     │
│                                                                          │
│  "I (the backend namespace) grant permission for                         │
│   HTTPRoutes from the frontend namespace                                 │
│   to send traffic to Services in my namespace."                          │
│                                                                          │
│  Without this, cross-namespace routing is BLOCKED! ❌                    │
│  This is a security feature — you opt IN to cross-namespace access.     │
│                                                                          │
│  Namespace: frontend          Namespace: backend                         │
│  ┌────────────────┐           ┌──────────────────────────┐              │
│  │ HTTPRoute      │           │ ReferenceGrant            │              │
│  │ backendRefs:   │  allows   │ from: frontend/HTTPRoute  │              │
│  │ - name: api    │ ←──────── │ to: Service               │              │
│  │   namespace:   │           │                            │              │
│  │     backend    │           │ Service: api               │              │
│  └────────────────┘           └──────────────────────────┘              │
└──────────────────────────────────────────────────────────────────────────┘
```

┌──────────────────────────────────────────────────────────────────────────┐
│  Feature                    Ingress              Gateway API             │
│  ──────────                 ──────────           ────────────            │
│  Path-based routing         ✅ Yes               ✅ Yes                  │
│  Host-based routing         ✅ Yes               ✅ Yes                  │
│  Header-based routing       ❌ Annotations only  ✅ Native               │
│  Query param routing        ❌ No                ✅ Yes                  │
│  Traffic splitting          ❌ Annotations only  ✅ Native (weights)     │
│  Request redirect           ❌ Annotations only  ✅ Native               │
│  URL rewriting              ❌ Annotations only  ✅ Native               │
│  Header modification        ❌ Annotations only  ✅ Native               │
│  Cross-namespace routing    ❌ No                ✅ Yes (ReferenceGrant) │
│  gRPC routing               ❌ No                ✅ GRPCRoute            │
│  TCP/UDP routing            ❌ No                ✅ TCPRoute/UDPRoute    │

## 📋 Complete Real-World Example

Here's a full production-like setup:

```yaml
# 1. GatewayClass (created by platform team, once)
apiVersion: gateway.networking.k8s.io/v1
kind: GatewayClass
metadata:
  name: production
spec:
  controllerName: gateway.envoyproxy.io/gatewayclass-controller
---
# 2. Gateway (created by network team)
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: production-gateway
  namespace: infra
spec:
  gatewayClassName: production
  listeners:
    - name: http
      protocol: HTTP
      port: 80
      allowedRoutes:
        namespaces:
          from: All
    - name: https
      protocol: HTTPS
      port: 443
      hostname: "*.example.com"
      tls:
        mode: Terminate
        certificateRefs:
          - name: wildcard-example-com
      allowedRoutes:
        namespaces:
          from: All
---
# 3. HTTPRoute for HTTP→HTTPS redirect
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: http-to-https-redirect
  namespace: infra
spec:
  parentRefs:
    - name: production-gateway
      sectionName: http
  rules:
    - filters:
        - type: RequestRedirect
          requestRedirect:
            scheme: https
            statusCode: 301
---
# 4. HTTPRoute for API (created by backend team)
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: api-routes
  namespace: backend
spec:
  parentRefs:
    - name: production-gateway
      namespace: infra
      sectionName: https
  hostnames:
    - "api.example.com"
  rules:
    # v2 canary: 10% of traffic
    - matches:
        - path:
            type: PathPrefix
            value: /api
      backendRefs:
        - name: api-v1
          port: 8080
          weight: 90
        - name: api-v2
          port: 8080
          weight: 10
---
# 5. HTTPRoute for web frontend (created by frontend team)
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: web-routes
  namespace: frontend
spec:
  parentRefs:
    - name: production-gateway
      namespace: infra
      sectionName: https
  hostnames:
    - "www.example.com"
  rules:
    - matches:
        - path:
            type: PathPrefix
            value: /
      backendRefs:
        - name: web-app
          port: 3000
```

## 🧠 Quick Summary

| Concept | What It Does |
|---------|-------------|
| **GatewayClass** | Defines which controller/implementation to use (like StorageClass for networking) |
| **Gateway** | Creates the actual load balancer with listeners (ports, TLS, hostname filters) |
| **HTTPRoute** | Defines HTTP routing rules (path, header, query param matching) |
| **GRPCRoute** | Routing for gRPC traffic |
| **TCPRoute / TLSRoute** | Routing for raw TCP and TLS passthrough (alpha) |
| **ReferenceGrant** | Permission to reference resources across namespaces |
| **Traffic Splitting** | Send weighted % of traffic to different backends (canary!) |
| **Filters** | Modify requests/responses (headers, redirects, URL rewrites) |
| **parentRefs** | How a Route attaches to a Gateway |
| **sectionName** | Target a specific listener on a Gateway |

## 🎯 Key Takeaways

1. **Gateway API is the future** of Kubernetes traffic management — Ingress won't get new features
2. **Three personas**: Infrastructure Provider (GatewayClass) → Cluster Operator (Gateway) → Developer (Routes)
3. **No more annotation hell** — features like header routing, traffic splitting, and URL rewriting are built-in
4. **Cross-namespace routing** is first-class with ReferenceGrant for security
5. **HTTPRoute and GRPCRoute are GA** (production-ready), TCP/UDP routes are still alpha
6. **Install CRDs first**, then a controller (Envoy Gateway, Istio, Nginx, Cilium, etc.)
7. **Traffic splitting with weights** makes canary deployments trivial
8. **Gateway API is portable** — switch from Envoy to Nginx without rewriting your routes!
