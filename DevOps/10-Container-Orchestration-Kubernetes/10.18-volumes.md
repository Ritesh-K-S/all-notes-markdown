# 10.18 — Volumes — emptyDir, hostPath, PersistentVolume (PV)

> **One Line Answer:** Kubernetes Volumes are the way to give your Pods **persistent or shared storage** — because by default, when a container dies, all its data dies with it. Kubernetes offers different volume types: **emptyDir** (temporary shared storage that lives as long as the Pod), **hostPath** (mounts a directory from the node's filesystem), and **PersistentVolume (PV)** (durable storage that survives Pod restarts and can be backed by cloud disks like EBS, Azure Disk, or NFS).

---

## 🤔 Why Do You Need Volumes? — The Ephemeral Problem

Let's understand the core problem first:

```
┌──────────────────────────────────────────────────────────────────────────┐
│  THE PROBLEM: Containers are EPHEMERAL (short-lived)                     │
│                                                                          │
│  Step 1: Your app writes data inside the container                       │
│  ┌───────────────────────────────┐                                      │
│  │  Container (nginx)            │                                      │
│  │                               │                                      │
│  │  /app/data/uploads/           │                                      │
│  │    photo1.jpg  ✅             │                                      │
│  │    photo2.jpg  ✅             │                                      │
│  │    photo3.jpg  ✅             │                                      │
│  │                               │                                      │
│  │  Everything looks great!      │                                      │
│  └───────────────────────────────┘                                      │
│                                                                          │
│  Step 2: Container crashes or gets restarted                             │
│  ┌───────────────────────────────┐                                      │
│  │  Container (nginx) — RESTART  │                                      │
│  │                               │                                      │
│  │  /app/data/uploads/           │                                      │
│  │    (empty)  😱                │                                      │
│  │                               │                                      │
│  │  ALL DATA IS GONE! 💀        │                                      │
│  └───────────────────────────────┘                                      │
│                                                                          │
│  This is because containers use a writable layer that's                  │
│  DESTROYED when the container is removed.                                │
│                                                                          │
│  SOLUTION: Volumes! 🎉                                                  │
│  Store data OUTSIDE the container so it survives restarts.               │
└──────────────────────────────────────────────────────────────────────────┘
```

## 1️⃣ emptyDir — Temporary Scratch Space

### What is emptyDir?

An `emptyDir` volume is created when a Pod is assigned to a node, and it **starts empty** (hence the name). It exists as long as the Pod is running on that node. When the Pod is **deleted** (for any reason), the `emptyDir` data is **permanently deleted**.

```
┌──────────────────────────────────────────────────────────────────────────┐
│  emptyDir Lifecycle:                                                      │
│                                                                          │
│  Pod Created → emptyDir Created (empty) → Containers Write Data         │
│       │                                         │                        │
│       │         Pod is Running                   │                        │
│       │    Data is shared between containers     │                        │
│       │                                         │                        │
│  Pod Deleted → emptyDir DESTROYED → Data GONE! 💀                       │
│                                                                          │
│  Key Points:                                                             │
│  ✅ Shared between ALL containers in the SAME Pod                       │
│  ✅ Survives container RESTARTS (within the same Pod)                    │
│  ❌ Does NOT survive Pod DELETION                                        │
│  ❌ Does NOT survive Pod RESCHEDULING (moving to another node)           │
│  📍 Stored on the node's disk (or in memory with medium: Memory)        │
└──────────────────────────────────────────────────────────────────────────┘
```

### When to Use emptyDir?

| Use Case | Example |
|----------|---------|
| **Scratch space** | Sorting/processing temporary data |
| **Cache** | Local cache that can be regenerated |
| **Sharing data between containers** | Sidecar pattern — one container writes logs, another reads them |
| **Checkpointing** | Long computation that needs to survive a crash |

### Basic emptyDir Example

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: emptydir-demo
spec:
  containers:
    - name: writer
      image: busybox
      command: ["sh", "-c", "echo 'Hello from writer!' > /data/message.txt; sleep 3600"]
      volumeMounts:
        - name: shared-data       # ← Mount the volume at /data
          mountPath: /data

    - name: reader
      image: busybox
      command: ["sh", "-c", "sleep 5; cat /data/message.txt; sleep 3600"]
      volumeMounts:
        - name: shared-data       # ← Same volume, mounted at /data
          mountPath: /data

  volumes:
    - name: shared-data           # ← Define the volume
      emptyDir: {}                # ← Type: emptyDir (starts empty)
```

Let's see what happens step by step:

```
┌──────────────────────────────────────────────────────────────────────────┐
│  Pod: emptydir-demo                                                      │
│                                                                          │
│  ┌─────────────────┐    shared-data     ┌─────────────────┐            │
│  │  writer          │    (emptyDir)      │  reader          │            │
│  │                  │   ┌──────────┐     │                  │            │
│  │  Writes to       │───│ /data/   │─────│  Reads from      │            │
│  │  /data/message   │   │          │     │  /data/message   │            │
│  │  .txt            │   │ message  │     │  .txt            │            │
│  │                  │   │ .txt     │     │                  │            │
│  └─────────────────┘   └──────────┘     └─────────────────┘            │
│                                                                          │
│  Both containers see the SAME files in /data/                           │
│  Writer creates a file → Reader can read it immediately! ✅             │
└──────────────────────────────────────────────────────────────────────────┘
```

```bash
# Apply and test
kubectl apply -f emptydir-demo.yaml

# Check what the reader sees:
kubectl logs emptydir-demo -c reader
# Hello from writer!

# The writer and reader are sharing data through the emptyDir volume! 🎉
```

### emptyDir with Memory (tmpfs)

You can store the emptyDir in **RAM** instead of disk — it's super fast but limited by available memory:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: emptydir-memory
spec:
  containers:
    - name: app
      image: busybox
      command: ["sh", "-c", "sleep 3600"]
      volumeMounts:
        - name: cache
          mountPath: /cache
  volumes:
    - name: cache
      emptyDir:
        medium: Memory          # ← Store in RAM! Super fast! 🚀
        sizeLimit: 256Mi        # ← Limit: max 256MB of RAM
```

```
┌──────────────────────────────────────────────────────────────────────────┐
│  emptyDir: medium comparison                                              │
│                                                                          │
│  medium: "" (default)          medium: Memory                            │
│  ─────────────────────         ────────────────────                      │
│  Stored on node's DISK         Stored in RAM (tmpfs)                     │
│  Slower (disk I/O)             MUCH faster (memory speed)                │
│  Survives node reboot?  ❌     Survives node reboot?  ❌                │
│  Size limit?  Node's disk      Size limit?  Pod's memory limit          │
│  Use: general scratch          Use: high-performance cache               │
│                                                                          │
│  ⚠️ Memory emptyDir counts against the container's memory limit!        │
│  If the emptyDir exceeds sizeLimit, the Pod is EVICTED.                 │
└──────────────────────────────────────────────────────────────────────────┘
```

### Real-World emptyDir: Sidecar Log Processing

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-with-log-sidecar
spec:
  containers:
    # Main application — writes logs to a file
    - name: app
      image: nginx
      volumeMounts:
        - name: log-volume
          mountPath: /var/log/nginx

    # Sidecar — reads logs and forwards them (e.g., to Elasticsearch)
    - name: log-forwarder
      image: busybox
      command: ["sh", "-c", "tail -f /logs/access.log"]
      volumeMounts:
        - name: log-volume
          mountPath: /logs       # ← Different mountPath, SAME volume!

  volumes:
    - name: log-volume
      emptyDir: {}
```

```
┌──────────────────────────────────────────────────────────────────────────┐
│  Sidecar Pattern with emptyDir:                                          │
│                                                                          │
│  ┌──────────────┐    log-volume      ┌──────────────────┐              │
│  │  nginx        │    (emptyDir)      │  log-forwarder    │              │
│  │               │   ┌────────────┐   │                   │              │
│  │  Writes to    │──▶│ access.log │──▶│  Reads logs and   │              │
│  │  /var/log/    │   │ error.log  │   │  forwards to      │              │
│  │  nginx/       │   └────────────┘   │  Elasticsearch    │              │
│  └──────────────┘                     └──────────────────┘              │
│                                                                          │
│  This is one of the most common patterns in Kubernetes!                  │
└──────────────────────────────────────────────────────────────────────────┘
```

## 2️⃣ hostPath — Mount the Node's Filesystem

### What is hostPath?

A `hostPath` volume mounts a **file or directory from the host node's filesystem** into a Pod. The data lives on the actual node, so it persists even after the Pod is deleted — but only on THAT specific node.

```
┌──────────────────────────────────────────────────────────────────────────┐
│  hostPath — What It Does:                                                │
│                                                                          │
│  Node (Worker Machine)                                                   │
│  ┌────────────────────────────────────────────────────────────┐         │
│  │  Node's Filesystem                                         │         │
│  │                                                            │         │
│  │  /var/data/my-app/   ← This directory on the node         │         │
│  │    file1.txt                                               │         │
│  │    file2.txt                                               │         │
│  │           │                                                │         │
│  │           │ hostPath mount                                 │         │
│  │           ▼                                                │         │
│  │  ┌─────────────────────────────────────┐                  │         │
│  │  │  Pod                                │                  │         │
│  │  │  ┌─────────────────────────────┐    │                  │         │
│  │  │  │  Container                  │    │                  │         │
│  │  │  │  /data/ ← mounted here     │    │                  │         │
│  │  │  │    file1.txt                │    │                  │         │
│  │  │  │    file2.txt                │    │                  │         │
│  │  │  └─────────────────────────────┘    │                  │         │
│  │  └─────────────────────────────────────┘                  │         │
│  └────────────────────────────────────────────────────────────┘         │
│                                                                          │
│  ⚠️ If Pod moves to a DIFFERENT node, it can't see the data!           │
│  The data is stuck on Node 1 — it doesn't follow the Pod.              │
└──────────────────────────────────────────────────────────────────────────┘
```

### When to Use hostPath?

```
┌──────────────────────────────────────────────────────────────────────────┐
│  ✅ Good Use Cases:                                                      │
│  • Access node-level system files (/var/log, /sys, /proc)               │
│  • DaemonSets that need node filesystem access (monitoring agents)      │
│  • Single-node development clusters (minikube, kind)                    │
│  • Docker socket mounting (/var/run/docker.sock)                        │
│                                                                          │
│  ❌ BAD Use Cases (Don't do this!):                                     │
│  • Storing application data in production                                │
│  • Databases                                                             │
│  • Anything that needs to survive Pod rescheduling                       │
│  • Multi-node clusters (Pod might move to a different node!)            │
│                                                                          │
│  ⚠️ SECURITY WARNING:                                                   │
│  hostPath gives the Pod access to the NODE's filesystem!                │
│  A malicious Pod could read sensitive files or even modify the host!    │
│  Most production clusters RESTRICT or BLOCK hostPath volumes.           │
└──────────────────────────────────────────────────────────────────────────┘
```

### Basic hostPath Example

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: hostpath-demo
spec:
  containers:
    - name: app
      image: busybox
      command: ["sh", "-c", "echo 'Data on the node!' > /data/test.txt; sleep 3600"]
      volumeMounts:
        - name: node-data
          mountPath: /data          # ← Inside the container
  volumes:
    - name: node-data
      hostPath:
        path: /tmp/k8s-demo        # ← Path on the HOST node
        type: DirectoryOrCreate     # ← Create directory if it doesn't exist
```

### hostPath Types

```yaml
# The "type" field controls validation behavior:

volumes:
  - name: my-volume
    hostPath:
      path: /some/path
      type: DirectoryOrCreate    # ← One of these types
```

```
┌──────────────────────────────────────────────────────────────────────────┐
│  hostPath Type             What It Does                                  │
│  ──────────────            ──────────────────────────────────────        │
│                                                                          │
│  "" (empty string)         No validation. Default. Anything goes.       │
│                                                                          │
│  DirectoryOrCreate         If path doesn't exist, create an empty       │
│                            directory (permissions: 0755, same owner     │
│                            as kubelet)                                   │
│                                                                          │
│  Directory                 Path MUST already exist as a directory.       │
│                            Pod fails to start if it doesn't exist.      │
│                                                                          │
│  FileOrCreate              If file doesn't exist, create an empty       │
│                            file (permissions: 0644)                     │
│                                                                          │
│  File                      Path MUST already exist as a file.           │
│                            Pod fails to start if it doesn't exist.      │
│                                                                          │
│  Socket                    Path MUST exist as a Unix socket.            │
│                                                                          │
│  CharDevice                Path MUST exist as a character device.       │
│                                                                          │
│  BlockDevice               Path MUST exist as a block device.           │
└──────────────────────────────────────────────────────────────────────────┘
```

### Real-World hostPath: Accessing Docker Socket

```yaml
# Common pattern: CI/CD tool that needs to build Docker images
apiVersion: v1
kind: Pod
metadata:
  name: docker-builder
spec:
  containers:
    - name: builder
      image: docker:24-cli
      command: ["sleep", "3600"]
      volumeMounts:
        - name: docker-socket
          mountPath: /var/run/docker.sock     # ← Mount Docker socket
  volumes:
    - name: docker-socket
      hostPath:
        path: /var/run/docker.sock            # ← Node's Docker socket
        type: Socket
```

### Real-World hostPath: DaemonSet for Log Collection

```yaml
# Collect logs from ALL nodes using a DaemonSet
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: log-collector
spec:
  selector:
    matchLabels:
      app: log-collector
  template:
    metadata:
      labels:
        app: log-collector
    spec:
      containers:
        - name: fluentd
          image: fluentd:latest
          volumeMounts:
            # Mount the node's /var/log directory
            - name: varlog
              mountPath: /var/log
              readOnly: true          # ← Read-only for safety!
            # Mount container logs
            - name: containers-log
              mountPath: /var/lib/docker/containers
              readOnly: true
      volumes:
        - name: varlog
          hostPath:
            path: /var/log
            type: Directory
        - name: containers-log
          hostPath:
            path: /var/lib/docker/containers
            type: Directory
```

## 3️⃣ PersistentVolume (PV) — True Persistent Storage

### What is a PersistentVolume?

A PersistentVolume (PV) is a piece of storage in the cluster that has been **provisioned by an administrator** or **dynamically provisioned** using a StorageClass. It's a cluster-level resource (not namespaced) that **lives independently of any Pod**.

```
┌──────────────────────────────────────────────────────────────────────────┐
│  The Relationship: PV → PVC → Pod                                        │
│                                                                          │
│  Think of it like renting a storage unit:                                │
│                                                                          │
│  PersistentVolume (PV)         = The actual storage unit                │
│                                  (exists in the storage facility)       │
│                                                                          │
│  PersistentVolumeClaim (PVC)   = Your rental agreement                  │
│                                  ("I need 10GB of fast storage")        │
│                                                                          │
│  Pod                           = You (the person using the storage)     │
│                                                                          │
│  ┌────────────────┐    binds to    ┌────────────┐    used by    ┌─────┐│
│  │       PV       │ ◄──────────── │    PVC     │ ◄─────────── │ Pod ││
│  │                │               │            │              │     ││
│  │ 50GB EBS Disk  │               │ "I need    │              │ app ││
│  │ AWS us-east-1  │               │  10GB,     │              │     ││
│  │                │               │  ReadWrite │              │     ││
│  └────────────────┘               └────────────┘              └─────┘│
│                                                                          │
│  Why this separation?                                                    │
│  • Admins manage storage (PVs) — they know the infrastructure           │
│  • Developers request storage (PVCs) — they just say "I need 10GB"     │
│  • Clean separation of concerns!                                        │
└──────────────────────────────────────────────────────────────────────────┘
```

### PV Lifecycle — The Full Journey

```
┌──────────────────────────────────────────────────────────────────────────┐
│  PV Lifecycle:                                                            │
│                                                                          │
│  1. PROVISIONING                                                         │
│     Admin creates PV (static) OR StorageClass creates it (dynamic)      │
│     Status: Available                                                    │
│                                                                          │
│  2. BINDING                                                              │
│     A PVC claims the PV (size and access mode must match)               │
│     Status: Bound                                                        │
│                                                                          │
│  3. USING                                                                │
│     Pod mounts the PVC and reads/writes data                            │
│     Status: Bound (in use)                                               │
│                                                                          │
│  4. RELEASING                                                            │
│     Pod is deleted, PVC is deleted                                       │
│     Status: Released                                                     │
│                                                                          │
│  5. RECLAIMING                                                           │
│     What happens to the data?                                            │
│     • Retain  → Data kept, PV must be manually cleaned up               │
│     • Delete  → PV and underlying storage are DELETED                   │
│     • Recycle → Data wiped (rm -rf), PV made Available again            │
│                 (⚠️ Deprecated! Use dynamic provisioning instead)       │
│                                                                          │
│  Available ──▶ Bound ──▶ Released ──▶ (Retain/Delete/Recycle)          │
│                                                                          │
└──────────────────────────────────────────────────────────────────────────┘
```

### Creating a PersistentVolume (Static Provisioning)

```yaml
# pv.yaml — Admin creates this
apiVersion: v1
kind: PersistentVolume
metadata:
  name: my-pv                    # ← PVs are NOT namespaced (cluster-wide)
spec:
  capacity:
    storage: 10Gi                # ← Size of the volume (10 GigaBytes)

  accessModes:
    - ReadWriteOnce              # ← How it can be accessed (see table below)

  persistentVolumeReclaimPolicy: Retain   # ← What happens when released

  storageClassName: manual       # ← Optional: group PVs into classes

  # The actual storage backend — this is where the data physically lives
  # Option A: NFS
  nfs:
    server: 192.168.1.100
    path: /exports/data

  # Option B: AWS EBS (uncomment to use)
  # awsElasticBlockStore:
  #   volumeID: vol-0123456789abcdef
  #   fsType: ext4

  # Option C: Host path (for testing only!)
  # hostPath:
  #   path: /mnt/data
```

```bash
# Apply the PV
kubectl apply -f pv.yaml

# Check PV status
kubectl get pv

# Output:
# NAME    CAPACITY   ACCESS MODES   RECLAIM POLICY   STATUS      CLAIM   STORAGECLASS   AGE
# my-pv   10Gi       RWO            Retain           Available           manual         10s
#                                                    ↑
#                                              Not claimed yet!
```

### Access Modes Explained

```
┌──────────────────────────────────────────────────────────────────────────┐
│  Access Modes — How Pods Can Use the Volume                              │
│                                                                          │
│  Mode                Short     What It Means                            │
│  ──────────────      ─────     ─────────────────────────────────────    │
│                                                                          │
│  ReadWriteOnce       RWO       Volume can be mounted as read-write      │
│                                by a SINGLE node.                        │
│                                Multiple Pods on the SAME node can       │
│                                use it.                                   │
│                                Most common for cloud disks (EBS,        │
│                                Azure Disk, GCE PD).                     │
│                                                                          │
│  ReadOnlyMany        ROX       Volume can be mounted as read-only       │
│                                by MANY nodes simultaneously.            │
│                                Good for shared config or static data.   │
│                                                                          │
│  ReadWriteMany       RWX       Volume can be mounted as read-write      │
│                                by MANY nodes simultaneously.            │
│                                Only supported by certain backends:      │
│                                NFS, CephFS, Azure Files, EFS.           │
│                                                                          │
│  ReadWriteOncePod    RWOP      Volume can be mounted as read-write      │
│                                by a SINGLE Pod (stricter than RWO).     │
│                                K8s 1.27+ only.                          │
│                                                                          │
│  ⚠️ Not all storage backends support all access modes!                  │
│  Cloud block storage (EBS, Azure Disk) usually only supports RWO.      │
│  For RWX, you need NFS or a shared filesystem.                          │
└──────────────────────────────────────────────────────────────────────────┘
```

### Creating a PersistentVolumeClaim (PVC)

A PVC is how a **developer requests storage**. Kubernetes will find a matching PV and bind them together:

```yaml
# pvc.yaml — Developer creates this
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-pvc
  namespace: default             # ← PVCs ARE namespaced!
spec:
  accessModes:
    - ReadWriteOnce              # ← Must match (or be a subset of) the PV

  resources:
    requests:
      storage: 5Gi               # ← "I need at least 5GB"
                                  # K8s will find a PV with >= 5GB

  storageClassName: manual       # ← Must match the PV's storageClassName
```

```bash
# Apply the PVC
kubectl apply -f pvc.yaml

# Check PVC status
kubectl get pvc

# Output:
# NAME     STATUS   VOLUME   CAPACITY   ACCESS MODES   STORAGECLASS   AGE
# my-pvc   Bound    my-pv    10Gi       RWO            manual         5s
#          ↑                 ↑
#          Bound!            Got the 10Gi PV (we asked for 5Gi, got 10Gi)

# Check PV status — it's now Bound!
kubectl get pv

# Output:
```

## 🔄 Dynamic Provisioning with StorageClass

In production, you rarely create PVs manually. Instead, you use **StorageClass** to dynamically create PVs when a PVC is created:

```yaml
# storageclass.yaml — Admin creates this once
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-ssd
provisioner: kubernetes.io/aws-ebs    # ← Which provisioner to use
parameters:
  type: gp3                            # ← EBS volume type
  fsType: ext4                         # ← Filesystem type
reclaimPolicy: Delete                  # ← Delete PV when PVC is deleted
volumeBindingMode: WaitForFirstConsumer # ← Create PV only when a Pod needs it
allowVolumeExpansion: true             # ← Allow resizing later
```

```yaml
# pvc-dynamic.yaml — Developer creates this (NO PV needed!)
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: dynamic-pvc
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 20Gi
  storageClassName: fast-ssd          # ← Use the "fast-ssd" StorageClass
```

```
┌──────────────────────────────────────────────────────────────────────────┐
│  Dynamic Provisioning Flow:                                               │
│                                                                          │
│  1. Dev creates PVC with storageClassName: fast-ssd                     │
│                                                                          │
│  2. Kubernetes sees: "No matching PV exists, but there's a              │
│     StorageClass called 'fast-ssd' — let me create a PV!"              │
│                                                                          │
│  3. StorageClass tells the provisioner (e.g., AWS EBS):                 │
│     "Create a 20Gi gp3 EBS volume!"                                    │
│                                                                          │
│  4. EBS volume is created in AWS                                         │
│                                                                          │
│  5. A PV is automatically created and bound to the PVC                  │
│                                                                          │
│  6. Pod can now use the PVC!                                             │
│                                                                          │
│  Static (Manual):                 Dynamic (Automatic):                  │
│  Admin creates PV ──▶ PVC binds   PVC created ──▶ PV auto-created!     │
│  (tedious, manual)                (magic! ✨)                           │
│                                                                          │
└──────────────────────────────────────────────────────────────────────────┘
```

```bash
# Check what StorageClasses are available in your cluster
kubectl get storageclass

# Output (example on EKS):
# NAME            PROVISIONER             RECLAIMPOLICY   VOLUMEBINDINGMODE      AGE
# gp2 (default)   kubernetes.io/aws-ebs   Delete          WaitForFirstConsumer   30d
# gp3             ebs.csi.aws.com         Delete          WaitForFirstConsumer   30d
```

## 📊 Reclaim Policies — What Happens When You Delete a PVC

```
┌──────────────────────────────────────────────────────────────────────────┐
│  Reclaim Policies:                                                        │
│                                                                          │
│  Policy      What Happens When PVC is Deleted        Use Case            │
│  ──────      ───────────────────────────────────     ────────────        │
│                                                                          │
│  Retain      PV is NOT deleted. Data is preserved.   Production data.   │
│              PV status becomes "Released".            Manual cleanup.    │
│              Admin must manually delete the PV                          │
│              and the underlying storage.                                 │
│              ✅ Safest — you never lose data!                           │
│                                                                          │
│  Delete      PV AND the underlying storage are        Dev/test envs.    │
│              automatically deleted. Data is GONE!     Ephemeral data.   │
│              ⚠️ Be careful — no recovery possible!                     │
│                                                                          │
│  Recycle     Data is wiped (rm -rf /volume/*),        DEPRECATED.       │
│              then PV is made Available again.         Don't use.        │
│              ⚠️ DEPRECATED — use dynamic provisioning                   │
│              instead.                                                    │
│                                                                          │
│  Recommendation:                                                         │
│  • Production databases → Retain                                        │
│  • Dynamic provisioning → Delete (PV auto-created, auto-destroyed)     │
│  • Never use Recycle                                                     │
└──────────────────────────────────────────────────────────────────────────┘
```

## ⚠️ Common Mistakes & Gotchas

### Mistake 1: Using emptyDir for Important Data

```yaml
# ❌ WRONG — Data will be lost when Pod is deleted!
volumes:
  - name: db-data
    emptyDir: {}

# ✅ CORRECT — Use PVC for important data
volumes:
  - name: db-data
    persistentVolumeClaim:
      claimName: db-pvc
```

### Mistake 2: Using hostPath in Multi-Node Clusters

```
❌ Problem:
  Pod on Node-1 writes data to /data/app on Node-1
  Pod gets rescheduled to Node-2
  /data/app on Node-2 is EMPTY — data is on Node-1!

✅ Solution:
  Use PersistentVolumes with network storage (NFS, EBS, etc.)
  that is accessible from ANY node.
```

### Mistake 3: PVC Stuck in Pending State

```bash
kubectl get pvc
# NAME     STATUS    VOLUME   CAPACITY   ACCESS MODES   STORAGECLASS   AGE
# my-pvc   Pending                                      fast-ssd       5m

# Common causes:
# 1. No PV matches the PVC requirements
kubectl get pv   # Check if any PV has the right size/access/storageClass

# 2. StorageClass doesn't exist
kubectl get storageclass   # Verify the StorageClass name

# 3. volumeBindingMode: WaitForFirstConsumer
# PV won't be created until a Pod actually uses the PVC!
# Create a Pod that mounts the PVC and it will work.

# 4. Provisioner not installed (for dynamic provisioning)
# E.g., EBS CSI driver not installed on EKS
```

### Mistake 4: Forgetting ReadOnly When Needed

```yaml
# When multiple Pods need to read the SAME data:
volumeMounts:
  - name: shared-config
    mountPath: /config
    readOnly: true          # ← Prevents accidental writes!
```

## 🧠 Quick Summary Table

| Volume Type | Lifetime | Shared Between Containers | Shared Between Pods | Survives Pod Delete | Production Use |
|-------------|----------|--------------------------|--------------------|--------------------|---------------|
| **emptyDir** | Pod lifetime | ✅ Same Pod only | ❌ No | ❌ No | Scratch/cache |
| **emptyDir (Memory)** | Pod lifetime | ✅ Same Pod only | ❌ No | ❌ No | High-speed cache |
| **hostPath** | Node lifetime | ✅ Same Node | ⚠️ Same Node only | ✅ On same node | DaemonSets, dev only |
| **PV (static)** | Independent | ✅ Via PVC | ✅ Via PVC | ✅ Yes | Databases, uploads |
| **PV (dynamic)** | Independent | ✅ Via PVC | ✅ Via PVC | ✅ Yes | **Recommended!** |

## 🎯 Key Takeaways

1. **Containers are ephemeral** — without volumes, data dies with the container
2. **emptyDir** is temporary scratch space — perfect for sharing data between containers in the same Pod, but data is lost when the Pod dies
3. **hostPath** mounts the node's filesystem — useful for DaemonSets and debugging, but a **security risk** and not portable across nodes
4. **PersistentVolume (PV)** is the real deal — cluster-level storage that survives Pod restarts and rescheduling
5. **PersistentVolumeClaim (PVC)** is how developers request storage — they don't need to know the infrastructure details
6. **Dynamic Provisioning** (with StorageClass) is the production standard — PVs are created automatically when a PVC is created
7. **Access Modes matter** — RWO for single-node access, RWX for multi-node (needs NFS or shared filesystem)
8. **Reclaim Policies** — use `Retain` for important data, `Delete` for ephemeral/replaceable data
