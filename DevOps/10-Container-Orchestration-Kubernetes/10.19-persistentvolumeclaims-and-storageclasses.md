# 10.19 — PersistentVolumeClaims (PVC) & StorageClasses

> **One Line Answer:** A **PersistentVolumeClaim (PVC)** is a **request for storage** by a Pod — it's like saying "Hey Kubernetes, I need 10GB of fast storage please!" — and a **StorageClass** is the **blueprint** that tells Kubernetes HOW to automatically create that storage (what type, what speed, what provider) so you don't have to manually create PersistentVolumes every time.

---

## 🤔 Why Do We Need PVCs and StorageClasses?

In the previous section (10.18), we learned about PersistentVolumes (PV). But there's a big problem with the manual PV approach:

### The Problem: Manual PV Management Is Painful

```
The Manual Way (Without StorageClasses):

Step 1: Dev says "I need 10GB storage for my database"
   👨‍💻 Developer → "Hey admin, I need storage!"

Step 2: Admin manually creates a PersistentVolume
   👨‍💼 Admin → Goes to AWS → Creates an EBS volume → 
              Writes PV YAML → Applies it to cluster
   (This takes time... maybe hours or days!)

Step 3: Dev creates a PVC that matches the PV
   👨‍💻 Developer → Creates PVC → Hopes it binds to the right PV

Step 4: Repeat for EVERY single storage request
   👨‍💼 Admin → 😩 "Not another storage ticket!"

   ┌──────────────────────────────────────┐
   │  Problems:                           │
   │  ❌ Slow — manual process            │
   │  ❌ Error-prone — wrong sizes/types  │
   │  ❌ Doesn't scale — 100 PVs? No way │
   │  ❌ Admin becomes bottleneck         │
   └──────────────────────────────────────┘
```

### The Solution: StorageClasses + Dynamic Provisioning

```
The Automatic Way (With StorageClasses):

Step 1: Admin creates a StorageClass ONCE
   👨‍💼 Admin → "Here's a StorageClass called 'fast-ssd' 
              that auto-creates EBS gp3 volumes"

Step 2: Dev creates a PVC requesting storage
   👨‍💻 Developer → "I need 10GB of 'fast-ssd' storage"

Step 3: Kubernetes AUTOMATICALLY creates the PV!
   🤖 Kubernetes → Creates EBS volume on AWS →
                    Creates PV → Binds PV to PVC
   (This takes seconds! ⚡)

Step 4: Pod uses the PVC — done!
   🎉 No admin intervention needed!

   ┌──────────────────────────────────────┐
   │  Benefits:                           │
   │  ✅ Fast — seconds, not hours        │
   │  ✅ Automated — no human errors      │
   │  ✅ Scales — 1000 PVCs? No problem   │
   │  ✅ Self-service for developers      │
   └──────────────────────────────────────┘
```

## 📋 PersistentVolumeClaim (PVC) — Deep Dive

### What Exactly Is a PVC?

A PVC is a **request for storage** by a user (Pod). Think of it as filling out a form:

```
PVC = Storage Request Form
┌─────────────────────────────────────────┐
│  📋 Storage Request                      │
│                                          │
│  How much space?     ___10Gi___          │
│  Access mode?        ___ReadWriteOnce___ │
│  Storage class?      ___fast-ssd___      │
│  Any specific PV?    ___no, auto___      │
│                                          │
│  [Submit Request]                        │
└─────────────────────────────────────────┘
```

### Basic PVC YAML

```yaml
# my-pvc.yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-app-data        # Name of the claim
  namespace: default
spec:
  accessModes:
    - ReadWriteOnce         # One node can read/write at a time
  resources:
    requests:
      storage: 10Gi         # I need 10 GiB of storage
  storageClassName: fast-ssd # Use this StorageClass to provision storage
```

Let's break down every field:

```
apiVersion: v1                    ← PVC is a core resource (v1)

kind: PersistentVolumeClaim       ← The resource type

metadata:
  name: my-app-data               ← Name you'll reference in your Pod

spec:
  accessModes:                    ← HOW the volume can be accessed
    - ReadWriteOnce               ← Only one node at a time (most common)
    
  resources:
    requests:
      storage: 10Gi               ← HOW MUCH storage you need (minimum)
      
  storageClassName: fast-ssd      ← WHICH StorageClass to use
                                    (determines the type of storage)
```

### Access Modes Explained

```
Access Modes — Who can read/write to the volume?

┌────────────────────────────────────────────────────────────────────────┐
│                                                                        │
│  ReadWriteOnce (RWO)          — Most Common!                           │
│  ────────────────                                                      │
│  ✅ ONE node can mount it as read-write                                │
│  ✅ Multiple pods on the SAME node can use it                          │
│  ❌ Pods on DIFFERENT nodes cannot use it simultaneously               │
│  📌 Use for: Databases (MySQL, PostgreSQL), single-app storage         │
│                                                                        │
│  ┌────────┐                                                            │
│  │ Node 1 │──→ Pod A ──→ 📁 Volume (read/write) ✅                    │
│  │        │──→ Pod B ──→ 📁 Volume (read/write) ✅ (same node = OK)   │
│  └────────┘                                                            │
│  ┌────────┐                                                            │
│  │ Node 2 │──→ Pod C ──→ 📁 Volume (read/write) ❌ (different node!)  │
│  └────────┘                                                            │
│                                                                        │
├────────────────────────────────────────────────────────────────────────┤
│                                                                        │
│  ReadOnlyMany (ROX)                                                    │
│  ──────────────────                                                    │
│  ✅ MANY nodes can mount it as read-only                               │
│  ❌ Nobody can write to it                                             │
│  📌 Use for: Shared config files, static content, ML models           │
│                                                                        │
│  ┌────────┐                                                            │
│  │ Node 1 │──→ Pod A ──→ 📁 Volume (read-only) ✅                     │
│  └────────┘                                                            │
│  ┌────────┐                                                            │
│  │ Node 2 │──→ Pod B ──→ 📁 Volume (read-only) ✅                     │
│  └────────┘                                                            │
│  ┌────────┐                                                            │
│  │ Node 3 │──→ Pod C ──→ 📁 Volume (read-only) ✅                     │
│  └────────┘                                                            │
│                                                                        │
├────────────────────────────────────────────────────────────────────────┤
│                                                                        │
│  ReadWriteMany (RWX)                                                   │
│  ──────────────────                                                    │
│  ✅ MANY nodes can mount it as read-write                              │
│  📌 Use for: Shared file storage, CMS, file uploads                   │
│  ⚠️  Not all storage types support this (NFS, EFS, Azure Files do)    │
│  ⚠️  EBS, GCE PD do NOT support RWX!                                  │
│                                                                        │
│  ┌────────┐                                                            │
│  │ Node 1 │──→ Pod A ──→ 📁 Volume (read/write) ✅                    │
│  └────────┘                                                            │
│  ┌────────┐                                                            │
│  │ Node 2 │──→ Pod B ──→ 📁 Volume (read/write) ✅                    │
│  └────────┘                                                            │
│                                                                        │
├────────────────────────────────────────────────────────────────────────┤
│                                                                        │
│  ReadWriteOncePod (RWOP)       — Kubernetes 1.22+                      │
│  ───────────────────                                                   │
│  ✅ Only ONE pod in the entire cluster can mount it                    │
│  📌 Use for: When you need strict single-writer guarantee             │
│  📌 More restrictive than RWO (which allows multiple pods on same node)│
│                                                                        │
│  ┌────────┐                                                            │
│  │ Node 1 │──→ Pod A ──→ 📁 Volume (read/write) ✅                    │
│  │        │──→ Pod B ──→ 📁 Volume ❌ (only one pod allowed!)         │
│  └────────┘                                                            │
│                                                                        │
└────────────────────────────────────────────────────────────────────────┘
```

### PVC Lifecycle — What Happens When You Create a PVC

```
PVC Lifecycle:

                    create PVC
                        │
                        ▼
               ┌─────────────────┐
               │    Pending      │ ← Waiting for a PV to bind
               │                 │   (StorageClass is provisioning
               │  "Looking for   │    OR waiting for matching PV)
               │   matching PV"  │
               └────────┬────────┘
                        │   PV found or created
                        ▼
               ┌─────────────────┐
               │     Bound       │ ← PVC is now linked to a PV
               │                 │   Ready to be used by a Pod!
               │  "I got my      │
               │   storage!"     │
               └────────┬────────┘
                        │   PVC deleted
                        ▼
               ┌─────────────────┐
               │   Released      │ ← PV still exists, but PVC is gone
               │                 │   What happens to PV depends on
               │  "PVC is gone,  │   the Reclaim Policy
               │   what now?"    │
               └────────┬────────┘
                        │
              ┌─────────┴──────────┐
              ▼                    ▼
    ┌─────────────────┐  ┌─────────────────┐
    │  Retain          │  │  Delete          │
    │                  │  │                  │
    │  PV & data kept  │  │  PV & data       │
    │  Admin decides   │  │  automatically   │
    │  what to do      │  │  deleted          │
    └─────────────────┘  └─────────────────┘
```

## 🗂️ StorageClass — Deep Dive

### What Exactly Is a StorageClass?

A StorageClass is a **blueprint/template** for creating storage. It tells Kubernetes:

1. **Which provisioner** to use (AWS EBS? NFS? Azure Disk?)
2. **What parameters** to use (disk type, IOPS, encryption?)
3. **What reclaim policy** (delete or retain when PVC is deleted?)

Think of it like a menu at a restaurant:

```
StorageClass = Menu Options for Storage

┌───────────────────────────────────────────────┐
│  🍽️  Kubernetes Storage Menu                   │
│                                                │
│  ┌─────────────────────────────────────────┐   │
│  │  📌 "standard" (Default)                │   │
│  │  Type: HDD (gp2)                       │   │
│  │  Speed: Normal                          │   │
│  │  Price: $ (Cheap)                       │   │
│  │  Good for: Dev/test, logs, backups      │   │
│  └─────────────────────────────────────────┘   │
│                                                │
│  ┌─────────────────────────────────────────┐   │
│  │  ⚡ "fast-ssd"                           │   │
│  │  Type: SSD (gp3)                        │   │
│  │  Speed: Fast                            │   │
│  │  Price: $$ (Moderate)                   │   │
│  │  Good for: Databases, APIs              │   │
│  └─────────────────────────────────────────┘   │
│                                                │
│  ┌─────────────────────────────────────────┐   │
│  │  🚀 "ultra-fast"                         │   │
│  │  Type: Provisioned IOPS SSD (io2)       │   │
│  │  Speed: Ultra Fast                      │   │
│  │  Price: $$$ (Expensive)                 │   │
│  │  Good for: High-perf databases          │   │

│                                                │
│  ┌─────────────────────────────────────────┐   │
│  │  📂 "shared-files"                      │   │
│  │  Type: NFS / EFS                        │   │
│  │  Speed: Moderate                        │   │
│  │  Access: ReadWriteMany (multi-node)     │   │
│  │  Good for: Shared file storage, CMS     │   │
│  └─────────────────────────────────────────┘   │
│                                                │
└───────────────────────────────────────────────┘
```

### Basic StorageClass YAML

```yaml
# storageclass-fast-ssd.yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-ssd                    # Name that PVCs will reference
provisioner: ebs.csi.aws.com        # Which plugin creates the actual disk
parameters:
  type: gp3                         # AWS EBS disk type (gp3 = General Purpose SSD)
  fsType: ext4                      # File system type
  encrypted: "true"                 # Encrypt the volume
reclaimPolicy: Delete               # Delete the PV when PVC is deleted
volumeBindingMode: WaitForFirstConsumer  # Don't create volume until a Pod needs it
allowVolumeExpansion: true          # Allow resizing PVCs later
```

Let's break down every field:

```
apiVersion: storage.k8s.io/v1       ← StorageClass belongs to the storage API group

kind: StorageClass                   ← Resource type

metadata:
  name: fast-ssd                     ← Name that PVCs reference in storageClassName

provisioner: ebs.csi.aws.com        ← The CSI driver / plugin that creates actual storage
                                       Examples:
                                       - ebs.csi.aws.com          (AWS EBS)
                                       - disk.csi.azure.com       (Azure Disk)
                                       - pd.csi.storage.gke.io    (GCE PD)
                                       - nfs.csi.k8s.io           (NFS)
                                       - kubernetes.io/no-provisioner (manual, no auto)

parameters:                          ← Provisioner-specific settings
  type: gp3                          ← Disk type (gp2, gp3, io1, io2, st1, sc1 for AWS)
  fsType: ext4                       ← File system (ext4, xfs, etc.)
  encrypted: "true"                  ← Enable encryption (string, not boolean!)

reclaimPolicy: Delete                ← What happens when PVC is deleted
                                       - Delete: PV AND underlying storage deleted
                                       - Retain: PV and storage kept (manual cleanup)

volumeBindingMode: WaitForFirstConsumer
                                     ← WHEN to create the volume
                                       - Immediate: Create right when PVC is created
                                       - WaitForFirstConsumer: Wait until a Pod needs it
                                         (better for topology/zone awareness!)

allowVolumeExpansion: true           ← Can PVCs resize? (increase storage later)
```

### Reclaim Policy — What Happens When PVC Is Deleted?

```
Reclaim Policy Comparison:

┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  reclaimPolicy: Delete (Default for dynamic provisioning)    │
│  ────────────────────                                        │
│                                                              │
│  1. PVC is deleted                                           │
│  2. PV is automatically deleted                              │
│  3. Underlying storage (EBS, disk) is also deleted           │
│  4. ALL DATA IS GONE! ⚠️                                     │
│                                                              │
│  Good for: Dev/test environments, temporary data             │
│  ⚠️  Careful: Don't use for important production data!       │
│                                                              │
│  PVC deleted → PV deleted → EBS volume deleted → Data GONE   │
│                                                              │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  reclaimPolicy: Retain                                       │
│  ────────────────────                                        │
│                                                              │
│  1. PVC is deleted                                           │
│  2. PV still exists (status: Released)                       │
│  3. Underlying storage (EBS, disk) still exists              │
│  4. Data is SAFE ✅                                          │
│  5. Admin must manually clean up or re-bind                  │
│                                                              │
│  Good for: Production databases, critical data               │
│  📌 Requires manual intervention to reuse                    │
│                                                              │
│  PVC deleted → PV kept (Released) → EBS volume kept → Data   │
│                                                     SAFE     │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### Volume Binding Mode — When to Create the Volume?

```
volumeBindingMode Comparison:

┌────────────────────────────────────────────────────────────────────┐
│                                                                    │
│  Immediate (Default)                                               │
│  ─────────────────                                                 │
│                                                                    │
│  PVC Created → Volume created RIGHT NOW                            │
│                                                                    │
│  Problem:                                                          │
│  PVC creates an EBS volume in us-east-1a                           │
│  But the Pod gets scheduled to a node in us-east-1b                │
│  Result: Pod can't access the volume! 💥                           │
│                                                                    │
│  ┌─────────┐    ┌─────────┐                                        │
│  │ Node    │    │ EBS Vol │                                        │
│  │ Zone: b │    │ Zone: a │  ← Different zones! Can't connect!     │
│  └─────────┘    └─────────┘                                        │
│                                                                    │
├────────────────────────────────────────────────────────────────────┤
│                                                                    │
│  WaitForFirstConsumer (Recommended! ✅)                            │
│  ─────────────────────                                             │
│                                                                    │
│  PVC Created → Nothing happens yet (PVC stays Pending)             │
│  Pod Created → Scheduler picks Node → THEN volume is created       │
│                in the SAME zone as the node!                        │
│                                                                    │
│  ┌─────────┐    ┌─────────┐                                        │
│  │ Node    │    │ EBS Vol │                                        │
│  │ Zone: b │    │ Zone: b │  ← Same zone! Perfect! ✅              │
│  └─────────┘    └─────────┘                                        │
│                                                                    │
└────────────────────────────────────────────────────────────────────┘
```

## 🔧 Hands-On Examples

### Example 1: Complete Flow — StorageClass → PVC → Pod

**Step 1: Create a StorageClass**

```yaml
# storageclass.yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-ssd
provisioner: ebs.csi.aws.com        # AWS EBS CSI driver
parameters:
  type: gp3                          # General Purpose SSD v3
  fsType: ext4
  encrypted: "true"
reclaimPolicy: Delete
volumeBindingMode: WaitForFirstConsumer
allowVolumeExpansion: true
```

```bash
kubectl apply -f storageclass.yaml

# Verify
kubectl get storageclass
# NAME       PROVISIONER       RECLAIMPOLICY   VOLUMEBINDINGMODE       AGE
# fast-ssd   ebs.csi.aws.com   Delete          WaitForFirstConsumer    5s
```

**Step 2: Create a PVC requesting storage from that StorageClass**

```yaml
# pvc.yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: postgres-data
spec:
  accessModes:
    - ReadWriteOnce             # One node at a time
  resources:
    requests:
      storage: 20Gi             # I need 20 GiB
  storageClassName: fast-ssd    # Use our "fast-ssd" StorageClass
```

```bash
kubectl apply -f pvc.yaml

# Check PVC status
kubectl get pvc
# NAME            STATUS    VOLUME   CAPACITY   ACCESS MODES   STORAGECLASS   AGE
# postgres-data   Pending                                      fast-ssd       5s
#
# Status is "Pending" because we used WaitForFirstConsumer
# It will stay Pending until a Pod references it!
```

**Step 3: Create a Pod that uses the PVC**

```yaml
# postgres-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: postgres
spec:
  containers:
    - name: postgres
      image: postgres:16
      env:
        - name: POSTGRES_PASSWORD
          value: "mysecretpassword"      # In production, use Secrets!
      ports:
        - containerPort: 5432
      volumeMounts:
        - name: postgres-storage        # Must match volumes[].name below
          mountPath: /var/lib/postgresql/data   # Where PostgreSQL stores data
          subPath: pgdata                # Subdirectory within the volume
  volumes:
    - name: postgres-storage            # Volume name (referenced in volumeMounts)
      persistentVolumeClaim:
        claimName: postgres-data         # Name of the PVC we created
```

```bash
kubectl apply -f postgres-pod.yaml

# Now check PVC — it should be Bound!
kubectl get pvc
# NAME            STATUS   VOLUME                                     CAPACITY   ACCESS MODES   STORAGECLASS   AGE
# postgres-data   Bound    pvc-a1b2c3d4-5678-90ab-cdef-1234567890ab   20Gi       RWO            fast-ssd       2m

# Check PV that was auto-created
kubectl get pv
# NAME                                       CAPACITY   ACCESS MODES   RECLAIM POLICY   STATUS   CLAIM                    STORAGECLASS   AGE
# pvc-a1b2c3d4-5678-90ab-cdef-1234567890ab   20Gi       RWO            Delete           Bound    default/postgres-data    fast-ssd       1m

# 🎉 Kubernetes automatically:
# 1. Created an EBS gp3 volume on AWS
# 2. Created a PV pointing to that EBS volume
# 3. Bound the PVC to the PV
# 4. Mounted it into the Pod
# All in seconds!
```

Let's understand the volume mount in the Pod:

```
Pod YAML Explained:

  volumes:                              ← Define volumes available to this Pod
    - name: postgres-storage            ← Give the volume a name (any name you want)
      persistentVolumeClaim:
        claimName: postgres-data        ← Reference the PVC by name
                                          (this is the bridge between Pod and storage)

  containers:
    - volumeMounts:                     ← Mount volumes into the container
        - name: postgres-storage        ← Must match volumes[].name above
          mountPath: /var/lib/postgresql/data   ← Path inside the container
          subPath: pgdata               ← Optional: subdirectory within the volume
                                          (some apps, like PostgreSQL, need this
                                           because the mount dir must not be empty)
```

### Example 2: PVC with a Deployment (Most Common Real-World Pattern)

```yaml
# postgres-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: postgres
spec:
  replicas: 1                            # ⚠️ Only 1 replica with RWO!
  selector:
    matchLabels:
      app: postgres
  template:
    metadata:
      labels:
        app: postgres
    spec:
      containers:
        - name: postgres
          image: postgres:16
          env:
            - name: POSTGRES_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: postgres-secret
                  key: password
          ports:
            - containerPort: 5432
          volumeMounts:
            - name: postgres-storage
              mountPath: /var/lib/postgresql/data
              subPath: pgdata
          resources:
            requests:
              memory: "256Mi"
              cpu: "250m"
            limits:
              memory: "512Mi"
              cpu: "500m"
      volumes:
        - name: postgres-storage
          persistentVolumeClaim:
            claimName: postgres-data     # Same PVC as before
---
apiVersion: v1
kind: Service
metadata:
  name: postgres
spec:
  selector:
    app: postgres
  ports:
    - port: 5432
      targetPort: 5432
  type: ClusterIP
```

### Example 3: StatefulSet with volumeClaimTemplates (Best for Databases!)

This is the **recommended pattern** for stateful applications:

```yaml
# postgres-statefulset.yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres
spec:
  serviceName: postgres
  replicas: 3                            # 3 replicas, each gets its OWN PVC!
  selector:
    matchLabels:
      app: postgres
  template:
    metadata:
      labels:
        app: postgres
    spec:
      containers:
        - name: postgres
          image: postgres:16
          ports:
            - containerPort: 5432
          volumeMounts:
            - name: data
              mountPath: /var/lib/postgresql/data
              subPath: pgdata
  volumeClaimTemplates:                  # ← This is the magic!
    - metadata:
        name: data                       # Volume name (referenced in volumeMounts)
      spec:
        accessModes:
          - ReadWriteOnce
        resources:
          requests:
            storage: 20Gi
        storageClassName: fast-ssd
```

```bash
kubectl apply -f postgres-statefulset.yaml

# What happens? Kubernetes creates 3 PVCs automatically:
kubectl get pvc
# NAME              STATUS   VOLUME         CAPACITY   STORAGECLASS   AGE
# data-postgres-0   Bound    pvc-abc123     20Gi       fast-ssd       1m
# data-postgres-1   Bound    pvc-def456     20Gi       fast-ssd       1m
# data-postgres-2   Bound    pvc-ghi789     20Gi       fast-ssd       1m

# Each Pod gets its OWN dedicated PVC:
# postgres-0 → data-postgres-0 → 20Gi EBS volume
# postgres-1 → data-postgres-1 → 20Gi EBS volume
# postgres-2 → data-postgres-2 → 20Gi EBS volume

# Even if pods restart, they ALWAYS reconnect to the SAME PVC!
# postgres-0 will ALWAYS use data-postgres-0 ✅
```

```
volumeClaimTemplates vs Regular PVC:

Regular PVC (in Deployment):
┌──────────────┐
│  Deployment  │
│  replicas: 3 │
│              │
│  Pod-0 ──┐   │     ┌──────────────┐
│  Pod-1 ──┼───────→ │  Shared PVC  │  ← All pods share ONE PVC
│  Pod-2 ──┘   │     │  (20Gi)      │     (only works with RWX!)
│              │     └──────────────┘
└──────────────┘

┌──────────────┐
│ StatefulSet  │
│  replicas: 3 │
│              │     ┌──────────────────┐
│  Pod-0 ──────────→ │  PVC-0 (20Gi)    │  ← Each pod gets its OWN PVC!
│              │     └──────────────────┘     (works with RWO!)
│              │     ┌──────────────────┐
│  Pod-1 ──────────→ │  PVC-1 (20Gi)    │
│              │     └──────────────────┘
│              │     ┌──────────────────┐
│  Pod-2 ──────────→ │  PVC-2 (20Gi)    │
│              │     └──────────────────┘
└──────────────┘
```

## 🔄 Resizing PVCs (Volume Expansion)

One of the best features — you can **increase PVC size** without downtime (if your StorageClass supports it).

### Requirements

1. StorageClass must have `allowVolumeExpansion: true`
2. The CSI driver must support expansion

### How to Resize

```bash
# Check current PVC size
kubectl get pvc postgres-data
# NAME            STATUS   VOLUME         CAPACITY   STORAGECLASS   AGE
# postgres-data   Bound    pvc-abc123     20Gi       fast-ssd       1h

# Edit the PVC to increase size (you CANNOT decrease!)
kubectl edit pvc postgres-data
```

```yaml
# Change this:
spec:
  resources:
    requests:
      storage: 20Gi    # OLD size

# To this:
spec:
  resources:
    requests:
      storage: 50Gi    # NEW size (only increase!)
```

Or use `kubectl patch`:

```bash
# One-liner to resize
kubectl patch pvc postgres-data -p '{"spec":{"resources":{"requests":{"storage":"50Gi"}}}}'

# Check the status — you might see conditions about resizing
kubectl describe pvc postgres-data

# Events:
#   Type     Reason                      Message
#   ----     ------                      -------
#   Normal   Resizing                    External resizer is resizing volume pvc-abc123
#   Normal   FileSystemResizeRequired    Require file system resize of volume on node
#   Normal   FileSystemResizeSuccessful  MountVolume.NodeExpandVolume succeeded
```

```
⚠️  Important Notes on Resizing:

✅ You CAN increase PVC size
❌ You CANNOT decrease PVC size (ever!)
✅ Most CSI drivers support online expansion (no restart needed)
⚠️  Some older drivers need a Pod restart for filesystem expansion
📌 Always check if your StorageClass has allowVolumeExpansion: true
```

## 🏷️ Default StorageClass

You can mark one StorageClass as the **default**. Any PVC that doesn't specify a `storageClassName` will use the default.

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: standard
  annotations:
    storageclass.kubernetes.io/is-default-class: "true"   # ← Makes this the default!
provisioner: ebs.csi.aws.com
parameters:
  type: gp3
reclaimPolicy: Delete
volumeBindingMode: WaitForFirstConsumer
```

```bash
# Check which StorageClass is default
kubectl get storageclass
# NAME                 PROVISIONER       RECLAIMPOLICY   VOLUMEBINDINGMODE       AGE
# standard (default)   ebs.csi.aws.com   Delete          WaitForFirstConsumer    30d
# fast-ssd             ebs.csi.aws.com   Delete          WaitForFirstConsumer    30d
# high-iops            ebs.csi.aws.com   Retain          WaitForFirstConsumer    30d

# PVC without storageClassName will use "standard" (default)
```

```yaml
# This PVC will use the default StorageClass
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-data
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 5Gi
  # No storageClassName specified → uses default StorageClass!
```

> **Important:** If you explicitly set `storageClassName: ""` (empty string), it means "I want NO StorageClass — I want to manually bind to a pre-existing PV." This is different from omitting the field!

## 📝 Key Takeaways

```
1. PVC = Request for storage (what you need)
   StorageClass = Blueprint for creating storage (how to create it)
   PV = Actual storage resource (what gets created)

2. Dynamic Provisioning = StorageClass + PVC → PV created automatically
   Static Provisioning = Admin creates PV manually → PVC binds to it

3. Always use WaitForFirstConsumer for cloud environments
   (avoids zone mismatch issues)

4. Use Retain reclaim policy for production databases
   Use Delete for dev/test environments

5. PVCs can be resized (only increase, never decrease!)
   StorageClass must have allowVolumeExpansion: true

6. For StatefulSets, use volumeClaimTemplates
   Each replica gets its own dedicated PVC

7. Access Modes matter:
   - RWO for single-node apps (databases)
   - RWX for shared storage (NFS/EFS only)
   - RWOP for strict single-pod access
```
