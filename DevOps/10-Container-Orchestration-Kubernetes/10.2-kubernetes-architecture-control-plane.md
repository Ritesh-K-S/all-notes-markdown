# 10.2 — Kubernetes Architecture — Control Plane (API Server, etcd, Scheduler, Controller Manager)

> **One Line Answer:** The Control Plane is the **brain of Kubernetes** — it consists of the **API Server** (the front door), **etcd** (the memory/database), **Scheduler** (the matchmaker that assigns pods to nodes), and **Controller Manager** (the supervisor that ensures everything is running as desired). Together, they make all the decisions about your cluster.

---

## 🤔 Why Do You Need to Know the Control Plane?

You might be thinking: *"I just want to deploy apps. Why do I need to know what's inside the control plane?"*

Here's why:

1. **Troubleshooting** — Your deployment isn't working? Understanding the control plane helps you figure out WHERE the problem is. Is the API Server rejecting your request? Is the Scheduler unable to find a node? Is etcd corrupted?

2. **Interviews** — "Explain the Kubernetes control plane components" is a **TOP 5** Kubernetes interview question. You WILL be asked this.

3. **Security** — The API Server is the single entry point to your cluster. Understanding it helps you secure it properly.

4. **Cluster Administration** — If you ever set up or maintain a K8s cluster (especially on-premise), you need to understand each component deeply.

5. **Debugging** — When `kubectl` commands hang or fail, knowing the internal flow helps you diagnose issues fast.

### Critical Rule: **Everything goes through the API Server**

This is the most important thing to understand:

```
kubectl ──► API Server ──► etcd
Scheduler ──► API Server ──► etcd  
Controller Manager ──► API Server ──► etcd
kubelet (on worker nodes) ──► API Server ──► etcd

NOBODY talks to etcd directly (except the API Server)
NOBODY talks to each other directly (everything goes through the API Server)
```

The API Server is the **central hub**. Period.

## 1️⃣ kube-apiserver — The Front Door

### What Is It?

The API Server is the **front door** to the entire Kubernetes cluster. It's the **only** component that directly interacts with **etcd** (the database). Every single request — from `kubectl`, from the scheduler, from the controller manager, from kubelets — goes through the API Server.

### Real-World Analogy 🏨

Think of a **hotel front desk**:

- You (the guest) don't go directly to the kitchen, the housekeeping room, or the laundry
- You go to the **front desk** and make your request
- The front desk routes your request to the right department
- The front desk keeps a record of everything

The API Server is that front desk.

### What Does It Do?

| Function | Description |
|----------|-------------|
| **REST API** | Exposes a RESTful API (HTTPS) that accepts requests |
| **Authentication** | Verifies WHO you are (certificates, tokens, OIDC) |
| **Authorization** | Checks if you're ALLOWED to do what you're asking (RBAC) |
| **Admission Control** | Applies policies and mutations to requests before saving |
| **Validation** | Checks if your YAML/JSON is valid and correct |
| **etcd Gateway** | The ONLY component that reads/writes to etcd |
| **Watch Mechanism** | Lets other components "watch" for changes in real-time |

### The Request Flow

When you run `kubectl apply -f deployment.yaml`, here's what happens step by step:

```
┌─────────┐     HTTPS      ┌──────────────┐
│ kubectl │ ──────────────► │  API Server  │
└─────────┘                 └──────┬───────┘
                                   │
                    Step 1: Authentication
                    "Who is this person?"
                    (checks certificates/tokens)
                                   │
                    Step 2: Authorization (RBAC)
                    "Are they allowed to create
                     a deployment in this namespace?"
                                   │
                    Step 3: Admission Controllers
                    "Apply any policies/mutations"
                    (e.g., add default resource limits,
                     inject sidecar containers)
                                   │
                    Step 4: Validation
                    "Is this YAML schema correct?"
                    (e.g., is 'replicas' a number?)
                                   │
                    Step 5: Persist to etcd
                    "Save this deployment object
                     to the database"
                                   │
                                   ▼
                            ┌──────────┐
                            │   etcd   │
                            └──────────┘
```

### Let's See It in Action

```bash
# When you run this command:
kubectl apply -f deployment.yaml

# Behind the scenes, kubectl converts it to a REST API call:
# POST https://<api-server>:6443/apis/apps/v1/namespaces/default/deployments

# The API Server processes it through the pipeline above
# and responds with:
# deployment.apps/my-app created
```

### The API Server is RESTful

The API Server exposes standard REST endpoints:

```
GET    /api/v1/namespaces/default/pods          → List all pods
GET    /api/v1/namespaces/default/pods/my-pod   → Get a specific pod
POST   /api/v1/namespaces/default/pods          → Create a pod
PUT    /api/v1/namespaces/default/pods/my-pod   → Update a pod
DELETE /api/v1/namespaces/default/pods/my-pod   → Delete a pod
PATCH  /api/v1/namespaces/default/pods/my-pod   → Partially update a pod
```

You can even call these directly (though you'd normally use kubectl):

```bash
# Using curl to talk to the API Server directly
# (you'd need proper authentication tokens)
kubectl proxy &
curl http://localhost:8001/api/v1/namespaces/default/pods
```

This returns a JSON response with all your pods. kubectl is just a friendly wrapper around these API calls.

### Admission Controllers — The Security Guards

Admission Controllers are plugins that intercept requests **after** authentication and authorization but **before** the object is persisted to etcd. They can:

1. **Mutate** (change) the request — called **Mutating Admission Webhooks**
2. **Validate** (accept/reject) the request — called **Validating Admission Webhooks**

```
Request → Authentication → Authorization → Mutating Admission → Validating Admission → etcd

Example flow:
1. You submit a pod without resource limits
2. Mutating webhook adds default limits: cpu=500m, memory=256Mi
3. Validating webhook checks: "Does this pod meet our security policy?"
4. If yes → saved to etcd
5. If no → request is REJECTED with an error message
```

Some built-in admission controllers:

| Controller | What It Does |
|-----------|--------------|
| **NamespaceLifecycle** | Prevents creating objects in non-existent namespaces |
| **LimitRanger** | Applies default resource limits |
| **ServiceAccount** | Automatically adds a ServiceAccount to pods |
| **DefaultStorageClass** | Assigns a default storage class to PVCs |
| **ResourceQuota** | Enforces resource quotas in namespaces |
| **PodSecurity** | Enforces Pod Security Standards |

### API Server Key Details

| Detail | Value |
|--------|-------|
| **Binary name** | `kube-apiserver` |
| **Default port** | `6443` (HTTPS) |
| **Protocol** | REST over HTTPS |
| **Stateless?** | Yes — all state is in etcd |
| **Can have multiple?** | Yes — can run multiple replicas for HA (High Availability) |
| **Config location** | `/etc/kubernetes/manifests/kube-apiserver.yaml` (kubeadm) |

## 2️⃣ etcd — The Brain's Memory

### What Is It?

**etcd** (pronounced "et-see-dee") is a **distributed, reliable key-value store** that serves as Kubernetes' **single source of truth**. Every piece of data about your cluster — pods, services, deployments, secrets, config maps, node information — it's ALL stored in etcd.

### Real-World Analogy 🗄️

Think of etcd as the **filing cabinet** in a hospital:

- Every patient record is there
- Every doctor's schedule is there
- Every medication inventory is there
- If the filing cabinet is destroyed, the hospital can't function
- So you keep **backup copies** of the filing cabinet (redundancy)

In Kubernetes:
- etcd is the filing cabinet
- The API Server is the only person allowed to access the filing cabinet
- If etcd goes down, the cluster can't make any new decisions
- So you ALWAYS run etcd with backups and redundancy

### What Does It Store?

**Everything.** Literally everything about the cluster:

```
etcd stores:
├── /registry/pods/default/my-web-app-xyz     → Pod definitions
├── /registry/deployments/default/my-app      → Deployment specs
├── /registry/services/default/my-service     → Service configurations
├── /registry/secrets/default/db-password     → Secrets (encrypted)
├── /registry/configmaps/default/app-config   → ConfigMaps
├── /registry/nodes/worker-1                  → Node information
├── /registry/namespaces/production           → Namespace info
├── /registry/serviceaccounts/default/default → Service accounts
└── ... (everything else)
```

### How etcd Works — Key-Value Store

etcd is a **key-value** store, like a dictionary or hash map:

```
Key                                          → Value (JSON)
─────────────────────────────────────────────────────────
/registry/pods/default/nginx-abc123          → { "name": "nginx-abc123", 
                                                 "image": "nginx:1.25",
                                                 "status": "Running", ... }

/registry/deployments/default/my-app         → { "replicas": 3,
                                                 "image": "my-app:v2",
                                                 "strategy": "RollingUpdate", ... }

/registry/services/default/web-service       → { "type": "ClusterIP",
                                                 "port": 80,
                                                 "targetPort": 8080, ... }
```

### Why etcd? (Why Not MySQL or PostgreSQL?)

| Feature | Why It Matters for K8s |
|---------|----------------------|
| **Distributed** | Runs on multiple nodes — no single point of failure |
| **Consistent** | Uses Raft consensus — all nodes agree on the data |
| **Fast** | Optimized for reads (K8s reads WAY more than it writes) |
| **Watch API** | Components can "watch" for changes in real-time |
| **Simple** | Key-value model is simpler than relational databases |
| **Reliable** | Battle-tested at massive scale |

### etcd and the Raft Consensus Algorithm

When you run etcd in a cluster (which you should in production), the nodes need to agree on the data. This is done using the **Raft consensus algorithm**:

```
etcd Cluster (3 nodes — minimum for HA):

┌──────────┐     ┌──────────┐     ┌──────────┐
│  etcd-1  │◄───►│  etcd-2  │◄───►│  etcd-3  │
│ (Leader) │     │(Follower)│     │(Follower)│
└──────────┘     └──────────┘     └──────────┘

Write request flow:
1. Client sends write to Leader (etcd-1)
2. Leader replicates to Followers (etcd-2, etcd-3)
3. Once MAJORITY (2 out of 3) confirm → write is committed
4. If Leader dies → Followers elect a new Leader automatically
```

### How Many etcd Nodes Should You Have?

| Number of etcd Nodes | Fault Tolerance | Can Lose | Recommended? |
|---------------------|-----------------|----------|-------------|
| 1 | None | 0 nodes | ❌ Dev/testing only |
| 2 | None (split brain risk) | 0 nodes | ❌ Never do this |
| **3** | **1 node failure** | **1 node** | **✅ Minimum for production** |
| 5 | 2 node failures | 2 nodes | ✅ Large production clusters |
| 7 | 3 node failures | 3 nodes | ✅ Very large / critical |

> **Why always odd numbers?** Because of the majority rule. With 3 nodes, the majority is 2. If 1 dies, 2 are still alive = majority = cluster works. With 4 nodes, majority is 3. If 2 die, only 2 left < 3 = cluster fails. So 4 nodes is NOT better than 3!

### Interacting with etcd

You can interact with etcd directly using `etcdctl` (but you should rarely need to):

```bash
# Check etcd cluster health
etcdctl endpoint health

# List all keys (be careful — lots of output!)
etcdctl get / --prefix --keys-only

# Get a specific key
etcdctl get /registry/pods/default/my-pod

# Take a snapshot (BACKUP — very important!)
etcdctl snapshot save /backup/etcd-snapshot.db

# Restore from a snapshot
etcdctl snapshot restore /backup/etcd-snapshot.db

# Check cluster member list
etcdctl member list
```

### etcd Backup — This is CRITICAL ⚠️

> **If you lose etcd data and have no backup, you lose your ENTIRE cluster configuration.** All deployments, services, secrets — everything. Gone.

```bash
# Take a backup (do this regularly!)
ETCDCTL_API=3 etcdctl snapshot save /backup/etcd-$(date +%Y%m%d).db \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key

# Verify the backup
ETCDCTL_API=3 etcdctl snapshot status /backup/etcd-$(date +%Y%m%d).db --write-table

# Output:
+----------+----------+------------+------------+
|   HASH   | REVISION | TOTAL KEYS | TOTAL SIZE |
+----------+----------+------------+------------+
| 5f12a14c |   128432 |       1287 |     5.1 MB |
+----------+----------+------------+------------+
```

### etcd Key Details

| Detail | Value |
|--------|-------|
| **Binary name** | `etcd` |
| **Client port** | `2379` |
| **Peer port** | `2380` (for etcd-to-etcd communication) |
| **Data directory** | `/var/lib/etcd` |
| **Config** | `/etc/kubernetes/manifests/etcd.yaml` (kubeadm) |
| **Written in** | Go |
| **Protocol** | gRPC + HTTP/2 |

## 3️⃣ kube-scheduler — The Matchmaker

### What Is It?

The Scheduler is responsible for **assigning newly created pods to worker nodes**. When a pod is created but not yet assigned to a node (it's in "Pending" state), the Scheduler picks the best node for it.

### Real-World Analogy 🏠

Think of the Scheduler as a **real estate agent**:

A family (pod) comes to the agent and says:
> "We need a house (node) with at least 3 bedrooms (CPU), 2 bathrooms (memory), near a school (specific label), and not in the flood zone (taint)."

The agent looks at all available houses, filters out the ones that don't meet the requirements, ranks the remaining ones, and picks the best match.

**The Scheduler does exactly this — but for pods and nodes.**

### What Does It Do?

1. **Watches** the API Server for newly created pods that have no node assigned
2. **Filters** out nodes that can't run the pod (not enough resources, wrong labels, taints, etc.)
3. **Scores** the remaining nodes to find the best one
4. **Binds** the pod to the chosen node (tells the API Server which node to use)

### The Scheduling Process — Step by Step

```
New pod created (no node assigned yet)
              │
              ▼
┌──────────────────────────┐
│  Step 1: FILTERING       │
│  "Which nodes CAN run    │
│   this pod?"             │
│                          │
│  Check:                  │
│  ✅ Enough CPU?          │
│  ✅ Enough Memory?       │
│  ✅ Correct labels?      │
│  ✅ No blocking taints?  │
│  ✅ Port available?      │
│  ✅ Volume can mount?    │
│                          │
│  Node 1: ✅ Passes       │
│  Node 2: ❌ Not enough RAM│
│  Node 3: ✅ Passes       │
│  Node 4: ❌ Has a taint  │
└────────────┬─────────────┘
             │
             ▼
┌──────────────────────────┐
│  Step 2: SCORING         │
│  "Which passing node     │
│   is the BEST?"          │
│                          │
│  Factors:                │
│  - Resource balance      │
│  - Affinity rules        │
│  - Data locality         │
│  - Spreading across zones│
│                          │
│  Node 1: Score 78/100    │
│  Node 3: Score 92/100    │
└────────────┬─────────────┘
             │
             ▼
┌──────────────────────────┐
│  Step 3: BINDING         │
│  "Assign pod to Node 3"  │
│                          │
│  Scheduler → API Server  │
│  "Put this pod on Node 3"│
│                          │
│  API Server → etcd       │
│  "Save: pod.nodeName =   │
│   node-3"                │
└──────────────────────────┘
```

### Filtering Phase — What Checks Are Done?

Here's a detailed look at the filters (called **predicates** in older versions or **filter plugins** in the scheduling framework):

| Filter | What It Checks |
|--------|---------------|
| **PodFitsResources** | Does the node have enough CPU and memory for this pod? |
| **PodFitsHostPorts** | Are the required host ports available on this node? |
| **NodeSelector** | Does the node have the labels the pod requires? |
| **NodeAffinity** | Does the node match the pod's affinity rules? |
| **Taints/Tolerations** | Does the pod tolerate the node's taints? |
| **PodAntiAffinity** | Is there a pod on this node that this pod wants to AVOID? |
| **VolumeZone** | Is the pod's volume in the same zone as this node? |
| **NoVolumeConflicts** | Does the node already have conflicting volumes mounted? |

### Scoring Phase — How Are Nodes Ranked?

After filtering, the remaining nodes are scored (called **priorities** or **score plugins**):

| Scoring Factor | What It Does |
|----------------|-------------|
| **LeastRequestedPriority** | Favors nodes with the MOST free resources (spreads load) |
| **BalancedResourceAllocation** | Favors nodes where CPU and memory usage are balanced |
| **NodeAffinityPriority** | Gives higher score to nodes matching affinity preferences |
| **PodTopologySpread** | Spreads pods across zones/racks for high availability |
| **ImageLocality** | Favors nodes that already have the container image cached |
| **InterPodAffinity** | Favors nodes where the pod's "friends" are already running |

### Example: Scheduling Decision

Let's say you have this pod:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
  - name: app
    image: my-app:v2
    resources:
      requests:
        cpu: "500m"       # Needs 0.5 CPU cores
        memory: "256Mi"   # Needs 256 MB RAM
  nodeSelector:
    disk: ssd             # Must be on a node with SSD
```

And these nodes:

```
Node 1: 4 CPU, 8Gi RAM, labels: {disk: hdd}       → ❌ FILTERED OUT (no SSD)
Node 2: 2 CPU, 4Gi RAM, labels: {disk: ssd}        → ✅ Passes filter
         (1.5 CPU free, 3Gi RAM free)                    Score: 75
Node 3: 8 CPU, 16Gi RAM, labels: {disk: ssd}       → ✅ Passes filter
         (6 CPU free, 12Gi RAM free)                     Score: 95
Node 4: 1 CPU, 2Gi RAM, labels: {disk: ssd}        → ✅ Passes filter
         (0.3 CPU free, 1Gi RAM free)                    ❌ Not enough CPU!
                                                          Actually filtered out.
```

**Result:** Pod is scheduled on **Node 3** (highest score, most resources available).

### What If No Node Fits?

If the Scheduler can't find ANY node that passes all filters, the pod stays in **Pending** state:

```bash
kubectl get pods
# Output:
NAME     READY   STATUS    RESTARTS   AGE
my-app   0/1     Pending   0          5m

kubectl describe pod my-app
# Events:
#   Warning  FailedScheduling  0/3 nodes are available:
#     1 Insufficient cpu,
#     1 Insufficient memory,
#     1 node(s) had taint {key: NoSchedule}
```

This is **one of the most common issues** you'll troubleshoot. The fix is usually:
1. Add more nodes (scale the cluster)
2. Free up resources (reduce other pods' requests)
3. Adjust the pod's resource requests
4. Remove taints or add tolerations

## 4️⃣ kube-controller-manager — The Supervisor

### What Is It?

The Controller Manager is a **collection of controllers** bundled into a single binary. Each controller is a **control loop** that watches the state of the cluster (via the API Server) and makes changes to move the **current state** toward the **desired state**.

### Real-World Analogy 🏭

Think of a **factory floor supervisor**:

The supervisor has a checklist:
- ✅ Assembly Line A should have 5 workers → checks → only 4 → calls in 1 more
- ✅ Quality check station should be running → checks → it's stopped → restarts it
- ✅ Warehouse should have 100 units in stock → checks → only 80 → orders production of 20 more

The Controller Manager is exactly this — a set of supervisors, each watching a different thing, each making sure reality matches the plan.

### The Control Loop Pattern

Every controller in Kubernetes follows this pattern:

```
         ┌─────────────────────────────────┐
         │                                 │
         ▼                                 │
┌─────────────────┐                        │
│  OBSERVE         │                        │
│  (Watch current  │                        │
│   state via API) │                        │
└────────┬────────┘                        │
         │                                 │
         ▼                                 │
┌─────────────────┐                        │
│  DIFF            │                        │
│  (Compare with   │                        │
│   desired state) │                        │
└────────┬────────┘                        │
         │                                 │
    ┌────┴────┐                            │
    │ Match?  │                            │
    └────┬────┘                            │
     Yes │  No                             │
         │   │                             │
         │   ▼                             │
         │ ┌─────────────────┐             │
         │ │  ACT             │             │
         │ │  (Take action to │             │
         │ │   fix the diff)  │─────────────┘
         │ └─────────────────┘
         │
         ▼
      (Do nothing,
       check again
       in a moment)
```

This loop runs **continuously** — that's how Kubernetes achieves self-healing.

### What Controllers Are Inside?

The `kube-controller-manager` binary contains **many** controllers. Here are the most important ones:

#### 🔄 Deployment Controller

**Watches:** Deployments
**Ensures:** The right number of ReplicaSets exist and are at the right version

```
You say: "I want 3 replicas of my-app:v2"

Deployment Controller checks:
→ Current state: ReplicaSet with 3 replicas of my-app:v1
→ Desired state: 3 replicas of my-app:v2
→ Action: Create a new ReplicaSet for v2, scale it up,
          scale down the v1 ReplicaSet (rolling update!)
```

#### 🔄 ReplicaSet Controller

**Watches:** ReplicaSets
**Ensures:** The correct number of pod replicas are running

```
ReplicaSet says: "I want 3 pods"

ReplicaSet Controller checks:
→ Current: 2 pods running
→ Desired: 3 pods
→ Action: Create 1 more pod

Later...
→ Current: 4 pods running (someone created one manually)
→ Desired: 3 pods
→ Action: Terminate 1 pod
```

This is the controller that enables **self-healing**. If a pod dies, the ReplicaSet Controller notices and creates a replacement.

#### 🖥️ Node Controller

**Watches:** Nodes
**Ensures:** Nodes are healthy and takes action when they're not

```
Node Controller checks every node:
→ Node 1: Last heartbeat 10 seconds ago ✅ Healthy
→ Node 2: Last heartbeat 5 seconds ago  ✅ Healthy
→ Node 3: Last heartbeat 5 MINUTES ago  ❌ Unreachable!

Action for Node 3:
1. Mark it as "NotReady"
2. Wait grace period (default: 5 minutes)
3. If still not responding → evict all pods from Node 3
4. Those pods' ReplicaSet controllers will recreate them on healthy nodes
```

#### 🔐 ServiceAccount Controller

**Watches:** Namespaces
**Ensures:** Every namespace has a default ServiceAccount

```
New namespace "production" created
→ ServiceAccount Controller: "I need to create a 
   'default' ServiceAccount in the 'production' namespace"
→ Creates it automatically
```

#### 🏷️ Namespace Controller

**Watches:** Namespaces
**Ensures:** When a namespace is deleted, ALL resources in it are cleaned up

```
You delete namespace "staging":
→ Namespace Controller: "I need to delete ALL pods, services,
   deployments, configmaps, secrets in namespace 'staging'"
→ Cleans up everything
```

#### 🔗 Endpoints Controller

**Watches:** Services and Pods
**Ensures:** Services have the correct endpoints (pod IPs)

```
Service "web-service" selects pods with label app=web
→ Pod-1 (10.0.1.5) has label app=web ✅
→ Pod-2 (10.0.1.6) has label app=web ✅
→ Pod-3 (10.0.2.3) has label app=web ✅

Endpoints Controller creates:
web-service endpoints: [10.0.1.5, 10.0.1.6, 10.0.2.3]

Pod-2 crashes:
→ Endpoints Controller removes 10.0.1.6 from the list
Updated endpoints: [10.0.1.5, 10.0.2.3]
```

#### 📋 Job Controller

**Watches:** Jobs
**Ensures:** The specified number of pods run to completion

```
Job says: "Run this batch task to completion"
→ Job Controller creates a pod
→ Pod completes successfully ✅
→ Job Controller marks the Job as "Complete"

If the pod fails:
→ Job Controller creates a new pod (retry)
→ Up to the configured retry limit
```

### All Controllers at a Glance

| Controller | Watches | Ensures |
|-----------|---------|---------|
| **Deployment** | Deployments | Correct ReplicaSets exist |
| **ReplicaSet** | ReplicaSets | Correct number of pods running |
| **Node** | Nodes | Nodes are healthy, evict pods from dead nodes |
| **ServiceAccount** | Namespaces | Default ServiceAccount exists |
| **Namespace** | Namespaces | Cleanup on namespace deletion |
| **Endpoints** | Services + Pods | Services have correct pod IPs |
| **Job** | Jobs | Batch tasks run to completion |
| **CronJob** | CronJobs | Scheduled jobs run on time |
| **StatefulSet** | StatefulSets | Ordered, sticky pod management |
| **DaemonSet** | DaemonSets | A pod runs on every (or selected) node |
| **PV Binder** | PVCs | PersistentVolumeClaims are bound to PersistentVolumes |
| **TTL** | Finished pods | Cleans up completed pods after TTL |

### Controller Manager Key Details

| Detail | Value |
|--------|-------|
| **Binary name** | `kube-controller-manager` |
| **Default port** | `10257` (HTTPS) |
| **Stateless?** | Yes |
| **Leader election?** | Yes (only one instance is active at a time) |
| **Config** | `/etc/kubernetes/manifests/kube-controller-manager.yaml` (kubeadm) |

## 5️⃣ cloud-controller-manager — The Cloud Translator

### What Is It?

The Cloud Controller Manager (CCM) is an **optional** component that only exists when your Kubernetes cluster runs on a **cloud provider** (AWS, Azure, GCP, etc.). It contains controllers that interact with the cloud provider's API.

### Why Does It Exist?

In the early days, all cloud-specific code was inside the `kube-controller-manager`. This was messy — every time AWS or Azure needed a change, they had to modify core Kubernetes code.

The solution? **Separate the cloud-specific stuff** into its own component:

```
Before (messy):
kube-controller-manager
├── Node Controller (+ AWS-specific code)
├── Route Controller (+ Azure-specific code)
├── Service Controller (+ GCP-specific code)
└── ... (cloud code mixed with core code)

After (clean):
kube-controller-manager              cloud-controller-manager
├── Node Controller (core only)      ├── Node Controller (cloud parts)
├── ReplicaSet Controller            ├── Route Controller
├── Deployment Controller            ├── Service Controller
└── ... (pure Kubernetes)            └── ... (cloud-specific)
```

### What Controllers Does It Have?

| Controller | What It Does |
|-----------|-------------|
| **Node Controller** | Checks with cloud API if a node VM still exists. If the VM was terminated, removes the node from K8s |
| **Route Controller** | Sets up network routes in the cloud VPC so pods can communicate across nodes |
| **Service Controller** | When you create a Service of type `LoadBalancer`, it provisions a cloud load balancer (AWS ELB, Azure LB, GCP LB) |

### Example: LoadBalancer Service

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-web-service
spec:
  type: LoadBalancer      # ← This triggers the cloud-controller-manager
  selector:
    app: web
  ports:
  - port: 80
    targetPort: 8080
```

When you apply this:
1. API Server saves it to etcd
2. Cloud Controller Manager's **Service Controller** sees: "type: LoadBalancer"
3. It calls the cloud provider's API (e.g., AWS API)
4. AWS creates an Elastic Load Balancer (ELB)
5. The ELB gets a public IP/DNS
6. Cloud Controller Manager updates the Service with the external IP

```bash
kubectl get svc my-web-service
# NAME             TYPE           CLUSTER-IP     EXTERNAL-IP        PORT(S)
# my-web-service   LoadBalancer   10.96.45.123   a1b2c3.elb.aws.com 80:31234/TCP
```

### Cloud Controller Manager Key Details

| Detail | Value |
|--------|-------|
| **Binary name** | `cloud-controller-manager` |
| **Required?** | No — only for cloud environments |
| **Provider-specific** | Each cloud has its own implementation |
| **Examples** | `aws-cloud-controller-manager`, `azure-cloud-controller-manager` |

## 🔄 How Everything Fits Together — Complete Request Flow

Let's trace a **complete flow** of deploying an application to see all control plane components working together:

### Scenario: You run `kubectl apply -f deployment.yaml`

```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
      - name: web
        image: nginx:1.25
        ports:
        - containerPort: 80
```

### The Complete Flow:

```
Step 1: kubectl → API Server
────────────────────────────
kubectl converts your YAML to a REST API call and sends it
to the API Server (port 6443).

Step 2: API Server processes the request
────────────────────────────
→ Authentication: Is this user valid? ✅
→ Authorization: Can they create deployments? ✅
→ Admission: Apply any policies ✅
→ Validation: Is the YAML correct? ✅
→ Save to etcd ✅

Step 3: Deployment Controller notices
────────────────────────────
The Deployment Controller (inside kube-controller-manager) 
is watching for new Deployments via the API Server.

"New Deployment detected! It says 3 replicas. 
 I need to create a ReplicaSet."

→ Creates a ReplicaSet object (via API Server → etcd)

Step 4: ReplicaSet Controller notices
────────────────────────────
The ReplicaSet Controller sees the new ReplicaSet.

"This ReplicaSet wants 3 pods. Currently 0 pods exist.
 I need to create 3 pods."

→ Creates 3 Pod objects (via API Server → etcd)
→ These pods have nodeName: "" (no node assigned yet)

Step 5: Scheduler notices
────────────────────────────
The Scheduler is watching for pods with no node assigned.

"3 new pods with no node! Let me find the best nodes."

→ Filters and scores nodes for each pod
→ Pod 1 → Node A (score: 95)
→ Pod 2 → Node B (score: 90)
→ Pod 3 → Node A (score: 88)

→ Updates pod.spec.nodeName via API Server → etcd

Step 6: kubelet on each node notices
────────────────────────────
The kubelet on Node A sees: "I have 2 new pods assigned to me"
The kubelet on Node B sees: "I have 1 new pod assigned to me"

→ Each kubelet pulls the container image and starts the container
→ Reports back: "Pod is running!" (via API Server → etcd)

Step 7: Done! 🎉
────────────────────────────
kubectl get pods
NAME                       READY   STATUS    NODE
web-app-5d8f9b7c4-abc12   1/1     Running   node-a
web-app-5d8f9b7c4-def34   1/1     Running   node-b
web-app-5d8f9b7c4-ghi56   1/1     Running   node-a
```

Here's the same flow as a diagram:

```
 kubectl                API Server              etcd
   │                       │                      │
   │── apply deployment ──►│                      │
   │                       │── save deployment ──►│
   │                       │                      │
   │                       │     Deployment Controller
   │                       │◄── "create ReplicaSet" ──
   │                       │── save ReplicaSet ──►│
   │                       │                      │
   │                       │     ReplicaSet Controller
   │                       │◄── "create 3 Pods" ──
   │                       │── save 3 Pods ──────►│
   │                       │                      │
   │                       │     Scheduler
   │                       │◄── "assign pods to nodes" ──
   │                       │── update nodeName ──►│
   │                       │                      │
   │                       │     kubelet (on nodes)
   │                       │◄── "pull image, run container" ──
   │                       │── update pod status ►│
   │                       │                      │
   │◄── deployment ready ──│                      │
```

## 🏥 High Availability (HA) for the Control Plane

In production, you **never** run a single control plane node. If it dies, you can't manage your cluster.

### HA Setup:

```
┌──────────────────────────────────────────────────────────┐
│                    LOAD BALANCER                         │
│              (for API Server traffic)                    │
└──────────────┬───────────────┬───────────────┬──────────┘
               │               │               │
      ┌────────▼──────┐ ┌─────▼───────┐ ┌─────▼───────┐
      │ Master Node 1 │ │ Master Node 2│ │ Master Node 3│
      │               │ │              │ │              │
      │ API Server    │ │ API Server   │ │ API Server   │
      │ Scheduler*    │ │ Scheduler    │ │ Scheduler    │
      │ Controller*   │ │ Controller   │ │ Controller   │
      │ etcd          │ │ etcd         │ │ etcd         │
      └───────────────┘ └──────────────┘ └──────────────┘
      
      * = Active (leader)
      Others = Standby (will take over if leader fails)
```

**Key points:**
- **API Server**: ALL instances are active (stateless, load balanced)
- **Scheduler**: Only ONE is active (leader election). Others are standby
- **Controller Manager**: Only ONE is active (leader election). Others are standby
- **etcd**: ALL are active (distributed, Raft consensus)

## 📍 Where Are Control Plane Components Running?

On a cluster set up with `kubeadm`, the control plane components run as **static pods** on the master node:

```bash
# Check control plane pods
kubectl get pods -n kube-system

# Output:
NAME                                  READY   STATUS    RESTARTS   AGE
etcd-master-node                      1/1     Running   0          30d
kube-apiserver-master-node            1/1     Running   0          30d
kube-controller-manager-master-node   1/1     Running   0          30d
kube-scheduler-master-node            1/1     Running   0          30d
```

Their configuration files (static pod manifests) are at:

```
/etc/kubernetes/manifests/
├── etcd.yaml
├── kube-apiserver.yaml
├── kube-controller-manager.yaml
└── kube-scheduler.yaml
```

On **managed Kubernetes** (EKS, AKS, GKE), the control plane is managed by the cloud provider — you don't even see these pods. You only manage worker nodes.

## 📋 Summary — Control Plane Components

| Component | Role | Analogy | Key Port |
|-----------|------|---------|----------|
| **kube-apiserver** | Front door, authentication, authorization, validation, etcd gateway | Hotel front desk | 6443 |
| **etcd** | Distributed key-value database, stores ALL cluster data | Filing cabinet | 2379, 2380 |
| **kube-scheduler** | Assigns unscheduled pods to the best available node | Real estate agent | 10259 |
| **kube-controller-manager** | Runs controllers that ensure desired state = current state | Factory supervisors | 10257 |
| **cloud-controller-manager** | Integrates with cloud provider APIs (load balancers, routes, VMs) | Cloud translator | varies |

### Key Takeaways:

1. **API Server is the center** — everything goes through it
2. **etcd is the brain's memory** — lose it without backup = disaster
3. **Scheduler is the matchmaker** — filters, scores, binds pods to nodes
4. **Controller Manager is the supervisor** — a collection of control loops that ensure desired state
5. **Everything is declarative** — you declare what you want, controllers make it happen
6. **High availability** — always run multiple control plane nodes in production
