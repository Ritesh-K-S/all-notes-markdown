# 10.20 — CSI Drivers — EBS, EFS, Azure Disk, GCE PD

> **One Line Answer:** **CSI (Container Storage Interface)** is a **standard plugin system** that lets Kubernetes talk to ANY storage provider — whether it's AWS EBS, Azure Disk, Google Persistent Disk, NFS, or anything else — through a common interface. CSI Drivers are the **plugins** that implement this interface, so Kubernetes doesn't need to know the specifics of each storage system — it just talks to the CSI driver, and the driver handles everything.

---

## 🤔 Why Do We Need CSI Drivers?

### The Old Way — In-Tree Plugins (The Problem)

Before CSI, storage drivers were **baked directly into the Kubernetes codebase** (called "in-tree" plugins). This caused serious problems:

```
The Old Way (In-Tree Storage Plugins):

┌─────────────────────────────────────────────────────┐
│              Kubernetes Core Codebase                │
│                                                     │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌────────┐ │
│  │ AWS EBS  │ │ GCE PD   │ │ Azure    │ │ NFS    │ │
│  │ plugin   │ │ plugin   │ │ Disk     │ │ plugin │ │
│  │          │ │          │ │ plugin   │ │        │ │
│  └──────────┘ └──────────┘ └──────────┘ └────────┘ │
│                                                     │
│  Problems:                                          │
│  ❌ Every new storage provider = change K8s code    │
│  ❌ Bug in a plugin? Wait for K8s release to fix    │
│  ❌ K8s binary becomes HUGE with all plugins        │
│  ❌ Plugin devs must follow K8s release cycle       │
│  ❌ Can't add custom storage without forking K8s    │
└─────────────────────────────────────────────────────┘
```

### The New Way — CSI (The Solution)

```
The New Way (CSI — Container Storage Interface):

┌─────────────────────────────┐
│    Kubernetes Core          │
│                             │
│    ┌─────────────────┐      │
│    │  CSI Interface   │      │     Standard API — same for ALL storage!
│    │  (standard API)  │      │
│    └────────┬────────┘      │
│             │               │
└─────────────┼───────────────┘
              │
    ┌─────────┼─────────────────────────────────┐
    │         │                                 │
    ▼         ▼              ▼                  ▼
┌────────┐ ┌────────┐ ┌───────────┐ ┌──────────────┐
│AWS EBS │ │GCE PD  │ │Azure Disk │ │ Any Custom   │
│CSI     │ │CSI     │ │CSI Driver │ │ CSI Driver   │
│Driver  │ │Driver  │ │           │ │ (NetApp,     │
│        │ │        │ │           │ │  Ceph, etc.) │
└────────┘ └────────┘ └───────────┘ └──────────────┘
    │           │           │              │
    ▼           ▼           ▼              ▼
 AWS EBS     GCE PD    Azure Disk    Any Storage
 Volumes     Disks     Managed Disks  System

Benefits:
✅ Storage providers develop their own drivers independently
✅ Bugs fixed by provider, not Kubernetes team
✅ K8s core stays lean and clean
✅ Anyone can build a CSI driver for ANY storage
✅ Drivers can be updated without updating K8s
```

## 🧠 How CSI Works — The Architecture

### CSI Components

Every CSI driver has two main components that run as Pods in your cluster:

```
CSI Driver Architecture:

┌──────────────────────────────────────────────────────────────────────┐
│                        Kubernetes Cluster                            │
│                                                                      │
│  ┌─────────────────────────────────────────────┐                     │
│  │        CSI Controller (Deployment)           │                     │
│  │        (Runs on any node — usually 1-2)      │                     │
│  │                                              │                     │
│  │  ┌─────────────────┐  ┌──────────────────┐   │                     │
│  │  │  CSI Controller  │  │  Sidecar          │   │                     │
│  │  │  Plugin          │  │  Containers:      │   │                     │
│  │  │                  │  │  - provisioner    │   │                     │
│  │  │  Handles:        │  │  - attacher       │   │                     │
│  │  │  - Create volume │  │  - snapshotter    │   │                     │
│  │  │  - Delete volume │  │  - resizer        │   │                     │
│  │  │  - Attach volume │  │                   │   │                     │
│  │  │  - Snapshot      │  │                   │   │                     │
│  │  └─────────────────┘  └──────────────────┘   │                     │
│  └─────────────────────────────────────────────┘                     │
│                                                                      │
│  ┌─────────────────────────────────────────────┐                     │
│  │        CSI Node Plugin (DaemonSet)           │  ← Runs on EVERY   │
│  │        (Runs on every worker node)           │     worker node     │
│  │                                              │                     │
│  │  ┌─────────────────┐  ┌──────────────────┐   │                     │
│  │  │  CSI Node        │  │  Sidecar:        │   │                     │
│  │  │  Plugin          │  │  - node-driver   │   │                     │
│  │  │                  │  │    -registrar     │   │                     │
│  │  │  Handles:        │  │                   │   │                     │
│  │  │  - Mount volume  │  │                   │   │                     │
│  │  │  - Unmount volume│  │                   │   │                     │
│  │  │  - Format disk   │  │                   │   │                     │
│  │  └─────────────────┘  └──────────────────┘   │                     │
│  └─────────────────────────────────────────────┘                     │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

### The Flow — What Happens When a Pod Needs Storage

```
Step-by-Step: PVC → CSI Driver → Actual Storage

1. Developer creates PVC
   ┌──────────────────────────┐
   │  PVC: "I need 20Gi of   │
   │   fast-ssd storage"     │
   └────────────┬─────────────┘
                │
                ▼
2. Kubernetes sees the PVC, checks the StorageClass
   ┌──────────────────────────┐
   │  StorageClass: fast-ssd  │
   │  Provisioner:            │
   │  ebs.csi.aws.com        │ ← "Use the EBS CSI driver"
   └────────────┬─────────────┘
                │
                ▼
3. CSI Controller creates the actual volume
   ┌──────────────────────────┐
   │  CSI Controller calls    │
   │  AWS API:                │
   │  "Create a 20Gi gp3     │
   │   EBS volume in         │
   │   us-east-1a"           │
   └────────────┬─────────────┘
                │
                ▼
4. PV is created and bound to PVC
   ┌──────────────────────────┐
   │  PV created:             │
   │  Name: pvc-abc123       │
   │  Size: 20Gi             │
   │  EBS ID: vol-0123456789 │
   │  Status: Bound          │
   └────────────┬─────────────┘
                │
                ▼
5. Pod is scheduled to a node
   ┌──────────────────────────┐
   │  Scheduler: "Put this    │
   │  pod on Node-2 in       │
   │  us-east-1a"            │
   └────────────┬─────────────┘
                │
                ▼
6. CSI Controller attaches volume to the node
   ┌──────────────────────────┐
   │  CSI Controller:         │
   │  "Attach EBS vol-012345 │
   │   to EC2 instance       │
   │   i-abcdef on Node-2"   │
   └────────────┬─────────────┘
                │
                ▼
7. CSI Node Plugin mounts the volume into the Pod
   ┌──────────────────────────┐
   │  CSI Node Plugin:        │
   │  "Format as ext4,       │
   │   mount at /data in     │
   │   the container"        │
   └──────────────────────────┘

   🎉 Pod now has storage!
```

## ☁️ AWS EBS CSI Driver

### What Is It?

The **AWS EBS CSI Driver** allows Kubernetes to create, attach, resize, and delete Amazon EBS (Elastic Block Store) volumes automatically.

```
AWS EBS Quick Facts:
┌────────────────────────────────────────────────┐
│  Access Mode:     ReadWriteOnce (RWO) only     │
│  Multi-node:      ❌ No (one node at a time)   │
│  Volume Types:    gp2, gp3, io1, io2, st1, sc1│
│  Max Size:        16 TiB                       │
│  Snapshots:       ✅ Yes                        │
│  Resize:          ✅ Yes (online)               │
│  Encryption:      ✅ Yes (AES-256, KMS)         │
│  Best For:        Databases, single-app storage│
└────────────────────────────────────────────────┘
```

### Installing AWS EBS CSI Driver

There are multiple ways to install it. Here's the most common approach using Helm:

```bash
# Method 1: Using eksctl (if you're on EKS)
# First, create an IAM OIDC provider for your cluster
eksctl utils associate-iam-oidc-provider \
  --cluster my-cluster \
  --approve

# Create an IAM role for the EBS CSI driver
eksctl create iamserviceaccount \
  --name ebs-csi-controller-sa \
  --namespace kube-system \
  --cluster my-cluster \
  --role-name AmazonEKS_EBS_CSI_DriverRole \
  --role-only \
  --attach-policy-arn arn:aws:iam::aws:policy/service-role/AmazonEBSCSIDriverPolicy \
  --approve

# Install the EBS CSI driver as an EKS add-on
eksctl create addon \
  --name aws-ebs-csi-driver \
  --cluster my-cluster \
  --service-account-role-arn arn:aws:iam::123456789012:role/AmazonEKS_EBS_CSI_DriverRole \
  --force
```

```bash
# Method 2: Using Helm (works on any K8s cluster on AWS)
helm repo add aws-ebs-csi-driver https://kubernetes-sigs.github.io/aws-ebs-csi-driver
helm repo update

helm install aws-ebs-csi-driver aws-ebs-csi-driver/aws-ebs-csi-driver \
  --namespace kube-system \
  --set controller.serviceAccount.annotations."eks\.amazonaws\.com/role-arn"="arn:aws:iam::123456789012:role/AmazonEKS_EBS_CSI_DriverRole"
```


### AWS EBS StorageClass Examples

```yaml
# gp3 — Best general purpose (recommended default)
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: ebs-gp3
  annotations:
    storageclass.kubernetes.io/is-default-class: "true"
provisioner: ebs.csi.aws.com
parameters:
  type: gp3                    # General Purpose SSD v3
  fsType: ext4
  encrypted: "true"            # Encrypt at rest
  # gp3 defaults: 3000 IOPS, 125 MiB/s throughput
  # You can customize:
  # iops: "5000"               # Up to 16,000 IOPS
  # throughput: "250"          # Up to 1,000 MiB/s
reclaimPolicy: Delete
volumeBindingMode: WaitForFirstConsumer
allowVolumeExpansion: true
---
# io2 — High performance (databases)
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: ebs-io2
provisioner: ebs.csi.aws.com
parameters:
  type: io2
  fsType: ext4
  encrypted: "true"
  iopsPerGB: "50"              # 50 IOPS per GB (max 64,000)
reclaimPolicy: Retain          # Keep data!
volumeBindingMode: WaitForFirstConsumer
allowVolumeExpansion: true
---
# st1 — Throughput optimized HDD (big data, logs)
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: ebs-st1
provisioner: ebs.csi.aws.com
parameters:
  type: st1                    # Throughput Optimized HDD
  fsType: ext4
reclaimPolicy: Delete
volumeBindingMode: WaitForFirstConsumer
```

### EBS Volume Snapshots

CSI drivers also support **volume snapshots** — point-in-time copies of your data:

```yaml
# First, create a VolumeSnapshotClass
apiVersion: snapshot.storage.k8s.io/v1
kind: VolumeSnapshotClass
metadata:
  name: ebs-snapshot-class
driver: ebs.csi.aws.com
deletionPolicy: Delete           # Delete snapshot when VolumeSnapshot is deleted
---
# Create a snapshot of an existing PVC
apiVersion: snapshot.storage.k8s.io/v1
kind: VolumeSnapshot
metadata:
  name: my-db-snapshot
spec:
  volumeSnapshotClassName: ebs-snapshot-class
  source:
    persistentVolumeClaimName: postgres-data   # PVC to snapshot
```

```bash
# Check snapshot status
kubectl get volumesnapshot
# NAME             READYTOUSE   SOURCEPVC       SNAPSHOTCONTENT          AGE
# my-db-snapshot   true         postgres-data   snapcontent-abc123       2m

# Restore from snapshot — create a new PVC from the snapshot
```

```yaml
# Restore a PVC from a snapshot
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: postgres-data-restored
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 20Gi                    # Must be >= snapshot size
  storageClassName: ebs-gp3
  dataSource:                          # ← This is the magic!
    name: my-db-snapshot               # Name of the VolumeSnapshot
    kind: VolumeSnapshot
    apiGroup: snapshot.storage.k8s.io
```

## ☁️ AWS EFS CSI Driver

### What Is It?

**Amazon EFS (Elastic File System)** is a managed NFS service. Unlike EBS, EFS supports **ReadWriteMany (RWX)** — multiple pods across multiple nodes can read/write simultaneously!

```
EBS vs EFS — When to Use What:

┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  AWS EBS (Block Storage)         AWS EFS (File Storage)      │
│  ──────────────────────         ─────────────────────        │
│                                                              │
│  📀 Like a USB hard drive       📂 Like a shared network     │
│     attached to ONE computer       drive everyone can access  │
│                                                              │
│  ✅ RWO (single node)           ✅ RWX (multi-node!)          │
│  ✅ Very fast (SSD)             ⚡ Moderate speed              │
│  ✅ Fixed size (20Gi, 100Gi)    ✅ Auto-scales (no size limit)│
│  💰 Pay per provisioned GB     💰 Pay per used GB            │
│  🔧 Zone-specific              🌐 Multi-AZ (regional)        │
│                                                              │
│  Best for:                      Best for:                     │
│  - Databases                    - Shared file uploads         │
│  - Single-instance apps         - CMS content                 │
│  - High IOPS workloads          - ML training data            │
│                                 - Multi-pod shared storage    │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### EFS StorageClass and Usage

```yaml
# First, you need an EFS file system. Create it in AWS Console or CLI:
# aws efs create-file-system --performance-mode generalPurpose --throughput-mode bursting

# StorageClass for EFS (Dynamic Provisioning)
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: efs-sc
provisioner: efs.csi.aws.com
parameters:
  provisioningMode: efs-ap          # Use EFS Access Points
  fileSystemId: fs-0123456789abcdef # Your EFS file system ID
  directoryPerms: "700"             # Permissions for the created directory
  gidRangeStart: "1000"             # Group ID range
  gidRangeEnd: "2000"
  basePath: "/dynamic_provisioning" # Base path in EFS
reclaimPolicy: Delete
---
# PVC using EFS (notice: ReadWriteMany!)
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: shared-uploads
spec:
  accessModes:
    - ReadWriteMany                  # ← Multiple nodes can read/write!
  resources:
    requests:
      storage: 5Gi                   # Note: EFS ignores size (auto-scales)
  storageClassName: efs-sc
---
# Deployment with multiple replicas sharing the same EFS volume
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
spec:
  replicas: 3                        # All 3 pods share the same storage!
  selector:
    matchLabels:
      app: web-app
  template:
    metadata:
      labels:
        app: web-app
    spec:
      containers:
        - name: web
          image: nginx:latest
          volumeMounts:
            - name: shared-storage
              mountPath: /usr/share/nginx/html/uploads
      volumes:
        - name: shared-storage
          persistentVolumeClaim:
            claimName: shared-uploads   # All pods mount the same PVC!
```

```
How EFS Works with Multiple Pods:

┌──────────┐     ┌──────────┐     ┌──────────┐
│  Node 1  │     │  Node 2  │     │  Node 3  │
│          │     │          │     │          │
│ ┌──────┐ │     │ ┌──────┐ │     │ ┌──────┐ │
│ │Pod-0 │ │     │ │Pod-1 │ │     │ │Pod-2 │ │
│ │      │ │     │ │      │ │     │ │      │ │
│ │/data │ │     │ │/data │ │     │ │/data │ │
│ └──┬───┘ │     │ └──┬───┘ │     │ └──┬───┘ │
│    │     │     │    │     │     │    │     │
└────┼─────┘     └────┼─────┘     └────┼─────┘
     │                │                │
     └────────────────┼────────────────┘
                      │
              ┌───────┴────────┐

              │                │
              │ All pods see   │
              │ the SAME files │
              │ simultaneously │
              └────────────────┘
```

## ☁️ Azure Disk CSI Driver

### What Is It?

**Azure Disk CSI Driver** lets Kubernetes use Azure Managed Disks as persistent storage. It's similar to AWS EBS — block storage attached to a single node.

```
Azure Disk Quick Facts:
┌────────────────────────────────────────────────────┐
│  Access Mode:     ReadWriteOnce (RWO)              │
│  Disk Types:      Standard_LRS, Standard_ZRS,      │
│                   Premium_LRS, Premium_ZRS,         │
│                   StandardSSD_LRS, StandardSSD_ZRS, │
│                   UltraSSD_LRS                      │
│  Max Size:        64 TiB (Ultra), 32 TiB (others)  │
│  Snapshots:       ✅ Yes                            │
│  Resize:          ✅ Yes (online)                   │
│  Encryption:      ✅ Yes (SSE, CMK)                 │
│  Best For:        Databases, single-app storage     │
└────────────────────────────────────────────────────┘
```

### Installation

On AKS (Azure Kubernetes Service), the Azure Disk CSI driver comes **pre-installed**! You don't need to install it manually.

```bash
# Check if it's already installed (on AKS)
kubectl get csidriver
# NAME                  ATTACHREQUIRED   PODINFOONMOUNT   AGE
# disk.csi.azure.com    true             false            30d
# file.csi.azure.com    false            true             30d

# If not on AKS, install via Helm:
helm repo add azuredisk-csi-driver https://raw.githubusercontent.com/kubernetes-sigs/azuredisk-csi-driver/master/charts
helm install azuredisk-csi-driver azuredisk-csi-driver/azuredisk-csi-driver \
  --namespace kube-system
```

### Azure Disk StorageClass Examples

```yaml
# Standard SSD (good default)
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: azure-standard-ssd

  skuName: StandardSSD_LRS         # Standard SSD, locally redundant
  fsType: ext4

---
# Premium SSD (high performance)
apiVersion: storage.k8s.io/v1

  name: azure-premium-ssd
provisioner: disk.csi.azure.com
parameters:
  skuName: Premium_LRS             # Premium SSD
  fsType: ext4
  cachingMode: ReadOnly            # Host caching (None, ReadOnly, ReadWrite)

allowVolumeExpansion: true
---
# Zone-redundant Premium SSD (high availability)
apiVersion: storage.k8s.io/v1

  name: azure-premium-zrs
provisioner: disk.csi.azure.com
parameters:
  skuName: Premium_ZRS             # Premium SSD, zone-redundant storage!
  fsType: ext4                     # Data replicated across 3 availability zones
reclaimPolicy: Retain
volumeBindingMode: WaitForFirstConsumer
allowVolumeExpansion: true
```

### Azure Files (Shared Storage — like AWS EFS)

Azure also has **Azure Files** for shared storage (ReadWriteMany):

```yaml
# Azure Files StorageClass (for shared storage)
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: azure-files
provisioner: file.csi.azure.com
parameters:
  skuName: Standard_LRS
  protocol: smb                    # SMB protocol (or nfs for NFS)
reclaimPolicy: Delete
volumeBindingMode: Immediate       # Azure Files doesn't have zone issues
allowVolumeExpansion: true
mountOptions:
  - dir_mode=0777
  - file_mode=0777
  - uid=0
  - gid=0
```


```yaml

  type: pd-ssd

allowVolumeExpansion: true
```

## 🔍 How to Check and Debug CSI Drivers

```bash
# ─── Check installed CSI drivers ───
kubectl get csidriver
# NAME                  ATTACHREQUIRED   PODINFOONMOUNT   STORAGECAPACITY   AGE
# ebs.csi.aws.com       true             false            false             30d
# efs.csi.aws.com       false            false            false             30d

# ─── Check CSI driver pods ───
kubectl get pods -n kube-system | grep csi
# ebs-csi-controller-xxx-yyy   6/6     Running   0   30d
# ebs-csi-node-abc             3/3     Running   0   30d
# efs-csi-controller-xxx       3/3     Running   0   30d
# efs-csi-node-def             3/3     Running   0   30d

# ─── Check CSI driver logs (for debugging) ───
# Controller logs (provisioning, attaching)
kubectl logs -n kube-system deployment/ebs-csi-controller -c csi-provisioner
kubectl logs -n kube-system deployment/ebs-csi-controller -c ebs-plugin

# Node logs (mounting, formatting)
kubectl logs -n kube-system daemonset/ebs-csi-node -c ebs-plugin

# ─── Check CSINode objects ───
kubectl get csinodes
# NAME       DRIVERS   AGE
# node-1     2         30d
# node-2     2         30d

kubectl describe csinode node-1
# Shows which CSI drivers are registered on each node

# ─── Check VolumeAttachments ───
kubectl get volumeattachments
# Shows which volumes are attached to which nodes

# ─── Common debug flow ───
# PVC stuck in Pending?
kubectl describe pvc my-pvc          # Check events
kubectl get sc                        # StorageClass exists?
kubectl get pods -n kube-system | grep csi  # CSI driver running?
kubectl logs -n kube-system deployment/ebs-csi-controller -c csi-provisioner  # Check logs
```

```

│ Problem                       │ Fix                                      │
├────────────────────────────────┼─────────────────────────────────────────┤
│ PVC stuck in Pending           │ 1. Check StorageClass exists             │
│                                │ 2. Check CSI controller is running       │
```

## 📝 Key Takeaways

```
1. CSI = Container Storage Interface
   A standard plugin system so Kubernetes can work with ANY storage provider
   through a common interface.

2. CSI replaced the old "in-tree" plugins
   Storage drivers are now independent, developed by storage vendors,
   and don't require Kubernetes code changes.

3. Every CSI driver has two parts:
   - Controller (Deployment) → Creates/deletes/attaches volumes
   - Node Plugin (DaemonSet) → Mounts/unmounts volumes on each node

4. Cloud providers:
   - AWS: EBS (block/RWO) + EFS (file/RWX)
   - Azure: Azure Disk (block/RWO) + Azure Files (file/RWX)
   - GCP: GCE PD (block/RWO) + Filestore (file/RWX)

5. For RWO (single node): Use block storage (EBS, Azure Disk, GCE PD)
   For RWX (multi node): Use file storage (EFS, Azure Files, Filestore, NFS)

6. CSI drivers also support:
   - Volume snapshots (point-in-time backup)
   - Volume cloning (create PVC from existing PVC)
   - Online resize (increase PVC size without downtime)
   - Encryption (at rest)

7. On managed Kubernetes (EKS, AKS, GKE):
   - AKS & GKE: CSI drivers pre-installed
   - EKS: Install as add-on (not pre-installed)
```
