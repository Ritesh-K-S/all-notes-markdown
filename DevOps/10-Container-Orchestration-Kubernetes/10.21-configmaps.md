# 10.21 — ConfigMaps — Externalizing Configuration

> **One Line Answer:** A **ConfigMap** is a Kubernetes object that stores **non-sensitive configuration data** (like database hostnames, feature flags, config files, or environment variables) as **key-value pairs** — so you can keep your configuration **separate from your container image** and change settings without rebuilding or redeploying your application.

## 🤔 Why Do We Need ConfigMaps?

### The Problem: Hardcoded Configuration

Imagine you have an app with configuration hardcoded inside it:

```
The Problem — Configuration Inside the Container Image:

┌────────────────────────────────────────────────────────┐
│  Dockerfile                                            │
│                                                        │
│  FROM node:20                                          │
│  COPY app.js /app/                                     │
│  # Config is HARDCODED in the code:                    │
│  # const DB_HOST = "db-prod.example.com"               │
│  # const DB_PORT = "5432"                              │
│  # const LOG_LEVEL = "info"                            │
│  # const FEATURE_FLAG = "true"                         │
│  CMD ["node", "/app/app.js"]                           │
└────────────────────────────────────────────────────────┘

Problems:
❌ Change DB_HOST? → Must rebuild the image!
❌ Different config for dev/staging/prod? → Need 3 different images!
❌ Quick config change? → Full CI/CD pipeline! (Build → Push → Deploy)
❌ Config mixed with code = messy and error-prone
```

### The Solution: ConfigMaps

```
The Solution — Configuration Outside the Container Image:

┌─────────────────────┐    ┌──────────────────────┐
│  Container Image    │    │  ConfigMap           │
│                     │    │                      │
│  FROM node:20       │    │  DB_HOST: db-prod... │
│  COPY app.js /app/  │    │  DB_PORT: 5432       │
│  CMD ["node", ...]  │    │  LOG_LEVEL: info     │
│                     │    │  FEATURE_FLAG: true   │
│  (No config here!)  │    │                      │
│  (Same image for    │    │  (Config lives here, │
│   ALL environments!)│    │   separate from code)│
└─────────────────────┘    └──────────────────────┘

Benefits:
✅ Change config? → Update ConfigMap, restart Pod (no rebuild!)
✅ Same image for dev/staging/prod — just different ConfigMaps!
✅ Quick changes — no CI/CD pipeline needed
✅ Clean separation of code and configuration
```

### The 12-Factor App Principle

This follows the famous **12-Factor App** methodology — specifically **Factor III: Config**:

```
"Store config in the environment"

Your app should read configuration from:
✅ Environment variables
✅ Config files mounted into the container
✅ NOT from hardcoded values in the code

ConfigMaps are Kubernetes' way of implementing this principle!
```

## 🧠 What Can You Store in a ConfigMap?

```
ConfigMap Can Store:

┌───────────────────────────────────────────────────────────────┐
│                                                               │
│  1. Simple Key-Value Pairs                                    │
│     DB_HOST=db.example.com                                    │
│     DB_PORT=5432                                              │
│     LOG_LEVEL=info                                            │
│                                                               │
│  2. Entire Configuration Files                                │
│     nginx.conf = "server { listen 80; ... }"                  │
│     app.properties = "spring.datasource.url=jdbc:..."         │
│     config.json = '{"port": 8080, "debug": false}'            │
│                                                               │
│  3. Multi-line Text                                           │
│     game.properties = |                                       │
│       enemies=aliens,monsters                                 │
│       lives=3                                                 │
│       level.initial=easy                                      │
│                                                               │
│  ⚠️  What NOT to store:                                       │
│     ❌ Passwords, API keys, tokens → Use Secrets instead!     │
│     ❌ Binary files (larger than 1MB) → Use Volumes           │
│     📌 ConfigMap max size = 1 MiB (1,048,576 bytes)          │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

## 🏗️ Creating ConfigMaps

There are multiple ways to create ConfigMaps. Let's learn all of them!

### Method 1: From a YAML File (Most Common)

```yaml
# configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
  namespace: default
data:                              # ← key-value pairs go here
  DB_HOST: "db.example.com"
  DB_PORT: "5432"
  LOG_LEVEL: "info"
  FEATURE_NEW_UI: "true"
  APP_COLOR: "blue"
```

```bash
kubectl apply -f configmap.yaml

# Verify
kubectl get configmap app-config
# NAME         DATA   AGE
# app-config   5      5s

kubectl describe configmap app-config
# Name:         app-config
# Namespace:    default
# Data
# ====
# DB_HOST:
# ----
# db.example.com
# DB_PORT:
# ----
# 5432
# LOG_LEVEL:
# ----
# info
# ...
```

Let's understand the YAML:

```
apiVersion: v1                    ← ConfigMap is a core resource
kind: ConfigMap                   ← Resource type
metadata:
  name: app-config                ← Name you'll reference later
  namespace: default              ← Namespace (ConfigMaps are namespaced!)
data:                             ← The actual configuration data
  DB_HOST: "db.example.com"      ← key: value (ALL values must be strings!)
  DB_PORT: "5432"                 ← Even numbers must be quoted as strings!
  LOG_LEVEL: "info"
  FEATURE_NEW_UI: "true"          ← Even booleans must be strings!
```

> **Important:** ALL values in `data` must be **strings**! Not numbers, not booleans — strings! That's why `"5432"` and `"true"` are quoted.

### Method 2: From the Command Line (Quick & Easy)

```bash
# Create ConfigMap with literal key-value pairs
kubectl create configmap app-config \
  --from-literal=DB_HOST=db.example.com \
  --from-literal=DB_PORT=5432 \
  --from-literal=LOG_LEVEL=info

# Verify
kubectl get configmap app-config -o yaml
```

This is equivalent to the YAML above but faster for quick creation.

### Method 3: From a File

Got an existing config file? You can create a ConfigMap from it directly!

```bash
# Let's say you have a config file:
cat > app.properties << 'EOF'
database.host=db.example.com
database.port=5432
database.name=myapp
log.level=info
feature.new-ui=true
EOF

# Create ConfigMap from the file
kubectl create configmap app-config --from-file=app.properties

# The ENTIRE file becomes ONE key in the ConfigMap:
# Key = filename (app.properties)
# Value = file contents
```

```bash
# You can also specify a custom key name:
kubectl create configmap app-config --from-file=my-config=app.properties
# Key = my-config (instead of app.properties)
# Value = file contents
```

### Method 4: From a Directory

```bash
# If you have multiple config files in a directory:
mkdir config-files
echo "server.port=8080" > config-files/server.properties
echo "log.level=debug" > config-files/logging.properties

# Create ConfigMap from all files in the directory
kubectl create configmap app-config --from-dir=config-files/

# Each file becomes a key:
# server.properties → "server.port=8080"
# logging.properties → "log.level=debug"
```

### Method 5: From an .env File

```bash
# If you have a .env file:
cat > app.env << 'EOF'
DB_HOST=db.example.com
DB_PORT=5432
LOG_LEVEL=info
EOF

# Create ConfigMap from env file
kubectl create configmap app-config --from-env-file=app.env

# Each line becomes a separate key-value pair in the ConfigMap!
# (Unlike --from-file which stores the whole file as one key)
```

### YAML with Multi-line Configuration Files

This is super useful when you need to store entire config files:

```yaml
# configmap-with-files.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: nginx-config
data:
  # Simple key-value pairs
  WORKER_PROCESSES: "4"
  
  # Entire nginx.conf file as a multi-line value
  nginx.conf: |
    server {
        listen 80;
        server_name example.com;
        
        location / {
            proxy_pass http://backend:8080;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
        }
        
        location /static {
            root /usr/share/nginx/html;
            expires 30d;
        }
    }
  
  # Another config file
  mime.types: |
    types {
        text/html  html;
        text/css   css;
        application/javascript js;
    }
```

```
The | symbol in YAML = "multi-line string, keep newlines"
                       Everything below it (indented) is the value

  nginx.conf: |              ← Key is "nginx.conf"
    server {                 ← This entire block
        listen 80;           ← is the value
        ...                  ← (preserving newlines)
    }                        ← End of value (next key starts at same indent level)
```

## 🔌 Using ConfigMaps in Pods

Now the important part — how do you actually USE the configuration in your Pods?

There are **three main ways**:

```
Three Ways to Use ConfigMaps in Pods:

┌───────────────────────────────────────────────────────────┐
│                                                           │
│  1. Environment Variables                                 │
│     → Inject individual keys as env vars                  │
│     → Or inject ALL keys at once                          │
│                                                           │
│  2. Volume Mounts (Files)                                 │
│     → Mount ConfigMap as files in the container           │
│     → Each key becomes a file, value = file contents      │
│                                                           │
│  3. Command-Line Arguments                                │
│     → Use ConfigMap values in container command/args       │
│                                                           │
└───────────────────────────────────────────────────────────┘
```

### Way 1: Environment Variables — Individual Keys

```yaml
# pod-env-specific.yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
    - name: my-app
      image: my-app:latest
      env:
        - name: DATABASE_HOST              # Env var name in the container
          valueFrom:
            configMapKeyRef:
              name: app-config             # ConfigMap name
              key: DB_HOST                 # Key in the ConfigMap
        - name: DATABASE_PORT
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: DB_PORT
        - name: LOG_LEVEL
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: LOG_LEVEL
```

```bash
# Inside the container:
echo $DATABASE_HOST    # → db.example.com
echo $DATABASE_PORT    # → 5432
echo $LOG_LEVEL        # → info
```

Let's break it down:

```
env:
  - name: DATABASE_HOST           ← Name of the env var INSIDE the container
    valueFrom:                    ← "Get the value from..."
      configMapKeyRef:            ← "...a ConfigMap"
        name: app-config          ← Which ConfigMap?
        key: DB_HOST              ← Which key in that ConfigMap?

So: ConfigMap "app-config" → key "DB_HOST" → value "db.example.com"
    Gets injected as env var "DATABASE_HOST" in the container
```

### Way 2: Environment Variables — ALL Keys at Once

Instead of mapping each key individually, inject everything:

```yaml
# pod-env-all.yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
    - name: my-app
      image: my-app:latest
      envFrom:                            # ← "envFrom" instead of "env"
        - configMapRef:
            name: app-config              # Inject ALL keys from this ConfigMap
```

```bash
# Inside the container — ALL keys become env vars:
echo $DB_HOST          # → db.example.com
echo $DB_PORT          # → 5432
echo $LOG_LEVEL        # → info
echo $FEATURE_NEW_UI   # → true
echo $APP_COLOR        # → blue

# The env var names match the ConfigMap keys exactly!
```

You can also add a **prefix**:

```yaml
envFrom:
  - configMapRef:
      name: app-config
    prefix: APP_                          # Add prefix to all env var names
```

```bash
# Now env vars have the APP_ prefix:
echo $APP_DB_HOST       # → db.example.com
echo $APP_DB_PORT       # → 5432
echo $APP_LOG_LEVEL     # → info
```

### Way 3: Volume Mount — Mount as Files

This is the best approach for **configuration files** (nginx.conf, app.properties, etc.):

```yaml
# pod-volume-mount.yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  containers:
    - name: nginx
      image: nginx:latest
      volumeMounts:
        - name: config-volume             # Must match volumes[].name
          mountPath: /etc/nginx/conf.d     # Directory in container
          readOnly: true                   # Good practice: read-only
  volumes:
    - name: config-volume                 # Volume name
      configMap:
        name: nginx-config                # ConfigMap name
```

```
What happens when you mount a ConfigMap as a volume:

ConfigMap "nginx-config" has:
  - Key: "nginx.conf"     → Value: "server { listen 80; ... }"
  - Key: "mime.types"     → Value: "types { text/html html; ... }"
  - Key: "WORKER_PROCESSES" → Value: "4"

After mounting at /etc/nginx/conf.d:

┌──────────────────────────────────────┐
│  Container File System               │
│                                      │
│  /etc/nginx/conf.d/                  │
│  ├── nginx.conf          ← File!    │
│  │   (contents: "server { ... }")   │
│  ├── mime.types          ← File!    │
│  │   (contents: "types { ... }")    │
│  └── WORKER_PROCESSES    ← File!    │
│      (contents: "4")                │
│                                      │
│  Each KEY becomes a FILENAME         │
│  Each VALUE becomes the FILE CONTENT │
└──────────────────────────────────────┘
```

### Mount Specific Keys Only

Don't want ALL keys as files? You can select specific ones:

```yaml
# Mount only specific keys
apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  containers:
    - name: nginx
      image: nginx:latest
      volumeMounts:
        - name: config-volume
          mountPath: /etc/nginx/conf.d
  volumes:
    - name: config-volume
      configMap:
        name: nginx-config
        items:                            # ← Select specific keys
          - key: nginx.conf               # Key from ConfigMap
            path: default.conf            # Filename in the mount directory
```

```
Result:
  /etc/nginx/conf.d/
  └── default.conf      ← Only this file (renamed from "nginx.conf" key)

  Other keys (mime.types, WORKER_PROCESSES) are NOT mounted
```

### Mount a Single File (Without Overwriting the Directory)

By default, mounting a ConfigMap **replaces the entire directory**. To mount just a single file without overwriting other files in the directory, use `subPath`:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  containers:
    - name: nginx
      image: nginx:latest
      volumeMounts:
        - name: config-volume
          mountPath: /etc/nginx/nginx.conf    # ← Exact file path!
          subPath: nginx.conf                  # ← Key from ConfigMap
  volumes:
    - name: config-volume
      configMap:
        name: nginx-config
```

```
Without subPath:
  mountPath: /etc/nginx/conf.d     → REPLACES entire directory!
  /etc/nginx/conf.d/               ← ONLY ConfigMap files here now
  └── nginx.conf                    (original files are GONE!)

With subPath:
  mountPath: /etc/nginx/nginx.conf → Only this ONE file is affected
  subPath: nginx.conf
  /etc/nginx/                       ← Other files are PRESERVED!
  ├── nginx.conf                    ← From ConfigMap
  ├── mime.types                    ← Original file (still here!)
  └── conf.d/                       ← Original directory (still here!)
  
⚠️  Important: subPath volumes do NOT receive automatic updates!
    If you update the ConfigMap, the file won't change until Pod restart.
```

### Way 4: Command-Line Arguments

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
    - name: my-app
      image: my-app:latest
      command: ["./start.sh"]
      args:
        - "--log-level=$(LOG_LEVEL)"       # Use env var in args
        - "--port=$(DB_PORT)"
      env:
        - name: LOG_LEVEL
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: LOG_LEVEL
        - name: DB_PORT
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: DB_PORT
```

## 🔄 Updating ConfigMaps

### How to Update

```bash
# Method 1: Edit directly
kubectl edit configmap app-config
# Opens in your default editor — make changes, save, done!

# Method 2: Replace from file
kubectl apply -f configmap.yaml    # After editing the YAML file

# Method 3: Patch specific values
kubectl patch configmap app-config -p '{"data":{"LOG_LEVEL":"debug"}}'

# Method 4: Replace entirely
kubectl create configmap app-config \
  --from-literal=DB_HOST=new-db.example.com \
  --from-literal=DB_PORT=5432 \
  --dry-run=client -o yaml | kubectl apply -f -
```

### What Happens When You Update a ConfigMap?

This is **super important** to understand:

```
ConfigMap Update Behavior:

┌──────────────────────────────────────────────────────────────────────┐
│                                                                      │
│  Method 1: Environment Variables (env / envFrom)                     │
│  ──────────────────────────────────                                  │
│  ❌ NOT automatically updated!                                       │
│  Env vars are set when the container STARTS                          │
│  Changing ConfigMap does NOT change running container's env vars     │
│  🔧 Fix: Restart the Pod (delete and recreate)                      │
│                                                                      │
│  kubectl rollout restart deployment my-app                           │
│                                                                      │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  Method 2: Volume Mount (WITHOUT subPath)                            │
│  ─────────────────────────────────────────                           │
│  ✅ Automatically updated! (eventually)                              │
│  Kubernetes syncs changes to mounted files                           │
│  ⏱️  Update delay: up to ~60-90 seconds (kubelet sync period)        │
│  📌 Your app needs to watch for file changes or re-read on demand   │
│                                                                      │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  Method 3: Volume Mount (WITH subPath)                               │
│  ──────────────────────────────────────                              │
│  ❌ NOT automatically updated!                                       │
│  subPath mounts are static — they don't receive updates             │
│  🔧 Fix: Restart the Pod                                            │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

### Force Rolling Update on ConfigMap Change

A common pattern is to trigger a Deployment rollout when a ConfigMap changes:

```bash
# Simple: Restart the deployment
kubectl rollout restart deployment my-app

# The pods will be recreated with the new ConfigMap values
```

A smarter approach — add a hash of the ConfigMap to the Pod template so Kubernetes automatically detects the change:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  template:
    metadata:
      annotations:
        # Change this value when ConfigMap changes to trigger rollout
        configmap-hash: "abc123"    # Update this manually or via CI/CD
    spec:
      containers:
        - name: my-app
          image: my-app:latest
          envFrom:
            - configMapRef:
                name: app-config
```

If you're using **Helm**, it handles this automatically:

```yaml
# In Helm templates:
spec:
  template:
    metadata:
      annotations:
        checksum/config: {{ include (print $.Template.BasePath "/configmap.yaml") . | sha256sum }}
```

### Example 1: Web Application with Multiple Environments

```yaml
# configmap-dev.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
  namespace: dev

  ENV: "development"
  DB_HOST: "db-dev.internal"

apiVersion: v1

  name: app-config
```

### Example 2: Nginx Reverse Proxy with ConfigMap

```yaml
# nginx-configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: nginx-config
data:
  default.conf: |
    upstream backend {
        server backend-svc:8080;
    }
    
    server {
        listen 80;
        server_name example.com;
        
        # Redirect HTTP to HTTPS
        return 301 https://$host$request_uri;
    }
    
    server {
        listen 443 ssl;
        server_name example.com;
        
        ssl_certificate /etc/nginx/ssl/tls.crt;
        ssl_certificate_key /etc/nginx/ssl/tls.key;
        
        location / {
            proxy_pass http://backend;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
        }
        
        location /health {
            return 200 'OK';
            add_header Content-Type text/plain;
        }
    }
---
# nginx-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
spec:
  replicas: 2
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
        - name: nginx
          image: nginx:latest
          ports:
            - containerPort: 80
            - containerPort: 443
          volumeMounts:
            - name: nginx-config
              mountPath: /etc/nginx/conf.d    # Mount config files here
              readOnly: true
      volumes:
        - name: nginx-config
          configMap:
            name: nginx-config
```

```yaml
          ddl-auto: validate
        show-sql: false
    logging:
      level:
        root: INFO
        com.myapp: DEBUG
    management:
      endpoints:
        web:
          exposure:
            include: health,info,metrics
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: spring-app
spec:
  replicas: 2
  selector:
    matchLabels:
      app: spring-app
  template:
    metadata:
      labels:
        app: spring-app
    spec:
      containers:
        - name: app
          image: my-spring-app:latest
          ports:
            - containerPort: 8080
          volumeMounts:
            - name: config
              mountPath: /app/config        # Spring Boot auto-reads from here
              readOnly: true
      volumes:
        - name: config
          configMap:
            name: spring-config
```

### Example 4: Using Multiple ConfigMaps in One Pod

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
    - name: my-app
      image: my-app:latest
      
      # Environment variables from multiple ConfigMaps
      envFrom:
        - configMapRef:
            name: db-config               # All keys from db-config
          prefix: DB_                      # Prefix with DB_
        - configMapRef:
            name: app-config              # All keys from app-config
          prefix: APP_

      # Also mount files from another ConfigMap
      volumeMounts:
        - name: nginx-conf
          mountPath: /etc/nginx/conf.d
          readOnly: true
        - name: app-properties
          mountPath: /app/config
          readOnly: true

  volumes:
    - name: nginx-conf
      configMap:
        name: nginx-config
    - name: app-properties
      configMap:
        name: spring-config
```

## 🔒 ConfigMap vs Secret — What's the Difference?

```
┌────────────────────────────────────────────────────────────────────┐
│                                                                    │
│  ConfigMap                           Secret                        │
│  ─────────                           ──────                        │
│                                                                    │
│  📋 Non-sensitive configuration      🔒 Sensitive data              │
│                                                                    │
│  Stores:                             Stores:                        │
│  - Database hostnames               - Passwords                     │
│  - Port numbers                     - API keys                      │
│  - Feature flags                    - TLS certificates              │
│  - Config files                     - Tokens                        │
│  - Log levels                       - SSH keys                      │
│                                                                    │
│  Stored as:                          Stored as:                      │
│  Plain text                          Base64 encoded                  │
│  (NOT encrypted)                     (NOT encrypted by default!)     │
│                                                                    │
│  Size limit: 1 MiB                   Size limit: 1 MiB              │

│                                                                    │
```

## ♻️ Immutable ConfigMaps

Starting from Kubernetes 1.21, you can make ConfigMaps **immutable** — meaning they can NEVER be changed after creation:

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config-v2
data:
  DB_HOST: "db.example.com"
  LOG_LEVEL: "info"
immutable: true                   # ← This ConfigMap can never be modified!
```

```
Why use immutable ConfigMaps?

✅ Performance: Kubernetes doesn't need to watch for changes
   (reduces load on API server in large clusters)
   
✅ Safety: Prevents accidental changes to critical config

✅ Versioning pattern: Create new ConfigMaps with version suffix
   app-config-v1 → app-config-v2 → app-config-v3
   
⚠️  To "update": Create a new ConfigMap and update Pod to reference it

kubectl delete configmap app-config-v1  # Delete old one (optional)
```

## 🔍 Useful kubectl Commands for ConfigMaps

```bash
# ─── Create ───
kubectl create configmap my-config --from-literal=key=value
kubectl create configmap my-config --from-file=config.properties
kubectl create configmap my-config --from-env-file=app.env
kubectl apply -f configmap.yaml

# ─── Read ───
kubectl get configmap                      # List all ConfigMaps
kubectl get configmap -A                   # All namespaces
kubectl get configmap my-config            # Specific ConfigMap
kubectl get configmap my-config -o yaml    # Full YAML output
kubectl describe configmap my-config       # Detailed view with events

# ─── View just the data ───
kubectl get configmap my-config -o jsonpath='{.data}'
kubectl get configmap my-config -o jsonpath='{.data.DB_HOST}'

# ─── Update ───
kubectl edit configmap my-config           # Edit in editor
kubectl patch configmap my-config -p '{"data":{"LOG_LEVEL":"debug"}}'

# ─── Delete ───
kubectl delete configmap my-config

# ─── Generate YAML (dry-run) ───
kubectl create configmap my-config \
  --from-literal=DB_HOST=localhost \
  --from-literal=DB_PORT=5432 \
  --dry-run=client -o yaml > configmap.yaml
# This creates the YAML file without actually creating the ConfigMap
# Great for generating YAML templates!

# ─── Check which Pods use a ConfigMap ───
kubectl get pods -o json | jq '.items[] | 
  select(.spec.volumes[]?.configMap.name == "my-config" or 
  .spec.containers[].envFrom[]?.configMapRef.name == "my-config") | 
  .metadata.name'
```

```
┌────────────────────────────────┬──────────────────────────────────────────┐
│ Problem                       │ Fix                                       │
├────────────────────────────────┼──────────────────────────────────────────┤
│ Pod can't start:              │ ConfigMap must exist BEFORE the Pod!      │

├────────────────────────────────┼──────────────────────────────────────────┤

│                               │ kubectl rollout restart deploy my-app     │

│                               │ Or check if using subPath (no auto-update)│
├────────────────────────────────┼──────────────────────────────────────────┤
│ Volume mount overwrites       │ Use subPath to mount single files         │
│ entire directory              │ without replacing the directory            │
├────────────────────────────────┼──────────────────────────────────────────┤
```

### Optional ConfigMap References

If a ConfigMap might not exist, you can make the reference optional:

```yaml
env:
  - name: DB_HOST
    valueFrom:
      configMapKeyRef:
        name: app-config
        key: DB_HOST
        optional: true              # ← Pod starts even if ConfigMap doesn't exist
                                    #    (env var just won't be set)
```

## 🎯 Best Practices

```
ConfigMap Best Practices:

1. ✅ Name ConfigMaps clearly
   Good: "postgres-config", "nginx-config", "frontend-config"
   Bad:  "config1", "my-config", "test"

2. ✅ Use namespaces
   Different environments = different namespaces = different ConfigMaps

3. ✅ Use volume mounts for config files
   Mount nginx.conf, application.yml as files (not env vars)

4. ✅ Use envFrom for simple key-value config
   Quick and easy for environment variables

5. ✅ Never store secrets in ConfigMaps
   Passwords, API keys, tokens → use Secrets!

6. ✅ Use immutable ConfigMaps for critical config
   Prevents accidental changes

7. ✅ Plan for updates
   Know that env vars need Pod restart
   Volume mounts (without subPath) auto-update

8. ✅ Keep ConfigMaps small
   Max 1 MiB — split large configs into multiple ConfigMaps

9. ✅ Use --dry-run=client -o yaml to generate YAML templates
   Don't write YAML from scratch!

10. ✅ Version your ConfigMaps in Git
    Treat them as code — review, track, audit
```

## 📝 Key Takeaways

```
1. ConfigMap = Kubernetes way to externalize non-sensitive configuration
   Keeps config SEPARATE from container images

2. Create from: YAML files, literal values, files, directories, .env files

3. Use in Pods via:
   - env (individual keys)
   - envFrom (all keys at once)
   - volume mounts (as files)
   - command arguments

4. Update behavior:
   - Env vars: NOT auto-updated → need Pod restart
   - Volume mounts: Auto-updated (~60-90s delay)
   - subPath mounts: NOT auto-updated → need Pod restart

5. ConfigMaps are namespaced (Pod and ConfigMap must be in same namespace)

6. Max size: 1 MiB

7. ALL values must be strings (use "5432" not 5432)

8. For sensitive data → use Secrets (next section!)

9. Same container image + different ConfigMaps = different environments
   (dev, staging, prod without rebuilding!)
```
