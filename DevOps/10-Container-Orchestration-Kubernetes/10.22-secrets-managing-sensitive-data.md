# 10.22 — Secrets — Managing Sensitive Data in Kubernetes

> **One Line Answer:** Kubernetes **Secrets** let you store and manage sensitive information — like passwords, API keys, tokens, and TLS certificates — **separately from your application code and Pod specs**, so you don't have to hardcode secrets into your YAML files or Docker images. They're base64-encoded (not encrypted by default!), and can be injected into Pods as environment variables or mounted as files.

---

## 🤔 Why Do We Need Secrets?

### The Problem: Hardcoded Sensitive Data

Imagine you have a web app that connects to a database. Without Secrets, you'd do something terrible like this:

```yaml
# ❌ BAD — NEVER do this!
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  template:
    spec:
      containers:
        - name: my-app
          image: my-app:v1
          env:
            - name: DB_PASSWORD
              value: "SuperSecret123!"     # 😱 Password in plain text!
            - name: API_KEY
              value: "sk-abc123xyz789"     # 😱 API key exposed!
```

**What's wrong with this?**

```
Problems with Hardcoded Secrets:

1. 🔓 Anyone with access to the YAML can see passwords
2. 📦 Secrets get committed to Git — exposed forever in history
3. 🔄 Changing a password means editing every Deployment YAML
4. 👀 kubectl get deployment -o yaml shows passwords in plain text
5. 🚫 No access control — everyone who can read the Deployment can see secrets
```

### The Solution: Kubernetes Secrets

```
How Secrets Fix This:

┌─────────────────────────────────┐
│          Kubernetes Secret       │
│  (Stored separately in etcd)    │
│                                  │
│  DB_PASSWORD = SuperSecret123!   │
│  API_KEY     = sk-abc123xyz789   │
└──────────────┬──────────────────┘
               │
               │ Referenced (not hardcoded!)
               │
┌──────────────▼──────────────────┐
│         Pod / Deployment         │
│                                  │
│  env:                            │
│    - name: DB_PASSWORD           │
│      valueFrom:                  │
│        secretKeyRef:             │
│          name: my-secret  ◄──── Points to Secret, not actual value
│          key: db-password        │
└──────────────────────────────────┘

Benefits:
✅ Secrets stored separately from app code
✅ Not committed to Git (only Secret name is in YAML)
✅ Change once, all Pods pick it up
✅ Can restrict who can read Secrets via RBAC
✅ Base64 encoded in API responses
```

## 📦 How Secrets Work Internally

Let's understand what happens behind the scenes:

```
Secret Lifecycle:

1. You create a Secret
   ┌──────────────────────────┐
   │  kubectl create secret    │
   │  or kubectl apply -f      │
   └────────────┬─────────────┘
                │
                ▼
2. API Server receives it
   ┌──────────────────────────┐
   │  Kubernetes API Server    │
   │  • Validates the Secret   │
   │  • Base64 encodes values  │
   └────────────┬─────────────┘
                │
                ▼
3. Stored in etcd
   ┌──────────────────────────┐
   │  etcd (key-value store)   │
   │  • Stored as base64       │
   │  • Can be encrypted at    │
   │    rest (if configured)   │
   └────────────┬─────────────┘
                │
                ▼
4. Pod requests the Secret
   ┌──────────────────────────┐
   │  kubelet on the Node      │
   │  • Fetches Secret from    │
   │    API Server              │
   │  • Mounts it in the Pod   │
   │    as env var or file      │
   └──────────────────────────┘
```

### Important: Base64 ≠ Encryption!

This is a **very common misconception**. Let me clear it up:

```
Base64 is NOT encryption. It's just ENCODING.

Original:     "MyPassword123"
Base64:       "TXlQYXNzd29yZDEyMw=="

Anyone can decode it:
$ echo "TXlQYXNzd29yZDEyMw==" | base64 --decode
MyPassword123

It's like writing a secret message in Pig Latin — 
anyone who knows the trick can read it!

🔐 For real security, you need:
   - Encryption at rest (EncryptionConfiguration)
   - External secret managers (Vault, AWS Secrets Manager)
   - RBAC to restrict who can read Secrets
```

```
├─────────────────────────────┼──────────────────────────────────────┤

└─────────────────────────────┴──────────────────────────────────────┘

Most Common: Opaque (you'll use this 90% of the time)
```

## ✏️ Creating Secrets — All Methods

### Method 1: Using `kubectl create secret` (Imperative — Command Line)

This is the quickest way, great for testing and development.

#### a) Create from literal values

```bash
# Syntax: kubectl create secret generic <name> --from-literal=<key>=<value>

kubectl create secret generic db-credentials \
  --from-literal=username=admin \
  --from-literal=password=SuperSecret123!
```

```
What this creates internally:

apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
type: Opaque
data:
  username: YWRtaW4=              # "admin" in base64
  password: U3VwZXJTZWNyZXQxMjMh  # "SuperSecret123!" in base64
```

#### b) Create from a file

```bash
# Let's say you have a config file with sensitive data
echo -n "SuperSecret123!" > ./password.txt
echo -n "admin" > ./username.txt

# Create secret from those files
kubectl create secret generic db-credentials \
  --from-file=password=./password.txt \
  --from-file=username=./username.txt
```

```
The key name becomes the filename (or what you specify before =)
The value becomes the file content.

Tip: Use -n with echo to avoid a trailing newline character!
     Without -n:  "SuperSecret123!\n" ← extra newline, causes bugs!
     With -n:     "SuperSecret123!"   ← clean value ✅
```

#### c) Create from an env file

```bash
# Create a .env file
cat <<EOF > ./secret.env
DB_HOST=mysql.example.com
DB_USER=admin
DB_PASS=SuperSecret123!
EOF

# Create secret from env file
kubectl create secret generic db-credentials \
  --from-env-file=./secret.env
```

### Method 2: Using a YAML File (Declarative — Recommended for Production)

#### a) With base64-encoded values (using `data`)

```yaml
# db-secret.yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
  namespace: default
type: Opaque
data:
  # Values MUST be base64-encoded
  username: YWRtaW4=                # echo -n "admin" | base64
  password: U3VwZXJTZWNyZXQxMjMh    # echo -n "SuperSecret123!" | base64
```

```bash
# How to base64 encode on the command line:
echo -n "admin" | base64
# Output: YWRtaW4=

echo -n "SuperSecret123!" | base64
# Output: U3VwZXJTZWNyZXQxMjMh

# Apply the secret
kubectl apply -f db-secret.yaml
```

#### b) With plain text values (using `stringData`) — MUCH Easier!

```yaml
# db-secret.yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
  namespace: default
type: Opaque
stringData:
  # Values in PLAIN TEXT — Kubernetes encodes them for you!
  username: admin
  password: SuperSecret123!
```

```
stringData vs data:

┌────────────┬──────────────────────────────────────────────┐
│  Field     │  Description                                 │
├────────────┼──────────────────────────────────────────────┤
│  data      │  Values must be base64-encoded by YOU        │
│  stringData│  Values in plain text, K8s encodes for you   │
└────────────┴──────────────────────────────────────────────┘

Both produce the same Secret in the cluster.
stringData is just a convenience — it gets converted to data
when stored in etcd.

⚠️ WARNING: If you use stringData in a YAML file and commit it
to Git, the password is still visible in plain text in the file!
```

```bash
# Apply it
kubectl apply -f db-secret.yaml
```

### Method 3: Creating TLS Secrets

```bash
# For HTTPS/TLS certificates
kubectl create secret tls my-tls-secret \
  --cert=path/to/tls.crt \
  --key=path/to/tls.key
```

```
This creates a Secret with two keys:
  - tls.crt → the certificate
  - tls.key → the private key

Used by Ingress controllers for HTTPS termination.
```

### Method 4: Creating Docker Registry Secrets

```bash
# For pulling images from private registries
kubectl create secret docker-registry my-registry-secret \
  --docker-server=https://index.docker.io/v1/ \
  --docker-username=myuser \
  --docker-password=mypassword \
  --docker-email=myemail@example.com
```

```bash
# Use it in a Pod:
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
    - name: my-app
      image: myregistry.com/my-private-image:v1
  imagePullSecrets:               # ← Tell K8s which secret has registry creds
    - name: my-registry-secret
```

```bash

```

```

====

```

### View the actual Secret values

```bash
# Get base64-encoded values
kubectl get secret db-credentials -o yaml
```

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
data:
  password: U3VwZXJTZWNyZXQxMjMh    # Still base64
  username: YWRtaW4=
```

```bash
# Decode a specific value
kubectl get secret db-credentials -o jsonpath='{.data.password}' | base64 --decode
# Output: SuperSecret123!

# Or decode all values at once with a neat trick
kubectl get secret db-credentials -o go-template='{{range $k,$v := .data}}{{$k}}: {{$v | base64decode}}{{"\n"}}{{end}}'
# Output:
# password: SuperSecret123!
# username: admin
```

### Delete a Secret

```bash
kubectl delete secret db-credentials
```

### Edit a Secret

```bash
# Opens in your default editor — values shown in base64
kubectl edit secret db-credentials
```

## 🚀 Using Secrets in Pods

There are **three main ways** to consume Secrets inside a Pod:

### Way 1: As Environment Variables (Individual Keys)

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
    - name: my-app
      image: my-app:v1
      env:
        - name: DB_USERNAME           # Name of the env var inside the container
          valueFrom:
            secretKeyRef:
              name: db-credentials    # Name of the Secret
              key: username           # Key inside the Secret
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: db-credentials
              key: password
```

```
Inside the container:

$ echo $DB_USERNAME
admin

$ echo $DB_PASSWORD
SuperSecret123!

The values are automatically decoded from base64!
```

### Way 2: As Environment Variables (All Keys at Once with envFrom)

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
    - name: my-app
      image: my-app:v1
      envFrom:
        - secretRef:
            name: db-credentials       # All keys become env vars
          prefix: DB_                   # Optional: adds prefix to all keys
```

```
Inside the container:

$ echo $DB_username    # key "username" becomes env var "DB_username"
admin

$ echo $DB_password    # key "password" becomes env var "DB_password"
SuperSecret123!

Note: The original key names from the Secret become the env var names.
With prefix "DB_", they become DB_username, DB_password.
```

### Way 3: As Files Mounted in a Volume

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
    - name: my-app
      image: my-app:v1
      volumeMounts:
        - name: secret-volume
          mountPath: /etc/secrets      # Directory where secrets will appear
          readOnly: true               # Always mount secrets as read-only!
  volumes:
    - name: secret-volume
      secret:
        secretName: db-credentials     # Name of the Secret
```

```
Inside the container:

$ ls /etc/secrets/
password  username

$ cat /etc/secrets/username
admin

$ cat /etc/secrets/password
SuperSecret123!

Each key becomes a FILE in the mount directory.
The filename = key name
The file content = decoded value
```

#### Mount only specific keys

```yaml
volumes:
  - name: secret-volume
    secret:
      secretName: db-credentials
      items:                          # Only mount specific keys
        - key: password               # Key from the Secret
          path: db-password.txt       # Filename in the mount directory
```

```
Inside the container:

$ ls /etc/secrets/
db-password.txt              # Only the specified key is mounted

$ cat /etc/secrets/db-password.txt
SuperSecret123!
```

#### Set file permissions

```yaml
volumes:
  - name: secret-volume
    secret:
      secretName: db-credentials
      defaultMode: 0400              # Read-only for owner (most secure)
```

## 🔄 Secret Updates — What Happens When You Change a Secret?

This is an important behavior you need to understand:

```
When you UPDATE a Secret:

┌──────────────────────────────────────────────────────────┐
│  Method of Consuming Secret  │  Auto-Updates?            │
├──────────────────────────────┼───────────────────────────┤
│  Environment Variables       │  ❌ NO — Pod must restart │
│  Volume Mounts               │  ✅ YES — Eventually      │
│                              │  (can take up to 60s +    │
│                              │   kubelet sync period)    │
└──────────────────────────────┴───────────────────────────┘

Why? Because env vars are set at container startup and
never change. But mounted files are updated by kubelet
periodically.
```

```bash
# To force a Pod to pick up new env-var-based Secrets:
kubectl rollout restart deployment my-app
```

## 🔒 Secret Security Best Practices

### 1. Enable Encryption at Rest

By default, Secrets are stored **unencrypted** in etcd. To fix this:

```yaml
# /etc/kubernetes/encryption-config.yaml (on the control plane)
apiVersion: apiserver.config.k8s.io/v1
kind: EncryptionConfiguration
resources:
  - resources:
      - secrets
    providers:
      - aescbc:                      # AES-CBC encryption
          keys:
            - name: key1
              secret: <base64-encoded-32-byte-key>
      - identity: {}                 # Fallback to read existing unencrypted secrets
```

```bash
# Then configure the API server to use it:
# Add to kube-apiserver flags:
--encryption-provider-config=/etc/kubernetes/encryption-config.yaml
```

### 2. Use RBAC to Restrict Access

```yaml
# Only allow specific ServiceAccount to read secrets
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: secret-reader
  namespace: default
rules:
  - apiGroups: [""]
    resources: ["secrets"]
    resourceNames: ["db-credentials"]     # Only this specific secret
    verbs: ["get"]                         # Only read, not list or watch
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-db-secrets
  namespace: default
subjects:
  - kind: ServiceAccount
    name: my-app-sa
    namespace: default
roleRef:
  kind: Role
  name: secret-reader
  apiGroup: rbac.authorization.k8s.io
```

### 3. Use External Secret Managers (Production Best Practice)

```
Instead of native K8s Secrets, use:

┌──────────────────────────────────────────────────────┐
│  Tool                    │  What It Does              │
├──────────────────────────┼────────────────────────────┤
│  HashiCorp Vault         │  Full-featured secret      │
│                          │  management (most popular) │
│  AWS Secrets Manager     │  AWS-native secrets        │
│  Azure Key Vault         │  Azure-native secrets      │
│  GCP Secret Manager      │  GCP-native secrets        │
│  External Secrets        │  K8s operator that syncs   │
│  Operator (ESO)          │  external secrets → K8s    │
│  Sealed Secrets          │  Encrypt secrets in Git    │
│  (Bitnami)               │  (GitOps-friendly)         │
└──────────────────────────┴────────────────────────────┘
```

### 4. Never Commit Secrets to Git

```bash
# Add to .gitignore
*-secret.yaml
*.secret
secrets/

# Better yet: use Sealed Secrets or External Secrets Operator
# so you CAN commit encrypted secrets to Git safely
```

### 5. Use `stringData` for Readability, but Be Careful

```
✅ Good for: local testing, quick experiments
❌ Bad for: committing to Git (values are in plain text)

Best practice: Use kubectl create secret, NOT YAML files with stringData,
for anything that might end up in version control.
```

## 🧪 Practical Example: Full Application with Secrets

Let's build a complete example — a web app connecting to MySQL using Secrets.

### Step 1: Create the Secret

```yaml
# mysql-secret.yaml
apiVersion: v1
kind: Secret
metadata:
  name: mysql-secret
  namespace: default
type: Opaque
stringData:
  MYSQL_ROOT_PASSWORD: rootpass123
  MYSQL_DATABASE: myappdb
  MYSQL_USER: appuser
  MYSQL_PASSWORD: apppass456
```

```bash
kubectl apply -f mysql-secret.yaml
```

### Step 2: Create MySQL Deployment using the Secret

```yaml
# mysql-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: mysql
spec:
  replicas: 1
  selector:
    matchLabels:
      app: mysql
  template:
    metadata:
      labels:
        app: mysql
    spec:
      containers:
        - name: mysql
          image: mysql:8.0
          ports:
            - containerPort: 3306
          envFrom:
            - secretRef:
                name: mysql-secret     # All keys injected as env vars
          volumeMounts:
            - name: mysql-storage
              mountPath: /var/lib/mysql
      volumes:
        - name: mysql-storage
          emptyDir: {}                  # Use PVC in production!
---
apiVersion: v1
kind: Service
metadata:
  name: mysql-service
spec:
  selector:
    app: mysql
  ports:
    - port: 3306
      targetPort: 3306
  type: ClusterIP
```

### Step 3: Create the App Deployment using individual Secret keys

```yaml
# app-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-web-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-web-app
  template:
    metadata:
      labels:
        app: my-web-app
    spec:
      containers:
        - name: my-web-app
          image: my-web-app:v1
          ports:
            - containerPort: 8080
          env:
            - name: DB_HOST
              value: mysql-service      # Not secret — plain value is fine
            - name: DB_PORT
              value: "3306"             # Not secret — plain value is fine
            - name: DB_NAME
              valueFrom:
                secretKeyRef:
                  name: mysql-secret
                  key: MYSQL_DATABASE
            - name: DB_USER
              valueFrom:
                secretKeyRef:
                  name: mysql-secret
                  key: MYSQL_USER
            - name: DB_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: mysql-secret
                  key: MYSQL_PASSWORD
```

```bash
kubectl apply -f mysql-deployment.yaml
kubectl apply -f app-deployment.yaml
```

### Step 4: Verify everything works

```bash
# Check the secrets
kubectl get secrets
kubectl describe secret mysql-secret

# Check the Pods
kubectl get pods

# Verify env vars inside the app Pod
kubectl exec -it <app-pod-name> -- env | grep DB_
# DB_HOST=mysql-service
# DB_PORT=3306
# DB_NAME=myappdb
# DB_USER=appuser
# DB_PASSWORD=apppass456
```

## 🔁 Immutable Secrets (Kubernetes v1.21+)

You can make a Secret **immutable** — meaning once created, it can never be changed. This is great for security and performance.

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: fixed-api-key
type: Opaque
stringData:
  api-key: "sk-abc123xyz789"
immutable: true                      # ← This makes it immutable!
```

```
Benefits of Immutable Secrets:

✅ Protects against accidental or malicious changes
✅ Better performance — kubelet doesn't need to watch for updates
✅ Reduces load on API server (no watch connections)

⚠️ To "update" an immutable Secret, you must:
   1. Delete the old Secret
   2. Create a new one
   3. Restart Pods that use it
```

## 🆚 Secrets vs ConfigMaps — When to Use What?

```
┌──────────────────┬────────────────────────┬─────────────────────────┐
│  Feature         │  ConfigMap             │  Secret                 │
├──────────────────┼────────────────────────┼─────────────────────────┤
│  Purpose         │  Non-sensitive config  │  Sensitive data         │
│  Data stored as  │  Plain text            │  Base64-encoded         │
│  Max size        │  1 MB                  │  1 MB                   │
│  Encryption      │  No                    │  Optional (at rest)     │
│  RBAC            │  Less restrictive      │  More restrictive       │
│  Use for         │  App config, feature   │  Passwords, API keys,   │
│                  │  flags, URLs           │  tokens, certificates   │
│  Can be mounted  │  ✅ Yes                │  ✅ Yes                 │
│  as volume       │                        │                         │
│  Can be env var  │  ✅ Yes                │  ✅ Yes                 │
│  Auto-update     │  ✅ Volume mounts      │  ✅ Volume mounts       │
│  (volumes)       │                        │                         │
└──────────────────┴────────────────────────┴─────────────────────────┘

Simple rule:
  - Is it a password, key, or token? → Secret
  - Is it a config value, URL, or setting? → ConfigMap
```

## ❓ Common Mistakes & Troubleshooting

### Mistake 1: Forgetting to base64 encode when using `data`

```yaml
# ❌ WRONG — plain text in data field
data:
  password: SuperSecret123!     # This is NOT valid base64!

# ✅ CORRECT — base64-encoded
data:
  password: U3VwZXJTZWNyZXQxMjMh

# Or just use stringData instead:
stringData:
  password: SuperSecret123!     # K8s encodes for you
```

### Mistake 2: Extra newline from `echo`

```bash
# ❌ WRONG — includes a trailing newline
echo "password123" | base64
# cGFzc3dvcmQxMjMK        ← includes \n

# ✅ CORRECT — no trailing newline
echo -n "password123" | base64
# cGFzc3dvcmQxMjM=        ← clean
```

### Mistake 3: Secret not found by Pod

```bash
# Check if the Secret exists in the same namespace as the Pod!
kubectl get secret db-credentials -n <namespace>

# Secrets are namespace-scoped!
# A Pod in namespace "production" can't access a Secret in namespace "default"
```

### Mistake 4: Thinking Secrets are truly secure by default

```
Remember:
- Base64 is encoding, NOT encryption
- Anyone with kubectl access can decode secrets
- Enable encryption at rest
- Use RBAC to restrict access
- Consider external secret managers for production
```

```bash
echo "dGV4dA==" | base64 --decode          # Decode
```

---

## 🎯 Key Takeaways

```
1. Secrets store sensitive data (passwords, keys, tokens) separately from Pod specs
2. They are base64-ENCODED, not encrypted — don't confuse the two!
3. Three ways to consume: env vars, envFrom, or volume mounts
4. Volume-mounted Secrets auto-update; env var Secrets need Pod restart
5. Always enable encryption at rest in production
6. Use RBAC to control who can read Secrets
7. For production: use external secret managers (Vault, AWS SM, etc.)
8. Never commit Secret YAML files to Git
9. Secrets are namespace-scoped — Pod and Secret must be in the same namespace
10. Use stringData for convenience, data for already-encoded values
```
