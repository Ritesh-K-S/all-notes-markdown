# 10.23 — Environment Variables, Volume Mounts, and Projections

> **One Line Answer:** Kubernetes gives you **three powerful ways** to pass configuration data into your containers — **Environment Variables** (key-value pairs available in the shell), **Volume Mounts** (files/directories injected into the container's filesystem), and **Projected Volumes** (a way to combine multiple sources like Secrets, ConfigMaps, and Pod info into a single mounted directory). Understanding when and how to use each one is essential for building well-configured, production-ready apps.

## 🤔 Why Do We Need Different Methods?

Different types of configuration data are best delivered in different ways:

```
Think of it like getting information in real life:

📋 Environment Variables = Sticky notes on your desk
   → Quick to read, simple key=value, great for small things
   → Example: DB_HOST=mysql-server, LOG_LEVEL=debug

📁 Volume Mounts = A folder full of documents
   → Complete files, configs, certificates
   → Example: nginx.conf, tls.crt, application.properties

📦 Projected Volumes = A combined binder from multiple sources
   → Merge Secrets + ConfigMaps + Pod info into one folder
   → Example: All config in /etc/app-config/ from different sources
```

```
When to Use What:

┌─────────────────────────┬────────────────────────────────────────────────┐
│  Method                 │  Best For                                      │
├─────────────────────────┼────────────────────────────────────────────────┤
│  Environment Variables  │  Simple values: URLs, ports, feature flags,    │
│                         │  database names, log levels                    │
│                         │                                                │
│  Volume Mounts          │  Config files: nginx.conf, app.properties,     │
│                         │  TLS certs, SSH keys, JSON/YAML configs        │
│                         │                                                │
│  Projected Volumes      │  Combining multiple config sources into one    │
│                         │  directory, or needing Pod identity info       │
└─────────────────────────┴────────────────────────────────────────────────┘
```

## 📋 Part 1: Environment Variables — The Basics

Environment variables are the simplest way to pass config to a container. Your app reads them using standard OS mechanisms.

### Method 1: Static Values (Hardcoded)

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
    - name: my-app
      image: my-app:v1
      env:
        - name: APP_ENV                # Name of the env var
          value: "production"          # Static value
        - name: LOG_LEVEL
          value: "info"
        - name: APP_PORT
          value: "8080"                # Always a string, even for numbers!
```

```
Inside the container:

$ echo $APP_ENV
production

$ echo $LOG_LEVEL
info

$ echo $APP_PORT
8080
```

```
⚠️ Important: Environment variable values are ALWAYS strings.
   Even if you write: value: 8080 (without quotes)
   Kubernetes treats it as the string "8080".
   Your app needs to parse it to an integer if needed.
```

### Method 2: From ConfigMaps

```yaml
# First, create a ConfigMap
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  APP_ENV: production
  LOG_LEVEL: info
  MAX_RETRIES: "3"
```

#### a) Individual keys with `valueFrom`

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
    - name: my-app
      image: my-app:v1
      env:
        - name: ENVIRONMENT              # You can rename the key!
          valueFrom:
            configMapKeyRef:
              name: app-config            # ConfigMap name
              key: APP_ENV                # Key in the ConfigMap
        - name: LOG_LEVEL
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: LOG_LEVEL
```

```
Inside the container:

$ echo $ENVIRONMENT      # Renamed from APP_ENV
production

$ echo $LOG_LEVEL
info
```

#### b) All keys at once with `envFrom`

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
    - name: my-app
      image: my-app:v1
      envFrom:
        - configMapRef:
            name: app-config          # All keys become env vars!
```

```
Inside the container:

$ env | sort
APP_ENV=production
LOG_LEVEL=info
MAX_RETRIES=3

All keys from the ConfigMap are injected as env vars.
Key names become env var names directly.
```

#### c) With a prefix

```yaml
envFrom:
  - configMapRef:
      name: app-config
    prefix: CONFIG_                   # Adds "CONFIG_" prefix to all keys
```

```
Inside the container:

$ env | grep CONFIG
CONFIG_APP_ENV=production
CONFIG_LOG_LEVEL=info
CONFIG_MAX_RETRIES=3
```

### Method 3: From Secrets

Works exactly like ConfigMaps, but uses `secretKeyRef` instead:

```yaml
env:
  - name: DB_PASSWORD
    valueFrom:
      secretKeyRef:                   # Secret instead of ConfigMap
        name: db-credentials
        key: password

# Or all at once:
envFrom:
  - secretRef:
      name: db-credentials
```

### Method 4: From Pod Fields (Downward API)

This is super cool — you can inject **information about the Pod itself** as env vars!

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
  labels:
    app: web
    version: v2
spec:
  containers:
    - name: my-app
      image: my-app:v1
      env:
        # Pod's own name
        - name: POD_NAME
          valueFrom:
            fieldRef:
              fieldPath: metadata.name

        # Pod's namespace
        - name: POD_NAMESPACE
          valueFrom:
            fieldRef:
              fieldPath: metadata.namespace

        # Pod's IP address
        - name: POD_IP
          valueFrom:
            fieldRef:
              fieldPath: status.podIP

        # Node the Pod is running on
        - name: NODE_NAME
          valueFrom:
            fieldRef:
              fieldPath: spec.nodeName

        # Pod's Service Account
        - name: SERVICE_ACCOUNT
          valueFrom:
            fieldRef:
              fieldPath: spec.serviceAccountName

        # Pod's UID
        - name: POD_UID
          valueFrom:
            fieldRef:
              fieldPath: metadata.uid

        # A specific label value
        - name: APP_LABEL
          valueFrom:
            fieldRef:
              fieldPath: metadata.labels['app']
```

```
Inside the container:

$ echo $POD_NAME
my-app

$ echo $POD_NAMESPACE
default

$ echo $POD_IP
10.244.0.15

$ echo $NODE_NAME
worker-node-1

$ echo $SERVICE_ACCOUNT
default

$ echo $APP_LABEL
web
```

### Method 5: From Container Resource Fields

You can also inject the container's **resource limits and requests** as env vars:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
    - name: my-app
      image: my-app:v1
      resources:
        requests:
          memory: "128Mi"
          cpu: "250m"
        limits:
          memory: "256Mi"
          cpu: "500m"
      env:
        - name: MY_CPU_REQUEST
          valueFrom:
            resourceFieldRef:
              containerName: my-app
              resource: requests.cpu

        - name: MY_MEM_LIMIT
          valueFrom:
            resourceFieldRef:
              containerName: my-app
              resource: limits.memory
              divisor: "1Mi"          # Convert to MiB (optional)
```

```
Inside the container:

$ echo $MY_CPU_REQUEST
1                          # 250m = 0.25 CPU, shown as 1 (ceiling)

$ echo $MY_MEM_LIMIT
256                        # 256Mi, shown as 256 because of divisor
```

```
Available Resource Fields:

requests.cpu, requests.memory, requests.ephemeral-storage
limits.cpu, limits.memory, limits.ephemeral-storage
```

## 📁 Part 2: Volume Mounts — Files and Directories

Volume Mounts let you inject **files and directories** into a container. This is essential for configuration files, certificates, and any data that's better represented as files rather than simple key-value pairs.

### How Volume Mounts Work

```
Two-Step Process:

Step 1: Define a Volume (in spec.volumes)
   → "Here's a volume called 'config-vol' that comes from ConfigMap 'app-config'"

Step 2: Mount it (in containers.volumeMounts)
   → "Mount 'config-vol' into the container at /etc/config"

┌─────────────────────────────────────────────────────────┐
│  Pod Spec                                               │
│                                                         │
│  volumes:                                               │
│    - name: config-vol         ← Step 1: Define volume   │
│      configMap:                                         │
│        name: app-config                                 │
│                                                         │
│  containers:                                            │
│    - name: my-app                                       │
│      volumeMounts:                                      │
│        - name: config-vol     ← Step 2: Mount it        │
│          mountPath: /etc/config                         │
└─────────────────────────────────────────────────────────┘
```

### Mount ConfigMap as Files

```yaml
# ConfigMap with a full config file
apiVersion: v1
kind: ConfigMap
metadata:
  name: nginx-config
data:
  nginx.conf: |
    server {
        listen 80;
        server_name example.com;
        location / {
            proxy_pass http://backend:3000;
        }
    }
  mime.types: |
    text/html  html htm;
    text/css   css;
```

```yaml
# Pod that mounts the ConfigMap as files
apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  containers:
    - name: nginx
      image: nginx:latest
      volumeMounts:
        - name: nginx-config-volume
          mountPath: /etc/nginx/conf.d    # Where files appear in container
          readOnly: true
  volumes:
    - name: nginx-config-volume
      configMap:
        name: nginx-config                # Name of the ConfigMap
```

```
Inside the container:

$ ls /etc/nginx/conf.d/
nginx.conf    mime.types

$ cat /etc/nginx/conf.d/nginx.conf
server {
    listen 80;
    server_name example.com;
    location / {
        proxy_pass http://backend:3000;
    }
}

Each KEY in the ConfigMap becomes a FILENAME.
Each VALUE becomes the FILE CONTENT.
```

### Mount Only Specific Keys

```yaml
volumes:
  - name: nginx-config-volume
    configMap:
      name: nginx-config
      items:                            # Only mount specific keys
        - key: nginx.conf               # Key from ConfigMap
          path: default.conf            # Filename in mount directory
```

```
Inside the container:

$ ls /etc/nginx/conf.d/
default.conf                 # Only the specified key, renamed

$ cat /etc/nginx/conf.d/default.conf
server {
    listen 80;
    ...
}
```

### Mount Secret as Files

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
    - name: my-app
      image: my-app:v1
      volumeMounts:
        - name: tls-certs
          mountPath: /etc/tls
          readOnly: true               # Always read-only for secrets!
  volumes:
    - name: tls-certs
      secret:
        secretName: my-tls-secret      # Secret containing tls.crt and tls.key
        defaultMode: 0400              # File permissions (read-only for owner)
```

```
Inside the container:

$ ls -la /etc/tls/
-r--------  1 root root  1234 Jan 1 00:00 tls.crt
-r--------  1 root root  5678 Jan 1 00:00 tls.key

Values are automatically decoded from base64!
```

### Mount a Single File WITHOUT Overwriting the Directory

By default, mounting a volume **replaces the entire directory**. This can be a problem!

```
The Problem:

mountPath: /etc/nginx
This REPLACES everything in /etc/nginx/ with ONLY the mounted files.
All original files in /etc/nginx/ are GONE!

The Solution: subPath
Mount a single file into an existing directory without destroying other files.
```

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  containers:
    - name: nginx
      image: nginx:latest
      volumeMounts:
        - name: nginx-config
          mountPath: /etc/nginx/nginx.conf    # Full path to the FILE
          subPath: nginx.conf                 # Key from the ConfigMap
  volumes:
    - name: nginx-config
      configMap:
        name: nginx-config
```

```
With subPath:
/etc/nginx/
  ├── nginx.conf        ← Our mounted file
  ├── conf.d/           ← Original directory still exists!
  ├── mime.types         ← Original file still exists!
  └── ...

Without subPath:
/etc/nginx/
  └── nginx.conf        ← ONLY our file exists, everything else is GONE!

⚠️ Important trade-off: subPath files do NOT auto-update when the
ConfigMap/Secret changes! Only regular volume mounts auto-update.
```

## 📦 Part 3: Projected Volumes — Combining Multiple Sources

**Projected Volumes** are like a superpowered volume mount that lets you **combine data from multiple sources** (ConfigMaps, Secrets, Downward API, and ServiceAccount tokens) into a **single directory**.

### Why Projected Volumes?

```
Without Projected Volumes:

You need 3 different mounts for 3 sources:
  /etc/config/app-settings/    ← from ConfigMap
  /etc/config/secrets/         ← from Secret
  /etc/config/pod-info/        ← from Downward API

That's messy! 😩

With Projected Volumes:

Everything in ONE directory:
  /etc/config/
    ├── app.conf               ← from ConfigMap
    ├── db-password            ← from Secret
    ├── labels                 ← from Downward API
    └── token                  ← from ServiceAccount

Clean and organized! ✨
```

### Basic Projected Volume Example

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
  labels:
    app: web
    tier: frontend
spec:
  containers:
    - name: my-app
      image: my-app:v1
      volumeMounts:
        - name: all-config
          mountPath: /etc/app-config
          readOnly: true
  volumes:
    - name: all-config
      projected:                          # ← Projected volume!
        sources:
          # Source 1: From a ConfigMap
          - configMap:
              name: app-config
              items:
                - key: app.conf
                  path: app.conf          # → /etc/app-config/app.conf

          # Source 2: From a Secret
          - secret:
              name: db-credentials
              items:
                - key: password
                  path: db-password       # → /etc/app-config/db-password

          # Source 3: From Downward API (Pod info)
          - downwardAPI:
              items:
                - path: labels            # → /etc/app-config/labels
                  fieldRef:
                    fieldPath: metadata.labels

                - path: cpu-request       # → /etc/app-config/cpu-request
                  resourceFieldRef:
                    containerName: my-app
                    resource: requests.cpu

          # Source 4: ServiceAccount token
          - serviceAccountToken:
              path: token                 # → /etc/app-config/token
              expirationSeconds: 3600     # Token expires in 1 hour
              audience: "api"             # Who the token is for
```

```
Inside the container:

$ ls /etc/app-config/
app.conf       ← from ConfigMap
db-password    ← from Secret
labels         ← from Downward API
cpu-request    ← from Downward API (resource info)
token          ← from ServiceAccount

$ cat /etc/app-config/app.conf
(contents of the ConfigMap key)

$ cat /etc/app-config/db-password
SuperSecret123!

$ cat /etc/app-config/labels
app="web"
tier="frontend"

$ cat /etc/app-config/token
eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...
```

### Projected Volume Sources Explained

```
┌────────────────────────┬───────────────────────────────────────────┐
│  Source Type            │  What It Provides                         │
├────────────────────────┼───────────────────────────────────────────┤
│  configMap             │  Keys from a ConfigMap as files            │
│  secret                │  Keys from a Secret as files               │
│  downwardAPI           │  Pod metadata (labels, annotations,        │
│                        │  name, namespace, IP, resources)           │
│  serviceAccountToken   │  Auto-rotating service account JWT tokens  │
│                        │  (used for Pod identity / API auth)        │
└────────────────────────┴───────────────────────────────────────────┘
```

## 🔽 Part 4: Downward API Deep Dive

The Downward API lets a Pod know about **itself** — its name, namespace, labels, resource limits, etc. There are two ways to use it:

### Way 1: As Environment Variables (covered above)

```yaml
env:
  - name: POD_NAME
    valueFrom:
      fieldRef:
        fieldPath: metadata.name
```

### Way 2: As Files via Volume Mount

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
  labels:
    app: web
    version: v3
  annotations:
    owner: "team-alpha"
    description: "Main web application"
spec:
  containers:
    - name: my-app
      image: my-app:v1
      resources:
        requests:
          memory: "64Mi"
          cpu: "250m"
        limits:
          memory: "128Mi"
          cpu: "500m"
      volumeMounts:
        - name: pod-info
          mountPath: /etc/pod-info
          readOnly: true
  volumes:
    - name: pod-info
      downwardAPI:
        items:
          - path: "name"
            fieldRef:
              fieldPath: metadata.name

          - path: "namespace"
            fieldRef:
              fieldPath: metadata.namespace

          - path: "labels"
            fieldRef:
              fieldPath: metadata.labels

          - path: "annotations"
            fieldRef:
              fieldPath: metadata.annotations

          - path: "cpu-request"
            resourceFieldRef:
              containerName: my-app
              resource: requests.cpu
              divisor: "1m"             # Show in millicores

          - path: "mem-limit"
            resourceFieldRef:
              containerName: my-app
              resource: limits.memory
              divisor: "1Mi"            # Show in MiB
```

```
Inside the container:

$ cat /etc/pod-info/name
my-app

$ cat /etc/pod-info/namespace
default

$ cat /etc/pod-info/labels
app="web"
version="v3"

$ cat /etc/pod-info/annotations
description="Main web application"
owner="team-alpha"

$ cat /etc/pod-info/cpu-request
250

$ cat /etc/pod-info/mem-limit
128
```

### Available Downward API Fields

```
Pod-level fields (via fieldRef):
┌─────────────────────────────────────┬─────────────────────────────┐
│  fieldPath                          │  What It Returns            │
├─────────────────────────────────────┼─────────────────────────────┤
│  metadata.name                      │  Pod name                   │
│  metadata.namespace                 │  Pod namespace              │
│  metadata.uid                       │  Pod UID                    │
│  metadata.labels                    │  All labels                 │
│  metadata.labels['<KEY>']           │  Specific label value       │
│  metadata.annotations               │  All annotations            │
│  metadata.annotations['<KEY>']      │  Specific annotation value  │
│  spec.nodeName                      │  Node the Pod runs on       │
│  spec.serviceAccountName            │  ServiceAccount name        │
│  status.hostIP                      │  Node's IP address          │
│  status.podIP                       │  Pod's IP address           │
│  status.podIPs                      │  All Pod IPs                │
└─────────────────────────────────────┴─────────────────────────────┘

Container-level fields (via resourceFieldRef):
┌─────────────────────────────────────┬─────────────────────────────┐
│  resource                           │  What It Returns            │
├─────────────────────────────────────┼─────────────────────────────┤
│  requests.cpu                       │  CPU request                │
│  limits.cpu                         │  CPU limit                  │
│  requests.memory                    │  Memory request             │
│  limits.memory                      │  Memory limit               │
│  requests.ephemeral-storage         │  Storage request            │
│  limits.ephemeral-storage           │  Storage limit              │
└─────────────────────────────────────┴─────────────────────────────┘
```

## 🧪 Practical Example: Full App with All Methods Combined

Let's build a real-world example that uses env vars, volume mounts, and projected volumes all together:

```yaml
# configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: webapp-config
data:
  APP_ENV: production
  LOG_LEVEL: info
  MAX_CONNECTIONS: "100"
  app-settings.json: |
    {
      "theme": "dark",
      "language": "en",
      "features": {
        "notifications": true,
        "analytics": false
      }
    }
```

```yaml
# secret.yaml
apiVersion: v1
kind: Secret
metadata:
  name: webapp-secrets
type: Opaque
stringData:
  DB_PASSWORD: "SuperSecret123!"
  API_KEY: "sk-abc123xyz789"
  tls.crt: |
    -----BEGIN CERTIFICATE-----
    MIIBxTCCAW...
    -----END CERTIFICATE-----
  tls.key: |
    -----BEGIN PRIVATE KEY-----
    MIIBvAIBADA...
    -----END PRIVATE KEY-----
```

```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: webapp
spec:
  replicas: 3
  selector:
    matchLabels:
      app: webapp
  template:
    metadata:
      labels:
        app: webapp
        version: v2
    spec:
      containers:
        - name: webapp
          image: webapp:v2
          ports:
            - containerPort: 8080
          resources:
            requests:
              cpu: "250m"
              memory: "128Mi"
            limits:
              cpu: "500m"
              memory: "256Mi"

          # ─── Method 1: Simple env vars ───
          env:
            - name: APP_PORT
              value: "8080"

          # ─── Method 2: Env vars from ConfigMap (all keys) ───
            - name: APP_ENV
              valueFrom:
                configMapKeyRef:
                  name: webapp-config
                  key: APP_ENV

          # ─── Method 3: Env vars from Secret ───
            - name: DB_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: webapp-secrets
                  key: DB_PASSWORD

          # ─── Method 4: Downward API env vars ───
            - name: POD_NAME
              valueFrom:
                fieldRef:
                  fieldPath: metadata.name
            - name: POD_IP
              valueFrom:
                fieldRef:
                  fieldPath: status.podIP

          # ─── Volume Mounts ───
          volumeMounts:
            # Config file from ConfigMap
            - name: app-settings
              mountPath: /app/config/settings.json
              subPath: app-settings.json
              readOnly: true

            # TLS certificates from Secret
            - name: tls-certs
              mountPath: /etc/tls
              readOnly: true

            # Projected volume with combined sources
            - name: combined-config
              mountPath: /etc/app-info
              readOnly: true

      volumes:
        # ConfigMap as a volume (for the settings file)
        - name: app-settings
          configMap:
            name: webapp-config

        # Secret as a volume (for TLS certs)
        - name: tls-certs
          secret:
            secretName: webapp-secrets
            items:
              - key: tls.crt
                path: tls.crt
              - key: tls.key
                path: tls.key
            defaultMode: 0400

        # Projected volume (combining multiple sources)
        - name: combined-config
          projected:
            sources:
              - configMap:
                  name: webapp-config
                  items:
                    - key: LOG_LEVEL
                      path: log-level
              - downwardAPI:
                  items:
                    - path: labels
                      fieldRef:
                        fieldPath: metadata.labels
                    - path: pod-name
                      fieldRef:
                        fieldPath: metadata.name
              - serviceAccountToken:
                  path: token
                  expirationSeconds: 3600
```

```
Inside the container — everything available:

# Env vars
$ echo $APP_PORT → 8080
$ echo $APP_ENV → production
$ echo $DB_PASSWORD → SuperSecret123!
$ echo $POD_NAME → webapp-6d4f5b7c8-abc12
$ echo $POD_IP → 10.244.0.23

# Config file (subPath mount — doesn't overwrite directory)
$ cat /app/config/settings.json
{
  "theme": "dark",
  "language": "en",
  ...
}

# TLS certs (from Secret)
$ ls /etc/tls/
tls.crt  tls.key

# Projected volume (combined sources)
$ ls /etc/app-info/
labels  log-level  pod-name  token

$ cat /etc/app-info/log-level → info
$ cat /etc/app-info/pod-name → webapp-6d4f5b7c8-abc12
$ cat /etc/app-info/labels → app="webapp"\nversion="v2"
```

## 🔄 Update Behavior — What Gets Auto-Updated?

This is critical to understand:

```
┌──────────────────────────────┬──────────────┬──────────────────────┐
│  Configuration Method        │  Auto-Update │  Notes               │
├──────────────────────────────┼──────────────┼──────────────────────┤
│  env / envFrom               │  ❌ No       │  Must restart Pod    │
│  Regular volume mount        │  ✅ Yes      │  ~60s delay + kubelet│
│                              │              │  sync period         │
│  subPath volume mount        │  ❌ No       │  Must restart Pod    │
│  Projected volume            │  ✅ Yes      │  ~60s delay + kubelet│
│                              │              │  sync period         │
│  Downward API volume         │  ✅ Yes      │  Labels/annotations  │
│                              │              │  update reflected    │
│  Downward API env var        │  ❌ No       │  Set at Pod creation │
└──────────────────────────────┴──────────────┴──────────────────────┘

Rule of thumb:
  📋 Env vars → NEVER auto-update (restart needed)
  📁 Regular volume mounts → auto-update (eventually)
  📁 subPath mounts → NEVER auto-update (restart needed)
```

```bash
# To force Pods to pick up new config:
kubectl rollout restart deployment webapp
```

## ⚠️ Common Gotchas & Best Practices

### Gotcha 1: Volume mount overwrites entire directory

```yaml
# ❌ This DELETES everything in /etc/nginx/
volumeMounts:
  - name: config
    mountPath: /etc/nginx

# ✅ Use subPath to add a single file
volumeMounts:
  - name: config
    mountPath: /etc/nginx/custom.conf
    subPath: custom.conf
```

### Gotcha 2: Invalid env var names from ConfigMap keys

```yaml
# ConfigMap key: "app.setting.name"
# ❌ This becomes env var: app.setting.name — INVALID! (dots not allowed)

# Solution: Use items with path when mounting as volume
# Or rename keys to use underscores: APP_SETTING_NAME
```

### Gotcha 3: Order matters with envFrom + env

```yaml
# If you use BOTH envFrom and env, individual env entries
# can OVERRIDE envFrom entries:

envFrom:
  - configMapRef:
      name: app-config        # Sets LOG_LEVEL=info

env:
  - name: LOG_LEVEL
    value: "debug"            # OVERRIDES the ConfigMap value!

# Result: LOG_LEVEL=debug (env wins over envFrom)
```

### Gotcha 4: Optional ConfigMaps and Secrets

```yaml
# If a ConfigMap or Secret doesn't exist, the Pod FAILS to start.
# Make it optional:

env:
  - name: OPTIONAL_VALUE
    valueFrom:
      configMapKeyRef:
        name: maybe-exists
        key: some-key
        optional: true          # Pod starts even if ConfigMap doesn't exist

envFrom:
  - configMapRef:
      name: maybe-exists
      optional: true            # Same for envFrom
```

## 📝 Quick Reference Cheatsheet

```yaml
# ─── ENV VARS ───

# Static value
env:
  - name: KEY
    value: "value"

# From ConfigMap (single key)
env:
  - name: KEY
    valueFrom:
      configMapKeyRef:
        name: config-name
        key: key-name

# From Secret (single key)
env:
  - name: KEY
    valueFrom:
      secretKeyRef:
        name: secret-name
        key: key-name

# All keys from ConfigMap
envFrom:
  - configMapRef:
      name: config-name

# All keys from Secret
envFrom:
  - secretRef:
      name: secret-name

# Pod info (Downward API)
env:
  - name: POD_NAME
    valueFrom:
      fieldRef:
        fieldPath: metadata.name

# Resource info
env:
  - name: CPU_LIMIT
    valueFrom:
      resourceFieldRef:
        resource: limits.cpu

# ─── VOLUME MOUNTS ───

# ConfigMap as files
volumes:
  - name: vol
    configMap:
      name: config-name
# Mount:
volumeMounts:
  - name: vol
    mountPath: /path

# Secret as files
volumes:
  - name: vol
    secret:
      secretName: secret-name
# Mount:
volumeMounts:
  - name: vol
    mountPath: /path
    readOnly: true

# Single file (subPath)
volumeMounts:
  - name: vol
    mountPath: /path/file.conf
    subPath: file.conf

# ─── PROJECTED ───
volumes:
  - name: vol
    projected:
      sources:
        - configMap: ...
        - secret: ...
        - downwardAPI: ...
        - serviceAccountToken: ...
```

## 🎯 Key Takeaways

```
1. Environment Variables: best for simple key-value config (URLs, ports, flags)
2. Volume Mounts: best for config files, certs, and complex data
3. Projected Volumes: combine ConfigMaps, Secrets, Pod info in one directory
4. Downward API: lets Pods know about themselves (name, IP, labels, resources)
5. subPath: mount a single file without wiping the directory — but loses auto-update
6. Env vars NEVER auto-update — must restart the Pod
7. Regular volume mounts DO auto-update (~60s delay)
8. env entries override envFrom entries for the same key
9. Use optional: true if the ConfigMap/Secret might not exist
10. Always mount Secrets as readOnly: true
```
