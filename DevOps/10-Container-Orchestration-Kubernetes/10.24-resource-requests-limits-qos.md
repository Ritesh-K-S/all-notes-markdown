# 10.24 — Resource Requests & Limits — CPU, Memory, QoS Classes

> **One Line Answer:** **Resource Requests** tell Kubernetes how much CPU and memory a container **needs at minimum** (used for scheduling decisions), while **Resource Limits** set the **maximum** a container can use (enforced at runtime). Together, they determine the **Quality of Service (QoS) class** of your Pod — which decides what gets killed first when the node runs low on resources. Getting this right is one of the most important things you'll do in production Kubernetes.

---

## 🤔 Why Do We Need Resource Requests & Limits?

### The Problem: No Resource Controls

Imagine a Kubernetes node with 4 CPU cores and 8 GB of RAM, running 5 Pods:

```
Without Resource Requests & Limits:

Node: 4 CPU, 8 GB RAM
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  Pod A (web app)       → uses 0.5 CPU, 512 MB ✅        │
│  Pod B (API server)    → uses 1 CPU, 1 GB ✅            │
│  Pod C (batch job)     → uses 0.2 CPU, 256 MB ✅        │
│  Pod D (ML training)   → wants ALL the CPU and memory!  │
│  Pod E (database)      → needs 2 CPU, 4 GB              │
│                                                         │
│  😱 Pod D goes rogue and consumes:                      │
│     - 3 CPU cores                                       │
│     - 6 GB RAM                                          │
│                                                         │
│  Result:                                                │
│     - Pod A, B, C → starved, slow, unresponsive         │
│     - Pod E → can't get enough memory, OOM killed!      │
│     - Your website is DOWN 💀                           │
│                                                         │
└─────────────────────────────────────────────────────────┘

One misbehaving container can take down an ENTIRE node!
This is called the "Noisy Neighbor" problem.
```

### The Solution: Resource Requests & Limits

```
With Resource Requests & Limits:

Node: 4 CPU, 8 GB RAM
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  Pod A: requests 0.5 CPU, 512 MB | limits 1 CPU, 1 GB  │
│  Pod B: requests 1 CPU, 1 GB     | limits 2 CPU, 2 GB  │
│  Pod C: requests 0.2 CPU, 256 MB | limits 0.5 CPU, 512M│
│  Pod D: requests 1 CPU, 2 GB     | limits 1 CPU, 2 GB  │
│  Pod E: requests 2 CPU, 4 GB     | limits 2 CPU, 4 GB  │
│                                                         │
│  ✅ Scheduler places Pods on nodes with enough resources│
│  ✅ No Pod can exceed its limits                        │
│  ✅ Misbehaving Pods get throttled or killed            │
│  ✅ Everyone gets their fair share                      │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 📐 Understanding CPU and Memory Units

Before we set requests and limits, let's understand the units:

### CPU Units

```
CPU is measured in "cores" or "millicores":

1 CPU    = 1 full core (of any CPU — vCPU on cloud, physical core on bare metal)
1000m    = 1 CPU (m = millicores)
500m     = 0.5 CPU (half a core)
250m     = 0.25 CPU (quarter of a core)
100m     = 0.1 CPU (one-tenth of a core)

Examples:
┌─────────────────┬──────────────┬────────────────────────────┐
│  Value           │  Equivalent  │  Description               │
├─────────────────┼──────────────┼────────────────────────────┤
│  cpu: "1"        │  1000m       │  One full CPU core         │
│  cpu: "0.5"      │  500m        │  Half a CPU core           │
│  cpu: "100m"     │  0.1 CPU     │  10% of a CPU core         │
│  cpu: "2"        │  2000m       │  Two full CPU cores        │
│  cpu: "250m"     │  0.25 CPU    │  Quarter of a CPU core     │
└─────────────────┴──────────────┴────────────────────────────┘

💡 In real-world apps:
   - A simple web server: 100m-500m
   - An API server: 250m-1000m
   - A database: 500m-2000m
   - ML/AI workload: 2000m-8000m+
```

### Memory Units

```
Memory is measured in bytes with standard suffixes:

Binary (powers of 2 — use these!):
┌──────────┬──────────────────┬────────────────────┐
│  Suffix  │  Meaning         │  Bytes             │
├──────────┼──────────────────┼────────────────────┤
│  Ki      │  Kibibytes       │  1024 bytes        │
│  Mi      │  Mebibytes       │  1024 Ki           │
│  Gi      │  Gibibytes       │  1024 Mi           │
│  Ti      │  Tebibytes       │  1024 Gi           │
└──────────┴──────────────────┴────────────────────┘

Decimal (powers of 10 — less common):
┌──────────┬──────────────────┬────────────────────┐
│  Suffix  │  Meaning         │  Bytes             │
├──────────┼──────────────────┼────────────────────┤
│  K       │  Kilobytes       │  1000 bytes        │
│  M       │  Megabytes       │  1000 K            │
│  G       │  Gigabytes       │  1000 M            │
└──────────┴──────────────────┴────────────────────┘

Examples:
  memory: "128Mi"    → 128 Mebibytes (~134 MB)
  memory: "1Gi"      → 1 Gibibyte (~1.07 GB)
  memory: "256Mi"    → 256 Mebibytes
  memory: "512Mi"    → 512 Mebibytes

💡 In real-world apps:
   - A simple web server: 64Mi-256Mi
   - An API server: 128Mi-512Mi
   - A database: 512Mi-4Gi
   - ML/AI workload: 2Gi-32Gi+

⚠️ Best Practice: Always use binary suffixes (Mi, Gi)
   because that's what the OS actually measures.
```

## 🛠️ Setting Requests and Limits

### Basic Syntax

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
    - name: my-app
      image: my-app:v1
      resources:
        requests:                     # MINIMUM guaranteed resources
          cpu: "250m"                 # At least 0.25 CPU
          memory: "128Mi"             # At least 128 MiB RAM
        limits:                       # MAXIMUM allowed resources
          cpu: "500m"                 # Can't use more than 0.5 CPU
          memory: "256Mi"             # Can't use more than 256 MiB RAM
```

### What Requests and Limits Actually Do

```
┌──────────────────────────────────────────────────────────────────┐
│                     REQUESTS vs LIMITS                           │
├────────────────┬────────────────────┬────────────────────────────┤
│                │  REQUESTS          │  LIMITS                    │
├────────────────┼────────────────────┼────────────────────────────┤
│  What is it?   │  Minimum guarantee │  Maximum ceiling           │
│                │                    │                            │
│  When used?    │  At SCHEDULING     │  At RUNTIME                │
│                │  time (deciding    │  (enforcing boundaries     │
│                │  which node)       │  on a running container)   │
│                │                    │                            │
│  CPU behavior  │  Scheduler ensures │  Container gets CPU-       │
│  if exceeded   │  node has enough   │  throttled (slowed down,   │
│                │  allocatable CPU   │  NOT killed)               │
│                │                    │                            │
│  Memory        │  Scheduler ensures │  Container gets OOM-       │
│  behavior      │  node has enough   │  killed (Out of Memory     │
│  if exceeded   │  allocatable memory│  killed) by the kernel     │
│                │                    │                            │
│  Can exceed?   │  Yes! A container  │  CPU: Throttled            │
│                │  can use MORE than │  Memory: KILLED!           │
│                │  its request if    │                            │
│                │  resources are free│                            │
└────────────────┴────────────────────┴────────────────────────────┘
```

Let me explain this with a simple analogy:

```
Think of it like booking a hotel room:

REQUEST = Your reservation
  → "I need at least a room with 1 bed" (guaranteed)
  → The hotel won't book you if no rooms are available

LIMIT = Room upgrade policy
  → "You can use a suite if available, but max 2 beds"
  → If you try to take 3 beds → you're kicked out (memory)
  → If you try to use the gym at full capacity → you're throttled (CPU)

Your container:
  Request 256Mi → "I NEED at least 256Mi to function"
  Limit 512Mi  → "I might burst up to 512Mi, but never more"
```

## 🔍 How the Scheduler Uses Requests

The Kubernetes Scheduler uses **requests** (NOT limits) to decide where to place Pods:

```
Scheduling Decision:

Node A: 4 CPU, 8Gi RAM
Already running:
  - Pod X: requests 1 CPU, 2Gi
  - Pod Y: requests 1.5 CPU, 3Gi
  
Available = Total - Sum of Requests
  CPU:    4 - 1 - 1.5 = 1.5 CPU available
  Memory: 8 - 2 - 3   = 3Gi available

New Pod Z wants: requests 2 CPU, 2Gi
  CPU:    2 > 1.5 → ❌ NOT ENOUGH CPU!
  Memory: 2 < 3   → ✅ Enough memory

Result: Pod Z will NOT be scheduled on Node A.
        Scheduler looks for another node.

If NO node has enough resources → Pod stays "Pending"!
```

```bash
# See why a Pod is Pending:
kubectl describe pod <pod-name>

# You'll see something like:
# Events:
#   Warning  FailedScheduling  0/3 nodes are available:
#   3 Insufficient cpu, 2 Insufficient memory
```

## 🎯 CPU Behavior — Throttling (Not Killing)

```
CPU is a COMPRESSIBLE resource — meaning it can be throttled.

If your container tries to use more CPU than its limit:

┌────────────────────────────────────────────────────────────┐
│  Container: requests 250m, limits 500m                     │
│                                                            │
│  Time 1: Container uses 300m → ✅ OK (under limit)         │
│  Time 2: Container uses 500m → ✅ OK (at limit)            │
│  Time 3: Container tries 800m → ⚠️ THROTTLED to 500m      │
│                                                            │
│  The container is NOT killed — just slowed down.            │
│  It gets exactly 500m of CPU time per scheduling period.    │
│  The rest of the time, it waits.                           │
│                                                            │
│  Effect: Your app becomes SLOWER, not dead.                │
│  Symptoms: Higher latency, slower response times.          │
└────────────────────────────────────────────────────────────┘
```

```
How CPU throttling works under the hood:

Kubernetes uses Linux CFS (Completely Fair Scheduler):
  - cfs_period_us = 100000 (100ms = 0.1 second)
  - cfs_quota_us  = limit × period

Example: limit = 500m (0.5 CPU)
  - quota = 0.5 × 100000 = 50000 microseconds
  - The container gets 50ms of CPU time per 100ms period
  - If it tries to use more → it's paused until the next period

┌──────── 100ms period ────────┐
│ [50ms active] [50ms waiting] │  ← 500m limit
│ [100ms active]               │  ← 1000m (1 CPU) limit
│ [25ms active] [75ms waiting] │  ← 250m limit
└──────────────────────────────┘
```

## 💀 Memory Behavior — OOM Kill (Instant Death)

```
Memory is an INCOMPRESSIBLE resource — you can't throttle it.
If you allocated memory, you're USING it. You can't slow it down.

If your container tries to use more memory than its limit:

┌─────────────────────────────────────────────────────────────┐
│  Container: requests 128Mi, limits 256Mi                    │
│                                                             │
│  Time 1: Container uses 100Mi  → ✅ Fine                    │
│  Time 2: Container uses 200Mi  → ✅ Fine                    │
│  Time 3: Container uses 256Mi  → ⚠️ At the limit            │
│  Time 4: Container tries 257Mi → 💀 OOM KILLED!             │
│                                                             │
│  The Linux kernel's OOM killer terminates the process.      │
│  The container is restarted (if restartPolicy allows).      │
│                                                             │
│  You'll see:                                                │
│  $ kubectl get pods                                         │
│  NAME    READY   STATUS      RESTARTS   AGE                 │
│  my-app  0/1     OOMKilled   3          5m                  │
│                                                             │
│  After multiple OOM kills → CrashLoopBackOff                │
└─────────────────────────────────────────────────────────────┘
```

```bash
# How to detect OOM kills:
kubectl describe pod my-app

# Look for:
#   Last State:     Terminated
#     Reason:       OOMKilled
#     Exit Code:    137

# Exit code 137 = 128 + 9 (SIGKILL signal)
# This means the kernel forcefully killed the process.
```

## ⭐ Quality of Service (QoS) Classes

Based on how you set requests and limits, Kubernetes assigns one of **three QoS classes** to each Pod. This determines which Pod gets killed **first** when the node runs out of resources.

### The Three QoS Classes

```
┌──────────────────────────────────────────────────────────────────────┐
│                    QoS Classes (Priority Order)                      │
│                                                                      │
│  🥇 Guaranteed  (HIGHEST priority — killed LAST)                    │
│  │   Every container has requests = limits for BOTH CPU and memory  │
│  │                                                                  │
│  🥈 Burstable   (MEDIUM priority)                                   │
│  │   At least one container has requests ≠ limits,                  │
│  │   or only requests OR limits set                                 │
│  │                                                                  │
│  🥉 BestEffort  (LOWEST priority — killed FIRST)                   │
│     No requests or limits set at all                                │
│                                                                      │
│  When node is under pressure:                                        │
│  Kill order: BestEffort → Burstable → Guaranteed                    │
│                                                                      │
│  "Guaranteed" Pods are like VIP passengers — last to leave the ship!│
└──────────────────────────────────────────────────────────────────────┘
```

### QoS Class: Guaranteed

```yaml
# GUARANTEED — requests EQUALS limits for ALL containers
apiVersion: v1
kind: Pod
metadata:
  name: guaranteed-pod
spec:
  containers:
    - name: app
      image: app:v1
      resources:
        requests:
          cpu: "500m"           # ← Same
          memory: "256Mi"       # ← Same
        limits:
          cpu: "500m"           # ← Same as request
          memory: "256Mi"       # ← Same as request
```

```
When do you get Guaranteed?

✅ Every container in the Pod MUST have:
   - requests.cpu = limits.cpu
   - requests.memory = limits.memory

Short version: requests == limits for everything.

💡 Tip: If you only set limits (no requests), Kubernetes
   automatically sets requests = limits. So this also works:

   resources:
     limits:
       cpu: "500m"
       memory: "256Mi"
   # requests automatically set to cpu: 500m, memory: 256Mi
   # → QoS = Guaranteed!
```

### QoS Class: Burstable

```yaml
# BURSTABLE — requests and limits are DIFFERENT
apiVersion: v1
kind: Pod
metadata:
  name: burstable-pod
spec:
  containers:
    - name: app
      image: app:v1
      resources:
        requests:
          cpu: "250m"           # ← Different
          memory: "128Mi"       # ← Different
        limits:
          cpu: "500m"           # ← from requests
          memory: "256Mi"       # ← from requests
```

```
When do you get Burstable?

✅ At least ONE container has requests ≠ limits
✅ OR only requests are set (no limits)
✅ OR only limits are set for some resources but not all

Any scenario between "nothing" and "everything equal" = Burstable.

Examples that are Burstable:
  requests: 250m  limits: 500m     ← different values
  requests: 250m  (no limits)      ← only requests
  (only cpu request, no memory)    ← partial
```

### QoS Class: BestEffort

```yaml
# BESTEFFORT — NO resources specified at all
apiVersion: v1
kind: Pod
metadata:
  name: besteffort-pod
spec:
  containers:
    - name: app
      image: app:v1
      # No resources section at all!
```

```
When do you get BestEffort?

✅ NO container in the Pod has ANY requests or limits.
   No cpu request, no cpu limit, no memory request, no memory limit.
   Completely empty resources section (or omitted entirely).

⚠️ This is DANGEROUS in production!
   - Pod can use unlimited resources (steal from others)
   - Pod gets killed FIRST when node is under pressure
   - Scheduler doesn't reserve any resources for it
```

### Check QoS Class

```bash
# See the QoS class of a Pod
kubectl get pod my-app -o yaml | grep qosClass

# Output:
# qosClass: Guaranteed    # or Burstable or BestEffort

# Or with describe:
kubectl describe pod my-app | grep "QoS Class"
# QoS Class: Guaranteed
```

### QoS Decision Flowchart

```
                    Does the Pod have 
                  resources set at all?
                         │
                    ┌────┴────┐
                    │         │
                   NO        YES
                    │         │
              ┌─────▼─────┐  │
              │ BestEffort │  │
              └───────────┘  │
                              │
                   Do ALL containers have
                  requests = limits for
                   BOTH cpu AND memory?
                         │
                    ┌────┴────┐
                    │         │
                   YES       NO
                    │         │
              ┌─────▼──────┐  ┌──────▼──────┐
              │ Guaranteed  │  │  Burstable  │
              └────────────┘  └─────────────┘
```

## 💥 What Happens When the Node Runs Out of Resources?

### CPU Pressure

```
When a node runs out of CPU:

The kernel uses CFS to give each container its fair share based on REQUESTS.

Example: Node has 2 CPU, all used up.
  Container A: requests 1 CPU  → gets 1 CPU (50%)
  Container B: requests 500m   → gets 500m (25%)
  Container C: requests 500m   → gets 500m (25%)

Nobody is killed! They're just slowed down proportionally.
Requests act as "weights" for CPU time sharing.

If Container A is idle, B and C can use its CPU.
When A wakes up, it takes back its share.
```

### Memory Pressure (Eviction)

```
When a node runs out of memory — things get serious:

1. kubelet detects memory pressure
2. kubelet starts EVICTING Pods based on QoS class:

   Phase 1: Kill BestEffort Pods first
            (they never promised they'd be well-behaved)
            
   Phase 2: Kill Burstable Pods that are using more than their request
            (sorted by usage above request, highest first)
            
   Phase 3: Kill Guaranteed Pods (only as last resort)
            (they followed all the rules — killed only in extreme cases)

Within same QoS class, Pods using MORE memory relative to 
their request get killed first.
```

```
Eviction Thresholds (defaults):

memory.available < 100Mi  → eviction starts
nodefs.available < 10%    → eviction starts (disk)
imagefs.available < 15%   → eviction starts (image disk)

These are configurable in kubelet settings.
```

## 🔄 LimitRange — Default Requests & Limits for a Namespace

Don't want to manually set resources on every Pod? Use **LimitRange** to set defaults!

```yaml
# limitrange.yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: default-limits
  namespace: development
spec:
  limits:
    - type: Container
      default:                     # Default LIMITS (if not specified)
        cpu: "500m"
        memory: "256Mi"
      defaultRequest:              # Default REQUESTS (if not specified)
        cpu: "100m"
        memory: "128Mi"
      max:                         # Maximum allowed limits
        cpu: "2"
        memory: "1Gi"
      min:                         # Minimum allowed requests
        cpu: "50m"
        memory: "64Mi"
```

```bash
kubectl apply -f limitrange.yaml
```

```
What this does:

1. default: If a container has no limits, it gets cpu: 500m, memory: 256Mi
2. defaultRequest: If a container has no requests, it gets cpu: 100m, memory: 128Mi
3. max: No container can have limits higher than 2 CPU, 1Gi memory
4. min: No container can have requests lower than 50m CPU, 64Mi memory

Now ANY Pod created in the "development" namespace will automatically
get these defaults if they don't specify their own resources!
```

```yaml
# This Pod in the "development" namespace:
apiVersion: v1
kind: Pod
metadata:
  name: my-app
  namespace: development
spec:
  containers:
    - name: my-app
      image: my-app:v1
      # No resources specified!

# Automatically gets:
# resources:
#   requests:
#     cpu: 100m
#     memory: 128Mi
#   limits:
#     cpu: 500m
#     memory: 256Mi
```

## 📊 ResourceQuota — Namespace-Level Resource Budgets

While LimitRange controls **individual Pods**, ResourceQuota controls the **total** for an entire namespace.

```yaml
# resource-quota.yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: team-quota
  namespace: development
spec:
  hard:
    # Total CPU and memory for the namespace
    requests.cpu: "10"            # All requests.cpu in namespace ≤ 10 CPU
    requests.memory: "20Gi"       # All requests.memory ≤ 20 GiB
    limits.cpu: "20"              # All limits.cpu ≤ 20 CPU
    limits.memory: "40Gi"         # All limits.memory ≤ 40 GiB

    # Pod count limits
    pods: "50"                    # Max 50 Pods in this namespace
    count/deployments.apps: "20"  # Max 20 Deployments
    count/services: "10"          # Max 10 Services

    # Storage limits
    requests.storage: "100Gi"     # Total PVC storage ≤ 100 GiB
    persistentvolumeclaims: "20"  # Max 20 PVCs
```

```bash
kubectl apply -f resource-quota.yaml

# Check current usage:
kubectl describe resourcequota team-quota -n development
```

```
Output:
Name:                   team-quota
Namespace:              development
Resource                Used    Hard
--------                ----    ----
limits.cpu              4       20
limits.memory           8Gi     40Gi
pods                    12      50
requests.cpu            2       10
requests.memory         4Gi     20Gi
requests.storage        30Gi    100Gi

This shows: the team is using 2 of 10 allowed CPU requests.
If they try to create a Pod that would exceed the quota → REJECTED!
```

```
⚠️ Important: When a ResourceQuota is active in a namespace,
ALL Pods MUST specify resource requests and limits!

If a Pod doesn't specify resources → it gets REJECTED.
(Unless LimitRange sets defaults for them.)

This is why LimitRange + ResourceQuota are used together:
  - ResourceQuota: Sets the budget for the team
  - LimitRange: Sets smart defaults so Pods aren't rejected
```

## 🧪 Practical Examples

### Example 1: Web Application (Low Resources)

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web-app
  template:
    metadata:
      labels:
        app: web-app
    spec:
      containers:
        - name: web-app
          image: nginx:latest
          ports:
            - containerPort: 80
          resources:
            requests:
              cpu: "100m"           # 10% of a CPU — enough for basic web serving
              memory: "128Mi"       # 128 MiB — enough for nginx
            limits:
              cpu: "250m"           # Can burst to 25% CPU
              memory: "256Mi"       # Max 256 MiB — kill if exceeds
```

### Example 2: API Server (Medium Resources)

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-server
spec:
  replicas: 2
  selector:
    matchLabels:
      app: api-server
  template:
    metadata:
      labels:
        app: api-server
    spec:
      containers:
        - name: api-server
          image: api-server:v1
          ports:
            - containerPort: 3000
          resources:
            requests:
              cpu: "250m"           # Quarter CPU minimum
              memory: "256Mi"       # 256 MiB minimum
            limits:
              cpu: "1"              # Can burst to 1 full CPU
              memory: "512Mi"       # Max 512 MiB
```

### Example 3: Database (High Resources, Guaranteed QoS)

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: postgres
spec:
  replicas: 1
  selector:
    matchLabels:
      app: postgres
  template:
    metadata:
      labels:
        app: postgres
    spec:
      containers:
        - name: postgres
          image: postgres:15
          ports:
            - containerPort: 5432
          resources:
            # requests = limits → Guaranteed QoS!
            requests:
              cpu: "1"              # 1 full CPU guaranteed
              memory: "2Gi"         # 2 GiB guaranteed
            limits:
              cpu: "1"              # Same as request
              memory: "2Gi"         # Same as request
```

### Example 4: Multi-Container Pod

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: multi-container-app
spec:
  containers:
    - name: main-app
      image: app:v1
      resources:
        requests:
          cpu: "500m"
          memory: "256Mi"
        limits:
          cpu: "1"
          memory: "512Mi"

    - name: log-shipper                # Sidecar container
      image: fluentd:latest
      resources:
        requests:
          cpu: "100m"
          memory: "64Mi"
        limits:
          cpu: "200m"
          memory: "128Mi"

# Total Pod resources:
# Requests: 600m CPU, 320Mi memory
# Limits: 1200m CPU, 640Mi memory
# QoS: Burstable (requests ≠ limits)
```

### Check allocatable resources on a node

```bash
kubectl describe node worker-01

# Look for:
# Capacity:
#   cpu:                4
#   memory:             8167344Ki

#   cpu:                3920m      ← available for Pods (some reserved for system)
```

```
Notice: Limits can exceed 100% of a node! This is called OVERCOMMITTING.

The scheduler places Pods based on REQUESTS, not limits.
So you can have:
  - Total requests: 3920m (100% of allocatable)
  - Total limits: 6000m (153% of allocatable!)

This works because not all containers will hit their limits
at the same time. But if they do → OOM kills and throttling.
```

## 🧠 How to Choose the Right Values

```
Step 1: Start with no limits (development/staging)
   → Let your app run and observe actual usage with kubectl top

Step 2: Set requests to the AVERAGE usage
   → What your app typically needs

Step 3: Set limits to the PEAK usage + buffer
   → What your app needs during traffic spikes

Example:
   Observed average: 200m CPU, 180Mi memory
   Observed peak:    450m CPU, 350Mi memory

   Recommended:
   requests:
     cpu: 200m          # Average usage
     memory: 200Mi      # Rounded up from average
   limits:
     cpu: 500m          # Peak + 10% buffer
     memory: 400Mi      # Peak + ~15% buffer
```

```
Common Mistakes:

❌ Setting requests too HIGH → wastes resources, fewer Pods fit on nodes
❌ Setting requests too LOW → scheduler overcommits, Pods get evicted
❌ Setting limits too LOW → CPU throttling, OOM kills
❌ Setting limits too HIGH → noisy neighbor problem
❌ No limits at all → rogue containers can kill the node
❌ requests > limits → invalid! Kubernetes rejects this.

Golden Rules:
✅ requests ≤ limits (always!)
✅ Observe before you set (don't guess)
✅ Use Guaranteed QoS for critical workloads (databases, etc.)
✅ Use LimitRange for defaults
✅ Use ResourceQuota for namespace budgets
```

## 📊 Overcommitting — A Double-Edged Sword

```
Overcommitting = sum of all requests < sum of all limits

Example Node: 4 CPU
  Pod A: requests 1 CPU, limits 2 CPU
  Pod B: requests 1 CPU, limits 2 CPU
  Pod C: requests 1 CPU, limits 2 CPU
  Pod D: requests 1 CPU, limits 2 CPU

  Total requests: 4 CPU (100%) ← Scheduler sees this, node is "full"
  Total limits: 8 CPU (200%)   ← If ALL pods burst at once... trouble!

  This is NORMAL and EXPECTED in Kubernetes.
  Most apps don't use their limits all the time.

  But if they do: CPU throttling + potential OOM kills.
```

## ❓ Common Issues & Troubleshooting

### Issue 1: Pod stuck in Pending (Insufficient resources)

```bash
kubectl describe pod my-app
# Events:
#   Warning  FailedScheduling  0/3 nodes are available:
#   3 Insufficient cpu

# Fix options:
# 1. Lower the requests
# 2. Add more nodes (Cluster Autoscaler)
# 3. Remove/scale down other workloads
# 4. Check if requests are set too high
```

### Issue 2: OOMKilled

```bash
kubectl describe pod my-app
# State:          Terminated
# Reason:         OOMKilled
# Exit Code:      137

# Fix options:
# 1. Increase memory limits
# 2. Fix memory leaks in your app
# 3. Check if your app has a -Xmx (Java) or similar memory setting
#    that should match the container limit
```

### Issue 3: CPU Throttling (slow app)

```bash
# Check for CPU throttling
kubectl top pod my-app
# If CPU usage is constantly at the limit → throttling

# Fix options:
# 1. Increase CPU limits
# 2. Optimize your app's CPU usage
# 3. Add more replicas (horizontal scaling)
```

### Issue 4: Pod Evicted

```bash
kubectl get pods
# NAME    STATUS    
# my-app  Evicted

kubectl describe pod my-app
# Status:    Failed
# Reason:    Evicted
# Message:   The node was low on resource: memory.

# Fix: Increase memory requests or add nodes
```

## 🎯 Key Takeaways

```
1.  REQUESTS = minimum guarantee (used for scheduling)
2.  LIMITS = maximum allowed (enforced at runtime)
3.  CPU is compressible → exceeding limit = throttled (slow)
4.  Memory is incompressible → exceeding limit = OOM killed (dead)
5.  Three QoS Classes: Guaranteed > Burstable > BestEffort
6.  Guaranteed = requests equals limits (safest for critical apps)
7.  BestEffort = no resources set (most dangerous, killed first)
8.  Scheduler uses REQUESTS to place Pods, not limits
9.  Use LimitRange for namespace defaults
10. Use ResourceQuota for namespace budgets
11. Always observe actual usage before setting values
12. requests must always be ≤ limits
13. Install Metrics Server for kubectl top to work
14. Overcommitting is normal — just plan for burst scenarios
```
