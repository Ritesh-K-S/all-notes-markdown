# 10.25 — Node Selectors, Affinity & Anti-Affinity

> **One Line Answer:** Node Selectors, Affinity, and Anti-Affinity are Kubernetes scheduling features that let you **control which nodes your Pods land on** — from simple label matching to complex rules like "put this Pod on a GPU node" or "don't put two replicas on the same node."

## 🤔 Why Do We Need This?

By default, Kubernetes is pretty smart. When you create a Pod, the **Scheduler** looks at all available nodes and picks the best one based on available CPU, memory, and other factors.

But sometimes, **"best available"** isn't good enough. You need **control** over where your Pods run. Here are some real-world scenarios:

| Scenario | What You Need |
|----------|---------------|
| Your ML model needs a GPU | Pod must go to a node with a GPU |
| Your app must run in a specific availability zone | Pod should land on nodes in `us-east-1a` |
| You have SSD nodes for databases | Database Pod should go to SSD nodes |
| You don't want two replicas on the same node | If one node dies, you don't lose both replicas |
| Your frontend and backend should be close | Put them on the same node or same zone for low latency |
| Two competing apps shouldn't share a node | Keep them apart to avoid resource contention |

Kubernetes gives you **three levels** of scheduling control, from simple to powerful:

```
Simple ──────────────────────────────────────────── Complex
  
  nodeSelector        Node Affinity         Pod Affinity / Anti-Affinity
  (basic)             (flexible)            (most powerful)
```

Let's learn each one step by step.

## 📌 Level 1: nodeSelector — The Simplest Way

`nodeSelector` is the **easiest** way to tell Kubernetes: "Hey, put my Pod on a node that has THIS label."

### How It Works

1. You **label** your nodes (key-value pairs)
2. You add `nodeSelector` to your Pod spec with the label you want
3. The Scheduler only considers nodes that **match** that label

### Step 1: Label Your Nodes

First, let's see what nodes we have:

```bash
kubectl get nodes
```

Output:
```
NAME        STATUS   ROLES           AGE   VERSION
node-1      Ready    control-plane   10d   v1.28.0
node-2      Ready    <none>          10d   v1.28.0
node-3      Ready    <none>          10d   v1.28.0
node-4      Ready    <none>          10d   v1.28.0
```

Now, let's say `node-2` has an SSD and `node-3` has a GPU. Let's label them:

```bash
# Label node-2 as having SSD storage
kubectl label nodes node-2 disktype=ssd

# Label node-3 as having a GPU
kubectl label nodes node-3 hardware=gpu

# Label node-4 as being in a specific zone
kubectl label nodes node-4 zone=us-east-1a
```

Verify the labels:

```bash
kubectl get nodes --show-labels
```

Or check a specific node:

```bash
kubectl describe node node-2 | grep -A 10 "Labels"
```

### Step 2: Use nodeSelector in Your Pod

```yaml
# pod-with-nodeselector.yaml
apiVersion: v1
kind: Pod
metadata:
  name: database-pod
spec:
  nodeSelector:
    disktype: ssd       # Only schedule on nodes with label disktype=ssd
  containers:
  - name: postgres
    image: postgres:15
    ports:
    - containerPort: 5432
```

**What happens here:**
- The Scheduler looks at ALL nodes
- It filters out any node that does NOT have the label `disktype=ssd`
- From the remaining nodes (only `node-2` in our case), it picks the best one
- If NO node has that label, the Pod stays in `Pending` state forever!

```bash
kubectl apply -f pod-with-nodeselector.yaml

# Check where it landed
kubectl get pod database-pod -o wide
```

Output:
```
NAME           READY   STATUS    NODE     
database-pod   1/1     Running   node-2   # ✅ Landed on the SSD node!
```

### Using nodeSelector with Deployments

You can use `nodeSelector` inside a Deployment template too:

```yaml
# deployment-gpu.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: ml-model
spec:
  replicas: 3
  selector:
    matchLabels:
      app: ml-model
  template:
    metadata:
      labels:
        app: ml-model
    spec:
      nodeSelector:
        hardware: gpu        # All 3 replicas go to GPU nodes only
      containers:
      - name: model-server
        image: my-ml-model:v1
        resources:
          limits:
            nvidia.com/gpu: 1
```

### Common Built-in Node Labels

Kubernetes automatically adds some labels to every node. You can use these without manually labeling:

```bash
kubectl describe node node-1 | grep -A 20 "Labels"
```

| Label | Example Value | What It Means |
|-------|--------------|---------------|
| `kubernetes.io/hostname` | `node-1` | The node's hostname |
| `kubernetes.io/os` | `linux` | Operating system |
| `kubernetes.io/arch` | `amd64` | CPU architecture |
| `topology.kubernetes.io/zone` | `us-east-1a` | Cloud availability zone |
| `topology.kubernetes.io/region` | `us-east-1` | Cloud region |
| `node.kubernetes.io/instance-type` | `m5.large` | Cloud instance type (AWS/GCP/Azure) |

Example — schedule a Pod only on ARM nodes:

```yaml
spec:
  nodeSelector:
    kubernetes.io/arch: arm64
```

### Removing a Label

```bash
# Notice the minus (-) at the end — that means "remove"
kubectl label nodes node-2 disktype-
```

### ⚠️ Limitations of nodeSelector

| Limitation | Explanation |
|-----------|-------------|
| **Only AND logic** | If you specify two labels, the node must have BOTH. You can't say "label A OR label B" |
| **Hard requirement** | If no matching node exists, Pod stays Pending forever. No soft preferences. |
| **No "NOT" logic** | You can't say "NOT on this node" |
| **No expressions** | Can't say "disk size > 100GB" or "zone IN (us-east-1a, us-east-1b)" |

This is where **Node Affinity** comes in! 🎯

## 🎯 Level 2: Node Affinity — The Powerful Version

Node Affinity is like `nodeSelector` on steroids. It gives you:
- ✅ **Hard rules** (required) — "MUST run on these nodes"
- ✅ **Soft rules** (preferred) — "I'd LIKE to run on these nodes, but it's okay if you can't"
- ✅ **Expressive operators** — In, NotIn, Exists, DoesNotExist, Gt, Lt
- ✅ **Multiple conditions** with AND/OR logic

### The Two Types of Node Affinity

| Type | Meaning | What Happens If No Match? |
|------|---------|---------------------------|
| `requiredDuringSchedulingIgnoredDuringExecution` | **Hard rule** — MUST match | Pod stays **Pending** |
| `preferredDuringSchedulingIgnoredDuringExecution` | **Soft rule** — TRY to match | Pod gets scheduled on **any available node** |

> 💡 **"IgnoredDuringExecution"** means: if the node's labels change AFTER the Pod is already running, Kubernetes **won't evict** the Pod. It only checks at scheduling time.

### Understanding the Operators

| Operator | Meaning | Example |
|----------|---------|---------|
| `In` | Label value is in a list | `zone In [us-east-1a, us-east-1b]` |
| `NotIn` | Label value is NOT in a list | `zone NotIn [us-west-2a]` |
| `Exists` | Label key exists (value doesn't matter) | `gpu Exists` |
| `DoesNotExist` | Label key does NOT exist | `spot DoesNotExist` |
| `Gt` | Label value is greater than (numeric) | `gpu-count Gt 2` |
| `Lt` | Label value is less than (numeric) | `gpu-count Lt 8` |

### Example 1: Hard Rule — Must Run on SSD Nodes

```yaml
# node-affinity-required.yaml
apiVersion: v1
kind: Pod
metadata:
  name: db-pod
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: disktype
            operator: In
            values:
            - ssd
            - nvme          # disktype must be "ssd" OR "nvme"
  containers:
  - name: postgres
    image: postgres:15
```

**What happens:**
- Scheduler checks: Does the node have label `disktype` with value `ssd` OR `nvme`?
- If YES → node is eligible
- If NO → node is filtered out
- If NO node matches at all → Pod stays `Pending`

**This is like nodeSelector**, but notice we can match **multiple values** (ssd OR nvme)! You can't do that with nodeSelector.

### Example 2: Soft Rule — Prefer SSD but Accept Anything

```yaml
# node-affinity-preferred.yaml
apiVersion: v1
kind: Pod
metadata:
  name: cache-pod
spec:
  affinity:
    nodeAffinity:
      preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 80                    # How important is this preference? (1-100)
        preference:
          matchExpressions:
          - key: disktype
            operator: In
            values:
            - ssd
  containers:
  - name: redis
    image: redis:7
```

**What happens:**
- Scheduler says: "I'd PREFER to put this on an SSD node"
- If an SSD node is available → it'll go there
- If NO SSD node is available → it'll go to ANY available node
- The Pod **never stays Pending** because of this rule — it's just a preference

### Example 3: Combining Hard + Soft Rules

This is where it gets really powerful. You can combine both!

```yaml
# node-affinity-combined.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
spec:
  replicas: 5
  selector:
    matchLabels:
      app: web-app
  template:
    metadata:
      labels:
        app: web-app
    spec:
      affinity:
        nodeAffinity:
          # HARD RULE: Must be in us-east region
          requiredDuringSchedulingIgnoredDuringExecution:
            nodeSelectorTerms:
            - matchExpressions:
              - key: topology.kubernetes.io/region
                operator: In
                values:
                - us-east-1
          # SOFT RULE: Prefer zone 1a, but 1b and 1c are okay too
          preferredDuringSchedulingIgnoredDuringExecution:
          - weight: 70
            preference:
              matchExpressions:
              - key: topology.kubernetes.io/zone
                operator: In
                values:
                - us-east-1a
          - weight: 30
            preference:
              matchExpressions:
              - key: topology.kubernetes.io/zone
                operator: In
                values:
                - us-east-1b
      containers:
      - name: web
        image: nginx:1.25
```

**What happens:**
1. **First**, the Scheduler filters: only nodes in `us-east-1` region (hard rule)
2. **Then**, among those nodes, it **scores** them:
   - Nodes in `us-east-1a` get **+70 points** (higher weight = more preferred)
   - Nodes in `us-east-1b` get **+30 points**
   - Nodes in `us-east-1c` get **+0 points** (still eligible, just no bonus)
3. Pods land on the highest-scoring nodes first

### Example 4: Using NotIn — Avoid Certain Nodes

```yaml
# avoid-spot-instances.yaml
apiVersion: v1
kind: Pod
metadata:
  name: critical-app
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: node-type
            operator: NotIn
            values:
            - spot              # Don't run on spot/preemptible instances!
  containers:
  - name: app
    image: my-critical-app:v1
```

### Example 5: Using Exists — Just Check If Label Exists

```yaml
# needs-gpu.yaml
apiVersion: v1
kind: Pod
metadata:
  name: training-job
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: nvidia.com/gpu
            operator: Exists      # We don't care about the value, just that GPU exists
  containers:
  - name: trainer
    image: my-ml-trainer:v1
```

### Multiple nodeSelectorTerms = OR Logic

```yaml
nodeSelectorTerms:
- matchExpressions:            # Term 1
  - key: disktype
    operator: In
    values: [ssd]
- matchExpressions:            # Term 2
  - key: disktype
    operator: In
    values: [nvme]
```

Term 1 **OR** Term 2 must match. (Either SSD or NVMe is fine.)

### Multiple matchExpressions in ONE Term = AND Logic

```yaml
nodeSelectorTerms:
- matchExpressions:
  - key: disktype              # Condition 1
    operator: In
    values: [ssd]
  - key: zone                  # Condition 2
    operator: In
    values: [us-east-1a]
```

Condition 1 **AND** Condition 2 must BOTH be true. (Must be SSD AND in us-east-1a.)

### Summary: OR vs AND

```
nodeSelectorTerms:
  - term1       ─┐
  - term2       ─┤──→  OR  (any term can match)
  - term3       ─┘
  
Each term:
  matchExpressions:
    - expr1     ─┐
    - expr2     ─┤──→  AND  (all expressions must match)
    - expr3     ─┘
```

## 🔗 Level 3: Pod Affinity & Pod Anti-Affinity — Schedule Relative to Other Pods

So far we've been saying "put my Pod on a node with THESE labels." But what if you want to say:

- "Put my Pod on the **same node** as another Pod" → **Pod Affinity**
- "Put my Pod on a **different node** than another Pod" → **Pod Anti-Affinity**

This isn't about node labels anymore — it's about **where other Pods are already running**.

### Real-World Scenarios

| Scenario | Affinity Type | Why |
|----------|--------------|-----|
| Web frontend + Redis cache on same node | Pod Affinity | Low latency between them |
| Two database replicas on different nodes | Pod Anti-Affinity | High availability |
| Monitoring agent co-located with the app | Pod Affinity | Needs access to app metrics |
| Two competing services on different nodes | Pod Anti-Affinity | Resource isolation |

### The Topology Key — Super Important!

Pod Affinity/Anti-Affinity uses a concept called `topologyKey`. This tells Kubernetes **what "same" means**:

| topologyKey | "Same" Means |
|-------------|-------------|
| `kubernetes.io/hostname` | Same **node** |
| `topology.kubernetes.io/zone` | Same **availability zone** |
| `topology.kubernetes.io/region` | Same **region** |

Think of it like this:
- `topologyKey: kubernetes.io/hostname` → "Put me on the **same machine**"
- `topologyKey: topology.kubernetes.io/zone` → "Put me in the **same data center zone**"

### Pod Affinity — "Schedule Me NEAR This Pod"

#### Example: Co-locate Frontend with Redis

```yaml
# pod-affinity-example.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-frontend
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web-frontend
  template:
    metadata:
      labels:
        app: web-frontend
    spec:
      affinity:
        podAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
          - labelSelector:
              matchExpressions:
              - key: app
                operator: In
                values:
                - redis-cache       # Find pods with label app=redis-cache
            topologyKey: kubernetes.io/hostname   # Put me on the SAME NODE
      containers:
      - name: frontend
        image: my-frontend:v1
```

**What happens:**
1. Scheduler looks for any running Pod with label `app=redis-cache`
2. Let's say that Pod is on `node-3`
3. Since `topologyKey` is `hostname`, "same topology" = "same node"
4. So, the frontend Pod gets scheduled on `node-3` too!

> ⚠️ If no Pod with `app=redis-cache` exists yet, the frontend Pod will stay **Pending**. Make sure the target Pod exists first!

#### Soft Pod Affinity — Prefer, Don't Require

```yaml
spec:
  affinity:
    podAffinity:
      preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 100
        podAffinityTerm:
          labelSelector:
            matchExpressions:
            - key: app
              operator: In
              values:
              - redis-cache
          topologyKey: kubernetes.io/hostname
```

Now if Redis isn't available on any node, the frontend Pod **still gets scheduled** somewhere. It's just a preference.

### Pod Anti-Affinity — "Schedule Me AWAY From This Pod"

This is the **most commonly used** feature in production. It ensures high availability by spreading replicas across nodes/zones.

#### Example: Spread Replicas Across Nodes

```yaml
# anti-affinity-spread.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web-app
  template:
    metadata:
      labels:
        app: web-app
    spec:
      affinity:
        podAntiAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
          - labelSelector:
              matchExpressions:
              - key: app
                operator: In
                values:
                - web-app           # Don't put me near OTHER web-app pods
            topologyKey: kubernetes.io/hostname   # "Near" = same node
      containers:
      - name: web
        image: nginx:1.25
```

**What happens:**
1. Replica 1 gets scheduled on `node-2` ✅
2. Replica 2: "Is there a `web-app` Pod on `node-2`? YES → skip. Is there one on `node-3`? NO → go here" ✅
3. Replica 3: "Is there a `web-app` Pod on `node-2`? YES. On `node-3`? YES. On `node-4`? NO → go here" ✅
4. Result: **one replica per node**!

```
node-2: [web-app-replica-1]
node-3: [web-app-replica-2]  
node-4: [web-app-replica-3]
```

> ⚠️ If you have 3 replicas but only 2 worker nodes, the 3rd replica will stay **Pending** with hard anti-affinity!

#### Spread Across Availability Zones

```yaml
spec:
  affinity:
    podAntiAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
      - labelSelector:
          matchExpressions:
          - key: app
            operator: In
            values:
            - web-app
        topologyKey: topology.kubernetes.io/zone   # Different ZONES, not just nodes
```

Now replicas get spread across **availability zones** — even better for disaster recovery!

#### Soft Anti-Affinity — Best Effort Spreading

In production, you often want soft anti-affinity so Pods **don't get stuck Pending**:

```yaml
# soft-anti-affinity.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-server
spec:
  replicas: 5
  selector:
    matchLabels:
      app: api-server
  template:
    metadata:
      labels:
        app: api-server
    spec:
      affinity:
        podAntiAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:
          - weight: 100
            podAffinityTerm:
              labelSelector:
                matchExpressions:
                - key: app
                  operator: In
                  values:
                  - api-server
              topologyKey: kubernetes.io/hostname
      containers:
      - name: api
        image: my-api:v1
```

**What happens:**
- Kubernetes **tries** to spread Pods across different nodes
- If there are only 3 nodes but 5 replicas, 2 nodes will get 2 Pods each
- No Pod stays Pending — it's a best-effort spread

## 🧩 Combining Everything — A Real-World Production Example

Here's a realistic example combining Node Affinity + Pod Anti-Affinity:

```yaml
# production-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: payment-service
  namespace: production
spec:
  replicas: 4
  selector:
    matchLabels:
      app: payment-service
  template:
    metadata:
      labels:
        app: payment-service
        tier: critical
    spec:
      affinity:
        # 1. NODE AFFINITY: Must run on production nodes, prefer us-east-1a
        nodeAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
            nodeSelectorTerms:
            - matchExpressions:
              - key: environment
                operator: In
                values:
                - production
              - key: node-type
                operator: NotIn
                values:
                - spot              # Never on spot instances!
          preferredDuringSchedulingIgnoredDuringExecution:
          - weight: 60
            preference:
              matchExpressions:
              - key: topology.kubernetes.io/zone
                operator: In
                values:
                - us-east-1a
          - weight: 40
            preference:
              matchExpressions:
              - key: topology.kubernetes.io/zone
                operator: In
                values:
                - us-east-1b
        
        # 2. POD ANTI-AFFINITY: Spread replicas across zones
        podAntiAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
          - labelSelector:
              matchExpressions:
              - key: app
                operator: In
                values:
                - payment-service
            topologyKey: kubernetes.io/hostname     # One per node (hard)
          preferredDuringSchedulingIgnoredDuringExecution:
          - weight: 100
            podAffinityTerm:
              labelSelector:
                matchExpressions:
                - key: app
                  operator: In
                  values:
                  - payment-service
              topologyKey: topology.kubernetes.io/zone  # Try different zones (soft)
        
        # 3. POD AFFINITY: Co-locate with the payment database
        podAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:
          - weight: 50
            podAffinityTerm:
              labelSelector:
                matchExpressions:
                - key: app
                  operator: In
                  values:
                  - payment-db
              topologyKey: topology.kubernetes.io/zone  # Same zone as DB
      
      containers:
      - name: payment
        image: payment-service:v2.1
        resources:
          requests:
            cpu: 500m
            memory: 512Mi
          limits:
            cpu: "1"
            memory: 1Gi
```

**What this does:**
1. ✅ Only runs on `production` nodes (never `staging` or `dev`)
2. ✅ Never runs on spot instances (critical service!)
3. ✅ Prefers zone `us-east-1a` (weight 60) over `us-east-1b` (weight 40)
4. ✅ One replica per node (hard anti-affinity by hostname)
5. ✅ Tries to spread across zones (soft anti-affinity by zone)
6. ✅ Prefers to be in the same zone as the payment database (low latency)

## 🔍 Debugging Scheduling Issues

### Pod Stuck in Pending?

```bash
# Check why a Pod isn't getting scheduled
kubectl describe pod <pod-name>
```

Look at the **Events** section at the bottom:

```
Events:
  Type     Reason            Message
  ----     ------            -------
  Warning  FailedScheduling  0/4 nodes are available: 
           1 node(s) had untolerated taint, 
           2 node(s) didn't match Pod's node affinity/selector, 
           1 node(s) didn't match pod anti-affinity rules.
```

This tells you exactly **why** the scheduler couldn't find a node.

### Check Node Labels

```bash
# See all labels on all nodes
kubectl get nodes --show-labels

# See labels on a specific node
kubectl get node node-2 -o jsonpath='{.metadata.labels}' | python -m json.tool

# Find nodes with a specific label
kubectl get nodes -l disktype=ssd
kubectl get nodes -l "topology.kubernetes.io/zone in (us-east-1a, us-east-1b)"
```

### Check Where Pods Are Running

```bash
# See which node each Pod is on
kubectl get pods -o wide

# See Pods on a specific node
kubectl get pods --field-selector spec.nodeName=node-2
```

## 📊 nodeSelector vs Node Affinity vs Pod Affinity/Anti-Affinity — Comparison

| Feature | nodeSelector | Node Affinity | Pod Affinity/Anti-Affinity |
|---------|-------------|---------------|---------------------------|
| **Complexity** | Simple | Medium | Complex |
| **Hard rules** | ✅ Yes | ✅ Yes | ✅ Yes |
| **Soft rules (preferences)** | ❌ No | ✅ Yes | ✅ Yes |
| **Operators (In, NotIn, etc.)** | ❌ No | ✅ Yes | ✅ Yes |
| **OR logic** | ❌ No | ✅ Yes | ✅ Yes |
| **Schedule relative to nodes** | ✅ Yes | ✅ Yes | ❌ No |
| **Schedule relative to Pods** | ❌ No | ❌ No | ✅ Yes |
| **Performance impact** | Minimal | Minimal | Can be high in large clusters |
| **When to use** | Simple, static needs | Flexible node targeting | HA, co-location, spreading |

---

## 💡 Best Practices

| Practice | Why |
|----------|-----|
| **Use soft (preferred) rules when possible** | Hard rules can make Pods permanently Pending |
| **Use Pod Anti-Affinity for HA** | Spread replicas across nodes/zones |
| **Don't overuse Pod Affinity** | It's expensive for the scheduler in large clusters (100+ nodes) |
| **Use built-in topology labels** | `topology.kubernetes.io/zone` etc. are already there |
| **Combine with resource requests** | Affinity doesn't check resources — a Pod can land on a matching but overloaded node |
| **Test with `kubectl describe`** | Always check Events if scheduling fails |
| **Prefer `preferredDuringScheduling` for non-critical workloads** | Avoids Pending pods when the cluster is tight |

## ⚡ Quick Reference — YAML Cheat Sheet

```yaml
# nodeSelector (simplest)
spec:
  nodeSelector:
    label-key: label-value

# Node Affinity (required)
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: KEY
            operator: In|NotIn|Exists|DoesNotExist|Gt|Lt
            values: [val1, val2]

# Node Affinity (preferred)
spec:
  affinity:
    nodeAffinity:
      preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 1-100
        preference:
          matchExpressions:
          - key: KEY
            operator: In
            values: [val1]

# Pod Affinity
spec:
  affinity:
    podAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
      - labelSelector:
          matchExpressions:
          - key: app
            operator: In
            values: [target-app]
        topologyKey: kubernetes.io/hostname

# Pod Anti-Affinity
spec:
  affinity:
    podAntiAffinity:
      preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 100
        podAffinityTerm:
          labelSelector:
            matchExpressions:
            - key: app
              operator: In
              values: [my-app]
          topologyKey: kubernetes.io/hostname
```

---

## 🧠 Key Takeaways

1. **nodeSelector** → Simple key-value matching. Good for basic needs, but limited.
2. **Node Affinity** → Powerful version of nodeSelector. Supports operators, soft/hard rules, OR/AND logic.
3. **Pod Affinity** → "Put me near this other Pod." Great for co-location (latency-sensitive apps).
4. **Pod Anti-Affinity** → "Keep me away from this other Pod." Essential for **high availability**.
5. **`topologyKey`** → Defines what "same location" means (same node, same zone, same region).
6. **Weight (1-100)** → For soft rules, higher weight = stronger preference.
7. **Always check `kubectl describe pod`** when scheduling fails — it tells you exactly what went wrong.
