# 10.26 — Taints & Tolerations

> **One Line Answer:** Taints are like a **"No Entry" sign** on a node — they repel Pods. Tolerations are like a **"VIP pass"** on a Pod — they allow that Pod to ignore the No Entry sign and still get scheduled on the tainted node.

## 🤔 Let's Understand the Concept First

In the previous chapter (10.25), we learned about **Node Affinity** — where a Pod says:

> "I **want** to go to THIS node."

Taints & Tolerations work the **opposite way** — a Node says:

> "I **don't want** any random Pod landing on me. Only special Pods that can **tolerate** my taint are allowed."

### Think of It Like a Nightclub 🎵

| Concept | Nightclub Analogy |
|---------|-------------------|
| **Node** | The nightclub |
| **Taint** | A sign at the door: "Members Only" or "VIP Only" |
| **Pod** | A person trying to enter |
| **Toleration** | A VIP membership card or guest pass |
| **No Toleration** | Person gets rejected at the door |

So:
- **Affinity** = A person says "I want to go to THAT nightclub" (Pod chooses node)
- **Taint** = A nightclub says "Only VIPs allowed" (Node rejects certain Pods)

**They're complementary systems.** In production, you usually use BOTH together.

## 🎯 Why Do We Need Taints & Tolerations?

Here are real-world scenarios:

| Scenario | How Taints Help |
|----------|-----------------|
| **Dedicated nodes for a specific team** | Taint the nodes; only that team's Pods have tolerations |
| **GPU nodes only for ML workloads** | Taint GPU nodes; only ML Pods can tolerate it |
| **Control plane nodes** | Kubernetes automatically taints them so your app Pods don't land there |
| **Node maintenance** | Taint a node to evict all Pods before upgrading |
| **Spot/preemptible instances** | Taint them; only fault-tolerant Pods go there |
| **Nodes with special hardware** | Taint SSD/NVMe nodes; only database Pods allowed |

## 📌 How Taints Work — Adding a Taint to a Node

### Taint Syntax

A taint has three parts:

```
key=value:effect
```

| Part | Description | Example |
|------|-------------|---------|
| **key** | A label-like identifier | `dedicated`, `hardware`, `env` |
| **value** | The value (optional) | `gpu`, `team-data`, `staging` |
| **effect** | What happens to Pods that can't tolerate it | `NoSchedule`, `PreferNoSchedule`, `NoExecute` |

### The Three Effects

| Effect | What It Does | Existing Pods? |
|--------|-------------|----------------|
| `NoSchedule` | **Don't schedule** new Pods here (unless they tolerate it) | Already running Pods are **NOT affected** |
| `PreferNoSchedule` | **Try not to schedule** here, but it's okay if there's no other option | Already running Pods are **NOT affected** |
| `NoExecute` | **Don't schedule** new Pods AND **evict** existing Pods that don't tolerate it | Already running Pods **get evicted** |

Let's visualize:

```
NoSchedule:
  ┌─────────────────────────┐
  │  Node (Tainted)         │
  │                         │
  │  [Existing Pod A] ✅    │  ← Already here, stays
  │  [New Pod B] ❌         │  ← Rejected! Can't come in
  │  [New Pod C + Pass] ✅  │  ← Has toleration, allowed!
  └─────────────────────────┘

NoExecute:
  ┌─────────────────────────┐
  │  Node (Tainted)         │
  │                         │
  │  [Existing Pod A] ❌ 🚪 │  ← EVICTED! Get out!
  │  [New Pod B] ❌         │  ← Rejected!
  │  [New Pod C + Pass] ✅  │  ← Has toleration, can stay
  └─────────────────────────┘

PreferNoSchedule:
  ┌─────────────────────────┐
  │  Node (Tainted)         │
  │                         │
  │  [Existing Pod A] ✅    │  ← Already here, stays  
  │  [New Pod B] ⚠️ Maybe   │  ← Last resort, if no other nodes
  │  [New Pod C + Pass] ✅  │  ← Has toleration, welcome!
  └─────────────────────────┘
```

### Adding a Taint to a Node

```bash
# Basic syntax
kubectl taint nodes <node-name> <key>=<value>:<effect>

# Example: Taint a node for GPU workloads only
kubectl taint nodes node-3 hardware=gpu:NoSchedule

# Example: Taint a node for team-data only
kubectl taint nodes node-4 dedicated=team-data:NoSchedule

# Example: Mark a node for maintenance (evicts existing Pods!)
kubectl taint nodes node-2 maintenance=true:NoExecute

# Taint without a value (just a key)
kubectl taint nodes node-3 gpu:NoSchedule
```

### Viewing Taints on a Node

```bash
# Method 1: describe the node
kubectl describe node node-3 | grep -A 5 "Taints"

# Output:
# Taints:  hardware=gpu:NoSchedule

# Method 2: JSON output
kubectl get node node-3 -o jsonpath='{.spec.taints}'
```

### Removing a Taint

Just add a **minus sign (-)** at the end:

```bash
# Remove a specific taint
kubectl taint nodes node-3 hardware=gpu:NoSchedule-

# Remove taint with just a key (no value)
kubectl taint nodes node-3 gpu:NoSchedule-

# Remove ALL taints with a specific key (regardless of effect)
kubectl taint nodes node-3 hardware-
```

## 🎫 How Tolerations Work — Adding a Toleration to a Pod

A toleration is added to a **Pod spec** and says: "I can tolerate this taint."

### Toleration Syntax

```yaml
spec:
  tolerations:
  - key: "key"
    operator: "Equal"      # or "Exists"
    value: "value"
    effect: "NoSchedule"   # or "PreferNoSchedule" or "NoExecute"
```

### The Two Operators

| Operator | Meaning | Value Required? |
|----------|---------|----------------|
| `Equal` | Key AND value must match the taint | Yes |
| `Exists` | Only the key needs to match (any value) | No |

### Example 1: Basic Taint + Toleration

**Step 1:** Taint the node:

```bash
kubectl taint nodes node-3 hardware=gpu:NoSchedule
```

**Step 2:** Create a Pod WITH the toleration:

```yaml
# gpu-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: ml-training
spec:
  tolerations:
  - key: "hardware"
    operator: "Equal"
    value: "gpu"
    effect: "NoSchedule"
  containers:
  - name: trainer
    image: my-ml-model:v1
```

```bash
kubectl apply -f gpu-pod.yaml
kubectl get pod ml-training -o wide
```

```
NAME          READY   STATUS    NODE
ml-training   1/1     Running   node-3    # ✅ Scheduled on the tainted GPU node!
```

**Step 3:** Try a Pod WITHOUT the toleration:

```yaml
# regular-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: regular-app
spec:
  containers:
  - name: app
    image: nginx:1.25
```

```bash
kubectl apply -f regular-pod.yaml
kubectl describe pod regular-app
```

```
Events:
  Warning  FailedScheduling  0/4 nodes are available:
    1 node(s) had untolerated taint {hardware: gpu}    # ❌ Rejected from node-3!
```

### Example 2: Using the Exists Operator

If you don't care about the taint's value, just that the key exists:

```yaml
spec:
  tolerations:
  - key: "hardware"
    operator: "Exists"       # Matches hardware=gpu, hardware=fpga, hardware=anything
    effect: "NoSchedule"
```

This tolerates ANY taint with the key `hardware`, regardless of its value.

### Example 3: Tolerate ALL Taints with a Specific Key

```yaml
spec:
  tolerations:
  - key: "hardware"
    operator: "Exists"
    # No effect specified = matches ALL effects (NoSchedule, PreferNoSchedule, NoExecute)
```

### Example 4: Tolerate EVERYTHING (Nuclear Option)

```yaml
spec:
  tolerations:
  - operator: "Exists"        # No key, no value, no effect = tolerate ALL taints
```

> ⚠️ This Pod will be scheduled on ANY node, even heavily tainted ones. Use with caution! (DaemonSets often use this — we'll see why later.)

### Example 5: NoExecute with tolerationSeconds

When you taint a node with `NoExecute`, existing Pods get evicted immediately. But you can give Pods a **grace period** with `tolerationSeconds`:

```yaml
spec:
  tolerations:
  - key: "maintenance"
    operator: "Equal"
    value: "true"
    effect: "NoExecute"
    tolerationSeconds: 300     # Stay for 300 seconds (5 minutes), then get evicted
```

**What happens:**
1. Node gets tainted with `maintenance=true:NoExecute`
2. Pods WITHOUT any toleration → evicted **immediately**
3. Pods with this toleration → stay for **5 more minutes**, then get evicted
4. Pods with the toleration but NO `tolerationSeconds` → stay **forever** (tolerate indefinitely)

This is super useful for **graceful node draining** — gives your app time to finish ongoing requests.

## 🏗️ Built-in Kubernetes Taints

Kubernetes automatically applies some taints. You should know about them:

### Control Plane Taint

```bash
kubectl describe node <control-plane-node> | grep Taint
```

```
Taints: node-role.kubernetes.io/control-plane:NoSchedule
```

This is why your application Pods **never** get scheduled on the control plane node! Kubernetes adds this taint automatically.

If you're running a single-node cluster (like Minikube) and WANT Pods on the control plane:

```bash
# Remove the control plane taint
kubectl taint nodes <control-plane-node> node-role.kubernetes.io/control-plane:NoSchedule-
```

### Automatic Taints Added by Kubernetes

Kubernetes can automatically taint nodes when problems are detected:

| Taint | When It's Added | Effect |
|-------|----------------|--------|
| `node.kubernetes.io/not-ready` | Node is not ready | `NoExecute` |
| `node.kubernetes.io/unreachable` | Node is unreachable | `NoExecute` |
| `node.kubernetes.io/disk-pressure` | Node disk is full | `NoSchedule` |
| `node.kubernetes.io/memory-pressure` | Node is running out of memory | `NoSchedule` |
| `node.kubernetes.io/pid-pressure` | Node is running out of process IDs | `NoSchedule` |
| `node.kubernetes.io/network-unavailable` | Node network is down | `NoSchedule` |
| `node.kubernetes.io/unschedulable` | Node is cordoned | `NoSchedule` |

> 💡 Kubernetes automatically adds tolerations for `not-ready` and `unreachable` to every Pod with `tolerationSeconds: 300` (5 minutes). That's why when a node goes down, Pods wait 5 minutes before being rescheduled to another node.

## 🔧 Real-World Scenarios with Full Examples

### Scenario 1: Dedicated Nodes for Different Teams

```bash
# Taint nodes for different teams
kubectl taint nodes node-2 team=frontend:NoSchedule
kubectl taint nodes node-3 team=backend:NoSchedule
kubectl taint nodes node-4 team=data:NoSchedule
```

Frontend team's Deployment:

```yaml
# frontend-deploy.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: frontend-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: frontend
  template:
    metadata:
      labels:
        app: frontend
    spec:
      tolerations:
      - key: "team"
        operator: "Equal"
        value: "frontend"
        effect: "NoSchedule"
      nodeSelector:
        team: frontend           # Also add nodeSelector to ATTRACT to these nodes
      containers:
      - name: frontend
        image: frontend:v1
```

> 🔑 **Important:** Tolerations **only allow** a Pod to go to a tainted node. They don't **force** the Pod to go there. Without `nodeSelector` or `nodeAffinity`, the Pod might still land on an untainted node! Always combine taints with affinity for dedicated nodes.

### Scenario 2: Node Maintenance — Graceful Drain

```bash
# Step 1: Cordon the node (prevents new Pods from being scheduled)
kubectl cordon node-2

# Step 2: Taint with NoExecute to evict existing Pods
kubectl taint nodes node-2 maintenance=upgrading:NoExecute

# Step 3: Do your maintenance (upgrade kernel, patches, etc.)
# ...

# Step 4: Uncordon and remove taint
kubectl taint nodes node-2 maintenance=upgrading:NoExecute-
kubectl uncordon node-2
```

Or use the built-in drain command (which does all of this for you):

```bash
kubectl drain node-2 --ignore-daemonsets --delete-emptydir-data
```

### Scenario 3: Spot/Preemptible Instances

```bash
# Taint spot instances
kubectl taint nodes spot-node-1 cloud.google.com/gke-spot=true:NoSchedule
kubectl taint nodes spot-node-2 cloud.google.com/gke-spot=true:NoSchedule
```

Only fault-tolerant, non-critical workloads should go here:

```yaml
# batch-job.yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: data-processing
spec:
  template:
    spec:
      tolerations:
      - key: "cloud.google.com/gke-spot"
        operator: "Equal"
        value: "true"
        effect: "NoSchedule"
      nodeSelector:
        cloud.google.com/gke-spot: "true"      # Attract to spot nodes (cheaper!)
      containers:
      - name: processor
        image: data-processor:v1
      restartPolicy: OnFailure
```

### Scenario 4: DaemonSets That Run Everywhere

DaemonSets (like monitoring agents) often need to run on EVERY node, even tainted ones:

```yaml
# monitoring-daemonset.yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: node-exporter
spec:
  selector:
    matchLabels:
      app: node-exporter
  template:
    metadata:
      labels:
        app: node-exporter
    spec:
      tolerations:
      - operator: "Exists"         # Tolerate ALL taints — run on every node
      containers:
      - name: node-exporter
        image: prom/node-exporter:latest
        ports:
        - containerPort: 9100
```

## 🆚 Taints & Tolerations vs Affinity — When to Use What?

| Feature | Taints & Tolerations | Node Affinity |
|---------|---------------------|---------------|
| **Direction** | Node repels Pods | Pod attracts to nodes |
| **Who decides?** | Node admin | Pod/Deployment author |
| **Default behavior** | Pods are EXCLUDED unless tolerated | Pods go ANYWHERE unless restricted |
| **Use case** | Restrict which Pods CAN run on a node | Specify which nodes a Pod WANTS to run on |
| **Can evict running Pods?** | ✅ Yes (NoExecute) | ❌ No |
| **Soft/Hard rules?** | PreferNoSchedule / NoSchedule | preferred / required |

### The Golden Rule

> Use **Taints** when the **Node admin** wants to control what runs on their nodes.
>
> Use **Affinity** when the **Pod owner** wants to choose where their Pod runs.
>
> Use **BOTH together** for dedicated nodes.

```
Dedicated Node Setup:
  
  1. TAINT the node     → "Only special Pods allowed" (repel others)
  2. AFFINITY on Pod    → "I want to go to THAT node" (attract Pod)
  3. TOLERATION on Pod  → "I can handle that node's taint" (VIP pass)
  
  All three together = Pod goes to the right node, and ONLY the right Pods go there.
```

## 🧪 Hands-On Lab: Testing Taints & Tolerations

Let's do a quick hands-on exercise:

```bash
# 1. Check current taints on all nodes
kubectl describe nodes | grep -A 3 "Taints"

# 2. Taint a node
kubectl taint nodes node-2 env=test:NoSchedule

# 3. Try to schedule a Pod (should avoid node-2)
kubectl run test-pod --image=nginx:1.25
kubectl get pod test-pod -o wide     # Check which node it's on

# 4. Create a Pod with toleration
cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: Pod
metadata:
  name: tolerant-pod
spec:
  tolerations:
  - key: "env"
    operator: "Equal"
    value: "test"
    effect: "NoSchedule"
  nodeSelector:
    # Add if you want to FORCE it to node-2
    kubernetes.io/hostname: node-2
  containers:
  - name: nginx
    image: nginx:1.25
EOF

kubectl get pod tolerant-pod -o wide     # Should be on node-2

# 5. Clean up
kubectl delete pod test-pod tolerant-pod
kubectl taint nodes node-2 env=test:NoSchedule-
```

## 💡 Best Practices

| Practice | Why |
|----------|-----|
| **Always combine taints with affinity for dedicated nodes** | Taints alone don't guarantee Pods go TO the node, only that unwanted Pods stay away |
| **Use `PreferNoSchedule` for soft restrictions** | Avoids Pending Pods when cluster resources are tight |
| **Be careful with `NoExecute`** | It evicts running Pods! Use `tolerationSeconds` for grace |
| **Document your taints** | Other teams need to know why their Pods aren't scheduling |
| **Use meaningful key-value pairs** | `team=ml`, `hardware=gpu`, `env=staging` — not `foo=bar` |
| **DaemonSets should tolerate all taints** | Monitoring/logging agents need to run everywhere |
| **Don't taint too aggressively** | Over-tainting leads to resource waste (empty nodes) |

## ⚡ Quick Reference

```bash
# Add a taint
kubectl taint nodes <node> key=value:NoSchedule
kubectl taint nodes <node> key=value:NoExecute
kubectl taint nodes <node> key=value:PreferNoSchedule

# Remove a taint (note the minus at the end)
kubectl taint nodes <node> key=value:NoSchedule-
kubectl taint nodes <node> key-                    # Remove all taints with that key

# View taints
kubectl describe node <node> | grep Taint
kubectl get nodes -o json | jq '.items[].spec.taints'
```

```yaml
# Toleration (Equal operator — must match key, value, and effect)
tolerations:
- key: "key"
  operator: "Equal"
  value: "value"
  effect: "NoSchedule"

# Toleration (Exists operator — matches any value for the key)
tolerations:
- key: "key"
  operator: "Exists"
  effect: "NoSchedule"

# Tolerate ALL taints (for DaemonSets)
tolerations:
- operator: "Exists"

# Tolerate with eviction grace period
tolerations:
- key: "key"
  operator: "Equal"
  value: "value"
  effect: "NoExecute"
  tolerationSeconds: 3600    # Stay for 1 hour, then get evicted
```

## 🧠 Key Takeaways

1. **Taints** go on **Nodes** — they **repel** Pods.
2. **Tolerations** go on **Pods** — they **allow** Pods to be scheduled on tainted nodes.
3. Three effects: **NoSchedule** (block new), **PreferNoSchedule** (soft block), **NoExecute** (block new + evict existing).
4. Tolerations **allow** but don't **require** — combine with affinity/nodeSelector for dedicated nodes.
5. Kubernetes auto-taints the control plane and unhealthy nodes.
6. Use `tolerationSeconds` with `NoExecute` for graceful eviction.
7. DaemonSets typically tolerate all taints (`operator: Exists`).
8. **Affinity** = Pod says "I want to go there." **Taint** = Node says "You can't come here."
