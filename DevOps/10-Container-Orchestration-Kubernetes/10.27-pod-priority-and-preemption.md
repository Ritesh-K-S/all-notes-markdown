# 10.27 — Pod Priority & Preemption

> **One Line Answer:** Pod Priority lets you assign **importance levels** to your Pods, and Preemption allows the Kubernetes scheduler to **evict (kick out) lower-priority Pods** to make room for higher-priority ones when the cluster is full.

---

## 🤔 Why Do We Need This?

Imagine your Kubernetes cluster is running at **full capacity** — every node is packed with Pods, no CPU or memory left. Now, a **critical Pod** needs to be deployed:

```
Cluster Status: FULL (no resources available)

┌──────────┐  ┌──────────┐  ┌──────────┐
│  node-1   │  │  node-2   │  │  node-3   │
│ [Pod A]   │  │ [Pod D]   │  │ [Pod G]   │
│ [Pod B]   │  │ [Pod E]   │  │ [Pod H]   │
│ [Pod C]   │  │ [Pod F]   │  │ [Pod I]   │
│  FULL!    │  │  FULL!    │  │  FULL!    │
└──────────┘  └──────────┘  └──────────┘

🚨 NEW POD: "payment-service" (CRITICAL!) needs to run!
   → Where does it go? Everything is full!
```

Without Priority & Preemption, the `payment-service` Pod would just stay in `Pending` forever. That's terrible for a critical service!

With Priority & Preemption, Kubernetes can say:

> "Okay, `payment-service` is a **high-priority** Pod. Let me evict some **low-priority** Pod (like a batch job or a dev workload) to make room."

```
After Preemption:

┌──────────┐  ┌──────────┐  ┌──────────┐
│  node-1   │  │  node-2   │  │  node-3   │
│ [Pod A]   │  │ [Pod D]   │  │ [Pod G]   │
│ [Pod B]   │  │ [Pod E]   │  │ [Pod H]   │
│ [Pod C]   │  │ [PAYMENT] │  │ [Pod I]   │
│  FULL!    │  │  ✅ Done!  │  │  FULL!    │
└──────────┘  └──────────┘  └──────────┘
                  ↑
              Pod F (low priority) was evicted to make room!
```

---

## 🎯 Real-World Use Cases

| Scenario | How Priority Helps |
|----------|-------------------|
| **Payment processing** is more important than **log aggregation** | Payment gets high priority; log jobs get evicted if cluster is full |
| **Production workloads** vs **dev/test Pods** on a shared cluster | Production is high priority; dev Pods get evicted first |
| **System daemons** (monitoring, DNS) must always run | System-level priority ensures they're never evicted |
| **Batch jobs** are okay to restart | Low priority; they'll be re-scheduled when resources free up |
| **Emergency hotfix deployments** | Critical priority; makes room immediately |

---

## 📌 How It Works — The Two Pieces

Pod Priority & Preemption has two components:

| Component | What It Is | Where It's Defined |
|-----------|------------|-------------------|
| **PriorityClass** | A cluster-wide object that defines a priority **level** (a number) | Cluster-level resource (like a StorageClass) |
| **priorityClassName** | A reference in the Pod spec that says "I use THIS priority" | Pod/Deployment spec |

### The Flow

```
1. Admin creates PriorityClasses (e.g., critical=1000, default=0, low=-100)
           ↓
2. Deployment references a PriorityClass
           ↓
3. Pod gets created with that priority value
           ↓
4. Scheduler tries to find a node with enough resources
           ↓
5a. If resources available → Schedule normally ✅
5b. If NO resources available → Check if lower-priority Pods can be evicted
           ↓
6. Evict lower-priority Pods → Schedule the high-priority Pod ✅
```

## 🔨 Step 1: Creating PriorityClasses

A `PriorityClass` is a simple object with a name and a number:

```yaml
# priority-classes.yaml
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: critical
value: 1000000          # Higher number = higher priority
globalDefault: false    # Is this the default for Pods that don't specify one?
preemptionPolicy: PreemptLowerPriority   # Can this priority preempt others?
description: "For mission-critical production services like payments, auth, etc."
---
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: high-priority
value: 100000
globalDefault: false
preemptionPolicy: PreemptLowerPriority
description: "For important production services."
---
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: default-priority
value: 0
globalDefault: true      # ← This is the DEFAULT for all Pods that don't specify one
preemptionPolicy: PreemptLowerPriority
description: "Default priority for regular workloads."
---
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: low-priority
value: -1000
globalDefault: false
preemptionPolicy: Never    # ← This priority CANNOT preempt others
description: "For batch jobs, dev/test workloads that can be evicted."
---
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: best-effort
value: -10000
globalDefault: false
preemptionPolicy: Never
description: "Lowest priority. For non-essential background tasks."
```

Apply them:

```bash
kubectl apply -f priority-classes.yaml

# View all PriorityClasses
kubectl get priorityclasses
```

Output:

```
NAME                            VALUE        GLOBAL-DEFAULT   AGE
best-effort                     -10000       false            5s
critical                        1000000      false            5s
default-priority                0            true             5s
high-priority                   100000       false            5s
low-priority                    -1000        false            5s
system-cluster-critical         2000000000   false            30d   ← Built-in!
system-node-critical            2000001000   false            30d   ← Built-in!
```

### Understanding the Fields

| Field | Description |
|-------|-------------|
| `value` | An integer. **Higher = more important.** Can be negative. Max is 1,000,000,000 (1 billion) for user-defined classes. |
| `globalDefault` | If `true`, Pods without a `priorityClassName` get this priority. Only ONE PriorityClass should have this set to `true`. |
| `preemptionPolicy` | `PreemptLowerPriority` (default) — can evict lower-priority Pods. `Never` — cannot evict others, but still has scheduling priority. |
| `description` | Human-readable description. Optional but recommended. |

### Built-in System Priority Classes

Kubernetes comes with two built-in PriorityClasses:

| Name | Value | Used By |
|------|-------|---------|
| `system-node-critical` | 2,000,001,000 | Critical node-level Pods (e.g., `kube-proxy`, `kube-flannel`) |
| `system-cluster-critical` | 2,000,000,000 | Critical cluster-level Pods (e.g., `coredns`, `kube-dns`) |

> ⚠️ **Never** set your PriorityClass value higher than 1,000,000,000. Values above that are reserved for system components. If you set your priority higher than system Pods, you could preempt `coredns` or `kube-proxy` and **break your cluster**!

## ⚙️ How Preemption Works — The Detailed Flow

Let's walk through exactly what happens when preemption kicks in:

### Step-by-Step Preemption Process

```
1. High-priority Pod is created
   └→ Scheduler tries to find a node with enough resources

2. No node has enough resources → Pod is "unschedulable"
   └→ Scheduler activates the PREEMPTION logic

3. Scheduler scans ALL nodes looking for preemption candidates
   └→ For each node, it asks: "If I evict some lower-priority Pods,
      will there be enough resources for the new Pod?"

4. Scheduler picks the BEST node to preempt from
   └→ Criteria:
      - Minimize the number of Pods to evict
      - Prefer evicting the LOWEST priority Pods
      - Prefer nodes where preemption breaks the fewest PodDisruptionBudgets

5. Selected low-priority Pods receive a DELETION signal
   └→ They get their graceful termination period (terminationGracePeriodSeconds)
   └→ Default: 30 seconds to shut down gracefully

6. Once Pods are terminated, the high-priority Pod gets scheduled
   └→ It's now running on the freed-up resources ✅

7. The evicted Pods go back to the scheduling queue
   └→ They'll be rescheduled when resources become available
   └→ (If they're part of a Deployment/Job, new Pods are created automatically)
```

### Visual Example

**Before:** Cluster is full

```
node-2 (4 CPU, 8Gi Memory total):
┌──────────────────────────────────────┐
│ Pod A (priority: 0)    - 1 CPU, 2Gi │
│ Pod B (priority: -1000) - 2 CPU, 4Gi│
│ Pod C (priority: 0)    - 1 CPU, 2Gi │
│ FREE: 0 CPU, 0 Gi                   │
└──────────────────────────────────────┘

New Pod: payment-service (priority: 1000000) needs 2 CPU, 3Gi
→ No room! Time to preempt!
```

**Preemption Decision:**

```
Scheduler thinks:
  "If I evict Pod B (priority -1000, using 2 CPU + 4Gi)..."
  "I free up 2 CPU and 4Gi"
  "payment-service needs 2 CPU and 3Gi"
  "That's enough! Let's evict Pod B."
  
  Why Pod B? Because it has the LOWEST priority (-1000).
  Pod A and Pod C have priority 0, which is higher than -1000.
```

**After Preemption:**

```
node-2:
┌──────────────────────────────────────┐
│ Pod A (priority: 0)    - 1 CPU, 2Gi │
│ payment-svc (1000000)  - 2 CPU, 3Gi │
│ Pod C (priority: 0)    - 1 CPU, 2Gi │
│ FREE: 0 CPU, 1Gi                    │
└──────────────────────────────────────┘

Pod B → evicted, goes back to the scheduling queue.
        Will be rescheduled when resources are available.
```

## 🛡️ preemptionPolicy — Control Who Can Preempt

The `preemptionPolicy` field has two options:

### PreemptLowerPriority (Default)

```yaml
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: high-priority
value: 100000
preemptionPolicy: PreemptLowerPriority    # Can evict lower-priority Pods
```

This Pod CAN preempt (evict) lower-priority Pods when the cluster is full.

### Never — Priority Without Preemption

```yaml
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: high-no-preempt
value: 100000
preemptionPolicy: Never    # High priority for SCHEDULING ORDER, but can't evict others
```

**What this does:**
- The Pod gets **scheduled before** lower-priority Pods in the scheduling queue
- But it **cannot evict** running Pods to make room
- If there's no room, it stays `Pending` until resources free up naturally

**When to use `Never`:**
- You want your Pod to be scheduled first when resources are available
- But you don't want it to disrupt (evict) any running workloads
- Good for important but not urgent workloads

### Scheduling Order vs Preemption — What's the Difference?

```
Scheduling Queue (ordered by priority):
  1. payment-service    (priority: 1000000) ← Scheduled first
  2. web-app            (priority: 100000)  ← Scheduled second
  3. batch-job          (priority: 0)       ← Scheduled third
  4. dev-test           (priority: -10000)  ← Scheduled last

This ordering happens REGARDLESS of preemptionPolicy.
PreemptionPolicy only controls whether a Pod can EVICT running Pods.
```

## ⚠️ Important Gotchas and Edge Cases

### 1. Preemption Doesn't Guarantee Immediate Scheduling

After evicting a Pod, the preempted Pod gets a **graceful termination period** (default 30 seconds). The high-priority Pod must wait until the resources are actually freed.

```yaml
# If a Pod has a long graceful termination period:
spec:
  terminationGracePeriodSeconds: 300   # 5 minutes!
```

The high-priority Pod will wait up to 5 minutes for this Pod to terminate.

### 2. PodDisruptionBudgets (PDBs) Are Respected

If a low-priority Pod is protected by a PDB, the scheduler **tries** to respect it. But if preemption is the only option and the PDB prevents it, the scheduler may still preempt — PDBs are best-effort during preemption.

### 3. No Guarantee of Same-Node Rescheduling

When a low-priority Pod is evicted, it goes back to the scheduling queue. There's **no guarantee** it'll be scheduled on the same node or even at all (if the cluster stays full).

### 4. Inter-Pod Affinity and Preemption

If evicting Pod X would break the affinity rules of Pod Y (which depends on Pod X being on the same node), the scheduler considers this. It tries not to create cascading disruptions.

### 5. Pods Without a PriorityClass

If you don't specify a `priorityClassName` and no PriorityClass has `globalDefault: true`, the Pod gets priority value **0**.

```bash
# Check if you have a global default
kubectl get priorityclasses -o custom-columns=NAME:.metadata.name,VALUE:.value,DEFAULT:.globalDefault
```

### 6. Can't Change Priority After Creation

Once a Pod is created, you **cannot change its priority**. To change priority, you must delete and recreate the Pod (or update the Deployment, which triggers a rolling update).

## 📊 Priority vs Other Scheduling Features — How They Interact

| Feature | Purpose | Works With Priority? |
|---------|---------|---------------------|
| **Resource Requests/Limits** | Define how much CPU/memory a Pod needs | Yes — preemption frees up resources |
| **Node Affinity** | Attract Pod to specific nodes | Yes — preemption only happens on nodes that match affinity |
| **Taints & Tolerations** | Restrict which Pods can go on a node | Yes — preempted Pod must tolerate node's taints |
| **Pod Anti-Affinity** | Spread Pods across nodes | Yes — scheduler considers anti-affinity during preemption |
| **PodDisruptionBudgets** | Limit voluntary disruptions | Best-effort during preemption |
| **HPA** | Auto-scale based on metrics | HPA may create high-priority Pods that trigger preemption |

---

## 💡 Best Practices

| Practice | Why |
|----------|-----|
| **Define a clear priority hierarchy** | 3-5 tiers is usually enough. Don't create dozens of levels. |
| **Set a `globalDefault` PriorityClass** | So Pods without a priority get a reasonable default |
| **Use `preemptionPolicy: Never` for batch/dev workloads** | They get high scheduling priority without disrupting others |
| **Never set priority > 1,000,000,000** | Reserved for system components. You could break your cluster! |
| **Always set resource requests** | Preemption depends on resource accounting. Without requests, the scheduler can't calculate what to free |
| **Use PodDisruptionBudgets for critical services** | Extra protection against accidental preemption |
| **Monitor preemption events** | Set up alerts for `Preempted` events in your monitoring stack |
| **Document your priority tiers** | Teams need to know which PriorityClass to use |
| **Test preemption in staging first** | Don't learn about preemption behavior in production! |

## 🧠 Key Takeaways

1. **PriorityClass** defines importance levels (a number). Higher number = more important.
2. **Preemption** lets high-priority Pods **evict** low-priority Pods when the cluster is full.
3. **Three built-in tiers**: `system-node-critical` > `system-cluster-critical` > your custom classes.
4. **`globalDefault: true`** sets the default priority for Pods that don't specify one.
5. **`preemptionPolicy: Never`** = high scheduling priority, but can't evict others. Great for batch/dev.
6. **Always set resource requests** — preemption needs them to calculate resource needs.
7. **Preemption is graceful** — evicted Pods get their `terminationGracePeriodSeconds`.
8. **Evicted Pods go back to the queue** — they'll be rescheduled when resources are available.
9. **Don't set priority > 1 billion** — that's reserved for Kubernetes system components.
10. **3-5 priority tiers** is the sweet spot. Don't over-complicate.
