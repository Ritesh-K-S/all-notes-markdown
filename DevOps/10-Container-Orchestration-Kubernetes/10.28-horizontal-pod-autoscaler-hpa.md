# 10.28 — Horizontal Pod Autoscaler (HPA)

> **One Line Answer:** The Horizontal Pod Autoscaler (HPA) **automatically increases or decreases the number of Pod replicas** in a Deployment, ReplicaSet, or StatefulSet based on observed metrics like CPU usage, memory usage, or custom metrics — so your app scales out when traffic spikes and scales in when things calm down, without you lifting a finger.

## 🤔 Why Do We Need HPA?

Let's understand the problem first.

### The Problem: Fixed Number of Pods

Imagine you have a web app running with 3 replicas. It works fine during normal hours. But what happens during a flash sale, or when a viral tweet sends a flood of users your way?

```
Normal Hours (3 pods = perfectly fine):

   👤👤👤👤👤  →  [Pod 1] ✅ handles 5 users
   👤👤👤👤👤  →  [Pod 2] ✅ handles 5 users
   👤👤👤👤👤  →  [Pod 3] ✅ handles 5 users

   Total: 15 users → 3 pods → Everyone is happy 😊


Flash Sale (3 pods = DISASTER):

   👤👤👤👤👤👤👤👤👤👤  →  [Pod 1] 😰 CPU: 95%! SLOW!
   👤👤👤👤👤👤👤👤👤👤  →  [Pod 2] 😰 CPU: 98%! CRASHING!
   👤👤👤👤👤👤👤👤👤👤  →  [Pod 3] 💀 CPU: 100%! OOMKilled!
   👤👤👤👤👤👤👤👤👤👤  →  ??? NO MORE PODS! Users get 503 errors!

   Total: 40 users → 3 pods → Website is DOWN 💥
```

And what about 3 AM when no one is using your app? Those 3 pods are still running, eating CPU and memory — wasting your cloud bill! 💸

### The Solution: HPA

```
With HPA — Auto Scales Based on Load:

Normal Hours:
   👤👤👤👤👤  →  [Pod 1] ✅
   👤👤👤👤👤  →  [Pod 2] ✅
   👤👤👤👤👤  →  [Pod 3] ✅
   HPA says: "CPU at 50%. 3 pods is fine." 😊


Flash Sale (traffic spikes!):
   HPA says: "CPU at 90%! I need MORE pods!"
   
   👤👤👤👤👤  →  [Pod 1] ✅
   👤👤👤👤👤  →  [Pod 2] ✅
   👤👤👤👤👤  →  [Pod 3] ✅
   👤👤👤👤👤  →  [Pod 4] ✅  ← HPA created this!
   👤👤👤👤👤  →  [Pod 5] ✅  ← HPA created this!
   👤👤👤👤👤  →  [Pod 6] ✅  ← HPA created this!
   👤👤👤👤👤  →  [Pod 7] ✅  ← HPA created this!
   👤👤👤👤👤  →  [Pod 8] ✅  ← HPA created this!

   Total: 40 users → 8 pods → Everyone is happy 😊


3 AM (no traffic):
   HPA says: "CPU at 5%. Too many pods. Scaling down."
   
   👤  →  [Pod 1] ✅
              [Pod 2] ✅
   
   Total: 1 user → 2 pods → Saving money 💰
```

## 🏗️ How Does HPA Work? — The Big Picture

Here's the full flow of what happens behind the scenes:

```
                    ┌─────────────────────────────────┐
                    │   HPA Controller                │
                    │   (runs in kube-controller-      │
                    │    manager)                       │
                    │                                   │
                    │   Every 15 seconds (default):     │
                    │   1. Query metrics                │
                    │   2. Calculate desired replicas   │
                    │   3. Scale up/down if needed      │
                    └──────────┬───────────┬────────────┘
                               │           │
              ┌────────────────▼──┐   ┌────▼──────────────────┐
              │  Metrics Server   │   │  Deployment /         │
              │  (or Prometheus   │   │  ReplicaSet /         │
              │   Adapter)        │   │  StatefulSet          │
              │                   │   │                       │
              │  Collects CPU,    │   │  HPA changes the      │
              │  Memory, Custom   │   │  .spec.replicas field │
              │  metrics from     │   │  to scale up/down     │
              │  every Pod        │   │                       │
              └───────────────────┘   └───────────────────────┘
```

### Step-by-Step:

1. **Metrics Server** (or a custom metrics provider) collects real-time CPU/memory usage from every Pod via the kubelet.
2. **HPA Controller** queries the Metrics API every **15 seconds** (configurable).
3. HPA **calculates** the desired number of replicas using a formula.
4. If the desired replicas ≠ current replicas, HPA **updates** the Deployment's `.spec.replicas`.
5. The Deployment controller notices the change and **creates or deletes Pods** accordingly.

## 📋 Prerequisites — You NEED Metrics Server

HPA **cannot work** without a metrics source. The most basic one is the **Metrics Server**.

### What is Metrics Server?

Metrics Server is a lightweight, in-memory metrics aggregator. It collects CPU and memory usage from kubelets and exposes them via the Kubernetes Metrics API (`metrics.k8s.io`).

### Install Metrics Server

```bash
# Install Metrics Server (one command)
kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml
```

> **Note for Minikube users:**
> ```bash
> minikube addons enable metrics-server
> ```

> **Note for Kind users:** You may need to add `--kubelet-insecure-tls` flag to the Metrics Server deployment.

### Verify Metrics Server is Running

```bash
# Check if metrics-server pod is running
kubectl get pods -n kube-system | grep metrics-server

# Test if metrics API is working — this should show CPU/memory of your nodes
kubectl top nodes

# Test pod metrics
kubectl top pods
```

If `kubectl top nodes` shows actual numbers, you're good to go! If it says "Metrics API not available", the Metrics Server isn't set up correctly.

## 🛠️ Creating Your First HPA — Step by Step

Let's do a complete hands-on example.

### Step 1: Create a Deployment with Resource Requests

**This is critical!** HPA needs resource `requests` to calculate utilization percentage. Without requests, HPA can't work for CPU/memory-based scaling.

```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-web-app
spec:
  replicas: 2                          # Starting with 2 pods
  selector:
    matchLabels:
      app: my-web-app
  template:
    metadata:
      labels:
        app: my-web-app
    spec:
      containers:
      - name: web
        image: nginx:latest
        ports:
        - containerPort: 80
        resources:
          requests:                     # ⚠️ REQUIRED for HPA!
            cpu: "100m"                # Each pod "requests" 100 millicores
            memory: "128Mi"            # Each pod "requests" 128 MiB
          limits:
            cpu: "500m"                # Each pod can use up to 500 millicores
            memory: "256Mi"            # Each pod can use up to 256 MiB
```

**What do these resource values mean?**
- `cpu: "100m"` → 100 millicores = 0.1 CPU core (10% of one core)
- `cpu: "500m"` → 500 millicores = 0.5 CPU core (50% of one core)
- `memory: "128Mi"` → 128 Mebibytes of RAM

```bash
# Apply the deployment
kubectl apply -f deployment.yaml
```

### Step 2: Create an HPA (Imperative Way — Quick Command)

The fastest way to create an HPA:

```bash
# Create HPA: scale between 2 and 10 pods, target 50% CPU utilization
kubectl autoscale deployment my-web-app \
  --min=2 \
  --max=10 \
  --cpu-percent=50
```

**What this means:**
- `--min=2` → Never go below 2 pods (even at 3 AM)
- `--max=10` → Never go above 10 pods (cost protection!)
- `--cpu-percent=50` → If average CPU across all pods goes above 50% of their request (50% of 100m = 50m), scale up. If it drops below, scale down.

### Step 3: Check the HPA

```bash
# View HPA status
kubectl get hpa

# Output:
# NAME         REFERENCE               TARGETS   MINPODS   MAXPODS   REPLICAS   AGE
# my-web-app   Deployment/my-web-app   10%/50%   2         10        2          30s
```

**Reading the output:**
- `TARGETS: 10%/50%` → Current CPU is 10%, target is 50%. Since 10% < 50%, no scaling needed.
- `MINPODS: 2` → Minimum replicas.
- `MAXPODS: 10` → Maximum replicas.
- `REPLICAS: 2` → Currently running 2 pods.

```bash
# For more details
kubectl describe hpa my-web-app
```

## 📝 HPA YAML Manifest — The Declarative Way

For production, always use YAML manifests (declarative approach).

### Basic HPA — CPU Only

```yaml
# hpa-basic.yaml
apiVersion: autoscaling/v2          # Use v2 (v1 is outdated, v2beta2 is deprecated)
kind: HorizontalPodAutoscaler
metadata:
  name: my-web-app-hpa
spec:
  scaleTargetRef:                    # What to scale?
    apiVersion: apps/v1
    kind: Deployment
    name: my-web-app                 # The name of your Deployment
  minReplicas: 2                     # Minimum pods
  maxReplicas: 10                    # Maximum pods
  metrics:                           # What metrics to use?
  - type: Resource                   # Built-in resource metrics (CPU/Memory)
    resource:
      name: cpu                      # Scale based on CPU
      target:
        type: Utilization            # Percentage of resource request
        averageUtilization: 50       # Target: 50% CPU utilization
```

```bash
kubectl apply -f hpa-basic.yaml
```

### HPA — CPU + Memory

```yaml
# hpa-cpu-memory.yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: my-web-app-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-web-app
  minReplicas: 2
  maxReplicas: 15
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 60       # Scale up if CPU > 60%
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 70       # Scale up if Memory > 70%
```

**Important:** When you specify multiple metrics, HPA evaluates ALL of them and picks the one that results in the **highest** number of replicas. This ensures your app can handle the worst case.

```
Example:
  CPU says:  "I need 4 pods"
  Memory says: "I need 6 pods"
  
  HPA picks: 6 pods (the higher one) ✅
```

## 🧮 The HPA Scaling Formula — How It Calculates Replicas

Understanding the math helps you tune HPA correctly.

### The Formula

```
desiredReplicas = ceil( currentReplicas × (currentMetricValue / desiredMetricValue) )
```

### Example Walkthrough

Let's say:
- You have **3 pods** running
- Each pod **requests 100m CPU**
- Target utilization = **50%** (which means 50m per pod)
- Current average CPU usage = **90%** (which means 90m per pod)

```
desiredReplicas = ceil( 3 × (90 / 50) )
                = ceil( 3 × 1.8 )
                = ceil( 5.4 )
                = 6

HPA will scale from 3 pods → 6 pods! 🚀
```

Another example — scaling down:
- You have **6 pods** running
- Target utilization = **50%**
- Current average CPU usage = **20%**

```
desiredReplicas = ceil( 6 × (20 / 50) )
                = ceil( 6 × 0.4 )
                = ceil( 2.4 )
                = 3

HPA will scale from 6 pods → 3 pods 📉
(But not below minReplicas!)
```

## ⚙️ HPA Behavior — Controlling Scale Up/Down Speed

By default, HPA can be a bit aggressive with scaling up and too slow with scaling down. Kubernetes v1.23+ lets you fine-tune this with the `behavior` field.

### Why Does This Matter?

```
Without behavior tuning:

  Traffic spike → HPA instantly adds 20 pods → spike is just 2 seconds → 
  Now you have 20 pods doing nothing → Takes 5 minutes to scale back down → 💸💸💸

With behavior tuning:

  Traffic spike → HPA adds pods gradually (max 4 per minute) → 
  If spike is temporary, it doesn't over-provision → 
  Scale down is also controlled → 💰 Saves money!
```

### HPA with Behavior Configuration

```yaml
# hpa-with-behavior.yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: my-web-app-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-web-app
  minReplicas: 2
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 50
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 30     # Wait 30 sec before scaling up
      policies:
      - type: Pods
        value: 4                          # Add max 4 pods at a time
        periodSeconds: 60                 # Per 60-second window
      - type: Percent
        value: 100                        # Or double the pods (100% increase)
        periodSeconds: 60
      selectPolicy: Max                   # Use whichever policy gives MORE pods
    scaleDown:
      stabilizationWindowSeconds: 300    # Wait 5 min before scaling down (be cautious!)
      policies:
      - type: Pods
        value: 2                          # Remove max 2 pods at a time
        periodSeconds: 60                 # Per 60-second window
      - type: Percent
        value: 10                         # Or remove 10% of pods
        periodSeconds: 60
      selectPolicy: Min                   # Use whichever policy removes FEWER pods
```

**Let's break this down:**

| Field | What It Does |
|-------|-------------|
| `stabilizationWindowSeconds` | How long HPA looks back at metrics before deciding to scale. Prevents flapping (rapid up/down). |
| `policies` | Rules that define HOW MUCH to scale in each period. |
| `type: Pods` | Scale by a fixed number of pods. |
| `type: Percent` | Scale by a percentage of current replicas. |
| `periodSeconds` | The time window for the policy. |
| `selectPolicy: Max` | When multiple policies exist, pick the one that allows the MOST scaling. |
| `selectPolicy: Min` | When multiple policies exist, pick the one that allows the LEAST scaling. |
| `selectPolicy: Disabled` | Disables scaling in that direction entirely! |

### Prevent Scale Down Entirely (Useful for Critical Apps)

```yaml
behavior:
  scaleDown:
    selectPolicy: Disabled    # HPA will NEVER scale down automatically
```

This is useful when you want to scale up automatically but only scale down manually (for mission-critical apps where you never want to accidentally lose capacity).

## 📊 HPA with Custom Metrics

CPU and memory are great, but sometimes you need to scale based on business-specific metrics like:
- Requests per second (RPS)
- Queue length (messages waiting in RabbitMQ/Kafka)
- Active WebSocket connections
- Response latency

### Types of Metrics HPA Supports

```yaml
metrics:
# Type 1: Resource — Built-in CPU/Memory (from Metrics Server)
- type: Resource
  resource:
    name: cpu
    target:
      type: Utilization
      averageUtilization: 50

# Type 2: Pods — Custom metric from each Pod (from Prometheus Adapter)
- type: Pods
  pods:
    metric:
      name: http_requests_per_second     # Custom metric name
    target:
      type: AverageValue
      averageValue: "1000"               # Scale if avg RPS per pod > 1000

# Type 3: Object — Metric from a Kubernetes object (like an Ingress)
- type: Object
  object:
    describedObject:
      apiVersion: networking.k8s.io/v1
      kind: Ingress
      name: my-ingress
    metric:
      name: requests_per_second
    target:
      type: Value
      value: "5000"                      # Total RPS on the Ingress

# Type 4: External — Metric from outside Kubernetes (cloud metrics, etc.)
- type: External
  external:
    metric:
      name: sqs_queue_length             # AWS SQS queue depth
      selector:
        matchLabels:
          queue: "order-processing"
    target:
      type: Value
      value: "100"                       # Scale if queue has > 100 messages
```

### Setting Up Custom Metrics with Prometheus

To use custom metrics, you need a **metrics adapter** that bridges Prometheus (or another monitoring system) with the Kubernetes Metrics API.

```
Your App → exposes metrics → Prometheus scrapes them → Prometheus Adapter → 
HPA reads from Metrics API → Scales your pods
```

**Step 1:** Install Prometheus (if not already installed)

```bash
# Using Helm
helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm install prometheus prometheus-community/kube-prometheus-stack
```

**Step 2:** Install Prometheus Adapter

```bash
helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm install prometheus-adapter prometheus-community/prometheus-adapter \
  --set prometheus.url=http://prometheus-kube-prometheus-prometheus.default.svc \
  --set prometheus.port=9090
```

**Step 3:** Configure the adapter to expose your custom metric (in the Helm values):

```yaml
# prometheus-adapter-values.yaml
rules:
- seriesQuery: 'http_requests_total{namespace!="",pod!=""}'
  resources:
    overrides:
      namespace: {resource: "namespace"}
      pod: {resource: "pod"}
  name:
    matches: "^(.*)_total$"
    as: "${1}_per_second"
  metricsQuery: 'rate(<<.Series>>{<<.LabelMatchers>>}[2m])'
```

**Step 4:** Use the custom metric in HPA:

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: custom-metric-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-web-app
  minReplicas: 2
  maxReplicas: 20
  metrics:
  - type: Pods
    pods:
      metric:
        name: http_requests_per_second
      target:
        type: AverageValue
        averageValue: "500"     # Each pod should handle ~500 RPS
```

## 🎯 Target Types Explained

There are three target types you can use:

### 1. Utilization (Percentage)

```yaml
target:
  type: Utilization
  averageUtilization: 50    # 50% of the resource request
```

- **Only works with:** `type: Resource` (CPU/Memory)
- **What it means:** "Scale when the average utilization across all pods exceeds 50% of their **request**"
- **Example:** If request is 100m CPU and target is 50%, HPA targets 50m CPU per pod

### 2. AverageValue (Absolute Value per Pod)

```yaml
target:
  type: AverageValue
  averageValue: "500m"      # 500 millicores per pod
```

- **Works with:** Resource, Pods, External
- **What it means:** "Scale when the average metric value per pod exceeds 500m"
- **Example:** Keep each pod's CPU at ~500m regardless of what they requested

### 3. Value (Total Value)

```yaml
target:
  type: Value
  value: "10000"            # Total value across the whole system
```

- **Works with:** Object, External
- **What it means:** "Scale when the total metric value exceeds 10000"
- **Example:** If SQS queue has > 10000 messages, scale up

## 🚫 Common Mistakes & Troubleshooting

### Mistake 1: No Resource Requests

```yaml
# ❌ BAD — HPA can't calculate utilization without requests!
containers:
- name: web
  image: nginx
  # No resources section at all!

# ✅ GOOD — Always set requests
containers:
- name: web
  image: nginx
  resources:
    requests:
      cpu: "100m"
      memory: "128Mi"
```

**HPA shows `<unknown>/50%`?** This is almost always because you forgot to set resource requests.

### Mistake 2: Metrics Server Not Installed

```bash
# Check if metrics-server is running
kubectl get deployment metrics-server -n kube-system

# If not installed, HPA will show <unknown> for targets
# Install it:
kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml
```

### Mistake 3: Min = Max (No Room to Scale)

```yaml
# ❌ BAD — HPA can't scale if min equals max!
minReplicas: 5
maxReplicas: 5

# ✅ GOOD — Give HPA room to work
minReplicas: 2
maxReplicas: 10
```

### Mistake 4: Conflicting with `replicas` Field

```yaml
# ❌ BAD — Don't set replicas in Deployment when using HPA
# HPA controls the replica count, so they will fight each other!
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3            # This conflicts with HPA!
```

**Best practice:** When using HPA, either:
- Remove `replicas` from your Deployment YAML, or
- Accept that HPA will override it anyway

### Mistake 5: HPA Flapping (Rapid Scale Up/Down)

```
2:00 PM — CPU high → Scale to 8 pods
2:01 PM — CPU drops → Scale to 3 pods  
2:02 PM — CPU high → Scale to 8 pods
2:03 PM — CPU drops → Scale to 3 pods
...this is called "flapping" or "thrashing"
```

**Fix:** Use the `behavior` field to add stabilization windows:

```yaml
behavior:
  scaleDown:
    stabilizationWindowSeconds: 300    # Wait 5 minutes before scaling down
  scaleUp:
    stabilizationWindowSeconds: 60     # Wait 1 minute before scaling up
```

## 🏆 Best Practices for HPA

| Practice | Why |
|----------|-----|
| **Always set resource `requests`** | HPA can't calculate utilization without them |
| **Set reasonable `minReplicas`** | Don't go below 2 for production (high availability) |
| **Set `maxReplicas` as a cost guard** | Prevents runaway scaling that burns your cloud bill |
| **Use `behavior` for stabilization** | Prevents flapping and unnecessary scaling events |
| **Combine CPU + Memory metrics** | Some apps are CPU-bound, others are memory-bound |
| **Use custom metrics for real scaling** | CPU/memory don't always reflect actual load (e.g., queue depth is better for workers) |
| **Don't set `replicas` in Deployment** | Let HPA fully manage the replica count |
| **Monitor HPA events** | `kubectl describe hpa` shows scaling decisions and why |
| **Test under load before production** | Use tools like `hey`, `wrk`, `k6` to simulate traffic |
| **Use `autoscaling/v2` API** | v1 only supports CPU; v2 supports memory, custom, and external metrics |

## 🔄 HPA vs Manual Scaling — Quick Comparison

| Feature | Manual Scaling | HPA |
|---------|---------------|-----|
| Speed | Slow (human reaction) | Fast (15-sec check interval) |
| Accuracy | Best-guess | Data-driven |
| Cost | Wastes money (over-provision) or risks downtime (under-provision) | Optimized — just right |
| Night/Weekend | Someone has to be on-call | Fully automatic |
| Effort | Manual `kubectl scale` | Set once, forget |

## 🔑 Key Takeaways

1. **HPA = Automatic horizontal scaling** — adds/removes Pod replicas based on metrics.
2. **Metrics Server is required** — install it first, or HPA won't know your Pod's CPU/memory usage.
3. **Resource `requests` are mandatory** — HPA calculates utilization as a percentage of requests.
4. **Use `autoscaling/v2`** — it supports CPU, memory, custom metrics, and external metrics.
5. **The formula is simple:** `desiredReplicas = ceil(currentReplicas × (current / target))`.
6. **Use `behavior`** to fine-tune how fast HPA scales up and down.
7. **Custom metrics** (via Prometheus Adapter) let you scale based on real business metrics like RPS or queue depth.
8. **HPA doesn't add nodes** — it only adds Pods. If your cluster is out of node capacity, you need the **Cluster Autoscaler** (covered in 10.30).
