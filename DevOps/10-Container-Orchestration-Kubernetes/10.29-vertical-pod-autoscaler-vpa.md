# 10.29 — Vertical Pod Autoscaler (VPA)

> **One Line Answer:** The Vertical Pod Autoscaler (VPA) **automatically adjusts the CPU and memory requests/limits** of your containers — making each Pod bigger or smaller — so you don't have to guess the right resource values anymore. While HPA adds more Pods, VPA makes existing Pods more powerful (or leaner).

---

## 🤔 Why Do We Need VPA?

### The Problem: Guessing Resource Requests

Every time you write a Deployment, you have to set CPU and memory requests. But how do you pick the right numbers?

```yaml
resources:
  requests:
    cpu: "???"          # What should I put here? 🤷
    memory: "???"       # No idea... let's guess? 😅
```

Most teams do one of two things — and both are bad:

```
Option A: Set requests TOO HIGH (over-provisioning)

   Your app needs:    100m CPU,  128Mi memory
   You set requests:  1000m CPU, 2Gi memory      ← 10x more than needed!
   
   ┌─────────────────────────────────────────────┐
   │  Pod                                         │
   │  ┌───────────┐                               │
   │  │ App uses  │                               │
   │  │ 100m CPU  │    WASTED SPACE!              │
   │  │ 128Mi mem │    (900m CPU, ~1.9Gi memory   │
   │  └───────────┘     reserved but unused)      │
   │                                              │
   │  Requested: 1000m CPU, 2Gi memory            │
   └─────────────────────────────────────────────┘
   
   Result: 💸 You're paying for resources you never use!
   
   10 pods × 900m wasted CPU = 9 whole CPU cores WASTED
   10 pods × 1.9Gi wasted memory = 19 GiB WASTED
   
   That's real money on your cloud bill! 💰💰💰


Option B: Set requests TOO LOW (under-provisioning)

   Your app needs:    500m CPU,  512Mi memory
   You set requests:  50m CPU,   64Mi memory      ← Way too low!
   
   ┌──────────┐
   │  Pod     │
   │  App is  │  
   │ CRUSHED! │  ← OOMKilled! CrashLoopBackOff! Slow responses!
   │  50m CPU │
   │  64Mi    │
   └──────────┘
   
   Result: 😱 Your app crashes, users get errors, pages load slow!
```

### The Solution: VPA Figures It Out For You

```
With VPA:

1. VPA watches your app's ACTUAL resource usage over time
   
   Day 1: App used 120m CPU, 200Mi memory
   Day 2: App used 180m CPU, 250Mi memory
   Day 3: App used 150m CPU, 220Mi memory
   
2. VPA calculates the RIGHT resource values
   
   Recommendation: cpu: 200m, memory: 280Mi
   (with some headroom for spikes)

3. VPA updates your Pod's requests to match reality
   
   Before VPA:  requests: cpu: 50m,  memory: 64Mi    ← Bad guess
   After VPA:   requests: cpu: 200m, memory: 280Mi   ← Perfect fit! ✅
```

## 🏗️ How Does VPA Work? — Architecture

VPA has **three main components** that work together:

```
┌──────────────────────────────────────────────────────────────────┐
│                          VPA System                               │
│                                                                   │
│  ┌─────────────────┐  ┌─────────────────┐  ┌──────────────────┐ │
│  │  VPA Recommender │  │  VPA Updater    │  │ VPA Admission    │ │
│  │                  │  │                  │  │ Controller       │ │
│  │  Watches actual  │  │  Evicts pods     │  │                  │ │
│  │  resource usage  │  │  that need       │  │ When a new pod   │ │
│  │  over time.      │  │  resource        │  │ is created, it   │ │
│  │                  │  │  changes.        │  │ injects the      │ │
│  │  Calculates      │  │                  │  │ recommended      │ │
│  │  recommendations │  │  (Pods get       │  │ resource values  │ │
│  │  for CPU and     │  │   recreated      │  │ into the pod     │ │
│  │  memory.         │  │   with new       │  │ spec.            │ │
│  │                  │  │   values)        │  │                  │ │
│  └────────┬─────────┘  └────────┬─────────┘  └────────┬─────────┘ │
│           │                     │                      │           │
│           ▼                     ▼                      ▼           │
│  ┌──────────────────────────────────────────────────────────────┐ │
│  │                    Metrics Server / Prometheus               │ │
│  │                    (Source of real usage data)                │ │
│  └──────────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────────┘
```

### Component Breakdown:

| Component | What It Does |
|-----------|-------------|
| **VPA Recommender** | Watches resource usage history. Calculates recommended CPU/memory requests. Stores recommendations in the VPA object status. |
| **VPA Updater** | Checks if running Pods match the recommendations. If not, it **evicts** (restarts) the Pods so they can be recreated with new values. |
| **VPA Admission Controller** | A mutating webhook. When a new Pod is created, it intercepts the request and **injects** the recommended resource values before the Pod starts. |

### The Flow:

```
1. Your app runs with some resource requests
2. VPA Recommender watches actual usage via Metrics Server
3. After collecting enough data, Recommender generates recommendations
4. VPA Updater sees that current requests ≠ recommendations
5. Updater EVICTS the pod (deletes it)
6. Deployment controller creates a NEW pod
7. VPA Admission Controller intercepts the new pod creation
8. Admission Controller INJECTS the recommended values
9. New pod starts with perfect resource requests! ✅
```

> ⚠️ **Important:** VPA works by **restarting Pods** to apply new values. It cannot change resources on a running Pod in-place (though Kubernetes is working on an "in-place resize" feature).

## 📦 Installing VPA

VPA is **NOT built into Kubernetes** by default. You need to install it separately.

### Option 1: Install from the Official Repository

```bash
# Clone the VPA repository
git clone https://github.com/kubernetes/autoscaler.git

# Navigate to the VPA directory
cd autoscaler/vertical-pod-autoscaler

# Install VPA components
./hack/vpa-up.sh
```

This installs all three components: Recommender, Updater, and Admission Controller.

### Option 2: Install via Helm

```bash
# Add the FairWinds Helm repo (popular VPA chart)
helm repo add fairwinds-stable https://charts.fairwinds.com/stable
helm repo update

# Install VPA
helm install vpa fairwinds-stable/vpa \
  --namespace vpa \
  --create-namespace
```

### Verify Installation

```bash
# Check VPA pods are running
kubectl get pods -n kube-system | grep vpa
# (or check the namespace where you installed it)

# You should see:
# vpa-admission-controller-xxxxx   1/1   Running
# vpa-recommender-xxxxx            1/1   Running
# vpa-updater-xxxxx                1/1   Running

# Check if VPA CRD is installed
kubectl get crd | grep verticalpodautoscaler
# Should show: verticalpodautoscalers.autoscaling.k8s.io
```

## 🛠️ Creating Your First VPA — Step by Step

### Step 1: Create a Deployment

```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 2
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
      - name: my-app
        image: nginx:latest
        ports:
        - containerPort: 80
        resources:
          requests:
            cpu: "50m"           # Our initial guess (probably wrong!)
            memory: "64Mi"
          limits:
            cpu: "500m"
            memory: "512Mi"
```

```bash
kubectl apply -f deployment.yaml
```

### Step 2: Create a VPA Object

```yaml
# vpa.yaml
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: my-app-vpa
spec:
  targetRef:                          # What to autoscale?
    apiVersion: apps/v1
    kind: Deployment
    name: my-app                      # Must match your Deployment name
  updatePolicy:
    updateMode: "Auto"                # VPA will automatically apply recommendations
```

```bash
kubectl apply -f vpa.yaml
```

### Step 3: Check VPA Recommendations

```bash
# View VPA status
kubectl get vpa

# Get detailed recommendations
kubectl describe vpa my-app-vpa
```

**Example output:**

```
Name:         my-app-vpa
Namespace:    default
Status:
  Recommendation:
    Container Recommendations:
      Container Name:  my-app
      Lower Bound:                    # Minimum recommended
        Cpu:     25m
        Memory:  131072k
      Target:                         # 💡 IDEAL recommendation
        Cpu:     100m
        Memory:  262144k
      Upper Bound:                    # Maximum recommended
        Cpu:     500m
        Memory:  524288k
      Uncapped Target:                # What VPA would recommend without limits
        Cpu:     100m
        Memory:  262144k
```

**Reading the recommendations:**

| Field | Meaning |
|-------|---------|
| **Lower Bound** | The minimum resources the container needs. Going below this risks instability. |
| **Target** | The sweet spot — VPA's best recommendation. This is what it will set. |
| **Upper Bound** | The maximum VPA thinks the container could ever need. |
| **Uncapped Target** | What VPA would recommend if there were no min/max bounds set. |

## 📝 VPA Update Modes — The Most Important Setting

The `updateMode` determines **what VPA actually does** with its recommendations:

### Mode 1: `"Off"` — Recommend Only (Safest)

```yaml
updatePolicy:
  updateMode: "Off"
```

- VPA **only generates recommendations** but does NOT apply them.
- Pods are **never restarted** or modified.
- You read the recommendations and manually update your Deployment.
- **Best for:** Getting started, understanding your app's needs, production apps where you don't want automatic restarts.

```bash
# Read VPA recommendations
kubectl get vpa my-app-vpa -o jsonpath='{.status.recommendation}' | jq .
```

### Mode 2: `"Initial"` — Set on Creation Only

```yaml
updatePolicy:
  updateMode: "Initial"
```

- VPA sets resource values **only when a Pod is first created**.
- It does NOT restart existing Pods to apply new recommendations.
- Useful when combined with HPA (HPA scales horizontally, VPA sets initial sizes).

### Mode 3: `"Auto"` — Fully Automatic

```yaml
updatePolicy:
  updateMode: "Auto"
```

- VPA **automatically evicts (restarts) Pods** and applies new recommendations.
- This means your Pods WILL be restarted when VPA decides resources need changing.
- **Best for:** Non-critical apps, development/staging environments.

> ⚠️ **Warning:** In `Auto` mode, VPA will restart your Pods! Make sure your app handles restarts gracefully (use multiple replicas, PodDisruptionBudgets, etc.).

### Comparison Table

| Mode | Generates Recommendations | Applies on Pod Creation | Evicts Running Pods | Use Case |
|------|--------------------------|------------------------|--------------------|----|
| `Off` | ✅ Yes | ❌ No | ❌ No | Monitoring/analysis only |
| `Initial` | ✅ Yes | ✅ Yes | ❌ No | Safe for production |
| `Auto` | ✅ Yes | ✅ Yes | ✅ Yes | Dev/staging, non-critical |

## 🎛️ Advanced VPA Configuration

### Setting Resource Boundaries (Min/Max)

You might not want VPA to recommend values that are too low or too high:

```yaml
# vpa-with-limits.yaml
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: my-app-vpa
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-app
  updatePolicy:
    updateMode: "Auto"
  resourcePolicy:
    containerPolicies:
    - containerName: my-app           # Apply to specific container
      minAllowed:                      # Don't go below these values
        cpu: "50m"
        memory: "64Mi"
      maxAllowed:                      # Don't go above these values
        cpu: "2"                       # 2 full CPU cores max
        memory: "4Gi"                  # 4 GiB max
      controlledResources: ["cpu", "memory"]  # Which resources to manage
```

### Excluding Specific Containers

In multi-container Pods, you might want VPA to manage only certain containers:

```yaml
resourcePolicy:
  containerPolicies:
  - containerName: main-app
    mode: "Auto"                # VPA manages this container
  - containerName: sidecar-proxy
    mode: "Off"                 # VPA ignores this container
```

### Controlling Which Resources VPA Manages

```yaml
resourcePolicy:
  containerPolicies:
  - containerName: my-app
    controlledResources: ["cpu"]     # Only manage CPU, leave memory alone
```

Or only manage memory:

```yaml
    controlledResources: ["memory"]  # Only manage memory, leave CPU alone
```

### Controlling How VPA Sets Limits

By default, VPA manages requests. But what about limits? You can control this:

```yaml
resourcePolicy:
  containerPolicies:
  - containerName: my-app
    controlledValues: RequestsAndLimits   # VPA sets both requests AND limits
    # Other option: RequestsOnly          # VPA only sets requests (default)
```

## 🔄 VPA + HPA — Can You Use Both Together?

This is a very common question, and the answer is: **yes, but with care**.

### The Problem

```
If both VPA and HPA use CPU metrics:

  HPA says: "CPU is high! Add more pods!"
  VPA says: "CPU is high! Give each pod more CPU!"

  They fight each other! 😵
  
  HPA adds pods → CPU drops → VPA reduces CPU requests → 
  CPU goes back up → HPA adds more pods → infinite loop!
```

### The Solution: Use Different Metrics

```
✅ The right way to combine VPA + HPA:

  VPA: Manages CPU and memory requests (right-sizes each pod)
  HPA: Scales based on CUSTOM METRICS (like requests per second, queue depth)

  They don't conflict because they look at different metrics!
```

### Example: VPA + HPA Working Together

```yaml
# VPA — manages resource sizing
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: my-app-vpa
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-app
  updatePolicy:
    updateMode: "Auto"
  resourcePolicy:
    containerPolicies:
    - containerName: my-app
      controlledResources: ["cpu", "memory"]
---
# HPA — scales pod count based on custom metric (NOT cpu/memory!)
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: my-app-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-app
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Pods
    pods:
      metric:
        name: http_requests_per_second    # Custom metric, NOT cpu!
      target:
        type: AverageValue
        averageValue: "1000"
```

### Another Safe Combo: VPA in "Off" Mode + HPA on CPU

```
VPA (mode: "Off")     → Just gives recommendations, doesn't change anything
HPA (cpu-percent: 50) → Scales pods normally based on CPU

You manually read VPA recommendations and update your Deployment periodically.
No conflict because VPA isn't touching anything automatically!
```

## 🧪 Hands-On Lab: VPA in Action

### Step 1: Install VPA (if not already installed)

```bash
# Quick install
git clone https://github.com/kubernetes/autoscaler.git
cd autoscaler/vertical-pod-autoscaler
./hack/vpa-up.sh
```

### Step 2: Deploy an App with Bad Resource Guesses

```yaml
# hamster.yaml — Official VPA test app
apiVersion: apps/v1
kind: Deployment
metadata:
  name: hamster
spec:
  replicas: 2
  selector:
    matchLabels:
      app: hamster
  template:
    metadata:
      labels:
        app: hamster
    spec:
      containers:
      - name: hamster
        image: registry.k8s.io/ubuntu-slim:0.1
        resources:
          requests:
            cpu: "100m"         # Intentionally wrong!
            memory: "50Mi"      # Intentionally wrong!
        command: ["/bin/sh"]
        args: ["-c", "while true; do timeout 0.5s yes >/dev/null; sleep 0.5s; done"]
```

This app alternates between doing CPU-intensive work and sleeping — simulating a real workload.

```bash
kubectl apply -f hamster.yaml
```

### Step 3: Create a VPA in "Off" Mode First (Safe!)

```yaml
# vpa-hamster.yaml
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: hamster-vpa
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: hamster
  updatePolicy:
    updateMode: "Off"      # Just recommend, don't change anything
```

```bash
kubectl apply -f vpa-hamster.yaml
```

### Step 4: Wait and Check Recommendations

```bash
# Wait a few minutes for VPA to collect data, then check:
kubectl get vpa hamster-vpa --output yaml

# Look at the recommendation section:
# recommendation:
#   containerRecommendations:
#   - containerName: hamster
#     lowerBound:
#       cpu: 400m
#       memory: 262144k
#     target:
#       cpu: 587m         ← VPA recommends 587m, you set 100m! Way too low!
#       memory: 262144k   ← VPA recommends ~256Mi, you set 50Mi! Way too low!
#     upperBound:
#       cpu: 1200m
#       memory: 524288k
```

### Step 5: Switch to Auto Mode and Watch It Work

```bash
# Edit the VPA to switch to Auto mode
kubectl patch vpa hamster-vpa --type='json' \
  -p='[{"op": "replace", "path": "/spec/updatePolicy/updateMode", "value": "Auto"}]'
```

Now watch VPA restart the pods with new values:

```bash
# Watch pods being evicted and recreated
kubectl get pods -l app=hamster --watch

# After pods are recreated, check their resource requests:
kubectl get pod -l app=hamster -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.spec.containers[0].resources.requests}{"\n"}{end}'

# Output should show the VPA-recommended values, not your original 100m/50Mi!
# Something like: cpu: 587m, memory: 262144k
```

### Step 6: Clean Up

```bash
kubectl delete deployment hamster
kubectl delete vpa hamster-vpa
```

## 🔍 Goldilocks — VPA Dashboard Tool

Manually checking VPA recommendations for every Deployment is tedious. **Goldilocks** by FairWinds is a tool that creates VPAs for all Deployments in a namespace and provides a nice web dashboard to see all recommendations at once.

### Install Goldilocks

```bash
# Add the FairWinds Helm repo
helm repo add fairwinds-stable https://charts.fairwinds.com/stable
helm repo update

# Install Goldilocks
helm install goldilocks fairwinds-stable/goldilocks \
  --namespace goldilocks \
  --create-namespace
```

### Enable Goldilocks for a Namespace

```bash
# Label the namespace you want Goldilocks to analyze
kubectl label namespace default goldilocks.fairwinds.com/enabled=true
```

Goldilocks will automatically create VPA objects (in `Off` mode) for every Deployment in that namespace and start collecting recommendations.

### Access the Dashboard

```bash
# Port-forward to the Goldilocks dashboard
kubectl port-forward -n goldilocks svc/goldilocks-dashboard 8080:80
```

Open `http://localhost:8080` in your browser. You'll see a beautiful dashboard showing recommended resource values for every container in every Deployment!

## 🚫 Common Mistakes & Troubleshooting

### Mistake 1: VPA Not Generating Recommendations

```bash
# Check if VPA recommender is running
kubectl get pods -n kube-system | grep vpa-recommender

# Check VPA events
kubectl describe vpa my-app-vpa

# Common cause: Metrics Server not installed!
kubectl top pods    # If this fails, install Metrics Server first
```

### Mistake 2: Pods Keep Getting Restarted

In `Auto` mode, VPA evicts pods to apply changes. If recommendations keep changing, pods keep restarting.

**Fix:** Use `"Initial"` mode or set `minAllowed`/`maxAllowed` boundaries:

```yaml
resourcePolicy:
  containerPolicies:
  - containerName: my-app
    minAllowed:
      cpu: "100m"
      memory: "128Mi"
    maxAllowed:
      cpu: "1"
      memory: "1Gi"
```

### Mistake 3: Using VPA with HPA on the Same CPU Metric

```
❌ BAD — Both using CPU:
  VPA: manages CPU requests
  HPA: scales based on CPU utilization
  
  They will conflict and create chaos!

✅ GOOD — Different metrics:
  VPA: manages CPU/memory requests
  HPA: scales based on custom metrics (RPS, queue depth)
```

### Mistake 4: Not Having Enough Replicas

In `Auto` mode, VPA evicts pods. If you only have 1 replica, your app goes DOWN during the update!

```yaml
# ❌ BAD — Only 1 replica + VPA Auto = downtime during eviction
spec:
  replicas: 1

# ✅ GOOD — Multiple replicas so evictions don't cause downtime
spec:
  replicas: 3
```

Also use **PodDisruptionBudgets** to control how many pods can be evicted at once:

```yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: my-app-pdb
spec:
  minAvailable: 1              # At least 1 pod must always be running
  selector:
    matchLabels:
      app: my-app
```

### Mistake 5: VPA Setting Resources Too High

VPA might recommend very high values based on spike usage. Use boundaries:

```yaml
resourcePolicy:
  containerPolicies:
  - containerName: my-app
    maxAllowed:
      cpu: "2"                 # Never recommend more than 2 cores
      memory: "2Gi"            # Never recommend more than 2 GiB
```

## 📊 VPA vs HPA — When to Use What?

| Feature | HPA | VPA |
|---------|-----|-----|
| **What it scales** | Number of Pods (horizontal) | Size of Pods (vertical) |
| **How it scales** | Adds/removes Pod replicas | Changes CPU/memory requests |
| **Pod restart needed?** | No | Yes (in Auto mode) |
| **Best for** | Stateless, horizontally scalable apps | Right-sizing resources, stateful apps |
| **Works with** | CPU, memory, custom metrics | CPU, memory |
| **Built into K8s?** | Yes | No (separate install) |
| **Risk** | Too many pods → cost | Pod restarts → brief disruption |

### When to Use Which?

```
Use HPA when:
  ✅ Your app can run as multiple replicas (stateless)
  ✅ You want to handle traffic spikes by adding pods
  ✅ Examples: web servers, API servers, microservices

Use VPA when:
  ✅ You don't know the right resource requests
  ✅ Your app can't be scaled horizontally (single-instance databases)
  ✅ You want to optimize costs by right-sizing pods
  ✅ Examples: databases, ML training jobs, batch processors

Use BOTH when:
  ✅ You want right-sized pods (VPA) AND horizontal scaling (HPA)
  ✅ But use different metrics! VPA for resources, HPA for custom metrics
```

## 🏆 Best Practices for VPA

| Practice | Why |
|----------|-----|
| **Start with `"Off"` mode** | Get recommendations without risking pod restarts |
| **Use `minAllowed`/`maxAllowed`** | Prevent VPA from setting crazy high or low values |
| **Always have multiple replicas** | So VPA evictions don't cause downtime |
| **Use PodDisruptionBudgets** | Control how many pods VPA can evict at once |
| **Don't combine VPA + HPA on same metric** | They'll fight each other |
| **Use Goldilocks for visibility** | See all recommendations in a dashboard |
| **Review recommendations regularly** | Even in Off mode, check what VPA suggests |
| **Prefer `"Initial"` mode for production** | Sets values at pod creation, no surprise restarts |

---

## 🔑 Key Takeaways

1. **VPA = Automatic vertical scaling** — adjusts CPU/memory requests per container.
2. **Three components:** Recommender (calculates), Updater (evicts pods), Admission Controller (injects values).
3. **Three modes:** `Off` (recommend only), `Initial` (set at creation), `Auto` (full auto with restarts).
4. **VPA is NOT built into Kubernetes** — you must install it separately.
5. **VPA restarts Pods** to apply changes (no in-place resize yet in most clusters).
6. **Don't use VPA + HPA on the same CPU metric** — they'll conflict.
7. **Use Goldilocks** for a dashboard view of VPA recommendations across all workloads.
8. **VPA doesn't add Pods** — it makes Pods bigger/smaller. For more Pods, use HPA. For more nodes, use Cluster Autoscaler (10.30).
