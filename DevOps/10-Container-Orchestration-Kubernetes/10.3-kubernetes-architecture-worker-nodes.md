# 10.3 — Kubernetes Architecture — Worker Nodes (kubelet, kube-proxy, Container Runtime)

> **One Line Answer:** Worker Nodes are the **muscles of Kubernetes** — they're the machines where your actual applications run. Each worker node has three key components: the **kubelet** (an agent that ensures pods are running), **kube-proxy** (handles networking and load balancing), and a **Container Runtime** (the engine that actually runs containers, like containerd or CRI-O).

## 1️⃣ kubelet — The Node Agent

### What Is It?

The **kubelet** is an agent that runs on **every single worker node** in the cluster. It's the component that actually makes sure your pods are running and healthy on that specific node.

Think of it this way: the control plane makes decisions, but the kubelet **executes** those decisions on each node.

### Real-World Analogy 👷

Imagine a **construction site foreman**:

- The **architect** (control plane) draws the blueprints: "Build 3 rooms on this floor"
- The **foreman** (kubelet) on that floor receives the blueprints and:
  - Gets the materials (pulls container images)
  - Builds the rooms (starts containers)
  - Checks every hour that the rooms are solid (health checks)
  - Reports back: "All 3 rooms are built and stable" (status updates)
  - If a room collapses, rebuilds it immediately (restarts crashed containers)

Each floor has its own foreman. Each node has its own kubelet.

### What Does the kubelet Do?

| Function | Description |
|----------|-------------|
| **Pod Lifecycle Management** | Creates, starts, stops, and deletes pods as told by the API Server |
| **Image Pulling** | Tells the container runtime to pull the required container images |
| **Container Health Monitoring** | Runs liveness, readiness, and startup probes |
| **Status Reporting** | Reports pod and node status back to the API Server |
| **Resource Monitoring** | Monitors CPU, memory, disk usage on the node |
| **Volume Mounting** | Mounts volumes (configmaps, secrets, persistent volumes) into pods |
| **Node Registration** | Registers the node with the API Server when it starts up |

### How Does the kubelet Work?

The kubelet follows this workflow:

```
┌──────────────────────────────────────────────────────┐
│                   kubelet Workflow                     │
│                                                      │
│  Step 1: WATCH the API Server                        │
│  ────────────────────────                            │
│  kubelet constantly watches the API Server for        │
│  pods assigned to THIS node (spec.nodeName = me)     │
│                                                      │
│  Step 2: COMPARE desired vs actual                   │
│  ────────────────────────                            │
│  "The API Server says I should have pods A, B, C.    │
│   Currently I'm running pods A and B.                │
│   Pod C is missing!"                                 │
│                                                      │
│  Step 3: RECONCILE                                   │
│  ────────────────────────                            │
│  "I need to create Pod C."                           │
│  → Tell container runtime to pull the image          │
│  → Tell container runtime to create the container    │
│  → Tell container runtime to start the container     │
│  → Mount any volumes                                 │
│  → Set up networking                                 │
│                                                      │
│  Step 4: REPORT status                               │
│  ────────────────────────                            │
│  "Pod C is now Running."                             │
│  → Send status update to API Server                  │
│                                                      │
│  Step 5: MONITOR health                              │
│  ────────────────────────                            │
│  Run health checks (probes) on all pods              │
│  If a pod fails health check → restart it            │
│                                                      │
│  (Repeat continuously...)                            │
└──────────────────────────────────────────────────────┘
```

### The PodSpec

The kubelet works with **PodSpecs** — YAML/JSON descriptions of what a pod should look like. It receives these from the API Server.

```yaml
# This is what the kubelet receives from the API Server
apiVersion: v1
kind: Pod
metadata:
  name: my-web-app
  namespace: default
spec:
  containers:
  - name: web
    image: nginx:1.25
    ports:
    - containerPort: 80
    resources:
      requests:
        cpu: "250m"
        memory: "128Mi"
      limits:
        cpu: "500m"
        memory: "256Mi"
    livenessProbe:
      httpGet:
        path: /healthz
        port: 80
      initialDelaySeconds: 10
      periodSeconds: 5
    readinessProbe:
      httpGet:
        path: /ready
        port: 80
      initialDelaySeconds: 5
      periodSeconds: 3
  volumes:
  - name: config
    configMap:
      name: app-config
```

The kubelet reads this PodSpec and:
1. Pulls the `nginx:1.25` image
2. Creates a container with the specified resource limits
3. Exposes port 80
4. Mounts the `app-config` ConfigMap as a volume
5. Starts running liveness and readiness probes

### Health Probes — The kubelet's Health Checks

The kubelet runs three types of health checks (probes) on containers:

#### 1. Liveness Probe — "Is the container alive?"

If this fails, the kubelet **restarts** the container.

```yaml
livenessProbe:
  httpGet:
    path: /healthz        # Hit this endpoint
    port: 8080
  initialDelaySeconds: 15  # Wait 15s before first check
  periodSeconds: 10        # Check every 10 seconds
  failureThreshold: 3      # Restart after 3 consecutive failures
```

**Use case:** Your app is running but **stuck** (deadlocked, infinite loop). The process is alive but not functioning. The liveness probe detects this and restarts it.

```
Timeline:
0s   → Container starts
15s  → First liveness check: GET /healthz → 200 OK ✅
25s  → Second check: GET /healthz → 200 OK ✅
35s  → Third check: GET /healthz → 200 OK ✅
...
120s → App gets stuck in deadlock
125s → Check: GET /healthz → timeout ❌ (failure 1)
135s → Check: GET /healthz → timeout ❌ (failure 2)
145s → Check: GET /healthz → timeout ❌ (failure 3)
145s → kubelet: "3 failures! Restarting container..."
146s → Container restarted ✅
```

#### 2. Readiness Probe — "Is the container ready to receive traffic?"

If this fails, the kubelet **removes the pod from the Service endpoints** (no traffic is sent to it), but does NOT restart the container.

```yaml
readinessProbe:
  httpGet:
    path: /ready
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 3
  failureThreshold: 3
```

**Use case:** Your app needs time to warm up (load caches, connect to database). It's alive but not ready to serve users. The readiness probe ensures no traffic is sent until it's truly ready.

```
Timeline:
0s  → Container starts
5s  → First readiness check: GET /ready → 503 ❌ (still warming up)
8s  → Second check: GET /ready → 503 ❌ (loading cache...)
11s → Third check: GET /ready → 200 OK ✅
    → kubelet: "Pod is ready! Adding to Service endpoints."
    → Traffic starts flowing to this pod
```

#### 3. Startup Probe — "Has the container finished starting up?"

Used for **slow-starting containers**. While the startup probe is running, liveness and readiness probes are disabled.

```yaml
startupProbe:
  httpGet:
    path: /started
    port: 8080
  failureThreshold: 30     # Try up to 30 times
  periodSeconds: 10        # Every 10 seconds = 300s (5 min) to start
```

**Use case:** Legacy apps or Java apps that take 2-3 minutes to start. Without a startup probe, the liveness probe might kill the container before it even finishes starting!

### Probe Types (How to Check)

The kubelet supports three ways to check health:

| Probe Type | How It Works | Example |
|-----------|-------------|---------|
| **HTTP GET** | Sends an HTTP GET request. 200-399 = success | `httpGet: { path: /health, port: 8080 }` |
| **TCP Socket** | Tries to open a TCP connection. Success = port is open | `tcpSocket: { port: 3306 }` (great for databases) |
| **Exec** | Runs a command inside the container. Exit code 0 = success | `exec: { command: ["cat", "/tmp/healthy"] }` |

```yaml
# HTTP GET probe (most common)
livenessProbe:
  httpGet:
    path: /healthz
    port: 8080

# TCP Socket probe (good for databases, Redis, etc.)
livenessProbe:
  tcpSocket:
    port: 3306

# Exec probe (run a custom command)
livenessProbe:
  exec:
    command:
    - cat
    - /tmp/healthy
```

### Node Heartbeats

The kubelet sends **heartbeats** to the API Server to let the control plane know the node is still alive:

```
kubelet → API Server: "I'm alive! Here's my status"
                       (every 10 seconds by default)

If the API Server doesn't hear from a kubelet for ~40 seconds:
→ Node Controller marks the node as "NotReady"
→ After 5 minutes of no heartbeat:
→ Node Controller evicts all pods from that node
→ ReplicaSet controllers recreate those pods on other nodes
```

This is configured via:
- `--node-status-update-frequency` (default: 10s) — how often kubelet sends heartbeats
- `--node-monitor-period` (default: 5s) — how often the node controller checks
- `--node-monitor-grace-period` (default: 40s) — how long to wait before marking NotReady
- `--pod-eviction-timeout` (default: 5m) — how long before evicting pods from NotReady node

### Static Pods — kubelet's Secret Feature

The kubelet can run pods **without the API Server**. These are called **static pods**.

```
Normal pods:    API Server → kubelet → Container Runtime
Static pods:    kubelet reads from local file → Container Runtime
```

The kubelet watches a directory (default: `/etc/kubernetes/manifests/`) for pod YAML files and runs them automatically.

**Where is this used?** The control plane components themselves! On a `kubeadm` cluster:

```
/etc/kubernetes/manifests/
├── etcd.yaml                    ← static pod
├── kube-apiserver.yaml          ← static pod
├── kube-controller-manager.yaml ← static pod
└── kube-scheduler.yaml          ← static pod
```

**Think about it:** The API Server runs as a pod, but pods need the API Server to run... 🤔 This is a chicken-and-egg problem! Static pods solve it — the kubelet can run these pods directly from local files, no API Server needed.

### kubelet Key Details

| Detail | Value |
|--------|-------|
| **Binary name** | `kubelet` |
| **Default port** | `10250` (HTTPS API), `10255` (read-only, usually disabled) |
| **Runs on** | Every node (both master and worker) |
| **Talks to** | API Server, Container Runtime (via CRI) |
| **Config** | `/var/lib/kubelet/config.yaml` |
| **Static pod path** | `/etc/kubernetes/manifests/` |
| **NOT a pod** | kubelet runs as a **system service** (systemd), not a pod |

That last point is important: **kubelet is NOT a Kubernetes pod**. It's a regular Linux process managed by systemd:

```bash
# Check kubelet service status
sudo systemctl status kubelet

# Output:
● kubelet.service - kubelet: The Kubernetes Node Agent
   Loaded: loaded (/etc/systemd/system/kubelet.service; enabled)
   Active: active (running) since Mon 2024-01-15 10:30:00 UTC
   ...

# View kubelet logs
sudo journalctl -u kubelet -f

# Restart kubelet
sudo systemctl restart kubelet
```

## 2️⃣ kube-proxy — The Network Plumber

### What Is It?

**kube-proxy** is a network component that runs on **every node** in the cluster. It's responsible for implementing **Kubernetes Services** — the mechanism that allows pods to communicate with each other and with the outside world.

### Real-World Analogy 📞

Think of kube-proxy as the **switchboard operator** in a large office building:

- Employee A calls the "Sales Department" (Service name)
- The switchboard operator (kube-proxy) knows that the Sales Department has 3 people: John, Jane, and Bob (3 pod replicas)
- The operator routes the call to one of them (load balancing)
- If Jane is on vacation (pod is down), the operator only routes to John and Bob
- The caller never needs to know individual phone numbers — just the department name

### What Does kube-proxy Do?

| Function | Description |
|----------|-------------|
| **Service Implementation** | Makes Kubernetes Services work by routing traffic to the right pods |
| **Load Balancing** | Distributes traffic across multiple pod replicas |
| **Network Rules** | Creates and manages iptables/IPVS rules on the node |
| **Service Discovery** | Ensures the Service ClusterIP routes to the correct pod IPs |

### Understanding the Problem kube-proxy Solves

Pods in Kubernetes are **ephemeral** — they come and go. Each time a pod is created, it gets a **new IP address**:

```
Time 0:
  Pod-1: 10.244.1.5
  Pod-2: 10.244.2.8
  Pod-3: 10.244.1.12

Time 1 (Pod-2 crashed and was recreated):
  Pod-1: 10.244.1.5
  Pod-2: 10.244.3.22    ← NEW IP!
  Pod-3: 10.244.1.12
```

If your frontend app hardcodes `10.244.2.8` to reach the backend, it's now pointing to a dead IP. **This breaks everything.**

The solution? **Kubernetes Services** — and kube-proxy is what makes them work.

### How Services Work (with kube-proxy)

```yaml
# A Service definition
apiVersion: v1
kind: Service
metadata:
  name: backend-service
spec:
  selector:
    app: backend       # Find pods with this label
  ports:
  - port: 80           # Service port
    targetPort: 8080   # Pod port
  type: ClusterIP      # Internal only
```

When you create this Service:

1. Kubernetes assigns it a **stable virtual IP** (ClusterIP), e.g., `10.96.45.123`
2. kube-proxy on every node sets up **network rules** so that any traffic to `10.96.45.123:80` gets forwarded to one of the backend pods on port `8080`

```
Frontend Pod sends request to:  10.96.45.123:80 (Service IP)
                                      │
                               kube-proxy intercepts
                                      │
                        ┌─────────────┼─────────────┐
                        │             │             │
                        ▼             ▼             ▼
                   Pod-1:8080    Pod-2:8080    Pod-3:8080
                  (10.244.1.5) (10.244.3.22) (10.244.1.12)
```

The frontend **never needs to know** the individual pod IPs. It just talks to the Service IP, and kube-proxy handles the routing.

### kube-proxy Modes

kube-proxy can operate in three different modes. This is important for interviews and understanding performance:

#### Mode 1: iptables (Default)

kube-proxy creates **iptables rules** on each node. iptables is the Linux kernel's built-in firewall/packet filter.

```
kube-proxy in iptables mode:

1. Watches API Server for Service and Endpoint changes
2. Creates iptables rules for each Service

For service "backend-service" (10.96.45.123):

iptables rules created:
┌────────────────────────────────────────────────────────────┐
│ IF destination = 10.96.45.123:80 THEN                     │
│   ├── 33% chance → forward to 10.244.1.5:8080  (Pod-1)   │
│   ├── 33% chance → forward to 10.244.3.22:8080 (Pod-2)   │
│   └── 34% chance → forward to 10.244.1.12:8080 (Pod-3)   │
└────────────────────────────────────────────────────────────┘
```

**How iptables mode works:**

```
Packet arrives → iptables in kernel → matches rule → forwards to pod
```

The packet never reaches kube-proxy itself — it's handled entirely in the **kernel** (which is fast!). kube-proxy just sets up the rules.

```bash
# You can see the iptables rules kube-proxy creates:
sudo iptables -t nat -L KUBE-SERVICES -n

# Output (simplified):
Chain KUBE-SERVICES
target          prot  source    destination
KUBE-SVC-xxx    tcp   0.0.0.0  10.96.45.123   /* backend-service */
KUBE-SVC-yyy    tcp   0.0.0.0  10.96.0.1      /* kubernetes */
...
```

**Pros & Cons of iptables mode:**

| Pros | Cons |
|------|------|
| ✅ Simple and well-understood | ❌ Not designed for load balancing |
| ✅ Handled in kernel (fast) | ❌ Rules are sequential — O(n) lookup |
| ✅ Default mode | ❌ With 10,000+ services, rule updates are SLOW |
| ✅ Reliable | ❌ Random load balancing only (no least-connections) |

#### Mode 2: IPVS (IP Virtual Server)

IPVS is a **purpose-built Linux kernel load balancer**. It's designed for load balancing, unlike iptables which is primarily a firewall.

```
kube-proxy in IPVS mode:

1. Watches API Server for Service and Endpoint changes
2. Creates IPVS virtual servers for each Service

For service "backend-service" (10.96.45.123):

IPVS configuration:
┌──────────────────────────────────────────────────────┐
│ Virtual Server: 10.96.45.123:80                      │
│   ├── Real Server: 10.244.1.5:8080  (weight: 1)     │
│   ├── Real Server: 10.244.3.22:8080 (weight: 1)     │
│   └── Real Server: 10.244.1.12:8080 (weight: 1)     │
│                                                      │
│ Algorithm: round-robin (or least-conn, etc.)         │
└──────────────────────────────────────────────────────┘
```

**Pros & Cons of IPVS mode:**

| Pros | Cons |
|------|------|
| ✅ Designed for load balancing | ❌ Requires IPVS kernel modules |
| ✅ Hash-based lookup — O(1) | ❌ More complex to debug |
| ✅ Multiple algorithms (RR, LC, SH, etc.) | ❌ Slightly more setup required |
| ✅ Scales to 10,000+ services easily | |
| ✅ Better performance at scale | |

**IPVS Load Balancing Algorithms:**

| Algorithm | How It Works |
|-----------|-------------|
| **Round Robin (rr)** | Rotates through pods in order |
| **Least Connection (lc)** | Sends to the pod with fewest active connections |
| **Source Hashing (sh)** | Same client IP always goes to same pod (sticky sessions) |
| **Destination Hashing (dh)** | Same destination always goes to same pod |
| **Shortest Expected Delay (sed)** | Sends to pod expected to respond fastest |

```bash
# Enable IPVS mode (in kube-proxy config):
# Edit the kube-proxy ConfigMap
kubectl edit configmap kube-proxy -n kube-system

# Change:
# mode: ""   →   mode: "ipvs"

# Check current IPVS rules
sudo ipvsadm -Ln

# Output:
IP Virtual Server version 1.2.1
TCP  10.96.45.123:80 rr
  -> 10.244.1.5:8080      Masq    1      0          0
  -> 10.244.3.22:8080     Masq    1      0          0
  -> 10.244.1.12:8080     Masq    1      0          0
```

#### Mode 3: kernelspace / userspace (Legacy)

In the old days, kube-proxy ran in **userspace mode** where packets were actually forwarded through the kube-proxy process itself. This was slow because packets had to go from kernel → user space → kernel.

```
Old way (userspace mode — DON'T USE):
Packet → kernel → kube-proxy process (user space) → kernel → pod

New way (iptables/IPVS):
Packet → kernel (iptables/IPVS rules) → pod
```

**Userspace mode is deprecated. Don't use it.** It's only mentioned here for completeness and in case it comes up in interviews.

### Comparison of kube-proxy Modes

| Feature | iptables | IPVS | userspace (legacy) |
|---------|----------|------|--------------------|
| **Performance** | Good | Excellent | Poor |
| **Scale** | Up to ~5,000 services | 10,000+ services | Very limited |
| **LB Algorithm** | Random only | RR, LC, SH, DH, SED | Round Robin |
| **Kernel handling** | Yes | Yes | No (user space) |
| **Default?** | ✅ Yes | No | No |
| **Recommended for production** | Small-medium clusters | Large clusters | ❌ Never |

### DNS-Based Service Discovery

While kube-proxy handles the IP-level routing, Kubernetes also has a **DNS server** (CoreDNS) that gives Services human-readable names:

```bash
# Instead of using the ClusterIP directly:
curl http://10.96.45.123:80

# You can use the DNS name:
curl http://backend-service:80

# Full DNS name (from a different namespace):
curl http://backend-service.default.svc.cluster.local:80
```

The DNS format is:
```
<service-name>.<namespace>.svc.cluster.local
```

CoreDNS resolves this name to the Service's ClusterIP, and then kube-proxy handles routing from there to the actual pods.

### kube-proxy Key Details

| Detail | Value |
|--------|-------|
| **Binary name** | `kube-proxy` |
| **Runs on** | Every node (as a DaemonSet pod) |
| **Default mode** | iptables |
| **Config** | ConfigMap `kube-proxy` in `kube-system` namespace |
| **Watches** | API Server for Service and Endpoint changes |
| **Default port** | `10256` (health check) |

```bash
# Check kube-proxy pods
kubectl get pods -n kube-system -l k8s-app=kube-proxy

# Output:
NAME               READY   STATUS    RESTARTS   AGE
kube-proxy-abc12   1/1     Running   0          30d
kube-proxy-def34   1/1     Running   0          30d
kube-proxy-ghi56   1/1     Running   0          30d

## 3️⃣ Container Runtime — The Engine

### What Is It?

The **Container Runtime** is the software that actually **pulls container images** and **runs containers** on the node. It's the lowest-level component — the actual engine that starts and stops containers.

### Real-World Analogy 🏗️

If the kubelet is the **foreman** giving orders, the container runtime is the **construction crew** that actually builds things:

- Foreman (kubelet): "Build a room using blueprint X" (PodSpec)
- Construction crew (container runtime): Actually lays bricks, installs wiring, etc. (pulls image, creates container, starts it)

### The Container Runtime Interface (CRI)

Kubernetes doesn't directly talk to container runtimes. Instead, it uses a standard interface called **CRI (Container Runtime Interface)**:

```
kubelet ──► CRI (standard interface) ──► Container Runtime
                                              │
                                    ┌─────────┼─────────┐
                                    │         │         │
                                containerd   CRI-O    Other
```

**Why CRI?** So Kubernetes doesn't depend on any specific runtime. Any runtime that implements the CRI can work with Kubernetes. Plug and play!

### The Big Docker Story 🐋

**Important history lesson** that comes up in interviews:

```
Timeline:
2014-2020: Kubernetes used Docker as its container runtime
2020:      Kubernetes announced Docker deprecation (dockershim removal)  
2022:      Kubernetes v1.24 officially removed dockershim
```

**What happened?**

Docker is a **full platform** — it includes the Docker CLI, Docker Daemon, Docker Compose, Docker Swarm, networking, volumes, build tools, etc. Kubernetes only needed the **container running** part.

Using Docker as a K8s runtime looked like this (the old way):

```
kubelet → dockershim → Docker Daemon → containerd → runc → container

That's a LOT of layers! 😬
```

The `dockershim` was a translation layer that Kubernetes maintained to convert CRI calls into Docker API calls. It was a maintenance burden.

The solution? **Cut out the middleman.** Kubernetes now talks directly to `containerd` (which was always the core container runtime inside Docker anyway):

```
Old way:  kubelet → dockershim → Docker → containerd → runc → container
New way:  kubelet → CRI → containerd → runc → container  ✅ (simpler!)
```

> **IMPORTANT:** This does NOT mean Docker images stopped working! Docker images are OCI-standard images. They work with ANY OCI-compliant runtime. You can still `docker build` your images and run them on Kubernetes with containerd. The only thing that changed is the **runtime** — not the **image format**.

### containerd — The Industry Standard Runtime

**containerd** (pronounced "container-dee") is the most widely used container runtime for Kubernetes.

```

### CRI-O — The Kubernetes-Native Runtime

**CRI-O** is a lightweight container runtime built **specifically** for Kubernetes. It implements the CRI and nothing else — no extras, no bloat.

```
containerd: General-purpose runtime (can be used outside K8s too)
CRI-O:      Purpose-built ONLY for Kubernetes
```

| Fact | Detail |
|------|--------|
| **Created by** | Red Hat, Intel, SUSE, others |
| **Status** | CNCF Incubating project |
| **Used by** | OpenShift (Red Hat's K8s distribution) |
| **Philosophy** | Minimal — only what K8s needs |
| **Written in** | Go |

### runc — The Low-Level Runtime

Both containerd and CRI-O use **runc** (or a compatible alternative) as the actual low-level runtime that creates containers:

```
High-level runtime (containerd/CRI-O):
  - Image management
  - Container lifecycle management
  - CRI implementation
  - Networking setup

Low-level runtime (runc):
  - Create Linux namespaces (pid, net, mnt, uts, ipc, user)
  - Set up cgroups (resource limits)
  - Set up the root filesystem
  - Execute the container process
```

```bash
# runc is the OCI reference implementation
# Created by Docker, donated to the Open Container Initiative (OCI)

# You rarely interact with runc directly, but it's there:
runc list           # List containers
runc state <id>     # Get container state
```

### Alternative Low-Level Runtimes

While runc is the default, there are alternatives for specific use cases:

| Runtime | Use Case |
|---------|----------|
| **runc** | Default, standard Linux containers |
| **gVisor (runsc)** | Security-focused — adds a user-space kernel sandbox |
| **Kata Containers** | Security-focused — each container runs in a lightweight VM |
| **crun** | Performance-focused — written in C (faster than runc which is in Go) |
| **youki** | Experimental — written in Rust |

```
Security spectrum:

Less secure ←──────────────────────────────────→ More secure
    runc           gVisor            Kata Containers
 (shared kernel)  (user-space       (hardware VM
                   kernel)           isolation)
    
Performance:
    Fast           Medium            Slower
```

### The Crash Scenario (Self-Healing)

```
Timeline:
─────────

T=0:    Pod is running normally
        kubelet: liveness check → ✅ OK
        kubelet: liveness check → ✅ OK

T=120s: Pod's app crashes (process exits)
        kubelet: "Container exited! Restart policy is 'Always'"
        kubelet → containerd: "Restart the container"
        Container restarts in ~1 second
        kubelet → API Server: "Pod restarted (restart count: 1)"

T=125s: kubelet: liveness check → ✅ OK (back to normal)

─── What if the container keeps crashing? ───

T=130s: Container crashes again
        kubelet restarts it (restart count: 2)

T=135s: Container crashes again
        kubelet: "This keeps crashing. Applying CrashLoopBackOff"
        kubelet waits 10 seconds before restarting

T=145s: Restart (count: 3)
T=146s: Crashes again
        kubelet waits 20 seconds (exponential backoff)

T=166s: Restart (count: 4)
T=167s: Crashes again
        kubelet waits 40 seconds

... backoff doubles each time up to 5 minutes max ...

This is the "CrashLoopBackOff" state you'll often see:
kubectl get pods
NAME       READY   STATUS             RESTARTS   AGE
web-app    0/1     CrashLoopBackOff   5          10m
```

## 🌐 Networking on Worker Nodes — CNI Plugins

We've talked about kube-proxy, but there's another important networking piece: **CNI (Container Network Interface) plugins**.

### What Is CNI?

CNI is a standard interface for configuring container networking. When a pod is created, the kubelet calls a **CNI plugin** to set up networking for that pod:

```
kubelet → CNI Plugin → "Set up networking for this pod"
                     → Assign IP address
                     → Set up routes
                     → Configure network namespace
```

### Popular CNI Plugins

| Plugin | Company | Key Feature |
|--------|---------|-------------|
| **Calico** | Tigera | Network policies, BGP routing, most popular |
| **Flannel** | CoreOS | Simple overlay network, easy to set up |
| **Cilium** | Isovalent | eBPF-based, advanced security & observability |
| **Weave Net** | Weaveworks | Easy setup, encrypted traffic |
| **AWS VPC CNI** | AWS | Uses native AWS VPC networking (for EKS) |
| **Azure CNI** | Microsoft | Uses native Azure networking (for AKS) |

```
Without CNI plugin: Pods can't communicate ❌
With CNI plugin:    Pods get IPs and can talk to each other ✅

Pod-to-Pod communication model:
┌──────────┐                    ┌──────────┐
│  Node 1  │                    │  Node 2  │
│          │                    │          │
│ ┌──────┐ │     CNI Plugin    │ ┌──────┐ │
│ │Pod A │ │ ←───(overlay/───→ │ │Pod C │ │
│ │10.244│ │      routing)     │ │10.244│ │
│ │.1.5  │ │                    │ │.2.8  │ │
│ └──────┘ │                    │ └──────┘ │
│ ┌──────┐ │                    │ ┌──────┐ │
│ │Pod B │ │                    │ │Pod D │ │
│ │10.244│ │                    │ │10.244│ │
│ │.1.6  │ │                    │ │.2.9  │ │
│ └──────┘ │                    │ └──────┘ │
└──────────┘                    └──────────┘

Every pod gets a unique IP!
Every pod can reach every other pod (across nodes)!
```

### The Kubernetes Networking Model

Kubernetes has a simple but powerful networking model with three rules:

1. **Every pod gets its own IP address**
2. **Pods on any node can communicate with pods on any other node WITHOUT NAT**
3. **The IP that a pod sees itself as is the SAME IP others see it as**

These rules are enforced by the CNI plugin.

## 📋 Summary — Worker Node Components

| Component | Role | Analogy | Key Port | Runs As |
|-----------|------|---------|----------|---------|
| **kubelet** | Node agent — ensures pods are running, health checks, status reporting | Construction foreman | 10250 | systemd service |
| **kube-proxy** | Network plumber — implements Services via iptables/IPVS rules | Switchboard operator | 10256 | DaemonSet pod |
| **Container Runtime** | Engine — pulls images and runs containers (containerd, CRI-O) | Construction crew | varies | systemd service |

### Key Takeaways:

1. **kubelet** is the most important component on a worker node — it's the bridge between the control plane and the actual containers
2. **kubelet is NOT a pod** — it's a systemd service, because it needs to run before any pods can start
3. **kube-proxy** makes Services work by creating network routing rules in the kernel
4. **iptables mode** is the default; **IPVS mode** is better for large clusters
5. **containerd** is the standard container runtime; Docker runtime was removed in K8s 1.24
6. **Docker images still work** — the image format didn't change, only the runtime
7. **CNI plugins** handle pod-to-pod networking across nodes
8. **Health probes** (liveness, readiness, startup) are run by the kubelet
9. **CrashLoopBackOff** is exponential backoff when a container keeps crashing
10. **Static pods** allow the kubelet to run pods without the API Server (used for control plane components)

### The Complete Request Flow (Deployment to Running Pods):

```
1. You:          kubectl apply -f deployment.yaml
2. API Server:   Authenticates → Authorizes → Validates → Saves to etcd
3. Controller:   Sees new Deployment → Creates ReplicaSet → Creates Pods
4. Scheduler:    Sees unscheduled pods → Filters/Scores nodes → Assigns pods
5. kubelet:      Sees assigned pods → Tells containerd to pull image & run
6. containerd:   Pulls image → Creates container → Starts process
7. kubelet:      Reports "Pod Running" → Starts health probes
8. kube-proxy:   Updates iptables/IPVS rules → Pod can receive traffic
9. Done!         Your app is running, accessible, self-healing, and scalable ✅
```
