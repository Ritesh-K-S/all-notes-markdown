# 10.30 — Cluster Autoscaler — Node Auto-Scaling

> **One Line Answer:** The Cluster Autoscaler **automatically adds or removes worker Nodes** from your Kubernetes cluster — when HPA wants to add more Pods but there's no room on existing Nodes, Cluster Autoscaler spins up new Nodes; when Nodes are underutilized and their Pods can fit elsewhere, it removes those Nodes to save money.

## 🤔 Why Do We Need Cluster Autoscaler?

### The Problem: HPA Creates Pods, But Nodes Run Out of Space

Remember HPA from 10.28? HPA scales Pods. But Pods need to run on Nodes. What happens when your Nodes are full?

```
Scenario: HPA wants to scale up, but the cluster is FULL

  Node 1 (4 CPU, 8Gi)              Node 2 (4 CPU, 8Gi)
  ┌─────────────────────┐          ┌─────────────────────┐
  │ [Pod A] 1 CPU, 2Gi  │          │ [Pod D] 1 CPU, 2Gi  │
  │ [Pod B] 1 CPU, 2Gi  │          │ [Pod E] 1 CPU, 2Gi  │
  │ [Pod C] 1 CPU, 2Gi  │          │ [Pod F] 1 CPU, 2Gi  │
  │ [System] 0.5 CPU    │          │ [System] 0.5 CPU    │
  │                      │          │                      │
  │ Free: 0.5 CPU, 2Gi  │          │ Free: 0.5 CPU, 2Gi  │
  └─────────────────────┘          └─────────────────────┘

  HPA: "CPU is at 95%! I need 4 more pods, each needs 1 CPU!"
  
  Scheduler: "Hmm... Node 1 has only 0.5 CPU free, Node 2 has only 0.5 CPU free."
  Scheduler: "I can't schedule ANY of these new pods!" 😰
  
  New Pods:
  [Pod G] 1 CPU → ⏳ PENDING (no node has enough room)
  [Pod H] 1 CPU → ⏳ PENDING
  [Pod I] 1 CPU → ⏳ PENDING
  [Pod J] 1 CPU → ⏳ PENDING
  
  These pods will stay PENDING forever! Your app can't scale! 💥
```

### The Other Problem: Paying for Idle Nodes

```
3 AM — No traffic. Most pods scaled down by HPA.

  Node 1 (4 CPU, 8Gi)              Node 2 (4 CPU, 8Gi)              Node 3 (4 CPU, 8Gi)
  ┌─────────────────────┐          ┌─────────────────────┐          ┌─────────────────────┐
  │ [Pod A] 0.5 CPU     │          │ [System] 0.5 CPU    │          │ [System] 0.5 CPU    │
  │ [System] 0.5 CPU    │          │                      │          │                      │
  │                      │          │                      │          │                      │
  │ Free: 3 CPU, 6Gi    │          │ Free: 3.5 CPU, 8Gi  │          │ Free: 3.5 CPU, 8Gi  │
  └─────────────────────┘          └─────────────────────┘          └─────────────────────┘
  
  Node 2 and Node 3 are almost EMPTY! But you're still paying for them! 💸💸
  
  At $0.10/hour per node × 3 nodes × 8 idle hours/night × 30 days = $72/month WASTED
  In a real cluster with 50+ nodes, this could be THOUSANDS per month!
```

### The Solution: Cluster Autoscaler

```
With Cluster Autoscaler:

SCALE UP — When pods are Pending:
  
  Cluster Autoscaler sees: "4 pods are Pending because no node has room."
  Cluster Autoscaler says: "I'll ask the cloud provider to add 2 new nodes!"
  
  Cloud Provider (AWS/GCP/Azure): "Sure! Here are 2 new EC2/GCE instances!"
  
  Node 1             Node 2             Node 3 (NEW!)       Node 4 (NEW!)
  ┌──────────┐      ┌──────────┐      ┌──────────┐        ┌──────────┐
  │ Pod A-C  │      │ Pod D-F  │      │ [Pod G]  │        │ [Pod I]  │
  │ (full)   │      │ (full)   │      │ [Pod H]  │        │ [Pod J]  │
  └──────────┘      └──────────┘      └──────────┘        └──────────┘
  
  All pods are running! Crisis averted! ✅


SCALE DOWN — When nodes are underutilized:

  Cluster Autoscaler sees: "Node 3 and Node 4 are nearly empty."
  Cluster Autoscaler says: "Let me move those pods to other nodes and remove these nodes."
  
  1. Pods on Node 3 and 4 are evicted and rescheduled on Node 1 and 2
  2. Cloud Provider terminates Node 3 and Node 4
  
  Node 1             Node 2
  ┌──────────┐      ┌──────────┐
  │ Pod A    │      │ Pod B    │
  └──────────┘      └──────────┘
  
  Fewer nodes = Lower cloud bill! 💰
```

## 🏗️ How Does Cluster Autoscaler Work?

### The Architecture

```
┌──────────────────────────────────────────────────────────────────┐
│                    Cluster Autoscaler                              │
│                                                                    │
│  Runs as a Deployment inside your cluster                         │
│  (usually in kube-system namespace)                               │
│                                                                    │
│  ┌──────────────────┐    ┌──────────────────────────────────┐    │
│  │  Scale Up Logic   │    │  Scale Down Logic                │    │
│  │                    │    │                                  │    │
│  │  Every 10 seconds: │    │  Every 10 seconds:              │    │
│  │  1. Check for      │    │  1. Check each node:            │    │
│  │     Pending pods   │    │     - Is utilization < 50%?     │    │
│  │  2. Can any node   │    │     - Can all pods move to      │    │
│  │     group fit them?│    │       other nodes?               │    │
│  │  3. If yes, add    │    │  2. If yes for 10 min straight, │    │
│  │     nodes to that  │    │     remove the node             │    │
│  │     node group     │    │                                  │    │
│  └────────┬───────────┘    └──────────┬───────────────────────┘    │
│           │                           │                            │
│           ▼                           ▼                            │
│  ┌──────────────────────────────────────────────────────────────┐ │
│  │            Cloud Provider API                                │ │
│  │  (AWS Auto Scaling Groups / GCP Managed Instance Groups /    │ │
│  │   Azure VM Scale Sets)                                       │ │
│  │                                                              │ │
│  │  Cluster Autoscaler calls the cloud API to:                  │ │
│  │  - Increase node group size (add nodes)                      │ │
│  │  - Decrease node group size (remove nodes)                   │ │
│  └──────────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────────┘
```

### Scale UP — Step by Step

```
Step 1: Pod gets created by HPA (or manually)
        [New Pod] → Status: Pending (no node can fit it)

Step 2: Cluster Autoscaler scans for Pending pods (every ~10 seconds)
        CA: "I found 3 Pending pods!"

Step 3: CA checks which node group can accommodate these pods
        CA: "Node group 'worker-pool-1' uses m5.large instances (2 CPU, 8Gi)."
        CA: "Each new node can fit 2 of these pods."
        CA: "I need 2 new nodes."

Step 4: CA calls the cloud API to increase the node group size
        CA → AWS API: "Set ASG 'worker-pool-1' desired count from 3 to 5"

Step 5: Cloud provider launches new instances (1-5 minutes)
        AWS: "Launching 2 new EC2 instances..."

Step 6: New nodes join the cluster (kubelet registers with API server)
        kubectl get nodes: "2 new nodes are Ready!"

Step 7: Scheduler places the Pending pods on the new nodes
        [Pod G] → Scheduled on Node 4 ✅
        [Pod H] → Scheduled on Node 4 ✅
        [Pod I] → Scheduled on Node 5 ✅
```

### Scale DOWN — Step by Step

```
Step 1: CA continuously monitors node utilization
        CA: "Node 4 is using only 20% of its resources."

Step 2: CA checks: "Can all pods on Node 4 move to other nodes?"
        CA: "Yes! Node 1 and Node 2 have enough free space."

Step 3: CA waits for the scale-down delay (default: 10 minutes)
        CA: "Node 4 has been underutilized for 10 minutes straight."

Step 4: CA cordons the node (marks it as unschedulable)
        kubectl cordon node-4

Step 5: CA drains the node (gracefully evicts all pods)
        kubectl drain node-4 --ignore-daemonsets --delete-emptydir-data

Step 6: Evicted pods get rescheduled on other nodes
        [Pod G] → Moved to Node 1 ✅
        [Pod H] → Moved to Node 2 ✅

Step 7: CA calls cloud API to terminate the empty node
        CA → AWS API: "Set ASG desired count from 5 to 4"
        AWS: "Terminating EC2 instance i-0abc123..."

Step 8: Node is gone. Money saved! 💰
```

## ☁️ Setting Up Cluster Autoscaler on AWS (EKS)

AWS is the most common cloud platform for Kubernetes, so let's start here.

### Prerequisites

- An EKS cluster
- Node groups (managed or self-managed) backed by Auto Scaling Groups (ASGs)
- IAM permissions for the Cluster Autoscaler to modify ASGs

### Step 1: Create an IAM Policy

Cluster Autoscaler needs permissions to describe and modify Auto Scaling Groups.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "autoscaling:DescribeAutoScalingGroups",
        "autoscaling:DescribeAutoScalingInstances",
        "autoscaling:DescribeLaunchConfigurations",
        "autoscaling:DescribeScalingActivities",
        "autoscaling:DescribeTags",
        "autoscaling:SetDesiredCapacity",
        "autoscaling:TerminateInstanceInAutoScalingGroup",
        "ec2:DescribeLaunchTemplateVersions",
        "ec2:DescribeInstanceTypes",
        "ec2:DescribeImages",
        "ec2:GetInstanceTypesFromInstanceRequirements",
        "eks:DescribeNodegroup"
      ],
      "Resource": ["*"]
    }
  ]
}
```

```bash
# Create the IAM policy
aws iam create-policy \
  --policy-name ClusterAutoscalerPolicy \
  --policy-document file://cluster-autoscaler-policy.json
```

### Step 2: Create an IAM Role for Service Account (IRSA)

```bash
# Replace with your values
CLUSTER_NAME=my-eks-cluster
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
OIDC_PROVIDER=$(aws eks describe-cluster --name $CLUSTER_NAME \
  --query "cluster.identity.oidc.issuer" --output text | sed 's|https://||')

# Create IAM role with trust policy for the service account
cat > trust-policy.json <<EOF
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Federated": "arn:aws:iam::${ACCOUNT_ID}:oidc-provider/${OIDC_PROVIDER}"
      },
      "Action": "sts:AssumeRoleWithWebIdentity",
      "Condition": {
        "StringEquals": {
          "${OIDC_PROVIDER}:aud": "sts.amazonaws.com",
          "${OIDC_PROVIDER}:sub": "system:serviceaccount:kube-system:cluster-autoscaler"
        }
      }
    }
  ]
}
EOF

aws iam create-role \
  --role-name ClusterAutoscalerRole \
  --assume-role-policy-document file://trust-policy.json

aws iam attach-role-policy \
  --role-name ClusterAutoscalerRole \
  --policy-arn arn:aws:iam::${ACCOUNT_ID}:policy/ClusterAutoscalerPolicy
```

### Step 3: Tag Your Auto Scaling Groups

Cluster Autoscaler finds node groups by looking for specific tags on ASGs:

```bash
# Tag your ASG (replace with your ASG name)
aws autoscaling create-or-update-tags --tags \
  ResourceId=my-asg-name,ResourceType=auto-scaling-group,Key=k8s.io/cluster-autoscaler/enabled,Value=true,PropagateAtLaunch=true \
  ResourceId=my-asg-name,ResourceType=auto-scaling-group,Key=k8s.io/cluster-autoscaler/my-eks-cluster,Value=owned,PropagateAtLaunch=true
```

### Step 4: Deploy Cluster Autoscaler

```yaml

kind: ServiceAccount
metadata:
  name: cluster-autoscaler
  namespace: kube-system
  annotations:
    eks.amazonaws.com/role-arn: arn:aws:iam::<ACCOUNT_ID>:role/ClusterAutoscalerRole
---
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: cluster-autoscaler

- apiGroups: [""]

- apiGroups: [""]

  verbs: ["watch", "list", "get", "update"]
- apiGroups: [""]

  verbs: ["watch", "list", "get"]
- apiGroups: ["extensions"]
  resources: ["replicasets", "daemonsets"]
  verbs: ["watch", "list", "get"]
- apiGroups: ["policy"]
  resources: ["poddisruptionbudgets"]
  verbs: ["watch", "list"]
- apiGroups: ["apps"]
  resources: ["statefulsets", "replicasets", "daemonsets"]
  verbs: ["watch", "list", "get"]

  verbs: ["watch", "list", "get"]
```

## ☁️ Setting Up Cluster Autoscaler on GCP (GKE)

GKE makes it super easy — Cluster Autoscaler is built-in!

### Enable During Cluster Creation

```bash
gcloud container clusters create my-cluster \
  --enable-autoscaling \
  --min-nodes=1 \
  --max-nodes=10 \
  --zone=us-central1-a
```

### Enable on an Existing Cluster

```bash
# Enable autoscaling on a specific node pool
gcloud container clusters update my-cluster \
  --enable-autoscaling \
  --min-nodes=1 \
  --max-nodes=10 \
  --node-pool=default-pool \
  --zone=us-central1-a
```

That's it! GKE manages the Cluster Autoscaler for you. No extra deployments needed.

## ⚙️ Key Configuration Parameters

Cluster Autoscaler has many flags to customize its behavior. Here are the most important ones:

### Scale Up Settings

| Flag | Default | What It Does |
|------|---------|-------------|
| `--scan-interval` | 10s | How often CA checks for pending pods |
| `--max-node-provision-time` | 15m | Max time to wait for a new node to become ready |
| `--max-nodes-total` | 0 (unlimited) | Maximum total nodes across all node groups |

### Scale Down Settings

| Flag | Default | What It Does |
|------|---------|-------------|
| `--scale-down-enabled` | true | Whether scale-down is enabled |
| `--scale-down-delay-after-add` | 10m | Don't scale down for 10 min after adding a node |
| `--scale-down-delay-after-delete` | 10s | Wait after deleting a node before considering more deletions |
| `--scale-down-delay-after-failure` | 3m | Wait after a failed scale-down attempt |
| `--scale-down-unneeded-time` | 10m | How long a node must be underutilized before removal |
| `--scale-down-utilization-threshold` | 0.5 | Node is "underutilized" if resource usage < 50% |
| `--scale-down-unready-time` | 20m | How long an unready node must be unready before removal |

### Expander Strategies

When multiple node groups can accommodate pending pods, the **expander** decides which one to use:

| Expander | What It Does |
|----------|-------------|
| `random` | Picks a random eligible node group |
| `most-pods` | Picks the node group that can schedule the most pending pods |
| `least-waste` | Picks the node group that will have the least idle resources after scheduling |
| `priority` | Uses a priority-based configuration (you define priorities) |
| `price` | Picks the cheapest node group (cloud provider dependent) |

```bash
# Set expander in Cluster Autoscaler flags
- --expander=least-waste
```

### Example: Production-Ready Configuration

```bash
# All these go as command flags in the CA Deployment
- --cloud-provider=aws
- --node-group-auto-discovery=asg:tag=k8s.io/cluster-autoscaler/enabled=true
- --expander=least-waste                      # Choose cheapest node type
- --scale-down-enabled=true
- --scale-down-delay-after-add=10m            # Don't remove nodes right after adding
- --scale-down-unneeded-time=10m              # Node must be idle for 10 min
- --scale-down-utilization-threshold=0.5      # Consider underutilized below 50%
- --skip-nodes-with-local-storage=false       # Allow scaling nodes with emptyDir
- --skip-nodes-with-system-pods=false         # Allow scaling nodes with system pods
- --balance-similar-node-groups=true          # Keep similar node groups balanced
- --max-node-provision-time=15m               # Allow 15 min for node to be ready
- --max-graceful-termination-sec=600          # 10 min for pods to terminate gracefully
```

## 🛡️ Pods That Prevent Scale Down

Cluster Autoscaler won't remove a node if it has pods that can't be moved. Here are the types of pods that block scale-down:

### 1. Pods with Local Storage (emptyDir)

```yaml
# This pod uses emptyDir — data is lost if pod moves
volumes:
- name: cache
  emptyDir: {}
```

**Fix:** Use `--skip-nodes-with-local-storage=false` to allow CA to evict these pods.

### 2. Pods Not Managed by a Controller

```bash
# Standalone pod (not part of a Deployment/ReplicaSet)
kubectl run my-pod --image=nginx
# CA won't remove the node because this pod can't be rescheduled!
```

**Fix:** Always use Deployments, not standalone Pods.

### 3. Pods with PodDisruptionBudget (PDB)

```yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: my-pdb
spec:
  minAvailable: 3        # At least 3 pods must be running
  selector:
    matchLabels:
      app: my-app
```

If evicting a pod would violate the PDB, CA won't remove the node.

### 4. Pods with `cluster-autoscaler.kubernetes.io/safe-to-evict: "false"` Annotation

```yaml
metadata:
  annotations:
    cluster-autoscaler.kubernetes.io/safe-to-evict: "false"
```

This annotation explicitly tells CA: "Don't evict this pod."

### 5. Kube-system Pods (by default)

System pods in `kube-system` block scale-down by default. 

**Fix:** Use `--skip-nodes-with-system-pods=false` flag.

### How to Mark Pods as Safe to Evict

```yaml
metadata:
  annotations:
    cluster-autoscaler.kubernetes.io/safe-to-evict: "true"
```

This tells CA: "It's okay to evict this pod for scale-down."

## 🔗 The Complete Autoscaling Stack — HPA + VPA + Cluster Autoscaler

All three autoscalers work together to form a complete scaling solution:

```
                    Traffic/Load Increases
                           │
                           ▼
              ┌─────────────────────────┐
              │  HPA (Pod Autoscaler)   │
              │                         │
              │  "More traffic? I need  │
              │   more Pod replicas!"   │
              │                         │
              │  Scale: 3 → 8 pods      │
              └────────────┬────────────┘
                           │
                           ▼
                    Scheduler tries to 
                    place new pods...
                           │
                    ┌──────┴──────┐
                    │             │
               Nodes have    Nodes are
               enough room   FULL! 😰
                    │             │
                    ▼             ▼
              Pods run! ✅   Pods are PENDING
                              │
                              ▼
              ┌─────────────────────────┐
              │  Cluster Autoscaler     │
              │  (Node Autoscaler)      │
              │                         │
              │  "Pods are pending!     │
              │   I need more Nodes!"   │
              │                         │
              │  Scale: 3 → 5 nodes     │
              └────────────┬────────────┘
                           │
                           ▼
                  New nodes join cluster
                  Pending pods get scheduled ✅

              ┌─────────────────────────┐
              │  VPA (Resource Sizer)   │
              │                         │
              │  "These pods are using  │
              │   more CPU than their   │
              │   requests. Let me      │
              │   right-size them."     │
              │                         │
              │  Requests: 100m → 250m  │
              └─────────────────────────┘
```

### How They Complement Each Other:

| Autoscaler | What It Scales | Direction | When It Acts |
|-----------|---------------|-----------|-------------|
| **HPA** | Number of Pods | Horizontal (more replicas) | When pod metrics exceed targets |
| **VPA** | Pod resource requests | Vertical (bigger pods) | When actual usage ≠ requests |
| **Cluster Autoscaler** | Number of Nodes | Infrastructure | When pods can't be scheduled |

### The Typical Flow:

```
1. Traffic increases
2. HPA detects high CPU/metrics → creates more pods
3. If nodes are full → pods stay Pending
4. Cluster Autoscaler detects Pending pods → adds nodes
5. Pods get scheduled on new nodes
6. VPA (in background) adjusts resource requests for future pods

Traffic decreases:
1. HPA detects low CPU/metrics → removes pods
2. Nodes become underutilized
3. Cluster Autoscaler detects idle nodes → removes them
4. VPA adjusts resource requests down for cost savings
```

## 🧪 Hands-On Lab: Watch Cluster Autoscaler in Action

### Setup (Using Minikube with Multi-Node)

> **Note:** True Cluster Autoscaler requires a cloud provider. For local testing, we'll simulate the behavior.

On a real cloud cluster (EKS/GKE/AKS):

### Step 1: Check Current Nodes

```bash
kubectl get nodes
# NAME                          STATUS   ROLES    AGE   VERSION
# node-1                        Ready    <none>   10d   v1.29.0
# node-2                        Ready    <none>   10d   v1.29.0
```

### Step 2: Create a Deployment That Fills Up Nodes

```yaml
# inflate.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: inflate
spec:
  replicas: 1
  selector:
    matchLabels:
      app: inflate
  template:
    metadata:
      labels:
        app: inflate
    spec:
      containers:
      - name: inflate
        image: public.ecr.aws/eks-distro/kubernetes/pause:3.7
        resources:
          requests:
            cpu: "1"            # Each pod requests 1 full CPU core!
            memory: "1Gi"
```

```bash
kubectl apply -f inflate.yaml
```

### Step 3: Scale Up to Force Pending Pods

```bash
# Scale to a number that exceeds your cluster capacity
kubectl scale deployment inflate --replicas=20

# Watch pods — some will be Pending!
kubectl get pods -l app=inflate --watch

# You'll see something like:
# inflate-xxx   Running   (on existing nodes)
# inflate-xxx   Running
# inflate-xxx   Pending   ← No room!
# inflate-xxx   Pending   ← No room!
# inflate-xxx   Pending   ← No room!
```

### Step 4: Watch Cluster Autoscaler Add Nodes

```bash
# Watch nodes being added
kubectl get nodes --watch

# Check CA logs
kubectl logs -n kube-system -l app=cluster-autoscaler --tail=50

# You should see:
# "Scale-up: setting group size from 2 to 5"
# "Successfully added 3 nodes"

# After a few minutes, new nodes appear:
# node-3   Ready   <none>   1m   v1.29.0
# node-4   Ready   <none>   1m   v1.29.0
# node-5   Ready   <none>   1m   v1.29.0

# And Pending pods start Running!
```

### Step 5: Scale Down and Watch Nodes Removed

```bash
# Scale down the deployment
kubectl scale deployment inflate --replicas=2

# Wait ~10 minutes (scale-down-unneeded-time)
# Watch nodes being removed
kubectl get nodes --watch

# CA logs will show:
# "Scale-down: node node-4 is unneeded for 10m0s"
# "Scale-down: removing node node-4"
# "Successfully removed 1 node"
```

### Step 6: Clean Up

```bash
kubectl delete deployment inflate
```

## 🔍 Monitoring Cluster Autoscaler

### Check CA Status ConfigMap

```bash
kubectl get configmap cluster-autoscaler-status -n kube-system -o yaml
```

This shows:
- Current state of each node group
- Pending scale-up/scale-down operations
- Health status

### Check CA Logs

```bash
# Follow CA logs in real-time
kubectl logs -n kube-system deployment/cluster-autoscaler -f

# Key log messages to look for:
# "Scale-up: setting group X desired size to Y"     ← Adding nodes
# "Scale-down: node X is unneeded"                   ← Considering removal
# "Scale-down: removing node X"                      ← Actually removing
# "No candidates for scale down"                     ← All nodes are needed
# "Pod X is unschedulable"                           ← Trigger for scale up
```

### CA Events

```bash
# Check events related to Cluster Autoscaler
kubectl get events --field-selector source=cluster-autoscaler -A

# Events on specific nodes
kubectl describe node <node-name> | grep -A 5 "cluster-autoscaler"
```

### Prometheus Metrics

Cluster Autoscaler exposes Prometheus metrics:

```
# Key metrics:
cluster_autoscaler_scaled_up_nodes_total           # Total nodes added
cluster_autoscaler_scaled_down_nodes_total          # Total nodes removed
cluster_autoscaler_unschedulable_pods_count         # Current pending pods
cluster_autoscaler_nodes_count                      # Current node count
cluster_autoscaler_function_duration_seconds         # CA loop duration
cluster_autoscaler_last_activity                    # Last scaling activity timestamp
```

## 🚫 Common Mistakes & Troubleshooting

### Mistake 1: Nodes Not Scaling Up

**Symptoms:** Pods are Pending, but no new nodes are being added.

```bash
# Debug checklist:
# 1. Is Cluster Autoscaler running?
kubectl get pods -n kube-system -l app=cluster-autoscaler

# 2. Check CA logs for errors
kubectl logs -n kube-system -l app=cluster-autoscaler --tail=100

# 3. Are you at max nodes already?
kubectl get nodes | wc -l
# Compare with your maxNodes setting

# 4. Does the node group have the right tags?
# (AWS) Check ASG tags include k8s.io/cluster-autoscaler/enabled=true

# 5. Does the pod have resource requests?
kubectl get pod <pending-pod> -o yaml | grep -A 5 resources
# If no requests, CA might not know how much capacity is needed
```

### Mistake 2: Nodes Not Scaling Down

**Symptoms:** Nodes are nearly empty but not being removed.

```bash
# Debug checklist:
# 1. Check what's preventing scale-down
kubectl logs -n kube-system -l app=cluster-autoscaler | grep "cannot be removed"

# 2. Common blockers:
#    - Standalone pods (not managed by a controller)
kubectl get pods --field-selector=status.phase=Running -o json | \
  jq '.items[] | select(.metadata.ownerReferences == null) | .metadata.name'

#    - Pods with local storage
kubectl get pods -o json | jq '.items[] | select(.spec.volumes[]?.emptyDir != null) | .metadata.name'

#    - Pods with safe-to-evict=false annotation
kubectl get pods -o json | jq '.items[] | select(.metadata.annotations["cluster-autoscaler.kubernetes.io/safe-to-evict"] == "false") | .metadata.name'

# 3. Is scale-down even enabled?
# Check CA flags for --scale-down-enabled=true
```

### Mistake 3: Scale-Down Is Too Aggressive

```bash
# Increase the delay and utilization threshold
- --scale-down-unneeded-time=20m              # Wait 20 min instead of 10
- --scale-down-delay-after-add=15m            # Wait 15 min after adding a node
- --scale-down-utilization-threshold=0.65     # Only scale down if usage < 65%
```

### Mistake 4: Wrong Node Type for Workload

If your pods need GPUs but your node group only has CPU nodes, CA will keep adding CPU nodes that can't schedule the GPU pods.

**Fix:** Create multiple node groups with different instance types:

```bash
# Node group 1: General purpose (CPU workloads)
# m5.large, min=2, max=20

# Node group 2: GPU workloads
# p3.2xlarge, min=0, max=5

# Node group 3: Memory-intensive workloads
# r5.2xlarge, min=0, max=10
```

Use `nodeSelector` or `nodeAffinity` to place pods on the right node group:

```yaml
spec:
  nodeSelector:
    node-type: gpu          # Only schedule on GPU nodes
```

### Mistake 5: Not Setting Resource Requests

```yaml
# ❌ BAD — No resource requests
containers:
- name: my-app
  image: my-app:latest
  # CA can't determine how much capacity each pod needs!

# ✅ GOOD — Always set resource requests
containers:
- name: my-app
  image: my-app:latest
  resources:
    requests:
      cpu: "250m"
      memory: "256Mi"
```

## 🆕 Karpenter — The Modern Alternative to Cluster Autoscaler

**Karpenter** is a newer, faster, and more flexible node autoscaler built by AWS (but it works with other providers too). It's becoming the preferred choice for many teams.

### Cluster Autoscaler vs Karpenter

| Feature | Cluster Autoscaler | Karpenter |
|---------|-------------------|-----------|
| **Speed** | Slow (works through ASGs) | Fast (directly provisions instances) |
| **Node type selection** | Fixed per node group | Dynamic — picks the best instance type per pod |
| **Configuration** | ASG min/max per node group | Provisioner CRDs — more flexible |
| **Bin-packing** | Basic | Advanced — better resource utilization |
| **Cloud support** | AWS, GCP, Azure, etc. | Originally AWS-only, now expanding |
| **Scale from zero** | Slow | Fast (no ASG warm-up needed) |
| **Spot instances** | Manual ASG config | Native support with fallback |

### How Karpenter Works (High Level)

```
1. Pod is Pending (no node can fit it)
2. Karpenter looks at the pod's requirements:
   - How much CPU/memory?
   - Any node selector or affinity?
   - Any GPU requirements?
   - Any topology spread constraints?
3. Karpenter DIRECTLY launches the best-fit EC2 instance
   (no ASG involved — it calls EC2 API directly!)
4. Node joins the cluster in ~60 seconds
5. Pod gets scheduled ✅
```

### Quick Karpenter Example

```yaml
# Karpenter NodePool (replaces CA's node group concept)
apiVersion: karpenter.sh/v1beta1
kind: NodePool
metadata:
  name: default
spec:
  template:
    spec:
      requirements:
      - key: "karpenter.k8s.aws/instance-category"
        operator: In
        values: ["c", "m", "r"]           # Allow c-type, m-type, r-type instances
      - key: "karpenter.k8s.aws/instance-size"
        operator: In
        values: ["medium", "large", "xlarge", "2xlarge"]
      - key: "kubernetes.io/arch"
        operator: In
        values: ["amd64"]
      - key: "karpenter.sh/capacity-type"
        operator: In
        values: ["on-demand", "spot"]     # Allow both on-demand and spot!
  limits:
    cpu: "1000"                           # Max 1000 CPU cores total
    memory: "1000Gi"                      # Max 1000 GiB memory total
  disruption:
    consolidationPolicy: WhenUnderutilized  # Automatically consolidate underused nodes
    expireAfter: 720h                     # Refresh nodes every 30 days
```

> **Note:** Karpenter is covered briefly here for awareness. If you're starting fresh on AWS, consider Karpenter over Cluster Autoscaler.

## 🏆 Best Practices for Cluster Autoscaler

| Practice | Why |
|----------|-----|
| **Always set resource requests on pods** | CA needs this to determine if pods can fit on nodes |
| **Use multiple node groups** | Different workloads need different instance types (CPU, GPU, memory) |
| **Set maxNodes as a cost guard** | Prevents runaway scaling from burning your cloud budget |
| **Use `least-waste` expander** | Picks the node type that wastes the least resources |
| **Set scale-down-delay-after-add** | Prevents removing nodes that were just added (thrashing) |
| **Use PodDisruptionBudgets** | Controls how many pods can be evicted during scale-down |
| **Don't use standalone Pods** | Use Deployments so pods can be rescheduled to other nodes |
| **Annotate pods as safe-to-evict** | If a pod can handle restarts, mark it so CA can scale down |
| **Monitor CA metrics and logs** | Know when scaling happens and catch issues early |
| **Consider Karpenter for AWS** | Faster, smarter, more flexible than traditional CA |

## 🔑 Key Takeaways

1. **Cluster Autoscaler = Automatic node scaling** — adds nodes when pods are Pending, removes nodes when they're underutilized.
2. **It works with cloud providers** — AWS ASGs, GCP MIGs, Azure VMSS. It calls cloud APIs to add/remove nodes.
3. **Scale-up is triggered by Pending pods** — if the scheduler can't place a pod, CA steps in.
4. **Scale-down checks utilization** — if a node is below 50% usage for 10 minutes and its pods can move elsewhere, the node gets removed.
5. **Some pods block scale-down** — standalone pods, pods with local storage, pods with PDBs, or pods with the `safe-to-evict: "false"` annotation.
6. **The complete stack:** HPA (scale pods) + VPA (size pods) + Cluster Autoscaler (scale nodes) = fully automated scaling.
7. **Karpenter** is a modern alternative (especially for AWS) that's faster and smarter than traditional CA.
8. **Always set resource requests** — without them, neither HPA, VPA, nor Cluster Autoscaler can make good decisions.

```
Key Flags:
  --scan-interval=10s                    How often to check for pending pods
  --scale-down-unneeded-time=10m         How long node must be idle before removal

  --expander=least-waste                 Strategy for choosing node groups
  --max-nodes-total=100                  Max total nodes (cost protection)

Scale Up Trigger:  Pods in Pending state (unschedulable)
Scale Down Trigger: Node utilization < threshold for unneeded-time

Prevent Eviction:  
  annotation: cluster-autoscaler.kubernetes.io/safe-to-evict: "false"
Allow Eviction:    
  annotation: cluster-autoscaler.kubernetes.io/safe-to-evict: "true"
```
