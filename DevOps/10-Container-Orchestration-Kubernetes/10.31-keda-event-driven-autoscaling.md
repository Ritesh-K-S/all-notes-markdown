## What is KEDA?

**KEDA** stands for **Kubernetes Event-Driven Autoscaling**. It's an open-source project (originally created by Microsoft and Red Hat, now a **CNCF Graduated** project) that lets you **scale your Kubernetes workloads based on external events** — not just CPU or memory.

### The Problem KEDA Solves

Let's say you have a worker application that processes messages from a **RabbitMQ queue** or **Kafka topic**. With the built-in Kubernetes **HPA (Horizontal Pod Autoscaler)**, you can only scale based on CPU or memory usage. But what you *really* want is:

> "If there are 500 messages waiting in the queue, spin up more pods. If the queue is empty, scale down to **zero**."

That's **exactly** what KEDA does. 🎯

### KEDA in One Sentence

> KEDA bridges the gap between **external event sources** (queues, databases, streams, HTTP requests, cron schedules, etc.) and **Kubernetes autoscaling** — including the ability to scale all the way down to **zero pods**.

---

## Why Not Just Use HPA?

| Feature | HPA (Built-in) | KEDA |
|---------|----------------|------|
| Scale based on CPU/Memory | ✅ Yes | ✅ Yes |
| Scale based on custom metrics | ⚠️ Requires Prometheus Adapter setup | ✅ Built-in, easy |
| Scale based on external events (queues, etc.) | ❌ No | ✅ Yes |
| Scale to zero | ❌ No (minimum 1 pod) | ✅ Yes |
| 60+ event source integrations | ❌ No | ✅ Yes |
| Works with existing HPA | N/A | ✅ Extends HPA |

**Bottom line:** KEDA doesn't replace HPA — it **extends** it. Under the hood, KEDA actually creates and manages HPA objects for you.

## KEDA Architecture — How It Works

KEDA has three main components:

```
┌─────────────────────────────────────────────────────────┐
│                    KEDA Components                       │
│                                                          │
│  ┌──────────────────┐   ┌──────────────────────────┐    │
│  │   KEDA Operator   │   │   KEDA Metrics Server     │    │
│  │  (keda-operator)  │   │ (keda-operator-metrics-   │    │
│  │                    │   │  apiserver)                │    │
│  │  - Watches Scaled  │   │  - Exposes external       │    │
│  │    Objects          │   │    metrics to Kubernetes  │    │
│  │  - Creates/Manages │   │    Metrics API             │    │
│  │    HPA              │   │  - HPA reads from here    │    │
│  │  - Scales to/from  │   │                            │    │
│  │    zero             │   │                            │    │
│  └──────────────────┘   └──────────────────────────┘    │
│                                                          │
│  ┌──────────────────────────────────────────────────┐   │
│  │              Admission Webhooks                    │   │
│  │  - Validates ScaledObject/ScaledJob configs        │   │
│  └──────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

### How scaling works step by step:

1. **You define a `ScaledObject`** (or `ScaledJob`) — telling KEDA what to scale and based on what event source.
2. **KEDA Operator** watches your ScaledObject and connects to the external event source (e.g., RabbitMQ).
3. **KEDA Metrics Server** exposes those external metrics through the Kubernetes Metrics API.
4. **KEDA creates an HPA** behind the scenes, feeding it the external metrics.
5. The **HPA scales your deployment** up or down based on the metric values.
6. If the metric value is **zero** (e.g., empty queue), KEDA scales the deployment to **zero pods** (HPA can't do this — KEDA handles it directly).
7. When a new event arrives, KEDA detects it and scales back up from zero — this is called **activation**.

## Core Concepts & Custom Resources

KEDA introduces **4 custom resource types** (CRDs):

### 1. `ScaledObject` — Scale Deployments/StatefulSets

This is the most commonly used resource. It tells KEDA: *"Watch this event source and scale this Deployment."*

### 2. `ScaledJob` — Scale Jobs

Instead of scaling a Deployment (long-running pods), this creates **Kubernetes Jobs** for each event — perfect for one-off processing tasks.

### 3. `TriggerAuthentication` — Auth within a Namespace

Stores authentication details (credentials, connection strings) for connecting to event sources. Scoped to a **single namespace**.

### 4. `ClusterTriggerAuthentication` — Auth Cluster-Wide

Same as TriggerAuthentication, but available **across all namespaces**.

## ScaledObject — Deep Dive

The `ScaledObject` is the heart of KEDA. Let's break down every field:

```yaml
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: my-scaledobject            # Name of this ScaledObject
  namespace: default               # Namespace where your workload lives
spec:
  scaleTargetRef:
    apiVersion: apps/v1            # API version of the target (optional, defaults to apps/v1)
    kind: Deployment               # What to scale — Deployment or StatefulSet
    name: my-worker-deployment     # Name of the Deployment to scale
  
  pollingInterval: 30              # How often KEDA checks the event source (seconds, default: 30)
  cooldownPeriod: 300              # Wait time before scaling to zero after last trigger (seconds, default: 300)
  
  minReplicaCount: 0              # Minimum replicas (0 = scale to zero, default: 0)
  maxReplicaCount: 100            # Maximum replicas (default: 100)
  
  fallback:                        # What to do if the scaler fails to get metrics
    failureThreshold: 3            # Number of failures before using fallback
    replicas: 5                    # Fallback replica count
  
  advanced:
    restoreToOriginalReplicaCount: false   # Restore original replicas when ScaledObject is deleted?
    horizontalPodAutoscalerConfig:
      behavior:                    # Same as HPA behavior spec
        scaleDown:
          stabilizationWindowSeconds: 300
          policies:
            - type: Percent
              value: 100
              periodSeconds: 15
  
  triggers:                        # One or more event sources to watch
    - type: rabbitmq               # The type of scaler (there are 60+ options)
      metadata:
        queueName: my-queue
        queueLength: "5"           # Scale up when there are more than 5 messages per pod
      authenticationRef:
        name: rabbitmq-auth        # Reference to TriggerAuthentication
```

### Key Fields Explained

| Field | What It Does | Default |
|-------|-------------|---------|
| `pollingInterval` | How often (in seconds) KEDA checks the event source for new data | 30 |
| `cooldownPeriod` | After the last trigger fires, how long to wait before scaling to zero | 300 |
| `minReplicaCount` | Minimum pods. Set to `0` to enable scale-to-zero | 0 |
| `maxReplicaCount` | Maximum pods KEDA will scale to | 100 |
| `fallback` | If KEDA can't reach the event source, use this replica count as a safety net | — |
| `triggers` | The actual event sources that drive scaling decisions | — |

## ScaledJob — For Batch Processing

While `ScaledObject` scales Deployments (long-running pods), `ScaledJob` creates **Kubernetes Jobs** for each batch of events. This is great for:

- Processing files uploaded to a storage bucket
- Handling messages where each message is a self-contained task
- Running one-off computations triggered by events

```yaml
apiVersion: keda.sh/v1alpha1
kind: ScaledJob
metadata:
  name: my-scaledjob
  namespace: default
spec:
  jobTargetRef:
    parallelism: 1                    # How many pods per job
    completions: 1                    # How many completions needed
    backoffLimit: 3                   # Retries on failure
    template:
      spec:
        containers:
          - name: worker
            image: my-worker:latest
            command: ["python", "process.py"]
        restartPolicy: Never
  
  pollingInterval: 30
  maxReplicaCount: 30                 # Max concurrent jobs
  successfulJobsHistoryLimit: 5       # How many completed jobs to keep
  failedJobsHistoryLimit: 3           # How many failed jobs to keep
  
  scalingStrategy:
    strategy: "default"               # Options: default, custom, accurate
  
  triggers:
    - type: azure-queue
      metadata:
        queueName: my-processing-queue
        queueLength: "1"              # 1 job per message
```

## Practical Examples

### Example 1: Scale Based on RabbitMQ Queue Length

**Scenario:** You have a worker deployment that processes messages from RabbitMQ. You want:
- Scale to 0 when the queue is empty
- 1 pod per 10 messages in the queue
- Maximum 20 pods

**Step 1: Create the TriggerAuthentication**

```yaml
# rabbitmq-auth.yaml
apiVersion: v1
kind: Secret
metadata:
  name: rabbitmq-secret
  namespace: default
type: Opaque
data:
  host: YW1xcDovL3VzZXI6cGFzc0ByYWJiaXRtcS5kZWZhdWx0LnN2Yy5jbHVzdGVyLmxvY2FsOjU2NzI=  # base64 encoded
---
apiVersion: keda.sh/v1alpha1
kind: TriggerAuthentication
metadata:
  name: rabbitmq-auth
  namespace: default
spec:
  secretTargetRef:
    - parameter: host
      name: rabbitmq-secret
      key: host
```

**Step 2: Create the worker Deployment**

```yaml
# worker-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: rabbitmq-worker
  namespace: default
spec:
  replicas: 1                # KEDA will manage this
  selector:
    matchLabels:
      app: rabbitmq-worker
  template:
    metadata:
      labels:
        app: rabbitmq-worker
    spec:
      containers:
        - name: worker
          image: my-rabbitmq-worker:latest
          env:
            - name: RABBITMQ_URL
              valueFrom:
                secretKeyRef:
                  name: rabbitmq-secret
                  key: host
```

**Step 3: Create the ScaledObject**

```yaml
# rabbitmq-scaledobject.yaml
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: rabbitmq-scaledobject
  namespace: default
spec:
  scaleTargetRef:
    name: rabbitmq-worker             # Name of the Deployment
  pollingInterval: 15                  # Check every 15 seconds
  cooldownPeriod: 120                  # Wait 2 minutes before scaling to zero
  minReplicaCount: 0                   # Allow scale to zero
  maxReplicaCount: 20                  # Max 20 pods
  triggers:
    - type: rabbitmq
      metadata:
        protocol: amqp
        queueName: task-queue
        mode: QueueLength
        value: "10"                    # 1 pod per 10 messages
      authenticationRef:
        name: rabbitmq-auth            # Reference the auth we created
```

**How it behaves:**

| Queue Messages | Pods Running |
|---------------|-------------|
| 0 | 0 (scaled to zero!) |
| 1–10 | 1 |
| 11–20 | 2 |
| 50 | 5 |
| 200 | 20 (max) |
| 500 | 20 (capped at max) |

### Example 2: Scale Based on Kafka Consumer Lag

**Scenario:** Your consumer group is falling behind. Scale up consumers when lag increases.

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: kafka-secret
type: Opaque
data:
  sasl: cGxhaW50ZXh0        # base64 of "plaintext"
  username: bXktdXNlcg==     # base64 of "my-user"
  password: bXktcGFzcw==     # base64 of "my-pass"
---
apiVersion: keda.sh/v1alpha1
kind: TriggerAuthentication
metadata:
  name: kafka-auth
spec:
  secretTargetRef:
    - parameter: sasl
      name: kafka-secret
      key: sasl
    - parameter: username
      name: kafka-secret
      key: username
    - parameter: password
      name: kafka-secret
      key: password
---
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: kafka-scaledobject
spec:
  scaleTargetRef:
    name: kafka-consumer
  pollingInterval: 10
  cooldownPeriod: 300
  minReplicaCount: 1                  # Keep at least 1 consumer always running
  maxReplicaCount: 50
  triggers:
    - type: kafka
      metadata:
        bootstrapServers: kafka-broker.default.svc.cluster.local:9092
        consumerGroup: my-consumer-group
        topic: my-events-topic
        lagThreshold: "100"           # Scale up when lag > 100 per partition
        offsetResetPolicy: latest
      authenticationRef:
        name: kafka-auth
```

**Explanation:**
- `lagThreshold: "100"` — If the total consumer lag across all partitions is 100, KEDA will set desired replicas to 1. If lag is 500, it'll want 5 replicas. If lag is 5000, it'll want 50 (capped at `maxReplicaCount`).
- `minReplicaCount: 1` — Unlike the RabbitMQ example, here we always want at least 1 consumer running.

### Example 3: Scale Based on Prometheus Metrics

**Scenario:** Scale a web API based on request rate (requests per second) from Prometheus.

```yaml
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: prometheus-scaledobject
spec:
  scaleTargetRef:
    name: my-web-api
  pollingInterval: 15
  cooldownPeriod: 60
  minReplicaCount: 2                   # Always keep at least 2 pods
  maxReplicaCount: 30
  triggers:
    - type: prometheus
      metadata:
        serverAddress: http://prometheus-server.monitoring.svc.cluster.local:9090
        metricName: http_requests_total
        query: sum(rate(http_requests_total{deployment="my-web-api"}[2m]))
        threshold: "50"               # Scale when rate exceeds 50 rps per pod
        activationThreshold: "5"      # Don't activate below 5 rps total
```

**Explanation:**
- `query` — This is a standard PromQL query. It calculates the total request rate across all pods over a 2-minute window.
- `threshold: "50"` — Each pod should handle 50 requests per second. If the total rate is 200 rps, KEDA wants 4 pods.
- `activationThreshold: "5"` — The metric value must exceed 5 before KEDA activates scaling (moves from 0 to `minReplicaCount`). Since `minReplicaCount` is 2 here, this mainly prevents unnecessary HPA activity during very low traffic.

### Example 4: Scale on a Cron Schedule

**Scenario:** You know your app gets heavy traffic every weekday between 9 AM and 6 PM. Pre-scale before the rush.

```yaml
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: cron-scaledobject
spec:
  scaleTargetRef:
    name: my-web-app
  minReplicaCount: 2
  maxReplicaCount: 20
  triggers:
    - type: cron
      metadata:
        timezone: Asia/Kolkata               # Your timezone
        start: 0 9 * * 1-5                   # 9:00 AM, Monday to Friday
        end: 0 18 * * 1-5                    # 6:00 PM, Monday to Friday
        desiredReplicas: "10"                # Scale to 10 pods during business hours
```

**Explanation:**
- Between 9 AM and 6 PM on weekdays → 10 pods
- Outside those hours → falls back to `minReplicaCount` (2 pods)
- You can combine cron triggers with other triggers (e.g., also scale on CPU) — KEDA uses the **highest** value from all triggers.

## Multiple Triggers — Combining Event Sources

You can define **multiple triggers** on a single ScaledObject. KEDA uses the **maximum** desired replica count from all triggers.

```yaml
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: multi-trigger-scaledobject
spec:
  scaleTargetRef:
    name: my-app
  minReplicaCount: 1
  maxReplicaCount: 50
  triggers:
    # Trigger 1: Scale based on CPU
    - type: cpu
      metricType: Utilization
      metadata:
        value: "70"                    # Scale when CPU > 70%
    
    # Trigger 2: Scale based on queue
    - type: rabbitmq
      metadata:
        queueName: task-queue
        queueLength: "10"
      authenticationRef:
        name: rabbitmq-auth
    
    # Trigger 3: Pre-scale for business hours
    - type: cron
      metadata:
        timezone: UTC
        start: 0 8 * * 1-5
        end: 0 20 * * 1-5
        desiredReplicas: "5"
```

**How KEDA decides:** If CPU says "need 3 pods", queue says "need 7 pods", and cron says "need 5 pods" → KEDA scales to **7 pods** (the maximum).

## TriggerAuthentication — Connecting Securely

Most event sources need credentials. KEDA supports multiple auth methods:

### Method 1: From Kubernetes Secrets

```yaml
apiVersion: keda.sh/v1alpha1
kind: TriggerAuthentication
metadata:
  name: my-auth
spec:
  secretTargetRef:
    - parameter: connection          # Parameter name expected by the scaler
      name: my-secret                # Kubernetes Secret name
      key: connection-string         # Key inside the Secret
```

### Method 2: Using Pod Identity (AWS IRSA / Azure Workload Identity)

```yaml
apiVersion: keda.sh/v1alpha1
kind: TriggerAuthentication
metadata:
  name: aws-pod-identity-auth
spec:
  podIdentity:
    provider: aws-eks                # Options: aws-eks, azure-workload, gcp, none
```

### Method 3: From HashiCorp Vault

```yaml
apiVersion: keda.sh/v1alpha1
kind: TriggerAuthentication
metadata:
  name: vault-auth
spec:
  hashiCorpVault:
    address: https://vault.example.com
    authentication: token
    credential:
      token: vault-token
    secrets:
      - parameter: connection
        key: secret/data/myapp
        path: connectionString
```

### Method 4: From Environment Variables

```yaml
apiVersion: keda.sh/v1alpha1
kind: TriggerAuthentication
metadata:
  name: env-auth
spec:
  env:
    - parameter: connection
      name: QUEUE_CONNECTION_STRING  # Env var from the scaled workload's container
      containerName: worker
```

## Scale-to-Zero — How It Works

This is KEDA's superpower. Here's exactly how it works:

```
Queue has 0 messages for 300 seconds (cooldownPeriod)
           │
           ▼
   KEDA scales Deployment to 0 replicas
           │
           ▼
   No pods running. Zero cost. 💰
           │
           ▼
   New message arrives in queue
           │
           ▼
   KEDA detects it (within pollingInterval)
           │
           ▼
   KEDA scales Deployment to 1 replica (activation)
           │
           ▼
   HPA takes over for further scaling (1 → N)
```

### Important Notes on Scale-to-Zero:

1. **Cold start:** When scaling from 0 → 1, there's a delay (pod scheduling + container startup). This could be 5–30 seconds depending on image size and pull policy.
2. **No requests lost (for queues):** Messages stay in the queue until a pod picks them up.
3. **Not suitable for HTTP services:** If your app serves HTTP traffic, scaling to zero means requests will fail during the cold start. Use `minReplicaCount: 1` for HTTP services (or look into **KEDA HTTP Add-on**).

### KEDA HTTP Add-on (for HTTP Workloads)

KEDA has an experimental HTTP add-on that can proxy HTTP requests and queue them during scale-from-zero:

```yaml
apiVersion: http.keda.sh/v1alpha1
kind: HTTPScaledObject
metadata:
  name: my-http-app
spec:
  hosts:
    - example.com
  targetPendingRequests: 100         # Scale based on pending HTTP requests
  scaleTargetRef:
    name: my-http-deployment
    kind: Deployment
    apiVersion: apps/v1
    service: my-http-service
    port: 8080
  replicas:
    min: 0
    max: 10
```

```bash
kubectl get scaledobject rabbitmq-scaledobject -o jsonpath='{.status.conditions}' | jq .
```

Key conditions:
- **Ready:** `True` = KEDA is actively managing this object
- **Active:** `True` = At least one trigger is reporting metrics above activation threshold
- **Fallback:** `True` = KEDA can't reach the event source and is using fallback replicas

## Pausing Autoscaling

Sometimes you want to temporarily stop KEDA from scaling (during maintenance, debugging, etc.):

### Method 1: Annotations

```bash
# Pause scaling
kubectl annotate scaledobject my-scaledobject autoscaling.keda.sh/paused-replicas="3"

# This will:
# - Stop all trigger polling
# - Scale the deployment to exactly 3 replicas
# - Keep it there until you remove the annotation

# Resume scaling
kubectl annotate scaledobject my-scaledobject autoscaling.keda.sh/paused-replicas-
```

### Method 2: Patch the ScaledObject

```bash
# Pause
kubectl patch scaledobject my-scaledobject --type='json' \
  -p='[{"op": "add", "path": "/metadata/annotations/autoscaling.keda.sh~1paused-replicas", "value": "5"}]'
```

## Best Practices

### 1. Always Set Fallback

```yaml
spec:
  fallback:
    failureThreshold: 3
    replicas: 5
```

If KEDA can't reach your event source (e.g., RabbitMQ is down), you don't want to accidentally scale to zero. The fallback ensures a safe number of replicas.

### 2. Tune Your Polling Interval

- **Too low** (e.g., 5s): Puts unnecessary load on your event source.
- **Too high** (e.g., 120s): Slow reaction to traffic spikes.
- **Sweet spot:** 15–30 seconds for most use cases.

### 3. Use `cooldownPeriod` Wisely

- Set higher (300–600s) for bursty workloads to avoid thrashing (scaling up and down repeatedly).
- Set lower (60–120s) if you want aggressive scale-down to save costs.

### 4. Don't Scale HTTP Services to Zero (Unless Using HTTP Add-on)

For HTTP workloads, use `minReplicaCount: 1` or `minReplicaCount: 2`. Scaling to zero means dropped requests during cold start.

### 5. Use HPA Behavior for Smooth Scaling

```yaml
spec:
  advanced:
    horizontalPodAutoscalerConfig:
      behavior:
        scaleDown:
          stabilizationWindowSeconds: 300     # Wait 5 min before scaling down
          policies:
            - type: Pods
              value: 2                        # Scale down max 2 pods at a time
              periodSeconds: 60
        scaleUp:
          stabilizationWindowSeconds: 0       # Scale up immediately
          policies:
            - type: Pods
              value: 5                        # Scale up max 5 pods at a time
              periodSeconds: 60
```

### 6. Secure Your TriggerAuthentication

- Never hardcode credentials in ScaledObject metadata.
- Use `TriggerAuthentication` with Kubernetes Secrets.
- Better yet, use **Pod Identity** (IRSA, Workload Identity) — no secrets needed.

## Troubleshooting Common Issues

### Issue 1: ScaledObject Not Scaling

```bash
# Check ScaledObject status
kubectl describe scaledobject <name>

# Look for these conditions:
# Ready = True?
# Active = True?

# Check KEDA operator logs
kubectl logs -n keda deployment/keda-operator | grep -i error
```

**Common causes:**
- Wrong scaler configuration (check metadata fields)
- Authentication failure (check TriggerAuthentication)
- Target Deployment doesn't exist or is in a different namespace

### Issue 2: Not Scaling to Zero

- Check `cooldownPeriod` — it must expire first
- Verify `minReplicaCount: 0` is set
- Ensure the metric value is actually reaching 0 (check scaler logs)

### Issue 3: Slow Scale-Up from Zero

This is expected — it takes time to schedule a pod, pull the image, and start the container. To minimize:

```yaml
# Use imagePullPolicy: IfNotPresent
# Use small container images
# Pre-pull images using a DaemonSet
# Set pollingInterval low (e.g., 10 seconds)
```

### Issue 4: Thrashing (Rapid Scale Up/Down)

```yaml
# Increase cooldownPeriod
cooldownPeriod: 600

# Add HPA stabilization
advanced:
  horizontalPodAutoscalerConfig:
    behavior:
      scaleDown:
        stabilizationWindowSeconds: 300
```

## Summary

| Concept | What It Means |
|---------|--------------|
| **KEDA** | Kubernetes Event-Driven Autoscaler — scales based on external events |
| **ScaledObject** | Tells KEDA what Deployment to scale and what triggers to use |
| **ScaledJob** | Creates Kubernetes Jobs per event instead of scaling a Deployment |
| **Triggers/Scalers** | 60+ event sources (queues, databases, metrics, cron, cloud services) |
| **TriggerAuthentication** | Securely stores credentials for connecting to event sources |
| **Scale to Zero** | KEDA's superpower — 0 pods when idle, scales up on events |
| **pollingInterval** | How often KEDA checks event sources |
| **cooldownPeriod** | Wait time before scaling back to zero |
| **Fallback** | Safety replicas when KEDA can't reach the event source |

> **Key Takeaway:** KEDA is the go-to solution when you need Kubernetes workloads to react to real-world events — not just CPU and memory. It's lightweight, integrates with everything, and the ability to scale to zero makes it a cost-saving champion. 🏆
