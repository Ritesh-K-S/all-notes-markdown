## What is RBAC?

**RBAC** stands for **Role-Based Access Control**. It's Kubernetes' built-in system for controlling **who can do what** inside your cluster.

Think of it like this: In a company, not everyone should have access to everything. The intern shouldn't be able to delete production databases, and the marketing team doesn't need access to the server room. RBAC is exactly that — but for Kubernetes.

### RBAC in One Sentence

> RBAC lets you define **rules** (what actions are allowed on what resources) and **bind** those rules to **users, groups, or service accounts**.

---

## Why Do We Need RBAC?

Without RBAC, anyone with access to the cluster can do **anything** — create pods, delete deployments, read secrets, even delete the entire namespace. That's terrifying in production. 😱

RBAC helps you:

- **Limit blast radius:** If someone's credentials are compromised, they can only do what their role allows.
- **Enforce least privilege:** Give people only the permissions they actually need.
- **Audit access:** You know exactly who has access to what.
- **Multi-tenancy:** Different teams can share a cluster without stepping on each other.

## RBAC Building Blocks — The Big Picture

RBAC has **4 main resources**. Here's the simple mental model:

```
┌─────────────────────────────────────────────────────────┐
│                     RBAC Model                          │
│                                                          │
│   WHO?              WHAT CAN THEY DO?        WHERE?     │
│   ─────             ──────────────────        ─────     │
│   User              Role (namespace-scoped)    Namespace │
│   Group             ClusterRole (cluster-wide) Cluster   │
│   ServiceAccount                                         │
│                                                          │
│   Connected by:                                          │
│   RoleBinding         → binds Role to WHO in a namespace │
│   ClusterRoleBinding  → binds ClusterRole to WHO cluster-│
│                         wide                             │
└─────────────────────────────────────────────────────────┘
```

Let me break this down into plain English:

| Resource | What It Does | Scope |
|----------|-------------|-------|
| **Role** | Defines a set of permissions (e.g., "can read pods") | Single namespace |
| **ClusterRole** | Defines a set of permissions | Entire cluster |
| **RoleBinding** | Connects a Role to a user/group/service account | Single namespace |
| **ClusterRoleBinding** | Connects a ClusterRole to a user/group/service account | Entire cluster |

## Understanding the Pieces

### Subjects — The "WHO"

In RBAC, the **subject** is the entity you're granting permissions to. There are three types:

| Subject Type | What It Is | Example |
|-------------|-----------|---------|
| **User** | A human user (authenticated externally — Kubernetes has no user database) | `ritesh`, `admin@company.com` |
| **Group** | A group of users (also from external authentication) | `dev-team`, `system:masters` |
| **ServiceAccount** | An identity for pods/applications running inside the cluster | `default`, `my-app-sa` |

> **Important:** Kubernetes does **not** have a built-in user management system. Users are authenticated through external systems (certificates, OIDC providers like Google/Azure AD, etc.). But ServiceAccounts ARE managed by Kubernetes.

### Resources — The "WHAT"

Resources are the Kubernetes objects you want to control access to:

- `pods`, `deployments`, `services`, `secrets`, `configmaps`, `namespaces`, `nodes`, `persistentvolumes`, etc.
- Also includes **subresources** like `pods/log`, `pods/exec`, `deployments/scale`

### Verbs — The "ACTIONS"

Verbs define what actions are allowed:

| Verb | What It Does | HTTP Method |
|------|-------------|-------------|
| `get` | Read a single resource | GET (specific) |
| `list` | List all resources of a type | GET (collection) |
| `watch` | Watch for changes (streaming) | GET (with watch) |
| `create` | Create a new resource | POST |
| `update` | Modify an existing resource (full replace) | PUT |
| `patch` | Partially modify a resource | PATCH |
| `delete` | Delete a single resource | DELETE |
| `deletecollection` | Delete all resources of a type | DELETE (collection) |

### API Groups — Where Resources Live

Kubernetes resources belong to API groups:

| API Group | Resources | In YAML |
|-----------|----------|---------|
| Core (empty string `""`) | pods, services, secrets, configmaps, namespaces, nodes, persistentvolumes | `apiGroups: [""]` |
| `apps` | deployments, statefulsets, daemonsets, replicasets | `apiGroups: ["apps"]` |
| `batch` | jobs, cronjobs | `apiGroups: ["batch"]` |
| `networking.k8s.io` | ingresses, networkpolicies | `apiGroups: ["networking.k8s.io"]` |
| `rbac.authorization.k8s.io` | roles, clusterroles, rolebindings, clusterrolebindings | `apiGroups: ["rbac.authorization.k8s.io"]` |
| `storage.k8s.io` | storageclasses, persistentvolumeclaims | `apiGroups: ["storage.k8s.io"]` |

> 💡 **How to find the API group for any resource:**
> ```bash
> kubectl api-resources
> ```
> This shows every resource, its API group, and whether it's namespaced.

## Role — Namespace-Scoped Permissions

A **Role** defines permissions within a **single namespace**. It says "these actions are allowed on these resources, but only in this namespace."

### Example 1: Read-Only Access to Pods

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: development         # This Role only works in the "development" namespace
rules:
  - apiGroups: [""]              # Core API group (pods are in the core group)
    resources: ["pods"]          # Resource type
    verbs: ["get", "list", "watch"]   # Allowed actions (read-only)
```

**What this Role allows:**
- ✅ `kubectl get pods -n development`
- ✅ `kubectl describe pod my-pod -n development`
- ❌ `kubectl delete pod my-pod -n development` (no `delete` verb)
- ❌ `kubectl get pods -n production` (wrong namespace)

### Example 2: Full Access to Deployments and Services

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: app-manager
  namespace: staging
rules:
  - apiGroups: ["apps"]          # Deployments are in the "apps" API group
    resources: ["deployments"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
  
  - apiGroups: [""]              # Services are in the core API group
    resources: ["services"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
```

### Example 3: Access to Pods + Pod Logs + Pod Exec

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-debugger
  namespace: production
rules:
  - apiGroups: [""]
    resources: ["pods"]
    verbs: ["get", "list", "watch"]
  
  - apiGroups: [""]
    resources: ["pods/log"]        # Subresource: pod logs
    verbs: ["get"]
  
  - apiGroups: [""]
    resources: ["pods/exec"]       # Subresource: exec into pods
    verbs: ["create"]             # exec is a "create" action (you create an exec session)
```

### Example 4: Access to Specific Resources by Name

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: specific-configmap-reader
  namespace: production
rules:
  - apiGroups: [""]
    resources: ["configmaps"]
    resourceNames: ["app-config", "feature-flags"]   # ONLY these specific ConfigMaps
    verbs: ["get", "watch"]
```

> `resourceNames` restricts access to **specific named resources** rather than all resources of that type. Very useful for fine-grained control.

## ClusterRole — Cluster-Wide Permissions

A **ClusterRole** is like a Role, but it works **across the entire cluster** — all namespaces. It can also grant access to **cluster-scoped resources** (like nodes, namespaces, persistent volumes) that don't belong to any namespace.

### When to Use ClusterRole Instead of Role

| Use Case | Role or ClusterRole? |
|----------|---------------------|
| Access to pods in a specific namespace | Role |
| Access to pods in **all** namespaces | ClusterRole |
| Access to nodes (cluster-scoped resource) | ClusterRole |
| Access to namespaces (cluster-scoped) | ClusterRole |
| Access to persistent volumes (cluster-scoped) | ClusterRole |
| Reusable permission template across namespaces | ClusterRole + RoleBinding |

### Example 1: Read All Pods Across the Cluster

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: cluster-pod-reader       # No namespace field — ClusterRoles are cluster-scoped
rules:
  - apiGroups: [""]
    resources: ["pods"]
    verbs: ["get", "list", "watch"]
```

### Example 2: Node Administrator

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: node-admin
rules:
  - apiGroups: [""]
    resources: ["nodes"]          # Nodes are cluster-scoped (not in any namespace)
    verbs: ["get", "list", "watch", "update", "patch"]
  
  - apiGroups: [""]
    resources: ["nodes/status"]
    verbs: ["get", "update", "patch"]
```

### Example 3: Full Cluster Admin (Everything)

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: super-admin
rules:
  - apiGroups: ["*"]             # ALL API groups
    resources: ["*"]             # ALL resources
    verbs: ["*"]                 # ALL actions
```

> ⚠️ **Warning:** This is equivalent to `cluster-admin`. Only give this to people who absolutely need it. In production, this should be extremely rare.

### Aggregated ClusterRoles

Kubernetes lets you **combine** multiple ClusterRoles into one using **aggregation labels**. This is how the built-in roles (`admin`, `edit`, `view`) work.

```yaml
# Base ClusterRole with aggregation label
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: monitoring-reader
  labels:
    rbac.authorization.k8s.io/aggregate-to-view: "true"    # Automatically added to "view" ClusterRole
    rbac.authorization.k8s.io/aggregate-to-edit: "true"    # Automatically added to "edit" ClusterRole
rules:
  - apiGroups: ["monitoring.coreos.com"]
    resources: ["prometheusrules", "servicemonitors"]
    verbs: ["get", "list", "watch"]
```

When you add the label `rbac.authorization.k8s.io/aggregate-to-view: "true"`, Kubernetes **automatically merges** these rules into the built-in `view` ClusterRole. So anyone with the `view` role gets these permissions too!

## RoleBinding — Connecting Roles to Users (Namespace-Scoped)

A Role by itself does nothing. You need to **bind** it to a subject (user, group, or service account). A **RoleBinding** does this within a single namespace.

### Example 1: Bind a Role to a User

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-pods-binding
  namespace: development
subjects:
  - kind: User
    name: ritesh                    # The user to grant access to
    apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role                        # Referencing a Role (not ClusterRole)
  name: pod-reader                  # The Role we created earlier
  apiGroup: rbac.authorization.k8s.io
```

**What this does:** User `ritesh` can now read pods in the `development` namespace.

### Example 2: Bind a Role to a Group

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: dev-team-binding
  namespace: development
subjects:
  - kind: Group
    name: dev-team                   # All users in the "dev-team" group
    apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: app-manager
  apiGroup: rbac.authorization.k8s.io
```

**What this does:** Everyone in the `dev-team` group can manage deployments and services in the `development` namespace.

### Example 3: Bind a Role to a Service Account

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: sa-pod-reader-binding
  namespace: production
subjects:
  - kind: ServiceAccount
    name: my-app-sa                  # Service Account name
    namespace: production            # Namespace where the SA lives
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
```

### Example 4: Bind a ClusterRole to a User in a Specific Namespace

This is a **powerful pattern**: You can use a RoleBinding to bind a **ClusterRole** to a user, but limit it to a **single namespace**.

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: dev-edit-binding
  namespace: development             # Only in the "development" namespace
subjects:
  - kind: User
    name: ritesh
    apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole                  # Using a ClusterRole here!
  name: edit                         # Built-in "edit" ClusterRole
  apiGroup: rbac.authorization.k8s.io
```

**What this does:** User `ritesh` gets the `edit` ClusterRole permissions, but **only in the development namespace**. This is the most common pattern in real clusters — you define ClusterRoles once and reuse them across namespaces with RoleBindings.

### Multiple Subjects in One Binding

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: multi-subject-binding
  namespace: staging
subjects:
  - kind: User
    name: alice
    apiGroup: rbac.authorization.k8s.io
  - kind: User
    name: bob
    apiGroup: rbac.authorization.k8s.io
  - kind: Group
    name: qa-team
    apiGroup: rbac.authorization.k8s.io
  - kind: ServiceAccount
    name: ci-pipeline
    namespace: staging
roleRef:
  kind: ClusterRole
  name: edit
  apiGroup: rbac.authorization.k8s.io
```

## ClusterRoleBinding — Cluster-Wide Binding

A **ClusterRoleBinding** binds a ClusterRole to subjects **across the entire cluster**.

### Example 1: Make a User Cluster Admin

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: ritesh-cluster-admin
subjects:
  - kind: User
    name: ritesh
    apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: cluster-admin              # Built-in ClusterRole — full access to everything
  apiGroup: rbac.authorization.k8s.io
```

> ⚠️ **Be very careful with this.** `cluster-admin` has god-mode permissions. In production, limit this to a tiny number of people.

### Example 2: Give All Developers Read-Only Access Cluster-Wide

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: all-devs-view
subjects:
  - kind: Group
    name: developers
    apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: view                      # Built-in read-only ClusterRole
  apiGroup: rbac.authorization.k8s.io
```

## Built-in ClusterRoles

Kubernetes comes with several built-in ClusterRoles. These are the most important ones:

| ClusterRole | Permissions | Common Use |
|-------------|-----------|------------|
| `cluster-admin` | Everything. Full control over entire cluster | Cluster administrators only |
| `admin` | Full access within a namespace (when used with RoleBinding) | Namespace admins |
| `edit` | Read/write access to most resources (no RBAC, no resource quotas) | Developers |
| `view` | Read-only access to most resources (no secrets) | Read-only users, dashboards |
| `system:node` | Permissions for kubelets | Automatically bound to nodes |
| `system:kube-scheduler` | Permissions for the scheduler | Automatically bound to scheduler |

### View Built-in Roles

```bash
# List all ClusterRoles
kubectl get clusterroles

# See what permissions "edit" has
kubectl describe clusterrole edit

# See what permissions "view" has  
kubectl describe clusterrole view

# See all RoleBindings in a namespace
kubectl get rolebindings -n development

# See all ClusterRoleBindings
kubectl get clusterrolebindings
```

## Practical Scenario: Setting Up RBAC for a Team

Let's set up a real-world RBAC structure for a company with:

- **Platform team:** Full cluster access
- **Backend team:** Full access to `backend` namespace, read-only elsewhere
- **Frontend team:** Full access to `frontend` namespace, read-only elsewhere
- **QA team:** Read-only access to `staging` and `production` namespaces
- **CI/CD pipeline:** Deploy to `staging` and `production`

### Step 1: Create the Namespaces

```bash
kubectl create namespace backend
kubectl create namespace frontend
kubectl create namespace staging
kubectl create namespace production
```

### Step 2: Platform Team — Cluster Admin

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: platform-team-admin
subjects:
  - kind: Group
    name: platform-team
    apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: cluster-admin
  apiGroup: rbac.authorization.k8s.io
```

### Step 3: Backend Team

```yaml
# Full access to backend namespace
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: backend-team-admin
  namespace: backend
subjects:
  - kind: Group
    name: backend-team
    apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: admin
  apiGroup: rbac.authorization.k8s.io
---
# Read-only access cluster-wide
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: backend-team-view
subjects:
  - kind: Group
    name: backend-team
    apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: view
  apiGroup: rbac.authorization.k8s.io
```

### Step 4: Frontend Team

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: frontend-team-admin
  namespace: frontend
subjects:
  - kind: Group
    name: frontend-team
    apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: admin
  apiGroup: rbac.authorization.k8s.io
---
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: frontend-team-view
subjects:
  - kind: Group
    name: frontend-team
    apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: view
  apiGroup: rbac.authorization.k8s.io
```

### Step 5: QA Team — Read-Only

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: qa-team-staging-view
  namespace: staging
subjects:
  - kind: Group
    name: qa-team
    apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: view
  apiGroup: rbac.authorization.k8s.io
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: qa-team-prod-view
  namespace: production
subjects:
  - kind: Group
    name: qa-team
    apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: view
  apiGroup: rbac.authorization.k8s.io
```

### Step 6: CI/CD Pipeline Service Account

```yaml
# Create a Service Account for CI/CD
apiVersion: v1
kind: ServiceAccount
metadata:
  name: ci-pipeline
  namespace: staging
---
# Custom Role for deployment
apiVersion: rbac.authorization.k8s.io/v1

  name: deployer
  namespace: staging

    resources: ["deployments"]
    verbs: ["get", "list", "watch", "create", "update", "patch"]
  - apiGroups: [""]
    resources: ["services", "configmaps"]
    verbs: ["get", "list", "watch", "create", "update", "patch"]
  - apiGroups: [""]
    resources: ["pods"]
    verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: ci-pipeline-deployer
  namespace: staging
subjects:
  - kind: ServiceAccount
    name: ci-pipeline
    namespace: staging
roleRef:
  kind: Role
  name: deployer
  apiGroup: rbac.authorization.k8s.io
```

## Checking Permissions — `kubectl auth can-i`

This is one of the most useful RBAC commands. It tells you whether a user can perform an action.

### Check Your Own Permissions

```bash
# Can I create pods in the default namespace?
kubectl auth can-i create pods
# Output: yes

# Can I delete nodes?
kubectl auth can-i delete nodes
# Output: no

# Can I get secrets in the production namespace?
kubectl auth can-i get secrets -n production
# Output: no
```

### Check Another User's Permissions (Requires Admin)

```bash
# Can user "ritesh" list pods in the development namespace?
kubectl auth can-i list pods -n development --as ritesh
# Output: yes

# Can the "qa-team" group delete deployments in production?
kubectl auth can-i delete deployments -n production --as-group qa-team
# Output: no

# Can the service account "ci-pipeline" create deployments in staging?
kubectl auth can-i create deployments -n staging --as system:serviceaccount:staging:ci-pipeline
# Output: yes
```

### List ALL Permissions for a User

```bash
# List everything the current user can do
kubectl auth can-i --list

# List what user "ritesh" can do in the "development" namespace
kubectl auth can-i --list -n development --as ritesh
```

Output looks like:

```
Resources                                   Non-Resource URLs   Resource Names   Verbs
pods                                        []                  []               [get list watch]
pods/log                                    []                  []               [get]
deployments.apps                            []                  []               [get list watch create update patch delete]
services                                    []                  []               [get list watch create update patch]
```

## Debugging RBAC Issues

### Common Error: "Forbidden"

```
Error from server (Forbidden): pods is forbidden: User "ritesh" cannot list resource 
"pods" in API group "" in the namespace "production"
```

**How to debug:**

```bash
# Step 1: Check what the user CAN do
kubectl auth can-i --list -n production --as ritesh

# Step 2: Check all RoleBindings in the namespace
kubectl get rolebindings -n production -o wide

# Step 3: Check all ClusterRoleBindings
kubectl get clusterrolebindings -o wide | grep ritesh

# Step 4: Describe a specific binding to see its role and subjects
kubectl describe rolebinding <name> -n production

# Step 5: Check the Role's rules
kubectl describe role <role-name> -n production
# or
kubectl describe clusterrole <clusterrole-name>
```

### Useful Debugging Commands

```bash
# See ALL ClusterRoles
kubectl get clusterroles

# See ALL ClusterRoleBindings with subjects
kubectl get clusterrolebindings -o wide

# See ALL RoleBindings in ALL namespaces
kubectl get rolebindings -A -o wide

# Find which roles/bindings reference a specific user
kubectl get clusterrolebindings -o json | jq '.items[] | select(.subjects[]?.name == "ritesh") | .metadata.name'

# See the full RBAC rules for a ClusterRole
kubectl get clusterrole edit -o yaml
```

## Important RBAC Rules & Gotchas

### 1. RBAC is Additive (No "Deny" Rules)

RBAC only has **allow** rules. There are **no deny rules**. If you have two roles — one that allows "get pods" and one that allows "delete pods" — you can do both.

**Implication:** You can't say "allow everything except deleting secrets." You'd have to explicitly list everything else.

### 2. You Can't Edit Your Own Permissions (Privilege Escalation Prevention)

Kubernetes prevents you from creating a Role or RoleBinding that gives you more permissions than you already have. This prevents privilege escalation.

```bash
# If you can only "get pods", you can't create a role that allows "delete pods"
# Kubernetes will reject it with: "cannot grant permissions you do not have"
```

### 3. RoleBinding Can Reference a ClusterRole

This is a very common and powerful pattern:

```yaml
kind: RoleBinding          # Namespace-scoped binding
roleRef:
  kind: ClusterRole        # But referencing a ClusterRole!
  name: edit
```

This means: "Apply the `edit` ClusterRole, but only in this specific namespace." You define the ClusterRole once and reuse it everywhere.

### 4. ClusterRoleBinding + ClusterRole = Cluster-Wide Access

```yaml
kind: ClusterRoleBinding   # Cluster-wide binding
roleRef:
  kind: ClusterRole        # Cluster-wide role
  name: view
```

This gives `view` access to **every namespace**.

### 5. RoleBinding CANNOT Reference a Role from Another Namespace

A RoleBinding in `namespace-a` cannot reference a Role from `namespace-b`. Roles are strictly namespace-scoped.

## RBAC for Non-Resource URLs

Some Kubernetes API endpoints aren't resources — they're URLs like `/healthz`, `/api`, `/version`. You can control access to these too:

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: health-checker
rules:
  - nonResourceURLs: ["/healthz", "/healthz/*", "/version"]
    verbs: ["get"]
```

## Creating Roles with kubectl (Imperative)

You don't always have to write YAML. kubectl can create roles directly:

```bash
# Create a Role that allows reading pods and services
kubectl create role pod-service-reader \
  --verb=get,list,watch \
  --resource=pods,services \
  -n development

# Create a ClusterRole that allows reading nodes
kubectl create clusterrole node-reader \
  --verb=get,list,watch \
  --resource=nodes

# Create a RoleBinding
kubectl create rolebinding dev-pod-reader \
  --role=pod-service-reader \
  --user=ritesh \
  -n development

# Create a ClusterRoleBinding
kubectl create clusterrolebinding global-node-reader \
  --clusterrole=node-reader \
  --group=developers

# Bind a ClusterRole to a ServiceAccount in a namespace
kubectl create rolebinding ci-deployer \
  --clusterrole=edit \
  --serviceaccount=staging:ci-pipeline \
  -n staging
```

## Enabling RBAC

RBAC is **enabled by default** in all modern Kubernetes distributions (1.8+). You can verify:

```bash
# Check if the RBAC API is available
kubectl api-versions | grep rbac

# Output should include:
# rbac.authorization.k8s.io/v1
```

If you're using kubeadm, RBAC is enabled automatically. For older clusters, you'd need to pass `--authorization-mode=RBAC` to the API server.

## Best Practices

### 1. Follow the Principle of Least Privilege

Give users/services only the **minimum permissions** they need. Start restrictive and add permissions as needed.

```yaml
# ✅ Good: Only what's needed
rules:
  - apiGroups: ["apps"]
    resources: ["deployments"]
    verbs: ["get", "list"]

# ❌ Bad: Wildcard everything
rules:
  - apiGroups: ["*"]
    resources: ["*"]
    verbs: ["*"]
```

### 2. Use Groups Instead of Individual Users

Binding roles to groups is much easier to manage than binding to individual users.

```yaml
# ✅ Good: Bind to a group
subjects:
  - kind: Group
    name: backend-team

# ❌ Bad: Bind to individual users
subjects:
  - kind: User
    name: alice
  - kind: User
    name: bob
  - kind: User
    name: charlie
  # ... 50 more users 😩
```

### 3. Use Built-in ClusterRoles When Possible

Don't reinvent the wheel. The built-in `view`, `edit`, and `admin` ClusterRoles cover most use cases.

### 4. Use ClusterRole + RoleBinding Pattern

Define ClusterRoles once, then bind them per-namespace with RoleBindings. This avoids duplicating Role definitions.

### 5. Audit RBAC Regularly

```bash
# Who has cluster-admin?
kubectl get clusterrolebindings -o json | \
  jq '.items[] | select(.roleRef.name == "cluster-admin") | {name: .metadata.name, subjects: .subjects}'

# What can a specific service account do?
kubectl auth can-i --list --as system:serviceaccount:default:my-sa
```

### 6. Use Separate Service Accounts for Each Application

Don't use the `default` service account. Create dedicated ones for each app with only the permissions it needs. (More on this in chapter 10.33.)

## Summary

| Concept | What It Does |
|---------|-------------|
| **RBAC** | Controls who can do what in your Kubernetes cluster |
| **Role** | Defines permissions in a single namespace |
| **ClusterRole** | Defines permissions cluster-wide |
| **RoleBinding** | Binds a Role (or ClusterRole) to users/groups/SAs in a namespace |
| **ClusterRoleBinding** | Binds a ClusterRole to users/groups/SAs cluster-wide |
| **Subjects** | Users, Groups, or ServiceAccounts (the "who") |
| **Rules** | apiGroups + resources + verbs (the "what") |
| **`kubectl auth can-i`** | Check permissions for yourself or others |
| **Built-in roles** | `cluster-admin`, `admin`, `edit`, `view` |

> **Key Takeaway:** RBAC is not optional — it's essential. Every production cluster should have a well-thought-out RBAC strategy. Remember: **no deny rules** in RBAC, it's all additive. Start with the built-in roles, follow least privilege, and use groups over individual users. 🛡️
