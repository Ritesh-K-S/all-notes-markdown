# 10.33 — Service Accounts 🤖

> **Level:** Intermediate
> **Category:** Security

---

## What is a Service Account?

In Kubernetes, **Service Accounts** provide an **identity for processes running inside pods**. When your application (running in a pod) needs to talk to the Kubernetes API — like reading ConfigMaps, listing pods, or creating deployments — it needs an identity. That identity is a Service Account.

### Simple Analogy

Think of it like a badge system in an office:

- **Users** (humans) have their own employee badges to get into the building.
- **Robots/Machines** (automated systems) also need badges — but they're different from human badges. They're machine badges.

In Kubernetes:
- **Users** = human operators who use `kubectl` (authenticated externally)
- **Service Accounts** = identities for **applications/pods** (managed directly by Kubernetes)

### Service Accounts in One Sentence

> A Service Account is a **Kubernetes-managed identity** that pods use to authenticate with the Kubernetes API server.

---

## Why Do We Need Service Accounts?

Not every pod needs to talk to the Kubernetes API, but many do:

| Use Case | Why It Needs API Access |
|----------|----------------------|
| **CI/CD pipeline** | Creates/updates Deployments during deployment |
| **Monitoring agent** (Prometheus) | Lists pods and services for auto-discovery |
| **Log collector** (Fluentd/Fluent Bit) | Reads pod metadata to enrich logs |
| **Custom operator** | Watches CRDs and manages resources |
| **Ingress controller** | Reads Ingress resources and configures routing |
| **Autoscaler (KEDA/HPA)** | Reads metrics and scales deployments |
| **Vault agent** | Reads secrets from Kubernetes API |

Without Service Accounts, there's no way for pods to securely authenticate with the API server.

---

## The Default Service Account

Every namespace in Kubernetes automatically gets a Service Account named `default`. When you create a pod without specifying a Service Account, it automatically uses the `default` one.

```bash
# See the default service account
kubectl get serviceaccounts -n default

# Output:
# NAME      SECRETS   AGE
# default   0         30d

# Every namespace has one
kubectl get serviceaccounts -A | grep default
```

### The Problem with the Default Service Account

The `default` service account has **minimal permissions** (basically none, unless someone added them). But here's the issue:

1. **Every pod uses it** if you don't specify otherwise — so if someone accidentally grants permissions to the `default` SA, ALL pods in that namespace get those permissions.
2. **No isolation** — all pods share the same identity.
3. **Hard to audit** — you can't tell which app is making API calls.

> **Best Practice:** Never use the `default` service account for your applications. Always create a **dedicated** service account for each application.

## Creating Service Accounts

### Method 1: Using YAML (Declarative)

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: my-app-sa               # Name of the Service Account
  namespace: production          # Namespace it belongs to
  labels:
    app: my-app                  # Optional labels for organization
  annotations:
    description: "Service account for my-app"   # Optional annotations
```

Apply it:

```bash
kubectl apply -f my-app-sa.yaml
```

### Method 2: Using kubectl (Imperative)

```bash
# Create a service account
kubectl create serviceaccount my-app-sa -n production

# Verify
kubectl get serviceaccount my-app-sa -n production
```

## Assigning a Service Account to a Pod

To use a Service Account, you specify it in the pod spec using `serviceAccountName`:

### In a Pod

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app-pod
  namespace: production
spec:
  serviceAccountName: my-app-sa       # Use our custom Service Account
  containers:
    - name: my-app
      image: my-app:latest
```

### In a Deployment

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
  namespace: production
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      serviceAccountName: my-app-sa     # Applied to all pods in this deployment
      containers:
        - name: my-app
          image: my-app:latest
```

> If you don't specify `serviceAccountName`, Kubernetes uses the `default` service account of the namespace.

## How Service Account Tokens Work

When a pod uses a Service Account, Kubernetes **automatically mounts a token** inside the pod. This token is what the pod uses to authenticate with the API server.

### Kubernetes 1.24+ (Modern — Projected Tokens)

Starting from Kubernetes 1.24, tokens are **projected volumes** with these properties:

- **Time-bound:** Tokens expire automatically (default: 1 hour, auto-refreshed by kubelet)
- **Audience-bound:** Tokens are valid only for a specific audience (the API server)
- **Object-bound:** Tokens are tied to the specific pod

```yaml
# Kubernetes automatically mounts this for every pod:
spec:
  containers:
    - name: my-app
      volumeMounts:
        - mountPath: /var/run/secrets/kubernetes.io/serviceaccount
          name: kube-api-access-xxxxx
          readOnly: true
  volumes:
    - name: kube-api-access-xxxxx
      projected:
        sources:
          - serviceAccountToken:
              expirationSeconds: 3607
              path: token
          - configMap:
              name: kube-root-ca.crt
              items:
                - key: ca.crt
                  path: ca.crt
          - downwardAPI:
              items:
                - fieldRef:
                    fieldPath: metadata.namespace
                  path: namespace
```

### What Gets Mounted Inside the Pod

Inside every pod at `/var/run/secrets/kubernetes.io/serviceaccount/`, you'll find:

```bash
# Exec into a pod and check
kubectl exec -it my-app-pod -- ls /var/run/secrets/kubernetes.io/serviceaccount/

# Output:
# ca.crt      → The cluster's CA certificate (to verify API server's identity)
# namespace   → The namespace the pod is running in (plain text)
# token       → The JWT token for authenticating with the API server
```

### Reading the Token

```bash
# See the token
kubectl exec -it my-app-pod -- cat /var/run/secrets/kubernetes.io/serviceaccount/token

# It's a JWT token — you can decode it:
# Header: {"alg":"RS256","kid":"..."}
# Payload: {
#   "aud": ["https://kubernetes.default.svc.cluster.local"],
#   "exp": 1735689600,
#   "iat": 1735686000,
#   "iss": "https://kubernetes.default.svc.cluster.local",
#   "kubernetes.io": {
#     "namespace": "production",
#     "pod": {"name": "my-app-pod", "uid": "..."},
#     "serviceaccount": {"name": "my-app-sa", "uid": "..."}
#   },
#   "sub": "system:serviceaccount:production:my-app-sa"
# }
```

### Using the Token from Inside a Pod

When your application inside a pod wants to call the Kubernetes API:

```bash
# From inside the pod, you can call the API server like this:

# The API server address
APISERVER=https://kubernetes.default.svc

# The token
TOKEN=$(cat /var/run/secrets/kubernetes.io/serviceaccount/token)

# The CA cert
CACERT=/var/run/secrets/kubernetes.io/serviceaccount/ca.crt

# Make an API call - list pods in the current namespace
curl --cacert $CACERT \
     --header "Authorization: Bearer $TOKEN" \
     $APISERVER/api/v1/namespaces/production/pods
```

> Most Kubernetes client libraries (Python `kubernetes`, Go `client-go`, etc.) automatically detect these mounted files and use them. You don't have to read the token manually in your code.

## Disabling Token Auto-Mounting

Sometimes your pod does NOT need to talk to the Kubernetes API at all. In that case, you should **disable** the automatic token mounting for security (principle of least privilege):

### Disable at Pod Level

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: simple-web-app
spec:
  serviceAccountName: my-app-sa
  automountServiceAccountToken: false    # Don't mount the token!
  containers:
    - name: web
      image: nginx:latest
```

### Disable at Service Account Level

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: no-api-access-sa
  namespace: production
automountServiceAccountToken: false       # All pods using this SA won't get tokens
```

> **Priority:** Pod-level setting overrides Service Account-level setting. If the SA says `false` but the pod says `true`, the token **will** be mounted.

| SA Setting | Pod Setting | Result |
|-----------|------------|--------|
| `true` (default) | Not set | Token mounted |
| `true` | `false` | Token NOT mounted |
| `false` | Not set | Token NOT mounted |
| `false` | `true` | Token mounted |

## Service Accounts + RBAC — Putting It Together

A Service Account alone doesn't give any permissions. You need to pair it with **RBAC** (covered in 10.32) to grant specific permissions.

### Complete Example: App That Reads ConfigMaps

**Step 1: Create the Service Account**

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: config-reader-sa
  namespace: production
```

**Step 2: Create a Role with the Needed Permissions**

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: configmap-reader
  namespace: production
rules:
  - apiGroups: [""]
    resources: ["configmaps"]
    verbs: ["get", "list", "watch"]
```

**Step 3: Bind the Role to the Service Account**

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: config-reader-binding
  namespace: production
subjects:
  - kind: ServiceAccount
    name: config-reader-sa
    namespace: production
roleRef:
  kind: Role
  name: configmap-reader
  apiGroup: rbac.authorization.k8s.io
```

**Step 4: Use the Service Account in Your Deployment**

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
  namespace: production
spec:
  replicas: 2
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      serviceAccountName: config-reader-sa     # This pod can now read ConfigMaps!
      containers:
        - name: my-app
          image: my-app:latest
```

**Result:** Pods in this deployment can:
- ✅ Read ConfigMaps in the `production` namespace
- ❌ Create or delete ConfigMaps
- ❌ Read Secrets
- ❌ Read ConfigMaps in other namespaces

### Complete Example: Monitoring Agent That Reads Pods Cluster-Wide

```yaml
# Service Account
apiVersion: v1
kind: ServiceAccount
metadata:
  name: monitoring-sa
  namespace: monitoring
---
# ClusterRole (cluster-wide permissions)
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: pod-reader
rules:
  - apiGroups: [""]
    resources: ["pods", "nodes", "services", "endpoints"]
    verbs: ["get", "list", "watch"]
  - apiGroups: ["apps"]
    resources: ["deployments", "statefulsets", "daemonsets"]
    verbs: ["get", "list", "watch"]
---
# ClusterRoleBinding (bind cluster-wide)
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: monitoring-pod-reader
subjects:
  - kind: ServiceAccount
    name: monitoring-sa
    namespace: monitoring
roleRef:
  kind: ClusterRole
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
---
# Deployment using the Service Account
apiVersion: apps/v1
kind: Deployment
metadata:
  name: monitoring-agent
  namespace: monitoring
spec:
  replicas: 1
  selector:
    matchLabels:
      app: monitoring-agent
  template:
    metadata:
      labels:
        app: monitoring-agent
    spec:
      serviceAccountName: monitoring-sa
      containers:
        - name: agent
          image: monitoring-agent:latest
```

## Long-Lived Tokens (Kubernetes 1.24+)

Before Kubernetes 1.24, when you created a Service Account, Kubernetes automatically created a permanent (non-expiring) Secret with the token. **This is no longer the case.**

Since 1.24:
- Tokens are **projected** and **auto-expire** (default 1 hour, auto-refreshed)
- No more auto-created token Secrets

But sometimes you **need** a long-lived token (for external systems, CI/CD tools, etc.). Here's how:

### Method 1: Create a Manual Token Secret

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: my-app-sa-token
  namespace: production
  annotations:
    kubernetes.io/service-account.name: my-app-sa   # Link to the Service Account
type: kubernetes.io/service-account-token            # This type triggers token generation
```

```bash
# Apply it
kubectl apply -f token-secret.yaml

# Get the token
kubectl get secret my-app-sa-token -n production -o jsonpath='{.data.token}' | base64 -d
```

> ⚠️ **Warning:** This token **never expires**. Use it only when absolutely necessary (e.g., external systems that can't handle token rotation). For pods, always use the default projected tokens.

### Method 2: Using `kubectl create token` (Short-Lived)

```bash
# Create a token that expires in 1 hour (3600 seconds)
kubectl create token my-app-sa -n production --duration=3600s

# Create a token that expires in 24 hours
kubectl create token my-app-sa -n production --duration=86400s

# Create a token with a specific audience
kubectl create token my-app-sa -n production --audience=https://my-external-service.com
```

## Token Request API (TokenRequest)

For programmatic token creation, Kubernetes has the **TokenRequest API**. This is what kubelet uses internally to get tokens for pods.

```yaml
apiVersion: authentication.k8s.io/v1
kind: TokenRequest
metadata:
  name: my-app-sa
  namespace: production
spec:
  audiences:
    - https://kubernetes.default.svc.cluster.local
  expirationSeconds: 3600
  boundObjectRef:                   # Optional: bind token to a specific object
    apiVersion: v1
    kind: Pod
    name: my-app-pod
    uid: "pod-uid-here"
```

## Service Accounts for Cloud Provider Integration

Service Accounts are the bridge between Kubernetes and cloud services. Each cloud provider has a way to link Kubernetes SAs to cloud IAM roles.

### AWS — IAM Roles for Service Accounts (IRSA)

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: s3-reader-sa
  namespace: production
  annotations:
    # This annotation links the Kubernetes SA to an AWS IAM Role
    eks.amazonaws.com/role-arn: arn:aws:iam::123456789:role/s3-reader-role
```

Now any pod using `s3-reader-sa` automatically gets AWS credentials for the `s3-reader-role` IAM role. No need to hardcode AWS access keys!

### Azure — Workload Identity

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: blob-reader-sa
  namespace: production
  annotations:
    azure.workload.identity/client-id: "your-azure-ad-app-client-id"
  labels:
    azure.workload.identity/use: "true"
```

### GCP — Workload Identity

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: gcs-reader-sa
  namespace: production
  annotations:
    iam.gke.io/gcp-service-account: gcs-reader@my-project.iam.gserviceaccount.com
```

## Image Pull Secrets with Service Accounts

If your container images are stored in a **private registry**, you can attach image pull secrets to a Service Account. All pods using that SA will automatically use those credentials to pull images.

### Step 1: Create the Image Pull Secret

```bash
kubectl create secret docker-registry my-registry-cred \
  --docker-server=registry.example.com \
  --docker-username=myuser \
  --docker-password=mypassword \
  --docker-email=me@example.com \
  -n production
```

### Step 2: Add It to the Service Account

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: my-app-sa
  namespace: production
imagePullSecrets:
  - name: my-registry-cred          # Automatically used by all pods with this SA
```

Or patch an existing SA:

```bash
kubectl patch serviceaccount my-app-sa -n production \
  -p '{"imagePullSecrets": [{"name": "my-registry-cred"}]}'
```

Now you don't need to specify `imagePullSecrets` in every pod spec — it's inherited from the Service Account.

## Verifying Service Account Permissions

```bash
# Check what a service account can do
kubectl auth can-i --list --as=system:serviceaccount:production:my-app-sa

# Can this SA read pods?
kubectl auth can-i get pods -n production --as=system:serviceaccount:production:my-app-sa
# Output: yes

# Can this SA delete deployments?
kubectl auth can-i delete deployments -n production --as=system:serviceaccount:production:my-app-sa
# Output: no

# Can this SA read secrets?
kubectl auth can-i get secrets -n production --as=system:serviceaccount:production:my-app-sa
# Output: no
```

> The `--as` flag uses the format: `system:serviceaccount:<namespace>:<sa-name>`

## Service Accounts in Different Contexts

### CI/CD Pipelines

```yaml
# Create a dedicated SA for your CI/CD pipeline
apiVersion: v1
kind: ServiceAccount
metadata:
  name: github-actions-deployer
  namespace: production
---
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: deployer-role
  namespace: production
rules:
  - apiGroups: ["apps"]
    resources: ["deployments"]
    verbs: ["get", "list", "update", "patch"]
  - apiGroups: [""]
    resources: ["pods"]
    verbs: ["get", "list"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: github-actions-deployer-binding
  namespace: production
subjects:
  - kind: ServiceAccount
    name: github-actions-deployer
    namespace: production
roleRef:
  kind: Role
  name: deployer-role
  apiGroup: rbac.authorization.k8s.io
```

Then generate a token for GitHub Actions:

```bash
# Create a long-lived token for CI/CD
kubectl create token github-actions-deployer -n production --duration=8760h
# Store this token as a GitHub Actions secret
```

### Kubernetes Operators

Operators need broad permissions to manage their custom resources:

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: my-operator-sa
  namespace: operator-system
---
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: my-operator-role
rules:
  # Watch for custom resources
  - apiGroups: ["myapp.example.com"]
    resources: ["myapps", "myapps/status"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
  # Manage workloads
  - apiGroups: ["apps"]
    resources: ["deployments"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
  # Manage services
  - apiGroups: [""]
    resources: ["services", "configmaps"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
  # Events
  - apiGroups: [""]
    resources: ["events"]
    verbs: ["create", "patch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: my-operator-binding
subjects:
  - kind: ServiceAccount
    name: my-operator-sa
    namespace: operator-system
roleRef:
  kind: ClusterRole
  name: my-operator-role
  apiGroup: rbac.authorization.k8s.io
```

## Managing Service Accounts

### List All Service Accounts

```bash
# In a specific namespace
kubectl get serviceaccounts -n production

# In all namespaces
kubectl get serviceaccounts -A

# With more details
kubectl get serviceaccounts -n production -o yaml
```

### Describe a Service Account

```bash
kubectl describe serviceaccount my-app-sa -n production

# Output:
# Name:                my-app-sa
# Namespace:           production
# Labels:              app=my-app
# Annotations:         <none>
# Image pull secrets:  my-registry-cred
# Mountable secrets:   <none>
# Tokens:              <none>
```

### Delete a Service Account

```bash
kubectl delete serviceaccount my-app-sa -n production
```

> When you delete a SA, any RoleBindings or ClusterRoleBindings referencing it still exist but become orphaned. Clean them up too.

## Best Practices

### 1. One Service Account per Application

```yaml
# ✅ Good
# my-web-app uses web-app-sa
# my-worker uses worker-sa
# my-cronjob uses cronjob-sa

# ❌ Bad
# Everything uses "default"
```

### 2. Disable Token Mounting When Not Needed

```yaml
# ✅ If your app doesn't talk to the Kubernetes API:
spec:
  serviceAccountName: my-app-sa
  automountServiceAccountToken: false
```

### 3. Use Short-Lived Tokens

Prefer the auto-projected tokens (they expire and rotate). Only use long-lived token Secrets when absolutely necessary for external systems.

### 4. Use Cloud Provider Workload Identity

Instead of storing cloud credentials as Kubernetes Secrets:

```yaml
# ✅ Good: Use IRSA / Workload Identity
annotations:
  eks.amazonaws.com/role-arn: arn:aws:iam::123456789:role/my-role

# ❌ Bad: Store AWS keys in a Secret
# env:
#   - name: AWS_ACCESS_KEY_ID
#     valueFrom:
#       secretKeyRef: ...
```

### 5. Follow Least Privilege for RBAC

```yaml
# ✅ Good: Only what the app needs
rules:
  - apiGroups: [""]
    resources: ["configmaps"]
    verbs: ["get", "watch"]

# ❌ Bad: Give everything just in case
rules:
  - apiGroups: ["*"]
    resources: ["*"]
    verbs: ["*"]
```

### 6. Attach Image Pull Secrets to Service Accounts

Instead of adding `imagePullSecrets` to every pod spec, attach them to the Service Account:

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: my-app-sa
imagePullSecrets:
  - name: my-registry-cred
```

### 7. Name Service Accounts Descriptively

```bash
# ✅ Good names
prometheus-server-sa
github-actions-deployer
order-service-sa

# ❌ Bad names
sa1
test-sa
my-sa
```

## Troubleshooting Service Account Issues

### Issue 1: Pod Can't Authenticate with API Server

**Symptom:** Your app logs show `401 Unauthorized` or `Forbidden` when calling the Kubernetes API.

**Debug steps:**

```bash
# 1. Check which SA the pod is using
kubectl get pod my-pod -o jsonpath='{.spec.serviceAccountName}'

# 2. Check if the SA exists
kubectl get serviceaccount my-app-sa -n production

# 3. Check if the token is mounted
kubectl exec my-pod -- ls /var/run/secrets/kubernetes.io/serviceaccount/
# Should show: ca.crt  namespace  token

# 4. Check if automountServiceAccountToken is false
kubectl get pod my-pod -o jsonpath='{.spec.automountServiceAccountToken}'

# 5. Check RBAC permissions
kubectl auth can-i --list --as=system:serviceaccount:production:my-app-sa -n production
```

### Issue 2: Image Pull Fails with Authentication Error

**Symptom:** `ErrImagePull` or `ImagePullBackOff` for private registry images.

```bash
# Check if imagePullSecrets are attached to the SA
kubectl get serviceaccount my-app-sa -n production -o yaml | grep -A 5 imagePullSecrets

# Check if the secret exists and has the right data
kubectl get secret my-registry-cred -n production -o yaml

# Verify the credentials manually
kubectl get secret my-registry-cred -n production -o jsonpath='{.data.\.dockerconfigjson}' | base64 -d
```

### Issue 3: Token Expired (External Systems)

If you're using a long-lived token for an external system and it stops working:

```bash
# Check if the secret still exists
kubectl get secret my-sa-token -n production

# Check if the SA still exists (deleting the SA invalidates all tokens)
kubectl get serviceaccount my-app-sa -n production

# Generate a new token
kubectl create token my-app-sa -n production --duration=8760h
```

## Summary

| Concept | What It Means |
|---------|--------------|
| **Service Account** | Kubernetes-managed identity for pods/applications |
| **default SA** | Auto-created in every namespace; don't use it for apps |
| **serviceAccountName** | Pod spec field to assign an SA to a pod |
| **Projected Token** | Auto-mounted, short-lived, auto-rotated JWT token (K8s 1.24+) |
| **automountServiceAccountToken** | Control whether the token is mounted in the pod |
| **Token Secret** | Manually created long-lived token (use sparingly) |
| **kubectl create token** | CLI command to generate a temporary token |
| **imagePullSecrets** | Attach registry credentials to an SA for private images |
| **Cloud Workload Identity** | Link K8s SA to cloud IAM roles (IRSA, Azure WI, GCP WI) |
| **system:serviceaccount:ns:name** | The full identity string format for RBAC |

> **Key Takeaway:** Service Accounts are the identity layer for your applications in Kubernetes. Every app should have its own Service Account with only the permissions it needs. Combine Service Accounts with RBAC for proper access control, and use cloud workload identity to eliminate hardcoded credentials. It's not glamorous, but getting this right is what separates secure clusters from breached ones. 🔒
