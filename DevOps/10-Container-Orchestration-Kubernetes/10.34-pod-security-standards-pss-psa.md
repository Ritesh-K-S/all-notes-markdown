## 1. The Big Picture — Why Do We Need Pod Security?

Imagine you have a Kubernetes cluster where many developers deploy their applications. Now, what if someone (accidentally or intentionally) creates a Pod that:

- Runs as **root** user? 🚨
- Mounts the **host's filesystem**?
- Runs in **privileged mode** (basically has full access to the node)?
- Can escalate its own privileges?

That's a **massive security risk**. One compromised Pod could take over the entire node — or even the entire cluster!

**Pod Security Standards** and **Pod Security Admission** are Kubernetes' built-in way to say:

> "Hey, these are the rules. Your Pods must follow them, or they won't be allowed to run."

Think of it like a **bouncer at a club** — if your Pod doesn't meet the dress code (security standards), it doesn't get in.

---

## 2. A Bit of History — From PodSecurityPolicy to PSA

Here's the short version of the story:

| Timeline | What Happened |
|----------|---------------|
| **Kubernetes ≤ 1.21** | We had `PodSecurityPolicy` (PSP) — it worked but was **complicated, confusing**, and hard to use correctly |
| **Kubernetes 1.22** | PSP was **deprecated** |
| **Kubernetes 1.23** | Pod Security Admission (PSA) was introduced as **beta** |
| **Kubernetes 1.25** | PSP was **removed entirely**. PSA became **stable (GA)** |

**Why was PSP replaced?**
- PSP was hard to understand — even experienced engineers struggled with it
- It was confusing which policy applied to which Pod
- There was no "dry-run" or "warning" mode
- The RBAC integration was messy

PSA is the **simpler, better replacement**. It's built right into Kubernetes — no extra installation needed!

## 3. What Are Pod Security Standards (PSS)?

**Pod Security Standards (PSS)** define **three levels of security** for Pods. Think of them as three "tiers" of rules — from most permissive to most restrictive:

```
┌─────────────────────────────────────────────────┐
│              Pod Security Standards             │
├────────────┬────────────────┬───────────────────┤
│ Privileged │   Baseline     │    Restricted     │
│ (No rules) │ (Basic rules)  │ (Strict rules)    │
│            │                │                   │
│ "Do        │ "Follow        │ "Follow all       │
│  whatever  │  the basic     │  the hardening    │
│  you want" │  safety rules" │  best practices"  │
└────────────┴────────────────┴───────────────────┘
     ↑              ↑                  ↑
  Most           Good for           Most
  Permissive     most apps          Secure
```

### Quick Summary:

| Level | What It Means | Use Case |
|-------|---------------|----------|
| **Privileged** | No restrictions at all | System-level Pods, CNI plugins, storage drivers, things that NEED full access |
| **Baseline** | Prevents the most dangerous things (like privileged mode, hostNetwork) | Most regular applications |
| **Restricted** | Enforces all hardening best practices (non-root, read-only root filesystem, drop capabilities, etc.) | Security-critical workloads, multi-tenant clusters |

---

## 4. The Three Security Levels in Detail

### 4.1 — Privileged Level

This is basically **"no restrictions"**. The Pod can do anything.

**When to use:** Only for infrastructure-level components that genuinely need full access to the node. Examples:
- CNI plugins (Calico, Cilium)
- CSI drivers
- Log collectors that mount host paths
- kube-proxy

**What's allowed:** Everything — privileged containers, hostNetwork, hostPID, hostIPC, any user, any volume type, etc.

---

### 4.2 — Baseline Level

This level blocks the **most dangerous configurations** while still being flexible enough for most applications.

**What is NOT allowed at Baseline level:**

| Field | Restriction |
|-------|-------------|
| `hostNetwork` | Must be `false` (or not set) |
| `hostPID` | Must be `false` (or not set) |
| `hostIPC` | Must be `false` (or not set) |
| `privileged` | Must be `false` (or not set) |
| `hostPorts` | Cannot use host ports (or limited range) |
| Capabilities | Cannot add dangerous capabilities like `NET_RAW`, `SYS_ADMIN` |
| Volume types | Cannot use `hostPath` volumes |
| `/proc` mount | Cannot change the default procMount |
| Seccomp profile | Cannot set to `Unconfined` |
| Sysctls | Only safe sysctls allowed |
| `appArmorProfile` | Cannot opt out of AppArmor |
| SELinux | Cannot set custom SELinux user/role |
| `windowsOptions` | Restricted host process |

**When to use:** Most standard applications — web servers, APIs, workers, etc.

---

### 4.3 — Restricted Level

This is the **most secure level**. It includes everything from Baseline PLUS additional hardening:

| Field | Restriction |
|-------|-------------|
| Volume types | Only allowed: `configMap`, `csi`, `downwardAPI`, `emptyDir`, `ephemeral`, `persistentVolumeClaim`, `projected`, `secret` |
| Privilege escalation | `allowPrivilegeEscalation` must be `false` |
| Running as root | Must run as non-root (`runAsNonRoot: true`) |
| User ID | Must specify `runAsUser` with non-zero value (Kubernetes 1.23+: allowed to use `runAsNonRoot` instead) |
| Seccomp profile | Must be set to `RuntimeDefault` or `Localhost` |
| Capabilities | Must drop `ALL`, can only add `NET_BIND_SERVICE` |

**When to use:** Production workloads in security-sensitive environments, multi-tenant clusters, regulated industries (finance, healthcare).

---

### Visual Comparison

```
Privileged:     ✅ hostNetwork  ✅ privileged  ✅ root   ✅ hostPath  ✅ any capability
Baseline:       ❌ hostNetwork  ❌ privileged  ✅ root   ❌ hostPath  ❌ dangerous caps
Restricted:     ❌ hostNetwork  ❌ privileged  ❌ root   ❌ hostPath  ❌ ALL caps (except NET_BIND_SERVICE)
```

## 5. What Is Pod Security Admission (PSA)?

**Pod Security Admission (PSA)** is the **enforcement mechanism** — it's the thing that actually **checks your Pods** against the Pod Security Standards and decides whether to allow them or not.

Think of it this way:
- **PSS** = the **rules** (what's allowed and what's not)
- **PSA** = the **enforcer** (the bouncer who checks against those rules)

PSA is built into Kubernetes as an **admission controller** — it intercepts every Pod creation/update request and evaluates it.

```
Developer creates Pod
        │
        ▼
   API Server receives request
        │
        ▼
   ┌─────────────────────────┐
   │  Pod Security Admission │  ← "Does this Pod match the
   │  (PSA) Controller       │     security standard applied
   │                         │     to this namespace?"
   └──────────┬──────────────┘
              │
     ┌────────┴────────┐
     │                 │
  ✅ PASS           ❌ FAIL
  Pod is created    Pod is rejected
                    (or warning/audit)
```

## 6. PSA Modes — Enforce, Audit, Warn

PSA doesn't just have one way to react. It has **three modes** that you can mix and match:

| Mode | What Happens | Effect |
|------|-------------|--------|
| **enforce** | Pod is **rejected** if it violates the standard | Pod is NOT created |
| **audit** | Violation is **logged** in the audit log, but Pod is still created | Pod IS created, violation is recorded |
| **warn** | A **warning** is shown to the user (in kubectl output), but Pod is still created | Pod IS created, user sees a warning |

### Why Three Modes?

This is **genius** because it lets you **gradually roll out** security policies:

1. **Start with `warn`** → Developers see warnings but nothing breaks
2. **Add `audit`** → You can track violations in your logs
3. **Finally switch to `enforce`** → Violations are blocked

This way, you don't suddenly break everyone's deployments!

### You Can Mix Modes and Levels!

Here's the cool part — you can set **different levels for different modes**:

```yaml
# Example: Enforce baseline, but warn about restricted violations
pod-security.kubernetes.io/enforce: baseline
pod-security.kubernetes.io/warn: restricted
pod-security.kubernetes.io/audit: restricted
```

This means:
- Pods that violate **baseline** → **BLOCKED** (enforce)
- Pods that violate **restricted** → **WARNING shown** (warn) + **audit log entry** (audit)
- So developers see what they need to fix to reach "restricted" level, but their basic apps still work

## 7. Enabling Pod Security Admission

### Good News — It's Already Enabled!

Starting with **Kubernetes 1.25**, PSA is enabled by default. You don't need to install anything!

For older versions (1.23-1.24), it was a beta feature that needed to be enabled via the `PodSecurity` feature gate:

```yaml
# In the API server configuration (kube-apiserver)
# (You usually don't need this for K8s 1.25+)
--feature-gates=PodSecurity=true
```

And the admission plugin needs to be in the list:

```yaml
--enable-admission-plugins=...,PodSecurity,...
```

### Verifying PSA Is Active

```bash
# Check the API server feature gates (on kubeadm clusters)
kubectl -n kube-system get pod kube-apiserver-<node> -o yaml | grep -i podsecurity

# Or simply try applying a label to a namespace (if it works, PSA is active)
kubectl label namespace default pod-security.kubernetes.io/warn=baseline
```

## 8. Applying PSS via Namespace Labels

Here's the beautiful simplicity of PSA — you apply security standards **using namespace labels**. That's it. No custom resources, no complicated YAML files.

### Label Format

```
pod-security.kubernetes.io/<MODE>: <LEVEL>
pod-security.kubernetes.io/<MODE>-version: <VERSION>
```

Where:
- `<MODE>` = `enforce`, `audit`, or `warn`
- `<LEVEL>` = `privileged`, `baseline`, or `restricted`
- `<VERSION>` = Kubernetes version (e.g., `v1.28`) or `latest`

### Example — Apply Baseline Enforcement to a Namespace

```bash
# Create a namespace
kubectl create namespace my-app

# Apply baseline enforcement
kubectl label namespace my-app \
  pod-security.kubernetes.io/enforce=baseline \
  pod-security.kubernetes.io/enforce-version=latest \
  pod-security.kubernetes.io/warn=restricted \
  pod-security.kubernetes.io/warn-version=latest \
  pod-security.kubernetes.io/audit=restricted \
  pod-security.kubernetes.io/audit-version=latest
```

What this does:
- **Blocks** any Pod that violates **baseline** rules
- **Warns** about Pods that violate **restricted** rules (but still allows them)
- **Audits** Pods that violate **restricted** rules

### Using YAML

```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: my-secure-app
  labels:
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/enforce-version: latest
    pod-security.kubernetes.io/warn: restricted
    pod-security.kubernetes.io/warn-version: latest
    pod-security.kubernetes.io/audit: restricted
    pod-security.kubernetes.io/audit-version: latest
```

```bash
kubectl apply -f namespace.yaml
```

## 9. Hands-On Examples

Let's get our hands dirty with some real examples! 🛠️

### Example 1: Create a Restricted Namespace and Try to Deploy a "Bad" Pod

**Step 1: Create the namespace with restricted enforcement**

```yaml
# restricted-namespace.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: restricted-ns
  labels:
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/enforce-version: latest
```

```bash
kubectl apply -f restricted-namespace.yaml
```

**Step 2: Try to create a Pod that runs as root**

```yaml
# bad-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: bad-pod
  namespace: restricted-ns
spec:
  containers:
    - name: nginx
      image: nginx:latest
```

```bash
kubectl apply -f bad-pod.yaml
```

**What happens?** You'll get an error like:

```
Error from server (Forbidden): error when creating "bad-pod.yaml": pods "bad-pod" is 
forbidden: violates PodSecurity "restricted:latest": 
  allowPrivilegeEscalation != false 
    (container "nginx" must set securityContext.allowPrivilegeEscalation=false), 
  unrestricted capabilities 
    (container "nginx" must set securityContext.capabilities.drop=["ALL"]), 
  runAsNonRoot != true 
    (pod or container "nginx" must set securityContext.runAsNonRoot=true), 
  seccompProfile 
    (pod or container "nginx" must set securityContext.seccompProfile.type to 
    "RuntimeDefault" or "Localhost")
```

The error is very descriptive — it tells you **exactly what to fix**!

**Step 3: Fix the Pod to comply with restricted standard**

```yaml
# good-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: good-pod
  namespace: restricted-ns
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1000
    seccompProfile:
      type: RuntimeDefault
  containers:
    - name: nginx
      image: nginx:latest
      securityContext:
        allowPrivilegeEscalation: false
        capabilities:
          drop:
            - ALL
```

```bash
kubectl apply -f good-pod.yaml
# pod/good-pod created ✅
```

---

### Example 2: Baseline Enforcement with Restricted Warnings

```yaml
# balanced-namespace.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: balanced-ns
  labels:
    pod-security.kubernetes.io/enforce: baseline
    pod-security.kubernetes.io/warn: restricted
    pod-security.kubernetes.io/audit: restricted
```

```bash
kubectl apply -f balanced-namespace.yaml
```

Now create a simple Pod:

```yaml
# simple-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
  namespace: balanced-ns
spec:
  containers:
    - name: app
      image: nginx:latest
```

```bash
kubectl apply -f simple-pod.yaml
```

**Output:**

```
Warning: would violate PodSecurity "restricted:latest": 
  allowPrivilegeEscalation != false, unrestricted capabilities, 
  runAsNonRoot != true, seccompProfile
pod/my-app created
```

See that? The Pod **was created** (because it passes baseline), but you got a **warning** that it doesn't meet restricted standards. Super helpful for gradual migration!

---

### Example 3: Trying a Privileged Pod in a Baseline Namespace

```yaml
# privileged-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: hacker-pod
  namespace: balanced-ns
spec:
  containers:
    - name: evil
      image: ubuntu:latest
      securityContext:
        privileged: true    # ← This is dangerous!
```

```bash
kubectl apply -f privileged-pod.yaml
```

**Output:**

```
Error from server (Forbidden): error when creating "privileged-pod.yaml": pods 
"hacker-pod" is forbidden: violates PodSecurity "baseline:latest": 
  privileged (container "evil" must not set securityContext.privileged=true)
```

Blocked! The baseline standard caught the privileged container.

## 10. Exemptions in PSA

Sometimes, certain Pods **need** to break the rules. For example:
- **kube-system** namespace Pods (CNI, CSI drivers)
- Monitoring agents that need host access
- Logging collectors

You can configure **exemptions** in the PSA admission controller's configuration file:

```yaml
# admission-configuration.yaml
apiVersion: apiserver.config.k8s.io/v1
kind: AdmissionConfiguration
plugins:
  - name: PodSecurity
    configuration:
      apiVersion: pod-security.admission.config.k8s.io/v1
      kind: PodSecurityConfiguration
      defaults:
        enforce: baseline
        enforce-version: latest
        audit: restricted
        audit-version: latest
        warn: restricted
        warn-version: latest
      exemptions:
        # Exempt specific usernames
        usernames:
          - system:serviceaccount:kube-system:calico-node
        
        # Exempt specific runtime class names
        runtimeClasses:
          - untrusted
        
        # Exempt specific namespaces
        namespaces:
          - kube-system
          - kube-node-lease
          - monitoring
```

### How to Apply This Configuration

For **kubeadm** clusters, reference this file in the API server manifest:

```yaml
# /etc/kubernetes/manifests/kube-apiserver.yaml
spec:
  containers:
    - command:
        - kube-apiserver
        - --admission-control-config-file=/etc/kubernetes/psa/admission-configuration.yaml
      volumeMounts:
        - mountPath: /etc/kubernetes/psa
          name: psa-config
          readOnly: true
  volumes:
    - hostPath:
        path: /etc/kubernetes/psa
        type: DirectoryOrCreate
      name: psa-config
```

For **managed clusters** (EKS, AKS, GKE), exemptions are typically handled through cluster-level settings or the namespace label approach.

## 11. Migrating from PodSecurityPolicy to PSA

If you're still on an older cluster with PSP, here's a migration plan:

### Step-by-Step Migration

```
Step 1: Review existing PSPs
        ↓
Step 2: Map PSPs to PSS levels (privileged/baseline/restricted)
        ↓
Step 3: Apply PSA labels in "warn" + "audit" mode
        ↓
Step 4: Monitor warnings and audit logs
        ↓
Step 5: Fix non-compliant workloads
        ↓
Step 6: Switch PSA mode to "enforce"
        ↓
Step 7: Remove old PSPs and disable PSP admission controller
```

### Useful Command — Dry Run

You can test what would happen if you applied a PSS level to a namespace:

```bash
# Check all Pods in namespace "default" against "restricted" standard
kubectl label --dry-run=server --overwrite namespace default \
  pod-security.kubernetes.io/enforce=restricted
```

This will show you which Pods would be rejected **without actually enforcing anything**.

## 12. Best Practices

### 1. Start Gradually
```
warn → audit → enforce
```
Never jump straight to enforce on existing namespaces.

### 2. Version Pin Your Standards
```yaml
pod-security.kubernetes.io/enforce-version: v1.28
```
This prevents surprises when upgrading Kubernetes (new versions may add stricter checks).

### 3. Use "latest" in Dev, Pin in Production
- **Dev/staging:** `enforce-version: latest` (to catch issues early)
- **Production:** `enforce-version: v1.28` (for stability)

### 4. Apply Restricted by Default
Make `restricted` the default for all new namespaces. Only relax to `baseline` or `privileged` when there's a clear justification.

### 5. Always Exempt kube-system
System components often need privileged access. Always exempt `kube-system`.

### 6. Combine with Other Security Tools
PSA is just one layer. Combine it with:
- **OPA/Gatekeeper** or **Kyverno** for more fine-grained policies
- **Network Policies** for network isolation
- **RBAC** for access control
- **Admission webhooks** for custom validations

### 7. Document Your Standards
Create a clear document that tells developers:
- What security level each namespace uses
- What they need to add to their Pod specs
- Examples of compliant Pods

## 13. Common Pitfalls & Troubleshooting

### Pitfall 1: "My Pod Won't Create But I Don't Know Why"

**Solution:** Add `warn` and `audit` labels first to see what's wrong:

```bash
kubectl label namespace my-ns \
  pod-security.kubernetes.io/warn=restricted \
  pod-security.kubernetes.io/audit=restricted
```

Then try creating the Pod — you'll see detailed warning messages.

### Pitfall 2: "I Applied Labels But Nothing Happens"

**Check:**
- Is the PSA admission controller enabled? (It's enabled by default in K8s 1.25+)
- Labels are **case-sensitive** — double-check spelling
- Labels apply to the **namespace**, not the Pod

### Pitfall 3: "Existing Pods Still Running After I Applied Enforce"

**This is by design!** PSA only checks Pods at **creation time**. It does NOT retroactively delete existing Pods. To enforce on existing workloads, you need to recreate them:

```bash
kubectl rollout restart deployment -n restricted-ns my-deployment
```

### Pitfall 4: "My Helm Chart Doesn't Work in Restricted Namespace"

Many Helm charts run as root by default. You'll need to override security context values:

```bash
helm install my-app my-chart \
  --set securityContext.runAsNonRoot=true \
  --set securityContext.runAsUser=1000 \
  --set containerSecurityContext.allowPrivilegeEscalation=false \
  --set containerSecurityContext.capabilities.drop='{ALL}'
```

## 14. Quick Reference Cheat Sheet

### Apply Security Standards

```bash
# Enforce baseline
kubectl label ns <namespace> pod-security.kubernetes.io/enforce=baseline

# Warn restricted
kubectl label ns <namespace> pod-security.kubernetes.io/warn=restricted

# Audit restricted
kubectl label ns <namespace> pod-security.kubernetes.io/audit=restricted

# Remove a label
kubectl label ns <namespace> pod-security.kubernetes.io/enforce-

# Dry-run check
kubectl label --dry-run=server --overwrite ns <namespace> \
  pod-security.kubernetes.io/enforce=restricted
```

### Minimum Restricted-Compliant Pod

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: secure-pod
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1000
    fsGroup: 1000
    seccompProfile:
      type: RuntimeDefault
  containers:
    - name: app
      image: my-app:latest
      securityContext:
        allowPrivilegeEscalation: false
        readOnlyRootFilesystem: true
        capabilities:
          drop:
            - ALL
      resources:
        requests:
          memory: "64Mi"
          cpu: "100m"
        limits:
          memory: "128Mi"
          cpu: "250m"
```

### Check Namespace Labels

```bash
# See all PSA labels on a namespace
kubectl get namespace <namespace> -o yaml | grep pod-security

# List all namespaces with their PSA labels
kubectl get namespaces -L pod-security.kubernetes.io/enforce
```

## 15. Summary

| Concept | What It Is |
|---------|-----------|
| **Pod Security Standards (PSS)** | Three tiers of security rules: Privileged, Baseline, Restricted |
| **Pod Security Admission (PSA)** | The built-in admission controller that enforces PSS |
| **Privileged** | No restrictions — for system components |
| **Baseline** | Blocks dangerous stuff — good for most apps |
| **Restricted** | Full hardening — for security-critical workloads |
| **Enforce mode** | Blocks non-compliant Pods |
| **Warn mode** | Shows warning but allows the Pod |
| **Audit mode** | Logs violation but allows the Pod |
| **Namespace labels** | How you apply PSS levels to namespaces |
| **Exemptions** | How you exclude certain users/namespaces/runtimes from checks |

### Key Takeaway

> PSA is Kubernetes' **built-in, zero-install, simple** way to enforce Pod security. Start with `warn`, move to `audit`, then `enforce`. Use namespace labels — it's that easy!
