## 1. The Problem — Why Do We Need Policy as Code?

Let's say you're running a Kubernetes cluster, and you want to make sure:

- Every Pod has **resource limits** set (so one app doesn't eat all the memory)
- Every resource has proper **labels** (like `team`, `environment`)
- No one uses the `latest` image tag (because that's unpredictable)
- Containers can't run as **root**
- Only approved container registries are used

You could put these rules in a Word document and say "Hey everyone, follow these rules!" — but we all know what happens... **people forget, people make mistakes, people ignore rules**.

**Policy as Code** means:

> "Write your rules as actual code, and let the system **automatically enforce** them."

Instead of a human reviewing every deployment, a **machine** checks it instantly. No more relying on memory or discipline!

## 2. What Is OPA (Open Policy Agent)?

**OPA** (pronounced "oh-pa") is a **general-purpose policy engine**. It's not specific to Kubernetes — you can use it for:

- Kubernetes admission control
- API authorization
- Terraform plan validation
- Microservice authorization
- Data filtering

Think of OPA as a **universal decision-making engine**:

```
┌─────────────┐     ┌────────────┐     ┌──────────┐
│  Question   │────▶│    OPA     │────▶│  Answer  │
│ (Input)     │     │  (Policy   │     │ (Allow/  │
│             │     │   Engine)  │     │  Deny)   │
└─────────────┘     └────────────┘     └──────────┘
                         ▲
                         │
                    ┌────────────┐
                    │  Policies  │
                    │  (written  │
                    │  in Rego)  │
                    └────────────┘
```

**How OPA works in simple terms:**

1. Something asks OPA a **question** (e.g., "Can this Pod be created?")
2. OPA evaluates the question against its **policies** (written in a language called **Rego**)
3. OPA gives an **answer** (yes/no, allow/deny, or any structured data)

OPA is a **CNCF Graduated Project** — meaning it's widely adopted and production-ready.

## 3. What Is Gatekeeper?

**Gatekeeper** is OPA's **Kubernetes-specific integration**. While OPA is a generic policy engine, Gatekeeper is built specifically to work with Kubernetes.

Think of it like this:

| Component | Analogy |
|-----------|---------|
| **OPA** | A general-purpose engine (like a car engine) |
| **Gatekeeper** | The complete car, designed specifically for Kubernetes roads |

Gatekeeper does these things:
- Runs as a **Kubernetes admission webhook** (catches API requests before they're processed)
- Uses **Custom Resource Definitions (CRDs)** to define policies (so you manage policies like any other K8s resource)
- Provides **audit** functionality (check existing resources against policies)
- Supports **mutation** (automatically modify resources to comply with policies)

## 4. OPA vs Gatekeeper — How Are They Related?

```
┌──────────────────────────────────────────────────┐
│                    Gatekeeper                     │
│  ┌──────────────────────────────────────────────┐│
│  │                                              ││
│  │   OPA Engine  (embedded inside Gatekeeper)   ││
│  │                                              ││
│  └──────────────────────────────────────────────┘│
│                                                  │
│  + Kubernetes CRDs (ConstraintTemplates)         │
│  + Admission Webhook                             │
│  + Audit Controller                              │
│  + Mutation Support                              │
│  + Status Reporting                              │
│                                                  │
└──────────────────────────────────────────────────┘
```

**Key difference:**

| Feature | OPA (standalone) | Gatekeeper |
|---------|------------------|------------|
| Kubernetes-native | ❌ Needs sidecar/webhook setup | ✅ CRDs + webhook built in |
| Policy management | Files/ConfigMaps | CRDs (ConstraintTemplates) |
| Audit existing resources | ❌ No | ✅ Yes |
| Mutation support | Custom code needed | ✅ Built-in |
| Learning curve | Higher (need to set up everything) | Lower (Kubernetes-native) |

**Bottom line:** For Kubernetes, always use **Gatekeeper** (not standalone OPA).

## 5. Gatekeeper Architecture — How It Works

```
                    Developer runs: kubectl apply -f pod.yaml
                                    │
                                    ▼
                            ┌──────────────┐
                            │  API Server  │
                            └──────┬───────┘
                                   │
                    ┌──────────────┼────────────────────┐
                    │              ▼                     │
                    │  ┌───────────────────────┐         │
                    │  │  Validating Admission │         │
                    │  │  Webhook              │         │
                    │  └───────────┬───────────┘         │
                    │              │                     │
                    │              ▼                     │
                    │  ┌───────────────────────┐         │
                    │  │     Gatekeeper        │         │
                    │  │  ┌─────────────────┐  │         │
                    │  │  │   OPA Engine    │  │         │
                    │  │  │  + Rego Policy  │  │         │
                    │  │  └─────────────────┘  │         │
                    │  │  ┌─────────────────┐  │         │
                    │  │  │ Constraint      │  │         │
                    │  │  │ Templates       │  │         │
                    │  │  └─────────────────┘  │         │
                    │  │  ┌─────────────────┐  │         │
                    │  │  │ Constraints     │  │         │
                    │  │  └─────────────────┘  │         │
                    │  └───────────┬───────────┘         │
                    │              │                     │
                    │         ✅ or ❌                    │
                    └──────────────┼────────────────────┘
                                   │
                              ┌────┴────┐
                              │         │
                           ✅ Allow   ❌ Deny
                           Pod created   Error returned
```

**Key Components:**

1. **Webhook Server** — Intercepts API requests from the API server
2. **OPA Engine** — Evaluates policies against the incoming request
3. **ConstraintTemplate** — Defines the **shape** of a policy (like a class definition)
4. **Constraint** — An **instance** of a ConstraintTemplate with specific parameters (like an object)
5. **Audit Controller** — Periodically checks existing resources against all constraints

## 7. Core Concepts — ConstraintTemplates and Constraints

This is the **most important concept** in Gatekeeper. Let's break it down with an analogy:

### Think of It Like a Cookie Cutter

| Concept | Analogy | Purpose |
|---------|---------|---------|
| **ConstraintTemplate** | The cookie cutter shape | Defines WHAT to check (the Rego logic) and WHAT parameters it accepts |
| **Constraint** | An actual cookie made with that cutter | Specifies the actual rules (parameter values) and WHERE to apply them |

```
ConstraintTemplate                    Constraint
(Defines the "what" and "how")        (Defines the "where" and specific values)
                                      
"Check if a label exists"      →      "Check if 'team' label exists
                                       on all Pods in namespace 'production'"
                                      
"Check if image tag is allowed" →     "Only allow images from
                                       'myregistry.com' in all namespaces"
```

### The Two-Step Process

```
Step 1: Create a ConstraintTemplate  →  "Here's the RULE LOGIC"
                                        (written in Rego)

Step 2: Create a Constraint           →  "Apply THIS RULE with THESE
                                          parameters to THESE resources"
```

## 8. Rego Language — The Basics

**Rego** (pronounced "ray-go") is the policy language used by OPA. Don't worry — you don't need to become a Rego expert! Here are the basics:

### Key Concepts

```rego
# A package declaration (required)
package k8srequiredlabels

# A "violation" rule — this is what Gatekeeper looks for
# If this rule produces a result, it means there's a violation
violation[{"msg": msg}] {
  # This is the condition to check
  # If ALL conditions in this block are true, it's a violation
  
  some label                                    # Iterate over something
  input.review.object.metadata.labels[label]    # Access the resource's labels
  not has_label(label)                          # Check if a required label is missing
  msg := sprintf("Missing required label: %v", [label])
}
```

### Important Rego Rules

```rego
# 1. Variables are assigned with :=
name := "hello"

# 2. Comparison uses ==
x == 5

# 3. "not" negates a condition
not input.review.object.metadata.labels["team"]

# 4. Everything in a rule body is AND'ed together
# All conditions must be true for the rule to match
violation[{"msg": msg}] {
  condition1    # AND
  condition2    # AND
  condition3    # ALL must be true
}

# 5. Multiple rule definitions with the same name are OR'ed
# If ANY of these matches, it's a violation
violation[{"msg": msg}] {
  # Check 1
}
violation[{"msg": msg}] {
  # Check 2 (alternative)
}

# 6. input.review.object is the Kubernetes resource being evaluated
# input.parameters contains the constraint's parameters
```

### Accessing Kubernetes Resources in Rego

```rego
# The Pod being created/updated
input.review.object

# The Pod's metadata
input.review.object.metadata

# The Pod's labels
input.review.object.metadata.labels

# The Pod's containers
input.review.object.spec.containers

# The kind of resource
input.review.kind.kind    # "Pod", "Deployment", etc.

# The parameters from the Constraint
input.parameters
```

## 9. Hands-On Examples

### Example 1: Require Specific Labels

This is the classic "hello world" of Gatekeeper.

**Step 1: Create the ConstraintTemplate**

```yaml
# require-labels-template.yaml
apiVersion: templates.gatekeeper.sh/v1
kind: ConstraintTemplate
metadata:
  name: k8srequiredlabels
spec:
  crd:
    spec:
      names:
        kind: K8sRequiredLabels    # This becomes the Constraint "kind"
      validation:
        openAPIV3Schema:
          type: object
          properties:
            labels:
              type: array
              description: "A list of required labels"
              items:
                type: string
  targets:
    - target: admission.k8s.gatekeeper.sh
      rego: |
        package k8srequiredlabels

        violation[{"msg": msg, "details": {"missing_labels": missing}}] {
          # Get the set of provided labels
          provided := {label | input.review.object.metadata.labels[label]}
          
          # Get the set of required labels (from constraint parameters)
          required := {label | label := input.parameters.labels[_]}
          
          # Find which required labels are missing
          missing := required - provided
          
          # If there are missing labels, it's a violation
          count(missing) > 0
          
          msg := sprintf("You are missing required labels: %v", [missing])
        }
```

Let's break down what's happening here:

- **`crd.spec`** — Defines the schema for the Constraint (what parameters it accepts). Here, it accepts a list of label names.
- **`targets.rego`** — The actual policy logic written in Rego.
- **`violation`** — If this rule produces a result, the request is denied.
- **`input.review.object`** — The Kubernetes resource being checked.
- **`input.parameters`** — The parameters provided by the Constraint.

```bash
kubectl apply -f require-labels-template.yaml
```

**Step 2: Create the Constraint**

```yaml
# require-team-label.yaml
apiVersion: constraints.gatekeeper.sh/v1beta1
kind: K8sRequiredLabels       # Must match the ConstraintTemplate's kind!
metadata:
  name: require-team-label
spec:
  enforcementAction: deny     # deny | dryrun | warn
  match:
    kinds:
      - apiGroups: [""]
        kinds: ["Namespace"]  # Apply this to Namespaces
    excludedNamespaces:
      - kube-system
      - gatekeeper-system
  parameters:
    labels:
      - "team"                # Require the "team" label
```

What this says: "Every Namespace (except kube-system and gatekeeper-system) MUST have a `team` label."

```bash
kubectl apply -f require-team-label.yaml
```

**Step 3: Test It!**

```bash
# Try to create a namespace without the "team" label
kubectl create namespace bad-namespace
```

**Output:**

```
Error from server (Forbidden): admission webhook "validation.gatekeeper.sh" denied 
the request: [require-team-label] You are missing required labels: {"team"}
```

Blocked! Now try with the label:

```bash
kubectl create namespace good-namespace --dry-run=server -o yaml
# That would fail too. Let's use YAML:
```

```yaml
# good-namespace.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: good-namespace
  labels:
    team: platform
```

```bash
kubectl apply -f good-namespace.yaml
# namespace/good-namespace created ✅
```

### Example 2: Block `latest` Image Tag

```yaml
# disallow-latest-tag-template.yaml
apiVersion: templates.gatekeeper.sh/v1
kind: ConstraintTemplate
metadata:
  name: k8sdisallowedtags
spec:
  crd:
    spec:
      names:
        kind: K8sDisallowedTags
      validation:
        openAPIV3Schema:
          type: object
          properties:
            tags:
              type: array
              description: "Disallowed image tags"
              items:
                type: string
  targets:
    - target: admission.k8s.gatekeeper.sh
      rego: |
        package k8sdisallowedtags

        violation[{"msg": msg}] {
          # Check each container in the Pod
          container := input.review.object.spec.containers[_]
          
          # Get the image tag
          tag := [t | split(container.image, ":"); t := _[count(_) - 1]][0]
          
          # Check if this tag is in the disallowed list
          disallowed := input.parameters.tags[_]
          tag == disallowed
          
          msg := sprintf("Container '%v' uses disallowed image tag '%v'. Please use a specific version tag.", [container.name, tag])
        }

        # Also catch images with no tag (which defaults to "latest")
        violation[{"msg": msg}] {
          container := input.review.object.spec.containers[_]
          
          # If the image doesn't contain a ":", it has no tag (defaults to latest)
          not contains(container.image, ":")
          
          # Check if "latest" is in the disallowed list
          "latest" == input.parameters.tags[_]
          
          msg := sprintf("Container '%v' has no image tag specified (defaults to 'latest'). Please use a specific version tag.", [container.name])
        }
```

```bash
kubectl apply -f disallow-latest-tag-template.yaml
```

Now create the Constraint:

```yaml
# no-latest-tag.yaml
apiVersion: constraints.gatekeeper.sh/v1beta1
kind: K8sDisallowedTags
metadata:
  name: no-latest-tag
spec:
  enforcementAction: deny
  match:
    kinds:
      - apiGroups: [""]
        kinds: ["Pod"]
      - apiGroups: ["apps"]
        kinds: ["Deployment", "StatefulSet", "DaemonSet"]
    excludedNamespaces:
      - kube-system
      - gatekeeper-system
  parameters:
    tags:
      - "latest"
```

```bash
kubectl apply -f no-latest-tag.yaml
```

Test it:

```bash
# This will be blocked
kubectl run test --image=nginx:latest
# Error: Container 'test' uses disallowed image tag 'latest'

# This will also be blocked (no tag = latest)
kubectl run test --image=nginx
# Error: Container 'test' has no image tag specified

# This will work
kubectl run test --image=nginx:1.25.3
# pod/test created ✅
```

### Example 3: Require Resource Limits

```yaml
# require-limits-template.yaml
apiVersion: templates.gatekeeper.sh/v1
kind: ConstraintTemplate
metadata:
  name: k8srequiredresources
spec:
  crd:
    spec:
      names:
        kind: K8sRequiredResources
      validation:
        openAPIV3Schema:
          type: object
          properties:
            resources:
              type: array
              description: "Required resource types"
              items:
                type: string
  targets:
    - target: admission.k8s.gatekeeper.sh
      rego: |
        package k8srequiredresources

        violation[{"msg": msg}] {
          container := input.review.object.spec.containers[_]
          resource_type := input.parameters.resources[_]
          
          # Check if limits are set for the required resource type
          not container.resources.limits[resource_type]
          
          msg := sprintf(
            "Container '%v' must have resource limits set for '%v'.",
            [container.name, resource_type]
          )

        violation[{"msg": msg}] {
          container := input.review.object.spec.containers[_]
          resource_type := input.parameters.resources[_]
          
          # Check if requests are set for the required resource type
          not container.resources.requests[resource_type]
          
          msg := sprintf(
            "Container '%v' must have resource requests set for '%v'.",
            [container.name, resource_type]
          )
        }
```

### Example 4: Allow Only Specific Container Registries

```yaml
# allowed-repos-template.yaml
apiVersion: templates.gatekeeper.sh/v1
kind: ConstraintTemplate
metadata:
  name: k8sallowedrepos
spec:
  crd:
    spec:
      names:
        kind: K8sAllowedRepos
      validation:
        openAPIV3Schema:
          type: object
          properties:
            repos:
              type: array
              description: "List of allowed container image repositories"
              items:
                type: string
  targets:
    - target: admission.k8s.gatekeeper.sh
      rego: |
        package k8sallowedrepos

        violation[{"msg": msg}] {
          container := input.review.object.spec.containers[_]
          
          # Check if the container image starts with any allowed repo
          not any_repo_match(container.image)
          
          msg := sprintf(
            "Container '%v' uses image '%v' which is not from an allowed repository. Allowed repos: %v",
            [container.name, container.image, input.parameters.repos]
          )
        }

        # Also check init containers
        violation[{"msg": msg}] {
          container := input.review.object.spec.initContainers[_]
          not any_repo_match(container.image)
          
          msg := sprintf(
            "Init container '%v' uses image '%v' which is not from an allowed repository. Allowed repos: %v",
            [container.name, container.image, input.parameters.repos]
          )
        }

        any_repo_match(image) {
          repo := input.parameters.repos[_]
          startswith(image, repo)
        }
```

```yaml
# only-company-registry.yaml

  name: only-company-registry

  enforcementAction: deny

    kinds:
      - apiGroups: [""]
        kinds: ["Pod"]
    excludedNamespaces:
      - kube-system
      - gatekeeper-system

    repos:
      - "mycompany.azurecr.io/"
      - "gcr.io/my-project/"
```

## 10. Audit Mode — Checking Existing Resources

One of Gatekeeper's killer features is **audit mode**. It doesn't just check new resources — it can scan **existing resources** too!

### How Audit Works

The `gatekeeper-audit` pod periodically scans all resources in the cluster and reports violations in the Constraint's `status` field.

```bash
# Check the audit results for a constraint
kubectl get k8srequiredlabels require-team-label -o yaml
```

Look at the `status.violations` section:

```yaml
status:
  auditTimestamp: "2024-01-15T10:30:00Z"
  totalViolations: 3
  violations:
    - enforcementAction: deny
      kind: Namespace
      name: default
      message: "You are missing required labels: {\"team\"}"
    - enforcementAction: deny
      kind: Namespace
      name: some-old-namespace
      message: "You are missing required labels: {\"team\"}"
```

### Dry-Run Mode

You can set a constraint to `dryrun` to see violations without blocking anything:

```yaml
spec:
  enforcementAction: dryrun   # Just audit, don't block
```

This is perfect for:
1. Testing new policies before enforcing them
2. Understanding the impact of a policy on existing resources
3. Gradual rollout (dryrun → warn → deny)

### Enforcement Actions Summary

| Action | Blocks New Resources? | Reports in Audit? | Shows Warnings? |
|--------|----------------------|-------------------|-----------------|
| `deny` | ✅ Yes | ✅ Yes | ✅ Yes |
| `warn` | ❌ No | ✅ Yes | ✅ Yes (in kubectl output) |
| `dryrun` | ❌ No | ✅ Yes | ❌ No |

## 11. Mutation with Gatekeeper

Gatekeeper can not only **validate** (allow/deny) but also **mutate** (modify) resources. This is super useful for automatically fixing things!

### Example: Automatically Add a Label

```yaml
# add-default-label.yaml
apiVersion: mutations.gatekeeper.sh/v1
kind: Assign
metadata:
  name: add-environment-label
spec:
  applyTo:
    - groups: [""]
      kinds: ["Pod"]
      versions: ["v1"]
  match:
    scope: Namespaced
    kinds:
      - apiGroups: [""]
        kinds: ["Pod"]
    excludedNamespaces:
      - kube-system
  location: "metadata.labels.environment"
  parameters:
    assign:
      value: "unknown"          # Default value if label is missing
    pathTests:
      - subPath: "metadata.labels.environment"
        condition: MustNotExist  # Only add if the label doesn't exist
```

This automatically adds `environment: unknown` to any Pod that doesn't already have an `environment` label.

### Example: Set Image Pull Policy

```yaml
# set-image-pull-policy.yaml
apiVersion: mutations.gatekeeper.sh/v1
kind: AssignMetadata
metadata:
  name: always-pull-images
spec:
  match:
    scope: Namespaced
    kinds:
      - apiGroups: [""]
        kinds: ["Pod"]
  location: "metadata.annotations.policy\\.applied"
  parameters:
    assign:
      value: "image-pull-always"
```

### Mutation Types

| Type | What It Does |
|------|-------------|
| `Assign` | Sets a field value (can be deep in the spec) |
| `AssignMetadata` | Sets metadata fields (labels, annotations) |
| `ModifySet` | Adds/removes items from a list |
| `AssignImage` | Specifically modifies container image references |

## 12. External Data with Gatekeeper

Sometimes your policies need to check against **external data** — like "Is this image in our approved list?" where the list comes from an external API or database.

Gatekeeper supports this through **External Data Providers**:

```yaml
apiVersion: externaldata.gatekeeper.sh/v1beta1
kind: Provider
metadata:
  name: image-scanner
spec:
  url: https://my-scanner.internal/validate
  timeout: 5
  caBundle: <base64-encoded-CA-cert>
```

Then in your Rego:

```rego
violation[{"msg": msg}] {
  container := input.review.object.spec.containers[_]
  
  # Call external data provider
  response := external_data({"provider": "image-scanner", "keys": [container.image]})
  
  # Check the response
  response.errors[_]
  msg := sprintf("Image '%v' failed security scan", [container.image])
}
```

## 13. Gatekeeper Library — Pre-Built Policies

Great news — you don't have to write everything from scratch! The **Gatekeeper Library** has dozens of pre-built policies ready to use.

### Where to Find Them

The official library is at: `https://github.com/open-policy-agent/gatekeeper-library`

### Categories Available

| Category | Example Policies |
|----------|-----------------|
| **General** | Required labels, allowed repos, disallowed tags, block NodePort |
| **Pod Security** | Privileged containers, host networking, read-only root filesystem |
| **RBAC** | Disallow wildcard verbs, require specific roles |

### Using a Library Policy

```bash
# Clone the library
git clone https://github.com/open-policy-agent/gatekeeper-library.git

# Apply a template and constraint
kubectl apply -f gatekeeper-library/library/general/allowedrepos/template.yaml
kubectl apply -f gatekeeper-library/library/general/allowedrepos/samples/repo-must-be-openpolicyagent/constraint.yaml
```

Or just copy the YAML from the repo and customize it!

## 14. Debugging and Troubleshooting

### Check If ConstraintTemplate Is Ready

```bash
kubectl get constrainttemplates
# NAME                  AGE
# k8srequiredlabels     5m
# k8sdisallowedtags     3m

# Check for errors
kubectl describe constrainttemplate k8srequiredlabels
```

Look at the `status` section — if there are Rego errors, they'll show up here.

### Check If Constraint Is Active

```bash
kubectl get constraints
# Or specifically:
kubectl get k8srequiredlabels

# Detailed view with audit results:
kubectl get k8srequiredlabels require-team-label -o yaml
```

### Check Gatekeeper Logs

```bash
# Controller manager logs (for admission decisions)
kubectl logs -n gatekeeper-system -l control-plane=controller-manager

# Audit logs (for audit scan results)
kubectl logs -n gatekeeper-system -l control-plane=audit-controller
```

### Common Error: "no matches for kind"

```
error: unable to recognize "constraint.yaml": no matches for kind "K8sRequiredLabels"
```

**Fix:** Make sure the ConstraintTemplate is applied BEFORE the Constraint. The template creates the CRD that the constraint uses.

```bash
# Correct order:
kubectl apply -f template.yaml    # First!
kubectl apply -f constraint.yaml  # Second!
```

### Common Error: Webhook Timeout

If Gatekeeper pods are down, ALL requests to the API server might fail because the webhook is still configured.

**Emergency fix:**

```bash
# Remove the webhook temporarily (use with caution!)
kubectl delete validatingwebhookconfiguration gatekeeper-validating-webhook-configuration
```

To prevent this, configure the webhook with `failurePolicy: Ignore`:

```bash
helm install gatekeeper gatekeeper/gatekeeper \
  --set webhook.failPolicy=Ignore
```

### Testing Policies Locally

Use the **Gatekeeper CLI** (`gator`) to test policies without a cluster:

```bash
# Install gator
go install github.com/open-policy-agent/gatekeeper/v3/cmd/gator@latest

# Test a policy
gator test -f template.yaml -f constraint.yaml -f test-resource.yaml

# Verify a template's Rego
gator verify -f template.yaml
```

## 15. Best Practices

### 1. Start with Dry Run
```yaml
spec:
  enforcementAction: dryrun    # Start here
  # enforcementAction: warn    # Then move here
  # enforcementAction: deny    # Finally enforce
```

### 2. Always Exclude System Namespaces
```yaml
spec:
  match:
    excludedNamespaces:
      - kube-system
      - kube-public
      - kube-node-lease
      - gatekeeper-system
```

### 3. Use Descriptive Error Messages
```rego
msg := sprintf(
  "Container '%v' in Pod '%v' must have CPU limits set. See docs at: https://wiki.company.com/k8s-limits",
  [container.name, input.review.object.metadata.name]
)
```

### 4. Version Control Your Policies
Store all ConstraintTemplates and Constraints in Git. Apply them via CI/CD (GitOps).

### 5. Test with Gator Before Deploying
```bash
gator test -f policies/ -f test-cases/
```

### 6. Monitor Audit Results
Set up alerts for increasing violation counts:
```bash
# Check total violations across all constraints
kubectl get constraints -o json | jq '.items[].status.totalViolations'
```

### 7. Use the Gatekeeper Library
Don't reinvent the wheel — start with the library and customize.

### 8. Set Webhook Failure Policy
```yaml
# For non-critical policies: allow requests if Gatekeeper is down
failurePolicy: Ignore

# For critical policies: block requests if Gatekeeper is down
failurePolicy: Fail
```

## 16. OPA/Gatekeeper vs PSA vs Kyverno

| Feature | PSA (Built-in) | Gatekeeper | Kyverno |
|---------|----------------|------------|---------|
| **Installation** | Built-in (nothing to install) | Helm/YAML | Helm/YAML |
| **Policy Language** | Labels only | Rego | YAML (native K8s) |
| **Learning Curve** | Very low | High (Rego) | Low (just YAML) |
| **Flexibility** | Limited (3 fixed levels) | Very high (custom Rego) | High (YAML-based) |
| **Mutation** | ❌ No | ✅ Yes | ✅ Yes |
| **Audit** | ❌ No | ✅ Yes | ✅ Yes |
| **Generation** | ❌ No | ❌ No | ✅ Yes (can create resources) |
| **Custom Policies** | ❌ No | ✅ Unlimited | ✅ Unlimited |
| **Scope** | Pod security only | Any K8s resource | Any K8s resource |
| **Best For** | Basic pod security | Complex/custom policies | Teams who prefer YAML |

**Recommendation:**
- **PSA** for basic pod security (use it alongside Gatekeeper or Kyverno)
- **Gatekeeper** when you need complex policies or already use OPA elsewhere
- **Kyverno** when your team prefers YAML over learning Rego (covered in next section)

## 17. Summary

| Concept | What It Is |
|---------|-----------|
| **OPA** | General-purpose policy engine, uses Rego language |
| **Gatekeeper** | OPA for Kubernetes — CRDs, webhooks, audit |
| **ConstraintTemplate** | Defines the policy logic (Rego code) + parameter schema |
| **Constraint** | Instance of a template — specifies parameters + scope |
| **Rego** | Policy language — `violation[{"msg": msg}] { conditions }` |
| **Enforcement Actions** | `deny` (block), `warn` (warning), `dryrun` (audit only) |
| **Audit** | Scans existing resources against all constraints |
| **Mutation** | Automatically modify resources (Assign, AssignMetadata, ModifySet) |
| **Gatekeeper Library** | Pre-built policies ready to use |
| **gator CLI** | Test policies locally without a cluster |

### Key Takeaway

> Gatekeeper brings **"policy as code"** to Kubernetes. Define your rules in **ConstraintTemplates** (using Rego), instantiate them with **Constraints**, and let Gatekeeper **enforce, audit, and report** — all as native Kubernetes resources. Start with `dryrun`, graduate to `deny`.
