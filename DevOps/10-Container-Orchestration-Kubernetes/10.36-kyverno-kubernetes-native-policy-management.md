## 1. What Is Kyverno?

**Kyverno** (Greek for "govern") is a **policy engine designed specifically for Kubernetes**. It lets you manage policies as Kubernetes resources — using plain YAML!

Here's the magic: **No new language to learn!** Unlike OPA/Gatekeeper (which requires Rego), Kyverno uses the same YAML you already use for everything else in Kubernetes.

```
┌──────────────────────────────────────────────────┐
│                    Kyverno                       │
│                                                  │
│  "Write policies in YAML.                        │
│   Validate, Mutate, Generate —                   │
│   all in Kubernetes-native style."               │
│                                                  │
│  📝 YAML in → 🛡️ Policy enforced               │
│                                                  │
└──────────────────────────────────────────────────┘
```

Kyverno is a **CNCF Incubating Project** — actively maintained, widely adopted, and production-ready.

**What can Kyverno do?**

| Capability | What It Means |
|-----------|---------------|
| **Validate** | Allow or deny resource requests based on rules |
| **Mutate** | Automatically modify resources (add labels, set defaults, etc.) |
| **Generate** | Automatically create new resources when something happens |
| **Verify Images** | Check container image signatures and attestations |

## 2. Why Kyverno? — The "Just YAML" Advantage

Let's compare how you'd write a "require labels" policy in Gatekeeper vs Kyverno:

### Gatekeeper Way (Rego + YAML — TWO resources needed)

```yaml
# 1. ConstraintTemplate (with Rego code inside)
apiVersion: templates.gatekeeper.sh/v1
kind: ConstraintTemplate
metadata:
  name: k8srequiredlabels
spec:
  crd:
    spec:
      names:
        kind: K8sRequiredLabels
      validation:
        openAPIV3Schema:
          type: object
          properties:
            labels:
              type: array
              items:
                type: string
  targets:
    - target: admission.k8s.gatekeeper.sh
      rego: |
        package k8srequiredlabels
        violation[{"msg": msg, "details": {"missing_labels": missing}}] {
          provided := {label | input.review.object.metadata.labels[label]}
          required := {label | label := input.parameters.labels[_]}
          missing := required - provided
          count(missing) > 0
          msg := sprintf("Missing labels: %v", [missing])
        }
---
# 2. Constraint (separate resource)
apiVersion: constraints.gatekeeper.sh/v1beta1
kind: K8sRequiredLabels
metadata:
  name: require-team-label
spec:
  match:
    kinds:
      - apiGroups: [""]
        kinds: ["Namespace"]
  parameters:
    labels: ["team"]
```

### Kyverno Way (Just YAML — ONE resource)

```yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: require-team-label
spec:
  validationFailureAction: Enforce
  rules:
    - name: check-team-label
      match:
        any:
          - resources:
              kinds:
                - Namespace
      validate:
        message: "The label 'team' is required."
        pattern:
          metadata:
            labels:
              team: "?*"    # "?*" means "must exist and not be empty"
```

See the difference? With Kyverno:
- **One resource** instead of two
- **Pure YAML** instead of Rego
- **Readable** even if you've never seen it before
- The **pattern matching** is intuitive — it literally mirrors the structure of the resource you're checking

## 3. Kyverno vs OPA/Gatekeeper — Side by Side

| Feature | OPA/Gatekeeper | Kyverno |
|---------|---------------|---------|
| **Policy Language** | Rego (custom language) | YAML (Kubernetes-native) |
| **Learning Curve** | Steep (learn Rego) | Gentle (just YAML) |
| **Resources Needed** | 2 per policy (Template + Constraint) | 1 per policy (ClusterPolicy/Policy) |
| **Validate** | ✅ Yes | ✅ Yes |
| **Mutate** | ✅ Yes (separate resource) | ✅ Yes (in same policy) |
| **Generate** | ❌ No | ✅ Yes! (create resources automatically) |
| **Image Verification** | ❌ No (need external tools) | ✅ Built-in |
| **Audit / Background Scan** | ✅ Yes | ✅ Yes |
| **Flexibility** | Very high (Rego is Turing-complete) | High (but YAML has limits) |
| **Complex Logic** | Better for very complex logic | Great for 90% of use cases |
| **Community** | CNCF Graduated | CNCF Incubating |
| **Best For** | Complex/enterprise policies, multi-system OPA usage | Teams preferring YAML, Kubernetes-focused policies |

### When to Choose Kyverno

- Your team is Kubernetes-focused and loves YAML
- You want to get started quickly without learning a new language
- You need generate policies (auto-create resources)
- You want built-in image verification
- 90% of your policies are standard validations/mutations

### When to Choose Gatekeeper

- You already use OPA in other systems (APIs, Terraform, etc.)
- You have very complex policy logic that needs a full programming language
- Your organization has invested in Rego expertise

```
                   Developer: kubectl apply -f deployment.yaml
                                       │
                                       ▼
                               ┌──────────────┐
                               │  API Server  │
                               └──────┬───────┘
                                      │
                 ┌────────────────────┼────────────────────┐
                 │                    │                     │
                 │    Webhooks        │                     │
                 │    ┌───────────────┼──────────────┐      │
                 │    │               ▼              │      │
                 │    │  ┌──────────────────────┐    │      │
                 │    │  │   Mutating Webhook   │    │      │
                 │    │  │   (Mutate policies)  │    │      │
                 │    │  └──────────┬───────────┘    │      │
                 │    │             │                │      │
                 │    │             ▼                │      │
                 │    │  ┌──────────────────────┐    │      │
                 │    │  │  Validating Webhook  │    │      │
                 │    │  │ (Validate policies)  │    │      │
                 │    │  └──────────┬───────────┘    │      │
                 │    └─────────────┼────────────────┘      │
                 │                  │                        │
                 │             ┌────┴────┐                  │
                 │             │         │                  │
                 │          ✅ Allow   ❌ Deny               │
                 └────────────────────────────────────────┘
                 
                 ┌──────────────────────────────────────────┐
                 │         Background Controller            │
                 │  (Scans existing resources periodically) │
                 │  (Handles generate policies)             │
                 └──────────────────────────────────────────┘
```

### Kyverno Components

| Component | What It Does |
|-----------|-------------|
| **Admission Controller** | Intercepts API requests (create/update/delete) and applies validate + mutate policies |
| **Background Controller** | Scans existing resources against policies, handles generate policies |
| **Reports Controller** | Creates policy reports (audit results) |
| **Cleanup Controller** | Handles cleanup policies (auto-delete resources) |

## 6. Core Concepts — Policy and ClusterPolicy

Kyverno has two types of policy resources:

| Resource | Scope | What It Means |
|----------|-------|---------------|
| **ClusterPolicy** | Cluster-wide | Applies to resources across ALL namespaces (or non-namespaced resources) |
| **Policy** | Namespace-scoped | Applies only to resources in a specific namespace |

### Basic Structure of a Kyverno Policy

```yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy              # or "Policy" for namespace-scoped
metadata:
  name: my-policy
  annotations:
    policies.kyverno.io/title: "My Policy Title"
    policies.kyverno.io/description: "What this policy does"
    policies.kyverno.io/category: "Best Practices"
    policies.kyverno.io/severity: "medium"
spec:
  validationFailureAction: Enforce    # Enforce or Audit
  background: true                    # Scan existing resources? (default: true)
  rules:
    - name: rule-name
      match:                          # WHAT resources to check
        any:
          - resources:
              kinds:
                - Pod
              namespaces:
                - production
      exclude:                        # WHAT to skip
        any:
          - resources:
              namespaces:
                - kube-system
      validate:                       # The actual check (or mutate/generate)
        message: "Error message shown to users"
        pattern:
          # Pattern matching goes here
```

### Key Fields

| Field | Values | What It Does |
|-------|--------|-------------|
| `validationFailureAction` | `Enforce` | Block resources that violate the policy |
| `validationFailureAction` | `Audit` | Allow but report violations (like a dry-run) |
| `background` | `true/false` | Whether to scan existing resources |
| `match` | resources, namespaces, labels, etc. | Which resources the policy applies to |
| `exclude` | Same as match | Which resources to skip |

## 7. Policy Types — Validate, Mutate, Generate, VerifyImages

Kyverno supports four types of rules in a single policy:

```
┌────────────────────────────────────────────────────┐
│                  Kyverno Policy                    │
│                                                    │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐         │
│  │ Validate │  │  Mutate  │  │ Generate │         │
│  │          │  │          │  │          │         │
│  │ "Is this │  │ "Change  │  │ "Create  │         │
│  │  OK?"    │  │  this."  │  │  that."  │         │
│  └──────────┘  └──────────┘  └──────────┘         │
│                                                    │
│  ┌──────────────┐                                  │
│  │ VerifyImages │                                  │
│  │              │                                  │
│  │ "Is this     │                                  │
│  │  image       │                                  │
│  │  signed?"    │                                  │
│  └──────────────┘                                  │
└────────────────────────────────────────────────────┘
```

| Type | What It Does | Example |
|------|-------------|---------|
| **validate** | Allow or deny based on conditions | "Pods must have resource limits" |
| **mutate** | Automatically modify resources | "Add default labels to all Pods" |
| **generate** | Create new resources automatically | "Create a NetworkPolicy for every new Namespace" |
| **verifyImages** | Check image signatures/attestations | "Only allow signed images from Cosign" |

## 8. Validate Policies — Hands-On Examples

### Example 1: Require Labels

```yaml
# require-labels.yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: require-labels
  annotations:
    policies.kyverno.io/title: "Require Labels"
    policies.kyverno.io/description: "All resources must have 'team' and 'app' labels."
    policies.kyverno.io/severity: "medium"
spec:
  validationFailureAction: Enforce
  background: true
  rules:
    - name: require-team-label
      match:
        any:
          - resources:
              kinds:
                - Deployment
                - StatefulSet
                - Service
      exclude:
        any:
          - resources:
              namespaces:
                - kube-system
                - kyverno
      validate:
        message: "The labels 'team' and 'app' are required. Add them to metadata.labels."
        pattern:
          metadata:
            labels:
              team: "?*"      # Must exist and not be empty
              app: "?*"       # Must exist and not be empty
```

**How pattern matching works:**
- `"?*"` → Must exist and have at least one character (not empty)
- `"*"` → Must exist (can be empty)
- `"value"` → Must be exactly "value"
- `"!value"` → Must NOT be "value"
- `"value1 | value2"` → Must be "value1" OR "value2"

```bash
kubectl apply -f require-labels.yaml
```

**Test it:**

```bash
# This will be BLOCKED
kubectl create deployment nginx --image=nginx:1.25

# Error: "The labels 'team' and 'app' are required."
```

```yaml
# This will WORK
# good-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
  labels:
    team: platform
    app: nginx
spec:
  replicas: 1
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
        team: platform
    spec:
      containers:
        - name: nginx
          image: nginx:1.25
```

### Example 2: Disallow `latest` Tag

```yaml
# disallow-latest-tag.yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: disallow-latest-tag
  annotations:
    policies.kyverno.io/title: "Disallow Latest Tag"
    policies.kyverno.io/description: "Container images must have a tag that is not 'latest'."
    policies.kyverno.io/severity: "high"
spec:
  validationFailureAction: Enforce
  rules:
    - name: validate-image-tag
      match:
        any:
          - resources:
              kinds:
                - Pod
      validate:
        message: >-
          An image tag is required. Using 'latest' is not allowed.
          Use a specific version tag like 'nginx:1.25'.
        pattern:
          spec:
            containers:
              - image: "*:*"       # Must have format "image:tag"
                                    # This blocks bare images like "nginx"
            =(initContainers):     # "=(" means "if exists, check this"
              - image: "*:*"
    - name: validate-not-latest
      match:
        any:
          - resources:
              kinds:
                - Pod
      validate:
        message: "The 'latest' tag is not allowed. Use a specific version."
        pattern:
          spec:
            containers:
              - image: "!*:latest"  # Must NOT end with ":latest"
            =(initContainers):
              - image: "!*:latest"
```

**Pattern magic explained:**
- `"*:*"` → Image must contain a colon (i.e., must have a tag)
- `"!*:latest"` → Image must NOT match anything ending with `:latest`
- `=(initContainers)` → Only check init containers **if they exist** (optional field)

### Example 4: Restrict to Allowed Registries

```yaml
# allowed-registries.yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: allowed-registries
  annotations:
    policies.kyverno.io/title: "Restrict Image Registries"
    policies.kyverno.io/severity: "high"
spec:
  validationFailureAction: Enforce
  rules:
    - name: validate-registries
      match:
        any:
          - resources:
              kinds:
                - Pod
      validate:
        message: >-
          Images must come from an approved registry.
          Allowed registries: mycompany.azurecr.io, gcr.io/my-project
        pattern:
          spec:
            containers:
              - image: "mycompany.azurecr.io/* | gcr.io/my-project/*"
            =(initContainers):
              - image: "mycompany.azurecr.io/* | gcr.io/my-project/*"
```

The `|` operator means OR — the image must match one of the allowed patterns.

### Example 5: Using `deny` with Conditions (Advanced)

For more complex validation that patterns can't handle, use `deny` with `conditions`:

```yaml
# deny-nodeport.yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: deny-nodeport
  annotations:
    policies.kyverno.io/title: "Disallow NodePort Services"
spec:
  validationFailureAction: Enforce
  rules:
    - name: no-nodeport
      match:
        any:
          - resources:
              kinds:
                - Service
      validate:
        message: "NodePort services are not allowed. Use ClusterIP or LoadBalancer instead."
        deny:
          conditions:
            any:
              - key: "{{ request.object.spec.type }}"
                operator: Equals
                value: NodePort
```

**The `{{ }}` syntax** — These are **JMESPath expressions** (Kyverno's way of accessing resource fields). Think of them like variables:

| Expression | What It Returns |
|-----------|----------------|
| `{{ request.object.metadata.name }}` | The resource's name |
| `{{ request.object.metadata.namespace }}` | The resource's namespace |
| `{{ request.object.spec.containers[0].image }}` | First container's image |
| `{{ request.object.metadata.labels.team }}` | The "team" label value |

## 9. Mutate Policies — Hands-On Examples

Mutation policies **automatically modify** resources. This is incredibly useful for setting defaults, adding labels, or injecting sidecars.

### Example 1: Add Default Labels

```yaml
# add-default-labels.yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: add-default-labels
spec:
  rules:
    - name: add-managed-by
      match:
        any:
          - resources:
              kinds:
                - Deployment
                - StatefulSet
                - DaemonSet
      mutate:
        patchStrategicMerge:
          metadata:
            labels:
              +(managed-by): kyverno    # "+" means "add only if not exists"
```

**Mutation operators:**
- `+(key): value` → Add the field only if it doesn't exist (won't overwrite)
- `-(key): value` → Remove the field
- No prefix → Always set this value (will overwrite)

---

### Example 2: Set Default Resource Requests

```yaml
# default-resources.yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: set-default-resources
spec:
  rules:
    - name: set-cpu-memory-defaults
      match:
        any:
          - resources:
              kinds:
                - Pod
      exclude:
        any:
          - resources:
              namespaces:
                - kube-system
                - kyverno
      mutate:
        patchStrategicMerge:
          spec:
            containers:
              - (name): "*"     # Match ALL containers (wildcard)
                resources:
                  requests:
                    +(memory): "64Mi"     # Add default memory request if not set
                    +(cpu): "100m"        # Add default CPU request if not set
                  limits:
                    +(memory): "128Mi"    # Add default memory limit if not set
                    +(cpu): "250m"        # Add default CPU limit if not set
```

This is powerful! If a developer forgets to set resource requests/limits, Kyverno automatically adds sensible defaults.

### Example 3: Set Image Pull Policy Based on Tag

```yaml
# set-image-pull-policy.yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: set-image-pull-policy
spec:
  rules:
    - name: set-always-for-latest
      match:
        any:
          - resources:
              kinds:
                - Pod
      preconditions:
        any:
          - key: "{{ request.object.spec.containers[].image }}"
            operator: AnyIn
            value: ["*:latest"]
      mutate:
        patchStrategicMerge:
          spec:
            containers:
              - (image): "*:latest"
                imagePullPolicy: Always
```

If someone uses a `latest` tag (which we should block — but if we allow it), at least ensure the pull policy is `Always`.

### Example 4: Inject a Sidecar Container

```yaml
# inject-logging-sidecar.yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: inject-logging-sidecar
spec:
  rules:
    - name: inject-fluentbit
      match:
        any:
          - resources:
              kinds:
                - Pod
              selector:
                matchLabels:
                  inject-logging: "true"    # Only Pods with this label
      mutate:
        patchStrategicMerge:
          spec:
            containers:
              - name: fluentbit-sidecar
                image: fluent/fluent-bit:2.2
                resources:
                  requests:
                    cpu: "50m"
                    memory: "32Mi"
                  limits:
                    cpu: "100m"
                    memory: "64Mi"
                volumeMounts:
                  - name: app-logs
                    mountPath: /var/log/app
            volumes:
              - name: app-logs
                emptyDir: {}
```

Now any Pod with the label `inject-logging: "true"` automatically gets a Fluent Bit sidecar!

## 10. Generate Policies — Hands-On Examples

This is Kyverno's **superpower** that Gatekeeper doesn't have. Generate policies automatically **create resources** when something happens.

### Example 1: Auto-Create NetworkPolicy for New Namespaces

```yaml
# generate-networkpolicy.yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: default-networkpolicy
spec:
  rules:
    - name: generate-default-deny
      match:
        any:
          - resources:
              kinds:
                - Namespace
      exclude:
        any:
          - resources:
              names:
                - kube-system
                - kube-public
                - kyverno
      generate:
        synchronize: true       # Keep the generated resource in sync
        apiVersion: networking.k8s.io/v1
        kind: NetworkPolicy
        name: default-deny-ingress
        namespace: "{{ request.object.metadata.name }}"
        data:
          spec:
            podSelector: {}     # Apply to all pods in namespace
            policyTypes:
              - Ingress         # Deny all ingress by default
```

**What happens:**
1. Someone creates a new namespace: `kubectl create namespace my-app`
2. Kyverno **automatically** creates a `default-deny-ingress` NetworkPolicy in that namespace
3. If `synchronize: true`, Kyverno keeps the NetworkPolicy in sync — if someone deletes it, Kyverno recreates it!

### Example 2: Auto-Create ResourceQuota for New Namespaces

```yaml
# generate-quota.yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: default-resource-quota
spec:
  rules:
    - name: generate-quota
      match:
        any:
          - resources:
              kinds:
                - Namespace
      exclude:
        any:
          - resources:
              names:
                - kube-system
                - kube-public
                - kyverno
      generate:
        synchronize: true
        apiVersion: v1
        kind: ResourceQuota
        name: default-quota
        namespace: "{{ request.object.metadata.name }}"
        data:
          spec:
            hard:
              requests.cpu: "4"
              requests.memory: "8Gi"
              limits.cpu: "8"
              limits.memory: "16Gi"
              pods: "20"
              services: "10"
              persistentvolumeclaims: "5"
```

Every new namespace automatically gets resource quotas! No manual work needed.

---

### Example 3: Auto-Create RBAC for New Namespaces

```yaml
# generate-rbac.yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: default-rbac
spec:
  rules:
    - name: generate-admin-rolebinding
      match:
        any:
          - resources:
              kinds:
                - Namespace
              selector:
                matchLabels:
                  team: "?*"        # Only if namespace has a "team" label
      generate:
        synchronize: true
        apiVersion: rbac.authorization.k8s.io/v1
        kind: RoleBinding
        name: team-admin
        namespace: "{{ request.object.metadata.name }}"
        data:
          roleRef:
            apiGroup: rbac.authorization.k8s.io
            kind: ClusterRole
            name: admin
          subjects:
            - kind: Group
              name: "team-{{ request.object.metadata.labels.team }}"
              apiGroup: rbac.authorization.k8s.io
```

When a namespace with label `team: platform` is created, Kyverno auto-creates a RoleBinding that gives the `team-platform` group admin access to that namespace. This is incredibly powerful for multi-tenant clusters!

## 11. Image Verification Policies

Kyverno can verify that container images are **signed** using tools like **Cosign** (from the Sigstore project).

```yaml
# verify-images.yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: verify-image-signatures
spec:
  validationFailureAction: Enforce
  rules:
    - name: verify-cosign-signature
      match:
        any:
          - resources:
              kinds:
                - Pod
      verifyImages:
        - imageReferences:
            - "myregistry.com/*"
          attestors:
            - entries:
                - keys:
                    publicKeys: |-
                      -----BEGIN PUBLIC KEY-----
                      MFkwEwYHKoZIzj0CAQYIKoZIzj0DAQcDQgAE...
                      -----END PUBLIC KEY-----
```

This ensures that only images signed with the specified public key are allowed. If someone tries to deploy an unsigned image from `myregistry.com`, it's blocked!

## 12. Policy Exceptions

Sometimes certain workloads need to be **exempt** from policies. Kyverno provides a clean way to do this:

```yaml
# exception.yaml
apiVersion: kyverno.io/v2beta1
kind: PolicyException
metadata:
  name: allow-monitoring-privileged
  namespace: monitoring
spec:
  exceptions:
    - policyName: disallow-privileged-containers
      ruleNames:
        - check-privileged
  match:
    any:
      - resources:
          kinds:
            - Pod
          namespaces:
            - monitoring
          names:
            - "prometheus-*"
            - "node-exporter-*"
```

This says: "Exempt Pods named `prometheus-*` and `node-exporter-*` in the `monitoring` namespace from the `disallow-privileged-containers` policy."

**Why use PolicyExceptions instead of just excluding in the policy?**
- Separation of concerns — the policy stays clean
- You can audit who requested exceptions
- Exceptions can be managed by different teams
- Easy to revoke (just delete the exception)

## 13. Background Scanning — Auditing Existing Resources

Kyverno doesn't just check new resources — it continuously scans **existing** resources too.

### How It Works

When `background: true` is set in a policy (default), the Background Controller periodically scans all matching resources and generates **PolicyReports**.

### Viewing Policy Reports

```bash
# View cluster-level policy reports
kubectl get clusterpolicyreport

# View namespace-level policy reports
kubectl get policyreport -A

# Detailed view
kubectl get policyreport -n default -o yaml
```

### Policy Report Structure

```yaml
apiVersion: wgpolicyk8s.io/v1alpha2
kind: PolicyReport
metadata:
  name: polr-ns-default
  namespace: default
results:
  - message: "The label 'team' is required."
    policy: require-labels
    rule: require-team-label
    result: fail                 # pass, fail, warn, error, skip
    resources:
      - apiVersion: apps/v1
        kind: Deployment
        name: my-deployment
        namespace: default
summary:
  pass: 12
  fail: 3
  warn: 0
  error: 0
  skip: 0
```

### Audit Mode vs Enforce Mode

```yaml
# Audit mode — allows but reports
spec:
  validationFailureAction: Audit

# Enforce mode — blocks and reports
spec:
  validationFailureAction: Enforce
```

**Pro tip:** Start with `Audit` to see the impact, then switch to `Enforce`:

```bash
# Check how many violations exist before enforcing
kubectl get policyreport -A -o json | \
  jq '.items[].summary.fail' | \
  paste -sd+ | bc
```

## 16. Advanced Features

### 16.1 — Cleanup Policies

Kyverno can automatically **delete** resources based on conditions:

```yaml
# cleanup-failed-pods.yaml
apiVersion: kyverno.io/v2beta1
kind: ClusterCleanupPolicy
metadata:
  name: cleanup-failed-pods
spec:
  match:
    any:
      - resources:
          kinds:
            - Pod
  conditions:
    any:
      - key: "{{ target.status.phase }}"
        operator: Equals
        value: Failed
  schedule: "*/15 * * * *"    # Every 15 minutes
```

This automatically cleans up failed Pods every 15 minutes!

---

### 16.2 — foreach — Iterate Over Lists

When you need to check every item in a list:

```yaml
# check-all-containers.yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: check-container-security
spec:
  validationFailureAction: Enforce
  rules:
    - name: check-security-context
      match:
        any:
          - resources:
              kinds:
                - Pod
      validate:
        message: "All containers must set allowPrivilegeEscalation to false."
        foreach:
          - list: "request.object.spec.containers"
            deny:
              conditions:
                any:
                  - key: "{{ element.securityContext.allowPrivilegeEscalation || 'true' }}"
                    operator: Equals
                    value: "true"
```

`foreach` iterates over `spec.containers` and checks each one individually. The current item is accessible as `element`.

---

### 16.3 — Context — External Data Lookups

Kyverno can pull data from the Kubernetes API or external sources:

```yaml
# check-namespace-quota.yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: check-existing-quota
spec:
  rules:
    - name: verify-quota-exists
      match:
        any:
          - resources:
              kinds:
                - Deployment
      context:
        - name: quotas
          apiCall:
            urlPath: "/api/v1/namespaces/{{ request.namespace }}/resourcequotas"
            jmesPath: "items | length(@)"
      validate:
        message: "Cannot create Deployments in a namespace without ResourceQuota."
        deny:
          conditions:
            any:
              - key: "{{ quotas }}"
                operator: Equals
                value: 0
```

This checks if the namespace has a ResourceQuota before allowing a Deployment!

---

### 16.4 — Chaining Rules

A single policy can have multiple rules that work together:

```yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: comprehensive-pod-security
spec:
  validationFailureAction: Enforce
  rules:
    - name: check-labels
      match:
        any:
          - resources:
              kinds: ["Pod"]
      validate:
        message: "Label 'app' is required."
        pattern:
          metadata:
            labels:
              app: "?*"
    
    - name: check-resources
      match:
        any:
          - resources:
              kinds: ["Pod"]
      validate:
        message: "Resource limits are required."
        pattern:
          spec:
            containers:
              - resources:
                  limits:
                    memory: "?*"
                    cpu: "?*"
    
    - name: disallow-root
      match:
        any:
          - resources:
              kinds: ["Pod"]
      validate:
        message: "Running as root is not allowed."
        pattern:
          spec:
            securityContext:
              runAsNonRoot: true
```

## 17. Best Practices

### 1. Start with Audit, Then Enforce

```yaml
# Phase 1: Audit
validationFailureAction: Audit

# Phase 2: Review policy reports
kubectl get policyreport -A

# Phase 3: Fix violations

# Phase 4: Enforce
validationFailureAction: Enforce
```

### 2. Use Annotations for Documentation

```yaml
metadata:
  annotations:
    policies.kyverno.io/title: "Require Resource Limits"
    policies.kyverno.io/description: "All containers must have CPU and memory limits."
    policies.kyverno.io/category: "Best Practices"
    policies.kyverno.io/severity: "high"          # low, medium, high, critical
    policies.kyverno.io/subject: "Pod"
    policies.kyverno.io/minversion: "1.6.0"
```

### 3. Always Exclude System Namespaces

```yaml
exclude:
  any:
    - resources:
        namespaces:
          - kube-system
          - kube-public
          - kube-node-lease
          - kyverno
```

### 4. Version Control Your Policies

Store all policies in Git and deploy via CI/CD or GitOps (ArgoCD/Flux).

### 5. Use the Kyverno Policy Library

Kyverno has a huge library of pre-built policies: `https://kyverno.io/policies/`

Categories include:
- Best Practices
- Pod Security Standards (PSS) equivalents
- Multi-tenancy
- Supply Chain Security
- EKS/AKS/GKE specific

### 6. Test Policies in CI/CD

```bash
# In your pipeline
kyverno test policies/
kyverno apply policies/ --resource manifests/ --detailed-results
```

### 7. Use Mutation to Set Sensible Defaults

Instead of just blocking Pods without resource limits, **auto-set defaults**:

```yaml
# Better UX: mutate first, then validate
rules:
  - name: set-defaults       # First: add defaults
    mutate: ...
  - name: validate-limits    # Then: validate
    validate: ...
```

### 8. Monitor with Policy Reporter

Always set up Policy Reporter for visibility into policy compliance across your cluster.

## 18. Common Pitfalls & Troubleshooting

### Pitfall 1: "Policy Not Working"

**Check order:**

```bash
# 1. Is the policy applied?
kubectl get clusterpolicy

# 2. Is it in the right state?
kubectl get clusterpolicy <name> -o yaml | grep -A5 status

# 3. Are the webhooks registered?
kubectl get validatingwebhookconfigurations | grep kyverno
kubectl get mutatingwebhookconfigurations | grep kyverno

# 4. Check Kyverno logs
kubectl logs -n kyverno -l app.kubernetes.io/component=admission-controller
```

### Pitfall 2: "Existing Resources Still Violating"

PSA only checks on create/update. For existing resources:
- Set `background: true` in the policy (default)
- Check policy reports: `kubectl get policyreport -A`
- Trigger a rollout: `kubectl rollout restart deployment -n <ns> <name>`

### Pitfall 3: "Pattern Not Matching"

Common issues with patterns:
- `"?*"` requires **at least one character** (empty string fails)
- `"*"` matches anything **including empty** 
- Anchors: `=(field)` means "if exists"; `(field)` is a regular match

### Pitfall 4: "Kyverno Webhook Blocking Everything"

If Kyverno pods crash, the webhook might block all requests.

**Emergency fix:**

```bash
# Option 1: Set failure policy to Ignore
helm upgrade kyverno kyverno/kyverno --set config.webhooks.failurePolicy=Ignore

# Option 2: Remove webhooks temporarily
kubectl delete validatingwebhookconfiguration kyverno-resource-validating-webhook-cfg
kubectl delete mutatingwebhookconfiguration kyverno-resource-mutating-webhook-cfg
```

### Pitfall 5: "Cannot Generate Resources — Permission Denied"

Kyverno needs RBAC permissions to create the resources specified in generate policies:

```bash
# Check Kyverno's cluster role
kubectl get clusterrole kyverno:admission-controller -o yaml
```

Make sure it has permissions for the resource types you're trying to generate.

## 19. Summary

| Concept | What It Is |
|---------|-----------|
| **Kyverno** | Kubernetes-native policy engine — policies written in YAML |
| **ClusterPolicy** | Cluster-wide policy (all namespaces) |
| **Policy** | Namespace-scoped policy |
| **Validate** | Allow/deny resources based on rules (pattern or deny+conditions) |
| **Mutate** | Automatically modify resources (add defaults, labels, sidecars) |
| **Generate** | Auto-create resources (NetworkPolicy, RBAC, Quota per namespace) |
| **VerifyImages** | Check container image signatures (Cosign/Notary) |
| **Pattern Matching** | `"?*"` (non-empty), `"!value"` (not this), `"a | b"` (either) |
| **PolicyException** | Exempt specific workloads from policies |
| **Policy Reports** | Audit results for existing resources |
| **Kyverno CLI** | Test policies locally / in CI/CD |
| **Policy Reporter** | UI + metrics for policy compliance |
| **Cleanup Policies** | Auto-delete resources on a schedule |

### Key Takeaway

> Kyverno is the **"just YAML"** policy engine for Kubernetes. It can **validate** (block bad configs), **mutate** (auto-fix resources), **generate** (auto-create resources), and **verify images** (supply chain security) — all using familiar Kubernetes YAML. No new languages needed!
