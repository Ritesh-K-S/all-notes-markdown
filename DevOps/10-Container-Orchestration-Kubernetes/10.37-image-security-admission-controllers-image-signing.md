## 🤔 Why Should You Care About Image Security?

Let's say you have a Kubernetes cluster running your company's app. Someone on your team (or a CI/CD pipeline) deploys a new image:

```
mycompany/payment-service:latest
```

But what if:

- That image was **built from a compromised base image** with malware? 🦠
- Someone **pushed a malicious image** to your registry pretending to be legit? 🎭
- A hacker performed a **man-in-the-middle attack** and swapped the image while it was being pulled? 🕵️
- Your developer accidentally deployed `nginx:latest` from **Docker Hub** instead of your **private registry**? 😱

Without image security:
- **Anyone** can deploy **anything** into your cluster
- You have **no proof** that the image is the exact same one your CI/CD built
- You're running **untrusted code** in production

That's a nightmare. Let's fix it! 🛡️

## 🚪 Part 1: Admission Controllers — The Bouncers of Kubernetes

### What is an Admission Controller?

An **Admission Controller** is a piece of code that intercepts requests to the **Kubernetes API server** BEFORE an object (like a Pod) is created — and decides whether to **allow**, **deny**, or **modify** it.

Think of it like a **bouncer at a nightclub**:

```
You (Developer) ──→ "I want to create a Pod with image X"
                          │
                          ▼
                 ┌──────────────────┐
                 │  API Server      │
                 │                  │
                 │  1. Authentication│ ← Are you who you say you are?
                 │  2. Authorization │ ← Are you ALLOWED to do this?
                 │  3. ADMISSION    │ ← Should we ACTUALLY let this through?
                 │     CONTROLLERS  │   ← THIS IS WHERE THE MAGIC HAPPENS
                 │                  │
                 │  4. Persist to   │
                 │     etcd         │
                 └──────────────────┘
```

Even if you're **authenticated** (you are who you say) and **authorized** (you have RBAC permissions), the Admission Controller can still say **"NOPE!"** 🚫

### Types of Admission Controllers

There are **two types**:

| Type | What It Does | Example |
|------|-------------|---------|
| **Validating** | Checks the request and **accepts or rejects** it. Doesn't change anything. | "Reject any Pod using `latest` tag" |
| **Mutating** | **Modifies** the request before it's persisted. Can add/change fields. | "Automatically add `imagePullPolicy: Always`" |

The order is: **Mutating → Validating** (mutate first, then validate the final result).

### Built-in Admission Controllers

Kubernetes has **many built-in** admission controllers. Here are the most important ones for image security:

| Controller | What It Does |
|-----------|-------------|
| `AlwaysPullImages` | Forces `imagePullPolicy: Always` on every Pod — so images are always pulled fresh from the registry (prevents using stale/cached images) |
| `ImagePolicyWebhook` | Calls an external webhook to decide if an image is allowed |
| `PodSecurity` | Enforces Pod Security Standards (Baseline, Restricted) |
| `ValidatingAdmissionWebhook` | Generic webhook — you define ANY validation logic |
| `MutatingAdmissionWebhook` | Generic webhook — you define ANY mutation logic |

### How to See Which Admission Controllers Are Enabled

```bash
# On a kubeadm cluster, check the API server manifest:
cat /etc/kubernetes/manifests/kube-apiserver.yaml | grep enable-admission-plugins

# Typical output:
# --enable-admission-plugins=NodeRestriction,MutatingAdmissionWebhook,ValidatingAdmissionWebhook
```

### Enabling the AlwaysPullImages Admission Controller

This is the **simplest image security measure**. It ensures that every Pod always pulls the image from the registry instead of using a cached version.

```yaml
# In /etc/kubernetes/manifests/kube-apiserver.yaml, add to the command:
- --enable-admission-plugins=NodeRestriction,AlwaysPullImages
```

**Why is this important?**
- Without it, if an image is cached on a node, ANY Pod on that node can use it — even if it doesn't have registry credentials
- With it, every Pod must authenticate to pull the image — so only authorized Pods can use it

### 🔌 Webhook Admission Controllers (The Real Power)

The built-in controllers are useful, but the **real power** comes from **webhooks** — you can write your OWN admission logic!

#### How Webhook Admission Controllers Work

```
Developer → kubectl apply → API Server
                                │
                    ┌───────────┴───────────┐
                    │ Mutating Webhooks     │  ← Called first
                    │ (can modify the Pod)  │
                    └───────────┬───────────┘
                                │
                    ┌───────────┴───────────┐
                    │ Validating Webhooks   │  ← Called second
                    │ (accept/reject only)  │
                    └───────────┬───────────┘
                                │
                          Save to etcd
```

#### Example: Creating a Validating Webhook

Let's say you want to **reject any Pod that uses an image NOT from your private registry**.

**Step 1: Write a webhook server** (a simple HTTP server that receives admission requests):

```python
# webhook-server.py (simplified example using Flask)
from flask import Flask, request, jsonify
import json

app = Flask(__name__)

ALLOWED_REGISTRIES = ["mycompany.azurecr.io", "gcr.io/myproject"]

@app.route('/validate', methods=['POST'])
def validate():
    admission_review = request.get_json()
    pod_spec = admission_review['request']['object']['spec']
    
    for container in pod_spec.get('containers', []):
        image = container['image']
        # Check if the image is from an allowed registry
        allowed = any(image.startswith(reg) for reg in ALLOWED_REGISTRIES)
        if not allowed:
            return jsonify({
                "apiVersion": "admission.k8s.io/v1",
                "kind": "AdmissionReview",
                "response": {
                    "uid": admission_review['request']['uid'],
                    "allowed": False,
                    "status": {
                        "message": f"Image '{image}' is not from an allowed registry. "
                                   f"Allowed: {ALLOWED_REGISTRIES}"
                    }
                }
            })
    
    # All images are from allowed registries
    return jsonify({
        "apiVersion": "admission.k8s.io/v1",
        "kind": "AdmissionReview",
        "response": {
            "uid": admission_review['request']['uid'],
            "allowed": True
        }
    })

if __name__ == '__main__':
    # In production, use TLS! Webhooks MUST use HTTPS.
    app.run(host='0.0.0.0', port=443, ssl_context=('cert.pem', 'key.pem'))
```

**What this code does:**
- It's a simple web server that listens for admission review requests from the Kubernetes API server
- When someone tries to create a Pod, the API server sends the Pod spec to this webhook
- The webhook checks if all container images come from allowed registries
- If any image is NOT from an allowed registry, it rejects the request with a clear error message
- If all images are fine, it allows the request

**Step 2: Deploy the webhook server as a Service in Kubernetes:**

```yaml
# webhook-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: image-policy-webhook
  namespace: webhook-system
spec:
  replicas: 2
  selector:
    matchLabels:
      app: image-policy-webhook
  template:
    metadata:
      labels:
        app: image-policy-webhook
    spec:
      containers:
      - name: webhook
        image: mycompany.azurecr.io/image-policy-webhook:v1.0
        ports:
        - containerPort: 443
        volumeMounts:
        - name: tls-certs
          mountPath: /certs
          readOnly: true
      volumes:
      - name: tls-certs
        secret:
          secretName: webhook-tls
---
apiVersion: v1
kind: Service
metadata:
  name: image-policy-webhook
  namespace: webhook-system
spec:
  selector:
    app: image-policy-webhook
  ports:
  - port: 443
    targetPort: 443
```

**Step 3: Register the webhook with Kubernetes:**

```yaml
# validating-webhook-config.yaml
apiVersion: admissionregistration.k8s.io/v1
kind: ValidatingWebhookConfiguration
metadata:
  name: image-policy-webhook
webhooks:
- name: image-policy.mycompany.com
  clientConfig:
    service:
      name: image-policy-webhook
      namespace: webhook-system
      path: /validate
    caBundle: <BASE64_ENCODED_CA_CERT>   # CA cert that signed the webhook's TLS cert
  rules:
  - apiGroups: [""]
    apiVersions: ["v1"]
    operations: ["CREATE", "UPDATE"]
    resources: ["pods"]
  admissionReviewVersions: ["v1"]
  sideEffects: None
  failurePolicy: Fail    # If webhook is down, REJECT all requests (safe default)
  # failurePolicy: Ignore  # If webhook is down, ALLOW all requests (less safe)
```

**Key fields explained:**

| Field | What It Means |
|-------|--------------|
| `clientConfig.service` | Where the webhook server lives (Kubernetes Service) |
| `clientConfig.caBundle` | The CA certificate to verify the webhook's TLS certificate |
| `rules` | What resources and operations trigger this webhook |
| `failurePolicy: Fail` | If the webhook is unreachable, DENY everything (safe but strict) |
| `failurePolicy: Ignore` | If the webhook is unreachable, ALLOW everything (risky) |
| `sideEffects: None` | This webhook doesn't modify anything outside the request |

#### Apply it:

```bash
kubectl apply -f validating-webhook-config.yaml
```

#### Test it:

```bash
# This should be REJECTED (image not from allowed registry):
kubectl run bad-pod --image=nginx:latest
# Error: Image 'nginx:latest' is not from an allowed registry.

# This should be ALLOWED:
kubectl run good-pod --image=mycompany.azurecr.io/nginx:1.25
# pod/good-pod created ✅
```

### Using OPA Gatekeeper or Kyverno for Image Policies

Instead of writing your own webhook server, you can use **policy engines** (covered in 10.35 and 10.36) to create image policies:

#### Kyverno Example — Block images without tags:

```yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: disallow-latest-tag
spec:
  validationFailureAction: Enforce
  rules:
  - name: require-image-tag
    match:
      any:
      - resources:
          kinds:
          - Pod
    validate:
      message: "An image tag is required. Using ':latest' is not allowed."
      pattern:
        spec:
          containers:
          - image: "!*:latest"    # Reject any image ending with :latest
```

**What this does:**
- It creates a cluster-wide policy using Kyverno (no need to write a webhook server!)
- Any Pod that uses an image with the `:latest` tag will be **rejected**
- The `Enforce` action means it will actively block, not just warn

#### OPA Gatekeeper Example — Only allow images from trusted registries:

```yaml
# First, create a ConstraintTemplate
apiVersion: templates.gatekeeper.sh/v1
kind: ConstraintTemplate
metadata:
  name: k8sallowedrepos
spec:
  crd:
    spec:
      names:
        kind: K8sAllowedRepos
      validation:
        openAPIV3Schema:
          type: object
          properties:
            repos:
              type: array
              items:
                type: string
  targets:
  - target: admission.k8s.gatekeeper.sh
    rego: |
      package k8sallowedrepos
      
      violation[{"msg": msg}] {
        container := input.review.object.spec.containers[_]
        not startswith(container.image, input.parameters.repos[_])
        msg := sprintf("Image '%v' is not from an allowed repository", [container.image])
      }
---
# Then, create a Constraint that uses the template
apiVersion: constraints.gatekeeper.sh/v1beta1
kind: K8sAllowedRepos
metadata:
  name: allowed-repos
spec:
  match:
    kinds:
    - apiGroups: [""]
      kinds: ["Pod"]
  parameters:
    repos:
    - "mycompany.azurecr.io/"
    - "gcr.io/myproject/"
```

**What this does:**
- The `ConstraintTemplate` defines the LOGIC (using Rego policy language): check if an image starts with an allowed repo prefix
- The `K8sAllowedRepos` constraint APPLIES that logic and specifies which repos are actually allowed
- Any Pod using an image from outside those repos is blocked

## 🔐 Part 2: Image Signing — Proving Your Images Are Legit

### The Problem Image Signing Solves

Even if you restrict registries, how do you know the image in your registry hasn't been **tampered with**?

```
Your CI/CD builds image → pushes to registry → Kubernetes pulls from registry
                                    ↑
                          🕵️ Attacker modifies the image
                             in the registry!
```

**Image signing** is like putting a **wax seal on a letter** — it proves:
1. **Authenticity** — The image was built by someone you trust
2. **Integrity** — The image hasn't been modified since it was signed

```
Without Signing:           With Signing:
┌─────────────┐           ┌─────────────┐
│   Image     │           │   Image     │
│   v1.0      │           │   v1.0      │
│             │           │ + Signature │ ← Cryptographic proof
│  Anyone     │           │   (Cosign)  │
│  could have │           │             │
│  changed it │           │  Verified!  │
└─────────────┘           └─────────────┘
```

---

### 🖊️ Cosign — The Modern Way to Sign Images

**Cosign** is part of the **Sigstore** project. It's the most popular and easiest way to sign container images today.

#### Why Cosign?

| Feature | Cosign | Old Tools (Docker Content Trust) |
|---------|--------|--------------------------------|
| Easy to use | ✅ Very simple CLI | ❌ Complex setup |
| Keyless signing | ✅ Yes (uses OIDC identity) | ❌ No |
| Works with any OCI registry | ✅ Yes | ❌ Limited |
| Stores signatures in registry | ✅ Yes (as OCI artifacts) | ❌ Separate Notary server |
| Kubernetes integration | ✅ Great (Kyverno, OPA, Sigstore Policy Controller) | ❌ Limited |
| Active development | ✅ Very active | ⚠️ Notary v1 is deprecated |

#### Installing Cosign

```bash
# Linux
curl -LO https://github.com/sigstore/cosign/releases/latest/download/cosign-linux-amd64
chmod +x cosign-linux-amd64
sudo mv cosign-linux-amd64 /usr/local/bin/cosign

# macOS
brew install cosign

# Verify installation
cosign version
```

#### Method 1: Signing with a Key Pair (Traditional)

```bash
# Step 1: Generate a key pair
cosign generate-key-pair

# This creates:
# cosign.key  (private key — KEEP THIS SECRET!)
# cosign.pub  (public key — share this for verification)

# Step 2: Sign an image
cosign sign --key cosign.key myregistry.io/myapp:v1.0

# What happens:
# 1. Cosign computes a digest (hash) of the image
# 2. Signs the digest with your private key
# 3. Pushes the signature to the registry as an OCI artifact
# 4. The signature is stored alongside the image

# Step 3: Verify the signature
cosign verify --key cosign.pub myregistry.io/myapp:v1.0

# Output (if valid):
# Verification for myregistry.io/myapp:v1.0 --
# The following checks were performed on each of these signatures:
#   - The cosign claims were validated
#   - The signatures were verified against the specified public key
```

**How this works behind the scenes:**

```
Signing:
┌──────────┐     ┌──────────────┐     ┌──────────────┐
│  Image   │────▶│  Compute     │────▶│  Sign with   │
│  Digest  │     │  SHA256 Hash │     │  Private Key │
└──────────┘     └──────────────┘     └──────┬───────┘
                                              │
                                              ▼
                                    ┌──────────────┐
                                    │  Push        │
                                    │  Signature   │
                                    │  to Registry │
                                    └──────────────┘

Verification:
┌──────────┐     ┌──────────────┐     ┌──────────────┐
│  Image   │────▶│  Compute     │────▶│  Verify with │
│  Digest  │     │  SHA256 Hash │     │  Public Key  │
└──────────┘     └──────────────┘     └──────┬───────┘
                                              │
                                       ✅ Match! Image is authentic
                                       ❌ No match! Image was tampered with
```

#### Method 2: Keyless Signing (The Cool Way) 🆕

This is the **game changer**. With keyless signing, you don't need to manage keys at all! Instead, you use your **identity** (like your GitHub account or Google account).

```bash
# Just sign — no keys needed!
cosign sign myregistry.io/myapp:v1.0

# What happens:
# 1. Cosign opens a browser for you to log in (OIDC)
# 2. You authenticate with GitHub/Google/Microsoft
# 3. Sigstore's Fulcio CA issues a short-lived certificate
# 4. The image is signed with that certificate
# 5. The signing event is recorded in Rekor (transparency log)
# 6. The certificate expires after ~20 minutes (but the signature is permanent)

# Verify — check that a specific identity signed it
cosign verify \
  --certificate-identity=user@example.com \
  --certificate-oidc-issuer=https://accounts.google.com \
  myregistry.io/myapp:v1.0
```

**Why keyless signing is amazing:**
- **No key management** — no need to protect, rotate, or distribute private keys
- **Identity-based** — you know WHO signed it (e.g., your CI/CD pipeline's service account)
- **Transparency log** — every signing event is publicly recorded and auditable
- **Short-lived certs** — even if compromised, the cert is already expired

#### Method 3: Signing in CI/CD (GitHub Actions Example)

This is how you'd sign images automatically in your pipeline:

```yaml
# .github/workflows/build-and-sign.yaml
name: Build, Push, and Sign Image

on:
  push:
    branches: [main]

permissions:
  contents: read
  id-token: write    # Needed for keyless signing with Sigstore

jobs:
  build-sign:
    runs-on: ubuntu-latest
    steps:
    - name: Checkout
      uses: actions/checkout@v4

    - name: Log in to Container Registry
      uses: docker/login-action@v3
      with:
        registry: ghcr.io
        username: ${{ github.actor }}
        password: ${{ secrets.GITHUB_TOKEN }}

    - name: Build and Push Image
      uses: docker/build-push-action@v5
      with:
        push: true
        tags: ghcr.io/${{ github.repository }}:${{ github.sha }}

    - name: Install Cosign
      uses: sigstore/cosign-installer@v3

    - name: Sign the Image (Keyless)
      run: |
        cosign sign --yes ghcr.io/${{ github.repository }}:${{ github.sha }}
      env:
        COSIGN_EXPERIMENTAL: "true"
```

**What this pipeline does:**
1. Checks out the code
2. Logs into GitHub Container Registry (ghcr.io)
3. Builds the Docker image and pushes it
4. Installs Cosign
5. Signs the image using **keyless signing** (the GitHub Actions OIDC token acts as the identity)

Now every image your CI/CD produces is **cryptographically signed**!

### 📜 Notary / TUF — The Enterprise Approach

**Notary** implements **The Update Framework (TUF)** — a well-established security framework for software distribution.

#### Notary v1 vs Notary v2

| Feature | Notary v1 (Docker Content Trust) | Notary v2 (Notation) |
|---------|--------------------------------|---------------------|
| Status | ⚠️ Deprecated / Maintenance mode | ✅ Active development |
| Complexity | High — needs a separate server | Lower — signatures stored in registry |
| CLI Tool | `docker trust` | `notation` |
| OCI Compatible | Limited | ✅ Full OCI compliance |
| Registry Support | Docker Hub, some others | Any OCI-compliant registry |

#### Using Notation (Notary v2)

```bash
# Install Notation
# Linux
curl -LO https://github.com/notaryproject/notation/releases/latest/download/notation_linux_amd64.tar.gz
tar xzf notation_linux_amd64.tar.gz
sudo mv notation /usr/local/bin/

# Generate a test key (for local testing)
notation cert generate-test --default "mycompany.io"

# Sign an image
notation sign myregistry.io/myapp:v1.0

# List signatures
notation ls myregistry.io/myapp:v1.0

# Verify an image
notation verify myregistry.io/myapp:v1.0
```

#### Notation with AWS Signer (Production Example)

```bash
# Install the AWS Signer plugin for Notation
```

## 🔗 Part 3: Enforcing Signed Images in Kubernetes

Signing images is great, but it's useless if Kubernetes doesn't **check** the signatures before running them!

### Option 1: Sigstore Policy Controller

The **Sigstore Policy Controller** is a Kubernetes admission controller that verifies image signatures before allowing Pods to run.

```bash
# Install Sigstore Policy Controller using Helm
helm repo add sigstore https://sigstore.github.io/helm-charts
helm install policy-controller sigstore/policy-controller \
  -n cosign-system --create-namespace
```

```yaml
# Create a ClusterImagePolicy
apiVersion: policy.sigstore.dev/v1beta1
kind: ClusterImagePolicy
metadata:
  name: require-signed-images
spec:
  images:
  - glob: "ghcr.io/mycompany/*"     # Apply to all images from your org
  authorities:
  - keyless:
      url: https://fulcio.sigstore.dev    # Sigstore's CA
      identities:
      - issuer: https://token.actions.githubusercontent.com
        subject: "https://github.com/mycompany/*"    # Only trust images signed by your GitHub Actions
```

```yaml
# Enable enforcement on a namespace
apiVersion: v1
kind: Namespace
metadata:
  name: production
  labels:
    policy.sigstore.dev/include: "true"    # Enforce signed images in this namespace
```

Now if someone tries to deploy an unsigned image in the `production` namespace:

```bash
kubectl -n production run test --image=ghcr.io/mycompany/app:v1.0
# Error: image ghcr.io/mycompany/app:v1.0 is not signed
```

### Option 2: Kyverno Image Verification

```yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: verify-image-signature
spec:
  validationFailureAction: Enforce
  webhookTimeoutSeconds: 30
  rules:
  - name: verify-cosign-signature
    match:
      any:
      - resources:
          kinds:
          - Pod
    verifyImages:
    - imageReferences:
      - "ghcr.io/mycompany/*"
      attestors:
      - entries:
        - keyless:
            subject: "https://github.com/mycompany/*"
            issuer: "https://token.actions.githubusercontent.com"
            rekor:
              url: https://rekor.sigstore.dev
```

**What this does:**
- Uses Kyverno (which you might already have from 10.36!)
- Checks every Pod's images against the signing policy
- Only allows images signed by your GitHub Actions pipelines
- Checks the Rekor transparency log to ensure the signature is valid

### Option 3: Connaisseur (Standalone Admission Controller)

```bash
# Install Connaisseur
helm repo add connaisseur https://sse-secure-systems.github.io/connaisseur/charts
helm install connaisseur connaisseur/connaisseur \
  -n connaisseur --create-namespace \
  --set validators[0].name=cosign \
  --set validators[0].type=cosign \
  --set validators[0].trustRoots[0].name=default \
  --set validators[0].trustRoots[0].key="$(cat cosign.pub)"
```

## 🔍 Part 4: Software Supply Chain Security — The Big Picture

Image signing is part of a bigger concept called **Software Supply Chain Security**. Here's how everything fits together:

```
Source Code     Build          Sign           Store          Deploy         Run
─────────────────────────────────────────────────────────────────────────────────
   Git      →  CI/CD     →  Cosign/    →  Registry   →  Admission   →  Kubernetes
   Commit      Pipeline     Notation      (signed      Controller      (verified
   (signed)    (trusted)    (sign        images)       (verify         and safe!)
               (SLSA)       image)                     signatures)

Security at each step:
├── Signed commits (GPG)
├── Trusted build (SLSA provenance)
├── Signed image (Cosign/Notation)
├── Scanned image (Trivy/Grype)
├── Verified at deploy time (Admission Controller)
└── Runtime security (Falco/Tetragon)
```

### SLSA (Supply-chain Levels for Software Artifacts)

**SLSA** (pronounced "salsa") is a framework that tells you how secure your build pipeline is:

| SLSA Level | Meaning |
|-----------|---------|
| Level 0 | No guarantees — anyone could have built anything |
| Level 1 | The build process is documented |
| Level 2 | The build service generates provenance (proof of how it was built) |
| Level 3 | The build platform is hardened (isolated, tamper-proof) |

```bash
# Generate SLSA provenance with Cosign
cosign attest --predicate provenance.json --type slsaprovenance \
  --key cosign.key myregistry.io/myapp:v1.0

# Verify the attestation
cosign verify-attestation --key cosign.pub \
  --type slsaprovenance myregistry.io/myapp:v1.0
```

```yaml
          kinds:
```

## 📋 Best Practices Checklist

| Practice | Why |
|---------|-----|
| ✅ Always use image digests (`@sha256:...`) instead of tags | Tags are mutable — digests are not |
| ✅ Sign all images in CI/CD automatically | Never deploy unsigned images |
| ✅ Use keyless signing in CI/CD | No key management headaches |
| ✅ Enforce signature verification in production namespaces | Don't just sign — verify! |
| ✅ Use `failurePolicy: Fail` on webhooks | If verification is broken, deny by default |
| ✅ Restrict image sources to trusted registries | Blocklist → Allowlist approach |
| ✅ Scan images for vulnerabilities (Trivy, Grype) | Signing proves authenticity, not safety |
| ✅ Store cosign private keys in a KMS (AWS KMS, HashiCorp Vault) | Never in Git or env vars |
| ✅ Set up SLSA provenance for builds | Know HOW your images were built |
| ✅ Monitor Rekor transparency logs | Audit who signed what and when |

## 🧠 Quick Recap

```
Image Security = Admission Controllers + Image Signing + Image Scanning

Admission Controllers:
├── Built-in: AlwaysPullImages, ImagePolicyWebhook
├── Webhook-based: Custom validation/mutation logic
└── Policy Engines: Kyverno, OPA Gatekeeper (easiest!)

Image Signing:
├── Cosign (Sigstore) ← RECOMMENDED
│   ├── Key-pair signing (traditional)
│   ├── Keyless signing (modern — uses OIDC identity)
│   └── CI/CD integration (GitHub Actions, GitLab CI)
├── Notation (Notary v2) ← Enterprise option
└── Docker Content Trust (Notary v1) ← Deprecated

Enforcement in Kubernetes:
├── Sigstore Policy Controller
├── Kyverno verifyImages
├── OPA Gatekeeper
└── Connaisseur

Supply Chain Security:
├── SLSA Framework
├── Cosign Attestations
└── Rekor Transparency Log
```
