# 10.38 — Helm — Package Manager for Kubernetes (Charts, Values, Templating, Hooks)

> **One Line Answer:** Helm is the **package manager for Kubernetes** — just like `apt` for Ubuntu or `npm` for Node.js. It lets you **package, share, and deploy** even the most complex Kubernetes applications with a single command, using reusable templates called **Charts**.

---

## 🤔 Why Do We Need Helm? — The Problem

Let's say you want to deploy a simple web application to Kubernetes. You'd need:

```
1. Deployment (for your app Pods)
2. Service (to expose the Pods)
3. ConfigMap (for configuration)
4. Secret (for passwords/API keys)
5. Ingress (for external access)
6. HPA (for autoscaling)
7. ServiceAccount (for RBAC)
8. PVC (for storage)
```

That's **8 YAML files**! And for each environment (dev, staging, prod), you need slightly different values — different image tags, replica counts, domain names, etc.

Now imagine you have **20 microservices**. That's **160 YAML files** to manage!

### The Pain Without Helm:

```
Without Helm:
├── app1/
│   ├── deployment.yaml        ← Copy-paste for each env
│   ├── service.yaml           ← Copy-paste for each env
│   ├── configmap.yaml         ← Copy-paste for each env
│   ├── deployment-staging.yaml   ← Almost identical!
│   ├── service-staging.yaml      ← Almost identical!
│   ├── deployment-prod.yaml      ← Almost identical!
│   └── ...                    ← YAML hell 🔥
├── app2/
│   ├── (same thing again...)
└── app3/
    └── (and again...)

Problems:
❌ Copy-paste everywhere
❌ Inconsistency between environments
❌ Hard to version or rollback
❌ No easy way to share configurations
❌ Updating one thing means editing 20 files
```

### The Joy With Helm:

```
With Helm:
├── my-app/                    ← One Helm Chart
│   ├── Chart.yaml             ← Metadata
│   ├── values.yaml            ← Default values (one place!)
│   ├── values-staging.yaml    ← Only the differences
│   ├── values-prod.yaml       ← Only the differences
│   └── templates/             ← Templates (write once, reuse forever)
│       ├── deployment.yaml
│       ├── service.yaml
│       └── configmap.yaml

# Deploy to dev:
helm install my-app ./my-app

# Deploy to staging:
helm install my-app ./my-app -f values-staging.yaml

# Deploy to prod:
helm install my-app ./my-app -f values-prod.yaml

# Rollback:
helm rollback my-app 1

# Upgrade:
helm upgrade my-app ./my-app --set image.tag=v2.0
```

**One chart. Multiple environments. Easy rollback. Life is good!** 🎉

## 📦 What is Helm? — The Basics

### Core Concepts

| Concept | What It Is | Real-World Analogy |
|---------|-----------|-------------------|
| **Chart** | A package of Kubernetes YAML templates + metadata | Like an `npm package` or `apt package` |
| **Values** | Variables that customize the chart for different environments | Like environment variables or config files |
| **Release** | A specific installation of a chart in your cluster | Like a running instance of an app |
| **Repository** | A place where charts are stored and shared | Like Docker Hub but for Helm charts |
| **Template** | A YAML file with Go template placeholders | Like a form with blank fields to fill |

### How Helm Works

```
┌──────────────┐    ┌───────────────┐    ┌─────────────┐
│  Chart       │    │   Values      │    │  Kubernetes  │
│  (Templates) │ +  │  (Variables)  │ =  │  Manifests   │
│              │    │               │    │  (Final YAML)│
└──────────────┘    └───────────────┘    └──────┬──────┘
                                                │
                                                ▼
                                       ┌──────────────┐
                                       │  Kubernetes   │
                                       │  API Server   │
                                       │  (kubectl     │
                                       │   apply)      │
                                       └──────────────┘
```

Helm takes your **templates** + **values** → renders them into **final YAML** → applies them to your cluster. Simple!

## 📁 Chart Structure — Anatomy of a Helm Chart

```bash
# Create a new chart
helm create my-webapp
```

This generates:

```
my-webapp/
├── Chart.yaml            # Chart metadata (name, version, description)
├── values.yaml           # Default configuration values
├── charts/               # Dependencies (sub-charts)
├── templates/            # Kubernetes manifest templates
│   ├── deployment.yaml   # Deployment template
│   ├── service.yaml      # Service template
│   ├── ingress.yaml      # Ingress template
│   ├── serviceaccount.yaml
│   ├── hpa.yaml          # HorizontalPodAutoscaler
│   ├── NOTES.txt         # Post-install message shown to user
│   ├── _helpers.tpl      # Template helper functions
│   └── tests/
│       └── test-connection.yaml
└── .helmignore           # Files to ignore when packaging
```

Let's look at each file in detail:

### Chart.yaml — The Identity Card

```yaml
# Chart.yaml
apiVersion: v2                    # Helm chart API version (v2 for Helm 3)
name: my-webapp                   # Chart name
description: A simple web application
type: application                 # "application" or "library"
version: 0.1.0                    # Chart version (follows SemVer)
appVersion: "1.0.0"               # Version of the app being deployed

# Optional fields
keywords:
  - web
  - nginx
home: https://github.com/mycompany/my-webapp
maintainers:
  - name: John Doe
    email: john@example.com

# Dependencies (other charts this chart needs)
dependencies:
  - name: postgresql
    version: "12.x.x"
    repository: "https://charts.bitnami.com/bitnami"
    condition: postgresql.enabled     # Only include if this value is true
  - name: redis
    version: "17.x.x"
    repository: "https://charts.bitnami.com/bitnami"
    condition: redis.enabled
```

**Key fields explained:**
- `apiVersion: v2` — Always `v2` for Helm 3 charts
- `version` — The CHART version. Bump this when you change the chart templates
- `appVersion` — The APPLICATION version. Bump this when you deploy a new version of your app
- `dependencies` — Other charts that this chart depends on (e.g., a database)

### values.yaml — The Configuration

```yaml
# values.yaml — Default values for the chart
# Users can override ANY of these values

replicaCount: 2

image:
  repository: nginx
  tag: "1.25"
  pullPolicy: IfNotPresent

imagePullSecrets: []

nameOverride: ""
fullnameOverride: ""

serviceAccount:
  create: true
  annotations: {}
  name: ""

service:
  type: ClusterIP
  port: 80

ingress:
  enabled: false
  className: "nginx"
  annotations: {}
  hosts:
    - host: myapp.example.com
      paths:
        - path: /
          pathType: Prefix
  tls: []

resources:
  limits:
    cpu: 200m
    memory: 256Mi
  requests:
    cpu: 100m
    memory: 128Mi

autoscaling:
  enabled: false
  minReplicas: 2
  maxReplicas: 10
  targetCPUUtilizationPercentage: 80

nodeSelector: {}
tolerations: []
affinity: {}

# Application-specific config
config:
  databaseUrl: "postgres://localhost:5432/mydb"
  logLevel: "info"
  
# Feature flags
postgresql:
  enabled: false
redis:
  enabled: false
```

## 🎨 Helm Templating — The Heart of Helm

Helm uses **Go templates** to turn your YAML templates into actual Kubernetes manifests. Let's learn the syntax:

### Basic Syntax

```yaml
# templates/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ .Release.Name }}-webapp       # Inserts the release name
  labels:
    app: {{ .Chart.Name }}               # Inserts the chart name
    version: {{ .Chart.AppVersion }}     # Inserts the app version
spec:
  replicas: {{ .Values.replicaCount }}   # Inserts the value from values.yaml
  selector:
    matchLabels:
      app: {{ .Chart.Name }}
  template:
    metadata:
      labels:
        app: {{ .Chart.Name }}
    spec:
      containers:
      - name: {{ .Chart.Name }}
        image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
        imagePullPolicy: {{ .Values.image.pullPolicy }}
        ports:
        - containerPort: 80
```

**The `{{ }}` syntax** is Go template syntax. Everything inside double curly braces gets replaced with actual values.

### Built-in Objects

| Object | What It Contains |
|--------|-----------------|
| `.Values` | Values from `values.yaml` and command-line overrides |
| `.Release` | Info about the release (name, namespace, revision, etc.) |
| `.Chart` | Info from `Chart.yaml` (name, version, appVersion) |
| `.Template` | Info about the current template file |
| `.Files` | Access to non-template files in the chart |
| `.Capabilities` | Info about the cluster's capabilities (API versions, etc.) |

```yaml
# Commonly used built-in values:
{{ .Release.Name }}         # Name of the release (e.g., "my-app")
{{ .Release.Namespace }}    # Namespace where release is installed
{{ .Release.Revision }}     # Revision number (increments on upgrade)
{{ .Release.IsInstall }}    # true if this is an install (not upgrade)
{{ .Release.IsUpgrade }}    # true if this is an upgrade
{{ .Chart.Name }}           # Chart name from Chart.yaml
{{ .Chart.Version }}        # Chart version from Chart.yaml
{{ .Chart.AppVersion }}     # App version from Chart.yaml
```

### Conditionals (if/else)

```yaml
# templates/ingress.yaml
{{- if .Values.ingress.enabled }}
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: {{ .Release.Name }}-ingress
  {{- if .Values.ingress.annotations }}
  annotations:
    {{- toYaml .Values.ingress.annotations | nindent 4 }}
  {{- end }}
spec:
  ingressClassName: {{ .Values.ingress.className }}
  rules:
  {{- range .Values.ingress.hosts }}
  - host: {{ .host }}
    http:
      paths:
      {{- range .paths }}
      - path: {{ .path }}
        pathType: {{ .pathType }}
        backend:
          service:
            name: {{ $.Release.Name }}-webapp
            port:
              number: 80
      {{- end }}
  {{- end }}
{{- end }}
```

**What this does:**
- The entire Ingress resource is only created if `ingress.enabled` is `true`
- Annotations are only added if they exist
- `range` loops through the hosts and paths arrays
- `$` refers to the root scope (needed inside `range` loops to access `.Release`)

### Loops (range)

```yaml
# Loop through a list
env:
{{- range .Values.env }}
- name: {{ .name }}
  value: {{ .value | quote }}
{{- end }}

# With values.yaml:
# env:
#   - name: LOG_LEVEL
#     value: info
#   - name: DB_HOST
#     value: postgres

# Loop through a map
{{- range $key, $value := .Values.labels }}
{{ $key }}: {{ $value }}
{{- end }}

# With values.yaml:
# labels:
#   team: backend
#   tier: web
```

### Template Functions & Pipes

Helm has TONS of useful template functions:

```yaml
# String functions
{{ .Values.image.tag | upper }}          # Convert to uppercase
{{ .Values.name | lower }}              # Convert to lowercase
{{ .Values.name | title }}              # Title Case
{{ .Values.name | quote }}              # Wrap in quotes: "value"
{{ .Values.name | trim }}               # Remove whitespace
{{ .Values.name | replace "-" "_" }}    # Replace characters
{{ .Values.name | default "nginx" }}    # Use default if empty

# Number functions
{{ .Values.port | int }}                # Convert to integer

# YAML/JSON conversion
{{ .Values.config | toYaml }}           # Convert to YAML string
{{ .Values.config | toJson }}           # Convert to JSON string

# Indentation (SUPER important in YAML!)
{{ .Values.resources | toYaml | nindent 8 }}   # Convert to YAML and indent 8 spaces
# nindent = "newline + indent"
# indent = just indent (no newline)

# Required values (fail if not set)
{{ required "image.repository is required!" .Values.image.repository }}

# Lookup (check if a resource exists in the cluster)
{{ lookup "v1" "Secret" "default" "my-secret" }}
```

### Whitespace Control

This is tricky but important! YAML is whitespace-sensitive, so Helm gives you control:

```yaml
# {{- removes whitespace BEFORE the tag
# -}} removes whitespace AFTER the tag

# Without whitespace control (notice the blank lines):
metadata:
  labels:
    {{ if .Values.team }}
    team: {{ .Values.team }}
    {{ end }}

# RESULT (ugly blank lines):
# metadata:
#   labels:
#     
#     team: backend
#     

# With whitespace control (clean output):
metadata:
  labels:
    {{- if .Values.team }}
    team: {{ .Values.team }}
    {{- end }}

# RESULT (clean):
# metadata:
#   labels:
#     team: backend
```

### Named Templates (_helpers.tpl)

**Named templates** (also called **partials**) let you reuse template snippets. They live in `_helpers.tpl`:

```yaml
# templates/_helpers.tpl

# Define a template for the app name
{{- define "my-webapp.name" -}}
{{- default .Chart.Name .Values.nameOverride | trunc 63 | trimSuffix "-" }}
{{- end }}

# Define a template for the full name
{{- define "my-webapp.fullname" -}}
{{- if .Values.fullnameOverride }}
{{- .Values.fullnameOverride | trunc 63 | trimSuffix "-" }}
{{- else }}
{{- $name := default .Chart.Name .Values.nameOverride }}
{{- printf "%s-%s" .Release.Name $name | trunc 63 | trimSuffix "-" }}
{{- end }}
{{- end }}

# Define common labels
{{- define "my-webapp.labels" -}}
helm.sh/chart: {{ .Chart.Name }}-{{ .Chart.Version | replace "+" "_" }}
app.kubernetes.io/name: {{ include "my-webapp.name" . }}
app.kubernetes.io/instance: {{ .Release.Name }}
app.kubernetes.io/version: {{ .Chart.AppVersion | quote }}
app.kubernetes.io/managed-by: {{ .Release.Service }}
{{- end }}

# Selector labels (subset of common labels)
{{- define "my-webapp.selectorLabels" -}}
app.kubernetes.io/name: {{ include "my-webapp.name" . }}
app.kubernetes.io/instance: {{ .Release.Name }}
{{- end }}
```

**Using named templates:**

```yaml
# templates/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ include "my-webapp.fullname" . }}
  labels:
    {{- include "my-webapp.labels" . | nindent 4 }}
spec:
  selector:
    matchLabels:
      {{- include "my-webapp.selectorLabels" . | nindent 6 }}
  template:
    metadata:
      labels:
        {{- include "my-webapp.selectorLabels" . | nindent 8 }}
```

**Key difference:**
- `{{ template "name" . }}` — inserts the template but does NOT allow piping
- `{{ include "name" . }}` — inserts the template AND allows piping (use this!)

## 🛠️ Helm CLI — Essential Commands

### Managing Repositories

```bash
# Add a repository
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo add ingress-nginx https://kubernetes.github.io/ingress-nginx
helm repo add prometheus https://prometheus-community.github.io/helm-charts

# Update repos (like apt update)

helm repo list

helm search repo nginx

# Search Artifact Hub (the public chart registry)
helm search hub wordpress
```

### Installing Charts

```bash
# Install from a repo
helm install my-release bitnami/nginx

# Install with a specific version
helm install my-release bitnami/nginx --version 15.0.0

# Install into a specific namespace
helm install my-release bitnami/nginx -n production --create-namespace

# Install with custom values file
helm install my-release bitnami/nginx -f my-values.yaml

# Install with individual value overrides
helm install my-release bitnami/nginx \
  --set replicaCount=3 \
  --set image.tag=1.25

# Install from a local chart directory
helm install my-release ./my-webapp

# Install with --dry-run to preview without actually installing
helm install my-release ./my-webapp --dry-run

# Install and wait for all resources to be ready
helm install my-release ./my-webapp --wait --timeout 5m
```

### Viewing & Debugging

```bash
# Show the rendered YAML (without installing)
helm template my-release ./my-webapp

# Show rendered YAML with specific values
helm template my-release ./my-webapp -f values-prod.yaml

# Show computed values (merged defaults + overrides)
helm get values my-release

# Show all values (including defaults)
helm get values my-release --all

# Show the rendered manifests of an installed release
helm get manifest my-release

# Show release notes
helm get notes my-release

# Show everything about a release
helm get all my-release

# Debug template rendering issues
helm template my-release ./my-webapp --debug
```

### Upgrading

```bash
# Upgrade with new values
helm upgrade my-release ./my-webapp --set image.tag=v2.0

# Upgrade with a values file
helm upgrade my-release ./my-webapp -f values-prod.yaml

# Install if not exists, upgrade if exists (idempotent!)
helm upgrade --install my-release ./my-webapp

# Upgrade and reuse previous values (+ apply new ones)
helm upgrade my-release ./my-webapp --reuse-values --set image.tag=v2.0

# Upgrade and reset values to defaults (then apply overrides)
helm upgrade my-release ./my-webapp --reset-values --set image.tag=v2.0
```

## 📝 Values — Customizing Charts for Different Environments

### Values Precedence (from lowest to highest)

```
1. values.yaml (in the chart)         ← Lowest priority (defaults)
2. Parent chart's values.yaml         ← Overrides child defaults
3. -f / --values flag (file)          ← Overrides chart defaults
4. --set flag (command line)          ← Highest priority
```

### Environment-Specific Values Files

```yaml
# values.yaml (defaults — for dev)
replicaCount: 1
image:
  repository: mycompany/webapp
  tag: "latest"
resources:
  requests:
    cpu: 100m
    memory: 128Mi
ingress:
  enabled: false
```

```yaml
# values-staging.yaml (only the differences!)
replicaCount: 2
image:
  tag: "v1.5.0-rc1"
resources:
  requests:
    cpu: 250m
    memory: 256Mi
ingress:
  enabled: true
  hosts:
    - host: staging.myapp.com
      paths:
        - path: /
          pathType: Prefix
```

```yaml
# values-prod.yaml
replicaCount: 5
image:
  tag: "v1.4.0"
resources:
  requests:
    cpu: 500m
    memory: 512Mi
  limits:
    cpu: "1"
    memory: "1Gi"
ingress:
  enabled: true
  hosts:
    - host: myapp.com
      paths:
        - path: /
          pathType: Prefix
  tls:
    - secretName: myapp-tls
      hosts:
        - myapp.com
autoscaling:
  enabled: true
  minReplicas: 5
  maxReplicas: 20
```

```bash
# Deploy to each environment:
helm upgrade --install webapp ./my-webapp -f values.yaml                    # Dev
helm upgrade --install webapp ./my-webapp -f values-staging.yaml -n staging # Staging
helm upgrade --install webapp ./my-webapp -f values-prod.yaml -n production # Prod
```

### Using --set for Individual Values

```bash
# Simple values
helm install app ./my-webapp --set replicaCount=3

# Nested values (use dots)
helm install app ./my-webapp --set image.tag=v2.0

# Strings with special characters (use quotes)
helm install app ./my-webapp --set config.url="http://db:5432/mydb"

# Arrays
helm install app ./my-webapp --set "env[0].name=LOG_LEVEL,env[0].value=debug"

# Multiple values
helm install app ./my-webapp \
  --set replicaCount=3 \
  --set image.tag=v2.0 \
  --set ingress.enabled=true
```

## 🪝 Helm Hooks — Running Jobs at Specific Points

**Hooks** let you run things at specific points in a release lifecycle — like running a database migration BEFORE an upgrade, or sending a notification AFTER a successful install.

### Available Hooks

| Hook | When It Runs |
|------|-------------|
| `pre-install` | Before any resources are created |
| `post-install` | After all resources are created |
| `pre-upgrade` | Before any resources are updated |
| `post-upgrade` | After all resources are updated |
| `pre-delete` | Before any resources are deleted |
| `post-delete` | After all resources are deleted |
| `pre-rollback` | Before a rollback |
| `post-rollback` | After a rollback |
| `test` | When `helm test` is run |

### Example: Database Migration Hook

```yaml
# templates/migration-job.yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: {{ .Release.Name }}-db-migrate
  annotations:
    "helm.sh/hook": pre-upgrade          # Run BEFORE upgrade
    "helm.sh/hook-weight": "1"           # Order (lower runs first)
    "helm.sh/hook-delete-policy": hook-succeeded   # Delete job after success
spec:
  template:
    spec:
      restartPolicy: Never
      containers:
      - name: migrate
        image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
        command: ["python", "manage.py", "migrate"]
        env:
        - name: DATABASE_URL
          valueFrom:
            secretKeyRef:
              name: {{ .Release.Name }}-db-secret
              key: url
  backoffLimit: 3
```

**What this does:**
- Before Helm upgrades your app, it runs a database migration job
- `hook-weight: "1"` means if you have multiple pre-upgrade hooks, this one runs first
- `hook-delete-policy: hook-succeeded` means the Job is automatically deleted after it succeeds (keeps your cluster clean)

### Hook Delete Policies

| Policy | What It Does |
|--------|-------------|
| `hook-succeeded` | Delete the hook resource after it succeeds |
| `hook-failed` | Delete the hook resource after it fails |
| `before-hook-creation` | Delete any existing hook resource before creating a new one |

### Example: Post-Install Notification

```yaml
# templates/notify-job.yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: {{ .Release.Name }}-notify
  annotations:
    "helm.sh/hook": post-install,post-upgrade   # Run after install OR upgrade
    "helm.sh/hook-weight": "10"
    "helm.sh/hook-delete-policy": hook-succeeded
spec:
  template:
    spec:
      restartPolicy: Never
      containers:
      - name: notify
        image: curlimages/curl:latest
        command:
        - /bin/sh
        - -c
        - |
          curl -X POST https://slack.webhook.url \
            -H 'Content-Type: application/json' \
            -d '{"text": "🚀 {{ .Release.Name }} deployed successfully (v{{ .Chart.AppVersion }})"}'
```

## 📦 Chart Dependencies (Sub-Charts)

Your app probably needs a database, a cache, or other services. Instead of deploying them separately, you can include them as **dependencies**:

### Declaring Dependencies

```yaml
# Chart.yaml
dependencies:
  - name: postgresql
    version: "12.5.8"
    repository: "https://charts.bitnami.com/bitnami"
    condition: postgresql.enabled          # Only include if enabled
  - name: redis
    version: "17.11.3"
    repository: "https://charts.bitnami.com/bitnami"
    condition: redis.enabled
    alias: cache                           # Use "cache" instead of "redis" in values
```

### Managing Dependencies

```bash
# Download dependencies (fetches into charts/ directory)
helm dependency update ./my-webapp

# List dependencies and their status
helm dependency list ./my-webapp

# Output:
# NAME         VERSION    REPOSITORY                              STATUS
# postgresql   12.5.8     https://charts.bitnami.com/bitnami      ok
# redis        17.11.3    https://charts.bitnami.com/bitnami      ok
```

### Configuring Dependencies in values.yaml

```yaml
# values.yaml
postgresql:
  enabled: true                          # Enable the PostgreSQL sub-chart
  auth:
    postgresPassword: "supersecret"      # PostgreSQL-specific values
    database: "myapp"
  primary:
    persistence:
      size: 10Gi

redis:
  enabled: true
  architecture: standalone
  auth:
    password: "redispassword"
```

The key insight: **values under the dependency name are passed to that sub-chart**. So `postgresql.auth.postgresPassword` in YOUR values.yaml becomes `auth.postgresPassword` inside the PostgreSQL chart.

## 📤 Packaging & Publishing Charts

### Package a Chart

```bash
# Package into a .tgz file
helm package ./my-webapp
# Creates: my-webapp-0.1.0.tgz

# Package with a specific version
helm package ./my-webapp --version 1.2.3
```

### Publish to a Chart Repository

```bash
# Option 1: Use GitHub Pages
# Create an index.yaml for your chart repository
helm repo index . --url https://mycompany.github.io/helm-charts

# Option 2: Use OCI Registry (recommended — Helm 3.8+)
# Login to your OCI registry
helm registry login myregistry.azurecr.io

# Push a chart to an OCI registry
helm push my-webapp-0.1.0.tgz oci://myregistry.azurecr.io/helm-charts

# Install from an OCI registry
helm install my-release oci://myregistry.azurecr.io/helm-charts/my-webapp --version 0.1.0
```

### Publish to Artifact Hub

**Artifact Hub** (https://artifacthub.io) is the public registry for Helm charts. To publish:

1. Host your chart repo (GitHub Pages, S3, or OCI registry)
2. Create an account on Artifact Hub
3. Register your repository

```yaml
  repository: mycompany/node-api
  tag: "2.3.0"
  pullPolicy: IfNotPresent

  type: ClusterIP
  port: 3000

ingress:
  enabled: false
```

```yaml
    {{- include "node-api.labels" . | nindent 4 }}

          containerPort: 3000

            path: /health

          httpGet:
            path: /ready
            port: http
          initialDelaySeconds: 5
          periodSeconds: 5
        env:
        {{- range .Values.env }}
        - name: {{ .name }}
          value: {{ .value | quote }}
        {{- end }}
        {{- if .Values.postgresql.enabled }}
        - name: DATABASE_URL
          value: "postgresql://postgres:{{ .Values.postgresql.auth.postgresPassword }}@{{ .Release.Name }}-postgresql:5432/{{ .Values.postgresql.auth.database }}"
        {{- end }}
        resources:
          {{- toYaml .Values.resources | nindent 10 }}
```

## ⚠️ Common Pitfalls & Tips

| Pitfall | Solution |
|---------|---------|
| Indentation errors in templates | Use `nindent` instead of `indent`, and always test with `helm template` |
| Forgetting quotes around values | Use `{{ .Values.tag | quote }}` for strings that might look like numbers |
| Using `latest` tag | Always pin to a specific version in `values.yaml` |
| Secrets in `values.yaml` checked into Git | Use external secret managers (Vault, AWS Secrets Manager) or `helm-secrets` plugin |
| Not using `--dry-run` before applying | Always preview changes before applying to production |
| Helm release names too long | Keep release names short — Kubernetes has 63-char limits on names |
| Not versioning your chart | Always bump `version` in `Chart.yaml` when templates change |

## 🧠 Quick Recap

```
Helm = Package Manager for Kubernetes

Core Concepts:
├── Chart      → Package of templates + metadata
├── Values     → Variables to customize the chart
├── Release    → An installed instance of a chart
├── Repository → Where charts are stored
└── Template   → YAML with Go template syntax

Chart Structure:
├── Chart.yaml      → Identity card
├── values.yaml     → Default config
├── templates/      → Kubernetes manifests with {{ }}
│   ├── _helpers.tpl  → Reusable template snippets
│   └── NOTES.txt     → Post-install message
└── charts/         → Dependencies

Key Commands:
├── helm create       → Scaffold a new chart
├── helm install      → Deploy a release
├── helm upgrade      → Update a release
├── helm rollback     → Undo an upgrade
├── helm template     → Preview rendered YAML
├── helm list         → List releases
└── helm uninstall    → Remove a release

Templating:
├── {{ .Values.x }}        → Insert a value
├── {{ if }}...{{ end }}   → Conditionals
├── {{ range }}...{{ end }}→ Loops
├── {{ include "name" . }} → Include named template
├── | nindent N            → Indent N spaces
└── | quote / default / required → Transform values

Hooks: pre-install, post-install, pre-upgrade, post-upgrade, test
```
