# 10.39 — Kustomize — Template-Free Configuration Management

> **One Line Answer:** Kustomize lets you **customize Kubernetes YAML files without using templates** — instead of adding `{{ }}` placeholders everywhere, you keep your original YAML clean and apply **patches, overlays, and transformations** on top of it for different environments.

---

## 🤔 Why Kustomize? — The Problem with Templates

We just learned Helm, and it's amazing. But Helm uses **Go templates** which can get messy:

```yaml
# Helm template (hard to read, right?)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ include "myapp.fullname" . }}
  labels:
    {{- include "myapp.labels" . | nindent 4 }}
spec:
  {{- if not .Values.autoscaling.enabled }}
  replicas: {{ .Values.replicaCount }}
  {{- end }}
  selector:
    matchLabels:
      {{- include "myapp.selectorLabels" . | nindent 6 }}
  template:
    metadata:
      labels:
        {{- include "myapp.selectorLabels" . | nindent 8 }}
    spec:
      containers:
      - name: {{ .Chart.Name }}
        image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
```

That's not valid YAML anymore! You can't just `kubectl apply` it. You can't validate it with a YAML linter. You can't easily read it.

### Kustomize's Philosophy

> **"Keep your YAML files as plain, valid Kubernetes manifests. Customize them by applying patches and overlays — no templates needed."**

```yaml
# Kustomize way (clean, readable, valid YAML!)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 2
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: myapp
        image: nginx:1.25
```

That's just **normal Kubernetes YAML**! You can `kubectl apply` it directly. Kustomize adds the customization layer **on top** of it.

---

## 📦 What is Kustomize?

Kustomize is a **configuration management tool** built into `kubectl` that lets you:

1. **Keep a "base"** — your standard Kubernetes YAML files
2. **Create "overlays"** — environment-specific customizations (dev, staging, prod)
3. **Apply patches** — add, modify, or delete specific fields
4. **Generate resources** — auto-create ConfigMaps and Secrets from files
5. **Transform resources** — add labels, annotations, name prefixes/suffixes

All **without modifying the original YAML files**!

### The Mental Model

```
┌─────────────────────────────────────────────┐
│                   BASE                       │
│  (Your clean, standard Kubernetes YAML)      │
│                                              │
│  deployment.yaml                             │
│  service.yaml                                │
│  configmap.yaml                              │
│  kustomization.yaml                          │
└──────────────┬──────────────────────────────┘
               │
    ┌──────────┼──────────┐
    │          │          │
    ▼          ▼          ▼
┌────────┐ ┌────────┐ ┌────────┐
│  DEV   │ │STAGING │ │  PROD  │
│Overlay │ │Overlay │ │Overlay │
│        │ │        │ │        │
│+1 rep  │ │+3 reps │ │+5 reps │
│+debug  │ │+staging│ │+prod   │
│ labels │ │ domain │ │ domain │
└────────┘ └────────┘ └────────┘

Each overlay PATCHES the base — the base is never modified!
```

### Kustomize vs Helm — Quick Comparison

| Feature | Kustomize | Helm |
|---------|-----------|------|
| Approach | Patching/overlaying | Templating |
| Valid YAML? | ✅ Always valid YAML | ❌ Templates are not valid YAML |
| Built into kubectl? | ✅ Yes (`kubectl apply -k`) | ❌ Separate tool |
| Learning curve | Low | Medium-High |
| Packaging & sharing | ❌ Not great | ✅ Charts + repos |
| Rollback | ❌ No built-in | ✅ Built-in |
| Hooks | ❌ No | ✅ Yes |
| Dependencies | ❌ No sub-charts | ✅ Sub-charts |
| Best for | Simple to moderate customization | Complex apps with many configs |

> **Pro tip:** You don't have to choose! Many teams use **Helm for third-party apps** (installing Nginx Ingress, Prometheus, etc.) and **Kustomize for their own apps** (because their YAML stays clean and readable).

## 🔧 Installing & Using Kustomize

Kustomize is **already built into kubectl** (since v1.14)! You can also install the standalone binary:

```bash
# Option 1: Use kubectl (already built-in!)
kubectl apply -k ./my-app/overlays/production
kubectl kustomize ./my-app/overlays/production    # Preview only

# Option 2: Install standalone kustomize (latest features)
# Linux
curl -s "https://raw.githubusercontent.com/kubernetes-sigs/kustomize/master/hack/install_kustomize.sh" | bash
sudo mv kustomize /usr/local/bin/

# macOS
brew install kustomize

# Verify
kustomize version
```

> **Note:** The standalone `kustomize` binary is usually newer than the one bundled in `kubectl`. If you need the latest features, use the standalone version.

## 📁 Project Structure — Base & Overlays

Here's the standard Kustomize directory structure:

```
my-app/
├── base/                         # The "base" — standard manifests
│   ├── kustomization.yaml        # Lists all resources in the base
│   ├── deployment.yaml           # Standard deployment
│   ├── service.yaml              # Standard service
│   └── configmap.yaml            # Standard configmap
│
└── overlays/                     # Environment-specific customizations
    ├── dev/
    │   ├── kustomization.yaml    # References base + dev-specific patches
    │   └── replica-patch.yaml    # Dev: 1 replica
    │
    ├── staging/
    │   ├── kustomization.yaml
    │   ├── replica-patch.yaml    # Staging: 3 replicas
    │   └── ingress.yaml          # Staging: add ingress
    │
    └── production/
        ├── kustomization.yaml
        ├── replica-patch.yaml    # Prod: 5 replicas
        ├── resource-patch.yaml   # Prod: more CPU/memory
        ├── hpa.yaml              # Prod: add autoscaling
        └── ingress.yaml          # Prod: production domain
```

## 🏗️ Building Your First Kustomize Project

### Step 1: Create the Base

```yaml
# base/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 1
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: myapp
        image: mycompany/myapp:latest
        ports:
        - containerPort: 8080
        env:
        - name: LOG_LEVEL
          value: "info"
        resources:
          requests:
            cpu: 100m
            memory: 128Mi
          limits:
            cpu: 200m
            memory: 256Mi
```

```yaml
# base/service.yaml
apiVersion: v1
kind: Service
metadata:
  name: myapp
spec:
  selector:
    app: myapp
  ports:
  - port: 80
    targetPort: 8080
  type: ClusterIP
```

```yaml
# base/configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: myapp-config
data:
  APP_ENV: "development"
  DB_HOST: "localhost"
  DB_PORT: "5432"
```

### Step 2: Create the Base kustomization.yaml

This is the **key file** — it tells Kustomize what resources to include:

```yaml
# base/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

# List all resources in the base
resources:
  - deployment.yaml
  - service.yaml
  - configmap.yaml

# Add common labels to ALL resources
commonLabels:
  managed-by: kustomize

# Add common annotations
commonAnnotations:
  team: backend
```

### Step 3: Test the Base

```bash
# Preview the rendered YAML
kubectl kustomize ./base

# Or with standalone kustomize:
kustomize build ./base

# Apply the base directly (for dev):
kubectl apply -k ./base
```

**Output:** All three resources rendered with the common labels and annotations added. The YAML files are unchanged — Kustomize adds the labels at build time!

### Step 4: Create the Dev Overlay

```yaml
# overlays/dev/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

# Reference the base
resources:
  - ../../base

# Add a name prefix for dev
namePrefix: dev-

# Add dev-specific labels
commonLabels:
  env: development

# Patch the deployment for dev
patches:
  - target:
      kind: Deployment
      name: myapp
    patch: |-
      - op: replace
        path: /spec/replicas
        value: 1
      - op: replace
        path: /spec/template/spec/containers/0/env/0/value
        value: "debug"
```

```bash
# Preview dev overlay
kubectl kustomize ./overlays/dev

# The Deployment name becomes "dev-myapp", replicas = 1, LOG_LEVEL = "debug"
```

### Step 5: Create the Production Overlay

```yaml
# overlays/production/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

# Reference the base
resources:
  - ../../base
  - hpa.yaml              # Add extra resources for prod

# Add a name prefix
namePrefix: prod-

# Use a different namespace
namespace: production

# Add prod-specific labels
commonLabels:
  env: production

# Set the image tag (no need to edit the deployment!)
images:
  - name: mycompany/myapp
    newTag: "v2.3.0"

# Patch using a strategic merge patch file
patches:
  - path: replica-patch.yaml
  - path: resource-patch.yaml
```

```yaml
# overlays/production/replica-patch.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp              # Must match the base resource name
spec:
  replicas: 5              # Override replicas to 5
```

```yaml
# overlays/production/resource-patch.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  template:
    spec:
      containers:
      - name: myapp        # Must match the container name
        resources:
          requests:
            cpu: 500m
            memory: 512Mi
          limits:
            cpu: "1"
            memory: "1Gi"
```

```yaml
# overlays/production/hpa.yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: myapp
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  minReplicas: 5
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 75
```

```bash
# Preview production overlay
kubectl kustomize ./overlays/production

# Apply production overlay
kubectl apply -k ./overlays/production
```

**What happens:**
- All resources get the `prod-` prefix (so `myapp` → `prod-myapp`)
- All resources go to the `production` namespace
- Image tag changes from `latest` to `v2.3.0`
- Deployment gets 5 replicas with beefier resources
- HPA is added (only in production!)
- Common labels include `env: production`

## 🔧 Kustomize Features Deep Dive

### 1. Image Transformations

Change image names, tags, or digests **without editing any YAML file**:

```yaml
# kustomization.yaml
images:
  # Change just the tag
  - name: mycompany/myapp
    newTag: "v3.0.0"

  # Change the entire image name
  - name: mycompany/myapp
    newName: ghcr.io/mycompany/myapp
    newTag: "v3.0.0"

  # Use a specific digest (most secure!)
  - name: mycompany/myapp
    newName: ghcr.io/mycompany/myapp
    digest: sha256:abc123def456...
```

This is **amazing** for CI/CD — your pipeline can update the image tag without touching any manifest:

```bash
# In CI/CD pipeline:
cd overlays/production
kustomize edit set image mycompany/myapp=mycompany/myapp:${GIT_SHA}
kustomize build . | kubectl apply -f -
```

### 2. ConfigMap and Secret Generators

Instead of creating ConfigMaps and Secrets manually, Kustomize can **generate them from files or literals** — and it automatically adds a **hash suffix** so changes trigger Pod restarts!

```yaml
# kustomization.yaml
configMapGenerator:
  # From literal key-value pairs
  - name: app-config
    literals:
      - APP_ENV=production
      - LOG_LEVEL=warn
      - DB_HOST=prod-db.internal

  # From a file
  - name: nginx-config
    files:
      - nginx.conf                          # Key = filename, value = file contents

  # From a .env file
  - name: env-config
    envs:
      - .env.production                     # Each line becomes a key-value pair

secretGenerator:
  # From literals
  - name: db-credentials
    literals:
      - username=admin
      - password=supersecretpassword
    type: Opaque

  # From files
  - name: tls-cert
    files:
      - tls.crt
      - tls.key
    type: kubernetes.io/tls
```

**The magic of hash suffixes:**

```bash
# Without Kustomize: If you change a ConfigMap, Pods don't restart
# because the ConfigMap name is the same!

# With Kustomize: ConfigMap name gets a hash suffix based on contents
# Original: app-config
# Generated: app-config-8m2k5h     ← Hash changes when content changes!

# When the hash changes, the Deployment reference changes too,
# which triggers a rolling update! Automatic Pod restarts on config changes!
```

If you DON'T want the hash suffix:

```yaml
# kustomization.yaml
generatorOptions:
  disableNameSuffixHash: true
```

### 3. Patches — Three Ways to Patch

Kustomize supports three types of patches:

#### a) Strategic Merge Patch (Most Common)

You write a **partial YAML** that gets merged with the original:

```yaml
# patches/deployment-patch.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp           # Must match the target resource
spec:
  replicas: 5           # Only this field is changed
  template:
    spec:
      containers:
      - name: myapp     # Must match the container name
        resources:
          limits:
            cpu: "2"
            memory: "2Gi"
```

```yaml
# kustomization.yaml
patches:
  - path: patches/deployment-patch.yaml
```

**How it works:** Kustomize takes the base deployment and **merges** this patch into it. Only the fields you specify are changed — everything else stays the same.

#### b) JSON Patch (Precise Field Targeting)

Use JSON Patch operations (`add`, `remove`, `replace`, `move`, `copy`, `test`) for precise changes:

```yaml
# kustomization.yaml
patches:
  - target:
      kind: Deployment
      name: myapp
    patch: |-
      # Replace replicas
      - op: replace
        path: /spec/replicas
        value: 5
      
      # Add a new environment variable
      - op: add
        path: /spec/template/spec/containers/0/env/-
        value:
          name: NEW_FEATURE
          value: "enabled"
      
      # Remove a specific env var (by index)
      - op: remove
        path: /spec/template/spec/containers/0/env/2
      
      # Add an annotation
      - op: add
        path: /metadata/annotations
        value:
          backup: "true"
```

**When to use JSON Patch:**
- When you need to **remove** something (strategic merge can't easily remove fields)
- When you need to target a specific array index
- When you need precise control over what changes

#### c) Inline Patches

You can write patches directly in `kustomization.yaml` without separate files:

```yaml
# kustomization.yaml
patches:
  # Inline strategic merge patch
  - patch: |-
      apiVersion: apps/v1
      kind: Deployment
      metadata:
        name: myapp
      spec:
        replicas: 5
    
  # Inline JSON patch with target
  - target:
      kind: Service
      name: myapp
    patch: |-
      - op: replace
        path: /spec/type
        value: LoadBalancer
```

### 4. Name Prefix & Suffix

```yaml
# kustomization.yaml
namePrefix: prod-           # All resource names get "prod-" prefix
nameSuffix: -v2              # All resource names get "-v2" suffix

# Result: "myapp" becomes "prod-myapp-v2"
# All references (Service selectors, HPA targets, etc.) are updated automatically!
```

### 5. Namespace Transformation

```yaml
# kustomization.yaml
namespace: production        # All resources get this namespace

# Every resource in the output will have:
# metadata:
#   namespace: production
```

### 6. Common Labels & Annotations

```yaml
# kustomization.yaml
commonLabels:
  app.kubernetes.io/part-of: my-platform
  env: production
  team: backend

commonAnnotations:
  owner: backend-team@mycompany.com
  cost-center: "12345"

# These labels/annotations are added to EVERY resource AND
# to selectors (matchLabels, selector, etc.) where appropriate
```

### 7. Replacements (Kustomize v4+)

**Replacements** let you copy a value from one resource and paste it into another:

```yaml
# kustomization.yaml
replacements:
  - source:
      kind: ConfigMap
      name: app-config
      fieldPath: data.DB_HOST
    targets:
      - select:
          kind: Deployment
          name: myapp
        fieldPaths:
          - spec.template.spec.containers.[name=myapp].env.[name=DB_HOST].value
```

This copies the `DB_HOST` value from the ConfigMap and puts it into the Deployment's environment variable. No duplication!

### 8. Components (Reusable Add-ons)

**Components** are reusable pieces of configuration that can be included in any overlay:

```
my-app/
├── base/
│   └── ...
├── components/
│   ├── monitoring/           # Add Prometheus annotations
│   │   └── kustomization.yaml
│   ├── tls/                  # Add TLS configuration
│   │   └── kustomization.yaml
│   └── rate-limiting/        # Add rate limiting annotations
│       └── kustomization.yaml
└── overlays/
    ├── dev/
    │   └── kustomization.yaml   # Uses: base + monitoring
    └── production/
        └── kustomization.yaml   # Uses: base + monitoring + tls + rate-limiting
```

```yaml
# components/monitoring/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1alpha1
kind: Component

patches:
  - patch: |-
      apiVersion: apps/v1
      kind: Deployment
      metadata:
        name: myapp
      spec:
        template:
          metadata:
            annotations:
              prometheus.io/scrape: "true"
              prometheus.io/port: "8080"
              prometheus.io/path: "/metrics"
```

```yaml
# overlays/production/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

resources:
  - ../../base

components:
  - ../../components/monitoring
  - ../../components/tls
  - ../../components/rate-limiting
```

## 🧪 Practical Example: Full Multi-Environment Setup

Let's build a complete real-world Kustomize project for a Python API:

### Base

```yaml
# base/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

resources:
  - deployment.yaml
  - service.yaml

configMapGenerator:
  - name: api-config
    literals:
      - APP_NAME=my-python-api
      - LOG_FORMAT=json
```

```yaml
# base/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: python-api
spec:
  replicas: 1
  selector:
    matchLabels:
      app: python-api
  template:
    metadata:
      labels:
        app: python-api
    spec:
      containers:
      - name: api
        image: mycompany/python-api:latest
        ports:
        - containerPort: 8000
        envFrom:
        - configMapRef:
            name: api-config
        resources:
          requests:
            cpu: 100m
            memory: 128Mi
          limits:
            cpu: 200m
            memory: 256Mi
        livenessProbe:
          httpGet:
            path: /health
            port: 8000
          initialDelaySeconds: 10
        readinessProbe:
          httpGet:
            path: /ready
            port: 8000
          initialDelaySeconds: 5
```

```yaml
# base/service.yaml
apiVersion: v1
kind: Service
metadata:
  name: python-api
spec:
  selector:
    app: python-api
  ports:
  - port: 80
    targetPort: 8000
  type: ClusterIP
```

### Dev Overlay

```yaml
# overlays/dev/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

resources:
  - ../../base

namePrefix: dev-
namespace: dev

commonLabels:
  env: dev

configMapGenerator:
  - name: api-config
    behavior: merge              # Merge with base ConfigMap (not replace!)
    literals:
      - LOG_LEVEL=debug
      - DB_HOST=dev-postgres.dev.svc.cluster.local

images:
  - name: mycompany/python-api
    newTag: "dev-latest"
```

### Staging Overlay

```yaml
# overlays/staging/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

resources:
  - ../../base
  - ingress.yaml

namePrefix: staging-
namespace: staging

commonLabels:
  env: staging

configMapGenerator:
  - name: api-config
    behavior: merge
    literals:
      - LOG_LEVEL=info
      - DB_HOST=staging-postgres.staging.svc.cluster.local

images:
  - name: mycompany/python-api
    newTag: "v1.5.0-rc1"

patches:
  - patch: |-
      apiVersion: apps/v1
      kind: Deployment
      metadata:
        name: python-api
      spec:
        replicas: 3
        template:
          spec:
            containers:
            - name: api
              resources:
                requests:
                  cpu: 250m
                  memory: 256Mi
                limits:
                  cpu: 500m
                  memory: 512Mi
```

```yaml
# overlays/staging/ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: python-api
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  rules:
  - host: staging-api.mycompany.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: python-api       # Kustomize auto-updates with namePrefix!
            port:
              number: 80
```

### Production Overlay

```yaml
# overlays/production/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

resources:
  - ../../base
  - ingress.yaml
  - hpa.yaml
  - pdb.yaml                    # PodDisruptionBudget for high availability

namePrefix: prod-
namespace: production

commonLabels:
  env: production

commonAnnotations:
  owner: platform-team@mycompany.com

  - name: api-config
    behavior: merge
    literals:
      - LOG_LEVEL=warn
      - DB_HOST=prod-postgres.production.svc.cluster.local

  - name: api-secrets
    literals:
      - DB_PASSWORD=prod-super-secret-password
      - API_KEY=sk-prod-1234567890

images:
  - name: mycompany/python-api
    newTag: "v1.4.2"

patches:
  - path: deployment-patch.yaml
```

```yaml
# overlays/production/deployment-patch.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: python-api
spec:
  replicas: 5
  template:
    spec:
      containers:
      - name: api
        resources:
          requests:
            cpu: 500m

          matchLabels:
            app: python-api
```

```yaml
  maxReplicas: 25

  - type: Resource
```

```yaml
# overlays/production/ingress.yaml

kind: Ingress
metadata:
  name: python-api
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
    nginx.ingress.kubernetes.io/rate-limit: "100"
    cert-manager.io/cluster-issuer: letsencrypt-prod
spec:
  ingressClassName: nginx
  rules:
  - host: api.mycompany.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: python-api
            port:
              number: 80
  tls:
  - hosts:
    - api.mycompany.com
    secretName: api-tls-cert
```

### Deploy!

```bash
# Preview dev
kubectl kustomize overlays/dev

# Preview production
kubectl kustomize overlays/production

# Apply dev
kubectl apply -k overlays/dev

# Apply staging
kubectl apply -k overlays/staging

# Apply production
kubectl apply -k overlays/production

# Delete everything from an overlay
kubectl delete -k overlays/dev
```

### GitHub Actions Example

```yaml
# .github/workflows/deploy.yaml
name: Deploy

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4

    - name: Set up kubectl

    - name: Build and push image
      run: |
        docker build -t mycompany/python-api:${{ github.sha }} .
        docker push mycompany/python-api:${{ github.sha }}

    - name: Update image tag in Kustomize
      run: |
        cd overlays/production
        kustomize edit set image mycompany/python-api=mycompany/python-api:${{ github.sha }}

    - name: Deploy to production
      run: |
        kubectl apply -k overlays/production

    # OR — for GitOps, commit the change and let ArgoCD/Flux deploy it:
    - name: Commit image tag update
      run: |
        git config user.name "GitHub Actions"
        git config user.email "actions@github.com"
        git add overlays/production/kustomization.yaml
        git commit -m "Update image to ${{ github.sha }}"
        git push
```

## ⚠️ Common Pitfalls & Tips

| Pitfall | Solution |
|---------|---------|
| Patch not working | Make sure `metadata.name` in the patch matches the base resource **exactly** |
| ConfigMap changes not triggering Pod restarts | Use `configMapGenerator` instead of plain ConfigMap — it adds hash suffixes |
| `namePrefix` breaking cross-references | Kustomize auto-updates most references, but check HPA `scaleTargetRef` and Ingress `backend.service.name` |
| Wanting to remove a field | Use JSON Patch with `op: remove` — strategic merge patches can't easily remove fields |
| Overlay not finding the base | Use correct relative path in `resources: [../../base]` |
| Labels mismatch after adding `commonLabels` | `commonLabels` adds labels to BOTH `metadata.labels` AND `selector.matchLabels` — this can cause issues if you add labels after initial deployment |
| Secrets in Git | Use sealed-secrets, SOPS, or external-secrets operator — don't commit raw secrets |

## 🤝 Kustomize + Helm Together

You can use both! This is actually a very common pattern:

```yaml
# kustomization.yaml
# Use Helm to render a chart, then customize with Kustomize!

helmCharts:
  - name: ingress-nginx
    repo: https://kubernetes.github.io/ingress-nginx
    version: 4.7.1
    releaseName: ingress
    namespace: ingress-nginx
    valuesFile: nginx-values.yaml

# Then apply Kustomize patches on top!
patches:
  - target:
      kind: Deployment
      name: ingress-ingress-nginx-controller
    patch: |-
      - op: add
        path: /spec/template/metadata/annotations/custom-annotation
        value: "my-value"
```

Or the more common approach — render Helm first, then Kustomize:

```bash
# Step 1: Render Helm chart to plain YAML
helm template my-release bitnami/nginx -f values.yaml > base/nginx-rendered.yaml

# Step 2: Use Kustomize to customize it
# kustomization.yaml
# resources:
#   - nginx-rendered.yaml
# patches:
#   - ...

kubectl apply -k .
```

## 🧠 Quick Recap

```
Kustomize = Template-Free Configuration Management

Philosophy: Keep YAML clean. Customize through patches and overlays.

Structure:
├── base/                    → Standard manifests (shared by all envs)
│   ├── kustomization.yaml   → Lists resources, generators, common config
│   ├── deployment.yaml      → Plain, valid Kubernetes YAML
│   └── service.yaml
└── overlays/                → Environment-specific customizations
    ├── dev/                 → Patches for dev (1 replica, debug mode)
    ├── staging/             → Patches for staging (3 replicas, staging domain)
    └── production/          → Patches for prod (5 replicas, HPA, TLS)

Key Features:
├── images           → Change image name/tag/digest without editing YAML
├── namePrefix       → Add prefix to all resource names
├── namespace        → Set namespace for all resources
├── commonLabels     → Add labels to everything
├── patches          → Strategic merge or JSON patches
├── configMapGenerator → Generate ConfigMaps with hash suffixes
├── secretGenerator  → Generate Secrets with hash suffixes
├── components       → Reusable configuration add-ons
└── replacements     → Copy values between resources

Patch Types:
├── Strategic Merge Patch → Write partial YAML, merge with base
├── JSON Patch           → Precise operations (add/remove/replace)
└── Inline Patch         → Write patches directly in kustomization.yaml

Commands:
├── kubectl apply -k ./path    → Apply a Kustomize directory
├── kubectl kustomize ./path   → Preview rendered YAML
├── kustomize edit set image   → Update image tag programmatically
└── kubectl diff -k ./path     → Preview changes before applying

When to use Kustomize:
✅ Your own apps with clean YAML
✅ Simple to moderate customization needs
✅ GitOps workflows (ArgoCD, Flux)
✅ When you want valid, readable YAML

When to use Helm instead:
✅ Third-party apps (install Prometheus, Nginx Ingress, etc.)
✅ Complex apps needing packaging, versioning, rollback
✅ Sharing charts across teams/organizations
✅ When you need hooks or dependencies
```
