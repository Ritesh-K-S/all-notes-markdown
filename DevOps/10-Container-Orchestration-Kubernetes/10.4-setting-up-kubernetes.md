## 🤔 The Big Picture — Why So Many Options?

Kubernetes is a **complex system** — it has a control plane, worker nodes, networking, storage, and more. Setting it all up from scratch is like building a car from individual parts.

So the community and cloud providers created **different tools for different needs**:

```
┌──────────────────────────────────────────────────────────────────┐
│                  How Do You Want Your Kubernetes?                 │
│                                                                  │
│   🏠 LOCAL (Learning/Dev)     🏗️ SELF-MANAGED      ☁️ MANAGED    │
│   ├── Minikube                ├── kubeadm           ├── EKS      │
│   ├── Kind                    ├── k3s (also local)  ├── AKS      │
│   └── k3s                    └── kops              └── GKE      │
│                                                                  │
│   Easy ◄──────────────────────────────────────────────► Complex  │
│   Free                                              Costs Money  │
│   Limited                                           Production   │
└──────────────────────────────────────────────────────────────────┘
```

**Which one should you use?**

| Your Goal | Best Option |
|-----------|-------------|
| Learning Kubernetes on your laptop | Minikube or Kind |
| CI/CD pipeline testing | Kind |
| Lightweight production (Edge, IoT, small servers) | k3s |
| Full production on bare-metal / VMs | kubeadm |
| Production on AWS | EKS |
| Production on Azure | AKS |
| Production on Google Cloud | GKE |

## 📋 Prerequisites — What You Need Before Starting

Before setting up any Kubernetes cluster, make sure you have:

| Requirement | Why |
|-------------|-----|
| **Docker** (or another container runtime) | Kubernetes runs containers — it needs a runtime |
| **kubectl** | The CLI tool to talk to your Kubernetes cluster |
| **Enough resources** | At least 2 CPU cores and 2 GB RAM for local clusters |
| **A terminal** | Bash, Zsh, PowerShell — whatever you're comfortable with |

### Install kubectl First (You'll Need It for Every Setup)

**On Linux:**
```bash
# Download the latest kubectl binary
curl -LO "https://dl.k8s.io/release/$(curl -L -s https://dl.k8s.io/release/stable.txt)/bin/linux/amd64/kubectl"

# Make it executable
chmod +x kubectl

# Move it to your PATH
sudo mv kubectl /usr/local/bin/

# Verify the installation
kubectl version --client
```

**Output:**
```
Client Version: v1.31.0
Kustomize Version: v5.4.2
```

**On macOS (using Homebrew):**
```bash
brew install kubectl
```

**On Windows (using Chocolatey):**
```powershell
choco install kubernetes-cli
```

**Or using winget:**
```powershell
winget install -e --id Kubernetes.kubectl
```

### Verify kubectl is working:
```bash
kubectl version --client
# Output: Client Version: v1.31.x
```

## 1️⃣ Minikube — Your First Kubernetes Cluster

### What Is Minikube?

Minikube is the **most popular tool for running Kubernetes locally**. It creates a **single-node Kubernetes cluster** inside a virtual machine (or container) on your laptop.

Think of it like this: instead of needing 5 real servers to set up Kubernetes, Minikube gives you a **miniature version** that fits right on your computer.

```
┌─────────────────────────────────────────────────┐
│               Your Laptop / Desktop              │
│                                                  │
│   ┌──────────────────────────────────────────┐   │
│   │            Minikube VM/Container         │   │
│   │                                          │   │
│   │   ┌──────────────────────────────────┐   │   │
│   │   │     Single-Node K8s Cluster      │   │   │
│   │   │                                  │   │   │
│   │   │   Control Plane + Worker Node    │   │   │
│   │   │   (All-in-one!)                  │   │   │
│   │   │                                  │   │   │
│   │   │   ┌─────┐ ┌─────┐ ┌─────┐       │   │   │
│   │   │   │Pod 1│ │Pod 2│ │Pod 3│       │   │   │
│   │   │   └─────┘ └─────┘ └─────┘       │   │   │
│   │   └──────────────────────────────────┘   │   │
│   └──────────────────────────────────────────┘   │
└─────────────────────────────────────────────────┘
```

### Installing Minikube

**On Linux:**
```bash
# Download the Minikube binary
curl -LO https://storage.googleapis.com/minikube/releases/latest/minikube-linux-amd64

# Install it
sudo install minikube-linux-amd64 /usr/local/bin/minikube

# Verify
minikube version
```

**On macOS:**
```bash
brew install minikube
```

**On Windows:**
```powershell
choco install minikube
# OR
winget install minikube
```

### Starting Your First Cluster

```bash
# Start a Minikube cluster (uses Docker driver by default)
minikube start
```

**Output:**
```

✨  Automatically selected the docker driver
📌  Using Docker driver with root privileges
👍  Starting control plane node minikube in cluster minikube
🚜  Pulling base image ...
🔥  Creating docker container (CPUs=2, Memory=2200MB) ...
🐳  Preparing Kubernetes v1.30.0 on Docker 26.0.1 ...
    ▪ Generating certificates and keys ...
```

**What just happened:**
1. Minikube downloaded a Docker image with Kubernetes pre-installed
2. It created a Docker container (or VM) on your machine

```bash
minikube status
```

**Output:**

**Output:**
```

```

You have a **single node** that acts as both the control plane and the worker!

### Useful Minikube Commands

```bash
# Stop the cluster (saves state — like hibernating your computer)
minikube stop

# Start it again (resumes from where you left off)
minikube start

# Delete the cluster completely (fresh start)
minikube delete

# Check what addons are available
minikube addons list

# Enable the dashboard (a web UI for Kubernetes!)
minikube addons enable dashboard

# Open the dashboard in your browser
minikube dashboard

# SSH into the Minikube node
minikube ssh

# Check the IP address of your Minikube cluster
minikube ip
```

### Minikube Multi-Node Cluster

Want to simulate a real multi-node cluster? Minikube can do that too!

```bash
# Create a cluster with 3 nodes
minikube start --nodes 3

# Check all nodes
kubectl get nodes
```

**Output:**
```
NAME           STATUS   ROLES           AGE   VERSION
minikube       Ready    control-plane   3m    v1.30.0
minikube-m02   Ready    <none>          2m    v1.30.0
minikube-m03   Ready    <none>          1m    v1.30.0
```

### Minikube with Custom Resources

```bash
# Start with more CPU and memory
minikube start --cpus 4 --memory 8192

# Start with a specific Kubernetes version
minikube start --kubernetes-version=v1.29.0

# Start with a specific driver
minikube start --driver=virtualbox    # Uses VirtualBox instead of Docker
minikube start --driver=docker        # Uses Docker (default on most systems)
minikube start --driver=hyperv        # Uses Hyper-V (Windows)
```

### Minikube Profiles — Run Multiple Clusters

```bash
# Create a cluster named "dev"
minikube start -p dev

# Create another cluster named "staging"
minikube start -p staging

# Switch between profiles
minikube profile dev
minikube profile staging

# List all profiles
minikube profile list

# Delete a specific profile
minikube delete -p staging
```

## 2️⃣ Kind (Kubernetes IN Docker) — Clusters for Testing

### What Is Kind?

Kind stands for **K**ubernetes **IN** **D**ocker. It runs Kubernetes clusters using **Docker containers as nodes**. Each "node" is actually a Docker container pretending to be a server.

Kind was originally designed for **testing Kubernetes itself**, but it's become super popular for:
- CI/CD pipeline testing
- Local development
- Quick throwaway clusters

```
┌─────────────────────────────────────────────────────────────┐
│                     Your Machine (Docker Host)               │
│                                                              │
│   ┌────────────────────┐   ┌──────────────┐ ┌──────────────┐│
│   │  Docker Container  │   │Docker Cont.  │ │Docker Cont.  ││
│   │  (Control Plane)   │   │(Worker #1)   │ │(Worker #2)   ││
│   │                    │   │              │ │              ││
│   │  ┌──────────────┐  │   │ ┌──────────┐ │ │ ┌──────────┐ ││
│   │  │ API Server   │  │   │ │ kubelet  │ │ │ │ kubelet  │ ││
│   │  │ etcd         │  │   │ │ pods...  │ │ │ │ pods...  │ ││
│   │  │ Scheduler    │  │   │ └──────────┘ │ │ └──────────┘ ││
│   │  │ Controller   │  │   │              │ │              ││
│   │  └──────────────┘  │   └──────────────┘ └──────────────┘│
│   └────────────────────┘                                     │
└─────────────────────────────────────────────────────────────┘
```

### Why Kind Over Minikube?

| Feature | Minikube | Kind |
|---------|----------|------|
| Speed | Moderate | **Very Fast** |
| Multi-node | Supported (newer versions) | **Built-in from day one** |
| CI/CD friendly | Somewhat | **Designed for CI/CD** |
| Resource usage | Higher (full VM option) | **Lower (Docker containers)** |
| Addons/Dashboard | Built-in addons | Manual setup |
| Best for | Learning, development | **Testing, CI/CD** |

### Installing Kind

**On Linux/macOS:**
```bash
# Using Go (if you have Go installed)
go install sigs.k8s.io/kind@v0.23.0

# Or download the binary directly
# For Linux:
curl -Lo ./kind https://kind.sigs.k8s.io/dl/v0.23.0/kind-linux-amd64
chmod +x ./kind
sudo mv ./kind /usr/local/bin/kind

# For macOS:
brew install kind
```

**On Windows:**
```powershell
choco install kind
```

### Creating a Cluster

```bash
# Create a basic single-node cluster
kind create cluster
```

**Output:**
```
Creating cluster "kind" ...
 ✓ Ensuring node image (kindest/node:v1.30.0) 🖼
 ✓ Preparing nodes 📦
 ✓ Writing configuration 📜
 ✓ Starting control-plane 🕹️
 ✓ Installing CNI 🔌
 ✓ Installing StorageClass 💾
Set kubectl context to "kind-kind"
You can now use your cluster with:

kubectl cluster-info --context kind-kind
```

**That's it!** A cluster in about 30 seconds!

### Creating a Multi-Node Cluster with a Config File

This is where Kind really shines. You define your cluster in a YAML file:

```yaml
# kind-config.yaml
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
nodes:
  - role: control-plane      # 1 control plane node
  - role: worker              # Worker node 1
  - role: worker              # Worker node 2
  - role: worker              # Worker node 3
```

```bash
# Create the cluster using the config file
kind create cluster --config kind-config.yaml --name my-cluster
```

**Output:**
```
Creating cluster "my-cluster" ...
 ✓ Ensuring node image (kindest/node:v1.30.0) 🖼
 ✓ Preparing nodes 📦 📦 📦 📦
 ✓ Writing configuration 📜
 ✓ Starting control-plane 🕹️
 ✓ Installing CNI 🔌
 ✓ Installing StorageClass 💾
 ✓ Joining worker nodes 🚜
Set kubectl context to "kind-my-cluster"
```

```bash
# Check the nodes
kubectl get nodes
```

**Output:**
```
NAME                       STATUS   ROLES           AGE   VERSION
my-cluster-control-plane   Ready    control-plane   2m    v1.30.0
my-cluster-worker          Ready    <none>          90s   v1.30.0
my-cluster-worker2         Ready    <none>          90s   v1.30.0
my-cluster-worker3         Ready    <none>          90s   v1.30.0
```

### High Availability (HA) Cluster with Kind

```yaml
# kind-ha-config.yaml
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
nodes:
  - role: control-plane    # Control Plane 1
  - role: control-plane    # Control Plane 2
  - role: control-plane    # Control Plane 3
  - role: worker           # Worker 1
  - role: worker           # Worker 2
  - role: worker           # Worker 3
```

```bash
kind create cluster --config kind-ha-config.yaml --name ha-cluster
```

Now you have a **3-control-plane, 3-worker** cluster — just like a real production setup!

### Port Mapping in Kind

To access services running in Kind from your host:

```yaml
# kind-with-ports.yaml
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
nodes:
  - role: control-plane
    extraPortMappings:
      - containerPort: 30000     # NodePort inside the cluster
        hostPort: 8080           # Port on your machine
        protocol: TCP
  - role: worker
```

### Loading Local Docker Images into Kind

Kind runs in Docker containers, so it **can't see your local Docker images** by default. You need to explicitly load them:

```bash
# Build your image locally
docker build -t my-app:latest .

# Load it into the Kind cluster
kind load docker-image my-app:latest --name my-cluster

# Now you can use "my-app:latest" in your K8s deployments!
```

This is **super useful** in CI/CD pipelines — build the image, load it into Kind, test it, done!

### Managing Kind Clusters

```bash
# List all Kind clusters
kind get clusters

# Delete a cluster
kind delete cluster --name my-cluster

# Delete the default cluster
kind delete cluster

# Export the kubeconfig
kind get kubeconfig --name my-cluster > kubeconfig.yaml

# Export logs for debugging
kind export logs --name my-cluster ./kind-logs/
```

## 3️⃣ k3s — Lightweight Kubernetes for Everything

### What Is k3s?

k3s is a **lightweight, certified Kubernetes distribution** created by Rancher (now part of SUSE). It's called "k3s" because it's **half the size of k8s** (Kubernetes — k + 8 letters + s).

k3s strips down Kubernetes by:
- Replacing **etcd** with **SQLite** (by default) — much simpler
- Bundling everything into a **single binary** (less than 100 MB!)
- Removing legacy/alpha features you probably don't need
- Including common stuff like a **load balancer** and **ingress controller** built-in

```
┌──────────────────────────────────────────────────────┐
│               k3s vs Standard Kubernetes              │
│                                                       │
│   Standard K8s:                k3s:                   │
│   ┌──────────────┐             ┌──────────────┐       │
│   │ API Server   │             │              │       │
│   │ etcd         │    →→→      │   Single     │       │
│   │ Scheduler    │             │   Binary     │       │
│   │ Controller   │             │   (~100 MB)  │       │
│   │ Cloud Ctrl   │             │              │       │
│   │ kube-proxy   │             │ Everything   │       │
│   │ kubelet      │             │ included!    │       │
│   └──────────────┘             └──────────────┘       │
│   ~500+ MB                     ~100 MB                │
└──────────────────────────────────────────────────────┘
```

### Where Is k3s Used?

| Use Case | Why k3s? |
|----------|----------|
| Edge computing | Tiny footprint, runs on ARM |
| IoT devices | Works on Raspberry Pi! |
| Development | Quick to start, easy to use |
| Small production clusters | All you need, nothing you don't |
| CI/CD testing | Fast cluster creation |

### Installing k3s

**On the Server (Master Node):**
```bash
# One single command to install k3s!
curl -sfL https://get.k3s.io | sh -
```

**Output:**
```
[INFO]  Finding release for channel stable
[INFO]  Using v1.30.0+k3s1 as release
[INFO]  Downloading hash https://github.com/k3s-io/k3s/...
[INFO]  Installing k3s to /usr/local/bin/k3s
[INFO]  systemd: Creating service file
[INFO]  systemd: Enabling k3s unit
[INFO]  systemd: Starting k3s
```

**That's literally it!** One command, and you have a fully functional Kubernetes cluster!

```bash
# Check if it's running
sudo k3s kubectl get nodes
```

**Output:**
```
NAME         STATUS   ROLES                  AGE   VERSION
my-server    Ready    control-plane,master   30s   v1.30.0+k3s1
```

### Accessing k3s with Regular kubectl

By default, k3s puts its kubeconfig at `/etc/rancher/k3s/k3s.yaml`. To use regular `kubectl`:

```bash
# Copy the kubeconfig to the default location
sudo cp /etc/rancher/k3s/k3s.yaml ~/.kube/config
sudo chown $USER:$USER ~/.kube/config

# Now you can use regular kubectl!
kubectl get nodes
```

### Adding Worker Nodes to k3s

On the **master node**, get the token:
```bash
sudo cat /var/lib/rancher/k3s/server/node-token
```

On each **worker node**, run:
```bash
# Replace <MASTER_IP> and <TOKEN> with actual values
curl -sfL https://get.k3s.io | K3S_URL=https://<MASTER_IP>:6443 K3S_TOKEN=<TOKEN> sh -
```

```bash
# Back on the master, check all nodes
kubectl get nodes
```

**Output:**
```
NAME         STATUS   ROLES                  AGE   VERSION
master       Ready    control-plane,master   5m    v1.30.0+k3s1
worker-1     Ready    <none>                 2m    v1.30.0+k3s1
worker-2     Ready    <none>                 1m    v1.30.0+k3s1
```

### Uninstalling k3s

```bash
# On the server (master)
/usr/local/bin/k3s-uninstall.sh

# On worker nodes
/usr/local/bin/k3s-agent-uninstall.sh
```

## 4️⃣ kubeadm — The Official Way to Set Up Production Clusters

### What Is kubeadm?

kubeadm is the **official Kubernetes tool** for bootstrapping (setting up) production-grade clusters. It's like the "official recipe" for building a Kubernetes cluster from scratch.

kubeadm handles:
- ✅ Initializing the control plane
- ✅ Setting up networking
- ✅ Generating certificates
- ✅ Joining worker nodes

kubeadm does **NOT** handle:
- ❌ Provisioning machines (you need VMs/servers ready)
- ❌ Installing Docker or container runtime (you do this first)
- ❌ Setting up networking plugins (you choose and install a CNI)
- ❌ Managing the cluster after setup (that's your job)

```
┌──────────────────────────────────────────────────────────────┐
│                 kubeadm Cluster Setup Flow                    │
│                                                               │
│   Step 1: Prepare machines (VMs, bare-metal)                  │
│       ↓                                                       │
│   Step 2: Install container runtime (containerd)              │
│       ↓                                                       │
│   Step 3: Install kubeadm, kubelet, kubectl on ALL nodes      │
│       ↓                                                       │
│   Step 4: kubeadm init (on master node)                       │
│       ↓                                                       │
│   Step 5: Install CNI plugin (Calico, Flannel, Cilium)        │
│       ↓                                                       │
│   Step 6: kubeadm join (on each worker node)                  │
│       ↓                                                       │
│   ✅ Cluster is ready!                                         │
└──────────────────────────────────────────────────────────────┘
```

### Step 1: Prepare Your Machines

You need at least:
- **1 machine** for the control plane (2+ CPU, 2+ GB RAM)
- **1+ machines** for worker nodes

These can be:
- Cloud VMs (AWS EC2, Azure VMs, GCP Compute Engine)
- On-premise servers
- VirtualBox / Vagrant VMs (for practice)

### Step 2: Prerequisites on ALL Nodes

Run these commands on **every** machine (master + workers):

```bash
# 1. Disable swap (Kubernetes requires this)
sudo swapoff -a
# Make it permanent — comment out swap line in /etc/fstab
sudo sed -i '/ swap / s/^\(.*\)$/#\1/g' /etc/fstab

# 2. Load required kernel modules
cat <<EOF | sudo tee /etc/modules-load.d/k8s.conf
overlay
br_netfilter
EOF

sudo modprobe overlay
sudo modprobe br_netfilter

# 3. Set required sysctl parameters
cat <<EOF | sudo tee /etc/sysctl.d/k8s.conf
net.bridge.bridge-nf-call-iptables  = 1
net.bridge.bridge-nf-call-ip6tables = 1
net.ipv4.ip_forward                 = 1
EOF

sudo sysctl --system
```

### Step 3: Install Container Runtime (containerd)

```bash
# Install containerd
sudo apt-get update
sudo apt-get install -y containerd

# Create default config
sudo mkdir -p /etc/containerd
containerd config default | sudo tee /etc/containerd/config.toml

# Enable SystemdCgroup (important!)
sudo sed -i 's/SystemdCgroup = false/SystemdCgroup = true/g' /etc/containerd/config.toml

# Restart containerd
sudo systemctl restart containerd
sudo systemctl enable containerd
```

### Step 4: Install kubeadm, kubelet, and kubectl

```bash
# Add Kubernetes repository
sudo apt-get update
sudo apt-get install -y apt-transport-https ca-certificates curl gpg

# Add the Kubernetes signing key
curl -fsSL https://pkgs.k8s.io/core:/stable:/v1.30/deb/Release.key | sudo gpg --dearmor -o /etc/apt/keyrings/kubernetes-apt-keyring.gpg

# Add the Kubernetes repository
echo 'deb [signed-by=/etc/apt/keyrings/kubernetes-apt-keyring.gpg] https://pkgs.k8s.io/core:/stable:/v1.30/deb/ /' | sudo tee /etc/apt/sources.list.d/kubernetes.list

# Install the tools
sudo apt-get update
sudo apt-get install -y kubelet kubeadm kubectl

# Prevent them from being auto-updated (important for stability!)
sudo apt-mark hold kubelet kubeadm kubectl
```

### Step 5: Initialize the Control Plane (Master Node Only)

```bash
# Initialize the cluster!
sudo kubeadm init --pod-network-cidr=10.244.0.0/16 --apiserver-advertise-address=<MASTER_IP>
```

Let's break down the flags:
- `--pod-network-cidr=10.244.0.0/16` → The IP range for pods (Flannel uses this range; Calico uses `192.168.0.0/16`)
- `--apiserver-advertise-address=<MASTER_IP>` → The IP address of your master node

**Output (abbreviated):**
```
[init] Using Kubernetes version: v1.30.0
[preflight] Running pre-flight checks
[certs] Generating certificates ...
[kubeconfig] Writing kubeconfig files ...
[control-plane] Creating static Pod manifests ...
[etcd] Creating static Pod manifest for local etcd ...
[wait-control-plane] Waiting for the kubelet to boot up the control plane ...
[kubelet-start] Writing kubelet configuration ...

Your Kubernetes control-plane has initialized successfully!

To start using your cluster, you need to run the following as a regular user:

  mkdir -p $HOME/.kube
  sudo cp -i /etc/kubernetes/admin.conf $HOME/.kube/config
  sudo chown $(id -u):$(id -g) $HOME/.kube/config

Then you can join any number of worker nodes by running the following
on each as root:

  kubeadm join 192.168.1.100:6443 --token abcdef.1234567890abcdef \
    --discovery-token-ca-cert-hash sha256:abc123def456...
```

### Step 6: Set Up kubectl Access

```bash
# Run these commands as a regular user (not root!)
mkdir -p $HOME/.kube
sudo cp -i /etc/kubernetes/admin.conf $HOME/.kube/config
sudo chown $(id -u):$(id -g) $HOME/.kube/config
```

### Step 7: Install a CNI (Container Network Interface) Plugin

Without a CNI plugin, your pods **can't communicate** with each other. Choose one:

**Option A: Calico (Most popular for production)**
```bash
kubectl apply -f https://raw.githubusercontent.com/projectcalico/calico/v3.27.0/manifests/calico.yaml
```

**Option B: Flannel (Simpler)**
```bash
kubectl apply -f https://github.com/flannel-io/flannel/releases/latest/download/kube-flannel.yml
```

**Option C: Cilium (Modern, eBPF-based)**
```bash
# Install Cilium CLI
curl -L --remote-name-all https://github.com/cilium/cilium-cli/releases/latest/download/cilium-linux-amd64.tar.gz
sudo tar xzvf cilium-linux-amd64.tar.gz -C /usr/local/bin
cilium install
```

### Step 8: Join Worker Nodes

On each **worker node**, run the join command from Step 5:

```bash
sudo kubeadm join 192.168.1.100:6443 --token abcdef.1234567890abcdef \
  --discovery-token-ca-cert-hash sha256:abc123def456...
```

**If you lost the join command**, regenerate it on the master:
```bash
kubeadm token create --print-join-command
```

### Step 9: Verify the Cluster

```bash
# On the master node
kubectl get nodes
```

**Output:**
```

worker-1   Ready    <none>          5m    v1.30.0
```

## 5️⃣ Managed Kubernetes — EKS, AKS, GKE

### What Is Managed Kubernetes?

With managed Kubernetes, the **cloud provider handles the control plane** for you. You don't need to worry about:
- Setting up the API server, etcd, scheduler
- Upgrading the control plane
- Making the control plane highly available
- Patching security vulnerabilities in control plane components

**You just focus on deploying your apps!**

```
┌───────────────────────────────────────────────────────────┐
│              Managed Kubernetes Architecture               │
│                                                            │
│   ☁️ Cloud Provider Manages This:                          │
│   ┌──────────────────────────────────┐                     │
│   │         Control Plane            │  ← You don't touch  │
│   │  API Server | etcd | Scheduler   │     this!           │
│   │  Controller Manager              │                     │
│   └──────────────────────────────────┘                     │
│                    │                                        │
│   👨‍💻 You Manage This:                                     │
│   ┌──────────────────────────────────┐                     │
│   │         Worker Nodes             │  ← Your VMs         │
│   │  ┌──────┐ ┌──────┐ ┌──────┐     │                     │
│   │  │Node 1│ │Node 2│ │Node 3│     │                     │
│   │  │Pods  │ │Pods  │ │Pods  │     │                     │

│   └──────────────────────────────────┘                     │
```

### Amazon EKS (Elastic Kubernetes Service)

**What:** AWS's managed Kubernetes service.

**Creating an EKS Cluster:**

**Option A: Using eksctl (recommended — easiest way)**
```bash
# Install eksctl
curl --silent --location "https://github.com/weaveworks/eksctl/releases/latest/download/eksctl_$(uname -s)_amd64.tar.gz" | tar xz -C /tmp
sudo mv /tmp/eksctl /usr/local/bin

# Create a cluster (this takes ~15 minutes)
eksctl create cluster \
  --name my-cluster \
  --region us-east-1 \
  --nodegroup-name my-nodes \
  --node-type t3.medium \
  --nodes 3 \
  --nodes-min 1 \
  --nodes-max 5 \
  --managed
```

Let's break down each flag:
- `--name my-cluster` → Name of your EKS cluster
- `--region us-east-1` → AWS region to deploy in
- `--nodegroup-name my-nodes` → Name for your group of worker nodes
- `--node-type t3.medium` → EC2 instance type (2 vCPUs, 4 GB RAM)
- `--nodes 3` → Start with 3 worker nodes
- `--nodes-min 1` / `--nodes-max 5` → Auto-scaling limits
- `--managed` → Use managed node groups (AWS handles node updates)

**Output:**
```
[ℹ]  eksctl version 0.176.0
[ℹ]  using region us-east-1
[ℹ]  setting availability zones to [us-east-1a us-east-1b]
[ℹ]  will create a CloudFormation stack for cluster itself and 1 nodegroup
[ℹ]  creating EKS cluster "my-cluster" in "us-east-1" region ...
[✔]  EKS cluster "my-cluster" in "us-east-1" region is ready
```

**Option B: Using AWS CLI**
```bash
# Create the EKS cluster
aws eks create-cluster \
  --name my-cluster \
  --role-arn arn:aws:iam::123456789012:role/EKSClusterRole \
  --resources-vpc-config subnetIds=subnet-abc123,subnet-def456,securityGroupIds=sg-xyz789

# Update kubeconfig to connect
aws eks update-kubeconfig --name my-cluster --region us-east-1
```

**Connecting to an existing EKS cluster:**
```bash
# Configure kubectl to talk to your EKS cluster
aws eks update-kubeconfig --name my-cluster --region us-east-1

# Verify connection
kubectl get nodes
```

**Deleting an EKS cluster:**
```bash
eksctl delete cluster --name my-cluster --region us-east-1
```

### Azure AKS (Azure Kubernetes Service)

**What:** Microsoft Azure's managed Kubernetes service.

```bash
# Login to Azure
az login

# Create a resource group
az group create --name myResourceGroup --location eastus

# Create an AKS cluster
az aks create \
  --resource-group myResourceGroup \
  --name myAKSCluster \
  --node-count 3 \
  --node-vm-size Standard_DS2_v2 \
  --enable-managed-identity \
  --generate-ssh-keys
```

Let's break down the flags:
- `--resource-group myResourceGroup` → Azure resource group
- `--name myAKSCluster` → Name of your AKS cluster
- `--node-count 3` → Number of worker nodes
- `--node-vm-size Standard_DS2_v2` → VM size (2 vCPUs, 7 GB RAM)
- `--enable-managed-identity` → Use Azure managed identity for authentication
- `--generate-ssh-keys` → Auto-generate SSH keys for node access

**Connecting to AKS:**
```bash
# Get credentials and configure kubectl
az aks get-credentials --resource-group myResourceGroup --name myAKSCluster

# Verify connection
kubectl get nodes
```

**Deleting an AKS cluster:**

### Google GKE (Google Kubernetes Engine)

**What:** Google Cloud's managed Kubernetes service. Google literally **invented Kubernetes** (it evolved from their internal system called Borg), so GKE is often considered the **most mature** managed K8s offering.

```bash
# Login to Google Cloud
gcloud auth login

# Set your project
gcloud config set project my-project-id

# Create a GKE cluster
gcloud container clusters create my-cluster \
  --zone us-central1-a \
  --num-nodes 3 \
  --machine-type e2-medium \
  --enable-autoscaling \
  --min-nodes 1 \
  --max-nodes 5
```

Let's break down the flags:
- `--zone us-central1-a` → Google Cloud zone
- `--num-nodes 3` → Number of nodes per zone
- `--machine-type e2-medium` → VM type (2 vCPUs, 4 GB RAM)
- `--enable-autoscaling` → Enable cluster auto-scaling
- `--min-nodes` / `--max-nodes` → Auto-scaling range

**GKE Autopilot (fully managed — even the nodes!):**
```bash
gcloud container clusters create-auto my-autopilot-cluster \
  --region us-central1
```

With Autopilot, Google manages **everything** — you just deploy pods and pay per pod resource usage!

**Connecting to GKE:**
```bash
# Get credentials
gcloud container clusters get-credentials my-cluster --zone us-central1-a

# Verify
kubectl get nodes
```

**Deleting a GKE cluster:**
```bash
gcloud container clusters delete my-cluster --zone us-central1-a
```

## 🔧 Switching Between Multiple Clusters (Contexts)

When you have multiple clusters, you need to **switch between them**. This is done using **kubectl contexts**.

```bash
# See all available contexts
kubectl config get-contexts
```

**Output:**
```
CURRENT   NAME                 CLUSTER              AUTHINFO             NAMESPACE
*         minikube             minikube             minikube             default
          kind-my-cluster      kind-my-cluster      kind-my-cluster      
          arn:aws:eks:...      my-eks-cluster       my-eks-user          
```

The `*` shows which cluster you're currently connected to.

```bash
# Switch to a different cluster
kubectl config use-context kind-my-cluster

# Now all kubectl commands go to the Kind cluster!
kubectl get nodes    # Shows Kind cluster nodes

# Switch back to Minikube
kubectl config use-context minikube

# Rename a context for convenience
kubectl config rename-context arn:aws:eks:us-east-1:123456:cluster/my-cluster eks-prod
```

### Using kubectx (Makes Switching Even Easier)

```bash
# Install kubectx
# On macOS:
brew install kubectx

# On Linux:
sudo apt-get install kubectx

# List all contexts
kubectx
# Output:
# minikube
# kind-my-cluster
# eks-prod

# Switch context
kubectx kind-my-cluster

# Switch back to the previous context
kubectx -
```

## 🎯 My Recommendation — Where to Start

If you're **learning Kubernetes**, here's the perfect order:

1. **Start with Minikube** — Install it, create a cluster, deploy your first app. Minikube is the most beginner-friendly.

2. **Try Kind** — Once comfortable with the basics, use Kind to spin up multi-node clusters. It's faster and helps you understand node-level concepts.

3. **Explore k3s** — Set it up on a VM or a Raspberry Pi. It gives you a feel for real multi-machine setups without the complexity of kubeadm.

4. **Graduate to kubeadm** — When you need to understand how Kubernetes is **really** set up from scratch. This knowledge is crucial for CKA (Certified Kubernetes Administrator) exam.

5. **Go Cloud** — Try a managed service (GKE has a generous free tier, AKS control plane is free). This is what you'll use in real-world production jobs.

## 📝 Key Takeaways

| Takeaway | Details |
|----------|---------|
| Start with **Minikube** or **Kind** | Perfect for learning on your laptop |
| **k3s** is great for lightweight production | Single binary, runs anywhere |
| **kubeadm** is the official production setup tool | Know this for the CKA exam |
| **Managed K8s** (EKS/AKS/GKE) is what companies use | Control plane is handled for you |
| **kubectl** works with ALL of these | Learn it once, use it everywhere |
| Use **contexts** to switch between clusters | `kubectl config use-context <name>` |
