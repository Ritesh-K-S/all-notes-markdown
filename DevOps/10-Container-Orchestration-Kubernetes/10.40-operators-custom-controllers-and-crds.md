# 10.40 — Operators — Custom Controllers & CRDs

> **One Line Answer:** A Kubernetes Operator is a **custom controller** that uses **Custom Resource Definitions (CRDs)** to automate the management of complex applications — essentially teaching Kubernetes how to run, scale, backup, and heal applications that it doesn't natively understand.

## 🤔 What Problem Do Operators Solve?

Let's think about this with a real-world scenario.

You want to run a **PostgreSQL database** on Kubernetes. Sure, you can create a Deployment, a Service, a PersistentVolumeClaim, and get it running. But then what?

- Who creates the **replicas** for high availability?
- Who handles **failover** when the primary database crashes?
- Who takes **backups** every night at 2 AM?
- Who performs a **version upgrade** from PostgreSQL 14 → 15 without downtime?
- Who manages **connection pooling**, **replication lag monitoring**, and **recovery**?

**None of this is built into Kubernetes.** Kubernetes knows how to run containers, restart them, and scale them. But it has **no idea** what PostgreSQL is or how to manage it properly.

This is where **Operators** come in.

> An Operator packages **human operational knowledge** into software that Kubernetes can execute automatically.

Think of it like this:

| Without Operator | With Operator |
|-----------------|---------------|
| You manually set up replication | Operator configures replicas automatically |
| You write cron jobs for backups | Operator handles scheduled backups |
| You SSH in and fix things when they break | Operator detects issues and self-heals |
| You read docs for upgrade procedures | Operator performs rolling upgrades safely |
| You are the DBA on-call at 3 AM | Operator is the 24/7 automated DBA |

## 📦 What Exactly is an Operator?

An Operator has **two main components**:

### 1. Custom Resource Definition (CRD)
This **extends** the Kubernetes API by creating a new "kind" of resource. Just like Kubernetes has built-in resources like `Pod`, `Service`, `Deployment` — a CRD lets you create your own custom resources like `PostgresCluster`, `RedisCluster`, `KafkaTopic`, etc.

### 2. Custom Controller
This is a program (usually a Pod running in your cluster) that **watches** your custom resources and takes action whenever they are created, updated, or deleted.

```
┌─────────────────────────────────────────────────────────┐
│                   OPERATOR = CRD + Controller            │
│                                                          │
│  ┌──────────────┐         ┌─────────────────────────┐    │
│  │     CRD      │         │   Custom Controller     │    │
│  │              │         │                         │    │
│  │ Defines new  │         │ Watches CRD resources   │    │
│  │ resource     │◄───────►│ and takes action:       │    │
│  │ types in K8s │         │  • Create Pods          │    │
│  │ API          │         │  • Setup replication    │    │
│  │              │         │  • Handle backups       │    │
│  │ e.g.,        │         │  • Perform upgrades     │    │
│  │ PostgresDB   │         │  • Self-heal            │    │
│  └──────────────┘         └─────────────────────────┘    │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### Real-World Analogy

Think of a **smart home system**:

- **CRD** = The "smart device" you add (like a smart thermostat, smart lock, smart camera). You're extending what your home can do.
- **Custom Controller** = The **automation rules** that act on those devices. "If temperature drops below 20°C → turn on heater", "If door unlocked after 11 PM → send alert."

The devices (CRDs) define **what exists**. The rules (controller) define **what happens**.

**An Operator is both the device AND the rules combined.**

## 🔄 The Operator Pattern — How It Works

Every Operator follows the same fundamental pattern called the **Reconciliation Loop** (also called the **Control Loop**):

```
     ┌────────────────────┐
     │   User creates/    │
     │   updates a CR     │
     │ (Custom Resource)  │
     └────────┬───────────┘
              │
              ▼
     ┌────────────────────┐
     │   Controller       │
     │   OBSERVES the     │
     │   current state    │
     │   (What IS)        │
     └────────┬───────────┘
              │
              ▼
     ┌────────────────────┐
     │   Controller       │
     │   COMPARES with    │
     │   desired state    │
     │   (What SHOULD be) │
     └────────┬───────────┘
              │
              ▼
     ┌────────────────────┐
     │   Controller       │
     │   ACTS to make     │
     │   current = desired│
     │   (Reconcile)      │
     └────────┬───────────┘
              │
              └──────────► Loop repeats continuously
```

### Step-by-Step:

1. **Observe** — The controller watches the Kubernetes API for changes to its custom resources.
2. **Analyze** — It compares the **current state** (what's actually running) vs. the **desired state** (what the CR says should be running).
3. **Act** — If there's a difference, it takes action to **reconcile** — create Pods, update configs, trigger backups, etc.
4. **Repeat** — This loop runs forever. The controller is always checking and correcting.

This is the exact same pattern that built-in Kubernetes controllers use:
- The **Deployment controller** watches Deployments and ensures the right number of ReplicaSets exist.
- The **ReplicaSet controller** watches ReplicaSets and ensures the right number of Pods exist.

Operators simply apply this same pattern to **your custom applications**.

## 📝 Let's See It in Action — A Simple Example

Let's say someone built a **PostgreSQL Operator**. Here's what using it looks like:

### Step 1: Install the CRD

First, the CRD gets installed (usually comes with the Operator). This tells Kubernetes about a new resource type:

```yaml
# This is a simplified CRD (the actual ones are much longer)
apiVersion: apiextensions.k8s.io/v1
kind: CustomResourceDefinition
metadata:
  name: postgresclusters.postgres.example.com   # Plural name + group
spec:
  group: postgres.example.com                   # API group
  names:
    kind: PostgresCluster                       # The "kind" you'll use in YAML
    plural: postgresclusters                    # kubectl get postgresclusters
    singular: postgrescluster
    shortNames:
      - pg                                      # kubectl get pg (shortcut!)
  scope: Namespaced                             # Can be per-namespace
  versions:
    - name: v1
      served: true                              # This version is available via API
      storage: true                             # This version is stored in etcd
      schema:
        openAPIV3Schema:
          type: object
          properties:
            spec:
              type: object
              properties:
                replicas:                        # How many PostgreSQL instances
                  type: integer
                version:                         # PostgreSQL version
                  type: string
                storage:
                  type: object
                  properties:
                    size:                         # Storage size
                      type: string
                backup:
                  type: object
                  properties:
                    schedule:                     # Cron schedule for backups
                      type: string
```

After applying this CRD:
```bash
kubectl apply -f postgres-crd.yaml
```

Kubernetes now **knows about a new resource type** called `PostgresCluster`. You can verify:
```bash
kubectl get crd
# NAME                                    CREATED AT
# postgresclusters.postgres.example.com   2024-01-15T10:30:00Z
```

### Step 2: Deploy the Operator (Controller)

The Operator itself runs as a Deployment inside your cluster:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: postgres-operator
  namespace: operators
spec:
  replicas: 1
  selector:
    matchLabels:
      app: postgres-operator
  template:
    metadata:
      labels:
        app: postgres-operator
    spec:
      serviceAccountName: postgres-operator-sa    # Needs RBAC permissions!
      containers:
        - name: operator
          image: example/postgres-operator:v1.0.0
          # This container runs the controller logic
          # It watches for PostgresCluster resources
          # and creates/manages actual PostgreSQL Pods
```

Along with the necessary RBAC (so the Operator has permission to create Pods, Services, etc.):

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: postgres-operator-role
rules:
  - apiGroups: [""]
    resources: ["pods", "services", "persistentvolumeclaims", "configmaps", "secrets"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
  - apiGroups: ["apps"]
    resources: ["statefulsets"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
  - apiGroups: ["postgres.example.com"]
    resources: ["postgresclusters"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: postgres-operator-binding
subjects:
  - kind: ServiceAccount
    name: postgres-operator-sa
    namespace: operators
roleRef:
  kind: ClusterRole
  name: postgres-operator-role
  apiGroup: rbac.authorization.k8s.io
```

### Step 3: Create Your Custom Resource (CR)

Now the magic! Instead of writing 10+ YAML files for StatefulSets, Services, ConfigMaps, PVCs, CronJobs for backup, etc., you write **one simple YAML**:

```yaml
apiVersion: postgres.example.com/v1
kind: PostgresCluster                # Our custom resource!
metadata:
  name: my-production-db
  namespace: production
spec:
  replicas: 3                        # 1 primary + 2 replicas
  version: "15.4"                    # PostgreSQL 15.4
  storage:
    size: "100Gi"                    # 100GB storage per instance
  backup:
    schedule: "0 2 * * *"            # Backup every night at 2 AM
```

```bash
kubectl apply -f my-postgres.yaml
```

**That's it!** The Operator sees this new `PostgresCluster` resource and automatically:

1. ✅ Creates a **StatefulSet** with 3 PostgreSQL Pods
2. ✅ Sets up **streaming replication** (1 primary → 2 replicas)
3. ✅ Creates **Services** for read/write and read-only connections
4. ✅ Provisions **PersistentVolumeClaims** with 100Gi each
5. ✅ Configures **automatic backups** at 2 AM daily
6. ✅ Sets up **health checks** and **automatic failover**
7. ✅ Continuously **monitors** and **self-heals**

### What Happens Behind the Scenes

```
You apply:                        Operator creates:
                                  
PostgresCluster                   ┌──────────────────────────┐
  replicas: 3      ──────────►    │ StatefulSet (3 replicas) │
  version: 15.4                   │ Service (read-write)     │
  storage: 100Gi                  │ Service (read-only)      │
  backup: 2 AM                   │ PVC × 3 (100Gi each)     │
                                  │ ConfigMap (pg config)    │
                                  │ Secret (passwords)       │
                                  │ CronJob (backups)        │
                                  │ Monitoring endpoints     │
                                  └──────────────────────────┘
```

## 🏗️ Understanding the Controller — The Brain of an Operator

The controller is where all the logic lives. Let me show you a simplified version of what a controller does, written in Go (the most common language for Operators):

```go
// This is simplified pseudocode to show the concept
// Real controllers use controller-runtime library

package main

import (
    "context"
    "fmt"
    
    ctrl "sigs.k8s.io/controller-runtime"
    "sigs.k8s.io/controller-runtime/pkg/client"
)

// PostgresClusterReconciler is our custom controller
type PostgresClusterReconciler struct {
    client.Client   // Kubernetes API client
}

// Reconcile is called whenever a PostgresCluster resource changes
// THIS IS THE HEART OF THE OPERATOR
func (r *PostgresClusterReconciler) Reconcile(ctx context.Context, req ctrl.Request) (ctrl.Result, error) {
    
    // Step 1: OBSERVE — Fetch the PostgresCluster resource
    var pgCluster PostgresCluster
    if err := r.Get(ctx, req.NamespacedName, &pgCluster); err != nil {
        // If the resource was deleted, clean up
        return ctrl.Result{}, client.IgnoreNotFound(err)
    }
    
    // Step 2: ANALYZE — Check what currently exists vs what should exist
    
    // Does the StatefulSet exist?
    existingStatefulSet, err := r.getStatefulSet(ctx, pgCluster)
    if err != nil {
        // StatefulSet doesn't exist — need to create it
        fmt.Println("StatefulSet not found, creating...")
        err = r.createStatefulSet(ctx, pgCluster)
        if err != nil {
            return ctrl.Result{}, err
        }
    }
    
    // Step 3: ACT — Reconcile the differences
    
    // Are the replicas correct?
    if existingStatefulSet.Spec.Replicas != pgCluster.Spec.Replicas {
        fmt.Printf("Scaling from %d to %d replicas\n", 
            existingStatefulSet.Spec.Replicas, pgCluster.Spec.Replicas)
        err = r.scaleStatefulSet(ctx, pgCluster)
        if err != nil {
            return ctrl.Result{}, err
        }
    }
    
    // Is the version correct? (upgrade needed?)
    if existingStatefulSet.CurrentVersion != pgCluster.Spec.Version {
        fmt.Printf("Upgrading from %s to %s\n",
            existingStatefulSet.CurrentVersion, pgCluster.Spec.Version)
        err = r.performRollingUpgrade(ctx, pgCluster)
        if err != nil {
            return ctrl.Result{}, err
        }
    }
    
    // Is replication healthy?
    if !r.isReplicationHealthy(ctx, pgCluster) {
        fmt.Println("Replication unhealthy, repairing...")
        err = r.repairReplication(ctx, pgCluster)
        if err != nil {
            return ctrl.Result{}, err
        }
    }
    
    // Are backups configured?
    err = r.ensureBackupCronJob(ctx, pgCluster)
    if err != nil {
        return ctrl.Result{}, err
    }
    
    // Update the status of the custom resource
    pgCluster.Status.Ready = true
    pgCluster.Status.Replicas = pgCluster.Spec.Replicas
    pgCluster.Status.CurrentVersion = pgCluster.Spec.Version
    r.Status().Update(ctx, &pgCluster)
    
    // Step 4: REPEAT — Return and wait for next trigger
    // The controller will be called again if anything changes
    return ctrl.Result{}, nil
}
```

### Key Takeaways from this Code:

1. **`Reconcile()` is the main function** — It's called every time something changes.
2. **It follows Observe → Analyze → Act** — Get current state, compare with desired, fix differences.
3. **It's idempotent** — You can run it 100 times, and it'll only make changes if needed.
4. **It updates status** — So you can do `kubectl get pg` and see the health of your database.

## 🎯 Operator Capability Levels

Not all Operators are created equal. The **Operator Capability Model** defines 5 levels of maturity:

```
Level 5: Auto Pilot          ★★★★★  ← Full lifecycle automation
Level 4: Deep Insights        ★★★★☆  ← Monitoring, alerting, analytics
Level 3: Full Lifecycle        ★★★☆☆  ← Backup, restore, upgrades
Level 2: Seamless Upgrades     ★★☆☆☆  ← Automated updates, patch management
Level 1: Basic Install         ★☆☆☆☆  ← Automated deployment only
```

### Level by Level:

| Level | Name | What It Does | Example |
|-------|------|-------------|---------|
| **Level 1** | Basic Install | Automates initial deployment | `kubectl apply` and it sets up the app |
| **Level 2** | Seamless Upgrades | Handles version upgrades automatically | Upgrades PostgreSQL 14 → 15 safely |
| **Level 3** | Full Lifecycle | Backup, restore, failure recovery | Nightly backups, point-in-time recovery |
| **Level 4** | Deep Insights | Metrics, alerts, log aggregation | Exposes Prometheus metrics, health dashboards |
| **Level 5** | Auto Pilot | Auto-scaling, auto-tuning, auto-healing | Detects slow queries → adjusts config automatically |

Most Operators you find in the wild are Level 1-3. Level 5 Operators are rare and impressive.

## 🆚 Operator vs. Helm Chart — What's the Difference?

This is a common question. Let me clear it up:

| Aspect | Helm Chart | Operator |
|--------|-----------|----------|
| **What it is** | A package manager / templating tool | A running controller with business logic |
| **When it runs** | Only at install/upgrade time | Continuously — it watches and reacts 24/7 |
| **Day 1 (Deploy)** | ✅ Great — installs all resources | ✅ Great — installs all resources |
| **Day 2 (Operate)** | ❌ Can't handle ongoing operations | ✅ Manages backups, failover, upgrades |
| **Self-healing** | ❌ Kubernetes restarts Pods, that's it | ✅ Application-aware self-healing |
| **Domain knowledge** | ❌ None — it's just templates | ✅ Encodes operational expertise |
| **Complexity** | Low — YAML templates | High — requires writing controller code |
| **Best for** | Stateless apps, simple deployments | Stateful apps, databases, complex systems |

### The Simple Rule:

- **Stateless app** (web server, API, frontend) → **Helm Chart** is usually enough.
- **Stateful app** (database, message queue, cache) → **Operator** is the way to go.

> **Pro Tip:** Many Operators are actually **installed via Helm Charts**! Helm installs the Operator, and then the Operator manages the application. They're complementary, not competing.

## 🔄 How Controllers Talk to the Kubernetes API

Understanding this flow is crucial:

```
┌──────────────┐     Watch      ┌────────────────────┐
│   Controller │◄───────────────│  Kubernetes API     │
│   (Operator) │                │  Server             │
│              │   GET/LIST     │                     │
│              │───────────────►│  Stores resources   │
│              │                │  in etcd            │
│              │  CREATE/UPDATE │                     │
│              │───────────────►│                     │
│              │                │                     │
│              │   DELETE       │                     │
│              │───────────────►│                     │
└──────────────┘                └────────────────────┘
        │
        │ Creates/Manages
        ▼
┌──────────────────┐
│ Pods, Services,  │
│ StatefulSets,    │
│ ConfigMaps, etc. │
└──────────────────┘
```

### The Watch Mechanism

Controllers don't poll the API server every second (that would be wasteful). Instead, they use **watches**:

1. The controller says: "Hey API server, tell me whenever a `PostgresCluster` resource changes."
2. The API server opens a **long-lived HTTP connection** (watch stream).
3. Whenever someone creates, updates, or deletes a `PostgresCluster`, the API server **pushes an event** to the controller.
4. The controller's `Reconcile()` function is triggered.

This is efficient and real-time!

### Informers and Work Queues

In practice, controllers use a more sophisticated mechanism:

```
┌───────────┐    Events     ┌────────────┐    Items    ┌────────────┐
│ API Server│──────────────►│  Informer  │────────────►│ Work Queue │
│           │               │  (Cache)   │             │            │
└───────────┘               └────────────┘             └─────┬──────┘
                                                              │
                                                              │ Dequeue
                                                              ▼
                                                       ┌────────────┐
                                                       │ Reconcile()│
                                                       │  function  │
                                                       └────────────┘
```

- **Informer**: Maintains a local cache of resources to avoid hitting the API server for every read.
- **Work Queue**: Ensures events are processed in order and handles retries if reconciliation fails.

## ⚡ When Should You Write Your Own Operator?

Before you jump into building an Operator, ask yourself:

### ✅ Build an Operator When:

- Your application requires **complex Day 2 operations** (backup, restore, upgrade, scaling with special logic).
- You need **application-specific self-healing** (not just restarting a Pod, but reconfiguring replication).
- You're managing a **stateful application** with complex lifecycle.
- You want to provide a **simple API** for your team to deploy complex systems. (`kubectl apply -f my-database.yaml` instead of 20 manual steps).
- You're building a **platform** and want to offer "as-a-service" experiences (Database-as-a-Service, Cache-as-a-Service).

### ❌ Don't Build an Operator When:

- A **Helm Chart** does the job fine (stateless apps, simple configs).
- You're deploying a **standard off-the-shelf** app that doesn't need custom management.
- An **existing Operator** already exists for your use case (check OperatorHub first!).
- You don't have the engineering bandwidth to **maintain** the Operator long-term.

## 🛡️ Best Practices for Working with Operators

### 1. Use Established Operators

Don't reinvent the wheel. If you need PostgreSQL, use CloudNativePG. If you need Kafka, use Strimzi. These are battle-tested.

### 2. Grant Least Privilege RBAC

Operators need RBAC permissions to create resources. Only grant what's needed:

```yaml
# ✅ Good — specific permissions
rules:
  - apiGroups: [""]
    resources: ["pods", "services"]
    verbs: ["get", "list", "create", "update"]

# ❌ Bad — wildcard everything
rules:
  - apiGroups: ["*"]
    resources: ["*"]
    verbs: ["*"]
```

### 3. Monitor Your Operators

Operators are running software — they can crash, have bugs, or get stuck. Monitor them like any other critical workload:
- Check Operator Pod logs: `kubectl logs -n operators deploy/my-operator`
- Set up alerts for Operator Pod restarts
- Watch for `CrashLoopBackOff` on Operator Pods

### 4. Version Your CRDs

When you update a CRD, use proper versioning (v1, v2alpha1, v2beta1) and provide conversion webhooks:

```yaml
versions:
  - name: v1       # Current stable version
    served: true
    storage: true
  - name: v2beta1  # New version being tested
    served: true
    storage: false
```

### 5. Implement Status Subresource

Always update the `.status` field of your custom resources so users can see what's happening:

```bash
kubectl get postgrescluster my-db
# NAME    REPLICAS   VERSION   STATUS   AGE
# my-db   3          15.4      Ready    7d
```

## 🔑 Key Takeaways

1. **Operators encode human knowledge** — They're like having an expert SRE running 24/7 inside your cluster.
2. **CRD + Controller = Operator** — CRD defines the "what", Controller defines the "how".
3. **Reconciliation Loop** is the core pattern — Observe, Compare, Act, Repeat. Always converge toward desired state.
4. **Don't build your own unless you have to** — Check OperatorHub.io first. Existing Operators are mature and battle-tested.
5. **Operators shine for stateful apps** — Databases, message queues, caches — anything that needs more than just "restart the Pod."
6. **Security matters** — Always use least-privilege RBAC and monitor your Operators.
