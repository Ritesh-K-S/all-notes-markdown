# 10.41 — Operator Framework & Operator SDK

> **One Line Answer:** The **Operator Framework** is an open-source toolkit by Red Hat/CNCF that gives you everything you need to **build, test, package, and manage** Kubernetes Operators — and the **Operator SDK** is its main tool that scaffolds Operator projects so you can focus on writing business logic instead of boilerplate.

---

## 🤔 What Problem Does the Operator Framework Solve?

In the previous section (10.40), we learned what Operators are — CRD + Custom Controller. But actually **building** one from scratch is *really* hard:

- You need to understand the **Kubernetes API** deeply.
- You need to set up **watches**, **informers**, **work queues**, and **caches**.
- You need to handle **error recovery**, **retries**, **leader election**, and **finalizers**.
- You need to implement proper **RBAC**, **webhooks**, **metrics**, and **status updates**.
- You need to **test**, **package**, and **distribute** the Operator.

Doing all this from scratch would take months for an experienced Go developer.

**The Operator Framework solves this** by giving you:

1. **Operator SDK** — A CLI tool that scaffolds (generates) the project structure and boilerplate code.
2. **Operator Lifecycle Manager (OLM)** — Manages the installation, upgrade, and lifecycle of Operators in a cluster.
3. **OperatorHub** — A marketplace to discover and share Operators.

```
┌─────────────────────────────────────────────────────────────┐
│                    OPERATOR FRAMEWORK                        │
│                                                              │
│  ┌──────────────┐  ┌─────────────┐  ┌───────────────────┐   │
│  │ Operator SDK │  │     OLM     │  │   OperatorHub     │   │
│  │              │  │             │  │                   │   │
│  │ BUILD your   │  │ MANAGE your │  │ DISCOVER & SHARE  │   │
│  │ Operator     │  │ Operator's  │  │ Operators with    │   │
│  │              │  │ lifecycle   │  │ the community     │   │
│  └──────────────┘  └─────────────┘  └───────────────────┘   │
│                                                              │
│       Development        Operations         Distribution     │
└─────────────────────────────────────────────────────────────┘
```

## 🧰 The Operator SDK — Your Main Tool

The Operator SDK is a command-line tool that helps you:
- **Scaffold** a new Operator project (generates all boilerplate code).
- **Generate** CRDs, RBAC rules, and Kubernetes manifests from your Go/Ansible/Helm code.
- **Build** and push the Operator container image.
- **Test** your Operator locally and in a cluster.
- **Bundle** your Operator for distribution via OLM.

### Three Ways to Build Operators

The SDK supports **three approaches** to building Operators. Each one suits different skill levels and use cases:

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│  ┌──────────┐       ┌──────────────┐       ┌──────────────┐     │
│  │   Helm   │       │   Ansible    │       │     Go       │     │
│  │  Based   │       │   Based      │       │   Based      │     │
│  │          │       │              │       │              │     │
│  │ Easiest  │       │ Medium       │       │ Most         │     │
│  │ Level 1  │       │ Level 1-2    │       │ Powerful     │     │
│  │          │       │              │       │ Level 1-5    │     │
│  └──────────┘       └──────────────┘       └──────────────┘     │
│                                                                  │
│  Wraps existing     Uses Ansible          Full controller-      │
│  Helm chart as      playbooks as          runtime written       │
│  an Operator        reconciliation        in Go                 │
│                     logic                                        │
└─────────────────────────────────────────────────────────────────┘
```

| Approach | Language/Skill | Capability Level | Best For |
|----------|---------------|-----------------|----------|
| **Helm-based** | Helm charts (YAML) | Level 1 (Basic Install) | Wrapping existing Helm charts as Operators |
| **Ansible-based** | Ansible playbooks | Level 1-2 (Install + Upgrades) | Teams already using Ansible, moderate Day 2 ops |
| **Go-based** | Go programming | Level 1-5 (Full Auto Pilot) | Maximum control, complex stateful apps |

### Which One Should You Choose?

```
Do you have an existing Helm chart?
  └─ YES → Start with Helm-based Operator
  └─ NO  → Do you know Ansible?
              └─ YES → Ansible-based Operator
              └─ NO  → Do you need complex Day 2 logic?
                          └─ YES → Go-based Operator
                          └─ NO  → Helm-based is probably fine
```

## 🚀 Installing the Operator SDK

### Prerequisites

Before installing the SDK, make sure you have:
- **Go** (version 1.21+) — For Go-based Operators
- **Docker** or **Podman** — For building container images
- **kubectl** — For interacting with Kubernetes
- **A Kubernetes cluster** — Minikube, Kind, or a cloud cluster

### Installation

#### On macOS:
```bash
brew install operator-sdk
```

#### On Linux:
```bash
# Download the latest release
export ARCH=$(case $(uname -m) in x86_64) echo -n amd64 ;; aarch64) echo -n arm64 ;; *) echo -n $(uname -m) ;; esac)
export OS=$(uname | awk '{print tolower($0)}')
export OPERATOR_SDK_DL_URL=https://github.com/operator-framework/operator-sdk/releases/download/v1.34.1

curl -LO ${OPERATOR_SDK_DL_URL}/operator-sdk_${OS}_${ARCH}

# Make it executable and move to PATH
chmod +x operator-sdk_${OS}_${ARCH}
sudo mv operator-sdk_${OS}_${ARCH} /usr/local/bin/operator-sdk

# Verify installation
operator-sdk version
# operator-sdk version: "v1.34.1", commit: "...", ...
```

#### On Windows:
```powershell
# Download from GitHub releases
# https://github.com/operator-framework/operator-sdk/releases
# Or use Go to install:
```

## 🏗️ Building a Go-Based Operator — Step by Step

This is the most powerful approach. Let's build a simple Operator that manages a custom **Guestbook** application.

### Step 1: Scaffold the Project

```bash
# Create a new directory for your Operator
mkdir guestbook-operator && cd guestbook-operator

# Initialize the project using the Operator SDK
operator-sdk init \
  --domain example.com \         # Your organization's domain
  --repo github.com/yourname/guestbook-operator  # Go module path
```

**What this creates:**

```
guestbook-operator/
├── Dockerfile               # Builds the Operator container image
├── Makefile                  # Build, test, deploy commands
├── PROJECT                   # Operator SDK project metadata
├── go.mod                    # Go module dependencies
├── go.sum                    # Dependency checksums
├── cmd/
│   └── main.go               # Entry point — starts the controller manager
├── config/
│   ├── default/               # Kustomize base configuration
│   ├── manager/               # Operator Deployment manifests
│   ├── rbac/                  # RBAC (Role, ClusterRole) manifests
│   └── prometheus/            # Prometheus monitoring config
└── internal/
    └── controller/            # Where your controller code will live
```

Let's look at the key generated files:

#### `cmd/main.go` — The Entry Point

```go
package main

import (
    "flag"
    "os"

    "k8s.io/apimachinery/pkg/runtime"
    ctrl "sigs.k8s.io/controller-runtime"
    "sigs.k8s.io/controller-runtime/pkg/log/zap"
    // ... more imports
)

func main() {
    // Setup logging
    opts := zap.Options{Development: true}
    opts.BindFlags(flag.CommandLine)
    flag.Parse()
    ctrl.SetLogger(zap.New(zap.UseFlagOptions(&opts)))

    // Create the Manager
    // The Manager is responsible for:
    // - Running all controllers
    // - Managing the shared cache (informers)
    // - Handling leader election (so only one Operator instance is active)
    mgr, err := ctrl.NewManager(ctrl.GetConfigOrDie(), ctrl.Options{
        Scheme:                 scheme,
        // ... configuration options
    })
    if err != nil {
        os.Exit(1)
    }

    // Register controllers with the Manager
    // (We'll add our controller here after creating it)

    // Start the Manager (blocks until stopped)
    if err := mgr.Start(ctrl.SetupSignalHandler()); err != nil {
        os.Exit(1)
    }
}
```

### Step 3: Define Your CRD Schema

Open `api/v1/guestbook_types.go` and define what your custom resource looks like:

```go
package v1

import (
    metav1 "k8s.io/apimachinery/pkg/apis/meta/v1"
)

// GuestbookSpec defines the desired state of Guestbook
// This is what users will write in their YAML files
type GuestbookSpec struct {
    // Size is the number of Guestbook Pod replicas
    // +kubebuilder:validation:Minimum=1
    // +kubebuilder:validation:Maximum=10
    Size int32 `json:"size"`

    // Image is the container image for the Guestbook app
    // +kubebuilder:default="gcr.io/google-samples/gb-frontend:v5"
    Image string `json:"image,omitempty"`

    // Port is the port the Guestbook app listens on
    // +kubebuilder:default=80
    Port int32 `json:"port,omitempty"`

    // RedisEnabled enables a Redis backend for the Guestbook
    // +kubebuilder:default=true
    RedisEnabled bool `json:"redisEnabled,omitempty"`
}

// GuestbookStatus defines the observed state of Guestbook
// This is what the Operator will update to report current status
type GuestbookStatus struct {
    // Replicas is the current number of running Pods
    Replicas int32 `json:"replicas,omitempty"`

    // Conditions represent the latest available observations
    Conditions []metav1.Condition `json:"conditions,omitempty"`

    // Ready indicates if the Guestbook is fully operational
    Ready bool `json:"ready,omitempty"`
}

// +kubebuilder:object:root=true
// +kubebuilder:subresource:status
// +kubebuilder:printcolumn:name="Size",type=integer,JSONPath=`.spec.size`
// +kubebuilder:printcolumn:name="Ready",type=boolean,JSONPath=`.status.ready`
// +kubebuilder:printcolumn:name="Age",type="date",JSONPath=".metadata.creationTimestamp"

// Guestbook is the Schema for the guestbooks API
type Guestbook struct {
    metav1.TypeMeta   `json:",inline"`
    metav1.ObjectMeta `json:"metadata,omitempty"`

    Spec   GuestbookSpec   `json:"spec,omitempty"`
    Status GuestbookStatus `json:"status,omitempty"`
}

// +kubebuilder:object:root=true

// GuestbookList contains a list of Guestbook
type GuestbookList struct {
    metav1.TypeMeta `json:",inline"`
    metav1.ListMeta `json:"metadata,omitempty"`
    Items           []Guestbook `json:"items"`
}

func init() {
    SchemeBuilder.Register(&Guestbook{}, &GuestbookList{})
}
```

#### Understanding the Markers (Comments Starting with `+`)

Those special comments (like `+kubebuilder:validation:Minimum=1`) are called **markers**. They're instructions for the code generator:

| Marker | What It Does |
|--------|-------------|
| `+kubebuilder:validation:Minimum=1` | Sets minimum value for the field |
| `+kubebuilder:validation:Maximum=10` | Sets maximum value for the field |
| `+kubebuilder:default="value"` | Sets a default value |
| `+kubebuilder:object:root=true` | Marks this as a root API object |
| `+kubebuilder:subresource:status` | Enables the `/status` subresource |
| `+kubebuilder:printcolumn:...` | Adds columns to `kubectl get` output |

After defining your types, regenerate the CRD manifests:

```bash
make manifests   # Generates CRD YAML from your Go types
make generate    # Generates DeepCopy methods
```

This produces `config/crd/bases/webapp.example.com_guestbooks.yaml` — a complete CRD YAML that you can apply to your cluster.

### Step 4: Write the Controller Logic

Open `internal/controller/guestbook_controller.go` and write the reconciliation logic:

```go
package controller

import (
    "context"
    "fmt"

    appsv1 "k8s.io/api/apps/v1"
    corev1 "k8s.io/api/core/v1"
    "k8s.io/apimachinery/pkg/api/errors"
    metav1 "k8s.io/apimachinery/pkg/apis/meta/v1"
    "k8s.io/apimachinery/pkg/runtime"
    "k8s.io/apimachinery/pkg/types"
    ctrl "sigs.k8s.io/controller-runtime"
    "sigs.k8s.io/controller-runtime/pkg/client"
    "sigs.k8s.io/controller-runtime/pkg/log"

    webappv1 "github.com/yourname/guestbook-operator/api/v1"
)

// GuestbookReconciler reconciles a Guestbook object
type GuestbookReconciler struct {
    client.Client
    Scheme *runtime.Scheme
}

// +kubebuilder:rbac:groups=webapp.example.com,resources=guestbooks,verbs=get;list;watch;create;update;patch;delete
// +kubebuilder:rbac:groups=webapp.example.com,resources=guestbooks/status,verbs=get;update;patch
// +kubebuilder:rbac:groups=apps,resources=deployments,verbs=get;list;watch;create;update;patch;delete
// +kubebuilder:rbac:groups="",resources=services,verbs=get;list;watch;create;update;patch;delete

// Reconcile is the main reconciliation loop
// It's called whenever a Guestbook resource is created, updated, or deleted
func (r *GuestbookReconciler) Reconcile(ctx context.Context, req ctrl.Request) (ctrl.Result, error) {
    logger := log.FromContext(ctx)

    // ==========================================
    // Step 1: FETCH the Guestbook custom resource
    // ==========================================
    var guestbook webappv1.Guestbook
    if err := r.Get(ctx, req.NamespacedName, &guestbook); err != nil {
        if errors.IsNotFound(err) {
            // The Guestbook was deleted — nothing to do
            logger.Info("Guestbook resource not found. Ignoring since it must be deleted.")
            return ctrl.Result{}, nil
        }
        // Error reading the object — requeue the request
        logger.Error(err, "Failed to get Guestbook")
        return ctrl.Result{}, err
    }

    logger.Info("Reconciling Guestbook", "name", guestbook.Name)

    // ==========================================
    // Step 2: CHECK if the Deployment already exists
    // ==========================================
    var existingDeployment appsv1.Deployment
    err := r.Get(ctx, types.NamespacedName{
        Name:      guestbook.Name + "-deployment",
        Namespace: guestbook.Namespace,
    }, &existingDeployment)

    if err != nil && errors.IsNotFound(err) {
        // Deployment doesn't exist — CREATE it
        logger.Info("Creating a new Deployment", "name", guestbook.Name)
        dep := r.createDeployment(&guestbook)
        if err := r.Create(ctx, dep); err != nil {
            logger.Error(err, "Failed to create Deployment")
            return ctrl.Result{}, err
        }
        logger.Info("Deployment created successfully")
    } else if err != nil {
        // Some other error occurred
        logger.Error(err, "Failed to get Deployment")
        return ctrl.Result{}, err
    } else {
        // Deployment exists — check if it needs updating
        
        // Check replica count
        if *existingDeployment.Spec.Replicas != guestbook.Spec.Size {
            logger.Info("Updating replica count",
                "current", *existingDeployment.Spec.Replicas,
                "desired", guestbook.Spec.Size)
            existingDeployment.Spec.Replicas = &guestbook.Spec.Size
            if err := r.Update(ctx, &existingDeployment); err != nil {
                logger.Error(err, "Failed to update Deployment")
                return ctrl.Result{}, err
            }
        }

        // Check container image
        currentImage := existingDeployment.Spec.Template.Spec.Containers[0].Image
        if currentImage != guestbook.Spec.Image {
            logger.Info("Updating container image",
                "current", currentImage,
                "desired", guestbook.Spec.Image)
            existingDeployment.Spec.Template.Spec.Containers[0].Image = guestbook.Spec.Image
            if err := r.Update(ctx, &existingDeployment); err != nil {
                logger.Error(err, "Failed to update Deployment image")
                return ctrl.Result{}, err
            }
        }
    }

    // ==========================================
    // Step 3: CREATE Service if it doesn't exist
    // ==========================================
    var existingService corev1.Service
    err = r.Get(ctx, types.NamespacedName{
        Name:      guestbook.Name + "-service",
        Namespace: guestbook.Namespace,
    }, &existingService)

    if err != nil && errors.IsNotFound(err) {
        svc := r.createService(&guestbook)
        if err := r.Create(ctx, svc); err != nil {
            logger.Error(err, "Failed to create Service")
            return ctrl.Result{}, err
        }
        logger.Info("Service created successfully")
    }

    // ==========================================
    // Step 4: UPDATE the Status of our custom resource
    // ==========================================
    guestbook.Status.Replicas = guestbook.Spec.Size
    guestbook.Status.Ready = true
    if err := r.Status().Update(ctx, &guestbook); err != nil {
        logger.Error(err, "Failed to update Guestbook status")
        return ctrl.Result{}, err
    }

    logger.Info("Reconciliation complete", "name", guestbook.Name)
    return ctrl.Result{}, nil
}

// createDeployment constructs a Deployment for the Guestbook
func (r *GuestbookReconciler) createDeployment(gb *webappv1.Guestbook) *appsv1.Deployment {
    labels := map[string]string{
        "app":        "guestbook",
        "guestbook":  gb.Name,
        "managed-by": "guestbook-operator",
    }

    dep := &appsv1.Deployment{
        ObjectMeta: metav1.ObjectMeta{
            Name:      gb.Name + "-deployment",
            Namespace: gb.Namespace,
            Labels:    labels,
        },
        Spec: appsv1.DeploymentSpec{
            Replicas: &gb.Spec.Size,
            Selector: &metav1.LabelSelector{
                MatchLabels: labels,
            },
            Template: corev1.PodTemplateSpec{
                ObjectMeta: metav1.ObjectMeta{
                    Labels: labels,
                },
                Spec: corev1.PodSpec{
                    Containers: []corev1.Container{
                        {
                            Name:  "guestbook",
                            Image: gb.Spec.Image,
                            Ports: []corev1.ContainerPort{
                                {
                                    ContainerPort: gb.Spec.Port,
                                },
                            },
                        },
                    },
                },
            },
        },
    }

    // Set the Guestbook as the owner of this Deployment
    // When the Guestbook CR is deleted, this Deployment will be garbage collected
    ctrl.SetControllerReference(gb, dep, r.Scheme)

    return dep
}

// createService constructs a Service for the Guestbook
func (r *GuestbookReconciler) createService(gb *webappv1.Guestbook) *corev1.Service {
    svc := &corev1.Service{
        ObjectMeta: metav1.ObjectMeta{
            Name:      gb.Name + "-service",
            Namespace: gb.Namespace,
        },
        Spec: corev1.ServiceSpec{
            Selector: map[string]string{
                "app":       "guestbook",
                "guestbook": gb.Name,
            },
            Ports: []corev1.ServicePort{
                {
                    Port:       80,
                    TargetPort: intstr.FromInt(int(gb.Spec.Port)),
                },
            },
            Type: corev1.ServiceTypeClusterIP,
        },
    }

    ctrl.SetControllerReference(gb, svc, r.Scheme)
    return svc
}

// SetupWithManager sets up the controller with the Manager.
// This tells the Manager to watch for Guestbook resources
// AND also watch Deployments that are owned by Guestbook resources
func (r *GuestbookReconciler) SetupWithManager(mgr ctrl.Manager) error {
    return ctrl.NewControllerManagedBy(mgr).
        For(&webappv1.Guestbook{}).           // Watch Guestbook CRs
        Owns(&appsv1.Deployment{}).            // Also watch Deployments owned by us
        Owns(&corev1.Service{}).               // Also watch Services owned by us
        Complete(r)
}
```

#### Breaking Down the Controller Code

Let me explain the important parts:

**1. RBAC Markers:**
```go
// +kubebuilder:rbac:groups=webapp.example.com,resources=guestbooks,verbs=get;list;watch;create;update;patch;delete
```
These generate the RBAC rules that the Operator needs. Run `make manifests` and these become ClusterRole YAML.

**2. Reconcile Function:**
This is THE function. Everything happens here:
- Fetch the CR → Check existing resources → Create/Update as needed → Update status.

**3. Owner References:**
```go
ctrl.SetControllerReference(gb, dep, r.Scheme)
```
This is crucial! It sets the Guestbook CR as the "owner" of the Deployment. When you delete the Guestbook CR, Kubernetes automatically deletes (garbage collects) the Deployment, Service, and everything else the Operator created.

**4. SetupWithManager:**
```go
For(&webappv1.Guestbook{}).    // Watch Guestbook CRs
Owns(&appsv1.Deployment{}).     // Watch owned Deployments
```
This tells the controller to react to changes on both the Guestbook CR AND any Deployments it owns. So if someone manually deletes the Deployment, the controller will detect this and recreate it!

## 🎯 Building a Helm-Based Operator

This is the **easiest** way to create an Operator. It wraps an existing Helm chart.

```bash
# Create a Helm-based Operator project
mkdir nginx-operator && cd nginx-operator

operator-sdk init \
  --plugins helm \
  --domain example.com

# Create an API from an existing Helm chart
operator-sdk create api \
  --group web \
  --version v1 \
  --kind Nginx \
  --helm-chart nginx \
  --helm-chart-repo https://charts.bitnami.com/bitnami
```

This generates a project where the Helm chart values become your CRD spec. Users create a `Nginx` CR, and the Operator runs `helm install` under the hood.

**Project structure:**

```
nginx-operator/
├── Dockerfile
├── Makefile
├── PROJECT
├── config/
│   ├── crd/
│   ├── default/
│   ├── manager/
│   ├── rbac/
│   └── samples/
│       └── web_v1_nginx.yaml    # Sample CR
├── helm-charts/
│   └── nginx/                   # The Helm chart lives here
│       ├── Chart.yaml
│       ├── values.yaml
│       └── templates/
└── watches.yaml                 # Maps CRs to Helm charts
```

**The `watches.yaml` file:**

```yaml
# This file tells the Operator which Helm chart to use
# for each custom resource
- group: web.example.com
  version: v1
  kind: Nginx
  chart: helm-charts/nginx       # Path to the Helm chart
  # You can also override values here:
  # overrideValues:
  #   image.repository: my-custom-nginx
```

**Sample CR:**

```yaml
apiVersion: web.example.com/v1
kind: Nginx
metadata:
  name: my-nginx
spec:
  # These values map directly to the Helm chart's values.yaml
  replicaCount: 3
  service:
    type: LoadBalancer
  image:
    repository: nginx
    tag: "1.25"
```

When you apply this CR, the Operator automatically does `helm install` with these values. When you update the CR, it does `helm upgrade`. When you delete the CR, it does `helm uninstall`. Simple!

## 🤖 Building an Ansible-Based Operator

If your team already uses Ansible, this is a great middle-ground approach.

```bash
# Create an Ansible-based Operator project
mkdir memcached-operator && cd memcached-operator

operator-sdk init \
  --plugins ansible \
  --domain example.com

# Create an API
operator-sdk create api \
  --group cache \
  --version v1 \
  --kind Memcached \
  --generate-role
```

## 📦 Operator Lifecycle Manager (OLM)

OLM is the second major component of the Operator Framework. It handles:

- **Installing** Operators in a cluster
- **Upgrading** Operators to new versions
- **Managing dependencies** between Operators
- **Providing a catalog** of available Operators

### Key OLM Concepts

```
┌─────────────────────────────────────────────────────────┐
│                    OLM Components                        │
│                                                          │
│  ┌──────────────────┐    ┌───────────────────────┐       │
│  │ CatalogSource    │    │ Subscription          │       │
│  │                  │    │                       │       │
│  │ "App Store"      │    │ "Auto-update rule"    │       │
│  │ Lists available  │    │ Install channel X     │       │
│  │ Operators        │    │ Auto-approve updates  │       │
│  └────────┬─────────┘    └───────────┬───────────┘       │
│           │                          │                    │
│           ▼                          ▼                    │
│  ┌──────────────────┐    ┌───────────────────────┐       │
│  │ ClusterService   │    │ InstallPlan           │       │
│  │ Version (CSV)    │    │                       │       │
│  │                  │    │ "Install recipe"      │       │
│  │ Describes one    │    │ What resources to     │       │
│  │ version of an    │    │ create for the        │       │
│  │ Operator         │    │ Operator              │       │
│  └──────────────────┘    └───────────────────────┘       │
└─────────────────────────────────────────────────────────┘
```

| OLM Resource | Purpose | Analogy |
|-------------|---------|---------|
| **CatalogSource** | A repository of available Operators | Like an app store (Google Play, App Store) |
| **Subscription** | Declares which Operator to install and how to update it | Like subscribing to an app for auto-updates |
| **ClusterServiceVersion (CSV)** | Describes a specific version of an Operator (metadata, permissions, install strategy) | Like an app listing with version info |
| **InstallPlan** | The plan for installing/upgrading an Operator | Like the download/install confirmation screen |

### Installing OLM

```bash
# Install OLM in your cluster
operator-sdk olm install

# Verify
operator-sdk olm status
```

### Creating an Operator Bundle for OLM

```bash
# Generate the bundle (CSV, CRD, package manifest)
make bundle IMG=yourregistry/guestbook-operator:v0.1.0

# Build the bundle image
make bundle-build BUNDLE_IMG=yourregistry/guestbook-operator-bundle:v0.1.0

# Push the bundle image
make bundle-push BUNDLE_IMG=yourregistry/guestbook-operator-bundle:v0.1.0

# Test it with OLM
operator-sdk run bundle yourregistry/guestbook-operator-bundle:v0.1.0
```

## 🧪 Testing Your Operator

Testing is crucial. The SDK supports multiple testing approaches:

### 1. Local Testing (Outside the Cluster)

Run the Operator on your machine, connecting to a remote cluster:

```bash
# Install the CRD
make install

# Run the Operator locally (not in a Pod)
make run

# In another terminal, create a CR and watch what happens
kubectl apply -f config/samples/webapp_v1_guestbook.yaml
```

This is the fastest way to develop and debug — you get immediate logs in your terminal and can set breakpoints in your IDE.

### 2. Unit Tests

```go
// internal/controller/guestbook_controller_test.go
package controller

import (
    "context"
    "testing"

    . "github.com/onsi/ginkgo/v2"
    . "github.com/onsi/gomega"
    appsv1 "k8s.io/api/apps/v1"
    metav1 "k8s.io/apimachinery/pkg/apis/meta/v1"
    "k8s.io/apimachinery/pkg/types"

    webappv1 "github.com/yourname/guestbook-operator/api/v1"
)

var _ = Describe("Guestbook Controller", func() {
    Context("When creating a Guestbook CR", func() {
        It("Should create a Deployment with correct replicas", func() {
            ctx := context.Background()

            // Create a Guestbook CR
            guestbook := &webappv1.Guestbook{
                ObjectMeta: metav1.ObjectMeta{
                    Name:      "test-guestbook",
                    Namespace: "default",
                },
                Spec: webappv1.GuestbookSpec{
                    Size:  3,
                    Image: "gcr.io/google-samples/gb-frontend:v5",
                    Port:  80,
                },
            }
            Expect(k8sClient.Create(ctx, guestbook)).To(Succeed())

            // Eventually, a Deployment should be created
            deployment := &appsv1.Deployment{}
            Eventually(func() error {
                return k8sClient.Get(ctx, types.NamespacedName{
                    Name:      "test-guestbook-deployment",
                    Namespace: "default",
                }, deployment)
            }).Should(Succeed())

            // Verify the Deployment has the correct number of replicas
            Expect(*deployment.Spec.Replicas).To(Equal(int32(3)))
        })
    })
})
```

Run tests:
```bash
make test
```

### 3. End-to-End Tests with `envtest`

The `envtest` package starts a real (but lightweight) API server and etcd for testing. It's faster than running a full cluster but still tests real Kubernetes API interactions.

### 4. Scorecard Tests

The SDK includes a **scorecard** tool that validates your Operator against best practices:

```bash
# Run scorecard tests
operator-sdk scorecard yourregistry/guestbook-operator-bundle:v0.1.0

# Output shows pass/fail for each test:
# ✅ Basic Tests - olm-crds-have-validation
# ✅ Basic Tests - olm-crds-have-resources
# ✅ OLM Tests - olm-spec-descriptors
# ❌ OLM Tests - olm-status-descriptors  (missing status descriptors)
```

## 🔧 Key Concepts You'll Encounter

### Finalizers

Finalizers let your Operator run **cleanup logic** before a resource is deleted:

```go
const finalizerName = "webapp.example.com/finalizer"

func (r *GuestbookReconciler) Reconcile(ctx context.Context, req ctrl.Request) (ctrl.Result, error) {
    var guestbook webappv1.Guestbook
    r.Get(ctx, req.NamespacedName, &guestbook)

    // Is the resource being deleted?
    if !guestbook.DeletionTimestamp.IsZero() {
        // Yes! Run cleanup logic
        if containsString(guestbook.Finalizers, finalizerName) {
            // Do cleanup: delete external resources, close connections, etc.
            r.cleanupExternalResources(&guestbook)
            
            // Remove the finalizer so Kubernetes can delete the resource
            guestbook.Finalizers = removeString(guestbook.Finalizers, finalizerName)
            r.Update(ctx, &guestbook)
        }
        return ctrl.Result{}, nil
    }

    // Resource is NOT being deleted — add finalizer if missing
    if !containsString(guestbook.Finalizers, finalizerName) {
        guestbook.Finalizers = append(guestbook.Finalizers, finalizerName)
        r.Update(ctx, &guestbook)
    }

    // ... normal reconciliation logic
}
```

### Leader Election

When running multiple replicas of your Operator for high availability, only ONE should be active:

```go
mgr, err := ctrl.NewManager(ctrl.GetConfigOrDie(), ctrl.Options{
    LeaderElection:   true,                               // Enable leader election
    LeaderElectionID: "guestbook-operator-leader-lock",   // Unique lock name
})
```

### Webhooks

Webhooks let you **validate** or **mutate** resources before they're saved:

```bash
# Create a webhook
operator-sdk create webhook \
  --group webapp \
  --version v1 \
  --kind Guestbook \
  --defaulting \          # Mutating webhook (set defaults)
  --programmatic-validation  # Validating webhook (reject bad configs)
```

```go
// Defaulting webhook — sets default values
func (r *Guestbook) Default() {
    if r.Spec.Size == 0 {
        r.Spec.Size = 1  // Default to 1 replica
    }
    if r.Spec.Port == 0 {
        r.Spec.Port = 80  // Default to port 80
    }
}

// Validating webhook — rejects invalid configurations
func (r *Guestbook) ValidateCreate() (admission.Warnings, error) {
    if r.Spec.Size > 10 {
        return nil, fmt.Errorf("size cannot exceed 10, got %d", r.Spec.Size)
    }
    return nil, nil
}
```

## 📊 Comparison: All Three Approaches

| Feature | Helm-Based | Ansible-Based | Go-Based |
|---------|-----------|--------------|----------|
| **Learning Curve** | Low | Medium | High |
| **Language** | YAML (Helm) | YAML (Ansible) | Go |
| **Max Capability Level** | Level 1 | Level 2 | Level 5 |
| **Day 2 Operations** | Limited | Moderate | Full |
| **Custom Logic** | No | Limited (Ansible modules) | Unlimited |
| **Webhooks** | ❌ | ❌ | ✅ |
| **Testing** | Basic | Basic | Full (envtest, unit, e2e) |
| **Performance** | Good | Moderate (Python overhead) | Best |
| **Best For** | Wrapping existing Helm charts | Teams with Ansible expertise | Complex stateful applications |

## 🔑 Key Takeaways

1. **Don't build from scratch** — The Operator SDK saves you months of boilerplate work.
2. **Choose the right approach** — Helm for simple wrappers, Ansible for moderate needs, Go for full power.
3. **Start simple** — Begin with a Helm-based Operator and upgrade to Go when you need more control.
4. **Test everything** — Use `make run` for local development, `envtest` for integration tests, and scorecard for best practice validation.
5. **OLM is optional but recommended** — It handles Operator lifecycle management in production clusters.
6. **Owner references are essential** — Always set them so garbage collection works when CRs are deleted.
