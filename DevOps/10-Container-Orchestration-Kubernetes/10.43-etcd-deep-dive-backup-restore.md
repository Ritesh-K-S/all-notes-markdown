## 🤔 Why Should You Care About etcd?

Imagine your Kubernetes cluster as a city:

- **Pods** are the people
- **Nodes** are the neighborhoods
- **Services** are the roads connecting everything
- **Deployments** are the city plans

Now ask yourself: **Where is ALL this information stored?** Where does Kubernetes remember that you want 3 replicas of your web app, or that Pod "frontend-abc123" should be running on Node 2?

**That's etcd.**

> etcd is the **single source of truth** for everything in your Kubernetes cluster. If etcd dies and you have no backup, your cluster effectively doesn't exist anymore — even if all your nodes are still running.

Let that sink in. 😳

## 📦 What is etcd?

### The Basics

| Fact | Detail |
|------|--------|
| **Full Name** | etcd (pronounced "et-see-dee") |
| **What it is** | A distributed, reliable key-value store |
| **Created By** | CoreOS (now part of Red Hat / IBM) |
| **Written In** | Go (Golang) |
| **Used By** | Kubernetes (as its only data store), and many other systems |
| **Data Model** | Key-Value pairs (like a giant dictionary / hash map) |
| **Consistency** | Strongly consistent (uses Raft consensus algorithm) |
| **License** | Apache License 2.0 (open source) |
| **Port** | 2379 (client communication), 2380 (peer-to-peer communication) |
| **CNCF Status** | Graduated project (highest maturity level) |

### What Does "Key-Value Store" Mean?

It's super simple. Think of it like a dictionary:

```
Key                           → Value
/registry/pods/default/nginx  → { entire Pod spec as JSON }
/registry/services/default/my-svc → { entire Service spec as JSON }
/registry/secrets/default/db-pass → { encrypted Secret data }
```

Every resource you create in Kubernetes (Pods, Deployments, Services, Secrets, ConfigMaps, Namespaces, etc.) gets stored as a **key-value pair** in etcd.

## 🏗️ Where Does etcd Sit in Kubernetes Architecture?

Let's look at the big picture:

```
┌──────────────────────────── Control Plane ─────────────────────────────┐
│                                                                        │
│  ┌─────────────┐    ┌──────────────┐    ┌──────────────────┐          │
│  │  API Server  │───▶│     etcd     │    │   Scheduler      │          │
│  │ (kube-apiserver)│  │  (Data Store) │    │ (kube-scheduler) │          │
│  └──────┬───────┘    └──────────────┘    └──────────────────┘          │
│         │                                                              │
│         │            ┌───────────────────────┐                         │
│         └───────────▶│  Controller Manager    │                         │
│                      │ (kube-controller-mgr)  │                         │
│                      └───────────────────────┘                         │
└────────────────────────────────────────────────────────────────────────┘
```

### The Critical Insight

**Only the API Server talks directly to etcd.** No other component (Scheduler, Controller Manager, kubelet, kubectl) ever touches etcd directly.

The flow is always:

```
kubectl → API Server → etcd
Scheduler → API Server → etcd
Controller Manager → API Server → etcd
kubelet → API Server → etcd
```

This is a deliberate design decision:
- **Security**: Only one component needs credentials to access the database
- **Consistency**: All changes go through one gateway (API Server)
- **Validation**: API Server validates every change before writing to etcd

## 🔬 etcd Internals — How Does It Work?

### 1. The Raft Consensus Algorithm

etcd uses a consensus algorithm called **Raft** to ensure all copies of data stay in sync. Let's understand it simply.

#### The Problem

Imagine you have 3 etcd servers (a cluster). A write request comes in: "Create a new Pod called nginx." How do you make sure all 3 servers agree on this change?

What if:
- Server 2 crashes mid-write?
- Two servers receive different writes at the same time?
- Network between servers goes down temporarily?

**Raft solves all of these problems.**

#### How Raft Works (Simplified)

Think of it like an election system:

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   etcd-1     │     │   etcd-2     │     │   etcd-3     │
│   LEADER ⭐  │────▶│   FOLLOWER   │     │   FOLLOWER   │
│              │     │              │     │              │
└──────────────┘     └──────────────┘     └──────────────┘
       │                                         ▲
       └─────────────────────────────────────────┘
```

1. **One node is elected as the Leader** — All writes go through the Leader
2. **Leader sends changes to Followers** — "Hey, write this data"
3. **Majority must agree** — If 2 out of 3 agree, the write is committed
4. **If the Leader dies** — A new election happens, a Follower becomes the new Leader
5. **Clients can read from any node** — But writes always go to the Leader

#### The Quorum Rule

For etcd to work (accept writes), it needs a **majority** (quorum) of nodes to be alive:

| Cluster Size | Quorum Needed | Can Tolerate Failures |
|-------------|---------------|----------------------|
| 1 node | 1 | 0 (don't do this in production!) |
| 3 nodes | 2 | 1 node failure |
| 5 nodes | 3 | 2 node failures |
| 7 nodes | 4 | 3 node failures |

**Recommendation:** Always use **odd numbers** (3 or 5) for etcd clusters. Even numbers don't give you any advantage and waste resources.

**Why 3 is the sweet spot for most clusters:**
- Can survive 1 node failure
- Not too many nodes to slow down consensus
- Most Kubernetes setups use this

**Why 5 for large/critical clusters:**
- Can survive 2 node failures
- More resilience at the cost of slightly slower writes

### 2. Revision and MVCC

etcd uses **Multi-Version Concurrency Control (MVCC)**. Every change gets a **revision number** — think of it like Git commits:

```
Revision 1: Created Pod nginx
Revision 2: Updated Pod nginx (changed image)
Revision 3: Created Service my-svc
Revision 4: Deleted Pod nginx
Revision 5: Created Deployment web-app
```

This means:
- You can **watch for changes** (Kubernetes does this heavily!)
- You can **go back in history** to see what changed
- Multiple reads can happen simultaneously without locking

### 3. Watch Mechanism

One of etcd's most powerful features. Kubernetes controllers constantly **watch** for changes:

```
Controller Manager: "Hey etcd, tell me whenever a Pod changes"
etcd: "Sure, I'll notify you on every change"

... time passes ...

etcd: "Hey! Pod nginx was deleted!" (Revision 4)
Controller Manager: "Got it! I'll create a new one to match the desired state"
```

This is how Kubernetes achieves its **declarative, self-healing** nature. Controllers watch etcd (through the API Server) and react to changes.

## 🔧 Working with etcd — Hands-On

### Installing etcdctl

`etcdctl` is the CLI tool for interacting with etcd directly.

**On a kubeadm cluster**, etcdctl is usually available inside the etcd Pod:

```bash
# Get a shell inside the etcd Pod
kubectl exec -it etcd-<control-plane-node-name> -n kube-system -- /bin/sh

# Or use etcdctl directly
kubectl exec -it etcd-<control-plane-node-name> -n kube-system -- etcdctl version
```

**Install on Ubuntu/Debian:**

```bash
# Download etcdctl (match your etcd version!)
ETCD_VER=v3.5.12

curl -L https://github.com/etcd-io/etcd/releases/download/${ETCD_VER}/etcd-${ETCD_VER}-linux-amd64.tar.gz -o etcd.tar.gz

tar xzf etcd.tar.gz
sudo mv etcd-${ETCD_VER}-linux-amd64/etcdctl /usr/local/bin/
etcdctl version
```

### Setting Up Authentication

etcd in Kubernetes uses **TLS certificates** for secure communication. You need to provide these when running etcdctl commands:

```bash
# The certificates are usually located at:
# /etc/kubernetes/pki/etcd/

# Set environment variables for convenience
export ETCDCTL_API=3
export ETCDCTL_ENDPOINTS=https://127.0.0.1:2379
export ETCDCTL_CACERT=/etc/kubernetes/pki/etcd/ca.crt
export ETCDCTL_CERT=/etc/kubernetes/pki/etcd/server.crt
export ETCDCTL_KEY=/etc/kubernetes/pki/etcd/server.key
```

Let me explain each:

| Variable | What It Is |
|----------|-----------|
| `ETCDCTL_API=3` | Use etcd v3 API (always use this, v2 is deprecated) |
| `ETCDCTL_ENDPOINTS` | Where etcd is listening |
| `ETCDCTL_CACERT` | Certificate Authority cert — to verify the server |
| `ETCDCTL_CERT` | Client cert — to prove who you are |
| `ETCDCTL_KEY` | Client key — private key for the client cert |

## 🌍 etcd Cluster Topologies in Kubernetes

There are two main ways to deploy etcd with Kubernetes:

### 1. Stacked etcd (Most Common)

etcd runs **on the same nodes** as the control plane components:

```
┌────── Control Plane Node 1 ──────┐
│  API Server                      │
│  Controller Manager              │
│  Scheduler                       │
│  etcd ◀━━━━━━━━━━━━━━━━━━━━━━━┓  │
└──────────────────────────────│──┘
                               │
┌────── Control Plane Node 2 ──│───┐
│  API Server                  │   │
│  Controller Manager          │   │
│  Scheduler                   │   │
│  etcd ◀━━━━━━━━━━━━━━━━━━━━━━┫   │
└──────────────────────────────│──┘
                               │
┌────── Control Plane Node 3 ──│───┐
│  API Server                  │   │
│  Controller Manager          │   │
│  Scheduler                   │   │
│  etcd ◀━━━━━━━━━━━━━━━━━━━━━━┛   │
└──────────────────────────────────┘
```

**Pros:**
- Simpler to set up (kubeadm does this by default)
- Fewer nodes to manage
- Lower cost

**Cons:**
- If a node dies, you lose both a control plane component AND an etcd member
- Coupled failure domains

### 2. External etcd (More Resilient)

etcd runs on **separate, dedicated nodes**:

```
┌── CP Node 1 ──┐  ┌── CP Node 2 ──┐  ┌── CP Node 3 ──┐
│  API Server    │  │  API Server    │  │  API Server    │
│  Controller    │  │  Controller    │  │  Controller    │
│  Scheduler     │  │  Scheduler     │  │  Scheduler     │
└───────┬────────┘  └───────┬────────┘  └───────┬────────┘
        │                   │                   │
        ▼                   ▼                   ▼
┌── etcd Node 1 ┐  ┌── etcd Node 2 ┐  ┌── etcd Node 3 ┐
│    etcd        │  │    etcd        │  │    etcd        │
└────────────────┘  └────────────────┘  └────────────────┘
```

**Pros:**
- Decoupled failure domains — etcd and control plane can fail independently
- Can use specialized hardware/disks for etcd (SSD recommended!)
- Better for large production clusters

**Cons:**
- More nodes to manage (6 instead of 3)
- More complex setup
- Higher cost

### Which Should You Use?

| Scenario | Recommendation |
|----------|---------------|
| Small/medium clusters (< 100 nodes) | Stacked etcd |
| Learning / Dev environments | Stacked etcd |
| Large production clusters (100+ nodes) | External etcd |
| Mission-critical workloads | External etcd |
| Managed Kubernetes (EKS, AKS, GKE) | Cloud provider handles it! |

## 🛡️ etcd Performance and Tuning

etcd is **very sensitive to disk latency**. This is the #1 source of etcd problems. Here's what you need to know:

### Hardware Recommendations

| Resource | Minimum | Recommended |
|----------|---------|------------|
| **CPU** | 2 cores | 4+ cores |
| **Memory** | 2 GB | 8 GB |
| **Disk** | HDD (slow!) | **SSD** (strongly recommended!) |
| **Disk IOPS** | 50 sequential | 500+ sequential |
| **Network** | 1 Gbps | 10 Gbps for large clusters |

### Key Performance Tuning Parameters

```yaml
# etcd configuration (usually in /etc/etcd/etcd.conf or as flags)

# Heartbeat interval — how often leader sends heartbeats to followers
# Default: 100ms, increase for high-latency networks
--heartbeat-interval=100

# Election timeout — how long a follower waits before starting an election
# Default: 1000ms (must be 5-10x heartbeat interval)
--election-timeout=1000

# Snapshot count — after how many writes to create a snapshot
# Default: 100000
--snapshot-count=100000

# Quota backend bytes — max etcd database size
# Default: 2GB, max recommended: 8GB
--quota-backend-bytes=8589934592

# Auto compaction — automatically clean up old revisions
--auto-compaction-mode=periodic
--auto-compaction-retention=1h
```

### Monitoring etcd Health

Key metrics to watch:

```bash
# Check database size
etcdctl endpoint status --write-out=table
# Watch the "DB SIZE" column — if it approaches quota, you're in trouble!

# Check for slow disk (fsync latency)
# etcd exposes Prometheus metrics at :2379/metrics
curl -s https://localhost:2379/metrics --cert /etc/kubernetes/pki/etcd/server.crt --key /etc/kubernetes/pki/etcd/server.key --cacert /etc/kubernetes/pki/etcd/ca.crt | grep wal_fsync

# Key metric: etcd_disk_wal_fsync_duration_seconds
# If the 99th percentile exceeds 10ms, your disk is too slow!

# Check leader changes (too many = instability)
curl -s https://localhost:2379/metrics ... | grep etcd_server_leader_changes_seen_total
```

### Compaction and Defragmentation

Over time, etcd accumulates old data (revisions). You need to clean it up:

```bash
# Compact — remove old revisions up to a specific revision
# First, get the current revision
CURRENT_REV=$(etcdctl endpoint status --write-out=json | python3 -c "import json,sys; print(json.load(sys.stdin)[0]['Status']['header']['revision'])")

# Compact up to current revision (removes old versions of keys)
etcdctl compact $CURRENT_REV

# Defragment — reclaim disk space after compaction
# Run on EACH member, one at a time (NOT in parallel!)
etcdctl defrag --endpoints=https://etcd-0:2379
etcdctl defrag --endpoints=https://etcd-1:2379
etcdctl defrag --endpoints=https://etcd-2:2379

# Check DB size after defrag — it should be smaller
etcdctl endpoint status --write-out=table
```

**⚠️ Warning:** Always defrag one member at a time. Defragging blocks the member temporarily, and you don't want to block a quorum.

## 💾 etcd Backup — The Most Critical Task

> **If you take away ONE thing from this entire note, let it be this: ALWAYS have etcd backups. Always. Always. ALWAYS.**

### Why Backups are Critical

| Scenario | Without Backup | With Backup |
|----------|---------------|-------------|
| etcd corruption | Entire cluster lost | Restore from backup ✅ |
| Accidental deletion (someone ran `kubectl delete all --all`) | Data gone forever | Restore from backup ✅ |
| Failed cluster upgrade | Cluster broken | Restore pre-upgrade backup ✅ |
| Ransomware attack | Start from scratch | Restore from backup ✅ |
| Disk failure | Cluster is dead | Restore from backup ✅ |

### Method 1: Snapshot Backup (Recommended)

This is the **official and most reliable** method.

```bash
# Take a snapshot backup
etcdctl snapshot save /backup/etcd-snapshot-$(date +%Y%m%d-%H%M%S).db

# Output:
# Snapshot saved at /backup/etcd-snapshot-20250529-143022.db

# Verify the snapshot
etcdctl snapshot status /backup/etcd-snapshot-20250529-143022.db --write-out=table

# Output:
# +----------+----------+------------+------------+
# |   HASH   | REVISION | TOTAL KEYS | TOTAL SIZE |
# +----------+----------+------------+------------+
# | b3fae224 |   412039 |       1258 |     4.1 MB |
# +----------+----------+------------+------------+
```

### Method 2: Backup from kubeadm Cluster (With TLS Certs)

In a kubeadm cluster, you need to specify the certificates:

```bash
ETCDCTL_API=3 etcdctl snapshot save /backup/etcd-backup.db \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key
```

### Method 3: Backup from Inside the etcd Pod

If you can't run etcdctl on the host, do it from inside the Pod:

```bash
# Step 1: Exec into the etcd Pod
kubectl exec -it etcd-<control-plane-node> -n kube-system -- /bin/sh

# Step 2: Take the snapshot (certs are already mounted inside the Pod)
etcdctl snapshot save /var/lib/etcd/backup.db \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key

# Step 3: Exit the Pod
exit

# Step 4: Copy the backup from the Pod to your local machine
kubectl cp kube-system/etcd-<control-plane-node>:/var/lib/etcd/backup.db ./etcd-backup.db
```

### Automated Backup Script

Here's a script you can run as a **cron job** to take regular backups:

```bash
#!/bin/bash
# etcd-backup.sh — Automated etcd Backup Script

# Configuration
BACKUP_DIR="/backup/etcd"
RETENTION_DAYS=7
ETCD_ENDPOINTS="https://127.0.0.1:2379"
ETCD_CACERT="/etc/kubernetes/pki/etcd/ca.crt"
ETCD_CERT="/etc/kubernetes/pki/etcd/server.crt"
ETCD_KEY="/etc/kubernetes/pki/etcd/server.key"
TIMESTAMP=$(date +%Y%m%d-%H%M%S)
BACKUP_FILE="${BACKUP_DIR}/etcd-snapshot-${TIMESTAMP}.db"

# Create backup directory if it doesn't exist
mkdir -p ${BACKUP_DIR}

# Take snapshot
echo "[$(date)] Starting etcd backup..."
ETCDCTL_API=3 etcdctl snapshot save ${BACKUP_FILE} \
  --endpoints=${ETCD_ENDPOINTS} \
  --cacert=${ETCD_CACERT} \
  --cert=${ETCD_CERT} \
  --key=${ETCD_KEY}

# Verify the backup
if ETCDCTL_API=3 etcdctl snapshot status ${BACKUP_FILE} > /dev/null 2>&1; then
    echo "[$(date)] ✅ Backup successful: ${BACKUP_FILE}"
    # Show backup info
    ETCDCTL_API=3 etcdctl snapshot status ${BACKUP_FILE} --write-out=table
else
    echo "[$(date)] ❌ Backup FAILED!"
    exit 1
fi

# Remove old backups (keep last N days)
echo "[$(date)] Cleaning up backups older than ${RETENTION_DAYS} days..."
find ${BACKUP_DIR} -name "etcd-snapshot-*.db" -mtime +${RETENTION_DAYS} -delete

echo "[$(date)] Backup process completed."
```

Set it up as a cron job:

```bash
# Run backup every 6 hours
crontab -e

# Add this line:
0 */6 * * * /usr/local/bin/etcd-backup.sh >> /var/log/etcd-backup.log 2>&1
```

### Backup Best Practices

| Practice | Why |
|----------|-----|
| Backup at least every **4-6 hours** | Minimize data loss |
| Store backups **off-cluster** (S3, GCS, Azure Blob) | If the cluster dies, backups survive |
| **Test your restores** regularly | A backup you can't restore is useless |
| Backup **before** cluster upgrades | So you can roll back |
| **Encrypt** backup files | They contain Secrets! |
| Keep at least **7 days** of backups | Catch issues that aren't noticed immediately |
| **Monitor** backup job success/failure | Use alerting if backup fails |

## 🔄 etcd Restore — Bringing Your Cluster Back

When disaster strikes, here's how to restore from a backup.

### Important Concepts Before Restoring

1. **Restore creates a NEW data directory** — it doesn't overwrite the existing one
2. **All members must restore from the SAME snapshot** — you can't mix snapshots
3. **The cluster ID and member IDs change** — it's essentially a new cluster
4. **Stop kube-apiserver first** — never restore while the API server is writing to etcd

### Single-Node Restore (kubeadm Cluster)

This is the most common scenario — your single control plane cluster needs a restore.

```bash
# Step 1: Stop the kube-apiserver and etcd
# On a kubeadm cluster, these are static Pods managed by kubelet
# Move the manifests out of the static pod directory to stop them

sudo mv /etc/kubernetes/manifests/kube-apiserver.yaml /tmp/
sudo mv /etc/kubernetes/manifests/etcd.yaml /tmp/

# Wait for them to stop
sleep 10

# Verify they've stopped
sudo crictl ps | grep -E "kube-apiserver|etcd"
# Should return nothing

# Step 2: Back up the current (broken) data directory (just in case!)
sudo mv /var/lib/etcd /var/lib/etcd.broken.bak

# Step 3: Restore the snapshot
ETCDCTL_API=3 etcdctl snapshot restore /backup/etcd-snapshot-20250529-143022.db \
  --data-dir=/var/lib/etcd \
  --name=<etcd-member-name> \
  --initial-cluster=<etcd-member-name>=https://<etcd-ip>:2380 \
  --initial-cluster-token=etcd-cluster-1 \
  --initial-advertise-peer-urls=https://<etcd-ip>:2380

# Example with actual values:
ETCDCTL_API=3 etcdctl snapshot restore /backup/etcd-snapshot-20250529-143022.db \
  --data-dir=/var/lib/etcd \
  --name=control-plane \
  --initial-cluster=control-plane=https://192.168.1.10:2380 \
  --initial-cluster-token=etcd-cluster-1 \
  --initial-advertise-peer-urls=https://192.168.1.10:2380

# Step 4: Set correct ownership on the restored data
sudo chown -R etcd:etcd /var/lib/etcd

# Step 5: Restart etcd and kube-apiserver by moving manifests back
sudo mv /tmp/etcd.yaml /etc/kubernetes/manifests/
sudo mv /tmp/kube-apiserver.yaml /etc/kubernetes/manifests/

# Step 6: Wait for them to start and verify
sleep 30
kubectl get nodes
kubectl get pods --all-namespaces
```

Let me explain each flag in the restore command:

| Flag | What It Does |
|------|-------------|
| `--data-dir` | Where to put the restored data (must be empty or non-existent) |
| `--name` | Name of this etcd member |
| `--initial-cluster` | Comma-separated list of all members and their peer URLs |
| `--initial-cluster-token` | A unique token to identify this cluster (prevents cross-cluster communication) |
| `--initial-advertise-peer-urls` | The URL this member advertises to other members |

### Multi-Node Restore (HA Cluster)

For a 3-node etcd cluster, you must restore on ALL nodes:

```bash
# ============ ON EVERY NODE ============
# Step 1: Stop kube-apiserver and etcd on ALL nodes first
sudo mv /etc/kubernetes/manifests/kube-apiserver.yaml /tmp/
sudo mv /etc/kubernetes/manifests/etcd.yaml /tmp/
sleep 10

# Step 2: Remove old data on ALL nodes
sudo mv /var/lib/etcd /var/lib/etcd.broken.bak

# ============ ON NODE 1 (etcd-0) ============
ETCDCTL_API=3 etcdctl snapshot restore /backup/etcd-backup.db \
  --data-dir=/var/lib/etcd \
  --name=etcd-0 \
  --initial-cluster=etcd-0=https://10.0.0.10:2380,etcd-1=https://10.0.0.11:2380,etcd-2=https://10.0.0.12:2380 \
  --initial-cluster-token=etcd-cluster-restored \
  --initial-advertise-peer-urls=https://10.0.0.10:2380

# ============ ON NODE 2 (etcd-1) ============
ETCDCTL_API=3 etcdctl snapshot restore /backup/etcd-backup.db \
  --data-dir=/var/lib/etcd \
  --name=etcd-1 \
  --initial-cluster=etcd-0=https://10.0.0.10:2380,etcd-1=https://10.0.0.11:2380,etcd-2=https://10.0.0.12:2380 \
  --initial-cluster-token=etcd-cluster-restored \
  --initial-advertise-peer-urls=https://10.0.0.11:2380

# ============ ON NODE 3 (etcd-2) ============
ETCDCTL_API=3 etcdctl snapshot restore /backup/etcd-backup.db \
  --data-dir=/var/lib/etcd \
  --name=etcd-2 \
  --initial-cluster=etcd-0=https://10.0.0.10:2380,etcd-1=https://10.0.0.11:2380,etcd-2=https://10.0.0.12:2380 \
  --initial-cluster-token=etcd-cluster-restored \
  --initial-advertise-peer-urls=https://10.0.0.12:2380

# ============ ON EVERY NODE ============
# Step 3: Fix ownership
sudo chown -R etcd:etcd /var/lib/etcd

# Step 4: Restart etcd and kube-apiserver
sudo mv /tmp/etcd.yaml /etc/kubernetes/manifests/
sudo mv /tmp/kube-apiserver.yaml /etc/kubernetes/manifests/

# Step 5: Verify
sleep 30
etcdctl member list --write-out=table
etcdctl endpoint health
kubectl get nodes
```

**Key Point:** Notice that the `--initial-cluster` is the same on all nodes (it contains ALL members), but `--name` and `--initial-advertise-peer-urls` are different on each node.

## 🔒 etcd Security Best Practices

### 1. Always Use TLS

etcd should **never** be exposed without TLS. In a kubeadm cluster, this is set up automatically:

```yaml
# From /etc/kubernetes/manifests/etcd.yaml (kubeadm generated)
spec:
  containers:
  - command:
    - etcd
    - --cert-file=/etc/kubernetes/pki/etcd/server.crt        # Server cert
    - --key-file=/etc/kubernetes/pki/etcd/server.key          # Server key
    - --peer-cert-file=/etc/kubernetes/pki/etcd/peer.crt      # Peer cert
    - --peer-key-file=/etc/kubernetes/pki/etcd/peer.key        # Peer key
    - --trusted-ca-file=/etc/kubernetes/pki/etcd/ca.crt       # CA cert
    - --peer-trusted-ca-file=/etc/kubernetes/pki/etcd/ca.crt  # Peer CA cert
    - --client-cert-auth=true                                 # Require client certs
    - --peer-client-cert-auth=true                            # Require peer certs
```

### 2. Restrict Network Access

```bash
# Only the API server should be able to reach etcd
# Use firewall rules to restrict access to ports 2379/2380

# Example with iptables:
# Allow API Server to reach etcd
iptables -A INPUT -p tcp --dport 2379 -s <api-server-ip> -j ACCEPT
# Allow etcd peers to communicate
iptables -A INPUT -p tcp --dport 2380 -s <etcd-peer-cidr> -j ACCEPT
# Block everything else to etcd
iptables -A INPUT -p tcp --dport 2379 -j DROP
iptables -A INPUT -p tcp --dport 2380 -j DROP
```

### 3. Encrypt Data at Rest

By default, Kubernetes Secrets are stored in etcd **base64 encoded but NOT encrypted**. Anyone with access to etcd can read them!

Enable encryption at rest:

```yaml
# /etc/kubernetes/encryption-config.yaml
apiVersion: apiserver.config.k8s.io/v1
kind: EncryptionConfiguration
resources:
  - resources:
      - secrets
    providers:
      - aescbc:
          keys:
            - name: key1
              secret: <base64-encoded-32-byte-key>  # Generate with: head -c 32 /dev/urandom | base64
      - identity: {}  # Fallback — allows reading unencrypted existing Secrets
```

Then configure the API Server to use it:

```yaml
# In kube-apiserver manifest, add:
- --encryption-provider-config=/etc/kubernetes/encryption-config.yaml
```

### 4. Encrypt Backups

```bash
# Encrypt your backup with gpg
gpg --symmetric --cipher-algo AES256 etcd-backup.db
# This creates etcd-backup.db.gpg

# To decrypt:
gpg --decrypt etcd-backup.db.gpg > etcd-backup-decrypted.db
```

Or with OpenSSL:

```bash
# Encrypt
openssl enc -aes-256-cbc -salt -pbkdf2 -in etcd-backup.db -out etcd-backup.db.enc

# Decrypt
openssl enc -aes-256-cbc -d -pbkdf2 -in etcd-backup.db.enc -out etcd-backup.db
```

## 🚨 Common etcd Issues and Troubleshooting

### Issue 1: "etcdserver: mvcc: database space exceeded"

**Problem:** etcd database has grown beyond its quota (default 2GB).

```bash
# Check current DB size
etcdctl endpoint status --write-out=table

# Fix: Compact and defrag
# Get current revision
REV=$(etcdctl endpoint status --write-out=json | python3 -c "import json,sys; print(json.load(sys.stdin)[0]['Status']['header']['revision'])")

# Compact
etcdctl compact $REV

# Defrag (one member at a time!)
etcdctl defrag

# Disable alarm (etcd sets an alarm when space is exceeded)
etcdctl alarm disarm

# Increase quota if needed
# Add to etcd flags: --quota-backend-bytes=8589934592 (8GB)
```

### Issue 2: Slow Disk (High fsync Latency)

**Problem:** etcd is slow because the disk is too slow.

```bash
# Check with fio (disk benchmark)
fio --rw=write --ioengine=sync --fdatasync=1 --directory=/var/lib/etcd \
    --size=22m --bs=2300 --name=etcd-benchmark

# If the 99th percentile fsync is > 10ms, your disk is too slow

# Fix: Use SSDs!
# Also check if other processes are using the same disk
```

### Issue 3: Leader Keeps Changing

**Problem:** Frequent leader elections indicate instability.

```bash
# Check leader changes
etcdctl endpoint status --write-out=table
# Look for rapid changes in the IS LEADER column

# Common causes:
# 1. Slow disk (most common!) — use SSDs
# 2. Network issues between members — check latency
# 3. Resource starvation — etcd needs dedicated CPU/memory
# 4. Heartbeat/election timeout too low — increase them
```

### Issue 4: Member Failed to Start After Crash

```bash
# Check etcd logs
journalctl -u etcd -l --no-pager | tail -50
# or for kubeadm:
kubectl logs etcd-<node-name> -n kube-system

# Common fix: remove corrupt WAL/snap files and rejoin
# (Only if you have a backup or other healthy members!)
sudo rm -rf /var/lib/etcd/member
# Then restart etcd — it will rejoin the cluster and sync
```

## 📋 CKA/CKAD Exam Tips for etcd

If you're preparing for the CKA (Certified Kubernetes Administrator) exam, etcd backup and restore is a **very common exam question**. Here's your cheat sheet:

```bash
# ========== BACKUP ==========
ETCDCTL_API=3 etcdctl snapshot save /opt/etcd-backup.db \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key

# Verify
ETCDCTL_API=3 etcdctl snapshot status /opt/etcd-backup.db --write-out=table

# ========== RESTORE ==========
# 1. Stop API server and etcd
sudo mv /etc/kubernetes/manifests/kube-apiserver.yaml /tmp/
sudo mv /etc/kubernetes/manifests/etcd.yaml /tmp/

# 2. Restore (to a NEW data dir)
ETCDCTL_API=3 etcdctl snapshot restore /opt/etcd-backup.db \
  --data-dir=/var/lib/etcd-restored

# 3. Update etcd manifest to use new data dir
sudo vi /tmp/etcd.yaml
# Change: --data-dir=/var/lib/etcd
# To:     --data-dir=/var/lib/etcd-restored
# Also update the hostPath volume mount

# 4. Move manifests back
sudo mv /tmp/etcd.yaml /etc/kubernetes/manifests/
sudo mv /tmp/kube-apiserver.yaml /etc/kubernetes/manifests/

# 5. Verify
sleep 30
kubectl get pods -A
```

**Exam Tip:** The exam might not have environment variables pre-set. You need to remember the `--endpoints`, `--cacert`, `--cert`, and `--key` flags!

## 🧠 Quick Summary

| Concept | Key Takeaway |
|---------|-------------|
| **What is etcd?** | Distributed key-value store — the brain of Kubernetes |
| **What it stores** | ALL cluster state — Pods, Services, Secrets, everything |
| **Who talks to it** | Only the API Server (never directly) |
| **Consensus** | Raft algorithm — needs majority (quorum) to work |
| **Cluster size** | 3 or 5 nodes (always odd numbers!) |
| **Port 2379** | Client communication (API Server → etcd) |
| **Port 2380** | Peer communication (etcd ↔ etcd) |
| **Backup command** | `etcdctl snapshot save <file>` |
| **Restore command** | `etcdctl snapshot restore <file> --data-dir=<new-dir>` |
| **#1 enemy** | Slow disks — always use SSDs for etcd |
| **#1 rule** | ALWAYS have backups. Test them. Store off-cluster. |
