# 10.44 — Kubernetes API — Client Libraries, Extending the API

> **One Line Answer:** The Kubernetes API is the **single gateway** through which everything in your cluster happens — every `kubectl` command, every controller action, every dashboard click is just an HTTP request to the API server, and you can use **client libraries** to talk to it programmatically or even **extend it** with your own custom endpoints.

---

## 🤔 Why Should You Understand the Kubernetes API?

You've been using `kubectl` this entire time. Every command you type:

```bash
kubectl get pods
kubectl create deployment nginx --image=nginx
kubectl delete service my-svc
```

Behind the scenes, **kubectl is just making HTTP API calls to the Kubernetes API server.** That's it. There's no magic.

```
kubectl get pods  →  GET https://api-server:6443/api/v1/namespaces/default/pods
```

Understanding the API gives you **superpowers**:

| Superpower | How |
|-----------|-----|
| **Automate anything** | Write scripts/apps that manage your cluster |
| **Build custom tools** | Dashboards, operators, CLI tools |
| **Debug deeper** | Understand exactly what's happening under the hood |
| **Extend Kubernetes** | Add your own resources and APIs |
| **Pass CKA/CKAD exams** | API knowledge is tested directly |

## 📦 The Kubernetes API — The Big Picture

### Everything is an API

Kubernetes is designed as an **API-first** system. The API server (`kube-apiserver`) is the central hub:

```
                          ┌──────────────┐
         kubectl ────────▶│              │
         Dashboard ──────▶│  API Server  │◀──── Controller Manager
         Your App ───────▶│  (REST API)  │◀──── Scheduler
         CI/CD ──────────▶│              │◀──── kubelet
                          └──────┬───────┘
                                 │
                                 ▼
                          ┌──────────────┐
                          │     etcd     │
                          └──────────────┘
```

**Every single interaction** with Kubernetes goes through the API server. There are no backdoors, no shortcuts.

### It's Just REST

The Kubernetes API is a standard **REST API** using HTTP:

| HTTP Method | Kubernetes Action | Example |
|------------|-------------------|---------|
| `GET` | Read / List | Get a Pod, list all Services |
| `POST` | Create | Create a Deployment |
| `PUT` | Update (full replace) | Update entire Pod spec |
| `PATCH` | Update (partial) | Change just the image field |
| `DELETE` | Delete | Delete a ConfigMap |

### API Response Format

Everything is JSON (or protobuf internally for performance):

```bash
# This kubectl command:
kubectl get pod nginx -o json

# Is actually this API call:
# GET /api/v1/namespaces/default/pods/nginx
# And returns something like:
{
  "apiVersion": "v1",
  "kind": "Pod",
  "metadata": {
    "name": "nginx",
    "namespace": "default",
    "uid": "12345-abcde-67890",
    "resourceVersion": "54321",
    "creationTimestamp": "2025-05-29T10:00:00Z"
  },
  "spec": {
    "containers": [
      {
        "name": "nginx",
        "image": "nginx:1.25"
      }
    ]
  },
  "status": {
    "phase": "Running"
  }
}
```

## 🗺️ API Structure — How URLs are Organized

The Kubernetes API is organized into **groups** and **versions**. Understanding the URL structure is key.

### Core API Group (Legacy)

The original resources (Pods, Services, ConfigMaps, etc.) live in the **core** group:

```
/api/v1/...

Examples:
/api/v1/namespaces                          → List all namespaces
/api/v1/namespaces/default/pods             → List pods in default namespace
/api/v1/namespaces/default/pods/nginx       → Get specific pod "nginx"
/api/v1/namespaces/default/services         → List services
/api/v1/nodes                               → List all nodes (not namespaced)
```

### Named API Groups

Newer resources are organized into **named groups** under `/apis/`:

```
/apis/<group>/<version>/...

Examples:
/apis/apps/v1/namespaces/default/deployments        → Deployments
/apis/apps/v1/namespaces/default/statefulsets        → StatefulSets
/apis/apps/v1/namespaces/default/daemonsets          → DaemonSets
/apis/batch/v1/namespaces/default/jobs               → Jobs
/apis/batch/v1/namespaces/default/cronjobs           → CronJobs
/apis/networking.k8s.io/v1/namespaces/default/ingresses → Ingresses
/apis/rbac.authorization.k8s.io/v1/clusterroles      → ClusterRoles
```

### The Pattern

```
Cluster-scoped:  /api(s)/<group>/<version>/<resource>
Namespaced:      /api(s)/<group>/<version>/namespaces/<ns>/<resource>
Specific:        /api(s)/<group>/<version>/namespaces/<ns>/<resource>/<name>
```

### How to Find the API Path for Any Resource

```bash
# Use kubectl api-resources to see all resources and their API groups
kubectl api-resources

# Output (shortened):
# NAME                  SHORTNAMES   APIVERSION                     NAMESPACED   KIND
# pods                  po           v1                             true         Pod
# services              svc          v1                             true         Service
# deployments           deploy       apps/v1                        true         Deployment
# ingresses             ing          networking.k8s.io/v1           true         Ingress
# clusterroles                       rbac.authorization.k8s.io/v1   false        ClusterRole

# You can also filter by API group
kubectl api-resources --api-group=apps
kubectl api-resources --api-group=batch
```

### API Versions Explained

| Version | Stability | Meaning |
|---------|-----------|---------|
| `v1alpha1` | Alpha | Experimental — may change or be removed, disabled by default |
| `v1beta1` | Beta | Pre-release — mostly stable, enabled by default |
| `v1` | Stable | Production-ready — won't change |
| `v2` | Stable v2 | Next major version of a stable API |

```bash
# See which API versions are available on your cluster
kubectl api-versions

# Output (partial):
# admissionregistration.k8s.io/v1
# apps/v1
# batch/v1
# certificates.k8s.io/v1
# networking.k8s.io/v1
# rbac.authorization.k8s.io/v1
# storage.k8s.io/v1
# v1
```

## 🔧 Talking to the API Directly

### Method 1: kubectl proxy (Easiest for Learning)

`kubectl proxy` creates a local proxy that handles authentication for you:

```bash
# Start the proxy (runs on localhost:8001 by default)
kubectl proxy &

# Now you can make API calls without any certificates!
# List all pods in default namespace
curl http://localhost:8001/api/v1/namespaces/default/pods

# Get a specific pod
curl http://localhost:8001/api/v1/namespaces/default/pods/nginx

# List all deployments
curl http://localhost:8001/apis/apps/v1/namespaces/default/deployments

# Get cluster info
curl http://localhost:8001/api/v1/nodes

# List all API resources
curl http://localhost:8001/apis
```

Pretty output with `jq`:

```bash
# List just the pod names
curl -s http://localhost:8001/api/v1/namespaces/default/pods | jq '.items[].metadata.name'

# Output:
# "nginx-abc123"
# "web-frontend-xyz789"
# "api-backend-def456"
```

### Method 2: Direct API Call with Certificates

Without the proxy, you need to provide authentication directly:

```bash
# Get the API server URL
APISERVER=$(kubectl config view --minify -o jsonpath='{.clusters[0].cluster.server}')

# Get a service account token
TOKEN=$(kubectl create token default)

# Make the API call
curl -s ${APISERVER}/api/v1/namespaces/default/pods \
  --header "Authorization: Bearer ${TOKEN}" \
  --cacert /etc/kubernetes/pki/ca.crt

# Or if you want to skip TLS verification (NOT for production!)
curl -sk ${APISERVER}/api/v1/namespaces/default/pods \
  --header "Authorization: Bearer ${TOKEN}"
```

### Method 3: Using kubectl --raw

The simplest way to make raw API calls:

```bash
# Raw GET request
kubectl get --raw /api/v1/namespaces/default/pods

# Pretty print
kubectl get --raw /api/v1/namespaces/default/pods | python3 -m json.tool

# Get API server health
kubectl get --raw /healthz
kubectl get --raw /readyz
kubectl get --raw /livez

# Get API server metrics
kubectl get --raw /metrics | head -50
```

### Creating Resources via the API

```bash
# Create a Pod using curl
kubectl proxy &

curl -X POST http://localhost:8001/api/v1/namespaces/default/pods \
  -H "Content-Type: application/json" \
  -d '{
    "apiVersion": "v1",
    "kind": "Pod",
    "metadata": {
      "name": "api-created-pod"
    },
    "spec": {
      "containers": [{
        "name": "nginx",
        "image": "nginx:1.25"
      }]
    }
  }'

# Delete it
curl -X DELETE http://localhost:8001/api/v1/namespaces/default/pods/api-created-pod
```

## 🔐 API Authentication and Authorization

Every API request goes through three stages:

```
Request → [Authentication] → [Authorization] → [Admission Control] → etcd
          "Who are you?"     "Can you do this?"  "Is this valid?"
```

### Authentication Methods

| Method | How it Works | Who Uses It |
|--------|-------------|-------------|
| **X.509 Client Certificates** | TLS client cert signed by cluster CA | kubectl (kubeconfig), kubelet |
| **Bearer Tokens** | Token in HTTP header | Service Accounts, OIDC |
| **Service Account Tokens** | JWT tokens mounted in Pods | Pods talking to API |
| **OIDC (OpenID Connect)** | External identity provider | Human users (Google, Azure AD, etc.) |
| **Webhook Token Auth** | External webhook validates tokens | Custom integrations |

### How kubectl Authenticates

Your `~/.kube/config` (kubeconfig) file contains everything:

```yaml
apiVersion: v1
kind: Config
clusters:
- cluster:
    certificate-authority-data: <base64-CA-cert>   # How to verify the API server
    server: https://192.168.1.10:6443               # Where the API server is
  name: my-cluster
users:
- name: admin
  user:
    client-certificate-data: <base64-client-cert>   # Who I am
    client-key-data: <base64-client-key>            # Proof of identity
contexts:
- context:
    cluster: my-cluster
    user: admin
    namespace: default                               # Default namespace
  name: my-context
current-context: my-context
```

### Service Account Tokens — How Pods Talk to the API

When a Pod runs inside Kubernetes, it automatically gets a **Service Account token** mounted at a known path:

```bash
# Inside any Pod, you'll find:
ls /var/run/secrets/kubernetes.io/serviceaccount/

# token     — JWT token for authentication
# ca.crt    — CA certificate to verify API server
# namespace — the namespace this Pod is in

# You can use these to talk to the API from inside a Pod!
TOKEN=$(cat /var/run/secrets/kubernetes.io/serviceaccount/token)
CACERT=/var/run/secrets/kubernetes.io/serviceaccount/ca.crt

curl -s --cacert ${CACERT} \
  -H "Authorization: Bearer ${TOKEN}" \
  https://kubernetes.default.svc/api/v1/namespaces/default/pods
```

**`kubernetes.default.svc`** is a built-in Service that points to the API server. Every Pod can reach it.

### Python Client — Getting Started

This is the most popular choice for scripting and automation.

#### Installation

```bash
pip install kubernetes
```

#### Example 1: List All Pods

```python
from kubernetes import client, config

# Load kubeconfig (works if you have ~/.kube/config set up)
config.load_kube_config()

# Or if running inside a Pod, use:
# config.load_incluster_config()

# Create an API client
v1 = client.CoreV1Api()

# List all pods in default namespace
pods = v1.list_namespaced_pod(namespace="default")

print(f"Found {len(pods.items)} pods in default namespace:\n")
for pod in pods.items:
    print(f"  Name: {pod.metadata.name}")
    print(f"  Status: {pod.status.phase}")
    print(f"  IP: {pod.status.pod_ip}")
    print(f"  Node: {pod.spec.node_name}")
    print("  ---")
```

Output:

```
Found 3 pods in default namespace:

  Name: nginx-abc123
  Status: Running
  IP: 10.244.1.5
  Node: worker-1
  ---
  Name: web-frontend-xyz789
  Status: Running
  IP: 10.244.2.3
  Node: worker-2
  ---
```

#### Example 2: Create a Deployment

```python
from kubernetes import client, config

config.load_kube_config()

# Create Apps API client (Deployments are in the "apps" group)
apps_v1 = client.AppsV1Api()

# Define the Deployment
deployment = client.V1Deployment(
    metadata=client.V1ObjectMeta(name="web-app"),
    spec=client.V1DeploymentSpec(
        replicas=3,
        selector=client.V1LabelSelector(
            match_labels={"app": "web-app"}
        ),
        template=client.V1PodTemplateSpec(
            metadata=client.V1ObjectMeta(
                labels={"app": "web-app"}
            ),
            spec=client.V1PodSpec(
                containers=[
                    client.V1Container(
                        name="nginx",
                        image="nginx:1.25",
                        ports=[client.V1ContainerPort(container_port=80)],
                        resources=client.V1ResourceRequirements(
                            requests={"cpu": "100m", "memory": "128Mi"},
                            limits={"cpu": "250m", "memory": "256Mi"}
                        )
                    )
                ]
            )
        )
    )
)

# Create it!
result = apps_v1.create_namespaced_deployment(
    namespace="default",
    body=deployment
)
print(f"Deployment '{result.metadata.name}' created with {result.spec.replicas} replicas!")
```

#### Example 3: Watch for Events (Real-Time)

```python
from kubernetes import client, config, watch

config.load_kube_config()
v1 = client.CoreV1Api()

# Create a watcher
w = watch.Watch()

print("Watching for Pod events (Ctrl+C to stop)...\n")

# Watch pods in all namespaces
for event in w.stream(v1.list_pod_for_all_namespaces):
    pod = event['object']
    event_type = event['type']    # ADDED, MODIFIED, DELETED
    
    print(f"[{event_type}] Pod: {pod.metadata.name} "
          f"Namespace: {pod.metadata.namespace} "
          f"Status: {pod.status.phase}")
```

Output (as events happen in real-time):

```
Watching for Pod events (Ctrl+C to stop)...

[ADDED] Pod: nginx-abc123 Namespace: default Status: Running
[MODIFIED] Pod: web-app-xyz Namespace: production Status: Running
[DELETED] Pod: old-pod-123 Namespace: staging Status: Succeeded
[ADDED] Pod: new-pod-456 Namespace: default Status: Pending
```

#### Example 4: Delete Old Completed Pods

```python
from kubernetes import client, config

config.load_kube_config()
v1 = client.CoreV1Api()

# Find and delete all completed (Succeeded) pods
pods = v1.list_pod_for_all_namespaces()

deleted_count = 0
for pod in pods.items:
    if pod.status.phase == "Succeeded":
        v1.delete_namespaced_pod(
            name=pod.metadata.name,
            namespace=pod.metadata.namespace
        )
        print(f"Deleted completed pod: {pod.metadata.name} in {pod.metadata.namespace}")
        deleted_count += 1

print(f"\nCleaned up {deleted_count} completed pods!")
```

#### Example 5: Scale a Deployment

```python
from kubernetes import client, config

config.load_kube_config()
apps_v1 = client.AppsV1Api()

# Scale the deployment to 5 replicas
body = {"spec": {"replicas": 5}}

result = apps_v1.patch_namespaced_deployment_scale(
    name="web-app",
    namespace="default",
    body=body
)

print(f"Scaled 'web-app' to {result.spec.replicas} replicas!")
```

### Go Client (client-go) — The Gold Standard

`client-go` is the most powerful Kubernetes client library. It's what Kubernetes itself is built with.

#### Installation

```bash
go get k8s.io/client-go@latest
go get k8s.io/apimachinery@latest
```

#### Example: List Pods in Go

```go
package main

import (
    "context"
    "fmt"
    "os"
    "path/filepath"

    metav1 "k8s.io/apimachinery/pkg/apis/meta/v1"
    "k8s.io/client-go/kubernetes"
    "k8s.io/client-go/tools/clientcmd"
)

func main() {
    // Load kubeconfig from default location
    kubeconfig := filepath.Join(os.Getenv("HOME"), ".kube", "config")
    config, err := clientcmd.BuildConfigFromFlags("", kubeconfig)
    if err != nil {
        panic(err)
    }

    // Create the clientset
    clientset, err := kubernetes.NewForConfig(config)
    if err != nil {
        panic(err)
    }

    // List pods in default namespace
    pods, err := clientset.CoreV1().Pods("default").List(
        context.TODO(),
        metav1.ListOptions{},
    )
    if err != nil {
        panic(err)
    }

    fmt.Printf("Found %d pods in default namespace:\n\n", len(pods.Items))
    for _, pod := range pods.Items {
        fmt.Printf("  Name: %s | Status: %s | IP: %s\n",
            pod.Name, pod.Status.Phase, pod.Status.PodIP)
    }
}
```

## 🔌 Extending the Kubernetes API

This is where things get really powerful. You can **add your own resources** to Kubernetes!

### Why Extend the API?

Imagine you want Kubernetes to manage something it doesn't natively support — like:
- **Databases** (e.g., a "PostgresCluster" resource)
- **Certificates** (e.g., a "Certificate" resource — this is what cert-manager does)
- **External DNS records** (e.g., a "DNSRecord" resource)
- **Application definitions** (e.g., an "Application" resource for ArgoCD)

You extend the API so you can write YAML like this:

```yaml
apiVersion: databases.example.com/v1
kind: PostgresCluster
metadata:
  name: my-database
spec:
  version: "15"
  replicas: 3
  storage:
    size: 100Gi
```

And Kubernetes treats it like a first-class resource, just like a Pod or Deployment!

### Method 1: Custom Resource Definitions (CRDs)

CRDs are the **easiest** way to extend the API. You define a new resource type and Kubernetes creates the API endpoints for you.

```yaml
# postgres-crd.yaml — Define a new "PostgresCluster" resource
apiVersion: apiextensions.k8s.io/v1
kind: CustomResourceDefinition
metadata:
  name: postgresclusters.databases.example.com
spec:
  group: databases.example.com         # API group
  versions:
    - name: v1                         # API version
      served: true                     # Is this version available?
      storage: true                    # Is this the storage version?
      schema:
        openAPIV3Schema:               # Validation schema
          type: object
          properties:
            spec:
              type: object
              properties:
                version:
                  type: string
                  description: "PostgreSQL version"
                replicas:
                  type: integer
                  minimum: 1
                  maximum: 10
                  description: "Number of replicas"
                storage:
                  type: object
                  properties:
                    size:
                      type: string
                      description: "Storage size (e.g., 100Gi)"
              required: ["version", "replicas"]
      additionalPrinterColumns:        # What kubectl shows in table output
        - name: Version
          type: string
          jsonPath: .spec.version
        - name: Replicas
          type: integer
          jsonPath: .spec.replicas
        - name: Age
          type: date
          jsonPath: .metadata.creationTimestamp
  scope: Namespaced                    # Can be Namespaced or Cluster
  names:
    plural: postgresclusters           # /apis/.../postgresclusters
    singular: postgrescluster          # Used in kubectl
    kind: PostgresCluster              # Resource kind
    shortNames:                        # Short names for kubectl
      - pg
      - pgc
```

Apply it:

```bash
# Create the CRD
kubectl apply -f postgres-crd.yaml

# Now the new API endpoint exists!
kubectl api-resources | grep postgres
# postgresclusters   pg,pgc   databases.example.com/v1   true   PostgresCluster
```

Now you can create instances of your custom resource:

```yaml
# my-postgres.yaml
apiVersion: databases.example.com/v1
kind: PostgresCluster
metadata:
  name: production-db
  namespace: default
spec:
  version: "15"
  replicas: 3
  storage:
    size: 100Gi
```

```bash
# Create it just like any other resource!
kubectl apply -f my-postgres.yaml

# List your custom resources
kubectl get postgresclusters
# Or use the short name:
kubectl get pg

# Output:
# NAME            VERSION   REPLICAS   AGE
# production-db   15        3          10s

# Describe it
kubectl describe pg production-db

# Delete it
kubectl delete pg production-db
```

**But wait** — the CRD only stores the data. It doesn't actually **do** anything (like create actual PostgreSQL Pods). For that, you need a **Controller** or **Operator** (covered in 10.40-10.42).

### Method 2: API Aggregation Layer

For more complex scenarios, you can build a complete **custom API server** that plugs into the main API server:

```
┌─────────────────────────────────────────────┐
│              kube-apiserver                  │
│                                             │
│  /api/v1/pods          → Built-in handler   │
│  /apis/apps/v1/...     → Built-in handler   │
│  /apis/metrics.k8s.io  → Aggregated ────────┼──▶ metrics-server (custom API)
│  /apis/custom.io/v1    → Aggregated ────────┼──▶ your-api-server (custom API)
│                                             │
└─────────────────────────────────────────────┘
```

This is done with an `APIService` resource:

```yaml
apiVersion: apiregistration.k8s.io/v1
kind: APIService
metadata:
  name: v1.custom.example.com
spec:
  group: custom.example.com
  version: v1
  service:
    name: my-custom-api        # K8s Service pointing to your API server
    namespace: kube-system
  groupPriorityMinimum: 1000
  versionPriority: 15
  insecureSkipTLSVerify: false
  caBundle: <base64-ca-cert>
```

**When to use which:**

| Approach | When to Use |
|----------|------------|
| **CRDs** | Most cases — simple, no extra infrastructure needed |
| **API Aggregation** | When you need custom storage, complex validation, custom sub-resources, or your own API server logic |

### Method 3: Admission Webhooks

Admission webhooks let you **intercept and modify** API requests before they're saved to etcd:

```
kubectl apply → API Server → [Mutating Webhooks] → [Validating Webhooks] → etcd
                               "Add a sidecar"      "Check policy"
```

There are two types:

#### Mutating Admission Webhook

Can **modify** the incoming request. Example: automatically inject a sidecar container:

```yaml
apiVersion: admissionregistration.k8s.io/v1
kind: MutatingWebhookConfiguration
metadata:
  name: sidecar-injector
webhooks:
  - name: sidecar.example.com
    clientConfig:
      service:
        name: sidecar-injector-svc
        namespace: kube-system
        path: "/inject"
      caBundle: <base64-ca-cert>
    rules:
      - apiGroups: [""]
        resources: ["pods"]
        apiVersions: ["v1"]
        operations: ["CREATE"]
    admissionReviewVersions: ["v1"]
    sideEffects: None
    failurePolicy: Ignore    # Don't block if webhook is down
```

#### Validating Admission Webhook

Can only **accept or reject** requests. Example: block containers running as root:

```yaml
apiVersion: admissionregistration.k8s.io/v1
kind: ValidatingWebhookConfiguration
metadata:
  name: no-root-containers
webhooks:
  - name: validate.security.example.com
    clientConfig:
      service:
        name: security-validator-svc
        namespace: kube-system
        path: "/validate"
      caBundle: <base64-ca-cert>
    rules:
      - apiGroups: [""]
        resources: ["pods"]
        apiVersions: ["v1"]
        operations: ["CREATE", "UPDATE"]
    admissionReviewVersions: ["v1"]
    sideEffects: None
    failurePolicy: Fail      # Block if webhook is down (strict)
```

**Real-world tools that use webhooks:**
- **Istio** — Mutating webhook to inject Envoy sidecar
- **OPA/Gatekeeper** — Validating webhook for policy enforcement
- **cert-manager** — Mutating webhook to inject TLS certificates
- **Kyverno** — Both mutating and validating for policy management

## 🔄 API Versioning and Deprecation

Kubernetes evolves, and APIs change over time. Here's how it works:

### The API Lifecycle

```
Alpha (v1alpha1) → Beta (v1beta1) → Stable (v1) → Deprecated → Removed
```

| Stage | What Happens |
|-------|-------------|
| **Alpha** | Feature is experimental, disabled by default, may be removed |
| **Beta** | Feature is mostly stable, enabled by default, schema may change |
| **Stable** | Feature is production-ready, won't have breaking changes |
| **Deprecated** | Still works but will be removed in a future version |
| **Removed** | Gone! Your old YAML will break |

### How to Check for Deprecated APIs

```bash
# Install kubent (Kube No Trouble) — finds deprecated APIs
# https://github.com/dober-man/kubent
kubectl get all -A -o json | kubent

# Or use kubectl convert plugin
kubectl convert -f old-deployment.yaml --output-version apps/v1

# Check which API versions a resource supports
kubectl explain deployment --api-version=apps/v1
```

### kubectl convert Example

If you have old YAML using a deprecated API:

```yaml
# OLD (deprecated)
apiVersion: extensions/v1beta1
kind: Ingress
...
```

Convert to the new API:

```bash
kubectl convert -f old-ingress.yaml --output-version networking.k8s.io/v1 > new-ingress.yaml
```

```yaml
# NEW (current)
apiVersion: networking.k8s.io/v1
kind: Ingress
...
```

## 🎯 API Request Flow — The Complete Picture

Here's what happens when you run `kubectl create deployment nginx --image=nginx`:

```
Step 1: kubectl reads ~/.kube/config for authentication info
        ↓
Step 2: kubectl constructs the API request
        POST /apis/apps/v1/namespaces/default/deployments
        Body: { Deployment JSON }
        ↓
Step 3: Request hits kube-apiserver
        ↓
Step 4: AUTHENTICATION — "Who is making this request?"
        (Validates the client certificate from kubeconfig)
        ↓
Step 5: AUTHORIZATION — "Is this user allowed to create deployments?"
        (Checks RBAC rules — Roles, ClusterRoles, Bindings)
        ↓
Step 6: MUTATING ADMISSION — "Should we modify this request?"
        (Webhooks may add labels, inject sidecars, set defaults)
        ↓
Step 7: SCHEMA VALIDATION — "Is the YAML/JSON valid?"
        (Checks against the OpenAPI schema)
        ↓
Step 8: VALIDATING ADMISSION — "Should we allow this?"
        (Webhooks may reject based on policies)
        ↓
Step 9: PERSIST — Write to etcd
        (The Deployment object is stored as a key-value pair)
        ↓
Step 10: RESPONSE — Return success to kubectl
         "deployment.apps/nginx created"
        ↓
Step 11: CONTROLLERS REACT
         - Deployment controller sees the new Deployment → creates a ReplicaSet
         - ReplicaSet controller sees the new ReplicaSet → creates Pods
         - Scheduler sees unscheduled Pods → assigns them to nodes
         - kubelet on each node sees assigned Pods → starts containers
```

## 🧠 Quick Summary

| Concept | Key Takeaway |
|---------|-------------|
| **Kubernetes API** | REST API — everything is an HTTP call |
| **API Server** | The only gateway to the cluster |
| **API Groups** | Core (`/api/v1`) and named (`/apis/<group>/<version>`) |
| **kubectl proxy** | Local proxy that handles auth — great for learning |
| **kubectl --raw** | Make raw API calls through kubectl |
| **Python client** | `pip install kubernetes` — most popular for automation |
| **Go client-go** | Most powerful, used by Kubernetes itself |
| **CRDs** | Add your own resource types to Kubernetes |
| **API Aggregation** | Plug in your own API server |
| **Mutating Webhooks** | Modify requests before they're saved |
| **Validating Webhooks** | Accept or reject requests based on policies |
| **API Versioning** | alpha → beta → stable → deprecated → removed |
