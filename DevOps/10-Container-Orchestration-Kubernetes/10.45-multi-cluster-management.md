# 10.45 — Multi-Cluster Management — Federation, Rancher, Cluster API

> **One Line Answer:** As your organization grows, one Kubernetes cluster is never enough — you'll need **multiple clusters** across regions, clouds, and environments, and tools like **Kubernetes Federation**, **Rancher**, and **Cluster API** help you manage them all from a single place instead of going crazy.

---

## 🤔 Why Would You Need Multiple Clusters?

When you're learning Kubernetes, one cluster feels like a lot. But in the real world, organizations run **many** clusters. Here's why:

### Real-World Scenarios

| Scenario | Why Multiple Clusters? |
|----------|----------------------|
| **Dev / Staging / Production** | You don't want a developer's experiment crashing your production workload |
| **Multi-Region** | Users in Europe need low latency → run a cluster in EU. Users in Asia → cluster in Asia |
| **Multi-Cloud** | Avoid vendor lock-in → run on AWS, Azure, AND GCP |
| **Compliance** | EU data must stay in EU (GDPR) → separate cluster in EU region |
| **Team Isolation** | Different teams get their own clusters for full autonomy |
| **Blast Radius** | If one cluster dies, others keep running |
| **Scale Limits** | A single cluster has limits (~5000 nodes, ~150000 pods) |
| **Edge Computing** | Clusters at the edge (stores, factories, IoT) + central cloud cluster |
| **Hybrid Cloud** | Some workloads on-premise + some in the cloud |

### The Problem This Creates

Imagine you have 15 clusters. Now you need to:

- Deploy the same app to all 15 clusters
- Manage RBAC across all 15 clusters
- Monitor all 15 clusters
- Upgrade all 15 clusters
- Handle DNS and traffic routing across clusters
- Keep configurations consistent

Doing this manually is a **nightmare**. You need tools to help you manage multiple clusters. Let's explore the main approaches.

---

## 1️⃣ Kubernetes Federation (KubeFed)

### What is Federation?

Kubernetes Federation (also known as **KubeFed v2**) lets you **coordinate configuration across multiple Kubernetes clusters** from a single control plane. Think of it as a "cluster of clusters."

```
                    ┌─────────────────────┐
                    │   Host Cluster      │
                    │   (Federation       │
                    │    Control Plane)   │
                    └──────────┬──────────┘
                               │
              ┌────────────────┼────────────────┐
              ▼                ▼                ▼
     ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
     │  Cluster A   │  │  Cluster B   │  │  Cluster C   │
     │  (US-East)   │  │  (EU-West)   │  │  (AP-South)  │
     └──────────────┘  └──────────────┘  └──────────────┘
```

### How Federation Works

Instead of creating resources directly in each cluster, you create **federated resources** in the host cluster, and KubeFed distributes them:

```
Regular Kubernetes:          Federation:
                              
kubectl → Cluster A           kubectl → Federation Control Plane
kubectl → Cluster B                         ↓
kubectl → Cluster C           FederatedDeployment → Cluster A
                                                  → Cluster B
(3 separate commands)                             → Cluster C
                              
                              (1 command, distributed automatically)
```

### Setting Up KubeFed

#### Step 1: Install KubeFed on the Host Cluster

```bash
# The host cluster is where the Federation control plane runs
# It can also be a member cluster

# Install KubeFed using Helm
helm repo add kubefed https://raw.githubusercontent.com/kubernetes-sigs/kubefed/master/charts

helm install kubefed kubefed/kubefed \
  --namespace kube-federation-system \
  --create-namespace \
  --version 0.10.0

# Verify installation
kubectl get pods -n kube-federation-system
# NAME                                          READY   STATUS    RESTARTS   AGE
# kubefed-admission-webhook-xxx                 1/1     Running   0          30s
# kubefed-controller-manager-xxx                1/1     Running   0          30s
```

#### Step 2: Install kubefedctl CLI

```bash
# Download kubefedctl
VERSION=0.10.0
curl -LO https://github.com/kubernetes-sigs/kubefed/releases/download/v${VERSION}/kubefedctl-${VERSION}-linux-amd64.tgz
tar xzf kubefedctl-${VERSION}-linux-amd64.tgz
sudo mv kubefedctl /usr/local/bin/
kubefedctl version
```

#### Step 3: Join Member Clusters

```bash
# First, make sure your kubeconfig has contexts for all clusters
kubectl config get-contexts
# CURRENT   NAME          CLUSTER       AUTHINFO      NAMESPACE
# *         host-cluster  host-cluster  admin         default
#           cluster-a     cluster-a     admin-a       default
#           cluster-b     cluster-b     admin-b       default
#           cluster-c     cluster-c     admin-c       default

# Join clusters to the federation
kubefedctl join cluster-a \
  --cluster-context=cluster-a \
  --host-cluster-context=host-cluster \
  --v=2

kubefedctl join cluster-b \
  --cluster-context=cluster-b \
  --host-cluster-context=host-cluster \
  --v=2

kubefedctl join cluster-c \
  --cluster-context=cluster-c \
  --host-cluster-context=host-cluster \
  --v=2

# Verify joined clusters
kubectl get kubefedclusters -n kube-federation-system
# NAME        AGE   READY
# cluster-a   30s   True
# cluster-b   25s   True
# cluster-c   20s   True
```

#### Step 4: Enable Federated Types

Tell KubeFed which resource types you want to federate:

```bash
# Enable federation for common resource types
kubefedctl enable deployments.apps
kubefedctl enable services
kubefedctl enable configmaps
kubefedctl enable secrets
kubefedctl enable namespaces
kubefedctl enable ingresses.networking.k8s.io

# Check what's enabled
kubectl get federatedtypeconfigs -n kube-federation-system
# NAME                                AGE
# configmaps                          10s
# deployments.apps                    10s
# ingresses.networking.k8s.io         10s
# namespaces                          10s
# secrets                             10s
# services                            10s
```

### Creating Federated Resources

Now let's deploy a workload across all clusters!

#### Federated Namespace

```yaml
# federated-namespace.yaml
apiVersion: types.kubefed.io/v1beta1
kind: FederatedNamespace
metadata:
  name: my-app
  namespace: my-app
spec:
  placement:
    clusters:
      - name: cluster-a
      - name: cluster-b
      - name: cluster-c
```

#### Federated Deployment

```yaml
# federated-deployment.yaml
apiVersion: types.kubefed.io/v1beta1
kind: FederatedDeployment
metadata:
  name: web-app
  namespace: my-app
spec:
  template:                          # The base Deployment template
    metadata:
      labels:
        app: web-app
    spec:
      replicas: 3
      selector:
        matchLabels:
          app: web-app
      template:
        metadata:
          labels:
            app: web-app
        spec:
          containers:
            - name: nginx
              image: nginx:1.25
              ports:
                - containerPort: 80
              resources:
                requests:
                  cpu: 100m
                  memory: 128Mi
                limits:
                  cpu: 250m
                  memory: 256Mi
  placement:                         # WHERE to deploy
    clusters:
      - name: cluster-a
      - name: cluster-b
      - name: cluster-c
  overrides:                         # Per-cluster customization
    - clusterName: cluster-a
      clusterOverrides:
        - path: "/spec/replicas"
          value: 5                    # More replicas in US (higher traffic)
    - clusterName: cluster-c
      clusterOverrides:
        - path: "/spec/replicas"
          value: 2                    # Fewer replicas in AP (lower traffic)
```

Let me break this down:

| Section | What It Does |
|---------|-------------|
| `template` | The base resource definition — same as a regular Deployment |
| `placement` | Which clusters should receive this resource |
| `overrides` | Per-cluster modifications (like different replica counts per region) |

```bash
# Apply the federated resources
kubectl apply -f federated-namespace.yaml
kubectl apply -f federated-deployment.yaml

# Verify — switch to each cluster and check
kubectl --context=cluster-a get deployment web-app -n my-app
# NAME      READY   UP-TO-DATE   AVAILABLE   AGE
# web-app   5/5     5            5           30s    ← 5 replicas (overridden)

kubectl --context=cluster-b get deployment web-app -n my-app
# NAME      READY   UP-TO-DATE   AVAILABLE   AGE
# web-app   3/3     3            3           30s    ← 3 replicas (default)

kubectl --context=cluster-c get deployment web-app -n my-app
# NAME      READY   UP-TO-DATE   AVAILABLE   AGE
# web-app   2/2     2            2           30s    ← 2 replicas (overridden)
```

#### Federated Service

```yaml
# federated-service.yaml
apiVersion: types.kubefed.io/v1beta1
kind: FederatedService
metadata:
  name: web-app
  namespace: my-app
spec:
  template:
    spec:
      selector:
        app: web-app
      ports:
        - port: 80
          targetPort: 80
      type: ClusterIP
  placement:
    clusters:
      - name: cluster-a
      - name: cluster-b
      - name: cluster-c
```

### Federation Pros and Cons

| Pros | Cons |
|------|------|
| Distribute workloads across clusters easily | Complex to set up and maintain |
| Per-cluster overrides for customization | KubeFed project is in maintenance mode (not very active) |
| Uses standard Kubernetes patterns | Limited ecosystem support |
| Automatic sync of resources | Debugging federated resources is harder |
| Supports many resource types | No built-in cross-cluster networking |

### Current Status of Federation

**Honest assessment:** KubeFed v2 is in **maintenance mode** and development has slowed significantly. The community has largely moved toward other approaches like **GitOps-based multi-cluster** (ArgoCD, Flux) and **Cluster API**. However, understanding the concepts is valuable.

## 2️⃣ Rancher — Full Multi-Cluster Management Platform

### What is Rancher?

Rancher is a **complete multi-cluster management platform** with a beautiful web UI and powerful CLI. It's like having a command center for all your Kubernetes clusters.

```
┌─────────────────────────────────────────────────────┐
│                    Rancher Server                     │
│                                                       │
│  ┌─────────────┐  ┌──────────┐  ┌─────────────────┐ │
│  │  Web UI      │  │  API     │  │  Catalog/Apps   │ │
│  │  Dashboard   │  │  Server  │  │  (Helm Charts)  │ │
│  └─────────────┘  └──────────┘  └─────────────────┘ │
│                                                       │
│  ┌──────────────────────────────────────────────┐    │
│  │  Cluster Management                          │    │
│  │  • Create new clusters                       │    │
│  │  • Import existing clusters                  │    │
│  │  • Upgrade clusters                          │    │
│  │  • Monitor all clusters                      │    │
│  │  • Manage RBAC across clusters               │    │
│  │  • Deploy apps to any cluster                │    │
│  └──────────────────────────────────────────────┘    │
└───────────────┬──────────────┬──────────────┬────────┘
                │              │              │
       ┌────────▼──────┐ ┌────▼──────┐ ┌────▼──────┐
       │  EKS Cluster  │ │ AKS       │ │ On-Prem   │
       │  (AWS)        │ │ Cluster   │ │ Cluster   │
       │               │ │ (Azure)   │ │ (Bare     │
       │  5 nodes      │ │ 3 nodes   │ │  Metal)   │
       └───────────────┘ └───────────┘ └───────────┘
```

### Key Features of Rancher

| Feature | Description |
|---------|-------------|
| **Cluster Provisioning** | Create K8s clusters on any infrastructure (AWS, Azure, GCP, vSphere, bare metal) |
| **Cluster Import** | Import and manage existing clusters (EKS, AKS, GKE, or any K8s cluster) |
| **Centralized RBAC** | Manage users and permissions across all clusters from one place |
| **App Catalog** | Deploy applications from Helm charts with a few clicks |
| **Monitoring** | Built-in Prometheus + Grafana integration for all clusters |
| **Logging** | Centralized logging setup across clusters |
| **Alerting** | Unified alerting across all managed clusters |
| **CIS Benchmarks** | Security scanning for all clusters |
| **Istio Integration** | Service mesh management built-in |
| **Fleet** | GitOps-at-scale for deploying to hundreds of clusters |

### Installing Rancher

#### Method 1: Docker (Quick Start / Testing Only)

```bash
# Quick start with Docker (NOT for production!)
docker run -d \
  --name rancher \
  --restart=unless-stopped \
  -p 80:80 \
  -p 443:443 \
  --privileged \
  rancher/rancher:latest

# Access at https://localhost
# Default password is generated — check logs:
docker logs rancher 2>&1 | grep "Bootstrap Password"
```

#### Method 2: Helm on Kubernetes (Production)

```bash
# Step 1: Install cert-manager (Rancher needs it for TLS)
kubectl apply -f https://github.com/cert-manager/cert-manager/releases/download/v1.14.0/cert-manager.yaml

# Wait for cert-manager to be ready
kubectl wait --for=condition=Available deployment --all -n cert-manager --timeout=300s

# Step 2: Add Rancher Helm repo
helm repo add rancher-latest https://releases.rancher.com/server-charts/latest
helm repo update

# Step 3: Create namespace
kubectl create namespace cattle-system

# Step 4: Install Rancher
helm install rancher rancher-latest/rancher \
  --namespace cattle-system \
  --set hostname=rancher.example.com \
  --set replicas=3 \
  --set bootstrapPassword=admin

# Step 5: Wait for Rancher to be ready
kubectl -n cattle-system rollout status deployment/rancher

# Step 6: Get the URL
echo "Access Rancher at: https://rancher.example.com"
```

### Managing Clusters with Rancher

#### Creating a New Cluster (RKE2 on Cloud VMs)

Through the UI, it's point-and-click. Through the API/CLI, you use the Rancher API:

```bash
# Using Rancher CLI
# First, login
rancher login https://rancher.example.com --token token-xxxxx

# Create a new cluster (RKE2 on AWS)
rancher cluster create my-new-cluster \
  --provider amazonec2 \
  --kubernetes-version v1.29.0+rke2r1
```

#### Importing an Existing Cluster

This is the most common operation — you have an existing EKS/AKS/GKE cluster and want Rancher to manage it.

```bash
# In the Rancher UI:
# 1. Click "Import Existing"
# 2. Give it a name
# 3. Rancher gives you a kubectl command to run on the target cluster:

# Run this on the cluster you want to import:
kubectl apply -f https://rancher.example.com/v3/import/xxxxx.yaml

# That's it! The cluster will appear in Rancher within a minute.
```

What happens behind the scenes:
1. The command installs a **Rancher agent** (called `cattle-cluster-agent`) in the target cluster
2. The agent connects back to the Rancher server
3. Rancher now has visibility and control over the cluster

#### Deploying Apps Across Clusters

Rancher includes **Fleet** — a GitOps engine for multi-cluster deployments:

```yaml
# fleet.yaml — Deploy to multiple clusters based on labels
apiVersion: fleet.cattle.io/v1alpha1
kind: GitRepo
metadata:
  name: web-app
  namespace: fleet-default
spec:
  repo: https://github.com/your-org/web-app-manifests
  branch: main
  paths:
    - manifests/
  targets:
    - name: production
      clusterSelector:
        matchLabels:
          environment: production
    - name: staging
      clusterSelector:
        matchLabels:
          environment: staging
```

This tells Fleet:
- Watch the Git repo for changes
- Deploy to all clusters labeled `environment: production`
- Deploy to all clusters labeled `environment: staging`
- Whenever you push to the `main` branch, Fleet automatically updates all matching clusters

#### Managing RBAC Across Clusters

One of Rancher's best features — centralized user management:

```
Rancher Global Roles:
├── Admin (full access to everything)
├── Standard User (can create clusters, manage own projects)
└── User-Base (read-only, can log in)

Cluster Roles (per cluster):
├── Cluster Owner (full access to one cluster)
├── Cluster Member (can create projects/namespaces)
└── Read-Only (view everything, change nothing)

Project Roles (per namespace group):
├── Project Owner
├── Project Member
└── Read-Only
```

You can integrate with external identity providers:

```
Rancher → Active Directory / LDAP
Rancher → GitHub
Rancher → Google OAuth
Rancher → Azure AD
Rancher → Okta / SAML
Rancher → OpenLDAP
```

### Rancher Pros and Cons

| Pros | Cons |
|------|------|
| Beautiful, intuitive web UI | Can be complex to install in production (HA setup) |
| Supports ANY Kubernetes cluster (any cloud, on-prem) | Resource overhead of Rancher server + agents |
| Built-in monitoring, logging, alerting | Learning curve for the full platform |
| Fleet for GitOps multi-cluster deployment | Rancher-specific concepts (Projects) on top of K8s |
| Strong RBAC and multi-tenancy | Vendor dependency (SUSE/Rancher) |
| Active development and large community | Can feel like "too much" for small teams |
| Free and open source (Rancher core) | |

### When to Use Rancher

| Scenario | Good Fit? |
|----------|----------|
| Managing 5-100+ clusters | Yes — Rancher shines here |
| Mix of cloud + on-premise clusters | Yes — cloud-agnostic |
| Need a unified dashboard for all clusters | Yes — best-in-class UI |
| Small team, 1-2 clusters | Maybe overkill — just use kubectl |
| Already deep in one cloud (only EKS or only AKS) | Consider cloud-native tools first |
| Need GitOps for multi-cluster | Yes — Fleet is built-in |

## 3️⃣ Cluster API (CAPI) — Declarative Cluster Lifecycle Management

### What is Cluster API?

Cluster API is a Kubernetes sub-project that lets you **create, configure, and manage Kubernetes clusters using Kubernetes itself**. Yes — you use Kubernetes to manage Kubernetes!

The key idea: **Treat clusters like cattle, not pets.**

```
Traditional approach:
  "Let me SSH into the server and manually install Kubernetes..."
  "Let me click through the AWS console to create an EKS cluster..."

Cluster API approach:
  "Let me write a YAML file that defines my cluster, and apply it."
  
  kubectl apply -f my-cluster.yaml    # ← A cluster is created! 🎉
  kubectl delete -f my-cluster.yaml   # ← The cluster is destroyed!
```

### How Cluster API Works

```
┌────────────────────────────────────┐
│       Management Cluster           │
│  (Where CAPI controllers run)      │
│                                    │
│  ┌────────────────────────────┐    │
│  │  CAPI Core Controllers    │    │
│  │  • Cluster Controller     │    │
│  │  • Machine Controller     │    │
│  │  • MachineSet Controller  │    │
│  └────────────────────────────┘    │
│                                    │
│  ┌────────────────────────────┐    │
│  │  Infrastructure Provider  │    │ ← Knows how to create VMs/infra
│  │  (e.g., AWS, Azure, GCP,  │    │    on a specific cloud
│  │   vSphere, Docker, etc.)  │    │
│  └────────────────────────────┘    │
│                                    │
│  ┌────────────────────────────┐    │
│  │  Bootstrap Provider       │    │ ← Knows how to install K8s
│  │  (e.g., kubeadm, RKE2,   │    │    on the VMs
│  │   Talos, MicroK8s)       │    │
│  └────────────────────────────┘    │
│                                    │
│  ┌────────────────────────────┐    │
│  │  Control Plane Provider   │    │ ← Manages the control plane
│  │  (e.g., KubeadmControlPlane,│  │    nodes specifically
│  │   EKS, AKS managed CP)   │    │
│  └────────────────────────────┘    │
└─────────────┬──────────────────────┘
              │
              │  Creates & Manages
              │
    ┌─────────▼──────────┐  ┌──────────────────┐
    │  Workload Cluster 1│  │ Workload Cluster 2│
    │  (Production)      │  │ (Staging)         │
    │  5 nodes           │  │ 3 nodes           │
    └────────────────────┘  └──────────────────┘
```

### Key Concepts

| Concept | What It Is | Analogy |
|---------|-----------|---------|
| **Management Cluster** | The K8s cluster that runs CAPI controllers | The factory that builds other factories |
| **Workload Cluster** | A cluster created and managed by CAPI | The factory that runs your actual apps |
| **Infrastructure Provider** | Plugin that knows how to create VMs on a specific cloud | The builder that knows how to work with specific materials |
| **Bootstrap Provider** | Plugin that knows how to install Kubernetes on VMs | The installer that sets up the machinery |
| **Control Plane Provider** | Plugin that manages control plane nodes | The architect of the factory |

### CAPI Resources (CRDs)

CAPI introduces its own Custom Resource Definitions:

```
Cluster                    → Defines the overall cluster
├── KubeadmControlPlane    → Manages control plane nodes
│   └── Machine            → Individual control plane VM
├── MachineDeployment      → Like a Deployment but for VMs (worker nodes)
│   └── MachineSet         → Like a ReplicaSet for VMs
│       └── Machine        → Individual worker VM
└── InfrastructureCluster  → Cloud-specific infrastructure (VPC, subnets, etc.)
```

### Setting Up Cluster API

#### Step 1: Install clusterctl CLI

```bash
# Download clusterctl
curl -L https://github.com/kubernetes-sigs/cluster-api/releases/download/v1.7.0/clusterctl-linux-amd64 -o clusterctl
chmod +x clusterctl
sudo mv clusterctl /usr/local/bin/
clusterctl version
```

#### Step 2: Initialize the Management Cluster

You need an existing Kubernetes cluster to use as the management cluster. A simple Kind cluster works:

```bash
# Create a Kind cluster as the management cluster
kind create cluster --name capi-management

# Initialize CAPI with your cloud provider
# For AWS:
export AWS_REGION=us-east-1
export AWS_ACCESS_KEY_ID=<your-access-key>
export AWS_SECRET_ACCESS_KEY=<your-secret-key>

clusterctl init --infrastructure aws

# For Azure:
# export AZURE_SUBSCRIPTION_ID=<sub-id>
# export AZURE_TENANT_ID=<tenant-id>
# export AZURE_CLIENT_ID=<client-id>
# export AZURE_CLIENT_SECRET=<client-secret>
# clusterctl init --infrastructure azure

# For Docker (local testing — creates clusters using Docker containers):
clusterctl init --infrastructure docker

# For vSphere:
# clusterctl init --infrastructure vsphere

# Verify CAPI controllers are running
kubectl get pods -A | grep cap
# NAMESPACE                           NAME                                           READY
# capi-system                         capi-controller-manager-xxx                     1/1
# capi-kubeadm-bootstrap-system       capi-kubeadm-bootstrap-controller-manager-xxx   1/1
# capi-kubeadm-control-plane-system   capi-kubeadm-control-plane-controller-manager-xxx 1/1
# capa-system                         capa-controller-manager-xxx                     1/1  ← AWS provider
```

#### Step 3: Create a Workload Cluster

The easiest way — use `clusterctl generate`:

```bash
# Generate cluster manifests
clusterctl generate cluster my-production-cluster \
  --kubernetes-version v1.29.0 \
  --control-plane-machine-count 3 \
  --worker-machine-count 5 \
  > my-production-cluster.yaml
```

This generates a YAML file with all the resources. Let's look at what's inside:

```yaml
# my-production-cluster.yaml (simplified)

# 1. The Cluster resource — defines the overall cluster
apiVersion: cluster.x-k8s.io/v1beta1
kind: Cluster
metadata:
  name: my-production-cluster
  namespace: default
spec:
  clusterNetwork:
    pods:
      cidrBlocks: ["192.168.0.0/16"]    # Pod network CIDR
    services:
      cidrBlocks: ["10.128.0.0/12"]     # Service network CIDR
  controlPlaneRef:
    apiVersion: controlplane.cluster.x-k8s.io/v1beta1
    kind: KubeadmControlPlane
    name: my-production-cluster-control-plane
  infrastructureRef:
    apiVersion: infrastructure.cluster.x-k8s.io/v1beta2
    kind: AWSCluster                     # Cloud-specific
    name: my-production-cluster

---
# 2. Cloud-specific infrastructure (AWS example)
apiVersion: infrastructure.cluster.x-k8s.io/v1beta2
kind: AWSCluster
metadata:
  name: my-production-cluster
spec:
  region: us-east-1
  sshKeyName: my-ssh-key
  network:
    vpc:
      cidrBlock: 10.0.0.0/16            # VPC CIDR
    subnets:
      - availabilityZone: us-east-1a
        cidrBlock: 10.0.1.0/24
        isPublic: true
      - availabilityZone: us-east-1b
        cidrBlock: 10.0.2.0/24
        isPublic: true

---
# 3. Control Plane definition
apiVersion: controlplane.cluster.x-k8s.io/v1beta1
kind: KubeadmControlPlane
metadata:
  name: my-production-cluster-control-plane
spec:
  replicas: 3                            # 3 control plane nodes (HA!)
  version: v1.29.0
  machineTemplate:
    infrastructureRef:
      apiVersion: infrastructure.cluster.x-k8s.io/v1beta2
      kind: AWSMachineTemplate
      name: my-production-cluster-control-plane
  kubeadmConfigSpec:
    initConfiguration:
      nodeRegistration:
        name: "{{ ds.meta_data.local_hostname }}"
    clusterConfiguration:
      apiServer:
        extraArgs:
          cloud-provider: external
      controllerManager:
        extraArgs:
          cloud-provider: external

---
# 4. Machine Template for Control Plane nodes
apiVersion: infrastructure.cluster.x-k8s.io/v1beta2
kind: AWSMachineTemplate
metadata:
  name: my-production-cluster-control-plane
spec:
  template:
    spec:
      instanceType: t3.large             # VM size
      iamInstanceProfile: control-plane.cluster-api-provider-aws.sigs.k8s.io
      sshKeyName: my-ssh-key
      rootVolume:
        size: 50                          # 50 GB disk

---
# 5. Worker nodes (MachineDeployment — like a Deployment for VMs!)
apiVersion: cluster.x-k8s.io/v1beta1
kind: MachineDeployment
metadata:
  name: my-production-cluster-workers
spec:
  clusterName: my-production-cluster
  replicas: 5                             # 5 worker nodes
  selector:
    matchLabels: {}
  template:
    spec:
      clusterName: my-production-cluster
      version: v1.29.0
      bootstrap:
        configRef:
          apiVersion: bootstrap.cluster.x-k8s.io/v1beta1
          kind: KubeadmConfigTemplate
          name: my-production-cluster-workers
      infrastructureRef:
        apiVersion: infrastructure.cluster.x-k8s.io/v1beta2
        kind: AWSMachineTemplate
        name: my-production-cluster-workers

---
# 6. Machine Template for Worker nodes
apiVersion: infrastructure.cluster.x-k8s.io/v1beta2
kind: AWSMachineTemplate
metadata:
  name: my-production-cluster-workers
spec:
  template:
    spec:
      instanceType: t3.xlarge            # Bigger VMs for workers
      iamInstanceProfile: nodes.cluster-api-provider-aws.sigs.k8s.io
      sshKeyName: my-ssh-key
      rootVolume:
        size: 100                         # 100 GB disk
```

### Day-2 Operations with Cluster API

#### Scaling Workers

```bash
# Scale from 5 to 10 workers — just like scaling a Deployment!
kubectl scale machinedeployment my-production-cluster-workers --replicas=10

# Or edit the YAML and re-apply
# Change replicas: 5 → replicas: 10
kubectl apply -f my-production-cluster.yaml

# Watch new machines being created
kubectl get machines --watch
```

#### Upgrading Kubernetes Version

```bash
# Upgrade control plane (just change the version!)
kubectl patch KubeadmControlPlane my-production-cluster-control-plane \
  --type merge \
  -p '{"spec": {"version": "v1.30.0"}}'

# CAPI will perform a rolling upgrade:
# 1. Create a new control plane node with v1.30.0
# 2. Wait for it to be healthy
# 3. Remove an old v1.29.0 node
# 4. Repeat until all nodes are upgraded

# Upgrade workers (update the MachineDeployment)
kubectl patch MachineDeployment my-production-cluster-workers \
  --type merge \
  -p '{"spec": {"template": {"spec": {"version": "v1.30.0"}}}}'

# Same rolling upgrade process for workers
```

#### Deleting a Cluster

```bash
# Delete the entire cluster — CAPI will clean up all infrastructure!
kubectl delete cluster my-production-cluster

# This will:
# 1. Delete all VMs (Machines)
# 2. Delete the VPC, subnets, security groups
# 3. Delete the load balancers
# 4. Clean up everything!
```

### Cluster API Pros and Cons

| Pros | Cons |
|------|------|
| Fully declarative — clusters as YAML | Steep learning curve |
| GitOps-friendly — store cluster definitions in Git | Complex initial setup of management cluster |
| Standardized across any cloud | Need to understand many CRDs |
| Rolling upgrades built-in | Not all providers are equally mature |
| Scale clusters like scaling Deployments | Overkill for a few simple clusters |
| Official Kubernetes project (strong community) | Requires understanding of infrastructure concepts |

## 🛠️ Practical: Managing Multiple Clusters with kubectl

Before reaching for any tool, you should master **kubectl contexts** — the built-in way to switch between clusters.

### Working with Multiple kubeconfig Files

```bash
# Method 1: Merge kubeconfigs into one file
export KUBECONFIG=~/.kube/config:~/.kube/cluster-a.config:~/.kube/cluster-b.config
kubectl config view --flatten > ~/.kube/merged-config
export KUBECONFIG=~/.kube/merged-config

# Method 2: Set KUBECONFIG to multiple files
export KUBECONFIG=~/.kube/config-cluster-a:~/.kube/config-cluster-b:~/.kube/config-cluster-c

# List all contexts
kubectl config get-contexts
# CURRENT   NAME          CLUSTER       AUTHINFO     NAMESPACE
# *         cluster-a     cluster-a     admin-a      default
#           cluster-b     cluster-b     admin-b      default
#           cluster-c     cluster-c     admin-c      default

# Switch context
kubectl config use-context cluster-b
# Switched to context "cluster-b".

# Or use --context flag without switching
kubectl --context=cluster-a get pods
kubectl --context=cluster-b get pods
kubectl --context=cluster-c get pods
```

### kubectx and kubens — Essential Tools

```bash
# Install kubectx and kubens
# macOS:
brew install kubectx

# Linux:
sudo apt install kubectx
# Or download from: https://github.com/ahmetb/kubectx

# Switch contexts easily
kubectx                    # List all contexts
kubectx cluster-a          # Switch to cluster-a
kubectx -                  # Switch to previous context

# Switch namespaces easily
kubens                     # List all namespaces
kubens production          # Switch to production namespace
kubens -                   # Switch to previous namespace
```

### Running Commands Across Multiple Clusters

```bash
# Simple bash loop
for ctx in cluster-a cluster-b cluster-c; do
  echo "=== $ctx ==="
  kubectl --context=$ctx get nodes
  echo ""
done

# Or use a tool like kubectl-foreach
# Install: https://github.com/ahmetb/kubectl-foreach
kubectl foreach get pods -A
```

## 🌐 Cross-Cluster Networking

One of the trickiest parts of multi-cluster is making Pods in different clusters talk to each other.

### Options for Cross-Cluster Networking

| Tool | How It Works |
|------|-------------|
| **Submariner** | Creates encrypted tunnels between clusters |
| **Cilium Cluster Mesh** | Extends Cilium's eBPF networking across clusters |
| **Istio Multi-Cluster** | Service mesh spanning multiple clusters |
| **Skupper** | Virtual Application Network — Layer 7 multi-cluster |
| **Linkerd Multi-Cluster** | Service mesh multi-cluster support |

### Example: Submariner (Simplified)

```bash
# Install subctl CLI
curl -Ls https://get.submariner.io | bash
export PATH=$PATH:~/.local/bin

# Deploy the Submariner broker on one cluster
subctl deploy-broker --context cluster-a

# Join other clusters
subctl join --context cluster-b broker-info.subm --clusterid cluster-b
subctl join --context cluster-c broker-info.subm --clusterid cluster-c

# Now Services can be discovered across clusters!
# Export a Service from cluster-a:
subctl export service --context cluster-a my-service -n my-namespace

# Access it from cluster-b:
# curl my-service.my-namespace.svc.clusterset.local
```
