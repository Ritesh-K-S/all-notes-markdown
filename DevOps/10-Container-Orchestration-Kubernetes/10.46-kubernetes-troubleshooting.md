## 🤔 Why Do You Need Troubleshooting Skills?

Let's be real — in Kubernetes, **things will break**. No matter how good your YAML is, at some point you'll face:

- A Pod stuck in `Pending` forever
- A Pod crashing with `CrashLoopBackOff`
- A Service that doesn't connect to anything
- An application that starts but doesn't respond
- A node that goes `NotReady`

**If you don't know how to troubleshoot, you'll be stuck.** This is the most practical and important skill you'll ever learn in Kubernetes.

> 💡 **Fun Fact:** In real-world DevOps interviews, troubleshooting scenarios are the #1 most asked Kubernetes questions. Interviewers love to say: *"Your Pod is in CrashLoopBackOff — walk me through how you'd fix it."*

---

## 🧭 The Troubleshooting Mindset — A General Framework

Before diving into specific commands, let's understand the **general approach** to troubleshooting anything in Kubernetes. Think of it as a flowchart:

```
Something is broken
       │
       ▼
What resource is affected?
  (Pod? Service? Node? Ingress?)
       │
       ▼
Check the STATUS of that resource
  (kubectl get <resource>)
       │
       ▼
Describe the resource for DETAILS
  (kubectl describe <resource>)
       │
       ▼
Check EVENTS for error messages
  (kubectl get events)
       │
       ▼
Check LOGS if it's a container issue
  (kubectl logs <pod>)
       │
       ▼
If needed, get INSIDE the container
  (kubectl exec or kubectl debug)
       │
       ▼
Fix the issue & apply changes
```

Let's learn each step in detail.

## 🔍 Step 1: Check Pod Status — `kubectl get pods`

This is always your **first command** when something is wrong. It tells you the current state of all your Pods.

```bash
# Get all pods in the default namespace
kubectl get pods

# Get all pods in a specific namespace
kubectl get pods -n my-namespace

# Get all pods in ALL namespaces
kubectl get pods -A

# Get pods with more details (node, IP, etc.)
kubectl get pods -o wide

# Watch pods in real-time (auto-refresh)
kubectl get pods -w
```

### Understanding Pod Statuses

When you run `kubectl get pods`, you'll see a `STATUS` column. Here's what each status means:

| Status | What It Means | Is It Normal? |
|--------|--------------|---------------|
| `Pending` | Pod is accepted but not yet running. Waiting for scheduling or image pull. | ⚠️ Temporary is OK. Stuck = Problem. |
| `Running` | Pod is running and at least one container is alive. | ✅ Normal |
| `Succeeded` | All containers in the Pod exited successfully (exit code 0). Common for Jobs. | ✅ Normal for Jobs |
| `Failed` | All containers terminated, and at least one exited with an error. | ❌ Problem |
| `CrashLoopBackOff` | Container keeps crashing and Kubernetes keeps restarting it (with increasing delay). | ❌ Major Problem |
| `ImagePullBackOff` | Kubernetes can't pull the container image. | ❌ Problem |
| `ErrImagePull` | Same as above — image pull failed. | ❌ Problem |
| `ContainerCreating` | The container is being set up (pulling image, mounting volumes). | ⚠️ Temporary is OK |
| `Terminating` | Pod is being shut down. | ⚠️ Temporary is OK |
| `Unknown` | The state of the Pod can't be determined (usually a node problem). | ❌ Problem |
| `Init:0/1` | Init container is running (not complete yet). | ⚠️ Temporary is OK |
| `Init:Error` | Init container failed. | ❌ Problem |
| `Init:CrashLoopBackOff` | Init container keeps crashing. | ❌ Problem |
| `OOMKilled` | Container was killed because it ran out of memory. | ❌ Problem |
| `Evicted` | Pod was kicked off the node due to resource pressure (disk, memory). | ❌ Problem |

### Example Output

```
NAME                        READY   STATUS             RESTARTS      AGE
web-app-6d5f8b4c77-abc12   1/1     Running            0             2d
api-server-7f9b4d-def34    0/1     CrashLoopBackOff   15 (3m ago)   1h
worker-job-xyz89           0/1     Pending            0             30m
frontend-5c8d9e-ghi56      0/1     ImagePullBackOff   0             10m
```

Just from this output, you can already see:
- `web-app` → ✅ Healthy
- `api-server` → ❌ Keeps crashing (15 restarts!)
- `worker-job` → ⚠️ Stuck in Pending (scheduling issue?)
- `frontend` → ❌ Can't pull the image

## 🔎 Step 2: Describe the Resource — `kubectl describe`

`kubectl describe` is your **best friend** for troubleshooting. It gives you a **detailed report** about any Kubernetes resource, including events at the bottom.

```bash
# Describe a specific pod
kubectl describe pod api-server-7f9b4d-def34

# Describe a pod in a specific namespace
kubectl describe pod my-pod -n my-namespace

# Describe a node
kubectl describe node worker-node-1

# Describe a service
kubectl describe svc my-service

# Describe a deployment
kubectl describe deployment my-app
```

### What `kubectl describe pod` Shows You

The output has several important sections:

```
Name:             api-server-7f9b4d-def34
Namespace:        default
Node:             worker-node-1/10.0.1.5
Status:           Running
IP:               172.17.0.4
Containers:
  api:
    Image:          mycompany/api-server:v2.1

    Ready:          False
    Restart Count:  15
    Limits:
      cpu:     500m
      memory:  256Mi
    Requests:
      cpu:     100m
      memory:  128Mi
    Liveness:  http-get http://:8080/health
    Readiness: http-get http://:8080/ready
Conditions:
  Type              Status
  Initialized       True
  Ready             False
  ContainersReady   False
  PodScheduled      True
Events:
  Type     Reason     Age                From               Message
  ----     ------     ----               ----               -------
  Normal   Scheduled  60m                default-scheduler  Successfully assigned default/api-server-7f9b4d-def34 to worker-node-1
  Normal   Pulled     5m (x16 over 60m)  kubelet            Container image "mycompany/api-server:v2.1" already present on machine
  Normal   Created    5m (x16 over 60m)  kubelet            Created container api

  Warning  BackOff    30s (x90 over 55m) kubelet            Back-off restarting failed container api in pod api-server-7f9b4d-def34
```

### How to Read This Output

| Section | What to Look For |
|---------|-----------------|
| **Node** | Which node is the Pod on? Is the node healthy? |
| **Status** | Is it Running, Pending, Failed? |
| **Containers → State** | Current state of each container (Running, Waiting, Terminated) |
| **Containers → Reason** | WHY is it in that state? (CrashLoopBackOff, OOMKilled, Error) |
| **Exit Code** | 0 = success, 1 = general error, 137 = OOMKilled/SIGKILL, 143 = SIGTERM |
| **Restart Count** | How many times has it restarted? High number = something keeps breaking |
| **Limits / Requests** | Are resources too low? Could be OOM issues |
| **Liveness / Readiness** | Are health check probes configured? Are they failing? |
| **Conditions** | Quick overview of Pod health |
| **Events** (most important!) | Timeline of what happened — look for `Warning` events |

> 💡 **Pro Tip:** Always scroll to the **Events** section at the bottom. That's where Kubernetes tells you exactly what went wrong.

## 📋 Step 3: Check Events — `kubectl get events`

Events are the **log book** of your cluster. Every time something important happens (scheduling, pulling images, crashing, etc.), Kubernetes records an event.

```bash
# Get all events in the default namespace
kubectl get events

# Get events sorted by time (most recent last)
kubectl get events --sort-by='.lastTimestamp'

# Get events for a specific namespace
kubectl get events -n kube-system

# Get events in all namespaces
kubectl get events -A

# Get events for a specific pod (using field selector)
kubectl get events --field-selector involvedObject.name=my-pod

# Get only Warning events (skip Normal ones)
kubectl get events --field-selector type=Warning

# Watch events in real-time
kubectl get events -w
```

### Example Events Output

```
LAST SEEN   TYPE      REASON              OBJECT                              MESSAGE
2m          Normal    Scheduled           pod/my-app-abc12                    Successfully assigned to worker-1
2m          Normal    Pulling             pod/my-app-abc12                    Pulling image "nginx:latest"
1m          Normal    Pulled              pod/my-app-abc12                    Successfully pulled image
1m          Normal    Created             pod/my-app-abc12                    Created container nginx
1m          Normal    Started             pod/my-app-abc12                    Started container nginx
30s         Warning   BackOff             pod/api-server-def34                Back-off restarting failed container
15s         Warning   FailedScheduling    pod/worker-xyz89                    0/3 nodes are available: 3 Insufficient cpu
10s         Warning   Failed              pod/frontend-ghi56                  Failed to pull image "myapp:v99": not found
```

> 💡 **Key Insight:** Look for **Warning** events. `Normal` events are usually fine. `Warning` events point to problems.

## 📝 Step 4: Check Logs — `kubectl logs`

If a container is crashing or misbehaving, the **application logs** will tell you why. This is like reading the console output of your application.

```bash
# Get logs from a pod (single container)
kubectl logs my-pod

# Get logs from a specific container in a multi-container pod
kubectl logs my-pod -c my-container

# Follow logs in real-time (like tail -f)
kubectl logs my-pod -f

# Get last 100 lines only
kubectl logs my-pod --tail=100

# Get logs from the last 1 hour
kubectl logs my-pod --since=1h

# Get logs from the last 30 minutes
kubectl logs my-pod --since=30m

# Get logs from a PREVIOUS (crashed) container
# THIS IS SUPER IMPORTANT for CrashLoopBackOff!
kubectl logs my-pod --previous
# or
kubectl logs my-pod -p

# Get logs from all pods matching a label
kubectl logs -l app=my-app

# Get logs with timestamps
kubectl logs my-pod --timestamps=true
```

### The `--previous` Flag — Your CrashLoopBackOff Lifesaver

When a container keeps crashing and restarting (`CrashLoopBackOff`), the **current** container has just started and has no useful logs. You need to see the logs from the **previous crashed container**:

```bash
# This shows logs from the CURRENT container (might be empty or just startup logs)
kubectl logs crashing-pod

# This shows logs from the PREVIOUS container that CRASHED 🎯
kubectl logs crashing-pod --previous
```

### Example: Reading Crash Logs

```bash
$ kubectl logs api-server-7f9b4d-def34 --previous

2024-03-15 10:45:01 INFO  Starting API Server v2.1...
2024-03-15 10:45:02 INFO  Connecting to database at postgres://db-host:5432/mydb...
2024-03-15 10:45:07 ERROR Connection refused: cannot connect to postgres://db-host:5432/mydb
2024-03-15 10:45:07 FATAL Database connection failed. Exiting.
```

**There's your answer!** The app can't connect to the database. Maybe:
- The database Pod isn't running
- The Service name `db-host` is wrong
- The database is in a different namespace
- Network policy is blocking the connection

## 🐛 Step 5: Get Inside the Container — `kubectl exec`

Sometimes you need to **get inside a running container** to investigate — check files, test network connections, check environment variables, etc.

```bash
# Open an interactive shell inside a pod
kubectl exec -it my-pod -- /bin/bash

# If bash is not available (minimal images), try sh
kubectl exec -it my-pod -- /bin/sh

# Run a specific command without entering the shell
kubectl exec my-pod -- cat /etc/config/app.conf

# Check environment variables
kubectl exec my-pod -- env

# Test DNS resolution from inside the pod
kubectl exec my-pod -- nslookup my-service

# Test connectivity to another service
kubectl exec my-pod -- curl http://my-service:8080/health

# Check if a file exists
kubectl exec my-pod -- ls -la /app/data/

# Check network connectivity
kubectl exec my-pod -- wget -qO- http://other-service:3000

# Exec into a specific container in a multi-container pod
kubectl exec -it my-pod -c sidecar-container -- /bin/sh
```

### Common Things to Check Inside a Container

| What to Check | Command | Why |
|---------------|---------|-----|
| Environment variables | `env` | Are DB_HOST, API_KEY, etc. set correctly? |
| DNS resolution | `nslookup my-service` | Can the pod resolve service names? |
| Network connectivity | `curl http://other-service:8080` | Can the pod reach other services? |
| File contents | `cat /etc/config/app.conf` | Is the ConfigMap/Secret mounted correctly? |
| Disk space | `df -h` | Is the disk full? |
| Running processes | `ps aux` | Is the application actually running? |
| Permissions | `ls -la /app/` | Are file permissions correct? |
| Hosts file | `cat /etc/hosts` | Any custom host entries? |
| DNS config | `cat /etc/resolv.conf` | Is DNS configured correctly? |

## 🛠️ Step 6: Ephemeral Debug Containers — `kubectl debug`

Sometimes the container is so minimal (like **distroless** or **scratch** images) that there's **no shell, no curl, no nothing** inside it. You can't `exec` into it because there are no debugging tools.

That's where **ephemeral debug containers** come in. You inject a temporary debugging container into a running Pod.

```bash
# Attach a debug container to a running pod
kubectl debug -it my-pod --image=busybox --target=my-container

# Use a more full-featured debug image
kubectl debug -it my-pod --image=nicolaka/netshoot --target=my-container

# Create a debug copy of the pod (doesn't modify the original)
kubectl debug my-pod -it --copy-to=my-pod-debug --image=ubuntu

# Debug a node directly (creates a privileged pod on the node)
kubectl debug node/worker-node-1 -it --image=ubuntu
```

### What is `nicolaka/netshoot`?

It's a popular Docker image packed with **every networking tool you'd ever need**:
- `curl`, `wget`
- `dig`, `nslookup`
- `ping`, `traceroute`
- `tcpdump`, `netstat`, `ss`
- `iptables`
- And many more

```bash
# Example: Debug network issues from a distroless pod
kubectl debug -it my-pod --image=nicolaka/netshoot --target=my-container

# Now inside the debug container:
ping other-service
curl http://api-service:8080/health
nslookup my-database
tcpdump -i eth0 port 5432
```

> 💡 **Note:** Ephemeral containers require Kubernetes 1.25+ (the feature went stable in 1.25).

## 🔴 Most Common Problems & How to Fix Them

Let's go through the most common Kubernetes issues one by one with a clear diagnosis and fix.

---

### Problem 1: `CrashLoopBackOff`

**What it means:** Your container starts, crashes, and Kubernetes restarts it. But it crashes again. And again. Kubernetes keeps trying with an **exponential backoff** delay (10s, 20s, 40s, 80s... up to 5 minutes).

**How to diagnose:**

```bash
# Step 1: Check the pod status
kubectl get pods
# OUTPUT: my-pod   0/1   CrashLoopBackOff   8   15m

# Step 2: Check the previous container's logs (most important!)
kubectl logs my-pod --previous

# Step 3: Describe the pod for more details
kubectl describe pod my-pod
# Look at: Exit Code, Reason, Events
```

**Common Causes & Fixes:**

| Cause | How You'd Know | Fix |
|-------|---------------|-----|
| Application error (bug in code) | Logs show exception/stack trace | Fix the application code |
| Missing environment variable | Logs show "undefined" or "env not set" | Add ConfigMap/Secret/env in YAML |
| Database connection failed | Logs show "connection refused" | Check DB pod, Service name, network policy |
| Wrong command/entrypoint | Logs show "command not found" | Fix `command` or `args` in Pod spec |
| Missing config file | Logs show "file not found" | Mount ConfigMap/Secret as volume |
| OOMKilled (Out of Memory) | `describe` shows Exit Code 137 | Increase memory limits |
| Liveness probe failing | Events show "Liveness probe failed" | Fix probe endpoint or increase timeout |
| Permission denied | Logs show "Permission denied" | Fix file permissions or securityContext |

**Example: CrashLoopBackOff due to missing env variable**

```bash
$ kubectl logs my-api --previous
Error: DATABASE_URL environment variable is not set
Process exited with code 1
```

**Fix:** Add the environment variable:

```yaml
spec:
  containers:
  - name: my-api
    image: mycompany/api:v2
    env:
    - name: DATABASE_URL
      value: "postgres://db-service:5432/mydb"
```

### Problem 2: `Pending` (Stuck)

**What it means:** The Pod has been created but Kubernetes can't find a node to schedule it on.

**How to diagnose:**

```bash
# Step 1: Check the pod
kubectl get pods
# OUTPUT: my-pod   0/1   Pending   0   10m

# Step 2: Describe the pod — look at Events
kubectl describe pod my-pod
# Look for "FailedScheduling" event

# Step 3: Check node resources
kubectl describe nodes | grep -A 5 "Allocated resources"

# Step 4: Check if there are any taints on nodes
kubectl describe nodes | grep -i taint
```

**Common Causes & Fixes:**

| Cause | Event Message | Fix |
|-------|--------------|-----|
| Not enough CPU/Memory | "Insufficient cpu" or "Insufficient memory" | Reduce resource requests, or add more nodes |
| No matching node selector | "didn't match Pod's node affinity/selector" | Fix `nodeSelector` or labels on nodes |
| Taints with no toleration | "node(s) had taints that the pod didn't tolerate" | Add toleration or remove taint |
| PVC not bound | "persistentvolumeclaim is not bound" | Create matching PV or check StorageClass |
| No available nodes | "0/0 nodes are available" | Add nodes to the cluster |

**Example: Pending due to insufficient resources**

```bash
$ kubectl describe pod my-pod
Events:
  Warning  FailedScheduling  default-scheduler  0/3 nodes are available: 
    1 Insufficient cpu, 2 Insufficient memory.
```

**Fix Options:**
1. Reduce resource requests in the Pod spec
2. Scale up existing nodes (more CPU/memory)
3. Add more nodes to the cluster
4. Delete unused Pods to free up resources

---

### Problem 3: `ImagePullBackOff` / `ErrImagePull`

**What it means:** Kubernetes can't download the container image.

**How to diagnose:**

```bash
# Describe the pod — look at Events
kubectl describe pod my-pod
```

**Common Causes & Fixes:**

| Cause | Event Message | Fix |
|-------|--------------|-----|
| Typo in image name | "repository does not exist" | Fix the image name in the YAML |
| Image tag doesn't exist | "manifest unknown" | Use a valid tag (check registry) |
| Private registry — no auth | "unauthorized: authentication required" | Create an `imagePullSecret` |
| Registry is down | "dial tcp: lookup registry.example.com: no such host" | Check if registry is accessible |
| Network issue | "i/o timeout" | Check cluster's internet/network access |

**Example: Fix ImagePullBackOff with imagePullSecret**

```bash
# Step 1: Create a secret for your private registry
kubectl create secret docker-registry my-registry-secret \
  --docker-server=registry.example.com \
  --docker-username=myuser \
  --docker-password=mypassword \
  --docker-email=me@example.com

# Step 2: Reference it in your Pod spec
```

```yaml
spec:
  imagePullSecrets:
  - name: my-registry-secret
  containers:
  - name: my-app
    image: registry.example.com/my-app:v1.0
```

### Problem 4: `OOMKilled` (Out of Memory)

**What it means:** The container used more memory than its `limits.memory` allows, so the Linux kernel killed it.

**How to diagnose:**

```bash
# Describe the pod
kubectl describe pod my-pod

# Look for:
# Last State: Terminated
# Reason: OOMKilled
# Exit Code: 137
```

**How to fix:**

```yaml
# Option 1: Increase the memory limit
spec:
  containers:
  - name: my-app
    image: my-app:v1
    resources:
      requests:
        memory: "256Mi"    # What the pod asks for
      limits:
        memory: "512Mi"    # Max the pod can use (increase this!)

# Option 2: Fix the memory leak in your application
# (check if the app is leaking memory over time)
```

> ⚠️ **Warning:** Don't just blindly increase limits. If your app has a memory leak, it'll eventually OOMKill at any limit. Fix the root cause!

**How to check memory usage:**

```bash
# Check current memory usage of pods (requires metrics-server)
kubectl top pods

# Check memory usage of nodes
kubectl top nodes

# Check which pods are using the most memory
kubectl top pods --sort-by=memory
```

---

### Problem 5: `CreateContainerConfigError`

**What it means:** Kubernetes can't create the container because something is wrong with the configuration — usually a missing ConfigMap or Secret.

**How to diagnose:**

```bash
kubectl describe pod my-pod
# Look for events like:
# "Error: configmap "my-config" not found"
# "Error: secret "my-secret" not found"
```

**How to fix:**

```bash
# Check if the ConfigMap exists
kubectl get configmap my-config

# Check if the Secret exists
kubectl get secret my-secret

# If missing, create them:
kubectl create configmap my-config --from-literal=key1=value1
kubectl create secret generic my-secret --from-literal=password=mysecretpassword
```

---

### Problem 6: Service Not Working (Can't Connect)

**What it means:** Your Pod is running but you can't reach it through the Service.

**How to diagnose:**

```bash
# Step 1: Check the service exists
kubectl get svc my-service

# Step 2: Check if the service has endpoints (Pods mapped to it)
kubectl get endpoints my-service

# If endpoints is <none>, the selector doesn't match any pods!

# Step 3: Check labels on your pods
kubectl get pods --show-labels

# Step 4: Check the service selector
kubectl describe svc my-service
# Look at "Selector" field

# Step 5: Compare — do the Pod labels match the Service selector?
```

**The Most Common Cause:**

```yaml
# Service selector says:
selector:
  app: my-app      # Looking for pods with label "app: my-app"

# But your Pod has:
labels:
  app: myapp        # Oops! "myapp" ≠ "my-app" (missing hyphen!)
```

**Fix:** Make sure labels match exactly.

```bash
# Quick check: Does the service have endpoints?
$ kubectl get endpoints my-service
NAME         ENDPOINTS            AGE
my-service   <none>               5m    # ❌ No endpoints! Selector mismatch!

# After fixing labels:
$ kubectl get endpoints my-service
NAME         ENDPOINTS            AGE
my-service   172.17.0.5:8080      1m    # ✅ Now it has endpoints
```

---

### Problem 7: Node `NotReady`

**What it means:** A worker node has stopped communicating with the control plane, or it has a critical problem.

**How to diagnose:**

```bash
# Step 1: Check node status
kubectl get nodes

# Step 2: Describe the problem node
kubectl describe node worker-node-3
# Look at Conditions section:
# Ready = False → node is down
# MemoryPressure = True → node running out of memory
# DiskPressure = True → node running out of disk
# PIDPressure = True → too many processes

# Step 3: SSH into the node and check
ssh worker-node-3

# Check kubelet status
systemctl status kubelet

# Check kubelet logs
journalctl -u kubelet -f

# Check disk space
df -h

# Check memory
free -h

# Check container runtime
systemctl status containerd
# or
systemctl status docker
```

**Common Causes & Fixes:**

| Cause | Fix |
|-------|-----|
| kubelet stopped | `systemctl restart kubelet` |
| Disk full | Free up disk space, clean old images: `crictl rmi --prune` |
| Memory pressure | Kill memory-heavy processes, add more RAM |
| Container runtime crashed | `systemctl restart containerd` |
| Network issue | Check network connectivity to control plane |
| Certificate expired | Renew kubelet certificates |

## 🧰 Handy Troubleshooting Commands — Cheat Sheet

Here's a quick reference of all the troubleshooting commands you'll use regularly:

### Pod Troubleshooting

```bash
# Get all pods with status
kubectl get pods -A -o wide

# Describe a pod (DETAILED info + events)
kubectl describe pod <pod-name>

# Get logs (current container)
kubectl logs <pod-name>

# Get logs (previous crashed container)
kubectl logs <pod-name> --previous

# Get logs (specific container in multi-container pod)
kubectl logs <pod-name> -c <container-name>

# Follow logs in real-time
kubectl logs <pod-name> -f

# Get logs from last 30 minutes
kubectl logs <pod-name> --since=30m

# Exec into a pod
kubectl exec -it <pod-name> -- /bin/sh

# Debug with ephemeral container
kubectl debug -it <pod-name> --image=busybox

# Get pod YAML (see the actual running config)
kubectl get pod <pod-name> -o yaml
```

### Event Troubleshooting

```bash
# All events (sorted by time)
kubectl get events --sort-by='.lastTimestamp'

kubectl get events --field-selector type=Warning

kubectl get events --field-selector involvedObject.name=<pod-name>

kubectl get events -w
```

### Node Troubleshooting

```bash
# Node status
```

## 🎯 Real-World Troubleshooting Scenarios

Let's walk through some real-world scenarios step by step.

### Scenario 1: "My app is deployed but users get 502 Bad Gateway"

```bash
# Step 1: Is the Pod running?
kubectl get pods -l app=my-web-app
# → Running ✅

# Step 2: Is the Pod healthy? (check readiness)
kubectl get pods -l app=my-web-app
# → Look at READY column: 0/1 means readiness probe is failing!

# Step 3: Check readiness probe
kubectl describe pod my-web-app-xyz
# → Events show "Readiness probe failed: HTTP probe failed with statuscode: 503"

# Step 4: Check what endpoint the readiness probe is hitting
kubectl get pod my-web-app-xyz -o yaml | grep -A 5 readinessProbe
# → It's checking /health on port 8080

# Step 5: Exec into the pod and test manually
kubectl exec -it my-web-app-xyz -- curl localhost:8080/health
# → "503 Service Unavailable: database connection failed"

# Root Cause: The app starts but can't connect to the database,
# so it reports unhealthy, so the Service removes it from endpoints,
# so Ingress returns 502.

# Fix: Fix the database connection (check DB pod, service, credentials)
```

### Scenario 2: "New deployment is stuck — old pods are still running"

```bash
# Step 1: Check the deployment
kubectl get deployment my-app
# → READY shows 3/5 (only 3 out of 5 desired replicas are ready)

# Step 2: Check the rollout status
kubectl rollout status deployment/my-app
# → "Waiting for deployment spec update to be observed..."

# Step 3: Check if new pods are starting
kubectl get pods -l app=my-app
# → Old pods: Running, New pods: CrashLoopBackOff or ImagePullBackOff

# Step 4: Fix the new pods or rollback
kubectl rollout undo deployment/my-app   # Rollback to previous version
# OR fix the issue and re-deploy
```

## 📊 Troubleshooting Flowchart — Decision Tree

Here's a complete troubleshooting decision tree you can follow:

```
Pod Status?
├── Pending
│   ├── Check: kubectl describe pod → Events
│   ├── Insufficient resources? → Reduce requests or add nodes
│   ├── Node selector/affinity mismatch? → Fix labels or selectors
│   ├── Taints not tolerated? → Add toleration
│   └── PVC not bound? → Create PV or check StorageClass
│
├── ImagePullBackOff / ErrImagePull
│   ├── Wrong image name/tag? → Fix the image reference
│   ├── Private registry? → Add imagePullSecret
│   └── Network issue? → Check cluster internet access
│
├── CrashLoopBackOff
│   ├── Check: kubectl logs <pod> --previous
│   ├── Application error? → Fix the code
│   ├── Missing config/env? → Add ConfigMap/Secret/env
│   ├── OOMKilled (exit code 137)? → Increase memory limits
│   └── Wrong command? → Fix command/args in spec
│
├── Running but not READY
│   ├── Readiness probe failing? → Fix probe or app health endpoint
│   └── Container still starting? → Increase initialDelaySeconds
│
├── Running and READY but not accessible
│   ├── Service endpoints empty? → Fix label selector match
│   ├── Wrong port in service? → Match targetPort with containerPort
│   ├── Network policy blocking? → Check/update network policies
│   └── Ingress misconfigured? → Check Ingress rules and annotations
│
└── Evicted
    ├── Node disk pressure? → Clean up disk, increase node size
    └── Node memory pressure? → Reduce pod memory usage, add nodes
```

## 💡 Pro Tips for Kubernetes Troubleshooting

1. **Always check events first** — `kubectl describe` + `kubectl get events` solve 80% of issues.

2. **Use `--previous` for crash logs** — When a pod is in CrashLoopBackOff, `kubectl logs --previous` shows you WHY it crashed.

3. **Install metrics-server** — Without it, `kubectl top` won't work, and you can't see resource usage.

4. **Know your exit codes:**
   - `0` = Success
   - `1` = General error
   - `126` = Command not executable
   - `127` = Command not found
   - `137` = OOMKilled (128 + 9 = SIGKILL)
   - `143` = Graceful termination (128 + 15 = SIGTERM)

5. **Use labels effectively** — Filter everything by label: `kubectl get pods -l app=my-app`

6. **Keep `nicolaka/netshoot` in your toolkit** — It's the best debugging image for network issues.

7. **Check both the Pod AND the Node** — Sometimes the problem isn't the Pod, it's the Node it's running on.

8. **Use `kubectl get events --sort-by='.lastTimestamp'` frequently** — It's like reading the cluster's diary.

9. **Don't panic on restarts** — A few restarts during initial deployment can be normal. Constant restarts = problem.

10. **Document your fixes** — When you solve a weird issue, write it down. You WILL see it again.
