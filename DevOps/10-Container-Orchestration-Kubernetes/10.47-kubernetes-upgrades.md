# 10.47 — Kubernetes Upgrades — kubeadm upgrade, Zero Downtime

> **One Line Answer:** Kubernetes upgrades are the process of **moving your cluster from one Kubernetes version to a newer one** — carefully upgrading the control plane first, then worker nodes one by one — using tools like `kubeadm upgrade` to keep your applications running with zero downtime.

## 🤔 Why Do You Need to Upgrade Kubernetes?

Kubernetes releases a **new minor version every ~4 months** (e.g., 1.28 → 1.29 → 1.30). Each version brings:

- **New features** (like new API versions, better autoscaling, etc.)
- **Bug fixes** (things that were broken get fixed)
- **Security patches** (critical vulnerabilities get patched)
- **Deprecations** (old features get removed)

### The Kubernetes Support Window

Kubernetes only **supports the last 3 minor versions** at any time. For example, if the latest is 1.30:

| Version | Support Status |
|---------|---------------|
| 1.30 | ✅ Supported (latest) |
| 1.29 | ✅ Supported |
| 1.28 | ✅ Supported |
| 1.27 | ❌ End of life — no more patches! |
| 1.26 | ❌ End of life |

> ⚠️ **If you're running an unsupported version, you're on your own.** No security patches, no bug fixes. This is dangerous in production!

### What Happens If You Don't Upgrade?

| Risk | What Could Happen |
|------|-------------------|
| **Security vulnerabilities** | Known CVEs won't be patched → your cluster is hackable |
| **Deprecated APIs removed** | Your YAML manifests may stop working after skipping versions |
| **No community support** | No one will help you with bugs on old versions |
| **Falling too far behind** | Upgrading from 1.24 → 1.30 is MUCH harder than 1.29 → 1.30 |
| **Compliance issues** | Many security standards require running supported software |

> 💡 **Bottom Line:** Upgrade regularly. Don't skip more than one minor version at a time!

## 📋 Kubernetes Version Numbering

Let's understand how Kubernetes versions work:

```
v1.29.3
 │  │  │
 │  │  └── Patch version (bug fixes, security patches)
 │  └───── Minor version (new features, may have deprecations)
 └──────── Major version (always 1, hasn't changed)
```

| Version Type | Example | What Changes | Risk |
|-------------|---------|--------------|------|
| **Patch** (1.29.2 → 1.29.3) | Bug fixes, security patches | Nothing breaks | Low |
| **Minor** (1.29 → 1.30) | New features, API changes, deprecations | Some things might need updating | Medium |
| **Major** (1.x → 2.x) | Everything could change | Hasn't happened yet! | N/A |

### The Golden Rule of Kubernetes Upgrades

> **You can only upgrade ONE minor version at a time.**

| Upgrade Path | Allowed? |
|-------------|----------|
| 1.28 → 1.29 | ✅ Yes |
| 1.29 → 1.30 | ✅ Yes |
| 1.28 → 1.30 | ❌ No! Must go 1.28 → 1.29 → 1.30 |
| 1.27 → 1.30 | ❌ No! Must go 1.27 → 1.28 → 1.29 → 1.30 |
| 1.29.1 → 1.29.5 | ✅ Yes (patch upgrades are always fine) |

## 🏗️ What Gets Upgraded?

A Kubernetes cluster has many components. Here's what needs upgrading and in what order:

### Upgrade Order (VERY IMPORTANT!)

```
Step 1: Control Plane Components
  ├── kube-apiserver
  ├── kube-controller-manager
  ├── kube-scheduler
  ├── etcd
  └── cloud-controller-manager (if applicable)

Step 2: Worker Node Components (one node at a time)
  ├── kubelet
  └── kube-proxy

Step 3: Add-ons (after cluster is upgraded)
  ├── CoreDNS
  ├── CNI plugin (Calico, Cilium, etc.)
  ├── Ingress controller
  └── Other add-ons
```

### Version Skew Policy

Not all components need to be the **exact same version**. Kubernetes allows a certain "version skew" between components:

| Component | Allowed Versions (if API server is 1.30) |
|-----------|------------------------------------------|
| kube-apiserver | **1.30** (must be upgraded first) |
| kube-controller-manager | 1.29 or 1.30 |
| kube-scheduler | 1.29 or 1.30 |
| kubelet | 1.28, 1.29, or 1.30 (can be up to 2 minor versions older) |
| kube-proxy | 1.28, 1.29, or 1.30 (same as kubelet) |
| kubectl (client) | 1.29, 1.30, or 1.31 (±1 from API server) |

> 💡 **Key Takeaway:** The API server must ALWAYS be the **newest component** in the cluster. Upgrade it first!

## 🔧 How to Upgrade with kubeadm — Step by Step

`kubeadm` is the official tool for creating and managing Kubernetes clusters. Here's the complete upgrade process.

### Prerequisites

Before you start:

```bash
# 1. Check your current version
kubectl version
kubeadm version
kubelet --version

# 2. Read the release notes for the target version
# Check: https://kubernetes.io/releases/

# 3. Read the changelog for deprecations and breaking changes
# Check: https://github.com/kubernetes/kubernetes/blob/master/CHANGELOG/

# 4. BACKUP etcd (CRITICAL!)
# If something goes terribly wrong, you need this backup
ETCDCTL_API=3 etcdctl snapshot save /backup/etcd-snapshot-$(date +%Y%m%d).db \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key

# 5. Verify the backup
ETCDCTL_API=3 etcdctl snapshot status /backup/etcd-snapshot-$(date +%Y%m%d).db --write-out=table
```

> ⚠️ **NEVER skip the etcd backup!** If the upgrade corrupts etcd, you lose your entire cluster state — all your deployments, services, secrets, everything. The backup is your safety net.

---

### Phase 1: Upgrade the First Control Plane Node

Let's say we're upgrading from **1.29 → 1.30** on Ubuntu/Debian:

#### Step 1: Update the package repository

```bash
# SSH into the first control plane node
ssh control-plane-1

# Check what kubeadm versions are available
sudo apt update

# For Ubuntu/Debian — update the Kubernetes apt repository for the new version
# Edit the repo to point to the new version
sudo nano /etc/apt/sources.list.d/kubernetes.list
# Change the version in the URL from 1.29 to 1.30:
# deb [signed-by=/etc/apt/keyrings/kubernetes-apt-keyring.gpg] https://pkgs.k8s.io/core:/stable:/v1.30/deb/ /

sudo apt update

# Check available versions
sudo apt-cache madison kubeadm
```

#### Step 2: Upgrade kubeadm

```bash
# Unhold kubeadm (if it was held to prevent accidental upgrades)
sudo apt-mark unhold kubeadm

# Install the new version of kubeadm
sudo apt-get install -y kubeadm=1.30.0-1.1

# Hold kubeadm again to prevent accidental upgrades
sudo apt-mark hold kubeadm

# Verify the new version
kubeadm version
```

#### Step 3: Plan the upgrade (dry run)

```bash
# This checks if the upgrade is possible and shows what will happen
sudo kubeadm upgrade plan
```

**Example output:**

```
[upgrade/config] Making sure the configuration is correct:
[preflight] Running pre-flight checks.
[upgrade] Running cluster health checks

Components that must be upgraded manually after you have upgraded the control plane with 'kubeadm upgrade apply':
COMPONENT   CURRENT       TARGET
kubelet     v1.29.0       v1.30.0

Upgrade to the latest version in the v1.30 series:

COMPONENT                 CURRENT    TARGET
kube-apiserver            v1.29.0    v1.30.0
kube-controller-manager   v1.29.0    v1.30.0
kube-scheduler            v1.29.0    v1.30.0
kube-proxy                v1.29.0    v1.30.0
CoreDNS                   v1.11.1    v1.11.3
etcd                      3.5.10     3.5.12

You can now apply the upgrade by executing the following command:
    kubeadm upgrade apply v1.30.0
```

> 💡 **`kubeadm upgrade plan` is safe** — it doesn't change anything. It just tells you what WILL happen. Always run it first!

#### Step 4: Apply the upgrade

```bash
# This is the actual upgrade! It upgrades the control plane components
sudo kubeadm upgrade apply v1.30.0
```

**What this does:**
1. Downloads new component images
2. Upgrades the static Pod manifests in `/etc/kubernetes/manifests/`
3. Upgrades etcd (if managed by kubeadm)
4. Updates the `kube-apiserver`, `kube-controller-manager`, `kube-scheduler`
5. Applies new `CoreDNS` and `kube-proxy` configurations
6. Updates certificates if needed

**Expected output:**

```
[upgrade/successful] SUCCESS! Your cluster was upgraded to "v1.30.0". Enjoy!

[upgrade/kubelet] Now that your control plane is upgraded, please proceed with upgrading your kubelets.
```

#### Step 5: Drain the control plane node

```bash
# This safely moves all workload Pods off the node before upgrading kubelet
kubectl drain control-plane-1 --ignore-daemonsets --delete-emptydir-data
```

**What `kubectl drain` does:**
- Marks the node as **unschedulable** (no new Pods will be placed here)
- **Evicts** all Pods from the node (they'll be rescheduled on other nodes)
- Keeps DaemonSet Pods (because they MUST run on every node)

#### Step 6: Upgrade kubelet and kubectl

```bash
# Unhold packages
sudo apt-mark unhold kubelet kubectl

# Install new versions
sudo apt-get install -y kubelet=1.30.0-1.1 kubectl=1.30.0-1.1

# Hold packages again
sudo apt-mark hold kubelet kubectl

# Restart kubelet
sudo systemctl daemon-reload
sudo systemctl restart kubelet
```

#### Step 7: Uncordon the node

```bash
# Allow Pods to be scheduled on this node again
kubectl uncordon control-plane-1

# Verify the node is Ready
kubectl get nodes
```

**Expected output:**

```
NAME              STATUS   ROLES           AGE    VERSION
control-plane-1   Ready    control-plane   365d   v1.30.0   ✅ Upgraded!
worker-1          Ready    <none>          365d   v1.29.0
worker-2          Ready    <none>          365d   v1.29.0
worker-3          Ready    <none>          365d   v1.29.0
```

---

### Phase 2: Upgrade Additional Control Plane Nodes (if HA setup)

If you have multiple control plane nodes (high availability), repeat for each additional control plane node:

```bash
# SSH into the next control plane node
ssh control-plane-2

# Upgrade kubeadm
sudo apt-mark unhold kubeadm
sudo apt-get install -y kubeadm=1.30.0-1.1
sudo apt-mark hold kubeadm

# For additional control plane nodes, use "node" instead of "apply"
sudo kubeadm upgrade node

# Drain the node
kubectl drain control-plane-2 --ignore-daemonsets --delete-emptydir-data

# Upgrade kubelet & kubectl
sudo apt-mark unhold kubelet kubectl
sudo apt-get install -y kubelet=1.30.0-1.1 kubectl=1.30.0-1.1
sudo apt-mark hold kubelet kubectl
sudo systemctl daemon-reload
sudo systemctl restart kubelet

# Uncordon
kubectl uncordon control-plane-2
```

> ⚠️ **Important:** For the **first** control plane node, use `kubeadm upgrade apply`. For **additional** control plane nodes, use `kubeadm upgrade node`.

---

### Phase 3: Upgrade Worker Nodes (One at a Time!)

Now upgrade each worker node. **Do them one at a time** to maintain application availability!

```bash
# === From your workstation (or first control plane node) ===

# Step 1: Drain the worker node
kubectl drain worker-1 --ignore-daemonsets --delete-emptydir-data
# Pods will be rescheduled on worker-2 and worker-3

# === SSH into the worker node ===
ssh worker-1

# Step 2: Upgrade kubeadm
sudo apt-mark unhold kubeadm
sudo apt-get install -y kubeadm=1.30.0-1.1
sudo apt-mark hold kubeadm

# Step 3: Upgrade the node configuration
sudo kubeadm upgrade node

# Step 4: Upgrade kubelet & kubectl
sudo apt-mark unhold kubelet kubectl
sudo apt-get install -y kubelet=1.30.0-1.1 kubectl=1.30.0-1.1
sudo apt-mark hold kubelet kubectl

# Step 5: Restart kubelet
sudo systemctl daemon-reload
sudo systemctl restart kubelet

# === Back on your workstation ===

# Step 6: Uncordon the node
kubectl uncordon worker-1

# Step 7: Verify
kubectl get nodes
```

**Repeat for worker-2, worker-3, etc.**

### After All Nodes Are Upgraded

```bash
$ kubectl get nodes
NAME              STATUS   ROLES           AGE    VERSION
control-plane-1   Ready    control-plane   365d   v1.30.0   ✅
control-plane-2   Ready    control-plane   365d   v1.30.0   ✅
control-plane-3   Ready    control-plane   365d   v1.30.0   ✅
worker-1          Ready    <none>          365d   v1.30.0   ✅
worker-2          Ready    <none>          365d   v1.30.0   ✅
worker-3          Ready    <none>          365d   v1.30.0   ✅
```

🎉 **All nodes upgraded!**

## 🔄 How to Upgrade on RHEL/CentOS/Fedora

The process is the same — only the package management commands are different:

```bash
# Instead of apt, use yum or dnf:

# Check available versions
sudo yum list --showduplicates kubeadm

# Upgrade kubeadm
sudo yum install -y kubeadm-1.30.0-150500.1.1

# Upgrade kubelet & kubectl
sudo yum install -y kubelet-1.30.0-150500.1.1 kubectl-1.30.0-150500.1.1
sudo systemctl daemon-reload
sudo systemctl restart kubelet
```

## ☁️ Upgrading Managed Kubernetes (EKS, AKS, GKE)

If you're using a cloud-managed Kubernetes service, the upgrade process is **much simpler** because the cloud provider handles the control plane for you.

### Amazon EKS

```bash
# Upgrade the control plane (managed by AWS)
aws eks update-cluster-version \
  --name my-cluster \
  --kubernetes-version 1.30

# Check status
aws eks describe-update --name my-cluster --update-id <update-id>

# Upgrade node groups (managed node groups)
aws eks update-nodegroup-version \
  --cluster-name my-cluster \
  --nodegroup-name my-nodegroup \
  --kubernetes-version 1.30

# For self-managed nodes: update the AMI and roll nodes manually
```

### Azure AKS

```bash
# Check available upgrade versions
az aks get-upgrades --resource-group myRG --name myCluster --output table

# Upgrade control plane + node pools
az aks upgrade \
  --resource-group myRG \
  --name myCluster \
  --kubernetes-version 1.30.0

# Upgrade a specific node pool
az aks nodepool upgrade \
  --resource-group myRG \
  --cluster-name myCluster \
  --name mynodepool \
  --kubernetes-version 1.30.0
```

### Google GKE

```bash
# GKE can auto-upgrade! But for manual control:

# Upgrade the control plane
gcloud container clusters upgrade my-cluster \
  --master \
  --cluster-version 1.30.0 \
  --zone us-central1-a

# Upgrade a node pool
gcloud container clusters upgrade my-cluster \
  --node-pool default-pool \
  --cluster-version 1.30.0 \
  --zone us-central1-a
```

### Cloud Provider Comparison

| Feature | EKS | AKS | GKE |
|---------|-----|-----|-----|
| Control plane upgrade | Manual | Manual (with `az aks upgrade`) | Manual or Auto |
| Node upgrade | Manual (managed/self-managed) | Manual or Auto | Manual or Auto |
| Auto-upgrade option | No (for control plane) | Yes | Yes |
| Zero-downtime built-in | Surge upgrades | Surge upgrades | Surge upgrades |
| Rollback support | Limited | Limited | No |

## 🛡️ Zero Downtime Upgrades — How to Avoid Outage

The whole point of doing node-by-node upgrades is to maintain **zero downtime**. Here's how to ensure your applications stay available during the upgrade:

### 1. Use Pod Disruption Budgets (PDBs)

A PDB tells Kubernetes: "Don't disrupt more than X pods at a time."

```yaml
# pdb.yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: my-app-pdb
spec:
  minAvailable: 2          # At least 2 pods must always be running
  # OR
  # maxUnavailable: 1      # At most 1 pod can be down at a time
  selector:
    matchLabels:
      app: my-app
```

```bash
kubectl apply -f pdb.yaml
```

**How it helps during upgrades:**
- When you `kubectl drain` a node, Kubernetes checks PDBs before evicting Pods
- If evicting a Pod would violate the PDB, the drain waits until it's safe
- This ensures your app always has enough replicas running

### 2. Run Multiple Replicas

```yaml
# If you have only 1 replica, draining a node = downtime
# Always run at least 2-3 replicas for production apps
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3    # At least 3 replicas for HA
  # ...
```

### 3. Spread Pods Across Nodes

Use **topology spread constraints** or **pod anti-affinity** to ensure replicas aren't all on the same node:

```yaml
spec:
  topologySpreadConstraints:
  - maxSkew: 1
    topologyKey: kubernetes.io/hostname
    whenUnsatisfiable: DoNotSchedule
    labelSelector:
      matchLabels:
        app: my-app
```

**Why?** If all 3 replicas are on `worker-1` and you drain `worker-1`, all replicas go down at once — even if you have 3 replicas!

### 4. Use Graceful Shutdown

Make sure your application handles `SIGTERM` gracefully:

```yaml
spec:
  terminationGracePeriodSeconds: 60   # Give the app 60 seconds to shut down
  containers:
  - name: my-app
    lifecycle:
      preStop:
        exec:
          command: ["/bin/sh", "-c", "sleep 10"]   # Wait for in-flight requests
```

### 5. Use Readiness Probes

```yaml
readinessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 5
```

Readiness probes ensure that:
- New Pods don't receive traffic until they're actually ready
- When a Pod is being terminated, it's removed from the Service before it shuts down

## 🔄 Handling Deprecated APIs During Upgrades

One of the trickiest parts of upgrading is **API deprecations**. Kubernetes regularly deprecates old API versions.

### Example: A Common Deprecation

```yaml
# Old (deprecated in 1.22):
apiVersion: extensions/v1beta1
kind: Ingress

# New (use this instead):
apiVersion: networking.k8s.io/v1
kind: Ingress
```

### How to Check for Deprecated APIs

```bash
# Use kubectl to check for deprecated APIs in your cluster
# Install "kubectl-deprecations" plugin (also known as "kubent")

# Using kubent (kube-no-trouble):
# Download from: https://github.com/dober-man/kubent
kubent

# Example output:
# >>> Deprecated APIs removed in 1.30 <<<
# KIND      NAMESPACE   NAME         API_VERSION                    REPLACE_WITH
# Ingress   default     my-ingress   extensions/v1beta1             networking.k8s.io/v1
# CronJob   default     my-job       batch/v1beta1                  batch/v1
```

### Using `kubectl convert` Plugin

```bash
# Convert old API versions to new ones
kubectl convert -f old-ingress.yaml --output-version networking.k8s.io/v1
```

### Checking API Deprecations Before Upgrade

```bash
# Check which APIs your cluster resources are using
kubectl api-resources

# Check what API versions are available
kubectl api-versions

# Find all resources using deprecated APIs
kubectl get ingress.extensions -A 2>/dev/null && echo "Still using extensions/v1beta1!"
```

## 🧪 Upgrade Testing — Dry Run Before Production

**Never upgrade production first!** Here's the recommended testing approach:

```
Step 1: Read Release Notes & Changelog
       │
Step 2: Upgrade Dev/Test Cluster First
       │
Step 3: Run All Tests (functional, integration, smoke)
       │
Step 4: Upgrade Staging Cluster
       │
Step 5: Run All Tests + Load Tests
       │
Step 6: Upgrade Production (during maintenance window)
       │
Step 7: Monitor Everything for 24-48 Hours
```

### Pre-Upgrade Checklist

| # | Check | Command |
|---|-------|---------|
| 1 | Current cluster version | `kubectl version` |
| 2 | All nodes healthy | `kubectl get nodes` |
| 3 | All pods running | `kubectl get pods -A` |
| 4 | etcd backup taken | `etcdctl snapshot save ...` |
| 5 | No deprecated APIs in use | `kubent` |
| 6 | PDBs configured for critical apps | `kubectl get pdb -A` |
| 7 | Multiple replicas for important apps | `kubectl get deployments -A` |
| 8 | Release notes reviewed | Manual check |
| 9 | Rollback plan documented | Manual check |
| 10 | Team notified | Communication |

## 🔙 Rollback — What If the Upgrade Goes Wrong?

### Rolling Back a Worker Node

If a worker node has issues after upgrading:

```bash
# Drain the node
kubectl drain worker-1 --ignore-daemonsets --delete-emptydir-data

# SSH into the node
ssh worker-1

# Downgrade kubelet & kubeadm
sudo apt-get install -y kubeadm=1.29.0-1.1 kubelet=1.29.0-1.1 kubectl=1.29.0-1.1
sudo kubeadm upgrade node
sudo systemctl daemon-reload
sudo systemctl restart kubelet

# Uncordon
kubectl uncordon worker-1
```

### Rolling Back the Control Plane

> ⚠️ **Rolling back the control plane is much harder and risky.** This is why the etcd backup is critical!

```bash
# If you need to restore the entire cluster from etcd backup:

# Stop the API server
sudo mv /etc/kubernetes/manifests/kube-apiserver.yaml /tmp/

# Restore etcd from backup
ETCDCTL_API=3 etcdctl snapshot restore /backup/etcd-snapshot.db \
  --data-dir=/var/lib/etcd-restored

# Update etcd to use restored data
# Edit /etc/kubernetes/manifests/etcd.yaml to point to /var/lib/etcd-restored

# Restart everything
sudo mv /tmp/kube-apiserver.yaml /etc/kubernetes/manifests/
sudo systemctl restart kubelet
```

> 💡 **Best Practice:** Instead of rolling back, it's usually better to **fix forward** — find and fix the issue on the new version rather than trying to go back.

## 💡 Pro Tips for Kubernetes Upgrades

1. **Never skip minor versions.** Go 1.28 → 1.29 → 1.30, not 1.28 → 1.30.

2. **Always backup etcd before upgrading.** This is your "undo" button.

3. **Upgrade dev → staging → production.** Never go straight to production.

4. **Use PDBs for all critical applications.** This prevents accidental downtime during drain.

5. **Drain one node at a time.** This keeps your applications available.

6. **Check for deprecated APIs with `kubent` before upgrading.** Fix them before they break.

7. **Monitor closely after upgrading.** Watch logs, events, and metrics for 24-48 hours.

8. **Keep your `kubeadm`, `kubelet`, and `kubectl` versions in sync.** Mismatched versions cause subtle bugs.

9. **Schedule upgrades regularly.** Don't let your cluster fall 3+ versions behind.

10. **Document your upgrade runbook.** Write down every step so anyone on the team can do it.

---

## 📚 Summary

| What | How |
|------|-----|
| Check current version | `kubectl version` / `kubeadm version` |
| Backup etcd | `etcdctl snapshot save` |
| Plan the upgrade | `kubeadm upgrade plan` |
| Upgrade control plane | `kubeadm upgrade apply v1.30.0` |
| Upgrade additional control planes | `kubeadm upgrade node` |
| Drain a node | `kubectl drain <node> --ignore-daemonsets` |
| Upgrade kubelet | `apt install kubelet=1.30.0-1.1` |
| Uncordon a node | `kubectl uncordon <node>` |
| Check deprecated APIs | `kubent` |
| Zero downtime | PDBs + multiple replicas + spread across nodes |
| Managed clusters | Use cloud CLI (aws eks, az aks, gcloud container) |

> **Remember:** Kubernetes upgrades are not scary — they're just a process. Follow the steps, backup etcd, drain one node at a time, and you'll be fine. The biggest mistake is **not upgrading at all** and falling too far behind!
