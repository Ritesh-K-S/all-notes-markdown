# 10.48 — Windows Containers on Kubernetes

> **One Line Answer:** Kubernetes can run **Windows containers alongside Linux containers** in the same cluster by adding **Windows worker nodes** — this lets you orchestrate legacy .NET Framework apps, Windows services, and other Windows-only workloads using the same Kubernetes tools and workflows you already know.

---

## 🤔 Why Would You Run Windows Containers on Kubernetes?

Great question! Most of the Kubernetes world is Linux. So why bother with Windows?

### The Reality of Enterprise Software

Many large companies have **decades of Windows-based applications**:

| Application Type | Example | Can It Run on Linux? |
|-----------------|---------|---------------------|
| .NET Framework 4.x apps | Legacy ASP.NET web apps | ❌ No — .NET Framework is Windows-only |
| Windows Communication Foundation (WCF) | Enterprise SOAP services | ❌ No — Windows-only |
| Windows Services | Background processing services | ❌ No — Windows-only |
| IIS-hosted applications | Classic web apps on Internet Information Services | ❌ No — Windows-only |
| SQL Server (older versions) | SQL Server 2016/2017 | ❌ Some versions are Windows-only |
| PowerShell-based automation | Windows-specific scripts | ⚠️ Partially (PowerShell Core works on Linux) |
| Active Directory integration | Apps that use AD/LDAP tightly | ⚠️ Very difficult on Linux |
| .NET Core / .NET 5+ apps | Modern cross-platform .NET | ✅ Yes! These run on Linux |

> 💡 **Key Insight:** If your app is built on **.NET Core** or **.NET 5+**, it runs on Linux — use Linux containers. Windows containers are for apps that **genuinely can't run on Linux** (like .NET Framework 4.x).

### The Business Case

Instead of maintaining **two separate systems** (Kubernetes for Linux apps + something else for Windows apps), you can:

- Run **everything in one Kubernetes cluster**
- Use the **same CI/CD pipelines**
- Use the **same monitoring and logging tools**
- Use the **same Kubernetes skills and processes**
- **Gradually modernize** Windows apps while keeping them running

### Critical Rule: Control Plane Is ALWAYS Linux

> ⚠️ **The Kubernetes control plane (API server, etcd, scheduler, controller manager) can ONLY run on Linux.** You cannot run the control plane on Windows. Windows nodes are **worker nodes only**.

## ⚙️ Windows Container Types

Before diving in, you need to understand the two types of Windows containers:

### 1. Windows Server Containers (Process Isolation)

- Share the Windows kernel with the host (like Linux containers)
- Lightweight and fast
- **Require the container OS version to match the host OS version**
- Used in production

### 2. Hyper-V Containers (Hyper-V Isolation)

- Each container gets its own Windows kernel via a lightweight VM
- Better security isolation
- Can run different Windows versions
- Heavier than process isolation
- **Not fully supported in Kubernetes** (limited support)

### Windows OS Version Compatibility

This is one of the trickiest parts of Windows containers:

| Host OS Version | Can Run These Container Images |
|----------------|-------------------------------|
| Windows Server 2019 (LTSC) | Windows Server 2019 containers only |
| Windows Server 2022 (LTSC) | Windows Server 2022 containers only |
| Windows Server (SAC) | Matching SAC version only |

> ⚠️ **Unlike Linux**, where any container runs on any kernel version, Windows containers are **strictly version-matched**. A Windows Server 2019 container **will NOT run** on a Windows Server 2022 host (with process isolation).

This means:
- Your Windows node OS version must match your container image OS version
- If you have mixed versions, you need separate node pools
- Always check the base image tag for the OS version

### Common Windows Base Images

```dockerfile
# Windows Server Core (full Windows, ~5 GB!)
FROM mcr.microsoft.com/windows/servercore:ltsc2022

# Windows Nano Server (minimal Windows, ~300 MB)
FROM mcr.microsoft.com/windows/nanoserver:ltsc2022

# .NET Framework (runs on Server Core)
FROM mcr.microsoft.com/dotnet/framework/aspnet:4.8-windowsservercore-ltsc2022

# .NET (cross-platform — prefer this if possible!)
FROM mcr.microsoft.com/dotnet/aspnet:8.0-nanoserver-ltsc2022
```

### Size Comparison

| Base Image | Size | Use Case |
|-----------|------|----------|
| `alpine` (Linux) | ~5 MB | Minimal Linux apps |
| `ubuntu` (Linux) | ~78 MB | General Linux apps |
| `nanoserver` (Windows) | ~300 MB | Minimal Windows apps (.NET Core) |
| `servercore` (Windows) | ~5 GB | Full .NET Framework, IIS, PowerShell |

> 💡 Windows containers are **much larger** than Linux containers. A simple IIS app image can be 5+ GB compared to an Nginx image at ~150 MB. Plan your storage and pull times accordingly.

## 🛠️ Setting Up Windows Nodes in Kubernetes

### Prerequisites

| Requirement | Details |
|-------------|---------|
| Kubernetes version | 1.22+ (Windows support went stable in 1.22) |
| Control plane | Must be Linux |
| Windows node OS | Windows Server 2019 or 2022 (Desktop editions NOT supported) |
| Container runtime | containerd (Docker is deprecated for K8s) |
| Networking (CNI) | Calico, Flannel (overlay or host-gw), or Azure CNI |

### Option 1: Adding Windows Nodes to a kubeadm Cluster

#### Step 1: Prepare the Windows Server

```powershell
# On the Windows Server (PowerShell as Administrator)

# Install the Containers feature
Install-WindowsFeature -Name Containers

# Install containerd
# Download from: https://github.com/containerd/containerd/releases

# Create containerd directory
mkdir "C:\Program Files\containerd"
cd "C:\Program Files\containerd"

# Extract and configure containerd
.\containerd.exe config default | Out-File config.toml -Encoding ascii

# Register containerd as a Windows service
.\containerd.exe --register-service

# Start containerd
Start-Service containerd
```

#### Step 2: Install Kubernetes Components on Windows

```powershell
# Download kubelet and kube-proxy
# These need to match your cluster version

$KubernetesVersion = "v1.30.0"

# Create directories
mkdir C:\k
cd C:\k

# Download kubelet
curl.exe -LO "https://dl.k8s.io/$KubernetesVersion/bin/windows/amd64/kubelet.exe"

# Download kube-proxy
curl.exe -LO "https://dl.k8s.io/$KubernetesVersion/bin/windows/amd64/kube-proxy.exe"

# Download kubectl (optional, for debugging)
curl.exe -LO "https://dl.k8s.io/$KubernetesVersion/bin/windows/amd64/kubectl.exe"

# Add to PATH
$env:Path += ";C:\k"
[Environment]::SetEnvironmentVariable("Path", $env:Path, [EnvironmentVariableTarget]::Machine)
```

#### Step 3: Join the Cluster

```powershell
# On the Linux control plane, generate a join token:
kubeadm token create --print-join-command

# On the Windows node, join the cluster:
# (Use the join command from above, but you'll need to configure
#  kubelet and kube-proxy manually on Windows)

# Register kubelet as a Windows service
sc.exe create kubelet binPath= "C:\k\kubelet.exe --config=C:\k\kubelet-config.yaml --kubeconfig=C:\k\kubeconfig --container-runtime-endpoint=npipe:////.//pipe//containerd-containerd" start= auto

# Start kubelet
Start-Service kubelet
```

> 💡 **In practice**, setting up Windows nodes manually is complex. Most people use managed services (EKS, AKS, GKE) or tools like **SIG Windows tools** that automate this.

### Option 2: Using Managed Kubernetes (Recommended!)

This is **much easier** and is the recommended approach for production.

#### Azure AKS (Best Windows Support)

```bash
# Create a cluster with a Linux node pool (required for system pods)
az aks create \
  --resource-group myRG \
  --name myCluster \
  --node-count 2 \
  --network-plugin azure \
  --generate-ssh-keys

# Add a Windows node pool
az aks nodepool add \
  --resource-group myRG \
  --cluster-name myCluster \
  --name winpool \
  --os-type Windows \
  --os-sku Windows2022 \
  --node-count 2 \
  --node-vm-size Standard_D4s_v3

# Get credentials
az aks get-credentials --resource-group myRG --name myCluster

# Verify — you should see both Linux and Windows nodes
kubectl get nodes -o wide
```

#### Amazon EKS

```bash
# Create cluster (control plane)
eksctl create cluster --name my-cluster --version 1.30

# Add a Windows node group
eksctl create nodegroup \
  --cluster my-cluster \
  --name windows-ng \
  --node-type m5.xlarge \
  --nodes 2 \
  --node-ami-family WindowsServer2022FullContainer

eksctl utils install-vpc-controllers --cluster my-cluster --approve
```

#### Google GKE

```bash
# Create cluster with Windows node pool
gcloud container clusters create my-cluster \
  --enable-ip-alias \
  --num-nodes=2 \
  --zone=us-central1-a

# Add Windows Server node pool
gcloud container node-pools create windows-pool \
  --cluster=my-cluster \
  --zone=us-central1-a \
  --image-type=WINDOWS_LTSC_CONTAINERD \
  --num-nodes=2 \
  --machine-type=n1-standard-4
```

## 📝 Deploying Windows Workloads — YAML Examples

### The Key: `nodeSelector`

Since your cluster has both Linux and Windows nodes, you **must** tell Kubernetes which OS your Pod needs:

```yaml
spec:
  nodeSelector:
    kubernetes.io/os: windows    # Schedule on Windows nodes
```

Without this, your Windows container might get scheduled on a Linux node and fail!

### Example 1: Simple IIS Web Server

```yaml
# iis-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: iis-web
  labels:
    app: iis-web
spec:
  replicas: 3
  selector:
    matchLabels:
      app: iis-web
  template:
    metadata:
      labels:
        app: iis-web
    spec:
      nodeSelector:
        kubernetes.io/os: windows          # MUST specify Windows!
      containers:
      - name: iis
        image: mcr.microsoft.com/windows/servercore/iis:windowsservercore-ltsc2022
        ports:
        - containerPort: 80
        resources:
          limits:
            cpu: "1"
            memory: "2Gi"
          requests:
            cpu: "500m"
            memory: "1Gi"
---
apiVersion: v1
kind: Service
metadata:
  name: iis-web-svc
spec:
  type: LoadBalancer
  selector:
    app: iis-web
  ports:
  - port: 80
    targetPort: 80
```

```bash
# Deploy it
kubectl apply -f iis-deployment.yaml

# Check pods — they should be on Windows nodes
kubectl get pods -o wide
# NAME                       READY   STATUS    NODE
# iis-web-abc12-xyz34        1/1     Running   aks-winpool-11111   ✅ Windows node!

# Get the external IP
kubectl get svc iis-web-svc
```

### Example 3: Custom .NET Framework App with Dockerfile

Here's how you'd containerize your own .NET Framework app:

```dockerfile
# Dockerfile (for .NET Framework 4.8)
FROM mcr.microsoft.com/dotnet/framework/sdk:4.8-windowsservercore-ltsc2022 AS build
WORKDIR /app

# Copy solution and restore packages
COPY *.sln .
COPY MyWebApp/*.csproj ./MyWebApp/
RUN nuget restore

# Copy everything else and build
COPY . .
WORKDIR /app/MyWebApp
RUN msbuild /p:Configuration=Release /p:OutputPath=/out

# Runtime image
FROM mcr.microsoft.com/dotnet/framework/aspnet:4.8-windowsservercore-ltsc2022
WORKDIR /inetpub/wwwroot
COPY --from=build /out/_PublishedWebsites/MyWebApp .
```

# Build and push

## 🚧 Limitations & Differences — What Doesn't Work on Windows

Not everything works the same on Windows. Here's what you need to know:

### Features NOT Supported on Windows

| Feature | Linux | Windows | Notes |
|---------|-------|---------|-------|
| **Privileged containers** | ✅ | ❌ | Windows has no equivalent |
| **HostNetwork** | ✅ | ❌ | Not supported |
| **HostPID / HostIPC** | ✅ | ❌ | Not supported |
| **DaemonSets** | ✅ | ✅ | Works, but limited use cases |
| **ConfigMaps as volumes** | ✅ | ✅ | Works |
| **Secrets as volumes** | ✅ | ✅ | Works |
| **emptyDir volumes** | ✅ | ✅ | Works |
| **hostPath volumes** | ✅ | ✅ | Works (Windows paths like `C:\data`) |
| **PersistentVolumes** | ✅ | ✅ | Works (CSI drivers needed) |
| **Resource limits (CPU/Memory)** | ✅ | ✅ | Works |
| **Readiness/Liveness probes** | ✅ | ✅ | Works |
| **Init containers** | ✅ | ✅ | Works |
| **Security contexts (runAsUser)** | ✅ | ❌ | Not supported — Windows uses different security model |
| **seccomp profiles** | ✅ | ❌ | Not supported |
| **AppArmor / SELinux** | ✅ | ❌ | Linux-specific |
| **Pod Security Standards** | ✅ | ⚠️ | Partial support |
| **IPv6 dual-stack** | ✅ | ✅ | Supported since 1.24 |

### Networking Differences

| Feature | Linux | Windows | Notes |
|---------|-------|---------|-------|
| **ClusterIP Services** | ✅ | ✅ | Works |
| **NodePort Services** | ✅ | ✅ | Works |
| **LoadBalancer Services** | ✅ | ✅ | Works |
| **Ingress** | ✅ | ✅ | Works |
| **Network Policies** | ✅ | ⚠️ | Limited CNI support (Calico supports it) |
| **DNS (CoreDNS)** | ✅ | ✅ | Works (CoreDNS runs on Linux but serves Windows pods) |
| **HostPort** | ✅ | ❌ | Not supported |
| **Service type: ExternalName** | ✅ | ✅ | Works |

### Windows-Specific Considerations

```yaml
# Windows containers use backslashes for paths:
# Linux:
# volumeMounts:
# - mountPath: /app/config
#   name: config-vol

# Windows:
volumeMounts:
- mountPath: "C:\\app\\config"    # Windows-style path!
  name: config-vol

# OR you can use forward slashes (Kubernetes translates them):
volumeMounts:
- mountPath: "/app/config"       # This also works on Windows
  name: config-vol
```

## 🏷️ Node Labeling & Scheduling Strategies

### Automatic Labels

Kubernetes automatically adds OS labels to nodes:

```bash
# Check labels on your nodes
kubectl get nodes --show-labels

# Windows nodes will have:
# kubernetes.io/os=windows
# kubernetes.io/arch=amd64

# Linux nodes will have:
# kubernetes.io/os=linux
# kubernetes.io/arch=amd64
```

### Using nodeSelector (Simple)

```yaml
# For Windows pods:
spec:
  nodeSelector:
    kubernetes.io/os: windows

# For Linux pods:
spec:
  nodeSelector:
    kubernetes.io/os: linux
```

### Using Node Affinity (Advanced)

```yaml
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: kubernetes.io/os
            operator: In
            values:
            - windows
          - key: node.kubernetes.io/windows-build
            operator: In
            values:
            - "10.0.20348"     # Windows Server 2022
```

### Using Taints & Tolerations

You can taint Windows nodes so that **only Windows workloads** get scheduled there:

```bash
# Taint all Windows nodes
kubectl taint nodes win-node-1 os=windows:NoSchedule
kubectl taint nodes win-node-2 os=windows:NoSchedule
```

```yaml
# Windows Pods need to tolerate this taint
spec:
  nodeSelector:
    kubernetes.io/os: windows
  tolerations:
  - key: "os"
    operator: "Equal"
    value: "windows"
    effect: "NoSchedule"
```

## 🔍 Troubleshooting Windows Containers on Kubernetes

### Common Issues

#### Issue 1: Pod Stuck in ContainerCreating

```bash
# This is usually because of image pull times — Windows images are HUGE
kubectl describe pod my-windows-pod

# Events might show:
# "pulling image mcr.microsoft.com/windows/servercore:ltsc2022"
# This can take 5-15 minutes for the first pull!

# Solution: Be patient on first pull, or pre-pull images on nodes
```

#### Issue 2: OS Version Mismatch

```
# Error: "The container operating system does not match the host operating system"

# This means your container image version doesn't match the node's Windows version
# Solution: Use the correct base image tag for your node's Windows version
```

#### Issue 3: Container Fails to Start

```bash
# Check logs
kubectl logs my-windows-pod

# If no logs, exec into the container (if possible)
kubectl exec -it my-windows-pod -- powershell

# Check Windows event logs inside the container
Get-EventLog -LogName Application -Newest 20
```

#### Issue 4: Networking Not Working

```bash
# Test DNS resolution from inside the Windows pod
kubectl exec -it my-windows-pod -- powershell -c "Resolve-DnsName my-service"

# Test connectivity
kubectl exec -it my-windows-pod -- powershell -c "Test-NetConnection my-service -Port 8080"

# Check if the Windows pod has an IP
kubectl get pod my-windows-pod -o wide
```

### Useful Debug Commands for Windows Pods

```bash
# Get a PowerShell shell inside a Windows pod
kubectl exec -it my-windows-pod -- powershell

# Once inside:

# Check network configuration
ipconfig /all

# Check DNS resolution
Resolve-DnsName kubernetes.default.svc.cluster.local

# Test connectivity to a service
Test-NetConnection my-service -Port 80

# Check running processes
Get-Process

# Check Windows version
[System.Environment]::OSVersion

# Check environment variables
Get-ChildItem Env:

# Check IIS status (if running IIS)
Get-Service W3SVC

# Check event logs
Get-EventLog -LogName Application -Newest 10
```

## 📊 Windows vs Linux Containers — Quick Comparison

| Aspect | Linux Containers | Windows Containers |
|--------|-----------------|-------------------|
| **Base image size** | 5 MB - 150 MB | 300 MB - 5+ GB |
| **Startup time** | Seconds | Seconds to minutes |
| **First pull time** | Fast (small images) | Slow (huge images) |
| **Control plane** | ✅ Can host | ❌ Cannot host |
| **Kernel sharing** | Same Linux kernel | Must match Windows version |
| **Security model** | Linux users, namespaces | Windows ACLs, no privileged mode |
| **Ecosystem** | Massive | Growing |
| **Cost** | Free OS | Windows Server license needed |
| **Shell** | bash / sh | PowerShell / cmd |
| **File paths** | `/app/config` | `C:\app\config` |
| **Line endings** | LF | CRLF (be careful with scripts!) |
| **Container runtime** | containerd, CRI-O | containerd |

## 🗺️ Migration Strategy — Moving Windows Apps to Kubernetes

If you have existing Windows applications, here's a recommended migration path:

### Phase 1: Assess

```
Questions to answer:
├── Can the app run on .NET Core/.NET 5+? → Use Linux containers instead!
├── Does it use .NET Framework 4.x features? → Must use Windows containers
├── Does it need Windows-specific features (COM, Registry, etc.)? → Windows containers
├── Does it need GUI? → Cannot containerize (containers are headless)
└── What Windows Server version does it need? → Determines base image
```

### Phase 2: Containerize

```dockerfile
# Example: Containerize a legacy ASP.NET 4.8 app
FROM mcr.microsoft.com/dotnet/framework/aspnet:4.8-windowsservercore-ltsc2022

# Copy published app to IIS wwwroot
WORKDIR /inetpub/wwwroot
COPY ./published-app .

# IIS is already configured in the base image
# The app will be served on port 80
EXPOSE 80
```

### Phase 3: Deploy to Kubernetes

```yaml
# Don't forget:
# 1. nodeSelector for Windows
# 2. Appropriate resource limits (Windows apps often need more memory)
# 3. Health checks (readiness/liveness probes)
# 4. Service for networking
```

### Phase 4: Modernize (Long-term)

```
The goal is to eventually move OFF Windows containers:
├── Port .NET Framework → .NET 8+ (cross-platform)
├── Replace WCF → gRPC or REST APIs
├── Replace Windows Services → Background workers on Linux
├── Replace IIS → Kestrel (built-in .NET web server)
└── End result: Everything on Linux containers (cheaper, smaller, faster)
```

| **OS License** | Free | Windows Server license required |
| **Cloud VM Cost** | Standard pricing | ~30-40% higher (includes Windows license) |
| **Image Storage** | Small images | Large images (more registry storage) |
| **Network Transfer** | Less data (small images) | More data (huge image pulls) |
| **Node Resources** | Lower overhead | Higher overhead (Windows OS is heavier) |

### Example: Azure AKS Cost Comparison

| Node Type | vCPUs | RAM | Monthly Cost (approx.) |
|-----------|-------|-----|----------------------|
| Linux (Standard_D4s_v3) | 4 | 16 GB | ~$140/month |
| Windows (Standard_D4s_v3) | 4 | 16 GB | ~$200/month |

> 💡 **Tip:** Minimize the number of Windows nodes. Run everything you can on Linux, and only use Windows nodes for applications that truly need Windows.

## 💡 Best Practices for Windows Containers on Kubernetes

1. **Always use `nodeSelector: kubernetes.io/os: windows`** — Without it, Windows pods may get scheduled on Linux nodes and fail.

2. **Use Nano Server base images when possible** — They're ~300 MB vs ~5 GB for Server Core. Much faster to pull and start.

3. **Match OS versions carefully** — Container image OS version must match the node OS version (for process isolation).

4. **Pre-pull images on nodes** — Windows images are huge. Pre-pulling avoids long startup times.

5. **Set generous resource requests** — Windows containers typically need more memory than Linux containers (1-2 GB minimum for IIS apps).

6. **Plan for modernization** — Use Windows containers as a **stepping stone**, not a permanent solution. Migrate to .NET 8+ and Linux containers over time.

7. **Use managed Kubernetes** — Setting up Windows nodes manually is painful. Use AKS, EKS, or GKE.

8. **Test OS upgrades in staging first** — Windows base image updates can break apps that depend on specific Windows features.

9. **Monitor Windows-specific metrics** — CPU, memory, IIS request queues, .NET CLR metrics.

10. **Keep Linux system components on Linux nodes** — CoreDNS, Ingress controllers, monitoring agents — these all run on Linux nodes.

## 📚 Summary

| What | Details |
|------|---------|
| **What** | Run Windows containers in a Kubernetes cluster alongside Linux containers |
| **Why** | Legacy .NET Framework apps, Windows Services, IIS apps that can't run on Linux |
| **Control Plane** | Always Linux (cannot run on Windows) |
| **Worker Nodes** | Can be Linux, Windows, or both |
| **Scheduling** | Use `nodeSelector: kubernetes.io/os: windows` |
| **Container Types** | Process isolation (common) and Hyper-V isolation (limited K8s support) |
| **OS Version Matching** | Container image OS must match node OS version |
| **Base Images** | Server Core (~5 GB) or Nano Server (~300 MB) |
| **Key Limitation** | No privileged containers, no HostNetwork, no HostPID |
| **Best Provider** | Azure AKS has the best Windows container support |
| **Long-term Goal** | Migrate to .NET 8+ and Linux containers when possible |

> **Remember:** Windows containers on Kubernetes is a powerful tool for **modernizing legacy Windows applications** without a complete rewrite. But the long-term goal should always be to **move toward Linux containers** whenever possible — they're smaller, faster, cheaper, and better supported. Use Windows containers as a **bridge**, not a destination!
