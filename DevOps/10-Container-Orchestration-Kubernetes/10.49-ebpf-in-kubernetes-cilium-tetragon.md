# 10.49 — eBPF in Kubernetes (Cilium, Tetragon)

> **One Line Answer:** eBPF (extended Berkeley Packet Filter) is a **revolutionary Linux kernel technology** that lets you run custom programs **directly inside the kernel** without changing kernel code — and in Kubernetes, tools like **Cilium** (networking, security, observability) and **Tetragon** (runtime security) use eBPF to make networking faster, security stronger, and observability deeper — all without the overhead of traditional approaches.

---

## 🤔 Before eBPF — Let's Understand the Problem

In Kubernetes, everything runs on Linux. When a Pod talks to another Pod, when a network policy blocks traffic, when you monitor system calls — all of that happens **inside the Linux kernel**.

Traditionally, to customize kernel behavior, you had three terrible options:

| Option | Problem |
|--------|---------|
| **Modify the kernel source code** | Takes years to upstream, risky, requires kernel expertise |
| **Write a kernel module** | Crash the whole system if there's a bug, no safety guarantees |
| **Use iptables / userspace proxies** | Slow, doesn't scale, adds latency |

### The Real Pain in Kubernetes

Let's look at how traditional Kubernetes networking works:

```
Pod A sends packet to Pod B
    ↓
Packet leaves Pod A's network namespace
    ↓
Hits iptables rules (kube-proxy maintains THOUSANDS of rules)
    ↓
iptables does NAT + destination lookup (linearly — O(n)!)
    ↓
Packet is forwarded to Pod B's network namespace
    ↓
Pod B receives the packet
```

When you have **10,000 Services** in your cluster, kube-proxy creates **tens of thousands of iptables rules**. Every single packet has to traverse these rules. Performance **degrades linearly**.

And if you want to add network policies, monitoring, or security on top? Even more iptables rules. More latency. More pain.

**What if you could just hook into the kernel at the exact right point and handle the packet in the most efficient way possible — without iptables, without userspace proxies, without kernel modifications?**

**That's exactly what eBPF does.**

---

## 🧬 What is eBPF?

### The Simple Explanation

Think of the Linux kernel as a **highway system** for your computer. All data (network packets, file access, system calls) travels through this highway.

Normally, you can't change how the highway works. The roads are built, the traffic lights are set, and you just use them as-is.

**eBPF lets you place tiny, smart "traffic cameras" and "traffic controllers" at specific points on the highway** — without rebuilding the road.

These tiny programs:
- **Watch** what's happening (observability)
- **Allow or block** specific traffic (security)
- **Redirect** traffic (networking)
- Run at **kernel speed** (no userspace overhead)
- Are **safe** — the kernel verifies them before running

### The Technical Definition

> **eBPF (extended Berkeley Packet Filter)** is a technology that allows you to run **sandboxed programs inside the Linux kernel** without modifying kernel source code or loading kernel modules. These programs are **verified for safety** by the kernel before execution, so they cannot crash the system.

### A Brief History

| Year | Event |
|------|-------|
| **1992** | BPF (Berkeley Packet Filter) created — a simple packet filtering mechanism for `tcpdump` |
| **2014** | Alexei Starovoitov (at PLUMgrid/Facebook) extended BPF massively → "eBPF" was born |
| **2016** | Facebook uses eBPF in production for load balancing (Katran) |
| **2017** | Cilium project starts — using eBPF for Kubernetes networking |
| **2018** | Netflix, Google, Cloudflare adopt eBPF for networking/security |
| **2021** | eBPF Foundation created (Linux Foundation) — backed by Google, Meta, Microsoft, Netflix, Isovalent |
| **2023** | Cilium becomes a CNCF Graduated project |
| **2024+** | eBPF is the standard for high-performance networking and security in Kubernetes |

## ⚙️ How eBPF Works — Step by Step

Let's walk through how an eBPF program goes from your code to running inside the kernel:

```
┌──────────────────────────────────────────────────────────┐
│                     USER SPACE                           │
│                                                          │
│   1. Write eBPF program (C or Rust)                      │
│           ↓                                              │
│   2. Compile to eBPF bytecode (LLVM/Clang)               │
│           ↓                                              │
│   3. Load into kernel via bpf() system call              │
│                                                          │
├──────────────────────────────────────────────────────────┤
│                     KERNEL SPACE                         │
│                                                          │
│   4. eBPF Verifier checks the program                    │
│      - No infinite loops?           ✅                   │
│      - No invalid memory access?    ✅                   │
│      - Does it terminate?           ✅                   │
│      - Is it safe?                  ✅                   │
│           ↓                                              │
│   5. JIT Compiler converts bytecode to native machine    │
│      code (runs at near-native speed!)                   │
│           ↓                                              │
│   6. Program is attached to a "hook point"               │
│      (network, syscall, tracepoint, etc.)                │
│           ↓                                              │
│   7. Every time the hook fires, your eBPF program runs!  │
│                                                          │
│   8. Results sent back to userspace via eBPF Maps        │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### Let's Break Down Each Step

#### Step 1–2: Writing and Compiling

eBPF programs are typically written in **C** (or Rust) and compiled to a special bytecode format:

```c
// A simple eBPF program that counts packets
// (This is a simplified example — real programs use helper libraries)

#include <linux/bpf.h>
#include <bpf/bpf_helpers.h>

// eBPF Map — like a shared hash table between kernel and userspace
struct {
    __uint(type, BPF_MAP_TYPE_ARRAY);
    __uint(max_entries, 1);
    __type(key, __u32);
    __type(value, __u64);
} packet_count SEC(".maps");

// This function runs every time a packet hits the network interface
SEC("xdp")
int count_packets(struct xdp_md *ctx) {
    __u32 key = 0;
    __u64 *count;

    // Look up the current count
    count = bpf_map_lookup_elem(&packet_count, &key);
    if (count) {
        // Increment the counter
        __sync_fetch_and_add(count, 1);
    }

    // XDP_PASS = let the packet through (don't drop it)
    return XDP_PASS;
}

char _license[] SEC("license") = "GPL";
```

**Don't worry if this looks complex!** You won't write raw eBPF programs in daily DevOps work. Tools like **Cilium** and **Tetragon** handle all of this for you. This is just to show you what's happening under the hood.

#### Step 3: Loading into the Kernel

The compiled bytecode is loaded using the `bpf()` system call. Libraries like `libbpf` make this easier.

#### Step 4: The Verifier (The Safety Net)

This is the **most important part** of eBPF. Before any program runs in the kernel, it must pass the **eBPF verifier**:

| Check | What It Does | Why? |
|-------|-------------|------|
| **No infinite loops** | Guarantees the program terminates | Prevents kernel hangs |
| **Bounded memory access** | All memory accesses are validated | Prevents crashes and security issues |
| **Valid helper functions** | Only approved kernel functions can be called | Prevents unauthorized access |
| **Program size limit** | Max ~1 million instructions | Keeps programs small and fast |
| **Stack size limit** | Max 512 bytes of stack | Prevents stack overflow in kernel |

If the verifier rejects the program, it **does not run**. Period. This is why eBPF is safe — a buggy eBPF program cannot crash your kernel.

#### Step 5: JIT Compilation

After verification, the bytecode is compiled to **native machine code** (x86, ARM, etc.) by the JIT (Just-In-Time) compiler. This means eBPF programs run at **near-native speed** — almost as fast as compiled C code running directly in the kernel.

#### Step 6: Hook Points

eBPF programs attach to **specific points** in the kernel. Here are the main hook types:

| Hook Type | Where It Fires | Used For |
|-----------|----------------|----------|
| **XDP** (eXpress Data Path) | Earliest point when a network packet arrives (before the kernel networking stack) | Packet filtering, DDoS mitigation, load balancing |
| **TC** (Traffic Control) | After initial packet processing | Network policies, traffic shaping |
| **Socket** | When sockets are created/used | Connection tracking, load balancing |
| **Tracepoints** | Predefined kernel instrumentation points | Performance monitoring, debugging |
| **Kprobes / Kretprobes** | Any kernel function entry/exit | Deep kernel observability |
| **Uprobes** | Userspace function entry/exit | Application-level tracing |
| **LSM** (Linux Security Module) | Security check points | Security enforcement |
| **cgroup** | When processes in a cgroup perform actions | Container-level policies |

#### Step 7–8: eBPF Maps

eBPF Maps are **shared data structures** between eBPF programs (kernel space) and user-space applications. Think of them like shared memory tables.

```
┌─────────────┐                    ┌─────────────┐
│  User Space  │  ← reads data ←  │   eBPF Map   │
│  (Cilium,    │                   │  (Hash Table │
│   Tetragon,  │                   │   Array,     │
│   CLI tools) │  → writes config →│   LRU, etc.) │
└─────────────┘                    └──────┬───────┘
                                          │
                                   ┌──────┴───────┐
                                   │  eBPF Program │
                                   │  (in Kernel)  │
                                   │  writes data  │
                                   └──────────────┘
```

Common Map types:

| Map Type | Description | Use Case |
|----------|-------------|----------|
| `BPF_MAP_TYPE_HASH` | Key-value hash table | Connection tracking, IP lookups |
| `BPF_MAP_TYPE_ARRAY` | Fixed-size array | Counters, statistics |
| `BPF_MAP_TYPE_LRU_HASH` | Hash table with LRU eviction | Caching, rate limiting |
| `BPF_MAP_TYPE_RINGBUF` | Ring buffer | Event streaming to userspace |
| `BPF_MAP_TYPE_LPM_TRIE` | Longest prefix match trie | CIDR-based routing/policy |

## 🔄 eBPF vs Traditional Approaches — Why It Matters

Let's compare eBPF with what it replaces:

### Networking: eBPF vs iptables

| Feature | iptables (Traditional) | eBPF (Cilium) |
|---------|----------------------|---------------|
| **Performance** | Degrades with rule count (O(n)) | Constant time lookups (O(1) hash tables) |
| **Rules at scale** | 10K+ services = very slow | 100K+ services = still fast |
| **Observability** | None built-in | Rich metrics, flow logs |
| **Protocol awareness** | L3/L4 only (IP, TCP/UDP) | L3/L4 + L7 (HTTP, gRPC, Kafka) |
| **NAT** | Conntrack + iptables NAT | eBPF-based NAT (faster) |
| **Update speed** | Rebuilds entire rule chain | Incremental updates |

### Security: eBPF vs Traditional

| Feature | Traditional (audit, seccomp) | eBPF (Tetragon) |
|---------|------------------------------|-----------------|
| **Granularity** | Process-level | Kubernetes-aware (Pod, namespace, label) |
| **Enforcement** | Limited to predefined rules | Custom policies at any kernel hook |
| **Overhead** | Moderate | Minimal (runs in kernel) |
| **Real-time** | Log-based (after the fact) | Real-time detection + enforcement |
| **Context** | PID, UID | Pod name, namespace, container, binary name |

## 🌐 Cilium — eBPF-Powered Networking for Kubernetes

### What is Cilium?

> **Cilium** is a Kubernetes **CNI (Container Network Interface) plugin** that uses eBPF to provide **networking, security, and observability** for your Kubernetes cluster — replacing kube-proxy, iptables, and traditional CNI plugins with a faster, smarter eBPF-based approach.

### Why Cilium Exists

| Traditional CNI (Calico, Flannel) | Cilium |
|----------------------------------|--------|
| Uses iptables for packet routing | Uses eBPF programs in the kernel |
| kube-proxy creates iptables rules | Replaces kube-proxy entirely |
| L3/L4 network policies only | L3/L4 + **L7** network policies (HTTP, gRPC, Kafka) |
| Limited observability | Deep observability with **Hubble** |
| Performance degrades at scale | High performance even with 50K+ services |

### Cilium Architecture

```

│                  Kubernetes Cluster                │
│                                                    │
│  ┌──────────────────────────────────────────────┐  │

│  │                   │    │                   │      │
```

### Cilium Components Explained

| Component | What It Does | How It Runs |
|-----------|-------------|-------------|
| **Cilium Agent** | Runs on every node, loads eBPF programs into the kernel, manages networking and policy | DaemonSet |
| **Cilium Operator** | Manages cluster-wide tasks (IPAM, CRD syncing, garbage collection) | Deployment (1-2 replicas) |

| **eBPF Programs** | The actual programs running in the kernel doing networking/security/LB | Loaded by Cilium Agent |

### Cilium Feature Deep Dive

#### 1. kube-proxy Replacement

Traditionally, **kube-proxy** runs on every node and creates iptables rules for Service load balancing. Cilium can **completely replace kube-proxy** with eBPF:

```
Traditional (kube-proxy):
  Packet → iptables (linear scan through rules) → Destination Pod
  ❌ Slow at scale, no observability

Cilium (eBPF):
  Packet → eBPF hash table lookup (O(1)) → Destination Pod
  ✅ Constant time, with full flow visibility
```

To verify kube-proxy replacement:

```bash
# Check if kube-proxy replacement is active
kubectl -n kube-system exec ds/cilium -- cilium-dbg status | grep KubeProxyReplacement

# Expected output:
# KubeProxyReplacement:   True   [eth0 (Direct Routing)]
```

#### 2. L7 Network Policies

This is one of Cilium's **superpowers**. Traditional Kubernetes NetworkPolicy only works at L3/L4 (IP and port). Cilium can enforce policies at **L7** (application layer):

**Example: Allow only GET requests to a specific API path**

```yaml
# cilium-l7-policy.yaml
apiVersion: cilium.io/v2
kind: CiliumNetworkPolicy
metadata:
  name: l7-rule
  namespace: default
spec:
  endpointSelector:
    matchLabels:
      app: backend-api
  ingress:
  - fromEndpoints:
    - matchLabels:
        app: frontend
    toPorts:
    - ports:
      - port: "80"
        protocol: TCP
      rules:
        http:
        - method: "GET"
          path: "/api/v1/public/.*"
        - method: "POST"
          path: "/api/v1/data"
          headers:
          - 'Content-Type: application/json'
```

**Let's break down what this policy does:**

| Field | What It Means |
|-------|--------------|
| `endpointSelector: app: backend-api` | This policy applies to Pods with label `app=backend-api` |
| `fromEndpoints: app: frontend` | Only Pods with label `app=frontend` can send traffic |
| `port: "80"` | Traffic must be on port 80 |
| `method: "GET"`, `path: "/api/v1/public/.*"` | Allow GET requests to any path starting with `/api/v1/public/` |
| `method: "POST"`, `path: "/api/v1/data"` | Allow POST requests ONLY to `/api/v1/data` |
| `headers` | The POST must have a JSON Content-Type header |

**This is impossible with standard Kubernetes NetworkPolicy!** Standard policies can only say "allow traffic from frontend to backend on port 80" — they can't distinguish between GET/POST or check the URL path.

Apply and test:

```bash
# Apply the policy
kubectl apply -f cilium-l7-policy.yaml

# Check the policy
kubectl get cnp  # cnp = CiliumNetworkPolicy

# Describe the policy
kubectl describe cnp l7-rule
```

#### 3. Cluster-Wide Network Policies

Cilium also provides `CiliumClusterwideNetworkPolicy` for cluster-scoped policies:

```yaml
# cluster-wide-deny-all-egress-to-internet.yaml
apiVersion: cilium.io/v2
kind: CiliumClusterwideNetworkPolicy
metadata:
  name: deny-external-traffic
spec:
  endpointSelector: {}    # Applies to ALL pods in the cluster
  egressDeny:
  - toCIDR:
    - "0.0.0.0/0"         # Deny all egress to the internet
  egress:
  - toEndpoints:
    - {}                   # But allow all pod-to-pod traffic within the cluster
  - toEntities:
    - kube-apiserver       # Allow traffic to the Kubernetes API server
  - toCIDR:
    - "10.0.0.0/8"        # Allow traffic within private networks
```

**What this does:** Blocks all Pods from reaching the public internet, while allowing internal cluster communication and access to the Kubernetes API. This is a great **security baseline** for any production cluster.

#### 4. Transparent Encryption (WireGuard)

Cilium can encrypt all pod-to-pod traffic using **WireGuard** — automatically, with zero config on your applications:

```bash
# Install Cilium with WireGuard encryption
cilium install --set encryption.enabled=true \
               --set encryption.type=wireguard
```

Or via Helm:

```yaml
# values.yaml for Helm
encryption:
  enabled: true
  type: wireguard
  wireguard:
    userspaceFallback: false
```

After enabling:

```bash
# Verify encryption is active
cilium-dbg encrypt status

# Expected output:
# Encryption: Wireguard
# Wireguard:
#   ...
#   Interface: cilium_wg0
#   Public Key: <key>
```

**All pod-to-pod traffic is now encrypted at the kernel level — your applications don't need to change anything!**

#### 5. Bandwidth Management

Cilium can enforce bandwidth limits on Pods using eBPF:

```yaml
# Pod with bandwidth annotations
apiVersion: v1
kind: Pod
metadata:
  name: bandwidth-limited-pod
  annotations:
    kubernetes.io/ingress-bandwidth: "10M"   # 10 Megabits/sec ingress limit
    kubernetes.io/egress-bandwidth: "5M"     # 5 Megabits/sec egress limit
spec:
  containers:
  - name: app
    image: nginx
```

Enable bandwidth manager in Cilium:

```bash
helm upgrade cilium cilium/cilium --namespace kube-system \
  --set bandwidthManager.enabled=true \
  --set bandwidthManager.bbr=true    # Use BBR congestion control
```


## 🛡️ Tetragon — eBPF-Based Runtime Security

### What is Tetragon?

> **Tetragon** is an eBPF-based **runtime security observability and enforcement** tool for Kubernetes. It lets you **detect and prevent** security threats at the **kernel level** — monitoring process execution, file access, network connections, and more, all with **full Kubernetes context** (Pod name, namespace, labels).

Think of it this way:
- **Cilium** = eBPF for **networking**
- **Tetragon** = eBPF for **security**

### Why Tetragon?

Traditional security tools have major limitations:

| Traditional Security | Tetragon (eBPF) |
|---------------------|-----------------|
| Runs in userspace — can be tampered with | Runs **inside the kernel** — cannot be bypassed |
| Works with PIDs (meaningless in K8s) | Works with **Pod names, namespaces, labels** |
| High performance overhead | **Minimal overhead** (kernel-level) |
| Detection only (log and alert) | **Detection AND enforcement** (can kill processes in real-time) |
| Delayed response (log → SIEM → alert → action) | **Real-time** (detect and act in milliseconds) |
| Limited visibility into kernel events | **Deep kernel-level visibility** (syscalls, file access, network) |

### TracingPolicy — The Heart of Tetragon

**TracingPolicy** is how you tell Tetragon **what to watch for**. It's a Kubernetes Custom Resource (CRD) that defines which kernel events to monitor and what to do about them.

#### Example 1: Monitor File Access to Sensitive Paths

```yaml
# tracing-policy-file-access.yaml
apiVersion: cilium.io/v1alpha1
kind: TracingPolicy
metadata:
  name: monitor-sensitive-files
spec:
  kprobes:
  - call: "fd_install"              # Kernel function called when a file is opened
    syscall: false
    args:
    - index: 0
      type: "int"                   # File descriptor
    - index: 1
      type: "file"                  # File path
    selectors:
    - matchArgs:
      - index: 1
        operator: "Prefix"
        values:
        - "/etc/shadow"             # Monitor access to password file
        - "/etc/passwd"             # Monitor access to user file
        - "/etc/kubernetes/"        # Monitor access to K8s configs
        - "/var/run/secrets/"       # Monitor access to K8s secrets
        - "/root/.ssh/"            # Monitor access to SSH keys
        - "/etc/ssl/private/"      # Monitor access to SSL private keys
```

**What this does:**
- Hooks into the `fd_install` kernel function — called every time a file descriptor is created (file opened)
- Filters for sensitive file paths only
- Logs every time ANY process in ANY Pod tries to read these files
- Output includes full Kubernetes context (Pod name, namespace, container, etc.)

Apply and test:

```bash
# Apply the policy
kubectl apply -f tracing-policy-file-access.yaml

# Watch for events
tetra getevents --output compact

# Now, in another terminal, exec into a pod and try to read /etc/shadow
kubectl exec -it nginx-pod -- cat /etc/shadow

# Tetragon output will show:
# 📬 open    default/nginx-pod /etc/shadow
# 🚀 process default/nginx-pod /bin/cat /etc/shadow
```

#### Example 2: Detect and BLOCK Malicious Processes

This is where Tetragon gets really powerful — it can **kill processes in real-time**:

```yaml
# tracing-policy-block-crypto-mining.yaml
apiVersion: cilium.io/v1alpha1
kind: TracingPolicy
metadata:
  name: block-crypto-miners
spec:
  kprobes:
  - call: "tcp_connect"              # Kernel function for outgoing TCP connections
    syscall: false
    args:
    - index: 0
      type: "sock"
    selectors:
    - matchArgs:
      - index: 0
        operator: "DPort"
        values:
        - "3333"                      # Stratum mining protocol port
        - "4444"                      # Common mining pool port
        - "5555"                      # Another mining pool port
        - "7777"                      # Another mining pool port
        - "8888"                      # Another mining pool port
        - "9999"                      # Another mining pool port
      matchActions:
      - action: Sigkill              # KILL the process immediately!
```

**What this does:**
- Monitors all outgoing TCP connections at the kernel level
- If any process tries to connect to known crypto mining ports (3333, 4444, etc.)
- **Immediately kills the process** with SIGKILL — before the connection is even established!
- This happens at kernel speed — the attacker has no chance to bypass it

```bash
# Apply the policy
kubectl apply -f tracing-policy-block-crypto-mining.yaml

# If a compromised Pod tries to connect to a mining pool:
# 💥 SIGKILL default/compromised-pod /usr/bin/xmrig (TCP connect to 1.2.3.4:3333)
```

#### Example 3: Monitor Network Connections

```yaml
# tracing-policy-network.yaml
apiVersion: cilium.io/v1alpha1
kind: TracingPolicy
metadata:
  name: monitor-network-connections
spec:
  kprobes:
  - call: "tcp_connect"
    syscall: false
    args:
    - index: 0
      type: "sock"
    selectors:
    - matchNamespaces:
      - namespace: default
        operator: In
        values:
        - "production"
        - "staging"
  - call: "tcp_close"
    syscall: false
    args:
    - index: 0
      type: "sock"
    selectors:
    - matchNamespaces:
      - namespace: default
        operator: In
        values:
        - "production"
        - "staging"
```

**What this does:** Monitors all TCP connection opens and closes in the `production` and `staging` namespaces, giving you full visibility into network activity.

#### Example 4: Detect Privilege Escalation

```yaml
# tracing-policy-privilege-escalation.yaml
apiVersion: cilium.io/v1alpha1
kind: TracingPolicy
metadata:
  name: detect-privilege-escalation
spec:
  kprobes:
  - call: "commit_creds"             # Kernel function that changes process credentials
    syscall: false
    args:
    - index: 0
      type: "cred"                   # The new credentials being applied
    selectors:
    - matchArgs:
      - index: 0
        operator: "Equal"
        values:
        - "0"                         # UID 0 = root
      matchActions:
      - action: Post                  # Log the event (use Sigkill to also kill the process)
```

**What this does:**
- Monitors the `commit_creds` kernel function — called when a process changes its credentials
- Specifically watches for any process changing to UID 0 (root)
- Logs the event with full Kubernetes context
- Change `action: Post` to `action: Sigkill` to **kill any process attempting privilege escalation**

## ❓ Common Questions & Troubleshooting

### Q1: Do I need to know C to use Cilium/Tetragon?

**No!** You never write eBPF programs directly. Cilium and Tetragon handle all the eBPF programming for you. You just write **YAML manifests** (CiliumNetworkPolicy, TracingPolicy) — the same way you write any other Kubernetes resource.

### Q2: Does eBPF add performance overhead?

**Minimal.** eBPF programs run inside the kernel at near-native speed. In fact, Cilium is often **faster** than traditional iptables-based networking because it uses efficient hash-table lookups instead of linear iptables rule scanning.

### Q3: What kernel version do I need?

| Feature | Minimum Kernel |
|---------|---------------|
| Cilium (basic) | 4.19+ |
| Cilium (full features) | 5.10+ |
| Tetragon | 4.19+ (5.x+ recommended) |
| kube-proxy replacement | 5.7+ |
| WireGuard encryption | 5.6+ |
| BBR bandwidth management | 5.18+ |

Most modern Linux distributions (Ubuntu 22.04+, RHEL 9+, Amazon Linux 2023) have a sufficiently new kernel.

Check your kernel version:
```bash
uname -r
# Example: 5.15.0-91-generic — this supports all features!
```

### Q4: Can I use Cilium with an existing cluster?

**Yes**, but it requires a migration:
1. Install Cilium alongside your current CNI
2. Gradually migrate workloads
3. Remove the old CNI

Cilium provides a [migration guide](https://docs.cilium.io/en/stable/installation/k8s-install-migration/) for this process.

### Q5: How do I debug Cilium issues?

```bash
# Check Cilium agent logs
kubectl -n kube-system logs ds/cilium

# Check Cilium status in detail
kubectl -n kube-system exec ds/cilium -- cilium-dbg status --verbose

# Check endpoint health
kubectl -n kube-system exec ds/cilium -- cilium-dbg endpoint health

# Check BPF maps
kubectl -n kube-system exec ds/cilium -- cilium-dbg bpf lb list
kubectl -n kube-system exec ds/cilium -- cilium-dbg bpf ct list global

# Check Cilium monitor for real-time events
kubectl -n kube-system exec ds/cilium -- cilium-dbg monitor --type drop
```

### Q6: How is Tetragon different from Falco?

| Feature | Tetragon | Falco |
|---------|----------|-------|
| **Technology** | eBPF (kernel-level) | eBPF or kernel module |
| **Enforcement** | ✅ Can kill processes (Sigkill) | ❌ Detection only (alerts) |
| **Performance** | Very low overhead | Moderate overhead |
| **Policy language** | TracingPolicy (K8s CRDs) | Falco Rules (custom YAML) |
| **K8s integration** | Native (CRDs, labels, namespaces) | Via plugins |
| **Community** | Cilium/Isovalent (CNCF) | Sysdig (CNCF) |

**In short:** Use Tetragon if you want **enforcement** (kill bad processes). Use Falco if you want **detection** with a mature rule ecosystem. Many teams use both!
