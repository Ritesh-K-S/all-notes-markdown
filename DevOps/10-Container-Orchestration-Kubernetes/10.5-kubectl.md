# 10.5 — kubectl — The Swiss Army Knife (get, describe, apply, delete, logs, exec)

> **One Line Answer:** `kubectl` (pronounced "kube-control" or "kube-cuddle" — yes, people actually debate this!) is the **command-line tool that lets you talk to your Kubernetes cluster** — it's how you deploy apps, check what's running, read logs, debug problems, and manage literally everything in Kubernetes.

---

## 🤔 The Big Picture — What Is kubectl?

Think of `kubectl` as your **remote control for Kubernetes**. Just like Docker has `docker` CLI, Kubernetes has `kubectl`. It sends commands to the **Kubernetes API Server**, which then makes things happen in your cluster.

```
┌─────────────────────────────────────────────────────────────┐
│                        YOU (DevOps Engineer)                 │
│                                                              │
│    $ kubectl get pods                                        │
│    $ kubectl apply -f deployment.yaml                        │
│    $ kubectl logs my-app                                     │
│    $ kubectl exec -it my-app -- bash                         │
└─────────────────────┬────────────────────────────────────────┘
                      │  (HTTPS request to API Server)
                      ▼
┌─────────────────────────────────────────────────────────────┐
│              Kubernetes API Server                           │
│                                                              │
│    Receives the command → Validates it → Executes it         │
│                                                              │
│    ┌──────────┐  ┌──────────┐  ┌──────────┐                 │
│    │  Node 1  │  │  Node 2  │  │  Node 3  │                 │
│    │  ┌────┐  │  │  ┌────┐  │  │  ┌────┐  │                 │
│    │  │Pods│  │  │  │Pods│  │  │  │Pods│  │                 │
│    │  └────┘  │  │  └────┘  │  │  └────┘  │                 │
│    └──────────┘  └──────────┘  └──────────┘                 │
└─────────────────────────────────────────────────────────────┘
```

### How kubectl Knows Which Cluster to Talk To

kubectl uses a **kubeconfig** file (located at `~/.kube/config` by default) that contains:
- **Clusters** — The API server addresses
- **Users** — Authentication credentials
- **Contexts** — Which user connects to which cluster

```bash
# See your current kubeconfig
kubectl config view

# See which cluster you're currently connected to
kubectl config current-context

# List all available contexts
kubectl config get-contexts
```

## 📋 Quick Reference — The Essential kubectl Commands

| Command | What It Does | Example |
|---------|-------------|---------|
| `kubectl get` | List resources | `kubectl get pods` |
| `kubectl describe` | Show detailed info about a resource | `kubectl describe pod my-app` |
| `kubectl apply` | Create or update resources from a file | `kubectl apply -f app.yaml` |
| `kubectl delete` | Delete resources | `kubectl delete pod my-app` |
| `kubectl logs` | Show container logs | `kubectl logs my-app` |
| `kubectl exec` | Run a command inside a container | `kubectl exec -it my-app -- bash` |
| `kubectl create` | Create a resource (imperative) | `kubectl create deployment nginx --image=nginx` |
| `kubectl edit` | Edit a resource in your editor | `kubectl edit deployment nginx` |
| `kubectl scale` | Scale a deployment up or down | `kubectl scale deployment nginx --replicas=5` |
| `kubectl port-forward` | Forward a port to your machine | `kubectl port-forward pod/my-app 8080:80` |
| `kubectl cp` | Copy files to/from containers | `kubectl cp my-app:/logs/app.log ./app.log` |

## 📐 The Basic Syntax

Every kubectl command follows this pattern:

```
kubectl [COMMAND] [RESOURCE_TYPE] [RESOURCE_NAME] [FLAGS]
```

For example:
```bash
kubectl    get       pods       my-app      -o wide
│          │         │          │            │
│          │         │          │            └── Flag: output format
│          │         │          └── Resource name (optional)
│          │         └── Resource type (what kind of thing)
│          └── Command (what you want to do)
└── The tool itself
```

## 1️⃣ `kubectl get` — List Resources

This is the command you'll use **most often**. It lists resources in your cluster.

### Getting Pods

```bash
# List all pods in the current namespace
kubectl get pods
```

**Output:**
```
NAME                     READY   STATUS    RESTARTS   AGE
nginx-7c5ddbdf54-abcde   1/1     Running   0          5m
nginx-7c5ddbdf54-fghij   1/1     Running   0          5m
redis-master-0            1/1     Running   0          10m
```

Let's understand each column:
- **NAME** — The name of the pod
- **READY** — `1/1` means 1 out of 1 container is ready (healthy)
- **STATUS** — Current state: `Running`, `Pending`, `CrashLoopBackOff`, `Error`, etc.
- **RESTARTS** — How many times the container restarted (high number = problem!)
- **AGE** — How long since the pod was created

### Getting More Details with `-o wide`

```bash
# Get pods with more info (IP, Node, etc.)
kubectl get pods -o wide
```

**Output:**
```
NAME                     READY   STATUS    RESTARTS   AGE   IP           NODE       NOMINATED NODE   READINESS GATES
nginx-7c5ddbdf54-abcde   1/1     Running   0          5m    10.244.1.5   worker-1   <none>           <none>
nginx-7c5ddbdf54-fghij   1/1     Running   0          5m    10.244.2.3   worker-2   <none>           <none>
```

Now you can see:
- **IP** — The pod's internal IP address
- **NODE** — Which worker node the pod is running on

### Getting Different Resource Types

```bash
# List all nodes in the cluster
kubectl get nodes

# List all services
kubectl get services
# OR shorthand:
kubectl get svc

# List all deployments
kubectl get deployments
# OR shorthand:
kubectl get deploy

# List all namespaces
kubectl get namespaces
# OR shorthand:
kubectl get ns

# List all configmaps
kubectl get configmaps
# OR shorthand:
kubectl get cm

# List all secrets
kubectl get secrets

# List ALL resources (pods, services, deployments, etc.)
kubectl get all
```


### Filtering by Namespace

By default, kubectl shows resources in the `default` namespace. Most real-world apps use custom namespaces.

```bash
# List pods in a specific namespace
kubectl get pods -n kube-system

# List pods in ALL namespaces
kubectl get pods --all-namespaces
# OR shorthand:
kubectl get pods -A
```

**Output of `kubectl get pods -A`:**
```

```

### Filtering by Labels

Labels are key-value pairs attached to resources. You can filter using them:

```bash
# Get pods with a specific label
kubectl get pods -l app=nginx

# Get pods with multiple labels
kubectl get pods -l app=nginx,environment=production

# Get pods where a label exists (any value)
kubectl get pods -l app

# Get pods where a label does NOT exist
kubectl get pods -l '!app'
```

### Output Formats

kubectl can output data in many formats:

```bash
# Default table format
kubectl get pods

# Wide format (more columns)
kubectl get pods -o wide

# YAML format (great for copying/learning)
kubectl get pod nginx-7c5ddbdf54-abcde -o yaml

# JSON format (great for scripting)
kubectl get pod nginx-7c5ddbdf54-abcde -o json

# Just the names
kubectl get pods -o name
# Output:
# pod/nginx-7c5ddbdf54-abcde
# pod/nginx-7c5ddbdf54-fghij

# Custom columns
kubectl get pods -o custom-columns=NAME:.metadata.name,STATUS:.status.phase,IP:.status.podIP
# Output:
# NAME                      STATUS    IP
# nginx-7c5ddbdf54-abcde    Running   10.244.1.5
# nginx-7c5ddbdf54-fghij    Running   10.244.2.3

# Using jsonpath (powerful for extracting specific fields)
kubectl get pods -o jsonpath='{.items[*].metadata.name}'
# Output: nginx-7c5ddbdf54-abcde nginx-7c5ddbdf54-fghij
```

### Watch Mode — Live Updates

```bash
# Watch pods in real-time (updates automatically!)
kubectl get pods --watch
# OR shorthand:
kubectl get pods -w
```

This is like a **live dashboard** — it shows you pod changes as they happen. Super useful when deploying or debugging! Press `Ctrl+C` to stop.

### Sorting Results

```bash
# Sort pods by creation time
kubectl get pods --sort-by=.metadata.creationTimestamp

# Sort pods by restart count
kubectl get pods --sort-by=.status.containerStatuses[0].restartCount

# Sort nodes by CPU capacity
kubectl get nodes --sort-by=.status.capacity.cpu
```

| Scripting and automation | `kubectl get -o json/yaml` |

## 3️⃣ `kubectl apply` — Create or Update Resources (Declarative)

`kubectl apply` is the **most important command for real-world Kubernetes**. It takes a YAML file and either **creates** the resource (if it doesn't exist) or **updates** it (if it does).

This is called the **declarative** approach — you declare what you want, and Kubernetes figures out how to make it happen.

### Creating a Simple Pod with YAML

First, create a YAML file:

```yaml
# nginx-pod.yaml
apiVersion: v1            # API version for Pods
kind: Pod                 # Type of resource
metadata:
  name: my-nginx          # Name of the pod
  labels:
    app: nginx            # Labels for organizing/selecting
spec:
  containers:
    - name: nginx         # Container name
      image: nginx:1.27   # Docker image to use
      ports:
        - containerPort: 80   # Port the container listens on
```

Now apply it:

```bash
kubectl apply -f nginx-pod.yaml
```

**Output:**
```
pod/my-nginx created
```

### Creating a Deployment

```yaml
# nginx-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
  labels:
    app: nginx
spec:
  replicas: 3                    # Run 3 copies of the pod
  selector:
    matchLabels:
      app: nginx                 # Select pods with this label
  template:
    metadata:
      labels:
        app: nginx               # Label each pod
    spec:
      containers:
        - name: nginx
          image: nginx:1.27
          ports:
            - containerPort: 80
          resources:
            requests:
              cpu: "100m"        # Request 0.1 CPU cores
              memory: "64Mi"     # Request 64 MB RAM
            limits:
              cpu: "200m"        # Max 0.2 CPU cores
              memory: "128Mi"    # Max 128 MB RAM
```

```bash
# Apply the deployment
kubectl apply -f nginx-deployment.yaml
```

**Output:**
```
deployment.apps/nginx-deployment created
```

```bash
# Check the deployment
kubectl get deployment nginx-deployment
```

**Output:**
```
NAME               READY   UP-TO-DATE   AVAILABLE   AGE
nginx-deployment   3/3     3            3            30s
```

### Updating a Resource

Just modify the YAML file and apply again:

```bash
# Change replicas from 3 to 5 in the YAML file, then:
kubectl apply -f nginx-deployment.yaml
```

**Output:**
```
deployment.apps/nginx-deployment configured
```

Notice it says `configured` (not `created`) — Kubernetes updated the existing resource!

### Applying Multiple Files

```bash
# Apply all YAML files in a directory
kubectl apply -f ./kubernetes/

# Apply multiple specific files
kubectl apply -f deployment.yaml -f service.yaml -f configmap.yaml

# Apply from a URL
kubectl apply -f https://raw.githubusercontent.com/kubernetes/website/main/content/en/examples/pods/simple-pod.yaml
```

### Dry Run — Test Before Applying

```bash
# Check what would happen without actually applying
kubectl apply -f nginx-deployment.yaml --dry-run=client

# Server-side dry run (validates against the actual API server)
kubectl apply -f nginx-deployment.yaml --dry-run=server
```

**Output:**
```
deployment.apps/nginx-deployment created (dry run)
```

Nothing actually changed — it just told you what **would** happen!

### Declarative vs Imperative — Why `apply` Is Preferred

| Approach | Command | Pros | Cons |
|----------|---------|------|------|
| **Declarative** (apply) | `kubectl apply -f file.yaml` | Trackable in Git, repeatable, reviewable | Need to write YAML files |
| **Imperative** (create/run) | `kubectl create deployment nginx --image=nginx` | Quick for testing | Not trackable, not repeatable |

**Rule of thumb:** Use `kubectl apply` for anything that matters. Use imperative commands only for quick throwaway tests.

## 4️⃣ `kubectl delete` — Remove Resources

### Delete a Specific Resource

```bash
# Delete a pod
kubectl delete pod my-nginx

# Delete a deployment
kubectl delete deployment nginx-deployment

# Delete a service
kubectl delete service nginx-service
```

**Output:**
```
pod "my-nginx" deleted
```

### Delete Using a YAML File

This is the cleanest way — delete exactly what you created:

```bash
# Delete everything defined in the YAML file
kubectl delete -f nginx-deployment.yaml
```

### Delete Multiple Resources

```bash
# Delete multiple pods
kubectl delete pod pod1 pod2 pod3

# Delete all pods in the current namespace
kubectl delete pods --all

# Delete all resources in a namespace
kubectl delete all --all -n my-namespace

# Delete all resources from a directory of YAML files
kubectl delete -f ./kubernetes/
```

### Delete by Label

```bash
# Delete all pods with label app=nginx
kubectl delete pods -l app=nginx

# Delete all deployments with environment=test
kubectl delete deployments -l environment=test
```

### Force Delete (Use With Caution!)

Sometimes a pod gets stuck in `Terminating` state. You can force delete it:

```bash
# Force delete a stuck pod
kubectl delete pod stuck-pod --grace-period=0 --force
```

**⚠️ Warning:** Force deleting skips the graceful shutdown process. Only use this when a pod is truly stuck and won't terminate on its own.

## 5️⃣ `kubectl logs` — Read Container Logs

Logs are your **best friend for debugging**. Every application inside a container writes logs, and `kubectl logs` lets you read them.

### Basic Log Reading

```bash
# Get logs of a pod
kubectl logs my-nginx
```

**Output:**
```
/docker-entrypoint.sh: /docker-entrypoint.d/ is not empty, will attempt to perform configuration
/docker-entrypoint.sh: Looking for shell scripts in /docker-entrypoint.d/
10.244.1.1 - - [15/May/2025:10:35:22 +0000] "GET / HTTP/1.1" 200 615 "-" "curl/7.88.1"
10.244.1.1 - - [15/May/2025:10:35:25 +0000] "GET /api HTTP/1.1" 404 153 "-" "curl/7.88.1"
```

### Follow Logs in Real-Time

```bash
# Stream logs as they come (like tail -f)
kubectl logs -f my-nginx
```

Press `Ctrl+C` to stop following.

### Tail — Show Last N Lines

```bash
# Show only the last 50 lines
kubectl logs my-nginx --tail=50

# Show only the last 100 lines
kubectl logs my-nginx --tail=100
```

### Logs from a Specific Time Period

```bash
# Show logs from the last 1 hour
kubectl logs my-nginx --since=1h

# Show logs from the last 30 minutes
kubectl logs my-nginx --since=30m

# Show logs from the last 10 seconds
kubectl logs my-nginx --since=10s

# Show logs since a specific timestamp
kubectl logs my-nginx --since-time="2025-05-15T10:00:00Z"
```

### Logs from a Specific Container (Multi-Container Pods)

If a pod has multiple containers, you need to specify which one:

```bash
# List containers in a pod
kubectl get pod my-app -o jsonpath='{.spec.containers[*].name}'
# Output: app-container sidecar-container

# Get logs from a specific container
kubectl logs my-app -c app-container
kubectl logs my-app -c sidecar-container
```

### Logs from Previous Container Instance

If a container crashed and restarted, you can see the **previous container's logs**:

```bash
# Get logs from the previous (crashed) container
kubectl logs my-app --previous
# OR shorthand:
kubectl logs my-app -p
```

This is **super useful** for debugging `CrashLoopBackOff` — you can see what error caused the crash!

### Logs from All Pods in a Deployment

```bash
# Get logs from all pods matching a label
kubectl logs -l app=nginx

# Follow logs from all pods with a label
kubectl logs -l app=nginx -f

# Show logs from all pods, prefixed with pod name
kubectl logs -l app=nginx --prefix=true
```

**Output with `--prefix`:**
```
[pod/nginx-7c5ddbdf54-abcde/nginx] 10.244.1.1 - - [15/May/2025:10:35:22 +0000] "GET / HTTP/1.1" 200 615
[pod/nginx-7c5ddbdf54-fghij/nginx] 10.244.2.1 - - [15/May/2025:10:35:23 +0000] "GET / HTTP/1.1" 200 615
```

## 6️⃣ `kubectl exec` — Run Commands Inside Containers

`kubectl exec` lets you **get inside a running container** — just like `docker exec`. This is essential for debugging.

### Run a Single Command

```bash
# Run 'ls' inside a pod
kubectl exec my-nginx -- ls /usr/share/nginx/html
```

**Output:**
```
50x.html
index.html
```

**Important:** Notice the `--` before the command. This separates kubectl flags from the command you want to run inside the container.

### Open an Interactive Shell

```bash
# Get a bash shell inside the container
kubectl exec -it my-nginx -- /bin/bash

# If bash isn't available, try sh
kubectl exec -it my-nginx -- /bin/sh
```

**Flags explained:**
- `-i` → Interactive (keep STDIN open)
- `-t` → Allocate a TTY (give you a terminal prompt)

**Output:**
```
root@my-nginx:/# 
```

Now you're **inside the container**! You can poke around:

```bash
root@my-nginx:/# whoami
root

root@my-nginx:/# cat /etc/nginx/nginx.conf
# Shows the Nginx configuration

root@my-nginx:/# curl localhost:80
# Shows the web page being served

root@my-nginx:/# ls -la /var/log/nginx/
# Check Nginx logs inside the container

root@my-nginx:/# exit    # Exit back to your terminal
```

### Exec into a Specific Container (Multi-Container Pods)

```bash
# Specify which container to exec into
kubectl exec -it my-app -c sidecar-container -- /bin/sh
```

### Common Debugging Commands Inside Containers

```bash
# Check DNS resolution
kubectl exec -it my-nginx -- nslookup kubernetes.default

# Check network connectivity
kubectl exec -it my-nginx -- curl -s http://other-service:8080/health

# Check environment variables
kubectl exec -it my-nginx -- env

# Check mounted files
kubectl exec -it my-nginx -- cat /etc/config/app.conf

# Check the running processes
kubectl exec -it my-nginx -- ps aux

# Check disk usage
kubectl exec -it my-nginx -- df -h
```

## 7️⃣ More Essential kubectl Commands

### `kubectl create` — Quick Imperative Creation

```bash
# Create a deployment
kubectl create deployment nginx --image=nginx

# Create a namespace
kubectl create namespace my-app

# Create a configmap from literal values
kubectl create configmap my-config --from-literal=key1=value1 --from-literal=key2=value2

# Create a secret from literal values
kubectl create secret generic my-secret --from-literal=password=mysecretpassword

# Create a service (expose a deployment)
kubectl expose deployment nginx --port=80 --type=NodePort
```

### `kubectl edit` — Edit a Resource Live

```bash
# Opens the resource in your default editor (vi/nano/code)
kubectl edit deployment nginx-deployment
```

This opens the **full YAML** of the resource in your editor. Make changes, save, and close — Kubernetes applies the changes immediately.

**Tip:** Set your preferred editor:
```bash
export KUBE_EDITOR="nano"    # Use nano instead of vi
# OR
export KUBE_EDITOR="code --wait"    # Use VS Code
```

### `kubectl scale` — Scale Up or Down

```bash
# Scale a deployment to 5 replicas
kubectl scale deployment nginx-deployment --replicas=5

# Scale down to 1 replica
kubectl scale deployment nginx-deployment --replicas=1

# Scale to zero (stops all pods but keeps the deployment)
kubectl scale deployment nginx-deployment --replicas=0
```

### `kubectl port-forward` — Access Services Locally

This forwards a port from your machine to a pod/service in the cluster:

```bash
# Forward local port 8080 to pod port 80
kubectl port-forward pod/my-nginx 8080:80

# Forward to a service
kubectl port-forward service/nginx-service 8080:80

# Forward to a deployment
kubectl port-forward deployment/nginx-deployment 8080:80
```

Now open `http://localhost:8080` in your browser — you're accessing the pod directly!

**Note:** This only works while the command is running. Press `Ctrl+C` to stop.

### `kubectl cp` — Copy Files To/From Containers

```bash
# Copy a file FROM a container to your local machine
kubectl cp my-nginx:/var/log/nginx/access.log ./access.log

# Copy a file TO a container from your local machine
kubectl cp ./config.json my-nginx:/app/config.json

# Copy from a specific container in a multi-container pod
kubectl cp my-app:/logs/app.log ./app.log -c app-container
```

### `kubectl top` — Resource Usage (Requires Metrics Server)

```bash
# Check CPU and memory usage of nodes
kubectl top nodes

# Output:
# NAME       CPU(cores)   CPU%   MEMORY(bytes)   MEMORY%
# worker-1   250m         12%    1024Mi           51%
# worker-2   180m         9%     768Mi            38%

# Check CPU and memory usage of pods
kubectl top pods

# Check all pods in all namespaces, sorted by CPU
kubectl top pods -A --sort-by=cpu

# Check a specific pod
kubectl top pod my-nginx
```

**Note:** You need the **Metrics Server** installed for `kubectl top` to work:
```bash
kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml
```

### `kubectl rollout` — Manage Deployments Updates

```bash
# Check rollout status
kubectl rollout status deployment/nginx-deployment

# View rollout history
kubectl rollout history deployment/nginx-deployment

# Undo the last rollout (rollback!)
kubectl rollout undo deployment/nginx-deployment

# Rollback to a specific revision
kubectl rollout undo deployment/nginx-deployment --to-revision=2

# Pause a rollout (for canary testing)
kubectl rollout pause deployment/nginx-deployment

# Resume a paused rollout
kubectl rollout resume deployment/nginx-deployment

# Restart a deployment (rolling restart of all pods)
kubectl rollout restart deployment/nginx-deployment
```

## 8️⃣ Helpful Tips and Tricks

### Auto-Completion (HUGE Time Saver!)

```bash
# For Bash:
echo 'source <(kubectl completion bash)' >> ~/.bashrc
source ~/.bashrc

# For Zsh:
echo 'source <(kubectl completion zsh)' >> ~/.zshrc
source ~/.zshrc

# For Fish:
kubectl completion fish | source
```

Now you can press `Tab` to auto-complete commands, resource names, and flags!

### Create an Alias (Because Typing kubectl Is Tiring)

```bash
# Add to your .bashrc or .zshrc
alias k='kubectl'
alias kgp='kubectl get pods'
alias kgs='kubectl get svc'
alias kgd='kubectl get deployments'
alias kga='kubectl get all'
alias kaf='kubectl apply -f'
alias kdf='kubectl delete -f'
alias kdp='kubectl describe pod'
alias kl='kubectl logs'
alias kli='kubectl logs -f'
alias ke='kubectl exec -it'
```

Now instead of:
```bash
kubectl get pods --all-namespaces
```

You type:
```bash
k get pods -A
```

### Generate YAML Templates (Instead of Writing from Scratch)

Don't remember the YAML format? Let kubectl generate it for you!

```bash
# Generate a pod YAML (without creating it)
kubectl run my-nginx --image=nginx --dry-run=client -o yaml > pod.yaml

# Generate a deployment YAML
kubectl create deployment nginx --image=nginx --dry-run=client -o yaml > deployment.yaml

# Generate a service YAML
kubectl expose deployment nginx --port=80 --type=NodePort --dry-run=client -o yaml > service.yaml

# Generate a job YAML
kubectl create job my-job --image=busybox --dry-run=client -o yaml -- echo "Hello World" > job.yaml
```

This is **incredibly useful** for the CKA exam and day-to-day work!

## 9️⃣ Real-World kubectl Workflow Example

Let's put it all together with a real scenario:

```bash
# 1. Create a namespace for your app
kubectl create namespace my-app

# 2. Deploy your application
kubectl apply -f deployment.yaml -n my-app

# 3. Check if pods are running
kubectl get pods -n my-app

# 4. Check deployment status
kubectl rollout status deployment/my-app -n my-app

# 5. Check logs to make sure the app started correctly
kubectl logs -l app=my-app -n my-app

# 6. If something's wrong, describe the pod
kubectl describe pod <pod-name> -n my-app

# 7. If you need to debug inside the container
kubectl exec -it <pod-name> -n my-app -- /bin/sh

# 8. Expose the app as a service
kubectl apply -f service.yaml -n my-app

# 9. Port-forward to test locally
kubectl port-forward service/my-app-service 8080:80 -n my-app

# 10. Scale up if everything looks good
kubectl scale deployment my-app --replicas=5 -n my-app
```

## 📝 Key Takeaways

| Takeaway | Details |
|----------|---------|
| **`kubectl get`** | Your go-to for listing resources — use `-o wide`, `-A`, `-l` |
| **`kubectl describe`** | Deep dive into a resource — **Events section is gold for debugging** |
| **`kubectl apply -f`** | The declarative way — always prefer this over imperative commands |
| **`kubectl delete`** | Remove resources — use `-f` to delete what you created |
| **`kubectl logs`** | Read container logs — use `-f` to follow, `--previous` for crashed containers |
| **`kubectl exec -it`** | Get inside a container — essential for debugging |
| **`--dry-run=client -o yaml`** | Generate YAML templates — saves so much time! |
| **Set up auto-completion** | Seriously, do this first — it's a massive time saver |
| **Use aliases** | `alias k='kubectl'` at minimum |
