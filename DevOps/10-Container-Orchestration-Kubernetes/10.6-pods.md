# 10.6 — Pods — The Smallest Unit (Single & Multi-Container Pods, Init Containers)

> **One Line Answer:** A **Pod** is the **smallest deployable unit in Kubernetes** — it's a wrapper around one or more containers that share the same network (IP address) and storage, and are always scheduled together on the same node. Think of a pod as a **tiny apartment** where containers live together as roommates.

## 🤔 The Big Picture — What Is a Pod?

In Docker, the basic unit is a **container**. In Kubernetes, the basic unit is a **Pod**.

But wait — why not just use containers directly? Why wrap them in a Pod?

Because Kubernetes needs a **higher-level abstraction** that can:
- Group related containers together
- Give them a **shared network** (same IP address)
- Give them **shared storage** (volumes)
- Manage their **lifecycle** together (start, stop, restart)

```
┌──────────────────────────────────────────────────────────────┐
│                     Docker World                              │
│                                                               │
│   Container 1      Container 2      Container 3              │
│   (isolated)       (isolated)       (isolated)               │
│   IP: 172.17.0.2   IP: 172.17.0.3   IP: 172.17.0.4          │
│                                                               │
│   Each container = its own little island                      │
└──────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────┐
│                    Kubernetes World                            │
│                                                               │
│   ┌────────────────────────────────┐  ┌────────────────────┐  │
│   │          Pod 1                 │  │      Pod 2         │  │
│   │  ┌────────┐  ┌────────┐       │  │  ┌────────┐        │  │
│   │  │ App    │  │ Sidecar│       │  │  │  App   │        │  │
│   │  │ Cont.  │  │ Cont.  │       │  │  │ Cont.  │        │  │
│   │  └────────┘  └────────┘       │  │  └────────┘        │  │
│   │  Shared IP: 10.244.1.5        │  │  IP: 10.244.1.6    │  │
│   │  Shared Volumes               │  │                    │  │
│   └────────────────────────────────┘  └────────────────────┘  │
│                                                               │
│   Containers in the same pod = roommates                      │
└──────────────────────────────────────────────────────────────┘
```

### The Apartment Analogy 🏠

Think of a Pod as an **apartment**:
- The **apartment** is the Pod
- The **roommates** are the containers
- All roommates share the same **address** (IP address)
- All roommates share the same **kitchen and bathroom** (volumes)
- Roommates can talk to each other by just **walking to the next room** (localhost)
- But to talk to people in **other apartments** (other pods), they need to use the **phone** (network)

## 📋 Pod Key Facts

| Fact | Details |
|------|---------|
| **Smallest unit** | You can't deploy a single container in K8s — it's always wrapped in a Pod |
| **One IP per pod** | All containers in a pod share the same IP address |
| **Localhost** | Containers in the same pod communicate via `localhost` |
| **Shared volumes** | Containers in the same pod can share storage volumes |
| **Same node** | All containers in a pod always run on the **same node** |
| **Ephemeral** | Pods are **temporary** — they can be killed and recreated anytime |
| **No self-healing** | A standalone pod won't restart if the node dies (use Deployments for that!) |
| **Unique name** | Each pod has a unique name within its namespace |

## 1️⃣ Single-Container Pods — The Most Common Pattern

The **vast majority** of pods you'll create have just **one container**. This is the most common pattern.

### Creating a Pod — The Imperative Way (Quick & Dirty)

```bash
# Create a pod quickly using kubectl run
kubectl run my-nginx --image=nginx:1.27
```

**Output:**
```
pod/my-nginx created
```

```bash
# Check it
kubectl get pods
```

**Output:**
```
NAME       READY   STATUS    RESTARTS   AGE
my-nginx   1/1     Running   0          10s
```

### Creating a Pod — The Declarative Way (The Right Way)

```yaml
# nginx-pod.yaml
apiVersion: v1                  # Which API version to use (v1 for Pods)
kind: Pod                       # What type of resource (Pod)
metadata:                       # Info about the pod
  name: my-nginx                # Pod name (must be unique in the namespace)
  labels:                       # Key-value tags for organizing/selecting
    app: nginx
    environment: dev
    tier: frontend
spec:                           # The specification — what you want
  containers:                   # List of containers in this pod
    - name: nginx               # Container name
      image: nginx:1.27         # Docker image to use
      ports:
        - containerPort: 80     # Port the app listens on
```

Let's break down **every single field**:

| Field | What It Means |
|-------|---------------|
| `apiVersion: v1` | Pods use the core `v1` API. Different resources use different API versions (e.g., Deployments use `apps/v1`) |
| `kind: Pod` | We're creating a Pod resource |
| `metadata.name` | The unique name of this pod. Used in all kubectl commands |
| `metadata.labels` | Key-value tags. These are used for selecting, filtering, and connecting resources together |
| `spec.containers` | A list of containers to run in this pod. Most pods have just 1 container |
| `containers[].name` | Name of this specific container within the pod |
| `containers[].image` | The Docker/OCI image to pull and run |
| `containers[].ports` | Ports the container exposes. This is **informational only** — it doesn't actually open the port (Services do that) |

```bash
# Apply it
kubectl apply -f nginx-pod.yaml

# Check it
kubectl get pods
```

### Pod with More Configuration

```yaml
# app-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
  labels:
    app: my-app
    version: "1.0"
spec:
  containers:
    - name: app
      image: my-app:1.0
      ports:
        - containerPort: 8080
          name: http                # Give the port a name
          protocol: TCP             # TCP (default) or UDP
      
      # Environment variables
      env:
        - name: DATABASE_HOST
          value: "postgres-service"
        - name: DATABASE_PORT
          value: "5432"
        - name: APP_MODE
          value: "production"
      
      # Resource requests and limits
      resources:
        requests:                   # Minimum resources guaranteed
          cpu: "100m"               # 0.1 CPU cores (100 millicores)
          memory: "128Mi"           # 128 Mebibytes of RAM
        limits:                     # Maximum resources allowed
          cpu: "500m"               # 0.5 CPU cores
          memory: "256Mi"           # 256 Mebibytes of RAM
      
      # Liveness probe — Is the container alive?
      livenessProbe:
        httpGet:
          path: /health
          port: 8080
        initialDelaySeconds: 15     # Wait 15 seconds before first check
        periodSeconds: 10           # Check every 10 seconds
      
      # Readiness probe — Is the container ready to receive traffic?
      readinessProbe:
        httpGet:
          path: /ready
          port: 8080
        initialDelaySeconds: 5
        periodSeconds: 5
  
  # Restart policy
  restartPolicy: Always             # Always | OnFailure | Never
```

Let's understand the new fields:

**Environment Variables (`env`):**
These are key-value pairs passed to the container — like configuration settings. The app reads them using `os.getenv("DATABASE_HOST")` (in Python) or `process.env.DATABASE_HOST` (in Node.js).

**Resources (`resources`):**
- `requests` = "I need **at least** this much." The scheduler uses this to find a node with enough room.
- `limits` = "Don't let me use **more** than this." If the container exceeds memory limits, it gets killed (OOMKilled).

```
CPU units:
  1 CPU   = 1000m (millicores)
  500m    = 0.5 CPU
  100m    = 0.1 CPU
  250m    = 0.25 CPU

Memory units:
  1 Gi    = 1024 Mi (Gibibytes → Mebibytes)
  128Mi   = 128 Mebibytes
  256Mi   = 256 Mebibytes
  1Gi     = 1024 Mebibytes
```

**Liveness Probe:**
Kubernetes constantly checks if your container is **alive and healthy**. If the liveness probe fails, Kubernetes **kills the container and restarts it**.

Think of it like a doctor checking your heartbeat — if no heartbeat → emergency restart!

**Readiness Probe:**
Checks if the container is **ready to serve traffic**. Even if the container is alive, it might not be ready (e.g., still loading data, waiting for database connection). If the readiness probe fails, Kubernetes **stops sending traffic** to that pod.

**Restart Policy:**
- `Always` — Always restart the container when it stops (default — used for long-running services)
- `OnFailure` — Only restart if the container exited with an error (used for Jobs)
- `Never` — Never restart (used for one-off tasks)

## 2️⃣ Pod Lifecycle — From Birth to Death

Every pod goes through a lifecycle:

```
┌─────────────────────────────────────────────────────────────┐
│                    Pod Lifecycle                              │
│                                                              │
│   Pending ──→ Running ──→ Succeeded                          │
│      │            │                                          │
│      │            └──→ Failed                                │
│      │                                                       │
│      └──→ Failed (can't schedule, can't pull image, etc.)    │
└─────────────────────────────────────────────────────────────┘
```

### Pod Phases

| Phase | What It Means |
|-------|---------------|
| **Pending** | Pod accepted by K8s, but containers not running yet. Could be downloading image, waiting for a node, etc. |
| **Running** | Pod bound to a node, at least one container is running |
| **Succeeded** | All containers finished successfully (exit code 0). Common for Jobs |
| **Failed** | All containers terminated, at least one failed (non-zero exit code) |
| **Unknown** | Pod state can't be determined (usually a node communication issue) |

### Common Pod Statuses You'll See

```bash
kubectl get pods
```

| Status | Meaning | What to Do |
|--------|---------|------------|
| `Running` | Everything is fine! | Nothing — you're good! |
| `Pending` | Waiting for something | Check with `kubectl describe` — might be insufficient resources |
| `ContainerCreating` | Image is being pulled | Wait, or check image name if it takes too long |
| `CrashLoopBackOff` | Container keeps crashing and restarting | Check logs: `kubectl logs <pod> --previous` |
| `ImagePullBackOff` | Can't pull the container image | Check image name, tag, registry credentials |
| `ErrImagePull` | Failed to pull the image | Same as above |
| `Terminating` | Pod is being deleted | Wait, or force delete if stuck |
| `OOMKilled` | Out of memory — killed by K8s | Increase memory limits |
| `Error` | Container exited with an error | Check logs |
| `Init:0/1` | Init container running | Wait for init containers to complete |
| `Evicted` | Node ran out of resources | Check node resources, set proper resource requests |

### Debugging Pod Issues

```bash
# Step 1: Check the pod status
kubectl get pod my-app

# Step 2: Describe the pod (look at Events section!)
kubectl describe pod my-app

# Step 3: Check logs
kubectl logs my-app

# Step 4: If it crashed, check previous logs
kubectl logs my-app --previous

# Step 5: If you need to investigate inside the container
kubectl exec -it my-app -- /bin/sh
```

## 3️⃣ Multi-Container Pods — Containers Living Together

While most pods have one container, sometimes you need **multiple containers in the same pod**. These containers work **tightly together** as a team.

### Why Multi-Container Pods?

There are specific patterns where it makes sense:

```
┌──────────────────────────────────────────────────────────────┐
│                     Multi-Container Pod                       │
│                                                               │
│   ┌────────────┐    ┌────────────┐    ┌────────────┐          │
│   │    Main     │    │  Sidecar   │    │    Log     │          │
│   │    App      │    │  Proxy     │    │  Shipper   │          │
│   │  Container  │    │ Container  │    │ Container  │          │
│   │             │    │            │    │            │          │
│   │  Port:8080  │    │  Port:9090 │    │  No port   │          │
│   └──────┬──────┘    └──────┬─────┘    └──────┬─────┘          │
│          │                  │                  │               │
│          └──────────────────┴──────────────────┘               │
│                   All share: IP, Volumes                       │
│                   Talk via: localhost                           │
│                   Pod IP: 10.244.1.5                           │
└──────────────────────────────────────────────────────────────┘
```

### Multi-Container Design Patterns

There are three main patterns for multi-container pods:

#### Pattern 1: Sidecar Pattern

The **sidecar** enhances or extends the main container's functionality. Think of it like a motorcycle sidecar — it rides alongside the main container.

**Common use cases:**
- Log collection (Fluentd/Filebeat sidecar collects logs from the main app)
- Monitoring (Prometheus exporter sidecar)
- Configuration syncing (sidecar watches for config changes)
- Service mesh proxy (Envoy sidecar — used by Istio)

```yaml
# sidecar-example.yaml
apiVersion: v1
kind: Pod
metadata:
  name: web-with-logging
  labels:
    app: web
spec:
  containers:
    # Main application container
    - name: web-app
      image: nginx:1.27
      ports:
        - containerPort: 80
      volumeMounts:
        - name: shared-logs          # Mount the shared volume
          mountPath: /var/log/nginx   # Nginx writes logs here

    # Sidecar container — ships logs to a central system
    - name: log-shipper
      image: busybox:1.36
      command: ['sh', '-c', 'tail -F /logs/access.log']   # Reads logs
      volumeMounts:
        - name: shared-logs          # Same shared volume!
          mountPath: /logs            # Reads from here

  volumes:
    - name: shared-logs              # Define the shared volume
      emptyDir: {}                   # Temporary storage — deleted when pod dies
```

**How it works:**
1. The `web-app` container (Nginx) writes access logs to `/var/log/nginx/`
2. Both containers share the `shared-logs` volume
3. The `log-shipper` sidecar reads logs from `/logs/` (same data!)
4. The sidecar can then forward these logs to a logging system (ELK, CloudWatch, etc.)

```bash
# Apply it
kubectl apply -f sidecar-example.yaml

# Check both containers are running
kubectl get pod web-with-logging
```

**Output:**
```
NAME               READY   STATUS    RESTARTS   AGE
web-with-logging   2/2     Running   0          30s
```

Notice **2/2** — both containers are ready!

```bash
# Check logs of the sidecar
kubectl logs web-with-logging -c log-shipper

# Check logs of the main app
kubectl logs web-with-logging -c web-app
```

#### Pattern 2: Ambassador Pattern

The **ambassador** acts as a proxy for the main container. It handles outbound connections, abstracts away complexity.

**Common use cases:**
- Database connection pooling
- API gateway
- Rate limiting outbound requests

```yaml
# ambassador-example.yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-with-proxy
spec:
  containers:
    # Main app talks to localhost:6379 thinking it's Redis
    - name: app
      image: my-app:1.0
      env:
        - name: REDIS_HOST
          value: "localhost"       # Talks to the ambassador!
        - name: REDIS_PORT
          value: "6379"

    # Ambassador: proxies Redis connections to the right cluster
    - name: redis-proxy
      image: redis-proxy:1.0
      ports:
        - containerPort: 6379
      env:
        - name: REDIS_CLUSTER_HOST
          value: "redis-cluster.production.svc.cluster.local"
```

**How it works:**
1. The main app connects to Redis on `localhost:6379`
2. The ambassador container listens on port 6379
3. The ambassador forwards the connection to the actual Redis cluster
4. The main app doesn't know (or care) about the real Redis location

#### Pattern 3: Adapter Pattern

The **adapter** transforms or standardizes the main container's output.

**Common use cases:**
- Converting log formats (e.g., app logs → JSON format for logging system)
- Exposing metrics in Prometheus format
- Data transformation

```yaml
# adapter-example.yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-with-adapter
spec:
  containers:
    # Main app writes logs in a custom format
    - name: app
      image: my-legacy-app:1.0
      volumeMounts:
        - name: app-logs
          mountPath: /var/log/app

    # Adapter: converts log format to JSON
    - name: log-adapter
      image: log-transformer:1.0
      volumeMounts:
        - name: app-logs
          mountPath: /var/log/input     # Reads raw logs
      env:
        - name: OUTPUT_FORMAT
          value: "json"

  volumes:
    - name: app-logs
      emptyDir: {}
```

### How Containers in the Same Pod Communicate

```
┌──────────────────────────────────────────────────────┐
│                        Pod                            │
│                                                       │
│   ┌──────────┐                      ┌──────────┐     │
│   │ Container│  ←── localhost ───→   │ Container│     │
│   │   App    │       :8080           │  Sidecar │     │
│   │ Port:8080│                       │ Port:9090│     │
│   └──────────┘                      └──────────┘     │
│                                                       │
│   App can call: curl http://localhost:9090             │
│   Sidecar can call: curl http://localhost:8080         │
│                                                       │
│   They share:                                         │
│   ✅ Same IP address (10.244.1.5)                     │
│   ✅ Same network namespace                           │
│   ✅ Same volumes (if mounted)                        │
│   ✅ Same node                                        │
│                                                       │
│   They DON'T share:                                   │
│   ❌ Filesystem (each has its own, unless volumes)     │
│   ❌ Process namespace (by default)                    │
│   ❌ Hostname (same hostname though)                   │
└──────────────────────────────────────────────────────┘
```

**Key point:** Containers in the same pod use `localhost` to talk to each other, but they must use **different ports** (just like two applications on the same computer can't both listen on port 8080).

## 4️⃣ Init Containers — Do Something Before the Main App Starts

### What Are Init Containers?

Init containers are **special containers that run before the main app containers**. They run to completion (they must finish and exit successfully) before the main containers start.

Think of init containers like a **setup crew** at a concert:
1. The setup crew arrives first and sets up the stage, lights, and sound
2. Only after the setup is complete, the band (main containers) starts performing
3. The setup crew leaves — they're done

```
┌──────────────────────────────────────────────────────────────┐
│                   Pod Startup Sequence                        │
│                                                               │
│   Time ──────────────────────────────────────────────────→    │
│                                                               │
│   ┌──────────┐  ┌──────────┐  ┌─────────────────────────┐    │
│   │  Init    │  │  Init    │  │     Main Containers     │    │
│   │ Cont. 1  │→ │ Cont. 2  │→ │                         │    │
│   │ (runs &  │  │ (runs &  │  │  App    Sidecar         │    │
│   │  exits)  │  │  exits)  │  │  (runs forever)         │    │
│   └──────────┘  └──────────┘  └─────────────────────────┘    │
│                                                               │
│   Init containers run ONE AT A TIME, in order.                │
│   Main containers start ONLY after ALL inits succeed.         │
└──────────────────────────────────────────────────────────────┘
```

### Why Use Init Containers?

| Use Case | Example |
|----------|---------|
| **Wait for a dependency** | Wait until the database is ready before starting the app |
| **Clone a repo** | Pull code from Git before the main app uses it |
| **Download config** | Fetch configuration files from a remote source |
| **Run migrations** | Run database migrations before the app starts |
| **Set up permissions** | Change file ownership/permissions before the app runs |
| **Register with service** | Register the pod with a service discovery system |

### Example 1: Wait for a Database

This is the **most common** use case — your app needs a database, but the database might not be ready yet.

```yaml
# app-with-init.yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
  labels:
    app: my-app
spec:
  # Init containers — run before main containers
  initContainers:
    - name: wait-for-db
      image: busybox:1.36
      command: ['sh', '-c']
      args:
        - |
          echo "Waiting for PostgreSQL to be ready..."
          until nc -z postgres-service 5432; do
            echo "PostgreSQL is not ready yet — sleeping 2 seconds"
            sleep 2
          done
          echo "PostgreSQL is ready! Starting the app..."

  # Main containers — start after init containers finish
  containers:
    - name: app
      image: my-app:1.0
      ports:
        - containerPort: 8080
      env:
        - name: DATABASE_URL
          value: "postgresql://user:pass@postgres-service:5432/mydb"
```

**What happens step by step:**
1. Kubernetes creates the pod
2. The `wait-for-db` init container starts
3. It keeps checking if PostgreSQL is accepting connections on port 5432
4. Once PostgreSQL is ready, the init container exits with code 0 (success)
5. Kubernetes starts the `app` main container
6. The app connects to the database — and it's already there! No connection errors!

```bash
# Apply and watch
kubectl apply -f app-with-init.yaml

# Check the pod status — you'll see the init container phase
kubectl get pod my-app
```

**Output (while init container is running):**
```
NAME     READY   STATUS     RESTARTS   AGE
my-app   0/1     Init:0/1   0          5s
```

`Init:0/1` means: 0 out of 1 init containers have completed.

**Output (after init container finishes):**
```
NAME     READY   STATUS    RESTARTS   AGE
my-app   1/1     Running   0          15s
```

### Example 2: Clone a Git Repository

```yaml
# app-with-git-clone.yaml
apiVersion: v1
kind: Pod
metadata:
  name: web-app
spec:
  initContainers:
    - name: git-clone
      image: alpine/git:latest
      command: ['git', 'clone', 'https://github.com/example/my-website.git', '/repo']
      volumeMounts:
        - name: website-content
          mountPath: /repo

  containers:
    - name: nginx
      image: nginx:1.27
      ports:
        - containerPort: 80
      volumeMounts:
        - name: website-content
          mountPath: /usr/share/nginx/html    # Serve the cloned website!

  volumes:
    - name: website-content
      emptyDir: {}
```

**How it works:**
1. The `git-clone` init container clones a repository into a shared volume
2. After cloning completes, the init container exits
3. Nginx starts and serves the cloned content from the shared volume

### Example 3: Multiple Init Containers (Run in Order)

```yaml
# multi-init-example.yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  initContainers:
    # Init 1: Wait for database
    - name: wait-for-db
      image: busybox:1.36
      command: ['sh', '-c', 'until nc -z postgres-service 5432; do sleep 2; done']

    # Init 2: Wait for cache
    - name: wait-for-cache
      image: busybox:1.36
      command: ['sh', '-c', 'until nc -z redis-service 6379; do sleep 2; done']

    # Init 3: Run database migrations
    - name: run-migrations
      image: my-app:1.0
      command: ['python', 'manage.py', 'migrate']
      env:
        - name: DATABASE_URL
          value: "postgresql://user:pass@postgres-service:5432/mydb"

  containers:
    - name: app
      image: my-app:1.0
      ports:
        - containerPort: 8080
```

**Execution order:**
1. `wait-for-db` runs → waits until PostgreSQL is ready → exits
2. `wait-for-cache` runs → waits until Redis is ready → exits
3. `run-migrations` runs → applies database schema changes → exits
4. `app` main container starts → the app is ready with all dependencies!

### Init Containers vs Regular Containers

| Feature | Init Containers | Regular Containers |
|---------|----------------|-------------------|
| **When they run** | Before main containers | After init containers |
| **Run to completion** | Yes — must exit with code 0 | No — run continuously |
| **Run in order** | One at a time, sequentially | All start together (in parallel) |
| **If one fails** | Pod restarts (init containers re-run) | Depends on restartPolicy |
| **Probes** | No liveness/readiness probes | Yes — probes are supported |
| **Resources** | Can have different resource limits | Have their own limits |

### What Happens If an Init Container Fails?

```
Init Container fails (exit code ≠ 0)
    ↓
Kubernetes restarts the entire init sequence (from the beginning!)
    ↓
If restartPolicy = Always or OnFailure → keeps retrying
    ↓
If restartPolicy = Never → pod goes to Failed state
```

```bash
# Check why an init container failed
kubectl describe pod my-app
# Look at the Events section and Init Containers section

# Check init container logs
kubectl logs my-app -c wait-for-db
```

## 5️⃣ Pod DNS and Networking

Every pod gets its own IP address and can communicate with other pods.

### How Pods Communicate

```
┌───────────────────────────────────────────────────────────────┐
│                    Kubernetes Cluster                          │
│                                                                │
│   ┌────────────────────┐         ┌────────────────────┐        │
│   │      Pod A         │         │      Pod B         │        │
│   │  IP: 10.244.1.5    │  ←───→  │  IP: 10.244.2.3    │        │
│   │  App on port 8080  │  network│  App on port 3000  │        │
│   └────────────────────┘         └────────────────────┘        │
│                                                                │
│   Pod A can reach Pod B: curl http://10.244.2.3:3000           │
│   Pod B can reach Pod A: curl http://10.244.1.5:8080           │
│                                                                │
│   ⚠️ But don't use Pod IPs directly!                           │
│   Pods are ephemeral — IPs change when pods restart!           │
│   Use Services instead! (covered in 10.12)                     │
└───────────────────────────────────────────────────────────────┘
```

### Pod DNS

Each pod can resolve other **Services** by name:

```bash
# From inside a pod, you can reach a service by its name:

# Or the full DNS name:

```

## 8️⃣ Probes Explained — Kubernetes Health Checks

Probes are one of the most important concepts for production pods. There are three types:

```
┌─────────────────────────────────────────────────────────────┐
│                    The Three Probes                           │
│                                                              │
│   ┌──────────────┐  ┌──────────────┐  ┌──────────────┐      │
│   │   Startup    │  │   Liveness   │  │  Readiness   │      │
│   │    Probe     │→ │    Probe     │→ │    Probe     │      │
│   │              │  │              │  │              │      │
│   │ "Has the app │  │ "Is the app  │  │ "Is the app  │      │
│   │  started     │  │  still       │  │  ready for   │      │
│   │  yet?"       │  │  alive?"     │  │  traffic?"   │      │
│   │              │  │              │  │              │      │
│   │ If fails:    │  │ If fails:    │  │ If fails:    │      │
│   │ Kill+restart │  │ Kill+restart │  │ Stop sending │      │
│   │              │  │              │  │ traffic      │      │
│   └──────────────┘  └──────────────┘  └──────────────┘      │
│                                                              │
│   startup runs first, then liveness & readiness run together │
└─────────────────────────────────────────────────────────────┘
```

### Probe Types (How to Check)

There are three ways Kubernetes can check health:

**1. HTTP GET Probe** (Most common for web apps)
```yaml
livenessProbe:
  httpGet:
    path: /healthz        # Endpoint to call
    port: 8080            # Port to call on
    httpHeaders:          # Optional custom headers
      - name: Custom-Header
        value: my-value
  initialDelaySeconds: 15
  periodSeconds: 10
```
Kubernetes sends an HTTP GET request. If it returns a status code between 200-399, it's healthy.

**2. TCP Socket Probe** (For non-HTTP services like databases)
```yaml
livenessProbe:
  tcpSocket:
    port: 3306            # Just check if the port is open
  initialDelaySeconds: 15
  periodSeconds: 10
```
Kubernetes tries to open a TCP connection. If it succeeds, the container is healthy.

**3. Exec Probe** (Run a command inside the container)
```yaml
livenessProbe:
  exec:
    command:
      - cat
      - /tmp/healthy      # If this file exists, it's healthy
  initialDelaySeconds: 5
  periodSeconds: 5
```
Kubernetes runs the command inside the container. If it exits with code 0, the container is healthy.

### Probe Parameters Explained

```yaml
livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
  initialDelaySeconds: 15    # Wait 15s after container starts before first probe
  periodSeconds: 10          # Check every 10 seconds
  timeoutSeconds: 3          # If no response in 3s, consider it failed
  successThreshold: 1        # How many successes to be considered healthy (liveness always 1)
  failureThreshold: 3        # After 3 failures, take action (restart or remove from service)
```

## 9️⃣ Pods vs Deployments — When to Use What

**You should almost NEVER create standalone pods in production!** Here's why:

| Feature | Standalone Pod | Deployment |
|---------|---------------|------------|
| Self-healing | ❌ Pod dies → gone forever | ✅ Pod dies → new one created |
| Scaling | ❌ Manual creation of each pod | ✅ `kubectl scale --replicas=5` |
| Rolling updates | ❌ No update mechanism | ✅ Zero-downtime updates |
| Rollback | ❌ No rollback | ✅ `kubectl rollout undo` |
| Node failure | ❌ Pod lost if node dies | ✅ Rescheduled to another node |

**Rule of thumb:**
- Use **standalone pods** only for testing, debugging, or one-off tasks
- Use **Deployments** (which manage pods for you) for everything else

```bash
# ❌ Don't do this in production:
kubectl run my-app --image=my-app:1.0

# ✅ Do this instead — create a Deployment:
kubectl create deployment my-app --image=my-app:1.0 --replicas=3
```

We'll cover Deployments in detail in **10.8 — Deployments**.

## 📝 Key Takeaways

| Takeaway | Details |
|----------|---------|
| **Pod = smallest unit** | You can't deploy a single container in K8s — it's always in a pod |
| **One IP per pod** | All containers share the same IP and can talk via `localhost` |
| **Single-container pods** | The most common pattern — 90% of your pods |
| **Multi-container pods** | Sidecar, Ambassador, Adapter patterns for tightly coupled containers |
| **Init containers** | Run before main containers — for setup, waiting on dependencies |
| **Pods are ephemeral** | They can die anytime — don't treat them as permanent |
| **Use Deployments, not bare Pods** | Deployments give you self-healing, scaling, and rolling updates |
| **Probes are essential** | Liveness, Readiness, and Startup probes keep your apps healthy |
| **Labels are how you organize** | Used for selection, filtering, and connecting resources |
