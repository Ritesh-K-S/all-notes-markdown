# 10.8 — Deployments — Rolling Updates, Rollbacks, Scaling

> **One Line Answer:** A Deployment is the **most commonly used Kubernetes controller** — it manages ReplicaSets for you and gives you superpowers like **rolling updates** (update without downtime), **rollbacks** (go back to a previous version), and **easy scaling** — all with a single YAML file.

---

## 🤔 Why Do We Need Deployments? — The Problem With ReplicaSets Alone

In the previous chapter (10.7), we learned that a ReplicaSet keeps a fixed number of Pods running. Great! But it has a **big limitation**:

> **When you update the Pod template (e.g., change the container image from `nginx:1.25` to `nginx:1.26`), the ReplicaSet does NOT update existing Pods.**

It only uses the new template for *future* Pod creations. Your old Pods just keep running with the old image forever.

So how do you **update your application** without downtime?

**Manual approach (painful):**
1. Delete all old Pods
2. Wait for the ReplicaSet to create new ones with the new image
3. Meanwhile your app is **completely DOWN** 💀

**Or:**
1. Create a new ReplicaSet with the new image
2. Slowly scale up the new RS while scaling down the old RS
3. Delete the old RS when done
4. Do all of this manually, hoping you don't mess up

This is tedious, error-prone, and scary. Enter: **Deployments!**

---

## 📦 What is a Deployment?

A **Deployment** is a higher-level Kubernetes object that **manages ReplicaSets automatically**. It gives you:

| Feature | What It Does |
|---------|-------------|
| **ReplicaSet Management** | Creates and manages ReplicaSets for you (you never touch RS directly) |
| **Rolling Updates** | Updates your app gradually — old Pods go down as new Pods come up — **zero downtime** |
| **Rollbacks** | Messed up an update? One command and you're back to the previous version |
| **Scaling** | Scale up or down easily |
| **Revision History** | Keeps track of every update as a "revision" — like Git history for your Pods |
| **Pause & Resume** | Pause a rollout, make multiple changes, then resume |

### The Hierarchy:

```
  You (Human)
    │
    ▼
  Deployment        ← You create THIS
    │
    ▼
  ReplicaSet        ← Deployment creates THIS automatically
    │
    ▼
  Pods (x3)         ← ReplicaSet creates THESE automatically
```

**You only interact with the Deployment. Everything else happens automatically.**

## 📝 Writing a Deployment YAML — Step by Step

Let's create a Deployment that runs **3 replicas** of an Nginx web server.

### The Full YAML:

```yaml
apiVersion: apps/v1
kind: Deployment              # 👈 Not ReplicaSet, not Pod — DEPLOYMENT
metadata:
  name: nginx-deployment
  labels:
    app: nginx
spec:
  replicas: 3                 # We want 3 Pod replicas
  selector:                   # How to find Pods managed by this Deployment
    matchLabels:
      app: nginx
  template:                   # Pod template (blueprint for each Pod)
    metadata:
      labels:
        app: nginx            # Must match selector above
    spec:
      containers:
      - name: nginx
        image: nginx:1.25     # Container image
        ports:
        - containerPort: 80
```

### Notice Something?

This YAML looks **almost identical** to a ReplicaSet! The only difference is `kind: Deployment` instead of `kind: ReplicaSet`. That's because a Deployment wraps a ReplicaSet — it has the same fields plus extra superpowers.

## 🔄 Rolling Updates — The Star Feature

This is where Deployments really shine. Let's update our Nginx from `1.25` to `1.26` **without any downtime**.

### Method 1: Update the YAML and Re-apply

Change the image in your YAML file:

```yaml
      containers:
      - name: nginx
        image: nginx:1.26     # Changed from 1.25 to 1.26
```

Then apply:

```bash
kubectl apply -f nginx-deployment.yaml
```

### Method 2: Use kubectl set image

```bash
kubectl set image deployment/nginx-deployment nginx=nginx:1.26
```

This command says: _"In the `nginx-deployment` Deployment, change the container named `nginx` to use image `nginx:1.26`"_

### What Happens Behind the Scenes?

This is the magic! When you update the image, here's what Kubernetes does step by step:

```
BEFORE UPDATE:
┌─────────────────────────────────────────────┐
│  ReplicaSet-OLD (image: nginx:1.25)         │
│  ┌──────┐  ┌──────┐  ┌──────┐              │
│  │Pod 1 │  │Pod 2 │  │Pod 3 │   replicas=3 │
│  │ v1.25│  │ v1.25│  │ v1.25│              │
│  └──────┘  └──────┘  └──────┘              │
└─────────────────────────────────────────────┘

DURING UPDATE (Rolling):
┌─────────────────────────────────────────────┐
│  ReplicaSet-OLD (scaling down)              │
│  ┌──────┐  ┌──────┐                         │
│  │Pod 1 │  │Pod 2 │             replicas=2  │
│  │ v1.25│  │ v1.25│                         │
│  └──────┘  └──────┘                         │
└─────────────────────────────────────────────┘
┌─────────────────────────────────────────────┐
│  ReplicaSet-NEW (scaling up)                │
│  ┌──────┐                                   │
│  │Pod A │                       replicas=1  │
│  │ v1.26│                                   │
│  └──────┘                                   │
└─────────────────────────────────────────────┘

...continues until...

AFTER UPDATE:
┌─────────────────────────────────────────────┐
│  ReplicaSet-OLD (kept for rollback)         │
│                                 replicas=0  │
└─────────────────────────────────────────────┘
┌─────────────────────────────────────────────┐
│  ReplicaSet-NEW (image: nginx:1.26)         │
│  ┌──────┐  ┌──────┐  ┌──────┐              │
│  │Pod A │  │Pod B │  │Pod C │   replicas=3 │
│  │ v1.26│  │ v1.26│  │ v1.26│              │
│  └──────┘  └──────┘  └──────┘              │
└─────────────────────────────────────────────┘
```

**Key insight:** The old ReplicaSet is **NOT deleted** — it's scaled to 0 and kept around. This is how rollbacks work!

### Watch the Rolling Update in Real-Time

```bash

```

Output:
```
Waiting for deployment "nginx-deployment" rollout to finish: 1 out of 3 new replicas have been updated...

```


```

```

## ⚙️ Rolling Update Strategy — Controlling the Update Speed

You can control **how fast** the rolling update happens using two parameters:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
spec:
  replicas: 3
  strategy:
    type: RollingUpdate           # This is the default
    rollingUpdate:
      maxSurge: 1                 # Max extra Pods during update
      maxUnavailable: 1           # Max Pods that can be down during update
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.26
        ports:
        - containerPort: 80
```

### What Do maxSurge and maxUnavailable Mean?

| Parameter | Meaning | Example (replicas=3) |
|-----------|---------|---------------------|
| `maxSurge` | How many **extra** Pods can exist above the desired count during update | `maxSurge: 1` means max 4 Pods can exist (3 desired + 1 extra) |
| `maxUnavailable` | How many Pods can be **unavailable** during update | `maxUnavailable: 1` means at least 2 Pods must always be running |

**You can set these as numbers or percentages:**

```yaml
maxSurge: 25%           # 25% of 3 = 0.75 → rounded UP to 1
maxUnavailable: 25%     # 25% of 3 = 0.75 → rounded DOWN to 0
```

### Common Strategies:

| Strategy | maxSurge | maxUnavailable | Behavior |
|----------|----------|----------------|----------|
| **Conservative** | 1 | 0 | Never have fewer Pods than desired. Add 1 new, then remove 1 old. Slowest but safest. |
| **Default** | 25% | 25% | Balanced speed and availability |
| **Aggressive** | 50% | 50% | Fast update, but more Pods are unavailable at once |
| **All at once** | 100% | 0 | Create all new Pods first, then remove old ones. Requires double the resources temporarily. |

## 🔙 Rollbacks — Undoing a Bad Update

You deployed `nginx:1.26` but it has a bug? No problem — **roll back** to the previous version!

### Check Rollout History

```bash
kubectl rollout history deployment/nginx-deployment
```

Output:
```
deployment.apps/nginx-deployment
REVISION  CHANGE-CAUSE
1         <none>
2         <none>
```

Each revision represents a version of your Deployment. Revision 1 was `nginx:1.25`, revision 2 was `nginx:1.26`.

**Pro tip:** Add `--record` when applying changes to record the command in the history (or use annotations):

```bash
kubectl set image deployment/nginx-deployment nginx=nginx:1.26 --record
```

Or better yet, annotate your changes:

```bash
kubectl annotate deployment/nginx-deployment kubernetes.io/change-cause="Updated nginx to 1.26"
```

Now `kubectl rollout history` shows:
```
REVISION  CHANGE-CAUSE
1         <none>
2         Updated nginx to 1.26
```

### See Details of a Specific Revision

```bash
kubectl rollout history deployment/nginx-deployment --revision=1
```

Output:
```
deployment.apps/nginx-deployment with revision #1
Pod Template:
  Labels:       app=nginx
                pod-template-hash=7fb96c846b
  Containers:
   nginx:
    Image:      nginx:1.25
    Port:       80/TCP
```

### Rollback to the Previous Version

```bash
kubectl rollout undo deployment/nginx-deployment
```

This goes back **one revision** (from revision 2 back to revision 1 → `nginx:1.25`).

### Rollback to a Specific Revision

```bash
kubectl rollout undo deployment/nginx-deployment --to-revision=1
```

### What Happens During Rollback?

It's basically another rolling update — but this time, Kubernetes:
1. Scales **up** the old ReplicaSet (which was kept at 0 replicas)
2. Scales **down** the current ReplicaSet
3. Does this gradually, just like a normal rolling update

```
Rollback = Rolling update using an old ReplicaSet's Pod template
```

## ⏸️ Pause and Resume — Making Multiple Changes

What if you need to make several changes (update image, change environment variables, adjust resources) and you don't want each change to trigger a separate rolling update?

### Pause the Deployment

```bash
kubectl rollout pause deployment/nginx-deployment
```

### Make Multiple Changes

```bash
kubectl set image deployment/nginx-deployment nginx=nginx:1.27
kubectl set resources deployment/nginx-deployment -c nginx --limits=cpu=200m,memory=256Mi
```

No rolling update happens yet — the Deployment is paused!

### Resume the Deployment

```bash
kubectl rollout resume deployment/nginx-deployment
```

Now **one single rolling update** happens with ALL your changes applied together. Clean!

## 📏 Scaling a Deployment

### Scale Using kubectl

```bash
# Scale up to 5 replicas
kubectl scale deployment/nginx-deployment --replicas=5

# Scale down to 2
kubectl scale deployment/nginx-deployment --replicas=2

# Scale to 0 (stops all Pods but keeps the Deployment)
kubectl scale deployment/nginx-deployment --replicas=0
```

### Scale in YAML

```yaml
spec:
  replicas: 5     # Change this number
```

```bash
kubectl apply -f nginx-deployment.yaml
```

### Conditional Scaling (Only If Current Count Matches)

```bash
kubectl scale deployment/nginx-deployment --current-replicas=3 --replicas=5
```

This only scales if the current replica count is 3. It's a safety check — if someone else already scaled it, this command won't override their change.

## 🔄 Deployment Strategies — RollingUpdate vs Recreate

Kubernetes supports **two** deployment strategies:

### 1. RollingUpdate (Default)

```yaml
spec:
  strategy:
    type: RollingUpdate
```

- Gradually replaces old Pods with new ones
- **Zero downtime** — some Pods are always running
- Best for most applications
- We already covered this in detail above

### 2. Recreate

```yaml
spec:
  strategy:
    type: Recreate
```

- **Kills ALL old Pods first**, then creates all new Pods
- There IS downtime — the app is completely down during the switch
- Use when your app **can't run two versions simultaneously** (e.g., database schema changes, apps that lock files)

```
Recreate Strategy:

Step 1: Kill all old Pods
  ❌ Pod 1  ❌ Pod 2  ❌ Pod 3    ← All down, app is OFFLINE

Step 2: Create new Pods
  ✅ Pod A  ✅ Pod B  ✅ Pod C    ← All up, app is back ONLINE
```

## 📝 Complete Deployment YAML — With All Options

Here's a production-ready Deployment with all the common fields:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
  namespace: production
  labels:
    app: my-app
    version: v2
  annotations:
    kubernetes.io/change-cause: "Deployed v2 with new login feature"
spec:
  replicas: 3
  revisionHistoryLimit: 10           # Keep last 10 ReplicaSets for rollback
  progressDeadlineSeconds: 600       # Fail if update takes more than 10 minutes
  minReadySeconds: 5                 # Wait 5 seconds before considering a Pod "ready"
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0              # Never have fewer than desired Pods
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
        version: v2
    spec:
      containers:
      - name: my-app
        image: my-registry/my-app:v2
        ports:
        - containerPort: 8080
        resources:
          requests:
            cpu: "100m"
            memory: "128Mi"
          limits:
            cpu: "500m"
            memory: "512Mi"
        readinessProbe:              # Is the Pod ready to receive traffic?
          httpGet:
            path: /health
            port: 8080
          initialDelaySeconds: 5
          periodSeconds: 10
        livenessProbe:               # Is the Pod still alive?
          httpGet:
            path: /health
            port: 8080
          initialDelaySeconds: 15
          periodSeconds: 20
```

### Key Fields Explained:

| Field | What It Does |
|-------|-------------|
| `revisionHistoryLimit` | How many old ReplicaSets to keep (for rollbacks). Default is 10. |
| `progressDeadlineSeconds` | How long Kubernetes waits for a rollout to progress before marking it as failed. Default is 600 (10 min). |
| `minReadySeconds` | After a Pod becomes Ready, wait this many seconds before considering the rollout successful. Helps catch Pods that crash shortly after starting. |
| `readinessProbe` | Tells Kubernetes when the Pod is **ready to receive traffic**. Critical for zero-downtime deployments! |
| `livenessProbe` | Tells Kubernetes if the Pod is **still alive**. If it fails, K8s restarts the Pod. |

> ⚠️ **Important:** Without a `readinessProbe`, Kubernetes sends traffic to new Pods immediately — even if your app hasn't finished starting up. This can cause errors during rolling updates. **Always use readiness probes in production!**

## 🧪 Real-World Example — Full Update Workflow

Let's walk through a complete update scenario:

### 1. Deploy v1 of your app

```bash
kubectl create deployment my-web --image=my-app:v1 --replicas=3
```

### 2. Verify it's running

```bash
kubectl get deploy my-web
kubectl get pods -l app=my-web
```

### 3. Update to v2

```bash
kubectl set image deployment/my-web my-app=my-app:v2
kubectl annotate deployment/my-web kubernetes.io/change-cause="Upgrade to v2 - new dashboard"
```

### 4. Watch the rollout

```bash
kubectl rollout status deployment/my-web
```

### 5. Something is wrong! Pods are crashing!

```bash
kubectl get pods
# You see CrashLoopBackOff errors

kubectl rollout undo deployment/my-web
# Boom — you're back to v1
```

### 6. Fix the v2 image and try again

```bash
kubectl set image deployment/my-web my-app=my-app:v2.1
kubectl annotate deployment/my-web kubernetes.io/change-cause="Upgrade to v2.1 - fixed crash bug"
```

### 7. Check revision history

```bash
kubectl rollout history deployment/my-web
```

```
REVISION  CHANGE-CAUSE
1         <none>
2         Upgrade to v2 - new dashboard
3         <none> (this was the rollback to v1)
4         Upgrade to v2.1 - fixed crash bug
```

## 🎯 Key Interview Questions

**Q: What is a Deployment in Kubernetes?**
> A Deployment is a higher-level controller that manages ReplicaSets and provides declarative updates, rolling updates, rollbacks, and scaling for stateless applications.

**Q: How does a rolling update work?**
> Kubernetes creates a new ReplicaSet with the updated Pod template. It gradually scales up the new RS and scales down the old RS, ensuring that at least `maxUnavailable` Pods are always available and at most `maxSurge` extra Pods exist during the transition.

**Q: What's the difference between Recreate and RollingUpdate strategies?**
> Recreate kills all old Pods before creating new ones (causes downtime). RollingUpdate gradually replaces old Pods with new ones (zero downtime).

**Q: How do rollbacks work?**
> Old ReplicaSets are not deleted — they're kept at 0 replicas. During a rollback, Kubernetes scales up the old RS and scales down the current RS, effectively reverting to the previous Pod template.

**Q: What is `revisionHistoryLimit`?**
> It controls how many old ReplicaSets are kept for rollback. Default is 10. Setting it to 0 means you can't roll back (old RS are deleted).

**Q: What happens if a rolling update gets stuck?**
> If the rollout doesn't progress within `progressDeadlineSeconds` (default 600 seconds), the Deployment marks the update as failed. You can then investigate or rollback.

**Q: What is `minReadySeconds`?**
> After a new Pod becomes Ready, Kubernetes waits this many seconds before moving to the next Pod in the rolling update. This gives the Pod time to prove it's stable (not crashing right after startup).

## 🧩 Summary

| Concept | Key Point |
|---------|-----------|
| **What** | A controller that manages ReplicaSets and provides rolling updates, rollbacks, and scaling |
| **Creates** | Deployment → ReplicaSet → Pods (hierarchy) |
