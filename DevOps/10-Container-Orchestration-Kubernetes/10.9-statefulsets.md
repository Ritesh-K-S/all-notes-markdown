# 10.9 — StatefulSets — Stateful Applications (Databases, Kafka)

> **One Line Answer:** A StatefulSet is a Kubernetes controller designed for **stateful applications** — apps that need **stable identities**, **persistent storage**, and **ordered startup/shutdown** — like databases (MySQL, PostgreSQL, MongoDB), message queues (Kafka, RabbitMQ), and distributed systems (Elasticsearch, ZooKeeper).

---

## 🤔 The Problem — Why Can't We Use Deployments for Everything?

In the previous chapter (10.8), we learned that Deployments are awesome. They manage Pods, do rolling updates, handle rollbacks. So why do we need anything else?

Because Deployments are designed for **stateless** applications — apps where:
- Every Pod is **identical** and **interchangeable**
- Pods can be created, destroyed, and replaced freely
- Pods don't need to remember anything (no persistent data)
- Pod names and order don't matter

But some applications are **stateful** — they need guarantees that Deployments **cannot provide**.

### Stateless vs Stateful — The Restaurant Analogy

**Stateless App (like a fast-food counter):**
- 5 counters are serving customers
- Any customer can go to any counter
- If counter #3 breaks, replace it with a new one — customers don't care
- All counters are identical
- **This is a Deployment.**

**Stateful App (like a bank with personal accounts):**
- Each teller has a specific window number (Window 1, Window 2, Window 3)
- Your account is assigned to Window 2 — you MUST go there
- If Window 2's teller is replaced, the new teller needs access to your account history
- The windows must open in order (Window 1 first, then 2, then 3)
- **This is a StatefulSet.**

---

## 🗄️ Real-World Examples — What Needs StatefulSets?

| Application | Why It's Stateful |
|-------------|-------------------|
| **MySQL / PostgreSQL** | Data is stored on disk. Each instance has unique data. A replacement needs the SAME data. |
| **MongoDB Replica Set** | Has a primary and secondaries. Each member has a specific role and identity. |
| **Apache Kafka** | Each broker has a unique ID and stores partitions. Brokers must know each other's identities. |
| **ZooKeeper** | Nodes are numbered (zk-0, zk-1, zk-2). They form a quorum and need to find each other by name. |
| **Elasticsearch** | Each node stores specific shards. Nodes need stable network names for cluster discovery. |
| **Redis (Cluster mode)** | Each node owns specific hash slots. Identity matters for replication and failover. |
| **RabbitMQ** | Nodes form a cluster using hostnames. Hostnames must be stable. |

## 📦 What Makes a StatefulSet Special?

A StatefulSet provides **four key guarantees** that Deployments don't:

### 1. 🏷️ Stable, Unique Pod Names (Ordinal Index)

**Deployment Pods:** Random names like `nginx-deployment-7fb96c846b-x4k2p` — unpredictable, changes on restart.

**StatefulSet Pods:** Ordered names like `mysql-0`, `mysql-1`, `mysql-2` — predictable, **always the same**.

```
Deployment Pods:                  StatefulSet Pods:
├── nginx-7fb96c-x4k2p           ├── mysql-0        ← Always #0
├── nginx-7fb96c-a8m3n           ├── mysql-1        ← Always #1
└── nginx-7fb96c-q9w5r           └── mysql-2        ← Always #2
     ↑ Random names                    ↑ Ordered, stable names
```

If `mysql-1` crashes and is restarted, the new Pod is **still called `mysql-1`** — same name, same identity.

### 2. 💾 Stable, Persistent Storage (Per-Pod Volumes)

In a Deployment, if a Pod dies and gets recreated, it loses its data (unless you use shared storage, but that's tricky for databases).

In a StatefulSet, **each Pod gets its own PersistentVolumeClaim (PVC)**:

```
StatefulSet: mysql (replicas: 3)

  mysql-0  →  PVC: data-mysql-0  →  PV: pv-001  →  Actual Disk
  mysql-1  →  PVC: data-mysql-1  →  PV: pv-002  →  Actual Disk
  mysql-2  →  PVC: data-mysql-2  →  PV: pv-003  →  Actual Disk
```

If `mysql-1` crashes and gets recreated, the new `mysql-1` Pod is **attached to the SAME PVC** (`data-mysql-1`), so it gets back all its data. No data loss!

### 3. 🌐 Stable Network Identity (DNS Names)

Each Pod in a StatefulSet gets a **predictable, stable DNS name** through a **Headless Service**:

```
Pod Name:     mysql-0
DNS Name:     mysql-0.mysql-headless.default.svc.cluster.local
              ──────  ──────────────  ───────  ───  ────────────
              Pod     Headless Svc    Namespace      Cluster Domain
              Name    Name
```

This means other Pods can always find `mysql-0` by its DNS name, even if the Pod IP changes after a restart.

### 4. 📊 Ordered, Graceful Operations

**Pods are created in order:** `mysql-0` first, then `mysql-1`, then `mysql-2`.

**Pods are deleted in reverse order:** `mysql-2` first, then `mysql-1`, then `mysql-0`.

This matters because many distributed systems need specific startup/shutdown sequences:

```
Startup Order:
  mysql-0 starts and becomes Primary     ← Must be first!
  mysql-1 starts and joins as Replica     ← Needs Primary running
  mysql-2 starts and joins as Replica     ← Needs Primary running

Shutdown Order:
  mysql-2 stops first                     ← Replica, safe to remove
  mysql-1 stops next                      ← Replica, safe to remove
  mysql-0 stops last                      ← Primary, stops last
```

## 📊 StatefulSet vs Deployment — Side-by-Side Comparison

| Feature | Deployment | StatefulSet |
|---------|-----------|-------------|
| **Pod Names** | Random (e.g., `app-7fb96c-x4k2p`) | Ordered (e.g., `app-0`, `app-1`, `app-2`) |
| **Pod Identity** | Interchangeable — any Pod can replace any other | Unique — `app-1` is always `app-1` |
| **Storage** | Shared or no persistent storage | Each Pod has its own persistent volume |
| **DNS** | All Pods share one Service IP | Each Pod gets its own DNS name |
| **Startup Order** | All Pods start simultaneously | Pods start one by one, in order (0, 1, 2...) |
| **Shutdown Order** | All Pods stop simultaneously | Pods stop in reverse order (2, 1, 0) |
| **Scaling Up** | Creates Pods in any order | Creates next ordinal (e.g., 3 → adds `app-3`) |
| **Scaling Down** | Removes any Pod | Removes highest ordinal first (e.g., `app-3` first) |
| **Updates** | Rolling updates (new RS) | Rolling updates (in reverse order by default) |
| **Use Case** | Stateless apps (web servers, APIs) | Stateful apps (databases, message queues) |

## 📝 Writing a StatefulSet YAML — Step by Step

Let's create a StatefulSet for a MySQL database with 3 replicas.

### Prerequisites — The Headless Service

Before creating a StatefulSet, you need a **Headless Service**. This is a regular Kubernetes Service but with `clusterIP: None`. It doesn't load-balance — instead, it creates individual DNS records for each Pod.

```yaml
apiVersion: v1
kind: Service
metadata:
  name: mysql-headless           # Name of the headless service
  labels:
    app: mysql
spec:
  ports:
  - port: 3306
    name: mysql
  clusterIP: None                # 👈 This makes it "headless"
  selector:
    app: mysql                   # Matches Pods with this label
```

**Why headless?** Because in a database cluster, clients often need to connect to a **specific** Pod (e.g., the primary), not just any Pod. The headless service gives each Pod its own DNS name so you can target them individually.

### The StatefulSet YAML:

```yaml
apiVersion: apps/v1
kind: StatefulSet                # 👈 Not Deployment!
metadata:
  name: mysql
spec:
  serviceName: mysql-headless    # 👈 Must reference the Headless Service
  replicas: 3
  selector:
    matchLabels:
      app: mysql
  template:
    metadata:
      labels:
        app: mysql
    spec:
      containers:
      - name: mysql
        image: mysql:8.0
        ports:
        - containerPort: 3306
          name: mysql
        env:
        - name: MYSQL_ROOT_PASSWORD
          value: "my-secret-pw"           # In production, use Secrets!
        volumeMounts:
        - name: mysql-data
          mountPath: /var/lib/mysql        # Where MySQL stores data
  volumeClaimTemplates:          # 👈 This is unique to StatefulSets!
  - metadata:
      name: mysql-data
    spec:
      accessModes: ["ReadWriteOnce"]
      resources:
        requests:
          storage: 10Gi          # Each Pod gets 10GB of storage
```

### Key Differences from Deployment YAML:

| Field | What's Different |
|-------|-----------------|
| `kind: StatefulSet` | We're creating a StatefulSet, not a Deployment |
| `spec.serviceName` | **Required!** Points to the Headless Service that provides DNS names |
| `volumeClaimTemplates` | **Unique to StatefulSets!** Creates a separate PVC for each Pod automatically |

### How volumeClaimTemplates Works:

Unlike Deployments where you define volumes directly, StatefulSets use `volumeClaimTemplates` to **automatically create a new PVC for each Pod**:

```
StatefulSet creates:
  Pod: mysql-0  →  PVC: mysql-data-mysql-0  (10Gi)
  Pod: mysql-1  →  PVC: mysql-data-mysql-1  (10Gi)
  Pod: mysql-2  →  PVC: mysql-data-mysql-2  (10Gi)

PVC naming pattern: <volumeClaimTemplate-name>-<pod-name>
                     mysql-data              -mysql-0
```

## 🚀 Hands-On — Creating and Managing a StatefulSet

### Step 1: Create the Headless Service and StatefulSet

Save both YAMLs in a single file (`mysql-statefulset.yaml`) separated by `---`:

```yaml
# Headless Service
apiVersion: v1
kind: Service
metadata:
  name: mysql-headless
  labels:
    app: mysql
spec:
  ports:
  - port: 3306
    name: mysql
  clusterIP: None
  selector:
    app: mysql
---
# StatefulSet
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: mysql
spec:
  serviceName: mysql-headless
  replicas: 3
  selector:
    matchLabels:
      app: mysql
  template:
    metadata:
      labels:
        app: mysql
    spec:
      containers:
      - name: mysql
        image: mysql:8.0
        ports:
        - containerPort: 3306
          name: mysql
        env:
        - name: MYSQL_ROOT_PASSWORD
          valueFrom:
            secretKeyRef:
              name: mysql-secret
              key: password
        volumeMounts:
        - name: mysql-data
          mountPath: /var/lib/mysql
  volumeClaimTemplates:
  - metadata:
      name: mysql-data
    spec:
      accessModes: ["ReadWriteOnce"]
      resources:
        requests:
          storage: 10Gi
```

Apply it:

```bash
kubectl apply -f mysql-statefulset.yaml
```

### Step 2: Watch Pods Being Created In Order

```bash
kubectl get pods -w -l app=mysql
```

Output (over time):
```
NAME      READY   STATUS    RESTARTS   AGE
mysql-0   0/1     Pending   0          0s
mysql-0   0/1     ContainerCreating   0   2s
mysql-0   1/1     Running   0          10s     ← mysql-0 is ready
mysql-1   0/1     Pending   0          0s       ← NOW mysql-1 starts
mysql-1   0/1     ContainerCreating   0   2s
mysql-1   1/1     Running   0          12s     ← mysql-1 is ready
mysql-2   0/1     Pending   0          0s       ← NOW mysql-2 starts
mysql-2   0/1     ContainerCreating   0   2s
mysql-2   1/1     Running   0          11s     ← mysql-2 is ready
```

**See the order?** `mysql-0` → `mysql-1` → `mysql-2`. Each Pod waits for the previous one to be Ready before starting!

### Step 3: Check the PVCs (Persistent Volume Claims)

```bash
kubectl get pvc
```

Output:
```
NAME                 STATUS   VOLUME          CAPACITY   ACCESS MODES   AGE
mysql-data-mysql-0   Bound    pv-random-001   10Gi       RWO            2m
mysql-data-mysql-1   Bound    pv-random-002   10Gi       RWO            1m
mysql-data-mysql-2   Bound    pv-random-003   10Gi       RWO            30s
```

Each Pod has its own PVC! They'll persist even if the Pod is deleted.

### Step 4: Test DNS Names

From any Pod in the cluster, you can reach each MySQL Pod by its DNS name:

```bash
# Enter a temporary Pod to test DNS
kubectl run dns-test --image=busybox --rm -it -- sh

# Inside the Pod, try DNS lookups:
nslookup mysql-0.mysql-headless.default.svc.cluster.local
nslookup mysql-1.mysql-headless.default.svc.cluster.local
nslookup mysql-2.mysql-headless.default.svc.cluster.local
```

Each query returns the **specific Pod's IP address** — not a load-balanced IP.

### Step 5: Test Pod Identity Persistence

Delete `mysql-1` and watch what happens:

```bash
kubectl delete pod mysql-1
kubectl get pods -w -l app=mysql
```

Output:
```
NAME      READY   STATUS        RESTARTS   AGE
mysql-0   1/1     Running       0          5m
mysql-1   1/1     Terminating   0          5m      ← Being deleted
mysql-2   1/1     Running       0          5m
mysql-1   0/1     Pending       0          0s       ← Recreated as mysql-1!
mysql-1   1/1     Running       0          8s       ← Same name, same PVC
```

The new Pod is **still called `mysql-1`** and it's **attached to the same PVC** (`mysql-data-mysql-1`). All the data is preserved!

## 🔄 Updating a StatefulSet

### Update Strategies

StatefulSets support two update strategies:

#### 1. RollingUpdate (Default)

```yaml
spec:
  updateStrategy:
    type: RollingUpdate
    rollingUpdate:
      partition: 0          # Update all Pods (we'll explain this below)
```

Updates Pods **one at a time**, in **reverse order** (highest ordinal first):

```
Update order: mysql-2 → mysql-1 → mysql-0
              (reverse of creation order)
```

This is the opposite of Deployments! It makes sense for databases — you usually want to update replicas first, and the primary last.

#### 2. OnDelete

```yaml
spec:
  updateStrategy:
    type: OnDelete
```

When you update the StatefulSet, **nothing happens automatically**. Pods are only updated when you **manually delete** them. The replacement Pod will use the new template.

This gives you complete manual control — useful for databases where you want to coordinate the update carefully.

### Partitioned Rolling Updates (Canary Updates)

The `partition` field is super useful. It tells Kubernetes: _"Only update Pods with ordinal >= partition"_

```yaml
spec:
  updateStrategy:
    type: RollingUpdate
    rollingUpdate:
      partition: 2           # Only update Pods with ordinal >= 2
```

With `partition: 2` and 3 replicas (mysql-0, mysql-1, mysql-2):
- `mysql-2` gets updated to the new version ✅
- `mysql-1` stays on the old version ❌
- `mysql-0` stays on the old version ❌

This is a **canary update** — you can test the new version on one Pod before rolling it out to all.

```
Step 1: Set partition=2 and update image
  mysql-0: v1  mysql-1: v1  mysql-2: v2    ← Only mysql-2 updated

Step 2: Test mysql-2. It works! Set partition=1
  mysql-0: v1  mysql-1: v2  mysql-2: v2    ← mysql-1 also updated

Step 3: Everything good! Set partition=0
  mysql-0: v2  mysql-1: v2  mysql-2: v2    ← All updated!
```

## 🔢 Scaling StatefulSets

### Scale Up

```bash
kubectl scale statefulset mysql --replicas=5
```

New Pods are created **in order**: `mysql-3`, then `mysql-4`. Each waits for the previous to be Ready.

### Scale Down

```bash
kubectl scale statefulset mysql --replicas=2
```

Pods are removed in **reverse order**: `mysql-4` first, then `mysql-3`, then `mysql-2`.

> ⚠️ **Important:** When you scale down, the **PVCs are NOT deleted**! The data is preserved. If you scale back up later, the new Pods reattach to their old PVCs and get their data back.

```
Scale down from 3 to 1:
  mysql-2: DELETED  →  PVC mysql-data-mysql-2: KEPT (data preserved)
  mysql-1: DELETED  →  PVC mysql-data-mysql-1: KEPT (data preserved)
  mysql-0: Running  →  PVC mysql-data-mysql-0: Active

Scale back up to 3:
  mysql-1: Created  →  Reattaches to PVC mysql-data-mysql-1 (data restored!)
  mysql-2: Created  →  Reattaches to PVC mysql-data-mysql-2 (data restored!)
```

## 🌐 Deep Dive — Headless Service & DNS

### Normal Service vs Headless Service

```
Normal Service (clusterIP: 10.96.0.50):
┌────────────────────────────┐
│  Service: mysql-service    │
│  IP: 10.96.0.50            │
│  DNS: mysql-service        │
│                            │
│  Load balances to:         │
│  ├── mysql-0 (10.0.1.10)   │
│  ├── mysql-1 (10.0.2.20)   │
│  └── mysql-2 (10.0.3.30)   │
└────────────────────────────┘
Client connects to 10.96.0.50 → gets random Pod

Headless Service (clusterIP: None):
┌────────────────────────────┐
│  Service: mysql-headless   │
│  IP: None (no ClusterIP!)  │
│  DNS:                      │
│  ├── mysql-0.mysql-headless│
│  │    → 10.0.1.10          │
│  ├── mysql-1.mysql-headless│
│  │    → 10.0.2.20          │
│  └── mysql-2.mysql-headless│
│       → 10.0.3.30          │
└────────────────────────────┘
Client connects to mysql-1.mysql-headless → always gets mysql-1
```

### DNS Records Created

For a StatefulSet named `mysql` with headless service `mysql-headless` in namespace `default`:

| Pod | Full DNS Name |
|-----|--------------|
| mysql-0 | `mysql-0.mysql-headless.default.svc.cluster.local` |
| mysql-1 | `mysql-1.mysql-headless.default.svc.cluster.local` |
| mysql-2 | `mysql-2.mysql-headless.default.svc.cluster.local` |

You can also use short names:
- `mysql-0.mysql-headless` (within the same namespace)
- `mysql-0.mysql-headless.default` (from other namespaces)

## 🧪 Real-World Example — ZooKeeper Ensemble

ZooKeeper is the classic StatefulSet example. It needs:
- Exactly 3 (or 5) nodes with stable identities
- Each node knows the others by name
- Ordered startup (node 0 starts first)

```yaml
apiVersion: v1
kind: Service
metadata:
  name: zk-headless
spec:
  ports:
  - port: 2181
    name: client
  - port: 2888
    name: server
  - port: 3888
    name: leader-election
  clusterIP: None
  selector:
    app: zookeeper
---
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: zk
spec:
  serviceName: zk-headless
  replicas: 3
  selector:
    matchLabels:
      app: zookeeper
  template:
    metadata:
      labels:
        app: zookeeper
    spec:
      containers:
      - name: zookeeper
        image: zookeeper:3.9
        ports:
        - containerPort: 2181
          name: client
        - containerPort: 2888
          name: server
        - containerPort: 3888
          name: leader-election
        env:
        - name: ZOO_MY_ID
          valueFrom:
            fieldRef:
              fieldPath: metadata.name    # Gets "zk-0", "zk-1", etc.
        - name: ZOO_SERVERS
          value: >-
            server.0=zk-0.zk-headless:2888:3888;2181
            server.1=zk-1.zk-headless:2888:3888;2181
            server.2=zk-2.zk-headless:2888:3888;2181
        volumeMounts:
        - name: zk-data
          mountPath: /data
  volumeClaimTemplates:
  - metadata:
      name: zk-data
    spec:
      accessModes: ["ReadWriteOnce"]
      resources:
        requests:
          storage: 5Gi
```

Notice how `ZOO_SERVERS` uses the stable DNS names (`zk-0.zk-headless`, `zk-1.zk-headless`, etc.) — this only works because StatefulSets provide stable DNS!

## ⚠️ Important Gotchas and Best Practices

### 1. PVCs Are NOT Deleted Automatically

When you delete a StatefulSet or scale down, the PVCs **persist**. This is a safety feature (you don't want to accidentally delete your database data!), but it means you need to **manually clean up PVCs** if you truly want to delete the data:

```bash
# Delete the StatefulSet
kubectl delete statefulset mysql

# PVCs still exist!
kubectl get pvc
# mysql-data-mysql-0   Bound   ...
# mysql-data-mysql-1   Bound   ...
# mysql-data-mysql-2   Bound   ...

# Manually delete PVCs if you want to remove the data
kubectl delete pvc mysql-data-mysql-0 mysql-data-mysql-1 mysql-data-mysql-2
```

### 2. Pod Management Policies

By default, Pods are created/deleted **one at a time, in order**. But sometimes you want parallel operations:

```yaml
spec:
  podManagementPolicy: OrderedReady    # Default — one at a time, in order
  # OR
  podManagementPolicy: Parallel        # All Pods start/stop simultaneously
```

Use `Parallel` when your Pods don't depend on each other for startup (e.g., they can self-discover cluster members after starting).

### 3. Always Use Readiness Probes

```yaml
readinessProbe:
  exec:
    command:
    - sh
    - -c
    - "mysql -u root -p$MYSQL_ROOT_PASSWORD -e 'SELECT 1'"
  initialDelaySeconds: 30
  periodSeconds: 10
```

Without readiness probes, the StatefulSet might move to creating the next Pod before the current one is truly ready — which can break ordered startup guarantees.

### 4. Use Pod Disruption Budgets (PDBs)

Protect your StatefulSet from too many Pods being disrupted at once (e.g., during node upgrades):

```yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: mysql-pdb
spec:
  minAvailable: 2            # At least 2 Pods must be running at all times
  selector:
    matchLabels:
      app: mysql
```

### 5. Use Anti-Affinity to Spread Pods Across Nodes

You don't want all your database Pods on the same node — if that node fails, you lose all replicas:

```yaml
spec:
  template:
    spec:
      affinity:
        podAntiAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
          - labelSelector:
              matchExpressions:
              - key: app
                operator: In
                values:
                - mysql
            topologyKey: "kubernetes.io/hostname"
```

This ensures each MySQL Pod runs on a **different** node.

## 🗑️ Deleting a StatefulSet

### Delete StatefulSet (Graceful — Default)

```bash
kubectl delete statefulset mysql
```

Pods are terminated in **reverse order**: `mysql-2` → `mysql-1` → `mysql-0`.

### Delete StatefulSet (Non-Cascading — Keep Pods)

```bash
kubectl delete statefulset mysql --cascade=orphan
```

The StatefulSet is deleted but Pods keep running as orphans. Useful when you want to recreate the StatefulSet with different settings without restarting Pods.

### Delete Everything Including PVCs

```bash
kubectl delete statefulset mysql
kubectl delete svc mysql-headless
kubectl delete pvc -l app=mysql     # Delete PVCs by label
```

## 🎯 Key Interview Questions

**Q: What is a StatefulSet and when do you use it?**
> A StatefulSet manages stateful applications that require stable network identities, persistent storage, and ordered deployment. Use it for databases, message queues, and distributed systems like MySQL, MongoDB, Kafka, ZooKeeper, etc.

**Q: What's the difference between a Deployment and a StatefulSet?**
> Deployments are for stateless apps — Pods are interchangeable, have random names, and share storage. StatefulSets are for stateful apps — Pods have stable ordered names (app-0, app-1), each gets its own persistent volume, and startup/shutdown is ordered.

**Q: What is a Headless Service and why does StatefulSet need it?**
> A Headless Service (clusterIP: None) doesn't load-balance. Instead, it creates individual DNS records for each Pod. StatefulSets need this so each Pod can be addressed by a predictable, stable DNS name (e.g., `mysql-0.mysql-headless`).

**Q: What happens to PVCs when a StatefulSet is deleted?**
> PVCs are NOT deleted. They persist even after the StatefulSet and its Pods are gone. This is by design to prevent accidental data loss. You must manually delete PVCs if you want to remove the data.

**Q: What is the partition field in StatefulSet update strategy?**
> The `partition` field enables canary updates. Only Pods with an ordinal index >= partition value get updated. This lets you test the new version on a subset of Pods before rolling it out to all.

**Q: In what order are StatefulSet Pods created and deleted?**
> Created in ascending order: 0, 1, 2, 3... (each waits for the previous to be Ready). Deleted in descending order: ...3, 2, 1, 0. This can be changed with `podManagementPolicy: Parallel`.

**Q: Can you use a Deployment for a database?**
> Technically yes, but it's not recommended. Deployments don't guarantee stable Pod identities, ordered startup, or per-Pod persistent storage — all of which are critical for databases. Use a StatefulSet.

| **Storage** | Each Pod gets its own PVC via `volumeClaimTemplates` |
| **DNS** | Each Pod gets a unique DNS name through a Headless Service |
| **Startup** | Ordered: 0 → 1 → 2 (each waits for the previous) |
| **Shutdown** | Reverse order: 2 → 1 → 0 |
| **Updates** | RollingUpdate (reverse order) or OnDelete (manual) |
| **Partition** | Canary updates — only update Pods with ordinal >= partition |
| **PVC Cleanup** | PVCs are NOT auto-deleted — manual cleanup required |
| **Headless Service** | Required — provides stable DNS names for Pods |
| **Use Cases** | Databases, message queues, distributed systems (MySQL, Kafka, ZooKeeper, etc.) |
