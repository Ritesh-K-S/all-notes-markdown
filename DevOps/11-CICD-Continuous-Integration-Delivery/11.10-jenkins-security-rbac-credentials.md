## 🤔 Why Does Jenkins Security Matter?

Let's think about it this way:

Jenkins is the **brain of your CI/CD pipeline**. It has access to:
- Your **source code** (via Git credentials)
- Your **cloud infrastructure** (via AWS/Azure/GCP credentials)
- Your **production servers** (via SSH keys)
- Your **Docker registries** (via auth tokens)
- Your **databases** (via connection strings and passwords)

If someone gains unauthorized access to Jenkins, they can:
- 🔥 **Deploy malicious code** to production
- 🔥 **Steal all your secrets** (API keys, passwords, tokens)
- 🔥 **Delete your infrastructure**
- 🔥 **Inject backdoors** into your builds

**Jenkins is a high-value target.** If it's not secured, your entire software supply chain is at risk.

> 💡 **Rule of thumb:** Treat Jenkins like the **keys to your kingdom**. Lock it down tight.

## 1️⃣ Jenkins Security Basics — The First Thing You Should Do

When you first install Jenkins, by default it's pretty open — which is **dangerous**. The very first thing you should do is **enable security**.

### Accessing Security Settings

```
Jenkins Dashboard → Manage Jenkins → Security (Configure Global Security)
```

You'll see a page with several sections:
- **Authentication** — how users prove who they are
- **Authorization** — what users are allowed to do
- **CSRF Protection** — protecting against cross-site request forgery
- **Agent → Controller Security** — securing communication between agents and controller

### The Initial Admin Password

When you first install Jenkins, it generates a one-time admin password stored in:

```bash
# Linux
cat /var/lib/jenkins/secrets/initialAdminPassword

# Docker
docker exec <container-id> cat /var/jenkins_home/secrets/initialAdminPassword
```

You use this password to log in for the first time and set up your admin account.

> ⚠️ **Important:** Change the default admin password immediately after first login!

## 2️⃣ Authentication — "Who Are You?"

Authentication is about **verifying identity** — proving that you are who you say you are.

Jenkins supports multiple authentication methods:

### Option 1: Jenkins' Own User Database (Default)

This is the simplest option. Jenkins manages its own list of users and passwords.

```
Configure Global Security → Security Realm → Jenkins' own user database
```

- ✅ Good for small teams (5-10 people)
- ✅ Easy to set up
- ❌ Doesn't scale well for large organizations
- ❌ Users have yet another password to remember

**How to create users:**

```
Manage Jenkins → Users → Create User
```

You fill in:
- Username
- Password
- Full Name
- Email

### Option 2: LDAP (Lightweight Directory Access Protocol)

If your company uses Active Directory (AD) or OpenLDAP, you can connect Jenkins to it.

```
Configure Global Security → Security Realm → LDAP
```

Configuration example:

```
Server: ldap://ldap.company.com
Root DN: dc=company,dc=com
User search base: ou=People
User search filter: uid={0}
Group search base: ou=Groups
Manager DN: cn=admin,dc=company,dc=com
Manager Password: ********
```

- ✅ Users log in with their corporate credentials
- ✅ Centralized user management
- ✅ Great for large organizations
- ❌ Requires LDAP/AD infrastructure

### Option 3: SSO with SAML / OAuth / OpenID Connect

For modern setups, you can integrate Jenkins with:
- **GitHub OAuth** — users log in with their GitHub account
- **Google OAuth** — users log in with their Google account
- **SAML 2.0** — integrate with Okta, Azure AD, OneLogin, etc.
- **OpenID Connect** — integrate with Keycloak, Auth0, etc.

**Example: GitHub OAuth Plugin Setup**

1. Install the "GitHub Authentication" plugin
2. Create an OAuth App on GitHub:
   - Go to GitHub → Settings → Developer Settings → OAuth Apps → New OAuth App
   - Set Authorization callback URL: `https://your-jenkins-url/securityRealm/finishLogin`
3. Configure in Jenkins:

```
Security Realm: GitHub Authentication Plugin
GitHub API URL: https://api.github.com
Client ID: <your-github-oauth-client-id>
Client Secret: <your-github-oauth-client-secret>
OAuth Scope: read:org
```

- ✅ No extra passwords for users
- ✅ Leverages existing identity providers
- ✅ Supports MFA (Multi-Factor Authentication) from the identity provider
- ❌ Requires plugin installation and configuration

## 3️⃣ Authorization — "What Can You Do?"

Once Jenkins knows **who you are** (authentication), it needs to decide **what you're allowed to do** (authorization).

Jenkins offers several authorization strategies:

| Strategy | Description | Best For |
|----------|-------------|----------|
| **Anyone can do anything** | No restrictions at all | ⚠️ Never use in production! |
| **Legacy mode** | Admin has full control, everyone else read-only | Very small teams |
| **Logged-in users can do anything** | All authenticated users have full access | Small trusted teams |
| **Matrix-based security** | Fine-grained permissions per user/group | Medium teams |
| **Project-based Matrix** | Permissions per project + global | Large teams |
| **Role-Based Strategy** | Roles assigned to users/groups (Plugin) | Enterprise teams |

Let's explore the important ones in detail.

---

## 4️⃣ Matrix-Based Security

This is Jenkins' built-in fine-grained permission system. You create a **matrix (table)** where:
- **Rows** = Users or Groups
- **Columns** = Permissions

### How to Enable It

```
Manage Jenkins → Security → Authorization → Matrix-based security
```

### The Permission Categories

Jenkins organizes permissions into these categories:

| Category | What It Controls |
|----------|-----------------|
| **Overall** | Administer, Read, RunScripts, ConfigureUpdateCenter |
| **Credentials** | Create, Delete, Manage, View, Update |
| **Agent** | Build, Configure, Connect, Create, Delete, Disconnect |
| **Job** | Build, Cancel, Configure, Create, Delete, Discover, Move, Read, Workspace |
| **Run** | Delete, Replay, Update |
| **View** | Configure, Create, Delete, Read |
| **SCM** | Tag |

### Example Matrix Configuration

Let's say you have three types of users:

```
┌──────────────────┬───────────┬──────────┬──────────┬──────────┬──────────┐
│ User/Group       │ Overall   │ Job      │ Job      │ Job      │ Job      │
│                  │ Read      │ Read     │ Build    │ Configure│ Create   │
├──────────────────┼───────────┼──────────┼──────────┼──────────┼──────────┤
│ admin            │ ✅ (all)  │ ✅       │ ✅       │ ✅       │ ✅       │
│ developers       │ ✅        │ ✅       │ ✅       │ ❌       │ ❌       │
│ qa-team          │ ✅        │ ✅       │ ✅       │ ❌       │ ❌       │
│ viewers          │ ✅        │ ✅       │ ❌       │ ❌       │ ❌       │
│ Anonymous        │ ❌        │ ❌       │ ❌       │ ❌       │ ❌       │
└──────────────────┴───────────┴──────────┴──────────┴──────────┴──────────┘
```

**What this means:**
- **admin** — can do everything (full access)
- **developers** — can see and trigger builds, but can't change job configs or create new jobs
- **qa-team** — same as developers (can trigger test pipelines)
- **viewers** — can only view, can't trigger anything
- **Anonymous** — no access at all (must log in)

> ⚠️ **Common Mistake:** Forgetting to give the `admin` user **Overall/Administer** permission. If you lock yourself out, you'll need to fix it via the config file on the server.

### What If You Lock Yourself Out?

Don't panic! Here's how to recover:

```bash
# 1. Stop Jenkins
sudo systemctl stop jenkins

# 2. Edit the Jenkins config file
sudo nano /var/lib/jenkins/config.xml

# 3. Find and change these lines:
#    Change <useSecurity>true</useSecurity> to:
#    <useSecurity>false</useSecurity>

# 4. Restart Jenkins
sudo systemctl start jenkins

# 5. Go to Jenkins UI, reconfigure security, and set up admin properly

# 6. Re-enable security
```

> 🔴 **Warning:** This temporarily disables ALL security. Do this only in emergencies and re-enable security immediately!

## 5️⃣ Role-Based Access Control (RBAC) — The Best Approach

Matrix-based security works, but it gets **messy** when you have many users. Imagine managing permissions for 100 users one by one — nightmare! 😅

That's where the **Role-Based Strategy Plugin** comes in. Instead of assigning permissions to individual users, you:

1. **Create Roles** (like "Admin", "Developer", "Viewer")
2. **Assign permissions to Roles**
3. **Assign users to Roles**

This is how **every enterprise** manages access.

### Step 1: Install the Plugin

```
Manage Jenkins → Plugins → Available Plugins → Search "Role-based Authorization Strategy" → Install
```

### Step 2: Enable Role-Based Strategy

```
Manage Jenkins → Security → Authorization → Role-Based Strategy
```

### Step 3: Manage Roles

```
Manage Jenkins → Security → Manage and Assign Roles
```

You'll see three types of roles:

#### A) Global Roles — Apply to the entire Jenkins instance

| Role Name | Overall Read | Job Read | Job Build | Job Configure | Job Create | Administer |
|-----------|-------------|----------|-----------|---------------|------------|------------|
| admin     | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| developer | ✅ | ✅ | ✅ | ❌ | ❌ | ❌ |
| viewer    | ✅ | ✅ | ❌ | ❌ | ❌ | ❌ |

#### B) Project Roles (Item Roles) — Apply to specific jobs/folders matching a pattern

| Role Name | Pattern (Regex) | Job Read | Job Build | Job Configure |
|-----------|-----------------|----------|-----------|---------------|
| frontend-dev | `frontend-.*` | ✅ | ✅ | ✅ |
| backend-dev  | `backend-.*`  | ✅ | ✅ | ✅ |
| qa-tester    | `.*-test-.*`  | ✅ | ✅ | ❌ |

**What this means:**
- `frontend-dev` role can see, build, and configure any job whose name starts with `frontend-`
- `backend-dev` role can do the same for jobs starting with `backend-`
- `qa-tester` can see and run any job with `-test-` in the name, but can't modify them

#### C) Agent Roles (Node Roles) — Control who can use which build agents

| Role Name | Pattern | Connect | Configure | Build |
|-----------|---------|---------|-----------|-------|
| linux-users | `linux-.*` | ✅ | ❌ | ✅ |
| k8s-admins  | `k8s-.*`   | ✅ | ✅ | ✅ |

### Step 4: Assign Roles to Users

```
Manage and Assign Roles → Assign Roles
```

| User/Group | Global Role | Project Role |
|------------|-------------|--------------|
| ritesh     | admin       | — |
| john       | developer   | frontend-dev |
| sarah      | developer   | backend-dev |
| mike       | viewer      | qa-tester |

Now:
- **ritesh** is a full admin
- **john** is a developer who can only configure frontend jobs
- **sarah** is a developer who can only configure backend jobs
- **mike** can only view Jenkins and trigger test jobs

> 💡 **This is beautiful!** Instead of managing 100 users individually, you manage 5-10 roles. New team member? Just assign them a role and they're good to go.

## 6️⃣ Project-Based Security

For even more granular control, you can set permissions **per job/pipeline**.

### Enable It

```
Configure Global Security → Authorization → Project-based Matrix Authorization Strategy
```

### How It Works

Each job gets its own permission matrix:

```
Job → Configure → Enable project-based security → Add users/groups with permissions
```

**Example:** You have a job called `deploy-production`:

| User/Group | Read | Build | Configure |
|------------|------|-------|-----------|
| deploy-team | ✅ | ✅ | ✅ |
| developers | ✅ | ❌ | ❌ |
| qa-team | ✅ | ❌ | ❌ |

Only the `deploy-team` can trigger production deployments. Everyone else can only view.

## 7️⃣ Credentials Management — Storing Secrets Safely

This is one of the **most critical** parts of Jenkins security. Your pipelines need secrets to work:
- Git credentials to pull code
- Docker registry credentials to push images
- AWS access keys to deploy to cloud
- SSH keys to connect to servers
- Database passwords
- API tokens

**The golden rule: NEVER hardcode secrets in your Jenkinsfile or scripts!**

```groovy
// ❌ NEVER DO THIS — secrets in plain text!!!
pipeline {
    agent any
    stages {
        stage('Deploy') {
            steps {
                sh 'docker login -u admin -p MySecretPassword123 registry.example.com'
                sh 'aws configure set aws_access_key_id AKIAIOSFODNN7EXAMPLE'
            }
        }
    }
}
```

**Why is this terrible?**
- Secrets are visible in **version control** (Git history)
- Secrets are visible in **console output** (build logs)
- Anyone with read access to the repo can see them
- They can **never** be truly removed from Git history

### How Jenkins Credentials Work

Jenkins has a built-in **Credentials Store** that:
- **Encrypts** secrets at rest using AES-128/AES-256
- **Masks** secrets in console output (replaces with `****`)
- **Controls access** to secrets via scopes and domains
- **Provides** a secure API for pipelines to use secrets

### Where to Add Credentials

```
Manage Jenkins → Credentials → System → Global credentials (unrestricted) → Add Credentials
```

Or for folder-scoped credentials:

```
Folder → Credentials → Add Credentials
```

## 8️⃣ Credential Types & Scopes

### Credential Types

Jenkins supports these credential types out of the box:

| Type | What It Stores | Use Case |
|------|---------------|----------|
| **Username with password** | Username + Password | Git repos, Docker registries, databases |
| **Secret text** | A single secret string | API tokens, webhook secrets |
| **Secret file** | An encrypted file | kubeconfig, service account JSON, certificates |
| **SSH Username with private key** | SSH private key + passphrase | SSH into servers, Git over SSH |
| **Certificate** | PKCS#12 certificate + password | TLS client authentication |
| **AWS Credentials** | Access Key ID + Secret Access Key (via plugin) | AWS deployments |

### Credential Scopes

Scopes control **where** a credential can be used:

| Scope | Visibility | Best For |
|-------|-----------|----------|
| **Global** | Available to all jobs across Jenkins | Shared credentials (Docker Hub, Git) |
| **System** | Only available to Jenkins system (not jobs) | Jenkins internal use (SMTP, LDAP) |
| **Folder** | Available only to jobs inside that folder | Team-specific secrets |

**Think of it like this:**

```
┌─────────────────────────────────────────────┐
│ Jenkins Instance                            │
│                                             │
│  🔑 Global Credentials                     │
│  ├── docker-hub-creds (all jobs can use)    │
│  └── github-token (all jobs can use)        │
│                                             │
│  🔒 System Credentials                     │
│  ├── smtp-password (Jenkins internal only)  │
│  └── ldap-bind-password (Jenkins only)      │
│                                             │
│  📁 Folder: "frontend-team"                │
│  │  🔑 Folder Credentials                  │
│  │  └── frontend-deploy-key (only frontend) │
│  │                                          │
│  📁 Folder: "backend-team"                 │
│  │  🔑 Folder Credentials                  │
│  │  └── backend-db-password (only backend)  │
│  │                                          │
└─────────────────────────────────────────────┘
```

> 💡 **Best Practice:** Use the **narrowest scope possible**. Don't make everything Global. If only the backend team needs database credentials, put them in the backend folder.

## 9️⃣ Using Credentials in Pipelines

Now the fun part — how to actually **use** these credentials in your Jenkinsfile!

### Method 1: `credentials()` Helper in `environment` Block

The simplest way. Works great for environment variables.

```groovy
pipeline {
    agent any
    
    environment {
        // For "Username with password" type
        // This creates THREE variables:
        // DOCKER_CREDS       → username:password
        // DOCKER_CREDS_USR   → username only
        // DOCKER_CREDS_PSW   → password only
        DOCKER_CREDS = credentials('docker-hub-credentials')
        
        // For "Secret text" type
        // Creates ONE variable with the secret value
        API_TOKEN = credentials('my-api-token')
        
        // For "Secret file" type
        // Creates ONE variable with the file path
        KUBECONFIG = credentials('my-kubeconfig')
    }
    
    stages {
        stage('Login to Docker') {
            steps {
                sh '''
                    echo $DOCKER_CREDS_PSW | docker login -u $DOCKER_CREDS_USR --password-stdin
                '''
            }
        }
        
        stage('Call API') {
            steps {
                sh '''
                    curl -H "Authorization: Bearer $API_TOKEN" https://api.example.com/data
                '''
            }
        }
        
        stage('Deploy to K8s') {
            steps {
                sh '''
                    kubectl --kubeconfig=$KUBECONFIG apply -f deployment.yaml
                '''
            }
        }
    }
}
```

**How the `credentials()` helper works — explained simply:**

```
credentials('credential-id') automatically does:

1. Fetches the credential from Jenkins' encrypted store
2. Decrypts it
3. Makes it available as an environment variable
4. Masks the value in console output (shows **** instead)
5. Cleans up after the pipeline finishes

For "Username with password" type, it creates 3 variables:
   VARIABLE_NAME      → "username:password"
   VARIABLE_NAME_USR  → "username"
   VARIABLE_NAME_PSW  → "password"

For "Secret text" type, it creates 1 variable:
   VARIABLE_NAME      → "the-secret-value"

For "Secret file" type, it creates 1 variable:
   VARIABLE_NAME      → "/path/to/temp/file"  (a temp file with the content)
```

### Method 2: `withCredentials` Block

More flexible. Use this when you need credentials in a specific stage only (not the whole pipeline).

```groovy
pipeline {
    agent any
    
    stages {
        stage('Deploy') {
            steps {
                // Username with password
                withCredentials([usernamePassword(
                    credentialsId: 'docker-hub-credentials',
                    usernameVariable: 'DOCKER_USER',
                    passwordVariable: 'DOCKER_PASS'
                )]) {
                    sh 'echo $DOCKER_PASS | docker login -u $DOCKER_USER --password-stdin'
                    sh 'docker push myapp:latest'
                }
                // After this block, DOCKER_USER and DOCKER_PASS are gone
            }
        }
        
        stage('SSH Deploy') {
            steps {
                // SSH Key
                withCredentials([sshUserPrivateKey(
                    credentialsId: 'production-ssh-key',
                    keyFileVariable: 'SSH_KEY',
                    usernameVariable: 'SSH_USER'
                )]) {
                    sh '''
                        ssh -i $SSH_KEY -o StrictHostKeyChecking=no $SSH_USER@production.example.com \
                        "cd /app && docker-compose pull && docker-compose up -d"
                    '''
                }
            }
        }
        
        stage('Cloud Deploy') {
            steps {
                // Secret text
                withCredentials([string(
                    credentialsId: 'aws-secret-key',
                    variable: 'AWS_SECRET'
                )]) {
                    sh 'aws s3 sync ./dist s3://my-bucket'
                }
            }
        }
        
        stage('Use Config File') {
            steps {
                // Secret file
                withCredentials([file(
                    credentialsId: 'my-kubeconfig',
                    variable: 'KUBECONFIG'
                )]) {
                    sh 'kubectl get pods'
                }
            }
        }
    }
}
```

**Why `withCredentials` is often better than `environment`:**
- Credentials are only available inside the `withCredentials` block
- Follows the **principle of least privilege** — secrets are available only when needed
- Variable names are explicit and clear
- Better for auditing and debugging

### Method 3: Credentials in Shared Libraries

For enterprise setups, you might use shared libraries to standardize credential usage:

```groovy
// vars/dockerDeploy.groovy (Shared Library)
def call(Map config) {
    withCredentials([usernamePassword(
        credentialsId: config.credentialsId ?: 'docker-hub-credentials',
        usernameVariable: 'DOCKER_USER',
        passwordVariable: 'DOCKER_PASS'
    )]) {
        sh "echo $DOCKER_PASS | docker login -u $DOCKER_USER --password-stdin ${config.registry}"
        sh "docker push ${config.image}:${config.tag}"
    }
}
```

```groovy
// Jenkinsfile (uses the shared library)
@Library('my-shared-lib') _

pipeline {
    agent any
    stages {
        stage('Push Image') {
            steps {
                dockerDeploy(
                    registry: 'registry.example.com',
                    image: 'my-app',
                    tag: 'v1.2.3',
                    credentialsId: 'private-registry-creds'
                )
            }
        }
    }
}
```

## 🔟 Security Best Practices — The Complete Checklist

Here's everything you should do to harden your Jenkins setup:

### Authentication Best Practices

```
✅ Enable authentication (never leave Jenkins open)
✅ Use SSO/LDAP/OAuth for centralized identity management
✅ Enforce strong passwords (if using Jenkins user database)
✅ Enable Multi-Factor Authentication (MFA) via identity provider
✅ Disable "Allow users to sign up" (prevent self-registration)
✅ Set session timeout to a reasonable value (e.g., 30 minutes)
```

### Authorization Best Practices

```
✅ Use Role-Based Access Control (RBAC) — not matrix for large teams
✅ Follow Principle of Least Privilege — give minimum required permissions
✅ Never give Anonymous users any access in production
✅ Regularly audit user permissions (remove ex-employees!)
✅ Use folder-based security to isolate team projects
✅ Restrict who can configure/create jobs vs who can just trigger them
```

### Credentials Best Practices

```
✅ NEVER hardcode secrets in Jenkinsfiles, scripts, or config
✅ Use the narrowest credential scope possible (Folder > Global)
✅ Rotate credentials regularly (every 90 days)
✅ Use withCredentials() instead of environment block when possible
✅ Audit credential usage — who uses what
✅ Use external secret managers (HashiCorp Vault, AWS Secrets Manager)
   for the highest security
✅ Delete unused credentials
```

### Infrastructure Security

```
✅ Run Jenkins behind a reverse proxy (Nginx/HAProxy) with TLS
✅ Use HTTPS everywhere — no plain HTTP
✅ Keep Jenkins and all plugins updated
✅ Disable Jenkins CLI (it's an attack vector)
✅ Enable CSRF Protection (enabled by default, don't disable it!)
✅ Restrict access to Jenkins from trusted networks only (firewall rules)
✅ Run Jenkins as a non-root user
✅ Enable audit logging
```

### Pipeline Security

```
✅ Use "Authorize Project" plugin to run jobs with specific user permissions
✅ Disable script approval for untrusted users
✅ Review Pipeline Groovy scripts for security issues
✅ Use Folder-level credential scoping
✅ Pin plugin versions and review before updating
✅ Enable the Script Security plugin (sandbox Groovy scripts)
```

### How to Disable the Jenkins CLI (Important!)

The Jenkins CLI is often used as an attack vector. Disable it:

```
Manage Jenkins → Security → CLI → Disable CLI over Remoting
```

Or in `jenkins.yaml` (JCasC - Jenkins Configuration as Code):

```yaml
jenkins:
  disableRememberMe: false
  remotingSecurity:
    enabled: true

security:
  cli:
    enabled: false
```

## 1️⃣1️⃣ Common Security Misconfigurations to Avoid

Here are the most common security mistakes people make with Jenkins — and how to fix them:

### Mistake 1: "Anyone can do anything" in production

```
❌ Authorization: Anyone can do anything

✅ Fix: Use Role-Based Strategy or Matrix-based security
```

### Mistake 2: Secrets in console output

```groovy
// ❌ This will print the secret in the build log!
sh "echo ${API_KEY}"
sh "echo My password is ${PASSWORD}"

// ✅ Jenkins automatically masks credentials() variables
//    But be careful with custom scripts that might log values
```

### Mistake 3: Running Jenkins as root

```bash
# ❌ Never run Jenkins as root
sudo java -jar jenkins.war

# ✅ Run as a dedicated jenkins user
sudo -u jenkins java -jar jenkins.war

# ✅ Or use the systemd service (runs as jenkins user by default)
sudo systemctl start jenkins
```

### Mistake 4: Jenkins exposed to the public internet without protection

```
❌ Jenkins at http://your-server:8080 — no firewall, no TLS

✅ Fix:
   1. Put Jenkins behind Nginx/HAProxy with TLS
   2. Restrict access via firewall (allow only office IPs or VPN)
   3. Use a VPN or bastion host to access Jenkins
```

**Example: Nginx reverse proxy for Jenkins:**

```nginx
server {
    listen 443 ssl;
    server_name jenkins.example.com;

    ssl_certificate     /etc/ssl/certs/jenkins.crt;
    ssl_certificate_key /etc/ssl/private/jenkins.key;

    location / {
        proxy_pass http://127.0.0.1:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto https;

        # WebSocket support for Jenkins
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
}
```

### Mistake 5: Not securing the Jenkins agent communication

```
❌ Agents connect to controller without encryption

✅ Fix: Use TCP with JNLP4-connect protocol (encrypted)
   Manage Jenkins → Security → Agents → Agent protocols
   Enable: JNLP4-connect (Java Web Start Agent protocol/4)
   Disable: JNLP-connect, JNLP2-connect (older, insecure)
```

### Mistake 6: Using inline script approvals without review

When a Groovy script needs approval:

```
Manage Jenkins → In-process Script Approval
```

**Always review what you're approving!** Malicious scripts can:
- Read the file system
- Execute system commands
- Access credentials
- Open network connections

## 🔗 Integrating with External Secret Managers

For enterprise-grade security, use external secret managers instead of (or alongside) Jenkins' built-in credential store:

### HashiCorp Vault Integration

Install the "HashiCorp Vault" plugin, then:

```groovy
pipeline {
    agent any
    stages {
        stage('Get Secrets from Vault') {
            steps {
                withVault(
                    configuration: [
                        vaultUrl: 'https://vault.example.com',
                        vaultCredentialId: 'vault-approle'
                    ],
                    vaultSecrets: [
                        [
                            path: 'secret/data/myapp',
                            secretValues: [
                                [envVar: 'DB_PASSWORD', vaultKey: 'db_password'],
                                [envVar: 'API_KEY', vaultKey: 'api_key']
                            ]
                        ]
                    ]
                ) {
                    sh 'echo "Database password is masked: $DB_PASSWORD"'
                    sh './deploy.sh'
                }
            }
        }
    }
}
```

### AWS Secrets Manager Integration

```groovy
pipeline {
    agent any
    stages {
        stage('Deploy') {
            steps {
                withAWSParameterStore(
                    credentialsId: 'aws-credentials',
                    regionName: 'us-east-1',
                    naming: 'basename',
                    path: '/myapp/production/'
                ) {
                    sh 'echo $DB_HOST'
                    sh './deploy.sh'
                }
            }
        }
    }
}
```

## 🎯 Key Takeaways

1. **Enable security immediately** — default Jenkins is wide open
2. **Use RBAC** (Role-Based Strategy plugin) for managing permissions at scale
3. **Never hardcode secrets** — always use Jenkins Credentials Store or external secret managers
4. **Use `withCredentials()`** for the most secure way to use secrets in pipelines
5. **Follow least privilege** — give users only what they need, nothing more
6. **Put Jenkins behind HTTPS** with a reverse proxy — never expose port 8080 directly
7. **Disable the CLI** and secure agent communication
8. **Rotate credentials** regularly and audit who has access
9. **Use folder-scoped credentials** to isolate team secrets
10. **Keep Jenkins and plugins updated** — security patches are released frequently
