# 11.11 — Jenkins — Scaling with Kubernetes (Jenkins on K8s)

> **One Line Answer:** Running Jenkins on Kubernetes means your **Jenkins controller lives inside a K8s cluster**, and every time a build triggers, Kubernetes **spins up a fresh pod as a build agent**, runs the job, and then **destroys the pod** — giving you **infinite scalability** and **zero idle resources**.

---

## 🤔 Why Run Jenkins on Kubernetes?

### The Problem with Traditional Jenkins

In a traditional Jenkins setup, you have:

```
┌────────────────────────────────────────────────────┐
│ Jenkins Controller (Master)                        │
│   - Manages jobs, schedules builds, stores config  │
│                                                    │
│   Permanent Agents (always running):               │
│   ┌──────────┐ ┌──────────┐ ┌──────────┐          │
│   │ Agent 1  │ │ Agent 2  │ │ Agent 3  │          │
│   │ (idle)   │ │ (busy)   │ │ (idle)   │          │
│   └──────────┘ └──────────┘ └──────────┘          │
│                                                    │
│   Cost: You pay for ALL agents 24/7               │
│   Even when they're doing NOTHING!                │
└────────────────────────────────────────────────────┘
```

**Problems with this approach:**

| Problem | Why It Hurts |
|---------|-------------|
| **Wasted resources** | Agents sit idle 70-80% of the time, but you pay for them 24/7 |
| **Scaling is manual** | Need more capacity? Manually provision new VMs, install Java, configure agents |
| **Scaling is slow** | Takes 10-15 minutes to bring up a new agent VM |
| **Inconsistent environments** | Agents accumulate "drift" — different packages, versions, leftover files |
| **Maintenance burden** | You have to patch, update, and maintain every agent machine |
| **Fixed capacity** | 10 agents means max 10 parallel builds. Rush hour? Builds queue up |

### The Kubernetes Solution

With Jenkins on Kubernetes:

```
┌────────────────────────────────────────────────────────────┐
│ Kubernetes Cluster                                         │
│                                                            │
│  ┌──────────────────────────────────────────┐              │
│  │ Jenkins Controller (Pod)                 │              │
│  │  - Manages jobs, schedules builds        │              │
│  │  - Has Kubernetes Plugin installed        │              │
│  └──────────────────┬───────────────────────┘              │
│                     │                                      │
│     When a build triggers, K8s creates a pod:              │
│                     │                                      │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐                   │
│  │ Agent    │ │ Agent    │ │ Agent    │ ← Created on      │
│  │ Pod 1   │ │ Pod 2   │ │ Pod 3   │   demand!           │
│  │ (build) │ │ (build) │ │ (build) │                     │
│  └──────────┘ └──────────┘ └──────────┘                   │
│       │              │            │                        │
│       ▼              ▼            ▼                        │
│   Build done     Build done   Build done                   │
│   Pod deleted!   Pod deleted! Pod deleted!                  │
│                                                            │
│   Cost: You pay ONLY for what you use!                    │
└────────────────────────────────────────────────────────────┘
```

**Benefits:**

| Benefit | Description |
|---------|-------------|
| **Auto-scaling** | Need 50 builds simultaneously? K8s spins up 50 agent pods |
| **Zero idle cost** | Pods are destroyed after the build — no wasted resources |
| **Clean environments** | Every build gets a fresh pod — no leftover state, no drift |
| **Fast spin-up** | New agent pod starts in 10-30 seconds (vs 10-15 min for a VM) |
| **No maintenance** | No OS patching, no manual updates — just update the container image |
| **Custom environments** | Different jobs can use different container images (Node, Python, Go, Java...) |
| **Built-in HA** | Kubernetes handles controller pod restarts automatically |

> 💡 **Think of it like this:** Traditional Jenkins agents are like **owning cars** that sit in your garage most of the time. Jenkins on K8s is like **Uber** — a car shows up when you need it and goes away when you're done. You only pay for the ride.

## 1️⃣ Architecture — How Jenkins on K8s Works

Let's understand every component:

```
┌─────────────────────────────────────────────────────────────────┐
│                    KUBERNETES CLUSTER                            │
│                                                                 │
│  Namespace: jenkins                                             │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │                                                           │  │
│  │  ┌─────────────────────────────────────┐                  │  │
│  │  │ Jenkins Controller Pod              │                  │  │
│  │  │ ┌─────────────────────────────────┐ │                  │  │
│  │  │ │ jenkins/jenkins:lts Container   │ │                  │  │
│  │  │ │                                 │ │                  │  │
│  │  │ │ - Jenkins Core (Java app)       │ │                  │  │
│  │  │ │ - Kubernetes Plugin             │ │                  │  │
│  │  │ │ - Job definitions               │ │                  │  │
│  │  │ │ - Build history                 │ │                  │  │
│  │  │ │ - Credentials store             │ │                  │  │
│  │  │ └─────────────────────────────────┘ │                  │  │
│  │  │                                     │                  │  │
│  │  │ Volume: /var/jenkins_home (PVC)     │                  │  │
│  │  └───────────────┬─────────────────────┘                  │  │
│  │                  │                                        │  │
│  │    Kubernetes API │ (creates/deletes pods)                │  │
│  │                  │                                        │  │
│  │    ┌─────────────▼─────────────┐                          │  │
│  │    │ Agent Pods (dynamic)      │                          │  │
│  │    │                           │                          │  │
│  │    │  ┌─────┐ ┌─────┐ ┌─────┐│                          │  │
│  │    │  │JNLP │ │Maven│ │Docker││                          │  │
│  │    │  │Agent│ │Cont.│ │Cont. ││                          │  │
│  │    │  └─────┘ └─────┘ └─────┘│                          │  │
│  │    │  (multi-container pod)   │                          │  │
│  │    └──────────────────────────┘                          │  │
│  │                                                           │  │
│  └───────────────────────────────────────────────────────────┘  │
│                                                                 │
│  Services:                                                      │
│  ├── jenkins-controller (ClusterIP/LoadBalancer) → UI :8080     │
│  └── jenkins-agent (ClusterIP) → JNLP agents :50000            │
│                                                                 │
│  PersistentVolumeClaim:                                         │
│  └── jenkins-pvc → /var/jenkins_home                            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### How the Flow Works (Step by Step)

```
1. Developer pushes code to GitHub
         │
         ▼
2. Webhook triggers Jenkins pipeline
         │
         ▼
3. Jenkins Controller receives the trigger
         │
         ▼
4. Controller sees the pipeline needs a Kubernetes agent
         │
         ▼
5. Controller calls Kubernetes API:
   "Hey K8s, please create a pod with these containers"
         │
         ▼
6. Kubernetes creates the Agent Pod:
   - JNLP container (connects back to Jenkins controller)
   - Build containers (maven, docker, node, etc.)
         │
         ▼
7. Agent Pod starts and connects to Jenkins via JNLP
         │
         ▼
8. Jenkins runs the pipeline steps inside the agent pod
         │
         ▼
9. Build completes (success or failure)
         │
         ▼
10. Jenkins tells Kubernetes: "Delete that pod"
         │
         ▼
11. Pod is destroyed — resources freed!
```

## 2️⃣ Installing Jenkins on Kubernetes (Using Helm)

Helm is the easiest way to deploy Jenkins on Kubernetes. Let's do it step by step.

### Prerequisites

```bash
# You need:
# 1. A running Kubernetes cluster (minikube, kind, EKS, AKS, GKE)
# 2. kubectl configured to talk to your cluster
# 3. Helm 3 installed

# Verify everything is ready
kubectl cluster-info
helm version
```

### Step 1: Add the Jenkins Helm Repository

```bash
# Add the official Jenkins Helm chart repository
helm repo add jenkins https://charts.jenkins.io

# Update repo index
helm repo update
```

### Step 2: Create a Namespace

```bash
# Create a dedicated namespace for Jenkins
kubectl create namespace jenkins
```

### Step 3: Create a Custom Values File

This is the most important step — customizing Jenkins for your needs.

```yaml
# jenkins-values.yaml

# ============================================================
# CONTROLLER CONFIGURATION
# ============================================================
controller:
  # Jenkins image
  image:
    registry: "docker.io"
    repository: "jenkins/jenkins"
    tag: "lts-jdk17"
  
  # Resource limits for the controller pod
  resources:
    requests:
      cpu: "500m"
      memory: "1Gi"
    limits:
      cpu: "2000m"
      memory: "4Gi"
  
  # Service type — how to access Jenkins UI
  # Use LoadBalancer for cloud, NodePort for local/minikube
  serviceType: LoadBalancer
  # servicePort: 8080
  
  # Admin user configuration
  adminUser: "admin"
  # adminPassword: "your-strong-password"  # If not set, auto-generated
  
  # Plugins to install
  installPlugins:
    - kubernetes
    - workflow-aggregator
    - git
    - configuration-as-code
    - job-dsl
    - credentials-binding
    - role-strategy
    - pipeline-stage-view
    - docker-workflow
    - blueocean
  
  # Jenkins Configuration as Code (JCasC)
  JCasC:
    defaultConfig: true
    configScripts:
      welcome-message: |
        jenkins:
          systemMessage: "Jenkins on Kubernetes - Configured via JCasC"
  
  # Ingress (optional — for production)
  # ingress:
  #   enabled: true
  #   hostName: jenkins.example.com
  #   annotations:
  #     kubernetes.io/ingress.class: nginx
  #     cert-manager.io/cluster-issuer: letsencrypt
  #   tls:
  #     - secretName: jenkins-tls
  #       hosts:
  #         - jenkins.example.com

# ============================================================
# AGENT CONFIGURATION
# ============================================================
agent:
  enabled: true
  # Default agent pod template
  image:
    repository: "jenkins/inbound-agent"
    tag: "latest"
  resources:
    requests:
      cpu: "256m"
      memory: "256Mi"
    limits:
      cpu: "1000m"
      memory: "1Gi"
  # Maximum number of concurrent agent pods
  containerCap: 20

# ============================================================
# PERSISTENCE
# ============================================================
persistence:
  enabled: true
  # StorageClass — use your cluster's default or specify one
  # storageClass: "gp2"     # AWS EBS
  # storageClass: "standard" # GKE
  size: "20Gi"
  accessMode: "ReadWriteOnce"

# ============================================================
# RBAC — Jenkins needs permissions to create/delete pods
# ============================================================
rbac:
  create: true
  readSecrets: true

serviceAccount:
  create: true
  name: "jenkins"
```

**Let's understand each section:**

| Section | What It Does |
|---------|-------------|
| `controller` | Configures the Jenkins controller pod — image, resources, plugins, admin user |
| `agent` | Default settings for dynamically created agent pods |
| `persistence` | Creates a PersistentVolumeClaim to store Jenkins data (jobs, config, build history) |
| `rbac` | Creates RBAC roles so Jenkins can interact with the Kubernetes API (create/delete pods) |
| `serviceAccount` | Creates a ServiceAccount for Jenkins to authenticate with Kubernetes |

### Step 4: Install Jenkins

```bash
# Install Jenkins using Helm with our custom values
helm install jenkins jenkins/jenkins \
  --namespace jenkins \
  --values jenkins-values.yaml \
  --wait
```

**What this command does:**
- `helm install jenkins` — install a release named "jenkins"
- `jenkins/jenkins` — from the jenkins/jenkins chart
- `--namespace jenkins` — into the jenkins namespace
- `--values jenkins-values.yaml` — using our custom configuration
- `--wait` — wait until all pods are ready

### Step 5: Get the Admin Password

```bash
# If you didn't set adminPassword in values, get the auto-generated one:
kubectl exec --namespace jenkins -it svc/jenkins -c jenkins -- \
  /bin/cat /run/secrets/additional/chart-admin-password && echo
```

### Step 6: Access Jenkins UI

```bash
# Option 1: Port forward (for testing/local)
kubectl --namespace jenkins port-forward svc/jenkins 8080:8080

# Now open: http://localhost:8080

# Option 2: Get the LoadBalancer URL (cloud)
kubectl get svc jenkins --namespace jenkins -o jsonpath='{.status.loadBalancer.ingress[0].hostname}'
```

### Verify Everything is Running

```bash
# Check pods
kubectl get pods -n jenkins

# Output:
# NAME                       READY   STATUS    RESTARTS   AGE
# jenkins-0                  2/2     Running   0          5m

# Check services
kubectl get svc -n jenkins

# Output:
# NAME            TYPE           CLUSTER-IP      EXTERNAL-IP      PORT(S)
# jenkins         LoadBalancer   10.100.200.50   a1b2c3d4...      8080:31234/TCP
# jenkins-agent   ClusterIP      10.100.200.51   <none>           50000/TCP

# Check PVC
kubectl get pvc -n jenkins

# Output:
# NAME      STATUS   VOLUME     CAPACITY   ACCESS MODES   STORAGECLASS
# jenkins   Bound    pvc-abc    20Gi       RWO            gp2
```

## 3️⃣ Configuring the Kubernetes Plugin

The **Kubernetes Plugin** is the magic that connects Jenkins to Kubernetes. It tells Jenkins how to create agent pods.

### Plugin Configuration Location

```
Manage Jenkins → Clouds → New cloud → Kubernetes
```

### Key Settings

```yaml
# These settings tell Jenkins how to talk to Kubernetes

Cloud name: kubernetes

# Kubernetes URL — where is the K8s API?
# If Jenkins is INSIDE the cluster (which it is), leave blank or use:
Kubernetes URL: https://kubernetes.default.svc.cluster.local

# Kubernetes Namespace — where to create agent pods
Kubernetes Namespace: jenkins

# Jenkins URL — how agents connect back to Jenkins
Jenkins URL: http://jenkins.jenkins.svc.cluster.local:8080

# Jenkins Tunnel — for JNLP agent communication
Jenkins tunnel: jenkins-agent.jenkins.svc.cluster.local:50000

# Concurrency Limit — max agent pods at once
Container Cap: 20

# Pod timeout — how long to wait for a pod to start
Pod Retention: Never (always delete after build)
```

**How does Jenkins authenticate with Kubernetes?**

When Jenkins runs inside the cluster, it uses the **ServiceAccount** automatically mounted in the pod. The Helm chart creates the necessary RBAC roles:

```yaml
# This is what Helm creates for you automatically:
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: jenkins
rules:
  - apiGroups: [""]
    resources: ["pods", "pods/exec", "pods/log"]
    verbs: ["get", "list", "watch", "create", "update", "delete"]
  - apiGroups: [""]
    resources: ["secrets"]
    verbs: ["get", "list"]
  - apiGroups: [""]
    resources: ["persistentvolumeclaims"]
    verbs: ["get", "list", "create", "delete"]
  - apiGroups: [""]
    resources: ["events"]
    verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: jenkins
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: jenkins
subjects:
  - kind: ServiceAccount
    name: jenkins
    namespace: jenkins
```

> 💡 **Translation:** This gives the Jenkins ServiceAccount permission to create, list, and delete pods in the cluster — which is exactly what it needs to spin up and tear down agent pods.

## 4️⃣ Pod Templates — Defining Agent Environments

Pod Templates define **what your agent pods look like** — which containers they run, what resources they get, and what volumes they mount.

### Configuring Pod Templates in the UI

```
Manage Jenkins → Clouds → kubernetes → Pod Templates → Add Pod Template
```

### Example: A Maven Build Agent

```
Name: maven
Labels: maven
Usage: Use this node as much as possible

Containers:
  Container 1:
    Name: jnlp
    Image: jenkins/inbound-agent:latest
    (This is the JNLP agent that connects to Jenkins)

  Container 2:
    Name: maven
    Image: maven:3.9-eclipse-temurin-17
    Command to run: sleep
    Arguments: infinity
    (This keeps the container alive so Jenkins can exec into it)
```

### Configuring Pod Templates via JCasC (Recommended)

Instead of clicking through the UI, define pod templates in code:

```yaml
# jenkins-values.yaml (add to the controller.JCasC.configScripts section)
controller:
  JCasC:
    configScripts:
      cloud-config: |
        jenkins:
          clouds:
            - kubernetes:
                name: "kubernetes"
                serverUrl: "https://kubernetes.default.svc.cluster.local"
                namespace: "jenkins"
                jenkinsUrl: "http://jenkins.jenkins.svc.cluster.local:8080"
                jenkinsTunnel: "jenkins-agent.jenkins.svc.cluster.local:50000"
                containerCap: 20
                podRetention: "never"
                templates:
                  # Template 1: Basic JNLP agent
                  - name: "default"
                    label: "default"
                    nodeUsageMode: "NORMAL"
                    containers:
                      - name: "jnlp"
                        image: "jenkins/inbound-agent:latest"
                        resourceRequestCpu: "256m"
                        resourceRequestMemory: "256Mi"
                        resourceLimitCpu: "500m"
                        resourceLimitMemory: "512Mi"
                  
                  # Template 2: Maven build agent
                  - name: "maven"
                    label: "maven"
                    nodeUsageMode: "EXCLUSIVE"
                    containers:
                      - name: "jnlp"
                        image: "jenkins/inbound-agent:latest"
                        resourceRequestCpu: "256m"
                        resourceRequestMemory: "256Mi"
                      - name: "maven"
                        image: "maven:3.9-eclipse-temurin-17"
                        command: "sleep"
                        args: "infinity"
                        resourceRequestCpu: "500m"
                        resourceRequestMemory: "512Mi"
                        resourceLimitCpu: "2000m"
                        resourceLimitMemory: "2Gi"
                    volumes:
                      - hostPathVolume:
                          mountPath: "/root/.m2/repository"
                          hostPath: "/tmp/maven-cache"
                  
                  # Template 3: Node.js agent
                  - name: "nodejs"
                    label: "nodejs"
                    nodeUsageMode: "EXCLUSIVE"
                    containers:
                      - name: "jnlp"
                        image: "jenkins/inbound-agent:latest"
                      - name: "node"
                        image: "node:20-alpine"
                        command: "sleep"
                        args: "infinity"
                        resourceRequestCpu: "500m"
                        resourceRequestMemory: "512Mi"
                  
                  # Template 4: Docker-in-Docker agent
                  - name: "docker"
                    label: "docker"
                    nodeUsageMode: "EXCLUSIVE"
                    containers:
                      - name: "jnlp"
                        image: "jenkins/inbound-agent:latest"
                      - name: "docker"
                        image: "docker:24-dind"
                        privileged: true
                        command: ""
                        args: ""
                        resourceRequestCpu: "500m"
                        resourceRequestMemory: "512Mi"
                    volumes:
                      - emptyDirVolume:
                          mountPath: "/var/lib/docker"
                          memory: false
```

**Understanding the pod template fields:**

| Field | What It Does |
|-------|-------------|
| `name` | Name of the template |
| `label` | Label used in Jenkinsfile to select this template (`agent { label 'maven' }`) |
| `nodeUsageMode` | `NORMAL` = use for any job, `EXCLUSIVE` = only when label matches |
| `containers` | List of containers in the pod |
| `jnlp` container | The Jenkins agent that communicates with the controller (always needed) |
| Other containers | Your build tools (maven, node, docker, etc.) |
| `command: sleep` + `args: infinity` | Keeps the container running so Jenkins can exec commands into it |
| `volumes` | Mount volumes for caching, Docker socket, etc. |

## 5️⃣ Writing Pipelines with Kubernetes Agents

Now the fun part — writing Jenkinsfiles that use Kubernetes agents!

### Method 1: Using Pod Template Labels

Reference the label you defined in pod templates:

```groovy
pipeline {
    // Use the "maven" pod template we defined earlier
    agent {
        label 'maven'
    }
    
    stages {
        stage('Build') {
            steps {
                // This runs inside the "maven" container
                container('maven') {
                    sh 'mvn --version'
                    sh 'mvn clean package -DskipTests'
                }
            }
        }
        
        stage('Test') {
            steps {
                container('maven') {
                    sh 'mvn test'
                }
            }
        }
    }
}
```

### Method 2: Inline Pod Definition with `kubernetes` Agent (Most Flexible)

Define the pod template directly in the Jenkinsfile:

```groovy
pipeline {
    agent {
        kubernetes {
            // Optional: give the pod a meaningful name
            label 'my-build-pod'
            
            // Define the pod inline using YAML
            yaml '''
apiVersion: v1
kind: Pod
metadata:
  labels:
    app: jenkins-agent
spec:
  containers:
    - name: maven
      image: maven:3.9-eclipse-temurin-17
      command:
        - sleep
      args:
        - infinity
      resources:
        requests:
          memory: "512Mi"
          cpu: "500m"
        limits:
          memory: "2Gi"
          cpu: "2000m"
      volumeMounts:
        - name: maven-cache
          mountPath: /root/.m2/repository
    
    - name: docker
      image: docker:24-cli
      command:
        - sleep
      args:
        - infinity
      env:
        - name: DOCKER_HOST
          value: tcp://localhost:2376
        - name: DOCKER_TLS_VERIFY
          value: "1"
        - name: DOCKER_CERT_PATH
          value: /certs/client
      volumeMounts:
        - name: docker-certs
          mountPath: /certs/client
          readOnly: true
    
    - name: dind
      image: docker:24-dind
      securityContext:
        privileged: true
      env:
        - name: DOCKER_TLS_CERTDIR
          value: /certs
      volumeMounts:
        - name: docker-certs
          mountPath: /certs
        - name: docker-storage
          mountPath: /var/lib/docker
    
    - name: kubectl
      image: bitnami/kubectl:latest
      command:
        - sleep
      args:
        - infinity
  
  volumes:
    - name: maven-cache
      emptyDir: {}
    - name: docker-certs
      emptyDir: {}
    - name: docker-storage
      emptyDir: {}
'''
        }
    }
    
    stages {
        stage('Build') {
            steps {
                container('maven') {
                    sh 'mvn clean package -DskipTests'
                }
            }
        }
        
        stage('Build Docker Image') {
            steps {
                container('docker') {
                    sh 'docker build -t myapp:${BUILD_NUMBER} .'
                    sh 'docker tag myapp:${BUILD_NUMBER} registry.example.com/myapp:${BUILD_NUMBER}'
                }
            }
        }
        
        stage('Push Image') {
            steps {
                container('docker') {
                    withCredentials([usernamePassword(
                        credentialsId: 'docker-registry-creds',
                        usernameVariable: 'DOCKER_USER',
                        passwordVariable: 'DOCKER_PASS'
                    )]) {
                        sh 'echo $DOCKER_PASS | docker login -u $DOCKER_USER --password-stdin registry.example.com'
                        sh 'docker push registry.example.com/myapp:${BUILD_NUMBER}'
                    }
                }
            }
        }
        
        stage('Deploy to K8s') {
            steps {
                container('kubectl') {
                    withCredentials([file(credentialsId: 'kubeconfig', variable: 'KUBECONFIG')]) {
                        sh '''
                            kubectl set image deployment/myapp \
                              myapp=registry.example.com/myapp:${BUILD_NUMBER} \
                              --namespace production
                            kubectl rollout status deployment/myapp --namespace production
                        '''
                    }
                }
            }
        }
    }
}
```

**What's happening here — explained step by step:**

```
1. Pipeline triggers
2. Jenkins creates a Pod with 4 containers:
   ┌──────────────────────────────────────────────────┐
   │ Agent Pod                                        │
   │                                                  │
   │  ┌──────────┐  ┌──────────┐  ┌──────────┐      │
   │  │  maven   │  │  docker  │  │  kubectl │      │
   │  │  (build) │  │  (cli)   │  │  (deploy)│      │
   │  └──────────┘  └──────────┘  └──────────┘      │
   │       │              │                           │
   │       │         ┌──────────┐                    │
   │       │         │   dind   │ ← Docker daemon    │
   │       │         │(Docker-  │   (for building    │
   │       │         │ in-Docker│    images)          │
   │       │         └──────────┘                    │
   │                                                  │
   │  + JNLP container (auto-added by Jenkins)       │
   └──────────────────────────────────────────────────┘

3. Each stage uses container('name') to specify which
   container to run commands in:
   - Stage 'Build' → runs in 'maven' container
   - Stage 'Build Docker Image' → runs in 'docker' container
   - Stage 'Deploy' → runs in 'kubectl' container

4. After all stages complete → Pod is destroyed
```

### Method 3: Using `podTemplate` Step (Scripted Pipeline)

```groovy
// Scripted Pipeline style
podTemplate(
    label: 'my-pod',
    containers: [
        containerTemplate(
            name: 'golang',
            image: 'golang:1.21',
            command: 'sleep',
            args: 'infinity'
        ),
        containerTemplate(
            name: 'docker',
            image: 'docker:24-cli',
            command: 'sleep',
            args: 'infinity'
        )
    ],
    volumes: [
        hostPathVolume(
            mountPath: '/var/run/docker.sock',
            hostPath: '/var/run/docker.sock'
        )
    ]
) {
    node('my-pod') {
        stage('Build Go App') {
            container('golang') {
                git url: 'https://github.com/example/go-app.git', branch: 'main'
                sh 'go build -o app .'
                sh 'go test ./...'
            }
        }
        
        stage('Containerize') {
            container('docker') {
                sh 'docker build -t go-app:latest .'
            }
        }
    }
}
```

## 6️⃣ Multi-Container Pods — The Sidecar Pattern

One of the most powerful features of Jenkins on K8s is running **multiple containers in a single pod**. Each container has a specific job:

### Common Multi-Container Patterns

```
┌─────────────────────────────────────────────────────────┐
│ Agent Pod                                               │
│                                                         │
│  ┌──────────┐  Main build tool                         │
│  │  jnlp    │  Jenkins agent (always present)          │
│  └──────────┘                                          │
│                                                         │
│  ┌──────────┐  Build tool                              │
│  │  maven   │  Compiles code, runs tests               │
│  └──────────┘                                          │
│                                                         │
│  ┌──────────┐  Container builder                       │
│  │  kaniko  │  Builds Docker images WITHOUT Docker     │
│  └──────────┘  daemon (more secure than DinD!)         │
│                                                         │
│  ┌──────────┐  Deployment tool                         │
│  │  kubectl │  Deploys to Kubernetes                   │
│  └──────────┘                                          │
│                                                         │
│  ┌──────────┐  Code quality                            │
│  │  sonar   │  SonarQube scanner                       │
│  └──────────┘                                          │
│                                                         │
│  All containers share the same network (localhost)      │
│  and can share volumes!                                │
└─────────────────────────────────────────────────────────┘
```

### Real-World Example: Full CI/CD Pipeline with Kaniko

Kaniko builds Docker images **without** needing a Docker daemon — much more secure than Docker-in-Docker!

```groovy
pipeline {
    agent {
        kubernetes {
            yaml '''
apiVersion: v1
kind: Pod
spec:
  containers:
    - name: maven
      image: maven:3.9-eclipse-temurin-17
      command: ["sleep"]
      args: ["infinity"]
      volumeMounts:
        - name: maven-cache
          mountPath: /root/.m2/repository
    
    - name: kaniko
      image: gcr.io/kaniko-project/executor:debug
      command: ["sleep"]
      args: ["infinity"]
      volumeMounts:
        - name: docker-config
          mountPath: /kaniko/.docker
    
    - name: kubectl
      image: bitnami/kubectl:latest
      command: ["sleep"]
      args: ["infinity"]
  
  volumes:
    - name: maven-cache
      persistentVolumeClaim:
        claimName: maven-cache-pvc
    - name: docker-config
      secret:
        secretName: docker-registry-credentials
        items:
          - key: .dockerconfigjson
            path: config.json
'''
        }
    }
    
    environment {
        REGISTRY = 'registry.example.com'
        IMAGE_NAME = 'myapp'
        IMAGE_TAG = "${BUILD_NUMBER}"
    }
    
    stages {
        stage('Checkout') {
            steps {
                git branch: 'main', url: 'https://github.com/example/myapp.git'
            }
        }
        
        stage('Build & Test') {
            steps {
                container('maven') {
                    sh 'mvn clean verify'
                }
            }
            post {
                always {
                    junit '**/target/surefire-reports/*.xml'
                }
            }
        }
        
        stage('Build & Push Image') {
            steps {
                container('kaniko') {
                    sh """
                        /kaniko/executor \
                          --context=\$(pwd) \
                          --dockerfile=Dockerfile \
                          --destination=${REGISTRY}/${IMAGE_NAME}:${IMAGE_TAG} \
                          --destination=${REGISTRY}/${IMAGE_NAME}:latest \
                          --cache=true \
                          --cache-repo=${REGISTRY}/${IMAGE_NAME}/cache
                    """
                }
            }
        }
        
        stage('Deploy') {
            steps {
                container('kubectl') {
                    sh """
                        kubectl set image deployment/${IMAGE_NAME} \
                          ${IMAGE_NAME}=${REGISTRY}/${IMAGE_NAME}:${IMAGE_TAG} \
                          -n production
                        kubectl rollout status deployment/${IMAGE_NAME} -n production --timeout=300s
                    """
                }
            }
        }
    }
}
```

## 7️⃣ Persistent Storage for Jenkins Controller

The Jenkins controller needs persistent storage for:
- Job configurations
- Build history
- Plugin data
- Credentials
- User configuration

**Without persistent storage, you'll lose everything when the pod restarts!**

### PersistentVolumeClaim (Handled by Helm)

The Helm chart creates this automatically:

```yaml
# This is what Helm creates:
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: jenkins
  namespace: jenkins
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 20Gi
  storageClassName: gp2  # or your cluster's default
```

### Backup Strategy

**Always back up Jenkins data!** Here are common approaches:

```bash
# Method 1: Volume Snapshots (cloud providers)
# AWS EBS Snapshot
aws ec2 create-snapshot \
  --volume-id vol-abc123 \
  --description "Jenkins backup $(date +%Y%m%d)"

# Method 2: Backup to S3 using CronJob
```

```yaml
# backup-cronjob.yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: jenkins-backup
  namespace: jenkins
spec:
  schedule: "0 2 * * *"  # Every day at 2 AM
  jobTemplate:
    spec:
      template:
        spec:
          containers:
            - name: backup
              image: amazon/aws-cli:latest
              command:
                - /bin/sh
                - -c
                - |
                  tar czf /tmp/jenkins-backup-$(date +%Y%m%d).tar.gz /var/jenkins_home
                  aws s3 cp /tmp/jenkins-backup-*.tar.gz s3://my-jenkins-backups/
              volumeMounts:
                - name: jenkins-data
                  mountPath: /var/jenkins_home
                  readOnly: true
          restartPolicy: OnFailure
          volumes:
            - name: jenkins-data
              persistentVolumeClaim:
                claimName: jenkins
```

## 8️⃣ Resource Management & Scaling

### Setting Resource Limits for Agent Pods

Always set resource requests and limits to prevent one build from eating all cluster resources:

```yaml
# In pod template
containers:
  - name: maven
    image: maven:3.9
    resources:
      requests:
        cpu: "500m"      # Guaranteed 0.5 CPU
        memory: "512Mi"  # Guaranteed 512 MB RAM
      limits:
        cpu: "2000m"     # Max 2 CPUs
        memory: "2Gi"    # Max 2 GB RAM (OOMKilled if exceeded)
```

**Resource guidelines by build type:**

| Build Type | CPU Request | CPU Limit | Memory Request | Memory Limit |
|-----------|-------------|-----------|----------------|-------------|
| Simple scripts | 250m | 500m | 256Mi | 512Mi |
| Java/Maven | 500m | 2000m | 512Mi | 2Gi |
| Node.js | 500m | 1000m | 512Mi | 1Gi |
| Docker builds | 500m | 2000m | 512Mi | 2Gi |
| Large compilations | 1000m | 4000m | 1Gi | 4Gi |

### Container Cap — Limiting Concurrent Builds

```yaml
# In Kubernetes cloud configuration
agent:
  containerCap: 20  # Maximum 20 agent pods at once
```

This prevents Jenkins from creating too many pods and overwhelming the cluster.

### Cluster Autoscaler Integration

If you're using a managed Kubernetes service (EKS, AKS, GKE), enable the **Cluster Autoscaler**:

```
Jenkins needs more agents → Creates more pods → Pods pending (no resources)
    → Cluster Autoscaler detects pending pods → Adds new nodes
    → Pods get scheduled → Builds run

Builds finish → Pods deleted → Nodes idle
    → Cluster Autoscaler removes idle nodes → Saves money!
```

```yaml
# Example: EKS Node Group with autoscaling
apiVersion: eksctl.io/v1alpha5
kind: ClusterConfig
metadata:
  name: my-cluster
  region: us-east-1
nodeGroups:
  - name: jenkins-agents
    instanceType: m5.xlarge
    desiredCapacity: 2
    minSize: 1
    maxSize: 10
    labels:
      role: jenkins-agent
    tags:
      k8s.io/cluster-autoscaler/enabled: "true"
```

## 9️⃣ Jenkins Configuration as Code (JCasC) on K8s

JCasC lets you define your **entire Jenkins configuration in YAML** — no manual UI clicking needed. This is **critical** for Jenkins on K8s because:

- If the controller pod crashes and restarts, configuration is auto-applied
- Configuration is version-controlled (GitOps!)
- Reproducible across environments (dev, staging, prod Jenkins)

### Complete JCasC Example

```yaml
# jenkins-values.yaml
controller:
  JCasC:
    configScripts:
      # Jenkins system configuration
      jenkins-config: |
        jenkins:
          systemMessage: "Production Jenkins - Managed by JCasC"
          numExecutors: 0  # Controller doesn't run builds
          mode: EXCLUSIVE
          
          securityRealm:
            local:
              allowsSignup: false
              users:
                - id: "admin"
                  password: "${chart-admin-password}"
          
          authorizationStrategy:
            roleBased:
              roles:
                global:
                  - name: "admin"
                    permissions:
                      - "Overall/Administer"
                    entries:
                      - user: "admin"
                  - name: "developer"
                    permissions:
                      - "Overall/Read"
                      - "Job/Read"
                      - "Job/Build"
                      - "Job/Cancel"
                    entries:
                      - group: "developers"
                  - name: "viewer"
                    permissions:
                      - "Overall/Read"
                      - "Job/Read"
                    entries:
                      - group: "viewers"
      
      # Cloud / Kubernetes configuration
      cloud-config: |
        jenkins:
          clouds:
            - kubernetes:
                name: "kubernetes"
                namespace: "jenkins"
                jenkinsUrl: "http://jenkins.jenkins.svc.cluster.local:8080"
                jenkinsTunnel: "jenkins-agent.jenkins.svc.cluster.local:50000"
                containerCap: 30
                retentionTimeout: 5
                templates:
                  - name: "default"
                    label: "default"
                    containers:
                      - name: "jnlp"
                        image: "jenkins/inbound-agent:latest"
                        resourceRequestCpu: "256m"
                        resourceRequestMemory: "256Mi"
                  - name: "maven"
                    label: "maven"
                    containers:
                      - name: "jnlp"
                        image: "jenkins/inbound-agent:latest"
                      - name: "maven"
                        image: "maven:3.9-eclipse-temurin-17"
                        command: "sleep"
                        args: "infinity"
                        resourceRequestCpu: "500m"
                        resourceRequestMemory: "1Gi"
                  - name: "nodejs"
                    label: "nodejs"
                    containers:
                      - name: "jnlp"
                        image: "jenkins/inbound-agent:latest"
                      - name: "node"
                        image: "node:20-alpine"
                        command: "sleep"
                        args: "infinity"
      
      # Credentials
      credentials-config: |
        credentials:
          system:
            domainCredentials:
              - credentials:
                  - usernamePassword:
                      scope: GLOBAL
                      id: "github-credentials"
                      username: "${GITHUB_USER}"
                      password: "${GITHUB_TOKEN}"
                      description: "GitHub credentials"
                  - string:
                      scope: GLOBAL
                      id: "slack-webhook"
                      secret: "${SLACK_WEBHOOK_URL}"
                      description: "Slack notification webhook"
      
      # Shared pipeline libraries
      libraries-config: |
        unclassified:
          globalLibraries:
            libraries:
              - name: "shared-pipeline-lib"
                defaultVersion: "main"
                retriever:
                  modernSCM:
                    scm:
                      git:
                        remote: "https://github.com/myorg/jenkins-shared-lib.git"
                        credentialsId: "github-credentials"
```

## 🔟 Production Best Practices

### 1. Controller Should Never Run Builds

```yaml
# Set executors to 0 on the controller
controller:
  numExecutors: 0
```

The controller should ONLY manage and schedule — never run builds itself. This keeps it stable and responsive.

### 2. Use Namespaces for Isolation

```bash
# Create separate namespaces
kubectl create namespace jenkins        # For the controller
kubectl create namespace jenkins-agents # For agent pods (optional)
```

### 3. Resource Quotas for Agent Namespace

Prevent Jenkins from eating the entire cluster:

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: jenkins-agents-quota
  namespace: jenkins
spec:
  hard:
    requests.cpu: "20"
    requests.memory: "40Gi"
    limits.cpu: "40"
    limits.memory: "80Gi"
    pods: "50"
```

### 4. Network Policies

Restrict what agent pods can access:

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: jenkins-agent-network-policy
  namespace: jenkins
spec:
  podSelector:
    matchLabels:
      jenkins/label: "agent"
  policyTypes:
    - Ingress
    - Egress
  ingress: []  # No incoming traffic needed
  egress:
    - to:
        - podSelector:
            matchLabels:
              app.kubernetes.io/component: jenkins-controller
      ports:
        - port: 50000  # JNLP
        - port: 8080   # Jenkins API
    - to: []  # Allow all outbound (for pulling deps, images)
      ports:
        - port: 443
        - port: 80
```

### 5. Use Pod Priority for Critical Builds

```yaml
# Priority Class
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: critical-build
value: 1000000
globalDefault: false
description: "For production deployment builds"

---
# Use in pod template
apiVersion: v1
kind: Pod
spec:
  priorityClassName: critical-build
  containers:
    - name: maven
      image: maven:3.9
```

### 6. Monitoring Jenkins on K8s

```yaml
# Enable Prometheus metrics
controller:
  installPlugins:
    - prometheus  # Exposes /prometheus endpoint
  
  # Prometheus ServiceMonitor
  prometheus:
    enabled: true
    serviceMonitorAdditionalLabels:
      release: prometheus
```

Key metrics to monitor:

| Metric | What It Tells You |
|--------|------------------|
| `jenkins_queue_size_value` | How many builds are waiting in queue |
| `jenkins_node_count_value` | Number of active agent nodes |
| `jenkins_executor_count_value` | Total executors available |
| `jenkins_executor_in_use_value` | Executors currently busy |
| `jenkins_job_building_duration` | How long builds take |
| `jenkins_plugins_active` | Active plugin count |

## 🎯 Key Takeaways

1. **Jenkins on K8s = dynamic agents** — pods are created on demand and destroyed after builds
2. **Use Helm** to install Jenkins on Kubernetes — it handles RBAC, PVCs, and configuration
3. **Pod Templates** define what containers your agents have — Maven, Docker, Node, kubectl, etc.
4. **Use `container('name')`** in your Jenkinsfile to run commands in specific containers
5. **Kaniko** is preferred over Docker-in-Docker for building images (more secure, no privileged mode)
6. **JCasC** (Configuration as Code) makes Jenkins reproducible and GitOps-friendly
7. **Always set resource requests/limits** to prevent builds from starving the cluster
8. **Controller should never run builds** — set executors to 0
9. **Use PersistentVolumes** for Jenkins data and back them up regularly
10. **Cluster Autoscaler** + Jenkins on K8s = truly elastic CI/CD that scales from 0 to hundreds of builds
