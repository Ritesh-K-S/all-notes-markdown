## 🤔 Why GitHub Actions? What's the Big Deal?

Before GitHub Actions, if you wanted CI/CD, you had to:

1. Set up a **separate CI/CD server** (Jenkins, CircleCI, Travis CI)
2. **Connect** it to your GitHub repo (webhooks, tokens)
3. **Maintain** that server (updates, plugins, scaling)
4. **Pay** for hosting and maintenance
5. **Switch** between GitHub and your CI tool constantly

GitHub Actions changed the game by putting CI/CD **directly inside GitHub**:

```
┌──────────────────────────────────────────────────────────┐
│                    BEFORE GITHUB ACTIONS                  │
│                                                          │
│  GitHub (Code)  ←→  Jenkins/CircleCI/Travis (CI/CD)     │
│  (separate)         (separate server to maintain)        │
│                                                          │
│  • Two systems to manage                                │
│  • Webhooks to configure                                │
│  • Credentials syncing                                  │
│  • Context switching                                    │
└──────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────┐
│                    WITH GITHUB ACTIONS                    │
│                                                          │
│  GitHub (Code + CI/CD — all in one place!)               │
│                                                          │
│  • Code and pipelines in the same repo                  │
│  • No separate server to maintain                       │
│  • Built-in secrets management                          │
│  • Tight integration with PRs, Issues, Releases         │
│  • Huge marketplace of pre-built actions                │
│  • Free for public repos!                               │
└──────────────────────────────────────────────────────────┘
```

### GitHub Actions vs Jenkins — Quick Comparison

| Feature | GitHub Actions | Jenkins |
|---------|---------------|---------|
| **Hosting** | GitHub-hosted (or self-hosted) | Self-hosted (you manage) |
| **Setup** | Zero setup — just add a YAML file | Install, configure, maintain |
| **Config** | YAML files in `.github/workflows/` | Jenkinsfile + UI config |
| **Marketplace** | 20,000+ pre-built actions | 1,800+ plugins |
| **Cost** | Free for public repos, free tier for private | Free (but you pay for infra) |
| **Scaling** | Auto-scaling (GitHub-hosted runners) | Manual or K8s plugin |
| **Best for** | GitHub-hosted projects, open source | Enterprise, complex pipelines |

## 1️⃣ Core Concepts — The Building Blocks

Think of GitHub Actions like a recipe book:

```
WORKFLOW (the recipe)
  └── EVENT/TRIGGER (when to cook — "when guests arrive")
  └── JOBS (the dishes — "appetizer", "main course", "dessert")
        └── STEPS (the cooking instructions)
              └── ACTIONS (pre-made ingredients / techniques)
              └── RUN COMMANDS (custom cooking steps)
```

Let's break it down:

### The Hierarchy

```
┌─────────────────────────────────────────────────────────┐
│ WORKFLOW (.github/workflows/ci.yml)                     │
│                                                         │
│  Triggered by: push, pull_request, schedule, etc.       │
│                                                         │
│  ┌───────────────────────────────────────────────────┐  │
│  │ JOB 1: build                                     │  │
│  │  Runs on: ubuntu-latest                          │  │
│  │                                                   │  │
│  │  ┌─────────────────────────────────────────────┐  │  │
│  │  │ STEP 1: Checkout code                       │  │  │
│  │  │   uses: actions/checkout@v4  (ACTION)       │  │  │
│  │  ├─────────────────────────────────────────────┤  │  │
│  │  │ STEP 2: Setup Node.js                       │  │  │
│  │  │   uses: actions/setup-node@v4  (ACTION)     │  │  │
│  │  ├─────────────────────────────────────────────┤  │  │
│  │  │ STEP 3: Install dependencies                │  │  │
│  │  │   run: npm install  (SHELL COMMAND)         │  │  │
│  │  ├─────────────────────────────────────────────┤  │  │
│  │  │ STEP 4: Run tests                           │  │  │
│  │  │   run: npm test  (SHELL COMMAND)            │  │  │
│  │  └─────────────────────────────────────────────┘  │  │
│  └───────────────────────────────────────────────────┘  │
│                                                         │
│  ┌───────────────────────────────────────────────────┐  │
│  │ JOB 2: deploy (depends on: build)                │  │
│  │  Runs on: ubuntu-latest                          │  │
│  │  ...                                             │  │
│  └───────────────────────────────────────────────────┘  │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### Key Terms Cheat Sheet

| Term | What It Is | Analogy |
|------|-----------|---------|
| **Workflow** | An automated process defined in YAML | The entire recipe |
| **Event/Trigger** | What causes the workflow to run | "Start cooking when guests arrive" |
| **Job** | A set of steps that run on the same runner | A dish in the meal |
| **Step** | A single task within a job | A cooking instruction |
| **Action** | A reusable, pre-built step | A pre-made sauce from the store |
| **Runner** | The machine where the job executes | The kitchen |
| **Artifact** | Files produced by a job | The cooked dish |

## 2️⃣ Your First Workflow — Hello World

Let's create your very first workflow! It's super simple.

### Step 1: Create the Workflow File

In your repository, create this file:

```
your-repo/
├── .github/
│   └── workflows/
│       └── hello.yml    ← This is your workflow!
├── src/
├── package.json
└── README.md
```

> 💡 **Important:** Workflow files MUST be in `.github/workflows/` directory and MUST be `.yml` or `.yaml` files.

### Step 2: Write the Workflow

```yaml
# .github/workflows/hello.yml

# Name of the workflow (shows up in GitHub UI)
name: Hello World

# When should this workflow run?
on:
  push:
    branches: [ main ]

# What work should be done?
jobs:
  say-hello:
    # What machine should this run on?
    runs-on: ubuntu-latest
    
    # The actual steps to perform
    steps:
      - name: Print Hello
        run: echo "Hello, World! 🎉"
      
      - name: Print date
        run: date
      
      - name: Print runner info
        run: |
          echo "Runner OS: $RUNNER_OS"
          echo "Runner Arch: $RUNNER_ARCH"
          echo "GitHub Repository: $GITHUB_REPOSITORY"
          echo "Branch: $GITHUB_REF"
          echo "Commit SHA: $GITHUB_SHA"
```

**Let's break down every single line:**

```yaml
name: Hello World
# ↑ This is the display name in the GitHub Actions tab
#   It can be anything — descriptive names are best

on:
  push:
    branches: [ main ]
# ↑ TRIGGER: Run this workflow when someone pushes to 'main' branch

jobs:
# ↑ Contains all the jobs in this workflow

  say-hello:
  # ↑ Job ID — a unique identifier (no spaces, use hyphens)
  #   This is used internally and in the GitHub UI
  
    runs-on: ubuntu-latest
    # ↑ What runner (machine) to use
    #   ubuntu-latest = a fresh Ubuntu VM provided by GitHub
    
    steps:
    # ↑ The list of steps (executed in order, top to bottom)
    
      - name: Print Hello
      # ↑ Display name for this step (shown in the GitHub UI)
        run: echo "Hello, World! 🎉"
      # ↑ Shell command to execute
      
      - name: Print runner info
        run: |
        # ↑ The pipe (|) symbol means "multiline string"
        #   Everything indented below is a single script
          echo "Runner OS: $RUNNER_OS"
          echo "GitHub Repository: $GITHUB_REPOSITORY"
```

### Step 3: Push and Watch It Run!

```bash
git add .github/workflows/hello.yml
git commit -m "Add first GitHub Actions workflow"
git push origin main
```

Now go to your repository on GitHub → **Actions** tab → You'll see your workflow running! 🎉

## 3️⃣ Triggers (Events) — When Should CI/CD Run?

The `on:` section defines **when** your workflow runs. GitHub Actions supports a TON of triggers.

### Common Triggers

#### Push Event — When code is pushed

```yaml
# Run on ANY push
on: push

# Run on push to specific branches
on:
  push:
    branches:
      - main
      - develop
      - 'release/**'    # Any branch starting with release/

# Run on push to specific branches, but NOT others
on:
  push:
    branches:
      - main
    branches-ignore:
      - 'docs/**'        # Don't run for documentation branches

# Run when specific files change
on:
  push:
    branches: [ main ]
    paths:
      - 'src/**'           # Only run if files in src/ changed
      - 'package.json'     # Or package.json changed
      - '!src/**/*.md'     # But NOT if only markdown files changed

# Ignore specific paths
on:
  push:
    paths-ignore:
      - '**.md'            # Don't run for markdown file changes
      - 'docs/**'          # Don't run for docs changes
      - '.gitignore'       # Don't run for .gitignore changes
```

#### Pull Request Event — When PRs are created/updated

```yaml
# Run on any PR activity
on: pull_request

# Run on specific PR events
on:
  pull_request:
    types:
      - opened        # PR was created
      - synchronize   # New commits pushed to the PR
      - reopened      # PR was reopened
    branches:
      - main          # Only PRs targeting the main branch

# Run on PR events for specific paths
on:
  pull_request:
    paths:
      - 'src/**'
      - 'tests/**'
```

**Common PR event types explained:**

| Type | When It Fires |
|------|--------------|
| `opened` | A new PR is created |
| `synchronize` | New commits are pushed to the PR branch |
| `reopened` | A closed PR is reopened |
| `closed` | PR is closed (merged or not) |
| `ready_for_review` | Draft PR is marked as ready |
| `review_requested` | Someone requests a code review |
| `labeled` | A label is added to the PR |

#### Schedule — Cron-based triggers

```yaml
# Run on a schedule (uses cron syntax)
on:
  schedule:
    # ┌────────── minute (0-59)
    # │ ┌──────── hour (0-23, UTC!)
    # │ │ ┌────── day of month (1-31)
    # │ │ │ ┌──── month (1-12)
    # │ │ │ │ ┌── day of week (0-6, Sunday=0)
    # │ │ │ │ │
    - cron: '0 6 * * 1-5'   # Every weekday at 6 AM UTC
    - cron: '0 0 * * 0'     # Every Sunday at midnight UTC
    - cron: '*/15 * * * *'  # Every 15 minutes (be careful! uses runner minutes)
```

> ⚠️ **Note:** Schedule times are in **UTC**! And scheduled workflows run on the **default branch** (usually `main`).

#### Manual Trigger — `workflow_dispatch`

```yaml
# Allow manual triggers from the GitHub UI
on:
  workflow_dispatch:
    # Optional: define input parameters
    inputs:
      environment:
        description: 'Target environment'
        required: true
        default: 'staging'
        type: choice
        options:
          - development
          - staging
          - production
      debug:
        description: 'Enable debug mode'
        required: false
        type: boolean
        default: false
      version:
        description: 'Version to deploy'
        required: true
        type: string

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Deploy
        run: |
          echo "Deploying to: ${{ github.event.inputs.environment }}"
          echo "Version: ${{ github.event.inputs.version }}"
          echo "Debug: ${{ github.event.inputs.debug }}"
```

This adds a "Run workflow" button in the GitHub Actions tab where you can fill in the inputs!

#### Release Event — When a release is published

```yaml
on:
  release:
    types: [ published ]

jobs:
  publish:
    runs-on: ubuntu-latest
    steps:
      - name: Get release info
        run: |
          echo "Tag: ${{ github.event.release.tag_name }}"
          echo "Name: ${{ github.event.release.name }}"
```

#### Multiple Triggers

You can combine multiple triggers:

```yaml
# Run on push, PR, and manual trigger
on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]
  workflow_dispatch:
```

### All Available Events (Reference)

Here are the most commonly used events:

| Event | When It Fires |
|-------|--------------|
| `push` | Code is pushed to a branch |
| `pull_request` | PR is opened, updated, closed |
| `workflow_dispatch` | Manual trigger from GitHub UI |
| `schedule` | Cron schedule |
| `release` | Release is created/published |
| `issues` | Issue is opened, edited, closed |
| `issue_comment` | Comment on an issue or PR |
| `create` | Branch or tag is created |
| `delete` | Branch or tag is deleted |
| `fork` | Repository is forked |
| `deployment` | Deployment is created |
| `repository_dispatch` | External trigger via API |
| `workflow_call` | Called by another workflow |
| `workflow_run` | Another workflow completes |

## 4️⃣ Jobs — Units of Work

Jobs are the **main building blocks** of a workflow. Each job:
- Runs on a **separate runner** (fresh VM or container)
- Has its own **file system** (doesn't share files with other jobs by default)
- Runs **in parallel** by default (unless you set dependencies)

### Basic Job Syntax

```yaml
jobs:
  # Job 1
  build:
    name: Build Application         # Display name (optional)
    runs-on: ubuntu-latest          # Runner to use
    steps:
      - uses: actions/checkout@v4
      - run: npm install
      - run: npm run build

  # Job 2
  test:
    name: Run Tests
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm install
      - run: npm test
```

**In this example, `build` and `test` run IN PARALLEL** because there's no dependency between them:

```
Workflow starts
    │
    ├──→ build (ubuntu-latest)  ─→ ✅ Done
    │
    └──→ test (ubuntu-latest)   ─→ ✅ Done
    
Both run at the same time!
```

### Job Dependencies — `needs`

Use `needs` to make jobs run sequentially:

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm run build

  test:
    runs-on: ubuntu-latest
    needs: build                    # ← Wait for 'build' to finish first
    steps:
      - uses: actions/checkout@v4
      - run: npm test

  deploy:
    runs-on: ubuntu-latest
    needs: [build, test]            # ← Wait for BOTH 'build' AND 'test'
    steps:
      - run: echo "Deploying..."
```

**Execution flow:**

```
Workflow starts
    │
    ▼
  build ─────────→ ✅ Done
    │
    ▼
  test ──────────→ ✅ Done
    │
    ▼
  deploy ────────→ ✅ Done

Sequential! Each waits for the previous.
```

### Complex Dependency Graph

```yaml
jobs:
  lint:
    runs-on: ubuntu-latest
    steps: [...]

  build:
    runs-on: ubuntu-latest
    steps: [...]

  unit-test:
    needs: build
    runs-on: ubuntu-latest
    steps: [...]

  integration-test:
    needs: build
    runs-on: ubuntu-latest
    steps: [...]

  deploy-staging:
    needs: [unit-test, integration-test]
    runs-on: ubuntu-latest
    steps: [...]

  deploy-production:
    needs: deploy-staging
    runs-on: ubuntu-latest
    steps: [...]
```

**Execution flow:**

```
          ┌── lint (parallel)
Start ────┤
          └── build
                │
          ┌─────┴──────┐
          ▼            ▼
    unit-test    integration-test  (parallel!)
          │            │
          └─────┬──────┘
                ▼
         deploy-staging
                │
                ▼
        deploy-production
```

### Job Outputs — Passing Data Between Jobs

Since each job runs on a **separate machine**, you need a way to pass data between them:

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    # Define outputs for this job
    outputs:
      version: ${{ steps.get-version.outputs.version }}
      image-tag: ${{ steps.get-version.outputs.image-tag }}
    steps:
      - uses: actions/checkout@v4
      
      - name: Get version
        id: get-version              # ← Give this step an ID
        run: |
          VERSION=$(cat package.json | jq -r '.version')
          echo "version=$VERSION" >> $GITHUB_OUTPUT
          echo "image-tag=myapp:$VERSION-${{ github.sha }}" >> $GITHUB_OUTPUT
      
      - run: echo "Version is ${{ steps.get-version.outputs.version }}"

  deploy:
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Use outputs from build job
        run: |
          echo "Deploying version: ${{ needs.build.outputs.version }}"
          echo "Image tag: ${{ needs.build.outputs.image-tag }}"
```

**How `$GITHUB_OUTPUT` works:**

```bash
# Old way (deprecated):
echo "::set-output name=key::value"

# New way (current):
echo "key=value" >> $GITHUB_OUTPUT

# For multiline values:
echo "key<<EOF" >> $GITHUB_OUTPUT
echo "line 1" >> $GITHUB_OUTPUT
echo "line 2" >> $GITHUB_OUTPUT
echo "EOF" >> $GITHUB_OUTPUT
```

### Matrix Strategy — Run Jobs with Multiple Configurations

This is one of the most powerful features! Run the same job with different configurations (Node versions, OS, etc.):

```yaml
jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
        node-version: [18, 20, 22]
      fail-fast: false    # Don't cancel others if one fails
    
    steps:
      - uses: actions/checkout@v4
      
      - name: Setup Node.js ${{ matrix.node-version }}
        uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
      
      - run: npm install
      - run: npm test
```

**This creates 9 parallel jobs:**

```
┌─────────────────────────────────────────────────┐
│ Matrix: 3 OS × 3 Node versions = 9 jobs        │
│                                                 │
│  ubuntu-latest  + Node 18  ── running ──→ ✅   │
│  ubuntu-latest  + Node 20  ── running ──→ ✅   │
│  ubuntu-latest  + Node 22  ── running ──→ ✅   │
│  windows-latest + Node 18  ── running ──→ ✅   │
│  windows-latest + Node 20  ── running ──→ ✅   │
│  windows-latest + Node 22  ── running ──→ ❌   │
│  macos-latest   + Node 18  ── running ──→ ✅   │
│  macos-latest   + Node 20  ── running ──→ ✅   │
│  macos-latest   + Node 22  ── running ──→ ✅   │
│                                                 │
│  fail-fast: false → All 9 run even if one fails │
└─────────────────────────────────────────────────┘
```

#### Advanced Matrix — Include / Exclude

```yaml
strategy:
  matrix:
    os: [ubuntu-latest, windows-latest]
    node: [18, 20]
    
    # Add specific combinations
    include:
      - os: ubuntu-latest
        node: 22
        experimental: true
    
    # Remove specific combinations
    exclude:
      - os: windows-latest
        node: 18
```

## 5️⃣ Steps — Individual Commands

Steps are the **smallest unit of work** inside a job. Each step either:
- **Runs a shell command** (`run:`)
- **Uses a pre-built action** (`uses:`)

### Shell Commands with `run`

```yaml
steps:
  # Single-line command
  - name: Simple command
    run: echo "Hello!"

  # Multi-line commands (use pipe |)
  - name: Multiple commands
    run: |
      echo "Step 1: Installing..."
      npm install
      echo "Step 2: Building..."
      npm run build
      echo "Step 3: Done!"

  # Different shell
  - name: PowerShell command (Windows)
    run: Write-Output "Hello from PowerShell!"
    shell: pwsh

  # Bash (explicit)
  - name: Bash command
    run: |
      echo "This runs in bash"
      for i in 1 2 3; do
        echo "Item $i"
      done
    shell: bash

  # Python script
  - name: Run Python
    run: |
      import json
      data = {"name": "GitHub Actions", "awesome": True}
      print(json.dumps(data, indent=2))
    shell: python
```

### Working Directory

```yaml
steps:
  - uses: actions/checkout@v4
  
  # Run command in a specific directory
  - name: Install in subdirectory
    run: npm install
    working-directory: ./frontend
  
  - name: Build backend
    run: npm run build
    working-directory: ./backend
```

### Conditional Steps — `if`

```yaml
steps:
  # Only run on main branch
  - name: Deploy to production
    if: github.ref == 'refs/heads/main'
    run: echo "Deploying to production..."

  # Only run on pull requests
  - name: Run PR checks
    if: github.event_name == 'pull_request'
    run: echo "Running PR checks..."

  # Only run when a previous step fails
  - name: Upload error logs
    if: failure()
    run: echo "Something failed! Uploading logs..."

  # Always run (even if previous steps failed)
  - name: Cleanup
    if: always()
    run: echo "Cleaning up..."

  # Run only if a specific step succeeded
  - name: Notify success
    if: steps.build.outcome == 'success'
    run: echo "Build succeeded!"
```

**Status check functions:**

| Function | Meaning |
|----------|---------|
| `success()` | All previous steps succeeded (default) |
| `failure()` | Any previous step failed |
| `always()` | Always run, regardless of previous results |
| `cancelled()` | The workflow was cancelled |

### Continue on Error

```yaml
steps:
  - name: Linting (non-blocking)
    run: npm run lint
    continue-on-error: true    # ← Even if lint fails, continue to next step
  
  - name: This still runs even if lint failed
    run: npm test
```

### Timeout

```yaml
steps:
  - name: Run tests (with timeout)
    run: npm test
    timeout-minutes: 10        # ← Kill the step if it runs longer than 10 minutes
```

## 6️⃣ Actions — Reusable Building Blocks

Actions are **pre-built, reusable steps** created by GitHub, third parties, or you. They're like plugins/modules you can plug into your workflow.

### Using Actions

```yaml
steps:
  # GitHub official action — checkout code
  - uses: actions/checkout@v4
  
  # GitHub official action — setup Node.js
  - uses: actions/setup-node@v4
    with:
      node-version: '20'
      cache: 'npm'            # ← Automatically cache npm packages!
  
  # Third-party action — upload to S3
  - uses: jakejarvis/s3-sync-action@v0.5.1
    with:
      args: --delete
    env:
      AWS_S3_BUCKET: my-bucket
      AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
      AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
```

### Action Format Explained

```yaml
uses: owner/repo@version
#     │      │     │
#     │      │     └── Version: @v4, @main, @sha-hash
#     │      └── Repository name
#     └── GitHub user or organization

# Examples:
uses: actions/checkout@v4              # Official GitHub action
uses: actions/setup-node@v4            # Official GitHub action
uses: docker/build-push-action@v5      # Docker's official action
uses: aws-actions/configure-aws-credentials@v4  # AWS official
uses: hashicorp/setup-terraform@v3     # HashiCorp official
```

> ⚠️ **Security Tip:** Always pin actions to a specific version (`@v4`) or even a commit SHA (`@abc123`), never use `@main` or `@latest` in production. A compromised action can steal your secrets!

### Most Essential GitHub Actions

Here are the actions you'll use in almost every workflow:

#### 1. `actions/checkout` — Clone your repo

```yaml
- uses: actions/checkout@v4
  with:
    # How many commits of history to fetch (0 = full history)
    fetch-depth: 0
    
    # Checkout a specific branch or tag
    ref: 'develop'
    
    # Use a token for private repos
    token: ${{ secrets.GITHUB_TOKEN }}
```

#### 2. `actions/setup-node` — Install Node.js

```yaml
- uses: actions/setup-node@v4
  with:
    node-version: '20'
    # Or use a version file
    node-version-file: '.nvmrc'
    # Cache dependencies
    cache: 'npm'    # or 'yarn' or 'pnpm'
```

#### 3. `actions/setup-python` — Install Python

```yaml
- uses: actions/setup-python@v5
  with:
    python-version: '3.12'
    cache: 'pip'
```

#### 4. `actions/setup-java` — Install Java

```yaml
- uses: actions/setup-java@v4
  with:
    distribution: 'temurin'
    java-version: '17'
    cache: 'maven'
```

#### 5. `actions/cache` — Cache files between runs

```yaml
- uses: actions/cache@v4
  with:
    path: ~/.npm
    key: ${{ runner.os }}-npm-${{ hashFiles('**/package-lock.json') }}
    restore-keys: |
      ${{ runner.os }}-npm-
```

#### 6. `actions/upload-artifact` & `actions/download-artifact`

```yaml
# Upload in build job
- uses: actions/upload-artifact@v4
  with:
    name: build-output
    path: dist/

# Download in deploy job
- uses: actions/download-artifact@v4
  with:
    name: build-output
    path: dist/
```

#### 7. `docker/build-push-action` — Build & push Docker images

```yaml
- uses: docker/build-push-action@v5
  with:
    context: .
    push: true
    tags: |
      myregistry/myapp:latest
      myregistry/myapp:${{ github.sha }}
```

```yaml
jobs:
```

For more control, you can run your own runner machines:

- You need larger machines
- You want to save money at scale
- You have compliance requirements (data must stay on-prem)

## 8️⃣ A Real-World CI/CD Pipeline

Let's build a complete, production-ready CI/CD pipeline for a Node.js application:

```yaml
# .github/workflows/ci-cd.yml
name: CI/CD Pipeline

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main ]

# Cancel in-progress runs for the same branch
concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

jobs:
  # ============================================
  # JOB 1: Lint & Code Quality
  # ============================================
  lint:
    name: Lint & Code Quality
    runs-on: ubuntu-latest
    
    steps:
      - name: Checkout code
        uses: actions/checkout@v4
      
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      
      - name: Install dependencies
        run: npm ci
        # npm ci is like npm install but:
        # - Faster (skips some steps)
        # - Uses exact versions from package-lock.json
        # - Deletes node_modules first (clean install)
      
      - name: Run ESLint
        run: npm run lint
      
      - name: Check formatting (Prettier)
        run: npm run format:check

  # ============================================
  # JOB 2: Run Tests
  # ============================================
  test:
    name: Test (Node ${{ matrix.node-version }})
    runs-on: ubuntu-latest
    
    strategy:
      matrix:
        node-version: [18, 20, 22]
      fail-fast: false
    
    steps:
      - name: Checkout code
        uses: actions/checkout@v4
      
      - name: Setup Node.js ${{ matrix.node-version }}
        uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
          cache: 'npm'
      
      - name: Install dependencies
        run: npm ci
      
      - name: Run unit tests
        run: npm test -- --coverage
      
      - name: Upload coverage report
        if: matrix.node-version == 20
        uses: actions/upload-artifact@v4
        with:
          name: coverage-report
          path: coverage/

  # ============================================
  # JOB 3: Build
  # ============================================
  build:
    name: Build Application
    runs-on: ubuntu-latest
    needs: [lint, test]    # Wait for lint and test to pass
    
    steps:
      - name: Checkout code
        uses: actions/checkout@v4
      
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      
      - name: Install dependencies
        run: npm ci
      
      - name: Build
        run: npm run build
      
      - name: Upload build artifact
        uses: actions/upload-artifact@v4
        with:
          name: build-output
          path: dist/
          retention-days: 7

  # ============================================
  # JOB 4: Build Docker Image
  # ============================================
  docker:
    name: Build & Push Docker Image
    runs-on: ubuntu-latest
    needs: build
    if: github.ref == 'refs/heads/main'    # Only on main branch
    
    permissions:
      contents: read
      packages: write    # Needed to push to GitHub Container Registry
    
    steps:
      - name: Checkout code
        uses: actions/checkout@v4
      
      - name: Download build artifact
        uses: actions/download-artifact@v4
        with:
          name: build-output
          path: dist/
      
      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3
      
      - name: Login to GitHub Container Registry
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
      
      - name: Build and push Docker image
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: |
            ghcr.io/${{ github.repository }}:latest
            ghcr.io/${{ github.repository }}:${{ github.sha }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

  # ============================================
  # JOB 5: Deploy to Staging
  # ============================================
  deploy-staging:
    name: Deploy to Staging
    runs-on: ubuntu-latest
    needs: docker
    environment: staging    # Uses GitHub Environments for approvals
    
    steps:
      - name: Deploy to staging
        run: |
          echo "Deploying to staging..."
          echo "Image: ghcr.io/${{ github.repository }}:${{ github.sha }}"
          # kubectl set image deployment/myapp myapp=ghcr.io/...
      
      - name: Run smoke tests
        run: |
          echo "Running smoke tests against staging..."
          # curl -f https://staging.example.com/health || exit 1

  # ============================================
  # JOB 6: Deploy to Production
  # ============================================
  deploy-production:
    name: Deploy to Production
    runs-on: ubuntu-latest
    needs: deploy-staging
    environment: production    # Requires manual approval!
    
    steps:
      - name: Deploy to production
        run: |
          echo "Deploying to production..."
          echo "Image: ghcr.io/${{ github.repository }}:${{ github.sha }}"
```

**The complete flow looks like this:**

```
Push to main
    │
    ├──→ lint ──────→ ✅
    │                   │
    ├──→ test (18) ──→ ✅├──→ build ──→ docker ──→ deploy-staging ──→ deploy-production
    │                   │                              │                     │
    ├──→ test (20) ──→ ✅│                        (auto deploy)      (manual approval!)
    │                   │
    └──→ test (22) ──→ ✅│
```

## 9️⃣ Environment Variables & Contexts

### Default Environment Variables

GitHub Actions automatically sets these variables:

```yaml
steps:
  - name: Show environment variables
    run: |
      echo "Repository: $GITHUB_REPOSITORY"        # owner/repo
      echo "Branch: $GITHUB_REF"                    # refs/heads/main
      echo "Branch Name: $GITHUB_REF_NAME"          # main
      echo "Commit SHA: $GITHUB_SHA"                # abc123...
      echo "Short SHA: ${GITHUB_SHA::7}"            # abc123
      echo "Actor: $GITHUB_ACTOR"                   # who triggered
      echo "Event: $GITHUB_EVENT_NAME"              # push, pull_request
      echo "Workspace: $GITHUB_WORKSPACE"           # /home/runner/work/repo
      echo "Run ID: $GITHUB_RUN_ID"                 # unique run ID
      echo "Run Number: $GITHUB_RUN_NUMBER"         # incremental number
      echo "Runner OS: $RUNNER_OS"                  # Linux, Windows, macOS
```

### Custom Environment Variables

```yaml
# Workflow-level (available to ALL jobs)
env:
  APP_NAME: myapp
  NODE_ENV: production

jobs:
  build:
    runs-on: ubuntu-latest
    
    # Job-level (available to all steps in this job)
    env:
      BUILD_TYPE: release
    
    steps:
      - name: Show vars
        # Step-level (available only to this step)
        env:
          STEP_VAR: hello
        run: |
          echo "$APP_NAME"     # myapp (workflow-level)
          echo "$BUILD_TYPE"   # release (job-level)
          echo "$STEP_VAR"     # hello (step-level)
```

### Contexts — Accessing Workflow Information

Contexts are **objects** that provide information about the workflow, job, steps, etc. You access them with `${{ context.property }}` syntax.

```yaml
steps:
  - name: Context examples
    run: |
      echo "Repo: ${{ github.repository }}"
      echo "Event: ${{ github.event_name }}"
      echo "Actor: ${{ github.actor }}"
      echo "SHA: ${{ github.sha }}"
      echo "Ref: ${{ github.ref }}"
      echo "Runner OS: ${{ runner.os }}"
      echo "Job status: ${{ job.status }}"
```

**Common contexts:**

| Context | What It Contains |
|---------|-----------------|
| `github` | Workflow trigger info (repo, event, actor, sha, ref) |
| `env` | Environment variables |
| `vars` | Repository/org variables (non-secret) |
| `secrets` | Encrypted secrets |
| `job` | Current job info |
| `steps` | Outputs from previous steps |
| `runner` | Runner machine info |
| `needs` | Outputs from dependent jobs |
| `matrix` | Current matrix values |
| `inputs` | Workflow dispatch inputs |

### Using Secrets

Secrets are encrypted values stored in your repository settings.

```
Repository → Settings → Secrets and variables → Actions → New repository secret
```

```yaml
steps:
  - name: Use secrets
    env:
      API_KEY: ${{ secrets.API_KEY }}
      DB_PASSWORD: ${{ secrets.DB_PASSWORD }}
    run: |
      # Secrets are masked in logs (show as ***)
      echo "API Key: $API_KEY"     # Output: API Key: ***
      curl -H "Authorization: Bearer $API_KEY" https://api.example.com
```

> ⚠️ **Security Notes:**
> - Secrets are **not** passed to workflows triggered from forked repos (prevents secret theft)
> - Secrets are **masked** in logs (replaced with `***`)
> - `GITHUB_TOKEN` is a special built-in secret — auto-generated for each workflow run

## 🔟 Conditionals — Running Things Based on Conditions

### Job-Level Conditionals

```yaml
jobs:
  deploy:
    runs-on: ubuntu-latest
    # Only deploy from main branch, not PRs
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    steps:
      - run: echo "Deploying..."

  notify-failure:
    runs-on: ubuntu-latest
    needs: [build, test]
    # Only run if a previous job failed
    if: failure()
    steps:
      - run: echo "Something failed! Sending notification..."
```

### Common Conditional Expressions

```yaml
# Branch checks
if: github.ref == 'refs/heads/main'
if: github.ref_name == 'develop'
if: startsWith(github.ref, 'refs/tags/')

# Event checks
if: github.event_name == 'push'
if: github.event_name == 'pull_request'
if: github.event_name == 'workflow_dispatch'

# PR specific
if: github.event.pull_request.merged == true
if: contains(github.event.pull_request.labels.*.name, 'deploy')

# Actor checks
if: github.actor == 'dependabot[bot]'
if: github.actor != 'bot-user'

# Combining conditions
if: github.ref == 'refs/heads/main' && github.event_name == 'push'
if: github.ref == 'refs/heads/main' || github.ref == 'refs/heads/develop'
if: "!(github.event_name == 'pull_request')"

# Status checks
if: success()     # Previous steps all passed
if: failure()     # Any previous step failed
if: always()      # Always run
if: cancelled()   # Workflow was cancelled

# Check step outputs
if: steps.check.outputs.changed == 'true'
```

## 1️⃣1️⃣ Artifacts — Passing Data Between Jobs

Since each job runs on a **separate machine**, files don't persist between jobs. Artifacts solve this:

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run build
      
      # Upload the built files as an artifact
      - name: Upload build artifact
        uses: actions/upload-artifact@v4
        with:
          name: my-build               # Artifact name
          path: |                       # Files/folders to upload
            dist/
            build/
          retention-days: 5             # How long to keep (default 90)
          if-no-files-found: error      # error, warn, or ignore

  deploy:
    runs-on: ubuntu-latest
    needs: build
    steps:
      # Download the artifact from the build job
      - name: Download build artifact
        uses: actions/download-artifact@v4
        with:
          name: my-build                # Same name as upload
          path: ./dist                  # Where to download
      
      - name: List downloaded files
        run: ls -la ./dist
      
      - name: Deploy
        run: echo "Deploying downloaded artifact..."
```

## 1️⃣2️⃣ Caching — Speed Up Your Builds

Caching saves files (like `node_modules/`, `.m2/`, `pip cache/`) between workflow runs so you don't have to download them every time.

### Method 1: Built-in Cache in Setup Actions

Many setup actions have built-in caching:

```yaml
# Node.js — caches npm automatically!
- uses: actions/setup-node@v4
  with:
    node-version: '20'
    cache: 'npm'           # Just add this line!

# Python — caches pip
- uses: actions/setup-python@v5
  with:
    python-version: '3.12'
    cache: 'pip'

# Java — caches Maven/Gradle
- uses: actions/setup-java@v4
  with:
    java-version: '17'
    distribution: 'temurin'
    cache: 'maven'         # or 'gradle'
```

### Method 2: Explicit `actions/cache`

For more control:

```yaml
steps:
  - uses: actions/checkout@v4
  
  - name: Cache node_modules
    uses: actions/cache@v4
    id: cache-npm                           # Give it an ID
    with:
      path: node_modules                    # What to cache
      key: ${{ runner.os }}-npm-${{ hashFiles('package-lock.json') }}
      # ↑ Cache key — if package-lock.json changes, cache is invalidated
      restore-keys: |
        ${{ runner.os }}-npm-
      # ↑ Fallback keys — if exact key doesn't match, try partial match
  
  - name: Install dependencies
    if: steps.cache-npm.outputs.cache-hit != 'true'
    # ↑ Only install if cache was NOT found!
    run: npm ci
  
  - name: Build
    run: npm run build
```

**How caching works:**

```
First run:
  1. Check cache → key "Linux-npm-abc123" → ❌ Not found
  2. Run npm ci → installs everything (slow)
  3. Save node_modules to cache with key "Linux-npm-abc123"

Second run (same package-lock.json):
  1. Check cache → key "Linux-npm-abc123" → ✅ Found!
  2. Restore node_modules from cache (fast!)
  3. Skip npm ci entirely

Third run (package-lock.json changed):
  1. Check cache → key "Linux-npm-def456" → ❌ Not found
  2. Try restore-keys → "Linux-npm-" → ✅ Partial match found
  3. Restore old cache (might need some updates)
  4. Run npm ci → only installs changed packages (faster than cold start)
  5. Save new cache with key "Linux-npm-def456"
```

## 1️⃣3️⃣ Common Patterns & Examples

### Pattern 1: Auto-label PRs

```yaml
name: Auto Label PRs

on:
  pull_request:
    types: [opened, synchronize]

jobs:

    runs-on: ubuntu-latest
    permissions:
      pull-requests: write

          repo-token: ${{ secrets.GITHUB_TOKEN }}
```

### Pattern 2: Create Release on Tag Push

```yaml
name: Release

on:
  push:
    tags:
      - 'v*'    # Triggers on v1.0.0, v2.1.3, etc.

jobs:
  release:
    runs-on: ubuntu-latest
    permissions:
      contents: write
    steps:
      - uses: actions/checkout@v4
      
      - name: Build
        run: |
          npm ci
          npm run build
          tar czf dist.tar.gz dist/
      
      - name: Create Release
        uses: softprops/action-gh-release@v2
        with:
          files: dist.tar.gz
          generate_release_notes: true
```

### Pattern 3: Deploy to Multiple Environments

```yaml
name: Deploy

on:
  push:
    branches: [main, develop]

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: ${{ github.ref_name == 'main' && 'production' || 'staging' }}
    
    steps:
      - name: Deploy
        run: |
          if [ "${{ github.ref_name }}" == "main" ]; then
            echo "Deploying to PRODUCTION"
```

```yaml
name: Backend CI

on:

      - 'backend/**'
      - 'shared/**'

  test-backend:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: cd backend && npm test
```

### Pattern 5: Notify on Slack

```yaml
jobs:
```

## 🎯 Key Takeaways

1. **GitHub Actions is CI/CD built into GitHub** — no separate servers to manage
2. **Workflows** are YAML files in `.github/workflows/` — version-controlled with your code
3. **Triggers (`on:`)** define WHEN workflows run — push, PR, schedule, manual, and many more
4. **Jobs** are independent units of work that run on separate machines, parallel by default
5. **Steps** are individual tasks inside a job — either shell commands (`run:`) or pre-built actions (`uses:`)
6. **Actions** are reusable building blocks from the marketplace — always pin to a version!
7. **Matrix strategy** lets you test across multiple OS/language versions automatically
8. **Secrets** are encrypted and masked in logs — never hardcode sensitive values
9. **Artifacts** let you pass files between jobs, **caching** speeds up repeated installs
10. **`needs:`** creates job dependencies, **`if:`** adds conditionals, **`concurrency:`** prevents duplicate runs
11. **GitHub-hosted runners** are free for public repos and come with tons of pre-installed tools
12. **Always use `npm ci`** instead of `npm install` in CI — it's faster and more reproducible
