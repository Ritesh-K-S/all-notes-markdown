## 2. GitHub Actions Secrets — The Basics

**Secrets** are encrypted values that you store in GitHub and use in your workflows. They are meant for sensitive data like:

- API keys and tokens
- Passwords and credentials
- SSH keys and certificates
- Connection strings
- Any value you **never** want visible in your code or logs

### How Secrets Work — The Big Picture:

```
YOU                           GITHUB                        WORKFLOW
 │                              │                              │
 │  1. Create a secret          │                              │
 │  (e.g., AWS_SECRET_KEY)      │                              │
 │──────────────────────────────►│                              │
 │                              │ 2. GitHub encrypts it        │
 │                              │    using libsodium            │
 │                              │    (sealed box encryption)   │
 │                              │                              │
 │                              │  3. Workflow runs             │
 │                              │──────────────────────────────►│
 │                              │                              │
 │                              │  4. GitHub decrypts and       │
 │                              │     injects as env var        │
 │                              │──────────────────────────────►│
 │                              │                              │
 │                              │  5. If the value appears     │
 │                              │     in logs, it's replaced    │
 │                              │     with ***                  │
 │                              │                              │
```

### Key Rules About Secrets:

```
📋 SECRET RULES:
├── 🔒 Encrypted at rest using libsodium
├── 👁️ Can NEVER be viewed after creation (only updated or deleted)
├── 🚫 Automatically masked in logs (replaced with ***)
├── 📦 Available as environment variables in workflow runs
├── 🔀 Not passed to workflows triggered by forks (security!)
├── 📏 Max 48 KB per secret
├── 📊 Max 100 org secrets, 100 repo secrets, 100 env secrets
└── 🏷️ Names: uppercase, alphanumeric + underscores, no GITHUB_ prefix
```

## 3. Types of Secrets — Repository, Environment, Organization

GitHub offers three levels of secrets. Think of them like nested circles:

```
┌──────────────────────────────────────────────────────────┐
│                  ORGANIZATION SECRETS                     │
│  (Shared across ALL or SELECTED repos in the org)        │
│                                                          │
│    ┌──────────────────────────────────────────────┐      │
│    │            REPOSITORY SECRETS                │      │
│    │  (Available to ALL workflows in this repo)   │      │
│    │                                              │      │
│    │    ┌──────────────────────────────────┐      │      │
│    │    │      ENVIRONMENT SECRETS         │      │      │
│    │    │  (Only available when a job      │      │      │
│    │    │   uses that specific environment)│      │      │
│    │    └──────────────────────────────────┘      │      │
│    │                                              │      │
│    └──────────────────────────────────────────────┘      │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### Comparison:

| Feature | Organization Secrets | Repository Secrets | Environment Secrets |
|---------|--------------------|--------------------|---------------------|
| **Scope** | All/selected repos in org | Single repository | Single environment in a repo |
| **Who sets them** | Org admins | Repo admins | Repo admins |
| **Use case** | Shared credentials (Docker Hub, AWS) | App-specific keys | Stage-specific keys (prod DB password) |
| **Visibility** | Configurable per repo | All workflows | Only jobs targeting that environment |
| **Priority** | Lowest | Medium | Highest (overrides others) |

### Secret Precedence (when names clash):

```
If the same secret name exists at multiple levels:

Environment secret  → WINS (highest priority)
  ↑ overrides
Repository secret
  ↑ overrides
Organization secret → LOWEST priority
```

This is actually very useful! You can have a generic `DATABASE_URL` at the org level and override it per environment.

## 4. Creating and Managing Secrets

### Creating Repository Secrets (via GitHub UI):

```
Steps:
1. Go to your repository on GitHub
2. Click "Settings" tab
3. Left sidebar → "Secrets and variables" → "Actions"
4. Click "New repository secret"
5. Enter:
   - Name: MY_SECRET_NAME  (uppercase, underscores)
   - Secret: the-actual-value
6. Click "Add secret"
```

### Creating Repository Secrets (via GitHub CLI):

```bash
# Set a secret from a value
gh secret set MY_API_KEY --body "sk-abc123def456"

# Set a secret from a file (great for SSH keys, certificates)
gh secret set SSH_PRIVATE_KEY < ~/.ssh/id_rsa

# Set a secret interactively (prompts for value — most secure)
gh secret set DATABASE_PASSWORD

# List all secrets (shows names only, never values!)
gh secret list

# Delete a secret
gh secret delete MY_API_KEY
```

### Creating Organization Secrets (via GitHub CLI):

```bash
# Set an org secret available to ALL repos
gh secret set DOCKER_PASSWORD --org my-org --visibility all

# Set an org secret available to SELECTED repos only
gh secret set AWS_SECRET_KEY --org my-org --visibility selected \
  --repos repo1,repo2,repo3

# Set an org secret available to only PRIVATE repos
gh secret set INTERNAL_TOKEN --org my-org --visibility private
```

### Creating Environment Secrets:

```
Steps:
1. Repository → Settings → Environments
2. Select an environment (e.g., "production")
3. Under "Environment secrets," click "Add secret"
4. Enter name and value
5. Click "Add secret"
```

Or via CLI:

```bash
# Set a secret for a specific environment
gh secret set DATABASE_URL --env production --body "postgres://prod-server/db"
gh secret set DATABASE_URL --env staging --body "postgres://staging-server/db"
```

## 5. Using Secrets in Workflows

### Basic Usage — `${{ secrets.SECRET_NAME }}`:

```yaml
name: Deploy

on: push

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Login to Docker Hub
        run: docker login -u ${{ secrets.DOCKER_USERNAME }} -p ${{ secrets.DOCKER_PASSWORD }}
        # ⚠️ This works but using run: with secrets directly is not the safest way
```

### Better Practice — Use Environment Variables:

```yaml
steps:
  - name: Login to Docker Hub
    env:
      DOCKER_USER: ${{ secrets.DOCKER_USERNAME }}
      DOCKER_PASS: ${{ secrets.DOCKER_PASSWORD }}
    run: echo "${DOCKER_PASS}" | docker login -u "${DOCKER_USER}" --password-stdin
    # ✅ Safer — prevents shell injection
```

### Passing Secrets to Actions:

```yaml
steps:
  - name: Deploy to AWS
    uses: aws-actions/configure-aws-credentials@v4
    with:
      aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
      aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
      aws-region: us-east-1
```

### Using Secrets in Multi-line Values (e.g., SSH keys):

```bash
# First, set the secret via CLI (preserves newlines)
gh secret set SSH_KEY < ~/.ssh/id_rsa
```

```yaml
# Then use it in a workflow
steps:
  - name: Setup SSH key
    run: |
      mkdir -p ~/.ssh
      echo "${{ secrets.SSH_KEY }}" > ~/.ssh/id_rsa
      chmod 600 ~/.ssh/id_rsa
      ssh-keyscan github.com >> ~/.ssh/known_hosts
```

### Using Secrets in JSON (e.g., service account keys):

```yaml
steps:
  - name: Authenticate with GCP
    env:
      GCP_CREDS: ${{ secrets.GCP_SERVICE_ACCOUNT_KEY }}
    run: |
      echo "${GCP_CREDS}" > /tmp/gcp-key.json
      gcloud auth activate-service-account --key-file=/tmp/gcp-key.json
      rm /tmp/gcp-key.json    # Clean up after use!
```

## 6. Secret Masking & Security Rules

### Automatic Masking:

GitHub automatically **masks** (hides) secret values in workflow logs:

```
# If your secret DATABASE_URL = "postgres://user:pass@db.example.com/mydb"

# What you'd expect to see in logs:
  Connecting to postgres://user:pass@db.example.com/mydb

# What you ACTUALLY see in logs:
  Connecting to ***
```

### But Masking Has Limits!

```
⚠️ MASKING LIMITATIONS:
├── Secrets shorter than 4 characters are NOT masked
├── Structured data (JSON) may be partially masked
├── If you transform a secret (base64 encode, etc.), the new value is NOT masked
├── Secrets in workflow artifacts are NOT masked
├── Secrets in step outputs might not be masked
└── Log commands like ::debug:: can sometimes leak secrets
```

### Manually Masking a Value:

If you create a derived value from a secret, you should manually mask it:

```yaml
steps:
  - name: Create a derived value
    run: |
      # Derive a new value from a secret
      DERIVED_TOKEN=$(echo "${{ secrets.TOKEN }}" | base64)

      # Manually mask it so it won't appear in logs
      echo "::add-mask::${DERIVED_TOKEN}"

      # Now use it safely
      echo "Token: ${DERIVED_TOKEN}"    # Will show *** in logs
```

### Security Rules to Always Follow:

```yaml
# ❌ NEVER DO THIS — Secret directly in run command
- run: curl -H "Authorization: Bearer ${{ secrets.API_TOKEN }}" https://api.example.com
  # Shell could log the expanded command

# ✅ ALWAYS DO THIS — Use environment variable
- run: curl -H "Authorization: Bearer ${API_TOKEN}" https://api.example.com
  env:
    API_TOKEN: ${{ secrets.API_TOKEN }}

# ❌ NEVER DO THIS — Print secrets for debugging
- run: echo "My secret is ${{ secrets.PASSWORD }}"

# ❌ NEVER DO THIS — Write secrets to files in artifacts
- run: echo "${{ secrets.KEY }}" > debug-output.txt
- uses: actions/upload-artifact@v4
  with:
    path: debug-output.txt    # Secret is now downloadable!

# ❌ NEVER DO THIS — Pass secrets to untrusted third-party actions
- uses: random-unknown-user/sketchy-action@main
  with:
    token: ${{ secrets.GITHUB_TOKEN }}
```

## 7. Using Secrets with Docker and Third-Party Actions

### Docker Build with Secrets:

```yaml
steps:
  - name: Build with Docker (passing secrets as build args)
    env:
      NPM_TOKEN: ${{ secrets.NPM_TOKEN }}
    run: |
      docker build \
        --secret id=npm_token,env=NPM_TOKEN \
        -t myapp:latest .
```

In the Dockerfile, use `--mount=type=secret`:

```dockerfile
# syntax=docker/dockerfile:1
FROM node:20-slim

# Mount the secret during build (not stored in the image layer!)
RUN --mount=type=secret,id=npm_token \
    NPM_TOKEN=$(cat /run/secrets/npm_token) && \
    echo "//registry.npmjs.org/:_authToken=${NPM_TOKEN}" > .npmrc && \
    npm ci && \
    rm .npmrc
```

### Passing Secrets to Composite Actions:

```yaml
# In the workflow
steps:
  - uses: ./my-composite-action
    with:
      api-key: ${{ secrets.API_KEY }}

# In the composite action's action.yml
inputs:
  api-key:
    required: true

runs:
  using: 'composite'
  steps:
    - shell: bash
      env:
        API_KEY: ${{ inputs.api-key }}
      run: echo "Using API key safely via env var"
```

## 8. GITHUB_TOKEN — The Automatic Secret

Every workflow run automatically gets a `GITHUB_TOKEN` — a temporary token that lets your workflow interact with the GitHub API.

### What GITHUB_TOKEN Can Do:

```
GITHUB_TOKEN permissions (configurable):
├── contents      → Read/write repo contents
├── issues        → Create/read/update issues
├── pull-requests → Create/read/update PRs
├── packages      → Push/pull from GitHub Packages
├── actions       → Manage workflow runs
├── deployments   → Create deployment statuses
├── statuses      → Update commit statuses
├── checks        → Create check runs
└── security-events → Upload SARIF results
```

### Setting GITHUB_TOKEN Permissions:

```yaml
# Method 1: At the workflow level (applies to all jobs)
permissions:
  contents: read
  pull-requests: write

# Method 2: At the job level (applies to that job only)
jobs:
  build:
    permissions:
      contents: read
    runs-on: ubuntu-latest
    steps: ...

  deploy:
    permissions:
      contents: read
      deployments: write
    runs-on: ubuntu-latest
    steps: ...
```

### Using GITHUB_TOKEN:

```yaml
steps:
  # Method 1: Explicitly pass it
  - uses: some-action@v1
    with:
      token: ${{ secrets.GITHUB_TOKEN }}

  # Method 2: It's also available as github.token
  - uses: some-action@v1
    with:
      token: ${{ github.token }}

  # Method 3: Available as an environment variable
  - run: |
      curl -H "Authorization: token ${GITHUB_TOKEN}" \
        https://api.github.com/repos/${{ github.repository }}/issues
    env:
      GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

### When GITHUB_TOKEN Isn't Enough:

```
GITHUB_TOKEN limitations:
├── Cannot trigger other workflows (prevents infinite loops)
├── Cannot access other repositories
├── Cannot create/manage GitHub Apps
├── Limited to the current repository
└── Expires when the workflow job finishes

Solution: Create a Personal Access Token (PAT) or GitHub App token
and store it as a secret.
```

## 9. Environments — What and Why

**Environments** represent deployment targets like `development`, `staging`, `production`, etc. They give you:

1. **Protection rules** — Require approvals before deploying
2. **Environment-specific secrets** — Different DB passwords per environment
3. **Wait timers** — Force a delay before deployment
4. **Branch restrictions** — Only deploy from `main` to production
5. **Deployment history** — See what's deployed where and when

### Visualizing Environments:

```
                    YOUR GITHUB REPO
                         │
              ┌──────────┼──────────┐
              │          │          │
         ┌────▼───┐ ┌───▼────┐ ┌──▼──────────┐
         │  dev   │ │staging │ │ production   │
         │        │ │        │ │              │
         │No rules│ │1 review│ │2 reviewers   │
         │        │ │        │ │+ wait timer  │
         │DB_URL= │ │DB_URL= │ │DB_URL=       │
         │dev-db  │ │stg-db  │ │prod-db       │
         │        │ │        │ │              │
         │Any     │ │main,   │ │main branch   │
         │branch  │ │develop │ │only          │
         └────────┘ └────────┘ └──────────────┘
```

## 11. Environment Protection Rules

Protection rules are what make environments powerful. They add safety gates before deployment.

### Available Protection Rules:

```
┌──────────────────────────────────────────────────────────────────┐
│               ENVIRONMENT PROTECTION RULES                       │
├───────────────────┬──────────────────────────────────────────────┤
│ Required          │ Specific people or teams must approve        │
│ Reviewers         │ the deployment before it proceeds.           │
│                   │ Up to 6 reviewers can be added.              │
├───────────────────┼──────────────────────────────────────────────┤
│ Wait Timer        │ A delay (in minutes, 0-43200 = 30 days)     │
│                   │ before the deployment starts. Useful for     │
│                   │ "cool down" periods.                         │
├───────────────────┼──────────────────────────────────────────────┤
│ Deployment        │ Only specific branches or tags can deploy    │
│ Branches / Tags   │ to this environment.                        │
│                   │ Options: All, Protected, Selected            │
├───────────────────┼──────────────────────────────────────────────┤
│ Custom Rules      │ Use GitHub Apps to create custom             │
│ (via GitHub Apps) │ deployment protection rules.                 │
└───────────────────┴──────────────────────────────────────────────┘
```

### How Required Reviewers Work:

```
Developer pushes to main
       │
       ▼
Workflow starts
       │
       ▼
Job reaches "production" environment
       │
       ▼
⏸️ PAUSED — Waiting for approval
       │
       ├── Reviewer gets notified (email + GitHub UI)
       │
       ├── Reviewer goes to the workflow run page
       │
       ├── Reviewer clicks "Review deployments"
       │
       ├── Reviewer selects "Approve" or "Reject"
       │
       ▼
✅ Approved → Job continues and deploys
❌ Rejected → Job fails
⏰ Timeout  → Job fails (default: 30 days)
```

### Configuring Branch Restrictions:

```
Via UI: Settings → Environments → [env] → Deployment branches

Options:
├── "All branches"          → Any branch can deploy here
├── "Protected branches"    → Only branches with protection rules
└── "Selected branches"     → You specify patterns:
    ├── main
    ├── release/*
    └── hotfix/*
```

## 12. Using Environments in Workflows

### Basic Usage:

```yaml
name: Deploy

on:
  push:
    branches: [main]

jobs:
  deploy-staging:
    runs-on: ubuntu-latest
    environment: staging          # ← This job targets the "staging" environment
    steps:
      - uses: actions/checkout@v4
      - run: echo "Deploying to staging..."
      - run: echo "Using DB: ${{ secrets.DATABASE_URL }}"
        # ↑ This uses the STAGING environment secret

  deploy-production:
    needs: deploy-staging
    runs-on: ubuntu-latest
    environment: production       # ← This job targets the "production" environment
    steps:
      - uses: actions/checkout@v4
      - run: echo "Deploying to production..."
      - run: echo "Using DB: ${{ secrets.DATABASE_URL }}"
        # ↑ This uses the PRODUCTION environment secret (different value!)
```

### What Happens When You Use `environment:`:

```
1. Job starts
2. GitHub checks: "Is this environment protected?"
   ├── If yes → Pauses and waits for approval / timer
   └── If no  → Continues immediately
3. GitHub loads environment-specific secrets
4. GitHub loads environment-specific variables
5. Job steps execute with those secrets/variables
6. GitHub updates the deployment status for this environment
```

### Sequential Deployment Pipeline:

```yaml
name: Progressive Deployment

on:
  push:
    branches: [main]

jobs:
  # Stage 1: Deploy to dev (no approval needed)
  deploy-dev:
    runs-on: ubuntu-latest
    environment: development
    steps:
      - uses: actions/checkout@v4
      - run: ./deploy.sh
        env:
          ENV: development
          API_URL: ${{ vars.API_URL }}         # Environment variable
          DB_URL: ${{ secrets.DATABASE_URL }}   # Environment secret

  # Stage 2: Deploy to staging (needs dev to pass)
  deploy-staging:
    needs: deploy-dev
    runs-on: ubuntu-latest
    environment: staging
    steps:
      - uses: actions/checkout@v4
      - run: ./deploy.sh
        env:
          ENV: staging
          API_URL: ${{ vars.API_URL }}
          DB_URL: ${{ secrets.DATABASE_URL }}

  # Stage 3: Deploy to production (needs staging + manual approval)
  deploy-production:
    needs: deploy-staging
    runs-on: ubuntu-latest
    environment: production
    steps:
      - uses: actions/checkout@v4
      - run: ./deploy.sh
        env:
          ENV: production
          API_URL: ${{ vars.API_URL }}
          DB_URL: ${{ secrets.DATABASE_URL }}
```

## 13. Environments + Secrets — The Perfect Combo

The real power comes when you combine environments with secrets. Each environment can have its **own set of secrets** with the same names but different values.

### Example Setup:

```
Repository Secret:
└── DOCKER_PASSWORD = "shared-docker-pass"     (same for all)

Environment: development
├── DATABASE_URL = "postgres://dev-db:5432/app"
├── API_KEY = "dev-key-12345"
└── REDIS_URL = "redis://dev-redis:6379"

Environment: staging
├── DATABASE_URL = "postgres://stg-db:5432/app"
├── API_KEY = "stg-key-67890"
└── REDIS_URL = "redis://stg-redis:6379"

Environment: production
├── DATABASE_URL = "postgres://prod-db:5432/app"
├── API_KEY = "prod-key-ABCDE"
└── REDIS_URL = "redis://prod-redis:6379"
```

### In Your Workflow:

```yaml
jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: ${{ github.ref == 'refs/heads/main' && 'production' || 'staging' }}
    # ↑ Dynamically choose environment based on branch!
    steps:
      - run: |
          echo "Deploying with:"
          echo "  Database: (secret — masked)"
          echo "  Environment: ${{ github.event.deployment.environment }}"
        env:
          DATABASE_URL: ${{ secrets.DATABASE_URL }}
          # The value automatically changes based on the environment!
```

### Environment Variables (Non-Secret Configuration):

GitHub also supports **environment variables** (not encrypted, visible) alongside secrets:

```yaml
# Set via: Settings → Environments → [env] → Environment variables

# Use in workflows:
jobs:
  deploy:
    environment: production
    runs-on: ubuntu-latest
    steps:
      - run: echo "Deploying to ${{ vars.DEPLOY_TARGET }}"
        # vars.DEPLOY_TARGET = different value per environment
```

```
Secrets (${{ secrets.NAME }})  → Encrypted, hidden, for sensitive data
Variables (${{ vars.NAME }})   → Plain text, visible, for configuration
```

## 15. Matrix Builds — What and Why

**Matrix builds** let you automatically run the same job across multiple configurations. Instead of writing separate jobs for each combination, you define a matrix and GitHub generates all the combinations for you.

### The Problem:

```yaml
# ❌ WITHOUT matrix — You'd have to write this:
jobs:
  test-node18-ubuntu:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/setup-node@v4
        with:
          node-version: '18'
      - run: npm test

  test-node20-ubuntu:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: npm test

  test-node22-ubuntu:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/setup-node@v4
        with:
          node-version: '22'
      - run: npm test

  # ... imagine adding macOS and Windows too 😩
```

### The Solution — Matrix!

```yaml
# ✅ WITH matrix — Clean and elegant:
jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node-version: [18, 20, 22]      # ← 3 versions
    steps:
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
      - run: npm test
```

This single job definition **automatically creates 3 parallel jobs** — one for each Node version!

## 16. Basic Matrix Strategy

### Single-Variable Matrix:

```yaml
jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        python-version: ['3.10', '3.11', '3.12']
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python-version }}
      - run: |
          pip install -r requirements.txt
          pytest
```

**This creates:**

```
Job 1: test (python-version=3.10) ─── Running ✅
Job 2: test (python-version=3.11) ─── Running ✅
Job 3: test (python-version=3.12) ─── Running ✅
```

### Accessing Matrix Values:

Use `${{ matrix.VARIABLE_NAME }}` anywhere in the job:

```yaml
strategy:
  matrix:
    os: [ubuntu-latest, windows-latest, macos-latest]

runs-on: ${{ matrix.os }}          # ← Use in runs-on
steps:
  - run: echo "Testing on ${{ matrix.os }}"   # ← Use in steps
```

## 17. Multi-Dimensional Matrix

The real magic happens when you combine multiple variables — GitHub creates the **Cartesian product** (all combinations):

```yaml
jobs:
  test:
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
        node-version: [18, 20, 22]

    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
      - run: npm ci
      - run: npm test
```

**This creates 3 × 3 = 9 parallel jobs:**

```
┌─────────────────┬──────────┬──────────┬──────────┐
│                 │ Node 18  │ Node 20  │ Node 22  │
├─────────────────┼──────────┼──────────┼──────────┤
│ ubuntu-latest   │   Job 1  │   Job 2  │   Job 3  │
│ windows-latest  │   Job 4  │   Job 5  │   Job 6  │
│ macos-latest    │   Job 7  │   Job 8  │   Job 9  │
└─────────────────┴──────────┴──────────┴──────────┘
```

All 9 jobs run **in parallel** (subject to your concurrency limit).

### Three-Dimensional Matrix:

```yaml
strategy:
  matrix:
    os: [ubuntu-latest, windows-latest]
    node: [18, 20]
    database: [postgres, mysql]

# This creates 2 × 2 × 2 = 8 jobs!
```

```
ubuntu  + Node 18 + postgres
ubuntu  + Node 18 + mysql
ubuntu  + Node 20 + postgres
ubuntu  + Node 20 + mysql
windows + Node 18 + postgres
windows + Node 18 + mysql
windows + Node 20 + postgres
windows + Node 20 + mysql
```

## 18. Include and Exclude in Matrix

Sometimes you don't want ALL combinations. You can fine-tune with `include` and `exclude`.

### Exclude — Remove Specific Combinations:

```yaml
strategy:
  matrix:
    os: [ubuntu-latest, windows-latest, macos-latest]
    node: [18, 20, 22]
    exclude:
      # Don't test Node 18 on macOS (not needed)
      - os: macos-latest
        node: 18
      # Don't test Node 22 on Windows (known issues)
      - os: windows-latest
        node: 22
```

```
Result: 9 total - 2 excluded = 7 jobs

┌─────────────────┬──────────┬──────────┬──────────┐
│                 │ Node 18  │ Node 20  │ Node 22  │
├─────────────────┼──────────┼──────────┼──────────┤
│ ubuntu-latest   │   ✅     │   ✅     │   ✅     │
│ windows-latest  │   ✅     │   ✅     │   ❌     │
│ macos-latest    │   ❌     │   ✅     │   ✅     │
└─────────────────┴──────────┴──────────┴──────────┘
```

### Include — Add Extra Combinations or Variables:

```yaml
strategy:
  matrix:
    os: [ubuntu-latest, windows-latest]
    node: [18, 20]
    include:
      # Add an extra combination that's not in the Cartesian product
      - os: macos-latest
        node: 22

      # Add extra variables to a specific combination
      - os: ubuntu-latest
        node: 20
        experimental: true       # ← Extra variable! Only for this combo
        npm-version: 10
```

### Include to Add Completely Custom Combinations:

```yaml
strategy:
  matrix:
    include:
      # No base arrays — each include entry is a separate job
      - os: ubuntu-latest
        node: 20
        name: "Linux Node 20"

      - os: windows-latest
        node: 20
        name: "Windows Node 20"

      - os: macos-latest
        node: 22
        name: "macOS Node 22 (experimental)"
```

## 19. Fail-Fast and Max-Parallel

### Fail-Fast — What Happens When One Job Fails?

By default, **fail-fast is `true`** — if any matrix job fails, GitHub **cancels all remaining jobs**.

```yaml
strategy:
  fail-fast: true    # Default — if one fails, cancel all others
  matrix:
    node: [18, 20, 22]
```

```
Scenario: Node 18 job fails

fail-fast: true (default)        fail-fast: false
├── Node 18: ❌ Failed           ├── Node 18: ❌ Failed
├── Node 20: 🚫 Cancelled       ├── Node 20: ✅ Passed
└── Node 22: 🚫 Cancelled       └── Node 22: ✅ Passed
```

**When to set `fail-fast: false`:**
- You want to see results across ALL configurations
- One failure shouldn't stop you from knowing if other versions work
- During release testing when you need the full picture

```yaml
strategy:
  fail-fast: false    # Let all jobs complete even if one fails
  matrix:
    os: [ubuntu-latest, windows-latest, macos-latest]
    node: [18, 20, 22]
```

### Max-Parallel — Control Concurrency:

By default, GitHub runs as many matrix jobs in parallel as possible. You can limit this:

```yaml
strategy:
  max-parallel: 2      # Only run 2 jobs at a time
  matrix:
    node: [18, 20, 22]
```

```
Without max-parallel:           With max-parallel: 2:
├── Node 18: Running            ├── Node 18: Running
├── Node 20: Running            ├── Node 20: Running
└── Node 22: Running            └── Node 22: Queued ⏳
    (all 3 in parallel)             (waits for a slot)
```

**When to use `max-parallel`:**
- You're hitting API rate limits
- Your tests need shared resources (databases, services)
- You want to reduce costs on self-hosted runners

## 20. Dynamic Matrix from JSON

Sometimes you want to generate the matrix dynamically (e.g., from a config file or another job's output).

### Method 1: From a JSON File:

```json
// matrix-config.json
{
  "include": [
    { "project": "frontend", "node": "20", "directory": "./frontend" },
    { "project": "backend", "node": "18", "directory": "./backend" },
    { "project": "api", "node": "22", "directory": "./api" }
  ]
}
```

```yaml
jobs:
  setup:
    runs-on: ubuntu-latest
    outputs:
      matrix: ${{ steps.set-matrix.outputs.matrix }}
    steps:
      - uses: actions/checkout@v4
      - id: set-matrix
        run: |
          MATRIX=$(cat matrix-config.json)
          echo "matrix=${MATRIX}" >> $GITHUB_OUTPUT

  build:
    needs: setup
    runs-on: ubuntu-latest
    strategy:
      matrix: ${{ fromJSON(needs.setup.outputs.matrix) }}
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: |
          cd ${{ matrix.directory }}
          npm ci && npm test
          echo "Tested ${{ matrix.project }}"
```

### Method 2: Dynamically Generated:

```yaml
jobs:
  detect-changes:
    runs-on: ubuntu-latest
    outputs:
      matrix: ${{ steps.changes.outputs.matrix }}
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - id: changes
        run: |
          # Find which services have changed files
          SERVICES=()
          if git diff --name-only HEAD~1 | grep -q "^frontend/"; then
            SERVICES+='"frontend"'
          fi
          if git diff --name-only HEAD~1 | grep -q "^backend/"; then
            SERVICES+='"backend"'
          fi
          if git diff --name-only HEAD~1 | grep -q "^api/"; then
            SERVICES+='"api"'
          fi

          # Build the matrix JSON
          MATRIX=$(printf '%s,' "${SERVICES[@]}" | sed 's/,$//')
          echo "matrix={\"service\":[${MATRIX}]}" >> $GITHUB_OUTPUT

  test:
    needs: detect-changes
    if: needs.detect-changes.outputs.matrix != '{"service":[]}'
    runs-on: ubuntu-latest
    strategy:
      matrix: ${{ fromJSON(needs.detect-changes.outputs.matrix) }}
    steps:
      - uses: actions/checkout@v4
      - run: |
          cd ${{ matrix.service }}
          npm test
```

## 21. Matrix with Environments

You can combine matrix builds with environments for multi-environment deployments:

```yaml
name: Deploy to All Environments

on:
  workflow_dispatch:    # Manual trigger

jobs:
  deploy:
    strategy:
      max-parallel: 1    # Deploy one at a time!
      matrix:
        environment: [development, staging, production]
    runs-on: ubuntu-latest
    environment: ${{ matrix.environment }}
    steps:
      - uses: actions/checkout@v4
      - name: Deploy
        run: |
          echo "Deploying to ${{ matrix.environment }}"
          echo "Using API URL: ${{ vars.API_URL }}"
          ./deploy.sh --env ${{ matrix.environment }}
        env:
          DATABASE_URL: ${{ secrets.DATABASE_URL }}
          API_KEY: ${{ secrets.API_KEY }}
```

**What happens:**

```
1. development job → Runs immediately (no protection rules)
2. staging job     → Waits for dev, then reviewer approval
3. production job  → Waits for staging, then reviewer approval + timer
```

## 23. Common Mistakes & Troubleshooting

### Mistake 1: Secret is Empty in Workflow

```yaml
# ❌ Problem: Secret appears empty
- run: echo "Token: ${{ secrets.MY_TOKEN }}"
  # Output: Token:

# Possible causes:
# 1. Secret name is wrong (case-sensitive!)
# 2. Secret was set in the wrong scope (org vs repo vs env)
# 3. Workflow is triggered by a fork PR (secrets are not available!)
# 4. Environment is not specified in the job

# ✅ Fix: Verify the secret name and check the scope
- run: echo "Has token: ${{ secrets.MY_TOKEN != '' }}"
  # Output: Has token: true   (if secret exists)
  # Output: Has token: false  (if secret is missing)
```

### Mistake 2: Matrix Values Must Be Quoted Properly

```yaml
# ❌ WRONG — These will be interpreted as floats!
matrix:
  python: [3.10, 3.11, 3.12]
  # 3.10 becomes 3.1 (trailing zero dropped!)

# ✅ CORRECT — Quote them as strings
matrix:
  python: ['3.10', '3.11', '3.12']
```

### Mistake 3: Forgetting `fromJSON` for Dynamic Matrix

```yaml
# ❌ WRONG — Treats the output as a plain string
strategy:
  matrix: ${{ needs.setup.outputs.matrix }}

# ✅ CORRECT — Parse the JSON string into an object
strategy:
  matrix: ${{ fromJSON(needs.setup.outputs.matrix) }}
```

### Mistake 4: Environment Name Mismatch

```yaml
# ❌ Names are case-sensitive!
environment: Production     # ← Capital P

# If your environment is named "production" (lowercase), this won't match
# and secrets won't be loaded!

# ✅ Always match exactly
environment: production
```

### Mistake 5: Max Matrix Size

```
GitHub limits:
├── Max 256 jobs per workflow run
├── Max 256 entries in a matrix (after include/exclude)
└── If you exceed this, the workflow will fail to start

Solution: Split into multiple workflows or reduce matrix dimensions
```
