## 1. The Problem — Why Do We Need Reusable Workflows?

Imagine you're working in a company that has **50 microservices**. Each one needs a CI/CD pipeline that does:

1. Run linting
2. Run tests
3. Build a Docker image
4. Push to a container registry
5. Deploy to staging

Without reusable workflows, you'd have to **copy-paste** the same YAML workflow into all 50 repos. Now imagine you need to change the Docker registry URL — you'd have to update **50 files** in **50 repos**. Nightmare! 😵

```
❌ Without Reusable Workflows:

Repo 1: .github/workflows/ci.yml  ──┐
Repo 2: .github/workflows/ci.yml  ──┤ All 50 files are identical
Repo 3: .github/workflows/ci.yml  ──┤ (or nearly identical).
...                                  │ Change one? Change ALL 50.
Repo 50: .github/workflows/ci.yml ──┘

✅ With Reusable Workflows:

Central Repo: .github/workflows/build.yml   ← Define ONCE
    ▲
    │ (calls)
    │
Repo 1: .github/workflows/ci.yml  ──┐
Repo 2: .github/workflows/ci.yml  ──┤ Each repo just calls
Repo 3: .github/workflows/ci.yml  ──┤ the central workflow.
...                                  │ Change once, applied everywhere.
Repo 50: .github/workflows/ci.yml ──┘
```

### Key benefits:
- **DRY** — Write once, use everywhere
- **Consistency** — Every repo follows the same pipeline
- **Maintainability** — Fix bugs or update steps in one place
- **Security** — Centralize secret management

## 2. Reusable Workflows — The Big Picture

A **Reusable Workflow** is a regular GitHub Actions workflow file that can be **called by other workflows**. Think of it like a **function** in programming — you define it once, and call it from wherever you need it.

### How it works:

```
┌──────────────────────┐        ┌──────────────────────────┐
│   CALLER Workflow    │        │   CALLED (Reusable)      │
│   (in any repo)      │        │   Workflow                │
│                      │        │   (in same or other repo) │
│  jobs:               │        │                          │
│    my-job:           │  ───►  │  on:                     │
│      uses: org/repo/ │        │    workflow_call:         │
│      .github/work... │        │      inputs: ...         │
│                      │        │      secrets: ...        │
│                      │        │  jobs:                   │
│                      │        │    build:                │
│                      │        │      steps: ...          │
└──────────────────────┘        └──────────────────────────┘
```

### Key terminology:

| Term | Meaning |
|------|---------|
| **Caller workflow** | The workflow that **calls** (uses) a reusable workflow |
| **Called workflow** | The reusable workflow that **gets called** |
| `workflow_call` | The trigger event that makes a workflow reusable |
| `uses` | The keyword in the caller that references the reusable workflow |

## 3. Creating Your First Reusable Workflow

To make a workflow reusable, you need to use the special trigger: `workflow_call`.

### Step 1: Create the reusable workflow

Let's say we want to create a reusable workflow for running tests. Create this file in a repository (let's call it `my-org/shared-workflows`):

**File: `.github/workflows/run-tests.yml`**

```yaml
# This is a REUSABLE workflow — notice the trigger is "workflow_call"
name: Run Tests

on:
  workflow_call:   # ← This is the magic! This makes it reusable.

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Install dependencies
        run: npm ci

      - name: Run tests
        run: npm test
```

That's it! This workflow is now reusable. The key is `on: workflow_call`.

### What makes it different from a normal workflow?

```yaml
# Normal workflow — triggers on push
on:
  push:
    branches: [main]

# Reusable workflow — triggers when CALLED by another workflow
on:
  workflow_call:
```

> **Important:** A workflow can have BOTH triggers! You can make it run on push AND be callable by other workflows:

```yaml
on:
  push:
    branches: [main]
  workflow_call:    # Also reusable!
```

## 4. Calling a Reusable Workflow

Now let's call the reusable workflow from another repository (or the same repo).

### Calling from another repo

**File: `.github/workflows/ci.yml`** (in any repo)

```yaml
name: CI Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  run-tests:
    uses: my-org/shared-workflows/.github/workflows/run-tests.yml@main
    #     ─────────┬──────────── ──────────────┬──────────────── ──┬──
    #              │                            │                   │
    #     org/repo name              path to workflow file      branch/tag/SHA
```

### Let's break down the `uses` syntax:

```
uses: {owner}/{repo}/.github/workflows/{filename}@{ref}
```

| Part | Example | Meaning |
|------|---------|---------|
| `{owner}/{repo}` | `my-org/shared-workflows` | The GitHub org and repo name |
| `.github/workflows/{filename}` | `.github/workflows/run-tests.yml` | Path to the workflow file |
| `@{ref}` | `@main`, `@v1`, `@a1b2c3d` | Branch name, tag, or commit SHA |

### Calling from the same repo

If the reusable workflow is in the **same repository**, you just use a relative path:

```yaml
jobs:
  run-tests:
    uses: ./.github/workflows/run-tests.yml
    #     ──────────────┬─────────────────
    #           Relative path (same repo)
```

### Calling with a specific version (tag)

```yaml
jobs:
  run-tests:
    uses: my-org/shared-workflows/.github/workflows/run-tests.yml@v1.0.0
    #                                                              ▲
    #                                                     Git tag (recommended!)
```

> **Best Practice:** Always pin to a **tag** or **commit SHA** in production. Using `@main` means your pipeline could break if someone pushes a bad change to the shared workflow.

## 5. Inputs and Secrets in Reusable Workflows

Most of the time, your reusable workflow needs some **configuration** — like "which Node.js version?" or "which environment to deploy to?". That's where **inputs** and **secrets** come in.

### Defining inputs in the reusable workflow

```yaml
name: Build and Test

on:
  workflow_call:
    inputs:
      node-version:
        description: 'Node.js version to use'
        required: false
        type: string
        default: '20'           # Default value if not provided
      
      environment:
        description: 'Deployment environment'
        required: true
        type: string            # Can be: string, boolean, number
      
      run-lint:
        description: 'Whether to run linting'
        required: false
        type: boolean
        default: true

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: ${{ inputs.node-version }}
          #                  ▲
          #       Access input values with inputs.<name>

      - name: Install deps
        run: npm ci

      - name: Run linter
        if: ${{ inputs.run-lint }}    # Conditional step based on input
        run: npm run lint

      - name: Run tests
        run: npm test

      - name: Deploy to ${{ inputs.environment }}
        run: echo "Deploying to ${{ inputs.environment }}"
```

### Passing inputs from the caller workflow

```yaml
name: CI

on:
  push:
    branches: [main]

jobs:
  build-and-test:
    uses: my-org/shared-workflows/.github/workflows/build-test.yml@v1
    with:                         # ← Use "with" to pass inputs
      node-version: '18'
      environment: 'staging'
      run-lint: true
```

### Defining and passing secrets

Secrets need to be declared separately from inputs:

**Reusable workflow (defining secrets it needs):**

```yaml
on:
  workflow_call:
    inputs:
      environment:
        required: true
        type: string
    secrets:
      DEPLOY_TOKEN:
        description: 'Token for deployment'
        required: true
      AWS_ACCESS_KEY:
        description: 'AWS access key'
        required: false

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Deploy
        run: ./deploy.sh
        env:
          TOKEN: ${{ secrets.DEPLOY_TOKEN }}
          AWS_KEY: ${{ secrets.AWS_ACCESS_KEY }}
```

**Caller workflow (passing secrets):**

```yaml
jobs:
  deploy:
    uses: my-org/shared-workflows/.github/workflows/deploy.yml@v1
    with:
      environment: 'production'
    secrets:                      # ← Use "secrets" to pass secrets
      DEPLOY_TOKEN: ${{ secrets.MY_DEPLOY_TOKEN }}
      AWS_ACCESS_KEY: ${{ secrets.AWS_KEY }}
```

### The `inherit` shortcut for secrets

If you want to pass **ALL** secrets from the caller to the reusable workflow automatically (without listing each one), use `secrets: inherit`:

```yaml
jobs:
  deploy:
    uses: my-org/shared-workflows/.github/workflows/deploy.yml@v1
    with:
      environment: 'production'
    secrets: inherit               # ← Pass ALL secrets automatically!
```

> **When to use `inherit`:**
> - Quick and easy — no need to list every secret
> - Good for workflows within the same org
>
> **When NOT to use `inherit`:**
> - When calling external/third-party workflows (security risk!)
> - When you want to explicitly control which secrets are shared

## 6. Outputs from Reusable Workflows

Reusable workflows can also **return data** back to the caller — just like a function returning a value.

### Defining outputs in the reusable workflow

```yaml
name: Build Image

on:
  workflow_call:
    inputs:
      image-name:
        required: true
        type: string
    outputs:
      image-tag:
        description: 'The tag of the built Docker image'
        value: ${{ jobs.build.outputs.tag }}
        #         ▲
        #   Must reference a job's output

jobs:
  build:
    runs-on: ubuntu-latest
    outputs:
      tag: ${{ steps.set-tag.outputs.image_tag }}
      #       ▲
      #   Must reference a step's output
    steps:
      - uses: actions/checkout@v4

      - name: Generate image tag
        id: set-tag              # ← Give the step an ID
        run: |
          TAG="${{ inputs.image-name }}:$(git rev-parse --short HEAD)"
          echo "image_tag=$TAG" >> $GITHUB_OUTPUT
          #                       ▲
          #   Set step output using GITHUB_OUTPUT

      - name: Build Docker image
        run: docker build -t ${{ steps.set-tag.outputs.image_tag }} .
```

### The output chain looks like this:

```
Step Output                Job Output              Workflow Output
─────────────────────    ─────────────────────    ─────────────────────
steps.set-tag.outputs    jobs.build.outputs       workflow outputs
  .image_tag        ──►    .tag             ──►    .image-tag
                                                       │
                                                       ▼
                                              Available to caller
```

### Using outputs in the caller workflow

```yaml
name: CI/CD

on:
  push:
    branches: [main]

jobs:
  # Step 1: Build the image (calls reusable workflow)
  build:
    uses: my-org/shared-workflows/.github/workflows/build-image.yml@v1
    with:
      image-name: 'my-app'

  # Step 2: Deploy using the image tag from step 1
  deploy:
    needs: build                   # ← Wait for build to finish
    runs-on: ubuntu-latest
    steps:
      - name: Deploy image
        run: |
          echo "Deploying image: ${{ needs.build.outputs.image-tag }}"
          #                       ▲
          #   Access reusable workflow output via needs.<job-name>.outputs.<output-name>
          kubectl set image deployment/my-app app=${{ needs.build.outputs.image-tag }}
```

## 7. Reusable Workflows — Nesting and Limitations

### Nesting reusable workflows

A reusable workflow **CAN** call another reusable workflow! But there are limits:

```
Caller Workflow
  └── Reusable Workflow (Level 1)
        └── Reusable Workflow (Level 2)
              └── Reusable Workflow (Level 3)
                    └── Reusable Workflow (Level 4)
                          └── ❌ MAXIMUM DEPTH REACHED (4 levels)
```

> **Limit:** You can nest reusable workflows up to **4 levels deep**.

### Other important limitations

| Limitation | Details |
|-----------|---------|
| **Max nesting depth** | 4 levels |
| **Max called workflows** | A single workflow file can call up to **20** reusable workflows |
| **Environment variables** | `env` variables defined in the caller are **NOT** passed to the called workflow |
| **Strategy/matrix** | The caller job that uses `uses:` **cannot** also have `strategy`, `container`, or `services` |
| **Cannot mix `uses` and `steps`** | A job either calls a reusable workflow (`uses:`) OR has `steps:`, never both |
| **Concurrency** | `concurrency` from the caller is NOT passed to the called workflow |

### Common mistake — trying to mix `uses` and `steps`:

```yaml
# ❌ WRONG — Cannot have both "uses" and "steps" in the same job
jobs:
  build:
    uses: my-org/shared-workflows/.github/workflows/build.yml@v1
    steps:                        # ← ERROR! Can't do this!
      - run: echo "Hello"

# ✅ CORRECT — Separate into two jobs
jobs:
  build:
    uses: my-org/shared-workflows/.github/workflows/build.yml@v1
  
  post-build:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - run: echo "Build is done!"
```

## 8. Reusable Workflows — Real-World Examples

### Example 1: Full CI/CD Pipeline with Reusable Workflows

**Shared reusable workflow: `build-and-push.yml`**

```yaml
name: Build and Push Docker Image

on:
  workflow_call:
    inputs:
      dockerfile:
        required: false
        type: string
        default: './Dockerfile'
      image-name:
        required: true
        type: string
      registry:
        required: false
        type: string
        default: 'ghcr.io'
    secrets:
      REGISTRY_USERNAME:
        required: true
      REGISTRY_PASSWORD:
        required: true
    outputs:
      image-digest:
        description: 'Image digest'
        value: ${{ jobs.build.outputs.digest }}

jobs:
  build:
    runs-on: ubuntu-latest
    outputs:
      digest: ${{ steps.push.outputs.digest }}
    steps:
      - uses: actions/checkout@v4

      - name: Log in to registry
        uses: docker/login-action@v3
        with:
          registry: ${{ inputs.registry }}
          username: ${{ secrets.REGISTRY_USERNAME }}
          password: ${{ secrets.REGISTRY_PASSWORD }}

      - name: Build and push
        id: push
        uses: docker/build-push-action@v5
        with:
          context: .
          file: ${{ inputs.dockerfile }}
          push: true
          tags: |
            ${{ inputs.registry }}/${{ inputs.image-name }}:latest
            ${{ inputs.registry }}/${{ inputs.image-name }}:${{ github.sha }}
```

**Caller workflow in any microservice repo:**

```yaml
name: CI/CD

on:
  push:
    branches: [main]

jobs:
  test:
    uses: my-org/shared-workflows/.github/workflows/run-tests.yml@v1
    with:
      node-version: '20'

  build:
    needs: test
    uses: my-org/shared-workflows/.github/workflows/build-and-push.yml@v1
    with:
      image-name: 'my-org/user-service'
    secrets:
      REGISTRY_USERNAME: ${{ secrets.GHCR_USERNAME }}
      REGISTRY_PASSWORD: ${{ secrets.GHCR_TOKEN }}

  deploy:
    needs: build
    uses: my-org/shared-workflows/.github/workflows/deploy-k8s.yml@v1
    with:
      environment: 'production'
      image-digest: ${{ needs.build.outputs.image-digest }}
    secrets: inherit
```

### Example 2: Matrix + Reusable Workflow

You **can** use matrix in the caller's other jobs, but NOT in the job that has `uses:`. However, you can pass matrix values as inputs:

```yaml
name: Multi-Service CI

on:
  push:
    branches: [main]

jobs:
  # Define services to build
  prepare:
    runs-on: ubuntu-latest
    outputs:
      services: ${{ steps.set.outputs.services }}
    steps:
      - id: set
        run: |
          echo 'services=["user-service", "order-service", "payment-service"]' >> $GITHUB_OUTPUT

  # Build each service (cannot use matrix directly with "uses:")
  # Instead, create separate jobs or use a different approach
  build-user:
    needs: prepare
    uses: my-org/shared-workflows/.github/workflows/build-and-push.yml@v1
    with:
      image-name: 'my-org/user-service'
    secrets: inherit

  build-order:
    needs: prepare
    uses: my-org/shared-workflows/.github/workflows/build-and-push.yml@v1
    with:
      image-name: 'my-org/order-service'
    secrets: inherit
```

## 9. What are Composite Actions?

While reusable workflows operate at the **workflow/job level**, Composite Actions work at the **step level**. They let you bundle multiple steps into a **single reusable action**.

Think of it this way:

```
Reusable Workflow = Reusable JOB (replaces an entire job)
Composite Action  = Reusable STEP (replaces one or more steps inside a job)
```

### Visual comparison:

```
┌─────────────────────────────────────────┐
│            WORKFLOW                      │
│  ┌───────────────────────────────────┐  │
│  │           JOB 1                   │  │
│  │  ┌─────────────────────────────┐  │  │
│  │  │ Step 1 ◄── Composite Action │  │  │  ← Composite Action replaces
│  │  │ (multiple sub-steps inside) │  │  │     one or more STEPS
│  │  └─────────────────────────────┘  │  │
│  │  ┌────────────┐                   │  │
│  │  │ Step 2     │                   │  │
│  │  └────────────┘                   │  │
│  └───────────────────────────────────┘  │
│  ┌───────────────────────────────────┐  │
│  │ JOB 2 ◄── Reusable Workflow      │  │  ← Reusable Workflow replaces
│  │ (entire job defined elsewhere)    │  │     an entire JOB
│  └───────────────────────────────────┘  │
└─────────────────────────────────────────┘
```

### Why use Composite Actions?

- You want to reuse a **group of steps** (not an entire job)
- You want to create a **custom action** without writing JavaScript or Docker
- You need something that works like a **function** you can call within any job
- You want to share logic **within the same job** alongside other steps

## 10. Creating Your First Composite Action

A composite action is defined in an `action.yml` (or `action.yaml`) file.

### File structure:

```
my-composite-action/
  ├── action.yml          ← The action definition
  └── scripts/            ← Optional: any scripts your action needs
        └── setup.sh
```

### Basic composite action

**File: `action.yml`**

```yaml
name: 'Setup and Test Node.js Project'
description: 'Checks out code, sets up Node.js, installs deps, and runs tests'
author: 'Your Name'

# No inputs for now (we'll add them later)

runs:
  using: 'composite'          # ← This is what makes it a COMPOSITE action
  steps:
    - name: Checkout code
      uses: actions/checkout@v4
      shell: bash               # ← Required for "run" steps in composite actions!

    - name: Setup Node.js
      uses: actions/setup-node@v4
      with:
        node-version: '20'

    - name: Install dependencies
      run: npm ci
      shell: bash               # ← MUST specify shell for every "run" step

    - name: Run tests
      run: npm test
      shell: bash
```

> **Critical Rule:** In composite actions, every `run` step **MUST** have a `shell` property. This is different from regular workflows where `shell` is optional (defaults to `bash`).

### Using the composite action in a workflow

**If the action is in the same repo:**

```yaml
name: CI

on: [push]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - name: Run setup and tests
        uses: ./.github/actions/my-composite-action
        #     ▲
        #     Relative path to the directory containing action.yml
```

**If the action is in another repo:**

```yaml
steps:
  - name: Run setup and tests
    uses: my-org/my-composite-action@v1
    #     ▲
    #     org/repo@ref (action.yml must be in the root of the repo)
```

## 11. Composite Actions — Inputs and Outputs

### Defining inputs

```yaml
name: 'Build Docker Image'
description: 'Builds a Docker image with configurable options'

inputs:
  image-name:
    description: 'Name for the Docker image'
    required: true
  tag:
    description: 'Tag for the Docker image'
    required: false
    default: 'latest'
  dockerfile:
    description: 'Path to the Dockerfile'
    required: false
    default: './Dockerfile'

runs:
  using: 'composite'
  steps:
    - name: Checkout
      uses: actions/checkout@v4

    - name: Build image
      run: |
        docker build \
          -f ${{ inputs.dockerfile }} \
          -t ${{ inputs.image-name }}:${{ inputs.tag }} \
          .
      shell: bash

    - name: Verify image
      run: docker images ${{ inputs.image-name }}:${{ inputs.tag }}
      shell: bash
```

**Using with inputs:**

```yaml
steps:
  - name: Build my image
    uses: my-org/docker-build-action@v1
    with:
      image-name: 'my-app'
      tag: ${{ github.sha }}
      dockerfile: './docker/Dockerfile.prod'
```

### Defining outputs

```yaml
name: 'Get Version Info'
description: 'Extracts version information from package.json'

outputs:
  version:
    description: 'The version from package.json'
    value: ${{ steps.get-version.outputs.version }}
  major:
    description: 'Major version number'
    value: ${{ steps.get-version.outputs.major }}

runs:
  using: 'composite'
  steps:
    - name: Get version
      id: get-version                   # ← Need an ID to reference outputs
      run: |
        VERSION=$(node -p "require('./package.json').version")
        MAJOR=$(echo $VERSION | cut -d. -f1)
        echo "version=$VERSION" >> $GITHUB_OUTPUT
        echo "major=$MAJOR" >> $GITHUB_OUTPUT
      shell: bash
```

**Using outputs in the workflow:**

```yaml
steps:
  - uses: actions/checkout@v4

  - name: Get version
    id: version-info              # ← Give the step an ID
    uses: my-org/get-version-action@v1

  - name: Use the version
    run: |
      echo "Full version: ${{ steps.version-info.outputs.version }}"
      echo "Major version: ${{ steps.version-info.outputs.major }}"
    shell: bash
```

## 12. Composite Actions — Using Shell Scripts

You can run external scripts from your composite action. This is great for keeping your `action.yml` clean when you have complex logic.

### File structure:

```
my-action/
  ├── action.yml
  └── scripts/
        ├── setup.sh
        └── validate.sh
```

### action.yml

```yaml
name: 'Setup Environment'
description: 'Sets up the development environment'

inputs:
  language:
    description: 'Programming language'
    required: true

runs:
  using: 'composite'
  steps:
    - name: Run setup script
      run: ${{ github.action_path }}/scripts/setup.sh
      #      ▲
      #      github.action_path = the directory where action.yml lives
      #      This is CRITICAL — without it, the script won't be found!
      shell: bash
      env:
        LANGUAGE: ${{ inputs.language }}

    - name: Validate setup
      run: ${{ github.action_path }}/scripts/validate.sh
      shell: bash
```

> **Important:** Always use `${{ github.action_path }}` to reference files inside your action directory. Without it, GitHub won't know where to find your scripts!

### setup.sh

```bash
#!/bin/bash
set -euo pipefail

echo "Setting up environment for $LANGUAGE..."

case "$LANGUAGE" in
  node)
    echo "Installing Node.js dependencies..."
    npm ci
    ;;
  python)
    echo "Installing Python dependencies..."
    pip install -r requirements.txt
    ;;
  go)
    echo "Installing Go dependencies..."
    go mod download
    ;;
  *)
    echo "Unknown language: $LANGUAGE"
    exit 1
    ;;
esac

echo "Setup complete!"
```

## 13. Composite Actions — Real-World Examples

### Example 1: Slack Notification Action

```yaml
# .github/actions/notify-slack/action.yml
name: 'Slack Notification'
description: 'Sends a formatted message to Slack'

inputs:
  webhook-url:
    description: 'Slack webhook URL'
    required: true
  status:
    description: 'Build status (success/failure)'
    required: true
  message:
    description: 'Custom message'
    required: false
    default: ''

runs:
  using: 'composite'
  steps:
    - name: Set color
      id: color
      run: |
        if [ "${{ inputs.status }}" = "success" ]; then
          echo "color=#36a64f" >> $GITHUB_OUTPUT
          echo "emoji=✅" >> $GITHUB_OUTPUT
        else
          echo "color=#dc3545" >> $GITHUB_OUTPUT
          echo "emoji=❌" >> $GITHUB_OUTPUT
        fi
      shell: bash

    - name: Send Slack notification
      run: |
        curl -X POST "${{ inputs.webhook-url }}" \
          -H "Content-Type: application/json" \
          -d '{
            "attachments": [{
              "color": "${{ steps.color.outputs.color }}",
              "title": "${{ steps.color.outputs.emoji }} Build ${{ inputs.status }} — ${{ github.repository }}",
              "text": "Branch: ${{ github.ref_name }}\nCommit: ${{ github.sha }}\n${{ inputs.message }}",
              "footer": "GitHub Actions",
              "ts": "'$(date +%s)'"
            }]
          }'
      shell: bash
```

**Using the Slack notification:**

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test
      
      - name: Notify on success
        if: success()
        uses: ./.github/actions/notify-slack
        with:
          webhook-url: ${{ secrets.SLACK_WEBHOOK }}
          status: 'success'
          message: 'All tests passed!'

      - name: Notify on failure
        if: failure()
        uses: ./.github/actions/notify-slack
        with:
          webhook-url: ${{ secrets.SLACK_WEBHOOK }}
          status: 'failure'
          message: 'Tests failed! Check the logs.'
```

### Example 2: Setup + Cache Action

```yaml
# .github/actions/setup-node-cached/action.yml
name: 'Setup Node.js with Cache'
description: 'Sets up Node.js with npm caching for faster CI runs'

inputs:
  node-version:
    description: 'Node.js version'
    required: false
    default: '20'

outputs:
  cache-hit:
    description: 'Whether the cache was hit'
    value: ${{ steps.cache.outputs.cache-hit }}

runs:
  using: 'composite'
  steps:
    - name: Setup Node.js
      uses: actions/setup-node@v4
      with:
        node-version: ${{ inputs.node-version }}

    - name: Get npm cache directory
      id: npm-cache-dir
      run: echo "dir=$(npm config get cache)" >> $GITHUB_OUTPUT
      shell: bash

    - name: Cache npm dependencies
      id: cache
      uses: actions/cache@v4
      with:
        path: ${{ steps.npm-cache-dir.outputs.dir }}
        key: ${{ runner.os }}-node-${{ inputs.node-version }}-${{ hashFiles('**/package-lock.json') }}
        restore-keys: |
          ${{ runner.os }}-node-${{ inputs.node-version }}-

    - name: Install dependencies
      run: npm ci
      shell: bash
```

**Using it in a workflow:**

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Setup Node with cache
        uses: ./.github/actions/setup-node-cached
        with:
          node-version: '18'
      
      # Dependencies are already installed! Just run your tests:
      - run: npm test
      - run: npm run build
```

## 14. Reusable Workflows vs Composite Actions — When to Use What

This is a really common question, so let's break it down clearly:

```
┌───────────────────────────────────────────────────────────────────┐
│                    DECISION FLOWCHART                              │
│                                                                   │
│  Do you need to reuse an ENTIRE JOB (with its own runner)?       │
│     │                                                             │
│     ├── YES → Use REUSABLE WORKFLOW                               │
│     │                                                             │
│     └── NO → Do you need to reuse a GROUP OF STEPS               │
│              within an existing job?                               │
│                │                                                  │
│                ├── YES → Use COMPOSITE ACTION                     │
│                │                                                  │
│                └── NO → Do you need to reuse a SINGLE complex    │
│                         step with its own runtime (JS/Docker)?    │
│                           │                                       │
│                           ├── YES → Use JS or Docker ACTION       │
│                           │                                       │
│                           └── NO → Just write a regular step      │
└───────────────────────────────────────────────────────────────────┘
```

### Side-by-side comparison:

| Feature | Reusable Workflow | Composite Action |
|---------|------------------|-----------------|
| **Scope** | Entire job(s) | Steps within a job |
| **Runs on** | Its own runner | Caller's runner |
| **Defined in** | `.github/workflows/*.yml` | `action.yml` in any directory |
| **Called with** | `jobs.<id>.uses:` | `steps[*].uses:` |
| **Can have its own `runs-on`?** | ✅ Yes | ❌ No (uses caller's runner) |
| **Can mix with other steps?** | ❌ No (takes over the whole job) | ✅ Yes |
| **Supports secrets?** | ✅ Yes (declared explicitly) | ❌ No (use env variables) |
| **Supports `if` conditions?** | ✅ On the calling job | ✅ On individual steps |
| **Max nesting** | 4 levels | No limit (but keep it sane) |
| **Best for** | Complete pipelines, multi-job flows | Setup routines, notifications, utilities |

### Quick rule of thumb:

```
"I want to standardize our entire build-test-deploy pipeline"
    → Reusable Workflow

"I want to create a reusable 'setup Node.js + cache + install' step"
    → Composite Action

"I need to run custom JavaScript logic or a Docker container"
    → JavaScript Action or Docker Action
```

## 15. Sharing Actions and Workflows Across Repos

### Option 1: Public repository

Create a public GitHub repository containing your actions/workflows. Any repo can use them:

```yaml
# Anyone can use a public repo's actions
uses: my-org/public-actions/setup-node@v1
```

### Option 2: Internal repository (GitHub Enterprise)

If you're on GitHub Enterprise, you can create an **internal** repository and allow other repos in the org to access it.

Go to the repo's **Settings → Actions → General** and select:
> "Allow access from repositories in the organization"

### Option 3: Private repository

For private repos, you need to explicitly grant access:

1. Go to the **shared workflow repo** → Settings → Actions → General
2. Under "Access", select "Accessible from repositories in the organization"

```yaml
# Works if access is granted
uses: my-org/private-workflows/.github/workflows/build.yml@v1
```

### Option 4: Same repository

Store composite actions in `.github/actions/` in the same repo:

```
.github/
  ├── actions/
  │     ├── setup-node/
  │     │     └── action.yml
  │     ├── notify-slack/
  │     │     └── action.yml
  │     └── deploy/
  │           └── action.yml
  └── workflows/
        ├── ci.yml
        └── cd.yml
```

## 16. Versioning Your Reusable Components

Proper versioning is crucial for stability. If someone changes a shared workflow and it breaks, it breaks ALL repos that use it.

### Use Git tags for versioning

```bash
# Create a version tag
git tag -a v1.0.0 -m "Initial release"
git push origin v1.0.0

# Create a major version tag (recommended for easy updates)
git tag -a v1 -m "Major version 1"
git push origin v1
```

### Recommended tag strategy:

```
v1.0.0  ← Specific version (most stable — use in production)
v1.0    ← Minor version (gets patch updates)
v1      ← Major version (gets minor + patch updates)
main    ← Latest (unstable — NEVER use in production)
```

### How callers reference versions:

```yaml
# Most stable — pinned to exact version
uses: my-org/actions/setup@v1.2.3

# Stable — gets patches automatically
uses: my-org/actions/setup@v1

# Most stable — pinned to commit SHA (immune to tag changes)
uses: my-org/actions/setup@a1b2c3d4e5f6789012345678901234567890abcd

# ⚠️ Unstable — don't use in production
uses: my-org/actions/setup@main
```

### Updating major version tags

When you release v1.1.0, update the `v1` tag to point to it:

```bash
# Delete old v1 tag
git tag -d v1
git push origin :refs/tags/v1

# Create new v1 tag pointing to v1.1.0
git tag -a v1 -m "Update v1 to v1.1.0"
git push origin v1
```

## 17. Best Practices

### 1. Start simple, then extract

Don't try to make everything reusable from day one. Start with regular workflows, notice patterns, then extract common parts.

```
Step 1: Write pipelines for 3-4 repos
Step 2: Notice patterns (same steps/jobs everywhere)
Step 3: Extract common patterns into reusable workflows/actions
Step 4: Refactor repos to use the shared components
```

### 2. Keep reusable workflows focused

```yaml
# ✅ Good — Single responsibility
name: Run Tests         # Does ONE thing
name: Build Image       # Does ONE thing
name: Deploy to K8s     # Does ONE thing

# ❌ Bad — Does too much
name: Build Test Deploy Monitor Alert    # Kitchen sink!
```

### 3. Always provide defaults for inputs

```yaml
inputs:
  node-version:
    required: false
    default: '20'          # ← Smart default = less config for callers
```

### 4. Document your actions and workflows

```yaml
name: 'My Awesome Action'
description: |
  This action does X, Y, and Z.
  
  ## Usage
  ```yaml
  - uses: my-org/my-action@v1
    with:
      input1: value1
  ```
  
  ## Inputs
  - `input1` (required): Description of input1
```

### 5. Test your reusable workflows

Create a test workflow in your shared-workflows repo:

```yaml
# .github/workflows/test-build-workflow.yml
name: Test Build Workflow

on:
  pull_request:
    paths:
      - '.github/workflows/build.yml'

jobs:
  test:
    uses: ./.github/workflows/build.yml
    with:
      image-name: 'test-image'
    secrets: inherit
```

### 6. Use `permissions` for security

```yaml
on:
  workflow_call:

permissions:
  contents: read           # Minimum required permissions
  packages: write          # Only what's needed

jobs:
  build:
    runs-on: ubuntu-latest
    steps: ...
```

## 18. Common Mistakes and Troubleshooting

### Mistake 1: Forgetting `shell` in composite actions

```yaml
# ❌ ERROR: "Missing required input 'shell'"
runs:
  using: 'composite'
  steps:
    - run: echo "Hello"       # No shell specified!

# ✅ FIX: Add shell to every run step
runs:
  using: 'composite'
  steps:
    - run: echo "Hello"
      shell: bash              # ← Required!
```

### Mistake 2: Trying to use `steps` in a caller job

```yaml
# ❌ ERROR
jobs:
  build:
    uses: ./.github/workflows/reusable.yml
    steps:                                   # ← Can't have both!
      - run: echo "hi"

# ✅ FIX: Use a separate job
jobs:
  build:
    uses: ./.github/workflows/reusable.yml
  after-build:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - run: echo "hi"
```

### Mistake 3: Wrong path to composite action

```yaml
# ❌ ERROR: action.yml not found
uses: ./.github/actions/my-action/action.yml    # Don't include action.yml!

# ✅ FIX: Point to the directory
uses: ./.github/actions/my-action               # Points to the directory
```

### Mistake 4: Env vars not available in reusable workflows

```yaml
# In caller — these env vars are NOT passed to the reusable workflow
env:
  MY_VAR: hello

jobs:
  build:
    uses: ./.github/workflows/reusable.yml      # MY_VAR is NOT available inside!

# ✅ FIX: Pass values as inputs
jobs:
  build:
    uses: ./.github/workflows/reusable.yml
    with:
      my-var: 'hello'                           # Pass as input instead
```

### Mistake 5: Referencing wrong context for outputs

```yaml
# ❌ Wrong — can't use "steps" in workflow outputs
on:
  workflow_call:
    outputs:
      result:
        value: ${{ steps.my-step.outputs.result }}    # Wrong!

# ✅ Correct — reference job outputs
on:
  workflow_call:
    outputs:
      result:
        value: ${{ jobs.my-job.outputs.result }}       # Correct!
```

### Quick reference card:

```
┌──────────────────────────────────────────────────────────────┐
│            REUSABLE WORKFLOWS & COMPOSITE ACTIONS            │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  REUSABLE WORKFLOW:                                          │
│    Define:   on: workflow_call                                │
│    Call:     jobs.<id>.uses: org/repo/path@ref               │
│    Scope:    Entire job(s) with their own runner             │
│    Inputs:   on.workflow_call.inputs.<name>                   │
│    Secrets:  on.workflow_call.secrets.<name>                  │
│    Outputs:  on.workflow_call.outputs.<name>                  │
│    Inherit:  secrets: inherit (pass all secrets)             │
│                                                              │
│  COMPOSITE ACTION:                                           │
│    Define:   runs.using: 'composite' in action.yml           │
│    Call:     steps[*].uses: org/repo@ref or ./path           │
│    Scope:    Steps within an existing job                    │
│    Inputs:   inputs.<name> in action.yml                     │
│    Outputs:  outputs.<name> in action.yml                    │
│    Scripts:  ${{ github.action_path }}/script.sh             │
│    Shell:    REQUIRED on every run step!                      │
│                                                              │
│  VERSIONING:                                                 │
│    @v1.2.3  → Exact version (most stable)                    │
│    @v1      → Major version (gets updates)                   │
│    @<sha>   → Commit SHA (immune to tag changes)             │
│    @main    → Latest (never use in production!)              │
│                                                              │
│  CHOOSE:                                                     │
│    Need entire job?      → Reusable Workflow                 │
│    Need group of steps?  → Composite Action                  │
│    Need custom runtime?  → JavaScript/Docker Action          │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```
