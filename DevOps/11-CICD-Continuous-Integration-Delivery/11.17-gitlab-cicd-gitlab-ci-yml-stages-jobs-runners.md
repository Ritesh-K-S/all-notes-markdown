## 1. What is GitLab CI/CD?

GitLab CI/CD is GitLab's **built-in** continuous integration and continuous delivery system. Unlike Jenkins (which is a separate tool), GitLab CI/CD is **part of GitLab itself** — there's nothing extra to install.

You define your pipeline in a file called `.gitlab-ci.yml`, push it to your repo, and GitLab automatically picks it up and runs your pipeline. That's it!

### Key differences from other CI/CD tools:

```
Jenkins:    Separate server → Install plugins → Configure via UI → Jenkinsfile
GitHub:     GitHub Actions → .github/workflows/*.yml → Marketplace actions
GitLab:     Built-in CI/CD → .gitlab-ci.yml → Everything in one platform
```

### What can you do with GitLab CI/CD?

- **Build** your application (compile, package, create Docker images)
- **Test** your code (unit tests, integration tests, security scans)
- **Deploy** to any environment (staging, production, Kubernetes)
- **Review** apps — create temporary environments for merge requests
- **Monitor** — track pipeline performance and metrics
- **Auto DevOps** — automatic pipeline detection and configuration

## 2. How GitLab CI/CD Works — The Big Picture

Here's the entire flow from code push to deployment:

```
Developer pushes code
        │
        ▼
┌──────────────────┐
│ GitLab detects   │
│ .gitlab-ci.yml   │
│ in the repo      │
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│ GitLab creates   │
│ a PIPELINE       │
│ (a collection    │
│  of stages)      │
└────────┬─────────┘
         │
         ▼
┌──────────────────┐     ┌──────────────────┐     ┌──────────────────┐
│   STAGE: build   │────►│   STAGE: test    │────►│   STAGE: deploy  │
│                  │     │                  │     │                  │
│ ┌──────────────┐ │     │ ┌──────────────┐ │     │ ┌──────────────┐ │
│ │  JOB: build  │ │     │ │ JOB: unit    │ │     │ │ JOB: deploy  │ │
│ │  the app     │ │     │ │ tests        │ │     │ │ to staging   │ │
│ └──────────────┘ │     │ ├──────────────┤ │     │ └──────────────┘ │
│                  │     │ │ JOB: lint    │ │     │                  │
│                  │     │ │ check        │ │     │                  │
│                  │     │ └──────────────┘ │     │                  │
└──────────────────┘     └──────────────────┘     └──────────────────┘
                               ▲
                               │
                     Jobs in the same stage
                     run in PARALLEL
```

### The hierarchy:

```
Pipeline
  └── Stage 1: build
  │     └── Job: compile
  │     └── Job: build-image
  └── Stage 2: test
  │     └── Job: unit-tests      ← These run in PARALLEL
  │     └── Job: lint            ← (same stage = parallel)
  │     └── Job: security-scan
  └── Stage 3: deploy
        └── Job: deploy-staging
        └── Job: deploy-prod
```

### Key concepts:

| Concept | What it is |
|---------|-----------|
| **Pipeline** | The top-level container — one pipeline per commit/push |
| **Stage** | A phase of the pipeline (build, test, deploy). Stages run **sequentially** |
| **Job** | A task that runs commands. Jobs in the same stage run in **parallel** |
| **Runner** | The machine (server/container) that actually executes the jobs |

## 3. The `.gitlab-ci.yml` File — Your Pipeline Blueprint

The `.gitlab-ci.yml` file is the **single source of truth** for your CI/CD pipeline. It lives in the **root** of your Git repository.

### Basic structure:

```yaml
# .gitlab-ci.yml

# 1. Define the stages (order matters!)
stages:
  - build
  - test
  - deploy

# 2. Define jobs (each job belongs to a stage)
build-job:
  stage: build
  script:
    - echo "Building the application..."
    - npm run build

test-job:
  stage: test
  script:
    - echo "Running tests..."
    - npm test

deploy-job:
  stage: deploy
  script:
    - echo "Deploying to production..."
    - ./deploy.sh
```

### Line-by-line breakdown:

```yaml
stages:           # ← Define the ORDER of stages
  - build         #    Stage 1 (runs first)
  - test          #    Stage 2 (runs after build completes)
  - deploy        #    Stage 3 (runs after test completes)

build-job:        # ← This is a JOB name (you choose any name)
  stage: build    # ← Which stage does this job belong to?
  script:         # ← The commands to run (required for every job!)
    - echo "Hi"   #    Each line is a shell command
    - npm build
```

> **Important Rules:**
> - The file MUST be named `.gitlab-ci.yml` (exactly this name)
> - It MUST be in the root of your repository
> - It uses YAML syntax
> - Every job MUST have a `script` keyword

## 4. Your First Pipeline — Hello World

Let's create the simplest possible pipeline:

```yaml
# .gitlab-ci.yml

hello-world:
  script:
    - echo "Hello, GitLab CI/CD! 🎉"
    - echo "My first pipeline is running!"
    - echo "Current directory: $(pwd)"
    - echo "Files in repo: $(ls)"
```

That's it! Just **one job** with a `script`. No stages needed (GitLab defaults to the `test` stage).

### What happens when you push this:

```
1. You push to GitLab
2. GitLab finds .gitlab-ci.yml
3. GitLab creates a pipeline with one job: "hello-world"
4. A Runner picks up the job
5. The Runner executes the script commands
6. You see the output in the pipeline UI

Pipeline → Stages → Jobs → Script Output
   │         │        │         │
   │         │        │         └── "Hello, GitLab CI/CD! 🎉"
   │         │        └── hello-world (job)
   │         └── test (default stage)
   └── Pipeline #1 (passed ✅)
```

### A slightly more useful first pipeline:

```yaml
# .gitlab-ci.yml

stages:
  - build
  - test

build:
  stage: build
  script:
    - echo "=== Building Application ==="
    - echo "Commit SHA: $CI_COMMIT_SHA"
    - echo "Branch: $CI_COMMIT_BRANCH"
    - echo "Project: $CI_PROJECT_NAME"

test:
  stage: test
  script:
    - echo "=== Running Tests ==="
    - echo "Pipeline ID: $CI_PIPELINE_ID"
    - echo "Job ID: $CI_JOB_ID"
```

> **Notice:** GitLab provides tons of **predefined variables** like `$CI_COMMIT_SHA`, `$CI_COMMIT_BRANCH`, etc. These are automatically available in every job. We'll cover them in detail in the next section (11.18).

## 5. Understanding Stages

Stages define the **order** of execution in your pipeline.

### Rules of stages:

1. **Stages run sequentially** (one after another)
2. **Jobs in the same stage run in parallel** (if enough runners are available)
3. **If a job in a stage fails, the next stage is NOT started** (by default)
4. **If ALL jobs in a stage pass, the next stage starts**

```yaml
stages:
  - build        # Runs first
  - test         # Runs after ALL build jobs pass
  - deploy       # Runs after ALL test jobs pass
```

### Visual example:

```yaml
stages:
  - build
  - test
  - deploy

compile:
  stage: build
  script: echo "Compiling..."

package:
  stage: build
  script: echo "Packaging..."

unit-tests:
  stage: test
  script: echo "Unit testing..."

integration-tests:
  stage: test
  script: echo "Integration testing..."

deploy-staging:
  stage: deploy
  script: echo "Deploying to staging..."
```

**Execution flow:**

```
Time ──────────────────────────────────────────────────────────►

Stage: build          Stage: test             Stage: deploy
┌─────────────┐       ┌──────────────────┐    ┌─────────────────┐
│ compile     │──┐    │ unit-tests       │──┐ │ deploy-staging  │
└─────────────┘  │    └──────────────────┘  │ └─────────────────┘
                 ├──►                       ├──►
┌─────────────┐  │    ┌──────────────────┐  │
│ package     │──┘    │ integration-tests│──┘
└─────────────┘       └──────────────────┘

   ◄── Parallel ──►      ◄── Parallel ──►      ◄── Sequential ──►
```

### Default stages

If you don't define `stages:`, GitLab uses these defaults:

```yaml
# Default stages (if you don't specify any):
stages:
  - .pre        # Always runs first (special stage)
  - build
  - test        # Default stage for jobs without a stage
  - deploy
  - .post       # Always runs last (special stage)
```

### The `.pre` and `.post` special stages

```yaml
stages:
  - build
  - test
  - deploy

# This job runs BEFORE everything else
setup:
  stage: .pre
  script: echo "Setting up environment..."

# This job runs AFTER everything else
cleanup:
  stage: .post
  script: echo "Cleaning up..."

build:
  stage: build
  script: echo "Building..."
```

**Execution order:** `.pre` → `build` → `test` → `deploy` → `.post`

## 6. Understanding Jobs

Jobs are the **building blocks** of your pipeline. Each job:
- Belongs to a **stage**
- Runs a **script** (one or more commands)
- Runs on a **runner**
- Is **independent** from other jobs (each job starts fresh)

### Job naming rules:

```yaml
# ✅ Valid job names
build-app:
  script: echo "Building..."

test_application:
  script: echo "Testing..."

deploy to staging:         # Spaces are allowed!
  script: echo "Deploying..."

my-job-123:
  script: echo "Numbers too!"

# ❌ Reserved keywords — CANNOT be used as job names:
# image, services, stages, types, before_script, after_script,
# variables, cache, include, default, pages, workflow
```

### Important: Each job runs independently

```
Job A starts → Fresh environment → Runs script → Ends
Job B starts → Fresh environment → Runs script → Ends

Each job gets a CLEAN workspace. Changes made in Job A
are NOT available in Job B (unless you use artifacts).
```

This is different from Jenkins where the workspace persists between stages!

```yaml
# ❌ This WON'T work as expected:
build:
  stage: build
  script:
    - mkdir output
    - echo "hello" > output/result.txt    # Creates a file

test:
  stage: test
  script:
    - cat output/result.txt               # ❌ ERROR! File doesn't exist!
    # Each job starts fresh — output/ from build job is GONE

# ✅ Use artifacts to pass files between jobs (covered in 11.18)
build:
  stage: build
  script:
    - mkdir output
    - echo "hello" > output/result.txt
  artifacts:
    paths:
      - output/                           # ← Save this for other jobs
```

## 7. Job Keywords Deep Dive

Here are the most important keywords you can use in a job:

### Complete job anatomy:

```yaml
my-job:                        # Job name
  stage: test                  # Which stage (default: test)
  image: node:20-alpine        # Docker image to use
  services:                    # Sidecar containers
    - postgres:15
  variables:                   # Job-level variables
    DB_HOST: postgres
  before_script:               # Runs before "script"
    - npm ci
  script:                      # The main commands (REQUIRED!)
    - npm test
  after_script:                # Runs after "script" (even if script fails)
    - echo "Cleanup..."
  artifacts:                   # Files to save/pass between jobs
    paths:
      - coverage/
  cache:                       # Files to cache between pipeline runs
    paths:
      - node_modules/
  tags:                        # Which runner to use
    - docker
    - linux
  rules:                       # When should this job run?
    - if: $CI_COMMIT_BRANCH == "main"
  allow_failure: false         # If true, pipeline continues even if this fails
  when: on_success             # When to run: on_success, on_failure, always, manual, delayed
  timeout: 30 minutes          # Max execution time
  retry:                       # Retry on failure
    max: 2
  needs: [build-job]           # Run after specific jobs (DAG)
  environment:                 # Deployment environment
    name: production
    url: https://myapp.com
  coverage: '/Coverage: \d+%/' # Regex to extract code coverage
  parallel: 3                  # Run N instances in parallel
```

Don't worry about memorizing all of these! We'll cover the important ones in detail.

## 8. The `script` Keyword — Running Commands

The `script` keyword is the **only required keyword** in a job. It defines the commands to execute.

### Single command:

```yaml
greet:
  script: echo "Hello!"
```

### Multiple commands:

```yaml
build:
  script:
    - echo "Step 1: Installing dependencies"
    - npm install
    - echo "Step 2: Building application"
    - npm run build
    - echo "Step 3: Done!"
```

### Multi-line commands:

```yaml
deploy:
  script:
    - |
      echo "Starting deployment..."
      if [ "$CI_COMMIT_BRANCH" = "main" ]; then
        echo "Deploying to production"
        ./deploy.sh production
      else
        echo "Deploying to staging"
        ./deploy.sh staging
      fi
    - echo "Deployment complete!"
```

### Important: Script failure behavior

```yaml
test:
  script:
    - echo "Step 1"         # Runs
    - false                  # ← This command fails (exit code 1)
    - echo "Step 3"         # ❌ NOT reached! Job stops here.
```

By default, if any command exits with a **non-zero exit code**, the job **stops immediately** and is marked as **failed**.

### Handling expected failures:

```yaml
test:
  script:
    - echo "Step 1"
    - command_that_might_fail || true    # ← Continue even if this fails
    - echo "Step 3"                      # ← This WILL run
```

## 9. The `before_script` and `after_script`

### `before_script` — Setup steps

Runs **before** the main `script`. Perfect for setup tasks like installing dependencies.

```yaml
test:
  before_script:
    - echo "=== Setting up ==="
    - apt-get update && apt-get install -y curl
    - npm ci
  script:
    - npm test
```

### `after_script` — Cleanup steps

Runs **after** the main `script`, **even if the script fails**! Perfect for cleanup.

```yaml
test:
  script:
    - npm test            # Even if this fails...
  after_script:
    - echo "Cleaning up..." # ...this STILL runs!
    - rm -rf temp/
```

### Global `before_script` and `after_script`

You can define them globally (applies to ALL jobs):

```yaml
# Global — applies to every job
default:
  before_script:
    - echo "Starting job: $CI_JOB_NAME"
  after_script:
    - echo "Finished job: $CI_JOB_NAME"

build:
  script:
    - npm run build

test:
  script:
    - npm test
  before_script:              # ← This OVERRIDES the global before_script
    - echo "Custom setup for test"
    - npm ci
```

> **Important:** If a job defines its own `before_script`, it **replaces** the global one (it doesn't add to it).

### Execution order:

```
1. before_script (runs first)
2. script (runs second)
3. after_script (runs last — ALWAYS, even on failure)
```

## 10. Job Dependencies and Ordering

### Default behavior: Stage-based ordering

By default, jobs are ordered by **stages**:

```yaml
stages:
  - build
  - test
  - deploy

build-app:
  stage: build
  script: echo "Building..."    # Runs first

run-tests:
  stage: test
  script: echo "Testing..."    # Runs after ALL build jobs pass

deploy-app:
  stage: deploy
  script: echo "Deploying..."  # Runs after ALL test jobs pass
```

### The `needs` keyword — DAG (Directed Acyclic Graph)

The `needs` keyword lets you create **custom dependencies** between jobs, regardless of stages. This can make your pipeline much faster!

```yaml
stages:
  - build
  - test
  - deploy

build-frontend:
  stage: build
  script: echo "Building frontend..."

build-backend:
  stage: build
  script: echo "Building backend..."

test-frontend:
  stage: test
  needs: [build-frontend]        # ← Only needs frontend build!
  script: echo "Testing frontend..."

test-backend:
  stage: test
  needs: [build-backend]         # ← Only needs backend build!
  script: echo "Testing backend..."

deploy:
  stage: deploy
  needs: [test-frontend, test-backend]
  script: echo "Deploying..."
```

### Without `needs` vs with `needs`:

```
WITHOUT needs (stage-based — slower):
┌─────────────┐  ┌─────────────┐     ┌──────────────┐  ┌──────────────┐     ┌────────┐
│ build-front │  │ build-back  │     │ test-front   │  │ test-back    │     │ deploy │
└─────────────┘  └─────────────┘     └──────────────┘  └──────────────┘     └────────┘
◄──── WAIT for both ────►  THEN  ◄──── WAIT for both ────►  THEN  ◄── run ──►

Total: build time + test time + deploy time = ⏱️ SLOW


WITH needs (DAG — faster):
┌─────────────┐  ┌──────────────┐
│ build-front │─►│ test-front   │──┐
└─────────────┘  └──────────────┘  │   ┌────────┐
                                    ├──►│ deploy │
┌─────────────┐  ┌──────────────┐  │   └────────┘
│ build-back  │─►│ test-back    │──┘
└─────────────┘  └──────────────┘

test-front starts as soon as build-front finishes (doesn't wait for build-back!)
Total: max(front path, back path) + deploy time = ⏱️ FASTER!
```

### The `dependencies` keyword

`dependencies` controls which **artifacts** are downloaded (not execution order):

```yaml
build:
  stage: build
  script: npm run build
  artifacts:
    paths: [dist/]

test:
  stage: test
  dependencies: [build]         # ← Download artifacts from "build" job
  script: ls dist/              # dist/ is available because of dependencies

deploy:
  stage: deploy
  dependencies: []              # ← Download NO artifacts (faster!)
  script: echo "Deploying..."
```

## 11. The `only` / `except` vs `rules` — Controlling When Jobs Run

### `only` and `except` (old way — still works but avoid in new projects)

```yaml
# Run only on the main branch
deploy:
  script: ./deploy.sh
  only:
    - main

# Run on all branches EXCEPT feature branches
build:
  script: npm run build
  except:
    - /^feature\/.*$/          # Regex pattern

# Run only on tags
release:
  script: ./release.sh
  only:
    - tags

# Run only on merge requests
test:
  script: npm test
  only:
    - merge_requests
```

> **⚠️ Warning:** `only`/`except` is the older syntax. GitLab recommends using `rules` instead (see next section). Don't mix `only`/`except` with `rules` in the same job!

## 12. The `rules` Keyword — Modern Approach

`rules` is the modern, more powerful way to control when jobs run. It uses **conditions** with `if`, `changes`, and `exists`.

### Basic `rules` with `if`:

```yaml
deploy-production:
  script: ./deploy.sh production
  rules:
    - if: $CI_COMMIT_BRANCH == "main"       # Only on main branch
      when: manual                           # Require manual click to start
    - if: $CI_COMMIT_BRANCH == "staging"
      when: on_success                       # Run automatically
    - when: never                            # Don't run on any other branch
```

### How `rules` works:

```
GitLab checks each rule from TOP to BOTTOM:
  Rule 1: Does the condition match? → YES → Use this rule's "when"
  Rule 1: Does the condition match? → NO  → Check Rule 2
  Rule 2: Does the condition match? → YES → Use this rule's "when"
  Rule 2: Does the condition match? → NO  → Check Rule 3
  ...
  No rules matched? → Job is NOT added to the pipeline
```

### `rules` with `changes` (file-based triggers):

```yaml
# Only run frontend tests when frontend files change
test-frontend:
  script: npm test
  rules:
    - changes:
        - "frontend/**/*"                   # Only when files in frontend/ change
        - "package.json"
        - "package-lock.json"

# Only rebuild Docker image when Dockerfile changes
build-image:
  script: docker build -t myapp .
  rules:
    - changes:
        - Dockerfile
        - "src/**/*"
```

### `rules` with `exists` (file existence):

```yaml
# Only run if a Dockerfile exists
build-docker:
  script: docker build .
  rules:
    - exists:
        - Dockerfile                        # Run only if Dockerfile exists

# Only run Python tests if requirements.txt exists
test-python:
  script: pytest
  rules:
    - exists:
        - requirements.txt
```

### Combining conditions:

```yaml
deploy:
  script: ./deploy.sh
  rules:
    # Deploy to production: only main branch, manual trigger
    - if: $CI_COMMIT_BRANCH == "main"
      when: manual
      allow_failure: true                    # Pipeline doesn't wait for this

    # Deploy to staging: only on merge requests to main
    - if: $CI_PIPELINE_SOURCE == "merge_request_event" && $CI_MERGE_REQUEST_TARGET_BRANCH_NAME == "main"
      when: on_success

    # Never deploy otherwise
    - when: never
```

### Common `rules` patterns:

```yaml
# Pattern 1: Run on main + MRs, skip everything else
rules:
  - if: $CI_COMMIT_BRANCH == "main"
  - if: $CI_PIPELINE_SOURCE == "merge_request_event"

# Pattern 2: Run always except on schedules
rules:
  - if: $CI_PIPELINE_SOURCE == "schedule"
    when: never
  - when: on_success

# Pattern 3: Run on tag pushes only
rules:
  - if: $CI_COMMIT_TAG

# Pattern 4: Manual deployment with environment protection
rules:
  - if: $CI_COMMIT_BRANCH == "main"
    when: manual
```

## 13. GitLab Runners — What Are They?

A **GitLab Runner** is the agent that **actually executes your CI/CD jobs**. Think of it like a worker in a factory — GitLab is the manager that decides WHAT needs to be done, and the Runner is the worker that DOES the work.

```
┌──────────────────┐         ┌──────────────────┐
│   GitLab Server  │         │   GitLab Runner  │
│   (Manager)      │         │   (Worker)       │
│                  │         │                  │
│  "Here's a job   │ ──────► │  "Got it! Let me │
│   to run. Build  │         │   execute those  │
│   this project." │         │   commands..."   │
│                  │ ◄────── │                  │
│  "Job finished   │         │  "Done! Here's   │
│   with status    │         │   the output."   │
│   and logs."     │         │                  │
└──────────────────┘         └──────────────────┘
      (gitlab.com              (Your server,
       or self-hosted)          cloud VM, or
                                local machine)
```

### How the Runner works:

```
1. Runner constantly POLLS GitLab: "Any jobs for me?"
2. GitLab responds: "Yes! Here's a job to run."
3. Runner CLONES the repository
4. Runner executes before_script → script → after_script
5. Runner sends logs and results back to GitLab
6. Runner goes back to polling: "Any more jobs?"
```

## 14. Types of Runners

### 1. Shared Runners (gitlab.com)

If you're using **gitlab.com** (the SaaS version), GitLab provides free shared runners. You don't need to set up anything!

```
Pros:
  ✅ No setup required
  ✅ Free for open-source projects
  ✅ Auto-scaling
  
Cons:
  ❌ Limited free minutes (400 min/month on free plan)
  ❌ Shared with other users (might be slow)
  ❌ Can't customize the environment
```

### 2. Group Runners

Available to all projects in a GitLab **group** (organization).

```
GitLab Group: "my-company"
  ├── Project A ── Can use group runner ✅
  ├── Project B ── Can use group runner ✅
  └── Project C ── Can use group runner ✅
```

### 3. Project-Specific Runners

Dedicated to a **single project**. Only that project can use it.

```
Runner "build-server-01"
  └── Only available to: my-company/my-app
```

### Comparison table:

| Type | Scope | Setup | Best For |
|------|-------|-------|----------|
| **Shared** | All projects on instance | Already done (gitlab.com) | Small projects, open source |
| **Group** | All projects in a group | Admin sets up once | Teams, organizations |
| **Project** | Single project | Per project | Specialized builds, security |

## 15. Runner Executors

The **executor** determines **how** the Runner executes your job. Think of it as the Runner's "engine".

### Common executors:

| Executor | How it runs jobs | Best For |
|----------|-----------------|----------|
| **Docker** | Each job runs in a fresh Docker container | Most common! Clean, isolated |
| **Shell** | Runs directly on the runner's host OS | Simple, direct access to host |
| **Docker Machine** | Auto-creates Docker hosts on cloud | Auto-scaling |
| **Kubernetes** | Runs jobs as Kubernetes pods | Cloud-native, K8s environments |
| **SSH** | Connects to remote server via SSH | Remote execution |
| **VirtualBox** | Runs in a virtual machine | Full OS isolation |
| **Parallels** | Runs in Parallels VM (macOS) | macOS/iOS builds |

### Docker Executor (most popular):

```
Job starts
    │
    ▼
┌───────────────────────────────┐
│ Docker Container              │
│ (fresh for every job!)        │
│                               │
│  ┌─────────────────────────┐  │
│  │ 1. Clone repository     │  │
│  │ 2. Run before_script    │  │
│  │ 3. Run script           │  │
│  │ 4. Run after_script     │  │
│  │ 5. Upload artifacts     │  │
│  └─────────────────────────┘  │
│                               │
└───────────────────────────────┘
    │
    ▼
Container destroyed (clean state for next job)
```

### Shell Executor:

```
Job starts
    │
    ▼
┌───────────────────────────────┐
│ Host Machine                  │
│ (Runner's own OS)             │
│                               │
│  Runs commands directly       │
│  in a shell (bash/sh/cmd)     │
│                               │
│  ⚠️ NOT isolated!             │
│  Changes persist between jobs │
│  Security risk if untrusted   │
│  code runs here               │
└───────────────────────────────┘
```

## 16. Installing and Registering a Runner

### Step 1: Install GitLab Runner

**On Ubuntu/Debian:**

```bash
# Add the GitLab Runner repository
curl -L "https://packages.gitlab.com/install/repositories/runner/gitlab-runner/script.deb.sh" | sudo bash

# Install GitLab Runner
sudo apt-get install gitlab-runner
```

**On CentOS/RHEL:**

```bash
curl -L "https://packages.gitlab.com/install/repositories/runner/gitlab-runner/script.rpm.sh" | sudo bash
sudo yum install gitlab-runner
```

**Using Docker:**

```bash
docker run -d \
  --name gitlab-runner \
  --restart always \
  -v /srv/gitlab-runner/config:/etc/gitlab-runner \
  -v /var/run/docker.sock:/var/run/docker.sock \
  gitlab/gitlab-runner:latest
```

### Step 2: Register the Runner

Registration connects your Runner to your GitLab instance.

```bash
sudo gitlab-runner register
```

You'll be asked a series of questions:

```
Enter the GitLab instance URL:
> https://gitlab.com/                    # or your self-hosted URL

Enter the registration token:
> glrt-xxxxxxxxxxxxxxxxxxxx              # Found in GitLab → Settings → CI/CD → Runners

Enter a description for the runner:
> my-docker-runner                       # A friendly name

Enter tags for the runner (comma-separated):
> docker,linux,node                      # Tags to match jobs (covered next)

Enter optional maintenance note:
>                                        # Press Enter to skip

Enter an executor:
> docker                                 # Choose: docker, shell, kubernetes, etc.

Enter the default Docker image:
> node:20-alpine                         # Default image for Docker executor
```

### Where to find the registration token:

```
For project runners:
  Project → Settings → CI/CD → Runners → "New project runner"

For group runners:
  Group → Settings → CI/CD → Runners → "New group runner"

For instance runners (admin only):
  Admin Area → CI/CD → Runners → "New instance runner"
```

### Step 3: Verify the runner is working

```bash
# Check runner status
sudo gitlab-runner status

# List registered runners
sudo gitlab-runner list

# Run a test
sudo gitlab-runner verify
```

### The runner config file

The runner configuration is stored at `/etc/gitlab-runner/config.toml`:

```toml
concurrent = 4               # Max concurrent jobs
check_interval = 3           # How often to check for new jobs (seconds)

[[runners]]
  name = "my-docker-runner"
  url = "https://gitlab.com/"
  token = "glrt-xxxxxxxxxxxx"
  executor = "docker"
  
  [runners.docker]
    image = "node:20-alpine"  # Default Docker image
    privileged = false        # Don't allow privileged containers
    volumes = ["/cache"]      # Mount cache volume
    shm_size = 0              # Shared memory size
```

## 17. Tags — Matching Jobs to Runners

Tags are like **labels** that connect jobs to the right runners.

### How tags work:

```
Runner "build-server" has tags: [docker, linux, gpu]
Runner "mac-mini" has tags: [macos, ios]

Job A has tags: [docker]     → Runs on "build-server" ✅
Job B has tags: [macos]      → Runs on "mac-mini" ✅
Job C has tags: [windows]    → ❌ No matching runner! Job stays pending.
```

### Using tags in `.gitlab-ci.yml`:

```yaml
# This job will ONLY run on a runner that has BOTH tags: "docker" AND "linux"
build:
  tags:
    - docker
    - linux
  script:
    - npm run build

# This job requires a macOS runner
build-ios:
  tags:
    - macos
    - xcode
  script:
    - xcodebuild -scheme MyApp

# This job requires a GPU runner
train-model:
  tags:
    - gpu
    - cuda
  script:
    - python train.py
```

### What happens if there's no matching runner?

```
Job status: "pending" (waiting for a runner with matching tags)
           ▼
After a while: "stuck" (still no matching runner)
           ▼
Eventually: timeout (if configured)
```

> **Tip:** If a job is stuck, check if the runner tags match the job tags!

## 18. The `image` Keyword — Docker-Based Pipelines

When using the Docker executor, the `image` keyword specifies which Docker image to use for the job.

### Setting the image:

```yaml
# Global image (applies to all jobs)
default:
  image: node:20-alpine

# Per-job image (overrides global)
build:
  image: node:20-alpine
  script: npm run build

test-python:
  image: python:3.12-slim
  script: pytest

deploy:
  image: alpine/k8s:1.28.0
  script: kubectl apply -f k8s/
```

### How the Docker image works:

```
1. Runner pulls the image: docker pull node:20-alpine
2. Runner creates a container from the image
3. Runner clones your repo INTO the container
4. Runner runs your script commands INSIDE the container
5. Runner destroys the container when done

Each job = Fresh container = Clean environment!
```

### Using a private registry image:

```yaml
build:
  image:
    name: registry.example.com/my-org/custom-image:latest
    entrypoint: [""]          # Override entrypoint if needed
  script:
    - my-custom-tool build
```

### Best practices for images:

```yaml
# ✅ Good — specific version, small image
image: node:20-alpine

# ⚠️ Avoid — "latest" can break unexpectedly
image: node:latest

# ✅ Good — use digest for maximum reproducibility
image: node@sha256:abc123...

# ✅ Good — alpine variants are smaller and faster to pull
image: python:3.12-alpine   # ~50MB
# vs
image: python:3.12           # ~900MB
```

## 19. Services — Running Sidecar Containers

The `services` keyword runs **additional Docker containers** alongside your job. These are perfect for databases, caches, and other services your tests need.

### Example: Running tests with a database:

```yaml
test:
  image: node:20-alpine
  services:
    - postgres:15                # ← Runs a PostgreSQL container
    - redis:7-alpine             # ← Runs a Redis container
  variables:
    POSTGRES_DB: test_db
    POSTGRES_USER: runner
    POSTGRES_PASSWORD: password
    DATABASE_URL: "postgresql://runner:password@postgres:5432/test_db"
    REDIS_URL: "redis://redis:6379"
  script:
    - npm ci
    - npm test
```

### How services work:

```
┌─────────────────────────────────────────────────┐
│                   Docker Network                 │
│                                                  │
│  ┌─────────────────┐  ┌──────────┐  ┌────────┐ │
│  │  Job Container   │  │ postgres │  │ redis  │ │
│  │  (node:20)       │  │  :15     │  │ :7     │ │
│  │                  │  │          │  │        │ │
│  │  Your code runs  │  │ Port:    │  │ Port:  │ │
│  │  here            │  │  5432    │  │  6379  │ │
│  └─────────────────┘  └──────────┘  └────────┘ │
│         │                   ▲            ▲      │
│         │                   │            │      │
│         └───── Can connect ─┴────────────┘      │
│               via hostname                       │
│         (e.g., "postgres", "redis")              │
└─────────────────────────────────────────────────┘
```

### Service hostnames:

The service is available at a hostname that matches the image name:

```yaml
services:
  - postgres:15           # Hostname: "postgres"
  - redis:7               # Hostname: "redis"
  - mysql:8.0             # Hostname: "mysql"
  - mongo:7               # Hostname: "mongo"

# For images with a registry:
services:
  - registry.example.com/my-db:latest   # Hostname: "registry.example.com__my-db"
                                         # (slashes become double underscores)
```

### Service with alias (custom hostname):

```yaml
services:
  - name: postgres:15
    alias: db                            # ← Use "db" as hostname instead
  - name: redis:7
    alias: cache                         # ← Use "cache" as hostname

test:
  script:
    - ping db                           # ← Now uses alias
    - redis-cli -h cache ping           # ← Now uses alias
```

## 21. Real-World Pipeline Examples

### Example 1: Node.js Application

```yaml
# .gitlab-ci.yml — Complete Node.js Pipeline

stages:
  - install
  - quality
  - build
  - deploy

default:
  image: node:20-alpine

# --- Cache node_modules across all jobs ---
cache:
  key: ${CI_COMMIT_REF_SLUG}
  paths:
    - node_modules/

# --- Install dependencies ---
install-deps:
  stage: install
  script:
    - npm ci
  artifacts:
    paths:
      - node_modules/
    expire_in: 1 hour

# --- Code Quality ---
lint:
  stage: quality
  needs: [install-deps]
  script:
    - npm run lint

unit-tests:
  stage: quality
  needs: [install-deps]
  services:
    - postgres:15
  variables:
    POSTGRES_DB: test
    POSTGRES_USER: test
    POSTGRES_PASSWORD: test
    DATABASE_URL: "postgresql://test:test@postgres:5432/test"
  script:
    - npm test
  coverage: '/All files\s*\|\s*(\d+\.?\d*)\s*\|/'
  artifacts:
    reports:
      junit: junit.xml
      coverage_report:
        coverage_format: cobertura
        path: coverage/cobertura-coverage.xml

# --- Build ---
build:
  stage: build
  needs: [lint, unit-tests]
  script:
    - npm run build
  artifacts:
    paths:
      - dist/
    expire_in: 1 day

# --- Deploy ---
deploy-staging:
  stage: deploy
  needs: [build]
  image: alpine/k8s:1.28.0
  script:
    - kubectl apply -f k8s/staging/
  environment:
    name: staging
    url: https://staging.myapp.com
  rules:
    - if: $CI_COMMIT_BRANCH == "main"

deploy-production:
  stage: deploy
  needs: [build]
  image: alpine/k8s:1.28.0
  script:
    - kubectl apply -f k8s/production/
  environment:
    name: production
    url: https://myapp.com
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
      when: manual                           # Manual deployment to production
```

## 23. `extends` — Inheriting Job Configuration

The `extends` keyword lets you create **base job templates** and inherit from them. It's like **class inheritance** in programming.

### Basic extends:

```yaml
# Base template (starts with a dot, so it's hidden — won't run as a job)
.test-template:
  image: node:20-alpine
  before_script:
    - npm ci
  cache:
    paths:
      - node_modules/

# Jobs that inherit from the template
unit-tests:
  extends: .test-template          # ← Inherits everything from .test-template
  script:
    - npm run test:unit

integration-tests:
  extends: .test-template          # ← Also inherits from .test-template
  script:
    - npm run test:integration
  services:
    - postgres:15                  # ← Can add extra stuff
```

### What `.unit-tests` actually becomes (after merging):

```yaml
# GitLab merges the template with the job:
unit-tests:
  image: node:20-alpine           # ← From .test-template
  before_script:                   # ← From .test-template
    - npm ci
  cache:                           # ← From .test-template
    paths:
      - node_modules/
  script:                          # ← From unit-tests (overrides template)
    - npm run test:unit
```

### Hidden jobs (dot prefix):

```yaml
# Jobs starting with a DOT are "hidden" — they DON'T run as jobs
.base:                             # ← Hidden (template only)
  image: node:20

my-job:                            # ← Visible (actually runs)
  extends: .base
  script: echo "Hello"
```

### Multi-level extends:

```yaml
.base:
  image: node:20
  tags:
    - docker

.test-base:
  extends: .base                  # ← Inherits from .base
  before_script:
    - npm ci

unit-test:
  extends: .test-base             # ← Inherits from .test-base (which inherits from .base)
  script:
    - npm test
```

## 24. Best Practices

### 1. Always define stages explicitly

```yaml
# ✅ Good — clear pipeline structure
stages:
  - build
  - test
  - deploy

# ❌ Bad — relying on defaults, confusing for team members
# (no stages defined)
```

### 2. Use `rules` instead of `only`/`except`

```yaml
# ✅ Modern approach
rules:
  - if: $CI_COMMIT_BRANCH == "main"

# ❌ Old approach (avoid in new projects)
only:
  - main
```

### 3. Keep jobs small and focused

```yaml
# ✅ Good — each job does ONE thing
lint:
  script: npm run lint

unit-test:
  script: npm test

build:
  script: npm run build

# ❌ Bad — one giant job
do-everything:
  script:
    - npm run lint
    - npm test
    - npm run build
    - npm run deploy
```

### 4. Use `needs` for faster pipelines

```yaml
# ✅ Good — parallel where possible
test:
  needs: [install]     # Starts as soon as install finishes

lint:
  needs: [install]     # Also starts as soon as install finishes
                       # (parallel with test!)
```

### 5. Use templates for common patterns

```yaml
# ✅ Good — DRY with extends
.deploy-template:
  image: alpine/k8s:1.28.0
  script: kubectl apply -f k8s/$ENV/

deploy-staging:
  extends: .deploy-template
  variables:
    ENV: staging

deploy-prod:
  extends: .deploy-template
  variables:
    ENV: production
```

### 6. Set appropriate timeouts

```yaml
# ✅ Good — prevent runaway jobs
build:
  timeout: 10 minutes
  script: npm run build

# Default timeout is 1 hour — that's often too long
```

## 25. Common Mistakes and Troubleshooting

### Mistake 1: Job name is a reserved word

```yaml
# ❌ ERROR — "image" is a reserved keyword
image:
  script: echo "Building image"

# ✅ FIX — Use a different name
build-image:
  script: echo "Building image"
```

### Mistake 2: Forgetting `script`

```yaml
# ❌ ERROR — Every job MUST have a "script"
build:
  image: node:20
  # No script! GitLab will show an error.

# ✅ FIX — Add a script
build:
  image: node:20
  script:
    - npm run build
```

### Mistake 3: Indentation errors (YAML!)

```yaml
# ❌ ERROR — Mixed tabs and spaces, wrong indentation
build:
script:                    # Should be indented under "build:"
- npm run build

# ✅ FIX — Proper YAML indentation (2 spaces)
build:
  script:
    - npm run build
```

### Mistake 4: Job stuck in "pending"

```
Problem: Job shows "pending" forever
Common causes:
  1. No runner with matching tags
  2. All runners are busy
  3. Runner is offline

Fix:
  - Check Settings → CI/CD → Runners (is runner online?)
  - Check job tags vs runner tags
  - Remove tags from job (if using shared runners)
```

### Mistake 5: Files from previous jobs not available

```yaml
# ❌ Files from "build" are NOT available in "test" by default!
build:
  script: npm run build    # Creates dist/

test:
  script: ls dist/         # ❌ dist/ doesn't exist!

# ✅ FIX — Use artifacts
build:
  script: npm run build
  artifacts:
    paths:
      - dist/

test:
  script: ls dist/         # ✅ dist/ is available via artifacts
```
