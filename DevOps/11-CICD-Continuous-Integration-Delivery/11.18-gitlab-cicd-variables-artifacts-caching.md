## 1. The Three Pillars — Variables, Artifacts, and Cache

Before we dive in, let's understand what each one does and when to use it:

```
┌─────────────────────────────────────────────────────────────────┐
│                   DATA FLOW IN GITLAB CI/CD                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  VARIABLES                                                       │
│    What: Key-value pairs (strings)                               │
│    For:  Configuration, secrets, environment settings            │
│    Scope: Pipeline, job, or environment                          │
│    Example: DATABASE_URL, API_KEY, NODE_ENV                      │
│                                                                  │
│  ARTIFACTS                                                       │
│    What: Files/directories saved AFTER a job finishes            │
│    For:  Passing build outputs between JOBS                      │
│    Scope: Within a pipeline (job to job)                         │
│    Example: dist/, coverage reports, compiled binaries           │
│                                                                  │
│  CACHE                                                           │
│    What: Files saved to speed up FUTURE pipeline runs            │
│    For:  Avoiding re-downloading dependencies                    │
│    Scope: Across PIPELINES (not guaranteed!)                     │
│    Example: node_modules/, .pip-cache/, .m2/                     │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### Quick decision guide:

```
"I need to pass a DATABASE_URL to my test job"
    → Use a VARIABLE

"I need to pass build output (dist/) from build job to deploy job"
    → Use ARTIFACTS

"I want to speed up npm install by saving node_modules/"
    → Use CACHE
```

## 2. Variables — Overview

Variables in GitLab CI/CD are like **environment variables** in Linux. They're key-value pairs that are available in your job scripts.

### Where can you define variables?

```
1. Predefined by GitLab     → $CI_COMMIT_SHA, $CI_PROJECT_NAME, etc.
2. In .gitlab-ci.yml         → Custom variables in your pipeline file
3. In GitLab UI (Settings)   → Project/Group settings → CI/CD → Variables
4. In the API                → Programmatically via GitLab API
5. When triggering manually  → Enter values when starting a pipeline
```

### How to use variables in your pipeline:

```yaml
my-job:
  script:
    # Method 1: Shell variable syntax (most common)
    - echo $MY_VARIABLE
    - echo ${MY_VARIABLE}

    # Method 2: GitLab CI variable syntax
    - echo $CI_COMMIT_SHA
    
    # In YAML values (not just scripts!)
    - echo "Branch is $CI_COMMIT_BRANCH"
```

## 3. Predefined CI/CD Variables

GitLab automatically provides **hundreds of predefined variables**. Here are the most important ones:

### Commit and Branch Info:

| Variable | Example Value | Description |
|----------|--------------|-------------|
| `$CI_COMMIT_SHA` | `a1b2c3d4e5f6...` | Full commit SHA |
| `$CI_COMMIT_SHORT_SHA` | `a1b2c3d4` | Short commit SHA (8 chars) |
| `$CI_COMMIT_BRANCH` | `main` | Branch name (empty for tags/MRs) |
| `$CI_COMMIT_TAG` | `v1.2.3` | Tag name (empty for branches) |
| `$CI_COMMIT_MESSAGE` | `Fix bug #123` | Full commit message |
| `$CI_COMMIT_TITLE` | `Fix bug #123` | First line of commit message |
| `$CI_COMMIT_AUTHOR` | `John Doe` | Commit author |
| `$CI_COMMIT_REF_NAME` | `main` or `v1.0` | Branch or tag name |
| `$CI_COMMIT_REF_SLUG` | `my-feature-branch` | URL-safe version of ref name |

### Project Info:

| Variable | Example Value | Description |
|----------|--------------|-------------|
| `$CI_PROJECT_NAME` | `my-app` | Project name |
| `$CI_PROJECT_PATH` | `my-org/my-app` | Full project path |
| `$CI_PROJECT_URL` | `https://gitlab.com/my-org/my-app` | Project URL |
| `$CI_PROJECT_DIR` | `/builds/my-org/my-app` | Directory where repo is cloned |
| `$CI_PROJECT_ID` | `12345` | Project ID number |

### Pipeline and Job Info:

| Variable | Example Value | Description |
|----------|--------------|-------------|
| `$CI_PIPELINE_ID` | `987654` | Pipeline ID |
| `$CI_PIPELINE_SOURCE` | `push`, `merge_request_event` | What triggered the pipeline |
| `$CI_JOB_ID` | `123456` | Job ID |
| `$CI_JOB_NAME` | `build-app` | Job name |
| `$CI_JOB_STAGE` | `build` | Current stage |
| `$CI_JOB_STATUS` | `success` | Job status (in after_script) |
| `$CI_JOB_TOKEN` | `token-value` | Auto-generated token for API |

### Registry Info:

| Variable | Example Value | Description |
|----------|--------------|-------------|
| `$CI_REGISTRY` | `registry.gitlab.com` | Container registry URL |
| `$CI_REGISTRY_IMAGE` | `registry.gitlab.com/org/app` | Full image path |
| `$CI_REGISTRY_USER` | `gitlab-ci-token` | Registry username |
| `$CI_REGISTRY_PASSWORD` | `token-value` | Registry password |

### Merge Request Info:

| Variable | Example Value | Description |
|----------|--------------|-------------|
| `$CI_MERGE_REQUEST_ID` | `42` | MR internal ID |
| `$CI_MERGE_REQUEST_IID` | `7` | MR display ID |
| `$CI_MERGE_REQUEST_SOURCE_BRANCH_NAME` | `feature/login` | Source branch |
| `$CI_MERGE_REQUEST_TARGET_BRANCH_NAME` | `main` | Target branch |
| `$CI_MERGE_REQUEST_TITLE` | `Add login feature` | MR title |

### Using predefined variables — Examples:

```yaml
build-image:
  script:
    # Build a Docker image tagged with the commit SHA
    - docker build -t $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA .
    
    # Log in to GitLab Container Registry (using built-in credentials!)
    - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $CI_REGISTRY
    
    # Push the image
    - docker push $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA

show-info:
  script:
    - echo "Project: $CI_PROJECT_NAME"
    - echo "Branch: $CI_COMMIT_BRANCH"
    - echo "Commit: $CI_COMMIT_SHORT_SHA"
    - echo "Pipeline: $CI_PIPELINE_ID"
    - echo "Triggered by: $CI_PIPELINE_SOURCE"
```

### How to see ALL available variables:

```yaml
debug:
  script:
    - export                   # Shows ALL environment variables
    # Or for a specific one:
    - echo $CI_COMMIT_SHA
```

## 4. Custom Variables — Defined in `.gitlab-ci.yml`

### Global variables (available to ALL jobs):

```yaml
variables:
  NODE_ENV: production
  APP_NAME: my-awesome-app
  API_URL: https://api.example.com

build:
  script:
    - echo "Building $APP_NAME in $NODE_ENV mode"
    - echo "API: $API_URL"

test:
  script:
    - echo "Testing $APP_NAME"          # Also has access to global variables
```

### Job-level variables (only available in that job):

```yaml
variables:
  GLOBAL_VAR: "I'm available everywhere"

build:
  variables:
    BUILD_TYPE: release                  # Only available in this job
  script:
    - echo $GLOBAL_VAR                   # ✅ Works
    - echo $BUILD_TYPE                   # ✅ Works

test:
  script:
    - echo $GLOBAL_VAR                   # ✅ Works
    - echo $BUILD_TYPE                   # ❌ Empty! Not available here.
```

### Job variables override global ones:

```yaml
variables:
  ENVIRONMENT: development               # Global default

deploy-staging:
  variables:
    ENVIRONMENT: staging                 # ← Overrides global for this job
  script:
    - echo "Deploying to $ENVIRONMENT"   # Outputs: "Deploying to staging"

deploy-prod:
  variables:
    ENVIRONMENT: production              # ← Overrides global for this job
  script:
    - echo "Deploying to $ENVIRONMENT"   # Outputs: "Deploying to production"
```

### Variables with special characters:

```yaml
variables:
  # Strings with spaces need quotes
  GREETING: "Hello World"
  
  # URLs are fine without quotes
  API_URL: https://api.example.com
  
  # Variables with dollar signs need single quotes or escaping
  PRICE: '$$9.99'                        # Use $$ to escape a literal $
  
  # Multi-line variables
  SSH_CONFIG: |
    Host *
      StrictHostKeyChecking no
      UserKnownHostsFile /dev/null
```

## 5. Variables — Defined in GitLab UI (Settings)

For **sensitive data** (passwords, API keys, tokens), **never put them in `.gitlab-ci.yml`**! Use the GitLab UI instead.

### Where to set them:

```
Project-level:  Project → Settings → CI/CD → Variables → "Add variable"
Group-level:    Group → Settings → CI/CD → Variables → "Add variable"
Instance-level: Admin Area → Settings → CI/CD → Variables (admin only)
```

### The variable form:

```
┌──────────────────────────────────────────┐
│          Add CI/CD Variable              │
│                                          │
│  Key:         [DEPLOY_TOKEN           ]  │
│  Value:       [**********************]   │
│  Type:        ○ Variable  ○ File         │
│  Environment: [All (default)         ▼]  │
│                                          │
│  Flags:                                  │
│  ☑ Protect variable                      │
│  ☑ Mask variable                         │
│  ☐ Expand variable reference             │
│                                          │
│           [Add variable]                 │
└──────────────────────────────────────────┘
```

### Using UI variables in your pipeline:

```yaml
# These variables are set in the GitLab UI, not in the YAML file
# They're automatically available as environment variables

deploy:
  script:
    - echo "Deploying..."
    - ./deploy.sh --token $DEPLOY_TOKEN    # ← Set in GitLab UI
    - aws s3 sync ./dist s3://my-bucket    # Uses $AWS_ACCESS_KEY_ID 
                                            #   and $AWS_SECRET_ACCESS_KEY
                                            #   set in GitLab UI
```

## 6. Variable Scopes and Precedence

Variables can be defined at multiple levels. When the same variable name is defined in multiple places, **the most specific one wins**.

### Precedence order (highest to lowest):

```
1. Trigger variables (API, manual pipeline)      ← HIGHEST priority
2. Project-level CI/CD variables (GitLab UI)
3. Group-level CI/CD variables (GitLab UI)
4. Instance-level CI/CD variables (admin)
5. Job-level variables (.gitlab-ci.yml)
6. Global variables (.gitlab-ci.yml)
7. Predefined GitLab variables                    ← LOWEST priority
```

### Example of precedence:

```yaml
# In .gitlab-ci.yml:
variables:
  DATABASE_URL: "postgres://dev:5432/devdb"     # Priority: 6

deploy:
  variables:
    DATABASE_URL: "postgres://prod:5432/proddb"  # Priority: 5 (wins over global)
  script:
    - echo $DATABASE_URL
    # If also set in GitLab UI → UI value wins (priority 2)
    # If also set in trigger → trigger value wins (priority 1)
```

### Environment-scoped variables:

You can restrict UI variables to specific environments:

```
Variable: DATABASE_URL
Value: postgres://prod-host:5432/proddb
Environment: production                    ← Only available in "production" env

Variable: DATABASE_URL
Value: postgres://staging-host:5432/stagingdb
Environment: staging                       ← Only available in "staging" env
```

```yaml
deploy-staging:
  environment:
    name: staging
  script:
    - echo $DATABASE_URL    # Gets staging URL

deploy-production:
  environment:
    name: production
  script:
    - echo $DATABASE_URL    # Gets production URL
```

## 7. Protected and Masked Variables

### Protected Variables

Protected variables are **only available on protected branches and tags**. This prevents someone from creating a random branch and accessing production secrets.

```
                          Protected Variable: PROD_API_KEY
                          
main branch (protected)     → ✅ $PROD_API_KEY available
v1.0.0 tag (protected)     → ✅ $PROD_API_KEY available
feature/login (not protected) → ❌ $PROD_API_KEY is EMPTY
```

### When to use protected variables:

```
✅ Use protected for:
  - Production database credentials
  - Deployment tokens
  - Cloud provider keys (AWS, GCP, Azure)
  - Any production secret

❌ Don't use protected for:
  - Variables needed in all branches (like test API keys)
  - Non-sensitive configuration
```

### Masked Variables

Masked variables have their values **hidden in job logs**. If the value appears in any log output, it's replaced with `[MASKED]`.

```
Without masking:
  $ echo "Token is abc123secret"
  Token is abc123secret          ← SECRET EXPOSED IN LOGS! 😱

With masking:
  $ echo "Token is abc123secret"
  Token is [MASKED]              ← Secret is hidden ✅
```

### Masking requirements:

```
The value MUST:
  - Be at least 8 characters long
  - Not contain newlines
  - Only use characters: a-z, A-Z, 0-9, @, :, ., ~, +, -, _, /, \

If the value doesn't meet these requirements, GitLab won't let you mask it.
```

### Combining protected + masked:

```
For maximum security, use BOTH:
  ☑ Protected  → Only available on protected branches
  ☑ Masked     → Hidden in logs

This is the recommended setting for all production secrets.
```

## 8. Variable Expansion and Nesting

GitLab supports **variable expansion** — using one variable inside another.

### Basic expansion:

```yaml
variables:
  REGISTRY: registry.gitlab.com
  IMAGE_NAME: my-org/my-app
  FULL_IMAGE: $REGISTRY/$IMAGE_NAME       # ← Expands to registry.gitlab.com/my-org/my-app

build:
  script:
    - docker build -t $FULL_IMAGE:$CI_COMMIT_SHORT_SHA .
    # Expands to: docker build -t registry.gitlab.com/my-org/my-app:a1b2c3d4 .
```

### Expansion in different contexts:

```yaml
variables:
  BASE_URL: https://example.com

deploy:
  variables:
    # ✅ Expansion works in variable definitions
    API_URL: $BASE_URL/api/v1
    
  script:
    # ✅ Expansion works in scripts
    - curl $API_URL/health
    
    # ✅ Expansion works inline
    - echo "Deploying to $BASE_URL"
```

### Preventing expansion:

```yaml
variables:
  # To use a literal dollar sign, use $$
  PRICE: '$$9.99'                # Becomes: $9.99
  
  # Or use single quotes in scripts
  PATTERN: '$$HOME'              # Becomes: $HOME (literal)
```

### Nested variable expansion (GitLab 14.3+):

```yaml
variables:
  ENV: staging
  DEPLOY_URL_staging: https://staging.example.com
  DEPLOY_URL_production: https://example.com

deploy:
  script:
    # This WON'T work in YAML (GitLab doesn't support this in YAML)
    # - echo $DEPLOY_URL_$ENV
    
    # But this WILL work in shell:
    - VARNAME="DEPLOY_URL_${ENV}"
    - echo "${!VARNAME}"           # Bash indirect expansion
```

## 9. Variables in `rules` and Conditions

Variables are super useful in `rules` to control when jobs run.

### Using variables in `if` conditions:

```yaml
deploy-production:
  script: ./deploy.sh
  rules:
    # Check branch name
    - if: $CI_COMMIT_BRANCH == "main"
    
    # Check pipeline source
    - if: $CI_PIPELINE_SOURCE == "schedule"
    
    # Check a custom variable
    - if: $DEPLOY_TARGET == "production"
    
    # Check if variable exists (is not empty)
    - if: $CI_COMMIT_TAG
    
    # Check if variable does NOT exist
    - if: $CI_COMMIT_TAG == null
    
    # Combine conditions with &&
    - if: $CI_COMMIT_BRANCH == "main" && $CI_PIPELINE_SOURCE == "push"
    
    # Use || for OR
    - if: $CI_COMMIT_BRANCH == "main" || $CI_COMMIT_BRANCH == "develop"
    
    # Regex matching with =~
    - if: $CI_COMMIT_BRANCH =~ /^release\/.*/
    
    # Negative regex
    - if: $CI_COMMIT_BRANCH !~ /^feature\/.*/
```

### Common patterns:

```yaml
# Pattern 1: Skip CI for certain commits
workflow:
  rules:
    - if: $CI_COMMIT_MESSAGE =~ /\[skip ci\]/
      when: never
    - when: always

# Pattern 2: Different behavior based on custom variable
variables:
  DEPLOY_ENV: staging          # Default

deploy:
  script: ./deploy.sh $DEPLOY_ENV
  rules:
    - if: $DEPLOY_ENV == "production"
      when: manual                         # Manual for production
    - when: on_success                     # Auto for everything else
```

## 10. File-Type Variables

Instead of a simple string value, a variable can contain the **path to a file** with the actual content. This is perfect for certificates, SSH keys, and config files.

### Setting a file variable (in GitLab UI):

```
Key:    KUBECONFIG
Value:  (paste your kubeconfig YAML content)
Type:   ○ Variable  ● File              ← Select "File"!
```

### How file variables work:

```
Regular variable:
  $MY_VAR = "the actual value"

File variable:
  $MY_VAR = "/tmp/gitlab_ci_random123/MY_VAR"   ← Path to a file
  Content of that file = "the actual value"
```

### Using file variables:

```yaml
deploy:
  script:
    # For kubeconfig — kubectl automatically reads $KUBECONFIG
    - kubectl get pods
    # $KUBECONFIG points to a temporary file containing the kubeconfig YAML
    
    # For SSH keys
    - cat $SSH_PRIVATE_KEY > ~/.ssh/id_rsa
    - chmod 600 ~/.ssh/id_rsa
    
    # For any config file
    - cp $APP_CONFIG ./config/app.yml
```

### Why use file variables?

```
✅ Use file variables for:
  - SSH private keys
  - Kubernetes kubeconfig files
  - TLS certificates
  - Any content that tools expect as a FILE

✅ Benefits:
  - Tools that expect file paths (like kubectl) work automatically
  - Content can be multi-line (no issues with newlines)
  - Easier to handle than escaping multi-line strings
```

## 11. Variables — Best Practices

### 1. Never put secrets in `.gitlab-ci.yml`

```yaml
# ❌ TERRIBLE — Secret in source code!
variables:
  API_KEY: "sk-abc123secret456"

# ✅ CORRECT — Use GitLab UI variables
deploy:
  script:
    - curl -H "Authorization: Bearer $API_KEY" https://api.example.com
    # $API_KEY is set in GitLab → Settings → CI/CD → Variables
```

### 2. Use meaningful variable names

```yaml
# ❌ Bad — What do these mean?
variables:
  X: "hello"
  A: "3000"

# ✅ Good — Self-documenting
variables:
  APP_NAME: "hello"
  SERVER_PORT: "3000"
```

### 3. Use variable groups for environments

```yaml
# Define defaults, override per environment
variables:
  REPLICAS: "2"
  LOG_LEVEL: info

deploy-staging:
  variables:
    REPLICAS: "1"
    LOG_LEVEL: debug

deploy-production:
  variables:
    REPLICAS: "5"
    LOG_LEVEL: warn
```

### 4. Always protect and mask production secrets

```
For any production credential:
  ☑ Protected  → Only on protected branches
  ☑ Masked     → Hidden in logs
```

### 5. Document your variables

```yaml
variables:
  # Application settings
  APP_NAME: my-service           # Name used for Docker image and deployment
  APP_PORT: "3000"               # Port the application listens on
  
  # Feature flags
  ENABLE_CACHE: "true"           # Set to "false" to disable Redis caching
  DEBUG_MODE: "false"            # Set to "true" for verbose logging
```

## 12. Artifacts — Overview

**Artifacts** are files created by a job that you want to:
1. **Pass to other jobs** in the same pipeline
2. **Download** from the GitLab UI
3. **Store** for a specific time (like build outputs, test reports)

### How artifacts work:

```
┌──────────────────┐        ┌──────────────────┐
│    BUILD Job     │        │    DEPLOY Job    │
│                  │        │                  │
│  npm run build   │        │  Uses dist/      │
│  Creates dist/   │        │  from build job  │
│                  │        │                  │
│  artifacts:      │  ───►  │  (automatically  │
│    paths:        │        │   downloaded)    │
│      - dist/     │        │                  │
└──────────────────┘        └──────────────────┘
        │
        ▼
   Files uploaded to
   GitLab server
   (downloadable from UI)
```

### Key facts about artifacts:

```
- Artifacts are uploaded to GitLab after the job finishes
- Other jobs in later stages automatically download them
- You can download artifacts from the GitLab UI
- Artifacts have an expiration time (default: 30 days)
- Artifacts are stored on the GitLab server (uses storage)
```

## 13. Basic Artifacts — Passing Files Between Jobs

### Saving artifacts:

```yaml
build:
  stage: build
  script:
    - npm ci
    - npm run build                    # Creates dist/ directory
  artifacts:
    paths:
      - dist/                          # ← Save the dist/ directory
      - build-info.txt                 # ← Save a specific file
```

### How the next job gets them:

```yaml
# By default, ALL artifacts from previous stages are downloaded automatically!

deploy:
  stage: deploy
  script:
    - ls dist/                         # ✅ dist/ is here! (from build job)
    - cat build-info.txt               # ✅ This too!
    - ./deploy.sh
```

### Saving specific files with patterns:

```yaml
build:
  artifacts:
    paths:
      - dist/                          # Entire directory
      - "*.jar"                        # All .jar files in root
      - "target/**/*.war"              # All .war files anywhere in target/
      - reports/coverage.html          # Specific file
```

### Excluding files:

```yaml
build:
  artifacts:
    paths:
      - build/
    exclude:
      - build/**/*.map                 # Don't save source maps
      - build/**/*.tmp                 # Don't save temp files
```

### Naming artifacts:

```yaml
build:
  artifacts:
    name: "build-${CI_COMMIT_REF_SLUG}-${CI_COMMIT_SHORT_SHA}"
    # Downloads as: build-main-a1b2c3d4.zip
    paths:
      - dist/
```

## 14. Artifact Expiration

By default, artifacts expire after **30 days** (configurable by admin). You should set explicit expiration times to save storage.

### Setting expiration:

```yaml
build:
  artifacts:
    paths:
      - dist/
    expire_in: 1 week               # ← Artifacts deleted after 1 week

test:
  artifacts:
    paths:
      - coverage/
    expire_in: 3 days                # ← Artifacts deleted after 3 days

release:
  artifacts:
    paths:
      - release/
    expire_in: never                 # ← Keep forever (use carefully!)
```

### Supported time formats:

```yaml
expire_in: 42                       # 42 seconds
expire_in: 42 seconds
expire_in: 5 mins                   # 5 minutes
expire_in: 3 hours
```

## 16. Controlling Artifact Downloads with `dependencies`

By default, **every job downloads artifacts from ALL previous jobs**. This can be slow if you have many artifacts. Use `dependencies` to control this.

### Download specific artifacts:

```yaml
build-frontend:
  stage: build
  script: npm run build
  artifacts:
    paths: [dist/frontend/]

build-backend:
  stage: build
  script: go build -o backend
  artifacts:
    paths: [backend]

# Only download frontend artifacts (skip backend)
deploy-frontend:
  stage: deploy
  dependencies:
    - build-frontend               # ← Only download from this job
  script:
    - ls dist/frontend/            # ✅ Available
    - ls backend                   # ❌ NOT available

# Download NO artifacts (fastest)
notify:
  stage: deploy
  dependencies: []                 # ← Download nothing!
  script:
    - curl -X POST $SLACK_WEBHOOK
```

### `dependencies` vs `needs`:

```yaml
# "needs" = controls EXECUTION ORDER (when the job starts)
# "dependencies" = controls ARTIFACT DOWNLOAD (which files to download)

# You can use both together:
deploy:
  needs: [build-frontend, test]           # Wait for these jobs
  dependencies: [build-frontend]          # But only download artifacts from this one
  script: ...
```

> **Note:** If you use `needs`, artifacts are only downloaded from the jobs listed in `needs` by default. You don't always need `dependencies` when using `needs`.

## 18. Artifacts — Best Practices

### 1. Always set `expire_in`

```yaml
# ✅ Good — explicit expiration
artifacts:
  paths: [dist/]
  expire_in: 1 week

# ❌ Bad — uses default (30 days), wastes storage
artifacts:
  paths: [dist/]
```

### 2. Only artifact what you need

```yaml
# ❌ Bad — artifacts the entire project!
artifacts:
  paths:
    - .                             # Everything! node_modules and all!

# ✅ Good — only the build output
artifacts:
  paths:
    - dist/
```

### 3. Use `dependencies: []` for jobs that don't need artifacts

```yaml
# ✅ Faster — skips unnecessary downloads
notify-slack:
  dependencies: []
  script: curl $SLACK_WEBHOOK
```

### 4. Use artifact reports for test/coverage data

```yaml
# ✅ GitLab shows results in the MR UI
artifacts:
  reports:
    junit: test-results.xml

# ❌ Just saving the file — GitLab can't parse it
artifacts:
  paths:
    - test-results.xml
```

### 5. Save debugging artifacts `on_failure`

```yaml
artifacts:
  paths: [logs/, screenshots/]
  when: on_failure                  # Only when you need them!
  expire_in: 1 day
```

## 19. Caching — Overview

**Caching** stores files between pipeline runs to **speed up** your jobs. The most common use case is caching dependencies (like `node_modules/` or `.pip-cache/`).

### Cache vs Artifacts — visual comparison:

```
ARTIFACTS — Pass files between JOBS (within one pipeline):

Pipeline #1:
  Build Job ──(artifacts)──► Test Job ──(artifacts)──► Deploy Job
                dist/                    dist/


CACHE — Speed up FUTURE pipelines (across multiple pipelines):

Pipeline #1:
  Install Job → npm install → (saves node_modules/ to cache) ⏱️ 60s

Pipeline #2:
  Install Job → npm install → (restores node_modules/ from cache) ⏱️ 5s!

Pipeline #3:
  Install Job → npm install → (restores node_modules/ from cache) ⏱️ 5s!
```

---

## 20. Cache vs Artifacts — The Key Difference

This is one of the **most confusing** topics in GitLab CI/CD, so let's make it crystal clear:

```
┌──────────────────────────────────────────────────────────────┐
│              CACHE vs ARTIFACTS                               │
├─────────────────────────┬────────────────────────────────────┤
│        CACHE            │          ARTIFACTS                  │
├─────────────────────────┼────────────────────────────────────┤
│ Speeds up pipelines     │ Passes data between jobs           │
│ Across multiple runs    │ Within a single pipeline           │
│ NOT guaranteed          │ GUARANTEED to be there             │
│ Stored on the Runner    │ Stored on GitLab server            │
│ For dependencies        │ For build outputs                  │
│ node_modules/, .cache/  │ dist/, binaries, reports           │
│ Optional (nice to have) │ Required (pipeline depends on it)  │
├─────────────────────────┼────────────────────────────────────┤
│ ⚠️ CACHE CAN DISAPPEAR │ ✅ ARTIFACTS ARE RELIABLE          │
│ at any time!            │ Always available until they expire │
└─────────────────────────┴────────────────────────────────────┘
```

### Golden rule:

```
"Will my pipeline BREAK if this data is missing?"

YES → Use ARTIFACTS (guaranteed)
NO  → Use CACHE (best-effort, speeds things up)
```

### Examples:

```yaml
# ✅ CACHE — node_modules can always be recreated with "npm install"
cache:
  paths:
    - node_modules/              # If cache is missing, npm install will recreate it

# ✅ ARTIFACTS — deploy job NEEDS the built files
build:
  artifacts:
    paths:
      - dist/                    # deploy job will FAIL without dist/

# ❌ WRONG — Using cache for build outputs
build:
  cache:
    paths:
      - dist/                   # WRONG! Cache is not guaranteed!
                                 # Deploy job might not get dist/!
```

## 21. Basic Cache Configuration

### Simple cache example:

```yaml
default:
  cache:
    paths:
      - node_modules/

install:
  stage: build
  script:
    - npm ci                    # First run: installs everything (slow)
                                 # Later runs: uses cache (fast!)

test:
  stage: test
  script:
    - npm test                  # node_modules/ available from cache
```

### Cache with a key:

```yaml
cache:
  key: ${CI_COMMIT_REF_SLUG}    # Different cache per branch
  paths:
    - node_modules/
```

### What happens step by step:

```
Pipeline #1 (first run):
  1. Job starts
  2. Try to restore cache → No cache found (first time)
  3. npm ci → Downloads ALL packages (60 seconds)
  4. Job finishes
  5. Save node_modules/ to cache ← CACHE SAVED

Pipeline #2 (second run):
  1. Job starts
  2. Try to restore cache → ✅ Cache found!
  3. Restore node_modules/ from cache (5 seconds)
  4. npm ci → Checks packages, nothing to download
  5. Job finishes
  6. Save updated node_modules/ to cache ← CACHE UPDATED
```

## 22. Cache Keys — Controlling Cache Invalidation

The **cache key** determines when to reuse or create a new cache. Think of it as the cache's **name tag**.

### Static key (simplest):

```yaml
cache:
  key: my-project-cache          # Same cache for ALL branches, ALL pipelines
  paths:
    - node_modules/
```

### Per-branch key (most common):

```yaml
cache:
  key: ${CI_COMMIT_REF_SLUG}    # Different cache for each branch
  paths:                          # main → one cache
    - node_modules/               # feature/login → different cache
```

### File-based key (smartest):

```yaml
cache:
  key:
    files:
      - package-lock.json        # ← Cache key is based on this file's HASH
  paths:
    - node_modules/
```

How this works:

```
package-lock.json hasn't changed → Same key → Reuse cache ✅ (fast!)
package-lock.json changed        → New key  → Fresh cache (npm install runs fully)
```

### Combined file + prefix key:

```yaml
cache:
  key:
    files:
      - package-lock.json
    prefix: ${CI_JOB_NAME}       # Combine with job name
  paths:
    - node_modules/
# Key becomes: "test-<hash-of-package-lock.json>"
```

### Multiple key examples:

```yaml
# Python project
cache:
  key:
    files:
      - requirements.txt
  paths:
    - .pip-cache/

# Go project
cache:
  key:
    files:
      - go.sum
  paths:
    - .go-cache/

# Java/Maven project
cache:
  key:
    files:
      - pom.xml
  paths:
    - .m2/repository/

# Java/Gradle project
cache:
  key:
    files:
      - gradle.lockfile
  paths:
    - .gradle/caches/
```

## 23. Cache Policy — Push, Pull, Pull-Push

The **cache policy** controls whether a job **reads** from cache, **writes** to cache, or both.

### The three policies:

```yaml
# pull-push (DEFAULT): Download cache before job, upload after job
cache:
  policy: pull-push              # Read + Write
  paths:
    - node_modules/

# pull: Only download cache, don't upload changes
cache:
  policy: pull                   # Read only
  paths:
    - node_modules/

# push: Only upload cache, don't try to download
cache:
  policy: push                   # Write only
  paths:
    - node_modules/
```

### When to use each policy:

```
pull-push (default):
  Use when: Job both uses AND updates the cache
  Example: install job that runs npm ci

pull:
  Use when: Job only READS from cache (faster — skips upload)
  Example: test jobs that just need node_modules

push:
  Use when: Job only CREATES the cache
  Example: Scheduled job that pre-warms the cache
```

### Optimized pipeline with cache policies:

```yaml
# This job creates/updates the cache
install:
  stage: build
  cache:
    key:
      files: [package-lock.json]
    paths:
      - node_modules/
    policy: pull-push              # ← Read existing cache + write updates
  script:
    - npm ci

# These jobs only READ from cache (faster!)
lint:
  stage: test
  cache:
    key:
      files: [package-lock.json]
    paths:
      - node_modules/
    policy: pull                   # ← Only read, don't waste time uploading
  script:
    - npm run lint

test:
  stage: test
  cache:
    key:
      files: [package-lock.json]
    paths:
      - node_modules/
    policy: pull                   # ← Only read
  script:
    - npm test
```

## 24. Cache `paths` vs `untracked`

### `paths` — Cache specific directories:

```yaml
cache:
  paths:
    - node_modules/              # Cache this directory
    - .npm/                      # And this one
```

### `untracked` — Cache all untracked files:

```yaml
cache:
  untracked: true                # Cache ALL files not tracked by Git
```

> **Warning:** `untracked: true` can cache a LOT of files and slow down your pipeline. Prefer explicit `paths` for better control.

### Combining both:

```yaml
cache:
  untracked: true
  paths:
    - extra-dir/                 # Also cache tracked files in this dir
```

## 26. Cache Fallback Keys

If the exact cache key isn't found, GitLab can fall back to an **older cache**. This is great for speeding up new branches.

### Using `fallback_keys`:

```yaml
cache:
  key: ${CI_COMMIT_REF_SLUG}-${CI_COMMIT_SHORT_SHA}
  fallback_keys:
    - ${CI_COMMIT_REF_SLUG}      # Try branch-level cache
    - main                       # Fall back to main branch cache
  paths:
    - node_modules/
```

### How fallback works:

```
Job on branch "feature/login":

1. Look for cache: feature-login-a1b2c3d4     → Not found
2. Look for cache: feature-login               → Not found (new branch)
3. Look for cache: main                        → ✅ Found! Use main's cache
4. After job: Save cache as feature-login-a1b2c3d4
```

This means **new branches start fast** because they inherit the cache from `main`!

## 28. Caching — Best Practices

### 1. Cache dependencies, not build outputs

```yaml
# ✅ Good — dependencies (can be regenerated)
cache:
  paths:
    - node_modules/
    - .pip-cache/

# ❌ Bad — build outputs should be artifacts
cache:
  paths:
    - dist/                      # Use artifacts for this!
```

### 2. Use file-based cache keys

```yaml
# ✅ Best — cache invalidates when lock file changes
cache:
  key:
    files: [package-lock.json]
  paths:
    - node_modules/

# ⚠️ OK — but cache might get stale
cache:
  key: ${CI_COMMIT_REF_SLUG}
  paths:
    - node_modules/
```

### 3. Use `pull` policy for read-only jobs

```yaml
# ✅ Good — test only reads, doesn't need to update cache
test:
  cache:
    policy: pull
    paths: [node_modules/]
  script: npm test
```

### 4. Set appropriate cache directories for each language

```yaml
# Node.js
cache:
  paths:
    - node_modules/
    # Or better: cache the npm cache directory
    - .npm/

# Python
variables:
  PIP_CACHE_DIR: "$CI_PROJECT_DIR/.pip-cache"
cache:
  paths:
    - .pip-cache/

# Go
variables:
  GOPATH: "$CI_PROJECT_DIR/.go"
cache:
  paths:
    - .go/pkg/mod/

# Maven (Java)
cache:
  paths:
    - .m2/repository/

# Gradle (Java)
cache:
  paths:
    - .gradle/caches/
    - .gradle/wrapper/
```

### 5. Don't cache everything

```yaml
# ❌ Bad — too much data, slow to upload/download
cache:
  untracked: true

# ✅ Good — specific directories only
cache:
  paths:
    - node_modules/
```

## 29. Real-World Pipeline — Putting It All Together

Here's a complete, production-ready pipeline that uses variables, artifacts, and caching together:

```yaml
# .gitlab-ci.yml — Full Pipeline with Variables, Artifacts, and Caching

# ═══════════════════════════════════════════════
# VARIABLES
# ═══════════════════════════════════════════════
variables:
  # Application settings
  APP_NAME: my-node-app
  NODE_VERSION: "20"
  
  # Docker settings
  DOCKER_IMAGE: $CI_REGISTRY_IMAGE
  DOCKER_TAG: $CI_COMMIT_SHORT_SHA
  
  # npm cache (for speed)
  NPM_CONFIG_CACHE: "$CI_PROJECT_DIR/.npm-cache"

# ═══════════════════════════════════════════════
# STAGES
# ═══════════════════════════════════════════════
stages:
  - install
  - quality
  - build
  - docker
  - deploy

# ═══════════════════════════════════════════════
# DEFAULT SETTINGS
# ═══════════════════════════════════════════════
default:
  image: node:${NODE_VERSION}-alpine

# ═══════════════════════════════════════════════
# GLOBAL CACHE
# ═══════════════════════════════════════════════
# Cache npm dependencies for all jobs
cache: &global_cache
  key:
    files:
      - package-lock.json
  paths:
    - .npm-cache/
    - node_modules/
  policy: pull                    # Default: jobs only READ the cache

# ═══════════════════════════════════════════════
# JOBS
# ═══════════════════════════════════════════════

# --- Stage: install ---
install-deps:
  stage: install
  cache:
    <<: *global_cache
    policy: pull-push              # This job WRITES the cache
  script:
    - npm ci --cache .npm-cache
  rules:
    - when: always

# --- Stage: quality ---
lint:
  stage: quality
  needs: [install-deps]
  script:
    - npm run lint
  rules:
    - when: always

unit-tests:
  stage: quality
  needs: [install-deps]
  services:
    - name: postgres:15
      alias: db
  variables:
    POSTGRES_DB: test
    POSTGRES_USER: runner
    POSTGRES_PASSWORD: testpass
    DATABASE_URL: "postgresql://runner:testpass@db:5432/test"
  script:
    - npm test -- --reporter junit --output test-results/junit.xml
    - npm run coverage
  artifacts:
    # Reports (displayed in GitLab UI)
    reports:
      junit: test-results/junit.xml
      coverage_report:
        coverage_format: cobertura
        path: coverage/cobertura-coverage.xml
    # Also save coverage HTML for browsing
    paths:
      - coverage/
    expire_in: 1 week
    when: always                  # Save test results even if tests fail!
  coverage: '/All files\s*\|\s*(\d+\.?\d*)\s*\|/'
  rules:
    - when: always

security-scan:
  stage: quality
  needs: [install-deps]
  script:
    - npm audit --production --json > gl-dependency-scanning.json || true
  artifacts:
    reports:
      dependency_scanning: gl-dependency-scanning.json
    expire_in: 1 week
  allow_failure: true             # Don't block pipeline for security warnings
  rules:
    - when: always

# --- Stage: build ---
build-app:
  stage: build
  needs: [lint, unit-tests]
  script:
    - npm run build
    - echo "BUILD_TIME=$(date -u +%Y-%m-%dT%H:%M:%SZ)" > build-info.env
    - echo "BUILD_SHA=$CI_COMMIT_SHORT_SHA" >> build-info.env
  artifacts:
    paths:
      - dist/                     # Build output → needed by deploy
      - build-info.env
    reports:
      dotenv: build-info.env      # Export variables to downstream jobs
    expire_in: 1 week
  rules:
    - when: on_success

# --- Stage: docker ---
build-docker:
  stage: docker
  image: docker:24
  services:
    - docker:24-dind
  needs: [build-app]
  dependencies: [build-app]       # Only download build-app artifacts
  variables:
    DOCKER_TLS_CERTDIR: "/certs"
  cache: []                       # No npm cache needed for Docker build
  before_script:
    - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $CI_REGISTRY
  script:
    - |
      docker build \
        --build-arg BUILD_DATE=$BUILD_TIME \
        --build-arg BUILD_SHA=$BUILD_SHA \
        -t $DOCKER_IMAGE:$DOCKER_TAG \
        -t $DOCKER_IMAGE:latest \
        .
    - docker push $DOCKER_IMAGE:$DOCKER_TAG
    - docker push $DOCKER_IMAGE:latest
  rules:
    - if: $CI_COMMIT_BRANCH == "main"

# --- Stage: deploy ---
deploy-staging:
  stage: deploy
  image: bitnami/kubectl:latest
  needs: [build-docker]
  dependencies: []                # Don't download any artifacts (not needed)
  cache: []                       # No cache needed
  variables:
    KUBE_NAMESPACE: staging
  script:
    - kubectl config use-context staging
    - kubectl set image deployment/$APP_NAME app=$DOCKER_IMAGE:$DOCKER_TAG -n $KUBE_NAMESPACE
    - kubectl rollout status deployment/$APP_NAME -n $KUBE_NAMESPACE --timeout=120s
  environment:
    name: staging
    url: https://staging.myapp.com
  rules:
    - if: $CI_COMMIT_BRANCH == "main"

deploy-production:
  stage: deploy
  image: bitnami/kubectl:latest
  needs: [deploy-staging]         # Must deploy to staging first!
  dependencies: []
  cache: []
  variables:
    KUBE_NAMESPACE: production
  script:
    - kubectl config use-context production
    - kubectl set image deployment/$APP_NAME app=$DOCKER_IMAGE:$DOCKER_TAG -n $KUBE_NAMESPACE
    - kubectl rollout status deployment/$APP_NAME -n $KUBE_NAMESPACE --timeout=180s
  environment:
    name: production
    url: https://myapp.com
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
      when: manual                 # ← Manual approval for production!
  allow_failure: false             # Pipeline doesn't auto-continue
```

### What this pipeline does:

```
1. install-deps → Installs npm packages (updates cache)
2. lint, unit-tests, security-scan → Run in PARALLEL (quality checks)
3. build-app → Builds the app (saves dist/ as artifact)
4. build-docker → Builds and pushes Docker image
5. deploy-staging → Auto-deploys to staging
6. deploy-production → MANUAL deploy to production

Data flow:
  Cache:     npm dependencies (node_modules/) → shared across all pipelines
  Artifact:  dist/ → from build-app to build-docker
  Variables: BUILD_TIME, BUILD_SHA → from build-app to build-docker (via dotenv)
  Secrets:   CI_REGISTRY_PASSWORD, KUBECONFIG → from GitLab UI
```

## 30. Common Mistakes and Troubleshooting

### Mistake 1: Using cache for build outputs

```yaml
# ❌ WRONG — Cache is NOT guaranteed!
build:
  cache:
    paths: [dist/]
  script: npm run build

deploy:
  script: ls dist/              # Might be EMPTY if cache was cleared!

# ✅ CORRECT — Use artifacts for build outputs
build:
  script: npm run build
  artifacts:
    paths: [dist/]

deploy:
  script: ls dist/              # ✅ Always available
```

### Mistake 2: Forgetting `expire_in` for artifacts

```yaml
# ❌ Bad — artifacts pile up using storage
artifacts:
  paths: [dist/]

# ✅ Good — auto-cleaned after 1 week
artifacts:
  paths: [dist/]
  expire_in: 1 week
```

### Mistake 3: Secrets in `.gitlab-ci.yml`

```yaml
# ❌ NEVER DO THIS!
variables:
  DATABASE_PASSWORD: "super_secret_123"

# ✅ Set in GitLab UI with Protected + Masked flags
```

### Mistake 4: Cache key too broad or too narrow

```yaml
# ❌ Too broad — ALL branches share one cache (conflicts!)
cache:
  key: "global"

# ❌ Too narrow — new cache for every commit (never reused!)
cache:
  key: ${CI_COMMIT_SHA}

# ✅ Just right — per branch, invalidated when dependencies change
cache:
  key:
    files: [package-lock.json]
    prefix: ${CI_COMMIT_REF_SLUG}
```

### Mistake 5: Not understanding variable precedence

```yaml
# In .gitlab-ci.yml:
variables:
  API_URL: https://dev.example.com

# In GitLab UI: API_URL = https://prod.example.com

deploy:
  script:
    - echo $API_URL
    # Output: https://prod.example.com  ← UI variable WINS!
```

### Mistake 6: Cache not working between jobs in the same pipeline

```
Remember: Cache is saved AFTER a job completes and restored 
at the START of the next job. But cache sharing between jobs 
in the SAME pipeline is not guaranteed on different runners!

If you need reliable file sharing → Use ARTIFACTS, not cache.
```

### Troubleshooting: "My cache isn't working!"

```
Checklist:
  ☐ Are cache keys the same between jobs? (compare keys)
  ☐ Are jobs running on the same runner? (shared cache is per-runner)
  ☐ Is the cache path correct? (relative to project dir)
  ☐ Has the cache expired or been cleared?
  ☐ Is the runner configured with a cache server? (for distributed cache)
  
Debug: Add this to your job to see what's happening:
  script:
    - echo "Cache key: ${CI_COMMIT_REF_SLUG}"
    - ls -la node_modules/ || echo "node_modules not found"
```

## 31. Summary

### Quick reference card:

```
┌──────────────────────────────────────────────────────────────┐
│         VARIABLES, ARTIFACTS & CACHING — CHEAT SHEET         │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  ═══ VARIABLES ═══                                           │
│  Predefined:    $CI_COMMIT_SHA, $CI_COMMIT_BRANCH, etc.     │
│  Custom YAML:   variables: { KEY: "value" }                  │
│  GitLab UI:     Settings → CI/CD → Variables                 │
│  Protected:     Only on protected branches/tags              │
│  Masked:        Hidden in job logs ([MASKED])                │
│  File type:     Value stored as file, var holds the path    │
│  Precedence:    Trigger > UI > Job-level > Global > Default │
│                                                              │
│  ═══ ARTIFACTS ═══                                           │
│  Purpose:    Pass files between JOBS (reliable!)             │
│  Define:     artifacts: { paths: [dist/] }                   │
│  Expiry:     expire_in: 1 week                               │
│  Reports:    junit, coverage_report, sast, dotenv            │
│  Download:   dependencies: [job-name] or dependencies: []    │
│  On fail:    when: always / on_failure / on_success           │
│                                                              │
│  ═══ CACHE ═══                                               │
│  Purpose:    Speed up FUTURE pipelines (best-effort!)        │
│  Define:     cache: { paths: [node_modules/] }               │
│  Key:        key: ${CI_COMMIT_REF_SLUG}                      │
│  Smart key:  key: { files: [package-lock.json] }             │
│  Policy:     pull-push (default) / pull / push               │
│  Fallback:   fallback_keys: [main]                           │
│  Clear:      Change key version or UI button                 │
│                                                              │
│  ═══ WHEN TO USE WHAT ═══                                    │
│  Need config/secrets?          → Variables                   │
│  Need to pass files to jobs?   → Artifacts                   │
│  Need to speed up installs?    → Cache                       │
│  Pipeline breaks without it?   → Artifacts (guaranteed)      │
│  Just a nice speedup?          → Cache (best-effort)         │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```
