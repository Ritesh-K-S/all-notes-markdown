## 1. Introduction

Alright, so you've already learned how to write `.gitlab-ci.yml` files, set up stages, jobs, variables, artifacts, and caching. That's awesome! But what if I told you that GitLab can **automatically** create an entire CI/CD pipeline for you without writing a single line of pipeline code? And what if your teammates could see a **live preview** of every merge request before merging?

That's exactly what **Auto DevOps** and **Review Apps** do. Let's break them down.

---

## 2. What is Auto DevOps?

**Auto DevOps** is GitLab's built-in, batteries-included CI/CD pipeline that automatically detects your project type, builds it, tests it, scans it for security issues, and deploys it — all without you writing a `.gitlab-ci.yml` file.

Think of it like this:

> **Manual CI/CD** = You cook a meal from scratch (write every stage yourself)
> **Auto DevOps** = You use a meal kit (GitLab gives you the recipe and ingredients, you just turn it on)

### What does Auto DevOps give you for free?

| Feature | What It Does |
|---------|-------------|
| Auto Build | Detects language and builds a Docker image |
| Auto Test | Runs tests automatically |
| Auto Code Quality | Checks your code quality |
| Auto SAST | Static security analysis |
| Auto DAST | Dynamic security analysis |
| Auto Dependency Scanning | Checks for vulnerable dependencies |
| Auto Container Scanning | Scans your Docker image for CVEs |
| Auto License Compliance | Checks license compatibility |
| Auto Deploy | Deploys to Kubernetes |
| Auto Monitoring | Sets up Prometheus monitoring |

That's 10 stages — all automated. Crazy, right?

---

## 3. How Auto DevOps Works — The Big Picture

Here's the flow in simple terms:

```
You push code
    ↓
GitLab detects the language (Ruby, Python, Node.js, Java, Go, etc.)
    ↓
It uses Heroku Buildpacks or Cloud Native Buildpacks to build a Docker image
    ↓
It runs tests
    ↓
It scans for security vulnerabilities
    ↓
It deploys to a Kubernetes cluster (if configured)
    ↓
It sets up monitoring (if Prometheus is connected)
```

Under the hood, Auto DevOps uses a **predefined `.gitlab-ci.yml` template** maintained by GitLab. You don't see it in your repo, but it's there behind the scenes.

The template lives at:
```
https://gitlab.com/gitlab-org/gitlab/-/blob/master/lib/gitlab/ci/templates/Auto-DevOps.gitlab-ci.yml
```

## 4. Enabling Auto DevOps

There are **three ways** to enable Auto DevOps:

### Method 1: Enable for a Single Project

1. Go to your GitLab project
2. Navigate to **Settings → CI/CD**
3. Expand the **Auto DevOps** section
4. Check **"Default to Auto DevOps pipeline"**
5. Optionally set the **deployment strategy** and **base domain**
6. Click **Save changes**

### Method 2: Enable at the Group Level

1. Go to your Group
2. Navigate to **Settings → CI/CD**
3. Expand **Auto DevOps**
4. Enable it

> This enables Auto DevOps for **all projects** in that group (unless a project has its own `.gitlab-ci.yml`).

### Method 3: Enable at the Instance Level (Self-Managed GitLab)

1. Go to **Admin Area → Settings → CI/CD**
2. Expand **Auto DevOps**
3. Check **"Default to Auto DevOps pipeline"**
4. Set the **Auto DevOps domain** (e.g., `auto-devops.example.com`)

### Important Rule:

> If your project already has a `.gitlab-ci.yml` file, Auto DevOps will **NOT** run. Your custom pipeline takes priority.

## 5. Auto DevOps Pipeline Stages — Deep Dive

Let's go through each stage that Auto DevOps creates for you:

### 5.1 Auto Build

**What it does:** Detects your app's language and creates a Docker image.

**How it works:**
- It uses **Cloud Native Buildpacks** (or Heroku Buildpacks as fallback)
- Detects the language by looking at files like `package.json` (Node.js), `requirements.txt` (Python), `Gemfile` (Ruby), `pom.xml` (Java), `go.mod` (Go), etc.
- Builds the image and pushes it to GitLab's built-in **Container Registry**

**If you want to use your own Dockerfile instead:**

```yaml
# Set this CI/CD variable in your project settings:
# AUTO_DEVOPS_BUILD_IMAGE_FORWARDED_CI_VARIABLES = ""
# Or simply add a Dockerfile to your repo root — Auto DevOps will prefer it!
```

When Auto DevOps finds a `Dockerfile` in your project root, it uses `docker build` instead of buildpacks.

### 5.2 Auto Test

**What it does:** Runs your project's test suite automatically.

**How it works:**
- For **Heroku Buildpacks**, it runs the standard test command for your language
- For example:
  - Node.js → `npm test`
  - Python → `python setup.py test` or `pytest`
  - Ruby → `rake test`
  - Java → `mvn test` or `gradle test`

**What if you have no tests?** The stage will simply pass (it won't fail).

### 5.3 Auto Code Quality

**What it does:** Analyzes your code quality using **Code Climate**.

**What it reports:**
- Code complexity
- Duplicated code
- Maintainability issues
- The report shows up directly in the **Merge Request widget**

### 5.4 Auto SAST

**What it does:** Runs **Static Application Security Testing** — scans your source code for security vulnerabilities without running it.

**Tools used (depends on language):**
- `semgrep` (multiple languages)
- `brakeman` (Ruby)
- `bandit` (Python)
- `gosec` (Go)
- `spotbugs` (Java)
- `eslint-plugin-security` (JavaScript)

**Results** show up in the **Security Dashboard** and **Merge Request widget**.

### 5.5 Auto DAST

**What it does:** Runs **Dynamic Application Security Testing** — actually hits your running application with security tests.

- Uses **OWASP ZAP** to scan your deployed application
- Finds vulnerabilities like XSS, SQL Injection, CSRF, etc.
- Requires a **running environment** (works best with Review Apps!)

### 5.6 Auto Dependency Scanning

**What it does:** Checks your project dependencies for known vulnerabilities.

- Scans `package-lock.json`, `Gemfile.lock`, `requirements.txt`, `pom.xml`, etc.
- Cross-references with vulnerability databases (NVD, GitLab Advisory Database)

### 5.7 Auto Container Scanning

**What it does:** Scans the Docker image that Auto Build created for known CVEs (Common Vulnerabilities and Exposures).

- Uses **Trivy** (or previously **Clair**) to scan the image layers
- Reports vulnerabilities in the base image and installed packages

### 5.8 Auto License Compliance

**What it does:** Checks the licenses of your dependencies to make sure they're compatible with your project.

- Detects licenses like MIT, Apache 2.0, GPL, LGPL, etc.
- You can set **allowed** and **denied** licenses in project settings

### 5.9 Auto Deploy

**What it does:** Deploys your application to a **Kubernetes cluster**.

**How it works:**
- Uses **Helm** charts to deploy
- Creates Kubernetes **Deployments**, **Services**, and **Ingress** resources
- Supports multiple environments: `staging`, `production`, `review`

**Deployment strategies:**
- **Continuous deployment to production** — every push to default branch goes live
- **Continuous deployment with manual step** — auto-deploys to staging, manual approval for production
- **Timed incremental rollout** — gradually shifts traffic (canary-style)

### 5.10 Auto Monitoring

**What it does:** Sets up **Prometheus monitoring** for your deployed app.

- Monitors response metrics (latency, error rate, throughput)
- Integrates with GitLab's built-in monitoring dashboard
- Requires Prometheus to be installed on the Kubernetes cluster

## 6. Customizing Auto DevOps

Auto DevOps is great out of the box, but you'll often need to tweak it. Here's how:

### Method 1: Using CI/CD Variables

You can control Auto DevOps behavior by setting CI/CD variables in **Settings → CI/CD → Variables**:

| Variable | What It Does | Example Value |
|----------|-------------|---------------|
| `AUTO_DEVOPS_DOMAIN` | Base domain for deployments | `apps.example.com` |
| `POSTGRES_ENABLED` | Enable/disable auto PostgreSQL | `false` |
| `STAGING_ENABLED` | Enable staging environment | `1` |
| `CANARY_ENABLED` | Enable canary deployments | `1` |
| `INCREMENTAL_ROLLOUT_MODE` | Set rollout mode | `timed` or `manual` |
| `TEST_DISABLED` | Skip auto tests | `1` |
| `CODE_QUALITY_DISABLED` | Skip code quality scan | `1` |
| `SAST_DISABLED` | Skip SAST | `1` |
| `DAST_DISABLED` | Skip DAST | `1` |
| `CONTAINER_SCANNING_DISABLED` | Skip container scanning | `1` |
| `DEPENDENCY_SCANNING_DISABLED` | Skip dependency scanning | `1` |
| `LICENSE_MANAGEMENT_DISABLED` | Skip license check | `1` |
| `K8S_SECRET_*` | Pass secrets to Kubernetes | `K8S_SECRET_DB_PASSWORD=xxx` |

### Method 2: Overriding with a `.gitlab-ci.yml`

You can include the Auto DevOps template and override specific jobs:

```yaml
# .gitlab-ci.yml

# Include the Auto DevOps template
include:
  - template: Auto-DevOps.gitlab-ci.yml

# Override the test job
test:
  script:
    - echo "Running my custom tests instead!"
    - npm run test:custom

# Override the build job variables
variables:
  AUTO_DEVOPS_BUILD_IMAGE_CNB_ENABLED: "false"
  DOCKERFILE_PATH: "docker/Dockerfile.production"
```

This gives you the best of both worlds — Auto DevOps handles the heavy lifting, and you customize only what you need.

### Method 3: Custom Helm Chart

By default, Auto DevOps uses GitLab's built-in Helm chart for deployment. But you can provide your own:

```
# Place your chart in your repo:
chart/
  Chart.yaml
  values.yaml
  templates/
    deployment.yaml
    service.yaml
    ingress.yaml
```

Auto DevOps will detect the `chart/` directory and use your custom Helm chart instead.

## 7. Auto DevOps with Kubernetes

For Auto DevOps to deploy, you need a **Kubernetes cluster connected to GitLab**.

### Connecting a Cluster

**Option 1: GitLab Kubernetes Agent (Recommended)**

```yaml
# .gitlab/agents/my-agent/config.yaml
ci_access:
  projects:
    - id: your-project-path
```

**Option 2: Certificate-Based Connection (Deprecated but still works)**

1. Go to **Infrastructure → Kubernetes clusters**
2. Click **Connect a cluster**
3. Provide the API URL, CA certificate, and service account token

### What Gets Deployed?

When Auto DevOps deploys to Kubernetes, it creates:

```yaml
# Deployment
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 1       # Configurable via REPLICAS variable
  template:
    spec:
      containers:
      - name: auto-deploy-app
        image: registry.gitlab.com/your-group/your-project:tag
        ports:
        - containerPort: 5000   # Default port
        env:
        - name: DATABASE_URL
          value: "..."   # Auto-provisioned if POSTGRES_ENABLED=true

---
# Service
apiVersion: v1
kind: Service
metadata:
  name: my-app
spec:
  type: ClusterIP
  ports:
  - port: 5000

---
# Ingress
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-app
spec:
  rules:
  - host: my-app.apps.example.com   # Uses AUTO_DEVOPS_DOMAIN
    http:
      paths:
      - path: /
        backend:
          service:
            name: my-app
            port:
              number: 5000
```

### Setting the Deployment Domain

Set the `AUTO_DEVOPS_DOMAIN` variable. For example, if your domain is `apps.example.com`:
- Production will be at: `my-project.apps.example.com`
- Staging will be at: `my-project-staging.apps.example.com`
- Review apps will be at: `my-project-review-branch-name.apps.example.com`

## 8. What are Review Apps?

Okay, here's one of the **coolest features** in GitLab — **Review Apps**.

Imagine this scenario:
1. A developer creates a merge request (MR) with a new feature
2. A reviewer opens the MR
3. Instead of checking out the branch locally and running it... there's a **live link** right in the MR
4. The reviewer clicks the link and sees the **actual running application** with the new changes
5. When the MR is merged or closed, the app is **automatically destroyed**

That's a Review App!

> **Review App** = A temporary, live deployment of a specific branch/merge request that lets anyone see and test the changes in a real environment.

### Why are Review Apps amazing?

| Benefit | Description |
|---------|-------------|
| **Visual review** | Reviewers can see UI changes live, not just read code |
| **QA testing** | QA team can test features before merge |
| **Stakeholder demos** | PMs/designers can preview features |
| **Isolated testing** | Each MR gets its own environment |
| **Automatic cleanup** | Environments are destroyed when MR is closed |
| **Faster feedback** | No need to deploy to staging and wait |

---

## 9. How Review Apps Work

Here's the flow:

```
Developer creates a branch → Pushes code
    ↓
CI/CD pipeline runs → Build stage creates a Docker image
    ↓
Deploy stage creates a NEW environment just for this branch
    ↓
The app is deployed to that environment (e.g., Kubernetes namespace, Docker, VM)
    ↓
A unique URL is generated (e.g., review-feature-xyz.apps.example.com)
    ↓
The URL shows up in the Merge Request
    ↓
Reviewer clicks the link → sees the live app
    ↓
When MR is merged/closed → the environment is automatically STOPPED and DESTROYED
```

The magic ingredients:
- **Dynamic environments** — each branch gets a unique environment name
- **`environment` keyword** in `.gitlab-ci.yml` — tells GitLab this job creates an environment
- **`on_stop` action** — defines how to destroy the environment when done

## 10. Setting Up Review Apps — Step by Step

Let's build a Review App setup from scratch. We'll start simple and add complexity.

### Basic Review App Setup

```yaml
# .gitlab-ci.yml

stages:
  - build
  - review
  - cleanup

variables:
  APP_NAME: my-app

# Build the Docker image
build:
  stage: build
  image: docker:24
  services:
    - docker:24-dind
  script:
    - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $CI_REGISTRY
    - docker build -t $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA .
    - docker push $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA
  only:
    - branches

# Deploy a Review App for every branch (except main)
review:
  stage: review
  script:
    - echo "Deploying review app for branch $CI_COMMIT_REF_SLUG..."
    - echo "App will be available at https://${CI_COMMIT_REF_SLUG}.apps.example.com"
    # Your actual deployment command goes here
    # e.g., kubectl apply, docker run, helm install, etc.
  environment:
    name: review/$CI_COMMIT_REF_SLUG          # Dynamic environment name
    url: https://${CI_COMMIT_REF_SLUG}.apps.example.com   # Dynamic URL
    on_stop: stop_review                       # What to run when stopping
    auto_stop_in: 1 week                       # Automatically stop after 1 week
  only:
    - branches
  except:
    - main

# Stop/destroy the Review App
stop_review:
  stage: cleanup
  script:
    - echo "Stopping review app for branch $CI_COMMIT_REF_SLUG..."
    # Your actual teardown command goes here
    # e.g., kubectl delete, docker rm, helm uninstall, etc.
  environment:
    name: review/$CI_COMMIT_REF_SLUG
    action: stop                               # This marks the environment as "stopped"
  when: manual                                 # Can be triggered manually
  only:
    - branches
  except:
    - main
```

Let me break this down:

### The `environment` Keyword

```yaml
environment:
  name: review/$CI_COMMIT_REF_SLUG     # 👈 Dynamic name using the branch slug
  url: https://${CI_COMMIT_REF_SLUG}.apps.example.com  # 👈 The live URL
  on_stop: stop_review                  # 👈 Which job to run when stopping
  auto_stop_in: 1 week                  # 👈 Auto-cleanup timer
```

- `name: review/$CI_COMMIT_REF_SLUG` → Creates a unique environment like `review/feature-login`, `review/fix-bug-123`, etc.
- `$CI_COMMIT_REF_SLUG` → A URL-safe version of your branch name (e.g., `feature/login-page` becomes `feature-login-page`)
- `on_stop: stop_review` → Points to the job that cleans up this environment
- `auto_stop_in: 1 week` → If no one stops it manually, GitLab will auto-stop it after 1 week

### The Stop Job

```yaml
stop_review:
  environment:
    name: review/$CI_COMMIT_REF_SLUG   # Must match the deploy job's environment name
    action: stop                        # 👈 This is the key — tells GitLab this STOPS the env
  when: manual                          # Triggered manually OR automatically when MR is merged/closed
```

**Important:** The `action: stop` tells GitLab "this job destroys the environment." GitLab will automatically trigger this job when:
- The merge request is **merged**
- The merge request is **closed**
- The `auto_stop_in` timer expires

## 11. Review Apps with Kubernetes

This is the most common setup in production environments. Each review app gets its own **Kubernetes namespace** (or at least its own set of resources).

```yaml
# .gitlab-ci.yml

stages:
  - build
  - review
  - cleanup

variables:
  KUBE_NAMESPACE: "review-${CI_COMMIT_REF_SLUG}"

# Build
build:
  stage: build
  image: docker:24
  services:
    - docker:24-dind
  script:
    - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $CI_REGISTRY
    - docker build -t $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA .
    - docker push $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA

# Deploy Review App to Kubernetes
deploy_review:
  stage: review
  image: bitnami/kubectl:latest
  script:
    # Create a namespace for this review app
    - kubectl create namespace $KUBE_NAMESPACE --dry-run=client -o yaml | kubectl apply -f -

    # Deploy the app
    - |
      cat <<EOF | kubectl apply -n $KUBE_NAMESPACE -f -
      apiVersion: apps/v1
      kind: Deployment
      metadata:
        name: review-app
      spec:
        replicas: 1
        selector:
          matchLabels:
            app: review-app
        template:
          metadata:
            labels:
              app: review-app
          spec:
            containers:
            - name: app
              image: $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA
              ports:
              - containerPort: 3000
              env:
              - name: DATABASE_URL
                value: "postgres://review:password@db:5432/reviewdb"
      ---
      apiVersion: v1
      kind: Service
      metadata:
        name: review-app
      spec:
        selector:
          app: review-app
        ports:
        - port: 80
          targetPort: 3000
      ---
      apiVersion: networking.k8s.io/v1
      kind: Ingress
      metadata:
        name: review-app
        annotations:
          nginx.ingress.kubernetes.io/rewrite-target: /
      spec:
        ingressClassName: nginx
        rules:
        - host: ${CI_COMMIT_REF_SLUG}.review.example.com
          http:
            paths:
            - path: /
              pathType: Prefix
              backend:
                service:
                  name: review-app
                  port:
                    number: 80
      EOF

    - echo "Review app deployed at https://${CI_COMMIT_REF_SLUG}.review.example.com"
  environment:
    name: review/$CI_COMMIT_REF_SLUG
    url: https://${CI_COMMIT_REF_SLUG}.review.example.com
    on_stop: stop_review
    auto_stop_in: 3 days
  rules:
    - if: '$CI_MERGE_REQUEST_IID'   # Only run for merge requests

# Cleanup
stop_review:
  stage: cleanup
  image: bitnami/kubectl:latest
  script:
    - kubectl delete namespace $KUBE_NAMESPACE --ignore-not-found
    - echo "Review app environment deleted."
  environment:
    name: review/$CI_COMMIT_REF_SLUG
    action: stop
  when: manual
  rules:
    - if: '$CI_MERGE_REQUEST_IID'
      when: manual
```

**What happens here:**
1. Each MR gets its own Kubernetes namespace: `review-feature-login`, `review-fix-bug-42`, etc.
2. A Deployment, Service, and Ingress are created inside that namespace
3. Wildcard DNS (`*.review.example.com`) routes traffic to the right app
4. When the MR is merged/closed, the entire namespace is deleted — clean and simple!

### Setting Up Wildcard DNS

For Review Apps with unique URLs, you need **wildcard DNS**:

```
*.review.example.com  →  A record  →  <your-ingress-controller-IP>
```

This way, `feature-login.review.example.com` and `fix-bug-42.review.example.com` all route to your Kubernetes Ingress Controller, which then routes to the correct review app based on the `host` rule.

## 12. Review Apps with Docker Compose

Not everyone uses Kubernetes. Here's how to do Review Apps with a simple server and Docker Compose:

```yaml
# .gitlab-ci.yml

stages:
  - build
  - review
  - cleanup

deploy_review:
  stage: review
  tags:
    - shell   # Runs on a shell runner (not Docker runner)
  script:
    # Create a unique project directory
    - export APP_DIR="/opt/review-apps/${CI_COMMIT_REF_SLUG}"
    - mkdir -p $APP_DIR

    # Generate a docker-compose file for this review app
    - |
      cat > $APP_DIR/docker-compose.yml <<EOF
      version: '3.8'
      services:
        app:
          image: $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA
          ports:
            - "${REVIEW_PORT}:3000"
          environment:
            - NODE_ENV=review
            - APP_URL=https://${CI_COMMIT_REF_SLUG}.review.example.com
          labels:
            - "traefik.enable=true"
            - "traefik.http.routers.${CI_COMMIT_REF_SLUG}.rule=Host(\`${CI_COMMIT_REF_SLUG}.review.example.com\`)"
            - "traefik.http.services.${CI_COMMIT_REF_SLUG}.loadbalancer.server.port=3000"
      EOF

    # Deploy using Docker Compose
    - cd $APP_DIR
    - docker compose pull
    - docker compose up -d
  environment:
    name: review/$CI_COMMIT_REF_SLUG
    url: https://${CI_COMMIT_REF_SLUG}.review.example.com
    on_stop: stop_review
    auto_stop_in: 5 days
  rules:
    - if: '$CI_MERGE_REQUEST_IID'

stop_review:
  stage: cleanup
  tags:
    - shell
  script:
    - export APP_DIR="/opt/review-apps/${CI_COMMIT_REF_SLUG}"
    - cd $APP_DIR && docker compose down --volumes --remove-orphans || true
    - rm -rf $APP_DIR
  environment:
    name: review/$CI_COMMIT_REF_SLUG
    action: stop
  when: manual
  rules:
    - if: '$CI_MERGE_REQUEST_IID'
      when: manual
```

This uses **Traefik** as a reverse proxy that automatically routes traffic based on Docker labels. Each review app gets a unique hostname.

## 13. Review Apps — Dynamic Environments

Let's understand **dynamic environments** more deeply because they're the backbone of Review Apps.

### GitLab's Predefined CI Variables

These variables are super useful for dynamic environments:

| Variable | Example Value | Description |
|----------|--------------|-------------|
| `$CI_COMMIT_REF_SLUG` | `feature-login-page` | URL-safe branch name |
| `$CI_COMMIT_REF_NAME` | `feature/login-page` | Original branch name |
| `$CI_COMMIT_SHORT_SHA` | `a1b2c3d4` | Short commit SHA |
| `$CI_MERGE_REQUEST_IID` | `42` | MR number |
| `$CI_PROJECT_NAME` | `my-app` | Project name |
| `$CI_ENVIRONMENT_SLUG` | `review-feature-lo-abc123` | URL-safe environment name |

### Environment Naming Patterns

```yaml
# Pattern 1: Using branch slug
environment:
  name: review/$CI_COMMIT_REF_SLUG
  # Creates: review/feature-login, review/fix-bug-42, etc.

# Pattern 2: Using MR number
environment:
  name: review/mr-$CI_MERGE_REQUEST_IID

  # Creates: review/my-app/feature-login, etc.
```

## 14. Cleaning Up Review App Environments

Review Apps are temporary by nature. You don't want hundreds of abandoned environments eating up resources. Here's how to manage cleanup:

### Method 1: `auto_stop_in`

```yaml
environment:
  name: review/$CI_COMMIT_REF_SLUG
  auto_stop_in: 3 days   # Auto-stops 3 days after last deployment
```

Supported formats:
- `1 hour`, `6 hours`
- `1 day`, `3 days`, `1 week`
- `30 minutes`

### Method 2: `on_stop` with Automatic Trigger

```yaml
# The deploy job
deploy_review:
  environment:
    name: review/$CI_COMMIT_REF_SLUG
    on_stop: stop_review   # GitLab runs this when MR is merged/closed

# The stop job
stop_review:
  environment:
    name: review/$CI_COMMIT_REF_SLUG
    action: stop
  when: manual
```

When the MR is merged or closed, GitLab automatically triggers the `stop_review` job.

### Method 3: Scheduled Cleanup Pipeline

Create a scheduled pipeline that cleans up old environments:

```yaml
# .gitlab-ci.yml

cleanup_old_environments:
  stage: cleanup
  image: curlimages/curl:latest
  script:
    # Use GitLab API to find and stop old environments
    - |
      # Get all active review environments
      ENVS=$(curl --silent --header "PRIVATE-TOKEN: $GITLAB_API_TOKEN" \
        "$CI_API_V4_URL/projects/$CI_PROJECT_ID/environments?states=available&search=review" \
        | jq -r '.[] | select(.updated_at < (now - 604800 | todate)) | .id')
      
      # Stop each old environment
      for ENV_ID in $ENVS; do
        echo "Stopping environment $ENV_ID"
        curl --request POST --header "PRIVATE-TOKEN: $GITLAB_API_TOKEN" \
          "$CI_API_V4_URL/projects/$CI_PROJECT_ID/environments/$ENV_ID/stop"
      done
  rules:
    - if: '$CI_PIPELINE_SOURCE == "schedule"'   # Only runs on schedule
```

Set this up as a **scheduled pipeline** (e.g., daily at midnight) in **Build → Pipeline schedules**.

## 16. Auto DevOps + Review Apps Together

When Auto DevOps is enabled and you have a Kubernetes cluster connected, Review Apps work **automatically**. You don't need to configure anything extra!

Here's what happens:

1. Developer creates a merge request
2. Auto DevOps builds the Docker image
3. Auto DevOps **automatically deploys a review app** to Kubernetes
4. A unique URL appears in the MR
5. DAST runs against the review app URL
6. When MR is merged → review app is destroyed
7. Auto DevOps deploys to staging (if `STAGING_ENABLED=1`)
8. Manual approval → Auto DevOps deploys to production

```
MR Created → Build → Test → Security Scans → Review App (live!) → DAST on Review App
                                                                      ↓
MR Merged → Review App destroyed → Staging deploy → Manual gate → Production deploy
```

### To enable this flow:

```
# CI/CD Variables to set:
AUTO_DEVOPS_DOMAIN = apps.example.com
STAGING_ENABLED = 1
KUBE_INGRESS_BASE_DOMAIN = apps.example.com
```

That's it. Auto DevOps handles everything else.

## 17. Real-World Example — Full Setup

Let's put it all together with a complete, production-ready setup:

```yaml
# .gitlab-ci.yml — Complete Review Apps + Multi-Environment Pipeline

stages:
  - build
  - test
  - review
  - staging
  - production
  - cleanup

variables:
  DOCKER_IMAGE: $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA

# ========== BUILD ==========
build:
  stage: build
  image: docker:24
  services:
    - docker:24-dind
  script:
    - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $CI_REGISTRY
    - docker build
        --build-arg BUILD_DATE=$(date -u +%Y-%m-%dT%H:%M:%SZ)
        --build-arg VCS_REF=$CI_COMMIT_SHORT_SHA
        -t $DOCKER_IMAGE .
    - docker push $DOCKER_IMAGE

# ========== TEST ==========
test:
  stage: test
  image: $DOCKER_IMAGE
  script:
    - npm test
  coverage: '/Lines\s*:\s*(\d+\.?\d*)%/'

# ========== REVIEW APP ==========
deploy_review:
  stage: review
  image:
    name: bitnami/kubectl:latest
    entrypoint: ['']
  before_script:
    - kubectl config use-context $KUBE_CONTEXT
  script:
    - export KUBE_NAMESPACE="review-${CI_COMMIT_REF_SLUG}"
    - kubectl create namespace $KUBE_NAMESPACE --dry-run=client -o yaml | kubectl apply -f -

    # Deploy using a Helm chart
    - |
      helm upgrade --install review-$CI_COMMIT_REF_SLUG ./chart \
        --namespace $KUBE_NAMESPACE \
        --set image.repository=$CI_REGISTRY_IMAGE \
        --set image.tag=$CI_COMMIT_SHORT_SHA \
        --set ingress.host=${CI_COMMIT_REF_SLUG}.review.example.com \
        --set replicas=1 \
        --set resources.requests.memory=128Mi \
        --set resources.requests.cpu=100m \
        --wait --timeout 300s
  environment:
    name: review/$CI_COMMIT_REF_SLUG
    url: https://${CI_COMMIT_REF_SLUG}.review.example.com
    on_stop: stop_review
    auto_stop_in: 5 days
  rules:
    - if: '$CI_MERGE_REQUEST_IID'

stop_review:
  stage: cleanup
  image:
    name: bitnami/kubectl:latest
    entrypoint: ['']
  before_script:
    - kubectl config use-context $KUBE_CONTEXT
  script:
    - export KUBE_NAMESPACE="review-${CI_COMMIT_REF_SLUG}"
    - helm uninstall review-$CI_COMMIT_REF_SLUG --namespace $KUBE_NAMESPACE || true
    - kubectl delete namespace $KUBE_NAMESPACE --ignore-not-found
  environment:
    name: review/$CI_COMMIT_REF_SLUG
    action: stop
  rules:
    - if: '$CI_MERGE_REQUEST_IID'
      when: manual

# ========== STAGING ==========
deploy_staging:
  stage: staging
  image:
    name: bitnami/kubectl:latest
    entrypoint: ['']
  script:
    - |
      helm upgrade --install my-app-staging ./chart \
        --namespace staging \
        --set image.repository=$CI_REGISTRY_IMAGE \
        --set image.tag=$CI_COMMIT_SHORT_SHA \
        --set ingress.host=staging.example.com \
        --set replicas=2 \
        --wait --timeout 300s
  environment:
    name: staging
    url: https://staging.example.com
  rules:
    - if: '$CI_COMMIT_BRANCH == "main"'

# ========== PRODUCTION ==========
deploy_production:
  stage: production
  image:
    name: bitnami/kubectl:latest
    entrypoint: ['']
  script:
    - |
      helm upgrade --install my-app-production ./chart \
        --namespace production \
        --set image.repository=$CI_REGISTRY_IMAGE \
        --set image.tag=$CI_COMMIT_SHORT_SHA \
        --set ingress.host=www.example.com \
        --set replicas=3 \
        --set resources.requests.memory=256Mi \
        --set resources.requests.cpu=250m \
        --wait --timeout 300s
  environment:
    name: production
    url: https://www.example.com
  rules:
    - if: '$CI_COMMIT_BRANCH == "main"'
      when: manual   # 👈 Requires manual approval
```

## 18. Troubleshooting Common Issues

### Issue 1: Review App URL returns 404 or connection refused

**Cause:** Wildcard DNS not configured, or Ingress not working.

**Fix:**
```bash
# Check if wildcard DNS resolves
dig *.review.example.com

# Check if Ingress resource exists
kubectl get ingress -n review-feature-login

# Check if the pod is running
kubectl get pods -n review-feature-login

# Check pod logs
kubectl logs -n review-feature-login deployment/review-app
```

### Issue 2: Stop job doesn't run when MR is merged

**Cause:** The stop job must have the same `rules` or `only/except` as the deploy job.

**Fix:**
```yaml
# ❌ Wrong — different rules
deploy_review:
  rules:
    - if: '$CI_MERGE_REQUEST_IID'

stop_review:
  only:
    - branches    # This doesn't match!

# ✅ Correct — same rules
deploy_review:
  rules:
    - if: '$CI_MERGE_REQUEST_IID'

stop_review:
  rules:
    - if: '$CI_MERGE_REQUEST_IID'
      when: manual
```

### Issue 3: Too many review app environments eating cluster resources

**Fix:**
```yaml
# Add resource limits
environment:
  auto_stop_in: 2 days   # Short auto-stop timer

# Set tight resource limits in your Kubernetes deployment
resources:
  requests:
    memory: "64Mi"
    cpu: "50m"
  limits:
    memory: "128Mi"
    cpu: "100m"
```

### Issue 4: Auto DevOps can't detect the language

**Fix:**
Make sure you have the appropriate file in your repo root:
- Node.js: `package.json`
- Python: `requirements.txt` or `Pipfile`
- Ruby: `Gemfile`
- Java: `pom.xml` or `build.gradle`
- Go: `go.mod`
- PHP: `composer.json`

### Issue 5: Auto DevOps build fails with "no buildpack"

**Fix:**
Either add a `Dockerfile` to your repo root, or set the buildpack explicitly:
```
# CI/CD Variable:
BUILDPACK_URL = https://github.com/heroku/heroku-buildpack-nodejs
```

## 19. Best Practices

### For Auto DevOps:

1. **Start with Auto DevOps, then customize** — Don't write everything from scratch. Let Auto DevOps do the heavy lifting, then override what you need.

2. **Always connect a Kubernetes cluster** — Auto DevOps is 10x more useful with Kubernetes for auto-deploy.

3. **Set `AUTO_DEVOPS_DOMAIN`** — Without it, deploy stages are skipped.

4. **Disable stages you don't need** — Use `*_DISABLED` variables to skip unnecessary scans in early stages.

5. **Use staging before production** — Set `STAGING_ENABLED=1` and use manual gates for production.

### For Review Apps:

1. **Always set `auto_stop_in`** — Prevent orphaned environments from wasting resources.

2. **Use lightweight configs for review apps** — 1 replica, minimal resources, no production databases.

3. **Use the same `rules` for deploy and stop jobs** — Otherwise, the stop job won't trigger.

4. **Set up wildcard DNS** — Required for unique review app URLs.

5. **Use merge request pipelines** — `rules: - if: '$CI_MERGE_REQUEST_IID'` is better than `only: branches` + `except: main`.

6. **Add resource quotas** — Use Kubernetes `ResourceQuota` and `LimitRange` to prevent review apps from consuming too many resources.

7. **Use Helm for deployment** — Makes cleanup much easier (`helm uninstall` removes everything).
