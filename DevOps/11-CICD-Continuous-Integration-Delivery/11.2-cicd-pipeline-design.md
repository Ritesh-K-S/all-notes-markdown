# 11.2 — CI/CD Pipeline Design — Stages, Gates, Artifacts

> **One Line Answer:** A CI/CD pipeline is a **series of automated steps** (stages) that your code goes through — from the moment you push it to the moment it reaches production — and designing it well is the difference between smooth deployments and total chaos.

---

## 🤔 Let's Start With an Analogy — The Car Manufacturing Line

Think about how cars are made in a factory:

```
Raw Materials → Chassis Assembly → Engine Install → Paint → Interior → Quality Check → Ship to Dealer
```

Each step is a **stage**. The car moves from one stage to the next. If the paint is bad, the car doesn't move forward — it gets **rejected** at that stage.

A CI/CD pipeline works **exactly the same way**, but for software:

```
Code → Build → Test → Security Scan → Package → Deploy to Staging → Deploy to Production
```

Each stage has a **purpose**, and the code only moves forward if it **passes** that stage. If it fails at any stage, the pipeline **stops**, the developer gets notified, and they fix it.

---

## 🏗️ What is a CI/CD Pipeline?

A CI/CD pipeline is an **automated workflow** that takes your code from **commit** to **production** through a series of defined stages.

### Think of It As:

```
┌────────────────────────────────────────────────────────────────────────┐
│                         CI/CD PIPELINE                                │
│                                                                        │
│   📝 Code  →  🔨 Build  →  🧪 Test  →  🔒 Scan  →  📦 Package      │
│                                                                        │
│   →  🚧 Deploy to Staging  →  ✅ Approve  →  🚀 Deploy to Prod      │
│                                                                        │
│   Each arrow = automatic. Each step = a "stage".                      │
│   The whole thing = a "pipeline".                                     │
└────────────────────────────────────────────────────────────────────────┘
```

### Key Terms You Need to Know:

| Term | What It Means | Example |
|------|-------------|---------|
| **Pipeline** | The entire automated workflow from start to finish | Your complete CI/CD process |
| **Stage** | A major phase of the pipeline (contains one or more jobs) | "Build", "Test", "Deploy" |
| **Job** | A specific task within a stage | "Run unit tests", "Build Docker image" |
| **Step** | An individual command or action within a job | `npm install`, `docker build` |
| **Trigger** | What starts the pipeline | Push to `main`, PR created, schedule (cron) |
| **Artifact** | Output files produced by a stage, passed to the next stage | Docker image, JAR file, test report |
| **Gate / Quality Gate** | A checkpoint that must pass before the pipeline continues | "All tests must pass", "No critical vulnerabilities" |
| **Runner / Agent** | The machine (physical or virtual) that executes the pipeline | GitHub-hosted runner, Jenkins agent, GitLab runner |

## 📐 The Standard Pipeline Stages — Explained in Detail

Let's go through each stage of a well-designed CI/CD pipeline:

### Stage 1: Source / Code (Trigger)

```
┌─────────────────────────────────────────┐
│  📝 SOURCE STAGE                       │
│                                         │
│  What happens:                          │
│  - Developer pushes code to Git         │
│  - Or creates a Pull Request            │
│  - Or merges to main branch             │
│                                         │
│  This TRIGGERS the pipeline             │
└─────────────────────────────────────────┘
```

**Common triggers:**

| Trigger | When Pipeline Runs |
|---------|-------------------|
| `push` to `main` | Every time code is merged to main |
| `pull_request` | When a PR is created or updated |
| `schedule` (cron) | On a schedule (e.g., nightly builds) |
| `manual` | When someone clicks a button |
| `tag` | When a Git tag is created (e.g., `v1.0.0`) |
| `webhook` | When an external event occurs |

**Example (GitHub Actions trigger):**

```yaml
on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]
  schedule:
    - cron: '0 2 * * *'  # Every day at 2 AM
```

**What this does:**
- Runs on every push to `main` or `develop`
- Runs when someone opens a PR targeting `main`
- Also runs automatically every night at 2 AM (nightly build)

---

### Stage 2: Build

```
┌─────────────────────────────────────────┐
│  🔨 BUILD STAGE                        │
│                                         │
│  What happens:                          │
│  - Install dependencies                 │
│  - Compile the source code              │
│  - Build the application binary         │
│  - Generate build artifacts             │
│                                         │
│  Goal: Turn source code into something  │
│  that can actually RUN                  │
└─────────────────────────────────────────┘
```

**Why is this stage important?**

If your code doesn't even **compile** or **build**, there's no point running tests or deploying. The build stage catches basic errors like:
- Syntax errors
- Missing imports
- Broken dependencies
- Compilation failures

**Example — Building different types of applications:**

```bash
# Java (Maven)
mvn clean package

# Node.js
npm install
npm run build

# Go
go build -o myapp ./cmd/main.go

# Python (create a distribution)
pip install -r requirements.txt
python setup.py sdist bdist_wheel

# Docker (build a container image)
docker build -t myapp:latest .
```

**Example (GitHub Actions build stage):**

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'   # Cache npm dependencies for faster builds

      - name: Install dependencies
        run: npm ci       # 'ci' is faster than 'install' for CI environments

      - name: Build
        run: npm run build

      - name: Upload build artifact
        uses: actions/upload-artifact@v4
        with:
          name: build-output
          path: dist/      # Save the build output for later stages
```

**Line-by-line explanation:**

| Line | What It Does |
|------|-------------|
| `runs-on: ubuntu-latest` | This job runs on a fresh Ubuntu machine |
| `actions/checkout@v4` | Clones the repository code |
| `cache: 'npm'` | Caches `node_modules` so next build is faster |
| `npm ci` | Installs dependencies from `package-lock.json` (deterministic, faster) |
| `npm run build` | Compiles/bundles the app (e.g., React → static files) |
| `upload-artifact` | Saves the `dist/` folder so the Test and Deploy stages can use it |

---

### Stage 3: Test

```
┌─────────────────────────────────────────┐
│  🧪 TEST STAGE                         │
│                                         │
│  What happens:                          │
│  - Run unit tests                       │
│  - Run integration tests                │
│  - Run end-to-end (E2E) tests           │
│  - Check code coverage                  │
│  - Run linting / code quality checks    │
│                                         │
│  Goal: Make sure the code WORKS and     │
│  doesn't break anything                 │
└─────────────────────────────────────────┘
```

**Types of tests in a pipeline:**

```
                    ┌──────────────────┐
                    │   E2E Tests      │  ← Slowest, but tests the whole system
                    │   (Selenium,     │
                    │    Cypress,      │
                    │    Playwright)   │
                  ┌─┴──────────────────┴─┐
                  │  Integration Tests   │  ← Tests how components work together
                  │  (API tests,         │
                  │   DB tests)          │
                ┌─┴──────────────────────┴─┐
                │     Unit Tests           │  ← Fastest, tests individual functions
                │     (Jest, JUnit,        │
                │      pytest, Go test)    │
                └──────────────────────────┘
                
        ◄── The "Test Pyramid" — More unit tests, fewer E2E tests ──►
```

**The Test Pyramid Explained:**

| Level | What It Tests | Speed | How Many? |
|-------|-------------|-------|-----------|
| **Unit Tests** | Individual functions/methods in isolation | Very fast (seconds) | Hundreds to thousands |
| **Integration Tests** | Multiple components working together (API + DB) | Medium (seconds to minutes) | Tens to hundreds |
| **E2E Tests** | The entire application from user perspective (browser) | Slow (minutes) | Few dozen |

**Example (Test stage in a pipeline):**

```yaml
  test:
    runs-on: ubuntu-latest
    needs: build           # This stage runs AFTER the build stage
    steps:
      - uses: actions/checkout@v4
      
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      
      - name: Install dependencies
        run: npm ci
      
      - name: Run linter
        run: npm run lint
      
      - name: Run unit tests
        run: npm test -- --coverage
      
      - name: Run integration tests
        run: npm run test:integration
      
      - name: Upload test results
        if: always()         # Upload even if tests fail
        uses: actions/upload-artifact@v4
        with:
          name: test-results
          path: coverage/
```

**Key points:**
- `needs: build` — This stage **waits** for the build stage to finish first
- `--coverage` — Generates a code coverage report (how much code is tested)
- `if: always()` — Uploads test reports even if tests fail (so you can see what went wrong)

### Stage 4: Security Scan

```
┌─────────────────────────────────────────┐
│  🔒 SECURITY SCAN STAGE               │
│                                         │
│  What happens:                          │
│  - SAST — Static code analysis          │
│  - SCA — Check for vulnerable deps      │
│  - Secret scanning — leaked passwords?  │
│  - Container image scanning             │
│                                         │
│  Goal: Catch security issues BEFORE     │
│  they reach production                  │
└─────────────────────────────────────────┘
```

**Types of security scans:**

| Scan Type | Full Name | What It Does | Example Tools |
|-----------|-----------|-------------|---------------|
| **SAST** | Static Application Security Testing | Analyzes source code for vulnerabilities (SQL injection, XSS, etc.) | SonarQube, Semgrep, CodeQL |
| **SCA** | Software Composition Analysis | Checks your dependencies for known vulnerabilities | Snyk, Dependabot, OWASP Dependency-Check |
| **DAST** | Dynamic Application Security Testing | Tests the running application for vulnerabilities | OWASP ZAP, Burp Suite |
| **Secret Scanning** | — | Looks for accidentally committed passwords, API keys, tokens | GitLeaks, TruffleHog, GitHub Secret Scanning |
| **Container Scanning** | — | Scans Docker images for vulnerabilities | Trivy, Grype, Snyk Container |

**Example (Security scan stage):**

```yaml
  security:
    runs-on: ubuntu-latest
    needs: build
    steps:
      - uses: actions/checkout@v4
      
      - name: Run SAST scan (Semgrep)
        uses: returntocorp/semgrep-action@v1
        with:
          config: p/owasp-top-ten
      
      - name: Check for secrets (GitLeaks)
        uses: gitleaks/gitleaks-action@v2
      
      - name: Scan dependencies (Snyk)
        uses: snyk/actions/node@master
        env:
          SNYK_TOKEN: ${{ secrets.SNYK_TOKEN }}
      
      - name: Scan Docker image (Trivy)
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: 'myapp:latest'
          severity: 'CRITICAL,HIGH'
          exit-code: '1'         # Fail the pipeline if critical vulnerabilities found
```

**What each scan does:**
- **Semgrep** — Reads your code and finds patterns that match known security issues (like SQL injection)
- **GitLeaks** — Scans commit history for passwords, API keys, tokens that shouldn't be there
- **Snyk** — Checks `package.json` and `package-lock.json` for dependencies with known CVEs
- **Trivy** — Scans the Docker image for vulnerable OS packages and libraries

### Stage 5: Package / Artifact

```
┌─────────────────────────────────────────┐
│  📦 PACKAGE STAGE                      │
│                                         │
│  What happens:                          │
│  - Build a Docker image                 │
│  - Push to container registry           │
│  - Or: Create a JAR/WAR/DEB/RPM        │
│  - Tag with version number              │
│  - Store as a versioned artifact        │
│                                         │
│  Goal: Create a deployable package      │
│  with a unique version                  │
└─────────────────────────────────────────┘
```

**Example (Docker package stage):**

```yaml
  package:
    runs-on: ubuntu-latest
    needs: [test, security]   # Runs AFTER both test AND security pass
    steps:
      - uses: actions/checkout@v4
      
      - name: Login to Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKER_USERNAME }}
          password: ${{ secrets.DOCKER_PASSWORD }}
      
      - name: Build and push Docker image
        uses: docker/build-push-action@v5
        with:
          push: true
          tags: |
            myapp:latest
            myapp:${{ github.sha }}
            myapp:v1.2.3
```

**Why three tags?**

| Tag | Purpose |
|-----|---------|
| `myapp:latest` | Always points to the newest version |
| `myapp:<git-sha>` | Tied to the exact commit — you can trace any image back to its code |
| `myapp:v1.2.3` | Semantic version — clear, human-readable version number |

### Stage 6: Deploy to Staging

```
┌─────────────────────────────────────────┐
│  🚧 STAGING DEPLOYMENT                │
│                                         │
│  What happens:                          │
│  - Deploy to a staging environment      │
│  - (Staging = a copy of production)     │
│  - Run smoke tests                      │
│  - Run acceptance tests                 │
│  - Verify everything works as expected  │
│                                         │
│  Goal: Test in a production-LIKE        │
│  environment before going live          │
└─────────────────────────────────────────┘
```

**What is a staging environment?**

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  Development → Staging → Production                        │
│                                                             │
│  Dev:      Where developers test locally                   │
│  Staging:  A copy of production (same config, same setup)  │
│  Prod:     The real thing — what users actually use        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**Example (Deploy to staging):**

```yaml
  deploy-staging:
    runs-on: ubuntu-latest
    needs: package
    environment: staging     # GitHub Environments feature
    steps:
      - name: Deploy to staging Kubernetes cluster
        run: |
          kubectl set image deployment/myapp \
            myapp=myapp:${{ github.sha }} \
            --namespace=staging
      
      - name: Wait for rollout
        run: |
          kubectl rollout status deployment/myapp \
            --namespace=staging \
            --timeout=300s
      
      - name: Run smoke tests
        run: |
          # Simple check — is the app responding?
          curl -f https://staging.myapp.com/health || exit 1
          
          # Run API smoke tests
          npm run test:smoke -- --base-url=https://staging.myapp.com
```

**What's happening:**
1. `kubectl set image` — Updates the staging deployment to use the new Docker image
2. `kubectl rollout status` — Waits up to 5 minutes for the deployment to complete
3. `curl /health` — Checks if the app is alive and responding
4. `test:smoke` — Runs a few critical tests against the staging URL

### Stage 7: Approval Gate (for Continuous Delivery)

```
┌─────────────────────────────────────────┐
│  ✅ APPROVAL GATE                      │
│                                         │
│  What happens:                          │
│  - Pipeline PAUSES here                 │
│  - Waits for human approval             │
│  - Reviewer checks staging              │
│  - Clicks "Approve" to continue         │
│                                         │
│  Goal: Give humans the final say        │
│  before going to production             │
└─────────────────────────────────────────┘
```

This is what separates **Continuous Delivery** from **Continuous Deployment**.

**How approval gates work in different tools:**

| Tool | How Approval Works |
|------|-------------------|
| **GitHub Actions** | Use `environment` with required reviewers |
| **GitLab CI** | Use `when: manual` on the deploy job |
| **Jenkins** | Use `input` step in the pipeline |
| **Azure DevOps** | Use approval gates on environments |

**Example (GitHub Actions — manual approval):**

```yaml
  deploy-production:
    runs-on: ubuntu-latest
    needs: deploy-staging
    environment:
      name: production        # This environment has required reviewers configured
      url: https://myapp.com
    steps:
      - name: Deploy to production
        run: |
          kubectl set image deployment/myapp \
            myapp=myapp:${{ github.sha }} \
            --namespace=production
```

In GitHub, you'd configure the `production` environment to require approval from specific people. The pipeline pauses, sends a notification, and waits for someone to click "Approve".

**Example (Jenkins — input step):**

```groovy
pipeline {
    agent any
    stages {
        stage('Deploy to Staging') {
            steps {
                sh 'kubectl apply -f k8s/staging/'
            }
        }
        stage('Approval') {
            steps {
                input message: 'Deploy to Production?', 
                      ok: 'Yes, Deploy!',
                      submitter: 'admin,lead-dev'
            }
        }
        stage('Deploy to Production') {
            steps {
                sh 'kubectl apply -f k8s/production/'
            }
        }
    }
}
```

**What's happening in this Jenkins pipeline:**
- `input message` — Pauses the pipeline and shows a prompt: "Deploy to Production?"
- `ok: 'Yes, Deploy!'` — The text on the approval button
- `submitter: 'admin,lead-dev'` — Only these users can approve

### Stage 8: Deploy to Production

```
┌─────────────────────────────────────────┐
│  🚀 PRODUCTION DEPLOYMENT             │
│                                         │
│  What happens:                          │
│  - Deploy the tested, approved artifact │
│  - Use a safe deployment strategy       │
│  - Monitor for errors                   │
│  - Be ready to rollback                 │
│                                         │
│  Goal: Get the code to users SAFELY     │
└─────────────────────────────────────────┘
```

**Common deployment strategies (quick overview — detailed in 11.25):**

```
Rolling Update:
  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐
  │ v1   │  │ v1   │  │ v1   │  │ v1   │   ← Start: all v1
  └──────┘  └──────┘  └──────┘  └──────┘
  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐
  │ v2 ✨│  │ v1   │  │ v1   │  │ v1   │   ← Replace one at a time
  └──────┘  └──────┘  └──────┘  └──────┘
  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐
  │ v2 ✨│  │ v2 ✨│  │ v2 ✨│  │ v2 ✨│   ← Done: all v2
  └──────┘  └──────┘  └──────┘  └──────┘

Blue/Green:
  BLUE (current) ← All traffic      GREEN (new) ← No traffic
  ┌──────────┐                      ┌──────────┐
  │   v1     │  ──── switch ────►   │   v2     │
  └──────────┘                      └──────────┘
  Now GREEN gets all traffic. BLUE is standby for rollback.

Canary:
  ┌──────────────────────────────────────────┐
  │  95% traffic → v1 (stable)              │
  │   5% traffic → v2 (canary)              │
  │                                          │
  │  Monitor the canary...                  │
  │  If OK → gradually increase to 100%     │
  │  If BAD → route all traffic back to v1  │
  └──────────────────────────────────────────┘
```

## 📦 Understanding Artifacts in Detail

Artifacts are one of the most important concepts in CI/CD. Let's really understand them.

### What is an Artifact?

An **artifact** is any **output produced by a pipeline stage** that can be:
- Passed to the next stage
- Stored for later use
- Downloaded for debugging
- Deployed to an environment

### Common Types of Artifacts:

| Artifact Type | Description | Example |
|--------------|-------------|---------|
| **Build artifact** | Compiled code or bundled application | `dist/` folder, `app.jar`, `main.exe` |
| **Docker image** | Containerized application | `myapp:v1.2.3` pushed to Docker Hub |
| **Test reports** | Results of test runs | `junit-report.xml`, `coverage/lcov.info` |
| **Helm chart** | Kubernetes package | `myapp-0.1.0.tgz` |
| **Binary/Executable** | Compiled program | `terraform`, `kubectl` custom plugin |
| **Configuration files** | Generated configs | `k8s-manifests/`, `.env.production` |
| **Documentation** | Generated docs | `docs/api.html`, `swagger.json` |

### How Artifacts Flow Through the Pipeline:

```
┌───────────────────────────────────────────────────────────────┐
│                    ARTIFACT FLOW                              │
│                                                               │
│  Build Stage                                                  │
│  ├── Produces: dist/ folder, Docker image                    │
│  └── Stores artifacts in artifact storage                    │
│          │                                                    │
│          ▼                                                    │
│  Test Stage                                                   │
│  ├── Downloads: dist/ folder                                 │
│  ├── Runs tests against it                                   │
│  ├── Produces: test-reports/, coverage/                      │
│  └── Stores test artifacts                                   │
│          │                                                    │
│          ▼                                                    │
│  Deploy Stage                                                 │
│  ├── Downloads: Docker image tag                             │
│  └── Deploys to Kubernetes/server                            │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

### Example (Passing artifacts between stages in GitHub Actions):

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run build
      
      # UPLOAD the build output as an artifact
      - name: Upload build artifact
        uses: actions/upload-artifact@v4
        with:
          name: app-build
          path: dist/
          retention-days: 5     # Keep for 5 days, then auto-delete

  test:
    runs-on: ubuntu-latest
    needs: build
    steps:
      # DOWNLOAD the artifact from the build stage
      - name: Download build artifact
        uses: actions/download-artifact@v4
        with:
          name: app-build
          path: dist/
      
      - run: npm test
```

**Why this matters:**
- The build stage creates the `dist/` folder and **uploads** it
- The test stage **downloads** that exact `dist/` folder
- This ensures the test stage tests the **exact same build** — no rebuilding!
- `retention-days: 5` — Artifacts take up storage, so we auto-delete after 5 days

## 🚦 Understanding Quality Gates

Quality Gates are **checkpoints** in your pipeline that enforce rules. If the rules aren't met, the pipeline **stops**.

### What is a Quality Gate?

Think of it like a tollbooth on a highway:

```
  🚗 → ──── [ QUALITY GATE ] ──── → 🚗 (passes if rules are met)
                   │
                   ↓
               ❌ BLOCKED (if rules are NOT met)
```

### Common Quality Gates:

| Quality Gate | Rule | What Happens if Failed |
|-------------|------|----------------------|
| **Test Coverage** | Coverage must be ≥ 80% | Pipeline fails — write more tests |
| **No Critical Bugs** | Zero critical/blocker bugs from SAST | Pipeline fails — fix the bugs |
| **No Security Vulnerabilities** | No HIGH or CRITICAL CVEs | Pipeline fails — update dependencies |
| **No Secrets in Code** | No passwords/keys in source code | Pipeline fails — remove the secrets |
| **Code Review** | At least 2 approvals on the PR | Can't merge until reviewed |
| **Performance** | Response time < 500ms | Pipeline fails — optimize the code |
| **Build Size** | Docker image < 500MB | Warning or fail — optimize the image |

### Example (SonarQube Quality Gate in a pipeline):

```yaml
  quality-gate:
    runs-on: ubuntu-latest
    needs: test
    steps:
      - name: SonarQube Scan
        uses: sonarcloud/sonarcloud-github-action@master
        env:
          SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
      
      - name: Check Quality Gate
        run: |
          # This will fail the pipeline if quality gate conditions aren't met
          # Quality gate rules (configured in SonarQube):
          # - Code coverage >= 80%
          # - No new critical/blocker issues
          # - Duplicated lines < 3%
          # - Maintainability rating = A
          
          STATUS=$(curl -s "https://sonarcloud.io/api/qualitygates/project_status?projectKey=myapp" \
            -H "Authorization: Bearer ${{ secrets.SONAR_TOKEN }}" \
            | jq -r '.projectStatus.status')
          
          if [ "$STATUS" != "OK" ]; then
            echo "❌ Quality Gate FAILED!"
            exit 1
          fi
          echo "✅ Quality Gate PASSED!"
```

## 🎨 Designing Your Pipeline — Best Practices

### 1. Fail Fast

Put the **fastest checks first**. If linting fails in 5 seconds, don't waste 10 minutes running all tests first.

```
Good Order:                          Bad Order:
  1. Lint (5 sec)         ✅          1. E2E Tests (10 min)    ❌
  2. Unit Tests (30 sec)              2. Integration Tests (5 min)
  3. Build (1 min)                    3. Build (1 min)
  4. Integration Tests (5 min)        4. Unit Tests (30 sec)
  5. E2E Tests (10 min)              5. Lint (5 sec)
```

### 2. Parallelize Independent Stages

If stages don't depend on each other, run them **at the same time**:

```
                    ┌── Unit Tests ──────┐
                    │                    │
Build ──────────────┼── Integration Tests┼────── Deploy
                    │                    │
                    ├── Security Scan ───┤
                    │                    │
                    └── Lint ────────────┘
                    
                    These 4 run in PARALLEL!
```

**Example (Parallel stages in GitHub Actions):**

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run build

  # These three jobs run IN PARALLEL after build
  unit-test:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test

  integration-test:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run test:integration

  security-scan:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm audit --audit-level=high

  # This job runs AFTER all three parallel jobs complete
  deploy:
    needs: [unit-test, integration-test, security-scan]
    runs-on: ubuntu-latest
    steps:
      - run: echo "All checks passed! Deploying..."
```

**How this runs:**

```
Timeline:
  0 min ──── build
  1 min ──── unit-test ─────┐
             integration ───┤──── deploy (starts after ALL three finish)
             security ──────┘
  4 min ──── done!

Without parallelization:
  0 min ──── build → unit → integration → security → deploy
  10 min ──── done!

Parallel is 2.5x faster in this example!
```

### 3. Use Caching

Don't download the same dependencies every time. Cache them!

```yaml
  - name: Cache node modules
    uses: actions/cache@v4
    with:
      path: ~/.npm
      key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }}
      restore-keys: |
        ${{ runner.os }}-node-
```

**What this does:**
- **First run:** Downloads all dependencies → stores them in cache
- **Next run:** Checks if `package-lock.json` changed → if not, uses the cache (way faster!)
- `hashFiles('**/package-lock.json')` — Creates a unique cache key based on your dependencies

### 4. Keep Pipelines Under 10 Minutes

The longer the pipeline, the less useful it becomes:

| Pipeline Duration | Developer Experience |
|-------------------|---------------------|
| < 5 minutes | "Fast! I'll wait for it." |
| 5-10 minutes | "I'll grab coffee." |
| 10-20 minutes | "I'll work on something else." |
| 20-30 minutes | "I forgot about my pipeline." |
| > 30 minutes | "Is CI still running? I deployed yesterday..." |

**Ways to speed up your pipeline:**
- Parallelize stages
- Cache dependencies
- Use smaller Docker base images
- Run only affected tests (test impact analysis)
- Use matrix builds for multi-platform testing

### 5. Use Matrix Builds for Multi-Version Testing

Need to test against multiple versions of Node.js? Use a matrix:

```yaml
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node-version: [18, 20, 22]    # Test against 3 versions
    steps:
      - uses: actions/checkout@v4
      - name: Use Node.js ${{ matrix.node-version }}
        uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
      - run: npm ci
      - run: npm test
```

**What happens:**

```
This creates 3 parallel jobs:
  ├── test (Node 18) ──── running in parallel
  ├── test (Node 20) ──── running in parallel
  └── test (Node 22) ──── running in parallel
```

## 🏗️ A Complete, Well-Designed Pipeline — Full Example

Let's put it all together. Here's a production-grade CI/CD pipeline:

```yaml
name: Production CI/CD Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  # ═══════════════════════════════════════
  # Stage 1: Lint & Code Quality (fastest)
  # ═══════════════════════════════════════
  lint:
    name: Lint & Code Quality
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - run: npm run lint
      - run: npm run format:check

  # ═══════════════════════════════════════
  # Stage 2: Build
  # ═══════════════════════════════════════
  build:
    name: Build Application
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - run: npm run build
      - uses: actions/upload-artifact@v4
        with:
          name: build-output
          path: dist/

  # ═══════════════════════════════════════
  # Stage 3: Tests (parallel with security)
  # ═══════════════════════════════════════
  unit-test:
    name: Unit Tests
    runs-on: ubuntu-latest
    needs: build
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - run: npm test -- --coverage
      - uses: actions/upload-artifact@v4
        if: always()
        with:
          name: coverage-report
          path: coverage/

  integration-test:
    name: Integration Tests
    runs-on: ubuntu-latest
    needs: build
    services:
      postgres:
        image: postgres:15
        env:
          POSTGRES_PASSWORD: testpassword
        ports:
          - 5432:5432
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - run: npm run test:integration
        env:
          DATABASE_URL: postgresql://postgres:testpassword@localhost:5432/testdb

  # ═══════════════════════════════════════
  # Stage 4: Security (parallel with tests)
  # ═══════════════════════════════════════
  security:
    name: Security Scan
    runs-on: ubuntu-latest
    needs: build
    steps:
      - uses: actions/checkout@v4
      - name: Run Trivy vulnerability scanner
        uses: aquasecurity/trivy-action@master
        with:
          scan-type: 'fs'
          severity: 'CRITICAL,HIGH'
          exit-code: '1'

  # ═══════════════════════════════════════
  # Stage 5: Package (Docker Image)
  # ═══════════════════════════════════════
  package:
    name: Build & Push Docker Image
    runs-on: ubuntu-latest
    needs: [lint, unit-test, integration-test, security]
    if: github.ref == 'refs/heads/main'    # Only on main branch
    permissions:
      contents: read
      packages: write
    steps:
      - uses: actions/checkout@v4
      - name: Log in to Container Registry
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          push: true
          tags: |
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:latest
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}

  # ═══════════════════════════════════════
  # Stage 6: Deploy to Staging
  # ═══════════════════════════════════════
  deploy-staging:
    name: Deploy to Staging
    runs-on: ubuntu-latest
    needs: package
    environment: staging
    steps:
      - name: Deploy to staging
        run: |
          echo "Deploying ${{ github.sha }} to staging..."
          kubectl set image deployment/myapp \
            myapp=${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }} \
            --namespace=staging
      - name: Smoke test
        run: curl -f https://staging.myapp.com/health

  # ═══════════════════════════════════════
  # Stage 7: Deploy to Production (manual)
  # ═══════════════════════════════════════
  deploy-production:
    name: Deploy to Production
    runs-on: ubuntu-latest
    needs: deploy-staging
    environment:
      name: production           # Requires manual approval
      url: https://myapp.com
    steps:
      - name: Deploy to production
        run: |
          echo "Deploying ${{ github.sha }} to production..."
          kubectl set image deployment/myapp \
            myapp=${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }} \
            --namespace=production
      - name: Verify deployment
        run: |
          kubectl rollout status deployment/myapp \
            --namespace=production --timeout=300s
          curl -f https://myapp.com/health
```

## 🧠 Key Takeaways

| Concept | Remember This |
|---------|--------------|
| **Pipeline** | Automated workflow from code to production |
| **Stages** | Major phases: Build → Test → Scan → Package → Deploy |
| **Artifacts** | Output files passed between stages (Docker images, JARs, test reports) |
| **Quality Gates** | Checkpoints that enforce rules (coverage ≥ 80%, no critical vulnerabilities) |
| **Fail Fast** | Put fastest checks first — don't waste time if linting fails |
| **Parallelize** | Run independent stages at the same time to save time |
| **Cache** | Don't re-download dependencies every run |
| **< 10 minutes** | Keep your pipeline fast or developers will ignore it |
