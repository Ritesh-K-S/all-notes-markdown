## 1. Introduction — What is Azure DevOps?

**Azure DevOps** is Microsoft's all-in-one platform for software development and DevOps. Think of it as a complete toolkit that gives you everything from project management to CI/CD pipelines to artifact storage — all under one roof.

It was previously known as:
- **Visual Studio Team Services (VSTS)** (2015-2018)
- **Team Foundation Server (TFS)** (on-premises version)

Today, it comes in two flavors:
- **Azure DevOps Services** — Cloud-hosted (SaaS), at `dev.azure.com`
- **Azure DevOps Server** — Self-hosted (on-premises)

> **Fun fact:** You do NOT need to use Azure cloud to use Azure DevOps. You can deploy to AWS, GCP, on-premises, or anywhere. The name is a bit misleading!

## 2. Azure DevOps Services Overview

Azure DevOps is not just CI/CD. It's a suite of **5 core services**:

| Service | What It Does | Comparable To |
|---------|-------------|---------------|
| **Azure Boards** | Project management, work items, sprints, Kanban boards | Jira, Trello |
| **Azure Repos** | Git repositories (also supports TFVC) | GitHub, GitLab repos |
| **Azure Pipelines** | CI/CD pipelines ← **This is our focus** | Jenkins, GitHub Actions, GitLab CI |
| **Azure Test Plans** | Manual and exploratory testing | TestRail |
| **Azure Artifacts** | Package management (npm, NuGet, Maven, pip, Cargo) | Nexus, Artifactory, GitHub Packages |

You can use all five together or pick and choose. For example, you can use **GitHub** for repos but **Azure Pipelines** for CI/CD. They integrate beautifully.

### Free Tier

Azure DevOps is **free** for:
- Up to **5 users** (Basic plan)
- **1 free Microsoft-hosted parallel job** (1800 minutes/month)
- Unlimited private repos
- Unlimited Azure Boards work items

## 3. Azure Pipelines — The CI/CD Engine

**Azure Pipelines** is the CI/CD component of Azure DevOps. Here's what makes it special:

### Key Features:

| Feature | Description |
|---------|-------------|
| **Multi-platform** | Build on Linux, macOS, and Windows agents |
| **Multi-language** | Supports every major language (Node.js, Python, Java, .NET, Go, Ruby, PHP, etc.) |
| **YAML Pipelines** | Pipeline as code — version controlled |
| **Classic Pipelines** | Visual/GUI-based pipeline editor (legacy but still available) |
| **Multi-stage** | Build, test, and deploy in a single pipeline |
| **Environments** | Approval gates, deployment history, Kubernetes integration |
| **Container support** | Run jobs in Docker containers, deploy to Kubernetes |
| **Parallel jobs** | Run multiple jobs simultaneously |
| **Integration** | Works with GitHub, Bitbucket, GitLab, SVN, and Azure Repos |

## 4. Getting Started — Setting Up Azure DevOps

### Step 1: Create an Organization

1. Go to [dev.azure.com](https://dev.azure.com)
2. Sign in with your Microsoft account (or create one — it's free)
3. Click **"New Organization"**
4. Give it a name (e.g., `my-company`)
5. Your URL will be: `https://dev.azure.com/my-company`

### Step 2: Create a Project

1. Inside your organization, click **"New Project"**
2. Set:
   - **Name:** `my-app`
   - **Visibility:** Private (or Public)
   - **Version control:** Git
   - **Work item process:** Agile (or Scrum/CMMI)
3. Click **Create**

### Step 3: Connect Your Code

You have 3 options:
- **Azure Repos** — Push code directly to Azure's Git repos
- **GitHub** — Connect your GitHub repository
- **Bitbucket / Other** — Import from external repos

For GitHub integration:
1. Go to **Pipelines → New Pipeline**
2. Select **GitHub**
3. Authorize Azure Pipelines (GitHub App)
4. Select your repository

## 5. Core Concepts — Pipeline Terminology

Before diving into YAML, let's understand the vocabulary:

```
Pipeline
  └── Stage (e.g., "Build", "Test", "Deploy-Staging", "Deploy-Production")
        └── Job (e.g., "build-linux", "build-windows")
              └── Step (e.g., "install dependencies", "run tests", "publish artifacts")
                    └── Task (a pre-built action) or Script (custom command)
```

| Term | What It Is | Example |
|------|-----------|---------|
| **Pipeline** | The entire CI/CD workflow | The whole `azure-pipelines.yml` file |
| **Stage** | A logical division (build, test, deploy) | `Build`, `Deploy-Prod` |
| **Job** | A unit of work that runs on an agent | `build-on-linux`, `test-on-windows` |
| **Step** | A single action within a job | `npm install`, `docker build` |
| **Task** | A pre-built, reusable action | `DotNetCoreCLI@2`, `Docker@2` |
| **Script** | A custom shell/PowerShell/bash command | `script: echo "Hello"` |
| **Agent** | The machine (VM) that runs the job | Microsoft-hosted or self-hosted |
| **Pool** | A collection of agents | `vmImage: 'ubuntu-latest'` |
| **Trigger** | What starts the pipeline | Push to `main`, PR created |
| **Artifact** | Output of a build (binaries, Docker images) | `.zip`, `.jar`, Docker image |
| **Environment** | A deployment target | `staging`, `production` |

## 6. Classic Pipelines vs YAML Pipelines

Azure DevOps supports two ways to define pipelines:

### Classic Pipelines (GUI-based)

- Created through the **visual editor** in the browser
- Drag-and-drop tasks
- Configuration stored **on the server** (not in your repo)
- Good for beginners but **not recommended** for teams

### YAML Pipelines (Code-based) ← **Recommended**

- Defined in an `azure-pipelines.yml` file in your repo
- Version controlled alongside your code
- Supports all features: stages, templates, conditions, etc.
- **Industry standard** — similar to GitHub Actions and GitLab CI

```
Classic Pipelines = Clicking buttons in a UI
YAML Pipelines = Writing code in a file (pipeline-as-code)
```

> **Recommendation:** Always use YAML pipelines. Classic pipelines are being de-emphasized by Microsoft.

## 7. Your First YAML Pipeline — Step by Step

Let's create a simple pipeline for a Node.js app.

### Step 1: Create `azure-pipelines.yml` in your repo root

```yaml
# azure-pipelines.yml — My First Pipeline

trigger:
  - main           # Run when code is pushed to main branch

pool:
  vmImage: 'ubuntu-latest'   # Use a Microsoft-hosted Ubuntu agent

steps:
  - task: NodeTool@0
    inputs:
      versionSpec: '18.x'
    displayName: 'Install Node.js 18'

  - script: npm install
    displayName: 'Install dependencies'

  - script: npm test
    displayName: 'Run tests'

  - script: npm run build
    displayName: 'Build the app'
```

### Step 2: Push to your repository

```bash
git add azure-pipelines.yml
git commit -m "Add Azure Pipeline"
git push
```

### Step 3: Azure DevOps auto-detects the file and runs the pipeline!

Go to **Pipelines** in Azure DevOps — you'll see your pipeline running.

### Let's break down what's happening:

```yaml
trigger:
  - main
```
> "Run this pipeline whenever someone pushes to the `main` branch."

```yaml
pool:
  vmImage: 'ubuntu-latest'
```
> "Use a Microsoft-hosted Ubuntu virtual machine to run this pipeline."

```yaml
steps:
  - task: NodeTool@0
    inputs:
      versionSpec: '18.x'
```
> "Install Node.js version 18 on the agent."

```yaml
  - script: npm install
    displayName: 'Install dependencies'
```
> "Run `npm install` and show 'Install dependencies' as the step name in the UI."

## 8. Pipeline Structure Deep Dive

Here's the **full structure** of an Azure Pipeline YAML file:

```yaml
# ==========================================
# Full Azure Pipeline Structure
# ==========================================

# Pipeline name (displayed in UI)
name: '$(Build.DefinitionName)_$(Date:yyyyMMdd)_$(Rev:r)'

# When to trigger
trigger:
  branches:
    include:
      - main
      - develop
    exclude:
      - feature/experimental/*
  paths:
    include:
      - src/*
    exclude:
      - docs/*

# PR triggers
pr:
  branches:
    include:
      - main

# Pipeline-level variables
variables:
  buildConfiguration: 'Release'
  nodeVersion: '18.x'

# Agent pool (can be overridden per job)
pool:
  vmImage: 'ubuntu-latest'

# Stages
stages:
  - stage: Build
    displayName: 'Build Stage'
    jobs:
      - job: BuildJob
        displayName: 'Build'
        steps:
          - script: echo "Building..."

  - stage: Test
    displayName: 'Test Stage'
    dependsOn: Build     # Runs after Build
    jobs:
      - job: TestJob
        steps:
          - script: echo "Testing..."

  - stage: Deploy
    displayName: 'Deploy Stage'
    dependsOn: Test      # Runs after Test
    condition: and(succeeded(), eq(variables['Build.SourceBranch'], 'refs/heads/main'))
    jobs:
      - deployment: DeployJob
        environment: 'production'
        strategy:
          runOnce:
            deploy:
              steps:
                - script: echo "Deploying to production..."
```

### Simple vs Multi-Stage

If your pipeline is simple (just one job, no stages), you can skip the `stages` and `jobs` wrapper:

```yaml
# Simple — no stages, no jobs (single job pipeline)
trigger:
  - main
pool:
  vmImage: 'ubuntu-latest'
steps:
  - script: echo "Hello, world!"
```

```yaml
# Multi-stage — explicit stages and jobs
stages:
  - stage: Build
    jobs:
      - job: BuildApp
        steps:
          - script: echo "Building..."
  - stage: Deploy
    jobs:
      - job: DeployApp
        steps:
          - script: echo "Deploying..."
```

## 9. Triggers — When Does the Pipeline Run?

### CI Trigger (Push Trigger)

Runs when code is pushed to specific branches:

```yaml
# Simple — trigger on main and develop
trigger:
  - main
  - develop

# Advanced — include/exclude branches and paths
trigger:
  branches:
    include:
      - main
      - release/*
    exclude:
      - feature/experimental/*
  paths:
    include:
      - src/**
      - package.json
    exclude:
      - docs/**
      - '**/*.md'
  tags:
    include:
      - v*        # Trigger on tags like v1.0, v2.3
```

### PR Trigger (Pull Request)

Runs when a pull request is created or updated:

```yaml
pr:
  branches:
    include:
      - main
      - develop
  paths:
    exclude:
      - docs/**

# Or simply:
pr:
  - main
```

### Scheduled Trigger (Cron)

```yaml
schedules:
  - cron: '0 2 * * *'         # Every day at 2:00 AM UTC
    displayName: 'Nightly Build'
    branches:
      include:
        - main
    always: true                # Run even if no code changes

  - cron: '0 12 * * 0'        # Every Sunday at noon
    displayName: 'Weekly Full Test'
    branches:
      include:
        - main
        - develop
```

### Manual Trigger

```yaml
# Disable automatic triggers completely
trigger: none
pr: none

# Pipeline can only be run manually from the Azure DevOps UI
```

### Pipeline Trigger (Pipeline Completion)

Trigger one pipeline after another completes:

```yaml
resources:
  pipelines:
    - pipeline: buildPipeline          # Alias
      source: 'My Build Pipeline'     # Name of the other pipeline
      trigger:
        branches:
          include:
            - main
```

## 10. Stages, Jobs, and Steps

### Stages

Stages are the **big blocks** of your pipeline. They run sequentially by default.

```yaml
stages:
  - stage: Build
    displayName: 'Build'
    jobs:
      - job: BuildApp
        steps:
          - script: echo "Building..."

  - stage: Test
    displayName: 'Test'
    dependsOn: Build        # Explicit dependency (default is sequential)
    jobs:
      - job: UnitTests
        steps:
          - script: echo "Running unit tests..."
      - job: IntegrationTests
        steps:
          - script: echo "Running integration tests..."

  - stage: Deploy
    displayName: 'Deploy'
    dependsOn:
      - Build
      - Test                # Depends on BOTH Build and Test
    condition: succeeded()  # Only if previous stages succeeded
```

### Parallel Stages

```yaml
stages:
  - stage: Build
    jobs:
      - job: BuildApp
        steps:
          - script: echo "Building..."

  # These two stages run IN PARALLEL (both depend only on Build)
  - stage: TestUnit
    dependsOn: Build
    jobs:
      - job: UnitTest
        steps:
          - script: echo "Unit tests..."

  - stage: TestIntegration
    dependsOn: Build
    jobs:
      - job: IntegrationTest
        steps:
          - script: echo "Integration tests..."

  # This waits for BOTH test stages
  - stage: Deploy
    dependsOn:
      - TestUnit
      - TestIntegration
```

### Jobs

Jobs run on agents. Multiple jobs in a stage run **in parallel** by default.

```yaml
jobs:
  # Regular job
  - job: BuildOnLinux
    pool:
      vmImage: 'ubuntu-latest'
    steps:
      - script: echo "Building on Linux"

  # This runs in PARALLEL with BuildOnLinux
  - job: BuildOnWindows
    pool:
      vmImage: 'windows-latest'
    steps:
      - script: echo "Building on Windows"

  # Deployment job (special type — has an environment)
  - deployment: DeployToStaging
    environment: 'staging'
    strategy:
      runOnce:
        deploy:
          steps:
            - script: echo "Deploying..."
```

### Steps

Steps are the individual actions. Two main types:

```yaml
steps:
  # Type 1: script — Run a command
  - script: |
      echo "Hello"
      npm install
      npm test
    displayName: 'Run tests'

  # Type 2: task — Use a pre-built task
  - task: Docker@2
    inputs:
      command: 'buildAndPush'
      repository: 'myapp'
      dockerfile: 'Dockerfile'
    displayName: 'Build Docker image'

  # Bash (explicitly use bash shell)
  - bash: echo "This runs in bash"

  # PowerShell
  - powershell: Write-Host "This runs in PowerShell"

  # pwsh (cross-platform PowerShell Core)
  - pwsh: Write-Host "This runs in PowerShell Core"

  # Checkout step (automatically included, but can be customized)
  - checkout: self
    clean: true
    fetchDepth: 1    # Shallow clone for speed
```

## 11. Variables and Parameters

### Pipeline Variables

```yaml
# Method 1: Simple key-value
variables:
  buildConfiguration: 'Release'
  nodeVersion: '18.x'

# Method 2: Variable with groups and explicit definitions
variables:
  - name: buildConfiguration
    value: 'Release'
  - name: isMainBranch
    value: ${{ eq(variables['Build.SourceBranch'], 'refs/heads/main') }}
  - group: my-variable-group    # Variables from a Library group
```

### Using Variables

```yaml
steps:
  - script: echo "Config is $(buildConfiguration)"    # Runtime macro syntax
  - script: echo "Branch is $(Build.SourceBranchName)" # Predefined variable
```

### Variable Scoping

```yaml
variables:
  globalVar: 'I am available everywhere'

stages:
  - stage: Build
    variables:
      stageVar: 'I am only available in Build stage'
    jobs:
      - job: BuildJob
        variables:
          jobVar: 'I am only available in BuildJob'
        steps:
          - script: |
              echo "Global: $(globalVar)"     # ✅ Works
              echo "Stage: $(stageVar)"       # ✅ Works
              echo "Job: $(jobVar)"           # ✅ Works
```

### Predefined Variables

Azure DevOps provides hundreds of predefined variables. Here are the most useful ones:

| Variable | Description | Example Value |
|----------|-------------|---------------|
| `$(Build.SourceBranch)` | Full branch ref | `refs/heads/main` |
| `$(Build.SourceBranchName)` | Branch name only | `main` |
| `$(Build.BuildId)` | Unique build ID | `1234` |
| `$(Build.BuildNumber)` | Build number | `20240115.1` |
| `$(Build.Repository.Name)` | Repo name | `my-app` |
| `$(System.DefaultWorkingDirectory)` | Working directory | `/home/vsts/work/1/s` |
| `$(Pipeline.Workspace)` | Pipeline workspace | `/home/vsts/work/1` |
| `$(Build.ArtifactStagingDirectory)` | Artifact staging directory | `/home/vsts/work/1/a` |
| `$(Agent.OS)` | Agent OS | `Linux`, `Windows_NT` |

### Parameters (Runtime Input)

Parameters let you provide input when manually running a pipeline:

```yaml
parameters:
  - name: environment
    displayName: 'Deploy to which environment?'
    type: string
    default: 'staging'
    values:
      - staging
      - production

  - name: runTests
    displayName: 'Run tests?'
    type: boolean
    default: true

  - name: nodeVersion
    displayName: 'Node.js version'
    type: string
    default: '18.x'

trigger: none   # Manual only

stages:
  - stage: Build
    jobs:
      - job: BuildJob
        steps:
          - task: NodeTool@0
            inputs:
              versionSpec: '${{ parameters.nodeVersion }}'

          - script: npm test
            condition: eq('${{ parameters.runTests }}', true)
            displayName: 'Run tests'

  - stage: Deploy
    jobs:
      - deployment: DeployJob
        environment: '${{ parameters.environment }}'
        strategy:
          runOnce:
            deploy:
              steps:
                - script: echo "Deploying to ${{ parameters.environment }}"
```

When you run this pipeline manually, Azure DevOps shows a **form** with dropdowns and checkboxes!

### Variable Syntax Cheat Sheet

| Syntax | When It's Evaluated | Use Case |
|--------|-------------------|----------|
| `$(variable)` | Runtime (macro) | Most common. Used in scripts and task inputs |
| `${{ variables.variable }}` | Compile time (template) | Used in conditions and template expressions |
| `$[variables.variable]` | Runtime (expression) | Used in conditions evaluated at runtime |

## 12. Agent Pools and Build Agents

An **agent** is the machine that actually runs your pipeline jobs.

### Microsoft-Hosted Agents

Microsoft provides pre-configured VMs. You just pick one:

```yaml
pool:
  vmImage: 'ubuntu-latest'          # Ubuntu 22.04 LTS
  # vmImage: 'ubuntu-20.04'         # Ubuntu 20.04 (specific version)
  # vmImage: 'windows-latest'       # Windows Server 2022
  # vmImage: 'windows-2019'         # Windows Server 2019
  # vmImage: 'macos-latest'         # macOS 13 (Ventura)
  # vmImage: 'macos-12'             # macOS 12 (Monterey)
```

**What's pre-installed?**
Each agent comes with a TON of pre-installed software:
- Docker, Docker Compose
- Node.js, Python, Java, .NET, Go, Ruby
- kubectl, Helm, Terraform
- Azure CLI, AWS CLI, gcloud
- Git, Curl, jq

Check the full list: [Microsoft-hosted agent software](https://learn.microsoft.com/en-us/azure/devops/pipelines/agents/hosted)

### Self-Hosted Agents

You can run your own agent on any machine:

```bash
# Download and configure a self-hosted agent (Linux)
mkdir myagent && cd myagent

curl -O https://vstsagentpackage.azureedge.net/agent/3.x.x/vsts-agent-linux-x64-3.x.x.tar.gz
tar zxvf vsts-agent-linux-x64-3.x.x.tar.gz

# Configure
./config.sh --url https://dev.azure.com/your-org --auth pat --token YOUR_PAT

# Run as a service
sudo ./svc.sh install
sudo ./svc.sh start
```

Then use it in your pipeline:

```yaml
pool:
  name: 'My Self-Hosted Pool'       # Custom pool name
  demands:
    - docker                          # Agent must have Docker
    - Agent.OS -equals Linux         # Must be Linux
```

### When to use Self-Hosted?

| Use Microsoft-Hosted When... | Use Self-Hosted When... |
|------------------------------|------------------------|
| Standard builds | Need special hardware (GPU, ARM) |
| No special requirements | Need access to internal network |
| Don't want to manage infrastructure | Need persistent caches |
| Want clean environments each run | Need more than 1800 min/month for free |

## 13. Tasks — The Building Blocks

**Tasks** are pre-built, reusable actions. Think of them like GitHub Actions' "actions" or Jenkins' "plugins."

### Common Tasks

```yaml
# Docker task — build and push images
- task: Docker@2
  inputs:
    containerRegistry: 'my-docker-hub-connection'
    repository: 'myapp'
    command: 'buildAndPush'
    Dockerfile: '**/Dockerfile'
    tags: |
      $(Build.BuildId)
      latest

# .NET build
- task: DotNetCoreCLI@2
  inputs:
    command: 'build'
    projects: '**/*.csproj'
    arguments: '--configuration $(buildConfiguration)'

# Node.js tool installer
- task: NodeTool@0
  inputs:
    versionSpec: '18.x'

# Publish test results
- task: PublishTestResults@2
  inputs:
    testResultsFormat: 'JUnit'
    testResultsFiles: '**/test-results.xml'

# Publish build artifacts
- task: PublishBuildArtifacts@1
  inputs:
    PathtoPublish: '$(Build.ArtifactStagingDirectory)'
    ArtifactName: 'drop'

# Copy files
- task: CopyFiles@2
  inputs:
    SourceFolder: 'dist'
    Contents: '**'
    TargetFolder: '$(Build.ArtifactStagingDirectory)'

# Azure Web App deployment
- task: AzureWebApp@1
  inputs:
    azureSubscription: 'my-azure-subscription'
    appType: 'webAppLinux'
    appName: 'my-web-app'
    package: '$(Build.ArtifactStagingDirectory)/**/*.zip'

# Kubernetes deploy
- task: KubernetesManifest@0
  inputs:
    action: 'deploy'
    kubernetesServiceConnection: 'my-k8s-connection'
    namespace: 'production'
    manifests: 'k8s/*.yml'
    containers: 'myregistry.azurecr.io/myapp:$(Build.BuildId)'

# Helm deploy
- task: HelmDeploy@0
  inputs:
    connectionType: 'Kubernetes Service Connection'
    kubernetesServiceConnection: 'my-k8s-connection'
    command: 'upgrade'
    chartType: 'FilePath'
    chartPath: './chart'
    releaseName: 'myapp'
    arguments: '--set image.tag=$(Build.BuildId)'
```

### Finding Tasks

- **Marketplace:** [marketplace.visualstudio.com](https://marketplace.visualstudio.com/azuredevops)
- **In pipeline editor:** Click "Show assistant" on the right side
- **Task reference:** `TaskName@Version` (e.g., `Docker@2`, `DotNetCoreCLI@2`)

## 14. Templates — Reusable Pipeline Code

Templates are one of the **most powerful features** of Azure Pipelines. They let you reuse pipeline code across multiple pipelines and projects.

### Step Template

```yaml
# templates/build-steps.yml
parameters:
  - name: nodeVersion
    type: string
    default: '18.x'

steps:
  - task: NodeTool@0
    inputs:
      versionSpec: ${{ parameters.nodeVersion }}
    displayName: 'Install Node.js ${{ parameters.nodeVersion }}'

  - script: npm ci
    displayName: 'Install dependencies'

  - script: npm run build
    displayName: 'Build'

  - script: npm test
    displayName: 'Test'
```

```yaml
# azure-pipelines.yml — Using the template
trigger:
  - main

pool:
  vmImage: 'ubuntu-latest'

steps:
  - template: templates/build-steps.yml
    parameters:
      nodeVersion: '20.x'
```

### Job Template

```yaml
# templates/build-job.yml
parameters:
  - name: os
    type: string
  - name: vmImage
    type: string

jobs:
  - job: Build_${{ parameters.os }}
    displayName: 'Build on ${{ parameters.os }}'
    pool:
      vmImage: ${{ parameters.vmImage }}
    steps:
      - script: echo "Building on ${{ parameters.os }}..."
      - script: npm ci
      - script: npm test
```

```yaml
# azure-pipelines.yml — Matrix build across OS
trigger:
  - main

stages:
  - stage: Build
    jobs:
      - template: templates/build-job.yml
        parameters:
          os: 'Linux'
```

## 15. Environments and Approvals

**Environments** are deployment targets with approval gates and deployment history.

### Creating an Environment

1. Go to **Pipelines → Environments**
2. Click **"New environment"**
3. Name it (e.g., `staging`, `production`)
4. Optionally link a Kubernetes cluster or VM resource

### Using Environments in Pipeline

```yaml
stages:
  - stage: DeployStaging
    jobs:
      - deployment: DeployToStaging
        environment: 'staging'           # 👈 References the environment
        strategy:
          runOnce:
            deploy:
              steps:
                - script: echo "Deploying to staging"

  - stage: DeployProd
    jobs:
      - deployment: DeployToProd
        environment: 'production'        # 👈 References production environment
        strategy:
          runOnce:
            deploy:
              steps:
                - script: echo "Deploying to production"
```

### Adding Approval Gates

1. Go to **Pipelines → Environments → production**
2. Click the **⋮** menu → **Approvals and checks**
3. Click **"+"** to add a check:
   - **Approvals** — Require specific people to approve
   - **Branch control** — Only allow deployments from certain branches
   - **Business hours** — Only deploy during working hours
   - **Exclusive lock** — Only one deployment at a time

When the pipeline reaches the production stage, it will **pause** and wait for approval.

### Deployment Strategies

```yaml
# runOnce — Deploy all at once (simple)
strategy:
  runOnce:
    deploy:
      steps:
        - script: echo "Deploying..."

# rolling — Update targets in batches
strategy:
  rolling:
    maxParallel: 2         # Update 2 targets at a time
    deploy:
      steps:
        - script: echo "Deploying to $(Agent.MachineName)"

# canary — Gradual rollout (percentage-based)
strategy:
  canary:
    increments: [10, 20]   # First 10%, then 20%, then full
    deploy:
      steps:
        - script: echo "Canary deploy"
    on:
      success:
        steps:
          - script: echo "Canary succeeded, continuing..."
      failure:
        steps:
          - script: echo "Canary failed, rolling back..."
```

## 16. Artifacts — Passing Data Between Stages

Stages run on **different agents**, so you need artifacts to pass data between them.

### Publishing Artifacts

```yaml
# Method 1: PublishBuildArtifacts task (classic)
- task: PublishBuildArtifacts@1
  inputs:
    PathtoPublish: '$(Build.ArtifactStagingDirectory)'
    ArtifactName: 'webapp'

# Method 2: publish shortcut (modern, simpler)
- publish: $(Build.ArtifactStagingDirectory)
  artifact: 'webapp'
  displayName: 'Publish build artifacts'
```

### Downloading Artifacts

```yaml
# In a later stage:
- download: current            # Download from current pipeline run
  artifact: 'webapp'          # Artifact name

# Or using the task:
- task: DownloadBuildArtifacts@1
  inputs:
    buildType: 'current'
    downloadType: 'single'
    artifactName: 'webapp'
    downloadPath: '$(System.ArtifactsDirectory)'
```

### Complete Example — Build and Deploy with Artifacts

```yaml
stages:
  - stage: Build
    jobs:
      - job: BuildApp
        pool:
          vmImage: 'ubuntu-latest'
        steps:
          - script: npm ci
          - script: npm run build

          # Copy build output to staging directory
          - task: CopyFiles@2
            inputs:
              SourceFolder: 'dist'
              Contents: '**'
              TargetFolder: '$(Build.ArtifactStagingDirectory)'

          # Publish artifacts
          - publish: $(Build.ArtifactStagingDirectory)
            artifact: 'webapp'

  - stage: Deploy
    dependsOn: Build
    jobs:
      - deployment: DeployApp
        environment: 'production'
        strategy:
          runOnce:
            deploy:
              steps:
                # Artifacts are auto-downloaded in deployment jobs
                # Available at: $(Pipeline.Workspace)/webapp/
                - script: ls $(Pipeline.Workspace)/webapp/
                  displayName: 'List artifacts'
                - script: echo "Deploying from $(Pipeline.Workspace)/webapp/"
```

## 18. Multi-Stage Pipeline — Build, Test, Deploy

Let's build a **complete, production-ready** multi-stage pipeline:

```yaml
# azure-pipelines.yml — Complete Multi-Stage Pipeline

trigger:
  branches:
    include:
      - main
      - develop
  paths:
    exclude:
      - docs/**
      - '**/*.md'

pr:
  branches:
    include:
      - main

variables:
  dockerRegistryServiceConnection: 'my-acr-connection'
  imageRepository: 'myapp'
  containerRegistry: 'myregistry.azurecr.io'
  dockerfilePath: 'Dockerfile'
  tag: '$(Build.BuildId)'

stages:
  # ========== BUILD ==========
  - stage: Build
    displayName: 'Build & Test'
    jobs:
      - job: Build
        pool:
          vmImage: 'ubuntu-latest'
        steps:
          # Install Node.js
          - task: NodeTool@0
            inputs:
              versionSpec: '18.x'

          # Install dependencies
          - script: npm ci
            displayName: 'Install dependencies'

          # Run linting
          - script: npm run lint
            displayName: 'Lint code'

          # Run unit tests
          - script: npm test -- --coverage --ci
            displayName: 'Run unit tests'

          # Publish test results
          - task: PublishTestResults@2
            inputs:
              testResultsFormat: 'JUnit'
              testResultsFiles: '**/junit.xml'
            condition: always()   # Publish even if tests fail

          # Publish code coverage
          - task: PublishCodeCoverageResults@1
            inputs:
              codeCoverageTool: 'Cobertura'
              summaryFileLocation: 'coverage/cobertura-coverage.xml'

          # Build Docker image
          - task: Docker@2
            displayName: 'Build Docker image'
            inputs:
              containerRegistry: $(dockerRegistryServiceConnection)
              repository: $(imageRepository)
              command: 'build'
              Dockerfile: $(dockerfilePath)
              tags: |
                $(tag)
                latest

          # Push Docker image
          - task: Docker@2
            displayName: 'Push Docker image'
            inputs:
              containerRegistry: $(dockerRegistryServiceConnection)
              repository: $(imageRepository)
              command: 'push'
              tags: |
                $(tag)
                latest

  # ========== DEPLOY TO STAGING ==========
  - stage: DeployStaging
    displayName: 'Deploy to Staging'
    dependsOn: Build
    condition: and(succeeded(), eq(variables['Build.SourceBranch'], 'refs/heads/main'))
    jobs:
      - deployment: DeployStaging
        environment: 'staging'
        pool:
          vmImage: 'ubuntu-latest'
        strategy:
          runOnce:
            deploy:
              steps:
                - task: KubernetesManifest@0
                  displayName: 'Deploy to Staging K8s'
                  inputs:
                    action: 'deploy'
                    kubernetesServiceConnection: 'staging-k8s'
                    namespace: 'staging'
                    manifests: 'k8s/staging/*.yml'
                    containers: '$(containerRegistry)/$(imageRepository):$(tag)'

                # Run smoke tests against staging
                - script: |
                    echo "Running smoke tests against staging..."
                    curl -f https://staging.example.com/health || exit 1
                  displayName: 'Smoke tests'

  # ========== DEPLOY TO PRODUCTION ==========
  - stage: DeployProduction
    displayName: 'Deploy to Production'
    dependsOn: DeployStaging
    condition: succeeded()
    jobs:
      - deployment: DeployProduction
        environment: 'production'    # 👈 Has manual approval configured
        pool:
          vmImage: 'ubuntu-latest'
        strategy:
          runOnce:
            deploy:
              steps:
                - task: KubernetesManifest@0
                  displayName: 'Deploy to Production K8s'
                  inputs:
                    action: 'deploy'
                    kubernetesServiceConnection: 'production-k8s'
                    namespace: 'production'
                    manifests: 'k8s/production/*.yml'
                    containers: '$(containerRegistry)/$(imageRepository):$(tag)'

                - script: |
                    echo "Production deployment complete!"
                    echo "Image: $(containerRegistry)/$(imageRepository):$(tag)"
                  displayName: 'Deployment summary'
```

## 20. Pipeline Caching and Optimization

### Caching Dependencies

```yaml
# Cache node_modules
steps:
  - task: Cache@2
    inputs:
      key: 'npm | "$(Agent.OS)" | package-lock.json'
      path: '$(System.DefaultWorkingDirectory)/node_modules'
      restoreKeys: |
        npm | "$(Agent.OS)"
    displayName: 'Cache npm packages'

  - script: npm ci
    displayName: 'Install dependencies'
    condition: ne(variables['CacheRestored'], 'true')   # Skip if cache hit
```

```yaml
# Cache pip packages
- task: Cache@2
  inputs:
    key: 'pip | "$(Agent.OS)" | requirements.txt'
    path: '$(PIP_CACHE_DIR)'
    restoreKeys: |
      pip | "$(Agent.OS)"

# Cache Maven
- task: Cache@2
  inputs:
    key: 'maven | "$(Agent.OS)" | pom.xml'
    path: '$(MAVEN_CACHE_FOLDER)'
```

### Other Optimization Tips

```yaml
# 1. Shallow clone (faster checkout)
steps:
  - checkout: self
    fetchDepth: 1

# 2. Only run on relevant changes
trigger:
  paths:
    include:
      - src/**
    exclude:
      - docs/**
      - '**/*.md'

# 3. Parallel jobs (matrix strategy)
jobs:
  - job: Test
    strategy:
      matrix:
        Node16:
          nodeVersion: '16.x'
        Node18:
          nodeVersion: '18.x'
        Node20:
          nodeVersion: '20.x'
    steps:
      - task: NodeTool@0
        inputs:
          versionSpec: $(nodeVersion)
      - script: npm test

# 4. Conditional stages (skip deploy on PRs)
- stage: Deploy
  condition: and(succeeded(), ne(variables['Build.Reason'], 'PullRequest'))
```

## 21. Security in Azure Pipelines

### Secret Variables

```yaml
# Option 1: Define in UI (Settings → Pipelines → Variables)
# Mark as "secret" — value is masked in logs

# Option 2: Variable groups (Library)
variables:
  - group: production-secrets   # Contains DB_PASSWORD, API_KEY, etc.

steps:
  - script: echo "Using $(DB_PASSWORD)"
    # The actual value is masked in logs: ***
```

### Azure Key Vault Integration

```yaml
variables:
  - group: my-keyvault-group   # Linked to Azure Key Vault

# Or use the task:
steps:
  - task: AzureKeyVault@2
    inputs:
      azureSubscription: 'my-azure-sub'
      KeyVaultName: 'my-keyvault'
      SecretsFilter: 'db-password,api-key'
    # Secrets are now available as $(db-password) and $(api-key)
```

### Branch Policies

Protect your main branch:
1. Go to **Repos → Branches → main → ⋮ → Branch policies**
2. Enable:
   - **Require a minimum number of reviewers**
   - **Check for linked work items**
   - **Check for comment resolution**
   - **Build validation** — Require pipeline to pass before merge

### Secure Files

Upload certificates, SSH keys, or other sensitive files:
1. Go to **Pipelines → Library → Secure files**
2. Upload your file
3. Use in pipeline:

```yaml
steps:
  - task: DownloadSecureFile@1
    inputs:
      secureFile: 'my-certificate.pem'
    name: myCert

  - script: |
      echo "Certificate downloaded to: $(myCert.secureFilePath)"

```

## 23. Troubleshooting Common Issues

### Issue 1: "No hosted parallelism has been purchased or granted"

**Cause:** Free tier requires requesting free parallel jobs for new organizations.

**Fix:**
1. Go to [aka.ms/azpipelines-parallelism-request](https://aka.ms/azpipelines-parallelism-request)
2. Fill out the form to request free parallelism
3. Wait 2-3 business days for approval

### Issue 2: Pipeline doesn't trigger on push

**Fix:**
```yaml
# Make sure trigger is configured correctly
trigger:
  branches:
    include:
      - main
      - develop   # Is your branch listed here?

# Check if paths exclude is too aggressive
trigger:
  paths:
    exclude:
      - '**/*.md'   # This might exclude your changes!
```

### Issue 3: Variables not available in a later stage

**Fix:** Variables set in one job are NOT available in another job/stage unless you use **output variables**:

```yaml
stages:
  - stage: Build
    jobs:
      - job: BuildJob
        steps:
          - script: echo "##vso[task.setvariable variable=imageTag;isOutput=true]v1.2.3"
            name: setVar

  - stage: Deploy
    dependsOn: Build
    variables:
      imageTag: $[ stageDependencies.Build.BuildJob.outputs['setVar.imageTag'] ]
    jobs:
      - job: DeployJob
        steps:
          - script: echo "Deploying $(imageTag)"
```

### Issue 4: Docker build fails — "permission denied"

**Fix:** Make sure Docker service is enabled:
```yaml
pool:
  vmImage: 'ubuntu-latest'

# Docker is pre-installed on Microsoft-hosted agents
# Just make sure you're not trying to use Docker-in-Docker unnecessarily
```

### Issue 5: Agent times out

**Fix:**
```yaml
jobs:
  - job: LongJob
    timeoutInMinutes: 120    # Default is 60 minutes
    cancelTimeoutInMinutes: 5
    steps:
      - script: long-running-command
```

## 24. Best Practices

### Pipeline Design

1. **Use YAML pipelines** — Never use Classic (GUI) pipelines for new work
2. **Use templates** — Extract reusable steps into templates
3. **Use stages** — Separate build, test, and deploy
4. **Use environments** — For approval gates and deployment history
5. **Keep pipelines fast** — Cache dependencies, shallow clone, parallel jobs

### Security

6. **Never hardcode secrets** — Use variable groups or Key Vault
7. **Use service connections** — Don't store credentials in variables
8. **Enable branch policies** — Require PR reviews and build validation
9. **Limit who can edit pipelines** — Use pipeline permissions

### Organization

10. **One pipeline per app** — Don't put multiple apps in one pipeline
11. **Use naming conventions** — `app-name-ci`, `app-name-cd`
12. **Store templates in a central repo** — Share across projects
13. **Tag your builds** — Use semantic versioning in Docker tags

### Monitoring

14. **Publish test results** — Use `PublishTestResults` task
15. **Publish code coverage** — Track trends over time
16. **Set up notifications** — Email/Slack alerts for failures
17. **Review pipeline analytics** — Identify slow stages
