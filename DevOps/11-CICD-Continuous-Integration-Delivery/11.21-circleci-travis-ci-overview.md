## 1. What is CircleCI?

**CircleCI** is one of the most popular cloud-native CI/CD platforms. It's known for being **fast**, **flexible**, and **developer-friendly**.

Think of CircleCI as a CI/CD platform that was built from day one to be cloud-first. You connect your GitHub or Bitbucket repo, add a config file, and boom — you have a CI/CD pipeline.

### Quick Facts

| Property | Details |
|----------|---------|
| **Founded** | 2011 |
| **Config file** | `.circleci/config.yml` |
| **Supported VCS** | GitHub, Bitbucket, GitLab |
| **Free tier** | 6,000 build minutes/month, 30 jobs, 1 self-hosted runner |
| **Hosted/Self-hosted** | Both (CircleCI Cloud + CircleCI Server) |
| **Best for** | Teams wanting fast, cloud-native CI/CD |

### Why do teams love CircleCI?

- **Speed** — Powerful caching, parallel execution, Docker layer caching
- **Orbs** — Reusable pipeline packages (like GitHub Actions' "actions")
- **SSH debugging** — You can SSH into a running build to debug it live!
- **Docker-first** — Deep Docker integration (run jobs in any Docker image)
- **Insights** — Built-in analytics dashboard showing pipeline performance

## 2. CircleCI Key Features

| Feature | Description |
|---------|-------------|
| **Docker-first execution** | Every job runs in a Docker container by default |
| **Orbs** | Reusable, shareable pipeline packages |
| **Workflows** | Orchestrate jobs with dependencies, parallelism, and approvals |
| **Caching** | Dependency caching to speed up builds |
| **Docker Layer Caching (DLC)** | Cache Docker build layers between builds |
| **Parallelism** | Split tests across multiple containers |
| **SSH debugging** | SSH into a build for real-time debugging |
| **Contexts** | Shared environment variables across projects |
| **Resource classes** | Choose CPU/RAM for your build containers |
| **Self-hosted runners** | Run builds on your own infrastructure |
| **Insights dashboard** | Pipeline analytics and optimization suggestions |

## 3. CircleCI Architecture — How It Works

Here's the flow:

```
You push code to GitHub/Bitbucket/GitLab
    ↓
CircleCI detects the push (via webhooks)
    ↓
CircleCI reads .circleci/config.yml from your repo
    ↓
It creates execution environments (Docker containers, Linux VMs, macOS VMs)
    ↓
Jobs run inside these environments
    ↓
Results are reported back (pass/fail, artifacts, test results)
    ↓
Status is shown in your GitHub PR / commit
```

### Execution Environments

CircleCI offers 3 types of execution environments:

| Type | What It Is | Best For |
|------|-----------|----------|
| **Docker** | Jobs run inside Docker containers | Most builds (Node.js, Python, Java, etc.) |
| **Machine** | Jobs run on a full Linux VM | Docker-in-Docker, integration tests |
| **macOS** | Jobs run on macOS VMs | iOS/macOS app builds |

## 4. Getting Started with CircleCI

### Step 1: Sign Up

1. Go to [circleci.com](https://circleci.com)
2. Click **"Sign Up"** → Sign in with GitHub (or Bitbucket/GitLab)
3. Authorize CircleCI to access your repos

### Step 2: Set Up Your Project

1. Click **"Projects"** in the sidebar
2. Find your repository → Click **"Set Up Project"**
3. CircleCI may suggest a config template based on your project type

### Step 3: Create the Config File

Create the file `.circleci/config.yml` in your repository:

```yaml
# .circleci/config.yml — My First CircleCI Pipeline

version: 2.1

jobs:
  build-and-test:
    docker:
      - image: cimg/node:18.0    # CircleCI's convenience Node.js image
    steps:
      - checkout                  # Checkout code from repo
      - run:
          name: Install dependencies
          command: npm install
      - run:
          name: Run tests
          command: npm test

workflows:
  main:
    jobs:
      - build-and-test
```

### Step 4: Push and Watch

```bash
mkdir -p .circleci
# Create the config.yml file shown above
git add .circleci/config.yml
git commit -m "Add CircleCI config"
git push
```

Go to the CircleCI dashboard — you'll see your pipeline running!

## 5. CircleCI Configuration Deep Dive

Let's understand the structure of `.circleci/config.yml`:

```yaml
version: 2.1                    # Config version (always use 2.1)

# ===== ORBS =====
orbs:                            # Reusable packages
  node: circleci/node@5.2

# ===== COMMANDS =====
commands:                        # Reusable step sequences
  install-deps:
    steps:
      - run: npm ci

# ===== EXECUTORS =====
executors:                       # Reusable execution environments
  node-executor:
    docker:
      - image: cimg/node:18.0
    working_directory: ~/project

# ===== JOBS =====
jobs:                            # Units of work
  build:
    executor: node-executor
    steps:
      - checkout
      - install-deps
      - run: npm run build

  test:
    executor: node-executor
    steps:
      - checkout
      - install-deps
      - run: npm test

# ===== WORKFLOWS =====
workflows:                       # Orchestration
  build-test-deploy:
    jobs:
      - build
      - test:
          requires:
            - build              # Test runs AFTER build
```

### The hierarchy:

```
config.yml
  ├── version (2.1)
  ├── orbs (optional — import reusable packages)
  ├── commands (optional — reusable step sequences)
  ├── executors (optional — reusable environment definitions)
  ├── jobs (the actual work)
  │     ├── job1
  │     │     ├── executor (Docker, machine, macOS)
  │     │     └── steps (checkout, run, save_cache, etc.)
  │     └── job2
  └── workflows (orchestration — order, dependencies, filters)
```

## 6. Jobs, Steps, and Commands

### Jobs

A **job** is a collection of steps that runs in a single execution environment.

```yaml
jobs:
  my-job:
    docker:
      - image: cimg/node:18.0      # Primary container
      - image: cimg/postgres:15.0   # Service container (database)
        environment:
          POSTGRES_USER: testuser
          POSTGRES_DB: testdb
          POSTGRES_PASSWORD: testpass
    working_directory: ~/my-project
    steps:
      - checkout
      - run: npm install
      - run: npm test
```

**Key points:**
- The **first image** is the primary container where your steps run
- Additional images are **service containers** (databases, Redis, etc.)
- `working_directory` sets where the code is checked out

### Steps

Steps are the individual actions within a job:

```yaml
steps:
  # Checkout code
  - checkout

  # Run a shell command
  - run:
      name: Install dependencies
      command: npm ci
      working_directory: ~/project/frontend   # Override working directory

  # Run with environment variables
  - run:
      name: Run tests
      command: npm test
      environment:
        NODE_ENV: test
        CI: true

  # Run with conditional (only on main branch)
  - run:
      name: Deploy
      command: ./deploy.sh
      when: on_success    # on_success, on_fail, always

  # Multi-line command
  - run:
      name: Complex setup
      command: |
        echo "Setting up..."
        mkdir -p dist
        npm run build
        echo "Build complete!"

  # Background step (doesn't block next steps)
  - run:
      name: Start server
      command: npm start
      background: true

  # Save cache
  - save_cache:
      key: deps-v1-{{ checksum "package-lock.json" }}
      paths:
        - node_modules

  # Restore cache
  - restore_cache:
      keys:
        - deps-v1-{{ checksum "package-lock.json" }}
        - deps-v1-   # Fallback key

  # Store artifacts (available for download after build)
  - store_artifacts:
      path: dist
      destination: build-output

  # Store test results (for CircleCI's test insights)
  - store_test_results:
      path: test-results
```

### Commands (Reusable Steps)

Commands let you define reusable step sequences:

```yaml
commands:
  install-and-build:
    description: "Install dependencies and build the app"
    parameters:
      node-version:
        type: string
        default: "18.x"
    steps:
      - run:
          name: Install Node.js << parameters.node-version >>
          command: |
            echo "Using Node.js << parameters.node-version >>"
      - restore_cache:
          keys:
            - deps-v1-{{ checksum "package-lock.json" }}
      - run: npm ci
      - save_cache:
          key: deps-v1-{{ checksum "package-lock.json" }}
          paths:
            - node_modules
      - run: npm run build

jobs:
  build:
    docker:
      - image: cimg/node:18.0
    steps:
      - checkout
      - install-and-build:    # 👈 Use the command
          node-version: "18.x"
```

## 7. Workflows — Orchestrating Jobs

**Workflows** define the order in which jobs run, their dependencies, and conditions.

### Sequential Workflow

```yaml
workflows:
  build-test-deploy:
    jobs:
      - build
      - test:
          requires:
            - build           # Test runs AFTER build
      - deploy:
          requires:
            - test            # Deploy runs AFTER test
```

Visualized:
```
build → test → deploy
```

### Parallel Workflow

```yaml
workflows:
  build-test-deploy:
    jobs:
      - build
      - unit-test:
          requires:
            - build
      - integration-test:
          requires:
            - build
      - deploy:
          requires:
            - unit-test
            - integration-test   # Wait for BOTH
```

Visualized:
```
         ┌→ unit-test ─────┐
build ───┤                  ├→ deploy
         └→ integration-test┘
```

### Fan-out / Fan-in

```yaml
workflows:
  build-and-test:
    jobs:
      - build
      - test-firefox:
          requires: [build]
      - test-chrome:
          requires: [build]
      - test-safari:
          requires: [build]
      - deploy:
          requires:
            - test-firefox
            - test-chrome
            - test-safari
```

### Manual Approval

```yaml
workflows:
  deploy-workflow:
    jobs:
      - build
      - test:
          requires: [build]
      - hold-for-approval:           # 👈 Manual gate
          type: approval
          requires: [test]
      - deploy-production:
          requires: [hold-for-approval]
```

When the pipeline reaches `hold-for-approval`, it **pauses** and shows an "Approve" button in the CircleCI UI. Someone must click it to continue.

### Branch Filtering

```yaml
workflows:
  build-test-deploy:
    jobs:
      - build                        # Runs on ALL branches
      - test:
          requires: [build]           # Runs on ALL branches
      - deploy-staging:
          requires: [test]
          filters:
            branches:
              only: develop           # Only on develop branch
      - deploy-production:
          requires: [test]
          filters:
            branches:
              only: main              # Only on main branch
```

### Tag Filtering

```yaml
workflows:
  release:
    jobs:
      - build:
          filters:
            tags:
              only: /^v.*/            # Only tags like v1.0, v2.3.1
            branches:
              ignore: /.*/            # Ignore all branches
      - publish:
          requires: [build]
          filters:
            tags:
              only: /^v.*/
            branches:
              ignore: /.*/
```

### Scheduled Workflows

```yaml
workflows:
  nightly:
    triggers:
      - schedule:
          cron: "0 2 * * *"           # Every day at 2 AM UTC
          filters:
            branches:
              only: main
    jobs:
      - full-test-suite
      - security-scan
```

## 8. Executors — Where Jobs Run

### Docker Executor (Most Common)

```yaml
jobs:
  build:
    docker:
      # Primary container (your code runs here)
      - image: cimg/node:18.0
        auth:
          username: $DOCKERHUB_USERNAME
          password: $DOCKERHUB_PASSWORD

      # Service containers (accessible via localhost)
      - image: cimg/postgres:15.0
        environment:
          POSTGRES_DB: mydb
          POSTGRES_PASSWORD: secret

      - image: cimg/redis:7.0
    steps:
      - checkout
      - run: npm test   # Can connect to postgres on localhost:5432
```

### CircleCI Convenience Images

CircleCI provides optimized Docker images prefixed with `cimg/`:

| Image | Language/Tool |
|-------|--------------|
| `cimg/node:18.0` | Node.js 18 |
| `cimg/python:3.11` | Python 3.11 |
| `cimg/go:1.21` | Go 1.21 |
| `cimg/ruby:3.2` | Ruby 3.2 |
| `cimg/openjdk:17.0` | Java 17 |
| `cimg/rust:1.73` | Rust 1.73 |
| `cimg/base:current` | Basic Ubuntu (for custom setups) |
| `cimg/postgres:15.0` | PostgreSQL 15 |
| `cimg/redis:7.0` | Redis 7 |
| `cimg/mysql:8.0` | MySQL 8 |

### Machine Executor (Full VM)

Use when you need full control or Docker-in-Docker:

```yaml
jobs:
  integration-test:
    machine:
      image: ubuntu-2204:current    # Full Ubuntu VM
    steps:
      - checkout
      - run: docker compose up -d
      - run: docker compose exec app npm test
      - run: docker compose down
```

### macOS Executor

```yaml
jobs:
  build-ios:
    macos:
      xcode: "14.2.0"              # Specific Xcode version
    steps:
      - checkout
      - run: xcodebuild -scheme MyApp -destination 'platform=iOS Simulator,name=iPhone 14'
```

### Reusable Executors

```yaml
executors:
  node-18:
    docker:
      - image: cimg/node:18.0
    working_directory: ~/project
    environment:
      NODE_ENV: test

  python-311:
    docker:
      - image: cimg/python:3.11
    working_directory: ~/project

jobs:
  build-node:
    executor: node-18              # 👈 Use the executor
    steps:
      - checkout
      - run: npm ci

  build-python:
    executor: python-311           # 👈 Use the executor
    steps:
      - checkout
      - run: pip install -r requirements.txt
```

### Resource Classes

Control how much CPU and RAM your jobs get:

```yaml
jobs:
  build:
    docker:
      - image: cimg/node:18.0
    resource_class: large           # More CPU and RAM

# Available resource classes for Docker:
# small    - 1 vCPU, 2GB RAM
# medium   - 2 vCPU, 4GB RAM (default)
# medium+  - 3 vCPU, 6GB RAM
# large    - 4 vCPU, 8GB RAM
# xlarge   - 8 vCPU, 16GB RAM
# 2xlarge  - 16 vCPU, 32GB RAM
```

## 9. Orbs — Reusable Pipeline Packages

**Orbs** are CircleCI's killer feature. They're reusable, shareable packages of pipeline configuration — like npm packages but for CI/CD.

### What's in an Orb?

An orb can contain:
- **Commands** — reusable step sequences
- **Jobs** — complete job definitions
- **Executors** — reusable execution environments

### Using Orbs

```yaml
version: 2.1

orbs:
  node: circleci/node@5.2          # Official Node.js orb
  docker: circleci/docker@2.4      # Official Docker orb
  k8s: circleci/kubernetes@1.3     # Official Kubernetes orb
  slack: circleci/slack@4.12       # Official Slack notification orb

jobs:
  build:
    executor: node/default         # 👈 Use the orb's executor
    steps:
      - checkout
      - node/install-packages      # 👈 Use the orb's command (handles caching!)
      - run: npm test

workflows:
  main:
    jobs:
      - build
      - docker/publish:            # 👈 Use the orb's job
          image: myuser/myapp
          tag: ${CIRCLE_SHA1}
```

### Popular Orbs

| Orb | What It Does |
|-----|-------------|
| `circleci/node` | Install Node.js, cache packages, run tests |
| `circleci/python` | Install Python, cache pip packages |
| `circleci/docker` | Build, push Docker images |
| `circleci/kubernetes` | Deploy to Kubernetes |
| `circleci/aws-cli` | Configure AWS CLI |
| `circleci/aws-ecr` | Push to Amazon ECR |
| `circleci/aws-ecs` | Deploy to Amazon ECS |
| `circleci/gcp-gcr` | Push to Google Container Registry |
| `circleci/slack` | Send Slack notifications |
| `circleci/jira` | Integrate with Jira |
| `circleci/terraform` | Run Terraform commands |
| `circleci/helm` | Deploy with Helm |

### Example: Using the Node Orb

Without the orb (manual):
```yaml
jobs:
  build:
    docker:
      - image: cimg/node:18.0
    steps:
      - checkout
      - restore_cache:
          keys:
            - deps-v1-{{ checksum "package-lock.json" }}
      - run: npm ci
      - save_cache:
          key: deps-v1-{{ checksum "package-lock.json" }}
          paths:
            - node_modules
      - run: npm test
```

With the orb (much simpler!):
```yaml
orbs:
  node: circleci/node@5.2

jobs:
  build:
    executor: node/default
    steps:
      - checkout
      - node/install-packages    # 👈 Handles caching automatically!
      - run: npm test
```

The orb handles all the caching logic for you. That's the beauty of orbs!

Visit the [CircleCI Orb Registry](https://circleci.com/developer/orbs) to browse available orbs.

## 10. Caching and Artifacts

### Dependency Caching

```yaml
steps:
  # Restore cache (try exact match first, then fallback)
  - restore_cache:
      keys:
        - v1-deps-{{ checksum "package-lock.json" }}   # Exact match
        - v1-deps-                                       # Fallback prefix

  - run: npm ci

  # Save cache
  - save_cache:
      key: v1-deps-{{ checksum "package-lock.json" }}
      paths:
        - node_modules

# For Python:
  - restore_cache:
      keys:
        - v1-pip-{{ checksum "requirements.txt" }}
  - run: pip install -r requirements.txt
  - save_cache:
      key: v1-pip-{{ checksum "requirements.txt" }}
      paths:
        - ~/.cache/pip
```

### Cache Key Templates

| Template | Description |
|----------|-------------|
| `{{ checksum "file" }}` | SHA256 hash of the file |
| `{{ .Branch }}` | Current branch name |
| `{{ .Revision }}` | Git commit SHA |
| `{{ epoch }}` | Current time in seconds |
| `{{ .Environment.variableName }}` | Environment variable value |

### Docker Layer Caching (DLC)

This caches Docker build layers between runs — **huge time saver** when building Docker images:

```yaml
jobs:
  build-docker:
    machine:
      image: ubuntu-2204:current
      docker_layer_caching: true     # 👈 Enable DLC
    steps:
      - checkout
      - run: docker build -t myapp .  # Layers are cached!
      - run: docker push myapp
```

> **Note:** DLC is a paid feature.

### Storing Artifacts

Artifacts are files available for download after the build:

```yaml
steps:
  - run: npm run build
  - run: npm test -- --reporter=junit --output=test-results/

  # Store build output as artifact
  - store_artifacts:
      path: dist
      destination: build             # Available at: Artifacts → build/

  # Store test results (CircleCI analyzes these)
  - store_test_results:
      path: test-results             # Must be JUnit XML format
```

### Passing Data Between Jobs with Workspaces

```yaml
jobs:
  build:
    docker:
      - image: cimg/node:18.0
    steps:
      - checkout
      - run: npm ci
      - run: npm run build
      - persist_to_workspace:        # 👈 Save to workspace
          root: .
          paths:
            - dist
            - node_modules

  test:
    docker:
      - image: cimg/node:18.0
    steps:
      - checkout
      - attach_workspace:            # 👈 Restore from workspace
          at: .
      - run: npm test                # node_modules and dist are already here!

  deploy:
    docker:
      - image: cimg/node:18.0
    steps:
      - attach_workspace:
          at: .
      - run: |
          echo "Deploying dist/ directory..."
          ls dist/

workflows:
  main:
    jobs:
      - build
      - test:
          requires: [build]
      - deploy:
          requires: [test]
```

**Workspace vs Cache vs Artifacts:**

| Feature | Purpose | Lifetime | Share Between |
|---------|---------|----------|---------------|
| **Workspace** | Pass data between jobs in same workflow | Single workflow run | Jobs in same workflow |
| **Cache** | Speed up repeated builds | 15 days | All workflow runs |
| **Artifacts** | Download build outputs | 30 days | Anyone with access |

## 11. Environment Variables and Contexts

### Setting Variables

**Method 1: In the config file**

```yaml
jobs:
  build:
    docker:
      - image: cimg/node:18.0
    environment:                      # Job-level variables
      NODE_ENV: test
      CI: true
    steps:
      - run:
          name: Run tests
          command: npm test
          environment:                # Step-level variables
            DATABASE_URL: "postgresql://localhost:5432/testdb"
```

**Method 2: In CircleCI project settings (for secrets)**

1. Go to your project in CircleCI
2. Click **Project Settings → Environment Variables**
3. Add variables like `AWS_ACCESS_KEY_ID`, `DOCKER_PASSWORD`, etc.

> These are **encrypted** and masked in logs. Perfect for secrets!

**Method 3: Contexts (shared across projects)**

1. Go to **Organization Settings → Contexts**
2. Create a context (e.g., `aws-production`)
3. Add variables to the context

```yaml
workflows:
  deploy:
    jobs:
      - deploy-production:
          context:
            - aws-production           # 👈 All variables in this context are available
            - docker-credentials
```

### CircleCI Built-in Variables

| Variable | Description |
|----------|-------------|
| `CIRCLE_BRANCH` | Current branch name |
| `CIRCLE_BUILD_NUM` | Build number |
| `CIRCLE_SHA1` | Full commit SHA |
| `CIRCLE_TAG` | Git tag (if triggered by tag) |
| `CIRCLE_PROJECT_REPONAME` | Repository name |
| `CIRCLE_PR_NUMBER` | Pull request number |
| `CIRCLE_WORKFLOW_ID` | Unique workflow run ID |
| `CIRCLE_JOB` | Current job name |

## 12. CircleCI with Docker and Kubernetes

### Building and Pushing Docker Images

```yaml
version: 2.1

orbs:
  docker: circleci/docker@2.4

jobs:
  build-and-push:
    machine:
      image: ubuntu-2204:current
      docker_layer_caching: true
    steps:
      - checkout
      - run:
          name: Build Docker image
          command: |
            docker build -t myregistry/myapp:${CIRCLE_SHA1} .
            docker tag myregistry/myapp:${CIRCLE_SHA1} myregistry/myapp:latest

      - run:
          name: Push to registry
          command: |
            echo "$DOCKER_PASSWORD" | docker login -u "$DOCKER_USERNAME" --password-stdin
            docker push myregistry/myapp:${CIRCLE_SHA1}
            docker push myregistry/myapp:latest
```

## 13. Advanced CircleCI Features

### Test Splitting (Parallelism)

Split your test suite across multiple containers to run faster:

```yaml
jobs:
  test:
    docker:
      - image: cimg/node:18.0
    parallelism: 4                     # 👈 Run 4 containers in parallel
    steps:
      - checkout
      - run: npm ci
      - run:
          name: Run tests
          command: |
            # CircleCI splits the test files across containers
            TESTS=$(circleci tests glob "tests/**/*.test.js" | circleci tests split --split-by=timings)
            npx jest $TESTS
      - store_test_results:
          path: test-results
```

CircleCI will:
1. Spin up 4 containers
2. Use historical timing data to evenly split tests
3. Each container runs ~25% of the tests
4. Total test time ≈ original time / 4

### SSH Debugging

When a build fails and you can't figure out why:

1. Click **"Rerun job with SSH"** in the CircleCI UI
2. CircleCI gives you an SSH command:
   ```bash
   ssh -p 12345 circleci@123.456.789.0
   ```
3. You're dropped into the **exact same container** where your build ran
4. Poke around, check logs, run commands, debug!

> This is **amazing** for debugging flaky tests or environment issues.

### Dynamic Configuration

Generate your pipeline config dynamically:

```yaml
# .circleci/config.yml
version: 2.1

setup: true                           # 👈 This is a setup config

orbs:
  continuation: circleci/continuation@1.0

jobs:
  generate-config:
    docker:
      - image: cimg/base:current
    steps:
      - checkout
      - run:
          name: Generate pipeline config
          command: |
            # Generate config based on what changed
            python .circleci/generate-config.py > generated-config.yml
      - continuation/continue:
          configuration_path: generated-config.yml
```

### Pipeline Parameters

```yaml
version: 2.1

  run-integration-tests:
    type: boolean
    default: false
  deploy-environment:
    type: string

            branches:
              only: main
```

## 15. What is Travis CI?

**Travis CI** was one of the **first** cloud-based CI/CD services. It pioneered the concept of "CI/CD as a service" and was hugely popular in the open-source community, especially between 2011–2020.

### Quick Facts

| Property | Details |
|----------|---------|
| **Founded** | 2011 |
| **Config file** | `.travis.yml` (in repo root) |
| **Supported VCS** | GitHub, Bitbucket, GitLab, Assembla |
| **Free tier** | Limited (was generous for open-source, now reduced) |
| **Two versions** | `travis-ci.org` (old, shut down) → `travis-ci.com` (current) |
| **Best for** | Simple projects, open-source (historically) |

### A Brief History

- **2011–2018:** Travis CI was THE go-to CI for open-source. Almost every GitHub project used it. The free tier for open-source was incredibly generous.
- **2018:** Acquired by Idera.
- **2020–2021:** Major changes to pricing. Free tier was significantly reduced. Many open-source projects migrated to **GitHub Actions**.
- **Today:** Still operational but usage has declined significantly. Many teams have moved to GitHub Actions, CircleCI, or GitLab CI.

> **Important:** We're covering Travis CI because you'll encounter it in many existing projects and it's good to understand its concepts. However, for new projects, consider GitHub Actions or CircleCI instead.

## 17. Travis CI Architecture

Travis CI's flow is straightforward:

```
You push code to GitHub
    ↓
GitHub sends a webhook to Travis CI
    ↓
Travis CI reads .travis.yml from your repo
    ↓
It spins up a clean VM (Linux, macOS, or Windows)
    ↓
It runs through the build lifecycle phases
    ↓
Reports results back to GitHub (green checkmark or red X)
```

### Build Environment

Each build gets a **fresh virtual machine**:
- **Linux:** Ubuntu (various versions)
- **macOS:** Xcode-configured macOS VMs
- **Windows:** Windows Server

The VM is **destroyed** after each build — completely clean slate every time.

## 20. Build Lifecycle — Phases

Travis CI builds go through specific **phases** in order. Understanding these is key:

```
1. apt addons       (install system packages)
2. cache            (restore cached dependencies)
3. before_install   (run commands before install)
4. install          (install dependencies)
5. before_script    (run commands before tests)
6. script           (run tests / build)
7. after_success    (run if script succeeds)    OR   after_failure (run if script fails)
8. before_deploy    (run before deployment)
9. deploy           (deploy to external service)
10. after_deploy    (run after deployment)
11. after_script    (always runs, regardless of result)
```

### Example Using All Phases

```yaml
language: node_js
node_js:
  - "18"

# System packages
addons:
  apt:
    packages:
      - libpng-dev

# Restore cached files
cache:
  directories:
    - node_modules

# Phase 1: Before install
before_install:
  - echo "Setting up environment..."
  - npm install -g npm@latest

# Phase 2: Install
install:
  - npm ci

# Phase 3: Before script
before_script:
  - echo "Starting database..."
  - psql -c 'CREATE DATABASE testdb;' -U postgres

# Phase 4: Script (the main build/test step)
script:
  - npm run lint
  - npm test
  - npm run build

# Phase 5a: After success (only if script passes)
after_success:
  - npm run coverage:upload
  - echo "All tests passed! 🎉"

# Phase 5b: After failure (only if script fails)
after_failure:
  - cat npm-debug.log
  - echo "Build failed 😢"

# Phase 6: Deploy
deploy:
  provider: heroku
  api_key:
    secure: "encrypted-api-key"
  app: my-production-app
  on:
    branch: main

# Phase 7: After script (always runs)
after_script:
  - echo "Build complete. Cleaning up..."
```

### Important Rules:

- If **`install`** fails → the build is **errored** (stops immediately)
- If **`script`** fails → the build is **failed** (but `after_failure` and `after_script` still run)
- **`after_success`** only runs if `script` succeeds
- **`after_failure`** only runs if `script` fails
- **`after_script`** always runs regardless

## 21. Build Matrix — Multi-Version Testing

This is one of Travis CI's **best features** — easily test across multiple versions:

### Simple Matrix

```yaml
language: node_js
node_js:
  - "16"
  - "18"
  - "20"

# This creates 3 builds, one for each Node.js version
```

### Multi-Dimension Matrix

```yaml
language: node_js
node_js:
  - "18"
  - "20"
os:
  - linux
  - osx

# This creates 4 builds:
# Node 18 on Linux
# Node 18 on macOS
# Node 20 on Linux
# Node 20 on macOS
```

### Custom Matrix with `env`

```yaml
language: node_js
node_js:
  - "18"

env:
  - DB=postgres TEST_SUITE=unit
  - DB=mysql TEST_SUITE=unit
  - DB=postgres TEST_SUITE=integration

# Creates 3 builds with different environment variables
```

### Matrix Include/Exclude

```yaml
language: node_js

# Explicit matrix entries
jobs:
  include:
    - node_js: "18"
      env: TEST_SUITE=unit
    - node_js: "20"
      env: TEST_SUITE=unit
    - node_js: "18"
      env: TEST_SUITE=integration
      os: linux

  # Exclude specific combinations
  exclude:
    - node_js: "16"
      os: osx

  # Allow some jobs to fail without failing the whole build
  allow_failures:
    - node_js: "21"        # Experimental, okay if it fails

  # Fast finish — don't wait for allowed_failures to complete
  fast_finish: true
```

### Stages (Sequential Groups of Jobs)

```yaml
jobs:
  include:
    # Stage 1: Lint (runs first)
    - stage: lint
      script: npm run lint

    # Stage 2: Test (runs after lint passes)
    - stage: test
      node_js: "18"
      script: npm test
    - stage: test
      node_js: "20"
      script: npm test

    # Stage 3: Deploy (runs after all tests pass)
    - stage: deploy
      script: skip     # No test to run
      deploy:
        provider: heroku
        api_key:
          secure: "encrypted-key"
        app: my-app
        on:
          branch: main

stages:
  - lint
  - test
  - name: deploy
    if: branch = main AND type != pull_request
```

```yaml
      name: "Deploy to Production"

        - echo "Deploying to production..."

          tags: true

# Notifications
notifications:
  slack:

      - secure: "encrypted-slack-webhook-url"
```

## 25. Current State of Travis CI

Let's be honest about where Travis CI stands today:

### The Good
- **Simple config** — `.travis.yml` is one of the easiest CI configs to write
- **Build matrix** — Still one of the best for multi-version testing
- **Built-in deploy providers** — Easy deployment to many services
- **Mature** — Well-documented, lots of Stack Overflow answers

### The Concerns
- **Reduced free tier** — Open-source projects no longer get unlimited free builds
- **Community migration** — Many popular open-source projects have moved to GitHub Actions
- **Slower innovation** — Not keeping up with competitors in terms of new features
- **Security incidents** — Had some security issues in 2021 that eroded trust

### Migration Paths

If you're moving away from Travis CI:

| Moving To | Difficulty | Notes |
|-----------|-----------|-------|
| **GitHub Actions** | Easy | Most similar model. GitHub provides a migration guide |
| **CircleCI** | Medium | Different config format but similar concepts |
| **GitLab CI** | Medium | Need to move repos to GitLab (or mirror them) |
| **Azure Pipelines** | Medium | More verbose config but very powerful |

### Quick Migration — Travis CI to GitHub Actions

Travis CI:
```yaml
# .travis.yml
language: node_js
node_js:
  - "18"
install:
  - npm ci
script:
  - npm test
deploy:
```

## 26. CircleCI vs Travis CI vs Others

| Feature | CircleCI | Travis CI | GitHub Actions | GitLab CI | Jenkins |
|---------|----------|-----------|----------------|-----------|---------|
| **Config file** | `.circleci/config.yml` | `.travis.yml` | `.github/workflows/*.yml` | `.gitlab-ci.yml` | `Jenkinsfile` |
| **Config complexity** | Medium | Simple | Medium | Medium | Complex |
| **Orbs/Reuse** | ✅ Orbs (excellent) | ❌ Limited | ✅ Actions marketplace | ✅ includes/extends | ✅ Shared libraries |
| **Free tier** | 6,000 min/mo | Limited | 2,000 min/mo (private) | 400 min/mo | Free (self-hosted) |
| **Docker support** | ✅ Excellent (native) | ✅ Good | ✅ Good | ✅ Excellent | ✅ Good (plugins) |
| **Build matrix** | ✅ Via workflows | ✅ Excellent (best) | ✅ Matrix strategy | ✅ parallel/matrix | ✅ Via scripting |
| **SSH debugging** | ✅ Yes! | ✅ Limited | ❌ No | ❌ No | ✅ Yes (agent) |
| **Parallelism** | ✅ Test splitting | ❌ No | ✅ Matrix | ✅ parallel keyword | ✅ Distributed builds |
| **Self-hosted** | ✅ CircleCI Server | ❌ No | ✅ Self-hosted runners | ✅ GitLab Runners | ✅ Always self-hosted |
| **Best at** | Speed, Docker, Orbs | Simplicity, Matrix | GitHub integration | All-in-one platform | Maximum flexibility |
| **Popularity (2024+)** | High | Declining | Very High | High | High |

## 27. When to Use What?

### Choose CircleCI when:
- You want the **fastest** CI/CD builds
- You use Docker heavily
- You want **Orbs** for reusable config
- You need **SSH debugging**
- You want powerful **test splitting** (parallelism)
- Your code is on GitHub, Bitbucket, or GitLab

### Choose Travis CI when:
- You have a **simple project** that just needs basic CI
- You need easy **multi-version testing** (build matrix)
- You have an existing project already using Travis CI
- You want the **simplest possible config**

### Choose GitHub Actions when:
- Your code is on **GitHub** (obvious choice)
- You want tight integration with GitHub PRs, issues, releases
- You want access to the **Actions marketplace**
- You want a good free tier (2,000 min/month for private repos)

### Choose GitLab CI when:
- Your code is on **GitLab**
- You want an **all-in-one** platform (repos + CI/CD + registry + monitoring)
- You want **Auto DevOps** (zero-config CI/CD)

### Choose Jenkins when:
- You need **maximum flexibility** and control
- You want to run everything **on-premises**
- You have complex, enterprise-grade requirements
- You don't mind managing your own infrastructure

### Choose Azure Pipelines when:
- You're in a **Microsoft/Azure** ecosystem
- You need **multi-OS** builds (Linux + Windows + macOS)
- You want powerful **templates** for large organizations
- You use Azure services for deployment

## 28. Summary

### CircleCI — Key Takeaways

| Concept | Description |
|---------|-------------|
| **Config file** | `.circleci/config.yml` |
| **Jobs** | Units of work running in Docker/VM/macOS |
| **Steps** | Individual actions (checkout, run, save_cache, etc.) |
| **Workflows** | Orchestrate jobs with dependencies and approvals |
| **Orbs** | Reusable pipeline packages (CircleCI's superpower) |
| **Executors** | Docker, Machine (VM), macOS |
| **Workspaces** | Share data between jobs |
| **Caching** | Speed up builds by caching dependencies |
| **Contexts** | Share secrets across projects |
| **SSH debugging** | SSH into running builds for debugging |

### Travis CI — Key Takeaways

| Concept | Description |
|---------|-------------|
| **Config file** | `.travis.yml` |
| **Build lifecycle** | before_install → install → script → deploy |
| **Build matrix** | Test across multiple language versions/OS easily |
| **Stages** | Sequential groups of parallel jobs |
| **Deploy providers** | Built-in deployment to Heroku, AWS, npm, etc. |
| **Simplicity** | Easiest CI config to learn |
| **Current status** | Declining. Consider alternatives for new projects |

### Quick Start Templates

**CircleCI:**
```yaml
version: 2.1
orbs:
  node: circleci/node@5.2
jobs:
  build-and-test:
    executor: node/default
    steps:
      - checkout
      - node/install-packages
      - run: npm test
workflows:
  main:
    jobs:
      - build-and-test
```

**Travis CI:**
```yaml
language: node_js
node_js:
  - "18"
script:
  - npm test
```
