## 📌 What is Tekton?

Tekton is an **open-source framework** for building CI/CD pipelines that run **natively on Kubernetes**. Instead of using a separate CI/CD server like Jenkins, Tekton turns your Kubernetes cluster itself into a CI/CD engine.

Think of it this way:

- **Jenkins** = A separate server that runs your pipelines  
- **Tekton** = Your Kubernetes cluster IS the pipeline runner

Tekton was originally created by Google as part of the **Knative** project, and later became its own standalone project under the **Continuous Delivery Foundation (CDF)** — the same foundation that backs Jenkins and Spinnaker.

### Why is Tekton Called "Cloud-Native"?

Because everything in Tekton is a **Kubernetes resource**. Tasks, Pipelines, Runs — they're all defined as Kubernetes YAML manifests (Custom Resource Definitions / CRDs). If you know Kubernetes, you already know how to work with Tekton.

---

## 🤔 Why Use Tekton? (When We Already Have Jenkins, GitHub Actions, etc.)

Great question! Here's why Tekton exists and when it shines:

| Reason | Explanation |
|--------|-------------|
| **Kubernetes-native** | Everything runs as Pods. No separate CI/CD server to maintain |
| **Portable** | Same pipeline runs on any Kubernetes cluster — EKS, GKE, AKS, on-prem |
| **Scalable** | Kubernetes handles scaling automatically. Need 100 parallel builds? K8s got it |
| **Serverless-style** | No always-on server. Pipelines spin up Pods, run, and die. Zero idle cost |
| **Reusable** | Tasks can be shared across teams/orgs via Tekton Hub |
| **Vendor-neutral** | Not locked into GitHub, GitLab, or any specific vendor |
| **Extensible** | Build custom tasks in any language/container image |

### When Should You Use Tekton?

✅ You're already running workloads on Kubernetes  
✅ You want your CI/CD to live in the same cluster as your apps  
✅ You need a vendor-neutral, portable CI/CD solution  
✅ You want fine-grained control over every step in your pipeline  
✅ You're building an internal developer platform  

### When Should You NOT Use Tekton?

❌ You're not using Kubernetes (Tekton requires K8s)  
❌ You want a simple, quick-to-set-up CI/CD (GitHub Actions is simpler)  
❌ You need a rich UI out of the box (Tekton's UI is basic)  
❌ Your team is small and doesn't need the complexity

## 🏗️ Tekton Architecture — The Big Picture

Tekton works by installing **Custom Resource Definitions (CRDs)** into your Kubernetes cluster. These CRDs define new resource types like `Task`, `Pipeline`, `PipelineRun`, etc.

Here's the architecture at a glance:

```
┌──────────────────────────────────────────────────────┐
│                  Kubernetes Cluster                    │
│                                                        │
│  ┌──────────────────────────────────────────────────┐ │
│  │           Tekton Pipelines Controller             │ │
│  │  (Watches for TaskRuns/PipelineRuns and           │ │
│  │   creates Pods to execute them)                   │ │
│  └──────────────────────────────────────────────────┘ │
│                                                        │
│  ┌──────────┐  ┌──────────┐  ┌──────────────────────┐│
│  │   Task   │  │ Pipeline │  │  PipelineRun /       ││
│  │  (CRD)   │  │  (CRD)   │  │  TaskRun (CRD)      ││
│  └──────────┘  └──────────┘  └──────────────────────┘│
│                                                        │
│  ┌──────────────────────────────────────────────────┐ │
│  │          Pods (Actual Execution)                   │ │
│  │  Each Step in a Task = One Container in a Pod     │ │
│  └──────────────────────────────────────────────────┘ │
│                                                        │
│  ┌──────────────────────────────────────────────────┐ │
│  │     Tekton Dashboard (Optional Web UI)            │ │
│  └──────────────────────────────────────────────────┘ │
│                                                        │
│  ┌──────────────────────────────────────────────────┐ │
│  │     Tekton Triggers (Webhooks / Event Listeners)  │ │
│  └──────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────┘
```

## 🧱 Tekton Building Blocks — Core Concepts

Tekton has a clear hierarchy of concepts. Let's learn them from the bottom up:

### 1. Step

A **Step** is the **smallest unit of work** in Tekton. It's a single command or script that runs inside a **container**.

Think of a Step as: *"Run this one specific thing."*

```yaml
steps:
  - name: say-hello
    image: ubuntu
    script: |
      echo "Hello from Tekton!"
```

Each Step runs in its own container within the same Pod.

---

### 2. Task

A **Task** is a **collection of Steps** that run sequentially in a single Pod. All Steps in a Task share the same Pod (same workspace, same volumes).

Think of a Task as: *"A single job — like 'build the code' or 'run the tests'."*

```yaml
apiVersion: tekton.dev/v1
kind: Task
metadata:
  name: build-and-test
spec:
  steps:
    - name: build
      image: node:18
      script: |
        npm install
        npm run build
    - name: test
      image: node:18
      script: |
        npm test
```

**Key points about Tasks:**
- Steps within a Task run **sequentially** (one after another)
- All Steps share the same **Pod** and can share files via workspaces
- Tasks are **reusable** — define once, use in many Pipelines

---

### 3. TaskRun

A **TaskRun** is what happens when you **actually execute** a Task. It's an instance of a Task.

Think of it like this:
- **Task** = The recipe (instructions)
- **TaskRun** = Actually cooking the dish (execution)

```yaml
apiVersion: tekton.dev/v1
kind: TaskRun
metadata:
  name: build-and-test-run-1
spec:
  taskRef:
    name: build-and-test
```

When you create a TaskRun, Tekton's controller:
1. Sees the TaskRun
2. Looks up the referenced Task
3. Creates a Pod with containers for each Step
4. Runs the Steps in order
5. Reports the result (success/failure)

---

### 4. Pipeline

A **Pipeline** is a **collection of Tasks** that run in a defined order (sequentially or in parallel).

Think of a Pipeline as: *"The full workflow — build, test, deploy — with multiple tasks chained together."*

```yaml
apiVersion: tekton.dev/v1
kind: Pipeline
metadata:
  name: my-cicd-pipeline
spec:
  tasks:
    - name: fetch-code
      taskRef:
        name: git-clone
    - name: build
      taskRef:
        name: build-app
      runAfter:
        - fetch-code
    - name: test
      taskRef:
        name: run-tests
      runAfter:
        - build
    - name: deploy
      taskRef:
        name: deploy-app
      runAfter:
        - test
```

**Key points about Pipelines:**
- `runAfter` controls the order (dependency graph)
- Tasks without `runAfter` can run **in parallel**
- Each Task runs in its **own Pod**

---

### 5. PipelineRun

A **PipelineRun** is an execution of a Pipeline — just like TaskRun is to Task.

```yaml
apiVersion: tekton.dev/v1
kind: PipelineRun
metadata:
  name: my-cicd-pipeline-run-1
spec:
  pipelineRef:
    name: my-cicd-pipeline
  workspaces:
    - name: shared-workspace
      volumeClaimTemplate:
        spec:
          accessModes:
            - ReadWriteOnce
          resources:
            requests:
              storage: 1Gi
```

---

### Quick Summary: The Hierarchy

```
PipelineRun
  └── Pipeline
        ├── Task 1
        │     ├── Step 1 (container)
        │     ├── Step 2 (container)
        │     └── Step 3 (container)
        ├── Task 2
        │     ├── Step 1
        │     └── Step 2
        └── Task 3
              └── Step 1
```

| Concept | What It Is | Analogy |
|---------|-----------|---------|
| **Step** | A single command in a container | A single instruction in a recipe |
| **Task** | A group of Steps (one Pod) | A complete recipe |
| **TaskRun** | An execution of a Task | Actually cooking the recipe |
| **Pipeline** | A group of Tasks in order | A full meal plan (multiple recipes) |
| **PipelineRun** | An execution of a Pipeline | Actually preparing the full meal |

## 🔧 Installing Tekton

### Prerequisites

- A running Kubernetes cluster (Minikube, Kind, EKS, GKE, etc.)
- `kubectl` configured and connected to your cluster

### Step 1: Install Tekton Pipelines

```bash
# Install the latest stable release of Tekton Pipelines
kubectl apply --filename https://storage.googleapis.com/tekton-releases/pipeline/latest/release.yaml
```

This installs:
- Tekton CRDs (Task, Pipeline, TaskRun, PipelineRun, etc.)
- Tekton Pipelines Controller
- Tekton Webhook (for validation)

### Step 2: Verify Installation

```bash
# Check that Tekton pods are running
kubectl get pods -n tekton-pipelines

# You should see something like:
# NAME                                          READY   STATUS    RESTARTS   AGE
# tekton-pipelines-controller-xxx               1/1     Running   0          30s
# tekton-pipelines-webhook-xxx                  1/1     Running   0          30s
```

### Step 3: Install Tekton CLI (`tkn`)

The `tkn` CLI makes it easy to work with Tekton from the command line.

```bash
# On macOS (Homebrew)
brew install tektoncd-cli

# On Linux
curl -LO https://github.com/tektoncd/cli/releases/latest/download/tkn_Linux_x86_64.tar.gz
tar xvzf tkn_Linux_x86_64.tar.gz -C /usr/local/bin/ tkn

# On Windows (Chocolatey)
choco install tektoncd-cli
```

Verify:

```bash
tkn version
# Client version: 0.x.x
# Pipeline version: v0.x.x
```

### Step 4: Install Tekton Dashboard (Optional)

```bash
kubectl apply --filename https://storage.googleapis.com/tekton-releases/dashboard/latest/release.yaml

# Access it via port-forward
kubectl port-forward -n tekton-pipelines svc/tekton-dashboard 9097:9097

# Open http://localhost:9097 in your browser
```

### Step 5: Install Tekton Triggers (Optional but Recommended)

Triggers let you automatically start PipelineRuns from events like Git pushes.

```bash
kubectl apply --filename https://storage.googleapis.com/tekton-releases/triggers/latest/release.yaml
kubectl apply --filename https://storage.googleapis.com/tekton-releases/triggers/latest/interceptors.yaml
```

## 🚀 Hands-On: Your First Tekton Pipeline (Step by Step)

Let's build a complete CI/CD pipeline that:
1. Clones a Git repository
2. Builds the application
3. Runs tests
4. Deploys (simulated)

### Step 1: Create a "Hello World" Task

```yaml
# hello-task.yaml
apiVersion: tekton.dev/v1
kind: Task
metadata:
  name: hello
spec:
  steps:
    - name: greet
      image: alpine:3.18
      script: |
        #!/bin/sh
        echo "========================================="
        echo "  Hello from Tekton! 🚀"
        echo "  This is my first Tekton Task!"
        echo "========================================="
```

Apply it:

```bash
kubectl apply -f hello-task.yaml
# task.tekton.dev/hello created

# Verify
kubectl get tasks
# NAME    AGE
# hello   5s
```

### Step 2: Run the Task

```yaml
# hello-taskrun.yaml
apiVersion: tekton.dev/v1
kind: TaskRun
metadata:
  name: hello-run-1
spec:
  taskRef:
    name: hello
```

Apply and watch:

```bash
kubectl apply -f hello-taskrun.yaml

# Watch the TaskRun
kubectl get taskruns
# NAME          SUCCEEDED   REASON      STARTTIME   COMPLETIONTIME
```

## 📥 Parameters — Making Tasks Dynamic

Hardcoded values are no fun. Let's make Tasks accept **parameters**.

### Task with Parameters

```yaml
# greet-task.yaml
apiVersion: tekton.dev/v1
kind: Task
metadata:
  name: greet
spec:
  params:
    - name: person
      type: string
      description: "Who to greet"
      default: "World"
    - name: greeting
      type: string
      description: "The greeting message"
      default: "Hello"
  steps:
    - name: greet
      image: alpine:3.18
      script: |
        #!/bin/sh
        echo "$(params.greeting), $(params.person)! Welcome to Tekton!"
```

**What's happening:**
- `params` defines the input parameters
- `$(params.person)` references a parameter inside a Step
- `default` provides a fallback value if not supplied

### TaskRun with Parameters

```yaml
# greet-taskrun.yaml
apiVersion: tekton.dev/v1
kind: TaskRun
metadata:
  name: greet-run-1
spec:
  taskRef:
    name: greet
  params:
    - name: person
      value: "DevOps Engineer"
    - name: greeting
      value: "Hey there"
```

Output:

```
[greet] Hey there, DevOps Engineer! Welcome to Tekton!
```

Or use the CLI:

```bash
tkn task start greet -p person="Ritesh" -p greeting="Namaste"
```

## 📂 Workspaces — Sharing Data Between Steps and Tasks

**Workspaces** are how Tekton handles shared storage. When you clone a repo in one Task and want to build it in another Task, you need a workspace to share the files.

Think of a workspace as a **shared folder** between Tasks.

### Defining Workspaces in a Task

```yaml
# clone-and-list-task.yaml
apiVersion: tekton.dev/v1
kind: Task
metadata:
  name: clone-and-list
spec:
  workspaces:
    - name: source
      description: "The workspace to clone the repo into"
  params:
    - name: repo-url
      type: string
  steps:
    - name: clone
      image: alpine/git:latest
      script: |
        git clone $(params.repo-url) $(workspaces.source.path)/repo
    - name: list-files
      image: alpine:3.18
      script: |
        echo "Files in the repository:"
        ls -la $(workspaces.source.path)/repo
```

### Providing Workspaces in a TaskRun

```yaml
# clone-and-list-taskrun.yaml
apiVersion: tekton.dev/v1
kind: TaskRun
metadata:
  name: clone-and-list-run
spec:
  taskRef:
    name: clone-and-list
  params:
    - name: repo-url
      value: "https://github.com/tektoncd/pipeline.git"
  workspaces:
    - name: source
      volumeClaimTemplate:
        spec:
          accessModes:
            - ReadWriteOnce
          resources:
            requests:
              storage: 1Gi
```

**Workspace types you can use:**

| Type | What It Is | Use Case |
|------|-----------|----------|
| `volumeClaimTemplate` | Creates a new PVC automatically | Most common — persistent storage |
| `persistentVolumeClaim` | Uses an existing PVC | When you have pre-created storage |
| `emptyDir` | Temporary in-memory or disk storage | Fast, but lost after the run |
| `configMap` | Mounts a ConfigMap as files | Configuration files |
| `secret` | Mounts a Secret as files | Credentials, keys |

## 🔗 Building a Complete Pipeline

Now let's put it all together — a real Pipeline with multiple Tasks:

### Task 1: Clone the Repository

```yaml
# task-git-clone.yaml
apiVersion: tekton.dev/v1
kind: Task
metadata:
  name: git-clone-custom
spec:
  workspaces:
    - name: output
  params:
    - name: url
      type: string
    - name: revision
      type: string
      default: "main"
  steps:
    - name: clone
      image: alpine/git:latest
      script: |
        cd $(workspaces.output.path)
        git clone -b $(params.revision) $(params.url) source
        echo "✅ Repository cloned successfully!"
        ls -la source/
```

### Task 2: Build the Application

```yaml
# task-build.yaml
apiVersion: tekton.dev/v1
kind: Task
metadata:
  name: build-app
spec:
  workspaces:
    - name: source
  steps:
    - name: build
      image: node:18-alpine
      script: |
        cd $(workspaces.source.path)/source
        echo "📦 Installing dependencies..."
        npm install 2>/dev/null || echo "No package.json found, skipping npm install"
        echo "🔨 Building application..."
        # npm run build  # Uncomment for real projects
        echo "✅ Build complete!"
```

### Task 3: Run Tests

```yaml
# task-test.yaml
apiVersion: tekton.dev/v1
kind: Task
metadata:
  name: run-tests
spec:
  workspaces:
    - name: source
  steps:
    - name: unit-tests
      image: node:18-alpine
      script: |
        cd $(workspaces.source.path)/source
        echo "🧪 Running unit tests..."
        # npm test  # Uncomment for real projects
        echo "✅ All tests passed!"
    - name: lint
      image: node:18-alpine
      script: |
        cd $(workspaces.source.path)/source
        echo "🔍 Running linter..."
        # npm run lint  # Uncomment for real projects
        echo "✅ Linting complete — no issues found!"
```

### Task 4: Deploy (Simulated)

```yaml
# task-deploy.yaml
apiVersion: tekton.dev/v1
kind: Task
metadata:
  name: deploy-app
spec:
  workspaces:
    - name: source
  params:
    - name: environment
      type: string
      default: "staging"
  steps:
    - name: deploy
      image: alpine:3.18
      script: |
        echo "🚀 Deploying to $(params.environment)..."
        echo "   Source: $(workspaces.source.path)/source"
        echo "   Target: $(params.environment) cluster"
        # kubectl apply -f k8s/  # Real deployment command
        echo "✅ Deployment to $(params.environment) successful!"
```

### The Pipeline — Connecting All Tasks

```yaml
# pipeline.yaml
apiVersion: tekton.dev/v1
kind: Pipeline
metadata:
  name: ci-cd-pipeline
spec:
  params:
    - name: repo-url
      type: string
      description: "The Git repository URL"
    - name: branch
      type: string
      default: "main"
    - name: deploy-env
      type: string
      default: "staging"
  workspaces:
    - name: shared-workspace
      description: "Workspace shared across all tasks"
  tasks:
    # Task 1: Clone the repository
    - name: clone
      taskRef:
        name: git-clone-custom
      workspaces:
        - name: output
          workspace: shared-workspace
      params:
        - name: url
          value: $(params.repo-url)
        - name: revision
          value: $(params.branch)

    # Task 2: Build (runs after clone)
    - name: build
      taskRef:
        name: build-app
      runAfter:
        - clone
      workspaces:
        - name: source
          workspace: shared-workspace

    # Task 3: Test (runs after build)
    - name: test
      taskRef:
        name: run-tests
      runAfter:
        - build
      workspaces:
        - name: source
          workspace: shared-workspace

    # Task 4: Deploy (runs after test)
    - name: deploy
      taskRef:
        name: deploy-app
      runAfter:
        - test
      workspaces:
        - name: source
          workspace: shared-workspace
      params:
        - name: environment
          value: $(params.deploy-env)
```

### Run the Pipeline

```yaml
# pipelinerun.yaml
apiVersion: tekton.dev/v1
kind: PipelineRun
metadata:
  name: ci-cd-pipeline-run-1
spec:
  pipelineRef:
    name: ci-cd-pipeline
  params:
    - name: repo-url
      value: "https://github.com/yourname/your-app.git"
    - name: branch
      value: "main"
    - name: deploy-env
      value: "staging"
  workspaces:
    - name: shared-workspace
      volumeClaimTemplate:
        spec:
          accessModes:
            - ReadWriteOnce
          resources:
            requests:
              storage: 2Gi
```

Apply everything:

```bash
# Apply all Tasks
kubectl apply -f task-git-clone.yaml
kubectl apply -f task-build.yaml
kubectl apply -f task-test.yaml
kubectl apply -f task-deploy.yaml

# Apply the Pipeline
kubectl apply -f pipeline.yaml

# Start the PipelineRun
kubectl apply -f pipelinerun.yaml

# Watch the progress
tkn pipelinerun logs ci-cd-pipeline-run-1 -f
```

Or use the CLI to start it directly:

```bash
tkn pipeline start ci-cd-pipeline \
  -p repo-url="https://github.com/yourname/your-app.git" \
  -p branch="main" \
  -p deploy-env="staging" \
  -w name=shared-workspace,volumeClaimTemplateFile=workspace-template.yaml
```

## ⚡ Parallel Tasks in Tekton

One of Tekton's strengths is running Tasks in parallel. If two Tasks don't depend on each other, they can run at the same time.

```yaml
apiVersion: tekton.dev/v1
kind: Pipeline
metadata:
  name: parallel-pipeline
spec:
  workspaces:
    - name: shared-workspace
  tasks:
    - name: clone
      taskRef:
        name: git-clone-custom
      workspaces:
        - name: output
          workspace: shared-workspace

    # These two run IN PARALLEL (both depend only on 'clone')
    - name: unit-tests
      taskRef:
        name: run-unit-tests
      runAfter:
        - clone
      workspaces:
        - name: source
          workspace: shared-workspace

    - name: integration-tests
      taskRef:
        name: run-integration-tests
      runAfter:
        - clone
      workspaces:
        - name: source
          workspace: shared-workspace

    # This runs AFTER both test tasks complete
    - name: deploy
      taskRef:
        name: deploy-app
      runAfter:
        - unit-tests
        - integration-tests
      workspaces:
        - name: source
          workspace: shared-workspace
```

Visual representation:

```
                  ┌─────────────────┐
                  │      clone      │
                  └────────┬────────┘
                           │
              ┌────────────┴────────────┐
              ▼                         ▼
    ┌─────────────────┐     ┌──────────────────────┐
    │   unit-tests    │     │  integration-tests    │
    └────────┬────────┘     └──────────┬───────────┘
              │                         │
              └────────────┬────────────┘
                           ▼
                  ┌─────────────────┐
                  │     deploy      │
                  └─────────────────┘
```

## 🔔 Tekton Triggers — Auto-Starting Pipelines from Events

Tekton Triggers let you **automatically start PipelineRuns** when something happens — like a Git push or a pull request.

### Components of Tekton Triggers

| Component | What It Does |
|-----------|-------------|
| **EventListener** | Listens for incoming HTTP events (webhooks) |
| **TriggerBinding** | Extracts data from the event payload (repo URL, branch, commit) |
| **TriggerTemplate** | Creates a PipelineRun using the extracted data |
| **Interceptor** | Processes/filters events (e.g., only trigger on `push` to `main`) |

### How It Works

```
GitHub/GitLab Webhook
        │
        ▼
  EventListener (K8s Service)
        │
        ▼
  Interceptor (Filter/Validate)
        │
        ▼
  TriggerBinding (Extract data from payload)
        │
        ▼
  TriggerTemplate (Create PipelineRun with extracted data)
        │
        ▼
  PipelineRun (Pipeline starts automatically!)
```

### Example: Trigger on GitHub Push

**1. TriggerBinding — Extract data from the GitHub webhook payload:**

```yaml
# trigger-binding.yaml
apiVersion: triggers.tekton.dev/v1beta1
kind: TriggerBinding
metadata:
  name: github-push-binding
spec:
  params:
    - name: git-repo-url
      value: $(body.repository.clone_url)
    - name: git-revision
      value: $(body.head_commit.id)
    - name: git-branch
      value: $(body.ref)
```

**2. TriggerTemplate — Create a PipelineRun:**

```yaml
# trigger-template.yaml
apiVersion: triggers.tekton.dev/v1beta1
kind: TriggerTemplate
metadata:
  name: github-push-template
spec:
  params:
    - name: git-repo-url
    - name: git-revision
    - name: git-branch
  resourcetemplates:
    - apiVersion: tekton.dev/v1
      kind: PipelineRun
      metadata:
        generateName: ci-cd-pipeline-run-
      spec:
        pipelineRef:
          name: ci-cd-pipeline
        params:
          - name: repo-url
            value: $(tt.params.git-repo-url)
          - name: branch
            value: $(tt.params.git-branch)
        workspaces:
          - name: shared-workspace
            volumeClaimTemplate:
              spec:
                accessModes:
                  - ReadWriteOnce
                resources:
                  requests:
                    storage: 2Gi
```

**3. EventListener — Listen for webhooks:**

```yaml
# event-listener.yaml
apiVersion: triggers.tekton.dev/v1beta1
kind: EventListener
metadata:
  name: github-listener
spec:
  serviceAccountName: tekton-triggers-sa
  triggers:
    - name: github-push
      interceptors:
        - ref:
            name: "github"
          params:
            - name: "eventTypes"
              value: ["push"]
      bindings:
        - ref: github-push-binding
      template:
        ref: github-push-template
```

**4. Expose the EventListener:**

```bash
# The EventListener creates a K8s Service automatically
kubectl get svc -l eventlistener=github-listener

# For testing, port-forward:
kubectl port-forward svc/el-github-listener 8080:8080

# In production, use an Ingress or LoadBalancer
```

Then configure your GitHub repository's webhook to point to your EventListener URL.

## 📊 Tekton Results — Storing Pipeline History

By default, Tekton stores PipelineRun/TaskRun results only as Kubernetes resources (which get garbage collected). **Tekton Results** provides long-term storage.

```bash
# Install Tekton Results
kubectl apply -f https://storage.googleapis.com/tekton-releases/results/latest/release.yaml
```

### Using Results to Pass Data Between Tasks

Tasks can emit **results** that other Tasks can consume:

```yaml
# Task that produces a result
apiVersion: tekton.dev/v1
kind: Task
metadata:
  name: generate-version
spec:
  results:
    - name: version
      description: "The generated version number"
  steps:
    - name: generate
      image: alpine:3.18
      script: |
        VERSION="1.0.$(date +%s)"
        echo -n "$VERSION" | tee $(results.version.path)
        echo "Generated version: $VERSION"
```

```yaml
# Pipeline that passes results between Tasks
apiVersion: tekton.dev/v1
kind: Pipeline
metadata:
  name: versioned-deploy
spec:
  tasks:
    - name: get-version
      taskRef:
        name: generate-version
    - name: build
      taskRef:
        name: build-with-version
      runAfter:
        - get-version
      params:
        - name: version
          value: $(tasks.get-version.results.version)
```

`$(tasks.get-version.results.version)` — This is how you reference a result from another Task. The pattern is `$(tasks.<task-name>.results.<result-name>)`.

## 🔒 Tekton Security Best Practices

| Practice | How |
|----------|-----|
| Use **ServiceAccounts** | Don't run tasks as cluster-admin |
| Use **Secrets** for credentials | Never hardcode passwords in Tasks |
| Use **signed images** | Verify task images come from trusted sources |
| **Limit RBAC** | Only give Tekton the permissions it needs |
| Use **Tekton Chains** | For supply chain security (signing, provenance) |

### Example: Using Secrets in Tekton

```yaml
# Create a secret
kubectl create secret generic git-credentials \
  --from-literal=username=myuser \
  --from-literal=password=mytoken

# Reference in a Task
apiVersion: tekton.dev/v1
kind: Task
metadata:
  name: authenticated-clone
spec:
  steps:
    - name: clone
      image: alpine/git
      env:
        - name: GIT_USERNAME
          valueFrom:
            secretKeyRef:
              name: git-credentials
              key: username
        - name: GIT_PASSWORD
          valueFrom:
            secretKeyRef:
              name: git-credentials
              key: password
      script: |
        git clone https://$GIT_USERNAME:$GIT_PASSWORD@github.com/org/repo.git
```

## 🆚 Tekton vs Other CI/CD Tools

| Feature | Tekton | Jenkins | GitHub Actions | GitLab CI |
|---------|--------|---------|----------------|-----------|
| **Runs on** | Kubernetes only | Any server | GitHub cloud/self-hosted | GitLab cloud/self-hosted |
| **Config format** | Kubernetes YAML | Jenkinsfile (Groovy) | YAML | YAML |
| **Setup effort** | Medium-High | Medium | Low | Low |
| **Scaling** | Kubernetes-native | Plugin-based | Managed / self-hosted | Managed / self-hosted |
| **UI** | Basic (Dashboard) | Rich | Built into GitHub | Built into GitLab |
| **Vendor lock-in** | None | None | GitHub | GitLab |
| **Learning curve** | Steep (need K8s knowledge) | Medium | Easy | Easy |
| **Reusable components** | Tekton Hub | Plugins/Shared Libraries | Actions Marketplace | CI templates |
| **Best for** | K8s-heavy orgs, platforms | Legacy + complex workflows | GitHub-based projects | GitLab-based projects |

## 🎯 Summary — Key Takeaways

| Point | Detail |
|-------|--------|
| Tekton is a **Kubernetes-native** CI/CD framework | Everything runs as K8s resources (Pods, CRDs) |
| **Step → Task → Pipeline** | The building block hierarchy |
| **TaskRun / PipelineRun** | Execution instances of Tasks and Pipelines |
| **Workspaces** | Shared storage between Steps and Tasks |
| **Parameters & Results** | Dynamic inputs and outputs for Tasks |
| **Triggers** | Auto-start pipelines from Git webhooks |
| **Tekton Hub** | Reusable community Tasks catalog |
| Best for | Kubernetes-heavy organizations building internal platforms |
| Not ideal for | Small teams or non-Kubernetes environments |
