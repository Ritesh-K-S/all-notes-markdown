## 📌 What is Drone CI?

**Drone** is a modern, **container-native** CI/CD platform that automatically builds, tests, and deploys your code. Every step of your pipeline runs inside an **isolated Docker container**, which makes pipelines clean, reproducible, and language-agnostic.

Think of Drone as the **minimalist's CI/CD tool** — it does what Jenkins does but with:
- Way less setup
- Way less complexity
- A beautiful, simple YAML config
- Containers everywhere

Drone was acquired by **Harness** in 2020, and its open-source version continues to thrive. The commercial version adds enterprise features, but the community edition is very capable on its own.

### Drone at a Glance

| Aspect | Details |
|--------|---------|
| **Created by** | Brad Rydzewski, now under Harness |
| **Written in** | Go |
| **Config file** | `.drone.yml` (in your repo root) |
| **License** | Apache 2.0 (Community Edition), Proprietary (Enterprise) |
| **Container runtime** | Docker |
| **Supported SCM** | GitHub, GitLab, Gitea, Bitbucket, Azure DevOps |

## 🤔 Why Drone CI? What Makes It Special?

| Feature | Why It Matters |
|---------|---------------|
| **Container-first** | Every pipeline step runs in its own Docker container. Clean, isolated, reproducible |
| **Simple YAML config** | One `.drone.yml` file — that's it. No complex UI configuration |
| **Self-service** | Developers activate repos themselves. No admin needed for every new project |
| **Lightweight** | Single Go binary. Minimal resource usage compared to Jenkins |
| **Built-in secrets** | Native secret management — no plugins needed |
| **Plugin ecosystem** | Plugins are just Docker images. Easy to create and share |
| **Multi-platform** | Supports Linux, Windows, macOS, and ARM builds |
| **Scalable** | Supports multiple runners for parallel execution |

### When to Use Drone CI?

✅ You want a simple, no-nonsense CI/CD system  
✅ Your team uses Docker already  
✅ You want developers to self-serve (activate their own repos)  
✅ You need something lightweight that doesn't eat your server's resources  
✅ You want pipeline config as code (`.drone.yml` in the repo)  

### When NOT to Use Drone CI?

❌ You need a massive plugin ecosystem like Jenkins  
❌ You need very complex pipeline orchestration (try Tekton or Jenkins)  
❌ You're in a large enterprise needing advanced RBAC, audit trails, etc.  
❌ You need deep integration with a specific cloud provider (try their native CI/CD)

## 🏗️ Drone Architecture

Drone has a clean, two-component architecture:

```
┌─────────────────────────────────────────────────────────┐
│                     Drone Server                         │
│                                                          │
│  ┌──────────────┐  ┌──────────────┐  ┌───────────────┐  │
│  │   Web UI     │  │   API        │  │  Webhook      │  │
│  │  (Dashboard) │  │  (REST)      │  │  Receiver     │  │
│  └──────────────┘  └──────────────┘  └───────────────┘  │
│                                                          │
│  ┌──────────────────────────────────────────────────┐    │
│  │              SQLite / PostgreSQL / MySQL          │    │
│  │              (Build history, users, repos)        │    │
│  └──────────────────────────────────────────────────┘    │
└─────────────────────────────┬───────────────────────────┘
                              │
                    (gRPC communication)
                              │
┌─────────────────────────────┴───────────────────────────┐
│                    Drone Runner(s)                        │
│                                                          │
│  ┌──────────────────────────────────────────────────┐    │
│  │  Receives pipeline jobs from Server               │    │
│  │  Spins up Docker containers for each step         │    │
│  │  Streams logs back to Server                      │    │
│  │  Reports build status (pass/fail)                 │    │
│  └──────────────────────────────────────────────────┘    │
│                                                          │
│  ┌────────┐  ┌────────┐  ┌────────┐  ┌────────┐        │
│  │ Step 1 │  │ Step 2 │  │ Step 3 │  │ Step 4 │        │
│  │(Docker)│  │(Docker)│  │(Docker)│  │(Docker)│        │
│  └────────┘  └────────┘  └────────┘  └────────┘        │
└─────────────────────────────────────────────────────────┘
```

### The Two Main Components

| Component | Role |
|-----------|------|
| **Drone Server** | The brain — handles webhooks, stores data, serves UI and API |
| **Drone Runner** | The muscle — picks up jobs and executes pipelines in containers |

### How It Works (Flow)

1. **Developer pushes code** to GitHub/GitLab/etc.
2. **Webhook fires** → hits the Drone Server
3. Drone Server reads `.drone.yml` from the repo
4. Drone Server **queues the build** and assigns it to a Runner
5. Runner **pulls Docker images** for each step
6. Runner **executes steps** sequentially in containers
7. **Logs stream back** to the Server in real-time
8. **Build status** is reported back to the Git provider (commit status)

## 🔧 Installing Drone CI

We'll set up Drone with GitHub as the source control provider using Docker Compose.

### Prerequisites

- Docker and Docker Compose installed
- A GitHub account
- A GitHub OAuth Application (we'll create one)

### Step 1: Create a GitHub OAuth Application

1. Go to **GitHub → Settings → Developer Settings → OAuth Apps → New OAuth App**
2. Fill in:
   - **Application name:** `Drone CI`
   - **Homepage URL:** `http://drone.yourdomain.com` (or `http://localhost:8080` for local)
   - **Authorization callback URL:** `http://drone.yourdomain.com/login` (or `http://localhost:8080/login`)
3. Click **Register application**
4. Note down the **Client ID** and generate a **Client Secret**

### Step 2: Generate a Shared Secret

The Server and Runner communicate using a shared secret (RPC secret).

```bash
# Generate a random secret
openssl rand -hex 16
# Example output: bea26a2221fd8090ea38720fc445eca6
```

### Step 3: Docker Compose File

```yaml
# docker-compose.yml
version: '3.8'

services:
  drone-server:
    image: drone/drone:2
    container_name: drone-server
    ports:
      - "8080:80"
      - "8443:443"
    volumes:
      - drone-data:/data
    restart: always
    environment:
      # GitHub OAuth
      - DRONE_GITHUB_CLIENT_ID=your_github_client_id
      - DRONE_GITHUB_CLIENT_SECRET=your_github_client_secret
      # Server settings
      - DRONE_RPC_SECRET=bea26a2221fd8090ea38720fc445eca6
      - DRONE_SERVER_HOST=localhost:8080
      - DRONE_SERVER_PROTO=http
      # Admin user (your GitHub username)
      - DRONE_USER_CREATE=username:yourgithubusername,admin:true

  drone-runner:
    image: drone/drone-runner-docker:1
    container_name: drone-runner
    restart: always
    depends_on:
      - drone-server
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock
    environment:
      - DRONE_RPC_PROTO=http
      - DRONE_RPC_HOST=drone-server
      - DRONE_RPC_SECRET=bea26a2221fd8090ea38720fc445eca6
      - DRONE_RUNNER_CAPACITY=2
      - DRONE_RUNNER_NAME=local-runner

volumes:
  drone-data:
```

**What each environment variable does:**

| Variable | Purpose |
|----------|---------|
| `DRONE_GITHUB_CLIENT_ID` | OAuth Client ID from GitHub |
| `DRONE_GITHUB_CLIENT_SECRET` | OAuth Client Secret from GitHub |
| `DRONE_RPC_SECRET` | Shared secret between Server and Runner |
| `DRONE_SERVER_HOST` | The hostname where Drone is accessible |
| `DRONE_SERVER_PROTO` | HTTP or HTTPS |
| `DRONE_USER_CREATE` | Auto-create an admin user |
| `DRONE_RUNNER_CAPACITY` | How many pipelines the runner can execute at once |

### Step 4: Start Drone

```bash
docker compose up -d

# Check logs
docker logs drone-server
docker logs drone-runner

# You should see:
# "successfully pinged the remote server" in the runner logs
```

### Step 5: Access the UI

1. Open `http://localhost:8080` in your browser
2. Click **"Continue"** to authenticate with GitHub
3. Authorize the OAuth app
4. You'll see your GitHub repositories listed
5. Click on a repo → Click **"Activate"** to enable CI/CD for it

That's it! Drone is running! 🎉

## 📝 The `.drone.yml` File — Pipeline Configuration

The `.drone.yml` file lives in the **root of your repository** and defines your entire pipeline. It's beautifully simple.

### Basic Structure

```yaml
---
kind: pipeline
type: docker
name: default

steps:
  - name: build
    image: node:18
    commands:
      - npm install
      - npm run build

  - name: test
    image: node:18
    commands:
      - npm test
```

**Breaking it down:**

| Field | What It Does |
|-------|-------------|
| `kind: pipeline` | This is a pipeline definition |
| `type: docker` | Use Docker containers for execution |
| `name: default` | The name of this pipeline |
| `steps` | The list of steps to execute |
| `steps[].name` | Human-readable name for the step |
| `steps[].image` | Docker image to run this step in |
| `steps[].commands` | Shell commands to execute inside the container |

### How Steps Work

- Each step runs in its **own Docker container**
- Steps run **sequentially** (one after another) by default
- All steps share a **common workspace** (your repo is automatically cloned)
- If any step fails, the pipeline **stops** (unless configured otherwise)

## 🔀 Triggers — When Should the Pipeline Run?

By default, Drone runs the pipeline on every push and pull request. You can control this with **trigger conditions**.

### Branch Triggers

```yaml
---
kind: pipeline
type: docker
name: default

trigger:
  branch:
    - main
    - develop
  event:
    - push
    - pull_request

steps:
  - name: build
    image: node:18
    commands:
      - npm install
      - npm run build
```

### Available Trigger Conditions

```yaml
trigger:
  # Which branches
  branch:
    include:
      - main
      - develop
    exclude:
      - feature/*

  # Which events
  event:
    - push              # Code push
    - pull_request      # PR opened/updated
    - tag               # Git tag created
    - promote           # Manual promotion
    - rollback          # Manual rollback
    - cron              # Scheduled (cron) builds

  # Only specific Git refs
  ref:
    - refs/heads/main
    - refs/tags/v*

  # Only for specific build status
  status:
    - success
    - failure
```

### Per-Step Conditions

You can also add conditions to **individual steps**:

```yaml
steps:
  - name: test
    image: node:18
    commands:
      - npm test
    # This step runs on ALL branches

  - name: deploy-staging
    image: plugins/ssh
    settings:
      host: staging.example.com
      script:
        - docker pull myapp:latest
        - docker-compose up -d
    when:
      branch:
        - develop
      event:
        - push
    # This step ONLY runs on pushes to 'develop'

  - name: deploy-production
    image: plugins/ssh
    settings:
      host: production.example.com
      script:
        - docker pull myapp:${DRONE_TAG}
        - docker-compose up -d
    when:
      event:
        - tag
      ref:
        - refs/tags/v*
    # This step ONLY runs when a version tag (v*) is created
```

## 🔐 Secrets — Managing Sensitive Data

Drone has built-in secret management. Secrets are stored in Drone (not in your YAML file) and injected into pipelines at runtime.

### Adding Secrets via UI

1. Go to your repository in the Drone UI
2. Click **Settings** → **Secrets**
3. Click **"New Secret"**
4. Enter the name and value
5. Click **Save**

### Adding Secrets via CLI

```bash
# Install Drone CLI
curl -L https://github.com/harness/drone-cli/releases/latest/download/drone_linux_amd64.tar.gz | tar zx
sudo install -t /usr/local/bin drone

# Configure CLI
export DRONE_SERVER=http://localhost:8080
export DRONE_TOKEN=your_personal_token  # From Drone UI → Account → Token

# Add a secret
drone secret add --repository yourname/yourrepo --name docker_password --data "supersecret123"

# List secrets
drone secret ls --repository yourname/yourrepo

# Remove a secret
drone secret rm --repository yourname/yourrepo --name docker_password
```

### Using Secrets in `.drone.yml`

```yaml
steps:
  - name: deploy
    image: plugins/docker
    settings:
      repo: myregistry/myapp
      username:
        from_secret: docker_username    # ← References the secret by name
      password:
        from_secret: docker_password    # ← References the secret by name
```

**Important security rules:**
- Secrets are **NOT** available in pull requests from forks (by default) — this prevents malicious PRs from stealing your secrets
- Secrets are **masked** in logs — Drone replaces the secret value with `***` in build output
- Secrets are only injected into steps that request them

## 🔌 Drone Plugins — Extending Functionality

In Drone, **plugins are just Docker images** that accept configuration via environment variables. This is a beautifully simple concept.

### How Plugins Work

A plugin is:
1. A Docker image (e.g., `plugins/docker`)
2. That reads configuration from environment variables
3. And performs a specific action (build image, send notification, deploy, etc.)

### Popular Drone Plugins

| Plugin | Image | What It Does |
|--------|-------|-------------|
| **Docker** | `plugins/docker` | Build and push Docker images |
| **ECR** | `plugins/ecr` | Push to AWS ECR |
| **GCR** | `plugins/gcr` | Push to Google Container Registry |
| **S3** | `plugins/s3` | Upload files to AWS S3 |
| **Slack** | `plugins/slack` | Send Slack notifications |
| **Email** | `plugins/email` | Send email notifications |
| **Webhook** | `plugins/webhook` | Call a webhook URL |
| **SSH** | `appleboy/drone-ssh` | Execute commands on remote servers via SSH |
| **SCP** | `appleboy/drone-scp` | Copy files to remote servers |
| **Helm** | `pelotech/drone-helm3` | Deploy Helm charts to Kubernetes |
| **Kubernetes** | `plugins/kube` | Deploy to Kubernetes |

### Plugin Usage Example

```yaml
steps:
  # Build and push Docker image
  - name: publish
    image: plugins/docker
    settings:
      repo: myregistry/myapp
      tags:
        - latest
        - ${DRONE_COMMIT_SHA:0:8}
      dockerfile: Dockerfile
      context: .
      username:
        from_secret: docker_username
      password:
        from_secret: docker_password

  # Send Slack notification
  - name: slack-notify
    image: plugins/slack
    settings:
      webhook:
        from_secret: slack_webhook
      channel: ci-notifications
      template: |
        {{#success build.status}}
          ✅ Build #{{build.number}} succeeded for {{repo.name}}
          Commit: {{build.commit}} by {{build.author}}
        {{else}}
          ❌ Build #{{build.number}} failed for {{repo.name}}
          Commit: {{build.commit}} by {{build.author}}
        {{/success}}
    when:
      status:
        - success
        - failure
```

### Creating Your Own Plugin

Since plugins are just Docker images, creating one is simple:

```bash
# my-plugin/main.sh
#!/bin/sh
echo "Hello from my custom plugin!"
echo "Repository: $DRONE_REPO"
echo "Commit: $DRONE_COMMIT"
echo "Branch: $DRONE_BRANCH"
echo "Custom Setting: $PLUGIN_MESSAGE"
```

```dockerfile
# my-plugin/Dockerfile
FROM alpine:3.18
COPY main.sh /bin/main.sh
RUN chmod +x /bin/main.sh
ENTRYPOINT ["/bin/main.sh"]
```

```bash
# Build and push
docker build -t myregistry/my-drone-plugin:latest .
docker push myregistry/my-drone-plugin:latest
```

Use it in your pipeline:

```yaml
steps:
  - name: custom-step
    image: myregistry/my-drone-plugin:latest
    settings:
      message: "This is amazing!"
```

The `settings` values are automatically converted to environment variables with a `PLUGIN_` prefix. So `message` becomes `PLUGIN_MESSAGE`.

## 🧩 Services — Running Background Containers

Need a database or Redis for your tests? Drone lets you spin up **service containers** that run alongside your pipeline.

```yaml
---
kind: pipeline
type: docker
name: default

services:
  - name: database
    image: postgres:15
    environment:
      POSTGRES_USER: testuser
      POSTGRES_PASSWORD: testpass
      POSTGRES_DB: testdb

  - name: redis
    image: redis:7-alpine

steps:
  - name: test
    image: node:18
    environment:
      DATABASE_URL: postgres://testuser:testpass@database:5432/testdb
      REDIS_URL: redis://redis:6379
    commands:
      - npm install
      - npm test
```

**How services work:**
- Service containers start **before** your steps
- They're accessible by their **name** as a hostname (e.g., `database`, `redis`)
- They run in the **same Docker network** as your step containers
- They're automatically **stopped** when the pipeline ends

This is like Docker Compose but built into your CI pipeline!

## 🔗 Multiple Pipelines in One File

You can define multiple pipelines in a single `.drone.yml`. They run **in parallel** by default.

```yaml
---
kind: pipeline
type: docker
name: backend

steps:
  - name: test-backend
    image: golang:1.21
    commands:
      - cd backend
      - go test ./...

  - name: build-backend
    image: golang:1.21
    commands:
      - cd backend
      - go build -o server .

---
kind: pipeline
type: docker
name: frontend

steps:
  - name: test-frontend
    image: node:18
    commands:
      - cd frontend
      - npm install
      - npm test

  - name: build-frontend
    image: node:18
    commands:
      - cd frontend
      - npm run build
```

### Pipeline Dependencies

Want one pipeline to wait for another? Use `depends_on`:

```yaml
---
kind: pipeline
type: docker
name: test

steps:
  - name: test
    image: node:18
    commands:
      - npm test

---
kind: pipeline
type: docker
name: deploy

depends_on:
  - test          # ← This pipeline waits for 'test' to succeed

steps:
  - name: deploy
    image: plugins/docker
    settings:
      repo: myregistry/myapp
      tags: latest
```

## 📦 Volumes — Sharing Data and Caching

### Host Volumes (Trusted Repos Only)

```yaml
---
kind: pipeline
type: docker
name: default

volumes:
  - name: cache
    host:
      path: /tmp/drone-cache

steps:
  - name: restore-cache
    image: node:18
    volumes:
      - name: cache
        path: /cache
    commands:
      - cp -r /cache/node_modules ./node_modules 2>/dev/null || true

  - name: install
    image: node:18
    commands:
      - npm install

  - name: save-cache
    image: node:18
    volumes:
      - name: cache
        path: /cache
    commands:
      - cp -r ./node_modules /cache/node_modules
```

### Temporary Volumes (Shared Between Steps)

```yaml
volumes:
  - name: shared
    temp: {}

steps:
  - name: generate
    image: alpine
    volumes:
      - name: shared
        path: /shared
    commands:
      - echo "Hello" > /shared/data.txt

  - name: consume
    image: alpine
    volumes:
      - name: shared
        path: /shared
    commands:
      - cat /shared/data.txt    # Outputs: Hello
```

## 🕐 Cron Jobs — Scheduled Builds

Drone supports scheduled builds using cron. Configure them in the Drone UI:

1. Go to your repository → **Settings** → **Cron Jobs**
2. Add a new cron job:
   - **Name:** `nightly-build`
   - **Branch:** `main`
   - **Schedule:** `@daily` (or a cron expression like `0 0 * * *`)

Then in your `.drone.yml`, you can add steps that only run on cron events:

```yaml
steps:
  - name: security-scan
    image: aquasec/trivy:latest
    commands:
      - trivy fs --severity HIGH,CRITICAL .
    when:
      event:
        - cron
      cron:
        - nightly-build
```

## 🖥️ Drone Runner Types

Drone supports different runner types for different environments:

| Runner | Use Case |
|--------|----------|
| **Docker Runner** | Most common — runs steps in Docker containers |
| **Kubernetes Runner** | Runs steps as Kubernetes Pods |
| **Exec Runner** | Runs steps directly on the host (no container isolation) |
| **SSH Runner** | Runs steps on remote machines via SSH |
| **DigitalOcean Runner** | Spins up DigitalOcean droplets for builds |
| **AWS Runner** | Spins up EC2 instances for builds |

### Kubernetes Runner Example

If you're running Drone on Kubernetes, you can use the Kubernetes runner:

```bash
# Install via Helm
helm repo add drone https://charts.drone.io
helm repo update

helm install drone-runner-kube drone/drone-runner-kube \
  --set env.DRONE_RPC_HOST=drone.yourdomain.com \
  --set env.DRONE_RPC_PROTO=https \
  --set env.DRONE_RPC_SECRET=your_rpc_secret
```

Pipeline for Kubernetes runner:

```yaml
---
kind: pipeline
type: kubernetes      # ← Different type
name: default

steps:
  - name: build
    image: node:18
    commands:
      - npm install
      - npm run build
```

## 🛡️ Drone CLI — Useful Commands

```bash
# ---- Authentication ----
export DRONE_SERVER=http://localhost:8080
export DRONE_TOKEN=your_token

# ---- Repository Management ----
drone repo ls                              # List repos
drone repo info yourname/yourrepo          # Repo info
drone repo enable yourname/yourrepo        # Activate repo
drone repo disable yourname/yourrepo       # Deactivate repo

# ---- Builds ----
drone build ls yourname/yourrepo           # List builds
drone build info yourname/yourrepo 42      # Build #42 info
drone build logs yourname/yourrepo 42 1 1  # Logs for build 42, stage 1, step 1
drone build promote yourname/yourrepo 42 production  # Promote build to production
drone build restart yourname/yourrepo 42   # Restart build #42

# ---- Secrets ----
drone secret add --repository yourname/yourrepo --name KEY --data VALUE
drone secret ls --repository yourname/yourrepo
drone secret rm --repository yourname/yourrepo --name KEY

# ---- Cron ----
drone cron ls yourname/yourrepo
drone cron add yourname/yourrepo nightly-build "0 0 * * *" --branch main

# ---- Lint / Validate ----
drone lint .drone.yml                      # Validate your pipeline config
drone fmt .drone.yml                       # Format your pipeline config

# ---- Convert (useful for debugging) ----
drone exec .drone.yml                      # Run pipeline locally (great for testing!)
```

### Running Pipelines Locally — `drone exec`

One of Drone's best features — test your pipeline **locally** before pushing:

```bash
drone exec .drone.yml
```

This runs your `.drone.yml` on your local machine using Docker, just like it would run on the Drone server. Super useful for debugging!

## 🆚 Drone CI vs Other CI/CD Tools

| Feature | Drone CI | Jenkins | GitHub Actions | Tekton |
|---------|----------|---------|----------------|--------|
| **Setup** | Very easy | Medium | Zero (hosted) | Complex (needs K8s) |
| **Config** | `.drone.yml` | Jenkinsfile | YAML workflows | K8s YAML |
| **Container-native** | Yes (every step) | Optional | Yes | Yes |
| **Resource usage** | Minimal | Heavy | N/A (hosted) | Medium |
| **UI** | Simple & clean | Feature-rich | Built into GitHub | Basic |
| **Plugin system** | Docker images | Java plugins | Actions (Docker/JS) | Tasks (K8s CRDs) |
| **Local testing** | `drone exec` | Not easy | `act` (3rd party) | Not easy |
| **Self-service** | Yes (repo activation) | Admin-managed | Yes | No |
| **Learning curve** | Low | Medium-High | Low | High |
| **Best for** | Small-mid teams, Docker users | Enterprises, complex needs | GitHub projects | K8s-native CI/CD |

## 📋 A Production-Ready `.drone.yml` Template

Here's a comprehensive template you can adapt:

```yaml
---
kind: pipeline
type: docker
name: ci

trigger:
  branch:
    - main
    - develop
    - feature/*
  event:
    - push
    - pull_request

steps:
  # Step 1: Install dependencies
  - name: install
    image: node:18-alpine
    commands:
      - npm ci
    
  # Step 2: Lint
  - name: lint
    image: node:18-alpine
    commands:
      - npm run lint

  # Step 3: Test with coverage
  - name: test
    image: node:18-alpine
    commands:
      - npm run test:coverage
    environment:
      DATABASE_URL: postgres://test:test@database:5432/testdb

  # Step 4: Build
  - name: build
    image: node:18-alpine
    commands:
      - npm run build

  # Step 5: Build and push Docker image (only on main)
  - name: docker-publish
    image: plugins/docker
    settings:
      repo: myregistry/myapp
      tags:
        - latest
        - ${DRONE_COMMIT_SHA:0:8}
        - ${DRONE_BUILD_NUMBER}
      username:
        from_secret: docker_username
      password:
        from_secret: docker_password
    when:
      branch:
        - main
      event:
        - push

  # Step 6: Deploy to staging (only on develop)
  - name: deploy-staging
    image: appleboy/drone-ssh
    settings:
      host:
        from_secret: staging_host
      username:
        from_secret: ssh_username
      key:
        from_secret: ssh_key
      script:
        - cd /opt/myapp
        - docker compose pull
        - docker compose up -d
    when:
      branch:
        - develop
      event:
        - push

  # Step 7: Notify on Slack
  - name: notify
    image: plugins/slack
    settings:
      webhook:
        from_secret: slack_webhook
      channel: ci-builds
    when:
      status:
        - success
        - failure

services:
  - name: database
    image: postgres:15
    environment:
      POSTGRES_USER: test
      POSTGRES_PASSWORD: test
      POSTGRES_DB: testdb

---
kind: pipeline
type: docker
name: deploy-production

trigger:
  event:
    - promote
  target:
    - production

depends_on:
  - ci

steps:
  - name: deploy
    image: appleboy/drone-ssh
    settings:
      host:
        from_secret: production_host
      username:
        from_secret: ssh_username
      key:
        from_secret: ssh_key
      script:
        - cd /opt/myapp
        - docker compose pull
        - docker compose up -d --no-deps myapp

  - name: notify
    image: plugins/slack
    settings:
      webhook:
        from_secret: slack_webhook
      channel: deployments
      template: "🚀 {{repo.name}} deployed to production by {{build.author}}"
```

To deploy to production, use:

```bash
drone build promote yourname/yourrepo 42 production
```

## 🎯 Summary — Key Takeaways

| Point | Detail |
|-------|--------|
| Drone is a **container-native** CI/CD platform | Every step runs in its own Docker container |
| **`.drone.yml`** is the only config file | Lives in your repo root — pipeline as code |
| **Server + Runner** architecture | Server handles UI/webhooks, Runner executes pipelines |
| **Plugins are Docker images** | Simple to use, simple to create |
| **Services** are background containers | Great for databases during tests |
| **Secrets** are built-in | Secure, masked in logs, not exposed to fork PRs |
| **`drone exec`** runs pipelines locally | Amazing for testing before pushing |
| **Self-service model** | Developers activate their own repos |
| Best for | Small-to-mid teams who love Docker and simplicity |
