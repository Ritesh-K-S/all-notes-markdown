## 📌 What is "Pipeline as Code"?

**Pipeline as Code** means defining your entire CI/CD pipeline — build, test, deploy — in a **version-controlled file** that lives **inside your source code repository**, instead of configuring it through a UI or admin console.

In simple words:

> *Your pipeline is just another file in your Git repo — reviewed, versioned, and managed like any other code.*

### The Old Way vs Pipeline as Code

**The Old Way (UI-based pipelines):**

```
Developer: "Hey Jenkins admin, can you add a test step to our pipeline?"
Admin:     "Sure, let me click through the UI... done."
Developer: "Wait, it broke. What changed?"
Admin:     "I don't know, I think I changed something else too..."
Developer: "Can we roll back?"
Admin:     "Uhh... I didn't save the old config..."
```

😰 No version control. No review process. No history. No rollback.

**The New Way (Pipeline as Code):**

```
Developer: "I'll add a test step to the Jenkinsfile and create a pull request."
Reviewer:  "Looks good! The new test step makes sense. Approved."
Developer: *merges PR*
CI/CD:     *automatically picks up the new pipeline definition*
```

😊 Version-controlled. Reviewed. History preserved. Easy rollback.

## 🤔 Why Pipeline as Code? The Benefits

| Benefit | Explanation |
|---------|-------------|
| **Version Control** | Pipeline changes are tracked in Git — who changed what, when, and why |
| **Code Review** | Pipeline changes go through PR/MR review, just like application code |
| **Reproducibility** | Anyone can recreate the exact same pipeline from the repo |
| **Rollback** | Bad pipeline change? Just `git revert`. Done |
| **Consistency** | Every branch can have its own pipeline version |
| **Self-Service** | Developers own their pipelines — no admin bottleneck |
| **Disaster Recovery** | Jenkins server dies? No problem — pipeline definition is in Git |
| **Testing** | You can test pipeline changes in a feature branch before merging |
| **Documentation** | The pipeline file IS the documentation of how your app is built and deployed |
| **Audit Trail** | Git log shows complete history of pipeline changes |

## 📂 Pipeline as Code Across Different Tools

Every major CI/CD tool supports Pipeline as Code. Here's the file each tool uses:

| CI/CD Tool | Pipeline File | Format |
|-----------|--------------|--------|
| **Jenkins** | `Jenkinsfile` | Groovy (Declarative or Scripted) |
| **GitHub Actions** | `.github/workflows/*.yml` | YAML |
| **GitLab CI** | `.gitlab-ci.yml` | YAML |
| **Azure DevOps** | `azure-pipelines.yml` | YAML |
| **CircleCI** | `.circleci/config.yml` | YAML |
| **Travis CI** | `.travis.yml` | YAML |
| **Drone CI** | `.drone.yml` | YAML |
| **Tekton** | YAML manifests (Tasks, Pipelines) | Kubernetes YAML |
| **Bitbucket Pipelines** | `bitbucket-pipelines.yml` | YAML |

### Quick Examples

**Jenkins (Jenkinsfile):**

```groovy
pipeline {
    agent any
    stages {
        stage('Build') {
            steps {
                sh 'npm install'
                sh 'npm run build'
            }
        }
        stage('Test') {
            steps {
                sh 'npm test'
            }
        }
        stage('Deploy') {
            when { branch 'main' }
            steps {
                sh './deploy.sh'
            }
        }
    }
}
```

**GitHub Actions (.github/workflows/ci.yml):**

```yaml
name: CI Pipeline
on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '18'
      - run: npm ci
      - run: npm test
      - run: npm run build
```

**GitLab CI (.gitlab-ci.yml):**

```yaml
stages:
  - build
  - test
  - deploy

build:
  stage: build
  image: node:18
  script:
    - npm ci
    - npm run build

test:
  stage: test
  image: node:18
  script:
    - npm test

deploy:
  stage: deploy
  script:
    - ./deploy.sh
  only:
    - main
```

**Drone CI (.drone.yml):**

```yaml
kind: pipeline
type: docker
name: default

steps:
  - name: build
    image: node:18
    commands:
      - npm ci
      - npm run build
  - name: test
    image: node:18
    commands:
      - npm test
```

The syntax varies, but the **concept is the same**: pipeline definition lives in your repo, checked into version control, reviewed and managed as code.

## 🏆 Best Practices — The Complete Guide

Now let's dive into the practices that separate a "good enough" pipeline from a **production-grade, maintainable, secure** pipeline.

---

### 1. Keep the Pipeline File in the Repository Root

Always store your pipeline configuration in the **root** of your repository (or the standard location for your CI tool).

```
my-project/
├── src/
├── tests/
├── Dockerfile
├── Jenkinsfile           ← Root of repo
├── .github/
│   └── workflows/
│       └── ci.yml        ← Standard location for GitHub Actions
└── .gitlab-ci.yml        ← Root of repo
```

**Why?**
- Everyone knows where to find it
- The CI tool automatically discovers it
- It's immediately visible when browsing the repo

**Don't do this:**
```
my-project/
├── src/
├── ci/
│   └── pipelines/
│       └── config/
│           └── Jenkinsfile    ← Buried too deep!
```

### 2. Use Declarative Syntax Over Scripted (When Possible)

Most CI tools offer both a declarative (structured) and a scripted (imperative) way to write pipelines. **Prefer declarative.**

**Declarative (Recommended) — Jenkins:**

```groovy
pipeline {
    agent any
    stages {
        stage('Build') {
            steps {
                sh 'npm install'
                sh 'npm run build'
            }
        }
        stage('Test') {
            steps {
                sh 'npm test'
            }
        }
    }
    post {
        always {
            cleanWs()
        }
    }
}
```

**Scripted (Avoid unless necessary) — Jenkins:**

```groovy
node {
    try {
        stage('Build') {
            sh 'npm install'
            sh 'npm run build'
        }
        stage('Test') {
            sh 'npm test'
        }
    } finally {
        cleanWs()
    }
}
```

**Why prefer declarative?**
- Easier to read and understand
- More structured — enforces consistent pipeline shape
- Better validation — catches errors before execution
- Easier for new team members to learn

**When to use scripted:**
- Very complex conditional logic
- Dynamic pipeline generation
- When declarative syntax is too limiting

### 3. Keep Pipelines DRY — Don't Repeat Yourself

If you have the same steps in multiple pipelines, **extract them** into reusable components.

**Jenkins — Shared Libraries:**

```groovy
// vars/standardBuild.groovy (in shared library repo)
def call(Map config = [:]) {
    pipeline {
        agent any
        stages {
            stage('Build') {
                steps {
                    sh "npm install"
                    sh "npm run build"
                }
            }
            stage('Test') {
                steps {
                    sh "npm test"
                }
            }
            stage('Deploy') {
                when { branch config.deployBranch ?: 'main' }
                steps {
                    sh "./deploy.sh ${config.environment ?: 'staging'}"
                }
            }
        }
    }
}
```

```groovy
// Jenkinsfile in each project
@Library('my-shared-lib') _
standardBuild(deployBranch: 'main', environment: 'production')
```

**GitHub Actions — Reusable Workflows:**

```yaml
# .github/workflows/reusable-build.yml (in a shared repo)
name: Reusable Build
on:
  workflow_call:
    inputs:
      node-version:
        type: string
        default: '18'
      deploy-env:
        type: string
        default: 'staging'
    secrets:
      DEPLOY_KEY:
        required: true

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ inputs.node-version }}
      - run: npm ci
      - run: npm test
      - run: npm run build
```

```yaml
# .github/workflows/ci.yml (in each project)
name: CI
on: push

jobs:
  build:
    uses: my-org/shared-workflows/.github/workflows/reusable-build.yml@main
    with:
      node-version: '20'
      deploy-env: 'production'
    secrets:
      DEPLOY_KEY: ${{ secrets.DEPLOY_KEY }}
```

**GitLab CI — Include Templates:**

```yaml
# shared-templates/build-template.yml (in a shared repo)
.build-template:
  stage: build
  script:
    - npm ci
    - npm run build
  artifacts:
    paths:
      - dist/

.test-template:
  stage: test
  script:
    - npm test
```

```yaml
# .gitlab-ci.yml (in each project)
include:
  - project: 'my-org/shared-templates'
    ref: main
    file: 'build-template.yml'

build:
  extends: .build-template

test:
  extends: .test-template
```

**Benefits of DRY pipelines:**
- Fix once, fix everywhere
- Consistent standards across all projects
- Less maintenance overhead
- New projects get a pipeline instantly

### 4. Parameterize Everything — Avoid Hardcoding

Never hardcode values that might change. Use parameters, variables, and environment variables.

**Bad — Hardcoded:**

```yaml
steps:
  - name: deploy
    image: myregistry.com/deployer:1.2.3
    commands:
      - kubectl apply -f deployment.yaml --namespace production
      - kubectl set image deployment/myapp myapp=myregistry.com/myapp:v1.5.0
```

**Good — Parameterized:**

```yaml
# GitHub Actions example
env:
  REGISTRY: myregistry.com
  APP_NAME: myapp
  K8S_NAMESPACE: ${{ github.ref == 'refs/heads/main' && 'production' || 'staging' }}

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Deploy
        run: |
          kubectl apply -f deployment.yaml --namespace ${{ env.K8S_NAMESPACE }}
          kubectl set image deployment/${{ env.APP_NAME }} \
            ${{ env.APP_NAME }}=${{ env.REGISTRY }}/${{ env.APP_NAME }}:${{ github.sha }}
```

**What to parameterize:**
- Docker image versions
- Environment names (dev, staging, production)
- Registry URLs
- Kubernetes namespaces
- Branch-specific behavior
- Timeouts and retry counts

### 5. Fail Fast — Order Steps by Speed

Put the **fastest, most likely to fail** steps first. Don't waste 20 minutes building a Docker image only to find out linting fails.

**Bad order:**

```yaml
stages:
  - build          # 5 minutes
  - docker-build   # 10 minutes
  - test           # 3 minutes
  - lint           # 30 seconds ← Should be FIRST!
```

**Good order:**

```yaml
stages:
  - lint           # 30 seconds — fast feedback
  - test           # 3 minutes — catch bugs early
  - build          # 5 minutes
  - docker-build   # 10 minutes — only if everything else passed
  - deploy         # 2 minutes — last step
```

**The principle:** If something is going to fail, let it fail as **early and quickly** as possible.

### 6. Use Caching Aggressively

Downloads and installations eat up pipeline time. **Cache dependencies** to speed things up.

**GitHub Actions — Dependency Caching:**

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - uses: actions/setup-node@v4
        with:
          node-version: '18'
          cache: 'npm'             # ← Built-in npm caching!
      
      - run: npm ci                # Uses cache if package-lock.json unchanged
      - run: npm test
```

**GitLab CI — Caching:**

```yaml
build:
  image: node:18
  cache:
    key: ${CI_COMMIT_REF_SLUG}     # Cache per branch
    paths:
      - node_modules/
    policy: pull-push               # Pull cache before build, push after
  script:
    - npm ci
    - npm run build
```

**Jenkins — Caching with Docker volumes:**

```groovy
pipeline {
    agent {
        docker {
            image 'node:18'
            args '-v npm-cache:/root/.npm'  // Mount a persistent volume for npm cache
        }
    }
    stages {
        stage('Install') {
            steps {
                sh 'npm ci'
            }
        }
    }
}
```

**What to cache:**
- `node_modules/` (Node.js)
- `.m2/repository/` (Maven/Java)
- `~/.cache/pip` (Python)
- `~/.cache/go-build` (Go)
- Docker layers
- Build outputs

### 7. Run Steps in Parallel When Possible

If steps don't depend on each other, run them **at the same time** to save precious minutes.

**GitHub Actions — Parallel Jobs:**

```yaml
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run lint

  unit-tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run test:unit

  integration-tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run test:integration

  # This job waits for ALL the above to pass
  deploy:
    needs: [lint, unit-tests, integration-tests]
    runs-on: ubuntu-latest
    steps:
      - run: ./deploy.sh
```

**Jenkins — Parallel Stages:**

```groovy
pipeline {
    agent any
    stages {
        stage('Build') {
            steps {
                sh 'npm install'
                sh 'npm run build'
            }
        }
        stage('Quality Checks') {
            parallel {
                stage('Unit Tests') {
                    steps {
                        sh 'npm run test:unit'
                    }
                }
                stage('Integration Tests') {
                    steps {
                        sh 'npm run test:integration'
                    }
                }
                stage('Lint') {
                    steps {
                        sh 'npm run lint'
                    }
                }
                stage('Security Scan') {
                    steps {
                        sh 'npm audit'
                    }
                }
            }
        }
        stage('Deploy') {
            steps {
                sh './deploy.sh'
            }
        }
    }
}
```

Visual representation:

```
    ┌──────────┐
    │  Build   │
    └────┬─────┘
         │
    ┌────┴────────────────────────────────────┐
    │         Quality Checks (Parallel)        │
    │                                          │
    │  ┌──────────┐  ┌───────────────┐        │
    │  │Unit Tests│  │Integration    │        │
    │  │          │  │Tests          │        │
    │  └──────────┘  └───────────────┘        │
    │  ┌──────────┐  ┌───────────────┐        │
    │  │  Lint    │  │Security Scan  │        │
    │  │          │  │               │        │
    │  └──────────┘  └───────────────┘        │
    └────┬────────────────────────────────────┘
         │
    ┌────┴─────┐
    │  Deploy  │
    └──────────┘
```

### 8. Use Matrix Builds for Multi-Version Testing

Test your code across multiple versions, operating systems, or configurations simultaneously.

**GitHub Actions — Matrix Strategy:**

```yaml
jobs:
  test:
    strategy:
      matrix:
        node-version: [16, 18, 20]
        os: [ubuntu-latest, windows-latest, macos-latest]
      fail-fast: false        # Don't cancel other jobs if one fails
    
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
      - run: npm ci
      - run: npm test
```

This creates **9 parallel jobs** (3 Node versions × 3 OS types)!

**GitLab CI — Matrix:**

```yaml
test:
  image: node:${NODE_VERSION}
  script:
    - npm ci
    - npm test
  parallel:
    matrix:
      - NODE_VERSION: ['16', '18', '20']
```

### 9. Handle Secrets Properly

This is **critical** for security. Never, ever, ever put secrets in your pipeline file.

**The Cardinal Rules of Pipeline Secrets:**

| Rule | Explanation |
|------|-------------|
| **Never hardcode secrets** | No passwords, tokens, or keys in `.yml` / `Jenkinsfile` |
| **Use the CI tool's secret store** | Every CI tool has built-in secret management |
| **Limit secret scope** | Give secrets only to the steps that need them |
| **Rotate secrets regularly** | Change passwords and tokens periodically |
| **Don't print secrets in logs** | Even accidentally via `echo $SECRET` |
| **Mask secrets in output** | Most CI tools auto-mask, but be careful with indirect exposure |

**Bad — Secrets in code:**

```yaml
# NEVER DO THIS!!! 🚫
steps:
  - name: deploy
    commands:
      - docker login -u admin -p SuperSecretPassword123 registry.example.com
```

**Good — Using secret stores:**

```yaml
# GitHub Actions
steps:
  - name: Login to Registry
    run: docker login -u ${{ secrets.REGISTRY_USER }} -p ${{ secrets.REGISTRY_PASSWORD }} registry.example.com
```

```yaml
# Drone CI
steps:
  - name: deploy
    image: plugins/docker
    settings:
      username:
        from_secret: registry_user
      password:
        from_secret: registry_password
```

```groovy
// Jenkins
pipeline {
    agent any
    stages {
        stage('Deploy') {
            steps {
                withCredentials([usernamePassword(
                    credentialsId: 'docker-registry',
                    usernameVariable: 'REG_USER',
                    passwordVariable: 'REG_PASS'
                )]) {
                    sh 'docker login -u $REG_USER -p $REG_PASS registry.example.com'
                }
            }
        }
    }
}
```

**Advanced — Use external secret managers:**

```yaml
# Integrate with HashiCorp Vault, AWS Secrets Manager, etc.
steps:
  - name: Get secrets from Vault
    uses: hashicorp/vault-action@v2
    with:
      url: https://vault.example.com
      method: token
      token: ${{ secrets.VAULT_TOKEN }}
      secrets: |
        secret/data/myapp db_password | DB_PASSWORD ;
        secret/data/myapp api_key | API_KEY
```

### 10. Implement Proper Error Handling and Notifications

Your pipeline should handle failures gracefully and notify the right people.

**Jenkins — Post-Build Actions:**

```groovy
pipeline {
    agent any
    stages {
        stage('Build') {
            steps {
                sh 'npm run build'
            }
        }
        stage('Test') {
            steps {
                sh 'npm test'
            }
        }
    }
    post {
        success {
            slackSend(color: 'good', message: "✅ Build #${env.BUILD_NUMBER} passed!")
        }
        failure {
            slackSend(color: 'danger', message: "❌ Build #${env.BUILD_NUMBER} failed!")
            emailext(
                subject: "Build Failed: ${env.JOB_NAME} #${env.BUILD_NUMBER}",
                body: "Check: ${env.BUILD_URL}",
                to: 'team@company.com'
            )
        }
        always {
            cleanWs()    // Clean workspace regardless of result
            junit 'reports/**/*.xml'    // Always publish test results
        }
        unstable {
            slackSend(color: 'warning', message: "⚠️ Build #${env.BUILD_NUMBER} is unstable")
        }
    }
}
```

**GitHub Actions — Notifications:**

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test
      
      # Always notify, regardless of success or failure
      - name: Notify Slack
        if: always()
        uses: slackapi/slack-github-action@v1
        with:
          payload: |
            {
              "text": "${{ job.status == 'success' && '✅' || '❌' }} Build ${{ job.status }} for ${{ github.repository }}"
            }
        env:
          SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK }}
```

### 11. Use Timeouts and Retry Logic

Prevent pipelines from running forever and handle transient failures.

**Jenkins — Timeouts:**

```groovy
pipeline {
    agent any
    options {
        timeout(time: 30, unit: 'MINUTES')    // Global timeout
        retry(2)                                // Retry entire pipeline up to 2 times
    }
    stages {
        stage('Build') {
            options {
                timeout(time: 10, unit: 'MINUTES')   // Stage-level timeout
            }
            steps {
                sh 'npm run build'
            }
        }
        stage('Deploy') {
            steps {
                retry(3) {                            // Retry this specific step
                    sh './deploy.sh'
                }
            }
        }
    }
}
```

**GitHub Actions — Timeouts:**

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    timeout-minutes: 30            # Job-level timeout
    steps:
      - uses: actions/checkout@v4
      - name: Build
        timeout-minutes: 10       # Step-level timeout
        run: npm run build
```

### 12. Lint and Validate Pipeline Files

Catch errors in your pipeline config **before** pushing. Don't waste a commit just to find a YAML syntax error.

**Tools for validation:**

| Tool | For | Command |
|------|-----|---------|
| `yamllint` | Any YAML-based CI | `yamllint .github/workflows/*.yml` |
| `actionlint` | GitHub Actions | `actionlint .github/workflows/*.yml` |
| `gitlab-ci-lint` | GitLab CI | Use GitLab API: `POST /api/v4/ci/lint` |
| `drone lint` | Drone CI | `drone lint .drone.yml` |
| `jenkins-pipeline-linter` | Jenkins | Use Jenkins API or IDE plugin |
| `circleci config validate` | CircleCI | `circleci config validate` |

**Example: Add linting to your workflow:**

```yaml
# .github/workflows/lint-workflows.yml
name: Lint CI Workflows

on:
  pull_request:
    paths:
      - '.github/workflows/**'

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Lint GitHub Actions workflows
        uses: raven-actions/actionlint@v1
```

Now if someone submits a PR that changes a workflow file, the linter will catch any mistakes!

### 13. Tag Pipeline Versions for Shared Pipelines

When using shared pipeline libraries or reusable workflows, always **pin to a specific version**.

**Bad — Using `main` branch (unstable):**

```yaml
# Could break at any time if the shared repo changes
uses: my-org/shared-workflows/.github/workflows/build.yml@main
```

**Good — Pin to a tag or SHA:**

```yaml
# Pinned to a specific version — stable and predictable
uses: my-org/shared-workflows/.github/workflows/build.yml@v2.1.0

# Or pin to a commit SHA for maximum safety
uses: my-org/shared-workflows/.github/workflows/build.yml@a1b2c3d4e5f6
```

```groovy
// Jenkins — Pin shared library version
@Library('my-shared-lib@v2.1.0') _
```

### 14. Keep Pipelines Fast — The Speed Budget

A fast pipeline means faster feedback, which means happier developers and more frequent deployments.

**Speed targets to aim for:**

| Stage | Target Time | Why |
|-------|-------------|-----|
| Lint | < 1 minute | Should be instant |
| Unit Tests | < 5 minutes | Fast feedback loop |
| Build | < 5 minutes | Developers wait for this |
| Integration Tests | < 10 minutes | Can be parallelized |
| Full Pipeline | < 15-20 minutes | Maximum acceptable for most teams |

**Techniques to speed up pipelines:**

| Technique | Impact |
|-----------|--------|
| **Cache dependencies** | Saves 2-5 minutes per run |
| **Parallel stages** | Cuts time proportionally to parallelism |
| **Smaller Docker images** | `alpine` vs `ubuntu` = faster pulls |
| **Incremental builds** | Only build what changed |
| **Skip unnecessary steps** | Don't run deploy on feature branches |
| **Matrix builds** | Test in parallel, not sequentially |
| **Pre-built base images** | Custom image with deps pre-installed |

### 15. Use Branch-Specific Pipeline Behavior

Not every branch needs the full pipeline. Feature branches need fast feedback; `main` needs the full treatment.

```yaml
# GitHub Actions — Branch-specific jobs
jobs:
  # Always run: lint and tests
  quality:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run lint
      - run: npm test

  # Only on main: build Docker image
  build-image:
    if: github.ref == 'refs/heads/main'
    needs: quality
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: docker build -t myapp:${{ github.sha }} .

  # Only on main: deploy to staging
  deploy-staging:
    if: github.ref == 'refs/heads/main'
    needs: build-image
    runs-on: ubuntu-latest
    steps:
      - run: ./deploy.sh staging

  # Only on tags: deploy to production
  deploy-production:
    if: startsWith(github.ref, 'refs/tags/v')
    needs: build-image
    runs-on: ubuntu-latest
    environment: production     # Requires manual approval
    steps:
      - run: ./deploy.sh production
```

```groovy
// Jenkins — Branch-specific stages
pipeline {
    agent any
    stages {
        stage('Test') {
            steps {
                sh 'npm test'
            }
        }
        stage('Deploy to Staging') {
            when {
                branch 'main'
            }
            steps {
                sh './deploy.sh staging'
            }
        }
        stage('Deploy to Production') {
            when {
                tag pattern: "v\\d+\\.\\d+\\.\\d+", comparator: "REGEXP"
            }
            input {
                message "Deploy to production?"
                ok "Yes, deploy!"
            }
            steps {
                sh './deploy.sh production'
            }
        }
    }
}
```

### 16. Document Your Pipeline

A well-documented pipeline saves hours of debugging and onboarding time.

**In-file comments:**

```yaml
# ============================================================
# CI/CD Pipeline for MyApp
# ============================================================
# Trigger: Push to main/develop, PRs to main
# Stages: lint → test → build → deploy
# Deploy: staging (develop), production (tags v*)
#
# Required Secrets:
#   - DOCKER_USERNAME: Docker Hub username
#   - DOCKER_PASSWORD: Docker Hub password
#   - DEPLOY_KEY: SSH key for deployment server
#   - SLACK_WEBHOOK: Slack notification webhook
#
# Maintainer: DevOps Team (devops@company.com)
# Last updated: 2024-01-15
# ============================================================

name: CI/CD Pipeline
on:
  push:
    branches: [main, develop]
```

**Separate documentation (recommended for complex pipelines):**

```markdown
# Pipeline Documentation

## Overview
This pipeline builds, tests, and deploys the MyApp service.

## Stages
1. **Lint** — Runs ESLint and Prettier checks
2. **Test** — Runs unit tests with Jest (requires PostgreSQL service)
3. **Build** — Creates Docker image and pushes to registry
4. **Deploy** — Deploys to staging (auto) or production (manual approval)

## Secrets Required
| Secret | Where to Get It | Who Manages It |
|--------|----------------|----------------|
| DOCKER_PASSWORD | Docker Hub | DevOps Team |
| DEPLOY_KEY | SSH keygen | DevOps Team |

## Troubleshooting
- **"npm ci" fails**: Check if package-lock.json is committed
- **Docker build fails**: Check if Dockerfile exists in repo root
- **Deploy fails**: Verify the deploy server is reachable
```

### 17. Implement Approval Gates for Production

Never auto-deploy to production without a human check (unless your test coverage and monitoring are exceptional).

**GitHub Actions — Environments with Approval:**

```yaml
jobs:
  deploy-production:
    runs-on: ubuntu-latest
    environment:
      name: production          # ← Requires approval
      url: https://myapp.com
    steps:
      - run: ./deploy.sh production
```

Set up the approval in GitHub: **Repository → Settings → Environments → production → Required reviewers**

**Jenkins — Input Step:**

```groovy
stage('Deploy to Production') {
    steps {
        input message: 'Deploy to production?', 
              ok: 'Deploy',
              submitter: 'admin,lead-dev'     // Only specific people can approve
        sh './deploy.sh production'
    }
}
```

**GitLab CI — Manual Jobs:**

```yaml
deploy-production:
  stage: deploy
  script:
    - ./deploy.sh production
  when: manual                    # ← Requires manual click
  only:
    - main
  allow_failure: false
```

## 📋 Pipeline as Code Checklist

Use this checklist when writing or reviewing a pipeline:

```
✅ Pipeline file is in the repo (version-controlled)
✅ Uses declarative syntax
✅ No hardcoded secrets — using CI tool's secret store
✅ No hardcoded values — everything parameterized
✅ Steps ordered by speed (fail fast)
✅ Dependencies are cached
✅ Independent steps run in parallel
✅ Proper error handling and notifications
✅ Timeouts set on long-running steps
✅ Branch-specific behavior configured
✅ Production deployment requires approval
✅ Pipeline file is documented (comments or separate doc)
✅ Pipeline config is validated/linted
✅ Shared pipelines pinned to specific versions
✅ Build artifacts are stored and accessible
✅ Test results are published and visible
```

## 🆚 Pipeline Configuration Approaches Compared

| Approach | Pros | Cons | Example |
|----------|------|------|---------|
| **Single file** | Simple, easy to find | Can get long/complex | `.gitlab-ci.yml` |
| **Multiple workflow files** | Separation of concerns | More files to manage | `.github/workflows/*.yml` |
| **Shared libraries** | DRY, consistent standards | Extra repo to maintain | Jenkins Shared Libraries |
| **Reusable workflows** | Composable, version-controlled | Harder to debug | GitHub reusable workflows |
| **Template includes** | Modular, team-specific overrides | Indirection makes reading harder | GitLab `include` |
| **Auto-generated** | Consistent, less human error | Less flexibility | GitLab Auto DevOps |

---

## 🎯 Summary — Key Takeaways

| Point | Detail |
|-------|--------|
| **Pipeline as Code** = pipeline config in your Git repo | Versioned, reviewed, reproducible |
| **Declarative > Scripted** | Easier to read, validate, and maintain |
| **DRY it up** | Shared libraries, reusable workflows, templates |
| **Fail fast** | Lint → Test → Build → Deploy (fastest first) |
| **Cache everything** | Dependencies, Docker layers, build artifacts |
| **Parallelize** | Independent steps should run simultaneously |
| **Secrets never in code** | Use the CI tool's secret store or external vault |
| **Set timeouts** | Prevent pipelines from running forever |
| **Approval gates** | Always require approval for production deploys |
| **Document** | Comments in the file + separate docs for complex pipelines |
| **Validate** | Lint your pipeline files before pushing |
| **Keep it fast** | Target < 15–20 minutes for the full pipeline |
