## 📌 Why Do Deployment Strategies Matter?

Imagine you've been working on a new feature for weeks. Your code is tested, your pipeline is green, and you're ready to deploy to production. But here's the million-dollar question:

**How do you push new code to production without breaking things for your users?**

That's where **deployment strategies** come in. They are techniques that control **how** your new version replaces the old one in production.

A bad deployment strategy = **downtime**, **angry users**, and **3 AM phone calls**.  
A good deployment strategy = **zero downtime**, **safe rollbacks**, and **happy engineers**.

Let's break down each strategy one by one. 👇

---

## 🔄 1. Recreate Deployment (The Simple but Risky Way)

### How It Works

This is the **simplest** strategy — and also the **most dangerous** for production:

1. **Stop** all instances of the **old version** (v1)
2. **Start** all instances of the **new version** (v2)

That's it. Kill everything → bring up the new stuff.

```
Timeline:
─────────────────────────────────────────────
   v1 running    │  DOWNTIME  │  v2 running
─────────────────│────────────│──────────────
                 ▼            ▼
           v1 stopped     v2 started
```

### Kubernetes Example

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3
  strategy:
    type: Recreate          # <-- This is the key line
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
      - name: my-app
        image: my-app:v2    # Update from v1 to v2
        ports:
        - containerPort: 8080
```

**What happens when you apply this?**

- Kubernetes will **terminate ALL** old pods first
- Then it will **create ALL** new pods
- During the gap = **downtime** ⚠️

### When to Use Recreate

| ✅ Use When | ❌ Avoid When |
|------------|-------------|
| Your app can't run two versions at once | You need zero downtime |
| Database schema changes that break old version | Production with real users |
| Dev/staging environments | High availability is required |
| You need a clean slate (no version mixing) | Business-critical applications |

### Pros & Cons

| Pros | Cons |
|------|------|
| Super simple to implement | Causes **downtime** |
| No version compatibility issues | Users get errors during transition |
| Clean state — no old/new mixing | Not suitable for production |

## 🔵🟢 2. Blue/Green Deployment

### The Big Idea

Blue/Green deployment is like having **two identical production environments**:

- **Blue** = The current live version (what users are using right now)
- **Green** = The new version (staged, tested, and ready to go)

When you're confident that Green works perfectly, you **switch traffic** from Blue to Green **all at once**. If something goes wrong, you switch back to Blue instantly.

```
BEFORE SWITCH:
┌──────────────┐        ┌──────────────┐
│   BLUE (v1)  │ ◄──── Traffic ────    │  GREEN (v2)  │
│   (LIVE)     │        │  (IDLE/TEST) │
└──────────────┘        └──────────────┘

AFTER SWITCH:
┌──────────────┐        ┌──────────────┐
│   BLUE (v1)  │        │  GREEN (v2)  │ ◄──── Traffic
│   (IDLE)     │        │   (LIVE)     │
└──────────────┘        └──────────────┘
```

### How It Works — Step by Step

1. **Blue is live** — Users are hitting the Blue environment (v1)
2. **Deploy Green** — Deploy v2 to the Green environment (identical infrastructure)
3. **Test Green** — Run smoke tests, health checks on Green (users don't see this)
4. **Switch traffic** — Update the load balancer / DNS / router to point to Green
5. **Monitor** — Watch metrics, logs, error rates
6. **If OK** → Green becomes the new "live", Blue becomes standby
7. **If NOT OK** → Switch traffic back to Blue immediately (instant rollback!)

### Implementation with Kubernetes Services

```yaml
# blue-deployment.yaml — Current live version
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app-blue
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
      version: blue
  template:
    metadata:
      labels:
        app: my-app
        version: blue
    spec:
      containers:
      - name: my-app
        image: my-app:v1
        ports:
        - containerPort: 8080
---
# green-deployment.yaml — New version being prepared
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app-green
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
      version: green
  template:
    metadata:
      labels:
        app: my-app
        version: green
    spec:
      containers:
      - name: my-app
        image: my-app:v2
        ports:
        - containerPort: 8080
---
# service.yaml — The single Service that routes traffic
apiVersion: v1
kind: Service
metadata:
  name: my-app-service
spec:
  selector:
    app: my-app
    version: blue      # <-- Currently pointing to BLUE
  ports:
  - port: 80
    targetPort: 8080
```

**To switch traffic from Blue to Green**, just update the Service selector:

```bash
# Switch traffic to Green
kubectl patch service my-app-service -p '{"spec":{"selector":{"version":"green"}}}'

# Oh no, something broke! Switch back to Blue instantly:
kubectl patch service my-app-service -p '{"spec":{"selector":{"version":"blue"}}}'
```

That's it! One command to switch. One command to rollback. **Zero downtime.**

### Implementation with Nginx (Traditional Servers)

```nginx
# /etc/nginx/conf.d/my-app.conf

# Define both upstream groups
upstream blue {
    server 10.0.1.10:8080;
    server 10.0.1.11:8080;
    server 10.0.1.12:8080;
}

upstream green {
    server 10.0.2.10:8080;
    server 10.0.2.11:8080;
    server 10.0.2.12:8080;
}

server {
    listen 80;
    server_name myapp.example.com;

    location / {
        proxy_pass http://blue;    # Change to 'green' to switch
    }
}
```

To switch: change `proxy_pass http://blue;` → `proxy_pass http://green;` and reload Nginx:

```bash
sudo nginx -s reload
```

### Blue/Green with AWS (Using ALB Target Groups)

```bash
# Create two target groups
aws elbv2 create-target-group --name blue-tg --protocol HTTP --port 8080 --vpc-id vpc-xxxx
aws elbv2 create-target-group --name green-tg --protocol HTTP --port 8080 --vpc-id vpc-xxxx

# Switch listener from blue to green
aws elbv2 modify-listener \
  --listener-arn arn:aws:elasticloadbalancing:...:listener/... \
  --default-actions Type=forward,TargetGroupArn=arn:aws:elasticloadbalancing:...:targetgroup/green-tg/...
```

### Pros & Cons

| Pros | Cons |
|------|------|
| **Zero downtime** | Requires **double the infrastructure** ($$) |
| **Instant rollback** (just switch back) | Database migrations can be tricky |
| Full testing of new version before going live | More complex to set up initially |
| Clear separation between old and new | Need to keep both environments in sync |

### When to Use Blue/Green

- Production deployments where **zero downtime** is critical
- When you want the ability to **instantly rollback**
- When you can afford the **extra infrastructure cost**
- Applications where you want **full confidence** before switching

## 🔄 3. Rolling Deployment

### The Big Idea

Instead of switching everything at once, a Rolling Deployment replaces your old version with the new version **gradually**, one instance (or a few instances) at a time.

Think of it like replacing tiles on a floor — you replace them **one by one**, and the floor is always usable.

```
Step 1: [v1] [v1] [v1] [v1]    ← All running v1
Step 2: [v2] [v1] [v1] [v1]    ← First pod updated
Step 3: [v2] [v2] [v1] [v1]    ← Second pod updated
Step 4: [v2] [v2] [v2] [v1]    ← Third pod updated
Step 5: [v2] [v2] [v2] [v2]    ← All running v2 ✅
```

At every step, your application is **always available** — some pods are running v1, some are running v2, but the service never goes down.

### How It Works

1. You have N pods running v1
2. Kubernetes (or your orchestrator) starts replacing pods **one by one**:
   - Kill one v1 pod → start one v2 pod
   - Wait for the v2 pod to be **healthy** (pass readiness probe)
   - Move to the next pod
3. Continue until all pods are running v2
4. If a v2 pod fails health checks, the rollout **pauses** or **rolls back**

### Kubernetes Example (This is the DEFAULT strategy!)

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 4
  strategy:
    type: RollingUpdate          # <-- Default strategy in Kubernetes!
    rollingUpdate:
      maxUnavailable: 1          # Max pods that can be down during update
      maxSurge: 1                # Max extra pods created during update
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
      - name: my-app
        image: my-app:v2
        ports:
        - containerPort: 8080
        readinessProbe:          # IMPORTANT for rolling updates!
          httpGet:
            path: /health
            port: 8080
          initialDelaySeconds: 5
          periodSeconds: 10
        livenessProbe:
          httpGet:
            path: /health
            port: 8080
          initialDelaySeconds: 15
          periodSeconds: 20
```

### Understanding maxUnavailable and maxSurge

These two settings control the **speed** and **safety** of your rolling update:

| Setting | What It Does | Example (4 replicas) |
|---------|-------------|---------------------|
| `maxUnavailable: 1` | At most 1 pod can be unavailable at any time | Minimum 3 pods always running |
| `maxSurge: 1` | At most 1 extra pod can be created beyond desired count | Max 5 pods temporarily |
| `maxUnavailable: 25%` | Up to 25% of pods can be down | With 4 replicas = 1 pod |
| `maxSurge: 25%` | Up to 25% extra pods allowed | With 4 replicas = 1 extra pod |

**Conservative settings** (slow but safe):
```yaml
maxUnavailable: 0    # Never have fewer pods than desired
maxSurge: 1          # Only create one extra at a time
```

**Aggressive settings** (fast but risky):
```yaml
maxUnavailable: 2    # 2 pods can be down simultaneously
maxSurge: 2          # 2 extra pods created simultaneously
```

### Rolling Update Commands

```bash
# Update the image (triggers a rolling update)
kubectl set image deployment/my-app my-app=my-app:v2

# Watch the rollout progress in real-time
kubectl rollout status deployment/my-app

# Check rollout history
kubectl rollout history deployment/my-app

# Undo the rollout (rollback to previous version)
kubectl rollout undo deployment/my-app

# Rollback to a specific revision
kubectl rollout undo deployment/my-app --to-revision=2

# Pause a rollout (useful for canary-like behavior)
kubectl rollout pause deployment/my-app

# Resume a paused rollout
kubectl rollout resume deployment/my-app
```

### Pros & Cons

| Pros | Cons |
|------|------|
| **Zero downtime** | Both versions run simultaneously (version compatibility needed) |
| **Default in Kubernetes** (no extra setup) | Slower than Blue/Green (gradual) |
| No extra infrastructure cost | Rollback is also gradual (not instant) |
| Automatic rollback if health checks fail | Harder to test the full new version in production |

### When to Use Rolling Deployment

- Most **standard production deployments**
- When you can't afford double infrastructure (unlike Blue/Green)
- When your app handles **version compatibility** well
- Kubernetes workloads (it's the default — just use it!)

## 🐤 4. Canary Deployment

### The Big Idea

A Canary deployment is like being **extra cautious**. Instead of rolling out to everyone, you first send a **small percentage of traffic** (say 5%) to the new version. If everything looks good, you gradually increase the traffic until 100% is on the new version.

The name comes from the old practice of coal miners bringing **canaries into mines** — if the canary died, the air was toxic and miners would evacuate. Similarly, your "canary" pods detect problems before they affect all users.

```
Step 1: 95% → v1    |  5% → v2     ← Small canary test
Step 2: 80% → v1    | 20% → v2     ← Looking good, increase
Step 3: 50% → v1    | 50% → v2     ← Half and half
Step 4:  0% → v1    |100% → v2     ← Full rollout ✅

At any step: metrics bad? → Route 100% back to v1 🔄
```

### How It Works — Step by Step

1. Deploy v2 to a **small number of pods** (canary pods)
2. Route a **small percentage of traffic** (5-10%) to v2
3. **Monitor closely** — error rates, latency, CPU, memory, logs
4. If metrics are **healthy** → gradually increase traffic to v2
5. If metrics are **bad** → route all traffic back to v1, kill canary pods
6. Continue increasing until v2 has 100% traffic
7. Remove v1 pods

### Simple Canary with Kubernetes (Using Replica Counts)

The simplest way — control traffic split by adjusting **replica counts**:

```yaml
# v1 deployment — 9 replicas (90% of traffic)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app-v1
spec:
  replicas: 9
  selector:
    matchLabels:
      app: my-app
      version: v1
  template:
    metadata:
      labels:
        app: my-app
        version: v1
    spec:
      containers:
      - name: my-app
        image: my-app:v1
---
# v2 deployment — 1 replica (10% of traffic)  ← The Canary!
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app-v2
spec:
  replicas: 1
  selector:
    matchLabels:
      app: my-app
      version: v2
  template:
    metadata:
      labels:
        app: my-app
        version: v2
    spec:
      containers:
      - name: my-app
        image: my-app:v2
---
# Service that selects BOTH versions (based on shared label)
apiVersion: v1
kind: Service
metadata:
  name: my-app-service
spec:
  selector:
    app: my-app       # Matches BOTH v1 and v2 pods!
  ports:
  - port: 80
    targetPort: 8080
```

Since the Service selects `app: my-app` (without specifying version), traffic is distributed across **all 10 pods** — 9 v1 pods get ~90% traffic, 1 v2 pod gets ~10% traffic.

To increase canary traffic, scale up v2 and scale down v1:

```bash
# Increase canary to 30%
kubectl scale deployment my-app-v2 --replicas=3
kubectl scale deployment my-app-v1 --replicas=7

# Increase canary to 50%
kubectl scale deployment my-app-v2 --replicas=5
kubectl scale deployment my-app-v1 --replicas=5

# Full rollout (100% v2)
kubectl scale deployment my-app-v2 --replicas=10
kubectl scale deployment my-app-v1 --replicas=0
```

### Advanced Canary with Istio (Weighted Traffic Routing)

For **precise traffic control**, use a service mesh like Istio:

```yaml
# Istio VirtualService — precise traffic splitting
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: my-app
spec:
  hosts:
  - my-app-service
  http:
  - route:
    - destination:
        host: my-app-service
        subset: v1
      weight: 95             # 95% to v1
    - destination:
        host: my-app-service
        subset: v2
      weight: 5              # 5% to v2 (canary)
---
# Istio DestinationRule — define subsets
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: my-app
spec:
  host: my-app-service
  subsets:
  - name: v1
    labels:
      version: v1
  - name: v2
    labels:
      version: v2
```

With Istio, you can control traffic at the **exact percentage** level — even 1% to canary.

### Automated Canary with Flagger

**Flagger** is a tool that automates canary deployments on Kubernetes. It watches metrics and automatically promotes or rolls back:

```yaml
# Flagger Canary resource
apiVersion: flagger.app/v1beta1
kind: Canary
metadata:
  name: my-app
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-app
  service:
    port: 8080
  analysis:
    interval: 1m              # Check every 1 minute
    threshold: 5              # Max 5 failed checks before rollback
    maxWeight: 50             # Max 50% traffic to canary
    stepWeight: 10            # Increase by 10% each step
    metrics:
    - name: request-success-rate
      thresholdRange:
        min: 99              # Must maintain 99% success rate
      interval: 1m
    - name: request-duration
      thresholdRange:
        max: 500             # Response time must be under 500ms
      interval: 1m
```

**What Flagger does automatically:**

1. Detects a new version of your Deployment
2. Creates canary pods
3. Routes 10% traffic → checks metrics → OK? → 20% → checks → OK? → 30%...
4. If success rate drops below 99% → **automatic rollback**
5. If all checks pass → **automatic promotion** to 100%

### Pros & Cons

| Pros | Cons |
|------|------|
| **Safest** deployment strategy | More complex to set up |
| Issues caught early (only small % affected) | Requires good **monitoring** and **metrics** |
| Data-driven decisions (metrics guide rollout) | Need service mesh for precise control (Istio, Linkerd) |
| Can be fully automated (Flagger, Argo Rollouts) | Both versions run simultaneously |

### When to Use Canary

- **Critical production deployments** where you want maximum safety
- When you have **good monitoring** (Prometheus, Grafana, Datadog)
- Large-scale applications with **many users**
- When combined with **automated analysis** (Flagger, Argo Rollouts)

## 🧪 5. A/B Testing Deployment

### The Big Idea

A/B Testing is **NOT really about deploying** — it's about **experimenting**. You run **two versions simultaneously** and route different **groups of users** to different versions to see which one performs better.

Think of it like a scientific experiment:
- **Group A** sees version A (the control)
- **Group B** sees version B (the experiment)
- You measure which group has better results (more clicks, more purchases, fewer errors)

The key difference from Canary:
- **Canary** = "Is this new version stable?" (technical decision)
- **A/B Testing** = "Which version do users prefer?" (business decision)

```
                        ┌─── Header: X-Version=A ──→ [Version A]
User Request ──→ Router │
                        └─── Header: X-Version=B ──→ [Version B]

         Routing based on: cookies, headers, user ID, location, etc.
```

### How A/B Testing Routing Works

Traffic is split based on **user attributes**, not random percentages:

| Routing Criteria | Example |
|-----------------|---------|
| **Cookie/Header** | Users with `beta=true` cookie get version B |
| **User ID** | User IDs ending in 0-4 get A, 5-9 get B |
| **Geolocation** | US users get A, EU users get B |
| **Device type** | Mobile users get A, desktop users get B |
| **Account type** | Free users get A, premium users get B |

### A/B Testing with Istio

```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: my-app
spec:
  hosts:
  - my-app-service
  http:
  # Rule 1: If header "x-user-group" is "beta", route to v2
  - match:
    - headers:
        x-user-group:
          exact: beta
    route:
    - destination:
        host: my-app-service
        subset: v2

  # Rule 2: Everyone else goes to v1
  - route:
    - destination:
        host: my-app-service
        subset: v1
```

### A/B Testing with Nginx

```nginx
# Split based on a cookie
map $cookie_ab_test $backend {
    "B"     green_backend;    # Cookie value "B" → version B
    default blue_backend;     # Everyone else → version A
}

upstream blue_backend {
    server 10.0.1.10:8080;
}

upstream green_backend {
    server 10.0.2.10:8080;
}

server {
    listen 80;
    server_name myapp.example.com;

    location / {
        proxy_pass http://$backend;
    }
}
```

### A/B Testing with Kubernetes Ingress (Nginx Ingress Controller)

```yaml
# Version A — Main Ingress
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-app-main
spec:
  rules:
  - host: myapp.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: my-app-v1
            port:
              number: 80
---
# Version B — Canary Ingress (header-based routing)
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-app-canary
  annotations:
    nginx.ingress.kubernetes.io/canary: "true"
    nginx.ingress.kubernetes.io/canary-by-header: "x-user-group"
    nginx.ingress.kubernetes.io/canary-by-header-value: "beta"
spec:
  rules:
  - host: myapp.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: my-app-v2
            port:
              number: 80
```

**How this works:**
- Users with header `x-user-group: beta` → routed to v2
- Everyone else → routed to v1

### Pros & Cons

| Pros | Cons |
|------|------|
| Make **data-driven** product decisions | Most complex to implement |
| Test features on real users | Need analytics tools to measure results |
| Control exactly **who** sees what | Both versions must run simultaneously |
| Measure business impact before full rollout | Routing logic can get complicated |

### When to Use A/B Testing

- **Product decisions** — testing UI changes, new features
- **E-commerce** — testing checkout flows, pricing pages
- When you need **business metrics**, not just technical metrics
- Combined with **feature flag** tools (covered in 11.26)

## 🔀 6. Shadow (Dark) Deployment

### The Big Idea

In a Shadow deployment, you run the new version alongside the old version, but the new version receives a **copy of real production traffic** without actually affecting users. Users always get responses from v1 — v2 just processes the same requests in the background for testing purposes.

```
                     ┌──→ [v1] ──→ Response to User ✅
User Request ──→ Router │
                     └──→ [v2] ──→ Response discarded 🗑️ (but logged for analysis)
```

### When to Use Shadow Deployment

- Testing **performance** of new version under real-world traffic
- Validating **data processing** logic (does v2 produce the same results as v1?)
- Machine learning model deployments (compare prediction accuracy)
- When you **cannot risk** any user impact at all

### Shadow Deployment with Istio

```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: my-app
spec:
  hosts:
  - my-app-service
  http:
  - route:
    - destination:
        host: my-app-v1        # Real traffic goes here
    mirror:
      host: my-app-v2          # Copy of traffic goes here
    mirrorPercentage:
      value: 100.0              # Mirror 100% of traffic
```

## 📊 Strategy Comparison — The Ultimate Cheat Sheet

| Strategy | Downtime | Rollback Speed | Cost | Complexity | Risk | Best For |
|----------|----------|---------------|------|-----------|------|----------|
| **Recreate** | YES ⚠️ | Slow (redeploy) | Low | Very Easy | High | Dev/staging |
| **Blue/Green** | No | **Instant** ⚡ | **High** (2x infra) | Medium | Low | Critical apps |
| **Rolling** | No | Gradual | Low | Easy | Medium | Default K8s |
| **Canary** | No | Fast | Medium | High | **Very Low** | High-traffic apps |
| **A/B Testing** | No | Fast | Medium | High | Low | Product experiments |
| **Shadow** | No | N/A | High | High | **None** | Testing in prod |

## 🔧 Real-World Tool: Argo Rollouts

**Argo Rollouts** is a Kubernetes controller that provides advanced deployment strategies (Blue/Green, Canary) with **automated analysis** and **progressive delivery**.

### Install Argo Rollouts

```bash
kubectl create namespace argo-rollouts
kubectl apply -n argo-rollouts -f https://github.com/argoproj/argo-rollouts/releases/latest/download/install.yaml

# Install the kubectl plugin
# (For analysis in CLI)
brew install argoproj/tap/kubectl-argo-rollouts    # Mac
# Or download from GitHub releases for Linux/Windows
```

### Canary Rollout with Argo Rollouts

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: my-app
spec:
  replicas: 5
  strategy:
    canary:
      steps:
      - setWeight: 10        # Send 10% traffic to canary
      - pause: {duration: 5m} # Wait 5 minutes
      - setWeight: 30        # Increase to 30%
      - pause: {duration: 5m}
      - setWeight: 50        # Increase to 50%
      - pause: {duration: 5m}
      - setWeight: 100       # Full rollout
      canaryService: my-app-canary
      stableService: my-app-stable
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
      - name: my-app
        image: my-app:v2
        ports:
        - containerPort: 8080
```

### Blue/Green Rollout with Argo Rollouts

apiVersion: argoproj.io/v1alpha1

  name: my-app

  replicas: 3
  strategy:
    blueGreen:
      activeService: my-app-active       # Service pointing to live version
      previewService: my-app-preview     # Service pointing to new version

## 🎯 How to Choose the Right Strategy

Ask yourself these questions:

```
1. Can your app tolerate downtime?
   └─ YES → Recreate (simplest)
   └─ NO  → Continue ↓

2. Do you need instant rollback?
   └─ YES → Blue/Green
   └─ NO  → Continue ↓

3. Do you want maximum safety with gradual rollout?
   └─ YES → Canary
   └─ NO  → Continue ↓

4. Do you want to test different features with different user groups?
   └─ YES → A/B Testing
   └─ NO  → Continue ↓

5. Just a regular deployment with zero downtime?
   └─ YES → Rolling Update (K8s default — you probably already have this!)
```

---

## 📝 Summary

| Strategy | One-Line Summary |
|----------|-----------------|
| **Recreate** | Kill old, start new. Simple but causes downtime. |
| **Blue/Green** | Two environments, instant switch, instant rollback. |
| **Rolling** | Replace pods one by one. K8s default. Zero downtime. |
| **Canary** | Small % first, watch metrics, then increase. Safest. |
| **A/B Testing** | Route by user attributes. Business experiments. |
| **Shadow** | Mirror traffic to new version. Zero user impact. |

The best DevOps engineers don't just pick one strategy — they use **different strategies for different situations**. A staging deploy might use Recreate, while a production deploy uses Canary with Argo Rollouts and automated metric analysis.
