## 📌 What Are Feature Flags?

Imagine you're working on a big new feature — say, a **new checkout flow** for your e-commerce app. The feature takes 3 weeks to build. During those 3 weeks, you're pushing code to `main` every day (because you practice CI/CD, right?). But the feature isn't ready for users yet!

**The old way:** Use a long-lived feature branch → merge nightmare after 3 weeks 😫  
**The feature flag way:** Wrap the incomplete feature in an `if` statement → deploy to production but **keep it hidden** 🎉

A **feature flag** (also called **feature toggle** or **feature switch**) is simply a way to **turn features on or off** without deploying new code.

```
// Without feature flag:
showNewCheckout()    // ← Everyone sees it, ready or not

// With feature flag:
if (featureFlags.isEnabled("new-checkout")) {
    showNewCheckout()    // ← Only shown when flag is ON
} else {
    showOldCheckout()    // ← Everyone else sees this
}
```

That's it at the core! It's just a **conditional check** that you can control **remotely** — from a dashboard, an API, or a config file — without redeploying your application.

---

## 🤔 Why Do Feature Flags Matter in CI/CD?

Feature flags solve some of the **biggest headaches** in software delivery:

### 1. Decouple Deployment from Release

| Without Flags | With Flags |
|--------------|------------|
| Deploying code = releasing feature to users | Deploy code anytime, release when ready |
| "We can't deploy until the feature is done!" | "Deploy daily, enable the feature when it's complete" |
| Deployment is scary | Deployment is boring (in a good way!) |

### 2. Trunk-Based Development

Feature flags make **trunk-based development** (everyone commits to `main`) possible. No more long-lived branches!

```
Old way:
feature-branch (3 weeks) ──────────────────→ merge to main → deploy → pray 🙏

New way:
main (daily commits with flags OFF) ──→ deploy daily → flip flag when ready ✅
```

### 3. Safe Rollouts

```
Day 1: Enable flag for internal team only → test in production
Day 2: Enable flag for 5% of users → monitor metrics
Day 3: Enable for 25% → looking good!
Day 4: Enable for 100% → full release 🎉

Oh no, metrics dropped? → Turn flag OFF → instant rollback 🔄
```

### 4. Kill Switch

Something broke in production at 2 AM? Instead of rolling back a deployment, just **turn off the flag**. No code changes, no deploy, no waiting. Just flip it off.

## 🏗️ Types of Feature Flags

Not all feature flags are the same. Here are the main types:

### 1. Release Flags (Short-lived)

**Purpose:** Hide incomplete features until they're ready.

```python
if feature_flags.is_enabled("new-dashboard"):
    return render_new_dashboard()
else:
    return render_old_dashboard()
```

**Lifecycle:** Create → develop behind flag → release → **remove flag and old code** ✅

> ⚠️ **Important:** Remove release flags after the feature is fully rolled out! Dead flags are tech debt.

### 2. Experiment Flags (A/B Tests)

**Purpose:** Test different variations to see which performs better.

```python
variant = feature_flags.get_variant("checkout-button-color")

if variant == "red":
    button_color = "#FF0000"
elif variant == "green":
    button_color = "#00FF00"
else:
    button_color = "#0000FF"  # default (control)
```

**Lifecycle:** Create → run experiment → analyze data → pick winner → **remove flag**

### 3. Ops Flags (Kill Switches)

**Purpose:** Disable features during incidents or high load.

```python
if feature_flags.is_enabled("enable-recommendations-engine"):
    recommendations = fetch_recommendations(user_id)  # Expensive call
else:
    recommendations = []  # Skip it — system is under load
```

**Lifecycle:** These may live **long-term** as operational controls.

### 4. Permission Flags (Entitlements)

**Purpose:** Control access based on user attributes (plan, role, region).

```python
if feature_flags.is_enabled("premium-analytics", user=current_user):
    show_premium_analytics()
else:
    show_upgrade_prompt()
```

**Lifecycle:** Long-term — tied to business logic.

## 🔧 Building a Simple Feature Flag System (DIY)

Before we look at tools, let's understand what's happening under the hood by building a simple flag system.

### The Simplest Feature Flag — Environment Variable

```python
import os

# Set in environment: FEATURE_NEW_UI=true
NEW_UI_ENABLED = os.getenv("FEATURE_NEW_UI", "false").lower() == "true"

def render_page():
    if NEW_UI_ENABLED:
        return render_new_ui()
    else:
        return render_old_ui()
```

**Pros:** Dead simple.  
**Cons:** Need to redeploy to change the flag. That defeats the purpose!

### Better — Config File Based

```yaml
# feature-flags.yaml
flags:
  new-checkout:
    enabled: true
    description: "New checkout flow"
  dark-mode:
    enabled: false
    description: "Dark mode theme"
  premium-reports:
    enabled: true
    users: ["user123", "user456"]  # Only for specific users
```

```python
import yaml

class FeatureFlags:
    def __init__(self, config_path="feature-flags.yaml"):
        with open(config_path) as f:
            self.config = yaml.safe_load(f)

    def is_enabled(self, flag_name, user_id=None):
        flag = self.config.get("flags", {}).get(flag_name)
        if not flag:
            return False

        # Check if flag is enabled at all
        if not flag.get("enabled", False):
            return False

        # Check if limited to specific users
        allowed_users = flag.get("users")
        if allowed_users and user_id not in allowed_users:
            return False

        return True

# Usage:
flags = FeatureFlags()

if flags.is_enabled("new-checkout"):
    show_new_checkout()

if flags.is_enabled("premium-reports", user_id="user123"):
    show_premium_reports()
```

**Pros:** Easy to update without code changes.  
**Cons:** Still need to restart app or rebuild config. No real-time changes.

### Even Better — Database/API Based

```python
import requests

class RemoteFeatureFlags:
    def __init__(self, api_url):
        self.api_url = api_url
        self._cache = {}
        self._cache_ttl = 60  # seconds

    def is_enabled(self, flag_name, user_context=None):
        """Check if a feature flag is enabled, with caching."""
        response = requests.post(
            f"{self.api_url}/evaluate",
            json={
                "flag": flag_name,
                "context": user_context or {}
            }
        )
        return response.json().get("enabled", False)

# Usage:
flags = RemoteFeatureFlags("https://flags.mycompany.com/api")

if flags.is_enabled("new-checkout", {"user_id": "123", "plan": "premium"}):
    show_new_checkout()
```

This is basically what tools like LaunchDarkly, Unleash, and Flagsmith do — but with **way more features**.

## 🚀 LaunchDarkly — The Industry Leader

LaunchDarkly is a **fully managed**, **enterprise-grade** feature flag platform. It's the most popular commercial feature flag service and is used by companies like IBM, Microsoft, and Atlassian.

### Key Features

| Feature | Description |
|---------|-------------|
| **Real-time flag updates** | Changes propagate in milliseconds via streaming |
| **Targeting rules** | Target by user, segment, percentage, custom attributes |
| **Environments** | Separate flag configs per environment (dev, staging, prod) |
| **Audit log** | Track who changed what, when |
| **Experimentation** | Built-in A/B testing and metric tracking |
| **SDKs** | 25+ SDKs (JavaScript, Python, Go, Java, React, Node, etc.) |
| **Approvals** | Require approval before flag changes in production |

### How to Use LaunchDarkly

#### Step 1: Install the SDK

```bash
# Python
pip install launchdarkly-server-sdk

# Node.js
npm install @launchdarkly/node-server-sdk

# Go
go get github.com/launchdarkly/go-server-sdk/v7
```

#### Step 2: Initialize the Client

```python
import ldclient
from ldclient.config import Config

# Initialize with your SDK key
ldclient.set_config(Config("YOUR_SDK_KEY"))
client = ldclient.get()

# Wait for initialization
if client.is_initialized():
    print("LaunchDarkly client initialized!")
```

#### Step 3: Evaluate a Flag

```python
from ldclient import Context

# Create a user context
context = Context.builder("user-123") \
    .set("email", "john@example.com") \
    .set("plan", "premium") \
    .set("country", "US") \
    .build()

# Check if flag is enabled (returns boolean)
show_new_ui = client.variation("new-ui-feature", context, False)
# False = default value if flag doesn't exist or SDK can't connect

if show_new_ui:
    render_new_ui()
else:
    render_old_ui()
```

#### Step 4: Use Multivariate Flags (Not Just Boolean!)

```python
# String flag — returns a variation like "control", "variant-a", "variant-b"
checkout_version = client.variation("checkout-experiment", context, "control")

if checkout_version == "variant-a":
    show_single_page_checkout()
elif checkout_version == "variant-b":
    show_multi_step_checkout()
else:
    show_original_checkout()
```

#### Step 5: Targeting Rules (Set in LaunchDarkly Dashboard)

In the LaunchDarkly dashboard, you can create rules like:

```
Rule 1: IF user.email ends with "@mycompany.com" → flag = TRUE
Rule 2: IF user.plan == "premium" → flag = TRUE  
Rule 3: IF user.country IN ["US", "CA"] → percentage rollout 50%
Default: flag = FALSE
```

#### Node.js Example

```javascript
import * as LaunchDarkly from '@launchdarkly/node-server-sdk';

const client = LaunchDarkly.init('YOUR_SDK_KEY');

await client.waitForInitialization();

const context = {
  kind: 'user',
  key: 'user-123',
  email: 'john@example.com',
  plan: 'premium',
  country: 'US'
};

// Boolean flag

// JSON flag (returns complex config)

  theme: 'light',
  maxItems: 10
});

console.log(`New feature: ${showNewFeature}`);

// Always close the client when done
await client.close();
```

## ✅ Feature Flag Best Practices

### 1. Clean Up Old Flags

Feature flags are **temporary** (mostly). Remove them after the feature is fully rolled out!

```python
# BAD — This flag was enabled 2 years ago and everyone forgot about it
if flags.is_enabled("new-login-page"):  # Always true, nobody remembers why
    show_new_login()
else:
    show_old_login()  # Dead code — never runs

# GOOD — After full rollout, clean up:
show_new_login()  # Just call it directly, remove the flag
```

**Tip:** Set a **flag expiry date** when you create it. Many tools support this.

### 2. Name Flags Consistently

```
✅ Good naming:
  enable-new-checkout
  experiment-checkout-button-color
  ops-disable-recommendation-engine
  permission-premium-analytics

❌ Bad naming:
  flag1
  test
  johns-feature
  temporary-fix-please-remove
```

### 3. Use a Naming Convention

| Prefix | Type | Example |
|--------|------|---------|
| `release-` | Release toggles | `release-new-dashboard` |
| `experiment-` | A/B tests | `experiment-pricing-page` |
| `ops-` | Operational flags | `ops-disable-email-notifications` |
| `permission-` | Entitlement flags | `permission-export-csv` |

### 4. Default to OFF

Always set the **default value** to `false` (off). This means if the flag service is down, the old behavior continues (safe fallback).

```python
# GOOD — default is False (safe fallback)
show_new_feature = client.variation("new-feature", context, False)

# BAD — default is True (if flag service is down, everyone gets the new feature!)
show_new_feature = client.variation("new-feature", context, True)
```

### 5. Test Both Paths

Always test what happens when the flag is ON **and** when it's OFF:

```python
# Test file: test_checkout.py

def test_new_checkout_when_flag_enabled(mock_flags):
    mock_flags.set("new-checkout", True)
    response = client.get("/checkout")
    assert "New Checkout" in response.text

def test_old_checkout_when_flag_disabled(mock_flags):
    mock_flags.set("new-checkout", False)
    response = client.get("/checkout")
    assert "Classic Checkout" in response.text
```

### 6. Monitor Flag Usage

Track how many times each flag is evaluated and clean up unused flags:

```
Flag: release-new-dashboard
  - Created: 2024-01-15
  - Last evaluated: 2024-01-20 (6 months ago!)
  - Status: Enabled for 100% of users
  
  ⚠️ This flag should be removed — it's always ON and hasn't been
     evaluated recently. Clean it up!
```

## 🔄 Feature Flags in CI/CD Pipelines

Feature flags can also be used **inside your CI/CD pipeline** itself:

```yaml
# .github/workflows/deploy.yml
name: Deploy

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4

    - name: Check deployment flag
      id: check-flag
      run: |
        # Query feature flag service to see if deployments are enabled
        DEPLOY_ENABLED=$(curl -s \
          -H "Authorization: Bearer ${{ secrets.FLAG_API_KEY }}" \
          "https://flags.mycompany.com/api/flags/enable-production-deploy" \
          | jq -r '.enabled')
        echo "deploy_enabled=$DEPLOY_ENABLED" >> $GITHUB_OUTPUT

    - name: Deploy to production
      if: steps.check-flag.outputs.deploy_enabled == 'true'
      run: |
        echo "Deploying to production..."
        # deployment commands here

    - name: Skip deployment
      if: steps.check-flag.outputs.deploy_enabled != 'true'
      run: |
        echo "⚠️ Production deployments are currently disabled via feature flag"
        echo "Enable the 'enable-production-deploy' flag to resume deployments"
```

This lets you **freeze deployments** during incidents or maintenance windows without touching your pipeline code!

---

## 📝 Summary

| Concept | Key Takeaway |
|---------|-------------|
| **Feature Flags** | Turn features on/off without redeploying |
| **Release Flags** | Hide incomplete features; remove after rollout |
| **Experiment Flags** | A/B test different variations |
| **Ops Flags** | Kill switches for incidents |
| **LaunchDarkly** | Best commercial tool, most features, SaaS only |
| **Unleash** | Best open-source, self-hostable, MIT license |
| **Flagsmith** | Open-source + great UI + remote config |
| **Best Practice** | Clean up flags, default to OFF, test both paths |

Feature flags are one of those things that seem simple on the surface but are **incredibly powerful** in practice. They change the way you think about deployments — you stop worrying about "will this break production?" because you can always just turn it off.
