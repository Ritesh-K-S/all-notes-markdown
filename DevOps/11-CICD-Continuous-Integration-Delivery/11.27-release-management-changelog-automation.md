## 📌 What is Release Management?

**Release management** is the process of planning, scheduling, and controlling the movement of software from development to production. It answers questions like:

- When do we release?
- What's included in this release?
- How do we track what changed?
- How do we communicate changes to users?
- How do we roll back if something goes wrong?

Think of it like launching a rocket 🚀:
- You don't just launch whenever you feel like it
- You have a **checklist**, a **countdown**, and a **plan**
- Everyone knows what's happening and when
- You have an **abort procedure** if something goes wrong

## 🔢 Semantic Versioning (SemVer)

Before we talk about releases, we need to talk about **version numbers**. The most widely used versioning scheme is **Semantic Versioning** (SemVer).

### The Format

```
MAJOR.MINOR.PATCH

Examples:
  1.0.0
  2.3.1
  10.4.22
```

### What Each Number Means

| Part | When to Increment | Example |
|------|-------------------|---------|
| **MAJOR** | Breaking changes (old code won't work with new version) | 1.0.0 → **2**.0.0 |
| **MINOR** | New features (backwards compatible — old code still works) | 1.0.0 → 1.**1**.0 |
| **PATCH** | Bug fixes (backwards compatible — nothing new, just fixes) | 1.0.0 → 1.0.**1** |

### Real-World Examples

```
v1.0.0 → v1.0.1  (Fixed a typo in error messages — PATCH)
v1.0.1 → v1.1.0  (Added dark mode — MINOR)
v1.1.0 → v1.1.1  (Fixed dark mode on Safari — PATCH)
v1.1.1 → v2.0.0  (Completely redesigned API, old clients will break — MAJOR)
```

### Pre-Release & Build Metadata

SemVer also supports pre-release tags and build metadata:

```
1.0.0-alpha          ← Alpha release (very early, unstable)
1.0.0-alpha.1        ← First alpha iteration
1.0.0-beta           ← Beta release (feature-complete, might have bugs)
1.0.0-beta.2         ← Second beta iteration
1.0.0-rc.1           ← Release Candidate 1 (almost ready!)
1.0.0-rc.2           ← Release Candidate 2 (fixed something from rc.1)
1.0.0                ← Stable release! 🎉

1.0.0+build.123      ← Build metadata (doesn't affect versioning)
1.0.0+20240115       ← Build date metadata
```

### Release Lifecycle

```
alpha → beta → rc (release candidate) → stable

alpha:   Early testing, expect bugs, features may change
beta:    Feature complete, testing for bugs
rc:      Almost ready, final testing
stable:  Production-ready, fully tested
```

## 🏷️ Git Tags & GitHub/GitLab Releases

### Git Tags — Marking Release Points

A **Git tag** is like a bookmark on a specific commit — it marks a point in your Git history as a release.

```bash
# Create a lightweight tag
git tag v1.0.0

# Create an annotated tag (recommended — includes metadata)
git tag -a v1.0.0 -m "Release version 1.0.0 - Initial stable release"

# Tag a specific commit (not just HEAD)
git tag -a v1.0.0 -m "Release 1.0.0" abc1234

# List all tags
git tag
git tag -l "v1.*"     # List only v1.x tags

# Show tag details
git show v1.0.0

# Push tags to remote
git push origin v1.0.0        # Push a specific tag
git push origin --tags         # Push ALL tags

# Delete a tag
git tag -d v1.0.0              # Delete locally
git push origin --delete v1.0.0  # Delete from remote
```

### Lightweight vs Annotated Tags

| Type | Command | Metadata | Use For |
|------|---------|----------|---------|
| **Lightweight** | `git tag v1.0.0` | Just a pointer to a commit | Temporary, personal marks |
| **Annotated** | `git tag -a v1.0.0 -m "msg"` | Author, date, message, can be signed | **Releases** (always use this) |

### GitHub Releases

GitHub Releases sit on top of Git tags and add:
- **Release notes** (description of what changed)
- **Binary assets** (downloadable files attached to the release)
- **Pre-release marking** (flag it as alpha/beta)

#### Creating a Release via GitHub CLI

```bash
# Install GitHub CLI
# https://cli.github.com/

# Create a release
gh release create v1.0.0 \
  --title "v1.0.0 - Initial Release" \
  --notes "## What's New
  - Added user authentication
  - Added dashboard
  - Fixed login bug (#42)
  
  ## Breaking Changes
  - None
  
  ## Contributors
  - @alice
  - @bob"

# Create a pre-release
gh release create v2.0.0-beta.1 \
  --title "v2.0.0 Beta 1" \
  --notes "Beta release for testing" \
  --prerelease

# Create a release with binary assets attached
gh release create v1.0.0 \
  --title "v1.0.0" \
  --notes "Release notes here" \
  ./dist/myapp-linux-amd64 \
  ./dist/myapp-darwin-amd64 \
  ./dist/myapp-windows-amd64.exe

# Create a release and auto-generate notes from commits
gh release create v1.1.0 --generate-notes

# List releases
gh release list

# Download release assets
gh release download v1.0.0
```

#### Creating a Release via GitHub Actions

```yaml
name: Release

on:
  push:
    tags:
      - 'v*'       # Trigger on any tag starting with 'v'

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4

    - name: Build
      run: |
        # Build your application
        make build

    - name: Create GitHub Release
      uses: softprops/action-gh-release@v2
      with:
        files: |
          dist/myapp-linux-amd64
          dist/myapp-darwin-amd64
          dist/myapp-windows-amd64.exe
        generate_release_notes: true
        prerelease: ${{ contains(github.ref, 'beta') || contains(github.ref, 'alpha') }}
      env:
        GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

**How this works:**
1. Developer creates and pushes a tag: `git tag -a v1.2.0 -m "Release" && git push origin v1.2.0`
2. GitHub Actions detects the `v*` tag push
3. Builds the project
4. Creates a GitHub Release with auto-generated notes and attached binaries

### GitLab Releases

```yaml
# .gitlab-ci.yml
release:
  stage: release
  image: registry.gitlab.com/gitlab-org/release-cli:latest
  rules:
    - if: $CI_COMMIT_TAG

        - name: "Docker Image"

```

## 📋 Changelogs — What Changed and Why

### What is a Changelog?

A **changelog** is a file (usually `CHANGELOG.md`) that lists all notable changes in each release. It's meant for **humans** — your users, your team, your future self.

### The Keep a Changelog Format

The most popular format is [Keep a Changelog](https://keepachangelog.com/):

```markdown
# Changelog

All notable changes to this project will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/),
and this project adheres to [Semantic Versioning](https://semver.org/).

## [Unreleased]
### Added
- Dark mode support

## [1.2.0] - 2024-03-15
### Added
- User profile page
- Export data to CSV
- API rate limiting

### Changed
- Improved dashboard loading speed by 40%
- Updated dependencies to latest versions

### Fixed
- Fixed login failure on Safari (#142)
- Fixed timezone display issue (#156)

### Security
- Updated jsonwebtoken to fix CVE-2024-xxxx

## [1.1.0] - 2024-02-01
### Added
- Two-factor authentication
- Email notifications

### Deprecated
- The `/api/v1/users` endpoint will be removed in v2.0.0. Use `/api/v2/users` instead.

### Removed
- Removed legacy export format

## [1.0.0] - 2024-01-01
### Added
- Initial release
- User registration and login
- Dashboard
- REST API

[Unreleased]: https://github.com/user/repo/compare/v1.2.0...HEAD
[1.2.0]: https://github.com/user/repo/compare/v1.1.0...v1.2.0
[1.1.0]: https://github.com/user/repo/compare/v1.0.0...v1.1.0
[1.0.0]: https://github.com/user/repo/releases/tag/v1.0.0
```

### Changelog Categories

| Category | What It Covers |
|----------|---------------|
| **Added** | New features |
| **Changed** | Changes in existing functionality |
| **Deprecated** | Features that will be removed soon |
| **Removed** | Features that were removed |
| **Fixed** | Bug fixes |
| **Security** | Security vulnerability fixes |

## 🤖 Automated Changelog Generation

Manually writing changelogs is boring and error-prone. Let's automate it!

### Conventional Commits — The Foundation

**Conventional Commits** is a commit message format that makes changelog automation possible:

```
<type>(<scope>): <description>

[optional body]

[optional footer(s)]
```

#### Types

| Type | Meaning | SemVer Impact |
|------|---------|--------------|
| `feat` | A new feature | **MINOR** bump |
| `fix` | A bug fix | **PATCH** bump |
| `docs` | Documentation changes | No bump |
| `style` | Code style (formatting, semicolons) | No bump |
| `refactor` | Code restructuring (no feature/fix) | No bump |
| `perf` | Performance improvement | **PATCH** bump |
| `test` | Adding/fixing tests | No bump |
| `chore` | Build tools, dependencies, CI changes | No bump |
| `ci` | CI/CD changes | No bump |
| `build` | Build system changes | No bump |
| `revert` | Reverts a previous commit | Depends |

**Breaking changes** → **MAJOR** bump (use `!` or `BREAKING CHANGE:` footer)

#### Examples

```bash
# Feature (MINOR bump)
git commit -m "feat(auth): add Google OAuth login"

# Bug fix (PATCH bump)
git commit -m "fix(dashboard): correct timezone display for UTC+12"

# Breaking change (MAJOR bump)
git commit -m "feat(api)!: redesign user endpoint response format

BREAKING CHANGE: The /api/users endpoint now returns an object instead of an array.
Old: [{ id: 1, name: 'Alice' }]
New: { users: [{ id: 1, name: 'Alice' }], total: 1 }"

# Documentation (no bump)
git commit -m "docs: update API documentation for v2 endpoints"

# Multiple scopes
git commit -m "fix(auth,session): handle expired tokens gracefully"

# Commit with body
git commit -m "feat(notifications): add email digest feature

Users can now opt-in to receive a daily email digest
summarizing their notifications instead of individual emails.

Closes #234"
```

### Enforcing Conventional Commits with commitlint

**commitlint** is a tool that checks if your commit messages follow the Conventional Commits format.

```bash
# Install commitlint
npm install --save-dev @commitlint/{cli,config-conventional}
```

```javascript
// commitlint.config.js
module.exports = {
  extends: ['@commitlint/config-conventional'],
  rules: {
    'type-enum': [2, 'always', [
      'feat', 'fix', 'docs', 'style', 'refactor',
      'perf', 'test', 'chore', 'ci', 'build', 'revert'
    ]],
    'subject-max-length': [2, 'always', 100],
    'body-max-line-length': [2, 'always', 200]
  }
};
```

Set up a Git hook to run it automatically:

```bash
# Install husky (Git hooks manager)
npm install --save-dev husky

# Initialize husky
npx husky init

# Add commitlint hook
echo "npx --no -- commitlint --edit \$1" > .husky/commit-msg
```

Now every commit will be validated:

```bash
git commit -m "updated stuff"
# ❌ Error: subject may not be empty
# ❌ Error: type may not be empty

git commit -m "feat(auth): add password reset flow"
# ✅ Commit passes validation!
```

## 🛠️ Tool 1: semantic-release — Fully Automated Releases

**semantic-release** is the gold standard for automated releases. It does EVERYTHING:

1. Analyzes commit messages (Conventional Commits)
2. Determines the next version number
3. Generates the changelog
4. Creates a Git tag
5. Creates a GitHub/GitLab release
6. Publishes to npm/PyPI/Docker Hub/etc.

### Installation

```bash
npm install --save-dev semantic-release \
  @semantic-release/changelog \
  @semantic-release/git
```

### Configuration

```json
// .releaserc.json
{
  "branches": ["main"],
  "plugins": [
    "@semantic-release/commit-analyzer",
    "@semantic-release/release-notes-generator",
    [
      "@semantic-release/changelog",
      {
        "changelogFile": "CHANGELOG.md"
      }
    ],
    [
      "@semantic-release/npm",
      {
        "npmPublish": false
      }
    ],
    [
      "@semantic-release/git",
      {
        "assets": ["CHANGELOG.md", "package.json"],
        "message": "chore(release): ${nextRelease.version} [skip ci]\n\n${nextRelease.notes}"
      }
    ],
    "@semantic-release/github"
  ]
}
```

### GitHub Actions Integration

```yaml
# .github/workflows/release.yml
name: Release

on:
  push:
    branches: [main]

permissions:
  contents: write
  issues: write
  pull-requests: write

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
      with:
        fetch-depth: 0      # Full history needed for changelog
        persist-credentials: false

    - uses: actions/setup-node@v4
      with:
        node-version: 20

    - name: Install dependencies
      run: npm ci

    - name: Run tests
      run: npm test

    - name: Release
      env:
        GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        NPM_TOKEN: ${{ secrets.NPM_TOKEN }}    # Only if publishing to npm
      run: npx semantic-release
```

### How It Works — Step by Step

```
1. Developer pushes commits to main:
   - "feat(auth): add SSO support"
   - "fix(dashboard): fix chart rendering"
   - "docs: update README"

2. semantic-release analyzes commits:
   - feat → MINOR bump
   - fix  → PATCH bump
   - docs → no bump
   - Highest = MINOR → next version = 1.2.0 (if current is 1.1.x)

3. Generates changelog:
   ## [1.2.0] - 2024-03-15
   ### Features
   - **auth:** add SSO support (abc1234)
   ### Bug Fixes
   - **dashboard:** fix chart rendering (def5678)

4. Creates Git tag: v1.2.0

5. Creates GitHub Release with generated notes

6. Publishes to npm (if configured)

7. Commits updated CHANGELOG.md and package.json back to main
```

The entire process is **completely automated**. No human intervention needed!

## 🛠️ Tool 2: standard-version / commit-and-tag-version

If you want more control (not fully automated — you decide when to release), use **commit-and-tag-version** (formerly standard-version):

```bash
npm install --save-dev commit-and-tag-version
```

```json
// package.json
{
  "scripts": {
    "release": "commit-and-tag-version",
    "release:minor": "commit-and-tag-version --release-as minor",
    "release:major": "commit-and-tag-version --release-as major",
    "release:patch": "commit-and-tag-version --release-as patch",
    "release:alpha": "commit-and-tag-version --prerelease alpha"
  }
}
```

```bash
# When you're ready to release:
npm run release

# What happens:
# 1. Reads commits since last tag
# 2. Determines version bump (from commit types)
# 3. Updates CHANGELOG.md
# 4. Bumps version in package.json
# 5. Creates a Git commit and tag

# Then push:
git push --follow-tags origin main
```

## 🛠️ Tool 3: release-please (by Google)

**release-please** is Google's tool for automated releases. It works by creating a **Release PR** that accumulates changes:

```yaml
# .github/workflows/release-please.yml
name: release-please

on:
  push:
    branches: [main]

permissions:
  contents: write
  pull-requests: write

jobs:
  release-please:
    runs-on: ubuntu-latest
    steps:
    - uses: googleapis/release-please-action@v4
      with:
        release-type: node     # or: python, go, java, etc.
        token: ${{ secrets.GITHUB_TOKEN }}
```

### How release-please Works

```
1. Developer pushes commits to main
2. release-please creates/updates a "Release PR" automatically:
   
   PR Title: "chore(main): release 1.2.0"
   PR Body:  Contains the generated changelog
   
3. The Release PR accumulates changes over time
4. When you're ready to release → MERGE the Release PR
5. release-please creates the Git tag and GitHub Release automatically
```

This gives you the **automation** of semantic-release but with a **manual approval step** (merging the PR).

## 🛠️ Tool 4: git-cliff — Universal Changelog Generator

**git-cliff** is a fast, flexible changelog generator written in Rust. It works with **any Git project** (not just Node.js).

### Installation

```bash
# macOS
brew install git-cliff

# Linux (cargo)
cargo install git-cliff

# npm
npm install -g git-cliff

# Docker
docker run -v "$(pwd)":/app orhunp/git-cliff
```

### Usage

```bash
# Generate changelog
git-cliff -o CHANGELOG.md

# Generate changelog for a specific range
git-cliff v1.0.0..v2.0.0

# Generate just the latest release notes
git-cliff --latest

# Generate unreleased changes
git-cliff --unreleased

# Output to stdout (for CI/CD)
git-cliff --latest --strip header
```

### Configuration

```toml
# cliff.toml
[changelog]
header = """
# Changelog\n
All notable changes to this project will be documented in this file.\n
"""
body = """
{% if version %}\
    ## [{{ version | trim_start_matches(pat="v") }}] - {{ timestamp | date(format="%Y-%m-%d") }}
{% else %}\
    ## [Unreleased]
{% endif %}\
{% for group, commits in commits | group_by(attribute="group") %}
    ### {{ group | striptags | trim | upper_first }}
    {% for commit in commits %}
        - {% if commit.scope %}**{{ commit.scope }}:** {% endif %}\
            {{ commit.message | upper_first }}\
            ([{{ commit.id | truncate(length=7, end="") }}](https://github.com/user/repo/commit/{{ commit.id }}))\
    {% endfor %}
{% endfor %}\n
"""
trim = true

[git]
conventional_commits = true
filter_unconventional = true
split_commits = false

commit_parsers = [
    { message = "^feat", group = "<!-- 0 -->🚀 Features" },
    { message = "^fix", group = "<!-- 1 -->🐛 Bug Fixes" },
    { message = "^doc", group = "<!-- 3 -->📚 Documentation" },
    { message = "^perf", group = "<!-- 4 -->⚡ Performance" },
    { message = "^refactor", group = "<!-- 2 -->🔧 Refactoring" },
    { message = "^style", group = "<!-- 5 -->🎨 Styling" },
    { message = "^test", group = "<!-- 6 -->🧪 Testing" },
    { message = "^chore\\(release\\)", skip = true },
    { message = "^chore", group = "<!-- 7 -->⚙️ Miscellaneous" },
]
```

### git-cliff in GitHub Actions

```yaml
name: Generate Changelog

on:
  push:
    tags:
      - 'v*'

jobs:
  changelog:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
      with:
        fetch-depth: 0

    - name: Generate changelog
      uses: orhun/git-cliff-action@v3
      id: changelog
      with:
        config: cliff.toml
        args: --latest --strip header

    - name: Create Release
      uses: softprops/action-gh-release@v2
      with:
        body: ${{ steps.changelog.outputs.content }}
      env:
        GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

## 📊 Tool Comparison

| Feature | semantic-release | release-please | commit-and-tag-version | git-cliff |
|---------|-----------------|----------------|----------------------|-----------|
| **Automation Level** | Fully automatic | Semi-auto (PR approval) | Manual trigger | Manual trigger |
| **Changelog** | ✅ Auto-generated | ✅ Auto-generated | ✅ Auto-generated | ✅ Auto-generated |
| **Version Bump** | ✅ Automatic | ✅ Automatic | ✅ Automatic | ❌ Just changelog |
| **Git Tag** | ✅ Automatic | ✅ Automatic | ✅ Automatic | ❌ You do it |
| **GitHub Release** | ✅ Automatic | ✅ Automatic | ❌ Manual | ❌ Manual |
| **npm Publish** | ✅ | ❌ | ❌ | ❌ |
| **Language** | Node.js | Any | Node.js | Any (Rust) |
| **Approval Step** | None (auto) | PR merge | Manual CLI | Manual CLI |
| **Best For** | Libraries, packages | Teams wanting control | Simple projects | Any project |

### Quick Decision Guide

```
Want EVERYTHING automated? → semantic-release
Want automation BUT with approval? → release-please  
Want simple, CLI-based? → commit-and-tag-version
Just need a changelog? → git-cliff
```

## 🔄 Complete Release Workflow Example

Here's a complete, production-ready release workflow combining everything:

```yaml
# .github/workflows/release.yml
name: Release Pipeline

on:
  push:
    branches: [main]
  workflow_dispatch:
    inputs:
      release_type:
        description: 'Release type (auto/major/minor/patch)'
        required: false
        default: 'auto'

permissions:
  contents: write
  pull-requests: write
  issues: write

jobs:
  # Step 1: Run tests
  test:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-node@v4
      with:
        node-version: 20
    - run: npm ci
    - run: npm test
    - run: npm run lint

  # Step 2: Create release
  release:
    needs: test
    runs-on: ubuntu-latest
    outputs:
      released: ${{ steps.release.outputs.new_release_published }}
      version: ${{ steps.release.outputs.new_release_version }}
    steps:
    - uses: actions/checkout@v4
      with:
        fetch-depth: 0
        persist-credentials: false

    - uses: actions/setup-node@v4
      with:
        node-version: 20

    - run: npm ci

    - name: Semantic Release
      id: release
      env:
        GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
      run: npx semantic-release

  # Step 3: Build and publish Docker image (only if a release was created)
  docker:
    needs: release
    if: needs.release.outputs.released == 'true'
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4

    - name: Login to Docker Hub
      uses: docker/login-action@v3
      with:
        username: ${{ secrets.DOCKER_USERNAME }}
        password: ${{ secrets.DOCKER_PASSWORD }}

    - name: Build and Push
      uses: docker/build-push-action@v5
      with:
        push: true
        tags: |
          myorg/myapp:${{ needs.release.outputs.version }}
          myorg/myapp:latest

  # Step 4: Notify the team
  notify:
    needs: [release, docker]
    if: needs.release.outputs.released == 'true'
    runs-on: ubuntu-latest
    steps:
    - name: Notify Slack
      uses: slackapi/slack-github-action@v1
      with:
        payload: |
          {
            "text": "🚀 *v${{ needs.release.outputs.version }}* has been released!\nSee: https://github.com/${{ github.repository }}/releases/tag/v${{ needs.release.outputs.version }}"
          }
      env:
        SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK }}
```

### The Flow

```
Developer pushes to main
        │
        ▼
    Run tests ──→ Tests pass?
        │              │
        ▼              ▼ (fail → stop)
  Analyze commits
        │
        ▼
  Determine version bump
  (feat → minor, fix → patch)
        │
        ▼
  Generate CHANGELOG.md
        │
        ▼
  Create Git tag + GitHub Release
        │
        ▼
  Build Docker image with version tag
        │
        ▼
  Notify team on Slack
        │
        ▼
     Done! 🎉
```

## ✅ Release Management Best Practices

### 1. Never Manually Write Version Numbers

Let your tools determine the version from commit messages. This eliminates "what version should this be?" debates.

### 2. Use Protected Branches

```
main branch → protected
  ✅ Require pull request reviews
  ✅ Require status checks to pass
  ✅ Require linear history
  ❌ Allow force pushes
```

### 3. Release from main Only

```
feature-branch → PR → main → auto-release

Never release from feature branches!
```

### 4. Tag Docker Images with Versions

```bash
# GOOD
myapp:1.2.3        # Specific version
myapp:1.2           # Latest patch of 1.2
myapp:latest        # Latest stable

# BAD
myapp:latest        # Only tag — which version is this?!
```

### 5. Keep a Human-Readable Changelog

Even with automation, make sure your changelog makes sense to **humans**:

```markdown
## [2.1.0] - 2024-03-15

### 🚀 Features
- **auth:** Users can now sign in with Google and GitHub OAuth
- **dashboard:** Added real-time notifications bell icon

### 🐛 Bug Fixes  
- **checkout:** Fixed cart items disappearing on page refresh (#234)
- **search:** Fixed search returning stale results (#256)
```

### 6. Document Breaking Changes Clearly

```markdown
## [3.0.0] - 2024-04-01

### ⚠️ BREAKING CHANGES

- **API:** The `/api/v2/users` endpoint now returns paginated results.
  
  **Before (v2.x):**
  ```json
  [{ "id": 1, "name": "Alice" }, { "id": 2, "name": "Bob" }]
  ```
  
  **After (v3.x):**
  ```json
  { "data": [{ "id": 1, "name": "Alice" }], "page": 1, "total": 50 }
  ```

  **Migration:** Update your client to handle the new response format.
  See [Migration Guide](docs/migration-v3.md).
```

## 📝 Summary

| Concept | Key Takeaway |
|---------|-------------|
| **Semantic Versioning** | MAJOR.MINOR.PATCH — breaking.feature.fix |
| **Git Tags** | Use annotated tags to mark releases |
| **Conventional Commits** | `feat:`, `fix:`, `docs:` — enables automation |
| **commitlint** | Enforces commit message format |
| **semantic-release** | Fully automated versioning + changelog + release |
| **release-please** | Automated with manual approval (Release PR) |
| **git-cliff** | Flexible changelog generator for any project |
| **Best Practices** | Automate everything, clean changelogs, tag Docker images |

The key insight is: **good commit messages → automatic everything**. If your team follows Conventional Commits, the entire release process — version bumping, changelog, Git tags, GitHub releases, Docker images, notifications — can be fully automated.
