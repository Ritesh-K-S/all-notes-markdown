## 🤔 What is Jenkins? — The Basics

### The Simple Answer:

Jenkins is a **server** that sits somewhere (your machine, a VM, a Kubernetes cluster) and **watches your Git repositories**. When you push code, Jenkins **automatically** builds, tests, and deploys your application — according to the pipeline you've defined.

Think of Jenkins as a **robot assistant** that does all the tedious, repetitive work for you:

```
Without Jenkins:
  Developer pushes code → manually builds → manually tests → manually deploys
  (Boring, error-prone, slow)

With Jenkins:
  Developer pushes code → Jenkins does EVERYTHING automatically
  (Developer goes for coffee ☕)
```

### Why Jenkins is So Popular:

| Reason | Details |
|--------|---------|
| **Free & Open Source** | No license costs. Community-driven. Thousands of contributors. |
| **Massive Plugin Ecosystem** | 1,800+ plugins — integrates with basically everything |
| **Language/Platform Agnostic** | Works with Java, Node.js, Python, Go, .NET, Docker, Kubernetes — anything |
| **Pipeline as Code** | Define your entire CI/CD process in a `Jenkinsfile` stored in Git |
| **Distributed Builds** | Run builds on multiple machines (agents) for speed and parallelism |
| **Battle-Tested** | Used since 2011. Trusted by Netflix, LinkedIn, eBay, NASA, and thousands more |
| **Self-Hosted** | You own the server, you control the data — no vendor lock-in |

### Jenkins in the CI/CD Ecosystem:

```
Developer pushes code to GitHub/GitLab/Bitbucket
           │
           ▼
    ┌──────────────┐
    │   JENKINS    │  ← Jenkins receives a webhook notification
    │   Server     │
    │              │
    │  "Oh, new    │
    │   code?      │
    │   Let me     │
    │   handle     │
    │   this!"     │
    └──────┬───────┘
           │
           ▼
    ┌──────────────────────────────────────────────┐
    │  Pipeline executes automatically:            │
    │  1. Pull latest code                         │
    │  2. Install dependencies                     │
    │  3. Build the application                    │
    │  4. Run tests                                │
    │  5. Build Docker image                       │
    │  6. Push to registry                         │
    │  7. Deploy to staging                        │
    │  8. Wait for approval                        │
    │  9. Deploy to production                     │
    └──────────────────────────────────────────────┘
```

## 🏗️ Jenkins Architecture — How It Works

Before installing, let's understand the pieces:

```
┌─────────────────────────────────────────────────────────────────┐
│                    JENKINS ARCHITECTURE                         │
│                                                                 │
│  ┌───────────────────────────────────────┐                     │
│  │        JENKINS CONTROLLER             │                     │
│  │        (Master Node)                  │                     │
│  │                                       │                     │
│  │  ┌─────────────┐  ┌──────────────┐   │                     │
│  │  │ Web UI      │  │ REST API     │   │                     │
│  │  │ (Dashboard) │  │              │   │                     │
│  │  └─────────────┘  └──────────────┘   │                     │
│  │  ┌─────────────┐  ┌──────────────┐   │                     │
│  │  │ Job Queue   │  │ Plugin       │   │                     │
│  │  │ & Scheduler │  │ Manager      │   │                     │
│  │  └─────────────┘  └──────────────┘   │                     │
│  │  ┌─────────────────────────────────┐ │                     │
│  │  │ Configuration & Credentials    │ │                     │
│  │  └─────────────────────────────────┘ │                     │
│  └──────────────────┬────────────────────┘                     │
│                     │                                           │
│          ┌──────────┼──────────┐                               │
│          │          │          │                                │
│    ┌─────▼────┐ ┌───▼────┐ ┌──▼──────┐                       │
│    │ Agent 1  │ │Agent 2 │ │ Agent 3 │                       │
│    │ (Linux)  │ │(Docker)│ │(Windows)│                       │
│    │          │ │        │ │         │                       │
│    │ Runs     │ │ Runs   │ │ Runs    │                       │
│    │ builds   │ │ builds │ │ builds  │                       │
│    └──────────┘ └────────┘ └─────────┘                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### Key Components:

| Component | What It Does |
|-----------|-------------|
| **Controller (Master)** | The brain. Manages the UI, schedules jobs, dispatches work to agents. Does NOT run builds (ideally). |
| **Agent (Worker/Slave)** | The muscle. Executes the actual build/test/deploy steps. Can be Linux, Windows, Mac, or Docker. |
| **Executor** | A "slot" on an agent that can run one job at a time. An agent can have multiple executors. |
| **Job/Project** | A defined task (e.g., "Build my app", "Run tests"). Contains the build configuration. |
| **Build** | A single execution of a job (e.g., "Build #42 — failed at test stage"). |
| **Pipeline** | A job defined as code (Jenkinsfile) — the modern way to use Jenkins. |
| **Plugin** | An add-on that extends Jenkins functionality (Git, Docker, Kubernetes, Slack, etc.). |
| **Workspace** | A directory on the agent where the code is checked out and builds happen. |

## 🛠️ Installing Jenkins — Step by Step

Jenkins can be installed in many ways. We'll cover the most common methods:

### Prerequisites

Jenkins is built with **Java**, so you need Java installed first.

| Requirement | Minimum | Recommended |
|-------------|---------|-------------|
| **Java** | Java 11 | Java 17 (LTS) |
| **RAM** | 256 MB | 1 GB+ (4 GB for production) |
| **Disk** | 1 GB | 50 GB+ (builds use lots of space) |
| **OS** | Linux, macOS, Windows | Ubuntu/Debian recommended |

---

### Method 1: Install on Ubuntu/Debian (Most Common)

This is the standard way to run Jenkins on a Linux server.

**Step 1: Install Java 17**

```bash
# Update package list
sudo apt update

# Install Java 17 (OpenJDK)
sudo apt install -y openjdk-17-jdk

# Verify installation
java -version
# Output: openjdk version "17.0.x" ...
```

**What we did:**
- `apt update` — Refreshed the package list so we get the latest versions
- `openjdk-17-jdk` — Installed Java 17 (the open-source version from Oracle/RedHat)
- `java -version` — Confirmed Java is installed correctly

**Step 2: Add the Jenkins Repository**

```bash
# Add Jenkins GPG key (to verify package authenticity)
sudo wget -O /usr/share/keyrings/jenkins-keyring.asc \
  https://pkg.jenkins.io/debian-stable/jenkins.io-2023.key

# Add Jenkins apt repository
echo "deb [signed-by=/usr/share/keyrings/jenkins-keyring.asc]" \
  https://pkg.jenkins.io/debian-stable binary/ | sudo tee \
  /etc/apt/sources.list.d/jenkins.list > /dev/null
```

**What we did:**
- Downloaded Jenkins' GPG key — this ensures the Jenkins packages we download are authentic and haven't been tampered with
- Added the official Jenkins repository to our apt sources — so `apt install jenkins` knows where to get it from

**Step 3: Install Jenkins**

```bash
# Update package list (now includes Jenkins repo)
sudo apt update

# Install Jenkins
sudo apt install -y jenkins

# Jenkins is now installed!
```

**Step 4: Start Jenkins**

```bash
# Start Jenkins service
sudo systemctl start jenkins

# Enable Jenkins to start on boot
sudo systemctl enable jenkins

# Check if Jenkins is running
sudo systemctl status jenkins
```

**Expected output:**

```
● jenkins.service - Jenkins Continuous Integration Server
     Loaded: loaded (/lib/systemd/system/jenkins.service; enabled)
     Active: active (running) since Mon 2026-05-29 10:30:00 UTC
   Main PID: 12345 (java)
     Memory: 512.0M
        CPU: 15.234s
```

**Step 5: Open the Firewall (if needed)**

```bash
# Jenkins runs on port 8080 by default
sudo ufw allow 8080
sudo ufw status
```

**Step 6: Access Jenkins in Browser**

Open your browser and go to:

```
http://your-server-ip:8080
```

Or if you're running locally:

```
http://localhost:8080
```

You'll see the **"Unlock Jenkins"** page. We'll set this up in the next section!

### Method 2: Install with Docker (Fastest & Cleanest)

This is the **recommended way** — it keeps Jenkins isolated in a container and is easy to manage.

**Step 1: Run Jenkins with Docker**

```bash
# Create a Docker volume for Jenkins data (so data survives container restarts)
docker volume create jenkins-data

# Run Jenkins
docker run -d \
  --name jenkins \
  --restart=unless-stopped \
  -p 8080:8080 \
  -p 50000:50000 \
  -v jenkins-data:/var/jenkins_home \
  -v /var/run/docker.sock:/var/run/docker.sock \
  jenkins/jenkins:lts-jdk17
```

**Let's break down every flag:**

| Flag | What It Does |
|------|-------------|
| `-d` | Run in detached mode (background) |
| `--name jenkins` | Give the container a name (easier to manage) |
| `--restart=unless-stopped` | Automatically restart Jenkins if it crashes or the server reboots |
| `-p 8080:8080` | Map port 8080 (Jenkins web UI) from container to host |
| `-p 50000:50000` | Map port 50000 (JNLP agent communication) from container to host |
| `-v jenkins-data:/var/jenkins_home` | Persist Jenkins data in a Docker volume |
| `-v /var/run/docker.sock:/var/run/docker.sock` | Allow Jenkins to use Docker (for Docker-in-Docker builds) |
| `jenkins/jenkins:lts-jdk17` | Use the official Jenkins LTS image with Java 17 |

**Step 2: Check if Jenkins is running**

```bash
# View container status
docker ps

# View Jenkins logs
docker logs jenkins
```

**Step 3: Access at `http://localhost:8080`**

### Method 3: Install with Docker Compose (Production-Ready)

For a more organized setup, use Docker Compose:

**Create a `docker-compose.yml` file:**

```yaml
version: '3.8'

services:
  jenkins:
    image: jenkins/jenkins:lts-jdk17
    container_name: jenkins
    restart: unless-stopped
    ports:
      - "8080:8080"       # Jenkins Web UI
      - "50000:50000"     # Agent communication
    volumes:
      - jenkins-data:/var/jenkins_home
      - /var/run/docker.sock:/var/run/docker.sock
    environment:
      - JAVA_OPTS=-Xmx2g -Xms512m    # Give Jenkins 2GB max RAM
    networks:
      - jenkins-network

volumes:
  jenkins-data:
    driver: local

networks:
  jenkins-network:
    driver: bridge
```

**Run it:**

```bash
# Start Jenkins
docker compose up -d

# Check status
docker compose ps

# View logs
docker compose logs -f jenkins

# Stop Jenkins
docker compose down

# Stop and DELETE all data (careful!)
# docker compose down -v
```

**What each part of docker-compose.yml does:**

| Section | What It Does |
|---------|-------------|
| `image: jenkins/jenkins:lts-jdk17` | Uses the official LTS image with Java 17 |
| `restart: unless-stopped` | Jenkins restarts automatically if it crashes |
| `ports` | Exposes port 8080 (UI) and 50000 (agents) |
| `volumes: jenkins-data` | Data is stored in a named volume — survives container restarts/rebuilds |
| `JAVA_OPTS` | Configures Java memory — 512MB initial, 2GB max |
| `networks` | Creates an isolated network for Jenkins |

## ⚙️ Essential Configuration After Installation

### 1. Configure Global Tools

Go to **Manage Jenkins → Tools**

Here you tell Jenkins where to find tools like JDK, Maven, Node.js, Git, Docker, etc.

**Example — Adding Node.js:**

```
Manage Jenkins → Tools → NodeJS installations → Add NodeJS

  Name:                    NodeJS-20
  Install automatically:   ✅ (checked)
  Version:                 20.x
```

**Example — Adding Maven:**

```
Manage Jenkins → Tools → Maven installations → Add Maven

  Name:                    Maven-3.9
  Install automatically:   ✅ (checked)
  Version:                 3.9.x
```

Now in your pipeline, you can reference these tools:

```groovy
pipeline {
    agent any
    tools {
        nodejs 'NodeJS-20'     // Uses the NodeJS we configured
        maven 'Maven-3.9'     // Uses the Maven we configured
    }
    stages {
        stage('Build') {
            steps {
                sh 'node --version'   // Will print v20.x
                sh 'mvn --version'    // Will print 3.9.x
            }
        }
    }
}
```

### 2. Configure Credentials

Go to **Manage Jenkins → Credentials → System → Global credentials**

Jenkins needs credentials to:
- Pull code from private Git repositories
- Push Docker images to registries
- Deploy to servers (SSH keys)
- Access cloud providers (AWS, Azure, GCP)

**Types of credentials Jenkins supports:**

| Type | Use Case | Example |
|------|----------|---------|
| **Username with password** | Git repos, Docker Hub, web services | GitHub username + token |
| **SSH Username with private key** | SSH into servers, Git over SSH | `id_rsa` private key |
| **Secret text** | API tokens, single secrets | Slack webhook URL |
| **Secret file** | Kubeconfig, service account JSON | `kubeconfig.yaml` |
| **Certificate** | Client certificates for mutual TLS | `.p12` or `.pfx` files |

**Example — Adding GitHub credentials:**

```
1. Go to: Manage Jenkins → Credentials → System → Global credentials
2. Click "Add Credentials"
3. Fill in:
   - Kind:        Username with password
   - Scope:       Global
   - Username:    your-github-username
   - Password:    your-github-personal-access-token (NOT your password!)
   - ID:          github-credentials
   - Description: GitHub Access Token
4. Click "Create"
```

**Using credentials in a pipeline:**

```groovy
pipeline {
    agent any
    stages {
        stage('Clone') {
            steps {
                // Jenkins automatically uses the credentials to authenticate
                git credentialsId: 'github-credentials',
                    url: 'https://github.com/your-org/your-repo.git',
                    branch: 'main'
            }
        }
        stage('Docker Push') {
            steps {
                // Use credentials in a block — they're available as env vars
                withCredentials([
                    usernamePassword(
                        credentialsId: 'docker-hub-creds',
                        usernameVariable: 'DOCKER_USER',
                        passwordVariable: 'DOCKER_PASS'
                    )
                ]) {
                    sh '''
                        echo $DOCKER_PASS | docker login -u $DOCKER_USER --password-stdin
                        docker push myapp:latest
                    '''
                }
            }
        }
    }
}
```

**What's happening:**
- `credentialsId: 'github-credentials'` — References the credential we created by its ID
- `withCredentials([...])` — Makes credentials available as environment variables inside the block
- `usernameVariable` / `passwordVariable` — Jenkins injects these as env vars
- The actual values are **never printed in logs** — Jenkins masks them automatically

### 3. Configure System Settings

Go to **Manage Jenkins → System**

Important settings here:

| Setting | What It Does | Recommended Value |
|---------|-------------|-------------------|
| **Jenkins URL** | Base URL for Jenkins | `https://jenkins.yourcompany.com/` |
| **System Admin e-mail** | "From" address for notification emails | `jenkins@yourcompany.com` |
| **# of executors** | How many builds can run on the controller | `0` (don't run builds on the controller!) |
| **Quiet period** | Seconds to wait before starting a build (prevents rapid re-triggers) | `5` |
| **SCM checkout retry count** | How many times to retry cloning if it fails | `3` |

**Why set executors to 0 on the controller?**

```
BAD setup:
  Controller runs builds → Uses its resources → Slows down the UI → Can crash

GOOD setup:
  Controller: 0 executors (only manages + schedules)
  Agents: Run all the actual builds (can be scaled up/down)
```

### Creating a Pipeline Job (The Modern Way):

Now let's create a proper pipeline job — this is how you'll use Jenkins in real projects.

```
1. Click "New Item" on the dashboard
2. Enter name: "my-first-pipeline"
3. Select "Pipeline"
4. Click "OK"
```

**In the Pipeline section, paste this script:**

```groovy
pipeline {
    agent any
    
    stages {
        stage('Checkout') {
            steps {
                echo '📥 Checking out code...'
                // In a real project, you'd use: git url: 'https://github.com/...'
                echo 'Code checked out!'
            }
        }
        
        stage('Build') {
            steps {
                echo '🔨 Building the application...'
                sh 'echo "Compiling source code..."'
                sh 'sleep 2'   // Simulate build time
                echo 'Build complete!'
            }
        }
        
        stage('Test') {
            parallel {
                stage('Unit Tests') {
                    steps {
                        echo '🧪 Running unit tests...'
                        sh 'sleep 3'
                        echo '✅ Unit tests passed!'
                    }
                }
                stage('Integration Tests') {
                    steps {
                        echo '🧪 Running integration tests...'
                        sh 'sleep 4'
                        echo '✅ Integration tests passed!'
                    }
                }
            }
        }
        
        stage('Deploy') {
            steps {
                echo '🚀 Deploying to staging...'
                sh 'sleep 2'
                echo '✅ Deployed successfully!'
            }
        }
    }
    
    post {
        success {
            echo '🎉 Pipeline completed successfully!'
        }
        failure {
            echo '❌ Pipeline failed. Check the logs!'
        }
        always {
            echo '📋 Pipeline finished. Cleaning up...'
        }
    }
}
```

**Let's break this down piece by piece:**

| Section | What It Does |
|---------|-------------|
| `pipeline { }` | The entire pipeline definition — everything goes inside this |
| `agent any` | Run on any available agent (for now, the controller itself) |
| `stages { }` | Contains all the stages of the pipeline |
| `stage('Checkout') { }` | First stage — normally checks out code from Git |
| `stage('Build') { }` | Second stage — builds the application |
| `stage('Test') { parallel { } }` | Third stage — runs unit and integration tests **in parallel** |
| `stage('Deploy') { }` | Fourth stage — deploys the application |
| `post { }` | Runs after all stages complete |
| `post → success { }` | Runs only if the entire pipeline succeeded |
| `post → failure { }` | Runs only if the pipeline failed |
| `post → always { }` | Runs no matter what (cleanup, notifications) |
| `sh 'command'` | Executes a shell command |
| `echo 'message'` | Prints a message to the console |

**Click "Save" and then "Build Now".**

You'll see a beautiful **Stage View**:

```
┌────────────┐   ┌────────────┐   ┌────────────────────────┐   ┌────────────┐
```

## 🔧 Important Jenkins Environment Variables

Jenkins provides many built-in environment variables you can use in your pipelines:

| Variable | Description | Example Value |
|----------|-------------|---------------|
| `BUILD_NUMBER` | Current build number | `42` |
| `BUILD_ID` | Same as BUILD_NUMBER (for newer Jenkins) | `42` |
| `BUILD_URL` | URL to this build's page | `http://jenkins:8080/job/myapp/42/` |
| `JOB_NAME` | Name of the job | `myapp` |
| `JOB_URL` | URL to the job | `http://jenkins:8080/job/myapp/` |
| `WORKSPACE` | Path to the workspace directory | `/var/jenkins_home/workspace/myapp` |
| `JENKINS_URL` | Base URL of Jenkins | `http://jenkins:8080/` |
| `GIT_BRANCH` | Git branch being built | `origin/main` |
| `GIT_COMMIT` | Git commit SHA | `a1b2c3d4e5f6...` |

## 🔒 Basic Security Configuration

Go to **Manage Jenkins → Security**

### Authentication (Who Can Log In?)

| Option | Description | Best For |
|--------|-------------|----------|
| **Jenkins' own user database** | Jenkins manages users internally | Small teams, getting started |
| **LDAP** | Authenticate against Active Directory / LDAP | Enterprise environments |
| **GitHub OAuth** | Login with GitHub accounts | GitHub-based teams |
| **GitLab OAuth** | Login with GitLab accounts | GitLab-based teams |

### Authorization (What Can They Do?)

| Option | Description | Best For |
|--------|-------------|----------|
| **Logged-in users can do anything** | Everyone has full access | Development/testing only |
| **Matrix-based security** | Per-user permissions (fine-grained) | Small teams with different roles |
| **Role-Based Strategy** | Define roles and assign users (requires plugin) | Production environments |

**Recommended production setup:**

```
Authentication: LDAP or GitHub/GitLab OAuth
Authorization: Role-Based Strategy

Roles example:
  ├── Admin:     Full control (manage plugins, security, agents)
  ├── Developer: Create/run jobs, view builds, manage own credentials
  ├── Viewer:    Read-only access to dashboards and build logs
  └── Deploy:    Can trigger production deployments (for senior devs/leads)
```

## 📂 Jenkins Directory Structure

Understanding where Jenkins stores its files helps with backup and troubleshooting:

```
/var/jenkins_home/                    ← JENKINS_HOME (all Jenkins data)
├── config.xml                        ← Main Jenkins configuration
├── credentials.xml                   ← Encrypted credentials
├── hudson.model.UpdateCenter.xml     ← Update center configuration
├── jobs/                             ← All job configurations and build data
│   ├── hello-world/
│   │   ├── config.xml               ← Job configuration
│   │   └── builds/                   ← Build history
│   │       ├── 1/
│   │       │   ├── log              ← Console output
│   │       │   └── build.xml        ← Build metadata
│   │       └── 2/
│   └── my-first-pipeline/
│       ├── config.xml
│       └── builds/
├── nodes/                            ← Agent configurations
├── plugins/                          ← Installed plugins (.jpi files)
├── secrets/                          ← Secret keys and master key
│   ├── initialAdminPassword         ← First-time setup password
│   └── master.key                    ← Encryption master key
├── users/                            ← User configurations
├── workspace/                        ← Build workspaces
│   ├── hello-world/                  ← Where code is checked out
│   └── my-first-pipeline/
└── logs/                             ← Jenkins logs
```

**Important for backups:**

```bash
# Backup JENKINS_HOME (this backs up EVERYTHING)
tar -czf jenkins-backup-$(date +%Y%m%d).tar.gz /var/jenkins_home/

# If using Docker volume:
docker run --rm -v jenkins-data:/data -v $(pwd):/backup \
  busybox tar czf /backup/jenkins-backup.tar.gz /data
```
