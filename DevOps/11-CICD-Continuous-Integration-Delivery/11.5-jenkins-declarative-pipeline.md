## 📌 What is a Declarative Pipeline?

A **Declarative Pipeline** is the **easier, more structured** way to write a Jenkinsfile. It was introduced in Jenkins Pipeline 2.0 and is the **recommended approach** by the Jenkins community.

Think of it like this:
- **Declarative** = "I declare WHAT I want" (structured, opinionated)
- **Scripted** = "I script HOW to do it" (flexible, Groovy-based) — covered in 11.6

Declarative Pipeline uses a **predefined structure** with specific keywords. You can't just write random code anywhere — everything has its place. And that's actually a **good thing** because it makes pipelines easier to read, write, and maintain.

---

## 🏗️ The Basic Structure

Every Declarative Pipeline follows this skeleton:

```groovy
pipeline {
    agent any

    stages {
        stage('Stage Name') {
            steps {
                // your commands here
            }
        }
    }
}
```

Let's break this down:

| Block | Required? | What It Does |
|-------|-----------|-------------|
| `pipeline { }` | ✅ Yes | The outermost wrapper — everything goes inside this |
| `agent` | ✅ Yes | WHERE should this pipeline run? |
| `stages { }` | ✅ Yes | Container for all your stages |
| `stage('Name') { }` | ✅ Yes (at least 1) | A logical phase of your pipeline (Build, Test, Deploy, etc.) |
| `steps { }` | ✅ Yes | The actual commands/actions inside a stage |

## 🖥️ Agent — Where Does It Run?

The `agent` directive tells Jenkins **where** (on which machine/node) the pipeline should execute.

### Agent Options

```groovy
// Run on ANY available agent
agent any

// Don't allocate an agent at the pipeline level
// (each stage will specify its own)
agent none

// Run on a specific labeled node
agent {
    label 'linux-node'
}

// Run inside a Docker container
agent {
    docker {
        image 'node:18-alpine'
    }
}

// Run inside a Docker container with more options
agent {
    docker {
        image 'maven:3.9.4-eclipse-temurin-17'
        label 'docker-enabled'           // which node has Docker
        args '-v /tmp:/tmp'              // Docker run arguments
    }
}

// Run inside a Kubernetes pod
agent {
    kubernetes {
        yaml '''
            apiVersion: v1
            kind: Pod
            spec:
              containers:
              - name: maven
                image: maven:3.9.4
                command:
                - sleep
                args:
                - infinity
        '''
    }
}
```

### Per-Stage Agent

You can set a different agent for each stage:

```groovy
pipeline {
    agent none  // No global agent

    stages {
        stage('Build') {
            agent {
                docker { image 'maven:3.9.4' }
            }
            steps {
                sh 'mvn clean package'
            }
        }

        stage('Test') {
            agent {
                docker { image 'node:18' }
            }
            steps {
                sh 'npm test'
            }
        }

        stage('Deploy') {
            agent { label 'production-server' }
            steps {
                sh './deploy.sh'
            }
        }
    }
}
```

This is super useful when different stages need different tools/environments!

## 📦 Stages & Steps — The Core of Your Pipeline

### Stages

Stages are the **logical phases** of your pipeline. They show up as separate blocks in the Jenkins UI, making it easy to see where things are and what failed.

Common stages:

```
Checkout → Build → Test → Code Analysis → Package → Deploy → Notify
```

### Steps

Steps are the **actual commands** you run inside a stage. Here are the most common ones:

```groovy
steps {
    // Print a message
    echo 'Hello, World!'

    // Run a shell command (Linux/Mac)
    sh 'npm install'

    // Run a batch command (Windows)
    bat 'dir'

    // Run a PowerShell command (Windows)
    powershell 'Get-ChildItem'

    // Checkout code from SCM (Git)
    checkout scm

    // Git checkout specific repo
    git branch: 'main', url: 'https://github.com/user/repo.git'

    // Run inside a specific directory
    dir('my-subdirectory') {
        sh 'ls -la'
    }

    // Sleep/wait
    sleep time: 10, unit: 'SECONDS'

    // Retry a block if it fails
    retry(3) {
        sh 'curl http://my-service/health'
    }

    // Timeout a block
    timeout(time: 5, unit: 'MINUTES') {
        sh 'npm test'
    }
}
```

## 🌍 Environment Variables

You can define environment variables that are available throughout the pipeline or within specific stages.

### Global Environment Variables

```groovy
pipeline {
    agent any

    environment {
        APP_NAME = 'my-awesome-app'
        APP_VERSION = '1.0.0'
        DEPLOY_ENV = 'staging'
    }

    stages {
        stage('Build') {
            steps {
                echo "Building ${APP_NAME} version ${APP_VERSION}"
                sh "docker build -t ${APP_NAME}:${APP_VERSION} ."
            }
        }

        stage('Deploy') {
            steps {
                echo "Deploying to ${DEPLOY_ENV}..."
            }
        }
    }
}
```

### Stage-Level Environment Variables

```groovy
stage('Deploy') {
    environment {
        DEPLOY_SERVER = 'prod-server-01'
        DEPLOY_PATH = '/opt/apps/my-app'
    }
    steps {
        echo "Deploying to ${DEPLOY_SERVER}:${DEPLOY_PATH}"
    }
}
```

### Using Credentials as Environment Variables

This is the **safe way** to use passwords, tokens, and secrets in Jenkins:

```groovy
pipeline {
    agent any

    environment {
        // Jenkins Credentials ID → Environment Variable
        DOCKER_CREDS = credentials('docker-hub-credentials')    // username + password
        AWS_ACCESS_KEY = credentials('aws-access-key-id')       // secret text
        GIT_TOKEN = credentials('github-token')                 // secret text
    }

    stages {
        stage('Docker Login') {
            steps {
                // For username/password credentials, Jenkins creates:
                // DOCKER_CREDS_USR = username
                // DOCKER_CREDS_PSW = password
                sh 'echo $DOCKER_CREDS_PSW | docker login -u $DOCKER_CREDS_USR --password-stdin'
            }
        }
    }
}
```

**How `credentials()` works:**

| Credential Type | What You Get |
|----------------|-------------|
| Secret Text | `MY_VAR` = the secret value |
| Username + Password | `MY_VAR_USR` = username, `MY_VAR_PSW` = password |
| Secret File | `MY_VAR` = path to the temp file containing the secret |
| SSH Key | `MY_VAR` = path to the key file, `MY_VAR_USR` = username |

### Built-in Jenkins Environment Variables

Jenkins provides a bunch of built-in variables you can use:

```groovy
steps {
    echo "Build Number: ${env.BUILD_NUMBER}"
    echo "Build URL: ${env.BUILD_URL}"
    echo "Job Name: ${env.JOB_NAME}"
    echo "Workspace: ${env.WORKSPACE}"
    echo "Node Name: ${env.NODE_NAME}"
    echo "Branch: ${env.BRANCH_NAME}"       // Multibranch only
    echo "Git Commit: ${env.GIT_COMMIT}"
    echo "Git Branch: ${env.GIT_BRANCH}"
}
```

> 💡 To see ALL available env vars, add a stage with: `sh 'printenv | sort'`

## 🔧 Parameters — User Input Before Build

Parameters let users **provide input** before running the pipeline (like choosing a branch, environment, or version).

```groovy
pipeline {
    agent any

    parameters {
        string(name: 'BRANCH_NAME', defaultValue: 'main', description: 'Which branch to build?')
        choice(name: 'ENVIRONMENT', choices: ['dev', 'staging', 'production'], description: 'Deploy to which env?')
        booleanParam(name: 'RUN_TESTS', defaultValue: true, description: 'Should we run tests?')
        text(name: 'RELEASE_NOTES', defaultValue: '', description: 'Enter release notes')
        password(name: 'API_KEY', defaultValue: '', description: 'Enter API key')
    }

    stages {
        stage('Build') {
            steps {
                echo "Building branch: ${params.BRANCH_NAME}"
                echo "Target environment: ${params.ENVIRONMENT}"
            }
        }

        stage('Test') {
            when {
                expression { params.RUN_TESTS == true }
            }
            steps {
                echo 'Running tests...'
                sh 'npm test'
            }
        }

        stage('Deploy') {
            steps {
                echo "Deploying to ${params.ENVIRONMENT}"
            }
        }
    }
}
```

When you click **"Build with Parameters"** in Jenkins UI, you'll see a nice form with all these fields!

## 🎯 Triggers — When Should the Pipeline Run?

```groovy
pipeline {
    agent any

    triggers {
        // Poll Git repo every 5 minutes for changes
        pollSCM('H/5 * * * *')

        // Run on a fixed schedule (like cron)
        cron('H 2 * * 1-5')  // 2 AM on weekdays

        // Triggered by upstream job
        upstream(upstreamProjects: 'build-job', threshold: hudson.model.Result.SUCCESS)
    }

    stages {
        stage('Build') {
            steps {
                echo 'Building...'
            }
        }
    }
}
```

> 💡 **Note:** Webhooks (from GitHub/GitLab) are configured in Jenkins settings, not in the Jenkinsfile. The Jenkinsfile `triggers` block is for polling and scheduling.

## 🔀 Conditional Execution — The `when` Block

The `when` directive lets you **skip stages** based on conditions. This is incredibly powerful!

```groovy
pipeline {
    agent any

    stages {
        // Only run on the main branch
        stage('Deploy to Production') {
            when {
                branch 'main'
            }
            steps {
                echo 'Deploying to production...'
            }
        }

        // Only run on develop branch
        stage('Deploy to Staging') {
            when {
                branch 'develop'
            }
            steps {
                echo 'Deploying to staging...'
            }
        }

        // Only run when a specific env var is set
        stage('Release') {
            when {
                environment name: 'DEPLOY_ENV', value: 'production'
            }
            steps {
                echo 'Creating release...'
            }
        }

        // Only run when the build parameter says so
        stage('Run Performance Tests') {
            when {
                expression { params.RUN_PERF_TESTS == true }
            }
            steps {
                echo 'Running performance tests...'
            }
        }

        // Combine conditions with allOf (AND) or anyOf (OR)
        stage('Full Deploy') {
            when {
                allOf {
                    branch 'main'
                    environment name: 'DEPLOY_ENV', value: 'production'
                }
            }
            steps {
                echo 'Full production deployment!'
            }
        }

        // NOT condition
        stage('Skip on PR') {
            when {
                not {
                    changeRequest()   // Skip if this is a Pull Request
                }
            }
            steps {
                echo 'Not a PR, proceeding...'
            }
        }
    }
}
```

### All `when` Conditions

| Condition | What It Checks |
|-----------|---------------|
| `branch 'main'` | Current branch name |
| `branch pattern: 'release-*', comparator: 'GLOB'` | Branch name matches pattern |
| `environment name: 'X', value: 'Y'` | Environment variable value |
| `expression { ... }` | Any Groovy expression that returns true/false |
| `changeRequest()` | Is this a Pull Request / Merge Request? |
| `tag 'v*'` | Is this a Git tag matching the pattern? |
| `changelog '.*bug fix.*'` | Commit message matches regex |
| `changeset '**/*.js'` | Changed files match the glob |
| `allOf { ... }` | ALL conditions must be true (AND) |
| `anyOf { ... }` | ANY condition must be true (OR) |
| `not { ... }` | Negation |
| `triggeredBy 'TimerTrigger'` | How the build was triggered |
| `equals expected: 2, actual: currentBuild.number` | Equality check |

## ⚡ Parallel Stages — Run Things Simultaneously

Instead of running stages one after another, you can run them **in parallel** to save time!

```groovy
pipeline {
    agent any

    stages {
        stage('Build') {
            steps {
                sh 'npm install'
                sh 'npm run build'
            }
        }

        stage('Tests') {
            parallel {
                stage('Unit Tests') {
                    steps {
                        sh 'npm run test:unit'
                    }
                }
                stage('Integration Tests') {
                    steps {
                        sh 'npm run test:integration'
                    }
                }
                stage('E2E Tests') {
                    agent { docker { image 'cypress/included:latest' } }
                    steps {
                        sh 'npm run test:e2e'
                    }
                }
            }
        }

        stage('Deploy') {
            steps {
                sh './deploy.sh'
            }
        }
    }
}
```

**What happens:**
```
Build ─────→ ┌─ Unit Tests      ─┐
             ├─ Integration Tests ├──→ Deploy
             └─ E2E Tests        ─┘
                (all 3 run at the same time!)
```

### `failFast` — Stop All if One Fails

```groovy
stage('Tests') {
    failFast true   // If any parallel stage fails, cancel the others immediately
    parallel {
        stage('Unit Tests') { ... }
        stage('Integration Tests') { ... }
    }
}
```

## 📮 Post — What Happens After the Pipeline

The `post` block defines actions that run **after** the pipeline (or stage) finishes — regardless of success or failure. Perfect for notifications, cleanup, etc.

```groovy
pipeline {
    agent any

    stages {
        stage('Build') {
            steps {
                sh 'npm install && npm run build'
            }
        }

        stage('Test') {
            steps {
                sh 'npm test'
            }
            // Post block at the STAGE level
            post {
                always {
                    junit 'test-results/**/*.xml'   // Publish test results
                }
            }
        }
    }

    // Post block at the PIPELINE level
    post {
        always {
            echo 'Pipeline finished. Cleaning up...'
            cleanWs()   // Clean workspace
        }

        success {
            echo '✅ Build succeeded!'
            // Send Slack notification
            slackSend channel: '#builds',
                      color: 'good',
                      message: "Build ${env.BUILD_NUMBER} succeeded! ✅"
        }

        failure {
            echo '❌ Build failed!'
            // Send email
            mail to: 'team@company.com',
                 subject: "FAILED: ${env.JOB_NAME} #${env.BUILD_NUMBER}",
                 body: "Check: ${env.BUILD_URL}"
        }

        unstable {
            echo '⚠️ Build is unstable (some tests failed).'
        }

        changed {
            echo 'Build status changed from previous build!'
        }

        aborted {
            echo '🛑 Build was aborted.'
        }
    }
}
```

### Post Conditions

| Condition | When It Runs |
|-----------|-------------|
| `always` | Every time — no matter what |
| `success` | Only if the pipeline/stage succeeded |
| `failure` | Only if the pipeline/stage failed |
| `unstable` | Only if marked unstable (e.g., test failures) |
| `changed` | Only if the current status is different from the last build |
| `aborted` | Only if the build was manually aborted |
| `fixed` | Only if the previous build failed but this one succeeded |
| `regression` | Only if the previous build succeeded but this one failed/is unstable |
| `cleanup` | Runs after every other post condition (for final cleanup) |

## 📁 Tools — Auto-Install Build Tools

If your Jenkins has tools configured (Maven, JDK, Node.js, etc.), you can reference them:

```groovy
pipeline {
    agent any

    tools {
        maven 'Maven-3.9.4'       // Name from Jenkins Global Tool Configuration
        jdk 'JDK-17'
        nodejs 'Node-18'
    }

    stages {
        stage('Build') {
            steps {
                sh 'mvn --version'
                sh 'java --version'
                sh 'node --version'
            }
        }
    }
}
```

Jenkins automatically installs and puts these tools on the `PATH` for you.

## ⏱️ Options — Pipeline-Level Settings

```groovy
pipeline {
    agent any

    options {
        timeout(time: 30, unit: 'MINUTES')       // Kill build if it takes > 30 min
        retry(2)                                   // Retry entire pipeline up to 2 times
        timestamps()                               // Add timestamps to console output
        buildDiscarder(logRotator(                 // Keep only last 10 builds
            numToKeepStr: '10',
            artifactNumToKeepStr: '5'
        ))
        disableConcurrentBuilds()                  // Don't run multiple builds at once
        skipDefaultCheckout()                      // Don't auto-checkout SCM
        ansiColor('xterm')                         // Color support in console
    }

    stages {
        stage('Build') {
            steps {
                checkout scm   // Manual checkout since we disabled default
                sh 'npm install'
            }
        }
    }
}
```

## 📥 Input — Manual Approval Gate

Need someone to approve before deploying to production? Use `input`:

```groovy
pipeline {
    agent any

    stages {
        stage('Build') {
            steps {
                sh 'npm install && npm run build'
            }
        }

        stage('Deploy to Staging') {
            steps {
                sh './deploy.sh staging'
            }
        }

        stage('Approval') {
            steps {
                input message: 'Deploy to Production?',
                      ok: 'Yes, deploy it!',
                      submitter: 'admin,devops-lead'
            }
        }

        stage('Deploy to Production') {
            steps {
                sh './deploy.sh production'
            }
        }
    }
}
```

The pipeline will **pause** at the "Approval" stage and wait for someone to click "Yes, deploy it!" in the Jenkins UI.

### Input with Parameters

```groovy
stage('Approval') {
    steps {
        script {
            def userInput = input(
                message: 'Ready for production?',
                parameters: [
                    choice(name: 'TARGET', choices: ['us-east-1', 'eu-west-1', 'ap-south-1'], description: 'Which region?'),
                    string(name: 'VERSION', defaultValue: '1.0.0', description: 'Version to deploy'),
                    booleanParam(name: 'SKIP_SMOKE', defaultValue: false, description: 'Skip smoke tests?')
                ]
            )
            env.DEPLOY_REGION = userInput.TARGET
            env.DEPLOY_VERSION = userInput.VERSION
        }
    }
}
```

## 🧩 Script Block — Escape Hatch to Groovy

Sometimes Declarative syntax is too rigid. The `script` block lets you write **raw Groovy** inside a Declarative Pipeline:

```groovy
stage('Dynamic Logic') {
    steps {
        script {
            // Full Groovy power here!
            def servers = ['web-01', 'web-02', 'web-03']

            for (server in servers) {
                echo "Deploying to ${server}..."
                sh "ssh deploy@${server} 'cd /opt/app && git pull && restart.sh'"
            }

            // If/else logic
            if (env.BRANCH_NAME == 'main') {
                echo 'This is the main branch!'
            } else {
                echo "This is branch: ${env.BRANCH_NAME}"
            }

            // Try/catch
            try {
                sh 'npm test'
            } catch (Exception e) {
                echo "Tests failed: ${e.getMessage()}"
                currentBuild.result = 'UNSTABLE'
            }
        }
    }
}
```

> 💡 **Tip:** Use `script` blocks sparingly. If you find yourself using them everywhere, consider switching to Scripted Pipeline (11.6).

## 🗄️ Stash & Unstash — Share Files Between Stages/Agents

When stages run on different agents, they don't share the same workspace. Use `stash`/`unstash` to pass files between them:

```groovy
pipeline {
    agent none

    stages {
        stage('Build') {
            agent { docker { image 'node:18' } }
            steps {
                sh 'npm install && npm run build'
                stash includes: 'dist/**', name: 'build-output'
            }
        }

        stage('Deploy') {
            agent { label 'production' }
            steps {
                unstash 'build-output'
                sh 'cp -r dist/* /var/www/html/'
            }
        }
    }
}
```

## 📤 Archiving Artifacts

Save build outputs (JARs, ZIPs, reports) that you might need later:

```groovy
stage('Build') {
    steps {
        sh 'mvn clean package'
    }
    post {
        success {
            archiveArtifacts artifacts: 'target/*.jar', fingerprint: true
        }
    }
}
```

These artifacts will be available on the build page in Jenkins for download.

## 🔥 Complete Real-World Jenkinsfile

Here's a **production-ready** Jenkinsfile that puts it all together:

```groovy
pipeline {
    agent any

    environment {
        APP_NAME = 'my-api'
        DOCKER_REGISTRY = 'registry.company.com'
        DOCKER_IMAGE = "${DOCKER_REGISTRY}/${APP_NAME}"
        DOCKER_CREDS = credentials('docker-registry-creds')
    }

    parameters {
        choice(name: 'ENVIRONMENT', choices: ['dev', 'staging', 'production'], description: 'Target environment')
        booleanParam(name: 'SKIP_TESTS', defaultValue: false, description: 'Skip test stage')
    }

    options {
        timeout(time: 30, unit: 'MINUTES')
        timestamps()
        buildDiscarder(logRotator(numToKeepStr: '20'))
        disableConcurrentBuilds()
    }

    stages {
        stage('Checkout') {
            steps {
                checkout scm
                script {
                    env.GIT_SHORT_COMMIT = sh(script: 'git rev-parse --short HEAD', returnStdout: true).trim()
                    env.IMAGE_TAG = "${env.BUILD_NUMBER}-${env.GIT_SHORT_COMMIT}"
                }
                echo "Building commit: ${env.GIT_SHORT_COMMIT}"
            }
        }

        stage('Build') {
            steps {
                sh 'npm ci'
                sh 'npm run build'
            }
        }

        stage('Test') {
            when {
                expression { !params.SKIP_TESTS }
            }
            parallel {
                stage('Unit Tests') {
                    steps {
                        sh 'npm run test:unit -- --reporter mocha-junit-reporter'
                    }
                    post {
                        always {
                            junit 'test-results/unit/*.xml'
                        }
                    }
                }
                stage('Lint') {
                    steps {
                        sh 'npm run lint'
                    }
                }
            }
        }

        stage('Docker Build & Push') {
            steps {
                sh """
                    echo \$DOCKER_CREDS_PSW | docker login ${DOCKER_REGISTRY} -u \$DOCKER_CREDS_USR --password-stdin
                    docker build -t ${DOCKER_IMAGE}:${IMAGE_TAG} .
                    docker push ${DOCKER_IMAGE}:${IMAGE_TAG}
                    docker tag ${DOCKER_IMAGE}:${IMAGE_TAG} ${DOCKER_IMAGE}:latest
                    docker push ${DOCKER_IMAGE}:latest
                """
            }
        }

        stage('Deploy to Staging') {
            when {
                anyOf {
                    branch 'main'
                    branch 'develop'
                    expression { params.ENVIRONMENT == 'staging' }
                }
            }
            steps {
                sh """
                    kubectl set image deployment/${APP_NAME} \
                        ${APP_NAME}=${DOCKER_IMAGE}:${IMAGE_TAG} \
                        -n staging
                    kubectl rollout status deployment/${APP_NAME} -n staging --timeout=300s
                """
            }
        }

        stage('Approval for Production') {
            when {
                allOf {
                    branch 'main'
                    expression { params.ENVIRONMENT == 'production' }
                }
            }
            steps {
                input message: "Deploy ${IMAGE_TAG} to PRODUCTION?",
                      ok: 'Deploy',
                      submitter: 'devops-team'
            }
        }

        stage('Deploy to Production') {
            when {
                allOf {
                    branch 'main'
                    expression { params.ENVIRONMENT == 'production' }
                }
            }
            steps {
                sh """
                    kubectl set image deployment/${APP_NAME} \
                        ${APP_NAME}=${DOCKER_IMAGE}:${IMAGE_TAG} \
                        -n production
                    kubectl rollout status deployment/${APP_NAME} -n production --timeout=300s
                """
            }
        }
    }

    post {
        always {
            cleanWs()
        }
        success {
            slackSend channel: '#deployments',
                      color: 'good',
                      message: "✅ ${APP_NAME} ${IMAGE_TAG} deployed to ${params.ENVIRONMENT}"
        }
        failure {
            slackSend channel: '#deployments',
                      color: 'danger',
                      message: "❌ ${APP_NAME} build #${env.BUILD_NUMBER} FAILED! ${env.BUILD_URL}"
        }
    }
}
```

**What this pipeline does:**
1. Checks out the code
2. Builds the Node.js app
3. Runs unit tests + lint **in parallel** (skippable)
4. Builds a Docker image and pushes it to registry
5. Deploys to Staging (for main/develop branches)
6. Waits for manual approval (only for production)
7. Deploys to Production
8. Sends Slack notifications
9. Cleans up workspace

---

## 🧠 Key Takeaways

1. **Declarative Pipeline** is the recommended way to write Jenkinsfiles — structured and easy to read.
2. Every pipeline needs at minimum: `pipeline`, `agent`, `stages`, and at least one `stage` with `steps`.
3. Use `environment` for variables, `credentials()` for secrets — **never hardcode passwords**.
4. Use `parameters` to make pipelines flexible and user-driven.
5. `when` blocks let you conditionally skip stages — branch-based deployments, feature flags, etc.
6. `parallel` runs stages simultaneously — great for running test suites faster.
7. `post` blocks handle notifications, cleanup, and reporting — `always`, `success`, `failure`.
8. `input` creates manual approval gates — essential for production deployments.
9. Use `script { }` blocks when you need Groovy logic, but keep them minimal.
10. `stash`/`unstash` let you share files across agents.

> 💡 **Pro tip:** Start simple. Get a basic pipeline working first, then add parallel stages, conditions, and notifications incrementally.
