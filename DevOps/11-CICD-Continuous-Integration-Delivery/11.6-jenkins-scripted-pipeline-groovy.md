# 11.6 — Jenkins Scripted Pipeline (Groovy)

> **Level:** Advanced  
> **Goal:** Master the Scripted Pipeline syntax — the flexible, Groovy-powered way to write Jenkins pipelines for complex and custom workflows.

---

## 📌 What is a Scripted Pipeline?

A **Scripted Pipeline** is the **older, more flexible** way to write Jenkinsfiles. While Declarative Pipeline (11.5) gives you a nice, structured format, Scripted Pipeline gives you the **full power of the Groovy programming language**.

Think of it like this:

| Feature | Declarative | Scripted |
|---------|------------|---------|
| Structure | Rigid, opinionated (must follow the `pipeline { }` format) | Freeform Groovy code |
| Learning curve | Easier | Harder (need Groovy knowledge) |
| Flexibility | Limited (use `script { }` for escape) | Unlimited |
| Readability | Very readable | Can get messy if not careful |
| Error handling | Built-in `post { }` | Manual `try/catch/finally` |
| When to use | 90% of the time | Complex/custom workflows |

---

## 🧱 Basic Structure

### Declarative vs Scripted — Side by Side

**Declarative (what you already know):**
```groovy
pipeline {
    agent any
    stages {
        stage('Build') {
            steps {
                sh 'npm install'
            }
        }
    }
}
```

**Scripted (same thing):**
```groovy
node {
    stage('Build') {
        sh 'npm install'
    }
}
```

Notice the differences:
- No `pipeline { }` wrapper
- No `steps { }` block
- Uses `node { }` instead of `agent`
- Commands go directly inside `stage { }`
- It's just... Groovy code!

### The Core Keywords

| Keyword | What It Does |
|---------|-------------|
| `node` | Allocates a Jenkins agent/executor — equivalent to `agent` in Declarative |
| `node('label')` | Allocates a specific agent with a matching label |
| `stage('Name')` | Defines a stage (shows in the Jenkins UI) |
| `sh 'command'` | Run a shell command |
| `echo 'message'` | Print to console |

---

## 🖥️ Node — Where Does It Run?

The `node` block is where you specify **which agent** to run on. It also allocates a **workspace** for your pipeline.

```groovy
// Run on any available agent
node {
    stage('Build') {
        sh 'echo "Running on any agent"'
    }
}

// Run on a specific labeled node
node('linux') {
    stage('Build') {
        sh 'echo "Running on a Linux node"'
    }
}

// Run different stages on different nodes
node('build-server') {
    stage('Build') {
        sh 'mvn clean package'
        stash includes: 'target/*.jar', name: 'build-artifacts'
    }
}

node('deploy-server') {
    stage('Deploy') {
        unstash 'build-artifacts'
        sh 'cp target/*.jar /opt/app/'
    }
}
```

### Nested Nodes

You can even nest `node` blocks for stages that need different agents:

```groovy
node('build-agent') {
    stage('Build') {
        sh 'npm install && npm run build'
        stash includes: 'dist/**', name: 'build-output'
    }

    // Switch to a different node for deployment
    node('production-server') {
        stage('Deploy') {
            unstash 'build-output'
            sh 'cp -r dist/* /var/www/html/'
        }
    }
}
```

## 🔧 Groovy Fundamentals for Scripted Pipelines

Since Scripted Pipelines are just Groovy code, you need to know some Groovy basics. Don't worry — if you know any programming language, Groovy will feel familiar!

### Variables

```groovy
node {
    // Define variables
    def appName = 'my-app'
    def version = '2.0.0'
    def buildNumber = env.BUILD_NUMBER

    stage('Info') {
        echo "App: ${appName}, Version: ${version}, Build: ${buildNumber}"
    }
}
```

### Strings

```groovy
node {
    // String interpolation (double quotes only!)
    def name = "World"
    echo "Hello, ${name}!"          // ✅ Works — Hello, World!
    echo 'Hello, ${name}!'          // ❌ Prints literally — Hello, ${name}!

    // Multi-line strings
    def message = """
        Building application...
        Environment: production
        Version: 1.0.0
    """
    echo message

    // Shell command output as string
    def gitCommit = sh(script: 'git rev-parse --short HEAD', returnStdout: true).trim()
    echo "Current commit: ${gitCommit}"
}
```

> ⚠️ **Important:** Use double quotes (`"..."`) for string interpolation. Single quotes (`'...'`) are literal strings in Groovy.

### If/Else

```groovy
node {
    stage('Conditional Deploy') {
        def branch = env.BRANCH_NAME ?: 'unknown'

        if (branch == 'main') {
            echo 'Deploying to PRODUCTION...'
            sh './deploy.sh production'
        } else if (branch == 'develop') {
            echo 'Deploying to STAGING...'
            sh './deploy.sh staging'
        } else {
            echo "Branch ${branch} — skipping deployment."
        }
    }
}
```

### Loops

```groovy
node {
    stage('Deploy to Multiple Servers') {
        def servers = ['web-01', 'web-02', 'web-03', 'web-04']

        for (server in servers) {
            echo "Deploying to ${server}..."
            sh "ssh deploy@${server} 'cd /opt/app && git pull && systemctl restart app'"
        }
    }
}
```

### Maps (Dictionaries)

```groovy
node {
    def config = [
        app_name: 'my-api',
        version: '3.1.0',
        port: 8080,
        replicas: 3
    ]

    stage('Build') {
        echo "Building ${config.app_name} v${config.version}"
        echo "Will deploy ${config.replicas} replicas on port ${config.port}"
    }
}
```

### Lists

```groovy
node {
    def environments = ['dev', 'staging', 'production']

    stage('Info') {
        echo "First env: ${environments[0]}"        // dev
        echo "Total envs: ${environments.size()}"    // 3
        echo "Has staging? ${environments.contains('staging')}"  // true
    }
}
```

### Functions (Methods)

You can define your own functions to reuse logic:

```groovy
// Define function outside node block
def deployToServer(String server, String version) {
    echo "Deploying version ${version} to ${server}..."
    sh """
        ssh deploy@${server} << 'EOF'
            cd /opt/app
            docker pull myregistry/myapp:${version}
            docker-compose up -d
        EOF
    """
}

def notifySlack(String message, String color = 'good') {
    slackSend channel: '#deployments', color: color, message: message
}

// Use the functions
node {
    stage('Build') {
        sh 'docker build -t myapp:latest .'
    }

    stage('Deploy') {
        deployToServer('prod-01.example.com', '2.5.0')
        deployToServer('prod-02.example.com', '2.5.0')
    }

    stage('Notify') {
        notifySlack("✅ Deployment complete!")
    }
}
```

## 🛡️ Error Handling — try/catch/finally

Unlike Declarative Pipeline which has a nice `post { }` block, in Scripted Pipeline you handle errors the **Java/Groovy way** with `try/catch/finally`.

### Basic Error Handling

```groovy
node {
    stage('Build') {
        try {
            sh 'npm install'
            sh 'npm run build'
        } catch (Exception e) {
            echo "Build failed: ${e.getMessage()}"
            currentBuild.result = 'FAILURE'
            throw e   // Re-throw to stop the pipeline
        }
    }
}
```

### Full Pipeline with Error Handling

```groovy
node {
    try {
        stage('Checkout') {
            checkout scm
        }

        stage('Build') {
            sh 'npm ci'
            sh 'npm run build'
        }

        stage('Test') {
            sh 'npm test'
        }

        stage('Deploy') {
            sh './deploy.sh'
        }

        // If we reach here, everything succeeded
        currentBuild.result = 'SUCCESS'

    } catch (Exception e) {
        // Something failed
        currentBuild.result = 'FAILURE'
        echo "Pipeline failed: ${e.getMessage()}"

        // Send failure notification
        mail to: 'team@company.com',
             subject: "FAILED: ${env.JOB_NAME} #${env.BUILD_NUMBER}",
             body: "Error: ${e.getMessage()}\nCheck: ${env.BUILD_URL}"

        throw e   // Re-throw so Jenkins marks the build as failed

    } finally {
        // This runs NO MATTER WHAT — like `post { always { } }` in Declarative
        echo 'Cleaning up workspace...'
        cleanWs()

        // Send notification based on result
        if (currentBuild.result == 'SUCCESS') {
            slackSend color: 'good', message: "✅ Build succeeded!"
        } else {
            slackSend color: 'danger', message: "❌ Build failed!"
        }
    }
}
```

### Handling Specific Errors

```groovy
node {
    stage('Deploy') {
        try {
            sh 'kubectl apply -f deployment.yaml'
            sh 'kubectl rollout status deployment/myapp --timeout=120s'

        } catch (hudson.AbortException e) {
            // The shell command returned non-zero exit code
            echo "Deployment command failed: ${e.getMessage()}"

            // Rollback!
            sh 'kubectl rollout undo deployment/myapp'
            throw e

        } catch (Exception e) {
            // Any other error
            echo "Unexpected error: ${e.getClass().getName()}: ${e.getMessage()}"
            throw e
        }
    }
}
```

### Continue on Failure (Don't Stop)

Sometimes you want the pipeline to **keep going** even if something fails:

```groovy
node {
    stage('Tests') {
        // Run tests but don't fail the build
        def testResult = sh(script: 'npm test', returnStatus: true)

        if (testResult != 0) {
            echo "⚠️ Tests failed but continuing..."
            currentBuild.result = 'UNSTABLE'
        }
    }

    stage('Deploy') {
        echo 'Deploying anyway...'
        sh './deploy.sh'
    }
}
```

The `returnStatus: true` parameter makes `sh` return the exit code instead of throwing an exception.

## ⚡ Parallel Execution

Running stages in parallel is where Scripted Pipeline really shines — you get much more control than Declarative.

### Basic Parallel

```groovy
node {
    stage('Build') {
        sh 'npm install && npm run build'
    }

    stage('Tests') {
        parallel(
            'Unit Tests': {
                sh 'npm run test:unit'
            },
            'Integration Tests': {
                sh 'npm run test:integration'
            },
            'Lint': {
                sh 'npm run lint'
            }
        )
    }

    stage('Deploy') {
        sh './deploy.sh'
    }
}
```

### Dynamic Parallel — Generated from a List

This is something you **can't easily do** in Declarative Pipeline:

```groovy
node {
    stage('Build') {
        sh 'npm install && npm run build'
    }

    stage('Deploy to All Regions') {
        def regions = ['us-east-1', 'eu-west-1', 'ap-south-1']
        def deployments = [:]

        for (region in regions) {
            // IMPORTANT: Capture the variable in a local scope!
            def r = region
            deployments["Deploy to ${r}"] = {
                node('deploy-agent') {
                    echo "Deploying to ${r}..."
                    sh "kubectl config use-context ${r}"
                    sh "kubectl apply -f deployment.yaml"
                }
            }
        }

        // Run all deployments in parallel
        parallel deployments
    }
}
```

> ⚠️ **Gotcha Alert!** Notice the `def r = region` line. In Groovy closures, you MUST capture loop variables in a local variable. Without it, all parallel branches would use the **last** value of the loop variable. This is a **very common bug**!

### Parallel with failFast

```groovy
stage('Tests') {
    parallel(
        failFast: true,   // If any branch fails, abort the rest
        'Unit Tests': {
            sh 'npm run test:unit'
        },
        'E2E Tests': {
            sh 'npm run test:e2e'
        }
    )
}
```

## 🔐 Working with Credentials

```groovy
node {
    stage('Deploy') {
        // Method 1: withCredentials block
        withCredentials([
            usernamePassword(
                credentialsId: 'docker-hub',
                usernameVariable: 'DOCKER_USER',
                passwordVariable: 'DOCKER_PASS'
            )
        ]) {
            sh 'echo $DOCKER_PASS | docker login -u $DOCKER_USER --password-stdin'
            sh 'docker push myregistry/myapp:latest'
        }
        // DOCKER_USER and DOCKER_PASS are NOT available here — they're scoped!

        // Method 2: Secret text
        withCredentials([string(credentialsId: 'slack-webhook', variable: 'WEBHOOK_URL')]) {
            sh "curl -X POST -H 'Content-type: application/json' --data '{\"text\":\"Deployed!\"}' $WEBHOOK_URL"
        }

        // Method 3: SSH Key
        withCredentials([sshUserPrivateKey(credentialsId: 'deploy-key', keyFileVariable: 'SSH_KEY', usernameVariable: 'SSH_USER')]) {
            sh "ssh -i $SSH_KEY ${SSH_USER}@production-server 'systemctl restart myapp'"
        }

        // Method 4: Secret file (e.g., kubeconfig)
        withCredentials([file(credentialsId: 'kubeconfig', variable: 'KUBECONFIG')]) {
            sh 'kubectl get pods'
        }

        // Method 5: Multiple credentials at once
        withCredentials([
            usernamePassword(credentialsId: 'docker-hub', usernameVariable: 'D_USER', passwordVariable: 'D_PASS'),
            string(credentialsId: 'sonar-token', variable: 'SONAR_TOKEN'),
            file(credentialsId: 'kubeconfig', variable: 'KUBECONFIG')
        ]) {
            sh 'echo $D_PASS | docker login -u $D_USER --password-stdin'
            sh "sonar-scanner -Dsonar.login=$SONAR_TOKEN"
            sh 'kubectl apply -f deployment.yaml'
        }
    }
}
```

> 🔒 **Security:** Jenkins automatically **masks** credential values in the console output. If `DOCKER_PASS` is `myS3cret`, you'll see `****` in the logs instead.

## 🏗️ Working with Docker in Scripted Pipeline

```groovy
node {
    stage('Build in Docker') {
        // Run commands inside a Docker container
        docker.image('node:18-alpine').inside {
            sh 'node --version'
            sh 'npm install'
            sh 'npm run build'
        }
    }

    stage('Build Docker Image') {
        def app = docker.build("mycompany/myapp:${env.BUILD_NUMBER}")

        // Push to registry
        docker.withRegistry('https://registry.company.com', 'docker-registry-creds') {
            app.push()
            app.push('latest')
        }
    }

    stage('Test with Docker Compose') {
        sh 'docker-compose -f docker-compose.test.yml up --abort-on-container-exit'
        sh 'docker-compose -f docker-compose.test.yml down'
    }
}
```

### Running Tests in a Specific Container

```groovy
node {
    stage('Test Matrix') {
        def nodeVersions = ['16', '18', '20']
        def testJobs = [:]

        for (ver in nodeVersions) {
            def v = ver  // Capture for closure
            testJobs["Node ${v}"] = {
                docker.image("node:${v}-alpine").inside {
                    sh 'npm install'
                    sh 'npm test'
                }
            }
        }

        parallel testJobs
    }
}
```

## 📂 Workspace Management

```groovy
node {
    // Checkout into a specific directory
    stage('Checkout') {
        checkout scm
    }

    // Work in a subdirectory
    stage('Build Frontend') {
        dir('frontend') {
            sh 'npm install'
            sh 'npm run build'
        }
    }

    stage('Build Backend') {
        dir('backend') {
            sh 'pip install -r requirements.txt'
            sh 'python setup.py build'
        }
    }

    // Get workspace path
    stage('Info') {
        echo "Workspace: ${env.WORKSPACE}"
        echo "Current dir: ${pwd()}"
    }

    // Write and read files
    stage('Files') {
        writeFile file: 'version.txt', text: "1.0.${env.BUILD_NUMBER}"
        def version = readFile('version.txt').trim()
        echo "Version: ${version}"

        // Read JSON
        writeFile file: 'config.json', text: '{"app": "myapp", "port": 3000}'
        def config = readJSON file: 'config.json'
        echo "App: ${config.app}, Port: ${config.port}"

        // Read YAML
        def yamlConfig = readYaml file: 'k8s/deployment.yaml'
        echo "Image: ${yamlConfig.spec.template.spec.containers[0].image}"
    }
}
```

## ⏳ Timeout & Retry

```groovy
node {
    stage('Flaky Test') {
        // Retry up to 3 times
        retry(3) {
            sh 'npm run test:e2e'
        }
    }

    stage('Wait for Service') {
        // Timeout after 5 minutes
        timeout(time: 5, unit: 'MINUTES') {
            // Wait until the service is healthy
            waitUntil {
                def status = sh(script: 'curl -s -o /dev/null -w "%{http_code}" http://myservice:8080/health', returnStdout: true).trim()
                return status == '200'
            }
        }
    }

    stage('Deploy with Timeout') {
        timeout(time: 10, unit: 'MINUTES') {
            sh 'kubectl rollout status deployment/myapp --timeout=600s'
        }
    }
}
```

## 🔄 Shared Libraries — Reuse Code Across Pipelines

When you have common functions used in many pipelines, put them in a **Shared Library** instead of copy-pasting.

### Shared Library Structure

```
my-shared-library/
├── vars/
│   ├── deployApp.groovy        ← Global function: deployApp()
│   ├── notifySlack.groovy      ← Global function: notifySlack()
│   └── buildDocker.groovy      ← Global function: buildDocker()
├── src/
│   └── com/
│       └── mycompany/
│           └── Utils.groovy    ← Class-based utilities
└── resources/
    └── templates/
        └── deployment.yaml     ← Static resources
```

### Defining a Shared Library Function

**`vars/deployApp.groovy`:**
```groovy
def call(String environment, String version) {
    echo "Deploying version ${version} to ${environment}..."

    if (environment == 'production') {
        input message: "Deploy ${version} to PRODUCTION?", ok: 'Deploy'
    }

    sh """
        kubectl config use-context ${environment}
        kubectl set image deployment/myapp myapp=myregistry/myapp:${version}
        kubectl rollout status deployment/myapp --timeout=300s
    """

    echo "✅ Deployed ${version} to ${environment} successfully!"
}
```

**`vars/notifySlack.groovy`:**
```groovy
def call(Map config = [:]) {
    def defaults = [
        channel: '#builds',
        color: 'good',
        message: 'Build completed'
    ]
    config = defaults + config

    slackSend(
        channel: config.channel,
        color: config.color,
        message: config.message
    )
}
```

### Using the Shared Library in a Pipeline

```groovy
// Load the library (configured in Jenkins → Manage Jenkins → Configure System → Global Pipeline Libraries)
@Library('my-shared-library') _

node {
    stage('Build') {
        sh 'npm install && npm run build'
    }

    stage('Deploy') {
        deployApp('staging', "1.0.${env.BUILD_NUMBER}")
    }

    stage('Notify') {
        notifySlack(
            channel: '#deployments',
            color: 'good',
            message: "✅ Build #${env.BUILD_NUMBER} deployed to staging!"
        )
    }
}
```

### Loading Library Dynamically

```groovy
// Load at runtime (doesn't need to be pre-configured in Jenkins)
library(
    identifier: 'my-shared-library@main',
    retriever: modernSCM([
        $class: 'GitSCMSource',
        remote: 'https://github.com/mycompany/jenkins-shared-library.git',
        credentialsId: 'github-token'
    ])
)

node {
    stage('Deploy') {
        deployApp('production', '2.5.0')
    }
}
```

## 🔥 Complete Real-World Scripted Pipeline

Here's a **production-grade** Scripted Pipeline that demonstrates all the concepts:

```groovy
#!groovy

// Load shared library
@Library('company-pipeline-library') _

// Global variables
def appName = 'order-service'
def dockerImage = ''
def imageTag = ''

node('build-agent') {
    try {
        // ===========================
        // STAGE 1: Checkout
        // ===========================
        stage('Checkout') {
            checkout scm

            // Get git info
            imageTag = "${env.BUILD_NUMBER}-${sh(script: 'git rev-parse --short HEAD', returnStdout: true).trim()}"
            echo "Building ${appName} — Tag: ${imageTag}"
        }

        // ===========================
        // STAGE 2: Build
        // ===========================
        stage('Build') {
            docker.image('maven:3.9.4-eclipse-temurin-17').inside('-v $HOME/.m2:/root/.m2') {
                sh 'mvn clean package -DskipTests'
            }
        }

        // ===========================
        // STAGE 3: Tests (Parallel)
        // ===========================
        stage('Tests') {
            parallel(
                'Unit Tests': {
                    docker.image('maven:3.9.4-eclipse-temurin-17').inside('-v $HOME/.m2:/root/.m2') {
                        sh 'mvn test'
                    }
                },
                'Code Analysis': {
                    docker.image('maven:3.9.4-eclipse-temurin-17').inside('-v $HOME/.m2:/root/.m2') {
                        withCredentials([string(credentialsId: 'sonarqube-token', variable: 'SONAR_TOKEN')]) {
                            sh """
                                mvn sonar:sonar \
                                    -Dsonar.host.url=https://sonar.company.com \
                                    -Dsonar.login=${SONAR_TOKEN}
                            """
                        }
                    }
                },
                'Security Scan': {
                    sh 'trivy fs --exit-code 1 --severity HIGH,CRITICAL .'
                }
            )
        }

        // ===========================
        // STAGE 4: Docker Build & Push
        // ===========================
        stage('Docker Build & Push') {
            dockerImage = docker.build("myregistry.com/${appName}:${imageTag}")

            docker.withRegistry('https://myregistry.com', 'docker-registry-creds') {
                dockerImage.push()
                dockerImage.push('latest')
            }
        }

        // ===========================
        // STAGE 5: Deploy to Staging
        // ===========================
        stage('Deploy to Staging') {
            withCredentials([file(credentialsId: 'staging-kubeconfig', variable: 'KUBECONFIG')]) {
                sh """
                    kubectl set image deployment/${appName} \
                        ${appName}=myregistry.com/${appName}:${imageTag} \
                        -n staging
                    kubectl rollout status deployment/${appName} -n staging --timeout=300s
                """
            }
        }

        // ===========================
        // STAGE 6: Integration Tests
        // ===========================
        stage('Integration Tests') {
            retry(2) {
                timeout(time: 10, unit: 'MINUTES') {
                    sh """
                        cd integration-tests
                        npm install
                        STAGING_URL=https://staging.company.com npm test
                    """
                }
            }
        }

        // ===========================
        // STAGE 7: Deploy to Production
        // ===========================
        if (env.BRANCH_NAME == 'main') {
            stage('Approval') {
                def approver = input(
                    message: "Deploy ${appName}:${imageTag} to PRODUCTION?",
                    ok: 'Deploy to Production',
                    submitter: 'devops-team,engineering-leads',
                    submitterParameter: 'approvedBy'
                )
                echo "Approved by: ${approver}"
            }

            stage('Deploy to Production') {
                withCredentials([file(credentialsId: 'prod-kubeconfig', variable: 'KUBECONFIG')]) {
                    // Deploy with canary strategy
                    sh """
                        # Update image
                        kubectl set image deployment/${appName} \
                            ${appName}=myregistry.com/${appName}:${imageTag} \
                            -n production

                        # Wait for rollout
                        kubectl rollout status deployment/${appName} \
                            -n production --timeout=300s
                    """
                }
            }

            stage('Smoke Tests') {
                retry(3) {
                    timeout(time: 2, unit: 'MINUTES') {
                        sh """
                            curl -sf https://api.company.com/health || exit 1
                            curl -sf https://api.company.com/readiness || exit 1
                        """
                    }
                }
            }
        }

        // ===========================
        // SUCCESS
        // ===========================
        currentBuild.result = 'SUCCESS'

    } catch (Exception e) {
        currentBuild.result = 'FAILURE'
        echo "❌ Pipeline failed: ${e.getMessage()}"

        // Auto-rollback on production failure
        if (env.BRANCH_NAME == 'main') {
            echo '🔄 Rolling back production deployment...'
            try {
                withCredentials([file(credentialsId: 'prod-kubeconfig', variable: 'KUBECONFIG')]) {
                    sh "kubectl rollout undo deployment/${appName} -n production"
                }
            } catch (Exception rollbackError) {
                echo "⚠️ Rollback also failed: ${rollbackError.getMessage()}"
            }
        }

        throw e

    } finally {
        // ===========================
        // ALWAYS — Cleanup & Notify
        // ===========================

        // Archive test results
        junit allowEmptyResults: true, testResults: '**/target/surefire-reports/*.xml'

        // Cleanup workspace
        cleanWs()

        // Notifications
        def buildStatus = currentBuild.result ?: 'SUCCESS'
        def color = buildStatus == 'SUCCESS' ? 'good' : 'danger'
        def emoji = buildStatus == 'SUCCESS' ? '✅' : '❌'

        notifySlack(
            channel: '#deployments',
            color: color,
            message: "${emoji} ${appName}:${imageTag} — Build #${env.BUILD_NUMBER} — ${buildStatus}\n${env.BUILD_URL}"
        )
    }
}
```

## 🤔 Scripted vs Declarative — When to Use What?

| Scenario | Use |
|----------|-----|
| Standard CI/CD pipeline (build → test → deploy) | **Declarative** ✅ |
| Need parallel deployments generated from a list | **Scripted** ✅ |
| Simple branch-based conditions | **Declarative** (`when` block) ✅ |
| Complex conditional logic with many if/else branches | **Scripted** ✅ |
| Team has Groovy experience | **Scripted** ✅ |
| Team is new to Jenkins | **Declarative** ✅ |
| Need custom error handling with rollbacks | **Scripted** ✅ |
| Standard post-build notifications | **Declarative** (`post` block) ✅ |
| Building a reusable pipeline template | **Scripted** ✅ |
| 90% of real-world use cases | **Declarative** ✅ |

> 💡 **Pro tip:** Start with Declarative. When you hit its limits, use `script { }` blocks. Only go full Scripted if your pipeline logic is genuinely complex.

## ⚠️ Common Pitfalls & Best Practices

### 1. Loop Variable Closure Bug (VERY Common!)

```groovy
// ❌ WRONG — All branches will deploy to 'ap-south-1' (the last value)
def regions = ['us-east-1', 'eu-west-1', 'ap-south-1']
def jobs = [:]
for (region in regions) {
    jobs[region] = {
        sh "deploy to ${region}"   // Bug! 'region' is shared
    }
}
parallel jobs

// ✅ CORRECT — Capture the variable
for (region in regions) {
    def r = region   // Local capture
    jobs[r] = {
        sh "deploy to ${r}"
    }
}
parallel jobs
```

### 2. Serializable Everything (CPS)

Jenkins pipelines are **serializable** — they can be paused and resumed (e.g., across Jenkins restarts). This means some Java/Groovy objects don't work normally.

```groovy
// ❌ WRONG — java.io.NotSerializableException
node {
    def json = new groovy.json.JsonSlurper().parseText('{"key": "value"}')
}

// ✅ CORRECT — Use @NonCPS annotation for non-serializable operations
@NonCPS
def parseJson(String text) {
    return new groovy.json.JsonSlurper().parseText(text)
}

node {
    def json = parseJson('{"key": "value"}')
    echo "Key: ${json.key}"
}

// ✅ ALSO CORRECT — Use Jenkins built-in readJSON
node {
    def json = readJSON text: '{"key": "value"}'
    echo "Key: ${json.key}"
}
```

### 3. Avoid Heavy Logic in the Jenkinsfile

```groovy
// ❌ Don't put 500 lines of Groovy logic in Jenkinsfile
// ✅ Move reusable logic to Shared Libraries
// ✅ Move complex build logic to build scripts (Makefile, scripts/, etc.)

// Good pattern:
node {
    stage('Build') {
        sh 'make build'              // Logic is in Makefile
    }
    stage('Test') {
        sh './scripts/run-tests.sh'  // Logic is in script
    }
    stage('Deploy') {
        deployApp('staging')          // Logic is in Shared Library
    }
}
```

### 4. Always Clean Up

```groovy
node {
    try {
        // ... pipeline stages ...
    } finally {
        // ALWAYS clean up — even if the pipeline fails
        cleanWs()

        // Remove Docker images to free disk space
        sh 'docker image prune -f'
    }
}
```

### 5. Use `returnStatus` and `returnStdout`

```groovy
node {
    // Get command output as a string
    def version = sh(script: 'cat VERSION', returnStdout: true).trim()

    // Get exit code instead of failing
    def exitCode = sh(script: 'npm test', returnStatus: true)

    if (exitCode != 0) {
        echo "Tests failed with exit code: ${exitCode}"
        currentBuild.result = 'UNSTABLE'
    }
}
```

## 🧠 Key Takeaways

1. **Scripted Pipeline** gives you the full power of **Groovy** — if/else, loops, functions, try/catch, classes.
2. Use `node { }` instead of `agent`, and write commands directly in `stage { }` (no `steps { }` needed).
3. Error handling is done with **`try/catch/finally`** — the `finally` block is your "post-always" equivalent.
4. **Dynamic parallel execution** (generated from lists/maps) is Scripted Pipeline's superpower.
5. Watch out for the **loop variable closure bug** — always capture loop vars with `def r = region`.
6. Use **`@NonCPS`** for non-serializable operations (JSON parsing, regex, etc.).
7. Keep Jenkinsfiles **thin** — move logic to Shared Libraries and build scripts.
8. **Declarative is recommended for 90% of use cases.** Only go Scripted when you genuinely need the flexibility.
9. Shared Libraries let you write functions once and reuse them across **all** your pipelines.
10. `withCredentials()` is the **safe, correct** way to handle secrets — never hardcode them.

> 💡 **Golden Rule:** Write Declarative first. Use `script { }` blocks when needed. Go full Scripted only for genuinely complex pipelines. And always, ALWAYS use Shared Libraries for reusable code.
