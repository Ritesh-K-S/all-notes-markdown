# 11.7 — Jenkins — Shared Libraries

> **One Line Answer:** Jenkins Shared Libraries let you **write reusable pipeline code once** and use it across **hundreds of Jenkinsfiles** — so instead of copy-pasting the same pipeline logic everywhere, you keep it in one Git repo and every pipeline just calls it like a function.

---

## 🤔 Why Do We Need Shared Libraries? — The Problem

Imagine you have 50 microservices, and each one has a Jenkinsfile. The build-test-deploy steps are almost identical for all of them:

```groovy
// Microservice A - Jenkinsfile
pipeline {
    agent any
    stages {
        stage('Build') {
            steps {
                sh 'docker build -t my-registry/service-a:${BUILD_NUMBER} .'
            }
        }
        stage('Test') {
            steps {
                sh 'docker run my-registry/service-a:${BUILD_NUMBER} npm test'
            }
        }
        stage('Push') {
            steps {
                sh 'docker push my-registry/service-a:${BUILD_NUMBER}'
            }
        }
        stage('Deploy') {
            steps {
                sh 'kubectl set image deployment/service-a app=my-registry/service-a:${BUILD_NUMBER}'
            }
        }
    }
}
```

Now copy-paste this for **50 services**, changing just the service name. What happens?

```
Problems Without Shared Libraries:
❌ 50 Jenkinsfiles with almost identical code
❌ Bug fix? Edit ALL 50 files manually
❌ New stage needed? Update ALL 50 files
❌ Different teams write pipelines differently — no consistency
❌ Security practices get missed in some pipelines
❌ Onboarding new teams is slow — "How do I write a Jenkinsfile?"
```

### The Joy With Shared Libraries:

```groovy
// Microservice A - Jenkinsfile (with Shared Library) — Just 3 lines!
@Library('my-shared-library') _

standardPipeline(
    serviceName: 'service-a',
    deployEnv: 'production'
)
```

```
Benefits With Shared Libraries:
✅ Write pipeline logic ONCE, reuse EVERYWHERE
✅ Fix a bug in ONE place — all 50 pipelines get the fix
✅ Enforce standards — security scanning, testing, notifications
✅ Jenkinsfiles become tiny and clean
✅ New teams just call the library — instant best practices
```

## 📁 Shared Library — Directory Structure

A Shared Library is just a **Git repository** with a specific folder structure. Jenkins expects it to look like this:

```
my-shared-library/          ← Git repo
├── vars/                   ← Global variables & functions (most commonly used)
│   ├── standardPipeline.groovy
│   ├── dockerBuild.groovy
│   ├── notifySlack.groovy
│   └── deployToK8s.groovy
├── src/                    ← More complex Groovy classes (OOP style)
│   └── org/
│       └── mycompany/
│           ├── Docker.groovy
│           ├── Kubernetes.groovy
│           └── Utils.groovy
├── resources/              ← Non-Groovy files (scripts, configs, templates)
│   ├── deploy.sh
│   ├── email-template.html
│   └── k8s-deployment.yaml
└── README.md
```

Let's understand each folder:

### 1. `vars/` — Global Variables (The Easy Way)

This is where **most people** put their shared library code. Each `.groovy` file in `vars/` becomes a **globally available function** in any Jenkinsfile.

**Rules:**
- Filename = function name (e.g., `dockerBuild.groovy` → you call `dockerBuild()` in Jenkinsfile)
- Must have a `call()` method (this is what runs when you call the function)
- Can also have other methods

**Example — `vars/dockerBuild.groovy`:**

```groovy
// vars/dockerBuild.groovy

def call(Map config = [:]) {
    // Default values
    def imageName = config.imageName ?: error("imageName is required")
    def tag = config.tag ?: env.BUILD_NUMBER
    def registry = config.registry ?: 'docker.io'
    def dockerfile = config.dockerfile ?: 'Dockerfile'

    echo "🐳 Building Docker image: ${registry}/${imageName}:${tag}"
    
    sh """
        docker build -t ${registry}/${imageName}:${tag} -f ${dockerfile} .
    """
    
    echo "✅ Docker image built successfully!"
    
    // Return the full image name for use in later stages
    return "${registry}/${imageName}:${tag}"
}
```

**How to use it in a Jenkinsfile:**

```groovy
@Library('my-shared-library') _

pipeline {
    agent any
    stages {
        stage('Build') {
            steps {
                script {
                    def image = dockerBuild(
                        imageName: 'my-app',
                        registry: 'myregistry.azurecr.io',
                        tag: "${env.BUILD_NUMBER}"
                    )
                    echo "Built image: ${image}"
                }
            }
        }
    }
}
```

**What's happening here?**
- `@Library('my-shared-library') _` — loads the shared library (the `_` is required — it imports everything)
- `dockerBuild(...)` — calls the `call()` method from `vars/dockerBuild.groovy`
- The `Map config` parameter receives all the key-value pairs you pass in

### 2. `src/` — Groovy Classes (The OOP Way)

For more complex logic, you can write proper Groovy classes in `src/`. These follow standard Java/Groovy package conventions.

**Example — `src/org/mycompany/Docker.groovy`:**

```groovy
// src/org/mycompany/Docker.groovy
package org.mycompany

class Docker implements Serializable {
    
    def steps  // This holds the Jenkins pipeline steps context
    
    Docker(steps) {
        this.steps = steps
    }
    
    def build(String imageName, String tag = 'latest', String dockerfile = 'Dockerfile') {
        steps.echo "🐳 Building image: ${imageName}:${tag}"
        steps.sh "docker build -t ${imageName}:${tag} -f ${dockerfile} ."
    }
    
    def push(String imageName, String tag = 'latest') {
        steps.echo "📤 Pushing image: ${imageName}:${tag}"
        steps.sh "docker push ${imageName}:${tag}"
    }
    
    def scan(String imageName, String tag = 'latest') {
        steps.echo "🔍 Scanning image for vulnerabilities: ${imageName}:${tag}"
        steps.sh "trivy image ${imageName}:${tag} --exit-code 1 --severity HIGH,CRITICAL"
    }
}
```

**Why `implements Serializable`?**
Jenkins pipelines can be paused and resumed (e.g., waiting for user approval). When paused, Jenkins **serializes** (saves) the state. If your class isn't `Serializable`, Jenkins will throw an error when it tries to save the pipeline state. So always add `implements Serializable` to your classes.

**Why do we pass `steps` to the constructor?**
Classes in `src/` don't automatically have access to Jenkins pipeline steps like `sh`, `echo`, `env`, etc. You need to pass the pipeline context (`this`) from the Jenkinsfile into your class. That's what `steps` is — it's a reference to the pipeline so your class can call `steps.sh(...)`, `steps.echo(...)`, etc.

**How to use `src/` classes in a Jenkinsfile:**

```groovy
@Library('my-shared-library') _
import org.mycompany.Docker

pipeline {
    agent any
    stages {
        stage('Build & Push') {
            steps {
                script {
                    def docker = new Docker(this)  // Pass pipeline context
                    docker.build('myregistry.io/my-app', env.BUILD_NUMBER)
                    docker.scan('myregistry.io/my-app', env.BUILD_NUMBER)
                    docker.push('myregistry.io/my-app', env.BUILD_NUMBER)
                }
            }
        }
    }
}
```

### 3. `resources/` — Static Files

For shell scripts, config templates, email templates, or any non-Groovy files.

**Example — `resources/k8s-deployment.yaml`:**

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: ${SERVICE_NAME}
spec:
  replicas: ${REPLICAS}
  selector:
    matchLabels:
      app: ${SERVICE_NAME}
  template:
    metadata:
      labels:
        app: ${SERVICE_NAME}
    spec:
      containers:
        - name: ${SERVICE_NAME}
          image: ${IMAGE}
          ports:
            - containerPort: ${PORT}
```

**How to load resources in your library code:**

```groovy
// vars/deployToK8s.groovy

def call(Map config) {
    // libraryResource() loads files from the resources/ directory
    def template = libraryResource('k8s-deployment.yaml')
    
    // Replace placeholders
    def manifest = template
        .replace('${SERVICE_NAME}', config.serviceName)
        .replace('${REPLICAS}', config.replicas.toString())
        .replace('${IMAGE}', config.image)
        .replace('${PORT}', config.port.toString())
    
    // Write the file and apply
    writeFile file: 'deployment.yaml', text: manifest
    sh 'kubectl apply -f deployment.yaml'
}
```

## ⚙️ Configuring Shared Libraries in Jenkins

There are **three ways** to configure a shared library in Jenkins. Let's go through each:

### Method 1: Global Shared Library (System-Wide)

This makes the library available to **ALL** pipelines on the Jenkins server.

**Steps:**
1. Go to **Manage Jenkins** → **System** (or **Configure System** in older versions)
2. Scroll down to **Global Pipeline Libraries**
3. Click **Add**
4. Fill in:

```
Name:           my-shared-library
Default Version: main          (branch name, tag, or commit hash)
Load Implicitly: ☐             (if checked, auto-loads in ALL pipelines)
Allow default version to be overridden: ☑
Include @Library changes in recent changes: ☑

Retrieval Method: Modern SCM
  → Git
    Project Repository: https://github.com/mycompany/jenkins-shared-library.git
    Credentials:        (select your Git credentials)
```

**What does "Load Implicitly" mean?**
- If **checked** → Every pipeline automatically gets the library. No need for `@Library` in Jenkinsfile
- If **unchecked** → Pipelines must explicitly load it with `@Library('my-shared-library') _`
- **Recommendation:** Leave it unchecked. Be explicit about what you load.

### Method 2: Folder-Level Library

If you organize your Jenkins jobs into folders (e.g., by team), you can set a library at the folder level.

**Steps:**
1. Go to the **folder** in Jenkins
2. Click **Configure**
3. Under **Pipeline Libraries**, add the library

This way, only jobs inside that folder can use the library. Great for team-specific libraries!

### Method 3: Load Directly in Jenkinsfile (Dynamic)

You can load a library directly from a Git repo without any Jenkins configuration:

```groovy
// Load from a specific Git repo and branch
library identifier: 'my-shared-library@main',
        retriever: modernSCM([
            $class: 'GitSCMSource',
            remote: 'https://github.com/mycompany/jenkins-shared-library.git',
            credentialsId: 'github-credentials'
        ])

pipeline {
    agent any
    stages {
        stage('Build') {
            steps {
                dockerBuild(imageName: 'my-app')
            }
        }
    }
}
```

**When to use this?** When you want to test a library change or use a one-off library without asking an admin to configure it globally.

## 🛠️ Real-World Example — Complete Shared Library

Let's build a real-world shared library from scratch that handles the entire CI/CD pipeline.

### Step 1: Create the Library Structure

```
jenkins-shared-library/
├── vars/
│   ├── standardPipeline.groovy       ← Main pipeline template
│   ├── dockerBuild.groovy            ← Docker build helper
│   ├── dockerPush.groovy             ← Docker push helper
│   ├── runTests.groovy               ← Test runner
│   ├── deployToK8s.groovy            ← Kubernetes deployment
│   ├── notifySlack.groovy            ← Slack notifications
│   └── qualityGate.groovy            ← SonarQube quality gate
├── src/
│   └── org/
│       └── mycompany/
│           └── Constants.groovy      ← Shared constants
├── resources/
│   └── k8s-deployment-template.yaml
└── README.md
```

### Step 2: Write the Library Code

**`src/org/mycompany/Constants.groovy`** — Shared Constants:

```groovy
package org.mycompany

class Constants {
    static final String DOCKER_REGISTRY = 'myregistry.azurecr.io'
    static final String K8S_NAMESPACE_DEV = 'development'
    static final String K8S_NAMESPACE_STAGING = 'staging'
    static final String K8S_NAMESPACE_PROD = 'production'
    static final String SLACK_CHANNEL = '#deployments'
    static final String SONAR_URL = 'https://sonarqube.mycompany.com'
}
```

**`vars/runTests.groovy`** — Test Runner:

```groovy
def call(Map config = [:]) {
    def testCommand = config.testCommand ?: 'npm test'
    def reportDir = config.reportDir ?: 'test-results'
    
    echo "🧪 Running tests..."
    
    try {
        sh testCommand
        echo "✅ All tests passed!"
    } catch (Exception e) {
        echo "❌ Tests failed!"
        throw e  // Re-throw so pipeline fails
    } finally {
        // Archive test results regardless of pass/fail
        if (fileExists(reportDir)) {
            junit "${reportDir}/**/*.xml"
        }
    }
}
```

**`vars/notifySlack.groovy`** — Slack Notifications:

```groovy
import org.mycompany.Constants

def call(Map config = [:]) {
    def status = config.status ?: 'UNKNOWN'
    def channel = config.channel ?: Constants.SLACK_CHANNEL
    def message = config.message ?: ''
    
    // Set color based on status
    def color
    switch (status) {
        case 'SUCCESS':
            color = 'good'       // Green
            message = message ?: "✅ *${env.JOB_NAME}* #${env.BUILD_NUMBER} succeeded!"
            break
        case 'FAILURE':
            color = 'danger'     // Red
            message = message ?: "❌ *${env.JOB_NAME}* #${env.BUILD_NUMBER} failed!"
            break
        case 'STARTED':
            color = '#439FE0'    // Blue
            message = message ?: "🚀 *${env.JOB_NAME}* #${env.BUILD_NUMBER} started..."
            break
        default:
            color = 'warning'    // Yellow
            message = message ?: "⚠️ *${env.JOB_NAME}* #${env.BUILD_NUMBER} — ${status}"
    }
    
    // Add build info
    message += "\n<${env.BUILD_URL}|View Build>"
    
    slackSend(channel: channel, color: color, message: message)
}
```

**`vars/qualityGate.groovy`** — SonarQube Quality Gate:

```groovy
import org.mycompany.Constants

def call(Map config = [:]) {
    def sonarUrl = config.sonarUrl ?: Constants.SONAR_URL
    def projectKey = config.projectKey ?: env.JOB_NAME
    
    echo "📊 Running SonarQube analysis..."
    
    withSonarQubeEnv('SonarQube') {
        sh """
            sonar-scanner \
                -Dsonar.projectKey=${projectKey} \
                -Dsonar.host.url=${sonarUrl} \
                -Dsonar.sources=src \
                -Dsonar.tests=test
        """
    }
    
    echo "⏳ Waiting for Quality Gate result..."
    
    timeout(time: 5, unit: 'MINUTES') {
        def qg = waitForQualityGate()
        if (qg.status != 'OK') {
            error "❌ Quality Gate failed: ${qg.status}"
        }
    }
    
    echo "✅ Quality Gate passed!"
}
```

**`vars/deployToK8s.groovy`** — Kubernetes Deployment:

```groovy
import org.mycompany.Constants

def call(Map config = [:]) {
    def serviceName = config.serviceName ?: error("serviceName is required")
    def image = config.image ?: error("image is required")
    def namespace = config.namespace ?: Constants.K8S_NAMESPACE_DEV
    def replicas = config.replicas ?: 2
    def port = config.port ?: 8080
    
    echo "🚀 Deploying ${serviceName} to ${namespace}..."
    echo "   Image: ${image}"
    echo "   Replicas: ${replicas}"
    
    // Load template from resources/
    def template = libraryResource('k8s-deployment-template.yaml')
    
    // Replace placeholders
    def manifest = template
        .replace('__SERVICE_NAME__', serviceName)
        .replace('__IMAGE__', image)
        .replace('__NAMESPACE__', namespace)
        .replace('__REPLICAS__', replicas.toString())
        .replace('__PORT__', port.toString())
    
    // Write and apply
    writeFile file: "deployment-${serviceName}.yaml", text: manifest
    
    sh """
        kubectl apply -f deployment-${serviceName}.yaml -n ${namespace}
        kubectl rollout status deployment/${serviceName} -n ${namespace} --timeout=300s
    """
    
    echo "✅ ${serviceName} deployed successfully to ${namespace}!"
}
```

**`vars/standardPipeline.groovy`** — The Main Pipeline Template:

```groovy
import org.mycompany.Constants

def call(Map config = [:]) {
    // Extract configuration with defaults
    def serviceName = config.serviceName ?: error("serviceName is required")
    def dockerfilePath = config.dockerfile ?: 'Dockerfile'
    def testCommand = config.testCommand ?: 'npm test'
    def deployEnvs = config.deployEnvs ?: ['dev']
    def port = config.port ?: 8080
    def runSonar = config.runSonar != null ? config.runSonar : true
    def notifyOnSlack = config.notifyOnSlack != null ? config.notifyOnSlack : true
    
    def fullImageName = ""
    
    pipeline {
        agent any
        
        environment {
            DOCKER_REGISTRY = Constants.DOCKER_REGISTRY
            IMAGE_TAG = "${env.BUILD_NUMBER}"
        }
        
        options {
            timeout(time: 30, unit: 'MINUTES')
            disableConcurrentBuilds()
            buildDiscarder(logRotator(numToKeepStr: '10'))
            timestamps()
        }
        
        stages {
            stage('🔔 Notify Start') {
                steps {
                    script {
                        if (notifyOnSlack) {
                            notifySlack(status: 'STARTED')
                        }
                    }
                }
            }
            
            stage('📥 Checkout') {
                steps {
                    checkout scm
                }
            }
            
            stage('🧪 Test') {
                steps {
                    runTests(testCommand: testCommand)
                }
            }
            
            stage('📊 Quality Gate') {
                when {
                    expression { return runSonar }
                }
                steps {
                    qualityGate(projectKey: serviceName)
                }
            }
            
            stage('🐳 Build Docker Image') {
                steps {
                    script {
                        fullImageName = dockerBuild(
                            imageName: serviceName,
                            tag: env.IMAGE_TAG,
                            registry: env.DOCKER_REGISTRY,
                            dockerfile: dockerfilePath
                        )
                    }
                }
            }
            
            stage('📤 Push Docker Image') {
                steps {
                    dockerPush(
                        imageName: serviceName,
                        tag: env.IMAGE_TAG,
                        registry: env.DOCKER_REGISTRY
                    )
                }
            }
            
            stage('🚀 Deploy') {
                steps {
                    script {
                        for (deployEnv in deployEnvs) {
                            def namespace
                            switch (deployEnv) {
                                case 'dev':
                                    namespace = Constants.K8S_NAMESPACE_DEV
                                    break
                                case 'staging':
                                    namespace = Constants.K8S_NAMESPACE_STAGING
                                    break
                                case 'prod':
                                    namespace = Constants.K8S_NAMESPACE_PROD
                                    break
                                default:
                                    error "Unknown environment: ${deployEnv}"
                            }
                            
                            deployToK8s(
                                serviceName: serviceName,
                                image: fullImageName,
                                namespace: namespace,
                                port: port
                            )
                        }
                    }
                }
            }
        }
        
        post {
            success {
                script {
                    if (notifyOnSlack) {
                        notifySlack(status: 'SUCCESS')
                    }
                }
            }
            failure {
                script {
                    if (notifyOnSlack) {
                        notifySlack(status: 'FAILURE')
                    }
                }
            }
            always {
                cleanWs()  // Clean workspace after build
            }
        }
    }
}
```

## 🔀 Version Control for Shared Libraries

Since the library lives in a Git repo, you can use branches and tags for versioning:

```groovy
// Use the 'main' branch (default)
@Library('my-shared-library') _

// Use a specific branch
@Library('my-shared-library@develop') _

// Use a specific tag (recommended for production)
@Library('my-shared-library@v2.1.0') _

// Use a specific commit hash
@Library('my-shared-library@a1b2c3d') _
```

### Best Practice: Semantic Versioning

```
v1.0.0  ← Initial release
v1.1.0  ← Added Slack notifications
v1.2.0  ← Added SonarQube quality gate
v2.0.0  ← Breaking change: renamed parameters
v2.0.1  ← Bug fix
```

**Why version your library?**
- Production pipelines use `@Library('my-shared-library@v2.1.0')` — stable, tested
- Dev pipelines can use `@Library('my-shared-library@develop')` — latest changes
- If a new version breaks something, you can pin to the old version while you fix it

## 🧪 Testing Shared Libraries

Yes, you can (and should!) test your shared library code!

### Unit Testing with JenkinsPipelineUnit

[JenkinsPipelineUnit](https://github.com/jenkinsci/JenkinsPipelineUnit) is a testing framework that lets you test pipeline code without a running Jenkins instance.

**`pom.xml` (or `build.gradle`) for tests:**

```xml
<dependency>
    <groupId>com.lesfurets</groupId>
    <artifactId>jenkins-pipeline-unit</artifactId>
    <version>1.9</version>
    <scope>test</scope>
</dependency>
```

**`test/groovy/DockerBuildTest.groovy`:**

```groovy
import com.lesfurets.jenkins.unit.BasePipelineTest
import org.junit.Before
import org.junit.Test
import static org.junit.Assert.*

class DockerBuildTest extends BasePipelineTest {

    @Before
    void setUp() {
        super.setUp()
        // Register vars/ scripts
        helper.registerAllowedMethod('sh', [String], null)
        helper.registerAllowedMethod('echo', [String], null)
    }

    @Test
    void testDockerBuildWithDefaults() {
        def script = loadScript('vars/dockerBuild.groovy')
        
        // Set up environment
        binding.setVariable('env', [BUILD_NUMBER: '42'])
        
        def result = script.call(imageName: 'my-app')
        
        // Verify the docker build command was called
        assertJobStatusSuccess()
    }

    @Test(expected = Exception)
    void testDockerBuildWithoutImageNameFails() {
        def script = loadScript('vars/dockerBuild.groovy')
        script.call([:])  // No imageName — should fail
    }
}
```

### Integration Testing — Use a Test Pipeline

Create a test Jenkinsfile that exercises all your library functions:

```groovy
// test/Jenkinsfile.test

                }
            }
        }
        stage('Test notifySlack') {
            steps {
                notifySlack(status: 'SUCCESS', message: 'Library test passed!')
            }
        }
    }
}
```

## 🏗️ Advanced Patterns

### Pattern 1: Multiple Libraries

You can load multiple libraries:

```groovy
@Library(['company-shared-library', 'team-utils-library']) _

pipeline {
    agent any
    stages {
        stage('Build') {
            steps {
                // From company-shared-library
                dockerBuild(imageName: 'my-app')
                
                // From team-utils-library
                teamSpecificDeploy(env: 'staging')
            }
        }
    }
}
```

**What if both libraries have a function with the same name?**
The library listed **first** takes priority. So order matters!

### Pattern 2: Conditional Library Loading

```groovy
// Load library only if on a specific branch
if (env.BRANCH_NAME == 'main') {
    library 'production-library@v1.0.0'
} else {
    library 'development-library@latest'
}
```

### Pattern 3: Wrapping the Entire Pipeline

Instead of defining stages in the library, you can just provide helper steps and let teams define their own stages:

```groovy
// vars/withStandardSetup.groovy
def call(Closure body) {
    // Setup that runs BEFORE the user's code
    echo "📋 Standard setup: checking tools..."
    sh 'docker --version'
    sh 'kubectl version --client'
    
    try {
        // Run the user's pipeline code
        body()
    } catch (Exception e) {
        notifySlack(status: 'FAILURE')
        throw e
    } finally {
        // Cleanup that runs AFTER
        echo "🧹 Cleaning up..."
        cleanWs()
    }
}
```

**Usage:**

```groovy
@Library('my-shared-library') _

withStandardSetup {
    node {
        stage('Build') {
            checkout scm
            sh 'docker build -t my-app .'
        }
        stage('Test') {
            sh 'npm test'
        }
    }
}
```

### Pattern 4: Using `@NonCPS` for Non-Serializable Code

Some Groovy operations (like certain list/map manipulations) aren't serializable. Mark them with `@NonCPS`:

```groovy
// vars/parseChangelog.groovy

@NonCPS
def call(String changelog) {
    // Regex operations are not serializable, so we use @NonCPS
    def entries = []
    changelog.eachLine { line ->
        def matcher = line =~ /^## \[(.+)\] - (.+)$/
        if (matcher.matches()) {
            entries.add([version: matcher[0][1], date: matcher[0][2]])
        }
    }
    return entries
}
```

**What does `@NonCPS` mean?**
- CPS = **Continuation Passing Style** — Jenkins' way of making pipelines pausable/resumable
- `@NonCPS` says "don't try to serialize this method" — it runs straight through without checkpoints
- Use it for utility methods that don't call any pipeline steps (no `sh`, `echo`, etc.)
- **Warning:** You CANNOT call pipeline steps (`sh`, `echo`, `writeFile`, etc.) inside a `@NonCPS` method!

## 🔐 Security Considerations

### 1. Approve Scripts in Script Security

When you first use a shared library, Jenkins' Script Security plugin might block certain method calls. An admin needs to approve them:

**Manage Jenkins → In-Process Script Approval**

```
Pending Approvals:
  method java.lang.String replace java.lang.String java.lang.String
  staticMethod org.mycompany.Constants DOCKER_REGISTRY
  ...
  [Approve] [Deny]
```

**Tip:** Global shared libraries configured by admins run in a **trusted sandbox** and bypass script security. Folder-level and dynamically-loaded libraries run in a **sandboxed** environment.

### 2. Credential Management in Libraries

Never hardcode credentials! Use Jenkins credentials:

```groovy
// vars/dockerPush.groovy

def call(Map config = [:]) {
    def imageName = config.imageName ?: error("imageName is required")
    def tag = config.tag ?: env.BUILD_NUMBER
    def registry = config.registry ?: 'docker.io'
    def credentialsId = config.credentialsId ?: 'docker-registry-credentials'
    
    withCredentials([usernamePassword(
        credentialsId: credentialsId,
        usernameVariable: 'DOCKER_USER',
        passwordVariable: 'DOCKER_PASS'
    )]) {
        sh """
            echo "\${DOCKER_PASS}" | docker login ${registry} -u "\${DOCKER_USER}" --password-stdin
            docker push ${registry}/${imageName}:${tag}
            docker logout ${registry}
        """
    }
}
```

### 3. Library Access Control

- **Global libraries:** Only Jenkins admins can configure them — trusted by default
- **Folder libraries:** Folder admins can configure — sandboxed by default
- **Dynamic libraries:** Anyone can load any repo — most risk

**Best Practice:** Use global libraries for company-wide standards. Don't allow untrusted repos as dynamic libraries.

## 🐛 Debugging Shared Libraries

### 1. Replay a Build

In Jenkins, go to a build → click **Replay**. You can edit the shared library code temporarily for that build only. Super useful for debugging without committing code!

### 2. Print Debug Info

```groovy
// vars/debugInfo.groovy
def call() {
    echo "=== Debug Info ==="
    echo "Job Name:    ${env.JOB_NAME}"
    echo "Build Number: ${env.BUILD_NUMBER}"
    echo "Branch:       ${env.BRANCH_NAME ?: 'N/A'}"
    echo "Node Name:    ${env.NODE_NAME}"
    echo "Workspace:    ${env.WORKSPACE}"
    echo "=================="
}
```

### 3. Check the Library Version Being Used

```groovy
// In your Jenkinsfile
echo "Library loaded from: ${library('my-shared-library').version}"
```

| What You Want | Where to Put It | How to Call It |
|---|---|---|

| Complex class with methods | `src/org/company/MyClass.groovy` | `new MyClass(this).method()` |

| Shared constants | `src/org/company/Constants.groovy` | `Constants.MY_CONSTANT` |

## ✅ Best Practices Summary

```
1. ONE shared library per organization (or at most 2-3)
   Don't create a library per team — that defeats the purpose

2. Version your library with semantic versioning
   Tag releases: v1.0.0, v1.1.0, v2.0.0

3. Pin versions in production Jenkinsfiles
   @Library('my-shared-library@v2.1.0') _

4. Keep vars/ functions small and focused
   Each file should do ONE thing well

5. Always add implements Serializable to src/ classes

6. Never hardcode credentials — use withCredentials()

7. Test your library before deploying to production
   Use JenkinsPipelineUnit or a test pipeline

8. Document your library
   README.md with examples for every function

9. Use Replay for debugging — don't commit half-baked fixes

10. Review changes like production code
    PRs, code reviews, automated tests
```
