# 11.8 — Jenkins — Agents, Distributed Builds, Docker Agents

> **One Line Answer:** Jenkins Agents (also called Nodes or Slaves) are **separate machines or containers** that run your build jobs, so the main Jenkins server (Controller) doesn't do the heavy lifting — think of the Controller as the **brain** that schedules work, and Agents as the **hands** that do the actual building, testing, and deploying.

## 🤔 Why Do We Need Agents? — The Problem

Imagine your Jenkins is running on a single server. Everything works fine when you have 5 jobs. But what happens when you grow?

```
Single Jenkins Server (No Agents):

🖥️ Jenkins Server (Controller)
 ├── Running Job 1: Building Java app...         ← Using CPU
 ├── Running Job 2: Running Selenium tests...    ← Using Memory
 ├── Running Job 3: Building Docker image...     ← Using Disk I/O
 ├── Running Job 4: Waiting...                   ← Queue! 😤
 ├── Running Job 5: Waiting...                   ← Queue! 😤
 ├── Running Job 6: Waiting...                   ← Queue! 😤
 └── Also serving the UI, scheduling, logs...    ← Overloaded! 🔥

Problems:
❌ Jobs queue up and wait — slow feedback to developers
❌ One bad job can crash the entire Jenkins server
❌ Can't build for different OS (Linux + Windows + Mac)
❌ Security risk — all jobs run on the same machine
❌ No isolation between jobs — they can interfere with each other
❌ Can't scale — you're limited to one machine's resources
```

### The Joy With Agents:

```
Jenkins With Agents:

🧠 Jenkins Controller (Brain)
 ├── Schedules jobs
 ├── Serves the UI
 ├── Stores configurations
 └── Sends work to agents ──────┐
                                │
        ┌───────────────────────┤
        │                       │
   🤖 Agent 1              🤖 Agent 2              🤖 Agent 3
   (Linux, Java)           (Linux, Node.js)        (Windows, .NET)
   ├── Job 1               ├── Job 2               ├── Job 3
   └── Job 4               └── Job 5               └── Job 6

   All running in parallel! ⚡
```

```
Benefits With Agents:
✅ Parallel execution — run many jobs simultaneously
✅ Isolation — jobs don't interfere with each other
✅ Different environments — Linux, Windows, Mac agents
✅ Scalability — add more agents when you need more capacity
✅ Security — Controller doesn't run untrusted code
✅ Specialized agents — GPU for ML, high-memory for large builds
```

## 📖 Terminology — Controller vs Agent

Let's get the naming straight (Jenkins changed the terminology over the years):

| Old Term | New Term | What It Means |
|---|---|---|
| Master | **Controller** | The main Jenkins server. Schedules jobs, serves UI, stores config |
| Slave | **Agent** (or Node) | A machine that executes jobs on behalf of the Controller |
| Node | **Node** | Any machine in the Jenkins environment (Controller + all Agents) |
| Executor | **Executor** | A "slot" on a node that can run one job. A node can have multiple executors |
| Label | **Label** | A tag assigned to agents for targeting jobs (e.g., `linux`, `docker`, `gpu`) |

**Executors Explained:**

```
🤖 Agent "build-server-1"
   Executors: 4

   [Executor 1] → Running Job A    ← Busy
   [Executor 2] → Running Job B    ← Busy
   [Executor 3] → Idle             ← Available
   [Executor 4] → Idle             ← Available

   This agent can run up to 4 jobs simultaneously!
```

**How many executors should an agent have?**
- Rule of thumb: **Number of CPU cores** = number of executors
- For I/O-heavy jobs (downloads, Docker builds): can go higher
- For CPU-heavy jobs (compilation, testing): keep it at CPU core count
- Start with 2-4 and adjust based on observation

## 🔧 Types of Agents

### 1. Permanent (Static) Agents

These are dedicated machines (physical or VMs) that are always connected to Jenkins.

```
Permanent Agent:
┌──────────────────────────────────┐
│ 🤖 Agent: "build-linux-01"       │
│ OS: Ubuntu 22.04                 │
│ IP: 192.168.1.100                │
│ Executors: 4                     │
│ Labels: linux, docker, java      │
│ Status: Always Online            │
│ Tools: JDK 17, Maven, Docker     │
└──────────────────────────────────┘

When to use:
✅ You have dedicated build servers
✅ Predictable and consistent workload
✅ Need specialized hardware (GPU, high memory)
✅ On-premise environments
```

### 2. Cloud Agents (Dynamic / Ephemeral)

These agents are created on-demand when a job needs one, and destroyed after the job finishes.

```
Cloud Agent (Dynamic):
                              Job starts
                                  │
                    ┌─────────────▼──────────────┐
                    │  Jenkins Controller         │
                    │  "I need a Linux agent!"    │
                    └─────────────┬──────────────┘
                                  │
                    ┌─────────────▼──────────────┐
                    │  Cloud Plugin (AWS/K8s/etc) │
                    │  "Creating agent..."        │
                    └─────────────┬──────────────┘
                                  │
                    ┌─────────────▼──────────────┐
                    │  🤖 New Agent Created        │
                    │  Running the job...          │
                    └─────────────┬──────────────┘
                                  │
                              Job finishes
                                  │
                    ┌─────────────▼──────────────┐
                    │  🗑️ Agent Destroyed          │
                    │  (No cost when idle!)        │
                    └──────────────────────────────┘

When to use:
✅ Variable workloads (busy during the day, quiet at night)
✅ Cloud environments (AWS, Azure, GCP)
✅ Kubernetes-based Jenkins
✅ Cost optimization — pay only when building
```

### 3. Docker Agents

Run each build inside a Docker container. The container IS the agent.

```
Docker Agent:
┌─────────────────────────────────────────┐
│ Host Machine                            │
│  ┌───────────────────────────────────┐  │
│  │ 🐳 Container: node:18             │  │
│  │ Running: npm test                  │  │
│  │ Isolated filesystem               │  │
│  │ Destroyed after build              │  │
│  └───────────────────────────────────┘  │
│  ┌───────────────────────────────────┐  │
│  │ 🐳 Container: maven:3.9-jdk-17   │  │
│  │ Running: mvn package              │  │
│  │ Isolated filesystem               │  │
│  │ Destroyed after build              │  │
│  └───────────────────────────────────┘  │
│  ┌───────────────────────────────────┐  │
│  │ 🐳 Container: python:3.12        │  │
│  │ Running: pytest                   │  │
│  │ Isolated filesystem               │  │
│  │ Destroyed after build              │  │
│  └───────────────────────────────────┘  │
└─────────────────────────────────────────┘

When to use:
✅ Different language/tool versions per project
✅ Clean environment for every build (no leftover state)
✅ Don't want to install tools on the agent machine
✅ Reproducible builds
```

## 🖥️ Setting Up a Permanent Agent

### Step 1: Prepare the Agent Machine

On the machine you want to use as an agent:

```bash
# 1. Install Java (Jenkins agents require Java)
sudo apt update
sudo apt install openjdk-17-jdk -y

# Verify Java
java -version
# openjdk version "17.0.x" ...

# 2. Create a Jenkins user
sudo useradd -m -d /home/jenkins -s /bin/bash jenkins

# 3. Create a workspace directory
sudo mkdir -p /var/jenkins
sudo chown jenkins:jenkins /var/jenkins

# 4. Set up SSH key authentication (for SSH launch method)
sudo -u jenkins ssh-keygen -t ed25519 -C "jenkins-agent" -f /home/jenkins/.ssh/id_ed25519 -N ""

# 5. Add the Controller's public key to authorized_keys
# (Copy the Controller's public key and paste it here)
sudo -u jenkins bash -c 'echo "ssh-ed25519 AAAA...controller-key..." >> /home/jenkins/.ssh/authorized_keys'
sudo chmod 700 /home/jenkins/.ssh
sudo chmod 600 /home/jenkins/.ssh/authorized_keys
```

### Step 2: Add the Agent in Jenkins UI

1. Go to **Manage Jenkins** → **Nodes** (or **Manage Nodes and Clouds**)
2. Click **New Node**
3. Enter a name: `build-linux-01`
4. Select **Permanent Agent**
5. Configure:

```
Name:                   build-linux-01
Description:            Linux build agent for Java and Docker projects
Number of Executors:    4
Remote Root Directory:  /var/jenkins
Labels:                 linux docker java
Usage:                  "Use this node as much as possible"
                        OR
                        "Only build jobs with label expressions matching this node"
Launch Method:          Launch agents via SSH
  Host:                 192.168.1.100
  Credentials:          (select or add SSH credentials)
  Host Key Verification: Known hosts file Verification Strategy
Availability:           Keep this agent online as much as possible
```

**Understanding "Usage" options:**

```
"Use this node as much as possible"
  → Any job can run here, even without a matching label
  → Good for general-purpose agents

"Only build jobs with label expressions matching this node"
  → Only jobs that explicitly request labels like "linux" or "docker" will run here
  → Good for specialized agents (e.g., GPU machines)
```

### Step 3: Launch Methods Explained

| Method | How It Works | Best For |
|---|---|---|
| **Launch via SSH** | Controller SSHs into the agent and starts the agent process | Linux agents, most common |
| **Launch via JNLP (Java Web Start)** | Agent connects TO the Controller (outbound connection) | Windows agents, agents behind firewalls |
| **Launch via command** | Run a custom command to start the agent | Custom setups |
| **Let Jenkins control this Windows agent** | Windows service management | Windows agents |

**SSH Launch (Most Common):**

```
Jenkins Controller ───SSH──→ Agent Machine
                              └── Starts agent.jar
                              └── Agent connects back to Controller
                              └── Ready to receive jobs!
```

**JNLP / Inbound Launch (Agent Initiates Connection):**

```
Agent Machine ───HTTPS──→ Jenkins Controller
                          └── "I'm agent build-linux-01!"
                          └── Controller: "Welcome! Here's work for you."

This is useful when:
- The agent is behind a firewall/NAT
- The Controller can't SSH into the agent
- Cloud environments where IPs change
```

**To connect via JNLP, run this on the agent machine:**

```bash
# Download agent.jar from Jenkins
curl -O https://your-jenkins-url/jnlpJars/agent.jar

# Connect to Jenkins
java -jar agent.jar \
    -url https://your-jenkins-url \
    -secret <agent-secret> \
    -name build-linux-01 \
    -workDir /var/jenkins
```

## 🏷️ Labels — Targeting the Right Agent

Labels are **tags** you assign to agents. Pipelines use **label expressions** to pick the right agent.

### Assigning Labels

When configuring an agent, set the Labels field:

```
Agent: build-linux-01
Labels: linux docker java maven
          │      │     │     │
          │      │     │     └── Has Maven installed
          │      │     └──────── Has JDK installed
          │      └────────────── Has Docker installed
          └───────────────────── Runs Linux OS
```

### Using Labels in Pipelines

**Declarative Pipeline:**

```groovy
pipeline {
    // Run on ANY agent with the "linux" label
    agent { label 'linux' }
    
    stages {
        stage('Build') {
            steps {
                sh 'echo "Running on a Linux agent!"'
            }
        }
    }
}
```

**Label Expressions — Boolean Logic:**

```groovy
// Run on an agent that has BOTH "linux" AND "docker" labels
agent { label 'linux && docker' }

// Run on an agent that has EITHER "linux" OR "mac" label
agent { label 'linux || mac' }

// Run on an agent that has "linux" but NOT "gpu"
agent { label 'linux && !gpu' }

// Complex: Linux with Docker OR Mac with Docker
agent { label '(linux || mac) && docker' }
```

**Different Agents for Different Stages:**

```groovy
pipeline {
    agent none  // Don't assign a global agent
    
    stages {
        stage('Build') {
            agent { label 'linux && docker' }  // Need Docker for building
            steps {
                sh 'docker build -t my-app .'
            }
        }
        
        stage('Test on Linux') {
            agent { label 'linux' }
            steps {
                sh './run-tests.sh'
            }
        }
        
        stage('Test on Windows') {
            agent { label 'windows' }
            steps {
                bat 'run-tests.bat'  // bat for Windows commands
            }
        }
        
        stage('Deploy') {
            agent { label 'linux && kubectl' }  // Need kubectl
            steps {
                sh 'kubectl apply -f deployment.yaml'
            }
        }
    }
}
```

## 🐳 Docker Agents — The Modern Way

Docker agents are the **most popular** way to run Jenkins builds today. Instead of installing tools on agent machines, you just use Docker images that already have everything you need.

### Method 1: Docker Agent in Declarative Pipeline

```groovy
pipeline {
    // Run the entire pipeline inside a Docker container
    agent {
        docker {
            image 'node:18-alpine'      // Use this Docker image
            args '-v /tmp:/tmp'         // Optional: mount volumes
        }
    }
    
    stages {
        stage('Install') {
            steps {
                sh 'node --version'     // Node.js 18 is available!
                sh 'npm install'
            }
        }
        stage('Test') {
            steps {
                sh 'npm test'
            }
        }
        stage('Build') {
            steps {
                sh 'npm run build'
            }
        }
    }
}
```

**What happens behind the scenes:**

```
1. Jenkins looks for an agent that has Docker installed
2. On that agent, it runs:
   docker run -d -v /workspace:/workspace node:18-alpine sleep infinity
3. All pipeline commands (sh 'npm install', etc.) run INSIDE this container
4. When the pipeline finishes, the container is removed
5. Clean slate for the next build!
```

### Method 2: Different Docker Images Per Stage

This is incredibly powerful — each stage can use a completely different environment:

```groovy
pipeline {
    agent none  // No global agent
    
    stages {
        stage('Build Frontend') {
            agent {
                docker { image 'node:18-alpine' }
            }
            steps {
                sh 'npm install'
                sh 'npm run build'
                stash includes: 'dist/**', name: 'frontend-build'
            }
        }
        
        stage('Build Backend') {
            agent {
                docker { image 'maven:3.9-eclipse-temurin-17' }
            }
            steps {
                sh 'mvn clean package -DskipTests'
                stash includes: 'target/*.jar', name: 'backend-build'
            }
        }
        
        stage('Test Backend') {
            agent {
                docker { image 'maven:3.9-eclipse-temurin-17' }
            }
            steps {
                sh 'mvn test'
            }
        }
        
        stage('Build Docker Image') {
            agent { label 'docker' }  // Need Docker on the host
            steps {
                unstash 'frontend-build'
                unstash 'backend-build'
                sh 'docker build -t my-app:${BUILD_NUMBER} .'
            }
        }
        
        stage('Deploy') {
            agent {
                docker { image 'bitnami/kubectl:latest' }
            }
            steps {
                sh 'kubectl apply -f k8s/'
            }
        }
    }
}
```

**Why is this amazing?**
- Frontend uses `node:18` — no need to install Node.js anywhere
- Backend uses `maven:3.9` — no need to install Maven or JDK
- Deploy uses `kubectl` — no need to install kubectl
- Each stage gets a fresh container — no leftover files or state

### Method 3: Custom Dockerfile as Agent

Don't want to use a public image? Build your own on-the-fly:

```groovy
pipeline {
    agent {
        dockerfile {
            filename 'Dockerfile.ci'   // Use this Dockerfile
            dir 'ci'                   // Look in the ci/ directory
            args '-v /var/run/docker.sock:/var/run/docker.sock'  // Docker-in-Docker
            additionalBuildArgs '--build-arg HTTP_PROXY=http://proxy:8080'
        }
    }
    
    stages {
        stage('Build') {
            steps {
                sh 'make build'
            }
        }
    }
}
```

**`ci/Dockerfile.ci`:**

```dockerfile
FROM ubuntu:22.04

# Install everything your build needs
RUN apt-get update && apt-get install -y \
    curl \
    git \
    jq \
    python3 \
    python3-pip \
    docker.io \
    && rm -rf /var/lib/apt/lists/*

# Install specific tools
RUN pip3 install awscli boto3

# Install kubectl
RUN curl -LO "https://dl.k8s.io/release/$(curl -L -s https://dl.k8s.io/release/stable.txt)/bin/linux/amd64/kubectl" \
    && chmod +x kubectl \
    && mv kubectl /usr/local/bin/
```

### Docker Agent — Advanced Options

```groovy
pipeline {
    agent {
        docker {
            image 'node:18'
            
            // Run on a specific host agent first (must have Docker)
            label 'docker-host'
            
            // Pass arguments to docker run
            args '''
                -v /home/jenkins/.npm:/root/.npm
                -v /var/run/docker.sock:/var/run/docker.sock
                -e "NODE_ENV=test"
                --network=host
                --memory=2g
                --cpus=2
            '''
            
            // Use a custom Docker registry
            registryUrl 'https://myregistry.azurecr.io'
            registryCredentialsId 'acr-credentials'
            
            // Reuse containers between builds (faster, but less isolated)
            reuseNode true
            
            // Custom workspace inside the container
            customWorkspace '/home/app'
        }
    }
    
    stages {
        stage('Build') {
            steps {
                sh 'npm install && npm run build'
            }
        }
    }
}
```

**What does `reuseNode true` do?**

```
Without reuseNode (default):
  Build 1: New container → npm install (slow, downloads everything)
  Build 2: New container → npm install (slow again!)
  Build 3: New container → npm install (slow again!)

With reuseNode true:
  Build 1: New container → npm install (slow first time)
  Build 2: Same container → npm install (fast! node_modules cached)
  Build 3: Same container → npm install (fast!)

Trade-off:
  ✅ Faster builds (caching)
  ❌ Less isolation (leftover state from previous builds)
```

## ☸️ Kubernetes Agents — Auto-Scaling Agents in K8s

The **Kubernetes plugin** for Jenkins creates Pods as agents on-demand. Each build gets its own Pod, and the Pod is deleted after the build. This is the most scalable approach.

### How It Works

```
                         ┌─────────────────────────────────┐
                         │  Kubernetes Cluster              │
                         │                                  │
  🧠 Jenkins Controller  │   ┌──────────────────────┐      │
  "I need an agent!" ────┼──►│  📦 Pod: build-abc    │      │
                         │   │  ├── 🐳 jnlp (agent) │      │
                         │   │  ├── 🐳 maven         │      │
                         │   │  └── 🐳 docker         │      │
                         │   └──────────────────────┘      │
                         │                                  │
                         │   ┌──────────────────────┐      │
                         │   │  📦 Pod: build-def    │      │
                         │   │  ├── 🐳 jnlp (agent) │      │
                         │   │  └── 🐳 node          │      │
                         │   └──────────────────────┘      │
                         │                                  │
                         └─────────────────────────────────┘

Each Pod:
- Has a JNLP container (connects to Jenkins as an agent)
- Has one or more tool containers (maven, docker, node, etc.)
- Runs the build
- Gets deleted when done
```

### Setting Up Kubernetes Cloud

1. Install the **Kubernetes plugin** in Jenkins
2. Go to **Manage Jenkins** → **Clouds** → **New Cloud** → **Kubernetes**
3. Configure:

```
Name:                   kubernetes
Kubernetes URL:         https://kubernetes.default.svc
                        (if Jenkins runs inside the cluster)
                        OR
                        https://your-k8s-api-server:6443
Kubernetes Namespace:   jenkins
Credentials:            (K8s service account or kubeconfig)
Jenkins URL:            http://jenkins.jenkins.svc.cluster.local:8080
Jenkins Tunnel:         jenkins-agent.jenkins.svc.cluster.local:50000
```

### Using Kubernetes Agents in Pipelines

**Basic — Single Container:**

```groovy
pipeline {
    agent {
        kubernetes {
            yaml '''
apiVersion: v1
kind: Pod
metadata:
  labels:
    app: jenkins-agent
spec:
  containers:
    - name: maven
      image: maven:3.9-eclipse-temurin-17
      command:
        - sleep
      args:
        - infinity
      resources:
        requests:
          memory: "512Mi"
          cpu: "500m"
        limits:
          memory: "1Gi"
          cpu: "1000m"
'''
        }
    }
    
    stages {
        stage('Build') {
            steps {
                container('maven') {
                    sh 'mvn --version'
                    sh 'mvn clean package'
                }
            }
        }
    }
}
```

**Advanced — Multi-Container Pod:**

```groovy
pipeline {
    agent {
        kubernetes {
            yaml '''
apiVersion: v1
kind: Pod
metadata:
  labels:
    app: jenkins-agent
spec:
  serviceAccountName: jenkins-agent
  containers:
    - name: maven
      image: maven:3.9-eclipse-temurin-17
      command: ["sleep"]
      args: ["infinity"]
      volumeMounts:
        - name: maven-cache
          mountPath: /root/.m2

    - name: docker
      image: docker:24-dind
      securityContext:
        privileged: true
      env:
        - name: DOCKER_TLS_CERTDIR
          value: ""
      volumeMounts:
        - name: docker-socket
          mountPath: /var/run

    - name: kubectl
      image: bitnami/kubectl:latest
      command: ["sleep"]
      args: ["infinity"]

  volumes:
    - name: maven-cache
      persistentVolumeClaim:
        claimName: maven-cache-pvc
    - name: docker-socket
      emptyDir: {}
'''
        }
    }
    
    stages {
        stage('Build') {
            steps {
                container('maven') {
                    sh 'mvn clean package -DskipTests'
                }
            }
        }
        
        stage('Test') {
            steps {
                container('maven') {
                    sh 'mvn test'
                }
            }
        }
        
        stage('Docker Build') {
            steps {
                container('docker') {
                    sh 'docker build -t my-app:${BUILD_NUMBER} .'
                    sh 'docker push my-app:${BUILD_NUMBER}'
                }
            }
        }
        
        stage('Deploy') {
            steps {
                container('kubectl') {
                    sh 'kubectl apply -f k8s/deployment.yaml'
                    sh 'kubectl rollout status deployment/my-app'
                }
            }
        }
    }
}
```

**What's the `container('name')` block?**
Each Pod has multiple containers. The `container('maven')` block says "run these steps inside the `maven` container." Without it, commands run in the default `jnlp` container (the Jenkins agent).

### Pod Templates — Reusable Agent Definitions

Instead of writing the Pod YAML in every Jenkinsfile, define templates in Jenkins configuration:

**In Jenkins UI:** Manage Jenkins → Clouds → Kubernetes → Pod Templates

```
Name:            maven-agent
Labels:          maven java
Namespace:       jenkins
Containers:
  - Name: maven
    Docker Image: maven:3.9-eclipse-temurin-17
    Command: sleep
    Args: infinity
    Resource Request CPU: 500m
    Resource Request Memory: 512Mi
    Resource Limit CPU: 1000m
    Resource Limit Memory: 1Gi
```

**Then in Jenkinsfile:**

```groovy
pipeline {
    agent { label 'maven' }  // Uses the pod template with label "maven"
    
    stages {
        stage('Build') {
            steps {
                container('maven') {
                    sh 'mvn clean package'
                }
            }
        }
    }
}
```

## 🔗 Stash & Unstash — Passing Files Between Agents

When different stages run on different agents, files don't automatically transfer. Use `stash` and `unstash`:

```groovy
pipeline {
    agent none
    
    stages {
        stage('Build on Linux') {
            agent { label 'linux' }
            steps {
                sh 'make build'
                // Save the build output to transfer to another agent
                stash includes: 'build/**', name: 'build-artifacts'
            }
        }
        
        stage('Test on Windows') {
            agent { label 'windows' }
            steps {
                // Get the build output from the previous stage
                unstash 'build-artifacts'
                bat 'run-tests.bat'
            }
        }
    }
}
```

**How stash/unstash works:**

```
Agent 1 (Linux):
  1. Build produces files in build/
  2. stash: Compresses build/ → sends to Jenkins Controller
  
Jenkins Controller:
  Holds the stashed files temporarily

Agent 2 (Windows):
  1. unstash: Gets the compressed files from Controller
  2. Extracts to the workspace
  3. Tests run against the build output
```

**Important Limitations:**
- Stash is for **small files** (< 100 MB). For large artifacts, use `archiveArtifacts` or external storage (S3, Nexus)
- Stash data is lost when the build finishes
- Each stash name must be unique within a build

## 🛡️ Security Best Practices for Agents

### 1. Controller Isolation

```
⚠️ NEVER run builds on the Controller!

Why?
- A malicious build script could access Jenkins config
- It could steal credentials
- It could modify other jobs
- It could crash Jenkins itself

How to enforce:
  Manage Jenkins → Nodes → Built-In Node → Configure
    Number of Executors: 0    ← Set to ZERO!
    Usage: "Only build jobs with label expressions matching this node"
```

### 2. Agent-to-Controller Security

```
Manage Jenkins → Security → Agent/Controller Access Control
  ☑ Enable Agent → Controller Access Control

This prevents agents from:
  ❌ Reading files on the Controller
  ❌ Executing commands on the Controller
  ❌ Accessing other agents' workspaces
```

### 3. Use Ephemeral Agents

```
Permanent Agent:
  ❌ Secrets from Build A might be accessible to Build B
  ❌ Malware from one build persists on the agent
  ❌ Hard to audit what's on the agent

Ephemeral Agent (Docker/K8s):
  ✅ Fresh environment for every build
  ✅ No leftover secrets or state
  ✅ Easier to audit — just check the image
  ✅ Compromise is limited to one build
```

### 4. Limit Agent Permissions

```bash
# The Jenkins agent user should have minimal permissions
# Don't run as root!

# Good: Dedicated jenkins user
User:  jenkins
Groups: docker (if Docker is needed)
Home:  /home/jenkins
Shell: /bin/bash

# Bad: Running as root
User: root  ← NEVER do this!
```

## 📊 Monitoring Agents

### Jenkins Built-In Monitoring

```
Manage Jenkins → Nodes

You'll see:
┌─────────────────┬─────────┬────────────┬──────────┐
│ Agent Name       │ Status  │ Executors  │ Labels   │
├─────────────────┼─────────┼────────────┼──────────┤

│ build-linux-02   │ ✅ Online│ 0/4 busy   │ linux    │
│ build-windows-01 │ ❌ Offline│ 0/2       │ windows  │
│ build-docker-01  │ ✅ Online│ 3/4 busy   │ docker   │
└─────────────────┴─────────┴────────────┴──────────┘
```

### Key Metrics to Watch

```
1. Queue Length — How many jobs are waiting for an agent?
   → If consistently > 0, you need more agents

2. Agent Utilization — What % of executors are busy?
   → > 80% consistently = add more agents
   → < 20% consistently = remove some agents (save cost)

3. Agent Uptime — Are agents going offline?
   → Check connectivity, disk space, memory

4. Build Times — Are builds getting slower?
   → Agent might need more resources (CPU, memory, faster disk)
```

## 📋 Quick Reference — Choosing the Right Agent Type

| Scenario | Recommended Agent Type |
|---|---|
| Small team, few jobs | Permanent agent (1-2 VMs) |
| Growing team, 10-50 jobs | Docker agents on permanent nodes |
| Large org, 100+ jobs | Kubernetes agents (auto-scaling) |
| Cloud-native company | Kubernetes agents |
| Mixed OS (Linux + Windows) | Permanent agents with labels |
| Cost-sensitive | Cloud agents (EC2, Azure VM Scale Sets) |
| Security-critical | Ephemeral (Docker/K8s) agents |
| GPU/ML workloads | Permanent agents with GPU labels |

## ✅ Best Practices Summary

```
1. NEVER run builds on the Controller
   Set Controller executors to 0

2. Use labels to organize and target agents
   linux, docker, java, gpu, windows, etc.

3. Prefer ephemeral agents (Docker/K8s) over permanent agents
   Clean environment, better security, easier scaling

4. Match executors to CPU cores
   Start with 2-4 and adjust

5. Use Docker agents for language-specific builds
   node:18 for JS, maven:3.9 for Java — no tool installation needed

6. Cache dependencies for faster builds
   Mount volumes for node_modules, .m2, pip cache

7. Monitor agent utilization
   Add agents when queue is consistently > 0

8. Keep agent images minimal
   Only install what's needed — smaller image = faster startup

9. Use Kubernetes agents for auto-scaling
   Scale to zero when idle — save cost

10. Secure your agents
    Non-root user, limited permissions, agent-to-controller security
```
