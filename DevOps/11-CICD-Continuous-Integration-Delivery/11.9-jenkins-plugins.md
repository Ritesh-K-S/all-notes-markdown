# 11.9 — Jenkins — Plugins (Blue Ocean, Credentials, Git, Docker, Kubernetes)

> **One Line Answer:** Jenkins plugins are **add-ons** that extend Jenkins' functionality — out of the box, Jenkins is like a basic phone, but with plugins it becomes a smartphone. There are **1,800+ plugins** for everything from Git integration to Kubernetes deployments to Slack notifications.

---

## 🤔 What Are Jenkins Plugins? — The Big Picture

Jenkins by itself is actually quite basic. It can run shell commands and that's about it. The real power comes from **plugins**. Almost every feature you use in Jenkins is actually a plugin:

```
Jenkins Without Plugins:
┌─────────────────────────────────┐
│  Jenkins Core                   │
│  ├── Run shell commands (sh)    │
│  ├── Basic UI                   │
│  ├── Job scheduling             │
│  └── That's... pretty much it   │
└─────────────────────────────────┘

Jenkins With Plugins:
┌─────────────────────────────────────────────────────────────┐
│  Jenkins Core + Plugins                                     │
│  ├── Git integration          (Git Plugin)                  │
│  ├── Pipeline as Code         (Pipeline Plugin)             │
│  ├── Docker builds            (Docker Plugin)               │
│  ├── Kubernetes agents        (Kubernetes Plugin)           │
│  ├── Credentials management   (Credentials Plugin)          │
│  ├── Beautiful UI             (Blue Ocean Plugin)           │
│  ├── Slack notifications      (Slack Notification Plugin)   │
│  ├── SonarQube integration    (SonarQube Scanner Plugin)    │
│  ├── AWS integration          (AWS Steps Plugin)            │
│  ├── LDAP authentication      (LDAP Plugin)                │
│  ├── Email notifications      (Email Extension Plugin)      │
│  ├── Test reports             (JUnit Plugin)                │
│  └── 1,800+ more...                                        │
└─────────────────────────────────────────────────────────────┘
```

## ⚙️ Managing Plugins — Installation & Updates

### Installing Plugins via UI

1. Go to **Manage Jenkins** → **Plugins** (or **Manage Plugins** in older versions)
2. You'll see four tabs:

```
┌──────────────┬───────────────┬──────────────┬──────────────┐
│  Updates     │  Available    │  Installed   │  Advanced    │
├──────────────┴───────────────┴──────────────┴──────────────┤
│                                                            │
│  Search: [docker_________________] 🔍                      │
│                                                            │
│  ☐ Docker Pipeline              Install                   │
│  ☐ Docker plugin                Install                   │
│  ☐ Docker Commons               Install                   │
│                                                            │
│  [Install without restart]  [Download now, install after]  │
└────────────────────────────────────────────────────────────┘
```

- **Updates:** Plugins with available updates
- **Available:** All plugins you haven't installed yet
- **Installed:** Currently installed plugins
- **Advanced:** Upload plugin manually (.hpi file), configure update site

### Installing Plugins via CLI

```bash
# Using Jenkins CLI
java -jar jenkins-cli.jar -s http://localhost:8080/ install-plugin docker-workflow

# Install multiple plugins at once
java -jar jenkins-cli.jar -s http://localhost:8080/ install-plugin \
    git \
    docker-workflow \
    kubernetes \
    credentials-binding \
    blueocean

# Restart Jenkins after installing
java -jar jenkins-cli.jar -s http://localhost:8080/ safe-restart
```

### Installing Plugins via Docker (Pre-baked)

When running Jenkins in Docker, you can pre-install plugins using a `plugins.txt` file:

**`plugins.txt`:**

```
# Core pipeline plugins
workflow-aggregator:latest
pipeline-stage-view:latest

# Git
git:latest
github-branch-source:latest

# Docker
docker-workflow:latest
docker-plugin:latest

# Kubernetes
kubernetes:latest

# Credentials
credentials-binding:latest
credentials:latest

# UI
blueocean:latest

# Notifications
slack:latest
email-ext:latest

# Code quality
sonar:latest

# Utils
timestamper:latest
ansicolor:latest
```

**`Dockerfile`:**

```dockerfile
FROM jenkins/jenkins:lts

# Skip initial setup wizard
ENV JAVA_OPTS="-Djenkins.install.runSetupWizard=false"

# Copy and install plugins
COPY plugins.txt /usr/share/jenkins/ref/plugins.txt
RUN jenkins-plugin-cli --plugin-file /usr/share/jenkins/ref/plugins.txt
```

**Why pre-install plugins?**
- Reproducible Jenkins setup — same plugins every time
- Faster startup — no manual installation needed
- Infrastructure as Code — your Jenkins config is in Git
- Easy to recreate if Jenkins crashes

## 🔑 Credentials Plugin — Managing Secrets Securely

The Credentials plugin is one of the most critical plugins. It provides a **secure way to store and use sensitive information** like passwords, API tokens, SSH keys, and certificates.

### Types of Credentials

```
Jenkins Credentials Types:
┌────────────────────────────────┬─────────────────────────────────────┐
│ Type                           │ Use Case                            │
├────────────────────────────────┼─────────────────────────────────────┤
│ Username with password         │ Git repos, Docker registries, APIs  │
│ Secret text                    │ API tokens, Slack tokens            │
│ Secret file                    │ Kubeconfig, service account JSON    │
│ SSH Username with private key  │ Git SSH, server access              │
│ Certificate                    │ Client certificates (PKCS#12)       │
│ AWS Credentials                │ AWS Access Key + Secret Key         │
└────────────────────────────────┴─────────────────────────────────────┘
```

### Adding Credentials

**Via UI:**

1. Go to **Manage Jenkins** → **Credentials**
2. Click on a **domain** (usually `(global)`)
3. Click **Add Credentials**
4. Fill in:

```
Kind:        Username with password
Scope:       Global (or System)
Username:    my-docker-user
Password:    ********** (hidden)
ID:          docker-registry-creds    ← This is what you reference in pipelines
Description: Docker Hub credentials for CI/CD
```

### Credential Scopes

```
System Scope:
  - Only available to Jenkins system (plugins, admin config)
  - Cannot be used in pipeline scripts
  - Use for: Jenkins-to-Jenkins connections, plugin configs

Global Scope:
  - Available everywhere — system + all pipelines
  - Use for: Shared credentials (Docker registry, Git, AWS)

Folder Scope:
  - Only available to jobs within a specific folder
  - Use for: Team-specific credentials
  
  Example:
    📁 Team-A/
      🔑 team-a-aws-creds (only Team A's jobs can use this)
    📁 Team-B/
      🔑 team-b-aws-creds (only Team B's jobs can use this)
```

### Using Credentials in Pipelines

**1. Username & Password:**

```groovy
pipeline {
    agent any
    stages {
        stage('Docker Login') {
            steps {
                withCredentials([usernamePassword(
                    credentialsId: 'docker-registry-creds',
                    usernameVariable: 'DOCKER_USER',
                    passwordVariable: 'DOCKER_PASS'
                )]) {
                    sh '''
                        echo "${DOCKER_PASS}" | docker login -u "${DOCKER_USER}" --password-stdin
                        docker push my-app:latest
                    '''
                    // Outside this block, DOCKER_USER and DOCKER_PASS don't exist
                }
            }
        }
    }
}
```

**What's happening here?**
- `withCredentials` fetches the credential from Jenkins securely
- Creates temporary environment variables (`DOCKER_USER`, `DOCKER_PASS`)
- Variables are **masked** in the console output — if the password appears in logs, it shows `****`
- Variables are only available inside the `withCredentials` block
- After the block ends, the variables are cleaned up

**2. Secret Text (API Token):**

```groovy
pipeline {
    agent any
    stages {
        stage('Call API') {
            steps {
                withCredentials([string(
                    credentialsId: 'slack-webhook-token',
                    variable: 'SLACK_TOKEN'
                )]) {
                    sh '''
                        curl -X POST https://hooks.slack.com/services/${SLACK_TOKEN} \
                            -H "Content-Type: application/json" \
                            -d '{"text": "Build completed!"}'
                    '''
                }
            }
        }
    }
}
```

**3. SSH Key:**

```groovy
pipeline {
    agent any
    stages {
        stage('Deploy via SSH') {
            steps {
                withCredentials([sshUserPrivateKey(
                    credentialsId: 'deploy-server-ssh',
                    keyFileVariable: 'SSH_KEY',
                    usernameVariable: 'SSH_USER'
                )]) {
                    sh '''
                        ssh -i "${SSH_KEY}" -o StrictHostKeyChecking=no \
                            ${SSH_USER}@production-server \
                            "cd /app && docker-compose pull && docker-compose up -d"
                    '''
                }
            }
        }
    }
}
```

**4. Secret File (Kubeconfig):**

```groovy
pipeline {
    agent any
    stages {
        stage('Deploy to K8s') {
            steps {
                withCredentials([file(
                    credentialsId: 'kubeconfig-production',
                    variable: 'KUBECONFIG'
                )]) {
                    sh '''
                        kubectl --kubeconfig=${KUBECONFIG} apply -f deployment.yaml
                        kubectl --kubeconfig=${KUBECONFIG} rollout status deployment/my-app
                    '''
                }
            }
        }
    }
}
```

**5. Multiple Credentials at Once:**

```groovy
pipeline {
    agent any
    stages {
        stage('Build and Push') {
            steps {
                withCredentials([
                    usernamePassword(
                        credentialsId: 'docker-creds',
                        usernameVariable: 'DOCKER_USER',
                        passwordVariable: 'DOCKER_PASS'
                    ),
                    string(
                        credentialsId: 'sonar-token',
                        variable: 'SONAR_TOKEN'
                    ),
                    file(
                        credentialsId: 'kubeconfig',
                        variable: 'KUBECONFIG'
                    )
                ]) {
                    sh '''
                        echo "${DOCKER_PASS}" | docker login -u "${DOCKER_USER}" --password-stdin
                        docker build -t my-app .
                        docker push my-app
                    '''
                }
            }
        }
    }
}
```

### The `credentials()` Helper (Shortcut)

For simpler cases, you can use the `credentials()` helper in the `environment` block:

```groovy
pipeline {
    agent any
    
    environment {
        // Username-password → creates _USR and _PSW variables automatically
        DOCKER_CREDS = credentials('docker-registry-creds')
        // Creates:
        //   DOCKER_CREDS       → username:password
        //   DOCKER_CREDS_USR   → username
        //   DOCKER_CREDS_PSW   → password
        
        // Secret text → creates a single variable
        SLACK_TOKEN = credentials('slack-webhook-token')
        // Creates:
        //   SLACK_TOKEN → the secret value
        
        // Secret file → creates path to temp file
        KUBECONFIG = credentials('kubeconfig-prod')
        // Creates:
        //   KUBECONFIG → /tmp/secretFiles/kubeconfig-prod-xxxxx
    }
    
    stages {
        stage('Deploy') {
            steps {
                sh '''
                    echo "${DOCKER_CREDS_PSW}" | docker login -u "${DOCKER_CREDS_USR}" --password-stdin
                    kubectl --kubeconfig=${KUBECONFIG} apply -f deployment.yaml
                '''
            }
        }
    }
}
```

### Credential Security Best Practices

```
1. Use FOLDER-scoped credentials for team isolation
   Team A can't accidentally use Team B's AWS keys

2. Use credential IDs that are descriptive
   Good: "production-aws-deploy-role"
   Bad:  "cred-1"

3. Rotate credentials regularly
   Update the credential in Jenkins → all pipelines auto-pick up the new value

4. Audit credential usage
   Manage Jenkins → Credentials → Usage tracking shows which jobs use which creds

5. Never print credentials in logs
   Jenkins masks them, but be careful with:
   sh "echo ${PASSWORD}"   ← BAD! Might leak
   
   Use:
   sh 'echo "${PASSWORD}"'  ← Better (single quotes prevent shell expansion in Groovy)

6. Use credential binding instead of environment variables when possible
   withCredentials is more secure than environment block

7. Consider external secret managers for production
   HashiCorp Vault, AWS Secrets Manager, Azure Key Vault
   (Jenkins has plugins for all of these)
```

## 🐙 Git Plugin — Source Code Integration

The Git plugin is probably the **most installed** Jenkins plugin. It enables Jenkins to clone repositories, track branches, and trigger builds on code changes.

### Basic Git Configuration in Pipeline

```groovy
pipeline {
    agent any
    
    stages {
        stage('Checkout') {
            steps {
                // Method 1: Simple checkout (uses the SCM configured in the job)
                checkout scm
                
                // Method 2: Explicit Git checkout
                git branch: 'main',
                    url: 'https://github.com/mycompany/my-app.git',
                    credentialsId: 'github-credentials'
                
                // Method 3: Advanced checkout with more options
                checkout([
                    $class: 'GitSCM',
                    branches: [[name: '*/main']],
                    extensions: [
                        [$class: 'CleanBeforeCheckout'],           // Clean workspace before checkout
                        [$class: 'CloneOption', 
                            depth: 1,                               // Shallow clone (faster)
                            noTags: true,                           // Don't fetch tags
                            shallow: true                           // Shallow clone
                        ],
                        [$class: 'SubmoduleOption',
                            recursiveSubmodules: true               // Checkout submodules
                        ]
                    ],
                    userRemoteConfigs: [[
                        url: 'https://github.com/mycompany/my-app.git',
                        credentialsId: 'github-credentials'
                    ]]
                ])
            }
        }
    }
}
```

### Multibranch Pipeline — Auto-Discover Branches

The Git plugin (combined with Branch Source plugins) enables **Multibranch Pipelines** — Jenkins automatically discovers all branches in a repo and creates pipeline jobs for each:

```
Repository:
  ├── main          → Jenkins creates pipeline: my-app/main
  ├── develop       → Jenkins creates pipeline: my-app/develop
  ├── feature/login → Jenkins creates pipeline: my-app/feature/login
  ├── PR #42        → Jenkins creates pipeline: my-app/PR-42
  └── PR #43        → Jenkins creates pipeline: my-app/PR-43

How to set up:
  1. New Item → Multibranch Pipeline
  2. Branch Sources → Add source → GitHub (or Git)
  3. Enter repo URL and credentials
  4. Jenkins scans the repo and creates jobs for each branch that has a Jenkinsfile
```

### Webhooks — Triggering Builds on Push

Instead of Jenkins polling Git for changes (wasteful), configure webhooks so Git notifies Jenkins:

**GitHub Webhook Setup:**

```
1. In GitHub repo → Settings → Webhooks → Add webhook
   Payload URL: https://your-jenkins.com/github-webhook/
   Content type: application/json
   Secret:       (optional, but recommended)
   Events:       Push events, Pull request events
   
2. In Jenkins pipeline:
```

```groovy
pipeline {
    agent any
    
    triggers {
        // GitHub webhook trigger
        githubPush()
        
        // OR: Poll SCM as fallback (checks every 5 minutes)
        pollSCM('H/5 * * * *')
    }
    
    stages {
        stage('Build') {
            steps {
                checkout scm
                sh 'make build'
            }
        }
    }
}
```

When Jenkins checks out a Git repo, it sets several useful environment variables:

## 🐳 Docker Plugin & Docker Pipeline Plugin

There are actually **two** Docker-related plugins you need to know:

```
1. Docker Plugin          → Use Docker containers as Jenkins agents
2. Docker Pipeline Plugin → Use Docker in pipeline steps (build, push, run)
```

### Docker Pipeline Plugin — Building & Pushing Images

```groovy
pipeline {
    agent any
    
    environment {
        DOCKER_REGISTRY = 'myregistry.azurecr.io'
        IMAGE_NAME = 'my-app'
        IMAGE_TAG = "${env.BUILD_NUMBER}"
    }
    
    stages {
        stage('Build Image') {
            steps {
                script {
                    // Build a Docker image
                    def appImage = docker.build("${DOCKER_REGISTRY}/${IMAGE_NAME}:${IMAGE_TAG}")
                    
                    // Build with custom Dockerfile and context
                    def appImage2 = docker.build(
                        "${DOCKER_REGISTRY}/${IMAGE_NAME}:${IMAGE_TAG}",
                        "-f Dockerfile.prod --build-arg ENV=production ."
                    )
                }
            }
        }
        
        stage('Test Image') {
            steps {
                script {
                    // Run tests inside the built image
                    docker.image("${DOCKER_REGISTRY}/${IMAGE_NAME}:${IMAGE_TAG}").inside {
                        sh 'npm test'
                    }
                }
            }
        }
        
        stage('Push Image') {
            steps {
                script {
                    // Push to a Docker registry
                    docker.withRegistry("https://${DOCKER_REGISTRY}", 'docker-credentials') {
                        def appImage = docker.build("${DOCKER_REGISTRY}/${IMAGE_NAME}:${IMAGE_TAG}")
                        appImage.push()                    // Push with build number tag
                        appImage.push('latest')            // Also push as latest
                    }
                }
            }
        }
    }
}
```

### Docker Plugin — Containers as Agents

This plugin lets Jenkins automatically provision Docker containers as build agents:

```groovy
pipeline {
    agent {
        docker {
            image 'node:18-alpine'
            label 'docker-host'     // Run on a host that has Docker
            args '-v /tmp:/tmp'     // Mount volumes
        }
    }
    
    stages {
        stage('Build') {
            steps {
                sh 'node --version'
                sh 'npm install'
                sh 'npm run build'
            }
        }
    }
}
```

### Docker Compose in Pipelines

For integration tests that need multiple services:

```groovy
pipeline {
    agent { label 'docker' }
    
    stages {
        stage('Start Services') {
            steps {
                sh 'docker-compose -f docker-compose.test.yml up -d'
                // Wait for services to be healthy
                sh 'sleep 10'
            }
        }
        
        stage('Run Integration Tests') {
            steps {
                sh 'npm run test:integration'
            }
        }
    }
    
    post {
        always {
            // Always clean up containers
            sh 'docker-compose -f docker-compose.test.yml down -v'
        }
    }
}
```

## ☸️ Kubernetes Plugin — Pods as Agents

The Kubernetes plugin is covered in detail in **11.8**, but here's a quick reference of the plugin's features:

### Key Features

```
┌─────────────────────────────────────────────────────┐
│  Kubernetes Plugin Features                          │
├─────────────────────────────────────────────────────┤
│  ✅ Create Pods on-demand as Jenkins agents           │
│  ✅ Multi-container Pods (sidecar pattern)            │
│  ✅ Pod templates (reusable agent definitions)        │
│  ✅ Resource requests/limits for each container       │
│  ✅ Volume mounts (PVC, ConfigMap, Secret)            │
│  ✅ Service account assignment                        │
│  ✅ Node selector / affinity / tolerations           │
│  ✅ Auto-cleanup after build finishes                 │
│  ✅ JNLP-based agent connection                      │
│  ✅ Inherited Pod templates (base + override)         │
└─────────────────────────────────────────────────────┘
```

### Quick Usage Example

```groovy
pipeline {
    agent {
        kubernetes {
            yaml '''
apiVersion: v1
kind: Pod
spec:
  containers:
    - name: node
      image: node:18
      command: ["sleep"]
      args: ["infinity"]
    - name: docker
      image: docker:24-dind
      securityContext:
        privileged: true
'''
        }
    }
    
    stages {
        stage('Build') {
            steps {
                container('node') {
                    sh 'npm install && npm run build'
                }
            }
        }
        stage('Dockerize') {
            steps {
                container('docker') {
                    sh 'docker build -t my-app .'
                }
            }
        }
    }
}
```

## 🔌 Other Essential Plugins — Quick Reference

Here's a reference of the most commonly used Jenkins plugins beyond the big five:

### Build & Pipeline Plugins

| Plugin | What It Does | Example Usage |
|---|---|---|
| **Pipeline** (`workflow-aggregator`) | Enables pipeline-as-code (Jenkinsfile) | Required for all pipeline jobs |
| **Pipeline Stage View** | Shows stages visually in classic UI | Visual pipeline progress |
| **Timestamper** | Adds timestamps to console output | `options { timestamps() }` |
| **AnsiColor** | Shows colored terminal output in Jenkins | `options { ansiColor('xterm') }` |
| **Build Timeout** | Auto-abort jobs that run too long | `options { timeout(time: 30, unit: 'MINUTES') }` |
| **Throttle Concurrent Builds** | Limit how many builds run in parallel | Prevent resource overload |

**Using Timestamper and AnsiColor:**

```groovy
pipeline {
    agent any
    
    options {
        timestamps()                          // Add timestamps to every line
        ansiColor('xterm')                    // Enable colored output
        timeout(time: 30, unit: 'MINUTES')    // Kill job after 30 min
        buildDiscarder(logRotator(numToKeepStr: '10'))  // Keep last 10 builds
        disableConcurrentBuilds()             // Only one build at a time
    }
    
    stages {
        stage('Build') {
            steps {
                // Colored output works now!
                sh 'echo -e "\\033[32mBuild successful!\\033[0m"'
            }
        }
    }
}
```

## 🛡️ Plugin Security Best Practices

```
1. Keep plugins updated
   Outdated plugins = security vulnerabilities
   Check Manage Jenkins → Plugins → Updates regularly

2. Only install plugins you need
   Each plugin increases attack surface
   Unused plugins waste resources and slow Jenkins down

3. Review plugin compatibility
   Check the plugin's compatibility with your Jenkins version
   https://plugins.jenkins.io/

4. Watch for plugin deprecation
   Some plugins are abandoned — look for active maintenance

5. Pin plugin versions in Docker
   plugins.txt: git:5.2.0 (not git:latest in production)

6. Security advisories
   Jenkins publishes security advisories for plugin vulnerabilities
   https://www.jenkins.io/security/advisories/
   Subscribe to the jenkins-security mailing list

7. Minimize admin-level plugins
   Some plugins require admin permissions — review what access they need
```

## 🐛 Troubleshooting Plugins

### Common Plugin Problems

```
Problem: "Plugin X requires Plugin Y version 2.0, but you have 1.5"
Solution: Update the dependency plugin first, then install/update Plugin X

Problem: "Jenkins fails to start after installing a plugin"
Solution: 
  1. Go to $JENKINS_HOME/plugins/
  2. Delete the problematic plugin's .jpi file
  3. Restart Jenkins
  # On Linux:
  rm /var/jenkins_home/plugins/bad-plugin.jpi
  systemctl restart jenkins

Problem: "Plugin is installed but not working"
Solution:
  1. Check if Jenkins needs a restart
  2. Check Manage Jenkins → System Log for errors
  3. Verify plugin configuration (Manage Jenkins → System)

Problem: "Too many plugins — Jenkins is slow"
Solution:
  1. Go to Manage Jenkins → Plugins → Installed
  2. Identify unused plugins
  3. Uninstall them (check dependencies first!)
  4. Restart Jenkins
```

### Checking Plugin Health

```
Manage Jenkins → System Information
  → Shows all installed plugins and versions

Manage Jenkins → System Log
  → Shows plugin errors and warnings

Manage Jenkins → About Jenkins
  → Shows plugin dependencies and potential issues
```
