# 12.10 — Terraform — Import Existing Infrastructure

> **One Line Answer:** `terraform import` lets you **bring your already-running cloud resources under Terraform's management** — so you can start using code to manage infrastructure that was originally created by hand (via console, CLI, or other tools).

## 🤔 Why Would You Need to Import Infrastructure?

Let's paint a picture.

You joined a company. They've been running on AWS for 3 years. They have:

- 50 EC2 instances
- 20 S3 buckets
- 10 RDS databases
- VPCs, security groups, IAM roles...

**But none of it was created with Terraform.** It was all created manually — by clicking buttons in the AWS Console.

Now the team says: *"We want to use Terraform for everything going forward."*

So what do you do? You have two options:

```
Option 1: Delete everything and recreate with Terraform
    → 😱 TERRIBLE IDEA — you'd have downtime, data loss, and angry users

Option 2: Import existing resources into Terraform
    → ✅ SMART — bring the existing stuff under Terraform's control
         without touching or destroying anything
```

**That's exactly what `terraform import` does.**

---

## 🧠 What Does "Import" Actually Mean?

When you run `terraform import`, Terraform does **one very specific thing**:

> It adds the real-world resource into your **Terraform state file** (`terraform.tfstate`).

That's it. It does NOT:

- ❌ Automatically write HCL code for you (in older versions)
- ❌ Create the `.tf` configuration files
- ❌ Modify the actual cloud resource

Think of it like this:

```
Before Import:
    Terraform: "I don't know about that EC2 instance. It doesn't exist to me."
    AWS: "That EC2 instance has been running for 2 years."

After Import:
    Terraform: "Oh! Now I know about that EC2 instance. It's in my state file."
    AWS: "Nothing changed on my side. The instance is the same."
```

So after importing, Terraform is **aware** of the resource, but you still need to write the matching HCL code so Terraform can **manage** it going forward.

## 📋 The Complete Import Workflow (Step by Step)

Here's the full process to import a resource. Let's use an **AWS S3 bucket** as our example.

### Step 1: Identify the Resource in the Cloud

First, find the resource you want to import. You need its **unique identifier**.

Every resource type has a different identifier:

| Resource Type | Identifier Example |
|---|---|
| AWS EC2 Instance | `i-0abcdef1234567890` |
| AWS S3 Bucket | `my-company-data-bucket` |
| AWS VPC | `vpc-0abc123def456` |
| AWS Security Group | `sg-0abc123def456` |
| AWS IAM User | `john.doe` |
| AWS RDS Instance | `my-database` |
| Azure Resource Group | `/subscriptions/xxx/resourceGroups/my-rg` |
| GCP Compute Instance | `projects/my-proj/zones/us-central1-a/instances/my-vm` |

You can find these identifiers in:
- The cloud provider's web console
- Using CLI commands like `aws ec2 describe-instances`
- Using the cloud provider's API

### Step 2: Write the Terraform Configuration (Empty Shell)

Before importing, you must write a **matching resource block** in your `.tf` file. Even if it's mostly empty.

```hcl
# main.tf

# Step 2: Write the resource block for the S3 bucket you want to import
resource "aws_s3_bucket" "my_data_bucket" {
  # We'll fill this in after import
}
```

The key things that MUST match:
- `aws_s3_bucket` → the resource **type** must match the actual resource
- `my_data_bucket` → this is the Terraform **name** (you can choose anything)

### Step 3: Run `terraform import`

Now connect the real-world resource to the Terraform resource block:

```bash
terraform import aws_s3_bucket.my_data_bucket my-company-data-bucket
```

Let's break this command down:

```
terraform import <RESOURCE_TYPE>.<RESOURCE_NAME> <REAL_WORLD_ID>
                 ↑                                ↑
                 This matches your .tf file        This is the actual
                 (aws_s3_bucket.my_data_bucket)    cloud resource ID
```

**Output:**

```
aws_s3_bucket.my_data_bucket: Importing from ID "my-company-data-bucket"...
aws_s3_bucket.my_data_bucket: Import prepared!
  Prepared aws_s3_bucket for import
aws_s3_bucket.my_data_bucket: Refreshing state... [id=my-company-data-bucket]

Import successful!

The resources that were imported are shown above. These resources are now in
your Terraform state and will henceforth be managed by Terraform.
```

### Step 4: Check What Terraform Knows

After importing, Terraform's state file now has all the details about that S3 bucket. You can see them:

```bash
terraform state show aws_s3_bucket.my_data_bucket
```

**Output:**

```hcl
# aws_s3_bucket.my_data_bucket:
resource "aws_s3_bucket" "my_data_bucket" {
    arn                         = "arn:aws:s3:::my-company-data-bucket"
    bucket                      = "my-company-data-bucket"
    bucket_domain_name          = "my-company-data-bucket.s3.amazonaws.com"
    hosted_zone_id              = "Z3AQBSTGFYJSTF"
    id                          = "my-company-data-bucket"
    region                      = "us-east-1"
    request_payer               = "BucketOwner"

    versioning {
        enabled = true
    }

    server_side_encryption_configuration {
        rule {
            apply_server_side_encryption_by_default {
                sse_algorithm = "aws:kms"
            }
        }
    }
}
```

### Step 5: Update Your `.tf` File to Match Reality

Now copy the attributes from the state into your `.tf` file so it matches the real resource:

```hcl
# main.tf — Updated to match the actual S3 bucket

resource "aws_s3_bucket" "my_data_bucket" {
  bucket = "my-company-data-bucket"
}

resource "aws_s3_bucket_versioning" "my_data_bucket" {
  bucket = aws_s3_bucket.my_data_bucket.id

  versioning_configuration {
    status = "Enabled"
  }
}

resource "aws_s3_bucket_server_side_encryption_configuration" "my_data_bucket" {
  bucket = aws_s3_bucket.my_data_bucket.id

  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "aws:kms"
    }
  }
}
```

### Step 6: Run `terraform plan` — Verify No Changes

This is the **critical** step. Run a plan to make sure Terraform agrees that your `.tf` file matches reality:

```bash
terraform plan
```

**The goal is to see:**

```
No changes. Your infrastructure matches the configuration.

Terraform has compared your real infrastructure against your configuration
and found no differences, so no changes are needed.
```

If you see **any changes**, it means your `.tf` file doesn't perfectly match the real resource. Fix the `.tf` file until the plan shows zero changes.

Common things you might miss:
- Tags
- Encryption settings
- Lifecycle rules
- Specific config options that were set via console

### Step 7: You're Done! 🎉

Now Terraform fully manages this resource. Any future changes should be made through Terraform code.

## 🆕 The `import` Block (Terraform 1.5+ — The Modern Way)

Starting from **Terraform 1.5**, there's a much better way to import resources — using an `import` block directly in your HCL code!

### Why is This Better?

The old `terraform import` CLI command had problems:

| Old Way (CLI) | New Way (import block) |
|---|---|
| Run from command line | Written in HCL code |
| Not version controlled | Version controlled (it's in your `.tf` files!) |
| Can't be reviewed in a PR | Can be reviewed in a PR |
| One resource at a time | Import many resources at once |
| Doesn't generate config | Can auto-generate config (Terraform 1.5+) |

### How to Use the `import` Block

```hcl
# imports.tf

import {
  to = aws_s3_bucket.my_data_bucket
  id = "my-company-data-bucket"
}

import {
  to = aws_instance.web_server
  id = "i-0abcdef1234567890"
}

import {
  to = aws_security_group.web_sg
  id = "sg-0abc123def456"
}
```

Then just write the matching resource blocks and run:

```bash
terraform plan
```

Terraform will show you that it wants to import these resources. Then:

```bash
terraform apply
```

It imports them. After that, you can **remove the import blocks** — they're only needed once.

## ✨ Auto-Generating Configuration (Terraform 1.5+)

This is the game-changer. You can tell Terraform to **generate the HCL code for you** during import!

### Step 1: Write the Import Block

```hcl
# imports.tf

import {
  to = aws_s3_bucket.my_data_bucket
  id = "my-company-data-bucket"
}
```

### Step 2: Run Plan with `-generate-config-out`

```bash
terraform plan -generate-config-out=generated.tf
```

### What Happens:

Terraform will:
1. Connect to AWS
2. Read all the attributes of the S3 bucket
3. Write a complete `resource` block into `generated.tf`

**generated.tf:**

```hcl
# __generated__ by Terraform
# Please review these resources and move them into your main configuration files.

resource "aws_s3_bucket" "my_data_bucket" {
  bucket              = "my-company-data-bucket"
  object_lock_enabled = false

  tags = {
    Environment = "production"
    Team        = "data-engineering"
  }
}
```

### Step 3: Review, Clean Up, and Apply

- Review the generated code
- Move it to the appropriate `.tf` file
- Remove any computed/read-only attributes that Terraform adds unnecessarily
- Run `terraform plan` to verify zero changes
- Run `terraform apply` to complete the import

## 📦 Importing Multiple Resources — Real-World Example

Let's say you want to import an **EC2 instance along with its security group and key pair**:

### Step 1: Write Import Blocks

```hcl
# imports.tf

import {
  to = aws_instance.web_server
  id = "i-0abcdef1234567890"
}

import {
  to = aws_security_group.web_sg
  id = "sg-0abc123def456"
}

import {
  to = aws_key_pair.my_key
  id = "my-ssh-key"
}
```

### Step 2: Write Resource Blocks

```hcl
# ec2.tf

resource "aws_instance" "web_server" {
  ami                    = "ami-0c55b159cbfafe1f0"
  instance_type          = "t3.medium"
  key_name               = aws_key_pair.my_key.key_name
  vpc_security_group_ids = [aws_security_group.web_sg.id]
  subnet_id              = "subnet-0abc123"

  root_block_device {
    volume_size = 50
    volume_type = "gp3"
  }

  tags = {
    Name        = "web-server-prod"
    Environment = "production"
  }
}

# sg.tf

resource "aws_security_group" "web_sg" {
  name        = "web-server-sg"
  description = "Security group for web server"
  vpc_id      = "vpc-0abc123"

  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  ingress {
    from_port   = 443
    to_port     = 443
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  ingress {
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["10.0.0.0/8"]   # SSH only from internal network
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }

  tags = {
    Name = "web-server-sg"
  }
}

# keypair.tf

resource "aws_key_pair" "my_key" {
  key_name   = "my-ssh-key"
  public_key = "ssh-rsa AAAAB3NzaC1yc2EAAAA..."  # Your actual public key
}
```

### Step 3: Apply

```bash
terraform apply
```

Terraform imports all three resources in one go.

## ⚠️ Common Pitfalls and How to Avoid Them

### Pitfall 1: Forgetting to Write the Resource Block First

```bash
$ terraform import aws_s3_bucket.my_bucket my-bucket-name

Error: resource address "aws_s3_bucket.my_bucket" does not exist 
in the configuration.
```

**Fix:** Always write the resource block in your `.tf` file before running import.

### Pitfall 2: Plan Shows Changes After Import

```bash
$ terraform plan

~ resource "aws_instance" "web_server" {
    ~ tags = {
        - "CreatedBy" = "manual" -> null     # Tag will be REMOVED!
      }
  }
```

**Why:** Your `.tf` file is missing some attributes (like tags) that exist on the real resource.

**Fix:** Add the missing attributes to your `.tf` file until `terraform plan` shows no changes.

### Pitfall 3: Importing into the Wrong State

If you're using **workspaces** or **remote state**, make sure you're importing into the correct one:

```bash
# Check which workspace you're in
terraform workspace show

# Switch if needed
terraform workspace select production

# Now import
terraform import aws_s3_bucket.data my-prod-bucket
```

### Pitfall 4: Dependent Resources

Some resources depend on others. For example, an EC2 instance depends on a security group. You should import them in order:

```
1. Import the VPC first
2. Import the subnet
3. Import the security group
4. Import the EC2 instance
```

If you import things out of order, you might get reference errors.

### Pitfall 5: Resources That Can't Be Imported

Not every resource supports import! Check the [Terraform documentation](https://registry.terraform.io/providers/hashicorp/aws/latest/docs) for each resource — there's an "Import" section at the bottom that tells you the ID format and whether import is supported.

## 🛠️ Useful Commands During Import

```bash
# See what's in your state file
terraform state list

# See details of a specific resource in state
terraform state show aws_s3_bucket.my_data_bucket

# Remove a resource from state (if you imported it by mistake)
# This does NOT delete the actual cloud resource!
terraform state rm aws_s3_bucket.my_data_bucket

# Move a resource in state (if you rename it in your code)
terraform state mv aws_s3_bucket.old_name aws_s3_bucket.new_name
```

## 🧰 Tools That Make Importing Easier

Manually importing hundreds of resources is painful. These tools help:

### 1. Terraformer (by Google)

Automatically generates Terraform code AND state for existing infrastructure.

```bash
# Install
brew install terraformer

# Import all EC2 instances from AWS
terraformer import aws --resources=ec2_instance --regions=us-east-1

# Import everything from AWS
terraformer import aws --resources=* --regions=us-east-1
```

It creates both the `.tf` files AND the state file — saving you tons of manual work.

### 2. Former2

A web-based tool that generates IaC code (Terraform, CloudFormation, CDK) from your existing AWS resources. It scans your AWS account and creates the code.

### 3. Azure Terrafy (aztfexport)

For Azure users — scans your Azure resource groups and generates Terraform code:

```bash
# Install
brew install aztfexport

# Export a resource group
aztfexport resource-group my-resource-group
```

### 4. `gcloud` Export (GCP)

Google Cloud has built-in export functionality:

```bash
gcloud beta resource-config bulk-export --resource-types=ComputeInstance
```

## 📋 Import Cheat Sheet

| Task | Command |
|------|---------|
| Import a single resource (CLI) | `terraform import <type>.<name> <id>` |
| Import using HCL block (1.5+) | `import { to = <type>.<name>  id = "<id>" }` |
| Generate config during import | `terraform plan -generate-config-out=generated.tf` |
| View imported resource in state | `terraform state show <type>.<name>` |
| List all resources in state | `terraform state list` |
| Remove from state (undo import) | `terraform state rm <type>.<name>` |
| Verify import is correct | `terraform plan` (should show no changes) |

## 🎯 Best Practices for Importing

1. **Import in small batches** — Don't try to import 200 resources at once. Do 5-10 at a time.
2. **Always run `terraform plan` after import** — Make sure it shows zero changes.
3. **Use the `import` block** (Terraform 1.5+) instead of the CLI command — it's reviewable and repeatable.
4. **Use `-generate-config-out`** — Let Terraform write the code for you.
5. **Tag everything during import** — Add consistent tags so you can track what was imported.
6. **Document what you imported** — Keep a log of imported resources for your team.
7. **Test with non-production first** — Practice importing in a dev/staging environment before touching production.
8. **Use version control** — Commit your `.tf` files after each successful import batch.

## 🧪 Quick Hands-On Exercise

Want to practice? Try this (requires AWS account):

1. **Manually create** an S3 bucket in the AWS Console called `test-terraform-import-<yourname>`
2. **Create** a `main.tf`:
   ```hcl
   provider "aws" {
     region = "us-east-1"
   }

   resource "aws_s3_bucket" "test" {}
   ```
3. **Run** `terraform init`
4. **Import** the bucket:
   ```bash
   terraform import aws_s3_bucket.test test-terraform-import-<yourname>
   ```
5. **View** the state: `terraform state show aws_s3_bucket.test`
6. **Update** `main.tf` with the actual bucket name
7. **Verify** with `terraform plan` — you should see "No changes"
8. **Clean up**: `terraform destroy` (or just delete the bucket manually and run `terraform state rm`)

## 🏁 Key Takeaways

| Point | Detail |
|-------|--------|
| **What** | `terraform import` brings existing cloud resources into Terraform state |
| **Why** | To manage hand-created infrastructure with Terraform without recreating it |
