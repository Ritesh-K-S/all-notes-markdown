## рҹӨ” Why Do Best Practices Matter?

Let's see what happens WITHOUT best practices:

```
your-terraform-project/
в”Ӯ
в”ңв”Җв”Җ main.tf            вҶҗ 2,500 lines long, contains EVERYTHING
в”ңв”Җв”Җ terraform.tfvars   вҶҗ hardcoded passwords (committed to Git рҹҳұ)
в””в”Җв”Җ README.md          вҶҗ says "TODO: add docs" (written 2 years ago)
```

What goes wrong:

- Nobody can find anything in the 2,500-line `main.tf`
- Merge conflicts happen on every pull request (everyone edits the same file)
- No idea which resources belong to which environment
- Cloud bill is a mystery вҖ” who created what? Why?
- New team members take weeks to understand the code
- One wrong `terraform apply` can destroy production

**Best practices prevent all of this.** Let's go through them one by one.

## рҹ“Ғ Part 1: Directory Structure

### The Simplest Structure (Small Projects / Learning)

If you're just learning or have a tiny project, this is fine:

```
my-project/
в”ңв”Җв”Җ main.tf           вҶҗ Resources
в”ңв”Җв”Җ variables.tf      вҶҗ Input variables
в”ңв”Җв”Җ outputs.tf        вҶҗ Output values
в”ңв”Җв”Җ providers.tf      вҶҗ Provider configuration
в”ңв”Җв”Җ terraform.tfvars  вҶҗ Variable values
в””в”Җв”Җ versions.tf       вҶҗ Required provider versions
```

Each file has a clear purpose:

| File | What Goes In It |
|------|-----------------|
| `main.tf` | Your actual resources (EC2, S3, VPC, etc.) |
| `variables.tf` | All `variable` blocks (inputs to your config) |
| `outputs.tf` | All `output` blocks (values you want to see after apply) |
| `providers.tf` | `provider` blocks (AWS, Azure, GCP config) |
| `terraform.tfvars` | Actual values for your variables |
| `versions.tf` | `terraform {}` block with required providers and versions |

**Example `versions.tf`:**

```hcl
terraform {
  required_version = ">= 1.5.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}
```

**Example `providers.tf`:**

```hcl
provider "aws" {
  region = var.aws_region

  default_tags {
    tags = {
      ManagedBy   = "terraform"
      Project     = var.project_name
      Environment = var.environment
    }
  }
}
```

**Example `variables.tf`:**

```hcl
variable "aws_region" {
  description = "AWS region to deploy resources"
  type        = string
  default     = "us-east-1"
}

variable "environment" {
  description = "Environment name (dev, staging, prod)"
  type        = string
}

variable "project_name" {
  description = "Name of the project"
  type        = string
}
```

### The Resource-Grouped Structure (Medium Projects)

When your project grows, split resources into **separate files by resource type or logical group**:

```
my-project/
в”ңв”Җв”Җ main.tf              вҶҗ Can be empty or contain locals
в”ңв”Җв”Җ variables.tf
в”ңв”Җв”Җ outputs.tf
в”ңв”Җв”Җ providers.tf
в”ңв”Җв”Җ versions.tf
в”ңв”Җв”Җ terraform.tfvars
в”Ӯ
в”ңв”Җв”Җ vpc.tf               вҶҗ VPC, subnets, route tables, IGW
в”ңв”Җв”Җ security-groups.tf   вҶҗ All security groups
в”ңв”Җв”Җ ec2.tf               вҶҗ EC2 instances, AMI lookups
в”ңв”Җв”Җ rds.tf               вҶҗ RDS databases
в”ңв”Җв”Җ s3.tf                вҶҗ S3 buckets
в”ңв”Җв”Җ iam.tf               вҶҗ IAM roles, policies, users
в”ңв”Җв”Җ alb.tf               вҶҗ Application Load Balancer
в”ңв”Җв”Җ route53.tf           вҶҗ DNS records
в””в”Җв”Җ cloudwatch.tf        вҶҗ Monitoring, alarms
```

**Why this is great:**
- Each file is small and focused
- Easy to find what you're looking for
- Fewer merge conflicts (different people edit different files)
- Code reviews are cleaner

**Rule of thumb:** If a `.tf` file is longer than **200 lines**, split it.

### The Environment-Based Structure (Real-World Projects)

This is the **most common structure** in production. You separate environments (dev, staging, prod) while sharing modules:

```
terraform/
в”Ӯ
в”ңв”Җв”Җ modules/                    вҶҗ Reusable modules (shared code)
в”Ӯ   в”ңв”Җв”Җ vpc/
в”Ӯ   в”Ӯ   в”ңв”Җв”Җ main.tf
в”Ӯ   в”Ӯ   в”ңв”Җв”Җ variables.tf
в”Ӯ   в”Ӯ   в””в”Җв”Җ outputs.tf
в”Ӯ   в”ңв”Җв”Җ ec2/
в”Ӯ   в”Ӯ   в”ңв”Җв”Җ main.tf
в”Ӯ   в”Ӯ   в”ңв”Җв”Җ variables.tf
в”Ӯ   в”Ӯ   в””в”Җв”Җ outputs.tf
в”Ӯ   в”ңв”Җв”Җ rds/
в”Ӯ   в”Ӯ   в”ңв”Җв”Җ main.tf
в”Ӯ   в”Ӯ   в”ңв”Җв”Җ variables.tf
в”Ӯ   в”Ӯ   в””в”Җв”Җ outputs.tf
в”Ӯ   в””в”Җв”Җ s3/
в”Ӯ       в”ңв”Җв”Җ main.tf
в”Ӯ       в”ңв”Җв”Җ variables.tf
в”Ӯ       в””в”Җв”Җ outputs.tf
в”Ӯ
в”ңв”Җв”Җ environments/               вҶҗ Environment-specific configs
в”Ӯ   в”ңв”Җв”Җ dev/
в”Ӯ   в”Ӯ   в”ңв”Җв”Җ main.tf            вҶҗ Calls modules with dev values
в”Ӯ   в”Ӯ   в”ңв”Җв”Җ variables.tf
в”Ӯ   в”Ӯ   в”ңв”Җв”Җ outputs.tf
в”Ӯ   в”Ӯ   в”ңв”Җв”Җ providers.tf
в”Ӯ   в”Ӯ   в”ңв”Җв”Җ backend.tf         вҶҗ Remote state config for dev
в”Ӯ   в”Ӯ   в””в”Җв”Җ terraform.tfvars
в”Ӯ   в”Ӯ
в”Ӯ   в”ңв”Җв”Җ staging/
в”Ӯ   в”Ӯ   в”ңв”Җв”Җ main.tf            вҶҗ Calls modules with staging values
в”Ӯ   в”Ӯ   в”ңв”Җв”Җ variables.tf
в”Ӯ   в”Ӯ   в”ңв”Җв”Җ outputs.tf
в”Ӯ   в”Ӯ   в”ңв”Җв”Җ providers.tf
в”Ӯ   в”Ӯ   в”ңв”Җв”Җ backend.tf
в”Ӯ   в”Ӯ   в””в”Җв”Җ terraform.tfvars
в”Ӯ   в”Ӯ
в”Ӯ   в””в”Җв”Җ prod/
в”Ӯ       в”ңв”Җв”Җ main.tf            вҶҗ Calls modules with prod values
в”Ӯ       в”ңв”Җв”Җ variables.tf
в”Ӯ       в”ңв”Җв”Җ outputs.tf
в”Ӯ       в”ңв”Җв”Җ providers.tf
в”Ӯ       в”ңв”Җв”Җ backend.tf
в”Ӯ       в””в”Җв”Җ terraform.tfvars
в”Ӯ
в””в”Җв”Җ README.md
```

**How it works:**

Each environment folder calls the **same modules** but with **different values**:

```hcl
# environments/dev/main.tf

module "vpc" {
  source = "../../modules/vpc"

  vpc_cidr     = "10.0.0.0/16"
  environment  = "dev"
  project_name = "myapp"
}

module "ec2" {
  source = "../../modules/ec2"

  instance_type = "t3.small"      # Small for dev
  vpc_id        = module.vpc.vpc_id
  subnet_id     = module.vpc.public_subnet_ids[0]
  environment   = "dev"
}
```

```hcl
# environments/prod/main.tf

module "vpc" {
  source = "../../modules/vpc"

  vpc_cidr     = "10.1.0.0/16"    # Different CIDR for prod
  environment  = "prod"
  project_name = "myapp"
}

module "ec2" {
  source = "../../modules/ec2"

  instance_type = "m5.xlarge"     # Bigger for prod!
  vpc_id        = module.vpc.vpc_id
  subnet_id     = module.vpc.public_subnet_ids[0]
  environment   = "prod"
}
```

**Benefits:**
- Same infrastructure code, different sizes/configs per environment
- Can't accidentally apply dev changes to prod (separate state files)
- Each environment can evolve independently
- Modules are tested once, used everywhere

### What NOT to Do (Anti-Patterns)

```
вқҢ BAD: Everything in one file
    в””в”Җв”Җ main.tf  (3000 lines)

вқҢ BAD: Environment values hardcoded
    resource "aws_instance" "web" {
      instance_type = "t3.large"  # What environment is this?
    }

вқҢ BAD: Using workspaces for environments
    # Workspaces share the same code AND state backend
    # One mistake can affect all environments

вқҢ BAD: No backend configuration
    # State file stored locally = disaster waiting to happen

вқҢ BAD: Secrets in terraform.tfvars committed to Git
    db_password = "SuperSecret123!"  # рҹҳұ
```

## рҹҸ·пёҸ Part 2: Naming Conventions

Consistent naming is like keeping a tidy kitchen вҖ” everything is where you expect it to be.

### Resource Names in Terraform Code

**Use snake_case** for Terraform resource names (the logical name in your code):

```hcl
# вң… GOOD вҖ” snake_case
resource "aws_instance" "web_server" {}
resource "aws_s3_bucket" "app_data" {}
resource "aws_security_group" "database_sg" {}

# вқҢ BAD вҖ” camelCase
resource "aws_instance" "webServer" {}

# вқҢ BAD вҖ” PascalCase
resource "aws_instance" "WebServer" {}

# вқҢ BAD вҖ” kebab-case (not allowed in Terraform)
resource "aws_instance" "web-server" {}

# вқҢ BAD вҖ” vague names
resource "aws_instance" "main" {}
resource "aws_instance" "this" {}
resource "aws_s3_bucket" "bucket" {}
```

### Cloud Resource Names (The Actual Name in AWS/Azure/GCP)

For the actual names that appear in your cloud console, use a **consistent naming pattern**:

```
{project}-{environment}-{resource_type}-{description}
```

**Examples:**

```hcl
resource "aws_instance" "web_server" {
  tags = {
    Name = "myapp-prod-ec2-web-server"
    #       вҶ‘       вҶ‘     вҶ‘    вҶ‘
    #    project   env   type  description
  }
}

resource "aws_s3_bucket" "app_logs" {
  bucket = "myapp-prod-s3-application-logs"
}

resource "aws_security_group" "database" {
  name = "myapp-prod-sg-database"
}

resource "aws_db_instance" "main" {
  identifier = "myapp-prod-rds-postgres"
}
```

### Using Locals for Consistent Naming

Don't repeat the naming pattern everywhere вҖ” use `locals`:

```hcl
locals {
  name_prefix = "${var.project_name}-${var.environment}"
}

resource "aws_instance" "web_server" {
  tags = {
    Name = "${local.name_prefix}-ec2-web-server"
  }
}

resource "aws_s3_bucket" "logs" {
  bucket = "${local.name_prefix}-s3-logs"
}

resource "aws_security_group" "web" {
  name = "${local.name_prefix}-sg-web"
}
```

Now if `project_name = "myapp"` and `environment = "prod"`, all resources will be named like `myapp-prod-*`.

### Variable Naming

```hcl
# вң… GOOD вҖ” descriptive, snake_case
variable "instance_type" {}
variable "vpc_cidr_block" {}
variable "database_password" {}
variable "enable_monitoring" {}
variable "max_instance_count" {}

# вқҢ BAD вҖ” too short, ambiguous
variable "type" {}
variable "cidr" {}
variable "pass" {}
variable "flag" {}
variable "count" {}   # 'count' is a reserved word in Terraform!
```

### Module Naming

```hcl
# вң… GOOD вҖ” clear purpose
module "vpc" { ... }
module "web_servers" { ... }
module "database" { ... }
module "monitoring" { ... }

# вқҢ BAD вҖ” vague
module "main" { ... }
module "infra" { ... }
module "stuff" { ... }
```

### Output Naming

Use a pattern: `{resource}_{attribute}`

```hcl
# вң… GOOD
output "vpc_id" {
  value = aws_vpc.main.id
}

output "web_server_public_ip" {
  value = aws_instance.web_server.public_ip
}

output "database_endpoint" {
  value = aws_db_instance.main.endpoint
}

# вқҢ BAD
output "id" {
  value = aws_vpc.main.id
}

output "ip" {
  value = aws_instance.web_server.public_ip
}
```

## рҹҸ·пёҸ Part 3: Tagging Strategy

Tags are **metadata labels** you attach to cloud resources. They're just key-value pairs.

### Why Tags Are Critical

```
Without tags:
    "Hey, what's this EC2 instance for?"
    "Who created it?"
    "Is it for dev or prod?"
    "Can we delete it?"
    "Who's paying for it?"
    вҶ’ Nobody knows. Welcome to cloud chaos. рҹ’ё

With tags:
    Name        = "myapp-prod-ec2-web-server"
    Environment = "production"
    Team        = "platform-engineering"
    Owner       = "jane.doe@company.com"
    Project     = "myapp"
    CostCenter  = "CC-1234"
    ManagedBy   = "terraform"
    вҶ’ Everything is crystal clear.
```

### Mandatory Tags (Every Resource Should Have These)

| Tag Key | Example Value | Purpose |
|---------|---------------|---------|
| `Name` | `myapp-prod-ec2-web` | Human-friendly name (shows in console) |
| `Environment` | `production` | Which environment (dev/staging/prod) |
| `Project` | `myapp` | Which project this belongs to |
| `Owner` | `team-platform@company.com` | Who's responsible |
| `ManagedBy` | `terraform` | How it was created |
| `CostCenter` | `CC-1234` | For billing/cost allocation |

### Optional but Useful Tags

| Tag Key | Example Value | Purpose |
|---------|---------------|---------|
| `Application` | `web-frontend` | Which application component |
| `Compliance` | `pci-dss` | Compliance requirements |
| `DataClassification` | `confidential` | Data sensitivity level |
| `BackupSchedule` | `daily` | Backup requirements |
| `AutoShutdown` | `true` | For cost savings (stop dev instances at night) |
| `ExpirationDate` | `2026-12-31` | Temporary resources |
| `Terraform` | `true` | Quick filter for Terraform-managed resources |
| `GitRepo` | `github.com/company/infra` | Link to source code |

### Implementing Tags in Terraform

#### Method 1: Tags on Every Resource (Basic)

```hcl
resource "aws_instance" "web_server" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.medium"

  tags = {
    Name        = "myapp-prod-ec2-web"
    Environment = "production"
    Project     = "myapp"
    Owner       = "team-platform@company.com"
    ManagedBy   = "terraform"
    CostCenter  = "CC-1234"
  }
}
```

**Problem:** You have to copy-paste these tags to EVERY resource. That's repetitive and error-prone.

#### Method 2: Using `locals` for Common Tags (Better)

```hcl
locals {
  common_tags = {
    Environment = var.environment
    Project     = var.project_name
    Owner       = var.team_email
    ManagedBy   = "terraform"
    CostCenter  = var.cost_center
  }
}

resource "aws_instance" "web_server" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.medium"

  tags = merge(local.common_tags, {
    Name = "${local.name_prefix}-ec2-web"
    Role = "web-server"
  })
}

resource "aws_s3_bucket" "logs" {
  bucket = "${local.name_prefix}-s3-logs"

  tags = merge(local.common_tags, {
    Name = "${local.name_prefix}-s3-logs"
    DataClassification = "internal"
  })
}
```

The `merge()` function combines the common tags with resource-specific tags. If there's a conflict, the second map wins.

#### Method 3: Using `default_tags` in the Provider (Best!)

AWS provider supports `default_tags` вҖ” these are automatically applied to EVERY resource:

```hcl
provider "aws" {
  region = var.aws_region

  default_tags {
    tags = {
      Environment = var.environment
      Project     = var.project_name
      Owner       = var.team_email
      ManagedBy   = "terraform"
      CostCenter  = var.cost_center
    }
  }
}

# Now you only need resource-specific tags!
resource "aws_instance" "web_server" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.medium"

  tags = {
    Name = "${local.name_prefix}-ec2-web"
    Role = "web-server"
  }
  # Environment, Project, Owner, ManagedBy, CostCenter
  # are AUTOMATICALLY added by default_tags!
}
```

**This is the recommended approach for AWS.** Azure and GCP have similar features.

## рҹ“қ Part 4: Other Essential Best Practices

### 1. Always Use Remote State

Never store the state file locally. Use a remote backend:

```hcl
# backend.tf

terraform {
  backend "s3" {
    bucket         = "mycompany-terraform-state"
    key            = "myapp/prod/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-state-lock"    # State locking!
    encrypt        = true                       # Encrypt at rest!
  }
}
```

**Why:**
- State file contains sensitive data (like database passwords)
- Multiple team members need access to the same state
- DynamoDB locking prevents two people from applying at the same time
- S3 versioning lets you recover if something goes wrong

### 2. Lock Provider Versions

Always pin your provider versions:

```hcl
# versions.tf

terraform {
  required_version = ">= 1.5.0, < 2.0.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.31"     # Allow 5.31.x, 5.32.x, etc. but NOT 6.0
    }
    random = {
      source  = "hashicorp/random"
      version = "~> 3.6"
    }
  }
}
```

**Why:** Without version pinning, `terraform init` might download a new provider version that breaks your code.

Also use the `.terraform.lock.hcl` file вҖ” **commit it to Git**! It locks exact versions.

### 3. Use Variables with Validation

Don't accept just any value вҖ” validate inputs:

```hcl
variable "environment" {
  description = "Environment name"
  type        = string

  validation {
    condition     = contains(["dev", "staging", "prod"], var.environment)
    error_message = "Environment must be one of: dev, staging, prod."
  }
}

variable "instance_type" {
  description = "EC2 instance type"
  type        = string
  default     = "t3.medium"

  validation {
    condition     = can(regex("^t3\\.|^m5\\.|^c5\\.", var.instance_type))
    error_message = "Only t3, m5, and c5 instance families are allowed."
  }
}

variable "vpc_cidr" {
  description = "CIDR block for the VPC"
  type        = string

  validation {
    condition     = can(cidrhost(var.vpc_cidr, 0))
    error_message = "Must be a valid CIDR block (e.g., 10.0.0.0/16)."
  }
}
```

### 4. Never Hardcode Secrets

```hcl
# вқҢ NEVER DO THIS
resource "aws_db_instance" "main" {
  password = "MyS3cretP@ssw0rd!"   # рҹҳұ This is in Git forever
}

# вң… Use variables (pass via CLI or environment)
variable "db_password" {
  description = "Database password"
  type        = string
  sensitive   = true      # This hides it from terraform plan output
}

resource "aws_db_instance" "main" {
  password = var.db_password
}
```

Ways to pass sensitive values:

```bash
# Option 1: Environment variable
export TF_VAR_db_password="MyS3cretP@ssw0rd!"
terraform apply

# Option 2: CLI flag
terraform apply -var="db_password=MyS3cretP@ssw0rd!"

# Option 3: Use a secrets manager
data "aws_secretsmanager_secret_version" "db_password" {
  secret_id = "myapp/prod/db-password"
}

resource "aws_db_instance" "main" {
  password = data.aws_secretsmanager_secret_version.db_password.secret_string
}
```

### 5. Use `.gitignore` for Terraform

Create a proper `.gitignore`:

```gitignore
# .gitignore for Terraform projects

# Local .terraform directory
.terraform/

# State files (if using local state вҖ” you shouldn't be!)
*.tfstate
*.tfstate.*

# Crash log files
crash.log
crash.*.log

# Variable files that might contain secrets
*.tfvars
!example.tfvars      # Keep the example file

# Override files
override.tf
```

### 6. Format and Validate Your Code

Always run these before committing:

```bash
# Auto-format all .tf files (fixes indentation, alignment, etc.)
terraform fmt -recursive

# Validate syntax and configuration
terraform validate

# Check the plan before applying
terraform plan -out=plan.tfplan

# Apply the saved plan (exact same changes, no surprises)
terraform apply plan.tfplan
```

### 7. Use `terraform plan -out` Before `terraform apply`

```bash
# Save the plan to a file
terraform plan -out=plan.tfplan

# Review the plan
# Then apply EXACTLY what was planned
terraform apply plan.tfplan
```

**Why:** If you just run `terraform apply`, the plan might change between when you review it and when you confirm. With `-out`, the apply uses the exact plan you reviewed.

### 8. Use `lifecycle` Rules When Needed

```hcl
resource "aws_instance" "web_server" {
  ami           = data.aws_ami.latest.id
  instance_type = "t3.medium"

  lifecycle {
    # Don't destroy this resource before creating the replacement
    create_before_destroy = true

    # Don't ever destroy this resource (even with terraform destroy)
    # prevent_destroy = true

    # Ignore changes to tags made outside Terraform
    ignore_changes = [
      tags["LastModified"],
      tags["ModifiedBy"],
    ]
  }
}
```

## рҹ“Ӣ Complete Best Practices Checklist

### Directory Structure
- [ ] Split resources into separate files (not one giant `main.tf`)
- [ ] Separate environments into different directories
- [ ] Use modules for reusable components
- [ ] Each module has `main.tf`, `variables.tf`, `outputs.tf`

### Naming
- [ ] Use `snake_case` for Terraform resource names
- [ ] Use consistent naming pattern for cloud resource names
- [ ] Use `locals` for name prefixes
- [ ] Variable names are descriptive (not single letters)

### Tagging
- [ ] Every resource has mandatory tags (Name, Environment, Project, Owner, ManagedBy)
- [ ] Use `default_tags` in the AWS provider
- [ ] Tags are consistent across all resources
- [ ] Cost allocation tags are set for billing

### State Management
- [ ] Remote state backend configured (S3 + DynamoDB for AWS)
- [ ] State file is encrypted
- [ ] State locking is enabled
- [ ] Separate state files per environment

### Security
- [ ] No secrets in `.tf` files or `.tfvars` committed to Git
- [ ] Sensitive variables marked with `sensitive = true`
- [ ] `.gitignore` properly configured
- [ ] Use secrets manager for passwords

### Code Quality
- [ ] `terraform fmt` run before every commit
- [ ] `terraform validate` passes
- [ ] Provider versions are pinned
- [ ] `.terraform.lock.hcl` is committed to Git
- [ ] Variables have descriptions and validation rules
- [ ] `terraform plan -out` used before `terraform apply`

| What | Where |
|------|-------|

| Output values | `outputs.tf` |

| Data sources | With related resources or in `data.tf` |
| Reusable code | `modules/` directory |
| Environment configs | `environments/{env}/` directories |
