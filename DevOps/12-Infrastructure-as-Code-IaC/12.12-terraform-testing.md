## 🤔 Why Test Terraform Code?

You might think: *"Terraform already has `terraform plan` — isn't that testing enough?"*

**Not even close.** Here's why:

```
What terraform plan catches:
    ✅ Syntax errors
    ✅ Missing required arguments
    ✅ Type mismatches
    ✅ What will change (create/update/destroy)

What terraform plan DOESN'T catch:
    ❌ Security misconfigurations (S3 bucket is public!)
    ❌ Best practice violations (no encryption on RDS!)
    ❌ Naming convention violations
    ❌ Whether the infrastructure actually WORKS after deployment
    ❌ Whether your app can connect to the database
    ❌ Whether your load balancer routes traffic correctly
    ❌ Compliance violations (PCI-DSS, HIPAA, SOC2)
    ❌ Cost issues (you're about to launch 50 m5.24xlarge instances!)
```

That's where testing tools come in. Think of them as layers of protection:

```
┌──────────────────────────────────────────────┐
│  Layer 1: tflint                             │
│  → Static analysis (catches errors BEFORE    │
│    you even run terraform plan)              │
├──────────────────────────────────────────────┤
│  Layer 2: Checkov / tfsec                    │
│  → Security & compliance scanning            │
│  → "Your S3 bucket is public!" type checks   │
├──────────────────────────────────────────────┤
│  Layer 3: terraform validate + plan          │
│  → Built-in Terraform checks                 │
├──────────────────────────────────────────────┤
│  Layer 4: Terratest                          │
│  → Integration testing — actually deploys    │
│    infrastructure and verifies it works      │
└──────────────────────────────────────────────┘
```

Let's learn each one.

## 🔍 Part 1: tflint — The Terraform Linter

### What is tflint?

**tflint** is a **static analysis tool** (linter) for Terraform. It reads your `.tf` files and finds errors, warnings, and best practice violations **without running anything**.

Think of it like a spell-checker for your Terraform code.

### What Does tflint Catch?

| Category | Example |
|----------|---------|
| **Invalid resource types** | `aws_ec2_instance` (should be `aws_instance`) |
| **Invalid instance types** | `instance_type = "t3.huuuge"` (doesn't exist) |
| **Invalid AMI IDs** | `ami = "ami-doesnotexist"` |
| **Deprecated features** | Using old argument names that were renamed |
| **Naming violations** | Custom rules for your team's conventions |
| **Missing required tags** | Resources without mandatory tags |

### Installation

```bash
# macOS
brew install tflint

# Linux (Ubuntu/Debian)
curl -s https://raw.githubusercontent.com/terraform-linters/tflint/master/install_linux.sh | bash

# Windows (Chocolatey)
choco install tflint

# Docker
docker run --rm -v $(pwd):/data -t ghcr.io/terraform-linters/tflint
```

Verify it's installed:

```bash
tflint --version
# TFLint version 0.50.3
```

### Basic Usage

```bash
# Navigate to your Terraform directory
cd my-terraform-project/

# Run tflint
tflint

# Run with initialization (downloads plugins)
tflint --init
tflint
```

### Configuration File — `.tflint.hcl`

Create a `.tflint.hcl` file in your project root to configure tflint:

```hcl
# .tflint.hcl

# Tell tflint which cloud provider plugins to use
plugin "aws" {
  enabled = true
  version = "0.30.0"
  source  = "github.com/terraform-linters/tflint-ruleset-aws"
}

# You can also add Azure or GCP plugins
# plugin "azurerm" {
#   enabled = true
#   version = "0.25.1"
#   source  = "github.com/terraform-linters/tflint-ruleset-azurerm"
# }

# Built-in rules configuration
rule "terraform_naming_convention" {
  enabled = true

  # Enforce snake_case for all resources
  custom_formats = {}
}

rule "terraform_documented_variables" {
  enabled = true
  # Every variable must have a description
}

rule "terraform_documented_outputs" {
  enabled = true
  # Every output must have a description
}

rule "terraform_typed_variables" {
  enabled = true
  # Every variable must have a type
}

rule "terraform_unused_declarations" {
  enabled = true
  # Flag unused variables, data sources, and locals
}

rule "terraform_standard_module_structure" {
  enabled = true
  # Modules should have main.tf, variables.tf, outputs.tf
}
```

After creating the config, initialize tflint:

```bash
tflint --init
```

### Example: tflint in Action

Let's say you have this Terraform code with some issues:

```hcl
# main.tf (with intentional mistakes)

resource "aws_instance" "web_server" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.huuuge"         # ← This instance type doesn't exist!

  tags = {
    Name = "web-server"
  }
}

resource "aws_s3_bucket" "data" {
  bucket = "my-bucket"
  acl    = "private"                   # ← This argument is deprecated!
}

variable "unused_var" {                # ← This variable is never used!
  type = string
}

variable "region" {                    # ← Missing description!
}
```

Running tflint:

```bash
$ tflint

  on main.tf line 4:

  on main.tf line 10:
  10:   acl    = "private"

  on main.tf line 13:
```

tflint found 4 issues — all BEFORE you even ran `terraform plan`!

### tflint in CI/CD

Add tflint to your pipeline:

```yaml
# GitHub Actions example
name: Terraform Lint

on: [pull_request]

jobs:
  tflint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: terraform-linters/setup-tflint@v4
        with:
          tflint_version: latest

      - name: Init TFLint
        run: tflint --init

      - name: Run TFLint
        run: tflint --format=compact --recursive
```

## 🛡️ Part 2: Checkov — Security & Compliance Scanner

### What is Checkov?

**Checkov** is a **security scanner** for Infrastructure as Code. It checks your Terraform code against **hundreds of security best practices and compliance rules**.

It's like having a security expert review every line of your Terraform code.

### What Does Checkov Catch?

```
🔴 CRITICAL ISSUES Checkov finds:
    → S3 buckets without encryption
    → S3 buckets that are publicly accessible
    → Security groups with 0.0.0.0/0 (open to the world)
    → RDS databases without encryption
    → RDS databases publicly accessible
    → EC2 instances with public IPs in private subnets
    → IAM policies that are too permissive ("*" permissions)
    → CloudTrail not enabled
    → VPC flow logs not enabled
    → No MFA on root account
    → Unencrypted EBS volumes
    → Secrets hardcoded in code
    → And hundreds more...
```

### Installation

```bash
# Python (pip) — most common way
pip install checkov

# Homebrew (macOS)
brew install checkov

# Docker
docker run --rm -v $(pwd):/tf bridgecrew/checkov -d /tf
```

Verify installation:

```bash
checkov --version
# 3.2.x
```

### Basic Usage

```bash
# Scan the current directory
checkov -d .

# Scan a specific file
checkov -f main.tf

# Scan and output in a specific format
checkov -d . -o json
checkov -d . -o sarif        # For GitHub Security tab
checkov -d . -o junit_xml    # For CI/CD test results

# Scan only Terraform files (Checkov also supports CloudFormation, K8s, Docker, etc.)
checkov -d . --framework terraform
```

### Fixing the Issues

```hcl
# main.tf — Secure version

resource "aws_s3_bucket" "data" {
  bucket = "my-company-data"

  tags = {
    Name = "my-company-data"
  }
}

# ✅ Enable encryption
resource "aws_s3_bucket_server_side_encryption_configuration" "data" {
  bucket = aws_s3_bucket.data.id

  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "aws:kms"
    }
    bucket_key_enabled = true
  }
}

# ✅ Enable versioning
resource "aws_s3_bucket_versioning" "data" {
  bucket = aws_s3_bucket.data.id
  versioning_configuration {
    status = "Enabled"
  }
}

# ✅ Block public access
resource "aws_s3_bucket_public_access_block" "data" {
  bucket = aws_s3_bucket.data.id

  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

# ✅ Enable access logging
resource "aws_s3_bucket_logging" "data" {
  bucket = aws_s3_bucket.data.id

  target_bucket = aws_s3_bucket.logs.id
  target_prefix = "s3-access-logs/"
}

resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.medium"

  # ✅ Encrypt root volume
  root_block_device {
    encrypted = true
  }

  # ✅ Use IMDSv2 (more secure)
  metadata_options {
    http_endpoint               = "enabled"
    http_tokens                 = "required"      # Forces IMDSv2
    http_put_response_hop_limit = 1
  }

  # ✅ Enable monitoring
  monitoring = true
}

resource "aws_security_group" "web" {
  name        = "web-sg"
  description = "Security group for web servers"

  # ✅ SSH only from internal network (not 0.0.0.0/0)
  ingress {
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["10.0.0.0/8"]     # Internal network only
    description = "SSH from internal network"
  }

  # ✅ HTTPS only (not all ports)
  ingress {
    from_port   = 443
    to_port     = 443
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
    description = "HTTPS from anywhere"
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
    description = "Allow all outbound"
  }
}

resource "aws_db_instance" "main" {
  engine         = "mysql"
  instance_class = "db.t3.medium"
  username       = "admin"
  password       = var.db_password   # ✅ No hardcoded password!

  # ✅ Encryption enabled
  storage_encrypted = true

  # ✅ Backups enabled
  backup_retention_period = 7

  # ✅ Not publicly accessible
  publicly_accessible = false

  # ✅ IAM authentication enabled
  iam_database_authentication_enabled = true

  # ✅ Delete protection
  deletion_protection = true
}
```

### Skipping Checks (When You Have a Good Reason)

Sometimes you intentionally want to skip a check. You can do this with inline comments:

```hcl
resource "aws_s3_bucket" "public_website" {
  bucket = "my-public-website"

  #checkov:skip=CKV_AWS_145:This is a public static website, encryption not needed
  #checkov:skip=CKV_AWS_21:Static website doesn't need versioning
}
```

Or skip via command line:

```bash
# Skip specific checks
checkov -d . --skip-check CKV_AWS_145,CKV_AWS_21

# Run only specific checks
checkov -d . --check CKV_AWS_24,CKV_AWS_8
```

Or use a config file:

```yaml
# .checkov.yaml

framework:
  - terraform

skip-check:
  - CKV_AWS_145   # We handle encryption differently
  - CKV_AWS_21    # Versioning not needed for this project

soft-fail-on:
  - CKV_AWS_18    # Access logging is nice-to-have, not blocking
```

### Checkov in CI/CD

```yaml
# GitHub Actions example
name: Security Scan

on: [pull_request]

jobs:
  checkov:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Run Checkov
        uses: bridgecrewio/checkov-action@master
        with:
          directory: .
          framework: terraform
          output_format: sarif
          soft_fail: false         # Fail the pipeline if checks fail
          skip_check: CKV_AWS_145  # Optional: skip specific checks
```

### Checkov vs tfsec

You might also hear about **tfsec** (now part of **Trivy** by Aqua Security). Both do similar things:

| Feature | Checkov | tfsec / Trivy |
|---------|---------|---------------|
| Language | Python | Go |
| Speed | Moderate | Fast |
| Multi-framework | Terraform, CloudFormation, K8s, Docker, etc. | Terraform (tfsec), All (Trivy) |
| Custom policies | Python | Rego/JSON |
| CI/CD integration | ✅ | ✅ |
| Free | ✅ | ✅ |

Both are excellent. Pick one and stick with it.

## 🧪 Part 3: Terratest — Integration Testing

### What is Terratest?

**Terratest** is a **Go library** that lets you write **automated tests for your Terraform code**. Unlike tflint and Checkov (which analyze code without running it), Terratest actually:

1. **Deploys** your infrastructure to a real cloud environment
2. **Validates** that everything works correctly
3. **Destroys** everything when the test finishes

It's the most thorough form of testing, but also the most expensive (it uses real cloud resources).

### When to Use Terratest

```
Use Terratest when:
    ✅ You're building Terraform modules used by many teams
    ✅ You need to verify that infrastructure actually works
    ✅ You want to test that your app can connect to the database
    ✅ You want to verify load balancer routing
    ✅ You need to test complex infrastructure interactions
    ✅ You're building a shared Terraform module library

Don't use Terratest when:
    ❌ You're writing a simple one-off configuration
    ❌ You don't have a test cloud account
    ❌ You can't afford the cloud costs of testing
    ❌ tflint + Checkov + terraform plan are enough for your needs
```

### Prerequisites

1. **Go** installed (version 1.21+)
2. **Terraform** installed
3. **Cloud credentials** configured (AWS/Azure/GCP)
4. A **test/sandbox cloud account** (don't test in production!)

### Installation

Terratest is a Go module — you add it to a Go project:

```bash
# Create a test directory
mkdir -p test
cd test

# Initialize a Go module
go mod init github.com/yourcompany/infrastructure/test

# Download Terratest
go get github.com/gruntwork-io/terratest/modules/terraform
go get github.com/stretchr/testify/assert
```

### Example 1: Testing an S3 Bucket Module

Let's say you have a Terraform module that creates an S3 bucket:

```hcl
# modules/s3-bucket/main.tf

resource "aws_s3_bucket" "this" {
  bucket = var.bucket_name

  tags = var.tags
}

resource "aws_s3_bucket_versioning" "this" {
  bucket = aws_s3_bucket.this.id

  versioning_configuration {
    status = "Enabled"
  }
}

resource "aws_s3_bucket_server_side_encryption_configuration" "this" {
  bucket = aws_s3_bucket.this.id

  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "aws:kms"
    }
  }
}

resource "aws_s3_bucket_public_access_block" "this" {
  bucket = aws_s3_bucket.this.id

  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}
```

```hcl
# modules/s3-bucket/variables.tf

variable "bucket_name" {
  description = "Name of the S3 bucket"
  type        = string
}

variable "tags" {
  description = "Tags to apply to the bucket"
  type        = map(string)
  default     = {}
}
```

```hcl
# modules/s3-bucket/outputs.tf

output "bucket_id" {
  value = aws_s3_bucket.this.id
}

output "bucket_arn" {
  value = aws_s3_bucket.this.arn
}

output "bucket_region" {
  value = aws_s3_bucket.this.region
}
```

Now let's write a **Terratest test** for this module:

```go
// test/s3_bucket_test.go

package test

import (
	"fmt"
	"testing"

	"github.com/gruntwork-io/terratest/modules/aws"
	"github.com/gruntwork-io/terratest/modules/random"
	"github.com/gruntwork-io/terratest/modules/terraform"
	"github.com/stretchr/testify/assert"
)

func TestS3BucketModule(t *testing.T) {
	t.Parallel() // Run tests in parallel for speed

	// Generate a unique bucket name to avoid conflicts
	uniqueID := random.UniqueId()
	bucketName := fmt.Sprintf("test-terratest-%s", uniqueID)

	// Set the AWS region for testing
	awsRegion := "us-east-1"

	// Configure Terraform options
	terraformOptions := terraform.WithDefaultRetryableErrors(t, &terraform.Options{
		// Path to the Terraform module we're testing
		TerraformDir: "../modules/s3-bucket",

		// Variables to pass to the module
		Vars: map[string]interface{}{
			"bucket_name": bucketName,
			"tags": map[string]string{
				"Environment": "test",
				"ManagedBy":   "terratest",
			},
		},

		// Environment variables (for AWS region)
		EnvVars: map[string]string{
			"AWS_DEFAULT_REGION": awsRegion,
		},
	})

	// ⭐ THE MOST IMPORTANT LINE ⭐
	// This ensures the infrastructure is DESTROYED after the test,
	// even if the test fails!
	defer terraform.Destroy(t, terraformOptions)

	// Deploy the infrastructure
	terraform.InitAndApply(t, terraformOptions)

	// ===== ASSERTIONS — Verify everything is correct =====

	// 1. Get the outputs from Terraform
	bucketID := terraform.Output(t, terraformOptions, "bucket_id")
	bucketARN := terraform.Output(t, terraformOptions, "bucket_arn")
	bucketRegion := terraform.Output(t, terraformOptions, "bucket_region")

	// 2. Verify the bucket name matches
	assert.Equal(t, bucketName, bucketID)

	// 3. Verify the bucket is in the correct region
	assert.Equal(t, awsRegion, bucketRegion)

	// 4. Verify the ARN looks correct
	expectedARN := fmt.Sprintf("arn:aws:s3:::%s", bucketName)
	assert.Equal(t, expectedARN, bucketARN)

	// 5. Verify versioning is enabled
	versioning := aws.GetS3BucketVersioning(t, awsRegion, bucketName)
	assert.Equal(t, "Enabled", versioning)

	// 6. Verify the bucket is NOT public
	// (Terratest has helper functions for common AWS checks)
	aws.AssertS3BucketPolicyExists(t, awsRegion, bucketName)
}
```

Let's break down the key parts:

```go
// 1. Setup — configure what to deploy and how
terraformOptions := terraform.WithDefaultRetryableErrors(t, &terraform.Options{
    TerraformDir: "../modules/s3-bucket",   // Where's the code?
    Vars: map[string]interface{}{            // What values to pass?
        "bucket_name": bucketName,
    },
})

// 2. Cleanup — ALWAYS destroy after testing (even if test fails!)
defer terraform.Destroy(t, terraformOptions)

// 3. Deploy — actually create the infrastructure
terraform.InitAndApply(t, terraformOptions)

// 4. Verify — check that things are correct
bucketID := terraform.Output(t, terraformOptions, "bucket_id")
assert.Equal(t, bucketName, bucketID)
```

### Running the Test

```bash
cd test/

# Run all tests
go test -v -timeout 30m

# Run a specific test
go test -v -timeout 30m -run TestS3BucketModule
```

### Example 2: Testing a VPC + EC2 Setup

Here's a more complex test that deploys a VPC with an EC2 instance and verifies connectivity:

```go
// test/vpc_ec2_test.go

package test

import (
	"fmt"
	"testing"
	"time"

	http_helper "github.com/gruntwork-io/terratest/modules/http-helper"
	"github.com/gruntwork-io/terratest/modules/random"
	"github.com/gruntwork-io/terratest/modules/terraform"
	"github.com/stretchr/testify/assert"
)

func TestVPCWithWebServer(t *testing.T) {
	t.Parallel()

	uniqueID := random.UniqueId()

	terraformOptions := terraform.WithDefaultRetryableErrors(t, &terraform.Options{
		TerraformDir: "../environments/test",

		Vars: map[string]interface{}{
			"environment":   "test",
			"project_name":  fmt.Sprintf("test-%s", uniqueID),
			"instance_type": "t3.micro",
		},
	})

	// Always clean up!
	defer terraform.Destroy(t, terraformOptions)

	// Deploy
	terraform.InitAndApply(t, terraformOptions)

	// Get outputs
	publicIP := terraform.Output(t, terraformOptions, "web_server_public_ip")
	vpcID := terraform.Output(t, terraformOptions, "vpc_id")

	// Verify VPC was created
	assert.NotEmpty(t, vpcID)

	// Verify the web server is accessible via HTTP
	// This retries for up to 5 minutes (the instance needs time to boot)
	url := fmt.Sprintf("http://%s:80", publicIP)
	http_helper.HttpGetWithRetry(
		t,
		url,
		nil,                    // TLS config (nil for HTTP)
		200,                    // Expected status code
		"Hello, World!",        // Expected body content
		30,                     // Max retries
		10*time.Second,         // Time between retries
	)
}
```

### Terratest Best Practices

#### 1. Always Use `defer terraform.Destroy()`

```go
// ⭐ This line MUST be right after creating terraformOptions
defer terraform.Destroy(t, terraformOptions)

// Without it, if your test fails, the infrastructure stays running
// and you'll get a surprise cloud bill! 💸
```

#### 2. Use Unique Names

```go
// ✅ GOOD — unique names prevent conflicts between parallel tests
uniqueID := random.UniqueId()
bucketName := fmt.Sprintf("test-%s", uniqueID)

// ❌ BAD — if two people run tests at the same time, they'll collide
bucketName := "my-test-bucket"
```

#### 3. Use a Dedicated Test Account

```
Production AWS Account → NEVER run tests here
Development AWS Account → Maybe, but risky
Test/Sandbox AWS Account → ✅ PERFECT — dedicated to testing
```

#### 4. Run Tests in Parallel

```go
func TestModuleA(t *testing.T) {
    t.Parallel()   // This test runs in parallel with others
    // ...
}

func TestModuleB(t *testing.T) {
    t.Parallel()   // This test also runs in parallel
    // ...
}
```

#### 5. Use Timeouts

```bash
# Give enough time for infrastructure to deploy
go test -v -timeout 60m
```

#### 6. Use Test Stages for Faster Iteration

When developing tests, you don't want to redeploy on every test run:

```go
func TestVPC(t *testing.T) {
    t.Parallel()

    terraformOptions := &terraform.Options{
        TerraformDir: "../modules/vpc",
    }

    // SKIP_deploy=true go test -v -run TestVPC
    // This skips the deploy step and reuses existing infrastructure
    defer terraform.Destroy(t, terraformOptions)

    // Stage 1: Deploy
    test_structure.RunTestStage(t, "deploy", func() {
        terraform.InitAndApply(t, terraformOptions)
    })

    // Stage 2: Validate
    test_structure.RunTestStage(t, "validate", func() {
        vpcID := terraform.Output(t, terraformOptions, "vpc_id")
        assert.NotEmpty(t, vpcID)
    })
}
```

## 🔄 Part 4: Putting It All Together — The Testing Pyramid

Here's how all these tools fit together in a real-world workflow:

```
                          ╱╲
                         ╱  ╲
                        ╱    ╲
                       ╱      ╲
                      ╱Terratest╲         ← Slow, expensive, thorough
                     ╱  (deploy   ╲          Run on: merge to main
                    ╱   & verify)   ╲
                   ╱─────────────────╲
                  ╱                    ╲
                 ╱   terraform plan     ╲   ← Medium speed
                ╱   (preview changes)    ╲     Run on: every PR
               ╱──────────────────────────╲
              ╱                             ╲
             ╱   Checkov / tfsec             ╲  ← Fast, security-focused
            ╱   (security & compliance)       ╲    Run on: every commit
           ╱───────────────────────────────────╲
          ╱                                      ╲
         ╱   tflint + terraform fmt + validate    ╲ ← Fastest
        ╱   (syntax, style, basic errors)          ╲   Run on: every commit
       ╱────────────────────────────────────────────╲
```

### Complete CI/CD Pipeline Example

```yaml
# .github/workflows/terraform.yml

name: Terraform CI/CD

on:
  pull_request:
    paths:
      - 'terraform/**'
  push:
    branches:
      - main

jobs:
  # ===== Layer 1: Linting (Fast — runs on every PR) =====
  lint:
    name: Lint & Format
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: hashicorp/setup-terraform@v3

      - name: Terraform Format Check
        run: terraform fmt -check -recursive terraform/

      - name: Terraform Init
        run: terraform init -backend=false
        working-directory: terraform/environments/dev

      - name: Terraform Validate
        run: terraform validate
        working-directory: terraform/environments/dev

      - uses: terraform-linters/setup-tflint@v4
      - name: TFLint Init
        run: tflint --init
        working-directory: terraform/
      - name: TFLint
        run: tflint --recursive
        working-directory: terraform/

  # ===== Layer 2: Security Scan (Fast — runs on every PR) =====
  security:
    name: Security Scan
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Checkov Scan
        uses: bridgecrewio/checkov-action@master
        with:
          directory: terraform/
          framework: terraform
          soft_fail: false

  # ===== Layer 3: Plan (Medium — runs on every PR) =====
  plan:
    name: Terraform Plan
    runs-on: ubuntu-latest
    needs: [lint, security]     # Only run if lint and security pass
    steps:
      - uses: actions/checkout@v4

      - uses: hashicorp/setup-terraform@v3

      - name: Terraform Init
        run: terraform init
        working-directory: terraform/environments/dev

      - name: Terraform Plan
        run: terraform plan -out=plan.tfplan
        working-directory: terraform/environments/dev

  # ===== Layer 4: Integration Test (Slow — only on main branch) =====
  integration-test:
    name: Terratest
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    needs: [plan]
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-go@v5
        with:
          go-version: '1.21'

      - uses: hashicorp/setup-terraform@v3

      - name: Run Terratest
        run: |
          cd test
          go test -v -timeout 60m
        env:
          AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
          AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          AWS_DEFAULT_REGION: us-east-1

  # ===== Layer 5: Apply (only on main after all tests pass) =====
  apply:
    name: Terraform Apply
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    needs: [integration-test]
    environment: production     # Requires manual approval
    steps:
      - uses: actions/checkout@v4

      - uses: hashicorp/setup-terraform@v3

      - name: Terraform Init
        run: terraform init
        working-directory: terraform/environments/prod

      - name: Terraform Apply
        run: terraform apply -auto-approve
        working-directory: terraform/environments/prod
```

## 📋 Tool Comparison Cheat Sheet

| Feature | tflint | Checkov | Terratest |
|---------|--------|---------|-----------|
| **Type** | Linter | Security scanner | Integration test |
| **Language** | Go | Python | Go |
| **Speed** | ⚡ Very fast | ⚡ Fast | 🐢 Slow (deploys real infra) |

| **Deploys?** | No | No | Yes |

## 🏁 Key Takeaways

| Tool | Purpose | Run When | Difficulty |
|------|---------|----------|------------|
| **tflint** | Catch code errors early | Every commit | Easy |
| **Checkov** | Find security issues | Every PR | Easy |
| **terraform plan** | Preview changes | Every PR | Built-in |
| **Terratest** | Verify infra actually works | Merge to main | Medium |
| **Infracost** | Know the cost impact | Every PR | Easy |

**The minimum you should do:**
1. `terraform fmt` — Format your code
2. `terraform validate` — Check syntax
3. `tflint` — Catch errors
4. `checkov` — Find security issues
5. `terraform plan` — Preview changes

**The ideal setup adds:**
6. Terratest for module testing
7. Infracost for cost visibility
8. All of the above in a CI/CD pipeline
