## 📌 Why Integrate Terraform with CI/CD?

Let's be honest — running `terraform apply` from your laptop is fine when you're learning, but in a real team environment, it's a **terrible idea**. Here's why:

- **No review process** — Someone can change infrastructure without anyone looking at it
- **No consistency** — Different people have different local setups (Terraform versions, credentials, plugins)
- **No audit trail** — Who ran what? When? From where?
- **Credential sprawl** — Everyone needs cloud credentials on their machines
- **No automation** — Manual processes are slow and error-prone
- **State conflicts** — Two people running `terraform apply` at the same time = disaster 💥

**The solution?** Put Terraform in a CI/CD pipeline, just like you do with application code.

```
The Journey of Infrastructure Changes:

MANUAL WAY (dangerous):
Developer → laptop → terraform apply → production
  "Hope nobody else is running this right now..." 😬

CI/CD WAY (safe & automated):
Developer → Git PR → Code Review → Automated Plan → Approval → Automated Apply → Production
  "Everyone can see what's changing, and it's all automated!" ✅
```

## 🔄 The Standard Terraform CI/CD Workflow

No matter which CI/CD tool you use, the workflow is almost always the same:

```
┌─────────────┐     ┌─────────────┐     ┌──────────────┐     ┌───────────────┐     ┌──────────────┐
│  Developer   │────▶│  Push Code  │────▶│  CI Pipeline │────▶│  Review &     │────▶│  Apply       │
│  writes HCL  │     │  to Git PR  │     │  runs plan   │     │  Approve      │     │  changes     │
└─────────────┘     └─────────────┘     └──────────────┘     └───────────────┘     └──────────────┘
```

### Step-by-Step:

1. **Developer** creates/modifies Terraform code locally
2. **Pushes** the changes to a Git branch and opens a **Pull Request** (PR)
3. **CI pipeline** automatically runs:
   - `terraform fmt -check` — Is the code formatted correctly?
   - `terraform validate` — Is the configuration valid?
   - `terraform plan` — What changes will be made?
4. **Plan output** is posted as a comment on the PR for everyone to review
5. **Team reviews** the plan — "Are these changes safe?"
6. After **approval**, `terraform apply` runs automatically (or with manual trigger)
7. **PR is merged** into the main branch

## 🏠 Option 1: DIY Terraform Pipeline (GitHub Actions Example)

Before we talk about specialized tools, let's see how you'd build a Terraform CI/CD pipeline yourself using **GitHub Actions**.

### Directory Structure

```
my-infra-repo/
├── .github/
│   └── workflows/
│       └── terraform.yml
├── environments/
│   ├── dev/
│   │   ├── main.tf
│   │   ├── variables.tf
│   │   └── terraform.tfvars
│   ├── staging/
│   │   ├── main.tf
│   │   ├── variables.tf
│   │   └── terraform.tfvars
│   └── prod/
│       ├── main.tf
│       ├── variables.tf
│       └── terraform.tfvars
└── modules/
    ├── vpc/
    ├── ec2/
    └── rds/
```

### GitHub Actions Workflow

```yaml
# .github/workflows/terraform.yml

name: Terraform CI/CD

on:
  pull_request:
    branches: [main]
    paths:
      - 'environments/**'
      - 'modules/**'
  push:
    branches: [main]
    paths:
      - 'environments/**'
      - 'modules/**'

# IMPORTANT: This prevents multiple workflows from running at the same time
# and potentially conflicting on terraform state
concurrency:
  group: terraform-${{ github.ref }}
  cancel-in-progress: false

env:
  TF_VERSION: "1.7.0"
  AWS_REGION: "us-east-1"

# These permissions are needed for GitHub Actions to:
# - Read the repo contents
# - Write comments on PRs
# - Authenticate with AWS using OIDC (no long-lived keys!)
permissions:
  contents: read
  pull-requests: write
  id-token: write

jobs:
  # ─────────────────────────────────────────────
  # JOB 1: Format Check
  # ─────────────────────────────────────────────
  format:
    name: "🔍 Terraform Format Check"
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Terraform
        uses: hashicorp/setup-terraform@v3
        with:
          terraform_version: ${{ env.TF_VERSION }}

      # Check if all .tf files are properly formatted
      # This fails the pipeline if someone didn't run 'terraform fmt'
      - name: Terraform Format Check
        run: terraform fmt -check -recursive -diff

  # ─────────────────────────────────────────────
  # JOB 2: Validate & Plan
  # ─────────────────────────────────────────────
  plan:
    name: "📋 Terraform Plan"
    runs-on: ubuntu-latest
    needs: format
    strategy:
      matrix:
        environment: [dev, staging, prod]
    
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Terraform
        uses: hashicorp/setup-terraform@v3
        with:
          terraform_version: ${{ env.TF_VERSION }}

      # Authenticate with AWS using OIDC — no hardcoded keys!
      # (You need to set up an OIDC provider in AWS first)
      - name: Configure AWS Credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::role/github-actions-terraform
          aws-region: ${{ env.AWS_REGION }}

      # Initialize Terraform (download providers, configure backend)
      - name: Terraform Init
        working-directory: environments/${{ matrix.environment }}
        run: terraform init

      # Validate the configuration
      - name: Terraform Validate
        working-directory: environments/${{ matrix.environment }}
        run: terraform validate

      # Generate a plan and save it to a file
      - name: Terraform Plan
        id: plan
        working-directory: environments/${{ matrix.environment }}
        run: |
          terraform plan -no-color -out=tfplan 2>&1 | tee plan_output.txt
        continue-on-error: true

      # Post the plan as a comment on the PR
      # This is super helpful for reviewers!
      - name: Post Plan to PR
        if: github.event_name == 'pull_request'
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            const planOutput = fs.readFileSync(
              'environments/${{ matrix.environment }}/plan_output.txt', 'utf8'
            );
            
            // Truncate if too long (GitHub has a comment size limit)
            const maxLength = 60000;
            const truncated = planOutput.length > maxLength 
              ? planOutput.substring(0, maxLength) + '\n... (truncated)'
              : planOutput;
            
            const body = `### 📋 Terraform Plan — \`${{ matrix.environment }}\`
            
            <details>
            <summary>Click to expand plan output</summary>
            
            \`\`\`hcl
            ${truncated}
            \`\`\`
            
            </details>
            
            **Status:** ${{ steps.plan.outcome == 'success' && '✅ Success' || '❌ Failed' }}`;
            
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: body
            });

      # Fail the job if plan failed
      - name: Check Plan Status
        if: steps.plan.outcome == 'failure'
        run: exit 1

      # Save the plan file as an artifact for the apply job
      - name: Upload Plan Artifact
        if: github.event_name == 'pull_request'
        uses: actions/upload-artifact@v4
        with:
          name: tfplan-${{ matrix.environment }}
          path: environments/${{ matrix.environment }}/tfplan

  # ─────────────────────────────────────────────
  # JOB 3: Apply (only on merge to main)
  # ─────────────────────────────────────────────
  apply:
    name: "🚀 Terraform Apply"
    runs-on: ubuntu-latest
    needs: plan
    # Only run on push to main (i.e., after PR is merged)
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    strategy:
      matrix:
        environment: [dev, staging, prod]
      # Apply one environment at a time, in order!
      max-parallel: 1
    
    environment: ${{ matrix.environment }}  # Uses GitHub Environments for approvals!
    
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Terraform
        uses: hashicorp/setup-terraform@v3
        with:
          terraform_version: ${{ env.TF_VERSION }}

      - name: Configure AWS Credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::role/github-actions-terraform
          aws-region: ${{ env.AWS_REGION }}

      - name: Terraform Init
        working-directory: environments/${{ matrix.environment }}
        run: terraform init

      # Apply the changes! 
      # -auto-approve because we already reviewed the plan in the PR
      - name: Terraform Apply
        working-directory: environments/${{ matrix.environment }}
        run: terraform apply -auto-approve
```

### What This Pipeline Does

Let me walk you through what happens:

```
Developer opens a PR:
  1. ✅ Format Check — Are all .tf files properly formatted?
  2. ✅ Validate — Is the HCL syntax correct?
  3. ✅ Plan — What changes will Terraform make?
  4. 💬 Plan is posted as a PR comment for review
  5. 👀 Team reviews the plan
  
After PR is merged to main:
  6. 🚀 Apply runs automatically (dev → staging → prod)
  7. Each environment can have manual approval gates (GitHub Environments)
```

## 🔑 Handling Credentials Securely

This is **critical** — never put AWS/Azure/GCP credentials in your code or plain environment variables!

### Method 1: OIDC (OpenID Connect) — Recommended ⭐

No long-lived credentials at all! GitHub Actions gets **temporary** tokens from AWS.

```yaml
# In your workflow:
- name: Configure AWS Credentials
  uses: aws-actions/configure-aws-credentials@v4
  with:
    role-to-assume: arn:aws:iam::role/github-actions-terraform
    aws-region: us-east-1
```

**How it works:**
```
GitHub Actions → "Hey AWS, I'm the repo 'myorg/infra', here's my OIDC token"
AWS → "Let me verify... ✅ Yep, that's legit. Here's a temporary credential (expires in 1 hour)"
GitHub Actions → Uses temporary credential to run Terraform
```

### Method 2: GitHub Secrets (If OIDC Not Available)

Store credentials as **encrypted secrets** in your repo settings:

```yaml
env:
  AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
  AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
```

> ⚠️ **Important:** OIDC is always preferred because there are no long-lived credentials to rotate or leak.

## 🏛️ Option 2: Atlantis — Pull Request Automation for Terraform

### What is Atlantis?

**Atlantis** is an open-source tool that automates Terraform via **pull requests**. It's a server that listens for GitHub/GitLab/Bitbucket webhooks and runs `terraform plan` and `terraform apply` directly from PRs.

Think of Atlantis as a **bot team member** that:
- Automatically runs `terraform plan` when you open a PR
- Posts the plan output as a comment
- Runs `terraform apply` when you comment `atlantis apply`
- Locks the directory so nobody else can run Terraform at the same time

```
How Atlantis Works:

Developer opens PR → GitHub sends webhook → Atlantis receives it
                                                    │
                                                    ▼
                                            Atlantis runs 'terraform plan'
                                                    │
                                                    ▼
                                            Posts plan as PR comment
                                                    │
                                                    ▼
                                     Reviewer comments: "atlantis apply"
                                                    │
                                                    ▼
                                            Atlantis runs 'terraform apply'
                                                    │
                                                    ▼
                                            Posts result as PR comment
                                                    │
                                                    ▼
                                            PR is merged ✅
```

### Installing Atlantis

#### Method 1: Docker (Simplest)

```bash
# Pull and run Atlantis
docker run -d \
  --name atlantis \
  -p 4141:4141 \
  -e ATLANTIS_GH_USER="my-atlantis-bot" \
  -e ATLANTIS_GH_TOKEN="ghp_xxxxxxxxxxxx" \
  -e ATLANTIS_GH_WEBHOOK_SECRET="my-webhook-secret" \
  -e ATLANTIS_REPO_ALLOWLIST="github.com/myorg/*" \
  -e AWS_ACCESS_KEY_ID="AKIA..." \
  -e AWS_SECRET_ACCESS_KEY="..." \
  ghcr.io/runatlantis/atlantis:latest \
  server
```

#### Method 2: Kubernetes (Production)

```yaml

            - name: atlantis-data
              mountPath: /atlantis-data

        - name: atlantis-data
          persistentVolumeClaim:
            claimName: atlantis-data-pvc
---
apiVersion: v1
kind: Service
metadata:
  name: atlantis
  namespace: atlantis
spec:
  type: ClusterIP
  ports:
    - port: 80
      targetPort: 4141
  selector:
    app: atlantis
```

### Configuring Atlantis

Atlantis uses two types of config files:

#### Server-Side Config (`atlantis.yaml` — in Atlantis server)

This controls **global** settings:

```yaml
# server-side atlantis.yaml
repos:
  # Allow all repos in your org
  - id: "github.com/myorg/*"
    # Require approval before apply
    apply_requirements: [approved, mergeable]
    # Allow repos to have their own atlantis.yaml
    allowed_overrides: [workflow, apply_requirements]
    # Delete the plan comment and source branch after merge
    delete_source_branch_on_merge: true
```

#### Repo-Side Config (`atlantis.yaml` — in your Terraform repo)

This controls **per-repo** behavior:

```yaml
# atlantis.yaml (in the root of your Terraform repo)
version: 3
automerge: false
parallel_plan: true     # Plan all projects in parallel
parallel_apply: false    # Apply one at a time (safer!)

projects:
  # Dev environment
  - name: dev
    dir: environments/dev
    workspace: default
    terraform_version: v1.7.0
    autoplan:
      when_modified:
        - "*.tf"
        - "*.tfvars"
        - "../../modules/**/*.tf"  # Also trigger when shared modules change!
      enabled: true
    apply_requirements: [approved]

  # Staging environment
  - name: staging
    dir: environments/staging
    workspace: default
    terraform_version: v1.7.0
    autoplan:
      when_modified:
        - "*.tf"
        - "*.tfvars"
        - "../../modules/**/*.tf"
      enabled: true
    apply_requirements: [approved]

  # Production environment — extra strict!
  - name: prod
    dir: environments/prod
    workspace: default
    terraform_version: v1.7.0
    autoplan:
      when_modified:
        - "*.tf"
        - "*.tfvars"
        - "../../modules/**/*.tf"
      enabled: true
    apply_requirements: [approved, mergeable]
    workflow: prod-workflow

# Custom workflow for production with extra checks
workflows:
  prod-workflow:
    plan:
      steps:
        - run: terraform fmt -check -diff
        - run: tflint
        - init
        - plan:
            extra_args: ["-lock=false"]
    apply:
      steps:
        - apply
```

Here's what working with Atlantis looks like in practice:

```
1. You open a PR that changes environments/dev/main.tf

2. Atlantis automatically comments:
   ┌─────────────────────────────────────────────┐
   │ 🤖 Atlantis — Plan for dir: environments/dev│
   │                                              │
   │ Terraform will perform the following actions:│
   │                                              │
   │   # aws_instance.web will be created         │
   │   + resource "aws_instance" "web" {          │

   │                                              │

5. Atlantis runs terraform apply and comments:
   ┌─────────────────────────────────────────────┐
   │ 🤖 Atlantis — Apply for dir: environments/dev│
   │                                              │
   │ aws_instance.web: Creating...                │

   │                                              │
   │ Apply complete! Resources: 1 added.          │
   │ 🔓 Lock released                            │
   └─────────────────────────────────────────────┘

6. You merge the PR! ✅
```

### Atlantis Commands You'll Use

| Command | What It Does |
|---------|-------------|
| `atlantis plan` | Run plan on all projects in the PR |
| `atlantis plan -d environments/dev` | Plan a specific directory |
| `atlantis apply` | Apply all planned projects |
| `atlantis apply -d environments/dev` | Apply a specific directory |
| `atlantis unlock` | Release the lock (if something went wrong) |
| `atlantis plan -- -target=aws_instance.web` | Plan with extra Terraform flags |
| `atlantis import ADDRESS ID` | Import existing resources |

### Atlantis Locking — Preventing Conflicts

One of Atlantis's best features is **directory locking**:

```
PR #42 changes environments/dev/
  → Atlantis LOCKS environments/dev/
  → No other PR can plan or apply in that directory

PR #43 also changes environments/dev/
  → Atlantis says: "🔒 This directory is locked by PR #42. 
     Please merge or close that PR first."

PR #42 is merged
  → Atlantis UNLOCKS environments/dev/
  → PR #43 can now proceed
```

This prevents the dreaded **state conflict** where two people apply different changes at the same time!

## ☁️ Option 3: Terraform Cloud

### What is Terraform Cloud?

**Terraform Cloud** is HashiCorp's **managed SaaS platform** for running Terraform. Instead of building your own CI/CD pipeline, Terraform Cloud handles everything for you:

- Remote execution (plan & apply run on HashiCorp's servers)
- State management (no need for S3 backends)
- Team access controls
- Policy enforcement (Sentinel / OPA)
- Private module registry
- Cost estimation
- Drift detection

```
                        ┌──────────────────────────────────┐
                        │         Terraform Cloud          │
                        │                                  │
Developer ──Git Push──▶ │  ┌─────────┐  ┌──────────────┐  │
                        │  │  Plan    │  │  State Store │  │
                        │  └────┬────┘  └──────────────┘  │
                        │       │                          │
                        │  ┌────▼────┐  ┌──────────────┐  │
                        │  │  Apply  │  │  Policy Check │  │
                        │  └─────────┘  └──────────────┘  │
                        │                                  │
                        │  ┌─────────┐  ┌──────────────┐  │
                        │  │  RBAC   │  │  Module Reg  │  │
                        │  └─────────┘  └──────────────┘  │
                        └──────────────────────────────────┘
```

### Terraform Cloud — VCS-Driven Workflow

The most popular way to use Terraform Cloud is with **VCS integration** (connecting your GitHub/GitLab repo):

```
How it works:

1. Connect your GitHub repo to a Terraform Cloud workspace
2. When you push to a branch / open a PR:
   → Terraform Cloud runs a plan automatically
   → Plan output appears in the PR as a status check
3. When the PR is merged:
   → Terraform Cloud runs apply automatically
```

**To set this up in Terraform Cloud UI:**

1. Go to your workspace → Settings → Version Control
2. Connect to GitHub/GitLab
3. Select your repo
4. Configure:
   - **VCS branch:** `main`
   - **Working directory:** `environments/dev` (if needed)
   - **Auto-apply on merge:** On/Off

### Terraform Cloud — Workspaces

In Terraform Cloud, **workspaces** are how you manage different environments:

```
Organization: my-company
├── Workspace: infra-dev        → environments/dev/
├── Workspace: infra-staging    → environments/staging/
├── Workspace: infra-prod       → environments/prod/
├── Workspace: networking       → networking/
└── Workspace: security         → security/
```

Each workspace has:
- Its own **state file**
- Its own **variables** (Terraform and environment variables)
- Its own **team access** permissions
- Its own **run history**

#### Setting Variables in Terraform Cloud

```
Workspace: infra-dev
├── Terraform Variables:
│   ├── instance_type = "t3.micro"
│   ├── environment   = "dev"
│   └── region        = "us-east-1"
│
├── Environment Variables:
│   ├── AWS_ACCESS_KEY_ID     = "AKIA..." (sensitive ✅)
│   ├── AWS_SECRET_ACCESS_KEY = "xxx..."  (sensitive ✅)
│   └── TF_LOG                = "INFO"
```

### Terraform Cloud — Run Triggers

You can chain workspaces so that applying one **triggers** another:

```
Example: When you update the "networking" workspace,
automatically trigger a plan in the "app" workspace.

┌─────────────┐     triggers     ┌──────────────┐     triggers     ┌──────────────┐
│  networking  │ ──────────────▶ │   app-dev    │ ──────────────▶ │  monitoring  │
└─────────────┘                  └──────────────┘                  └──────────────┘

Use case: Your app depends on VPC/subnet IDs from the networking workspace.
When networking changes, the app workspace automatically picks up the new values.
```

### Terraform Cloud — Sentinel Policies (Policy as Code)

Sentinel is HashiCorp's **policy-as-code** framework. It lets you enforce rules before Terraform can apply:

```python
# Example Sentinel policy: only allow t3 instance types

import "tfplan/v2" as tfplan

# Find all AWS instances in the plan
aws_instances = filter tfplan.resource_changes as _, rc {
    rc.type is "aws_instance" and
    rc.mode is "managed" and
    (rc.change.actions contains "create" or rc.change.actions contains "update")
}

# Check that all instances use t3 family
main = rule {
    all aws_instances as _, instance {
        instance.change.after.instance_type matches "^t3\\."
    }
}

# What this does:
# ✅ t3.micro    → Allowed
# ✅ t3.medium   → Allowed  
# ❌ m5.xlarge   → BLOCKED! "Policy check failed"
# ❌ c5.2xlarge  → BLOCKED! "Policy check failed"
```

More Sentinel policy examples:

```python
# Policy: All S3 buckets must have encryption enabled
import "tfplan/v2" as tfplan

s3_buckets = filter tfplan.resource_changes as _, rc {
    rc.type is "aws_s3_bucket" and
    rc.mode is "managed" and
    (rc.change.actions contains "create" or rc.change.actions contains "update")
}

main = rule {
    all s3_buckets as _, bucket {
        bucket.change.after.server_side_encryption_configuration is not null
    }
}
```

```python
# Policy: Require specific tags on all resources
import "tfplan/v2" as tfplan

required_tags = ["Environment", "Team", "CostCenter"]

all_resources = filter tfplan.resource_changes as _, rc {
    rc.change.after.tags is not null
}

main = rule {
    all all_resources as _, resource {
        all required_tags as tag {
            resource.change.after.tags contains tag
        }
    }
}
```

## 🏢 Terraform Enterprise

**Terraform Enterprise** is the **self-hosted** version of Terraform Cloud. It has all the features of Terraform Cloud Business, plus:

| Feature | Cloud | Enterprise |
|---------|-------|-----------|
| Hosting | HashiCorp SaaS | Your own servers |
| Air-gapped support | ❌ | ✅ |
| Custom SAML/SSO | ✅ (Business) | ✅ |
| Audit logging | ✅ (Business) | ✅ |
| Sentinel policies | ✅ (Business) | ✅ |
| Private networking | Limited | Full control |
| Compliance (FedRAMP, etc.) | Limited | Full control |

### When to Use Enterprise vs Cloud

```
Use Terraform Cloud if:
  ✅ You're a small-to-medium team
  ✅ You're comfortable with SaaS
  ✅ You don't have strict data residency requirements
  ✅ You want zero maintenance

Use Terraform Enterprise if:
  ✅ You need air-gapped environments (no internet)
  ✅ You have strict compliance requirements (FedRAMP, HIPAA, etc.)
  ✅ You need full control over networking
  ✅ You need to run in your own VPC/data center
  ✅ You're a large enterprise with hundreds of teams
```

## 🔧 Option 4: GitLab CI/CD with Terraform

GitLab has **built-in** Terraform integration. Here's how to set it up:

```yaml
# .gitlab-ci.yml

image:
  name: hashicorp/terraform:1.7.0
  entrypoint: [""]

variables:
  TF_ROOT: environments/dev
  TF_STATE_NAME: dev

cache:
  key: terraform-${TF_STATE_NAME}
  paths:
    - ${TF_ROOT}/.terraform/

stages:
  - validate
  - plan
  - apply

# ─────────────────────────────────────
# Stage 1: Validate
# ─────────────────────────────────────
validate:
  stage: validate
  script:
    - cd ${TF_ROOT}
    - terraform init -backend=false
    - terraform fmt -check
    - terraform validate
  rules:
    - if: $CI_MERGE_REQUEST_IID
    - if: $CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH

# ─────────────────────────────────────
# Stage 2: Plan
# ─────────────────────────────────────
plan:
  stage: plan
  script:
    - cd ${TF_ROOT}
    - terraform init
    - terraform plan -out=plan.cache
  artifacts:
    paths:
      - ${TF_ROOT}/plan.cache
    reports:
      # This creates a nice Terraform report in the MR!
      terraform: ${TF_ROOT}/plan.json
  rules:
    - if: $CI_MERGE_REQUEST_IID
    - if: $CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH
  before_script:
    - cd ${TF_ROOT}
    - terraform plan -out=plan.cache
    - terraform show -json plan.cache > plan.json

# ─────────────────────────────────────
# Stage 3: Apply (manual approval)
# ─────────────────────────────────────
apply:
  stage: apply
  script:
    - cd ${TF_ROOT}
    - terraform init
    - terraform apply plan.cache
  dependencies:
    - plan
  rules:
    - if: $CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH
  when: manual  # Requires manual click to apply!
  environment:
    name: dev
```

## 💡 Best Practices for Terraform CI/CD

### 1. Always Use Remote State

```hcl
# Never store state locally in CI/CD!
terraform {
  backend "s3" {
    bucket         = "my-terraform-state"
    key            = "dev/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-locks"  # For state locking!
    encrypt        = true
  }
}
```

### 2. Use State Locking

```
Without locking:
  Pipeline A: terraform apply   →  Reads state  →  Applies changes
  Pipeline B: terraform apply   →  Reads state  →  Applies changes  (CONFLICT! 💥)

With locking:
  Pipeline A: terraform apply   →  Acquires lock  →  Reads state  →  Applies  →  Releases lock
  Pipeline B: terraform apply   →  Tries to lock  →  "State is locked!" → Waits...
```

### 3. Pin Provider & Terraform Versions

```hcl
terraform {
  required_version = "= 1.7.0"  # Exact version! No surprises

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.30"  # Allow minor updates only
    }
  }
}
```

### 4. Separate Plan and Apply

```
❌ BAD:  terraform plan && terraform apply -auto-approve
   (What if something changed between plan and apply?)

✅ GOOD: terraform plan -out=tfplan  →  Review  →  terraform apply tfplan
   (Apply exactly what was planned, nothing more, nothing less)
```

### 5. Use Environment Protection Rules

```yaml
# GitHub Environments — require approval for prod
# Settings → Environments → prod:
#   ✅ Required reviewers: @team-leads
#   ✅ Wait timer: 5 minutes
#   ✅ Deployment branches: main only
```

### 6. Add Terraform Security Scanning

```yaml
# Add these to your pipeline:
- name: tflint
  run: |
    tflint --init
    tflint --recursive

- name: tfsec
  run: tfsec .

- name: checkov
  run: checkov -d . --framework terraform
```

### 7. Use .terraform.lock.hcl

Always commit the lock file! It ensures everyone uses the same provider versions:

```bash
# This file is auto-generated by terraform init
# COMMIT IT to your repo!
.terraform.lock.hcl
```
