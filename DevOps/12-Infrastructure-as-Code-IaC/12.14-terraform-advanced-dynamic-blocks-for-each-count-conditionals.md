# 12.14 — Terraform Advanced: Dynamic Blocks, for_each, count & Conditionals

> **Level:** Advanced  
> **Goal:** Master Terraform's most powerful features that help you write DRY (Don't Repeat Yourself), flexible, and reusable infrastructure code. Learn how to use `count`, `for_each`, dynamic blocks, conditionals, and advanced expressions to handle real-world complexity.

---

## 📌 Why Do You Need These Features?

When you start writing Terraform code, you'll quickly run into situations where you need to:

- Create **multiple similar resources** (e.g., 5 EC2 instances, 3 S3 buckets)
- **Conditionally** create resources (e.g., only create a bastion host in dev, not in prod)
- Create resources from a **list or map** of values
- Dynamically generate **nested blocks** inside a resource

Without these features, you'd end up **copy-pasting** resource blocks everywhere — which is messy, error-prone, and hard to maintain.

```
WITHOUT advanced features:                    WITH advanced features:

resource "aws_instance" "web1" {              resource "aws_instance" "web" {
  ami           = "ami-123"                     for_each = var.instances
  instance_type = "t3.micro"                    ami           = each.value.ami
}                                               instance_type = each.value.type
resource "aws_instance" "web2" {              }
  ami           = "ami-123"                   
  instance_type = "t3.micro"                  # 2 lines vs 12 lines!
}                                             # And it scales to 100 instances!
resource "aws_instance" "web3" {
  ami           = "ami-123"
  instance_type = "t3.micro"
}
```

## 🔢 count — Create Multiple Copies of a Resource

### Basic count

`count` is the simplest way to create multiple copies of a resource. You give it a number, and Terraform creates that many copies.

```hcl
# Create 3 EC2 instances
resource "aws_instance" "web" {
  count = 3                                # ← Create 3 copies!

  ami           = "ami-12345678"
  instance_type = "t3.micro"

  tags = {
    Name = "web-server-${count.index}"     # ← count.index gives 0, 1, 2
  }
}
```

**What Terraform creates:**

```
aws_instance.web[0]  →  Name: "web-server-0"
aws_instance.web[1]  →  Name: "web-server-1"
aws_instance.web[2]  →  Name: "web-server-2"
```

### count.index — The Current Number

`count.index` starts at **0** and goes up to `count - 1`:

```hcl
resource "aws_subnet" "private" {
  count = 3

  vpc_id            = aws_vpc.main.id
  cidr_block        = "10.0.${count.index + 1}.0/24"   # 10.0.1.0/24, 10.0.2.0/24, 10.0.3.0/24
  availability_zone = data.aws_availability_zones.available.names[count.index]

  tags = {
    Name = "private-subnet-${count.index + 1}"
  }
}

# Results:
# aws_subnet.private[0] → CIDR: 10.0.1.0/24, AZ: us-east-1a
# aws_subnet.private[1] → CIDR: 10.0.2.0/24, AZ: us-east-1b
# aws_subnet.private[2] → CIDR: 10.0.3.0/24, AZ: us-east-1c
```

### count with Variables

```hcl
variable "instance_count" {
  description = "Number of instances to create"
  type        = number
  default     = 2
}

resource "aws_instance" "web" {
  count = var.instance_count

  ami           = "ami-12345678"
  instance_type = "t3.micro"

  tags = {
    Name = "web-${count.index + 1}"
  }
}
```

### Referencing count Resources

When you use `count`, the resource becomes a **list**:

```hcl
# Reference a specific instance
output "first_instance_id" {
  value = aws_instance.web[0].id
}

# Reference ALL instances (returns a list)
output "all_instance_ids" {
  value = aws_instance.web[*].id
}

# The [*] is called "splat expression" — it means "all of them"
# It's equivalent to: [for i in aws_instance.web : i.id]
```

### ⚠️ The Problem with count

`count` has a **major limitation** — it uses **index-based** ordering. If you remove an item from the middle, everything after it **shifts**:

```
Imagine you have 3 servers:
  web[0] = "frontend"
  web[1] = "backend"    ← You want to remove this
  web[2] = "api"

After removing web[1]:
  web[0] = "frontend"   (unchanged ✅)
  web[1] = "api"         (was web[2], now gets DESTROYED and RECREATED! 😱)

Terraform thinks:
  - web[1] changed from "backend" to "api" → destroy old, create new
  - web[2] no longer exists → destroy it
  
That's NOT what you wanted!
```

**This is why `for_each` was invented.** Use `count` only when:
- Resources are **identical** (no differences between copies)
- You're doing **conditional** creation (count = 0 or 1)
- The order/list will **never change**

## 🔄 for_each — Create Resources from a Map or Set

### Why for_each?

`for_each` solves the ordering problem. Instead of using numeric indexes, it uses **keys** (names). If you remove an item, only that item is destroyed — nothing else is affected.

### for_each with a Set of Strings

```hcl
# Create S3 buckets from a set of names
resource "aws_s3_bucket" "buckets" {
  for_each = toset(["logs", "backups", "artifacts"])

  bucket = "my-company-${each.key}"

  tags = {
    Name    = each.key
    Purpose = each.key
  }
}

# What Terraform creates:
# aws_s3_bucket.buckets["logs"]      → bucket: "my-company-logs"
# aws_s3_bucket.buckets["backups"]   → bucket: "my-company-backups"
# aws_s3_bucket.buckets["artifacts"] → bucket: "my-company-artifacts"
```

**Key difference from count:**
```
With count:    aws_instance.web[0], web[1], web[2]       ← Numbers
With for_each: aws_s3_bucket.buckets["logs"], ["backups"] ← Names!

If you remove "backups" from the set:
  - Only "backups" bucket is destroyed
  - "logs" and "artifacts" are completely untouched ✅
```

### for_each with a Map

This is the most powerful pattern — each item can have **different** properties:

```hcl
variable "instances" {
  description = "Map of instances to create"
  type = map(object({
    instance_type = string
    ami           = string
    environment   = string
  }))
  default = {
    "web-server" = {
      instance_type = "t3.micro"
      ami           = "ami-12345678"
      environment   = "production"
    }
    "api-server" = {
      instance_type = "t3.small"
      ami           = "ami-12345678"
      environment   = "production"
    }
    "worker" = {
      instance_type = "t3.medium"
      ami           = "ami-87654321"
      environment   = "production"
    }
  }
}

resource "aws_instance" "servers" {
  for_each = var.instances

  ami           = each.value.ami            # ← each.value = the object
  instance_type = each.value.instance_type

  tags = {
    Name        = each.key                   # ← each.key = "web-server", "api-server", etc.
    Environment = each.value.environment
  }
}

# What Terraform creates:
# aws_instance.servers["web-server"] → t3.micro
# aws_instance.servers["api-server"] → t3.small
# aws_instance.servers["worker"]     → t3.medium
```

### each.key vs each.value

| Expression | What It Returns |
|-----------|----------------|
| `each.key` | The **key** of the current item (string) |
| `each.value` | The **value** of the current item (can be string, object, etc.) |

```hcl
# For a set:     each.key == each.value (both are the string)
# For a map:     each.key = the key, each.value = the value
```

### Referencing for_each Resources

```hcl
# Reference a specific resource by key
output "web_server_id" {
  value = aws_instance.servers["web-server"].id
}

# Reference ALL resources (returns a map)
output "all_server_ids" {
  value = { for k, v in aws_instance.servers : k => v.id }
}
# Output: { "web-server" = "i-abc", "api-server" = "i-def", "worker" = "i-ghi" }

# Get just the values as a list
output "all_ips" {
  value = [for v in aws_instance.servers : v.private_ip]
}
```

## 🎭 Dynamic Blocks — Dynamically Generate Nested Blocks

### The Problem

Some Terraform resources have **nested blocks** that you might need to repeat multiple times. For example, `ingress` rules in a security group:

```hcl
# Without dynamic blocks — lots of repetition!
resource "aws_security_group" "web" {
  name   = "web-sg"
  vpc_id = aws_vpc.main.id

  ingress {
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["10.0.0.0/8"]
  }

  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  ingress {
    from_port   = 443
    to_port     = 443
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  # What if you need 20 rules? 50 rules? 😩
}
```

### The Solution: Dynamic Blocks

A `dynamic` block lets you **generate** nested blocks from a list or map:

```hcl
variable "ingress_rules" {
  type = list(object({
    port        = number
    protocol    = string
    cidr_blocks = list(string)
    description = string
  }))
  default = [
    {
      port        = 22
      protocol    = "tcp"
      cidr_blocks = ["10.0.0.0/8"]
      description = "SSH from internal"
    },
    {
      port        = 80
      protocol    = "tcp"
      cidr_blocks = ["0.0.0.0/0"]
      description = "HTTP"
    },
    {
      port        = 443
      protocol    = "tcp"
      cidr_blocks = ["0.0.0.0/0"]
      description = "HTTPS"
    }
  ]
}

resource "aws_security_group" "web" {
  name        = "web-sg"
  description = "Web server security group"
  vpc_id      = aws_vpc.main.id

  # This generates one "ingress" block for each item in the list
  dynamic "ingress" {
    for_each = var.ingress_rules          # ← What to iterate over
    
    content {                              # ← What each block looks like
      from_port   = ingress.value.port     # ← ingress.value = current item
      to_port     = ingress.value.port
      protocol    = ingress.value.protocol
      cidr_blocks = ingress.value.cidr_blocks
      description = ingress.value.description
    }
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}
```

### How Dynamic Blocks Work

```
dynamic "BLOCK_NAME" {
  for_each = COLLECTION           ← List, set, or map to iterate over
  iterator = CUSTOM_NAME          ← Optional: rename the iterator (default = BLOCK_NAME)
  
  content {                       ← The content of each generated block
    attribute = CUSTOM_NAME.value ← Access the current item
  }
}
```

### Dynamic Block with Iterator

By default, the iterator variable has the same name as the block. You can rename it with `iterator`:

```hcl
resource "aws_security_group" "web" {
  name   = "web-sg"
  vpc_id = aws_vpc.main.id

  dynamic "ingress" {
    for_each = var.ingress_rules
    iterator = rule                       # ← Now use "rule" instead of "ingress"

    content {
      from_port   = rule.value.port       # ← "rule" instead of "ingress"
      to_port     = rule.value.port
      protocol    = rule.value.protocol
      cidr_blocks = rule.value.cidr_blocks
    }
  }
}
```

### Dynamic Blocks with Maps

```hcl
variable "tags_map" {
  type = map(string)
  default = {
    "Environment" = "production"
    "Team"        = "devops"
    "ManagedBy"   = "terraform"
  }
}

resource "aws_autoscaling_group" "web" {
  # ... other config ...

  dynamic "tag" {
    for_each = var.tags_map

    content {
      key                 = tag.key        # ← "Environment", "Team", etc.
      value               = tag.value      # ← "production", "devops", etc.
      propagate_at_launch = true
    }
  }
}
```

### Nested Dynamic Blocks

You can nest dynamic blocks inside each other:

```hcl
variable "load_balancer_listeners" {
  type = list(object({
    port     = number
    protocol = string
    actions = list(object({
      type             = string
      target_group_arn = string
    }))
  }))
}

resource "aws_lb_listener" "front_end" {
  for_each = { for l in var.load_balancer_listeners : l.port => l }

  load_balancer_arn = aws_lb.main.arn
  port              = each.value.port
  protocol          = each.value.protocol

  dynamic "default_action" {
    for_each = each.value.actions

    content {
      type             = default_action.value.type
      target_group_arn = default_action.value.target_group_arn
    }
  }
}
```

> ⚠️ **Warning:** Don't go crazy with nested dynamic blocks! If your code becomes hard to read, consider simplifying your data structure or using modules instead.

## ❓ Conditionals — Create Resources Only When Needed

### Conditional Resource Creation (count = 0 or 1)

The most common pattern: use `count` to **conditionally create** a resource:

```hcl
variable "create_bastion" {
  description = "Whether to create a bastion host"
  type        = bool
  default     = true
}

# This resource is created only if create_bastion is true
resource "aws_instance" "bastion" {
  count = var.create_bastion ? 1 : 0      # ← Ternary: condition ? true_value : false_value

  ami           = "ami-12345678"
  instance_type = "t3.micro"

  tags = {
    Name = "bastion-host"
  }
}
```

**How it works:**
```
var.create_bastion = true   →  count = 1  →  Resource IS created
var.create_bastion = false  →  count = 0  →  Resource is NOT created (doesn't exist!)
```

### Conditional Values (Ternary Operator)

```hcl
variable "environment" {
  type    = string
  default = "dev"
}

resource "aws_instance" "web" {
  ami = "ami-12345678"

  # Use a bigger instance type in production
  instance_type = var.environment == "prod" ? "t3.large" : "t3.micro"

  # Only enable detailed monitoring in production
  monitoring = var.environment == "prod" ? true : false

  # Different subnet based on environment
  subnet_id = var.environment == "prod" ? aws_subnet.prod.id : aws_subnet.dev.id

  tags = {
    Name        = "web-${var.environment}"
    Environment = var.environment
  }
}
```

### Conditional with for_each

```hcl
variable "enable_monitoring" {
  type    = bool
  default = false
}

variable "alarm_configs" {
  type = map(object({
    metric    = string
    threshold = number
  }))
  default = {
    "high-cpu" = {
      metric    = "CPUUtilization"
      threshold = 80
    }
    "high-memory" = {
      metric    = "MemoryUtilization"
      threshold = 90
    }
  }
}

# Only create alarms if monitoring is enabled
resource "aws_cloudwatch_metric_alarm" "alarms" {
  for_each = var.enable_monitoring ? var.alarm_configs : {}   # ← Empty map = no resources!

  alarm_name  = each.key
  metric_name = each.value.metric
  threshold   = each.value.threshold
  # ... other config
}
```

### Conditional Outputs

```hcl
output "bastion_ip" {
  value       = var.create_bastion ? aws_instance.bastion[0].public_ip : "N/A"
  description = "Public IP of the bastion host"
}

# Or using try() for safety
output "bastion_ip_safe" {
  value       = try(aws_instance.bastion[0].public_ip, "No bastion created")
  description = "Public IP of the bastion host"
}
```

## 🔁 for Expressions — Transform Collections

### Basic for Expression (List)

`for` expressions let you **transform** one collection into another:

```hcl
variable "names" {
  type    = list(string)
  default = ["alice", "bob", "charlie"]
}

# Transform to uppercase
locals {
  upper_names = [for name in var.names : upper(name)]
  # Result: ["ALICE", "BOB", "CHARLIE"]
}

# Add a prefix
locals {
  user_emails = [for name in var.names : "${name}@company.com"]
  # Result: ["alice@company.com", "bob@company.com", "charlie@company.com"]
}
```

### for Expression (Map)

Use `=>` to create a map:

```hcl
variable "users" {
  type = list(object({
    name  = string
    role  = string
  }))
  default = [
    { name = "alice",   role = "admin" },
    { name = "bob",     role = "developer" },
    { name = "charlie", role = "developer" }
  ]
}

# Create a map from name → role
locals {
  user_roles = { for user in var.users : user.name => user.role }
  # Result: { "alice" = "admin", "bob" = "developer", "charlie" = "developer" }
}
```

### for Expression with Filtering (if)

Add `if` at the end to filter:

```hcl
variable "instances" {
  type = map(object({
    type        = string
    environment = string
  }))
  default = {
    "web-1"    = { type = "t3.micro", environment = "dev" }
    "web-2"    = { type = "t3.small", environment = "prod" }
    "api-1"    = { type = "t3.micro", environment = "dev" }
    "worker-1" = { type = "t3.large", environment = "prod" }
  }
}

# Get only production instances
locals {
  prod_instances = {
    for name, config in var.instances : name => config
    if config.environment == "prod"
  }
  # Result: { "web-2" = {...}, "worker-1" = {...} }
}

# Get just the names of dev instances
locals {
  dev_instance_names = [
    for name, config in var.instances : name
    if config.environment == "dev"
  ]
  # Result: ["web-1", "api-1"]
}
```

### for Expression with Grouping

Use `...` to group values by key:

```hcl
variable "users" {
  type = list(object({
    name = string
    role = string
  }))
  default = [
    { name = "alice",   role = "admin" },
    { name = "bob",     role = "developer" },
    { name = "charlie", role = "developer" },
    { name = "diana",   role = "admin" }
  ]
}

# Group users by role
locals {
  users_by_role = {
    for user in var.users : user.role => user.name...
  }
  # Result: {
  #   "admin"     = ["alice", "diana"]
  #   "developer" = ["bob", "charlie"]
  # }
}
```

## 🧮 Putting It All Together — Real-World Examples

### Example 1: Multi-Environment VPC with Subnets

```hcl
variable "vpc_config" {
  type = object({
    cidr_block = string
    subnets = map(object({
      cidr        = string
      az          = string
      public      = bool
    }))
  })
  default = {
    cidr_block = "10.0.0.0/16"
    subnets = {
      "public-1"  = { cidr = "10.0.1.0/24", az = "us-east-1a", public = true }
      "public-2"  = { cidr = "10.0.2.0/24", az = "us-east-1b", public = true }
      "private-1" = { cidr = "10.0.3.0/24", az = "us-east-1a", public = false }
      "private-2" = { cidr = "10.0.4.0/24", az = "us-east-1b", public = false }
    }
  }
}

resource "aws_vpc" "main" {
  cidr_block = var.vpc_config.cidr_block
}

# Create all subnets from the map
resource "aws_subnet" "subnets" {
  for_each = var.vpc_config.subnets

  vpc_id                  = aws_vpc.main.id
  cidr_block              = each.value.cidr
  availability_zone       = each.value.az
  map_public_ip_on_launch = each.value.public

  tags = {
    Name   = each.key
    Type   = each.value.public ? "public" : "private"
  }
}

# Create route table associations only for public subnets
resource "aws_route_table_association" "public" {
  for_each = {
    for name, config in var.vpc_config.subnets : name => config
    if config.public == true
  }

  subnet_id      = aws_subnet.subnets[each.key].id
  route_table_id = aws_route_table.public.id
}
```

### Example 2: IAM Users with Policies

```hcl
variable "team_members" {
  type = map(object({
    groups   = list(string)
    policies = list(string)
    tags     = map(string)
  }))
  default = {
    "alice" = {
      groups   = ["admins", "developers"]
      policies = ["arn:aws:iam::policy/AdministratorAccess"]
      tags     = { team = "platform", role = "lead" }
    }
    "bob" = {
      groups   = ["developers"]
      policies = ["arn:aws:iam::policy/PowerUserAccess"]
      tags     = { team = "backend", role = "engineer" }
    }
    "charlie" = {
      groups   = ["developers", "readonly"]
      policies = ["arn:aws:iam::policy/ReadOnlyAccess"]
      tags     = { team = "frontend", role = "engineer" }
    }
  }
}

# Create IAM users
resource "aws_iam_user" "users" {
  for_each = var.team_members
  name     = each.key
  tags     = each.value.tags
}

# Attach policies to users
# We need to "flatten" the map because each user can have multiple policies
locals {
  user_policy_attachments = flatten([
    for user_name, user_config in var.team_members : [
      for policy_arn in user_config.policies : {
        user   = user_name
        policy = policy_arn
      }
    ]
  ])
}

resource "aws_iam_user_policy_attachment" "user_policies" {
  for_each = {
    for attachment in local.user_policy_attachments :
    "${attachment.user}-${attachment.policy}" => attachment
  }

  user       = aws_iam_user.users[each.value.user].name
  policy_arn = each.value.policy
}

# Add users to groups
locals {
  user_group_memberships = flatten([
    for user_name, user_config in var.team_members : [
      for group in user_config.groups : {
        user  = user_name
        group = group
      }
    ]
  ])
}

resource "aws_iam_user_group_membership" "memberships" {
  for_each = {
    for membership in local.user_group_memberships :
    "${membership.user}-${membership.group}" => membership
  }

  user   = aws_iam_user.users[each.value.user].name
  groups = [each.value.group]
}
```

### Example 3: Dynamic Security Group with Conditional Rules

```hcl
variable "environment" {
  type    = string
  default = "dev"
}

variable "enable_ssh" {
  type    = bool
  default = true
}

locals {
  # Base rules that always apply
  base_rules = {
    "http" = {
      port        = 80
      protocol    = "tcp"
      cidr_blocks = ["0.0.0.0/0"]
      description = "HTTP"
    }
    "https" = {
      port        = 443
      protocol    = "tcp"
      cidr_blocks = ["0.0.0.0/0"]
      description = "HTTPS"
    }
  }

  # SSH rule (conditional)
  ssh_rule = var.enable_ssh ? {
    "ssh" = {
      port        = 22
      protocol    = "tcp"
      cidr_blocks = var.environment == "prod" ? ["10.0.0.0/8"] : ["0.0.0.0/0"]
      description = "SSH"
    }
  } : {}

  # Merge all rules together
  all_rules = merge(local.base_rules, local.ssh_rule)
}

resource "aws_security_group" "app" {
  name        = "app-sg-${var.environment}"
  description = "Application security group for ${var.environment}"
  vpc_id      = aws_vpc.main.id

  dynamic "ingress" {
    for_each = local.all_rules

    content {
      from_port   = ingress.value.port
      to_port     = ingress.value.port
      protocol    = ingress.value.protocol
      cidr_blocks = ingress.value.cidr_blocks
      description = ingress.value.description
    }
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}
```

## 🧰 Useful Functions for Working with Collections

### flatten()

Converts nested lists into a single flat list:

```hcl
locals {
  nested = [["a", "b"], ["c", "d"], ["e"]]
  flat   = flatten(local.nested)
  # Result: ["a", "b", "c", "d", "e"]
}
```

### merge()

Combines multiple maps into one:

```hcl
locals {
  default_tags = { ManagedBy = "terraform", Environment = "dev" }
  custom_tags  = { Team = "platform", Application = "web" }
  all_tags     = merge(local.default_tags, local.custom_tags)
  # Result: { ManagedBy = "terraform", Environment = "dev", Team = "platform", Application = "web" }
}
```

### lookup()

Get a value from a map with a default fallback:

```hcl
variable "instance_types" {
  type = map(string)
  default = {
    dev     = "t3.micro"
    staging = "t3.small"
    prod    = "t3.large"
  }
}

locals {
  instance_type = lookup(var.instance_types, var.environment, "t3.micro")
  # If environment is "dev" → "t3.micro"
  # If environment is "prod" → "t3.large"
  # If environment is "unknown" → "t3.micro" (default)
}
```

### try()

Try to evaluate an expression, return a default if it fails:

```hcl
locals {
  # Safe access to potentially missing values
  db_port = try(var.config.database.port, 5432)
  
  # Try multiple expressions in order
  name = try(var.custom_name, var.default_name, "unnamed")
}
```

### coalesce()

Returns the first non-null, non-empty value:

```hcl
locals {
  name = coalesce(var.custom_name, var.default_name, "fallback")
  # Returns the first one that's not null or ""
}
```

### zipmap()

Creates a map from two lists (keys + values):

```hcl
locals {
  keys   = ["name", "age", "city"]
  values = ["Alice", "30", "NYC"]
  result = zipmap(local.keys, local.values)
  # Result: { name = "Alice", age = "30", city = "NYC" }
}
```

## 🚫 Common Mistakes & Gotchas

### 1. count and for_each Can't Be Used Together

```hcl
# ❌ This does NOT work!
resource "aws_instance" "web" {
  count    = 3
  for_each = var.instances   # ERROR! Can't use both!
}
```

### 2. for_each Only Accepts Maps and Sets of Strings

```hcl
# ❌ This does NOT work!
resource "aws_instance" "web" {
  for_each = ["a", "b", "c"]   # Lists are NOT allowed directly!
}

# ✅ Convert list to set
resource "aws_instance" "web" {
  for_each = toset(["a", "b", "c"])   # Use toset()!
}
```

### 3. for_each Keys Must Be Known at Plan Time

```hcl
# ❌ This might NOT work!
resource "aws_instance" "web" {
  for_each = {
    for subnet in aws_subnet.dynamic : subnet.id => subnet
    # subnet.id is not known until AFTER apply! 
  }
}

# ✅ Use values that are known at plan time
resource "aws_instance" "web" {
  for_each = var.subnet_configs   # Variable values are always known at plan time
}
```

### 4. Don't Overuse Dynamic Blocks

```hcl
# ❌ Too complex — hard to read and maintain
resource "aws_thing" "complex" {
  dynamic "block_a" {
    for_each = var.a
    content {
      dynamic "block_b" {
        for_each = block_a.value.items
        content {
          dynamic "block_c" {
            for_each = block_b.value.subitems
            content {
              # 🤯 What is even happening here?
            }
          }
        }
      }
    }
  }
}

# ✅ Better: Use a module or simplify the data structure
```

## 📊 count vs for_each — When to Use Which

| Scenario | Use `count` | Use `for_each` |
|----------|------------|---------------|
| Identical resources | ✅ | ✅ |
| Resources with different configs | ❌ | ✅ |
| Conditional creation (0 or 1) | ✅ | ✅ (empty map) |
| Items might be removed from middle | ❌ | ✅ |
| Need to reference by name | ❌ | ✅ |
| Simple "create N copies" | ✅ | Overkill |

**Rule of thumb:**
- Use **`count`** for: identical resources or conditional (0/1) creation
- Use **`for_each`** for: everything else (especially when items have different configs or might change)

## 🎯 Summary

| Feature | Purpose | Syntax |
|---------|---------|--------|
| **count** | Create N identical copies | `count = 3` |
| **count.index** | Current copy number (0-based) | `count.index` |
| **for_each** | Create resources from map/set | `for_each = var.map` |
| **each.key / each.value** | Current item's key and value | `each.key`, `each.value` |
| **dynamic blocks** | Generate nested blocks dynamically | `dynamic "block" { ... }` |
| **conditionals** | Create/skip resources based on condition | `count = var.x ? 1 : 0` |
| **for expressions** | Transform collections | `[for x in list : ...]` |
| **flatten()** | Flatten nested lists | `flatten([["a"], ["b"]])` |
| **merge()** | Combine maps | `merge(map1, map2)` |
