# 12.15 — Terraform State Manipulation: mv, rm, taint, untaint & More

> **Level:** Advanced  
> **Goal:** Learn how to safely manipulate Terraform state when things go wrong or when you need to reorganize your infrastructure code. Understand `terraform state mv`, `terraform state rm`, `terraform taint`, `terraform untaint`, `terraform state pull/push`, `terraform import`, and when to use each one.

---

## 📌 What is Terraform State?

Before we manipulate state, let's make sure we understand what it is.

**Terraform state** (`terraform.tfstate`) is a JSON file that acts as a **map** between your Terraform code and the real infrastructure in the cloud.

```
Your Code (main.tf)              State File                    Real Cloud Resources
┌─────────────────────┐     ┌─────────────────────┐     ┌─────────────────────┐
│ resource "aws_       │     │ {                   │     │                     │
│   instance" "web" {  │◄───▶│   "aws_instance.   │◄───▶│  EC2 Instance       │
│   ami = "ami-123"    │     │    web": {          │     │  i-0abc123def456    │
│   type = "t3.micro"  │     │     "id": "i-0abc"  │     │  ami-123            │
│ }                    │     │   }                 │     │  t3.micro           │
└─────────────────────┘     └─────────────────────┘     └─────────────────────┘
```

**Think of it this way:**
- Your **code** says _"I want this resource to exist"_
- The **state** says _"This resource already exists, and here's its ID"_
- The **cloud** is where the resource actually lives

When you run `terraform plan`, Terraform:
1. Reads your **code** (what you want)
2. Reads the **state** (what it thinks exists)
3. Checks the **cloud** (what actually exists)
4. Calculates the **difference** and shows you what needs to change

## 🤔 Why Would You Need to Manipulate State?

Here are the most common real-world scenarios:

| Scenario | What You Need |
|----------|--------------|
| Renamed a resource in code | `terraform state mv` |
| Moved a resource into a module | `terraform state mv` |
| Want Terraform to forget a resource (without destroying it) | `terraform state rm` |
| Brought an existing resource under Terraform management | `terraform import` |
| Need to force-recreate a resource | `terraform taint` (or `-replace`) |
| Accidentally tainted a resource | `terraform untaint` |
| Need to inspect the full state | `terraform state pull` |
| Need to manually fix state (dangerous!) | `terraform state push` |
| Need to see all resources in state | `terraform state list` |
| Need details about a specific resource | `terraform state show` |

## 🚚 terraform state mv — Move/Rename Resources in State

This is one of the most useful state commands. It renames a resource in state **without** destroying and recreating it.

### Scenario 1: You Renamed a Resource in Code

You had this:
```hcl
resource "aws_instance" "web_server" {
  ami           = "ami-12345678"
  instance_type = "t3.micro"
}
```

You renamed it to:
```hcl
resource "aws_instance" "app_server" {    # ← Changed the name!
  ami           = "ami-12345678"
  instance_type = "t3.micro"
}
```

**Without `state mv`**, Terraform would:
```
Plan: 1 to add, 0 to change, 1 to destroy.
  - Destroy aws_instance.web_server    (the old name)
  + Create aws_instance.app_server     (the new name)

😱 Your running server would be DESTROYED and a new one created!
```

**With `state mv`**, you tell Terraform: _"Hey, the resource you knew as `web_server` is now called `app_server`. It's the same resource, just a new name."_

```bash
# Move the resource in state
$ terraform state mv aws_instance.web_server aws_instance.app_server

Move "aws_instance.web_server" to "aws_instance.app_server"
Successfully moved 1 object(s).

# Now run plan
$ terraform plan
No changes. Your infrastructure matches the configuration.
# ✅ No destruction! The instance keeps running!
```

### Scenario 2: Moving a Resource INTO a Module

You started with:
```hcl
# main.tf
resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"
}
```

Now you're refactoring to use a module:
```hcl
# main.tf
module "networking" {
  source = "./modules/networking"
}

# modules/networking/main.tf
resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"
}
```

```bash
# Move the resource into the module's state
$ terraform state mv aws_vpc.main module.networking.aws_vpc.main

Move "aws_vpc.main" to "module.networking.aws_vpc.main"
Successfully moved 1 object(s).

$ terraform plan
No changes. ✅
```

### Scenario 3: Moving a Resource OUT of a Module

The reverse — you're removing a module and bringing the resource back to root:

```bash
$ terraform state mv module.networking.aws_vpc.main aws_vpc.main

Move "module.networking.aws_vpc.main" to "aws_vpc.main"
Successfully moved 1 object(s).
```

### Scenario 4: Moving Between Modules

```bash
$ terraform state mv module.old_module.aws_instance.web module.new_module.aws_instance.web
```

### Scenario 5: Renaming a for_each Key

You changed a key in your map:

```hcl
# Before:
variable "instances" {
  default = {
    "old-name" = { ... }
  }
}

# After:
variable "instances" {
  default = {
    "new-name" = { ... }   # Changed the key!
  }
}
```

```bash
# Move the state entry to the new key
$ terraform state mv 'aws_instance.servers["old-name"]' 'aws_instance.servers["new-name"]'
```

> ⚠️ **Note on quoting:** On Linux/Mac, use single quotes around addresses with brackets. On Windows PowerShell, use double quotes and escape the inner quotes.

### Scenario 6: Moving to a Different State File

You can move resources between **completely different** state files (useful for splitting a monolithic state):

```bash
# Move a resource to a different state file
$ terraform state mv -state-out=../other-project/terraform.tfstate \
    aws_instance.web aws_instance.web
```

### Using moved Blocks (Terraform 1.1+) — The Better Way!

Instead of manually running `terraform state mv`, you can use **moved blocks** in your code. This is **declarative** and works in CI/CD too:

```hcl
# Tell Terraform about the rename
moved {
  from = aws_instance.web_server
  to   = aws_instance.app_server
}

resource "aws_instance" "app_server" {
  ami           = "ami-12345678"
  instance_type = "t3.micro"
}
```

```bash
$ terraform plan

Terraform will perform the following actions:

  # aws_instance.web_server has moved to aws_instance.app_server
    resource "aws_instance" "app_server" {
        id            = "i-0abc123def456"
        # (no changes)
    }

Plan: 0 to add, 0 to change, 0 to destroy.
# ✅ Terraform knows it's a rename, not a destroy/create!
```

**moved blocks are better because:**
- They're **version controlled** (in your .tf files)
- They work in **CI/CD** (no manual state commands needed)
- Your **teammates** automatically get the state migration when they pull
- They serve as **documentation** of what changed

## 🗑️ terraform state rm — Remove a Resource from State

`terraform state rm` tells Terraform: _"Forget about this resource. Stop managing it. But don't destroy it in the cloud."_

### When Would You Use This?

| Scenario | Why |
|----------|-----|
| Migrating a resource to a different Terraform project | Remove from old state, import into new state |
| You want to stop managing a resource with Terraform | Resource was manually created and you don't want Terraform touching it |
| A resource was created outside Terraform | And someone accidentally imported it |
| Splitting a monolithic state into smaller states | Remove from the big state, import into smaller ones |

### Basic Usage

```bash
# Remove a single resource
$ terraform state rm aws_instance.web

Removed aws_instance.web
Successfully removed 1 resource instance(s).

# Now if you run plan:
$ terraform plan

  + aws_instance.web will be created
  
Plan: 1 to add, 0 to change, 0 to destroy.

# ⚠️ Terraform wants to CREATE it because it doesn't know it exists!
# If you don't want that, also remove the resource from your .tf code.
```

### Remove a for_each Instance

```bash
# Remove a specific instance from a for_each resource
$ terraform state rm 'aws_instance.servers["app-1"]'

Removed aws_instance.servers["app-1"]
Successfully removed 1 resource instance(s).
```

### Remove an Entire Module

```bash
# Remove all resources in a module
$ terraform state rm module.networking

Removed module.networking.aws_vpc.main
Removed module.networking.aws_subnet.public[0]
Removed module.networking.aws_subnet.public[1]
Successfully removed 3 resource instance(s).
```

### Important: The Resource Still Exists in the Cloud!

```
After terraform state rm:

State File              Cloud
┌──────────┐      ┌──────────────────┐
│           │      │  EC2 Instance    │
│  (empty)  │      │  i-0abc123       │
│           │      │  Still running!  │
└──────────┘      └──────────────────┘

Terraform doesn't know      The resource still
about the resource anymore   exists in AWS
```

**This is the key difference:**
- `terraform destroy` → Removes from state AND destroys in cloud
- `terraform state rm` → Removes from state ONLY (cloud resource untouched)

### Using removed Blocks (Terraform 1.7+) — The Better Way!

Just like `moved`, you can now use a **removed block** instead of the CLI command:

```hcl
# Tell Terraform to stop managing this resource (but don't destroy it)
removed {
  from = aws_instance.old_legacy_server

  lifecycle {
    destroy = false    # ← Don't destroy it, just forget it
  }
}
```

```bash
$ terraform plan

  # aws_instance.old_legacy_server will no longer be managed by Terraform
  
Plan: 0 to add, 0 to change, 0 to destroy.
# ✅ Resource is removed from state but not destroyed!
```

## 💀 terraform taint — Mark a Resource for Recreation

> ⚠️ **Note:** `terraform taint` is **deprecated** as of Terraform 0.15.2. Use `terraform apply -replace` instead. But you should still know about taint because you'll see it in older codebases and documentation.

### What Does Taint Do?

Tainting marks a resource as **"damaged"** or **"needs recreation"**. On the next `terraform apply`, Terraform will **destroy and recreate** that resource.

### When Would You Use This?

| Scenario | Why |
|----------|-----|
| A server's config was manually changed | You want to recreate it from scratch to match your code |
| A resource is in a broken state | Something went wrong and you need a fresh start |
| You need to rotate credentials | Force recreation of a resource that generates credentials |
| Testing your infrastructure | Verify that recreation works smoothly |
| A resource depends on external state | That external thing changed and you need to pick up the change |

### Using taint (Old Way)

```bash
# Mark a resource for recreation
$ terraform taint aws_instance.web

Resource instance aws_instance.web has been marked as tainted.

# Now see what will happen
$ terraform plan

  -/+ aws_instance.web (tainted, so must be replaced)

        ~ ami           = "ami-12345678" -> "ami-12345678"
        ~ id            = "i-0abc123" -> (known after apply)
        ~ instance_type = "t3.micro" -> "t3.micro"
        ...

Plan: 1 to add, 0 to change, 1 to destroy.

# Apply to recreate
$ terraform apply
```

### Using -replace (New Way — Recommended!) ⭐

```bash
# Replace a specific resource on the next apply
$ terraform plan -replace="aws_instance.web"

  # aws_instance.web will be replaced, as requested

  -/+ resource "aws_instance" "web" {
        ~ id            = "i-0abc123" -> (known after apply)
        # ... (forces replacement)
    }

Plan: 1 to add, 0 to change, 1 to destroy.

# Apply with the replacement
$ terraform apply -replace="aws_instance.web"
```

**Why -replace is better than taint:**
- It's a **single command** (no separate taint → apply steps)
- It doesn't modify state until apply is confirmed
- It's **safer** — you see the plan before committing
- It works with `plan` for review before apply
- It's not deprecated!

### Replace with for_each

```bash
# Replace a specific instance
$ terraform apply -replace='aws_instance.web["app-1"]'
```

### Replace Multiple Resources

```bash
# Replace multiple resources at once
$ terraform apply \
  -replace="aws_instance.web" \
  -replace="aws_security_group.web"
```

## ✅ terraform untaint — Cancel a Taint

If you accidentally tainted a resource (or changed your mind), you can untaint it:

```bash
# Oops, I tainted the wrong resource!
$ terraform taint aws_db_instance.production
Resource instance aws_db_instance.production has been marked as tainted.

# Wait, no! That's the production database! 😱
$ terraform untaint aws_db_instance.production
Resource instance aws_db_instance.production has been successfully untainted.

# Phew! Now plan shows no changes
$ terraform plan
No changes. ✅
```

## 📤 terraform state push — Upload State (Dangerous!)

`terraform state push` uploads a local state file to the remote backend. This is **rarely needed** and **very dangerous**.

```bash
# Upload a state file to the remote backend
$ terraform state push terraform.tfstate

# Force push (even if serial number doesn't match)
$ terraform state push -force terraform.tfstate
```

### When Would You Use This?

Honestly? Almost never. But here are some rare cases:

| Scenario | Why |
|----------|-----|
| Restoring from a backup | Your state got corrupted and you have a backup |
| Migrating between backends | Moving from local to S3, or S3 to Terraform Cloud |
| Emergency state repair | Something went very wrong and you need to manually fix state |

> ⚠️ **WARNING:** Using `state push` incorrectly can **permanently lose** your state or cause Terraform to think resources don't exist (leading to recreation). Always make a backup first!

```bash
# ALWAYS backup before pushing state!
$ terraform state pull > state_backup_$(date +%Y%m%d_%H%M%S).json

# Then carefully push
$ terraform state push fixed_state.json
```

## 📦 terraform import — Bring Existing Resources Under Terraform

`terraform import` tells Terraform: _"There's a resource in the cloud that I want you to start managing. Here's its ID."_

### When Would You Use This?

| Scenario | Why |
|----------|-----|
| Migrating from ClickOps to IaC | Resources were created in the AWS console |
| Another team created resources | You want to manage them with Terraform now |
| Disaster recovery | You need to rebuild state from scratch |
| Splitting/merging Terraform projects | Moving resources between state files |

### Step-by-Step Import Process

#### Step 1: Write the Resource Block

You **must** write the resource block in your code first (import doesn't generate code):

```hcl
# main.tf
resource "aws_instance" "existing_server" {
  # We'll fill in the details after import
  ami           = "ami-12345678"
  instance_type = "t3.micro"
}
```

#### Step 2: Run the Import Command

```bash
# terraform import <RESOURCE_ADDRESS> <CLOUD_RESOURCE_ID>
$ terraform import aws_instance.existing_server i-0abc123def456

aws_instance.existing_server: Importing from ID "i-0abc123def456"...
aws_instance.existing_server: Import prepared!
aws_instance.existing_server: Refreshing state...

Import successful!
```

#### Step 3: Update Your Code to Match Reality

```bash
# See what the imported resource looks like
$ terraform state show aws_instance.existing_server

# Update your .tf code to match ALL the attributes
# Then run plan to verify no changes:
$ terraform plan
No changes. ✅    # This means your code matches the real resource!
```

### Import with for_each

```bash
# Import into a for_each resource
$ terraform import 'aws_instance.servers["app-1"]' i-0abc123def456
```

### Import into a Module

```bash
$ terraform import module.networking.aws_vpc.main vpc-abc123
```

### Import Block (Terraform 1.5+) — The Better Way! ⭐

Instead of running CLI commands, you can use **import blocks** in your code:

```hcl
# This tells Terraform to import the resource
import {
  to = aws_instance.existing_server
  id = "i-0abc123def456"
}

resource "aws_instance" "existing_server" {
  ami           = "ami-12345678"
  instance_type = "t3.micro"
  
  tags = {
    Name = "existing-server"
  }
}
```

```bash
$ terraform plan

  # aws_instance.existing_server will be imported
    resource "aws_instance" "existing_server" {
        id            = "i-0abc123def456"
        ami           = "ami-12345678"
        instance_type = "t3.micro"
        ...
    }

Plan: 1 to import, 0 to add, 0 to change, 0 to destroy.
```

### Generate Code from Import (Terraform 1.5+) ⭐⭐

Terraform can now **generate the HCL code** for imported resources automatically:

```bash
# Generate code for the imported resource
$ terraform plan -generate-config-out=generated.tf

# This creates a generated.tf file with the correct HCL!
```

```hcl
# generated.tf (auto-generated!)
resource "aws_instance" "existing_server" {
  ami                         = "ami-12345678"
  instance_type               = "t3.micro"
  subnet_id                   = "subnet-abc123"
  vpc_security_group_ids      = ["sg-def456"]
  associate_public_ip_address = true
  
  tags = {
    Name = "existing-server"
  }
  
  # ... all attributes filled in automatically!
}
```

> **This is amazing** because it saves you from manually writing the resource block and guessing all the attributes!

## 🔧 terraform force-unlock — Release a Stuck State Lock

When someone (or a CI pipeline) crashes while running Terraform, the state lock might get stuck:

```bash
$ terraform plan

Error: Error locking state: Error acquiring the state lock: 
  Lock Info:
    ID:        abc123-def456
    Path:      s3://my-bucket/terraform.tfstate
    Operation: OperationTypePlan
    Who:       user@laptop
    Created:   2024-01-15 10:30:00

Terraform acquires a state lock to protect against two processes
writing to the state file at the same time.
```

### How to Fix It

```bash
# First, make sure nobody else is actually running Terraform!
# If you're sure it's a stuck lock, force-unlock it:

$ terraform force-unlock abc123-def456

Do you really want to force-unlock?
  Terraform will remove the lock on the remote state.
  This will allow local Terraform commands to modify this state, even though it
  may still be in use. Only 'yes' will be accepted to confirm.

  Enter a value: yes

Terraform state has been successfully unlocked!
```

> ⚠️ **Be careful!** Only use this when you're **absolutely sure** no other process is running. If two processes modify state simultaneously, you can corrupt it.

## 🛡️ State Manipulation Safety Checklist

Before doing ANY state manipulation, follow this checklist:

```
✅ 1. BACKUP STATE FIRST!
   $ terraform state pull > backup_$(date +%Y%m%d_%H%M%S).json

✅ 2. Understand what you're doing
   - What resource are you moving/removing?
   - What will happen to the real cloud resource?
   - Does anyone else have pending changes?

✅ 3. Communicate with your team
   - Let people know you're doing state surgery
   - Make sure nobody else is running Terraform

✅ 4. Do a dry run if possible
   - terraform state mv -dry-run SOURCE DEST
   - terraform plan (after the change)

✅ 5. Verify with terraform plan
   - After any state change, run terraform plan
   - It should show NO changes (or only expected changes)

✅ 6. Prefer declarative alternatives
   - Use 'moved' blocks instead of 'terraform state mv'
   - Use 'removed' blocks instead of 'terraform state rm'  
   - Use 'import' blocks instead of 'terraform import'
   - Use '-replace' instead of 'terraform taint'
```

## 🎯 Command Quick Reference

| Command | What It Does | Destructive? |
|---------|-------------|-------------|
| `terraform state list` | List all resources in state | No |
| `terraform state show ADDR` | Show details of a resource | No |
| `terraform state pull` | Download full state as JSON | No |
| `terraform state push FILE` | Upload state (DANGEROUS!) | ⚠️ YES |
| `terraform state mv SRC DEST` | Rename/move a resource in state | Low risk |
| `terraform state rm ADDR` | Remove resource from state (keeps cloud resource) | Medium risk |
| `terraform taint ADDR` | Mark resource for recreation (deprecated) | Medium risk |
| `terraform untaint ADDR` | Cancel a taint | No |
| `terraform apply -replace=ADDR` | Replace a resource (preferred over taint) | Medium risk |
| `terraform import ADDR ID` | Import existing cloud resource into state | Low risk |
| `terraform force-unlock ID` | Release a stuck state lock | Low risk |

## 🔄 Real-World Workflows

### Workflow 1: Refactoring — Renaming Resources

```bash
# Situation: You renamed "web_server" to "app_server" in your code

# Step 1: Backup
terraform state pull > backup.json

# Step 2: Move in state
terraform state mv aws_instance.web_server aws_instance.app_server

# Step 3: Verify
terraform plan
# Should show: No changes. ✅
```

**Or better — use moved blocks:**

```hcl
moved {
  from = aws_instance.web_server
  to   = aws_instance.app_server
}
```

### Workflow 2: Splitting a Monolith State

```bash
# Situation: Your single state file manages EVERYTHING. 
# You want to split networking into a separate project.

# Step 1: In the NEW project, write the resource blocks + import blocks

# Step 2: In the new project, run:
terraform plan    # Shows imports
terraform apply   # Imports the resources

# Step 3: In the OLD project, add removed blocks:
# removed { from = aws_vpc.main; lifecycle { destroy = false } }
terraform apply   # Removes from old state without destroying

# Step 4: Verify both states
# OLD project: terraform plan → No changes ✅
# NEW project: terraform plan → No changes ✅
```

### Workflow 3: Recovering from a Botched Apply

```bash
# Situation: terraform apply partially failed. Some resources were created, 
# others weren't. State is in a weird half-applied state.

# Step 1: DON'T PANIC! Check what's in state
terraform state list

# Step 2: Check what's in the cloud (use AWS CLI)
aws ec2 describe-instances --filters "Name=tag:Name,Values=my-server"

# Step 3: If a resource exists in cloud but not in state, import it
terraform import aws_instance.web i-0abc123

# Step 4: If a resource is in state but not in cloud, remove it from state
terraform state rm aws_instance.failed_resource

# Step 5: Run plan to see where things stand
terraform plan

# Step 6: Apply to reach desired state
terraform apply
```

### Workflow 4: Force-Recreating a Problematic Instance

```bash
# Situation: An EC2 instance is behaving strangely. You want to 
# recreate it from scratch.

# Option A: Using -replace (recommended)
terraform apply -replace="aws_instance.web"

# Option B: Using taint (deprecated but still works)
terraform taint aws_instance.web
terraform apply
```

## 🎯 Summary

| Concept | Key Takeaway |
|---------|-------------|
| **State** | Map between your code and real cloud resources |
| **state list** | See all tracked resources |
| **state show** | Inspect a specific resource's details |
| **state mv** | Rename/move resources without destroying them |
| **state rm** | Remove from state WITHOUT destroying in cloud |
| **taint/-replace** | Force recreation of a resource |
| **import** | Bring existing cloud resources under Terraform management |
| **state pull/push** | Download/upload raw state (push is dangerous!) |
| **moved blocks** | Declarative alternative to `state mv` (preferred!) |
| **removed blocks** | Declarative alternative to `state rm` (preferred!) |
| **import blocks** | Declarative alternative to `terraform import` (preferred!) |
| **Safety** | ALWAYS backup state before manipulation! |
