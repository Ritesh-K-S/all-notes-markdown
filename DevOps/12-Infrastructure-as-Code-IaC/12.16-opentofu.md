## 1. What is OpenTofu?

**OpenTofu** is a **free, open-source** Infrastructure as Code (IaC) tool. Think of it as Terraform's twin — it works almost exactly the same way, uses the same language (HCL), the same providers, and the same workflow. But there's one crucial difference: **OpenTofu is truly open-source** under the **MPL 2.0 license** (Mozilla Public License) and is managed by the **Linux Foundation**, not a single company.

### In Simple Words:

Imagine you've been using a recipe book (Terraform) that was free for everyone. One day, the publisher says, "Hey, this book is no longer fully free — some recipes are now behind a paywall." The community says, "No thanks, we'll copy the free version and keep it open forever." That copy is **OpenTofu**.

### Key Facts at a Glance

| Feature | Details |
|---------|---------|
| **Full Name** | OpenTofu |
| **Forked From** | Terraform v1.5.x (the last truly open-source version) |
| **License** | MPL 2.0 (truly open-source) |
| **Governed By** | Linux Foundation |
| **Language** | HCL (HashiCorp Configuration Language) |
| **First Stable Release** | January 2024 |
| **Website** | opentofu.org |

## 2. Why Was OpenTofu Created? — The Backstory

This is an important story because it teaches you about **open-source licensing** and why it matters in the DevOps world.

### The Timeline

**Before August 2023:**
- Terraform was licensed under **MPL 2.0** (open-source).
- Anyone could use it, modify it, build products on top of it — no restrictions.
- Companies like Spacelift, env0, Scalr built their businesses around Terraform.

**August 10, 2023 — The License Change:**
- HashiCorp (the company that created Terraform) changed Terraform's license from **MPL 2.0** to **BSL 1.1** (Business Source License).
- BSL is **NOT** an open-source license. It restricts who can use Terraform commercially — specifically, you can't build competing products with it.

**September 2023 — The Community Responds:**
- Over 140 companies and 800+ individuals signed the **OpenTF Manifesto** asking HashiCorp to reverse the change.
- HashiCorp said no.
- The community forked Terraform's last open-source version (v1.5.7) and created **OpenTofu**.

**September 20, 2023:**
- OpenTofu was accepted into the **Linux Foundation** — giving it vendor-neutral governance.

**January 2024:**
- **OpenTofu 1.6.0** — first stable release.

### Why Does This Matter to You?

- If you're learning IaC today, **you have a choice** between Terraform and OpenTofu.
- Everything you learn about Terraform works with OpenTofu (and vice versa).
- OpenTofu guarantees it will **always** be free and open-source.

## 3. OpenTofu vs Terraform — What's Different?

### What's the SAME?

Almost everything! Here's what works identically:

| Feature | Same? |
|---------|-------|
| HCL language syntax | ✅ Yes |
| Provider ecosystem (AWS, Azure, GCP, etc.) | ✅ Yes |
| State files & state management | ✅ Yes |
| Modules | ✅ Yes |
| CLI workflow (init, plan, apply, destroy) | ✅ Yes |
| Variables, outputs, locals | ✅ Yes |
| Workspaces | ✅ Yes |
| Remote backends (S3, GCS, etc.) | ✅ Yes |

### What's DIFFERENT?

| Feature | Terraform | OpenTofu |
|---------|-----------|----------|
| **License** | BSL 1.1 (not open-source) | MPL 2.0 (open-source) |
| **Governance** | HashiCorp (single company) | Linux Foundation (community) |
| **CLI binary** | `terraform` | `tofu` |
| **State Encryption** | ❌ Not built-in | ✅ Built-in (since 1.7) |
| **Registry** | registry.terraform.io | registry.opentofu.org (+ mirrors Terraform registry) |
| **Provider-defined Functions** | Added later | ✅ Added early |
| **Early Variable/Locals Evaluation** | ❌ | ✅ (since 1.8) |
| **`.tofu` files** | ❌ | ✅ Supported alongside `.tf` |

### The Big Takeaway

If you know Terraform, **you already know OpenTofu**. The switch is mostly about replacing the `terraform` command with `tofu`.

## 4. Installing OpenTofu

### On Linux (Debian/Ubuntu)

```bash
# Add the OpenTofu repository
curl --proto '=https' --tlsv1.2 -fsSL https://get.opentofu.org/install-opentofu.sh -o install-opentofu.sh

# Inspect the script (always a good practice!)
chmod +x install-opentofu.sh

# Install using the standalone method
./install-opentofu.sh --install-method standalone

# Verify installation
tofu --version
```

### On macOS (Homebrew)

```bash
brew install opentofu
tofu --version
```

### On Windows (Chocolatey)

```powershell
choco install opentofu
tofu --version
```

### On Windows (Scoop)

```powershell
scoop bucket add main
scoop install opentofu
tofu --version
```

### Using Docker

```bash
docker run --rm -it ghcr.io/opentofu/opentofu:latest --version
```

### Verify It Works

```bash
$ tofu --version
OpenTofu v1.8.x
on linux_amd64
```

> **Note:** The `tofu` command replaces the `terraform` command. Everything else stays the same!

## 5. Your First OpenTofu Project

Let's build something real! We'll create a simple project that creates a local file (no cloud account needed).

### Step 1: Create a Project Directory

```bash
mkdir my-first-opentofu
cd my-first-opentofu
```

### Step 2: Create the Main Configuration File

Create a file called `main.tf`:

```hcl
# main.tf

# Tell OpenTofu which provider we want to use
# The "local" provider lets us work with local files
terraform {
  required_providers {
    local = {
      source  = "hashicorp/local"
      version = "~> 2.4"
    }
  }
  required_version = ">= 1.6.0"
}

# Create a local file on your machine
resource "local_file" "hello" {
  content  = "Hello from OpenTofu! 🎉\nThis file was created by Infrastructure as Code."
  filename = "${path.module}/hello.txt"
}

# Output the file path so we can see it
output "file_path" {
  value       = local_file.hello.filename
  description = "Path to the created file"
}
```

**What's happening here?**
- `terraform { }` block — Yes, it still says `terraform`! OpenTofu is backward-compatible, so it reads the same block names.
- `required_providers` — We're using the "local" provider (to create local files).
- `resource "local_file" "hello"` — This creates a file called `hello.txt` with some content.
- `output "file_path"` — This prints the file path after creation.

### Step 3: Initialize the Project

```bash
tofu init
```

**What does `tofu init` do?**
- Downloads the providers you specified (in this case, the `local` provider).
- Sets up the `.terraform` directory with plugins.
- Creates `.terraform.lock.hcl` to lock provider versions.

**Output:**

```
Initializing the backend...
Initializing provider plugins...
- Finding hashicorp/local versions matching "~> 2.4"...
- Installing hashicorp/local v2.4.1...
- Installed hashicorp/local v2.4.1 (signed, key ID 0C0AF313E5FD9F80)

OpenTofu has been successfully initialized!
```

### Step 4: Preview the Changes

```bash
tofu plan
```

**What does `tofu plan` do?**
- Shows you what OpenTofu WILL do — without actually doing it.
- Think of it as a "dry run" or a "preview."

**Output:**

```
OpenTofu will perform the following actions:

  # local_file.hello will be created
  + resource "local_file" "hello" {
      + content              = "Hello from OpenTofu! 🎉\nThis file was created by Infrastructure as Code."
      + filename             = "./hello.txt"
      + id                   = (known after apply)
    }

Plan: 1 to add, 0 to change, 0 to destroy.
```

The `+` sign means "this resource will be CREATED." 

### Step 5: Apply the Changes

```bash
tofu apply
```

OpenTofu will show you the plan again and ask for confirmation:

```
Do you want to perform these actions?
  OpenTofu will perform the actions described above.
  Only 'yes' will be accepted to approve.

  Enter a value: yes
```

Type `yes` and press Enter.

```
local_file.hello: Creating...
local_file.hello: Creation complete after 0s [id=abc123...]

Apply complete! Resources: 1 added, 0 changed, 0 destroyed.

Outputs:

file_path = "./hello.txt"
```

### Step 6: Check the Result

```bash
cat hello.txt
```

Output:

```
Hello from OpenTofu! 🎉
This file was created by Infrastructure as Code.
```

**It worked!** 🎉 You just used OpenTofu to manage infrastructure (in this case, a file).

### Step 7: Clean Up

```bash
tofu destroy
```

This removes everything OpenTofu created. Type `yes` to confirm.

## 6. Key Features of OpenTofu

### 6.1 — State Encryption (The Big One!)

This is OpenTofu's **biggest unique feature**. In Terraform, your state file (which contains ALL your infrastructure details — sometimes including passwords and secrets) is stored **in plain text**. OpenTofu lets you **encrypt** it.

We'll cover this in detail in the next section.

### 6.2 — Early Variable and Locals Evaluation

In Terraform, you can't use variables in `backend` blocks or `module` sources. This is a huge annoyance! OpenTofu fixed this starting in version 1.8:

```hcl
# This DOESN'T work in Terraform, but DOES work in OpenTofu 1.8+
variable "environment" {
  default = "dev"
}

terraform {
  backend "s3" {
    bucket = "my-state-${var.environment}"    # ← Works in OpenTofu!
    key    = "terraform.tfstate"
    region = "us-east-1"
  }
}
```

This is a game-changer for managing multiple environments.

### 6.3 — Provider-Defined Functions

OpenTofu allows providers to ship their own custom functions that you can use in your HCL code:

```hcl
# Example: Using a provider-defined function
output "decoded" {
  value = provider::aws::arn_parse("arn:aws:iam::123456789012:role/my-role")
}
```

### 6.4 — `.tofu` File Support

OpenTofu reads both `.tf` and `.tofu` files. If both exist, `.tofu` files take precedence. This lets you have OpenTofu-specific overrides without breaking Terraform compatibility:

```
my-project/
├── main.tf          # Shared configuration
├── variables.tf     # Shared variables
├── backend.tofu     # OpenTofu-specific backend (with variables!)
└── providers.tf     # Shared providers
```

### 6.5 — Removed Blocks (for Refactoring)

When you want to remove a resource from your code without destroying it in the cloud:

```hcl
# This tells OpenTofu: "I removed this from code, but DON'T destroy it"
removed {
  from = aws_instance.old_server

  lifecycle {
    destroy = false
  }
}
```

## 7. State Encryption — OpenTofu's Killer Feature

### Why State Encryption Matters

Your state file (`terraform.tfstate`) contains **everything** about your infrastructure:
- Resource IDs
- IP addresses
- Database passwords (if passed as variables)
- API keys
- ARNs, connection strings, etc.

In Terraform, this file is **plain text JSON**. If someone gets access to it, they know EVERYTHING about your infrastructure.

OpenTofu's **built-in state encryption** solves this.

### How to Enable State Encryption

Add an `encryption` block to your OpenTofu configuration:

```hcl
# encryption.tf

terraform {
  encryption {
    ## Define the key provider — where the encryption key comes from
    key_provider "pbkdf2" "my_passphrase" {
      passphrase = var.state_passphrase
    }

    ## Encrypt the state file
    method "aes_gcm" "my_method" {
      keys = key_provider.pbkdf2.my_passphrase
    }

    state {
      method = method.aes_gcm.my_method
    }

    ## Also encrypt plan files
    plan {
      method = method.aes_gcm.my_method
    }
  }
}

variable "state_passphrase" {
  type      = string
  sensitive = true
}
```

**What's happening here?**

1. **`key_provider "pbkdf2"`** — Uses a passphrase to derive an encryption key. PBKDF2 is a secure key derivation function.
2. **`method "aes_gcm"`** — Uses AES-GCM encryption (industry-standard, very secure).
3. **`state { method = ... }`** — Tells OpenTofu to encrypt the state file.
4. **`plan { method = ... }`** — Also encrypts plan files.

### Available Key Providers

| Key Provider | Description | Use Case |
|-------------|-------------|----------|
| `pbkdf2` | Password-based key derivation | Simple setups, local dev |
| `aws_kms` | AWS Key Management Service | Production on AWS |
| `gcp_kms` | Google Cloud KMS | Production on GCP |
| `openbao` | OpenBao (open-source Vault fork) | Self-hosted secret management |

### Example: Using AWS KMS for Encryption

```hcl
terraform {
  encryption {
    key_provider "aws_kms" "main" {
      kms_key_id = "arn:aws:kms:us-east-1:123456789012:key/abcd-1234-efgh-5678"
      region     = "us-east-1"
      key_spec   = "AES_256"
    }

    method "aes_gcm" "main" {
      keys = key_provider.aws_kms.main
    }

    state {
      method = method.aes_gcm.main
    }

    plan {
      method = method.aes_gcm.main
    }
  }
}
```

### What Encrypted State Looks Like

Before encryption (plain text):
```json
{
  "version": 4,
  "resources": [
    {
      "type": "aws_instance",
      "name": "web",
      "instances": [
        {
          "attributes": {
            "ami": "ami-0c55b159cbfafe1f0",
            "private_ip": "10.0.1.50"
          }
        }
      ]
    }
  ]
}
```

After encryption:
```json
{
  "serial": 1,
  "lineage": "abc-123",
  "encrypted_data": "U2FsdGVkX1+AAAABBBBCCCC... (unreadable encrypted blob)"
}
```

Nobody can read your state without the encryption key! 🔐

## 8. Migrating from Terraform to OpenTofu

### The Good News

Migration is **surprisingly easy** because OpenTofu was forked from Terraform v1.5.x.

### Step-by-Step Migration

#### Step 1: Check Your Terraform Version

```bash
terraform version
```

If you're on Terraform 1.5.x or earlier, migration is seamless. If you're on 1.6+, there might be some differences (but OpenTofu has been adding parity features).

#### Step 2: Install OpenTofu

(See installation section above)

#### Step 3: Replace the Binary

The simplest approach — just use `tofu` instead of `terraform`:

```bash
# Before:
terraform init
terraform plan
terraform apply

# After:
tofu init
tofu plan
tofu apply
```

#### Step 4: Re-initialize

```bash
cd your-project
tofu init
```

OpenTofu will re-download providers and set up the `.terraform` directory.

#### Step 5: Verify State

```bash
tofu plan
```

If the plan shows **"No changes. Your infrastructure matches the configuration."** — you're done! The migration was successful.

### Migration Helper Script

Here's a simple bash script to help:

```bash
#!/bin/bash
# migrate-to-opentofu.sh

echo "🔄 Starting migration from Terraform to OpenTofu..."

# Check if tofu is installed
if ! command -v tofu &> /dev/null; then
    echo "❌ OpenTofu is not installed. Install it first!"
    exit 1
fi

# Check OpenTofu version
echo "📦 OpenTofu version: $(tofu --version | head -1)"

# Remove old .terraform directory (providers will be re-downloaded)
echo "🗑️  Removing old .terraform directory..."
rm -rf .terraform
rm -f .terraform.lock.hcl

# Initialize with OpenTofu
echo "🔧 Running tofu init..."
tofu init

# Verify
echo "🔍 Running tofu plan to verify..."
tofu plan

echo "✅ Migration check complete!"
echo "If the plan shows no changes, migration is successful!"
```

### Things to Watch Out For

| Issue | Solution |
|-------|----------|
| `.terraform.lock.hcl` differences | Delete it and run `tofu init` to regenerate |
| Terraform Cloud backend | OpenTofu doesn't support Terraform Cloud (use other backends) |
| BSL-licensed providers | Most providers are still MPL-licensed and work fine |
| Wrapper tools (Terragrunt, etc.) | Most now support OpenTofu natively |

## 9. Provider and Registry Support

### How Do Providers Work?

OpenTofu uses the same providers as Terraform. Providers are plugins that let OpenTofu talk to cloud APIs (AWS, Azure, GCP) or other services.

### The OpenTofu Registry

OpenTofu has its own registry at **registry.opentofu.org** that mirrors the Terraform registry:

```hcl
# This works exactly the same way
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"     # Same source!
      version = "~> 5.0"
    }
    kubernetes = {
      source  = "hashicorp/kubernetes"
      version = "~> 2.25"
    }
  }
}
```

### Popular Providers (All Work with OpenTofu)

| Provider | Source | Purpose |
|----------|--------|---------|
| AWS | `hashicorp/aws` | Amazon Web Services |
| Azure | `hashicorp/azurerm` | Microsoft Azure |
| GCP | `hashicorp/google` | Google Cloud Platform |
| Kubernetes | `hashicorp/kubernetes` | K8s resources |
| Helm | `hashicorp/helm` | Helm charts |
| Docker | `kreuzwerker/docker` | Docker containers |
| GitHub | `integrations/github` | GitHub repos, teams |
| Cloudflare | `cloudflare/cloudflare` | DNS, CDN |
| Datadog | `DataDog/datadog` | Monitoring |

## 10. OpenTofu CLI Commands Reference

Here's a quick reference of all the commands. They work exactly like Terraform's commands:

### Core Workflow Commands

```bash
# Initialize a project (download providers, set up backend)
tofu init

# Preview changes (dry run)
tofu plan

# Apply changes (make it happen!)
tofu apply

# Destroy everything
tofu destroy
```

### State Management Commands

```bash
# List all resources in state
tofu state list

# Show details of a specific resource
tofu state show aws_instance.web

# Move a resource (rename in state)
tofu state mv aws_instance.old aws_instance.new

# Remove a resource from state (without destroying it)
tofu state rm aws_instance.web

# Pull remote state to local
tofu state pull

# Push local state to remote
tofu state push
```

### Other Useful Commands

```bash
# Format code nicely
tofu fmt

# Validate configuration files
tofu validate

# Show the current state
tofu show

# Create a visual graph of resources
tofu graph | dot -Tpng > graph.png

# Show outputs
tofu output

# Import existing infrastructure
tofu import aws_instance.web i-1234567890abcdef0

# Mark a resource for recreation
tofu taint aws_instance.web     # deprecated, use -replace
tofu apply -replace="aws_instance.web"

# Show providers in use
tofu providers

# Upgrade provider versions
tofu init -upgrade
```

### Auto-Approve (Skip Confirmation)

```bash
# Use in CI/CD — skips the "yes" prompt
tofu apply -auto-approve
tofu destroy -auto-approve
```

> **⚠️ Warning:** Only use `-auto-approve` in automated pipelines, never in manual operations!

## 11. Using OpenTofu with CI/CD

### GitHub Actions Example

```yaml
# .github/workflows/opentofu.yml
name: OpenTofu CI/CD

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  plan:
    name: "OpenTofu Plan"
    runs-on: ubuntu-latest
    
    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Setup OpenTofu
        uses: opentofu/setup-opentofu@v1
        with:
          tofu_version: "1.8.0"

      - name: OpenTofu Init
        run: tofu init

      - name: OpenTofu Format Check
        run: tofu fmt -check

      - name: OpenTofu Validate
        run: tofu validate

      - name: OpenTofu Plan
        run: tofu plan -out=tfplan
        env:
          AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
          AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}

  apply:
    name: "OpenTofu Apply"
    needs: plan
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    
    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Setup OpenTofu
        uses: opentofu/setup-opentofu@v1

      - name: OpenTofu Init
        run: tofu init

      - name: OpenTofu Apply
        run: tofu apply -auto-approve
        env:
          AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
          AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
```

**What this pipeline does:**
1. On pull requests: runs `plan` to show what would change.
2. On push to `main`: runs `apply` to make the changes.

## 12. Real-World Example — AWS Infrastructure with OpenTofu

Let's create a proper AWS project with VPC, subnet, and an EC2 instance.

### Project Structure

```
aws-infra/
├── main.tf           # Main resources
├── variables.tf      # Input variables
├── outputs.tf        # Output values
├── providers.tf      # Provider configuration
└── terraform.tfvars  # Variable values
```

### providers.tf

```hcl
terraform {
  required_version = ">= 1.6.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }

  # Store state in S3
  backend "s3" {
    bucket = "my-opentofu-state"
    key    = "prod/terraform.tfstate"
    region = "us-east-1"
  }
}

provider "aws" {
  region = var.aws_region

  default_tags {
    tags = {
      ManagedBy   = "OpenTofu"
      Environment = var.environment
      Project     = var.project_name
    }
  }
}
```

### variables.tf

```hcl
variable "aws_region" {
  description = "AWS region to deploy resources"
  type        = string
  default     = "us-east-1"
}

variable "environment" {
  description = "Environment name (dev, staging, prod)"
  type        = string
  default     = "dev"
}

variable "project_name" {
  description = "Name of the project"
  type        = string
  default     = "my-opentofu-project"
}

variable "instance_type" {
  description = "EC2 instance type"
  type        = string
  default     = "t3.micro"
}

variable "vpc_cidr" {
  description = "CIDR block for the VPC"
  type        = string
  default     = "10.0.0.0/16"
}
```

### main.tf

```hcl
# --------------------------------------------------
# VPC
# --------------------------------------------------
resource "aws_vpc" "main" {
  cidr_block           = var.vpc_cidr
  enable_dns_hostnames = true
  enable_dns_support   = true

  tags = {
    Name = "${var.project_name}-vpc"
  }
}

# --------------------------------------------------
# Public Subnet
# --------------------------------------------------
resource "aws_subnet" "public" {
  vpc_id                  = aws_vpc.main.id
  cidr_block              = "10.0.1.0/24"
  availability_zone       = "${var.aws_region}a"
  map_public_ip_on_launch = true

  tags = {
    Name = "${var.project_name}-public-subnet"
  }
}

# --------------------------------------------------
# Internet Gateway (allows internet access)
# --------------------------------------------------
resource "aws_internet_gateway" "igw" {
  vpc_id = aws_vpc.main.id

  tags = {
    Name = "${var.project_name}-igw"
  }
}

# --------------------------------------------------
# Route Table (routes traffic to the internet)
# --------------------------------------------------
resource "aws_route_table" "public" {
  vpc_id = aws_vpc.main.id

  route {

data "aws_ami" "amazon_linux" {

    name   = "name"

resource "aws_instance" "web" {
```

## 14. When to Choose OpenTofu vs Terraform

### Choose OpenTofu If:

- ✅ You want **truly open-source** IaC
- ✅ You need **state encryption** built-in
- ✅ You want variables in **backend blocks**
- ✅ You're worried about vendor lock-in
- ✅ Your organization has open-source mandates
- ✅ You're starting fresh and want future flexibility

### Choose Terraform If:

- ✅ You need **Terraform Cloud** or **Terraform Enterprise**
- ✅ You have existing Terraform Cloud workflows
- ✅ Your team is already standardized on Terraform
- ✅ You need specific Terraform-only features (like Sentinel policies)

### The Balanced View

Both tools are excellent. Since they share the same foundation, **you can switch between them easily**. Learning one means you've learned both.

## 15. Common Issues and Troubleshooting

### Issue: "Provider not found"

```
Error: Failed to query available provider packages
```

**Fix:** Make sure you've run `tofu init` or check your `required_providers` block.

### Issue: State Lock Error

```
Error: Error acquiring the state lock
```

**Fix:**

```bash
# Force unlock (use with caution!)
tofu force-unlock LOCK_ID
```

### Issue: Version Mismatch

```
Error: This configuration requires OpenTofu >= 1.8.0
```

**Fix:** Upgrade OpenTofu to the required version.

### Issue: `.terraform.lock.hcl` Conflicts After Migration

**Fix:**

```bash
rm .terraform.lock.hcl
tofu init
```

| **What** | OpenTofu is an open-source fork of Terraform |

> **OpenTofu = Terraform, but truly open-source, with state encryption, managed by the Linux Foundation.**
