## 1. What is AWS CloudFormation?

**AWS CloudFormation** is Amazon's **native Infrastructure as Code** service. It lets you define your entire AWS infrastructure in a **template file** (written in JSON or YAML), and CloudFormation creates, updates, and deletes everything for you.

### Simple Analogy

Think of CloudFormation like ordering furniture from IKEA:

1. **Template** = The IKEA catalog order (what you want)
2. **Stack** = Your actual assembled furniture (what you have)
3. **CloudFormation Service** = The IKEA delivery and assembly team (does the work)

You tell CloudFormation **WHAT** you want (declarative), and it figures out **HOW** to create it.

### Key Facts

| Feature | Details |
|---------|---------|
| **Created By** | Amazon Web Services |
| **Type** | Fully managed AWS service |
| **Cost** | **Free!** (you only pay for the resources it creates) |
| **Template Format** | YAML or JSON |
| **Language** | Declarative (you describe desired state) |
| **Scope** | AWS only (that's both its strength and limitation) |
| **Alternative** | AWS CDK (writes code that generates CloudFormation) |

## 2. Why CloudFormation? — When to Use It

### Advantages

| Advantage | Why It Matters |
|-----------|---------------|
| **Native AWS integration** | First-class support for every AWS service, often on launch day |
| **No extra tools needed** | Built into AWS — nothing to install |
| **Free to use** | No licensing costs |
| **Automatic rollback** | If something fails, CloudFormation undoes everything |
| **Drift detection** | Tells you if someone changed things manually |
| **StackSets** | Deploy to 100+ accounts/regions at once |
| **Deep AWS support** | Supports resources that Terraform doesn't yet |

### When to Use CloudFormation

- ✅ You're **100% on AWS** (not multi-cloud)
- ✅ You want **native integration** with AWS services
- ✅ You need **StackSets** for multi-account deployments
- ✅ Your organization mandates AWS-native tools
- ✅ You want **automatic rollback** on failures
- ✅ You need new AWS features on day one

### When NOT to Use CloudFormation

- ❌ You need **multi-cloud** support (use Terraform or Pulumi instead)
- ❌ You find YAML/JSON templates too verbose (consider AWS CDK)
- ❌ You need complex logic (loops, conditions are limited in CloudFormation)

## 3. CloudFormation vs Terraform — Comparison

| Feature | CloudFormation | Terraform |
|---------|---------------|-----------|
| **Cloud Support** | AWS only | Multi-cloud (AWS, Azure, GCP, etc.) |
| **Language** | YAML / JSON | HCL |
| **State Management** | Managed by AWS (automatic!) | You manage it (S3, local, etc.) |
| **Rollback** | Automatic on failure | Manual |
| **Preview Changes** | Change Sets | `terraform plan` |
| **Drift Detection** | Built-in | `terraform plan` shows drift |
| **Multi-Account** | StackSets (amazing!) | Needs extra tooling |
| **New AWS Features** | Day-one support | Some delay |
| **Modularity** | Nested Stacks, Modules | Modules |
| **Cost** | Free | Free (open-source CLI) |
| **Learning Curve** | Moderate (verbose YAML) | Moderate (HCL) |
| **Max Template Size** | 1 MB | No limit |

### The Verdict

- If you're **AWS-only** → CloudFormation is a great choice.
- If you're **multi-cloud** → Terraform/OpenTofu is better.
- If you want **programming power** → Pulumi or AWS CDK.

## 4. Core Concepts — The Building Blocks

Let's understand the main concepts before diving into code:

### 4.1 — Template

A **template** is a YAML or JSON file that describes your desired infrastructure. Think of it as a **blueprint**.

```yaml
# This is a template — a blueprint for infrastructure
AWSTemplateFormatVersion: "2010-09-09"
Description: My first template
Resources:
  MyBucket:
    Type: AWS::S3::Bucket
```

### 4.2 — Stack

A **stack** is a **running instance** of a template. When you deploy a template, CloudFormation creates a stack.

```
Template (blueprint)  →  Stack (actual resources)
     main.yaml       →  "my-app-dev" stack
                     →  "my-app-prod" stack   (same template, different stack!)
```

### 4.3 — Resource

A **resource** is a single AWS thing — an S3 bucket, an EC2 instance, a VPC, etc.

### 4.4 — Stack Events

Every action CloudFormation takes is logged as an **event**. You can see:
- What resource is being created
- Whether it succeeded or failed
- Error messages if something went wrong

### 4.5 — Change Set

A **change set** is a preview of what will happen before you update a stack. Like `terraform plan`.

### 4.6 — StackSet

A **StackSet** lets you deploy the same template to **multiple AWS accounts and regions** at once. Incredibly powerful for enterprise setups.

## 5. CloudFormation Templates — Deep Dive

### YAML vs JSON

CloudFormation templates can be written in either YAML or JSON. **YAML is strongly recommended** because it's more readable and supports comments.

**YAML (recommended):**

```yaml
Resources:
  MyBucket:
    Type: AWS::S3::Bucket
    Properties:
      BucketName: my-awesome-bucket
      # This is a comment — YAML supports comments!
```

**JSON (verbose, no comments):**

```json
{
  "Resources": {
    "MyBucket": {
      "Type": "AWS::S3::Bucket",
      "Properties": {
        "BucketName": "my-awesome-bucket"
      }
    }
  }
}
```

**See why everyone uses YAML?** 😄

## 6. Template Anatomy — Every Section Explained

A CloudFormation template has several sections. Let's look at ALL of them:

```yaml
# =====================================================
# Full CloudFormation Template Structure
# =====================================================

# 1. FORMAT VERSION (optional but recommended)
AWSTemplateFormatVersion: "2010-09-09"

# 2. DESCRIPTION (optional — what does this template do?)
Description: >
  This template creates a web application infrastructure
  with VPC, subnets, and EC2 instances.

# 3. METADATA (optional — extra info for tools)
Metadata:
  AWS::CloudFormation::Interface:
    ParameterGroups:
      - Label:
          default: "Network Configuration"
        Parameters:
          - VpcCidr
          - SubnetCidr

# 4. PARAMETERS (optional — user inputs)
Parameters:
  EnvironmentType:
    Type: String
    Default: dev
    AllowedValues:
      - dev
      - staging
      - prod
    Description: The environment type

# 5. MAPPINGS (optional — lookup tables)
Mappings:
  RegionMap:
    us-east-1:
      AMI: ami-0c55b159cbfafe1f0
    us-west-2:
      AMI: ami-0892d3c7ee96c0bf7

# 6. CONDITIONS (optional — if/else logic)
Conditions:
  IsProduction: !Equals [!Ref EnvironmentType, prod]

# 7. TRANSFORM (optional — for macros like SAM)
# Transform: AWS::Serverless-2016-10-31

# 8. RESOURCES (required — the main section!)
Resources:
  MyBucket:
    Type: AWS::S3::Bucket
    Properties:
      BucketName: !Sub "${AWS::StackName}-bucket"

# 9. OUTPUTS (optional — values to display/export)
Outputs:
  BucketArn:
    Value: !GetAtt MyBucket.Arn
    Description: The ARN of the S3 bucket
    Export:
      Name: !Sub "${AWS::StackName}-bucket-arn"
```

### Section Summary

| Section | Required? | Purpose |
|---------|-----------|---------|
| `AWSTemplateFormatVersion` | No | Template version (always "2010-09-09") |
| `Description` | No | Human-readable description |
| `Metadata` | No | Extra info for tools and UI |
| `Parameters` | No | User inputs (like variables) |
| `Mappings` | No | Static lookup tables |
| `Conditions` | No | If/else logic |
| `Transform` | No | Macros (like AWS SAM) |
| `Resources` | **YES** | The actual infrastructure! |
| `Outputs` | No | Values to display or export |

> **Remember:** Only `Resources` is required! Everything else is optional.

## 7. Your First CloudFormation Stack

Let's create a real stack step by step!

### Step 1: Create a Template File

Create `first-stack.yaml`:

```yaml
AWSTemplateFormatVersion: "2010-09-09"
Description: My very first CloudFormation stack - creates an S3 bucket

Resources:
  # Create an S3 bucket
  MyFirstBucket:
    Type: AWS::S3::Bucket
    Properties:
      BucketName: !Sub "my-first-cf-bucket-${AWS::AccountId}"
      Tags:
        - Key: Environment
          Value: learning
        - Key: ManagedBy
          Value: CloudFormation

Outputs:
  BucketName:
    Value: !Ref MyFirstBucket
    Description: Name of the created bucket
  
  BucketArn:
    Value: !GetAtt MyFirstBucket.Arn
    Description: ARN of the created bucket
  
  BucketDomainName:
    Value: !GetAtt MyFirstBucket.DomainName
    Description: Domain name of the bucket
```

**Let's break it down:**

- `Type: AWS::S3::Bucket` — Every resource has a **Type** that tells CloudFormation what to create. The format is `AWS::<Service>::<Resource>`.
- `!Sub "..."` — The `!Sub` function substitutes variables. `${AWS::AccountId}` is a **pseudo parameter** that auto-fills with your AWS account ID.
- `!Ref MyFirstBucket` — Returns the bucket name.
- `!GetAtt MyFirstBucket.Arn` — Gets a specific attribute of the resource (like ARN).

### Step 2: Deploy Using AWS CLI

```bash
# Create the stack
aws cloudformation create-stack \
  --stack-name my-first-stack \
  --template-body file://first-stack.yaml

# Watch the progress
aws cloudformation describe-stack-events \
  --stack-name my-first-stack \
  --query 'StackEvents[*].[Timestamp,LogicalResourceId,ResourceStatus,ResourceStatusReason]' \
  --output table

# Wait for completion
aws cloudformation wait stack-create-complete \
  --stack-name my-first-stack

# See the outputs
aws cloudformation describe-stacks \
  --stack-name my-first-stack \
  --query 'Stacks[0].Outputs' \
  --output table
```

### Step 3: Deploy Using AWS Console (GUI)

1. Go to **AWS Console** → **CloudFormation**
2. Click **"Create Stack"** → **"With new resources"**
3. Upload your template file
4. Enter a stack name: `my-first-stack`
5. Click through the wizard → **"Create Stack"**
6. Watch the **Events** tab as resources are created

### Step 4: Clean Up

```bash
# Delete the stack (and all its resources)
aws cloudformation delete-stack \
  --stack-name my-first-stack
```

## 8. Parameters — Making Templates Reusable

Parameters let users provide **input values** when creating a stack — making templates reusable.

### Basic Parameters

```yaml
Parameters:
  # Text parameter
  EnvironmentName:
    Type: String
    Default: dev
    AllowedValues:
      - dev
      - staging
      - prod
    Description: Choose the environment
    ConstraintDescription: Must be dev, staging, or prod

  # Number parameter
  InstanceCount:
    Type: Number
    Default: 1
    MinValue: 1
    MaxValue: 10
    Description: Number of EC2 instances

  # AWS-specific parameter type (auto-validates!)
  VpcId:
    Type: AWS::EC2::VPC::Id
    Description: Select a VPC

  # List of subnet IDs
  SubnetIds:
    Type: List<AWS::EC2::Subnet::Id>
    Description: Select subnets

  # Key pair
  KeyPairName:
    Type: AWS::EC2::KeyPair::KeyName
    Description: Select an SSH key pair

  # Password (won't be shown in console)
  DatabasePassword:
    Type: String
    NoEcho: true              # Hides the value in the console!
    MinLength: 8
    MaxLength: 64
    AllowedPattern: "[a-zA-Z0-9]*"
    Description: Database password (min 8 chars, alphanumeric only)
```

### Using Parameters in Resources

```yaml
Resources:
  WebServer:
    Type: AWS::EC2::Instance
    Properties:
      InstanceType: !Ref InstanceType        # Use parameter value
      KeyName: !Ref KeyPairName              # Use parameter value
      SubnetId: !Select [0, !Ref SubnetIds]  # First subnet from the list
      Tags:
        - Key: Environment
          Value: !Ref EnvironmentName        # Use parameter value
```

### AWS-Specific Parameter Types (Super Useful!)

These types auto-populate dropdowns in the AWS Console:

| Type | What It Does |
|------|-------------|
| `AWS::EC2::VPC::Id` | Shows a dropdown of your VPCs |
| `AWS::EC2::Subnet::Id` | Shows a dropdown of your subnets |
| `AWS::EC2::SecurityGroup::Id` | Shows a dropdown of security groups |
| `AWS::EC2::KeyPair::KeyName` | Shows a dropdown of key pairs |
| `AWS::EC2::Image::Id` | Shows AMI IDs |
| `AWS::SSM::Parameter::Value<String>` | Reads from SSM Parameter Store |

### Deploying with Parameters

```bash
# Via CLI
aws cloudformation create-stack \
  --stack-name my-stack \
  --template-body file://template.yaml \
  --parameters \
    ParameterKey=EnvironmentName,ParameterValue=prod \
    ParameterKey=InstanceCount,ParameterValue=3

# Or using a parameters file
aws cloudformation create-stack \
  --stack-name my-stack \
  --template-body file://template.yaml \
  --parameters file://params-prod.json
```

**params-prod.json:**

```json
[
  {
    "ParameterKey": "EnvironmentName",
    "ParameterValue": "prod"
  },
  {
    "ParameterKey": "InstanceCount",
    "ParameterValue": "3"
  }
]
```

## 9. Mappings — Lookup Tables

Mappings are like **dictionaries/lookup tables**. They let you define fixed values based on keys (like region, environment, etc.).

### Example: Different AMIs per Region

```yaml
Mappings:
  RegionAMI:
    us-east-1:
      HVM64: ami-0c55b159cbfafe1f0
      HVM32: ami-0a1b2c3d4e5f67890
    us-west-2:
      HVM64: ami-0892d3c7ee96c0bf7
      HVM32: ami-0f1g2h3i4j5k67890
    eu-west-1:
      HVM64: ami-0d729a60dc2e30ac6
      HVM32: ami-0e2f3g4h5i6j78901

  EnvironmentConfig:
    dev:
      InstanceType: t3.micro
      MinSize: "1"
      MaxSize: "2"
    staging:
      InstanceType: t3.small
      MinSize: "2"
      MaxSize: "4"
    prod:
      InstanceType: t3.large
      MinSize: "3"
      MaxSize: "10"

Resources:
  WebServer:
    Type: AWS::EC2::Instance
    Properties:
      # Look up the AMI for the current region
      ImageId: !FindInMap [RegionAMI, !Ref "AWS::Region", HVM64]
      
      # Look up the instance type for the environment
      InstanceType: !FindInMap [EnvironmentConfig, !Ref EnvironmentName, InstanceType]
```

**How `!FindInMap` works:**

```
!FindInMap [MapName, FirstLevelKey, SecondLevelKey]

!FindInMap [RegionAMI, us-east-1, HVM64]
         → ami-0c55b159cbfafe1f0
```

It's like: `RegionAMI["us-east-1"]["HVM64"]` in Python.

## 10. Conditions — If/Else Logic

Conditions let you create resources only when certain criteria are met.

### Defining Conditions

```yaml
Parameters:
  EnvironmentType:
    Type: String
    Default: dev
    AllowedValues: [dev, staging, prod]

  EnableMonitoring:
    Type: String
    Default: "false"
    AllowedValues: ["true", "false"]

Conditions:
  # Is this the production environment?
  IsProduction: !Equals [!Ref EnvironmentType, prod]
  
  # Is this NOT production?
  IsNotProduction: !Not [!Equals [!Ref EnvironmentType, prod]]
  
  # Is monitoring enabled?
  MonitoringEnabled: !Equals [!Ref EnableMonitoring, "true"]
  
  # Is this production AND monitoring is enabled?
  ProdWithMonitoring: !And
    - !Condition IsProduction
    - !Condition MonitoringEnabled
  
  # Is this production OR staging?
  IsHigherEnvironment: !Or
    - !Equals [!Ref EnvironmentType, prod]
    - !Equals [!Ref EnvironmentType, staging]
```

### Using Conditions

```yaml
Resources:
  # This EC2 instance is ONLY created in production
  ProductionServer:
    Type: AWS::EC2::Instance
    Condition: IsProduction          # ← Only if IsProduction is true
    Properties:
      InstanceType: t3.large
      ImageId: ami-0c55b159cbfafe1f0

  # This CloudWatch alarm is ONLY created if monitoring is enabled
  CPUAlarm:
    Type: AWS::CloudWatch::Alarm
    Condition: MonitoringEnabled     # ← Only if monitoring is enabled
    Properties:
      AlarmName: HighCPU
      MetricName: CPUUtilization
      Namespace: AWS/EC2
      Threshold: 80
      Period: 300
      EvaluationPeriods: 2
      Statistic: Average
      ComparisonOperator: GreaterThanThreshold

  # Use conditions in property values with !If
  WebServer:
    Type: AWS::EC2::Instance
    Properties:
      # Different instance type based on environment
      InstanceType: !If [IsProduction, t3.large, t3.micro]
      
      # Different number of volumes
      BlockDeviceMappings:
        - DeviceName: /dev/xvda
          Ebs:
            VolumeSize: !If [IsProduction, 100, 20]    # 100GB for prod, 20GB for dev
```

### Condition Functions Reference

| Function | Purpose | Example |
|----------|---------|---------|
| `!Equals` | Compare two values | `!Equals [!Ref Env, prod]` |
| `!Not` | Negate a condition | `!Not [!Equals [!Ref Env, prod]]` |
| `!And` | ALL must be true | `!And [Cond1, Cond2]` |
| `!Or` | ANY must be true | `!Or [Cond1, Cond2]` |
| `!If` | Ternary (if/else) | `!If [Cond, ValueIfTrue, ValueIfFalse]` |

## 11. Intrinsic Functions — The Power Tools

Intrinsic functions are built-in functions that do transformations and lookups. They're the **most important thing to learn** in CloudFormation!

### 11.1 — !Ref (Reference)

Returns the value of a parameter or the ID/name of a resource:

```yaml
# Reference a parameter
InstanceType: !Ref MyInstanceTypeParam

# Reference a resource (returns its ID/name)
SubnetId: !Ref MySubnet
```

### 11.2 — !GetAtt (Get Attribute)

Gets a specific attribute from a resource:

```yaml
# Get the ARN of an S3 bucket
BucketArn: !GetAtt MyBucket.Arn

# Get the Availability Zone of an EC2 instance
AZ: !GetAtt MyInstance.AvailabilityZone

# Get the DNS name of a load balancer
DNSName: !GetAtt MyALB.DNSName
```

### 11.3 — !Sub (Substitute)

String interpolation — replaces `${Variable}` with actual values:

```yaml
# Simple substitution
BucketName: !Sub "my-app-${EnvironmentName}-${AWS::Region}"
# Result: "my-app-prod-us-east-1"

# Multi-line substitution
UserData:
  Fn::Base64:
    !Sub |
      #!/bin/bash
      echo "Environment: ${EnvironmentName}"
      echo "Region: ${AWS::Region}"
      echo "Stack: ${AWS::StackName}"
      yum update -y
      yum install -y httpd
      systemctl start httpd
```

### 11.4 — !Join (Join Strings)

Joins a list of strings with a separator:

```yaml
# Join with comma
SecurityGroupIds: !Join [",", [!Ref SG1, !Ref SG2, !Ref SG3]]
# Result: "sg-111,sg-222,sg-333"

# Join with empty string
Hostname: !Join ["", ["app", ".", !Ref Domain]]
# Result: "app.example.com"
```

### 11.5 — !Split (Split String)

Splits a string into a list:

```yaml
# Split a comma-separated string
!Split [",", "subnet-1,subnet-2,subnet-3"]
# Result: ["subnet-1", "subnet-2", "subnet-3"]
```

### 11.6 — !Select (Select from List)

Picks an item from a list by index:

```yaml
# Get the first AZ in the region
AvailabilityZone: !Select [0, !GetAZs ""]
# Result: "us-east-1a"

# Get the second subnet
SubnetId: !Select [1, !Ref SubnetIds]
```

### 11.7 — !GetAZs (Get Availability Zones)

Returns a list of AZs for a region:

```yaml
# Get AZs for the current region
AZs: !GetAZs ""    # Empty string = current region
# Result: ["us-east-1a", "us-east-1b", "us-east-1c", ...]
```

### 11.8 — !Cidr (Generate CIDR Blocks)

Generates CIDR blocks from a VPC CIDR:

```yaml
# Generate 4 subnets from a /16 VPC
Subnets: !Cidr [!GetAtt VPC.CidrBlock, 4, 8]
# From 10.0.0.0/16, generates:
# 10.0.0.0/24, 10.0.1.0/24, 10.0.2.0/24, 10.0.3.0/24
```

### 11.9 — !ImportValue (Cross-Stack Import)

Import a value exported by another stack:

```yaml
# In stack A (exports a value):
Outputs:
  VpcId:
    Value: !Ref MyVPC
    Export:
      Name: shared-vpc-id

# In stack B (imports the value):
Resources:
  MySubnet:
    Type: AWS::EC2::Subnet
    Properties:
      VpcId: !ImportValue shared-vpc-id    # ← Import from stack A!
```

### 11.10 — !Base64

Encodes a string to Base64 (needed for EC2 UserData):

```yaml
UserData:
  Fn::Base64: !Sub |
    #!/bin/bash
    echo "Hello from ${AWS::StackName}"
```

### Quick Reference Table

| Function | Purpose | Example |
|----------|---------|---------|
| `!Ref` | Get parameter/resource value | `!Ref MyParam` |
| `!GetAtt` | Get resource attribute | `!GetAtt MyBucket.Arn` |
| `!Sub` | String interpolation | `!Sub "Hello ${Name}"` |
| `!Join` | Join strings | `!Join ["-", ["a", "b"]]` |
| `!Split` | Split string | `!Split [",", "a,b,c"]` |
| `!Select` | Pick from list | `!Select [0, !Ref List]` |
| `!GetAZs` | List AZs | `!GetAZs ""` |
| `!FindInMap` | Lookup value | `!FindInMap [Map, Key1, Key2]` |
| `!If` | Conditional | `!If [Cond, Yes, No]` |
| `!Equals` | Compare | `!Equals [a, b]` |
| `!ImportValue` | Cross-stack | `!ImportValue ExportName` |
| `!Base64` | Encode | `Fn::Base64: "text"` |
| `!Cidr` | Generate CIDRs | `!Cidr [CIDR, Count, Bits]` |

## 12. Outputs and Cross-Stack References

### Defining Outputs

```yaml
Outputs:
  VpcId:
    Description: The ID of the VPC
    Value: !Ref MyVPC
    
  PublicSubnetId:
    Description: The ID of the public subnet
    Value: !Ref PublicSubnet
    
  WebServerPublicIP:
    Description: Public IP of the web server
    Value: !GetAtt WebServer.PublicIp
    Condition: IsProduction    # Only show this output in production
```

### Exporting Values (for Cross-Stack References)

```yaml
# Stack: networking-stack
Outputs:
  VpcId:
    Value: !Ref MyVPC
    Export:
      Name: !Sub "${AWS::StackName}-VpcId"    # Export name must be unique per region
    
  PublicSubnetId:
    Value: !Ref PublicSubnet
    Export:
      Name: !Sub "${AWS::StackName}-PublicSubnetId"
```

### Importing Values in Another Stack

```yaml
# Stack: application-stack
Resources:
  WebServer:
    Type: AWS::EC2::Instance
    Properties:
      SubnetId: !ImportValue "networking-stack-PublicSubnetId"
      # ...
```

### Viewing Exports

```bash
# List all exports in a region
aws cloudformation list-exports \
  --query 'Exports[*].[Name,Value]' \
  --output table
```

## 13. Stacks — Creating, Updating, Deleting

### Creating a Stack

```bash
# Simple create
aws cloudformation create-stack \
  --stack-name my-app \
  --template-body file://template.yaml

# With parameters
aws cloudformation create-stack \
  --stack-name my-app \
  --template-body file://template.yaml \
  --parameters ParameterKey=Env,ParameterValue=prod

# With IAM capabilities (needed when creating IAM resources)
aws cloudformation create-stack \
  --stack-name my-app \
```

## 14. Nested Stacks — Breaking Down Large Templates

When your template gets too big (>1000 resources, or just hard to manage), break it into **nested stacks**.

### How It Works

```
Parent Stack (root.yaml)
├── Nested Stack: networking (networking.yaml)
│   ├── VPC
│   ├── Subnets
│   └── Internet Gateway
├── Nested Stack: security (security.yaml)
│   ├── Security Groups
│   └── IAM Roles
└── Nested Stack: application (application.yaml)
    ├── EC2 Instances
    ├── ALB
    └── Auto Scaling Group
```

### Parent Stack

```yaml
# root.yaml
AWSTemplateFormatVersion: "2010-09-09"
Description: Parent stack that creates nested stacks

Parameters:
  Environment:
    Type: String
    Default: dev

Resources:
  # Nested stack for networking
  NetworkStack:
    Type: AWS::CloudFormation::Stack
    Properties:
      TemplateURL: https://s3.amazonaws.com/my-templates/networking.yaml
      Parameters:
        Environment: !Ref Environment
        VpcCidr: "10.0.0.0/16"

  # Nested stack for application
  ApplicationStack:
    Type: AWS::CloudFormation::Stack
    DependsOn: NetworkStack    # Wait for networking first!
    Properties:
      TemplateURL: https://s3.amazonaws.com/my-templates/application.yaml
      Parameters:
        Environment: !Ref Environment
        # Pass outputs from networking stack as inputs!
        VpcId: !GetAtt NetworkStack.Outputs.VpcId
        SubnetId: !GetAtt NetworkStack.Outputs.PublicSubnetId

Outputs:
  WebsiteURL:
    Value: !GetAtt ApplicationStack.Outputs.WebsiteURL
```

### Child Stack (networking.yaml)

```yaml
# networking.yaml
AWSTemplateFormatVersion: "2010-09-09"
Description: Networking resources

Parameters:
  Environment:
    Type: String
  VpcCidr:
    Type: String

Resources:
  VPC:
    Type: AWS::EC2::VPC
    Properties:
      CidrBlock: !Ref VpcCidr
      Tags:
        - Key: Name
          Value: !Sub "${Environment}-vpc"

  PublicSubnet:
    Type: AWS::EC2::Subnet
    Properties:
      VpcId: !Ref VPC
      CidrBlock: "10.0.1.0/24"

Outputs:
  VpcId:
    Value: !Ref VPC
  PublicSubnetId:
    Value: !Ref PublicSubnet
```

> **Important:** Nested stack templates must be stored in **S3** — you can't use local files!

### Uploading Templates to S3

```bash
# Upload templates to S3
aws s3 cp networking.yaml s3://my-templates/networking.yaml
aws s3 cp application.yaml s3://my-templates/application.yaml
aws s3 cp root.yaml s3://my-templates/root.yaml

# Deploy the parent stack
aws cloudformation create-stack \
  --stack-name my-app \
  --template-url https://s3.amazonaws.com/my-templates/root.yaml \
  --capabilities CAPABILITY_IAM
```

## 15. StackSets — Multi-Account, Multi-Region Deployments

**StackSets** is one of CloudFormation's most powerful features. It lets you deploy the **same template to multiple AWS accounts and regions** in a single operation.

### Use Cases

- Deploy security baselines to all 50+ accounts in your organization
- Set up CloudTrail logging in every region
- Create IAM roles across all accounts
- Deploy compliance rules everywhere

### How It Works

```
StackSet (defined in admin account)
├── Stack Instance: Account 111111111111 / us-east-1
├── Stack Instance: Account 111111111111 / eu-west-1
├── Stack Instance: Account 222222222222 / us-east-1
├── Stack Instance: Account 222222222222 / eu-west-1
├── Stack Instance: Account 333333333333 / us-east-1
└── Stack Instance: Account 333333333333 / eu-west-1
```

### Creating a StackSet

```yaml
# security-baseline.yaml — Deploy to ALL accounts
AWSTemplateFormatVersion: "2010-09-09"
Description: Security baseline for all accounts

Resources:
  # Enable CloudTrail in every account
  CloudTrail:
    Type: AWS::CloudTrail::Trail
    Properties:
      TrailName: organization-trail
      S3BucketName: !Ref TrailBucket
      IsMultiRegionTrail: true
      EnableLogFileValidation: true
      IsLogging: true

  # Create a security SNS topic in every account
  SecurityAlerts:
    Type: AWS::SNS::Topic
    Properties:
      TopicName: security-alerts

  # Create a standard IAM password policy
  PasswordPolicy:
    Type: AWS::IAM::AccountPasswordPolicy  
    Properties:
      MinimumPasswordLength: 14
      RequireSymbols: true
      RequireNumbers: true
      RequireUppercaseCharacters: true
      RequireLowercaseCharacters: true
      MaxPasswordAge: 90
```

### CLI Commands for StackSets

```bash
# Create a StackSet
aws cloudformation create-stack-set \
  --stack-set-name security-baseline \
  --template-body file://security-baseline.yaml \
  --permission-model SERVICE_MANAGED \
  --auto-deployment Enabled=true,RetainStacksOnAccountRemoval=false

# Deploy to specific accounts and regions
aws cloudformation create-stack-instances \
  --stack-set-name security-baseline \
  --accounts '["111111111111", "222222222222", "333333333333"]' \
  --regions '["us-east-1", "eu-west-1", "ap-southeast-1"]'

# Deploy to an entire AWS Organization OU
aws cloudformation create-stack-instances \
  --stack-set-name security-baseline \
  --deployment-targets OrganizationalUnitIds='["ou-abc123"]' \
  --regions '["us-east-1", "eu-west-1"]'

# Check status
aws cloudformation describe-stack-set-operation \
  --stack-set-name security-baseline \
  --operation-id abc-123-def

# Update all instances
aws cloudformation update-stack-set \
  --stack-set-name security-baseline \

# Delete stack instances

  --stack-set-name security-baseline \
  --accounts '["111111111111"]' \
  --regions '["us-east-1"]' \

# Delete the StackSet
aws cloudformation delete-stack-set \
  --stack-set-name security-baseline
```

|-------|-------------|----------|

| **Service-managed** | Uses AWS Organizations — automatic trust | Enterprise setups with Organizations |

## 16. Change Sets — Preview Before You Deploy

**Change Sets** let you preview what CloudFormation will do **BEFORE** making changes. It's like `terraform plan`.

### Creating a Change Set

```bash
# Create a change set (doesn't apply anything yet!)
aws cloudformation create-change-set \
  --stack-name my-app \
  --template-body file://template-v2.yaml \
  --change-set-name my-update-v2

# View the change set
aws cloudformation describe-change-set \
  --stack-name my-app \
  --change-set-name my-update-v2
```

### What You See

```json
{
  "Changes": [
    {
      "Type": "Resource",
      "ResourceChange": {
        "Action": "Modify",              
        "LogicalResourceId": "WebServer",
        "ResourceType": "AWS::EC2::Instance",
        "Replacement": "False",           
        "Details": [
          {
            "Target": {
              "Attribute": "Properties",
              "Name": "InstanceType"
            },
            "ChangeSource": "DirectModification"
          }
        ]
      }
    },
    {
      "Type": "Resource",
      "ResourceChange": {
        "Action": "Add",                  
        "LogicalResourceId": "NewBucket",
        "ResourceType": "AWS::S3::Bucket"
      }
    }
  ]
}
```

### Change Set Actions

| Action | Meaning | Risk |
|--------|---------|------|
| `Add` | New resource will be created | Low ✅ |
| `Modify` | Existing resource will be updated | Medium ⚠️ |
| `Remove` | Resource will be deleted | High 🔴 |
| `Replacement: True` | Resource will be REPLACED (deleted and recreated!) | Very High 🔴🔴 |

### Execute or Delete the Change Set

```bash
# If you're happy with the changes — APPLY them!
aws cloudformation execute-change-set \
  --stack-name my-app \
  --change-set-name my-update-v2

# If you don't want these changes — DELETE the change set
aws cloudformation delete-change-set \
  --stack-name my-app \
  --change-set-name my-update-v2
```

> **Best Practice:** ALWAYS use change sets for production updates. Never do direct `update-stack` in prod!

## 17. Drift Detection — Has Someone Changed Things Manually?

**Drift detection** tells you if someone (or something) has modified your resources **outside of CloudFormation** — like someone manually changing an EC2 instance type in the AWS Console.

### Why Is Drift Bad?

Imagine this scenario:
1. CloudFormation creates an EC2 instance with type `t3.micro`.
2. Someone manually changes it to `t3.large` in the AWS Console.
3. Now CloudFormation thinks it's `t3.micro`, but it's actually `t3.large`.
4. This mismatch is called **drift** — and it can cause surprises!

### Running Drift Detection

```bash
# Start drift detection
aws cloudformation detect-stack-drift \
  --stack-name my-app

# Returns a detection ID
# {
#   "StackDriftDetectionId": "abc-123-def"
# }

# Check the detection status
aws cloudformation describe-stack-drift-detection-status \
  --stack-drift-detection-id abc-123-def

# View drift results for all resources
aws cloudformation describe-stack-resource-drifts \
  --stack-name my-app \
  --stack-resource-drift-status-filters MODIFIED DELETED
```

### Drift Status Values

| Status | Meaning |
|--------|---------|
| `IN_SYNC` | Resource matches the template ✅ |
| `MODIFIED` | Properties have been changed ⚠️ |
| `DELETED` | Resource was deleted outside CloudFormation 🔴 |
| `NOT_CHECKED` | Drift detection not supported for this resource type |

### What Drift Looks Like

```json
{
  "StackResourceDriftStatus": "MODIFIED",
  "LogicalResourceId": "WebServer",
  "ResourceType": "AWS::EC2::Instance",
  "PropertyDifferences": [
    {
      "PropertyPath": "/InstanceType",
      "ExpectedValue": "t3.micro",       
      "ActualValue": "t3.large",         
      "DifferenceType": "NOT_EQUAL"
    }
  ]
}
```

This tells you exactly:
- **What** changed: `InstanceType`
- **Expected** value: `t3.micro` (what CloudFormation thinks)
- **Actual** value: `t3.large` (what's really there)

### Fixing Drift

You have two options:

**Option 1: Update the template to match reality**

```yaml
# Change template to match the manual change
WebServer:
  Type: AWS::EC2::Instance
  Properties:
    InstanceType: t3.large    # Update to match actual value
```

**Option 2: Update the stack to enforce the template**

```bash
# Force CloudFormation to reset the resource to match the template
aws cloudformation update-stack \
  --stack-name my-app \
  --template-body file://template.yaml \
  --use-previous-template
```

### Drift Detection in the Console

1. Go to **CloudFormation** → Select your stack
2. Click **Stack actions** → **Detect drift**
3. Wait for detection to complete
4. Click **View drift results**
5. Click on individual resources to see property differences

## 18. Rollback and Failure Handling

### Automatic Rollback (Default Behavior)

By default, if ANY resource fails to create, CloudFormation **rolls back EVERYTHING**:

```
1. Create VPC         ✅ Success
2. Create Subnet      ✅ Success
3. Create EC2         ❌ Failed (bad AMI ID)
4. ROLLBACK starts!
5. Delete Subnet      ✅ Deleted
6. Delete VPC         ✅ Deleted
Stack status: ROLLBACK_COMPLETE
```

### Disable Rollback (for Debugging)

Sometimes you want to keep the failed resources to debug:

```bash
aws cloudformation create-stack \
  --stack-name my-app \
  --template-body file://template.yaml \
  --disable-rollback    # ← Keep resources even if creation fails
```

### Stack Rollback Configuration

```bash
# Set a CloudWatch alarm as a rollback trigger
aws cloudformation create-stack \
  --stack-name my-app \
  --template-body file://template.yaml \
  --rollback-configuration '{
    "RollbackTriggers": [
      {
        "Arn": "arn:aws:cloudwatch:us-east-1:123456789012:alarm:HighErrorRate",
        "Type": "AWS::CloudWatch::Alarm"
      }
    ],
    "MonitoringTimeInMinutes": 10
  }'
```

This watches the alarm for 10 minutes after deployment. If the alarm fires, CloudFormation automatically rolls back!

### DeletionPolicy — Protect Resources from Deletion

```yaml
Resources:
  ProductionDatabase:
    Type: AWS::RDS::DBInstance
    DeletionPolicy: Retain       # ← NEVER delete this, even if stack is deleted!
    Properties:
      DBInstanceClass: db.t3.large
      Engine: mysql
      # ...

  LogBucket:
    Type: AWS::S3::Bucket
    DeletionPolicy: Retain       # Keep the bucket even if stack is deleted
    
  TempBucket:
    Type: AWS::S3::Bucket
    DeletionPolicy: Delete       # Default — delete when stack is deleted
```

| DeletionPolicy | Behavior |
|---------------|----------|
| `Delete` | Resource is deleted when stack is deleted (default) |
| `Retain` | Resource is kept — CloudFormation stops managing it |
| `Snapshot` | Take a snapshot before deleting (RDS, EBS, etc.) |

### UpdateReplacePolicy

Same idea, but for when CloudFormation needs to **replace** a resource during an update:

```yaml
Resources:
  Database:
    Type: AWS::RDS::DBInstance
    UpdateReplacePolicy: Snapshot    # Snapshot before replacement
    DeletionPolicy: Snapshot         # Snapshot before deletion
```

## 19. CloudFormation Helper Scripts

CloudFormation provides helper scripts for EC2 instances to handle complex bootstrapping.

### cfn-init — Configure Instances Declaratively

Instead of writing long bash scripts in UserData, use `cfn-init` for structured configuration:

```yaml
Resources:
  WebServer:
    Type: AWS::EC2::Instance
    Metadata:
      AWS::CloudFormation::Init:
        config:
          # 1. Install packages
          packages:
            yum:
              httpd: []
              php: []
              mysql: []
          
          # 2. Create files
          files:
            /var/www/html/index.html:
              content: |
                <h1>Hello from CloudFormation!</h1>
                <p>This was configured by cfn-init</p>
              mode: "000644"
              owner: apache
              group: apache
            
            /etc/cfn/cfn-hup.conf:
              content: !Sub |
                [main]
                stack=${AWS::StackName}
                region=${AWS::Region}
                interval=5
              mode: "000400"
              owner: root
              group: root
          
          # 3. Start services
          services:
            sysvinit:
              httpd:
                enabled: true
                ensureRunning: true
    
    Properties:
      ImageId: ami-0c55b159cbfafe1f0
      InstanceType: t3.micro
      UserData:
        Fn::Base64: !Sub |
          #!/bin/bash -xe
          # Install cfn-init
          yum install -y aws-cfn-bootstrap
          
          # Run cfn-init
          /opt/aws/bin/cfn-init -v \
            --stack ${AWS::StackName} \
            --resource WebServer \
            --region ${AWS::Region}
          
          # Signal success/failure to CloudFormation
          /opt/aws/bin/cfn-signal -e $? \
            --stack ${AWS::StackName} \
            --resource WebServer \
            --region ${AWS::Region}
    
    CreationPolicy:
      ResourceSignal:
        Timeout: PT5M    # Wait 5 minutes for the signal
```

### Helper Scripts Summary

| Script | Purpose |
|--------|---------|
| `cfn-init` | Reads metadata and configures the instance |
| `cfn-signal` | Tells CloudFormation if setup succeeded or failed |
| `cfn-hup` | Daemon that detects metadata changes and re-runs cfn-init |
| `cfn-get-metadata` | Retrieves metadata for a resource |

## 20. Real-World Example — Complete 3-Tier Architecture

Here's a production-ready template that creates a proper web application infrastructure:

```yaml
AWSTemplateFormatVersion: "2010-09-09"
Description: >
  Complete 3-tier web application:
  VPC → ALB → EC2 Auto Scaling Group → RDS Database

# =====================================================
# PARAMETERS
# =====================================================
Parameters:
  EnvironmentName:
    Type: String
    Default: dev
    AllowedValues: [dev, staging, prod]
    Description: Environment name

  InstanceType:
    Type: String
    Default: t3.micro
    AllowedValues: [t3.micro, t3.small, t3.medium, t3.large]
    Description: EC2 instance type

  KeyPairName:
    Type: AWS::EC2::KeyPair::KeyName
    Description: SSH key pair

  DBPassword:
    Type: String
    NoEcho: true
    MinLength: 8
    Description: Database master password

# =====================================================
# MAPPINGS
# =====================================================
Mappings:
  EnvironmentConfig:
    dev:
      MinSize: "1"
      MaxSize: "2"
      DBInstanceClass: db.t3.micro
    staging:
      MinSize: "2"
      MaxSize: "4"
      DBInstanceClass: db.t3.small
    prod:
      MinSize: "3"
      MaxSize: "10"
      DBInstanceClass: db.t3.medium

# =====================================================
# CONDITIONS
# =====================================================
Conditions:
  IsProduction: !Equals [!Ref EnvironmentName, prod]

# =====================================================
# RESOURCES
# =====================================================
Resources:

  # ---------- NETWORKING ----------

  VPC:
    Type: AWS::EC2::VPC
    Properties:
      CidrBlock: "10.0.0.0/16"
      EnableDnsHostnames: true
      EnableDnsSupport: true
      Tags:
        - Key: Name
          Value: !Sub "${EnvironmentName}-vpc"

  InternetGateway:
    Type: AWS::EC2::InternetGateway
    Properties:
      Tags:
        - Key: Name
          Value: !Sub "${EnvironmentName}-igw"

  AttachGateway:
    Type: AWS::EC2::VPCGatewayAttachment
    Properties:
      VpcId: !Ref VPC
      InternetGatewayId: !Ref InternetGateway

  PublicSubnet1:
    Type: AWS::EC2::Subnet
    Properties:
      VpcId: !Ref VPC
      CidrBlock: "10.0.1.0/24"
      AvailabilityZone: !Select [0, !GetAZs ""]
      MapPublicIpOnLaunch: true
      Tags:
        - Key: Name
          Value: !Sub "${EnvironmentName}-public-1"

  PublicSubnet2:
    Type: AWS::EC2::Subnet
    Properties:
      VpcId: !Ref VPC
      CidrBlock: "10.0.2.0/24"
      AvailabilityZone: !Select [1, !GetAZs ""]
      MapPublicIpOnLaunch: true
      Tags:
        - Key: Name
          Value: !Sub "${EnvironmentName}-public-2"

  PrivateSubnet1:
    Type: AWS::EC2::Subnet
    Properties:
      VpcId: !Ref VPC
      CidrBlock: "10.0.3.0/24"
      AvailabilityZone: !Select [0, !GetAZs ""]
      Tags:
        - Key: Name
          Value: !Sub "${EnvironmentName}-private-1"

  PrivateSubnet2:
    Type: AWS::EC2::Subnet
    Properties:
      VpcId: !Ref VPC
      CidrBlock: "10.0.4.0/24"
      AvailabilityZone: !Select [1, !GetAZs ""]
      Tags:
        - Key: Name
          Value: !Sub "${EnvironmentName}-private-2"

  PublicRouteTable:
    Type: AWS::EC2::RouteTable
    Properties:
      VpcId: !Ref VPC
      Tags:
        - Key: Name
          Value: !Sub "${EnvironmentName}-public-rt"

  PublicRoute:
    Type: AWS::EC2::Route
    DependsOn: AttachGateway
    Properties:
      RouteTableId: !Ref PublicRouteTable
      DestinationCidrBlock: "0.0.0.0/0"
      GatewayId: !Ref InternetGateway

  PublicSubnet1RTAssociation:
    Type: AWS::EC2::SubnetRouteTableAssociation
    Properties:
      SubnetId: !Ref PublicSubnet1
      RouteTableId: !Ref PublicRouteTable

  PublicSubnet2RTAssociation:
    Type: AWS::EC2::SubnetRouteTableAssociation
    Properties:
      SubnetId: !Ref PublicSubnet2
      RouteTableId: !Ref PublicRouteTable

  # ---------- SECURITY GROUPS ----------

  ALBSecurityGroup:
    Type: AWS::EC2::SecurityGroup
    Properties:
      GroupDescription: Allow HTTP/HTTPS to ALB
      VpcId: !Ref VPC
      SecurityGroupIngress:
        - IpProtocol: tcp
          FromPort: 80
          ToPort: 80
          CidrIp: "0.0.0.0/0"
        - IpProtocol: tcp
          FromPort: 443
          ToPort: 443
          CidrIp: "0.0.0.0/0"
      Tags:
        - Key: Name
          Value: !Sub "${EnvironmentName}-alb-sg"

  AppSecurityGroup:
    Type: AWS::EC2::SecurityGroup
    Properties:
      GroupDescription: Allow traffic from ALB only
      VpcId: !Ref VPC
      SecurityGroupIngress:
        - IpProtocol: tcp
          FromPort: 80
          ToPort: 80
          SourceSecurityGroupId: !Ref ALBSecurityGroup
      Tags:
        - Key: Name
          Value: !Sub "${EnvironmentName}-app-sg"

  DBSecurityGroup:
    Type: AWS::EC2::SecurityGroup
    Properties:
      GroupDescription: Allow MySQL from app tier only
      VpcId: !Ref VPC
      SecurityGroupIngress:
        - IpProtocol: tcp
          FromPort: 3306
          ToPort: 3306
          SourceSecurityGroupId: !Ref AppSecurityGroup
      Tags:
        - Key: Name
          Value: !Sub "${EnvironmentName}-db-sg"

  # ---------- APPLICATION LOAD BALANCER ----------

  ApplicationLoadBalancer:
    Type: AWS::ElasticLoadBalancingV2::LoadBalancer
    Properties:
      Name: !Sub "${EnvironmentName}-alb"
      Scheme: internet-facing
      Type: application
      SecurityGroups:
        - !Ref ALBSecurityGroup
      Subnets:
        - !Ref PublicSubnet1
        - !Ref PublicSubnet2

  ALBTargetGroup:
    Type: AWS::ElasticLoadBalancingV2::TargetGroup
    Properties:
      Name: !Sub "${EnvironmentName}-tg"
      Port: 80
      Protocol: HTTP
      VpcId: !Ref VPC
      HealthCheckPath: /
      HealthCheckIntervalSeconds: 30
      HealthyThresholdCount: 2
      UnhealthyThresholdCount: 3
      TargetType: instance

  ALBListener:
    Type: AWS::ElasticLoadBalancingV2::Listener
    Properties:
      LoadBalancerArn: !Ref ApplicationLoadBalancer
      Port: 80
      Protocol: HTTP
      DefaultActions:
        - Type: forward
          TargetGroupArn: !Ref ALBTargetGroup

  # ---------- AUTO SCALING GROUP ----------

  LaunchTemplate:
    Type: AWS::EC2::LaunchTemplate
    Properties:
      LaunchTemplateName: !Sub "${EnvironmentName}-lt"
      LaunchTemplateData:
        ImageId: ami-0c55b159cbfafe1f0
        InstanceType: !Ref InstanceType
        KeyName: !Ref KeyPairName
        SecurityGroupIds:
          - !Ref AppSecurityGroup
        UserData:
          Fn::Base64: !Sub |
            #!/bin/bash -xe
            yum update -y
            yum install -y httpd
            systemctl start httpd
            systemctl enable httpd
            
            # Create a simple page showing instance info
            INSTANCE_ID=$(curl -s http://169.254.169.254/latest/meta-data/instance-id)
            cat > /var/www/html/index.html <<EOF
            <h1>Hello from ${EnvironmentName}!</h1>
            <p>Instance: $INSTANCE_ID</p>
            <p>Managed by CloudFormation</p>
            EOF

  AutoScalingGroup:
    Type: AWS::AutoScaling::AutoScalingGroup
    Properties:
      AutoScalingGroupName: !Sub "${EnvironmentName}-asg"
      LaunchTemplate:
        LaunchTemplateId: !Ref LaunchTemplate
        Version: !GetAtt LaunchTemplate.LatestVersionNumber
      MinSize: !FindInMap [EnvironmentConfig, !Ref EnvironmentName, MinSize]
      MaxSize: !FindInMap [EnvironmentConfig, !Ref EnvironmentName, MaxSize]
      DesiredCapacity: !FindInMap [EnvironmentConfig, !Ref EnvironmentName, MinSize]
      VPCZoneIdentifier:
        - !Ref PublicSubnet1
        - !Ref PublicSubnet2
      TargetGroupARNs:
        - !Ref ALBTargetGroup
      Tags:
        - Key: Name
          Value: !Sub "${EnvironmentName}-web-server"
          PropagateAtLaunch: true

  # Scaling Policy
  ScaleUpPolicy:
    Type: AWS::AutoScaling::ScalingPolicy
    Properties:
      AutoScalingGroupName: !Ref AutoScalingGroup
      PolicyType: TargetTrackingScaling
      TargetTrackingConfiguration:
        PredefinedMetricSpecification:
          PredefinedMetricType: ASGAverageCPUUtilization
        TargetValue: 70

  # ---------- DATABASE (RDS) ----------

  DBSubnetGroup:
    Type: AWS::RDS::DBSubnetGroup
    Properties:
      DBSubnetGroupDescription: Subnet group for RDS
      SubnetIds:
        - !Ref PrivateSubnet1
        - !Ref PrivateSubnet2

  Database:
    Type: AWS::RDS::DBInstance
    DeletionPolicy: !If [IsProduction, Snapshot, Delete]
    Properties:
      DBInstanceIdentifier: !Sub "${EnvironmentName}-database"
      DBInstanceClass: !FindInMap [EnvironmentConfig, !Ref EnvironmentName, DBInstanceClass]
      Engine: mysql
      EngineVersion: "8.0"
      MasterUsername: admin
      MasterUserPassword: !Ref DBPassword
      AllocatedStorage: "20"
      StorageType: gp3
      DBSubnetGroupName: !Ref DBSubnetGroup
      VPCSecurityGroups:
        - !Ref DBSecurityGroup
      MultiAZ: !If [IsProduction, true, false]
      BackupRetentionPeriod: !If [IsProduction, 7, 1]
      Tags:
        - Key: Name
          Value: !Sub "${EnvironmentName}-db"

# =====================================================
# OUTPUTS
# =====================================================
Outputs:
  VpcId:
    Description: VPC ID
    Value: !Ref VPC
    Export:
      Name: !Sub "${AWS::StackName}-VpcId"

  WebsiteURL:
    Description: Application URL
    Value: !Sub "http://${ApplicationLoadBalancer.DNSName}"

  DatabaseEndpoint:
    Description: RDS Endpoint
    Value: !GetAtt Database.Endpoint.Address

  DatabasePort:
    Description: RDS Port
    Value: !GetAtt Database.Endpoint.Port
```

**What this template creates:**

1. **VPC** with 2 public + 2 private subnets across 2 AZs
2. **Internet Gateway** + Route Table for internet access
3. **3 Security Groups** (ALB → App → DB, each only allows traffic from the previous tier)
4. **Application Load Balancer** in public subnets
5. **Auto Scaling Group** with launch template and scaling policy
6. **RDS MySQL Database** in private subnets with Multi-AZ in production

## 21. CloudFormation with CI/CD

### GitHub Actions

```yaml
# .github/workflows/cloudformation.yml
name: CloudFormation Deploy

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  validate:
    name: Validate Template
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Configure AWS Credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: us-east-1

      - name: Validate CloudFormation Template
        run: |
          aws cloudformation validate-template \
            --template-body file://template.yaml

      - name: Lint with cfn-lint
        run: |
          pip install cfn-lint
          cfn-lint template.yaml

  deploy:
    name: Deploy Stack
    needs: validate
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    
    steps:
      - uses: actions/checkout@v4
      
      - name: Configure AWS Credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: us-east-1

      - name: Deploy CloudFormation Stack
        run: |
          aws cloudformation deploy \
            --template-file template.yaml \
            --stack-name my-app-prod \
            --parameter-overrides \
              EnvironmentName=prod \
            --capabilities CAPABILITY_IAM \
            --no-fail-on-empty-changeset
```

### The `aws cloudformation deploy` Command

This is the **recommended** way to deploy. It's simpler than create-stack/update-stack because it handles both cases:

```bash
# deploy = create OR update (whichever is needed)
aws cloudformation deploy \
  --template-file template.yaml \
  --stack-name my-app \
  --parameter-overrides \
    Env=prod \
    InstanceType=t3.large \
  --capabilities CAPABILITY_IAM \
  --tags \
    Project=MyApp \
    Team=DevOps \
  --no-fail-on-empty-changeset    # Don't fail if nothing changed
```

## 22. Best Practices

### 1. Always Use Change Sets for Production

```bash
# NEVER do this in production:
aws cloudformation update-stack ...   # ❌ Direct update

# ALWAYS do this:
aws cloudformation create-change-set ...   # ✅ Preview first
aws cloudformation execute-change-set ...  # ✅ Then apply
```

### 2. Use DeletionPolicy on Critical Resources

```yaml
Database:
  Type: AWS::RDS::DBInstance
  DeletionPolicy: Snapshot        # ✅ Always snapshot before delete
```

### 3. Validate Templates Before Deploying

```bash
# AWS CLI validation
aws cloudformation validate-template --template-body file://template.yaml

# cfn-lint (more thorough)
pip install cfn-lint
cfn-lint template.yaml

# cfn-nag (security checks)
gem install cfn-nag
cfn_nag_scan --input-path template.yaml
```

### 4. Use Parameters and Mappings — Don't Hardcode

```yaml
# ❌ BAD — hardcoded values
InstanceType: t3.large

# ✅ GOOD — parameterized
InstanceType: !Ref InstanceTypeParam
```

### 5. Tag Everything

```yaml
Tags:
  - Key: Environment
    Value: !Ref EnvironmentName
  - Key: ManagedBy
    Value: CloudFormation
  - Key: Project
    Value: !Ref AWS::StackName
```

### 6. Use Exports Sparingly

Exports create **dependencies** between stacks. You can't delete a stack that has exports being used by another stack.

### 7. Template File Organization

```
cloudformation/
├── templates/
│   ├── root.yaml           # Parent stack
│   ├── networking.yaml      # VPC, subnets
│   ├── security.yaml        # SGs, IAM
│   └── application.yaml     # EC2, ALB, ASG
├── parameters/
│   ├── dev.json             # Dev parameters
│   ├── staging.json         # Staging parameters
│   └── prod.json            # Prod parameters
├── scripts/
│   ├── deploy.sh            # Deployment script
│   └── validate.sh          # Validation script
└── README.md
```

### 8. Use `aws cloudformation deploy` Instead of `create/update`

```bash
# This single command handles both create and update!
aws cloudformation deploy \
  --template-file template.yaml \
  --stack-name my-app \
  --no-fail-on-empty-changeset
```

## 23. Common Errors and Troubleshooting

### Error: "Template format error"

```
An error occurred: Template format error: [/Resources/MyBucket] 
Every template must include a Resources section
```

**Fix:** Check your YAML indentation! YAML is very sensitive to spaces.

### Error: "Insufficient IAM permissions"

```
API: ec2:RunInstances You are not authorized to perform this operation
```

**Fix:** Add `--capabilities CAPABILITY_IAM` to your CLI command, and make sure your IAM role/user has the necessary permissions.

### Error: "Resource limit exceeded"

```
Maximum number of resources per stack: 500
```

**Fix:** Break your template into nested stacks.

### Error: "Stack is in ROLLBACK_COMPLETE state"

```
Stack [my-app] is in ROLLBACK_COMPLETE state and cannot be updated
```

**Fix:** You must delete the stack and create a new one:

```bash
aws cloudformation delete-stack --stack-name my-app
aws cloudformation wait stack-delete-complete --stack-name my-app
aws cloudformation create-stack --stack-name my-app --template-body file://template.yaml
```

### Error: "Export with name already exists"

**Fix:** Export names must be unique per region. Use the stack name in export names:

```yaml
Export:
  Name: !Sub "${AWS::StackName}-VpcId"    # Unique per stack!
```

### Error: "Circular dependency"

**Fix:** Resources can't reference each other in a loop. Use `DependsOn` explicitly or reorganize your resources.

### Debugging Tips

```bash
# Check stack events (most recent first)
aws cloudformation describe-stack-events \
  --stack-name my-app \
  --query 'StackEvents[?ResourceStatus==`CREATE_FAILED`].[LogicalResourceId,ResourceStatusReason]' \
  --output table

# Get the template of an existing stack
aws cloudformation get-template --stack-name my-app

# Estimate the cost of a template
aws cloudformation estimate-template-cost \
  --template-body file://template.yaml
```

## 24. Summary

| Topic | Key Takeaway |
|-------|-------------|
| **What** | AWS's native IaC service |
| **Language** | YAML or JSON templates |
| **Cost** | Free! (you pay only for resources created) |
| **Scope** | AWS only |
| **Templates** | Blueprint defining desired infrastructure |
| **Stacks** | Running instance of a template |
| **StackSets** | Deploy to multiple accounts/regions at once |
| **Change Sets** | Preview before you apply (like `terraform plan`) |
| **Drift Detection** | Find resources changed outside CloudFormation |
| **Rollback** | Automatic on failure |
| **Key Functions** | `!Ref`, `!GetAtt`, `!Sub`, `!FindInMap`, `!If`, `!Join`, `!Select` |
| **Cross-Stack** | Export/ImportValue for sharing between stacks |
| **Best For** | AWS-only shops, multi-account enterprise deployments |

### One-Liner Summary

> **AWS CloudFormation = AWS's free, native IaC service that lets you define your entire AWS infrastructure in YAML, with automatic rollback, drift detection, and multi-account deployments via StackSets.**
