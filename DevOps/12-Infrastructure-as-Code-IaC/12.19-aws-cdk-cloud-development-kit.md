## 🤔 Why Do We Even Need CDK? A Story First

Imagine you've been writing CloudFormation templates for months now. Your YAML files are **5,000 lines long**. You can't use loops properly. You can't write an `if` statement like a normal programmer. You copy-paste the same blocks over and over. You dream about YAML indentation errors at night 😫.

One day, your teammate says:

> "What if we could just write infrastructure in **Python** or **TypeScript** — with real loops, real conditions, real functions — and have it automatically turn into CloudFormation?"

That's **exactly** what AWS CDK does. It's like having a **translator** — you write code in a language you love, and CDK translates it into CloudFormation JSON/YAML behind the scenes.

---

## 📜 What Exactly is AWS CDK?

**AWS CDK (Cloud Development Kit)** is an **open-source software development framework** from AWS that lets you define cloud infrastructure using familiar programming languages.

```
┌─────────────────────────────────────────────────────┐
│                   YOU WRITE                          │
│         TypeScript / Python / Java / C# / Go        │
│                 (CDK Code)                           │
└───────────────────────┬─────────────────────────────┘
                        │
                        ▼
              ┌─────────────────────┐
              │   CDK Synthesizes   │
              │   (cdk synth)       │
              └─────────┬───────────┘
                        │
                        ▼
              ┌─────────────────────┐
              │  CloudFormation     │
              │  Template (JSON)    │
              └─────────┬───────────┘
                        │
                        ▼
              ┌─────────────────────┐
              │  AWS Deploys It     │
              │  (CloudFormation)   │
              └─────────────────────┘
```

**Key point:** CDK is **NOT** a replacement for CloudFormation — it's a **layer on top of it**. It generates CloudFormation templates and uses CloudFormation to actually deploy.

## 🆚 CDK vs CloudFormation vs Terraform — Quick Comparison

| Feature | CloudFormation | Terraform | AWS CDK |
|---------|---------------|-----------|---------|
| **Language** | YAML / JSON | HCL | TypeScript, Python, Java, C#, Go |
| **Cloud Support** | AWS only | Multi-cloud | AWS only |
| **Abstraction Level** | Low (verbose) | Medium | High (constructs!) |
| **Loops / Conditions** | Limited (`Fn::If`) | `for_each`, `count` | Full programming language power |
| **State** | Managed by AWS | State file (you manage) | Managed by AWS (via CFN) |
| **Reusability** | Nested stacks | Modules | Constructs (OOP classes!) |
| **Learning Curve** | YAML pain 😩 | Learn HCL | Know a programming language ✅ |
| **Testing** | Hard | Terratest | Jest, pytest, etc. |

## 🧱 CDK Core Concepts — The Building Blocks

Before we write any code, let's understand the key concepts. Think of CDK like building with LEGO blocks:

### 1. **App** — The Entire LEGO Set

The App is the **root of everything**. It's the container that holds all your stacks.

```
App
 ├── Stack A (e.g., Networking)
 ├── Stack B (e.g., Compute)
 └── Stack C (e.g., Database)
```

### 2. **Stack** — A Box of Related LEGO Pieces

A Stack maps **directly** to a CloudFormation stack. It's a collection of related AWS resources. Each stack is deployed independently.

### 3. **Construct** — An Individual LEGO Piece (or a pre-built section!)

This is the **most important concept** in CDK. A construct is a building block that represents one or more AWS resources.

Constructs come in **3 levels**:

| Level | Name | What It Is | Example |
|-------|------|-----------|---------|
| **L1** | CFN Resources | 1:1 mapping to CloudFormation resources. Very verbose. | `CfnBucket` (you set every single property) |
| **L2** | Curated Constructs | Sensible defaults + helper methods. **Most commonly used.** | `Bucket` (auto-encrypts, adds policies easily) |
| **L3** | Patterns | Multiple resources combined into a common pattern. | `LambdaRestApi` (creates Lambda + API Gateway together!) |

Think of it like this:
- **L1** = Raw LEGO bricks (tiny, lots of manual work)
- **L2** = Pre-built LEGO sections (walls, doors — the sweet spot)
- **L3** = Pre-built LEGO buildings (a whole house in one piece!)

## 🛠️ Setting Up CDK — Let's Get Started!

### Prerequisites

You need:
1. **Node.js** (v14+ recommended) — CDK CLI is built on Node.js regardless of your app language
2. **AWS CLI** configured with credentials
3. **CDK CLI** installed globally

### Step 1: Install CDK CLI

```bash
# Install CDK globally
npm install -g aws-cdk

# Verify installation
cdk --version
# Output: 2.x.x (build xxxxxxx)
```

### Step 2: Bootstrap Your AWS Account

Before deploying anything, CDK needs to set up some resources in your AWS account (S3 bucket for assets, IAM roles, etc.). This is a **one-time setup per account per region**.

```bash
# Bootstrap CDK in your account
cdk bootstrap aws://ACCOUNT_ID/REGION

# Example:
cdk bootstrap aws://123456789012/us-east-1
```

**What does bootstrapping do?**
- Creates an S3 bucket to store CDK assets (Lambda code, Docker images, etc.)
- Creates IAM roles for CDK to deploy on your behalf
- Creates an ECR repository for Docker images
- You only do this **once per account/region combo**

### Step 3: Create a New CDK Project

```bash
# Create a new directory
mkdir my-cdk-app && cd my-cdk-app

# Initialize a CDK project (TypeScript)
cdk init app --language typescript

# For Python:
# cdk init app --language python
```

This generates a project structure:

```
my-cdk-app/
├── bin/
│   └── my-cdk-app.ts          ← Entry point (creates the App)
├── lib/
│   └── my-cdk-app-stack.ts    ← Your stack definition (THIS is where you code!)
├── test/
│   └── my-cdk-app.test.ts     ← Tests for your stack
├── cdk.json                    ← CDK configuration
├── package.json                ← Node.js dependencies
├── tsconfig.json               ← TypeScript config
└── node_modules/
```

## 🚀 Your First CDK App — Creating an S3 Bucket (TypeScript)

Let's create a simple S3 bucket. Open `lib/my-cdk-app-stack.ts`:

```typescript
// lib/my-cdk-app-stack.ts

import * as cdk from 'aws-cdk-lib';
import * as s3 from 'aws-cdk-lib/aws-s3';
import { Construct } from 'constructs';

// Every stack extends cdk.Stack
export class MyCdkAppStack extends cdk.Stack {
  constructor(scope: Construct, id: string, props?: cdk.StackProps) {
    super(scope, id, props);

    // 🎯 Create an S3 Bucket — just ONE line of real config!
    const myBucket = new s3.Bucket(this, 'MyFirstBucket', {
      bucketName: 'my-awesome-cdk-bucket-2024',   // Bucket name
      versioned: true,                              // Enable versioning
      encryption: s3.BucketEncryption.S3_MANAGED,   // Enable encryption
      removalPolicy: cdk.RemovalPolicy.DESTROY,     // Delete bucket when stack is deleted
      autoDeleteObjects: true,                       // Auto-delete objects on bucket removal
    });
  }
}
```

**Let's break down what's happening:**

| Line | What It Does |
|------|-------------|
| `import * as s3 from 'aws-cdk-lib/aws-s3'` | Import the S3 module from CDK library |
| `export class MyCdkAppStack extends cdk.Stack` | Define a stack (maps to a CloudFormation stack) |
| `new s3.Bucket(this, 'MyFirstBucket', {...})` | Create an S3 bucket (L2 construct — smart defaults!) |
| `this` | The scope — this bucket belongs to THIS stack |
| `'MyFirstBucket'` | Logical ID — unique identifier within the stack |
| `versioned: true` | Turns on S3 versioning |
| `removalPolicy: cdk.RemovalPolicy.DESTROY` | When you destroy the stack, delete the bucket too |

**Compare this to CloudFormation YAML for the same thing:**

```yaml
# CloudFormation — 20+ lines for the same thing!
Resources:
  MyFirstBucket:
    Type: AWS::S3::Bucket
    Properties:
      BucketName: my-awesome-cdk-bucket-2024
      VersioningConfiguration:
        Status: Enabled
      BucketEncryption:
        ServerSideEncryptionConfiguration:
          - ServerSideEncryptionByDefault:
              SSEAlgorithm: AES256
    DeletionPolicy: Delete
  # Plus you'd need a custom resource for auto-delete objects...
```

See how much **cleaner** CDK is? 🎉

## 🔧 CDK CLI Commands — Your Daily Toolkit

Here are the commands you'll use every day:

```bash
# 1. Synthesize — Generate the CloudFormation template (don't deploy yet)
cdk synth
# This creates the cdk.out/ folder with the CloudFormation JSON

# 2. Diff — See what changes will happen BEFORE deploying
cdk diff
# Shows added/removed/changed resources (like terraform plan!)

# 3. Deploy — Actually deploy your stack to AWS
cdk deploy
# Deploys the CloudFormation stack

# 4. Deploy a specific stack (when you have multiple stacks)
cdk deploy MyNetworkStack

# 5. Deploy ALL stacks
cdk deploy --all

# 6. Destroy — Tear down the stack
cdk destroy

# 7. List — Show all stacks in the app
cdk ls

# 8. Doctor — Check for common issues
cdk doctor
```

**The typical CDK workflow:**

```
Write Code → cdk synth → cdk diff → cdk deploy → 🎉
```

## 🏗️ Real-World Example — VPC + EC2 + Security Group

Let's build something more realistic — a VPC with a public subnet, security group, and an EC2 instance:

```typescript
// lib/my-infra-stack.ts

import * as cdk from 'aws-cdk-lib';
import * as ec2 from 'aws-cdk-lib/aws-ec2';
import { Construct } from 'constructs';

export class MyInfraStack extends cdk.Stack {
  constructor(scope: Construct, id: string, props?: cdk.StackProps) {
    super(scope, id, props);

    // ========== 1. Create a VPC ==========
    // L2 construct — creates VPC, subnets, route tables, NAT gateways, IGW automatically!
    const vpc = new ec2.Vpc(this, 'MyVpc', {
      maxAzs: 2,                           // Use 2 Availability Zones
      natGateways: 1,                       // 1 NAT Gateway (saves cost)
      subnetConfiguration: [
        {
          name: 'Public',
          subnetType: ec2.SubnetType.PUBLIC,
          cidrMask: 24,                     // /24 subnet = 256 IPs
        },
        {
          name: 'Private',
          subnetType: ec2.SubnetType.PRIVATE_WITH_EGRESS,
          cidrMask: 24,
        },
      ],
    });

    // ========== 2. Create a Security Group ==========
    const webSg = new ec2.SecurityGroup(this, 'WebSecurityGroup', {
      vpc: vpc,                             // Attach to our VPC
      description: 'Allow HTTP and SSH access',
      allowAllOutbound: true,               // Allow all outbound traffic
    });

    // Allow HTTP from anywhere
    webSg.addIngressRule(
      ec2.Peer.anyIpv4(),                   // From: 0.0.0.0/0
      ec2.Port.tcp(80),                     // Port: 80
      'Allow HTTP from anywhere'            // Description
    );

    // Allow SSH from anywhere (in production, restrict this!)
    webSg.addIngressRule(
      ec2.Peer.anyIpv4(),
      ec2.Port.tcp(22),
      'Allow SSH from anywhere'
    );

    // ========== 3. Create an EC2 Instance ==========
    const instance = new ec2.Instance(this, 'WebServer', {
      vpc: vpc,
      vpcSubnets: { subnetType: ec2.SubnetType.PUBLIC },
      instanceType: ec2.InstanceType.of(
        ec2.InstanceClass.T3,               // Instance family
        ec2.InstanceSize.MICRO              // Instance size
      ),
      machineImage: ec2.MachineImage.latestAmazonLinux2(), // Amazon Linux 2 AMI
      securityGroup: webSg,
      keyName: 'my-key-pair',               // Your SSH key pair name
    });

    // ========== 4. Add User Data (run commands on boot) ==========
    instance.addUserData(
      'yum update -y',
      'yum install -y httpd',
      'systemctl start httpd',
      'systemctl enable httpd',
      'echo "<h1>Hello from CDK!</h1>" > /var/www/html/index.html'
    );

    // ========== 5. Output the public IP ==========
    new cdk.CfnOutput(this, 'InstancePublicIP', {
      value: instance.instancePublicIp,
      description: 'Public IP of the web server',
    });
  }
}
```

**Look how powerful this is!** With `new ec2.Vpc(...)`, CDK automatically creates:
- The VPC itself
- Public subnets (with Internet Gateway and route tables)
- Private subnets (with NAT Gateway and route tables)
- Route tables with proper routes
- All the wiring between them

In CloudFormation, this would be **200+ lines of YAML**. In CDK? About **10 lines**. That's the magic of L2 constructs! 🪄

## 🔗 Cross-Stack References — Sharing Resources Between Stacks

In real projects, you split infrastructure into multiple stacks. For example:
- **NetworkStack** — VPC, subnets
- **ComputeStack** — EC2 instances (needs the VPC from NetworkStack)

```typescript
// ==========================================
// lib/network-stack.ts — Creates the VPC
// ==========================================
import * as cdk from 'aws-cdk-lib';
import * as ec2 from 'aws-cdk-lib/aws-ec2';
import { Construct } from 'constructs';

export class NetworkStack extends cdk.Stack {
  // 👇 Expose the VPC as a public property so other stacks can use it
  public readonly vpc: ec2.Vpc;

  constructor(scope: Construct, id: string, props?: cdk.StackProps) {
    super(scope, id, props);

    this.vpc = new ec2.Vpc(this, 'SharedVpc', {
      maxAzs: 2,
    });
  }
}

// ==========================================
// lib/compute-stack.ts — Uses the VPC
// ==========================================
import * as cdk from 'aws-cdk-lib';
import * as ec2 from 'aws-cdk-lib/aws-ec2';
import { Construct } from 'constructs';

// Custom props interface to accept the VPC
interface ComputeStackProps extends cdk.StackProps {
  vpc: ec2.Vpc;
}

export class ComputeStack extends cdk.Stack {
  constructor(scope: Construct, id: string, props: ComputeStackProps) {
    super(scope, id, props);

    // 👇 Use the VPC from NetworkStack
    new ec2.Instance(this, 'WebServer', {
      vpc: props.vpc,   // <-- VPC passed from another stack!
      instanceType: ec2.InstanceType.of(ec2.InstanceClass.T3, ec2.InstanceSize.MICRO),
      machineImage: ec2.MachineImage.latestAmazonLinux2(),
    });
  }
}

// ==========================================
// bin/my-cdk-app.ts — Wire them together
// ==========================================
import * as cdk from 'aws-cdk-lib';
import { NetworkStack } from '../lib/network-stack';
import { ComputeStack } from '../lib/compute-stack';

const app = new cdk.App();

// Create NetworkStack first
const networkStack = new NetworkStack(app, 'NetworkStack');

// Pass the VPC to ComputeStack
new ComputeStack(app, 'ComputeStack', {
  vpc: networkStack.vpc,   // 👈 Cross-stack reference!
});
```

CDK automatically handles CloudFormation exports/imports under the hood. You just pass objects around like normal programming! 🎯

## 📦 Creating Custom Constructs — Reusable Components

One of CDK's superpowers is creating **reusable constructs** — like creating your own LEGO pieces:

```typescript
// lib/constructs/secure-bucket.ts

import * as cdk from 'aws-cdk-lib';
import * as s3 from 'aws-cdk-lib/aws-s3';
import { Construct } from 'constructs';

// Define properties for our custom construct
interface SecureBucketProps {
  bucketName?: string;
  expirationDays?: number;
}

// Our custom construct — a "SecureBucket" with best practices built-in
export class SecureBucket extends Construct {
  public readonly bucket: s3.Bucket;

  constructor(scope: Construct, id: string, props?: SecureBucketProps) {
    super(scope, id);

    this.bucket = new s3.Bucket(this, 'Bucket', {
      bucketName: props?.bucketName,
      // 🔒 Security best practices baked in
      encryption: s3.BucketEncryption.S3_MANAGED,
      blockPublicAccess: s3.BlockPublicAccess.BLOCK_ALL,
      enforceSSL: true,
      versioned: true,
      // 🗑️ Lifecycle rules
      lifecycleRules: [
        {
          expiration: cdk.Duration.days(props?.expirationDays ?? 90),
        },
      ],
      // 📝 Access logging
      serverAccessLogsPrefix: 'access-logs/',
    });
  }
}
```

Now use it in any stack:

```typescript
// lib/my-stack.ts

import { SecureBucket } from './constructs/secure-bucket';

export class MyStack extends cdk.Stack {
  constructor(scope: Construct, id: string, props?: cdk.StackProps) {
    super(scope, id, props);

    // One line to create a fully secure, compliant bucket!
    const dataBucket = new SecureBucket(this, 'DataBucket', {
      bucketName: 'my-secure-data-2024',
      expirationDays: 365,
    });
  }
}
```

**Why is this amazing?** 
- Your security team defines the `SecureBucket` construct with all compliance rules
- Developers just use `new SecureBucket(...)` — they can't accidentally create insecure buckets
- Update the construct once → all buckets across all projects get the update

## 🧪 Testing CDK Code — Yes, You Can Test Infrastructure!

One of the biggest advantages of CDK over YAML — you can write **real unit tests**!

### Using TypeScript + Jest:

```typescript
// test/my-cdk-app.test.ts

import * as cdk from 'aws-cdk-lib';
import { Template } from 'aws-cdk-lib/assertions';
import { MyCdkAppStack } from '../lib/my-cdk-app-stack';

describe('MyCdkAppStack', () => {
  let template: Template;

  beforeAll(() => {
    // Create the stack
    const app = new cdk.App();
    const stack = new MyCdkAppStack(app, 'TestStack');
    // Get the CloudFormation template
    template = Template.fromStack(stack);
  });

  // Test 1: Does our stack create an S3 bucket?
  test('should create an S3 bucket', () => {
    template.resourceCountIs('AWS::S3::Bucket', 1);
  });

  // Test 2: Is versioning enabled?
  test('S3 bucket should have versioning enabled', () => {
    template.hasResourceProperties('AWS::S3::Bucket', {
      VersioningConfiguration: {
        Status: 'Enabled',
      },
    });
  });

  // Test 3: Is encryption enabled?
  test('S3 bucket should have encryption', () => {
    template.hasResourceProperties('AWS::S3::Bucket', {
      BucketEncryption: {
        ServerSideEncryptionConfiguration: [
          {
            ServerSideEncryptionByDefault: {
              SSEAlgorithm: 'AES256',
            },
          },
        ],
      },
    });
  });

  // Test 4: Snapshot test — detect any unexpected changes
  test('should match snapshot', () => {
    expect(template.toJSON()).toMatchSnapshot();
  });
});
```

Run tests:

```bash
# Run tests
npm test

# Output:
# PASS  test/my-cdk-app.test.ts
#   MyCdkAppStack
#     ✓ should create an S3 bucket (5 ms)
#     ✓ S3 bucket should have versioning enabled (3 ms)
#     ✓ S3 bucket should have encryption (2 ms)
#     ✓ should match snapshot (4 ms)
```

**This is HUGE!** You can catch infrastructure bugs **before deploying**. Try doing this with YAML... 😅

## 🔐 CDK and IAM — Permissions Made Easy

CDK has **grant methods** that make IAM so much simpler:

```typescript
import * as s3 from 'aws-cdk-lib/aws-s3';
import * as lambda from 'aws-cdk-lib/aws-lambda';

// Create a bucket and Lambda function
const bucket = new s3.Bucket(this, 'DataBucket');

const myFunction = new lambda.Function(this, 'MyFunction', {
  runtime: lambda.Runtime.NODEJS_18_X,
  handler: 'index.handler',
  code: lambda.Code.fromAsset('lambda/'),
});

// 🎯 One line to grant read/write access!
bucket.grantReadWrite(myFunction);
// CDK automatically:
// 1. Creates the IAM policy with s3:GetObject, s3:PutObject, etc.
// 2. Attaches it to the Lambda's execution role
// 3. Adds the bucket ARN as the resource

// Compare to doing this manually in CloudFormation:
// - Create IAM Policy document (10+ lines)
// - Create IAM Role (5+ lines)
// - Attach policy to role (5+ lines)
// - Reference role in Lambda (2+ lines)
// CDK does ALL of this with ONE LINE! 🤯
```

Other grant methods:
```typescript
bucket.grantRead(myFunction);         // Read-only access
bucket.grantWrite(myFunction);        // Write-only access
bucket.grantDelete(myFunction);       // Delete access
bucket.grantPublicAccess();           // Make bucket public (careful!)

// Works for other services too:
table.grantReadWriteData(myFunction); // DynamoDB
queue.grantSendMessages(myFunction);  // SQS
topic.grantPublish(myFunction);       // SNS
```

## 🌐 CDK Pipelines — Self-Mutating CI/CD

CDK Pipelines is a construct that creates a CI/CD pipeline which **automatically deploys your CDK app** — and even **updates itself** when you change the pipeline code!

```typescript
import * as cdk from 'aws-cdk-lib';
import { CodePipeline, CodePipelineSource, ShellStep } from 'aws-cdk-lib/pipelines';

export class PipelineStack extends cdk.Stack {
  constructor(scope: Construct, id: string, props?: cdk.StackProps) {
    super(scope, id, props);

    const pipeline = new CodePipeline(this, 'Pipeline', {
      pipelineName: 'MyAppPipeline',
      
      // Source: GitHub repository
      synth: new ShellStep('Synth', {
        input: CodePipelineSource.gitHub('my-org/my-repo', 'main'),
        commands: [
          'npm ci',           // Install dependencies
          'npm run build',    // Compile TypeScript
          'npx cdk synth',    // Generate CloudFormation templates
        ],
      }),
    });

    // Add deployment stages
    pipeline.addStage(new MyApplicationStage(this, 'Dev', {
      env: { account: '111111111111', region: 'us-east-1' },
    }));

    pipeline.addStage(new MyApplicationStage(this, 'Prod', {
      env: { account: '222222222222', region: 'us-east-1' },
    }));
  }
}
```

## 📂 Best Practices for CDK Projects

### 1. Project Structure

```
my-cdk-project/
├── bin/
│   └── app.ts                    ← App entry point
├── lib/
│   ├── stacks/
│   │   ├── network-stack.ts      ← One file per stack
│   │   ├── compute-stack.ts
│   │   └── database-stack.ts
│   └── constructs/
│       ├── secure-bucket.ts      ← Reusable constructs
│       └── standard-vpc.ts
├── test/
│   ├── network-stack.test.ts     ← Tests per stack
│   └── compute-stack.test.ts
├── lambda/                        ← Lambda function code
│   └── handler.ts
├── cdk.json
└── package.json
```

### 2. Do's and Don'ts

| ✅ Do | ❌ Don't |
|-------|----------|
| Use L2 constructs whenever possible | Drop down to L1 (CfnXxx) unless you must |
| Create custom constructs for reuse | Copy-paste resource definitions |
| Write tests for your stacks | Deploy without testing |
| Use `cdk diff` before every deploy | Deploy blindly |
| Tag all resources | Leave resources untagged |
| Use environment-specific stacks | Hardcode account/region |
| Store CDK code in version control | Make manual changes in console |

### 3. Tagging All Resources

```typescript
// Apply tags to everything in the stack
cdk.Tags.of(this).add('Environment', 'production');
cdk.Tags.of(this).add('Team', 'platform');
cdk.Tags.of(this).add('ManagedBy', 'cdk');
```

## 🆚 CDK v1 vs CDK v2

| Feature | CDK v1 (Legacy) | CDK v2 (Current) |
|---------|-----------------|-------------------|
| **Status** | ⚠️ End of support (June 2023) | ✅ Active development |
| **Packages** | Separate package per service (`@aws-cdk/aws-s3`) | Single package (`aws-cdk-lib`) |
| **Imports** | `import * as s3 from '@aws-cdk/aws-s3'` | `import * as s3 from 'aws-cdk-lib/aws-s3'` |
| **Constructs** | Built-in | Separate `constructs` package |

**Always use CDK v2!** If you see tutorials with `@aws-cdk/aws-xxx` imports, they're v1 — use `aws-cdk-lib/aws-xxx` instead.

## 🧠 Key Takeaways

✅ **AWS CDK** lets you write infrastructure in real programming languages (TypeScript, Python, Java, C#, Go)

✅ It generates **CloudFormation templates** under the hood — CDK is a layer ON TOP of CloudFormation

✅ **Constructs** are the key abstraction — L1 (raw), L2 (smart defaults), L3 (patterns)

✅ You get **loops, conditions, classes, functions** — everything a programming language offers

✅ **Grant methods** make IAM permissions dead simple (`bucket.grantReadWrite(lambda)`)

✅ You can write **real unit tests** for your infrastructure using Jest, pytest, etc.

✅ **Custom constructs** let you package and share infrastructure patterns across your org

✅ Use `cdk synth` → `cdk diff` → `cdk deploy` as your deployment workflow

✅ CDK is **AWS-only** — for multi-cloud, look at Terraform, Pulumi, or Crossplane
