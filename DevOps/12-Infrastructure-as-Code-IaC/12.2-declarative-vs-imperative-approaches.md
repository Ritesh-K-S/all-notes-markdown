# 12.2 — Declarative vs Imperative Approaches

> **One Line Answer:** **Imperative** means you tell the computer **HOW** to do something (step-by-step instructions), while **Declarative** means you tell the computer **WHAT** you want (the end result) and it figures out the HOW by itself — and most modern IaC tools use the **declarative** approach because it's simpler, safer, and more predictable.

---

## 🤔 Let's Start With a Real-Life Analogy

Imagine you walk into a restaurant. You're hungry and you want a pizza. 🍕

### 🔵 The Imperative Way (HOW):

You walk into the kitchen and start giving the chef step-by-step instructions:

```
1. Take 200g of flour
2. Add 100ml of water
3. Add 5g of yeast
4. Mix it together
5. Knead the dough for 10 minutes
6. Let it rise for 1 hour
7. Roll the dough into a circle
8. Spread tomato sauce on top
9. Add mozzarella cheese
10. Add pepperoni
11. Put it in the oven at 450°F
12. Bake for 12 minutes
13. Take it out
14. Slice it into 8 pieces
15. Serve on a plate
```

You told the chef **exactly HOW** to make the pizza — every single step.

### 🟢 The Declarative Way (WHAT):

You sit down at a table and tell the waiter:

```
"I want a large pepperoni pizza, please."
```

That's it. You told the waiter **WHAT** you want. The chef knows HOW to make it. You don't care about the steps — you just care about the **end result**.

**That's the core difference between imperative and declarative.**

## 💻 Now Let's See This in Infrastructure Terms

Let's say you need **3 web servers running Nginx** in AWS.

### 🔵 Imperative Approach — Step-by-Step Commands

With an imperative approach, you write a **script** that tells the computer exactly what to do, step by step:

```bash
#!/bin/bash
# imperative-setup.sh — Create 3 web servers (imperative style)

echo "Creating web servers..."

# Step 1: Create server 1
aws ec2 run-instances \
  --image-id ami-0c55b159cbfafe1f0 \
  --instance-type t2.medium \
  --count 1 \
  --tag-specifications 'ResourceType=instance,Tags=[{Key=Name,Value=web-server-1}]'

# Step 2: Create server 2
aws ec2 run-instances \
  --image-id ami-0c55b159cbfafe1f0 \
  --instance-type t2.medium \
  --count 1 \
  --tag-specifications 'ResourceType=instance,Tags=[{Key=Name,Value=web-server-2}]'

# Step 3: Create server 3
aws ec2 run-instances \
  --image-id ami-0c55b159cbfafe1f0 \
  --instance-type t2.medium \
  --count 1 \
  --tag-specifications 'ResourceType=instance,Tags=[{Key=Name,Value=web-server-3}]'

echo "Installing Nginx on each server..."

# Step 4: SSH into server 1 and install Nginx
ssh ubuntu@<server-1-ip> "sudo apt update && sudo apt install -y nginx && sudo systemctl start nginx"

# Step 5: SSH into server 2 and install Nginx
ssh ubuntu@<server-2-ip> "sudo apt update && sudo apt install -y nginx && sudo systemctl start nginx"

# Step 6: SSH into server 3 and install Nginx
ssh ubuntu@<server-3-ip> "sudo apt update && sudo apt install -y nginx && sudo systemctl start nginx"

echo "Done! 3 web servers with Nginx are running."
```

**What's happening here:**
- You're telling the computer **every single step** — create this, then create that, then SSH here, then run this command
- YOU are responsible for the order, the logic, and handling everything

### The Big Problem — What Happens If You Run This Script Again?

```bash
# First run:  → Creates 3 servers ✅
# Second run: → Creates 3 MORE servers = Now you have 6! 💥
# Third run:  → Creates 3 MORE servers = Now you have 9! 💥💥💥
```

The script doesn't know that servers already exist. It just blindly runs the commands every time. You'd have to add extra logic:

```bash
# Now you need to handle this yourself:
EXISTING_SERVERS=$(aws ec2 describe-instances --filters "Name=tag:Name,Values=web-server-*" --query "Reservations[].Instances[].InstanceId" --output text)
EXISTING_COUNT=$(echo $EXISTING_SERVERS | wc -w)

if [ "$EXISTING_COUNT" -lt 3 ]; then
    NEED=$((3 - EXISTING_COUNT))
    echo "Need to create $NEED more servers..."
    # ... create only the missing ones
elif [ "$EXISTING_COUNT" -gt 3 ]; then
    echo "Too many servers! Need to delete some..."
    # ... delete the extra ones
else
    echo "Already have 3 servers. Nothing to do."
fi
```

See how complicated this gets? You're writing **all the logic** yourself — checking current state, figuring out what needs to change, handling edge cases...

### 🟢 Declarative Approach — Just Describe What You Want

With a declarative approach, you simply **describe the desired end state**:

```hcl
# main.tf — Terraform (Declarative)
# "I want 3 web servers running Nginx. Make it happen."

resource "aws_instance" "web_server" {
  count         = 3                          # I want exactly 3 servers
  ami           = "ami-0c55b159cbfafe1f0"    # Ubuntu 22.04
  instance_type = "t2.medium"               # 2 vCPUs, 4GB RAM

  user_data = <<-EOF
    #!/bin/bash
    apt update
    apt install -y nginx
    systemctl start nginx
    systemctl enable nginx
  EOF

  tags = {
    Name = "web-server-${count.index + 1}"   # web-server-1, web-server-2, web-server-3
  }
}
```

**What's happening here:**

| Line | What It Means |
|------|-------------|
| `count = 3` | "I want exactly 3 of these servers" |
| `ami = "ami-..."` | "Use this Ubuntu image" |
| `instance_type = "t2.medium"` | "Each should be this size" |
| `user_data = ...` | "After the server boots, run this script to install Nginx" |
| `Name = "web-server-${count.index + 1}"` | "Name them web-server-1, web-server-2, web-server-3" |

**Now, the magic — what happens when you run this multiple times?**

```bash
# First run:
terraform apply
# Terraform: "You want 3 servers. You have 0. I'll create 3." ✅
# Result: 3 servers

# Second run:
terraform apply
# Terraform: "You want 3 servers. You already have 3. Nothing to do." ✅
# Result: Still 3 servers (no changes!)

# Change count to 5 and run again:
terraform apply
# Terraform: "You want 5 servers. You have 3. I'll create 2 more." ✅
# Result: 5 servers

# Change count back to 3:
terraform apply
# Terraform: "You want 3 servers. You have 5. I'll delete 2." ✅
# Result: 3 servers
```

**Terraform figures out the HOW automatically!** You just tell it WHAT you want, and it:
1. Checks the **current state** (what exists now)
2. Compares it to the **desired state** (what you wrote in code)
3. Calculates the **difference** (what needs to change)
4. Makes **only the necessary changes**

This is called **"desired state management"** — and it's the superpower of declarative IaC.

## 📊 Side-by-Side Comparison

| Aspect | 🔵 Imperative | 🟢 Declarative |
|--------|--------------|-----------------|
| **You define** | HOW to do it (steps) | WHAT you want (end result) |
| **Example** | "Create a server, then install Nginx, then open port 80" | "I want a server with Nginx on port 80" |
| **Re-running** | Might create duplicates or fail | Safe to run multiple times (idempotent) |
| **State awareness** | You manage state manually | Tool tracks state automatically |
| **Complexity** | You handle all the logic | Tool handles the logic |
| **Order of operations** | You define the order | Tool figures out the order |
| **Error handling** | You write error handling code | Tool handles errors and rollbacks |
| **Learning curve** | Familiar (like regular scripting) | New concept to learn |
| **Flexibility** | Very flexible (full programming) | Less flexible (limited to what tool supports) |
| **Readability** | Can get messy with complex logic | Clean and easy to read |

## 🧪 Idempotency — The Key Concept

One of the biggest advantages of declarative tools is **idempotency**.

### What Does Idempotent Mean?

> **Idempotent** = Running the same operation **multiple times** produces the **same result** as running it once.

**Real-life example:**
- **Not idempotent:** "Add a spoon of sugar to my coffee" → Run it 5 times = 5 spoons of sugar! ☠️
- **Idempotent:** "Make sure my coffee has exactly 1 spoon of sugar" → Run it 5 times = still 1 spoon ✅

**In infrastructure terms:**

```
🔵 Imperative (NOT idempotent):
   "Create a server" → Run 5 times = 5 servers 💥

🟢 Declarative (Idempotent):
   "I want 1 server" → Run 5 times = still 1 server ✅
```

This is HUGE for reliability. You can safely re-run your IaC code without fear of breaking things.

## 🔍 Declarative vs Imperative — With Real Tool Examples

Let's see how different tools approach the same task: **"Install and start Nginx on a server."**

### 🔵 Imperative — Bash Script

```bash
#!/bin/bash
# You tell the system EXACTLY what to do, step by step

# Step 1: Update package list
sudo apt update

# Step 2: Install nginx
sudo apt install -y nginx

# Step 3: Start nginx
sudo systemctl start nginx

# Step 4: Enable nginx to start on boot
sudo systemctl enable nginx

# Step 5: Check if it's running
sudo systemctl status nginx
```

**Problems:**
- If nginx is already installed, `apt install` runs again (wastes time, might cause issues)
- If nginx is already running, `systemctl start` might error
- No awareness of current state
- Run it 10 times? It tries all steps 10 times

---

### 🟢 Declarative — Ansible Playbook

```yaml
# nginx-setup.yml — Ansible Playbook (Declarative)
---
- name: Ensure Nginx is installed and running
  hosts: web_servers
  become: yes      # Run as root (sudo)

  tasks:
    - name: Ensure nginx is installed
      apt:
        name: nginx
        state: present        # "Make sure nginx IS installed"
        update_cache: yes     # "Make sure package cache is updated"

    - name: Ensure nginx is running and enabled
      service:
        name: nginx
        state: started        # "Make sure nginx IS running"
        enabled: yes          # "Make sure nginx WILL start on boot"
```

**What's happening here (line by line):**

| Line | What It Means |
|------|-------------|
| `hosts: web_servers` | Run this on all servers in the `web_servers` group |
| `become: yes` | Use sudo (run as root) |
| `state: present` | "Make sure nginx **is installed**" — if already installed, do nothing |
| `update_cache: yes` | "Make sure the package list is up to date" |
| `state: started` | "Make sure nginx **is running**" — if already running, do nothing |
| `enabled: yes` | "Make sure nginx **starts on boot**" — if already enabled, do nothing |

**The Magic:**
```
First run:  "Nginx not installed → installing. Not running → starting." ✅ (changes made)
Second run: "Nginx already installed. Already running. Already enabled." ✅ (no changes!)
Third run:  Same as second. No changes. ✅
```

See the keyword `state`? That's the declarative approach. You're saying **"this is the state I want"**, not "do this step."

### 🟢 Declarative — Terraform

```hcl
# main.tf — Terraform (Declarative)
# "I want this infrastructure to exist"

# I want a VPC
resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"

  tags = {
    Name = "my-vpc"
  }
}

# I want a subnet inside that VPC
resource "aws_subnet" "web" {
  vpc_id     = aws_vpc.main.id    # Put it in the VPC I just defined
  cidr_block = "10.0.1.0/24"

  tags = {
    Name = "web-subnet"
  }
}

# I want a security group that allows HTTP traffic
resource "aws_security_group" "web_sg" {
  name   = "web-sg"
  vpc_id = aws_vpc.main.id

  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]    # Allow HTTP from anywhere
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]    # Allow all outbound traffic
  }
}

# I want a web server in that subnet with that security group
resource "aws_instance" "web" {
  ami                    = "ami-0c55b159cbfafe1f0"
  instance_type          = "t2.medium"
  subnet_id              = aws_subnet.web.id
  vpc_security_group_ids = [aws_security_group.web_sg.id]

  tags = {
    Name = "web-server"
  }
}
```

**What's happening here:**
- You **declared** 4 resources: a VPC, a subnet, a security group, and an EC2 instance
- You didn't specify the ORDER of creation — Terraform figures that out automatically
- Terraform knows that the subnet needs the VPC first, and the EC2 instance needs the subnet and security group
- It builds a **dependency graph** and creates everything in the right order

```
Terraform's Dependency Graph (figured out automatically):

    aws_vpc.main
        ↓
    ┌───┴───────────────┐
    ↓                   ↓
aws_subnet.web    aws_security_group.web_sg
    ↓                   ↓
    └───────┬───────────┘
            ↓
    aws_instance.web
```

## 🔄 The Terraform Workflow — How Declarative Works in Practice

Here's how Terraform uses the declarative approach in a real workflow:

```
┌─────────────────────────────────────────────────────────┐
│                 Terraform Workflow                        │
│                                                          │
│  1. You write code:   main.tf                            │
│     "I want 3 servers, 1 VPC, 2 subnets"                │
│                                                          │
│  2. terraform plan                                       │
│     Terraform compares:                                  │
│     ┌──────────────┐    ┌──────────────┐                │
│     │ Desired State │ vs │ Current State│                │
│     │ (your code)   │    │ (state file) │                │
│     └──────────────┘    └──────────────┘                │
│            ↓                                             │
│     "I need to create 3 servers, 1 VPC, 2 subnets"      │
│                                                          │
│  3. terraform apply                                      │
│     Terraform makes the changes                          │
│     Updates the state file                               │
│                                                          │
│  4. Next time you run terraform plan:                    │
│     "Desired state matches current state.                │
│      No changes needed." ✅                              │
└─────────────────────────────────────────────────────────┘
```

### The State File

The "secret sauce" of declarative IaC tools like Terraform is the **state file**. It keeps track of what currently exists in the real world.

```
┌────────────────┐     ┌──────────────────┐     ┌────────────────┐
│   Your Code    │     │   State File     │     │   Real World   │
│   (Desired)    │     │ (What Terraform  │     │  (AWS/Azure/   │
│                │     │  thinks exists)  │     │   GCP)         │
│ 3 servers      │     │ 3 servers        │     │ 3 servers      │
│ 1 VPC          │     │ 1 VPC            │     │ 1 VPC          │
│ 2 subnets      │     │ 2 subnets        │     │ 2 subnets      │
└────────────────┘     └──────────────────┘     └────────────────┘
        ↕                      ↕                        ↕
   terraform plan         terraform              actual cloud
   compares these        state file              infrastructure
```

When you run `terraform plan`, it compares:
1. **Your code** (what you want) vs **State file** (what Terraform thinks exists)
2. Calculates the difference
3. Shows you a plan of what will change

When you run `terraform apply`:
1. Makes the changes in the **real world** (AWS/Azure/GCP)
2. Updates the **state file** to match

## 🧩 When to Use Which Approach?

Both approaches have their place. Here's a practical guide:

### Use Declarative When:

| Scenario | Why Declarative |
|----------|----------------|
| Managing cloud infrastructure (VMs, networks, databases) | You want consistent, repeatable infrastructure |
| Setting up Kubernetes resources | YAML manifests are declarative by nature |
| Configuring servers at scale (100+ servers) | Idempotency ensures consistency |
| Team collaboration | Code is readable, reviewable, and predictable |
| Production environments | Safety of idempotency prevents accidental changes |

### Use Imperative When:

| Scenario | Why Imperative |
|----------|----------------|
| One-time setup scripts | Quick and straightforward |
| Complex logic with conditions | Sometimes you need if/else logic that declarative can't handle |
| Migration scripts | "Move data from old DB to new DB" is inherently step-by-step |
| Quick debugging/fixes | SSH in and run a command is faster than writing IaC |
| Glue scripts | Connecting different tools together |

### The Real-World Answer:

Most teams use **BOTH**:

```
Production infrastructure → Declarative (Terraform)
Server configuration     → Declarative (Ansible)
CI/CD pipelines          → Mix of both
One-time migrations      → Imperative (Bash/Python scripts)
Quick debugging          → Imperative (SSH + commands)
```

## 🎓 A Kubernetes Example — Declarative in Action

Kubernetes is a great example of declarative infrastructure. You've probably seen this already:

```yaml
# deployment.yaml — Kubernetes Deployment (Declarative)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
spec:
  replicas: 3                    # "I want 3 pods running"
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.25        # "Each pod should run this image"
        ports:
        - containerPort: 80      # "Expose port 80"
```

**What's happening here:**
- You're NOT saying "create pod 1, then create pod 2, then create pod 3"
- You're saying "I want 3 replicas of this pod running at all times"
- Kubernetes **continuously** works to match reality with your declaration:

```
You declare: replicas: 3

Kubernetes checks:
  - 3 pods running? → Do nothing ✅
  - Only 2 pods running? (one crashed) → Create 1 more ✅
  - 4 pods running? (somehow) → Delete 1 ✅
  - 0 pods running? → Create 3 ✅
```

This is called a **reconciliation loop** — the system constantly checks reality against the desired state and makes corrections. Terraform does the same thing when you run `terraform apply`.

## 📝 Key Takeaways

```
✅ Imperative = You describe HOW (step-by-step instructions)
   → Like a recipe: "Do step 1, then step 2, then step 3"
   → Tools: Bash scripts, Python scripts, AWS CLI
   → Problem: Not idempotent, doesn't track state, order matters

✅ Declarative = You describe WHAT (desired end state)
   → Like ordering food: "I want a pizza"
   → Tools: Terraform, Ansible, CloudFormation, Kubernetes YAML
   → Benefit: Idempotent, tracks state, tool figures out the order

✅ Idempotent = Running the same thing multiple times 
   gives the SAME result as running it once
   → Declarative tools are idempotent by design
   → Imperative scripts need extra work to be idempotent

✅ State File = How declarative tools track what exists
   → Compares desired state (code) vs current state (state file)
   → Calculates the difference and makes only necessary changes

✅ Most teams use BOTH approaches:
   → Declarative for infrastructure and configuration
   → Imperative for one-time scripts and migrations

✅ The industry has moved HEAVILY toward declarative
   → Terraform, Kubernetes, Ansible, CloudFormation are all declarative
   → Declarative is safer, more predictable, and easier to manage at scale
```
