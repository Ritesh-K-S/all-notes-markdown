# 12.20 — Crossplane — Kubernetes-Native IaC

> **One Line Answer:** Crossplane lets you **manage cloud infrastructure** (AWS, Azure, GCP, and more) **directly from Kubernetes** — using the same `kubectl` and YAML manifests you already know.

---

## 🤔 What Problem Does Crossplane Solve? Let's Start With a Story

Imagine you're a DevOps engineer on a team that has gone **all-in on Kubernetes**. Your apps run on Kubernetes. Your CI/CD runs on Kubernetes. Your monitoring runs on Kubernetes. Everything is managed with `kubectl` and YAML.

But then... you need to create an **RDS database** on AWS, or an **S3 bucket**, or a **Redis cluster**. Suddenly you have to:

1. Switch to Terraform (learn HCL, manage state files, set up backends...)
2. Or go to the AWS Console (click, click, click... 😩)
3. Or write CloudFormation (YAML headaches again...)

You think:

> "I wish I could just write a YAML manifest like I do for a Pod or Deployment... and `kubectl apply` it... and Kubernetes would create the RDS database for me..."

**That's EXACTLY what Crossplane does!** 🎯

```
Without Crossplane:
┌─────────────┐  kubectl   ┌─────────────┐
│  Developer   │ ────────→ │ Kubernetes   │  (only K8s resources)
└─────────────┘            └─────────────┘
       │
       │  terraform apply   ┌─────────────┐
       └──────────────────→ │ Terraform    │  (cloud resources — separate tool!)
                            └─────────────┘

With Crossplane:
┌─────────────┐  kubectl   ┌──────────────────┐
│  Developer   │ ────────→ │   Kubernetes      │
└─────────────┘            │  + Crossplane     │  (EVERYTHING!)
                           │   ├── Pods        │
                           │   ├── Deployments │
                           │   ├── RDS         │  ← Cloud resources too!
                           │   ├── S3 Bucket   │
                           │   └── VPC         │
                           └──────────────────┘
```

## 📜 What Exactly is Crossplane?

**Crossplane** is an **open-source** Kubernetes add-on that turns your Kubernetes cluster into a **universal control plane** for managing infrastructure across **any cloud provider**.

Think of it this way:
- **Kubernetes** manages your **containers** (Pods, Deployments, Services)
- **Crossplane** extends Kubernetes to also manage your **cloud infrastructure** (databases, buckets, VPCs, DNS, etc.)

### How Does It Work?

Crossplane uses **Kubernetes Custom Resource Definitions (CRDs)** to represent cloud resources. When you `kubectl apply` a Crossplane manifest, Crossplane's controllers talk to the cloud provider API and create the actual resource.

```
┌──────────────────────────────────────────────────────┐
│              Kubernetes Cluster                       │
│                                                       │
│  ┌─────────────┐    ┌────────────────────┐           │
│  │   Your App   │    │  Crossplane        │           │
│  │   (Pods,     │    │  Controllers       │           │
│  │   Deploys)   │    │                    │           │
│  └─────────────┘    └────────┬───────────┘           │
│                               │                       │
│         Kubernetes API        │                       │
│         (kubectl apply)       │                       │
└───────────────────────────────┼───────────────────────┘
                                │
                                │  API Calls
                    ┌───────────┼───────────┐
                    │           │           │
                    ▼           ▼           ▼
              ┌─────────┐ ┌─────────┐ ┌─────────┐
              │   AWS    │ │  Azure  │ │  GCP    │
              │ (RDS, S3)│ │(SQL DB) │ │(GCS)   │
              └─────────┘ └─────────┘ └─────────┘
```

## 🆚 Crossplane vs Terraform vs CDK — When to Use What?

| Feature | Terraform | AWS CDK | Crossplane |
|---------|-----------|---------|------------|
| **Interface** | HCL files + CLI | Programming languages | Kubernetes YAML + kubectl |
| **State Management** | State file (you manage) | CloudFormation (AWS manages) | Kubernetes etcd (K8s manages) |
| **Multi-Cloud** | ✅ Yes | ❌ AWS only | ✅ Yes |
| **Drift Detection** | `terraform plan` | `cdk diff` | **Continuous** (reconciliation loop!) |
| **GitOps Friendly** | Needs wrapper (Atlantis) | Needs CI/CD pipeline | ✅ Native (ArgoCD, Flux) |
| **Prerequisite** | Terraform CLI | Node.js + AWS CLI | Kubernetes cluster |
| **Best For** | General IaC | AWS-heavy teams who code | K8s-native teams, GitOps |
| **Learning Curve** | Learn HCL | Know a language | Know Kubernetes |

### The Killer Feature: Continuous Reconciliation 🔄

This is what makes Crossplane special:

- **Terraform:** You run `terraform apply` once. If someone manually changes the resource in the console, Terraform doesn't know until you run `terraform plan` again.
- **Crossplane:** It runs a **continuous reconciliation loop** (just like Kubernetes does for Pods!). If someone manually deletes your S3 bucket, Crossplane will **recreate it automatically**. It constantly ensures the actual state matches the desired state.

```
Terraform:
  Apply → Done → 😴 Sleeping... (drift can happen!)
  
Crossplane:
  Apply → Reconcile → Reconcile → Reconcile → ♾️ (always watching!)
```

## 🧱 Crossplane Core Concepts

### 1. **Providers** — Cloud Connectors

Providers are the plugins that let Crossplane talk to different clouds:

| Provider | What It Manages |
|----------|----------------|
| `provider-aws` | AWS resources (EC2, RDS, S3, VPC, IAM, etc.) |
| `provider-azure` | Azure resources (VMs, SQL, Blob, VNet, etc.) |
| `provider-gcp` | GCP resources (GCE, GCS, Cloud SQL, VPC, etc.) |
| `provider-kubernetes` | Kubernetes resources in remote clusters |
| `provider-helm` | Helm releases |
| `provider-sql` | SQL databases directly |

Think of providers like Terraform providers — they're how Crossplane knows how to talk to each cloud.

### 2. **Managed Resources (MRs)** — Individual Cloud Resources

These are the CRDs that represent **a single cloud resource**. Like a single S3 bucket or a single RDS instance.

### 3. **Composite Resources (XRs)** — Your Custom Abstractions

These let you **combine multiple resources** into a single custom API. Like a "Database" that creates RDS + Security Group + Subnet Group + Parameter Group all at once.

### 4. **Compositions** — The Recipe

A Composition defines **how** a Composite Resource is built — which Managed Resources to create and how to configure them.

### 5. **Claims (XRCs)** — What Developers Use

Claims are **namespace-scoped** simplified versions of Composite Resources. Developers use Claims to request infrastructure without knowing the details.

```
Here's how it all fits together:

Developer creates a CLAIM:
  "I need a PostgreSQL database, size medium"
        │
        ▼
COMPOSITE RESOURCE (XR) is created:
  The platform team's custom "Database" resource
        │
        ▼
COMPOSITION maps it to MANAGED RESOURCES:
  ├── AWS RDS Instance (db.t3.medium, PostgreSQL 15)
  ├── Security Group (allow port 5432)
  ├── Subnet Group (private subnets)
  └── Parameter Group (tuned settings)
        │
        ▼
PROVIDER creates actual AWS resources:
  Real RDS, Security Groups, etc. on AWS
```

## 🛠️ Installing Crossplane — Step by Step

### Prerequisites
- A running Kubernetes cluster (Minikube, Kind, EKS, AKS, GKE)
- `kubectl` configured
- Helm installed

### Step 1: Install Crossplane Using Helm

```bash
# Add the Crossplane Helm repository
helm repo add crossplane-stable https://charts.crossplane.io/stable
helm repo update

# Create a namespace for Crossplane
kubectl create namespace crossplane-system

# Install Crossplane
helm install crossplane \
  crossplane-stable/crossplane \
  --namespace crossplane-system \
  --wait

# Verify installation
kubectl get pods -n crossplane-system

# Output:
# NAME                                      READY   STATUS    RESTARTS   AGE
# crossplane-xxxxxxxxx-xxxxx                1/1     Running   0          30s
# crossplane-rbac-manager-xxxxxxx-xxxxx     1/1     Running   0          30s
```

### Step 2: Install the Crossplane CLI (Optional but Helpful)

```bash
# Install the Crossplane kubectl plugin
curl -sL "https://raw.githubusercontent.com/crossplane/crossplane/master/install.sh" | sh

# Or with brew
brew install crossplane/tap/crossplane
```

## ☁️ Setting Up an AWS Provider

### Step 1: Install the AWS Provider

```yaml
# aws-provider.yaml
apiVersion: pkg.crossplane.io/v1
kind: Provider
metadata:
  name: provider-aws-s3
spec:
  package: xpkg.upbound.io/upbound/provider-aws-s3:v1.1.0
```

```bash
kubectl apply -f aws-provider.yaml

# Wait for the provider to become healthy
kubectl get providers

# Output:
# NAME              INSTALLED   HEALTHY   PACKAGE                                         AGE
# provider-aws-s3   True        True      xpkg.upbound.io/upbound/provider-aws-s3:v1.1.0  60s
```

### Step 2: Configure AWS Credentials

First, create a Kubernetes Secret with your AWS credentials:

```bash
# Create a credentials file (use environment variables or AWS profile)
# IMPORTANT: In production, use IRSA (IAM Roles for Service Accounts) instead!

cat > aws-credentials.txt << EOF
[default]
aws_access_key_id = YOUR_ACCESS_KEY_ID
aws_secret_access_key = YOUR_SECRET_ACCESS_KEY
EOF

# Create a Kubernetes Secret from the credentials file
kubectl create secret generic aws-creds \
  -n crossplane-system \
  --from-file=creds=./aws-credentials.txt

# Clean up the credentials file
rm aws-credentials.txt
```

### Step 3: Create a ProviderConfig

```yaml
# aws-provider-config.yaml
apiVersion: aws.upbound.io/v1beta1
kind: ProviderConfig
metadata:
  name: default
spec:
  credentials:
    source: Secret
    secretRef:
      namespace: crossplane-system
      name: aws-creds
      key: creds
```

```bash
kubectl apply -f aws-provider-config.yaml
```

## 🚀 Your First Crossplane Resource — Creating an S3 Bucket

Now the fun part! Let's create an S3 bucket using just `kubectl apply`:

```yaml
# s3-bucket.yaml
apiVersion: s3.aws.upbound.io/v1beta1
kind: Bucket
metadata:
  name: my-crossplane-bucket
spec:
  forProvider:
    region: us-east-1
    tags:
      Environment: dev
      ManagedBy: crossplane
  providerConfigRef:
    name: default
```

```bash
# Apply it — just like creating a Pod!
kubectl apply -f s3-bucket.yaml

# Check the status
kubectl get bucket

# Output:
# NAME                    READY   SYNCED   EXTERNAL-NAME           AGE
# my-crossplane-bucket    True    True     my-crossplane-bucket    60s
```

**That's it!** An actual S3 bucket was created on AWS! 🎉

Let's break down the YAML:

| Field | What It Does |
|-------|-------------|
| `apiVersion: s3.aws.upbound.io/v1beta1` | The API group for AWS S3 resources (installed by the provider) |
| `kind: Bucket` | The type of resource — an S3 Bucket |
| `metadata.name` | The name in Kubernetes |
| `spec.forProvider` | Configuration that gets sent **to the cloud provider** (AWS) |
| `spec.forProvider.region` | Which AWS region to create the bucket in |
| `spec.providerConfigRef` | Which credentials to use |

### Check the Bucket Details

```bash
# Describe to see full details
kubectl describe bucket my-crossplane-bucket

# You'll see:
# Status:
#   Conditions:
#     - Type: Ready
#       Status: True         ← Bucket exists and is healthy
#     - Type: Synced
#       Status: True         ← Crossplane is in sync with AWS
#   AtProvider:
#     arn: arn:aws:s3:::my-crossplane-bucket
#     id: my-crossplane-bucket
```

### Delete the Bucket

```bash
# Just like deleting a Pod!
kubectl delete bucket my-crossplane-bucket
# Crossplane will delete the actual S3 bucket on AWS
```

## 🏗️ Real-World Example — RDS Database with Crossplane

Let's create a full RDS PostgreSQL database:

```yaml
# rds-instance.yaml
apiVersion: rds.aws.upbound.io/v1beta1
kind: Instance
metadata:
  name: my-postgres-db
spec:
  forProvider:
    region: us-east-1
    engine: postgres
    engineVersion: "15"
    instanceClass: db.t3.micro
    allocatedStorage: 20
    dbName: myappdb
    username: admin
    autoGeneratePassword: true          # Crossplane generates a secure password!
    passwordSecretRef:
      name: db-password
      namespace: default
      key: password
    skipFinalSnapshot: true
    publiclyAccessible: false
    storageEncrypted: true
    storageType: gp3
    tags:
      Environment: dev
      ManagedBy: crossplane
  providerConfigRef:
    name: default
  writeConnectionSecretToRef:            # Store connection details as a K8s Secret!
    name: db-connection-secret
    namespace: default
```

```bash
kubectl apply -f rds-instance.yaml

# Monitor the creation (RDS takes a few minutes)
kubectl get instance.rds my-postgres-db -w

# Output (watch it go from creating to ready):
# NAME             READY   SYNCED   AGE
# my-postgres-db   False   True     30s
# my-postgres-db   False   True     2m
# my-postgres-db   True    True     5m    ← Database is ready!
```

**The best part:** Crossplane stores the connection details as a Kubernetes Secret!

```bash
# View the connection secret
kubectl get secret db-connection-secret -o yaml

# Your app can now mount this secret as environment variables:
```

```yaml
# Use the database connection in your app Pod
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
    - name: app
      image: my-app:latest
      envFrom:
        - secretRef:
            name: db-connection-secret  # All connection details injected!
```

**This is beautiful!** Developer creates a database with `kubectl apply`, gets connection details as a Secret, uses them in their app — all within Kubernetes. No Terraform state files, no separate tools. 🎯

## 📦 Compositions — Building Custom Abstractions

This is where Crossplane gets **really** powerful. Platform teams can create custom APIs that hide complexity from developers.

### The Goal

Instead of developers writing detailed RDS + SecurityGroup + SubnetGroup YAML, they just say:

```yaml
# What the developer writes (simple!)
apiVersion: database.example.com/v1alpha1
kind: PostgreSQLInstance
metadata:
  name: my-db
spec:
  size: small
  version: "15"
```

And behind the scenes, Crossplane creates **everything** — RDS, Security Group, Subnet Group, Parameter Group, etc.

### Step 1: Define the Composite Resource Definition (XRD)

The XRD defines the **API schema** — what fields developers can set:

```yaml
# xrd-postgres.yaml
apiVersion: apiextensions.crossplane.io/v1
kind: CompositeResourceDefinition
metadata:
  name: xpostgresqlinstances.database.example.com
spec:
  group: database.example.com
  names:
    kind: XPostgreSQLInstance          # Cluster-scoped resource name
    plural: xpostgresqlinstances
  claimNames:
    kind: PostgreSQLInstance           # Namespace-scoped claim name (what devs use)
    plural: postgresqlinstances
  versions:
    - name: v1alpha1
      served: true
      referenceable: true
      schema:
        openAPIV3Schema:
          type: object
          properties:
            spec:
              type: object
              properties:
                size:                  # Developers just pick a size!
                  type: string
                  enum: ["small", "medium", "large"]
                  description: "Database size"
                version:
                  type: string
                  default: "15"
                  description: "PostgreSQL version"
              required:
                - size
```

```bash
kubectl apply -f xrd-postgres.yaml
```

### Step 2: Define the Composition

The Composition defines **what gets created** for each size:

```yaml
# composition-postgres.yaml
apiVersion: apiextensions.crossplane.io/v1
kind: Composition
metadata:
  name: postgres-on-aws
  labels:
    provider: aws
spec:
  compositeTypeRef:
    apiVersion: database.example.com/v1alpha1
    kind: XPostgreSQLInstance
  resources:
    # Resource 1: The RDS Instance
    - name: rds-instance
      base:
        apiVersion: rds.aws.upbound.io/v1beta1
        kind: Instance
        spec:
          forProvider:
            region: us-east-1
            engine: postgres
            publiclyAccessible: false
            storageEncrypted: true
            storageType: gp3
            skipFinalSnapshot: true
          providerConfigRef:
            name: default
      patches:
        # Map "size" to actual instance class
        - type: FromCompositeFieldPath
          fromFieldPath: "spec.size"
          toFieldPath: "spec.forProvider.instanceClass"
          transforms:
            - type: map
              map:
                small: db.t3.micro        # small → db.t3.micro
                medium: db.t3.medium      # medium → db.t3.medium
                large: db.r6g.large       # large → db.r6g.large
        # Map "size" to storage
        - type: FromCompositeFieldPath
          fromFieldPath: "spec.size"
          toFieldPath: "spec.forProvider.allocatedStorage"
          transforms:
            - type: map
              map:
                small: 20
                medium: 50
                large: 100
        # Map PostgreSQL version
        - type: FromCompositeFieldPath
          fromFieldPath: "spec.version"
          toFieldPath: "spec.forProvider.engineVersion"
```

```bash
kubectl apply -f composition-postgres.yaml
```

### Step 3: Developer Creates a Claim

Now developers can request a database with a **super simple** manifest:

```yaml
# my-database-claim.yaml
apiVersion: database.example.com/v1alpha1
kind: PostgreSQLInstance           # The claim (namespace-scoped, developer-friendly)
metadata:
  name: my-app-database
  namespace: team-alpha
spec:
  size: medium                     # That's it! Just pick a size!
  version: "15"
```

```bash
kubectl apply -f my-database-claim.yaml

# Check the claim
kubectl get postgresqlinstance -n team-alpha

# Output:
# NAME              READY   CONNECTION-SECRET   AGE
# my-app-database   True    my-app-database     5m
```

**The platform team** controls how databases are provisioned (instance types, security settings, networking). **Developers** just say "I need a medium database" — no AWS knowledge needed! This is called a **Platform API** or **Internal Developer Platform (IDP)**.

## 🔄 GitOps with Crossplane + ArgoCD

Crossplane is a **perfect match** for GitOps because everything is Kubernetes YAML:

```
┌─────────────────────────────────────────────────────┐
│                    Git Repository                    │
│                                                      │
│  infrastructure/                                     │
│  ├── databases/                                      │
│  │   ├── production-db.yaml    (PostgreSQLInstance)  │
│  │   └── staging-db.yaml       (PostgreSQLInstance)  │
│  ├── storage/                                        │
│  │   ├── data-bucket.yaml      (Bucket)             │
│  │   └── logs-bucket.yaml      (Bucket)             │
│  └── networking/                                     │
│      └── vpc.yaml              (VPC)                 │
└───────────────────────┬─────────────────────────────┘
                        │
                        │  ArgoCD syncs
                        ▼
              ┌─────────────────────┐
              │  Kubernetes Cluster  │
              │  + Crossplane        │
              │                      │
              │  ArgoCD watches Git  │
              │  Crossplane watches  │
              │  Kubernetes watches  │
              └─────────┬───────────┘
                        │
                        │  Crossplane reconciles
                        ▼
              ┌─────────────────────┐
              │   AWS / Azure / GCP  │
              │   (Actual Resources) │
              └─────────────────────┘
```

**Workflow:**
1. Developer opens a **Pull Request** to add/modify infrastructure YAML
2. Team reviews the PR
3. PR gets merged → ArgoCD detects the change
4. ArgoCD syncs the YAML to Kubernetes
5. Crossplane sees the new/updated CRs → creates/updates cloud resources
6. Everything is tracked in Git history 📝

## 🐛 Troubleshooting Crossplane

### Common Commands

```bash
# Check if providers are healthy
kubectl get providers

# Check managed resources
kubectl get managed

# Check all Crossplane resources
kubectl get crossplane

# Describe a resource to see errors
kubectl describe bucket my-bucket

# Check Crossplane controller logs
kubectl logs -n crossplane-system -l app=crossplane

# Check provider controller logs
kubectl logs -n crossplane-system -l pkg.crossplane.io/revision
```

### Common Issues

| Problem | Cause | Solution |
|---------|-------|----------|
| Resource stuck in `SYNCED=False` | Wrong credentials or permissions | Check ProviderConfig and IAM permissions |
| Provider not `HEALTHY` | Package version mismatch or network issue | Check provider logs, try different version |
| Resource stuck in `READY=False` | Cloud-side issue (quota, name conflict) | `kubectl describe` the resource for events |
| Claims not creating XRs | No matching Composition | Check Composition `compositeTypeRef` matches XRD |
| Slow reconciliation | Too many resources, provider rate limits | Tune `pollInterval` in ProviderConfig |

## ✅ When Should You Use Crossplane?

### Use Crossplane When:
- ✅ Your team is already **heavily invested in Kubernetes**
- ✅ You want a **GitOps-native** infrastructure workflow
- ✅ You want **continuous reconciliation** (self-healing infrastructure)
- ✅ You want to build an **Internal Developer Platform** (custom APIs for devs)
- ✅ You need **multi-cloud** support with a unified interface
- ✅ You want infrastructure and application manifests in the **same repo**

### Don't Use Crossplane When:
- ❌ Your team doesn't use Kubernetes
- ❌ You need a quick, simple IaC tool (Terraform is easier to start with)
- ❌ You don't have a team to maintain the Crossplane platform
- ❌ Your cloud resources are very simple (a couple of S3 buckets)

---

## 🧠 Key Takeaways

✅ **Crossplane** extends Kubernetes to manage cloud infrastructure using `kubectl` and YAML

✅ It installs **Providers** (like Terraform providers) to talk to AWS, Azure, GCP, etc.

✅ **Managed Resources** are CRDs that represent individual cloud resources (1:1 mapping)

✅ **Compositions** let platform teams create custom abstractions — developers just request "a database" without knowing cloud details

✅ **Claims** are the developer-friendly, namespace-scoped way to request infrastructure

✅ Crossplane has **continuous reconciliation** — it constantly ensures actual state matches desired state (self-healing!)

✅ It's **perfect for GitOps** — works great with ArgoCD and Flux

✅ State lives in **Kubernetes etcd** — no separate state files to manage

✅ Best suited for teams that are **Kubernetes-native** and want a unified control plane
