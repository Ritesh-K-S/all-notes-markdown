# 12.21 — Bicep — Azure Infrastructure as Code

> **One Line Answer:** Bicep is **Azure's own IaC language** — a cleaner, simpler, more readable alternative to ARM templates for deploying Azure resources. Think of it as "ARM templates, but actually enjoyable to write."

---

## 🤔 Why Does Bicep Exist? A Story First

If you've ever worked with Azure, you've probably met **ARM templates** (Azure Resource Manager templates). They look like this:

```json
{
  "$schema": "https://schema.management.azure.com/schemas/2019-04-01/deploymentTemplate.json#",
  "contentVersion": "1.0.0.0",
  "resources": [
    {
      "type": "Microsoft.Storage/storageAccounts",
      "apiVersion": "2021-02-01",
      "name": "mystorageaccount",
      "location": "[resourceGroup().location]",
      "sku": {
        "name": "Standard_LRS"
      },
      "kind": "StorageV2",
      "properties": {
        "supportsHttpsTrafficOnly": true,
        "minimumTlsVersion": "TLS1_2"
      }
    }
  ]
}
```

That's **a lot of JSON** for just one storage account! 😩 ARM templates are:
- **Verbose** — hundreds of lines for simple stuff
- **Hard to read** — deeply nested JSON
- **Painful to write** — no comments in JSON, bracket matching nightmares
- **No modularity** — reusing code is awkward (linked templates with URLs?)

Microsoft saw developers suffering and said:

> "Let's create a **new language** that compiles down to ARM templates, but is actually pleasant to write."

That language is **Bicep** 💪 (yes, like the muscle — because it's ARM's stronger, better-looking sibling! 😄)

---

## 📜 What Exactly is Bicep?

**Bicep** is a **domain-specific language (DSL)** for deploying Azure resources. You write `.bicep` files, and the Bicep compiler **transpiles** (converts) them into ARM template JSON.

```
┌─────────────────────────┐
│   You Write Bicep       │
│   (main.bicep)          │
│   Clean, readable DSL   │
└────────────┬────────────┘
             │
             │  bicep build
             ▼
┌─────────────────────────┐
│   ARM Template JSON     │
│   (main.json)           │
│   Auto-generated        │
└────────────┬────────────┘
             │
             │  az deployment
             ▼
┌─────────────────────────┐
│   Azure Resource        │
│   Manager (ARM)         │
│   Deploys to Azure      │
└─────────────────────────┘
```

**Key point:** Bicep is **NOT** a separate deployment engine. It's a **better way to write** ARM templates. The actual deployment still happens through ARM — just like CDK generates CloudFormation.

## 🆚 Bicep vs ARM vs Terraform — Quick Comparison

| Feature | ARM Templates | Bicep | Terraform |
|---------|--------------|-------|-----------|
| **Language** | JSON | Bicep DSL | HCL |
| **Cloud Support** | Azure only | Azure only | Multi-cloud |
| **Readability** | 😩 Very verbose | 😊 Clean & concise | 😊 Clean |
| **Type Safety** | None | ✅ Built-in | Limited |
| **IntelliSense** | Limited | ✅ Excellent (VS Code) | Good |
| **State File** | No (Azure tracks state) | No (Azure tracks state) | Yes (you manage!) |
| **Modularity** | Linked templates (URLs) | Modules (file-based) | Modules |
| **Learning Curve** | Steep | Easy! | Medium |
| **Day-0 Support** | Eventually | ✅ Immediate (same API!) | Wait for provider updates |
| **File Size** | ~3x larger | 1x (baseline) | ~1.5x |

### Why Pick Bicep Over Terraform for Azure?

- **Day-0 support:** When Azure releases a new service, Bicep supports it **immediately** (same API). Terraform's Azure provider might take weeks/months to catch up.
- **No state file:** ARM handles state. No S3 backends, no state locking headaches.
- **Azure-native:** Works perfectly with Azure DevOps, Azure CLI, Azure Portal.
- **If you're Azure-only**, Bicep is the simplest choice.

### Why Pick Terraform Over Bicep?

- **Multi-cloud:** If you use AWS + Azure + GCP, Terraform is the unified tool.
- **Larger ecosystem:** More community modules and providers.
- **Cloud-agnostic skills:** Learn once, use everywhere.

## 🚀 Your First Bicep File — Creating a Storage Account

Create a file called `main.bicep`:

```bicep
// main.bicep
// This creates an Azure Storage Account — look how clean this is!

// ========== Parameters ==========
// Parameters are inputs — like function arguments
@description('The Azure region to deploy to')
param location string = resourceGroup().location

@description('The name of the storage account')
@minLength(3)
@maxLength(24)
param storageAccountName string = 'mystorageaccount${uniqueString(resourceGroup().id)}'

@description('The SKU for the storage account')
@allowed([
  'Standard_LRS'
  'Standard_GRS'
  'Standard_ZRS'
  'Premium_LRS'
])
param storageSku string = 'Standard_LRS'

// ========== Resources ==========
// This is the actual resource — just ONE block!
resource storageAccount 'Microsoft.Storage/storageAccounts@2023-01-01' = {
  name: storageAccountName
  location: location
  sku: {
    name: storageSku
  }
  kind: 'StorageV2'
  properties: {
    supportsHttpsTrafficOnly: true
    minimumTlsVersion: 'TLS1_2'
    allowBlobPublicAccess: false
  }
}

// ========== Outputs ==========
// Outputs are return values — like function return statements
output storageId string = storageAccount.id
output storageName string = storageAccount.name
output primaryEndpoint string = storageAccount.properties.primaryEndpoints.blob
```

**Let's break down every part:**

| Part | What It Does |
|------|-------------|
| `param location string` | Declares an input parameter of type `string` |
| `= resourceGroup().location` | Default value — uses the resource group's location |
| `@description(...)` | Adds a description (shows in IntelliSense and docs) |
| `@allowed([...])` | Restricts values to a specific list |
| `@minLength(3)` | Validation — minimum 3 characters |
| `resource storageAccount` | `storageAccount` is the **symbolic name** (used to reference this resource in Bicep) |
| `'Microsoft.Storage/storageAccounts@2023-01-01'` | The **resource type** and **API version** |
| `= { ... }` | The resource body with name, location, sku, properties |
| `output storageId string` | Declares an output that returns the resource ID |
| `storageAccount.id` | References the resource's ID using the symbolic name |

### Compare: ARM Template vs Bicep

```
ARM Template (JSON):    ~35 lines  😩
Bicep:                  ~25 lines  😊 (and much more readable!)
```

## 🔧 Deploying Bicep — Multiple Ways

### Method 1: Azure CLI (Most Common)

```bash
# First, create a resource group (if you don't have one)
az group create --name my-rg --location eastus

# Deploy the Bicep file to the resource group
az deployment group create \
  --resource-group my-rg \
  --template-file main.bicep

# Deploy with parameter overrides
az deployment group create \
  --resource-group my-rg \
  --template-file main.bicep \
  --parameters storageSku='Standard_GRS'

# Preview changes without deploying (like terraform plan / cdk diff)
az deployment group what-if \
  --resource-group my-rg \
  --template-file main.bicep
```

**The `what-if` command is your best friend!** It shows you exactly what will change before you deploy:

```
Resource and property changes are indicated with these symbols:
  + Create
  ~ Modify
  - Delete

The deployment will update the following scope:
Scope: /subscriptions/xxx/resourceGroups/my-rg

  + Microsoft.Storage/storageAccounts/mystorageaccount123 [2023-01-01]
      name:          "mystorageaccount123"
      location:      "eastus"
      sku.name:      "Standard_LRS"
      kind:          "StorageV2"
```

### Method 2: Azure PowerShell

```powershell
# Deploy with PowerShell
New-AzResourceGroupDeployment `
  -ResourceGroupName "my-rg" `
  -TemplateFile "main.bicep"
```

### Method 3: Azure DevOps Pipeline

```yaml

  - main

pool:
  vmImage: 'ubuntu-latest'

  - task: AzureCLI@2
```


## 🧱 Bicep Language Features — Everything You Need

### 1. Variables — Computed Values

```bicep
// Variables are computed at deployment time
var storageName = 'st${uniqueString(resourceGroup().id)}'
var tags = {
  environment: 'production'
  team: 'platform'
  managedBy: 'bicep'
}

resource storageAccount 'Microsoft.Storage/storageAccounts@2023-01-01' = {
  name: storageName
  location: resourceGroup().location
  tags: tags              // Use the variable
  sku: {
    name: 'Standard_LRS'
  }
  kind: 'StorageV2'
  properties: {}
}
```

### 2. Conditions — Deploy Conditionally

```bicep
param deployStorage bool = true
param environment string = 'dev'

// Only create this resource if deployStorage is true
resource storageAccount 'Microsoft.Storage/storageAccounts@2023-01-01' = if (deployStorage) {
  name: 'mystorage${uniqueString(resourceGroup().id)}'
  location: resourceGroup().location
  sku: {
    name: environment == 'production' ? 'Standard_GRS' : 'Standard_LRS'  // Ternary!
  }
  kind: 'StorageV2'
  properties: {}
}
```

### 3. Loops — Create Multiple Resources

```bicep
// Create 3 storage accounts using a loop
param storageNames array = [
  'storagedev'
  'storagestaging'
  'storageprod'
]

resource storageAccounts 'Microsoft.Storage/storageAccounts@2023-01-01' = [for name in storageNames: {
  name: '${name}${uniqueString(resourceGroup().id)}'
  location: resourceGroup().location
  sku: {
    name: 'Standard_LRS'
  }
  kind: 'StorageV2'
  properties: {}
}]

// Access a specific resource from the loop
output firstStorageId string = storageAccounts[0].id
```

### Loop with Index

```bicep
// Loop with index
resource storageAccounts 'Microsoft.Storage/storageAccounts@2023-01-01' = [for (name, index) in storageNames: {
  name: '${name}${uniqueString(resourceGroup().id)}'
  location: resourceGroup().location
  tags: {
    order: string(index)    // 0, 1, 2
  }
  sku: {
    name: 'Standard_LRS'
  }
  kind: 'StorageV2'
  properties: {}
}]
```

### 4. String Interpolation

```bicep
param prefix string = 'myapp'
param env string = 'dev'

// String interpolation — just like JavaScript template literals!
var name = '${prefix}-${env}-storage'
var greeting = 'Deploying to ${resourceGroup().location}'
```

### 5. Existing Resources — Reference Things That Already Exist

```bicep
// Reference an existing resource (don't create it, just look it up)
resource existingVnet 'Microsoft.Network/virtualNetworks@2023-05-01' existing = {
  name: 'my-existing-vnet'
}

// Now use it
output vnetId string = existingVnet.id
output vnetAddressSpace array = existingVnet.properties.addressSpace.addressPrefixes
```

## 📦 Bicep Modules — Reusable Components

Modules are Bicep's answer to code reusability. Split your infrastructure into **separate files** and call them like functions.

### Creating a Module

```bicep
// modules/storage-account.bicep

@description('The name of the storage account')
param name string

@description('The location for the storage account')
param location string = resourceGroup().location

@description('The SKU tier')
param sku string = 'Standard_LRS'

@description('Tags to apply')
param tags object = {}

resource storageAccount 'Microsoft.Storage/storageAccounts@2023-01-01' = {
  name: name
  location: location
  tags: tags
  sku: {
    name: sku
  }
  kind: 'StorageV2'
  properties: {
    supportsHttpsTrafficOnly: true
    minimumTlsVersion: 'TLS1_2'
    allowBlobPublicAccess: false
  }
}

// Expose outputs for the parent to use
output id string = storageAccount.id
output name string = storageAccount.name
output primaryEndpoint string = storageAccount.properties.primaryEndpoints.blob
```

### Using the Module

```bicep
// main.bicep

param environment string = 'dev'
param location string = resourceGroup().location

var tags = {
  environment: environment
  managedBy: 'bicep'
}

// Call the module — like calling a function!
module appStorage 'modules/storage-account.bicep' = {
  name: 'deploy-app-storage'            // Deployment name (for tracking)
  params: {
    name: 'appdata${uniqueString(resourceGroup().id)}'
    location: location
    sku: environment == 'production' ? 'Standard_GRS' : 'Standard_LRS'
    tags: tags
  }
}

module logsStorage 'modules/storage-account.bicep' = {
  name: 'deploy-logs-storage'
  params: {
    name: 'logs${uniqueString(resourceGroup().id)}'
    location: location
    tags: tags
  }
}

// Access module outputs
output appStorageEndpoint string = appStorage.outputs.primaryEndpoint
output logsStorageEndpoint string = logsStorage.outputs.primaryEndpoint
```

```
Project structure:
my-bicep-project/
├── main.bicep                ← Entry point
├── modules/
│   ├── storage-account.bicep ← Reusable storage module
│   ├── vnet.bicep            ← Reusable VNet module
│   └── web-app.bicep         ← Reusable Web App module
└── parameters/
    ├── dev.bicepparam        ← Dev parameters
    └── prod.bicepparam       ← Prod parameters
```

## 🏗️ Real-World Example — Full Web App Infrastructure

Let's build something realistic — a Web App with SQL Database, App Service Plan, and Application Insights:

```bicep
// main.bicep — Complete Web App Infrastructure

// ========== Parameters ==========
@description('The environment name')
@allowed(['dev', 'staging', 'production'])
param environment string

@description('The Azure region')
param location string = resourceGroup().location

@description('The SQL admin username')
param sqlAdminLogin string

@description('The SQL admin password')
@secure()                              // 🔒 @secure() hides this from logs!
param sqlAdminPassword string

// ========== Variables ==========
var prefix = 'myapp-${environment}'
var tags = {
  environment: environment
  project: 'myapp'
  managedBy: 'bicep'
}

// SKU mapping based on environment
var appServiceSkuMap = {
  dev: {
    name: 'F1'              // Free tier for dev
    tier: 'Free'
  }
  staging: {
    name: 'B1'              // Basic for staging
    tier: 'Basic'
  }
  production: {
    name: 'P1v3'            // Premium for production
    tier: 'PremiumV3'
  }
}

// ========== 1. App Service Plan ==========
resource appServicePlan 'Microsoft.Web/serverfarms@2023-01-01' = {
  name: '${prefix}-plan'
  location: location
  tags: tags
  sku: appServiceSkuMap[environment]
  properties: {
    reserved: true           // Required for Linux
  }
}

// ========== 2. Application Insights ==========
resource appInsights 'Microsoft.Insights/components@2020-02-02' = {
  name: '${prefix}-insights'
  location: location
  tags: tags
  kind: 'web'
  properties: {
    Application_Type: 'web'
    RetentionInDays: environment == 'production' ? 90 : 30
  }
}

// ========== 3. SQL Server ==========
resource sqlServer 'Microsoft.Sql/servers@2023-05-01-preview' = {
  name: '${prefix}-sql'
  location: location
  tags: tags
  properties: {
    administratorLogin: sqlAdminLogin
    administratorLoginPassword: sqlAdminPassword
    minimalTlsVersion: '1.2'
  }
}

// ========== 4. SQL Database ==========
resource sqlDatabase 'Microsoft.Sql/servers/databases@2023-05-01-preview' = {
  parent: sqlServer                    // 👈 parent-child relationship!
  name: '${prefix}-db'
  location: location
  tags: tags
  sku: {
    name: environment == 'production' ? 'S1' : 'Basic'
  }
  properties: {
    collation: 'SQL_Latin1_General_CP1_CI_AS'
    maxSizeBytes: environment == 'production' ? 268435456000 : 2147483648
  }
}

// ========== 5. SQL Firewall Rule ==========
// Allow Azure services to access SQL
resource sqlFirewallRule 'Microsoft.Sql/servers/firewallRules@2023-05-01-preview' = {
  parent: sqlServer
  name: 'AllowAzureServices'
  properties: {
    startIpAddress: '0.0.0.0'
    endIpAddress: '0.0.0.0'
  }
}

// ========== 6. Web App ==========
resource webApp 'Microsoft.Web/sites@2023-01-01' = {
  name: '${prefix}-webapp'
  location: location
  tags: tags
  properties: {
    serverFarmId: appServicePlan.id    // 👈 Reference by symbolic name!
    httpsOnly: true
    siteConfig: {
      linuxFxVersion: 'NODE|18-lts'
      alwaysOn: environment == 'production'
      minTlsVersion: '1.2'
      appSettings: [
        {
          name: 'APPINSIGHTS_INSTRUMENTATIONKEY'
          value: appInsights.properties.InstrumentationKey  // 👈 Reference output!
        }
        {
          name: 'DATABASE_URL'
          value: 'Server=tcp:${sqlServer.properties.fullyQualifiedDomainName},1433;Database=${sqlDatabase.name};'
        }
      ]
    }
  }
}

// ========== Outputs ==========
output webAppUrl string = 'https://${webApp.properties.defaultHostName}'
output sqlServerFqdn string = sqlServer.properties.fullyQualifiedDomainName
output appInsightsKey string = appInsights.properties.InstrumentationKey
```

### Deploy It

```bash
# Deploy to dev
az deployment group create \
  --resource-group myapp-dev-rg \
  --template-file main.bicep \
  --parameters environment='dev' \
               sqlAdminLogin='appadmin' \

# Preview production deployment
az deployment group what-if \
  --resource-group myapp-prod-rg \
  --template-file main.bicep \
  --parameters environment='production' \
```

## 📋 Parameter Files — Separate Config from Code

Instead of passing parameters on the command line, use parameter files:

### Bicep Parameter Files (`.bicepparam`) — New Way!

```bicep
// parameters/dev.bicepparam
using '../main.bicep'              // Links to the Bicep file!

param environment = 'dev'
param sqlAdminLogin = 'devadmin'
// Don't put passwords here! Use Key Vault or CI/CD secrets.
```

```bash
# Deploy with parameter file
az deployment group create \
  --resource-group myapp-dev-rg \
  --template-file main.bicep \
  --parameters parameters/dev.bicepparam \
  --parameters sqlAdminPassword='FromCICD!'   # Override sensitive params
```

### JSON Parameter Files — Classic Way

```json
// parameters/dev.parameters.json
{
  "$schema": "https://schema.management.azure.com/schemas/2019-04-01/deploymentParameters.json#",
  "contentVersion": "1.0.0.0",
  "parameters": {
    "environment": {
      "value": "dev"
    },
    "sqlAdminLogin": {
      "value": "devadmin"
    }
  }
}
```

## 🔐 Security Best Practices

### 1. Use `@secure()` for Sensitive Parameters

```bicep
@secure()
param password string         // Won't appear in deployment logs or outputs

@secure()
param connectionString string  // Same for connection strings
```

### 2. Reference Key Vault Secrets

```bicep
// Reference secrets from Azure Key Vault — never hardcode!
resource keyVault 'Microsoft.KeyVault/vaults@2023-02-01' existing = {
  name: 'my-keyvault'
  scope: resourceGroup()
}

module webapp 'modules/webapp.bicep' = {
  name: 'deploy-webapp'
  params: {
    sqlPassword: keyVault.getSecret('sql-admin-password')  // 🔒 From Key Vault!
  }
}
```

### 3. Deployment Scopes

```bicep
// Bicep can deploy at different scopes:

// Resource Group scope (default)
// az deployment group create ...
targetScope = 'resourceGroup'

// Subscription scope (for creating resource groups, policies)
// az deployment sub create ...
targetScope = 'subscription'

// Management Group scope (for organization-wide policies)
// az deployment mg create ...
targetScope = 'managementGroup'

// Tenant scope (for management groups, subscriptions)
// az deployment tenant create ...
targetScope = 'tenant'
```

Example — creating a resource group at subscription scope:

```bicep
// rg.bicep
targetScope = 'subscription'

param rgName string = 'my-app-rg'
param location string = 'eastus'

resource resourceGroup 'Microsoft.Resources/resourceGroups@2023-07-01' = {
  name: rgName
  location: location
  tags: {
    environment: 'dev'
  }
}
```

## 🧪 Linting and Validation

### Bicep Linter

Bicep has a built-in linter that catches issues:

```bash
# Build (compile to ARM) — also runs linter
az bicep build --file main.bicep

# Lint specifically
az bicep lint --file main.bicep
```

Configure linting rules in `bicepconfig.json`:

```json
// bicepconfig.json
{
  "analyzers": {
    "core": {
      "rules": {
        "no-unused-params": {
          "level": "warning"
        },
        "no-unused-vars": {
          "level": "warning"
        },
        "prefer-interpolation": {
          "level": "warning"
        },
        "secure-parameter-default": {
          "level": "error"
        },
        "adminusername-should-not-be-literal": {
          "level": "warning"
        }
      }
    }
  }
}
```

### Validate Before Deploy

```bash
# Validate the template (checks syntax + Azure API validation)
az deployment group validate \
  --resource-group my-rg \
  --template-file main.bicep

# What-if — see exactly what will change
az deployment group what-if \
  --resource-group my-rg \
  --template-file main.bicep
```

## 🔧 Useful Bicep CLI Commands

```bash
# Build — Compile Bicep to ARM JSON
az bicep build --file main.bicep
# Outputs: main.json

# Decompile — Convert existing ARM JSON to Bicep!
az bicep decompile --file template.json
# Outputs: template.bicep (great for migrating!)

# Format — Auto-format your Bicep files
az bicep format --file main.bicep

# Lint — Check for issues
az bicep lint --file main.bicep

# Publish — Publish a module to a registry (Azure Container Registry)
az bicep publish --file modules/storage.bicep \
  --target br:myregistry.azurecr.io/bicep/modules/storage:v1.0

# Use a published module
module storage 'br:myregistry.azurecr.io/bicep/modules/storage:v1.0' = {
  name: 'storage-deployment'
  params: {
    name: 'mystorage'
  }
}
```

### Decompile — Migrate ARM Templates to Bicep!

If you have existing ARM templates, you can **convert them to Bicep**:

```bash
# Convert ARM JSON to Bicep
az bicep decompile --file old-arm-template.json

# This generates old-arm-template.bicep
# Review and clean up — decompilation isn't always perfect
```

This is a **huge migration helper**! You don't have to rewrite everything from scratch.

## 🆚 Bicep Type System & Decorators

### Data Types

```bicep
// Basic types
param myString string = 'hello'
param myInt int = 42
param myBool bool = true
param myArray array = ['a', 'b', 'c']
param myObject object = {
  key1: 'value1'
  key2: 42
}

// User-defined types (Bicep v0.12+)
type Environment = 'dev' | 'staging' | 'production'
param env Environment = 'dev'

type StorageConfig = {
  name: string
  sku: 'Standard_LRS' | 'Standard_GRS'
  enableVersioning: bool
}
param storageConfig StorageConfig
```

### Decorators — Metadata and Validation

```bicep
@description('The name of the resource')     // Description
@minLength(3)                                  // Min length
@maxLength(24)                                 // Max length
param name string

@description('The environment')
@allowed(['dev', 'staging', 'prod'])          // Allowed values
param env string

@description('Number of instances')
@minValue(1)                                   // Min value
@maxValue(10)                                  // Max value
param instanceCount int = 1

@secure()                                      // Hide from logs
param password string

@metadata({
  example: 'myapp-dev-storage'                // Example value in docs
})
param storageName string
```

## 🧠 Key Takeaways

✅ **Bicep** is Azure's **native IaC language** — cleaner and simpler than ARM templates

✅ It **compiles to ARM JSON** — so it has the **same capabilities** but with much better syntax

✅ **No state file** — Azure Resource Manager tracks state for you (unlike Terraform!)

✅ **Day-0 support** — new Azure services work in Bicep immediately

✅ **Modules** let you create reusable components (like Terraform modules)

✅ Use `@secure()` for passwords, `@allowed()` for validation, `@description()` for docs

✅ **`az deployment group what-if`** is your preview command (like `terraform plan`)

✅ You can **decompile** existing ARM templates to Bicep (`az bicep decompile`)

✅ Best for teams that are **Azure-only** — for multi-cloud, consider Terraform

✅ VS Code + Bicep extension gives you **amazing IntelliSense** and error checking

✅ Use **parameter files** (`.bicepparam`) to separate config from code
