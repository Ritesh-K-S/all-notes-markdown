## 🤔 What Is Terraform and Why Does Everyone Use It?

Imagine you need to set up infrastructure in **multiple clouds** — some servers in AWS, a database in Azure, DNS in Cloudflare, and monitoring in Datadog. Without Terraform, you'd need to:

```
Without Terraform:
    → Learn the AWS Console AND the AWS CLI
    → Learn the Azure Portal AND the Azure CLI
    → Learn the Cloudflare Dashboard AND the Cloudflare API
    → Learn the Datadog UI AND the Datadog API
    → Write separate scripts for each platform
    → Each platform has different commands, syntax, and workflows
    → Managing all of this is a NIGHTMARE 😱
```

Now with Terraform:

```
With Terraform:
    → Learn ONE language (HCL)
    → Learn ONE workflow (init → plan → apply)
    → Use ONE tool for ALL platforms
    → Write .tf files for AWS, Azure, Cloudflare, Datadog — all in the same format
    → Terraform talks to each platform's API for you
    → Everything in one place, one language, one workflow ✅
```

### Why Terraform Became the #1 IaC Tool:

| Reason | Details |
|--------|---------|
| **Multi-Cloud** | Works with AWS, Azure, GCP, and 3000+ other services |
| **One Language** | HCL (HashiCorp Configuration Language) — easy to learn |
| **Declarative** | You describe WHAT you want, Terraform figures out HOW |
| **State Management** | Tracks what exists, makes only necessary changes |
| **Plan Before Apply** | `terraform plan` shows you exactly what will change before doing anything |
| **Open Source** | Free to use, massive community |
| **Huge Ecosystem** | 3000+ providers, thousands of modules, rich documentation |
| **Industry Standard** | Used by Netflix, Uber, Slack, Stripe, and almost every tech company |

## 🏗️ Terraform Architecture — How It Works Under the Hood

Let's understand how Terraform actually works. Here's the big picture:

```
┌──────────────────────────────────────────────────────────────────┐
│                    TERRAFORM ARCHITECTURE                         │
│                                                                  │
│  ┌──────────────┐                                                │
│  │  YOU (User)   │                                               │
│  │  Write .tf    │                                               │
│  │  files        │                                               │
│  └──────┬───────┘                                                │
│         ↓                                                        │
│  ┌──────────────┐      ┌──────────────┐      ┌───────────────┐  │
│  │  Terraform   │ ───→ │  Providers   │ ───→ │  Cloud APIs   │  │
│  │  Core        │      │  (Plugins)   │      │  (AWS, Azure, │  │
│  │              │      │              │      │   GCP, etc.)  │  │
│  │  - HCL       │      │  - aws       │      │               │  │
│  │    Parser     │      │  - azurerm   │      │  Actually     │  │
│  │  - State      │      │  - google    │      │  creates the  │  │
│  │    Manager    │      │  - kubernetes│      │  resources    │  │
│  │  - Plan       │      │  - cloudflare│      │               │  │
│  │    Engine     │      │  - datadog   │      │               │  │
│  │  - Graph      │      │  - github    │      │               │  │
│  │    Builder    │      │  - ...3000+  │      │               │  │
│  └──────┬───────┘      └──────────────┘      └───────────────┘  │
│         ↓                                                        │
│  ┌──────────────┐                                                │
│  │  State File  │  ← Keeps track of what Terraform has created   │
│  │  (.tfstate)  │     in the real world                          │
│  └──────────────┘                                                │
└──────────────────────────────────────────────────────────────────┘
```

Let's break down each component:

---

### 1️⃣ Terraform Core

This is the **brain** of Terraform — the main binary/executable that you install on your machine.

**Terraform Core is responsible for:**

| Responsibility | What It Does |
|---------------|-------------|
| **Reading your code** | Parses `.tf` files written in HCL |
| **Building a dependency graph** | Figures out the order to create resources (VPC before subnet, subnet before EC2, etc.) |
| **State management** | Reads and writes the state file to track what exists |
| **Planning** | Compares desired state (code) vs current state (state file) and calculates changes |
| **Applying** | Tells providers to create/update/delete resources |

Think of Terraform Core as the **project manager** — it doesn't actually build anything itself. Instead, it reads the blueprint (your code), makes a plan, and tells the workers (providers) what to do.

---

### 2️⃣ Providers (The Most Important Concept!)

**Providers** are **plugins** that allow Terraform to talk to specific platforms and services.

Think of it this way:
- **Terraform Core** = The universal translator
- **Providers** = Language modules for each cloud/service

```
Terraform Core doesn't know how to talk to AWS directly.
It needs the AWS PROVIDER plugin to translate:

Your code (HCL)           AWS Provider              AWS API
─────────────             ──────────                ────────
resource "aws_instance"  →  Translates to  →  POST /api/ec2/run-instances
resource "aws_s3_bucket" →  Translates to  →  PUT /api/s3/create-bucket
resource "aws_vpc"       →  Translates to  →  POST /api/vpc/create-vpc
```

**Every cloud and service has its own provider:**

| Provider | What It Manages | Provider Name |
|----------|----------------|---------------|
| **AWS** | EC2, S3, RDS, VPC, Lambda, EKS, etc. | `hashicorp/aws` |
| **Azure** | VMs, Storage, AKS, SQL Database, etc. | `hashicorp/azurerm` |
| **Google Cloud** | Compute, GKE, Cloud SQL, BigQuery, etc. | `hashicorp/google` |
| **Kubernetes** | Deployments, Services, ConfigMaps, etc. | `hashicorp/kubernetes` |
| **Cloudflare** | DNS, CDN, Firewall Rules, etc. | `cloudflare/cloudflare` |
| **GitHub** | Repos, Teams, Branch Protections, etc. | `integrations/github` |
| **Datadog** | Monitors, Dashboards, Alerts, etc. | `datadog/datadog` |
| **Docker** | Containers, Images, Networks, etc. | `kreuzwerker/docker` |
| **Helm** | Helm chart releases on Kubernetes | `hashicorp/helm` |
| **Vault** | Secrets, Policies, Auth Methods, etc. | `hashicorp/vault` |
| **Cloudflare** | DNS records, Page Rules, Workers | `cloudflare/cloudflare` |

There are **3000+ providers** available in the [Terraform Registry](https://registry.terraform.io/browse/providers).

#### How to Declare Providers in Your Code:

```hcl
# providers.tf — Tell Terraform which providers you need

# This block tells Terraform which providers to download
terraform {
  required_version = ">= 1.0"        # Minimum Terraform version

  required_providers {
    aws = {
      source  = "hashicorp/aws"       # Where to download the provider from
      version = "~> 5.0"              # Use version 5.x (any minor/patch version)
    }
    azurerm = {
      source  = "hashicorp/azurerm"
      version = "~> 3.0"
    }
  }
}

# Configure the AWS Provider
provider "aws" {
  region = "us-east-1"                # Which AWS region to use
}

# Configure the Azure Provider
provider "azurerm" {
  features {}                         # Required (even if empty) for Azure provider
}
```

**What's happening here (line by line):**

| Line | What It Does |
|------|-------------|
| `terraform { }` | Global Terraform settings block |
| `required_version = ">= 1.0"` | "This code needs Terraform version 1.0 or higher" |
| `required_providers { }` | Lists which provider plugins Terraform needs to download |
| `source = "hashicorp/aws"` | Download the AWS provider from the Terraform Registry (by HashiCorp) |
| `version = "~> 5.0"` | Use any version 5.x (5.0, 5.1, 5.25, etc. but NOT 6.0) — the `~>` means "approximately equal" |
| `provider "aws" { }` | Configuration for the AWS provider — credentials, region, etc. |
| `region = "us-east-1"` | Create resources in the US East (Virginia) AWS region |
| `provider "azurerm" { }` | Configuration for the Azure provider |
| `features {}` | Required block for Azure provider (even if empty) |

### 3️⃣ State File (terraform.tfstate)

The **state file** is how Terraform keeps track of what it has created in the real world.

```
State File = Terraform's Memory

It's a JSON file that maps:
  Your Code (resource names) → Real World (actual resource IDs)
```

**Example state file (simplified):**

```json
{
  "version": 4,
  "resources": [
    {
      "type": "aws_instance",
      "name": "web_server",
      "instances": [
        {
          "attributes": {
            "id": "i-0a1b2c3d4e5f67890",
            "ami": "ami-0c55b159cbfafe1f0",
            "instance_type": "t2.medium",
            "public_ip": "54.123.45.67",
            "private_ip": "10.0.1.50",
            "tags": {
              "Name": "web-server-1"
            }
          }
        }
      ]
    }
  ]
}
```

**What's happening here:**
- Terraform created an EC2 instance from your code
- The real AWS instance ID is `i-0a1b2c3d4e5f67890`
- Its public IP is `54.123.45.67`
- Terraform saved all this in the state file so it can **find** and **manage** this resource later

**Why the state file matters:**

| Without State File | With State File |
|-------------------|-----------------|
| Terraform doesn't know what exists | Terraform knows exactly what exists |
| Running `apply` creates duplicates | Running `apply` only makes necessary changes |
| Can't update or delete resources | Can update, modify, or delete existing resources |
| Can't detect drift | Can compare code vs reality |

> ⚠️ **Important:** The state file can contain **sensitive data** (passwords, keys). NEVER commit it to Git. Store it remotely in S3, Azure Blob, or Terraform Cloud. We'll cover this in detail in section 12.6.

### 4️⃣ Dependency Graph

Terraform builds a **graph** of all your resources and their dependencies. This is how it knows what order to create things.

**Example:** You write this code:

```hcl
# VPC
resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"
}

# Subnet (depends on VPC)
resource "aws_subnet" "web" {
  vpc_id     = aws_vpc.main.id        # ← References the VPC
  cidr_block = "10.0.1.0/24"
}

# Security Group (depends on VPC)
resource "aws_security_group" "web_sg" {
  vpc_id = aws_vpc.main.id            # ← References the VPC

  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

# EC2 Instance (depends on Subnet AND Security Group)
resource "aws_instance" "web" {
  ami                    = "ami-0c55b159cbfafe1f0"
  instance_type          = "t2.medium"
  subnet_id              = aws_subnet.web.id                  # ← References the Subnet
  vpc_security_group_ids = [aws_security_group.web_sg.id]     # ← References the SG
}
```

**Terraform automatically builds this dependency graph:**

```
        aws_vpc.main
            ↓
    ┌───────┴───────────┐
    ↓                   ↓
aws_subnet.web    aws_security_group.web_sg
    ↓                   ↓
    └───────┬───────────┘
            ↓
    aws_instance.web
```

**What this means:**
1. Terraform creates the **VPC first** (no dependencies)
2. Then creates the **Subnet** and **Security Group** **in parallel** (both depend on VPC only)
3. Finally creates the **EC2 instance** (depends on both subnet and security group)

**During destruction, the order is reversed:**
1. Destroy EC2 instance first
2. Then destroy Subnet and Security Group
3. Finally destroy VPC

You NEVER have to think about the order — Terraform figures it out from the references in your code!

## 🔄 The Terraform Workflow — init, plan, apply, destroy

Every Terraform project follows the same workflow. Let's walk through each step:

```
┌─────────┐     ┌─────────┐     ┌─────────┐     ┌──────────┐
│  write   │ ──→ │  init   │ ──→ │  plan   │ ──→ │  apply   │
│  code    │     │         │     │         │     │          │
│ (.tf)    │     │download │     │preview  │     │create/   │
│          │     │providers│     │changes  │     │update    │
└─────────┘     └─────────┘     └─────────┘     └──────────┘
                                                      │
                                                      ↓
                                                ┌──────────┐
                                                │ destroy  │
                                                │(optional)│
                                                │tear down │
                                                └──────────┘
```

### Step 0: Write Your Code

Create your `.tf` files:

```hcl
# main.tf — Your infrastructure definition
resource "aws_instance" "my_server" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"

  tags = {
    Name = "my-first-terraform-server"
  }
}
```

### Step 1: `terraform init` — Initialize the Project

```bash
terraform init
```

**What this does:**
- Downloads the required **provider plugins** (e.g., AWS provider)
- Sets up the **backend** (where state is stored)
- Creates a `.terraform/` directory with downloaded plugins
- Creates a `.terraform.lock.hcl` file (locks provider versions)

**Output:**
```
Initializing the backend...

Initializing provider plugins...
- Finding hashicorp/aws versions matching "~> 5.0"...
- Installing hashicorp/aws v5.31.0...
- Installed hashicorp/aws v5.31.0 (signed by HashiCorp)

Terraform has been successfully initialized!
```

**Analogy:** `terraform init` is like `npm install` for Node.js or `pip install -r requirements.txt` for Python — it downloads all the dependencies.

> 💡 You only need to run `terraform init` once per project (or when you add new providers/modules).

---

### Step 2: `terraform plan` — Preview Changes

```bash
terraform plan
```

**What this does:**
- Reads your `.tf` files (desired state)
- Reads the state file (current state)
- Compares them and shows you what will change
- **Does NOT make any changes** — it's a dry run

**Output:**
```
Terraform will perform the following actions:

  # aws_instance.my_server will be created
  + resource "aws_instance" "my_server" {
      + ami                          = "ami-0c55b159cbfafe1f0"
      + arn                          = (known after apply)
      + id                           = (known after apply)
      + instance_type                = "t2.micro"
      + public_ip                    = (known after apply)
      + tags                         = {
          + "Name" = "my-first-terraform-server"
        }
    }

Plan: 1 to add, 0 to change, 0 to destroy.
```

**Reading the plan output:**

| Symbol | Meaning |
|--------|---------|
| `+` (green) | Resource will be **created** |
| `-` (red) | Resource will be **destroyed** |
| `~` (yellow) | Resource will be **modified** (updated in-place) |
| `-/+` (red/green) | Resource will be **replaced** (destroyed and re-created) |
| `(known after apply)` | This value will only be known after the resource is created (like an IP address) |

**Analogy:** `terraform plan` is like looking at the bill before paying at a restaurant — you see exactly what you're about to get and what it'll cost. You can say "actually, cancel that expensive dish" before committing.

---

### Step 3: `terraform apply` — Make It Happen

```bash
terraform apply
```

**What this does:**
- Shows the plan (same as `terraform plan`)
- Asks for confirmation: `Do you want to perform these actions?`
- If you type `yes`, it:
  - Creates/updates/deletes resources in the real world
  - Updates the state file

**Output:**
```
Terraform will perform the following actions:

  # aws_instance.my_server will be created
  + resource "aws_instance" "my_server" {
      + ami           = "ami-0c55b159cbfafe1f0"
      + instance_type = "t2.micro"
      + tags          = {
          + "Name" = "my-first-terraform-server"
        }
    }

Plan: 1 to add, 0 to change, 0 to destroy.

Do you want to perform these actions?
  Terraform will perform the actions described above.
  Only 'yes' will be accepted to approve.

  Enter a value: yes

aws_instance.my_server: Creating...
aws_instance.my_server: Still creating... [10s elapsed]
aws_instance.my_server: Still creating... [20s elapsed]
aws_instance.my_server: Creation complete after 32s [id=i-0a1b2c3d4e5f67890]

Apply complete! Resources: 1 added, 0 changed, 0 destroyed.
```

Your server is now running in AWS! 🎉

> 💡 **Pro tip:** Use `terraform apply -auto-approve` to skip the confirmation prompt (useful in CI/CD pipelines, **not** in production manual runs).

---

### Step 4: `terraform destroy` — Tear Everything Down

```bash
terraform destroy
```

**What this does:**
- Shows you everything that will be destroyed
- Asks for confirmation
- Deletes ALL resources managed by this Terraform project

**Output:**
```
Terraform will perform the following actions:

  # aws_instance.my_server will be destroyed
  - resource "aws_instance" "my_server" {
      - ami           = "ami-0c55b159cbfafe1f0"
      - instance_type = "t2.micro"
      - id            = "i-0a1b2c3d4e5f67890"
    }

Plan: 0 to add, 0 to change, 1 to destroy.

Do you want to perform these actions?
  Enter a value: yes

aws_instance.my_server: Destroying... [id=i-0a1b2c3d4e5f67890]
aws_instance.my_server: Destruction complete after 45s

Destroy complete! Resources: 1 destroyed.
```

**When to use `destroy`:**
- Tearing down test/dev environments
- Cleaning up after experiments
- Saving money by removing unused infrastructure
- End-of-life for a project

## 📁 Terraform Project Structure

Here's what a typical Terraform project looks like:

```
my-terraform-project/
├── main.tf              # Main infrastructure definitions
├── variables.tf         # Input variable declarations
├── outputs.tf           # Output value declarations
├── providers.tf         # Provider configurations
├── terraform.tfvars     # Variable values (your specific settings)
├── .terraform/          # Downloaded providers (auto-generated, DO NOT commit to Git)
│   └── providers/
│       └── hashicorp/
│           └── aws/
├── .terraform.lock.hcl  # Provider version lock file (COMMIT this to Git)
└── terraform.tfstate    # State file (DO NOT commit to Git — use remote state)
```

**What each file does:**

| File | Purpose | Commit to Git? |
|------|---------|---------------|
| `main.tf` | Your main infrastructure code | ✅ Yes |
| `variables.tf` | Declares input variables (what can be customized) | ✅ Yes |
| `outputs.tf` | Declares output values (what to display after apply) | ✅ Yes |
| `providers.tf` | Provider configurations (which cloud, which region) | ✅ Yes |
| `terraform.tfvars` | Actual values for variables | ⚠️ Maybe (not if it has secrets) |
| `.terraform/` | Downloaded provider plugins | ❌ No (auto-generated) |
| `.terraform.lock.hcl` | Locks provider versions | ✅ Yes |
| `terraform.tfstate` | State file | ❌ NEVER (use remote state) |

**A complete example project:**

```hcl
# ============================================
# providers.tf — Provider Configuration
# ============================================

terraform {
  required_version = ">= 1.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region = var.aws_region    # Uses a variable (defined in variables.tf)
}
```

```hcl
# ============================================
# variables.tf — Input Variables
# ============================================

variable "aws_region" {
  description = "The AWS region to create resources in"
  type        = string
  default     = "us-east-1"
}

variable "instance_type" {
  description = "The EC2 instance type"
  type        = string
  default     = "t2.micro"
}

variable "server_name" {
  description = "Name tag for the EC2 instance"
  type        = string
  default     = "my-web-server"
}
```

```hcl
# ============================================
# main.tf — Main Infrastructure
# ============================================

# Create an EC2 instance
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = var.instance_type    # Uses the variable

  tags = {
    Name        = var.server_name      # Uses the variable
    Environment = "development"
    ManagedBy   = "terraform"
  }
}
```

```hcl
# ============================================
# outputs.tf — Output Values
# ============================================

output "instance_id" {
  description = "The ID of the EC2 instance"
  value       = aws_instance.web.id
}

output "public_ip" {
  description = "The public IP of the EC2 instance"
  value       = aws_instance.web.public_ip
}

output "instance_state" {
  description = "The state of the EC2 instance"
  value       = aws_instance.web.instance_state
}
```

```hcl
# ============================================
# terraform.tfvars — Variable Values
# ============================================

aws_region    = "us-east-1"
instance_type = "t2.medium"
server_name   = "production-web-server"
```

**Running this project:**

```bash
# Step 1: Initialize (download AWS provider)
terraform init

# Step 2: Preview what will be created
terraform plan

# Output:
# Plan: 1 to add, 0 to change, 0 to destroy.

# Step 3: Create the infrastructure
terraform apply

# Output:
# Apply complete! Resources: 1 added, 0 changed, 0 destroyed.
#
# Outputs:
# instance_id    = "i-0a1b2c3d4e5f67890"
# public_ip      = "54.123.45.67"
# instance_state = "running"
```

## 🔌 Deep Dive Into Providers

Providers are the heart of Terraform. Let's understand them better.

### Where Do Providers Come From?

All providers are hosted on the **Terraform Registry**: [registry.terraform.io](https://registry.terraform.io)

There are three types of providers:

```
┌────────────────────────────────────────────────────────┐
│                  Provider Types                         │
├────────────────────┬───────────────────────────────────┤
│ Official           │ Built and maintained by HashiCorp  │
│                    │ e.g., aws, azurerm, google         │
├────────────────────┼───────────────────────────────────┤
│ Partner            │ Built by the technology company    │
│                    │ itself, reviewed by HashiCorp      │
│                    │ e.g., cloudflare, datadog, mongodb │
├────────────────────┼───────────────────────────────────┤
│ Community          │ Built by individual developers     │
│                    │ Not officially supported           │
│                    │ Use with caution                   │
└────────────────────┴───────────────────────────────────┘
```

### Using Multiple Providers

One of Terraform's superpowers is using **multiple providers** in the same project:

```hcl
# providers.tf — Multi-Cloud Setup!

terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
    cloudflare = {
      source  = "cloudflare/cloudflare"
      version = "~> 4.0"
    }
    datadog = {
      source  = "datadog/datadog"
      version = "~> 3.0"
    }
  }
}

# AWS — for servers and infrastructure
provider "aws" {
  region = "us-east-1"
}

# Cloudflare — for DNS and CDN
provider "cloudflare" {
  api_token = var.cloudflare_api_token
}

# Datadog — for monitoring
provider "datadog" {
  api_key = var.datadog_api_key
  app_key = var.datadog_app_key
}
```

Now you can manage resources across ALL three platforms in one Terraform project:

```hcl
# main.tf — Resources across multiple providers

# AWS: Create a web server
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.medium"

  tags = {
    Name = "web-server"
  }
}

# Cloudflare: Create a DNS record pointing to the server
resource "cloudflare_record" "web" {
  zone_id = var.cloudflare_zone_id
  name    = "www"
  content = aws_instance.web.public_ip    # ← Uses the IP from the AWS resource!
  type    = "A"
  ttl     = 300
}

# Datadog: Create a monitor to alert if the server goes down
resource "datadog_monitor" "web_uptime" {
  name    = "Web Server Uptime"
  type    = "service check"
  message = "Web server is DOWN! @slack-devops-alerts"

  query = "\"http.can_connect\".over(\"instance:web-server\").by(\"host\",\"instance\").last(2).count_by_status()"

  tags = ["env:production", "service:web"]
}
```

**What's happening here:**
1. **AWS** creates the server → gets a public IP
2. **Cloudflare** creates a DNS record → `www.example.com` points to the server's IP
3. **Datadog** creates a monitor → alerts the team if the server goes down

All in ONE Terraform project. ONE `terraform apply` command. **That's powerful.**

### Multiple Provider Configurations (Aliases)

Sometimes you need to use the same provider with different configurations. For example, creating resources in **two different AWS regions**:

```hcl
# Default AWS provider — us-east-1
provider "aws" {
  region = "us-east-1"
}

# Second AWS provider — eu-west-1 (with an alias)
provider "aws" {
  alias  = "europe"           # ← Give it a name
  region = "eu-west-1"
}

# Server in US (uses default provider)
resource "aws_instance" "us_server" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.medium"

  tags = {
    Name = "us-web-server"
  }
}

# Server in Europe (uses the aliased provider)
resource "aws_instance" "eu_server" {
  provider      = aws.europe         # ← Use the Europe provider
  ami           = "ami-0abc12345def"  # Different AMI for EU region
  instance_type = "t2.medium"

  tags = {
    Name = "eu-web-server"
  }
}
```

**What's happening here:**
- Two AWS providers: one for US, one for Europe
- The Europe provider has an `alias = "europe"` so Terraform can tell them apart
- The EU server uses `provider = aws.europe` to specify which provider to use
- The US server uses the default (no alias needed)

## 🧪 Your First Terraform Project — Step by Step

Let's build something real! We'll create an AWS S3 bucket (cloud storage) using Terraform.

### Prerequisites:
- Terraform installed (`terraform --version`)
- AWS CLI configured (`aws configure`) with your credentials

### Step 1: Create a Project Directory

```bash
mkdir my-first-terraform
cd my-first-terraform
```

### Step 2: Create the Terraform Files

```hcl
# main.tf

terraform {
  required_version = ">= 1.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region = "us-east-1"
}

# Create an S3 bucket
resource "aws_s3_bucket" "my_bucket" {
  bucket = "my-first-terraform-bucket-12345"   # Must be globally unique!

  tags = {
    Name        = "My First Terraform Bucket"
    Environment = "learning"
    ManagedBy   = "terraform"
  }
}

# Enable versioning on the bucket
resource "aws_s3_bucket_versioning" "my_bucket_versioning" {
  bucket = aws_s3_bucket.my_bucket.id         # References the bucket above

  versioning_configuration {
    status = "Enabled"
  }
}

# Output the bucket name and ARN
output "bucket_name" {
  value = aws_s3_bucket.my_bucket.bucket
}

output "bucket_arn" {
  value = aws_s3_bucket.my_bucket.arn
}
```

**What each resource does:**

| Resource | Purpose |
|----------|---------|
| `aws_s3_bucket.my_bucket` | Creates an S3 bucket (like a cloud folder for storing files) |
| `aws_s3_bucket_versioning` | Enables versioning — S3 keeps old versions of files when you update them |
| `output "bucket_name"` | After `apply`, displays the bucket name |
| `output "bucket_arn"` | After `apply`, displays the bucket's ARN (Amazon Resource Name — unique identifier) |

## 🌐 How Authentication Works With Providers

Every provider needs **credentials** to authenticate with the cloud/service. Here's how it works for the major providers:

### AWS Authentication

Terraform uses the same credentials as the AWS CLI. There are multiple ways to authenticate:

```hcl
# Method 1: Environment Variables (RECOMMENDED for CI/CD)
# Set these before running Terraform:
# export AWS_ACCESS_KEY_ID="your-access-key"
# export AWS_SECRET_ACCESS_KEY="your-secret-key"
# export AWS_REGION="us-east-1"

provider "aws" {
  region = "us-east-1"
  # Terraform automatically picks up the environment variables
}

# Method 2: Shared Credentials File (RECOMMENDED for local development)
# Uses ~/.aws/credentials (set up by `aws configure`)

provider "aws" {
  region  = "us-east-1"
  profile = "default"        # Or any named profile from ~/.aws/credentials
}

# Method 3: IAM Role (RECOMMENDED for EC2 instances / EKS)
# No credentials needed — the EC2 instance's IAM role is used automatically

provider "aws" {
  region = "us-east-1"
  # No explicit credentials — uses the instance role
}
```

> ⚠️ **NEVER hardcode credentials in your `.tf` files!** This is a serious security risk. Always use environment variables, credential files, or IAM roles.

```hcl
# ❌ NEVER DO THIS:
provider "aws" {
  region     = "us-east-1"
  access_key = "AKIAIOSFODNN7EXAMPLE"        # NEVER hardcode!
  secret_key = "wJalrXUtnFEMI/K7MDENG/bPx..."  # NEVER hardcode!
}
```

### Azure Authentication

```hcl
# Method 1: Azure CLI (for local development)
# First run: az login
provider "azurerm" {
  features {}
  # Uses Azure CLI credentials automatically
}

# Method 2: Service Principal (for CI/CD)
# export ARM_CLIENT_ID="your-client-id"
# export ARM_CLIENT_SECRET="your-client-secret"
# export ARM_SUBSCRIPTION_ID="your-subscription-id"
# export ARM_TENANT_ID="your-tenant-id"

provider "azurerm" {
  features {}
}
```

### GCP Authentication

```hcl
# Method 1: gcloud CLI (for local development)
# First run: gcloud auth application-default login
provider "google" {
  project = "my-project-id"
  region  = "us-central1"
}

# Method 2: Service Account Key (for CI/CD)
# export GOOGLE_CREDENTIALS="/path/to/service-account-key.json"
provider "google" {
  project = "my-project-id"
  region  = "us-central1"
}
```

## 📊 Terraform Version Constraints — Understanding the `~>` Operator

You'll see version constraints everywhere in Terraform. Here's what they mean:

| Constraint | Meaning | Allows | Doesn't Allow |
|-----------|---------|--------|---------------|
| `= 5.0.0` | Exactly this version | 5.0.0 only | Everything else |
| `>= 5.0` | This version or higher | 5.0, 5.1, 6.0, 7.0... | 4.x |
| `<= 5.0` | This version or lower | 5.0, 4.x, 3.x... | 6.0+ |
| `~> 5.0` | Any 5.x version | 5.0, 5.1, 5.99 | 6.0+ |
| `~> 5.1` | Any 5.1.x version | 5.1.0, 5.1.1, 5.1.99 | 5.2+, 6.0+ |
| `>= 5.0, < 6.0` | Between 5.0 and 5.x | 5.0 to 5.99 | 6.0+ |

The most common one is **`~>`** (called the "pessimistic constraint operator"):
- `~> 5.0` means "give me any version 5.x" (allows minor updates, blocks major version changes)
- This is the **recommended** approach — you get bug fixes and features without breaking changes

## 🗺️ How Terraform Resources Are Named

Every Terraform resource follows this pattern:

```
resource "<PROVIDER>_<RESOURCE_TYPE>" "<LOCAL_NAME>" {
  # Configuration
}
```

**Breaking it down:**

```hcl
resource "aws_instance" "web_server" {
#         ↑              ↑
#    Provider + Type   Your local name
#    (what to create)  (how YOU refer to it)
}
```

| Part | Example | Meaning |
|------|---------|---------|
| Provider prefix | `aws_` | This resource belongs to the AWS provider |
| Resource type | `instance` | We're creating an EC2 instance |
| Local name | `web_server` | This is YOUR name for this resource in your code |
| Full address | `aws_instance.web_server` | How you reference this resource elsewhere |

**More examples:**

```hcl
resource "aws_s3_bucket" "photos"       { }  # AWS S3 bucket, locally named "photos"
resource "aws_vpc" "main"               { }  # AWS VPC, locally named "main"
resource "azurerm_resource_group" "rg"  { }  # Azure Resource Group, locally named "rg"
resource "google_compute_instance" "vm" { }  # GCP Compute Instance, locally named "vm"
resource "cloudflare_record" "dns"      { }  # Cloudflare DNS Record, locally named "dns"
```

To reference one resource from another, use the full address:

```hcl
resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"
}

resource "aws_subnet" "web" {
  vpc_id = aws_vpc.main.id         # ← References the VPC's ID
  #        ↑          ↑
  #   Full address   Attribute
}
```

## 📝 Key Takeaways

```
✅ Terraform = Open-source IaC tool by HashiCorp
   → Declarative, multi-cloud, state-managed
   → Industry standard for infrastructure provisioning

✅ Architecture has 4 main components:
   1. Terraform Core — The brain (parses code, plans, manages state)
   2. Providers — Plugins that talk to cloud APIs (AWS, Azure, GCP, etc.)
   3. State File — Tracks what Terraform has created (terraform.tfstate)
   4. Dependency Graph — Figures out the correct order to create resources

✅ The Terraform Workflow:
   → terraform init    — Download providers (run once)
   → terraform plan    — Preview changes (dry run)
   → terraform apply   — Create/update infrastructure
   → terraform destroy — Tear down everything

✅ Providers are plugins:
   → 3000+ available in the Terraform Registry
   → Official (HashiCorp), Partner, and Community providers
   → Configured in the `terraform {}` and `provider {}` blocks
   → Can use multiple providers in one project
   → Can use the same provider with different configs (aliases)

✅ Project structure:
   → main.tf, variables.tf, outputs.tf, providers.tf
   → terraform.tfvars for variable values
   → .terraform/ (auto-generated, don't commit)
   → terraform.tfstate (NEVER commit, use remote state)

✅ Authentication:
   → NEVER hardcode credentials
   → Use environment variables, CLI profiles, or IAM roles
   → Different methods for different providers

✅ Version constraints:
   → ~> 5.0 means "any 5.x version" (most common)
   → Always pin your provider versions
```
