## 🤔 Before We Start — What Are We Actually Learning?

Think of Terraform like ordering food at a restaurant:

```
📝 HCL         = The LANGUAGE you use to write your order
🍕 Resources   = Things you want the kitchen to MAKE for you (new dishes)
📖 Data Sources = Things already on the menu that you just want to LOOK AT or REFERENCE
```

So:
- **HCL** is HOW you write Terraform code
- **Resources** are WHAT you want Terraform to create
- **Data Sources** are things that already exist and you want to use their info

Let's dive into each one!

## 📘 Part 1: HCL (HashiCorp Configuration Language)

### What is HCL?

HCL stands for **HashiCorp Configuration Language**. It's the language Terraform uses.

Think of it as a **simple, clean way to describe infrastructure** — like writing a recipe card instead of a 50-page manual.

### Why Not Just Use JSON or YAML?

Great question! HashiCorp COULD have used JSON or YAML, but:

| Feature | JSON | YAML | HCL |
|---------|------|------|-----|
| Human Readable | ❌ Lots of braces & quotes | ✅ Clean | ✅ Very Clean |
| Comments | ❌ Not supported | ✅ Supported | ✅ Supported |
| Logic (loops, conditions) | ❌ No | ❌ No | ✅ Yes |
| Functions | ❌ No | ❌ No | ✅ Yes |
| Purpose-built for IaC | ❌ No | ❌ No | ✅ Yes |

HCL gives you the **readability of YAML** with the **power of a programming language**.

> 💡 Fun Fact: Terraform CAN actually accept JSON format (`.tf.json` files), but nobody uses it because HCL is so much nicer.

---

### HCL Basic Syntax — The Building Blocks

Every HCL file is made up of **blocks**, **arguments**, and **expressions**. Let's learn each one.

---

#### 1️⃣ Blocks

A **block** is a container. It groups related settings together.

```hcl
# Basic block structure:
block_type "label_1" "label_2" {
  # arguments go inside here
}
```

**Real Example:**

```hcl
resource "aws_instance" "my_server" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"
}
```

Let's break this down:

```
resource              → Block TYPE (what kind of block is this?)
"aws_instance"        → FIRST label (the resource type — an AWS EC2 instance)
"my_server"           → SECOND label (YOUR name for this resource — you pick it!)
{                     → Block BODY starts
  ami = "..."         → An ARGUMENT inside the block
  instance_type = "..." → Another ARGUMENT
}                     → Block BODY ends
```

#### Types of Blocks You'll See:

| Block Type | Purpose | Example |
|-----------|---------|---------|
| `terraform` | Configure Terraform itself | Required version, backend |
| `provider` | Configure a cloud provider | AWS, Azure, GCP |
| `resource` | Create something NEW | EC2 instance, S3 bucket |
| `data` | Read something that ALREADY EXISTS | Get AMI ID, get VPC info |
| `variable` | Define input variables | Region, instance size |
| `output` | Define outputs | Show IP address after creation |
| `locals` | Define local variables | Computed/reusable values |
| `module` | Use a reusable module | Call a pre-built set of resources |

---

#### 2️⃣ Arguments

An **argument** assigns a value to a name. It's like a setting.

```hcl
# Syntax:
name = value
```

**Examples:**

```hcl
instance_type = "t2.micro"           # String
count         = 3                     # Number
enable_dns    = true                  # Boolean
tags          = {                     # Map (key-value pairs)
  Name = "MyServer"
  Env  = "Production"
}
availability_zones = ["us-east-1a", "us-east-1b"]  # List
```

---

#### 3️⃣ Expressions

**Expressions** are values or computations. They can be simple or complex.

**Simple expressions (just values):**

```hcl
"hello"         # String
42              # Number
true            # Boolean
["a", "b"]      # List
{ key = "val" } # Map
```

**Complex expressions (computed values):**

```hcl
# String interpolation — put variables inside strings
name = "server-${var.environment}"

# Conditional — like an if/else in one line
instance_type = var.environment == "prod" ? "t2.large" : "t2.micro"

# Function call
upper_name = upper("hello")   # Result: "HELLO"
```

---

### HCL Comments

You can add comments in three ways:

```hcl
# This is a single-line comment (most common)

// This is also a single-line comment (less common)

/*
  This is a
  multi-line comment
  for longer explanations
*/
```

### HCL Data Types

Here are ALL the data types in HCL:

#### Primitive Types:

```hcl
# String — text wrapped in quotes
name = "web-server"

# Number — integers or decimals
count = 5
price = 19.99

# Boolean — true or false
enabled = true
```

#### Complex Types:

```hcl
# List (ordered collection) — use square brackets
availability_zones = ["us-east-1a", "us-east-1b", "us-east-1c"]

# Map (key-value pairs) — use curly braces
tags = {
  Name        = "WebServer"
  Environment = "production"
  Team        = "devops"
}

# Set (like a list but no duplicates, no order)
# (You'll see this less often — mostly in variable type definitions)

# Object — a map with a defined structure
# (Used in variable type constraints)

# Tuple — a list with specific types per position
# (Rarely used directly, but good to know)
```

#### The `null` Value:

```hcl
# null means "no value" — Terraform treats it as "use the default" or "skip this"
special_setting = null
```

---

### HCL Operators

Just like any language, HCL has operators:

```hcl
# Arithmetic
result = 5 + 3      # 8
result = 10 - 4     # 6
result = 3 * 2      # 6
result = 10 / 3     # 3.333...
result = 10 % 3     # 1 (remainder)

# Comparison
equal     = (5 == 5)    # true
not_equal = (5 != 3)    # true
greater   = (5 > 3)     # true
less      = (3 < 5)     # true
gte       = (5 >= 5)    # true
lte       = (3 <= 5)    # true

# Logical
and_result = (true && false)   # false
or_result  = (true || false)   # true
not_result = !true             # false
```

---

### String Interpolation & Templates

This is one of HCL's superpowers — embedding variables and expressions inside strings:

```hcl
# Basic interpolation — use ${...} inside double quotes
name = "server-${var.environment}-${var.region}"
# If environment="prod" and region="us-east-1"
# Result: "server-prod-us-east-1"

# Expression inside interpolation
greeting = "We have ${var.count + 1} servers running"

# Multi-line strings — use <<EOF ... EOF (heredoc)
user_data = <<EOF
#!/bin/bash
echo "Hello from ${var.server_name}"
apt-get update
apt-get install -y nginx
EOF

# Indented heredoc — use <<- (strips leading whitespace)
user_data = <<-EOF
  #!/bin/bash
  echo "Hello from ${var.server_name}"
  apt-get update
  apt-get install -y nginx
EOF
```

### Putting It All Together — A Complete HCL Example

Here's a complete, real-world Terraform file that uses everything we've learned:

```hcl
# ----- terraform block: configure Terraform itself -----
terraform {
  required_version = ">= 1.5.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

# ----- provider block: configure AWS -----
provider "aws" {
  region = "us-east-1"
}

# ----- resource block: create an EC2 instance -----
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"

  tags = {
    Name        = "web-server-${format("%02d", 1)}"  # "web-server-01"
    Environment = "production"
    ManagedBy   = "Terraform"
  }
}
```

## 📘 Part 2: Resources — The Heart of Terraform

### What is a Resource?

A **Resource** is the most important thing in Terraform. It tells Terraform:

> **"Hey Terraform, please CREATE and MANAGE this thing for me."**

Every resource represents a **real piece of infrastructure** — a server, a database, a DNS record, a firewall rule, a storage bucket, etc.

---

### Resource Block Syntax

```hcl
resource "<PROVIDER>_<TYPE>" "<NAME>" {
  # configuration arguments
}
```

Let's decode each part:

```
resource          → Keyword that says "I want to create something"
"<PROVIDER>_<TYPE>" → What kind of thing to create
                     └── "aws_instance" = an EC2 instance in AWS
                     └── "aws_s3_bucket" = an S3 bucket in AWS
                     └── "azurerm_virtual_machine" = a VM in Azure
                     └── "google_compute_instance" = a VM in GCP
"<NAME>"          → YOUR friendly name for this resource (used to refer to it in your code)
{ ... }           → Configuration arguments (settings)
```

---

### Your First Resource — Creating an AWS EC2 Instance

```hcl
# Tell Terraform we're using AWS
provider "aws" {
  region = "us-east-1"
}

# Create an EC2 instance (a virtual server in AWS)
resource "aws_instance" "my_first_server" {
  ami           = "ami-0c55b159cbfafe1f0"    # Which operating system image to use
  instance_type = "t2.micro"                  # How powerful (1 CPU, 1 GB RAM — free tier!)

  tags = {
    Name = "MyFirstServer"
  }
}
```

**What happens when you run `terraform apply`?**

```
Terraform will:
  1. Contact AWS using your credentials
  2. Create a new EC2 instance with:
     - Ubuntu OS (that's what the AMI is)
     - t2.micro size (small, free-tier eligible)
     - Named "MyFirstServer"
  3. Save the details in the state file
  4. Show you the result
```

### More Resource Examples

#### 🪣 Create an S3 Bucket:

```hcl
resource "aws_s3_bucket" "my_bucket" {
  bucket = "my-unique-bucket-name-12345"

  tags = {
    Name        = "MyStorage"
    Environment = "dev"
  }
}

# Configure versioning separately (Terraform v4+ pattern)
resource "aws_s3_bucket_versioning" "my_bucket_versioning" {
  bucket = aws_s3_bucket.my_bucket.id    # Reference the bucket we created above!

  versioning_configuration {
    status = "Enabled"
  }
}
```

Notice how we **reference** the bucket in the versioning resource:
```
aws_s3_bucket.my_bucket.id
     ↑              ↑        ↑
  resource type   our name   attribute (auto-generated by AWS)
```

This is called a **resource reference** — it creates a **dependency**. Terraform knows to create the bucket FIRST, then the versioning.

---

#### 🔒 Create a Security Group:

```hcl
resource "aws_security_group" "web_sg" {
  name        = "web-server-sg"
  description = "Allow HTTP and SSH traffic"
  vpc_id      = "vpc-12345678"

  # Allow HTTP from anywhere
  ingress {
    description = "HTTP"
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  # Allow SSH from my IP only
  ingress {
    description = "SSH"
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["203.0.113.50/32"]    # Replace with your IP
  }

  # Allow all outbound traffic
  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }

  tags = {
    Name = "web-server-sg"
  }
}
```

#### 🌐 Create a VPC:

```hcl
resource "aws_vpc" "main" {
  cidr_block           = "10.0.0.0/16"
  enable_dns_support   = true
  enable_dns_hostnames = true

  tags = {
    Name = "main-vpc"
  }
}

# Create a subnet inside the VPC
resource "aws_subnet" "public" {
  vpc_id                  = aws_vpc.main.id       # Reference the VPC above
  cidr_block              = "10.0.1.0/24"
  availability_zone       = "us-east-1a"
  map_public_ip_on_launch = true

  tags = {
    Name = "public-subnet-1"
  }
}
```

### Resource Attributes — Inputs vs Outputs

Every resource has two kinds of attributes:

#### Input Arguments (you provide these):

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"    # YOU specify this
  instance_type = "t2.micro"                  # YOU specify this
  tags          = { Name = "web" }            # YOU specify this
}
```

#### Output Attributes (Terraform fills these AFTER creation):

```hcl
# After the instance is created, Terraform knows things like:
aws_instance.web.id                # "i-0abc123def456789"
aws_instance.web.public_ip         # "54.23.104.67"
aws_instance.web.private_ip        # "10.0.1.15"
aws_instance.web.arn               # "arn:aws:ec2:..."
aws_instance.web.availability_zone # "us-east-1a"
```

You can **use** these output attributes in OTHER resources:

```hcl
# Create an Elastic IP
resource "aws_eip" "web_ip" {
  instance = aws_instance.web.id    # Use the instance ID from above!
}
```

### Resource Dependencies

Terraform is smart. It figures out the **order** to create things by looking at references.

```hcl
# Step 1: VPC is created FIRST (no dependencies)
resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"
}

# Step 2: Subnet is created SECOND (depends on VPC)
resource "aws_subnet" "public" {
  vpc_id     = aws_vpc.main.id      # ← This reference tells Terraform:
  cidr_block = "10.0.1.0/24"        #    "Create the VPC first!"
}

# Step 3: Instance is created THIRD (depends on Subnet)
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"
  subnet_id     = aws_subnet.public.id   # ← "Create the Subnet first!"
}
```

Terraform builds a **dependency graph** and figures out the order automatically:

```
aws_vpc.main
      ↓
aws_subnet.public
      ↓
aws_instance.web
```

#### Explicit Dependencies with `depends_on`:

Sometimes Terraform can't figure out the dependency automatically. You can force it:

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"

  depends_on = [
    aws_iam_role_policy.web_policy    # Wait for this to be created first!
  ]
}
```

> ⚠️ **Best Practice:** Only use `depends_on` when you absolutely have to. Let Terraform figure out dependencies from references whenever possible.

### Resource Meta-Arguments

Meta-arguments are **special arguments** that work with ANY resource (not specific to any provider). These are super powerful:

#### `count` — Create Multiple Copies:

```hcl
resource "aws_instance" "web" {
  count         = 3                           # Create 3 instances!
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"

  tags = {
    Name = "web-server-${count.index}"       # web-server-0, web-server-1, web-server-2
  }
}

# Reference a specific one:
# aws_instance.web[0].public_ip    ← First instance
# aws_instance.web[1].public_ip    ← Second instance
# aws_instance.web[*].public_ip    ← ALL instances (splat expression)
```

#### `for_each` — Create One for Each Item in a Collection:

```hcl
resource "aws_s3_bucket" "buckets" {
  for_each = toset(["logs", "uploads", "backups"])

  bucket = "myapp-${each.value}-bucket"

  tags = {
    Name    = each.value
    Purpose = each.value
  }
}

# With a map:
resource "aws_instance" "servers" {
  for_each = {
    web = "t2.micro"
    app = "t2.small"
    db  = "t2.medium"
  }

  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = each.value       # The value from the map

  tags = {
    Name = "${each.key}-server"    # The key from the map
  }
}

# Reference:
# aws_instance.servers["web"].public_ip
# aws_instance.servers["db"].public_ip
```

> 💡 **`count` vs `for_each`:** Use `for_each` when each item is unique (different names, settings). Use `count` when they're all identical. `for_each` is generally preferred because adding/removing items doesn't shift indices.

#### `lifecycle` — Control How Resources are Updated/Destroyed:

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"

  lifecycle {
    # Don't destroy this before creating the replacement
    create_before_destroy = true

    # Never destroy this resource (even if you run terraform destroy)
    prevent_destroy = true

    # Ignore changes to these attributes (useful when something changes outside Terraform)
    ignore_changes = [
      tags,
      ami,
    ]

    # Custom conditions that must be true
    precondition {
      condition     = var.instance_type != "t2.nano"
      error_message = "t2.nano is too small for this workload!"
    }

    postcondition {
      condition     = self.public_ip != ""
      error_message = "The instance must have a public IP!"
    }
  }
}
```

#### `provider` — Use a Specific Provider Configuration:

```hcl
# If you have multiple AWS regions:
provider "aws" {
  region = "us-east-1"
}

provider "aws" {
  alias  = "west"
  region = "us-west-2"
}

# Use the default provider (us-east-1)
resource "aws_instance" "east_server" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"
}

# Use the "west" provider (us-west-2)
resource "aws_instance" "west_server" {
  provider      = aws.west                    # ← Use the aliased provider
  ami           = "ami-0892d3c7ee96c0bf7"
  instance_type = "t2.micro"
}
```

### Resource Lifecycle — What Happens Behind the Scenes

When you run Terraform commands, here's what happens to resources:

```
terraform plan    → "Here's what I WOULD do" (dry run)
terraform apply   → "I'm doing it NOW" (create/update/destroy)
terraform destroy → "I'm deleting EVERYTHING" (clean up)
```

The lifecycle of a resource:

```
1. CREATE  → Resource doesn't exist yet → Terraform creates it
2. READ    → Resource exists → Terraform checks its current state
3. UPDATE  → Something changed in your code → Terraform modifies it
4. DELETE  → You removed it from code (or ran destroy) → Terraform deletes it
```

This is called **CRUD** — and Terraform does it all for you!

## 📘 Part 3: Data Sources — Reading Existing Infrastructure

### What is a Data Source?

A **Data Source** lets you **READ information** from infrastructure that **already exists** — things you DIDN'T create with Terraform, or things created by a DIFFERENT Terraform configuration.

Think of it like a **lookup** or **search query**:

```
Resource    = "Build me a new house"
Data Source = "Tell me the address of the house that's already on Elm Street"
```

---

### Data Source Syntax

```hcl
data "<PROVIDER>_<TYPE>" "<NAME>" {
  # query/filter arguments
}
```

Notice it starts with `data` instead of `resource`!

---

### Data Source Examples

#### 🔍 Find the Latest Amazon Linux AMI:

This is one of the MOST common uses of data sources:

```hcl
data "aws_ami" "amazon_linux" {
  most_recent = true
  owners      = ["amazon"]

  filter {
    name   = "name"
    values = ["amzn2-ami-hvm-*-x86_64-gp2"]
  }

  filter {
    name   = "state"
    values = ["available"]
  }
}

# Now USE it in a resource:
resource "aws_instance" "web" {
  ami           = data.aws_ami.amazon_linux.id    # ← The AMI ID found by the data source!
  instance_type = "t2.micro"

  tags = {
    Name = "web-server"
  }
}
```

**What's happening here?**

```
1. The data source SEARCHES AWS for AMIs matching our criteria:
   - Owned by Amazon
   - Name matches "amzn2-ami-hvm-*-x86_64-gp2"
   - Status is "available"
   - Get the most recent one

2. Terraform stores the result, including the AMI ID

3. We reference it as: data.aws_ami.amazon_linux.id
   Notice the "data." prefix — that's how Terraform knows it's a data source
```

> 💡 **Why use this instead of hardcoding the AMI ID?** Because AMI IDs change by region and are updated frequently. This data source always gets the LATEST one automatically!

---

#### 🌐 Get Information About an Existing VPC:

```hcl
# Look up a VPC by its tag name
data "aws_vpc" "existing" {
  filter {
    name   = "tag:Name"
    values = ["production-vpc"]
  }
}

# Use the VPC's ID in a new resource
resource "aws_subnet" "new_subnet" {
  vpc_id     = data.aws_vpc.existing.id          # ← VPC ID from data source
  cidr_block = "10.0.99.0/24"

  tags = {
    Name = "new-subnet-in-existing-vpc"
  }
}
```

### Resource vs Data Source — Side by Side

| Feature | Resource | Data Source |
|---------|----------|-------------|
| **Keyword** | `resource` | `data` |
| **Purpose** | CREATE & manage infrastructure | READ existing infrastructure |
| **Reference** | `aws_instance.web.id` | `data.aws_ami.latest.id` |
| **State** | Stored in state file | Read every time (not stored permanently) |
| **Lifecycle** | Create, Update, Delete | Read only |
| **Example** | Create a new EC2 instance | Find the latest Ubuntu AMI |

## 🧪 Hands-On Example — Putting It All Together

Here's a complete, real-world example that uses HCL syntax, resources, AND data sources together:

```hcl
# ============================================================
# main.tf — Create a web server with all the bells and whistles
# ============================================================

# --- Terraform Configuration ---
terraform {
  required_version = ">= 1.5.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

# --- Provider ---
provider "aws" {
  region = "us-east-1"
}

# --- DATA SOURCES (read existing info) ---

# Get the latest Amazon Linux 2 AMI
data "aws_ami" "amazon_linux" {
  most_recent = true
  owners      = ["amazon"]

  filter {
    name   = "name"
    values = ["amzn2-ami-hvm-*-x86_64-gp2"]
  }
}

# Get available AZs
data "aws_availability_zones" "available" {
  state = "available"
}

# Get current account info
data "aws_caller_identity" "current" {}

# --- RESOURCES (create new infrastructure) ---

# 1. Create a VPC
resource "aws_vpc" "main" {
  cidr_block           = "10.0.0.0/16"
  enable_dns_support   = true
  enable_dns_hostnames = true

  tags = {
    Name = "my-vpc"
  }
}

# 2. Create a public subnet
resource "aws_subnet" "public" {
  vpc_id                  = aws_vpc.main.id
  cidr_block              = "10.0.1.0/24"
  availability_zone       = data.aws_availability_zones.available.names[0]
  map_public_ip_on_launch = true

  tags = {
    Name = "public-subnet"
  }
}

# 3. Create an Internet Gateway
resource "aws_internet_gateway" "igw" {
  vpc_id = aws_vpc.main.id

  tags = {
    Name = "my-igw"
  }
}

# 4. Create a route table
resource "aws_route_table" "public" {
  vpc_id = aws_vpc.main.id

  route {
    cidr_block = "0.0.0.0/0"
    gateway_id = aws_internet_gateway.igw.id
  }

  tags = {
    Name = "public-rt"
  }
}

# 5. Associate subnet with route table
resource "aws_route_table_association" "public" {
  subnet_id      = aws_subnet.public.id
  route_table_id = aws_route_table.public.id
}

# 6. Create a security group
resource "aws_security_group" "web" {
  name        = "web-sg"
  description = "Allow HTTP and SSH"
  vpc_id      = aws_vpc.main.id

  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  ingress {
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }

  tags = {
    Name = "web-sg"
  }
}

# 7. Create the EC2 instance
resource "aws_instance" "web" {
  ami                    = data.aws_ami.amazon_linux.id           # ← Data source!
  instance_type          = "t2.micro"
  subnet_id              = aws_subnet.public.id                   # ← Resource reference!
  vpc_security_group_ids = [aws_security_group.web.id]            # ← Resource reference!

  user_data = <<-EOF
    #!/bin/bash
    yum update -y
    yum install -y httpd
    systemctl start httpd
    systemctl enable httpd
    echo "<h1>Hello from Terraform! Account: ${data.aws_caller_identity.current.account_id}</h1>" > /var/www/html/index.html
  EOF

  tags = {
    Name      = "web-server"
    ManagedBy = "Terraform"
  }
}
```

## 📝 Terraform File Structure — How to Organize Your Code

You can put everything in one file, but best practice is to split it:

```
my-terraform-project/
├── main.tf           # Resources and data sources
├── providers.tf      # Provider configuration
├── variables.tf      # Input variable definitions
├── outputs.tf        # Output definitions
├── terraform.tfvars  # Variable values
└── README.md         # Documentation
```

> 💡 **Important:** Terraform reads ALL `.tf` files in a directory automatically. The filenames don't matter — Terraform treats them as one big configuration. But organizing them this way makes your code clean and easy to navigate.

## 🎯 Quick Reference Cheat Sheet

```
HCL SYNTAX:
  block_type "label" { ... }     # Block
  name = value                    # Argument
  "${var.name}"                   # Interpolation
```

## ❓ Common Interview Questions

**Q: What is HCL?**
> HCL is HashiCorp Configuration Language — a declarative language designed for defining infrastructure. It's human-readable, supports comments, variables, functions, and logic — making it more powerful than JSON/YAML for infrastructure as code.

**Q: What's the difference between a Resource and a Data Source?**
> A Resource creates and manages new infrastructure (CRUD). A Data Source only reads information from existing infrastructure — it never creates, updates, or deletes anything.

**Q: How does Terraform determine the order of resource creation?**
> Terraform builds a dependency graph by analyzing resource references. If Resource B uses an attribute from Resource A (`resource_a.name.id`), Terraform knows to create A first. You can also use `depends_on` for explicit dependencies.

**Q: What is the difference between `count` and `for_each`?**
> `count` creates resources by index number (0, 1, 2...) and is best for identical copies. `for_each` creates resources by key and is best for unique items. With `for_each`, adding/removing items doesn't affect other resources; with `count`, removing an item shifts all indices.

**Q: Can you use Terraform with JSON instead of HCL?**
> Yes! Terraform supports `.tf.json` files with the same structure expressed in JSON. However, HCL is the standard because it's more readable and supports comments and logic.
