# 12.5 — Terraform — Variables, Outputs, Locals

> **One Line Answer:** **Variables** let you make your Terraform code reusable by accepting different inputs, **Outputs** let you display or share useful information after resources are created, and **Locals** let you define computed values you can reuse inside your configuration — together, they make your code **flexible, DRY, and clean**.

---

## 🤔 Why Do We Need Variables, Outputs, and Locals?

Let's say you wrote this Terraform code:

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"

  tags = {
    Name        = "web-server"
    Environment = "production"
  }
}
```

It works! But there are problems:

```
❌ What if you want to use this code for "dev", "staging", AND "production"?
   → You'd have to change "production" manually every time.

❌ What if a different team wants t2.large instead of t2.micro?
   → They'd have to dig into your code and modify it.

❌ What if you need the server's IP address after creation?
   → You'd have to go to the AWS console to find it.

❌ What if "web-server-production" is used in 10 different places?
   → You'd have to type it out 10 times, and if it changes, update all 10.
```

**The solution:**

```
Variables → Make it configurable (accept inputs)
Outputs  → Show useful info after creation
Locals   → Compute values and reuse them everywhere
```

---

## 📘 Part 1: Variables (Input Variables)

### What Are Variables?

Variables are like **parameters** to your Terraform code. They let you say:

> "I need a value here, but the person using this code gets to choose what it is."

It's like a restaurant menu where you can **customize your order**:

```
🍔 Burger → Size: ___  |  Cheese: yes/no  |  Sauce: ___
```

Instead of hardcoding everything, you let the user fill in the blanks.

---

### Variable Declaration — How to Define a Variable

You declare variables in a `variable` block:

```hcl
variable "variable_name" {
  description = "What this variable is for"     # Optional but HIGHLY recommended
  type        = string                           # Optional — what kind of value
  default     = "some-default-value"             # Optional — fallback value
}
```

#### Simple Examples:

```hcl
# A string variable (text)
variable "region" {
  description = "The AWS region to deploy resources in"
  type        = string
  default     = "us-east-1"
}

# A number variable
variable "instance_count" {
  description = "How many EC2 instances to create"
  type        = number
  default     = 1
}

# A boolean variable (true/false)
variable "enable_monitoring" {
  description = "Whether to enable detailed monitoring"
  type        = bool
  default     = false
}
```

---

### Using Variables in Your Code

Once declared, use `var.variable_name` to reference a variable:

```hcl
# Declare variables
variable "region" {
  default = "us-east-1"
}

variable "instance_type" {
  default = "t2.micro"
}

variable "environment" {
  default = "dev"
}

# USE them in resources
provider "aws" {
  region = var.region                # ← Use the "region" variable
}

resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = var.instance_type    # ← Use the "instance_type" variable

  tags = {
    Name        = "web-server"
    Environment = var.environment      # ← Use the "environment" variable
  }
}
```

Now this code works for ANY environment, ANY region, and ANY instance size — just by changing the variable values!

---

### Variable Types — All the Types You Can Use

#### Primitive Types:

```hcl
# string — text
variable "name" {
  type    = string
  default = "web-server"
}

# number — integers or decimals
variable "port" {
  type    = number
  default = 8080
}

# bool — true or false
variable "is_public" {
  type    = bool
  default = true
}
```

#### Complex Types:

```hcl
# list — ordered collection of same type
variable "availability_zones" {
  type    = list(string)
  default = ["us-east-1a", "us-east-1b", "us-east-1c"]
}
# Access: var.availability_zones[0] → "us-east-1a"

# list of numbers
variable "ports" {
  type    = list(number)
  default = [80, 443, 8080]
}

# set — like list but no duplicates, no order
variable "unique_tags" {
  type    = set(string)
  default = ["web", "prod", "team-a"]
}

# map — key-value pairs (like a dictionary)
variable "tags" {
  type = map(string)
  default = {
    Environment = "dev"
    Team        = "devops"
    ManagedBy   = "terraform"
  }
}
# Access: var.tags["Environment"] → "dev"

# map of numbers
variable "instance_counts" {
  type = map(number)
  default = {
    dev     = 1
    staging = 2
    prod    = 5
  }
}

# object — a map with a defined structure (specific keys with specific types)
variable "server_config" {
  type = object({
    name          = string
    instance_type = string
    disk_size     = number
    is_public     = bool
  })
  default = {
    name          = "web-server"
    instance_type = "t2.micro"
    disk_size     = 20
    is_public     = true
  }
}
# Access: var.server_config.name → "web-server"
# Access: var.server_config.disk_size → 20

# tuple — a list with specific types per position
variable "mixed_data" {
  type    = tuple([string, number, bool])
  default = ["hello", 42, true]
}
# Access: var.mixed_data[0] → "hello"

# any — accept any type (not recommended — use specific types)
variable "flexible" {
  type    = any
  default = "could be anything"
}
```

#### Nested Complex Types:

```hcl
# List of objects — VERY common in real projects
variable "servers" {
  type = list(object({
    name          = string
    instance_type = string
    subnet_id     = string
  }))
  default = [
    {
      name          = "web-1"
      instance_type = "t2.micro"
      subnet_id     = "subnet-abc"
    },
    {
      name          = "web-2"
      instance_type = "t2.small"
      subnet_id     = "subnet-def"
    }
  ]
}

# Map of objects
variable "environments" {
  type = map(object({
    instance_type = string
    instance_count = number
    enable_monitoring = bool
  }))
  default = {
    dev = {
      instance_type     = "t2.micro"
      instance_count    = 1
      enable_monitoring = false
    }
    prod = {
      instance_type     = "t2.large"
      instance_count    = 3
      enable_monitoring = true
    }
  }
}
# Access: var.environments["prod"].instance_type → "t2.large"
```

### How to Set Variable Values — 5 Ways!

There are **multiple ways** to give variables their values. Here they are, from highest to lowest priority:

#### 1️⃣ Command Line `-var` Flag (Highest Priority):

```bash
terraform apply -var="region=us-west-2" -var="instance_type=t2.large"
```

#### 2️⃣ Command Line `-var-file` Flag:

```bash
terraform apply -var-file="production.tfvars"
```

#### 3️⃣ `terraform.tfvars` or `*.auto.tfvars` Files (Auto-loaded):

**`terraform.tfvars`** — Terraform loads this automatically:

```hcl
# terraform.tfvars
region        = "us-east-1"
instance_type = "t2.micro"
environment   = "production"
instance_count = 3
enable_monitoring = true
```

**`*.auto.tfvars`** — Any file ending in `.auto.tfvars` is also loaded automatically:

```hcl
# production.auto.tfvars
region = "us-east-1"
environment = "production"
```

**Custom `.tfvars` file** — Loaded with `-var-file`:

```hcl
# dev.tfvars
region        = "us-west-2"
instance_type = "t2.micro"
environment   = "dev"

# staging.tfvars
region        = "us-east-1"
instance_type = "t2.small"
environment   = "staging"
```

```bash
# Deploy to dev
terraform apply -var-file="dev.tfvars"

# Deploy to staging
terraform apply -var-file="staging.tfvars"
```

#### 4️⃣ Environment Variables:

Set environment variables with the `TF_VAR_` prefix:

```bash
# Linux / macOS
export TF_VAR_region="us-east-1"
export TF_VAR_instance_type="t2.micro"
export TF_VAR_environment="production"

# Windows PowerShell
$env:TF_VAR_region = "us-east-1"
$env:TF_VAR_instance_type = "t2.micro"

# Then just run terraform — it picks them up automatically!
terraform apply
```

> 💡 This is great for **CI/CD pipelines** where you set environment variables in your pipeline config.

#### 5️⃣ Default Value (Lowest Priority):

```hcl
variable "region" {
  default = "us-east-1"    # Used if nothing else sets it
}
```

#### 6️⃣ Interactive Prompt (If No Value Is Provided):

If a variable has **no default** and you don't provide a value, Terraform will ASK you:

```
$ terraform apply
var.region
  The AWS region to deploy in

  Enter a value: █
```

---

### Variable Priority Order

When the same variable is set in multiple places, Terraform uses this priority (highest wins):

```
1. -var flag on command line          ← HIGHEST priority
2. -var-file flag on command line
3. *.auto.tfvars files (alphabetical)
4. terraform.tfvars
5. TF_VAR_ environment variables
6. Default value in variable block
7. Interactive prompt (if no default)  ← LOWEST priority
```

### Variable Validation

You can add **validation rules** to make sure people provide sensible values:

```hcl
variable "instance_type" {
  description = "EC2 instance type"
  type        = string
  default     = "t2.micro"

  validation {
    condition     = contains(["t2.micro", "t2.small", "t2.medium", "t2.large"], var.instance_type)
    error_message = "Instance type must be one of: t2.micro, t2.small, t2.medium, t2.large."
  }
}

variable "environment" {
  description = "Environment name"
  type        = string

  validation {
    condition     = contains(["dev", "staging", "production"], var.environment)
    error_message = "Environment must be 'dev', 'staging', or 'production'."
  }
}

variable "cidr_block" {
  description = "VPC CIDR block"
  type        = string
  default     = "10.0.0.0/16"

  validation {
    condition     = can(cidrhost(var.cidr_block, 0))
    error_message = "Must be a valid CIDR block (e.g., 10.0.0.0/16)."
  }
}

variable "server_name" {
  description = "Name of the server"
  type        = string

  validation {
    condition     = length(var.server_name) >= 3 && length(var.server_name) <= 50
    error_message = "Server name must be between 3 and 50 characters."
  }

  validation {
    condition     = can(regex("^[a-zA-Z][a-zA-Z0-9-]*$", var.server_name))
    error_message = "Server name must start with a letter and only contain letters, numbers, and hyphens."
  }
}
```

Notice you can have **multiple validation blocks** on a single variable!

---

### Sensitive Variables — Hiding Secrets

For passwords, API keys, and other secrets, mark variables as `sensitive`:

```hcl
variable "db_password" {
  description = "Database password"
  type        = string
  sensitive   = true       # ← This hides the value in logs and terminal output!
}
```

What `sensitive = true` does:

```
✅ Value is HIDDEN in terraform plan output
✅ Value is HIDDEN in terraform apply output
✅ Value is HIDDEN in error messages
❌ Value is NOT encrypted in the state file! (use remote state with encryption for that)
```

Example output with sensitive:

```
# terraform plan
  + resource "aws_db_instance" "main" {
      + password = (sensitive value)     # ← Hidden!
    }
```

---

### Nullable Variables

By default, variables can accept `null`. You can control this:

```hcl
variable "additional_tags" {
  type     = map(string)
  default  = null
  nullable = true          # Default — allows null

  # OR

  nullable = false         # Terraform will raise an error if null is passed
}
```

## 📘 Part 2: Outputs

### What Are Outputs?

Outputs let you **display important information** after Terraform creates your infrastructure.

Think of it like a **receipt** after you order something online:

```
🧾 Order Confirmation:
   - Server IP: 54.23.104.67
   - Database Endpoint: mydb.abc123.us-east-1.rds.amazonaws.com
   - Load Balancer URL: http://my-alb-123456.us-east-1.elb.amazonaws.com
```

Without outputs, you'd have to log into AWS to find this information!

---

### Output Declaration

```hcl
output "output_name" {
  description = "What this output shows"   # Optional but recommended
  value       = expression                  # Required — the value to display
  sensitive   = false                       # Optional — hide in terminal
}
```

---

### Basic Output Examples

```hcl
# Show the instance's public IP
output "instance_public_ip" {
  description = "The public IP address of the web server"
  value       = aws_instance.web.public_ip
}

# Show the instance ID
output "instance_id" {
  description = "The ID of the EC2 instance"
  value       = aws_instance.web.id
}

# Show the S3 bucket name
output "bucket_name" {
  description = "The name of the S3 bucket"
  value       = aws_s3_bucket.my_bucket.bucket
}

# Show the bucket's ARN
output "bucket_arn" {
  description = "The ARN of the S3 bucket"
  value       = aws_s3_bucket.my_bucket.arn
}
```

After running `terraform apply`, you'll see:

```
Apply complete! Resources: 2 added, 0 changed, 0 destroyed.

Outputs:

instance_public_ip = "54.23.104.67"
instance_id = "i-0abc123def456789"
bucket_name = "my-unique-bucket-12345"
bucket_arn = "arn:aws:s3:::my-unique-bucket-12345"
```

---

### Advanced Output Examples

#### Output a Map/Object:

```hcl
output "server_info" {
  description = "All important info about the web server"
  value = {
    id         = aws_instance.web.id
    public_ip  = aws_instance.web.public_ip
    private_ip = aws_instance.web.private_ip
    az         = aws_instance.web.availability_zone
  }
}
```

#### Output a List (When Using `count`):

```hcl
resource "aws_instance" "web" {
  count         = 3
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"
}

# Output ALL public IPs
output "all_instance_ips" {
  description = "Public IPs of all web servers"
  value       = aws_instance.web[*].public_ip    # Splat expression!
}

# Result: ["54.23.1.1", "54.23.1.2", "54.23.1.3"]
```

#### Output a Map (When Using `for_each`):

```hcl
resource "aws_instance" "servers" {
  for_each = {
    web = "t2.micro"
    app = "t2.small"
    db  = "t2.medium"
  }

  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = each.value
}

# Output a map of name → IP
output "server_ips" {
  description = "Map of server name to public IP"
  value       = { for name, instance in aws_instance.servers : name => instance.public_ip }
}

# Result: { web = "54.23.1.1", app = "54.23.1.2", db = "54.23.1.3" }
```

---

### Sensitive Outputs

Just like variables, you can mark outputs as sensitive:

```hcl
output "db_connection_string" {
  description = "Database connection string"
  value       = "postgresql://${var.db_username}:${var.db_password}@${aws_db_instance.main.endpoint}/mydb"
  sensitive   = true       # ← Hidden in terminal output
}
```

Terminal shows:

```
Outputs:

db_connection_string = <sensitive>
```

To see it, run:

```bash
terraform output db_connection_string
```

---

### Using Outputs in the Terminal

```bash
# Show ALL outputs
terraform output

# Show a SPECIFIC output
terraform output instance_public_ip

# Show output as raw value (no quotes) — great for scripting
terraform output -raw instance_public_ip

# Show outputs as JSON
terraform output -json
```

**Using outputs in shell scripts:**

```bash
# Get the IP and SSH into the server
SERVER_IP=$(terraform output -raw instance_public_ip)
ssh ec2-user@$SERVER_IP

# Use in other commands
curl http://$(terraform output -raw load_balancer_dns)
```

---

### Outputs for Module Communication

This is a BIG use case. When you create **modules** (reusable Terraform packages), outputs are how child modules pass information BACK to the parent:

```
Parent Module
  └── calls Child Module (e.g., "networking" module)
       └── Child creates a VPC
       └── Child OUTPUTS the VPC ID
  └── Parent uses the VPC ID in another resource
```

```hcl
# In the child module (modules/networking/outputs.tf):
output "vpc_id" {
  value = aws_vpc.main.id
}

# In the parent (main.tf):
module "networking" {
  source = "./modules/networking"
}

resource "aws_instance" "web" {
  subnet_id = module.networking.vpc_id     # ← Using the module's output!
}
```

> 💡 We'll cover modules in detail in section 12.7!

## 📘 Part 3: Locals

### What Are Locals?

**Locals** (short for "local values") are like **private variables** that you compute inside your Terraform code. They're great for:

- Avoiding repetition (DRY — Don't Repeat Yourself)
- Computing values from variables
- Making complex expressions readable

Think of locals as **scratch paper** where you work out calculations before writing the final answer:

```
Variable  = "What's your name?" (the user provides this)
Local     = "OK, let me compute the full name tag" (internal calculation)
Output    = "Here's the result!" (showing something back)
```

---

### Locals Declaration

```hcl
locals {
  name1 = expression1
  name2 = expression2
  name3 = expression3
}
```

Use them with `local.name` (note: `local`, not `locals`!):

```hcl
locals {
  project_name = "myapp"
}

# Use it:
resource "aws_instance" "web" {
  tags = {
    Project = local.project_name     # ← local.project_name (singular!)
  }
}
```

---

### Why Use Locals? — Real Examples

#### Example 1: Common Tags (THE Most Common Use Case)

Without locals (repetitive and error-prone):

```hcl
resource "aws_instance" "web" {
  # ...
  tags = {
    Project     = "myapp"
    Environment = "production"
    Team        = "devops"
    ManagedBy   = "terraform"
  }
}

resource "aws_s3_bucket" "logs" {
  # ...
  tags = {
    Project     = "myapp"           # ← Same tags repeated!
    Environment = "production"      # ← What if you need to change one?
    Team        = "devops"          # ← You'd have to update EVERY resource!
    ManagedBy   = "terraform"
  }
}

resource "aws_vpc" "main" {
  # ...
  tags = {
    Project     = "myapp"           # ← Again!
    Environment = "production"
    Team        = "devops"
    ManagedBy   = "terraform"
  }
}
```

With locals (clean and DRY):

```hcl
locals {
  common_tags = {
    Project     = var.project_name
    Environment = var.environment
    Team        = "devops"
    ManagedBy   = "terraform"
  }
}

resource "aws_instance" "web" {
  # ...
  tags = merge(local.common_tags, {
    Name = "web-server"             # ← Add resource-specific tags
  })
}

resource "aws_s3_bucket" "logs" {
  # ...
  tags = merge(local.common_tags, {
    Name = "log-bucket"
  })
}

resource "aws_vpc" "main" {
  # ...
  tags = merge(local.common_tags, {
    Name = "main-vpc"
  })
}
```

Now if you need to change a tag, you change it in **ONE place**!

---

#### Example 2: Computing Names

```hcl
variable "project" {
  default = "myapp"
}

variable "environment" {
  default = "production"
}

variable "region" {
  default = "us-east-1"
}

locals {
  # Compute a name prefix used everywhere
  name_prefix = "${var.project}-${var.environment}"

  # Short region name
  region_short = {
    "us-east-1" = "use1"
    "us-west-2" = "usw2"
    "eu-west-1" = "euw1"
  }

  # Full resource name
  full_name = "${local.name_prefix}-${local.region_short[var.region]}"
}

resource "aws_instance" "web" {
  # ...
  tags = {
    Name = "${local.full_name}-web"    # "myapp-production-use1-web"
  }
}

resource "aws_s3_bucket" "logs" {
  bucket = "${local.full_name}-logs"   # "myapp-production-use1-logs"
}
```

---

#### Example 3: Conditional Logic

```hcl
variable "environment" {
  default = "dev"
}

locals {
  # Different settings based on environment
  is_production = var.environment == "production"

  instance_type = local.is_production ? "t2.large" : "t2.micro"
  instance_count = local.is_production ? 3 : 1
  enable_monitoring = local.is_production

  # Max AZs based on environment
  max_azs = local.is_production ? 3 : 1
}

resource "aws_instance" "web" {
  count         = local.instance_count
  instance_type = local.instance_type

  monitoring = local.enable_monitoring

  tags = {
    Name = "web-${count.index}"
  }
}
```

---

#### Example 4: Flattening Complex Data

```hcl
variable "vpc_config" {
  default = {
    vpc_cidr = "10.0.0.0/16"
    public_subnets = {
      "us-east-1a" = "10.0.1.0/24"
      "us-east-1b" = "10.0.2.0/24"
    }
    private_subnets = {
      "us-east-1a" = "10.0.10.0/24"
      "us-east-1b" = "10.0.20.0/24"
    }
  }
}

locals {
  # Flatten into a list of subnet objects for easy iteration
  all_subnets = flatten([
    for type, subnets in {
      public  = var.vpc_config.public_subnets
      private = var.vpc_config.private_subnets
    } : [
      for az, cidr in subnets : {
        type = type
        az   = az
        cidr = cidr
      }
    ]
  ])
}

# Result:
# [
#   { type = "public",  az = "us-east-1a", cidr = "10.0.1.0/24" },
#   { type = "public",  az = "us-east-1b", cidr = "10.0.2.0/24" },
#   { type = "private", az = "us-east-1a", cidr = "10.0.10.0/24" },
#   { type = "private", az = "us-east-1b", cidr = "10.0.20.0/24" },
# ]
```

### Locals vs Variables — When to Use Which?

| Feature | Variable (`var.`) | Local (`local.`) |
|---------|-------------------|-------------------|
| **Who sets it?** | The USER (external input) | The CODE (internal computation) |
| **Purpose** | Accept input from outside | Compute values inside |
| **Can be overridden?** | ✅ Yes (tfvars, CLI, env vars) | ❌ No (hardcoded in the config) |
| **Has a `default`?** | Optional | N/A (always has a value) |
| **Validation?** | ✅ Yes (validation blocks) | ❌ No |
| **Sensitive?** | ✅ Yes (sensitive = true) | ❌ No (but values from sensitive vars stay sensitive) |

**Rule of Thumb:**

```
→ If the value comes from OUTSIDE (user, CI/CD, tfvars) → use a VARIABLE
→ If the value is COMPUTED INSIDE your code → use a LOCAL
→ If the value needs to be shown to the user → use an OUTPUT
```

## 🧪 Complete Real-World Example — Putting It All Together

Here's a complete project using variables, outputs, AND locals:

### File: `variables.tf`

```hcl
# ============================================================
# variables.tf — All input variables for our infrastructure
# ============================================================

variable "project_name" {
  description = "Name of the project"
  type        = string
  default     = "webapp"

  validation {
    condition     = length(var.project_name) >= 2 && length(var.project_name) <= 20
    error_message = "Project name must be 2-20 characters."
  }
}

variable "environment" {
  description = "Deployment environment"
  type        = string
  default     = "dev"

  validation {
    condition     = contains(["dev", "staging", "production"], var.environment)
    error_message = "Must be dev, staging, or production."
  }
}

variable "region" {
  description = "AWS region"
  type        = string
  default     = "us-east-1"
}

variable "instance_type" {
  description = "EC2 instance type"
  type        = string
  default     = "t2.micro"
}

variable "db_password" {
  description = "Database password"
  type        = string
  sensitive   = true

    from_port   = 0
    to_port     = 0

## 🎯 Quick Reference Cheat Sheet

```
VARIABLES:
  variable "name" { }          # Declare a variable
  var.name                      # Reference a variable
  type = string | number | bool | list() | map() | object({}) | set()
  default = value               # Fallback value
  sensitive = true              # Hide in output
  validation { }                # Input validation
  
  Ways to set:
    -var="name=value"           # CLI flag
    -var-file="file.tfvars"     # CLI var file
    terraform.tfvars            # Auto-loaded
    *.auto.tfvars               # Auto-loaded
    TF_VAR_name                 # Environment variable
    default = value             # Default value

OUTPUTS:
  output "name" { }             # Declare an output
  value = expression            # What to show
  sensitive = true              # Hide in terminal
  terraform output              # View all outputs
  terraform output -raw name    # View raw value
  module.name.output_name       # Use module output

LOCALS:
  locals { name = expr }        # Declare locals
  local.name                    # Reference a local (singular!)
  
  Common patterns:
    common_tags = { ... }       # Shared tags
    name_prefix = "..."         # Name computation
    is_prod = var.env == "prod" # Conditional logic
    merge(map1, map2)           # Merge maps

## ❓ Common Interview Questions

**Q: What are the different ways to pass variables in Terraform?**
> CLI `-var` flag, `-var-file` flag, `terraform.tfvars` (auto-loaded), `*.auto.tfvars` (auto-loaded), `TF_VAR_` environment variables, and `default` value in the variable block. Priority goes from CLI flags (highest) to defaults (lowest).

**Q: What is the difference between `variable` and `local`?**
> Variables are external inputs — users provide them via CLI, tfvars files, or env vars. Locals are internal computed values defined in the Terraform code. Variables can be overridden; locals cannot.

**Q: How do you hide sensitive data in Terraform?**
> Mark variables and outputs with `sensitive = true`. This prevents values from showing in plan/apply output. However, sensitive values ARE still stored in the state file, so you must also secure your state file (use remote state with encryption).

**Q: What is a splat expression?**
> A splat expression (`[*]`) is a shorthand for extracting a single attribute from all instances of a resource. For example, `aws_instance.web[*].public_ip` returns a list of all public IPs when using `count`.

**Q: Can a local reference a variable?**
> Yes! Locals commonly reference variables to compute derived values. For example: `local.name_prefix = "${var.project}-${var.environment}"`.
