# 12.6 — Terraform — State Management, Remote State (S3, Consul, Terraform Cloud)

> **One Line Answer:** Terraform **state** is a file that keeps track of everything Terraform has created — it's Terraform's **memory**. Without it, Terraform wouldn't know what infrastructure exists, what needs to change, or what to delete. **Remote state** means storing this file in a shared, secure location (like S3 or Terraform Cloud) instead of on your local machine.

---

## 🤔 What Problem Does State Solve?

Let's say you write this Terraform code and run `terraform apply`:

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"
  tags = {
    Name = "web-server"
  }
}
```

Terraform creates an EC2 instance. Great!

Now the big question: **When you run `terraform apply` again, how does Terraform know the instance already exists?**

```
❌ Does it scan your entire AWS account looking for it? → NO (too slow, too unreliable)
❌ Does it check the resource tags? → NO (tags can be duplicated)
✅ It checks its STATE FILE → YES! This is the answer.
```

The **state file** is Terraform's memory. It says:

```
"Hey, I created an EC2 instance with ID i-0abc123def456789.
 Here's everything about it. Don't create another one."
```

---

## 📘 Part 1: Understanding Terraform State

### What is the State File?

When you run `terraform apply`, Terraform creates a file called `terraform.tfstate` in your working directory. This is a **JSON file** that maps your Terraform code to real infrastructure.

Let's peek inside:

```json
{
  "version": 4,
  "terraform_version": "1.7.0",
  "serial": 3,
  "lineage": "abc123-def456-ghi789",
  "outputs": {
    "instance_ip": {
      "value": "54.23.104.67",
      "type": "string"
    }
  },
  "resources": [
    {
      "mode": "managed",
      "type": "aws_instance",
      "name": "web",
      "provider": "provider[\"registry.terraform.io/hashicorp/aws\"]",
      "instances": [
        {
          "schema_version": 1,
          "attributes": {
            "id": "i-0abc123def456789",
            "ami": "ami-0c55b159cbfafe1f0",
            "instance_type": "t2.micro",
            "public_ip": "54.23.104.67",
            "private_ip": "10.0.1.15",
            "tags": {
              "Name": "web-server"
            }
          }
        }
      ]
    }
  ]
}
```

**What's in here?**

| Field | What It Is |
|-------|-----------|
| `version` | State file format version |
| `terraform_version` | Which version of Terraform created this |
| `serial` | A counter that increases every time state changes |
| `lineage` | Unique ID for this state — prevents accidentally mixing states |
| `outputs` | Values from your `output` blocks |
| `resources` | ALL resources Terraform manages — their types, names, IDs, and every attribute |

---

### What Does State Actually DO?

The state file serves **four critical purposes**:

#### 1️⃣ Mapping — "Which code = which real resource?"

```
Your Code:                          Real World:
resource "aws_instance" "web"  →→→  i-0abc123def456789 (EC2 in AWS)
resource "aws_s3_bucket" "logs" →→→ my-log-bucket-12345 (S3 in AWS)
```

Without state, Terraform wouldn't know that `aws_instance.web` in your code = instance `i-0abc123def456789` in AWS.

#### 2️⃣ Tracking Metadata — Dependencies & Ordering

State stores which resources depend on which, so Terraform knows the order to create/destroy things.

#### 3️⃣ Performance — Caching

Instead of calling the AWS API for EVERY resource during `terraform plan`, Terraform reads the state file first. For large infrastructure (hundreds of resources), this is MUCH faster.

#### 4️⃣ Detecting Drift — "Did someone change things manually?"

Terraform compares:
- What's in your **code** (desired state)
- What's in the **state file** (last known state)
- What's actually in **AWS** (real state)

If they don't match, Terraform tells you and can fix it:

```
~ aws_instance.web
    instance_type: "t2.micro" → "t2.small"
    # Someone changed this manually! Terraform will change it back.
```

---

### State File Location — Where Does It Live?

By default, Terraform stores state **locally**:

```
my-terraform-project/
├── main.tf
├── variables.tf
├── outputs.tf
├── terraform.tfstate          ← Current state
└── terraform.tfstate.backup   ← Previous state (automatic backup)
```

---

### The `terraform.tfstate.backup` File

Every time Terraform updates the state, it automatically saves the PREVIOUS version as `terraform.tfstate.backup`. This is your safety net:

```
terraform apply (first time)
  → Creates terraform.tfstate (serial: 1)

terraform apply (second time)
  → Moves terraform.tfstate → terraform.tfstate.backup (serial: 1)
  → Creates new terraform.tfstate (serial: 2)
```

## ⚠️ Part 2: Problems with Local State

Storing state locally works fine when you're learning or working alone. But in a **real team**, local state is a **disaster**:

### Problem 1: No Sharing

```
Developer A: Has terraform.tfstate on their laptop
Developer B: Has a DIFFERENT terraform.tfstate on their laptop

Both run terraform apply at the same time...

Developer A creates 3 instances
Developer B also creates 3 instances (doesn't know about A's!)

Result: 6 instances instead of 3. Double the cost! 💸
```

### Problem 2: No Locking

```
Developer A: terraform apply (modifying state...)
Developer B: terraform apply (also modifying state at the SAME time!)

Result: State file is CORRUPTED. Infrastructure is in an unknown state. 💥
```

### Problem 3: Security Risk

```
terraform.tfstate contains:
  - Database passwords
  - API keys
  - IP addresses
  - ARNs and account IDs
  
If this file is on someone's laptop:
  - Laptop gets stolen → secrets leaked
  - File accidentally pushed to GitHub → secrets exposed
  - No encryption → anyone with file access can read everything
```

### Problem 4: No Version History

```
Accidentally deleted terraform.tfstate?
  → Terraform thinks NOTHING exists
  → Running terraform apply will try to create EVERYTHING again
  → Duplicate resources, conflicts, chaos! 🔥
```

**The solution to ALL these problems: Remote State.**

---

## 📘 Part 3: Remote State — Storing State in the Cloud

### What is Remote State?

Instead of storing `terraform.tfstate` on your local machine, you store it in a **shared, central location** that everyone on the team can access.

```
LOCAL STATE (bad for teams):
  Developer A → laptop → terraform.tfstate
  Developer B → laptop → terraform.tfstate (different!)

REMOTE STATE (good for teams):
  Developer A ┐
              ├→ S3 Bucket → terraform.tfstate (single source of truth!)
  Developer B ┘
```

### Remote State Benefits

| Feature | Local State | Remote State |
|---------|------------|--------------|
| **Shared access** | ❌ Only on one machine | ✅ Anyone on the team |
| **Locking** | ❌ No protection | ✅ Only one person at a time |
| **Encryption** | ❌ Plain text file | ✅ Encrypted at rest |
| **Versioning** | ❌ Only .backup file | ✅ Full version history |
| **Backup** | ❌ Manual | ✅ Automatic |
| **CI/CD friendly** | ❌ Hard to use in pipelines | ✅ Perfect for automation |

---

### How to Configure Remote State — The `backend` Block

You configure remote state in the `terraform` block using a `backend`:

```hcl
terraform {
  backend "backend_type" {
    # backend-specific configuration
  }
}
```

Let's look at the most popular backends.

## 🪣 Part 4: AWS S3 Backend (Most Popular)

The **S3 backend** stores state in an AWS S3 bucket and uses **DynamoDB for state locking**.

### Why S3 + DynamoDB?

```
S3 Bucket      → Stores the state file (with versioning and encryption)
DynamoDB Table → Provides LOCKING (prevents two people from modifying state at once)
```

### Step 1: Create the S3 Bucket and DynamoDB Table

**You need to create these BEFORE using them as a backend.** You can do this manually in AWS, or with a separate Terraform config (the "bootstrap" approach):

```hcl
# bootstrap/main.tf — Run this FIRST to create the backend infrastructure

provider "aws" {
  region = "us-east-1"
}

# S3 bucket for storing state
resource "aws_s3_bucket" "terraform_state" {
  bucket = "my-company-terraform-state"     # Must be globally unique!

  # Prevent accidental deletion
  lifecycle {
    prevent_destroy = true
  }

  tags = {
    Name      = "Terraform State Bucket"
    ManagedBy = "terraform"
  }
}

# Enable versioning — so you can recover old state files
resource "aws_s3_bucket_versioning" "terraform_state" {
  bucket = aws_s3_bucket.terraform_state.id

  versioning_configuration {
    status = "Enabled"
  }
}

# Enable server-side encryption by default
resource "aws_s3_bucket_server_side_encryption_configuration" "terraform_state" {
  bucket = aws_s3_bucket.terraform_state.id

  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "aws:kms"
    }
  }
}

# Block ALL public access — state files should NEVER be public!
resource "aws_s3_bucket_public_access_block" "terraform_state" {
  bucket = aws_s3_bucket.terraform_state.id

  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

# DynamoDB table for state locking
resource "aws_dynamodb_table" "terraform_locks" {
  name         = "terraform-state-locks"
  billing_mode = "PAY_PER_REQUEST"       # No need to provision capacity
  hash_key     = "LockID"               # Required — Terraform uses this key

  attribute {
    name = "LockID"
    type = "S"                           # S = String
  }

  tags = {
    Name      = "Terraform State Lock Table"
    ManagedBy = "terraform"
  }
}

# Outputs — you'll need these values for your backend config
output "state_bucket_name" {
  value = aws_s3_bucket.terraform_state.bucket
}

output "dynamodb_table_name" {
  value = aws_dynamodb_table.terraform_locks.name
}
```

Run this first:

```bash
cd bootstrap
terraform init
terraform apply
```

### Step 2: Configure Your Project to Use the S3 Backend

Now in your actual project:

```hcl
# main.tf (or backend.tf)
terraform {
  required_version = ">= 1.5.0"

  backend "s3" {
    bucket         = "my-company-terraform-state"    # S3 bucket name
    key            = "projects/webapp/terraform.tfstate"  # Path INSIDE the bucket
    region         = "us-east-1"                     # Bucket's region
    dynamodb_table = "terraform-state-locks"         # DynamoDB table for locking
    encrypt        = true                            # Encrypt state at rest
  }

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}
```

### Step 3: Initialize with the New Backend

```bash
terraform init
```

Terraform will ask:

```
Initializing the backend...

Do you want to copy existing state to the new backend?
  Enter a value: yes

Successfully configured the backend "s3"!
```

Your state is now stored in S3!

---

### Understanding the `key` — Organizing State Files

The `key` is the **path** inside the S3 bucket where the state file is stored. Use it to organize multiple projects:

```
S3 Bucket: my-company-terraform-state/
├── projects/
│   ├── webapp/
│   │   └── terraform.tfstate        ← key = "projects/webapp/terraform.tfstate"
│   ├── api/
│   │   └── terraform.tfstate        ← key = "projects/api/terraform.tfstate"
│   └── database/
│       └── terraform.tfstate        ← key = "projects/database/terraform.tfstate"
├── environments/
│   ├── dev/
│   │   └── terraform.tfstate        ← key = "environments/dev/terraform.tfstate"
│   ├── staging/
│   │   └── terraform.tfstate        ← key = "environments/staging/terraform.tfstate"
│   └── production/
│       └── terraform.tfstate        ← key = "environments/production/terraform.tfstate"
```

> 💡 **Best Practice:** Use a clear naming convention like `{project}/{environment}/terraform.tfstate` so you can find any state file easily.

### How State Locking Works with DynamoDB

When someone runs `terraform apply`:

```
1. Terraform tries to CREATE a lock in DynamoDB:
   { LockID: "my-company-terraform-state/projects/webapp/terraform.tfstate" }

2. If the lock doesn't exist:
   → Lock is created ✅
   → Terraform proceeds with apply
   → After apply, lock is DELETED

3. If the lock ALREADY exists (someone else is running apply):
   → Terraform WAITS and shows:
     "Error: Error acquiring the state lock"
     "Lock Info: ID: abc-123, Who: user@laptop, Operation: OperationTypeApply"
   → You have to wait until the other person finishes
```

This prevents two people from modifying infrastructure at the same time!

#### Force Unlocking (Emergency Only!):

If a lock is stuck (e.g., someone's laptop crashed mid-apply):

```bash
terraform force-unlock LOCK_ID
```

Example:

```bash
terraform force-unlock "abc123-def456-ghi789"
```

> ⚠️ **WARNING:** Only use `force-unlock` if you're SURE nobody is actually running Terraform. Force-unlocking while someone is mid-apply can corrupt your state!

### S3 Backend — Complete Configuration Reference

Here are ALL the options for the S3 backend:

```hcl
terraform {
  backend "s3" {
    # Required
    bucket = "my-terraform-state"        # S3 bucket name
    key    = "path/to/terraform.tfstate" # Path in bucket
    region = "us-east-1"                 # Bucket region

    # Locking (highly recommended)
    dynamodb_table = "terraform-locks"   # DynamoDB table for locking

    # Encryption (highly recommended)
    encrypt    = true                    # Encrypt state at rest
    kms_key_id = "arn:aws:kms:..."      # Optional: custom KMS key

    # Authentication (optional — usually uses AWS CLI credentials)
    profile = "my-aws-profile"           # AWS CLI profile name
    # OR
    access_key = "AKIA..."              # Not recommended — use env vars!
    secret_key = "secret..."            # Not recommended — use env vars!
    
    # Advanced
    workspace_key_prefix = "workspaces" # Prefix for workspace state files
    skip_metadata_api_check = false     # Skip EC2 metadata API check
  }
}
```

## 📗 Part 5: Other Remote State Backends

### HashiCorp Consul Backend

[Consul](https://www.consul.io/) is HashiCorp's service discovery and configuration tool. It can also store Terraform state.

```hcl
terraform {
  backend "consul" {
    address = "consul.example.com:8500"  # Consul server address
    scheme  = "https"                     # Use HTTPS
    path    = "terraform/webapp/state"    # Key path in Consul's KV store
    lock    = true                        # Enable locking (built-in!)
  }
}
```

**When to use Consul:**

```
✅ You already use Consul for service discovery
✅ You want built-in locking (no DynamoDB needed)
✅ You want to keep everything in the HashiCorp ecosystem
❌ You don't want to manage another service (Consul itself)
```

### Azure Backend (azurerm)

For Azure users, store state in an Azure Storage Account:

```hcl
terraform {
  backend "azurerm" {
    resource_group_name  = "terraform-state-rg"
    storage_account_name = "tfstatestorageacct"
    container_name       = "tfstate"
    key                  = "webapp/terraform.tfstate"
  }
}
```

Locking is **built-in** with Azure Blob Storage (using blob leases) — no extra table needed!

For GCP users:

terraform {

Locking is **built-in** with GCS.

## 📘 Part 6: State Operations — Working with State

### Viewing State

```bash
# List all resources in state
terraform state list

# Output:
# aws_instance.web
# aws_s3_bucket.logs
# aws_vpc.main
# aws_subnet.public

# Show detailed info about a specific resource
terraform state show aws_instance.web

# Output:
# resource "aws_instance" "web" {
#     ami                    = "ami-0c55b159cbfafe1f0"
#     arn                    = "arn:aws:ec2:us-east-1:123456789:instance/i-0abc123"
#     instance_type          = "t2.micro"
#     public_ip              = "54.23.104.67"
#     ...
# }

# Pull the entire state file (useful for debugging)
terraform state pull

# Show the output values
terraform output
```

---

### Moving Resources in State

Sometimes you rename a resource in your code. Without state manipulation, Terraform would **destroy the old one and create a new one**:

```hcl
# BEFORE: 
resource "aws_instance" "web" { ... }

# AFTER (you renamed it):
resource "aws_instance" "web_server" { ... }
```

Terraform sees this as "delete `web`, create `web_server`" — destroying your running server!

**Fix it with `terraform state mv`:**

```bash
# Tell Terraform: "web" is now called "web_server" — same resource, just renamed
terraform state mv aws_instance.web aws_instance.web_server
```

Now Terraform knows the existing instance should be tracked under the new name — no destruction needed!

**More `state mv` examples:**

```bash
# Move a resource into a module
terraform state mv aws_instance.web module.compute.aws_instance.web

# Move from one module to another
terraform state mv module.old.aws_instance.web module.new.aws_instance.web

# Move an indexed resource
terraform state mv 'aws_instance.web[0]' 'aws_instance.web["primary"]'
```

---

### Removing Resources from State

Sometimes you want Terraform to **stop managing** a resource without deleting it:

```bash
# Remove from state (resource still exists in AWS, just Terraform forgets about it)
terraform state rm aws_instance.web
```

**When would you do this?**

```
→ You want to manage the resource manually from now on
→ You're moving the resource to a different Terraform project
→ You imported something by mistake
```

> ⚠️ After `state rm`, the resource still exists in AWS! Terraform just stops tracking it.

---

### Importing Existing Resources

What if you have infrastructure that was created **manually** (in the AWS console) or by another tool, and you want Terraform to manage it now?

```bash
# Import an existing EC2 instance into Terraform state
terraform import aws_instance.web i-0abc123def456789
```

**The workflow:**

```
1. Write the resource block in your .tf file (empty is fine):
   resource "aws_instance" "web" { }

2. Run terraform import:
   terraform import aws_instance.web i-0abc123def456789

3. Terraform adds the resource to state

4. Run terraform plan to see what attributes need to be filled in

5. Update your .tf file to match the real resource

6. Run terraform plan again — should show "No changes"
```

**Terraform 1.5+ has a better way — import blocks:**

```hcl
# In your .tf file:
import {
  to = aws_instance.web
  id = "i-0abc123def456789"
}

resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"
  tags = {
    Name = "web-server"
  }
}
```

Then just run:

```bash
terraform plan    # Shows what will be imported
terraform apply   # Imports it
```

> 💡 The `import` block approach is **much better** because it's declarative, can be reviewed in PRs, and generates import plans.

---

### Tainting / Replacing Resources

Sometimes a resource is "broken" and you want to force Terraform to **destroy and recreate** it:

```bash
# Old way (deprecated in Terraform 1.x):
terraform taint aws_instance.web
terraform apply

# New way (recommended):
terraform apply -replace="aws_instance.web"
```

This tells Terraform: "On the next apply, destroy this resource and create a fresh one."

---

### Refreshing State

If someone changed infrastructure manually (outside Terraform), your state file is **out of date**. You can refresh it:

```bash
# Update state to match real infrastructure (no changes to infrastructure)
terraform apply -refresh-only
```

This compares the state file with reality and updates the state — without creating, changing, or deleting anything.

## 📘 Part 7: Remote State Data Source — Reading Another Project's State

Here's a powerful concept: you can **read the state** of another Terraform project! This is how different projects share information.

### The Problem

```
Project A: Creates the VPC
Project B: Creates EC2 instances (needs the VPC ID from Project A)

How does Project B get the VPC ID? 🤔
```

### The Solution: `terraform_remote_state` Data Source

**Project A** (creates VPC and outputs the VPC ID):

```hcl
# Project A — networking/main.tf
resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"
  tags = { Name = "main-vpc" }
}

# Project A — networking/outputs.tf
output "vpc_id" {
  value = aws_vpc.main.id
}

output "public_subnet_ids" {
  value = aws_subnet.public[*].id
}
```

**Project B** (reads Project A's state to get the VPC ID):

```hcl
# Project B — application/main.tf

# Read the state from the networking project
data "terraform_remote_state" "networking" {
  backend = "s3"

  config = {
    bucket = "my-company-terraform-state"
    key    = "projects/networking/terraform.tfstate"
    region = "us-east-1"
  }
}

# Use the VPC ID from Project A's state
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"
  subnet_id     = data.terraform_remote_state.networking.outputs.public_subnet_ids[0]

  tags = {
    Name = "web-in-shared-vpc"
  }
}
```

**How it works:**

```
Project A:
  → Creates VPC
  → Saves state to S3 (including outputs)

Project B:
  → Reads Project A's state from S3
  → Gets the VPC ID from the outputs
  → Uses it in its own resources
```

This is how large organizations split infrastructure into **separate, manageable projects** that can still reference each other!

## 📘 Part 8: State File Security — Keeping State Safe

### Why State Security Matters

Your state file contains **EVERYTHING** about your infrastructure:

```
😱 What's in terraform.tfstate:
  - Database passwords (in plain text!)
  - API keys and secrets
  - Private IP addresses
  - Resource ARNs and IDs
  - Account numbers
  - SSL certificate details
  - Everything Terraform manages
```

### Security Best Practices

#### 1. NEVER Commit State to Git

```gitignore
# .gitignore — ALWAYS add these:
*.tfstate
*.tfstate.*
*.tfstate.backup
.terraform/
```

#### 2. Use Remote State with Encryption

```hcl
backend "s3" {
  bucket  = "my-terraform-state"
  key     = "terraform.tfstate"
  region  = "us-east-1"
  encrypt = true                     # ← Enable encryption
  kms_key_id = "arn:aws:kms:..."    # ← Use your own KMS key
}
```

#### 3. Enable Versioning on S3

```hcl
resource "aws_s3_bucket_versioning" "state" {
  bucket = aws_s3_bucket.terraform_state.id
  versioning_configuration {
    status = "Enabled"               # ← Can recover old state versions
  }
}
```

#### 4. Block Public Access

```hcl
resource "aws_s3_bucket_public_access_block" "state" {
  bucket = aws_s3_bucket.terraform_state.id

  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}
```

#### 5. Restrict Access with IAM

```hcl
# Only specific IAM roles can access the state bucket
resource "aws_s3_bucket_policy" "state" {
  bucket = aws_s3_bucket.terraform_state.id

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Effect    = "Allow"
        Principal = { AWS = "arn:aws:iam::123456789012:role/TerraformRole" }
        Action    = ["s3:GetObject", "s3:PutObject"]
        Resource  = "${aws_s3_bucket.terraform_state.arn}/*"
      },
      {
        Effect    = "Allow"
        Principal = { AWS = "arn:aws:iam::123456789012:role/TerraformRole" }
        Action    = "s3:ListBucket"
        Resource  = aws_s3_bucket.terraform_state.arn
      }
    ]
  })
}
```

#### 6. Enable DynamoDB Encryption

```hcl
resource "aws_dynamodb_table" "terraform_locks" {
  name         = "terraform-locks"
  billing_mode = "PAY_PER_REQUEST"
  hash_key     = "LockID"

  attribute {
    name = "LockID"
    type = "S"
  }

  server_side_encryption {
    enabled = true                   # ← Encrypt lock table too
  }
}
```

## 📘 Part 9: Workspaces — Multiple Environments, One Config

Terraform **workspaces** let you maintain separate state files for different environments using the SAME code:

```bash
# Create workspaces
terraform workspace new dev
terraform workspace new staging
terraform workspace new production

# Switch between workspaces
terraform workspace select dev

# List workspaces
terraform workspace list
# * dev
#   staging
#   production

# Show current workspace
terraform workspace show
# dev
```

**How workspaces work with remote state:**

```
S3 Bucket:
├── env:/
│   ├── dev/
│   │   └── terraform.tfstate        ← dev workspace state
│   ├── staging/
│   │   └── terraform.tfstate        ← staging workspace state
│   └── production/
│       └── terraform.tfstate        ← production workspace state
```

**Using workspace in your code:**

```hcl
locals {
  environment = terraform.workspace

  instance_type = {
    dev        = "t2.micro"
    staging    = "t2.small"
    production = "t2.large"
  }
}

resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = local.instance_type[terraform.workspace]

  tags = {
    Name        = "web-${terraform.workspace}"
    Environment = terraform.workspace
  }
}
```

> ⚠️ **Note:** Many teams prefer **separate directories** (one per environment) over workspaces, because:
> - Workspaces share the same backend config
> - It's easy to accidentally apply to the wrong workspace
> - Separate directories give clearer separation
>
> Both approaches are valid — choose what fits your team!

## ❓ Common Interview Questions

**Q: What is Terraform state and why is it important?**
> Terraform state is a JSON file that maps your Terraform configuration to real-world infrastructure. It's Terraform's source of truth — without it, Terraform wouldn't know what resources exist, their current attributes, or the dependencies between them. It enables plan/apply to work correctly.

**Q: What are the problems with local state?**
> Local state doesn't support team collaboration (no sharing), has no locking (concurrent modifications can corrupt it), isn't encrypted (secrets are in plain text), and has minimal versioning (only one backup file). Remote state solves all of these.

**Q: How do you configure remote state with S3?**
> Use the `backend "s3"` block in the `terraform` configuration. Specify the bucket name, key (path), region, and a DynamoDB table for locking. Enable encryption with `encrypt = true`. Run `terraform init` to migrate from local to remote state.

**Q: What is state locking and why is it important?**
> State locking prevents multiple people from modifying the state file simultaneously. Without it, concurrent `terraform apply` runs could corrupt the state or create conflicting infrastructure. With S3 backend, locking is provided by DynamoDB.

**Q: How do you import existing infrastructure into Terraform?**
> Use `terraform import aws_instance.web i-0abc123` to import an existing resource by its cloud provider ID. In Terraform 1.5+, you can also use `import` blocks in your code for a declarative approach that can be reviewed in pull requests.

**Q: What is the `terraform_remote_state` data source?**
> It allows one Terraform project to read the outputs of another project's state. This enables cross-project references — for example, a networking project outputs VPC IDs, and an application project reads them to deploy instances in that VPC.

**Q: How do you recover from a corrupted state file?**
> If using S3 with versioning enabled, you can restore a previous version of the state file. You can also use `terraform state pull` and `terraform state push` to manually fix state. As a last resort, you can delete the state and re-import all resources.
