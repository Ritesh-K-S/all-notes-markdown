# 12.7 — Terraform Modules — Writing, Publishing & Using

> **Level:** Intermediate  
> **Goal:** Master Terraform Modules — learn how to write reusable, shareable infrastructure code that keeps your projects clean, DRY, and scalable.

---

## 📌 What is a Terraform Module?

Okay, let's start with the simplest explanation:

A **Terraform module** is just a **folder with `.tf` files** in it. That's it. Seriously.

Every time you've been writing Terraform code, you've already been using a module — the **root module**. It's the folder where you run `terraform apply`.

But the real power comes when you **create your own reusable modules** — think of them like **functions in programming**. Instead of copy-pasting the same VPC code in 10 projects, you write it once as a module and just **call** it wherever you need.

### Why Use Modules?

| Problem Without Modules | Solution With Modules |
|---|---|
| Copy-pasting the same code everywhere | Write once, use everywhere |
| Huge `main.tf` files with 1000+ lines | Break into small, focused modules |
| Changing one thing means editing 10 files | Change the module, all consumers get updated |
| No standardization across teams | Modules enforce consistent patterns |
| Hard to test infrastructure | Test modules independently |

### The Mental Model

Think of modules like **LEGO blocks**:

```
🧱 VPC Module       → Creates your network
🧱 EC2 Module       → Creates your servers
🧱 RDS Module       → Creates your databases
🧱 S3 Module        → Creates your storage

Your Project = Snap these blocks together!
```

## 🏗️ Module Structure — How a Module Looks

Every well-structured module has these files:

```
modules/
└── vpc/
    ├── main.tf          # The actual resources
    ├── variables.tf     # Input variables (parameters)
    ├── outputs.tf       # Output values (return values)
    ├── versions.tf      # Provider & Terraform version constraints
    └── README.md        # Documentation (what does this module do?)
```

Let's understand each file:

| File | Purpose | Like in Programming |
|------|---------|-------------------|
| `main.tf` | Contains the actual resources | The function body |
| `variables.tf` | Defines what inputs the module accepts | Function parameters |
| `outputs.tf` | Defines what values the module returns | Return values |
| `versions.tf` | Locks Terraform and provider versions | Dependency versions |
| `README.md` | Explains how to use the module | API documentation |

## ✍️ Writing Your First Module — Step by Step

Let's build a **VPC module** for AWS. This module will create a VPC with subnets.

### Step 1: Create the Directory Structure

```
project/
├── main.tf              # Root module — calls our VPC module
├── variables.tf         # Root-level variables
├── outputs.tf           # Root-level outputs
└── modules/
    └── vpc/
        ├── main.tf      # VPC resources
        ├── variables.tf # VPC module inputs
        └── outputs.tf   # VPC module outputs
```

### Step 2: Write the Module — `modules/vpc/variables.tf`

First, define what inputs your module needs:

```hcl
# modules/vpc/variables.tf

variable "vpc_cidr" {
  description = "The CIDR block for the VPC"
  type        = string
  default     = "10.0.0.0/16"
}

variable "vpc_name" {
  description = "Name tag for the VPC"
  type        = string
}

variable "public_subnet_cidrs" {
  description = "List of CIDR blocks for public subnets"
  type        = list(string)
  default     = ["10.0.1.0/24", "10.0.2.0/24"]
}

variable "availability_zones" {
  description = "List of availability zones to use"
  type        = list(string)
  default     = ["us-east-1a", "us-east-1b"]
}

variable "environment" {
  description = "Environment name (dev, staging, prod)"
  type        = string
}
```

**What's happening here?**
- We're defining 5 input variables that users of this module can set
- Each variable has a `description` (always add this — future you will thank you!)
- `type` tells Terraform what kind of value to expect
- `default` provides a fallback value if the user doesn't specify one
- Variables without a `default` are **required** — the user MUST provide them

### Step 3: Write the Module — `modules/vpc/main.tf`

Now write the actual resources:

```hcl
# modules/vpc/main.tf

# Create the VPC
resource "aws_vpc" "this" {
  cidr_block           = var.vpc_cidr
  enable_dns_hostnames = true
  enable_dns_support   = true

  tags = {
    Name        = "${var.vpc_name}-vpc"
    Environment = var.environment
  }
}

# Create an Internet Gateway
resource "aws_internet_gateway" "this" {
  vpc_id = aws_vpc.this.id

  tags = {
    Name        = "${var.vpc_name}-igw"
    Environment = var.environment
  }
}

# Create Public Subnets
resource "aws_subnet" "public" {
  count = length(var.public_subnet_cidrs)

  vpc_id                  = aws_vpc.this.id
  cidr_block              = var.public_subnet_cidrs[count.index]
  availability_zone       = var.availability_zones[count.index]
  map_public_ip_on_launch = true

  tags = {
    Name        = "${var.vpc_name}-public-subnet-${count.index + 1}"
    Environment = var.environment
    Type        = "Public"
  }
}

# Create a Route Table for Public Subnets
resource "aws_route_table" "public" {
  vpc_id = aws_vpc.this.id

  route {
    cidr_block = "0.0.0.0/0"
    gateway_id = aws_internet_gateway.this.id
  }

  tags = {
    Name        = "${var.vpc_name}-public-rt"
    Environment = var.environment
  }
}

# Associate Public Subnets with Route Table
resource "aws_route_table_association" "public" {
  count = length(var.public_subnet_cidrs)

  subnet_id      = aws_subnet.public[count.index].id
  route_table_id = aws_route_table.public.id
}
```

**What's happening here?**
- `var.vpc_cidr` — We're reading values from `variables.tf` using `var.` prefix
- `aws_vpc.this` — We name resources `this` inside modules (it's a convention since the module name already tells you what it is)
- `count = length(var.public_subnet_cidrs)` — Creates as many subnets as there are CIDRs in the list
- `count.index` — The current iteration number (0, 1, 2...)
- We're tagging everything consistently — that's a huge benefit of modules!

### Step 4: Write the Module — `modules/vpc/outputs.tf`

Define what information the module exposes to the caller:

```hcl
# modules/vpc/outputs.tf

output "vpc_id" {
  description = "The ID of the VPC"
  value       = aws_vpc.this.id
}

output "vpc_cidr" {
  description = "The CIDR block of the VPC"
  value       = aws_vpc.this.cidr_block
}

output "public_subnet_ids" {
  description = "List of public subnet IDs"
  value       = aws_subnet.public[*].id
}

output "internet_gateway_id" {
  description = "The ID of the Internet Gateway"
  value       = aws_internet_gateway.this.id
}
```

**What's happening here?**
- Outputs are like **return values** — they let the caller access important data from the module
- `aws_subnet.public[*].id` — The `[*]` is a **splat expression** — it collects the `id` from ALL subnets into a list
- Other modules or the root module can use these outputs

## 📞 Calling / Using a Module

Now let's use our VPC module from the root module:

### Root `main.tf`

```hcl
# main.tf (Root Module)

provider "aws" {
  region = "us-east-1"
}

# Call our VPC module
module "my_vpc" {
  source = "./modules/vpc"     # Where the module code lives

  vpc_name            = "my-app"
  vpc_cidr            = "10.0.0.0/16"
  public_subnet_cidrs = ["10.0.1.0/24", "10.0.2.0/24", "10.0.3.0/24"]
  availability_zones  = ["us-east-1a", "us-east-1b", "us-east-1c"]
  environment         = "production"
}

# Use outputs from the module
output "vpc_id" {
  value = module.my_vpc.vpc_id
}

output "subnet_ids" {
  value = module.my_vpc.public_subnet_ids
}
```

**Let's break this down:**

| Part | What It Does |
|------|-------------|
| `module "my_vpc"` | Gives this module instance a name (`my_vpc`) |
| `source = "./modules/vpc"` | Points to where the module code lives |
| `vpc_name = "my-app"` | Sets the `vpc_name` variable in the module |
| `module.my_vpc.vpc_id` | Reads the `vpc_id` output FROM the module |

### Key Rule

> After adding or changing a module source, you **MUST** run `terraform init` (or `terraform init -upgrade`). This tells Terraform to download/link the module.

## 📦 Module Sources — Where Can Modules Come From?

Terraform modules can come from **many places**:

### 1. Local Path

```hcl
module "vpc" {
  source = "./modules/vpc"
}
```
- Simplest option — module lives in your project
- No download needed (but still need `terraform init`)

### 2. Terraform Registry (Public)

```hcl
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.1.2"

  name = "my-vpc"
  cidr = "10.0.0.0/16"
  # ... other inputs
}
```
- The **official module registry** at [registry.terraform.io](https://registry.terraform.io)
- Thousands of community-maintained modules
- Always pin the `version`!

### 3. GitHub

```hcl
module "vpc" {
  source = "github.com/my-org/terraform-modules//vpc?ref=v1.2.0"
}
```
- `//vpc` — Subfolder within the repo
- `?ref=v1.2.0` — Pin to a specific git tag/branch/commit

### 4. Git (Generic)

```hcl
module "vpc" {
  source = "git::https://example.com/modules.git//vpc?ref=main"
}

# SSH
module "vpc" {
  source = "git::ssh://git@github.com/my-org/modules.git//vpc?ref=v2.0.0"
}
```

### 5. S3 Bucket

```hcl
module "vpc" {
  source = "s3::https://s3-eu-west-1.amazonaws.com/my-terraform-modules/vpc.zip"
}
```

### 6. Terraform Cloud / Enterprise Private Registry

```hcl
module "vpc" {
  source  = "app.terraform.io/my-org/vpc/aws"
  version = "1.0.0"
}
```

### Quick Reference

| Source | When to Use |
|--------|------------|
| Local path | Development, small projects |
| Terraform Registry | Production — use battle-tested community modules |
| GitHub/Git | Private org modules, custom modules |
| S3/GCS | Air-gapped environments, strict compliance |
| TFC Private Registry | Enterprise, multi-team orgs |

## 🔄 Module Versioning — Don't Break Prod!

**ALWAYS** pin your module versions. Never use `latest` or leave version unspecified for remote modules.

### Terraform Registry Versioning

```hcl
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.1.2"          # Exact version (recommended)
}

# Version constraints (use carefully)
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "~> 5.1"         # Any 5.x but >= 5.1 (pessimistic constraint)
}

module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = ">= 5.0, < 6.0"  # Range
}
```

### Git Tag Versioning

```hcl
module "vpc" {
  source = "git::https://github.com/org/modules.git//vpc?ref=v1.2.0"
}
```

### Why Version Pinning Matters

```
Without pinning:
  Monday:  terraform apply → Uses module v1.0 → Works ✅
  Tuesday: Author pushes breaking changes → Now v2.0
  Tuesday: terraform apply → Uses module v2.0 → BREAKS! 💥

With pinning:
  Monday:  terraform apply → Uses module v1.0 → Works ✅
  Tuesday: Author pushes v2.0
  Tuesday: terraform apply → Still uses v1.0 → Still works ✅
  When ready: You upgrade intentionally → Test → Deploy
```

## 🧩 Passing Data Between Modules

In real projects, modules need to talk to each other. The VPC module creates subnets, and the EC2 module needs those subnet IDs. Here's how:

```hcl
# main.tf

# Module 1: Create the network
module "network" {
  source = "./modules/vpc"

  vpc_name    = "my-app"
  environment = "prod"
}

# Module 2: Create the servers — uses output from Module 1
module "webserver" {
  source = "./modules/ec2"

  subnet_id     = module.network.public_subnet_ids[0]   # ← Uses VPC module's output!
  vpc_id        = module.network.vpc_id                  # ← Uses VPC module's output!
  instance_type = "t3.medium"
  environment   = "prod"
}

# Module 3: Create the database — also uses VPC output
module "database" {
  source = "./modules/rds"

  subnet_ids  = module.network.public_subnet_ids         # ← Passes the full list
  vpc_id      = module.network.vpc_id
  db_name     = "myappdb"
  environment = "prod"
}
```

**The flow:**
```
VPC Module ──outputs──→ EC2 Module (needs subnet_id, vpc_id)
     │
     └──outputs──→ RDS Module (needs subnet_ids, vpc_id)
```

This creates an **implicit dependency**. Terraform knows it must create the VPC first, then the EC2 and RDS (which can be created in parallel since they don't depend on each other).

## 🔁 Using `for_each` and `count` with Modules

### Creating Multiple Instances with `for_each`

```hcl
# Create VPCs for multiple environments
module "vpc" {
  for_each = {
    dev     = { cidr = "10.0.0.0/16", env = "dev" }
    staging = { cidr = "10.1.0.0/16", env = "staging" }
    prod    = { cidr = "10.2.0.0/16", env = "prod" }
  }

  source = "./modules/vpc"

  vpc_name    = "myapp-${each.key}"
  vpc_cidr    = each.value.cidr
  environment = each.value.env
}

# Access outputs
output "dev_vpc_id" {
  value = module.vpc["dev"].vpc_id
}

output "all_vpc_ids" {
  value = { for k, v in module.vpc : k => v.vpc_id }
}
```

**What's happening here?**
- `for_each` creates **3 separate VPCs** — one for dev, staging, and prod
- `each.key` = `"dev"`, `"staging"`, or `"prod"`
- `each.value.cidr` = the CIDR for that environment
- Access outputs using `module.vpc["dev"].vpc_id` (map-style access)

### Creating Multiple Instances with `count`

```hcl
variable "regions" {
  default = ["us-east-1", "eu-west-1", "ap-southeast-1"]
}

module "vpc" {
  count  = length(var.regions)
  source = "./modules/vpc"

  vpc_name    = "myapp-${var.regions[count.index]}"
  vpc_cidr    = "10.${count.index}.0.0/16"
  environment = "prod"
}

# Access outputs
output "first_vpc_id" {
  value = module.vpc[0].vpc_id
}
```

### `for_each` vs `count` — Which to Use?

| Feature | `count` | `for_each` |
|---------|---------|------------|
| Access pattern | By index (`module.vpc[0]`) | By key (`module.vpc["dev"]`) |
| Remove middle item? | Shifts indexes — can destroy/recreate resources! 😱 | Safe — only the removed key is affected ✅ |
| Best for | Lists where order matters | Maps / sets where identity matters |
| **Recommendation** | Use sparingly | **Prefer this** |

## 📝 Module Best Practices

### 1. Keep Modules Focused

```
❌ BAD: One giant module that creates VPC + EC2 + RDS + S3 + IAM
✅ GOOD: Separate modules for VPC, EC2, RDS, S3, IAM
```

Each module should do **one thing** well. Like Unix philosophy: "Do one thing and do it well."

### 2. Always Add Descriptions

```hcl
# ❌ BAD
variable "cidr" {
  type = string
}

# ✅ GOOD
variable "cidr" {
  description = "The CIDR block for the VPC (e.g., 10.0.0.0/16)"
  type        = string
}
```

### 3. Use Validation Rules

```hcl
variable "environment" {
  description = "Environment name"
  type        = string

  validation {
    condition     = contains(["dev", "staging", "prod"], var.environment)
    error_message = "Environment must be dev, staging, or prod."
  }
}

variable "vpc_cidr" {
  description = "VPC CIDR block"
  type        = string

  validation {
    condition     = can(cidrhost(var.vpc_cidr, 0))
    error_message = "Must be a valid CIDR block (e.g., 10.0.0.0/16)."
  }
}
```

### 4. Use Sensible Defaults

```hcl
variable "instance_type" {
  description = "EC2 instance type"
  type        = string
  default     = "t3.micro"     # Safe, cheap default
}
```

### 5. Tag Everything Consistently

```hcl
variable "tags" {
  description = "Common tags to apply to all resources"
  type        = map(string)
  default     = {}
}

# In main.tf
resource "aws_vpc" "this" {
  cidr_block = var.vpc_cidr

  tags = merge(
    var.tags,
    {
      Name        = var.vpc_name
      Environment = var.environment
      ManagedBy   = "terraform"
    }
  )
}
```

### 6. Naming Convention for Modules

If you're publishing a module to the Terraform Registry, follow this naming pattern:

```
terraform-<PROVIDER>-<NAME>

Examples:
  terraform-aws-vpc
  terraform-aws-ec2-instance
  terraform-azurerm-resource-group
  terraform-google-network
```

## 🌍 Publishing a Module to Terraform Registry

Want to share your module with the world? Here's how:

### Prerequisites

1. Your module must be in a **public GitHub repo**
2. Repo name must follow `terraform-<PROVIDER>-<NAME>` format
3. Repo must have **tags** (semantic versioning — `v1.0.0`)
4. Must have these files at the root:
   - `main.tf`
   - `variables.tf`
   - `outputs.tf`
   - `README.md`

### Steps

```
1. Create a GitHub repo named: terraform-aws-my-vpc
2. Push your module code
3. Create a release tag: git tag v1.0.0 && git push --tags
4. Go to registry.terraform.io
5. Sign in with GitHub
6. Click "Publish" → Select your repo
7. Done! 🎉
```

### After Publishing — How Others Use It

```hcl
module "my_vpc" {
  source  = "your-github-username/my-vpc/aws"
  version = "1.0.0"

  vpc_name    = "production"
  environment = "prod"
}
```

## 🧪 A Complete Real-World Example

Let's put it all together — a project that uses multiple modules:

### Project Structure

```
infrastructure/
├── main.tf
├── variables.tf
├── outputs.tf
├── terraform.tfvars
├── versions.tf
└── modules/
    ├── vpc/
    │   ├── main.tf
    │   ├── variables.tf
    │   └── outputs.tf
    ├── security-group/
    │   ├── main.tf
    │   ├── variables.tf
    │   └── outputs.tf
    └── ec2/
        ├── main.tf
        ├── variables.tf
        └── outputs.tf
```

### `versions.tf`

```hcl
terraform {
  required_version = ">= 1.5.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}
```

### `variables.tf`

```hcl
variable "region" {
  description = "AWS region"
  type        = string
  default     = "us-east-1"
}

variable "environment" {
  description = "Environment (dev/staging/prod)"
  type        = string
}

variable "project_name" {
  description = "Project name for resource naming"
  type        = string
}
```

### `terraform.tfvars`

```hcl
region       = "us-east-1"
environment  = "production"
project_name = "myapp"
```

### `main.tf` (Root)

```hcl
provider "aws" {
  region = var.region
}

# Step 1: Create the VPC
module "vpc" {
  source = "./modules/vpc"

  vpc_name    = var.project_name
  environment = var.environment
}

# Step 2: Create Security Group (needs VPC)
module "web_sg" {
  source = "./modules/security-group"

  vpc_id      = module.vpc.vpc_id          # ← Output from VPC module
  name        = "${var.project_name}-web-sg"
  description = "Security group for web servers"
  
  ingress_rules = [
    {
      port        = 80
      protocol    = "tcp"
      cidr_blocks = ["0.0.0.0/0"]
    },
    {
      port        = 443
      protocol    = "tcp"
      cidr_blocks = ["0.0.0.0/0"]
    }
  ]

  environment = var.environment
}

# Step 3: Create EC2 (needs VPC subnet + Security Group)
module "webserver" {
  source = "./modules/ec2"

  instance_type       = "t3.medium"
  subnet_id           = module.vpc.public_subnet_ids[0]   # ← From VPC
  security_group_ids  = [module.web_sg.security_group_id]  # ← From SG
  key_name            = "my-key-pair"
  
  name        = "${var.project_name}-web"
  environment = var.environment
}
```

### `outputs.tf` (Root)

```hcl
output "vpc_id" {
  description = "The VPC ID"
  value       = module.vpc.vpc_id
}

output "web_server_public_ip" {
  description = "Public IP of the web server"
  value       = module.webserver.public_ip
}

output "web_security_group_id" {
  description = "Security Group ID for web servers"
  value       = module.web_sg.security_group_id
}
```

## 🧰 Useful Terraform Commands for Modules

```bash
# Initialize modules (REQUIRED after adding/changing modules)
terraform init

# Re-download modules (if source changed)
terraform init -upgrade

# See what modules are being used
terraform providers

# View the full plan with module resources
terraform plan

# Show all outputs (including module outputs)
terraform output

# Target a specific module resource
terraform apply -target=module.vpc

# Show state of a module resource
terraform state show module.vpc.aws_vpc.this

# List all resources in state (see module prefixes)
terraform state list
# Output:
# module.vpc.aws_vpc.this
# module.vpc.aws_subnet.public[0]
# module.vpc.aws_subnet.public[1]
# module.web_sg.aws_security_group.this
# module.webserver.aws_instance.this
```

## ⚠️ Common Mistakes & How to Avoid Them

### Mistake 1: Forgetting `terraform init` After Adding a Module

```bash
# You add a new module block, then run plan
$ terraform plan
# ERROR: Module not installed

# Fix: Always run init first!
$ terraform init
$ terraform plan
```

### Mistake 2: Not Pinning Module Versions

```hcl
# ❌ BAD — Can break at any time
module "vpc" {
  source = "terraform-aws-modules/vpc/aws"
}

# ✅ GOOD — Predictable, reproducible
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.1.2"
}
```

### Mistake 3: Making Modules Too Generic or Too Specific

```hcl
# ❌ TOO GENERIC — Basically re-creates the provider
module "anything" {
  source        = "./modules/generic"
  resource_type = "aws_instance"
  config        = { ... }
}

# ❌ TOO SPECIFIC — Can't reuse
module "johns_vpc_for_project_x" {
  source = "./modules/project-x-vpc"
  # hardcoded everything inside, no variables
}

# ✅ JUST RIGHT — Focused but flexible
module "vpc" {
  source = "./modules/vpc"
  cidr   = "10.0.0.0/16"
  name   = "my-vpc"
  # sensible defaults for everything else
}
```

### Mistake 4: Circular Dependencies

```hcl
# ❌ Module A uses output of Module B, and Module B uses output of Module A
# This creates a deadlock — Terraform can't figure out what to create first!

# ✅ Design your modules so data flows in ONE direction
# VPC → Security Group → EC2 (never backwards)
```

## 🎯 Summary

| Concept | Key Point |
|---------|-----------|
| What's a module? | A folder with `.tf` files — reusable infrastructure code |
| Why use modules? | DRY code, consistency, testability, collaboration |
| Module structure | `main.tf` + `variables.tf` + `outputs.tf` (minimum) |
| How to call | `module "name" { source = "..." }` |
| Sources | Local path, Registry, GitHub, Git, S3, TFC |
| Version pinning | ALWAYS pin versions for remote modules |
| Data flow | Pass data between modules via inputs and outputs |
| `for_each` vs `count` | Prefer `for_each` — it's safer for additions/removals |
| Publishing | Repo name: `terraform-<PROVIDER>-<NAME>`, push to GitHub, register |
| Best practices | Keep focused, add descriptions, validate inputs, tag resources |
