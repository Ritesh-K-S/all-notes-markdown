## 📌 The Problem — Why Do We Need This?

Here's a scenario every team faces:

You have **one application** but need to deploy it to **multiple environments**:
- **Dev** — for developers to test quickly
- **Staging** — for QA and pre-production testing
- **Prod** — the real deal, customer-facing

Each environment might differ in:
- Instance sizes (dev = `t3.micro`, prod = `t3.xlarge`)
- Number of instances (dev = 1, prod = 3)
- Region or availability zones
- Database size
- Monitoring levels

**The big question:** How do you manage this without copy-pasting your entire Terraform code 3 times?

There are **two main approaches:**

```
Approach 1: Terraform Workspaces (built-in feature)
Approach 2: Directory-based structure (separate folders per env)
```

Let's learn both, and then I'll tell you which one most teams actually use in production.

---

## 🔧 Approach 1: Terraform Workspaces

### What Are Workspaces?

Terraform **Workspaces** are a built-in feature that lets you have **multiple state files** within the same configuration directory.

Think of it like this:
- Same code → Different state → Different infrastructure

By default, every Terraform project starts in a workspace called `default`.

```
Without workspaces:
  terraform.tfstate → ONE set of infrastructure

With workspaces:
  terraform.tfstate.d/
    ├── dev/terraform.tfstate      → Dev infrastructure
    ├── staging/terraform.tfstate  → Staging infrastructure
    └── prod/terraform.tfstate     → Prod infrastructure
```

### Workspace Commands

```bash
# See which workspace you're in
terraform workspace show
# Output: default

# List all workspaces
terraform workspace list
# Output:
# * default

# Create a new workspace
terraform workspace new dev
# Output: Created and switched to workspace "dev"!

terraform workspace new staging
terraform workspace new prod

# List all workspaces now
terraform workspace list
# Output:
#   default
#   dev
# * staging
#   prod

# Switch to a workspace
terraform workspace select prod
# Output: Switched to workspace "prod".

# Delete a workspace (must switch away first)
terraform workspace select default
terraform workspace delete dev
```

### Using Workspaces in Your Code

The magic variable is `terraform.workspace` — it gives you the name of the current workspace.

```hcl
# variables.tf

variable "instance_type" {
  description = "EC2 instance type per environment"
  type        = map(string)
  default = {
    dev     = "t3.micro"
    staging = "t3.small"
    prod    = "t3.xlarge"
  }
}

variable "instance_count" {
  description = "Number of instances per environment"
  type        = map(number)
  default = {
    dev     = 1
    staging = 2
    prod    = 3
  }
}
```

```hcl
# main.tf

provider "aws" {
  region = "us-east-1"
}

resource "aws_instance" "web" {
  count = var.instance_count[terraform.workspace]    # ← Uses workspace name!

  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = var.instance_type[terraform.workspace]   # ← Uses workspace name!

  tags = {
    Name        = "web-${terraform.workspace}-${count.index + 1}"
    Environment = terraform.workspace
  }
}
```

**What's happening here?**
- `terraform.workspace` returns `"dev"`, `"staging"`, or `"prod"` depending on which workspace you've selected
- We use it as a **map key** to look up environment-specific values
- In `dev` workspace → `t3.micro`, 1 instance
- In `prod` workspace → `t3.xlarge`, 3 instances

### Workflow with Workspaces

```bash
# Deploy to Dev
terraform workspace select dev
terraform plan      # Shows: 1x t3.micro
terraform apply

# Deploy to Staging
terraform workspace select staging
terraform plan      # Shows: 2x t3.small
terraform apply

# Deploy to Prod
terraform workspace select prod
terraform plan      # Shows: 3x t3.xlarge
terraform apply
```

### Where Does the State Go?

With **local state**, workspaces create subdirectories:

```
my-project/
├── main.tf
├── variables.tf
├── terraform.tfstate.d/
│   ├── dev/
│   │   └── terraform.tfstate
│   ├── staging/
│   │   └── terraform.tfstate
│   └── prod/
│       └── terraform.tfstate
```

With **remote state** (S3), you typically use the workspace name in the key:

```hcl
terraform {
  backend "s3" {
    bucket = "my-terraform-state"
    key    = "myapp/terraform.tfstate"   # Workspace name is auto-appended
    region = "us-east-1"
  }
}
```

The actual S3 path becomes:
```
s3://my-terraform-state/env:/dev/myapp/terraform.tfstate
s3://my-terraform-state/env:/staging/myapp/terraform.tfstate
s3://my-terraform-state/env:/prod/myapp/terraform.tfstate
```

### Using Locals with Workspaces

For more complex configurations, use `locals`:

```hcl
locals {
  environment = terraform.workspace

  config = {
    dev = {
      instance_type  = "t3.micro"
      instance_count = 1
      db_size        = "db.t3.micro"
      enable_backup  = false
      domain         = "dev.myapp.com"
    }
    staging = {
      instance_type  = "t3.small"
      instance_count = 2
      db_size        = "db.t3.small"
      enable_backup  = true
      domain         = "staging.myapp.com"
    }
    prod = {
      instance_type  = "t3.xlarge"
      instance_count = 3
      db_size        = "db.r5.large"
      enable_backup  = true
      domain         = "myapp.com"
    }
  }

  # Shortcut to current environment's config
  env = local.config[local.environment]
}

# Usage
resource "aws_instance" "web" {
  count         = local.env.instance_count
  instance_type = local.env.instance_type
  # ...
}

resource "aws_db_instance" "main" {
  instance_class = local.env.db_size
  backup_retention_period = local.env.enable_backup ? 7 : 0
  # ...
}
```

**This is clean!** All environment configs are in one place, and you access them with `local.env.<property>`.

## ⚠️ Problems with Workspaces

Now, here's the honest truth — **many production teams avoid workspaces**. Here's why:

### Problem 1: Same Code, Same Backend

All environments share the same backend configuration. You can't have dev state in one S3 bucket and prod state in another (without workarounds).

### Problem 2: Accidental Destruction

```bash
# You THINK you're in dev...
terraform destroy -auto-approve

# But you were actually in prod! 💀
# There's no visual warning — just a workspace name
```

### Problem 3: No Structural Differences

What if `prod` needs a WAF and CloudFront, but `dev` doesn't? You end up with lots of conditionals:

```hcl
resource "aws_cloudfront_distribution" "main" {
  count = terraform.workspace == "prod" ? 1 : 0
  # ... 50 lines of config for something only prod has
}

resource "aws_wafv2_web_acl" "main" {
  count = terraform.workspace == "prod" ? 1 : 0
  # ... another 100 lines only for prod
}
```

This gets **messy fast**.

### Problem 4: CI/CD Complexity

In your CI/CD pipeline, you need to manage workspace selection:

```yaml
# Who sets the workspace? What if it forgets?
- run: terraform workspace select $ENV
- run: terraform apply
```

### When Workspaces ARE Good

| Good For | Not Good For |
|----------|-------------|
| Small projects with identical environments | Environments with structural differences |
| Quick prototyping | Production deployments with strict isolation |
| Personal/learning projects | Multi-team setups |
| Testing module changes across envs | When dev and prod need different backends |

## 📁 Approach 2: Directory-Based Structure (Recommended for Production)

This is what **most production teams** use. Instead of workspaces, you create **separate directories** for each environment.

### Basic Structure

```
infrastructure/
├── modules/                    # Shared modules
│   ├── vpc/
│   ├── ec2/
│   └── rds/
├── environments/
│   ├── dev/
│   │   ├── main.tf
│   │   ├── variables.tf
│   │   ├── outputs.tf
│   │   ├── terraform.tfvars
│   │   └── backend.tf
│   ├── staging/
│   │   ├── main.tf
│   │   ├── variables.tf
│   │   ├── outputs.tf
│   │   ├── terraform.tfvars
│   │   └── backend.tf
│   └── prod/
│       ├── main.tf
│       ├── variables.tf
│       ├── outputs.tf
│       ├── terraform.tfvars
│       └── backend.tf
```

### How It Works

Each environment directory:
1. Has its own `backend.tf` → **Completely separate state** ✅
2. Calls the **same shared modules** → **DRY code** ✅
3. Has its own `terraform.tfvars` → **Environment-specific values** ✅
4. Can have unique resources → **Structural differences** ✅

### Example: Dev Environment

```hcl
# environments/dev/backend.tf
terraform {
  backend "s3" {
    bucket         = "mycompany-terraform-state-dev"    # Dev-specific bucket!
    key            = "myapp/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-locks-dev"
    encrypt        = true
  }
}
```

```hcl
# environments/dev/main.tf
provider "aws" {
  region = var.region
}

module "vpc" {
  source = "../../modules/vpc"         # Points to shared module

  vpc_name    = "myapp-dev"
  vpc_cidr    = "10.0.0.0/16"
  environment = "dev"
}

module "webserver" {
  source = "../../modules/ec2"

  instance_type = "t3.micro"           # Small for dev
  instance_count = 1                   # Only 1 for dev
  subnet_id     = module.vpc.public_subnet_ids[0]
  environment   = "dev"
}

# No WAF, no CloudFront — dev doesn't need it
```

```hcl
# environments/dev/terraform.tfvars
region      = "us-east-1"
environment = "dev"
```

### Example: Prod Environment

```hcl
# environments/prod/backend.tf
terraform {
  backend "s3" {
    bucket         = "mycompany-terraform-state-prod"   # Prod-specific bucket!
    key            = "myapp/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-locks-prod"
    encrypt        = true
  }
}
```

```hcl
# environments/prod/main.tf
provider "aws" {
  region = var.region
}

module "vpc" {
  source = "../../modules/vpc"

  vpc_name    = "myapp-prod"
  vpc_cidr    = "10.2.0.0/16"        # Different CIDR for prod
  environment = "prod"
}

module "webserver" {
  source = "../../modules/ec2"

  instance_type  = "t3.xlarge"        # Bigger for prod
  instance_count = 3                  # 3 instances for prod
  subnet_id      = module.vpc.public_subnet_ids[0]
  environment    = "prod"
}

# Prod-only resources
module "cdn" {
  source = "../../modules/cloudfront"

  domain_name = "myapp.com"
  origin      = module.webserver.lb_dns_name
}

module "waf" {
  source = "../../modules/waf"

  cloudfront_arn = module.cdn.distribution_arn
}
```

```hcl
# environments/prod/terraform.tfvars
region      = "us-east-1"
environment = "prod"
```

### Workflow with Directory Structure

```bash
# Deploy to Dev
cd infrastructure/environments/dev
terraform init
terraform plan
terraform apply

# Deploy to Prod (completely separate — can't accidentally affect prod)
cd infrastructure/environments/prod
terraform init
terraform plan
terraform apply
```

### CI/CD with Directory Structure

```yaml
# GitHub Actions example
name: Deploy Infrastructure

on:
  push:
    branches: [main]

jobs:
  deploy-dev:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: infrastructure/environments/dev    # Just set the directory!
    steps:
      - uses: actions/checkout@v4
      - uses: hashicorp/setup-terraform@v3
      - run: terraform init
      - run: terraform plan
      - run: terraform apply -auto-approve

  deploy-prod:
    needs: deploy-dev       # Only after dev succeeds
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: infrastructure/environments/prod
    steps:
      - uses: actions/checkout@v4
      - uses: hashicorp/setup-terraform@v3
      - run: terraform init
      - run: terraform plan
      # Manual approval in production (not auto-approve!)
```

## 🆚 Workspaces vs Directory Structure — Head to Head

| Feature | Workspaces | Directory Structure |
|---------|-----------|-------------------|
| **State isolation** | Same backend, different state files | Completely separate backends ✅ |
| **Code duplication** | None — same code | Minimal — `main.tf` per env, shared modules |
| **Structural differences** | Painful (lots of `count = workspace == "prod" ? 1 : 0`) | Easy — just add/remove module blocks ✅ |
| **Risk of cross-env damage** | Higher (wrong workspace selected) | Lower (separate directories) ✅ |
| **CI/CD simplicity** | Need workspace selection logic | Just change directory ✅ |
| **Setup effort** | Low | Moderate |
| **Best for** | Small/simple projects | Production, teams, complex infra ✅ |
| **HashiCorp recommendation** | "Not recommended for environment separation" | Recommended ✅ |

### The Verdict

> **Use workspaces** for quick prototyping, learning, and simple projects.  
> **Use directory structure** for production, teams, and anything that matters.

## 🔀 Approach 3: Terragrunt (Bonus — Advanced)

**Terragrunt** is a thin wrapper around Terraform that solves the "repeated code" problem with the directory-based approach. It's very popular in production.

### The Problem Terragrunt Solves

With directory structure, you still have some repeated code — each environment has its own `main.tf`, `backend.tf`, `provider.tf` that are almost identical.

### Terragrunt Structure

```
infrastructure/
├── terragrunt.hcl                # Root config (common settings)
├── modules/
│   ├── vpc/
│   └── ec2/
└── environments/
    ├── dev/
    │   └── terragrunt.hcl        # Just the differences!
    ├── staging/
    │   └── terragrunt.hcl
    └── prod/
        └── terragrunt.hcl
```

### Root `terragrunt.hcl`

```hcl
# infrastructure/terragrunt.hcl

# Auto-generate backend configuration
remote_state {
  backend = "s3"
  generate = {
    path      = "backend.tf"
    if_exists = "overwrite"
  }
  config = {
    bucket         = "mycompany-terraform-state"
    key            = "${path_relative_to_include()}/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
    dynamodb_table = "terraform-locks"
  }
}

# Auto-generate provider configuration
generate "provider" {
  path      = "provider.tf"
  if_exists = "overwrite"
  contents  = <<EOF
provider "aws" {
  region = "us-east-1"
}
EOF
}
```

### Environment `terragrunt.hcl`

```hcl
# environments/dev/terragrunt.hcl

include "root" {
  path = find_in_parent_folders()    # Inherits root config
}

terraform {
  source = "../../modules/vpc"       # Which module to use
}

inputs = {
  vpc_name    = "myapp-dev"
  vpc_cidr    = "10.0.0.0/16"
  environment = "dev"
}
```

```hcl
# environments/prod/terragrunt.hcl

include "root" {
  path = find_in_parent_folders()
}

terraform {
  source = "../../modules/vpc"
}

inputs = {
  vpc_name    = "myapp-prod"
  vpc_cidr    = "10.2.0.0/16"
  environment = "prod"
}
```

### Terragrunt Commands

```bash
# Deploy single environment
cd environments/dev
terragrunt apply

# Deploy ALL environments at once
cd environments
terragrunt run-all apply

# Plan all environments
terragrunt run-all plan
```

### Install Terragrunt

```bash
# macOS
brew install terragrunt

# Linux
curl -L https://github.com/gruntwork-io/terragrunt/releases/latest/download/terragrunt_linux_amd64 -o /usr/local/bin/terragrunt
chmod +x /usr/local/bin/terragrunt

# Windows (Chocolatey)
choco install terragrunt

# Verify
terragrunt --version
```

## 📋 Managing `.tfvars` Files Per Environment

Even without Terragrunt, you can use `.tfvars` files effectively:

### Structure

```
infrastructure/
├── main.tf
├── variables.tf
├── outputs.tf
├── envs/
│   ├── dev.tfvars
│   ├── staging.tfvars
│   └── prod.tfvars
```

### `envs/dev.tfvars`

```hcl
environment    = "dev"
instance_type  = "t3.micro"
instance_count = 1
db_size        = "db.t3.micro"
enable_cdn     = false
domain         = "dev.myapp.com"
```

### `envs/prod.tfvars`

```hcl
environment    = "prod"
instance_type  = "t3.xlarge"
instance_count = 3
db_size        = "db.r5.large"
enable_cdn     = true
domain         = "myapp.com"
```

### Usage

```bash
# Deploy to dev
terraform plan -var-file="envs/dev.tfvars"
terraform apply -var-file="envs/dev.tfvars"

# Deploy to prod
terraform plan -var-file="envs/prod.tfvars"
terraform apply -var-file="envs/prod.tfvars"
```

**⚠️ Warning:** This approach shares the same state file for all environments unless you also change the backend, so it's less safe than the directory-based approach. It works best when combined with workspaces.

### Combining `.tfvars` with Workspaces

```bash
# Create workspaces
terraform workspace new dev
terraform workspace new prod

# Deploy dev
terraform workspace select dev
terraform apply -var-file="envs/dev.tfvars"

# Deploy prod
terraform workspace select prod
terraform apply -var-file="envs/prod.tfvars"
```

Now each environment has separate state AND separate variable values.

## 🏗️ Real-World Recommended Structure

Here's what a production-grade multi-environment setup looks like:

```
infrastructure/
├── README.md
├── modules/                          # Shared, versioned modules
│   ├── networking/
│   │   ├── vpc/
│   │   └── security-groups/
│   ├── compute/
│   │   ├── ec2/
│   │   └── ecs/
│   ├── data/
│   │   ├── rds/
│   │   └── elasticache/
│   └── monitoring/
│       ├── cloudwatch/
│       └── alarms/
├── environments/
│   ├── dev/
│   │   ├── us-east-1/               # Regional separation
│   │   │   ├── main.tf
│   │   │   ├── backend.tf
│   │   │   └── terraform.tfvars
│   │   └── eu-west-1/
│   │       ├── main.tf
│   │       ├── backend.tf
│   │       └── terraform.tfvars
│   ├── staging/
│   │   └── us-east-1/
│   │       ├── main.tf
│   │       ├── backend.tf
│   │       └── terraform.tfvars
│   └── prod/
│       ├── us-east-1/
│       │   ├── main.tf
│       │   ├── backend.tf
│       │   └── terraform.tfvars
│       └── eu-west-1/
│           ├── main.tf
│           ├── backend.tf
│           └── terraform.tfvars
└── global/                           # Global resources (IAM, Route53)
    ├── iam/
    │   ├── main.tf
    │   └── backend.tf
    └── dns/
        ├── main.tf
        └── backend.tf
```

**Key decisions in this structure:**
- **Modules** are shared across all environments
- **Environments** are completely isolated (separate state, backends)
- **Regions** have their own configs (multi-region support)
- **Global** resources (like IAM roles, DNS) are managed separately

## 🎯 Summary

| Approach | When to Use | Isolation Level |
|----------|------------|-----------------|
| **Workspaces** | Small projects, learning, prototyping | Low — shared backend |
| **Directory-based** | Production, teams, complex infra | High — separate everything |
| **Terragrunt** | Large-scale production, DRY fanatics | High — separate + DRY code |
| **`.tfvars` files** | Simple projects with similar environments | Depends on backend setup |

### Key Commands Quick Reference

```bash
# Workspaces
terraform workspace list
terraform workspace new <name>
terraform workspace select <name>
terraform workspace show
terraform workspace delete <name>

# Directory-based
cd environments/<env>
terraform init
terraform plan
terraform apply

# Terragrunt
terragrunt apply
terragrunt run-all apply

# tfvars
terraform apply -var-file="envs/<env>.tfvars"
```

### Golden Rules

1. **Production** → Use directory-based structure (or Terragrunt)
2. **Always** separate state between environments
3. **Never** run `terraform apply` in prod without reviewing the plan
4. **Pin** module versions in all environments
5. **Use** remote state with locking (S3 + DynamoDB or similar)
