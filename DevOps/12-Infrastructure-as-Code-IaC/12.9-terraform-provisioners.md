# 12.9 — Terraform Provisioners — null_resource, local-exec & remote-exec

> **Level:** Intermediate  
> **Goal:** Understand Terraform Provisioners — how to run scripts and commands during infrastructure creation, when to use them, and (more importantly) when NOT to use them.

---

## 📌 What Are Provisioners?

So you've created an EC2 instance with Terraform — great! But the instance is **empty**. It has no software installed, no configuration applied, no application running. 

**Provisioners** are Terraform's way of running **scripts or commands** on a resource **after it's created** (or before it's destroyed).

Think of it like this:

```
Terraform creates the server → Provisioner runs your setup script → Server is ready!
```

### Real-World Example

```
1. Terraform creates an EC2 instance        ← Resource creation
2. Provisioner SSHs into the instance        ← Connection
3. Provisioner runs "apt install nginx"      ← Provisioning
4. Provisioner copies config files           ← Configuration
5. Server is now ready to serve traffic!     ← Done
```

### Types of Provisioners

| Provisioner | What It Does | Where It Runs |
|-------------|-------------|--------------|
| `local-exec` | Runs a command on the **machine running Terraform** (your laptop/CI server) | Locally |
| `remote-exec` | Runs a command on the **remote resource** (the server Terraform just created) | On the remote server via SSH/WinRM |
| `file` | Copies files or directories from local to the **remote resource** | Local → Remote |

---

## ⚠️ The Big Warning — Use Provisioners as a LAST RESORT

Before we dive in, here's something really important that HashiCorp (the creators of Terraform) say themselves:

> **"Provisioners are a Last Resort"** — [Terraform Documentation](https://developer.hashicorp.com/terraform/language/resources/provisioners/syntax)

### Why?

| Problem | Explanation |
|---------|------------|
| **Not in state** | Terraform doesn't track what provisioners do. If a script fails halfway, Terraform has no idea what state the server is in |
| **Not idempotent** | Running `terraform apply` again won't re-run the provisioner (unless you taint the resource) |
| **Fragile** | SSH connections can timeout, scripts can fail, packages might not install |
| **Hard to debug** | When something fails, figuring out what went wrong is painful |
| **Slow** | Provisioners run sequentially and can take a long time |

### What to Use Instead

| Instead of Provisioner | Use This |
|-----------------------|----------|
| Installing software on EC2 | **User Data** (cloud-init) or **AMIs** (Packer) |
| Configuring servers | **Ansible**, **Chef**, **Puppet** |
| Running post-deploy scripts | **CI/CD pipeline steps** |
| Triggering APIs after resource creation | **Terraform `http` provider** or **`null_resource` + local-exec** (acceptable) |

**That said** — provisioners still have valid use cases, and you WILL encounter them in the real world. So let's learn them properly!

## 🖥️ `local-exec` Provisioner — Run Commands Locally

`local-exec` runs a command on the machine where Terraform is executing (your laptop, CI/CD runner, etc.) — **NOT on the remote server**.

### Basic Syntax

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.micro"

  provisioner "local-exec" {
    command = "echo 'Server created with IP: ${self.private_ip}'"
  }
}
```

**What's happening here?**
- After the EC2 instance is created, Terraform runs the `echo` command **on YOUR machine**
- `${self.private_ip}` — `self` refers to the resource that contains this provisioner. It gives you access to that resource's attributes

### Common Use Cases for `local-exec`

#### 1. Write Output to a File

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.micro"

  provisioner "local-exec" {
    command = "echo '${self.public_ip}' >> inventory.txt"
  }
}
```

This adds the server's public IP to a local file — useful for generating Ansible inventories!

#### 2. Run an Ansible Playbook After Instance Creation

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.micro"
  key_name      = "my-key"

  provisioner "local-exec" {
    command = "ansible-playbook -i '${self.public_ip},' --private-key=~/.ssh/my-key.pem playbook.yml"
  }
}
```

**This is actually a very common and legitimate use case!** Terraform creates the infrastructure, Ansible configures it.

#### 3. Send a Notification

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.micro"

  provisioner "local-exec" {
    command = "curl -X POST -H 'Content-type: application/json' --data '{\"text\":\"New server created: ${self.public_ip}\"}' https://hooks.slack.com/services/YOUR/SLACK/WEBHOOK"
  }
}
```

#### 4. Specifying the Interpreter

By default, `local-exec` uses `/bin/sh` on Linux/Mac and `cmd.exe` on Windows. You can change this:

```hcl
# Use bash explicitly
provisioner "local-exec" {
  command     = "echo $HOME"
  interpreter = ["/bin/bash", "-c"]
}

# Use PowerShell on Windows
provisioner "local-exec" {
  command     = "Write-Host 'Hello from PowerShell'"
  interpreter = ["PowerShell", "-Command"]
}

# Use Python
provisioner "local-exec" {
  command     = "print('Hello from Python')"
  interpreter = ["python3", "-c"]
}
```

#### 5. Setting Working Directory and Environment Variables

```hcl
provisioner "local-exec" {
  command     = "./setup.sh"
  working_dir = "/path/to/scripts"

  environment = {
    SERVER_IP   = self.public_ip
    SERVER_NAME = "web-01"
    ENV         = "production"
  }
}
```

## 🌐 `remote-exec` Provisioner — Run Commands on the Remote Server

`remote-exec` SSHs (or WinRMs) into the remote server and runs commands **directly on it**.

### Prerequisites

For `remote-exec` to work, you need:
1. **Network access** from Terraform host to the remote server (SSH port 22 open)
2. **Authentication** — SSH key or password
3. **A `connection` block** telling Terraform how to connect

### Basic Syntax

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.micro"
  key_name      = "my-key-pair"

  # Security group that allows SSH
  vpc_security_group_ids = [aws_security_group.allow_ssh.id]

  # Tell Terraform HOW to connect to this instance
  connection {
    type        = "ssh"
    user        = "ubuntu"                          # Ubuntu AMIs use "ubuntu" user
    private_key = file("~/.ssh/my-key-pair.pem")    # Your SSH private key
    host        = self.public_ip                     # Connect to this IP
  }

  # Run commands on the remote server
  provisioner "remote-exec" {
    inline = [
      "sudo apt update",
      "sudo apt install -y nginx",
      "sudo systemctl start nginx",
      "sudo systemctl enable nginx",
      "echo '<h1>Hello from Terraform!</h1>' | sudo tee /var/www/html/index.html"
    ]
  }
}

# Security group to allow SSH access
resource "aws_security_group" "allow_ssh" {
  name        = "allow-ssh"
  description = "Allow SSH inbound"

  ingress {
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]   # In production, restrict this!
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}
```

**What's happening here?**

1. Terraform creates the EC2 instance
2. Terraform waits for the instance to be reachable via SSH
3. Terraform SSHs in as the `ubuntu` user using the private key
4. Terraform runs each command in `inline` sequentially on the server
5. Nginx gets installed, started, and a custom homepage is set

### The `connection` Block Explained

```hcl
connection {
  type        = "ssh"              # "ssh" for Linux, "winrm" for Windows
  user        = "ubuntu"           # SSH username
  private_key = file("~/.ssh/key.pem")   # SSH private key content
  host        = self.public_ip     # IP to connect to
  port        = 22                 # SSH port (default: 22)
  timeout     = "5m"              # How long to wait for connection
}
```

#### Common SSH Users by AMI

| AMI | Default User |
|-----|-------------|
| Ubuntu | `ubuntu` |
| Amazon Linux / Amazon Linux 2 | `ec2-user` |
| CentOS | `centos` |
| RHEL | `ec2-user` |
| Debian | `admin` |
| SUSE | `ec2-user` |

### Three Ways to Use `remote-exec`

#### 1. `inline` — List of Commands

```hcl
provisioner "remote-exec" {
  inline = [
    "sudo apt update",
    "sudo apt install -y docker.io",
    "sudo usermod -aG docker ubuntu",
    "sudo systemctl enable docker"
  ]
}
```

Each string is a separate command. They run in order.

#### 2. `script` — Run a Local Script on the Remote Server

```hcl
provisioner "remote-exec" {
  script = "scripts/setup.sh"
}
```

Terraform copies `scripts/setup.sh` from your local machine to the remote server, then executes it. The script must be executable and have a shebang (`#!/bin/bash`).

Example `scripts/setup.sh`:
```bash
#!/bin/bash
set -e    # Exit on error

echo "Updating packages..."
sudo apt update && sudo apt upgrade -y

echo "Installing Docker..."
sudo apt install -y docker.io docker-compose
sudo systemctl enable docker
sudo systemctl start docker

echo "Setup complete!"
```

#### 3. `scripts` — Run Multiple Local Scripts

```hcl
provisioner "remote-exec" {
  scripts = [
    "scripts/01-update.sh",
    "scripts/02-install-docker.sh",
    "scripts/03-configure-app.sh"
  ]
}
```

Scripts run in the order listed.

## 📁 `file` Provisioner — Copy Files to the Remote Server

The `file` provisioner copies files or directories from your local machine to the remote server.

### Copy a Single File

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.micro"
  key_name      = "my-key-pair"

  connection {
    type        = "ssh"
    user        = "ubuntu"
    private_key = file("~/.ssh/my-key-pair.pem")
    host        = self.public_ip
  }

  # Copy a config file
  provisioner "file" {
    source      = "configs/nginx.conf"         # Local path
    destination = "/tmp/nginx.conf"            # Remote path
  }

  # Then move it to the right place (needs sudo)
  provisioner "remote-exec" {
    inline = [
      "sudo mv /tmp/nginx.conf /etc/nginx/nginx.conf",
      "sudo systemctl restart nginx"
    ]
  }
}
```

**Why copy to `/tmp` first?** Because the SSH user (`ubuntu`) might not have permission to write directly to `/etc/nginx/`. So we copy to `/tmp` and then use `sudo mv`.

### Copy a Directory

```hcl
provisioner "file" {
  source      = "app/"               # Local directory (trailing slash = copy contents)
  destination = "/home/ubuntu/app"   # Remote directory
}
```

**Important:** 
- `source = "app/"` (with trailing `/`) — copies the **contents** of `app/`
- `source = "app"` (without trailing `/`) — copies the **directory itself**

### Copy Content from a String

```hcl
provisioner "file" {
  content     = "DATABASE_URL=postgresql://user:pass@db:5432/myapp\nREDIS_URL=redis://cache:6379"
  destination = "/home/ubuntu/.env"
}
```

Instead of `source`, you can use `content` to write a string directly to a file on the remote server.

## 🎭 `null_resource` — The Swiss Army Knife

`null_resource` is a special resource that **doesn't create any infrastructure**. It's just a container for provisioners and triggers. Think of it as a "fake" resource that lets you run arbitrary code.

### Why Do We Need It?

Sometimes you want to run a command that isn't tied to a specific resource. For example:
- Run a script after ALL resources are created
- Trigger an external API
- Generate a file from multiple resource outputs
- Run something whenever a specific value changes

### Basic Example

```hcl
resource "null_resource" "post_deploy" {
  # Run after the instance and database are both created
  depends_on = [aws_instance.web, aws_db_instance.main]

  provisioner "local-exec" {
    command = "echo 'All infrastructure is ready! Server: ${aws_instance.web.public_ip}, DB: ${aws_db_instance.main.endpoint}'"
  }
}
```

### Using `triggers` — Re-run When Something Changes

By default, `null_resource` provisioners run **only once** — during creation. The `triggers` argument lets you force it to re-run when specific values change.

```hcl
resource "null_resource" "deploy_app" {
  # Re-run this whenever the app version changes
  triggers = {
    app_version = var.app_version
  }

  provisioner "local-exec" {
    command = "deploy.sh ${var.app_version} ${aws_instance.web.public_ip}"
  }
}
```

**What happens:**
1. First `terraform apply` → triggers = `{app_version = "1.0"}` → runs the deploy script
2. Second `terraform apply` (same version) → triggers haven't changed → **skips** ✅
3. Third `terraform apply` (version changes to "1.1") → triggers changed → **re-runs** the deploy script!

### Advanced Triggers Example

```hcl
resource "null_resource" "configure_cluster" {
  # Re-run whenever any of these change
  triggers = {
    cluster_id     = aws_eks_cluster.main.id
    node_group_id  = aws_eks_node_group.workers.id
    config_hash    = md5(file("k8s-config/deployment.yaml"))   # Re-run if config file changes!
  }

  provisioner "local-exec" {
    command = <<-EOT
      aws eks update-kubeconfig --name ${aws_eks_cluster.main.name} --region us-east-1
      kubectl apply -f k8s-config/
    EOT
  }

  depends_on = [aws_eks_node_group.workers]
}
```

**What's happening here?**
- `triggers` contains 3 values
- If ANY of them change → the `null_resource` is destroyed and recreated → provisioner runs again
- `md5(file(...))` calculates a hash of a file — if the file content changes, the hash changes
- After the EKS cluster and node group are ready, it configures kubectl and deploys K8s manifests

### Always-Run Pattern

Want a provisioner to run **every single time** you do `terraform apply`? Use a timestamp trigger:

```hcl
resource "null_resource" "always_run" {
  triggers = {
    always_run = timestamp()    # Always different → always re-runs
  }

  provisioner "local-exec" {
    command = "echo 'This runs every time! Current time: ${timestamp()}'"
  }
}
```

**⚠️ Use this sparingly** — it makes your apply non-idempotent.

## 🔄 `terraform_data` — The Modern Alternative to `null_resource`

Starting from **Terraform 1.4**, there's a built-in replacement for `null_resource` called `terraform_data`. It works the same way but doesn't require the `null` provider.

```hcl
# Old way (requires hashicorp/null provider)
resource "null_resource" "example" {
  triggers = {
    app_version = var.app_version
  }

  provisioner "local-exec" {
    command = "echo 'Deploying version ${var.app_version}'"
  }
}

# New way (built-in, no extra provider needed)
resource "terraform_data" "example" {
  triggers_replace = [var.app_version]    # Note: slightly different syntax

  provisioner "local-exec" {
    command = "echo 'Deploying version ${var.app_version}'"
  }
}
```

### Key Differences

| Feature | `null_resource` | `terraform_data` |
|---------|----------------|------------------|
| Provider needed | Yes (`hashicorp/null`) | No (built-in) |
| Triggers syntax | `triggers = { key = value }` | `triggers_replace = [value]` |
| Data storage | No | Yes (`input`/`output`) |
| Available since | Forever | Terraform 1.4+ |

### `terraform_data` with Input/Output

```hcl
resource "terraform_data" "app_config" {
  input = {
    version = var.app_version
    port    = 8080
    env     = "production"
  }

  provisioner "local-exec" {
    command = "echo 'Config: ${jsonencode(self.output)}'"
  }
}

# Access stored data
output "app_config" {
  value = terraform_data.app_config.output
}
```

## ⏰ `when` — Creation vs Destruction Provisioners

Provisioners can run at two moments:

### 1. `when = create` (Default)

Runs **when the resource is created**:

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.micro"

  provisioner "local-exec" {
    command = "echo 'Server ${self.public_ip} is UP!'"
    # when = create is the default, so you don't need to write it
  }
}
```

### 2. `when = destroy`

Runs **when the resource is being destroyed** (before it's gone):

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.micro"

  provisioner "local-exec" {
    when    = destroy
    command = "echo 'Server ${self.public_ip} is being DESTROYED!' >> decommission.log"
  }
}
```

### Common Destroy Use Cases

```hcl
# Deregister from load balancer before destroying
provisioner "remote-exec" {
  when = destroy
  inline = [
    "sudo /opt/scripts/deregister-from-lb.sh",
    "sudo /opt/scripts/drain-connections.sh"
  ]
}

# Send cleanup notification
provisioner "local-exec" {
  when    = destroy
  command = "curl -X POST https://api.myapp.com/servers/deregister -d 'ip=${self.private_ip}'"
}

# Remove DNS record
provisioner "local-exec" {
  when    = destroy
  command = "aws route53 change-resource-record-sets --hosted-zone-id ZXXXXX --change-batch file://delete-dns.json"
}
```

**Important:** Destroy provisioners have limitations:
- They can only reference `self` (the resource being destroyed) and variables from the `connection` block
- They **cannot** reference other resources (because they might already be destroyed)

## ❌ `on_failure` — What Happens When a Provisioner Fails?

By default, if a provisioner fails, Terraform marks the resource as **tainted** and stops. You can control this behavior:

### `on_failure = fail` (Default)

```hcl
provisioner "remote-exec" {
  on_failure = fail     # Default — stop everything if this fails

  inline = [
    "sudo apt install -y some-critical-package"
  ]
}
```

If the command fails → Terraform marks the resource as **tainted** → Next `terraform apply` will **destroy and recreate** it.

### `on_failure = continue`

```hcl
provisioner "local-exec" {
  on_failure = continue    # Ignore failures — keep going

  command = "send-notification.sh 'Server created'"
  # If this fails, no big deal — the server is still fine
}
```

If the command fails → Terraform **ignores the error** and continues. Use this for non-critical operations like notifications.

## 🧪 Complete Real-World Example — Full Server Setup

Here's a comprehensive example that uses everything we've learned:

```hcl
# variables.tf
variable "key_name" {
  description = "SSH key pair name"
  type        = string
}

variable "private_key_path" {
  description = "Path to the SSH private key"
  type        = string
}

variable "app_version" {
  description = "Application version to deploy"
  type        = string
  default     = "1.0.0"
}
```

```hcl
# main.tf

provider "aws" {
  region = "us-east-1"
}

# --- Security Group ---
resource "aws_security_group" "web" {
  name        = "web-server-sg"
  description = "Allow HTTP, HTTPS, and SSH"

  ingress {
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["YOUR_IP/32"]   # Restrict SSH to your IP!
  }

  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  ingress {
    from_port   = 443
    to_port     = 443
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

# --- EC2 Instance ---
resource "aws_instance" "web" {
  ami                    = "ami-0c55b159cbfafe1f0"
  instance_type          = "t3.small"
  key_name               = var.key_name
  vpc_security_group_ids = [aws_security_group.web.id]

  tags = {
    Name = "web-server"
  }

  # Connection settings (used by remote-exec and file provisioners)
  connection {
    type        = "ssh"
    user        = "ubuntu"
    private_key = file(var.private_key_path)
    host        = self.public_ip
    timeout     = "5m"
  }

  # Step 1: Copy application files to the server
  provisioner "file" {
    source      = "app/"
    destination = "/home/ubuntu/app"
  }

  # Step 2: Copy nginx config
  provisioner "file" {
    source      = "configs/nginx.conf"
    destination = "/tmp/nginx.conf"
  }

  # Step 3: Install software and configure the server
  provisioner "remote-exec" {
    inline = [
      # Update packages
      "sudo apt update && sudo apt upgrade -y",

      # Install Nginx
      "sudo apt install -y nginx",
      "sudo mv /tmp/nginx.conf /etc/nginx/sites-available/default",
      "sudo systemctl restart nginx",

      # Install Node.js
      "curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -",
      "sudo apt install -y nodejs",

      # Install app dependencies and start
      "cd /home/ubuntu/app && npm install --production",
      "sudo npm install -g pm2",
      "cd /home/ubuntu/app && pm2 start server.js --name myapp",
      "pm2 save",
      "sudo pm2 startup systemd -u ubuntu --hp /home/ubuntu"
    ]
  }

  # Step 4 (on destroy): Graceful shutdown
  provisioner "remote-exec" {
    when = destroy
    inline = [
      "pm2 stop myapp || true",
      "pm2 delete myapp || true"
    ]
  }

  # Step 5 (on destroy): Log decommission locally
  provisioner "local-exec" {
    when    = destroy
    command = "echo '${self.public_ip} decommissioned at ${timestamp()}' >> decommission.log"
  }
}

# --- Post-Deploy: Write inventory file ---
resource "null_resource" "generate_inventory" {
  triggers = {
    instance_id = aws_instance.web.id
  }

  provisioner "local-exec" {
    command = <<-EOT
      cat > inventory.ini <<EOF
      [webservers]
      ${aws_instance.web.public_ip} ansible_user=ubuntu ansible_ssh_private_key_file=${var.private_key_path}
      EOF
    EOT
  }
}

# --- Post-Deploy: Health Check ---
resource "null_resource" "health_check" {
  depends_on = [aws_instance.web]

  triggers = {
    instance_id = aws_instance.web.id
  }

  provisioner "local-exec" {
    command = <<-EOT
      echo "Waiting for server to be ready..."
      for i in $(seq 1 30); do
        if curl -s -o /dev/null -w "%%{http_code}" http://${aws_instance.web.public_ip} | grep -q "200"; then
          echo "Server is UP! ✅"
          exit 0
        fi
        echo "Attempt $i: Not ready yet..."
        sleep 10
      done
      echo "Server failed to start! ❌"
      exit 1
    EOT
  }
}

# --- Deploy new app version (triggered by version change) ---
resource "null_resource" "deploy_app" {
  triggers = {
    app_version = var.app_version
  }

  connection {
    type        = "ssh"
    user        = "ubuntu"
    private_key = file(var.private_key_path)
    host        = aws_instance.web.public_ip
  }

  provisioner "remote-exec" {
    inline = [
      "cd /home/ubuntu/app",
      "git pull origin main",
      "npm install --production",
      "pm2 restart myapp",
      "echo 'Deployed version ${var.app_version}'"
    ]
  }
}
```

```hcl
output "web_server_ip" {
  value = aws_instance.web.public_ip
}

  value = "http://${aws_instance.web.public_ip}"
}
```

## 🆚 Provisioners vs Alternatives — When to Use What

### Decision Tree

```
Need to run something during Terraform apply?
│
├── Is it about server configuration (packages, config files)?
│   ├── YES → Use cloud-init / User Data (best for initial setup)
│   │         Or use Packer to bake an AMI (best for complex setups)
│   │         Or use Ansible after Terraform (best for ongoing config)
│   └── NO ↓
│
├── Is it about triggering an external API or script?
│   ├── YES → Use null_resource + local-exec (acceptable ✅)
│   └── NO ↓
│
├── Is it about deploying application code?
│   ├── YES → Use your CI/CD pipeline (Jenkins, GitHub Actions, etc.)
│   └── NO ↓
│
└── None of the above fit?
    └── OK, use a provisioner — but document WHY
```

### cloud-init / User Data (Preferred Over remote-exec)

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.micro"

  # This is BETTER than remote-exec for initial server setup
  user_data = <<-EOF
    #!/bin/bash
    apt update
    apt install -y nginx
    systemctl start nginx
    echo "<h1>Hello World</h1>" > /var/www/html/index.html
  EOF

  user_data_replace_on_change = true   # Recreate instance if user_data changes

  tags = {
    Name = "web-server"
  }
}
```

**Why this is better:**
- Runs at instance boot time (no SSH needed)
- Cloud-init is built into most Linux AMIs
- No network access from Terraform needed
- More reliable
- Shows up in instance metadata

### Using `templatefile()` with User Data

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.micro"

  user_data = templatefile("scripts/user-data.sh", {
    app_version = var.app_version
    db_endpoint = aws_db_instance.main.endpoint
    environment = var.environment
  })

  tags = {
    Name = "web-${var.environment}"
  }
}
```

`scripts/user-data.sh`:
```bash
#!/bin/bash
set -e

echo "Deploying version ${app_version} in ${environment}"
echo "Database: ${db_endpoint}"

# Install packages
apt update && apt install -y nginx nodejs npm

# Configure app
cat > /home/ubuntu/.env <<EOL
APP_VERSION=${app_version}
DATABASE_URL=postgresql://${db_endpoint}/myapp
NODE_ENV=${environment}
EOL

# Start the application
cd /home/ubuntu/app
npm install --production
npm start
```

## 🧰 Useful Patterns with Provisioners

### Pattern 1: Wait for Resource to Be Ready

```hcl
resource "null_resource" "wait_for_db" {
  depends_on = [aws_db_instance.main]

  provisioner "local-exec" {
    command = <<-EOT
      echo "Waiting for database to accept connections..."
      for i in $(seq 1 60); do
        nc -z ${aws_db_instance.main.address} ${aws_db_instance.main.port} && echo "DB is ready!" && exit 0
        sleep 5
      done
      echo "DB never became ready!"
      exit 1
    EOT
  }
}
```

### Pattern 2: Run Database Migrations

```hcl
resource "null_resource" "db_migration" {
  triggers = {
    migration_hash = md5(file("migrations/latest.sql"))
  }

  depends_on = [aws_db_instance.main]

  provisioner "local-exec" {
    command = "psql -h ${aws_db_instance.main.address} -U admin -d myapp -f migrations/latest.sql"

    environment = {
      PGPASSWORD = var.db_password
    }
  }
}
```

### Pattern 3: Generate Files from Templates

```hcl
resource "null_resource" "generate_kubeconfig" {
  triggers = {
    cluster_id = aws_eks_cluster.main.id
  }

  provisioner "local-exec" {
    command = "aws eks update-kubeconfig --name ${aws_eks_cluster.main.name} --region us-east-1 --kubeconfig ./kubeconfig"
  }
}
```

### Pattern 4: Clean Up External Resources on Destroy

```hcl
resource "null_resource" "cleanup" {
  triggers = {
    cluster_name = aws_eks_cluster.main.name
    region       = var.region
  }

  provisioner "local-exec" {
    when    = destroy
    command = <<-EOT
      echo "Cleaning up load balancers..."
      aws elbv2 describe-load-balancers --region ${self.triggers.region} \
        --query "LoadBalancers[?contains(LoadBalancerName, '${self.triggers.cluster_name}')].LoadBalancerArn" \
        --output text | xargs -I {} aws elbv2 delete-load-balancer --load-balancer-arn {} --region ${self.triggers.region}
    EOT
  }
}
```

**Note:** In destroy provisioners, you can only access `self.triggers`, not other resources. That's why we store values in `triggers`.

## ⚠️ Common Mistakes & Troubleshooting

### Mistake 1: Provisioner Fails But Resource Was Created

```
Error: remote-exec provisioner error
  on main.tf line 25, in resource "aws_instance" "web":
  connection refused
```

The instance was created but provisioner failed. Terraform marks it as **tainted**.

```bash
# Check the tainted resource
terraform state list
# aws_instance.web (tainted)

# Next apply will DESTROY and RECREATE it
terraform apply
# Plan: 1 to destroy, 1 to create

# If you just want to re-run the provisioner without recreating:
# 1. Remove the taint
terraform untaint aws_instance.web
# 2. Use terraform apply -replace (Terraform 1.0+)
terraform apply -replace="null_resource.deploy"
```

### Mistake 2: SSH Connection Times Out

Common reasons:
- Security group doesn't allow SSH (port 22)
- Instance is in a private subnet (no public IP)
- Wrong SSH user for the AMI
- Wrong private key

```hcl
# Debug: Add timeout and verbose logging
connection {
  type        = "ssh"
  user        = "ubuntu"
  private_key = file("~/.ssh/key.pem")
  host        = self.public_ip
  timeout     = "10m"     # Increase timeout
}
```

### Mistake 3: Using Self Reference in Destroy Provisioners

```hcl
# ❌ BAD — Can't reference other resources in destroy provisioner
provisioner "local-exec" {
  when    = destroy
  command = "echo ${aws_security_group.web.id}"    # ERROR!
}

# ✅ GOOD — Store values in triggers, then use self.triggers
resource "null_resource" "cleanup" {
  triggers = {
    sg_id = aws_security_group.web.id    # Save it here
  }

  provisioner "local-exec" {
    when    = destroy
    command = "echo ${self.triggers.sg_id}"    # Access via self.triggers
  }
}
```

## 🎯 Summary

| Provisioner | Runs Where | Use For |
|-------------|-----------|---------|
| `local-exec` | Your machine | API calls, generating files, running Ansible, notifications |
| `remote-exec` | Remote server | Installing software, running scripts (last resort!) |
| `file` | Local → Remote | Copying config files, scripts, application code |
| `null_resource` | N/A (container) | Grouping provisioners, running arbitrary code with triggers |
| `terraform_data` | N/A (built-in) | Same as null_resource but no extra provider needed (Terraform 1.4+) |

### Key Takeaways

| Point | Details |
|-------|---------|
| **Use provisioners as last resort** | Prefer cloud-init, Packer, Ansible, CI/CD |
| **`local-exec` is the most common** | Triggering external tools is a valid use case |
| **`remote-exec` is fragile** | SSH can fail, scripts can fail, state is unknown |
| **`null_resource` + triggers** | Great for running things when specific values change |
| **`when = destroy`** | Cleanup logic before resources are destroyed |
| **`on_failure = continue`** | For non-critical operations (notifications, logging) |
| **Store values in triggers** | Destroy provisioners can only access `self.triggers` |
