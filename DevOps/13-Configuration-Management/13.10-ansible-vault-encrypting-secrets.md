## рҹӨ” Why Do We Need Ansible Vault? A Story First

Imagine this: You're writing an Ansible playbook to deploy a web application. Your playbook needs:
- A **database password** to configure the app
- An **API key** for a third-party service
- A **TLS certificate** for HTTPS

So you do something like this:

```yaml
# vars/main.yml вҖ” DON'T DO THIS! рҹҳұ
db_password: "SuperSecret123!"
api_key: "sk-abc123xyz789"
tls_private_key: |
  -----BEGIN PRIVATE KEY-----
  MIIEvQIBADANBgkqhkiG9w0BAQE...
  -----END PRIVATE KEY-----
```

Then you commit it to Git. **Congratulations, your secrets are now visible to everyone with access to the repo!** рҹҺүрҹ’Җ

You might think: *"I'll just put them in environment variables!"* вҖ” Sure, but then:
- How do you share them across the team?
- How do you version-control them?
- What if someone leaves the team?

This is where **Ansible Vault** comes in. It lets you **encrypt** those files or variables so they look like this:

```
$ANSIBLE_VAULT;1.1;AES256
62313365396662343061393464336163383764316462616562653830653831356330333664613938
3664663737613536663635633238326264643831613534310a393166343964386538643335346566
30316330653637393466313365303432303165626337636234613863333733366334613762373531
```

**Nobody can read them** without the vault password. But Ansible can **decrypt them on-the-fly** when running your playbooks. Best of both worlds! рҹҺҜ

## рҹ“ң What Exactly is Ansible Vault?

Ansible Vault is a **built-in feature** of Ansible (not a separate tool) that provides:

1. **Encryption** вҖ” Encrypt entire files or individual variables using AES-256 encryption
2. **Decryption** вҖ” Automatically decrypt at runtime when you provide the vault password
3. **Version Control Safety** вҖ” Encrypted files can be safely committed to Git
4. **No Extra Infrastructure** вҖ” No external secrets manager needed (though you can integrate with those too)

```
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ

в”Ӯ  inventory/                        в”Ӯ
в”Ӯ                                    в”Ӯ
в”Ӯ  ALL committed to Git safely вң…    в”Ӯ
```

### How It Works Under the Hood

- Uses **AES-256** symmetric encryption (same key to encrypt and decrypt)
- The encrypted file starts with a header: `$ANSIBLE_VAULT;1.1;AES256`
- The rest is hex-encoded encrypted data
- Ansible decrypts in memory вҖ” secrets are **never written to disk** in plain text during execution

## 2пёҸвғЈ Encrypting an Existing File

Already have a `secrets.yml` in plain text? No worries:

```bash
ansible-vault encrypt secrets.yml
```

```
New Vault password: ********
Confirm New Vault password: ********
Encryption successful
```

The file is now encrypted in-place. The original plain text is **gone** вҖ” replaced by the encrypted version.

### Encrypt Multiple Files at Once

```bash
ansible-vault encrypt secrets.yml api_keys.yml tls_certs.yml
```

All three files get encrypted with the **same vault password**.

## 4пёҸвғЈ Editing an Encrypted File

Need to update a secret?

```bash
ansible-vault edit secrets.yml
```

**What happens:**
1. Asks for vault password
2. Decrypts the file temporarily in memory
3. Opens it in your text editor
4. When you save and exit, **re-encrypts automatically**

The file is **never stored in plain text on disk** during this process.

This is one of the **most powerful** features! Instead of encrypting an entire file, you can encrypt just **one variable**:

Now you can paste this **directly into a regular (unencrypted) YAML file**:


- You can have **public variables and encrypted variables in the same file**
- You don't need to encrypt the entire file
- Team members can see the non-sensitive variables without needing the vault password
- `git diff` works for the non-encrypted variables

## рҹ”‘ How to Provide the Vault Password at Runtime

When you run a playbook that uses encrypted data, Ansible needs the vault password. There are several ways to provide it:

### Method 1: Ask at Runtime (Interactive)

```bash
ansible-playbook playbook.yml --ask-vault-pass
```

You'll be prompted:

```
Vault password: ********
```

**Pros:** Simple, secure (password never stored on disk)  
**Cons:** Can't be automated (CI/CD pipelines need automation!)

---

### Method 2: Vault Password File

Store the password in a file:

```bash
# Create a password file
echo "MyVaultPassword123" > ~/.vault_pass.txt

# IMPORTANT: Restrict permissions!
chmod 600 ~/.vault_pass.txt

# Use it
ansible-playbook playbook.yml --vault-password-file ~/.vault_pass.txt
```

**вҡ пёҸ NEVER commit this file to Git!** Add it to `.gitignore`:

```gitignore
# .gitignore
.vault_pass.txt
*.vault_pass
```

**Pros:** Can be automated (great for CI/CD)  
**Cons:** Password is in a file вҖ” must protect it carefully

---

### Method 3: Environment Variable

```bash
export ANSIBLE_VAULT_PASSWORD_FILE=~/.vault_pass.txt
ansible-playbook playbook.yml   # No extra flags needed!
```

Or in your `ansible.cfg`:

```ini
[defaults]
vault_password_file = ~/.vault_pass.txt
```

**Pros:** Set it once, forget it  
**Cons:** Same as password file вҖ” must protect the file

---

### Method 4: Script That Returns the Password

Instead of a plain text file, you can use a **script** that fetches the password from a secure source:

```bash
#!/bin/bash
# vault_pass.sh вҖ” Gets password from macOS Keychain / AWS SSM / etc.

# Example: Get from environment variable
echo "$VAULT_PASSWORD"

# Example: Get from AWS SSM Parameter Store
# aws ssm get-parameter --name "/ansible/vault-password" --with-decryption --query "Parameter.Value" --output text

# Example: Get from HashiCorp Vault
# vault kv get -field=password secret/ansible/vault-pass
```

```bash
chmod +x vault_pass.sh
ansible-playbook playbook.yml --vault-password-file ./vault_pass.sh
```

**Pros:** Most secure вҖ” password can come from any secrets manager  
**Cons:** More setup required

## рҹҶ” Multiple Vault IDs вҖ” Using Different Passwords for Different Secrets

In real projects, you might want **different passwords** for different environments or teams:

```
vault password "dev"  вҶ’ dev secrets
vault password "prod" вҶ’ prod secrets (only ops team knows this)
vault password "certs" вҶ’ TLS certificates (only security team knows this)
```

### Encrypting with a Vault ID

```bash
# Encrypt dev secrets with vault-id "dev"
ansible-vault encrypt --vault-id dev@prompt secrets_dev.yml

# Encrypt prod secrets with vault-id "prod"
ansible-vault encrypt --vault-id prod@prompt secrets_prod.yml

# Encrypt using password files
ansible-vault encrypt --vault-id dev@~/.vault_pass_dev.txt secrets_dev.yml
ansible-vault encrypt --vault-id prod@~/.vault_pass_prod.txt secrets_prod.yml
```

### The Vault ID Format

```
--vault-id <label>@<source>
```

Where `<source>` can be:
- `prompt` вҖ” Ask for password interactively
- `/path/to/file` вҖ” Read from a file
- `/path/to/script.sh` вҖ” Run a script that outputs the password

### Encrypting a String with Vault ID

```bash
ansible-vault encrypt_string --vault-id prod@prompt 'ProdDbPass!' --name 'db_password'
```

Output:

```yaml
db_password: !vault |
  $ANSIBLE_VAULT;1.2;AES256;prod
  ...
```

Notice the header now includes `prod` вҖ” Ansible knows which vault password to use!

### Running with Multiple Vault IDs

```bash
ansible-playbook playbook.yml \
  --vault-id dev@~/.vault_pass_dev.txt \
  --vault-id prod@~/.vault_pass_prod.txt
```

Ansible will **automatically match** each encrypted item with the correct vault password based on the vault ID label.

Here's how to organize vault files in a real Ansible project:

### Approach 1: Separate Encrypted File (Recommended for Beginners)

```
my-ansible-project/
в”ңв”Җв”Җ ansible.cfg
в”ңв”Җв”Җ inventory/
в”Ӯ   в”ңв”Җв”Җ dev
в”Ӯ   в””в”Җв”Җ prod
в”ңв”Җв”Җ group_vars/
в”Ӯ   в”ңв”Җв”Җ all/
в”Ӯ   в”Ӯ   в”ңв”Җв”Җ vars.yml          вҶҗ Public variables (unencrypted)
в”Ӯ   в”Ӯ   в””в”Җв”Җ vault.yml         вҶҗ Secrets (encrypted with ansible-vault)
в”Ӯ   в”ңв”Җв”Җ dev/
в”Ӯ   в”Ӯ   в”ңв”Җв”Җ vars.yml
в”Ӯ   в”Ӯ   в””в”Җв”Җ vault.yml
в”Ӯ   в””в”Җв”Җ prod/
в”Ӯ       в”ңв”Җв”Җ vars.yml
в”Ӯ       в””в”Җв”Җ vault.yml
в”ңв”Җв”Җ roles/
в”ңв”Җв”Җ playbook.yml
в””в”Җв”Җ .gitignore
```

**`group_vars/all/vars.yml`** вҖ” Public variables:

```yaml
# group_vars/all/vars.yml вҖ” NOT encrypted
app_name: "MyWebApp"
app_port: 8080
db_host: "db.example.com"
db_user: "appuser"

# Reference the vault variable (notice the naming convention!)
db_password: "{{ vault_db_password }}"
api_key: "{{ vault_api_key }}"
```

**`group_vars/all/vault.yml`** вҖ” Encrypted secrets:

```yaml
# group_vars/all/vault.yml вҖ” ENCRYPTED with ansible-vault
vault_db_password: "SuperSecret123!"
vault_api_key: "sk-abc123xyz789"
vault_smtp_password: "mailpass456"
```

### The `vault_` Prefix Convention

Notice how we prefix vault variables with `vault_`? This is a **best practice**:

```yaml
# In vars.yml (unencrypted) вҖ” readable, shows what variables exist
db_password: "{{ vault_db_password }}"

# In vault.yml (encrypted) вҖ” holds the actual secrets
vault_db_password: "SuperSecret123!"
```

**Why?**
1. **Clarity** вҖ” Anyone can see `vars.yml` and know what variables exist
2. **Traceability** вҖ” You can `grep` for `vault_` to find all secrets
3. **Separation** вҖ” Public config and secrets are clearly separated
4. **Ease of Use** вҖ” In playbooks, you just use `db_password` вҖ” you don't need to know it's from vault

## рҹ”’ Ansible Vault + `no_log` вҖ” Preventing Secret Leaks in Output

Even after decryption, Ansible might **print secrets in the output** (especially in debug messages or when tasks fail). Use `no_log: true` to prevent this:

```yaml
- name: Set database password
  mysql_user:
    name: appuser
    password: "{{ db_password }}"
    priv: "mydb.*:ALL"
  no_log: true   # вҶҗ This prevents Ansible from showing the password in output!
```

**Without `no_log`:**

```
TASK [Set database password] *****
changed: [db-server] => {"changed": true, "password": "SuperSecret123!"}
```

**With `no_log: true`:**

```
TASK [Set database password] *****
changed: [db-server] => {"censored": "the output has been hidden due to the fact that 'no_log: true' was set"}
```

**Always use `no_log: true` on tasks that handle secrets!** рҹ”җ

## рҹҶҡ Ansible Vault vs External Secrets Managers

| Feature | Ansible Vault | HashiCorp Vault | AWS Secrets Manager |
|---------|--------------|-----------------|---------------------|
| **What it is** | File encryption tool | Full secrets platform | Cloud secrets service |
| **Cost** | Free (built into Ansible) | Free (OSS) / Paid (Enterprise) | Pay per secret + API calls |
| **Encryption** | AES-256 | Multiple algorithms | AWS KMS |
| **Access Control** | Vault password | Policies, tokens, RBAC | IAM policies |
| **Rotation** | Manual | вң… Automatic | вң… Automatic |
| **Audit Logging** | вқҢ No | вң… Yes | вң… Yes |
| **Dynamic Secrets** | вқҢ No | вң… Yes (DB credentials, AWS keys) | Limited |
| **Best For** | Small-medium teams, simple setups | Enterprise, multi-tool environments | AWS-native environments |

### Using HashiCorp Vault with Ansible

You can actually use Ansible Vault (for file encryption) **AND** HashiCorp Vault (for dynamic secrets) together:

```yaml
# Lookup a secret from HashiCorp Vault at runtime
- name: Get database password from HashiCorp Vault
  set_fact:
    db_password: "{{ lookup('hashi_vault', 'secret/data/myapp/db:password') }}"
  no_log: true
```

## вҡ пёҸ Common Mistakes & Gotchas

### Mistake 1: Committing the Vault Password File


### Mistake 2: Forgetting to Encrypt Before Committing

```bash
# Oops, I edited the file and forgot to re-encrypt!
git diff
# Shows plain-text secrets рҹҳұ

# Always double-check:
head -1 group_vars/all/vault.yml
# Should show: $ANSIBLE_VAULT;1.1;AES256
```

### Mistake 3: Same Vault Password for All Environments

```bash
# Using one password for dev AND prod? Bad idea!
# If the dev password leaks, prod secrets are compromised too.

# Use vault IDs to separate them:
ansible-vault encrypt --vault-id dev@prompt group_vars/dev/vault.yml
ansible-vault encrypt --vault-id prod@prompt group_vars/prod/vault.yml
```

### Mistake 4: Not Using `no_log`

If a task fails, Ansible dumps all the variables вҖ” including secrets! Always use `no_log: true` on sensitive tasks.

### Mistake 5: Encrypting Too Much

Don't encrypt your entire `vars.yml` file if only 2 variables are secrets. Use inline encryption (`encrypt_string`) or the separate `vault.yml` approach.

```bash
# --- Create & Encrypt ---

ansible-vault encrypt existing.yml                    # Encrypt existing file
ansible-vault encrypt file1.yml file2.yml             # Encrypt multiple files
ansible-vault encrypt_string 'secret' --name 'var'    # Encrypt single variable

# --- View & Edit ---
ansible-vault view secrets.yml                        # View without decrypting file
ansible-vault edit secrets.yml                        # Edit encrypted file

# --- Decrypt ---
ansible-vault decrypt secrets.yml                     # Decrypt to plain text

# --- Rekey (change password) ---
ansible-vault rekey secrets.yml                       # Change vault password
ansible-vault rekey file1.yml file2.yml               # Rekey multiple files

# --- Running Playbooks ---
ansible-playbook site.yml --ask-vault-pass                           # Interactive
ansible-playbook site.yml --vault-password-file ~/.vault_pass.txt    # Password file
ansible-playbook site.yml --vault-id dev@prompt                      # Vault ID + prompt
ansible-playbook site.yml --vault-id dev@~/.vault_dev.txt            # Vault ID + file

# --- ansible.cfg shortcut ---
# [defaults]
# vault_password_file = ~/.vault_pass.txt

# --- Environment variable ---
export ANSIBLE_VAULT_PASSWORD_FILE=~/.vault_pass.txt
```

## рҹ§  Key Takeaways

1. **Ansible Vault encrypts secrets** using AES-256 вҖ” safe to commit to Git
2. **Two approaches:** Encrypt entire files (`ansible-vault encrypt`) or single variables (`encrypt_string`)
3. **Use the `vault_` prefix** convention to keep secrets traceable
4. **Separate vault files** per environment using vault IDs
5. **Never commit** the vault password file вҖ” always add it to `.gitignore`
6. **Always use `no_log: true`** on tasks that handle secrets
7. **Vault password scripts** can integrate with external secrets managers for CI/CD
8. **Ansible Vault is great for small-medium teams** вҖ” for enterprise needs, consider combining it with HashiCorp Vault or AWS Secrets Manager
