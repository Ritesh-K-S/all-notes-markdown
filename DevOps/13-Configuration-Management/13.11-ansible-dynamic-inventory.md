## 🤔 The Problem — Why Static Inventory Breaks Down

In the previous notes, you learned about Ansible inventory files like this:

```ini
# inventory/hosts — Static Inventory
[webservers]
192.168.1.10
192.168.1.11
192.168.1.12

[databases]
192.168.1.20
192.168.1.21
```

This works fine when you have **5-10 servers** that rarely change. But in the real cloud world:

- **Auto Scaling Groups** spin up/down servers based on traffic — you don't know how many web servers exist at any moment!
- **IP addresses change** every time an instance is recreated
- **Multiple environments** (dev, staging, prod) have different servers
- **Hundreds of servers** across multiple regions — maintaining a static file is a nightmare

Imagine this scenario:

```
Monday:    3 web servers running (10.0.1.5, 10.0.1.6, 10.0.1.7)
Tuesday:   Traffic spike → Auto Scaling adds 5 more servers
Wednesday: Traffic drops → Auto Scaling removes 4 servers
Thursday:  One server crashes, new one spins up with a different IP
```

**Are you going to edit your inventory file every single time?** Of course not! 😅

That's where **Dynamic Inventory** comes in.

---

## 📜 What is Dynamic Inventory?

Dynamic inventory is a mechanism where Ansible **queries an external source** (like AWS, Azure, GCP, or even a database) to get the list of hosts **at runtime**.

```
┌─────────────────────────────────────────────┐
│              Static Inventory                │
│                                              │
│  You write hosts manually in a file          │
│  inventory/hosts → [webservers] 10.0.1.5...  │
│                                              │
│  ❌ Outdated the moment a server changes     │
└─────────────────────────────────────────────┘

              vs.

┌─────────────────────────────────────────────┐
│             Dynamic Inventory                │
│                                              │
│  Ansible asks the cloud: "What's running?"   │
│                                              │
│  ansible-playbook → plugin → AWS API         │
│                              ↓               │
│                     "Here are your           │
│                      12 running EC2s,        │
│                      grouped by tags,        │
│                      regions, etc."          │
│                                              │
│  ✅ Always real-time, always accurate        │
└─────────────────────────────────────────────┘
```

### Two Ways to Do Dynamic Inventory

| Method | How It Works | Recommended? |
|--------|-------------|-------------|
| **Inventory Scripts** (Legacy) | A Python/Bash script that outputs JSON | ❌ Old way |
| **Inventory Plugins** (Modern) | A YAML config file + built-in Ansible plugin | ✅ Use this! |

We'll focus on **inventory plugins** — they're the modern, recommended approach.

---

## ⚙️ How Inventory Plugins Work

1. You create a YAML file (e.g., `aws_ec2.yml`) with configuration
2. The file name must end with a **specific pattern** that Ansible recognizes
3. When you run Ansible, the plugin calls the cloud provider's API
4. The API returns a list of running instances with their metadata
5. Ansible groups the instances based on tags, regions, instance types, etc.
6. Your playbook runs against these dynamically discovered hosts

```
┌──────────────────┐     ┌──────────────────┐     ┌──────────────────┐
│  aws_ec2.yml     │ ──→ │  amazon.aws.     │ ──→ │  AWS EC2 API     │
│  (config file)   │     │  aws_ec2 plugin  │     │  (returns hosts) │
└──────────────────┘     └──────────────────┘     └──────────────────┘
                                                          │
                                                          ▼
                                                  ┌──────────────────┐
                                                  │  Ansible gets:   │
                                                  │  - Host IPs      │
                                                  │  - Tags          │
                                                  │  - Regions       │
                                                  │  - Instance type │
                                                  │  - Security grps │
                                                  │  - And more...   │
                                                  └──────────────────┘
```

## ☁️ AWS Dynamic Inventory — Complete Guide

### Prerequisites

```bash
# 1. Install the AWS Ansible collection
ansible-galaxy collection install amazon.aws

# 2. Install boto3 (AWS SDK for Python) — the plugin needs this
pip install boto3 botocore

# 3. Configure AWS credentials (any of these methods):

# Method A: Environment variables
export AWS_ACCESS_KEY_ID="AKIA..."
export AWS_SECRET_ACCESS_KEY="wJalr..."
export AWS_DEFAULT_REGION="us-east-1"

# Method B: AWS CLI profile
aws configure
# Enter your access key, secret key, region

# Method C: IAM Role (if running on an EC2 instance — best for production!)
# No config needed — Ansible auto-detects the instance role
```

### Step 1: Create the Inventory Plugin File

The file **must** end with `aws_ec2.yml` or `aws_ec2.yaml`:

```yaml
# inventory/aws_ec2.yml
---
plugin: amazon.aws.aws_ec2

# Which AWS regions to search
regions:
  - us-east-1
  - us-west-2
  - eu-west-1

# Only include running instances (not stopped, terminated, etc.)
filters:
  instance-state-name: running

# What extra info to fetch for each instance
hostnames:
  # Try private IP first, fall back to public IP
  - private-ip-address
  - dns-name

# How to group instances (this is the magic! ✨)
keyed_groups:
  # Group by AWS tags (e.g., tag "Environment=prod" → group "env_prod")
  - key: tags.Environment
    prefix: env
    separator: "_"

  # Group by instance type (e.g., "instance_type_t3_micro")
  - key: instance_type
    prefix: instance_type
    separator: "_"

  # Group by region
  - key: placement.region
    prefix: aws_region
    separator: "_"

  # Group by a custom "Role" tag (e.g., tag "Role=webserver" → group "role_webserver")
  - key: tags.Role
    prefix: role
    separator: "_"

# Add all instances to groups based on their tags
groups:
  # Custom group: all instances with tag "Application=myapp"
  myapp: "'myapp' in (tags.Application | default(''))"

# Compose additional host variables
compose:
  # Set ansible_host to the private IP
  ansible_host: private_ip_address
  # Set connection user based on AMI
  ansible_user: "'ubuntu' if 'ubuntu' in (tags.OS | default('')) else 'ec2-user'"
```

Let's break down each section:

**`plugin`** — Tells Ansible which plugin to use. Must be `amazon.aws.aws_ec2`.

**`regions`** — Which AWS regions to search. You can list multiple.

**`filters`** — AWS EC2 filters to narrow down instances. Uses the same filter names as the AWS API:

```yaml
filters:
  instance-state-name: running
  # Only instances with specific tags:
  tag:Environment: prod
  # Only specific instance types:
  instance-type:
    - t3.micro
    - t3.small
  # Only in a specific VPC:
  vpc-id: vpc-0abc123def456
```

**`hostnames`** — How to identify each host. This determines what Ansible uses as the "hostname":

```yaml
hostnames:
  - private-ip-address         # Use private IP (for VPN/VPC access)
  - public-ip-address          # Use public IP (for direct internet access)
  - dns-name                   # Use public DNS name
  - private-dns-name           # Use private DNS name
  - tag:Name                   # Use the "Name" tag value
  - ip-address                 # Alias for public-ip-address
```

**`keyed_groups`** — Automatically create groups based on instance attributes:

```yaml
keyed_groups:
  - key: tags.Environment      # The attribute to group by
    prefix: env                # Group name prefix
    separator: "_"             # Separator between prefix and value
    # Result: env_prod, env_staging, env_dev
```

**`compose`** — Create or override host variables:

```yaml
compose:
  ansible_host: private_ip_address
  ansible_user: "'ubuntu'"
  ansible_ssh_private_key_file: "'/home/user/.ssh/aws-key.pem'"
```

### Step 3: Use It in a Playbook

```yaml
# playbook.yml
---
- name: Configure web servers
  hosts: role_webserver        # ← Uses the dynamic group!
  become: yes

  tasks:
    - name: Install Nginx
      apt:
        name: nginx
        state: present

    - name: Show instance info
      debug:
        msg: "I am {{ inventory_hostname }} in {{ placement.region }}"

- name: Configure databases
  hosts: role_database         # ← Another dynamic group!
  become: yes

  tasks:
    - name: Install PostgreSQL
      apt:
        name: postgresql
        state: present
```

```bash
# Run it!
ansible-playbook -i inventory/aws_ec2.yml playbook.yml
```

### Combining Static and Dynamic Inventory

You can use **both** at the same time! Just put them in the same directory:

```
inventory/
├── aws_ec2.yml          ← Dynamic (AWS instances)
├── static_hosts         ← Static (on-prem servers, bastion hosts, etc.)
└── group_vars/
    ├── role_webserver.yml
    └── role_database.yml
```

```ini
# inventory/static_hosts
[bastion]
bastion.example.com ansible_user=admin

[monitoring]
prometheus.internal.com
grafana.internal.com
```

```bash
# Point Ansible to the entire directory — it reads ALL files!
ansible-playbook -i inventory/ playbook.yml
```

## 🔷 Azure Dynamic Inventory — Complete Guide

### Prerequisites

```bash
# 1. Install the Azure Ansible collection
ansible-galaxy collection install azure.azcollection

# 2. Install Azure SDK dependencies
pip install azure-identity azure-mgmt-compute azure-mgmt-network azure-mgmt-resource

# 3. Configure Azure credentials (any of these methods):

# Method A: Environment variables
export AZURE_SUBSCRIPTION_ID="xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
export AZURE_CLIENT_ID="xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
export AZURE_SECRET="your-client-secret"
export AZURE_TENANT="xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"

# Method B: Azure CLI (easiest for local development)
az login

# Method C: Managed Identity (when running on an Azure VM — best for production!)
# No config needed — auto-detects the managed identity
```

### Create the Azure Inventory Plugin File

The file **must** end with `azure_rm.yml` or `azure_rm.yaml`:

```yaml
# inventory/azure_rm.yml
---
plugin: azure.azcollection.azure_rm

# Include all VMs from these resource groups
include_vm_resource_groups:
  - myapp-rg-prod
  - myapp-rg-staging

# Authentication (can also use environment variables)
auth_source: auto   # auto-detect: env vars → CLI → MSI

# Use private IP instead of public
plain_host_names: true

# Filters — only include running VMs
# exclude_host_filters:
#   - powerstate != 'running'

# How to group VMs
keyed_groups:
  # Group by resource group
  - key: resource_group
    prefix: rg
    separator: "_"

  # Group by tags
  - key: tags.Environment | default('untagged')
    prefix: env
    separator: "_"

  # Group by location/region
  - key: location
    prefix: azure_region
    separator: "_"

  # Group by VM size
  - key: vm_size
    prefix: size
    separator: "_"

  # Group by OS type
  - key: os_disk.operating_system_type
    prefix: os
    separator: "_"

# Additional host variables
hostnames:
  - default_inventory_hostname

compose:
  ansible_host: private_ipv4_addresses[0] | default(public_ip_addresses[0], true)
  ansible_user: "'azureuser'"
```

### Test It

```bash
ansible-inventory -i inventory/azure_rm.yml --graph
```

```
@all:
  |--@env_prod:
  |  |--myapp-web-vm-01
  |  |--myapp-web-vm-02
  |  |--myapp-db-vm-01
  |--@env_staging:
  |  |--myapp-staging-vm-01
  |--@rg_myapp_rg_prod:
  |  |--myapp-web-vm-01
  |  |--myapp-web-vm-02
  |  |--myapp-db-vm-01
  |--@azure_region_eastus:
  |  |--myapp-web-vm-01
  |  |--myapp-web-vm-02

  |  |--myapp-web-vm-01
  |  |--myapp-web-vm-02
  |  |--myapp-db-vm-01
  |  |--myapp-staging-vm-01
```

### Use It in a Playbook

```yaml
# playbook.yml
---
- name: Configure Azure web servers
  hosts: env_prod:&rg_myapp_rg_prod   # Intersection: prod AND in this resource group
  become: yes

  tasks:
    - name: Install Nginx
      apt:
        name: nginx
        state: present
      when: os_disk.operating_system_type == "Linux"
```

## 🟢 GCP Dynamic Inventory — Complete Guide

### Prerequisites

```bash
# 1. Install the GCP Ansible collection
ansible-galaxy collection install google.cloud

# 2. Install GCP SDK
pip install google-auth requests

# 3. Configure GCP credentials:

# Method A: Service account key file
export GCP_SERVICE_ACCOUNT_FILE="/path/to/service-account-key.json"

# Method B: Application Default Credentials
gcloud auth application-default login

# Method C: On a GCE instance — uses the instance's service account automatically
```

### Create the GCP Inventory Plugin File

The file **must** end with `gcp.yml` or `gcp.yaml`:

```yaml
# inventory/gcp.yml
---
plugin: google.cloud.gcp_compute

# Your GCP project(s)
projects:
  - my-gcp-project-id
  - another-project-id

# Which zones to search
zones:
  - us-central1-a
  - us-east1-b
  - europe-west1-b

# Filters (GCP filter syntax)
filters:
  - status = RUNNING

# Authentication
auth_kind: serviceaccount
service_account_file: /path/to/service-account-key.json
# OR use: auth_kind: application (for gcloud auth)

# How to group instances
keyed_groups:
  # Group by labels (GCP's version of tags)
  - key: labels.environment
    prefix: env
    separator: "_"

  # Group by machine type
  - key: machineType | regex_replace('.*/', '')
    prefix: machine
    separator: "_"

  # Group by zone
  - key: zone | regex_replace('.*/', '')
    prefix: zone
    separator: "_"

  # Group by network tags
  - key: tags.items | default([])
    prefix: tag
    separator: "_"

# Host variables
compose:
  ansible_host: networkInterfaces[0].accessConfigs[0].natIP | default(networkInterfaces[0].networkIP)
  ansible_user: "'ubuntu'"

# Use instance name as hostname
hostnames:
  - name
  # OR use private IP:
  # - private_ip
```

## 🔧 Advanced Dynamic Inventory Techniques

### 1. Caching — Speed Up Inventory Queries

Cloud API calls can be slow (especially with hundreds of instances). Enable caching:

```yaml
# inventory/aws_ec2.yml
plugin: amazon.aws.aws_ec2
regions:
  - us-east-1

# Enable caching
cache: true
cache_plugin: ansible.builtin.jsonfile
cache_connection: /tmp/ansible_inventory_cache
cache_timeout: 300   # Cache for 5 minutes (300 seconds)
```

Now Ansible will use the cached inventory for 5 minutes before querying AWS again. Much faster for repeated runs!

```bash
# Force refresh the cache
ansible-inventory -i inventory/aws_ec2.yml --list --flush-cache
```

### 2. Multiple Cloud Providers Together

You can query **multiple clouds** in the same project:

```
inventory/
├── aws_ec2.yml       ← AWS instances
├── azure_rm.yml      ← Azure VMs
├── gcp.yml           ← GCP instances
├── static_hosts      ← On-prem servers
└── group_vars/
    ├── env_prod.yml
    └── env_staging.yml
```

```bash
# Ansible reads ALL inventory sources in the directory
ansible-playbook -i inventory/ playbook.yml
```

Your playbook can now target hosts from **any cloud**:

```yaml
- name: Update all production web servers (regardless of cloud!)
  hosts: env_prod:&role_webserver
  tasks:
    - name: Update packages
      package:
        name: "*"
        state: latest
```

### 3. Custom Grouping with Conditionals

```yaml
# aws_ec2.yml
groups:
  # Group instances with "web" in their Name tag
  webservers: "'web' in (tags.Name | default(''))"

  # Group instances in production VPC
  production: "vpc_id == 'vpc-0abc123'"

  # Group large instances (for special config)
  large_instances: "instance_type.startswith('m5') or instance_type.startswith('c5')"

  # Group by custom logic
  needs_patching: "(tags.Patching | default('no')) == 'yes'"
```

### 4. Using `constructed` Plugin for Advanced Grouping

The `constructed` inventory plugin lets you create groups **after** the dynamic inventory runs:

```yaml
# inventory/constructed.yml
plugin: ansible.builtin.constructed
strict: false
use_vars_plugins: true

groups:
  # Combine cloud groups into a unified group
  all_webservers: >-
    'role_webserver' in group_names or
    'tag_http_server' in group_names

compose:
  # Standardize connection user across clouds
  ansible_user: >-
    'ubuntu' if 'aws_region' in group_names | join(' ')
    else 'azureuser' if 'azure_region' in group_names | join(' ')
    else 'admin'
```

### 5. Writing a Custom Dynamic Inventory Script (Legacy but Useful)

Sometimes you need inventory from a source that doesn't have a plugin (like a custom CMDB or database). You can write a script:

```python
#!/usr/bin/env python3
# inventory/custom_inventory.py
"""
Custom dynamic inventory script.
Must output JSON when called with --list.
Must output host vars when called with --host <hostname>.
"""

import json
import sys
import requests  # or any method to fetch your data

def get_inventory():
    """Query your custom source and return Ansible inventory format."""

    # Example: Fetch from an internal API
    # response = requests.get("https://cmdb.internal.com/api/servers")
    # servers = response.json()

    # For demo, here's a static example of the required JSON format:
    inventory = {
        "webservers": {
            "hosts": ["web1.example.com", "web2.example.com"],
            "vars": {
                "http_port": 80,
                "ansible_user": "deploy"
            }
        },
        "databases": {
            "hosts": ["db1.example.com"],
            "vars": {
                "db_port": 5432
            }
        },
        "_meta": {
            "hostvars": {
                "web1.example.com": {
                    "ansible_host": "10.0.1.10",
                    "server_role": "primary"
                },
                "web2.example.com": {
                    "ansible_host": "10.0.1.11",
                    "server_role": "secondary"
                },
                "db1.example.com": {
                    "ansible_host": "10.0.2.10",
                    "server_role": "primary"
                }
            }
        }
    }
    return inventory

def get_host_vars(hostname):
    """Return variables for a specific host."""
    inventory = get_inventory()
    return inventory.get("_meta", {}).get("hostvars", {}).get(hostname, {})

if __name__ == "__main__":
    if len(sys.argv) == 2 and sys.argv[1] == "--list":
        print(json.dumps(get_inventory(), indent=2))
    elif len(sys.argv) == 3 and sys.argv[1] == "--host":
        print(json.dumps(get_host_vars(sys.argv[2]), indent=2))
    else:
        print(json.dumps({}))
```

```bash
# Make it executable
chmod +x inventory/custom_inventory.py

# Test it
./inventory/custom_inventory.py --list

# Use it with Ansible
ansible-playbook -i inventory/custom_inventory.py playbook.yml
```

## 📋 Complete Working Example — AWS EC2 Dynamic Inventory

Let's build a full example from scratch:

### Project Structure

```
my-project/
├── ansible.cfg
├── inventory/
│   ├── aws_ec2.yml              ← Dynamic inventory config
│   └── group_vars/
│       ├── env_prod.yml          ← Variables for prod group
│       ├── env_staging.yml       ← Variables for staging group
│       └── role_webserver.yml    ← Variables for webserver group
├── playbooks/
│   ├── deploy.yml
│   └── patch.yml
└── requirements.yml
```

### `ansible.cfg`

```ini
[defaults]
inventory = inventory/
remote_user = ec2-user
private_key_file = ~/.ssh/aws-key.pem
host_key_checking = False
```

### `inventory/aws_ec2.yml`

```yaml
---
plugin: amazon.aws.aws_ec2
regions:
  - us-east-1
filters:
  instance-state-name: running
  "tag:ManagedBy": "ansible"
hostnames:
  - private-ip-address
keyed_groups:
  - key: tags.Environment
    prefix: env
    separator: "_"
  - key: tags.Role
    prefix: role
    separator: "_"
compose:
  ansible_host: private_ip_address
  ansible_user: "'ec2-user'"
cache: true
cache_plugin: ansible.builtin.jsonfile
cache_connection: /tmp/aws_inventory_cache
cache_timeout: 300
```

### `inventory/group_vars/env_prod.yml`

```yaml
---
nginx_worker_processes: 4
app_debug: false
log_level: "warn"
```

### `inventory/group_vars/env_staging.yml`

```yaml
---
nginx_worker_processes: 2
app_debug: true
log_level: "debug"
```

### `inventory/group_vars/role_webserver.yml`

```yaml
---
nginx_enabled: true
app_port: 8080
health_check_path: "/health"
```

### `playbooks/deploy.yml`

```yaml
---
- name: Deploy to production web servers
  hosts: env_prod:&role_webserver    # Production AND webserver role
  become: yes
  serial: "25%"                       # Rolling deploy — 25% at a time

  tasks:
    - name: Show discovered host info
      debug:
        msg: >
          Host: {{ inventory_hostname }}
          IP: {{ ansible_host }}
          Instance Type: {{ instance_type }}
          Region: {{ placement.region }}

    - name: Install Nginx
      package:
        name: nginx
        state: present

    - name: Deploy app config
      template:
        src: templates/app.conf.j2
        dest: /etc/myapp/config.yml
      notify: restart app

  handlers:
    - name: restart app
      systemd:
        name: myapp
        state: restarted
```

### Running the Playbook

```bash
# First, verify what inventory sees
ansible-inventory -i inventory/ --graph

# Ping all discovered hosts
ansible -i inventory/ all -m ping

# Run the deploy playbook
ansible-playbook -i inventory/ playbooks/deploy.yml

# Run only against staging
ansible-playbook -i inventory/ playbooks/deploy.yml --limit "env_staging"
```

## ⚠️ Common Mistakes & Gotchas

### Mistake 1: Wrong File Name

```bash
# WRONG — Ansible won't recognize these:
inventory/my_aws.yml         # Missing aws_ec2 in the name
inventory/aws.yml            # Missing _ec2

# RIGHT — Must end with the correct pattern:
inventory/aws_ec2.yml        ✅
inventory/my_aws_ec2.yml     ✅
inventory/prod_aws_ec2.yaml  ✅
```

### Mistake 2: Missing Python Dependencies

```bash
# AWS: "No module named 'boto3'"
pip install boto3 botocore

# Azure: "No module named 'azure'"
pip install azure-identity azure-mgmt-compute azure-mgmt-network azure-mgmt-resource

# GCP: "No module named 'google.auth'"
pip install google-auth requests
```

### Mistake 3: Missing Collection

```bash
# "plugin: amazon.aws.aws_ec2 — plugin not found"
ansible-galaxy collection install amazon.aws

# "plugin: azure.azcollection.azure_rm — plugin not found"
ansible-galaxy collection install azure.azcollection

# "plugin: google.cloud.gcp_compute — plugin not found"
ansible-galaxy collection install google.cloud
```

### Mistake 4: Not Tagging Your Cloud Resources

Dynamic inventory is only useful if your servers are **properly tagged**:

```bash
# Tag your EC2 instances properly!
aws ec2 create-tags --resources i-0abc123 \
  --tags Key=Environment,Value=prod Key=Role,Value=webserver Key=ManagedBy,Value=ansible
```

Without tags, you can't group servers meaningfully. **Tagging is essential!**

### Mistake 5: SSH Key Not Matching

Dynamic inventory finds the hosts, but you still need SSH access:

```yaml
# Make sure compose sets the correct SSH settings
compose:
  ansible_host: private_ip_address
  ansible_user: "'ec2-user'"                              # Match your AMI's default user
  ansible_ssh_private_key_file: "'/home/user/.ssh/key.pem'"  # Match your key pair
```

## 📋 Quick Reference — Cheat Sheet

```bash
# ═══════════════════════════════════════════════
# DYNAMIC INVENTORY — CHEAT SHEET
# ═══════════════════════════════════════════════

# --- Install Collections ---
ansible-galaxy collection install amazon.aws        # AWS
ansible-galaxy collection install azure.azcollection # Azure
ansible-galaxy collection install google.cloud       # GCP

# --- Install Python Dependencies ---
pip install boto3 botocore                           # AWS
pip install azure-identity azure-mgmt-compute azure-mgmt-network azure-mgmt-resource  # Azure
pip install google-auth requests                     # GCP

# --- File Naming ---
# AWS:   *aws_ec2.yml  or  *aws_ec2.yaml
# Azure: *azure_rm.yml or  *azure_rm.yaml
# GCP:   *gcp.yml      or  *gcp.yaml

# --- Test Commands ---
ansible-inventory -i inventory/aws_ec2.yml --list          # Full JSON output
ansible-inventory -i inventory/aws_ec2.yml --graph         # Tree view
ansible-inventory -i inventory/aws_ec2.yml --host <name>   # Single host vars
ansible-inventory -i inventory/ --graph                    # All sources in directory

# --- Flush Cache ---
ansible-inventory -i inventory/aws_ec2.yml --list --flush-cache

# --- Run Playbook ---
ansible-playbook -i inventory/ playbook.yml
ansible-playbook -i inventory/ playbook.yml --limit "env_prod"
ansible-playbook -i inventory/ playbook.yml --limit "env_prod:&role_webserver"

# --- Useful Patterns ---
# hosts: env_prod                    → All prod hosts
# hosts: role_webserver              → All webservers
# hosts: env_prod:&role_webserver    → Prod webservers (intersection)
# hosts: env_prod:!role_database     → Prod, but NOT databases (exclusion)
# hosts: env_prod:env_staging        → Prod OR staging (union)
```

## 🧠 Key Takeaways

1. **Dynamic inventory queries cloud APIs** at runtime — no more stale host lists
2. **Use inventory plugins** (YAML files), not legacy scripts
3. **File naming matters** — must end with `aws_ec2.yml`, `azure_rm.yml`, or `gcp.yml`
4. **`keyed_groups`** automatically creates Ansible groups from cloud metadata (tags, regions, etc.)
5. **Tag your cloud resources** — without tags, dynamic inventory is nearly useless
6. **Enable caching** for faster repeated runs
7. **You can combine** static and dynamic inventory in the same directory
8. **Multi-cloud is possible** — put AWS, Azure, and GCP inventory files in the same directory
