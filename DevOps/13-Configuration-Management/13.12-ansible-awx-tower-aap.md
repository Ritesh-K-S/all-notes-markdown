## 🤔 Why Do We Need a UI for Ansible? A Story First

Picture this: You've been doing amazing work with Ansible. You've got 50 playbooks, dynamic inventory across 3 cloud providers, and everything runs beautifully... from **your laptop's terminal**.

Now your boss asks:

> "Hey, can the junior developer run the deployment playbook too?"

You: *"Sure, they just need to SSH into the control node, make sure they have the right vault password, set up the AWS credentials, know which inventory to use, remember the exact command..."*

Boss: *"Can't they just... click a button?"*

Or worse:

> "The deployment failed last night at 3 AM. Who ran it? What changed? Can we see the logs?"

You: *"Ummm... I think Dave ran it from his laptop? Maybe? The logs are... somewhere on his machine?"*

This is exactly why tools like **AWX, Ansible Tower, and AAP** exist. They give Ansible a proper **web interface, access control, audit logging, and job scheduling** — turning your terminal-only automation into an enterprise-ready platform.

## 🚀 Installing AWX — Step by Step

AWX runs on **Kubernetes** (since AWX 18.0+). The easiest way to install it is using the **AWX Operator**.

### Prerequisites

- A Kubernetes cluster (Minikube, Kind, k3s, or a real cluster)
- `kubectl` configured
- At least **4 GB RAM** and **2 CPUs** available

### Method 1: AWX on Minikube (Great for Learning!)

```bash
# Step 1: Start Minikube with enough resources
minikube start --cpus=4 --memory=6g --addons=ingress

# Step 2: Install the AWX Operator using kubectl
# Create a namespace for AWX
kubectl create namespace awx

# Step 3: Deploy the AWX Operator
# First, install the operator from the official release
kubectl apply -f https://raw.githubusercontent.com/ansible/awx-operator/devel/deploy/awx-operator.yaml -n awx
```

Wait for the operator pod to be running:

```bash
kubectl get pods -n awx
# NAME                                    READY   STATUS    RESTARTS   AGE
# awx-operator-controller-manager-xxx     2/2     Running   0          2m
```

### Step 4: Create an AWX Instance

```yaml
# awx-instance.yml
---
apiVersion: awx.ansible.com/v1beta1
kind: AWX
metadata:
  name: awx-demo
  namespace: awx
spec:
  service_type: NodePort
  # Optional: Set the admin password (if not set, it's auto-generated)
  # admin_password_secret: awx-admin-password
```

```bash
kubectl apply -f awx-instance.yml -n awx
```

### Step 5: Wait for AWX to Come Up

```bash
# Watch the pods (this takes 3-5 minutes)
kubectl get pods -n awx -w

# You should eventually see:
# NAME                       READY   STATUS    RESTARTS   AGE
# awx-demo-postgres-xxx      1/1     Running   0          5m
# awx-demo-task-xxx          4/4     Running   0          4m
# awx-demo-web-xxx           3/3     Running   0          4m
```

### Step 6: Access AWX

```bash
# Get the NodePort
kubectl get svc -n awx
# NAME                TYPE        PORT(S)
# awx-demo-service    NodePort    80:32000/TCP

# Get the URL (Minikube)
minikube service awx-demo-service -n awx --url
# http://192.168.49.2:32000

# Get the admin password (if auto-generated)
kubectl get secret awx-demo-admin-password -n awx -o jsonpath="{.data.password}" | base64 --decode
# Output: something like "J8kF2mPqR9"
```

Open your browser, go to the URL, and log in:
- **Username:** `admin`
- **Password:** *(from the secret above)*

🎉 **You now have AWX running!**

## 🎯 AWX Core Concepts — Detailed Walkthrough

Let's walk through each concept with clear explanations:

### 1. Organizations

Think of organizations as **top-level folders** that separate teams or business units.

```
AWX
├── Organization: "DevOps Team"
│   ├── Users: alice, bob, charlie
│   ├── Projects: infra-playbooks, app-deploy
│   ├── Inventories: aws-prod, aws-staging
│   └── Credentials: aws-keys, ssh-keys
│
├── Organization: "Security Team"
│   ├── Users: dave, eve
│   ├── Projects: security-hardening
│   └── Credentials: security-scanner-api
│
└── Organization: "QA Team"
    ├── Users: frank, grace
    └── Projects: testing-playbooks
```

**Each organization has its own users, projects, inventories, and credentials — isolated from each other.**

---

### 2. Projects

A project is a **link to a Git repository** containing your Ansible playbooks.

**Setting up a project in AWX:**

| Field | Value | Explanation |
|-------|-------|------------|
| Name | `infra-playbooks` | Friendly name |
| Organization | `DevOps Team` | Which org owns it |
| Source Control Type | `Git` | Where the code lives |
| Source Control URL | `https://github.com/myorg/ansible-playbooks.git` | Repo URL |
| Source Control Branch | `main` | Which branch to use |
| Source Control Credential | `github-token` | PAT or SSH key for private repos |
| Update on Launch | ✅ | Pull latest code before each job |

When AWX syncs a project, it does a `git clone` (or `git pull`) of your repo. Your playbooks are then available to use in job templates.

**Why this is awesome:**
- Your playbooks live in Git (version controlled!)
- AWX always runs the latest version
- No need to manually copy playbooks to a server

---

### 3. Inventories

AWX supports both **static** and **dynamic** inventories:

**Static Inventory:**
- Manually add hosts and groups through the UI
- Good for small, stable environments

**Dynamic Inventory (Sourced):**
- Point AWX to a cloud provider (AWS, Azure, GCP)
- AWX runs the inventory plugin and discovers hosts automatically
- Can be scheduled to refresh (e.g., every 5 minutes)

**Setting up a dynamic AWS inventory in AWX:**

| Field | Value |
|-------|-------|
| Name | `aws-production` |
| Organization | `DevOps Team` |
| Source | `Amazon EC2` |
| Credential | `aws-api-keys` |
| Regions | `us-east-1, us-west-2` |
| Instance Filters | `tag:Environment=prod` |
| Update on Launch | ✅ |
| Overwrite | ✅ |

AWX will call the AWS API, discover all running EC2 instances with `Environment=prod` tag, and create the host entries automatically. **Same as the dynamic inventory we learned in 13.11, but with a nice UI!**

---

### 4. Credentials

AWX has a **secure credential store** — no more vault password files on someone's laptop!

**Types of credentials AWX supports:**

| Credential Type | Used For |
|----------------|----------|
| **Machine** | SSH keys / passwords to connect to hosts |
| **Source Control** | Git repo access (GitHub token, SSH key) |
| **Cloud** | AWS keys, Azure service principal, GCP service account |
| **Vault** | Ansible Vault password |
| **Container Registry** | Docker Hub, ECR, ACR credentials |
| **Network** | Network device credentials (Cisco, Juniper) |
| **Custom** | Any custom credential type you define |

**Key security features:**
- Credentials are **encrypted at rest** in the database
- Users can **use** credentials without **seeing** them
- Full **audit trail** of who used what credentials and when
- No secrets stored in Git or on laptops!

```
Traditional Ansible:
  "Hey Dave, what's the vault password?"
  "It's MySecr3t — don't tell anyone!"
  → Secrets shared over Slack/email 😱

AWX/AAP:
  Admin adds credential → Assigns it to a template
  Users click "Launch" → AWX injects the credential at runtime
  → Nobody ever sees the actual password! 🔐
```

---

### 5. Job Templates

A job template is the **central concept** in AWX. It ties everything together:

```
Job Template = Project (playbook) + Inventory (hosts) + Credentials (access)
```

**Setting up a job template:**

| Field | Value | Explanation |
|-------|-------|------------|
| Name | `Deploy Web App to Production` | What this job does |
| Job Type | `Run` | Run or Check (dry-run) |
| Inventory | `aws-production` | Which hosts to target |
| Project | `infra-playbooks` | Which Git repo |
| Playbook | `playbooks/deploy.yml` | Which playbook from the repo |
| Credentials | `ssh-key-prod`, `vault-pass` | Machine + Vault credentials |
| Limit | `role_webserver` | (Optional) Only target webservers |
| Verbosity | `0 (Normal)` | How detailed the output is |
| Extra Variables | `app_version: "2.1.0"` | Runtime variables |
| Enable Privilege Escalation | ✅ | `become: yes` |

**Now anyone with permission can click "Launch" and deploy the app!** No terminal, no SSH, no remembering commands.

### Survey — Making Templates Interactive

Want users to input values before launching? Add a **survey**:

```
Survey Questions:
┌────────────────────────────────────────────┐
│ What version to deploy?                     │
│ [Text field: app_version] Default: "latest" │
│                                             │
│ Which environment?                          │
│ [Dropdown: dev / staging / prod]            │
│                                             │
│ Enable maintenance mode?                    │
│ [Checkbox: yes / no] Default: no            │
└────────────────────────────────────────────┘
```

When a user launches the template, they see a nice form with these questions. Their answers become `extra_vars` passed to the playbook.

### 6. Workflows

Workflows let you chain **multiple job templates** together with conditional logic:

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  Run Tests   │ ──→ │  Deploy App  │ ──→ │  Smoke Test  │
│              │     │              │     │              │
└──────────────┘     └──────────────┘     └──────┬───────┘
                                                  │
                                          ┌───────┴───────┐
                                          │               │
                                     On Success      On Failure
                                          │               │
                                          ▼               ▼
                                   ┌──────────┐    ┌──────────────┐
                                   │  Notify  │    │  Rollback    │
                                   │  Slack ✅│    │  & Alert 🚨  │
                                   └──────────┘    └──────────────┘
```

**Workflow features:**
- **Success/Failure/Always paths** — run different templates based on results
- **Convergence** — wait for multiple parallel jobs to finish before proceeding
- **Approval nodes** — pause and wait for human approval before continuing
- **Inventory/credential override** — different templates can use different inventories

**Example real-world workflow:**

```
1. [Sync Inventory]           → Refresh dynamic inventory from AWS
       ↓ (on success)
2. [Run Compliance Check]     → Check if servers meet security baseline
       ↓ (on success)                ↓ (on failure)
3. [Deploy Application]       → [Send Alert to Security Team]
       ↓ (on success)
4. [Run Integration Tests]
       ↓ (on success)                ↓ (on failure)
5. [Notify Team - Success]    → [Rollback + Notify Team - Failure]
```

### 8. RBAC — Role-Based Access Control

AWX has a powerful permission system:

**Built-in Roles:**

| Role | Can Do |
|------|--------|
| **System Admin** | Everything. Full control over entire AWX. |
| **System Auditor** | View everything but change nothing. |
| **Org Admin** | Full control within their organization. |
| **Org Member** | Basic access within their organization. |
| **Team Admin** | Manage team members and permissions. |
| **Project Admin** | Manage projects (add/edit/delete). |
| **Inventory Admin** | Manage inventories. |
| **Job Template Admin** | Manage job templates. |
| **Execute** | Can launch (run) a job template but not edit it. |
| **Read** | Can view but not run or edit. |

**Real-world RBAC example:**

```
DevOps Team:
├── Alice (Admin)     → Can create/edit templates, manage credentials
├── Bob (Execute)     → Can launch deployments but can't edit playbooks
└── Charlie (Read)    → Can view job history/logs but can't run anything

QA Team:
├── Dave (Execute on "Run Tests" template only)
└── Eve (Read on everything)

Management:
└── Frank (System Auditor) → Can see everything across all orgs for compliance
```

**This means:**
- Junior developers can run deployments without accessing production SSH keys
- QA can trigger test runs without seeing infrastructure details
- Managers can review audit logs without any access to change things

## 🔌 AWX REST API — Automation on Top of Automation

One of AWX's most powerful features is its **REST API**. Every action you can do in the UI, you can do via API.

### Common API Endpoints

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `/api/v2/job_templates/` | GET | List all job templates |
| `/api/v2/job_templates/{id}/launch/` | POST | Launch a job template |
| `/api/v2/jobs/{id}/` | GET | Get job status and details |
| `/api/v2/jobs/{id}/stdout/` | GET | Get job output/logs |
| `/api/v2/inventories/` | GET | List inventories |
| `/api/v2/projects/` | GET | List projects |

### Example: Launch a Job from CI/CD Pipeline

```bash
# Launch a job template (ID=5) with extra variables
curl -X POST \
  https://awx.example.com/api/v2/job_templates/5/launch/ \
  -H "Authorization: Bearer YOUR_API_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "extra_vars": {
      "app_version": "2.1.0",
      "environment": "production"
    }
  }'

# Response:
# {
#   "job": 42,
#   "status": "pending",
#   ...
# }
```

```bash
# Check job status
curl -s \
  https://awx.example.com/api/v2/jobs/42/ \
  -H "Authorization: Bearer YOUR_API_TOKEN" \
  | jq '.status'

# "successful"
```

### Integration with CI/CD Pipelines

**GitHub Actions example — trigger AWX job after code merge:**

```yaml
# .github/workflows/deploy.yml
name: Deploy via AWX
on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Trigger AWX Deployment
        run: |
          # Launch the job template
          RESPONSE=$(curl -s -X POST \
            "${{ secrets.AWX_URL }}/api/v2/job_templates/${{ secrets.AWX_TEMPLATE_ID }}/launch/" \
            -H "Authorization: Bearer ${{ secrets.AWX_TOKEN }}" \
            -H "Content-Type: application/json" \
            -d '{"extra_vars": {"app_version": "${{ github.sha }}"}}')

          JOB_ID=$(echo $RESPONSE | jq -r '.job')
          echo "AWX Job ID: $JOB_ID"

          # Poll for completion
          while true; do
            STATUS=$(curl -s \
              "${{ secrets.AWX_URL }}/api/v2/jobs/$JOB_ID/" \
              -H "Authorization: Bearer ${{ secrets.AWX_TOKEN }}" \
              | jq -r '.status')
            echo "Status: $STATUS"
            if [ "$STATUS" = "successful" ]; then
              echo "Deployment successful!"
              exit 0
            elif [ "$STATUS" = "failed" ] || [ "$STATUS" = "error" ]; then
              echo "Deployment failed!"
              exit 1
            fi
            sleep 10
          done
```

**Jenkins Pipeline example:**

```groovy
// Jenkinsfile
pipeline {
    agent any
    stages {
        stage('Deploy via AWX') {
            steps {
                script {
                    def response = httpRequest(
                        url: "${AWX_URL}/api/v2/job_templates/${AWX_TEMPLATE_ID}/launch/",
                        httpMode: 'POST',
                        customHeaders: [[name: 'Authorization', value: "Bearer ${AWX_TOKEN}"]],
                        contentType: 'APPLICATION_JSON',
                        requestBody: '{"extra_vars": {"app_version": "' + env.BUILD_NUMBER + '"}}'
                    )
                    echo "AWX Job launched: ${response.content}"
                }
            }
        }
    }
}
```

### Execution Environments (EE) — A Game Changer

In the old days, you had "dependency hell":

```
Developer's laptop: Python 3.9, boto3 1.26, ansible 2.12
AWX server: Python 3.8, boto3 1.20, ansible 2.11
CI/CD runner: Python 3.10, boto3 1.28, ansible 2.14
→ "It works on my machine!" 😭
```

**Execution Environments** solve this by packaging everything in a container:

```dockerfile
# execution-environment.yml — Definition file
---
version: 3
dependencies:
  galaxy:
    collections:
      - amazon.aws
      - azure.azcollection
      - google.cloud
  python:
    - boto3>=1.26
    - azure-identity
    - google-auth
  system:
    - openssh-clients
    - sshpass

build_arg_defaults:
  ANSIBLE_GALAXY_CLI_COLLECTION_OPTS: "--force"

images:
  base_image:
    name: "quay.io/ansible/ansible-runner:latest"
```

```bash
# Build the execution environment
ansible-builder build --tag my-custom-ee:latest

# Use it with ansible-navigator
ansible-navigator run playbook.yml --execution-environment-image my-custom-ee:latest

# Push to a registry for AWX/AAP to use
docker push registry.example.com/my-custom-ee:latest
```

Now **everyone** uses the same container image — same Python version, same libraries, same Ansible version. No more "works on my machine"!

### Event-Driven Ansible (EDA) — React and Automate

EDA lets you **automatically trigger** Ansible actions based on events:

```yaml
# rulebook.yml — EDA rulebook
---
- name: Auto-remediate high CPU
  hosts: all
  sources:
    - ansible.eda.alertmanager:
        host: 0.0.0.0
        port: 5000

  rules:
    - name: Restart service on high CPU alert
      condition: event.alert.labels.alertname == "HighCPU"
      action:
        run_job_template:
          name: "Restart Application Service"
          organization: "DevOps Team"
          job_args:
            extra_vars:
              target_host: "{{ event.alert.labels.instance }}"
```

**Flow:**
```
Prometheus detects high CPU → Sends alert to AlertManager
    → AlertManager webhooks to EDA
    → EDA matches the rule
    → EDA triggers AWX job template "Restart Application Service"
    → Service is restarted automatically! 🎉
```

No human intervention needed. The system **heals itself**.

## ⚠️ Common Mistakes & Best Practices

### Mistake 1: Running AWX Without Kubernetes

```
AWX 18+ REQUIRES Kubernetes.
You can't just pip install awx anymore!

Options:
- Minikube (learning)
- K3s (lightweight production)
- Kind (CI/CD testing)
- EKS/AKS/GKE (enterprise production)
```

### Mistake 2: Not Using Projects (Git)

```
❌ Bad: Manually uploading playbooks to AWX
✅ Good: Link AWX to a Git repo → Playbooks are always in sync
```

### Mistake 3: Sharing Credentials

```
❌ Bad: Giving everyone the SSH key or vault password
✅ Good: Store credentials in AWX → Assign "Use" permission to teams
         Users can launch jobs without ever seeing the actual secrets
```

### Mistake 4: Not Using Workflows

```
❌ Bad: One giant playbook that does everything
✅ Good: Smaller, focused templates chained together in a workflow
         Each step can have different inventory, credentials, and failure handling
```

### Best Practices Checklist

- ✅ Use **Git for all playbooks** — never write playbooks directly in AWX
- ✅ Use **dynamic inventory** over static whenever possible
- ✅ Store **all credentials in AWX** — remove them from laptops and servers
- ✅ Set up **RBAC from day one** — don't let everyone be admin
- ✅ Enable **notifications** for failed jobs (Slack, email, PagerDuty)
- ✅ Use **workflows** for multi-step processes
- ✅ Use **surveys** to let non-technical users provide input safely
- ✅ **Audit regularly** — review job history and access logs
- ✅ Use **Execution Environments** for consistent runtime
- ✅ **Back up AWX** regularly (especially the PostgreSQL database)

## 🧠 Key Takeaways

1. **AWX = free, open-source UI for Ansible** — perfect for learning and small teams
2. **AAP = enterprise version** — adds Automation Hub, EDA, Execution Environments, and Red Hat support

5. **REST API** enables CI/CD integration — trigger Ansible from GitHub Actions, Jenkins, etc.

7. **Execution Environments** eliminate "works on my machine" by containerizing Ansible + dependencies
8. **Event-Driven Ansible (EDA)** enables auto-remediation — systems heal themselves
9. **AWX requires Kubernetes** since version 18 — use Minikube or K3s for learning
10. **Store everything in Git + AWX** — playbooks in Git, credentials in AWX, no secrets on laptops
