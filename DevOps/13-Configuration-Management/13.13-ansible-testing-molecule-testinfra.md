## 🤔 Why Test Ansible Code?

You might be thinking: *"I just ran the playbook and it worked. Isn't that enough?"*

**Nope. Not even close.** Here's why:

```
What "running the playbook" tells you:
    ✅ The YAML syntax is correct
    ✅ The tasks executed without errors (this time)
    ✅ Ansible didn't crash

What "running the playbook" DOESN'T tell you:
    ❌ Will it work on a FRESH server? (not just YOUR server)
    ❌ Is the role idempotent? (running it twice = same result?)
    ❌ Does it work on Ubuntu AND CentOS?
    ❌ Did the service ACTUALLY start and stay running?
    ❌ Is the config file ACTUALLY correct (not just placed)?
    ❌ Can the application ACTUALLY serve traffic?
    ❌ Will your changes break OTHER roles?
    ❌ Does it work with different variable combinations?
```

Without proper testing, you're basically saying: *"It works on my machine!"* — and we all know how that ends. 😅

### The Testing Pyramid for Ansible

```
                    ╱╲
                   ╱  ╲
                  ╱    ╲
                 ╱ Inte-╲
                ╱ gration╲        ← Molecule + Testinfra
               ╱  Tests   ╲         (Full role testing)
              ╱─────────────╲
             ╱   Functional   ╲    ← Molecule verify
            ╱     Tests        ╲     (Does it actually work?)
           ╱────────────────────╲
          ╱    Syntax & Lint      ╲  ← ansible-lint, yamllint
         ╱      (Static Tests)     ╲   (Catch errors early)
        ╱───────────────────────────╲
```

Let's start from the bottom and work our way up.

## 🧹 Part 1: Static Testing (The Quick Checks)

Before we even get to Molecule, there are quick tests you should ALWAYS run.

### 1.1 — YAML Lint

YAML is picky. One wrong indent and everything breaks.

```bash
# Install yamllint
pip install yamllint

# Lint your playbook
yamllint playbook.yml

# Lint all YAML files in a directory
yamllint roles/
```

**Example yamllint configuration** (`.yamllint.yml`):

```yaml
---
extends: default

rules:
  line-length:
    max: 160          # Allow longer lines (default is 80)
    level: warning     # Don't fail, just warn
  truthy:
    allowed-values: ['true', 'false', 'yes', 'no']
  comments:
    require-starting-space: true
    min-spaces-from-content: 1
  indentation:
    spaces: 2          # Ansible standard is 2 spaces
```

### 1.2 — Ansible Lint

`ansible-lint` is like a senior engineer reviewing your playbooks. It checks for best practices, not just syntax.

```bash
# Install
pip install ansible-lint

# Lint a playbook
ansible-lint playbook.yml

# Lint a role
ansible-lint roles/my_role/
```

**What does it catch?**

```
[risky-file-permissions]    → You forgot to set file permissions
[no-changed-when]           → A command/shell task without changed_when
[yaml[truthy]]              → Using "yes" instead of "true"
[name[missing]]             → Task without a name
[package-latest]            → Using state: latest (unpredictable)
[command-instead-of-module] → Using 'command: apt install' instead of apt module
[no-jinja-when]             → Using Jinja2 {{ }} inside when (not needed)
```

**Example — Bad vs Good:**

```yaml
# ❌ BAD — ansible-lint will flag all of these
- shell: systemctl restart nginx              # no 'name', should use 'service' module
- apt: name=nginx state=latest                # 'latest' is risky
- file: path=/etc/foo                         # no permissions set
- command: cat /etc/hostname                  # 'command' without changed_when

# ✅ GOOD — clean, best-practice code
- name: Restart Nginx
  ansible.builtin.service:
    name: nginx
    state: restarted

- name: Install Nginx
  ansible.builtin.apt:
    name: nginx
    state: present                             # 'present' not 'latest'

- name: Create config directory
  ansible.builtin.file:
    path: /etc/foo
    state: directory
    mode: "0755"

- name: Get hostname
  ansible.builtin.command: cat /etc/hostname
  changed_when: false                          # Tells Ansible this doesn't change anything
```

### 1.3 — Ansible Syntax Check

Built right into Ansible:

```bash
# Quick syntax check
ansible-playbook playbook.yml --syntax-check

# Dry run (check mode) — shows what WOULD change
ansible-playbook playbook.yml --check

# Dry run + diff — shows what files WOULD change and how
ansible-playbook playbook.yml --check --diff
```

## 🧪 Part 2: Molecule — The Complete Testing Framework

### What is Molecule?

**Molecule** is the official testing framework for Ansible roles. Think of it as a robot that:

1. **Creates** a fresh test environment (Docker container, VM, cloud instance)
2. **Runs** your Ansible role inside it
3. **Verifies** everything worked correctly
4. **Destroys** the environment when done

It's like having a QA team that tests your roles automatically, every single time.

```
┌──────────────────────────────────────────────────────────┐
│                  Molecule Workflow                        │
│                                                          │
│  1. CREATE     → Spin up fresh container/VM              │
│       ↓                                                  │
│  2. CONVERGE   → Run your Ansible role                   │
│       ↓                                                  │
│  3. IDEMPOTENCE→ Run the role AGAIN (should = no changes)│
│       ↓                                                  │
│  4. VERIFY     → Run tests (Testinfra/Ansible)           │
│       ↓                                                  │
│  5. DESTROY    → Tear down the environment               │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### Installation

```bash
# Install Molecule with Docker support
pip install molecule molecule-plugins[docker]

# Verify installation
molecule --version
```

**What gets installed:**
- `molecule` — the core framework
- `molecule-plugins[docker]` — the Docker driver (so Molecule can create Docker containers)

> 💡 **Other drivers:** You can also use `vagrant`, `ec2`, `azure`, `gce`, `podman`, etc. Docker is the most popular because it's fast and lightweight.

### Initializing Molecule in an Existing Role

Let's say you have a role called `nginx`:

```
roles/
└── nginx/
    ├── tasks/
    │   └── main.yml
    ├── handlers/
    │   └── main.yml
    ├── templates/
    │   └── nginx.conf.j2
    ├── defaults/
    │   └── main.yml
    └── meta/
        └── main.yml
```

**Add Molecule to it:**

```bash
cd roles/nginx
molecule init scenario --driver-name docker
```

This creates a `molecule/` directory inside your role:

```
roles/
└── nginx/
    ├── tasks/
    ├── handlers/
    ├── templates/
    ├── defaults/
    ├── meta/
    └── molecule/
        └── default/                  ← "default" scenario
            ├── molecule.yml          ← Main config file
            ├── converge.yml          ← Playbook to run your role
            ├── verify.yml            ← Verification playbook
            └── prepare.yml           ← (optional) Pre-setup tasks
```

### Creating a Brand New Role with Molecule

```bash
# Create a new role with Molecule built-in from the start
molecule init role my_namespace.my_role --driver-name docker
```

## ⚙️ Part 3: Molecule Configuration Deep Dive

### The molecule.yml File

This is the **brain** of Molecule. It tells Molecule what to create, how to test, and what to verify.

```yaml
---
# molecule/default/molecule.yml

dependency:
  name: galaxy                        # Install role dependencies from Ansible Galaxy

driver:
  name: docker                        # Use Docker to create test environments

platforms:
  - name: ubuntu-test                 # Name of the test container
    image: ubuntu:22.04               # Docker image to use
    pre_build_image: true             # Use image as-is (don't build a new one)
    command: /sbin/init               # Keep container running (for systemd)
    tmpfs:
      - /run
      - /tmp
    volumes:
      - /sys/fs/cgroup:/sys/fs/cgroup:rw
    privileged: true                  # Needed for systemd inside Docker
    
  - name: centos-test                 # Second test platform!
    image: centos:stream9
    pre_build_image: true
    command: /usr/sbin/init
    privileged: true

provisioner:
  name: ansible                       # Use Ansible as the provisioner
  config_options:
    defaults:
      callbacks_enabled: profile_tasks  # Show task timing
  playbooks:
    converge: converge.yml            # The playbook that applies your role
    verify: verify.yml                # The playbook that verifies results
  inventory:
    host_vars:
      ubuntu-test:
        nginx_port: 8080              # Different variables per platform
      centos-test:
        nginx_port: 80

verifier:
  name: ansible                       # Use Ansible for verification (default)
  # name: testinfra                   # Or use Testinfra (Python-based tests)

scenario:
  name: default
  test_sequence:                      # Customize the test steps
    - dependency
    - lint
    - cleanup
    - destroy
    - syntax
    - create
    - prepare
    - converge
    - idempotence
    - verify
    - cleanup
    - destroy
```

**Let's break this down:**

| Section | What It Does |
|---------|-------------|
| `dependency` | Installs any required Ansible Galaxy roles before testing |
| `driver` | Tells Molecule HOW to create test instances (Docker, Vagrant, etc.) |
| `platforms` | WHAT to create — which OS images, how many instances |
| `provisioner` | HOW to run Ansible — which playbooks, variables, settings |
| `verifier` | HOW to verify — Ansible playbook or Testinfra tests |
| `scenario` | The order of test steps |

### The converge.yml File

This is the playbook that actually runs your role:

```yaml
---
# molecule/default/converge.yml

- name: Converge
  hosts: all
  become: true                        # Run as root (sudo)
  
  vars:
    nginx_port: 80                    # Default variable (can be overridden per platform)

  roles:
    - role: "{{ lookup('env', 'MOLECULE_PROJECT_DIRECTORY') | basename }}"
```

**What's happening?**
- `hosts: all` → Run on all test instances Molecule created
- `become: true` → Use sudo (most roles need root)
- The `role` line dynamically picks up the current role being tested

### The prepare.yml File (Optional)

Sometimes your role expects certain things to already be installed. `prepare.yml` sets those up:

```yaml
---
# molecule/default/prepare.yml

- name: Prepare
  hosts: all
  become: true
  
  tasks:
    - name: Install Python (required for Ansible on some images)
      ansible.builtin.raw: apt-get update && apt-get install -y python3
      changed_when: false

    - name: Install required packages
      ansible.builtin.package:
        name:
          - curl
          - wget
        state: present
```

## 🏃 Part 4: Running Molecule

### The Key Molecule Commands

```bash
# Run the FULL test cycle (create → converge → verify → destroy)
molecule test

# Just create the test instances
molecule create

# Run your role (converge) — doesn't destroy after
molecule converge

# Log into a test container (super useful for debugging!)
molecule login --host ubuntu-test

# Run verification tests only
molecule verify

# Check idempotence (run role twice, second should have 0 changes)
molecule idempotence

# Destroy test instances
molecule destroy

# List all instances and their status
molecule list

# Run everything in debug mode
molecule --debug test
```

### The Full Test Sequence Explained

When you run `molecule test`, here's what happens step by step:

```
$ molecule test

Step 1: DEPENDENCY
    → Downloads role dependencies from Ansible Galaxy
    → Like running: ansible-galaxy install -r requirements.yml

Step 2: LINT (if configured)
    → Runs yamllint + ansible-lint on your code
    → Catches syntax errors and bad practices

Step 3: CLEANUP
    → Runs cleanup.yml if it exists
    → Cleans up any leftover resources from previous runs

Step 4: DESTROY
    → Destroys any existing test instances
    → Ensures a clean starting point

Step 5: SYNTAX
    → Runs ansible-playbook --syntax-check
    → Quick syntax validation

Step 6: CREATE
    → Creates fresh test instances (Docker containers, VMs, etc.)
    → You now have clean, empty servers to test on

Step 7: PREPARE
    → Runs prepare.yml (if it exists)
    → Installs prerequisites your role needs

Step 8: CONVERGE
    → Runs converge.yml — YOUR ROLE gets applied here!
    → This is the main event

Step 9: IDEMPOTENCE
    → Runs converge.yml AGAIN
    → If ANY task shows "changed" → TEST FAILS
    → This proves your role is idempotent (safe to run multiple times)

Step 10: VERIFY
    → Runs your verification tests
    → Checks that everything is actually working

Step 11: CLEANUP
    → Runs cleanup.yml if it exists

Step 12: DESTROY
    → Tears down all test instances
    → Clean up after yourself!
```

### Practical Example: Testing an Nginx Role

Let's walk through a complete example. Here's our Nginx role:

**Role structure:**
```
roles/nginx/
├── defaults/
│   └── main.yml
├── handlers/
│   └── main.yml
├── tasks/
│   └── main.yml
├── templates/
│   └── nginx.conf.j2
└── molecule/
    └── default/
        ├── molecule.yml
        ├── converge.yml
        └── verify.yml
```

**defaults/main.yml:**
```yaml
---
nginx_port: 80
nginx_server_name: localhost
nginx_worker_processes: auto
```

**tasks/main.yml:**
```yaml
---
- name: Install Nginx
  ansible.builtin.apt:
    name: nginx
    state: present
    update_cache: true

- name: Deploy Nginx configuration
  ansible.builtin.template:
    src: nginx.conf.j2
    dest: /etc/nginx/nginx.conf
    mode: "0644"
  notify: Restart Nginx

- name: Ensure Nginx is started and enabled
  ansible.builtin.service:
    name: nginx
    state: started
    enabled: true
```

**handlers/main.yml:**
```yaml
---
- name: Restart Nginx
  ansible.builtin.service:
    name: nginx
    state: restarted
```

**templates/nginx.conf.j2:**
```nginx
worker_processes {{ nginx_worker_processes }};

events {
    worker_connections 1024;
}

http {
    server {
        listen {{ nginx_port }};
        server_name {{ nginx_server_name }};

        location / {
            root /usr/share/nginx/html;
            index index.html;
        }
    }
}
```

**molecule/default/molecule.yml:**
```yaml
---
dependency:
  name: galaxy

driver:
  name: docker

platforms:
  - name: nginx-ubuntu
    image: geerlingguy/docker-ubuntu2204-ansible
    pre_build_image: true
    command: ""
    tmpfs:
      - /run
      - /tmp
    volumes:
      - /sys/fs/cgroup:/sys/fs/cgroup:rw
    privileged: true

provisioner:
  name: ansible

verifier:
  name: ansible
```

> 💡 **Pro tip:** We're using `geerlingguy/docker-ubuntu2204-ansible` instead of plain `ubuntu:22.04`. Jeff Geerlingy (a.k.a. the Ansible guru) maintains Docker images that have systemd and Ansible pre-installed — perfect for testing!

**molecule/default/converge.yml:**
```yaml
---
- name: Converge
  hosts: all
  become: true

  roles:
    - role: nginx
```

**molecule/default/verify.yml:**
```yaml
---
- name: Verify
  hosts: all
  become: true

  tasks:
    - name: Check that Nginx package is installed
      ansible.builtin.package:
        name: nginx
        state: present
      check_mode: true
      register: nginx_installed
      failed_when: nginx_installed.changed

    - name: Check that Nginx service is running
      ansible.builtin.service:
        name: nginx
        state: started
      check_mode: true
      register: nginx_running
      failed_when: nginx_running.changed

    - name: Check that Nginx is listening on port 80
      ansible.builtin.wait_for:
        port: 80
        timeout: 5

    - name: Check Nginx config file exists
      ansible.builtin.stat:
        path: /etc/nginx/nginx.conf
      register: nginx_conf
      failed_when: not nginx_conf.stat.exists

    - name: Verify Nginx responds to HTTP request
      ansible.builtin.uri:
        url: http://localhost:80
        status_code: 200
      register: result
      until: result.status == 200
      retries: 3
      delay: 2
```

**Now run the test:**

```bash
cd roles/nginx
molecule test
```

## 🐍 Part 5: Testinfra — Python-Based Testing

### What is Testinfra?

**Testinfra** is a Python library that lets you write tests for your infrastructure using **pytest**. Instead of writing verification in YAML (Ansible), you write it in Python — which gives you more power and flexibility.

Think of it this way:
- **Ansible verify** = Simple checks in YAML (good for basic stuff)
- **Testinfra** = Powerful Python tests (good for complex validation)

### Installation

```bash
pip install pytest-testinfra
```

### Setting Up Molecule with Testinfra

Update your `molecule.yml` to use Testinfra as the verifier:

```yaml
---
# molecule/default/molecule.yml

dependency:
  name: galaxy

driver:
  name: docker

platforms:
  - name: nginx-ubuntu
    image: geerlingguy/docker-ubuntu2204-ansible
    pre_build_image: true
    command: ""
    privileged: true

provisioner:
  name: ansible

verifier:
  name: testinfra                    # ← Changed from 'ansible' to 'testinfra'
```

### Writing Testinfra Tests

Create a test file at `molecule/default/tests/test_default.py`:

```python
"""Tests for the Nginx role."""
import pytest


def test_nginx_is_installed(host):
    """Check that Nginx package is installed."""
    nginx = host.package("nginx")
    assert nginx.is_installed


def test_nginx_service_is_running(host):
    """Check that Nginx service is running."""
    nginx_service = host.service("nginx")
    assert nginx_service.is_running
    assert nginx_service.is_enabled


def test_nginx_config_file_exists(host):
    """Check that Nginx config file exists with correct permissions."""
    config = host.file("/etc/nginx/nginx.conf")
    assert config.exists
    assert config.is_file
    assert config.user == "root"
    assert config.group == "root"
    assert oct(config.mode) == "0o644"


def test_nginx_is_listening(host):
    """Check that Nginx is listening on port 80."""
    socket = host.socket("tcp://0.0.0.0:80")
    assert socket.is_listening


def test_nginx_config_contains_correct_port(host):
    """Check that the Nginx config has the right port."""
    config = host.file("/etc/nginx/nginx.conf")
    assert config.contains("listen 80")


def test_nginx_responds_to_http_request(host):
    """Check that Nginx actually serves a page."""
    cmd = host.run("curl -s -o /dev/null -w '%{http_code}' http://localhost:80")
    assert cmd.stdout == "200"


def test_nginx_html_directory_exists(host):
    """Check that the default HTML directory exists."""
    html_dir = host.file("/usr/share/nginx/html")
    assert html_dir.exists
    assert html_dir.is_directory
```

### Understanding Testinfra's `host` Fixture

The `host` parameter is **magical**. It's automatically injected by Molecule and gives you access to the test container. Here's what you can do with it:

```python
# ─── Packages ───
host.package("nginx")                     # Check if a package is installed
host.package("nginx").version             # Get package version

# ─── Services ───
host.service("nginx")                     # Check service status
host.service("nginx").is_running          # Is it running?
host.service("nginx").is_enabled          # Will it start on boot?

# ─── Files ───
host.file("/etc/nginx/nginx.conf")        # File operations
host.file("/etc/nginx").is_directory      # Is it a directory?
host.file("/etc/nginx/nginx.conf").content_string  # Read file content
host.file("/etc/nginx/nginx.conf").contains("listen 80")  # Search in file
host.file("/etc/nginx/nginx.conf").mode   # File permissions (as int)
host.file("/etc/nginx/nginx.conf").user   # File owner
host.file("/etc/nginx/nginx.conf").group  # File group

# ─── Sockets ───
host.socket("tcp://0.0.0.0:80")           # Check listening ports
host.socket("tcp://:::443")               # IPv6 socket
host.socket("unix:///var/run/nginx.sock") # Unix socket

# ─── Commands ───
result = host.run("nginx -t")             # Run a command
result.stdout                             # Command output
result.stderr                             # Command errors
result.rc                                 # Return code (0 = success)

# ─── Users & Groups ───
host.user("www-data")                     # User info
host.user("www-data").exists              # Does user exist?
host.user("www-data").groups              # User's groups
host.group("www-data").exists             # Does group exist?

# ─── System Info ───
host.system_info.type                     # "linux"
host.system_info.distribution             # "ubuntu"
host.system_info.release                  # "22.04"

# ─── Processes ───
host.process.filter(comm="nginx")         # Find running processes

# ─── Iptables ───
host.iptables                             # Check firewall rules

# ─── Environment Variables ───
host.environment()                        # Get all env vars
```

### Advanced Testinfra Patterns

#### Using Pytest Parametrize (Test Multiple Values)

```python
import pytest


@pytest.mark.parametrize("package_name", [
    "nginx",
    "curl",
    "vim",
])
def test_packages_are_installed(host, package_name):
    """Check that all required packages are installed."""
    pkg = host.package(package_name)
    assert pkg.is_installed


@pytest.mark.parametrize("port", [80, 443])
def test_ports_are_listening(host, port):
    """Check that all required ports are listening."""
    socket = host.socket(f"tcp://0.0.0.0:{port}")
    assert socket.is_listening


@pytest.mark.parametrize("config_file,expected_content", [
    ("/etc/nginx/nginx.conf", "worker_processes"),
    ("/etc/nginx/nginx.conf", "listen 80"),
])
def test_config_contains_expected_content(host, config_file, expected_content):
    """Check that config files contain expected content."""
    f = host.file(config_file)
    assert f.contains(expected_content)
```

#### Testing Across Multiple Platforms

When you have multiple platforms in `molecule.yml`, Testinfra runs on ALL of them:

```python
def test_nginx_is_installed(host):
    """Works on both Ubuntu and CentOS!"""
    nginx = host.package("nginx")
    assert nginx.is_installed


def test_correct_package_manager_was_used(host):
    """Platform-specific test."""
    distro = host.system_info.distribution
    
    if distro == "ubuntu":
        # Check apt was used
        result = host.run("dpkg -l nginx")
        assert result.rc == 0
    elif distro in ("centos", "redhat"):
        # Check yum/dnf was used
        result = host.run("rpm -q nginx")
        assert result.rc == 0
```

## 🎭 Part 6: Multiple Scenarios

Sometimes you want to test your role in different configurations. That's what **scenarios** are for.

### Creating Multiple Scenarios

```bash
# Default scenario (already exists)
molecule init scenario --driver-name docker

# Create a scenario for testing with custom variables
molecule init scenario --scenario-name custom-port --driver-name docker

# Create a scenario for testing on multiple OSes
molecule init scenario --scenario-name multi-os --driver-name docker
```

Your role structure now looks like:

```
roles/nginx/
├── tasks/
├── defaults/
├── molecule/
│   ├── default/              ← Default scenario
│   │   ├── molecule.yml
│   │   ├── converge.yml
│   │   └── verify.yml
│   ├── custom-port/          ← Custom port scenario
│   │   ├── molecule.yml
│   │   ├── converge.yml
│   │   └── verify.yml
│   └── multi-os/             ← Multi-OS scenario
│       ├── molecule.yml
│       ├── converge.yml
│       └── verify.yml
```

### Custom Port Scenario Example

**molecule/custom-port/converge.yml:**
```yaml
---
- name: Converge with custom port
  hosts: all
  become: true
  
  vars:
    nginx_port: 8080              # ← Test with a non-default port!

  roles:
    - role: nginx
```

**molecule/custom-port/verify.yml:**
```yaml
---
- name: Verify custom port
  hosts: all
  become: true

  tasks:
    - name: Check Nginx is listening on port 8080
      ansible.builtin.wait_for:
        port: 8080
        timeout: 5

    - name: Verify config has port 8080
      ansible.builtin.lineinfile:
        path: /etc/nginx/nginx.conf
        line: "        listen 8080;"
      check_mode: true
      register: port_check
      failed_when: port_check.changed
```

```yaml
driver:
```

### Running Specific Scenarios


## 🔗 Part 7: Molecule in CI/CD

The real power of Molecule is running it **automatically** in your CI/CD pipeline.

### GitHub Actions Example

```yaml
# .github/workflows/molecule.yml

name: Ansible Role Testing

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  molecule:
    runs-on: ubuntu-latest
    
    strategy:
      matrix:
        scenario:
          - default
          - custom-port
          - multi-os
    
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'

      - name: Install dependencies
        run: |
          python -m pip install --upgrade pip
          pip install ansible molecule molecule-plugins[docker] pytest-testinfra

      - name: Run Molecule tests
        run: |
          cd roles/nginx
          molecule test --scenario-name ${{ matrix.scenario }}
        env:
          PY_COLORS: '1'              # Colored output
          ANSIBLE_FORCE_COLOR: '1'    # Colored Ansible output
```

### GitLab CI Example

```yaml
# .gitlab-ci.yml

stages:
  - test

ansible-molecule:
  stage: test
  image: python:3.11
  services:
    - docker:dind
  variables:
    DOCKER_HOST: tcp://docker:2375
  before_script:
    - pip install ansible molecule molecule-plugins[docker] pytest-testinfra
  script:
    - cd roles/nginx
    - molecule test
  parallel:
    matrix:
      - MOLECULE_SCENARIO:
          - default
          - custom-port
```

## 🛠️ Part 8: Debugging Molecule Tests

### When Things Go Wrong

```bash
# Don't destroy on failure — keep the container for debugging
molecule test --destroy=never

# Log into the container to investigate
molecule login --host nginx-ubuntu

# Inside the container, check things manually
systemctl status nginx
cat /etc/nginx/nginx.conf
nginx -t
curl localhost:80

# When done, destroy manually
molecule destroy
```

### Common Errors and Fixes

**1. "systemd not available" errors:**
```yaml
# Fix: Use systemd-enabled images and privileged mode
platforms:
  - name: test
    image: geerlingguy/docker-ubuntu2204-ansible  # ← systemd-enabled image
    command: ""
    privileged: true                                # ← required for systemd
```

**2. "Package not found" errors:**
```yaml
# Fix: Add prepare.yml to update package cache
# molecule/default/prepare.yml
- name: Prepare
  hosts: all
  become: true
  tasks:
    - name: Update apt cache
      ansible.builtin.apt:
        update_cache: true
      when: ansible_os_family == "Debian"
```

**3. Idempotence failures:**
```
# The second run shows "changed" tasks — your role isn't idempotent!
# Common causes:
#   - Using 'command' or 'shell' without changed_when
#   - Using 'state: latest' instead of 'state: present'
#   - Templates that include timestamps or random values
#   - Handlers that always fire
```

**4. Testinfra connection errors:**
```bash
# Fix: Make sure the Docker driver is installed
pip install molecule-plugins[docker]

# Verify containers are running
molecule list
```

## 📝 Part 9: Best Practices for Ansible Testing

### The Testing Checklist

```
✅ Every role should have at LEAST a default Molecule scenario
✅ Run ansible-lint BEFORE molecule test (fail fast)
✅ Always test idempotence (it's free with Molecule!)
✅ Test on multiple OS platforms if your role supports them
✅ Use Testinfra for complex verification, Ansible verify for simple checks
✅ Run Molecule in CI/CD — don't rely on "I tested it locally"
✅ Use --destroy=never when debugging failures
✅ Keep tests simple — test BEHAVIOR, not implementation
✅ Use meaningful test names (test_nginx_serves_homepage, not test_1)
✅ Test default variables AND custom variable combinations
```

### What to Test vs What NOT to Test

```
DO TEST:
    ✅ Packages are installed
    ✅ Services are running and enabled
    ✅ Config files exist with correct content and permissions
    ✅ Ports are listening
    ✅ HTTP endpoints respond correctly
    ✅ Users/groups exist
    ✅ Firewall rules are applied
    ✅ Cron jobs are created
    ✅ The application actually WORKS

DON'T TEST:
    ❌ Ansible itself (trust that apt module installs packages)
    ❌ The Docker image (test YOUR code, not others')
    ❌ Internal implementation details (test results, not steps)
    ❌ Things unrelated to your role
```

### Test File Organization

```
molecule/
├── default/
│   ├── molecule.yml
│   ├── converge.yml
│   ├── verify.yml
│   ├── prepare.yml
│   └── tests/
│       ├── test_default.py         # Main Testinfra tests
│       ├── test_config.py          # Config file tests
│       ├── test_service.py         # Service tests
│       └── conftest.py             # Shared fixtures
├── custom-config/
│   ├── molecule.yml
│   ├── converge.yml
│   └── verify.yml
└── multi-os/
    ├── molecule.yml
    ├── converge.yml
    └── verify.yml
```

## 🎯 Quick Reference Cheat Sheet

### Molecule Commands

| Command | What It Does |
|---------|-------------|
| `molecule init scenario -d docker` | Initialize Molecule in a role |
| `molecule test` | Full test cycle (create → converge → verify → destroy) |
| `molecule converge` | Just run the role (no destroy) |
| `molecule verify` | Just run verification tests |
| `molecule idempotence` | Check idempotence only |
| `molecule login -h <host>` | SSH/exec into a test container |
| `molecule destroy` | Destroy test instances |
| `molecule list` | Show status of test instances |
| `molecule test -s <scenario>` | Run a specific scenario |
| `molecule test --all` | Run ALL scenarios |
| `molecule test --destroy=never` | Don't destroy on failure (for debugging) |

### Testinfra Quick Reference

| What to Test | Testinfra Code |
|-------------|---------------|
| Package installed | `host.package("nginx").is_installed` |
| Service running | `host.service("nginx").is_running` |
| Service enabled | `host.service("nginx").is_enabled` |
| File exists | `host.file("/etc/nginx.conf").exists` |
| File content | `host.file("/etc/nginx.conf").contains("listen 80")` |
| File permissions | `oct(host.file("/etc/nginx.conf").mode) == "0o644"` |
| Port listening | `host.socket("tcp://0.0.0.0:80").is_listening` |
| Run command | `host.run("nginx -t").rc == 0` |
| User exists | `host.user("www-data").exists` |
| Group exists | `host.group("www-data").exists` |

## 🧠 Summary

```
┌──────────────────────────────────────────────────────────────────┐
│                  Ansible Testing Stack                           │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  yamllint        → Check YAML syntax                             │
│  ansible-lint    → Check Ansible best practices                  │
│  --syntax-check  → Check playbook syntax                         │
│  --check --diff  → Dry run — see what would change               │
│                                                                  │
│  Molecule        → FULL testing framework                        │
│    ├── create    → Spin up test environments                     │
│    ├── converge  → Run your role                                 │
│    ├── idempotence → Run again (should = 0 changes)              │
│    ├── verify    → Check results (Ansible or Testinfra)          │
│    └── destroy   → Clean up                                      │
│                                                                  │
│  Testinfra       → Python-based infrastructure tests             │
│    → More powerful than Ansible verify                            │
│    → Uses pytest (parametrize, fixtures, etc.)                   │
│    → Tests packages, services, files, ports, commands            │
│                                                                  │
│  CI/CD           → Run Molecule in GitHub Actions / GitLab CI    │
│    → Automated testing on every push/PR                          │
│    → Test multiple scenarios in parallel                         │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

**Bottom line:** If you're writing Ansible roles for production, **Molecule + Testinfra is non-negotiable**. It takes a bit of setup, but it saves you from the *"it works on my machine"* nightmare. Set it up once, run it forever.
