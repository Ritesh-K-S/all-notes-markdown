# 13.14 — Ansible — Advanced Patterns — Delegation, Serial, Strategies

> **One Line Answer:** Ansible advanced patterns let you control **where** tasks run (delegation), **how many** hosts get updated at once (serial), and **in what order** tasks execute (strategies) — giving you fine-grained control over complex deployments and zero-downtime updates.

---

## 🤔 Why Do You Need Advanced Patterns?

Imagine this: you have 100 web servers behind a load balancer. You want to:

1. Remove a server from the load balancer
2. Update it
3. Test it
4. Add it back to the load balancer
5. Move to the next server

If Ansible just ran everything on all 100 servers at once... your entire website goes down during the update. 💀

**That's where these advanced patterns come in:**

```
┌──────────────────────────────────────────────────────────────┐
│              Advanced Ansible Patterns                       │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  DELEGATION   → "Run this task on a DIFFERENT host"          │
│                 (e.g., update DNS from the web server play)  │
│                                                              │
│  SERIAL       → "Update hosts X at a time, not all at once"  │
│                 (e.g., update 5 servers, then 5 more, etc.)  │
│                                                              │
│  STRATEGIES   → "HOW to execute tasks across hosts"          │
│                 (linear, free, host_pinned, debug)           │
│                                                              │
│  RUN_ONCE     → "Run this task only ONCE, not per host"      │
│                 (e.g., run a DB migration once)              │
│                                                              │
│  THROTTLE     → "Limit concurrent task execution"            │
│                 (e.g., max 3 hosts downloading at once)      │
│                                                              │
│  MAX_FAIL     → "How many hosts can fail before we stop?"    │
│                 (e.g., stop if more than 20% fail)           │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

Let's learn each one with real, practical examples.

## 🎯 Part 1: Delegation — "Run This Task Somewhere Else"

### What is Delegation?

Normally, every task runs on the host it's targeting. But sometimes, you need a task to run on a **different** machine while still being part of the current play.

**Real-world examples:**
- Remove a server from the load balancer → run on the **load balancer**, not the web server
- Create a DNS record → run on the **DNS server** (or call an API from localhost)
- Send a notification to Slack → run on **localhost** (your Ansible controller)
- Grab an auth token from a vault server → run on **localhost**

### delegate_to — Basic Usage

```yaml
---
- name: Rolling update of web servers
  hosts: webservers
  become: true
  serial: 1

  tasks:
    # This runs on the LOAD BALANCER, not the web server
    - name: Remove server from load balancer
      ansible.builtin.command: >
        /usr/local/bin/remove-from-lb {{ inventory_hostname }}
      delegate_to: lb.example.com
    
    # This runs on the WEB SERVER (normal behavior)
    - name: Update application
      ansible.builtin.apt:
        name: myapp
        state: latest
    
    # This runs on the WEB SERVER (normal behavior)
    - name: Restart application
      ansible.builtin.service:
        name: myapp
        state: restarted
    
    # This runs on the WEB SERVER
    - name: Wait for app to be ready
      ansible.builtin.uri:
        url: "http://localhost:8080/health"
        status_code: 200
      retries: 10
      delay: 5

    # This runs on the LOAD BALANCER again
    - name: Add server back to load balancer
      ansible.builtin.command: >
        /usr/local/bin/add-to-lb {{ inventory_hostname }}
      delegate_to: lb.example.com
```

**What's happening step by step:**

```
For web1.example.com:
  1. [lb.example.com]   → Remove web1 from load balancer
  2. [web1.example.com] → Update application
  3. [web1.example.com] → Restart application
  4. [web1.example.com] → Health check
  5. [lb.example.com]   → Add web1 back to load balancer

Then for web2.example.com:
  1. [lb.example.com]   → Remove web2 from load balancer
  2. [web2.example.com] → Update application
  ... and so on
```

### delegate_to: localhost

The most common delegation target is **localhost** — your Ansible controller machine:

```yaml
- name: Deploy and notify
  hosts: webservers
  become: true

  tasks:
    - name: Deploy application
      ansible.builtin.copy:
        src: app.tar.gz
        dest: /opt/app/
    
    # This runs on YOUR machine (the Ansible controller)
    - name: Send Slack notification
      ansible.builtin.uri:
        url: https://hooks.slack.com/services/YOUR/WEBHOOK/URL
        method: POST
        body_format: json
        body:
          text: "Deployed app to {{ inventory_hostname }}"
      delegate_to: localhost
      become: false                    # Don't need sudo on localhost
    
    # This runs on YOUR machine
    - name: Update monitoring system
      ansible.builtin.uri:
        url: "https://monitoring.example.com/api/deploy"
        method: POST
        body_format: json
        body:
          server: "{{ inventory_hostname }}"
          version: "{{ app_version }}"
      delegate_to: localhost
      become: false
```

### local_action — Shorthand for delegate_to: localhost

Instead of writing `delegate_to: localhost` every time, you can use `local_action`:

```yaml
# These two are IDENTICAL:

# Option 1: delegate_to
- name: Send notification
  ansible.builtin.uri:
    url: https://hooks.slack.com/services/TOKEN
    method: POST
    body_format: json
    body:
      text: "Done!"
  delegate_to: localhost

# Option 2: local_action (shorter)
- name: Send notification
  local_action:
    module: ansible.builtin.uri
    url: https://hooks.slack.com/services/TOKEN
    method: POST
    body_format: json
    body:
      text: "Done!"
```

> 💡 **Which to use?** Most people prefer `delegate_to: localhost` because it's clearer and more consistent.

### delegate_facts — Store Facts on the Delegated Host

By default, when you delegate a task, the **facts** (results) are still stored on the **original** host. If you want them stored on the **delegated** host instead:

```yaml
- name: Gather facts from database server
  ansible.builtin.setup:
  delegate_to: db.example.com
  delegate_facts: true              # ← Store facts under db.example.com, not current host

# Now you can use db server facts in your play
- name: Show database server memory
  ansible.builtin.debug:
    msg: "DB server has {{ hostvars['db.example.com'].ansible_memtotal_mb }} MB RAM"
```

**When is this useful?**
- You're running a play on web servers but need facts about the database server
- You need to gather info from one host and use it in tasks on another

### Connection: local — Another Way to Run Locally

For playbooks that run **entirely** on localhost:

```yaml
- name: Cloud infrastructure setup
  hosts: localhost
  connection: local                   # ← Everything runs locally
  
  tasks:
    - name: Create AWS EC2 instance
      amazon.aws.ec2_instance:
        name: my-server
        instance_type: t3.micro
        image_id: ami-12345678
```

## 🔢 Part 2: Serial — "How Many Hosts at a Time?"

### What is Serial?

By default, Ansible runs a play on **ALL** hosts at the same time. The `serial` keyword limits how many hosts are updated at once.

This is **essential** for:
- Rolling updates (zero-downtime deployments)
- Database migrations (run on one server first, verify, then continue)
- Rate-limited operations (API calls with rate limits)

### Basic Serial Usage

```yaml
---
# Without serial: ALL 100 servers updated simultaneously
# With serial: 5 servers at a time

- name: Rolling update
  hosts: webservers          # Let's say this has 100 hosts
  become: true
  serial: 5                  # ← Update 5 at a time

  tasks:
    - name: Update application
      ansible.builtin.apt:
        name: myapp
        state: latest

    - name: Restart service
      ansible.builtin.service:
        name: myapp
        state: restarted
```

**What happens:**

```
Batch 1: web01, web02, web03, web04, web05
  → Update app on all 5
  → Restart service on all 5
  → ✅ All good? Continue.

Batch 2: web06, web07, web08, web09, web10
  → Update app on all 5
  → Restart service on all 5
  → ✅ All good? Continue.

... continues until all 100 are done ...

Batch 20: web96, web97, web98, web99, web100
  → Update app on all 5
  → Restart service on all 5
  → ✅ Done!
```

### Serial with Percentage

Instead of a fixed number, use a percentage:

```yaml
- name: Rolling update (percentage)
  hosts: webservers
  serial: "25%"              # ← Update 25% of hosts at a time

  tasks:
    - name: Update application
      ansible.builtin.apt:
        name: myapp
        state: latest
```

If you have 100 hosts: batches of 25 → 25 → 25 → 25

### Serial with Ramp-Up (Canary Deployments!)

This is the **really powerful** feature. You can start with a small batch (canary), verify it works, then increase:

```yaml
- name: Canary deployment
  hosts: webservers
  become: true
  serial:
    - 1                       # First: update just 1 server (the canary)
    - 5                       # Then: update 5 more
    - "25%"                   # Then: update 25% of remaining
    # The rest get the final batch

  tasks:
    - name: Update application
      ansible.builtin.apt:
        name: myapp
        state: latest

    - name: Health check
      ansible.builtin.uri:
        url: "http://{{ inventory_hostname }}:8080/health"
        status_code: 200
      retries: 5
      delay: 3
```

**What happens with 100 hosts:**

```
Batch 1: web01                        (1 server — the canary 🐤)
  → Update, health check
  → ✅ Canary is healthy? Continue.

Batch 2: web02, web03, web04, web05, web06   (5 servers)
  → Update, health check
  → ✅ All healthy? Continue.

Batch 3: web07 through web30          (25% = ~24 servers)
  → Update, health check
  → ✅ All healthy? Continue.

Batch 4: web31 through web100         (remaining ~70 servers)
  → Update, health check
  → ✅ Done!
```

**This is exactly how companies like Netflix, Amazon, and Google deploy updates.** Start small, gain confidence, then go big.

### Combining Serial with Delegation

The classic rolling update pattern:

```yaml
---
- name: Zero-downtime rolling update
  hosts: webservers
  become: true
  serial: 1                  # One server at a time — safest approach

  pre_tasks:
    # Step 1: Remove from load balancer (runs on LB)
    - name: Disable server in HAProxy
      community.general.haproxy:
        state: disabled
        host: "{{ inventory_hostname }}"
        socket: /var/run/haproxy/admin.sock
      delegate_to: "{{ item }}"
      loop: "{{ groups['loadbalancers'] }}"
    
    # Step 2: Wait for connections to drain
    - name: Wait for connections to drain
      ansible.builtin.wait_for:
        timeout: 30

  tasks:
    # Step 3: Update the application
    - name: Pull latest code
      ansible.builtin.git:
        repo: https://github.com/company/app.git
        dest: /opt/app
        version: "{{ app_version }}"
    
    - name: Install dependencies
      ansible.builtin.pip:
        requirements: /opt/app/requirements.txt
        virtualenv: /opt/app/venv

    - name: Restart application
      ansible.builtin.service:
        name: myapp
        state: restarted

  post_tasks:
    # Step 4: Wait for app to be ready
    - name: Wait for application to be ready
      ansible.builtin.uri:
        url: "http://{{ inventory_hostname }}:8080/health"
        status_code: 200
      register: health
      until: health.status == 200
      retries: 30
      delay: 5

    # Step 5: Add back to load balancer (runs on LB)
    - name: Enable server in HAProxy
      community.general.haproxy:
        state: enabled
        host: "{{ inventory_hostname }}"
        socket: /var/run/haproxy/admin.sock
      delegate_to: "{{ item }}"
      loop: "{{ groups['loadbalancers'] }}"
```

**The flow for each server:**

```
web01:
  ┌─────────────────────────────┐
  │ 1. Remove from load balancer │  ← delegate_to: lb
  │ 2. Wait for drain            │
  │ 3. Update code               │
  │ 4. Install deps              │
  │ 5. Restart app               │
  │ 6. Health check              │
  │ 7. Add back to load balancer │  ← delegate_to: lb
  └─────────────────────────────┘
        ↓ Only when web01 is done
web02:
  ┌─────────────────────────────┐
  │ 1. Remove from load balancer │
  │ ... same steps ...           │
  │ 7. Add back to load balancer │
  └─────────────────────────────┘
        ↓ ... and so on
```

**Zero downtime achieved!** At no point are ALL servers down. 🎉

## ⚡ Part 3: Strategies — "HOW to Execute Tasks"

### What is a Strategy?

A **strategy** controls the ORDER in which Ansible executes tasks across hosts. Think of it as the execution algorithm.

### Strategy 1: linear (DEFAULT)

This is what Ansible does by default. It runs **one task on ALL hosts** before moving to the next task.

```yaml
- name: Linear strategy (default)
  hosts: webservers
  strategy: linear              # ← This is the default, you don't need to write it

  tasks:
    - name: Task 1 - Install packages
      ansible.builtin.apt:
        name: nginx
        state: present

    - name: Task 2 - Copy config
      ansible.builtin.copy:
        src: nginx.conf
        dest: /etc/nginx/nginx.conf

    - name: Task 3 - Start service
      ansible.builtin.service:
        name: nginx
        state: started
```

**Execution order with 3 hosts:**

```
Task 1 → web01, web02, web03    (all hosts run Task 1)
         ↓ wait for ALL to finish
Task 2 → web01, web02, web03    (all hosts run Task 2)
         ↓ wait for ALL to finish
Task 3 → web01, web02, web03    (all hosts run Task 3)
```

**The problem:** If web03 is slow at Task 1, web01 and web02 sit around waiting. 🐌

### Strategy 2: free — "Go as Fast as You Can"

Each host runs through ALL tasks independently, as fast as possible, without waiting for other hosts.

```yaml
- name: Free strategy — maximum speed
  hosts: webservers
  strategy: free                 # ← Each host goes at its own pace

  tasks:
    - name: Task 1 - Install packages
      ansible.builtin.apt:
        name: nginx
        state: present

    - name: Task 2 - Copy config
      ansible.builtin.copy:
        src: nginx.conf
        dest: /etc/nginx/nginx.conf

    - name: Task 3 - Start service
      ansible.builtin.service:
        name: nginx
        state: started
```

**Execution order with 3 hosts:**

```
web01 (fast):     Task 1 → Task 2 → Task 3    ✅ Done!
web02 (medium):   Task 1 ──→ Task 2 ──→ Task 3    ✅ Done!
web03 (slow):     Task 1 ────→ Task 2 ────→ Task 3    ✅ Done!
```

Each host does its own thing without waiting for the others.

**When to use `free`:**
- Hosts are independent (no cross-host dependencies)
- You want maximum speed
- Tasks on one host don't depend on tasks completing on another host

**When NOT to use `free`:**
- Tasks depend on all hosts being at the same state
- You need coordinated deployments
- You're using `serial` (serial overrides strategy)

### Strategy 3: host_pinned — "Finish One Host Before Starting Another"

Like `free`, but Ansible won't start a new host until the current host finishes or gets blocked.

```yaml
- name: Host-pinned strategy
  hosts: webservers
  strategy: host_pinned

  tasks:
    - name: Task 1
      ansible.builtin.debug:
        msg: "Running on {{ inventory_hostname }}"

    - name: Task 2
      ansible.builtin.debug:
        msg: "Still on {{ inventory_hostname }}"
```

**Execution:**

```
web01: Task 1 → Task 2   (all tasks on web01 first)
         ↓
web02: Task 1 → Task 2   (then all tasks on web02)
         ↓
web03: Task 1 → Task 2   (then all tasks on web03)
```

### Strategy 4: debug — Interactive Debugging

Lets you **pause and inspect** task execution interactively. Amazing for debugging complex plays.

```yaml
- name: Debug strategy
  hosts: webservers
  strategy: debug

  tasks:
    - name: This task might fail
      ansible.builtin.command: /bin/false
```

When a task fails, you get a debug prompt:

```
TASK [This task might fail] ****
fatal: [web01]: FAILED! => {"changed": true, "cmd": ["/bin/false"], "rc": 1}

[web01] TASK: This task might fail (debug)> 
```

**Debug commands:**

| Command | What It Does |
|---------|-------------|
| `p task` | Print the task definition |
| `p task.args` | Print task arguments |
| `p host` | Print current host info |
| `p result` | Print the task result |
| `p vars` | Print all variables |
| `r` | Retry the failed task |
| `c` | Continue to next task |
| `q` | Quit the playbook |

```
[web01] TASK: This task might fail (debug)> p result._result
{'changed': True, 'cmd': ['/bin/false'], 'rc': 1, 'stdout': '', 'stderr': ''}

[web01] TASK: This task might fail (debug)> p task.args
{'_raw_params': '/bin/false'}

[web01] TASK: This task might fail (debug)> c    # Continue to next task
```

### Comparing All Strategies

```
┌──────────────────────────────────────────────────────────────────┐
│                    Strategy Comparison                            │
├──────────────┬───────────────────────────────────────────────────┤
│  linear      │ Task 1 on ALL hosts → Task 2 on ALL → Task 3... │
│  (default)   │ 🔒 Synchronized — hosts wait for each other     │
│              │ 📌 Use when: tasks depend on all hosts being     │
│              │    at the same step                              │
├──────────────┼───────────────────────────────────────────────────┤
│  free        │ Each host runs all tasks independently           │
│              │ 🚀 Fastest — no waiting                          │
│              │ 📌 Use when: hosts are independent, speed matters│
├──────────────┼───────────────────────────────────────────────────┤
│  host_pinned │ Complete all tasks on one host before moving on  │
│              │ 🎯 Focused — one host at a time                  │
│              │ 📌 Use when: you want serial-like behavior       │
│              │    with strategy-level control                   │
├──────────────┼───────────────────────────────────────────────────┤
│  debug       │ Pauses on failure for interactive debugging      │
│              │ 🔍 Debugging only — never use in production      │
│              │ 📌 Use when: troubleshooting complex plays       │
└──────────────┴───────────────────────────────────────────────────┘
```

### Setting Strategy Globally (in ansible.cfg)

```ini
# ansible.cfg
[defaults]
strategy = free                # Change default strategy for ALL plays
```

## 1️⃣ Part 4: run_once — "Run This Task Only Once"

### What is run_once?

Some tasks should only run **once**, regardless of how many hosts are in the play. For example:
- Running a database migration
- Sending a Slack notification
- Creating a shared resource

```yaml
- name: Deploy application
  hosts: webservers            # 20 servers
  become: true

  tasks:
    # This runs on ALL 20 servers
    - name: Update application code
      ansible.builtin.git:
        repo: https://github.com/company/app.git
        dest: /opt/app
        version: main

    # This runs on ONLY ONE server (the first one in inventory)
    - name: Run database migrations
      ansible.builtin.command: python manage.py migrate
      args:
        chdir: /opt/app
      run_once: true             # ← Only runs ONCE!
    
    # This runs on ALL 20 servers
    - name: Restart application
      ansible.builtin.service:
        name: myapp
        state: restarted
    
    # This runs ONCE on localhost
    - name: Notify team on Slack
      ansible.builtin.uri:
        url: https://hooks.slack.com/services/TOKEN
        method: POST
        body_format: json
        body:
          text: "Deployment complete! Version: {{ app_version }}"
      delegate_to: localhost
      run_once: true             # ← Only one notification, not 20!
      become: false
```

### run_once + serial

When using `serial`, `run_once` runs once **per batch**, not once for the entire play:

```yaml
- name: Batch deployment
  hosts: webservers
  serial: 5                     # 5 at a time
  
  tasks:
    - name: Database check
      ansible.builtin.command: check_db.sh
      run_once: true             # Runs ONCE per batch of 5!
```

```
Batch 1 (web01-05): → run_once runs on web01
Batch 2 (web06-10): → run_once runs on web06  ← Runs AGAIN!
Batch 3 (web11-15): → run_once runs on web11  ← Runs AGAIN!
```

> ⚠️ **Watch out!** If you want something to truly run once across ALL batches, use `delegate_to: localhost` with a conditional:

```yaml
- name: Truly run once across all batches
  ansible.builtin.command: run_migration.sh
  delegate_to: localhost
  run_once: true
  when: migration_done is not defined

- name: Set migration flag
  ansible.builtin.set_fact:
    migration_done: true
  run_once: true
```

## 🚦 Part 5: Throttle — "Limit Concurrent Execution"

### What is Throttle?

`throttle` limits how many hosts can run a **specific task** concurrently. This is different from `serial`, which limits the entire play.

```yaml
- name: Update all servers
  hosts: webservers              # 50 servers
  become: true

  tasks:
    # This runs on all 50 servers concurrently (default)
    - name: Install prerequisites
      ansible.builtin.apt:
        name: curl
        state: present

    # This runs on only 3 servers at a time
    - name: Download large artifact (bandwidth limited)
      ansible.builtin.get_url:
        url: https://releases.example.com/app-v2.0.tar.gz
        dest: /tmp/app.tar.gz
      throttle: 3                 # ← Only 3 downloads at once!

    # This runs on all 50 servers concurrently again
    - name: Extract and install
      ansible.builtin.unarchive:
        src: /tmp/app.tar.gz
        dest: /opt/app
        remote_src: true
```

**Why is this useful?**
- **Bandwidth limits:** Don't DDoS your artifact server with 50 simultaneous downloads
- **API rate limits:** External APIs might block you if you hit them too fast
- **Shared resources:** Database connections, license servers, etc.

### Throttle vs Serial — What's the Difference?

```
serial: 5
  → Limits the ENTIRE PLAY to 5 hosts at a time
  → All tasks run on 5 hosts, then the next 5, etc.

throttle: 5 (on a specific task)
  → Only limits THAT ONE TASK to 5 concurrent executions
  → Other tasks still run on all hosts
```

```yaml
- name: Example showing the difference
  hosts: webservers              # 20 hosts
  
  tasks:
    - name: Task 1 - runs on all 20 concurrently
      ansible.builtin.debug:
        msg: "Hello from {{ inventory_hostname }}"

    - name: Task 2 - runs on 3 at a time only
      ansible.builtin.debug:
        msg: "Throttled on {{ inventory_hostname }}"
      throttle: 3

    - name: Task 3 - runs on all 20 concurrently again
      ansible.builtin.debug:
        msg: "Back to full speed on {{ inventory_hostname }}"
```

## 💥 Part 6: max_fail_percentage — "How Many Failures are Acceptable?"

### What is max_fail_percentage?

By default, if ANY host fails, Ansible stops the play for that host but continues on others. `max_fail_percentage` lets you stop the ENTIRE play if too many hosts fail.

### Basic Usage

```yaml
- name: Rolling update with failure threshold
  hosts: webservers              # 100 servers
  become: true
  serial: 10                    # 10 at a time
  max_fail_percentage: 20       # ← Stop if more than 20% fail in a batch

  tasks:
    - name: Update application
      ansible.builtin.apt:
        name: myapp
        state: latest

    - name: Health check
      ansible.builtin.uri:
        url: "http://{{ inventory_hostname }}:8080/health"
        status_code: 200
```

**What happens:**
- Batch of 10 servers
- If 2 fail (20%) → continue (barely at the limit)
- If 3 fail (30%) → **STOP EVERYTHING** — something is seriously wrong

### any_errors_fatal — Stop on ANY Failure

If you want to be extra careful (zero tolerance for failures):

```yaml
- name: Critical deployment
  hosts: database_servers
  any_errors_fatal: true        # ← ONE failure = stop the ENTIRE play

  tasks:
    - name: Backup database
      ansible.builtin.command: pg_dump mydb > /backup/mydb.sql

    - name: Apply schema migration
      ansible.builtin.command: psql -f migration.sql mydb
```

### max_fail_percentage vs any_errors_fatal

```
┌────────────────────────────────────────────────────────┐
│  max_fail_percentage: 0    →  Same as any_errors_fatal │
│  max_fail_percentage: 20   →  Allow up to 20% failures │
│  max_fail_percentage: 100  →  Allow ALL failures       │
│  any_errors_fatal: true    →  Zero tolerance           │
│  any_errors_fatal: false   →  Default (continue on     │
│                               remaining hosts)          │
└────────────────────────────────────────────────────────┘
```

## 🔄 Part 7: Order — Controlling Host Execution Order

### The `order` Parameter

Control the order in which hosts are processed:

```yaml
- name: Update servers in specific order
  hosts: webservers
  order: sorted                  # ← Alphabetical order

  tasks:
    - name: Show execution order
      ansible.builtin.debug:
        msg: "Updating {{ inventory_hostname }}"
```

### Available Order Values

| Order | Description | Example |
|-------|-------------|---------|
| `inventory` | As listed in inventory (default) | web01, web03, web02 |
| `sorted` | Alphabetical | web01, web02, web03 |
| `reverse_sorted` | Reverse alphabetical | web03, web02, web01 |
| `reverse_inventory` | Reverse of inventory order | web02, web03, web01 |
| `shuffle` | Random order every time | web03, web01, web02 (random) |

### When to Use Each Order

```yaml
# Use 'shuffle' for rolling updates — spreads risk across racks/zones
- name: Rolling update (random order)
  hosts: webservers
  serial: 5
  order: shuffle
  tasks: [...]

# Use 'sorted' for predictable, reproducible deployments
- name: Predictable deployment
  hosts: webservers
  order: sorted
  tasks: [...]
```

## 🎮 Part 8: Pre-tasks, Post-tasks, and Handlers

### pre_tasks and post_tasks

These let you run tasks **before** and **after** the main `tasks` section. Handlers are flushed between each section.

```yaml
- name: Full deployment lifecycle
  hosts: webservers
  become: true
  serial: 5

  pre_tasks:
    - name: Disable monitoring alerts
      ansible.builtin.uri:
        url: "https://monitoring.example.com/api/silence"
        method: POST
        body_format: json
        body:
          host: "{{ inventory_hostname }}"
          duration: "30m"
      delegate_to: localhost
      become: false

    - name: Remove from load balancer
      community.general.haproxy:
        state: disabled
        host: "{{ inventory_hostname }}"
        socket: /var/run/haproxy/admin.sock
      delegate_to: lb.example.com

  # Handlers triggered in pre_tasks are flushed HERE

  tasks:
    - name: Update application
      ansible.builtin.apt:
        name: myapp
        state: latest
      notify: Restart application

    - name: Deploy config
      ansible.builtin.template:
        src: app.conf.j2
        dest: /etc/myapp/app.conf
      notify: Restart application

  # Handlers triggered in tasks are flushed HERE

  post_tasks:
    - name: Health check
      ansible.builtin.uri:
        url: "http://{{ inventory_hostname }}:8080/health"
        status_code: 200
      retries: 10
      delay: 5

    - name: Add back to load balancer
      community.general.haproxy:
        state: enabled
        host: "{{ inventory_hostname }}"
        socket: /var/run/haproxy/admin.sock
      delegate_to: lb.example.com

    - name: Re-enable monitoring alerts
      ansible.builtin.uri:
        url: "https://monitoring.example.com/api/unsilence"
        method: POST
        body_format: json
        body:
          host: "{{ inventory_hostname }}"
      delegate_to: localhost
      become: false

  # Handlers triggered in post_tasks are flushed HERE

  handlers:
    - name: Restart application
      ansible.builtin.service:
        name: myapp
        state: restarted
```

**Handler flush order:**

```
1. pre_tasks execute
2. Handlers triggered by pre_tasks are flushed (run)
3. tasks execute
4. Handlers triggered by tasks are flushed (run)
5. post_tasks execute
6. Handlers triggered by post_tasks are flushed (run)
```

## 🧩 Part 9: Putting It All Together — Real-World Patterns

### Pattern 1: Blue-Green Deployment

```yaml
---
- name: Blue-Green Deployment
  hosts: all
  gather_facts: false

  tasks:
    - name: Determine current active group (blue or green)
      ansible.builtin.command: cat /etc/active_environment
      register: active_env
      delegate_to: lb.example.com
      run_once: true

    - name: Set inactive group
      ansible.builtin.set_fact:
        deploy_group: "{{ 'green' if active_env.stdout == 'blue' else 'blue' }}"
      run_once: true

- name: Deploy to inactive environment
  hosts: "{{ deploy_group }}_servers"
  become: true

  tasks:
    - name: Deploy new version
      ansible.builtin.git:
        repo: https://github.com/company/app.git
        dest: /opt/app
        version: "{{ app_version }}"

    - name: Start application
      ansible.builtin.service:
        name: myapp
        state: restarted

    - name: Health check
      ansible.builtin.uri:
        url: "http://{{ inventory_hostname }}:8080/health"
        status_code: 200
      retries: 15
      delay: 5

- name: Switch traffic
  hosts: loadbalancers
  become: true

  tasks:
    - name: Switch load balancer to new environment
      ansible.builtin.template:
        src: haproxy.cfg.j2
        dest: /etc/haproxy/haproxy.cfg
      notify: Reload HAProxy

  handlers:
    - name: Reload HAProxy
      ansible.builtin.service:
        name: haproxy
        state: reloaded
```

### Pattern 2: Database Migration with App Update

```yaml
---
- name: Database migration and app update
  hosts: webservers
  become: true
  serial:
    - 1                          # Canary first
    - "100%"                     # Then all the rest

  pre_tasks:
    # Remove from LB
    - name: Drain connections
      ansible.builtin.command: >
        /usr/local/bin/lb-drain {{ inventory_hostname }}
      delegate_to: lb.example.com

  tasks:
    # Deploy code to ALL servers
    - name: Deploy application code
      ansible.builtin.git:
        repo: https://github.com/company/app.git
        dest: /opt/app
        version: "{{ app_version }}"

    # Run migration ONCE (only from the canary server)
    - name: Run database migrations
      ansible.builtin.command: >
        /opt/app/venv/bin/python manage.py migrate --no-input
      args:
        chdir: /opt/app
      run_once: true
      register: migration_result

    # Show migration result (once)
    - name: Display migration results
      ansible.builtin.debug:
        var: migration_result.stdout_lines
      run_once: true

    # Restart app on current batch
    - name: Restart application
      ansible.builtin.service:
        name: myapp
        state: restarted

  post_tasks:
    - name: Health check
      ansible.builtin.uri:
        url: "http://{{ inventory_hostname }}:8080/health"
        status_code: 200
      retries: 10
      delay: 5

    - name: Add back to load balancer
      ansible.builtin.command: >
        /usr/local/bin/lb-enable {{ inventory_hostname }}
      delegate_to: lb.example.com
```

### Pattern 3: Multi-Tier Application Deployment

```yaml
---
# Step 1: Update databases first (one at a time)
- name: Update database servers
  hosts: databases
  become: true
  serial: 1
  any_errors_fatal: true         # Stop everything if DB update fails

  tasks:
    - name: Backup database
      ansible.builtin.command: pg_dump mydb > /backup/mydb_{{ ansible_date_time.iso8601 }}.sql
      
    - name: Apply schema changes
      ansible.builtin.command: psql -f /opt/migrations/latest.sql mydb
      run_once: true

# Step 2: Update application servers (rolling)
- name: Update application servers
  hosts: appservers
  become: true
  serial:
    - 1
    - "25%"
    - "100%"
  max_fail_percentage: 10

  pre_tasks:
    - name: Disable in load balancer
      ansible.builtin.command: >
        /usr/local/bin/lb-disable {{ inventory_hostname }}
      delegate_to: lb.example.com

  tasks:
    - name: Deploy application
      ansible.builtin.unarchive:
        src: "https://releases.example.com/app-{{ app_version }}.tar.gz"
        dest: /opt/app
        remote_src: true

    - name: Restart application
      ansible.builtin.service:
        name: myapp
        state: restarted

  post_tasks:
    - name: Health check
      ansible.builtin.uri:
        url: "http://{{ inventory_hostname }}:8080/health"
        status_code: 200
      retries: 10
      delay: 3

    - name: Enable in load balancer
      ansible.builtin.command: >
        /usr/local/bin/lb-enable {{ inventory_hostname }}
      delegate_to: lb.example.com

# Step 3: Update cache servers (all at once — they rebuild from DB)
- name: Update cache servers
  hosts: cacheservers
  become: true
  strategy: free                 # As fast as possible

  tasks:
    - name: Flush and rebuild cache
      ansible.builtin.command: redis-cli FLUSHALL

    - name: Warm cache
      ansible.builtin.command: /opt/scripts/warm-cache.sh

# Step 4: Notify team
- name: Send notification
  hosts: localhost
  connection: local

  tasks:
    - name: Send Slack notification
      ansible.builtin.uri:
        url: https://hooks.slack.com/services/TOKEN
        method: POST
        body_format: json
        body:
          text: "Deployment of v{{ app_version }} complete! All {{ groups['appservers'] | length }} servers updated."
```

## 🎯 Quick Reference Cheat Sheet

### Delegation

| Feature | Syntax | Use Case |
|---------|--------|----------|
| `delegate_to: host` | Runs task on another host | LB operations, API calls |
| `delegate_to: localhost` | Runs task on controller | API calls, notifications |
| `local_action` | Shorthand for localhost | Same as above |
| `delegate_facts: true` | Store facts on delegate | Cross-host fact gathering |
| `connection: local` | Entire play runs locally | Cloud provisioning |

### Serial

| Feature | Syntax | What It Does |
|---------|--------|-------------|
| `serial: 5` | Fixed number | 5 hosts per batch |
| `serial: "25%"` | Percentage | 25% of hosts per batch |
| `serial: [1, 5, "50%"]` | Ramp-up | Canary → small batch → big batch |

### Strategies

| Strategy | Behavior | Use Case |
|----------|----------|----------|
| `linear` (default) | Task-by-task across all hosts | Most playbooks |
| `free` | Each host goes independently | Speed, independent hosts |
| `host_pinned` | All tasks on one host first | Focused execution |
| `debug` | Interactive debugging | Troubleshooting |

### Execution Control

| Feature | Syntax | What It Does |
|---------|--------|-------------|
| `run_once: true` | Run task once | DB migrations, notifications |
| `throttle: N` | Limit per task | Rate-limited operations |
| `max_fail_percentage: N` | Failure threshold | Stop if too many fail |
| `any_errors_fatal: true` | Zero tolerance | Critical operations |
| `order: shuffle` | Random host order | Risk distribution |

## 🧠 Summary

```
┌──────────────────────────────────────────────────────────────────┐
│           Ansible Advanced Patterns Summary                      │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  DELEGATION    delegate_to: host                                 │
│                → Run a task on a DIFFERENT machine                │
│                → Common: LB operations, API calls, notifications │
│                                                                  │
│  SERIAL        serial: 5  |  serial: "25%"  |  serial: [1,5,50%]│
│                → Control batch size for rolling updates          │
│                → Ramp-up pattern = canary deployments             │
│                                                                  │
│  STRATEGIES    linear (default) | free | host_pinned | debug     │
│                → Control HOW tasks flow across hosts             │
│                                                                  │
│  RUN_ONCE      run_once: true                                    │
│                → Execute a task only once, not per host           │
│                → ⚠️ Runs once PER BATCH when using serial       │
│                                                                  │
│  THROTTLE      throttle: 3                                       │
│                → Limit concurrent execution of a single task     │
│                                                                  │
│  FAILURE       max_fail_percentage: 20  |  any_errors_fatal      │
│                → Control what happens when things go wrong       │
│                                                                  │
│  ORDER         order: sorted | shuffle | inventory               │
│                → Control host processing order                   │
│                                                                  │
│  PRE/POST      pre_tasks → tasks → post_tasks                    │
│                → Lifecycle hooks with handler flushing            │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

**The golden rule:** When deploying to production, **always use `serial`** with a ramp-up pattern, **always have health checks**, and **always have a way to stop** if things go wrong (`max_fail_percentage`). Your future self (and your on-call team) will thank you.
