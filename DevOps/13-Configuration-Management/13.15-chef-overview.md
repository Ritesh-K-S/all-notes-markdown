## 1. What is Chef?

**Chef** is a **configuration management tool** that lets you write code (in Ruby) to describe how your servers (called **nodes**) should be configured. Chef then ensures every node matches that description — automatically, consistently, and repeatedly.

### The Big Idea

Instead of SSH-ing into 100 servers and manually installing Nginx, configuring firewalls, and setting up users... you write a **recipe** once, and Chef applies it to ALL your servers.

> Think of Chef like a **real chef in a kitchen** 🍳 — you give them a **recipe** (step-by-step instructions), and they cook the dish exactly the same way every time. No guessing, no "I think I added salt last time."

### Simple Analogy

| Real Kitchen | Chef (the tool) |
|-------------|----------------|
| A recipe (steps to cook a dish) | A **Recipe** (Ruby code describing server configuration) |
| A cookbook (collection of recipes) | A **Cookbook** (collection of related recipes) |
| Kitchen tools (knife, oven, stove) | **Knife** (CLI tool), **Chef Client** (the engine) |
| The head chef checking everything | **Chef Server** (central hub managing everything) |
| The ingredients | **Resources** (packages, files, services, users) |

### Key Facts

| Feature | Details |
|---------|---------|
| **Created By** | Adam Jacob (2009), now maintained by Progress Software |
| **Written In** | Ruby & Erlang |
| **Configuration Language** | Ruby DSL (Domain Specific Language) |
| **Model** | Pull-based (nodes pull config from Chef Server) |
| **Agent Required?** | Yes — `chef-client` runs on every managed node |
| **Communication** | HTTPS (Chef Client → Chef Server) |
| **Open Source?** | Yes — Chef Infra is open source (Apache 2.0 license) |
| **Commercial Version** | Chef Enterprise / Progress Chef |

## 2. Why Chef? — The Problem It Solves

### Without Chef (The Manual Nightmare)

```
Scenario: You have 200 web servers, and you need to:
  1. Install Nginx
  2. Deploy a specific config file
  3. Create a "deploy" user
  4. Set up log rotation
  5. Install security patches

The manual way:
  → SSH into server-1, run 15 commands, edit 3 files
  → SSH into server-2, run 15 commands, edit 3 files  
  → Repeat 198 more times... 
  → Server-47 has a typo in nginx.conf 💥
  → Server-123 was missed entirely 😱
  → 3 days later, you're still not done
  → 3 months later, half the servers have drifted from the original config
```

### With Chef (The Automated Way)

```ruby
# Write it ONCE in a recipe:
package 'nginx' do
  action :install
end

template '/etc/nginx/nginx.conf' do
  source 'nginx.conf.erb'
  notifies :restart, 'service[nginx]'
end

user 'deploy' do
  shell '/bin/bash'
  home '/home/deploy'
end

service 'nginx' do
  action [:enable, :start]
end

# Apply to ALL 200 servers → Done in minutes
# Every server is IDENTICAL — guaranteed
# Run it again next week → It only fixes what drifted
```

### Key Benefits

| Benefit | Explanation |
|---------|------------|
| **Consistency** | Every server is configured exactly the same |
| **Idempotency** | Run it 1 time or 100 times — same result (doesn't re-install what's already there) |
| **Version Control** | Your infrastructure config is in Git — full history, code reviews, rollback |
| **Scalability** | Configure 5 servers or 5,000 — same effort |
| **Self-Healing** | If someone manually changes something, Chef reverts it back on the next run |
| **Compliance** | Prove your servers meet security standards — the code IS the documentation |

## 3. Chef vs Ansible vs Puppet — Quick Comparison

| Feature | Chef | Ansible | Puppet |
|---------|------|---------|--------|
| **Language** | Ruby DSL | YAML | Puppet DSL |
| **Architecture** | Pull (agent-based) | Push (agentless, SSH) | Pull (agent-based) |
| **Learning Curve** | Steep (need Ruby knowledge) | Gentle (YAML is simple) | Moderate |
| **Agent Required** | Yes (`chef-client`) | No (just SSH) | Yes (`puppet-agent`) |
| **Central Server** | Chef Server (optional with Chef Solo) | No (control node only) | Puppet Server |
| **Idempotent** | Yes | Yes | Yes |
| **Best For** | Complex, large infra (Ruby shops) | Quick automation, any team | Enterprise compliance |
| **Community** | Good (Chef Supermarket) | Excellent (Ansible Galaxy) | Good (Puppet Forge) |
| **Speed** | Moderate | Fast (for small infra) | Moderate |

### When to pick Chef over others?

- Your team **already knows Ruby**
- You need **fine-grained control** with a full programming language
- You're managing **large-scale infrastructure** (thousands of nodes)
- You want a **pull-based model** where nodes self-configure
- You need **Test-Driven Infrastructure** (Chef has excellent testing tools)

## 4. How Chef Works — Architecture

### The Three Main Components

```
┌──────────────────────────────────────────────────────────────────┐
│                        CHEF ARCHITECTURE                         │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────┐    Upload     ┌───────────────┐                │
│  │  WORKSTATION │ ──────────→  │  CHEF SERVER   │                │
│  │  (Your PC)   │   (knife)    │  (Central Hub) │                │
│  └─────────────┘               └───────┬───────┘                │
│                                        │                         │
│        You write code here        Stores everything:             │
│        and push to server         - Cookbooks                    │
│                                   - Node data                    │
│                                   - Roles                        │
│                                   - Environments                 │
│                                   - Data bags                    │
│                                        │                         │
│                          ┌─────────────┼──────────────┐          │
│                          │             │              │          │
│                          ▼             ▼              ▼          │
│                    ┌──────────┐  ┌──────────┐  ┌──────────┐     │
│                    │  NODE 1  │  │  NODE 2  │  │  NODE 3  │     │
│                    │ (Server) │  │ (Server) │  │ (Server) │     │
│                    │          │  │          │  │          │     │
│                    │ chef-    │  │ chef-    │  │ chef-    │     │
│                    │ client   │  │ client   │  │ client   │     │
│                    └──────────┘  └──────────┘  └──────────┘     │
│                                                                  │
│                    Each node PULLS its config                    │
│                    from Chef Server periodically                 │
│                    (every 30 min by default)                     │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### How the Flow Works (Step by Step)

```
Step 1: You (the DevOps engineer) write recipes/cookbooks on your WORKSTATION
            ↓
Step 2: You use "knife" (CLI tool) to UPLOAD cookbooks to the CHEF SERVER
            ↓
Step 3: The Chef Server stores everything — cookbooks, node info, roles, etc.
            ↓
Step 4: Each NODE has "chef-client" installed (an agent)
            ↓
Step 5: chef-client on each node contacts Chef Server every 30 minutes
            ↓
Step 6: Chef Server tells the node: "Here's your RUN LIST — these are the 
        recipes you should apply"
            ↓
Step 7: The node downloads the recipes and CONVERGES — meaning it checks 
        its current state against the desired state and makes changes if needed
            ↓
Step 8: Node reports back to Chef Server: "I'm all good!" or "Something failed"
```

### What is "Convergence"?

Convergence is Chef's way of saying: **"Make the node look exactly like the recipe describes."**

```
If the recipe says:     "Nginx should be installed"
And the node:           Already has Nginx installed
Then Chef:              Does NOTHING (already converged ✅)

If the recipe says:     "Nginx should be installed"
And the node:           Does NOT have Nginx
Then Chef:              Installs Nginx (converging 🔄)
```

This is **idempotency** in action — run it as many times as you want, same result.

## 5. Chef Key Terminology — The Vocabulary

Before we dive deeper, let's get the vocabulary straight. Chef uses cooking metaphors (because, well, it's called Chef!):

| Term | What It Means | Real-World Analogy |
|------|--------------|-------------------|
| **Node** | Any server/machine managed by Chef | A kitchen where food is prepared |
| **Recipe** | A Ruby file describing what a node should look like | A cooking recipe (steps to make a dish) |
| **Cookbook** | A collection of related recipes, templates, files, and attributes | A cookbook with many recipes |
| **Resource** | A single piece of configuration (a package, file, service, user) | A single step in a recipe ("add 2 cups flour") |
| **Run List** | The ordered list of recipes/roles to apply to a node | The menu for the day |
| **Knife** | CLI tool used to interact with Chef Server | The chef's main kitchen tool |
| **Chef Client** | The agent that runs on each node | The sous-chef who follows the recipe |
| **Chef Server** | Central hub storing all cookbooks, node data, etc. | The restaurant manager |
| **Chef Workstation** | Your laptop/PC where you write and test cookbooks | The recipe-writing desk |
| **Ohai** | A tool that collects system info (OS, IP, CPU, RAM) from nodes | The inspector who checks the pantry |
| **Attribute** | Variables that customize recipe behavior per node/environment | Adjusting ingredients for serving size |
| **Template** | A dynamic configuration file (ERB format) | A recipe template where you fill in blanks |
| **Data Bag** | Global JSON data accessible by any recipe | The shared pantry |
| **Role** | A group of recipes and attributes for a server type (e.g., "webserver") | Job description for a kitchen station |
| **Environment** | Represents a stage like dev, staging, production | Different restaurants in a chain |
| **Supermarket** | Community repository of pre-built cookbooks | A grocery store for recipes |

Chef Workstation includes everything you need: `chef`, `knife`, `cookstyle`, `test kitchen`, `inspec`, and more.

**On Ubuntu/Debian:**

```bash
# Download and install Chef Workstation
wget https://packages.chef.io/files/stable/chef-workstation/23.10.1066/ubuntu/22.04/chef-workstation_23.10.1066-1_amd64.deb

sudo dpkg -i chef-workstation_23.10.1066-1_amd64.deb

# Verify installation
chef --version
```

**On macOS:**

```bash
# Using the installer
brew install --cask chef-workstation

# Verify
chef --version
```

**On Windows:**

```powershell
# Download the MSI from https://www.chef.io/downloads
# Or use Chocolatey:
choco install chef-workstation

# Verify
chef --version
```

### What Gets Installed

```
Chef Workstation includes:
├── chef-client          → The agent (also useful for local testing)
├── knife                → CLI to manage Chef Server
├── chef (CLI)           → Generate cookbooks, recipes, etc.
├── cookstyle            → Ruby linter for Chef code
├── Test Kitchen         → Testing framework for cookbooks

```

## 7. Recipes — The Heart of Chef

A **recipe** is a Ruby file (`.rb`) that describes the desired state of a node. It's the most fundamental building block in Chef.

### Your First Recipe

```ruby
# my_first_recipe.rb
# This recipe installs Nginx and ensures it's running

# Step 1: Install the nginx package
package 'nginx' do
  action :install
end

# Step 2: Start the nginx service and enable it on boot
service 'nginx' do
  action [:enable, :start]
end

# Step 3: Create a custom index page
file '/var/www/html/index.html' do
  content '<h1>Hello from Chef! 🍳</h1>'
  mode '0644'
  owner 'www-data'
  group 'www-data'
end
```

### Breaking Down the Recipe (Line by Line)

| Code | What It Does |
|------|-------------|
| `package 'nginx' do ... end` | This is a **resource block**. It tells Chef: "I want the package called 'nginx'" |
| `action :install` | The action to take — install the package. If it's already installed, Chef does nothing |
| `service 'nginx' do ... end` | Another resource block — this one manages the Nginx service |
| `action [:enable, :start]` | Two actions: **enable** (start on boot) and **start** (start right now) |
| `file '/var/www/html/index.html' do ... end` | Creates a file with specific content, permissions, and ownership |
| `content '<h1>Hello from Chef! 🍳</h1>'` | The content to put inside the file |
| `mode '0644'` | File permissions (owner: read/write, group: read, others: read) |
| `owner 'www-data'` | The user who owns the file |
| `group 'www-data'` | The group that owns the file |

### Running a Recipe Locally (Chef Solo / Chef Zero)

You don't need a Chef Server to test! Use `chef-client` in local mode:

```bash
# Run a recipe locally (great for testing)
sudo chef-client --local-mode my_first_recipe.rb

# Or using the short form
sudo chef-client -z my_first_recipe.rb
```

**What you'll see:**

```
Starting Chef Infra Client, version 18.3.0
resolving cookbooks for run list: []
Synchronizing Cookbooks:
Installing Cookbook Gems:
Compiling Cookbooks...
Converging 3 resources
Recipe: @recipe_files::/home/user/my_first_recipe.rb
  * apt_package[nginx] action install
    - install version 1.18.0-6ubuntu14 of package nginx
  * service[nginx] action enable
    - enable service service[nginx]
  * service[nginx] action start
    - start service service[nginx]
  * file[/var/www/html/index.html] action create
    - create new file /var/www/html/index.html
    - update content in file /var/www/html/index.html from none to 5d4133

Chef Infra Client finished, 4/4 resources updated in 12 seconds
```

**Run it again — see idempotency in action:**

```
Chef Infra Client finished, 0/4 resources updated in 3 seconds
```

Nothing changed! Everything was already in the desired state. ✅

### Recipe Ordering

Recipes run **top-to-bottom** — the order of resources matters:

```ruby
# ✅ CORRECT — install FIRST, then start
package 'nginx' do
  action :install
end

service 'nginx' do
  action :start
end

# ❌ WRONG — can't start something that isn't installed yet
service 'nginx' do
  action :start
end

package 'nginx' do
  action :install
end
```

## 8. Resources — Building Blocks of Recipes

**Resources** are the individual building blocks of recipes. Each resource describes **one piece** of the system's desired state.

### Resource Syntax

Every resource follows this pattern:

```ruby
resource_type 'name' do
  property1  value1
  property2  value2
  action     :action_name
end
```

### Common Resources (With Examples)

#### 📦 `package` — Install/Remove Software

```ruby
# Install a single package
package 'git' do
  action :install
end

# Install a specific version
package 'nginx' do
  version '1.18.0-6ubuntu14'
  action :install
end

# Install multiple packages at once
%w(curl wget vim htop).each do |pkg|
  package pkg do
    action :install
  end
end

# Remove a package
package 'apache2' do
  action :remove
end
```

#### 📄 `file` — Create/Manage Files

```ruby
# Create a simple file
file '/tmp/hello.txt' do
  content 'Hello from Chef!'
  mode '0644'
  owner 'root'
  group 'root'
  action :create
end

# Delete a file
file '/tmp/unwanted.txt' do
  action :delete
end
```

#### 📁 `directory` — Create/Manage Directories

```ruby
# Create a directory
directory '/opt/myapp' do
  owner 'deploy'
  group 'deploy'
  mode '0755'
  recursive true     # Create parent directories too
  action :create
end
```

#### 🔧 `service` — Manage Services

```ruby
# Start and enable a service
service 'nginx' do
  action [:enable, :start]
end

# Restart a service
service 'nginx' do
  action :restart
end

# Stop and disable a service
service 'apache2' do
  action [:stop, :disable]
end
```

#### 👤 `user` — Manage Users

```ruby
# Create a user
user 'deploy' do
  comment 'Deployment User'
  home '/home/deploy'
  shell '/bin/bash'
  manage_home true    # Create the home directory
  action :create
end

# Remove a user
user 'olduser' do
  action :remove
end
```

#### 👥 `group` — Manage Groups

```ruby
# Create a group and add members
group 'developers' do
  members ['deploy', 'alice', 'bob']
  action :create
end
```

#### ⚡ `execute` — Run Shell Commands

```ruby
# Run a command (use sparingly — prefer specific resources)
execute 'update_apt' do
  command 'apt-get update'
  action :run
end

# Run a command only if a condition is met
execute 'initialize_database' do
  command '/opt/myapp/bin/init_db.sh'
  not_if { ::File.exist?('/opt/myapp/.db_initialized') }
end
```

#### 📋 `cron` — Manage Cron Jobs

```ruby
# Create a cron job
cron 'cleanup_logs' do
  minute '0'
  hour '2'
  command '/usr/local/bin/cleanup_logs.sh'
  user 'root'
  action :create
end
```

#### 🔗 `link` — Create Symbolic Links

```ruby
# Create a symlink
link '/usr/local/bin/myapp' do
  to '/opt/myapp/current/bin/myapp'
  action :create
end
```

### Guards — Conditional Execution

Guards let you control WHEN a resource runs:

```ruby
# only_if — Run this resource ONLY IF the condition is true
package 'nginx' do
  action :install
  only_if { node['platform'] == 'ubuntu' }
end

# not_if — Run this resource ONLY IF the condition is NOT true
execute 'apt-get update' do
  command 'apt-get update'
  not_if { ::File.exist?('/tmp/apt_updated_today') }
end

# Guards can also be shell commands (string)
execute 'create_database' do
  command 'createdb myapp_production'
  not_if 'psql -lqt | cut -d \| -f 1 | grep -qw myapp_production'
end
```

### Notifications — Triggering Actions Between Resources

```ruby
# When the config file changes, restart Nginx
template '/etc/nginx/nginx.conf' do
  source 'nginx.conf.erb'
  notifies :restart, 'service[nginx]', :delayed   # Restart at the END of the run
end

# Or restart IMMEDIATELY
template '/etc/nginx/nginx.conf' do
  source 'nginx.conf.erb'
  notifies :restart, 'service[nginx]', :immediately
end

# subscribes — the reverse (the service watches the template)
service 'nginx' do
  action [:enable, :start]
  subscribes :restart, 'template[/etc/nginx/nginx.conf]', :delayed
end
```

**What's the difference between `:delayed` and `:immediately`?**

| Timing | Behavior |
|--------|----------|
| `:delayed` (default) | The notification runs at the **end** of the Chef run. If multiple resources notify the same service, it only restarts **once** |
| `:immediately` | The notification runs **right away**, before Chef moves to the next resource |

## 9. Cookbooks — Organizing Your Recipes

A **cookbook** is the fundamental unit of configuration in Chef. It's a directory containing recipes, templates, files, attributes, and more.

### Generating a Cookbook

```bash
# Generate a new cookbook
chef generate cookbook my_webserver

# What gets created:
my_webserver/
├── CHANGELOG.md           # Track changes
├── LICENSE                 # License info
├── Policyfile.rb          # Dependencies (modern approach)
├── README.md              # Documentation
├── chefignore             # Files to ignore when uploading
├── compliance/            # InSpec compliance profiles
│   └── ...
├── kitchen.yml            # Test Kitchen configuration
├── metadata.rb            # Cookbook metadata (name, version, deps)
├── recipes/               # Your recipes go here
│   └── default.rb         # The default recipe
├── spec/                  # Unit tests
│   └── ...
└── test/                  # Integration tests
    └── ...
```

### metadata.rb — The Cookbook's Identity Card

```ruby
# metadata.rb
name 'my_webserver'
maintainer 'Your Name'
maintainer_email 'you@example.com'
license 'Apache-2.0'
description 'Installs and configures a web server'
version '1.0.0'

# Dependencies — other cookbooks this one needs
depends 'nginx', '~> 12.0'
depends 'firewall', '>= 2.0'

# Supported platforms
supports 'ubuntu', '>= 20.04'
supports 'centos', '>= 7.0'
```

### Cookbook Structure — A Detailed Look

```
my_webserver/
├── attributes/
│   └── default.rb         # Default attribute values
├── files/
│   └── default/
│       └── motd.txt       # Static files to copy to nodes
├── libraries/
│   └── helpers.rb         # Custom Ruby helpers
├── recipes/
│   ├── default.rb         # Main recipe (runs when cookbook is in run list)
│   ├── install.rb         # Recipe to install packages
│   ├── configure.rb       # Recipe to configure the app
│   └── firewall.rb        # Recipe to set up firewall rules
├── templates/
│   └── default/
│       ├── nginx.conf.erb     # Dynamic config template
│       └── index.html.erb     # Dynamic HTML template
├── test/
│   └── integration/
│       └── default/
│           └── default_test.rb  # Integration tests
├── metadata.rb
├── Policyfile.rb
└── kitchen.yml
```

### A Real Cookbook Example

Let's create a proper `my_webserver` cookbook:

**`recipes/default.rb`** — The main recipe:

```ruby
# recipes/default.rb
# This is the entry point — it calls other recipes in order

include_recipe 'my_webserver::install'
include_recipe 'my_webserver::configure'
include_recipe 'my_webserver::firewall'
```

**`recipes/install.rb`** — Install Nginx:

```ruby
# recipes/install.rb

# Update package cache
apt_update 'update' do
  frequency 86400      # Only update if it's been more than 24 hours
  action :periodic
end

# Install Nginx
package 'nginx' do
  action :install
end

# Install additional useful packages
%w(curl vim htop).each do |pkg|
  package pkg do
    action :install
  end
end
```

**`recipes/configure.rb`** — Configure Nginx:

```ruby
# recipes/configure.rb

# Create the web root directory
directory node['my_webserver']['doc_root'] do
  owner 'www-data'
  group 'www-data'
  mode '0755'
  recursive true
  action :create
end

# Deploy the Nginx configuration
template '/etc/nginx/sites-available/default' do
  source 'default-site.conf.erb'
  owner 'root'
  group 'root'
  mode '0644'
  variables(
    server_name: node['my_webserver']['server_name'],
    doc_root: node['my_webserver']['doc_root'],
    port: node['my_webserver']['port']
  )
  notifies :restart, 'service[nginx]', :delayed
end

# Deploy the index page
template "#{node['my_webserver']['doc_root']}/index.html" do
  source 'index.html.erb'
  owner 'www-data'
  group 'www-data'
  mode '0644'
end

# Ensure Nginx is running and enabled
service 'nginx' do
  action [:enable, :start]
end
```

**`attributes/default.rb`** — Default values:

```ruby
# attributes/default.rb

default['my_webserver']['doc_root'] = '/var/www/mysite'
default['my_webserver']['server_name'] = 'localhost'
default['my_webserver']['port'] = 80
```

**`templates/default/default-site.conf.erb`** — Nginx config template:

```erb
# Managed by Chef — DO NOT EDIT MANUALLY
# Any manual changes will be overwritten on the next Chef run

server {
    listen <%= @port %>;
    server_name <%= @server_name %>;

    root <%= @doc_root %>;
    index index.html;

    location / {
        try_files $uri $uri/ =404;
    }

    access_log /var/log/nginx/<%= @server_name %>-access.log;
    error_log /var/log/nginx/<%= @server_name %>-error.log;
}
```

**What's happening with `<%= @variable %>`?**

| Syntax | Meaning |
|--------|---------|
| `<%= @port %>` | Insert the value of the `port` variable here |
| `<%= @server_name %>` | Insert the value of `server_name` here |
| `<%= @doc_root %>` | Insert the value of `doc_root` here |

These variables come from the `variables()` block in the `template` resource. Chef processes the `.erb` file and replaces these placeholders with real values.

## 10. Knife — Chef's Command-Line Swiss Army Knife

**Knife** is the CLI tool you use to interact with the Chef Server from your workstation. It's how you upload cookbooks, manage nodes, bootstrap servers, and more.

### Setting Up Knife

First, you need a `knife.rb` (or `config.rb`) configuration file:

```ruby
# ~/.chef/config.rb (or knife.rb)

current_dir = File.dirname(__FILE__)

log_level                :info
log_location             STDOUT
node_name                'your_username'
client_key               "#{current_dir}/your_username.pem"
chef_server_url          'https://chef-server.example.com/organizations/myorg'
cookbook_path             ["#{current_dir}/../cookbooks"]
```

### Essential Knife Commands

#### Managing Cookbooks

```bash
# Upload a cookbook to Chef Server
knife cookbook upload my_webserver

# Upload ALL cookbooks
knife cookbook upload --all

# List cookbooks on the server
knife cookbook list

# Show details of a specific cookbook
knife cookbook show my_webserver

# Download a cookbook from the server
knife cookbook download my_webserver

# Delete a cookbook from the server
knife cookbook delete my_webserver
```

#### Managing Nodes

```bash
# List all nodes
knife node list

# Show detailed info about a node
knife node show web-server-1

# Show node info as JSON
knife node show web-server-1 -F json

# Edit a node's run list
knife node run_list add web-server-1 'recipe[my_webserver]'

# Remove a recipe from a node's run list
knife node run_list remove web-server-1 'recipe[my_webserver]'

# Set a node's run list (replace everything)
knife node run_list set web-server-1 'role[webserver],recipe[my_webserver::firewall]'

# Delete a node
knife node delete web-server-1
```

#### Bootstrapping Nodes (Adding New Servers to Chef)

Bootstrapping = installing `chef-client` on a node AND registering it with Chef Server.

```bash
# Bootstrap a new node via SSH
knife bootstrap 192.168.1.100 \
  --ssh-user ubuntu \
  --sudo \
  --identity-file ~/.ssh/my-key.pem \
  --node-name web-server-1 \
  --run-list 'recipe[my_webserver]'

# What this does:
# 1. SSH into 192.168.1.100
# 2. Install chef-client on the server
# 3. Register it with Chef Server as "web-server-1"
# 4. Set its run list to include the my_webserver recipe
# 5. Run chef-client for the first time
```

#### Searching (Super Powerful!)

Knife can search across all your nodes:

```bash
# Find all Ubuntu nodes
knife search node 'platform:ubuntu'

# Find all nodes with Nginx installed
knife search node 'recipes:my_webserver'

# Find all nodes in the production environment
knife search node 'chef_environment:production'

# Find all nodes with more than 8GB RAM
knife search node 'memory_total:*' -a memory.total

# Get the IP addresses of all web servers
knife search node 'role:webserver' -a ipaddress
```

#### Running Commands on Nodes

```bash
# Run a command on all nodes
knife ssh 'name:*' 'uptime' --ssh-user ubuntu

# Run a command on specific nodes
knife ssh 'role:webserver' 'sudo systemctl status nginx' --ssh-user ubuntu

# Run a command on nodes matching a search
knife ssh 'platform:ubuntu AND chef_environment:production' 'df -h' --ssh-user ubuntu
```


## 12. Attributes — Customizing Behavior

**Attributes** are variables that let you customize how recipes behave. They follow a **precedence hierarchy** — some attribute sources override others.

### Attribute Precedence (Low → High)

```
LOWEST PRIORITY                              HIGHEST PRIORITY
     ↓                                              ↓
  default  →  normal  →  override  →  force_override  →  automatic (Ohai)

Where are they defined?
  default:         attributes/default.rb, recipes, environments, roles
  normal:          recipes, JSON node data
  override:        attributes/default.rb, recipes, environments, roles
  force_override:  attributes/default.rb, recipes
  automatic:       Ohai (system info — you can't set these, they're auto-detected)
```

### Defining Attributes

```ruby
# attributes/default.rb

# Use "default" level — can be overridden by roles/environments
default['my_webserver']['port'] = 80
default['my_webserver']['server_name'] = 'localhost'
default['my_webserver']['doc_root'] = '/var/www/mysite'
default['my_webserver']['max_connections'] = 1024

# Use "override" level — harder to override
override['my_webserver']['security']['ssl_enabled'] = true
```

### Using Attributes in Recipes

```ruby
# recipes/configure.rb

# Access attributes using node['cookbook']['attribute']
template '/etc/nginx/sites-available/default' do
  source 'default-site.conf.erb'
  variables(
    port: node['my_webserver']['port'],
    server_name: node['my_webserver']['server_name'],
    doc_root: node['my_webserver']['doc_root']
  )
end

# You can also use attributes in conditionals
if node['my_webserver']['security']['ssl_enabled']
  # Set up SSL configuration
  include_recipe 'my_webserver::ssl'
end
```

### Ohai — Automatic Attributes

**Ohai** automatically detects system info. You can use these in your recipes:

```ruby
# Available automatically — no setup needed!
node['platform']           # → "ubuntu"
node['platform_version']   # → "22.04"
node['hostname']           # → "web-server-1"
node['fqdn']               # → "web-server-1.example.com"
node['ipaddress']          # → "192.168.1.100"
node['memory']['total']    # → "16384MB"
node['cpu']['total']       # → 4
node['os']                 # → "linux"
node['kernel']['release']  # → "5.15.0-76-generic"

# Use Ohai data in your recipes
template '/etc/nginx/nginx.conf' do
  source 'nginx.conf.erb'
  variables(
    worker_processes: node['cpu']['total'],    # Match CPU cores
    hostname: node['hostname']
  )
end
```

## 13. Templates — Dynamic Configuration Files

Templates are **ERB (Embedded Ruby)** files that generate configuration files dynamically. They're way more powerful than static files because they can include variables, logic, and loops.

### Template vs File

```
file resource       → Copies a STATIC file (same content every time)
template resource   → Generates a DYNAMIC file (content changes based on variables)
```

### ERB Syntax Cheat Sheet

```erb
<%= expression %>      → Output the result (insert into the file)
<% code %>             → Execute Ruby code (no output)
<%# comment %>         → Comment (not included in output)
<%= @variable %>       → Output a variable passed from the recipe
<%= node['attr'] %>    → Output a node attribute directly
```

### Example Template

**`templates/default/nginx.conf.erb`:**

```erb
<%# This is managed by Chef — DO NOT EDIT MANUALLY %>

user www-data;
worker_processes <%= @worker_processes %>;
pid /run/nginx.pid;

events {
    worker_connections <%= @max_connections %>;
}

http {
    sendfile on;
    tcp_nopush on;
    keepalive_timeout 65;

    <% if @ssl_enabled %>
    # SSL Configuration
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;
    ssl_prefer_server_ciphers on;
    <% end %>

    <% @upstream_servers.each do |server| %>
    upstream backend {
        server <%= server['host'] %>:<%= server['port'] %>;
    }
    <% end %>

    include /etc/nginx/sites-enabled/*;
}
```

**Using the template in a recipe:**

```ruby
# recipes/configure.rb

template '/etc/nginx/nginx.conf' do
  source 'nginx.conf.erb'
  owner 'root'
  group 'root'
  mode '0644'
  variables(
    worker_processes: node['cpu']['total'],
    max_connections: node['my_webserver']['max_connections'],
    ssl_enabled: node['my_webserver']['security']['ssl_enabled'],
    upstream_servers: [
      { 'host' => '10.0.1.10', 'port' => 8080 },
      { 'host' => '10.0.1.11', 'port' => 8080 },
      { 'host' => '10.0.1.12', 'port' => 8080 }
    ]
  )
  notifies :restart, 'service[nginx]', :delayed
end
```

```ruby
  'my_webserver' => {
    'security' => {
      'ssl_enabled' => true
    }
```

## 15. Data Bags — Storing Global Data

**Data bags** are global JSON data stores that can be accessed by any recipe on any node. They're perfect for storing shared data like user accounts, API keys, or application configurations.

### Creating Data Bags

```bash
# Create a data bag called "users"
knife data bag create users

# Create an item in the data bag
knife data bag create users alice
```

**`data_bags/users/alice.json`:**

```json
{
  "id": "alice",
  "comment": "Alice - Developer",
  "uid": 2001,
  "gid": "developers",
  "home": "/home/alice",
  "shell": "/bin/bash",
  "ssh_keys": [
    "ssh-rsa AAAAB3NzaC1yc2EAAAADAQABAAABAQDf... alice@laptop"
  ]
}
```

**Using data bags in a recipe:**

```ruby
# recipes/users.rb

# Load all items from the "users" data bag
users = data_bag('users')

users.each do |username|
  # Load each user's data
  user_data = data_bag_item('users', username)
  
  # Create the user
  user user_data['id'] do
    comment user_data['comment']
    uid user_data['uid']
    home user_data['home']
    shell user_data['shell']
    manage_home true
    action :create
  end
  
  # Set up SSH authorized keys
  directory "#{user_data['home']}/.ssh" do
    owner user_data['id']
    group user_data['gid']
    mode '0700'
  end
  
  file "#{user_data['home']}/.ssh/authorized_keys" do
    content user_data['ssh_keys'].join("\n")
    owner user_data['id']
    group user_data['gid']
    mode '0600'
  end
end
```

### Encrypted Data Bags (For Secrets)

```bash
# Create a secret key
openssl rand -base64 512 | tr -d '\r\n' > /etc/chef/encrypted_data_bag_secret

# Create an encrypted data bag item
knife data bag create secrets database --secret-file /etc/chef/encrypted_data_bag_secret
```

**`data_bags/secrets/database.json` (encrypted on server):**

```json
{
  "id": "database",
  "password": "super_secret_password_123",
  "host": "db.internal.example.com",
  "port": 5432
}
```

**Using encrypted data bags:**

```ruby
# The decryption happens automatically if the key is on the node
db_secrets = data_bag_item('secrets', 'database')

template '/etc/myapp/database.yml' do
  source 'database.yml.erb'
  variables(
    db_host: db_secrets['host'],
    db_port: db_secrets['port'],
    db_password: db_secrets['password']
  )
  mode '0600'    # Restrictive permissions for secret files!
  owner 'myapp'
end
```

## 17. A Complete Real-World Example

Let's put it all together — a complete cookbook that sets up a production web server:

**Goal:** Set up an Nginx web server with:
- Nginx installed and configured
- Custom website deployed
- Firewall configured (only ports 22, 80, 443 open)
- A deploy user created
- Log rotation configured

```bash
# Generate the cookbook
chef generate cookbook production_webserver
cd production_webserver
```

**`metadata.rb`:**

```ruby
name 'production_webserver'
maintainer 'DevOps Team'
maintainer_email 'devops@company.com'
license 'Apache-2.0'
description 'Sets up a production-ready Nginx web server'
version '1.0.0'

supports 'ubuntu', '>= 20.04'
```

**`attributes/default.rb`:**

```ruby
default['production_webserver']['port'] = 80
default['production_webserver']['server_name'] = node['fqdn']
default['production_webserver']['doc_root'] = '/var/www/production'
default['production_webserver']['deploy_user'] = 'deploy'
default['production_webserver']['max_workers'] = node['cpu']['total']
```

**`recipes/default.rb`:**

```ruby
# recipes/default.rb — The main entry point
# Calls all sub-recipes in the correct order

include_recipe 'production_webserver::users'
include_recipe 'production_webserver::install'
include_recipe 'production_webserver::configure'
include_recipe 'production_webserver::firewall'
```

**`recipes/users.rb`:**

```ruby
# recipes/users.rb — Create the deploy user

  comment 'Deployment user'
  home "/home/#{node['production_webserver']['deploy_user']}"
  shell '/bin/bash'
  manage_home true
  action :create
end

  members [node['production_webserver']['deploy_user']]
  append true
  action :modify
end
```

**`recipes/install.rb`:**

```ruby

  frequency 86400
  action :periodic
end

  action :install
end

# Install useful tools
%w(curl vim htop unzip).each do |pkg|
  package pkg do
    action :install
  end
end
```

**`recipes/configure.rb`:**

```ruby
# recipes/configure.rb — Configure Nginx and deploy website

directory node['production_webserver']['doc_root'] do
  owner 'www-data'
  group 'www-data'
  mode '0755'
  recursive true
end

# Deploy Nginx config
template '/etc/nginx/sites-available/production' do
  source 'site.conf.erb'
  owner 'root'
  group 'root'
  mode '0644'
  variables(
    port: node['production_webserver']['port'],

  to '/etc/nginx/sites-available/production'

  action :delete
  notifies :restart, 'service[nginx]', :delayed

  source 'index.html.erb'
  owner 'www-data'
  group 'www-data'
  mode '0644'

service 'nginx' do
  action [:enable, :start]
```

## 20. Common Gotchas and Best Practices

### ❌ Common Mistakes

```ruby
# ❌ DON'T use execute for everything
execute 'install_nginx' do
  command 'apt-get install -y nginx'
end

# ✅ DO use specific resources
package 'nginx' do
  action :install
end

# ❌ DON'T hardcode values
template '/etc/nginx/nginx.conf' do
  variables(port: 80)   # Hardcoded!
end

# ✅ DO use attributes
template '/etc/nginx/nginx.conf' do
  variables(port: node['my_webserver']['port'])   # Configurable!
end

# ❌ DON'T forget notifications
template '/etc/nginx/nginx.conf' do
  source 'nginx.conf.erb'
  # Forgot to notify nginx to restart! Config changes won't apply until reboot
end

# ✅ DO add notifications
template '/etc/nginx/nginx.conf' do
  source 'nginx.conf.erb'
  notifies :restart, 'service[nginx]', :delayed
end
```

### ✅ Best Practices

1. **Use community cookbooks** — Don't reinvent the wheel
2. **Test everything** — Use Test Kitchen + InSpec + ChefSpec
3. **Use attributes** — Never hardcode values in recipes
4. **Keep recipes small** — One recipe per concern (install, configure, deploy)
5. **Version your cookbooks** — Use semantic versioning
6. **Pin cookbook versions** in environments — Never run untested versions in production
7. **Use `notifies`** — Always notify services when their config files change
8. **Use Policyfile** — It's the modern way to manage dependencies (replaces Berkshelf + roles + environments)
9. **Use encrypted data bags** (or Chef Vault) for secrets — Never put passwords in recipes
10. **Lint your code** — Use `cookstyle` (Chef's Ruby linter)

## 21. When to Choose Chef

### ✅ Choose Chef When:

- Your team is **comfortable with Ruby**
- You're managing **large-scale infrastructure** (1000+ nodes)
- You need a **pull-based model** (nodes self-configure on a schedule)
- You want **excellent testing tools** (Test Kitchen, InSpec, ChefSpec)
- You need **compliance automation** (InSpec is industry-leading)
- You want **fine-grained control** with a real programming language

### ❌ Maybe Skip Chef When:

- Your team **doesn't know Ruby** (consider Ansible — it uses YAML)
- You're managing a **small number of servers** (Ansible is simpler)
- You need a **quick, one-time** setup (Ansible ad-hoc commands are faster)
- You don't want to manage an **agent** on every server (Ansible is agentless)
- You're looking for the **simplest tool** to get started (Ansible wins here)

## 22. Summary

| Concept | What It Is |
|---------|-----------|
| **Chef** | Configuration management tool using Ruby DSL |
| **Recipe** | A Ruby file describing desired state of a node |
| **Resource** | A single piece of config (package, file, service, user) |
| **Cookbook** | A collection of recipes, templates, attributes, files |
| **Knife** | CLI tool to manage Chef Server (upload, bootstrap, search) |
| **Node** | A server/machine managed by Chef |
| **Attribute** | Variables that customize recipe behavior |
| **Template** | ERB file that generates dynamic config files |
| **Role** | A bundle of recipes + attributes for a server type |
| **Environment** | Represents dev/staging/production |
| **Data Bag** | Global JSON data store (users, secrets, configs) |
| **Ohai** | Auto-detects system info (OS, IP, CPU, memory) |
| **Chef Supermarket** | Community repository of cookbooks |
| **Convergence** | Making a node match its desired state |
| **Idempotency** | Running the same recipe multiple times = same result |

### The Chef Workflow in a Nutshell

```
1. Write cookbooks/recipes on your WORKSTATION
2. Test locally with Chef Zero + Test Kitchen
3. Upload to CHEF SERVER with knife
4. Bootstrap NODES to register with Chef Server
5. Nodes run chef-client on a schedule (every 30 min)
6. Each run: pull recipes → check current state → converge → report back
7. Repeat forever — your infrastructure stays consistent! 🎉
```
