## 1. What is Puppet?

**Puppet** is a **configuration management tool** that uses its own **declarative language (Puppet DSL)** to describe the desired state of your infrastructure. You write what you WANT your servers to look like, and Puppet figures out HOW to get there.

### The Big Idea

Instead of writing step-by-step instructions ("install this, then copy that, then start this"), you **declare** the end result you want. Puppet reads your declaration and makes it happen.

> Think of Puppet like **ordering food at a restaurant** 🍽️ — you say "I want a cheeseburger with fries" (the desired state), and the kitchen figures out all the steps to make it. You don't tell them "heat the grill to 400°F, take the beef, form a patty..."

### Declarative vs Imperative — The Core Difference

```
IMPERATIVE (like a shell script — step by step):
  1. Run apt-get update
  2. Run apt-get install nginx
  3. Copy nginx.conf to /etc/nginx/
  4. Run systemctl enable nginx
  5. Run systemctl start nginx

DECLARATIVE (like Puppet — describe the end state):
  "I want nginx installed, the config file at /etc/nginx/nginx.conf,
   and the nginx service running and enabled."
  
  → Puppet figures out the steps automatically
  → If nginx is ALREADY installed, it skips that step
  → If the config file ALREADY matches, it skips that too
  → Only does what's NEEDED
```

### Key Facts

| Feature | Details |
|---------|---------|
| **Created By** | Luke Kanies (2005), now maintained by Perforce (via Puppet, Inc.) |
| **Written In** | Ruby & Clojure (Puppet Server) |
| **Configuration Language** | Puppet DSL (its own declarative language) |
| **Model** | Pull-based (agents pull config from Puppet Server) |
| **Agent Required?** | Yes — `puppet-agent` runs on every managed node |
| **Communication** | HTTPS with SSL certificates (mutual TLS) |
| **Open Source?** | Yes — Puppet Community Edition (Apache 2.0) |
| **Commercial Version** | Puppet Enterprise (PE) |
| **Default Run Interval** | Every 30 minutes |

## 2. Why Puppet? — The Problem It Solves

### The Story

```
Scenario: You manage 500 servers across 3 environments (dev, staging, production).

Without Puppet:
  → You write Bash scripts to configure servers
  → Script A works on Ubuntu but breaks on CentOS
  → 3 months later, servers have drifted — some have different configs
  → A new team member joins — "Where's the documentation?"
  → You: "It's in... various scripts... spread across... multiple repos... 😅"
  → Someone manually changed something on server-217 → now it's different from the rest
  → Nobody knows why server-217 is special
  → Production incident at 3 AM → "Why is this server different?!"

With Puppet:
  → You DECLARE the desired state in Puppet code
  → Works on Ubuntu AND CentOS (Puppet handles the differences)
  → Every 30 minutes, Puppet checks every server and FIXES any drift
  → Someone manually changes server-217 → Puppet reverts it in 30 min
  → New team member joins → "Read the Puppet code — that IS the documentation"
  → Production at 3 AM → Every server is identical, as declared in code ✅
```

### Key Benefits

| Benefit | Explanation |
|---------|------------|
| **Declarative** | Describe WHAT you want, not HOW to get there — Puppet handles the details |
| **Idempotent** | Run it 1000 times — same result. Only changes what needs changing |
| **Self-Healing** | If someone manually changes a server, Puppet reverts it on the next run |
| **Cross-Platform** | Same code works on Linux (Ubuntu, CentOS, RHEL) and Windows |
| **Scalable** | Manages 10 servers or 10,000+ servers with the same ease |
| **Mature & Proven** | Been around since 2005 — battle-tested in enterprises worldwide |
| **Strong Ecosystem** | Puppet Forge has 7,000+ community modules |

## 3. Puppet vs Ansible vs Chef — Quick Comparison

| Feature | Puppet | Ansible | Chef |
|---------|--------|---------|------|
| **Language** | Puppet DSL (custom) | YAML | Ruby DSL |
| **Approach** | Declarative | Procedural (mostly) | Procedural (Ruby) |
| **Architecture** | Pull (agent-based) | Push (agentless, SSH) | Pull (agent-based) |
| **Learning Curve** | Moderate (must learn Puppet DSL) | Gentle (YAML is simple) | Steep (must know Ruby) |
| **Agent Required** | Yes (`puppet-agent`) | No (just SSH) | Yes (`chef-client`) |
| **Central Server** | Puppet Server | Not required | Chef Server |
| **Best For** | Enterprise compliance, large infra | Quick automation, any team | Complex infra, Ruby shops |
| **Reporting/UI** | Puppet Enterprise Console | AWX/Tower | Chef Automate |
| **Community** | Puppet Forge (7,000+ modules) | Ansible Galaxy | Chef Supermarket |
| **First Released** | 2005 | 2012 | 2009 |

### When to Pick Puppet Over Others?

- You need **enterprise-grade compliance and reporting**
- You're managing a **very large infrastructure** (thousands of nodes)
- You want **true declarative configuration** (model-driven, not procedural)
- You need **automatic drift detection and remediation** (self-healing)
- Your organization is already in the **Puppet ecosystem**
- You need **strong Windows support** alongside Linux

## 4. How Puppet Works — Architecture

### The Two Main Components

```
┌──────────────────────────────────────────────────────────────────┐
│                       PUPPET ARCHITECTURE                        │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌───────────────────────────────────────────────┐               │
│  │              PUPPET SERVER (Primary)           │               │
│  │                                               │               │
│  │  What it stores:                              │               │
│  │  ├── Manifests (Puppet code)                  │               │
│  │  ├── Modules (organized code bundles)         │               │
│  │  ├── Hiera data (configuration data)          │               │
│  │  ├── Node classifications                     │               │
│  │  ├── SSL Certificates (CA)                    │               │
│  │  └── File server (templates, static files)    │               │
│  │                                               │               │
│  │  What it does:                                │               │
│  │  ├── Compiles CATALOGS for each node          │               │
│  │  ├── Serves files and templates               │               │
│  │  └── Collects REPORTS from nodes              │               │
│  └───────────────────┬───────────────────────────┘               │
│                      │                                           │
│         Every 30 min │ (HTTPS / SSL mutual auth)                 │
│                      │                                           │
│          ┌───────────┼───────────┬───────────┐                   │
│          ▼           ▼           ▼           ▼                   │
│    ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐          │
│    │  NODE 1  │ │  NODE 2  │ │  NODE 3  │ │  NODE N  │          │
│    │          │ │          │ │          │ │          │          │
│    │ puppet-  │ │ puppet-  │ │ puppet-  │ │ puppet-  │          │
│    │ agent    │ │ agent    │ │ agent    │ │ agent    │          │
│    │          │ │          │ │          │ │          │          │
│    │ facter   │ │ facter   │ │ facter   │ │ facter   │          │
│    └──────────┘ └──────────┘ └──────────┘ └──────────┘          │
│                                                                  │
│    Each node has an AGENT that:                                  │
│    1. Collects system facts (Facter)                             │
│    2. Sends facts to Puppet Server                               │
│    3. Receives a compiled CATALOG                                │
│    4. Applies the catalog (converges)                            │
│    5. Sends a REPORT back to the server                          │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### The Puppet Run — Step by Step

```
Step 1: FACTER runs on the node
        → Collects system facts: OS, IP, hostname, CPU, RAM, etc.
        → "I'm Ubuntu 22.04, my IP is 10.0.1.50, I have 4 CPUs..."
            ↓
Step 2: AGENT sends facts to PUPPET SERVER
        → "Hey Server, here are my facts. What should I look like?"
            ↓
Step 3: Puppet Server COMPILES a CATALOG
        → Looks at the node's name, facts, and classifications
        → Reads the manifests and modules
        → Generates a CATALOG — a JSON document describing the
           EXACT desired state for THIS specific node
            ↓
Step 4: Server sends the CATALOG back to the agent
        → "Here's your catalog — this is what you should look like"
            ↓
Step 5: AGENT applies the catalog
        → Compares current state vs desired state
        → Makes ONLY the changes needed (idempotent!)
        → Installs packages, creates files, starts services, etc.
            ↓
Step 6: Agent sends a REPORT to the server
        → "I changed 3 things, 0 failures, here's what happened"
            ↓
Step 7: WAIT 30 minutes, then repeat from Step 1
```

### What is a Catalog?

A **catalog** is a compiled, node-specific document that tells the agent exactly what the desired state is. Think of it as a personalized to-do list for that specific server.

```
Puppet Code (human-readable)
    ↓ Compiled by Server
Catalog (machine-readable JSON)
    ↓ Applied by Agent
Actual Server State
```

The key insight: **the Server does the thinking (compiling), the Agent does the doing (applying).**

## 5. Puppet Key Terminology

| Term | What It Means | Analogy |
|------|--------------|---------|
| **Manifest** | A `.pp` file containing Puppet code | A blueprint |
| **Resource** | A single unit of configuration (package, file, service) | A single instruction on the blueprint |
| **Class** | A named block of Puppet code that can be included | A chapter in the blueprint |
| **Module** | A self-contained bundle of manifests, templates, files, data | A complete blueprint package |
| **Node** | A server/machine managed by Puppet | A building being constructed |
| **Catalog** | Compiled desired state for a specific node | The finalized construction plan |
| **Facter** | Tool that collects system facts from nodes | A building inspector |
| **Hiera** | Data lookup tool — separates data from code | A reference manual |
| **Puppet Server** | Central server that compiles catalogs | The architect's office |
| **Puppet Agent** | Software running on each managed node | The construction crew |
| **Puppet Forge** | Community repository of pre-built modules | A building supply store |
| **Environment** | A set of modules/manifests (dev, staging, prod) | Different construction projects |
| **Report** | Summary of what happened during a Puppet run | Construction progress report |
| **Certificate** | SSL cert used for agent ↔ server authentication | The crew's ID badge |

## 6. Installing Puppet

### Installing Puppet Server (The Primary/Master)

```bash
# On Ubuntu 22.04

# Step 1: Add the Puppet repository
wget https://apt.puppet.com/puppet8-release-jammy.deb
sudo dpkg -i puppet8-release-jammy.deb
sudo apt-get update

# Step 2: Install Puppet Server
sudo apt-get install -y puppetserver

# Step 3: Configure memory allocation (JVM)
# Edit /etc/default/puppetserver
# Default is 2GB — adjust based on your server's RAM
sudo sed -i 's/JAVA_ARGS="-Xms2g -Xmx2g"/JAVA_ARGS="-Xms1g -Xmx1g"/' /etc/default/puppetserver

# Step 4: Start Puppet Server
sudo systemctl start puppetserver
sudo systemctl enable puppetserver

# Step 5: Verify
sudo systemctl status puppetserver
sudo /opt/puppetlabs/bin/puppetserver --version
```

### Installing Puppet Agent (On Managed Nodes)

```bash
# On Ubuntu 22.04

# Step 1: Add the Puppet repository
wget https://apt.puppet.com/puppet8-release-jammy.deb
sudo dpkg -i puppet8-release-jammy.deb
sudo apt-get update

# Step 2: Install the Puppet Agent
sudo apt-get install -y puppet-agent

# Step 3: Configure the agent to point to your Puppet Server
sudo /opt/puppetlabs/bin/puppet config set server puppet-server.example.com --section main

# Step 4: Start the agent
sudo systemctl start puppet
sudo systemctl enable puppet

# Step 5: Verify
sudo /opt/puppetlabs/bin/puppet --version
```

### Certificate Signing (Important!)

Puppet uses mutual TLS — both the server and agent must trust each other via SSL certificates.

```bash
# On the AGENT — trigger a certificate request
sudo /opt/puppetlabs/bin/puppet agent --test --waitforcert 60

# On the SERVER — list pending certificate requests
sudo /opt/puppetlabs/bin/puppetserver ca list

# On the SERVER — sign the agent's certificate
sudo /opt/puppetlabs/bin/puppetserver ca sign --certname web-server-1.example.com

# Or sign ALL pending requests (be careful in production!)
sudo /opt/puppetlabs/bin/puppetserver ca sign --all

# Now the agent can communicate with the server ✅
```

### Using Puppet Without a Server (Puppet Apply)

For learning and testing, you can run Puppet locally without a server:

```bash
# Run a manifest directly on the local machine
sudo /opt/puppetlabs/bin/puppet apply my_manifest.pp

# Run in dry-run mode (show what WOULD change, but don't actually change it)
sudo /opt/puppetlabs/bin/puppet apply my_manifest.pp --noop
```

## 7. Puppet DSL — The Language

The Puppet DSL (Domain Specific Language) is a **declarative language** designed specifically for describing infrastructure. It's NOT Ruby, it's NOT YAML — it's its own thing.

### Basic Syntax

```puppet
# This is a comment

# A resource declaration
resource_type { 'title':
  property1 => value1,
  property2 => value2,
}

# Example: Install a package
package { 'nginx':
  ensure => installed,
}

# Example: Manage a service
service { 'nginx':
  ensure => running,
  enable => true,
}

# Example: Create a file
file { '/etc/motd':
  ensure  => file,
  content => "Welcome to this server!\n",
  owner   => 'root',
  group   => 'root',
  mode    => '0644',
}
```

### Puppet Syntax Rules

```
1. Resources use curly braces { }
2. Properties use fat arrow =>  (not = or :)
3. Strings use single quotes 'hello' or double quotes "hello ${variable}"
4. Each property ends with a COMMA (including the last one — it's optional but good practice)
5. Resource titles must be UNIQUE within a catalog
6. Everything is CASE-SENSITIVE
7. Manifests use the .pp file extension
```

## 8. Resources — The Atomic Unit

A **resource** is the fundamental building block of Puppet. Each resource describes ONE piece of the system.

### Resource Syntax

```puppet
resource_type { 'title':
  attribute1 => value1,
  attribute2 => value2,
  ...
}
```

### Common Resource Types

#### 📦 `package` — Install/Remove Software

```puppet
# Install a package
package { 'nginx':
  ensure => installed,        # or 'present', 'latest', '1.18.0', 'absent'
}

# Install a specific version
package { 'docker-ce':
  ensure => '24.0.7',
}

# Remove a package
package { 'apache2':
  ensure => absent,
}

# Install multiple packages (array of titles)
package { ['curl', 'wget', 'vim', 'htop', 'git']:
  ensure => installed,
}
```

#### 📄 `file` — Manage Files and Directories

```puppet
# Create a file with inline content
file { '/etc/motd':
  ensure  => file,
  content => "Welcome to ${facts['hostname']}!\nManaged by Puppet.\n",
  owner   => 'root',
  group   => 'root',
  mode    => '0644',
}

# Create a file from a source (static file)
file { '/etc/nginx/nginx.conf':
  ensure => file,
  source => 'puppet:///modules/my_webserver/nginx.conf',
  owner  => 'root',
  group  => 'root',
  mode   => '0644',
}

# Create a file from a template (dynamic content)
file { '/etc/nginx/sites-available/default':
  ensure  => file,
  content => epp('my_webserver/site.conf.epp'),
  owner   => 'root',
  group   => 'root',
  mode    => '0644',
}

# Create a directory
file { '/opt/myapp':
  ensure => directory,
  owner  => 'deploy',
  group  => 'deploy',
  mode   => '0755',
}

# Create a symlink
file { '/usr/local/bin/myapp':
  ensure => link,
  target => '/opt/myapp/current/bin/myapp',
}

# Delete a file
file { '/tmp/unwanted.txt':
  ensure => absent,
}

# Manage an entire directory tree recursively
file { '/opt/myapp/config':
  ensure  => directory,
  source  => 'puppet:///modules/my_webserver/config/',
  recurse => true,
  purge   => true,         # Remove files not managed by Puppet
  owner   => 'deploy',
  group   => 'deploy',
}
```

#### 🔧 `service` — Manage Services

```puppet
# Ensure a service is running and enabled on boot
service { 'nginx':
  ensure => running,
  enable => true,
}

# Stop and disable a service
service { 'apache2':
  ensure => stopped,
  enable => false,
}
```

#### 👤 `user` — Manage Users

```puppet
# Create a user
user { 'deploy':
  ensure     => present,
  comment    => 'Deployment User',
  home       => '/home/deploy',
  shell      => '/bin/bash',
  managehome => true,
  groups     => ['www-data', 'sudo'],
}

# Remove a user
user { 'olduser':
  ensure => absent,
}
```

#### 👥 `group` — Manage Groups

```puppet
# Create a group
group { 'developers':
  ensure => present,
  gid    => 3000,
}
```

#### ⚡ `exec` — Run Commands (Use Sparingly!)

```puppet
# Run a command
exec { 'apt-update':
  command => '/usr/bin/apt-get update',
  # Run only if the file doesn't exist (guard)
}

# Run a command with conditions
exec { 'initialize-database':
  command => '/opt/myapp/bin/init_db.sh',
  creates => '/opt/myapp/.db_initialized',   # Only run if this file doesn't exist
  user    => 'myapp',
  path    => ['/usr/bin', '/usr/local/bin'],
}

# Run a command only when triggered by another resource
exec { 'nginx-reload':
  command     => '/usr/sbin/nginx -s reload',
  refreshonly => true,    # Only runs when notified by another resource
}
```

#### 📋 `cron` — Manage Cron Jobs

```puppet
# Create a cron job
cron { 'cleanup-logs':
  ensure  => present,
  command => '/usr/local/bin/cleanup_logs.sh',
  user    => 'root',
  hour    => 2,
  minute  => 0,
}
```

### The `ensure` Property

Almost every resource has an `ensure` property. It defines what state the resource should be in:

| Resource Type | `ensure` Values |
|--------------|----------------|
| `package` | `installed` (or `present`), `latest`, `absent`, specific version |
| `file` | `file`, `directory`, `link`, `absent` |
| `service` | `running`, `stopped` |
| `user` | `present`, `absent` |
| `group` | `present`, `absent` |
| `cron` | `present`, `absent` |

## 9. Manifests — Writing Your Infrastructure Code

A **manifest** is a file with the `.pp` extension that contains Puppet code. It's where you write your resource declarations.

### Your First Manifest

```puppet
# webserver.pp
# This manifest sets up a basic Nginx web server

# Step 1: Install Nginx
package { 'nginx':
  ensure => installed,
}

# Step 2: Deploy the configuration
file { '/etc/nginx/sites-available/default':
  ensure  => file,
  content => "server {
    listen 80;
    server_name localhost;
    root /var/www/html;
    index index.html;
}\n",
  owner   => 'root',
  group   => 'root',
  mode    => '0644',
  require => Package['nginx'],          # Don't run until nginx is installed
  notify  => Service['nginx'],          # Restart nginx if this file changes
}

# Step 3: Create the website content
file { '/var/www/html/index.html':
  ensure  => file,
  content => "<h1>Hello from Puppet!</h1>\n<p>Managed by Puppet on ${facts['hostname']}</p>\n",
  owner   => 'www-data',
  group   => 'www-data',
  mode    => '0644',
  require => Package['nginx'],
}

# Step 4: Ensure Nginx is running
service { 'nginx':
  ensure    => running,
  enable    => true,
  require   => Package['nginx'],        # Don't start until nginx is installed
  subscribe => File['/etc/nginx/sites-available/default'],  # Restart if config changes
}
```

### Running the Manifest

```bash
# Apply the manifest locally
sudo /opt/puppetlabs/bin/puppet apply webserver.pp

# Dry-run (see what WOULD change without actually changing anything)
sudo /opt/puppetlabs/bin/puppet apply webserver.pp --noop

# Verbose output
sudo /opt/puppetlabs/bin/puppet apply webserver.pp --verbose

# Debug output (very detailed)
sudo /opt/puppetlabs/bin/puppet apply webserver.pp --debug
```

**Output:**

```
Notice: Compiled catalog for web-server-1 in environment production in 0.5 seconds
Notice: /Stage[main]/Main/Package[nginx]/ensure: created
Notice: /Stage[main]/Main/File[/etc/nginx/sites-available/default]/content: content changed
Notice: /Stage[main]/Main/File[/var/www/html/index.html]/ensure: defined content as '{md5}a1b2c3...'
Notice: /Stage[main]/Main/Service[nginx]/ensure: ensure changed 'stopped' to 'running'
Notice: Applied catalog in 8.42 seconds
```

### The Site Manifest (`site.pp`)

On a Puppet Server, the main entry point is `site.pp` (usually at `/etc/puppetlabs/code/environments/production/manifests/site.pp`):

```puppet
# site.pp — The main entry point for Puppet Server

# Default node — applies to ANY node not specifically listed
node default {
  notify { 'Hello': message => 'This node is managed by Puppet!' }
}

# Specific node definitions
node 'web-server-1.example.com' {
  include my_webserver
  include monitoring::node_exporter
}

node 'db-server-1.example.com' {
  include postgresql
  include monitoring::node_exporter
}

# Pattern matching (regex)
node /^web-.*\.example\.com$/ {
  include my_webserver
  include security::hardening
}
```

## 10. Classes — Organizing Your Code

A **class** is a named block of Puppet code that groups related resources together. Think of it like a function — you define it once, then "include" it wherever you need it.

### Defining a Class

```puppet
# Define a class
class my_webserver {
  package { 'nginx':
    ensure => installed,
  }

  service { 'nginx':
    ensure  => running,
    enable  => true,
    require => Package['nginx'],
  }
  
  file { '/var/www/html/index.html':
    ensure  => file,
    content => "<h1>Hello from Puppet!</h1>\n",
    owner   => 'www-data',
    group   => 'www-data',
    require => Package['nginx'],
  }
}
```

### Using (Including) a Class

```puppet
# Method 1: include (most common)
include my_webserver

# Method 2: require (like include, but adds ordering)
require my_webserver

# Method 3: contain (for use inside other classes)
contain my_webserver

# Method 4: class declaration with parameters
class { 'my_webserver':
  port        => 8080,
  server_name => 'mysite.com',
}
```

### Parameterized Classes (Super Useful!)

```puppet
# Define a class with parameters
class my_webserver (
  Integer $port        = 80,
  String  $server_name = 'localhost',
  String  $doc_root    = '/var/www/html',
  Boolean $ssl_enabled = false,
) {

  package { 'nginx':
    ensure => installed,
  }

  file { '/etc/nginx/sites-available/default':
    ensure  => file,
    content => epp('my_webserver/site.conf.epp', {
      'port'        => $port,
      'server_name' => $server_name,
      'doc_root'    => $doc_root,
    }),
    require => Package['nginx'],
    notify  => Service['nginx'],
  }

  if $ssl_enabled {
    # Include SSL configuration
    include my_webserver::ssl
  }

  service { 'nginx':
    ensure => running,
    enable => true,
  }
}
```

## 11. Modules — The Building Blocks

A **module** is a self-contained, reusable unit of Puppet code. It bundles manifests, templates, files, data, and tests into a standardized directory structure.

### Module Structure

```
my_webserver/
├── manifests/                 # Puppet code (.pp files)
│   ├── init.pp               # Main class (class my_webserver)
│   ├── install.pp            # class my_webserver::install
│   ├── config.pp             # class my_webserver::config
│   └── service.pp            # class my_webserver::service
├── templates/                 # Template files
│   ├── nginx.conf.epp        # EPP template (modern)
│   └── site.conf.erb         # ERB template (legacy)
├── files/                     # Static files
│   └── custom_error_page.html
├── lib/                       # Custom Puppet functions and facts
│   ├── facter/               # Custom facts
│   └── puppet/

├── data/                      # Hiera data specific to this module
│   └── common.yaml
├── examples/                  # Usage examples
│   └── init.pp
├── spec/                      # Tests
│   ├── classes/
│   │   └── my_webserver_spec.rb
│   └── spec_helper.rb
├── metadata.json              # Module metadata
├── README.md                  # Documentation
├── CHANGELOG.md               # Change history
└── Gemfile                    # Ruby dependencies for testing
```

### File Naming Convention (Critical!)

The mapping between **class names** and **file paths** is strict:

```
Class Name                    → File Path
─────────────────────────────────────────────────

my_webserver::firewall::rules → manifests/firewall/rules.pp
```

> ⚠️ **This is the #1 source of errors for Puppet beginners!** If the class name doesn't match the file path, Puppet can't find it.

## 16. Templates — Dynamic File Content

Templates let you generate configuration files dynamically, with variables, loops, and conditionals embedded.

### EPP Templates (Modern — Recommended)

EPP (Embedded Puppet) templates use Puppet's own syntax:

**`templates/nginx.conf.epp`:**

```epp
<%- | 
  String  $server_name,
  Integer $port,
  String  $doc_root,
  Integer $worker_processes,
  Boolean $ssl_enabled = false,
  Array[Hash] $upstream_servers = [],
| -%>
# Managed by Puppet — DO NOT EDIT MANUALLY
# Any manual changes will be overwritten on the next Puppet run

user www-data;
worker_processes <%= $worker_processes %>;
pid /run/nginx.pid;

events {
    worker_connections 1024;
}

http {
    sendfile on;
    tcp_nopush on;
    keepalive_timeout 65;

<% if $ssl_enabled { -%>
    # SSL Configuration
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;
    ssl_prefer_server_ciphers on;
<% } -%>

<% unless $upstream_servers.empty { -%>
    upstream backend {
<%   $upstream_servers.each |$server| { -%>
        server <%= $server['host'] %>:<%= $server['port'] %>;
<%   } -%>
    }
<% } -%>

    server {
        listen <%= $port %>;
        server_name <%= $server_name %>;
        root <%= $doc_root %>;
        index index.html;
    }
}
```

**Using the EPP template in a manifest:**

```puppet
file { '/etc/nginx/nginx.conf':
  ensure  => file,
  content => epp('my_webserver/nginx.conf.epp', {
    'server_name'      => $server_name,
    'port'             => $port,
    'doc_root'         => $doc_root,
    'worker_processes' => $facts['processors']['count'],
    'ssl_enabled'      => $ssl_enabled,
    'upstream_servers'  => [
      { 'host' => '10.0.1.10', 'port' => 8080 },
      { 'host' => '10.0.1.11', 'port' => 8080 },
    ],
  }),
  notify => Service['nginx'],
}
```

### EPP Syntax Cheat Sheet

```epp
<%= $variable %>           → Output a variable's value
<% puppet_code %>          → Execute Puppet code (no output)
<%# comment %>             → Comment (not in output)
<%- ... -%>                → Trim whitespace/newlines around tags
```

## 17. Hiera — Separating Data from Code

**Hiera** is Puppet's built-in data lookup system. It lets you **separate your data (values) from your code (logic)** — which is a HUGE best practice.

### Why Hiera?

```
WITHOUT Hiera (data mixed with code — messy! 😩):
  class { 'my_webserver':
    port => 443,                        # Hardcoded in the manifest
    server_name => 'prod.example.com',  # Mixed with code
    ssl_enabled => true,                # Hard to manage per-environment
  }

WITH Hiera (data separated from code — clean! ✅):
  Manifest: include my_webserver        # Just includes the class
  Hiera:    my_webserver::port: 443     # Data lives in YAML files
            my_webserver::server_name: prod.example.com
            my_webserver::ssl_enabled: true
```

### How Hiera Works

```
Puppet asks:  "What's the value of my_webserver::port?"
              ↓
Hiera searches through a HIERARCHY of YAML files:
              ↓
  1. nodes/web-server-1.yaml     → Not found, try next level
  2. environments/production.yaml → Found! port: 443
  3. common.yaml                  → (Would have checked this if not found above)
              ↓
Returns: 443
```

### Hiera Configuration

**`hiera.yaml`** (at the environment level):

```yaml
# hiera.yaml
---
version: 5

defaults:
  datadir: data
  data_hash: yaml_data

hierarchy:
  - name: "Per-node data"
    path: "nodes/%{trusted.certname}.yaml"
  
  - name: "Per-OS data"
    path: "os/%{facts.os.family}.yaml"
  
  - name: "Per-environment data"
    path: "environments/%{environment}.yaml"
  
  - name: "Common data"
    path: "common.yaml"
```

### Hiera Data Files

```yaml
# data/common.yaml (applies to ALL nodes)
---
my_webserver::port: 80
my_webserver::server_name: 'localhost'
my_webserver::ssl_enabled: false
my_webserver::packages:
  - nginx
  - curl
  - vim

# data/environments/production.yaml
---
my_webserver::port: 443
my_webserver::server_name: 'www.example.com'
my_webserver::ssl_enabled: true

# data/nodes/web-server-1.example.com.yaml
---
my_webserver::server_name: 'special-server.example.com'
my_webserver::max_connections: 4096
```

### Automatic Parameter Lookup

Hiera integrates directly with class parameters. If your class has a parameter `$port`, Hiera will automatically look up `<class_name>::port`:

```puppet
# The class definition (in the module)
class my_webserver (
  Integer $port        = 80,
  String  $server_name = 'localhost',
  Boolean $ssl_enabled = false,
) {
  # ...
}

# In site.pp — just include, no parameters needed!
include my_webserver

# Hiera automatically looks up:
#   my_webserver::port
#   my_webserver::server_name
#   my_webserver::ssl_enabled
# And passes them to the class! Magic! ✨
```

### Manual Hiera Lookups

```puppet
# In Puppet code — explicitly look up a value
$db_password = lookup('database::password')

# With a default value
$port = lookup('my_webserver::port', Integer, 'first', 80)

# Look up from the command line

```

## 19. Relationships and Ordering

By default, Puppet applies resources in **no guaranteed order**. You need to explicitly define relationships when order matters.

### Relationship Metaparameters

```puppet
# require — "I need this OTHER resource to be applied FIRST"
service { 'nginx':
  ensure  => running,
  require => Package['nginx'],    # Install nginx BEFORE starting the service
}

# before — "I should be applied BEFORE this other resource"
package { 'nginx':
  ensure => installed,
  before => Service['nginx'],     # Same as above, but from the package's perspective
}

# notify — "When I change, trigger a refresh on this other resource"
file { '/etc/nginx/nginx.conf':
  ensure => file,
  source => 'puppet:///modules/my_webserver/nginx.conf',
  notify => Service['nginx'],     # Restart nginx when config changes
}

# subscribe — "Watch this other resource, and refresh me when it changes"
service { 'nginx':
  ensure    => running,
  subscribe => File['/etc/nginx/nginx.conf'],   # Same as notify, but reversed
}
```

### Chaining Arrows (Concise Syntax)

```puppet
# -> means "before" (ordering)
Package['nginx'] -> File['/etc/nginx/nginx.conf'] -> Service['nginx']

# ~> means "notify" (ordering + trigger refresh)
Package['nginx'] -> File['/etc/nginx/nginx.conf'] ~> Service['nginx']

# Read it like this:
# Install nginx THEN deploy config THEN (re)start the service
# If the config changes, NOTIFY the service to restart
```

### The Classic Pattern: Package → Config → Service

```puppet
# This is the most common pattern in Puppet
package { 'nginx':
  ensure => installed,
}

file { '/etc/nginx/nginx.conf':
  ensure  => file,
  source  => 'puppet:///modules/my_webserver/nginx.conf',
  require => Package['nginx'],
  notify  => Service['nginx'],
}

service { 'nginx':
  ensure    => running,
  enable    => true,
  require   => Package['nginx'],
  subscribe => File['/etc/nginx/nginx.conf'],
}
```

## 20. Defined Types — Reusable Resource Wrappers

A **defined type** is like a class, but you can declare it **multiple times** with different parameters. Think of it as a reusable template for a set of resources.

### Example: Managing Multiple Virtual Hosts

```puppet
# manifests/vhost.pp
define my_webserver::vhost (
  Integer $port     = 80,
  String  $doc_root = "/var/www/${title}",
  String  $ensure   = 'present',
) {
  # $title is automatically set to the resource title

  # Create document root
  file { $doc_root:
    ensure => directory,
    owner  => 'www-data',
    group  => 'www-data',
    mode   => '0755',
  }

  # Create vhost config
  file { "/etc/nginx/sites-available/${title}.conf":
    ensure  => file,
    content => epp('my_webserver/vhost.conf.epp', {
      'server_name' => $title,
      'port'        => $port,
      'doc_root'    => $doc_root,
    }),
    notify  => Service['nginx'],
  }

  # Enable the site (symlink)
  file { "/etc/nginx/sites-enabled/${title}.conf":
    ensure  => link,
    target  => "/etc/nginx/sites-available/${title}.conf",
    notify  => Service['nginx'],
  }
}
```

**Using the defined type (multiple times!):**

```puppet
# Create multiple virtual hosts
my_webserver::vhost { 'blog.example.com':
  port => 80,
}

my_webserver::vhost { 'api.example.com':
  port     => 8080,
  doc_root => '/opt/api/public',
}

my_webserver::vhost { 'admin.example.com':
  port => 443,
}
```

## 22. A Complete Real-World Example

Let's build a complete Puppet module for a production web server:

**Module: `my_webserver`**

**`manifests/init.pp`:**

```puppet
# Main class — entry point
class my_webserver (
  Integer            $port           = 80,
  String             $server_name    = $facts['fqdn'],
  String             $doc_root       = '/var/www/production',
  Boolean            $ssl_enabled    = false,
  String             $deploy_user    = 'deploy',
  Array[String]      $extra_packages = [],
) {
  contain my_webserver::install
  contain my_webserver::config
  contain my_webserver::service

  Class['my_webserver::install']
  -> Class['my_webserver::config']
  ~> Class['my_webserver::service']
}
```

**`manifests/install.pp`:**

```puppet
# Installation class
class my_webserver::install {
  # Ensure package list is up to date (Debian/Ubuntu only)
  if $facts['os']['family'] == 'Debian' {
    exec { 'apt-update':
      command => '/usr/bin/apt-get update',
      unless  => "test $(( $(date +%s) - $(stat -c %Y /var/lib/apt/lists/) )) -lt 86400",
    }
  }

  # Install Nginx
  package { 'nginx':
    ensure => installed,
  }

  # Install additional packages
  $base_packages = ['curl', 'vim', 'htop', 'unzip']
  $all_packages = $base_packages + $my_webserver::extra_packages

  package { $all_packages:
    ensure => installed,
  }

  # Create deploy user
  user { $my_webserver::deploy_user:
    ensure     => present,
    comment    => 'Deployment user managed by Puppet',
    home       => "/home/${my_webserver::deploy_user}",
    shell      => '/bin/bash',
    managehome => true,
    groups     => ['www-data'],
  }
}
```

**`manifests/config.pp`:**

```puppet
# Configuration class
class my_webserver::config {
  # Create document root
  file { $my_webserver::doc_root:
    ensure => directory,
    owner  => 'www-data',
    group  => 'www-data',
    mode   => '0755',
  }

  # Deploy Nginx site configuration
  file { '/etc/nginx/sites-available/production':
    ensure  => file,
    content => epp('my_webserver/site.conf.epp', {
      'port'        => $my_webserver::port,
      'server_name' => $my_webserver::server_name,
      'doc_root'    => $my_webserver::doc_root,
      'ssl_enabled' => $my_webserver::ssl_enabled,
    }),
    owner   => 'root',
    group   => 'root',
    mode    => '0644',
  }

  # Enable the site
  file { '/etc/nginx/sites-enabled/production':
    ensure => link,
    target => '/etc/nginx/sites-available/production',
  }

  # Remove default site
  file { '/etc/nginx/sites-enabled/default':
    ensure => absent,
  }

  # Deploy index page
  file { "${my_webserver::doc_root}/index.html":
    ensure  => file,
    content => epp('my_webserver/index.html.epp', {
      'server_name' => $my_webserver::server_name,
      'hostname'    => $facts['hostname'],
    }),
    owner   => 'www-data',
    group   => 'www-data',
    mode    => '0644',
  }
}
```

**`manifests/service.pp`:**

```puppet
# Service management class
class my_webserver::service {
  service { 'nginx':
    ensure => running,
    enable => true,
  }
}
```

**`templates/site.conf.epp`:**

```epp
<%- |
  Integer $port,
  String  $server_name,
  String  $doc_root,
  Boolean $ssl_enabled,
| -%>
# Managed by Puppet — DO NOT EDIT MANUALLY

server {
<% if $ssl_enabled { -%>
    listen <%= $port %> ssl;
    ssl_certificate     /etc/ssl/certs/<%= $server_name %>.pem;
    ssl_certificate_key /etc/ssl/private/<%= $server_name %>.key;
    ssl_protocols       TLSv1.2 TLSv1.3;
<% } else { -%>
    listen <%= $port %>;
<% } -%>

    server_name <%= $server_name %>;
    root        <%= $doc_root %>;
    index       index.html;

    location / {
        try_files $uri $uri/ =404;
    }

    access_log /var/log/nginx/<%= $server_name %>-access.log;
    error_log  /var/log/nginx/<%= $server_name %>-error.log;
}
```

**`templates/index.html.epp`:**

```epp
<%- |
  String $server_name,
  String $hostname,
| -%>
<!DOCTYPE html>
<html>
<head><title><%= $server_name %></title></head>
<body>
  <h1>Welcome to <%= $server_name %></h1>
  <p>Served from host: <strong><%= $hostname %></strong></p>
  <p>Managed by Puppet</p>
</body>
</html>
```

## 24. Common Gotchas and Best Practices

### ❌ Common Mistakes

```puppet
# ❌ DON'T duplicate resource titles (Puppet will error!)
package { 'nginx':
  ensure => installed,
}
package { 'nginx':     # ERROR: Duplicate resource declaration!
  ensure => latest,
}

# ✅ DO declare each resource ONCE
package { 'nginx':
  ensure => installed,
}

# ❌ DON'T use exec for everything
exec { 'install-nginx':
  command => 'apt-get install -y nginx',
  path    => '/usr/bin',
}

# ✅ DO use specific resources
package { 'nginx':
  ensure => installed,
}

# ❌ DON'T hardcode data in manifests
class my_webserver {
  $port = 80                    # Hardcoded!
  $server_name = 'mysite.com'  # Hardcoded!
}

# ✅ DO use Hiera for data
class my_webserver (
  Integer $port,           # Comes from Hiera
  String  $server_name,    # Comes from Hiera
) { ... }

# ❌ DON'T forget ordering
service { 'nginx': ensure => running }
package { 'nginx': ensure => installed }
# Puppet might try to start nginx before installing it!

# ✅ DO define relationships
package { 'nginx': ensure => installed }
-> service { 'nginx': ensure => running }
```

### ✅ Best Practices

1. **Separate data from code** — Use Hiera for ALL configuration values
2. **Use the Roles and Profiles pattern** — Roles define server types, profiles wrap modules
3. **Follow naming conventions** — Class names MUST match file paths
4. **Test everything** — Use PDK, rspec-puppet, puppet-lint
5. **Use Puppet Forge modules** — Don't reinvent the wheel
6. **Pin module versions** — In Puppetfile and metadata.json
7. **Use environments** — Dev/staging/production with r10k
8. **One resource per title** — Never declare the same resource twice
9. **Prefer EPP over ERB** — EPP is the modern template format
10. **Use `--noop` first** — Always do a dry run before applying changes in production

### The Roles and Profiles Pattern

This is the recommended way to organize your Puppet code:

```
Roles (what a server IS)      → role::webserver, role::database
  ↓ includes
Profiles (what a server HAS)  → profile::nginx, profile::monitoring, profile::firewall
  ↓ wraps
Modules (how to do things)    → puppetlabs-nginx, puppetlabs-firewall
```

```puppet
# A Role — one per server type, includes profiles
class role::webserver {
  include profile::base
  include profile::nginx
  include profile::monitoring
  include profile::firewall
}

# A Profile — wraps one or more modules with site-specific config
class profile::nginx {
  class { 'nginx':
    manage_repo => true,
  }

  nginx::resource::server { $facts['fqdn']:
    listen_port => 443,
    ssl         => true,
    www_root    => '/var/www/html',
  }
}

# Node classification — just assign roles
node 'web-server-1.example.com' {
  include role::webserver
}
```

## 25. When to Choose Puppet

### ✅ Choose Puppet When:

- You need **true declarative** configuration management
- You're managing **large-scale infrastructure** (thousands of nodes)
- **Compliance and auditing** are critical (Puppet Enterprise has great reporting)
- You need **automatic drift remediation** — Puppet self-heals every 30 minutes
- You want a **mature, proven** tool (20+ years in the industry)
- You need **cross-platform support** (Linux AND Windows)
- Your organization values **strong typing** and **code quality** (Puppet DSL enforces structure)

### ❌ Maybe Skip Puppet When:

- You're managing a **small number of servers** (Ansible is simpler to start with)
- You need **ad-hoc** one-time tasks (Ansible is better for this)
- Your team doesn't want to **learn a new DSL** (Ansible uses YAML, Chef uses Ruby)
- You want an **agentless** solution (Ansible doesn't need agents)
- **Speed of setup** is critical (Puppet Server takes more time to set up than Ansible)

## 26. Summary

| Concept | What It Is |
|---------|-----------|
| **Puppet** | Declarative configuration management tool with its own DSL |
| **Manifest** | A `.pp` file containing Puppet code |
| **Resource** | Single unit of configuration (package, file, service, user) |
| **Class** | Named block of Puppet code, can have parameters |
| **Module** | Self-contained bundle of manifests, templates, files, data |
| **Puppet Forge** | Community repository of 7,000+ pre-built modules |
| **Facter** | Tool that auto-detects system facts (OS, IP, CPU, RAM) |
| **Hiera** | Data lookup system — separates data from code |
| **Catalog** | Compiled desired state for a specific node (JSON) |
| **Puppet Server** | Central server that compiles catalogs for nodes |
| **Puppet Agent** | Software on each node that applies the catalog |
| **EPP Template** | Dynamic config file using Puppet syntax |
| **Environment** | Separate set of code/data (dev, staging, production) |
| **r10k** | Tool that maps Git branches to Puppet environments |
| **PDK** | Puppet Development Kit — create, test, validate modules |

### The Puppet Workflow in a Nutshell

```
1. Write modules with manifests, templates, and Hiera data
2. Test with PDK (lint, unit tests, validation)
3. Push to Git → r10k deploys to Puppet Server environments
4. Nodes run puppet-agent every 30 minutes
5. Agent sends facts → Server compiles catalog → Agent applies catalog
6. Agent reports back → Server stores the report
7. Drift detected? → Puppet auto-remediates on next run
8. Repeat forever — your infrastructure stays consistent! 🎉
```
