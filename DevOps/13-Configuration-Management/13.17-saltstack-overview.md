## 1. What is SaltStack?

**SaltStack** (usually just called **Salt**) is an **open-source configuration management and remote execution tool**. It lets you manage thousands of servers simultaneously, execute commands on them in real-time, and ensure they're all configured exactly how you want.

### The Big Idea

Salt does two main things:

1. **Remote Execution** — Run commands on thousands of servers at once, in seconds
2. **Configuration Management** — Define the desired state of your servers and ensure they match

> Think of Salt like a **military radio system** 📻 — the general (Salt Master) sends a command, and EVERY soldier (Salt Minion) hears it and acts on it **instantly**. No delays, no going one-by-one.

### What Makes Salt Special?

Salt's killer feature is **SPEED**. While other tools (like Ansible) connect to servers one-by-one over SSH, Salt uses a persistent connection via **ZeroMQ** (a high-speed messaging library). This means:

```
Ansible:   "Hey server 1, do this... okay done. Hey server 2, do this... okay done."
           → Takes minutes for 500 servers

Salt:      "Hey ALL 500 servers, do this NOW!"
           → Takes seconds for 500 servers
```

### Key Facts

| Feature | Details |
|---------|---------|
| **Created By** | Thomas S. Hatch (2011) |
| **Now Maintained By** | VMware (acquired in 2020), open-source community |
| **Written In** | Python |
| **Configuration Language** | YAML + Jinja2 templates |
| **Model** | Both! Push (remote execution) AND Pull (states via schedule) |
| **Agent Required?** | Optional — has both agent (minion) and agentless (salt-ssh) modes |
| **Communication** | ZeroMQ (default) or TCP — blazing fast |
| **Open Source?** | Yes — Apache 2.0 license |
| **Commercial Version** | VMware Aria Automation Config (formerly vRealize SaltStack Config) |

## 2. Why SaltStack? — The Problem It Solves

### The Speed Problem

```
Scenario: It's 3 AM. A critical security vulnerability (CVE) just dropped.
You need to patch ALL 2,000 servers. RIGHT NOW.

With SSH scripts:
  → Loop through 2,000 servers, one by one
  → Each connection takes 2-5 seconds
  → Running the patch takes 30 seconds per server
  → Total time: ~19 HOURS 😱
  → By then, you've probably been hacked

With Ansible:
  → Forks to maybe 50 servers at a time (default is 5)
  → Better, but still takes a while for 2,000 servers
  → Total time: ~30-60 minutes

With Salt:
  → One command from the Salt Master
  → ALL 2,000 minions receive it simultaneously via ZeroMQ
  → ALL 2,000 servers start patching at the SAME time
  → Total time: ~2-3 MINUTES 🚀
```

### The Two Superpowers of Salt

**Superpower 1: Remote Execution (Ad-Hoc Commands)**

```bash
# Check uptime on ALL servers — result comes back in seconds
salt '*' cmd.run 'uptime'

# Install a security patch on all web servers
salt 'web-*' pkg.install 'openssl' version='3.0.12'

# Restart nginx on all web servers
salt 'web-*' service.restart 'nginx'

# Get disk usage on all database servers
salt 'db-*' disk.usage
```

**Superpower 2: Configuration Management (States)**

```yaml
# Define the desired state — YAML (simple!)
nginx_installed:
  pkg.installed:
    - name: nginx

nginx_running:
  service.running:
    - name: nginx
    - enable: True
    - require:
      - pkg: nginx_installed
```

### Key Benefits

| Benefit | Explanation |
|---------|------------|
| **Speed** | ZeroMQ messaging = near-instant execution across thousands of servers |
| **Scalability** | Designed for 10,000+ nodes from day one |
| **Real-Time** | Execute commands and get results in seconds |
| **Flexibility** | Both push (remote execution) and pull (scheduled states) |
| **YAML + Jinja** | Easy to learn — YAML for structure, Jinja2 for templating |
| **Event-Driven** | React to events automatically (beacons + reactors) |
| **Python-Based** | Easy to extend with custom Python modules |
| **Agentless Option** | Salt SSH works without installing agents |

## 3. SaltStack vs Ansible vs Puppet vs Chef — Comparison

| Feature | Salt | Ansible | Puppet | Chef |
|---------|------|---------|--------|------|
| **Language** | YAML + Jinja2 | YAML + Jinja2 | Puppet DSL | Ruby DSL |
| **Architecture** | Master-Minion (ZeroMQ) | Agentless (SSH) | Server-Agent | Server-Client |
| **Speed** | ⚡ Fastest (ZeroMQ) | Moderate (SSH) | Moderate | Moderate |
| **Agent** | Optional (minion) | No agent needed | Yes (puppet-agent) | Yes (chef-client) |
| **Remote Execution** | ✅ Built-in & excellent | Ad-hoc commands | Limited | Limited |
| **Config Management** | ✅ States (YAML) | ✅ Playbooks (YAML) | ✅ Manifests | ✅ Recipes (Ruby) |
| **Event System** | ✅ Built-in (beacons, reactors) | Limited | Limited (reports) | Limited |
| **Learning Curve** | Moderate | Gentle | Moderate | Steep |
| **Best For** | Speed, large scale, real-time | Simplicity, quick start | Enterprise compliance | Complex infra, Ruby |
| **Scalability** | Excellent (10,000+) | Good (with tuning) | Excellent | Good |

### When to Pick Salt Over Others?

- **Speed is critical** — You need to execute across thousands of servers in seconds
- **Real-time remote execution** — You want both config management AND ad-hoc commands
- **Event-driven automation** — You want to react to events automatically
- **Very large scale** — 5,000+ nodes
- **Python shop** — Your team knows Python and wants to extend the tool
- **Hybrid approach** — You want both push (execute now) and pull (scheduled) models

## 4. How Salt Works — Architecture

### The Master-Minion Model

```
┌──────────────────────────────────────────────────────────────────┐
│                       SALT ARCHITECTURE                          │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│                    ┌───────────────────┐                          │
│                    │    SALT MASTER    │                          │
│                    │                   │                          │
│                    │  What it stores:  │                          │
│                    │  ├── States       │                          │
│                    │  ├── Pillar data  │                          │
│                    │  ├── Top files    │                          │
│                    │  ├── Formulas     │                          │
│                    │  └── Templates    │                          │
│                    │                   │                          │
│                    │  What it does:    │                          │
│                    │  ├── Sends cmds   │                          │
│                    │  ├── Compiles     │                          │
│                    │  │   states       │                          │
│                    │  ├── Manages keys │                          │
│                    │  └── Event bus    │                          │
│                    └─────────┬─────────┘                          │
│                              │                                   │
│                     ZeroMQ (port 4505/4506)                      │
│                   (persistent, fast connection)                  │
│                              │                                   │
│          ┌───────────────────┼───────────────────┐               │
│          │                   │                   │               │
│          ▼                   ▼                   ▼               │
│   ┌────────────┐     ┌────────────┐     ┌────────────┐          │
│   │  MINION 1  │     │  MINION 2  │     │  MINION N  │          │
│   │ (web-srv-1)│     │ (db-srv-1) │     │   (...  )  │          │
│   │            │     │            │     │            │          │
│   │ salt-minion│     │ salt-minion│     │ salt-minion│          │
│   └────────────┘     └────────────┘     └────────────┘          │
│                                                                  │
│   Two ZeroMQ ports:                                              │
│   Port 4505 → PUBLISH (Master broadcasts commands to ALL)        │
│   Port 4506 → RETURN (Minions send results back to Master)       │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### How ZeroMQ Makes Salt Fast

```
Traditional approach (SSH-based like Ansible):
  Master → SSH to server-1 → run command → get result
  Master → SSH to server-2 → run command → get result  
  Master → SSH to server-3 → run command → get result
  ... (one at a time, or limited parallel)

Salt's approach (ZeroMQ message bus):
  Master PUBLISHES command on port 4505 → ALL minions receive it INSTANTLY
  All minions execute simultaneously
  All minions RETURN results on port 4506

It's like:
  📢 SSH = Making individual phone calls (one at a time)
  📢 Salt = Broadcasting on a radio (everyone hears at once)
```

### The Salt Run — Step by Step

```
Step 1: You type a command on the Salt Master
        $ salt '*' pkg.install nginx
            ↓
Step 2: Master PUBLISHES the command to ALL minions via ZeroMQ (port 4505)
        Every minion with a matching target receives it instantly
            ↓
Step 3: Each minion EXECUTES the command locally
        → Checks if nginx is installed
        → If not, installs it
        → Collects the result
            ↓
Step 4: Each minion RETURNS the result to the Master via ZeroMQ (port 4506)
        → "web-server-1: nginx 1.18.0 installed successfully"
        → "web-server-2: nginx 1.18.0 already installed"
            ↓
Step 5: Master displays ALL results to you
        Done! All 500 servers responded in 3 seconds ⚡
```

### The Event Bus

Salt has a built-in **event bus** — everything that happens generates events:

```
Events:
  → Minion connected → event
  → Minion ran a command → event  
  → State applied → event
  → Key accepted → event
  → Beacon triggered → event
  
You can:
  → Watch events in real-time
  → React to events automatically (Reactors)
  → Build event-driven automation
```

## 5. Salt Key Terminology

| Term | What It Means | Analogy |
|------|--------------|---------|
| **Master** | Central server that controls everything | The general |
| **Minion** | Agent running on managed servers | The soldier |
| **State** | YAML file describing desired configuration | A blueprint |
| **Grain** | Static system information about a minion (OS, CPU, IP) | A soldier's ID card |
| **Pillar** | Secure, minion-specific data (passwords, settings) | A sealed orders envelope |
| **Top File** | Maps states/pillars to minions | The assignment board |
| **Module** | Python code that does stuff (execution modules, state modules) | A toolbox |
| **Formula** | A reusable collection of states (like an Ansible role) | A pre-built construction kit |
| **Highstate** | Applying ALL assigned states to a minion | Full inspection + repair |
| **Beacon** | Monitors something on a minion and fires events | A sensor/alarm |
| **Reactor** | Automatically responds to events | An automated response system |
| **Runner** | Commands that run on the Master itself | HQ-level operations |
| **Mine** | Data shared between minions | The shared intelligence database |
| **Salt SSH** | Agentless mode — uses SSH instead of ZeroMQ | Radio-less mode (walkie-talkies instead) |
| **Returner** | Sends results to external systems (DB, Elasticsearch, etc.) | A reporting channel |

## 6. Installing Salt

### Installing the Salt Master

```bash
# On Ubuntu 22.04

# Step 1: Add the Salt repository
sudo curl -fsSL https://packages.broadcom.com/artifactory/api/security/keypair/SaltProjectKey/public | sudo tee /etc/apt/keyrings/salt-archive-keyring.pgp
echo "deb [signed-by=/etc/apt/keyrings/salt-archive-keyring.pgp] https://packages.broadcom.com/artifactory/saltproject-deb stable main" | sudo tee /etc/apt/sources.list.d/salt.list

sudo apt-get update

# Step 2: Install Salt Master
sudo apt-get install -y salt-master

# Step 3: Start the Salt Master
sudo systemctl start salt-master
sudo systemctl enable salt-master

# Step 4: Verify
salt --version
sudo systemctl status salt-master
```

### Installing the Salt Minion

```bash
# On Ubuntu 22.04

# Step 1: Add the Salt repository (same as above)
sudo curl -fsSL https://packages.broadcom.com/artifactory/api/security/keypair/SaltProjectKey/public | sudo tee /etc/apt/keyrings/salt-archive-keyring.pgp
echo "deb [signed-by=/etc/apt/keyrings/salt-archive-keyring.pgp] https://packages.broadcom.com/artifactory/saltproject-deb stable main" | sudo tee /etc/apt/sources.list.d/salt.list

sudo apt-get update

# Step 2: Install Salt Minion
sudo apt-get install -y salt-minion

# Step 3: Point the minion to the master
# Edit /etc/salt/minion
echo "master: salt-master.example.com" | sudo tee -a /etc/salt/minion

# Optional: Set the minion ID (defaults to hostname)
echo "id: web-server-1" | sudo tee -a /etc/salt/minion

# Step 4: Start the minion
sudo systemctl start salt-minion
sudo systemctl enable salt-minion

# Step 5: Verify
salt-minion --version
```

### Accepting Minion Keys

Salt uses public key cryptography. When a minion starts, it sends its public key to the master. You must accept it before the master will communicate with it.

```bash
# On the MASTER — list all key statuses
sudo salt-key -L

# Output:
# Accepted Keys:
# Denied Keys:
# Unaccepted Keys:
#   web-server-1
#   web-server-2
# Rejected Keys:

# Accept a specific key
sudo salt-key -a web-server-1

# Accept ALL pending keys (careful in production!)
sudo salt-key -A

# Reject a key
sudo salt-key -r suspicious-server

# Delete a key
sudo salt-key -d old-server

# Test connectivity
sudo salt '*' test.ping
# web-server-1:
#     True
# web-server-2:
#     True
```

### Quick Test — Is It Working?

```bash
# Ping all minions
salt '*' test.ping

# Get the Salt version on all minions
salt '*' test.version

# Run a simple command on all minions
salt '*' cmd.run 'hostname'

# If you see responses — you're good to go! ✅
```

## 7. Salt Commands — Remote Execution

Remote execution is Salt's superpower. You can run ANY command on ANY set of minions instantly.

### Basic Syntax

```
salt '<target>' <module>.<function> [arguments]
```

| Part | What It Means |
|------|--------------|
| `salt` | The Salt CLI command |
| `'<target>'` | Which minions to target (`'*'` = all, `'web-*'` = matching pattern) |
| `<module>.<function>` | What to do (e.g., `cmd.run`, `pkg.install`, `service.restart`) |
| `[arguments]` | Parameters for the function |

### Common Execution Modules

#### Running Shell Commands

```bash
# Run a command on all minions
salt '*' cmd.run 'uptime'

# Run a command on specific minions
salt 'web-server-1' cmd.run 'free -m'

# Run as a specific user
salt '*' cmd.run 'whoami' runas='deploy'

# Run with environment variables
salt '*' cmd.run 'echo $MY_VAR' env='{"MY_VAR": "hello"}'
```

#### Package Management

```bash
# Install a package
salt '*' pkg.install nginx

# Install a specific version
salt '*' pkg.install nginx version='1.18.0-6ubuntu14'

# Remove a package
salt '*' pkg.remove apache2

# Upgrade a package
salt '*' pkg.upgrade nginx

# Upgrade ALL packages
salt '*' pkg.upgrade

# List installed packages
salt '*' pkg.list_pkgs

# Check if a package is installed
salt '*' pkg.version nginx
```

#### Service Management

```bash
# Start a service
salt '*' service.start nginx

# Stop a service
salt '*' service.stop nginx

# Restart a service
salt '*' service.restart nginx

# Reload a service (without downtime)
salt '*' service.reload nginx

# Check status
salt '*' service.status nginx

# Enable on boot
salt '*' service.enable nginx

# Disable on boot
salt '*' service.disable nginx
```

#### File Operations

```bash
# Check if a file exists
salt '*' file.file_exists /etc/nginx/nginx.conf

# Read a file
salt '*' file.read /etc/hostname

# Write to a file
salt '*' file.write /tmp/hello.txt "Hello from Salt!"

# Get file stats
salt '*' file.stats /etc/nginx/nginx.conf

# Create a directory
salt '*' file.mkdir /opt/myapp

# Copy a file
salt '*' file.copy /etc/nginx/nginx.conf /etc/nginx/nginx.conf.bak
```

#### System Information

```bash
# Get all system information
salt '*' grains.items

# Get specific info
salt '*' grains.item os osrelease

# Get network interfaces
salt '*' network.interfaces

# Get disk usage
salt '*' disk.usage

# Get memory info
salt '*' status.meminfo

# Get CPU info
salt '*' status.cpuinfo

# Get the uptime
salt '*' status.uptime
```

#### User Management

```bash
# Create a user
salt '*' user.add deploy shell=/bin/bash createhome=True

# Delete a user
salt '*' user.delete olduser remove=True

# Check if a user exists
salt '*' user.info deploy

# Add user to a group
salt '*' group.adduser developers deploy
```

### Output Formatting

```bash
# Default output (nested)
salt '*' test.ping

# JSON output
salt '*' test.ping --out=json

# YAML output
salt '*' test.ping --out=yaml

# Table output
salt '*' test.ping --out=table

# Text output (one line per minion)
salt '*' test.ping --out=txt

# Only show results (no minion names)
salt '*' cmd.run 'hostname' --out=raw
```

## 8. Targeting Minions — Who Gets the Command?

Salt has powerful targeting options so you can precisely select which minions get a command.

### Glob Matching (Default)

```bash
# All minions
salt '*' test.ping

# Matching a pattern
salt 'web-*' test.ping          # web-server-1, web-server-2, web-prod-01, etc.
salt 'db-srv-?' test.ping       # db-srv-1, db-srv-2 (single character)
salt 'web-server-[1-3]' test.ping   # web-server-1, web-server-2, web-server-3
```

### List Targeting

```bash
# Specific minions (comma-separated)
salt -L 'web-server-1,web-server-2,db-server-1' test.ping
```

### Grain Targeting (Based on System Facts)

```bash
# All Ubuntu servers
salt -G 'os:Ubuntu' test.ping

# All 64-bit servers
salt -G 'osarch:amd64' test.ping

# All servers with 4+ CPUs
salt -G 'num_cpus:4' test.ping

# All servers in a specific data center (custom grain)
salt -G 'datacenter:us-east-1' test.ping
```

### Pillar Targeting (Based on Pillar Data)

```bash
# Target based on pillar values
salt -I 'role:webserver' test.ping

# Target based on nested pillar
salt -I 'app:environment:production' test.ping
```

### Compound Targeting (Combine Multiple Criteria)

```bash
# Combine different targeting methods
salt -C 'G@os:Ubuntu and web-*' test.ping
# Translation: "Ubuntu servers whose name starts with web-"

salt -C 'G@os:Ubuntu and not G@osrelease:18.04' test.ping
# Translation: "Ubuntu servers that are NOT version 18.04"

salt -C 'G@os:Ubuntu or G@os:CentOS' test.ping
# Translation: "All Ubuntu OR CentOS servers"
```

### Regex Targeting

```bash
# Regex matching on minion ID
salt -E 'web-server-[0-9]+' test.ping
```

### PCRE Targeting (Perl-Compatible Regular Expressions)

```bash
salt -P 'web-(prod|staging)-\d+' test.ping
```

### Targeting Cheat Sheet

| Flag | Method | Example |
|------|--------|---------|
| (none) | Glob | `salt 'web-*' test.ping` |
| `-L` | List | `salt -L 'srv1,srv2,srv3' test.ping` |
| `-G` | Grain | `salt -G 'os:Ubuntu' test.ping` |
| `-I` | Pillar | `salt -I 'role:webserver' test.ping` |
| `-E` | Regex | `salt -E 'web-[0-9]+' test.ping` |
| `-C` | Compound | `salt -C 'G@os:Ubuntu and web-*' test.ping` |
| `-N` | Nodegroup | `salt -N 'webservers' test.ping` |

### Nodegroups (Pre-Defined Groups)

Define groups in the master config:

```yaml
# /etc/salt/master.d/nodegroups.conf
nodegroups:
  webservers: 'web-*'
  databases: 'db-*'
  production: 'G@environment:production'
  ubuntu_web: 'G@os:Ubuntu and web-*'
```

```bash
# Use nodegroups
salt -N 'webservers' test.ping
salt -N 'production' cmd.run 'uptime'
```

## 9. Grains — System Information

**Grains** are static pieces of information about a minion — things like the operating system, CPU count, IP addresses, and more. They're automatically collected when the minion starts.

### Viewing Grains

```bash
# Show ALL grains for a minion (there are tons!)
salt 'web-server-1' grains.items

# Show specific grains
salt 'web-server-1' grains.item os osrelease ip4_interfaces

# Common grains you'll use:
salt '*' grains.item os            # Ubuntu, CentOS, Windows
salt '*' grains.item osrelease     # 22.04, 8.5
salt '*' grains.item os_family     # Debian, RedHat, Windows
salt '*' grains.item osfinger      # Ubuntu-22.04
salt '*' grains.item fqdn          # web-server-1.example.com
salt '*' grains.item hostname      # web-server-1
salt '*' grains.item ip4_interfaces   # Network interfaces & IPs
salt '*' grains.item num_cpus      # 4
salt '*' grains.item mem_total     # 16384 (in MB)
salt '*' grains.item kernel        # Linux
salt '*' grains.item virtual       # physical, kvm, xen, VirtualBox
salt '*' grains.item init          # systemd
```

### Using Grains in Targeting

```bash
# Run command only on Ubuntu servers
salt -G 'os:Ubuntu' cmd.run 'apt-get update'

# Run only on servers with 8+ CPUs
salt -G 'num_cpus:8' cmd.run 'echo "I am powerful!"'

# Run on all virtual machines
salt -G 'virtual:kvm' test.ping
```

### Custom Grains

You can add your own custom grains:

**Method 1: In the minion config (`/etc/salt/minion.d/custom_grains.conf`):**

```yaml
grains:
  role: webserver
  environment: production
  datacenter: us-east-1
  team: platform
```

**Method 2: As a Python grain module (`/srv/salt/_grains/my_grains.py`):**

```python
# _grains/my_grains.py
import os

def app_grains():
    """Custom grains for our application."""
    grains = {}
    
    # Check if our app is installed
    if os.path.exists('/opt/myapp/VERSION'):
        with open('/opt/myapp/VERSION', 'r') as f:
            grains['myapp_version'] = f.read().strip()
    else:
        grains['myapp_version'] = 'not_installed'
    
    # Determine the server role based on hostname
    hostname = os.uname()[1]
    if hostname.startswith('web-'):
        grains['role'] = 'webserver'
    elif hostname.startswith('db-'):
        grains['role'] = 'database'
    else:
        grains['role'] = 'unknown'
    
    return grains
```

```bash
# Sync custom grains to all minions
salt '*' saltutil.sync_grains

# Now use them!
salt -G 'role:webserver' test.ping
salt -G 'myapp_version:2.1.0' test.ping
```

## 10. States — Declarative Configuration

**States** are Salt's configuration management feature. They're YAML files that describe the **desired state** of a system.

### Where Do States Live?

```
/srv/salt/              ← The "file root" (default location)
├── top.sls             ← The top file (maps states to minions)
├── webserver/
│   ├── init.sls        ← Default state for "webserver"
│   ├── install.sls     ← Installation state
│   ├── config.sls      ← Configuration state
│   └── files/
│       └── nginx.conf  ← Static/template files
├── database/
│   ├── init.sls
│   └── ...
└── common/
    ├── init.sls
    └── packages.sls
```

> **Note:** Salt state files use the `.sls` extension (SaLt State). They're just YAML files!

### Your First State

```yaml
# /srv/salt/webserver/init.sls

# Step 1: Install Nginx
install_nginx:
  pkg.installed:
    - name: nginx

# Step 2: Deploy the configuration file
nginx_config:
  file.managed:
    - name: /etc/nginx/sites-available/default
    - source: salt://webserver/files/default.conf
    - user: root
    - group: root
    - mode: '0644'
    - require:
      - pkg: install_nginx

# Step 3: Make sure Nginx is running and enabled
nginx_service:
  service.running:
    - name: nginx
    - enable: True
    - require:
      - pkg: install_nginx
    - watch:
      - file: nginx_config
```

### Breaking Down the State (Line by Line)

```yaml
install_nginx:                    # ← State ID (unique name for this state block)
  pkg.installed:                  # ← State module.function
    - name: nginx                 # ← The package to install

nginx_config:                     # ← Another state ID
  file.managed:                   # ← Manage a file
    - name: /etc/nginx/...        # ← Path of the file on the minion
    - source: salt://webserver/...# ← Source file on the Salt Master
    - user: root                  # ← File owner
    - group: root                 # ← File group
    - mode: '0644'                # ← File permissions
    - require:                    # ← This state REQUIRES install_nginx to run first
      - pkg: install_nginx

nginx_service:                    # ← Another state ID
  service.running:                # ← Ensure the service is running
    - name: nginx                 # ← Service name
    - enable: True                # ← Start on boot
    - require:                    # ← Requires nginx to be installed
      - pkg: install_nginx
    - watch:                      # ← If nginx_config changes, restart this service
      - file: nginx_config
```

### Applying States

```bash
# Apply a specific state to specific minions
salt 'web-*' state.apply webserver

# Apply a specific state to a single minion
salt 'web-server-1' state.apply webserver

# Apply ALL states assigned in the top file (highstate)
salt '*' state.highstate

# Dry run (test mode — show what WOULD change)
salt 'web-server-1' state.apply webserver test=True

# Apply state locally (without a master)
sudo salt-call --local state.apply webserver
```

## 12. Top File — Mapping States to Minions

The **top file** (`top.sls`) is the "assignment board" — it maps which states get applied to which minions.

### Basic Top File

```yaml
# /srv/salt/top.sls

base:                               # ← The environment name
  '*':                              # ← Target: ALL minions
    - common                        # ← Apply the "common" state
    - common.packages               # ← Apply the "common.packages" state

  'web-*':                          # ← Target: minions matching "web-*"
    - webserver                     # ← Apply the "webserver" state

  'db-*':                           # ← Target: minions matching "db-*"
    - database                      # ← Apply the "database" state

  'G@os:Ubuntu':                    # ← Target: all Ubuntu servers (grain match)
    - ubuntu_specific               # ← Apply Ubuntu-specific config

  'web-server-1':                   # ← Target: one specific minion
    - special_config                # ← Apply special configuration
```

### How to Read the Top File

```
When you run: salt '*' state.highstate

Salt says:
  1. "Who are you?" → web-server-1 (minion ID)
  2. Checks top.sls: 
     - '*' matches → apply: common, common.packages
     - 'web-*' matches → apply: webserver
     - web-server-1 matches → apply: special_config
     - 'G@os:Ubuntu' → checks grain... yes, Ubuntu → apply: ubuntu_specific
  3. Applies ALL matched states in order
```

### Advanced Top File

```yaml
# /srv/salt/top.sls

base:
  # All servers get base config
  '*':
    - common
    - common.users
    - common.ssh
    - monitoring.node_exporter

  # Web servers
  'G@role:webserver':
    - match: grain
    - webserver
    - webserver.firewall

  # Database servers
  'I@role:database':
    - match: pillar
    - database
    - database.backup

  # Production only
  'G@environment:production':
    - match: grain
    - security.hardening
    - monitoring.alerts

  # Compound match
  'G@os:Ubuntu and web-*':
    - match: compound
    - ubuntu_web_specific
```

## 13. Pillar — Secure Data Management

**Pillar** is Salt's system for managing **secure, minion-specific data**. It's like Hiera in Puppet or Vault in Ansible — a way to separate sensitive data from your state files.

### Why Pillar?

```
WITHOUT Pillar (secrets in state files — BAD! 😱):
  database_config:
    file.managed:
      - name: /etc/myapp/database.yml
      - contents: |
          host: db.example.com
          password: super_secret_123    ← EXPOSED in your Git repo!

WITH Pillar (secrets separated — GOOD! ✅):
  database_config:
    file.managed:
      - name: /etc/myapp/database.yml
      - contents: |
          host: {{ pillar['database']['host'] }}
          password: {{ pillar['database']['password'] }}
          
  # Pillar data is stored separately, encrypted, and only sent to the specific minion
```

### Pillar Structure

```
/srv/pillar/                    ← Pillar root (separate from states!)
├── top.sls                     ← Maps pillar data to minions
├── common.sls                  ← Common data for all minions
├── webserver.sls               ← Data for web servers
├── database.sls                ← Data for database servers
└── secrets/
    └── production.sls          ← Production secrets
```

### Pillar Top File

```yaml
# /srv/pillar/top.sls

base:
  '*':
    - common

  'web-*':
    - webserver

  'db-*':
    - database

  'G@environment:production':
    - match: grain
    - secrets.production
```

### Pillar Data Files

```yaml
# /srv/pillar/common.sls
timezone: 'UTC'
ntp_servers:
  - 0.pool.ntp.org
  - 1.pool.ntp.org
dns_servers:
  - 8.8.8.8
  - 8.8.4.4

# /srv/pillar/webserver.sls
nginx:
  port: 80
  server_name: {{ grains['fqdn'] }}
  worker_processes: {{ grains['num_cpus'] }}
  doc_root: /var/www/html

# /srv/pillar/database.sls
postgresql:
  version: '15'
  max_connections: 200
  shared_buffers: '256MB'

# /srv/pillar/secrets/production.sls
database:
  host: db.internal.example.com
  port: 5432
  username: myapp_prod
  password: 'very_secret_password_123!'    # Only sent to targeted minions!

ssl:
  certificate: |
    -----BEGIN CERTIFICATE-----
    MIIDXTCCAkWgAwIBAgI...
    -----END CERTIFICATE-----
  private_key: |
    -----BEGIN PRIVATE KEY-----
    MIIEvgIBADANBgkqhki...
    -----END PRIVATE KEY-----
```

### Using Pillar in States

```yaml
# /srv/salt/webserver/config.sls

nginx_config:
  file.managed:
    - name: /etc/nginx/sites-available/default
    - source: salt://webserver/templates/site.conf.jinja
    - template: jinja
    - context:
        port: {{ pillar['nginx']['port'] }}
        server_name: {{ pillar['nginx']['server_name'] }}
        doc_root: {{ pillar['nginx']['doc_root'] }}

database_config:
  file.managed:
    - name: /etc/myapp/database.yml
    - contents: |
        host: {{ pillar['database']['host'] }}
        port: {{ pillar['database']['port'] }}
        username: {{ pillar['database']['username'] }}
        password: {{ pillar['database']['password'] }}
    - user: myapp
    - group: myapp
    - mode: '0600'              # Restrictive permissions for secrets!
```

### Pillar Commands

```bash
# Refresh pillar data on all minions
salt '*' saltutil.refresh_pillar

# View ALL pillar data for a minion
salt 'web-server-1' pillar.items

# View a specific pillar value
salt 'web-server-1' pillar.get nginx:port

# View pillar data in a specific format
salt 'web-server-1' pillar.items --out=json
```

### Pillar Security

```
Key point: Pillar data is ONLY sent to the minion it's targeted to!

If your top.sls says:
  'web-*':
    - webserver_secrets

Then ONLY minions matching 'web-*' will receive the webserver_secrets data.
A database server will NEVER see it. 🔒
```

## 14. Jinja Templating in Salt

Salt uses **Jinja2** (a Python templating engine) for dynamic content in states and files. If you've used Ansible, you already know Jinja!

### Jinja in State Files

```yaml
# /srv/salt/webserver/install.sls

# Install the correct package based on OS
{% if grains['os_family'] == 'Debian' %}
install_nginx:
  pkg.installed:
    - name: nginx
{% elif grains['os_family'] == 'RedHat' %}
install_nginx:
  pkg.installed:
    - name: nginx
    - require:
      - pkg: epel-release

epel:
  pkg.installed:
    - name: epel-release
{% endif %}

# Install multiple packages using a loop
{% set packages = ['curl', 'wget', 'vim', 'htop', 'git'] %}
{% for pkg in packages %}
install_{{ pkg }}:
  pkg.installed:
    - name: {{ pkg }}
{% endfor %}

# Use pillar data
nginx_config:
  file.managed:
    - name: /etc/nginx/nginx.conf
    - source: salt://webserver/templates/nginx.conf.jinja
    - template: jinja
    - context:
        port: {{ pillar.get('nginx:port', 80) }}
        worker_processes: {{ grains['num_cpus'] }}
```

### Jinja in Template Files

**`/srv/salt/webserver/templates/nginx.conf.jinja`:**

```jinja
{# This is a Jinja comment — won't appear in the output #}
# Managed by SaltStack — DO NOT EDIT MANUALLY

user www-data;
worker_processes {{ grains['num_cpus'] }};
pid /run/nginx.pid;

events {
    worker_connections {{ pillar.get('nginx:max_connections', 1024) }};
}

http {
    sendfile on;
    tcp_nopush on;
    keepalive_timeout 65;

{% if pillar.get('nginx:ssl_enabled', False) %}
    # SSL Configuration
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;
    ssl_prefer_server_ciphers on;
{% endif %}

{% if pillar.get('nginx:upstream_servers') %}
    upstream backend {
{% for server in pillar['nginx']['upstream_servers'] %}
        server {{ server.host }}:{{ server.port }};
{% endfor %}
    }
{% endif %}

    server {
        listen {{ pillar.get('nginx:port', 80) }};
        server_name {{ grains['fqdn'] }};
        root {{ pillar.get('nginx:doc_root', '/var/www/html') }};
        index index.html;

        location / {
            try_files $uri $uri/ =404;
        }
    }
}
```

### Jinja Syntax Cheat Sheet

```jinja
{{ variable }}              → Output a variable's value
{% if condition %}          → If statement
{% elif condition %}        → Else if
{% else %}                  → Else
{% endif %}                 → End if

{% for item in list %}      → For loop
{% endfor %}                → End for

{# This is a comment #}    → Comment (not in output)

{{ grains['key'] }}         → Access a grain
{{ pillar['key'] }}         → Access a pillar value
{{ pillar.get('key', 'default') }}  → Pillar with default value

{{ salt['cmd.run']('hostname') }}   → Run a Salt function inline

{% set my_var = 'value' %}  → Set a variable

{{ 'hello' | upper }}       → Filter: HELLO
{{ list | join(', ') }}     → Filter: join list items
{{ value | default('fallback') }}  → Filter: default value
```

## 16. Salt Mine — Sharing Data Between Minions

## 17. Reactors and Beacons — Event-Driven Automation

This is one of Salt's most powerful and unique features!

### Beacons — Monitor & Fire Events

**Beacons** run on minions and watch for things happening on the system. When something happens, they fire an event.

```yaml
# /etc/salt/minion.d/beacons.conf

beacons:
  # Watch for file changes
  inotify:
    - files:
        /etc/nginx/nginx.conf:
          mask:
            - modify
            - delete
    - disable_during_state_run: True

  # Watch for disk usage > 90%
  diskusage:
    - /: '90%'
    - /var: '85%'
    - interval: 300           # Check every 5 minutes

  # Watch for service status changes
  service:
    - services:
        nginx:
          onchangeonly: True
    - interval: 60

  # Watch for high load
  load:
    - averages:
        1m:
          - 2.0
        5m:
          - 1.5
        15m:
          - 1.0
```

### Reactors — Respond to Events Automatically

**Reactors** run on the master and automatically respond to events.

```yaml
# /etc/salt/master.d/reactor.conf

reactor:
  # When a minion starts up, apply its highstate
  - 'salt/minion/*/start':
    - /srv/reactor/highstate.sls

  # When disk usage is high, clean up logs
  - 'salt/beacon/*/diskusage/':
    - /srv/reactor/disk_cleanup.sls

  # When nginx stops unexpectedly, restart it
  - 'salt/beacon/*/service/nginx':
    - /srv/reactor/restart_nginx.sls
```

**Reactor state files:**

```yaml
# /srv/reactor/highstate.sls
apply_highstate:
  local.state.highstate:
    - tgt: {{ data['id'] }}         # Target the minion that triggered the event

# /srv/reactor/disk_cleanup.sls
cleanup_logs:
  local.cmd.run:
    - tgt: {{ data['id'] }}
    - arg:
      - 'find /var/log -name "*.gz" -mtime +7 -delete'

# /srv/reactor/restart_nginx.sls
restart_nginx:
  local.service.restart:
    - tgt: {{ data['id'] }}
    - arg:
      - nginx
```

### The Event System in Practice

```
Flow:
  1. Beacon on web-server-1 detects: nginx stopped!
  2. Beacon fires event: salt/beacon/web-server-1/service/nginx
  3. Master's reactor catches the event
  4. Reactor runs: salt 'web-server-1' service.restart nginx
  5. Nginx is back up — automatically! 🎉
  6. Total time: ~2 seconds

No human intervention needed!
```

### Watching Events in Real-Time

```bash
# Open this on the master to see ALL events as they happen
salt-run state.event pretty=True

# You'll see events like:
# salt/minion/web-server-1/start
# salt/job/20240101120000000000/new
# salt/beacon/web-server-1/diskusage/
# salt/auth
# etc.
```

## 18. Salt SSH — Agentless Mode

Don't want to install agents? **Salt SSH** works over SSH, just like Ansible — no minion required!

### When to Use Salt SSH

- Quick, one-time tasks on servers without Salt minions
- Testing Salt states before deploying minions
- Environments where you can't install agents
- Managing a small number of servers

### Setting Up Salt SSH

```yaml
# /etc/salt/roster — defines your SSH targets

web-server-1:
  host: 192.168.1.100
  user: ubuntu
  priv: /home/admin/.ssh/id_rsa
  sudo: True

web-server-2:
  host: 192.168.1.101
  user: ubuntu
  priv: /home/admin/.ssh/id_rsa
  sudo: True

db-server-1:
  host: 192.168.1.200
  user: ubuntu
  priv: /home/admin/.ssh/id_rsa
  sudo: True
  minion_opts:
    grains:
      role: database
```

### Running Commands with Salt SSH

```bash
# Ping all hosts in the roster
salt-ssh '*' test.ping

# Run a command
salt-ssh '*' cmd.run 'uptime'

# Apply a state
salt-ssh 'web-*' state.apply webserver

# Apply highstate
salt-ssh '*' state.highstate
```

### Salt SSH vs Regular Salt

| Feature | Salt (ZeroMQ) | Salt SSH |
|---------|--------------|----------|
| **Speed** | ⚡ Very fast | 🐢 Slower (SSH overhead) |
| **Agent** | Yes (salt-minion) | No (agentless) |
| **Scale** | 10,000+ nodes | Hundreds of nodes |
| **Setup** | Install minion on each node | Just need SSH access |
| **Events/Reactors** | ✅ Full support | ❌ Not available |
| **Beacons** | ✅ Full support | ❌ Not available |
| **Mine** | ✅ Full support | ❌ Not available |
| **Use Case** | Production, large infra | Quick tasks, testing |

## 19. Orchestration — Multi-System Workflows

Salt **orchestration** lets you coordinate operations across multiple systems in a specific order. This is different from highstate (which applies states independently to each minion).

### Example: Deploy a Web Application

```yaml
# /srv/salt/orch/deploy_app.sls

# Step 1: Put web servers in maintenance mode
maintenance_mode:
  salt.function:
    - name: cmd.run
    - tgt: 'web-*'
    - arg:
      - 'touch /var/www/html/maintenance.html'

# Step 2: Run database migrations
run_migrations:
  salt.state:
    - tgt: 'db-server-1'
    - sls:
      - database.migrate
    - require:
      - salt: maintenance_mode

# Step 3: Deploy new code to web servers
deploy_code:
  salt.state:
    - tgt: 'web-*'
    - sls:
      - myapp.deploy
    - require:
      - salt: run_migrations

# Step 4: Restart application servers (one at a time — rolling restart)
restart_app:
  salt.function:
    - name: service.restart
    - tgt: 'web-*'
    - arg:
      - myapp
    - batch: 1                    # One server at a time!
    - require:
      - salt: deploy_code

# Step 5: Remove maintenance mode
remove_maintenance:
  salt.function:
    - name: cmd.run
    - tgt: 'web-*'
    - arg:
      - 'rm -f /var/www/html/maintenance.html'
    - require:
      - salt: restart_app

# Step 6: Run health checks
health_check:
  salt.function:
    - name: http.query
    - tgt: 'web-server-1'
    - arg:
      - 'http://localhost/health'
    - require:
      - salt: remove_maintenance
```

### Running an Orchestration

```bash
# Run the orchestration
salt-run state.orchestrate orch.deploy_app

# Run with pillar data
salt-run state.orchestrate orch.deploy_app pillar='{"version": "2.1.0"}'
```

## 22. Common Gotchas and Best Practices

### ❌ Common Mistakes

```yaml
# ❌ DON'T use duplicate state IDs (Salt will error!)
install_nginx:
  pkg.installed:
    - name: nginx

install_nginx:            # ERROR: Duplicate ID!
  pkg.installed:
    - name: curl

# ✅ DO use unique IDs
install_nginx:
  pkg.installed:
    - name: nginx

install_curl:
  pkg.installed:
    - name: curl

# ❌ DON'T forget to quote mode values
myfile:
  file.managed:
    - mode: 0644          # This is an integer! Interpreted as 420 in decimal!

# ✅ DO quote mode values
myfile:
  file.managed:
    - mode: '0644'        # String — correct!

# ❌ DON'T forget ordering when it matters
nginx_running:
  service.running:
    - name: nginx         # Might fail if nginx isn't installed yet!

nginx_installed:
  pkg.installed:
    - name: nginx

# ✅ DO add require/watch relationships
nginx_running:
  service.running:
    - name: nginx
    - require:
      - pkg: nginx_installed
    - watch:
      - file: nginx_config

nginx_installed:
  pkg.installed:
    - name: nginx
```

### ✅ Best Practices

1. **Use pillar for ALL configuration values** — Never hardcode in states
2. **Use pillar for ALL secrets** — Passwords, API keys, certificates
3. **Quote file modes** — `'0644'` not `0644`
4. **Use formulas** — Don't reinvent the wheel
5. **Test with `test=True`** — Always dry-run before applying
6. **Use the `map.jinja` pattern** — Handle OS differences cleanly
7. **Keep states small** — One concern per state file (install, config, service)
8. **Use `require` and `watch`** — Define explicit relationships
9. **Use grains for targeting** — Set custom grains for roles and environments
10. **Use nodegroups** — Define logical groups of servers
11. **Version control everything** — States, pillars, formulas — all in Git
12. **Use `salt-call --local`** — For quick local testing without a master

## 23. When to Choose SaltStack

### ✅ Choose Salt When:

- **Speed is critical** — You need sub-second execution across thousands of servers
- **Real-time remote execution** — You need BOTH ad-hoc commands AND config management
- **Event-driven automation** — You want to react to events automatically (beacons + reactors)
- **Very large scale** — 5,000+ nodes (Salt was built for this)
- **Python expertise** — Your team knows Python and wants easy extensibility
- **Hybrid push/pull** — You want both on-demand (push) and scheduled (pull) modes
- **Orchestration** — You need to coordinate multi-system deployments

### ❌ Maybe Skip Salt When:

- You have a **small infrastructure** (< 50 servers) — Ansible is simpler
- You want the **simplest possible tool** — Ansible has a lower learning curve
- You don't want to manage a **master server** — Ansible doesn't need one
- You need the **largest community** — Ansible has the biggest ecosystem
- Your team doesn't know **Python or YAML** — other tools might fit better
- You need strong **enterprise compliance reporting** out of the box — Puppet Enterprise is better

## 24. Summary

| Concept | What It Is |
|---------|-----------|
| **Salt** | Fast configuration management + remote execution tool |
| **Master** | Central server that manages everything |
| **Minion** | Agent running on each managed server |
| **State** | YAML file describing desired configuration (`.sls`) |
| **Grain** | Static system info about a minion (OS, IP, CPU) |
| **Pillar** | Secure, minion-specific data (secrets, settings) |
| **Top File** | Maps states/pillars to minions |
| **Module** | Python code that does stuff (execution + state modules) |
| **Formula** | Reusable collection of states (like an Ansible role) |
| **Highstate** | Apply ALL assigned states to a minion |
| **Beacon** | Monitor something on a minion, fire events |
| **Reactor** | Automatically respond to events on the master |
| **Mine** | Share data between minions |
| **Salt SSH** | Agentless mode (SSH instead of ZeroMQ) |
| **Orchestration** | Coordinate operations across multiple systems |
| **ZeroMQ** | The fast messaging system that makes Salt blazing fast |
| **Jinja** | Templating engine for dynamic content in states/files |

### The Salt Workflow in a Nutshell

```
1. Install salt-master on your control server
2. Install salt-minion on managed servers → accept keys
3. Write states (YAML) defining desired configuration
4. Write pillar data (YAML) for secrets and variables
5. Create a top.sls mapping states to minions
6. Test with: salt '*' state.highstate test=True
7. Apply with: salt '*' state.highstate
8. Use remote execution for ad-hoc tasks: salt '*' cmd.run 'uptime'
9. Set up beacons for monitoring, reactors for auto-response
10. Everything happens in seconds — even across thousands of servers ⚡
```

### Quick Decision Matrix: Which Tool Should I Use?

```
Small team, simple needs, quick start     → Ansible
Large enterprise, compliance focus        → Puppet
Ruby expertise, testing culture           → Chef
Speed, scale, event-driven automation     → SaltStack
```
