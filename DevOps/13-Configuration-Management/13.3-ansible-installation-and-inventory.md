## 🛠️ Part 1: Installing Ansible

Ansible only needs to be installed on the **Control Node** (the machine you run Ansible FROM). The managed servers need nothing — just SSH and Python.

### Pre-requisites

Before installing Ansible, make sure you have:

```
✅ A Linux or macOS machine (or WSL2 on Windows)
✅ Python 3.9 or higher
✅ pip (Python package manager)
✅ SSH client (built into Linux/macOS)

Note: You CANNOT install Ansible natively on Windows.
      But you CAN use WSL2 (Windows Subsystem for Linux) — which is basically Linux inside Windows.
```

---

### Method 1: Install Using pip (Recommended — Works Everywhere)

This is the **cleanest and most portable** way to install Ansible.

```bash
# Step 1: Make sure Python 3 and pip are installed
python3 --version     # Should show Python 3.9+
pip3 --version        # Should show pip for Python 3

# If pip is not installed:
sudo apt update
sudo apt install -y python3-pip

# Step 2: Install Ansible using pip
pip3 install ansible

# Step 3: Verify the installation
ansible --version
```

**What you'll see:**

```
$ ansible --version
ansible [core 2.16.3]
  config file = None
  configured module search path = ['/home/user/.ansible/plugins/modules']
  ansible python module location = /home/user/.local/lib/python3.11/site-packages/ansible
  ansible collection location = /home/user/.ansible/collections
  executable location = /home/user/.local/bin/ansible
  python version = 3.11.6
  jinja version = 3.1.3
  libyaml = True
```

**Line by line — what this tells you:**

| Line | What It Means |
|------|--------------|
| `ansible [core 2.16.3]` | Ansible core version installed |
| `config file = None` | No ansible.cfg found yet (we'll create one) |
| `python version = 3.11.6` | Python version Ansible is using |
| `jinja version = 3.1.3` | Jinja2 templating engine version (used in templates) |

### Using a Virtual Environment (Best Practice)

It's a good practice to install Ansible in a Python virtual environment so it doesn't conflict with system packages:

```bash
# Create a virtual environment
python3 -m venv ~/ansible-env

# Activate it
source ~/ansible-env/bin/activate

# Install Ansible inside the virtual environment
pip install ansible

# Verify
ansible --version

# When you're done working, deactivate:
deactivate

# Next time, just activate again:
source ~/ansible-env/bin/activate
```

### Method 2: Install Using apt (Ubuntu/Debian)

```bash
# Add the Ansible PPA (Personal Package Archive)
sudo apt update
sudo apt install -y software-properties-common
sudo add-apt-repository --yes --update ppa:ansible/ansible

# Install Ansible
sudo apt install -y ansible

```

**When to use this method:**
- When you want Ansible to be available system-wide
- When you don't want to manage Python environments
- Note: This might give you an older version compared to pip

```bash

sudo dnf install -y ansible

# Verify

```

### Method 4: Install on macOS

```bash

brew install ansible

# Or using pip (recommended)
pip3 install ansible

# Verify
ansible --version
```

---

### Method 5: Windows — Using WSL2

You **cannot** install Ansible natively on Windows, but you can use WSL2:

```powershell
# Step 1: Install WSL2 (in PowerShell as Administrator)
wsl --install

# Step 2: Restart your computer

# Step 3: Open WSL (Ubuntu) terminal

sudo apt update
sudo apt install -y python3-pip
pip3 install ansible

# Step 5: Verify
ansible --version
```

### Installing ansible-core vs ansible — What's the Difference?

```
"ansible-core" (minimal):
    → Contains the Ansible engine (ansible, ansible-playbook, etc.)
    → Contains only a small set of built-in modules
    → Lightweight (~10-15 MB)
    → Install: pip install ansible-core

"ansible" (full package):
    → Includes ansible-core
    → PLUS the Ansible Community Package (thousands of modules & collections)
    → Includes modules for AWS, Azure, GCP, Docker, Kubernetes, etc.
    → Larger (~100+ MB)
    → Install: pip install ansible

Recommendation:
    → For learning: Install "ansible" (full package) — you get everything
    → For production: Install "ansible-core" + only the collections you need
```

## 📋 Part 2: Ansible Inventory — The Heart of "Who to Manage"

The **Inventory** is one of the most important concepts in Ansible. It tells Ansible **which servers to connect to** and **how to connect to them**.

### What is an Inventory?

```
Think of the inventory as your CONTACT LIST for servers:

Phone Contact List:
    → Mom — 555-0101
    → Dad — 555-0102
    → Boss — 555-0103

Ansible Inventory:
    → web-server-1 — 192.168.1.10
    → web-server-2 — 192.168.1.11
    → db-server-1 — 10.0.0.50

Just like your phone knows who to call and their numbers,
Ansible's inventory knows which servers to connect to and how.
```

---

### Inventory Formats

Ansible supports two inventory formats:

1. **INI format** (`.ini`) — Simple, human-readable
2. **YAML format** (`.yml`) — More structured, good for complex inventories

Let's learn both.

---

### INI Format Inventory (Most Common for Beginners)

```ini
# inventory.ini — The simplest possible inventory

# Just list server IPs or hostnames, one per line:
192.168.1.10
192.168.1.11
192.168.1.12
web4.example.com
```

That's it! You can now run Ansible against these servers:

```bash
ansible all -i inventory.ini -m ping
```

But wait — this isn't very organized. What if you have web servers, database servers, and monitoring servers? You don't want to install Nginx on your database servers!

#### Grouping Servers

```ini
# inventory.ini — Organized with groups

# =============================
# Web Servers
# =============================
[webservers]
web1.example.com
web2.example.com
web3.example.com
192.168.1.10

# =============================
# Database Servers
# =============================
[databases]
db1.example.com
db2.example.com

# =============================
# Monitoring Servers
# =============================
[monitoring]
grafana.example.com
prometheus.example.com

# =============================
# Load Balancers
# =============================
[loadbalancers]
lb1.example.com
```

Now you can target specific groups:

```bash
# Ping only web servers
ansible webservers -i inventory.ini -m ping

# Ping only database servers
ansible databases -i inventory.ini -m ping

# Ping ALL servers
ansible all -i inventory.ini -m ping
```

#### Adding Connection Details

Often, your servers need specific connection information:

```ini
# inventory.ini — With connection details

[webservers]
# Format: hostname ansible_host=IP ansible_user=USER ansible_port=PORT
web1 ansible_host=192.168.1.10 ansible_user=ubuntu ansible_port=22
web2 ansible_host=192.168.1.11 ansible_user=ubuntu
web3 ansible_host=192.168.1.12 ansible_user=ec2-user    # This one uses a different user

[databases]
db1 ansible_host=10.0.0.50 ansible_user=admin ansible_port=2222
db2 ansible_host=10.0.0.51 ansible_user=admin

[monitoring]
grafana ansible_host=172.16.0.10 ansible_user=ubuntu
```

**What each parameter means:**

| Parameter | What It Does | Example |
|-----------|-------------|---------|
| `ansible_host` | The IP address or hostname to connect to | `192.168.1.10` |
| `ansible_user` | The SSH username | `ubuntu`, `ec2-user`, `admin` |
| `ansible_port` | The SSH port (default: 22) | `2222` |
| `ansible_ssh_private_key_file` | Path to the SSH private key | `~/.ssh/mykey.pem` |
| `ansible_become` | Whether to use sudo | `yes` |
| `ansible_become_user` | Which user to become with sudo | `root` |
| `ansible_become_password` | Sudo password (avoid in plain text!) | `MyP@ssw0rd` |
| `ansible_python_interpreter` | Path to Python on the managed node | `/usr/bin/python3` |
| `ansible_connection` | Connection type | `ssh`, `local`, `winrm` |

#### Example with SSH Keys (Common in AWS/Cloud)

```ini
# inventory.ini — AWS EC2 instances

[webservers]
web1 ansible_host=54.123.45.67 ansible_user=ec2-user ansible_ssh_private_key_file=~/.ssh/aws-key.pem
web2 ansible_host=54.123.45.68 ansible_user=ec2-user ansible_ssh_private_key_file=~/.ssh/aws-key.pem

[databases]
db1 ansible_host=10.0.1.50 ansible_user=ubuntu ansible_ssh_private_key_file=~/.ssh/aws-key.pem
```

#### Group Variables (Avoid Repeating Yourself)

If all servers in a group use the same user and key, you can set **group variables** instead of repeating them:

```ini
# inventory.ini — Using group variables

[webservers]
web1 ansible_host=54.123.45.67
web2 ansible_host=54.123.45.68
web3 ansible_host=54.123.45.69

[webservers:vars]                          # Variables for ALL servers in "webservers"
ansible_user=ec2-user
ansible_ssh_private_key_file=~/.ssh/aws-key.pem
ansible_become=yes

[databases]
db1 ansible_host=10.0.1.50
db2 ansible_host=10.0.1.51

[databases:vars]                           # Variables for ALL servers in "databases"
ansible_user=ubuntu
ansible_ssh_private_key_file=~/.ssh/db-key.pem
ansible_become=yes
```

**What `[webservers:vars]` means:** "Apply these variables to every server in the `webservers` group." This saves you from repeating `ansible_user=ec2-user` on every single line.

#### Groups of Groups (Parent Groups)

You can create parent groups that contain other groups:

```ini
# inventory.ini — Nested groups

[webservers]
web1 ansible_host=192.168.1.10
web2 ansible_host=192.168.1.11

[appservers]
app1 ansible_host=192.168.1.20
app2 ansible_host=192.168.1.21

[databases]
db1 ansible_host=192.168.1.30
db2 ansible_host=192.168.1.31

# Parent group: "production" contains webservers, appservers, and databases
[production:children]
webservers
appservers
databases

# Now you can target ALL production servers at once:
# ansible production -i inventory.ini -m ping

# Another parent group
[staging]
staging-web1 ansible_host=10.0.0.10
staging-app1 ansible_host=10.0.0.20
staging-db1 ansible_host=10.0.0.30

# Grand-parent group: "all_environments" contains everything
[all_environments:children]
production
staging
```

**How groups work visually:**

```
                    all (built-in, always exists)
                    ├── production
                    │   ├── webservers
                    │   │   ├── web1
                    │   │   └── web2
                    │   ├── appservers
                    │   │   ├── app1
                    │   │   └── app2
                    │   └── databases
                    │       ├── db1
                    │       └── db2
                    ├── staging
                    │   ├── staging-web1
                    │   ├── staging-app1
                    │   └── staging-db1
                    └── ungrouped
                        └── (any servers not in a group)
```

**Built-in groups:** Ansible automatically creates two groups:
- `all` — contains every server in the inventory
- `ungrouped` — contains servers that aren't in any group

---

### YAML Format Inventory

The same inventory can be written in YAML (some people prefer this for complex inventories):

```yaml
# inventory.yml — YAML format inventory
all:
  children:
    production:
      children:
        webservers:
          hosts:
            web1:
              ansible_host: 192.168.1.10
            web2:
              ansible_host: 192.168.1.11
            web3:
              ansible_host: 192.168.1.12
          vars:                              # Group variables for webservers
            ansible_user: ec2-user
            ansible_ssh_private_key_file: ~/.ssh/aws-key.pem
            http_port: 80

        databases:
          hosts:
            db1:
              ansible_host: 10.0.1.50
              db_port: 5432                  # Host-specific variable
            db2:
              ansible_host: 10.0.1.51
              db_port: 3306                  # Different database port
          vars:
            ansible_user: ubuntu
            ansible_become: yes

        monitoring:
          hosts:
            grafana:
              ansible_host: 172.16.0.10
            prometheus:
              ansible_host: 172.16.0.11

    staging:
      hosts:
        staging-web:
          ansible_host: 10.0.0.10
        staging-db:
          ansible_host: 10.0.0.20
      vars:
        ansible_user: admin
        environment: staging
```

**INI vs YAML — Which to Use?**

| Aspect | INI Format | YAML Format |
|--------|-----------|-------------|
| **Simplicity** | ✅ Simpler for small inventories | More verbose |
| **Readability** | Good for flat lists | ✅ Better for nested structures |
| **Variables** | Limited nesting | ✅ Full nesting support |
| **When to use** | Learning, small environments | Large, complex environments |
| **File extension** | `.ini` or no extension | `.yml` or `.yaml` |

**My recommendation:** Start with INI format (simpler), switch to YAML when your inventory gets complex.

### Using group_vars and host_vars Directories

Instead of putting variables directly in the inventory file (which can get messy), you can use **directories**:

```
my-ansible-project/
├── inventory.ini
├── group_vars/                   # Variables by GROUP
│   ├── all.yml                   # Applied to ALL servers
│   ├── webservers.yml            # Applied to webservers group
│   └── databases.yml             # Applied to databases group
└── host_vars/                    # Variables by individual HOST
    ├── web1.yml                  # Applied to web1 only
    └── db1.yml                   # Applied to db1 only
```

```yaml
# group_vars/all.yml — Applied to EVERY server
---
ntp_server: time.google.com
timezone: UTC
ssh_port: 22
```

```yaml
# group_vars/webservers.yml — Applied to all web servers
---
http_port: 80
https_port: 443
nginx_worker_processes: auto
```

```yaml
# group_vars/databases.yml — Applied to all database servers
---
db_backup_enabled: true
db_backup_time: "02:00"
max_connections: 200
```

```yaml
# host_vars/web1.yml — Applied to web1 ONLY
---
nginx_worker_connections: 2048    # This server needs more connections
custom_domain: web1.example.com
```

**Why use directories instead of inline variables?**
- Cleaner, more organized
- Easier to manage in Git
- Each file can be edited independently
- Better for team collaboration

### Variable Precedence — Which Variable Wins?

When the same variable is defined in multiple places, Ansible follows a **precedence order**. Here's a simplified version (lower number = lower priority):

```
Priority (low → high):
    1. group_vars/all.yml           ← Lowest priority (can be overridden by anything)
    2. group_vars/<group>.yml       ← Group-specific
    3. host_vars/<host>.yml         ← Host-specific (overrides group vars)
    4. Playbook vars                ← In the playbook itself
    5. Command-line vars (-e)       ← Highest priority (overrides everything)
```

**Example:**

```yaml
# group_vars/all.yml
http_port: 80

# group_vars/webservers.yml
http_port: 8080           # Overrides all.yml for webservers

# host_vars/web1.yml
http_port: 9090            # Overrides webservers.yml for web1 only
```

```bash
# Command line override (highest priority):
ansible-playbook site.yml -e "http_port=3000"    # Overrides EVERYTHING
```

## 🧪 Hands-On Lab: Your First Ansible Setup

Let's put everything together with a practical exercise. We'll set up Ansible and test it.

### Step 1: Install Ansible

```bash
# On Ubuntu/Debian:
sudo apt update
sudo apt install -y python3-pip
pip3 install ansible

# Verify:
ansible --version
```

### Step 2: Create a Project Directory

```bash
mkdir ~/my-first-ansible
cd ~/my-first-ansible
```

### Step 3: Create the ansible.cfg File

```bash
cat > ansible.cfg << 'EOF'
[defaults]
inventory = ./inventory.ini
remote_user = ubuntu
host_key_checking = False
timeout = 30
forks = 10

[privilege_escalation]
become = True
become_method = sudo
become_user = root
become_ask_pass = False
EOF
```

**What each section means:**

| Setting | Meaning |
|---------|---------|
| `inventory = ./inventory.ini` | Default inventory file location |
| `remote_user = ubuntu` | SSH as "ubuntu" by default |
| `host_key_checking = False` | Don't prompt for SSH key verification |
| `become = True` | Use sudo by default |
| `become_method = sudo` | Use sudo (not su or other methods) |
| `become_ask_pass = False` | Don't ask for sudo password (assumes passwordless sudo) |

### Step 4: Create the Inventory File

If you don't have remote servers, you can test with **localhost**:

```ini
# inventory.ini — Testing with localhost

[local]
localhost ansible_connection=local

# If you have remote servers, add them here:
# [webservers]
# server1 ansible_host=192.168.1.10 ansible_user=ubuntu
# server2 ansible_host=192.168.1.11 ansible_user=ubuntu
```

**What `ansible_connection=local` means:** Instead of connecting via SSH, Ansible will run commands directly on the local machine. Perfect for testing!

### Step 5: Test the Connection

```bash
# Ping all hosts in the inventory
ansible all -m ping
```

**Expected output:**

```
localhost | SUCCESS => {
    "changed": false,
    "ping": "pong"
}
```

🎉 **Ansible is working!**

### Step 6: Run Some Ad-Hoc Commands

Ad-hoc commands are one-liner Ansible commands (no playbook needed). Great for quick tasks:

```bash
# Check the OS of all servers
ansible all -m command -a "uname -a"

# Check disk usage
ansible all -m command -a "df -h"

# Check memory usage
ansible all -m command -a "free -m"

# Gather all facts about the servers
ansible all -m setup

# Gather only specific facts
ansible all -m setup -a "filter=ansible_os_family"

# Install a package (on localhost)
ansible all -m apt -a "name=htop state=present" --become

# Create a file
ansible all -m file -a "path=/tmp/ansible-test.txt state=touch"

# Copy content to a file
ansible all -m copy -a "content='Hello from Ansible!' dest=/tmp/hello.txt"

# Check if the file was created
ansible all -m command -a "cat /tmp/hello.txt"
```

**Understanding the ad-hoc command syntax:**

```
ansible <target> -m <module> -a "<arguments>"

Where:
    <target>    = Which servers (all, webservers, db1, etc.)
    -m <module> = Which module to use (ping, command, apt, copy, etc.)
    -a "<args>" = Arguments for the module
    --become    = Run with sudo
    -i <file>   = Specify inventory file (if not in ansible.cfg)
```

### Step 7: Test with a Simple Playbook

Let's create your first playbook!

```yaml

  hosts: local

    - name: Show the date

        msg: "The current date is: {{ current_date.stdout }}"

    - name: Create a directory
      file:

        state: directory
        mode: '0755'

      command: cat /tmp/ansible-demo/info.txt
      register: file_content
```

## 🌐 Dynamic Inventory — For Cloud Environments

So far, we've been using **static inventory** — a file where you manually list servers. But in cloud environments (AWS, Azure, GCP), servers come and go all the time. You don't want to manually update the inventory every time a server is created or destroyed.

That's where **Dynamic Inventory** comes in.

### What is Dynamic Inventory?

```
Static Inventory:
    → You manually write the server list
    → File stays the same until YOU change it
    → Good for fixed environments (on-premise, stable infrastructure)

Dynamic Inventory:
    → A script or plugin that AUTOMATICALLY discovers servers
    → Queries your cloud provider's API: "Hey AWS, what EC2 instances are running?"
    → Generates the inventory on the fly
    → Always up-to-date!
    → Good for cloud environments where servers are created/destroyed frequently
```

### Example: AWS Dynamic Inventory

```yaml
# aws_ec2.yml — Dynamic inventory plugin for AWS
plugin: amazon.aws.aws_ec2
regions:
  - us-east-1
  - us-west-2

# Filter: Only get running instances
filters:
  instance-state-name: running

# Create groups based on tags
keyed_groups:
  - key: tags.Environment       # Group by "Environment" tag
    prefix: env                  # Results in groups like: env_production, env_staging
  - key: tags.Role              # Group by "Role" tag
    prefix: role                 # Results in groups like: role_webserver, role_database
  - key: instance_type          # Group by instance type
    prefix: type                 # Results in groups like: type_t2_medium

# How to connect
compose:
  ansible_host: public_ip_address    # Use the public IP to connect
```

```bash
# Use the dynamic inventory:
ansible-inventory -i aws_ec2.yml --list    # Show all discovered servers
ansible-inventory -i aws_ec2.yml --graph   # Show the group hierarchy

# Run a playbook with dynamic inventory:
ansible-playbook -i aws_ec2.yml setup.yml
```

**Output might look like:**

```
@all:
  |--@env_production:
  |  |--ec2-54-123-45-67.compute.amazonaws.com
```

## 🔍 Inventory Commands — Useful Day-to-Day

Here are the most useful inventory-related commands:

```bash
# List ALL hosts in the inventory
ansible all --list-hosts

# List hosts in a specific group
ansible webservers --list-hosts

# Show the complete inventory as JSON
ansible-inventory --list

# Show the inventory as a tree/graph
ansible-inventory --graph

# Show variables for a specific host
ansible-inventory --host web1

# Verify the inventory is valid
ansible-inventory --list > /dev/null && echo "Inventory is valid!" || echo "Inventory has errors!"
```

## 🧠 Inventory Patterns — Targeting Specific Hosts

When running Ansible commands, you can use **patterns** to target specific hosts:

```bash
# Target ALL hosts
ansible all -m ping

# Target a specific group
ansible webservers -m ping

# Target a specific host
ansible web1 -m ping

# Target multiple groups (OR — servers in webservers OR databases)
ansible 'webservers:databases' -m ping

# Target intersection (AND — servers in BOTH webservers AND production)
ansible 'webservers:&production' -m ping

# Exclude a group (webservers but NOT the "maintenance" group)
ansible 'webservers:!maintenance' -m ping

# Use wildcards
ansible 'web*' -m ping             # All hosts starting with "web"
ansible '192.168.1.*' -m ping      # All hosts in the 192.168.1.x subnet

# Target by index (first web server)
ansible 'webservers[0]' -m ping

# Target a range of hosts
ansible 'webservers[0:2]' -m ping  # First 3 web servers

# Use regex
ansible '~web[0-9]+\.example\.com' -m ping
```

## ⚙️ The ansible.cfg File — Configuring Ansible's Behavior

Ansible looks for its configuration in this order (first found wins):

```
1. ANSIBLE_CONFIG  (environment variable — highest priority)
2. ./ansible.cfg   (current directory)
3. ~/.ansible.cfg  (home directory)
4. /etc/ansible/ansible.cfg  (system-wide — lowest priority)

Recommendation: Put ansible.cfg in your PROJECT DIRECTORY
                 and version-control it with Git!
```

### Complete ansible.cfg Example with Explanations

```ini
# ansible.cfg — Full configuration example

[defaults]
# ===== INVENTORY =====
inventory = ./inventory/production.ini    # Default inventory file

# ===== CONNECTIONS =====
remote_user = ubuntu                      # Default SSH user
timeout = 30                              # SSH connection timeout (seconds)
forks = 20                                # Max parallel connections
host_key_checking = False                 # Don't verify SSH host keys

# ===== OUTPUT =====
stdout_callback = yaml                    # Pretty YAML output (much nicer!)
callback_whitelist = timer, profile_tasks # Show timing info for tasks
display_skipped_hosts = False             # Don't show skipped hosts in output
nocows = True                             # Disable cowsay (yes, really!)

# ===== ROLES =====
roles_path = ./roles                      # Where to find roles

# ===== RETRY FILES =====
retry_files_enabled = False               # Don't create .retry files

# ===== FACTS =====
gathering = smart                         # Only gather facts when needed
fact_caching = jsonfile                   # Cache facts to speed things up
fact_caching_connection = /tmp/ansible_facts_cache
fact_caching_timeout = 3600               # Cache for 1 hour

[privilege_escalation]
become = True                             # Use sudo by default
become_method = sudo                      # Use sudo (not su, pbrun, etc.)
become_user = root                        # Become root
become_ask_pass = False                   # Don't ask for sudo password

[ssh_connection]
pipelining = True                         # Speed up execution
ssh_args = -o ControlMaster=auto -o ControlPersist=60s -o ForwardAgent=yes
control_path_dir = /tmp/.ansible/cp       # SSH control path
```

**Key settings explained:**

| Setting | What It Does | Recommended Value |
|---------|-------------|-------------------|
| `forks = 20` | How many servers to configure in parallel | 10-50 (depends on your control node's resources) |
| `stdout_callback = yaml` | Makes the output pretty and readable | `yaml` (much nicer than default) |
| `pipelining = True` | Reduces SSH operations (faster execution) | `True` (but may need `requiretty` disabled in sudoers) |
| `gathering = smart` | Only gather facts when they haven't been cached | `smart` (saves time) |
| `host_key_checking = False` | Don't ask "Are you sure?" for new SSH connections | `False` for labs; `True` for high-security environments |

## 📝 Summary

| Concept | Key Takeaway |
|---------|-------------|
| **Installation** | `pip install ansible` on the control node — nothing on managed servers |
| **Control Node** | Must be Linux/macOS (or WSL2 on Windows) |
| **ansible vs ansible-core** | `ansible` = full package with all collections; `ansible-core` = minimal |
| **Inventory** | A file listing all servers to manage, organized into groups |
| **INI vs YAML** | INI is simpler; YAML is better for complex inventories |
| **Groups** | Organize servers by role (webservers, databases, monitoring) |
| **Group variables** | `[group:vars]` or `group_vars/` directory — apply vars to entire groups |
| **Host variables** | `host_vars/` directory — apply vars to specific hosts |
| **Children groups** | `[parent:children]` — create group hierarchies |
| **Dynamic Inventory** | Plugins that auto-discover servers from cloud providers |
| **ansible.cfg** | Configuration file — put in project directory, version-control it |
| **Patterns** | Target specific hosts using groups, wildcards, regex, set operations |
| **Ad-hoc commands** | Quick one-liners: `ansible <target> -m <module> -a "<args>"` |
