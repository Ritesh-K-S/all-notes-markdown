# 13.4 — Ansible — Ad-Hoc Commands

> **One Line Answer:** Ansible ad-hoc commands are **quick, one-liner commands** that you run directly from your terminal to perform **simple tasks on remote servers** — like checking uptime, installing a package, or restarting a service — **without writing a playbook**. Think of them as the "quick text message" vs the "formal email" (playbooks).

## 🤔 What Are Ad-Hoc Commands?

Let's say you have 50 servers and you want to quickly check if all of them are reachable. Do you really need to write a whole playbook (a YAML file with tasks, structure, etc.) just for that? **No!**

That's where **ad-hoc commands** come in. They're quick, one-time commands you fire from your terminal. They use Ansible modules behind the scenes, but you don't need to create any files.

### When to Use Ad-Hoc vs Playbooks:

| Scenario | Use Ad-Hoc? | Use Playbook? |
|----------|-------------|---------------|
| Check if 50 servers are alive | ✅ Yes | ❌ Overkill |
| Restart a single service | ✅ Yes | ❌ Overkill |
| Set up an entire web server with Nginx, SSL, configs | ❌ Too complex | ✅ Yes |
| Install one package quickly | ✅ Yes | ❌ Overkill |
| Deploy an entire application stack | ❌ Too complex | ✅ Yes |
| Check disk space across servers | ✅ Yes | ❌ Overkill |
| Configure users, permissions, firewall rules | ❌ Too many tasks | ✅ Yes |

**Rule of thumb:** If it's a **quick, one-off task** → ad-hoc command. If it's **repeatable or multi-step** → playbook.

## 📝 Syntax of an Ad-Hoc Command

Here's the general structure:

```bash
ansible <host-pattern> -m <module-name> -a "<module-arguments>" [options]
```

Let's break this down piece by piece:

```
ansible          → The Ansible CLI command
<host-pattern>   → WHICH servers to target (a group name, hostname, IP, or "all")
-m               → Specifies which MODULE to use
<module-name>    → The actual module (ping, shell, copy, apt, yum, service, etc.)
-a               → Specifies the ARGUMENTS to pass to the module
"<arguments>"    → The actual arguments (in quotes)
[options]        → Extra options like -i (inventory), -u (user), -b (become/sudo), etc.
```

### Visual breakdown:

```
ansible   webservers   -m   apt   -a   "name=nginx state=present"   -b
  │          │          │    │      │           │                      │
  │          │          │    │      │           │                      └─ Run as sudo (become)
  │          │          │    │      │           └─ Arguments: install nginx
  │          │          │    │      └─ Flag: arguments follow
  │          │          │    └─ Module: apt (Debian/Ubuntu package manager)
  │          │          └─ Flag: module follows
  │          └─ Target: the "webservers" group from inventory
  └─ The Ansible command
```

## 🎯 Common Options You'll Use

Before we dive into examples, let's understand the most common flags:

| Option | Short | What It Does | Example |
|--------|-------|-------------|---------|
| `--module-name` | `-m` | Which module to use | `-m ping` |
| `--args` | `-a` | Arguments for the module | `-a "name=nginx"` |
| `--inventory` | `-i` | Path to inventory file | `-i /etc/ansible/hosts` |
| `--become` | `-b` | Run with sudo/root privileges | `-b` |
| `--become-user` | | Specify which user to become | `--become-user root` |
| `--user` | `-u` | Connect as this remote user | `-u ubuntu` |
| `--ask-become-pass` | `-K` | Prompt for the sudo password | `-K` |
| `--ask-pass` | `-k` | Prompt for SSH password | `-k` |
| `--forks` | `-f` | How many servers to run on in parallel | `-f 10` |
| `--limit` | `-l` | Further limit the hosts | `-l server1` |
| `--check` | `-C` | Dry run — don't actually make changes | `-C` |
| `--verbose` | `-v` | Show more output (use -vvv for max) | `-vvv` |
| `--private-key` | | Path to SSH private key | `--private-key ~/.ssh/id_rsa` |

## 📋 Let's Set Up a Sample Inventory First

Before running ad-hoc commands, we need an **inventory file** — this tells Ansible which servers to manage.

Let's create a simple inventory file:

```ini
# File: /etc/ansible/hosts   (or you can put it anywhere and use -i flag)

[webservers]
web1 ansible_host=192.168.1.10 ansible_user=ubuntu
web2 ansible_host=192.168.1.11 ansible_user=ubuntu

[dbservers]
db1 ansible_host=192.168.1.20 ansible_user=centos
db2 ansible_host=192.168.1.21 ansible_user=centos

[loadbalancers]
lb1 ansible_host=192.168.1.30 ansible_user=ubuntu

[all:vars]
ansible_python_interpreter=/usr/bin/python3
ansible_ssh_private_key_file=~/.ssh/id_rsa
```

Now let's see what each part means:

```
[webservers]                  ← Group name (you pick any name)
web1                          ← Alias for the server (you pick any name)
  ansible_host=192.168.1.10   ← Actual IP or hostname of the server
  ansible_user=ubuntu          ← Username to SSH as

[all:vars]                    ← Variables applied to ALL hosts
  ansible_python_interpreter   ← Where Python is on the remote server
  ansible_ssh_private_key_file ← SSH key to use
```

## 🟢 Category 1: Connectivity & Information Gathering

### 1. Ping All Servers (Test Connectivity)

```bash
ansible all -m ping
```

This doesn't do a network ping (ICMP). It's Ansible's way of checking:
- ✅ Can I SSH into the server?
- ✅ Is Python installed on the server?
- ✅ Can Ansible run things on the server?

**Output (success):**
```json
web1 | SUCCESS => {
    "changed": false,
    "ping": "pong"
}
web2 | SUCCESS => {
    "changed": false,
    "ping": "pong"
}
```

**Output (failure):**
```json
web1 | UNREACHABLE! => {
    "changed": false,
    "msg": "Failed to connect to the host via ssh",
    "unreachable": true
}
```

### 2. Ping Only a Specific Group

```bash
# Only webservers
ansible webservers -m ping

# Only dbservers
ansible dbservers -m ping

# Only a single host
ansible web1 -m ping
```

### 3. Gather Facts (System Information)

```bash
ansible web1 -m setup
```

This returns a **massive** JSON with EVERYTHING about the server:
- OS, kernel version, architecture
- CPU, RAM, disk info
- Network interfaces, IP addresses
- Hostname, FQDN
- And much more...

**Too much info? Filter it:**

```bash
# Only get memory info
ansible web1 -m setup -a "filter=ansible_memtotal_mb"

# Only get OS-related info
ansible web1 -m setup -a "filter=ansible_os_family"

# Only get network-related info
ansible web1 -m setup -a "filter=ansible_default_ipv4"

# Wildcard filter — get all "ansible_eth*" facts
ansible web1 -m setup -a "filter=ansible_eth*"
```

**Example output for memory filter:**
```json
web1 | SUCCESS => {
    "ansible_facts": {
        "ansible_memtotal_mb": 3946
    },
    "changed": false
}
```

### 4. Check Uptime

```bash
ansible all -m command -a "uptime"
```

**Output:**
```
web1 | CHANGED | rc=0 >>
 14:23:45 up 32 days,  6:12,  1 user,  load average: 0.15, 0.10, 0.08

web2 | CHANGED | rc=0 >>
 14:23:45 up 15 days,  2:30,  1 user,  load average: 0.23, 0.18, 0.12
```

### 5. Check Disk Space

```bash
ansible all -m command -a "df -h"
```

### 6. Check Free Memory

```bash
ansible all -m shell -a "free -m"
```

## 🔵 Category 2: command vs shell Module — What's the Difference?

This confuses a lot of people, so let's clear it up:

### `command` Module (Default)

```bash
ansible all -m command -a "ls /tmp"
```

- Executes a command **directly** (no shell involved)
- ❌ Does NOT support pipes (`|`), redirects (`>`), environment variables (`$HOME`), or wildcards (`*`)
- ✅ Safer — no risk of shell injection attacks
- This is the **default module** — if you don't specify `-m`, Ansible uses `command`

### `shell` Module

```bash
ansible all -m shell -a "cat /var/log/syslog | grep error | tail -5"
```

- Executes command **through a shell** (`/bin/sh`)
- ✅ Supports pipes, redirects, environment variables, wildcards
- ⚠️ Less safe — possible shell injection if using untrusted input

### When to Use Which?

```
Need pipes, redirects, or env vars?
├── YES → Use shell module
│         ansible all -m shell -a "cat /var/log/syslog | grep error"
│
└── NO  → Use command module (or just skip -m, since it's the default)
          ansible all -m command -a "ls /tmp"
          ansible all -a "ls /tmp"    ← Same thing! command is default
```

### `raw` Module — For Extreme Cases

```bash
ansible all -m raw -a "apt-get install -y python3"
```

- Runs a raw SSH command — **doesn't even require Python** on the remote server
- Useful for bootstrapping (installing Python before Ansible can work properly)
- Very rarely used

## 🟡 Category 3: File & Directory Operations

### 1. Create a Directory

```bash
ansible webservers -m file -a "path=/opt/myapp state=directory mode=0755 owner=ubuntu group=ubuntu" -b
```

Breakdown:
- `path=/opt/myapp` → Path to create
- `state=directory` → We want a directory (not a file)
- `mode=0755` → Permissions (rwxr-xr-x)
- `owner=ubuntu` → Owner of the directory
- `group=ubuntu` → Group of the directory
- `-b` → Use sudo (because /opt requires root)

### 2. Create an Empty File

```bash
ansible webservers -m file -a "path=/tmp/hello.txt state=touch mode=0644"
```

### 3. Delete a File or Directory

```bash
# Delete a file
ansible webservers -m file -a "path=/tmp/hello.txt state=absent"

# Delete a directory (and everything inside it)
ansible webservers -m file -a "path=/opt/old-app state=absent" -b
```

`state=absent` = "make sure this does NOT exist" (deletes it if it does)

### 4. Copy a File FROM Your Machine TO Remote Servers

```bash
ansible webservers -m copy -a "src=/home/user/config.conf dest=/etc/myapp/config.conf owner=root group=root mode=0644" -b
```

Breakdown:
- `src=` → File on YOUR local machine (the control node)
- `dest=` → Where to put it on the REMOTE servers
- `owner/group/mode` → Set permissions on the remote file
- `-b` → Use sudo

### 5. Fetch a File FROM Remote Servers TO Your Machine

```bash
ansible webservers -m fetch -a "src=/var/log/syslog dest=/tmp/logs/ flat=no"
```

This is the **reverse of copy** — it pulls files from remote servers to your machine.
- `flat=no` → Saves as `/tmp/logs/web1/var/log/syslog` (preserves directory structure per host)
- `flat=yes` → Saves as `/tmp/logs/syslog` (but only works with a single host)

### 6. Change File Permissions

```bash
ansible webservers -m file -a "path=/opt/myapp/script.sh mode=0755" -b
```

### 7. Create a Symbolic Link

```bash
ansible webservers -m file -a "src=/opt/myapp/current dest=/opt/myapp/latest state=link"
```

## 🟠 Category 4: Package Management

### 1. Install a Package (Debian/Ubuntu — apt)

```bash
ansible webservers -m apt -a "name=nginx state=present update_cache=yes" -b
```

Breakdown:
- `name=nginx` → Package to install
- `state=present` → Make sure it's installed (if already installed, do nothing)
- `update_cache=yes` → Run `apt-get update` first
- `-b` → Use sudo

### 2. Install a Package (CentOS/RHEL — yum)

```bash
ansible dbservers -m yum -a "name=httpd state=present" -b
```

### 3. Install a Specific Version

```bash
ansible webservers -m apt -a "name=nginx=1.18.0-0ubuntu1 state=present" -b
```

### 4. Install Multiple Packages

```bash
ansible webservers -m apt -a "name=git,curl,wget,vim state=present" -b
```

### 5. Remove (Uninstall) a Package

```bash
ansible webservers -m apt -a "name=nginx state=absent" -b
```

`state=absent` → "Make sure this is NOT installed" (removes it if it is)

### 6. Upgrade All Packages

```bash
ansible webservers -m apt -a "upgrade=yes update_cache=yes" -b
```

### 7. Install a Package with the Generic `package` Module

```bash
ansible all -m package -a "name=git state=present" -b
```

The `package` module automatically detects whether to use `apt`, `yum`, `dnf`, etc., based on the OS. **Very useful when you have mixed environments!**

## 🔴 Category 5: Service Management

### 1. Start a Service

```bash
ansible webservers -m service -a "name=nginx state=started" -b
```

### 2. Stop a Service

```bash
ansible webservers -m service -a "name=nginx state=stopped" -b
```

### 3. Restart a Service

```bash
ansible webservers -m service -a "name=nginx state=restarted" -b
```

### 4. Enable a Service to Start at Boot

```bash
ansible webservers -m service -a "name=nginx enabled=yes" -b
```

### 5. Start AND Enable (Common Combo)

```bash
ansible webservers -m service -a "name=nginx state=started enabled=yes" -b
```

This means:
- Start Nginx right now
- AND make sure it auto-starts whenever the server reboots

### 6. Check Service Status (Using shell)

```bash
ansible webservers -m shell -a "systemctl status nginx"
```

## 🟣 Category 6: User & Group Management

### 1. Create a User

```bash
ansible webservers -m user -a "name=deploy state=present shell=/bin/bash createhome=yes" -b
```

Breakdown:
- `name=deploy` → Username
- `state=present` → Make sure the user exists
- `shell=/bin/bash` → Default shell
- `createhome=yes` → Create a home directory at /home/deploy

### 2. Create a User with a Specific UID

```bash
ansible webservers -m user -a "name=appuser uid=1050 group=www-data" -b
```

### 3. Delete a User

```bash
ansible webservers -m user -a "name=olduser state=absent remove=yes" -b
```

- `state=absent` → Remove the user
- `remove=yes` → Also delete their home directory

### 4. Add a User to a Group

```bash
ansible webservers -m user -a "name=deploy groups=sudo,docker append=yes" -b
```

- `groups=sudo,docker` → Add to these groups
- `append=yes` → **IMPORTANT!** Add to these groups WITHOUT removing from existing groups

### 5. Create a Group

```bash
ansible webservers -m group -a "name=developers state=present" -b
```

## ⚡ Category 7: Advanced Ad-Hoc Patterns

### 1. Targeting Multiple Groups

```bash
# Servers in BOTH webservers AND dbservers
ansible 'webservers:dbservers' -m ping

# Servers in webservers BUT NOT dbservers
ansible 'webservers:!dbservers' -m ping

# Servers in webservers AND dbservers (intersection)
ansible 'webservers:&dbservers' -m ping
```

### 2. Using Wildcards

```bash
# All servers starting with "web"
ansible 'web*' -m ping

# All servers with a specific IP range
ansible '192.168.1.*' -m ping
```

### 3. Limit to Specific Hosts

```bash
ansible webservers -m ping --limit web1
```

This targets the `webservers` group but only runs on `web1`.

### 4. Run with Increased Parallelism

```bash
# Default is 5 parallel connections, bump it up
ansible all -m ping -f 20
```

This runs the command on **20 servers simultaneously** instead of the default 5.

### 5. Dry Run (Check Mode)

```bash
ansible webservers -m apt -a "name=nginx state=present" -b --check
```

This shows you **what WOULD happen** without actually doing anything. Super useful before making changes!

### 6. Run a Command as a Different User

```bash
ansible webservers -m command -a "whoami" --become --become-user=www-data
```

### 7. Using a Custom Inventory File

```bash
ansible -i /path/to/my/inventory all -m ping
```

### 8. One-liner to Check a Specific Port

```bash
ansible webservers -m shell -a "ss -tlnp | grep ':80'"
```

## 🧠 Understanding Idempotency in Ad-Hoc Commands

This is a **super important concept** in Ansible. Let me explain it simply:

**Idempotent** = Running the same command multiple times always gives the **same result** without unintended side effects.

```
Example:
    "Install nginx" → Run it 1 time → Installs nginx    ✅
    "Install nginx" → Run it 5 more times → Does nothing (already installed) ✅
    
    That's idempotent! It checks first, then acts only if needed.
```

### Idempotent vs Non-Idempotent Modules:

| Module | Idempotent? | Why? |
|--------|------------|------|
| `apt` / `yum` | ✅ Yes | Checks if package is already installed |
| `service` | ✅ Yes | Checks if service is already running |
| `copy` | ✅ Yes | Checks if file already exists with same content |
| `file` | ✅ Yes | Checks if directory/file already exists |
| `user` | ✅ Yes | Checks if user already exists |
| `command` | ❌ No | Always runs the command (no way to check) |
| `shell` | ❌ No | Always runs the command (no way to check) |
| `raw` | ❌ No | Always runs the command (no way to check) |

**Best Practice:** Prefer idempotent modules (`apt`, `service`, `copy`, `file`, `user`) over `command`/`shell` whenever possible.

If you MUST use `command`/`shell`, use `creates` or `removes` to make them idempotent:

```bash
# Only run if /opt/myapp does NOT exist
ansible webservers -m command -a "tar xzf /tmp/myapp.tar.gz -C /opt/" -a "creates=/opt/myapp"

# Only run if /tmp/cleanup-needed exists
ansible webservers -m command -a "/opt/scripts/cleanup.sh" -a "removes=/tmp/cleanup-needed"
```

## 📊 Understanding the Output — Return Values

When you run an ad-hoc command, Ansible gives you color-coded output:

```
🟢 GREEN  = SUCCESS (no changes made — already in desired state)
🟡 YELLOW = CHANGED (success AND something was changed)
🔴 RED    = FAILED  (something went wrong)
🟤 DARK RED = UNREACHABLE (can't connect to the server)
```

**Example output with details:**

```json
web1 | CHANGED | rc=0 >>
Hello World
```

- `web1` → Which host
- `CHANGED` → Something was modified
- `rc=0` → Return code (0 = success, anything else = failure)
- `>> Hello World` → The actual command output

**For module-based output:**

```json
web1 | SUCCESS => {
    "ansible_facts": {
        "discovered_interpreter_python": "/usr/bin/python3"
    },
    "changed": false,
    "ping": "pong"
}
```

- `changed: false` → Nothing was modified (already in desired state)
- `changed: true` → Something was modified

## 🏆 Real-World Scenarios — Putting It All Together

### Scenario 1: You're on-call and need to check the health of all servers

```bash
# Step 1: Can I reach all servers?
ansible all -m ping

# Step 2: Check disk space on all servers
ansible all -m shell -a "df -h / | tail -1"

# Step 3: Check memory on all servers
ansible all -m shell -a "free -m | head -2"

# Step 4: Check if critical services are running
ansible webservers -m shell -a "systemctl is-active nginx"
ansible dbservers -m shell -a "systemctl is-active mysql"
```

### Scenario 2: Emergency security patch — install an update on all servers NOW

```bash
# Step 1: Check current version
ansible all -m shell -a "openssl version"

# Step 2: Update the package
ansible all -m apt -a "name=openssl state=latest update_cache=yes" -b

# Step 3: Verify the update
ansible all -m shell -a "openssl version"
```

### Scenario 3: New developer joins — create their account on all servers

```bash
# Step 1: Create the user
ansible all -m user -a "name=john state=present shell=/bin/bash createhome=yes" -b

# Step 2: Add their SSH key
ansible all -m authorized_key -a "user=john key='ssh-rsa AAAA...your-key-here...'" -b

# Step 3: Add to sudo group
ansible all -m user -a "name=john groups=sudo append=yes" -b
```

### Scenario 4: Deploy a quick config change

```bash
# Step 1: Backup the current config
ansible webservers -m shell -a "cp /etc/nginx/nginx.conf /etc/nginx/nginx.conf.bak" -b

# Step 2: Copy the new config
ansible webservers -m copy -a "src=./new-nginx.conf dest=/etc/nginx/nginx.conf owner=root mode=0644" -b

# Step 3: Test nginx config
ansible webservers -m command -a "nginx -t" -b

# Step 4: Reload nginx
ansible webservers -m service -a "name=nginx state=reloaded" -b
```

## ⚠️ Common Mistakes & Troubleshooting

### Mistake 1: Forgetting `-b` (become/sudo)

```bash
# ❌ FAILS — Permission denied
ansible webservers -m apt -a "name=nginx state=present"

# ✅ WORKS — Runs with sudo
ansible webservers -m apt -a "name=nginx state=present" -b
```

### Mistake 2: Using `command` module when you need pipes

```bash
# ❌ FAILS — command module doesn't support pipes
ansible all -m command -a "ps aux | grep nginx"

# ✅ WORKS — shell module supports pipes
ansible all -m shell -a "ps aux | grep nginx"
```

### Mistake 3: Not quoting arguments properly

```bash
# ❌ MIGHT BREAK — spaces in arguments
ansible all -m shell -a cat /var/log/syslog | grep error

# ✅ WORKS — properly quoted
ansible all -m shell -a "cat /var/log/syslog | grep error"
```

### Mistake 4: SSH Connection Issues

```bash
# If you're getting SSH errors, try:
ansible all -m ping -vvv   # Verbose output to see what's going wrong

# Common fixes:
# 1. Make sure SSH key is set up:
ssh-copy-id user@server-ip

# 2. Check your inventory file has correct IPs and users
# 3. Make sure the remote server allows SSH
# 4. Check if Python is installed on remote server
```

Ad-hoc commands are great for quick tasks, but when you need to:

- Make your work **repeatable** and **version-controlled**
