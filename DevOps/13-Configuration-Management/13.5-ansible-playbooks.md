# 13.5 — Ansible — Playbooks (Tasks, Handlers, Variables, Templates)

> **One Line Answer:** An Ansible Playbook is a **YAML file** that contains a **list of tasks** you want to run on your servers — it's like a **recipe book** where each "play" targets specific servers and each "task" is a step in the recipe, and it can include **handlers** (actions triggered by changes), **variables** (dynamic values), and **templates** (files with placeholders) to make your automation powerful and reusable.

## 🤔 What Is a Playbook and Why Do We Need It?

Remember ad-hoc commands from the previous section? They're great for quick, one-off tasks. But what if you need to:

1. Install Nginx
2. Copy a custom config file
3. Start Nginx
4. Enable Nginx to start on boot
5. Open port 80 in the firewall

That's 5 separate ad-hoc commands! And what if you need to do this on 100 servers? And what if you need to do it again next month when you set up new servers?

**That's where playbooks come in.** A playbook lets you write ALL of those steps in **one file**, run it with **one command**, and it does everything **in order**, on **all your servers**, **every time**.

### Ad-Hoc vs Playbook — The Difference

```
Ad-Hoc Commands:
    "Hey Ansible, go install nginx on all webservers real quick"
    → One command, one task, done and forgotten
    → Like sending a text message

Playbooks:
    "Hey Ansible, here's a complete step-by-step recipe:
     1. Install nginx
     2. Copy my custom config
     3. Start the service
     4. Enable it on boot
     5. Open the firewall port"
    → Multiple tasks, saved in a file, reusable forever
    → Like writing a formal document
```

## 📝 Your First Playbook — The Basics

Let's create the simplest possible playbook:

```yaml
# File: first-playbook.yml
---
- name: My First Playbook
  hosts: webservers
  become: yes

  tasks:
    - name: Install nginx
      apt:
        name: nginx
        state: present
        update_cache: yes

    - name: Start nginx
      service:
        name: nginx
        state: started
        enabled: yes
```

Now let's break this down **line by line**:

```yaml
---                              # Every YAML file starts with three dashes
- name: My First Playbook        # Name of this "play" (human-readable description)
  hosts: webservers              # Which servers to target (from your inventory)
  become: yes                    # Run all tasks with sudo

  tasks:                         # Here come the actual things to do
    - name: Install nginx        # Name of this task (shows in output)
      apt:                       # Module to use (apt for Ubuntu/Debian)
        name: nginx              # Package name
        state: present           # Make sure it's installed
        update_cache: yes        # Run apt-get update first

    - name: Start nginx          # Second task
      service:                   # Module: service management
        name: nginx              # Service name
        state: started           # Make sure it's running
        enabled: yes             # Make sure it starts on boot
```

### How to Run a Playbook

```bash
ansible-playbook first-playbook.yml
```

That's it! One command runs all the tasks.

**With extra options:**

```bash
# Specify a custom inventory file
ansible-playbook -i inventory.ini first-playbook.yml

# Dry run (check mode) — see what WOULD change without doing it
ansible-playbook first-playbook.yml --check

# Verbose output
ansible-playbook first-playbook.yml -v     # Some details
ansible-playbook first-playbook.yml -vv    # More details
ansible-playbook first-playbook.yml -vvv   # Maximum details

# Limit to specific hosts
ansible-playbook first-playbook.yml --limit web1

# Ask for sudo password
ansible-playbook first-playbook.yml -K

# Start from a specific task (skip earlier tasks)
ansible-playbook first-playbook.yml --start-at-task="Start nginx"

# Step through tasks one by one (asks before each)
ansible-playbook first-playbook.yml --step

# List all tasks without running them
ansible-playbook first-playbook.yml --list-tasks

# List all hosts that would be affected
ansible-playbook first-playbook.yml --list-hosts
```

## 🏗️ Playbook Structure — The Big Picture

Let's understand the full structure of a playbook:

```
Playbook (.yml file)
│
├── Play 1  ─────────────── targets: webservers
│   ├── Task 1              Install nginx
│   ├── Task 2              Copy config file
│   ├── Task 3              Start nginx
│   └── Handler 1           Restart nginx (only if config changed)
│
├── Play 2  ─────────────── targets: dbservers
│   ├── Task 1              Install MySQL
│   ├── Task 2              Copy my.cnf
│   └── Task 3              Start MySQL
│
└── Play 3  ─────────────── targets: all
    └── Task 1              Check uptime
```

**Key concepts:**
- A **playbook** can contain **multiple plays**
- Each **play** targets a **different group of servers**
- Each **play** contains **multiple tasks**
- **Tasks run in order** — top to bottom
- If a task **fails**, Ansible **stops** on that host (but continues on other hosts)

### Multiple Plays in One Playbook

```yaml
---
# Play 1: Set up web servers
- name: Configure Web Servers
  hosts: webservers
  become: yes

  tasks:
    - name: Install nginx
      apt:
        name: nginx
        state: present
        update_cache: yes

    - name: Start nginx
      service:
        name: nginx
        state: started

# Play 2: Set up database servers
- name: Configure Database Servers
  hosts: dbservers
  become: yes

  tasks:
    - name: Install MySQL
      apt:
        name: mysql-server
        state: present

    - name: Start MySQL
      service:
        name: mysql
        state: started

# Play 3: Common tasks for everyone
- name: Common Setup for All Servers
  hosts: all
  become: yes

  tasks:
    - name: Ensure NTP is installed
      apt:
        name: ntp
        state: present

    - name: Ensure NTP is running
      service:
        name: ntp
        state: started
        enabled: yes
```

## 📦 Tasks — The Heart of Playbooks

Tasks are the actual **actions** Ansible performs. Each task uses a **module** and has a **name**.

### Task Syntax

```yaml
tasks:
  # Method 1: Key-value format (one line per argument)
  - name: Install nginx
    apt:
      name: nginx
      state: present
      update_cache: yes

  # Method 2: Inline format (arguments as a string — less readable)
  - name: Install nginx
    apt: name=nginx state=present update_cache=yes
```

**Always prefer Method 1** — it's much easier to read.

### Task With `register` — Capturing Output

Sometimes you want to **save the output** of a task and use it later:

```yaml
tasks:
  - name: Check disk space
    command: df -h /
    register: disk_output

  - name: Show disk space
    debug:
      msg: "Disk space info: {{ disk_output.stdout }}"
```

What's happening:
1. First task runs `df -h /` and saves the result in a variable called `disk_output`
2. Second task prints that result using the `debug` module
3. `{{ disk_output.stdout }}` is Jinja2 syntax — it accesses the `stdout` field of the saved result

### Task With `when` — Conditional Execution

Run a task **only if** a condition is true:

```yaml
tasks:
  - name: Install nginx (only on Ubuntu)
    apt:
      name: nginx
      state: present
    when: ansible_os_family == "Debian"

  - name: Install nginx (only on CentOS)
    yum:
      name: nginx
      state: present
    when: ansible_os_family == "RedHat"
```

More `when` examples:

```yaml
tasks:
  # Only if a variable is defined
  - name: Print custom message
    debug:
      msg: "{{ custom_message }}"
    when: custom_message is defined

  # Only if a previous task changed something
  - name: Check if config exists
    stat:
      path: /etc/nginx/nginx.conf
    register: config_file

  - name: Create default config
    copy:
      src: nginx.conf
      dest: /etc/nginx/nginx.conf
    when: not config_file.stat.exists

  # Multiple conditions (AND)
  - name: Only on Ubuntu 22.04
    apt:
      name: special-package
      state: present
    when:
      - ansible_os_family == "Debian"
      - ansible_distribution_version == "22.04"

  # OR condition
  - name: Install on Debian or Ubuntu
    apt:
      name: nginx
      state: present
    when: ansible_distribution == "Debian" or ansible_distribution == "Ubuntu"
```

### Task With `loop` — Doing the Same Thing Multiple Times

```yaml
tasks:
  # Install multiple packages
  - name: Install required packages
    apt:
      name: "{{ item }}"
      state: present
    loop:
      - nginx
      - git
      - curl
      - vim
      - htop

  # Create multiple users
  - name: Create application users
    user:
      name: "{{ item.name }}"
      shell: "{{ item.shell }}"
      groups: "{{ item.groups }}"
    loop:
      - { name: 'deploy', shell: '/bin/bash', groups: 'sudo' }
      - { name: 'appuser', shell: '/bin/sh', groups: 'www-data' }
      - { name: 'monitor', shell: '/bin/bash', groups: 'sudo' }

  # Create multiple directories
  - name: Create app directories
    file:
      path: "{{ item }}"
      state: directory
      mode: '0755'
    loop:
      - /opt/myapp
      - /opt/myapp/logs
      - /opt/myapp/config
      - /opt/myapp/data
```

**Older syntax (deprecated but you'll see it in older playbooks):**

```yaml
  # OLD WAY — with_items (still works but loop is preferred)
  - name: Install packages
    apt:
      name: "{{ item }}"
      state: present
    with_items:
      - nginx
      - git
      - curl
```

**Even better — install multiple packages at once (more efficient):**

```yaml
  # BEST WAY — pass a list directly to the name parameter
  - name: Install required packages
    apt:
      name:
        - nginx
        - git
        - curl
        - vim
        - htop
      state: present
      update_cache: yes
```

This is faster because it runs `apt-get install nginx git curl vim htop` in one go, instead of running `apt-get install` five separate times.

### Task With `ignore_errors` — Don't Stop If This Fails

```yaml
tasks:
  - name: Try to stop old service (might not exist)
    service:
      name: old-app
      state: stopped
    ignore_errors: yes

  - name: This task will run even if the above failed
    debug:
      msg: "Continuing with deployment..."
```

### Task With `tags` — Run Only Specific Tasks

```yaml
tasks:
  - name: Install nginx
    apt:
      name: nginx
      state: present
    tags:
      - install
      - nginx

  - name: Copy nginx config
    copy:
      src: nginx.conf
      dest: /etc/nginx/nginx.conf
    tags:
      - config
      - nginx

  - name: Start nginx
    service:
      name: nginx
      state: started
    tags:
      - service
      - nginx
```

Now you can run only specific tagged tasks:

```bash
# Run only tasks tagged "install"
ansible-playbook site.yml --tags "install"

# Run only tasks tagged "config"
ansible-playbook site.yml --tags "config"

# Run everything EXCEPT tasks tagged "install"
ansible-playbook site.yml --skip-tags "install"

# Run tasks with multiple tags
ansible-playbook site.yml --tags "install,config"
```

## 🔔 Handlers — Actions Triggered by Changes

Handlers are **special tasks** that only run when **notified** by another task. The most common use case: **restart a service only if its config file changed**.

### The Problem Handlers Solve

Imagine this scenario:
1. You copy a new Nginx config file to the server
2. If the config actually changed, you need to restart Nginx
3. If the config was already the same (no change), there's no need to restart

Without handlers, you'd restart Nginx **every time** — even if nothing changed. That's wasteful and causes unnecessary downtime.

### How Handlers Work

```yaml
---
- name: Configure Web Server
  hosts: webservers
  become: yes

  tasks:
    - name: Install nginx
      apt:
        name: nginx
        state: present
        update_cache: yes

    - name: Copy nginx config
      copy:
        src: files/nginx.conf
        dest: /etc/nginx/nginx.conf
        owner: root
        group: root
        mode: '0644'
      notify: Restart nginx          # ← "Hey handler, I might need you!"

    - name: Copy site config
      template:
        src: templates/default.conf.j2
        dest: /etc/nginx/sites-available/default
      notify: Restart nginx          # ← Same handler can be notified multiple times

    - name: Ensure nginx is running
      service:
        name: nginx
        state: started
        enabled: yes

  handlers:                           # ← Handlers section (after tasks)
    - name: Restart nginx             # ← Must match the "notify" name EXACTLY
      service:
        name: nginx
        state: restarted
```

### How the Flow Works:

```
Task: "Copy nginx config"
  ├── Config file CHANGED?
  │   ├── YES → Mark "Restart nginx" handler as NOTIFIED
  │   └── NO  → Don't notify (skip handler)
  │
Task: "Copy site config"
  ├── Site config CHANGED?
  │   ├── YES → Mark "Restart nginx" handler as NOTIFIED
  │   └── NO  → Don't notify (skip handler)
  │
ALL TASKS COMPLETE → Now run notified handlers
  │
Handler: "Restart nginx"
  ├── Was I notified? 
  │   ├── YES → Restart nginx (runs ONCE even if notified twice)
  │   └── NO  → Don't run at all
```

**Key rules about handlers:**
1. Handlers **only run if notified** (i.e., only if the notifying task made a change)
2. Handlers run **at the END** of all tasks (not immediately after the notify)
3. Even if notified **multiple times**, a handler runs **only once**
4. The `notify` name must **exactly match** the handler name
5. Handlers are tasks — they use the same module syntax

### Multiple Handlers

```yaml
  tasks:
    - name: Update app code
      git:
        repo: https://github.com/myorg/myapp.git
        dest: /opt/myapp
      notify:
        - Restart application
        - Clear cache

    - name: Update nginx config
      template:
        src: nginx.conf.j2
        dest: /etc/nginx/nginx.conf
      notify:
        - Restart nginx

  handlers:
    - name: Restart application
      service:
        name: myapp
        state: restarted

    - name: Clear cache
      command: /opt/myapp/clear-cache.sh

    - name: Restart nginx
      service:
        name: nginx
        state: restarted
```

### Force Handlers to Run Immediately (flush_handlers)

Sometimes you need a handler to run before the next task, not at the end:

```yaml
  tasks:
    - name: Copy new nginx config
      copy:
        src: nginx.conf
        dest: /etc/nginx/nginx.conf
      notify: Restart nginx

    - name: Force handler to run NOW (not at the end)
      meta: flush_handlers

    - name: Check if nginx is healthy
      uri:
        url: http://localhost
        status_code: 200
```

## 🔤 Variables — Making Playbooks Dynamic

Variables let you **avoid hardcoding values** and make your playbooks **flexible and reusable**.

### Where Can You Define Variables?

There are **many** places to define variables in Ansible. Here are the most common:

### 1. Directly in the Playbook (`vars` section)

```yaml
---
- name: Deploy Application
  hosts: webservers
  become: yes

  vars:
    app_name: myapp
    app_version: "2.1.0"
    app_port: 8080
    app_user: deploy
    app_dir: "/opt/{{ app_name }}"     # You can use other variables!

  tasks:
    - name: Create app directory
      file:
        path: "{{ app_dir }}"
        state: directory
        owner: "{{ app_user }}"

    - name: Download app
      get_url:
        url: "https://releases.example.com/{{ app_name }}/{{ app_version }}/{{ app_name }}.tar.gz"
        dest: "/tmp/{{ app_name }}.tar.gz"

    - name: Display info
      debug:
        msg: "Deploying {{ app_name }} version {{ app_version }} on port {{ app_port }}"
```

**Important:** Always wrap variable references in **double quotes** when they start a value:

```yaml
# ✅ CORRECT
path: "{{ app_dir }}"

# ❌ WRONG — YAML will break
path: {{ app_dir }}

# ✅ OK if variable is in the middle of a string (quotes still recommended)
msg: "The app {{ app_name }} is running on port {{ app_port }}"
```

### 2. Variables in a Separate File (`vars_files`)

For cleaner organization, put variables in separate files:

```yaml
# File: vars/app_config.yml
app_name: myapp
app_version: "2.1.0"
app_port: 8080
app_user: deploy
app_dir: "/opt/{{ app_name }}"
packages:
  - nginx
  - git
  - curl
```

```yaml
# File: deploy.yml (playbook)
---
- name: Deploy Application
  hosts: webservers
  become: yes
  vars_files:
    - vars/app_config.yml

  tasks:
    - name: Create app directory
      file:
        path: "{{ app_dir }}"
        state: directory
        owner: "{{ app_user }}"

    - name: Install packages
      apt:
        name: "{{ packages }}"
        state: present
```

### 3. Variables from the Command Line (`--extra-vars` or `-e`)

```bash
# Pass variables when running the playbook
ansible-playbook deploy.yml -e "app_version=3.0.0"

# Pass multiple variables
ansible-playbook deploy.yml -e "app_version=3.0.0 app_port=9090"

# Pass variables from a JSON file
ansible-playbook deploy.yml -e "@vars/production.json"
```

**Command line variables have the HIGHEST priority** — they override everything else!

### 4. Inventory Variables

```ini
# File: inventory.ini
[webservers]
web1 ansible_host=192.168.1.10 http_port=80
web2 ansible_host=192.168.1.11 http_port=8080

[webservers:vars]
app_env=production
nginx_workers=4

[dbservers]
db1 ansible_host=192.168.1.20

[dbservers:vars]
db_port=3306
```

### 5. Group Variables and Host Variables (Files)

The best practice for inventory variables — use **directories**:

```
project/
├── inventory.ini
├── group_vars/
│   ├── all.yml           ← Variables for ALL hosts
│   ├── webservers.yml    ← Variables for the webservers group
│   └── dbservers.yml     ← Variables for the dbservers group
├── host_vars/
│   ├── web1.yml          ← Variables ONLY for web1
│   └── db1.yml           ← Variables ONLY for db1
└── site.yml
```

```yaml
# File: group_vars/all.yml
ntp_server: time.google.com
dns_server: 8.8.8.8
app_env: staging

# File: group_vars/webservers.yml
nginx_port: 80
nginx_workers: 4
ssl_enabled: true

# File: host_vars/web1.yml
nginx_port: 8080    # Override just for web1
custom_domain: web1.example.com
```

Ansible automatically picks up these files — no need to reference them in the playbook!

### 6. Registered Variables (from task output)

```yaml
tasks:
  - name: Get server hostname
    command: hostname
    register: hostname_result

  - name: Display hostname
    debug:
      msg: "This server's hostname is: {{ hostname_result.stdout }}"

  - name: Check if Docker is installed
    command: docker --version
    register: docker_check
    ignore_errors: yes

  - name: Install Docker if not present
    apt:
      name: docker.io
      state: present
    when: docker_check.rc != 0    # rc = return code, non-zero means failure
```

### 7. Facts — Variables Ansible Gathers Automatically

Ansible automatically collects **facts** about each server (OS, IP, CPU, RAM, etc.):

```yaml
tasks:
  - name: Show OS family
    debug:
      msg: "This server runs {{ ansible_os_family }}"

  - name: Show IP address
    debug:
      msg: "IP: {{ ansible_default_ipv4.address }}"

  - name: Show total RAM
    debug:
      msg: "RAM: {{ ansible_memtotal_mb }} MB"

  - name: Install based on OS
    apt:
      name: nginx
      state: present
    when: ansible_os_family == "Debian"
```

Common facts you'll use:

| Fact | What It Contains |
|------|-----------------|
| `ansible_os_family` | "Debian", "RedHat", "Suse", etc. |
| `ansible_distribution` | "Ubuntu", "CentOS", "Fedora", etc. |
| `ansible_distribution_version` | "22.04", "8", "38", etc. |
| `ansible_hostname` | Server hostname |
| `ansible_default_ipv4.address` | Primary IP address |
| `ansible_memtotal_mb` | Total RAM in MB |
| `ansible_processor_vcpus` | Number of CPU cores |
| `ansible_architecture` | "x86_64", "aarch64", etc. |

### Variable Precedence (Priority Order)

When the same variable is defined in multiple places, which one wins? Here's the order from **lowest** to **highest** priority:

```
Priority (lowest → highest):
    1.  Role defaults (defaults/main.yml)
    2.  Inventory group_vars/all
    3.  Inventory group_vars/<group>
    4.  Inventory host_vars/<host>
    5.  Play vars
    6.  Play vars_files
    7.  Task vars
    8.  Role vars (vars/main.yml)
    9.  Block vars
    10. set_fact / registered vars
    11. Extra vars (-e on command line)     ← ALWAYS WINS!
```

**Simple rule:** Command line (`-e`) always wins. Role defaults always lose. Everything else is in between.

## 📄 Templates — Dynamic Configuration Files (Jinja2)

Templates are one of the **most powerful features** of Ansible. They let you create **dynamic configuration files** that change based on variables, facts, and logic.

### What's the Problem Templates Solve?

Imagine you have an Nginx config that needs to be **slightly different** on each server:
- Different server names
- Different ports
- Different number of worker processes

Without templates, you'd need a **separate config file for every server**. With 100 servers, that's 100 files! 😱

With templates, you create **ONE template** with **placeholders**, and Ansible fills in the correct values for each server.

### How Templates Work

```
Template File (.j2)                    Ansible fills in values
─────────────────                      ──────────────────────

server {                               server {
    listen {{ nginx_port }};     →         listen 80;
    server_name {{ domain }};    →         server_name example.com;
    worker_processes {{ cpus }}; →         worker_processes 4;
}                                      }
```

### Creating and Using Templates

**Step 1:** Create a template file (with `.j2` extension — stands for Jinja2):

```jinja2
# File: templates/nginx.conf.j2

# Nginx Configuration — Managed by Ansible
# DO NOT EDIT MANUALLY — Changes will be overwritten!
# Generated for: {{ ansible_hostname }}
# Date: {{ ansible_date_time.iso8601 }}

user www-data;
worker_processes {{ ansible_processor_vcpus }};
pid /run/nginx.pid;

events {
    worker_connections {{ nginx_worker_connections | default(1024) }};
}

http {
    sendfile on;
    tcp_nopush on;
    tcp_nodelay on;
    keepalive_timeout {{ nginx_keepalive_timeout | default(65) }};

    server {
        listen {{ nginx_port }};
        server_name {{ server_domain }};

        root {{ app_dir }}/public;

        location / {
            proxy_pass http://127.0.0.1:{{ app_port }};
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
        }

{% if ssl_enabled %}
        listen 443 ssl;
        ssl_certificate /etc/ssl/certs/{{ server_domain }}.crt;
        ssl_certificate_key /etc/ssl/private/{{ server_domain }}.key;
{% endif %}

{% for location in custom_locations | default([]) %}
        location {{ location.path }} {
            proxy_pass {{ location.backend }};
        }
{% endfor %}
    }
}
```

**Step 2:** Use the `template` module in your playbook:

```yaml
---
- name: Configure Nginx
  hosts: webservers
  become: yes

  vars:
    nginx_port: 80
    server_domain: example.com
    app_port: 3000
    app_dir: /opt/myapp
    ssl_enabled: true
    nginx_worker_connections: 2048
    nginx_keepalive_timeout: 75
    custom_locations:
      - path: /api
        backend: http://127.0.0.1:5000
      - path: /admin
        backend: http://127.0.0.1:8000

  tasks:
    - name: Deploy nginx config from template
      template:
        src: templates/nginx.conf.j2
        dest: /etc/nginx/nginx.conf
        owner: root
        group: root
        mode: '0644'
        validate: "nginx -t -c %s"     # Validate config before applying!
      notify: Restart nginx

  handlers:
    - name: Restart nginx
      service:
        name: nginx
        state: restarted
```

### Jinja2 Syntax You'll Use in Templates

#### 1. Variables — `{{ }}`

```jinja2
# Simple variable
server_name {{ domain }};

# Variable with a default value (if not defined, use the default)
worker_processes {{ workers | default(4) }};

# Accessing nested variables
IP: {{ ansible_default_ipv4.address }}
```

#### 2. Conditionals — `{% if %}`

```jinja2
{% if ssl_enabled %}
listen 443 ssl;
ssl_certificate /etc/ssl/{{ domain }}.crt;
{% else %}
listen 80;
{% endif %}

{% if app_env == "production" %}
log_level warn;
{% elif app_env == "staging" %}
log_level info;
{% else %}
log_level debug;
{% endif %}
```

#### 3. Loops — `{% for %}`

```jinja2
# Simple loop
{% for server in backend_servers %}
server {{ server }}:{{ app_port }};
{% endfor %}

# Loop with index
{% for user in allowed_users %}
# User {{ loop.index }}: {{ user }}
allow {{ user }};
{% endfor %}

# Loop with conditional
{% for port in open_ports %}
{% if port != 22 %}
allow {{ port }}/tcp;
{% endif %}
{% endfor %}
```

#### 4. Filters — Transforming Values

```jinja2
# Convert to uppercase
{{ hostname | upper }}

# Convert to lowercase
{{ hostname | lower }}

# Default value
{{ timeout | default(30) }}

# Join a list into a string
{{ dns_servers | join(', ') }}

# Get the first/last item
{{ servers | first }}
{{ servers | last }}

# Math
{{ workers * 2 }}
{{ (ansible_memtotal_mb / 1024) | round(1) }} GB
```

#### 5. Comments — `{# #}`

```jinja2
{# This is a Jinja2 comment — it won't appear in the generated file #}
server_name {{ domain }};
```

## 🏆 Complete Real-World Playbook — Putting It ALL Together

Let's build a **complete, production-quality playbook** that sets up a web server. This combines everything we've learned:

### Project Structure

```
├── group_vars/
│   └── webservers.yml
├── templates/
```

### inventory.ini

```ini

web2 ansible_host=192.168.1.11

[webservers:vars]
ansible_user=ubuntu
```

### group_vars/webservers.yml

```yaml
# Application Settings
app_name: mywebapp
app_port: 3000
app_dir: /opt/mywebapp
app_user: deploy
app_env: production

# Nginx Settings
nginx_port: 80
nginx_worker_connections: 1024
server_domain: mywebapp.example.com
ssl_enabled: false

# Packages to install
required_packages:
  - nginx
  - git
  - curl
  - ufw
```

### templates/nginx.conf.j2

```jinja2
# Nginx Configuration for {{ app_name }}
# Managed by Ansible — DO NOT EDIT MANUALLY
# Host: {{ ansible_hostname }} | Generated: {{ ansible_date_time.iso8601 }}

user www-data;
worker_processes {{ ansible_processor_vcpus }};
pid /run/nginx.pid;

events {
    worker_connections {{ nginx_worker_connections }};
}

http {
    include /etc/nginx/mime.types;
    default_type application/octet-stream;

    # Logging
    access_log /var/log/nginx/{{ app_name }}_access.log;
    error_log /var/log/nginx/{{ app_name }}_error.log;

    server {
        listen {{ nginx_port }};
        server_name {{ server_domain }};

        location / {
            proxy_pass http://127.0.0.1:{{ app_port }};
            proxy_http_version 1.1;
            proxy_set_header Upgrade $http_upgrade;
            proxy_set_header Connection 'upgrade';
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_cache_bypass $http_upgrade;
        }
    }
}
```

### templates/index.html.j2

```html

<head><title>{{ app_name }}</title></head>
<body>
    <h1>Welcome to {{ app_name }}!</h1>
    <p>Server: {{ ansible_hostname }}</p>
    <p>Environment: {{ app_env }}</p>
    <p>IP: {{ ansible_default_ipv4.address }}</p>
</body>
</html>
```

### site.yml — The Main Playbook

```yaml
---
- name: Full Web Server Setup
  hosts: webservers
  become: yes
  gather_facts: yes

  tasks:
    # ===== SYSTEM SETUP =====
    - name: Update apt cache
      apt:
        update_cache: yes
        cache_valid_time: 3600
      tags: system

    - name: Install required packages
      apt:
        name: "{{ required_packages }}"
        state: present
      tags: system

    # ===== USER SETUP =====
    - name: Create application user
      user:
        name: "{{ app_user }}"
        shell: /bin/bash
        createhome: yes
        state: present
      tags: user

    # ===== APPLICATION DIRECTORY =====
    - name: Create application directories
      file:
        path: "{{ item }}"
        state: directory
        owner: "{{ app_user }}"
        group: "{{ app_user }}"
        mode: '0755'
      loop:
        - "{{ app_dir }}"
        - "{{ app_dir }}/public"
        - "{{ app_dir }}/logs"
        - "{{ app_dir }}/config"
      tags: app

    # ===== NGINX CONFIGURATION =====
    - name: Deploy nginx configuration
      template:
        src: templates/nginx.conf.j2
        dest: /etc/nginx/nginx.conf
        owner: root
        group: root
        mode: '0644'
      notify: Restart nginx
      tags: nginx

    - name: Deploy default page
      template:
        src: templates/index.html.j2
        dest: "{{ app_dir }}/public/index.html"
        owner: "{{ app_user }}"
        group: "{{ app_user }}"
        mode: '0644'
      tags: nginx

    - name: Remove default nginx site
      file:
        path: /etc/nginx/sites-enabled/default
        state: absent
      notify: Restart nginx
      tags: nginx

    # ===== FIREWALL =====
    - name: Allow SSH through firewall
      ufw:
        rule: allow
        port: '22'
        proto: tcp
      tags: firewall

    - name: Allow HTTP through firewall
      ufw:
        rule: allow
        port: "{{ nginx_port | string }}"
        proto: tcp
      tags: firewall

    - name: Enable UFW
      ufw:
        state: enabled
        policy: deny
      tags: firewall

    # ===== START SERVICES =====
    - name: Ensure nginx is running and enabled
      service:
        name: nginx
        state: started
        enabled: yes
      tags: nginx

    # ===== VERIFICATION =====
    - name: Verify nginx is responding
      uri:
        url: "http://localhost:{{ nginx_port }}"
        status_code: 200
      register: nginx_check
      tags: verify

    - name: Display deployment result
      debug:
        msg: |
          ✅ Deployment Complete!
          Server: {{ ansible_hostname }}
          IP: {{ ansible_default_ipv4.address }}
          App: {{ app_name }}
          Port: {{ nginx_port }}
          Nginx Status: {{ nginx_check.status }}
      tags: verify

  # ===== HANDLERS =====
  handlers:
    - name: Restart nginx
      service:
        name: nginx
        state: restarted
```

### Running This Playbook

```bash
# Full run
ansible-playbook -i inventory.ini site.yml

# Dry run first
ansible-playbook -i inventory.ini site.yml --check

# Only nginx tasks
ansible-playbook -i inventory.ini site.yml --tags nginx

# Only on web1
ansible-playbook -i inventory.ini site.yml --limit web1

# Override a variable
ansible-playbook -i inventory.ini site.yml -e "app_env=staging nginx_port=8080"
```

## ⚠️ Common Mistakes & Best Practices

### Mistake 1: YAML Indentation Errors

```yaml
# ❌ WRONG — inconsistent indentation
tasks:
  - name: Install nginx
    apt:
    name: nginx        # This is at the wrong level!

# ✅ CORRECT — proper indentation (2 spaces per level)
tasks:
  - name: Install nginx
    apt:
      name: nginx
      state: present
```

### Mistake 2: Forgetting Quotes Around Variables

```yaml
# ❌ WRONG — YAML breaks when a value starts with {{
path: {{ app_dir }}

# ✅ CORRECT — wrap in quotes
path: "{{ app_dir }}"
```

### Mistake 3: Handler Name Mismatch

```yaml
# ❌ WRONG — names don't match
tasks:
  - name: Copy config
    copy:
      src: nginx.conf
      dest: /etc/nginx/nginx.conf
    notify: restart nginx              # lowercase "r"

handlers:
  - name: Restart nginx               # uppercase "R" — MISMATCH!
    service:
      name: nginx
      state: restarted

# ✅ CORRECT — names match exactly
notify: Restart nginx
# ...
- name: Restart nginx
```

### Best Practices Summary

| Practice | Why? |
|----------|------|
| Always use `name:` for every task | Makes output readable |
| Use `--check` before applying | Preview changes safely |
| Keep variables in `group_vars`/`host_vars` | Organized and clean |
| Use templates instead of `copy` for dynamic files | Flexible configs |
| Use handlers for service restarts | Only restart when needed |
| Use tags on tasks | Run specific parts of playbook |
| Use `validate` on templates | Catch config errors before applying |
| Use version control (Git) for playbooks | Track changes, collaborate |
