# 13.6 — Ansible — Roles — Structure, Galaxy, Best Practices

> **One Line Answer:** An Ansible Role is a **pre-defined, reusable, and shareable way to organize your playbooks** into a standard directory structure — think of it like a **plugin or package** for Ansible. Instead of writing one giant playbook with everything crammed in, you break it into roles like `nginx`, `mysql`, `docker`, etc., each with its own tasks, handlers, variables, templates, and files — making your automation **clean, modular, and reusable** across projects.

---

## 🤔 Why Do We Need Roles? What Problem Do They Solve?

Imagine you've been writing Ansible playbooks for a while. Your `site.yml` now looks like this:

```yaml
# site.yml — This is getting out of hand...
---
- name: Configure Everything
  hosts: webservers
  become: yes

  vars:
    nginx_port: 80
    app_port: 3000
    db_host: 192.168.1.20
    # ... 50 more variables ...

  tasks:
    # Install system packages (5 tasks)
    # Create users (3 tasks)
    # Install and configure nginx (8 tasks)
    # Install and configure Node.js (6 tasks)
    # Deploy the application (10 tasks)
    # Configure firewall (4 tasks)
    # Set up monitoring agent (5 tasks)
    # Configure log rotation (3 tasks)
    # ... 44 tasks total in one file!

  handlers:
    # 8 handlers all mixed together
```

This has **multiple problems:**

| Problem | Why It's Bad |
|---------|-------------|
| **Huge file** | 500+ lines in one YAML file — hard to read and maintain |
| **No reusability** | Can't reuse the "nginx setup" part in another project |
| **No sharing** | Can't share the "Docker setup" part with your team |
| **Tight coupling** | Everything is tangled together — changing one thing might break another |
| **No testing** | Can't test the "database setup" part independently |
| **No organization** | Variables, tasks, templates, handlers — all mixed in one place |

**Roles solve ALL of these problems.**

### With Roles, the same playbook becomes:

```yaml
# site.yml — Clean and simple!
---
- name: Configure Web Servers
  hosts: webservers
  become: yes

  roles:
    - common         # Install base packages, create users
    - nginx          # Install and configure Nginx
    - nodejs         # Install Node.js
    - app_deploy     # Deploy the application
    - firewall       # Configure UFW firewall
    - monitoring     # Set up monitoring agent
```

Each role is **self-contained** with its own tasks, variables, templates, handlers — everything neatly organized in its own directory. Beautiful! 🎉

## 📁 Role Directory Structure — The Standard Layout

Every role follows the **same standard directory structure**. This is a convention that ALL Ansible roles follow:

```
roles/
└── nginx/                     ← Role name
    ├── defaults/
    │   └── main.yml           ← Default variables (lowest priority)
    ├── vars/
    │   └── main.yml           ← Role variables (higher priority)
    ├── tasks/
    │   └── main.yml           ← The actual tasks to execute
    ├── handlers/
    │   └── main.yml           ← Handlers (triggered by notify)
    ├── templates/
    │   └── nginx.conf.j2      ← Jinja2 template files
    ├── files/
    │   └── index.html         ← Static files to copy
    ├── meta/
    │   └── main.yml           ← Role metadata (dependencies, info)
    └── README.md              ← Documentation
```

Let's understand **each directory** in detail:

### 📂 `tasks/main.yml` — The Heart of the Role

This is where your **actual tasks** go. When Ansible runs this role, it executes `tasks/main.yml`.

```yaml
# roles/nginx/tasks/main.yml
---
- name: Install nginx
  apt:
    name: nginx
    state: present
    update_cache: yes

- name: Deploy nginx configuration
  template:
    src: nginx.conf.j2          # Automatically looks in templates/ directory
    dest: /etc/nginx/nginx.conf
    owner: root
    group: root
    mode: '0644'
  notify: Restart nginx

- name: Ensure nginx is running
  service:
    name: nginx
    state: started
    enabled: yes
```

**Key point:** When you reference `src: nginx.conf.j2` in a role, Ansible automatically looks in the role's `templates/` directory. Same for `files/` with the `copy` module. You don't need to specify the full path!

### 📂 `handlers/main.yml` — Handlers for This Role

```yaml
# roles/nginx/handlers/main.yml
---
- name: Restart nginx
  service:
    name: nginx
    state: restarted

- name: Reload nginx
  service:
    name: nginx
    state: reloaded
```

### 📂 `defaults/main.yml` — Default Variables (Lowest Priority)

These are the **default values** for your role's variables. Users can easily override these.

```yaml
# roles/nginx/defaults/main.yml
---
nginx_port: 80
nginx_worker_processes: auto
nginx_worker_connections: 1024
nginx_keepalive_timeout: 65
nginx_server_name: localhost
nginx_root: /var/www/html
nginx_log_dir: /var/log/nginx
nginx_ssl_enabled: false
nginx_ssl_cert: ""
nginx_ssl_key: ""
```

**Why `defaults/` has the LOWEST priority:** So that users can override them easily from their playbook, inventory, or command line. The defaults just provide sensible starting values.

### 📂 `vars/main.yml` — Role Variables (Higher Priority)

These are variables that **should NOT be easily overridden** — internal values the role needs.

```yaml
# roles/nginx/vars/main.yml
---
nginx_package_name: nginx
nginx_service_name: nginx
nginx_config_dir: /etc/nginx
nginx_sites_available: /etc/nginx/sites-available
nginx_sites_enabled: /etc/nginx/sites-enabled
```

**When to use `defaults/` vs `vars/`:**

```
defaults/main.yml:
    → Values the USER should customize
    → "What port should nginx listen on?" → defaults/
    → "What domain name?" → defaults/
    → Easy to override ✅

vars/main.yml:
    → Internal values the role needs to work
    → "Where is the nginx config directory?" → vars/
    → "What's the package name?" → vars/
    → Hard to override (by design) ⚠️
```

### 📂 `templates/` — Jinja2 Template Files

```jinja2
# roles/nginx/templates/nginx.conf.j2

user www-data;
worker_processes {{ nginx_worker_processes }};
pid /run/nginx.pid;

events {
    worker_connections {{ nginx_worker_connections }};
}

http {
    include /etc/nginx/mime.types;
    default_type application/octet-stream;

    access_log {{ nginx_log_dir }}/access.log;
    error_log {{ nginx_log_dir }}/error.log;

    keepalive_timeout {{ nginx_keepalive_timeout }};

    server {
        listen {{ nginx_port }};
        server_name {{ nginx_server_name }};
        root {{ nginx_root }};

{% if nginx_ssl_enabled %}
        listen 443 ssl;
        ssl_certificate {{ nginx_ssl_cert }};
        ssl_certificate_key {{ nginx_ssl_key }};
{% endif %}
    }
}
```

### 📂 `files/` — Static Files

Files that should be copied **as-is** (no template processing):

```
roles/nginx/files/
├── index.html
├── 50x.html
└── ssl-params.conf
```

```yaml
# In tasks/main.yml, you can reference these files directly:
- name: Copy default page
  copy:
    src: index.html           # Ansible looks in roles/nginx/files/ automatically
    dest: /var/www/html/index.html
```

### 📂 `meta/main.yml` — Role Metadata & Dependencies

```yaml
# roles/nginx/meta/main.yml
---
galaxy_info:
  author: Your Name
  description: Install and configure Nginx web server
  license: MIT
  min_ansible_version: "2.9"
  platforms:
    - name: Ubuntu
      versions:
        - focal
        - jammy
    - name: Debian
      versions:
        - bullseye
        - bookworm
  galaxy_tags:
    - nginx
    - webserver
    - proxy

dependencies:
  - role: common            # This role depends on the "common" role
  # - role: firewall        # Uncomment if you want auto-dependency
```

**Dependencies** are powerful — if your `nginx` role depends on the `common` role, Ansible will automatically run `common` first, even if you don't list it in your playbook!

## 🛠️ Creating a Role — Step by Step

### Method 1: Create Manually

Just create the directories and files yourself:

```bash
mkdir -p roles/nginx/{tasks,handlers,templates,files,vars,defaults,meta}
touch roles/nginx/{tasks,handlers,vars,defaults,meta}/main.yml
```

### Method 2: Use `ansible-galaxy init` (Recommended)

```bash
ansible-galaxy init roles/nginx
```

This creates the **entire standard directory structure** automatically:

```
roles/nginx/
├── README.md
├── defaults/
│   └── main.yml
├── files/
├── handlers/
│   └── main.yml
├── meta/
│   └── main.yml
├── tasks/
│   └── main.yml
├── templates/
├── tests/
│   ├── inventory
│   └── test.yml
└── vars/
    └── main.yml
```

It even creates a `tests/` directory for testing!

## 🎯 Using Roles in Playbooks

### Method 1: Simple Role List (Most Common)

```yaml
---
- name: Configure Servers
  hosts: webservers
  become: yes

  roles:
    - common
    - nginx
    - app_deploy
```

### Method 2: Roles with Variables

```yaml
---
- name: Configure Servers
  hosts: webservers
  become: yes

  roles:
    - common
    - role: nginx
      vars:
        nginx_port: 8080
        nginx_server_name: myapp.example.com
    - role: app_deploy
      vars:
        app_version: "2.1.0"
```

### Method 3: Roles with Tags

```yaml
---
- name: Configure Servers
  hosts: webservers
  become: yes

  roles:
    - role: common
      tags: common
    - role: nginx
      tags: nginx
    - role: app_deploy
      tags: deploy
```

Now you can run only specific roles:

```bash
# Only run the nginx role
ansible-playbook site.yml --tags nginx

# Run everything except deploy
ansible-playbook site.yml --skip-tags deploy
```

### Method 4: Conditional Roles

```yaml
---
- name: Configure Servers
  hosts: all
  become: yes

  roles:
    - common
    - role: nginx
      when: "'webservers' in group_names"
    - role: mysql
      when: "'dbservers' in group_names"
```

### Method 5: Mixing Roles with Tasks (`pre_tasks` and `post_tasks`)

```yaml
---
- name: Deploy Application
  hosts: webservers
  become: yes

  pre_tasks:                        # Run BEFORE roles
    - name: Update apt cache
      apt:
        update_cache: yes

  roles:                            # Run roles in order
    - common
    - nginx
    - app_deploy

  post_tasks:                       # Run AFTER roles
    - name: Verify deployment
      uri:
        url: "http://localhost"
        status_code: 200

    - name: Send notification
      debug:
        msg: "Deployment complete! ✅"
```

**Execution order:**
```
1. pre_tasks     ← First
2. roles         ← Second (in listed order)
3. tasks         ← Third (if any)
4. post_tasks    ← Last
```

### Method 6: Dynamic Role Inclusion

```yaml
# Include a role inside tasks (dynamic — at runtime)
tasks:
  - name: Include the nginx role
    include_role:
      name: nginx
    vars:
      nginx_port: 8080

  - name: Import the nginx role
    import_role:
      name: nginx
    vars:
      nginx_port: 8080
```

**`include_role` vs `import_role`:**

| Feature | `import_role` | `include_role` |
|---------|--------------|----------------|
| **When processed** | At playbook **parsing time** (static) | At **runtime** (dynamic) |
| **Tags** | Tags apply to all tasks inside | Tags only apply to the include itself |
| **Conditionals** | `when` is applied to each task | `when` is applied once to the include |
| **Loops** | ❌ Can't loop | ✅ Can loop |
| **Use when** | You know which role to run at design time | You need conditional/dynamic role inclusion |

## 🏗️ Complete Role Example — Building a Real `nginx` Role

Let's build a complete, production-quality Nginx role from scratch:

### Project Structure

```
project/
├── inventory.ini
├── group_vars/
│   ├── all.yml
│   └── webservers.yml
├── roles/
│   └── nginx/
│       ├── defaults/
│       │   └── main.yml
│       ├── vars/
│       │   └── main.yml
│       ├── tasks/
│       │   ├── main.yml
│       │   ├── install.yml
│       │   ├── configure.yml
│       │   └── service.yml
│       ├── handlers/
│       │   └── main.yml
│       ├── templates/
│       │   ├── nginx.conf.j2
│       │   └── vhost.conf.j2
│       ├── files/
│       │   └── index.html
│       └── meta/
│           └── main.yml
└── site.yml
```

### roles/nginx/defaults/main.yml

```yaml
---
# User-customizable defaults
nginx_port: 80
nginx_server_name: "_"
nginx_root: /var/www/html
nginx_worker_processes: auto
nginx_worker_connections: 1024
nginx_keepalive_timeout: 65
nginx_client_max_body_size: "10m"
nginx_ssl_enabled: false
nginx_ssl_cert: ""
nginx_ssl_key: ""
nginx_proxy_enabled: false
nginx_proxy_pass: ""
nginx_remove_default_site: true
nginx_extra_packages: []
```

### roles/nginx/vars/main.yml

```yaml
---
# Internal variables — not meant to be overridden
nginx_package: nginx
nginx_service: nginx
nginx_conf_dir: /etc/nginx
nginx_sites_available_dir: /etc/nginx/sites-available
nginx_sites_enabled_dir: /etc/nginx/sites-enabled
nginx_log_dir: /var/log/nginx
nginx_user: www-data
```

### roles/nginx/tasks/main.yml

Splitting tasks into separate files keeps things organized:

```yaml
# roles/nginx/tasks/main.yml
---
- name: Include installation tasks
  import_tasks: install.yml
  tags: nginx_install

- name: Include configuration tasks
  import_tasks: configure.yml
  tags: nginx_config

- name: Include service tasks
  import_tasks: service.yml
  tags: nginx_service
```

### roles/nginx/tasks/install.yml

```yaml
# roles/nginx/tasks/install.yml
---
- name: Update apt cache
  apt:
    update_cache: yes
    cache_valid_time: 3600

- name: Install nginx
  apt:
    name: "{{ nginx_package }}"
    state: present

- name: Install extra packages
  apt:
    name: "{{ nginx_extra_packages }}"
    state: present
  when: nginx_extra_packages | length > 0
```

### roles/nginx/tasks/configure.yml

```yaml
# roles/nginx/tasks/configure.yml
---
- name: Deploy main nginx configuration
  template:
    src: nginx.conf.j2
    dest: "{{ nginx_conf_dir }}/nginx.conf"
    owner: root
    group: root
    mode: '0644'
    validate: "nginx -t -c %s"
  notify: Restart nginx

- name: Deploy virtual host configuration
  template:
    src: vhost.conf.j2
    dest: "{{ nginx_sites_available_dir }}/{{ nginx_server_name }}.conf"
    owner: root
    group: root
    mode: '0644'
  notify: Reload nginx

- name: Enable virtual host
  file:
    src: "{{ nginx_sites_available_dir }}/{{ nginx_server_name }}.conf"
    dest: "{{ nginx_sites_enabled_dir }}/{{ nginx_server_name }}.conf"
    state: link
  notify: Reload nginx

- name: Remove default site
  file:
    path: "{{ nginx_sites_enabled_dir }}/default"
    state: absent
  when: nginx_remove_default_site
  notify: Reload nginx

- name: Deploy default index page
  copy:
    src: index.html
    dest: "{{ nginx_root }}/index.html"
    owner: "{{ nginx_user }}"
    group: "{{ nginx_user }}"
    mode: '0644'
  when: not nginx_proxy_enabled

- name: Create log directory
  file:
    path: "{{ nginx_log_dir }}"
    state: directory
    owner: "{{ nginx_user }}"
    group: adm
    mode: '0755'
```

### roles/nginx/tasks/service.yml

```yaml
# roles/nginx/tasks/service.yml
---
- name: Ensure nginx is running
  service:
    name: "{{ nginx_service }}"
    state: started
    enabled: yes

- name: Verify nginx is responding
  uri:
    url: "http://localhost:{{ nginx_port }}"
    status_code: 200
  register: nginx_health
  retries: 3
  delay: 5
  until: nginx_health.status == 200
```

### roles/nginx/handlers/main.yml

```yaml
# roles/nginx/handlers/main.yml
---
- name: Restart nginx
  service:
    name: "{{ nginx_service }}"
    state: restarted

- name: Reload nginx
  service:
    name: "{{ nginx_service }}"
    state: reloaded
```

### roles/nginx/templates/nginx.conf.j2

```jinja2
# {{ ansible_managed }}
# Nginx main configuration

user {{ nginx_user }};
worker_processes {{ nginx_worker_processes }};
pid /run/nginx.pid;
include /etc/nginx/modules-enabled/*.conf;

events {
    worker_connections {{ nginx_worker_connections }};
    multi_accept on;
}

http {
    # Basic Settings
    sendfile on;
    tcp_nopush on;
    tcp_nodelay on;
    keepalive_timeout {{ nginx_keepalive_timeout }};
    types_hash_max_size 2048;
    client_max_body_size {{ nginx_client_max_body_size }};

    include /etc/nginx/mime.types;
    default_type application/octet-stream;

    # Logging
    access_log {{ nginx_log_dir }}/access.log;
    error_log {{ nginx_log_dir }}/error.log;

    # Gzip Compression
    gzip on;
    gzip_vary on;
    gzip_proxied any;
    gzip_comp_level 6;
    gzip_types text/plain text/css application/json application/javascript text/xml application/xml;

    # Virtual Host Configs
    include /etc/nginx/conf.d/*.conf;
    include /etc/nginx/sites-enabled/*;
}
```

### roles/nginx/templates/vhost.conf.j2

```jinja2
# {{ ansible_managed }}
# Virtual Host: {{ nginx_server_name }}

server {
    listen {{ nginx_port }};
    server_name {{ nginx_server_name }};

{% if nginx_ssl_enabled %}
    listen 443 ssl http2;
    ssl_certificate {{ nginx_ssl_cert }};
    ssl_certificate_key {{ nginx_ssl_key }};
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;
{% endif %}

{% if nginx_proxy_enabled %}
    location / {
        proxy_pass {{ nginx_proxy_pass }};
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;
    }
{% else %}
    root {{ nginx_root }};
    index index.html index.htm;

    location / {
        try_files $uri $uri/ =404;
    }
{% endif %}

    # Error pages
    error_page 500 502 503 504 /50x.html;
    location = /50x.html {
        root /usr/share/nginx/html;
    }

    # Deny access to hidden files
    location ~ /\. {
        deny all;
        access_log off;
        log_not_found off;
    }
}
```

### roles/nginx/meta/main.yml

```yaml
# roles/nginx/meta/main.yml
---
galaxy_info:
  author: DevOps Team
  description: Install and configure Nginx
  license: MIT
  min_ansible_version: "2.9"
  platforms:
    - name: Ubuntu
      versions:
        - focal
        - jammy
  galaxy_tags:
    - nginx
    - web
    - proxy

dependencies: []
```

### site.yml — Using the Role

```yaml
# site.yml
---
- name: Setup Web Servers
  hosts: webservers
  become: yes

  roles:
    - role: nginx
      vars:
        nginx_port: 80
        nginx_server_name: myapp.example.com
        nginx_proxy_enabled: true
        nginx_proxy_pass: "http://127.0.0.1:3000"
```

That's it! One clean playbook that sets up a fully configured Nginx server with a reverse proxy.

## 🌐 Ansible Galaxy — The Role Marketplace

**Ansible Galaxy** is a **public repository** of community-created roles. Instead of writing everything from scratch, you can **download pre-built roles** for common tasks.

Think of it like:
- **npm** for Node.js packages
- **pip** for Python packages
- **Docker Hub** for container images
- **Ansible Galaxy** for Ansible roles

### Galaxy Website

Browse roles at: [https://galaxy.ansible.com](https://galaxy.ansible.com)

### Common Galaxy Commands

#### 1. Search for Roles

```bash
# Search for nginx roles
ansible-galaxy search nginx

# Search with platform filter
ansible-galaxy search nginx --platforms Ubuntu

# Search by author
ansible-galaxy search --author geerlingguy
```

#### 2. Install a Role

```bash
# Install a role from Galaxy
ansible-galaxy install geerlingguy.nginx

# Install a specific version
ansible-galaxy install geerlingguy.nginx,3.1.0

# Install to a custom directory
ansible-galaxy install geerlingguy.nginx -p ./roles/

# Install from a Git repository
ansible-galaxy install git+https://github.com/geerlingguy/ansible-role-nginx.git

# Install from a Git repo with a specific branch/tag
ansible-galaxy install git+https://github.com/geerlingguy/ansible-role-nginx.git,main
```

#### 3. Install Multiple Roles from a Requirements File

Create a `requirements.yml`:

```yaml
# requirements.yml
---
roles:
  - name: geerlingguy.nginx
    version: "3.1.0"

  - name: geerlingguy.docker
    version: "6.1.0"

  - name: geerlingguy.mysql
    version: "4.0.0"

  - name: geerlingguy.certbot
    version: "5.0.0"

  # From a Git repository
  - name: my-custom-role
    src: git+https://github.com/myorg/ansible-role-custom.git
    version: v1.2.0

  # From a local path (useful in development)
  # - name: local-role
  #   src: /path/to/local/role
```

Install all roles at once:

```bash
ansible-galaxy install -r requirements.yml

# Force reinstall (update)
ansible-galaxy install -r requirements.yml --force

# Install to a specific directory
ansible-galaxy install -r requirements.yml -p ./roles/
```

#### 4. List Installed Roles

```bash
ansible-galaxy list
```

#### 5. Remove a Role

```bash
ansible-galaxy remove geerlingguy.nginx
```

#### 6. View Role Info

```bash
ansible-galaxy info geerlingguy.nginx
```

## 📂 Ansible Collections — The Evolution Beyond Roles

Starting from Ansible 2.9+, **Collections** became the new way to distribute Ansible content. Collections can include:
- Roles
- Modules
- Plugins
- Playbooks

Think of it this way:
```
Role       = A reusable set of tasks (like an npm package)
Collection = A bundle of roles + modules + plugins (like an npm org scope)
```

### Installing Collections

```bash
# Install from Galaxy
ansible-galaxy collection install community.general

# Install from requirements file
# requirements.yml
collections:
  - name: community.general
    version: ">=5.0.0"
  - name: amazon.aws
    version: "5.0.0"

ansible-galaxy collection install -r requirements.yml
```

## 🏆 Best Practices for Ansible Roles

### 1. Keep Roles Focused — One Role, One Purpose

```
✅ GOOD:
    roles/nginx/        → Only Nginx stuff
    roles/mysql/        → Only MySQL stuff
    roles/app_deploy/   → Only app deployment stuff

❌ BAD:
    roles/web_and_db_and_monitoring/  → Too many responsibilities!
```

### 2. Use `defaults/main.yml` for User-Configurable Variables

```yaml
# ✅ GOOD — Easy for users to override
# defaults/main.yml
nginx_port: 80
nginx_worker_processes: auto

# ❌ BAD — Hardcoded in tasks
# tasks/main.yml
- name: Deploy config
  template:
    src: nginx.conf.j2
    dest: /etc/nginx/nginx.conf
  # Port 80 is hardcoded in the template — user can't change it!
```

### 3. Name Everything — Tasks, Variables, Handlers

```yaml
# ✅ GOOD — Clear names
- name: Install Nginx web server
  apt:
    name: nginx
    state: present

# ❌ BAD — No name
- apt:
    name: nginx
    state: present
```

### 4. Use `{{ ansible_managed }}` in Templates

```jinja2
# ✅ GOOD — Warns people not to edit manually
# {{ ansible_managed }}
# This file is managed by Ansible. Manual changes will be overwritten.

server {
    listen {{ nginx_port }};
}
```

This adds a comment like: `# Ansible managed: /home/user/roles/nginx/templates/nginx.conf.j2 modified on 2024-01-15`

### 5. Prefix Variable Names with the Role Name

```yaml
# ✅ GOOD — No variable name collisions
nginx_port: 80
nginx_root: /var/www/html
nginx_worker_processes: auto

# ❌ BAD — Generic names might collide with other roles
port: 80
root: /var/www/html
workers: auto
```

Imagine if both your `nginx` and `apache` roles used a variable called `port`. They'd overwrite each other! Prefixing prevents this.

### 6. Split Large Task Files

```yaml
# ✅ GOOD — Split into logical files
# tasks/main.yml
- import_tasks: install.yml
- import_tasks: configure.yml
- import_tasks: service.yml
- import_tasks: firewall.yml

# ❌ BAD — Everything in one massive file
# tasks/main.yml → 200 lines of mixed tasks
```

### 7. Include a README.md

Every role should have a `README.md` explaining:
- What the role does
- What variables are available
- Example usage
- Supported platforms

```markdown
# Nginx Role

Installs and configures Nginx.

## Variables

| Variable | Default | Description |
|----------|---------|-------------|
| `nginx_port` | `80` | HTTP listen port |
| `nginx_server_name` | `_` | Server name |
| `nginx_proxy_enabled` | `false` | Enable reverse proxy |

## Example

    - role: nginx
      vars:
        nginx_port: 80
        nginx_server_name: example.com
```

### 8. Use `validate` in Template Tasks

```yaml
# ✅ GOOD — Validate before applying
- name: Deploy nginx config
  template:
    src: nginx.conf.j2
    dest: /etc/nginx/nginx.conf
    validate: "nginx -t -c %s"          # Test config before applying!

# ❌ BAD — No validation (broken config = broken server)
- name: Deploy nginx config
  template:
    src: nginx.conf.j2
    dest: /etc/nginx/nginx.conf
```

### 9. Use Meta Dependencies Wisely

```yaml
# roles/app/meta/main.yml
dependencies:
  - role: common
  - role: nginx
    vars:
      nginx_proxy_enabled: true
      nginx_proxy_pass: "http://127.0.0.1:3000"
```

Be careful with dependencies — they can create **unexpected execution orders** and **duplicate runs**. Use them when a role truly **can't work** without another role.

### 10. Test Your Roles with Molecule

```bash
# Install Molecule
pip install molecule molecule-docker

# Initialize testing for a role
cd roles/nginx
molecule init scenario --driver-name docker

# Run tests
molecule test
```

Molecule creates a test environment (using Docker), applies your role, and verifies it works correctly — all automated!

## 📊 Role vs Playbook vs Collection — Summary

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│  PLAYBOOK                                                       │
│  ├── A YAML file that defines WHAT to do on WHICH servers       │
│  ├── Can contain tasks directly                                 │
│  ├── Can use roles                                              │
│  └── Entry point for ansible-playbook command                   │
│                                                                 │
│  ROLE                                                           │
│  ├── A reusable, self-contained package of automation           │
│  ├── Standard directory structure                               │
│  ├── Contains tasks, handlers, variables, templates, files      │
│  ├── Shared via Ansible Galaxy                                  │
│  └── Used INSIDE playbooks                                      │
│                                                                 │
│  COLLECTION                                                     │
│  ├── A bundle that can contain roles + modules + plugins        │
│  ├── The modern distribution format (Ansible 2.9+)              │
│  ├── Namespace: community.general, amazon.aws, etc.             │
│  └── Installed via ansible-galaxy collection install             │
│                                                                 │
│  Analogy:                                                       │
│  ├── Playbook  = Recipe (instructions for one meal)             │
│  ├── Role      = Recipe component (how to make the sauce)       │
│  └── Collection = Cookbook (complete set of recipes & tools)     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```
