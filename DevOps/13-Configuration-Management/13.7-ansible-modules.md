## 📌 What is an Ansible Module?

Alright, let's keep this simple.

A **module** in Ansible is basically a **small program** that Ansible runs on the target machine (or sometimes on the control node itself) to **do something specific** — like install a package, copy a file, start a service, create a user, etc.

Think of modules like **tools in a toolbox**:

```
🔧 apt module       → Install/remove packages on Debian/Ubuntu
🔧 yum module       → Install/remove packages on RHEL/CentOS
🔧 copy module      → Copy a file from your machine to the server
🔧 template module  → Copy a file with dynamic values (Jinja2)
🔧 service module   → Start, stop, restart services
🔧 file module      → Create files, directories, set permissions
🔧 docker module    → Manage Docker containers & images
🔧 k8s module       → Manage Kubernetes resources
```

You don't write the low-level code. You just **tell the module what you want**, and it figures out how to do it. That's the beauty of Ansible — **declarative automation**.

### How Many Modules Are There?

Ansible has **thousands** of modules. They're organized into **collections**:

| Collection | Example Modules | Purpose |
|---|---|---|
| `ansible.builtin` | file, copy, template, service, apt, yum, user, group | Core OS tasks |
| `community.docker` | docker_container, docker_image, docker_network | Docker management |
| `kubernetes.core` | k8s, k8s_info, helm | Kubernetes management |
| `amazon.aws` | ec2_instance, s3_bucket, iam_role | AWS resources |
| `azure.azcollection` | azure_rm_virtualmachine | Azure resources |
| `community.mysql` | mysql_db, mysql_user | MySQL databases |

### How a Module Actually Works — Behind the Scenes

Here's what happens when Ansible runs a module:

```
1. You write a task in your playbook → "install nginx using apt"

2. Ansible takes that task and packages it into a small Python script

3. Ansible connects to the target machine via SSH

4. Ansible copies that Python script to the target machine (/tmp/.ansible/...)

5. Ansible executes the script on the target machine

6. The script does its thing (installs nginx) and returns a JSON result

7. Ansible reads the result → success ✅ or failure ❌

8. Ansible cleans up the temporary script

9. Ansible moves to the next task
```

> **Key Point:** Most modules are **idempotent** — meaning if you run them 10 times, they'll only make changes the first time (when the desired state doesn't match). After that, they'll say "ok, already done" and skip. This is HUGE — it means you can safely re-run your playbooks without breaking things.

## 📁 The `file` Module — Managing Files & Directories

The `file` module is your go-to for creating, deleting, and managing files and directories on remote machines.

### What Can It Do?

| Action | Description |
|---|---|
| Create a file | Creates an empty file |
| Create a directory | Creates a directory (like `mkdir -p`) |
| Delete a file/directory | Removes files or directories |
| Set permissions | Changes `chmod` (e.g., 0755) |
| Set ownership | Changes `chown` (owner:group) |
| Create symlinks | Creates symbolic links |

### Basic Examples

#### 1. Create a Directory

```yaml
- name: Create the application directory
  ansible.builtin.file:
    path: /opt/myapp
    state: directory
    owner: deploy
    group: deploy
    mode: '0755'
```

**What's happening here?**
- `path` → Where to create the directory
- `state: directory` → "I want this to be a directory" (creates it if missing)
- `owner: deploy` → Set the owner to the `deploy` user
- `group: deploy` → Set the group to `deploy`
- `mode: '0755'` → Set permissions (rwxr-xr-x)

> **Tip:** Always quote the `mode` value as a string (e.g., `'0755'`). If you write `0755` without quotes, YAML may interpret it as an octal number and things can go wrong.

#### 2. Create an Empty File

```yaml
- name: Create an empty config file
  ansible.builtin.file:
    path: /opt/myapp/config.ini
    state: touch
    owner: deploy
    group: deploy
    mode: '0644'
```

`state: touch` works just like the `touch` command in Linux — creates the file if it doesn't exist, updates the timestamp if it does.

#### 3. Create a Symbolic Link

```yaml
- name: Create a symlink for the current release
  ansible.builtin.file:
    src: /opt/myapp/releases/v2.1.0
    dest: /opt/myapp/current
    state: link
```

This creates: `/opt/myapp/current → /opt/myapp/releases/v2.1.0`

#### 4. Delete a File

```yaml
- name: Remove the old log file
  ansible.builtin.file:
    path: /var/log/myapp/old.log
    state: absent
```

`state: absent` = "make sure this doesn't exist." If it's there, delete it. If it's already gone, do nothing.

#### 5. Set Permissions Recursively on a Directory

```yaml
- name: Set ownership recursively on app directory
  ansible.builtin.file:
    path: /opt/myapp
    state: directory
    owner: deploy
    group: deploy
    recurse: yes
```

`recurse: yes` → Apply the ownership change to ALL files and subdirectories inside `/opt/myapp`.

### All `state` Options for the `file` Module

| State | What It Does |
|---|---|
| `file` | Verifies a file exists (does NOT create it). Fails if missing. |
| `directory` | Creates a directory (like `mkdir -p`) |
| `touch` | Creates an empty file (like `touch`) |
| `link` | Creates a symbolic link |
| `hard` | Creates a hard link |
| `absent` | Deletes the file or directory |

## 📋 The `copy` Module — Copying Files to Remote Machines

The `copy` module copies a file from your **Ansible control node** (your laptop/server) to the **target machine**.

### Basic Example

```yaml
- name: Copy the application config file
  ansible.builtin.copy:
    src: files/app.conf
    dest: /etc/myapp/app.conf
    owner: root
    group: root
    mode: '0644'
```

**What's happening?**
- `src` → The file on YOUR machine (relative to the playbook or role's `files/` directory)
- `dest` → Where to put it on the TARGET machine
- `owner`, `group`, `mode` → Set permissions on the remote file

> **Important:** The `src` path is relative to the playbook directory (or the role's `files/` directory). So if your playbook is at `/home/user/ansible/site.yml`, and you say `src: files/app.conf`, Ansible looks for `/home/user/ansible/files/app.conf`.

### Copy with Content (Inline Content)

Sometimes you don't have a file to copy — you just want to create a file with specific content:

```yaml
- name: Create a motd file with custom content
  ansible.builtin.copy:
    content: |
      ==========================================
      Welcome to {{ inventory_hostname }}
      Environment: {{ env }}
      Managed by Ansible — DO NOT EDIT MANUALLY
      ==========================================
    dest: /etc/motd
    owner: root
    group: root
    mode: '0644'
```

`content` replaces `src` — instead of copying a file, it creates the file with this content directly.

### Copy with Backup

```yaml
- name: Copy nginx config with backup
  ansible.builtin.copy:
    src: files/nginx.conf
    dest: /etc/nginx/nginx.conf
    owner: root
    group: root
    mode: '0644'
    backup: yes
```

`backup: yes` → Before overwriting the existing file, Ansible saves a backup copy with a timestamp (e.g., `nginx.conf.2026-05-30@12:30:45~`). Super useful for safety!

### Copy a Whole Directory

```yaml
- name: Copy the entire config directory
  ansible.builtin.copy:
    src: files/configs/
    dest: /opt/myapp/configs/
    owner: deploy
    group: deploy
    mode: '0644'
    directory_mode: '0755'
```

> **Trailing slash matters!** If `src` ends with `/`, it copies the **contents** of the directory. Without the trailing slash, it copies the directory itself.

### Validate Before Copying

```yaml
- name: Copy sudoers file with validation
  ansible.builtin.copy:
    src: files/sudoers
    dest: /etc/sudoers
    owner: root
    group: root
    mode: '0440'
    validate: /usr/sbin/visudo -cf %s
```

`validate` → Before writing the file to its final destination, Ansible runs this command against the **temporary file** (`%s` = temp file path). If validation fails, the file is NOT copied. This prevents you from deploying a broken sudoers file and locking yourself out!

## 🎨 The `template` Module — Dynamic Files with Jinja2

The `template` module is like `copy`, but **on steroids**. It takes a **Jinja2 template** file and fills in variables before copying it to the remote machine.

### Why Use `template` Instead of `copy`?

| `copy` | `template` |
|---|---|
| Copies the file as-is | Processes Jinja2 variables/logic first |
| Static content | Dynamic content |
| Use when file doesn't change per host | Use when file varies per host/environment |

### Basic Example

**Step 1: Create the template file** — `templates/nginx.conf.j2`

```jinja2
# Nginx Configuration — Managed by Ansible
# Server: {{ inventory_hostname }}
# Generated on: {{ ansible_date_time.date }}

worker_processes {{ ansible_processor_vcpus }};

events {
    worker_connections {{ nginx_worker_connections | default(1024) }};
}

http {
    server {
        listen {{ nginx_port }};
        server_name {{ server_name }};

        root {{ document_root }};

        location / {
            try_files $uri $uri/ =404;
        }

        {% if enable_ssl | default(false) %}
        listen 443 ssl;
        ssl_certificate {{ ssl_cert_path }};
        ssl_certificate_key {{ ssl_key_path }};
        {% endif %}
    }
}
```

**Step 2: Use the template in your playbook**

```yaml
- name: Deploy nginx configuration
  ansible.builtin.template:
    src: templates/nginx.conf.j2
    dest: /etc/nginx/nginx.conf
    owner: root
    group: root
    mode: '0644'
    backup: yes
  notify: Restart nginx
```

**What's happening?**
1. Ansible reads `nginx.conf.j2`
2. Replaces `{{ inventory_hostname }}` with the actual hostname
3. Replaces `{{ ansible_processor_vcpus }}` with the number of CPUs on that server
4. Evaluates the `{% if enable_ssl %}` condition
5. Writes the final rendered file to `/etc/nginx/nginx.conf` on the remote machine

> **Convention:** Template files use the `.j2` extension (Jinja2). They live in the `templates/` directory.

### Template with Loops

```jinja2
# /etc/hosts — Managed by Ansible
127.0.0.1   localhost

{% for host in groups['webservers'] %}
{{ hostvars[host]['ansible_host'] }}   {{ host }}
{% endfor %}
```

This generates a dynamic `/etc/hosts` file with all your webservers listed!

### Template with Filters

```jinja2
# Application Config
APP_NAME={{ app_name | upper }}
DB_HOST={{ db_host | default('localhost') }}
SECRET_KEY={{ secret_key | hash('sha256') }}
WORKERS={{ ansible_processor_vcpus * 2 }}
ALLOWED_HOSTS={{ allowed_hosts | join(',') }}
```

Common Jinja2 filters:

| Filter | What It Does | Example |
|---|---|---|
| `upper` | Converts to uppercase | `{{ name | upper }}` → `MYAPP` |
| `lower` | Converts to lowercase | `{{ name | lower }}` → `myapp` |
| `default(val)` | Sets a default if variable is undefined | `{{ port | default(8080) }}` |
| `join(',')` | Joins a list into a string | `{{ list | join(',') }}` |
| `int` | Converts to integer | `{{ val | int }}` |
| `hash('sha256')` | Hashes a string | `{{ pass | hash('sha256') }}` |
| `to_yaml` | Converts to YAML string | `{{ dict | to_yaml }}` |
| `to_json` | Converts to JSON string | `{{ dict | to_json }}` |
| `regex_replace` | Regex find & replace | `{{ str | regex_replace('old','new') }}` |

### Validate Templates Before Deploying

```yaml
- name: Deploy Apache config
  ansible.builtin.template:
    src: templates/httpd.conf.j2
    dest: /etc/httpd/conf/httpd.conf
    validate: /usr/sbin/httpd -t -f %s
```

Just like `copy`, you can validate the rendered template before writing it.

## ⚙️ The `service` Module — Managing System Services

The `service` module starts, stops, restarts, and enables/disables system services (like nginx, mysql, docker, etc.).

### Basic Examples

#### Start and Enable a Service

```yaml
- name: Start and enable nginx
  ansible.builtin.service:
    name: nginx
    state: started
    enabled: yes
```

- `state: started` → Make sure nginx is running. If it's already running, do nothing.
- `enabled: yes` → Make sure nginx starts automatically on boot.

#### Stop a Service

```yaml
- name: Stop the old application server
  ansible.builtin.service:
    name: old-app
    state: stopped
    enabled: no
```

#### Restart a Service

```yaml
- name: Restart nginx after config change
  ansible.builtin.service:
    name: nginx
    state: restarted
```

> **Warning:** `state: restarted` will ALWAYS restart the service, even if nothing changed. For smarter behavior, use **handlers** (covered below).

#### Reload a Service (Graceful)

```yaml
- name: Reload nginx configuration
  ansible.builtin.service:
    name: nginx
    state: reloaded
```

`reloaded` is gentler than `restarted` — it tells the service to re-read its config without fully stopping and starting. No downtime!

### All Service States

| State | Behavior |
|---|---|
| `started` | Start if not running. Do nothing if already running. |
| `stopped` | Stop if running. Do nothing if already stopped. |
| `restarted` | Always restart (stop + start). Not idempotent! |
| `reloaded` | Reload config without stopping. |

### Using Handlers (The Smart Way to Restart)

Instead of always restarting, use **handlers** — they only trigger when something actually changes:

```yaml
# In your tasks
- name: Deploy nginx config
  ansible.builtin.template:
    src: templates/nginx.conf.j2
    dest: /etc/nginx/nginx.conf
  notify: Restart nginx      # Only triggers if the config file actually changed

# In your handlers
handlers:
  - name: Restart nginx
    ansible.builtin.service:
      name: nginx
      state: restarted
```

How this works:
1. If the template produces a **new** config file → handler gets **notified** → nginx restarts at the end of the play
2. If the template produces the **same** config file → handler is NOT notified → nginx stays running untouched

### `service` vs `systemd` Module

Ansible also has a `systemd` module for more advanced control:

```yaml
# Using the systemd module for more control
- name: Restart and daemon-reload
  ansible.builtin.systemd:
    name: myapp
    state: restarted
    daemon_reload: yes    # Equivalent to: systemctl daemon-reload
    scope: system         # system, user, or global
```

| Feature | `service` module | `systemd` module |
|---|---|---|
| Works with SysVinit | ✅ | ❌ |
| Works with systemd | ✅ | ✅ |
| `daemon_reload` | ❌ | ✅ |
| Scope control | ❌ | ✅ |
| Masking services | ❌ | ✅ |

**Rule of thumb:** Use `service` for basic start/stop/restart. Use `systemd` when you need systemd-specific features like `daemon_reload`.

## 📦 The `apt` Module — Package Management (Debian/Ubuntu)

The `apt` module manages packages on Debian-based systems (Ubuntu, Debian, Linux Mint, etc.).

### Install a Package

```yaml
- name: Install nginx
  ansible.builtin.apt:
    name: nginx
    state: present
```

`state: present` = "Make sure this package is installed." If it's already there, do nothing.

### Install a Specific Version

```yaml
- name: Install a specific version of nginx
  ansible.builtin.apt:
    name: nginx=1.24.0-1ubuntu1
    state: present
```

### Install Multiple Packages

```yaml
- name: Install required packages
  ansible.builtin.apt:
    name:
      - nginx
      - git
      - curl
      - wget
      - vim
      - htop
      - unzip
    state: present
    update_cache: yes
```

`update_cache: yes` → Runs `apt-get update` before installing. Always a good idea to make sure the package index is fresh.

### Update Cache with Time Limit

```yaml
- name: Install packages (update cache if older than 1 hour)
  ansible.builtin.apt:
    name: nginx
    state: present
    update_cache: yes
    cache_valid_time: 3600    # Don't update if cache is less than 3600 seconds old
```

`cache_valid_time: 3600` → Only runs `apt-get update` if the cache is older than 1 hour. Saves time on repeated runs!

### Remove a Package

```yaml
- name: Remove apache2 (we use nginx instead)
  ansible.builtin.apt:
    name: apache2
    state: absent
```

### Remove a Package and Its Config Files

```yaml
- name: Purge apache2 completely
  ansible.builtin.apt:
    name: apache2
    state: absent
    purge: yes
    autoremove: yes
```

- `purge: yes` → Remove the package AND its configuration files
- `autoremove: yes` → Remove any dependencies that are no longer needed

### Upgrade All Packages

```yaml
- name: Upgrade all packages to latest version
  ansible.builtin.apt:
    upgrade: dist
    update_cache: yes
```

| Upgrade Type | What It Does |
|---|---|
| `yes` | `apt-get upgrade` (safe upgrade) |
| `safe` | Same as `yes` |
| `full` | `apt-get full-upgrade` |
| `dist` | `apt-get dist-upgrade` (handles dependency changes) |

### Install a .deb File

```yaml
- name: Install a local .deb package
  ansible.builtin.apt:
    deb: /tmp/myapp_1.0.0_amd64.deb
```

### Add a Repository and Install

```yaml
- name: Add Docker GPG key
  ansible.builtin.apt_key:
    url: https://download.docker.com/linux/ubuntu/gpg
    state: present

- name: Add Docker repository
  ansible.builtin.apt_repository:
    repo: "deb [arch=amd64] https://download.docker.com/linux/ubuntu {{ ansible_distribution_release }} stable"
    state: present

- name: Install Docker
  ansible.builtin.apt:
    name: docker-ce
    state: present
    update_cache: yes
```

## 📦 The `yum` Module — Package Management (RHEL/CentOS)

The `yum` module does the same thing as `apt`, but for Red Hat-based systems (RHEL, CentOS, Fedora, Amazon Linux).

### Install a Package

```yaml
- name: Install nginx on CentOS
  ansible.builtin.yum:
    name: nginx
    state: present
```

### Install Multiple Packages

```yaml
- name: Install required packages
  ansible.builtin.yum:
    name:
      - nginx
      - git
      - curl
      - vim
      - htop
    state: present
```

### Install a Specific Version

```yaml
- name: Install specific version
  ansible.builtin.yum:
    name: nginx-1.24.0
    state: present
```

### Remove a Package

```yaml
- name: Remove httpd
  ansible.builtin.yum:
    name: httpd
    state: absent
```

### Install from a Specific Repository

```yaml
- name: Install from EPEL repo
  ansible.builtin.yum:
    name: htop
    state: present
    enablerepo: epel
```

### Update All Packages

```yaml
- name: Update all packages
  ansible.builtin.yum:
    name: '*'
    state: latest
```

### Install an RPM from URL

```yaml
- name: Install an RPM from URL
  ansible.builtin.yum:
    name: https://example.com/myapp-1.0.0.rpm
    state: present
```

### `dnf` Module (Fedora / RHEL 8+)

For newer RHEL-based systems, use the `dnf` module instead:

```yaml
- name: Install with dnf
  ansible.builtin.dnf:
    name: nginx
    state: present
```

The syntax is almost identical to `yum`. Just swap the module name.

### `apt` vs `yum` — Quick Comparison

| Feature | `apt` (Debian/Ubuntu) | `yum` (RHEL/CentOS) |
|---|---|---|
| Update cache | `update_cache: yes` | Not needed (auto) |
| Purge config | `purge: yes` | Not available |
| Auto-remove deps | `autoremove: yes` | `autoremove: yes` |
| Install .deb/.rpm | `deb: /path/file.deb` | `name: /path/file.rpm` |
| Enable repo | Not applicable | `enablerepo: epel` |

> **Pro Tip:** To write OS-agnostic playbooks, use the `package` module:
> ```yaml
> - name: Install git (works on any OS)
>   ansible.builtin.package:
>     name: git
>     state: present
> ```
> The `package` module auto-detects whether to use `apt`, `yum`, or `dnf`.

## 🐳 The `docker_container` Module — Managing Docker Containers

The `community.docker` collection lets you manage Docker containers, images, networks, and volumes through Ansible.

### Prerequisites

First, install the Docker collection:

```bash
ansible-galaxy collection install community.docker
```

Also, the target machine needs Docker installed and the `docker` Python SDK:

```yaml
- name: Install Docker Python SDK
  ansible.builtin.pip:
    name: docker
    state: present
```

### Run a Container

```yaml
- name: Run an nginx container
  community.docker.docker_container:
    name: my-nginx
    image: nginx:latest
    state: started
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - /opt/nginx/html:/usr/share/nginx/html:ro
      - /opt/nginx/conf:/etc/nginx/conf.d:ro
    restart_policy: unless-stopped
    env:
      NGINX_HOST: "example.com"
      NGINX_PORT: "80"
```

**Explanation:**
- `name` → Container name (like `docker run --name my-nginx`)
- `image` → Docker image to use
- `state: started` → Make sure the container is running
- `ports` → Port mappings (host:container)
- `volumes` → Volume mounts (host:container:options)
- `restart_policy` → Auto-restart behavior
- `env` → Environment variables

### All Container States

| State | What It Does |
|---|---|
| `started` | Create and start (or just start if exists) |
| `stopped` | Stop the container (keep it) |
| `present` | Create but don't start |
| `absent` | Remove the container |

### Stop and Remove a Container

```yaml
- name: Stop and remove the old container
  community.docker.docker_container:
    name: old-app
    state: absent
    force_kill: yes
```

### Run a Container with Health Check

```yaml
- name: Run app with health check
  community.docker.docker_container:
    name: my-app
    image: myapp:v2.0
    state: started
    ports:
      - "8080:8080"
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s
```

### Pull a Docker Image

```yaml
- name: Pull the latest nginx image
  community.docker.docker_image:
    name: nginx
    tag: latest
    source: pull
```

### Create a Docker Network

```yaml
- name: Create an application network
  community.docker.docker_network:
    name: app-network
    driver: bridge
    ipam_config:
      - subnet: 172.20.0.0/16
        gateway: 172.20.0.1
```

### Create a Docker Volume

```yaml
- name: Create a data volume
  community.docker.docker_volume:
    name: app-data
    state: present
```

### Full Docker Compose-Style Playbook

Here's a real-world example deploying a complete application stack:

```yaml
---
- name: Deploy Application Stack with Docker
  hosts: docker_servers
  become: yes

  tasks:
    - name: Create app network
      community.docker.docker_network:
        name: app-net
        state: present

    - name: Create data volume for PostgreSQL
      community.docker.docker_volume:
        name: pg-data
        state: present

    - name: Run PostgreSQL
      community.docker.docker_container:
        name: postgres
        image: postgres:16
        state: started
        restart_policy: unless-stopped
        networks:
          - name: app-net
        volumes:
          - pg-data:/var/lib/postgresql/data
        env:
          POSTGRES_DB: myapp
          POSTGRES_USER: appuser
          POSTGRES_PASSWORD: "{{ vault_db_password }}"

    - name: Run Redis
      community.docker.docker_container:
        name: redis
        image: redis:7-alpine
        state: started
        restart_policy: unless-stopped
        networks:
          - name: app-net

    - name: Run Application
      community.docker.docker_container:
        name: myapp
        image: "myapp:{{ app_version }}"
        state: started
        restart_policy: unless-stopped
        ports:
          - "8080:8080"
        networks:
          - name: app-net
        env:
          DATABASE_URL: "postgresql://appuser:{{ vault_db_password }}@postgres:5432/myapp"
          REDIS_URL: "redis://redis:6379"
          APP_ENV: "{{ environment }}"
```

## ☸️ The `k8s` Module — Managing Kubernetes Resources

The `kubernetes.core` collection lets you manage Kubernetes resources with Ansible — create deployments, services, configmaps, and anything else you'd normally do with `kubectl apply`.

### Prerequisites

Install the Kubernetes collection and Python dependencies:

```bash
# Install the collection
ansible-galaxy collection install kubernetes.core

# Install Python dependencies
pip install kubernetes
```

### Apply a Kubernetes Manifest (Inline)

```yaml
- name: Create a Namespace
  kubernetes.core.k8s:
    state: present
    definition:
      apiVersion: v1
      kind: Namespace
      metadata:
        name: my-app
        labels:
          environment: production
```

This is like running `kubectl apply -f namespace.yaml`, but defined right inside your playbook!

### Create a Deployment

```yaml
- name: Deploy nginx to Kubernetes
  kubernetes.core.k8s:
    state: present
    definition:
      apiVersion: apps/v1
      kind: Deployment
      metadata:
        name: nginx-deployment
        namespace: my-app
        labels:
          app: nginx
      spec:
        replicas: 3
        selector:
          matchLabels:
            app: nginx
        template:
          metadata:
            labels:
              app: nginx
          spec:
            containers:
              - name: nginx
                image: nginx:1.25
                ports:
                  - containerPort: 80
                resources:
                  requests:
                    memory: "64Mi"
                    cpu: "250m"
                  limits:
                    memory: "128Mi"
                    cpu: "500m"
```

### Create a Service

```yaml
- name: Expose nginx as a service
  kubernetes.core.k8s:
    state: present
    definition:
      apiVersion: v1
      kind: Service
      metadata:
        name: nginx-service
        namespace: my-app
      spec:
        selector:
          app: nginx
        ports:
          - port: 80
            targetPort: 80
        type: LoadBalancer
```

### Apply from a YAML File

Instead of writing the definition inline, you can apply an existing YAML file:

```yaml
- name: Apply deployment from file
  kubernetes.core.k8s:
    state: present
    src: files/deployment.yaml
```

### Apply from a Jinja2 Template

```yaml
- name: Apply deployment from template
  kubernetes.core.k8s:
    state: present
    template: templates/deployment.yaml.j2
```

This is super powerful — you can template your Kubernetes manifests with variables!

### Delete a Resource

```yaml
- name: Delete the old deployment
  kubernetes.core.k8s:
    state: absent
    api_version: apps/v1
    kind: Deployment
    namespace: my-app
    name: old-deployment
```

### Get Information About Resources

```yaml
- name: Get all pods in namespace
  kubernetes.core.k8s_info:
    api_version: v1
    kind: Pod
    namespace: my-app
  register: pod_list

- name: Display pod names
  ansible.builtin.debug:
    msg: "Pod: {{ item.metadata.name }} — Status: {{ item.status.phase }}"
  loop: "{{ pod_list.resources }}"
```

### Wait for a Deployment to Be Ready

```yaml
- name: Deploy application
  kubernetes.core.k8s:
    state: present
    src: files/deployment.yaml
    wait: yes
    wait_timeout: 300
    wait_condition:
      type: Available
      status: "True"
```

`wait: yes` → Ansible will wait until the deployment is fully ready before moving on. No more guessing!

    release_namespace: monitoring

      grafana:
        adminPassword: "{{ vault_grafana_password }}"

## 🧩 Other Commonly Used Modules — Quick Reference

Here's a quick reference for other modules you'll use ALL the time:

### `user` Module — Managing Users

```yaml
- name: Create a deploy user
  ansible.builtin.user:
    name: deploy
    shell: /bin/bash
    groups: docker,sudo
    append: yes
    create_home: yes
    password: "{{ user_password | password_hash('sha512') }}"
```

### `group` Module — Managing Groups

```yaml
- name: Create the docker group
  ansible.builtin.group:
    name: docker
    state: present
```

### `lineinfile` Module — Edit a Single Line in a File

```yaml
- name: Enable IP forwarding in sysctl
  ansible.builtin.lineinfile:
    path: /etc/sysctl.conf
    regexp: '^#?net.ipv4.ip_forward'
    line: 'net.ipv4.ip_forward = 1'
    state: present
```

### `blockinfile` Module — Insert a Block of Text

```yaml
- name: Add SSH banner
  ansible.builtin.blockinfile:
    path: /etc/ssh/sshd_config
    block: |
      Banner /etc/ssh/banner
      MaxAuthTries 3
      PermitRootLogin no
    marker: "# {mark} ANSIBLE MANAGED - SSH SECURITY"
```

### `cron` Module — Managing Cron Jobs

```yaml
- name: Add a backup cron job
  ansible.builtin.cron:
    name: "Database backup"
    minute: "0"
    hour: "2"
    job: "/opt/scripts/backup.sh >> /var/log/backup.log 2>&1"
    user: root
```

### `git` Module — Clone Repositories

```yaml
- name: Clone the application repository
  ansible.builtin.git:
    repo: https://github.com/myorg/myapp.git
    dest: /opt/myapp
    version: main
    force: yes
```

### `command` and `shell` Modules — Running Commands

```yaml
# Use command for simple commands (safer, no shell features)
- name: Check disk space
  ansible.builtin.command: df -h
  register: disk_space

# Use shell when you need pipes, redirects, or shell features
- name: Find large log files
  ansible.builtin.shell: find /var/log -name "*.log" -size +100M | head -10
  register: large_logs
```

> **Best Practice:** Prefer **specific modules** (apt, service, file) over `command`/`shell`. Modules are idempotent, `command`/`shell` are NOT.

### `debug` Module — Print Variables for Debugging

```yaml
- name: Show a variable
  ansible.builtin.debug:
    var: ansible_distribution

- name: Show a message
  ansible.builtin.debug:
    msg: "Running on {{ ansible_distribution }} {{ ansible_distribution_version }}"
```

### `wait_for` Module — Wait for Conditions

```yaml
- name: Wait for port 8080 to be available
  ansible.builtin.wait_for:
    port: 8080
    host: localhost
    timeout: 60
    delay: 5
```

### `uri` Module — Make HTTP Requests

```yaml
- name: Check if the application is healthy
  ansible.builtin.uri:
    url: "http://localhost:8080/health"
    method: GET
    status_code: 200
    return_content: yes
  register: health_check
  retries: 5
  delay: 10
  until: health_check.status == 200
```

## 📝 Summary — Which Module to Use When?

| I want to... | Use this module |
|---|---|
| Create/delete files and directories | `file` |
| Copy a static file to remote | `copy` |
| Copy a file with dynamic variables | `template` |
| Start/stop/restart services | `service` / `systemd` |
| Install packages (Debian/Ubuntu) | `apt` |
| Install packages (RHEL/CentOS) | `yum` / `dnf` |
| Install packages (any OS) | `package` |
| Manage Docker containers | `docker_container` |
| Manage Kubernetes resources | `k8s` |
| Create users | `user` |
| Edit a single line in a file | `lineinfile` |
| Insert a block of text | `blockinfile` |
| Clone a git repo | `git` |
| Run a raw command | `command` / `shell` |
| Make HTTP requests | `uri` |
| Schedule cron jobs | `cron` |
| Debug/print variables | `debug` |
| Wait for something | `wait_for` |

> **Golden Rule:** Always use a **specific module** when one exists. Only fall back to `command` or `shell` when there's no module for what you need. Modules are idempotent, safer, and give better output.

---

## 🎯 Key Takeaways

1. **Modules are the building blocks** — Every task in a playbook uses a module
2. **Most modules are idempotent** — They only make changes when needed. Safe to re-run!
3. **Use `template` over `copy`** when your files need dynamic content
4. **Use handlers with `service`** to avoid unnecessary restarts
5. **Use `package` module** for OS-agnostic playbooks
6. **Docker & K8s modules** let you manage containers and clusters right from Ansible
7. **Prefer specific modules** over `command`/`shell` — always
8. **`validate` parameter** on `copy`/`template` prevents broken configs from being deployed
