# 13.8 — Ansible Variables — Facts, Vault & Precedence

> **Level:** Intermediate  
> **Goal:** Understand every way to use variables in Ansible — where to define them, how they work, how Ansible auto-discovers system facts, how to encrypt sensitive data with Vault, and the all-important variable precedence order.

---

## 📌 What Are Variables in Ansible?

Variables in Ansible are exactly what you'd expect — **named containers that hold values**. They let you write flexible, reusable playbooks that work across different servers, environments, and scenarios.

Instead of hardcoding things like:

```yaml
# ❌ Bad — hardcoded values everywhere
- name: Install nginx
  apt:
    name: nginx

- name: Set listen port
  template:
    src: nginx.conf.j2
    dest: /etc/nginx/nginx.conf
  # But what if dev uses port 8080 and prod uses port 80?
```

You use variables:

```yaml
# ✅ Good — flexible with variables
- name: Install web server
  apt:
    name: "{{ web_server_package }}"

- name: Set listen port
  template:
    src: nginx.conf.j2
    dest: /etc/nginx/nginx.conf
  # Template uses {{ listen_port }} — different per environment!
```

Now the same playbook works for dev, staging, AND production. Just change the variables!

## 🏗️ Where Can You Define Variables? (All the Places!)

Ansible is VERY flexible about where you can define variables. Here's every place:

### 1. In the Playbook (Play-Level Variables)

```yaml
---
- name: Setup web servers
  hosts: webservers
  vars:
    http_port: 80
    app_name: myapp
    max_connections: 1000
    document_root: /var/www/html

  tasks:
    - name: Show the app name
      debug:
        msg: "Deploying {{ app_name }} on port {{ http_port }}"
```

### 2. In a Separate Variables File (`vars_files`)

```yaml
---
- name: Setup web servers
  hosts: webservers
  vars_files:
    - vars/common.yml
    - vars/{{ environment }}.yml    # Dynamically loads dev.yml, staging.yml, or prod.yml!

  tasks:
    - name: Deploy application
      debug:
        msg: "Deploying to {{ environment }} with DB at {{ db_host }}"
```

**vars/common.yml:**
```yaml
app_name: myapp
app_user: deploy
log_level: info
```

**vars/prod.yml:**
```yaml
environment: production
db_host: prod-db.example.com
db_port: 5432
http_port: 80
replicas: 5
```

**vars/dev.yml:**
```yaml
environment: development
db_host: localhost
db_port: 5432
http_port: 8080
replicas: 1
```

### 3. In the Inventory File

**INI format:**
```ini
[webservers]
web1.example.com http_port=80 app_env=production
web2.example.com http_port=8080 app_env=staging

[webservers:vars]
nginx_worker_processes=auto
max_upload_size=50M

[all:vars]
ansible_user=deploy
ansible_python_interpreter=/usr/bin/python3
```

**YAML format:**
```yaml
all:
  vars:
    ansible_user: deploy
    ansible_python_interpreter: /usr/bin/python3
  children:
    webservers:
      vars:
        nginx_worker_processes: auto
        max_upload_size: 50M
      hosts:
        web1.example.com:
          http_port: 80
          app_env: production
        web2.example.com:
          http_port: 8080
          app_env: staging
```

### 4. In `host_vars` and `group_vars` Directories

This is the **recommended best practice** for organizing variables. Create directories alongside your inventory:

```
inventory/
├── hosts.yml
├── group_vars/
│   ├── all.yml              # Variables for ALL hosts
│   ├── webservers.yml       # Variables for webservers group
│   ├── dbservers.yml        # Variables for dbservers group
│   └── webservers/
│       ├── vars.yml         # Main variables (you can split into multiple files!)
│       └── vault.yml        # Encrypted variables (passwords, keys)
└── host_vars/
    ├── web1.example.com.yml  # Variables for this specific host only
    └── web2.example.com.yml
```

**group_vars/all.yml:**
```yaml
# These apply to EVERY host
ntp_server: time.google.com
dns_servers:
  - 8.8.8.8
  - 8.8.4.4
timezone: UTC
```

**group_vars/webservers.yml:**
```yaml
# These apply only to hosts in the 'webservers' group
nginx_worker_processes: auto
nginx_worker_connections: 1024
ssl_cert_path: /etc/ssl/certs/server.crt
ssl_key_path: /etc/ssl/private/server.key
```

**host_vars/web1.example.com.yml:**
```yaml
# These apply ONLY to web1.example.com
server_id: 1
primary_server: true
custom_header: "X-Server: web1"
```

> **Why is this the best practice?** Because it keeps variables organized, easy to find, and easy to manage. When you're wondering "what variables does the webservers group use?", you just open `group_vars/webservers.yml`.

### 5. Registered Variables (Capture Task Output)

```yaml
- name: Check if Docker is installed
  ansible.builtin.command: docker --version
  register: docker_version
  ignore_errors: yes

- name: Show Docker version
  ansible.builtin.debug:
    var: docker_version.stdout
  when: docker_version.rc == 0

- name: Install Docker if not present
  ansible.builtin.apt:
    name: docker.io
    state: present
  when: docker_version.rc != 0
```

When you `register` a variable, it captures the **entire result** of the task:

```yaml
docker_version:
  stdout: "Docker version 24.0.7, build afdd53b"
  stderr: ""
  rc: 0                    # Return code (0 = success)
  changed: false
  failed: false
  stdout_lines:            # stdout split into a list
    - "Docker version 24.0.7, build afdd53b"
```

### 6. Command-Line Variables (Extra Vars)

The most powerful override — pass variables when running the playbook:

```bash
# Single variable
ansible-playbook site.yml -e "environment=production"

# Multiple variables
ansible-playbook site.yml -e "environment=production version=2.1.0 debug=false"

# From a file
ansible-playbook site.yml -e "@vars/production.yml"

# JSON format
ansible-playbook site.yml -e '{"environment": "production", "replicas": 5}'
```

> **Important:** Extra vars (`-e`) have the **highest priority** and override EVERYTHING else. More on this in the precedence section below.

### 7. `set_fact` — Set Variables During Playbook Execution

```yaml
- name: Calculate worker count based on CPUs
  ansible.builtin.set_fact:
    worker_count: "{{ ansible_processor_vcpus * 2 }}"
    cache_size: "{{ (ansible_memtotal_mb * 0.25) | int }}MB"

- name: Use the calculated values
  ansible.builtin.debug:
    msg: "Using {{ worker_count }} workers with {{ cache_size }} cache"
```

`set_fact` creates a variable that's available for the rest of the play (for that host).

### 8. `include_vars` — Load Variables Dynamically

```yaml
- name: Load OS-specific variables
  ansible.builtin.include_vars:
    file: "{{ ansible_os_family }}.yml"
  # Loads RedHat.yml or Debian.yml based on the OS!

- name: Load environment-specific variables
  ansible.builtin.include_vars:
    dir: vars/
    files_matching: "{{ environment }}.yml"
```

### 9. Role Defaults and Role Variables

When using roles:

```
roles/
└── nginx/
    ├── defaults/
    │   └── main.yml      # Default variables (LOWEST priority — easily overridden)
    ├── vars/
    │   └── main.yml      # Role variables (HIGH priority — harder to override)
    ├── tasks/
    │   └── main.yml
    └── templates/
        └── nginx.conf.j2
```

**defaults/main.yml:**
```yaml
# These are "soft" defaults — users of the role can easily override them
nginx_port: 80
nginx_worker_processes: auto
nginx_worker_connections: 1024
nginx_log_format: combined
```

**vars/main.yml:**
```yaml
# These are "hard" variables — not meant to be overridden
nginx_config_path: /etc/nginx/nginx.conf
nginx_pid_path: /run/nginx.pid
nginx_binary: /usr/sbin/nginx
```

> **Rule of thumb:** Put variables that users SHOULD customize in `defaults/`. Put variables that SHOULDN'T be changed in `vars/`.

## 🔍 Variable Data Types

Ansible supports several data types:

### Strings

```yaml
app_name: myapp
greeting: "Hello, World!"
multiline: |
  This is a
  multiline string
```

### Numbers

```yaml
http_port: 80           # Integer
timeout: 30.5           # Float
max_retries: 3
```

### Booleans

```yaml
enable_ssl: true
debug_mode: false
is_primary: yes          # Also valid (YAML)
force_restart: no        # Also valid (YAML)
```

### Lists (Arrays)

```yaml
packages:
  - nginx
  - git
  - curl
  - vim

# Or inline
packages: [nginx, git, curl, vim]
```

Accessing list items:
```yaml
- debug:
    msg: "First package: {{ packages[0] }}"     # nginx
    msg: "Last package: {{ packages[-1] }}"      # vim
```

### Dictionaries (Maps)

```yaml
database:
  host: db.example.com
  port: 5432
  name: myapp
  user: admin

# Or inline
database: {host: db.example.com, port: 5432, name: myapp}
```

Accessing dictionary values:
```yaml
- debug:
    msg: "DB host: {{ database.host }}"          # Dot notation
    msg: "DB port: {{ database['port'] }}"       # Bracket notation
```

> **When to use bracket notation:** Use `database['port']` instead of `database.port` when the key contains special characters, starts with a number, or conflicts with Python attributes/methods.

### Nested Structures

```yaml
servers:
  web:
    host: web.example.com
    port: 80
    backends:
      - { host: app1, port: 8080 }
      - { host: app2, port: 8080 }
  db:
    host: db.example.com
    port: 5432
```

Accessing nested values:
```yaml
- debug:
    msg: "Web server: {{ servers.web.host }}:{{ servers.web.port }}"
    msg: "First backend: {{ servers.web.backends[0].host }}"
```

## 🤖 Ansible Facts — Auto-Discovered System Information

**Facts** are variables that Ansible **automatically collects** from each target machine at the beginning of every play. They contain a TON of useful information about the system.

### How Facts Are Gathered

When you run a playbook, the very first thing Ansible does is run the **setup module** on each host:

```
PLAY [webservers] ****************************************************

TASK [Gathering Facts] ***********************************************
ok: [web1.example.com]
ok: [web2.example.com]
```

This "Gathering Facts" step collects hundreds of variables about each host.

### Viewing All Facts

```bash
# See ALL facts for a host
ansible web1.example.com -m setup

# Filter facts
ansible web1.example.com -m setup -a "filter=ansible_distribution*"
```

Or in a playbook:
```yaml
- name: Show all facts
  ansible.builtin.debug:
    var: ansible_facts
```

### Most Useful Facts

Here are the facts you'll use most often:

#### OS Information

```yaml
{{ ansible_distribution }}              # Ubuntu, CentOS, RedHat, Debian
{{ ansible_distribution_version }}      # 22.04, 8.5, 9.2
{{ ansible_distribution_release }}      # jammy, focal, bullseye
{{ ansible_distribution_major_version }} # 22, 8, 9
{{ ansible_os_family }}                 # Debian, RedHat, Suse
{{ ansible_system }}                    # Linux, Windows
{{ ansible_architecture }}              # x86_64, aarch64
{{ ansible_kernel }}                    # 5.15.0-76-generic
```

#### Network Information

```yaml
{{ ansible_hostname }}                  # web1 (short hostname)
{{ ansible_fqdn }}                      # web1.example.com (full hostname)
{{ ansible_default_ipv4.address }}      # 192.168.1.10
{{ ansible_default_ipv4.gateway }}      # 192.168.1.1
{{ ansible_default_ipv4.interface }}    # eth0
{{ ansible_all_ipv4_addresses }}        # ["192.168.1.10", "10.0.0.5"]
{{ ansible_dns.nameservers }}           # ["8.8.8.8", "8.8.4.4"]
```

#### Hardware Information

```yaml
{{ ansible_processor_vcpus }}           # 4
{{ ansible_processor_cores }}           # 2
{{ ansible_memtotal_mb }}               # 8192
{{ ansible_memfree_mb }}                # 4096
{{ ansible_swaptotal_mb }}              # 2048
{{ ansible_devices }}                   # Disk devices info
```

#### Date & Time

```yaml
{{ ansible_date_time.date }}            # 2026-05-30
{{ ansible_date_time.time }}            # 14:30:45
{{ ansible_date_time.epoch }}           # 1780300245
{{ ansible_date_time.tz }}              # UTC
```

#### Environment

```yaml
{{ ansible_env.HOME }}                  # /root
{{ ansible_env.PATH }}                  # /usr/local/sbin:/usr/local/bin:...
{{ ansible_env.USER }}                  # root
{{ ansible_user_id }}                   # root
```

### Using Facts in Real Playbooks

#### Example 1: OS-Specific Package Installation

```yaml
- name: Install Apache (OS-specific)
  ansible.builtin.apt:
    name: apache2
    state: present
  when: ansible_os_family == "Debian"

- name: Install Apache (OS-specific)
  ansible.builtin.yum:
    name: httpd
    state: present
  when: ansible_os_family == "RedHat"
```

#### Example 2: Dynamic Configuration Based on Hardware

```yaml
- name: Configure worker processes based on CPUs
  ansible.builtin.template:
    src: nginx.conf.j2
    dest: /etc/nginx/nginx.conf
  # In the template:
  # worker_processes {{ ansible_processor_vcpus }};
```

#### Example 3: Conditional Logic Based on Memory

```yaml
- name: Set JVM heap size based on available memory
  ansible.builtin.set_fact:
    jvm_heap: "{{ (ansible_memtotal_mb * 0.5) | int }}m"

- name: Deploy Java app config
  ansible.builtin.template:
    src: jvm.conf.j2
    dest: /opt/app/jvm.conf
  # Template uses: -Xmx{{ jvm_heap }}
```

### Custom Facts

You can create your own facts on target machines! Place a script or JSON/INI file in `/etc/ansible/facts.d/`:

**On the target machine — `/etc/ansible/facts.d/app_info.fact`:**
```ini
[app]
name=myapp
version=2.1.0
environment=production
```

Or as a script — `/etc/ansible/facts.d/app_info.fact` (must be executable):
```bash
#!/bin/bash
echo '{"app": {"name": "myapp", "version": "2.1.0", "uptime": "'$(uptime -s)'"}}'
```

**Access in your playbook:**
```yaml
- name: Show custom fact
  ansible.builtin.debug:
    msg: "App version: {{ ansible_local.app_info.app.version }}"
```

Custom facts are available under `ansible_local`.

### Disabling Fact Gathering

If you don't need facts (e.g., for a quick task), you can skip gathering to speed things up:

```yaml
---
- name: Quick task — no facts needed
  hosts: all
  gather_facts: no    # Skips the "Gathering Facts" step

  tasks:
    - name: Just restart the service
      service:
        name: nginx
        state: restarted
```

### Caching Facts

For large inventories, gathering facts every time can be slow. You can cache them:

**ansible.cfg:**
```ini
[defaults]
gathering = smart                 # Only gather if facts are not cached
fact_caching = jsonfile            # Cache backend
fact_caching_connection = /tmp/ansible_facts_cache
fact_caching_timeout = 86400      # Cache for 24 hours (in seconds)
```

Other caching backends: `redis`, `memcached`, `yaml`, `mongodb`.

## 🔐 Ansible Vault — Encrypting Sensitive Data

Ansible Vault is Ansible's built-in tool for **encrypting sensitive data** — passwords, API keys, certificates, tokens — anything you don't want sitting in plain text in your repo.

> **Why?** Because storing passwords in plain text in a Git repo is a TERRIBLE idea. Ansible Vault encrypts them so they're safe even if someone sees your repo.

### Creating an Encrypted File

```bash
# Create a new encrypted file
ansible-vault create secrets.yml
```

This opens your editor. Type in your sensitive variables:

```yaml
db_password: SuperSecretP@ssw0rd!
api_key: sk-1234567890abcdef
ssl_private_key: |
  -----BEGIN PRIVATE KEY-----
  MIIEvgIBADANBgkqhkiG9w0BAQEFAASCBKgw...
  -----END PRIVATE KEY-----
```

When you save and close, the file gets **encrypted** with AES-256. It looks like this:

```
$ANSIBLE_VAULT;1.1;AES256
61626364656667686970717273747576777879...
```

### Encrypting an Existing File

```bash
# Encrypt a file that already exists
ansible-vault encrypt vars/production.yml
```

### Editing an Encrypted File

```bash
# Open encrypted file for editing
ansible-vault edit secrets.yml
```

### Viewing an Encrypted File

```bash
# View without editing
ansible-vault view secrets.yml
```

### Decrypting a File

```bash
# Permanently decrypt (back to plain text)
ansible-vault decrypt secrets.yml
```

### Changing the Vault Password

```bash
ansible-vault rekey secrets.yml
```

### Running Playbooks with Vault-Encrypted Files

```bash
# Prompt for the vault password
ansible-playbook site.yml --ask-vault-pass

# Use a password file (better for automation/CI)
ansible-playbook site.yml --vault-password-file ~/.vault_password

# Use an environment variable
export ANSIBLE_VAULT_PASSWORD_FILE=~/.vault_password
ansible-playbook site.yml
```

### Encrypting Individual Variables (Inline Vault)

Instead of encrypting an entire file, you can encrypt **just one variable**:

```bash
# Encrypt a single string
ansible-vault encrypt_string 'SuperSecretP@ssw0rd!' --name 'db_password'
```

Output:
```yaml
db_password: !vault |
          $ANSIBLE_VAULT;1.1;AES256
          61626364656667686970717273747576777879...
```

Now paste this into your regular (unencrypted) variables file:

```yaml
# vars/production.yml
app_name: myapp
http_port: 80
db_host: db.example.com
db_user: admin
db_password: !vault |
          $ANSIBLE_VAULT;1.1;AES256
          61626364656667686970717273747576777879...
```

**This is the best practice!** You can see all your variables at a glance, but the sensitive ones are encrypted inline.

### Using Multiple Vault Passwords (Vault IDs)

For complex environments, you might want different passwords for different environments:

```bash
# Encrypt with a vault ID
ansible-vault encrypt --vault-id prod@prompt secrets-prod.yml
ansible-vault encrypt --vault-id dev@prompt secrets-dev.yml

# Run with multiple vault IDs
ansible-playbook site.yml \
  --vault-id dev@~/.vault_pass_dev \
  --vault-id prod@~/.vault_pass_prod
```

### Best Practices for Vault

```
✅ DO:
   • Keep vault passwords in a file outside the repo (~/.vault_password)
   • Add the password file to .gitignore
   • Use inline vault (encrypt_string) for individual secrets in shared files
   • Use different vault passwords for different environments
   • Use vault-id for multi-environment setups
   • Commit encrypted files to Git (that's the whole point!)

❌ DON'T:
   • Don't commit vault password files to Git
   • Don't share vault passwords in plain text (use a password manager)
   • Don't encrypt everything — only sensitive data
   • Don't use simple/weak vault passwords
```

### Vault in a Real Project — File Structure

```
project/
├── ansible.cfg
├── site.yml
├── inventory/
│   ├── hosts.yml
│   ├── group_vars/
│   │   ├── all/
│   │   │   ├── vars.yml           # Regular variables
│   │   │   └── vault.yml          # 🔐 Encrypted variables
│   │   ├── webservers/
│   │   │   ├── vars.yml
│   │   │   └── vault.yml          # 🔐 
│   │   └── dbservers/
│   │       ├── vars.yml
│   │       └── vault.yml          # 🔐
│   └── host_vars/
│       └── web1.example.com/
│           ├── vars.yml
│           └── vault.yml          # 🔐
└── .gitignore                     # Contains: .vault_password
```

A common pattern is to **prefix vault variables** to make them easy to identify:

**group_vars/webservers/vault.yml (encrypted):**
```yaml
vault_db_password: SuperSecret123
vault_api_key: sk-abcdef1234567890
vault_ssl_key: |
  -----BEGIN PRIVATE KEY-----
  ...
```

**group_vars/webservers/vars.yml (plain text):**
```yaml
# Reference vault variables — easy to see what's used
db_password: "{{ vault_db_password }}"
api_key: "{{ vault_api_key }}"
ssl_key: "{{ vault_ssl_key }}"

# Regular variables
db_host: db.example.com
http_port: 80
```

This way, anyone looking at `vars.yml` can see all variables, even if they can't decrypt the vault file.

## 📊 Variable Precedence — The Order of Priority

This is one of the **most important concepts** in Ansible. When the same variable is defined in multiple places, which one wins?

Ansible has a **22-level precedence order**. Here it is from **lowest** (easily overridden) to **highest** (overrides everything):

```
LOWEST PRIORITY (most easily overridden)
├── 1.  command line values (e.g., -u user — not variables)
├── 2.  role defaults (roles/myrole/defaults/main.yml)
├── 3.  inventory file or script group vars
├── 4.  inventory group_vars/all
├── 5.  playbook group_vars/all
├── 6.  inventory group_vars/*
├── 7.  playbook group_vars/*
├── 8.  inventory file or script host vars
├── 9.  inventory host_vars/*
├── 10. playbook host_vars/*
├── 11. host facts / cached set_facts
├── 12. play vars
├── 13. play vars_prompt
├── 14. play vars_files
├── 15. role vars (roles/myrole/vars/main.yml)
├── 16. block vars (in a block/rescue/always)
├── 17. task vars (only for that specific task)
├── 18. include_vars
├── 19. set_facts / registered vars
├── 20. role params (when calling a role with parameters)
├── 21. include params
├── 22. extra vars (-e "key=value") — ALWAYS WIN!
HIGHEST PRIORITY (overrides everything)
```

### But You Don't Need to Memorize All 22!

In practice, remember these **5 key levels**:

```
📌 Simplified Precedence (most important):

1. Extra vars (-e)           → HIGHEST — always wins, great for overrides
2. Task/Block/Role vars      → HIGH — specific to that context
3. Play vars/vars_files      → MEDIUM — apply to the whole play
4. Host vars (host_vars/)    → MEDIUM — specific to one host
5. Group vars (group_vars/)  → LOW-MEDIUM — apply to a group
6. Role defaults             → LOWEST — designed to be overridden
```

### Precedence Examples

#### Example 1: Role Default vs Play Variable

```yaml
# roles/nginx/defaults/main.yml
nginx_port: 80                    # Priority: 2 (role default)

# site.yml
- hosts: webservers
  vars:
    nginx_port: 8080              # Priority: 12 (play vars)
  roles:
    - nginx

# Result: nginx_port = 8080 ✅ (play vars override role defaults)
```

#### Example 2: group_vars vs host_vars

```yaml
# group_vars/webservers.yml
http_port: 80                     # Priority: 6

# host_vars/web1.example.com.yml
http_port: 8080                   # Priority: 9

# For web1: http_port = 8080 ✅ (host_vars override group_vars)
# For web2: http_port = 80 ✅ (only group_vars applies)
```

#### Example 3: Everything vs Extra Vars

```yaml
# group_vars/all.yml
app_version: "1.0.0"

# group_vars/webservers.yml
app_version: "2.0.0"

# host_vars/web1.yml
app_version: "3.0.0"

# play vars
vars:
  app_version: "4.0.0"

# Command line
ansible-playbook site.yml -e "app_version=5.0.0"

# Result for web1: app_version = "5.0.0" ✅
# Extra vars (-e) ALWAYS WIN. Period.
```

### Practical Tips for Variable Precedence

```
✅ Best Practices:
   • Use role defaults/ for "sensible defaults" users should override
   • Use group_vars/ for environment-specific settings
   • Use host_vars/ for host-specific overrides
   • Use extra vars (-e) for one-time overrides in CI/CD
   • Keep it simple — don't define the same variable in 5 places!

❌ Common Mistakes:
   • Putting variables in role vars/ when they should be in defaults/
   • Wondering why group_vars aren't working (check: is host_vars overriding?)
   • Forgetting that -e always wins
```

## 🔧 Special Variables (Magic Variables)

Ansible provides several built-in "magic" variables that are always available:

| Variable | Description |
|---|---|
| `inventory_hostname` | The name of the host as defined in inventory |
| `ansible_host` | The actual IP/hostname to connect to |
| `groups` | Dictionary of all groups and their hosts |
| `group_names` | List of groups the current host belongs to |
| `hostvars` | Dictionary of all host variables (access other hosts' vars) |
| `play_hosts` | List of hosts in the current play |
| `ansible_play_batch` | List of hosts in the current batch |
| `inventory_dir` | Directory of the inventory file |
| `playbook_dir` | Directory of the playbook |
| `role_path` | Path to the current role |
| `ansible_check_mode` | `true` if running in check mode (--check) |
| `ansible_version` | Dictionary with Ansible version info |

### Using `hostvars` to Access Other Hosts' Variables

```yaml
# Access the IP of the database server from a webserver task
- name: Configure app to connect to DB
  ansible.builtin.template:
    src: app.conf.j2
    dest: /opt/app/app.conf
  vars:
    db_ip: "{{ hostvars['db1.example.com']['ansible_default_ipv4']['address'] }}"
```

### Using `groups` to Loop Over Group Members

```yaml
- name: Generate load balancer config
  ansible.builtin.template:
    src: haproxy.cfg.j2
    dest: /etc/haproxy/haproxy.cfg
```

**haproxy.cfg.j2:**
```jinja2
backend web_servers
{% for host in groups['webservers'] %}
    server {{ host }} {{ hostvars[host]['ansible_default_ipv4']['address'] }}:80 check
{% endfor %}
```

Output:
```
backend web_servers
    server web1.example.com 192.168.1.10:80 check
    server web2.example.com 192.168.1.11:80 check
    server web3.example.com 192.168.1.12:80 check
```

### Using `group_names` for Conditional Logic

```yaml
- name: Install monitoring agent only on production servers
  ansible.builtin.apt:
    name: datadog-agent
    state: present
  when: "'production' in group_names"
```

## 📝 Variable Naming Best Practices

```yaml
# ✅ Good variable names
db_host: localhost
nginx_worker_connections: 1024
app_version: "2.1.0"
enable_ssl: true
max_upload_size_mb: 50

# ❌ Bad variable names
x: localhost                       # Too vague
host: localhost                    # Conflicts with Ansible keywords
my-variable: value                 # Hyphens not allowed in variable names
1st_server: web1                   # Can't start with a number
server name: web1                  # Spaces not allowed
```

Rules:
- Use **snake_case** (lowercase with underscores)
- Start with a letter
- Use descriptive names
- Prefix with the role/app name to avoid conflicts (e.g., `nginx_port`, not just `port`)
- Don't use Ansible reserved words (`hosts`, `tasks`, `vars`, `name`, etc.)

## 🎯 Key Takeaways

1. **Variables can be defined in many places** — inventory, group_vars, host_vars, playbooks, roles, command line
2. **group_vars/ and host_vars/ directories** are the recommended way to organize variables
3. **Facts** are auto-collected system info — OS, network, hardware, time
4. **Custom facts** can be placed in `/etc/ansible/facts.d/` on target machines
5. **Ansible Vault** encrypts sensitive data with AES-256 — commit encrypted files to Git safely
6. **Inline vault** (`encrypt_string`) is the best practice for individual secrets
7. **Extra vars (-e)** always have the highest priority and override everything
8. **Role defaults** have the lowest priority — designed to be easily overridden
9. **Prefix vault variables** with `vault_` for clarity
10. **Keep it simple** — don't define the same variable in too many places
