# 13.9 — Ansible Conditionals, Loops & Error Handling

> **Level:** Intermediate  
> **Goal:** Master how to add decision-making logic to your playbooks with conditionals (`when`), repeat tasks efficiently with loops, and handle errors gracefully so your automation doesn't break when something unexpected happens.

---

## 📌 Why Do We Need Logic in Playbooks?

So far, we've been writing tasks that run one after another — do this, then do that. But real-world automation needs **logic**:

- "Only install Apache on Debian servers" → **Conditional**
- "Install these 10 packages" → **Loop**
- "If this task fails, do this other thing instead" → **Error Handling**

Without these, your playbooks would be rigid, repetitive, and fragile. Let's make them smart!

---

## 🔀 Conditionals — The `when` Statement

The `when` statement lets you **skip or execute a task** based on a condition. It's like an `if` statement in programming.

### Basic Syntax

```yaml
- name: Do something only if condition is true
  ansible.builtin.some_module:
    some_param: value
  when: some_condition
```

> **Important:** `when` uses **raw Jinja2 expressions** — you do NOT need `{{ }}` around the condition. Just write the expression directly.

### Example 1: Based on Operating System

```yaml
- name: Install Apache on Debian/Ubuntu
  ansible.builtin.apt:
    name: apache2
    state: present
  when: ansible_os_family == "Debian"

- name: Install Apache on RHEL/CentOS
  ansible.builtin.yum:
    name: httpd
    state: present
  when: ansible_os_family == "RedHat"
```

How this works:
- Ansible checks the `ansible_os_family` fact
- If the server is Ubuntu → runs the `apt` task, skips the `yum` task
- If the server is CentOS → skips the `apt` task, runs the `yum` task

In the output, skipped tasks show as:
```
TASK [Install Apache on Debian/Ubuntu] ****
ok: [ubuntu-server]
skipping: [centos-server]

TASK [Install Apache on RHEL/CentOS] ****
skipping: [ubuntu-server]
ok: [centos-server]
```

### Example 2: Based on a Variable

```yaml
- name: Enable SSL only if requested
  ansible.builtin.template:
    src: ssl.conf.j2
    dest: /etc/nginx/conf.d/ssl.conf
  when: enable_ssl | default(false)
```

If `enable_ssl` is `true` → task runs. If `false` or undefined → task is skipped.

### Example 3: Based on a Registered Variable

```yaml
- name: Check if Docker is installed
  ansible.builtin.command: docker --version
  register: docker_check
  ignore_errors: yes
  changed_when: false    # Don't report this check as "changed"

- name: Install Docker if not present
  ansible.builtin.apt:
    name: docker.io
    state: present
  when: docker_check.rc != 0    # rc = return code. 0 = success, anything else = not found
```

### Comparison Operators

| Operator | Meaning | Example |
|---|---|---|
| `==` | Equal to | `when: ansible_os_family == "Debian"` |
| `!=` | Not equal to | `when: environment != "production"` |
| `>` | Greater than | `when: ansible_memtotal_mb > 4096` |
| `<` | Less than | `when: disk_usage < 80` |
| `>=` | Greater than or equal | `when: ansible_processor_vcpus >= 4` |
| `<=` | Less than or equal | `when: replicas <= 10` |

### Logical Operators (AND, OR, NOT)

#### AND — Both conditions must be true

```yaml
- name: Only on Ubuntu 22.04
  ansible.builtin.debug:
    msg: "This is Ubuntu 22.04!"
  when:
    - ansible_distribution == "Ubuntu"
    - ansible_distribution_version == "22.04"
```

When you list multiple conditions, they're combined with **AND** automatically. You can also write:

```yaml
  when: ansible_distribution == "Ubuntu" and ansible_distribution_version == "22.04"
```

#### OR — At least one condition must be true

```yaml
- name: Install on either Debian or Ubuntu
  ansible.builtin.apt:
    name: nginx
    state: present
  when: ansible_distribution == "Debian" or ansible_distribution == "Ubuntu"
```

#### NOT — Negate a condition

```yaml
- name: Do NOT run on production
  ansible.builtin.debug:
    msg: "This is NOT production"
  when: environment != "production"

# Or using 'not':
- name: Run if SSL is disabled
  ansible.builtin.debug:
    msg: "SSL is not enabled"
  when: not enable_ssl
```

#### Complex Conditions (Combining AND, OR, NOT)

```yaml
- name: Complex condition
  ansible.builtin.debug:
    msg: "Meets all criteria"
  when: >
    (ansible_os_family == "Debian" or ansible_os_family == "RedHat")
    and ansible_memtotal_mb >= 2048
    and environment == "production"
```

Use parentheses `()` to group conditions, just like in math or programming.

### Testing if a Variable is Defined

```yaml
# Check if a variable exists
- name: Use custom port if defined
  ansible.builtin.debug:
    msg: "Using port {{ custom_port }}"
  when: custom_port is defined

# Check if a variable is NOT defined
- name: Use default port
  ansible.builtin.debug:
    msg: "Using default port 80"
  when: custom_port is not defined
```

### Testing with `in` — Check if Value is in a List

```yaml
- name: Only run on specific distributions
  ansible.builtin.debug:
    msg: "Supported OS!"
  when: ansible_distribution in ["Ubuntu", "Debian", "CentOS"]

- name: Skip certain environments
  ansible.builtin.debug:
    msg: "Not a test environment"
  when: environment not in ["test", "ci", "sandbox"]
```

### Boolean Testing

```yaml
# Check if a boolean variable is true
- name: Enable debug logging
  ansible.builtin.template:
    src: debug.conf.j2
    dest: /etc/myapp/debug.conf
  when: debug_mode | bool       # Ensures string "true"/"false" is converted to boolean

# Check if a string is empty
- name: Skip if no email configured
  ansible.builtin.debug:
    msg: "Sending notification to {{ admin_email }}"
  when: admin_email | length > 0
```

### `when` with Blocks — Apply Condition to Multiple Tasks

Instead of repeating `when` on every task:

```yaml
# ❌ Repetitive
- name: Task 1
  apt: name=nginx state=present
  when: ansible_os_family == "Debian"

- name: Task 2
  template: src=nginx.conf.j2 dest=/etc/nginx/nginx.conf
  when: ansible_os_family == "Debian"

- name: Task 3
  service: name=nginx state=started
  when: ansible_os_family == "Debian"
```

Use a **block**:

```yaml
# ✅ Clean — condition applies to all tasks in the block
- name: Setup Nginx on Debian systems
  block:
    - name: Install nginx
      ansible.builtin.apt:
        name: nginx
        state: present

    - name: Deploy nginx config
      ansible.builtin.template:
        src: nginx.conf.j2
        dest: /etc/nginx/nginx.conf

    - name: Start nginx
      ansible.builtin.service:
        name: nginx
        state: started
        enabled: yes
  when: ansible_os_family == "Debian"
```

All three tasks run (or are all skipped) based on the single `when` condition on the block.

## 🔁 Loops — Repeating Tasks

Loops let you **run the same task multiple times** with different values. Instead of writing 10 separate tasks to install 10 packages, you write ONE task with a loop.

### Basic Loop with `loop`

```yaml
- name: Install multiple packages
  ansible.builtin.apt:
    name: "{{ item }}"
    state: present
  loop:
    - nginx
    - git
    - curl
    - vim
    - htop
    - unzip
```

**What happens:**
1. The task runs for `item = nginx`
2. The task runs for `item = git`
3. The task runs for `item = curl`
4. ...and so on for each item in the list

The magic variable `{{ item }}` holds the current value from the loop.

> **Pro Tip:** For the `apt` and `yum` modules, you can pass a list directly to `name:` — it's even faster because it runs a single package manager command:
> ```yaml
> - name: Install packages (optimized)
>   ansible.builtin.apt:
>     name:
>       - nginx
>       - git
>       - curl
>     state: present
> ```

### Loop Over a Variable

```yaml
vars:
  packages:
    - nginx
    - git
    - curl
    - vim

tasks:
  - name: Install all required packages
    ansible.builtin.apt:
      name: "{{ item }}"
      state: present
    loop: "{{ packages }}"
```

### Loop Over a Dictionary List

```yaml
- name: Create multiple users
  ansible.builtin.user:
    name: "{{ item.name }}"
    shell: "{{ item.shell }}"
    groups: "{{ item.groups }}"
    state: present
  loop:
    - { name: 'alice', shell: '/bin/bash', groups: 'developers' }
    - { name: 'bob', shell: '/bin/bash', groups: 'developers,docker' }
    - { name: 'charlie', shell: '/bin/zsh', groups: 'ops,docker,sudo' }
```

Each `item` is a dictionary, so you access fields with `item.name`, `item.shell`, etc.

### Loop Over a Dictionary (key-value pairs)

```yaml
vars:
  app_config:
    APP_NAME: myapp
    APP_PORT: "8080"
    APP_ENV: production
    LOG_LEVEL: info

tasks:
  - name: Set environment variables
    ansible.builtin.lineinfile:
      path: /etc/environment
      regexp: "^{{ item.key }}="
      line: "{{ item.key }}={{ item.value }}"
    loop: "{{ app_config | dict2items }}"
```

The `dict2items` filter converts a dictionary to a list of `{key: ..., value: ...}` items.

### Nested Loops with `loop` + `subelements`

```yaml
vars:
  users:
    - name: alice
      ssh_keys:
        - "ssh-rsa AAAA...alice-laptop"
        - "ssh-rsa AAAA...alice-desktop"
    - name: bob
      ssh_keys:
        - "ssh-rsa AAAA...bob-laptop"

tasks:
  - name: Add SSH keys for each user
    ansible.posix.authorized_key:
      user: "{{ item.0.name }}"
      key: "{{ item.1 }}"
      state: present
    loop: "{{ users | subelements('ssh_keys') }}"
```

This loops through each user AND each of their SSH keys. `item.0` is the user dictionary, `item.1` is the individual SSH key.

### Loop with Index — `loop_control` + `index_var`

```yaml
- name: Create numbered config files
  ansible.builtin.template:
    src: worker.conf.j2
    dest: "/etc/myapp/worker-{{ my_idx }}.conf"
  loop: "{{ range(1, 6) | list }}"   # [1, 2, 3, 4, 5]
  loop_control:
    index_var: my_idx
```

### Loop with Custom Label

By default, loop output shows the entire `item` dictionary, which can be messy. Use `label` to clean it up:

```yaml
- name: Create users with custom label
  ansible.builtin.user:
    name: "{{ item.name }}"
    shell: "{{ item.shell }}"
    groups: "{{ item.groups }}"
  loop:
    - { name: 'alice', shell: '/bin/bash', groups: 'dev', password: 'secret123' }
    - { name: 'bob', shell: '/bin/bash', groups: 'ops', password: 'secret456' }
  loop_control:
    label: "{{ item.name }}"    # Only show the name in output, not the whole dict (including password!)
```

Without label:
```
ok: [server] => (item={'name': 'alice', 'shell': '/bin/bash', 'groups': 'dev', 'password': 'secret123'})
```

With label:
```
ok: [server] => (item=alice)
```

Much cleaner! And no password leaked in the output.

### Loop with `pause` — Add Delay Between Iterations

```yaml
- name: Restart services one at a time with delay
  ansible.builtin.service:
    name: "{{ item }}"
    state: restarted
  loop:
    - app-worker-1
    - app-worker-2
    - app-worker-3
  loop_control:
    pause: 10    # Wait 10 seconds between each restart
```

### Conditional Loops — `when` + `loop` Together

```yaml
- name: Install packages only if not already present
  ansible.builtin.apt:
    name: "{{ item.name }}"
    state: present
  loop:
    - { name: 'nginx', install: true }
    - { name: 'apache2', install: false }
    - { name: 'git', install: true }
    - { name: 'svn', install: false }
  when: item.install    # Only install if install is true
```

### `until` Loop — Retry Until Success

This is different from `loop` — it **retries the same task** until a condition is met:

```yaml
- name: Wait for application to be ready
  ansible.builtin.uri:
    url: "http://localhost:8080/health"
    status_code: 200
  register: health_result
  until: health_result.status == 200
  retries: 30            # Try up to 30 times
  delay: 10              # Wait 10 seconds between retries
```

This keeps checking the health endpoint every 10 seconds, up to 30 times (5 minutes total), until it gets a 200 response.

### Loop Quick Reference

| I want to... | Use |
|---|---|
| Loop over a simple list | `loop: [a, b, c]` |
| Loop over a list variable | `loop: "{{ my_list }}"` |
| Loop over dict list | `loop: [{name: a, val: 1}, ...]` |
| Loop over key-value pairs | `loop: "{{ my_dict \| dict2items }}"` |
| Nested loop | `loop: "{{ list \| subelements('subkey') }}"` |
| Loop with index | `loop_control: index_var: idx` |
| Loop with delay | `loop_control: pause: 5` |
| Clean loop output | `loop_control: label: "{{ item.name }}"` |
| Retry until success | `until: condition` + `retries:` + `delay:` |
| Loop over a range | `loop: "{{ range(1, 10) \| list }}"` |
| Loop over files | `loop: "{{ query('fileglob', 'files/*.conf') }}"` |

## ⚠️ Error Handling — Dealing with Failures Gracefully

By default, when a task **fails**, Ansible **stops executing** on that host. That's usually what you want — if installing a package fails, there's no point continuing with configuration.

But sometimes you need more control. Let's look at all the error handling tools Ansible gives you.

### `ignore_errors` — Keep Going Even If It Fails

```yaml
- name: Try to stop old service (might not exist)
  ansible.builtin.service:
    name: old-app
    state: stopped
  ignore_errors: yes

- name: This task runs even if the above failed
  ansible.builtin.debug:
    msg: "Continuing with deployment..."
```

> **Use sparingly!** `ignore_errors: yes` means Ansible won't stop on failure, but the error is still logged. Only use this when you **expect** the task might fail and that's okay.

### `failed_when` — Custom Failure Conditions

Sometimes Ansible thinks a task succeeded, but YOU know it actually failed based on the output:

```yaml
- name: Run database migration
  ansible.builtin.command: /opt/app/migrate.sh
  register: migration_result
  failed_when: "'ERROR' in migration_result.stdout"
```

Or the reverse — Ansible thinks it failed, but you know it's actually fine:

```yaml
- name: Check if user exists
  ansible.builtin.command: id deploy_user
  register: user_check
  failed_when: user_check.rc not in [0, 1]
  # rc=0 means user exists, rc=1 means user doesn't exist
  # Both are fine! Only fail on other return codes (real errors)
```

### `changed_when` — Control When a Task Reports "Changed"

By default, `command` and `shell` modules always report "changed" (yellow in output). You can control this:

```yaml
- name: Check current app version
  ansible.builtin.command: /opt/app/version.sh
  register: app_version
  changed_when: false    # This is just a check, never report as changed

- name: Run deployment script
  ansible.builtin.command: /opt/app/deploy.sh
  register: deploy_result
  changed_when: "'Deployed successfully' in deploy_result.stdout"
  # Only report "changed" if the deploy actually did something
```

### `block`, `rescue`, `always` — Try/Catch/Finally for Ansible

This is Ansible's equivalent of try/catch/finally in programming. It's incredibly powerful:

```yaml
- name: Deploy Application with Error Recovery
  block:
    # ============ TRY — Normal operations ============
    - name: Pull latest code
      ansible.builtin.git:
        repo: https://github.com/myorg/myapp.git
        dest: /opt/myapp
        version: "{{ app_version }}"

    - name: Install dependencies
      ansible.builtin.command:
        cmd: pip install -r requirements.txt
        chdir: /opt/myapp

    - name: Run database migrations
      ansible.builtin.command:
        cmd: python manage.py migrate
        chdir: /opt/myapp

    - name: Restart application
      ansible.builtin.service:
        name: myapp
        state: restarted

  rescue:
    # ============ CATCH — If anything in block fails ============
    - name: Deployment failed! Rolling back...
      ansible.builtin.debug:
        msg: "Deployment failed. Initiating rollback."

    - name: Rollback to previous version
      ansible.builtin.command:
        cmd: git checkout {{ previous_version }}
        chdir: /opt/myapp

    - name: Restart application with old version
      ansible.builtin.service:
        name: myapp
        state: restarted

    - name: Send failure notification
      ansible.builtin.uri:
        url: "https://hooks.slack.com/services/YOUR/WEBHOOK"
        method: POST
        body_format: json
        body:
          text: "🚨 Deployment of {{ app_version }} FAILED on {{ inventory_hostname }}. Rolled back to {{ previous_version }}."

  always:
    # ============ FINALLY — Always runs, success or failure ============
    - name: Clean up temp files
      ansible.builtin.file:
        path: /tmp/deploy-artifacts
        state: absent

    - name: Log deployment result
      ansible.builtin.lineinfile:
        path: /var/log/deployments.log
        line: "{{ ansible_date_time.iso8601 }} - {{ app_version }} - {{ 'FAILED' if ansible_failed_task is defined else 'SUCCESS' }}"
        create: yes
```

**How it works:**

```
block:    → Run these tasks normally (the "try" part)
            ↓
            If ALL succeed → Skip rescue, run always
            If ANY fails  → Jump to rescue
            ↓
rescue:   → Run these tasks only if block failed (the "catch" part)
            ↓
always:   → ALWAYS run these tasks, no matter what happened
```

### Another Block/Rescue Example — Service Health Check

```yaml
- name: Deploy and verify
  block:
    - name: Deploy new version
      ansible.builtin.template:
        src: app.conf.j2
        dest: /etc/myapp/app.conf

    - name: Restart application
      ansible.builtin.service:
        name: myapp
        state: restarted

    - name: Wait for app to be healthy
      ansible.builtin.uri:
        url: http://localhost:8080/health
        status_code: 200
      register: health
      until: health.status == 200
      retries: 10
      delay: 5

  rescue:
    - name: Health check failed — reverting config
      ansible.builtin.copy:
        src: /etc/myapp/app.conf.bak
        dest: /etc/myapp/app.conf
        remote_src: yes

    - name: Restart with old config
      ansible.builtin.service:
        name: myapp
        state: restarted
```

### `any_errors_fatal` — Stop EVERYTHING If One Host Fails

By default, if a task fails on one host, Ansible stops on THAT host but continues on others. Sometimes you want to stop ALL hosts:

```yaml
---
- name: Critical deployment
  hosts: webservers
  any_errors_fatal: true    # If ANY host fails, stop ALL hosts immediately

  tasks:
    - name: Run pre-flight checks
      ansible.builtin.command: /opt/scripts/preflight.sh

    - name: Deploy application
      ansible.builtin.command: /opt/scripts/deploy.sh
```

Use this for deployments where you need **all-or-nothing** consistency.

### `max_fail_percentage` — Allow Some Failures

```yaml
---
- name: Rolling update
  hosts: webservers
  max_fail_percentage: 30    # Stop only if more than 30% of hosts fail
  serial: 5                  # Process 5 hosts at a time

  tasks:
    - name: Deploy application
      ansible.builtin.command: /opt/scripts/deploy.sh
```

If you have 20 servers and 7+ fail (>30%), Ansible stops. If only 6 fail, it continues. Useful for rolling deployments where a few failures are acceptable.

### `assert` Module — Validate Conditions

The `assert` module lets you verify that conditions are true before proceeding. It's like putting up guardrails:

```yaml
- name: Pre-flight checks
  ansible.builtin.assert:
    that:
      - ansible_memtotal_mb >= 2048
      - ansible_processor_vcpus >= 2
      - ansible_distribution == "Ubuntu"
    fail_msg: "Server does not meet minimum requirements!"
    success_msg: "All pre-flight checks passed ✅"
```

If any condition is false, the task fails with your custom error message.

### `fail` Module — Intentionally Fail

```yaml
- name: Fail if environment is not set
  ansible.builtin.fail:
    msg: "ERROR: 'environment' variable must be set to 'dev', 'staging', or 'production'"
  when: environment is not defined or environment not in ['dev', 'staging', 'production']
```

Use `fail` to create clear, helpful error messages when something is wrong.

## 🧩 Putting It All Together — Real-World Examples

### Example 1: Complete Application Deployment

```yaml
---
- name: Deploy Web Application
  hosts: webservers
  become: yes
  vars:
    app_name: myapp
    app_version: "{{ version | default('latest') }}"
    app_port: 8080

  pre_tasks:
    # ── Validate inputs ──
    - name: Verify required variables
      ansible.builtin.assert:
        that:
          - app_name is defined
          - app_port | int > 0
        fail_msg: "Required variables are missing!"

  tasks:
    # ── Install dependencies based on OS ──
    - name: Install dependencies (Debian)
      ansible.builtin.apt:
        name: "{{ item }}"
        state: present
        update_cache: yes
      loop:
        - python3
        - python3-pip
        - nginx
        - git
      when: ansible_os_family == "Debian"

    - name: Install dependencies (RedHat)
      ansible.builtin.yum:
        name: "{{ item }}"
        state: present
      loop:
        - python3
        - python3-pip
        - nginx
        - git
      when: ansible_os_family == "RedHat"

    # ── Deploy with error recovery ──
    - name: Deploy application
      block:
        - name: Clone/update repository
          ansible.builtin.git:
            repo: "https://github.com/myorg/{{ app_name }}.git"
            dest: "/opt/{{ app_name }}"
            version: "{{ app_version }}"
          register: git_result

        - name: Install Python dependencies
          ansible.builtin.pip:
            requirements: "/opt/{{ app_name }}/requirements.txt"
            virtualenv: "/opt/{{ app_name }}/venv"
          when: git_result.changed

        - name: Deploy configuration
          ansible.builtin.template:
            src: app.conf.j2
            dest: "/etc/{{ app_name }}/config.yml"
            owner: "{{ app_name }}"
            mode: '0640'
          notify: Restart application

        - name: Ensure service is running
          ansible.builtin.service:
            name: "{{ app_name }}"
            state: started
            enabled: yes

        - name: Health check
          ansible.builtin.uri:
            url: "http://localhost:{{ app_port }}/health"
            status_code: 200
          register: health
          until: health.status == 200
          retries: 12
          delay: 5

      rescue:
        - name: Deployment failed!
          ansible.builtin.debug:
            msg: "Deployment of {{ app_name }} v{{ app_version }} FAILED on {{ inventory_hostname }}"

        - name: Rollback to previous version
          ansible.builtin.command:
            cmd: git checkout HEAD~1
            chdir: "/opt/{{ app_name }}"
          ignore_errors: yes

      always:
        - name: Log deployment attempt
          ansible.builtin.lineinfile:
            path: /var/log/deployment.log
            line: "{{ ansible_date_time.iso8601 }} | {{ app_name }} | {{ app_version }} | {{ inventory_hostname }} | {{ 'FAIL' if ansible_failed_task is defined else 'OK' }}"
            create: yes

  handlers:
    - name: Restart application
      ansible.builtin.service:
        name: "{{ app_name }}"
        state: restarted
```

### Example 2: Multi-OS User Setup with Loops and Conditionals

```yaml
---
- name: Setup Users Across All Servers
  hosts: all
  become: yes

  vars:
    users:
      - name: alice
        groups: ['developers', 'docker']
        shell: /bin/bash
        state: present
        admin: true
      - name: bob
        groups: ['developers']
        shell: /bin/bash
        state: present
        admin: false
      - name: charlie
        groups: ['ops', 'docker', 'sudo']
        shell: /bin/zsh
        state: present
        admin: true
      - name: olduser
        groups: []
        shell: /bin/bash
        state: absent        # Remove this user!
        admin: false

  tasks:
    - name: Ensure required groups exist
      ansible.builtin.group:
        name: "{{ item }}"
        state: present
      loop:
        - developers
        - ops
        - docker

    - name: Create or remove users
      ansible.builtin.user:
        name: "{{ item.name }}"
        shell: "{{ item.shell }}"
        groups: "{{ item.groups | join(',') }}"
        state: "{{ item.state }}"
        remove: "{{ 'yes' if item.state == 'absent' else 'no' }}"
        append: yes
      loop: "{{ users }}"
      loop_control:
        label: "{{ item.name }} ({{ item.state }})"

    - name: Add admin users to sudoers
      ansible.builtin.lineinfile:
        path: /etc/sudoers.d/{{ item.name }}
        line: "{{ item.name }} ALL=(ALL) NOPASSWD:ALL"
        create: yes
        mode: '0440'
        validate: '/usr/sbin/visudo -cf %s'
      loop: "{{ users | selectattr('admin') | selectattr('state', 'equalto', 'present') | list }}"
      loop_control:
        label: "{{ item.name }}"
```

Notice the filter chain: `selectattr('admin')` picks users where `admin` is true, `selectattr('state', 'equalto', 'present')` then filters to only active users.

### Example 3: Container Deployment with Health Checks

```yaml
---
- name: Deploy Docker Services
  hosts: docker_servers
  become: yes

  vars:
    services:
      - name: postgres
        image: postgres:16
        port: "5432:5432"
        env:
          POSTGRES_DB: appdb
          POSTGRES_USER: admin
          POSTGRES_PASSWORD: "{{ vault_db_password }}"
        healthcheck: "pg_isready -U admin"
        critical: true

      - name: redis
        image: redis:7-alpine
        port: "6379:6379"
        env: {}
        healthcheck: "redis-cli ping"
        critical: true

      - name: app
        image: "myapp:{{ app_version | default('latest') }}"
        port: "8080:8080"
        env:
          DATABASE_URL: "postgresql://admin:{{ vault_db_password }}@postgres:5432/appdb"
          REDIS_URL: "redis://redis:6379"
        healthcheck: "curl -f http://localhost:8080/health"
        critical: true

      - name: metrics-exporter
        image: prom/node-exporter:latest
        port: "9100:9100"
        env: {}
        healthcheck: "wget -q --spider http://localhost:9100/metrics"
        critical: false        # Non-critical — failure is acceptable

  tasks:
    - name: Deploy each service
      block:
        - name: Run container — {{ item.name }}
          community.docker.docker_container:
            name: "{{ item.name }}"
            image: "{{ item.image }}"
            state: started
            restart_policy: unless-stopped
            ports:
              - "{{ item.port }}"
            env: "{{ item.env }}"
          loop: "{{ services }}"
          loop_control:
            label: "{{ item.name }}"

        - name: Wait for critical services to be healthy
          ansible.builtin.command: "docker exec {{ item.name }} {{ item.healthcheck }}"
          register: health
          until: health.rc == 0
          retries: 15
          delay: 5
          loop: "{{ services | selectattr('critical') | list }}"
          loop_control:
            label: "{{ item.name }}"

      rescue:
        - name: Critical service failed — stopping all containers
          community.docker.docker_container:
            name: "{{ item.name }}"
            state: stopped
          loop: "{{ services }}"
          loop_control:
            label: "{{ item.name }}"
          ignore_errors: yes

        - name: Fail with clear message
          ansible.builtin.fail:
            msg: "Deployment failed! One or more critical services did not pass health checks."
```

## 📋 Quick Reference Cheat Sheet

### Conditionals

```yaml
# Simple condition
when: ansible_os_family == "Debian"

# Multiple conditions (AND)
when:
  - condition_1
  - condition_2

# OR condition
when: condition_1 or condition_2

# NOT
when: not my_variable

# Variable defined/undefined
when: my_var is defined
when: my_var is not defined

# Check if value is in list
when: my_var in ['a', 'b', 'c']

# Based on registered result
when: result.rc == 0
when: result.stdout | length > 0
when: "'error' in result.stderr"
```

### Loops

```yaml
# Simple list
loop: [a, b, c]

# Variable list
loop: "{{ my_list }}"

# Dict list
loop:
  - { name: a, val: 1 }
  - { name: b, val: 2 }

# Dictionary key-value
loop: "{{ my_dict | dict2items }}"

# Range
loop: "{{ range(1, 11) | list }}"

# With index
loop_control:
  index_var: idx

# With custom label
loop_control:
  label: "{{ item.name }}"

# With pause
loop_control:
  pause: 5

# Retry until
until: result.status == 200
retries: 10
delay: 5
```

### Error Handling

```yaml
# Ignore errors
ignore_errors: yes

# Custom failure condition
failed_when: "'ERROR' in result.stdout"

# Custom changed condition
changed_when: false

# Block/rescue/always (try/catch/finally)
block:
  - task_1
  - task_2
rescue:
  - recovery_task
always:
  - cleanup_task

# Stop all hosts on any failure
any_errors_fatal: true

# Allow some failures
max_fail_percentage: 25

# Assert conditions
assert:
  that:
    - condition_1
    - condition_2
  fail_msg: "Custom error message"

# Intentional failure
fail:
  msg: "Something is wrong!"
```

## 🎯 Key Takeaways

1. **`when` is your if-statement** — Use it to make tasks conditional based on facts, variables, or registered results
2. **Don't use `{{ }}` in `when`** — It takes raw Jinja2 expressions directly
3. **Use `block`** to apply the same condition to multiple tasks at once
4. **`loop` replaces the old `with_*` syntax** — Use `loop` for all new playbooks
5. **`loop_control`** gives you index, labels, and pause between iterations
6. **`until` + `retries` + `delay`** = retry until a condition is met (perfect for health checks)
7. **`block`/`rescue`/`always`** = try/catch/finally — essential for deployment rollbacks
8. **`ignore_errors`** should be used sparingly — don't hide real problems!
9. **`failed_when` and `changed_when`** give you precise control over task status
10. **`assert`** is your pre-flight check — validate conditions before doing dangerous operations
