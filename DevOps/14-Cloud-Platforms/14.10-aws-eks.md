## рҹӨ” What Problem Does EKS Solve?

Imagine you've learned Kubernetes вҖ” you know Pods, Deployments, Services, etc. Now you want to run Kubernetes **in production** for a real company.

Here's the problem:

```
Running Kubernetes yourself (self-managed):
    в”ңв”Җв”Җ You need to set up the control plane (API server, etcd, scheduler, controller manager)
    в”ңв”Җв”Җ You need to manage etcd backups
    в”ңв”Җв”Җ You need to handle upgrades (very painful!)
    в”ңв”Җв”Җ You need to patch security vulnerabilities
    в”ңв”Җв”Җ You need to make sure etcd doesn't crash (if it does, your whole cluster is dead)
    в””в”Җв”Җ You need to handle high availability across multiple zones
    
    Result: You spend 60% of your time managing Kubernetes instead of running apps on it рҹҳ«
```

**EKS says:** "Hey, let me handle all that control plane stuff for you. You just focus on deploying your apps."

---

## вң… What is AWS EKS?

**Amazon Elastic Kubernetes Service (EKS)** is a managed Kubernetes service where:

| AWS Manages (Control Plane) | You Manage (Data Plane) |
|---|---|
| API Server | Worker Nodes (EC2 instances or Fargate) |
| etcd (data store) | Your application Pods |
| Scheduler | Networking configuration |
| Controller Manager | Storage configuration |
| Security patches & upgrades | Deploying your workloads |
| High Availability (multi-AZ) | Monitoring your apps |

### The Simple Version:

```
EKS = Kubernetes + AWS manages the hard parts (control plane)
You just bring your apps and deploy them using standard kubectl commands
```

### Key Architectural Points:

1. **Control Plane** вҖ” Fully managed by AWS, runs across **3 Availability Zones** for HA
2. **Data Plane** вҖ” Your worker nodes where Pods actually run (you pick EC2 or Fargate)
3. **API Endpoint** вҖ” EKS gives you an API server endpoint that you connect to with `kubectl`
4. **etcd** вҖ” Managed and backed up by AWS automatically

## рҹӣ пёҸ Ways to Run Worker Nodes in EKS

EKS gives you **three options** for running your worker nodes:

### Option 1: Managed Node Groups (Most Common)

```
You tell EKS: "I want 3 nodes of type t3.medium"
EKS creates EC2 instances, installs kubelet, joins them to the cluster automatically
EKS also handles rolling upgrades of the nodes for you
```

**Best for:** Most production workloads

### Option 2: Self-Managed Node Groups

```
You create your own EC2 instances
You install kubelet and join them to the cluster yourself
Full control, but more work
```

**Best for:** When you need custom AMIs, GPU instances, or specific configurations

### Option 3: AWS Fargate

```
No EC2 instances at all!
Each Pod runs in its own isolated micro-VM
You don't manage any servers вҖ” truly serverless Kubernetes
AWS charges per Pod (vCPU + memory per second)
```

**Best for:** Small workloads, batch jobs, when you don't want to manage nodes at all

### Comparison Table:

| Feature | Managed Node Groups | Self-Managed | Fargate |
|---|---|---|---|
| Node management | AWS handles | You handle | No nodes! |
| Upgrades | Automated rolling | Manual | Automatic |
| Scaling | Cluster Autoscaler / Karpenter | Cluster Autoscaler | Automatic |
| DaemonSets | вң… Yes | вң… Yes | вқҢ No |
| Cost | Pay for EC2 | Pay for EC2 | Pay per Pod |
| Best for | Production | Custom needs | Serverless/batch |

## рҹҡҖ Setting Up an EKS Cluster вҖ” Step by Step

### Prerequisites

Before you start, you need:

1. **AWS CLI** installed and configured
2. **kubectl** installed
3. **eksctl** installed (the official CLI tool for EKS)

### Install eksctl

```bash
# On Linux
curl --silent --location "https://github.com/weaveworks/eksctl/releases/latest/download/eksctl_$(uname -s)_amd64.tar.gz" | tar xz -C /tmp
sudo mv /tmp/eksctl /usr/local/bin

# On Mac
brew tap weaveworks/tap
brew install weaveworks/tap/eksctl

# Verify
eksctl version
```

### Method 1: Create Cluster with eksctl (Easiest)

```bash
# Create a basic EKS cluster with managed node group
eksctl create cluster \
  --name my-first-cluster \
  --region us-east-1 \
  --version 1.28 \
  --nodegroup-name my-nodes \
  --node-type t3.medium \
  --nodes 3 \
  --nodes-min 1 \
  --nodes-max 5 \
  --managed
```

**What this command does:**
- Creates a VPC with public and private subnets across 2 AZs
- Creates the EKS control plane
- Creates a managed node group with 3 `t3.medium` instances
- Configures autoscaling (min 1, max 5)
- Updates your `~/.kube/config` automatically

```
вҸі This takes about 15-20 minutes to complete
```

### Method 2: Create Cluster with eksctl Config File (Production Way)

Instead of a long command, use a YAML config file:

```yaml
# cluster-config.yaml
apiVersion: eksctl.io/v1alpha5
kind: ClusterConfig

metadata:
  name: production-cluster
  region: us-east-1
  version: "1.28"

# VPC configuration
vpc:
  cidr: 10.0.0.0/16
  nat:
    gateway: Single  # or HighlyAvailable for production

# Managed Node Groups
managedNodeGroups:
  - name: general-workers
    instanceType: t3.large
    minSize: 2
    maxSize: 10
    desiredCapacity: 3
    volumeSize: 50
    volumeType: gp3
    privateNetworking: true  # Nodes in private subnets
    labels:
      role: general
    tags:
      Environment: production
    iam:
      withAddonPolicies:
        albIngress: true      # For AWS Load Balancer Controller
        cloudWatch: true      # For CloudWatch logging
        ebs: true             # For EBS CSI driver

  - name: spot-workers
    instanceTypes:
      - t3.large
      - t3.xlarge
      - m5.large
    spot: true               # Use spot instances for cost savings
    minSize: 0
    maxSize: 20
    desiredCapacity: 2
    labels:
      role: spot-worker
    taints:
      - key: spot
        value: "true"
        effect: NoSchedule

# Fargate Profiles (optional)
fargateProfiles:
  - name: fp-default
    selectors:
      - namespace: fargate-workloads
        labels:
          compute: fargate

# CloudWatch Logging
cloudWatch:
  clusterLogging:
    enableTypes:
      - api
      - audit
      - authenticator
      - controllerManager
      - scheduler
```

```bash
# Create the cluster from config file
eksctl create cluster -f cluster-config.yaml
```

## рҹ“Ұ Deploying Your First App on EKS

Let's deploy a simple nginx app:

```yaml
# nginx-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-app
  labels:
    app: nginx
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.25
        ports:
        - containerPort: 80
        resources:
          requests:
            cpu: 100m
            memory: 128Mi
          limits:
            cpu: 250m
            memory: 256Mi
---
apiVersion: v1
kind: Service
metadata:
  name: nginx-service
  annotations:
    # This annotation makes EKS create an AWS Network Load Balancer
    service.beta.kubernetes.io/aws-load-balancer-type: nlb
spec:
  type: LoadBalancer
  selector:
    app: nginx
  ports:
  - port: 80
    targetPort: 80
```

```bash
# Deploy it
kubectl apply -f nginx-deployment.yaml

# Check the deployment
kubectl get deployments
kubectl get pods
kubectl get svc

# Wait for the LoadBalancer to get an external IP/DNS
kubectl get svc nginx-service -w

# nginx-service   LoadBalancer   172.20.45.123   a1b2c3-1234.us-east-1.elb.amazonaws.com       80:31234/TCP

# Now you can access your app at the EXTERNAL-IP URL!
curl http://a1b2c3-1234.us-east-1.elb.amazonaws.com
```

## рҹҢҗ EKS Networking Deep Dive

### VPC CNI Plugin (Amazon VPC CNI)

EKS uses a special networking plugin called **Amazon VPC CNI** that gives every Pod a **real IP address from your VPC**:

```
Traditional Kubernetes:
  Pod gets a virtual overlay IP (e.g., 10.244.1.5)
  Needs NAT/tunneling to talk to other AWS services

EKS with VPC CNI:
  Pod gets a REAL VPC IP (e.g., 10.0.1.45)
  Can directly talk to any AWS service (RDS, ElastiCache, etc.)
  No overlay network overhead вҖ” better performance!
```

### How VPC CNI Works:

```
EC2 Instance (t3.large)
в”ңв”Җв”Җ Primary ENI (Elastic Network Interface)
в”Ӯ   в”ңв”Җв”Җ Primary IP: 10.0.1.10 (node IP)
в”Ӯ   в”ңв”Җв”Җ Secondary IP: 10.0.1.11 вҶ’ assigned to Pod A
в”Ӯ   в”ңв”Җв”Җ Secondary IP: 10.0.1.12 вҶ’ assigned to Pod B
в”Ӯ   в””в”Җв”Җ Secondary IP: 10.0.1.13 вҶ’ assigned to Pod C
в”Ӯ
в”ңв”Җв”Җ Secondary ENI
в”Ӯ   в”ңв”Җв”Җ Primary IP: 10.0.1.20
в”Ӯ   в”ңв”Җв”Җ Secondary IP: 10.0.1.21 вҶ’ assigned to Pod D
в”Ӯ   в””в”Җв”Җ Secondary IP: 10.0.1.22 вҶ’ assigned to Pod E
в”Ӯ
в””в”Җв”Җ Each ENI can hold a limited number of IPs
    (depends on instance type)
```

**Important:** The number of Pods per node is LIMITED by the instance type:

| Instance Type | Max ENIs | IPs per ENI | Max Pods |
|---|---|---|---|
| t3.micro | 2 | 2 | 4 |
| t3.medium | 3 | 6 | 17 |
| t3.large | 3 | 12 | 35 |
| m5.large | 3 | 10 | 29 |
| m5.xlarge | 4 | 15 | 58 |

> **Pro Tip:** This is a common gotcha! If your nodes are "full" but have plenty of CPU/memory, you might have hit the **Pod IP limit**.

## рҹ’ҫ EKS Storage вҖ” EBS CSI Driver

To use persistent storage in EKS, you need the **Amazon EBS CSI Driver**:

```bash
# Install the EBS CSI Driver add-on
eksctl create addon \
  --name aws-ebs-csi-driver \
  --cluster my-first-cluster \
  --region us-east-1 \
  --force
```

Now create a StorageClass and use it:

```yaml
# storage-class.yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: ebs-gp3
provisioner: ebs.csi.aws.com
parameters:
  type: gp3
  fsType: ext4
  encrypted: "true"
reclaimPolicy: Delete
volumeBindingMode: WaitForFirstConsumer
allowVolumeExpansion: true
---
# pvc.yaml вҖ” Requesting storage
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-app-data
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: ebs-gp3
  resources:
    requests:
      storage: 20Gi
---
# pod-with-storage.yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-with-storage
spec:
  containers:
  - name: app
    image: nginx
    volumeMounts:
    - name: data
      mountPath: /data
  volumes:
  - name: data
    persistentVolumeClaim:
      claimName: my-app-data
```

```bash
kubectl apply -f storage-class.yaml
kubectl apply -f pvc.yaml
kubectl apply -f pod-with-storage.yaml

# Check the PVC
kubectl get pvc
# Output:
# NAME          STATUS   VOLUME         CAPACITY   ACCESS MODES   STORAGECLASS
# my-app-data   Bound    pvc-abc123..   20Gi       RWO            ebs-gp3
```

## рҹ”җ EKS IAM & Security

### IAM Roles for Service Accounts (IRSA)

This is the **recommended way** to give your Pods access to AWS services:

```
Old way (bad): Attach IAM role to the EC2 node вҶ’ ALL Pods on that node get the same permissions рҹҳұ
New way (IRSA): Each Pod gets its own IAM role through its ServiceAccount вҶ’ least privilege вң…
```

**How to set up IRSA:**

```bash
# Step 1: Create an IAM OIDC provider for your cluster
eksctl utils associate-iam-oidc-provider \
  --cluster my-first-cluster \
  --region us-east-1 \
  --approve

# Step 2: Create an IAM role and link it to a Kubernetes ServiceAccount
eksctl create iamserviceaccount \
  --name s3-reader \
  --namespace default \
  --cluster my-first-cluster \
  --region us-east-1 \
  --attach-policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess \
  --approve
```

```yaml
# Now use the ServiceAccount in your Pod
apiVersion: v1
kind: Pod
metadata:
  name: s3-reader-pod
spec:
  serviceAccountName: s3-reader    # вҶҗ This SA has S3 read access!
  containers:
  - name: app
    image: amazon/aws-cli
    command: ["aws", "s3", "ls"]   # This will work because of IRSA
```

### EKS Security Best Practices:

```
1. вң… Use IRSA (not node-level IAM roles)
2. вң… Enable control plane logging (API, audit, authenticator)
3. вң… Use private endpoint access (disable public if possible)
4. вң… Use Network Policies (Calico or Cilium)
5. вң… Enable encryption for secrets (using AWS KMS)
6. вң… Regularly update the cluster and node versions
7. вң… Use Pod Security Standards (Restricted)
8. вң… Scan container images for vulnerabilities
```

## рҹ“Ҳ EKS Autoscaling вҖ” Cluster Autoscaler vs Karpenter

### Cluster Autoscaler (Traditional)

```bash
# Install Cluster Autoscaler
kubectl apply -f https://raw.githubusercontent.com/kubernetes/autoscaler/master/cluster-autoscaler/cloudprovider/aws/examples/cluster-autoscaler-autodiscover.yaml

# Configure it
kubectl -n kube-system annotate deployment cluster-autoscaler \
  cluster-autoscaler.kubernetes.io/safe-to-evict="false"
```

**How it works:**

```
Pod can't be scheduled (not enough resources)
        вҶ“
Cluster Autoscaler detects "pending" Pods
        вҶ“
Adds more nodes to the node group
        вҶ“
Pod gets scheduled on the new node
        вҶ“
When nodes are underutilized, it removes them
```

### Karpenter (Modern вҖ” AWS Recommended)

Karpenter is a **next-generation** node autoscaler built by AWS:

```
Cluster Autoscaler limitations:
  - Works with Node Groups (predefined instance types)
  - Slow to scale (minutes)
  - Limited flexibility

Karpenter advantages:
  - No node groups needed!
  - Picks the best instance type automatically based on Pod requirements
  - Scales in seconds, not minutes
  - Supports spot instances intelligently
  - Consolidates workloads to reduce costs
```

```yaml
# Karpenter NodePool (replaces Node Groups concept)
apiVersion: karpenter.sh/v1beta1
kind: NodePool
metadata:
  name: default
spec:
  template:
    spec:
      requirements:
        - key: "karpenter.k8s.aws/instance-category"
          operator: In
          values: ["c", "m", "r"]    # Compute, Memory, or General purpose
        - key: "karpenter.k8s.aws/instance-size"
          operator: In
          values: ["medium", "large", "xlarge"]
        - key: "karpenter.sh/capacity-type"
          operator: In
          values: ["on-demand", "spot"]  # Use spot when possible!
        - key: "topology.kubernetes.io/zone"
          operator: In
          values: ["us-east-1a", "us-east-1b", "us-east-1c"]
      nodeClassRef:
        name: default
  limits:
    cpu: "1000"        # Total CPU limit across all nodes
    memory: 1000Gi     # Total memory limit
  disruption:
    consolidationPolicy: WhenUnderutilized   # Remove underused nodes
    expireAfter: 720h  # Replace nodes every 30 days (for security patches)
```

```
EKS Pricing Breakdown:

в”ңв”Җв”Җ Worker Nodes (Fargate): Per Pod pricing
```

### Cost Optimization Tips:

## рҹ—‘пёҸ Cleaning Up вҖ” Deleting an EKS Cluster

```bash
# Delete all Kubernetes resources first (important to avoid orphaned AWS resources!)
kubectl delete svc --all
kubectl delete ingress --all
kubectl delete pvc --all

# Delete the cluster
eksctl delete cluster --name my-first-cluster --region us-east-1

# вҡ пёҸ This takes about 10-15 minutes
# It will delete: VPC, subnets, node groups, security groups, IAM roles, etc.
```

> **Warning:** Always delete Kubernetes Services of type `LoadBalancer` and Ingresses BEFORE deleting the cluster. Otherwise, the ALBs/NLBs will become **orphaned** and keep costing you money!

## рҹҶҡ EKS vs ECS vs Fargate вҖ” When to Use What?

| Feature | EKS | ECS | Fargate |
|---|---|---|---|
| What it is | Managed Kubernetes | AWS-native container orchestrator | Serverless compute engine |
| Learning curve | Steep (need to know K8s) | Moderate | Easy |
| Portability | High (standard K8s) | Low (AWS-only) | Low (AWS-only) |
| Community | Huge (CNCF ecosystem) | Smaller | N/A |
| Control | Full K8s control | AWS-managed | Least control |
| Use with | EKS on EC2 or Fargate | ECS on EC2 or Fargate | ECS or EKS |
| Best for | Large/complex apps, multi-cloud | Simple AWS-only apps | Serverless containers |

> **Rule of thumb:** If you already know Kubernetes and want portability вҶ’ **EKS**. If you're all-in on AWS and want simplicity вҶ’ **ECS**. If you don't want to manage nodes at all вҶ’ **Fargate** (with either EKS or ECS).
