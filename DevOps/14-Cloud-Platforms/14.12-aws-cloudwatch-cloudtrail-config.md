# 14.12 — AWS CloudWatch, CloudTrail & AWS Config

> **One Line Answer:** **CloudWatch** monitors your resources and apps (metrics, logs, alarms), **CloudTrail** records every API call made in your AWS account (who did what, when), and **AWS Config** tracks the configuration of your resources over time (is your infrastructure compliant?).

---

## 🤔 Why Do You Need All Three?

Think of it this way — you're running a company's entire infrastructure on AWS. You need answers to three very different questions:

```
Question 1: "Is my application healthy RIGHT NOW?"
  → CloudWatch (metrics, logs, alarms, dashboards)

Question 2: "WHO made changes to my AWS account and WHAT did they do?"
  → CloudTrail (audit trail of all API calls)

Question 3: "Are my resources configured correctly? Are they compliant with our rules?"
  → AWS Config (configuration history, compliance rules)
```

# ☁️ Part 1: Amazon CloudWatch

## ✅ What is CloudWatch?

CloudWatch is AWS's **monitoring and observability service**. It collects and tracks:

| What | Example |
|---|---|
| **Metrics** | CPU usage, memory, disk I/O, request count, latency |
| **Logs** | Application logs, server logs, Lambda logs |
| **Alarms** | Alert when CPU > 80%, or error rate > 5% |
| **Dashboards** | Visual graphs and charts of your metrics |
| **Events** | React to changes (e.g., EC2 instance stopped → trigger Lambda) |

### The Simple Version:

```
CloudWatch = Your AWS monitoring dashboard
             + Log aggregation
             + Alerting system
             All in one service
```

---

## 📊 CloudWatch Metrics

### What Are Metrics?

Metrics are **numerical data points** about your resources over time:

```
Example Metrics:
  EC2:
    - CPUUtilization: 45.2% (right now)
    - NetworkIn: 500 MB (last hour)
    - DiskReadOps: 1,234 (last 5 minutes)
  
  RDS:
    - DatabaseConnections: 42
    - ReadLatency: 0.5ms
    - FreeStorageSpace: 15 GB
  
  ALB:
    - RequestCount: 10,432
    - TargetResponseTime: 0.120s
    - HTTPCode_Target_5XX_Count: 3
  
  Lambda:
    - Invocations: 5,000
    - Duration: 200ms (average)
    - Errors: 12
    - Throttles: 0
```

### Default Metrics vs Custom Metrics

```
Default Metrics (FREE — AWS sends them automatically):
  ├── EC2: CPU, Network, Disk, Status Checks
  ├── RDS: Connections, Latency, Storage
  ├── ALB: Request Count, Response Time, Error Codes
  ├── Lambda: Invocations, Duration, Errors
  └── Sent every 5 minutes (1-minute with detailed monitoring)

Custom Metrics (YOU send them — costs money):
  ├── Application-specific metrics
  ├── Memory usage (EC2 doesn't send this by default!)
  ├── Disk space usage
  ├── Active users count
  └── Business metrics (orders per minute, revenue, etc.)
```

> **Important gotcha:** EC2 does NOT send **memory** or **disk usage** metrics by default! You need the CloudWatch Agent for that.

### Sending Custom Metrics

```bash
# Using AWS CLI — Send a custom metric
aws cloudwatch put-metric-data \
  --namespace "MyApp" \
  --metric-name "ActiveUsers" \
  --value 42 \

  --dimensions Environment=Production,Service=WebApp
```

## 📝 CloudWatch Logs

### What Are CloudWatch Logs?

CloudWatch Logs is a **centralized log management service**:

```
Structure:
  Log Group: /aws/lambda/my-function        ← Container for log streams
    ├── Log Stream: 2024/01/15/[$LATEST]abc123  ← One stream per source
    │   ├── Log Event: "START RequestId: xyz..."
    │   ├── Log Event: "Processing order 1234..."
    │   └── Log Event: "END RequestId: xyz..."
    │
    └── Log Stream: 2024/01/15/[$LATEST]def456
        ├── Log Event: "START RequestId: uvw..."
        └── Log Event: "ERROR: Database timeout..."
```

### Where Do Logs Come From?

```
Automatic (no setup needed):
  ├── Lambda functions → /aws/lambda/<function-name>
  ├── API Gateway → /aws/apigateway/<api-id>
  ├── ECS/Fargate → /ecs/<task>
  └── CloudTrail → /aws/cloudtrail

Manual (you set up):
  ├── EC2 instances → CloudWatch Agent
  ├── EKS → Fluent Bit or CloudWatch Agent
  ├── On-premises servers → CloudWatch Agent
  └── Custom applications → AWS SDK
```

### Querying Logs with CloudWatch Logs Insights

CloudWatch Logs Insights lets you **search and analyze logs** using a query language:

```sql
-- Find all ERROR logs in the last hour
fields @timestamp, @message
| filter @message like /ERROR/
| sort @timestamp desc
| limit 50
```

```sql
-- Count errors per Lambda function in the last 24 hours
fields @timestamp, @message
| filter @message like /ERROR/
| stats count(*) as error_count by @logStream
| sort error_count desc
```

```sql
-- Find the slowest Lambda invocations
fields @timestamp, @duration, @memorySize, @maxMemoryUsed
| filter @type = "REPORT"
| sort @duration desc
| limit 20
```

```sql
-- Find all 5XX errors from API Gateway
fields @timestamp, @message
| filter @message like /5\d{2}/
| parse @message '"statusCode":*,' as statusCode
| filter statusCode >= 500
| stats count(*) by statusCode
```

```sql
-- Find specific IP addresses in access logs
fields @timestamp, @message
| filter @message like /192\.168\.1\./
| sort @timestamp desc
| limit 100
```

### Log Retention — How Long to Keep Logs?

```bash
# Set log retention (default is FOREVER — which costs money!)
aws logs put-retention-policy \
  --log-group-name /aws/lambda/my-function \
  --retention-in-days 30

# Common retention periods:
#   1, 3, 5, 7, 14, 30, 60, 90, 120, 150, 180, 365, 400, 545, 731, 1096, 1827, 2192, 2557, 2922, 3288, 3653
```

> **Pro Tip:** Always set retention! CloudWatch Logs can get VERY expensive if you never delete old logs. For most apps, 30-90 days is sufficient. Archive important logs to S3 for cheaper long-term storage.

### Exporting Logs to S3 (For Cheap Storage)

```bash
# Export logs from CloudWatch to S3
aws logs create-export-task \
  --log-group-name /aws/lambda/my-function \
  --from 1704067200000 \
  --to 1704153600000 \
  --destination my-logs-bucket \
  --destination-prefix "lambda-logs"
```

## 🚨 CloudWatch Alarms

### What Are Alarms?

Alarms watch a metric and take action when it crosses a threshold:

```
Example:
  Watch: EC2 CPUUtilization
  Threshold: > 80% for 5 minutes
  Action: Send SNS notification → Email/Slack/PagerDuty
  
  States:
    OK        → Metric is within the threshold (all good ✅)
    ALARM     → Metric has crossed the threshold (something's wrong! 🚨)
    INSUFFICIENT_DATA → Not enough data to determine state
```

### Creating Alarms

```bash
# Alarm: CPU > 80% for 5 minutes → Send email
aws cloudwatch put-metric-alarm \
  --alarm-name "High-CPU-Alarm" \
  --metric-name CPUUtilization \
  --namespace AWS/EC2 \
  --statistic Average \
  --period 300 \
  --threshold 80 \
  --comparison-operator GreaterThanThreshold \
  --evaluation-periods 1 \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:my-alerts-topic \
  --dimensions Name=InstanceId,Value=i-0abc123def456 \
  --alarm-description "Alert when CPU exceeds 80%"
```

### Creating Alarms with Terraform

```hcl
# SNS Topic for alerts
resource "aws_sns_topic" "alerts" {
  name = "infrastructure-alerts"
}

resource "aws_sns_topic_subscription" "email" {
  topic_arn = aws_sns_topic.alerts.arn
  protocol  = "email"
  endpoint  = "devops-team@company.com"
}

# Alarm: EC2 CPU > 80%
resource "aws_cloudwatch_metric_alarm" "high_cpu" {
  alarm_name          = "high-cpu-alarm"
  comparison_operator = "GreaterThanThreshold"
  evaluation_periods  = 2

  namespace           = "AWS/EC2"

  statistic           = "Average"

  alarm_actions       = [aws_sns_topic.alerts.arn]

  dimensions = {
    InstanceId = aws_instance.web.id
  }
}

# Alarm: ALB 5XX errors > 10 in 5 minutes

  evaluation_periods  = 1
  metric_name         = "HTTPCode_Target_5XX_Count"
  namespace           = "AWS/ApplicationELB"
  period              = 300
  statistic           = "Sum"
  threshold           = 10
  alarm_description   = "More than 10 5XX errors in 5 minutes"

}

  metric_name         = "Errors"
  namespace           = "AWS/Lambda"
  period              = 300
  statistic           = "Sum"
  threshold           = 5
  alarm_description   = "Lambda function errors exceeded threshold"
  alarm_actions       = [aws_sns_topic.alerts.arn]
  
  dimensions = {
    FunctionName = aws_lambda_function.my_function.function_name
  }
}

# Alarm: RDS Free Storage < 5 GB
resource "aws_cloudwatch_metric_alarm" "rds_storage" {
  alarm_name          = "rds-low-storage"
  comparison_operator = "LessThanThreshold"
  evaluation_periods  = 1
  metric_name         = "FreeStorageSpace"
  namespace           = "AWS/RDS"
  period              = 300
  statistic           = "Average"
  threshold           = 5000000000   # 5 GB in bytes
  alarm_description   = "RDS free storage is below 5 GB"

  dimensions = {
    DBInstanceIdentifier = aws_db_instance.main.id
  }
}
```

### Composite Alarms (Combine Multiple Alarms)

```hcl
# Trigger ONLY when BOTH CPU is high AND errors are high
resource "aws_cloudwatch_composite_alarm" "critical" {
  alarm_name = "critical-composite-alarm"
  
  alarm_rule = "ALARM(${aws_cloudwatch_metric_alarm.high_cpu.alarm_name}) AND ALARM(${aws_cloudwatch_metric_alarm.alb_5xx.alarm_name})"
  
  alarm_actions = [aws_sns_topic.critical_alerts.arn]
  alarm_description = "Both high CPU and high error rate — likely a real incident"
}
```

## 🔔 CloudWatch Events / Amazon EventBridge

> **Note:** CloudWatch Events has been rebranded as **Amazon EventBridge**. They're the same service — EventBridge is the newer, more powerful version.

EventBridge lets you **react to events** in your AWS account:

```
Event: EC2 instance stopped
  → Action: Send Slack notification

Event: Every day at 6 AM UTC
  → Action: Trigger a Lambda function to create backups

Event: AWS Health event (service degradation)
  → Action: Page the on-call engineer

Event: IAM user created
  → Action: Log it and send security alert
```

### Example: EventBridge Rule (Terraform)

```hcl
# Notify when an EC2 instance changes state
resource "aws_cloudwatch_event_rule" "ec2_state_change" {
  name        = "ec2-state-change"
  description = "Capture EC2 instance state changes"
  
  event_pattern = jsonencode({
    source      = ["aws.ec2"]
    detail-type = ["EC2 Instance State-change Notification"]
    detail = {
      state = ["stopped", "terminated"]
    }
  })
}

resource "aws_cloudwatch_event_target" "notify" {
  rule = aws_cloudwatch_event_rule.ec2_state_change.name
  arn  = aws_sns_topic.alerts.arn
}
```

## ✅ What is CloudTrail?

CloudTrail is AWS's **audit logging service**. It records **every API call** made in your AWS account:

```
Who?    → Which IAM user or role made the call
What?   → What action was performed (e.g., RunInstances, DeleteBucket)
When?   → Exact timestamp
Where?  → Which region, which resource
How?    → From which IP address, using which method (Console, CLI, SDK)
Result? → Did it succeed or fail?
```

### The Simple Version:

```
CloudTrail = A security camera for your AWS account 📹
Every action is recorded. Nothing goes unnoticed.
```

### Example CloudTrail Event:

```json
{
    "eventVersion": "1.08",
    "userIdentity": {
        "type": "IAMUser",
        "principalId": "AIDAEXAMPLE123",
        "arn": "arn:aws:iam::123456789012:user/alice",
        "accountId": "123456789012",
```

## 🛠️ Setting Up CloudTrail

### Important: CloudTrail Basics

```
Event Types:
├── Management Events (default: ON, free for 1 trail)
│   ├── API calls that manage your AWS resources
│   ├── Examples: CreateBucket, RunInstances, CreateUser
│   └── Also called "Control Plane" events
│
├── Data Events (optional: OFF by default, costs money)
│   ├── API calls on data WITHIN resources
│   ├── Examples: S3 GetObject/PutObject, Lambda Invoke
│   └── High volume — can generate millions of events
│
└── Insights Events (optional: OFF by default)
    ├── Detects UNUSUAL activity automatically
    └── Examples: Spike in API calls, unusual error rates
```

### Create a Trail (AWS CLI)

```bash
# Step 1: Create an S3 bucket for CloudTrail logs
aws s3 mb s3://my-cloudtrail-logs-bucket

# Step 2: Apply the bucket policy (CloudTrail needs permission to write)
aws s3api put-bucket-policy \
  --bucket my-cloudtrail-logs-bucket \
  --policy '{
    "Version": "2012-10-17",
    "Statement": [
      {
        "Sid": "AWSCloudTrailAclCheck",
        "Effect": "Allow",
        "Principal": {"Service": "cloudtrail.amazonaws.com"},
        "Action": "s3:GetBucketAcl",
        "Resource": "arn:aws:s3:::my-cloudtrail-logs-bucket"
      },
      {
        "Sid": "AWSCloudTrailWrite",
        "Effect": "Allow",
        "Principal": {"Service": "cloudtrail.amazonaws.com"},
        "Action": "s3:PutObject",
        "Resource": "arn:aws:s3:::my-cloudtrail-logs-bucket/AWSLogs/*",
        "Condition": {
          "StringEquals": {"s3:x-amz-acl": "bucket-owner-full-control"}
        }
      }
    ]
  }'

# Step 3: Create the trail
aws cloudtrail create-trail \
  --name my-organization-trail \
  --s3-bucket-name my-cloudtrail-logs-bucket \
  --is-multi-region-trail \
  --enable-log-file-validation

# Step 4: Start logging
aws cloudtrail start-logging --name my-organization-trail

# Step 5: Verify
aws cloudtrail get-trail-status --name my-organization-trail
```

### Create a Trail with Terraform

```hcl

      },
      {
        Sid       = "AWSCloudTrailWrite"
        Effect    = "Allow"

          StringEquals = {
            "s3:x-amz-acl" = "bucket-owner-full-control"
          }
        }
      }
    ]
  })
}

# CloudTrail
resource "aws_cloudtrail" "main" {
  name                       = "organization-trail"
  s3_bucket_name             = aws_s3_bucket.cloudtrail.id
  is_multi_region_trail      = true        # Log from ALL regions
  enable_log_file_validation = true        # Detect log tampering
  include_global_service_events = true     # IAM, STS, CloudFront events
  
  # Send logs to CloudWatch for easy searching
  cloud_watch_logs_group_arn = "${aws_cloudwatch_log_group.cloudtrail.arn}:*"
  cloud_watch_logs_role_arn  = aws_iam_role.cloudtrail_cloudwatch.arn
  
  # Enable data events for S3 (optional — can be expensive)

    read_write_type           = "All"
    include_management_events = true
    
    data_resource {
      type   = "AWS::S3::Object"
      values = ["arn:aws:s3:::sensitive-data-bucket/"]   # Only for specific buckets
    }
  }
  
  # Enable CloudTrail Insights
  insight_selector {
    insight_type = "ApiCallRateInsight"
  }
}

# CloudWatch Log Group for CloudTrail
resource "aws_cloudwatch_log_group" "cloudtrail" {
  name              = "/aws/cloudtrail/organization-trail"
  retention_in_days = 90
}
```

## 🔎 Searching CloudTrail Events

### Using AWS CLI

```bash
# Find who launched EC2 instances in the last 24 hours
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=RunInstances \
  --start-time $(date -d '24 hours ago' -u +%Y-%m-%dT%H:%M:%SZ) \
  --max-results 20

# Find all actions by a specific user
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=Username,AttributeValue=alice \
  --start-time 2024-01-01T00:00:00Z
```

## ✅ What is AWS Config?

AWS Config is a service that:

1. **Records** the configuration of your AWS resources over time
2. **Evaluates** whether your resources comply with desired configurations
3. **Alerts** you when something is non-compliant

```
Think of AWS Config as a "Configuration Compliance Police" 👮

It constantly watches your resources and checks:
  ✅ "Is S3 bucket encryption turned ON?"
  ✅ "Do all EC2 instances have required tags?"
  ✅ "Are security groups NOT allowing 0.0.0.0/0 on port 22?"
  ❌ "This RDS instance has no backup enabled!" → NON-COMPLIANT → ALERT!
```

### CloudTrail vs AWS Config — What's the Difference?

```
CloudTrail:                        AWS Config:
"WHO did WHAT, WHEN?"              "HOW is this resource configured NOW
                                    and how has it changed over time?"

"Alice deleted the S3 bucket       "S3 bucket 'my-bucket' had encryption
 at 2:30 PM"                        turned OFF on Jan 15, which violates
                                     our compliance rule"

Records API CALLS                  Records RESOURCE CONFIGURATIONS
Answer: "What happened?"           Answer: "Is this configured correctly?"
```

## 📏 AWS Config Rules — Compliance Checking

### What Are Config Rules?

Config Rules are **compliance checks** that automatically evaluate your resources:

```
Rule: "s3-bucket-server-side-encryption-enabled"
  ↓
AWS Config checks EVERY S3 bucket in your account
  ↓
  ├── my-encrypted-bucket → Encryption: ON → ✅ COMPLIANT
  ├── my-other-bucket     → Encryption: ON → ✅ COMPLIANT
  └── my-dev-bucket       → Encryption: OFF → ❌ NON-COMPLIANT → ALERT!
```

### Types of Config Rules:

```
1. AWS Managed Rules (150+ pre-built rules by AWS)
   ├── s3-bucket-server-side-encryption-enabled
   ├── ec2-instance-no-public-ip
   ├── rds-instance-public-access-check
   ├── iam-password-policy
   ├── restricted-ssh (no 0.0.0.0/0 on port 22)
   └── required-tags
   
2. Custom Rules (you write them)
   ├── Lambda-backed rules (your Lambda evaluates compliance)
   └── Guard rules (using CloudFormation Guard policy language)
```

### Custom Config Rule (Lambda-Backed)

For rules that AWS doesn't provide out of the box, you write your own:

```python
# lambda_function.py — Custom Config Rule
# Check: EC2 instances must use only approved AMIs

import json
import boto3

config = boto3.client('config')

APPROVED_AMIS = [
    'ami-0abc123def456',  # Ubuntu 22.04 (hardened)
    'ami-0def456abc789',  # Amazon Linux 2023 (hardened)
]

def lambda_handler(event, context):
    """
    AWS Config calls this Lambda for each resource it evaluates.
    """
    # Parse the invoking event
    invoking_event = json.loads(event['invokingEvent'])
    configuration_item = invoking_event['configurationItem']
    
    # Get the resource info
    resource_type = configuration_item['resourceType']
    resource_id = configuration_item['resourceId']
    configuration = configuration_item['configuration']
    
    # Only evaluate EC2 instances
    if resource_type != 'AWS::EC2::Instance':
        return
    
    # Check if the AMI is approved
    ami_id = configuration.get('imageId', '')
    
    if ami_id in APPROVED_AMIS:
        compliance = 'COMPLIANT'
        annotation = f'Instance uses approved AMI: {ami_id}'
    else:
        compliance = 'NON_COMPLIANT'
        annotation = f'Instance uses unapproved AMI: {ami_id}. Approved: {APPROVED_AMIS}'
    
    # Report the result back to AWS Config
    config.put_evaluations(
        Evaluations=[
            {
                'ComplianceResourceType': resource_type,
                'ComplianceResourceId': resource_id,
                'ComplianceType': compliance,
                'Annotation': annotation,
                'OrderingTimestamp': configuration_item['configurationItemCaptureTime']
            }
        ],
        ResultToken=event['resultToken']
    )
```


## 🔧 AWS Config Remediation — Auto-Fix Non-Compliance

You can configure AWS Config to **automatically fix** non-compliant resources:

```hcl
# Auto-remediation: If S3 encryption is off, turn it on automatically!
resource "aws_config_remediation_configuration" "s3_encryption" {
  config_rule_name = aws_config_config_rule.s3_encryption.name
  
  target_type    = "SSM_DOCUMENT"
  target_id      = "AWS-EnableS3BucketEncryption"   # Pre-built SSM automation
  
  parameter {
    name           = "BucketName"
    resource_value = "RESOURCE_ID"
  }
  parameter {
    name         = "SSEAlgorithm"
    static_value = "AES256"
  }
  
  automatic                  = true    # Auto-fix without manual approval
  maximum_automatic_attempts = 3
  retry_attempt_seconds      = 60
}
```

```
How Auto-Remediation Works:

1. AWS Config detects: S3 bucket "dev-logs" has encryption OFF → NON-COMPLIANT ❌
2. Remediation triggers automatically
3. SSM Automation runs → Enables encryption on "dev-logs"
4. AWS Config re-evaluates → "dev-logs" has encryption ON → COMPLIANT ✅
5. 🎉 Fixed automatically without human intervention!
```

## 🆚 CloudWatch vs CloudTrail vs AWS Config — Summary

| Feature | CloudWatch | CloudTrail | AWS Config |
|---|---|---|---|
| **Purpose** | Monitoring & alerting | Audit trail (who did what) | Configuration compliance |
| **What it tracks** | Metrics, logs, events | API calls | Resource configurations |
| **Key question** | "How is it performing?" | "Who changed it?" | "Is it configured correctly?" |
| **Real-time?** | Yes (metrics/alarms) | Near real-time (15 min) | Periodic or on change |
| **Storage** | CloudWatch (logs/metrics) | S3 bucket | S3 bucket |
| **Cost** | Per metric/log/alarm | Free (1 trail, mgmt events) | Per rule evaluation |
| **Common use** | CPU alerts, error monitoring | Security auditing | Compliance checking |
| **Example** | "CPU is at 92%!" | "Alice deleted the DB" | "DB has no encryption" |
