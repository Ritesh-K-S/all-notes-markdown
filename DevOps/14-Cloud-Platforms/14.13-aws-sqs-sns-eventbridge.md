## 1. Why Do We Need Messaging Services?

Imagine you have a web app where a user places an order. That order needs to:
1. Save to a database
2. Send a confirmation email
3. Notify the warehouse
4. Update inventory
5. Send analytics data

### The Problem with Direct Communication (Synchronous)

```
User → Order Service → Email Service → Warehouse Service → Inventory Service
```

If the Email Service goes down, **the whole chain breaks**. The user gets an error even though the order was valid. This is called **tight coupling** — services are too dependent on each other.

### The Solution: Messaging (Asynchronous)

```
User → Order Service → Message Queue/Topic → Email Service
                                            → Warehouse Service
                                            → Inventory Service
```

Now the Order Service just drops a message into a queue or topic, and all the other services pick it up **when they're ready**. If one service is down, the message waits. No one breaks!

### Key Benefits

| Benefit | What It Means |
|---------|---------------|
| **Decoupling** | Services don't need to know about each other |
| **Resilience** | If a consumer is down, messages wait in the queue |
| **Scalability** | Add more consumers to handle more messages |
| **Buffering** | Handle traffic spikes without overwhelming services |
| **Ordering** | Process things in the right sequence (with FIFO queues) |

## 2. Amazon SQS — Simple Queue Service

### What is SQS?

**SQS** (Simple Queue Service) is a **fully managed message queue** service by AWS. Think of it like a **mailbox** — a producer (sender) puts messages in, and a consumer (receiver) picks them up.

```
┌──────────┐     ┌──────────────┐     ┌──────────┐
│ Producer  │────▶│  SQS Queue   │────▶│ Consumer │
│ (Sender)  │     │  (Mailbox)   │     │(Receiver)│
└──────────┘     └──────────────┘     └──────────┘
```

### Key Characteristics

- **Fully managed** — No servers to maintain, AWS handles everything
- **Serverless** — Scales automatically from 1 message to billions
- **Highly available** — Messages are stored redundantly across multiple AZs
- **Pay per use** — You pay for what you use (first 1 million requests/month are FREE)
- **Message size** — Up to **256 KB** per message (use S3 for larger payloads)
- **Retention** — Messages stay in the queue for **1 minute to 14 days** (default is 4 days)
- **At-least-once delivery** — A message might be delivered more than once (Standard queue)

### How SQS Works (Step by Step)

```
Step 1: Producer sends a message to the SQS queue
Step 2: The message sits in the queue waiting to be processed
Step 3: A consumer polls (asks) the queue for messages
Step 4: Consumer receives the message and processes it
Step 5: Consumer deletes the message from the queue (to mark it as "done")
```

> **Important:** SQS is a **pull-based** system. Consumers have to come and ask for messages — SQS doesn't push messages to consumers.

## 3. SQS Queue Types — Standard vs FIFO

### Standard Queue

| Feature | Details |
|---------|---------|
| **Throughput** | **Unlimited** — nearly unlimited messages per second |
| **Ordering** | **Best-effort ordering** — messages *might* come out of order |
| **Delivery** | **At-least-once** — a message *might* be delivered more than once |
| **Use case** | When order doesn't matter (e.g., log processing, batch jobs) |

```
Messages In:  A → B → C → D
Messages Out: A → C → B → D  (order may vary)
               A might come twice!
```

### FIFO Queue

| Feature | Details |
|---------|---------|
| **Throughput** | **300 messages/sec** (without batching), **3000/sec** (with batching) |
| **Ordering** | **Strict FIFO** — First In, First Out, guaranteed |
| **Delivery** | **Exactly-once** — no duplicate delivery |
| **Naming** | Queue name **must end with `.fifo`** (e.g., `orders.fifo`) |
| **Use case** | When order matters (e.g., banking transactions, order processing) |

```
Messages In:  A → B → C → D
Messages Out: A → B → C → D  (exact same order, guaranteed!)
```

### When to Choose What?

| Scenario | Choose |
|----------|--------|
| Processing log files | Standard |
| Order processing in e-commerce | FIFO |
| Sending email notifications | Standard |
| Banking transactions | FIFO |
| Image thumbnail generation | Standard |
| Chat messages | FIFO |

## 4. SQS — Key Concepts

### Message Lifecycle

```
┌─────────┐   Send    ┌───────────┐  Receive   ┌──────────┐  Delete   ┌──────────┐
│ Producer │ ────────▶ │   Queue   │ ─────────▶ │ Consumer │ ────────▶ │ Removed  │
└─────────┘           │ (waiting) │            │(process) │           │  (done)  │
                      └───────────┘            └──────────┘           └──────────┘
```

### Visibility Timeout

When a consumer picks up a message, the message becomes **invisible** to other consumers for a period called the **Visibility Timeout** (default: 30 seconds).

```
Consumer A picks up Message → Message becomes invisible for 30s
     ↓
If Consumer A processes it and deletes it → Done! ✅
If Consumer A crashes before deleting it → After 30s, message reappears → Another consumer can pick it up
```

- Default: **30 seconds**
- Range: **0 seconds to 12 hours**
- Set it longer than your processing time!

### Long Polling vs Short Polling

| Polling Type | How It Works | Cost |
|-------------|--------------|------|
| **Short Polling** | Consumer asks "Any messages?" → Gets immediate response (even if empty) | More API calls = More cost |
| **Long Polling** | Consumer asks "Any messages?" → Waits up to 20 seconds for a message to arrive | Fewer API calls = Less cost |

**Always use Long Polling!** Set `WaitTimeSeconds` to a value between 1-20.

### Message Attributes

You can attach metadata to messages:

```json
{
  "MessageBody": "Order #12345 placed",
  "MessageAttributes": {
    "OrderType": {
      "DataType": "String",
      "StringValue": "Premium"
    },
    "Amount": {
      "DataType": "Number",
      "StringValue": "99.99"
    }
  }
}
```

### Delay Queues

You can delay the delivery of messages. The message is "hidden" for a specified time after being sent.

- **Queue-level delay:** 0 to 15 minutes (applies to ALL messages)
- **Message-level delay:** 0 to 15 minutes (per-message, only for Standard queues)

Use case: "Process this order 5 minutes after it's placed" (gives customers time to cancel).

## 5. SQS — Hands-On with AWS CLI

### Create a Standard Queue

```bash
# Create a simple standard queue
aws sqs create-queue \
  --queue-name my-first-queue

# Output:
# {
#     "QueueUrl": "https://sqs.us-east-1.amazonaws.com/123456789012/my-first-queue"
# }
```

### Create a FIFO Queue

```bash
# Create a FIFO queue (name MUST end with .fifo)
aws sqs create-queue \
  --queue-name my-order-queue.fifo \
  --attributes '{"FifoQueue": "true", "ContentBasedDeduplication": "true"}'

# ContentBasedDeduplication = AWS uses the message body to detect duplicates
```

### Send a Message

```bash
# Send a message to a Standard queue
aws sqs send-message \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/my-first-queue \
  --message-body "Hello from SQS! Order #12345 has been placed."

# Output:
# {
#     "MD5OfMessageBody": "abc123...",
#     "MessageId": "msg-id-12345"
# }
```

```bash
# Send a message to a FIFO queue
aws sqs send-message \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/my-order-queue.fifo \
  --message-body "Order #12345 placed" \
  --message-group-id "orders"

# --message-group-id: Messages in the same group are delivered in order
# Think of it like a "lane" — messages in the same lane stay in order
```

### Receive Messages

```bash
# Receive messages (long polling with 10 second wait)
aws sqs receive-message \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/my-first-queue \
  --max-number-of-messages 5 \
  --wait-time-seconds 10

# Output:
# {
#     "Messages": [
#         {
#             "MessageId": "msg-id-12345",
#             "ReceiptHandle": "AQEBwJ...",  ← You need this to delete the message!
#             "Body": "Hello from SQS! Order #12345 has been placed."
#         }
#     ]
# }
```

### Delete a Message (After Processing)

```bash
# Delete the message after you've processed it
aws sqs delete-message \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/my-first-queue \
  --receipt-handle "AQEBwJ..."

# If you don't delete it, it will reappear after the Visibility Timeout!
```

### Check Queue Attributes

```bash
# See how many messages are in the queue
aws sqs get-queue-attributes \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/my-first-queue \
  --attribute-names ApproximateNumberOfMessages ApproximateNumberOfMessagesNotVisible

# Output:
# {
#     "Attributes": {
#         "ApproximateNumberOfMessages": "5",           ← Messages waiting to be picked up
#         "ApproximateNumberOfMessagesNotVisible": "2"   ← Messages being processed right now
#     }
# }
```

### Delete a Queue

```bash
# Delete the queue (WARNING: all messages in it will be lost!)
aws sqs delete-queue \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/my-first-queue
```

## 6. SQS — Dead Letter Queues (DLQ)

### What is a Dead Letter Queue?

Sometimes a message keeps failing — maybe the consumer crashes every time it tries to process that particular message. After a certain number of failed attempts, you don't want that "poison pill" message blocking everything.

A **Dead Letter Queue (DLQ)** is a special queue where these "problem" messages are sent after they fail too many times.

```
┌──────────┐     ┌──────────────┐     ┌──────────┐
│ Producer  │────▶│  Main Queue  │────▶│ Consumer │  (tries to process, fails)
└──────────┘     └──────┬───────┘     └──────────┘
                        │
                        │ After 3 failed attempts
                        ▼
                 ┌──────────────┐
                 │  Dead Letter  │  ← "Problem messages land here"
                 │    Queue      │
                 └──────────────┘
```

### Setting Up a DLQ

```bash
# Step 1: Create the DLQ first
aws sqs create-queue --queue-name my-queue-dlq

# Step 2: Get the DLQ's ARN
aws sqs get-queue-attributes \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/my-queue-dlq \
  --attribute-names QueueArn

# Step 3: Create the main queue with a Redrive Policy pointing to the DLQ
aws sqs create-queue \
  --queue-name my-main-queue \
  --attributes '{
    "RedrivePolicy": "{\"deadLetterTargetArn\":\"arn:aws:sqs:us-east-1:123456789012:my-queue-dlq\",\"maxReceiveCount\":\"3\"}"
  }'

# maxReceiveCount: 3 → After 3 failed attempts, message goes to DLQ
```

### Key Points About DLQs

- Always create a DLQ for important queues
- Set up **CloudWatch Alarms** on the DLQ to get notified when messages land there
- The DLQ must be the **same type** as the main queue (Standard → Standard, FIFO → FIFO)
- Regularly check your DLQ and investigate why messages are failing
- You can **redrive** messages from the DLQ back to the main queue (retry them)

## 7. SQS — Visibility Timeout & Long Polling

### Visibility Timeout — Deep Dive

```
Timeline:
=========

0s          Consumer receives message
             ↓
             Message becomes INVISIBLE to other consumers
             ↓
0-30s       Consumer is processing the message...
             ↓
30s         Visibility Timeout expires!
             ↓
If NOT deleted → Message becomes VISIBLE again → Another consumer might pick it up
If DELETED     → Message is gone forever ✅
```

#### Changing Visibility Timeout

```bash
# Change visibility timeout for a specific message (extend processing time)
aws sqs change-message-visibility \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/my-queue \
  --receipt-handle "AQEBwJ..." \
  --visibility-timeout 60

# Now this specific message will stay invisible for 60 more seconds
```

**Pro tip:** If your processing takes a variable amount of time, start with a reasonable timeout and extend it using `ChangeMessageVisibility` if you need more time.

### Long Polling — Deep Dive

```
Short Polling (BAD):
====================
Consumer: "Any messages?" → Server: "Nope" (empty response)
Consumer: "Any messages?" → Server: "Nope" (empty response)
Consumer: "Any messages?" → Server: "Nope" (empty response)
Consumer: "Any messages?" → Server: "Here's one!"
Result: 4 API calls, 3 were wasted $$

Long Polling (GOOD):
====================
Consumer: "Any messages? I'll wait 20 seconds..."
Server: ...waiting...waiting... "Here's one!"
Result: 1 API call $$
```

```bash
# Enable long polling at the queue level (all consumers)
aws sqs set-queue-attributes \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/my-queue \
  --attributes '{"ReceiveMessageWaitTimeSeconds": "20"}'

# Or per-request
aws sqs receive-message \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/my-queue \
  --wait-time-seconds 20
```

## 8. Amazon SNS — Simple Notification Service

### What is SNS?

**SNS** (Simple Notification Service) is a **fully managed pub/sub (publish/subscribe)** messaging service. Think of it like a **radio broadcast** — one publisher sends a message, and everyone who's subscribed ("tuned in") receives it.

```
                         ┌──────────────┐
                    ┌───▶│  Email Sub    │  (gets the message)
                    │    └──────────────┘
┌──────────┐  Pub  │    ┌──────────────┐
│ Publisher │──────▶├───▶│  SQS Queue   │  (gets the message)
└──────────┘       │    └──────────────┘
     │             │    ┌──────────────┐
     ▼             ├───▶│  Lambda Fn   │  (gets the message)
  SNS Topic        │    └──────────────┘
                   │    ┌──────────────┐
                   └───▶│  HTTP/HTTPS  │  (gets the message)
                        └──────────────┘
```

### SQS vs SNS — The Core Difference

| Feature | SQS (Queue) | SNS (Topic) |
|---------|-------------|-------------|
| **Model** | Pull (consumers poll for messages) | Push (SNS pushes to subscribers) |
| **Consumers** | One consumer processes each message | All subscribers get every message |
| **Persistence** | Messages are stored until consumed | No persistence (fire & forget) |
| **Use case** | Worker queues, job processing | Notifications, fan-out, alerts |
| **Analogy** | Mailbox (one person reads the letter) | Radio broadcast (everyone hears it) |

### SNS Key Characteristics

- **Fully managed** — No servers, scales automatically
- **Push-based** — Messages are pushed to subscribers immediately
- **Up to 12.5 million subscriptions** per topic
- **Up to 100,000 topics** per account
- **Message size** — Up to **256 KB**
- **Message filtering** — Subscribers can filter which messages they receive
- **Multiple protocols** — Email, SMS, HTTP/HTTPS, SQS, Lambda, Firehose

## 9. SNS — Key Concepts

### Topics

A **Topic** is a communication channel. Publishers send messages to a topic, and all subscribers to that topic receive the message.

```
Topic: "order-events"
  ├── Subscriber 1: Email (admin@company.com)
  ├── Subscriber 2: SQS Queue (order-processing-queue)
  ├── Subscriber 3: Lambda (send-confirmation-email)
  └── Subscriber 4: HTTP Endpoint (https://webhook.site/abc123)
```

### Subscription Protocols

| Protocol | Description | Example |
|----------|-------------|---------|
| **Email** | Sends email notifications | admin@company.com |
| **Email-JSON** | Sends email with raw JSON | admin@company.com |
| **SMS** | Sends text messages | +1-555-123-4567 |
| **HTTP/HTTPS** | Sends HTTP POST requests | https://myapp.com/webhook |
| **SQS** | Sends messages to an SQS queue | arn:aws:sqs:...:my-queue |
| **Lambda** | Invokes a Lambda function | arn:aws:lambda:...:my-function |
| **Firehose** | Sends to Kinesis Data Firehose | arn:aws:firehose:...:my-stream |
| **Application** | Mobile push notifications (iOS, Android) | Platform endpoint ARN |

### Message Format

When SNS delivers a message, it wraps it in a standard format:

```json
{
  "Type": "Notification",
  "MessageId": "msg-12345",
  "TopicArn": "arn:aws:sns:us-east-1:123456789012:order-events",
  "Subject": "New Order",
  "Message": "Order #12345 has been placed!",
  "Timestamp": "2024-01-15T10:30:00.000Z",
  "SignatureVersion": "1",
  "Signature": "base64-encoded-signature",
  "SigningCertURL": "https://...",
  "UnsubscribeURL": "https://..."
}
```

## 10. SNS — Hands-On with AWS CLI

### Create a Topic

```bash
# Create an SNS topic
aws sns create-topic --name order-events

# Output:
# {
#     "TopicArn": "arn:aws:sns:us-east-1:123456789012:order-events"
# }
```

### Subscribe to a Topic

```bash
# Subscribe with Email
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123456789012:order-events \
  --protocol email \
  --notification-endpoint admin@company.com

# The subscriber will get a confirmation email — they must click "Confirm" to start receiving!

# Subscribe with SQS
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123456789012:order-events \
  --protocol sqs \
  --notification-endpoint arn:aws:sqs:us-east-1:123456789012:order-processing-queue

# Subscribe with Lambda
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123456789012:order-events \
  --protocol lambda \
  --notification-endpoint arn:aws:lambda:us-east-1:123456789012:function:process-order

# Subscribe with HTTP endpoint
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123456789012:order-events \
  --protocol https \
  --notification-endpoint https://myapp.com/webhook/orders
```

### Publish a Message

```bash
# Publish a message to the topic (ALL subscribers will receive it)
aws sns publish \
  --topic-arn arn:aws:sns:us-east-1:123456789012:order-events \
  --subject "New Order Placed" \
  --message "Order #12345 has been placed by customer John Doe. Total: $99.99"

# Output:
# {
#     "MessageId": "msg-id-67890"
# }
```

### Publish with Message Attributes

```bash

  --topic-arn arn:aws:sns:us-east-1:123456789012:order-events \

```

### List Topics and Subscriptions

```bash

# List subscriptions for a specific topic

  --topic-arn arn:aws:sns:us-east-1:123456789012:order-events

  --subscription-arn arn:aws:sns:us-east-1:123456789012:order-events:sub-id-12345
```

### Delete a Topic

```bash

  --topic-arn arn:aws:sns:us-east-1:123456789012:order-events
```

## 11. SNS — Message Filtering

### What is Message Filtering?

Without filtering, **every subscriber gets every message**. With filtering, subscribers can say "I only want messages that match certain criteria."

### Example Scenario

You have a topic `order-events`. Three subscribers:
1. **Premium team** — only wants Premium orders
2. **Warehouse** — only wants orders over $50
3. **Analytics** — wants everything

### Setting Up a Filter Policy

```bash
# Subscribe the premium team with a filter
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123456789012:order-events \
  --protocol sqs \
  --notification-endpoint arn:aws:sqs:us-east-1:123456789012:premium-orders-queue \
  --attributes '{
    "FilterPolicy": "{\"OrderType\": [\"Premium\"]}"
  }'

# This subscriber will ONLY receive messages where OrderType = "Premium"
```

### Filter Policy Examples

```json
// Match exact value
{"OrderType": ["Premium"]}

// Match multiple values (OR)
{"OrderType": ["Premium", "Enterprise"]}

// Numeric matching
{"Amount": [{"numeric": [">=", 50]}]}

// Prefix matching
{"CustomerName": [{"prefix": "John"}]}

// Anything-but (NOT)
{"OrderType": [{"anything-but": "Free"}]}

// Exists (attribute must be present)
{"OrderType": [{"exists": true}]}

// Combine multiple conditions (AND)
{
  "OrderType": ["Premium"],
  "Amount": [{"numeric": [">=", 100]}]
}
// This means: OrderType IS "Premium" AND Amount >= 100
```

## 12. SNS + SQS — Fan-Out Pattern

### What is Fan-Out?

The **Fan-Out Pattern** is one of the most common and powerful patterns in AWS. You publish a message to an SNS topic, and it "fans out" to multiple SQS queues. Each queue has its own consumer that processes the message independently.

```
                              ┌─────────────────────┐     ┌─────────────────┐
                         ┌───▶│ SQS: Email Queue     │────▶│ Email Service   │
                         │    └─────────────────────┘     └─────────────────┘
                         │
┌──────────┐    Pub      │    ┌─────────────────────┐     ┌─────────────────┐
│  Order    │───────▶ SNS├───▶│ SQS: Warehouse Queue│────▶│ Warehouse Svc   │
│  Service  │   Topic    │    └─────────────────────┘     └─────────────────┘
└──────────┘             │
                         │    ┌─────────────────────┐     ┌─────────────────┐
                         └───▶│ SQS: Analytics Queue │────▶│ Analytics Svc   │
                              └─────────────────────┘     └─────────────────┘
```

### Why SNS + SQS Together?

| Feature | SNS Alone | SQS Alone | SNS + SQS |
|---------|-----------|-----------|-----------|
| Push to many consumers | ✅ | ❌ | ✅ |
| Message persistence | ❌ | ✅ | ✅ |
| Independent processing | ❌ | ✅ | ✅ |
| Retry on failure | ❌ | ✅ | ✅ |
| Dead letter queues | ❌ | ✅ | ✅ |

**SNS + SQS = Best of both worlds!** SNS broadcasts to multiple queues, and each queue independently buffers and retries.

### Setting Up Fan-Out

```bash
# Step 1: Create the SNS Topic
aws sns create-topic --name order-events

# Step 2: Create multiple SQS Queues
aws sqs create-queue --queue-name email-queue
aws sqs create-queue --queue-name warehouse-queue
aws sqs create-queue --queue-name analytics-queue

# Step 3: Subscribe each queue to the topic
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123456789012:order-events \
  --protocol sqs \
  --notification-endpoint arn:aws:sqs:us-east-1:123456789012:email-queue

aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123456789012:order-events \
  --protocol sqs \
  --notification-endpoint arn:aws:sqs:us-east-1:123456789012:warehouse-queue

aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123456789012:order-events \
  --protocol sqs \
  --notification-endpoint arn:aws:sqs:us-east-1:123456789012:analytics-queue

# Step 4: You also need to add an SQS access policy to allow SNS to send messages!
# (See Terraform section below for the cleaner approach)
```

> **Important:** You need to add an **SQS Queue Policy** allowing the SNS topic to send messages to the queue. Otherwise, the messages won't be delivered!

## 13. Amazon EventBridge

### What is EventBridge?

**Amazon EventBridge** (formerly called CloudWatch Events) is a **serverless event bus** service. Think of it as an **upgrade to SNS** with superpowers:

- It can receive events from **AWS services** automatically (like "an EC2 instance was terminated")
- It can receive events from **third-party SaaS** apps (Shopify, Zendesk, Datadog, etc.)
- It can receive **custom events** from your own apps
- It has **powerful event filtering and routing** using JSON-based rules
- It supports **event schemas** and a **schema registry**

### EventBridge vs SNS — Think of It This Way

| Feature | SNS | EventBridge |
|---------|-----|-------------|
| **Message format** | Any (text, JSON) | Structured JSON events |
| **Filtering** | Simple attribute-based | Advanced JSON content-based filtering |
| **AWS integration** | Manual setup | Automatic (100+ AWS services emit events) |
| **SaaS integration** | ❌ | ✅ (Shopify, Zendesk, Auth0, etc.) |
| **Schema Registry** | ❌ | ✅ (discover & validate event schemas) |
| **Archive & Replay** | ❌ | ✅ (store and replay past events) |
| **Throughput** | Very high | High (but slightly lower than SNS) |
| **Number of targets** | 12.5M subscriptions | 5 targets per rule (300 rules per bus) |

> **Simple rule of thumb:**
> - Use **SNS** for high-throughput pub/sub (app-to-app messaging)
> - Use **EventBridge** for event-driven architectures (reacting to things that happen)

## 14. EventBridge — Key Concepts

### Event Bus

An **Event Bus** is where events flow through. Think of it as a highway for events.

| Bus Type | Description |
|----------|-------------|
| **Default Event Bus** | Every AWS account has one. AWS services send events here automatically |
| **Custom Event Bus** | You create these for your own application events |
| **Partner Event Bus** | Created when you integrate with SaaS partners |

### Events

An **Event** is a JSON object that describes something that happened:

```json
{
  "version": "0",
  "id": "12345-abcde-67890",
  "detail-type": "EC2 Instance State-change Notification",
  "source": "aws.ec2",
  "account": "123456789012",
  "time": "2024-01-15T10:30:00Z",
  "region": "us-east-1",
  "resources": [
    "arn:aws:ec2:us-east-1:123456789012:instance/i-1234567890abcdef0"
  ],
  "detail": {
    "instance-id": "i-1234567890abcdef0",
    "state": "terminated"
  }
}
```

### Rules

A **Rule** matches incoming events and routes them to targets. Think of it as an "if this, then that" — "IF this type of event comes in, THEN send it to this target."

### Targets

A **Target** is where you want the event to go. One rule can have up to **5 targets**.

Supported targets: Lambda, SQS, SNS, Step Functions, Kinesis, ECS Task, API Gateway, CloudWatch Logs, and more!

## 15. EventBridge — Rules & Patterns

### Event Patterns

Rules use **event patterns** to match events. If the pattern matches, the event is routed to the targets.

```json
// Match ALL EC2 state change events
{
  "source": ["aws.ec2"],
  "detail-type": ["EC2 Instance State-change Notification"]
}
```

```json
// Match only when an EC2 instance is TERMINATED
{
  "source": ["aws.ec2"],
  "detail-type": ["EC2 Instance State-change Notification"],
  "detail": {
    "state": ["terminated"]
  }
}
```

```json
// Match multiple states (terminated OR stopped)
{
  "source": ["aws.ec2"],
  "detail-type": ["EC2 Instance State-change Notification"],
  "detail": {
    "state": ["terminated", "stopped"]
  }
}
```

### Advanced Pattern Matching

```json
// Prefix matching — any source starting with "aws."
{
  "source": [{"prefix": "aws."}]
}

// Numeric matching — only orders over $100
{
  "source": ["my-app.orders"],
  "detail": {
    "amount": [{"numeric": [">", 100]}]
  }
}

// Anything-but — exclude "test" events
{
  "source": [{"anything-but": "test-app"}]
}

// Exists — only events that have a "priority" field
{
  "detail": {
    "priority": [{"exists": true}]
  }
}
```

### Scheduled Rules (Cron/Rate)

EventBridge can also trigger targets on a **schedule** (like cron jobs):

```bash
# Run every 5 minutes
aws events put-rule \
  --name "every-5-minutes" \
  --schedule-expression "rate(5 minutes)"

# Run every day at 8 AM UTC
aws events put-rule \
  --name "daily-8am" \
  --schedule-expression "cron(0 8 * * ? *)"

# Cron format: cron(minutes hours day-of-month month day-of-week year)
```

## 16. EventBridge — Hands-On with AWS CLI

### Create a Custom Event Bus

```bash
# Create a custom event bus
aws events create-event-bus --name my-app-events

# List all event buses
aws events list-event-buses
```

### Create a Rule

```bash
# Create a rule on the default bus (match EC2 terminations)
aws events put-rule \
  --name "ec2-terminated-rule" \
  --event-pattern '{
    "source": ["aws.ec2"],
    "detail-type": ["EC2 Instance State-change Notification"],
    "detail": {
      "state": ["terminated"]
    }
  }' \
  --description "Triggers when an EC2 instance is terminated"
```

### Add a Target to the Rule

```bash
# Add an SNS topic as a target
aws events put-targets \
  --rule "ec2-terminated-rule" \
  --targets '[{
    "Id": "sns-target",
    "Arn": "arn:aws:sns:us-east-1:123456789012:alerts-topic"
  }]'

# Add a Lambda function as a target
aws events put-targets \
  --rule "ec2-terminated-rule" \
  --targets '[{
    "Id": "lambda-target",
    "Arn": "arn:aws:lambda:us-east-1:123456789012:function:handle-ec2-termination"
  }]'
```

### Send a Custom Event

```bash
# Send a custom event from your application
aws events put-events --entries '[
  {
    "Source": "my-app.orders",
    "DetailType": "Order Placed",
    "Detail": "{\"orderId\": \"12345\", \"customerName\": \"John Doe\", \"amount\": 99.99, \"orderType\": \"Premium\"}",
    "EventBusName": "my-app-events"
  }
]'

# Output:
# {
#     "FailedEntryCount": 0,
#     "Entries": [
#         {
#             "EventId": "event-id-12345"
#         }
#     ]
# }
```

### Create a Rule for Custom Events

```bash
# Match only Premium orders over $50
aws events put-rule \
  --name "premium-large-orders" \
  --event-bus-name "my-app-events" \
  --event-pattern '{
    "source": ["my-app.orders"],
    "detail-type": ["Order Placed"],
    "detail": {
      "orderType": ["Premium"],
      "amount": [{"numeric": [">=", 50]}]
    }
  }'
```

### Archive & Replay Events

```bash
# Create an archive (store events for replay later)
aws events create-archive \
  --archive-name "order-events-archive" \
  --source-arn "arn:aws:events:us-east-1:123456789012:event-bus/my-app-events" \
  --retention-days 30

# Replay events from the archive (e.g., re-process all events from last week)
aws events start-replay \
  --replay-name "replay-last-week" \
  --event-source-arn "arn:aws:events:us-east-1:123456789012:event-bus/my-app-events" \
  --destination '{"Arn": "arn:aws:events:us-east-1:123456789012:event-bus/my-app-events"}' \
  --event-start-time "2024-01-08T00:00:00Z" \
  --event-end-time "2024-01-15T00:00:00Z"
```

## 17. EventBridge vs SNS vs SQS — When to Use What?

### Decision Guide

```
Do you need to REACT to AWS service events (EC2, S3, RDS, etc.)?
  └── YES → EventBridge

Do you need to integrate with SaaS apps (Shopify, Zendesk)?
  └── YES → EventBridge

Do you need to BROADCAST a message to many subscribers?
  └── YES → SNS (or EventBridge for event-driven)

Do you need a MESSAGE QUEUE where consumers pull and process messages?
  └── YES → SQS

Do you need EXACTLY-ONCE processing in strict order?
  └── YES → SQS FIFO

Do you need HIGH THROUGHPUT (millions of messages/sec)?
  └── YES → SNS + SQS Fan-Out

Do you need to REPLAY past events?
  └── YES → EventBridge (with archive)

Do you need SCHEDULED triggers (cron jobs)?
  └── YES → EventBridge Scheduler
```

### Comparison Table

| Feature | SQS | SNS | EventBridge |
|---------|-----|-----|-------------|
| **Type** | Message Queue | Pub/Sub | Event Bus |
| **Model** | Pull | Push | Push |
| **Use for** | Task queues, buffering | Notifications, fan-out | Event-driven architecture |
| **Ordering** | FIFO available | FIFO available | Ordering per rule |
| **Filtering** | ❌ | Attribute-based | Content-based (powerful) |
| **AWS events** | Manual | Manual | Automatic |
| **Persistence** | ✅ (14 days) | ❌ | ✅ (Archive) |
| **Replay** | ❌ | ❌ | ✅ |
| **DLQ support** | ✅ | ✅ | ✅ |
| **Max message size** | 256 KB | 256 KB | 256 KB |

### Common Combinations

| Pattern | Services | Use Case |
|---------|----------|----------|
| **Fan-Out** | SNS → Multiple SQS | One event, multiple independent processors |
| **Event-Driven** | EventBridge → Lambda | React to AWS/app events |
| **Buffering** | API → SQS → Lambda | Handle traffic spikes |
| **Notification** | SNS → Email/SMS | Alert humans |
| **CQRS** | EventBridge → SQS → Different read models | Command-query separation |

## 19. Terraform Examples

### SQS Queue with DLQ

```hcl
# Dead Letter Queue
resource "aws_sqs_queue" "dlq" {
  name = "order-processing-dlq"

  # Keep failed messages for 14 days (maximum)
  message_retention_seconds = 1209600
}

# Main Queue
resource "aws_sqs_queue" "main" {
  name = "order-processing-queue"

  # How long a message stays invisible after being received
  visibility_timeout_seconds = 60

  # How long messages are retained if not consumed (4 days)
  message_retention_seconds = 345600

  # Enable long polling (20 seconds)
  receive_wait_time_seconds = 20

  # After 3 failed attempts, send to DLQ
  redrive_policy = jsonencode({
    deadLetterTargetArn = aws_sqs_queue.dlq.arn
    maxReceiveCount     = 3
  })

  tags = {
    Environment = "production"
    Team        = "backend"
  }
}
```

### SNS Topic with SQS Fan-Out

```hcl
# SNS Topic
resource "aws_sns_topic" "order_events" {
  name = "order-events"
}

# SQS Queue for Email Service
resource "aws_sqs_queue" "email_queue" {
  name = "email-processing-queue"
}

# SQS Queue for Warehouse Service
resource "aws_sqs_queue" "warehouse_queue" {
  name = "warehouse-processing-queue"
}

# Allow SNS to send messages to the email queue
resource "aws_sqs_queue_policy" "email_queue_policy" {
  queue_url = aws_sqs_queue.email_queue.id

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Effect    = "Allow"
        Principal = { Service = "sns.amazonaws.com" }
        Action    = "sqs:SendMessage"
        Resource  = aws_sqs_queue.email_queue.arn
        Condition = {
          ArnEquals = {
            "aws:SourceArn" = aws_sns_topic.order_events.arn
          }
        }
      }
    ]
  })
}

# Allow SNS to send messages to the warehouse queue
resource "aws_sqs_queue_policy" "warehouse_queue_policy" {
  queue_url = aws_sqs_queue.warehouse_queue.id

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Effect    = "Allow"
        Principal = { Service = "sns.amazonaws.com" }
        Action    = "sqs:SendMessage"
        Resource  = aws_sqs_queue.warehouse_queue.arn
        Condition = {
          ArnEquals = {
            "aws:SourceArn" = aws_sns_topic.order_events.arn
          }
        }
      }
    ]
  })
}

# Subscribe queues to the SNS topic
resource "aws_sns_topic_subscription" "email_sub" {
  topic_arn = aws_sns_topic.order_events.arn
  protocol  = "sqs"
  endpoint  = aws_sqs_queue.email_queue.arn
}

resource "aws_sns_topic_subscription" "warehouse_sub" {
  topic_arn = aws_sns_topic.order_events.arn
  protocol  = "sqs"
  endpoint  = aws_sqs_queue.warehouse_queue.arn

  # Only receive Premium orders
  filter_policy = jsonencode({
    OrderType = ["Premium"]
  })
}
```

### EventBridge Rule with Lambda Target

```hcl
# EventBridge Rule — trigger on EC2 terminations
resource "aws_cloudwatch_event_rule" "ec2_terminated" {
  name        = "ec2-instance-terminated"
  description = "Triggers when an EC2 instance is terminated"

    source      = ["aws.ec2"]
    detail-type = ["EC2 Instance State-change Notification"]
    detail = {
      state = ["terminated"]
    }
  })

# Lambda function as the target

resource "aws_lambda_permission" "allow_eventbridge" {
  statement_id  = "AllowEventBridgeInvoke"
  action        = "lambda:InvokeFunction"

  principal     = "events.amazonaws.com"
  source_arn    = aws_cloudwatch_event_rule.ec2_terminated.arn
}
```

## 20. Best Practices

### SQS Best Practices

| Practice | Why |
|----------|-----|
| Always use **Long Polling** (WaitTimeSeconds = 20) | Reduces API calls and costs |
| Always set up a **Dead Letter Queue** | Catch and investigate failed messages |
| Set **Visibility Timeout > Processing Time** | Prevent duplicate processing |
| Use **Batch Operations** (send/receive/delete up to 10 messages at once) | Better performance and lower costs |
| Use **Message Deduplication** (FIFO) or idempotent consumers (Standard) | Prevent duplicates |
| Enable **Server-Side Encryption (SSE)** | Protect message data at rest |
| Monitor **ApproximateNumberOfMessages** in CloudWatch | Know your queue depth |
| Monitor **ApproximateAgeOfOldestMessage** | Detect stuck/slow consumers |

### SNS Best Practices

| Practice | Why |
|----------|-----|
| Use **Message Filtering** | Reduce unnecessary message delivery |
| Pair with **SQS for durability** (Fan-Out pattern) | SNS alone doesn't persist messages |
| Enable **Server-Side Encryption** | Protect message data |
| Use **Dead Letter Queues** for Lambda/SQS subscriptions | Handle delivery failures |
| Avoid using **Email/SMS** for high-volume messages | They're rate-limited and expensive |

### EventBridge Best Practices

| Practice | Why |
|----------|-----|
| Use the **default event bus** for AWS events | It's already set up and receives events automatically |
| Use **custom event buses** for application events | Keeps things organized |
| Make event patterns **as specific as possible** | Better performance, less noise |
| Enable **Archive** for critical events | Replay capability for debugging and recovery |
| Use **Schema Registry** | Discover and validate event formats |
| Set up **DLQ on rules** | Catch delivery failures |

```
│        → Use for: ALWAYS. Every queue should have one.   │
```
