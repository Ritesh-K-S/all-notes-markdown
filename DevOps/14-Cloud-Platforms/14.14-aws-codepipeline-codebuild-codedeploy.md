## 1. What is CI/CD on AWS?

**CI/CD** stands for **Continuous Integration / Continuous Delivery (or Deployment)**. Let's break it down simply:

| Term | What It Means | Real-World Analogy |
|------|---------------|-------------------|
| **Continuous Integration (CI)** | Every time a developer pushes code, it's automatically built and tested | Like spell-checking every time you type a paragraph |
| **Continuous Delivery (CD)** | Code is always ready to be deployed (but humans approve) | Like having a package ready at the door — you decide when to ship it |
| **Continuous Deployment (CD)** | Code is automatically deployed to production after tests pass | Like auto-shipping the package the moment it's packed |

```
Developer pushes code
       │
       ▼
┌──────────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│    Source     │────▶│    Build     │────▶│    Test      │────▶│   Deploy     │
│  (CodeCommit │     │  (CodeBuild) │     │  (CodeBuild) │     │ (CodeDeploy) │
│   / GitHub)  │     │              │     │              │     │              │
└──────────────┘     └──────────────┘     └──────────────┘     └──────────────┘
         ◄────────── CodePipeline orchestrates the entire flow ──────────────►
```

## 4. AWS CodeBuild — Build & Test Your Code

### What is CodeBuild?

**CodeBuild** is a **fully managed build service** that compiles your source code, runs tests, and produces deployable artifacts (like JAR files, ZIP packages, or Docker images).

Think of it like this: You give CodeBuild your source code and a set of instructions (buildspec.yml), and it gives you back a built and tested artifact.

```
┌──────────────┐                                    ┌──────────────┐
│  Source Code  │                                    │   Artifact   │
│  (from Git)   │──▶ CodeBuild (build + test) ──▶   │  (ZIP, JAR,  │
│               │                                    │  Docker img) │
└──────────────┘                                    └──────────────┘
```

### Key Characteristics

| Feature | Details |
|---------|---------|
| **Fully managed** | No build servers to manage — AWS runs everything |
| **Pay per use** | You pay only for the build minutes you consume |
| **Scalable** | Runs builds in isolated containers — scales automatically |
| **Build environments** | Pre-built images for Java, Python, Node.js, Go, .NET, Ruby, Docker, etc. |
| **Custom environments** | Use your own Docker image as the build environment |
| **Caching** | Cache dependencies (node_modules, .m2, pip packages) to speed up builds |
| **Timeout** | Default 60 minutes, max 8 hours |
| **Concurrent builds** | Default 20 concurrent builds (can request increase) |

### How CodeBuild Works

```
Step 1: CodeBuild downloads your source code (from GitHub, S3, CodeCommit, etc.)
Step 2: It spins up a fresh Docker container with your chosen build environment
Step 3: It runs your build commands (defined in buildspec.yml)
Step 4: It uploads the output artifacts to S3
Step 5: The container is destroyed (clean slate every time!)
```

### Build Environment — Compute Types

| Compute Type | vCPUs | Memory | Use Case |
|-------------|-------|--------|----------|
| **BUILD_GENERAL1_SMALL** | 3 | 3 GB | Small projects, scripts |
| **BUILD_GENERAL1_MEDIUM** | 7 | 15 GB | Most applications |
| **BUILD_GENERAL1_LARGE** | 15 | 145 GB | Large Java/C++ projects |
| **BUILD_GENERAL1_2XLARGE** | 72 | 145 GB | Massive builds |
| **BUILD_LAMBDA_1GB** | 2 | 1 GB | Quick builds (Lambda-based) |
| **BUILD_LAMBDA_10GB** | 2 | 10 GB | Medium Lambda-based builds |

## 5. CodeBuild — buildspec.yml Explained

The `buildspec.yml` file is the **heart of CodeBuild**. It tells CodeBuild exactly what to do. It must be placed at the **root of your source code repository**.

### Full buildspec.yml Structure

```yaml
# buildspec.yml — This file lives in the ROOT of your repository
version: 0.2  # Always use 0.2 (the latest version)

# ---------- ENVIRONMENT VARIABLES ----------
env:
  # Plain text variables
  variables:
    NODE_ENV: "production"
    APP_NAME: "my-awesome-app"
  
  # Variables from AWS Parameter Store (secure)
  parameter-store:
    DB_PASSWORD: "/myapp/db/password"
  
  # Variables from AWS Secrets Manager (even more secure)
  secrets-manager:
    API_KEY: "myapp/api-key:api_key"

# ---------- BUILD PHASES ----------
phases:

  # Phase 1: Install dependencies
  install:
    runtime-versions:
      nodejs: 18          # Use Node.js 18
    commands:
      - echo "Installing dependencies..."
      - npm ci             # Clean install (faster & more reliable than npm install)

  # Phase 2: Before the build (optional)
  pre_build:
    commands:
      - echo "Running pre-build steps..."
      - echo "Logging into ECR..."
      - aws ecr get-login-password --region us-east-1 | docker login --username AWS --password-stdin 123456789012.dkr.ecr.us-east-1.amazonaws.com

  # Phase 3: The actual build
  build:
    commands:
      - echo "Building the application..."
      - npm run build
      - echo "Building Docker image..."
      - docker build -t my-app:latest .
      - docker tag my-app:latest 123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app:latest

  # Phase 4: After the build (optional)
  post_build:
    commands:
      - echo "Pushing Docker image to ECR..."
      - docker push 123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app:latest
      - echo "Build completed on $(date)"

# ---------- ARTIFACTS ----------
# What to save/output after the build
artifacts:
  files:
    - '**/*'                    # Include all files
  base-directory: build         # From the "build" directory
  discard-paths: no             # Keep the directory structure

# ---------- CACHE ----------
# Speed up future builds by caching dependencies
cache:
  paths:
    - 'node_modules/**/*'       # Cache node_modules
    - '/root/.npm/**/*'         # Cache npm cache
```

### Simpler Example — Python Application

```yaml
version: 0.2

phases:
  install:
    runtime-versions:
      python: 3.11
    commands:
      - pip install -r requirements.txt

  pre_build:
    commands:
      - echo "Running tests..."
      - python -m pytest tests/ --verbose

  build:
    commands:
      - echo "Building application..."
      - python setup.py bdist_wheel

  post_build:
    commands:
      - echo "Build completed!"

artifacts:
  files:
    - dist/*.whl
    - appspec.yml
    - scripts/**/*
```

### Understanding Build Phases

```
┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────────┐
│ install   │────▶│ pre_build │────▶│  build   │────▶│  post_build  │
│           │     │           │     │          │     │              │
│ Install   │     │ Login to  │     │ Compile  │     │ Push image,  │
│ runtime & │     │ registry, │     │ code,    │     │ cleanup,     │
│ deps      │     │ run tests │     │ build    │     │ notifications│
│           │     │           │     │ image    │     │              │
└──────────┘     └──────────┘     └──────────┘     └──────────────┘
                                                           │
                                                           ▼
                                                    ┌──────────────┐
                                                    │  Artifacts   │
                                                    │ uploaded to  │
                                                    │     S3       │
                                                    └──────────────┘
```

If any phase **fails**, the build stops immediately and is marked as **FAILED**.

## 6. CodeBuild — Hands-On with AWS CLI

### Create a CodeBuild Project

```bash
# Create a build project that pulls code from GitHub
aws codebuild create-project \
  --name "my-app-build" \
  --description "Build my Node.js application" \
  --source '{
    "type": "GITHUB",
    "location": "https://github.com/myorg/my-app.git",
    "buildspec": "buildspec.yml"
  }' \
  --artifacts '{
    "type": "S3",
    "location": "my-build-artifacts-bucket",
    "name": "my-app-build",
    "packaging": "ZIP"
  }' \
  --environment '{
    "type": "LINUX_CONTAINER",
    "image": "aws/codebuild/amazonlinux2-x86_64-standard:5.0",
    "computeType": "BUILD_GENERAL1_MEDIUM",
    "privilegedMode": true
  }' \
  --service-role "arn:aws:iam::123456789012:role/codebuild-service-role"

# "privilegedMode": true → Required if you're building Docker images inside CodeBuild
```

### Start a Build

```bash
# Trigger a build manually
aws codebuild start-build --project-name "my-app-build"

# Start a build with environment variable overrides
aws codebuild start-build \
  --project-name "my-app-build" \
  --environment-variables-override '[
    {"name": "DEPLOY_ENV", "value": "staging", "type": "PLAINTEXT"},
    {"name": "DB_PASSWORD", "value": "/myapp/db/password", "type": "PARAMETER_STORE"}
  ]'

# Output:
# {
#     "build": {
#         "id": "my-app-build:build-id-12345",
#         "buildNumber": 1,
#         "buildStatus": "IN_PROGRESS",
#         ...
#     }
# }
```

### Check Build Status

```bash
# Get details of a specific build
aws codebuild batch-get-builds --ids "my-app-build:build-id-12345"

# List recent builds for a project
aws codebuild list-builds-for-project \
  --project-name "my-app-build" \
  --sort-order DESCENDING

# Watch build logs (in CloudWatch Logs)
# Build logs are automatically sent to CloudWatch Logs group:
#   /aws/codebuild/my-app-build
```

### Stop a Build

```bash
aws codebuild stop-build --id "my-app-build:build-id-12345"
```

### Delete a Build Project

```bash
aws codebuild delete-project --name "my-app-build"
```

## 7. CodeBuild — Environment Variables & Secrets

### Types of Environment Variables

| Type | Storage | Use Case | Example |
|------|---------|----------|---------|
| **PLAINTEXT** | In the buildspec or project config | Non-sensitive values | `NODE_ENV=production` |
| **PARAMETER_STORE** | AWS Systems Manager Parameter Store | Sensitive config | Database passwords |
| **SECRETS_MANAGER** | AWS Secrets Manager | Highly sensitive secrets | API keys, tokens |

### Using Environment Variables in buildspec.yml

```yaml
version: 0.2

env:
  # Plain text (visible in build logs and project config)
  variables:
    APP_NAME: "my-app"
    REGION: "us-east-1"

  # From Parameter Store (secure, not shown in logs)
  parameter-store:
    DB_HOST: "/myapp/production/db-host"
    DB_PASSWORD: "/myapp/production/db-password"

  # From Secrets Manager (most secure)
  secrets-manager:
    # Format: SECRET_NAME:JSON_KEY
    GITHUB_TOKEN: "myapp/github:token"
    API_KEY: "myapp/api:key"

  # Exported variables (pass to next CodePipeline stage)
  exported-variables:
    - IMAGE_TAG
    - BUILD_ID

phases:
  build:
    commands:
      # Use them like normal environment variables
      - echo "Building $APP_NAME in $REGION"
      - echo "DB Host is $DB_HOST"

      # Create a variable that gets exported to the next pipeline stage
      - export IMAGE_TAG=$(echo $CODEBUILD_RESOLVED_SOURCE_VERSION | head -c 8)
      - export BUILD_ID=$CODEBUILD_BUILD_NUMBER
```

### Built-in Environment Variables

CodeBuild provides many useful variables automatically:

| Variable | Description |
|----------|-------------|
| `CODEBUILD_BUILD_ID` | Unique build ID |
| `CODEBUILD_BUILD_NUMBER` | Build number (increments) |
| `CODEBUILD_SOURCE_VERSION` | Git commit SHA or branch name |
| `CODEBUILD_RESOLVED_SOURCE_VERSION` | Full commit SHA |
| `CODEBUILD_SRC_DIR` | Directory where source code is downloaded |
| `CODEBUILD_BUILD_ARN` | ARN of the build |
| `CODEBUILD_WEBHOOK_HEAD_REF` | Branch for webhook-triggered builds |
| `CODEBUILD_WEBHOOK_EVENT` | Webhook event type (PUSH, PULL_REQUEST, etc.) |

## 8. CodeBuild — Caching & Optimization

### Why Cache?

Without caching, every build downloads all dependencies from scratch. That's slow and wasteful!

```
Without Cache:                    With Cache:
=============                     ==========
npm ci → 45 seconds               npm ci → 5 seconds (from cache!)
pip install → 30 seconds          pip install → 3 seconds (from cache!)
Total build: 5 minutes            Total build: 2 minutes
```

### Cache Types

| Cache Type | How It Works | Best For |
|-----------|--------------|----------|
| **S3 Cache** | Uploads cache to S3 after build, downloads before next build | Shared cache across builds |
| **Local Cache** | Keeps cache on the build machine (if same machine is reused) | Fast, but not guaranteed |

### S3 Caching in buildspec.yml

```yaml
version: 0.2

phases:
  install:
    commands:
      - npm ci   # This uses cached node_modules if available!

  build:
    commands:
      - npm run build

cache:
  paths:
    - 'node_modules/**/*'      # Cache node_modules
    - '/root/.npm/**/*'        # Cache npm's global cache
    - '/root/.cache/pip/**/*'  # Cache pip packages (for Python)
    - '/root/.m2/**/*'         # Cache Maven dependencies (for Java)
    - '/root/.gradle/**/*'     # Cache Gradle dependencies
```

### Build Optimization Tips

| Tip | Impact |
|-----|--------|
| Use caching for dependencies | 2-5x faster builds |
| Use `npm ci` instead of `npm install` | Faster and more reliable |
| Use multi-stage Docker builds | Smaller images, faster uploads |
| Use a larger compute type for big projects | Faster compilation |
| Enable concurrent builds | Run multiple builds in parallel |
| Use `--only=production` for npm | Skip dev dependencies in production builds |

## 9. AWS CodeDeploy — Deploy Your Application

### What is CodeDeploy?

**CodeDeploy** is a **fully managed deployment service** that automates deploying your application to various compute platforms. Think of it as the "delivery truck" that takes your built artifact and puts it on your servers.

```
Built Artifact (from CodeBuild)
       │
       ▼
┌──────────────┐
│  CodeDeploy  │  → Deploys to EC2, Lambda, or ECS
└──────┬───────┘
       │
       ├──▶ EC2 Instance 1 ✅
       ├──▶ EC2 Instance 2 ✅
       ├──▶ EC2 Instance 3 ✅
       └──▶ EC2 Instance 4 ✅
```

### Key Characteristics

| Feature | Details |
|---------|---------|
| **Fully managed** | No deployment servers to manage |
| **Multiple targets** | Deploy to EC2, On-premises, Lambda, ECS |
| **Deployment strategies** | In-Place, Blue/Green, Canary, Linear |
| **Automatic rollback** | Roll back if deployment fails or alarms trigger |
| **Health monitoring** | Integrates with CloudWatch Alarms and ELB health checks |
| **No downtime** | Blue/Green deployments ensure zero downtime |

### Compute Platforms

| Platform | How CodeDeploy Deploys |
|----------|----------------------|
| **EC2/On-Premises** | Installs a CodeDeploy Agent on instances → agent pulls & deploys code |
| **Lambda** | Shifts traffic between Lambda function versions (Canary/Linear) |
| **ECS** | Shifts traffic between ECS task sets (Blue/Green with ALB) |

### Key Components

| Component | What It Is |
|-----------|-----------|
| **Application** | A container/name for your deployment configuration |
| **Deployment Group** | A set of EC2 instances (or Lambda/ECS targets) to deploy to |
| **Deployment Configuration** | Rules for how to deploy (all at once, half at a time, one at a time) |
| **Revision** | The version of your app to deploy (from S3 or GitHub) |
| **AppSpec file** | The instruction file that tells CodeDeploy HOW to deploy |

## 10. CodeDeploy — Deployment Types

### In-Place Deployment (EC2 Only)

The application is stopped on each instance, the new version is installed, and the application is restarted. One instance at a time (or a batch).

```
Before:  [v1] [v1] [v1] [v1]    ← All instances running v1

Step 1:  [⬇️ ] [v1] [v1] [v1]    ← Instance 1 taken out, updated
Step 2:  [v2] [⬇️ ] [v1] [v1]    ← Instance 1 back, Instance 2 updating
Step 3:  [v2] [v2] [⬇️ ] [v1]    ← Instance 3 updating
Step 4:  [v2] [v2] [v2] [⬇️ ]    ← Instance 4 updating

After:   [v2] [v2] [v2] [v2]    ← All instances running v2
```

**Pros:** Simple, no extra cost (no duplicate infrastructure)
**Cons:** Brief downtime per instance, can't instantly rollback

### Blue/Green Deployment

A completely new set of instances is created with the new version. Traffic is switched from old (Blue) to new (Green). If anything goes wrong, switch back instantly.

```
Before:
  Blue  (current):  [v1] [v1] [v1] [v1]  ← Serving traffic
  Green (new):      (doesn't exist yet)

Step 1:
  Blue  (current):  [v1] [v1] [v1] [v1]  ← Still serving traffic
  Green (new):      [v2] [v2] [v2] [v2]  ← Created and tested

Step 2: Switch traffic!
  Blue  (old):      [v1] [v1] [v1] [v1]  ← Kept as backup (terminated later)
  Green (current):  [v2] [v2] [v2] [v2]  ← Now serving traffic ✅

Rollback? Just switch traffic back to Blue!
```

**Pros:** Zero downtime, instant rollback, new instances are tested before traffic switch
**Cons:** More expensive (double the infrastructure during deployment)

### Deployment Configurations

| Configuration | How It Works |
|--------------|-------------|
| **AllAtOnce** | Deploy to ALL instances at the same time (fastest, most risky) |
| **OneAtATime** | Deploy to one instance at a time (slowest, safest) |
| **HalfAtATime** | Deploy to half the instances, then the other half |
| **Canary10Percent5Minutes** | Deploy to 10% first, wait 5 min, then deploy to the rest |
| **Linear10PercentEvery1Minute** | Deploy to 10% every minute until all are done |
| **Custom** | You define the percentage and intervals |

### Choosing a Deployment Strategy

```
Is downtime acceptable?
  ├── YES → In-Place (AllAtOnce or HalfAtATime)
  └── NO → Blue/Green

Do you need instant rollback?
  ├── YES → Blue/Green
  └── NO → In-Place (OneAtATime)

Is cost a concern?
  ├── YES → In-Place (no extra infrastructure)
  └── NO → Blue/Green (safest)

Do you want gradual traffic shift?
  └── YES → Canary or Linear (especially for Lambda/ECS)
```

## 11. CodeDeploy — appspec.yml Explained

The `appspec.yml` (or `appspec.yaml`) file tells CodeDeploy **what to deploy and how**. It must be in the **root of your revision** (the artifact).

### For EC2/On-Premises Deployments

```yaml
# appspec.yml — for EC2 deployments
version: 0.0  # Always 0.0 for EC2

os: linux      # or "windows"

# ---------- FILES ----------
# Which files to copy and where
files:
  # Copy everything from the "app" folder in the artifact to /var/www/myapp on the server
  - source: /app
    destination: /var/www/myapp

  # Copy the config file
  - source: /config/app.conf
    destination: /etc/myapp/

# ---------- PERMISSIONS ----------
# Set file permissions after copying (Linux only)
permissions:
  - object: /var/www/myapp
    owner: www-data
    group: www-data
    mode: 755
    type:
      - directory
  - object: /var/www/myapp
    owner: www-data
    group: www-data
    mode: 644
    pattern: "**/*.js"
    type:
      - file

# ---------- HOOKS (Lifecycle Events) ----------
# Scripts to run at different stages of the deployment
hooks:
  # Before the old application is stopped
  BeforeInstall:
    - location: scripts/before_install.sh
      timeout: 300      # seconds
      runas: root

  # After files are copied
  AfterInstall:
    - location: scripts/after_install.sh
      timeout: 300
      runas: root

  # Start the application
  ApplicationStart:
    - location: scripts/start_server.sh
      timeout: 300
      runas: root

  # Verify the deployment is healthy
  ValidateService:
    - location: scripts/validate.sh
      timeout: 300
      runas: root
```

### Example Deployment Scripts

**scripts/before_install.sh** — Prepare the server

```bash
#!/bin/bash
set -e

echo "=== Before Install ==="

# Stop the existing application
if systemctl is-active --quiet myapp; then
    echo "Stopping existing application..."
    systemctl stop myapp
fi

# Clean up old files
echo "Cleaning up old deployment..."
rm -rf /var/www/myapp/*

# Install system dependencies if needed
echo "Installing system packages..."
yum install -y nodejs npm
```

**scripts/after_install.sh** — Install app dependencies

```bash
#!/bin/bash
set -e

echo "=== After Install ==="

# Navigate to the application directory
cd /var/www/myapp

# Install application dependencies
echo "Installing Node.js dependencies..."
npm ci --production

# Set proper permissions
echo "Setting permissions..."
chown -R www-data:www-data /var/www/myapp
```

**scripts/start_server.sh** — Start the application

```bash
#!/bin/bash
set -e

echo "=== Starting Application ==="

# Start the application using systemd
systemctl start myapp
systemctl enable myapp

echo "Application started successfully!"
```

**scripts/validate.sh** — Verify the deployment

```bash
#!/bin/bash
set -e

echo "=== Validating Deployment ==="

# Wait for the application to be ready
sleep 5

# Check if the application is responding
HTTP_STATUS=$(curl -s -o /dev/null -w "%{http_code}" http://localhost:3000/health)

if [ "$HTTP_STATUS" -eq 200 ]; then
    echo "✅ Application is healthy! (HTTP $HTTP_STATUS)"
    exit 0
else
    echo "❌ Application is NOT healthy! (HTTP $HTTP_STATUS)"
    exit 1  # This will cause the deployment to fail and trigger rollback
fi
```

### EC2/On-Premises Lifecycle (In-Place Deployment)

```
┌─────────────────────────────────────────────────────────┐
│              Deployment Lifecycle Hooks                   │
│                                                          │
│  ┌──────────────────┐                                    │
│  │ ApplicationStop   │  ← Stop the running application   │
│  └────────┬─────────┘                                    │
│           ▼                                              │
│  ┌──────────────────┐                                    │
│  │ DownloadBundle    │  ← CodeDeploy downloads the new   │
│  └────────┬─────────┘    revision (you can't hook this)  │
│           ▼                                              │
│  ┌──────────────────┐                                    │
│  │ BeforeInstall     │  ← Pre-install tasks (backup,     │
│  └────────┬─────────┘    decrypt, cleanup)               │
│           ▼                                              │
│  ┌──────────────────┐                                    │
│  │ Install           │  ← CodeDeploy copies files        │
│  └────────┬─────────┘    (you can't hook this)           │
│           ▼                                              │
│  ┌──────────────────┐                                    │
│  │ AfterInstall      │  ← Post-install (set permissions, │
│  └────────┬─────────┘    configure app, install deps)    │
│           ▼                                              │
│  ┌──────────────────┐                                    │
│  │ ApplicationStart  │  ← Start the application          │
│  └────────┬─────────┘                                    │
│           ▼                                              │
│  ┌──────────────────┐                                    │
│  │ ValidateService   │  ← Verify the deployment works    │
│  └──────────────────┘    (health check, smoke test)      │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### Blue/Green Deployment — Additional Hooks

```
Blue/Green adds these extra hooks:

  BeforeBlockTraffic   ← Before taking the instance out of the load balancer
  BlockTraffic         ← Remove instance from ELB (automatic)
  AfterBlockTraffic    ← After instance is removed from ELB

  ... (normal lifecycle hooks happen here) ...

  BeforeAllowTraffic   ← Before adding instance back to the load balancer
  AllowTraffic         ← Add instance to ELB (automatic)
  AfterAllowTraffic    ← After instance is added back to ELB
```

## 14. AWS CodePipeline — Orchestrate Everything

### What is CodePipeline?

**CodePipeline** is a **fully managed CI/CD orchestration service**. It ties together all the other services (source, build, test, deploy) into a **pipeline** — an automated workflow that runs every time code changes.

Think of it as the **assembly line manager** that says "First get the code, then build it, then test it, then deploy it — and if anything fails, stop everything!"

```
┌─────────────────────────────────────────────────────────────────┐
│                     CodePipeline                                 │
│                                                                  │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐     │
│  │  Source   │──▶│  Build   │──▶│  Test    │──▶│  Deploy  │     │
│  │          │   │          │   │          │   │          │     │
│  │ GitHub   │   │CodeBuild │   │CodeBuild │   │CodeDeploy│     │
│  │ S3       │   │ Jenkins  │   │ 3rd party│   │ ECS      │     │
│  │ ECR      │   │          │   │          │   │ Lambda   │     │
│  └──────────┘   └──────────┘   └──────────┘   └──────────┘     │
│                                                                  │
│  ← Automatic trigger on code push                                │
│  ← Automatic rollback on failure                                 │
│  ← Manual approval gates between stages                          │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### Key Characteristics

| Feature | Details |
|---------|---------|
| **Fully managed** | No servers to run |
| **Visual pipeline** | Nice UI in AWS Console showing pipeline stages |
| **Automatic triggers** | Starts automatically when source code changes |
| **Manual approvals** | Add approval gates (e.g., "Manager must approve before production") |
| **Multiple sources** | GitHub, Bitbucket, CodeCommit, S3, ECR |
| **Extensible** | Integrates with CodeBuild, CodeDeploy, Jenkins, Lambda, ECS, CloudFormation |
| **Parallel actions** | Run multiple actions in parallel within a stage |
| **Cross-region** | Deploy to multiple AWS regions |
| **Cross-account** | Deploy to different AWS accounts |

## 15. CodePipeline — Key Concepts

### Pipeline Structure

```
Pipeline
  │
  ├── Stage 1: Source
  │     └── Action: GitHub Source
  │
  ├── Stage 2: Build
  │     └── Action: CodeBuild
  │
  ├── Stage 3: Test
  │     ├── Action: Unit Tests (CodeBuild)     ← These run in parallel!
  │     └── Action: Integration Tests (CodeBuild)
  │
  ├── Stage 4: Approval
  │     └── Action: Manual Approval (SNS notification)
  │
  └── Stage 5: Deploy
        ├── Action: Deploy to Staging (CodeDeploy)
        └── Action: Deploy to Production (CodeDeploy)
```

### Key Terms

| Term | Meaning |
|------|---------|
| **Pipeline** | The entire CI/CD workflow from source to deploy |
| **Stage** | A logical group of actions (Source, Build, Test, Deploy) |
| **Action** | A specific task within a stage (e.g., run CodeBuild, deploy with CodeDeploy) |
| **Artifact** | Files passed between stages (source code, built binaries, etc.) |
| **Transition** | The connection between stages (can be enabled/disabled) |
| **Trigger** | What starts the pipeline (code push, manual, scheduled, etc.) |

### Artifact Flow

Artifacts are how stages pass data to each other:

```
Source Stage
  └── Output: "SourceArtifact" (your source code as a ZIP)
        │
        ▼
Build Stage
  ├── Input: "SourceArtifact"
  └── Output: "BuildArtifact" (compiled code, Docker image tag, etc.)
        │
        ▼
Deploy Stage
  └── Input: "BuildArtifact"
```

CodePipeline stores artifacts in an **S3 bucket** (it creates one automatically, or you can specify your own).

### Action Types

| Category | Action Provider | What It Does |
|----------|----------------|-------------|
| **Source** | GitHub, Bitbucket, CodeCommit, S3, ECR | Fetches source code |
| **Build** | CodeBuild, Jenkins | Builds & tests code |
| **Test** | CodeBuild, Device Farm, 3rd party | Runs tests |
| **Deploy** | CodeDeploy, ECS, S3, CloudFormation, Lambda | Deploys application |
| **Approval** | Manual Approval | Human approval gate |
| **Invoke** | Lambda, Step Functions | Run custom logic |

```bash
  --pipeline-name "my-app-pipeline" \
  --stage-name "Approval" \
  --action-name "ManualApproval" \

# Disable a transition (pause pipeline between stages)
aws codepipeline disable-stage-transition \
  --pipeline-name "my-app-pipeline" \
  --stage-name "Deploy" \
  --transition-type "Inbound" \
  --reason "Maintenance window — pausing deployments"

aws codepipeline enable-stage-transition \
  --pipeline-name "my-app-pipeline" \
  --stage-name "Deploy" \
  --transition-type "Inbound"

# Delete a pipeline
aws codepipeline delete-pipeline --name "my-app-pipeline"
```

## 17. Full Pipeline Example — GitHub → CodeBuild → CodeDeploy

Here's a complete, real-world example of a pipeline for a Node.js application.

### Project Structure

```
my-app/
├── src/
│   └── app.js
├── tests/
│   └── app.test.js
├── scripts/
│   ├── before_install.sh
│   ├── after_install.sh
│   ├── start_server.sh
│   └── validate.sh
├── buildspec.yml        ← CodeBuild instructions
├── appspec.yml          ← CodeDeploy instructions
├── package.json
└── Dockerfile
```

### buildspec.yml

```yaml
version: 0.2

phases:
  install:
    runtime-versions:
      nodejs: 18
    commands:
      - echo "Installing dependencies..."
      - npm ci

  pre_build:
    commands:
      - echo "Running tests..."
      - npm test
      - echo "Running linting..."
      - npm run lint

  build:
    commands:
      - echo "Building application..."
      - npm run build
      - echo "Build complete!"

  post_build:
    commands:
      - echo "Creating deployment package..."

artifacts:
  files:
    - 'build/**/*'
    - 'package.json'
    - 'package-lock.json'
    - 'appspec.yml'
    - 'scripts/**/*'
  discard-paths: no

cache:
  paths:
    - 'node_modules/**/*'
```

### appspec.yml

```yaml
version: 0.0
os: linux

files:
  - source: /
    destination: /var/www/myapp

permissions:
  - object: /var/www/myapp
    owner: ec2-user
    group: ec2-user
    mode: 755

hooks:
  BeforeInstall:
    - location: scripts/before_install.sh
      timeout: 300
      runas: root
  AfterInstall:
    - location: scripts/after_install.sh
      timeout: 300
      runas: root
  ApplicationStart:
    - location: scripts/start_server.sh
      timeout: 300
      runas: root
  ValidateService:
    - location: scripts/validate.sh
      timeout: 300
      runas: root
```

### The Flow

```
1. Developer pushes code to GitHub (main branch)
      │
      ▼
2. CodePipeline detects the change (via webhook/polling)
      │
      ▼
3. Source Stage: Downloads code from GitHub → SourceArtifact
      │
      ▼
4. Build Stage: CodeBuild runs buildspec.yml
   - Installs Node.js 18
   - Runs npm ci (install deps)
   - Runs tests
   - Runs linting
   - Builds the app (npm run build)
   - Packages everything → BuildArtifact
      │
      ▼
5. (Optional) Approval Stage: SNS sends email to manager
   - Manager clicks "Approve" in AWS Console
      │
      ▼
6. Deploy Stage: CodeDeploy deploys to EC2 instances
   - Runs before_install.sh (stop old app, cleanup)
   - Copies files to /var/www/myapp
   - Runs after_install.sh (npm ci --production)
   - Runs start_server.sh (systemctl start myapp)
   - Runs validate.sh (curl health check)
      │
      ▼
7. Done! 🎉 Application is live with the new version.
   If validation fails → Auto-rollback to previous version.
```

```hcl
        DeploymentGroupName = aws_codedeploy_deployment_group.production.deployment_group_name
```

## 19. CodePipeline vs Jenkins vs GitHub Actions

| Feature | CodePipeline | Jenkins | GitHub Actions |
|---------|-------------|---------|----------------|
| **Hosting** | Fully managed (AWS) | Self-hosted (or managed) | Fully managed (GitHub) |
| **Setup** | Easy (AWS Console/CLI) | Medium (install + plugins) | Easy (YAML in repo) |
| **Maintenance** | None | High (updates, plugins, scaling) | None |
| **Cost** | $1/pipeline/month + build costs | Free (but server costs) | Free tier + per-minute |
| **Build service** | CodeBuild (separate) | Built-in | Built-in (runners) |
| **Extensibility** | Limited (AWS services) | 1800+ plugins | 15000+ marketplace actions |
| **AWS integration** | Best (native) | Good (with plugins) | Good (with actions) |
| **Multi-cloud** | AWS only | Any cloud | Any cloud |
| **Pipeline as Code** | JSON/YAML or Console | Jenkinsfile (Groovy) | YAML in repo |
| **Visual editor** | ✅ (Console) | ✅ (Blue Ocean) | ❌ |
| **Self-hosted runners** | ❌ | ✅ | ✅ |

### When to Use What?

| Use Case | Best Choice |
|----------|------------|
| All-in on AWS | **CodePipeline** |
| Complex pipelines with many plugins | **Jenkins** |
| GitHub-based repos, simple pipelines | **GitHub Actions** |
| Need visual pipeline editor | **CodePipeline** |
| Multi-cloud deployments | **Jenkins** or **GitHub Actions** |
| Small team, simple workflow | **GitHub Actions** |
| Enterprise with strict AWS compliance | **CodePipeline** |

## 20. Best Practices

### CodeBuild Best Practices

| Practice | Why |
|----------|-----|
| Use **buildspec.yml in the repo** (not inline) | Version-controlled, reviewable |
| Enable **S3 caching** for dependencies | 2-5x faster builds |
| Use **Parameter Store/Secrets Manager** for secrets | Never hardcode secrets in buildspec |
| Use **privileged mode only when needed** (Docker builds) | Follows least privilege |
| Set **appropriate timeout** | Don't let stuck builds run forever |
| Use **exported variables** to pass data between stages | Clean data flow |
| Use **multi-stage Docker builds** | Smaller, faster images |

### CodeDeploy Best Practices

| Practice | Why |
|----------|-----|
| Always enable **auto-rollback** | Automatic recovery from failed deployments |
| Use **Blue/Green for production** | Zero downtime, instant rollback |
| Write proper **validation scripts** | Catch issues before serving traffic |
| Use **deployment groups for environments** | Separate staging vs production |
| Tag your EC2 instances consistently | Makes deployment groups easier |
| Keep **appspec.yml and scripts in the repo** | Version-controlled deployment process |

### CodePipeline Best Practices

| Practice | Why |
|----------|-----|
| Use **manual approval** before production | Human safety net |
| Set up **SNS notifications** | Know when pipeline fails |
| Use **separate pipelines** for different environments | Don't mix staging and production |
| Enable **CloudWatch alarms** for deployment health | Auto-rollback on metrics |
| Store pipeline definition **in version control** (Terraform/CloudFormation) | Reproducible pipelines |
| Use **parallel actions** where possible | Faster pipelines |
| Keep **artifact bucket versioning** enabled | Audit trail and rollback |
