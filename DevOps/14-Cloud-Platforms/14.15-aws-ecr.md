## 1. What is a Container Registry?

Before we talk about ECR, let's understand what a **Container Registry** is.

When you build a Docker image, it exists only on your local machine. But what if you want to:
- Deploy it to a server?
- Share it with your team?
- Use it in CI/CD pipelines?
- Pull it on Kubernetes?

You need a place to **store** and **distribute** your Docker images. That place is a **Container Registry**.

```
Think of it like this:

  GitHub       → stores your SOURCE CODE      → others can "git clone" it
  Docker Hub   → stores your DOCKER IMAGES    → others can "docker pull" it
  AWS ECR      → stores your DOCKER IMAGES    → your AWS services can "docker pull" it
```

### How a Registry Works

```
Developer's Machine                    Container Registry                    Production Server
┌──────────────────┐                  ┌──────────────────┐                  ┌──────────────────┐
│                  │  docker push     │                  │  docker pull     │                  │
│  Build Docker    │ ──────────────▶  │  Store & Manage  │ ──────────────▶  │  Run Container   │
│  Image locally   │                  │  Docker Images   │                  │  from image      │
│                  │                  │                  │                  │                  │
└──────────────────┘                  └──────────────────┘                  └──────────────────┘
```

## 2. What is AWS ECR?

**Amazon Elastic Container Registry (ECR)** is a **fully managed Docker container registry** provided by AWS. It's like Docker Hub, but it runs entirely within your AWS account with tight integration into the AWS ecosystem.

### Key Characteristics

| Feature | Details |
|---------|---------|
| **Fully managed** | No registry servers to manage, scale, or patch |
| **Highly available** | Images stored across multiple Availability Zones |
| **Secure** | Encryption at rest (AES-256), in transit (HTTPS), IAM-based access control |
| **Integrated** | Works seamlessly with ECS, EKS, Lambda, CodeBuild, and more |
| **Image scanning** | Built-in vulnerability scanning (powered by Amazon Inspector) |
| **Lifecycle policies** | Automatically clean up old/unused images |
| **Replication** | Cross-region and cross-account replication |
| **Pull-through cache** | Cache images from public registries (Docker Hub, GitHub, Quay) |
| **OCI compatible** | Supports Docker images and OCI (Open Container Initiative) artifacts |
| **Image size** | Each image layer can be up to 42 GB; no total limit on image size |

### Pricing (Simple)

| Component | Cost |
|-----------|------|
| **Storage** | $0.10 per GB per month |
| **Data transfer (out)** | $0.09 per GB (same as S3; free within same region to AWS services) |
| **Data transfer (in)** | Free (pushing images is free!) |
| **ECR Public** | 50 GB free storage + 500 GB/month free bandwidth |

> For most teams, ECR costs only a few dollars per month. Much cheaper than running your own registry!

## 3. ECR vs Docker Hub vs Other Registries

| Feature | AWS ECR | Docker Hub | GitHub Container Registry | Harbor |
|---------|---------|------------|--------------------------|--------|
| **Type** | Managed (AWS) | Managed (Docker) | Managed (GitHub) | Self-hosted |
| **Private repos** | Unlimited | 1 free, then paid | Unlimited (free) | Unlimited |
| **Public repos** | ✅ (ECR Public) | ✅ | ✅ | ✅ |
| **Pull rate limits** | No limits (within AWS) | 100 pulls/6hrs (free) | 1000 pulls/day | No limits |
| **Image scanning** | ✅ (built-in) | ✅ (paid) | ❌ | ✅ (Trivy) |
| **Lifecycle policies** | ✅ | ❌ | ❌ | ✅ |
| **AWS integration** | Best | Good | Good | Manual |
| **IAM access control** | ✅ | ❌ | ❌ | LDAP/OIDC |
| **Encryption** | KMS + AES-256 | ❌ | ❌ | ✅ |
| **Replication** | ✅ (built-in) | ❌ | ❌ | ✅ |
| **Cost** | Pay per use | Free tier + $5-25/mo | Free | Free (self-hosted) |

### When to Use ECR

| Scenario | Best Choice |
|----------|------------|
| You're running containers on AWS (ECS, EKS, Lambda) | **ECR** (best integration, no pull limits) |
| Open-source project, public images | **Docker Hub** or **ECR Public** |
| GitHub-based CI/CD | **GitHub Container Registry** |
| Multi-cloud or on-premises | **Harbor** (self-hosted) |
| You want zero config, just works with AWS | **ECR** |

## 4. ECR Architecture — How It Works

### Components

```
AWS Account
├── ECR Registry (one per account per region)
│   ├── Repository: "my-app"
│   │   ├── Image: my-app:latest    (tag: latest)
│   │   ├── Image: my-app:v1.0.0    (tag: v1.0.0)
│   │   ├── Image: my-app:v1.1.0    (tag: v1.1.0)
│   │   └── Image: my-app:abc1234   (tag: commit SHA)
│   │
│   ├── Repository: "my-api"
│   │   ├── Image: my-api:latest
│   │   └── Image: my-api:v2.0.0
│   │
│   └── Repository: "nginx-custom"
│       └── Image: nginx-custom:latest
```

### Understanding the Image URI

```
123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app:v1.0.0
│              │         │                    │       │
│              │         │                    │       └── Tag (version)
│              │         │                    └── Repository name
│              │         └── Region
│              └── ECR service identifier
└── AWS Account ID
```

Let's break this down:
- `123456789012` — Your AWS Account ID
- `dkr.ecr` — ECR service (dkr = Docker)
- `us-east-1` — The AWS region where your registry lives
- `amazonaws.com` — AWS domain
- `my-app` — Your repository name (like a project folder)
- `v1.0.0` — The image tag (version label)

### Image vs Tag vs Digest

| Concept | What It Is | Example |
|---------|-----------|---------|
| **Repository** | A collection of related images (like a project) | `my-app` |
| **Image** | A specific Docker image identified by digest | `sha256:abc123...` |
| **Tag** | A human-readable label pointing to an image | `latest`, `v1.0.0`, `abc1234` |
| **Digest** | An immutable, unique SHA256 hash of the image | `sha256:a3ed95c...` |

> **Important:** Tags are **mutable** — you can move `latest` to point to a different image. Digests are **immutable** — they always point to the same image.

```
Repository: my-app
├── Tag "latest"   → points to → Image sha256:abc123...
├── Tag "v1.0.0"   → points to → Image sha256:abc123...  (same image, two tags!)
├── Tag "v0.9.0"   → points to → Image sha256:def456...
└── (untagged)     → exists as → Image sha256:ghi789...  (old image that lost its tag)
```

## 5. ECR — Public vs Private Repositories

### Private Registry (Default)

- Only accessible within your AWS account (or accounts you explicitly grant access to)
- Requires **IAM authentication** to push/pull
- URL format: `<account-id>.dkr.ecr.<region>.amazonaws.com/<repo-name>`
- Use for: Your application images, internal tools, anything proprietary

### Public Registry (ECR Public)

- Accessible to everyone on the internet (no authentication needed to pull)
- Hosted on **Amazon ECR Public Gallery** (gallery.ecr.aws)
- URL format: `public.ecr.aws/<alias>/<repo-name>`
- Use for: Open-source projects, shared tools, public Docker images
- Free tier: 50 GB storage + 500 GB bandwidth/month

```
Private: 123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app:v1.0.0
Public:  public.ecr.aws/mycompany/my-app:v1.0.0
```

## 6. Setting Up ECR — Hands-On with AWS CLI

### Create a Private Repository

```bash
# Create a repository for your application
aws ecr create-repository \
  --repository-name my-app \
  --image-scanning-configuration scanOnPush=true \
  --encryption-configuration encryptionType=AES256 \
  --region us-east-1

# Output:
# {
#     "repository": {
#         "repositoryArn": "arn:aws:ecr:us-east-1:123456789012:repository/my-app",
#         "registryId": "123456789012",
#         "repositoryName": "my-app",
#         "repositoryUri": "123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app",
#         "createdAt": "2024-01-15T10:00:00+00:00",
#         "imageTagMutability": "MUTABLE",
#         "imageScanningConfiguration": {
#             "scanOnPush": true    ← Images will be scanned automatically when pushed!
#         },
#         "encryptionConfiguration": {
#             "encryptionType": "AES256"
#         }
#     }
# }
```

### Create a Repository with Immutable Tags

```bash
# Immutable tags = once you push my-app:v1.0.0, nobody can overwrite it
# This is GREAT for production — ensures tag v1.0.0 always means the same image
aws ecr create-repository \
  --repository-name my-app-prod \
  --image-tag-mutability IMMUTABLE \
  --image-scanning-configuration scanOnPush=true
```

### List Repositories

```bash
# List all repositories in your registry
aws ecr describe-repositories

# Output:
# {
#     "repositories": [
#         {
#             "repositoryName": "my-app",
#             "repositoryUri": "123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app",
#             "imageScanningConfiguration": { "scanOnPush": true },
#             "imageTagMutability": "MUTABLE"
#         }
#     ]
# }
```

### Delete a Repository

```bash
# Delete a repository (must be empty, or use --force)
aws ecr delete-repository --repository-name my-app

# Force delete (deletes all images inside too!)
aws ecr delete-repository --repository-name my-app --force
```

## 7. Building, Tagging & Pushing Docker Images to ECR

This is the most common workflow. Let's go step by step.

### Step 1: Authenticate Docker to ECR

Before you can push images, you need to **log in** to ECR. ECR uses temporary tokens (valid for 12 hours).

```bash
# Login to ECR (this pipes the password to docker login)
aws ecr get-login-password --region us-east-1 | \
  docker login --username AWS --password-stdin 123456789012.dkr.ecr.us-east-1.amazonaws.com

# Output: Login Succeeded

# What's happening here:
# 1. aws ecr get-login-password → Gets a temporary auth token from AWS
# 2. | → Pipes it to the next command
# 3. docker login → Authenticates Docker with that token
# --username AWS → Always "AWS" for ECR
# --password-stdin → Read password from pipe (secure, not on command line)
```

### Step 2: Build Your Docker Image

```bash
# You have a Dockerfile in your project directory
# Let's say it's a simple Node.js app

# Build the image locally
docker build -t my-app .

# You can also build with a specific Dockerfile
docker build -f Dockerfile.production -t my-app .
```

Example **Dockerfile**:

```dockerfile
# Simple Node.js Dockerfile
FROM node:18-alpine

WORKDIR /app

# Copy package files first (for better caching)
COPY package.json package-lock.json ./

# Install dependencies
RUN npm ci --production

# Copy application code
COPY . .

# Expose the port
EXPOSE 3000

# Health check
HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \
  CMD wget --no-verbose --tries=1 --spider http://localhost:3000/health || exit 1

# Start the application
CMD ["node", "src/app.js"]
```

### Step 3: Tag the Image for ECR

Docker needs to know WHERE to push the image. You do this by **tagging** the image with the ECR repository URI.

```bash
# Tag format: <ECR_URI>:<TAG>

# Tag with "latest"
docker tag my-app:latest 123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app:latest

# Tag with a version number
docker tag my-app:latest 123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app:v1.0.0

# Tag with the Git commit SHA (great for traceability!)
GIT_SHA=$(git rev-parse --short HEAD)
docker tag my-app:latest 123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app:${GIT_SHA}

# You can (and should!) apply multiple tags to the same image
# The image is stored ONCE, tags are just pointers (no extra storage cost)
```

### Step 4: Push the Image to ECR

```bash
# Push all tags
docker push 123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app:latest
docker push 123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app:v1.0.0
docker push 123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app:${GIT_SHA}

# Output:
# The push refers to repository [123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app]
# abc123: Pushed
# def456: Pushed
# v1.0.0: digest: sha256:... size: 1234
```

### Complete Script (All Steps Together)

```bash
#!/bin/bash
# deploy-to-ecr.sh — Build and push Docker image to ECR

set -e  # Exit on any error

# Configuration
AWS_ACCOUNT_ID="123456789012"
AWS_REGION="us-east-1"
REPO_NAME="my-app"
ECR_URI="${AWS_ACCOUNT_ID}.dkr.ecr.${AWS_REGION}.amazonaws.com/${REPO_NAME}"

# Get version from git
GIT_SHA=$(git rev-parse --short HEAD)
VERSION=$(cat VERSION 2>/dev/null || echo "dev")

echo "=== Step 1: Login to ECR ==="
aws ecr get-login-password --region ${AWS_REGION} | \
  docker login --username AWS --password-stdin ${AWS_ACCOUNT_ID}.dkr.ecr.${AWS_REGION}.amazonaws.com

echo "=== Step 2: Build Docker Image ==="
docker build -t ${REPO_NAME}:latest .

echo "=== Step 3: Tag Image ==="
docker tag ${REPO_NAME}:latest ${ECR_URI}:latest
docker tag ${REPO_NAME}:latest ${ECR_URI}:${VERSION}
docker tag ${REPO_NAME}:latest ${ECR_URI}:${GIT_SHA}

echo "=== Step 4: Push Image to ECR ==="
docker push ${ECR_URI}:latest
docker push ${ECR_URI}:${VERSION}
docker push ${ECR_URI}:${GIT_SHA}

echo "=== Done! Image pushed to: ${ECR_URI} ==="
echo "Tags: latest, ${VERSION}, ${GIT_SHA}"
```

## 8. Pulling Images from ECR

### Pull from Same AWS Account

```bash
# First, login to ECR
aws ecr get-login-password --region us-east-1 | \
  docker login --username AWS --password-stdin 123456789012.dkr.ecr.us-east-1.amazonaws.com

# Pull the image
docker pull 123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app:v1.0.0

# Run the image
docker run -d -p 3000:3000 123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app:v1.0.0
```

### Pull from ECS (Automatic)

When running on **ECS**, you don't need to manually login. ECS automatically authenticates with ECR using the **Task Execution Role**. Just specify the ECR URI in your task definition:

```json
{
  "containerDefinitions": [
    {
      "name": "my-app",
      "image": "123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app:v1.0.0",
      "portMappings": [
        {
          "containerPort": 3000
        }
      ]
    }
  ]
}
```

### Pull from EKS (Kubernetes)

EKS nodes can pull from ECR in the **same account and region** automatically (if the node IAM role has the right permissions). For cross-account or cross-region, you need to set up credentials.

```yaml
# Kubernetes deployment using ECR image
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
        - name: my-app
          image: 123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app:v1.0.0
          ports:
            - containerPort: 3000
```

### List Images in a Repository

```bash
# List all images in a repository
aws ecr list-images --repository-name my-app

# Output:
# {
#     "imageIds": [
#         { "imageDigest": "sha256:abc123...", "imageTag": "latest" },
#         { "imageDigest": "sha256:abc123...", "imageTag": "v1.0.0" },
#         { "imageDigest": "sha256:def456...", "imageTag": "v0.9.0" }
#     ]
# }

# Get detailed info about images
aws ecr describe-images --repository-name my-app

# Output includes: image size, push date, scan status, tags, digest
```

### Delete an Image

```bash
# Delete a specific image by tag
aws ecr batch-delete-image \
  --repository-name my-app \
  --image-ids imageTag=v0.9.0

# Delete by digest (removes ALL tags pointing to that image)
aws ecr batch-delete-image \
  --repository-name my-app \
  --image-ids imageDigest=sha256:def456...
```

## 9. ECR Image Tags & Best Practices

### Tagging Strategies

| Strategy | Tag Example | Pros | Cons |
|----------|-------------|------|------|
| **latest** | `my-app:latest` | Simple | Can't tell which version is running |
| **Semantic Version** | `my-app:v1.2.3` | Clear versioning | Must manage versions manually |
| **Git SHA** | `my-app:abc1234` | Exact traceability to commit | Not human-friendly |
| **Build Number** | `my-app:build-42` | Sequential, easy to understand | Tied to CI system |
| **Date** | `my-app:2024-01-15` | Know when it was built | Multiple builds per day? |
| **Combined** | `my-app:v1.2.3-abc1234` | Best of both worlds | Longer tag names |

### Recommended Approach

```bash
# Apply MULTIPLE tags to every image:

# 1. Semantic version (for humans)
docker tag my-app 123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app:v1.2.3

# 2. Git SHA (for traceability)
docker tag my-app 123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app:abc1234

# 3. "latest" (for convenience, but NEVER use in production!)
docker tag my-app 123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app:latest

# In your Kubernetes/ECS deployment, always use the SPECIFIC tag:
# GOOD:  image: my-app:v1.2.3
# BAD:   image: my-app:latest  ← Which version is this?? Nobody knows!
```

### Tag Immutability

```bash
# Enable immutable tags on a repository
# Once you push my-app:v1.0.0, nobody can overwrite it with a different image!
aws ecr put-image-tag-mutability \
  --repository-name my-app \
  --image-tag-mutability IMMUTABLE

# Try to push the same tag again → ERROR!
# This is GREAT for production repos where you need guarantees.
```

## 10. ECR Lifecycle Policies — Auto-Cleanup Old Images

### The Problem

Over time, your ECR repository fills up with old images. If you push a new image every build, you might have hundreds of old images taking up space and costing money.

```
my-app repository:
  v1.0.0  → 150 MB (pushed 6 months ago) ← Do you still need this?
  v1.0.1  → 150 MB (pushed 5 months ago) ← How about this?
  v1.0.2  → 155 MB
  v1.1.0  → 160 MB
  ...
  v2.5.3  → 180 MB (latest) ← This is the one you actually use!
  
  Total: 500 images × 150 MB = 75 GB → $7.50/month just sitting there!
```

### The Solution: Lifecycle Policies

Lifecycle policies let you **automatically delete old images** based on rules you define.

### Common Lifecycle Rules

```bash
# Set a lifecycle policy on a repository
aws ecr put-lifecycle-policy \
  --repository-name my-app \
  --lifecycle-policy-text '{
    "rules": [
      {
        "rulePriority": 1,
        "description": "Keep only last 10 tagged images",
        "selection": {
          "tagStatus": "tagged",
          "tagPrefixList": ["v"],
          "countType": "imageCountMoreThan",
          "countNumber": 10
        },
        "action": {
          "type": "expire"
        }
      },
      {
        "rulePriority": 2,
        "description": "Delete untagged images older than 1 day",
        "selection": {
          "tagStatus": "untagged",
          "countType": "sinceImagePushed",
          "countUnit": "days",
          "countNumber": 1
        },
        "action": {
          "type": "expire"
        }
      }
    ]
  }'
```

### Lifecycle Policy Examples

#### Keep Only the Last N Images

```json
{
  "rules": [
    {
      "rulePriority": 1,
      "description": "Keep only the last 20 images",
      "selection": {
        "tagStatus": "any",
        "countType": "imageCountMoreThan",
        "countNumber": 20
      },
      "action": { "type": "expire" }
    }
  ]
}
```

#### Delete Images Older Than 30 Days

```json
{
  "rules": [
    {
      "rulePriority": 1,
      "description": "Remove images older than 30 days",
      "selection": {
        "tagStatus": "any",
        "countType": "sinceImagePushed",
        "countUnit": "days",
        "countNumber": 30
      },
      "action": { "type": "expire" }
    }
  ]
}
```

#### Production-Ready Policy (Recommended)

```json
{
  "rules": [
    {
      "rulePriority": 1,
      "description": "Keep last 5 production release images (v*)",
      "selection": {
        "tagStatus": "tagged",
        "tagPrefixList": ["v"],
        "countType": "imageCountMoreThan",
        "countNumber": 5
      },
      "action": { "type": "expire" }
    },
    {
      "rulePriority": 2,
      "description": "Keep last 10 dev/feature images",
      "selection": {
        "tagStatus": "tagged",
        "tagPrefixList": ["dev-", "feature-"],
        "countType": "imageCountMoreThan",
        "countNumber": 10
      },
      "action": { "type": "expire" }
    },
    {
      "rulePriority": 10,
      "description": "Delete untagged images after 1 day",
      "selection": {
        "tagStatus": "untagged",
        "countType": "sinceImagePushed",
        "countUnit": "days",
        "countNumber": 1
      },
      "action": { "type": "expire" }
    }
  ]
}
```

### Preview Lifecycle Policy (Dry Run)

```bash
# See what WOULD be deleted (without actually deleting)
aws ecr get-lifecycle-policy-preview \
  --repository-name my-app

# Start a preview
aws ecr start-lifecycle-policy-preview \
  --repository-name my-app \
  --lifecycle-policy-text file://lifecycle-policy.json
```

## 11. ECR Image Scanning — Find Vulnerabilities

### What is Image Scanning?

ECR can scan your Docker images for **known security vulnerabilities** (CVEs). It checks the OS packages and application dependencies in your image against a database of known vulnerabilities.

### Types of Scanning

| Type | How It Works | Cost |
|------|-------------|------|
| **Basic Scanning** | Uses the open-source Clair scanner. Scans OS packages only | Free |
| **Enhanced Scanning** | Uses Amazon Inspector. Scans OS + application packages (npm, pip, etc.) | Paid (per scan) |

### Enable Scan on Push

```bash
# Enable basic scanning when images are pushed
aws ecr put-image-scanning-configuration \
  --repository-name my-app \
  --image-scanning-configuration scanOnPush=true
```

### Manually Trigger a Scan

```bash
# Scan a specific image
aws ecr start-image-scan \
  --repository-name my-app \
```

## 12. ECR Replication — Cross-Region & Cross-Account

### Why Replicate?

| Reason | Example |
|--------|---------|
| **Multi-region deployment** | Your app runs in us-east-1 AND eu-west-1 → replicate images to both regions |
| **Disaster recovery** | If one region goes down, images are safe in another |
| **Multi-account** | Dev, Staging, Prod are in different AWS accounts → share images |
| **Faster pulls** | Pull from a nearby region = faster container startup |

### Set Up Cross-Region Replication

```bash
# Replicate all repositories from us-east-1 to eu-west-1
aws ecr put-replication-configuration \
  --replication-configuration '{
    "rules": [
      {
        "destinations": [
          {
            "region": "eu-west-1",
            "registryId": "123456789012"
          }
        ],
        "repositoryFilters": [
          {
            "filter": "prod-",
            "filterType": "PREFIX_MATCH"
          }
        ]
      }
    ]
  }'

# This replicates all repos starting with "prod-" to eu-west-1
# When you push to us-east-1, it automatically appears in eu-west-1!
```

### Cross-Account Replication

```bash
# Allow account 999888777666 to pull images from your registry
aws ecr put-replication-configuration \
  --replication-configuration '{
    "rules": [
      {
        "destinations": [
          {
            "region": "us-east-1",
            "registryId": "999888777666"
          }
        ]
      }
    ]
  }'

# The destination account also needs a registry policy to allow this
```

### Pull-Through Cache

ECR can **cache images from public registries** (Docker Hub, GitHub, Quay). This means:
- No Docker Hub pull rate limits
- Faster pulls (from your local AWS region)
- Images are cached permanently

```bash
# Create a pull-through cache rule for Docker Hub
aws ecr create-pull-through-cache-rule \
  --ecr-repository-prefix docker-hub \
  --upstream-registry-url registry-1.docker.io

# Now instead of:
#   docker pull nginx:latest                         (from Docker Hub)
# You can do:
#   docker pull 123456789012.dkr.ecr.us-east-1.amazonaws.com/docker-hub/library/nginx:latest
# First pull goes to Docker Hub and caches it. Subsequent pulls come from ECR!
```

## 15. ECR with CodeBuild — CI/CD Integration

This is one of the most common use cases: **build a Docker image in CodeBuild and push it to ECR**.

### buildspec.yml for ECR Push

```yaml
version: 0.2

env:
  variables:
    AWS_ACCOUNT_ID: "123456789012"
    AWS_DEFAULT_REGION: "us-east-1"
    IMAGE_REPO_NAME: "my-app"
  exported-variables:
    - IMAGE_TAG
    - IMAGE_URI

phases:
  pre_build:
    commands:
      - echo "Logging into Amazon ECR..."
      - aws ecr get-login-password --region $AWS_DEFAULT_REGION | docker login --username AWS --password-stdin $AWS_ACCOUNT_ID.dkr.ecr.$AWS_DEFAULT_REGION.amazonaws.com
      
      # Generate a unique tag using the git commit SHA
      - IMAGE_TAG=$(echo $CODEBUILD_RESOLVED_SOURCE_VERSION | head -c 8)
      - IMAGE_URI=$AWS_ACCOUNT_ID.dkr.ecr.$AWS_DEFAULT_REGION.amazonaws.com/$IMAGE_REPO_NAME:$IMAGE_TAG
      
      - echo "Image will be tagged as: $IMAGE_TAG"
      - echo "Full URI: $IMAGE_URI"

  build:
    commands:
      - echo "Building Docker image..."
      - docker build -t $IMAGE_REPO_NAME:latest .
      
      # Tag for ECR
      - docker tag $IMAGE_REPO_NAME:latest $IMAGE_URI
      - docker tag $IMAGE_REPO_NAME:latest $AWS_ACCOUNT_ID.dkr.ecr.$AWS_DEFAULT_REGION.amazonaws.com/$IMAGE_REPO_NAME:latest

  post_build:
    commands:
      - echo "Pushing Docker image to ECR..."
      - docker push $IMAGE_URI
      - docker push $AWS_ACCOUNT_ID.dkr.ecr.$AWS_DEFAULT_REGION.amazonaws.com/$IMAGE_REPO_NAME:latest
      
      # Write the image URI to a file (for CodeDeploy/ECS to use)
      - echo "Writing image definitions file..."
      - printf '[{"name":"my-app","imageUri":"%s"}]' $IMAGE_URI > imagedefinitions.json
      
      - echo "Build completed successfully!"

artifacts:
  files:
    - imagedefinitions.json    # This file tells ECS which image to deploy
    - appspec.yaml             # For CodeDeploy (ECS Blue/Green)
    - taskdef.json             # ECS task definition
```

### The imagedefinitions.json File

When deploying to **ECS via CodePipeline**, the pipeline needs to know which Docker image to deploy. This file tells it:

```json
[
  {
    "name": "my-app",
    "imageUri": "123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app:abc12345"
  }
]
```

- `name` → Must match the container name in your ECS Task Definition
- `imageUri` → The full ECR URI with the tag to deploy

## 17. ECR Encryption & Security

### Encryption at Rest

| Type | How It Works | Cost |
|------|-------------|------|
| **AES-256** (default) | AWS manages encryption keys, always on | Free |
| **KMS** | You control the encryption key (AWS KMS) | KMS key costs apply |

```bash
# Create a repo with KMS encryption
aws ecr create-repository \
  --repository-name my-secure-app \
  --encryption-configuration '{
    "encryptionType": "KMS",
    "kmsKey": "arn:aws:kms:us-east-1:123456789012:key/my-key-id"
  }'
```

### Encryption in Transit

All communication with ECR is over **HTTPS** (TLS). This is automatic — you can't disable it.

### Security Best Practices

| Practice | Why |
|----------|-----|
| Enable **scan on push** | Catch vulnerabilities before they reach production |
| Use **immutable tags** for production repos | Prevent tag overwriting |
| Use **lifecycle policies** | Remove old, potentially vulnerable images |
| Use **least-privilege IAM policies** | Only grant the permissions needed |
| Use **VPC endpoints** for ECR | Keep traffic within AWS network (no internet) |
| Use **KMS encryption** for sensitive workloads | Control your own encryption keys |
| Regularly **rotate ECR credentials** | Minimize risk of compromised tokens |
| Use **ECR pull-through cache** | Avoid Docker Hub rate limits |

### VPC Endpoint for ECR (Private Link)

Keep ECR traffic **entirely within your VPC** (never touches the internet):

```bash
# You need TWO VPC endpoints for ECR:

# 1. ECR API endpoint (for auth, describe, list operations)
aws ec2 create-vpc-endpoint \
  --vpc-id vpc-12345 \
  --service-name com.amazonaws.us-east-1.ecr.api \
  --vpc-endpoint-type Interface \
  --subnet-ids subnet-abc123 \
  --security-group-ids sg-xyz789

# 2. ECR Docker endpoint (for actual image pull/push)
aws ec2 create-vpc-endpoint \
  --vpc-id vpc-12345 \
  --service-name com.amazonaws.us-east-1.ecr.dkr \
  --vpc-endpoint-type Interface \
  --subnet-ids subnet-abc123 \
  --security-group-ids sg-xyz789

# 3. S3 Gateway endpoint (ECR stores layers in S3 behind the scenes)
aws ec2 create-vpc-endpoint \
  --vpc-id vpc-12345 \
  --service-name com.amazonaws.us-east-1.s3 \
  --vpc-endpoint-type Gateway \
  --route-table-ids rtb-123456
```

## 19. Troubleshooting Common Issues

### "no basic auth credentials" Error

```bash
# Problem: You forgot to login, or the token expired (12 hours)
# Solution: Login again!
aws ecr get-login-password --region us-east-1 | \
  docker login --username AWS --password-stdin 123456789012.dkr.ecr.us-east-1.amazonaws.com
```

### "repository does not exist" Error

```bash
# Problem: The repository hasn't been created yet
# Solution: Create it first!
aws ecr create-repository --repository-name my-app

# Unlike Docker Hub, ECR requires you to create the repository BEFORE pushing.
# Docker Hub creates repos automatically on first push. ECR does NOT.
```

### "denied: User is not authorized" Error

```bash
# Problem: Your IAM user/role doesn't have ECR permissions
# Solution: Attach the right IAM policy

# Quick fix: Attach the AWS managed policy
aws iam attach-user-policy \
  --user-name my-user \
  --policy-arn arn:aws:iam::aws:policy/AmazonEC2ContainerRegistryPowerUser

# This grants push AND pull access to ALL repositories in your account
```

### "image tag already exists" Error

```bash
# Problem: You're trying to push a tag that already exists on an IMMUTABLE repository
# Solution: Use a different tag, or switch to MUTABLE

# Check mutability setting
aws ecr describe-repositories --repository-name my-app

# Change to MUTABLE (if you want to allow overwriting)
aws ecr put-image-tag-mutability \
  --repository-name my-app \
  --image-tag-mutability MUTABLE
```

### Slow Push/Pull

```bash
# Possible causes:
# 1. Large image → Use multi-stage builds, smaller base images (alpine)
# 2. Network → Use VPC endpoints for private connectivity
# 3. Cross-region → Enable replication to pull from a closer region
# 4. No layer caching → Structure Dockerfile for better caching

# Check image size
aws ecr describe-images \
  --repository-name my-app \
  --image-ids imageTag=latest \
  --query 'imageDetails[0].imageSizeInBytes'
```

### EKS Can't Pull Images

```bash
# Check 1: Does the node IAM role have ECR permissions?
# The node role needs: ecr:GetAuthorizationToken, ecr:BatchGetImage, etc.

# Check 2: Is the image URI correct?
kubectl describe pod my-pod  # Look at the "Events" section for errors

# Check 3: Is the image tag correct? (typos are common!)
aws ecr list-images --repository-name my-app  # Verify the tag exists

# Check 4: Is the repository in the same region as EKS?
# Cross-region pulls work but need proper IAM setup
```

## 20. Best Practices

### Repository Management

| Practice | Why |
|----------|-----|
| **One repository per microservice** | Clean separation: `frontend`, `backend`, `worker` |
| **Use meaningful names** | `my-company/api-gateway` not `repo1` |
| **Enable scan on push** | Catch vulnerabilities immediately |
| **Set lifecycle policies** | Prevent storage costs from growing indefinitely |
| **Use immutable tags for production** | Guarantee `v1.0.0` always means the same thing |

### Image Tagging

| Practice | Why |
|----------|-----|
| **Never use `latest` in production** | You can't tell which version is deployed |
| **Use semantic versioning** (`v1.2.3`) | Clear, human-readable versions |
| **Include git SHA** (`abc1234`) | Trace back to the exact commit |
| **Apply multiple tags** per image | Flexibility without extra storage |
| **Automate tagging in CI/CD** | Consistency, no human error |

### Security

| Practice | Why |
|----------|-----|
| **Use VPC endpoints** | Keep ECR traffic private |
| **Restrict IAM permissions** | Least privilege — only what's needed |
| **Enable encryption** | Protect images at rest |
| **Monitor scan findings** | Set up EventBridge alerts for CRITICAL/HIGH |
| **Use repository policies** for cross-account | Fine-grained access control |
| **Rotate auth tokens regularly** | ECR tokens expire in 12 hours by default |

### Cost Optimization

| Practice | Savings |
|----------|---------|
| **Set lifecycle policies** | Delete old images automatically |
| **Use small base images** (Alpine, Distroless) | Smaller = cheaper storage |
| **Multi-stage Docker builds** | Dramatically smaller final images |
| **Use pull-through cache** | Avoid external registry costs |
| **Clean up untagged images** | Remove orphaned layers |
