# 14.16 — AWS Secrets Manager & Systems Manager (SSM)

> **One Line Answer:** **Secrets Manager** is AWS's service for securely storing and automatically rotating sensitive data like passwords, API keys, and tokens, while **Systems Manager (SSM)** is a massive toolkit for managing, configuring, patching, and automating your entire AWS infrastructure — servers, containers, and more — all from one place.

---

## 🤔 Why Do We Even Need These Services?

Let's paint a picture.

You're running a web application on AWS. Your app connects to a database, calls third-party APIs, and uses encryption keys. That means you have **secrets** — passwords, API keys, tokens, certificates.

Where do most beginners put these?

```
# ❌ DON'T DO THIS — EVER!
# Hardcoded in your code
DB_PASSWORD = "SuperSecret123!"
API_KEY = "sk-abc123def456"
```

Or even worse:

```
# ❌ Pushed to Git
# .env file committed to the repo
DATABASE_URL=postgres://admin:SuperSecret123!@mydb.amazonaws.com:5432/myapp
```

**Why is this terrible?**

| Problem | What Happens |
|---------|-------------|
| Secrets in code | Anyone who reads the code sees your passwords |
| Secrets in Git | Even if you delete them later, they're in Git history **forever** |
| Rotating secrets | You have to manually change passwords in every server, every config |
| No audit trail | You don't know who accessed what secret and when |
| No encryption at rest | Plain text files on disk = vulnerable |

AWS built **Secrets Manager** and **Systems Manager Parameter Store** to solve these problems properly.

# Part 1: AWS Secrets Manager

---

## 📦 What is AWS Secrets Manager?

AWS Secrets Manager is a fully managed service that helps you:

1. **Store** secrets (passwords, API keys, tokens, database credentials)
2. **Encrypt** them automatically using AWS KMS (Key Management Service)
3. **Rotate** them automatically on a schedule
4. **Access** them programmatically from your applications
5. **Audit** who accessed what secret and when (via CloudTrail)

### Think of It Like This:

Imagine a **high-security vault** in a bank. You put your valuables (secrets) inside. Only people with the right key (IAM permissions) can open specific lockers. Every time someone opens a locker, a camera (CloudTrail) records it. And the bank can automatically change your lock combination (rotation) on a schedule.

That's Secrets Manager.

## 🏗️ How Secrets Manager Works — Architecture

```
┌──────────────────────────────────────────────────────────┐
│                    YOUR APPLICATION                       │
│                                                          │
│    app.py / server.js / main.go                          │
│         │                                                │
│         │ 1. "Hey, give me the DB password"              │
│         ▼                                                │
│  ┌──────────────────────┐                                │
│  │  AWS SDK / CLI Call   │                               │
│  │  GetSecretValue()     │                               │
│  └──────────┬───────────┘                                │
│             │                                            │
└─────────────┼────────────────────────────────────────────┘
              │
              ▼ 2. Request goes to Secrets Manager API
┌──────────────────────────────────────────────────────────┐
│              AWS SECRETS MANAGER                          │
│                                                          │
│  ┌────────────────┐  ┌────────────────┐                  │
│  │  Secret: my-db │  │  Secret: api-  │                  │
│  │  password      │  │  keys/stripe   │                  │
│  │  ┌──────────┐  │  │  ┌──────────┐  │                  │
│  │  │ Encrypted│  │  │  │ Encrypted│  │                  │
│  │  │ with KMS │  │  │  │ with KMS │  │                  │
│  │  └──────────┘  │  │  └──────────┘  │                  │
│  └────────────────┘  └────────────────┘                  │
│                                                          │
│  3. Checks IAM permissions ✅                            │
│  4. Decrypts with KMS                                    │
│  5. Returns the secret value                             │
│                                                          │
└──────────────────────────────────────────────────────────┘
              │
              ▼
┌──────────────────────────────────────────────────────────┐
│                    AWS KMS                                │
│         (Handles encryption/decryption)                  │
└──────────────────────────────────────────────────────────┘
```

**Flow:**
1. Your app calls `GetSecretValue()` via the AWS SDK
2. Secrets Manager checks if the caller has the right IAM permissions
3. If allowed, it decrypts the secret using KMS
4. Returns the secret value to your app
5. CloudTrail logs the entire access event

## 🛠️ Creating Secrets — AWS Console & CLI

### Method 1: AWS CLI

```bash
# Create a simple secret (key-value pair)
aws secretsmanager create-secret \
    --name "prod/myapp/db-password" \
    --description "Production database password for MyApp" \
    --secret-string '{"username":"admin","password":"SuperSecret123!","host":"mydb.us-east-1.rds.amazonaws.com","port":"5432","dbname":"myapp"}'
```

**What's happening here?**

| Parameter | What It Does |
|-----------|-------------|
| `--name` | A unique name/path for your secret. Using `/` creates a hierarchy (like folders) |
| `--description` | Human-readable description so your team knows what this is |
| `--secret-string` | The actual secret data — can be plain text or JSON |

**Output:**
```json
{
    "ARN": "arn:aws:secretsmanager:us-east-1:123456789012:secret:prod/myapp/db-password-AbCdEf",
    "Name": "prod/myapp/db-password",
    "VersionId": "a1b2c3d4-5678-90ab-cdef-EXAMPLE11111"
}
```

### Method 2: Store a Binary Secret (like a certificate)

```bash
# Store a binary file as a secret
aws secretsmanager create-secret \
    --name "prod/myapp/tls-cert" \
    --secret-binary fileb://my-certificate.p12
```

### Method 3: Using AWS Console

1. Go to **AWS Console → Secrets Manager**
2. Click **"Store a new secret"**
3. Choose secret type:
   - **Credentials for RDS database** (auto-fills connection details)
   - **Credentials for other database**
   - **Other type of secret** (API keys, tokens, etc.)
4. Enter your key-value pairs
5. Name your secret (e.g., `prod/myapp/api-keys`)
6. Configure rotation (optional)
7. Click **Store**

## 🔍 Retrieving Secrets

### Using AWS CLI

```bash
# Get the secret value
aws secretsmanager get-secret-value \
    --secret-id "prod/myapp/db-password"
```

**Output:**
```json
{
    "ARN": "arn:aws:secretsmanager:us-east-1:123456789012:secret:prod/myapp/db-password-AbCdEf",
    "Name": "prod/myapp/db-password",
    "VersionId": "a1b2c3d4-5678-90ab-cdef-EXAMPLE11111",
    "SecretString": "{\"username\":\"admin\",\"password\":\"SuperSecret123!\",\"host\":\"mydb.us-east-1.rds.amazonaws.com\",\"port\":\"5432\",\"dbname\":\"myapp\"}",
    "VersionStages": ["AWSCURRENT"],
    "CreatedDate": "2024-01-15T10:30:00Z"
}
```

### Using Python (boto3)

```python
import json
import boto3
from botocore.exceptions import ClientError

def get_secret(secret_name, region_name="us-east-1"):
    """
    Retrieve a secret from AWS Secrets Manager.
    
    Args:
        secret_name: The name/ARN of the secret
        region_name: AWS region where the secret is stored
    
    Returns:
        dict: The secret key-value pairs
    """
    # Create a Secrets Manager client
    # This uses your AWS credentials (from env vars, ~/.aws/credentials, or IAM role)
    client = boto3.client(
        service_name='secretsmanager',
        region_name=region_name
    )

    try:
        # Call the API to get the secret value
        response = client.get_secret_value(SecretId=secret_name)
        
        # The secret is returned as a JSON string — parse it
        secret = json.loads(response['SecretString'])
        return secret
        
    except ClientError as e:
        # Handle specific errors
        if e.response['Error']['Code'] == 'ResourceNotFoundException':
            print(f"Secret '{secret_name}' not found!")
        elif e.response['Error']['Code'] == 'AccessDeniedException':
            print(f"No permission to access '{secret_name}'!")
        else:
            print(f"Error: {e}")
        raise

# Usage
secret = get_secret("prod/myapp/db-password")
db_host = secret['host']          # "mydb.us-east-1.rds.amazonaws.com"
db_user = secret['username']      # "admin"
db_password = secret['password']  # "SuperSecret123!"
db_name = secret['dbname']        # "myapp"

# Now connect to your database using these credentials
# import psycopg2
# conn = psycopg2.connect(
#     host=db_host,
#     user=db_user,
#     password=db_password,
#     dbname=db_name
# )
```

### Using Node.js

```javascript
const { SecretsManagerClient, GetSecretValueCommand } = require("@aws-sdk/client-secrets-manager");

async function getSecret(secretName) {
    // Create the client
    const client = new SecretsManagerClient({ region: "us-east-1" });
    
    try {
        // Make the API call
        const response = await client.send(
            new GetSecretValueCommand({ SecretId: secretName })
        );
        
        // Parse the JSON string into an object
        return JSON.parse(response.SecretString);
    } catch (error) {
        console.error(`Failed to retrieve secret: ${error.message}`);
        throw error;
    }
}

// Usage
(async () => {
    const secret = await getSecret("prod/myapp/db-password");
    console.log(`Connecting to ${secret.host} as ${secret.username}`);
})();
```

## 🔄 Secret Rotation — The Killer Feature

This is where Secrets Manager **really** shines. It can **automatically rotate** your secrets on a schedule — no human intervention needed.

### Why Rotate Secrets?

| Without Rotation | With Rotation |
|-----------------|---------------|
| Same password for months/years | New password every 30/60/90 days |
| If leaked, attacker has permanent access | If leaked, it expires automatically |
| Manual process, error-prone | Fully automated |
| Compliance violations | Meets SOC2, PCI-DSS, HIPAA requirements |

### How Rotation Works

```
┌──────────────────────────────────────────────────────┐
│                  ROTATION FLOW                        │
│                                                      │
│  1. Schedule triggers (e.g., every 30 days)          │
│         │                                            │
│         ▼                                            │
│  2. Secrets Manager invokes a Lambda function        │
│         │                                            │
│         ▼                                            │
│  3. Lambda generates a new password                  │
│         │                                            │
│         ▼                                            │
│  4. Lambda updates the password in the database      │
│         │                                            │
│         ▼                                            │
│  5. Lambda stores the new password in Secrets Mgr    │
│         │                                            │
│         ▼                                            │
│  6. New version becomes "AWSCURRENT"                 │
│     Old version becomes "AWSPREVIOUS"                │
│         │                                            │
│         ▼                                            │
│  7. Next time your app calls GetSecretValue(),       │
│     it gets the NEW password automatically!          │
│                                                      │
└──────────────────────────────────────────────────────┘
```

### Setting Up Rotation for an RDS Database

```bash
# Enable rotation for an RDS secret
aws secretsmanager rotate-secret \
    --secret-id "prod/myapp/db-password" \
    --rotation-lambda-arn "arn:aws:lambda:us-east-1:123456789012:function:SecretsManagerRDSRotation" \
    --rotation-rules '{"AutomaticallyAfterDays": 30}'
```

**What each part does:**

| Parameter | Meaning |
|-----------|---------|
| `--secret-id` | Which secret to rotate |
| `--rotation-lambda-arn` | The Lambda function that knows how to change the DB password |
| `--rotation-rules` | How often to rotate (every 30 days here) |

> **Note:** For RDS, Aurora, Redshift, and DocumentDB, AWS provides **pre-built Lambda rotation functions**. For custom secrets (API keys, etc.), you write your own Lambda function.

### Rotation Stages (Versioning)

Every secret can have multiple versions with labels:

| Stage | Meaning |
|-------|---------|
| `AWSCURRENT` | The **active** version — this is what your app gets by default |
| `AWSPREVIOUS` | The **last** version — kept as a fallback |
| `AWSPENDING` | The **new** version being set up during rotation (temporary) |

```bash
# Get a specific version
aws secretsmanager get-secret-value \
    --secret-id "prod/myapp/db-password" \
    --version-stage "AWSPREVIOUS"
```

## 🔐 IAM Policies for Secrets Manager

### Policy: Allow an App to Read a Specific Secret

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": [
                "secretsmanager:GetSecretValue"
            ],
            "Resource": "arn:aws:secretsmanager:us-east-1:123456789012:secret:prod/myapp/*"
        }
    ]
}
```

**What this does:**
- ✅ Allows reading secrets that start with `prod/myapp/`
- ❌ Cannot create, update, or delete secrets
- ❌ Cannot access secrets from other apps or environments

### Policy: Deny Access to Secrets Without Encryption

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Deny",
            "Action": "secretsmanager:GetSecretValue",
            "Resource": "*",
            "Condition": {
                "StringNotEquals": {
                    "secretsmanager:ResourceTag/Environment": "production"
                }
            }
        }
    ]
}
```

## 🏪 SSM Parameter Store — The Free Secrets Store

Parameter Store is where you store **configuration data** and **secrets** as key-value pairs. It's like a simpler, cheaper version of Secrets Manager.

### When to Use Parameter Store vs Secrets Manager?

| Feature | Parameter Store | Secrets Manager |
|---------|----------------|-----------------|
| **Cost** | ✅ **Free** (standard) | 💰 $0.40/secret/month |
| **Automatic Rotation** | ❌ No built-in rotation | ✅ Built-in rotation |
| **Max Size** | 4 KB (standard) / 8 KB (advanced) | 64 KB |
| **Encryption** | ✅ Optional (KMS) | ✅ Always encrypted |
| **Versioning** | ✅ Yes | ✅ Yes |
| **Cross-account access** | ❌ Limited | ✅ Yes |
| **Best For** | Config values, feature flags, non-rotating secrets | Database passwords, API keys that need rotation |

### Parameter Types

| Type | What It Is | Example |
|------|-----------|---------|
| **String** | Plain text value | `"us-east-1"` |
| **StringList** | Comma-separated values | `"us-east-1,us-west-2,eu-west-1"` |
| **SecureString** | Encrypted with KMS | `"SuperSecret123!"` (stored encrypted) |

### Creating Parameters — CLI

```bash
# Create a plain text parameter (String)
aws ssm put-parameter \
    --name "/myapp/prod/db-host" \
    --value "mydb.us-east-1.rds.amazonaws.com" \
    --type "String" \
    --description "Production database hostname"

# Create an encrypted parameter (SecureString)
aws ssm put-parameter \
    --name "/myapp/prod/db-password" \
    --value "SuperSecret123!" \
    --type "SecureString" \
    --description "Production database password"
# The value gets encrypted automatically using the default AWS KMS key

# Create a parameter with a custom KMS key
aws ssm put-parameter \
    --name "/myapp/prod/api-key" \
    --value "sk-abc123def456" \
    --type "SecureString" \
    --key-id "alias/my-custom-key" \
    --description "Stripe API key"

# Create a StringList parameter
aws ssm put-parameter \
    --name "/myapp/prod/allowed-regions" \
    --value "us-east-1,us-west-2,eu-west-1" \
    --type "StringList"
```

### Retrieving Parameters — CLI

```bash
# Get a single parameter
aws ssm get-parameter \
    --name "/myapp/prod/db-host"

# Output:
# {
#     "Parameter": {
#         "Name": "/myapp/prod/db-host",
#         "Type": "String",
#         "Value": "mydb.us-east-1.rds.amazonaws.com",
#         "Version": 1,
#         "LastModifiedDate": "2024-01-15T10:30:00Z"
#     }
# }

# WITH --with-decryption, you get the actual password

    --path "/myapp/prod/" \
    --recursive \
    --with-decryption

# This returns ALL parameters under /myapp/prod/ — super handy!
```

### Retrieving Parameters — Python

```python
import boto3

def get_parameter(name, decrypt=True):
    """
    Get a single parameter from SSM Parameter Store.
    
    Args:
        name: The parameter name (e.g., "/myapp/prod/db-host")
        decrypt: Whether to decrypt SecureString parameters
    
    Returns:
        str: The parameter value
    """
    ssm = boto3.client('ssm', region_name='us-east-1')
    
    response = ssm.get_parameter(
        Name=name,
        WithDecryption=decrypt  # Set to True for SecureString params
    )
    
    return response['Parameter']['Value']

def get_all_parameters(path, decrypt=True):
    """
    Get ALL parameters under a given path.
    Great for loading all config for an environment at once.
    
    Args:
        path: The parameter path prefix (e.g., "/myapp/prod/")
        decrypt: Whether to decrypt SecureString parameters
    
    Returns:
        dict: Parameter names and their values
    """
    ssm = boto3.client('ssm', region_name='us-east-1')
    parameters = {}
    
    # Use paginator because there might be many parameters
    paginator = ssm.get_paginator('get_parameters_by_path')
    
    for page in paginator.paginate(
        Path=path,
        Recursive=True,
        WithDecryption=decrypt
    ):
        for param in page['Parameters']:
            # Extract just the key name (remove the path prefix)
            key = param['Name'].split('/')[-1]
            parameters[key] = param['Value']
    
    return parameters

# Usage
db_host = get_parameter("/myapp/prod/db-host")
print(f"DB Host: {db_host}")

# Load all prod configs at once
config = get_all_parameters("/myapp/prod/")
print(f"DB Host: {config['db-host']}")
print(f"DB Password: {config['db-password']}")
```

## 🖥️ SSM Session Manager — SSH Without SSH

This is one of the **coolest** SSM features. It lets you connect to your EC2 instances **without**:
- Opening SSH port (port 22) in your security group
- Managing SSH key pairs
- Setting up a bastion host / jump box
- Having a public IP on the instance

### How It Works

```
Traditional SSH:
┌──────┐    Port 22     ┌──────────┐
│ Your │ ───────────► │ EC2      │   ❌ Requires port 22 open
│ PC   │    SSH Key    │ Instance │   ❌ Requires key management
└──────┘               └──────────┘   ❌ Requires public IP or bastion

SSM Session Manager:
┌──────┐    HTTPS      ┌──────────────┐     SSM Agent    ┌──────────┐
│ Your │ ───────────► │ SSM Service  │ ◄──────────────  │ EC2      │
│ PC   │   (Port 443) │ (AWS Cloud)  │   (Outbound 443) │ Instance │
└──────┘               └──────────────┘                  └──────────┘
                                                          ✅ No port 22
                                                          ✅ No SSH keys
                                                          ✅ No public IP needed
                                                          ✅ Full audit logging
```

### Prerequisites

1. **SSM Agent** must be installed on the EC2 instance (pre-installed on Amazon Linux 2, Ubuntu 20.04+)
2. **IAM Role** on the EC2 instance with the `AmazonSSMManagedInstanceCore` policy
3. **Outbound HTTPS (443)** access from the instance to SSM endpoints

### Setting Up Session Manager

```bash
# Step 1: Create an IAM role for EC2 with SSM permissions
aws iam create-role \
    --role-name EC2-SSM-Role \
    --assume-role-policy-document '{
        "Version": "2012-10-17",
        "Statement": [{
            "Effect": "Allow",
            "Principal": {"Service": "ec2.amazonaws.com"},
            "Action": "sts:AssumeRole"
        }]
    }'

# Step 2: Attach the SSM managed policy
aws iam attach-role-policy \
    --role-name EC2-SSM-Role \
    --policy-arn "arn:aws:iam::aws:policy/AmazonSSMManagedInstanceCore"

# Step 3: Create an instance profile (required for EC2)
aws iam create-instance-profile \
    --instance-profile-name EC2-SSM-Profile

aws iam add-role-to-instance-profile \
    --instance-profile-name EC2-SSM-Profile \
    --role-name EC2-SSM-Role

# Step 4: Launch an EC2 instance with this profile
aws ec2 run-instances \
    --image-id ami-0abcdef1234567890 \
    --instance-type t3.micro \
    --iam-instance-profile Name=EC2-SSM-Profile \
    --tag-specifications 'ResourceType=instance,Tags=[{Key=Name,Value=MyServer}]'
```

### Connecting via Session Manager

```bash
# Start an interactive shell session
aws ssm start-session --target i-0abc123def456789

# You'll see something like:
# Starting session with SessionId: user@example.com-0abc123def
```

### Port Forwarding Through SSM (Super Useful!)

```bash
# Forward a port from EC2 to your local machine
# Example: Access RDS database through an EC2 instance
aws ssm start-session \
    --target i-0abc123def456789 \
    --document-name AWS-StartPortForwardingSessionToRemoteHost \
    --parameters '{"host":["mydb.us-east-1.rds.amazonaws.com"],"portNumber":["5432"],"localPortNumber":["5432"]}'

# Now you can connect to localhost:5432 and it goes through
# the EC2 instance to your RDS database — no VPN needed!
```

## 📝 SSM Run Command — Execute Commands at Scale

Run Command lets you execute commands across **hundreds of instances** simultaneously without logging into each one.

```bash
# Run a command on multiple instances
aws ssm send-command \
    --targets "Key=tag:Environment,Values=production" \
    --document-name "AWS-RunShellScript" \
    --parameters 'commands=["yum update -y","systemctl restart nginx"]' \
    --comment "Update packages and restart nginx on all prod servers"

# What this does:
# 1. Finds ALL instances tagged Environment=production
# 2. Runs "yum update -y" on each one
# 3. Then runs "systemctl restart nginx"
# 4. Collects output from all instances
# 5. Reports success/failure for each
```

### Check Command Output

```bash
# Get the results of a command
aws ssm list-command-invocations \
    --command-id "a1b2c3d4-5678-90ab-cdef-EXAMPLE" \
    --details

# Shows output from each instance:
# Instance i-0abc123: SUCCESS - "nginx restarted"
# Instance i-0def456: SUCCESS - "nginx restarted"  
# Instance i-0ghi789: FAILED - "nginx not installed"
```

## 📑 SSM Documents — Reusable Automation Scripts

SSM Documents (formerly called "SSM Documents" or "Command Documents") are JSON or YAML definitions that tell SSM **what to do**. Think of them as reusable scripts.

### Built-in Documents

AWS provides many pre-built documents:

| Document | What It Does |
|----------|-------------|
| `AWS-RunShellScript` | Run bash commands on Linux |
| `AWS-RunPowerShellScript` | Run PowerShell commands on Windows |
| `AWS-UpdateSSMAgent` | Update the SSM Agent |
| `AWS-ApplyPatchBaseline` | Apply OS patches |
| `AWS-ConfigureAWSPackage` | Install/uninstall AWS packages |

### Creating a Custom Document

```yaml
# my-app-deploy.yaml
schemaVersion: "2.2"
description: "Deploy MyApp to instances"
parameters:
  AppVersion:
    type: "String"
    description: "Version to deploy"
    default: "latest"
  Environment:
    type: "String"
    description: "Environment (prod/staging/dev)"
    allowedValues:
      - "prod"
      - "staging"
      - "dev"
mainSteps:
  - action: "aws:runShellScript"
    name: "StopApp"
    inputs:
      runCommand:
        - "systemctl stop myapp || true"

  - action: "aws:runShellScript"
    name: "DeployApp"
    inputs:
      runCommand:
        - "aws s3 cp s3://my-deploys/myapp-{{ AppVersion }}.tar.gz /tmp/"
        - "tar -xzf /tmp/myapp-{{ AppVersion }}.tar.gz -C /opt/myapp/"
        - "chown -R myapp:myapp /opt/myapp/"

  - action: "aws:runShellScript"
    name: "StartApp"
    inputs:
      runCommand:
        - "systemctl start myapp"
        - "sleep 5"
        - "systemctl status myapp"
```

```bash
# Register the document
aws ssm create-document \
    --name "MyApp-Deploy" \
    --content file://my-app-deploy.yaml \
    --document-type "Command" \
    --document-format "YAML"

# Use it!
aws ssm send-command \
    --document-name "MyApp-Deploy" \
    --targets "Key=tag:Application,Values=myapp" \
    --parameters '{"AppVersion":["v2.1.0"],"Environment":["prod"]}'
```

## 🔄 SSM Automation — Complex Workflows

Automation lets you build **multi-step workflows** (Runbooks) for common tasks like:
- Creating golden AMIs
- Remediation of non-compliant instances
- Rolling deployments

```yaml
# golden-ami-automation.yaml
schemaVersion: "0.3"
description: "Create a Golden AMI"
parameters:
  SourceAmiId:
    type: "String"
    description: "Source AMI to start from"
  InstanceType:
    type: "String"
    default: "t3.medium"
mainSteps:
  - name: LaunchInstance
    action: "aws:runInstances"
    inputs:
      ImageId: "{{ SourceAmiId }}"
      InstanceType: "{{ InstanceType }}"
      MinInstanceCount: 1
      MaxInstanceCount: 1

  - name: InstallSoftware
    action: "aws:runCommand"
    inputs:
      DocumentName: "AWS-RunShellScript"
      InstanceIds:
        - "{{ LaunchInstance.InstanceIds }}"
      Parameters:
        commands:
          - "yum update -y"
          - "yum install -y nginx docker"
          - "systemctl enable nginx docker"

  - name: CreateImage
    action: "aws:createImage"
    inputs:
      InstanceId: "{{ LaunchInstance.InstanceIds }}"
      ImageName: "GoldenAMI-{{ global:DATE_TIME }}"
      NoReboot: false

  - name: TerminateInstance
    action: "aws:changeInstanceState"
    inputs:
      InstanceIds:
        - "{{ LaunchInstance.InstanceIds }}"
      DesiredState: "terminated"

outputs:
  - "CreateImage.ImageId"
```

## 🆚 Secrets Manager vs Parameter Store — Quick Decision Guide

```
Need automatic secret rotation?
├── YES → Use Secrets Manager
└── NO
    ├── Is it a database credential (RDS/Aurora)?
    │   ├── YES → Use Secrets Manager (built-in RDS rotation)
    │   └── NO
    │       ├── Is it a secret (password, API key)?
    │       │   ├── YES, but rotation not needed → Use Parameter Store (SecureString)
    │       │   └── YES, and rotation needed → Use Secrets Manager
    │       └── Is it a config value (not sensitive)?
    │           └── YES → Use Parameter Store (String)
    └── Budget is tight?
        └── YES → Use Parameter Store (it's free!)
```

## 🧠 Key Takeaways

| Concept | Remember This |
|---------|--------------|
| **Secrets Manager** | Stores & rotates secrets. Costs $0.40/secret/month. Best for DB passwords & API keys that need rotation |
| **Parameter Store** | Stores config & secrets. **Free** for standard. No auto-rotation. Best for config values & secrets you rotate manually |
| **Session Manager** | SSH without SSH. No port 22, no keys, no bastion. Just IAM + SSM Agent |
| **Run Command** | Run commands across hundreds of servers at once. Tag-based targeting |
| **Patch Manager** | Auto-patch your OS. Schedule maintenance windows |
| **Automation** | Multi-step runbooks. Golden AMI creation, remediation workflows |
| **Documents** | Reusable scripts in YAML/JSON. AWS provides many built-in ones |
| **Inventory** | Automatic tracking of what's installed on every instance |

## ❓ Common Interview Questions

**Q1: What's the difference between Secrets Manager and Parameter Store?**
> Secrets Manager is designed for secrets that need **automatic rotation** (like database passwords). It costs $0.40/secret/month. Parameter Store is free, handles both config values and secrets, but doesn't have built-in rotation. Use Secrets Manager for critical credentials; use Parameter Store for everything else.

**Q2: How does Session Manager work without opening port 22?**
> The SSM Agent on the instance makes an **outbound HTTPS connection** (port 443) to the SSM service. The service then brokers the session. Since the connection is outbound from the instance, no inbound ports need to be opened. All sessions are logged to CloudTrail and optionally to S3/CloudWatch.

**Q3: How do you access Secrets Manager from a Lambda function?**
> The Lambda function needs an IAM role with the `secretsmanager:GetSecretValue` permission. In the code, use the AWS SDK (`boto3` for Python) to call `get_secret_value()`. For better performance, cache the secret value and use the Secrets Manager caching library.

**Q4: Can you use Parameter Store parameters in CloudFormation?**
> Yes! CloudFormation can resolve SSM parameters using `{{resolve:ssm:parameter-name}}` syntax. For SecureStrings, use `{{resolve:ssm-secure:parameter-name}}`. This is great for referencing AMI IDs, VPC IDs, and other values without hardcoding them.
