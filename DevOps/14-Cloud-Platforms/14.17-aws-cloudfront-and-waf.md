# 14.17 — AWS CloudFront & WAF

> **One Line Answer:** **CloudFront** is AWS's Content Delivery Network (CDN) that caches and delivers your content (websites, APIs, videos) from edge locations around the world for blazing-fast performance, while **WAF** (Web Application Firewall) sits in front of CloudFront (or ALB) and protects your applications from common web attacks like SQL injection, XSS, and DDoS.

---

## 🤔 Let's Start With the Problem

Imagine you're running a website hosted on a server in **us-east-1 (Virginia, USA)**. A user in **Tokyo, Japan** visits your site. What happens?

```
User in Tokyo                                    Server in Virginia
    │                                                    │
    │  ──── Request travels ~11,000 km ──────────────►  │
    │                                                    │
    │  ◄── Response travels ~11,000 km back ──────────  │
    │                                                    │
    Total: ~22,000 km round trip
    Latency: 200-400ms (at the speed of light through fiber!)
```

Now multiply this by every image, CSS file, JavaScript file, and API call. Your Tokyo user is waiting **seconds** for your page to load.

**What if we could put a copy of your content IN Tokyo?**

```
User in Tokyo          Edge Location in Tokyo          Server in Virginia
    │                         │                              │
    │  ── 5 km ──►           │  (Already has a cached copy) │
    │                         │                              │
    │  ◄── 5 km ──           │                              │
    │                         │                              │
    Total: ~10 km round trip
    Latency: 5-20ms 🚀
```

That's what CloudFront does. And **WAF** makes sure that while your content flies fast, hackers can't come along for the ride.

# Part 1: AWS CloudFront (CDN)

---

## 📦 What is CloudFront?

CloudFront is AWS's **Content Delivery Network (CDN)**. It has **450+ Points of Presence (PoPs)** — also called **edge locations** — spread across the globe. When someone requests your content, CloudFront serves it from the **nearest edge location** instead of going all the way to your origin server.

### Key Concepts — The Vocabulary

| Term | What It Means | Analogy |
|------|-------------|---------|
| **Distribution** | A CloudFront configuration/deployment — your CDN setup | A franchise business plan |
| **Origin** | Where your original content lives (S3, EC2, ALB, or any HTTP server) | The headquarters / main warehouse |
| **Edge Location** | A data center where content gets cached around the world (~450+) | Local branch offices |
| **Regional Edge Cache** | A larger cache between edge locations and origin (~13 globally) | Regional warehouses |
| **Behavior** | Rules that define how CloudFront handles different URL patterns | Traffic rules for different roads |
| **TTL** | Time To Live — how long content stays cached | Expiry date on food |
| **Invalidation** | Manually force-remove cached content before TTL expires | Recalling a product from stores |
| **Origin Access Control (OAC)** | Restricts S3 access so only CloudFront can read from it | A VIP-only entrance |

## 🏗️ How CloudFront Works — Step by Step

```
┌─────────────────────────────────────────────────────────────────────┐
│                        CLOUDFRONT FLOW                               │
│                                                                      │
│  1. User in Mumbai requests https://cdn.mysite.com/logo.png         │
│         │                                                            │
│         ▼                                                            │
│  2. DNS resolves to the NEAREST edge location (Mumbai)              │
│         │                                                            │
│         ▼                                                            │
│  3. Edge location checks: "Do I have /logo.png cached?"            │
│         │                                                            │
│         ├── YES (Cache HIT) ────► Return cached content instantly   │
│         │                          Latency: 5-20ms ⚡               │
│         │                                                            │
│         └── NO (Cache MISS) ───► 4. Edge asks Regional Edge Cache   │
│                                        │                             │
│                                        ├── HIT ──► Return content   │
│                                        │                             │
│                                        └── MISS ─► 5. Fetch from   │
│                                                       ORIGIN        │
│                                                       (S3 / ALB)    │
│                                                         │           │
│                                                         ▼           │
│                                               6. Origin responds    │
│                                               7. Cache at edge      │
│                                               8. Return to user     │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

**On subsequent requests** for the same content from the same region — it's already cached. Instant delivery! 🚀

## 🛠️ Setting Up CloudFront — Step by Step

### Scenario 1: CloudFront + S3 (Static Website)

This is the most common setup — serving a static website (React, Angular, Vue, Hugo, etc.) through CloudFront.

#### Step 1: Create an S3 Bucket

```bash
# Create the bucket
aws s3 mb s3://my-awesome-website-2024

# Upload your website files
aws s3 sync ./build/ s3://my-awesome-website-2024/ \
    --delete

# The --delete flag removes files from S3 that are no longer in your local build/
```

#### Step 2: Create a CloudFront Distribution

```bash
aws cloudfront create-distribution \
    --distribution-config '{
        "CallerReference": "my-website-2024-01",
        "Comment": "My awesome website CDN",
        "DefaultCacheBehavior": {
            "TargetOriginId": "S3-my-awesome-website",
            "ViewerProtocolPolicy": "redirect-to-https",
            "AllowedMethods": {
                "Quantity": 2,
                "Items": ["GET", "HEAD"]
            },
            "CachePolicyId": "658327ea-f89d-4fab-a63d-7e88639e58f6",
            "Compress": true
        },
        "Origins": {
            "Quantity": 1,
            "Items": [{
                "Id": "S3-my-awesome-website",
                "DomainName": "my-awesome-website-2024.s3.us-east-1.amazonaws.com",
                "S3OriginConfig": {
                    "OriginAccessIdentity": ""
                },
                "OriginAccessControlId": "E2QWRUHEXAMPLE"
            }]
        },
        "Enabled": true,
        "DefaultRootObject": "index.html"
    }'
```

**Let's break down each setting:**

| Setting | What It Does |
|---------|-------------|
| `CallerReference` | A unique ID for this distribution (prevents duplicates) |
| `ViewerProtocolPolicy: redirect-to-https` | Forces HTTPS — anyone visiting HTTP gets redirected |
| `AllowedMethods: GET, HEAD` | Only allow read operations (perfect for static sites) |
| `CachePolicyId` | Uses the "CachingOptimized" managed policy for best performance |
| `Compress: true` | Automatically gzip/brotli compress content (smaller = faster) |
| `DefaultRootObject: index.html` | When someone visits `/`, serve `index.html` |

#### Step 3: Create Origin Access Control (OAC)

OAC ensures that **only CloudFront** can access your S3 bucket — users can't bypass the CDN and go directly to S3.

```bash
# Create OAC
aws cloudfront create-origin-access-control \
    --origin-access-control-config '{
        "Name": "my-website-oac",
        "Description": "OAC for my website S3 bucket",
        "SigningProtocol": "sigv4",
        "SigningBehavior": "always",
        "OriginAccessControlOriginType": "s3"
    }'
```

Then add this bucket policy to S3:

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "AllowCloudFrontServicePrincipal",
            "Effect": "Allow",
            "Principal": {
                "Service": "cloudfront.amazonaws.com"
            },
            "Action": "s3:GetObject",
            "Resource": "arn:aws:s3:::my-awesome-website-2024/*",
            "Condition": {
                "StringEquals": {
                    "AWS:SourceArn": "arn:aws:cloudfront::123456789012:distribution/E1ABCDEF123456"
                }
            }
        }
    ]
}
```

**What this policy says:**
- ✅ CloudFront distribution `E1ABCDEF123456` can read objects from this bucket
- ❌ Nobody else can — not even someone who knows the S3 URL directly

## 🔧 CloudFront with Terraform

```hcl
# Create a CloudFront distribution for a static website
resource "aws_cloudfront_distribution" "website" {
  enabled             = true
  is_ipv6_enabled     = true
  comment             = "My website CDN"
  default_root_object = "index.html"
  price_class         = "PriceClass_100"  # Only use edge locations in US, Canada, Europe (cheapest)
  
  # Aliases (custom domain names)
  aliases = ["www.mysite.com", "mysite.com"]

  # Origin: S3 bucket
  origin {
    domain_name              = aws_s3_bucket.website.bucket_regional_domain_name
    origin_id                = "S3-Website"
    origin_access_control_id = aws_cloudfront_origin_access_control.website.id
  }

  # Default behavior (for all paths)
  default_cache_behavior {
    allowed_methods        = ["GET", "HEAD", "OPTIONS"]
    cached_methods         = ["GET", "HEAD"]
    target_origin_id       = "S3-Website"
    viewer_protocol_policy = "redirect-to-https"
    compress               = true

    # Use a managed cache policy
    cache_policy_id = "658327ea-f89d-4fab-a63d-7e88639e58f6"  # CachingOptimized
  }

  # Custom error responses (for SPA routing)
  custom_error_response {
    error_code            = 403
    response_code         = 200
    response_page_path    = "/index.html"
    error_caching_min_ttl = 10
  }

  custom_error_response {
    error_code            = 404
    response_code         = 200
    response_page_path    = "/index.html"
    error_caching_min_ttl = 10
  }

  # SSL certificate (must be in us-east-1 for CloudFront!)
  viewer_certificate {
    acm_certificate_arn      = aws_acm_certificate.website.arn
    ssl_support_method       = "sni-only"
    minimum_protocol_version = "TLSv1.2_2021"
  }

  # Geo-restriction (optional)
  restrictions {
    geo_restriction {
      restriction_type = "none"  # or "whitelist" / "blacklist"
    }
  }

  tags = {
    Environment = "production"
    Project     = "my-website"
  }
}

# Origin Access Control
resource "aws_cloudfront_origin_access_control" "website" {
  name                              = "website-oac"
  description                       = "OAC for website S3 bucket"
  origin_access_control_origin_type = "s3"
  signing_behavior                  = "always"
  signing_protocol                  = "sigv4"
}

# S3 bucket policy to allow CloudFront
resource "aws_s3_bucket_policy" "website" {
  bucket = aws_s3_bucket.website.id

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Sid    = "AllowCloudFront"
        Effect = "Allow"
        Principal = {
          Service = "cloudfront.amazonaws.com"
        }
        Action   = "s3:GetObject"
        Resource = "${aws_s3_bucket.website.arn}/*"
        Condition = {
          StringEquals = {
            "AWS:SourceArn" = aws_cloudfront_distribution.website.arn
          }
        }
      }
    ]
  })
}
```

**Why the custom error responses for 403 and 404?**

If you're running a Single Page Application (React, Angular, Vue), the routing is handled client-side. When a user goes to `/about`, CloudFront looks for a file called `about` in S3, doesn't find it, and returns a 403/404. By mapping these errors to `/index.html` with a 200 status, CloudFront serves the SPA and lets the client-side router handle the URL.

## 🗑️ Cache Invalidation — When You Need Fresh Content NOW

When you deploy new code, CloudFront might still serve the **old cached version** until the TTL expires. Invalidation forces CloudFront to throw away the cached version.

```bash
# Invalidate specific files
aws cloudfront create-invalidation \
    --distribution-id E1ABCDEF123456 \
    --paths "/index.html" "/static/js/*" "/static/css/*"

# Invalidate EVERYTHING (nuclear option)
aws cloudfront create-invalidation \
    --distribution-id E1ABCDEF123456 \
    --paths "/*"

# Check invalidation status
aws cloudfront get-invalidation \
    --distribution-id E1ABCDEF123456 \
    --id I1ABCDEF234567
```

**Important Notes:**
- First 1,000 invalidation paths per month are **free**
- After that, $0.005 per path
- Wildcard `/*` counts as **one** path (pro tip!)
- Invalidations take 5-15 minutes to propagate globally

### Better Alternative: Cache Busting

Instead of invalidating, use **versioned file names**:

```
# Instead of:
/static/js/app.js        ← Same name, CloudFront caches forever

# Use:
/static/js/app.a1b2c3.js ← Hash in filename, new deploy = new hash = new URL
```

Most build tools (Webpack, Vite) do this automatically. CloudFront sees a new URL and fetches from origin.

## ⚡ CloudFront Functions vs Lambda@Edge

CloudFront can run code **at the edge** — right where the user is. There are two options:

### CloudFront Functions (Lightweight)

- **Runs at:** All 450+ edge locations
- **Execution time:** < 1 millisecond
- **Memory:** 2 MB
- **Language:** JavaScript (ES 5.1)
- **Cost:** $0.10 per million invocations
- **Use cases:** URL redirects, header manipulation, simple auth, URL rewrites

```javascript
// CloudFront Function: Add security headers to every response
function handler(event) {
    var response = event.response;
    var headers = response.headers;

    // Add security headers
    headers['strict-transport-security'] = { value: 'max-age=63072000; includeSubdomains; preload' };
    headers['x-content-type-options'] = { value: 'nosniff' };
    headers['x-frame-options'] = { value: 'DENY' };
    headers['x-xss-protection'] = { value: '1; mode=block' };
    headers['referrer-policy'] = { value: 'strict-origin-when-cross-origin' };

    return response;
}
```

```javascript
// CloudFront Function: Redirect www to non-www
function handler(event) {
    var request = event.request;
    var host = request.headers.host.value;

    if (host.startsWith('www.')) {
        var newHost = host.replace('www.', '');
        return {
            statusCode: 301,
            statusDescription: 'Moved Permanently',
            headers: {
                'location': { value: 'https://' + newHost + request.uri }
            }
        };
    }

    return request;
}
```

### Lambda@Edge (Heavy Lifting)

- **Runs at:** 13 Regional Edge Caches
- **Execution time:** Up to 30 seconds (viewer) / 30 seconds (origin)
- **Memory:** Up to 10 GB
- **Language:** Node.js, Python
- **Cost:** Higher (Lambda pricing)
- **Use cases:** Complex auth (JWT validation), image resizing, A/B testing, dynamic content generation

```javascript
// Lambda@Edge: Resize images on the fly
exports.handler = async (event) => {
    const request = event.Records[0].cf.request;
    const uri = request.uri;
    
    // Check if the request is for an image with size parameters
    // e.g., /images/photo.jpg?w=300&h=200
    const params = new URLSearchParams(request.querystring);
    const width = params.get('w');
    const height = params.get('h');
    
    if (width || height) {
        // Modify the origin request to fetch from a resizing service
        request.uri = `/resize/${width}x${height}${uri}`;
    }
    
    return request;
};
```

### Quick Comparison

| Feature | CloudFront Functions | Lambda@Edge |
|---------|---------------------|-------------|
| **Speed** | Sub-millisecond | Milliseconds to seconds |
| **Scale** | Millions of req/sec | Thousands of req/sec |
| **Languages** | JavaScript only | Node.js, Python |
| **Network access** | ❌ No | ✅ Yes |
| **Body access** | ❌ No | ✅ Yes |
| **Cost** | Very cheap | More expensive |
| **Best for** | Simple transformations | Complex logic |

## 🔐 CloudFront Signed URLs & Signed Cookies

For **private content** (paid videos, premium downloads), you can restrict access using signed URLs or cookies.

### Signed URLs — Per-File Access

```python
import boto3
from datetime import datetime, timedelta
from botocore.signers import CloudFrontSigner
import rsa

def generate_signed_url(url, key_pair_id, private_key_path, expiry_hours=1):
    """
    Generate a signed CloudFront URL for private content.
    
    Args:
        url: The CloudFront URL to sign
        key_pair_id: The CloudFront key pair ID
        private_key_path: Path to the private key file
        expiry_hours: How long the URL is valid
    
    Returns:
        str: A signed URL that expires after the specified time
    """
    # Read the private key
    with open(private_key_path, 'rb') as key_file:
        private_key = rsa.PrivateKey.load_pkcs1(key_file.read())
    
    # Create a signer
    def rsa_signer(message):
        return rsa.sign(message, private_key, 'SHA-1')
    
    signer = CloudFrontSigner(key_pair_id, rsa_signer)
    
    # Set expiry time
    expire_date = datetime.utcnow() + timedelta(hours=expiry_hours)
    
    # Generate signed URL
    signed_url = signer.generate_presigned_url(
        url,
        date_less_than=expire_date
    )
    
    return signed_url

# Usage
signed = generate_signed_url(
    url="https://d1234567890.cloudfront.net/premium/video.mp4",
    key_pair_id="KAPKAJ1EXAMPLE",
    private_key_path="./private_key.pem",
    expiry_hours=2
)
print(f"Access this URL (valid for 2 hours): {signed}")
```

### When to Use What?

| Use Signed URLs When... | Use Signed Cookies When... |
|------------------------|---------------------------|
| Restricting access to individual files | Restricting access to multiple files |
| Users need to download a specific file | Users need access to an entire section |
| Client doesn't support cookies | Client supports cookies (web browsers) |
| Example: Download link for a purchased file | Example: Paid subscription video library |

# Part 2: AWS WAF (Web Application Firewall)

---

## 🛡️ What is AWS WAF?

AWS WAF is a **Web Application Firewall** that protects your web applications from common attacks. It sits in front of CloudFront, ALB, or API Gateway and inspects every incoming request.

### Think of It Like This:

Imagine a nightclub. CloudFront is the **nightclub** serving drinks (content) to customers (users). WAF is the **bouncer at the door** — checking IDs, dress codes, and making sure troublemakers don't get in.

```
Without WAF:
┌──────┐              ┌────────────┐
│ Good │ ──────────► │            │
│ User │              │ CloudFront │ → Everyone gets in
│      │              │ / ALB      │
│ Bad  │ ──────────► │            │
│ Actor│              └────────────┘
└──────┘

With WAF:
┌──────┐       ┌──────────┐       ┌────────────┐
│ Good │ ────► │          │ ────► │            │
│ User │       │   WAF    │       │ CloudFront │ → Only good traffic gets in
│      │       │ (Bouncer)│       │ / ALB      │
│ Bad  │ ──X─► │          │       │            │
│ Actor│       │ BLOCKED! │       └────────────┘
└──────┘       └──────────┘
```

---

## 🎯 What Attacks Does WAF Protect Against?

| Attack | What It Is | Example |
|--------|-----------|---------|
| **SQL Injection** | Inserting SQL code into inputs to manipulate databases | `' OR 1=1 --` in a login form |
| **XSS** (Cross-Site Scripting) | Injecting JavaScript into web pages | `<script>steal(cookies)</script>` |
| **DDoS** (Layer 7) | Flooding your app with HTTP requests | 1 million requests/second |
| **Bad Bots** | Automated scrapers, credential stuffers | Scraping your prices, brute-forcing logins |
| **Path Traversal** | Trying to access server files | `../../etc/passwd` |
| **IP-based attacks** | Attacks from known malicious IPs | Traffic from botnets |

## 🏗️ WAF Architecture — The Building Blocks

```
┌──────────────────────────────────────────────────────┐
│                   AWS WAF                             │
│                                                      │
│  ┌────────────────────────────────────────────────┐  │
│  │              WEB ACL                            │  │
│  │  (The main container — a set of rules)          │  │
│  │                                                  │  │
│  │  ┌──────────────────────────────────────────┐   │  │
│  │  │ Rule Group 1: AWS Managed Rules          │   │  │
│  │  │  ├── AWSManagedRulesCommonRuleSet        │   │  │
│  │  │  ├── AWSManagedRulesSQLiRuleSet          │   │  │
│  │  │  └── AWSManagedRulesKnownBadInputs       │   │  │
│  │  └──────────────────────────────────────────┘   │  │
│  │                                                  │  │
│  │  ┌──────────────────────────────────────────┐   │  │
│  │  │ Rule Group 2: Custom Rules               │   │  │
│  │  │  ├── Block specific countries            │   │  │
│  │  │  ├── Rate limiting (100 req/5min/IP)     │   │  │
│  │  │  └── Block specific User-Agents          │   │  │
│  │  └──────────────────────────────────────────┘   │  │
│  │                                                  │  │
│  │  Default Action: ALLOW (if no rules match)      │  │
│  │                                                  │  │
│  └────────────────────────────────────────────────┘  │
│                                                      │
│  Associated with: CloudFront / ALB / API Gateway     │
│                                                      │
└──────────────────────────────────────────────────────┘
```

### Key Components

| Component | What It Is |
|-----------|-----------|
| **Web ACL** | The top-level container. You attach it to CloudFront/ALB/API Gateway. Contains rules |
| **Rule** | A condition + action. "If request matches X, then ALLOW/BLOCK/COUNT" |
| **Rule Group** | A reusable collection of rules. AWS provides managed rule groups |
| **Managed Rules** | Pre-built rule groups by AWS and marketplace vendors |
| **IP Set** | A list of IP addresses to allow or block |
| **Regex Pattern Set** | A set of regex patterns to match against requests |

## 🛠️ Setting Up WAF — Step by Step

### Step 1: Create a Web ACL with AWS Managed Rules

```bash
# Create a Web ACL for CloudFront (must be in us-east-1)
aws wafv2 create-web-acl \
    --name "MyWebsite-WAF" \
    --scope "CLOUDFRONT" \
    --region "us-east-1" \
    --default-action '{"Allow": {}}' \
    --description "WAF for my website" \
    --rules '[
        {
            "Name": "AWS-AWSManagedRulesCommonRuleSet",
            "Priority": 1,
            "Statement": {
                "ManagedRuleGroupStatement": {
                    "VendorName": "AWS",
                    "Name": "AWSManagedRulesCommonRuleSet"
                }
            },
            "OverrideAction": {"None": {}},
            "VisibilityConfig": {
                "SampledRequestsEnabled": true,
                "CloudWatchMetricsEnabled": true,
                "MetricName": "AWSCommonRules"
            }
        },
        {
            "Name": "AWS-AWSManagedRulesSQLiRuleSet",
            "Priority": 2,
            "Statement": {
                "ManagedRuleGroupStatement": {
                    "VendorName": "AWS",
                    "Name": "AWSManagedRulesSQLiRuleSet"
                }
            },
            "OverrideAction": {"None": {}},
            "VisibilityConfig": {
                "SampledRequestsEnabled": true,
                "CloudWatchMetricsEnabled": true,
                "MetricName": "AWSSQLiRules"
            }
        },
        {
            "Name": "RateLimit",
            "Priority": 3,
            "Statement": {
                "RateBasedStatement": {
                    "Limit": 2000,
                    "AggregateKeyType": "IP"
```

**What each rule does:**

| Rule | What It Does |
|------|-------------|
| `AWSManagedRulesCommonRuleSet` | Blocks common exploits like XSS, path traversal, bad bots |

## 📊 AWS Managed Rule Groups — The Essentials

AWS provides several managed rule groups. Here are the most important ones:

| Rule Group | What It Protects Against | WCU Cost |
|-----------|------------------------|----------|
| `AWSManagedRulesCommonRuleSet` | XSS, path traversal, bad bots, HTTP violations | 700 |
| `AWSManagedRulesSQLiRuleSet` | SQL injection attacks | 200 |
| `AWSManagedRulesKnownBadInputsRuleSet` | Log4j, Java deserialization, other known exploits | 200 |
| `AWSManagedRulesAmazonIpReputationList` | IPs with bad reputation (botnets, scanners) | 25 |
| `AWSManagedRulesAnonymousIpList` | VPNs, Tor, proxies — anonymous access | 50 |

| `AWSManagedRulesWordPressRuleSet` | WordPress-specific attacks | 100 |

> **WCU = Web ACL Capacity Units.** Each Web ACL has a maximum of **5,000 WCU**. This limits how many rules you can add. Choose wisely!

## 🧠 Key Takeaways

| Concept | Remember This |
|---------|--------------|
| **CloudFront** | CDN with 450+ edge locations. Caches and delivers content globally |
| **Origin** | Where your original content lives (S3, ALB, any HTTP server) |
| **OAC** | Restricts S3 so only CloudFront can access it |
| **Invalidation** | Force-removes cached content. Use `/*` for everything (counts as 1 path) |
| **Cache Busting** | Better than invalidation — use hashed filenames |
| **CloudFront Functions** | Lightweight edge code (< 1ms). Great for headers, redirects |
| **Lambda@Edge** | Heavy edge code. Great for auth, image resizing, A/B testing |
| **WAF** | Web Application Firewall. Blocks SQL injection, XSS, bad bots, DDoS |
| **Web ACL** | Container for WAF rules. Attached to CloudFront/ALB/API Gateway |
| **Managed Rules** | Pre-built rule sets by AWS. Use `CommonRuleSet` + `SQLiRuleSet` minimum |
| **Rate Limiting** | Block IPs that make too many requests. Essential for DDoS protection |
| **Shield Standard** | Free DDoS protection that works automatically with CloudFront |

## ❓ Common Interview Questions

**Q1: What's the difference between CloudFront Functions and Lambda@Edge?**
> CloudFront Functions are ultra-lightweight (< 1ms, JavaScript only, no network access) and run at all 450+ edge locations — great for simple tasks like header manipulation and URL redirects. Lambda@Edge runs at 13 regional edge caches, supports Node.js/Python, can access network and request bodies, and runs up to 30 seconds — ideal for complex auth, image resizing, and A/B testing.

**Q2: How do you serve a React SPA through CloudFront?**
> Upload the build output to S3, create a CloudFront distribution with S3 as the origin, set `DefaultRootObject` to `index.html`, and create custom error responses mapping 403/404 to `/index.html` with status 200 (for client-side routing). Use OAC to restrict S3 access to CloudFront only. Use ACM for a free SSL cert (must be in us-east-1).

**Q3: How does WAF rate limiting work?**
> WAF evaluates requests over a 5-minute window. If a single IP exceeds the configured limit (e.g., 2000 requests), WAF blocks subsequent requests from that IP until the rate drops below the threshold. It uses the `RateBasedStatement` rule type and can aggregate by IP, forwarded IP, or custom keys.

**Q4: Can you use WAF without CloudFront?**
> Yes! WAF can be attached to ALB (Application Load Balancer), API Gateway, AppSync, and Cognito User Pools. Use `scope = "REGIONAL"` instead of `"CLOUDFRONT"`. This is useful when your application doesn't need a CDN but still needs protection.
