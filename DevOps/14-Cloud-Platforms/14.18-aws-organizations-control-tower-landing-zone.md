# 14.18 — AWS Organizations, Control Tower & Landing Zone

> **One Line Answer:** **AWS Organizations** lets you manage multiple AWS accounts from one central place with consolidated billing and policies, **Control Tower** automates the setup of a secure, well-architected multi-account environment, and **Landing Zone** is the overall design pattern for how your multi-account structure is organized — think of it as the blueprint for your entire AWS estate.

---

## 🤔 Why Do You Need Multiple AWS Accounts?

Let's start with the most common beginner question: **"Why not just use one AWS account for everything?"**

Imagine you have one house (one AWS account) where:
- Your **development team** is building new experiments in the living room
- Your **production workloads** are running in the kitchen
- Your **finance team** is managing billing in the bedroom
- Your **security team** is trying to lock everything down

What happens?

```
Single AWS Account — The Chaos:

┌──────────────────────────────────────────────────┐
│              ONE AWS ACCOUNT                      │
│                                                   │
│  ⚠️ Dev accidentally deletes prod database        │
│  ⚠️ Intern spins up expensive GPU instances       │
│  ⚠️ Can't tell which team is spending what        │
│  ⚠️ One security breach exposes EVERYTHING        │
│  ⚠️ IAM policies become impossibly complex        │
│  ⚠️ Service limits shared across all workloads    │
│  ⚠️ Compliance audit? Nightmare.                  │
│                                                   │
└──────────────────────────────────────────────────┘
```

Now imagine you have **separate houses** (accounts) for each purpose:

```
Multi-Account Strategy — The Order:

┌─────────────┐ ┌─────────────┐ ┌─────────────┐
│  Dev Account│ │ Staging Acct│ │ Prod Account│
│             │ │             │ │             │
│ Devs can    │ │ QA testing  │ │ Locked down │
│ experiment  │ │ environment │ │ Minimal     │
│ freely      │ │             │ │ access      │
└─────────────┘ └─────────────┘ └─────────────┘

┌─────────────┐ ┌─────────────┐ ┌─────────────┐
│ Security    │ │  Logging    │ │  Shared     │
│ Account     │ │  Account    │ │  Services   │
│             │ │             │ │             │
│ GuardDuty,  │ │ CloudTrail, │ │ DNS, AD,    │
│ SecurityHub │ │ Config logs │ │ VPN, Transit│
│ WAF rules   │ │ centralized │ │ Gateway     │
└─────────────┘ └─────────────┘ └─────────────┘
```

### Benefits of Multi-Account Strategy

| Benefit | Explanation |
|---------|------------|
| **Blast Radius Reduction** | If dev account is compromised, prod is untouched |
| **Cost Visibility** | Each account's costs are tracked separately |
| **Service Limits** | Each account has its own limits (more capacity!) |
| **Compliance** | Easier to prove which resources are in scope for audits |
| **Team Autonomy** | Each team manages their own account with guardrails |
| **Security Boundaries** | Strongest isolation AWS offers — accounts are hard boundaries |

## 📦 What is AWS Organizations?

AWS Organizations is a service that lets you **create and manage multiple AWS accounts** from a single central account (called the **management account** or **master account**).

Think of it like a **corporate headquarters** managing multiple branch offices:

### Key Terminology

| Term | What It Is | Analogy |
|------|-----------|---------|
| **Management Account** | The root/master account that created the Organization | CEO's office |

## 🛠️ Setting Up AWS Organizations

### Step 1: Create the Organization

```bash
# From your management account, create the organization
aws organizations create-organization --feature-set ALL

# Output:
# {
#     "Organization": {
#         "Id": "o-abc123def4",
#         "Arn": "arn:aws:organizations::111111111111:organization/o-abc123def4",
#         "FeatureSet": "ALL",
#         "MasterAccountId": "111111111111",
#         "MasterAccountEmail": "admin@mycompany.com"
#     }
# }
```

**Feature Sets:**

| Feature Set | What You Get |
|-------------|-------------|
| `ALL` | Full features including SCPs, tag policies, backup policies. **Use this.** |
| `CONSOLIDATED_BILLING` | Only billing consolidation, no SCPs. Limited and rarely used. |

### Step 2: Create Organizational Units (OUs)

```bash
# Get the root ID first
aws organizations list-roots
# Returns: r-abcd

# Create OUs under the root
aws organizations create-organizational-unit \
    --parent-id r-abcd \
    --name "Production"
# Returns: ou-abcd-11111111

aws organizations create-organizational-unit \
    --parent-id r-abcd \
    --name "Development"
# Returns: ou-abcd-22222222

aws organizations create-organizational-unit \
    --parent-id r-abcd \
    --name "Security"
# Returns: ou-abcd-33333333

aws organizations create-organizational-unit \
    --parent-id r-abcd \
    --name "SharedServices"
# Returns: ou-abcd-44444444

# You can also create nested OUs
aws organizations create-organizational-unit \
    --parent-id ou-abcd-11111111 \
    --name "Production-Critical"
```

### Step 3: Create Member Accounts

```bash
# Create a new AWS account within the Organization
aws organizations create-account \
    --email "prod@mycompany.com" \
    --account-name "Production Workloads"

# Output:
# {
#     "CreateAccountStatus": {
#         "Id": "car-abc123",
#         "AccountName": "Production Workloads",
#         "State": "IN_PROGRESS"
#     }
# }

# Check the status
aws organizations describe-create-account-status \
    --create-account-request-id "car-abc123"
# Wait until State is "SUCCEEDED"

# Create more accounts
aws organizations create-account \
    --email "dev@mycompany.com" \
    --account-name "Development"

aws organizations create-account \
    --email "security@mycompany.com" \
    --account-name "Security & Audit"

aws organizations create-account \
    --email "logging@mycompany.com" \
    --account-name "Log Archive"
```

> **Important:** Each account needs a **unique email address**. Many companies use email aliases: `aws+prod@mycompany.com`, `aws+dev@mycompany.com`, etc.

### Step 4: Move Accounts into OUs

```bash
# Move the production account into the Production OU
aws organizations move-account \
    --account-id "222222222222" \
    --source-parent-id "r-abcd" \
    --destination-parent-id "ou-abcd-11111111"

# Move the dev account into the Development OU
aws organizations move-account \
    --account-id "333333333333" \
    --source-parent-id "r-abcd" \
    --destination-parent-id "ou-abcd-22222222"
```

## 🛡️ Service Control Policies (SCPs) — The Guardrails

SCPs are the **most powerful feature** of AWS Organizations. They define the **maximum permissions** that accounts in the OU can have. Think of them as **boundaries** that even admin users can't cross.

### How SCPs Work

```
Without SCP:
IAM Policy says: "Allow EC2, S3, RDS, Lambda, everything"
Result: User can do anything their IAM policy allows ✅

With SCP:
SCP says: "Only allow EC2, S3, RDS"
IAM Policy says: "Allow EC2, S3, RDS, Lambda, everything"
Result: User can ONLY use EC2, S3, RDS ✅
         Lambda is BLOCKED even though IAM allows it ❌

┌─────────────────────────────────────────┐
│         EFFECTIVE PERMISSIONS            │
│                                         │
│    IAM Policy     ∩     SCP             │
│   ┌──────────┐   ┌──────────┐          │
│   │ EC2  ✅  │   │ EC2  ✅  │          │
│   │ S3   ✅  │   │ S3   ✅  │  = Only  │
│   │ RDS  ✅  │   │ RDS  ✅  │  EC2,S3, │
│   │ Lambda✅ │   │          │  RDS     │
│   │ DynamoDB✅│   │          │          │
│   └──────────┘   └──────────┘          │
│                                         │
│  SCP is the CEILING, not the FLOOR      │
│  SCP doesn't GRANT permissions,         │
│  it LIMITS them                         │
└─────────────────────────────────────────┘
```

> **Key Point:** SCPs don't grant permissions — they only **limit** what's possible. You still need IAM policies to actually give users access. SCPs set the boundary; IAM policies work within that boundary.

### SCP Examples

#### SCP 1: Deny Leaving the Organization

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "PreventLeavingOrganization",
            "Effect": "Deny",
            "Action": [
                "organizations:LeaveOrganization"
            ],
            "Resource": "*"
        }
    ]
}
```

**Why?** Prevents a rogue admin from detaching an account from the Organization, which would remove all SCPs and guardrails.

#### SCP 2: Restrict to Specific AWS Regions

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "DenyOutsideAllowedRegions",
            "Effect": "Deny",
            "NotAction": [
                "iam:*",
                "organizations:*",
                "sts:*",
                "support:*",
                "budgets:*",
                "cloudfront:*",
                "route53:*",
                "waf:*",
                "wafv2:*"
            ],
            "Resource": "*",
            "Condition": {
                "StringNotEquals": {
                    "aws:RequestedRegion": [
                        "us-east-1",
                        "us-west-2",
                        "eu-west-1"
                    ]
                }
            }
        }
    ]
}
```

**What this does:**
- ✅ Allows resources in `us-east-1`, `us-west-2`, `eu-west-1`
- ❌ Blocks creating resources in any other region (like `ap-southeast-1`)
- ✅ Allows global services (IAM, CloudFront, Route53) regardless of region
- 🎯 Prevents someone from spinning up instances in an unexpected region (common attack vector)

#### SCP 3: Prevent Disabling CloudTrail

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "ProtectCloudTrail",
            "Effect": "Deny",
            "Action": [
                "cloudtrail:StopLogging",
                "cloudtrail:DeleteTrail",
                "cloudtrail:UpdateTrail"
            ],
            "Resource": "*"
        }
    ]
}
```

**Why?** If an attacker gets into an account, the first thing they do is **turn off logging**. This SCP prevents anyone — even root — from stopping CloudTrail.

#### SCP 4: Deny Root User Access

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "DenyRootUser",
            "Effect": "Deny",
            "Action": "*",
            "Resource": "*",
            "Condition": {
                "StringLike": {
                    "aws:PrincipalArn": "arn:aws:iam::*:root"
                }
            }
        }
    ]
}
```

**Why?** The root user has unlimited power. By denying all actions for root via SCP, you force everyone to use IAM users/roles. Root should only be used for account recovery.

#### SCP 5: Prevent Creating IAM Users (Force SSO)

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "DenyIAMUserCreation",
            "Effect": "Deny",
            "Action": [
                "iam:CreateUser",
                "iam:CreateAccessKey"
            ],
            "Resource": "*"
        }
    ]
}
```

**Why?** Forces everyone to use AWS IAM Identity Center (SSO) instead of creating per-account IAM users. Centralizes identity management.

### Applying SCPs

```bash
# Create the SCP
aws organizations create-policy \
    --name "DenyRootAccess" \
    --description "Prevents root user from performing any actions" \
    --type "SERVICE_CONTROL_POLICY" \
    --content file://deny-root-scp.json

# Attach SCP to an OU (applies to ALL accounts in that OU)
aws organizations attach-policy \
    --policy-id "p-abc123" \
    --target-id "ou-abcd-11111111"

# Attach SCP to a specific account
aws organizations attach-policy \
    --policy-id "p-abc123" \
    --target-id "222222222222"

# SCPs are inherited! If you attach to Production OU,
# all accounts inside it (and nested OUs) get the policy
```

### SCP Inheritance Flow

```
Root (r-abcd)
├── SCP: FullAWSAccess (default)
│
├── OU: Production
│   ├── SCP: DenyRootAccess
│   ├── SCP: RestrictRegions  
│   │
│   ├── Account: Prod-App1
│   │   └── Effective: FullAccess ∩ DenyRoot ∩ RestrictRegions
│   │
│   └── OU: Production-Critical
│       ├── SCP: DenyDeleteResources
│       │
│       └── Account: Prod-Database
│           └── Effective: FullAccess ∩ DenyRoot ∩ RestrictRegions ∩ DenyDelete
│                          (All parent SCPs stack up!)
│
└── OU: Development
    ├── SCP: LimitInstanceTypes (max t3.xlarge)
    │
    └── Account: Dev-Team1
        └── Effective: FullAccess ∩ LimitInstanceTypes
```

## 💰 Consolidated Billing

One of the simplest but most valuable features of Organizations:

```
Without Organizations:
┌──────────────────┐
│ Account 1: $500  │ → Bill 1
│ Account 2: $300  │ → Bill 2  
│ Account 3: $200  │ → Bill 3
│                  │
│ Total: $1,000    │ No volume discounts
└──────────────────┘

With Organizations (Consolidated Billing):
┌──────────────────────────────┐
│ Management Account           │
│                              │
│ Account 1: $500 ─┐          │
│ Account 2: $300 ─┼── $1,000 │ → Single Bill
│ Account 3: $200 ─┘          │   Volume discounts!
│                              │   Reserved Instances shared!
│ Savings Plan shared across  │   Savings Plans shared!
│ all accounts!               │
└──────────────────────────────┘
```

### Key Benefits:
- **One bill** for all accounts
- **Volume discounts** — combined usage gets cheaper per-unit pricing
- **Reserved Instance sharing** — buy an RI in one account, use it in another
- **Savings Plans sharing** — same as above
- **Per-account cost tracking** — still see costs broken down by account

## 🔄 Cross-Account Access with Organizations

### Using IAM Roles for Cross-Account Access

```json
// In the PROD account (222222222222) — create a role
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Principal": {
                "AWS": "arn:aws:iam::111111111111:root"
            },
            "Action": "sts:AssumeRole",
            "Condition": {
                "StringEquals": {
                    "aws:PrincipalOrgID": "o-abc123def4"
                }
            }
        }
    ]
}
```

**What this does:** Allows users from the management account (111111111111) to assume a role in the prod account — but ONLY if they're part of the same Organization. The `aws:PrincipalOrgID` condition is a security best practice.

### Switching Roles via CLI

```bash
# From the management account, assume a role in the prod account
aws sts assume-role \
    --role-arn "arn:aws:iam::222222222222:role/OrganizationAdmin" \
    --role-session-name "admin-session"

# Or configure it in ~/.aws/config for easy switching:
# [profile prod]
# role_arn = arn:aws:iam::222222222222:role/OrganizationAdmin
# source_profile = default

# Then use it:
aws s3 ls --profile prod
```

## 📦 What is AWS Control Tower?

If AWS Organizations is the **tool** for managing multiple accounts, **Control Tower** is the **automated setup wizard** that creates a well-architected multi-account environment for you.

Think of it like this:

| Analogy | Organizations | Control Tower |
|---------|--------------|---------------|
| Building a house | Buying individual bricks, cement, and doing it yourself | Hiring a contractor who follows best practices automatically |
| Cooking | Buying ingredients and cooking from scratch | Using a meal kit with pre-portioned ingredients and step-by-step instructions |

### What Control Tower Does Automatically

```
When you enable Control Tower, it:

1. ✅ Creates the Organization (if not already)
2. ✅ Creates two core accounts:
   ├── Log Archive Account (centralized logging)
   └── Audit Account (security audit & compliance)
3. ✅ Sets up foundational OUs:
   ├── Security OU (for core accounts)
   └── Sandbox OU (for experiments)
4. ✅ Applies mandatory guardrails (SCPs)
5. ✅ Enables CloudTrail in all accounts
6. ✅ Configures AWS Config in all accounts
7. ✅ Sets up IAM Identity Center (SSO)
8. ✅ Creates a Landing Zone dashboard
9. ✅ Enables drift detection
```

## 🛠️ Setting Up Control Tower

### Through the AWS Console (Recommended)

1. Go to **AWS Console → Control Tower**
2. Click **"Set up landing zone"**
3. Review the pricing and prerequisites
4. Configure:
   - **Home Region** (where Control Tower manages from)
   - **Additional Regions** (optional — which other regions to govern)
   - **Log Archive account email** (e.g., `aws-logs@mycompany.com`)
   - **Audit account email** (e.g., `aws-audit@mycompany.com`)
5. Review and click **"Set up landing zone"**
6. Wait ~30-60 minutes for setup to complete

### Using Terraform (AWS Control Tower Account Factory for Terraform — AFT)

```hcl
# AFT is the official way to create accounts in Control Tower via Terraform
```

## 🛡️ Control Tower Guardrails (Controls)

Guardrails are the rules Control Tower enforces across your accounts. They come in three types:

### Types of Guardrails

| Type | How It Works | Enforcement |
|------|-------------|-------------|
| **Preventive** | Uses SCPs to prevent actions | Hard block — can't be bypassed |
| **Detective** | Uses AWS Config rules to detect non-compliance | Alerts — shows violations but doesn't block |
| **Proactive** | Uses CloudFormation hooks to check before creation | Pre-creation validation |

### Guardrail Categories

| Category | Meaning |
|----------|---------|
| **Mandatory** | Always enabled. Can't be turned off. Core security |
| **Strongly Recommended** | Not enabled by default, but AWS strongly suggests them |
| **Elective** | Optional. Enable based on your needs |

### Key Mandatory Guardrails

| Guardrail | What It Does |
|-----------|-------------|
| Disallow changes to CloudTrail configuration | Nobody can turn off logging |
| Disallow changes to AWS Config rules | Nobody can disable compliance checking |
| Disallow deletion of log archive S3 buckets | Logs can never be deleted |
| Enable CloudTrail in all available regions | Logging is always on everywhere |
| Enable integrity validation for CloudTrail | Logs can't be tampered with |

### Key Strongly Recommended Guardrails

| Guardrail | What It Does |
|-----------|-------------|
| Disallow public read access to S3 buckets | Prevents data leaks |
| Disallow public write access to S3 buckets | Prevents data corruption |
| Enable MFA for root user | Adds extra security to root |
| Disallow internet gateways | Prevents direct internet access (for sensitive OUs) |
| Enable encryption for EBS volumes at rest | Data at rest is always encrypted |

### Enabling Guardrails via CLI

```bash
# Enable a guardrail on an OU
aws controltower enable-control \
    --control-identifier "arn:aws:controltower:us-east-1::control/AWS-GR_DISALLOW_PUBLIC_S3" \
    --target-identifier "arn:aws:organizations::111111111111:ou/o-abc123/ou-abcd-11111111"

# List enabled guardrails
aws controltower list-enabled-controls \
    --target-identifier "arn:aws:organizations::111111111111:ou/o-abc123/ou-abcd-11111111"
```

## 🔐 IAM Identity Center (SSO) — Centralized Access

Control Tower sets up **IAM Identity Center** (formerly AWS SSO) so users can log in once and access multiple accounts.

```
Without SSO:
┌──────┐    User: admin-dev    ┌─────────────┐
│ User │ ──── Password A ────► │ Dev Account  │
│      │                       └─────────────┘
│      │    User: admin-prod   ┌─────────────┐
│      │ ──── Password B ────► │ Prod Account │
│      │                       └─────────────┘
│      │    User: admin-sec    ┌─────────────┐
│      │ ──── Password C ────► │ Sec Account  │
└──────┘                       └─────────────┘
😫 Three usernames! Three passwords! Three MFA tokens!

With IAM Identity Center:
┌──────┐                    ┌─────────────────────┐
│ User │ ── One Login ────► │ IAM Identity Center │
│      │    (+ one MFA)     │ (SSO Portal)        │
└──────┘                    │                     │
                            │ Choose your account: │
                            │ ┌─────────────────┐  │
                            │ │ Dev Account      │──► Dev (ReadOnly role)

                            │ │ Security Account │──► Security (Auditor role)
                            │ └─────────────────┘  │
                            └─────────────────────┘
```

### Setting Up Permission Sets

Permission Sets define what a user can do when they access an account:

```bash
# Create a Permission Set (like a role template)
aws sso-admin create-permission-set \
    --instance-arn "arn:aws:sso:::instance/ssoins-1234567890" \
    --name "DeveloperAccess" \
    --description "Full access for developers in dev accounts" \
    --session-duration "PT8H"

# Attach a managed policy to the permission set
aws sso-admin attach-managed-policy-to-permission-set \
    --instance-arn "arn:aws:sso:::instance/ssoins-1234567890" \
    --permission-set-arn "arn:aws:sso:::permissionSet/ssoins-1234567890/ps-abc123" \
    --managed-policy-arn "arn:aws:iam::aws:policy/PowerUserAccess"

# Assign the permission set to a group for a specific account
aws sso-admin create-account-assignment \
    --instance-arn "arn:aws:sso:::instance/ssoins-1234567890" \
    --target-id "333333333333" \
    --target-type "AWS_ACCOUNT" \
    --permission-set-arn "arn:aws:sso:::permissionSet/ssoins-1234567890/ps-abc123" \
    --principal-type "GROUP" \
    --principal-id "g-1234567890"
```

### Transit Gateway Architecture

```
┌──────────────────────────────────────────────────────────────┐
│                   NETWORK ACCOUNT (HUB)                       │
│                                                               │
│  ┌────────────────────────────────────────────────────────┐  │
│  │               AWS TRANSIT GATEWAY                       │  │
│  │                                                         │  │

│  │                                                         │  │
│  └──────┬──────────┬──────────┬──────────┬───────────────┘  │

│  │ VPN /   │ │ Shared  │ │ Prod    │ │ Dev        │       │
│  │ Direct  │ │ Services│ │ VPC     │ │ VPC        │       │
│  │ Connect │ │ VPC     │ │         │ │            │       │
│  │         │ │         │ │         │ │            │       │
```

### Terraform for Transit Gateway

```hcl
# In the Network account
resource "aws_ec2_transit_gateway" "main" {
  description = "Central Transit Gateway"
  
  default_route_table_association = "disable"
  default_route_table_propagation = "disable"
  
  tags = {
    Name = "central-tgw"
  }

# Share the Transit Gateway with other accounts via RAM
resource "aws_ram_resource_share" "tgw" {
  name                      = "transit-gateway-share"
  allow_external_principals = false  # Only within the Organization

  tags = {
    Name = "TGW-Share"
  }
}

  resource_arn       = aws_ec2_transit_gateway.main.arn
  resource_share_arn = aws_ram_resource_share.tgw.arn
}

resource "aws_ram_principal_association" "org" {
  principal          = "arn:aws:organizations::111111111111:organization/o-abc123def4"
  resource_share_arn = aws_ram_resource_share.tgw.arn
}

# Create route tables for different traffic flows
resource "aws_ec2_transit_gateway_route_table" "prod" {
  transit_gateway_id = aws_ec2_transit_gateway.main.id

}

resource "aws_ec2_transit_gateway_route_table" "dev" {
  transit_gateway_id = aws_ec2_transit_gateway.main.id

  transit_gateway_id = aws_ec2_transit_gateway.main.id
```

## 🔍 Drift Detection — Are Accounts Still Compliant?

Control Tower continuously monitors for **drift** — when something changes from the expected configuration.

### Types of Drift

| Drift Type | What Happened |
|-----------|--------------|
| Moved account | An account was moved to a different OU |
| Removed account | An account was removed from the Organization |
| SCP changes | An SCP was modified or detached |
| CloudTrail changes | CloudTrail logging was modified |
| Config changes | AWS Config rules were modified |

```bash
# Check Landing Zone drift status
aws controltower get-landing-zone \
    --landing-zone-identifier "arn:aws:controltower:us-east-1:111111111111:landingzone/abc123"

# If drift is detected, you'll see:
# "DriftStatus": {
#     "DriftType": "IN_SYNC"  or  "DRIFTED"
# }
```

When drift is detected, Control Tower shows it in the dashboard and you can **reset** the configuration to bring everything back in line.

## ❓ Common Interview Questions

**Q1: Why use multiple AWS accounts instead of one?**
> Multiple accounts provide the strongest isolation boundary AWS offers. Benefits include: reduced blast radius (compromise in dev doesn't affect prod), clear cost attribution per team/project, separate service limits per account, easier compliance (audit scope is per-account), and team autonomy with guardrails. AWS itself recommends multi-account strategy as a best practice.

**Q2: What's the difference between SCPs and IAM policies?**
> SCPs set the **maximum boundary** of what's possible in an account — they're a ceiling, not a floor. They don't grant permissions; they restrict them. IAM policies grant actual permissions within that boundary. Effective permissions = IAM Policy ∩ SCP. Even if IAM says "Allow *", if the SCP denies it, it's denied. SCPs don't affect the management account — only member accounts.

**Q3: What does Control Tower set up automatically?**
> Control Tower creates: (1) An Organization with a Security OU and Sandbox OU, (2) A Log Archive account for centralized CloudTrail and Config logs, (3) An Audit account for security tooling, (4) Mandatory guardrails (protect logging, prevent SCP changes), (5) IAM Identity Center (SSO) for centralized access, (6) AWS Config rules for compliance monitoring, and (7) A dashboard showing compliance status and drift detection.

**Q4: How do you handle networking across multiple accounts?**
> Use a dedicated Network account with a **Transit Gateway** as a hub. Each account's VPC attaches to the Transit Gateway via Transit Gateway Attachments. Route tables control which VPCs can communicate. Typically: prod can reach shared services but not dev, dev can reach shared services but not prod. Use AWS RAM (Resource Access Manager) to share the Transit Gateway across accounts in the Organization.

**Q5: What is Landing Zone drift and how do you fix it?**
> Drift occurs when the actual configuration of your landing zone deviates from the expected baseline — for example, if someone modifies an SCP, moves an account to a different OU, or changes CloudTrail settings. Control Tower's drift detection monitors for these changes and alerts you via the dashboard. To fix drift, you can use the "Reset" or "Re-register" option in Control Tower to bring the configuration back to the expected state.
