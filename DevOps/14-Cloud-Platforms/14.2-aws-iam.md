# 14.2 — AWS — Account Setup, IAM (Users, Groups, Roles, Policies)

> **One Line Answer:** IAM (Identity and Access Management) is AWS's **security gatekeeper** — it controls **WHO** can access your AWS account and **WHAT** they can do. Think of it as the **bouncer + key card system** for your entire cloud infrastructure.

---

## 🤔 Let's Start With the Big Picture

Imagine you own a huge office building (your AWS account). You need to control:

- **Who can enter the building?** (Authentication — proving who you are)
- **Which rooms can they access?** (Authorization — what you're allowed to do)
- **Can someone enter temporarily without a permanent badge?** (Roles — temporary access)

**IAM handles ALL of this for AWS.**

Without IAM, anyone with your AWS credentials could:
- Launch 1000 expensive servers 💸
- Delete all your databases 💀
- Access all your customer data 😱

**IAM is NOT optional. It's the FIRST thing you set up on AWS.**

## 🏁 Part 1: AWS Account Setup

### Step 1: Create an AWS Account

1. Go to [https://aws.amazon.com](https://aws.amazon.com)
2. Click **"Create an AWS Account"**
3. Enter your email address and account name
4. Provide payment information (credit/debit card — they charge $1 for verification, then refund it)
5. Choose a support plan (start with **Basic — FREE**)
6. Done! You now have an AWS account 🎉

### ⚠️ The Root User — Handle With Extreme Care!

The **Root User** is the email address you used to create the AWS account. It has **GOD-LEVEL access** — it can do absolutely anything, including:
- Close the entire AWS account
- Change the billing information
- Access every single service and resource

**Golden Rules for Root User:**

```
╔═══════════════════════════════════════════════════════════╗
║  ROOT USER BEST PRACTICES — FOLLOW THESE RELIGIOUSLY!    ║
╠═══════════════════════════════════════════════════════════╣
║  1. NEVER use root user for daily tasks                  ║
║  2. Enable MFA (Multi-Factor Authentication) IMMEDIATELY ║
║  3. Create an IAM admin user and use THAT instead        ║
║  4. Don't create access keys for root user               ║
║  5. Use root only for account-level tasks                ║
║     (like changing billing or closing the account)       ║
╚═══════════════════════════════════════════════════════════╝
```

### Step 2: Enable MFA on Root User

MFA = Multi-Factor Authentication = Something you **know** (password) + Something you **have** (phone/hardware key)

```
How to enable MFA:
──────────────────
1. Sign in to AWS Console as Root User
2. Go to → IAM Dashboard (or search "IAM" in the search bar)
3. Click → "Add MFA" under Security Recommendations
4. Choose MFA device type:
   → Virtual MFA device (Google Authenticator, Authy) ← Most common
   → Hardware MFA device (YubiKey)
   → SMS (not recommended)
5. Scan the QR code with your authenticator app
6. Enter two consecutive MFA codes
7. Done! Root user is now protected 🔐
```

### Step 3: Create an IAM Admin User (Your Daily Driver)

```
Why?
────
Root user = Master key 🔑 → Keep it locked away
IAM Admin = Manager key 🗝️ → Use this for daily work

How to create:
──────────────
1. Go to IAM Console → Users → "Create user"
2. Username: "admin" (or your name, e.g., "ritesh-admin")
3. Check ✅ "Provide user access to the AWS Management Console"
4. Set a password
5. Click Next → Attach policies directly
6. Search and select: "AdministratorAccess" ← This gives full access
7. Create user
8. Sign out of root, sign back in as your new IAM user 🎉
```

## 🔐 Part 2: AWS IAM — Deep Dive

### What is IAM?

> **AWS Identity and Access Management (IAM)** is a FREE service that lets you securely control access to AWS services and resources.

**IAM answers two questions:**
1. **Authentication** — WHO are you? (Identity verification)
2. **Authorization** — WHAT are you allowed to do? (Permissions)

### IAM Architecture — The Big Picture


| Component | What It Is | Real-World Analogy |

| **Policies** | JSON documents that define permissions | A list of rules on what each badge can unlock |

## 👤 IAM Users

> An **IAM User** represents a **person or application** that interacts with AWS.

### Key Points:
- Each user has a **unique name** within the AWS account
- Users can access AWS via:
  - **Console access** — Username + Password (for humans using the web console)
  - **Programmatic access** — Access Key ID + Secret Access Key (for CLI, SDK, API)
- **New users have NO permissions by default** — they can't do anything until you grant them permissions
- You can have up to **5,000 IAM users** per account

### Creating a User (AWS CLI):

```bash
# Create a new IAM user
aws iam create-user --user-name ritesh

# Output:
# {
#     "User": {
#         "UserName": "ritesh",
#         "UserId": "AIDAJEXAMPLEID",
#         "Arn": "arn:aws:iam::123456789012:user/ritesh",
#         "Path": "/",
#         "CreateDate": "2024-01-15T10:30:00Z"
#     }
# }
```

Let's break down that output:
- **UserName**: The name you gave the user
- **UserId**: A unique ID AWS generates internally
- **Arn**: Amazon Resource Name — a unique identifier for EVERY resource in AWS
- **Path**: Used for organizing users (like folders). Default is `/`

### Creating Console Access (Password):

```bash
# Give the user a password for console login
aws iam create-login-profile \
    --user-name ritesh \
    --password "MyStr0ng!P@ssw0rd" \
    --password-reset-required
    
# --password-reset-required forces the user to change password on first login
# This is a security best practice!
```

### Creating Programmatic Access (Access Keys):

```bash
# Create access keys for CLI/API access
aws iam create-access-key --user-name ritesh

# Output:
# {
#     "AccessKey": {
#         "UserName": "ritesh",
#         "AccessKeyId": "AKIAIOSFODNN7EXAMPLE",
#         "SecretAccessKey": "wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY",
#         "Status": "Active"
#     }
# }

# ⚠️ IMPORTANT: Save the SecretAccessKey NOW!
# AWS will NEVER show it again. If you lose it, you must create a new one.
```

**Understanding Access Keys:**

```
Access Key ID:     AKIAIOSFODNN7EXAMPLE        ← This is like your USERNAME
Secret Access Key: wJalrXUtnFEMI/K7MDENG/...   ← This is like your PASSWORD

You need BOTH to authenticate via CLI/API.
NEVER share your Secret Access Key. NEVER commit it to Git!
```

### Configuring AWS CLI with Access Keys:

```bash
# Configure AWS CLI on your machine
aws configure

# It will ask:
# AWS Access Key ID [None]: AKIAIOSFODNN7EXAMPLE
# AWS Secret Access Key [None]: wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY
# Default region name [None]: us-east-1
# Default output format [None]: json

# This stores your credentials in ~/.aws/credentials
# and config in ~/.aws/config

# Verify it works:
aws sts get-caller-identity
# {
#     "UserId": "AIDAJEXAMPLEID",
#     "Account": "123456789012",
#     "Arn": "arn:aws:iam::123456789012:user/ritesh"
# }
```

### Best Practices for Users:

```
✅ DO:
  → Create individual users for each person (don't share accounts)
  → Use strong passwords
  → Enable MFA for all users
  → Rotate access keys regularly (every 90 days)
  → Delete unused users and access keys

❌ DON'T:
  → Share access keys between users
  → Embed access keys in code (use IAM Roles or environment variables)
  → Give users more permissions than they need
  → Use the root user for daily tasks
```

## 👥 IAM Groups

> A **Group** is a **collection of IAM users**. You attach policies to groups, and all users in that group inherit those permissions.

### Why Groups?

Imagine you have 50 developers. Without groups:

```
❌ Without Groups:
   → Attach "EC2FullAccess" policy to user-1
   → Attach "EC2FullAccess" policy to user-2
   → Attach "EC2FullAccess" policy to user-3
   → ... (repeat 50 times!)
   → New developer joins? Manually add all policies again.
   → Policy needs to change? Update all 50 users!
   → 😱 Nightmare!
```

With groups:

```
✅ With Groups:
   → Create group: "Developers"
   → Attach "EC2FullAccess" policy to the group
   → Add all 50 users to the "Developers" group
   → All 50 users automatically get EC2 access!
   → New developer joins? Just add them to the group.
   → Policy needs to change? Update it once on the group.
   → 😎 Easy!
```

### Group Characteristics:
- A user can belong to **multiple groups** (e.g., "Developers" AND "DevOps")
- Groups **CANNOT be nested** (a group cannot contain another group)
- Groups are for **users only** — you cannot add roles or other AWS services to a group
- A group can have **up to 10 policies** attached
- An account can have up to **300 groups**

### Creating Groups (AWS CLI):

```bash
# Create a group
aws iam create-group --group-name Developers

# Attach a policy to the group
aws iam attach-group-policy \
    --group-name Developers \
    --policy-arn arn:aws:iam::aws:policy/AmazonEC2FullAccess

# Add a user to the group
aws iam add-user-to-group \
    --user-name ritesh \
    --group-name Developers

# List users in the group
aws iam get-group --group-name Developers

# List groups a user belongs to
aws iam list-groups-for-user --user-name ritesh
```

### Common Group Structure:

```
AWS Account
├── Group: Admins
│   ├── Policy: AdministratorAccess
│   └── Users: alice, bob (DevOps leads)
│
├── Group: Developers
│   ├── Policy: AmazonEC2FullAccess
│   ├── Policy: AmazonS3ReadOnlyAccess
│   └── Users: charlie, dave, eve, frank
│
├── Group: QA
│   ├── Policy: AmazonEC2ReadOnlyAccess
│   ├── Policy: CloudWatchReadOnlyAccess
│   └── Users: grace, henry
│
├── Group: Finance
│   ├── Policy: AWSBillingReadOnlyAccess
│   └── Users: irene
│
└── Group: Interns
    ├── Policy: ReadOnlyAccess
    └── Users: jack, kate
```

## 🎭 IAM Roles

> A **Role** is an IAM identity with **temporary credentials** that can be **assumed** by users, applications, or AWS services.

### The Key Difference: Users vs Roles

| Feature | User | Role |
|---------|------|------|
| **Who uses it?** | A specific person or app | Anyone/anything that is allowed |
| **Credentials** | Permanent (password/access keys) | Temporary (auto-expire) |
| **Duration** | Until you delete or rotate them | Minutes to hours (configurable) |
| **Best for** | Humans, long-term identities | Services, cross-account, temporary access |

### Why Do We Need Roles?

**Scenario 1: EC2 Instance Needs to Access S3**

```
❌ Bad Way (Without Roles):
   → Create an IAM user
   → Generate access keys
   → SSH into the EC2 instance
   → Run "aws configure" and paste the access keys
   → Keys are now STORED on the server 😱
   → If someone hacks the server, they get the keys!

✅ Good Way (With Roles):
   → Create an IAM Role with S3 permissions
   → Attach the Role to the EC2 instance
   → EC2 automatically gets temporary credentials
   → No keys to manage, no keys to leak!
   → Credentials auto-rotate every few hours
```

**Scenario 2: Cross-Account Access**

```
Your company has TWO AWS accounts:
   → Account A: Development (Account ID: 111111111111)
   → Account B: Production  (Account ID: 222222222222)

A developer in Account A needs to access resources in Account B.

Solution:
   → Create a Role in Account B that Account A users can assume
   → Developer in Account A "assumes" the role
   → Gets temporary credentials for Account B
   → Does their work, credentials expire automatically
```

**Scenario 3: AWS Services Need Permissions**

```
AWS Lambda function needs to read from DynamoDB:
   → Lambda CAN'T use access keys (it's a service, not a person)
   → Create an IAM Role: "LambdaDynamoDBRole"
   → Attach DynamoDB read permissions to the role
   → Assign the role to the Lambda function
   → Lambda automatically gets temporary credentials to read DynamoDB
```

### Creating a Role (AWS CLI):

**Step 1: Create a Trust Policy (WHO can assume this role):**

```json
// trust-policy.json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Principal": {
                "Service": "ec2.amazonaws.com"
            },
            "Action": "sts:AssumeRole"
        }
    ]
}
```

Let's break this down:
- **Version**: Policy language version (always use "2012-10-17")
- **Statement**: Array of permission rules
- **Effect**: "Allow" or "Deny"
- **Principal**: WHO can assume this role — here, the EC2 service
- **Action**: "sts:AssumeRole" — this is what allows assuming the role

**Step 2: Create the Role:**

```bash
# Create the role with the trust policy
aws iam create-role \
    --role-name EC2-S3-Access-Role \
    --assume-role-policy-document file://trust-policy.json

# Attach a permissions policy to the role
aws iam attach-role-policy \
    --role-name EC2-S3-Access-Role \
    --policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess

# Create an instance profile (needed to attach role to EC2)
aws iam create-instance-profile \
    --instance-profile-name EC2-S3-Access-Profile

# Add the role to the instance profile
aws iam add-role-to-instance-profile \
    --instance-profile-name EC2-S3-Access-Profile \
    --role-name EC2-S3-Access-Role
```

**Step 3: Attach the Role to an EC2 Instance:**

```bash
# When launching a new EC2 instance
aws ec2 run-instances \
    --image-id ami-0abcdef1234567890 \
    --instance-type t3.micro \
    --iam-instance-profile Name=EC2-S3-Access-Profile

# Or attach to an existing instance
aws ec2 associate-iam-instance-profile \
    --instance-id i-1234567890abcdef0 \
    --iam-instance-profile Name=EC2-S3-Access-Profile
```

### Assuming a Role (for Users):

```bash
# A user assumes a role to get temporary credentials
aws sts assume-role \
    --role-arn arn:aws:iam::222222222222:role/CrossAccountRole \
    --role-session-name my-session

# Output:
# {
#     "Credentials": {
#         "AccessKeyId": "ASIAJEXAMPLEID",        ← Temporary Access Key
#         "SecretAccessKey": "wJalrXUtnFEMI/...",  ← Temporary Secret
#         "SessionToken": "FwoGZXIvYXdzEE...",     ← Session Token (required!)
#         "Expiration": "2024-01-15T12:30:00Z"     ← Expires automatically!
#     }
# }

# Use these temporary credentials:
export AWS_ACCESS_KEY_ID="ASIAJEXAMPLEID"
export AWS_SECRET_ACCESS_KEY="wJalrXUtnFEMI/..."
export AWS_SESSION_TOKEN="FwoGZXIvYXdzEE..."

# Now your CLI commands use the assumed role's permissions
aws s3 ls  # Works if the role has S3 permissions
```

### Common Role Types:

| Role Type | Who Assumes It | Example Use Case |
|-----------|---------------|-----------------|
| **Service Role** | AWS Services (EC2, Lambda, ECS) | EC2 instance accessing S3 |
| **Cross-Account Role** | Users/services in another AWS account | Dev account accessing Prod |
| **Identity Provider Role** | External users (Google, SAML, OIDC) | SSO login for employees |
| **Service-Linked Role** | Specific AWS service (auto-created) | AWS Autoscaling managing EC2 |

## 📜 IAM Policies — The Heart of Permissions

> A **Policy** is a **JSON document** that defines **what actions are allowed or denied** on **which AWS resources**.

### Policy Structure:

Every IAM policy has this structure:

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "DescriptiveStatementName",
            "Effect": "Allow",
            "Action": [
                "service:ActionName"
            ],
            "Resource": [
                "arn:aws:service:region:account:resource"
            ],
            "Condition": {
                "ConditionOperator": {
                    "ConditionKey": "ConditionValue"
                }
            }
        }
    ]
}
```

Let's break down each part:

| Field | What It Means | Example |
|-------|--------------|---------|
| **Version** | Policy language version | Always `"2012-10-17"` |
| **Statement** | Array of individual permission rules | Can have multiple statements |
| **Sid** | Statement ID — a label for the rule (optional) | `"AllowS3Read"` |
| **Effect** | Allow or Deny | `"Allow"` or `"Deny"` |
| **Action** | What API actions are permitted/denied | `"s3:GetObject"`, `"ec2:StartInstances"` |
| **Resource** | Which specific resources the rule applies to | `"arn:aws:s3:::my-bucket/*"` |
| **Condition** | Extra conditions that must be true (optional) | Only from specific IP, only with MFA |

### Example Policies (From Simple to Complex):

**Example 1: Allow Reading Objects from a Specific S3 Bucket**

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "AllowReadFromMyBucket",
            "Effect": "Allow",
            "Action": [
                "s3:GetObject",
                "s3:ListBucket"
            ],
            "Resource": [
                "arn:aws:s3:::my-company-data",
                "arn:aws:s3:::my-company-data/*"
            ]
        }
    ]
}
```

Breaking it down:
- **Effect**: Allow ✅
- **Action**: `s3:GetObject` (download files), `s3:ListBucket` (list files)
- **Resource**: Only the bucket `my-company-data` and everything inside it (`/*`)

Note: We need TWO resources here:
- `arn:aws:s3:::my-company-data` — The bucket itself (for ListBucket)
- `arn:aws:s3:::my-company-data/*` — The objects inside (for GetObject)

**Example 2: Allow EC2 Management (Start, Stop, Describe)**

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "AllowEC2Management",
            "Effect": "Allow",
            "Action": [
                "ec2:DescribeInstances",
                "ec2:StartInstances",
                "ec2:StopInstances",
                "ec2:RebootInstances"
            ],
            "Resource": "*"
        }
    ]
}
```

- `Resource: "*"` means ALL EC2 instances. You can restrict to specific instances using their ARN.

**Example 3: Deny Deleting ANY Resource (Guardrail Policy)**

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "DenyAllDeletions",
            "Effect": "Deny",
            "Action": [
                "ec2:TerminateInstances",
                "s3:DeleteBucket",
                "s3:DeleteObject",
                "rds:DeleteDBInstance"
            ],
            "Resource": "*"
        }
    ]
}
```

- **Deny ALWAYS wins over Allow.** Even if another policy says "Allow", this Deny will override it.

**Example 4: Policy with Conditions (Allow EC2 Only in Specific Region)**

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "AllowEC2OnlyInUSEast1",
            "Effect": "Allow",
            "Action": "ec2:*",
            "Resource": "*",
            "Condition": {
                "StringEquals": {
                    "aws:RequestedRegion": "us-east-1"
                }
            }
        }
    ]
}
```

- This allows ALL EC2 actions, but ONLY in the `us-east-1` region.
- If someone tries to launch an instance in `eu-west-1`, it will be **denied**.

**Example 5: Require MFA for Sensitive Actions**

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "AllowAllActionsWithMFA",
            "Effect": "Allow",
            "Action": "*",
            "Resource": "*",
            "Condition": {
                "Bool": {
                    "aws:MultiFactorAuthPresent": "true"
                }
            }
        },
        {
            "Sid": "DenyAllWithoutMFA",
            "Effect": "Deny",
            "Action": "*",
            "Resource": "*",
            "Condition": {
                "BoolIfExists": {
                    "aws:MultiFactorAuthPresent": "false"
                }
            }
        }
    ]
}
```

- First statement: Allow everything IF MFA is present
- Second statement: Deny everything IF MFA is NOT present
- Result: Users MUST have MFA enabled to do anything

### Types of Policies:

| Policy Type | What It Is | Who Creates It | Where It's Attached |
|------------|-----------|---------------|-------------------|
| **AWS Managed Policies** | Pre-built by AWS | AWS | Users, Groups, Roles |
| **Customer Managed Policies** | Created by you | You | Users, Groups, Roles |
| **Inline Policies** | Embedded directly in a user/group/role | You | Single User, Group, or Role |

**AWS Managed Policies (Most Common Ones):**

```
AdministratorAccess          → Full access to everything (GOD mode)
PowerUserAccess              → Full access EXCEPT IAM management
ReadOnlyAccess               → Read-only access to all services
AmazonEC2FullAccess          → Full EC2 access
AmazonS3FullAccess           → Full S3 access
AmazonS3ReadOnlyAccess       → Read-only S3 access
AmazonRDSFullAccess          → Full RDS access
AmazonEKSClusterPolicy       → EKS cluster management
CloudWatchFullAccess          → Full CloudWatch access
AmazonVPCFullAccess          → Full VPC access
IAMFullAccess                → Full IAM access (create users, roles, etc.)
```

**When to use which:**

```
AWS Managed → When a pre-built policy fits your needs (most common)
Customer Managed → When you need custom permissions specific to your org
Inline → AVOID unless it's a one-off, tight coupling to a single entity
```

### Creating a Custom Policy (AWS CLI):

```bash
# Create a custom policy from a JSON file
aws iam create-policy \
    --policy-name CustomDevOpsPolicy \
    --policy-document file://custom-policy.json

# custom-policy.json:
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "AllowEC2AndS3",
            "Effect": "Allow",
            "Action": [
                "ec2:Describe*",
                "ec2:StartInstances",
                "ec2:StopInstances",
                "s3:GetObject",
                "s3:PutObject",
                "s3:ListBucket"
            ],
            "Resource": "*"
        },
        {
            "Sid": "DenyIAMChanges",
            "Effect": "Deny",
            "Action": "iam:*",
            "Resource": "*"
        }
    ]
}

# This policy:
# ✅ Allows describing EC2 instances
# ✅ Allows starting/stopping EC2 instances
# ✅ Allows reading/writing S3 objects
# ❌ Denies ALL IAM actions (can't mess with permissions!)
```

### Attaching Policies:

```bash
# Attach a policy to a USER
aws iam attach-user-policy \
    --user-name ritesh \
    --policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess

# Attach a policy to a GROUP
aws iam attach-group-policy \
    --group-name Developers \
    --policy-arn arn:aws:iam::aws:policy/AmazonEC2FullAccess

# Attach a policy to a ROLE
aws iam attach-role-policy \
    --role-name EC2-S3-Access-Role \
    --policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess
```

## 🧩 Understanding ARN (Amazon Resource Name)

Every resource in AWS has a unique ARN. It's like a **full address** for any AWS resource.

```
Format:
arn:aws:service:region:account-id:resource-type/resource-id

Examples:
───────
arn:aws:s3:::my-bucket                          ← S3 bucket (global, no region)
arn:aws:s3:::my-bucket/docs/file.pdf            ← Specific object in S3
arn:aws:ec2:us-east-1:123456789012:instance/i-12345  ← Specific EC2 instance
arn:aws:iam::123456789012:user/ritesh            ← IAM user (global, no region)
arn:aws:iam::123456789012:role/MyRole             ← IAM role
arn:aws:lambda:us-east-1:123456789012:function:my-func ← Lambda function

Breaking Down:
─────────────
arn:aws:ec2:us-east-1:123456789012:instance/i-12345
│   │   │    │         │              │         │
│   │   │    │         │              │         └── Resource ID
│   │   │    │         │              └── Resource Type
│   │   │    │         └── Account ID
│   │   │    └── Region
│   │   └── Service
│   └── Partition (aws, aws-cn for China, aws-us-gov for GovCloud)
└── Always starts with "arn"
```

## 🔒 IAM Policy Evaluation Logic

When a user makes a request, AWS evaluates ALL applicable policies. Here's how:

```
User makes a request (e.g., "Delete this S3 bucket")
    │
    ▼
Step 1: Is there an EXPLICIT DENY?
    │
    ├── YES → ❌ REQUEST DENIED (Deny ALWAYS wins)
    │
    └── NO → Continue to Step 2
              │
              ▼
Step 2: Is there an EXPLICIT ALLOW?
    │
    ├── YES → ✅ REQUEST ALLOWED
    │
    └── NO → ❌ REQUEST DENIED (Implicit deny — no permission = no access)
```

**The Golden Rule:**

```
╔════════════════════════════════════════════╗
║  DENY > ALLOW > Default DENY              ║
║                                            ║
║  1. Explicit DENY always wins              ║
║  2. Explicit ALLOW grants access           ║
║  3. If neither, DEFAULT is DENY            ║
║                                            ║
║  In other words:                           ║
║  Everything is denied unless explicitly     ║
║  allowed, and explicit denies override      ║
║  any allows.                               ║
╚════════════════════════════════════════════╝
```

## 🔑 IAM Security Best Practices — The Checklist

```
╔══════════════════════════════════════════════════════════════╗
║                IAM SECURITY BEST PRACTICES                  ║
╠══════════════════════════════════════════════════════════════╣
║                                                              ║
║  1. ✅ Enable MFA everywhere (root + all users)              ║
║  2. ✅ Follow Least Privilege Principle                       ║
║        → Give MINIMUM permissions needed, nothing more       ║
║  3. ✅ Use Groups — don't attach policies to individual users║
║  4. ✅ Use Roles for AWS services (not access keys)          ║
║  5. ✅ Use Roles for cross-account access                    ║
║  6. ✅ Rotate credentials regularly (90-day max)             ║
║  7. ✅ Use strong password policy                            ║
║  8. ✅ Remove unused users and access keys                   ║
║  9. ✅ Never use root user for daily tasks                   ║
║  10. ✅ Use IAM Access Analyzer to find unused permissions   ║
║  11. ✅ Never hardcode credentials in code                   ║
║  12. ✅ Use AWS Organizations SCPs for guardrails            ║
║  13. ✅ Monitor IAM with CloudTrail (log every API call)     ║
║  14. ✅ Use IAM Credential Report for auditing               ║
║                                                              ║
╚══════════════════════════════════════════════════════════════╝
```

### Setting a Password Policy:

```bash
# Configure a strong password policy for your account
aws iam update-account-password-policy \
    --minimum-password-length 14 \
    --require-symbols \
    --require-numbers \
    --require-uppercase-characters \
    --require-lowercase-characters \
    --allow-users-to-change-password \
    --max-password-age 90 \
    --password-reuse-prevention 12
```

### Generating a Credential Report:

```bash
# Generate a credential report (shows all users, MFA status, key age, etc.)
aws iam generate-credential-report

# Download the report
aws iam get-credential-report --output text --query Content | base64 --decode > report.csv

# This CSV shows:
# - Which users have MFA enabled
# - When access keys were last used
# - When passwords were last rotated
# - Which users have console access
# → Very useful for security audits!
```

## 📝 Summary — Key Takeaways

| Concept | Remember This |
|---------|--------------|
| **Root User** | God-level access. Enable MFA. NEVER use for daily tasks. |
| **IAM User** | Represents a person or app. Has permanent credentials. |
| **IAM Group** | Collection of users. Attach policies to groups, not users. |
| **IAM Role** | Temporary identity. Used by AWS services and cross-account. |
| **IAM Policy** | JSON document defining permissions (Allow/Deny + Actions + Resources). |
| **Policy Evaluation** | Deny > Allow > Default Deny |
| **Least Privilege** | Give minimum permissions needed. No more. |
| **MFA** | Multi-Factor Auth. Enable it everywhere. Always. |
| **Access Keys** | For CLI/API access. Rotate regularly. Never commit to Git. |
| **ARN** | Unique identifier for every AWS resource. |
