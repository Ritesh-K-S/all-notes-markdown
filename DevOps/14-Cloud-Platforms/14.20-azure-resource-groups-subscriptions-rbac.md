## 1. Welcome to Azure — The Big Picture

Azure is Microsoft's cloud platform, and just like any large organization, it needs a way to **organize**, **control**, and **secure** all the stuff people create on it.

Think of it like a big company:
- The **company** (Azure Tenant) has different **departments** (Management Groups)
- Each department has **budgets** (Subscriptions)
- Each budget funds **projects** (Resource Groups)
- Each project contains **stuff** (Resources — VMs, databases, etc.)
- And there are **access rules** about who can do what (RBAC)

Let's break each of these down!

## 2. Azure's Organizational Hierarchy

```
┌──────────────────────────────────────────────────────┐
│              Azure AD Tenant (Entra ID)              │  ← Your organization's identity
│              (e.g., yourcompany.onmicrosoft.com)      │
├──────────────────────────────────────────────────────┤
│                                                      │
│  ┌───────────────────────────────────────────────┐   │
│  │           Root Management Group               │   │  ← Top-level grouping
│  │                                               │   │
│  │  ┌─────────────────┐  ┌─────────────────────┐ │   │
│  │  │ MG: Production  │  │ MG: Non-Production  │ │   │  ← Management Groups
│  │  │                 │  │                     │ │   │
│  │  │ ┌─────────────┐ │  │ ┌─────────────────┐ │ │   │
│  │  │ │ Sub: Prod   │ │  │ │ Sub: Dev        │ │ │   │  ← Subscriptions
│  │  │ │             │ │  │ │                 │ │ │   │
│  │  │ │ ┌─────────┐ │ │  │ │ ┌─────────────┐ │ │ │   │
│  │  │ │ │ RG:     │ │ │  │ │ │ RG:         │ │ │ │   │  ← Resource Groups
│  │  │ │ │ web-app │ │ │  │ │ │ dev-testing │ │ │ │   │
│  │  │ │ │         │ │ │  │ │ │             │ │ │ │   │
│  │  │ │ │ VM, DB  │ │ │  │ │ │ VM, Storage │ │ │ │   │  ← Resources
│  │  │ │ │ Storage │ │ │  │ │ │             │ │ │ │   │
│  │  │ │ └─────────┘ │ │  │ │ └─────────────┘ │ │ │   │
│  │  │ └─────────────┘ │  │ └─────────────────┘ │ │   │
│  │  └─────────────────┘  └─────────────────────┘ │   │
│  └───────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────┘
```

**Key Rule:** Permissions, policies, and settings **inherit downward**. If you set a policy on a Management Group, it applies to ALL subscriptions, resource groups, and resources below it.

## 3. Azure Account & Tenant (Azure AD / Entra ID)

### What is a Tenant?

A **tenant** is your organization's dedicated instance of **Azure Active Directory** (now called **Microsoft Entra ID**). It's the top-level identity container.

Think of it like this:
- Your **tenant** = your company's identity system
- It holds all your **users**, **groups**, **applications**, and **service principals**
- Every Azure subscription belongs to ONE tenant

### Simple Analogy

```
Tenant = Your company's building
├── Users = Employees who work in the building
├── Groups = Teams/departments (Engineering, Finance, etc.)
├── Applications = Software registered with your company
└── Service Principals = Robot employees (automated scripts/apps)
```

### Key Facts

- When you sign up for Azure, a tenant is created automatically
- Tenant has a unique ID (GUID) and a domain (e.g., `yourcompany.onmicrosoft.com`)
- You can add custom domains (e.g., `yourcompany.com`)
- One tenant can have **many** subscriptions
- But one subscription belongs to only **one** tenant

```bash
# Check your current tenant
az account show --query tenantId -o tsv
# Output: 72f988bf-86f1-41af-91ab-2d7cd011db47
```

## 4. Management Groups

### What Are They?

Management Groups are **containers** that help you manage access, policy, and compliance across **multiple subscriptions**. They sit above subscriptions in the hierarchy.

### Why Do You Need Them?

If you have 10+ subscriptions, managing policies on each one individually is painful. Management Groups let you apply things **once** at the group level.

### Simple Analogy

```
Without Management Groups:
├── Subscription: Prod-Web     → Apply security policy
├── Subscription: Prod-API     → Apply security policy (again!)
├── Subscription: Prod-Data    → Apply security policy (again!!)
├── Subscription: Dev-Web      → Apply dev policy
├── Subscription: Dev-API      → Apply dev policy (again!)
└── ...this doesn't scale ❌

With Management Groups:
├── MG: Production
│   ├── Sub: Prod-Web      ┐
│   ├── Sub: Prod-API      ├── Security policy applied ONCE to MG
│   └── Sub: Prod-Data     ┘
├── MG: Non-Production
│   ├── Sub: Dev-Web       ┐
│   └── Sub: Dev-API       ├── Dev policy applied ONCE to MG
└── ✅ Clean and scalable!
```

### Key Rules

- Up to **6 levels** of depth (not counting root level)
- 10,000 management groups per directory
- A management group can have many children, but only **one** parent
- All subscriptions and management groups are under the **Root Management Group**

```bash
# Create a management group
az account management-group create \
    --name "Production" \
    --display-name "Production Workloads"

# Move a subscription into a management group
az account management-group subscription add \
    --name "Production" \
    --subscription "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"

# List all management groups
az account management-group list -o table
```

## 5. Subscriptions — Deep Dive

### What is a Subscription?

A **subscription** is a billing boundary and an access control boundary in Azure. Every resource you create lives inside a subscription.

### Simple Analogy

Think of subscriptions like **credit cards**:
- Each credit card has its own billing
- Each card can have different spending limits
- Different people can be authorized to use different cards
- You might have one for "business expenses" and another for "personal use"

### Why Multiple Subscriptions?

| Reason | Example |
|---|---|
| **Billing separation** | Prod costs vs Dev costs |
| **Access control boundary** | Different teams, different permissions |
| **Subscription limits** | Some resources have per-subscription limits |
| **Compliance** | Isolate regulated workloads (PCI, HIPAA) |
| **Environment separation** | Production, Staging, Development |

### Common Subscription Patterns

```
Pattern 1: By Environment
├── Subscription: Production
├── Subscription: Staging
└── Subscription: Development

Pattern 2: By Team/Department
├── Subscription: Engineering
├── Subscription: Data Science
└── Subscription: Marketing

Pattern 3: By Application
├── Subscription: E-Commerce Platform
├── Subscription: Internal Tools
└── Subscription: Data Analytics

Pattern 4: Combined (most common for larger orgs)
├── MG: Production
│   ├── Sub: Prod-Platform
│   └── Sub: Prod-Data
├── MG: Non-Prod
│   ├── Sub: Dev
│   └── Sub: Staging
└── MG: Sandbox
    └── Sub: Experiments
```

### Subscription Limits (Important!)

Each subscription has limits (also called **quotas**):

| Resource | Default Limit |
|---|---|
| VMs per region | 25,000 |
| Storage accounts per region | 250 |
| Resource Groups per subscription | 980 |
| Virtual Networks per region | 1,000 |
| Public IP addresses per region | 1,000 |
| Network Security Groups per region | 5,000 |

> You can request limit increases through Azure Support!

```bash
# List all subscriptions
az account list -o table

# Set a specific subscription as active
az account set --subscription "My-Dev-Subscription"

# Show current subscription details
az account show -o table
```

## 6. Resource Groups — Deep Dive

### What is a Resource Group?

A **Resource Group (RG)** is a logical container for Azure resources. Every single resource in Azure MUST belong to exactly one resource group.

### Simple Analogy

Think of Resource Groups like **folders on your computer**:
- You organize files into folders
- Each file belongs to ONE folder
- You can delete a folder and all files inside it are deleted
- You can set permissions on a folder, and they apply to everything inside

### Key Rules

| Rule | Explanation |
|---|---|
| Every resource must be in exactly **one** RG | No resource can exist outside an RG |
| An RG can contain resources from **different regions** | The RG location is just where its metadata is stored |
| Resources can communicate **across** RGs | RGs are organizational, not network boundaries |
| Deleting an RG **deletes everything** inside it | Be careful! |
| RG names must be **unique within a subscription** | But can be the same across subscriptions |
| RGs **cannot** be nested | No RG inside another RG |

### How to Organize Resource Groups

```
Option 1: By Application / Workload (Most Common ✅)
├── rg-ecommerce-prod        ← All e-commerce prod resources
│   ├── VM: web-server
│   ├── SQL Database: products-db
│   ├── Storage: product-images
│   └── App Service: checkout-api
├── rg-ecommerce-dev          ← All e-commerce dev resources
└── rg-monitoring             ← Shared monitoring resources

Option 2: By Resource Type
├── rg-virtual-machines
├── rg-databases
├── rg-storage
└── rg-networking
# ^^^ Not recommended — hard to manage lifecycle

Option 3: By Lifecycle (deploy/delete together)
├── rg-project-alpha          ← Delete when project is done
├── rg-sprint-17-testing      ← Delete after sprint
└── rg-shared-infrastructure  ← Keep forever
```

### Best Practice: Group by Lifecycle

Resources that are **created together, managed together, and deleted together** should be in the **same resource group**.

```bash
# Create a resource group
az group create \
    --name "rg-webapp-prod" \
    --location "eastus" \
    --tags Environment=Production Team=WebApp

# List all resource groups
az group list -o table

# Show details of a specific RG
az group show --name "rg-webapp-prod"

# List all resources in a resource group
az resource list --resource-group "rg-webapp-prod" -o table

# Delete a resource group (deletes ALL resources inside!)
# ⚠️ Be VERY careful with this!
az group delete --name "rg-temp-testing" --yes --no-wait
```

### Resource Group vs AWS

| Azure | AWS Equivalent |
|---|---|
| Resource Group | No direct equivalent (closest: Tags + CloudFormation Stack) |
| Subscription | AWS Account |
| Management Group | AWS Organizations OU |
| Tenant (Entra ID) | AWS Organizations Root |

## 7. Resources

A **resource** is any individual service instance you create in Azure:

```
Examples of Resources:
├── Virtual Machine (VM)
├── Storage Account
├── SQL Database
├── Virtual Network
├── App Service
├── Key Vault
├── Load Balancer
├── Container Registry
├── Kubernetes Cluster (AKS)
├── Function App
└── ... basically anything you create in Azure
```

### Key Facts About Resources

- Every resource has a unique **Resource ID**:
  ```
  /subscriptions/{sub-id}/resourceGroups/{rg-name}/providers/{provider}/{type}/{name}
  
  Example:
  /subscriptions/12345678-abcd-efgh-ijkl-123456789012/resourceGroups/rg-webapp-prod/providers/Microsoft.Compute/virtualMachines/web-server-01
  ```
- Resources can be in a **different region** than their resource group
- Resources can **move** between resource groups (with some exceptions)

```bash
# Create a resource (example: storage account)
az storage account create \
    --name "mystorageaccount2024" \
    --resource-group "rg-webapp-prod" \
    --location "eastus" \
    --sku "Standard_LRS"

# Move a resource to a different resource group
az resource move \
    --destination-group "rg-webapp-staging" \
    --ids "/subscriptions/.../resourceGroups/rg-webapp-prod/providers/Microsoft.Storage/storageAccounts/mystorageaccount2024"
```

## 8. Tagging in Azure

Tags are **name-value pairs** you can apply to resources, resource groups, and subscriptions for organization, cost tracking, and automation.

### Why Tags Matter

```
Without Tags:
You have 500 resources. Quick, which ones belong to the 
marketing team? Which are for production? Which can be 
deleted? 🤷 No idea!

With Tags:
Every resource has:
  Team = marketing
  Environment = production
  CostCenter = CC-1234
  Project = campaign-dashboard
  
Now you can filter, report, and automate! ✅
```

### Common Tag Strategy

| Tag Name | Purpose | Example Values |
|---|---|---|
| `Environment` | Which environment | production, staging, dev |
| `Team` / `Owner` | Who owns it | platform, backend, data |
| `Project` | Which project | checkout, search, analytics |
| `CostCenter` | For billing allocation | CC-1001, CC-2002 |
| `ManagedBy` | How was it created | terraform, manual, bicep |
| `CreatedDate` | When was it created | 2024-01-15 |
| `AutoShutdown` | Should it auto-stop? | true, false |

```bash
# Add tags to a resource group
az group update \
    --name "rg-webapp-prod" \
    --tags Environment=Production Team=WebApp CostCenter=CC-1001

# Add tags to a specific resource
az resource tag \
    --tags Environment=Production Team=WebApp \
    --ids "/subscriptions/.../resourceGroups/rg-webapp-prod/providers/Microsoft.Compute/virtualMachines/web-server-01"

# Find all resources with a specific tag
az resource list --tag Environment=Production -o table
```

### Important: Tags Are NOT Inherited

Unlike RBAC and policies, tags do **NOT** automatically flow from resource groups to resources. If you tag a resource group with `Environment=Production`, the resources inside it do NOT automatically get that tag.

> **Solution:** Use **Azure Policy** to enforce tagging!

```json
// Azure Policy: Require 'Environment' tag on all resources
{
    "if": {
        "field": "[concat('tags[', 'Environment', ']')]",
        "exists": "false"
    },
    "then": {
        "effect": "deny"
    }
}
// This policy BLOCKS the creation of any resource that 
// doesn't have an 'Environment' tag. Simple and effective!
```

## 9. Azure RBAC — Role-Based Access Control

### What is RBAC?

RBAC stands for **Role-Based Access Control**. It's how you control **who** can do **what** on **which resources** in Azure.

### The Three Questions of RBAC

Every RBAC assignment answers three questions:

```
WHO?     →  Security Principal (User, Group, Service Principal, Managed Identity)
WHAT?    →  Role Definition (Reader, Contributor, Owner, Custom Role)
WHERE?   →  Scope (Management Group, Subscription, Resource Group, Resource)

Example:
WHO:   "Alice" (user)
WHAT:  "Contributor" (can create/modify/delete resources)
WHERE: "rg-webapp-prod" (only this resource group)

= Alice can create, modify, and delete resources 
  in the rg-webapp-prod resource group only.
```

### Visual Representation

```
┌───────────────────────────────────────────────────────────────┐
│                    ROLE ASSIGNMENT                             │
│                                                               │
│  ┌──────────────┐   ┌──────────────┐   ┌──────────────────┐  │
│  │    WHO?       │ + │    WHAT?      │ + │     WHERE?       │  │
│  │              │   │              │   │                  │  │
│  │ Security     │   │ Role         │   │ Scope            │  │
│  │ Principal    │   │ Definition   │   │                  │  │
│  │              │   │              │   │ • Mgmt Group     │  │
│  │ • User       │   │ • Owner      │   │ • Subscription   │  │
│  │ • Group      │   │ • Contributor│   │ • Resource Group  │  │
│  │ • Service    │   │ • Reader     │   │ • Resource       │  │
│  │   Principal  │   │ • Custom     │   │                  │  │
│  │ • Managed    │   │              │   │                  │  │
│  │   Identity   │   │              │   │                  │  │
│  └──────────────┘   └──────────────┘   └──────────────────┘  │
│                                                               │
│                    = ROLE ASSIGNMENT                           │
└───────────────────────────────────────────────────────────────┘
```

### Security Principals Explained

| Type | What It Is | Use When |
|---|---|---|
| **User** | An individual person in Azure AD/Entra ID | A specific person needs access |
| **Group** | A collection of users | A team needs the same access |
| **Service Principal** | An identity for an application | An app/script needs access |
| **Managed Identity** | An auto-managed service principal | Azure resources accessing other Azure resources |

```
Think of it like a building:
User           = "Employee badge" — for individual people
Group          = "Department keycard" — for entire teams
Service Princ. = "Robot badge" — for automated systems
Managed Id.    = "Auto-generated robot badge" — Azure manages the password for you
```

### Scope and Inheritance

RBAC permissions **inherit downward**:

```
Management Group: "Production"        ← Alice is Owner HERE
    │
    ├── Subscription: "Prod-Web"      ← Alice is ALSO Owner here (inherited)
    │   │
    │   ├── RG: "rg-web-frontend"     ← Alice is ALSO Owner here (inherited)
    │   │   ├── VM: web-01            ← Alice is ALSO Owner here (inherited)
    │   │   └── Storage: images       ← Alice is ALSO Owner here (inherited)
    │   │
    │   └── RG: "rg-web-backend"      ← Alice is ALSO Owner here (inherited)
    │
    └── Subscription: "Prod-Data"     ← Alice is ALSO Owner here (inherited)

RULE: Assign at the HIGHEST level needed, but the LOWEST level possible.
```

## 10. Built-in Roles

Azure has **100+** built-in roles. Here are the most important ones:

### The Big Three

| Role | What They Can Do | Use When |
|---|---|---|
| **Owner** | Full access + can manage RBAC (assign roles to others) | Subscription administrators |
| **Contributor** | Can create/modify/delete resources, but CANNOT manage RBAC | Developers, DevOps engineers |
| **Reader** | Can view everything, but CANNOT change anything | Auditors, managers, read-only access |

### Visual Comparison

```
┌─────────────────────────────────────────────┐
│              OWNER                          │
│  ✅ Read     ✅ Write    ✅ Delete           │
│  ✅ Manage RBAC (assign roles to others)    │
├─────────────────────────────────────────────┤
│              CONTRIBUTOR                     │
│  ✅ Read     ✅ Write    ✅ Delete           │
│  ❌ Manage RBAC                             │
├─────────────────────────────────────────────┤
│              READER                          │
│  ✅ Read     ❌ Write    ❌ Delete           │
│  ❌ Manage RBAC                             │
├─────────────────────────────────────────────┤
│              USER ACCESS ADMINISTRATOR       │
│  ❌ Read*    ❌ Write*   ❌ Delete*          │
│  ✅ Manage RBAC (assign roles to others)    │
│  *Can only manage access, not resources     │
└─────────────────────────────────────────────┘
```

### Other Common Built-in Roles

| Role | Scope | Purpose |
|---|---|---|
| **Virtual Machine Contributor** | Compute | Manage VMs, but not the network/storage they connect to |
| **Network Contributor** | Network | Manage networks, subnets, NSGs, load balancers |
| **Storage Blob Data Reader** | Storage | Read blobs in storage accounts |
| **Storage Blob Data Contributor** | Storage | Read, write, delete blobs |
| **Key Vault Secrets Officer** | Key Vault | Manage secrets in Key Vault |
| **AKS Cluster Admin** | Kubernetes | Full access to AKS clusters |
| **SQL DB Contributor** | Database | Manage SQL databases, not access/security |
| **Monitoring Reader** | Monitoring | Read monitoring data (metrics, logs) |
| **Cost Management Reader** | Billing | View cost and billing data |

```bash
# List all built-in roles
az role definition list --output table

# Show details of a specific role
az role definition list --name "Contributor" --output json
```

## 11. Custom Roles

When built-in roles don't fit your needs, you can create **custom roles**.

### When to Use Custom Roles

```
Scenario: Your developers need to:
✅ Start/Stop VMs
✅ View VM details  
❌ Create new VMs
❌ Delete VMs
❌ Modify VM size

The built-in "Virtual Machine Contributor" allows ALL VM operations.
That's too much! → Create a custom role with just Start/Stop/View.
```

### Creating a Custom Role

```json
// custom-role-vm-operator.json
{
    "Name": "VM Operator",
    "Description": "Can start, stop, and restart VMs. Cannot create or delete.",
    "Actions": [
        "Microsoft.Compute/virtualMachines/start/action",
        "Microsoft.Compute/virtualMachines/powerOff/action",
        "Microsoft.Compute/virtualMachines/restart/action",
        "Microsoft.Compute/virtualMachines/read",
        "Microsoft.Compute/virtualMachines/instanceView/read",
        "Microsoft.Resources/subscriptions/resourceGroups/read"
    ],
    "NotActions": [],
    "DataActions": [],
    "NotDataActions": [],
    "AssignableScopes": [
        "/subscriptions/xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
    ]
}
```

**What's happening here?**
- `Actions`: Lists the exact operations this role CAN do
- `NotActions`: Operations explicitly denied (we don't need any here)
- `DataActions`: Data-plane operations (like reading blob data)
- `AssignableScopes`: Where this role can be assigned (this subscription only)

```bash
# Create the custom role
az role definition create --role-definition @custom-role-vm-operator.json

# Assign the custom role to a user
az role assignment create \
    --assignee "developer@yourcompany.com" \
    --role "VM Operator" \
    --scope "/subscriptions/xxx/resourceGroups/rg-webapp-prod"
```

### Understanding Actions vs DataActions

```
Actions (Control Plane):
├── Managing resources themselves
├── Example: Create VM, Delete Storage Account, Modify Network
└── "I can manage the CONTAINER"

DataActions (Data Plane):
├── Working with data INSIDE resources
├── Example: Read blob data, Write to queue, Read secrets
└── "I can access the CONTENT"

Example:
Storage Account Contributor (Actions):
→ Can CREATE/DELETE storage accounts
→ CANNOT read the actual files inside

Storage Blob Data Reader (DataActions):
→ CANNOT create/delete storage accounts  
→ CAN read the actual files inside
```

## 12. Role Assignments — Putting It All Together

### Creating Role Assignments

```bash
# Assign "Reader" role to a user at the subscription level
az role assignment create \
    --assignee "alice@yourcompany.com" \
    --role "Reader" \
    --scope "/subscriptions/xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"

# Assign "Contributor" to a group at the resource group level
az role assignment create \
    --assignee-object-id "group-object-id-here" \
    --assignee-principal-type "Group" \
    --role "Contributor" \
    --scope "/subscriptions/xxx/resourceGroups/rg-webapp-prod"

# Assign role to a service principal (for CI/CD pipelines)
az role assignment create \
    --assignee "service-principal-app-id" \
    --role "Contributor" \
    --scope "/subscriptions/xxx/resourceGroups/rg-webapp-prod"

# List all role assignments for a resource group
az role assignment list \
    --resource-group "rg-webapp-prod" \
    --output table

# Remove a role assignment
az role assignment delete \
    --assignee "alice@yourcompany.com" \
    --role "Reader" \
    --scope "/subscriptions/xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
```

### Real-World Example: Setting Up a Team

```bash
# SCENARIO: Set up access for a web development team

# Step 1: Create a security group for the team
az ad group create \
    --display-name "WebDev-Team" \
    --mail-nickname "webdev-team"

# Step 2: Add team members to the group
az ad group member add \
    --group "WebDev-Team" \
    --member-id "alice-object-id"
az ad group member add \
    --group "WebDev-Team" \
    --member-id "bob-object-id"

# Step 3: Assign Contributor on their resource groups
az role assignment create \
    --assignee-object-id "$(az ad group show --group 'WebDev-Team' --query id -o tsv)" \
    --assignee-principal-type "Group" \
    --role "Contributor" \
    --scope "/subscriptions/xxx/resourceGroups/rg-webapp-dev"

# Step 4: Assign Reader on production (can view but not change)
az role assignment create \
    --assignee-object-id "$(az ad group show --group 'WebDev-Team' --query id -o tsv)" \
    --assignee-principal-type "Group" \
    --role "Reader" \
    --scope "/subscriptions/xxx/resourceGroups/rg-webapp-prod"

# Now:
# ✅ All WebDev team members can create/modify resources in DEV
# ✅ All WebDev team members can VIEW resources in PROD
# ❌ No one on the team can CHANGE production resources
# ✅ When a new person joins, just add them to the group!
```

### Best Practice: Use Groups, Not Individual Users

```
❌ BAD: Assign roles to individual users
   "Alice" → Contributor → rg-webapp
   "Bob"   → Contributor → rg-webapp
   "Carol" → Contributor → rg-webapp
   # When Dave joins, you need to create ANOTHER assignment
   # When Alice leaves, you need to REMEMBER to remove her
   # With 50 people and 20 resource groups = 1000 assignments! 😱

✅ GOOD: Assign roles to groups
   "WebDev-Team" group → Contributor → rg-webapp
   # When Dave joins → add him to the group. Done!
   # When Alice leaves → remove from group. Done!
   # 50 people, 20 resource groups = still just 20 assignments! 👍
```

## 13. Azure Policy — Enforcing Rules

Azure Policy is different from RBAC. While RBAC controls **who can do what**, Policy controls **what can be done** regardless of who's doing it.

### RBAC vs Policy

```
RBAC:  "Alice CAN create VMs" (focus on the PERSON)
Policy: "VMs MUST be in East US region" (focus on the RULE)

Together:
Alice CAN create VMs, BUT only in East US region!
```

### Common Policy Examples

```json
// Policy 1: Only allow certain VM sizes (save money!)
{
    "if": {
        "allOf": [
            {
                "field": "type",
                "equals": "Microsoft.Compute/virtualMachines"
            },
            {
                "not": {
                    "field": "Microsoft.Compute/virtualMachines/sku.name",
                    "in": [
                        "Standard_B1s",
                        "Standard_B2s",
                        "Standard_D2s_v3",
                        "Standard_D4s_v3"
                    ]
                }
            }
        ]
    },
    "then": {
        "effect": "deny"
    }
}
// This policy says: "You can ONLY create VMs that are 
// B1s, B2s, D2s_v3, or D4s_v3. Anything else is BLOCKED."
// This prevents someone from spinning up a $10K/month VM!
```

### Policy Effects

| Effect | What Happens |
|---|---|
| **Deny** | Blocks the action entirely |
| **Audit** | Allows the action but flags it as non-compliant |
| **Append** | Adds extra properties (e.g., auto-add tags) |
| **Modify** | Changes properties on existing resources |
| **DeployIfNotExists** | Deploys a resource if a condition is met |
| **AuditIfNotExists** | Flags if a related resource doesn't exist |
| **Disabled** | Policy exists but isn't active |

```bash
# Assign a built-in policy: "Require a tag on resources"
az policy assignment create \
    --name "require-environment-tag" \
    --display-name "Require Environment Tag" \
    --policy "/providers/Microsoft.Authorization/policyDefinitions/871b6d14-10aa-478d-b590-94f262ecfa99" \
    --scope "/subscriptions/xxx/resourceGroups/rg-webapp-prod" \
    --params '{"tagName": {"value": "Environment"}}'
```

## 14. Locks — Preventing Accidental Deletion

Resource locks prevent accidental modification or deletion of critical resources.

### Two Types of Locks

```
┌────────────────────────────────────────────────┐
│              LOCK TYPES                        │
├────────────────────────────────────────────────┤
│                                                │
│  🔒 ReadOnly Lock                              │
│     • Can: READ resources                      │
│     • Cannot: MODIFY or DELETE resources        │
│     • Like: "Look but don't touch"              │
│                                                │
│  🔒 CanNotDelete Lock                          │
│     • Can: READ and MODIFY resources            │
│     • Cannot: DELETE resources                  │
│     • Like: "Change it, but don't destroy it"   │
│                                                │
└────────────────────────────────────────────────┘
```

### Why Locks Are Critical

```
HORROR STORY (without locks):
Junior DevOps engineer: "Let me clean up some old resource groups..."
*deletes rg-production-database*
"Wait... that was PRODUCTION! 😱"

WITH LOCKS:
Junior DevOps engineer: "Let me clean up some old resource groups..."
Azure: "❌ Cannot delete rg-production-database — it has a CanNotDelete lock"
"Phew! 😮‍💨"
```

```bash
# Create a CanNotDelete lock on a resource group
az lock create \
    --name "protect-prod" \
    --resource-group "rg-webapp-prod" \
    --lock-type CanNotDelete \
    --notes "Production environment — do not delete!"

# Create a ReadOnly lock
az lock create \
    --name "freeze-network" \
    --resource-group "rg-networking" \
    --lock-type ReadOnly \
    --notes "Network config frozen — change management required"

# List locks
az lock list --resource-group "rg-webapp-prod" -o table

# Delete a lock (only when you REALLY need to)
az lock delete --name "protect-prod" --resource-group "rg-webapp-prod"
```

> **Key Point:** Locks are inherited. A lock on a resource group applies to ALL resources inside it.

## 17. Best Practices

### Resource Groups

| Do | Don't |
|---|---|
| Group resources by lifecycle | Group by resource type |
| Use consistent naming: `rg-{app}-{env}` | Use random names: `myRG1` |
| Apply tags to every RG | Leave RGs untagged |
| Use locks on production RGs | Leave production unprotected |
| Clean up unused RGs regularly | Let empty RGs pile up |

### RBAC

| Do | Don't |
|---|---|
| Assign roles to **groups**, not users | Assign roles to individual users |
| Use **least privilege** — give minimum needed | Give Owner to everyone "just in case" |
| Use **Managed Identities** for Azure resources | Use service principal secrets in code |
| Audit role assignments regularly | Set and forget |
| Use custom roles when built-in are too broad | Create custom roles for everything |
| Assign at the lowest scope needed | Assign at subscription level when RG is enough |

### Subscriptions

| Do | Don't |
|---|---|
| Separate prod from non-prod | Put everything in one subscription |
| Use management groups for governance | Apply policies subscription by subscription |
| Set up budget alerts | Let costs run unchecked |
| Apply naming conventions | Use default names |

## 18. Common Mistakes to Avoid

```
❌ MISTAKE 1: Giving Owner role at subscription level to developers
   → They can assign roles, delete anything, change billing
   ✅ FIX: Give Contributor at resource group level

❌ MISTAKE 2: Not using resource locks on production
   → One accidental "az group delete" and production is gone
   ✅ FIX: Apply CanNotDelete locks on all production RGs

❌ MISTAKE 3: Organizing resource groups by type instead of lifecycle
   → Deleting a project means hunting across multiple RGs
   ✅ FIX: Group by application/project lifecycle

❌ MISTAKE 4: Not using tags
   → Can't track costs, can't find resources, can't automate
   ✅ FIX: Enforce tagging with Azure Policy

❌ MISTAKE 5: Using the same subscription for everything
   → Billing is a mess, can't apply different policies
   ✅ FIX: Separate subscriptions for prod, non-prod, sandbox

❌ MISTAKE 6: Hardcoding service principal secrets
   → Secrets expire, get leaked, hard to rotate
   ✅ FIX: Use Managed Identities — Azure handles the credentials

❌ MISTAKE 7: Not reviewing role assignments
   → Former employees still have access, role creep
   ✅ FIX: Quarterly access reviews using Azure AD Access Reviews
```

## 19. Interview Questions

### Basic Level

**Q: What is a Resource Group in Azure?**
> A: A Resource Group is a logical container that holds related Azure resources. Every Azure resource must belong to exactly one resource group. It's used for organization, access control, lifecycle management, and cost tracking.

**Q: What is the difference between Owner, Contributor, and Reader roles?**
> A: 
> - **Owner**: Full access to all resources + can manage RBAC (assign roles)
> - **Contributor**: Can create, modify, and delete resources, but cannot manage RBAC
> - **Reader**: Can only view resources, cannot make any changes

**Q: Can a resource be in multiple resource groups?**
> A: No. A resource can only belong to exactly one resource group at a time. However, you can move resources between resource groups.

### Intermediate Level

**Q: What's the difference between RBAC and Azure Policy?**
> A: RBAC controls **who** can do actions (identity-based access control). Azure Policy controls **what** actions are allowed regardless of who does them (compliance/governance). Example: RBAC says "Alice can create VMs," Policy says "VMs must be in East US region."

**Q: Explain the Azure management hierarchy.**
> A: From top to bottom: Tenant (Azure AD/Entra ID) → Management Groups → Subscriptions → Resource Groups → Resources. Permissions and policies inherit downward.

**Q: What happens when you delete a resource group?**
> A: All resources inside the resource group are deleted. This is irreversible. That's why resource locks are critical for production environments.

### Advanced Level

**Q: A developer says they can't create a VM. How do you troubleshoot?**
> A: I'd check in this order:
> 1. **RBAC**: Does the user have a role with `Microsoft.Compute/virtualMachines/write` permission at the right scope?
> 2. **Azure Policy**: Is there a policy denying VM creation (e.g., restricting VM sizes, regions)?
> 3. **Subscription quotas**: Has the subscription hit its VM limit for that region?
> 4. **Resource locks**: Is there a ReadOnly lock on the resource group?
> 5. **Conditional Access**: Are there Entra ID Conditional Access policies blocking the operation?

**Q: How would you set up Azure for a company with 100 developers, 3 environments, and 5 product teams?**
> A: 
> - **Tenant**: One Entra ID tenant
> - **Management Groups**: Production, Non-Production, Sandbox
> - **Subscriptions**: One per environment per team (or shared subscriptions with RBAC separation)
> - **Resource Groups**: By application lifecycle within each subscription
> - **RBAC**: Security groups per team, Contributor on their resources, Reader on prod
> - **Policy**: Enforce tagging, allowed regions, allowed VM sizes
> - **Locks**: CanNotDelete on all production resource groups
> - **Budgets**: Per subscription with alerts at 80%

```
Azure's organizational hierarchy:

Tenant (Identity)
  └── Management Groups (Governance at scale)
       └── Subscriptions (Billing + access boundary)
            └── Resource Groups (Logical container)
                 └── Resources (The actual stuff — VMs, DBs, etc.)

Key governance tools:
├── RBAC: WHO can do WHAT on WHICH resources
├── Policy: WHAT is allowed (regardless of who)
├── Tags: Metadata for organization and cost tracking
```
