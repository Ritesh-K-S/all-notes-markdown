## 1. What is an Azure Virtual Machine?

An Azure Virtual Machine (VM) is basically a **computer in the cloud**. Instead of buying a physical server, plugging it in, installing an OS, and maintaining it — you click a few buttons (or run a command) and Azure gives you a ready-to-use computer in seconds.

### Physical Server vs Azure VM

```
Physical Server:                    Azure VM:
├── Buy hardware ($5,000+)          ├── Create in 2 minutes
├── Wait for delivery (weeks)       ├── Pay by the second
├── Rack it in data center          ├── Delete when done
├── Install OS                      ├── Resize anytime
├── Configure networking            ├── Backup with a click
├── Maintain hardware               ├── No hardware maintenance
├── Replace when it breaks          ├── Azure handles failures
└── Dispose after 5 years           └── Available globally
```

### When to Use VMs

| Use VMs When | Don't Use VMs When |
|---|---|
| You need full OS control | A managed service exists (PaaS) |
| Running legacy applications | Building new cloud-native apps |
| Custom software that can't run on PaaS | Simple web apps (use App Service) |
| Lift-and-shift from on-premises | Stateless workloads (use Functions) |
| Specific OS/kernel requirements | Container workloads (use AKS) |
| Development and testing | Background jobs (use Azure Functions) |

## 2. VM Components — What Makes Up a VM?

When you create a VM, Azure actually creates several resources:

```
┌────────────────────────────────────────────────────┐
│                 Azure VM                           │
│                                                    │
│  ┌──────────────┐   ┌──────────────────────────┐  │
│  │   VM itself   │   │  OS Disk (Managed Disk)  │  │
│  │  (compute)    │   │  (30GB - 4TB)            │  │
│  └──────┬───────┘   └──────────────────────────┘  │
│         │                                          │
│  ┌──────┴──────────────────────────────────────┐  │
│  │         Network Interface (NIC)              │  │
│  └──────┬──────────────────────────────────────┘  │
│         │                                          │
│  ┌──────┴───────┐   ┌──────────────────────────┐  │
│  │  Public IP    │   │   Network Security Group │  │
│  │  (optional)   │   │   (NSG — firewall rules) │  │
│  └──────────────┘   └──────────────────────────┘  │
│                                                    │
│  ┌──────────────────────────────────────────────┐  │
│  │         Virtual Network (VNet) + Subnet      │  │
│  └──────────────────────────────────────────────┘  │
│                                                    │
│  Optional:                                         │
│  ┌──────────────┐   ┌──────────────────────────┐  │
│  │  Data Disks   │   │  Availability Set/Zone   │  │
│  │  (extra       │   │  (high availability)     │  │
│  │   storage)    │   │                          │  │
│  └──────────────┘   └──────────────────────────┘  │
└────────────────────────────────────────────────────┘
```

**Each of these is a separate Azure resource** — they're created together but can be managed individually. When you delete a VM, by default the NIC, public IP, and disks may NOT be automatically deleted (you need to clean them up).

## 3. Creating Your First VM

### Using Azure CLI

```bash
# Step 1: Create a resource group (if you don't have one)
az group create \
    --name "rg-demo-vms" \
    --location "eastus"

# Step 2: Create the VM
az vm create \
    --resource-group "rg-demo-vms" \
    --name "my-first-vm" \
    --image "Ubuntu2204" \
    --size "Standard_B2s" \
    --admin-username "azureuser" \
    --generate-ssh-keys \
    --public-ip-sku Standard \
    --tags Environment=Dev Team=Learning

# What this does:
# --name "my-first-vm"        → Names the VM
# --image "Ubuntu2204"        → Uses Ubuntu 22.04 LTS
# --size "Standard_B2s"       → 2 vCPUs, 4 GB RAM (~$30/month)
# --admin-username "azureuser"→ Sets the login username
# --generate-ssh-keys         → Creates SSH key pair for you
# --public-ip-sku Standard    → Assigns a public IP
# --tags ...                  → Adds tags for organization

# OUTPUT will show something like:
# {
#   "publicIpAddress": "20.123.45.67",
#   "resourceGroup": "rg-demo-vms",
#   "powerState": "VM running",
#   ...
# }
```

### Using Azure PowerShell

```powershell
# Create a resource group
New-AzResourceGroup -Name "rg-demo-vms" -Location "eastus"

# Create credentials
$cred = Get-Credential  # Will prompt for username/password

# Create the VM
New-AzVM `
    -ResourceGroupName "rg-demo-vms" `
    -Name "my-first-vm" `
    -Location "eastus" `
    -Image "Ubuntu2204" `
    -Size "Standard_B2s" `
    -Credential $cred `
    -OpenPorts 22
```

### Using an ARM Template (Infrastructure as Code)

```json
{
    "$schema": "https://schema.management.azure.com/schemas/2019-04-01/deploymentTemplate.json#",
    "contentVersion": "1.0.0.0",
    "parameters": {
        "vmName": {
            "type": "string",
            "defaultValue": "my-vm",
            "metadata": {
                "description": "Name of the virtual machine"
            }
        },
        "adminUsername": {
            "type": "string",
            "metadata": {
                "description": "Admin username"
            }
        },
        "adminPassword": {
            "type": "securestring",
            "metadata": {
                "description": "Admin password"
            }
        }
    },
    "variables": {
        "vmSize": "Standard_B2s",
        "vnetName": "[concat(parameters('vmName'), '-vnet')]",
        "subnetName": "default",
        "nicName": "[concat(parameters('vmName'), '-nic')]",
        "publicIPName": "[concat(parameters('vmName'), '-pip')]",
        "nsgName": "[concat(parameters('vmName'), '-nsg')]"
    },
    "resources": [
        {
            "type": "Microsoft.Network/networkSecurityGroups",
            "apiVersion": "2023-04-01",
            "name": "[variables('nsgName')]",
            "location": "[resourceGroup().location]",
            "properties": {
                "securityRules": [
                    {
                        "name": "AllowSSH",
                        "properties": {
                            "priority": 1000,
                            "protocol": "Tcp",
                            "access": "Allow",
                            "direction": "Inbound",
                            "destinationPortRange": "22",
                            "sourceAddressPrefix": "*",
                            "destinationAddressPrefix": "*",
                            "sourcePortRange": "*"
                        }
                    }
                ]
            }
        },
        {
            "type": "Microsoft.Compute/virtualMachines",
            "apiVersion": "2023-07-01",
            "name": "[parameters('vmName')]",
            "location": "[resourceGroup().location]",
            "dependsOn": [
                "[resourceId('Microsoft.Network/networkInterfaces', variables('nicName'))]"
            ],
            "properties": {
                "hardwareProfile": {
                    "vmSize": "[variables('vmSize')]"
                },
                "storageProfile": {
                    "imageReference": {
                        "publisher": "Canonical",
                        "offer": "0001-com-ubuntu-server-jammy",
                        "sku": "22_04-lts-gen2",
                        "version": "latest"
                    },
                    "osDisk": {
                        "createOption": "FromImage",
                        "managedDisk": {
                            "storageAccountType": "Standard_LRS"
                        }
                    }
                },
                "osProfile": {
                    "computerName": "[parameters('vmName')]",
                    "adminUsername": "[parameters('adminUsername')]",
                    "adminPassword": "[parameters('adminPassword')]"
                },
                "networkProfile": {
                    "networkInterfaces": [
                        {
                            "id": "[resourceId('Microsoft.Network/networkInterfaces', variables('nicName'))]"
                        }
                    ]
                }
            }
        }
    ]
}
```

**What's happening here?**
- This ARM template defines EVERYTHING needed: NSG (firewall), NIC (network card), Public IP, VNet, and the VM itself
- You deploy this once, and Azure creates all the resources in the right order (the `dependsOn` tells Azure what to create first)
- This is Infrastructure as Code — repeatable, version-controlled, and consistent

## 4. VM Sizes & Series — Picking the Right One

Azure VMs come in different "sizes" (think of them like T-shirt sizes, but for computers). Each size has different amounts of CPU, RAM, storage, and network capacity.

### VM Series Overview

```
┌──────────────────────────────────────────────────────────────────┐
│                    AZURE VM SERIES                               │
├──────────┬───────────────────┬───────────────────────────────────┤
│ Series   │ Optimized For     │ Use Cases                        │
├──────────┼───────────────────┼───────────────────────────────────┤
│ B-series │ Burstable         │ Dev/test, small websites,        │
│          │ (variable CPU)    │ micro-services                   │
├──────────┼───────────────────┼───────────────────────────────────┤
│ D-series │ General Purpose   │ Most workloads, web servers,     │
│          │ (balanced)        │ app servers, databases           │
├──────────┼───────────────────┼───────────────────────────────────┤
│ E-series │ Memory Optimized  │ Large databases, in-memory       │
│          │ (high RAM)        │ caching, SAP HANA                │
├──────────┼───────────────────┼───────────────────────────────────┤
│ F-series │ Compute Optimized │ Batch processing, gaming,        │
│          │ (high CPU)        │ analytics, CI/CD agents          │
├──────────┼───────────────────┼───────────────────────────────────┤
│ L-series │ Storage Optimized │ Big data, large databases,       │
│          │ (high I/O)        │ data warehousing                 │
├──────────┼───────────────────┼───────────────────────────────────┤
│ N-series │ GPU               │ Machine learning, video          │
│          │ (GPU-equipped)    │ rendering, graphics              │
├──────────┼───────────────────┼───────────────────────────────────┤
│ M-series │ Memory-Intensive  │ SAP HANA, very large databases   │
│          │ (MASSIVE RAM)     │ (up to 4 TB RAM!)                │
└──────────┴───────────────────┴───────────────────────────────────┘
```

### Understanding VM Size Names

```
Standard_D4s_v5
    │     │ │ │
    │     │ │ └── v5 = Generation 5 (newer = better)
    │     │ └──── s = Supports Premium Storage (SSD)
    │     └────── 4 = Number of vCPUs
    └──────────── D = Series (General Purpose)

More examples:
Standard_B2s     → Burstable, 2 vCPUs, Premium Storage
Standard_E8s_v5  → Memory-optimized, 8 vCPUs, Gen 5
Standard_F16s_v2 → Compute-optimized, 16 vCPUs, Gen 2
Standard_NC6     → GPU (N-series, Compute), 6 vCPUs
```

### Common VM Sizes with Pricing (Approximate, East US)

| Size | vCPUs | RAM | Temp Storage | ~Monthly Cost |
|---|---|---|---|---|
| Standard_B1s | 1 | 1 GB | 4 GB | ~$7 |
| Standard_B2s | 2 | 4 GB | 8 GB | ~$30 |
| Standard_D2s_v5 | 2 | 8 GB | None | ~$70 |
| Standard_D4s_v5 | 4 | 16 GB | None | ~$140 |
| Standard_D8s_v5 | 8 | 32 GB | None | ~$280 |
| Standard_E4s_v5 | 4 | 32 GB | None | ~$182 |
| Standard_F4s_v2 | 4 | 8 GB | 32 GB | ~$122 |

### How to Pick the Right Size

```
Decision Tree:

Start: What's your workload?
│
├── Dev/Test or low-traffic website?
│   └── B-series (B1s, B2s) — cheapest, burstable CPU
│
├── General web app or API server?
│   └── D-series (D2s_v5, D4s_v5) — balanced CPU & RAM
│
├── Database or in-memory caching?
│   └── E-series (E4s_v5, E8s_v5) — lots of RAM
│
├── Data processing, batch jobs, CI/CD?
│   └── F-series (F4s_v2, F8s_v2) — lots of CPU
│
├── Big data, data warehousing?
│   └── L-series (L8s_v3) — high disk I/O
│
└── Machine learning, GPU rendering?
    └── N-series (NC, ND, NV) — GPU attached
```

```bash
# List available VM sizes in a region
az vm list-sizes --location "eastus" -o table

# List available VM sizes for an existing VM (what you can resize to)
az vm list-vm-resize-options \
    --resource-group "rg-demo-vms" \
    --name "my-first-vm" \
    -o table

# Resize a VM (VM must be stopped/deallocated first for some sizes)
az vm resize \
    --resource-group "rg-demo-vms" \
    --name "my-first-vm" \
    --size "Standard_D4s_v5"
```

## 5. VM Images — OS Options

An **image** is the template used to create a VM. It includes the OS and sometimes pre-installed software.

### Common Images

```bash
# List popular images
az vm image list -o table

# Common ones:
# Ubuntu:    Ubuntu2204 (22.04 LTS)
# Debian:    Debian11
# CentOS:    CentOS85  (end-of-life, use alternatives)
# RHEL:      RHEL9-LVM
# Windows:   Win2022Datacenter
# Windows:   Win2019Datacenter
```

### Custom Images

You can create your own images with pre-installed software:

```bash
# Step 1: Create a VM and install everything you need
az vm create --name "template-vm" --image "Ubuntu2204" ...
# SSH in, install your app, configure everything

# Step 2: Generalize the VM (prepare for imaging)
# Inside the VM:
sudo waagent -deprovision+user -force

# Step 3: Deallocate and generalize in Azure
az vm deallocate --resource-group "rg-demo" --name "template-vm"
az vm generalize --resource-group "rg-demo" --name "template-vm"

# Step 4: Create the image
az image create \
    --resource-group "rg-demo" \
    --name "my-app-image-v1" \
    --source "template-vm"

# Step 5: Create new VMs from your custom image
az vm create \
    --resource-group "rg-demo" \
    --name "app-server-01" \
    --image "my-app-image-v1" \
    --size "Standard_D2s_v5"
# This VM will have ALL your pre-installed software!
```

### Azure Compute Gallery (formerly Shared Image Gallery)

For managing images at scale across regions and subscriptions:

```
Azure Compute Gallery:
├── Image Definition: "my-web-app"
│   ├── Version 1.0.0 → replicated to East US, West US
│   ├── Version 1.1.0 → replicated to East US, West US, Europe
│   └── Version 1.2.0 → replicated to all regions
│
├── Image Definition: "my-api-server"
│   ├── Version 1.0.0
```

## 6. VM Storage — Disks Explained

### Types of Disks

```
┌───────────────────────────────────────────────────────────────┐
│                    AZURE MANAGED DISKS                        │
├───────────────┬───────────┬────────────┬──────────────────────┤
│ Type          │ IOPS      │ Throughput │ Use Case             │
├───────────────┼───────────┼────────────┼──────────────────────┤
│ Ultra Disk    │ Up to     │ Up to      │ SAP HANA, top-tier   │
│ (SSD)        │ 160,000   │ 4,000 MB/s │ databases            │
├───────────────┼───────────┼────────────┼──────────────────────┤
│ Premium SSD   │ Up to     │ Up to      │ Production workloads,│
│ v2           │ 80,000    │ 1,200 MB/s │ databases            │
├───────────────┼───────────┼────────────┼──────────────────────┤
│ Premium SSD   │ Up to     │ Up to      │ Production web       │
│              │ 20,000    │ 900 MB/s   │ servers, apps        │
├───────────────┼───────────┼────────────┼──────────────────────┤
│ Standard SSD  │ Up to     │ Up to      │ Dev/test, low-traffic│
│              │ 6,000     │ 750 MB/s   │ web servers          │
├───────────────┼───────────┼────────────┼──────────────────────┤
│ Standard HDD  │ Up to     │ Up to      │ Backups, archival,   │
│              │ 2,000     │ 500 MB/s   │ infrequent access    │
└───────────────┴───────────┴────────────┴──────────────────────┘

Price comparison (128 GB disk, East US, approximate):
Ultra Disk:    ~$75/month
Premium SSD:   ~$19/month
Standard SSD:  ~$10/month
Standard HDD:  ~$5/month
```

### OS Disk vs Data Disk vs Temp Disk

```
OS Disk:
├── Contains the operating system
├── Created automatically with the VM
├── Persistent (data survives VM restart)
├── Usually 30 GB (Linux) or 127 GB (Windows)
└── Should be Premium SSD for production

Data Disk:
├── Extra disks you attach to the VM
├── Persistent (data survives VM restart)
├── Can attach multiple data disks (depends on VM size)
├── Used for: databases, application data, logs
└── Can be detached and attached to different VMs

Temp Disk (Ephemeral):
├── Comes with most VM sizes (size varies)
├── ⚠️ NOT persistent (data LOST on deallocate/resize!)
├── Mounted at /dev/sdb (Linux) or D:\ (Windows)
├── Good for: temp files, swap, cache
└── FREE — included with the VM
```

```bash
# Attach a data disk to a VM
az vm disk attach \
    --resource-group "rg-demo-vms" \
    --vm-name "my-first-vm" \
    --name "my-data-disk" \
    --size-gb 128 \
    --sku Premium_LRS \
    --new

# Inside the VM, you need to partition, format, and mount it:
# (SSH into the VM first)

# Find the new disk
lsblk
# Output shows /dev/sdc (or similar) — the new disk

# Partition it
sudo fdisk /dev/sdc
# Type: n (new), p (primary), 1 (partition 1), Enter, Enter, w (write)

# Format it
sudo mkfs.ext4 /dev/sdc1

# Create mount point and mount
sudo mkdir /datadisk
sudo mount /dev/sdc1 /datadisk

# Make it persistent (survives reboot)
echo '/dev/sdc1 /datadisk ext4 defaults 0 2' | sudo tee -a /etc/fstab
```

## 7. VM Networking

### Network Components

```
┌─────────────────────────────────────────────────┐
│                Internet                          │
└───────────────────┬─────────────────────────────┘
                    │
          ┌─────────┴─────────┐
          │    Public IP       │  ← Optional, for internet access
          │  (20.123.45.67)    │
          └─────────┬─────────┘
                    │
          ┌─────────┴─────────┐
          │    NSG (Firewall)  │  ← Controls inbound/outbound traffic
          │  Allow: SSH (22)   │
          │  Allow: HTTP (80)  │
          │  Deny: Everything  │
          └─────────┬─────────┘
                    │
          ┌─────────┴─────────┐
          │    NIC             │  ← Network Interface Card
          │  Private IP:       │
          │  10.0.1.4          │
          └─────────┬─────────┘
                    │
          ┌─────────┴─────────┐
          │    Subnet          │  ← 10.0.1.0/24
          │    (within VNet)   │
          └─────────┬─────────┘
                    │
          ┌─────────┴─────────┐
          │    VNet            │  ← 10.0.0.0/16
          │  (Virtual Network) │
          └───────────────────┘
```

### NSG Rules

```bash
# Create an NSG
az network nsg create \
    --resource-group "rg-demo-vms" \
    --name "my-vm-nsg"

# Allow SSH (port 22)
az network nsg rule create \
    --resource-group "rg-demo-vms" \
    --nsg-name "my-vm-nsg" \
    --name "AllowSSH" \
    --priority 1000 \
    --protocol Tcp \
    --destination-port-ranges 22 \
    --access Allow \
    --direction Inbound

# Allow HTTP (port 80)
az network nsg rule create \
    --resource-group "rg-demo-vms" \
    --nsg-name "my-vm-nsg" \
    --name "AllowHTTP" \
    --priority 1100 \
    --protocol Tcp \
    --destination-port-ranges 80 \
    --access Allow \
    --direction Inbound

# Allow HTTPS (port 443)
az network nsg rule create \
    --resource-group "rg-demo-vms" \
    --nsg-name "my-vm-nsg" \
    --name "AllowHTTPS" \
    --priority 1200 \
    --protocol Tcp \
    --destination-port-ranges 443 \
    --access Allow \
    --direction Inbound

# ⚠️ Best practice: Restrict SSH to YOUR IP only
az network nsg rule create \
    --resource-group "rg-demo-vms" \
    --nsg-name "my-vm-nsg" \
    --name "AllowSSH-MyIP" \
    --priority 1000 \
    --protocol Tcp \
    --destination-port-ranges 22 \
    --source-address-prefixes "203.0.113.50" \
    --access Allow \
    --direction Inbound
# ^^^ Only allows SSH from your specific IP address
```

## 8. Connecting to VMs

### Linux VMs — SSH

```bash
# Connect using SSH key (most secure)
ssh azureuser@20.123.45.67

# Connect using a specific SSH key file
ssh -i ~/.ssh/my-key.pem azureuser@20.123.45.67

# Connect through Azure Bastion (no public IP needed!)
az network bastion ssh \
    --name "my-bastion" \
    --resource-group "rg-demo-vms" \
    --target-resource-id "/subscriptions/.../virtualMachines/my-vm" \
    --auth-type ssh-key \
    --username azureuser \
    --ssh-key ~/.ssh/id_rsa
```

### Windows VMs — RDP

```bash
# Get the public IP
az vm show -d \
    --resource-group "rg-demo-vms" \
    --name "my-windows-vm" \
    --query publicIps -o tsv

# Open Remote Desktop Connection:
# 1. Press Win+R, type "mstsc"
# 2. Enter the public IP
# 3. Enter username and password
```

### Azure Bastion — The Secure Way

```
Without Bastion (❌ risky):
Internet → Public IP → VM
• VM has a public IP exposed to the internet
• Anyone can try to SSH/RDP to it
• Need to manage NSG rules carefully

With Bastion (✅ secure):
Browser → Azure Portal → Bastion → Private IP → VM
• VM has NO public IP at all!
• Connect through Azure Portal in your browser
• Bastion handles the security
• No need to manage SSH keys or RDP files
```

```bash
# Create Azure Bastion
az network bastion create \
    --name "my-bastion" \
    --resource-group "rg-demo-vms" \
    --vnet-name "my-vnet" \
    --public-ip-address "bastion-pip" \
    --sku Standard
```

## 9. VM States & Billing

### VM Power States

```
┌──────────────────────────────────────────────────────────────┐
│                     VM POWER STATES                          │
├──────────────┬───────────────────┬───────────────────────────┤
│ State        │ Compute Billing   │ What's Happening          │
├──────────────┼───────────────────┼───────────────────────────┤
│ Starting     │ 💰 YES            │ VM is booting up          │
│ Running      │ 💰 YES            │ VM is on and working      │
│ Stopping     │ 💰 YES            │ VM is shutting down       │
│ Stopped      │ 💰 YES ⚠️         │ OS shutdown but not       │
│              │                   │ deallocated!              │
│ Deallocating │ 🚫 NO             │ Releasing compute         │
│ Deallocated  │ 🚫 NO             │ Compute released          │
└──────────────┴───────────────────┴───────────────────────────┘

⚠️ IMPORTANT:
"Stopped" ≠ "Deallocated"!

If you shut down from INSIDE the VM (sudo shutdown -h now):
→ VM is "Stopped" → YOU ARE STILL BEING CHARGED! 💸

If you deallocate from Azure (az vm deallocate):
→ VM is "Deallocated" → No compute charges ✅
→ BUT you still pay for disks and public IP
```

```bash
# Stop AND deallocate (stop paying for compute)
az vm deallocate \
    --resource-group "rg-demo-vms" \
    --name "my-first-vm"
# ^^^ This is what you want to save money!

# Just stop (still paying for compute!)
az vm stop \
    --resource-group "rg-demo-vms" \
    --name "my-first-vm"
# ^^^ DON'T use this to save money!

# Start the VM back up
az vm start \
    --resource-group "rg-demo-vms" \
    --name "my-first-vm"

# Restart the VM
az vm restart \
    --resource-group "rg-demo-vms" \
    --name "my-first-vm"

# Check VM power state
az vm get-instance-view \
    --resource-group "rg-demo-vms" \
    --name "my-first-vm" \
    --query instanceView.statuses[1].displayStatus -o tsv
```

### What You Pay For When VM is Deallocated

```
VM Deallocated:
├── Compute:    🚫 FREE (no charge)
├── OS Disk:    💰 Still charged (persistent storage)
├── Data Disks: 💰 Still charged (persistent storage)
├── Public IP:  💰 Still charged (static IP reservation)
└── Total:      Much less than running, but not zero!

To truly stop ALL charges:
→ Delete the VM and ALL associated resources
```

## 12. Availability Sets

### The Problem

```
Imagine all your VMs are in the same physical server rack:

Server Rack A:
├── VM-1 (your web server)
├── VM-2 (your web server)
└── VM-3 (your web server)

If the rack loses power or the top-of-rack switch fails:
→ ALL your VMs go down at once! 😱
→ Your website is completely offline!
```

### The Solution: Availability Sets

An **Availability Set** tells Azure: "Please don't put all my VMs in the same place." Azure distributes your VMs across different hardware to avoid single points of failure.

### How It Works — Fault Domains & Update Domains

```
AVAILABILITY SET
├── Fault Domains (FDs): Protection against HARDWARE failure
│   Think: Different physical server racks
│   
│   ┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│   │ Fault       │  │ Fault       │  │ Fault       │
│   │ Domain 0    │  │ Domain 1    │  │ Domain 2    │
│   │             │  │             │  │             │
│   │ ┌────────┐  │  │ ┌────────┐  │  │ ┌────────┐  │
│   │ │ VM-1   │  │  │ │ VM-2   │  │  │ │ VM-3   │  │
│   │ └────────┘  │  │ └────────┘  │  │ └────────┘  │
│   │             │  │             │  │             │
│   │ Own power   │  │ Own power   │  │ Own power   │
│   │ Own network │  │ Own network │  │ Own network │
│   └─────────────┘  └─────────────┘  └─────────────┘
│   
│   If Rack 0 loses power → VM-2 and VM-3 keep running ✅
│
└── Update Domains (UDs): Protection against MAINTENANCE reboot
    Think: Groups that reboot at different times
    
    During planned maintenance, Azure reboots ONE update domain 
    at a time. So not all your VMs reboot simultaneously.
    
    UD-0: VM-1, VM-4  ← Reboot these first
    UD-1: VM-2, VM-5  ← Then these
    UD-2: VM-3, VM-6  ← Then these
    
    At any given time, at least 2/3 of your VMs are running!
```

### Key Numbers

| Property | Value |
|---|---|
| Max Fault Domains | 2 or 3 (depends on region) |
| Max Update Domains | 20 |
| Max VMs per Availability Set | 200 |

### Creating an Availability Set

```bash
# Create an availability set
az vm availability-set create \
    --resource-group "rg-demo-vms" \
    --name "avset-webservers" \
    --platform-fault-domain-count 3 \
    --platform-update-domain-count 5

# Create VMs inside the availability set
az vm create \
    --resource-group "rg-demo-vms" \
    --name "web-server-01" \
    --image "Ubuntu2204" \
    --size "Standard_D2s_v5" \
    --availability-set "avset-webservers" \
    --admin-username "azureuser" \
    --generate-ssh-keys

az vm create \
    --resource-group "rg-demo-vms" \
    --name "web-server-02" \
    --image "Ubuntu2204" \
    --size "Standard_D2s_v5" \
    --availability-set "avset-webservers" \
    --admin-username "azureuser" \
    --generate-ssh-keys

# Azure automatically places these VMs in different fault domains!
```

### Important Rules

```
⚠️ AVAILABILITY SET RULES:
1. All VMs in a set MUST be the same size (or same size family)
2. You MUST add VMs when creating them (can't move existing VMs in)
3. An availability set is in ONE region only
4. You get 99.95% SLA with 2+ VMs in an availability set
5. Availability sets are FREE — you only pay for the VMs
```

## 15. Virtual Machine Scale Sets (VMSS)

### What is VMSS?

A **Virtual Machine Scale Set** is a group of identical VMs that can **automatically scale up or down** based on demand. Think of it as "auto-scaling VMs."

### Simple Analogy

```
Without VMSS:
Monday:    Traffic is low → You have 2 VMs → Paying for 2 VMs ✅
Friday:    Traffic spikes → You have 2 VMs → SITE IS SLOW! 😱
           You frantically create 3 more VMs manually...
Weekend:   Traffic drops → You have 5 VMs → Paying for 5 VMs 💸
           You forget to delete the extra ones...

With VMSS:
Monday:    Traffic is low → VMSS runs 2 VMs → Paying for 2 ✅
Friday:    Traffic spikes → VMSS auto-adds 3 more → 5 VMs → FAST! ✅
Weekend:   Traffic drops → VMSS auto-removes 3 → 2 VMs → SAVED $! ✅
           All automatic. No human intervention needed! 🎉
```

### Key Features

```
VMSS Features:
├── 🔄 Auto-scaling — add/remove VMs based on metrics
├── 🏗️ All VMs are identical — same image, size, config
├── 🌐 Integrated with Load Balancer — traffic distributed
├── 📊 Up to 1,000 VMs per scale set
├── 🔒 Supports Availability Zones (cross-zone)
├── 🔄 Rolling updates — update VMs without downtime
├── 💰 Supports Spot VMs — save up to 90%
└── 📈 Supports both manual and auto-scaling
```

### Creating a VMSS

```bash
# Create a VMSS with 3 initial instances
az vmss create \
    --resource-group "rg-demo-vms" \
    --name "webserver-vmss" \
    --image "Ubuntu2204" \
    --vm-sku "Standard_D2s_v5" \
    --instance-count 3 \
    --admin-username "azureuser" \
    --generate-ssh-keys \
    --load-balancer "web-lb" \
    --zones 1 2 3 \
    --upgrade-policy-mode Automatic \
    --custom-data cloud-init.yaml \
    --tags Environment=Production Team=WebApp

# What this does:
# --instance-count 3      → Start with 3 identical VMs
# --load-balancer "web-lb"→ Auto-creates a load balancer
# --zones 1 2 3           → Spread VMs across all 3 zones
# --upgrade-policy-mode   → How updates are applied
# --custom-data           → Cloud-init runs on each VM at boot
```

### Managing VMSS Instances

```bash
# List all instances in the scale set
az vmss list-instances \
    --resource-group "rg-demo-vms" \
    --name "webserver-vmss" \
    -o table

# Manually scale to 5 instances
az vmss scale \
    --resource-group "rg-demo-vms" \
    --name "webserver-vmss" \
    --new-capacity 5

# Get the current instance count
az vmss show \
    --resource-group "rg-demo-vms" \
    --name "webserver-vmss" \
    --query sku.capacity

# Restart a specific instance
az vmss restart \
    --resource-group "rg-demo-vms" \
    --name "webserver-vmss" \
    --instance-ids 3

# Delete a specific instance (VMSS will replace it if min count isn't met)
az vmss delete-instances \
    --resource-group "rg-demo-vms" \
    --name "webserver-vmss" \
    --instance-ids 3

# Update the image on all instances
az vmss update \
    --resource-group "rg-demo-vms" \
    --name "webserver-vmss" \
    --set virtualMachineProfile.storageProfile.imageReference.version=latest
```

## 16. VMSS — Auto-Scaling

### How Auto-Scaling Works

```
                  Auto-Scale Rule
                  ┌──────────────────────┐
                  │ IF CPU > 75%         │
                  │   for 10 minutes     │
                  │ THEN add 2 VMs       │
                  │                      │
                  │ IF CPU < 25%         │
                  │   for 10 minutes     │
                  │ THEN remove 1 VM     │
                  │                      │
                  │ Min instances: 2     │
                  │ Max instances: 10    │
                  └──────────────────────┘

Timeline:
8:00 AM  - Traffic low     → 2 VMs (minimum)
9:00 AM  - Traffic rises   → 2 VMs (CPU at 50%, below threshold)
10:00 AM - Traffic spikes  → 4 VMs (CPU was >75%, added 2)
11:00 AM - Traffic peaks   → 6 VMs (CPU still >75%, added 2 more)
2:00 PM  - Traffic drops   → 5 VMs (CPU <25%, removed 1)
6:00 PM  - Traffic low     → 2 VMs (gradually scaled down)
```

### Setting Up Auto-Scaling

```bash
# Create auto-scale settings
az monitor autoscale create \
    --resource-group "rg-demo-vms" \
    --resource "webserver-vmss" \
    --resource-type Microsoft.Compute/virtualMachineScaleSets \
    --name "autoscale-cpu" \
    --min-count 2 \
    --max-count 10 \
    --count 3

# Scale OUT rule: Add 2 VMs when average CPU > 75% for 10 minutes
az monitor autoscale rule create \
    --resource-group "rg-demo-vms" \
    --autoscale-name "autoscale-cpu" \
    --condition "Percentage CPU > 75 avg 10m" \
    --scale out 2

# Scale IN rule: Remove 1 VM when average CPU < 25% for 10 minutes
az monitor autoscale rule create \
    --resource-group "rg-demo-vms" \
    --autoscale-name "autoscale-cpu" \
    --condition "Percentage CPU < 25 avg 10m" \
    --scale in 1
```

### Scaling Metrics

You can scale on various metrics, not just CPU:

| Metric | When to Use |
|---|---|
| **Percentage CPU** | General compute workloads |
| **Available Memory Bytes** | Memory-intensive apps |
| **Network In/Out** | Network-heavy workloads |
| **Disk Read/Write Operations** | I/O-heavy workloads |
| **HTTP Queue Length** | Web servers behind load balancer |
| **Custom Metrics** | App-specific (e.g., messages in queue) |

### Schedule-Based Scaling

```bash
# Scale to 5 instances every weekday at 8 AM
az monitor autoscale profile create \
    --resource-group "rg-demo-vms" \
    --autoscale-name "autoscale-cpu" \
    --name "business-hours" \
    --min-count 5 \
    --max-count 10 \
    --count 5 \
    --recurrence week Mon Tue Wed Thu Fri \
    --start "08:00" \
    --end "18:00" \
    --timezone "Eastern Standard Time"

# Scale to 2 instances on weekends and evenings
# (This happens automatically as the default profile)
```

## 17. VMSS — Update Policies

When you update the VMSS model (change image, size, etc.), how should the changes roll out to existing instances?

### Three Update Policies

```
┌──────────────────────────────────────────────────────────────┐
│                    UPDATE POLICIES                           │
├──────────────┬───────────────────────────────────────────────┤
│              │                                               │
│  AUTOMATIC   │  All instances update immediately             │
│              │  ⚡ Fast, but ALL VMs reboot at once           │
│              │  ⚠️ Possible brief downtime                    │
│              │  Best for: Dev/Test environments               │
│              │                                               │
├──────────────┼───────────────────────────────────────────────┤
│              │                                               │
│  ROLLING     │  Updates in batches (e.g., 20% at a time)    │
│              │  ✅ Zero downtime (if configured right)        │
│              │  ⏱️ Takes longer to complete                   │
│              │  Best for: Production environments             │
│              │                                               │
├──────────────┼───────────────────────────────────────────────┤
│              │                                               │
│  MANUAL      │  You decide when to update each instance      │
│              │  🎮 Full control                               │
│              │  👀 Requires manual intervention               │
│              │  Best for: Careful, controlled deployments     │
│              │                                               │
└──────────────┴───────────────────────────────────────────────┘
```

### Rolling Update Configuration

```bash
# Configure rolling update policy
az vmss update \
    --resource-group "rg-demo-vms" \
    --name "webserver-vmss" \
    --set upgradePolicy.mode=Rolling \
    --set upgradePolicy.rollingUpgradePolicy.maxBatchInstancePercent=20 \
    --set upgradePolicy.rollingUpgradePolicy.maxUnhealthyInstancePercent=20 \
    --set upgradePolicy.rollingUpgradePolicy.pauseTimeBetweenBatches="PT5S"

# What this means:
# - Update 20% of instances at a time
# - If more than 20% become unhealthy, STOP the rollout
# - Wait 5 seconds between batches
# - While one batch is updating, the other 80% handle traffic

# Manually trigger update on specific instances
az vmss update-instances \
    --resource-group "rg-demo-vms" \
    --name "webserver-vmss" \
    --instance-ids 0 1 2
```

## 19. Spot VMs — Save Up to 90%

### What Are Spot VMs?

Spot VMs use Azure's **spare/unused capacity** at massive discounts (up to 90% off!). The catch? Azure can **evict** (take back) your VM at any time when it needs the capacity.

```
Regular VM:   $140/month for D4s_v5
Spot VM:      $14/month for D4s_v5 (90% off!)

But...
Azure: "Hey, I need that capacity back."
Your Spot VM: *gets evicted* 💀

This is fine for:
✅ Batch processing (can restart where you left off)
✅ CI/CD build agents (just restart the build)
✅ Dev/test environments (not critical)
✅ Rendering/encoding jobs (checkpointed)
✅ Stateless workloads behind a load balancer

This is NOT fine for:
❌ Production web servers (users will see downtime)
❌ Databases (data could be lost)
❌ Anything that can't handle being killed randomly
```

### Eviction Policies

```
When Azure needs capacity back, you choose what happens:

┌──────────────────────────────────────────────────┐
│  Eviction Policy: Stop/Deallocate                │
│  → VM stops, disks are kept                      │
│  → VM will restart when capacity is available     │
│  → You still pay for disks                        │
│  → Best for: Jobs you want to resume              │
├──────────────────────────────────────────────────┤
│  Eviction Policy: Delete                          │
│  → VM AND disks are deleted                       │
│  → You stop paying entirely                       │
│  → Best for: Truly disposable workloads           │
└──────────────────────────────────────────────────┘
```

```bash
# Create a Spot VM
az vm create \
    --resource-group "rg-demo-vms" \
    --name "spot-worker" \
    --image "Ubuntu2204" \
    --size "Standard_D4s_v5" \
    --priority Spot \
    --eviction-policy Deallocate \
    --max-price 0.05 \
    --admin-username "azureuser" \
    --generate-ssh-keys

# --priority Spot          → Makes it a Spot VM
# --eviction-policy        → What happens when evicted
# --max-price 0.05         → Maximum hourly price you're willing to pay
#                             If spot price exceeds this, VM gets evicted
#                             Use -1 for "I'll pay up to on-demand price"

# Create a VMSS with Spot VMs
az vmss create \
    --resource-group "rg-demo-vms" \
    --name "spot-vmss" \
    --image "Ubuntu2204" \
    --vm-sku "Standard_D2s_v5" \
    --instance-count 5 \
    --priority Spot \
    --eviction-policy Delete \
    --max-price -1 \
    --admin-username "azureuser" \
    --generate-ssh-keys
```

## 24. Troubleshooting Common VM Issues

### Can't SSH / RDP into VM

```
Checklist:
1. Is the VM running?
   az vm get-instance-view -g rg -n vm --query instanceView.statuses[1]

2. Does the VM have a public IP?
   az vm show -d -g rg -n vm --query publicIps

3. Are NSG rules allowing the connection?
   az network nsg rule list -g rg --nsg-name vm-nsg -o table
   → Is port 22 (SSH) or 3389 (RDP) allowed from your IP?

4. Is the firewall inside the VM blocking?
   → Use Azure Serial Console to check
   → sudo ufw status (Linux)

5. Is the SSH service running inside the VM?
   → Use Azure Run Command:
   az vm run-command invoke -g rg -n vm \
       --command-id RunShellScript \
       --scripts "systemctl status sshd"
```

### VM Won't Start

```
Checklist:
1. Check VM status:
   az vm get-instance-view -g rg -n vm

2. Check boot diagnostics:
   az vm boot-diagnostics get-boot-log -g rg -n vm

3. Is there a quota issue?
   az vm list-usage --location eastus -o table

4. Disk issues?
   → Try swapping the OS disk
   → Attach disk to a rescue VM

5. Run the VM Repair extension:
   az vm repair create -g rg -n vm --repair-username repair
```

### VM is Slow

```
Checklist:
1. Check CPU usage:
   az vm run-command invoke -g rg -n vm \
       --command-id RunShellScript \
       --scripts "top -bn1 | head -20"

2. Check memory:
   az vm run-command invoke -g rg -n vm \
       --command-id RunShellScript \
       --scripts "free -m"

3. Check disk I/O:
   az vm run-command invoke -g rg -n vm \
       --command-id RunShellScript \
       --scripts "iostat -x 1 5"

4. Is the VM size too small? → Resize it!
   az vm resize -g rg -n vm --size Standard_D4s_v5

5. Is the disk throttled? → Upgrade disk type
   Standard HDD → Standard SSD → Premium SSD
```

### Setup Script for This Architecture

```bash
#!/bin/bash
# Production Web App Setup Script

RESOURCE_GROUP="rg-webapp-prod"
LOCATION="eastus"
VMSS_NAME="webapp-vmss"

# 1. Create Resource Group
az group create -n $RESOURCE_GROUP -l $LOCATION \
    --tags Environment=Production Team=WebApp

# 2. Create VNet
az network vnet create \
    -g $RESOURCE_GROUP -n webapp-vnet \
    --address-prefix 10.0.0.0/16 \
    --subnet-name web-subnet \
    --subnet-prefix 10.0.1.0/24

# 3. Create NSG with rules
az network nsg create -g $RESOURCE_GROUP -n webapp-nsg

az network nsg rule create -g $RESOURCE_GROUP --nsg-name webapp-nsg \
    -n AllowHTTP --priority 1000 --protocol Tcp \
    --destination-port-ranges 80 --access Allow --direction Inbound

az network nsg rule create -g $RESOURCE_GROUP --nsg-name webapp-nsg \
    -n AllowHTTPS --priority 1100 --protocol Tcp \
    --destination-port-ranges 443 --access Allow --direction Inbound

# 4. Create VMSS across 3 Availability Zones
az vmss create \
    -g $RESOURCE_GROUP -n $VMSS_NAME \
    --image "Ubuntu2204" \
    --vm-sku "Standard_D2s_v5" \
    --instance-count 3 \
    --zones 1 2 3 \
    --upgrade-policy-mode Rolling \
    --load-balancer webapp-lb \
    --admin-username azureuser \
    --generate-ssh-keys \
    --vnet-name webapp-vnet \
    --subnet web-subnet \
    --nsg webapp-nsg \
    --custom-data cloud-init.yaml \
    --tags Environment=Production Team=WebApp

# 5. Configure Auto-Scaling
az monitor autoscale create \
    -g $RESOURCE_GROUP \
    --resource $VMSS_NAME \
    --resource-type Microsoft.Compute/virtualMachineScaleSets \
    --name webapp-autoscale \
    --min-count 3 --max-count 12 --count 3

az monitor autoscale rule create \
    -g $RESOURCE_GROUP --autoscale-name webapp-autoscale \
    --condition "Percentage CPU > 75 avg 10m" \
    --scale out 2

az monitor autoscale rule create \
    -g $RESOURCE_GROUP --autoscale-name webapp-autoscale \
    --condition "Percentage CPU < 25 avg 10m" \
    --scale in 1

# 6. Apply Resource Lock
az lock create -n protect-prod -g $RESOURCE_GROUP \
    --lock-type CanNotDelete \
    --notes "Production environment - do not delete!"

echo "✅ Production web application deployed successfully!"
echo "Load Balancer IP:"
az network public-ip show \
    -g $RESOURCE_GROUP -n webapp-lbPublicIP \
    --query ipAddress -o tsv
```

## 27. Interview Questions

### Basic Level

**Q: What is a VM in Azure?**
> A: An Azure Virtual Machine is a cloud-based compute resource that provides a fully functional computer (CPU, RAM, storage, networking) without needing physical hardware. You can run Windows or Linux and pay by the second.

**Q: What's the difference between "Stopped" and "Deallocated"?**
> A: "Stopped" means the OS is shut down but compute resources are still allocated — you're STILL being billed. "Deallocated" means compute resources are released — you're NOT billed for compute (but still billed for disks and static IPs).

**Q: What are VM series and how do you pick one?**
> A: VM series are categories optimized for specific workloads: B-series (burstable/dev), D-series (general purpose), E-series (memory-optimized), F-series (compute-optimized), N-series (GPU). Pick based on your workload's CPU, memory, and I/O requirements.

### Intermediate Level

**Q: What is the difference between Availability Sets and Availability Zones?**
> A: 
> - **Availability Sets** protect against rack-level hardware failures within a single data center using Fault Domains and Update Domains. SLA: 99.95%.
> - **Availability Zones** protect against entire data center failures by distributing VMs across physically separate data centers within a region. SLA: 99.99%.
> Zones provide stronger protection and are recommended for new deployments.

**Q: What is a Virtual Machine Scale Set (VMSS)?**
> A: VMSS is a service that lets you create and manage a group of identical, auto-scaling VMs. It automatically adds VMs when demand increases and removes them when demand decreases. It integrates with Azure Load Balancer for traffic distribution and supports Availability Zones for high availability.

**Q: How would you save costs on Azure VMs?**
> A: Multiple strategies:
> 1. **Right-sizing**: Use Azure Advisor to find over-provisioned VMs
> 2. **Reserved Instances**: Commit for 1-3 years (up to 72% savings)
> 3. **Spot VMs**: Use spare capacity (up to 90% savings for fault-tolerant workloads)
> 4. **Auto-scaling**: Scale down during off-peak hours
> 5. **Deallocate** non-production VMs at night/weekends
> 6. **Azure Hybrid Benefit**: Use existing Windows/SQL licenses

### Advanced Level

**Q: Design a highly available, auto-scaling web application architecture using Azure VMs.**
> A: I would use:
> - **VMSS** across 3 **Availability Zones** for compute (99.99% SLA)
> - **Azure Load Balancer** (Standard SKU, zone-redundant) for traffic distribution
> - **Auto-scaling rules** based on CPU metrics (min 3, max 12 instances)
> - **Rolling update policy** (20% batch size) for zero-downtime deployments
> - **Custom image** from Azure Compute Gallery for consistent deployments
> - **Premium SSD** managed disks for OS
> - **Azure Monitor** with alerts for CPU, memory, and availability
> - **NSG** rules allowing only necessary traffic (80, 443)
> - **Azure Bastion** for management access (no public IPs on VMs)
> - **Azure Backup** with daily snapshots
> - **Resource locks** to prevent accidental deletion

**Q: A VMSS instance is failing health checks and keeps getting replaced. How do you troubleshoot?**
> A: Step-by-step:
> 1. Check **VMSS instance view** for error messages
> 2. Check **boot diagnostics** of the failing instance
> 3. Use **Azure Run Command** to inspect logs inside the VM
> 4. Check the **cloud-init** or custom script output
> 5. Verify the **health probe** configuration (correct port, path, protocol)
> 6. Check **NSG rules** aren't blocking the health probe
> 7. Test the application locally in a standalone VM
> 8. Check if the **custom image** is corrupted — try rebuilding
> 9. Review **extension** logs if using VM extensions
