## 1. Azure Virtual Network (VNet) — The Big Picture

### What is a VNet?

Think of an **Azure Virtual Network (VNet)** as your own **private, isolated network inside Azure**. It's like having your own building in a huge city (Azure). Inside your building, you decide the rooms (subnets), who can enter (security rules), and how people inside your building talk to the outside world.

**In simple terms:** A VNet is a logically isolated network in Azure where you deploy your resources (VMs, databases, app services, etc.) and control how they communicate.

### Why Do You Need a VNet?

Without a VNet, your resources would be floating around with no structure, no security, and no control. A VNet gives you:

| Benefit | What It Means |
|---------|---------------|
| **Isolation** | Your resources are separated from other customers' resources |
| **Segmentation** | You can divide your network into subnets (e.g., frontend, backend, database) |
| **Security** | Control who can talk to whom using NSGs and firewalls |
| **Connectivity** | Connect to on-premises networks, other VNets, or the internet |
| **Name Resolution** | Built-in DNS for your resources |

### Key Characteristics of a VNet

- A VNet belongs to a **single Azure region** (e.g., East US, West Europe)
- A VNet belongs to a **single resource group**
- A VNet has an **address space** defined in CIDR notation (e.g., `10.0.0.0/16`)
- VNets are **free** — you don't pay for VNets themselves (you pay for resources inside them)
- A VNet can have **multiple subnets**
- VNets can be **peered** (connected) to other VNets, even across regions

## 2. Creating a VNet

### Using Azure Portal (GUI Way)

1. Go to **Azure Portal** → Search for **"Virtual Networks"**
2. Click **"+ Create"**
3. Fill in:
   - **Subscription**: Your subscription
   - **Resource Group**: Select or create one
   - **Name**: `my-vnet`
   - **Region**: `East US`
4. Go to **IP Addresses** tab:
   - **Address space**: `10.0.0.0/16` (gives you 65,536 addresses)
   - **Add a subnet**: `default` with `10.0.1.0/24` (256 addresses)
5. Review + Create

### Using Azure CLI

```bash
# Step 1: Create a resource group
az group create \
  --name my-resource-group \
  --location eastus

# Step 2: Create the VNet with a subnet
az network vnet create \
  --resource-group my-resource-group \
  --name my-vnet \
  --address-prefix 10.0.0.0/16 \
  --subnet-name frontend-subnet \
  --subnet-prefix 10.0.1.0/24

# Verify the VNet
az network vnet show \
  --resource-group my-resource-group \
  --name my-vnet \
  --output table
```

**What this does:**
- `--address-prefix 10.0.0.0/16` → The whole VNet gets the address range `10.0.0.0` to `10.0.255.255`
- `--subnet-name frontend-subnet` → Creates a subnet inside the VNet
- `--subnet-prefix 10.0.1.0/24` → That subnet gets addresses `10.0.1.0` to `10.0.1.255`

### Using Terraform

```hcl
# Provider configuration
provider "azurerm" {
  features {}
}

# Resource Group
resource "azurerm_resource_group" "example" {
  name     = "my-resource-group"
  location = "East US"
}

# Virtual Network
resource "azurerm_virtual_network" "example" {
  name                = "my-vnet"
  location            = azurerm_resource_group.example.location
  resource_group_name = azurerm_resource_group.example.name
  address_space       = ["10.0.0.0/16"]   # The whole network range

  tags = {
    environment = "dev"
  }
}

# Subnet
resource "azurerm_subnet" "frontend" {
  name                 = "frontend-subnet"
  resource_group_name  = azurerm_resource_group.example.name
  virtual_network_name = azurerm_virtual_network.example.name
  address_prefixes     = ["10.0.1.0/24"]   # Subset of the VNet range
}
```

**Line-by-line explanation:**
- `address_space = ["10.0.0.0/16"]` — Defines how many IP addresses your VNet has. `/16` = 65,536 addresses
- `address_prefixes = ["10.0.1.0/24"]` — This subnet gets 256 of those addresses (from `10.0.1.0` to `10.0.1.255`)

## 3. Subnets — Dividing Your Network

### What is a Subnet?

A **subnet** is a smaller network inside your VNet. It's like dividing a floor of your building into different rooms — one room for web servers, one for app servers, one for databases.

### Why Use Subnets?

| Reason | Example |
|--------|---------|
| **Organize resources** | Frontend servers in one subnet, backend in another |
| **Apply different security rules** | Database subnet blocks all internet traffic |
| **Compliance** | Keep sensitive data in a separate subnet |
| **Route traffic differently** | Public-facing subnet has internet access, backend doesn't |

### Common Subnet Architecture

```
VNet: 10.0.0.0/16
│
├── frontend-subnet:    10.0.1.0/24  (Web servers, public-facing)
├── backend-subnet:     10.0.2.0/24  (App servers, APIs)
├── database-subnet:    10.0.3.0/24  (Databases, no internet)
├── management-subnet:  10.0.4.0/24  (Jump boxes, bastion host)
└── gateway-subnet:     10.0.255.0/24 (VPN Gateway — MUST be named "GatewaySubnet")
```

### Creating Multiple Subnets

```bash
# Add more subnets to existing VNet
az network vnet subnet create \
  --resource-group my-resource-group \
  --vnet-name my-vnet \
  --name backend-subnet \
  --address-prefixes 10.0.2.0/24

az network vnet subnet create \
  --resource-group my-resource-group \
  --vnet-name my-vnet \
  --name database-subnet \
  --address-prefixes 10.0.3.0/24

# List all subnets
az network vnet subnet list \
  --resource-group my-resource-group \
  --vnet-name my-vnet \
  --output table
```

### Reserved Addresses in Every Subnet

Azure reserves **5 IP addresses** in every subnet. So if you create a `/24` subnet (256 addresses), only **251** are usable.

| Reserved IP | Purpose |
|-------------|---------|
| `x.x.x.0` | Network address |
| `x.x.x.1` | Default gateway |
| `x.x.x.2, x.x.x.3` | Azure DNS |
| `x.x.x.255` | Broadcast |

**Pro tip:** Don't make subnets too small! A `/28` subnet gives you only 16 - 5 = 11 usable IPs.

## 4. IP Addressing in Azure

### Public IP vs Private IP

| Feature | Private IP | Public IP |
|---------|-----------|-----------|
| **Where?** | Inside VNet only | Accessible from internet |
| **Range** | 10.x.x.x, 172.16.x.x, 192.168.x.x | Globally unique IP |
| **Cost** | Free | Paid (small cost) |
| **Auto-assigned?** | Yes (DHCP from subnet) | Must create separately |
| **Use case** | Internal communication | Web servers, SSH access |

### SKUs (Standard vs Basic)

| Feature | Basic Public IP | Standard Public IP |
|---------|----------------|-------------------|
| **Allocation** | Dynamic or Static | Static only |
| **Availability Zones** | No | Yes (zone-redundant) |
| **Security** | Open by default | Closed by default (need NSG) |
| **Load Balancer** | Basic LB only | Standard LB only |
| **Cost** | Cheaper | Slightly more |
| **Recommended?** | No (being retired) | **Yes — always use Standard** |

### Creating a Public IP

```bash
# Create a Standard static public IP
az network public-ip create \
  --resource-group my-resource-group \
  --name my-public-ip \
  --sku Standard \
  --allocation-method Static \
  --version IPv4

# Output shows the allocated IP
az network public-ip show \
  --resource-group my-resource-group \
  --name my-public-ip \
  --query ipAddress \
  --output tsv
```

## 5. Network Security Groups (NSG) — Your Virtual Firewall

### What is an NSG?

An **NSG (Network Security Group)** is Azure's basic firewall. It contains a list of **security rules** that allow or deny network traffic to resources in your VNet.

Think of it like a **bouncer at a club** — checking every person (packet) trying to get in or out, and deciding whether to let them through based on a list of rules.

### Where Can You Attach an NSG?

You can attach an NSG to:

1. **A Subnet** — Rules apply to ALL resources in that subnet
2. **A Network Interface (NIC)** — Rules apply to one specific VM

**Best Practice:** Attach NSGs to **subnets** (not individual NICs). It's easier to manage and less error-prone.

```
Internet
   │
   ▼
┌─────────────────────────────────────┐
│ NSG (attached to subnet)            │ ← Checks ALL traffic to/from subnet
│ ┌─────────────────────────────────┐ │
│ │ Subnet: 10.0.1.0/24            │ │
│ │   VM1 (NIC → optional NSG)     │ │ ← Can have its own NSG too
│ │   VM2 (NIC → optional NSG)     │ │
│ └─────────────────────────────────┘ │
└─────────────────────────────────────┘
```

**If you attach NSGs at both levels:** Traffic must pass BOTH NSGs. The subnet NSG is checked first (inbound), then the NIC NSG.

### Creating an NSG

```bash
# Create an NSG
az network nsg create \
  --resource-group my-resource-group \
  --name frontend-nsg

# Associate NSG with a subnet
az network vnet subnet update \
  --resource-group my-resource-group \
  --vnet-name my-vnet \
  --name frontend-subnet \
  --network-security-group frontend-nsg
```

## 6. NSG Rules Deep Dive

### Rule Components

Every NSG rule has these parts:

| Component | Description | Example |
|-----------|-------------|---------|
| **Priority** | Lower number = higher priority (100-4096) | `100` |
| **Name** | Descriptive name | `Allow-HTTP` |
| **Direction** | Inbound or Outbound | `Inbound` |
| **Source** | Where traffic comes FROM | `*` (any), `10.0.1.0/24`, `Internet` |
| **Source Port** | Source port range | `*` (any) |
| **Destination** | Where traffic goes TO | `*`, `10.0.2.0/24`, `VirtualNetwork` |
| **Destination Port** | Port(s) to allow/deny | `80`, `443`, `22`, `3389` |
| **Protocol** | TCP, UDP, ICMP, or Any | `TCP` |
| **Action** | Allow or Deny | `Allow` |

### Default Rules (Built-in)

Every NSG comes with these default rules that you CANNOT delete:

**Inbound defaults:**

| Priority | Name | Source | Destination | Action |
|----------|------|--------|-------------|--------|
| 65000 | AllowVnetInBound | VirtualNetwork | VirtualNetwork | Allow |
| 65001 | AllowAzureLoadBalancerInBound | AzureLoadBalancer | * | Allow |
| 65500 | DenyAllInBound | * | * | **Deny** |

**Outbound defaults:**

| Priority | Name | Source | Destination | Action |
|----------|------|--------|-------------|--------|
| 65000 | AllowVnetOutBound | VirtualNetwork | VirtualNetwork | Allow |
| 65001 | AllowInternetOutBound | * | Internet | Allow |
| 65500 | DenyAllOutBound | * | * | **Deny** |

**What this means:**
- By default, VMs in the same VNet **can** talk to each other ✅
- By default, VMs **can** access the internet (outbound) ✅
- By default, **nothing from the internet can reach your VMs** (inbound denied) ❌
- You must add rules to allow inbound traffic!

### Adding Custom Rules

```bash
# Allow HTTP (port 80) from anywhere
az network nsg rule create \
  --resource-group my-resource-group \
  --nsg-name frontend-nsg \
  --name Allow-HTTP \
  --priority 100 \
  --direction Inbound \
  --access Allow \
  --protocol TCP \
  --source-address-prefixes '*' \
  --source-port-ranges '*' \
  --destination-address-prefixes '*' \
  --destination-port-ranges 80

# Allow HTTPS (port 443) from anywhere
az network nsg rule create \
  --resource-group my-resource-group \
  --nsg-name frontend-nsg \
  --name Allow-HTTPS \
  --priority 110 \
  --direction Inbound \
  --access Allow \
  --protocol TCP \
  --source-address-prefixes '*' \
  --source-port-ranges '*' \
  --destination-address-prefixes '*' \
  --destination-port-ranges 443

# Allow SSH (port 22) only from YOUR IP
az network nsg rule create \
  --resource-group my-resource-group \
  --nsg-name frontend-nsg \
  --name Allow-SSH-MyIP \
  --priority 120 \
  --direction Inbound \
  --access Allow \
  --protocol TCP \
  --source-address-prefixes '203.0.113.50' \
  --source-port-ranges '*' \
  --destination-address-prefixes '*' \
  --destination-port-ranges 22

# Deny all other inbound (explicit — already default, but good practice)
az network nsg rule create \
  --resource-group my-resource-group \
  --nsg-name frontend-nsg \
  --name Deny-All-Other \
  --priority 4000 \
  --direction Inbound \
  --access Deny \
  --protocol '*' \
  --source-address-prefixes '*' \
  --source-port-ranges '*' \
  --destination-address-prefixes '*' \
  --destination-port-ranges '*'

# List all rules
az network nsg rule list \
  --resource-group my-resource-group \
  --nsg-name frontend-nsg \
  --output table
```

### NSG with Terraform

```hcl
resource "azurerm_network_security_group" "frontend" {
  name                = "frontend-nsg"
  location            = azurerm_resource_group.example.location
  resource_group_name = azurerm_resource_group.example.name

  # Rule 1: Allow HTTP
  security_rule {
    name                       = "Allow-HTTP"
    priority                   = 100
    direction                  = "Inbound"
    access                     = "Allow"
    protocol                   = "Tcp"
    source_port_range          = "*"          # Any source port
    destination_port_range     = "80"         # Only port 80
    source_address_prefix      = "*"          # From anywhere
    destination_address_prefix = "*"          # To any resource in this NSG
  }

  # Rule 2: Allow HTTPS
  security_rule {
    name                       = "Allow-HTTPS"
    priority                   = 110
    direction                  = "Inbound"
    access                     = "Allow"
    protocol                   = "Tcp"
    source_port_range          = "*"
    destination_port_range     = "443"
    source_address_prefix      = "*"
    destination_address_prefix = "*"
  }

  # Rule 3: Allow SSH from specific IP only
  security_rule {
    name                       = "Allow-SSH"
    priority                   = 120
    direction                  = "Inbound"
    access                     = "Allow"
    protocol                   = "Tcp"
    source_port_range          = "*"
    destination_port_range     = "22"
    source_address_prefix      = "203.0.113.50"  # Your IP only!
    destination_address_prefix = "*"
  }
}

# Associate NSG with subnet
resource "azurerm_subnet_network_security_group_association" "frontend" {
  subnet_id                 = azurerm_subnet.frontend.id
  network_security_group_id = azurerm_network_security_group.frontend.id
}
```

### How Rules Are Evaluated

Rules are processed in **priority order** (lowest number first). As soon as a match is found, processing STOPS.

```
Incoming packet → Port 80, TCP, from Internet
   │
   ▼
Priority 100: Allow HTTP (port 80, TCP) → ✅ MATCH → ALLOW → STOP
Priority 110: Allow HTTPS (port 443)    → (never checked)
Priority 120: Allow SSH (port 22)       → (never checked)
Priority 4000: Deny All                 → (never checked)
```

```
Incoming packet → Port 3306, TCP, from Internet
   │
   ▼
Priority 100: Allow HTTP (port 80)      → ❌ no match
Priority 110: Allow HTTPS (port 443)    → ❌ no match
Priority 120: Allow SSH (port 22)       → ❌ no match
Priority 4000: Deny All                 → ✅ MATCH → DENY → STOP
```

## 7. Application Security Groups (ASG)

### The Problem ASGs Solve

Imagine you have 50 web servers and 20 database servers. Without ASGs, you'd need to list all 50 IP addresses in your NSG rules. If you add a new web server, you have to update the rule.

**ASGs let you group VMs by role** and use the group name in NSG rules instead of IP addresses.

### How ASGs Work

```
Instead of this (messy):
  Allow traffic from 10.0.1.4, 10.0.1.5, 10.0.1.6, 10.0.1.7 → port 3306

You do this (clean):
  Allow traffic from ASG "web-servers" → ASG "db-servers" on port 3306
```

### Creating and Using ASGs

```bash
# Create ASGs
az network asg create --resource-group my-resource-group --name web-servers-asg
az network asg create --resource-group my-resource-group --name db-servers-asg

# Create NSG rule using ASGs
az network nsg rule create \
  --resource-group my-resource-group \
  --nsg-name backend-nsg \
  --name Allow-Web-to-DB \
  --priority 100 \
  --direction Inbound \
  --access Allow \
  --protocol TCP \
  --source-asgs web-servers-asg \
  --destination-asgs db-servers-asg \
  --destination-port-ranges 3306
```

Then, when you create a VM's NIC, you assign it to an ASG. No need to track individual IPs!

## 8. Azure Load Balancer — Layer 4 Load Balancing

### What is Azure Load Balancer?

The **Azure Load Balancer** distributes incoming network traffic across multiple VMs. It works at **Layer 4 (TCP/UDP)** — meaning it doesn't look at HTTP headers, URLs, or cookies. It just routes packets based on IP and port.

Think of it like a **traffic police officer** at a junction, sending cars (packets) to different roads (VMs) to avoid traffic jams.

### Public vs Internal Load Balancer

| Feature | Public Load Balancer | Internal Load Balancer |
|---------|---------------------|----------------------|
| **Frontend IP** | Public IP (internet-facing) | Private IP (internal only) |
| **Use case** | Web servers, APIs exposed to internet | Internal services, databases |
| **Who can access?** | Anyone on the internet | Only resources in the VNet |

```
                Internet
                   │
                   ▼
        ┌──────────────────┐
        │  Public Load     │  ← Has a public IP
        │  Balancer        │
        └──────┬───────────┘
               │
    ┌──────────┼──────────┐
    ▼          ▼          ▼
  VM1        VM2        VM3     (Frontend Web Servers)
    │          │          │
    └──────────┼──────────┘
               │
        ┌──────▼───────────┐
        │  Internal Load   │  ← Has a private IP (e.g., 10.0.2.100)
        │  Balancer        │
        └──────┬───────────┘
               │
    ┌──────────┼──────────┐
    ▼          ▼          ▼
  VM4        VM5        VM6     (Backend App Servers)
```

### SKUs: Basic vs Standard

| Feature | Basic (old) | Standard (recommended) |
|---------|-------------|----------------------|
| **Backend pool size** | Up to 300 | Up to 1000 |
| **Health probes** | TCP, HTTP | TCP, HTTP, HTTPS |
| **Availability Zones** | No | Yes |
| **SLA** | No SLA | 99.99% SLA |
| **Security** | Open by default | Closed by default (need NSG) |
| **Cost** | Free | Paid |
| **Should I use it?** | No (being deprecated) | **Always use Standard** |

### Key Components of a Load Balancer

```
┌─────────────────────────────────────────────────────────┐
│                    LOAD BALANCER                         │
│                                                         │
│  ┌──────────────┐   ┌──────────────┐   ┌─────────────┐ │
│  │ Frontend IP  │──▶│ Load Balance │──▶│ Backend     │ │
│  │ Configuration│   │ Rules        │   │ Pool        │ │
│  └──────────────┘   └──────────────┘   └─────────────┘ │
│                          │                    │         │
│                     ┌────▼────┐          ┌────▼────┐   │
│                     │ Health  │          │ VM1,VM2 │   │
│                     │ Probes  │          │ VM3...  │   │
│                     └─────────┘          └─────────┘   │
└─────────────────────────────────────────────────────────┘
```

- **Frontend IP** — The IP address clients connect to
- **Backend Pool** — The group of VMs that receive traffic
- **Load Balancing Rules** — Map frontend IP:port to backend pool:port
- **Health Probes** — Check if VMs are healthy; unhealthy VMs stop receiving traffic

### Creating a Load Balancer (CLI)

```bash
# Step 1: Create a public IP for the load balancer
az network public-ip create \
  --resource-group my-resource-group \
  --name lb-public-ip \
  --sku Standard \
  --allocation-method Static

# Step 2: Create the load balancer
az network lb create \
  --resource-group my-resource-group \
  --name my-load-balancer \
  --sku Standard \
  --public-ip-address lb-public-ip \
  --frontend-ip-name lb-frontend \
  --backend-pool-name lb-backend-pool

# Step 3: Create a health probe
az network lb probe create \
  --resource-group my-resource-group \
  --lb-name my-load-balancer \
  --name http-health-probe \
  --protocol Http \
  --port 80 \
  --path /health \
  --interval 15 \
  --threshold 2

# Step 4: Create a load balancing rule
az network lb rule create \
  --resource-group my-resource-group \
  --lb-name my-load-balancer \
  --name http-rule \
  --protocol Tcp \
  --frontend-port 80 \
  --backend-port 80 \
  --frontend-ip-name lb-frontend \
  --backend-pool-name lb-backend-pool \
  --probe-name http-health-probe \
  --idle-timeout 4
```

**What each step does:**
- **Step 1:** Creates a public IP that clients will use to reach your service
- **Step 2:** Creates the LB with a frontend (the public IP) and a backend pool (empty for now)
- **Step 3:** Creates a health check — LB will hit `/health` on port 80 every 15 seconds. If 2 consecutive checks fail, the VM is marked unhealthy
- **Step 4:** Creates a rule: "Any traffic coming to frontend port 80, send it to backend pool port 80"

### Load Balancing Algorithms

Azure Load Balancer uses a **5-tuple hash** by default:

```
Hash = Source IP + Source Port + Destination IP + Destination Port + Protocol
```

This means the same client connection always goes to the same VM. You can also configure:

| Mode | Description |
|------|-------------|
| **5-tuple (default)** | Best distribution, no stickiness |
| **Source IP (2-tuple)** | Same source IP → same VM |
| **Source IP + Protocol (3-tuple)** | Same source IP + protocol → same VM |

### Load Balancer with Terraform

```hcl
# Public IP for Load Balancer
resource "azurerm_public_ip" "lb" {
  name                = "lb-public-ip"
  location            = azurerm_resource_group.example.location
  resource_group_name = azurerm_resource_group.example.name
  allocation_method   = "Static"
  sku                 = "Standard"
}

# Load Balancer
resource "azurerm_lb" "example" {
  name                = "my-load-balancer"
  location            = azurerm_resource_group.example.location
  resource_group_name = azurerm_resource_group.example.name
  sku                 = "Standard"

  frontend_ip_configuration {
    name                 = "lb-frontend"
    public_ip_address_id = azurerm_public_ip.lb.id
  }
}

# Backend Pool
resource "azurerm_lb_backend_address_pool" "example" {
  loadbalancer_id = azurerm_lb.example.id
  name            = "lb-backend-pool"
}

# Health Probe
resource "azurerm_lb_probe" "example" {
  loadbalancer_id     = azurerm_lb.example.id
  name                = "http-health-probe"
  protocol            = "Http"
  port                = 80
  request_path        = "/health"
  interval_in_seconds = 15
  number_of_probes    = 2
}

# Load Balancing Rule
resource "azurerm_lb_rule" "example" {
  loadbalancer_id                = azurerm_lb.example.id
  name                           = "http-rule"
  protocol                       = "Tcp"
  frontend_port                  = 80
  backend_port                   = 80
  frontend_ip_configuration_name = "lb-frontend"
  backend_address_pool_ids       = [azurerm_lb_backend_address_pool.example.id]
  probe_id                       = azurerm_lb_probe.example.id
  idle_timeout_in_minutes        = 4
}
```

## 9. Azure Application Gateway — Layer 7 Load Balancing

### What is Application Gateway?

While Azure Load Balancer works at Layer 4 (TCP/UDP), **Application Gateway** works at **Layer 7 (HTTP/HTTPS)**. This means it understands HTTP — it can read URLs, headers, cookies, and make smarter routing decisions.

### When to Use Load Balancer vs Application Gateway?

| Feature | Azure Load Balancer | Application Gateway |
|---------|-------------------|-------------------|
| **Layer** | Layer 4 (TCP/UDP) | Layer 7 (HTTP/HTTPS) |
| **Understands HTTP?** | No | Yes |
| **URL-based routing?** | No | Yes (`/api` → pool A, `/images` → pool B) |
| **SSL termination?** | No | Yes (offload SSL from backend VMs) |
| **WAF (Web Application Firewall)?** | No | Yes (v2) |
| **Cookie-based affinity?** | No | Yes |
| **WebSocket support?** | No | Yes |
| **Cost** | Cheaper | More expensive |
| **Use case** | Non-HTTP traffic, simple LB | Web applications, APIs |

### Key Features of Application Gateway

1. **URL Path-Based Routing**
   ```
   www.example.com/images/* → Image Server Pool
   www.example.com/api/*    → API Server Pool
   www.example.com/*        → Default Server Pool
   ```

2. **Multi-Site Hosting**
   ```
   www.siteA.com → Pool A
   www.siteB.com → Pool B
   ```

3. **SSL/TLS Termination** — Decrypt HTTPS at the gateway, forward HTTP to backend (saves backend CPU)

4. **WAF (Web Application Firewall)** — Protects against SQL injection, XSS, and other OWASP Top 10 attacks

5. **Autoscaling** — Automatically scale up/down based on traffic

6. **Session Affinity** — Stick a user to the same backend server using cookies

### Architecture of Application Gateway

```
Internet
   │
   ▼
┌──────────────────────────────────────────────────────────┐
│                APPLICATION GATEWAY                        │
│                                                          │
│  ┌──────────────┐   ┌──────────────┐   ┌──────────────┐ │
│  │  Listeners   │──▶│   Rules      │──▶│ Backend      │ │
│  │ (port, proto │   │ (path-based, │   │ Pools        │ │
│  │  hostname)   │   │  basic)      │   │              │ │
│  └──────────────┘   └──────────────┘   └──────────────┘ │
│         │                                    │           │
│    ┌────▼────┐                          ┌────▼────┐     │
│    │   SSL   │                          │ Health  │     │
│    │ Certs   │                          │ Probes  │     │
│    └─────────┘                          └─────────┘     │
│                                                          │
│  ┌──────────────────────────────────────────────────────┐│
│  │ WAF (Web Application Firewall) — optional           ││
│  └──────────────────────────────────────────────────────┘│
└──────────────────────────────────────────────────────────┘
         │              │              │
         ▼              ▼              ▼
      VM Pool 1     VM Pool 2     VM Pool 3
      (/images)      (/api)       (default)
```

### Application Gateway Components

| Component | What It Does |
|-----------|-------------|
| **Frontend IP** | Public or private IP where clients connect |
| **Listener** | Listens on a specific port + protocol (HTTP:80 or HTTPS:443) |
| **Rule** | Connects a listener to a backend pool (basic or path-based) |
| **Backend Pool** | Group of VMs, VMSS, App Services, or IPs |
| **HTTP Settings** | Port, protocol, cookie affinity, connection draining |
| **Health Probe** | Checks if backend servers are healthy |
| **WAF Policy** | Firewall rules (OWASP rule sets) |

### Creating Application Gateway (CLI)

```bash
# Application Gateway needs its own subnet (dedicated)
az network vnet subnet create \
  --resource-group my-resource-group \
  --vnet-name my-vnet \
  --name appgw-subnet \
  --address-prefixes 10.0.10.0/24

# Create a public IP for the App Gateway
az network public-ip create \
  --resource-group my-resource-group \
  --name appgw-public-ip \
  --sku Standard \
  --allocation-method Static

# Create the Application Gateway (v2)
az network application-gateway create \
  --resource-group my-resource-group \
  --name my-app-gateway \
  --location eastus \
  --sku Standard_v2 \
  --capacity 2 \
  --vnet-name my-vnet \
  --subnet appgw-subnet \
  --public-ip-address appgw-public-ip \
  --frontend-port 80 \
  --http-settings-port 80 \
  --http-settings-protocol Http \
  --servers 10.0.1.4 10.0.1.5
```

**Important:** Application Gateway requires its **own dedicated subnet**. Don't put VMs in the same subnet as the App Gateway!

### Adding URL Path-Based Routing

```bash
# Create a URL path map
az network application-gateway url-path-map create \
  --resource-group my-resource-group \
  --gateway-name my-app-gateway \
  --name my-path-map \
  --default-address-pool defaultPool \
  --default-http-settings defaultSettings \
  --paths /api/* \
  --address-pool apiPool \
  --http-settings apiSettings \
  --rule-name api-rule
```

### Enabling WAF (Web Application Firewall)

To enable WAF, change the SKU from `Standard_v2` to `WAF_v2`:

```hcl

  name     = "WAF_v2"
  tier     = "WAF_v2"
  capacity = 2
}

waf_configuration {
  enabled          = true

  rule_set_version = "3.2"
```

**Firewall modes:**
- **Detection** — Logs malicious requests but doesn't block them (good for testing)
- **Prevention** — Actually blocks malicious requests (use in production)

## 10. VNet Peering — Connecting VNets Together

### What is VNet Peering?

VNet Peering connects two VNets so they can communicate over Azure's backbone network (not the internet). It's like building a bridge between two buildings.

### Types of Peering

| Type | Description |
|------|-------------|
| **Regional Peering** | Two VNets in the same Azure region |
| **Global Peering** | Two VNets in different regions |

### Key Points

- Peering is **non-transitive**: If VNet A peers with VNet B, and VNet B peers with VNet C, VNet A **cannot** talk to VNet C automatically
- Address spaces must **not overlap**
- Peering must be set up **on both sides** (VNet A → B, AND VNet B → A)
- Traffic stays on Azure's private backbone (low latency, high bandwidth)

### Creating VNet Peering

```bash
# Peer VNet A → VNet B
az network vnet peering create \
  --resource-group rg-A \
  --name vnetA-to-vnetB \
  --vnet-name vnet-A \
  --remote-vnet /subscriptions/<sub-id>/resourceGroups/rg-B/providers/Microsoft.Network/virtualNetworks/vnet-B \
  --allow-vnet-access

# Peer VNet B → VNet A (must do both sides!)
az network vnet peering create \
  --resource-group rg-B \
  --name vnetB-to-vnetA \
  --vnet-name vnet-B \
  --remote-vnet /subscriptions/<sub-id>/resourceGroups/rg-A/providers/Microsoft.Network/virtualNetworks/vnet-A \
  --allow-vnet-access
```

## 11. VPN Gateway & ExpressRoute

### VPN Gateway — Connecting On-Premises to Azure

A **VPN Gateway** creates an encrypted tunnel over the internet between your on-premises network and your Azure VNet.

| Type | Description |
|------|-------------|
| **Site-to-Site (S2S)** | Connects your entire on-premises network to Azure VNet |
| **Point-to-Site (P2S)** | Connects a single computer (e.g., laptop) to Azure VNet |
| **VNet-to-VNet** | Connects two Azure VNets via encrypted tunnel |

### ExpressRoute — Dedicated Private Connection

**ExpressRoute** is a private, dedicated connection between your data center and Azure — it does NOT go over the internet. Think of it as a private highway vs a public road.

| Feature | VPN Gateway | ExpressRoute |
|---------|------------|-------------|
| **Connection** | Over internet (encrypted) | Private dedicated line |
| **Bandwidth** | Up to ~1.25 Gbps | Up to 100 Gbps |
| **Latency** | Higher (internet) | Lower (private) |
| **Reliability** | Good | Excellent |
| **Cost** | Cheaper | Expensive |
| **Use case** | Small/medium businesses | Large enterprises |

## 12. Azure DNS

### What is Azure DNS?

Azure DNS lets you host your DNS domains in Azure and manage DNS records using the same tools and billing as your other Azure resources.

```bash
# Create a DNS zone
az network dns zone create \
  --resource-group my-resource-group \
  --name example.com

# Add an A record
az network dns record-set a add-record \
  --resource-group my-resource-group \
  --zone-name example.com \
  --record-set-name www \
  --ipv4-address 20.30.40.50

# Add a CNAME record
az network dns record-set cname set-record \
  --resource-group my-resource-group \
  --zone-name example.com \
  --record-set-name blog \
  --cname blog.wordpress.com
```

### Private DNS Zones

For internal name resolution within your VNet:

```bash
# Create a private DNS zone
az network private-dns zone create \
  --resource-group my-resource-group \
  --name internal.mycompany.com

# Link it to your VNet
az network private-dns link vnet create \
  --resource-group my-resource-group \
  --zone-name internal.mycompany.com \
  --name my-vnet-link \
  --virtual-network my-vnet \
  --registration-enabled true
```

With `--registration-enabled true`, VMs in the VNet automatically get DNS records in the private zone!

## 13. Service Endpoints & Private Endpoints

### Service Endpoints

A **Service Endpoint** provides a direct, optimized route from your VNet to Azure services (like Storage, SQL) over Azure's backbone network.

```bash
# Enable service endpoint for Azure Storage on a subnet
az network vnet subnet update \
  --resource-group my-resource-group \
  --vnet-name my-vnet \
  --name backend-subnet \
  --service-endpoints Microsoft.Storage Microsoft.Sql
```

**How it works:** Traffic from your VNet to Azure Storage stays on Azure's internal network instead of going through the internet.

### Private Endpoints

A **Private Endpoint** goes further — it gives an Azure service (like Storage, SQL, Key Vault) a **private IP address inside your VNet**. The service essentially becomes part of your VNet.

```
Without Private Endpoint:
  VM (10.0.1.4) → Internet → mystorageaccount.blob.core.windows.net (public IP)

With Private Endpoint:
  VM (10.0.1.4) → mystorageaccount.blob.core.windows.net (10.0.3.5 — private IP in your VNet!)
```

| Feature | Service Endpoint | Private Endpoint |
|---------|-----------------|-----------------|
| **Traffic path** | Azure backbone | Private IP in your VNet |
| **Access from on-prem** | No | Yes (via VPN/ExpressRoute) |
| **DNS** | Public DNS name | Private DNS zone |
| **Cost** | Free | Small per-hour cost |

## 14. NAT Gateway

### What is NAT Gateway?

A **NAT Gateway** provides outbound internet connectivity for VMs in a subnet that don't have public IPs. All outbound traffic from the subnet goes through the NAT Gateway's public IP.

**Why use it?**
- Your VMs don't need public IPs (more secure!)
- All outbound traffic uses a predictable public IP (great for whitelisting)
- Handles port exhaustion automatically

```bash
# Create a public IP for NAT Gateway
az network public-ip create \
  --resource-group my-resource-group \
  --name nat-public-ip \
  --sku Standard \
  --allocation-method Static

# Create NAT Gateway
az network nat gateway create \
  --resource-group my-resource-group \
  --name my-nat-gateway \
  --public-ip-addresses nat-public-ip \
  --idle-timeout 4

# Associate with a subnet
az network vnet subnet update \
  --resource-group my-resource-group \
  --vnet-name my-vnet \
  --name backend-subnet \
  --nat-gateway my-nat-gateway
```

## 15. Azure Firewall vs NSG — When to Use What

| Feature | NSG | Azure Firewall |
|---------|-----|---------------|
| **Layer** | Layer 3/4 | Layer 3/4/7 |
| **Scope** | Subnet or NIC | Centralized for entire VNet |
| **FQDN filtering** | No | Yes (e.g., allow *.github.com) |
| **Threat intelligence** | No | Yes |
| **Logging** | Basic | Rich (Azure Monitor, Log Analytics) |
| **Cost** | Free | ~$1.25/hour + data |
| **Use case** | Basic traffic control | Enterprise-grade security |

**Rule of thumb:**
- Use **NSGs** for basic, subnet-level traffic filtering (always)
- Add **Azure Firewall** for centralized, advanced filtering (enterprise)

## 17. Common Interview Questions

**Q: What's the difference between NSG and Azure Firewall?**
NSG is a basic, free, subnet/NIC-level packet filter (L3/L4). Azure Firewall is a centralized, managed, stateful firewall with L7 capabilities, FQDN filtering, and threat intelligence.

**Q: Can VMs in the same VNet but different subnets communicate?**
Yes! By default, subnets within the same VNet can communicate (due to the default `AllowVnetInBound` rule). You can restrict this using NSGs.

**Q: What's the difference between Azure Load Balancer and Application Gateway?**
Load Balancer = Layer 4 (TCP/UDP), simple traffic distribution. Application Gateway = Layer 7 (HTTP/HTTPS), URL-based routing, SSL termination, WAF.

**Q: What is VNet Peering? Is it transitive?**
VNet Peering connects two VNets over Azure's backbone. It is NOT transitive — if A peers with B, and B peers with C, A can't talk to C unless A-C peering is also created.

**Q: How many IPs does Azure reserve in each subnet?**
5 IPs — first four (.0, .1, .2, .3) and last (.255).

**Q: What's the difference between Service Endpoint and Private Endpoint?**
Service Endpoint routes traffic to Azure services over Azure's backbone (but the service keeps its public IP). Private Endpoint gives the service a private IP in your VNet, making it truly private.
