## 1. What is AKS? — The Big Picture

### The Problem

Running Kubernetes yourself is HARD. You need to:
- Set up and maintain the **control plane** (API server, etcd, scheduler, controller manager)
- Handle **upgrades** without downtime
- Manage **certificates**, **networking**, and **storage**
- Deal with **scaling**, **patching**, and **security**

That's a full-time job for a team of engineers!

### The Solution: AKS

**Azure Kubernetes Service (AKS)** is a **managed Kubernetes** offering. Microsoft takes care of the control plane — you just worry about your workloads.

| What Microsoft Manages (Free!) | What You Manage (Your Job) |
|-------------------------------|---------------------------|
| API Server | Worker nodes (VMs) |
| etcd (cluster database) | Your applications (Pods, Deployments) |
| Scheduler | Node pool sizing & scaling |
| Controller Manager | Application security |
| Control plane upgrades | Ingress, networking choices |
| Control plane high availability | Monitoring & alerting |

**You only pay for the worker nodes (VMs) and associated resources (disks, networking). The control plane is FREE!**

### When to Use AKS?

✅ You need to run containerized microservices at scale
✅ You want Kubernetes but don't want to manage the control plane
✅ You're already in the Azure ecosystem
✅ You need auto-scaling, self-healing, rolling updates
✅ You want to integrate with Azure DevOps, Azure Monitor, ACR, etc.

❌ You have a simple app (use Azure App Service instead)
❌ You just need to run a single container (use Azure Container Instances)
❌ You're running serverless workloads (use Azure Functions)

### Key Architecture Points

- **Control Plane** is fully managed and FREE. You can't SSH into it
- **Worker Nodes** are Azure VMs (you pick the VM size). You CAN SSH into them (but usually shouldn't)
- **Node Pools** are groups of worker nodes with the same VM size and configuration
- **kubelet** runs on each node, talks to the API server
- **containerd** is the container runtime (AKS uses containerd, not Docker)
- **kube-proxy** handles networking rules on each node

## 4. Creating Your First AKS Cluster

### Method 1: Azure CLI (Recommended for Learning)

```bash
# Step 1: Create a resource group
az group create \
  --name aks-resource-group \
  --location eastus

# Step 2: Create the AKS cluster
az aks create \
  --resource-group aks-resource-group \
  --name my-aks-cluster \
  --node-count 2 \
  --node-vm-size Standard_B2s \
  --generate-ssh-keys \
  --network-plugin azure \
  --enable-managed-identity \
  --enable-addons monitoring \
  --output table
```

**Let's break down every flag:**

| Flag | What It Does |
|------|-------------|
| `--resource-group` | Which resource group to put the cluster in |
| `--name` | Name of your AKS cluster |
| `--node-count 2` | Start with 2 worker nodes |
| `--node-vm-size Standard_B2s` | VM size for nodes (2 vCPU, 4 GB RAM — good for dev) |
| `--generate-ssh-keys` | Creates SSH keys for node access (stored in `~/.ssh/`) |
| `--network-plugin azure` | Use Azure CNI networking (pods get VNet IPs) |
| `--enable-managed-identity` | Use managed identity (no service principal secrets!) |
| `--enable-addons monitoring` | Enable Azure Monitor / Container Insights |

**This takes 5-10 minutes.** Azure is creating VMs, networking, the control plane, installing Kubernetes, and more.

### Common VM Sizes for AKS

| VM Size | vCPU | RAM | Best For |
|---------|------|-----|----------|
| `Standard_B2s` | 2 | 4 GB | Dev/test, learning |
| `Standard_DS2_v2` | 2 | 7 GB | Small workloads |
| `Standard_D4s_v3` | 4 | 16 GB | General workloads |
| `Standard_D8s_v3` | 8 | 32 GB | Medium workloads |
| `Standard_D16s_v3` | 16 | 64 GB | Large workloads |
| `Standard_E4s_v3` | 4 | 32 GB | Memory-intensive (Redis, Elasticsearch) |
| `Standard_F4s_v2` | 4 | 8 GB | CPU-intensive (CI/CD, batch processing) |
| `Standard_NC6s_v3` | 6 | 112 GB | GPU workloads (ML/AI) |

### Method 2: Azure Portal (GUI)

1. Go to Azure Portal → Search **"Kubernetes services"**
2. Click **"+ Create"** → **"Kubernetes cluster"**
3. Fill in:
   - **Subscription & Resource Group**
   - **Cluster name**: `my-aks-cluster`
   - **Region**: `East US`
   - **Kubernetes version**: Latest stable
   - **Node size**: `Standard_B2s`
   - **Node count**: `2`
4. In **Networking** tab:
   - Network configuration: **Azure CNI**
5. In **Integrations** tab:
   - Container Registry: Link your ACR (optional)
6. Review + Create

## 6. Node Pools — The Brains Behind the Operation

### What is a Node Pool?

A **Node Pool** is a group of worker nodes with the **same VM size** and **same configuration**. AKS always has at least one node pool (the "system" pool).

### Why Multiple Node Pools?

| Scenario | Solution |
|----------|----------|
| Different workload types | Small nodes for lightweight apps, large nodes for databases |
| Cost optimization | Spot (preemptible) nodes for batch jobs |
| GPU workloads | GPU node pool for ML training |
| OS requirements | Linux pool + Windows pool |
| Isolation | Dedicated pool for critical apps |

### System vs User Node Pools

| Feature | System Node Pool | User Node Pool |
|---------|-----------------|---------------|
| **Purpose** | Run system pods (CoreDNS, kube-proxy, etc.) | Run your application pods |
| **Required?** | Yes (at least 1) | Optional |
| **Can delete?** | No (at least 1 must exist) | Yes |
| **Taints** | `CriticalAddonsOnly` (optional) | Any custom taints |

### Creating & Managing Node Pools

```bash
# Add a new node pool (larger VMs for heavy workloads)
az aks nodepool add \
  --resource-group aks-resource-group \
  --cluster-name my-aks-cluster \
  --name heavypool \
  --node-count 3 \
  --node-vm-size Standard_D4s_v3 \
  --labels workload=heavy \
  --node-taints dedicated=heavy:NoSchedule

# Add a spot (preemptible) node pool — MUCH cheaper, but can be evicted
az aks nodepool add \
  --resource-group aks-resource-group \
  --cluster-name my-aks-cluster \
  --name spotpool \
  --node-count 2 \
  --node-vm-size Standard_D4s_v3 \
  --priority Spot \
  --spot-max-price -1 \
  --eviction-policy Delete \
  --labels workload=batch

# Add a Windows node pool (for Windows containers)
az aks nodepool add \
  --resource-group aks-resource-group \
  --cluster-name my-aks-cluster \
  --name winpool \
  --node-count 2 \
  --node-vm-size Standard_D4s_v3 \
  --os-type Windows

# Scale a node pool
az aks nodepool scale \
  --resource-group aks-resource-group \
  --cluster-name my-aks-cluster \
  --name heavypool \
  --node-count 5

# List all node pools
az aks nodepool list \
  --resource-group aks-resource-group \
  --cluster-name my-aks-cluster \
  --output table

# Delete a node pool
az aks nodepool delete \
  --resource-group aks-resource-group \
  --cluster-name my-aks-cluster \
  --name spotpool
```

### Targeting Specific Node Pools

Use **nodeSelector** or **affinity** in your Pod spec to schedule pods on specific node pools:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: heavy-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: heavy-app
  template:
    metadata:
      labels:
        app: heavy-app
    spec:
      # Schedule on nodes with label "workload=heavy"
      nodeSelector:
        workload: heavy

      # Tolerate the taint on heavypool
      tolerations:
      - key: "dedicated"
        operator: "Equal"
        value: "heavy"
        effect: "NoSchedule"

      containers:
      - name: heavy-app
        image: myapp:latest
        resources:
          requests:
            cpu: "2"
            memory: "4Gi"
          limits:
            cpu: "4"
            memory: "8Gi"
```

## 7. Deploying Applications on AKS

### Step-by-Step: Deploy a Simple Web App

#### 1. Create a Deployment

```yaml
# nginx-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-web
  labels:
    app: nginx-web
spec:
  replicas: 3                    # Run 3 copies of the app
  selector:
    matchLabels:
      app: nginx-web
  template:
    metadata:
      labels:
        app: nginx-web
    spec:
      containers:
      - name: nginx
        image: nginx:1.25        # Container image
        ports:
        - containerPort: 80      # Container listens on port 80
        resources:
          requests:              # Minimum resources guaranteed
            cpu: "100m"          # 0.1 CPU core
            memory: "128Mi"      # 128 MB RAM
          limits:                # Maximum resources allowed
            cpu: "250m"
            memory: "256Mi"
        livenessProbe:           # Restart container if this fails
          httpGet:
            path: /
            port: 80
          initialDelaySeconds: 10
          periodSeconds: 10
        readinessProbe:          # Don't send traffic until this passes
          httpGet:
            path: /
            port: 80
          initialDelaySeconds: 5
          periodSeconds: 5
```

#### 2. Create a Service (Expose the Deployment)

```yaml
# nginx-service.yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx-web-service
spec:
  type: LoadBalancer            # Creates an Azure Load Balancer with public IP
  selector:
    app: nginx-web              # Route traffic to pods with this label
  ports:
  - protocol: TCP
    port: 80                    # Service port (what clients connect to)
    targetPort: 80              # Container port (where traffic is forwarded)
```

#### 3. Deploy!

```bash
# Apply the deployment
kubectl apply -f nginx-deployment.yaml

# Apply the service
kubectl apply -f nginx-service.yaml

# Watch the deployment roll out
kubectl rollout status deployment/nginx-web

# Check pods
kubectl get pods -l app=nginx-web

# Output:
# NAME                         READY   STATUS    RESTARTS   AGE
# nginx-web-6d4f5c8b-abc12    1/1     Running   0          30s
# nginx-web-6d4f5c8b-def34    1/1     Running   0          30s
# nginx-web-6d4f5c8b-ghi56    1/1     Running   0          30s

# Check the service (wait for EXTERNAL-IP)
kubectl get service nginx-web-service --watch

# Output:
# NAME                TYPE           CLUSTER-IP    EXTERNAL-IP    PORT(S)        AGE
# nginx-web-service   LoadBalancer   10.0.45.123   20.85.67.89    80:31234/TCP   2m

# Now open http://20.85.67.89 in your browser — you'll see the Nginx welcome page!
```

#### 4. Using Ingress Instead of LoadBalancer

Instead of creating a separate Load Balancer for each service (expensive!), use an **Ingress Controller**:

```bash
# Install NGINX Ingress Controller using Helm
helm repo add ingress-nginx https://kubernetes.github.io/ingress-nginx
helm repo update

helm install ingress-nginx ingress-nginx/ingress-nginx \
  --namespace ingress-nginx \
  --create-namespace \
  --set controller.service.annotations."service\.beta\.kubernetes\.io/azure-load-balancer-health-probe-request-path"=/healthz
```

Then create an Ingress resource:

```yaml
# nginx-ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: nginx-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  rules:
  - host: myapp.example.com          # Route based on hostname
    http:
      paths:
      - path: /                       # All paths
        pathType: Prefix
        backend:
          service:
            name: nginx-web-service   # Forward to this service
            port:
              number: 80
  - host: api.example.com            # Different hostname
    http:
      paths:
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: api-service
            port:
              number: 8080
```

**Advantage:** One Load Balancer (Ingress Controller) handles routing for ALL your services. Much cheaper and more manageable!

## 8. AKS Networking

### Network Models: kubenet vs Azure CNI

AKS supports two network plugins:

| Feature | kubenet | Azure CNI |
|---------|---------|-----------|
| **Pod IPs** | Pod gets IP from a separate range (NAT'ed) | Pod gets a real IP from the VNet subnet |
| **Pod-to-VNet communication** | Through NAT | Direct (pods are first-class VNet citizens) |
| **Performance** | Slightly lower (NAT overhead) | Better (no NAT) |
| **IP consumption** | Lower (pods share node IPs) | Higher (each pod needs a VNet IP) |
| **Network Policies** | Calico only | Azure + Calico |
| **Max pods per node** | 110 | 30 (default, configurable to 250) |
| **Recommended for** | Small clusters, dev/test | Production, enterprise |

### Azure CNI vs Azure CNI Overlay

There's also a newer option — **Azure CNI Overlay**:

| Feature | Azure CNI | Azure CNI Overlay |
|---------|-----------|-------------------|
| **Pod IPs from** | VNet subnet | Separate overlay CIDR |
| **IP consumption** | High (needs large subnets) | Low (overlay network) |
| **Max pods per node** | 250 | 250 |
| **Peering/VPN access to pods** | Direct | Via NAT |
| **Best for** | When pods need VNet IPs | Large clusters (save IPs) |

### Choosing the Right Network Model

```bash
# Create cluster with Azure CNI (recommended for production)
az aks create \
  --resource-group aks-resource-group \
  --name my-aks-cluster \
  --network-plugin azure \
  --vnet-subnet-id /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.Network/virtualNetworks/<vnet>/subnets/<subnet> \
  --service-cidr 10.2.0.0/16 \
  --dns-service-ip 10.2.0.10

# Create cluster with Azure CNI Overlay
az aks create \
  --resource-group aks-resource-group \
  --name my-aks-cluster \
  --network-plugin azure \
  --network-plugin-mode overlay \
  --pod-cidr 192.168.0.0/16
```

### Network Policies in AKS

Network Policies control pod-to-pod traffic:

```bash
# Enable network policies during cluster creation
az aks create \
  --resource-group aks-resource-group \
  --name my-aks-cluster \
  --network-plugin azure \
  --network-policy azure        # or "calico"
```

Example Network Policy:

```yaml
# Only allow traffic to backend pods from frontend pods
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: backend-allow-frontend
  namespace: default
spec:
  podSelector:
    matchLabels:
      app: backend            # Apply to backend pods
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend       # Only allow from frontend pods
    ports:
    - protocol: TCP
      port: 8080
```

## 9. AKS Storage

### Storage Options in AKS

| Storage Type | Backed By | Access Mode | Best For |
|-------------|-----------|-------------|----------|
| **Azure Disk** | Managed Disk (SSD/HDD) | ReadWriteOnce (single node) | Databases, stateful apps |
| **Azure Files** | SMB/NFS file share | ReadWriteMany (multiple nodes) | Shared config, CMS, legacy apps |
| **Azure Blob** | Blob Storage (via CSI) | ReadWriteMany | Large datasets, ML training data |

### Using Azure Disk (Default for StatefulSets)

```yaml
# pvc-azure-disk.yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-disk-pvc
spec:
  accessModes:
    - ReadWriteOnce             # Disk can be mounted to one node
  storageClassName: managed-csi  # Built-in AKS StorageClass for Azure Disk
  resources:
    requests:
      storage: 10Gi             # 10 GB disk
---
# Pod using the PVC
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
  - name: app
    image: nginx
    volumeMounts:
    - mountPath: /data           # Mount the disk here inside the container
      name: my-volume
  volumes:
  - name: my-volume
    persistentVolumeClaim:
      claimName: my-disk-pvc     # Reference the PVC
```

### Built-in Storage Classes in AKS

```bash
# List available storage classes
kubectl get storageclass

# Output:
# NAME                    PROVISIONER          RECLAIMPOLICY   VOLUMEBINDINGMODE
# managed-csi (default)   disk.csi.azure.com   Delete          WaitForFirstConsumer
# managed-csi-premium     disk.csi.azure.com   Delete          WaitForFirstConsumer
# azurefile-csi           file.csi.azure.com   Delete          Immediate
# azurefile-csi-premium   file.csi.azure.com   Delete          Immediate
```

| Storage Class | Disk Type | IOPS | Best For |
|---------------|-----------|------|----------|
| `managed-csi` | Standard SSD | Lower | Dev/test, low-traffic workloads |
| `managed-csi-premium` | Premium SSD | Higher | Production databases |
| `azurefile-csi` | Standard Files | Lower | Shared storage (SMB) |
| `azurefile-csi-premium` | Premium Files | Higher | High-perf shared storage |

## 10. AKS Identity & Access Control (RBAC)

### Authentication: Who Are You?

AKS integrates with **Microsoft Entra ID** (formerly Azure AD) for authentication:

```bash
# Create cluster with Entra ID integration
az aks create \
  --resource-group aks-resource-group \
  --name my-aks-cluster \
  --enable-aad \
  --aad-admin-group-object-ids <group-object-id> \
  --enable-azure-rbac
```

### Authorization: What Can You Do?

Two options:

| Mode | Description |
|------|-------------|
| **Kubernetes RBAC** | Traditional K8s Roles and RoleBindings |
| **Azure RBAC for K8s** | Use Azure role assignments (recommended) |

### Azure RBAC Built-in Roles

| Role | What It Can Do |
|------|---------------|
| `Azure Kubernetes Service Cluster Admin` | Full access to the cluster |
| `Azure Kubernetes Service Cluster User` | Read kubeconfig, limited access |
| `Azure Kubernetes Service RBAC Admin` | Manage RBAC within the cluster |
| `Azure Kubernetes Service RBAC Writer` | Read/write most resources |
| `Azure Kubernetes Service RBAC Reader` | Read-only access |

### Managed Identity

AKS uses **managed identities** to interact with Azure resources (like pulling images from ACR, creating load balancers, managing disks):

```bash
# The cluster has TWO managed identities:

# 1. Cluster identity — used by the control plane
az aks show --resource-group aks-resource-group --name my-aks-cluster \
  --query identity

# 2. Kubelet identity — used by worker nodes (to pull images, etc.)
az aks show --resource-group aks-resource-group --name my-aks-cluster \
  --query identityProfile.kubeletidentity
```

### Workload Identity (Pod-Level Identity)

For pods that need to access Azure services (like Key Vault, Storage):

```bash
# Enable workload identity on the cluster
az aks update \
  --resource-group aks-resource-group \
  --name my-aks-cluster \
  --enable-oidc-issuer \
  --enable-workload-identity
```

Then create a federated credential and use it in your pod:

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: my-app-sa
  annotations:
    azure.workload.identity/client-id: "<your-managed-identity-client-id>"
---
apiVersion: v1
kind: Pod
metadata:
  name: my-app
  labels:
    azure.workload.identity/use: "true"
spec:
  serviceAccountName: my-app-sa
  containers:
  - name: app
    image: myapp:latest
```

No passwords or secrets needed! The pod automatically gets an Azure token.

## 11. Azure Container Registry (ACR) Integration

### Why ACR?

**Azure Container Registry (ACR)** is Azure's private Docker registry. Integrating it with AKS means your cluster can pull images securely without storing Docker passwords.

### Creating and Linking ACR to AKS

```bash
# Create an ACR
az acr create \
  --resource-group aks-resource-group \
  --name myaksregistry \
  --sku Basic

# Attach ACR to AKS (grants pull access automatically)
az aks update \
  --resource-group aks-resource-group \
  --name my-aks-cluster \
  --attach-acr myaksregistry
```

That's it! Now your AKS cluster can pull images from `myaksregistry.azurecr.io` without any secrets.

### Pushing Images to ACR

```bash
# Login to ACR
az acr login --name myaksregistry

# Tag your image
docker tag my-app:latest myaksregistry.azurecr.io/my-app:v1.0

# Push to ACR
docker push myaksregistry.azurecr.io/my-app:v1.0

# Use in your Kubernetes deployment
# image: myaksregistry.azurecr.io/my-app:v1.0
```

### Build Images Directly in ACR (No Local Docker Needed!)

```bash
# ACR Tasks — build in the cloud
az acr build \
  --registry myaksregistry \
  --image my-app:v1.0 \
  --file Dockerfile \
  .
```

This sends your code to ACR, builds the image in the cloud, and stores it — no Docker Desktop needed on your machine!

## 12. AKS Scaling

### Three Types of Scaling in AKS

```
┌─────────────────────────────────────────────────────┐
│                 SCALING IN AKS                       │
│                                                     │
│  1. HPA (Horizontal Pod Autoscaler)                 │
│     Scale PODS based on CPU/memory/custom metrics   │
│     "I need more copies of my app"                  │
│                                                     │
│  2. Cluster Autoscaler                              │
│     Scale NODES based on pending pods               │
│     "I need more VMs to run my pods"                │
│                                                     │
│  3. KEDA (Event-Driven Autoscaler)                  │
│     Scale PODS based on events (queues, etc.)       │
│     "Scale based on Kafka/SQS queue length"         │
└─────────────────────────────────────────────────────┘
```

### HPA (Horizontal Pod Autoscaler)

```bash
# Create an HPA
kubectl autoscale deployment nginx-web \
  --cpu-percent=50 \
  --min=2 \
  --max=10

# Or with YAML:
```

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: nginx-web-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: nginx-web
  minReplicas: 2               # Minimum pods
  maxReplicas: 10              # Maximum pods
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 50   # Scale up when CPU > 50%
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 70   # Scale up when memory > 70%
```

### Cluster Autoscaler

```bash
# Enable cluster autoscaler on the default node pool
az aks update \
  --resource-group aks-resource-group \
  --name my-aks-cluster \
  --enable-cluster-autoscaler \
  --min-count 2 \
  --max-count 10

# Or on a specific node pool
az aks nodepool update \
  --resource-group aks-resource-group \
  --cluster-name my-aks-cluster \
  --name heavypool \
  --enable-cluster-autoscaler \
  --min-count 1 \
  --max-count 5
```

**How they work together:**
1. Traffic increases → HPA creates more pods
2. Not enough nodes to schedule pods → Cluster Autoscaler adds more nodes
3. Traffic decreases → HPA removes pods
4. Nodes become empty → Cluster Autoscaler removes nodes (saves cost!)

### KEDA

```bash
# Install KEDA add-on
az aks update \
  --resource-group aks-resource-group \
  --name my-aks-cluster \
  --enable-keda
```

## 13. AKS Monitoring & Logging

### Container Insights (Built-in)

```bash

  --resource-group aks-resource-group \
  --name my-aks-cluster \
  --addons monitoring \
  --workspace-resource-id /subscriptions/<sub-id>/resourceGroups/<rg>/providers/Microsoft.OperationalInsights/workspaces/<workspace-name>
```

This gives you:
- **Node metrics** — CPU, memory, disk for each node
- **Pod metrics** — CPU, memory, restarts for each pod
- **Container logs** — stdout/stderr from all containers
- **Kubernetes events** — Deployments, scaling, failures
- **Pre-built dashboards** in Azure Portal

### Querying Logs with KQL (Kusto Query Language)

Go to Azure Portal → AKS Cluster → **Logs** and run queries:

```kql
// Get all container logs from the last hour
ContainerLogV2
| where TimeGenerated > ago(1h)
| where ContainerName == "nginx"
| project TimeGenerated, LogMessage, ContainerName, PodName
| order by TimeGenerated desc
| take 100

// Find pods with high CPU usage
KubePodInventory
| where TimeGenerated > ago(1h)
| join (
    Perf
    | where ObjectName == "K8SContainer"
    | where CounterName == "cpuUsageNanoCores"
) on Computer
| summarize AvgCPU = avg(CounterValue) by PodName, Namespace
| order by AvgCPU desc

// Find CrashLoopBackOff pods
KubeEvents
| where TimeGenerated > ago(24h)
| where Reason == "BackOff"
| summarize Count = count() by Name, Namespace, Reason
| order by Count desc
```

## 14. AKS Upgrades & Maintenance

### Check Available Versions

```bash
# See available Kubernetes versions for your region
az aks get-versions \
  --location eastus \
  --output table

# Check current version
az aks show \
  --resource-group aks-resource-group \
  --name my-aks-cluster \
  --query kubernetesVersion
```

### Upgrade Strategy

AKS upgrades happen in two phases:

1. **Control Plane Upgrade** — Azure handles this (no downtime for your apps)
2. **Node Pool Upgrade** — Nodes are cordoned, drained, and replaced one by one

```bash
# Upgrade the entire cluster (control plane + all node pools)
az aks upgrade \
  --resource-group aks-resource-group \
  --name my-aks-cluster \
  --kubernetes-version 1.29.0

# Upgrade control plane only (no node changes)
az aks upgrade \
  --resource-group aks-resource-group \
  --name my-aks-cluster \
  --kubernetes-version 1.29.0 \
  --control-plane-only

# Upgrade a specific node pool
az aks nodepool upgrade \
  --resource-group aks-resource-group \
  --cluster-name my-aks-cluster \
  --name nodepool1 \
  --kubernetes-version 1.29.0
```

### Auto-Upgrade (Recommended)

```bash
# Set up auto-upgrade channel
az aks update \
  --resource-group aks-resource-group \
  --name my-aks-cluster \
  --auto-upgrade-channel stable
```

| Channel | What Gets Upgraded |
|---------|-------------------|
| `none` | No auto-upgrades |
| `patch` | Auto-upgrade to latest patch (e.g., 1.28.3 → 1.28.5) |
| `stable` | Auto-upgrade to latest stable minor version |
| `rapid` | Auto-upgrade to latest supported version |
| `node-image` | Auto-upgrade node OS image only |

### Maintenance Windows

```bash
# Set a maintenance window (upgrades only happen during this time)
az aks maintenancewindow add \
  --resource-group aks-resource-group \
  --cluster-name my-aks-cluster \
  --name default \
  --schedule-type Weekly \
  --day-of-week Saturday \
  --start-time 02:00 \
  --duration 4
```

## 15. AKS Security Best Practices

### 1. Use Private Clusters

```bash
# Create a private cluster (API server not accessible from internet)
az aks create \
  --resource-group aks-resource-group \
  --name private-aks \
  --enable-private-cluster \
  --node-count 2
```

With a private cluster, the API server has a private IP. You access it via VPN, ExpressRoute, or a jump box in the same VNet.

### 2. Use Managed Identities (Not Service Principals)

```bash
# ✅ Good — Managed Identity (no secrets to rotate)
az aks create --enable-managed-identity ...

# ❌ Bad — Service Principal (secrets expire, need rotation)
az aks create --service-principal ... --client-secret ...
```

### 3. Enable Defender for Containers

```bash
az security pricing create \
  --name Containers \
  --tier Standard
```

### 4. Limit API Server Access

```bash
# Only allow specific IPs to access the API server
az aks update \
  --resource-group aks-resource-group \
  --name my-aks-cluster \
  --api-server-authorized-ip-ranges "203.0.113.0/24,198.51.100.0/24"
```

### 5. Use Pod Security Standards

```yaml
# Enforce restricted pod security standard on a namespace
apiVersion: v1
kind: Namespace
metadata:
  name: production
  labels:
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/warn: restricted
    pod-security.kubernetes.io/audit: restricted
```

### Security Checklist

| Item | How |
|------|-----|
| ✅ Private cluster | `--enable-private-cluster` |
| ✅ Managed identity | `--enable-managed-identity` |
| ✅ Entra ID integration | `--enable-aad` |
| ✅ Azure RBAC | `--enable-azure-rbac` |
| ✅ Network policies | `--network-policy azure` |
| ✅ API server IP restrictions | `--api-server-authorized-ip-ranges` |
| ✅ Workload identity | `--enable-workload-identity` |
| ✅ Image scanning | ACR + Defender for Containers |
| ✅ Pod Security Standards | Namespace labels |
| ✅ Secrets in Key Vault | CSI Secrets Store Driver |

## 18. Common Interview Questions

**Q: What is AKS and how is it different from running Kubernetes yourself?**
AKS is Azure's managed Kubernetes. Microsoft manages the control plane (API server, etcd, scheduler) for free. You only manage and pay for worker nodes. Self-managed K8s means you handle everything, including HA, upgrades, and certificate rotation.

**Q: How does AKS networking work? What's the difference between kubenet and Azure CNI?**
kubenet gives pods IPs from a separate overlay network (through NAT). Azure CNI gives pods real IPs from the VNet subnet, making them first-class VNet citizens. Azure CNI is recommended for production.

**Q: How does scaling work in AKS?**
Three mechanisms: HPA scales pods based on CPU/memory, Cluster Autoscaler scales nodes when pods can't be scheduled, and KEDA scales based on external events like queue depth.

**Q: What is a node pool and why would you use multiple?**
A node pool is a group of nodes with the same VM size and config. Use multiple pools for: different VM sizes (small for web, large for DB), spot instances for batch jobs, GPU nodes for ML, or Windows + Linux nodes.

**Q: How do you secure an AKS cluster?**
Private cluster, Entra ID integration, Azure RBAC, Network Policies, API server IP restrictions, Workload Identity, Defender for Containers, Pod Security Standards, and image scanning.

**Q: How do you upgrade AKS?**
Use `az aks upgrade`. Control plane upgrades first, then nodes are cordoned, drained, and replaced one by one. Enable auto-upgrade channels for automated upgrades.
