## 1. What is Azure DevOps? — The Big Picture

### Azure DevOps Has 5 Main Services

Think of Azure DevOps as a **toolbox** with five tools inside:

```
┌─────────────────────────────────────────────────────────────────┐
│                       AZURE DEVOPS                               │
│                                                                 │
│  ┌───────────┐  ┌───────────┐  ┌───────────┐                   │
│  │  BOARDS   │  │   REPOS   │  │ PIPELINES │                   │
│  │           │  │           │  │           │                   │
│  │ Plan work │  │Store code │  │ Build &   │                   │
│  │ Track     │  │ Git repos │  │ Deploy    │                   │
│  │ Sprints   │  │ PRs       │  │ CI/CD     │                   │
│  └───────────┘  └───────────┘  └───────────┘                   │
│                                                                 │
│  ┌───────────┐  ┌───────────┐                                   │
│  │TEST PLANS │  │ ARTIFACTS │                                   │
│  │           │  │           │                                   │
│  │ Manual &  │  │ Package   │                                   │
│  │ Auto tests│  │ feeds     │                                   │
│  │           │  │ (npm,NuGet│                                   │
│  └───────────┘  └───────────┘                                   │
└─────────────────────────────────────────────────────────────────┘
```

| Service | What It Does | Similar To |
|---------|-------------|-----------|
| **Azure Boards** | Plan & track work (tasks, bugs, sprints) | Jira, Trello |
| **Azure Repos** | Store code in Git repositories | GitHub, GitLab |
| **Azure Pipelines** | Build, test, deploy (CI/CD) | Jenkins, GitHub Actions |
| **Azure Test Plans** | Manual & automated testing | TestRail |
| **Azure Artifacts** | Host packages (npm, NuGet, Maven, pip) | Nexus, Artifactory |

### Why Azure DevOps?

- **All-in-one** — Everything in one place (planning, code, CI/CD, testing, packages)
- **Deep Azure integration** — Deploy to Azure services seamlessly
- **Free tier** — Free for up to 5 users (with some limits)
- **Enterprise-grade** — RBAC, audit logs, compliance
- **Flexible** — Use all services together, or pick and choose (you can use Azure Pipelines with GitHub repos!)

## 2. Azure DevOps Services vs Azure DevOps Server

| Feature | Azure DevOps Services (Cloud) | Azure DevOps Server (On-Premises) |
|---------|------------------------------|----------------------------------|
| **Hosted by** | Microsoft (SaaS) | You (on your servers) |
| **URL** | `dev.azure.com/your-org` | `your-server/tfs` |
| **Updates** | Automatic (always latest) | Manual upgrades |
| **Scaling** | Azure handles it | You manage hardware |
| **Best for** | Most teams | Government, air-gapped networks |
| **Pricing** | Free for 5 users, then per-user | License-based |

**For 99% of teams, use Azure DevOps Services (cloud).**

## 3. Setting Up Azure DevOps

### Step 1: Create an Organization

1. Go to [dev.azure.com](https://dev.azure.com)
2. Sign in with your Microsoft account
3. Click **"New Organization"**
4. Give it a name (e.g., `my-company`)
5. Choose a region (closest to your team)

Your URL will be: `https://dev.azure.com/my-company`

### Step 2: Create a Project

1. Click **"+ New Project"**
2. Fill in:
   - **Name**: `my-web-app`
   - **Visibility**: Private (or Public for open source)
   - **Version control**: Git
   - **Work item process**: Agile (or Scrum/CMMI)
3. Click **Create**

### Step 3: Invite Team Members

1. Go to **Organization Settings** → **Users**
2. Click **Add users**
3. Enter email addresses and assign access level:
   - **Basic** — Full access (paid after 5 users)
   - **Stakeholder** — Free, limited (can view boards, but not edit code)
   - **Visual Studio Subscriber** — Included with VS subscription

### Using Azure CLI (az devops)

```bash
# Install the Azure DevOps CLI extension
az extension add --name azure-devops

# Login and set defaults
az devops configure --defaults organization=https://dev.azure.com/my-company project=my-web-app

# Create a new project
az devops project create \
  --name my-web-app \
  --organization https://dev.azure.com/my-company \
  --visibility private \
  --process Agile
```

## 4. Azure Repos — Source Code Management

### What is Azure Repos?

Azure Repos is a **Git hosting service** (like GitHub or GitLab). Each project can have multiple Git repositories.

### Creating a Repository

```bash
# Create a new repo in your project
az repos create --name backend-api --project my-web-app

# List all repos
az repos list --project my-web-app --output table
```

### Cloning and Working with a Repo

```bash
# Clone the repo
git clone https://dev.azure.com/my-company/my-web-app/_git/backend-api

# Navigate into it
cd backend-api

# Create a file
echo "# Backend API" > README.md

# Commit and push
git add .
git commit -m "Initial commit"
git push origin main
```

### Authentication Methods

| Method | Best For | How |
|--------|----------|-----|
| **HTTPS + PAT** | Quick setup, CI/CD | Generate Personal Access Token in Azure DevOps |
| **SSH Keys** | Developers (no password prompts) | Add SSH public key to Azure DevOps profile |
| **Git Credential Manager** | Windows/Mac developers | Automatically handles auth (recommended) |

### Generating a Personal Access Token (PAT)

1. Go to Azure DevOps → Click your **profile icon** (top right)
2. Click **Personal Access Tokens**
3. Click **+ New Token**
4. Give it a name, select scopes (Code: Read & Write), set expiration
5. Copy the token (you won't see it again!)

```bash
# Use PAT for HTTPS clone
git clone https://dev.azure.com/my-company/my-web-app/_git/backend-api
# Username: anything (e.g., your email)
# Password: paste your PAT
```

## 5. Azure Repos — Branching Strategies

### Recommended Branching Strategy for Azure DevOps

```
main (production)
  │
  ├── release/1.0     (stabilize for release)
  │     │
  │     └── bugfix/fix-login-error
  │
  ├── develop          (integration branch)
  │     │
  │     ├── feature/user-auth
  │     ├── feature/payment-gateway
  │     └── feature/dashboard-redesign
  │
  └── hotfix/security-patch  (emergency fixes → goes directly to main)
```

### Branch Policies (Protecting main Branch)

This is CRITICAL. Branch policies prevent direct pushes to `main` and enforce quality gates.

**Setting up via Azure Portal:**

1. Go to Repos → Branches → ⋮ (three dots) on `main` → **Branch policies**
2. Enable:
   - ✅ **Require a minimum number of reviewers** (at least 1)
   - ✅ **Check for linked work items** (connect commits to tasks)
   - ✅ **Check for comment resolution** (all PR comments must be resolved)
   - ✅ **Build validation** (PR must pass a build pipeline before merging)

**Setting up via CLI:**

```bash
# Add a build validation policy
az repos policy build create \
  --branch main \
  --repository-id <repo-id> \
  --build-definition-id <pipeline-id> \
  --enabled true \
  --blocking true \
  --display-name "Build Validation" \
  --queue-on-source-update-only false \
  --valid-duration 720

# Require minimum 2 reviewers
az repos policy approver-count create \
  --branch main \
  --repository-id <repo-id> \
  --minimum-approver-count 2 \
  --enabled true \
  --blocking true \
  --creator-vote-counts false \
  --allow-downvotes false \
  --reset-on-source-push true
```

### What Each Policy Does

| Policy | What It Enforces |
|--------|-----------------|
| **Minimum reviewers** | At least X people must approve the PR |
| **Build validation** | A CI pipeline must pass before merge is allowed |
| **Comment resolution** | All review comments must be resolved |
| **Linked work items** | PR must be linked to a work item (for traceability) |
| **Merge strategy** | Enforce squash merge, no fast-forward, etc. |
| **Status checks** | External status checks must pass |

## 6. Azure Repos — Pull Requests & Code Reviews

### Creating a Pull Request

```bash
# Create a branch and make changes
git checkout -b feature/add-login
# ... make code changes ...
git add .
git commit -m "Add login functionality"
git push origin feature/add-login

# Create a PR via CLI
az repos pr create \
  --source-branch feature/add-login \
  --target-branch main \
  --title "Add login functionality" \
  --description "This PR adds user login with OAuth2 support" \
  --reviewers "user1@company.com" "user2@company.com" \
  --work-items 1234
```

### PR Review Workflow

```
Developer                    Reviewer                    CI Pipeline
   │                            │                            │
   ├── Create PR ──────────────▶│                            │
   │                            │                            │
   │                            │◀── Pipeline triggered ────│
   │                            │                            │
   │                            │──── Run tests ────────────▶│
   │                            │                            │
   │                            │◀── Tests passed ──────────│
   │                            │                            │
   │◀── Comments & suggestions ─│                            │
   │                            │                            │
   ├── Fix comments, push ─────▶│                            │
   │                            │                            │
   │◀── Approved ──────────────│                            │
   │                            │                            │
   ├── Merge PR (squash) ──────▶│                            │
   │                            │                            │
   ▼                            ▼                            ▼
                     Code is now in main!
```

### PR Auto-Complete

You can set a PR to **auto-complete** — it will automatically merge once all policies are satisfied:

```bash
# Set auto-complete on a PR
az repos pr update \
  --id <pr-id> \
  --auto-complete true \
  --merge-commit-message "Merge feature/add-login (#1234)" \
  --delete-source-branch true \
  --squash true
```

## 7. Azure Pipelines — CI/CD Overview

### What is Azure Pipelines?

Azure Pipelines automates the process of **building**, **testing**, and **deploying** your code. Every time you push code, a pipeline can automatically:

1. **Build** the code (compile, install dependencies)
2. **Test** it (unit tests, integration tests)
3. **Package** it (Docker image, artifact)
4. **Deploy** it (to Azure, AWS, on-prem, Kubernetes)

### Classic Pipelines vs YAML Pipelines

| Feature | Classic (GUI) | YAML (Code) |
|---------|--------------|-------------|
| **How to create** | Click buttons in UI | Write YAML file in your repo |
| **Version control** | Not in your repo | In your repo (tracked in Git!) |
| **Code review** | Can't review pipeline changes | Pipeline changes go through PRs |
| **Recommended** | No (legacy) | **Yes — always use YAML** |
| **Complexity** | Simple to start | Slightly harder to start, much better long-term |

**Always use YAML Pipelines.** Classic pipelines are being phased out.

### Pipeline Structure

```yaml
# High-level structure of a YAML pipeline

trigger:             # WHEN to run (on push, PR, schedule)
  branches:
    include:
      - main

pool:                # WHERE to run (what agent/machine)
  vmImage: 'ubuntu-latest'

variables:           # VARIABLES used in the pipeline
  buildConfiguration: 'Release'

stages:              # HIGH-LEVEL STAGES (Build, Test, Deploy)
  - stage: Build
    jobs:            # JOBS within a stage (run in parallel by default)
      - job: BuildJob
        steps:       # STEPS within a job (run sequentially)
          - script: echo "Building..."
          - script: echo "Testing..."

  - stage: Deploy
    dependsOn: Build
    jobs:
      - job: DeployJob
        steps:
          - script: echo "Deploying..."
```

### Hierarchy

```
Pipeline
  └── Stage (e.g., "Build", "Deploy to Dev", "Deploy to Prod")
        └── Job (e.g., "Build-Linux", "Build-Windows")
              └── Step (e.g., "Install dependencies", "Run tests", "Build Docker image")
```

## 8. Azure Pipelines — YAML Pipeline Deep Dive

### Triggers — When Does the Pipeline Run?

```yaml
# 1. CI Trigger — Run on push to specific branches
trigger:
  branches:
    include:
      - main
      - release/*
    exclude:
      - experimental/*
  paths:
    include:
      - src/*            # Only trigger if files in src/ changed
    exclude:
      - docs/*           # Don't trigger for docs changes

# 2. PR Trigger — Run on pull requests
pr:
  branches:
    include:
      - main
  paths:
    include:
      - src/*

# 3. Scheduled Trigger — Run on a schedule
schedules:
  - cron: "0 2 * * *"        # Every day at 2 AM UTC
    displayName: "Nightly Build"
    branches:
      include:
        - main
    always: true              # Run even if no code changes

# 4. Manual Trigger — Disable automatic triggers
trigger: none
pr: none
# Must be triggered manually from the UI or API
```

### Pool — Where Does It Run?

```yaml
# Microsoft-hosted agents (managed by Microsoft — easiest)
pool:
  vmImage: 'ubuntu-latest'     # Linux (recommended)
  # vmImage: 'windows-latest'  # Windows
  # vmImage: 'macos-latest'    # macOS

# Self-hosted agent (your own machine/VM)
pool:
  name: 'My-Self-Hosted-Pool'
  demands:
    - docker                    # Agent must have Docker installed
```

### Available Microsoft-Hosted Agent Images

| Image | OS | Pre-installed |
|-------|------|--------------|
| `ubuntu-latest` | Ubuntu 22.04 | Docker, Node, Python, .NET, Java, Go, kubectl, Terraform |
| `ubuntu-20.04` | Ubuntu 20.04 | Same as above |
| `windows-latest` | Windows Server 2022 | Visual Studio, .NET, Node, Python |
| `macos-latest` | macOS 14 | Xcode, CocoaPods, Flutter |

### Steps — The Actual Work

```yaml
steps:
  # 1. Script step — run any command
  - script: |
      echo "Hello from the pipeline!"
      npm install
      npm test
    displayName: 'Install and Test'

  # 2. Bash step (same as script on Linux)
  - bash: |
      echo "This runs in bash"
    displayName: 'Bash Step'

  # 3. PowerShell step
  - powershell: |
      Write-Host "This runs in PowerShell"
    displayName: 'PowerShell Step'

  # 4. Task step — use a pre-built task from the marketplace
  - task: NodeTool@0
    inputs:
      versionSpec: '20.x'
    displayName: 'Install Node.js 20'

  # 5. Checkout step (happens automatically, but you can customize)
  - checkout: self
    fetchDepth: 1            # Shallow clone (faster)

  # 6. Download artifacts from another pipeline
  - download: current
    artifact: drop
```

## 9. Azure Pipelines — Build (CI) Pipeline

### Complete CI Pipeline for a Node.js App

```yaml
# azure-pipelines.yml

trigger:
  branches:
    include:
      - main
      - feature/*

pr:
  branches:
    include:
      - main

pool:
  vmImage: 'ubuntu-latest'

variables:
  nodeVersion: '20.x'

stages:
  - stage: Build
    displayName: 'Build & Test'
    jobs:
      - job: BuildAndTest
        displayName: 'Build and Test'
        steps:
          # Step 1: Install Node.js
          - task: NodeTool@0
            inputs:
              versionSpec: $(nodeVersion)
            displayName: 'Install Node.js $(nodeVersion)'

          # Step 2: Install dependencies
          - script: npm ci
            displayName: 'Install dependencies'
            # npm ci is faster than npm install — uses package-lock.json exactly

          # Step 3: Run linting
          - script: npm run lint
            displayName: 'Run ESLint'

          # Step 4: Run unit tests
          - script: npm test -- --coverage
            displayName: 'Run unit tests'

          # Step 5: Publish test results
          - task: PublishTestResults@2
            inputs:
              testResultsFormat: 'JUnit'
              testResultsFiles: '**/test-results.xml'
            displayName: 'Publish test results'
            condition: always()    # Publish even if tests fail

          # Step 6: Publish code coverage
          - task: PublishCodeCoverageResults@1
            inputs:
              codeCoverageTool: 'Cobertura'
              summaryFileLocation: '**/coverage/cobertura-coverage.xml'
            displayName: 'Publish code coverage'
            condition: always()

          # Step 7: Build the app
          - script: npm run build
            displayName: 'Build application'

          # Step 8: Publish build artifact
          - task: PublishBuildArtifacts@1
            inputs:
              PathtoPublish: 'dist'
              ArtifactName: 'app-build'
            displayName: 'Publish build artifact'
```

### CI Pipeline for a Docker Application

```yaml
# azure-pipelines.yml

trigger:
  branches:
    include:
      - main

pool:
  vmImage: 'ubuntu-latest'

variables:
  dockerRegistryServiceConnection: 'acr-connection'
  imageRepository: 'myapp'
  containerRegistry: 'myregistry.azurecr.io'
  dockerfilePath: 'Dockerfile'
  tag: '$(Build.BuildId)'       # Use build number as image tag

stages:
  - stage: Build
    displayName: 'Build & Push Docker Image'
    jobs:
      - job: Docker
        steps:
          # Step 1: Build the Docker image
          - task: Docker@2
            displayName: 'Build Docker image'
            inputs:
              containerRegistry: $(dockerRegistryServiceConnection)
              repository: $(imageRepository)
              command: 'build'
              Dockerfile: $(dockerfilePath)
              tags: |
                $(tag)
                latest

          # Step 2: Push to ACR
          - task: Docker@2
            displayName: 'Push to ACR'
            inputs:
              containerRegistry: $(dockerRegistryServiceConnection)
              repository: $(imageRepository)
              command: 'push'
              tags: |
                $(tag)
                latest
```

**Setting up the ACR Service Connection:**
1. Go to Project Settings → Service Connections
2. Click **New service connection** → **Docker Registry**
3. Select **Azure Container Registry**
4. Choose your subscription and ACR
5. Name it `acr-connection`

## 10. Azure Pipelines — Release (CD) Pipeline

### Deploying to Azure App Service

```yaml
stages:
  - stage: Build
    # ... (build steps from above)

  - stage: DeployDev
    displayName: 'Deploy to Dev'
    dependsOn: Build              # Only run after Build succeeds
    condition: succeeded()         # Only if previous stage passed
    jobs:
      - deployment: DeployDev      # "deployment" job type (not "job")
        displayName: 'Deploy to Dev Environment'
        environment: 'dev'         # Links to an Environment in Azure DevOps
        strategy:
          runOnce:                 # Deployment strategy
            deploy:
              steps:
                - task: AzureWebApp@1
                  inputs:
                    azureSubscription: 'my-azure-subscription'
                    appType: 'webAppLinux'
                    appName: 'my-web-app-dev'
                    package: '$(Pipeline.Workspace)/app-build/**'

  - stage: DeployProd
    displayName: 'Deploy to Production'
    dependsOn: DeployDev
    condition: and(succeeded(), eq(variables['Build.SourceBranch'], 'refs/heads/main'))
    jobs:
      - deployment: DeployProd
        displayName: 'Deploy to Production'
        environment: 'production'  # This environment can have approval gates!
        strategy:
          runOnce:
            deploy:
              steps:
                - task: AzureWebApp@1
                  inputs:
                    azureSubscription: 'my-azure-subscription'
                    appType: 'webAppLinux'
                    appName: 'my-web-app-prod'
                    package: '$(Pipeline.Workspace)/app-build/**'
```

### Deployment Strategies

```yaml
# 1. Run Once (simple — deploy everything at once)
strategy:
  runOnce:
    deploy:
      steps: [...]

# 2. Rolling (deploy to a few targets at a time)
strategy:
  rolling:
    maxParallel: 2           # Deploy to 2 VMs at a time
    deploy:
      steps: [...]

# 3. Canary (deploy to a small % first, then gradually more)
strategy:
  canary:
    increments: [10, 20]     # First 10%, then 20%, then rest
    deploy:
      steps: [...]
```

## 11. Azure Pipelines — Variables & Secrets

### Types of Variables

```yaml
# 1. Inline variables (in the YAML file)
variables:
  buildConfiguration: 'Release'
  nodeVersion: '20.x'

# 2. Variable groups (shared across pipelines)
variables:
  - group: 'my-variable-group'    # Defined in Pipelines → Library

# 3. Template variables (from another file)
variables:
  - template: vars/common.yml

# 4. Runtime variables (set during pipeline run)
variables:
  - name: environment
    value: ${{ parameters.env }}
```

### Predefined Variables (Built-in)

Azure Pipelines provides many built-in variables:

| Variable | Description | Example Value |
|----------|-------------|---------------|
| `$(Build.BuildId)` | Unique build number | `1234` |
| `$(Build.BuildNumber)` | Build number string | `20240115.1` |
| `$(Build.SourceBranch)` | Full branch name | `refs/heads/main` |
| `$(Build.SourceBranchName)` | Short branch name | `main` |
| `$(Build.Repository.Name)` | Repo name | `backend-api` |
| `$(System.DefaultWorkingDirectory)` | Working directory path | `/home/vsts/work/1/s` |
| `$(Pipeline.Workspace)` | Pipeline workspace path | `/home/vsts/work/1` |
| `$(Agent.OS)` | Agent OS | `Linux` |

### Secrets — Handling Sensitive Data

**NEVER put secrets in your YAML file.** Use one of these approaches:

#### Method 1: Variable Groups (UI)

1. Go to Pipelines → Library → **+ Variable group**
2. Add variables and click the **lock icon** 🔒 to make them secret
3. Reference in YAML:

```yaml
variables:
  - group: 'my-secrets'        # Contains DB_PASSWORD, API_KEY, etc.

steps:
  - script: |
      echo "Connecting to database..."
      # Secret variables are automatically masked in logs
      ./deploy.sh --password $(DB_PASSWORD)
    displayName: 'Deploy'
    env:
      DB_PASSWORD: $(DB_PASSWORD)     # Pass as environment variable
```

#### Method 2: Azure Key Vault Integration

```yaml
variables:
  - group: 'keyvault-secrets'    # Variable group linked to Key Vault

# In the Variable Group settings:
# 1. Toggle "Link secrets from an Azure key vault"
# 2. Select your Key Vault
# 3. Choose which secrets to include
# Secrets are automatically fetched at runtime!
```

#### Method 3: Pipeline UI Variables

1. Go to your pipeline → **Edit** → **Variables** (top right)
2. Add a variable, check **"Keep this value secret"**

**Important:** Secret variables are:
- ✅ Masked in logs (shown as `***`)
- ✅ Not passed to forked repo PRs (security)
- ❌ Cannot be used in `condition:` expressions
- ❌ Not available in pipeline YAML output variables by default

## 12. Azure Pipelines — Templates & Reusable Code

### Why Templates?

If you have 20 pipelines that all do similar things (install Node, run tests, build Docker), you'd be copying the same YAML everywhere. Templates let you write it once and reuse it.

### Step Template

```yaml
# templates/build-node.yml
parameters:
  - name: nodeVersion
    type: string
    default: '20.x'

steps:
  - task: NodeTool@0
    inputs:
      versionSpec: ${{ parameters.nodeVersion }}
    displayName: 'Install Node.js ${{ parameters.nodeVersion }}'

  - script: npm ci
    displayName: 'Install dependencies'

  - script: npm run lint
    displayName: 'Lint'

  - script: npm test
    displayName: 'Test'

  - script: npm run build
    displayName: 'Build'
```

Use it:

```yaml
# azure-pipelines.yml
trigger:
  - main

pool:
  vmImage: 'ubuntu-latest'

steps:
  - template: templates/build-node.yml
    parameters:
      nodeVersion: '20.x'
```

### Job Template

```yaml
# templates/docker-build-push.yml
parameters:
  - name: registry
    type: string
  - name: repository
    type: string
  - name: tag
    type: string
    default: '$(Build.BuildId)'

jobs:
  - job: DockerBuildPush
    displayName: 'Build & Push Docker Image'
    steps:
      - task: Docker@2
        inputs:
          containerRegistry: ${{ parameters.registry }}
          repository: ${{ parameters.repository }}
          command: 'buildAndPush'
          tags: |
            ${{ parameters.tag }}
            latest
```

Use it:

```yaml
# azure-pipelines.yml
stages:
  - stage: Build
    jobs:
      - template: templates/docker-build-push.yml
        parameters:
          registry: 'acr-connection'
          repository: 'myapp'
```

### Stage Template

```yaml
# templates/deploy-stage.yml
parameters:
  - name: environment
    type: string
  - name: appName
    type: string

stages:
  - stage: Deploy_${{ parameters.environment }}
    displayName: 'Deploy to ${{ parameters.environment }}'
    jobs:
      - deployment: Deploy
        environment: ${{ parameters.environment }}
        strategy:
          runOnce:
            deploy:
              steps:
                - task: AzureWebApp@1
                  inputs:
                    azureSubscription: 'my-subscription'
                    appName: ${{ parameters.appName }}
                    package: '$(Pipeline.Workspace)/drop/**'
```

Use it:

```yaml
# azure-pipelines.yml
stages:
  - stage: Build
    jobs: [...]

  - template: templates/deploy-stage.yml
    parameters:
      environment: 'dev'
      appName: 'my-app-dev'

  - template: templates/deploy-stage.yml
    parameters:
      environment: 'staging'
      appName: 'my-app-staging'

  - template: templates/deploy-stage.yml
    parameters:
      environment: 'production'
      appName: 'my-app-prod'
```

**Beautiful!** Three identical deployment stages, written once.

### Extends Template (Enforcing Standards)

This is powerful — you can create a template that FORCES all pipelines to follow your rules:

```yaml
# templates/pipeline-base.yml
parameters:
  - name: stages
    type: stageList
    default: []

stages:
  - stage: SecurityScan
    displayName: 'Security Scanning (Required)'
    jobs:
      - job: Scan
        steps:
          - script: |
              echo "Running security scan..."
              # trivy, Snyk, etc.
            displayName: 'Run security scan'

  - ${{ each stage in parameters.stages }}:
    - ${{ stage }}
```

Use it:

```yaml
# azure-pipelines.yml
extends:
  template: templates/pipeline-base.yml
  parameters:
    stages:
      - stage: Build
        jobs: [...]
      - stage: Deploy
        jobs: [...]
```

Now EVERY pipeline that uses this template will automatically get the Security Scan stage!

## 13. Azure Pipelines — Agents (Self-Hosted & Microsoft-Hosted)

### Microsoft-Hosted Agents

These are VMs managed by Microsoft. Each job gets a **fresh VM** that's destroyed after the job.

| Pros | Cons |
|------|------|
| Zero maintenance | Limited free minutes (1800 min/month for private) |
| Pre-installed tools | Can't customize the environment |
| Always up-to-date | Slower start (new VM each time) |
| No infrastructure to manage | No persistent state between jobs |

### Self-Hosted Agents

You install the agent on your own machine (VM, on-prem server, or container).

| Pros | Cons |
|------|------|
| Unlimited free minutes | You maintain the machine |
| Persistent cache (faster builds) | You install & update tools |
| Access to internal networks | Security responsibility |
| Full customization | Infrastructure cost |

### Setting Up a Self-Hosted Agent (Linux)

```bash
# 1. Download the agent
mkdir myagent && cd myagent
curl -O https://vstsagentpackage.azureedge.net/agent/3.230.0/vsts-agent-linux-x64-3.230.0.tar.gz
tar zxvf vsts-agent-linux-x64-3.230.0.tar.gz

# 2. Configure the agent
./config.sh
# Enter:
#   Server URL: https://dev.azure.com/my-company
#   PAT: (your personal access token)
#   Agent pool: Default (or custom pool name)
#   Agent name: my-build-agent

# 3. Run the agent
./run.sh              # Foreground (for testing)
# OR
sudo ./svc.sh install # Install as a systemd service
sudo ./svc.sh start   # Start the service
```

### Self-Hosted Agent in Docker

```dockerfile
FROM ubuntu:22.04

RUN apt-get update && apt-get install -y \
    curl \
    git \
    jq \
    docker.io \
    nodejs \
    npm \
    && rm -rf /var/lib/apt/lists/*

WORKDIR /agent
COPY start.sh .
RUN chmod +x start.sh

ENTRYPOINT ["./start.sh"]
```

### Using Self-Hosted Agent in Pipeline

```yaml
pool:
  name: 'My-Self-Hosted-Pool'    # Your agent pool name
  demands:
    - docker                       # Agent must have Docker
    - kubectl                      # Agent must have kubectl
```

## 14. Azure Pipelines — Environments & Approvals

### What is an Environment?

An **Environment** in Azure DevOps represents a deployment target (dev, staging, production). It provides:
- **Approval gates** — Require human approval before deploying
- **Deployment history** — See what was deployed and when
- **Resource tracking** — Link to actual Kubernetes clusters, VMs

### Creating Environments

1. Go to Pipelines → **Environments**
2. Click **New environment**
3. Name: `production`
4. Resource: None (or link a Kubernetes cluster)

### Adding Approval Gates

1. Open the `production` environment
2. Click ⋮ → **Approvals and checks**
3. Click **+ Add check** → **Approvals**
4. Add approvers (e.g., `team-lead@company.com`)
5. Set timeout (e.g., 72 hours)
6. Optional: Require minimum 2 approvers

Now when the pipeline reaches the `production` stage, it PAUSES and waits for approval:

```
Build ──▶ Deploy Dev ──▶ Deploy Staging ──▶ ⏸️ Waiting for approval ──▶ Deploy Prod
                                               ↑
                                     Approver gets email notification
                                     Must click "Approve" to continue
```

### Other Checks You Can Add

| Check | What It Does |
|-------|-------------|
| **Approvals** | Require human approval |
| **Business Hours** | Only deploy during work hours (e.g., Mon-Fri 9am-5pm) |
| **Branch Control** | Only allow deployments from `main` branch |
| **Template Required** | Pipeline must extend a specific template |
| **Exclusive Lock** | Only one deployment at a time |
| **Invoke Azure Function** | Call a custom function for validation |

### Using Environments in YAML

```yaml
stages:
  - stage: DeployProd
    jobs:
      - deployment: DeployToProduction
        environment: 'production'          # ← This triggers the approval!
        strategy:
          runOnce:
            deploy:
              steps:
                - script: echo "Deploying to prod..."
```

## 15. Azure Pipelines — Multi-Stage Pipeline (Full Example)

Here's a complete, production-ready pipeline that builds, tests, and deploys to dev → staging → production:

```yaml
# azure-pipelines.yml
# Complete multi-stage CI/CD pipeline

trigger:
  branches:
    include:
      - main
  paths:
    exclude:
      - docs/*
      - README.md

pr:
  branches:
    include:
      - main

variables:
  - name: dockerRegistry
    value: 'myregistry.azurecr.io'
  - name: imageName
    value: 'myapp'
  - name: tag
    value: '$(Build.BuildId)'
  - group: 'app-secrets'           # From variable group / Key Vault

# ============================================================
# STAGE 1: BUILD & TEST
# ============================================================
stages:
  - stage: Build
    displayName: '🔨 Build & Test'
    jobs:
      - job: BuildTestPush
        displayName: 'Build, Test & Push Image'
        pool:
          vmImage: 'ubuntu-latest'
        steps:
          # Checkout code
          - checkout: self
            fetchDepth: 1

          # Run unit tests
          - script: |
              npm ci
              npm test -- --coverage --ci
            displayName: '🧪 Run Unit Tests'

          # Publish test results
          - task: PublishTestResults@2
            inputs:
              testResultsFormat: 'JUnit'
              testResultsFiles: '**/junit.xml'
            condition: always()

          # Build & push Docker image
          - task: Docker@2
            displayName: '🐳 Build & Push Docker Image'
            inputs:
              containerRegistry: 'acr-service-connection'
              repository: $(imageName)
              command: 'buildAndPush'
              Dockerfile: 'Dockerfile'
              tags: |
                $(tag)
                latest

          # Publish Kubernetes manifests as artifact
          - publish: k8s/
            artifact: k8s-manifests

# ============================================================
# STAGE 2: DEPLOY TO DEV
# ============================================================
  - stage: DeployDev
    displayName: '🚀 Deploy to Dev'
    dependsOn: Build
    condition: succeeded()
    jobs:
      - deployment: DeployDev
        displayName: 'Deploy to Dev'
        pool:
          vmImage: 'ubuntu-latest'
        environment: 'dev'
        strategy:
          runOnce:
            deploy:
              steps:
                - download: current
                  artifact: k8s-manifests

                - task: KubernetesManifest@1
                  displayName: 'Deploy to AKS Dev'
                  inputs:
                    action: 'deploy'
                    connectionType: 'azureResourceManager'
                    azureSubscriptionConnection: 'my-azure-sub'
                    azureResourceGroup: 'dev-rg'
                    kubernetesCluster: 'dev-aks'
                    namespace: 'myapp'
                    manifests: '$(Pipeline.Workspace)/k8s-manifests/*.yaml'
                    containers: '$(dockerRegistry)/$(imageName):$(tag)'

                - script: |
                    echo "Running smoke tests against dev..."
                    curl -f http://dev.myapp.com/health || exit 1
                  displayName: '🔍 Smoke Test'

# ============================================================
# STAGE 3: DEPLOY TO STAGING
# ============================================================
  - stage: DeployStaging
    displayName: '🎭 Deploy to Staging'
    dependsOn: DeployDev
    condition: succeeded()
    jobs:
      - deployment: DeployStaging
        displayName: 'Deploy to Staging'
        pool:
          vmImage: 'ubuntu-latest'
        environment: 'staging'
        strategy:
          runOnce:
            deploy:
              steps:
                - download: current
                  artifact: k8s-manifests

                - task: KubernetesManifest@1
                  displayName: 'Deploy to AKS Staging'
                  inputs:
                    action: 'deploy'
                    connectionType: 'azureResourceManager'
                    azureSubscriptionConnection: 'my-azure-sub'
                    azureResourceGroup: 'staging-rg'
                    kubernetesCluster: 'staging-aks'
                    namespace: 'myapp'
                    manifests: '$(Pipeline.Workspace)/k8s-manifests/*.yaml'
                    containers: '$(dockerRegistry)/$(imageName):$(tag)'

                - script: |
                    echo "Running integration tests..."
                    npm run test:integration
                  displayName: '🧪 Integration Tests'

# ============================================================
# STAGE 4: DEPLOY TO PRODUCTION (with approval!)
# ============================================================
  - stage: DeployProd
    displayName: '🏭 Deploy to Production'
    dependsOn: DeployStaging
    condition: and(succeeded(), eq(variables['Build.SourceBranch'], 'refs/heads/main'))
    jobs:
      - deployment: DeployProd
        displayName: 'Deploy to Production'
        pool:
          vmImage: 'ubuntu-latest'
        environment: 'production'        # ← Has approval gate!
        strategy:
          runOnce:
            deploy:
              steps:
                - download: current
                  artifact: k8s-manifests

                - task: KubernetesManifest@1
                  displayName: 'Deploy to AKS Production'
                  inputs:
                    action: 'deploy'
                    connectionType: 'azureResourceManager'
                    azureSubscriptionConnection: 'my-azure-sub'
                    azureResourceGroup: 'prod-rg'
                    kubernetesCluster: 'prod-aks'
                    namespace: 'myapp'
                    manifests: '$(Pipeline.Workspace)/k8s-manifests/*.yaml'
                    containers: '$(dockerRegistry)/$(imageName):$(tag)'

                - script: |
                    echo "Verifying production deployment..."
                    curl -f https://www.myapp.com/health || exit 1
                  displayName: '✅ Production Health Check'
```

**This pipeline:**
1. **Builds & tests** the code on every push
2. **Builds a Docker image** and pushes to ACR
3. **Deploys to Dev** automatically (with smoke test)
4. **Deploys to Staging** (with integration tests)
5. **Deploys to Production** only from `main`, after human approval

## 17. Azure Boards — Work Item Tracking

### What is Azure Boards?

Azure Boards is a **work tracking system** — like Jira or Trello. You create work items to track what needs to be done, assign them to people, and track progress through sprints.

### Work Item Types

Depending on your process template (Agile, Scrum, or CMMI):

**Agile Process:**

```
Epic (Big feature, takes months)
  └── Feature (Medium feature, takes weeks)
        └── User Story (Small piece of work, takes days)
              └── Task (Technical task, takes hours)
              └── Bug (Defect to fix)
```

**Scrum Process:**

```
Epic
  └── Feature
        └── Product Backlog Item (PBI) — equivalent to User Story
              └── Task
              └── Bug
```

### Creating Work Items

**Via UI:**
1. Go to Boards → **Work Items** → **+ New Work Item**
2. Select type: User Story
3. Fill in:
   - **Title**: "As a user, I can login with Google OAuth"
   - **Description**: Detailed acceptance criteria
   - **Assigned To**: Developer name
   - **State**: New → Active → Resolved → Closed
   - **Priority**: 1 (Critical), 2 (High), 3 (Medium), 4 (Low)
   - **Story Points**: 5 (effort estimation)

**Via CLI:**

```bash
# Create a user story
az boards work-item create \
  --type "User Story" \
  --title "As a user, I can login with Google OAuth" \
  --assigned-to "dev@company.com" \
  --fields "Microsoft.VSTS.Common.Priority=2" "Microsoft.VSTS.Scheduling.StoryPoints=5"

# Create a bug
az boards work-item create \
  --type Bug \
  --title "Login button not working on mobile" \
  --assigned-to "dev@company.com" \
  --fields "Microsoft.VSTS.Common.Severity=2 - High"

# Create a task under a user story
az boards work-item create \
  --type Task \
  --title "Set up Google OAuth credentials" \
  --assigned-to "dev@company.com" \
  --parent 1234         # Parent work item ID
```

### Linking Work Items to Code

This is one of the BEST features of Azure DevOps — full traceability:

```bash
# Link a work item in your commit message
git commit -m "Implement Google OAuth login #1234"
# Azure DevOps automatically links this commit to work item #1234!

# Or in a PR
az repos pr create \
  --source-branch feature/google-oauth \
  --target-branch main \
  --title "Google OAuth login" \
  --work-items 1234 1235      # Link multiple work items
```

Now you can see:
- Which commits relate to which work item
- Which PRs relate to which work item  
- Which builds/deployments include which work items

### Queries — Finding Work Items

```bash
# Create a query via CLI
az boards query \
  --wiql "SELECT [System.Id], [System.Title], [System.State] 
          FROM WorkItems 
          WHERE [System.AssignedTo] = @Me 
          AND [System.State] = 'Active'
          ORDER BY [Microsoft.VSTS.Common.Priority]"
```

**Common queries:**

| Query | WIQL Filter |
|-------|------------|
| My active items | `[Assigned To] = @Me AND [State] = 'Active'` |
| All open bugs | `[Work Item Type] = 'Bug' AND [State] <> 'Closed'` |
| Sprint backlog | `[Iteration Path] = @CurrentIteration` |
| Unassigned items | `[Assigned To] = '' AND [State] = 'New'` |
| High priority | `[Priority] <= 2 AND [State] <> 'Closed'` |

## 21. Common Interview Questions

**Q: What is Azure DevOps? Name its main services.**
Azure DevOps is Microsoft's end-to-end DevOps platform with 5 services: Azure Boards (work tracking), Azure Repos (Git repositories), Azure Pipelines (CI/CD), Azure Test Plans (testing), and Azure Artifacts (package management).

**Q: What's the difference between Classic and YAML pipelines?**
Classic pipelines are built in the UI (clicks), not version-controlled. YAML pipelines are defined as code in your repo, version-controlled, reviewable through PRs, and are the recommended approach. Classic is being deprecated.

**Q: Explain the pipeline hierarchy.**
Pipeline → Stages → Jobs → Steps. A pipeline has one or more stages (Build, Deploy). Each stage has one or more jobs (which run on agents). Each job has steps (the actual tasks/scripts).

**Q: What are environments in Azure Pipelines?**
Environments represent deployment targets (dev, staging, production). They provide approval gates, deployment history, and resource tracking. You can add approval checks so production deployments require human approval.

**Q: How do you store secrets in Azure Pipelines?**
Three ways: (1) Pipeline variables marked as secret, (2) Variable Groups linked to Azure Key Vault, (3) Variable Groups with locked variables. Never put secrets in YAML files.

**Q: What are pipeline templates and why use them?**
Templates are reusable YAML snippets (step, job, or stage level). They eliminate duplication, enforce standards (using `extends`), and make pipelines consistent across teams.

**Q: How do you deploy to AKS from Azure Pipelines?**
Use the KubernetesManifest task, Kubectl task, or HelmDeploy task. Set up a service connection to your AKS cluster. Use environments for approval gates before production deployment.

**Q: What's the difference between Microsoft-hosted and self-hosted agents?**
Microsoft-hosted: managed by Microsoft, fresh VM per job, zero maintenance, limited free minutes. Self-hosted: your own machines, persistent cache (faster builds), unlimited minutes, but you maintain them.

**Q: How does Azure Boards support Agile?**
Azure Boards supports Agile, Scrum, and CMMI processes. It has backlogs, sprint planning, Kanban boards, burndown charts, and velocity tracking. Work items (User Stories, Tasks, Bugs) can be linked to code, PRs, and builds for full traceability.
