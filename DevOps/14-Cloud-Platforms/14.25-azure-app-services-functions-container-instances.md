## 1. Introduction — What Are We Covering?

Azure gives you **multiple ways** to run your applications in the cloud. You don't always need a full Virtual Machine. In fact, most modern apps use one of these three lighter-weight options:

| Service | One-Line Description |
|---------|---------------------|
| **App Service** | A fully managed platform to host web apps, APIs, and mobile backends |
| **Azure Functions** | Run small pieces of code (functions) without managing any servers |
| **Container Instances** | Run Docker containers in Azure without managing VMs or orchestrators |

Think of it like this:
- **App Service** = "I have a web app, just host it for me."
- **Azure Functions** = "I have a tiny piece of code, run it when something happens."
- **ACI** = "I have a Docker container, just run it."

Let's dive deep into each one! 🚀

## 2. Azure App Service

### What is Azure App Service?

Azure App Service is a **fully managed Platform-as-a-Service (PaaS)** for hosting web applications, REST APIs, and mobile back-ends. 

In simple words: **You give Azure your code, and Azure takes care of everything else** — the servers, the OS, the patching, the load balancing, the scaling. You just focus on your application.

**Supported Languages/Frameworks:**
- .NET / .NET Core
- Java
- Node.js
- Python
- PHP
- Ruby
- Go (via custom containers)
- Any language (via custom Docker containers)

**Key Features:**
- Built-in auto-scaling
- CI/CD integration (GitHub, Azure DevOps, Bitbucket, Docker Hub)
- Custom domains with free SSL certificates
- Deployment slots (for zero-downtime deployments)
- Built-in authentication (Azure AD, Google, Facebook, etc.)
- WebSockets, HTTP/2 support
- Hybrid connections to on-premises resources

---

### App Service Plans

An **App Service Plan** is the underlying compute infrastructure for your app. Think of it as the "server" (but managed) that your app runs on.

Every App Service app must live inside an App Service Plan. **Multiple apps can share the same plan.**

**Tiers Available:**

| Tier | Use Case | Key Features |
|------|----------|--------------|
| **Free (F1)** | Testing & learning | 1 GB disk, 60 min/day CPU, no custom domains, no SSL |
| **Shared (D1)** | Dev/test | Custom domains, 240 min/day CPU, still shared infra |
| **Basic (B1, B2, B3)** | Dev/test workloads | Dedicated VMs, manual scaling, custom domains + SSL |
| **Standard (S1, S2, S3)** | Production workloads | Auto-scale, deployment slots, VNet integration |
| **Premium (P1v3, P2v3, P3v3)** | High-performance | More memory/CPU, more slots, zone redundancy |
| **Isolated (I1v2, I2v2, I3v2)** | Enterprise/compliance | Runs in your own isolated environment (App Service Environment) |

**How to choose?**
- Just learning? → **Free** or **Basic**
- Production web app? → **Standard** or **Premium**
- Strict network isolation needed? → **Isolated**

### Creating an App Service (Portal)

1. Go to **Azure Portal** → **Create a Resource** → **Web App**
2. Fill in the details:
   - **Subscription:** Your subscription
   - **Resource Group:** Create new or pick existing
   - **Name:** Must be globally unique (becomes `yourname.azurewebsites.net`)
   - **Publish:** Code or Docker Container
   - **Runtime Stack:** e.g., Node.js 20 LTS, Python 3.11, .NET 8
   - **Region:** Pick one close to your users
   - **App Service Plan:** Select a tier
3. Click **Review + Create** → **Create**

That's it! Your app will have a URL like `https://my-cool-app.azurewebsites.net`.

### Creating an App Service (Azure CLI)

```bash
# Step 1: Login to Azure
az login

# Step 2: Create a Resource Group (skip if you already have one)
az group create \
  --name myResourceGroup \
  --location eastus

# Step 3: Create an App Service Plan
az appservice plan create \
  --name myAppServicePlan \
  --resource-group myResourceGroup \
  --sku S1 \
  --is-linux
# --sku S1       → Standard tier
# --is-linux     → Linux-based plan (remove for Windows)

# Step 4: Create the Web App
az webapp create \
  --name my-unique-webapp-12345 \
  --resource-group myResourceGroup \
  --plan myAppServicePlan \
  --runtime "NODE:20-lts"
# --name         → Must be globally unique
# --runtime      → The language/version stack

# Step 5: Check it's running
az webapp show \
  --name my-unique-webapp-12345 \
  --resource-group myResourceGroup \
  --query "defaultHostName" \
  --output tsv
# Output: my-unique-webapp-12345.azurewebsites.net
```

**To see available runtimes:**

```bash
az webapp list-runtimes --os linux
# Shows: NODE:20-lts, PYTHON:3.11, DOTNETCORE:8.0, etc.
```

```bash
  --name my-unique-webapp-12345 \
```

### Deployment Slots

This is one of the **most powerful features** of App Service. Deployment slots let you deploy to a staging environment first, test it, and then **swap** it to production with **zero downtime**.

**How It Works:**

```
┌──────────────────────┐     ┌──────────────────────┐
│   Production Slot    │     │    Staging Slot       │
│   (live traffic)     │     │   (new version)       │
│                      │     │                       │
│  v1.0 of your app    │     │  v2.0 of your app     │
└──────────────────────┘     └──────────────────────┘
                    │              │
                    └──── SWAP ────┘
                         ↓
┌──────────────────────┐     ┌──────────────────────┐
│   Production Slot    │     │    Staging Slot       │
│   (live traffic)     │     │   (old version)       │
│                      │     │                       │
│  v2.0 of your app    │     │  v1.0 of your app     │
└──────────────────────┘     └──────────────────────┘
```

**Why is this great?**
- Zero downtime deployments
- If something goes wrong, just swap back!
- Test in a production-like environment before going live

```bash
# Create a staging slot
az webapp deployment slot create \
  --name my-unique-webapp-12345 \
  --resource-group myResourceGroup \
  --slot staging

# Deploy to the staging slot
az webapp deploy \
  --name my-unique-webapp-12345 \
  --resource-group myResourceGroup \
  --slot staging \
  --src-path ./myapp.zip \
  --type zip

# The staging slot gets its own URL:
# my-unique-webapp-12345-staging.azurewebsites.net

# When you're happy with staging, SWAP it to production
az webapp deployment slot swap \
  --name my-unique-webapp-12345 \
  --resource-group myResourceGroup \
  --slot staging \
  --target-slot production
# 🎉 Instant swap! Zero downtime!
```

> **Note:** Deployment slots are available in **Standard tier and above**. Free and Basic plans don't have slots.

### Custom Domains & SSL/TLS

By default, your app gets `yourapp.azurewebsites.net`. But you probably want `www.yourcompany.com`.

```bash
# Step 1: Add your custom domain
az webapp config hostname add \
  --webapp-name my-unique-webapp-12345 \
  --resource-group myResourceGroup \
  --hostname www.mycustomdomain.com

# Step 2: You'll need to verify ownership by adding a CNAME or TXT record
# in your DNS provider pointing to your app:
# CNAME: www → my-unique-webapp-12345.azurewebsites.net
# or
# TXT: asuid.www → (the verification ID from Azure)

# Step 3: Add a free managed SSL certificate
az webapp config ssl create \
  --name my-unique-webapp-12345 \
  --resource-group myResourceGroup \
  --hostname www.mycustomdomain.com

# Step 4: Bind the certificate
az webapp config ssl bind \
  --name my-unique-webapp-12345 \
  --resource-group myResourceGroup \
  --certificate-thumbprint <thumbprint> \
  --ssl-type SNI
```

> **Free SSL!** Azure App Service provides **free managed certificates** for custom domains. No need to buy one!

### App Settings & Connection Strings

App settings are like **environment variables** for your App Service. They override values in your app's configuration files.

```bash
# Set an app setting
az webapp config appsettings set \
  --name my-unique-webapp-12345 \
  --resource-group myResourceGroup \
  --settings DB_HOST=mydb.mysql.database.azure.com API_KEY=abc123

# View all app settings
az webapp config appsettings list \
  --name my-unique-webapp-12345 \
  --resource-group myResourceGroup
```

**In your code, access them like regular environment variables:**

```javascript
// Node.js
const dbHost = process.env.DB_HOST;
const apiKey = process.env.API_KEY;
```

```python
# Python
import os
db_host = os.environ.get('DB_HOST')
api_key = os.environ.get('API_KEY')
```

**Slot-Specific Settings:**

Some settings should **stick to a slot** (not swap). For example, you want your staging slot to always point to the staging database, not the production database.

```bash
# Mark a setting as "slot setting" (sticky to the slot)
az webapp config appsettings set \
  --name my-unique-webapp-12345 \
  --resource-group myResourceGroup \
  --slot staging \
  --slot-settings DB_HOST=staging-db.mysql.database.azure.com
# This setting stays with the staging slot even after a swap!
```

### Scaling — Scale Up vs Scale Out

**Scale Up** = Move to a bigger machine (more CPU, more RAM).

```bash
# Scale up to a bigger plan
az appservice plan update \
  --name myAppServicePlan \
  --resource-group myResourceGroup \
  --sku P1v3
# P1v3 = Premium tier with more resources
```

**Scale Out** = Add more instances (horizontal scaling).

```bash
# Manually scale out to 3 instances
az appservice plan update \
  --name myAppServicePlan \
  --resource-group myResourceGroup \
  --number-of-workers 3
```

**Auto-Scale** (the smart way):

```bash
# Create an autoscale rule
az monitor autoscale create \
  --resource-group myResourceGroup \
  --resource myAppServicePlan \
  --resource-type Microsoft.Web/serverfarms \
  --name myAutoScale \
  --min-count 1 \
  --max-count 10 \
  --count 2

# Add a scale-out rule (when CPU > 70%)
az monitor autoscale rule create \
  --resource-group myResourceGroup \
  --autoscale-name myAutoScale \
  --condition "CpuPercentage > 70 avg 5m" \
  --scale out 1

# Add a scale-in rule (when CPU < 30%)
az monitor autoscale rule create \
  --resource-group myResourceGroup \
  --autoscale-name myAutoScale \
  --condition "CpuPercentage < 30 avg 5m" \
  --scale in 1
```

```
Auto-Scaling Visualization:

CPU < 30%        CPU Normal        CPU > 70%
   │                  │                 │
   ▼                  ▼                 ▼
┌──────┐          ┌──────┐          ┌──────┐  ┌──────┐  ┌──────┐
│ VM 1 │          │ VM 1 │          │ VM 1 │  │ VM 2 │  │ VM 3 │
└──────┘          │ VM 2 │          └──────┘  └──────┘  └──────┘
  (1 instance)    └──────┘           (3 instances)
                  (2 instances)
```

### WebJobs

WebJobs let you run **background tasks** inside your App Service. Think of them as long-running scripts or programs that run alongside your web app.

**Types:**
- **Continuous** — Runs forever (like a worker process)
- **Triggered** — Runs on a schedule (like a cron job) or manually

```bash
# Upload a WebJob (a zip file containing your script)
az webapp webjob triggered create \
  --name my-unique-webapp-12345 \
  --resource-group myResourceGroup \
  --webjob-name myCleanupJob \
  --webjob-type triggered \
  --file-path ./cleanup-job.zip
```

> **Tip:** For new projects, prefer **Azure Functions** over WebJobs. Functions are more flexible and cost-effective. WebJobs are useful when your background task is tightly coupled with your web app.

### App Service Best Practices

1. **Always use deployment slots** for production apps — never deploy directly to production
2. **Use managed identities** instead of storing credentials in app settings
3. **Enable "Always On"** for production apps (prevents the app from going idle)
4. **Use Application Insights** for monitoring (covered in 14.27)
5. **Enable diagnostic logging** — you'll thank yourself when debugging
6. **Set up health check endpoints** — App Service can auto-heal unhealthy instances
7. **Use VNet integration** when your app needs to reach private resources
8. **Use 64-bit** worker process for apps that need more memory
9. **Enable HTTPS Only** — redirect all HTTP traffic to HTTPS
10. **Back up your app** — App Service has built-in backup feature

```bash
# Enable HTTPS Only
az webapp update \
  --name my-unique-webapp-12345 \
  --resource-group myResourceGroup \
  --https-only true

# Enable Always On
az webapp config set \
  --name my-unique-webapp-12345 \
  --resource-group myResourceGroup \
  --always-on true

# Enable Health Check
az webapp config set \
  --name my-unique-webapp-12345 \
  --resource-group myResourceGroup \
  --generic-configurations '{"healthCheckPath": "/api/health"}'
```

## 3. Azure Functions

### What is Azure Functions?

Azure Functions is a **serverless compute service** that lets you run small pieces of code (called "functions") without worrying about infrastructure. You write the code, define when it should run, and Azure handles everything else.

**Think of it like this:**
- A regular web server is like owning a restaurant — you manage everything: the building, the kitchen, the staff.
- Azure Functions is like a food truck that appears only when someone orders food and disappears when they're done. You only pay for the time the food truck is actually serving food.

---

### Serverless — The Big Idea

**Serverless doesn't mean "no servers."** It means **you don't think about servers.** Azure manages the servers for you.

**Key Principles of Serverless:**

| Principle | What It Means |
|-----------|--------------|
| **No server management** | You never provision, patch, or maintain servers |
| **Event-driven** | Code runs in response to events (HTTP request, file upload, timer, etc.) |
| **Auto-scale** | Scales from 0 to thousands of instances automatically |
| **Pay-per-use** | You pay only for the time your code actually runs (down to milliseconds) |
| **Stateless** | Each function execution is independent |

**When to use Azure Functions:**
- Processing files when they're uploaded
- Responding to HTTP requests (lightweight APIs)
- Running scheduled tasks (cron-like)
- Processing messages from a queue
- Reacting to database changes
- Webhooks
- IoT data processing

---

### How Azure Functions Work

```
            ┌─────────────┐
            │   Trigger    │  (What starts the function?)
            │  HTTP, Timer │
            │  Queue, Blob │
            └──────┬──────┘
                   │
                   ▼
            ┌─────────────┐
            │  Your Code   │  (What does the function do?)
            │  (Function)  │
            └──────┬──────┘
                   │
                   ▼
            ┌─────────────┐
            │   Bindings   │  (Where does data go?)
            │  Output to   │
            │  Queue, DB,  │
            │  Blob, etc.  │
            └─────────────┘
```

**Three Key Concepts:**

1. **Trigger** — The event that starts the function (every function has exactly ONE trigger)
2. **Input Binding** — Data the function reads from (optional, zero or more)
3. **Output Binding** — Where the function writes data to (optional, zero or more)

---

### Triggers & Bindings

**Available Triggers:**

| Trigger | When It Fires | Example |
|---------|---------------|---------|
| **HTTP** | When an HTTP request is received | API endpoint, webhook |
| **Timer** | On a schedule (CRON expression) | Run every 5 minutes |
| **Blob Storage** | When a file is added/modified in blob storage | Process uploaded images |
| **Queue Storage** | When a message arrives in a queue | Process orders |
| **Service Bus** | When a message arrives in Service Bus | Enterprise messaging |
| **Event Hub** | When an event arrives in Event Hub | IoT data streams |
| **Event Grid** | When an Event Grid event fires | React to Azure resource changes |
| **Cosmos DB** | When a document changes in Cosmos DB | Sync data across services |
| **SignalR** | When a SignalR connection event happens | Real-time notifications |

**Available Bindings (Input & Output):**

| Binding | Direction | What It Does |
|---------|-----------|-------------|
| **Blob Storage** | Input / Output | Read/write files from/to blob storage |
| **Queue Storage** | Output | Send messages to a queue |
| **Table Storage** | Input / Output | Read/write data to table storage |
| **Cosmos DB** | Input / Output | Read/write documents |
| **Service Bus** | Output | Send messages to Service Bus |
| **SendGrid** | Output | Send emails |
| **SignalR** | Output | Send real-time messages |
| **HTTP** | Output | Return HTTP response |

---

### Hosting Plans for Functions

| Plan | How It Scales | Pricing | Best For |
|------|---------------|---------|----------|
| **Consumption** | Auto-scales from 0 to ∞ | Pay per execution (first 1M executions free/month) | Most use cases, event-driven, low traffic |
| **Premium (Flex)** | Pre-warmed instances + auto-scale | Pay per vCPU-second + memory | Avoiding cold starts, VNet, longer execution |
| **Dedicated (App Service)** | Uses App Service Plan | Pay for the plan (always running) | When you already have an App Service plan |
| **Container Apps** | Auto-scales, Kubernetes-based | Pay per use | Microservices, when you need containers |

**What's a "cold start"?**

When your function hasn't run in a while (Consumption plan), Azure deallocates the resources. The next time someone calls your function, Azure needs to spin up a new instance. This startup time is called a **cold start** and can add a few seconds of delay.

→ Use **Premium plan** if cold starts are unacceptable for your use case.

### Writing Your First Function

**Example 1: HTTP Trigger (JavaScript)**

```javascript
// src/functions/hello.js

const { app } = require('@azure/functions');

// Register an HTTP-triggered function named "hello"
app.http('hello', {
    methods: ['GET', 'POST'],     // Accept both GET and POST
    authLevel: 'anonymous',       // No authentication required
    handler: async (request, context) => {
        // Log that the function was triggered
        context.log('HTTP function processed a request.');

        // Try to get the name from query string or body
        const name = request.query.get('name') 
                     || (await request.text()) 
                     || 'World';

        // Return the response
        return {
            status: 200,
            body: `Hello, ${name}! Welcome to Azure Functions! 🎉`
        };
    }
});
```

**How to call it:**

```bash
# GET request with query parameter
curl "https://my-awesome-func-app.azurewebsites.net/api/hello?name=Ritesh"
# Response: Hello, Ritesh! Welcome to Azure Functions! 🎉

# POST request with body
curl -X POST "https://my-awesome-func-app.azurewebsites.net/api/hello" \
  -d "DevOps Engineer"
# Response: Hello, DevOps Engineer! Welcome to Azure Functions! 🎉
```

**Example 2: Timer Trigger (Runs Every 5 Minutes)**

```javascript
// src/functions/scheduled-cleanup.js

const { app } = require('@azure/functions');

app.timer('scheduledCleanup', {
    // CRON expression: second minute hour day month day-of-week
    schedule: '0 */5 * * * *',   // Every 5 minutes
    handler: async (myTimer, context) => {
        const timestamp = new Date().toISOString();
        context.log(`Cleanup function ran at: ${timestamp}`);

        if (myTimer.isPastDue) {
            context.log('Timer is past due! Catching up...');
        }

        // Your cleanup logic here
        // e.g., delete old records, clean temp files, etc.
        context.log('Cleanup completed successfully!');
    }
});
```

> **CRON format in Azure Functions:** `{second} {minute} {hour} {day} {month} {day-of-week}`
> - `0 */5 * * * *` = Every 5 minutes
> - `0 0 * * * *` = Every hour
> - `0 0 9 * * 1-5` = Every weekday at 9:00 AM

**Example 3: Blob Trigger (Process Uploaded Files)**

```javascript
// src/functions/process-image.js

const { app } = require('@azure/functions');

app.storageBlob('processImage', {
    path: 'uploads/{filename}',           // Watch the "uploads" container
    connection: 'AzureWebJobsStorage',    // Connection string setting name
    handler: async (blob, context) => {
        // blob = the contents of the uploaded file
        // context.triggerMetadata.filename = the file name

        const fileName = context.triggerMetadata.filename;
        const blobSize = blob.length;

        context.log(`New file uploaded: ${fileName}`);
        context.log(`File size: ${blobSize} bytes`);

        // Process the file (resize image, extract text, etc.)
        // ...

        context.log(`Processing complete for: ${fileName}`);
    }
});
```

**Example 4: HTTP Trigger with Queue Output Binding (Python)**

```python
# function_app.py

import azure.functions as func
import json
import logging

app = func.FunctionApp()

@app.route(route="order", methods=["POST"], auth_level=func.AuthLevel.ANONYMOUS)
@app.queue_output(arg_name="msg", queue_name="orders", connection="AzureWebJobsStorage")
def create_order(req: func.HttpRequest, msg: func.Out[str]) -> func.HttpResponse:
    """
    Receives an order via HTTP POST and puts it in a queue for processing.
    """
    logging.info('New order received!')

    try:
        # Parse the order from the request body
        order = req.get_json()
        order_id = order.get('id', 'unknown')
        
        # Send the order to the queue (output binding does this!)
        msg.set(json.dumps(order))
        
        logging.info(f'Order {order_id} added to queue for processing.')
        
        return func.HttpResponse(
            json.dumps({"message": f"Order {order_id} received!"}),
            status_code=202,
            mimetype="application/json"
        )
    except ValueError:
        return func.HttpResponse(
            "Invalid JSON in request body",
            status_code=400
        )
```

**What's happening here?**
1. Someone sends a POST request with order data
2. The function reads the order
3. The `@app.queue_output` binding **automatically** sends the order to an Azure Storage Queue called "orders"
4. Another function (or service) can then pick up messages from that queue and process orders

This is a super common **decoupling pattern** in microservices!

### Durable Functions

Regular Azure Functions are **stateless** — they run, do their thing, and forget. But what if you need a function that:
- Runs a long workflow with multiple steps?
- Waits for external events?
- Runs multiple functions in parallel and waits for all to finish?

That's where **Durable Functions** come in! They add **stateful** capabilities to Azure Functions.

**Patterns:**

| Pattern | What It Does | Example |
|---------|-------------|---------|
| **Function Chaining** | Run functions one after another | Order → Validate → Charge → Ship |
| **Fan-out/Fan-in** | Run many functions in parallel, wait for all | Process 100 images simultaneously |
| **Async HTTP** | Start a long task, return immediately, check status later | File processing, report generation |
| **Monitor** | Periodically check a condition | Poll an API until a job completes |
| **Human Interaction** | Wait for a human to approve | Expense approval workflow |

**Simple Chaining Example (JavaScript):**

```javascript
// Orchestrator function — defines the workflow
const df = require('durable-functions');

df.app.orchestration('orderProcessing', function* (context) {
    // Step 1: Validate the order
    const isValid = yield context.df.callActivity('validateOrder', context.df.input);
    
    // Step 2: Charge the customer
    if (isValid) {
        const payment = yield context.df.callActivity('chargeCustomer', context.df.input);
    
        // Step 3: Ship the order
        const shipment = yield context.df.callActivity('shipOrder', {
            orderId: context.df.input.orderId,
            paymentId: payment.id
        });
    
        return shipment;
    }

    return { error: 'Order validation failed' };
});

// Activity functions — the actual work
df.app.activity('validateOrder', {
    handler: async (input) => {
        // Validate the order...
        return true;
    }
});

df.app.activity('chargeCustomer', {
    handler: async (input) => {
        // Charge the customer...
        return { id: 'PAY-12345' };
    }
});

df.app.activity('shipOrder', {
    handler: async (input) => {
        // Ship the order...
        return { trackingNumber: 'TRACK-67890' };
    }
});
```

### Azure Functions Best Practices

1. **Keep functions small and focused** — one function = one job
2. **Use bindings** instead of writing SDK code for common Azure services
3. **Don't make functions stateful** — use Durable Functions if you need state
4. **Set appropriate timeouts** — Consumption plan has a 5-min default (max 10 min)
5. **Use Application Insights** for monitoring and debugging
6. **Avoid long-running functions** on Consumption plan — use Premium or Dedicated
7. **Use managed identities** for authentication instead of connection strings
8. **Handle cold starts** — keep functions lightweight, minimize dependencies
9. **Use async/await** properly — don't block the event loop
10. **Version your function endpoints** — `/api/v1/orders` instead of `/api/orders`

## 4. Azure Container Instances (ACI)

### What is ACI?

Azure Container Instances (ACI) is the **fastest and simplest way to run a Docker container in Azure**. No VMs to manage. No Kubernetes to set up. Just give it a container image, and it runs.

**Think of it like this:**
- **Docker on your laptop** = runs containers locally
- **ACI** = runs containers in the cloud with one command
- **AKS (Kubernetes)** = runs and orchestrates hundreds/thousands of containers

ACI is perfect when you just need to **run a container** without all the complexity of an orchestrator.

---

### ACI vs AKS vs App Service

| Feature | ACI | AKS (Kubernetes) | App Service |
|---------|-----|-------------------|-------------|
| **Complexity** | Very simple | Complex | Simple |
| **Startup time** | Seconds | Minutes (cluster setup) | Seconds |
| **Orchestration** | None | Full orchestration | None |
| **Scaling** | Manual (or with KEDA) | Auto-scaling | Auto-scaling |
| **Use case** | Quick tasks, burst, simple apps | Microservices at scale | Web apps & APIs |
| **Pricing** | Per-second (CPU + memory) | Pay for VMs/nodes | Pay for plan |
| **Persistent storage** | Azure Files only | Full storage options | Built-in |
| **Networking** | Basic | Advanced (Network Policies) | VNet integration |

**When to use ACI:**
- Run a quick batch job or script
- CI/CD build agents
- Dev/test environments
- Burst workloads from AKS (virtual nodes)
- Simple containerized apps
- One-off tasks

**When NOT to use ACI:**
- Complex microservices architectures → use **AKS**
- You need advanced orchestration → use **AKS**
- You're running a web app without containers → use **App Service**

---

### Creating a Container Instance (Portal)

1. Go to **Azure Portal** → **Create a Resource** → **Container Instances**
2. Fill in:
   - **Container Name:** my-container
   - **Image Source:** Docker Hub, Azure Container Registry, or other
   - **Image:** e.g., `nginx:latest` or `mcr.microsoft.com/azuredocs/aci-helloworld`
   - **OS Type:** Linux or Windows
   - **Size:** CPU cores and memory
3. Configure **Networking:** Public IP, DNS name label, ports
4. Click **Create**

### Creating a Container Instance (Azure CLI)

**Example 1: Run a simple container**

```bash
# Run an Nginx container in Azure
az container create \

# Output:

# --------  ------------------------------------------  -----------

# Visit: http://my-nginx-demo-12345.eastus.azurecontainer.io
```

**Example 2: Run a container with environment variables**

```bash
az container create \
  --name my-app \
  --resource-group myResourceGroup \
  --image myregistry.azurecr.io/myapp:v1 \
  --cpu 2 \
  --memory 4 \
  --ports 8080 \
  --environment-variables \
    DB_HOST=mydb.mysql.database.azure.com \
    APP_ENV=production \
  --secure-environment-variables \
    DB_PASSWORD=SuperSecret123 \
  --dns-name-label my-app-demo
# --environment-variables         → Regular env vars (visible in logs)
# --secure-environment-variables  → Secret env vars (hidden from view)
```

**Example 3: Run a one-time task (no restart)**

```bash
az container create \
  --name data-migration \
  --resource-group myResourceGroup \
  --image myregistry.azurecr.io/migration-script:v1 \
  --restart-policy Never \
  --cpu 2 \
  --memory 4
# --restart-policy Never → Run once and stop
# Other options: Always (default), OnFailure

# Check logs
az container logs \
  --name data-migration \
  --resource-group myResourceGroup

# Stream logs in real-time
az container logs \
  --name data-migration \
  --resource-group myResourceGroup \
  --follow

# Execute a command inside the running container
az container exec \
  --name data-migration \
  --resource-group myResourceGroup \
  --exec-command "/bin/bash"
```

**Restart Policies:**

| Policy | Behavior | Use Case |
|--------|----------|----------|
| `Always` | Restart when container stops | Long-running services (web server) |
| `OnFailure` | Restart only on non-zero exit code | Batch jobs that might fail |
| `Never` | Never restart | One-time tasks, migrations |

### Container Groups

A **Container Group** is ACI's equivalent of a Kubernetes Pod. It's a collection of containers that:
- Share the same lifecycle (start/stop together)
- Share the same network (same IP, same ports)
- Share storage volumes
- Run on the same host

```
Container Group (shared IP: 20.62.x.x)
┌──────────────────────────────────────────────┐
│                                              │
│  ┌──────────────┐    ┌──────────────┐        │
│  │ Web App      │    │ Sidecar      │        │
│  │ Container    │    │ Container    │        │
│  │ (port 80)    │◄──►│ (log shipper)│        │
│  └──────────────┘    └──────────────┘        │
│                                              │
│  ┌─────────────────────────────────────┐     │
│  │ Shared Azure Files Volume           │     │
│  └─────────────────────────────────────┘     │
│                                              │
└──────────────────────────────────────────────┘
```

**Deploy a Container Group using YAML:**

```yaml
# container-group.yaml
apiVersion: '2021-09-01'
name: my-container-group
location: eastus
properties:
  osType: Linux
  containers:
    # Main application container
    - name: web-app
      properties:
        image: nginx:latest
        ports:
          - port: 80
        resources:
          requests:
            cpu: 1.0
            memoryInGb: 1.5
        volumeMounts:
          - name: shared-data
            mountPath: /usr/share/nginx/html
    # Sidecar container (e.g., log shipper)
    - name: log-shipper
      properties:
        image: fluentd:latest
        resources:
          requests:
            cpu: 0.5
            memoryInGb: 0.5
        volumeMounts:
          - name: shared-data
            mountPath: /var/log/app
  ipAddress:
    type: Public
    ports:
      - port: 80
        protocol: TCP
    dnsNameLabel: my-group-demo
  volumes:
    - name: shared-data
      azureFile:
        shareName: myshare
        storageAccountName: mystorageaccount
        storageAccountKey: <your-storage-key>
```

```bash
# Deploy the container group
az container create \
  --resource-group myResourceGroup \
  --file container-group.yaml
```

### Persistent Storage with ACI

By default, container storage is **ephemeral** — when the container stops, the data is gone. To keep data, mount an **Azure Files share**.

```bash
# Step 1: Create a storage account
az storage account create \
  --name mystorageacct123 \
  --resource-group myResourceGroup \
  --location eastus \
  --sku Standard_LRS

# Step 2: Create a file share
az storage share create \
  --name myshare \
  --account-name mystorageacct123

# Step 3: Get the storage account key
STORAGE_KEY=$(az storage account keys list \
  --resource-group myResourceGroup \
  --account-name mystorageacct123 \
  --query "[0].value" \
  --output tsv)

# Step 4: Create a container with the file share mounted
az container create \
  --name my-persistent-app \
  --resource-group myResourceGroup \
  --image nginx:latest \
  --ports 80 \
  --azure-file-volume-share-name myshare \
  --azure-file-volume-account-name mystorageacct123 \
  --azure-file-volume-account-key $STORAGE_KEY \
  --azure-file-volume-mount-path /data
# The /data directory in the container now persists data to Azure Files!
```

### ACI Networking

**Public IP:** (default)
- Container gets a public IP
- Optional DNS name label (FQDN)

**Private (VNet):**
- Container runs inside your Virtual Network
- No public IP — only accessible from within VNet

```bash
# Deploy ACI into a VNet (private)

  --image myapp:latest \
  --cpu 1 \
  --memory 1.5 \
  --vnet myVNet \
  --subnet aci-subnet \
  --ports 8080
# No public IP — only reachable from within the VNet
```

### ACI with Azure Container Registry (ACR)

Most real-world scenarios pull images from a **private** Azure Container Registry.

```bash
# Step 1: Create an ACR
az acr create \
  --name myacr12345 \
  --resource-group myResourceGroup \
  --sku Basic

# Step 2: Build and push an image to ACR
az acr build \
  --registry myacr12345 \
  --image myapp:v1 \
  --file Dockerfile .

# Step 3: Run the image from ACR in ACI
az container create \
  --name my-acr-app \
  --resource-group myResourceGroup \
  --image myacr12345.azurecr.io/myapp:v1 \
  --cpu 1 \
  --memory 1.5 \
  --ports 8080 \
  --registry-login-server myacr12345.azurecr.io \
  --registry-username $(az acr credential show --name myacr12345 --query username -o tsv) \
  --registry-password $(az acr credential show --name myacr12345 --query "passwords[0].value" -o tsv)
```

> **Better approach:** Use **Managed Identity** instead of username/password:

```bash
# Assign a managed identity to ACI and grant it ACR pull access
az container create \
  --name my-acr-app \
  --resource-group myResourceGroup \
  --image myacr12345.azurecr.io/myapp:v1 \
  --cpu 1 \
  --memory 1.5 \
  --ports 8080 \
  --acr-identity [system] \
  --assign-identity
# [system] tells ACI to use a system-assigned managed identity
# Azure automatically grants ACI permission to pull from ACR
```

### ACI Best Practices

1. **Use ACI for the right workload** — it's not a replacement for Kubernetes
2. **Set restart policies appropriately** — `Never` for batch jobs, `Always` for services
3. **Use secure environment variables** for secrets (or better, use Key Vault)
4. **Set CPU/memory limits** thoughtfully — you pay for what you allocate
5. **Use managed identities** instead of passing credentials
6. **Pull images from ACR** (private) instead of Docker Hub (public) for production
7. **Use container groups** for sidecar patterns (logging, monitoring)
8. **Monitor with Azure Monitor** and **Log Analytics**
9. **Use VNet integration** when containers need to talk to private resources
10. **Clean up stopped containers** — you still pay for allocated resources even if stopped

```bash
# Delete a container instance
az container delete \
  --name my-nginx \
  --resource-group myResourceGroup \
  --yes

# List all container instances
az container list \
  --resource-group myResourceGroup \
  --output table
```

## 5. Comparison Table — App Service vs Functions vs ACI

| Feature | App Service | Azure Functions | ACI |
|---------|------------|----------------|-----|
| **Type** | PaaS | Serverless/FaaS | CaaS (Container-as-a-Service) |
| **Best for** | Web apps, APIs | Event-driven code, small tasks | Run containers quickly |
| **Code/Container** | Both | Both | Container only |
| **Scaling** | Auto-scale (plan-based) | Auto-scale (event-based, 0 to ∞) | Manual |

| **Networking** | VNet integration | VNet integration | VNet integration |
| **Custom domains** | Yes | Yes | DNS label only |
| **SSL/TLS** | Free managed certs | Free managed certs | Manual |
| **Pricing** | Per plan (always running) | Per execution (can be $0) | Per second (CPU + memory) |
| **Deployment slots** | Yes | Yes | No |
| **Complexity** | Low | Low-Medium | Very Low |

## 6. Real-World Scenarios

### Scenario 1: E-commerce Website
**Use App Service** for the main web application + **Azure Functions** for:
- Sending order confirmation emails (Queue trigger)
- Generating invoices (Blob trigger when order is placed)
- Nightly inventory sync (Timer trigger)

### Scenario 2: Image Processing Pipeline
1. User uploads image to **Blob Storage**
2. **Azure Function** (Blob trigger) picks it up
3. Function sends image to **ACI** for heavy processing (AI model inference)
4. Processed image is saved back to Blob Storage
5. **Azure Function** (Blob trigger) notifies the user

### Scenario 3: CI/CD Build Agent
Use **ACI** to spin up containerized build agents on demand:
- CI pipeline starts → Spin up ACI with build tools
- Build completes → ACI stops and is deleted
- Pay only for the build time

### Scenario 4: Microservices with Mixed Workloads
- **App Service** for the main customer-facing web app
- **Azure Functions** for background processing and event handling
- **ACI** for ad-hoc batch processing and data migration jobs
- **AKS** for the core microservices that need full orchestration

## 7. Summary

| Service | Remember This |
|---------|--------------|
| **App Service** | "I have a web app — host it for me with scaling, slots, and custom domains" |
| **Azure Functions** | "I have small code — run it when X happens, and I'll pay only when it runs" |
| **ACI** | "I have a container — just run it in the cloud, no orchestration needed" |

**Quick Decision Guide:**

```
Do you have a web app?
  → Yes → Azure App Service

Do you need to react to events or run small tasks?
  → Yes → Azure Functions

Do you have a Docker container and just want to run it?
  → Yes → ACI (simple) or AKS (complex)

Do you need to orchestrate many containers?
  → Yes → AKS (Kubernetes), not ACI
```
