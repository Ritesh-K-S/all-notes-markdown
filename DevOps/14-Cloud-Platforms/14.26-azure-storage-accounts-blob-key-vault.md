## 1. Introduction — Why Storage & Secrets Matter

Every application needs two things:

1. **A place to store data** — files, images, backups, logs, configuration, etc.
2. **A safe place for secrets** — API keys, database passwords, certificates, encryption keys

Azure provides elegant solutions for both:

| Need | Azure Service |
|------|--------------|
| Store files, blobs, objects | **Azure Blob Storage** |
| Store structured data (NoSQL) | **Azure Table Storage** |
| Message queuing | **Azure Queue Storage** |
| Shared file system (SMB/NFS) | **Azure Files** |
| Secrets, keys, certificates | **Azure Key Vault** |

All of the first four live inside an **Azure Storage Account**. Key Vault is a separate service specifically for secrets and encryption.

Let's explore each one!

## 2. Azure Storage Accounts

### What is a Storage Account?

A Storage Account is a **container for all your Azure Storage services**. Think of it as a "bucket" (if you're coming from AWS S3 world) but much broader — it holds blobs, files, queues, and tables.

```
┌─────────────────────────────────────────────┐
│           Azure Storage Account             │
│     (mystorageacct12345)                    │
│                                             │
│  ┌───────────┐  ┌───────────┐               │
│  │   Blob    │  │   File    │               │
│  │  Storage  │  │  Shares   │               │
│  └───────────┘  └───────────┘               │
│                                             │
│  ┌───────────┐  ┌───────────┐               │
│  │   Queue   │  │   Table   │               │
│  │  Storage  │  │  Storage  │               │
│  └───────────┘  └───────────┘               │
│                                             │
└─────────────────────────────────────────────┘
```

**Key Facts:**
- Storage account name must be **globally unique** (3-24 characters, lowercase + numbers only)
- Endpoint format: `https://<account-name>.<service>.core.windows.net`
  - Blob: `https://mystorageacct.blob.core.windows.net`
  - Files: `https://mystorageacct.file.core.windows.net`
  - Queue: `https://mystorageacct.queue.core.windows.net`
  - Table: `https://mystorageacct.table.core.windows.net`

---

### Types of Storage Services

| Service | What It Stores | Think of It As | Use Case |
|---------|---------------|----------------|----------|
| **Blob Storage** | Unstructured data (files, images, videos, backups) | AWS S3 / Google Cloud Storage | Store anything: images, videos, logs, backups |
| **File Storage** | Shared file systems (SMB/NFS) | A network drive in the cloud | Lift-and-shift file shares, shared config |
| **Queue Storage** | Messages (up to 64 KB each) | A simple message queue | Decouple microservices, task queuing |
| **Table Storage** | Structured NoSQL data (key-value) | A simple NoSQL database | Logs, user sessions, device data |

---

### Storage Account Types & Redundancy

**Account Types:**

| Type | Description | Supports |
|------|-------------|----------|
| **Standard general-purpose v2** | Most common, recommended for most scenarios | Blob, File, Queue, Table |
| **Premium block blobs** | High-performance for block blobs | Blob only (SSD-backed) |
| **Premium file shares** | High-performance for file shares | File shares only (SSD-backed) |
| **Premium page blobs** | High-performance for page blobs (VM disks) | Page blobs only |

> **Default choice:** Almost always use **Standard general-purpose v2**. Only go premium when you need high IOPS or low latency.

**Redundancy (How Your Data is Protected):**

Azure always keeps **multiple copies** of your data. You choose how many and where:

| Redundancy | Copies | Regions | Durability | Cost |
|-----------|--------|---------|------------|------|
| **LRS** (Locally Redundant) | 3 copies in **one data center** | 1 | 99.999999999% (11 9's) | Lowest |
| **ZRS** (Zone Redundant) | 3 copies across **3 availability zones** | 1 | 99.9999999999% (12 9's) | Medium |
| **GRS** (Geo Redundant) | 6 copies: 3 local + 3 in **another region** | 2 | 99.99999999999999% (16 9's) | Higher |
| **GZRS** (Geo-Zone Redundant) | 6 copies: 3 across zones + 3 in another region | 2 | 99.99999999999999% (16 9's) | Highest |
| **RA-GRS** (Read-Access Geo Redundant) | Same as GRS but you can **read from the secondary** | 2 | 16 9's | Higher |
| **RA-GZRS** (Read-Access Geo-Zone Redundant) | Same as GZRS but you can **read from the secondary** | 2 | 16 9's | Highest |

```
LRS: All 3 copies in one data center
┌─────────────────────────────┐
│    Data Center A            │
│  [Copy 1] [Copy 2] [Copy 3]│
└─────────────────────────────┘

ZRS: 3 copies across availability zones
┌──────────┐ ┌──────────┐ ┌──────────┐
│  Zone 1  │ │  Zone 2  │ │  Zone 3  │
│ [Copy 1] │ │ [Copy 2] │ │ [Copy 3] │
└──────────┘ └──────────┘ └──────────┘

GRS: 3 local + 3 in a paired region
┌──────────────────┐    ┌──────────────────┐
│  Primary Region  │    │ Secondary Region │
│ [C1] [C2] [C3]   │───►│ [C4] [C5] [C6]   │
└──────────────────┘    └──────────────────┘
```

**How to choose:**
- Dev/test? → **LRS** (cheapest)
- Production, single region? → **ZRS** (protects against zone failure)
- Critical data, need DR? → **GRS** or **GZRS** (protects against region failure)

### Creating a Storage Account (Portal)

1. **Azure Portal** → **Create a Resource** → **Storage Account**
2. Fill in:
   - **Subscription & Resource Group**
   - **Storage Account Name:** Must be globally unique, lowercase, 3-24 chars
   - **Region:** Pick one close to your users
   - **Performance:** Standard (HDD) or Premium (SSD)
   - **Redundancy:** LRS, ZRS, GRS, etc.
3. Click **Review + Create** → **Create**

### Creating a Storage Account (Azure CLI)

```bash
# Create a storage account
az storage account create \
  --name mystorageacct12345 \
  --resource-group myResourceGroup \
  --location eastus \
  --sku Standard_ZRS \
  --kind StorageV2 \
  --access-tier Hot
# --name          → Must be globally unique (lowercase + numbers only)
# --sku           → Standard_LRS, Standard_ZRS, Standard_GRS, Standard_RAGRS, etc.
# --kind          → StorageV2 (recommended), BlobStorage, FileStorage
# --access-tier   → Hot (default), Cool

# View the storage account details
az storage account show \
  --name mystorageacct12345 \
  --resource-group myResourceGroup \
  --output table

# List all storage accounts in a resource group
az storage account list \
  --resource-group myResourceGroup \
  --output table

# Get connection string (needed to connect from code)
az storage account show-connection-string \
  --name mystorageacct12345 \
  --resource-group myResourceGroup \
  --output tsv
# Output: DefaultEndpointsProtocol=https;AccountName=mystorageacct12345;AccountKey=xxxxx...

# Get storage account keys
az storage account keys list \
  --name mystorageacct12345 \
  --resource-group myResourceGroup \
  --output table
# Two keys are provided (key1 and key2) for rotation purposes
```

### Access Tiers — Hot, Cool, Cold, Archive

Azure lets you save money by choosing the right **access tier** based on how frequently you access data.

| Tier | Access Frequency | Storage Cost | Access Cost | Min. Storage Duration | Use Case |
|------|-----------------|-------------|-------------|----------------------|----------|
| **Hot** | Frequently accessed | Highest | Lowest | None | Active data, websites, apps |
| **Cool** | Infrequently accessed | Lower | Higher | 30 days | Backups, short-term archive |
| **Cold** | Rarely accessed | Even Lower | Even Higher | 90 days | Compliance data, cold backups |
| **Archive** | Almost never accessed | Lowest | Highest | 180 days | Long-term archive, regulatory |

**Key Points:**
- **Hot** → Data you access daily (app data, current logs)
- **Cool** → Data accessed monthly (last month's logs, weekly backups)
- **Cold** → Data accessed quarterly (quarterly reports, older backups)
- **Archive** → Data you rarely need but must keep (legal records, 7-year audit data)

> **Warning:** Archive tier data is **offline**. You must "rehydrate" it before you can access it, which takes **hours** (up to 15 hours for Standard, up to 48 hours for high priority).

```bash
# Set the default access tier for a storage account
az storage account update \
  --name mystorageacct12345 \
  --resource-group myResourceGroup \
  --access-tier Cool

# Set tier for a specific blob
az storage blob set-tier \
  --account-name mystorageacct12345 \
  --container-name mycontainer \
  --name old-report.pdf \
  --tier Archive
```

### Storage Account Security

**1. Access Keys (Shared Key):**
Every storage account has **two access keys**. Anyone with these keys has **full access** to everything in the account.

```bash
# List access keys
az storage account keys list \
  --name mystorageacct12345 \
  --resource-group myResourceGroup

# Rotate (regenerate) a key
az storage account keys renew \
  --name mystorageacct12345 \
  --resource-group myResourceGroup \
  --key key1
```

> **Why two keys?** So you can rotate keys without downtime. Update your apps to use key2, then rotate key1. Then switch back.

**2. Shared Access Signatures (SAS):**
SAS tokens give **limited, time-bound access** to specific resources. Much safer than sharing your full access key.

```bash
# Generate a SAS token for a container (read-only, expires in 1 day)
az storage container generate-sas \
  --account-name mystorageacct12345 \
  --name mycontainer \
  --permissions r \
  --expiry 2026-06-01T00:00:00Z \
  --output tsv
# Output: se=2026-06-01&sp=r&sv=2022-11-02&sr=c&sig=xxxxx...

# Use the SAS token to access a blob
# https://mystorageacct12345.blob.core.windows.net/mycontainer/file.txt?<SAS-token>
```

**SAS Permission Types:**
| Permission | Letter | Description |
|-----------|--------|-------------|
| Read | `r` | Read blob content |
| Write | `w` | Write/create blobs |
| Delete | `d` | Delete blobs |
| List | `l` | List blobs in container |
| Add | `a` | Add blocks to append blob |
| Create | `c` | Create new blobs |

**3. Managed Identities (Best Practice!):**
Instead of connection strings or keys, let Azure services authenticate to storage using **managed identities**. No credentials to manage!

```bash
# Grant an App Service identity access to a storage account
az role assignment create \
  --assignee <app-service-identity-id> \
  --role "Storage Blob Data Contributor" \
  --scope /subscriptions/<sub>/resourceGroups/<rg>/providers/Microsoft.Storage/storageAccounts/mystorageacct12345
```

**4. Firewall & Network Rules:**

```bash
# Allow access only from a specific VNet/subnet
az storage account network-rule add \
  --account-name mystorageacct12345 \
  --resource-group myResourceGroup \
  --vnet-name myVNet \
  --subnet mySubnet

# Deny all traffic by default (then whitelist)
az storage account update \
  --name mystorageacct12345 \
  --resource-group myResourceGroup \
  --default-action Deny

# Allow access from a specific IP
az storage account network-rule add \
  --account-name mystorageacct12345 \
  --resource-group myResourceGroup \
  --ip-address 203.0.113.50
```

**5. Encryption:**
- All data in Azure Storage is **encrypted at rest by default** (256-bit AES)
- You can use **Microsoft-managed keys** (default) or **customer-managed keys** (stored in Key Vault)
- Data in transit is always encrypted via **HTTPS**

```bash
# Enforce HTTPS only (disable HTTP)
az storage account update \
  --name mystorageacct12345 \
  --resource-group myResourceGroup \
  --https-only true

# Enforce minimum TLS version
az storage account update \
  --name mystorageacct12345 \
  --resource-group myResourceGroup \
  --min-tls-version TLS1_2
```

## 3. Azure Blob Storage

### What is Blob Storage?

**Blob** stands for **Binary Large Object**. Blob Storage is Azure's object storage service for storing massive amounts of unstructured data — anything that doesn't fit neatly into rows and columns.

**Common Use Cases:**
- Storing images, videos, and media files
- Application log files
- Backup and disaster recovery data
- Static website hosting
- Data lake storage for analytics
- VM disk snapshots
- Container images (via Azure Container Registry, which uses blob storage underneath)

**Blob Storage vs File Storage:**
- **Blob Storage** = Object storage (like AWS S3). Flat structure. Access via REST API/SDK.
- **File Storage** = File system (like a network drive). Hierarchical. Access via SMB/NFS.

---

### Blob Types — Block, Append, Page

| Blob Type | How It Works | Max Size | Use Case |
|-----------|-------------|----------|----------|
| **Block Blob** | Made of blocks (chunks). Upload large files in pieces. | 190.7 TB | Files, images, videos, backups — **most common** |
| **Append Blob** | Can only append data (no modify/delete in middle) | 195 GB | Log files, audit trails |
| **Page Blob** | Random read/write access to pages (512-byte aligned) | 8 TB | VM disks (VHD files) |

> **Default:** Almost always use **Block Blobs**. They're the most versatile and most commonly used.

---

### Containers & Blobs — The Structure

```
Storage Account: mystorageacct12345
│
├── Container: "images"
│   ├── photo1.jpg
│   ├── photo2.png
│   └── thumbnails/
│       ├── photo1-thumb.jpg
│       └── photo2-thumb.png
│
├── Container: "logs"
│   ├── 2026/01/app.log
│   ├── 2026/02/app.log
│   └── 2026/03/app.log
│
└── Container: "backups"
    ├── db-backup-2026-01-01.sql.gz
    └── db-backup-2026-02-01.sql.gz
```

**Containers** are like top-level folders. **Blobs** are the actual files inside containers.

> **Note:** Blob storage is technically **flat** — there are no real folders. The "/" in `thumbnails/photo1-thumb.jpg` is just part of the blob name. But tools and the portal display them as folders for convenience.

**Container Access Levels:**

| Level | Who Can Access |
|-------|---------------|
| **Private** (default) | Only the storage account owner (with key, SAS, or RBAC) |
| **Blob** | Anonymous read access for blobs only (not listing) |
| **Container** | Anonymous read access for blobs AND listing |

> **For production:** Always keep containers **Private** and use SAS tokens or managed identities for access.

### Working with Blob Storage (Azure CLI)

```bash
# ============================================================
# SETUP: Set environment variable for convenience
# ============================================================
export AZURE_STORAGE_ACCOUNT=mystorageacct12345
# Or use --account-name in every command

# ============================================================
# CONTAINERS
# ============================================================

# Create a container
az storage container create \
  --name images \
  --account-name mystorageacct12345 \
  --public-access off
# --public-access off → Private (default and recommended)

# List all containers
az storage container list \
  --account-name mystorageacct12345 \
  --output table

# Delete a container (CAREFUL! Deletes all blobs inside)
az storage container delete \
  --name old-container \
  --account-name mystorageacct12345

# ============================================================
# UPLOAD BLOBS
# ============================================================

# Upload a single file
az storage blob upload \
  --account-name mystorageacct12345 \
  --container-name images \
  --name photo1.jpg \
  --file ./photo1.jpg \
  --tier Hot
# --name → The blob name in Azure
# --file → The local file path

# Upload all files from a directory
az storage blob upload-batch \
  --account-name mystorageacct12345 \
  --destination images \
  --source ./my-photos/ \
  --pattern "*.jpg"
# Uploads all .jpg files from ./my-photos/ to the "images" container

# Upload with a "folder" prefix
az storage blob upload \
  --account-name mystorageacct12345 \
  --container-name images \
  --name thumbnails/photo1-thumb.jpg \
  --file ./photo1-thumb.jpg

# ============================================================
# DOWNLOAD BLOBS
# ============================================================

# Download a single blob
az storage blob download \
  --account-name mystorageacct12345 \
  --container-name images \
  --name photo1.jpg \
  --file ./downloaded-photo1.jpg

# Download all blobs from a container
az storage blob download-batch \
  --account-name mystorageacct12345 \
  --source images \
  --destination ./downloaded-images/

# ============================================================
# LIST BLOBS
# ============================================================

# List all blobs in a container
az storage blob list \
  --account-name mystorageacct12345 \
  --container-name images \
  --output table

# List blobs with a prefix (like listing a "folder")
az storage blob list \
  --account-name mystorageacct12345 \
  --container-name images \
  --prefix "thumbnails/" \
  --output table

# ============================================================
# DELETE BLOBS
# ============================================================

# Delete a single blob
az storage blob delete \
  --account-name mystorageacct12345 \
  --container-name images \
  --name old-photo.jpg

# Delete all blobs matching a pattern
az storage blob delete-batch \
  --account-name mystorageacct12345 \
  --source images \
  --pattern "thumbnails/*"

# ============================================================
# BLOB PROPERTIES & METADATA
# ============================================================

# Show blob properties (size, type, last modified, etc.)
az storage blob show \
  --account-name mystorageacct12345 \
  --container-name images \
  --name photo1.jpg

# Set custom metadata on a blob
az storage blob metadata update \
  --account-name mystorageacct12345 \
  --container-name images \
  --name photo1.jpg \
  --metadata category=vacation year=2026

# ============================================================
# GENERATE A DOWNLOAD URL
# ============================================================

# Generate a URL with SAS token (valid for 1 hour)
az storage blob generate-sas \
  --account-name mystorageacct12345 \
  --container-name images \
  --name photo1.jpg \
  --permissions r \
  --expiry $(date -u -d "1 hour" '+%Y-%m-%dT%H:%MZ') \
  --full-uri \
  --output tsv
# Output: https://mystorageacct12345.blob.core.windows.net/images/photo1.jpg?se=...&sp=r&sig=...
# Anyone with this URL can download the file for 1 hour
```

### Working with Blob Storage (Python SDK)

```python
# pip install azure-storage-blob azure-identity

from azure.storage.blob import BlobServiceClient, BlobClient, ContainerClient
from azure.identity import DefaultAzureCredential

# ============================================================
# CONNECT TO BLOB STORAGE
# ============================================================

# Option 1: Connection string (simple but less secure)
connection_string = "DefaultEndpointsProtocol=https;AccountName=mystorageacct12345;AccountKey=xxxxx"
blob_service_client = BlobServiceClient.from_connection_string(connection_string)

# Option 2: Managed Identity (recommended for production!)
# No credentials needed — Azure handles authentication
credential = DefaultAzureCredential()
blob_service_client = BlobServiceClient(
    account_url="https://mystorageacct12345.blob.core.windows.net",
    credential=credential
)

# ============================================================
# CREATE A CONTAINER
# ============================================================
container_client = blob_service_client.create_container("my-uploads")
print("Container 'my-uploads' created!")

# ============================================================
# UPLOAD A FILE
# ============================================================
blob_client = blob_service_client.get_blob_client(
    container="my-uploads",
    blob="documents/report.pdf"
)

# Upload from a local file
with open("./report.pdf", "rb") as data:
    blob_client.upload_blob(data, overwrite=True)
print("File uploaded successfully!")

# Upload a string as a blob
blob_client2 = blob_service_client.get_blob_client(
    container="my-uploads",
    blob="logs/app.log"
)
blob_client2.upload_blob("This is a log entry\n", overwrite=True)

# ============================================================
# DOWNLOAD A FILE
# ============================================================
blob_client = blob_service_client.get_blob_client(
    container="my-uploads",
    blob="documents/report.pdf"
)

# Download to a local file
with open("./downloaded-report.pdf", "wb") as download_file:
    download_data = blob_client.download_blob()
    download_file.write(download_data.readall())
print("File downloaded successfully!")

# Download as a string (for text files)
blob_client2 = blob_service_client.get_blob_client(
    container="my-uploads",
    blob="logs/app.log"
)
log_content = blob_client2.download_blob().readall().decode('utf-8')
print(f"Log contents: {log_content}")

# ============================================================
# LIST ALL BLOBS IN A CONTAINER
# ============================================================
container_client = blob_service_client.get_container_client("my-uploads")

print("Blobs in 'my-uploads':")
for blob in container_client.list_blobs():
    print(f"  - {blob.name} ({blob.size} bytes, tier: {blob.blob_tier})")

# List blobs with a prefix (like a "folder")
print("\nBlobs in 'documents/' folder:")
for blob in container_client.list_blobs(name_starts_with="documents/"):
    print(f"  - {blob.name}")

# ============================================================
# DELETE A BLOB
# ============================================================
blob_client = blob_service_client.get_blob_client(
    container="my-uploads",
    blob="logs/app.log"
)
blob_client.delete_blob()
print("Blob deleted!")

# ============================================================
# SET BLOB METADATA
# ============================================================
blob_client = blob_service_client.get_blob_client(
    container="my-uploads",
    blob="documents/report.pdf"
)
blob_client.set_blob_metadata(metadata={
    "author": "Ritesh",
    "department": "DevOps",
    "year": "2026"
})
print("Metadata set!")

# Read metadata
properties = blob_client.get_blob_properties()
print(f"Metadata: {properties.metadata}")
# Output: Metadata: {'author': 'Ritesh', 'department': 'DevOps', 'year': '2026'}
```

### Working with Blob Storage (AzCopy)

**AzCopy** is a command-line tool optimized for high-performance data transfers to/from Azure Storage. It's **much faster** than the Azure CLI for large transfers.

```bash
# Download AzCopy: https://learn.microsoft.com/en-us/azure/storage/common/storage-use-azcopy-v10

# Login with Azure AD
azcopy login

# ============================================================
# UPLOAD
# ============================================================

# Upload a single file
azcopy copy "./report.pdf" "https://mystorageacct12345.blob.core.windows.net/my-uploads/report.pdf"

# Upload an entire directory
azcopy copy "./my-folder" "https://mystorageacct12345.blob.core.windows.net/my-uploads/" --recursive

# Upload with SAS token
azcopy copy "./report.pdf" "https://mystorageacct12345.blob.core.windows.net/my-uploads/report.pdf?<SAS-TOKEN>"

# ============================================================
# DOWNLOAD
# ============================================================

# Download a single file
azcopy copy "https://mystorageacct12345.blob.core.windows.net/my-uploads/report.pdf" "./downloaded-report.pdf"

# Download an entire container
azcopy copy "https://mystorageacct12345.blob.core.windows.net/my-uploads/" "./downloaded/" --recursive

# ============================================================
# SYNC (like rsync — only copies changed files)
# ============================================================

# Sync local folder to blob container
azcopy sync "./my-folder" "https://mystorageacct12345.blob.core.windows.net/my-uploads/" --recursive

# Sync blob container to local folder
azcopy sync "https://mystorageacct12345.blob.core.windows.net/my-uploads/" "./my-folder" --recursive
```

### Blob Versioning & Soft Delete

**Blob Versioning** keeps previous versions of your blobs automatically. Every time a blob is modified, a new version is created.

**Soft Delete** keeps deleted blobs around for a specified retention period so you can recover them.

```bash
# Enable blob versioning
az storage account blob-service-properties update \
  --account-name mystorageacct12345 \
  --resource-group myResourceGroup \
  --enable-versioning true

# Enable soft delete (keep deleted blobs for 30 days)
az storage account blob-service-properties update \
  --account-name mystorageacct12345 \
  --resource-group myResourceGroup \
  --delete-retention-days 30 \
  --enable-delete-retention true

# Enable container soft delete (keep deleted containers for 7 days)
az storage account blob-service-properties update \
  --account-name mystorageacct12345 \
  --resource-group myResourceGroup \
  --container-delete-retention-days 7 \
  --enable-container-delete-retention true
```

**Why is this important?**
- Someone accidentally deletes a file → Recover it from soft delete
- Someone overwrites a file with bad data → Restore the previous version
- Ransomware encrypts your blobs → Restore from versioning

### Lifecycle Management Policies

Automatically move or delete blobs based on age. Save money without manual effort!

```bash
# Create a lifecycle policy (JSON)
cat > lifecycle-policy.json << 'EOF'
{
  "rules": [
    {
      "name": "move-to-cool-after-30-days",
      "enabled": true,
      "type": "Lifecycle",
      "definition": {
        "filters": {
          "blobTypes": ["blockBlob"],
          "prefixMatch": ["logs/"]
        },
        "actions": {
          "baseBlob": {
            "tierToCool": {
              "daysAfterModificationGreaterThan": 30
            },
            "tierToArchive": {
              "daysAfterModificationGreaterThan": 90
            },
            "delete": {
              "daysAfterModificationGreaterThan": 365
            }
          }
        }
      }
    }
  ]
}
EOF

# Apply the policy
az storage account management-policy create \
  --account-name mystorageacct12345 \
  --resource-group myResourceGroup \
  --policy @lifecycle-policy.json
```

**What this policy does:**
1. Blobs in the "logs/" prefix that are **30+ days old** → Move to **Cool** tier
2. Blobs **90+ days old** → Move to **Archive** tier
3. Blobs **365+ days old** → **Delete** them

This saves a LOT of money on log storage! 💰

### Blob Storage Best Practices

1. **Use managed identities** instead of connection strings/keys in applications
2. **Enable soft delete and versioning** — protect against accidental deletion
3. **Use lifecycle policies** — automatically move old data to cheaper tiers
4. **Disable public access** unless you specifically need it (static websites)

8. **Set appropriate access tiers** — don't pay Hot prices for data you access once a year
9. **Enable HTTPS only** and enforce TLS 1.2 minimum
10. **Use immutable storage** for compliance (WORM — Write Once, Read Many)

## 7. Azure Key Vault

### What is Azure Key Vault?

Azure Key Vault is a **cloud service for securely storing and managing secrets, encryption keys, and certificates**. 

Think of it as a **digital safe** for your sensitive data. Instead of putting passwords in config files or environment variables, you store them in Key Vault and let your applications retrieve them securely.

**Why do you need Key Vault?**
- ❌ Hardcoding secrets in code → **Security nightmare**
- ❌ Storing secrets in environment variables → **Better, but still visible**
- ❌ Storing secrets in config files → **Can be accidentally committed to Git**
- ✅ Storing secrets in Key Vault → **Centralized, audited, encrypted, access-controlled**

---

### What Can You Store in Key Vault?

| Type | What It Is | Example |
|------|-----------|---------|
| **Secrets** | Any string value you want to keep safe | Database passwords, API keys, connection strings |
| **Keys** | Cryptographic keys for encryption/signing | RSA keys, EC keys for data encryption |
| **Certificates** | SSL/TLS certificates | HTTPS certs for your web apps |

---

### Key Vault Tiers

| Tier | Description | HSM Type | Use Case |
|------|-------------|----------|----------|
| **Standard** | Software-protected keys | Software | Most use cases |
| **Premium** | HSM-protected keys | FIPS 140-2 Level 2 HSMs | Compliance, regulated industries |

> **HSM** = Hardware Security Module. A physical device that protects cryptographic keys. Premium tier uses dedicated hardware for key protection.

---

### Managing Secrets

```bash

# CREATE / SET A SECRET
# ============================================================

# Set a simple secret

  --vault-name mykeyvault12345 \
  --name "DatabasePassword" \
  --value "SuperSecretP@ssw0rd!"

  --name "ApiKey" \
  --value "sk-abc123xyz789" \

# Set a secret from a file (e.g., a cert or key file)

  --vault-name mykeyvault12345 \

# ============================================================
# READ / GET A SECRET
# ============================================================

# Get the latest version of a secret
az keyvault secret show \
  --vault-name mykeyvault12345 \

  --query "value" \

# LIST ALL SECRETS
# ============================================================

# ============================================================

# ============================================================

  --vault-name mykeyvault12345 \
  --name "OldApiKey"

# Permanently delete (purge) — CANNOT be recovered
az keyvault secret purge \
```

### Access Policies vs RBAC

There are two ways to control who can access your Key Vault:

**Option 1: Access Policies (Legacy)**

```bash
# Grant a user access to secrets
az keyvault set-policy \
  --name mykeyvault12345 \
  --upn user@company.com \
  --secret-permissions get list set delete

# Grant an app/service principal access
az keyvault set-policy \
  --name mykeyvault12345 \
  --spn <app-id> \
  --secret-permissions get list
```

**Option 2: Azure RBAC (Recommended!)**

```bash
# Grant a user the "Key Vault Secrets User" role (can read secrets)
az role assignment create \
  --role "Key Vault Secrets User" \
  --assignee user@company.com \
  --scope /subscriptions/<sub>/resourceGroups/<rg>/providers/Microsoft.KeyVault/vaults/mykeyvault12345

# Grant a managed identity the "Key Vault Secrets Officer" role (can manage secrets)
az role assignment create \
  --role "Key Vault Secrets Officer" \
  --assignee <managed-identity-principal-id> \
  --scope /subscriptions/<sub>/resourceGroups/<rg>/providers/Microsoft.KeyVault/vaults/mykeyvault12345
```

**Key Vault RBAC Roles:**

| Role | What It Can Do |
|------|---------------|
| Key Vault Administrator | Full access to everything |
| Key Vault Secrets User | Read secrets only |
| Key Vault Secrets Officer | Manage (CRUD) secrets |
| Key Vault Certificates Officer | Manage certificates |
| Key Vault Crypto Officer | Manage keys and crypto operations |
| Key Vault Reader | Read metadata (not secret values) |

> **Best Practice:** Always use **RBAC** over Access Policies. RBAC is more granular, supports conditions, and is consistent with the rest of Azure.

### Key Vault with App Service

You can reference Key Vault secrets directly in App Service **app settings** — no code changes needed!

```bash
# Step 1: Enable system-assigned managed identity on App Service
az webapp identity assign \
  --name my-webapp \
  --resource-group myResourceGroup

# Step 2: Grant the identity access to Key Vault
az role assignment create \
  --role "Key Vault Secrets User" \
  --assignee $(az webapp identity show --name my-webapp --resource-group myResourceGroup --query principalId -o tsv) \
  --scope /subscriptions/<sub>/resourceGroups/<rg>/providers/Microsoft.KeyVault/vaults/mykeyvault12345

# Step 3: Reference Key Vault secret in app settings
az webapp config appsettings set \
  --name my-webapp \
  --resource-group myResourceGroup \
  --settings "DB_PASSWORD=@Microsoft.KeyVault(SecretUri=https://mykeyvault12345.vault.azure.net/secrets/DatabasePassword/)"
```

**What happens:**
1. When App Service starts, it sees the `@Microsoft.KeyVault(...)` reference
2. It uses its managed identity to fetch the secret from Key Vault
3. The actual secret value is injected as an environment variable
4. Your code reads `process.env.DB_PASSWORD` as normal!

This is **the best approach** because:
- No secrets in your code or config files
- Secrets are centrally managed in Key Vault
- Automatic rotation — update the secret in Key Vault, restart the app

### Key Vault Best Practices

1. **Use RBAC** instead of access policies — more granular and consistent
2. **Use managed identities** for applications accessing Key Vault — no credentials to manage
3. **Enable purge protection** — prevents accidental permanent deletion
4. **Enable soft delete** (on by default) — 90-day recovery window
5. **Set expiration dates** on secrets — forces rotation
6. **Rotate secrets regularly** — automate with Azure Event Grid + Functions
7. **Use separate vaults** for different environments (dev, staging, prod)
8. **Enable diagnostic logging** — audit who accessed what secrets
9. **Use Key Vault references** in App Service — don't copy secrets to app settings
10. **Limit access** — use least-privilege principle. Not everyone needs access to secrets
11. **Enable network restrictions** — allow access only from specific VNets/IPs
12. **Don't store large data** in Key Vault — it's for secrets (max 25KB per secret), not file storage

## 8. Integration Patterns — Storage + Key Vault Together

### Pattern 1: Store Storage Account Key in Key Vault

```bash
# Get the storage account key
STORAGE_KEY=$(az storage account keys list \
  --name mystorageacct12345 \
  --resource-group myResourceGroup \
  --query "[0].value" -o tsv)

# Store it in Key Vault
az keyvault secret set \
  --vault-name mykeyvault12345 \
  --name "StorageAccountKey" \
  --value "$STORAGE_KEY"

# Now your apps fetch the key from Key Vault instead of hardcoding it
```

### Pattern 2: Customer-Managed Keys for Storage Encryption

Instead of Microsoft-managed encryption keys, use your **own key from Key Vault** to encrypt storage data:

```bash
# Create an encryption key in Key Vault
az keyvault key create \
  --vault-name mykeyvault12345 \
  --name "StorageEncryptionKey" \
  --kty RSA \
  --size 2048

# Configure storage account to use the Key Vault key
az storage account update \
  --name mystorageacct12345 \
  --resource-group myResourceGroup \
  --encryption-key-source Microsoft.Keyvault \
  --encryption-key-vault https://mykeyvault12345.vault.azure.net \
  --encryption-key-name StorageEncryptionKey
```

### Pattern 3: Auto-Rotate Storage Keys with Azure Functions

```
┌──────────────────┐     ┌──────────────────┐     ┌──────────────────┐
│   Event Grid     │     │  Azure Function  │     │   Key Vault      │
│  (Key Expiry     │────►│ (Rotates the key │────►│ (Stores new key) │
│   Notification)  │     │  in Storage Acct) │     │                  │
└──────────────────┘     └──────────────────┘     └──────────────────┘
```

1. Key Vault sends a "secret nearing expiry" event via Event Grid
2. Azure Function triggers and rotates the storage account key
3. Function stores the new key back in Key Vault
4. Apps seamlessly use the new key

| **Secrets** | Store passwords, API keys, connection strings |
| **Keys** | Store and manage encryption/signing keys |
| **Certificates** | Store and auto-renew SSL/TLS certificates |
| **Access** | Use RBAC + Managed Identities (not access policies) |
| **Integration** | Works with App Service, AKS, Functions, VMs, and more |

### Quick Decision Guide

```
Need to store files/images/videos?
  → Azure Blob Storage

Need a network file share?
  → Azure Files

Need a message queue?
  → Azure Queue Storage (simple) or Service Bus (enterprise)

Need to store structured data cheaply?
  → Azure Table Storage (or Cosmos DB for more features)

Need to store passwords, API keys, certificates?
  → Azure Key Vault (always!)

Where should my storage encryption keys live?
  → Azure Key Vault (customer-managed keys)
```
