## 1. Introduction — Why Monitoring Matters

Imagine you deploy an application to production. Users start complaining it's slow. Or worse — it's down and nobody knows. That's a nightmare.

**Monitoring answers three critical questions:**

1. **Is my app working?** → Availability monitoring
2. **How is my app performing?** → Performance monitoring
3. **What went wrong?** → Troubleshooting and diagnostics

```
Without Monitoring:                  With Monitoring:
                                     
"Is the app working?"                ✅ Uptime: 99.97%
"I don't know... let me check"       ✅ Avg response: 120ms
                                     ✅ Error rate: 0.02%
"The app is slow!"                   ⚠️  Alert: CPU > 80%
"Which part? No idea..."             📊 Dashboard: App Service CPU graph
                                     🔍 Query: Top 10 slowest requests
"It's broken, what happened?"        
"Check the logs... wait, where       📋 Log Analytics: Full logs,
 are the logs?"                         searchable with KQL
```

Azure gives you a **complete monitoring platform** with three main components:

| Component | What It Does |
|-----------|-------------|
| **Azure Monitor** | The umbrella platform — collects metrics, logs, and alerts |
| **Log Analytics** | A powerful log database where you query logs using KQL |
| **Application Insights** | Deep application performance monitoring (APM) for your code |

## 2. Azure Monitor — The Big Picture

### What is Azure Monitor?

Azure Monitor is the **central monitoring platform** for everything in Azure. It collects data from your applications, infrastructure, network, and Azure platform itself, and gives you tools to analyze, visualize, and act on that data.

Think of Azure Monitor as the **control center** — it collects all monitoring data and connects to all the specialized tools.

**What Azure Monitor Can Monitor:**

| Layer | What You Monitor | Example |
|-------|-----------------|---------|
| **Application** | Your code, APIs, requests | Response times, error rates, dependencies |
| **Guest OS** | VM operating system | CPU, memory, disk usage, OS logs |
| **Azure Resource** | Azure services you use | App Service health, Storage latency, DB throughput |
| **Subscription** | Azure subscription events | Service health, resource changes |
| **Tenant** | Azure AD, tenant-wide | Sign-in logs, audit logs |

### Data Types — Metrics vs Logs

Azure Monitor collects two fundamental types of data:

| | Metrics | Logs |
|---|---------|------|
| **What** | Numerical values at regular intervals | Text records with rich structured data |
| **Format** | Time-series (timestamp + number) | Structured records (JSON-like) |
| **Speed** | Near real-time (1 minute granularity) | Can have slight delay (depends on config) |
| **Retention** | 93 days (automatic) | Configurable (30 days to 2 years) |
| **Query Language** | Metrics Explorer (GUI) or REST API | KQL (Kusto Query Language) |
| **Cost** | Free (platform metrics) | Based on data ingested (GB/day) |
| **Example** | CPU = 72% at 2:30 PM | "User john logged in from IP 10.0.0.5 at 2:30 PM" |
| **Best For** | Real-time dashboards, alerts on thresholds | Deep analysis, troubleshooting, auditing |

**Simple analogy:**
- **Metrics** = The speedometer and fuel gauge in your car (numbers, real-time)
- **Logs** = The car's trip log and maintenance records (detailed text records)

### Azure Monitor Metrics

Metrics are **numerical values collected at regular intervals**. Every Azure resource automatically emits platform metrics.

**Common Metrics:**

| Resource | Metric | What It Measures |
|----------|--------|-----------------|
| VM | Percentage CPU | CPU utilization |
| VM | Available Memory Bytes | Free RAM |
| App Service | Http Server Errors | Count of 5xx errors |
| App Service | Response Time | Average response time |
| App Service | Requests | Total request count |
| Storage | Transactions | Number of storage operations |
| SQL Database | DTU Percentage | Database load |
| AKS | Node CPU Utilization | Kubernetes node CPU |

```bash
# View metrics for an App Service
az monitor metrics list \
  --resource /subscriptions/<sub>/resourceGroups/<rg>/providers/Microsoft.Web/sites/my-webapp \
  --metric "HttpServerErrors" \
  --interval PT1H \
  --output table
# --interval PT1H → 1-hour granularity
# --interval PT5M → 5-minute granularity
# --interval PT1M → 1-minute granularity

# View CPU metrics for a VM
az monitor metrics list \
  --resource /subscriptions/<sub>/resourceGroups/<rg>/providers/Microsoft.Compute/virtualMachines/my-vm \
  --metric "Percentage CPU" \
  --interval PT5M \
  --start-time 2026-05-30T00:00:00Z \
  --end-time 2026-05-30T12:00:00Z \
  --output table

# List all available metrics for a resource
az monitor metrics list-definitions \
  --resource /subscriptions/<sub>/resourceGroups/<rg>/providers/Microsoft.Web/sites/my-webapp \
  --output table
```

**Metrics Explorer (Portal):**

The best way to explore metrics is through the Azure Portal:
1. Go to your resource → **Metrics**
2. Select a metric (e.g., "CPU Percentage")
3. Choose aggregation (Avg, Sum, Max, Min, Count)
4. Set the time range
5. Add more metrics to the same chart
6. Pin to a dashboard!

### Alerts

Alerts let you get **notified when something goes wrong** — before your users notice.

**Types of Alerts:**

| Alert Type | Triggered By | Example |
|------------|-------------|---------|
| **Metric Alert** | A metric crosses a threshold | CPU > 80% for 5 minutes |
| **Log Alert** | A log query returns results | Error count > 100 in last hour |
| **Activity Log Alert** | An Azure activity occurs | Someone deleted a resource |
| **Smart Detection** | AI detects anomalies | Sudden spike in failure rate |

**Creating Metric Alerts (Azure CLI):**

```bash
# Alert when CPU > 80% for 5 minutes
az monitor metrics alert create \
  --name "High-CPU-Alert" \
  --resource-group myResourceGroup \
  --scopes /subscriptions/<sub>/resourceGroups/<rg>/providers/Microsoft.Web/sites/my-webapp \
  --condition "avg Percentage CPU > 80" \
  --window-size 5m \
  --evaluation-frequency 1m \
  --severity 2 \
  --description "CPU usage is above 80%" \
  --action myActionGroup
# --window-size 5m         → Look at the last 5 minutes
# --evaluation-frequency 1m → Check every 1 minute
# --severity 0-4           → 0=Critical, 1=Error, 2=Warning, 3=Informational, 4=Verbose

# Alert when HTTP 5xx errors > 10
az monitor metrics alert create \
  --name "High-Error-Rate" \
  --resource-group myResourceGroup \
  --scopes /subscriptions/<sub>/resourceGroups/<rg>/providers/Microsoft.Web/sites/my-webapp \
  --condition "total Http5xx > 10" \
  --window-size 5m \
  --evaluation-frequency 1m \
  --severity 1 \
  --action myActionGroup

# Alert when response time > 3 seconds
az monitor metrics alert create \
  --name "Slow-Response-Alert" \
  --resource-group myResourceGroup \
  --scopes /subscriptions/<sub>/resourceGroups/<rg>/providers/Microsoft.Web/sites/my-webapp \
  --condition "avg HttpResponseTime > 3" \
  --window-size 5m \
  --evaluation-frequency 1m \
  --severity 2 \
  --action myActionGroup
```

**Creating Log-Based Alerts:**

```bash
# Alert when there are more than 50 errors in the last hour
az monitor scheduled-query create \
  --name "Error-Spike-Alert" \
  --resource-group myResourceGroup \
  --scopes /subscriptions/<sub>/resourceGroups/<rg>/providers/Microsoft.OperationalInsights/workspaces/myWorkspace \
  --condition "count 'Heartbeat' > 50" \
  --condition-query "AppExceptions | where TimeGenerated > ago(1h)" \
  --evaluation-frequency 5m \
  --window-size 1h \
  --severity 1 \
  --action-groups /subscriptions/<sub>/resourceGroups/<rg>/providers/Microsoft.Insights/actionGroups/myActionGroup
```

**Alert Severity Levels:**

| Severity | Level | Use For |
|----------|-------|---------|
| 0 | Critical | System is down, immediate action needed |
| 1 | Error | Major functionality broken |
| 2 | Warning | Something could become a problem |
| 3 | Informational | FYI, no action needed |
| 4 | Verbose | Detailed info, usually for debugging |

### Action Groups

An **Action Group** defines **what happens when an alert fires**. It's the "do this when X happens" part.

```bash
# Create an action group
az monitor action-group create \
  --name myActionGroup \
  --resource-group myResourceGroup \
  --short-name "OpsTeam" \
  --action email devops-team devops@company.com \
  --action sms oncall-sms 1 5551234567 \
  --action webhook pagerduty "https://events.pagerduty.com/integration/xxx/enqueue"

# Add a Logic App or Azure Function action
az monitor action-group update \
  --name myActionGroup \
  --resource-group myResourceGroup \
  --add-action azurefunction auto-remediate <function-app-resource-id> <function-name> <http-trigger-url>
```

**Available Action Types:**

| Action | What It Does |
|--------|-------------|
| **Email** | Send email to a person or group |
| **SMS** | Send text message |
| **Voice** | Automated phone call |
| **Push Notification** | Azure mobile app notification |
| **Webhook** | Call an HTTP endpoint (PagerDuty, Slack, custom) |
| **Azure Function** | Run a function (auto-remediation!) |
| **Logic App** | Trigger a Logic App workflow |
| **ITSM** | Create a ticket in ServiceNow, etc. |
| **Automation Runbook** | Run an Azure Automation runbook |

**Example: Send Alerts to Slack:**

```bash
# Create a Slack incoming webhook URL first (from Slack admin)
# Then add it as an action

az monitor action-group create \
  --name slack-alerts \
  --resource-group myResourceGroup \
  --short-name "Slack" \
  --action webhook slack-notify "https://hooks.slack.com/services/T00000000/B00000000/XXXXXXXXXXXXXXXXXXXXXXXX"
```

### Autoscale with Azure Monitor

Azure Monitor powers **autoscale** for resources like App Service, VMSS, and more.

```bash
# Create autoscale settings for an App Service Plan
az monitor autoscale create \
  --name my-autoscale \
  --resource-group myResourceGroup \
  --resource myAppServicePlan \
  --resource-type Microsoft.Web/serverfarms \
  --min-count 2 \
  --max-count 10 \
  --count 3

# Scale out when CPU > 70%
az monitor autoscale rule create \
  --resource-group myResourceGroup \
  --autoscale-name my-autoscale \
  --condition "CpuPercentage > 70 avg 5m" \
  --scale out 2
# Adds 2 more instances when avg CPU > 70% over 5 minutes

# Scale in when CPU < 30%
az monitor autoscale rule create \
  --resource-group myResourceGroup \
  --autoscale-name my-autoscale \
  --condition "CpuPercentage < 30 avg 10m" \
  --scale in 1

# Schedule-based scaling (e.g., scale up during business hours)
az monitor autoscale profile create \
  --autoscale-name my-autoscale \
  --resource-group myResourceGroup \
  --name "business-hours" \
  --min-count 5 \
  --max-count 20 \
  --count 10 \
  --recurrence week "Monday Tuesday Wednesday Thursday Friday" \
  --start 08:00 \
  --end 18:00 \
  --timezone "Eastern Standard Time"
```

## 3. Log Analytics

### What is Log Analytics?

Log Analytics is a **powerful log database and query engine** in Azure. It's where all your logs from Azure Monitor end up, and you query them using **KQL (Kusto Query Language)**.

Think of it as:
- **Storage** → A database that stores all your logs
- **Query Engine** → A way to search, filter, and analyze those logs
- **Visualization** → Built-in charts and tables

**Simple analogy:** If Azure Monitor is the security camera system, Log Analytics is the **recording device + search tool** where you can go back and find exactly what happened at any point in time.

---

### Log Analytics Workspace

A **workspace** is the logical container for your log data. All logs from various sources are sent to a workspace.

```

│              Log Analytics Workspace                         │

│                                                             │

│  │ VM Logs   │ │ App Svc   │ │ K8s Logs  │ │ Custom     │  │
│  │           │ │ Logs      │ │           │ │ Logs       │  │
│  └───────────┘ └───────────┘ └───────────┘ └────────────┘  │
│                                                             │
│  ┌───────────────────────────────────────────────────────┐  │
│  │                   KQL Query Engine                     │  │
│  │  "Show me all errors from the last 24 hours"          │  │
```

**Best Practice:** For most organizations, use **one workspace per environment** (or even one workspace for everything). It's easier to correlate data when everything is in one place.

---

### Creating a Workspace (Azure CLI)

```bash
# Create a Log Analytics Workspace
az monitor log-analytics workspace create \
  --resource-group myResourceGroup \
  --workspace-name myLogWorkspace \
  --location eastus \
  --retention-time 90 \
  --sku PerGB2018
# --retention-time 90 → Keep logs for 90 days (default: 30, max: 730)
# --sku PerGB2018     → Pay per GB of data ingested (most common)

# Get the workspace ID (needed for connecting data sources)
az monitor log-analytics workspace show \
  --resource-group myResourceGroup \
  --workspace-name myLogWorkspace \
  --query "customerId" \
  --output tsv

# Get the workspace key (needed for agents)
az monitor log-analytics workspace get-shared-keys \
  --resource-group myResourceGroup \
  --workspace-name myLogWorkspace \
  --output table
```

### KQL — Kusto Query Language

**KQL (Kusto Query Language)** is the query language used in Log Analytics. It's like SQL but designed specifically for log data and time-series analysis.

**Why KQL is great:**
- Very readable and intuitive
- Powerful for time-based analysis
- Built-in visualization (charts, graphs)
- Used across Azure Monitor, Application Insights, Microsoft Sentinel, Azure Data Explorer

**KQL vs SQL — Quick Comparison:**

| SQL | KQL |
|-----|-----|
| `SELECT * FROM Users WHERE age > 25` | `Users \| where age > 25` |
| `SELECT COUNT(*) FROM Logs GROUP BY Level` | `Logs \| summarize count() by Level` |
| `SELECT TOP 10 * FROM Logs ORDER BY Time DESC` | `Logs \| top 10 by Time desc` |

**Key difference:** In KQL, you start with a **table name** and pipe (`|`) data through operators, like a pipeline.

---

### KQL Basics — Your First Queries

Open **Log Analytics** in the Azure Portal (Monitor → Logs, or go to your workspace → Logs).

```kusto
// ============================================================
// BASIC QUERIES
// ============================================================

// 1. See all data in a table (first 50 rows by default)
Heartbeat

// 2. Filter rows with "where"
Heartbeat
| where TimeGenerated > ago(1h)
// ago(1h) = 1 hour ago. Also: ago(30m), ago(1d), ago(7d)

// 3. Select specific columns with "project"
Heartbeat
| where TimeGenerated > ago(1h)
| project TimeGenerated, Computer, OSType, Version

// 4. Count rows
Heartbeat
| where TimeGenerated > ago(24h)
| count
// Output: Count = 1,234

// 5. Limit results with "take" (random sample) or "top" (ordered)
Heartbeat
| take 10
// Returns any 10 rows (random)

Heartbeat
| top 10 by TimeGenerated desc
// Returns 10 most recent rows

// 6. Sort results
Heartbeat
| where TimeGenerated > ago(1h)
| sort by TimeGenerated desc

// 7. Search across all columns (powerful!)
search "error"
// Searches for "error" in ALL tables and ALL columns

search in (Syslog, Event) "connection refused"
// Search only in Syslog and Event tables

// 8. Distinct/unique values
Heartbeat
| distinct Computer
// List all unique computer names

// 9. Rename columns
Heartbeat
| where TimeGenerated > ago(1h)
| project Timestamp = TimeGenerated, Server = Computer, OS = OSType
```

### KQL Intermediate — Filtering, Aggregation & Joins

```kusto
// ============================================================
// FILTERING
// ============================================================

// String comparison (case-sensitive by default)
Syslog
| where SeverityLevel == "err"

// Case-insensitive comparison
Syslog
| where SeverityLevel =~ "ERR"

// Contains (substring search)
Syslog
| where SyslogMessage contains "connection"

// Starts with / ends with
Syslog
| where SyslogMessage startswith "Failed"
| where Computer endswith ".local"

// Multiple conditions
Syslog
| where SeverityLevel == "err" and Facility == "auth"
| where TimeGenerated > ago(24h)

// In operator (like SQL IN)
Syslog
| where SeverityLevel in ("err", "crit", "alert", "emerg")

// Not equal
Syslog
| where SeverityLevel != "info"

// Regex matching
Syslog
| where SyslogMessage matches regex @"error|fail|timeout"

// ============================================================
// AGGREGATION (summarize)
// ============================================================

// Count events by severity
Syslog
| where TimeGenerated > ago(24h)
| summarize Count = count() by SeverityLevel
| sort by Count desc
// Output:
// SeverityLevel  Count
// info           45,000
// warning        2,300
// err            180
// crit           5

// Count by time bucket (time-series)
Syslog
| where TimeGenerated > ago(24h)
| summarize Count = count() by bin(TimeGenerated, 1h)
// Counts events per hour — great for trend graphs!

// Average, Min, Max, Sum
Perf
| where ObjectName == "Processor" and CounterName == "% Processor Time"
| where TimeGenerated > ago(1h)
| summarize 
    AvgCPU = avg(CounterValue),
    MaxCPU = max(CounterValue),
    MinCPU = min(CounterValue)
    by Computer
| sort by AvgCPU desc

// Count distinct values
Syslog
| where TimeGenerated > ago(24h)
| summarize UniqueServers = dcount(Computer)

// Percentiles
Perf
| where CounterName == "% Processor Time"
| summarize 
    p50 = percentile(CounterValue, 50),
    p90 = percentile(CounterValue, 90),
    p99 = percentile(CounterValue, 99)
    by Computer

// ============================================================
// JOINS
// ============================================================

// Join Heartbeat with Perf to get OS info alongside performance data
Perf
| where CounterName == "% Processor Time"
| where TimeGenerated > ago(1h)
| summarize AvgCPU = avg(CounterValue) by Computer
| join kind=inner (
    Heartbeat
    | where TimeGenerated > ago(1h)
    | distinct Computer, OSType
) on Computer
| project Computer, OSType, AvgCPU
| sort by AvgCPU desc

// ============================================================
// EXTENDING DATA (calculated columns)
// ============================================================

// Add a new calculated column
Perf
| where CounterName == "% Processor Time"
| extend CPUStatus = iff(CounterValue > 80, "HIGH", "Normal")
| project TimeGenerated, Computer, CounterValue, CPUStatus

// Parse structured data from strings
Syslog
| where SyslogMessage contains "login"
| extend User = extract(@"user=(\w+)", 1, SyslogMessage)
| extend IP = extract(@"from (\d+\.\d+\.\d+\.\d+)", 1, SyslogMessage)
| project TimeGenerated, User, IP
```

### Log Analytics Best Practices

1. **Start every query with a time filter** — `| where TimeGenerated > ago(1h)`. Without it, you scan ALL data (expensive and slow).
2. **Filter early, aggregate late** — Put `where` clauses before `summarize` to reduce data processed.
3. **Use `project` to select only columns you need** — reduces data transferred.
4. **Use `take` for exploration, `top` for ordered results** — `take` is random, `top` is sorted.
5. **Set appropriate retention periods** — 30 days for dev, 90-365 for production, based on compliance needs.
6. **Use Data Collection Rules** to control what data is collected (don't collect everything).
7. **Create saved queries** — save queries you run frequently for reuse.
8. **Use workbooks** for complex dashboards — more powerful than pinned charts.
9. **Set daily caps** to avoid surprise costs — especially in dev/test environments.
10. **Archive old data** to cheaper storage if you need long-term retention but rare access.

```bash
# Set a daily data ingestion cap (avoid surprise costs)
az monitor log-analytics workspace update \
  --resource-group myResourceGroup \
  --workspace-name myLogWorkspace \
  --quota 5
# --quota 5 → Cap at 5 GB/day

# Update retention period
az monitor log-analytics workspace update \
  --resource-group myResourceGroup \
  --workspace-name myLogWorkspace \
  --retention-time 180
# Keep data for 180 days
```

## 4. Application Insights

### What is Application Insights?

Application Insights is an **Application Performance Management (APM)** service. While Azure Monitor and Log Analytics monitor infrastructure, Application Insights monitors **your application code**.

It answers questions like:
- How many requests is my app handling?
- What's the average response time?
- Which endpoints are the slowest?
- What exceptions are being thrown?
- Which dependencies (database, API, cache) are slow?
- Are my users having a good experience?

**Think of it this way:**
- **Azure Monitor** = Monitors the car engine (infrastructure)
- **Application Insights** = Monitors the driver experience (application performance)

### Instrumenting a Node.js App

```javascript
// Step 1: Install the SDK
// npm install applicationinsights

// Step 2: Initialize at the VERY TOP of your app (before any other imports!)
// app.js or index.js

const appInsights = require('applicationinsights');

// Initialize with connection string
appInsights.setup(process.env.APPLICATIONINSIGHTS_CONNECTION_STRING)
    .setAutoCollectRequests(true)          // Track incoming HTTP requests
    .setAutoCollectPerformance(true)       // Track performance counters
    .setAutoCollectExceptions(true)        // Track unhandled exceptions
    .setAutoCollectDependencies(true)      // Track outgoing HTTP, SQL, etc.
    .setAutoCollectConsole(true)           // Track console.log() as traces
    .setUseDiskRetryCaching(true)          // Cache telemetry on disk if upload fails
    .setSendLiveMetrics(true)              // Enable Live Metrics Stream
    .start();

// Step 3: Get the default client for custom telemetry
const client = appInsights.defaultClient;

// ============================================================
// Your normal Express app
// ============================================================
const express = require('express');
const app = express();

app.get('/api/users', async (req, res) => {
    try {
        // Application Insights automatically tracks this request!
        
        // Track a custom event
        client.trackEvent({
            name: "UserListViewed",
            properties: { 
                userId: req.query.userId,
                page: req.query.page || "1"
            }
        });

        // Track a custom metric
        client.trackMetric({
            name: "UsersRequested",
            value: 1
        });

        const users = await getUsers(); // Auto-tracked as a dependency
        res.json(users);
    } catch (error) {
        // Track the exception manually (auto-collection also catches unhandled ones)
        client.trackException({
            exception: error,
            properties: { endpoint: "/api/users" }
        });
        res.status(500).json({ error: "Internal server error" });
    }
});

app.get('/api/orders/:id', async (req, res) => {
    // Track a trace (log message)
    client.trackTrace({
        message: `Order ${req.params.id} requested`,
        severity: appInsights.Contracts.SeverityLevel.Information
    });

    const order = await getOrder(req.params.id);
    res.json(order);
});

app.listen(3000, () => {
    console.log('Server running on port 3000');
});
```

```kusto
| where timestamp > ago(24h)
| summarize 
    Count = count(),
    AvgDuration = avg(duration),
    P95Duration = percentile(duration, 95),
    FailRate = countif(success == false) * 100.0 / count()
    by name
| sort by Count desc
| render table

// Failed requests over time
requests
| where timestamp > ago(24h)
| where success == false
| summarize FailedRequests = count() by bin(timestamp, 1h)
| render timechart

// Response time distribution (histogram)
requests
| where timestamp > ago(24h)
| summarize count() by bin(duration, 100)
| render barchart

// ============================================================
// EXCEPTIONS
// ============================================================

// Top exceptions
exceptions
| where timestamp > ago(24h)
| summarize Count = count() by type, outerMessage
| sort by Count desc

// Exception details with stack trace
exceptions
| where timestamp > ago(1h)
| where type contains "NullReference" or type contains "TypeError"
| project timestamp, type, outerMessage, innermostMessage, 
    details[0].parsedStack
| take 20

// ============================================================
// DEPENDENCIES
// ============================================================

// Slowest external dependencies
dependencies
| where timestamp > ago(24h)
| summarize 
    AvgDuration = avg(duration),
    P95 = percentile(duration, 95),
    FailRate = countif(success == false) * 100.0 / count(),
    Count = count()
    by target, type, name
| sort by AvgDuration desc

// Database query performance
dependencies
| where type == "SQL"
| where timestamp > ago(24h)
| summarize 
    AvgDuration = avg(duration),
    MaxDuration = max(duration),
    Count = count()
    by name
| sort by AvgDuration desc

// Failed dependencies
dependencies
| where success == false
| where timestamp > ago(24h)
| summarize FailCount = count() by target, type, resultCode
| sort by FailCount desc

// ============================================================
// END-TO-END TRANSACTION — Trace a single request
// ============================================================

// Find a specific request and all related telemetry
let operationId = "abc123";  // Get this from a failed request
union requests, dependencies, exceptions, traces
| where operation_Id == operationId
| sort by timestamp asc
| project timestamp, itemType, name, duration, success, 
    resultCode, message

// ============================================================
// AVAILABILITY
// ============================================================

// Availability test results
availabilityResults
| where timestamp > ago(7d)
| summarize 
    Availability = countif(success == true) * 100.0 / count()
    by name, bin(timestamp, 1d)
| render timechart

// Failed availability tests
availabilityResults
| where success == false
| where timestamp > ago(24h)
| project timestamp, name, location, message, duration
| sort by timestamp desc
```

### Application Insights Best Practices

1. **Use workspace-based Application Insights** (connects to Log Analytics) — not classic
2. **Set sampling rate** for high-traffic apps — don't collect 100% of requests
3. **Use connection strings** (not just Instrumentation Key) — connection strings are newer and more flexible
4. **Track custom events** for business metrics — not just technical metrics
5. **Use correlation IDs** — Application Insights auto-correlates requests → dependencies → exceptions
6. **Set up availability tests** — know when your app is down before users tell you
7. **Use Live Metrics** during deployments — watch for problems in real-time
8. **Review the Application Map** regularly — spot architecture issues and bottlenecks
9. **Create alerts on failure rate** — get notified when error rate spikes
10. **Use Smart Detection** — it's free and catches issues automatically
11. **Don't log sensitive data** — PII, passwords, tokens should never appear in telemetry
12. **Set up daily caps** to control costs — especially in development

```bash
# Set sampling rate (e.g., collect 25% of telemetry)
az webapp config appsettings set \
  --name my-webapp \
  --resource-group myResourceGroup \
  --settings APPLICATIONINSIGHTS_SAMPLING_PERCENTAGE=25

# Set daily data cap for Application Insights
az monitor app-insights component billing update \
  --app my-app-insights \
  --resource-group myResourceGroup \
  --cap 5 \
  --stop-on-cap true
# --cap 5 → Max 5 GB/day
# --stop-on-cap → Stop collecting when cap is reached
```

### Cost Management for Monitoring

Monitoring costs come from **data ingestion** into Log Analytics. Here's how to keep costs under control:

```bash
# Check current data ingestion volume
az monitor log-analytics workspace show \
  --resource-group myResourceGroup \
  --workspace-name myLogWorkspace \
  --query "sku" \
  --output table

# Set a daily cap
az monitor log-analytics workspace update \
  --resource-group myResourceGroup \
  --workspace-name myLogWorkspace \
  --quota 10
# 10 GB/day cap
```

**Cost-Saving Tips:**

| Tip | Impact |
|-----|--------|
| Set daily caps on workspaces | Prevents unexpected charges |
| Use sampling in Application Insights | Reduce data by 50-90% |
| Reduce retention period | 30 days for dev, 90 for prod |
| Use Data Collection Rules to filter | Only collect what you need |
| Archive old data | Move to cheaper archive storage |
| Use Basic Logs tier | Cheaper for high-volume, rarely queried data |
| Commit to usage tier | Discounts for 100+ GB/day |

```kusto
// Find which tables are consuming the most data
Usage
| where TimeGenerated > ago(30d)
| summarize DataGB = sum(Quantity) / 1024 by DataType
| sort by DataGB desc
| render barchart
// This shows you WHERE your data costs come from
```

## 6. Comparison — Azure Monitor vs Other Tools

| Feature | Azure Monitor + App Insights | Datadog | New Relic | Prometheus + Grafana |
|---------|------------------------------|---------|-----------|---------------------|
| **Type** | Cloud-native (Azure) | SaaS | SaaS | Open-source (self-hosted) |
| **APM** | Application Insights | Datadog APM | New Relic APM | N/A (need Jaeger/Tempo) |
| **Logs** | Log Analytics (KQL) | Log Management | Logs | Loki |
| **Metrics** | Azure Monitor Metrics | Metrics | Metrics | Prometheus |
| **Dashboards** | Azure Dashboards + Grafana | Datadog Dashboards | New Relic Dashboards | Grafana |
| **Alerting** | Azure Alerts | Monitors | Alerts | Alertmanager |
| **Cost** | Pay-per-GB ingested | Per host/GB | Per GB | Free (+ hosting costs) |
| **Best for** | Azure-native workloads | Multi-cloud, comprehensive | Modern apps, full stack | Kubernetes, cost-sensitive |
| **Query Language** | KQL | Datadog Query Language | NRQL | PromQL |

**When to use Azure Monitor:**
- You're running workloads primarily on Azure
- You want a single pane of glass for Azure resources
- You need deep integration with Azure services
- You want to avoid managing monitoring infrastructure

## 7. Summary

### Quick Reference

| Component | What It Does | Key Feature |
|-----------|-------------|-------------|
| **Azure Monitor** | Central monitoring platform | Collects metrics + logs from everything |
| **Metrics** | Numerical time-series data | Real-time dashboards, threshold alerts |
| **Alerts** | Notifications when things go wrong | Email, SMS, Slack, PagerDuty, auto-remediation |
| **Log Analytics** | Log database + KQL query engine | Deep analysis, troubleshooting, auditing |
| **KQL** | Query language for logs | Powerful, readable, built-in visualization |
| **Application Insights** | Application Performance Management | Request tracking, dependency mapping, exceptions |
| **Application Map** | Visual topology of your app | See bottlenecks at a glance |
| **Live Metrics** | Real-time telemetry stream | Watch during deployments |
| **Availability Tests** | Synthetic monitoring | Know when your app is down before users |
| **Smart Detection** | AI-powered anomaly detection | Auto-detects performance degradation |

### The Golden Monitoring Rules

1. **Monitor everything** — infrastructure, application, and business metrics
2. **Alert on symptoms, not causes** — alert on "error rate > 5%", not "CPU > 80%"
3. **Use dashboards** — make health visible to the whole team
4. **Log strategically** — too few logs = can't debug, too many = expensive
5. **Set up availability tests** — be the first to know when things break
6. **Review Application Map** — understand your architecture's weak points
7. **Use KQL** — it's your superpower for troubleshooting
8. **Control costs** — set caps, use sampling, archive old data
