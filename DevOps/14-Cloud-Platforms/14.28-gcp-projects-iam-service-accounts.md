# 14.28 — GCP — Projects, IAM, Service Accounts

> **One Line Summary:** Google Cloud Platform organizes everything inside **Projects**, controls who can do what with **IAM (Identity and Access Management)**, and lets applications authenticate using **Service Accounts** — these three things are the foundation of everything you'll ever do in GCP.

---

## 🤔 What is Google Cloud Platform (GCP)?

Google Cloud Platform is Google's cloud computing service — just like AWS is Amazon's and Azure is Microsoft's.

GCP gives you:
- Virtual machines (servers)
- Databases
- Kubernetes clusters
- Storage
- Networking
- Machine learning tools
- And hundreds more services

But before you use ANY of these services, you need to understand three fundamental things:

1. **Projects** — How GCP organizes your stuff
2. **IAM** — Who is allowed to do what
3. **Service Accounts** — How your applications authenticate

Let's dive into each one. 🚀

---

## 🗂️ Part 1: GCP Projects

### What is a Project?

A **Project** is the most fundamental organizing unit in GCP. Think of it as a **container** that holds all your cloud resources.

Every single thing you create in GCP — a VM, a database, a Kubernetes cluster, a storage bucket — lives inside a Project.

```
Google Cloud
└── Organization (your company)
    ├── Folder: "Production"
    │   ├── Project: "prod-web-app"
    │   │   ├── VM Instance
    │   │   ├── Cloud SQL Database
    │   │   └── GKE Cluster
    │   └── Project: "prod-api"
    │       ├── Cloud Run Service
    │       └── Cloud Storage Bucket
    ├── Folder: "Staging"
    │   └── Project: "staging-web-app"
    │       ├── VM Instance
    │       └── Cloud SQL Database
    └── Folder: "Development"
        └── Project: "dev-playground"
            └── VM Instance
```

### Why Do We Need Projects?

| Reason | Explanation |
|--------|-------------|
| **Organization** | Keep different apps/environments separate |
| **Billing** | Each project can have its own billing. You know exactly what each project costs |
| **Access Control** | You can give different people access to different projects |
| **Quotas & Limits** | Each project has its own resource quotas |
| **Isolation** | Resources in one project can't accidentally mess with another project |

### Real-World Example

Imagine your company has:
- A **web application** running in production
- A **staging environment** for testing
- A **development sandbox** for devs to experiment

You'd create **3 separate projects** so they're completely isolated from each other. A developer breaking something in the dev project won't affect production.

---

### Project Identifiers — Every Project Has Three

When you create a project, it gets three identifiers:

| Identifier | What It Is | Example | Can Change? |
|-----------|-----------|---------|-------------|
| **Project Name** | Human-friendly name (for display) | "My Web App" | ✅ Yes |
| **Project ID** | Globally unique ID (used in commands/APIs) | `my-web-app-12345` | ❌ No (set once) |
| **Project Number** | Auto-generated number by Google | `415104042345` | ❌ No (auto-assigned) |

> **Important:** The **Project ID** is what you'll use most. It's globally unique across ALL of Google Cloud (no two projects in the world can have the same ID). Choose it wisely because you can't change it later!

### Creating a Project

#### Using the Console (Web UI):
1. Go to [console.cloud.google.com](https://console.cloud.google.com)
2. Click the project dropdown at the top
3. Click "New Project"
4. Enter a name and project ID
5. Select billing account and organization (if applicable)
6. Click "Create"

#### Using the gcloud CLI:

First, install the Google Cloud SDK (gcloud CLI):

```bash
# On Ubuntu/Debian
curl https://sdk.cloud.google.com | bash
exec -l $SHELL
gcloud init

# On macOS
brew install google-cloud-sdk

# On Windows — download the installer from:
# https://cloud.google.com/sdk/docs/install
```

Now create a project:

```bash
# Create a new project
gcloud projects create my-web-app-2024 \
    --name="My Web App" \
    --organization=123456789

# What this does:
# gcloud projects create  → creates a new project
# my-web-app-2024         → the Project ID (globally unique, can't change later)
# --name="My Web App"     → the display name (can change later)
# --organization=123456   → your organization ID (optional)
```

```bash
# List all your projects
gcloud projects list

# Output:
# PROJECT_ID          NAME            PROJECT_NUMBER
# my-web-app-2024     My Web App      123456789012
# staging-app-2024    Staging App     234567890123
# dev-playground      Dev Playground  345678901234
```

```bash
# Set a default project (so you don't have to specify it every time)
gcloud config set project my-web-app-2024

# Now all your gcloud commands will use this project by default
```

```bash
# Delete a project (careful! this deletes EVERYTHING in it)
gcloud projects delete my-web-app-2024
# Note: Projects are soft-deleted for 30 days. You can recover them within that window.
```

### GCP Resource Hierarchy

GCP organizes things in a hierarchy. Understanding this is crucial:

```
Organization (Top Level — your company's Google Cloud)
│
├── Folders (optional — for grouping projects)
│   ├── Sub-Folders (you can nest folders)
│   │   └── Projects
│   └── Projects
│
└── Projects (the base container for all resources)
    └── Resources (VMs, databases, buckets, etc.)
```

**Why does this hierarchy matter?**

Because **IAM policies INHERIT downward**:

```
Organization: "Allow team-leads@company.com to view everything"
    │
    └── This permission flows DOWN to every folder and project!
         │
         ├── Folder: Production
         │   └── team-leads can view everything here (inherited!)
         │
         └── Folder: Development
             └── team-leads can view everything here too (inherited!)
```

This means you can set a permission once at the top and it applies everywhere below. Super powerful, but also dangerous if not done carefully!

### Enabling APIs

Here's something that surprises many newcomers: In GCP, you have to **enable APIs** before you can use a service.

For example, before you can create a VM, you need to enable the Compute Engine API.

```bash
# Enable Compute Engine API
gcloud services enable compute.googleapis.com

# Enable Kubernetes Engine API
gcloud services enable container.googleapis.com

# Enable Cloud Storage API (usually enabled by default)
gcloud services enable storage.googleapis.com

# Enable Cloud Run API
gcloud services enable run.googleapis.com

# List all enabled APIs in your project
gcloud services list --enabled

# List all available APIs (there are hundreds!)
gcloud services list --available
```

> **Why does GCP do this?** For security and billing clarity. You only get charged for services you explicitly enable, and it prevents accidental usage.

## 🔐 Part 2: IAM — Identity and Access Management

### What is IAM?

IAM answers the fundamental question:

> **"Who can do what on which resource?"**

- **Who** → A person, a team, or an application (called a **Principal** or **Member**)
- **Can do what** → A specific action like "create VMs" or "read files" (called a **Role**)
- **On which resource** → A specific project, folder, VM, bucket, etc. (called a **Resource**)

```
IAM = WHO  +  CAN DO WHAT  +  ON WHICH RESOURCE

Example:
"alice@company.com" + "can create VMs" + "in project prod-web-app"
```

---

### IAM Building Blocks

#### 1. Principals (Who?)

A principal is the "who" — the identity that wants to do something.

| Principal Type | Example | Use Case |
|---------------|---------|----------|
| **Google Account** | `alice@gmail.com` | Individual person |
| **Service Account** | `my-app@project.iam.gserviceaccount.com` | Application/service |
| **Google Group** | `devops-team@company.com` | Group of people |
| **Google Workspace Domain** | `company.com` | Everyone in the company |
| **Cloud Identity Domain** | `company.com` | Everyone in the domain |
| **allAuthenticatedUsers** | (special) | Anyone with a Google account |
| **allUsers** | (special) | Anyone on the internet (public!) |

> ⚠️ **Warning:** Never use `allUsers` or `allAuthenticatedUsers` unless you intentionally want public access. This is a common security mistake!

---

#### 2. Roles (Can Do What?)

A role is a **collection of permissions**. Instead of assigning individual permissions (which would be tedious), you assign roles.

There are **three types of roles** in GCP:

##### a) Basic Roles (Primitive Roles) — AVOID in Production!

These are very broad, legacy roles:

| Role | What It Can Do |
|------|---------------|
| `roles/viewer` | Read-only access to everything |
| `roles/editor` | Read + write access to everything (can create/modify resources) |
| `roles/owner` | Editor + manage IAM + billing |

```
⚠️ NEVER use these in production!

Why? Because "Editor" means they can modify EVERY service in the project.
That's way too much power. A developer who just needs to deploy containers
shouldn't be able to delete your databases!
```

##### b) Predefined Roles — USE THESE!

Google creates these for specific services. They follow the **principle of least privilege**.

| Role | What It Can Do |
|------|---------------|
| `roles/compute.instanceAdmin` | Manage VMs only |
| `roles/storage.objectViewer` | Read objects from Cloud Storage only |
| `roles/container.clusterAdmin` | Manage GKE clusters only |
| `roles/cloudsql.editor` | Manage Cloud SQL databases only |
| `roles/run.developer` | Deploy to Cloud Run only |
| `roles/logging.viewer` | View logs only |

There are **hundreds** of predefined roles — one for almost every use case.

##### c) Custom Roles — When Predefined Isn't Enough

You can create your own roles with exactly the permissions you want:

```bash
# Create a custom role
gcloud iam roles create customStorageReader \
    --project=my-web-app-2024 \
    --title="Custom Storage Reader" \
    --description="Can only list and read storage buckets" \
    --permissions=storage.buckets.list,storage.objects.get,storage.objects.list

# What this does:
# Creates a role called "customStorageReader"
# This role can ONLY:
#   - List storage buckets
#   - Read objects from buckets
#   - List objects in buckets
# Nothing else! Very specific.
```

---

#### 3. IAM Policy (Putting It All Together)

An IAM Policy is a **binding** that connects a Principal to a Role on a Resource.

```
Policy = Principal + Role + Resource

"alice@company.com has roles/compute.instanceAdmin on project my-web-app-2024"
```

This is expressed as a JSON document called an **IAM Policy**:

```json
{
  "bindings": [
    {
      "role": "roles/compute.instanceAdmin",
      "members": [
        "user:alice@company.com",
        "group:devops-team@company.com"
      ]
    },
    {
      "role": "roles/storage.objectViewer",
      "members": [
        "user:bob@company.com",
        "serviceAccount:my-app@my-web-app-2024.iam.gserviceaccount.com"
      ]
    }
  ]
}
```

**Reading this policy:**
- Alice and the devops-team can **manage VMs** in this project
- Bob and the my-app service account can **read from storage buckets**
- Nobody else can do anything (deny by default!)

---

### Managing IAM with gcloud

```bash
# Grant a role to a user on a project
gcloud projects add-iam-policy-binding my-web-app-2024 \
    --member="user:alice@company.com" \
    --role="roles/compute.instanceAdmin"

# What this does:
# Gives alice@company.com the ability to manage VMs in the project my-web-app-2024
```

```bash
# Grant a role to a Google Group
gcloud projects add-iam-policy-binding my-web-app-2024 \
    --member="group:devops-team@company.com" \
    --role="roles/container.clusterAdmin"

# Now everyone in the devops-team group can manage GKE clusters
```

```bash
# Remove a role from a user
gcloud projects remove-iam-policy-binding my-web-app-2024 \
    --member="user:alice@company.com" \
    --role="roles/compute.instanceAdmin"
```

```bash
# View the full IAM policy for a project
gcloud projects get-iam-policy my-web-app-2024

# Output shows all bindings (who has what role)
```

```bash
# Check what roles a specific user has
gcloud asset search-all-iam-policies \
    --query="policy:alice@company.com" \
    --scope="projects/my-web-app-2024"
```

---

### IAM Conditions — Even More Granular Control

You can add **conditions** to IAM bindings. For example:
- "Alice can manage VMs, but **only between 9 AM and 5 PM**"
- "Bob can access storage, but **only from a specific IP range**"
- "The service account can read data, but **only resources tagged as 'non-sensitive'**"

```bash
# Grant a role with a condition (only during business hours)
gcloud projects add-iam-policy-binding my-web-app-2024 \
    --member="user:alice@company.com" \
    --role="roles/compute.instanceAdmin" \
    --condition='expression=request.time.getHours("America/New_York") >= 9 && request.time.getHours("America/New_York") <= 17,title=business-hours-only,description=Only during business hours'
```

---

### IAM Best Practices

| Practice | Why |
|----------|-----|
| **Use least privilege** | Give only the minimum permissions needed |
| **Use predefined roles, not basic roles** | `roles/editor` is too broad. Use specific roles like `roles/compute.instanceAdmin` |
| **Use Groups, not individual users** | Add `devops-team@company.com` to a role, not individual emails. Easier to manage |
| **Use Service Accounts for apps** | Applications should never use human credentials |
| **Audit regularly** | Use IAM Recommender to find overprivileged accounts |
| **Enable Organization Policies** | Set guardrails at the org level |
| **Use Conditions** | Add time-based or IP-based conditions for extra security |

## 🤖 Part 3: Service Accounts

### What is a Service Account?

A Service Account is a special type of account used by **applications, VMs, and services** — not humans.

Think of it this way:
- **Human Account:** alice@company.com → A real person logs in with a password
- **Service Account:** my-app@project-id.iam.gserviceaccount.com → An application authenticates automatically

### Why Do We Need Service Accounts?

Imagine you have an application that needs to:
- Read data from Cloud Storage
- Write logs to Cloud Logging
- Access a database

You can't have a person sitting there typing their password every time the app needs to access something. The app needs its own identity — that's a Service Account!

```
Without Service Accounts:
App → "Hey Google Cloud, I want to read from storage"
GCP → "Who are you? Show me your credentials!"
App → "Umm... I don't have any? Can you ask Bob to log in?"
❌ This doesn't work!

With Service Accounts:
App → "Hey Google Cloud, I want to read from storage. Here's my service account key."
GCP → "Ah, you're my-app@project.iam.gserviceaccount.com. Let me check... 
        yes, you have storage.objectViewer role. Here's your data!"
✅ Works automatically, no human needed!
```

---

### Types of Service Accounts

| Type | Description | Example |
|------|------------|---------|
| **Default Service Accounts** | Auto-created when you enable certain APIs | `PROJECT_NUMBER-compute@developer.gserviceaccount.com` |
| **User-Managed Service Accounts** | You create them for specific purposes | `my-app@my-project.iam.gserviceaccount.com` |
| **Google-Managed Service Accounts** | Google creates them internally for GCP services | Used by Google behind the scenes |

> ⚠️ **Important:** Default service accounts often have the **Editor** role (way too much permission!). Always create your own service accounts with specific, minimal permissions.

---

### Creating and Managing Service Accounts

```bash
# Create a service account
gcloud iam service-accounts create my-app-sa \
    --display-name="My Application Service Account" \
    --description="Used by the web app to access storage and databases"

# What this creates:
# Email: my-app-sa@my-web-app-2024.iam.gserviceaccount.com
# This is the "identity" of your application
```

```bash
# List all service accounts in a project
gcloud iam service-accounts list

# Output:
# EMAIL                                                  DISPLAY_NAME
# my-app-sa@my-web-app-2024.iam.gserviceaccount.com     My Application Service Account
# 123456-compute@developer.gserviceaccount.com           Compute Engine default service account
```

```bash
# Grant roles to the service account
# "This service account can read from Cloud Storage"
gcloud projects add-iam-policy-binding my-web-app-2024 \
    --member="serviceAccount:my-app-sa@my-web-app-2024.iam.gserviceaccount.com" \
    --role="roles/storage.objectViewer"

# "This service account can also write to Cloud SQL"
gcloud projects add-iam-policy-binding my-web-app-2024 \
    --member="serviceAccount:my-app-sa@my-web-app-2024.iam.gserviceaccount.com" \
    --role="roles/cloudsql.client"
```

---

### How Do Applications Use Service Accounts?

There are **two main ways** an application can authenticate with a service account:

#### Method 1: Service Account Keys (JSON Key File)

You download a JSON key file and your app uses it to authenticate.

```bash
# Create a key file for the service account
gcloud iam service-accounts keys create key.json \
    --iam-account=my-app-sa@my-web-app-2024.iam.gserviceaccount.com

# This creates a file called key.json that looks like:
```

```json
{
  "type": "service_account",
  "project_id": "my-web-app-2024",
  "private_key_id": "abc123...",
  "private_key": "-----BEGIN PRIVATE KEY-----\n...\n-----END PRIVATE KEY-----\n",
  "client_email": "my-app-sa@my-web-app-2024.iam.gserviceaccount.com",
  "client_id": "123456789",
  "auth_uri": "https://accounts.google.com/o/oauth2/auth",
  "token_uri": "https://oauth2.googleapis.com/token"
}
```

```bash
# Your application uses this key by setting an environment variable:
export GOOGLE_APPLICATION_CREDENTIALS="/path/to/key.json"

# Now any Google Cloud library in your app will automatically use this service account!
```

Using the key in a Python application:

```python
from google.cloud import storage

# This automatically uses GOOGLE_APPLICATION_CREDENTIALS
client = storage.Client()

# List all buckets
buckets = client.list_buckets()
for bucket in buckets:
    print(bucket.name)
```

> ⚠️ **SECURITY WARNING:** Service account key files are **extremely sensitive**!
> - NEVER commit them to Git
> - NEVER share them over email or Slack
> - Rotate them regularly
> - Use Workload Identity (Method 2) whenever possible instead

```bash
# List all keys for a service account (to audit)
gcloud iam service-accounts keys list \
    --iam-account=my-app-sa@my-web-app-2024.iam.gserviceaccount.com

# Delete an old key
gcloud iam service-accounts keys delete KEY_ID \
    --iam-account=my-app-sa@my-web-app-2024.iam.gserviceaccount.com
```

---

#### Method 2: Workload Identity / Attached Service Accounts (Preferred! 🌟)

Instead of downloading a key file, you **attach the service account directly** to your GCP resource (VM, Cloud Run, GKE pod, etc.).

The resource automatically gets the service account's identity — no key file needed!

```bash
# Create a VM with a specific service account attached
gcloud compute instances create my-vm \
    --zone=us-central1-a \
    --service-account=my-app-sa@my-web-app-2024.iam.gserviceaccount.com \
    --scopes=cloud-platform

# Now any code running on this VM automatically has the service account's permissions
# No key file needed!
```

```bash
# Deploy a Cloud Run service with a specific service account
gcloud run deploy my-service \
    --image=gcr.io/my-web-app-2024/my-app:latest \
    --service-account=my-app-sa@my-web-app-2024.iam.gserviceaccount.com

# The Cloud Run service automatically authenticates as this service account
```

**For GKE (Kubernetes), use Workload Identity:**

```bash
# Enable Workload Identity on your GKE cluster
gcloud container clusters update my-cluster \
    --zone=us-central1-a \
    --workload-pool=my-web-app-2024.svc.id.goog

# Create a Kubernetes Service Account
kubectl create serviceaccount my-k8s-sa --namespace default

# Bind the Kubernetes SA to the GCP SA
gcloud iam service-accounts add-iam-policy-binding \
    my-app-sa@my-web-app-2024.iam.gserviceaccount.com \
    --role="roles/iam.workloadIdentityUser" \
    --member="serviceAccount:my-web-app-2024.svc.id.goog[default/my-k8s-sa]"

# Annotate the Kubernetes SA
kubectl annotate serviceaccount my-k8s-sa \
    iam.gke.io/gcp-service-account=my-app-sa@my-web-app-2024.iam.gserviceaccount.com
```

Now any pod using `my-k8s-sa` will automatically have the GCP service account's permissions! No keys, no secrets to manage.

---

### Service Account Impersonation

Sometimes, instead of downloading a key, you want to **temporarily act as** a service account. This is called impersonation.

```bash
# Impersonate a service account
gcloud auth print-access-token \
    --impersonate-service-account=my-app-sa@my-web-app-2024.iam.gserviceaccount.com

# Run a command as the service account
gcloud storage ls \
    --impersonate-service-account=my-app-sa@my-web-app-2024.iam.gserviceaccount.com
```

For impersonation to work, you need the `roles/iam.serviceAccountTokenCreator` role on the target service account.

---

### Service Account Best Practices

| Practice | Why |
|----------|-----|
| **Create purpose-specific service accounts** | Don't use the default SA. Create one per application/service |
| **Follow least privilege** | Only give the exact permissions the app needs |
| **Avoid key files when possible** | Use Workload Identity or attached SAs instead |
| **Rotate keys regularly** | If you must use keys, rotate them every 90 days |
| **Disable unused service accounts** | Don't leave stale SAs lying around |
| **Monitor SA usage** | Use Cloud Audit Logs to track what SAs are doing |
| **Never use default SAs in production** | They usually have Editor role — way too much power |

```bash
# Disable a service account (without deleting)
gcloud iam service-accounts disable \
    my-app-sa@my-web-app-2024.iam.gserviceaccount.com

# Re-enable it later
gcloud iam service-accounts enable \
    my-app-sa@my-web-app-2024.iam.gserviceaccount.com

# Delete a service account
gcloud iam service-accounts delete \
    my-app-sa@my-web-app-2024.iam.gserviceaccount.com
```

## 🧪 Hands-On Lab: Setting Up a GCP Project with Proper IAM

Let's put everything together in a real-world scenario:

**Scenario:** Your company needs a new project for a web application. You need to:
1. Create the project
2. Set up proper IAM
3. Create a service account for the app

```bash
# Step 1: Create the project
gcloud projects create webapp-prod-2024 \
    --name="Production Web App" \
    --organization=YOUR_ORG_ID

# Step 2: Set the project as default
gcloud config set project webapp-prod-2024

# Step 3: Link a billing account
gcloud billing projects link webapp-prod-2024 \
    --billing-account=BILLING_ACCOUNT_ID

# Step 4: Enable required APIs
gcloud services enable compute.googleapis.com
gcloud services enable container.googleapis.com
gcloud services enable storage.googleapis.com
gcloud services enable sqladmin.googleapis.com

# Step 5: Create a service account for the web app
gcloud iam service-accounts create webapp-sa \
    --display-name="Web App Service Account" \
    --description="Used by the production web application"

# Step 6: Grant specific roles to the service account
# Can read from storage
gcloud projects add-iam-policy-binding webapp-prod-2024 \
    --member="serviceAccount:webapp-sa@webapp-prod-2024.iam.gserviceaccount.com" \
    --role="roles/storage.objectViewer"

# Can connect to Cloud SQL
gcloud projects add-iam-policy-binding webapp-prod-2024 \
    --member="serviceAccount:webapp-sa@webapp-prod-2024.iam.gserviceaccount.com" \
    --role="roles/cloudsql.client"

# Can write logs
gcloud projects add-iam-policy-binding webapp-prod-2024 \
    --member="serviceAccount:webapp-sa@webapp-prod-2024.iam.gserviceaccount.com" \
    --role="roles/logging.logWriter"

# Step 7: Grant team access
# DevOps team can manage Kubernetes
gcloud projects add-iam-policy-binding webapp-prod-2024 \
    --member="group:devops@company.com" \
    --role="roles/container.clusterAdmin"

# Developers can view but not modify
gcloud projects add-iam-policy-binding webapp-prod-2024 \
    --member="group:developers@company.com" \
    --role="roles/viewer"

# Step 8: Verify everything
gcloud projects get-iam-policy webapp-prod-2024
gcloud iam service-accounts list
gcloud services list --enabled
```

## 🧠 Summary — The Big Picture

```
Google Cloud
│
├── Organization
│   ├── Folders (organize projects by env/team)
│   └── Projects (container for all resources)
│       ├── Resources (VMs, DBs, buckets...)
│       └── IAM Policies
│           ├── WHO? → Principals (users, groups, service accounts)
│           ├── CAN DO WHAT? → Roles (predefined, custom)
│           └── ON WHAT? → Resources (project, folder, specific resource)
│
└── Service Accounts
    ├── Used by applications (not humans)
    ├── Authenticate via:
    │   ├── Key files (JSON) — works but risky
    │   └── Workload Identity (preferred — no keys!)
    └── Follow least privilege — only give what's needed
```

**Key Takeaways:**
1. **Everything lives in a Project** — always create projects for isolation
2. **IAM = Who + What + Where** — use predefined roles, avoid basic roles
3. **Service Accounts are for apps** — prefer Workload Identity over key files
4. **Principle of Least Privilege** — give the minimum access needed, nothing more
5. **Use Groups** — manage access through groups, not individual users
