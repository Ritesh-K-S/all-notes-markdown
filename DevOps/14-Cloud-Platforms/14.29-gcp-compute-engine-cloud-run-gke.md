## 🤔 The Big Question: Where Do I Run My Application?

When you want to run an application on Google Cloud, you have several options. Think of it like choosing transportation:

```
🚗 Compute Engine (VMs)    → You buy a car. You drive it, maintain it, fuel it.
                              Full control, full responsibility.

🚕 Cloud Run (Serverless)  → You call a taxi. It shows up when you need it,
                              disappears when you don't. You just ride.

🚂 GKE (Kubernetes)        → You run a train system. Complex but powerful.
                              Best for big, complex operations with many services.
```

| Feature | Compute Engine | Cloud Run | GKE |
|---------|---------------|-----------|-----|
| **What it is** | Virtual Machines | Serverless Containers | Managed Kubernetes |
| **Control Level** | Full (OS, network, everything) | Minimal (just your container) | High (Kubernetes features) |
| **Scaling** | Manual or Auto Scaling Groups | Automatic (including to zero!) | Automatic (HPA, VPA, Cluster Autoscaler) |
| **Cost** | Pay for running VMs (even if idle) | Pay only when handling requests | Pay for nodes (VMs running K8s) |
| **Best For** | Legacy apps, custom OS needs | Stateless APIs, microservices | Complex microservice architectures |
| **Learning Curve** | Low-Medium | Low | High |
| **Management Overhead** | High (you manage the OS) | Very Low | Medium (Google manages control plane) |

## 🖥️ Part 1: Compute Engine — Virtual Machines

### What is Compute Engine?

Compute Engine lets you create and run **Virtual Machines (VMs)** on Google's infrastructure. It's like renting a computer in Google's data center.

You get:
- A full operating system (Linux or Windows)
- CPU and RAM that you choose
- Disk storage
- A network interface
- Complete control over everything running on it

### When to Use Compute Engine?

- You need a traditional server (install whatever you want)
- Running applications that aren't containerized
- You need a specific OS configuration
- Legacy applications that can't run in containers
- You need GPU/TPU for machine learning
- You want full control over the environment

---

### Machine Types — Choosing the Right Size

GCP offers many machine types. Think of it like picking a size of pizza:

| Family | Use Case | Example |
|--------|----------|---------|
| **General Purpose (E2, N2, N2D)** | Everyday workloads, web servers, small databases | `e2-medium`, `n2-standard-4` |
| **Compute Optimized (C2, C2D)** | CPU-heavy tasks: gaming, HPC, batch processing | `c2-standard-8` |
| **Memory Optimized (M2, M3)** | Memory-heavy: large databases, in-memory caching | `m2-ultramem-416` |
| **Accelerator Optimized (A2, G2)** | GPU workloads: ML training, video processing | `a2-highgpu-1g` |
| **Shared-core (e2-micro, e2-small)** | Tiny workloads, dev/test, lightweight apps | `e2-micro` (free tier!) |

#### Understanding Machine Type Names:

```
n2-standard-4
│  │         │
│  │         └── 4 vCPUs
│  └── standard = balanced CPU/memory ratio
└── n2 = machine family (2nd gen general purpose)

Machine type naming: FAMILY-TYPE-VCPUS
```

**Common machine types:**

| Machine Type | vCPUs | Memory | Good For |
|-------------|-------|--------|----------|
| `e2-micro` | 0.25 | 1 GB | Free tier, tiny apps |
| `e2-small` | 0.5 | 2 GB | Dev/test, small services |
| `e2-medium` | 1 | 4 GB | Light web servers |
| `e2-standard-2` | 2 | 8 GB | General web apps |
| `e2-standard-4` | 4 | 16 GB | Medium web apps |
| `n2-standard-8` | 8 | 32 GB | Production workloads |
| `n2-standard-16` | 16 | 64 GB | Heavy applications |

You can also create **custom machine types** with exactly the CPU and memory you want:

```bash
# Create a custom machine type with 6 vCPUs and 20 GB RAM
gcloud compute instances create my-vm \
    --custom-cpu=6 \
    --custom-memory=20GB \
    --zone=us-central1-a
```

### Creating a VM — Step by Step

#### Using gcloud CLI:

```bash
# Create a basic VM
gcloud compute instances create my-first-vm \
    --zone=us-central1-a \
    --machine-type=e2-medium \
    --image-family=ubuntu-2204-lts \
    --image-project=ubuntu-os-cloud \
    --boot-disk-size=20GB \
    --tags=http-server,https-server

# Let's break this down:
# gcloud compute instances create  → command to create a VM
# my-first-vm                      → name of the VM
# --zone=us-central1-a             → which data center to create it in
# --machine-type=e2-medium         → size of the VM (1 vCPU, 4 GB RAM)
# --image-family=ubuntu-2204-lts   → operating system (Ubuntu 22.04)
# --image-project=ubuntu-os-cloud  → where the OS image comes from
# --boot-disk-size=20GB            → disk size
# --tags=http-server,https-server  → network tags (for firewall rules)
```

```bash
# SSH into your VM
gcloud compute ssh my-first-vm --zone=us-central1-a

# This opens an SSH session directly into your VM!
# You're now "inside" the virtual machine, like sitting at a remote computer.
```

```bash
# List all your VMs
gcloud compute instances list

# Output:
# NAME          ZONE            MACHINE_TYPE  STATUS   INTERNAL_IP  EXTERNAL_IP
# my-first-vm   us-central1-a   e2-medium     RUNNING  10.128.0.2   34.68.123.45
```

```bash
# Stop a VM (to save money when not using it)
gcloud compute instances stop my-first-vm --zone=us-central1-a

# Start it again
gcloud compute instances start my-first-vm --zone=us-central1-a

# Delete a VM (permanently removes it)
gcloud compute instances delete my-first-vm --zone=us-central1-a
```

### Startup Scripts — Auto-Configure Your VM

You can pass a script that runs automatically when the VM starts for the first time:

```bash
# Create a VM with a startup script
gcloud compute instances create web-server \
    --zone=us-central1-a \
    --machine-type=e2-medium \
    --image-family=ubuntu-2204-lts \
    --image-project=ubuntu-os-cloud \
    --metadata=startup-script='#!/bin/bash
apt-get update
apt-get install -y nginx
systemctl start nginx
systemctl enable nginx
echo "<h1>Hello from GCP!</h1>" > /var/www/html/index.html'

# This VM will automatically:
# 1. Update packages
# 2. Install Nginx web server
# 3. Start Nginx
# 4. Create a simple web page
# All without you having to SSH in!
```

For longer scripts, use a file:

```bash
# Create the startup script file
cat > startup.sh << 'EOF'
#!/bin/bash
apt-get update
apt-get install -y nginx docker.io
systemctl start nginx
systemctl start docker

# Deploy your application
docker pull mycompany/webapp:latest
docker run -d -p 8080:8080 mycompany/webapp:latest
EOF

# Create the VM with the script file
gcloud compute instances create web-server \
    --zone=us-central1-a \
    --machine-type=e2-standard-2 \
    --image-family=ubuntu-2204-lts \
    --image-project=ubuntu-os-cloud \
    --metadata-from-file=startup-script=startup.sh
```

### Disks and Storage

Each VM needs at least a **boot disk** (where the OS lives), but you can attach additional disks:

```bash
# Create an additional disk
gcloud compute disks create my-data-disk \
    --size=100GB \
    --type=pd-ssd \
    --zone=us-central1-a

# Disk types:
# pd-standard → HDD (cheap, slower)
# pd-balanced → Balanced SSD (good default)
# pd-ssd      → SSD (fast, more expensive)
# pd-extreme  → Extreme performance SSD

# Attach the disk to a VM
gcloud compute instances attach-disk my-first-vm \
    --disk=my-data-disk \
    --zone=us-central1-a

# Inside the VM, you'd then format and mount it:
# sudo mkfs.ext4 /dev/sdb
# sudo mkdir /mnt/data
# sudo mount /dev/sdb /mnt/data
```

### Preemptible & Spot VMs — Save Up to 90%!

If your workload can handle interruptions (batch processing, CI/CD, data processing), you can use **Spot VMs** that cost up to 90% less:

```bash
# Create a Spot VM (can be stopped by Google with 30-second notice)
gcloud compute instances create batch-worker \
    --zone=us-central1-a \
    --machine-type=e2-standard-4 \
    --provisioning-model=SPOT \
    --instance-termination-action=STOP \
    --image-family=ubuntu-2204-lts \
    --image-project=ubuntu-os-cloud

# This is the same VM but costs ~60-91% less!
# The trade-off: Google can stop it at any time if they need the capacity back.
# Great for: CI/CD runners, batch jobs, testing, data processing
# Bad for: Production web servers, databases
```

### Instance Groups & Auto-Scaling

For production, you rarely run just one VM. You use **Instance Groups** that automatically scale:

```bash
# Step 1: Create an Instance Template (blueprint for VMs)
gcloud compute instance-templates create web-template \
    --machine-type=e2-medium \
    --image-family=ubuntu-2204-lts \
    --image-project=ubuntu-os-cloud \
    --metadata=startup-script='#!/bin/bash
apt-get update && apt-get install -y nginx
echo "Hello from $(hostname)" > /var/www/html/index.html'

# Step 2: Create a Managed Instance Group
gcloud compute instance-groups managed create web-group \
    --template=web-template \
    --size=3 \
    --zone=us-central1-a

# This creates 3 identical VMs from the template

# Step 3: Set up Auto-Scaling
gcloud compute instance-groups managed set-autoscaling web-group \
    --zone=us-central1-a \
    --min-num-replicas=2 \
    --max-num-replicas=10 \
    --target-cpu-utilization=0.60

# Now GCP will:
# - Keep at least 2 VMs running
# - Scale up to 10 VMs if CPU goes above 60%
# - Scale back down when load decreases
```

## 🚀 Part 2: Cloud Run — Serverless Containers

### What is Cloud Run?

Cloud Run is a **serverless platform** that runs your **containers** without you having to manage any servers. You just give it a container image and it handles everything else.

```
You: "Here's my container image. Run it."
Cloud Run: "Got it! I'll:
  - Run it
  - Auto-scale it (even down to ZERO instances when nobody's using it)
  - Give it an HTTPS URL
  - Handle SSL certificates
  - Do load balancing
  - And you only pay when someone actually sends a request!"
```

### When to Use Cloud Run?

- Stateless web applications and APIs
- Microservices
- Webhooks and event handlers
- Lightweight backends
- Any containerized app that doesn't need persistent local storage
- You want to pay only for what you use (great for low-traffic apps!)

### When NOT to Use Cloud Run?

- Applications that need persistent local disk storage
- Long-running processes (max timeout is 60 minutes)
- Applications that need to maintain WebSocket connections for very long periods
- Heavy stateful workloads (use GKE instead)

### Deploying to Cloud Run — Step by Step

#### Step 1: Create Your Application

Let's create a simple Node.js app as an example:

```javascript
// app.js
const express = require('express');
const app = express();

app.get('/', (req, res) => {
  res.send(`Hello from Cloud Run! 
    Hostname: ${require('os').hostname()}
    Time: ${new Date().toISOString()}`);
});

app.get('/health', (req, res) => {
  res.status(200).send('OK');
});

const PORT = process.env.PORT || 8080;
app.listen(PORT, () => {
  console.log(`Server running on port ${PORT}`);
});
```

```json
// package.json
{
  "name": "cloud-run-demo",
  "version": "1.0.0",
  "scripts": {
    "start": "node app.js"
  },
  "dependencies": {
    "express": "^4.18.0"
  }
}
```

#### Step 2: Create a Dockerfile

```dockerfile
# Dockerfile
FROM node:18-slim

WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY . .

# Cloud Run provides the PORT environment variable
ENV PORT=8080
EXPOSE 8080

CMD ["npm", "start"]
```

#### Step 3: Build and Push the Container Image

```bash
# Option A: Build locally and push to Google Container Registry
gcloud builds submit --tag gcr.io/my-web-app-2024/cloud-run-demo

# What this does:
# 1. Sends your code to Cloud Build (Google's build service)
# 2. Builds the Docker image in the cloud
# 3. Pushes it to Google Container Registry (gcr.io)
# You don't even need Docker installed locally!

# Option B: Use Artifact Registry (newer, recommended)
gcloud builds submit --tag us-central1-docker.pkg.dev/my-web-app-2024/my-repo/cloud-run-demo
```

#### Step 4: Deploy to Cloud Run

```bash
# Deploy the container to Cloud Run
gcloud run deploy cloud-run-demo \
    --image=gcr.io/my-web-app-2024/cloud-run-demo \
    --region=us-central1 \
    --platform=managed \
    --allow-unauthenticated \
    --port=8080 \
    --memory=256Mi \
    --cpu=1 \
    --min-instances=0 \
    --max-instances=10

# Let's break this down:
# --image             → the container image to deploy
# --region            → which region to deploy in
# --platform=managed  → use fully managed Cloud Run (not on GKE)
# --allow-unauthenticated → anyone can access it (public API)
# --port=8080         → the port your app listens on
# --memory=256Mi      → max memory per instance
# --cpu=1             → max CPUs per instance
# --min-instances=0   → scale to ZERO when no traffic (saves money!)
# --max-instances=10  → maximum 10 instances during high traffic
```

**Output:**
```
Deploying container to Cloud Run service [cloud-run-demo] in project [my-web-app-2024] region [us-central1]
✓ Deploying...
  ✓ Creating Revision...
  ✓ Routing traffic...
Done.

Service [cloud-run-demo] revision [cloud-run-demo-00001-abc] has been deployed and is serving 100% of traffic.

Service URL: https://cloud-run-demo-abc123-uc.a.run.app
```

That's it! Your app is live at an HTTPS URL. No servers to manage, no SSL certificates to worry about.

### Cloud Run Key Features

#### Environment Variables

```bash
# Set environment variables
gcloud run deploy cloud-run-demo \
    --image=gcr.io/my-web-app-2024/cloud-run-demo \
    --region=us-central1 \
    --set-env-vars="DATABASE_URL=postgres://...,API_KEY=abc123,NODE_ENV=production"

# Update just one env var without redeploying
gcloud run services update cloud-run-demo \
    --region=us-central1 \
    --update-env-vars="LOG_LEVEL=debug"
```

#### Secrets from Secret Manager

```bash
# First, create a secret in Secret Manager
echo -n "my-super-secret-password" | gcloud secrets create db-password --data-file=-

# Mount the secret as an environment variable in Cloud Run
gcloud run deploy cloud-run-demo \
    --image=gcr.io/my-web-app-2024/cloud-run-demo \
    --region=us-central1 \
    --set-secrets="DB_PASSWORD=db-password:latest"

# Your app can now access it as process.env.DB_PASSWORD
# The secret is never stored in the container image!
```

#### Traffic Splitting (Canary Deployments)

```bash
# Deploy a new revision
gcloud run deploy cloud-run-demo \
    --image=gcr.io/my-web-app-2024/cloud-run-demo:v2 \
    --region=us-central1 \
    --no-traffic  # Don't send any traffic to this version yet

# Send 10% of traffic to the new version (canary deployment)
gcloud run services update-traffic cloud-run-demo \
    --region=us-central1 \
    --to-revisions=cloud-run-demo-00002-xyz=10

# Everything looks good? Send 100% to new version
gcloud run services update-traffic cloud-run-demo \
    --region=us-central1 \
    --to-latest
```

#### Custom Domains

```bash
# Map a custom domain to your Cloud Run service
gcloud run domain-mappings create \
    --service=cloud-run-demo \
    --domain=api.mycompany.com \
    --region=us-central1
```

### Cloud Run Pricing — Pay Per Request

This is what makes Cloud Run special:

```
Pricing = CPU time + Memory time + Number of requests

Example:
- 1 million requests/month
- Each request takes 200ms
- 256MB memory, 1 vCPU

Cost ≈ $0.50 - $2.00/month  ← Compare this to a VM running 24/7!

If your app gets ZERO traffic:
Cost = $0.00  ← Literally free! (min-instances=0)
```

## ☸️ Part 3: GKE — Google Kubernetes Engine

### What is GKE?

GKE is Google's **managed Kubernetes service**. Since Google invented Kubernetes (it came from Google's internal Borg system), GKE is often considered the best managed Kubernetes offering.

**What "managed" means:**
- Google manages the **control plane** (API server, etcd, scheduler, controller manager)
- You manage the **worker nodes** and your **applications**
- Google handles upgrades, patches, and high availability of the control plane

```
Without GKE (self-managed Kubernetes):
You: Set up control plane + worker nodes + networking + upgrades + patches + monitoring
     = Months of work, constant maintenance

With GKE:
Google: "Here's a working Kubernetes cluster. Control plane is ready."
You: "Great, I'll just deploy my apps!"
     = Cluster ready in minutes
```

### When to Use GKE?

- You have multiple microservices that need to communicate
- You need Kubernetes features (auto-scaling, self-healing, rolling updates)
- Your team already knows Kubernetes
- You need multi-cloud portability (K8s runs everywhere)
- Complex applications with many interconnected services
- You need fine-grained control over networking, scheduling, and resources

---

### GKE Modes — Standard vs Autopilot

GKE has two modes:

| Feature | GKE Standard | GKE Autopilot |
|---------|-------------|---------------|
| **Node Management** | You manage nodes | Google manages nodes |
| **Pricing** | Pay per node (VM) | Pay per pod (resources used) |
| **Control** | Full control over nodes | Limited node configuration |
| **Scaling** | You configure autoscaling | Automatic |
| **Security** | You harden nodes | Pre-hardened by Google |
| **Best For** | Full control, custom configs | Hands-off, serverless-like K8s |

```
GKE Standard = You manage the worker nodes (VMs)
GKE Autopilot = Google manages EVERYTHING, you just deploy pods
```

### Creating a GKE Cluster

#### GKE Standard Cluster:

```bash
# Create a GKE Standard cluster
gcloud container clusters create my-cluster \
    --zone=us-central1-a \
    --num-nodes=3 \
    --machine-type=e2-standard-2 \
    --disk-size=50GB \
    --enable-autoscaling \
    --min-nodes=2 \
    --max-nodes=10 \
    --enable-autorepair \
    --enable-autoupgrade \
    --release-channel=regular

# Let's break this down:
# --zone=us-central1-a      → single zone cluster (cheaper, not HA)
# --num-nodes=3             → start with 3 worker nodes
# --machine-type=e2-standard-2 → each node: 2 vCPUs, 8 GB RAM
# --disk-size=50GB          → disk per node
# --enable-autoscaling      → automatically add/remove nodes based on load
# --min-nodes=2             → never go below 2 nodes
# --max-nodes=10            → never go above 10 nodes
# --enable-autorepair       → automatically replace unhealthy nodes
# --enable-autoupgrade      → automatically upgrade node versions
# --release-channel=regular → balanced between new features and stability
```

For **high availability** (multi-zone):

```bash
# Create a regional (multi-zone) cluster
gcloud container clusters create prod-cluster \
    --region=us-central1 \
    --num-nodes=1 \
    --machine-type=e2-standard-4 \
    --enable-autoscaling \
    --min-nodes=1 \
    --max-nodes=5 \
    --enable-autorepair \
    --enable-autoupgrade

# --region=us-central1 → creates nodes across 3 zones
# (us-central1-a, us-central1-b, us-central1-c)
# --num-nodes=1 → 1 node PER ZONE = 3 nodes total
# If one zone goes down, your app keeps running!
```

#### GKE Autopilot Cluster (Simpler!):

```bash
# Create a GKE Autopilot cluster
gcloud container clusters create-auto my-autopilot-cluster \
    --region=us-central1

# That's it! Google handles EVERYTHING about the nodes.
# You just deploy pods and Google figures out the rest.
```

### Deploying an Application to GKE

Let's deploy a web application:

```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
  labels:
    app: web-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web-app
  template:
    metadata:
      labels:
        app: web-app
    spec:
      containers:
      - name: web-app
        image: gcr.io/my-web-app-2024/web-app:latest
        ports:
        - containerPort: 8080
        resources:
          requests:
            cpu: "100m"        # 0.1 CPU
            memory: "128Mi"    # 128 MB RAM
          limits:
            cpu: "500m"        # 0.5 CPU max
            memory: "256Mi"    # 256 MB RAM max
        livenessProbe:
          httpGet:
            path: /health
            port: 8080
          initialDelaySeconds: 10
          periodSeconds: 30
        readinessProbe:
          httpGet:
            path: /ready
            port: 8080
          initialDelaySeconds: 5
          periodSeconds: 10
        env:
        - name: NODE_ENV
          value: "production"
```

```yaml
# service.yaml
apiVersion: v1
kind: Service
metadata:
  name: web-app-service
spec:
  type: LoadBalancer    # Creates a Google Cloud Load Balancer
  selector:
    app: web-app
  ports:
  - protocol: TCP
    port: 80            # External port
    targetPort: 8080    # Container port
```

```bash
# Deploy the application
kubectl apply -f deployment.yaml
kubectl apply -f service.yaml

# Check the deployment
kubectl get deployments

# Output:

# NAME              TYPE           CLUSTER-IP     EXTERNAL-IP     PORT(S)        AGE

```

### GKE Node Pools

Node pools let you have **different types of machines** in the same cluster:

```bash
# Add a node pool with high-memory machines (for databases)
gcloud container node-pools create high-memory-pool \
    --cluster=my-cluster \
    --zone=us-central1-a \
    --machine-type=n2-highmem-4 \
    --num-nodes=2 \
    --enable-autoscaling \
    --min-nodes=1 \
    --max-nodes=5

# Add a node pool with Spot VMs (for batch jobs, save money!)
gcloud container node-pools create spot-pool \
    --cluster=my-cluster \
    --zone=us-central1-a \
    --machine-type=e2-standard-4 \
    --spot \
    --num-nodes=3 \
    --enable-autoscaling \
    --min-nodes=0 \
    --max-nodes=20
```

### GKE Workload Identity (Connecting GKE to GCP Services)

Instead of using service account key files in your pods, use Workload Identity:

```bash
# Step 1: Enable Workload Identity on the cluster
gcloud container clusters update my-cluster \
    --zone=us-central1-a \
    --workload-pool=my-web-app-2024.svc.id.goog

# Step 2: Create a GCP service account
gcloud iam service-accounts create gke-app-sa \
    --display-name="GKE App Service Account"

# Step 3: Grant the GCP SA permissions
gcloud projects add-iam-policy-binding my-web-app-2024 \
    --member="serviceAccount:gke-app-sa@my-web-app-2024.iam.gserviceaccount.com" \
    --role="roles/storage.objectViewer"

# Step 4: Create a Kubernetes service account
kubectl create serviceaccount app-ksa

# Step 5: Bind the K8s SA to the GCP SA
gcloud iam service-accounts add-iam-policy-binding \
    gke-app-sa@my-web-app-2024.iam.gserviceaccount.com \
    --role="roles/iam.workloadIdentityUser" \
    --member="serviceAccount:my-web-app-2024.svc.id.goog[default/app-ksa]"

# Step 6: Annotate the K8s SA
kubectl annotate serviceaccount app-ksa \
    iam.gke.io/gcp-service-account=gke-app-sa@my-web-app-2024.iam.gserviceaccount.com
```

```yaml
# Step 7: Use it in your deployment
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 2
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      serviceAccountName: app-ksa    # Uses Workload Identity!
      containers:
      - name: my-app
        image: gcr.io/my-web-app-2024/my-app:latest
        # No key files needed! The pod automatically has GCP permissions.
```

## 🤔 Which One Should You Choose?

Here's a simple decision tree:

```
Do you need full control over the OS?
├── Yes → Compute Engine (VMs)
└── No
    ├── Is your app containerized?
    │   ├── No → Compute Engine or Cloud Run (Cloud Run can build from source too)
    │   └── Yes
    │       ├── Is it a simple stateless service?
    │       │   ├── Yes → Cloud Run ✨ (simplest, cheapest for variable traffic)
    │       │   └── No
    │       │       ├── Do you have many microservices?
    │       │       │   ├── Yes → GKE 
    │       │       │   └── No → Cloud Run (probably still fine)
    │       │       └── Do you need Kubernetes features?
    │       │           ├── Yes → GKE
    │       │           └── No → Cloud Run
    └── Not sure → Start with Cloud Run. Migrate to GKE if needed.
```

### Real-World Usage Patterns

| Scenario | Best Choice | Why |
|----------|-------------|-----|
| Company blog / marketing website | Cloud Run | Simple, low traffic, cheap |
| REST API for mobile app | Cloud Run | Stateless, auto-scales |
| E-commerce platform (10+ services) | GKE | Complex, many services, need K8s |
| Legacy Java application | Compute Engine | Needs specific JVM/OS config |
| ML model serving | Cloud Run or GKE | Depends on complexity |
| Jenkins CI/CD server | Compute Engine | Needs persistent state, specific setup |
| Batch data processing | GKE (Spot nodes) or Compute Engine (Spot) | Cost-effective for batch |

## 🧠 Summary

```
Compute Engine (VMs)
├── Full control over OS and environment
├── You manage everything (OS, patches, scaling)
├── Pay for the VM 24/7 (even when idle)
├── Best for: legacy apps, custom OS needs, persistent state
└── Think: "I need a server"

Cloud Run (Serverless Containers)
├── Give it a container, it runs it
├── Auto-scales (even to zero!)
├── Pay only for request processing time
├── HTTPS URL, SSL, load balancing — all automatic
├── Best for: APIs, microservices, webhooks
└── Think: "I just want to run my code"

GKE (Managed Kubernetes)
├── Full Kubernetes on Google Cloud
├── Google manages control plane, you manage workloads
├── Autopilot mode = even less management
├── Best for: complex multi-service architectures
└── Think: "I need orchestration at scale"
```
