## 🤔 Why Do You Need Networking in the Cloud?

When you create VMs, containers, or databases on GCP, they all need to **talk to each other** and to the **outside world**. Networking is the plumbing that makes it all work.

Think of it like building a house:
- **VPC** = The property/land where your house sits (your private space)
- **Subnets** = Rooms inside the house (different areas for different purposes)
- **Firewall Rules** = Doors and locks (who can enter which room)
- **Load Balancer** = A receptionist at the front desk (directs visitors to the right person)
- **Cloud DNS** = Your house's address in the phone book (so people can find you)

```
Internet
    │
    ▼
Cloud DNS (maps myapp.com → IP address)
    │
    ▼
Cloud Load Balancer (distributes traffic)
    │
    ├────────────┼────────────┐
    ▼            ▼            ▼
  VM-1         VM-2         VM-3
  (in Subnet-A, inside your VPC)
```

## 🌐 Part 1: VPC — Virtual Private Cloud

### What is a VPC?

A **VPC (Virtual Private Cloud)** is your own **private, isolated network** inside Google Cloud. It's like having your own private section of the internet where only your resources can communicate.

**Key facts:**
- A VPC is a **global resource** in GCP (it spans all regions!)
- Every GCP project gets a **default VPC** automatically
- Resources inside the same VPC can talk to each other using **private IP addresses**
- Resources in different VPCs cannot talk to each other (unless you set up peering)

```
Google Cloud
│
├── VPC: "production-vpc" (Global — spans all regions)
│   ├── Subnet: "us-subnet" (us-central1) → 10.0.1.0/24
│   │   ├── VM-1 (10.0.1.2)
│   │   └── VM-2 (10.0.1.3)
│   ├── Subnet: "eu-subnet" (europe-west1) → 10.0.2.0/24
│   │   └── VM-3 (10.0.2.2)
│   └── Subnet: "asia-subnet" (asia-east1) → 10.0.3.0/24
│       └── VM-4 (10.0.3.2)
│
└── VPC: "staging-vpc" (completely isolated from production!)
    └── Subnet: "staging-subnet" (us-central1) → 10.1.1.0/24
        └── VM-5 (10.1.1.2)

VM-1 can talk to VM-3 (same VPC, different regions) ✅
VM-1 CANNOT talk to VM-5 (different VPCs) ❌
```

### VPC vs Traditional Networking

| Traditional Data Center | GCP VPC |
|------------------------|---------|
| Buy physical routers, switches, cables | All virtual — click or command to create |
| Takes weeks to set up | Takes seconds |
| Limited by physical location | Spans the entire globe |
| Expensive hardware | Pay as you go |
| Hard to change | Modify instantly |

### GCP's Default VPC

When you create a GCP project, it comes with a **default VPC** that has:
- One subnet in every region (auto-created)
- Default firewall rules (allow SSH, RDP, ICMP)
- Auto mode subnets with predefined IP ranges

```bash
# See the default VPC
gcloud compute networks list

# Output:
# NAME     SUBNET_MODE  BGP_ROUTING_MODE  IPV4_RANGE  GATEWAY_IPV4
# default  AUTO         REGIONAL

# See subnets in the default VPC
gcloud compute networks subnets list --network=default

# Output (auto subnets in every region):
# NAME     REGION           NETWORK  RANGE
# default  us-central1      default  10.128.0.0/20
# default  europe-west1     default  10.132.0.0/20
# default  asia-east1       default  10.140.0.0/20
# ... (one in every region)
```

> ⚠️ **For production, always create a custom VPC!** The default VPC has overly permissive firewall rules and auto-assigned IP ranges that you can't control.

---

### Creating a Custom VPC

```bash
# Step 1: Create a custom VPC
gcloud compute networks create prod-vpc \
    --subnet-mode=custom \
    --bgp-routing-mode=regional

# --subnet-mode=custom → YOU define the subnets (recommended for production)
# --subnet-mode=auto   → Google auto-creates subnets in every region
# --bgp-routing-mode   → How routes are advertised (regional or global)
```

```bash
# Step 2: Create subnets in the regions you need

# US subnet
gcloud compute networks subnets create us-subnet \
    --network=prod-vpc \
    --region=us-central1 \
    --range=10.0.1.0/24 \
    --enable-private-ip-google-access

# Europe subnet
gcloud compute networks subnets create eu-subnet \
    --network=prod-vpc \
    --region=europe-west1 \
    --range=10.0.2.0/24 \
    --enable-private-ip-google-access

# Asia subnet
gcloud compute networks subnets create asia-subnet \
    --network=prod-vpc \
    --region=asia-east1 \
    --range=10.0.3.0/24 \
    --enable-private-ip-google-access

# Let's break down the subnet creation:
# --network=prod-vpc        → which VPC this subnet belongs to
# --region=us-central1      → which region
# --range=10.0.1.0/24       → IP address range (256 IPs: 10.0.1.0 to 10.0.1.255)
# --enable-private-ip-google-access → VMs without external IPs can still reach Google APIs
```

**Understanding CIDR notation:**

```
10.0.1.0/24
│       │
│       └── /24 means the first 24 bits are the network part
│           This gives you 256 IP addresses (10.0.1.0 to 10.0.1.255)
└── The network address

Common subnet sizes:
/24 = 256 IPs    (small — good for most subnets)
/20 = 4,096 IPs  (medium)
/16 = 65,536 IPs (large)
/8  = 16 million IPs (huge)
```

### Secondary IP Ranges (for GKE)

When using GKE, pods and services need their own IP ranges. You add these as secondary ranges:

```bash
# Create a subnet with secondary ranges for GKE
gcloud compute networks subnets create gke-subnet \
    --network=prod-vpc \
    --region=us-central1 \
    --range=10.0.10.0/24 \
    --secondary-range pods=10.1.0.0/16,services=10.2.0.0/20

# --range=10.0.10.0/24              → Node IPs (256 IPs for nodes)
# --secondary-range pods=10.1.0.0/16    → Pod IPs (65,536 IPs for pods)
# --secondary-range services=10.2.0.0/20 → Service IPs (4,096 IPs for K8s services)
```

### Firewall Rules — Controlling Traffic

Firewall rules control **what traffic is allowed in and out** of your VPC. By default, all incoming traffic is **denied** and all outgoing traffic is **allowed**.

```bash
# Allow SSH (port 22) from anywhere
gcloud compute firewall-rules create allow-ssh \
    --network=prod-vpc \
    --allow=tcp:22 \
    --source-ranges=0.0.0.0/0 \
    --target-tags=ssh-enabled \
    --description="Allow SSH from anywhere to instances tagged ssh-enabled"

# Let's break this down:
# --network=prod-vpc          → apply to this VPC
# --allow=tcp:22              → allow TCP traffic on port 22 (SSH)
# --source-ranges=0.0.0.0/0   → from anywhere (all IPs)
# --target-tags=ssh-enabled    → only apply to VMs with the tag "ssh-enabled"
```

```bash
# Allow HTTP and HTTPS from anywhere
gcloud compute firewall-rules create allow-web \
    --network=prod-vpc \
    --allow=tcp:80,tcp:443 \
    --source-ranges=0.0.0.0/0 \
    --target-tags=web-server

# Allow internal communication between all VMs in the VPC
gcloud compute firewall-rules create allow-internal \
    --network=prod-vpc \
    --allow=tcp:0-65535,udp:0-65535,icmp \
    --source-ranges=10.0.0.0/8

# This allows ALL traffic between resources with private IPs starting with 10.x.x.x
# Meaning: all VMs in your VPC can freely talk to each other
```

```bash
# Allow a specific application port only from a load balancer
gcloud compute firewall-rules create allow-app-from-lb \
    --network=prod-vpc \
    --allow=tcp:8080 \
    --source-ranges=130.211.0.0/22,35.191.0.0/16 \
    --target-tags=app-server

# The source-ranges here are Google's load balancer IP ranges
# This means ONLY the load balancer can reach port 8080 on your app servers
# Nobody else can directly access them! (Good security practice)
```

```bash
# Deny all traffic from a specific IP (block an attacker)
gcloud compute firewall-rules create deny-bad-ip \
    --network=prod-vpc \
    --action=DENY \
    --rules=all \
    --source-ranges=203.0.113.100/32 \
    --priority=100

# --action=DENY → block this traffic
# --priority=100 → lower number = higher priority (this rule wins over others)
```

```bash
# List all firewall rules
gcloud compute firewall-rules list --filter="network=prod-vpc"

# Delete a firewall rule
gcloud compute firewall-rules delete allow-ssh
```

**Firewall Rule Priority:**

```
Priority 0    = Highest priority (evaluated first)
Priority 65535 = Lowest priority (evaluated last)

If two rules conflict, the one with LOWER priority number wins.

Example:
Rule 1 (priority 100): DENY all traffic from 10.0.0.0/8
Rule 2 (priority 200): ALLOW tcp:22 from 10.0.0.0/8

Result: Traffic from 10.0.0.0/8 is DENIED (Rule 1 wins because 100 < 200)
```

### VPC Peering — Connecting Two VPCs

Sometimes you need VPCs to talk to each other. VPC Peering creates a private connection between two VPCs:

```bash
# Create peering from prod-vpc to staging-vpc
gcloud compute networks peerings create prod-to-staging \
    --network=prod-vpc \
    --peer-network=staging-vpc

# Create peering from staging-vpc to prod-vpc (must be done from both sides!)
gcloud compute networks peerings create staging-to-prod \
    --network=staging-vpc \
    --peer-network=prod-vpc

# Now VMs in prod-vpc can talk to VMs in staging-vpc using private IPs
```

> **Important:** VPC Peering is NOT transitive! If A peers with B and B peers with C, A cannot talk to C through B.

### Private Google Access & Cloud NAT

#### Private Google Access

Allows VMs **without external IPs** to reach Google APIs (like Cloud Storage, BigQuery):

```bash
# Enable Private Google Access on a subnet
gcloud compute networks subnets update us-subnet \
    --region=us-central1 \
    --enable-private-ip-google-access
```

#### Cloud NAT — Internet Access for Private VMs

VMs without external IPs can't reach the internet by default. Cloud NAT gives them outbound internet access without exposing them:

```bash
# Step 1: Create a Cloud Router
gcloud compute routers create my-router \
    --network=prod-vpc \
    --region=us-central1

# Step 2: Create Cloud NAT
gcloud compute routers nats create my-nat \
    --router=my-router \
    --region=us-central1 \
    --nat-all-subnet-ip-ranges \
    --auto-allocate-nat-external-ips

# Now VMs without external IPs can:
# ✅ Download packages (apt-get, yum)
# ✅ Pull Docker images
# ✅ Make API calls to external services
# ❌ Cannot be reached from the internet (they're still private!)
```

This is a **security best practice** — your VMs are hidden from the internet but can still reach out when needed.

In large organizations, you might want a central networking team to manage the VPC while other teams use it:

```bash
# In the host project (managed by the networking team):
gcloud compute shared-vpc enable HOST_PROJECT_ID

# Associate a service project (used by development teams):
gcloud compute shared-vpc associated-projects add SERVICE_PROJECT_ID \
    --host-project=HOST_PROJECT_ID
```

```
Organization
│
├── Host Project (networking team manages VPCs)

│       ├── Subnet-A (for Team 1)

│   └── Creates VMs/GKE in Subnet-A
│
└── Service Project: Team-2 (uses Subnet-B)
    └── Creates VMs/GKE in Subnet-B
```

## ⚖️ Part 2: Cloud Load Balancing

### What is Load Balancing?

Imagine a restaurant with one cashier. If 100 customers show up, there's a massive line. But if you have 5 cashiers, customers are spread across them and everyone gets served faster.

**Load Balancing = Distributing incoming traffic across multiple servers so no single server is overwhelmed.**

```
Without Load Balancer:
All traffic → VM-1 (overloaded! 💥) 
              VM-2 (idle... 😴)
              VM-3 (idle... 😴)

With Load Balancer:
All traffic → Load Balancer → VM-1 (33% traffic)
                             → VM-2 (33% traffic)
                             → VM-3 (33% traffic)
                             = Everyone happy! ✅
```

### Benefits of Load Balancing

| Benefit | Explanation |
|---------|-------------|
| **High Availability** | If one VM dies, traffic goes to the others |
| **Scalability** | Add more VMs behind the load balancer as traffic grows |
| **Performance** | No single server gets overwhelmed |
| **Health Checks** | Automatically stops sending traffic to unhealthy VMs |
| **SSL Termination** | Handle HTTPS at the load balancer, not at each VM |
| **Global Reach** | Route users to the nearest server |

---

### Types of Load Balancers in GCP

GCP has several types of load balancers. The choice depends on your use case:

#### By Layer:

```
Layer 7 (Application Layer — HTTP/HTTPS):
├── Understands HTTP headers, URLs, cookies
├── Can route based on URL path (/api → backend, /images → CDN)
├── Can do SSL termination
└── Best for: web applications, APIs

Layer 4 (Transport Layer — TCP/UDP):
├── Works with raw TCP/UDP connections
├── Doesn't understand HTTP
├── Lower latency (no HTTP parsing)
└── Best for: databases, game servers, non-HTTP services
```

#### GCP Load Balancer Types:

| Load Balancer | Layer | Scope | Use Case |
|--------------|-------|-------|----------|
| **External HTTP(S) LB** | L7 | Global | Web apps, APIs (most common!) |
| **Internal HTTP(S) LB** | L7 | Regional | Internal microservices |
| **External TCP/UDP Network LB** | L4 | Regional | Non-HTTP traffic |
| **Internal TCP/UDP LB** | L4 | Regional | Internal non-HTTP services |
| **External TCP Proxy LB** | L4 | Global | Global TCP traffic (non-HTTP) |
| **External SSL Proxy LB** | L4 | Global | Global SSL traffic |

**The most commonly used one is the External HTTP(S) Load Balancer** — let's focus on that.

---

### Setting Up an External HTTP(S) Load Balancer

This is the most common setup — distributing web traffic across multiple VMs:

```
Internet → Cloud DNS → HTTP(S) Load Balancer → Backend Service → Instance Group → VMs
```

Here's how to set it up step by step:

#### Step 1: Create VMs (we'll use a Managed Instance Group)

```bash
# Create an instance template
gcloud compute instance-templates create web-template \
    --machine-type=e2-medium \
    --image-family=ubuntu-2204-lts \
    --image-project=ubuntu-os-cloud \
    --tags=http-server \
    --metadata=startup-script='#!/bin/bash
apt-get update
apt-get install -y nginx
HOSTNAME=$(hostname)
echo "<h1>Hello from $HOSTNAME</h1>" > /var/www/html/index.html
systemctl start nginx'

# Create a managed instance group
gcloud compute instance-groups managed create web-group \
    --template=web-template \
    --size=3 \
    --zone=us-central1-a
```

#### Step 2: Create a Health Check

The load balancer uses health checks to know which VMs are healthy:

```bash
# Create a health check
gcloud compute health-checks create http web-health-check \
    --port=80 \
    --request-path=/

# What this does:
# Every few seconds, the LB sends a GET / request to port 80 on each VM
# If a VM responds with 200 OK → it's healthy (receives traffic)
# If a VM doesn't respond → it's unhealthy (no more traffic until it recovers)
```

#### Step 3: Create a Backend Service

```bash
# Create a backend service
gcloud compute backend-services create web-backend \
    --protocol=HTTP \
    --health-checks=web-health-check \
    --global

# Add the instance group as a backend
gcloud compute backend-services add-backend web-backend \
    --instance-group=web-group \
    --instance-group-zone=us-central1-a \
    --global
```

#### Step 4: Create a URL Map

The URL map tells the load balancer where to send requests:

```bash
# Simple URL map (all traffic goes to web-backend)
gcloud compute url-maps create web-url-map \
    --default-service=web-backend
```

For more complex routing:

```bash
# Create multiple backend services
gcloud compute backend-services create api-backend \
    --protocol=HTTP \
    --health-checks=web-health-check \
    --global

gcloud compute backend-services create static-backend \
    --protocol=HTTP \
    --health-checks=web-health-check \
    --global

# Create a URL map with path-based routing
gcloud compute url-maps create complex-url-map \
    --default-service=web-backend

# Add path rules
gcloud compute url-maps add-path-matcher complex-url-map \
    --path-matcher-name=my-paths \
    --default-service=web-backend \
    --path-rules="/api/*=api-backend,/static/*=static-backend"

# Now:
# myapp.com/api/*    → goes to api-backend
# myapp.com/static/* → goes to static-backend  
# myapp.com/*        → goes to web-backend (default)
```

#### Step 5: Create an HTTPS Proxy (with SSL)

```bash
# For HTTP only (no SSL):
gcloud compute target-http-proxies create web-http-proxy \
    --url-map=web-url-map

# For HTTPS (recommended for production!):
# First, create a managed SSL certificate
gcloud compute ssl-certificates create my-cert \
    --domains=myapp.com,www.myapp.com \
    --global

# Then create the HTTPS proxy
gcloud compute target-https-proxies create web-https-proxy \
    --url-map=web-url-map \
    --ssl-certificates=my-cert
```

#### Step 6: Create a Global Forwarding Rule (the front door)

```bash
# Reserve a static external IP
gcloud compute addresses create web-lb-ip --global

# Get the IP address
gcloud compute addresses describe web-lb-ip --global

# Create the forwarding rule (HTTP)
gcloud compute forwarding-rules create web-http-rule \
    --address=web-lb-ip \
    --global \
    --target-http-proxy=web-http-proxy \
    --ports=80

# Create the forwarding rule (HTTPS)
gcloud compute forwarding-rules create web-https-rule \
    --address=web-lb-ip \
    --global \
    --target-https-proxy=web-https-proxy \
    --ports=443
```

**That's it!** Your load balancer is now live. Traffic hitting the external IP gets distributed across your VMs.

### Internal Load Balancer (for Microservices)

For internal services that shouldn't be exposed to the internet:

```bash
# Create an internal HTTP(S) load balancer

    --region=us-central1 \
    --load-balancing-scheme=INTERNAL_MANAGED \

# This LB gets a PRIVATE IP (e.g., 10.0.1.100)
# Only resources inside your VPC can reach it
# Perfect for: microservice A calling microservice B
```

### Network Load Balancer (Layer 4 — TCP/UDP)

For non-HTTP traffic (databases, game servers, etc.):

```bash
# Create a TCP network load balancer
gcloud compute forwarding-rules create tcp-lb \
    --region=us-central1 \
    --ports=5432 \
    --target-pool=db-pool

# Use for: PostgreSQL, MySQL, Redis, custom TCP protocols
```

### Load Balancing Algorithms

GCP uses several methods to distribute traffic:

| Algorithm | How It Works |
|-----------|-------------|
| **Round Robin** | Send to each server in turn: 1, 2, 3, 1, 2, 3... |
| **Least Connections** | Send to the server with the fewest active connections |
| **CPU Utilization** | Send to the server using the least CPU |
| **Requests Per Second** | Balance based on request rate |

```
Without CDN:

With CDN:
User in Tokyo → Cached at Tokyo edge → Response (fast! 10ms)
```

## 🌍 Part 3: Cloud DNS

### What is Cloud DNS?

Cloud DNS is Google's **managed DNS service**. It translates human-friendly domain names (like `myapp.com`) into IP addresses (like `34.120.0.1`) so browsers know where to send requests.

```
User types: www.myapp.com
    │
    ▼
DNS Resolver: "What's the IP for www.myapp.com?"
    │
    ▼
Cloud DNS: "It's 34.120.0.1"
    │
    ▼
Browser connects to 34.120.0.1
    │
    ▼
Your application responds! ✅
```

### DNS Record Types

Before we set up Cloud DNS, let's understand the different record types:

| Record Type | Purpose | Example |
|------------|---------|---------|
| **A** | Maps domain to IPv4 address | `myapp.com → 34.120.0.1` |
| **AAAA** | Maps domain to IPv6 address | `myapp.com → 2001:db8::1` |
| **CNAME** | Maps domain to another domain (alias) | `www.myapp.com → myapp.com` |
| **MX** | Mail server records | `myapp.com → mail.google.com` |
| **TXT** | Text records (verification, SPF, DKIM) | `myapp.com → "v=spf1 include:..."` |
| **NS** | Name server records | `myapp.com → ns-cloud-a1.googledomains.com` |
| **SOA** | Start of Authority (zone metadata) | Auto-created by Cloud DNS |
| **SRV** | Service location records | Used for service discovery |

---

### Setting Up Cloud DNS

#### Step 1: Create a DNS Zone

A DNS zone is a container for all DNS records for a domain:

```bash
# Create a public DNS zone
gcloud dns managed-zones create myapp-zone \
    --dns-name="myapp.com." \
    --description="DNS zone for myapp.com" \
    --visibility=public

# Note the trailing dot in --dns-name="myapp.com."
# This is standard DNS notation!
```

```bash
# View the zone details
gcloud dns managed-zones describe myapp-zone

# Output:
# creationTime: '2024-01-15T10:00:00.000Z'
# description: DNS zone for myapp.com
# dnsName: myapp.com.
# nameServers:
# - ns-cloud-a1.googledomains.com.
# - ns-cloud-a2.googledomains.com.
# - ns-cloud-a3.googledomains.com.
# - ns-cloud-a4.googledomains.com.
# visibility: public

# IMPORTANT: You need to update your domain registrar (GoDaddy, Namecheap, etc.)
# to point to these name servers!
```

#### Step 2: Add DNS Records

```bash
# Add an A record (domain → IP address)
gcloud dns record-sets create myapp.com. \
    --zone=myapp-zone \
    --type=A \
    --ttl=300 \
    --rrdatas="34.120.0.1"

# --type=A          → A record (IPv4)
# --ttl=300         → Time To Live (300 seconds = 5 minutes)
#                     After 5 minutes, DNS resolvers will re-check this record
# --rrdatas=34.120.0.1 → The IP address
```

```bash
# Add a CNAME record (www → root domain)
gcloud dns record-sets create www.myapp.com. \
    --zone=myapp-zone \
    --type=CNAME \
    --ttl=300 \
    --rrdatas="myapp.com."

# Now www.myapp.com points to myapp.com
# Which points to 34.120.0.1
```

```bash
# Add an A record for a subdomain
gcloud dns record-sets create api.myapp.com. \
    --zone=myapp-zone \
    --type=A \
    --ttl=300 \
    --rrdatas="34.120.0.2"

# Now api.myapp.com points to a different IP
```


### Using Transaction for Batch Changes

When you need to make multiple DNS changes atomically (all at once):

```bash
# Start a transaction
gcloud dns record-sets transaction start --zone=myapp-zone

# Add records
gcloud dns record-sets transaction add "34.120.0.1" \
    --name=myapp.com. \
    --type=A \
    --ttl=300 \
    --zone=myapp-zone

gcloud dns record-sets transaction add "myapp.com." \
    --name=www.myapp.com. \
    --type=CNAME \
    --ttl=300 \
    --zone=myapp-zone

# Remove a record
gcloud dns record-sets transaction remove "34.120.0.2" \
    --name=old.myapp.com. \
    --type=A \
    --ttl=300 \
    --zone=myapp-zone

# Execute all changes at once
gcloud dns record-sets transaction execute --zone=myapp-zone

# If something goes wrong, abort:
gcloud dns record-sets transaction abort --zone=myapp-zone
```

### DNS Routing Policies

Cloud DNS supports different routing policies:

#### Geolocation Routing

Route users to the nearest server based on their location:

```bash
# Create a geolocation routing policy
gcloud dns record-sets create myapp.com. \
    --zone=myapp-zone \
    --type=A \
    --ttl=300 \
    --routing-policy-type=GEO \
    --routing-policy-data="us-central1=34.120.0.1;europe-west1=34.121.0.1;asia-east1=34.122.0.1"

# Now:
# Users in North America → 34.120.0.1 (US server)
# Users in Europe        → 34.121.0.1 (EU server)
# Users in Asia          → 34.122.0.1 (Asia server)
```

#### Weighted Round Robin

Distribute traffic by percentage:

```bash
# Send 80% of traffic to one IP, 20% to another
gcloud dns record-sets create myapp.com. \
    --zone=myapp-zone \
    --type=A \
    --ttl=300 \
    --routing-policy-type=WRR \
    --routing-policy-data="0.8=34.120.0.1;0.2=34.120.0.2"

# 80% traffic → 34.120.0.1 (main server)
# 20% traffic → 34.120.0.2 (new server — canary testing!)
```

#### Failover

Route to a backup server if the primary is down:

```bash
# Primary and backup configuration
gcloud dns record-sets create myapp.com. \
    --zone=myapp-zone \
    --type=A \
    --ttl=60 \
    --routing-policy-type=FAILOVER \
    --routing-policy-primary-data="34.120.0.1" \
    --routing-policy-backup-data-type=GEO \
    --routing-policy-backup-data="us-central1=34.120.0.2" \
    --backup-data-trickle-ratio=0.0 \
    --enable-health-checking

# Normal: All traffic → 34.120.0.1 (primary)
# If primary is down: All traffic → 34.120.0.2 (backup)
```

## 🧪 Putting It All Together — Complete Networking Setup

Let's build a complete, production-ready networking setup:

```bash
# ==========================================
# STEP 1: Create the VPC and Subnets
# ==========================================

gcloud compute networks create prod-vpc \
    --subnet-mode=custom

gcloud compute networks subnets create web-subnet \
    --network=prod-vpc \
    --region=us-central1 \
    --range=10.0.1.0/24

gcloud compute networks subnets create db-subnet \
    --network=prod-vpc \
    --region=us-central1 \
    --range=10.0.2.0/24

# ==========================================
# STEP 2: Firewall Rules
# ==========================================

# Allow health checks from Google's LB
gcloud compute firewall-rules create allow-health-check \
    --network=prod-vpc \
    --allow=tcp:80,tcp:443,tcp:8080 \
    --source-ranges=130.211.0.0/22,35.191.0.0/16

# Allow SSH for management
gcloud compute firewall-rules create allow-ssh \
    --network=prod-vpc \
    --allow=tcp:22 \
    --source-ranges=YOUR_OFFICE_IP/32 \
    --target-tags=ssh-enabled

# Allow internal communication
gcloud compute firewall-rules create allow-internal \
    --network=prod-vpc \
    --allow=tcp:0-65535,udp:0-65535,icmp \
    --source-ranges=10.0.0.0/8

# ==========================================
# STEP 3: Create VMs with Instance Group
# ==========================================

gcloud compute instance-templates create web-template \
    --machine-type=e2-medium \
    --network=prod-vpc \
    --subnet=web-subnet \
    --image-family=ubuntu-2204-lts \
    --image-project=ubuntu-os-cloud \
    --tags=http-server,ssh-enabled \
    --metadata=startup-script='#!/bin/bash
apt-get update && apt-get install -y nginx
echo "<h1>Hello from $(hostname)</h1>" > /var/www/html/index.html'

gcloud compute instance-groups managed create web-group \
    --template=web-template \
    --size=3 \
    --zone=us-central1-a

gcloud compute instance-groups managed set-named-ports web-group \
    --named-ports=http:80 \
    --zone=us-central1-a

# ==========================================
# STEP 4: Create Load Balancer
# ==========================================

gcloud compute health-checks create http web-hc --port=80

gcloud compute backend-services create web-backend \
    --protocol=HTTP \
    --health-checks=web-hc \
    --global

gcloud compute backend-services add-backend web-backend \
    --instance-group=web-group \
    --instance-group-zone=us-central1-a \
    --global

gcloud compute url-maps create web-map \
    --default-service=web-backend

gcloud compute ssl-certificates create web-cert \
    --domains=myapp.com \
    --global

gcloud compute target-https-proxies create web-proxy \
    --url-map=web-map \
    --ssl-certificates=web-cert

gcloud compute addresses create web-ip --global

gcloud compute forwarding-rules create web-rule \
    --address=web-ip \
    --global \
    --target-https-proxy=web-proxy \
    --ports=443

# ==========================================
# STEP 5: Set Up DNS
# ==========================================

gcloud dns managed-zones create myapp-zone \
    --dns-name=myapp.com. \
    --visibility=public

# Get the load balancer IP
LB_IP=$(gcloud compute addresses describe web-ip --global --format="get(address)")

gcloud dns record-sets create myapp.com. \
    --zone=myapp-zone \
    --type=A \
    --ttl=300 \
    --rrdatas="$LB_IP"

gcloud dns record-sets create www.myapp.com. \
    --zone=myapp-zone \
    --type=CNAME \
    --ttl=300 \
    --rrdatas="myapp.com."

# ==========================================
# DONE! Your app is now accessible at:
# https://myapp.com
# https://www.myapp.com
# ==========================================
```

## 🧠 Summary

```
VPC (Virtual Private Cloud)
├── Your private, isolated network in GCP
├── Global resource (spans all regions)
├── Contains subnets (regional, with IP ranges)
├── Firewall rules control traffic in/out
├── VPC Peering connects two VPCs
├── Cloud NAT gives private VMs internet access
└── Best practice: Custom VPC for production (not default!)

Cloud Load Balancing
├── Distributes traffic across multiple servers
├── L7 (HTTP/S) → understands URLs, headers (most common)
├── L4 (TCP/UDP) → raw connections (databases, games)
├── Health checks automatically remove unhealthy servers
├── SSL termination handles HTTPS for you
├── Cloud CDN caches content at edge locations
└── Components: Forwarding Rule → Proxy → URL Map → Backend → Instance Group

Cloud DNS
├── Managed DNS service (translates domains → IPs)
├── Public zones → internet-facing domains
├── Private zones → internal service discovery
├── Supports: A, AAAA, CNAME, MX, TXT, SRV, NS records
├── Routing policies: Geolocation, Weighted, Failover
├── DNSSEC for security
└── 100% uptime SLA!
```

**Key Takeaways:**
1. **Always use a custom VPC** in production — the default VPC is too permissive
2. **Use firewall rules** to restrict traffic to only what's needed
3. **Use Cloud NAT** so private VMs can reach the internet without being exposed
4. **HTTP(S) Load Balancer** is your go-to for web applications
5. **Cloud DNS** with routing policies enables global traffic management
6. **Everything connects:** DNS → Load Balancer → VPC → VMs
