## 1. Introduction — Why Data Storage Matters in GCP

Every application needs a place to store its data — whether it's images, database records, user profiles, or log files. Google Cloud Platform gives you **multiple storage options**, each designed for a different kind of data and access pattern.

Think of it like this:
- Need to store **files** (images, videos, backups, logs)? → **Cloud Storage** (like Google Drive, but for your apps)
- Need a traditional **relational database** (tables, rows, SQL queries)? → **Cloud SQL**
- Need a **flexible, scalable NoSQL database** (documents, collections)? → **Firestore**

The key is picking the **right tool for the job**. Let's dive into each one.

---

## 2. GCP Cloud Storage (Object Storage)

### 2.1 What is Cloud Storage?

**Cloud Storage** is Google's **object storage** service. It lets you store and retrieve any amount of data — files, images, videos, backups, Terraform state files, container images… anything.

Think of it like a giant, infinitely scalable folder system in the cloud. It's similar to:
- **AWS S3** (Amazon's object storage)
- **Azure Blob Storage** (Microsoft's object storage)

**Object Storage** means:
- You don't deal with folders/directories on a file system
- Every file is an **"object"** stored inside a **"bucket"**
- Each object has a unique key (basically its path/name)
- It's **flat** — folders are simulated using `/` in the key name

---

### 2.2 Key Concepts

| Concept | What It Is | Example |
|---------|-----------|---------|
| **Bucket** | A container that holds your objects. Globally unique name. | `my-company-backups` |
| **Object** | A single file stored in a bucket | `logs/2024/app.log` |
| **Region** | Where your bucket lives geographically | `us-central1`, `asia-south1` |
| **Multi-Region** | Data replicated across multiple regions for high availability | `us`, `eu`, `asia` |
| **Dual-Region** | Data in exactly two regions (balance between cost & availability) | `us-central1` + `us-east1` |

#### Bucket Naming Rules
- Must be **globally unique** across all of GCP (like a domain name)
- Only lowercase letters, numbers, hyphens, underscores, and dots
- Must start and end with a letter or number
- Between 3–63 characters

---

### 2.3 Storage Classes

Not all data is accessed equally. Some data you need every second, some you might not look at for months. GCP charges you based on **how often** you access data and **how long** you store it.

| Storage Class | Best For | Min Storage Duration | Access Cost | Storage Cost |
|--------------|---------|---------------------|-------------|-------------|
| **Standard** | Frequently accessed data (hot data) | None | Lowest | Highest |
| **Nearline** | Data accessed less than once a month | 30 days | Medium | Medium |
| **Coldline** | Data accessed less than once a quarter | 90 days | Higher | Lower |
| **Archive** | Data accessed less than once a year (compliance, backups) | 365 days | Highest | Lowest |

**Simple Way to Remember:**
- Hot data → **Standard** (you touch it a lot)
- Warm data → **Nearline** (once a month-ish)
- Cold data → **Coldline** (once every few months)
- Frozen data → **Archive** (almost never, but need to keep it)

> **Note:** "Min Storage Duration" means if you delete an object before that time, you still pay for the full duration. For example, if you store something in Coldline and delete it after 30 days, you pay for 90 days.

---

### 2.4 Access Control

Cloud Storage gives you **multiple ways to control who can access your data**:

#### a) IAM (Identity and Access Management) — Recommended

This controls access at the **bucket level** using roles:

| Role | What It Can Do |
|------|---------------|
| `roles/storage.objectViewer` | Read objects |
| `roles/storage.objectCreator` | Create/upload objects |
| `roles/storage.objectAdmin` | Read, write, delete objects |
| `roles/storage.admin` | Full control over buckets + objects |

```bash
# Grant a user read access to a bucket
gcloud storage buckets add-iam-policy-binding gs://my-bucket \
  --member="user:john@example.com" \
  --role="roles/storage.objectViewer"
```

#### b) ACLs (Access Control Lists) — Legacy

Fine-grained, per-object access control. Not recommended for new projects — use IAM instead.

#### c) Signed URLs — Temporary Access

Give someone **time-limited access** to a specific object without needing a Google account. Perfect for sharing files temporarily.

```bash
# Generate a signed URL that expires in 1 hour
gcloud storage sign-url gs://my-bucket/report.pdf \
  --duration=1h \
  --private-key-file=key.json
```

This generates a long URL like:
```
https://storage.googleapis.com/my-bucket/report.pdf?X-Goog-Signature=abc123...&X-Goog-Expires=3600
```

Anyone with this URL can download the file for 1 hour. After that, the link stops working.

#### d) Uniform vs Fine-Grained Access

| Mode | Description |
|------|------------|
| **Uniform** | All objects in the bucket share the same IAM permissions. Simpler, recommended. |
| **Fine-Grained** | Each object can have its own ACL. More complex, legacy approach. |

### 2.5 Versioning & Lifecycle Management

#### Object Versioning

When you enable versioning, GCP **keeps old copies** of objects when you overwrite or delete them.

```bash
# Enable versioning on a bucket
gcloud storage buckets update gs://my-bucket --versioning

# List all versions of an object
gcloud storage ls --all-versions gs://my-bucket/config.yaml
```

**Why is this useful?**
- Accidentally deleted a file? Restore the old version!
- Someone overwrote a config? Get the previous one back!
- Great for **Terraform state files** — never lose your state

#### Lifecycle Rules

You can automatically **move, delete, or change storage class** of objects based on rules. This saves money!

**Example: Automatically move old logs to cheaper storage**

```json
{
  "lifecycle": {
    "rule": [
      {
        "action": {
          "type": "SetStorageClass",
          "storageClass": "NEARLINE"
        },
        "condition": {
          "age": 30
        }
      },
      {
        "action": {
          "type": "SetStorageClass",
          "storageClass": "COLDLINE"
        },
        "condition": {
          "age": 90
        }
      },
      {
        "action": {
          "type": "Delete"
        },
        "condition": {
          "age": 365
        }
      }
    ]
  }
}
```

What this does:
1. After **30 days** → Move to **Nearline** (cheaper)
2. After **90 days** → Move to **Coldline** (even cheaper)
3. After **365 days** → **Delete** the object

Apply it:
```bash
gcloud storage buckets update gs://my-logs-bucket \
  --lifecycle-file=lifecycle.json
```

### 2.6 Hands-On with gsutil & Console

`gsutil` is the old CLI tool, but Google now recommends `gcloud storage`. Both work, but let's use the modern one:

#### Creating a Bucket

```bash
# Create a bucket in us-central1 with Standard storage
gcloud storage buckets create gs://my-devops-bucket-2024 \
  --location=us-central1 \
  --default-storage-class=STANDARD \
  --uniform-bucket-level-access
```

#### Uploading Files

```bash
# Upload a single file
gcloud storage cp ./backup.tar.gz gs://my-devops-bucket-2024/backups/

# Upload an entire directory (recursively)
gcloud storage cp -r ./logs/ gs://my-devops-bucket-2024/logs/

# Upload with a specific storage class
gcloud storage cp ./archive.zip gs://my-devops-bucket-2024/ \
  --storage-class=COLDLINE
```

#### Downloading Files

```bash
# Download a file
gcloud storage cp gs://my-devops-bucket-2024/backups/backup.tar.gz ./

# Download entire folder
gcloud storage cp -r gs://my-devops-bucket-2024/logs/ ./local-logs/
```

#### Listing & Deleting

```bash
# List all buckets
gcloud storage buckets list

# List objects in a bucket
gcloud storage ls gs://my-devops-bucket-2024/

# List with details (size, date)
gcloud storage ls -l gs://my-devops-bucket-2024/

# Delete a file
gcloud storage rm gs://my-devops-bucket-2024/old-file.txt

# Delete entire bucket (must be empty first, or use --recursive)
gcloud storage rm -r gs://my-devops-bucket-2024/
```

#### Syncing (Like rsync)

```bash
# Sync local directory with bucket (only uploads changes)
gcloud storage rsync ./my-website/ gs://my-website-bucket/ --recursive

# Sync and delete files in bucket that don't exist locally
gcloud storage rsync ./my-website/ gs://my-website-bucket/ \
  --recursive --delete-unmatched-destination-objects
```

### 2.7 Terraform Example

Here's how you'd create a Cloud Storage bucket using Terraform (Infrastructure as Code):

```hcl
# Create a Cloud Storage bucket
resource "google_storage_bucket" "app_backups" {
  name          = "my-company-app-backups-2024"
  location      = "US"                        # Multi-region
  storage_class = "STANDARD"
  
  # Enable uniform access (IAM only, no ACLs)
  uniform_bucket_level_access = true
  
  # Enable versioning
  versioning {
    enabled = true
  }
  
  # Lifecycle rules — auto-manage old data
  lifecycle_rule {
    condition {
      age = 30    # After 30 days
    }
    action {
      type          = "SetStorageClass"
      storage_class = "NEARLINE"
    }
  }
  
  lifecycle_rule {
    condition {
      age = 90    # After 90 days
    }
    action {
      type          = "SetStorageClass"
      storage_class = "COLDLINE"
    }
  }
  
  lifecycle_rule {
    condition {
      age = 365   # After 1 year
    }
    action {
      type = "Delete"
    }
  }
  
  # Labels for organization
  labels = {
    environment = "production"
    team        = "devops"
  }
}

# Upload an object to the bucket
resource "google_storage_bucket_object" "readme" {
  name   = "README.md"
  bucket = google_storage_bucket.app_backups.name
  source = "./README.md"
}

# Grant a service account access to the bucket
resource "google_storage_bucket_iam_member" "viewer" {
  bucket = google_storage_bucket.app_backups.name
  role   = "roles/storage.objectViewer"
  member = "serviceAccount:my-app@my-project.iam.gserviceaccount.com"
}
```

**What this does:**
- Creates a multi-region bucket in the US
- Enables versioning (keeps old copies of files)
- Automatically moves data to cheaper storage as it ages
- Uploads a README file
- Gives a service account read-only access

## 3. GCP Cloud SQL

### 3.1 What is Cloud SQL?

**Cloud SQL** is Google's **fully managed relational database** service. It runs traditional SQL databases (MySQL, PostgreSQL, SQL Server) without you having to manage the server, patching, backups, or replication.

Think of it like this:
- You want a **MySQL or PostgreSQL database**
- But you **don't want to install it on a VM**, manage updates, handle backups, set up replication manually
- Cloud SQL does all of that for you — you just use the database

**It's similar to:**
- **AWS RDS** (Amazon's managed database)
- **Azure Database for MySQL/PostgreSQL**

---

### 3.2 Supported Databases

| Database | Versions | Use Case |
|----------|---------|----------|
| **MySQL** | 5.7, 8.0 | Most popular, widely used, great for web apps |
| **PostgreSQL** | 12, 13, 14, 15, 16 | Advanced features, JSON support, extensions |
| **SQL Server** | 2017, 2019, 2022 | Microsoft ecosystem, .NET applications |

> **Which one should you pick?**
> - **MySQL** → Simple, fast, widely supported. Great default choice.
> - **PostgreSQL** → More features, better for complex queries, JSON data, geospatial data.
> - **SQL Server** → Only if you're already in the Microsoft ecosystem.

---

### 3.3 Key Features

| Feature | What It Does |
|---------|-------------|
| **Automated Backups** | Daily backups, on-demand backups, point-in-time recovery |
| **High Availability (HA)** | Automatic failover to standby instance in another zone |
| **Read Replicas** | Scale read-heavy workloads by creating read-only copies |
| **Encryption** | Data encrypted at rest and in transit (by default!) |
| **Private IP** | Connect via VPC private networking (no public internet) |
| **Maintenance Windows** | You choose when GCP applies patches/updates |
| **Storage Auto-Resize** | Disk grows automatically as data grows |
| **IAM Authentication** | Use GCP IAM instead of database passwords |
| **Cloud SQL Proxy** | Secure, encrypted connection without whitelisting IPs |

---

### 3.4 Creating & Connecting to Cloud SQL

#### Creating an Instance via CLI

```bash
# Create a PostgreSQL instance
gcloud sql instances create my-postgres-db \
  --database-version=POSTGRES_15 \
  --tier=db-custom-2-8192 \
  --region=us-central1 \
  --availability-type=REGIONAL \
  --storage-size=50GB \
  --storage-auto-increase \
  --backup-start-time=03:00 \
  --enable-bin-log \
  --root-password=MySecurePassword123!
```

Let's break this down:
- `--database-version=POSTGRES_15` → Use PostgreSQL 15
- `--tier=db-custom-2-8192` → 2 vCPUs, 8 GB RAM
- `--region=us-central1` → Deploy in US Central
- `--availability-type=REGIONAL` → High availability (standby in another zone)
- `--storage-size=50GB` → Start with 50 GB disk
- `--storage-auto-increase` → Disk grows automatically if it fills up
- `--backup-start-time=03:00` → Daily backup at 3 AM UTC
- `--root-password` → Password for the `postgres` user

#### Common Machine Tiers

| Tier | vCPUs | RAM | Good For |
|------|-------|-----|----------|
| `db-f1-micro` | Shared | 0.6 GB | Testing only |
| `db-g1-small` | Shared | 1.7 GB | Small dev environments |
| `db-custom-1-3840` | 1 | 3.75 GB | Light production |
| `db-custom-2-8192` | 2 | 8 GB | Medium production |
| `db-custom-4-16384` | 4 | 16 GB | Heavy production |
| `db-custom-8-32768` | 8 | 32 GB | Large-scale production |

#### Creating a Database & User

```bash
# Create a database inside the instance
gcloud sql databases create myapp \
  --instance=my-postgres-db

# Create a user
gcloud sql users create appuser \
  --instance=my-postgres-db \
  --password=AppUserPassword456!
```

#### Connecting to Cloud SQL

**Option 1: Cloud SQL Auth Proxy (Recommended)**

The Cloud SQL Auth Proxy creates a secure, encrypted tunnel to your database without exposing it to the public internet.

```bash
# Download the proxy
curl -o cloud-sql-proxy \
  https://storage.googleapis.com/cloud-sql-connectors/cloud-sql-proxy/v2.8.0/cloud-sql-proxy.linux.amd64
chmod +x cloud-sql-proxy

# Start the proxy (it listens on localhost:5432)
./cloud-sql-proxy my-project:us-central1:my-postgres-db

# Now connect using any PostgreSQL client
psql -h 127.0.0.1 -U appuser -d myapp
```

**Option 2: Private IP (Within VPC)**

If your app runs on GCP (GKE, GCE, Cloud Run), use Private IP to connect directly through the VPC without going over the internet.

```bash
# Get the private IP of your Cloud SQL instance
gcloud sql instances describe my-postgres-db \
  --format='value(ipAddresses.ipAddress)'
```

**Option 3: Public IP + Authorized Networks**

Not recommended for production, but useful for quick testing:

```bash
# Whitelist your IP address
gcloud sql instances patch my-postgres-db \
  --authorized-networks=203.0.113.50/32

# Connect directly
psql -h <PUBLIC_IP> -U appuser -d myapp
```

### 3.5 High Availability & Replicas

#### High Availability (HA)

When you enable HA (`--availability-type=REGIONAL`), GCP creates:
1. A **primary instance** in one zone (e.g., `us-central1-a`)
2. A **standby instance** in another zone (e.g., `us-central1-b`)

If the primary fails, GCP **automatically fails over** to the standby. Your connection string stays the same — the switch is transparent.

```
Normal Operation:
┌─────────────┐     ┌─────────────┐
│   Primary    │────▶│   Standby   │
│ us-central1-a│     │ us-central1-b│
│  (Active)    │sync │  (Passive)  │
└─────────────┘     └─────────────┘
        ▲
        │
   Your App

After Failover:
┌─────────────┐     ┌─────────────┐
│   Primary    │     │   Standby   │
│ us-central1-a│     │ us-central1-b│
│   (Down!)    │     │  (Now Active)│
└─────────────┘     └─────────────┘
                          ▲
                          │
                     Your App (auto-reconnects)
```

#### Read Replicas

Read replicas are **read-only copies** of your database. They help you scale read-heavy workloads (like dashboards, reports, analytics).

```bash
# Create a read replica
gcloud sql instances create my-postgres-replica \
  --master-instance-name=my-postgres-db \
  --region=us-central1 \
  --tier=db-custom-2-8192

# Your app architecture:
# Writes → Primary instance
# Reads  → Read replica(s)
```

You can create replicas in:
- **Same region** → Low latency reads
- **Different region** → Cross-region disaster recovery

### 3.6 Backups & Maintenance

#### Automated Backups

Cloud SQL automatically takes **daily backups** and keeps them for 7 days (configurable up to 365 days).

```bash
# Configure backup settings
gcloud sql instances patch my-postgres-db \
  --backup-start-time=02:00 \
  --retained-backups-count=30 \
  --enable-point-in-time-recovery

# Create an on-demand backup
gcloud sql backups create --instance=my-postgres-db

# List backups
gcloud sql backups list --instance=my-postgres-db

# Restore from a backup
gcloud sql backups restore <BACKUP_ID> \
  --restore-instance=my-postgres-db
```

#### Point-in-Time Recovery (PITR)

PITR lets you restore your database to **any specific second** within the retention period. It uses binary logs (MySQL) or WAL logs (PostgreSQL).

```bash
# Restore to a specific time
gcloud sql instances clone my-postgres-db my-postgres-restored \
  --point-in-time='2024-06-15T10:30:00Z'
```

This is like a time machine for your database! 🕐

#### Maintenance Windows

```bash
# Set maintenance to Saturday at 2 AM
gcloud sql instances patch my-postgres-db \
  --maintenance-window-day=SAT \
  --maintenance-window-hour=2 \
  --maintenance-release-channel=production
```

### 3.7 Terraform Example

```hcl
# Cloud SQL PostgreSQL Instance
resource "google_sql_database_instance" "main" {
  name             = "my-app-postgres"
  database_version = "POSTGRES_15"
  region           = "us-central1"
  
  # Prevent accidental deletion
  deletion_protection = true
  
  settings {
    tier              = "db-custom-2-8192"   # 2 vCPU, 8GB RAM
    availability_type = "REGIONAL"            # High Availability
    disk_size         = 50                    # GB
    disk_autoresize   = true
    disk_type         = "PD_SSD"
    
    # Backup configuration
    backup_configuration {
      enabled                        = true
      start_time                     = "03:00"
      point_in_time_recovery_enabled = true
      transaction_log_retention_days = 7
      
      backup_retention_settings {
        retained_backups = 30
        retention_unit   = "COUNT"
      }
    }
    
    # Maintenance window
    maintenance_window {
      day          = 7    # Sunday
      hour         = 3    # 3 AM
      update_track = "stable"
    }
    
    # Private IP (no public access)
    ip_configuration {
      ipv4_enabled    = false          # No public IP
      private_network = "projects/my-project/global/networks/my-vpc"
      require_ssl     = true
    }
    
    # Database flags (PostgreSQL tuning)
    database_flags {
      name  = "max_connections"
      value = "200"
    }
    
    database_flags {
      name  = "log_min_duration_statement"
      value = "1000"   # Log queries taking more than 1 second
    }
  }
}

# Create a database
resource "google_sql_database" "app_db" {
  name     = "myapp"
  instance = google_sql_database_instance.main.name
}

# Create a user
resource "google_sql_user" "app_user" {
  name     = "appuser"
  instance = google_sql_database_instance.main.name
  password = var.db_password   # Use a variable, never hardcode!
}

# Read Replica
resource "google_sql_database_instance" "read_replica" {
  name                 = "my-app-postgres-replica"
  master_instance_name = google_sql_database_instance.main.name
  region               = "us-central1"
  database_version     = "POSTGRES_15"
  
  replica_configuration {
    failover_target = false
  }
  
  settings {
    tier            = "db-custom-2-8192"
    disk_autoresize = true
    
    ip_configuration {
      ipv4_enabled    = false
      private_network = "projects/my-project/global/networks/my-vpc"
    }
  }
}
```

**What this does:**
- Creates a **HA PostgreSQL 15** instance with private IP (no public internet access)
- Enables **daily backups** with 30-day retention and point-in-time recovery
- Sets a **maintenance window** for Sunday at 3 AM
- Creates a **database** and **user**
- Creates a **read replica** for scaling read operations
- Requires **SSL** for all connections

## 4. GCP Firestore

### 4.1 What is Firestore?

**Firestore** is Google's **fully managed, serverless, NoSQL document database**. It stores data as **documents** inside **collections** — think of it like a giant JSON database.

It's designed for:
- **Real-time applications** (chat apps, live dashboards)
- **Mobile and web apps** (direct client-side access)
- **Serverless backends** (pairs perfectly with Cloud Functions)
- **Flexible schemas** (no fixed table structure — each document can have different fields)

**Similar to:**
- **MongoDB** (document-based NoSQL)
- **AWS DynamoDB** (managed NoSQL, but different data model)
- **Azure Cosmos DB** (multi-model NoSQL)

---

### 4.2 Firestore vs Datastore

Firestore actually **replaced** Google Cloud Datastore. Here's how they compare:

| Feature | Firestore (Native Mode) | Datastore (Legacy / Firestore in Datastore Mode) |
|---------|------------------------|--------------------------------------------------|
| Real-time listeners | ✅ Yes | ❌ No |
| Offline support | ✅ Yes | ❌ No |
| Mobile/Web SDKs | ✅ Direct client access | ❌ Server-side only |
| Strongly consistent | ✅ Always | ⚠️ Eventually consistent (some queries) |
| Recommended for | New projects | Legacy projects already using Datastore |

> **Bottom line:** For new projects, always use **Firestore in Native Mode**. Datastore mode is just for backward compatibility.

---

### 4.3 Data Model

Firestore organizes data in a hierarchy:

```
Firestore Database
│
├── Collection: "users"
│   ├── Document: "user123"
│   │   ├── name: "Ritesh"
│   │   ├── email: "ritesh@example.com"
│   │   ├── age: 28
│   │   └── Sub-Collection: "orders"
│   │       ├── Document: "order001"
│   │       │   ├── product: "Laptop"
│   │       │   ├── price: 999.99
│   │       │   └── date: 2024-06-15
│   │       └── Document: "order002"
│   │           ├── product: "Mouse"
│   │           ├── price: 25.00
│   │           └── date: 2024-06-20
│   │
│   └── Document: "user456"
│       ├── name: "Priya"
│       ├── email: "priya@example.com"
│       └── age: 25
│
├── Collection: "products"
│   ├── Document: "prod001"
│   │   ├── name: "Laptop"
│   │   ├── category: "Electronics"
│   │   └── stock: 50
│   └── ...
```

**Key concepts:**
- **Collection** = A group of documents (like a table in SQL, but flexible)
- **Document** = A single record (like a row in SQL, but stored as a JSON-like object)
- **Field** = A key-value pair inside a document (like a column)
- **Sub-collection** = A collection inside a document (nesting!)

**Important rules:**
- Documents must live inside collections
- Collections must contain documents (they can't be empty and persist)
- Document IDs must be unique within a collection
- Documents have a **1 MB** max size
- You can nest **up to 100 levels** of sub-collections

#### Using Python SDK

```python
from google.cloud import firestore

# Initialize client
db = firestore.Client()

# --- CREATE / SET a document ---
doc_ref = db.collection("users").document("user123")
doc_ref.set({
    "name": "Ritesh",
    "email": "ritesh@example.com",
    "age": 28,
    "city": "Mumbai",
    "tags": ["devops", "cloud", "kubernetes"]
})

# --- READ a document ---
doc = db.collection("users").document("user123").get()
if doc.exists:
    print(f"User data: {doc.to_dict()}")
else:
    print("No such document!")

# --- UPDATE a document ---
doc_ref.update({
    "age": 29,
    "city": "Delhi"
})

# --- DELETE a document ---
db.collection("users").document("user123").delete()

# --- QUERY documents ---
# Get all users older than 25
users_ref = db.collection("users")
query = users_ref.where("age", ">", 25).order_by("age")

for doc in query.stream():
    print(f"{doc.id} => {doc.to_dict()}")

# Get users in a specific city

    print(f"{doc.id} => {doc.to_dict()}")
```

### 4.5 Security Rules

Firestore has its own **security rules language** that controls who can read/write data. This is mostly used for **client-side access** (mobile/web apps talking directly to Firestore).

```javascript
// firestore.rules
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    
    // Users can only read/write their own data
    match /users/{userId} {
      allow read, write: if request.auth != null 
                         && request.auth.uid == userId;
    }
    
    // Anyone can read products, only admins can write
    match /products/{productId} {
      allow read: if true;
      allow write: if request.auth != null 
                   && request.auth.token.admin == true;
    }
    
    // Orders: users can read their own, create new ones
    match /users/{userId}/orders/{orderId} {
      allow read: if request.auth.uid == userId;
      allow create: if request.auth.uid == userId
                    && request.resource.data.price > 0;
    }
  }
}
```

> **For DevOps / server-side use:** You typically use a **service account** with the Admin SDK, which bypasses security rules. Security rules are mainly for when client apps (browser, mobile) talk to Firestore directly.

## 5. Cloud Storage vs Cloud SQL vs Firestore — When to Use What?

| Need | Service | Why |
|------|---------|-----|
| Store files, images, backups, logs | **Cloud Storage** | Object storage, cheap, unlimited |
| Traditional SQL database with tables | **Cloud SQL** | Managed MySQL/PostgreSQL |
| Flexible NoSQL with real-time sync | **Firestore** | Document DB, great for apps |
| Terraform state file | **Cloud Storage** | Standard practice in GCP |
| User profiles, settings | **Firestore** | Flexible schema, fast reads |
| Financial transactions, relational data | **Cloud SQL** | ACID compliance, JOINs |
| Static website hosting | **Cloud Storage** | Cheap, fast, CDN-ready |
| Time-series metrics | **Neither** → Use **Cloud Bigtable** or **Cloud Monitoring** | Specialized tools exist |
| Data warehouse / analytics | **Neither** → Use **BigQuery** | Built for analytics at scale |

**Simple decision tree:**

```
Is it a file/blob? ─────────────── YES → Cloud Storage
         │
         NO
         │
Do you need SQL (JOINs, tables)? ── YES → Cloud SQL
         │
         NO
         │
Do you need flexible, real-time? ── YES → Firestore
```

## 6. Best Practices

### Cloud Storage
- ✅ Always enable **uniform bucket-level access** (simpler, more secure)
- ✅ Use **lifecycle rules** to move old data to cheaper storage
- ✅ Enable **versioning** for important buckets (Terraform state, configs)
- ✅ Use **private access** — avoid making buckets public unless necessary
- ✅ Enable **object lock** for compliance data (immutable backups)
- ❌ Don't use predictable bucket names with sensitive info

### Cloud SQL
- ✅ Always use **Private IP** — never expose to public internet in production
- ✅ Enable **HA** for production databases
- ✅ Use **Cloud SQL Auth Proxy** for secure connections
- ✅ Enable **automated backups + point-in-time recovery**
- ✅ Set **connection limits** and use **connection pooling**
- ✅ Use **read replicas** for read-heavy workloads
- ❌ Don't use `db-f1-micro` for production

### Firestore
- ✅ Design your data model around your **queries** (not the other way around)
- ✅ Keep documents **small** (< 1 MB)
- ✅ Use **batch operations** for multiple writes
- ✅ Create **composite indexes** for complex queries
- ✅ Use **server-side SDKs** (not security rules) for backend services
- ❌ Don't store large blobs in Firestore — use Cloud Storage

## 7. Summary

| Service | Type | Best For | Pricing Model |
|---------|------|----------|---------------|
| **Cloud Storage** | Object Storage | Files, backups, static content, Terraform state | Per GB stored + per operation |
| **Cloud SQL** | Managed Relational DB | Structured data, transactions, SQL queries | Per instance hour + storage |
| **Firestore** | Managed NoSQL Document DB | Real-time apps, flexible schemas, serverless | Per read/write/delete operation |

**Key Takeaways:**
1. **Cloud Storage** is your go-to for any file-based storage. Use lifecycle rules to save money.
2. **Cloud SQL** is a fully managed MySQL/PostgreSQL — use HA for production, Cloud SQL Proxy for secure connections.
3. **Firestore** is a serverless NoSQL database — great for apps that need real-time data and flexible schemas.
4. Pick the **right tool for the right job** — don't force SQL into a document model or vice versa.
5. Always use **private networking** and **encryption** — GCP provides both by default, just don't disable them!
