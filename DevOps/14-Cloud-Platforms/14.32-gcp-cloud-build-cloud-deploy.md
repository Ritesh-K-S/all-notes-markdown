## 1. Introduction — CI/CD on Google Cloud

When you write code, you need a way to:
1. **Build** it (compile, test, create artifacts)
2. **Deploy** it (push it to servers/Kubernetes/Cloud Run)

Google Cloud provides **two services** that work together to handle this:

| Service | Role | Think of It As |
|---------|------|---------------|
| **Cloud Build** | CI (Continuous Integration) — Build, test, create container images | The **kitchen** where you cook the food (build the app) |
| **Cloud Deploy** | CD (Continuous Delivery) — Deploy to environments safely | The **delivery service** that takes the food to customers (deploy to prod) |

```
Code Push → Cloud Build (build & test) → Cloud Deploy (deploy to dev → staging → prod)
```

They can work **independently** or **together**. Let's dive into each one.

## 2. Google Cloud Build

### 2.1 What is Cloud Build?

**Cloud Build** is Google's **fully managed CI/CD platform**. It executes your builds on Google's infrastructure — no servers to manage, no Jenkins to maintain.

You give it instructions ("build this Docker image", "run these tests", "deploy this app"), and it runs them in isolated containers on Google's servers.

**It's similar to:**
- **GitHub Actions** — Workflow-based CI/CD
- **GitLab CI** — Built-in CI/CD
- **Jenkins** — But without managing servers
- **AWS CodeBuild** — Amazon's equivalent

**Why Cloud Build?**
- ✅ **Serverless** — No servers to manage
- ✅ **Fast** — Runs on Google's infrastructure with fast networking
- ✅ **Deeply integrated** — Works natively with GCP services (GKE, Cloud Run, Artifact Registry)
- ✅ **Free tier** — 120 build-minutes/day for free!
- ✅ **Custom machine types** — From small to 32 vCPU machines

---

### 2.2 How Cloud Build Works

Here's the flow:

```
1. You push code to a repo (GitHub, GitLab, Cloud Source Repositories)
         │
         ▼
2. A trigger fires and starts a Cloud Build
         │
         ▼
3. Cloud Build reads your cloudbuild.yaml file
         │
         ▼
4. It runs each "step" in order, inside containers
         │
         ▼
5. Artifacts (Docker images, binaries) are pushed to Artifact Registry
         │
         ▼
6. Optionally, Cloud Deploy picks up and deploys the artifact
```

Each **step** runs inside a **Docker container**. You specify which container image to use for each step. Steps run sequentially by default but can run in parallel.

---

### 2.3 Key Concepts

| Concept | What It Is |
|---------|-----------|
| **Build** | A single execution of your build process |
| **Step** | One action in a build (e.g., "compile code", "run tests", "push image") |
| **Builder** | A Docker image used to run a step (e.g., `gcr.io/cloud-builders/docker`) |
| **Trigger** | Automatically starts a build when code changes (push, PR, tag) |
| **cloudbuild.yaml** | The configuration file that defines your build steps |
| **Substitution** | Variables you can use in your build config (`$PROJECT_ID`, `$COMMIT_SHA`) |
| **Artifact** | The output of your build (Docker image, JAR file, etc.) |
| **Worker Pool** | The machines your build runs on (default or custom) |
| **Build Log** | Detailed logs of every step (streamed in real-time) |

### 2.4 cloudbuild.yaml — The Build Configuration

This is the heart of Cloud Build. It's a YAML file that defines **what to do** during a build.

#### Basic Structure

```yaml
# cloudbuild.yaml
steps:
  # Step 1: Print a message
  - name: 'ubuntu'
    args: ['echo', 'Hello from Cloud Build!']

  # Step 2: List files
  - name: 'ubuntu'
    args: ['ls', '-la']

  # Step 3: Run a custom script
  - name: 'bash'
    args: ['bash', '-c', 'echo "Build number: $BUILD_ID"']
```

Let me explain each part:
- `steps:` — A list of things to do (executed in order)
- `name:` — The Docker image to use for this step
- `args:` — The command to run inside that container
- Each step runs in its own container, but they **share a workspace** directory (`/workspace`)

#### Real-World Example — Build a Node.js App

```yaml
# cloudbuild.yaml
steps:
  # Step 1: Install dependencies
  - name: 'node:18'
    entrypoint: 'npm'
    args: ['install']

  # Step 2: Run linter
  - name: 'node:18'
    entrypoint: 'npm'
    args: ['run', 'lint']

  # Step 3: Run tests
  - name: 'node:18'
    entrypoint: 'npm'
    args: ['test']

  # Step 4: Build the Docker image
  - name: 'gcr.io/cloud-builders/docker'
    args: [
      'build',
      '-t', 'us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/my-app:$COMMIT_SHA',
      '.'
    ]

  # Step 5: Push the image to Artifact Registry
  - name: 'gcr.io/cloud-builders/docker'
    args: [
      'push',
      'us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/my-app:$COMMIT_SHA'
    ]

# Images to be pushed (alternative to explicit push step)
images:
  - 'us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/my-app:$COMMIT_SHA'

# Timeout for the entire build (default: 10 minutes)
timeout: '1200s'  # 20 minutes

# Build options
options:
  machineType: 'E2_HIGHCPU_8'    # Faster machine
  logging: 'CLOUD_LOGGING_ONLY'
```

**What this does:**
1. Installs Node.js dependencies
2. Runs the linter to check code quality
3. Runs tests
4. Builds a Docker image tagged with the commit SHA
5. Pushes the image to Artifact Registry

```yaml
steps:
  # Use the official Python image
  - name: 'python:3.11'
    entrypoint: 'python'
    args: ['-m', 'pytest', 'tests/']

  - name: 'hashicorp/terraform:1.7'

  - name: 'hashicorp/terraform:1.7'

  - name: 'hashicorp/terraform:1.7'
```

#### Creating Custom Builders

You can create your own builder for specialized tools:

```dockerfile
# Dockerfile for a custom builder
FROM ubuntu:22.04
RUN apt-get update && apt-get install -y \
    curl \
    jq \
    python3 \
    python3-pip
RUN pip3 install awscli ansible
```

```bash
# Build and push your custom builder
docker build -t us-central1-docker.pkg.dev/my-project/builders/my-custom-builder:latest .
docker push us-central1-docker.pkg.dev/my-project/builders/my-custom-builder:latest
```

Then use it in your build:

```yaml
steps:
  - name: 'us-central1-docker.pkg.dev/my-project/builders/my-custom-builder:latest'
    args: ['ansible-playbook', 'deploy.yml']
```

### 2.6 Triggers — Automate Your Builds

Triggers **automatically start builds** when something happens in your source code repository.

#### Trigger Types

| Trigger Type | When It Fires |
|-------------|--------------|
| **Push to branch** | Code pushed to a specific branch (e.g., `main`, `develop`) |
| **Push tag** | A Git tag is created (e.g., `v1.0.0`) |
| **Pull Request** | A PR is opened or updated |
| **Manual** | You manually click "Run" or use the CLI |
| **Pub/Sub** | An event from a Pub/Sub topic |
| **Webhook** | An HTTP webhook from an external system |

#### Creating a Trigger via CLI

```bash
# Trigger on push to main branch (GitHub repo)
gcloud builds triggers create github \
  --name="build-on-push-main" \
  --repo-name="my-app" \
  --repo-owner="my-org" \
  --branch-pattern="^main$" \
  --build-config="cloudbuild.yaml" \
  --description="Build and test on push to main"

# Trigger on Git tags (e.g., v1.0.0)
gcloud builds triggers create github \
  --name="build-on-tag" \
  --repo-name="my-app" \
  --repo-owner="my-org" \
  --tag-pattern="^v.*" \
  --build-config="cloudbuild-release.yaml"

# Trigger on pull requests
gcloud builds triggers create github \
  --name="pr-checks" \
  --repo-name="my-app" \
  --repo-owner="my-org" \
  --pull-request-pattern="^main$" \
  --build-config="cloudbuild-pr.yaml" \
  --comment-control="COMMENTS_ENABLED"
```

#### Branch Filtering

```yaml
# Only trigger on specific branches
# ^main$ → exactly "main"
# ^release-.*$ → any branch starting with "release-"
# ^feature/.*$ → any branch starting with "feature/"
```

#### Included/Excluded Files

You can filter triggers based on which files changed:

```bash
gcloud builds triggers create github \
  --name="build-backend" \
  --repo-name="my-app" \
  --repo-owner="my-org" \
  --branch-pattern="^main$" \
  --build-config="cloudbuild.yaml" \
  --included-files="backend/**,shared/**" \
  --ignored-files="docs/**,*.md"
```

This trigger only fires if files in `backend/` or `shared/` change. Changes to `docs/` or markdown files are ignored.

### 2.7 Substitution Variables

Cloud Build provides **built-in variables** and lets you define **custom ones**.

#### Built-in Variables

| Variable | Value |
|----------|-------|
| `$PROJECT_ID` | Your GCP project ID |
| `$BUILD_ID` | Unique build ID |
| `$COMMIT_SHA` | Full Git commit SHA |
| `$SHORT_SHA` | Short Git commit SHA (first 7 chars) |
| `$BRANCH_NAME` | Git branch name |
| `$TAG_NAME` | Git tag name |
| `$REPO_NAME` | Repository name |
| `$REVISION_ID` | Same as COMMIT_SHA |
| `$_CUSTOM_VAR` | User-defined variable (starts with `_`) |

#### Using Substitutions

```yaml
steps:
  - name: 'gcr.io/cloud-builders/docker'
    args: [
      'build',
      '-t', 'us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/my-app:$SHORT_SHA',
      '-t', 'us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/my-app:latest',
      '--build-arg', 'VERSION=$TAG_NAME',
      '--build-arg', 'ENVIRONMENT=$_ENVIRONMENT',
      '.'
    ]

# Define custom substitution variables with defaults
substitutions:
  _ENVIRONMENT: 'development'
  _REGION: 'us-central1'
```

Override at runtime:

```bash
# Submit a build with custom substitutions
gcloud builds submit \
  --config=cloudbuild.yaml \
  --substitutions=_ENVIRONMENT=production,_REGION=europe-west1
```

### 2.8 Building & Pushing Docker Images

The most common Cloud Build use case — building Docker images.

#### Simple Docker Build

```yaml
# cloudbuild.yaml
steps:
  # Build the image
  - name: 'gcr.io/cloud-builders/docker'
    args: [
      'build',
      '-t', 'us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/api:$SHORT_SHA',
      '-t', 'us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/api:latest',
      '-f', 'Dockerfile',
      '.'
    ]

images:
  - 'us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/api:$SHORT_SHA'
  - 'us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/api:latest'
```

#### Multi-Stage Build with Caching

```yaml
steps:
  # Pull the cache image (if it exists)
  - name: 'gcr.io/cloud-builders/docker'
    entrypoint: 'bash'
    args:
      - '-c'
      - |
        docker pull us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/api:latest || exit 0

  # Build with cache
  - name: 'gcr.io/cloud-builders/docker'
    args: [
      'build',
      '--cache-from', 'us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/api:latest',
      '-t', 'us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/api:$SHORT_SHA',
      '-t', 'us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/api:latest',
      '.'
    ]

images:
  - 'us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/api:$SHORT_SHA'
  - 'us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/api:latest'

options:
  machineType: 'E2_HIGHCPU_8'
```

#### Building Multiple Services (Monorepo)

```yaml
steps:
  # Build API service
  - name: 'gcr.io/cloud-builders/docker'
    args: ['build', '-t', 'us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/api:$SHORT_SHA', './api']
    id: 'build-api'

  # Build Frontend service (runs in PARALLEL with API build)
  - name: 'gcr.io/cloud-builders/docker'
    args: ['build', '-t', 'us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/frontend:$SHORT_SHA', './frontend']
    id: 'build-frontend'
    waitFor: ['-']   # Don't wait for previous step, run in parallel!

  # Build Worker service (also in parallel)
  - name: 'gcr.io/cloud-builders/docker'
    args: ['build', '-t', 'us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/worker:$SHORT_SHA', './worker']
    id: 'build-worker'
    waitFor: ['-']

  # Push all images (wait for all builds to finish)
  - name: 'gcr.io/cloud-builders/docker'
    args: ['push', '--all-tags', 'us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/api']
    waitFor: ['build-api']

  - name: 'gcr.io/cloud-builders/docker'
    args: ['push', '--all-tags', 'us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/frontend']
    waitFor: ['build-frontend']

  - name: 'gcr.io/cloud-builders/docker'
    args: ['push', '--all-tags', 'us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/worker']
    waitFor: ['build-worker']
```

**Key feature:** `waitFor: ['-']` means "don't wait for anything, start immediately" — this enables **parallel builds**!

### 2.9 Running Tests in Cloud Build

```yaml
steps:
  # Install dependencies
  - name: 'python:3.11'
    entrypoint: 'pip'
    args: ['install', '-r', 'requirements.txt', '--user']

  # Run unit tests
  - name: 'python:3.11'
    entrypoint: 'python'
    args: ['-m', 'pytest', 'tests/unit/', '-v', '--junitxml=test-results/unit.xml']
    id: 'unit-tests'

  # Run integration tests (in parallel with unit tests)
  - name: 'python:3.11'
    entrypoint: 'python'
    args: ['-m', 'pytest', 'tests/integration/', '-v', '--junitxml=test-results/integration.xml']
    id: 'integration-tests'
    waitFor: ['-']

  # Run security scan
  - name: 'aquasec/trivy:latest'
    args: ['fs', '--severity', 'HIGH,CRITICAL', '--exit-code', '1', '.']
    id: 'security-scan'
    waitFor: ['-']

  # Build image only if all tests pass
  - name: 'gcr.io/cloud-builders/docker'
    args: ['build', '-t', 'us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/app:$SHORT_SHA', '.']
    waitFor: ['unit-tests', 'integration-tests', 'security-scan']

# Save test results as artifacts
artifacts:
  objects:
    location: 'gs://$PROJECT_ID-build-artifacts/$BUILD_ID/'
    paths: ['test-results/*.xml']
```

### 2.10 Private Pools & Worker Pools

By default, Cloud Build runs on Google-managed workers (shared infrastructure). For more control, you can use **private pools**.

| Feature | Default Pool | Private Pool |
|---------|-------------|-------------|
| **Network** | Google-managed | Your VPC |
| **Machine size** | Up to E2_HIGHCPU_32 | Custom sizes |
| **Access** | Public internet | Private VPC resources |
| **Use case** | General builds | Builds needing private resources |

```bash
# Create a private pool
gcloud builds worker-pools create my-pool \
  --region=us-central1 \
  --peered-network="projects/my-project/global/networks/my-vpc"
```

```yaml
# Use private pool in cloudbuild.yaml
options:
  pool:
    name: 'projects/my-project/locations/us-central1/workerPools/my-pool'
```

**When to use private pools:**
- Your build needs to access private databases or APIs
- Compliance requirements (data can't leave your VPC)
- You need bigger machines
- You want consistent build performance

### 2.12 Secrets in Cloud Build

You often need secrets during builds (API keys, database passwords, npm tokens). **Never hardcode them!**

#### Using Secret Manager

```yaml
steps:
  - name: 'gcr.io/cloud-builders/docker'
    args: [
      'build',
      '--build-arg', 'NPM_TOKEN=$$NPM_TOKEN',
      '-t', 'us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/app:$SHORT_SHA',
      '.'
    ]
    secretEnv: ['NPM_TOKEN']

availableSecrets:
  secretManager:
    - versionName: 'projects/my-project/secrets/npm-token/versions/latest'
      env: 'NPM_TOKEN'
```

**How it works:**
1. You store the secret in **Secret Manager** (GCP's secrets vault)
2. In `cloudbuild.yaml`, you reference it using `availableSecrets`
3. Cloud Build fetches the secret and makes it available as an environment variable
4. `$$NPM_TOKEN` (double dollar sign) references the secret env var

```bash
# Store a secret in Secret Manager
echo -n "my-super-secret-token" | \
  gcloud secrets create npm-token --data-file=-

# Grant Cloud Build access to the secret
gcloud secrets add-iam-policy-binding npm-token \
  --member="serviceAccount:$(gcloud projects describe $PROJECT_ID \
    --format='value(projectNumber)')@cloudbuild.gserviceaccount.com" \
  --role="roles/secretmanager.secretAccessor"
```

## 3. Google Cloud Deploy

### 3.1 What is Cloud Deploy?

**Cloud Deploy** is Google's **managed continuous delivery** service. While Cloud Build *builds* your app, Cloud Deploy *delivers* it to your environments (dev → staging → production).

It's designed for:
- **Progressive delivery** — Move through environments safely
- **Approvals** — Require human approval before production
- **Rollbacks** — Easily go back to a previous version
- **Audit trail** — Track who deployed what, when, and where

**It's similar to:**
- **AWS CodeDeploy** + **CodePipeline**
- **Spinnaker** — But managed and simpler
- **Argo Rollouts** — But GCP-native

---

### 3.2 How Cloud Deploy Works

```
Cloud Build                    Cloud Deploy
┌──────────┐                  ┌──────────────────────────────────────────┐
│  Build &  │   Creates a     │                                          │
│  Test     │──────────────▶  │  Release                                 │
│  App      │   "Release"     │    │                                     │
└──────────┘                  │    ▼                                     │
                              │  ┌─────┐   ┌─────────┐   ┌──────────┐  │
                              │  │ Dev │──▶│ Staging │──▶│Production│  │
                              │  └─────┘   └─────────┘   └──────────┘  │
                              │  (auto)    (auto)        (approval     │
                              │                          required)     │
                              └──────────────────────────────────────────┘
```

1. **Cloud Build** creates a Docker image
2. Cloud Build creates a **Release** in Cloud Deploy
3. The release gets deployed to the **first target** (dev)
4. If dev looks good, it gets **promoted** to the next target (staging)
5. After approval, it gets promoted to **production**

---

### 3.3 Key Concepts

| Concept | What It Is | Example |
|---------|-----------|---------|
| **Delivery Pipeline** | The full journey from dev to prod | `my-app-pipeline` |
| **Target** | A destination environment | `dev`, `staging`, `production` |
| **Release** | A specific version of your app | `release-abc123` |
| **Rollout** | The act of deploying a release to a target | "deploy release-abc123 to staging" |
| **Promotion** | Moving a release from one target to the next | dev → staging → prod |
| **Approval** | A human gate before deploying to a target | "Approve for production?" |
| **Skaffold** | The rendering engine Cloud Deploy uses | Defines how to deploy (kubectl, Helm, kustomize) |

#### What is Skaffold?

Cloud Deploy uses **Skaffold** under the hood to render and deploy your Kubernetes manifests. Skaffold is an open-source tool by Google that handles the "how to deploy" part.

You write a `skaffold.yaml` that tells Cloud Deploy:
- Where your Kubernetes manifests are
- How to render them (raw kubectl, Helm, or Kustomize)
- What images to use

---

### 3.4 Setting Up a Delivery Pipeline

#### Step 1: Create a `skaffold.yaml`

```yaml
# skaffold.yaml
apiVersion: skaffold/v4beta7
kind: Config
metadata:
  name: my-app
manifests:
  rawYaml:
    - k8s-manifests/*.yaml    # Path to your Kubernetes YAML files
deploy:
  kubectl: {}                 # Use kubectl to apply manifests
```

For Helm:

```yaml
# skaffold.yaml (Helm version)
apiVersion: skaffold/v4beta7
kind: Config
metadata:
  name: my-app
manifests:
  helm:
    releases:
      - name: my-app
        chartPath: ./helm/my-app
        valuesFiles:
          - ./helm/my-app/values.yaml
deploy:
  kubectl: {}
```

#### Step 2: Create Your Kubernetes Manifests

```yaml
# k8s-manifests/deployment.yaml
apiVersion: apps/v1
```

#### Step 3: Define the Delivery Pipeline

```yaml
# clouddeploy.yaml
apiVersion: deploy.cloud.google.com/v1
kind: DeliveryPipeline
metadata:
  name: my-app-pipeline
description: "Pipeline for deploying my-app"
serialPipeline:
  stages:
    - targetId: dev
      profiles: [dev]
    - targetId: staging
      profiles: [staging]
    - targetId: production
      profiles: [production]
      strategy:
        standard:
          verify: true     # Run verification after deploy
---
# Dev Target
apiVersion: deploy.cloud.google.com/v1
kind: Target
metadata:
  name: dev
description: "Development environment"
gke:
  cluster: projects/my-project/locations/us-central1/clusters/dev-cluster
---
# Staging Target
apiVersion: deploy.cloud.google.com/v1
kind: Target
metadata:
  name: staging
description: "Staging environment"
gke:
  cluster: projects/my-project/locations/us-central1/clusters/staging-cluster
---
# Production Target
apiVersion: deploy.cloud.google.com/v1
kind: Target
metadata:
  name: production
description: "Production environment"
requireApproval: true        # Require manual approval!
gke:
  cluster: projects/my-project/locations/us-central1/clusters/prod-cluster
```

**What this does:**
- Creates a pipeline with **3 stages**: dev → staging → production
- Dev and staging **auto-deploy** (no approval needed)
- Production **requires manual approval** before deploying
- Each target points to a different GKE cluster

#### Step 4: Apply the Pipeline

```bash
# Register the pipeline and targets
gcloud deploy apply --file=clouddeploy.yaml \
  --region=us-central1 \
  --project=my-project
```

### 3.7 Approvals & Promotion

#### Promoting a Release

After your release successfully deploys to dev, promote it to the next stage:

```bash
# Promote from dev to staging
gcloud deploy releases promote \
  --delivery-pipeline=my-app-pipeline \
  --release=release-v1-0-0 \
  --region=us-central1

# Promote from staging to production (will wait for approval)
gcloud deploy releases promote \
  --delivery-pipeline=my-app-pipeline \
  --release=release-v1-0-0 \
  --region=us-central1
```

#### Approving a Release

When production requires approval:

```bash
# List pending approvals
gcloud deploy rollouts list \
  --delivery-pipeline=my-app-pipeline \
  --release=release-v1-0-0 \
  --region=us-central1 \
  --filter="approvalState=NEEDS_APPROVAL"

# Approve the rollout
gcloud deploy rollouts approve <ROLLOUT_NAME> \
  --delivery-pipeline=my-app-pipeline \
  --release=release-v1-0-0 \
  --region=us-central1

# Reject the rollout
gcloud deploy rollouts reject <ROLLOUT_NAME> \
  --delivery-pipeline=my-app-pipeline \
  --release=release-v1-0-0 \
  --region=us-central1
```

### 3.9 Deploy Hooks & Verification

You can run **verification** after deployment to make sure everything works:

```yaml
# skaffold.yaml with verification
apiVersion: skaffold/v4beta7
kind: Config
metadata:
  name: my-app
manifests:
  rawYaml:
    - k8s-manifests/*.yaml
deploy:
  kubectl: {}
verify:
  - name: smoke-test
    container:
      name: smoke-test
      image: curlimages/curl:latest
      command: ["curl"]
      args: ["-f", "http://my-app.default.svc.cluster.local/health"]
  - name: integration-test
    container:
      name: integration-test
      image: us-central1-docker.pkg.dev/my-project/my-repo/test-runner:latest
      command: ["pytest"]
      args: ["tests/smoke/", "-v"]
```

**What this does:**
- After deploying, run a **smoke test** (curl the health endpoint)
- Then run **integration tests**
- If any test fails, the rollout is marked as **failed**

## 4. Cloud Build + Cloud Deploy — End-to-End Pipeline

Here's how you connect Cloud Build and Cloud Deploy into a complete CI/CD pipeline:

```yaml
# cloudbuild.yaml — The Complete Pipeline
steps:
  # Step 1: Run tests
  - name: 'node:18'
    entrypoint: 'npm'
    args: ['ci']
    id: 'install'

  - name: 'node:18'
    entrypoint: 'npm'
    args: ['test']
    id: 'test'
    waitFor: ['install']

  # Step 2: Build Docker image
  - name: 'gcr.io/cloud-builders/docker'
    args: [
      'build',
      '-t', 'us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/my-app:$SHORT_SHA',
      '.'
    ]
    id: 'build'
    waitFor: ['test']

  # Step 3: Push image to Artifact Registry
  - name: 'gcr.io/cloud-builders/docker'
    args: [
      'push',
      'us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/my-app:$SHORT_SHA'
    ]
    id: 'push'
    waitFor: ['build']

  # Step 4: Create a Cloud Deploy release
  - name: 'gcr.io/cloud-builders/gcloud'
    args: [
      'deploy', 'releases', 'create', 'release-$SHORT_SHA',
      '--delivery-pipeline=my-app-pipeline',
      '--region=us-central1',
      '--images=my-app=us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/my-app:$SHORT_SHA'
    ]
    id: 'deploy'
    waitFor: ['push']

options:
  machineType: 'E2_HIGHCPU_8'
```

**The flow:**
1. Developer pushes code to `main`
2. Cloud Build trigger fires
3. Cloud Build installs deps → runs tests → builds Docker image → pushes to registry
4. Cloud Build creates a **Cloud Deploy release**
5. Cloud Deploy deploys to **dev** (automatically)
6. Team verifies dev, then promotes to **staging**
7. After staging looks good, someone **approves** for **production**
8. Cloud Deploy deploys to production
9. Verification tests run automatically

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│   Push   │───▶│  Build   │───▶│  Push    │───▶│  Create  │───▶│  Deploy  │
│   Code   │    │  & Test  │    │  Image   │    │ Release  │    │  to Dev  │
└──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
                                                                     │
                              Cloud Build                            │
                ─────────────────────────────────────         Cloud Deploy
                                                                     │
                                                                     ▼
                                                 ┌──────────┐  ┌──────────┐
                                                 │ Promote  │──│  Deploy  │
                                                 │ (manual) │  │ Staging  │
                                                 └──────────┘  └──────────┘
                                                                     │
                                                                     ▼
                                                 ┌──────────┐  ┌──────────┐
                                                 │ Approve  │──│  Deploy  │
                                                 │ (manual) │  │   Prod   │
                                                 └──────────┘  └──────────┘
```

## 6. Cloud Build vs Other CI/CD Tools

| Feature | Cloud Build | GitHub Actions | Jenkins | GitLab CI |
|---------|------------|----------------|---------|-----------|
| **Managed?** | ✅ Fully | ✅ Fully | ❌ Self-hosted | ⚠️ Runners needed |
| **Free tier** | 120 min/day | 2000 min/month | Free (self-hosted) | 400 min/month |
| **GCP integration** | ⭐ Native | 🔧 Via actions | 🔧 Via plugins | 🔧 Via config |
| **Docker support** | ⭐ Excellent | ✅ Good | ✅ Good | ✅ Good |
| **Parallel steps** | ✅ Yes | ✅ Yes (matrix) | ✅ Yes | ✅ Yes |
| **Secrets** | Secret Manager | GitHub Secrets | Credentials plugin | CI Variables |
| **Deployment** | Cloud Deploy | Custom actions | Custom pipelines | Environments |
| **Best for** | GCP-heavy teams | GitHub-hosted projects | Complex enterprise | GitLab users |

**When to choose Cloud Build:**
- Your infrastructure is **primarily on GCP**
- You want **deep integration** with GKE, Cloud Run, Artifact Registry
- You want a **managed solution** (no Jenkins servers to maintain)
- You need **private pools** (builds within your VPC)

## 7. Best Practices

### Cloud Build
- ✅ **Use specific image tags** — Don't use `:latest` for builders
- ✅ **Enable build caching** — Pull previous images for `--cache-from`
- ✅ **Use `waitFor`** to parallelize independent steps
- ✅ **Store secrets in Secret Manager** — Never in `cloudbuild.yaml`
- ✅ **Set timeouts** — Don't let broken builds run forever
- ✅ **Use substitution variables** — Keep configs reusable
- ✅ **Pin builder versions** — Avoid surprises from updated builders
- ❌ Don't put sensitive info in build logs

### Cloud Deploy
- ✅ **Always require approval for production** — `requireApproval: true`
- ✅ **Use verification** — Run smoke tests after every deployment
- ✅ **Tag releases meaningfully** — Use commit SHA or semantic version
- ✅ **Use Skaffold profiles** — Different configs for dev/staging/prod
- ✅ **Monitor rollouts** — Set up alerts for failed deployments
- ✅ **Practice rollbacks** — Test your rollback procedure regularly
- ❌ Don't skip staging — Always test in a pre-prod environment first

## 8. Summary

| Service | What It Does | Think of It As |
|---------|-------------|---------------|
| **Cloud Build** | Builds, tests, and creates artifacts (Docker images) | Your automated build factory |
| **Cloud Deploy** | Delivers releases through environments (dev → staging → prod) | Your deployment conveyor belt |
| **Artifact Registry** | Stores Docker images and packages | Your package warehouse |
| **Skaffold** | Renders and applies Kubernetes manifests | The deployment script engine |

**Key Takeaways:**
1. **Cloud Build** is serverless CI — no servers to manage, 120 free minutes/day
2. **cloudbuild.yaml** defines your build steps — each step runs in a container
3. **Triggers** automate builds on code push, PRs, or tags
4. **Cloud Deploy** handles progressive delivery with approvals and rollbacks
5. **Together**, Cloud Build + Cloud Deploy give you a complete, GCP-native CI/CD pipeline
6. Use **Artifact Registry** to store your Docker images
7. Always use **Secret Manager** for sensitive data in builds
8. **Require approval** for production deployments — safety first!
