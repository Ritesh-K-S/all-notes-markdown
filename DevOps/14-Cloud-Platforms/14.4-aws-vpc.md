# 14.4 — AWS VPC (Virtual Private Cloud)

> **Level:** Intermediate
> **Topic:** Subnets, Route Tables, Internet Gateway, NAT Gateway, Security Groups, NACLs

---

## What is a VPC?

A **VPC (Virtual Private Cloud)** is basically your **own private network inside AWS**. Think of it like renting a floor in a huge building (AWS cloud). That entire floor is yours — you decide how to divide the rooms, who can enter, and what doors lead where.

Without a VPC, your resources (like EC2 instances, databases, etc.) would all be sitting on the open internet with no isolation. That's dangerous! A VPC gives you **complete control** over:

- Your **IP address range** (which addresses your resources use)
- Your **subnets** (how you divide your network)
- Your **route tables** (where traffic goes)
- Your **gateways** (how traffic enters/leaves)
- Your **security rules** (who can talk to whom)

> **Simple analogy:** VPC = Your own private apartment complex. You decide the layout, the locks on the doors, and who gets a key.

---

## VPC — Key Concepts at a Glance

| Concept | What It Does |
|---------|-------------|
| VPC | Your private network in AWS |
| Subnet | A subdivision of your VPC (like rooms in your apartment) |
| Route Table | Rules that decide where network traffic goes |
| Internet Gateway (IGW) | The front door — connects your VPC to the internet |
| NAT Gateway | Lets private resources access the internet without being exposed |
| Security Group (SG) | A firewall around individual resources (like a bodyguard) |
| NACL | A firewall around an entire subnet (like a security gate at the entrance) |

---

## Default VPC vs Custom VPC

When you create an AWS account, AWS automatically gives you a **Default VPC** in every region. It comes pre-configured so you can start launching resources right away.

| Feature | Default VPC | Custom VPC |
|---------|------------|------------|
| Created by | AWS (automatically) | You (manually) |
| CIDR Block | `172.31.0.0/16` | You choose (e.g., `10.0.0.0/16`) |
| Subnets | One public subnet per AZ | You create your own |
| Internet Gateway | Attached automatically | You attach manually |
| Use case | Quick testing & learning | Production workloads |

> **Best Practice:** Never use the default VPC for production. Always create a custom VPC with proper planning.

## CIDR Blocks — Choosing Your IP Range

When you create a VPC, the first thing you need to decide is the **CIDR block** — this is the range of IP addresses your VPC will use.

**CIDR** stands for **Classless Inter-Domain Routing**. It's a way of expressing a range of IP addresses.

### How CIDR Works

```
10.0.0.0/16
│         │
│         └── The "/16" means the first 16 bits are fixed (the network part)
└──────────── The remaining bits are for hosts (your resources)
```

| CIDR Block | Total IPs | Usable IPs | Use Case |
|-----------|-----------|-----------|----------|
| `/16` | 65,536 | 65,531 | Large VPC (most common) |
| `/20` | 4,096 | 4,091 | Medium subnet |
| `/24` | 256 | 251 | Small subnet |
| `/28` | 16 | 11 | Very small subnet |

> **Why fewer usable IPs?** AWS reserves 5 IPs in every subnet:
> - `.0` — Network address
> - `.1` — VPC router
> - `.2` — DNS server
> - `.3` — Reserved for future use
> - `.255` — Broadcast address (AWS doesn't support broadcast, but reserves it anyway)

### Common CIDR Choices

```
VPC:        10.0.0.0/16      (65,536 IPs — plenty of room)
Subnet 1:   10.0.1.0/24      (256 IPs)
Subnet 2:   10.0.2.0/24      (256 IPs)
Subnet 3:   10.0.3.0/24      (256 IPs)
```

> **Tip:** Plan your CIDR blocks carefully. You **cannot change** a VPC's primary CIDR block after creation (you can add secondary ones, though).

## Subnets — Dividing Your VPC

A **subnet** is a smaller section within your VPC. Think of it as dividing your apartment into rooms — each room serves a different purpose.

### Public Subnet vs Private Subnet

| Type | Can reach the internet? | Resources exposed to the internet? | Use Case |
|------|------------------------|-----------------------------------|----------|
| **Public Subnet** | ✅ Yes | ✅ Yes (via public IP) | Web servers, load balancers |
| **Private Subnet** | ✅ Yes (via NAT) | ❌ No | Databases, application servers |

> **Important:** What makes a subnet "public" or "private" is NOT a setting on the subnet itself. It's determined by its **route table** — specifically whether the route table has a route to an **Internet Gateway**.

### Availability Zones (AZs)

Each subnet lives in **one specific Availability Zone**. An AZ is a physically separate data center within a region.

```
Region: us-east-1
├── AZ: us-east-1a
│   ├── Public Subnet  (10.0.1.0/24)
│   └── Private Subnet (10.0.4.0/24)
├── AZ: us-east-1b
│   ├── Public Subnet  (10.0.2.0/24)
│   └── Private Subnet (10.0.5.0/24)
└── AZ: us-east-1c
    ├── Public Subnet  (10.0.3.0/24)
    └── Private Subnet (10.0.6.0/24)
```

> **Best Practice:** Always spread your subnets across **at least 2 AZs** for high availability. If one AZ goes down, your app keeps running in the other.

### Creating a Subnet (AWS CLI)

```bash
# Create a public subnet in us-east-1a
aws ec2 create-subnet \
  --vpc-id vpc-0abc123def456 \
  --cidr-block 10.0.1.0/24 \
  --availability-zone us-east-1a \
  --tag-specifications 'ResourceType=subnet,Tags=[{Key=Name,Value=public-subnet-1a}]'

# Create a private subnet in us-east-1a
aws ec2 create-subnet \
  --vpc-id vpc-0abc123def456 \
  --cidr-block 10.0.4.0/24 \
  --availability-zone us-east-1a \
  --tag-specifications 'ResourceType=subnet,Tags=[{Key=Name,Value=private-subnet-1a}]'
```

## Internet Gateway (IGW) — The Front Door

An **Internet Gateway** is what connects your VPC to the public internet. Without it, nothing in your VPC can reach the internet, and nobody from the internet can reach your VPC.

### Key Facts About IGW

- **One IGW per VPC** — you can only attach one
- It's **horizontally scaled**, redundant, and highly available (AWS manages this)
- It's **free** — no charge for the IGW itself
- It performs **NAT** for instances that have a public IP

### How It Works

```
Internet
   │
   ▼
┌─────────────────┐
│ Internet Gateway │  ◄── The door between your VPC and the internet
└────────┬────────┘
         │
    ┌────▼────┐
    │   VPC    │
    │          │
    │ ┌──────┐ │
    │ │Public│ │  ◄── Instances here CAN reach the internet
    │ │Subnet│ │      (if route table points to IGW)
    │ └──────┘ │
    │          │
    │ ┌───────┐│
    │ │Private││  ◄── Instances here CANNOT reach the internet directly
    │ │Subnet ││
    │ └───────┘│
    └──────────┘
```

### Attaching an IGW (AWS CLI)

```bash
# Step 1: Create an Internet Gateway
aws ec2 create-internet-gateway \
  --tag-specifications 'ResourceType=internet-gateway,Tags=[{Key=Name,Value=my-igw}]'

# Step 2: Attach it to your VPC
aws ec2 attach-internet-gateway \
  --internet-gateway-id igw-0abc123 \
  --vpc-id vpc-0abc123def456
```

> **Remember:** Just attaching an IGW is NOT enough. You also need to update the **route table** to send internet-bound traffic to the IGW.

## Route Tables — Traffic Directions

A **route table** is like a GPS for your network traffic. It contains rules (called **routes**) that tell traffic where to go.

Every subnet **must** be associated with a route table. If you don't associate one explicitly, it uses the **main route table** (the default one AWS creates with your VPC).

### What a Route Table Looks Like

**Public Subnet Route Table:**

| Destination | Target | Meaning |
|------------|--------|---------|
| `10.0.0.0/16` | `local` | Traffic within the VPC stays local |
| `0.0.0.0/0` | `igw-0abc123` | Everything else → Internet Gateway |

**Private Subnet Route Table:**

| Destination | Target | Meaning |
|------------|--------|---------|
| `10.0.0.0/16` | `local` | Traffic within the VPC stays local |
| `0.0.0.0/0` | `nat-0xyz789` | Everything else → NAT Gateway |

The `0.0.0.0/0` route is the **default route** — it catches all traffic that doesn't match any other rule. This is the key difference:

- Public subnet → default route goes to **IGW** (direct internet access)
- Private subnet → default route goes to **NAT Gateway** (internet access, but hidden)

### Creating and Associating Route Tables (AWS CLI)

```bash
# Step 1: Create a route table
aws ec2 create-route-table \
  --vpc-id vpc-0abc123def456 \
  --tag-specifications 'ResourceType=route-table,Tags=[{Key=Name,Value=public-rt}]'

# Step 2: Add a route to the Internet Gateway
aws ec2 create-route \
  --route-table-id rtb-0abc123 \
  --destination-cidr-block 0.0.0.0/0 \
  --gateway-id igw-0abc123

# Step 3: Associate the route table with a subnet
aws ec2 associate-route-table \
  --route-table-id rtb-0abc123 \
  --subnet-id subnet-0abc123
```

## NAT Gateway — Internet for Private Resources

A **NAT Gateway** (Network Address Translation) lets resources in your **private subnet** access the internet **without exposing them** to incoming traffic from the internet.

### Why Do You Need NAT?

Your database server sits in a private subnet (good — it's safe from the internet). But it needs to download software updates or talk to an external API. Without NAT, it's completely isolated.

NAT Gateway solves this:
- **Outbound traffic:** ✅ Allowed (your private instance can reach the internet)
- **Inbound traffic:** ❌ Blocked (nobody from the internet can reach your private instance)

### How NAT Gateway Works

```
Private Instance (10.0.4.10)
        │
        │ "I need to download updates"
        ▼
┌──────────────┐
│  NAT Gateway  │  (Lives in the PUBLIC subnet)
│  (10.0.1.50)  │  (Has an Elastic IP: 52.1.2.3)
└──────┬───────┘
       │
       ▼
┌──────────────┐
│Internet Gateway│
└──────┬───────┘
       │
       ▼
   Internet
   
The internet sees traffic coming from 52.1.2.3 (the NAT's Elastic IP),
NOT from 10.0.4.10 (the private instance).
```

### NAT Gateway vs NAT Instance

| Feature | NAT Gateway | NAT Instance |
|---------|------------|-------------|
| Managed by | AWS (fully managed) | You (EC2 instance) |
| Availability | Highly available in an AZ | You manage failover |
| Bandwidth | Up to 100 Gbps | Depends on instance type |
| Cost | ~$0.045/hr + data transfer | EC2 instance cost |
| Maintenance | None | You patch/update |
| Best for | Production | Cost-saving (small setups) |

> **Best Practice:** Use **NAT Gateway** for production. Use NAT Instance only if you're on a very tight budget.

### Creating a NAT Gateway (AWS CLI)

```bash
# Step 1: Allocate an Elastic IP
aws ec2 allocate-address --domain vpc

# Step 2: Create NAT Gateway in a PUBLIC subnet
aws ec2 create-nat-gateway \
  --subnet-id subnet-0abc123 \
  --allocation-id eipalloc-0abc123 \
  --tag-specifications 'ResourceType=natgateway,Tags=[{Key=Name,Value=my-nat-gw}]'

# Step 3: Update private subnet's route table
aws ec2 create-route \
  --route-table-id rtb-private-0abc123 \
  --destination-cidr-block 0.0.0.0/0 \
  --nat-gateway-id nat-0abc123
```

> **Important:** NAT Gateway must be created in a **public subnet** and needs an **Elastic IP**.

### High Availability for NAT Gateway

NAT Gateway is AZ-scoped — if the AZ goes down, resources in other AZs using that NAT lose internet access. For HA:

```
AZ-1a                          AZ-1b
┌─────────────────┐            ┌─────────────────┐
│  Public Subnet   │            │  Public Subnet   │
│  NAT Gateway A   │            │  NAT Gateway B   │
└────────┬────────┘            └────────┬────────┘
         │                              │
┌────────▼────────┐            ┌────────▼────────┐
│  Private Subnet  │            │  Private Subnet  │
│  Route → NAT-A   │            │  Route → NAT-B   │
└─────────────────┘            └─────────────────┘
```

> Create one NAT Gateway **per AZ** and update each private subnet's route table to point to the NAT Gateway in the same AZ.

## Security Groups (SG) — Instance-Level Firewall

A **Security Group** is like a **personal bodyguard** for your EC2 instance (or any resource). It controls what traffic is allowed **in** and **out** of that specific resource.

### Key Characteristics

| Feature | Details |
|---------|---------|
| Operates at | **Instance level** (attached to ENI — Elastic Network Interface) |
| Type | **Stateful** — if you allow traffic in, the response is automatically allowed out |
| Default behavior | **Deny all inbound**, **Allow all outbound** |
| Rules | Only **ALLOW** rules (you can't create explicit DENY rules) |
| Evaluation | **All rules are evaluated** (not in order) |

### What "Stateful" Means

```
Inbound Rule: Allow port 80 (HTTP)

1. Client sends request to port 80     → ✅ ALLOWED (matches inbound rule)
2. Server sends response back to client → ✅ AUTOMATICALLY ALLOWED (stateful!)

You don't need to create an outbound rule for the response.
The Security Group "remembers" the connection.
```

### Common Security Group Examples

**Web Server Security Group:**

| Type | Protocol | Port Range | Source | Description |
|------|----------|-----------|--------|-------------|
| Inbound | TCP | 80 | `0.0.0.0/0` | HTTP from anywhere |
| Inbound | TCP | 443 | `0.0.0.0/0` | HTTPS from anywhere |
| Inbound | TCP | 22 | `203.0.113.0/24` | SSH from office IP only |
| Outbound | All | All | `0.0.0.0/0` | Allow all outbound |

**Database Security Group:**

| Type | Protocol | Port Range | Source | Description |
|------|----------|-----------|--------|-------------|
| Inbound | TCP | 3306 | `sg-webserver` | MySQL from web server SG only |
| Outbound | All | All | `0.0.0.0/0` | Allow all outbound |

> **Notice:** The database SG allows inbound only from the **web server's Security Group** (not an IP). This is called **SG referencing** — a powerful and recommended pattern.

### Creating a Security Group (AWS CLI)

```bash
# Create a security group
aws ec2 create-security-group \
  --group-name web-server-sg \
  --description "Security group for web servers" \
  --vpc-id vpc-0abc123def456

# Add inbound rule - allow HTTP
aws ec2 authorize-security-group-ingress \
  --group-id sg-0abc123 \
  --protocol tcp \
  --port 80 \
  --cidr 0.0.0.0/0

# Add inbound rule - allow SSH from specific IP
aws ec2 authorize-security-group-ingress \
  --group-id sg-0abc123 \
  --protocol tcp \
  --port 22 \
  --cidr 203.0.113.50/32

# Add inbound rule - allow traffic from another security group
aws ec2 authorize-security-group-ingress \
  --group-id sg-database-0abc123 \
  --protocol tcp \
  --port 3306 \
  --source-group sg-webserver-0abc123
```

## NACLs (Network Access Control Lists) — Subnet-Level Firewall

A **NACL** (pronounced "nackle") is a firewall that operates at the **subnet level**. It's like the security gate at the entrance of a building — everyone entering or leaving the subnet must pass through it.

### Security Group vs NACL — The Big Comparison

| Feature | Security Group | NACL |
|---------|---------------|------|
| Operates at | Instance level | Subnet level |
| Stateful/Stateless | **Stateful** | **Stateless** |
| Rules | Allow rules only | Allow AND Deny rules |
| Default | Deny all inbound, allow all outbound | Allow all (default NACL) |
| Rule evaluation | All rules evaluated together | Rules evaluated **in order** (by rule number) |
| Association | Attached to instance/ENI | Associated with subnet |

### What "Stateless" Means

```
NACL is STATELESS — it doesn't "remember" connections.

Inbound Rule: Allow port 80
Outbound Rule: ??? (you MUST define this separately!)

1. Client sends request to port 80     → ✅ ALLOWED (matches inbound rule)
2. Server sends response on port 49152  → ❌ BLOCKED (no outbound rule for this!)

You MUST add an outbound rule for ephemeral ports (1024-65535) 
to allow response traffic back!
```

### Ephemeral Ports — The Gotcha

When a client connects to your server, the response goes back on a random **ephemeral port** (typically 1024-65535). Since NACLs are stateless, you must explicitly allow these:

```
Outbound Rule: Allow TCP ports 1024-65535 to 0.0.0.0/0
```

### NACL Rule Evaluation — Order Matters!

NACL rules are evaluated from the **lowest rule number to the highest**. As soon as a match is found, that rule is applied and evaluation stops.

| Rule # | Type | Protocol | Port | Source | Action |
|--------|------|----------|------|--------|--------|
| 100 | Inbound | TCP | 80 | `0.0.0.0/0` | **ALLOW** |
| 110 | Inbound | TCP | 443 | `0.0.0.0/0` | **ALLOW** |
| 120 | Inbound | TCP | 22 | `203.0.113.0/24` | **ALLOW** |
| 130 | Inbound | TCP | 22 | `0.0.0.0/0` | **DENY** |
| `*` | Inbound | All | All | `0.0.0.0/0` | **DENY** |

In this example:
- SSH from `203.0.113.0/24` → Matches rule 120 → **ALLOWED**
- SSH from `1.2.3.4` → Skips 100, 110, 120 → Matches 130 → **DENIED**
- The `*` rule is the default catch-all — it denies everything that doesn't match above

> **Tip:** Use rule numbers in increments of 10 or 100 (like 100, 110, 120). This gives you room to insert new rules between existing ones later.

### Creating NACL Rules (AWS CLI)

```bash
# Create a NACL
aws ec2 create-network-acl \
  --vpc-id vpc-0abc123def456 \
  --tag-specifications 'ResourceType=network-acl,Tags=[{Key=Name,Value=public-nacl}]'

# Add inbound rule — allow HTTP
aws ec2 create-network-acl-entry \
  --network-acl-id acl-0abc123 \
  --rule-number 100 \
  --protocol tcp \
  --port-range From=80,To=80 \
  --cidr-block 0.0.0.0/0 \
  --rule-action allow \
  --ingress

# Add outbound rule — allow ephemeral ports (for responses)
aws ec2 create-network-acl-entry \
  --network-acl-id acl-0abc123 \
  --rule-number 100 \
  --protocol tcp \
  --port-range From=1024,To=65535 \
  --cidr-block 0.0.0.0/0 \
  --rule-action allow \
  --egress

# Associate NACL with a subnet
aws ec2 replace-network-acl-association \
  --association-id aclassoc-0abc123 \
  --network-acl-id acl-0abc123
```

```
                                 │
                             Internet
```

### Typical 3-Tier Architecture

| Tier | Subnet Type | Resources | Security |
|------|------------|-----------|----------|
| **Web Tier** | Public | Load Balancer, Bastion Host | SG: Allow 80, 443 from internet |
| **App Tier** | Private | Application Servers | SG: Allow traffic from LB only |
| **Data Tier** | Private | Databases (RDS, ElastiCache) | SG: Allow traffic from App tier only |

## Creating a Complete VPC with Terraform

Here's a real-world Terraform example that creates a full VPC setup:

```hcl
# ─── VPC ───────────────────────────────────────────────
resource "aws_vpc" "main" {
  cidr_block           = "10.0.0.0/16"
  enable_dns_support   = true          # Enables DNS resolution in the VPC
  enable_dns_hostnames = true          # Gives EC2 instances public DNS names

  tags = {
    Name = "main-vpc"
  }
}

# ─── Internet Gateway ─────────────────────────────────
resource "aws_internet_gateway" "igw" {
  vpc_id = aws_vpc.main.id            # Attach to our VPC

  tags = {
    Name = "main-igw"
  }
}

# ─── Public Subnets ───────────────────────────────────
resource "aws_subnet" "public_1" {
  vpc_id                  = aws_vpc.main.id
  cidr_block              = "10.0.1.0/24"
  availability_zone       = "us-east-1a"
  map_public_ip_on_launch = true       # Auto-assign public IPs to instances

  tags = {
    Name = "public-subnet-1a"
  }
}

resource "aws_subnet" "public_2" {
  vpc_id                  = aws_vpc.main.id
  cidr_block              = "10.0.2.0/24"
  availability_zone       = "us-east-1b"
  map_public_ip_on_launch = true

  tags = {
    Name = "public-subnet-1b"
  }
}

# ─── Private Subnets ──────────────────────────────────
resource "aws_subnet" "private_1" {
  vpc_id            = aws_vpc.main.id
  cidr_block        = "10.0.3.0/24"
  availability_zone = "us-east-1a"

  tags = {
    Name = "private-subnet-1a"
  }
}

resource "aws_subnet" "private_2" {
  vpc_id            = aws_vpc.main.id
  cidr_block        = "10.0.4.0/24"
  availability_zone = "us-east-1b"

  tags = {
    Name = "private-subnet-1b"
  }
}

# ─── Elastic IP for NAT Gateway ────────────────────────
resource "aws_eip" "nat" {
  domain = "vpc"

  tags = {
    Name = "nat-eip"
  }
}

# ─── NAT Gateway (in Public Subnet) ───────────────────
resource "aws_nat_gateway" "nat" {
  allocation_id = aws_eip.nat.id
  subnet_id     = aws_subnet.public_1.id   # NAT lives in the PUBLIC subnet

  tags = {
    Name = "main-nat-gw"
  }

  depends_on = [aws_internet_gateway.igw]   # IGW must exist first
}

# ─── Public Route Table ───────────────────────────────
resource "aws_route_table" "public" {
  vpc_id = aws_vpc.main.id

  route {
    cidr_block = "0.0.0.0/0"
    gateway_id = aws_internet_gateway.igw.id   # Internet → IGW
  }

  tags = {
    Name = "public-rt"
  }
}
```

## VPC Peering — Connecting Two VPCs

**VPC Peering** lets two VPCs communicate with each other as if they were on the same network.

### Key Facts

- Works **across accounts** and **across regions**
- **No transitive peering** — if VPC-A peers with VPC-B, and VPC-B peers with VPC-C, VPC-A can **NOT** talk to VPC-C through VPC-B
- CIDR blocks must **not overlap**
- You must update **route tables on both sides**

```
┌──────────┐     VPC Peering     ┌──────────┐
│  VPC-A   │◄───────────────────►│  VPC-B   │
│10.0.0.0/16│    Connection     │10.1.0.0/16│
└──────────┘                     └──────────┘

VPC-A Route Table:                VPC-B Route Table:
10.1.0.0/16 → pcx-0abc123        10.0.0.0/16 → pcx-0abc123
```

## VPC Endpoints — Private Access to AWS Services

Instead of going through the internet to reach AWS services (like S3 or DynamoDB), you can use **VPC Endpoints** to access them **privately** within your VPC.

### Two Types of VPC Endpoints

| Type | Use Case | How It Works | Cost |
|------|---------|-------------|------|
| **Gateway Endpoint** | S3, DynamoDB | Adds a route in route table | **Free** |
| **Interface Endpoint** | Most other AWS services | Creates an ENI in your subnet | ~$0.01/hr |

```bash
# Create a Gateway Endpoint for S3
aws ec2 create-vpc-endpoint \
  --vpc-id vpc-0abc123def456 \
  --service-name com.amazonaws.us-east-1.s3 \
  --route-table-ids rtb-private-0abc123
```

> **Why use VPC Endpoints?**
> - **Security:** Traffic stays within AWS network (never touches the internet)
> - **Performance:** Lower latency
> - **Cost:** No NAT Gateway charges for S3/DynamoDB traffic

## VPC Flow Logs — Network Traffic Monitoring

**VPC Flow Logs** capture information about IP traffic going to and from network interfaces in your VPC. Think of it as a security camera for your network traffic.

### What They Capture

```
version account-id interface-id srcaddr dstaddr srcport dstport protocol packets bytes start end action log-status

2 123456789012 eni-0abc123 10.0.1.10 10.0.2.20 49152 80 6 10 840 1620140761 1620140821 ACCEPT OK
2 123456789012 eni-0abc123 203.0.113.5 10.0.1.10 12345 22 6 5 400 1620140761 1620140821 REJECT OK
```

| Field | Meaning |
|-------|---------|
| `srcaddr` | Source IP |
| `dstaddr` | Destination IP |
| `srcport` | Source port |
| `dstport` | Destination port |
| `action` | ACCEPT or REJECT |
| `protocol` | 6 = TCP, 17 = UDP, 1 = ICMP |

### Where to Store Flow Logs

- **CloudWatch Logs** — for real-time monitoring and alarms
- **S3** — for long-term storage and analysis
- **Kinesis Data Firehose** — for streaming to third-party tools

```bash
# Create flow logs for the entire VPC → send to CloudWatch
aws ec2 create-flow-logs \
  --resource-type VPC \
  --resource-ids vpc-0abc123def456 \
  --traffic-type ALL \
  --log-destination-type cloud-watch-logs \
  --log-group-name /aws/vpc/flow-logs \
  --deliver-logs-permission-arn arn:aws:iam::123456789012:role/flow-logs-role
```

## Quick Recap — VPC Decision Cheat Sheet

| Question | Answer |
|----------|--------|
| Resource needs internet access AND needs to be reachable? | Put in **public subnet** |
| Resource needs internet access but NOT reachable from internet? | Put in **private subnet** + **NAT Gateway** |
| Resource should have NO internet access at all? | Put in **private subnet**, no NAT route |
| Need firewall for a specific instance? | Use **Security Group** |
| Need firewall for an entire subnet? | Use **NACL** |
| Need two VPCs to communicate? | Use **VPC Peering** |
| Need to access S3 privately? | Use **Gateway VPC Endpoint** |
| Need to access other AWS services privately? | Use **Interface VPC Endpoint** |
| Need to see who's talking to whom? | Enable **VPC Flow Logs** |

## Common Mistakes to Avoid

1. **Forgetting ephemeral ports in NACLs** — NACLs are stateless! Always allow outbound ports 1024-65535
2. **Using the default VPC for production** — Create a custom VPC with proper planning
3. **Single AZ deployment** — Always use at least 2 AZs for high availability
4. **Overlapping CIDR blocks** — Plan ahead, especially if you'll need VPC peering
5. **Overly permissive Security Groups** — Don't use `0.0.0.0/0` for SSH! Restrict to known IPs
6. **One NAT Gateway for all AZs** — Creates a single point of failure. Use one per AZ
7. **Not using VPC Endpoints for S3** — You're paying NAT Gateway data charges unnecessarily

## Summary

| Component | Role | Key Thing to Remember |
|-----------|------|----------------------|
| **VPC** | Your private cloud network | Choose CIDR carefully — can't change later |
| **Subnet** | Division within VPC | Public = route to IGW, Private = route to NAT |
| **IGW** | Internet access for VPC | One per VPC, free, highly available |
| **NAT Gateway** | Internet for private resources | Lives in public subnet, needs Elastic IP |
| **Route Table** | Traffic directions | Every subnet needs one |
| **Security Group** | Instance firewall | Stateful, allow-only rules |
| **NACL** | Subnet firewall | Stateless, allow + deny rules, order matters |
