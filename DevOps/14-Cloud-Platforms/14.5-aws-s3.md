# 14.5 — AWS S3 (Simple Storage Service)

> **Level:** Beginner
> **Topic:** Buckets, Versioning, Lifecycle Policies, Replication, Encryption

---

## What is S3?

**Amazon S3 (Simple Storage Service)** is AWS's **object storage** service. It's one of the oldest and most popular AWS services — launched in 2006, and today it stores **trillions** of objects.

Think of S3 as an **unlimited hard drive in the cloud** where you can store any file — images, videos, backups, logs, application data, static websites, anything!

### Why Is S3 So Popular?

- **Unlimited storage** — no capacity limits
- **99.999999999% durability** (11 nines!) — AWS almost guarantees your data won't get lost
- **99.99% availability** — your data is almost always accessible
- **Extremely cheap** — starts at $0.023/GB/month
- **Built-in security** — encryption, access policies, versioning
- **Highly scalable** — handles any amount of data and traffic

> **Simple analogy:** S3 is like Google Drive or Dropbox, but designed for developers and applications, with way more features and almost infinite storage.

## Core Concepts

### Objects and Buckets

S3 has two main concepts:

| Concept | What It Is | Example |
|---------|-----------|---------|
| **Bucket** | A container (like a folder) that holds your files | `my-company-backups` |
| **Object** | A file stored in a bucket | `backups/database-2024-01-15.sql` |

```
S3
├── Bucket: my-app-assets
│   ├── images/logo.png           (Object)
│   ├── images/banner.jpg         (Object)
│   └── css/style.css             (Object)
│
├── Bucket: my-company-backups
│   ├── db/backup-2024-01-15.sql  (Object)
│   └── db/backup-2024-01-16.sql  (Object)
│
└── Bucket: my-app-logs
    ├── 2024/01/access.log        (Object)
    └── 2024/01/error.log         (Object)
```

### Object Anatomy

Every S3 object has these parts:

| Part | Description | Example |
|------|------------|---------|
| **Key** | The full path/name of the object | `images/photos/cat.jpg` |
| **Value** | The actual data (file content) | The cat image bytes |
| **Metadata** | Extra info about the object | Content-type, creation date |
| **Version ID** | If versioning is enabled | `abc123def456` |
| **Size** | Up to **5 TB** per object | Any size up to 5 TB |

> **Important:** S3 is a **flat storage** — there are no real folders. What looks like `images/photos/cat.jpg` is actually just a key (filename) that contains slashes. The AWS Console just shows it as if it were folders for convenience.

## Bucket Rules and Naming

### Bucket Naming Rules

- Must be **globally unique** across ALL AWS accounts (yes, the whole world!)
- 3 to 63 characters
- Only lowercase letters, numbers, and hyphens
- Must start with a letter or number
- No uppercase, no underscores, no spaces

```
✅ Good names: my-app-bucket, company-logs-2024, prod-backup-us-east-1
❌ Bad names:  My_Bucket, my bucket, MYBUCKET, -my-bucket
```

### Bucket Region

Each bucket lives in a specific **AWS Region**. Choose a region close to your users for lower latency.

```bash
# Create a bucket in us-east-1
aws s3 mb s3://my-awesome-bucket --region us-east-1

# Create a bucket in ap-south-1 (Mumbai)
aws s3 mb s3://my-india-bucket --region ap-south-1
```

## Working with S3 — AWS CLI

### Basic Operations

```bash
# ─── Create a bucket ──────────────────────────────────
aws s3 mb s3://my-app-bucket

# ─── List all buckets ─────────────────────────────────
aws s3 ls

# ─── Upload a file ────────────────────────────────────
aws s3 cp myfile.txt s3://my-app-bucket/
# Uploads myfile.txt to the root of the bucket

# ─── Upload to a "folder" ─────────────────────────────
aws s3 cp myfile.txt s3://my-app-bucket/documents/
# Uploads as: documents/myfile.txt

# ─── Upload an entire directory ───────────────────────
aws s3 cp ./my-folder s3://my-app-bucket/my-folder/ --recursive
# Uploads all files in ./my-folder to the bucket

# ─── Download a file ──────────────────────────────────
aws s3 cp s3://my-app-bucket/myfile.txt ./downloaded-file.txt

# ─── List objects in a bucket ─────────────────────────
aws s3 ls s3://my-app-bucket/
aws s3 ls s3://my-app-bucket/documents/

# ─── Delete a file ────────────────────────────────────
aws s3 rm s3://my-app-bucket/myfile.txt

# ─── Delete all files in a bucket ─────────────────────
aws s3 rm s3://my-app-bucket/ --recursive

# ─── Delete a bucket (must be empty first) ────────────
aws s3 rb s3://my-app-bucket

# ─── Delete a bucket AND all its contents ─────────────
aws s3 rb s3://my-app-bucket --force

# ─── Sync local folder with S3 (like rsync) ──────────
aws s3 sync ./local-folder s3://my-app-bucket/backup/
# Only uploads new/changed files — very efficient!

# ─── Sync S3 to local ─────────────────────────────────
aws s3 sync s3://my-app-bucket/backup/ ./local-folder
```

### Using `s3api` for More Control

The `s3api` commands give you lower-level control:

```bash
# Upload with specific content type and metadata
aws s3api put-object \
  --bucket my-app-bucket \
  --key images/logo.png \
  --body ./logo.png \
  --content-type image/png \
  --metadata '{"project":"myapp","version":"2.0"}'

# Get object metadata without downloading the file
aws s3api head-object \
  --bucket my-app-bucket \
  --key images/logo.png
```

## S3 Storage Classes — Saving Money

Not all data is accessed equally. S3 offers different **storage classes** with different prices and access patterns:

| Storage Class | Use Case | Retrieval Time | Cost (Storage) | Cost (Retrieval) |
|--------------|---------|---------------|---------------|-----------------|
| **S3 Standard** | Frequently accessed data | Instant | $$$ | Free |
| **S3 Intelligent-Tiering** | Unknown/changing access patterns | Instant | $$ | Small monitoring fee |
| **S3 Standard-IA** | Infrequent access (but needs fast retrieval) | Instant | $$ | Per-GB fee |
| **S3 One Zone-IA** | Infrequent access, re-creatable data | Instant | $ | Per-GB fee |
| **S3 Glacier Instant** | Archive with instant access | Instant | $ | Higher per-GB |
| **S3 Glacier Flexible** | Archive, accessed 1-2 times/year | 1-12 hours | ¢¢ | Per-GB + per-request |
| **S3 Glacier Deep Archive** | Long-term archive (7-10 years) | 12-48 hours | ¢ | Per-GB + per-request |

### Choosing the Right Class

```
Ask yourself:

1. How often will I access this data?
   ├── Multiple times a day      → S3 Standard
   ├── Don't know / changes      → Intelligent-Tiering
   ├── Once a month              → Standard-IA
   ├── Rarely (backup/archive)   → Glacier Flexible
   └── Almost never (compliance) → Glacier Deep Archive

2. Can I recreate this data if lost?
   ├── Yes  → One Zone-IA (cheaper, single AZ)
   └── No   → Multi-AZ options (Standard, Standard-IA, etc.)
```

### Setting Storage Class

```bash
# Upload with a specific storage class
aws s3 cp myfile.txt s3://my-bucket/ --storage-class STANDARD_IA

# Change storage class of an existing object (copies in-place)
aws s3 cp s3://my-bucket/myfile.txt s3://my-bucket/myfile.txt --storage-class GLACIER
```

## S3 Versioning — Never Lose a File Again

**Versioning** keeps **every version** of every object in your bucket. If you overwrite or delete a file, the old version is still there!

### How Versioning Works

```
Without Versioning:
Upload v1 → Upload v2 → v1 is GONE forever!

With Versioning:
Upload v1 → Upload v2 → Both v1 AND v2 exist!
                          You can retrieve either one.
```

### Enabling Versioning

```bash
# Enable versioning on a bucket
aws s3api put-bucket-versioning \
  --bucket my-app-bucket \
  --versioning-configuration Status=Enabled

# Check versioning status
aws s3api get-bucket-versioning --bucket my-app-bucket
```

### Working with Versions

```bash
# List all versions of all objects
aws s3api list-object-versions --bucket my-app-bucket

# Get a specific version of an object
aws s3api get-object \
  --bucket my-app-bucket \
  --key myfile.txt \
  --version-id "abc123def456" \
  downloaded-file.txt

# Delete a specific version (permanently)
aws s3api delete-object \
  --bucket my-app-bucket \
  --key myfile.txt \
  --version-id "abc123def456"
```

### What Happens When You "Delete" with Versioning?

When versioning is ON and you "delete" an object, S3 doesn't actually remove it. Instead, it places a **delete marker** — a placeholder that says "this file appears deleted."

```
Timeline:
1. Upload report.pdf        → Version ID: v1 (current)
2. Upload report.pdf again  → Version ID: v2 (current), v1 still exists
3. Delete report.pdf        → Delete Marker added (v1 and v2 still exist!)
4. List objects              → report.pdf appears GONE
5. List versions             → v1, v2, and Delete Marker all visible

To restore: just delete the Delete Marker!
```

> **Important:**
> - Versioning can only be **enabled** or **suspended** — never fully disabled once turned on
> - Every version is stored and **billed separately** — this can increase costs!

## S3 Lifecycle Policies — Automatic Cost Optimization

**Lifecycle policies** automatically move or delete objects based on rules you define. This saves money by transitioning old data to cheaper storage classes or cleaning up expired data.

### Common Lifecycle Scenarios

| Scenario | Rule |
|---------|------|
| Move logs to IA after 30 days | Transition to `STANDARD_IA` after 30 days |
| Archive old backups after 90 days | Transition to `GLACIER` after 90 days |
| Delete temp files after 7 days | Expire objects after 7 days |
| Clean up old versions | Expire non-current versions after 30 days |
| Clean up incomplete uploads | Abort multipart uploads after 7 days |

### Lifecycle Policy Example (JSON)

```json
{
  "Rules": [
    {
      "ID": "MoveToIAAfter30Days",
      "Status": "Enabled",
      "Filter": {
        "Prefix": "logs/"
      },
      "Transitions": [
        {
          "Days": 30,
          "StorageClass": "STANDARD_IA"
        }
      ]
    },
    {
      "ID": "ArchiveAfter90Days",
      "Status": "Enabled",
      "Filter": {
        "Prefix": "backups/"
      },
      "Transitions": [
        {
          "Days": 90,
          "StorageClass": "GLACIER"
        },
        {
          "Days": 365,
          "StorageClass": "DEEP_ARCHIVE"
        }
      ]
    },
    {
      "ID": "DeleteTempFilesAfter7Days",
      "Status": "Enabled",
      "Filter": {
        "Prefix": "tmp/"
      },
      "Expiration": {
        "Days": 7
      }
    },
    {
      "ID": "CleanupOldVersions",
      "Status": "Enabled",
      "Filter": {},
      "NoncurrentVersionTransitions": [
        {
          "NoncurrentDays": 30,
          "StorageClass": "STANDARD_IA"
        }
      ],
      "NoncurrentVersionExpiration": {
        "NoncurrentDays": 90
      }
    },
    {
      "ID": "AbortIncompleteUploads",
      "Status": "Enabled",
      "Filter": {},
      "AbortIncompleteMultipartUpload": {
        "DaysAfterInitiation": 7
      }
    }
  ]
}
```

### Applying Lifecycle Policy

```bash
# Apply the lifecycle policy
aws s3api put-bucket-lifecycle-configuration \
  --bucket my-app-bucket \
  --lifecycle-configuration file://lifecycle-policy.json

# View current lifecycle policy
aws s3api get-bucket-lifecycle-configuration --bucket my-app-bucket
```

### Lifecycle Transition Flow

```
Day 0          Day 30              Day 90              Day 365
  │               │                   │                   │
  ▼               ▼                   ▼                   ▼
┌────────┐   ┌──────────┐   ┌────────────────┐   ┌──────────────┐
│Standard │──►│Standard-IA│──►│Glacier Flexible│──►│Deep Archive  │
│  $$$    │   │    $$     │   │      ¢¢        │   │     ¢        │
└────────┘   └──────────┘   └────────────────┘   └──────────────┘

Cost goes DOWN ───────────────────────────────────────────────►
Retrieval time goes UP ───────────────────────────────────────►
```

## S3 Replication — Copying Data Across Buckets

**Replication** automatically copies objects from one bucket to another. There are two types:

| Type | What It Does | Use Case |
|------|-------------|---------|
| **CRR** (Cross-Region Replication) | Copies to a bucket in a **different region** | Disaster recovery, lower latency |
| **SRR** (Same-Region Replication) | Copies to a bucket in the **same region** | Log aggregation, compliance |

### Requirements for Replication

1. **Versioning must be enabled** on BOTH source and destination buckets
2. Buckets can be in different AWS accounts
3. S3 needs an **IAM role** with permissions to replicate
4. Only **new objects** are replicated (existing objects are NOT retroactively copied)

### Setting Up Cross-Region Replication

```bash
# Step 1: Enable versioning on source bucket
aws s3api put-bucket-versioning \
  --bucket source-bucket \
  --versioning-configuration Status=Enabled

# Step 2: Enable versioning on destination bucket
aws s3api put-bucket-versioning \
  --bucket destination-bucket \
  --versioning-configuration Status=Enabled

# Step 3: Create replication configuration
```

**Replication Configuration (JSON):**

```json
{
  "Role": "arn:aws:iam::123456789012:role/s3-replication-role",
  "Rules": [
    {
      "ID": "ReplicateEverything",
      "Status": "Enabled",
      "Priority": 1,
      "Filter": {},
      "Destination": {
        "Bucket": "arn:aws:s3:::destination-bucket",
        "StorageClass": "STANDARD_IA"
      },
      "DeleteMarkerReplication": {
        "Status": "Disabled"
      }
    }
  ]
}
```

```bash
# Step 4: Apply replication configuration
aws s3api put-bucket-replication \
  --bucket source-bucket \
  --replication-configuration file://replication-config.json
```

### Replication with Terraform

```hcl
resource "aws_s3_bucket" "source" {
  bucket = "my-source-bucket"
}

resource "aws_s3_bucket_versioning" "source" {
  bucket = aws_s3_bucket.source.id
  versioning_configuration {
    status = "Enabled"
  }
}

resource "aws_s3_bucket" "destination" {
  provider = aws.us-west-2          # Different region
  bucket   = "my-destination-bucket"
}

resource "aws_s3_bucket_versioning" "destination" {
  provider = aws.us-west-2
  bucket   = aws_s3_bucket.destination.id
  versioning_configuration {
    status = "Enabled"
  }
}

resource "aws_s3_bucket_replication_configuration" "replication" {
  bucket = aws_s3_bucket.source.id
  role   = aws_iam_role.replication.arn

  rule {
    id     = "replicate-all"
    status = "Enabled"

    destination {
      bucket        = aws_s3_bucket.destination.arn
      storage_class = "STANDARD_IA"
    }
  }
}
```

## S3 Encryption — Protecting Your Data

S3 offers multiple encryption options to protect your data at rest and in transit.

### Encryption in Transit

All S3 traffic uses **HTTPS** by default. You can enforce HTTPS-only with a bucket policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyHTTP",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:*",
      "Resource": [
        "arn:aws:s3:::my-bucket",
        "arn:aws:s3:::my-bucket/*"
      ],
      "Condition": {
        "Bool": {
          "aws:SecureTransport": "false"
        }
      }
    }
  ]
}
```

### Encryption at Rest — Types

| Type | Key Managed By | Key Storage | Cost |
|------|---------------|------------|------|
| **SSE-S3** | AWS (automatic) | AWS manages everything | Free |
| **SSE-KMS** | AWS KMS | You control via KMS policies | KMS key cost + API calls |
| **SSE-C** | You (customer) | You provide the key with every request | Free (you manage keys) |
| **Client-Side** | You | Encrypt before uploading | Free (you do all the work) |

### SSE-S3 (Default — Recommended for Most Use Cases)

AWS manages everything. Every object is encrypted with a unique key, and that key is itself encrypted with a master key that AWS rotates regularly.

```bash
# As of January 2023, S3 encrypts ALL new objects by default with SSE-S3!
# You don't need to do anything — it just works.

# But you can explicitly specify it:
aws s3 cp myfile.txt s3://my-bucket/ --sse AES256
```

### SSE-KMS (When You Need More Control)

Uses AWS Key Management Service. You get audit trails (who used which key when), and you can control key access with IAM policies.

```bash
# Upload with SSE-KMS using default AWS-managed KMS key
aws s3 cp myfile.txt s3://my-bucket/ --sse aws:kms

# Upload with SSE-KMS using YOUR custom KMS key
aws s3 cp myfile.txt s3://my-bucket/ \
  --sse aws:kms \
  --sse-kms-key-id arn:aws:kms:us-east-1:123456789012:key/my-key-id
```

### Setting Default Encryption on a Bucket

```bash
# Set default encryption to SSE-S3
aws s3api put-bucket-encryption \
  --bucket my-bucket \
  --server-side-encryption-configuration '{
    "Rules": [{
      "ApplyServerSideEncryptionByDefault": {
        "SSEAlgorithm": "AES256"
      },
      "BucketKeyEnabled": true
    }]
  }'

# Set default encryption to SSE-KMS
aws s3api put-bucket-encryption \
  --bucket my-bucket \
  --server-side-encryption-configuration '{
    "Rules": [{
      "ApplyServerSideEncryptionByDefault": {
        "SSEAlgorithm": "aws:kms",
        "KMSMasterKeyID": "arn:aws:kms:us-east-1:123456789012:key/my-key-id"
      },
      "BucketKeyEnabled": true
    }]
  }'
```

> **Bucket Key:** Enabling `BucketKeyEnabled` reduces KMS API call costs by using a bucket-level key for encryption. Always enable this when using SSE-KMS.

## S3 Access Control — Who Can Access What?

S3 has multiple layers of access control. Let's understand each one:

### 1. Bucket Policies (Most Common)

JSON-based policies attached to the bucket. They define who can do what.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "PublicReadForWebsite",
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-website-bucket/*"
    },
    {
      "Sid": "DenyDeleteForEveryone",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:DeleteObject",
      "Resource": "arn:aws:s3:::my-website-bucket/*",
      "Condition": {
        "StringNotEquals": {
          "aws:PrincipalArn": "arn:aws:iam::123456789012:role/admin-role"
        }
      }
    }
  ]
}
```

### 2. IAM Policies

Attached to IAM users, groups, or roles. Controls what S3 actions the user/role can perform.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::my-app-bucket",
        "arn:aws:s3:::my-app-bucket/*"
      ]
    }
  ]
}
```

### 3. S3 Block Public Access (Safety Net)

A bucket-level or account-level setting that **blocks public access** regardless of what bucket policies or ACLs say. Enabled by default.

```bash
# Check block public access settings
aws s3api get-public-access-block --bucket my-bucket

# Enable all block public access settings (recommended!)
aws s3api put-public-access-block \
  --bucket my-bucket \
  --public-access-block-configuration \
    BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true
```

> **Best Practice:** Keep Block Public Access **ON** for all buckets unless you specifically need public access (like a static website).

### 4. ACLs (Legacy — Avoid)

Access Control Lists are the old way of controlling access. AWS recommends using bucket policies instead.

```bash
# Disable ACLs (recommended for new buckets)
aws s3api put-bucket-ownership-controls \
  --bucket my-bucket \
  --ownership-controls Rules=[{ObjectOwnership=BucketOwnerEnforced}]
```

## S3 Static Website Hosting

S3 can host **static websites** (HTML, CSS, JS) directly — no web server needed!

```bash
# Step 1: Enable static website hosting
aws s3 website s3://my-website-bucket/ \
  --index-document index.html \
  --error-document error.html

# Step 2: Upload website files
aws s3 sync ./website-folder s3://my-website-bucket/

# Step 3: Disable block public access (needed for public website)
aws s3api put-public-access-block \
  --bucket my-website-bucket \
  --public-access-block-configuration \
    BlockPublicAcls=false,IgnorePublicAcls=false,BlockPublicPolicy=false,RestrictPublicBuckets=false

# Step 4: Add bucket policy for public read
aws s3api put-bucket-policy \
  --bucket my-website-bucket \
  --policy '{
    "Version": "2012-10-17",
    "Statement": [{
      "Sid": "PublicReadGetObject",
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-website-bucket/*"
    }]
  }'
```

Your website will be available at:
```
http://my-website-bucket.s3-website-us-east-1.amazonaws.com
```

> **Pro Tip:** For production, put **CloudFront** (CDN) in front of your S3 website for HTTPS, caching, and better performance.

## S3 Presigned URLs — Temporary Access

**Presigned URLs** let you give someone temporary access to a private S3 object without making it public or sharing your credentials.

```bash
# Generate a presigned URL (valid for 1 hour by default)
aws s3 presign s3://my-private-bucket/secret-report.pdf

# Generate with custom expiration (e.g., 5 minutes = 300 seconds)
aws s3 presign s3://my-private-bucket/secret-report.pdf --expires-in 300
```

**Output:**
```
https://my-private-bucket.s3.amazonaws.com/secret-report.pdf?X-Amz-Algorithm=AWS4-HMAC-SHA256&X-Amz-Credential=...&X-Amz-Expires=300&X-Amz-Signature=...
```

Anyone with this URL can download the file for the next 5 minutes, after which the link expires.

### Use Cases for Presigned URLs

- Share a private file with a client temporarily
- Allow users to upload files directly to S3 (presigned PUT URL)
- Generate download links in your application

## Multipart Upload — Handling Large Files

For files larger than **100 MB**, AWS recommends **multipart upload**. It splits the file into smaller parts and uploads them in parallel.

```
Large File (5 GB)
       │
       ▼
┌──────────────┐
│  Split into   │
│  parts        │
└──────┬───────┘
       │
  ┌────┼────┐
  ▼    ▼    ▼
Part1 Part2 Part3  ──► Uploaded in parallel
  │    │    │
  └────┼────┘
       ▼
┌──────────────┐
│  S3 combines  │
│  into 1 file  │
└──────────────┘
```

### Benefits

- **Faster uploads** — parallel parts
- **Resume on failure** — retry only failed parts, not the whole file
- **Required** for files > 5 GB

```bash
# The AWS CLI automatically uses multipart upload for large files
aws s3 cp large-file.zip s3://my-bucket/

# You can configure the multipart threshold and chunk size
aws configure set default.s3.multipart_threshold 64MB
aws configure set default.s3.multipart_chunksize 16MB
```

## S3 Event Notifications — React to Changes

S3 can send notifications when certain events happen in your bucket:

| Event | Trigger |
|-------|---------|
| `s3:ObjectCreated:*` | When a new object is uploaded |
| `s3:ObjectRemoved:*` | When an object is deleted |
| `s3:ObjectRestore:*` | When a Glacier object is restored |
| `s3:Replication:*` | When replication fails |

### Destinations for Notifications

- **AWS Lambda** — Run code when a file is uploaded
- **SQS** — Queue the event for processing
- **SNS** — Send notifications (email, SMS, etc.)
- **EventBridge** — Route to any AWS service

### Example: Trigger Lambda on Upload

```json
{
  "LambdaFunctionConfigurations": [
    {
      "Id": "ProcessNewImage",
      "LambdaFunctionArn": "arn:aws:lambda:us-east-1:123456789012:function:resize-image",
      "Events": ["s3:ObjectCreated:*"],
      "Filter": {
        "Key": {
          "FilterRules": [
            {
              "Name": "prefix",
              "Value": "uploads/"
            },
            {
              "Name": "suffix",
              "Value": ".jpg"
            }
          ]
        }
      }
    }
  ]
}
```

This triggers the Lambda function whenever a `.jpg` file is uploaded to the `uploads/` prefix.

## S3 Performance Optimization

### Transfer Acceleration

Speeds up uploads by routing through AWS CloudFront edge locations:

```bash
# Enable transfer acceleration
aws s3api put-bucket-accelerate-configuration \
  --bucket my-bucket \
  --accelerate-configuration Status=Enabled

# Upload using accelerated endpoint
aws s3 cp large-file.zip s3://my-bucket/ --endpoint-url https://my-bucket.s3-accelerate.amazonaws.com
```

### Performance Tips

| Tip | Details |
|-----|---------|
| **Multipart upload** | Use for files > 100 MB |
| **Transfer Acceleration** | Use for cross-region uploads |
| **S3 Select** | Query only the data you need from CSV/JSON files |
| **Byte-range fetches** | Download only part of a file |
| **Prefix distribution** | Spread objects across different prefixes for better throughput |

## Quick Recap — S3 Cheat Sheet

| Feature | What It Does | Key Thing to Remember |
|---------|-------------|----------------------|
| **Buckets** | Containers for objects | Globally unique names |
| **Storage Classes** | Different cost tiers | Match class to access pattern |
| **Versioning** | Keeps all versions | Can't be fully disabled once on |
| **Lifecycle** | Auto-transition/delete | Great for cost savings |
| **Replication** | Copy across regions/buckets | Needs versioning on both ends |
| **Encryption** | Protect data at rest | SSE-S3 is default since Jan 2023 |
| **Bucket Policies** | Access control | Use instead of ACLs |
| **Block Public Access** | Safety net | Keep ON unless you need public |
| **Presigned URLs** | Temporary access | Great for sharing private files |
| **Event Notifications** | React to changes | Trigger Lambda, SQS, SNS |

## Common Mistakes to Avoid

1. **Making buckets public accidentally** — Always enable Block Public Access
2. **Not using lifecycle policies** — You're paying for data you don't actively use
3. **Ignoring versioning costs** — Every version is stored separately and billed
4. **Not enabling encryption** — AWS does SSE-S3 by default now, but verify for compliance
5. **Using ACLs** — They're legacy. Use bucket policies instead
6. **Not cleaning up incomplete multipart uploads** — These silently accumulate and cost money
7. **Putting everything in S3 Standard** — Use Intelligent-Tiering if unsure about access patterns
