# 14.6 — AWS Route 53 (DNS Service)

> **Level:** Intermediate
> **Topic:** DNS, Hosted Zones, Record Types, Routing Policies, Health Checks

---

## What is Route 53?

**Amazon Route 53** is AWS's **DNS (Domain Name System)** service. It does three main things:

1. **Domain Registration** — Buy domain names (like `myapp.com`)
2. **DNS Routing** — Translate domain names to IP addresses (so `myapp.com` → `54.23.1.100`)
3. **Health Checking** — Monitor the health of your resources and route traffic away from unhealthy ones

### Why "Route 53"?

The name comes from **Port 53**, which is the port that DNS uses. That's it — just a nerdy naming choice by AWS!

> **Simple analogy:** Route 53 is like the **phone book of the internet**. When someone types `myapp.com`, Route 53 looks up the "phone number" (IP address) and connects them to the right server.

### Why Not Just Use Any DNS Provider?

You totally can use Cloudflare, GoDaddy, etc. But Route 53 has some big advantages:

- **Deeply integrated with AWS** — works seamlessly with ALB, S3, CloudFront, EC2
- **100% availability SLA** — the only AWS service with this guarantee
- **Alias records** — a special Route 53 feature for AWS resources (free, no extra DNS queries)
- **Advanced routing** — latency-based, geolocation, failover, weighted
- **Health checks** — built-in monitoring with automatic failover

## How DNS Works — Quick Refresher

Before diving into Route 53, let's quickly understand how DNS works:

```
You type: www.myapp.com

1. Your browser asks: "What's the IP for www.myapp.com?"
        │
        ▼
2. DNS Resolver (your ISP) checks its cache
   ├── Found? → Returns the IP immediately
   └── Not found? → Asks the root servers
        │
        ▼
3. Root DNS Server: "I don't know, but ask the .com TLD server"
        │
        ▼
4. .com TLD Server: "I don't know, but ask Route 53 (the authoritative server for myapp.com)"
        │
        ▼
5. Route 53: "www.myapp.com = 54.23.1.100" ✅
        │
        ▼
6. Browser connects to 54.23.1.100
```

### Key DNS Terms

| Term | Meaning | Example |
|------|---------|---------|
| **Domain** | The website address | `myapp.com` |
| **Subdomain** | A prefix to the domain | `api.myapp.com`, `blog.myapp.com` |
| **Record** | A DNS entry mapping a name to a value | `myapp.com → 54.23.1.100` |
| **TTL** | Time To Live — how long to cache the record | `300` (5 minutes) |
| **Hosted Zone** | A container for DNS records of a domain | All records for `myapp.com` |
| **Authoritative DNS** | The final authority for a domain's DNS | Route 53 for your domain |

## Hosted Zones — Where Your DNS Records Live

A **Hosted Zone** is a container that holds all the DNS records for a domain. Think of it as a folder that contains all the "directions" for your domain.

### Two Types of Hosted Zones

| Type | What It Does | Accessible From | Cost |
|------|-------------|-----------------|------|
| **Public Hosted Zone** | Routes traffic from the internet | Anyone on the internet | $0.50/month per zone |
| **Private Hosted Zone** | Routes traffic within a VPC | Only within your VPC(s) | $0.50/month per zone |

### Public Hosted Zone

Used for domains accessible from the internet (most common):

```
Public Hosted Zone: myapp.com
├── A Record:     myapp.com        → 54.23.1.100
├── A Record:     api.myapp.com    → 54.23.1.200
├── CNAME Record: www.myapp.com    → myapp.com
├── MX Record:    myapp.com        → mail.myapp.com
└── TXT Record:   myapp.com        → "v=spf1 include:_spf.google.com ~all"
```

### Private Hosted Zone

Used for internal DNS within your VPC — your private resources can have friendly names:

```
Private Hosted Zone: internal.myapp.com (attached to VPC)
├── A Record: db.internal.myapp.com      → 10.0.3.50  (RDS instance)
├── A Record: cache.internal.myapp.com   → 10.0.3.100 (ElastiCache)
└── A Record: api.internal.myapp.com     → 10.0.2.25  (Internal API)
```

> Your app servers can use `db.internal.myapp.com` instead of remembering `10.0.3.50`. Much easier!

### Creating a Hosted Zone (AWS CLI)

```bash
# Create a public hosted zone
aws route53 create-hosted-zone \
  --name myapp.com \
  --caller-reference "unique-string-$(date +%s)"

# Create a private hosted zone (attached to a VPC)
aws route53 create-hosted-zone \
  --name internal.myapp.com \
  --caller-reference "unique-string-$(date +%s)" \
  --vpc VPCRegion=us-east-1,VPCId=vpc-0abc123def456 \
  --hosted-zone-config PrivateZone=true

# List all hosted zones
aws route53 list-hosted-zones
```

## DNS Record Types — The Complete Guide

Route 53 supports many DNS record types. Here's every important one explained:

### A Record (Address)

Maps a domain name to an **IPv4 address**.

```
myapp.com  →  54.23.1.100
```

This is the most basic and common record type. When someone visits `myapp.com`, this tells their browser which server to connect to.

### AAAA Record (IPv6 Address)

Same as A record, but for **IPv6 addresses**.

```
myapp.com  →  2600:1f18:4b3:ab00::1
```

### CNAME Record (Canonical Name)

Maps a domain name to **another domain name** (not an IP).

```
www.myapp.com  →  myapp.com
blog.myapp.com →  myapp.blog-platform.com
```

> **Important limitation:** You CANNOT create a CNAME for the **root domain** (apex domain). For example, you can't create a CNAME for `myapp.com` — only for subdomains like `www.myapp.com`. This is a DNS standard limitation, not just Route 53.

### Alias Record (Route 53 Special!)

An **Alias** record is a Route 53-specific extension. It works like a CNAME but:
- ✅ Can be used for the **root domain** (`myapp.com`)
- ✅ **Free** — no charge for DNS queries
- ✅ Automatically resolves to the resource's IP

```
myapp.com → d1234.cloudfront.net     (Alias to CloudFront)
myapp.com → my-alb-1234.elb.amazonaws.com  (Alias to ALB)
```

### Alias vs CNAME — When to Use Which

| Feature | CNAME | Alias |
|---------|-------|-------|
| Root domain (apex)? | ❌ Not allowed | ✅ Allowed |
| Query cost? | Standard charges | **Free** |
| AWS resources? | Works but not ideal | **Built for this** |
| External domains? | ✅ Works | ❌ Only AWS resources |
| Health check? | Not directly | ✅ Built-in |

**Supported Alias Targets:**
- Elastic Load Balancer (ALB, NLB, CLB)
- CloudFront Distribution
- S3 Website Endpoint
- API Gateway
- VPC Interface Endpoint
- Another Route 53 record in the same hosted zone
- AWS Global Accelerator
- Elastic Beanstalk

> **Rule of thumb:** If you're pointing to an AWS resource, **always use Alias**. If you're pointing to an external domain, use CNAME.

### MX Record (Mail Exchange)

Specifies the mail servers for your domain:

```
myapp.com  →  10 mail1.myapp.com
               20 mail2.myapp.com
```

The number (10, 20) is the **priority** — lower number = higher priority. Mail goes to `mail1` first; if it's down, it goes to `mail2`.

### TXT Record (Text)

Stores any text. Commonly used for verification and email security:

```
myapp.com  →  "v=spf1 include:_spf.google.com ~all"        (Email SPF)
myapp.com  →  "google-site-verification=abc123..."           (Domain verification)
myapp.com  →  "v=DKIM1; k=rsa; p=MIGfMA0GCSqGSIb3..."      (Email DKIM)
```

### NS Record (Name Server)

Identifies the authoritative DNS servers for your domain. AWS creates these automatically when you create a hosted zone.

```
myapp.com  →  ns-123.awsdns-45.com
               ns-678.awsdns-90.net
               ns-234.awsdns-56.org
               ns-901.awsdns-12.co.uk
```

> These are the servers that Route 53 uses to answer DNS queries for your domain. You need to configure these at your domain registrar.

### SOA Record (Start of Authority)

Contains administrative information about your hosted zone. AWS manages this automatically — you rarely need to touch it.

### SRV Record (Service)

Used for service discovery. Specifies the host and port for specific services:

```
_sip._tcp.myapp.com  →  10 60 5060 sip.myapp.com
```

Format: `priority weight port target`

### PTR Record (Pointer)

Reverse DNS — maps an IP address back to a domain name. Used in email servers for spam verification.

## Creating DNS Records — Practical Examples

### Using AWS CLI

Route 53 CLI uses a JSON file for record changes:

```bash
# Create an A record
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --change-batch '{
    "Changes": [{
      "Action": "CREATE",
      "ResourceRecordSet": {
        "Name": "myapp.com",
        "Type": "A",
        "TTL": 300,
        "ResourceRecords": [{"Value": "54.23.1.100"}]
      }
    }]
  }'

# Create a CNAME record
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --change-batch '{
    "Changes": [{
      "Action": "CREATE",
      "ResourceRecordSet": {
        "Name": "www.myapp.com",
        "Type": "CNAME",
        "TTL": 300,
        "ResourceRecords": [{"Value": "myapp.com"}]
      }
    }]
  }'

# Create an Alias record pointing to an ALB
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --change-batch '{
    "Changes": [{
      "Action": "CREATE",
      "ResourceRecordSet": {
        "Name": "myapp.com",
        "Type": "A",
        "AliasTarget": {
          "HostedZoneId": "Z35SXDOTRQ7X7K",
          "DNSName": "my-alb-1234.us-east-1.elb.amazonaws.com",
          "EvaluateTargetHealth": true
        }
      }
    }]
  }'

# Create an MX record (for email)
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --change-batch '{
    "Changes": [{
      "Action": "CREATE",
      "ResourceRecordSet": {
        "Name": "myapp.com",
        "Type": "MX",
        "TTL": 300,
        "ResourceRecords": [
          {"Value": "10 mail1.myapp.com"},
          {"Value": "20 mail2.myapp.com"}
        ]
      }
    }]
  }'

# List all records in a hosted zone
aws route53 list-resource-record-sets --hosted-zone-id Z1234567890ABC
```

### Actions Explained

| Action | What It Does |
|--------|-------------|
| `CREATE` | Create a new record (fails if record exists) |
| `DELETE` | Delete an existing record |
| `UPSERT` | Create if doesn't exist, update if it does (safest option!) |

> **Tip:** Always use `UPSERT` instead of `CREATE` — it's idempotent and won't fail if the record already exists.

## TTL (Time to Live) — How Long to Cache

**TTL** tells DNS resolvers how long to cache your DNS record (in seconds).

```
High TTL (e.g., 86400 = 24 hours):
├── ✅ Fewer DNS queries (cheaper, faster)
├── ❌ Changes take longer to propagate
└── Best for: Stable records that rarely change

Low TTL (e.g., 60 = 1 minute):
├── ❌ More DNS queries (slightly more cost)
├── ✅ Changes propagate quickly
└── Best for: Records you might change (failover, migrations)
```

### TTL Strategy for Migrations

When you're about to change a DNS record (like during a server migration):

```
1. BEFORE migration: Lower TTL to 60 seconds
   (Wait 24-48 hours for old TTL to expire everywhere)

2. DURING migration: Change the DNS record
   (Within 60 seconds, everyone gets the new IP)

3. AFTER migration: Raise TTL back to 3600 or higher
   (Reduce DNS query costs)
```

> **Alias records don't have TTL** — Route 53 automatically uses the TTL of the target resource.

## Routing Policies — The Power of Route 53

This is where Route 53 really shines. Routing policies determine **how Route 53 responds** to DNS queries.

### 1. Simple Routing

The most basic — one record, one or more values. If multiple values, Route 53 returns all of them in random order.

```
myapp.com → 54.23.1.100
              54.23.1.200
              54.23.1.300

Client gets all three IPs in random order.
The client picks one (usually the first).
```

**Use case:** Single resource, or simple round-robin across multiple resources.

> **Limitation:** No health checks with simple routing.

### 2. Weighted Routing

Distribute traffic based on **weights** you assign. Perfect for gradual rollouts!

```
Record 1: myapp.com → 54.23.1.100  (Weight: 70)  ← 70% of traffic
Record 2: myapp.com → 54.23.1.200  (Weight: 20)  ← 20% of traffic
Record 3: myapp.com → 54.23.1.300  (Weight: 10)  ← 10% of traffic
```

**Use cases:**
- **Blue/Green deployment** — Send 90% to blue, 10% to green, gradually shift
- **A/B testing** — Send different percentages to different versions
- **Load distribution** — Distribute traffic across regions

```bash
# Create weighted record (70% weight)
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --change-batch '{
    "Changes": [{
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "myapp.com",
        "Type": "A",
        "SetIdentifier": "primary-server",
        "Weight": 70,
        "TTL": 300,
        "ResourceRecords": [{"Value": "54.23.1.100"}]
      }
    }]
  }'

# Create weighted record (30% weight)
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --change-batch '{
    "Changes": [{
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "myapp.com",
        "Type": "A",
        "SetIdentifier": "secondary-server",
        "Weight": 30,
        "TTL": 300,
        "ResourceRecords": [{"Value": "54.23.1.200"}]
      }
    }]
  }'
```

> **Note:** `SetIdentifier` must be unique for each weighted record with the same name and type.

### 3. Latency-Based Routing

Routes users to the region with the **lowest latency** (fastest response time).

```
User in New York  → us-east-1 (closest)  → 54.23.1.100
User in London    → eu-west-1 (closest)  → 52.48.2.200
User in Tokyo     → ap-northeast-1       → 13.114.3.300
```

Route 53 maintains a latency database and automatically routes users to the region that will give them the fastest experience.

```bash
# Latency-based record for us-east-1
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --change-batch '{
    "Changes": [{
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "myapp.com",
        "Type": "A",
        "SetIdentifier": "us-east-1",
        "Region": "us-east-1",
        "TTL": 300,
        "ResourceRecords": [{"Value": "54.23.1.100"}]
      }
    }]
  }'

# Latency-based record for eu-west-1
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --change-batch '{
    "Changes": [{
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "myapp.com",
        "Type": "A",
        "SetIdentifier": "eu-west-1",
        "Region": "eu-west-1",
        "TTL": 300,
        "ResourceRecords": [{"Value": "52.48.2.200"}]
      }
    }]
  }'
```

### 4. Failover Routing

Routes traffic to a **primary** resource, and automatically switches to a **secondary** (backup) resource if the primary is unhealthy.

```
Normal:
User → myapp.com → Primary (54.23.1.100) ✅

Primary goes down:
User → myapp.com → Secondary (54.23.1.200) ✅ (automatic failover!)
```

```bash
# Primary record (with health check)
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --change-batch '{
    "Changes": [{
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "myapp.com",
        "Type": "A",
        "SetIdentifier": "primary",
        "Failover": "PRIMARY",
        "TTL": 60,
        "ResourceRecords": [{"Value": "54.23.1.100"}],
        "HealthCheckId": "health-check-id-123"
      }
    }]
  }'

# Secondary record (backup — can point to S3 static "sorry" page)
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --change-batch '{
    "Changes": [{
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "myapp.com",
        "Type": "A",
        "SetIdentifier": "secondary",
        "Failover": "SECONDARY",
        "TTL": 60,
        "ResourceRecords": [{"Value": "54.23.1.200"}]
      }
    }]
  }'
```

### 5. Geolocation Routing

Routes traffic based on the **geographic location** of the user.

```
User in Europe     → EU server
User in Asia       → Asia server
User in US         → US server
User anywhere else → Default server
```

**Use cases:**
- Content localization (different language/content per region)
- Legal compliance (keep data in specific regions)
- Restricting access by country

```bash
# Route European users to EU server
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --change-batch '{
    "Changes": [{
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "myapp.com",
        "Type": "A",
        "SetIdentifier": "europe",
        "GeoLocation": {"ContinentCode": "EU"},
        "TTL": 300,
        "ResourceRecords": [{"Value": "52.48.2.200"}]
      }
    }]
  }'

# Default record (MUST have this — catches everyone else)
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1234567890ABC \
  --change-batch '{
    "Changes": [{
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "myapp.com",
        "Type": "A",
        "SetIdentifier": "default",
        "GeoLocation": {"CountryCode": "*"},
        "TTL": 300,
        "ResourceRecords": [{"Value": "54.23.1.100"}]
      }
    }]
  }'
```

> **Important:** Always create a **default** geolocation record. Without it, users from unmatched locations get no response!

### 6. Geoproximity Routing (Traffic Flow)

Similar to geolocation but with a **bias** — you can expand or shrink a region's "catchment area" to shift more or less traffic to it.

```
US-East bias: +25  (attracts more traffic)
US-West bias: -10  (pushes traffic away)
```

> This requires **Route 53 Traffic Flow**, which is a more advanced (and paid) feature.

### 7. Multivalue Answer Routing

Returns **multiple healthy IPs** in response to a query. Each record can have a health check, and Route 53 only returns healthy ones.

```
myapp.com → 54.23.1.100 (healthy ✅) → returned
myapp.com → 54.23.1.200 (unhealthy ❌) → NOT returned
myapp.com → 54.23.1.300 (healthy ✅) → returned
```

It's like simple routing + health checks. Not a replacement for a load balancer, but a simple client-side load distribution.

### Routing Policy Comparison

| Policy | Use Case | Health Check? |
|--------|---------|--------------|
| **Simple** | Single resource | ❌ No |
| **Weighted** | Traffic distribution / A/B testing | ✅ Yes |
| **Latency** | Fastest region | ✅ Yes |
| **Failover** | Active-passive disaster recovery | ✅ Yes (required) |
| **Geolocation** | Route by user location | ✅ Yes |
| **Geoproximity** | Route by location with bias | ✅ Yes |
| **Multivalue** | Simple load distribution | ✅ Yes |

## Health Checks — Is My Server Alive?

Route 53 health checks monitor the health of your resources and can automatically route traffic away from unhealthy ones.

### Types of Health Checks

| Type | What It Checks | Example |
|------|---------------|---------|
| **Endpoint** | Monitors an IP or domain | Check if `54.23.1.100:80` responds |
| **Calculated** | Combines multiple health checks | Healthy if 2 out of 3 checks pass |
| **CloudWatch Alarm** | Based on a CloudWatch alarm | Healthy if CPU < 90% |

### How Endpoint Health Checks Work

```
Route 53 Health Checkers (from multiple locations worldwide)
├── Checker in US-East    → sends request to your server every 30s
├── Checker in US-West    → sends request to your server every 30s
├── Checker in Europe     → sends request to your server every 30s
├── Checker in Asia       → sends request to your server every 30s
└── ... more checkers

If >18% of checkers report unhealthy → Resource marked UNHEALTHY
Route 53 stops routing traffic to it
```

### Health Check Settings

| Setting | Options | Default |
|---------|---------|---------|
| **Protocol** | HTTP, HTTPS, TCP | HTTP |
| **Port** | Any port | 80 |
| **Path** | URL path to check | `/` |
| **Interval** | 10s or 30s | 30s |
| **Failure Threshold** | 1-10 | 3 |
| **String Matching** | Check if response contains specific text | Disabled |
| **Regions** | Which regions to check from | All |

### Creating a Health Check (AWS CLI)

```bash
# Create an HTTP health check
aws route53 create-health-check \
  --caller-reference "health-check-$(date +%s)" \
  --health-check-config '{
    "Type": "HTTP",
    "ResourcePath": "/health",
    "FullyQualifiedDomainName": "myapp.com",
    "Port": 80,
    "RequestInterval": 30,
    "FailureThreshold": 3
  }'

# Create an HTTPS health check with string matching
aws route53 create-health-check \
  --caller-reference "health-check-https-$(date +%s)" \
  --health-check-config '{
    "Type": "HTTP_STR_MATCH",
    "ResourcePath": "/health",
    "FullyQualifiedDomainName": "myapp.com",
    "Port": 443,
    "RequestInterval": 10,
    "FailureThreshold": 2,
    "SearchString": "\"status\":\"ok\""
  }'
```

### Health Checks for Private Resources

Route 53 health checkers are on the public internet — they **can't reach private resources** (like instances in a private subnet). 

**Solution:** Use a **CloudWatch Alarm** health check instead:

```
Private Instance → CloudWatch Agent → CloudWatch Metric → CloudWatch Alarm → Route 53 Health Check
```

```bash
# Create health check based on CloudWatch alarm
aws route53 create-health-check \
  --caller-reference "cw-health-check-$(date +%s)" \
  --health-check-config '{
    "Type": "CLOUDWATCH_METRIC",
    "AlarmIdentifier": {
      "Region": "us-east-1",
      "Name": "my-instance-health-alarm"
    },
    "InsufficientDataHealthStatus": "Unhealthy"
  }'
```

## Domain Registration with Route 53

You can buy domain names directly through Route 53:

```bash
# Check if a domain is available
aws route53domains check-domain-availability \
  --domain-name myawesomeapp.com

# Register a domain (this costs money!)
aws route53domains register-domain \
  --domain-name myawesomeapp.com \
  --duration-in-years 1 \
  --admin-contact '{...}' \
  --registrant-contact '{...}' \
  --tech-contact '{...}'
```

### Using a Domain from Another Registrar

If you bought your domain from GoDaddy, Namecheap, etc., you can still use Route 53 for DNS:

```
Step 1: Create a hosted zone in Route 53 for your domain
Step 2: Route 53 gives you 4 NS (Name Server) records
Step 3: Go to your registrar and update the nameservers to Route 53's NS records
Step 4: DNS queries for your domain now go to Route 53!
```

```
Route 53 gives you:
ns-123.awsdns-45.com
ns-678.awsdns-90.net
ns-234.awsdns-56.org
ns-901.awsdns-12.co.uk

→ Enter these at GoDaddy/Namecheap as your domain's nameservers
```

## Route 53 with Terraform — Complete Examples

### Basic DNS Setup

```hcl
# ─── Hosted Zone ───────────────────────────────────────
resource "aws_route53_zone" "main" {
  name = "myapp.com"

  tags = {
    Environment = "production"
  }
}

# ─── A Record (direct IP) ─────────────────────────────
resource "aws_route53_record" "root" {
  zone_id = aws_route53_zone.main.zone_id
  name    = "myapp.com"
  type    = "A"
  ttl     = 300
  records = ["54.23.1.100"]
}

# ─── CNAME Record (www → root) ────────────────────────
resource "aws_route53_record" "www" {
  zone_id = aws_route53_zone.main.zone_id
  name    = "www.myapp.com"
  type    = "CNAME"
  ttl     = 300
  records = ["myapp.com"]
}

# ─── Alias Record (root → ALB) ────────────────────────
resource "aws_route53_record" "root_alias" {
  zone_id = aws_route53_zone.main.zone_id
  name    = "myapp.com"
  type    = "A"

  alias {
    name                   = aws_lb.main.dns_name         # ALB DNS name
    zone_id                = aws_lb.main.zone_id          # ALB hosted zone ID
    evaluate_target_health = true                          # Use ALB health checks
  }
}

# ─── Alias Record (root → S3 Website) ─────────────────
resource "aws_route53_record" "static_site" {
  zone_id = aws_route53_zone.main.zone_id
  name    = "docs.myapp.com"
  type    = "A"

  alias {
    name                   = aws_s3_bucket_website_configuration.docs.website_domain
    zone_id                = aws_s3_bucket.docs.hosted_zone_id
    evaluate_target_health = false
  }
}

# ─── Alias Record (root → CloudFront) ─────────────────

  type    = "A"

  }
```

## Real-World Architecture — Putting It All Together

Here's how Route 53 fits into a typical production setup:

```
User types: www.myapp.com
       │
       ▼
┌──────────────────┐
│    Route 53       │  DNS Resolution
│   (Hosted Zone)   │  
│                   │  Routing Policy: Latency-Based
│                   │  Health Checks: Enabled
└────────┬─────────┘
         │
    ┌────┴────┐
    ▼         ▼
┌────────┐ ┌────────┐
│us-east-1│ │eu-west-1│
│         │ │         │
│ ┌─────┐ │ │ ┌─────┐ │
│ │ ALB │ │ │ │ ALB │ │
│ └──┬──┘ │ │ └──┬──┘ │
│    │    │ │    │    │
│ ┌──▼──┐ │ │ ┌──▼──┐ │
│ │ EC2 │ │ │ │ EC2 │ │
│ └─────┘ │ │ └─────┘ │
└────────┘ └────────┘

US users → us-east-1 ALB (lower latency)
EU users → eu-west-1 ALB (lower latency)

If us-east-1 ALB goes down:
Route 53 health check detects failure → ALL users go to eu-west-1
```

## Route 53 Pricing

| Component | Cost |
|-----------|------|
| Hosted Zone | $0.50/month per zone |
| Standard queries | $0.40 per million queries |
| Latency-based queries | $0.60 per million queries |
| Geo/Geoproximity queries | $0.70 per million queries |
| Alias queries to AWS resources | **FREE** |
| Health checks (basic) | $0.50/month per check |
| Health checks (with string matching) | $1.00/month per check |
| Health checks (fast interval — 10s) | $1.00/month per check |
| Domain registration | Varies by TLD ($12/year for .com) |

> **Cost tip:** Use **Alias records** wherever possible — they're free for DNS queries to AWS resources!

## Quick Recap — Route 53 Cheat Sheet

| Concept | Key Takeaway |
|---------|-------------|
| **Hosted Zone** | Container for DNS records; public or private |
| **A Record** | Domain → IPv4 address |
| **CNAME** | Domain → another domain (not for root domain!) |
| **Alias** | Like CNAME but for root domain + free + AWS-only |
| **Simple Routing** | Basic DNS, no health checks |
| **Weighted** | Split traffic by percentage (great for canary deploys) |
| **Latency** | Route to fastest region |
| **Failover** | Automatic switch to backup if primary fails |
| **Geolocation** | Route by user's country/continent |
| **Health Checks** | Monitor resources, trigger failover |
| **TTL** | Lower = faster propagation, higher = less cost |

## Common Mistakes to Avoid

1. **Using CNAME for root domain** — Use Alias instead (CNAME on apex domain is not allowed by DNS spec)
2. **Forgetting the default geolocation record** — Users from unmatched locations get NO response
3. **Not setting up health checks with failover** — Without health checks, failover routing won't work
4. **High TTL before a migration** — Lower TTL days before changing DNS records
5. **Not using Alias for AWS resources** — You're paying for DNS queries unnecessarily
6. **Forgetting to update NS records at registrar** — Your hosted zone won't receive queries unless the registrar points to Route 53's nameservers
7. **Single health check region** — Use multiple regions for accurate health checking
