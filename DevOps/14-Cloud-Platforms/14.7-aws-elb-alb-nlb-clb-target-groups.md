# 14.7 — AWS — ELB (ALB, NLB, CLB) & Target Groups

> **One Line Answer:** AWS **Elastic Load Balancer (ELB)** is a fully managed service that **automatically distributes incoming traffic** across multiple targets (EC2 instances, containers, IPs, Lambda functions) — and it comes in three flavors: **Application Load Balancer (ALB)** for HTTP/HTTPS, **Network Load Balancer (NLB)** for ultra-fast TCP/UDP, and the legacy **Classic Load Balancer (CLB)**.

---

## 🤔 What Is Load Balancing and Why Do We Need It?

Let's say you have a website — maybe an e-commerce store. At first, one server handles all the traffic. But what happens when **thousands of users** hit your site at the same time? That single server will **crash**, get **slow**, or become **unresponsive**.

```
Without Load Balancer:
    → All traffic hits ONE server
    → Server gets overwhelmed
    → Website goes DOWN 💀
    → Users leave, money lost

    Users ──────→ [ Single Server ] ──→ 💥 CRASH!
```

Now imagine putting a **load balancer** in front of multiple servers:

```
With Load Balancer:
    → Traffic is DISTRIBUTED across multiple servers
    → No single server is overwhelmed
    → If one server dies, traffic goes to the others
    → Website stays UP and FAST ✅

                         ┌──→ [ Server 1 ] ✅
    Users ──→ [ ELB ] ──┼──→ [ Server 2 ] ✅
                         └──→ [ Server 3 ] ✅
```

**That's load balancing in a nutshell** — spreading the load so no single server breaks under pressure.

---

## 🏗️ What Is AWS Elastic Load Balancer (ELB)?

AWS ELB is Amazon's **managed load balancing service**. You don't need to install, maintain, or scale the load balancer yourself — AWS does it all for you.

### Key Benefits of ELB:

| Benefit | What It Means |
|---------|---------------|
| **Fully Managed** | AWS handles the setup, scaling, and maintenance of the load balancer |
| **High Availability** | ELB automatically spans multiple Availability Zones (AZs) |
| **Auto Scaling** | ELB scales itself based on traffic — no manual work |
| **Health Checks** | ELB continuously checks if your servers are healthy and removes unhealthy ones |
| **Security** | Integrates with Security Groups, ACM (SSL certificates), and WAF |
| **Sticky Sessions** | Can send a user to the same server every time (useful for sessions) |
| **Logging & Monitoring** | Access logs, CloudWatch metrics, request tracing |

AWS gives you **three types** of load balancers. Think of them as different tools for different jobs:

│                                                                     │

## 🟢 1. Application Load Balancer (ALB) — The Smart One

The ALB works at **Layer 7 (Application Layer)** of the OSI model. This means it understands **HTTP and HTTPS** — it can look inside the request (URL path, headers, cookies, query strings) and make smart routing decisions.

### When to Use ALB:

- Web applications
- REST APIs
- Microservices
- WebSocket connections
- gRPC services
- Anything that speaks HTTP/HTTPS

### Key Features of ALB:

| Feature | What It Does |
|---------|--------------|
| **Path-Based Routing** | Route `/api/*` to one group, `/images/*` to another |
| **Host-Based Routing** | Route `api.example.com` to one group, `www.example.com` to another |
| **Header-Based Routing** | Route based on HTTP headers (e.g., `X-Custom-Header: mobile`) |
| **Query String Routing** | Route based on query params (e.g., `?platform=ios`) |
| **Weighted Target Groups** | Send 80% traffic to v1, 20% to v2 (for canary deployments!) |
| **Fixed Response** | Return a custom response without hitting any target |
| **Redirects** | Redirect HTTP to HTTPS, or one URL to another |
| **WebSocket Support** | Full support for WebSocket connections |
| **gRPC Support** | Route gRPC traffic |
| **SSL/TLS Termination** | Handle HTTPS at the load balancer level |
| **Sticky Sessions** | Send a user to the same target using cookies |
| **Authentication** | Built-in integration with Cognito and OIDC providers |

### How ALB Routing Works — Visual:

```
                                 ┌───────────────────────────┐
                                 │     TARGET GROUP 1        │
                         ┌──────→│  (API Servers)            │
                         │       │  EC2-1, EC2-2, EC2-3      │
                         │       └───────────────────────────┘
                         │
    Users ──→ [ ALB ] ───┤  Rules:
                         │  • /api/*     → Target Group 1
                         │  • /images/*  → Target Group 2
                         │  • /*         → Target Group 3 (default)
                         │
                         │       ┌───────────────────────────┐
                         ├──────→│     TARGET GROUP 2        │
                         │       │  (Image Servers / S3)     │
                         │       └───────────────────────────┘
                         │
                         │       ┌───────────────────────────┐
                         └──────→│     TARGET GROUP 3        │
                                 │  (Frontend Servers)       │
                                 └───────────────────────────┘
```

### ALB Listener Rules Example:

When you set up an ALB, you create **listeners** (ports the ALB listens on) and **rules** (conditions to route traffic). Here's how it works:

```
Listener: Port 443 (HTTPS)
│
├─ Rule 1: IF path is /api/*        → Forward to API Target Group
├─ Rule 2: IF path is /admin/*      → Forward to Admin Target Group
├─ Rule 3: IF host is m.example.com → Forward to Mobile Target Group
├─ Rule 4: IF path is /health       → Return fixed response "OK" (200)
└─ Default Rule:                    → Forward to Frontend Target Group
```

### Creating an ALB with AWS CLI:

```bash
# Step 1: Create the Application Load Balancer
aws elbv2 create-load-balancer \
  --name my-web-alb \
  --subnets subnet-0abc1234 subnet-0def5678 \
  --security-groups sg-0abc1234 \
  --scheme internet-facing \
  --type application \
  --ip-address-type ipv4

# Explanation:
# --name             : A friendly name for your load balancer
# --subnets          : At least 2 subnets in different AZs (for high availability)
# --security-groups  : Which security group controls access to this ALB
# --scheme           : "internet-facing" (public) or "internal" (private)
# --type             : "application" for ALB
# --ip-address-type  : "ipv4" or "dualstack" (ipv4 + ipv6)
```

```bash
# Step 2: Create a Target Group
aws elbv2 create-target-group \
  --name my-api-targets \
  --protocol HTTP \
  --port 80 \
  --vpc-id vpc-0abc1234 \
  --target-type instance \
  --health-check-path /health \
  --health-check-interval-seconds 30 \
  --healthy-threshold-count 3 \
  --unhealthy-threshold-count 3

# Explanation:
# --name                         : A friendly name for the target group
# --protocol                     : The protocol used to route traffic to targets
# --port                         : The port on which targets receive traffic
# --vpc-id                       : The VPC where your targets live
# --target-type                  : "instance" (EC2), "ip" (any IP), or "lambda"
# --health-check-path            : The URL path that ALB will hit to check health
# --health-check-interval-seconds: How often (in seconds) to check health
# --healthy-threshold-count      : How many successful checks before marking healthy
# --unhealthy-threshold-count    : How many failed checks before marking unhealthy
```

```bash
# Step 3: Register targets (EC2 instances) with the target group
aws elbv2 register-targets \
  --target-group-arn arn:aws:elasticloadbalancing:us-east-1:123456:targetgroup/my-api-targets/abc123 \
  --targets Id=i-0abc1111 Id=i-0abc2222 Id=i-0abc3333

# This tells the ALB: "Here are the EC2 instances that should receive traffic"
```

```bash
# Step 4: Create a Listener (tells ALB which port to listen on)
aws elbv2 create-listener \
  --load-balancer-arn arn:aws:elasticloadbalancing:us-east-1:123456:loadbalancer/app/my-web-alb/xyz789 \
  --protocol HTTPS \
  --port 443 \
  --certificates CertificateArn=arn:aws:acm:us-east-1:123456:certificate/cert-abc123 \
  --default-actions Type=forward,TargetGroupArn=arn:aws:elasticloadbalancing:us-east-1:123456:targetgroup/my-api-targets/abc123

# Explanation:
# --protocol         : HTTPS (use HTTP for non-encrypted, but always prefer HTTPS)
# --port             : The port the ALB listens on (443 = standard HTTPS)
# --certificates     : The SSL certificate ARN from AWS Certificate Manager (ACM)
# --default-actions  : What to do if no rules match — here, forward to target group
```

```bash
# Step 5: Add a routing rule (path-based routing)
aws elbv2 create-rule \
  --listener-arn arn:aws:elasticloadbalancing:us-east-1:123456:listener/app/my-web-alb/xyz789/listener123 \
  --conditions Field=path-pattern,Values='/api/*' \
  --priority 10 \
  --actions Type=forward,TargetGroupArn=arn:aws:elasticloadbalancing:us-east-1:123456:targetgroup/my-api-targets/abc123

# This says: "If the URL path starts with /api/, send traffic to the API target group"
# --priority : Lower number = evaluated first (1-50000)
```

## 🔵 2. Network Load Balancer (NLB) — The Fast One

The NLB works at **Layer 4 (Transport Layer)** — it routes traffic based on **IP addresses and ports** only. It doesn't look inside the packet (no URL paths, headers, cookies). But in return, it's **insanely fast**.

### When to Use NLB:

- When you need **ultra-low latency** (millions of requests per second)
- TCP/UDP traffic (non-HTTP)
- Gaming servers
- IoT devices
- Financial trading applications
- When you need a **static IP** for your load balancer
- When you need to preserve the **client's source IP**

### Key Features of NLB:

| Feature | What It Does |
|---------|--------------|
| **Ultra-Low Latency** | Handles millions of requests per second with minimal delay |
| **Static IP** | Each NLB gets one static IP per AZ (or you can attach Elastic IPs) |
| **Preserves Source IP** | The target sees the real client IP (not the LB's IP) |
| **TCP, UDP, TLS** | Supports TCP, UDP, and TLS protocols |
| **Long-Lived Connections** | Great for WebSocket and long TCP connections |
| **Cross-Zone LB** | Can distribute traffic evenly across all AZs |
| **PrivateLink** | Expose services to other VPCs via AWS PrivateLink |
| **TLS Termination** | Can handle TLS at the NLB level |
| **Health Checks** | TCP, HTTP, and HTTPS health checks |

### NLB vs ALB — When Traffic Arrives:

```
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  ALB (Layer 7 — Application):                               │
│  "Let me READ this HTTP request... hmm, the path is /api,   │
│   the header says mobile... OK, I'll send it to the API     │
│   target group for mobile clients."                          │
│                                                              │
│  → Smart routing, but takes a tiny bit more time             │
│                                                              │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  NLB (Layer 4 — Transport):                                 │
│  "Traffic on port 443? FORWARD. Done."                       │
│                                                              │
│  → Dumb routing, but BLAZING fast                           │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### Creating an NLB with AWS CLI:

```bash
# Create a Network Load Balancer
aws elbv2 create-load-balancer \
  --name my-tcp-nlb \
  --subnets subnet-0abc1234 subnet-0def5678 \
  --scheme internet-facing \
  --type network \
  --ip-address-type ipv4

# Key difference from ALB:
# --type network           : This makes it an NLB
# No --security-groups     : NLB doesn't use security groups by default
#                            (traffic goes directly to targets, which have their own SGs)
```

```bash
# Create a TCP Target Group for NLB
aws elbv2 create-target-group \
  --name my-tcp-targets \
  --protocol TCP \
  --port 8080 \
  --vpc-id vpc-0abc1234 \
  --target-type instance \
  --health-check-protocol TCP \
  --health-check-port 8080

# Note: NLB target groups use TCP/UDP/TLS protocols, not HTTP
```

```bash
# Create a TCP Listener
aws elbv2 create-listener \
  --load-balancer-arn arn:aws:elasticloadbalancing:us-east-1:123456:loadbalancer/net/my-tcp-nlb/xyz789 \
  --protocol TCP \
  --port 80 \
  --default-actions Type=forward,TargetGroupArn=arn:aws:elasticloadbalancing:us-east-1:123456:targetgroup/my-tcp-targets/abc123
```

### NLB with Static / Elastic IP:

One unique feature of NLB is **static IPs**. Every NLB gets a static IP per AZ automatically. You can also attach your own **Elastic IPs**:

```bash
# Create NLB with Elastic IPs
aws elbv2 create-load-balancer \
  --name my-static-nlb \
  --type network \
  --subnet-mappings \
    SubnetId=subnet-0abc1234,AllocationId=eipalloc-111111 \
    SubnetId=subnet-0def5678,AllocationId=eipalloc-222222

# Now your NLB has FIXED IP addresses that never change.
# This is great when clients need to whitelist your IPs.
```

## 🟡 3. Classic Load Balancer (CLB) — The Old One

The CLB is the **original** AWS load balancer. It works at both Layer 4 and basic Layer 7, but it's **outdated** and lacks modern features.

### Should You Use CLB?

```
Short answer: NO ❌

CLB is legacy. AWS recommends migrating to ALB or NLB.
You'll only see CLB in:
  → Old production systems that haven't been migrated
  → Legacy tutorials and documentation
  → Some certification exam questions
```

### Why CLB Is Outdated:

| CLB Limitation | ALB/NLB Solution |
|----------------|-------------------|
| No path-based routing | ALB has full path/host/header routing |
| No host-based routing | ALB supports multiple domains on one LB |
| One SSL cert per LB | ALB supports SNI (multiple SSL certs) |
| No WebSocket support | ALB has native WebSocket support |
| No HTTP/2 | ALB supports HTTP/2 |
| Limited target types | ALB/NLB support instances, IPs, Lambda |
| No container support | ALB/NLB work great with ECS/EKS |

### CLB Quick Example (just to know the syntax):

```bash
# Create a Classic Load Balancer (legacy — avoid for new projects)
aws elb create-load-balancer \
  --load-balancer-name my-classic-lb \
  --listeners "Protocol=HTTP,LoadBalancerPort=80,InstanceProtocol=HTTP,InstancePort=80" \
  --subnets subnet-0abc1234 \
  --security-groups sg-0abc1234

# Note: CLB uses the "aws elb" command, not "aws elbv2"
# "elbv2" = ALB and NLB (version 2)
# "elb"   = CLB (version 1, legacy)
```

## 🎯 ALB vs NLB vs CLB — The Ultimate Comparison

| Feature | ALB | NLB | CLB |
|---------|-----|-----|-----|
| **OSI Layer** | Layer 7 (Application) | Layer 4 (Transport) | Layer 4 + basic Layer 7 |
| **Protocols** | HTTP, HTTPS, gRPC, WebSocket | TCP, UDP, TLS | TCP, SSL, HTTP, HTTPS |
| **Routing** | Path, host, header, query string | IP + port only | Basic (no path/host routing) |
| **Performance** | Good | Ultra-fast (millions of req/sec) | Moderate |
| **Static IP** | ❌ (use Global Accelerator) | ✅ (one per AZ) | ❌ |
| **Preserves Source IP** | ❌ (via X-Forwarded-For header) | ✅ (natively) | ❌ |
| **SSL Termination** | ✅ | ✅ | ✅ |
| **SNI (Multiple Certs)** | ✅ | ✅ | ❌ (one cert per LB) |
| **Sticky Sessions** | ✅ (cookie-based) | ✅ (connection-based) | ✅ (cookie-based) |
| **WebSocket** | ✅ | ✅ | ❌ |
| **HTTP/2** | ✅ | ❌ (Layer 4) | ❌ |
| **Lambda Targets** | ✅ | ❌ | ❌ |
| **PrivateLink** | ❌ | ✅ | ❌ |
| **Cross-Zone LB** | Always on (free) | Optional (can cost extra) | Optional |
| **Use For** | Web apps, APIs, microservices | Gaming, IoT, extreme perf | Legacy only |

### Quick Decision Guide:

```
Is your traffic HTTP/HTTPS?
  ├─ YES → Use ALB ✅
  └─ NO (TCP/UDP) → Use NLB ✅

Need ultra-low latency?
  └─ YES → Use NLB ✅

Need static IP?
  └─ YES → Use NLB ✅

Need smart routing (path/host/header)?
  └─ YES → Use ALB ✅

Is it an existing old system using CLB?
  └─ Migrate to ALB or NLB when possible
```

## 🎯 Target Groups — Deep Dive

A **Target Group** is a collection of targets (servers, containers, IPs, or Lambda functions) that the load balancer routes traffic to. Think of it as a "bucket" of destinations.

### Target Types:

```
┌─────────────────────────────────────────────────────────────┐
│                    TARGET TYPES                              │
│                                                             │
│  1. instance  → EC2 instances (identified by instance ID)  │
│  2. ip        → Any IP address (even outside AWS!)          │
│  3. lambda    → AWS Lambda functions (ALB only)            │
│  4. alb       → Another ALB (NLB → ALB chaining)          │
└─────────────────────────────────────────────────────────────┘
```

### Why Different Target Types?

```
instance type:
  → Best for EC2 instances
  → ALB routes to the instance, and the instance's SG applies
  → Example: 3 EC2 instances running your web app

ip type:
  → Best for containers (ECS with awsvpc network mode)
  → Best for on-premises servers (via Direct Connect / VPN)
  → You specify the actual IP addresses
  → Example: 10 Docker containers, each with their own IP

lambda type (ALB only):
  → Route HTTP requests directly to a Lambda function
  → Great for serverless APIs behind an ALB
  → Example: ALB → Lambda → return JSON response

alb type (NLB only):
  → Chain NLB in front of ALB
  → Use case: Need NLB's static IP + ALB's Layer 7 routing
  → Example: NLB (static IP) → ALB (path-based routing) → EC2
```

## ❤️ Health Checks — How ELB Knows If a Target Is Healthy

Health checks are **critical**. The load balancer periodically sends a request to each target to check if it's alive. If a target fails the health check, the LB **stops sending traffic** to it.

### Health Check Configuration:

```
┌────────────────────────────────────────────────────────────┐
│                    HEALTH CHECK SETTINGS                    │
│                                                            │
│  Protocol:            HTTP, HTTPS, or TCP                  │
│  Path:                /health (or any endpoint you choose) │
│  Port:                80 (or the traffic port)             │
│  Interval:            30 seconds (how often to check)      │
│  Timeout:             5 seconds (how long to wait)         │
│  Healthy Threshold:   3 (consecutive successes = healthy)  │
│  Unhealthy Threshold: 3 (consecutive failures = unhealthy)│
│  Success Codes:       200 (or 200-299)                     │
└────────────────────────────────────────────────────────────┘
```

### Health Check Flow:

```
ALB sends GET /health to Target every 30 seconds
│
├─ Target responds with 200 OK
│   → Count: 1 success
│   → After 3 consecutive successes → HEALTHY ✅
│   → ALB sends traffic to this target
│
├─ Target responds with 500 or no response
│   → Count: 1 failure
│   → After 3 consecutive failures → UNHEALTHY ❌
│   → ALB STOPS sending traffic to this target
│
└─ Target recovers and responds 200 again
    → After 3 consecutive successes → HEALTHY ✅ again
    → ALB starts sending traffic again
```

### A Simple Health Check Endpoint (Node.js Example):

```javascript
// This is a simple Express.js health check endpoint
// The ALB will hit this URL to check if your app is alive

const express = require('express');
const app = express();

// Health check endpoint — the ALB will call this
app.get('/health', (req, res) => {
  // You can add more checks here:
  // - Is the database connected?
  // - Is Redis reachable?
  // - Are critical services up?

  const isHealthy = true; // Replace with real checks

  if (isHealthy) {
    res.status(200).json({ status: 'healthy' });
  } else {
    res.status(503).json({ status: 'unhealthy' });
  }
});

app.listen(80, () => {
  console.log('Server running on port 80');
});
```

## 🔒 SSL/TLS Termination

**SSL Termination** means the load balancer handles the encryption/decryption of HTTPS traffic, so your backend servers don't have to.

```
Without SSL Termination:
    User ──HTTPS──→ [ ALB ] ──HTTPS──→ [ Server ]
    (Server needs SSL cert, more CPU usage)

With SSL Termination (recommended):
    User ──HTTPS──→ [ ALB ] ──HTTP──→ [ Server ]
    (ALB handles SSL, server only deals with plain HTTP)
    (Less CPU on the server, easier cert management)
```

### How to Set Up SSL with ALB:

1. **Get a certificate** from AWS Certificate Manager (ACM) — it's **free** for use with AWS services!
2. **Attach the certificate** to the ALB listener on port 443
3. **Optional:** Add a redirect rule from HTTP (80) → HTTPS (443)

```bash
# Request a free SSL certificate from ACM
aws acm request-certificate \
  --domain-name example.com \
  --subject-alternative-names "*.example.com" \
  --validation-method DNS

# After DNS validation, attach it to the ALB listener (shown earlier)
```

### HTTP to HTTPS Redirect Rule:

```bash
# Create an HTTP listener that redirects ALL traffic to HTTPS
aws elbv2 create-listener \
  --load-balancer-arn arn:aws:elasticloadbalancing:us-east-1:123456:loadbalancer/app/my-web-alb/xyz789 \
  --protocol HTTP \
  --port 80 \
  --default-actions Type=redirect,RedirectConfig='{Protocol=HTTPS,Port=443,StatusCode=HTTP_301}'

# Now if anyone hits http://example.com, they'll be automatically
# redirected to https://example.com — nice and secure! 🔒
```

## 🧩 Sticky Sessions (Session Affinity)

By default, a load balancer can send each request to a **different server**. But what if your app stores session data in memory (not recommended, but common)?

**Sticky sessions** ensure that a user is always sent to the **same target** for the duration of their session.

### How Sticky Sessions Work:

```
Without Sticky Sessions:
    Request 1 → Server A (login, session created on Server A)
    Request 2 → Server B (no session here! user sees login page again 😢)
    Request 3 → Server C (again, no session! broken experience)

With Sticky Sessions:
    Request 1 → Server A (login, session created, cookie set)
    Request 2 → Server A (cookie tells ALB: "send me to A")
    Request 3 → Server A (same server, session works! ✅)
```

### ALB Sticky Session Types:

| Type | How It Works |
|------|-------------|
| **Duration-Based** | ALB generates a cookie (`AWSALB`) with an expiry time you set |
| **Application-Based** | Your app generates its own cookie, ALB uses it for stickiness |

```bash
# Enable duration-based sticky sessions on a target group
aws elbv2 modify-target-group-attributes \
  --target-group-arn arn:aws:elasticloadbalancing:us-east-1:123456:targetgroup/my-api-targets/abc123 \
  --attributes \
    Key=stickiness.enabled,Value=true \
    Key=stickiness.type,Value=lb_cookie \
    Key=stickiness.lb_cookie.duration_seconds,Value=3600

# This means: once a user hits a target, they'll stay on that target for 3600 seconds (1 hour)
```

> **Pro Tip:** The best practice is to store sessions in a shared store (Redis, DynamoDB) so you don't need sticky sessions at all. This way, any server can handle any request.

## 🌐 Cross-Zone Load Balancing

By default, each AZ's load balancer node only sends traffic to targets **in its own AZ**. This can cause **uneven distribution** if you have different numbers of targets in each AZ.

```
Without Cross-Zone LB:
    AZ-a has 2 instances, AZ-b has 8 instances
    Each AZ gets 50% of traffic

    AZ-a: 50% traffic ÷ 2 instances = 25% per instance 😰 (overloaded!)
    AZ-b: 50% traffic ÷ 8 instances = 6.25% per instance 😴 (underutilized)

With Cross-Zone LB:
    All 10 instances share traffic equally
    Each instance gets 10% of traffic ✅ (balanced!)
```

| LB Type | Cross-Zone Default | Cost |
|---------|-------------------|------|
| ALB | Always enabled | Free |
| NLB | Disabled by default | May incur data transfer charges |
| CLB | Disabled by default | Free |

## 📊 Connection Draining (Deregistration Delay)

When you remove a target from the target group (for deployment, scaling down, etc.), you don't want to **kill active connections** immediately. **Connection draining** gives the target time to finish ongoing requests.

```
Without Connection Draining:
    Target removed → Active requests KILLED instantly → Users see errors 😡

With Connection Draining (default: 300 seconds):
    Target removed → Target marked "draining"
    → New requests go to OTHER targets
    → Existing requests have 300 seconds to complete
    → After 300s (or when all requests finish) → Target fully removed ✅
```

```bash
# Set deregistration delay to 120 seconds
aws elbv2 modify-target-group-attributes \
  --target-group-arn arn:aws:elasticloadbalancing:us-east-1:123456:targetgroup/my-api-targets/abc123 \
  --attributes Key=deregistration_delay.timeout_seconds,Value=120
```

## 🏗️ ALB with Terraform — Real-World Example

Here's a complete, production-ready ALB setup using Terraform:

```hcl
# =========================================
# ALB — Application Load Balancer
# =========================================

# Create the ALB itself
resource "aws_lb" "web_alb" {
  name               = "my-web-alb"
  internal           = false            # false = internet-facing, true = internal
  load_balancer_type = "application"    # "application" for ALB
  security_groups    = [aws_security_group.alb_sg.id]
  subnets            = [                # At least 2 subnets in different AZs
    aws_subnet.public_a.id,
    aws_subnet.public_b.id
  ]

  enable_deletion_protection = true     # Prevent accidental deletion

  tags = {
    Name        = "my-web-alb"
    Environment = "production"
  }
}

# =========================================
# Security Group for ALB
# =========================================
resource "aws_security_group" "alb_sg" {
  name        = "alb-security-group"
  description = "Allow HTTP and HTTPS traffic to ALB"
  vpc_id      = aws_vpc.main.id

  # Allow HTTPS from anywhere
  ingress {
    from_port   = 443
    to_port     = 443
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  # Allow HTTP from anywhere (will redirect to HTTPS)
  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  # Allow all outbound traffic
  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

# =========================================
# Target Group — Where traffic goes
# =========================================
resource "aws_lb_target_group" "api_tg" {
  name        = "api-target-group"
  port        = 80
  protocol    = "HTTP"
  vpc_id      = aws_vpc.main.id
  target_type = "instance"

  # Health check configuration
  health_check {
    enabled             = true
    path                = "/health"        # Your app's health endpoint
    port                = "traffic-port"   # Use the same port as traffic
    protocol            = "HTTP"
    healthy_threshold   = 3                # 3 successful checks = healthy
    unhealthy_threshold = 3                # 3 failed checks = unhealthy
    timeout             = 5                # Wait 5 seconds for response
    interval            = 30               # Check every 30 seconds
    matcher             = "200"            # Expect HTTP 200 response
  }

  # Connection draining
  deregistration_delay = 120               # 120 seconds to drain connections

  tags = {
    Name = "api-target-group"
  }
}

# =========================================
# HTTPS Listener (Port 443)
# =========================================
resource "aws_lb_listener" "https" {
  load_balancer_arn = aws_lb.web_alb.arn
  port              = 443
  protocol          = "HTTPS"
  ssl_policy        = "ELBSecurityPolicy-TLS13-1-2-2021-06"  # Modern TLS policy
  certificate_arn   = aws_acm_certificate.main.arn             # Your SSL cert

  default_action {
    type             = "forward"
    target_group_arn = aws_lb_target_group.api_tg.arn
  }
}

# =========================================
# HTTP Listener (Port 80) — Redirect to HTTPS
# =========================================
resource "aws_lb_listener" "http_redirect" {
  load_balancer_arn = aws_lb.web_alb.arn
  port              = 80
  protocol          = "HTTP"

  default_action {
    type = "redirect"
    redirect {
      port        = "443"
      protocol    = "HTTPS"
      status_code = "HTTP_301"    # Permanent redirect
    }
  }
}

# =========================================
# Listener Rule — Path-based routing
# =========================================
resource "aws_lb_listener_rule" "api_rule" {
  listener_arn = aws_lb_listener.https.arn
  priority     = 100

  action {
    type             = "forward"
    target_group_arn = aws_lb_target_group.api_tg.arn
  }

  condition {
    path_pattern {
      values = ["/api/*"]    # Route /api/* to API target group
    }
  }
}

# =========================================
# Register EC2 instances with Target Group
# =========================================
resource "aws_lb_target_group_attachment" "target1" {
  target_group_arn = aws_lb_target_group.api_tg.arn
  target_id        = aws_instance.web_server_1.id
  port             = 80
}

resource "aws_lb_target_group_attachment" "target2" {
  target_group_arn = aws_lb_target_group.api_tg.arn
  target_id        = aws_instance.web_server_2.id
  port             = 80
}
```

## 📊 ELB Access Logs

ELB can write detailed **access logs** to an S3 bucket. This is extremely useful for debugging, security analysis, and compliance.

### What's in the Access Log:

```
Each log entry contains:
  → Timestamp
  → Client IP and port
  → Target IP and port
  → Request processing time
  → Target processing time
  → Response processing time
  → ELB status code
  → Target status code
  → Bytes received and sent
  → HTTP method, URL, protocol
  → User agent
  → SSL cipher and protocol
  → Trace ID
```

### Enable Access Logs:

```bash
aws elbv2 modify-load-balancer-attributes \
  --load-balancer-arn arn:aws:elasticloadbalancing:us-east-1:123456:loadbalancer/app/my-web-alb/xyz789 \
  --attributes \
    Key=access_logs.s3.enabled,Value=true \
    Key=access_logs.s3.bucket,Value=my-alb-logs-bucket \
    Key=access_logs.s3.prefix,Value=alb-logs
```

## 🧠 Important Concepts for Interviews & Exams

### 1. SNI (Server Name Indication)

ALB and NLB support **SNI**, which means you can host **multiple SSL certificates** on a single load balancer. Each certificate is for a different domain.

```
Without SNI (CLB):
    1 Load Balancer = 1 SSL Certificate = 1 Domain
    Want 5 domains? Need 5 Load Balancers! 💸

With SNI (ALB/NLB):
    1 Load Balancer = Multiple SSL Certificates = Multiple Domains
    api.example.com    → cert-1
    www.example.com    → cert-2
    admin.example.com  → cert-3
    All on ONE ALB! ✅
```

### 2. X-Forwarded Headers (ALB)

Since ALB terminates the connection, the backend server sees the ALB's IP, not the client's IP. ALB adds special headers:

| Header | Contains |
|--------|----------|
| `X-Forwarded-For` | Client's real IP address |
| `X-Forwarded-Proto` | Protocol used (HTTP or HTTPS) |
| `X-Forwarded-Port` | Port used by the client |

```javascript
// In your backend code, get the real client IP:
const clientIP = req.headers['x-forwarded-for'] || req.connection.remoteAddress;
console.log('Real client IP:', clientIP);
```

### 3. Idle Timeout

The load balancer has an **idle timeout** (default: 60 seconds for ALB). If no data is sent over the connection for this duration, the LB closes it.

```bash
# Change idle timeout to 120 seconds
aws elbv2 modify-load-balancer-attributes \
  --load-balancer-arn arn:aws:elasticloadbalancing:us-east-1:123456:loadbalancer/app/my-web-alb/xyz789 \
  --attributes Key=idle_timeout.timeout_seconds,Value=120
```

> **Tip:** Make sure your app's keep-alive timeout is **longer** than the ALB's idle timeout to avoid connection drops.

```
┌───────────────────────────────────────────────────────────────┐
│                    ELB CHEAT SHEET                             │
│                                                               │
│  ALB = HTTP/HTTPS (Layer 7) = Smart routing = Web apps       │
│  NLB = TCP/UDP (Layer 4)    = Ultra-fast    = Gaming/IoT     │
│  CLB = Legacy               = Don't use     = Migrate away   │
│                                                               │
│  Target Group = Bucket of targets (instances, IPs, Lambda)   │
│  Listener     = Port + protocol the LB listens on            │
│  Rule         = Condition + action (route /api/* to group X) │
│  Health Check = "Are you alive?" check on targets            │
│                                                               │
│  ALB: Path routing, host routing, header routing, redirects  │
│  NLB: Static IP, source IP preservation, ultra-low latency   │
│                                                               │
│  SSL: Use ACM (free!) + attach to ALB/NLB listener           │
│  Sticky Sessions: Cookie-based (ALB) or connection (NLB)     │
│  Cross-Zone: ALB (always on), NLB (optional), CLB (optional) │
│  Connection Draining: Default 300s, customize as needed       │
│                                                               │
│  CLI: aws elbv2 (ALB/NLB) vs aws elb (CLB/legacy)           │
└───────────────────────────────────────────────────────────────┘
```
