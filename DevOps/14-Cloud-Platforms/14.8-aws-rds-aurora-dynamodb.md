## 🤔 Why Managed Databases? Why Not Just Install MySQL on EC2?

Great question! You *could* install MySQL on an EC2 instance. But then YOU are responsible for **everything**:

```
Self-Managed Database (MySQL on EC2):
    → YOU install and configure the database
    → YOU handle patches and updates
    → YOU set up backups (and pray they work)
    → YOU configure replication for high availability
    → YOU monitor performance
    → YOU handle failover when things break at 3 AM 😴
    → YOU scale the storage when it runs out
    → YOU manage security patches

Managed Database (RDS/Aurora/DynamoDB):
    → AWS does ALL of the above for you ✅
    → You just use the database
    → Focus on your application, not database administration
    → Sleep peacefully at night 😴💤
```

### Analogy:

```
Self-Managed = Cooking at home
    You buy ingredients, cook, clean, maintain the kitchen

Managed Database = Eating at a restaurant
    You just order food and eat. Chef handles everything else.

You're paying more, but saving TONS of time and avoiding headaches.
```

# 📘 Part 1: Amazon RDS (Relational Database Service)

## What Is RDS?

Amazon RDS is a **managed relational database service**. It takes popular database engines (MySQL, PostgreSQL, etc.) and runs them for you, handling all the boring admin stuff.

### Supported Database Engines:

| Engine | Description |
|--------|-------------|
| **MySQL** | Most popular open-source relational DB |
| **PostgreSQL** | Advanced open-source DB with powerful features |
| **MariaDB** | MySQL fork, community-driven |
| **Oracle** | Enterprise database (license required) |
| **SQL Server** | Microsoft's enterprise database |
| **Aurora** | AWS's own engine (covered separately below) |

## 🔑 RDS Key Features

### 1. Multi-AZ Deployment (High Availability)

This is how RDS stays **highly available**. AWS creates a **standby replica** in a different Availability Zone. If the primary fails, AWS **automatically** switches to the standby.

```
Normal Operation:
    App ──→ [ Primary DB (AZ-a) ]
                    │
                    │ synchronous replication
                    ▼
            [ Standby DB (AZ-b) ]  ← exact copy, always in sync

When Primary Fails:
    App ──→ [ Primary DB (AZ-a) ] 💥 DOWN!
                                        
    AWS automatically switches DNS endpoint...
                    
    App ──→ [ Standby DB (AZ-b) ] ✅ Now becomes the new primary!
    
    Failover takes about 60-120 seconds. No manual intervention needed!
```

**Key Points about Multi-AZ:**
- Synchronous replication (data is written to both at the same time)
- Automatic failover (no manual work)
- Same endpoint — your app doesn't need to change anything
- The standby is NOT used for reads (it's purely for failover)
- Costs roughly 2x (you're running two instances)

### 2. Read Replicas (Scalability)

Read Replicas solve a different problem — **too many reads**. If your database is overwhelmed by read queries (SELECT statements), you can create Read Replicas to handle the read traffic.

```
Without Read Replicas:
    App → [ Primary DB ] ← handles ALL reads AND writes
                           (gets overloaded with reads)

With Read Replicas:
    Writes ──→ [ Primary DB ]
                    │
                    │ asynchronous replication
                    ├──→ [ Read Replica 1 ] ← reads go here
                    ├──→ [ Read Replica 2 ] ← reads go here
                    └──→ [ Read Replica 3 ] ← reads go here
    
    Reads  ──→ [ Any Read Replica ]

    Now the primary only handles writes!
    Reads are spread across replicas! ✅
```

**Key Points about Read Replicas:**
- **Asynchronous** replication (slight delay — milliseconds to seconds)
- Can create up to **5 Read Replicas**
- Read Replicas can be in the **same AZ, different AZ, or different Region**
- Can be **promoted** to a standalone database
- You connect to Read Replicas using a **separate endpoint**
- Great for read-heavy workloads (reporting, analytics, dashboards)

### 3. Automated Backups

```
RDS Automated Backups:
    → Daily full backup during your chosen backup window
    → Transaction logs backed up every 5 minutes
    → Retained for 1-35 days (you choose)
    → Point-in-time restore: restore to any second within retention period
    → Backups stored in S3 (managed by AWS, you don't see the bucket)
```

### 4. Manual Snapshots

```
Manual Snapshots:
    → You trigger them manually (or via script/Lambda)
    → Retained FOREVER until you delete them
    → Can be shared with other AWS accounts
    → Can be copied to other regions (for disaster recovery)
    → Great for: before major changes, before upgrades, archival
```

### 5. Encryption

```
At-Rest Encryption:
    → Uses AWS KMS (Key Management Service)
    → Encrypts data, backups, snapshots, and read replicas
    → Must be enabled at creation time (can't encrypt an existing unencrypted DB)
    → Workaround: Take snapshot → Copy snapshot with encryption → Restore

In-Transit Encryption:
    → Use SSL/TLS connections
    → Download the RDS CA certificate
    → Configure your app to connect with SSL
```

## 💻 RDS — Hands-On with AWS CLI

### Create an RDS Instance:

```bash
# Create a MySQL RDS instance
aws rds create-db-instance \
  --db-instance-identifier my-mysql-db \
  --db-instance-class db.t3.medium \
  --engine mysql \
  --engine-version 8.0 \
  --master-username admin \
  --master-user-password 'MySecurePassword123!' \
  --allocated-storage 100 \
  --storage-type gp3 \
  --vpc-security-group-ids sg-0abc1234 \
  --db-subnet-group-name my-db-subnet-group \
  --multi-az \
  --backup-retention-period 7 \
  --storage-encrypted \
  --kms-key-id alias/aws/rds \
  --no-publicly-accessible

# Let's break down each parameter:
#
# --db-instance-identifier : Unique name for your DB instance
# --db-instance-class      : The "size" of your DB (CPU, RAM)
#                             db.t3.medium = 2 vCPU, 4 GB RAM (good for dev/small prod)
#                             db.r6g.xlarge = 4 vCPU, 32 GB RAM (good for production)
# --engine                 : Which database engine (mysql, postgres, mariadb, etc.)
# --engine-version         : Specific version of the engine
# --master-username        : The admin username
# --master-user-password   : The admin password (use Secrets Manager in production!)
# --allocated-storage      : Storage size in GB
# --storage-type           : gp3 (general SSD), io1/io2 (high-performance SSD)
# --vpc-security-group-ids : Security group controlling who can connect
# --db-subnet-group-name   : Which subnets the DB can be placed in
# --multi-az               : Enable Multi-AZ for high availability
# --backup-retention-period: Keep automated backups for 7 days
# --storage-encrypted      : Encrypt data at rest
# --no-publicly-accessible : Database is NOT accessible from the internet (good!)
```

### Create a Read Replica:

```bash
# Create a Read Replica of your existing DB
aws rds create-db-instance-read-replica \
  --db-instance-identifier my-mysql-db-read-1 \
  --source-db-instance-identifier my-mysql-db \
  --db-instance-class db.t3.medium \
  --availability-zone us-east-1b

# This creates a copy of your DB that handles read queries.
# The replica automatically syncs with the primary.
```

### Create a Manual Snapshot:

```bash
# Take a manual snapshot
aws rds create-db-snapshot \
  --db-instance-identifier my-mysql-db \
  --db-snapshot-identifier my-mysql-db-snapshot-2024-01-15

# Restore from snapshot
aws rds restore-db-instance-from-db-snapshot \
  --db-instance-identifier my-mysql-db-restored \
  --db-snapshot-identifier my-mysql-db-snapshot-2024-01-15 \
  --db-instance-class db.t3.medium
```

### Connect to RDS from Your Application:

```python
# Python example — connecting to RDS MySQL
import mysql.connector

# Connection details
connection = mysql.connector.connect(
    host="my-mysql-db.abc123.us-east-1.rds.amazonaws.com",  # RDS endpoint
    port=3306,                                                # MySQL default port
    user="admin",                                             # Master username
    password="MySecurePassword123!",                          # Use Secrets Manager!
    database="myapp",                                         # Database name
    ssl_ca="rds-combined-ca-bundle.pem"                       # SSL certificate
)

cursor = connection.cursor()
cursor.execute("SELECT * FROM users LIMIT 10")
results = cursor.fetchall()

for row in results:
    print(row)

cursor.close()
connection.close()
```

# 📗 Part 2: Amazon Aurora

## What Is Aurora?

Aurora is Amazon's **own relational database engine**. It's **compatible** with MySQL and PostgreSQL, meaning you can use the same SQL queries, tools, and drivers — but under the hood, Aurora is **completely re-engineered** for the cloud.

### Why Aurora Over Standard RDS?

```
Standard RDS MySQL:
    → Good, reliable, managed
    → Uses EBS storage (single AZ storage)
    → Manual scaling
    → Standard MySQL performance

Aurora MySQL:
    → 5x FASTER than standard MySQL ⚡
    → Storage automatically grows (10 GB → 128 TB, no downtime)
    → Data replicated 6 ways across 3 AZs (super durable)
    → Up to 15 Read Replicas (RDS only allows 5)
    → Sub-10ms replica lag (vs seconds for RDS)
    → Automatic failover in <30 seconds
    → Same price as RDS or cheaper at scale
```

---

## 🏗️ Aurora Architecture — The Magic Behind It

Aurora's architecture is fundamentally different from standard RDS. The **compute** and **storage** are **separated**.

```
┌────────────────────────────────────────────────────────────────┐
│                    AURORA ARCHITECTURE                          │
│                                                                │
│  ┌──────────────────────────────────────────┐                  │
│  │           COMPUTE LAYER                   │                 │
│  │                                           │                 │
│  │  ┌──────────┐  ┌──────────┐  ┌─────────┐│                 │
│  │  │ Primary  │  │ Replica  │  │ Replica ││                 │
│  │  │ (Writer) │  │ 1 (Read) │  │ 2 (Read)││                 │
│  │  │  AZ-a    │  │  AZ-b    │  │  AZ-c   ││                 │
│  │  └────┬─────┘  └────┬─────┘  └───┬─────┘│                 │
│  └───────┼──────────────┼────────────┼──────┘                 │
│          │              │            │                          │
│          ▼              ▼            ▼                          │
│  ┌────────────────────────────────────────────┐                │
│  │          SHARED STORAGE LAYER               │               │
│  │     (Distributed across 3 AZs)             │               │
│  │                                             │               │
│  │  ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ │               │
│  │  │Copy │ │Copy │ │Copy │ │Copy │ │Copy │  │               │
│  │  │  1  │ │  2  │ │  3  │ │  4  │ │  5  │  │               │
│  │  │AZ-a │ │AZ-a │ │AZ-b │ │AZ-b │ │AZ-c │  │               │
│  │  └─────┘ └─────┘ └─────┘ └─────┘ └─────┘  │               │
│  │  ┌─────┐                                    │               │
│  │  │Copy │  ← 6 copies across 3 AZs!        │               │
│  │  │  6  │     Can lose 2 copies and still   │               │
│  │  │AZ-c │     write. Lose 3 and still read! │               │
│  │  └─────┘                                    │               │
│  │                                             │               │
│  │  Auto-grows: 10 GB → 128 TB                │               │
│  │  No downtime for storage scaling!           │               │
│  └─────────────────────────────────────────────┘               │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### Why This Architecture Is Amazing:

| Feature | How Aurora Does It |
|---------|-------------------|
| **6 copies of data** | Your data is stored 6 times across 3 AZs |
| **Self-healing** | If a copy is corrupted, Aurora automatically repairs it |
| **Write tolerance** | Can lose 2 out of 6 copies and still write |
| **Read tolerance** | Can lose 3 out of 6 copies and still read |
| **Auto-growing storage** | Starts at 10 GB, grows automatically up to 128 TB |
| **No storage provisioning** | You never have to say "I need 500 GB" — it just grows |

## 🔑 Aurora Key Features

### Aurora Endpoints:

Aurora gives you **different endpoints** for different purposes:

```
┌─────────────────────────────────────────────────────────────┐
│                    AURORA ENDPOINTS                          │
│                                                             │
│  1. Cluster Endpoint (Writer Endpoint)                     │
│     → Always points to the PRIMARY instance                │
│     → Use for: INSERT, UPDATE, DELETE (writes)             │
│     → Example: my-cluster.cluster-abc123.us-east-1.        │
│                rds.amazonaws.com                            │
│                                                             │
│  2. Reader Endpoint                                        │
│     → Load-balances across ALL Read Replicas               │
│     → Use for: SELECT queries (reads)                      │
│     → Example: my-cluster.cluster-ro-abc123.us-east-1.     │
│                rds.amazonaws.com                            │
│                                                             │
│  3. Custom Endpoints                                       │
│     → You define which instances are in the group           │
│     → Use for: Sending analytics queries to specific       │
│       larger instances, isolating workloads                 │
│                                                             │
│  4. Instance Endpoint                                      │
│     → Direct connection to a specific instance             │
│     → Use for: Debugging, testing a specific replica       │
└─────────────────────────────────────────────────────────────┘
```

### Aurora Serverless

**Aurora Serverless** is the pay-per-use version of Aurora. Instead of choosing an instance size, you define a **capacity range** and Aurora **auto-scales** based on your actual workload.

```
Traditional Aurora:
    → You pick db.r6g.large (2 vCPU, 16 GB RAM)
    → You pay for it 24/7 whether it's busy or idle
    → Need more? Manually change instance class

Aurora Serverless v2:
    → You set a range: min 0.5 ACU to max 16 ACU
    → Aurora automatically scales UP when busy
    → Aurora automatically scales DOWN when idle
    → You pay only for what you use (per ACU-hour)
    → No manual intervention needed!
    
    ACU = Aurora Capacity Unit (roughly 2 GB RAM + proportional CPU)
```

**When to use Aurora Serverless:**
- Development and testing environments
- Infrequent or unpredictable workloads
- New applications where you don't know the traffic pattern
- Applications with spiky traffic (idle at night, busy during day)

### Aurora Global Database

For applications that need to serve users across **multiple regions** with low-latency reads:

```
┌────────────────────────────────────────────────────────────┐
│               AURORA GLOBAL DATABASE                       │
│                                                            │
│  US-EAST-1 (Primary Region)                               │
│  ┌────────────────────┐                                    │
│  │  Primary Cluster   │ ← All writes go here              │
│  │  (Writer + Readers)│                                    │
│  └────────┬───────────┘                                    │
│           │ replication (<1 second lag)                     │
│           ▼                                                │
│  EU-WEST-1 (Secondary Region)                              │
│  ┌────────────────────┐                                    │
│  │ Secondary Cluster  │ ← Local reads for European users  │
│  │  (Readers only)    │   Can be promoted if primary fails │
│  └────────────────────┘                                    │
│           │ replication                                     │
│           ▼                                                │
│  AP-SOUTHEAST-1 (Secondary Region)                         │
│  ┌────────────────────┐                                    │
│  │ Secondary Cluster  │ ← Local reads for Asian users     │
│  │  (Readers only)    │                                    │
│  └────────────────────┘                                    │
│                                                            │
│  Cross-region replication: typically <1 second!            │

```

## 🔄 RDS vs Aurora — Side-by-Side Comparison

| Feature | RDS | Aurora |
|---------|-----|--------|
| **Engines** | MySQL, PostgreSQL, MariaDB, Oracle, SQL Server | MySQL-compatible, PostgreSQL-compatible |
| **Performance** | Standard | 5x faster (MySQL), 3x faster (PostgreSQL) |
| **Storage** | EBS-based (up to 64 TB) | Shared, distributed (up to 128 TB) |
| **Data Copies** | 2 (with Multi-AZ) | 6 copies across 3 AZs |
| **Read Replicas** | Up to 5 | Up to 15 |
| **Replica Lag** | Seconds | Milliseconds (<10ms) |
| **Failover Time** | 60-120 seconds | <30 seconds |
| **Storage Scaling** | Manual (resize EBS) | Automatic (10 GB → 128 TB) |
| **Backtrack** | ❌ | ✅ (rewind DB to any point in time) |
| **Serverless** | ❌ | ✅ (Aurora Serverless v2) |
| **Global DB** | Cross-region Read Replicas | Aurora Global Database (<1s lag) |
| **Cost** | Lower starting cost | Higher, but better value at scale |

### When to Choose What:

```
Choose RDS when:
    → You need Oracle or SQL Server (Aurora doesn't support them)
    → You want the lowest possible cost for small workloads
    → You need exact version compatibility with existing systems
    → Simple, small-to-medium databases

Choose Aurora when:
    → You need maximum performance
    → You need high availability (6 copies, fast failover)
    → You have read-heavy workloads (15 replicas)
    → You need auto-scaling storage
    → You need global read access (Aurora Global Database)
    → You want serverless database (Aurora Serverless v2)
```

# 📙 Part 3: Amazon DynamoDB

## What Is DynamoDB?

DynamoDB is a **fully managed NoSQL database** from AWS. Unlike RDS and Aurora (which use SQL and tables with rows/columns), DynamoDB stores data as **key-value pairs** and **documents** (JSON-like).

### SQL vs NoSQL — Quick Refresher:

```
SQL (Relational — RDS, Aurora):
    → Data stored in tables with fixed columns
    → Relationships between tables (foreign keys, JOINs)
    → Schema is rigid (must define columns upfront)
    → Great for: complex queries, transactions, relationships
    → Example: Users table, Orders table, joined by user_id

NoSQL (DynamoDB):
    → Data stored as key-value pairs or documents
    → No fixed schema (each item can have different attributes)
    → No JOINs (denormalized data)
    → Great for: simple lookups, massive scale, flexible data
    → Example: User profile stored as a single JSON document
```

### Why DynamoDB?

```
Traditional Database at Scale:

    → 100 million?   Database on fire 🔥 Need a whole team of DBAs

DynamoDB:
    → 100 users?       Works great.
    → 10,000 users?    Works great.
    → 1 million?       Works great.
    → 100 million?     STILL works great. Same single-digit millisecond latency.
    → 1 BILLION items? Still fast. DynamoDB was BUILT for this. 🚀
```

## 🏗️ DynamoDB Architecture & Concepts

### Core Concepts:

```
┌──────────────────────────────────────────────────────────────┐
│                  DYNAMODB CONCEPTS                            │
│                                                              │
│  Table     = A collection of items (like a SQL table)       │
│  Item      = A single record (like a SQL row)               │
│  Attribute = A field in an item (like a SQL column)         │
│                                                              │
│  BUT unlike SQL:                                            │
│  → Items don't need the same attributes!                    │
│  → No schema enforced (schemaless)                          │
│  → Each item is basically a JSON document                   │
│                                                              │
│  Example Table: "Users"                                     │
│  ┌────────────────────────────────────────────────────────┐ │
│  │ Item 1: { userId: "U001", name: "Alice", age: 30 }    │ │
│  │ Item 2: { userId: "U002", name: "Bob", email: "..." } │ │
│  │ Item 3: { userId: "U003", name: "Charlie", score: 95 }│ │
│  └────────────────────────────────────────────────────────┘ │
│  Notice: Item 1 has "age", Item 2 has "email",             │
│          Item 3 has "score". All different! That's OK!      │
└──────────────────────────────────────────────────────────────┘
```

### Primary Key — The Most Important Concept:

Every DynamoDB table must have a **Primary Key**. It's how DynamoDB uniquely identifies and retrieves items. There are two types:

```
1. PARTITION KEY (Simple Primary Key):
   → A single attribute that uniquely identifies an item
   → DynamoDB uses it to determine WHERE to store the item

   Example: userId = "U001"
   
   ┌───────────────────────────────────┐
   │ Partition Key (userId) │ Data     │
   │───────────────────────│──────────│
   │ U001                   │ Alice... │
   │ U002                   │ Bob...   │
   │ U003                   │ Charlie..│
   └───────────────────────────────────┘

2. PARTITION KEY + SORT KEY (Composite Primary Key):
   → Two attributes together uniquely identify an item
   → Partition Key groups related items
   → Sort Key orders items within a partition
   
   Example: userId = "U001", timestamp = "2024-01-15T10:30:00"
   
   ┌──────────────────────────────────────────────────┐
   │ Partition Key  │ Sort Key        │ Data           │
   │ (userId)       │ (timestamp)     │                │
   │────────────────│─────────────────│────────────────│
   │ U001           │ 2024-01-15 10:30│ Ordered Pizza  │
   │ U001           │ 2024-01-15 14:00│ Ordered Coffee │
   │ U001           │ 2024-01-16 09:00│ Ordered Burger │
   │ U002           │ 2024-01-15 11:00│ Ordered Sushi  │
   └──────────────────────────────────────────────────┘
   
   Now you can:
   → Get ALL orders for U001 (query by Partition Key)
   → Get orders for U001 on a specific date (query by Partition + Sort range)
   → Get the LATEST order for U001 (query with ScanIndexForward=false)
```

### Secondary Indexes — Query Flexibility:

```
Problem: You defined userId as the Partition Key.
         But now you want to query by email. You can't!
         DynamoDB only allows queries by Primary Key.

Solution: Create a Secondary Index!

1. Global Secondary Index (GSI):
   → Completely different Partition Key and Sort Key
   → Can query across ALL partitions
   → Has its own provisioned capacity
   → Example: GSI with email as Partition Key
   
2. Local Secondary Index (LSI):
   → Same Partition Key, but different Sort Key
   → Must be created at table creation time
   → Example: Same userId partition, but sorted by orderTotal instead of timestamp
```

## 📊 DynamoDB Capacity Modes

DynamoDB gives you two ways to handle capacity:

```
┌──────────────────────────────────────────────────────────────┐
│              CAPACITY MODES                                   │
│                                                              │
│  1. ON-DEMAND MODE (Pay-per-request)                        │
│     → You pay for each read and write                       │
│     → DynamoDB auto-scales instantly                        │
│     → No capacity planning needed                           │
│     → Best for: unpredictable traffic, new apps             │
│     → More expensive per request, but no waste              │
│                                                              │
│  2. PROVISIONED MODE (with Auto Scaling)                    │
│     → You specify: "I need 100 reads/sec and 50 writes/sec"│
│     → DynamoDB reserves that capacity                       │
│     → Auto Scaling can adjust within min/max limits         │
│     → Best for: predictable traffic, cost optimization      │
│     → Cheaper per request, but risk of throttling           │
│                                                              │
│  Units:                                                      │
│  → RCU (Read Capacity Unit):  1 strongly consistent read    │
│    of up to 4 KB per second                                  │
│  → WCU (Write Capacity Unit): 1 write of up to 1 KB/second │
└──────────────────────────────────────────────────────────────┘
```

### RCU/WCU Calculation Examples:

```
Read Example:
    Item size: 8 KB
    Reads per second: 100
    Consistency: Strongly consistent

    RCUs needed = (Item Size / 4 KB) × Reads per second
    RCUs needed = (8 / 4) × 100 = 200 RCUs

    If Eventually Consistent (default): 200 / 2 = 100 RCUs
    (Eventually consistent reads cost HALF the RCUs!)

Write Example:
    Item size: 3 KB
    Writes per second: 50

    WCUs needed = ceil(Item Size / 1 KB) × Writes per second
    WCUs needed = 3 × 50 = 150 WCUs
```

## 💻 DynamoDB — Hands-On with AWS CLI

### Create a DynamoDB Table:

```bash
# Create a simple table with just a Partition Key
aws dynamodb create-table \
  --table-name Users \
  --attribute-definitions \
    AttributeName=userId,AttributeType=S \
  --key-schema \
    AttributeName=userId,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST

# Explanation:
# --table-name           : Name of the table
# --attribute-definitions: Define the attributes used in keys
#                          S = String, N = Number, B = Binary
# --key-schema           : Define the primary key
#                          HASH = Partition Key, RANGE = Sort Key
# --billing-mode         : PAY_PER_REQUEST (on-demand) or PROVISIONED
```

```bash
# Create a table with Partition Key + Sort Key (composite key)
aws dynamodb create-table \
  --table-name Orders \
  --attribute-definitions \
    AttributeName=userId,AttributeType=S \
    AttributeName=orderDate,AttributeType=S \
  --key-schema \
    AttributeName=userId,KeyType=HASH \
    AttributeName=orderDate,KeyType=RANGE \
  --billing-mode PAY_PER_REQUEST

# Now you can query: "Give me all orders for userId U001, sorted by orderDate"
```

### Put Items (Insert Data):

```bash
# Insert a single item
aws dynamodb put-item \
  --table-name Users \
  --item '{
    "userId": {"S": "U001"},
    "name": {"S": "Alice Johnson"},
    "email": {"S": "alice@example.com"},
    "age": {"N": "30"},
    "isPremium": {"BOOL": true},
    "tags": {"L": [{"S": "devops"}, {"S": "cloud"}]}
  }'

# Data types:
# S    = String
# N    = Number (stored as string, treated as number)
# BOOL = Boolean (true/false)
# L    = List (array)
# M    = Map (nested object)
# SS   = String Set
# NS   = Number Set
# NULL = Null
```

### Get Items (Read Data):

```bash
# Get a single item by primary key (fastest operation in DynamoDB)
aws dynamodb get-item \
  --table-name Users \
  --key '{"userId": {"S": "U001"}}' \
  --consistent-read

# --consistent-read: Get the most up-to-date data (uses more RCUs)
# Without it: Eventually consistent (cheaper, but might be slightly stale)
```

### Query Items:

```bash
# Query all orders for a specific user
aws dynamodb query \
  --table-name Orders \
  --key-condition-expression "userId = :uid AND orderDate BETWEEN :start AND :end" \
  --expression-attribute-values '{
    ":uid": {"S": "U001"},
    ":start": {"S": "2024-01-01"},
    ":end": {"S": "2024-12-31"}
  }'

# This returns all orders for user U001 in 2024, sorted by date
# Query is FAST because it uses the primary key
```

### Scan (Full Table Scan — Use Sparingly!):

```bash
# Scan the entire table (EXPENSIVE — reads every item!)
aws dynamodb scan \
  --table-name Users \
  --filter-expression "age > :min_age" \
  --expression-attribute-values '{":min_age": {"N": "25"}}'

# ⚠️ WARNING: Scan reads EVERY item in the table, then filters.
# For large tables, this is very expensive and slow.
# Always prefer Query over Scan when possible!
```

### Update an Item:

```bash
# Update a specific attribute
aws dynamodb update-item \
  --table-name Users \
  --key '{"userId": {"S": "U001"}}' \
  --update-expression "SET age = :new_age, #n = :new_name" \
  --expression-attribute-names '{"#n": "name"}' \
  --expression-attribute-values '{
    ":new_age": {"N": "31"},
    ":new_name": {"S": "Alice Smith"}
  }'

# Note: "name" is a reserved word in DynamoDB, so we use
# expression-attribute-names to create an alias "#n" for it
```

### Delete an Item:

```bash
# Delete an item by primary key
aws dynamodb delete-item \
  --table-name Users \
  --key '{"userId": {"S": "U001"}}'
```

## 🔑 DynamoDB Key Features

### 1. DynamoDB Streams

Capture a **time-ordered sequence** of changes to your table. Every insert, update, or delete triggers a stream record.

```
DynamoDB Streams:
    
    Table change → Stream Record → Lambda function → Do something!

    Use cases:
    → Trigger Lambda when a new order is placed
    → Replicate data to another table or region
    → Build event-driven architectures
    → Send notifications when data changes
    → Maintain materialized views
```

### 2. DynamoDB Accelerator (DAX)

DAX is an **in-memory cache** that sits in front of DynamoDB. It caches reads, reducing latency from **milliseconds to microseconds**.

```
Without DAX:
    App → DynamoDB (1-10 ms response time)

With DAX:
    App → DAX Cache → DynamoDB
    
    Cache HIT:  App → DAX (microseconds! ⚡)
    Cache MISS: App → DAX → DynamoDB → DAX caches it → App
    
    DAX is compatible with DynamoDB API — just change the endpoint!
    No code changes needed (almost).
```

### 3. Time to Live (TTL)

Automatically **delete expired items** from your table at no cost.

```bash
# Enable TTL on the "expiresAt" attribute
aws dynamodb update-time-to-live \
  --table-name Sessions \
  --time-to-live-specification "Enabled=true,AttributeName=expiresAt"

# Now any item with expiresAt < current time will be automatically deleted.
# Great for: session data, temporary tokens, cache entries
```

### 4. Global Tables (Multi-Region)

Automatically replicate your DynamoDB table across multiple AWS regions. Users in each region get **low-latency reads AND writes**.

```
Global Tables:
    ┌────────────┐     ┌────────────┐     ┌────────────┐
    │ US-EAST-1  │ ←──→│ EU-WEST-1  │ ←──→│ AP-SOUTH-1 │
    │ (replica)  │     │ (replica)  │     │ (replica)  │
    └────────────┘     └────────────┘     └────────────┘
    
    All replicas are read-write (active-active!)
    Replication latency: typically <1 second
    Conflict resolution: last-writer-wins
```

### 5. Point-in-Time Recovery (PITR)

```
PITR lets you restore your table to any second in the last 35 days.
    
    "Oh no, someone accidentally deleted all the data at 2:30 PM!"
    
    → Restore to 2:29 PM. Data is back! 🎉
    
    Enable it:
    aws dynamodb update-continuous-backups \
      --table-name Users \
      --point-in-time-recovery-specification PointInTimeRecoveryEnabled=true
```

## 🔄 RDS vs Aurora vs DynamoDB — The Big Comparison

| Feature | RDS | Aurora | DynamoDB |
|---------|-----|--------|----------|
| **Type** | Relational (SQL) | Relational (SQL) | NoSQL (Key-Value/Document) |
| **Engines** | MySQL, PostgreSQL, MariaDB, Oracle, SQL Server | MySQL-compatible, PostgreSQL-compatible | Proprietary (AWS) |
| **Query Language** | SQL | SQL | DynamoDB API / PartiQL |
| **Schema** | Fixed (tables, columns) | Fixed (tables, columns) | Flexible (schemaless) |
| **JOINs** | ✅ | ✅ | ❌ (denormalize instead) |
| **Transactions** | ✅ | ✅ | ✅ (limited) |
| **Max Storage** | 64 TB | 128 TB | Unlimited |
| **Scaling** | Vertical (change instance) | Vertical + Auto storage | Horizontal (auto-partitioning) |
| **Read Replicas** | Up to 5 | Up to 15 | N/A (auto-scaled) |
| **Latency** | Low (ms) | Very low (ms) | Single-digit ms (µs with DAX) |
| **Managed** | Partially | Fully | Fully (zero admin) |
| **Serverless** | ❌ | ✅ (Aurora Serverless v2) | ✅ (On-Demand mode) |
| **Global** | Cross-region replicas | Aurora Global DB | Global Tables |
| **Cost Model** | Per-instance-hour | Per-instance-hour | Per-request or provisioned |

### When to Use What — Decision Tree:

```
Do you need complex SQL queries, JOINs, and relationships?
├─ YES → Do you need maximum performance and auto-scaling storage?
│         ├─ YES → AURORA ✅
│         └─ NO  → RDS ✅ (cheaper for simple needs)
│
└─ NO → Do you need:
         → Simple key-value lookups?
         → Massive scale (millions of requests/sec)?
         → Flexible schema?
         → Fully serverless (zero management)?
         → Single-digit millisecond latency?
         └─ YES to any → DYNAMODB ✅
```

```hcl
  server_side_encryption {
    enabled = true
  }
```

## 🧠 Common Interview Questions

### Q: Can you convert an unencrypted RDS instance to encrypted?
```
Not directly. You need to:
1. Take a snapshot of the unencrypted instance
2. Copy the snapshot with encryption enabled
3. Restore a new instance from the encrypted snapshot
4. Update your application to point to the new instance
5. Delete the old unencrypted instance
```

### Q: What's the difference between Multi-AZ and Read Replicas?
```
Multi-AZ:
    → Purpose: HIGH AVAILABILITY (disaster recovery)
    → Synchronous replication
    → Automatic failover
    → Standby is NOT readable
    → Same region only

Read Replicas:
    → Purpose: SCALABILITY (handle more reads)
    → Asynchronous replication
    → No automatic failover (but can be promoted manually)
    → Replicas ARE readable
    → Can be cross-region
```

### Q: When would you choose DynamoDB over RDS?
```
Choose DynamoDB when:
    → You need simple key-value access patterns
    → You need unlimited horizontal scaling
    → You need single-digit millisecond latency at any scale
    → Your data model doesn't require JOINs
    → You want zero database administration
    → You need a serverless database

Choose RDS when:
    → You need complex SQL queries and JOINs
    → Your data has strong relationships
    → You're migrating from an existing SQL database
    → Your team knows SQL well
```
