## 🤔 Why Do We Need Container Orchestration?

You already know Docker — you can build images and run containers. But in production, you don't just run ONE container. You run **hundreds or thousands** of containers. And managing all of them manually is a nightmare:

```
Running Containers Manually (the hard way):
    → Which server should this container run on?
    → This server is out of memory — now what?
    → Container crashed — who restarts it?
    → I need 10 copies of this container for traffic spikes
    → How do I update all containers without downtime?
    → How do I route traffic to the right containers?
    → How do I monitor all these containers?
    → Server died — what happens to the containers on it?
    
    Answer: You need a CONTAINER ORCHESTRATOR! 🎯
```

The two most popular container orchestrators in the AWS ecosystem are:

```
1. Amazon ECS    → AWS's own container orchestrator (simpler, AWS-native)
2. Amazon EKS    → Managed Kubernetes (more complex, industry standard)

This note focuses on ECS. We'll cover EKS in a later note.
```

## 🏗️ What Is Amazon ECS?

Amazon ECS (Elastic Container Service) is AWS's **fully managed container orchestration** platform. It handles:

- **Where** to run your containers
- **How many** copies to run
- **Restarting** crashed containers
- **Scaling** up/down based on demand
- **Load balancing** traffic across containers
- **Rolling updates** for zero-downtime deployments

### ECS vs Kubernetes — Quick Comparison:

| Aspect | ECS | Kubernetes (EKS) |
|--------|-----|-------------------|
| **Complexity** | Simpler, fewer concepts | More complex, steeper learning curve |
| **AWS Integration** | Deep, native AWS integration | Works on any cloud |
| **Portability** | AWS only | Multi-cloud, on-premises |
| **Learning Curve** | Easier (especially with Fargate) | Harder, but more powerful |
| **Best For** | AWS-only shops, simpler needs | Multi-cloud, complex needs |

## 📦 ECS Core Concepts — Explained Simply

Let's go through each concept one by one:

### 1. Cluster

A **cluster** is a logical grouping — it's the "box" that holds everything. Think of it as your workspace or environment.

```
Cluster = "My Production Environment"
    → Contains services, tasks, and the underlying compute
    → You can have multiple clusters: dev, staging, production
    → A cluster can use EC2, Fargate, or both
```

### 2. Task Definition

A **task definition** is like a **blueprint** or **recipe** for running your containers. It describes:
- Which Docker image to use
- How much CPU and memory
- Which ports to expose
- Environment variables
- Logging configuration
- Volumes
- And more...

Think of it like a **docker-compose.yml** but for ECS.

```
Task Definition = "Here's HOW to run my container"

Example:
{
    "name": "my-web-app",
    "image": "nginx:latest",
    "cpu": 256,
    "memory": 512,
    "port": 80,
    "environment": [
        { "name": "ENV", "value": "production" }
    ]
}
```

### 3. Task

A **task** is a **running instance** of a task definition. If the task definition is the recipe, the task is the actual dish being served.

```
Task Definition = Blueprint / Recipe
Task            = Running instance of that blueprint

One Task Definition → can run many Tasks (like Docker Compose up --scale)
```

A task can contain **one or more containers** (sidecar pattern):

```
Task (single container):
    ┌────────────────┐
    │  Web App       │
    │  Container     │
    └────────────────┘

Task (multiple containers — sidecar pattern):
    ┌────────────────────────────┐
    │  ┌──────────┐  ┌────────┐ │
    │  │ Web App  │  │ Log    │ │
    │  │ Container│  │ Sidecar│ │
    │  └──────────┘  └────────┘ │
    │  These share network &    │
    │  can share volumes        │
    └────────────────────────────┘
```

### 4. Service

A **service** maintains a **desired number of tasks** running at all times. If a task crashes, the service **automatically replaces** it.

```
Service = "I want 3 copies of my web app running at all times"

    Service Config: desiredCount = 3
    
    ┌────────┐  ┌────────┐  ┌────────┐
    │ Task 1 │  │ Task 2 │  │ Task 3 │
    │  ✅    │  │  ✅    │  │  ✅    │
    └────────┘  └────────┘  └────────┘
    
    If Task 2 crashes:
    ┌────────┐  ┌────────┐  ┌────────┐
    │ Task 1 │  │ Task 2 │  │ Task 3 │
    │  ✅    │  │  💥    │  │  ✅    │
    └────────┘  └────────┘  └────────┘
    
    ECS Service automatically starts a replacement:
    ┌────────┐  ┌────────┐  ┌────────┐
    │ Task 1 │  │ Task 2′ │  │ Task 3 │
    │  ✅    │  │  ✅     │  │  ✅    │
    └────────┘  └─────────┘  └────────┘
    
    Self-healing! 🎉
```

### 5. Container Instance (EC2 Launch Type only)

If you use the **EC2 launch type**, the containers run on **EC2 instances** that you manage. Each EC2 instance runs the **ECS Agent** — a small program that communicates with the ECS service.

```
EC2 Launch Type:
    YOU manage the EC2 instances (patching, scaling, AMIs)
    ECS Agent runs on each instance and manages containers

    ┌───────────────────────────────────┐
    │  EC2 Instance                     │
    │  ┌───────────┐  ┌─────────────┐  │
    │  │ ECS Agent │  │ Docker      │  │
    │  │ (manages  │  │ Engine      │  │
    │  │  tasks)   │  │             │  │
    │  └───────────┘  └─────────────┘  │
    │  ┌────────┐ ┌────────┐ ┌──────┐  │
    │  │Task 1  │ │Task 2  │ │Task 3│  │
    │  └────────┘ └────────┘ └──────┘  │
    └───────────────────────────────────┘
```

## 🚀 EC2 Launch Type vs Fargate Launch Type

This is one of the most important decisions when using ECS. Let's compare:

```
┌──────────────────────────────────────────────────────────────────┐
│                   LAUNCH TYPES COMPARED                          │
│                                                                  │
│  EC2 LAUNCH TYPE:                                               │
│  ┌────────────────────────────────────────┐                     │
│  │  You manage EC2 instances              │                     │
│  │  → You choose instance types           │                     │
│  │  → You handle OS patches               │                     │
│  │  → You scale instances up/down          │                     │
│  │  → You monitor instance health          │                     │
│  │  → You pay for EC2 instances (even idle)│                     │
│  │  → More control, more responsibility    │                     │
│  │                                         │                     │
│  │  Best for:                              │                     │
│  │  → Cost optimization (reserved/spot)    │                     │
│  │  → GPU workloads                        │                     │
│  │  → Specific OS/kernel requirements      │                     │
│  │  → Very large containers                │                     │
│  └────────────────────────────────────────┘                     │
│                                                                  │
│  FARGATE LAUNCH TYPE:                                           │
│  ┌────────────────────────────────────────┐                     │
│  │  AWS manages everything                │                     │
│  │  → No servers to manage at all!        │                     │
│  │  → No patching, no scaling instances   │                     │
│  │  → Just define CPU/memory per task     │                     │
│  │  → Pay only for task resources used    │                     │
│  │  → Less control, less headaches 😌     │                     │
│  │                                         │                     │
│  │  Best for:                              │                     │
│  │  → Most workloads (web apps, APIs)     │                     │
│  │  → Teams that want simplicity          │                     │
│  │  → Bursty/unpredictable workloads      │                     │
│  │  → Batch processing                    │                     │
│  └────────────────────────────────────────┘                     │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### Side-by-Side Comparison:

| Feature | EC2 Launch Type | Fargate Launch Type |
|---------|----------------|---------------------|
| **Server Management** | You manage EC2 instances | No servers to manage |
| **Patching** | Your responsibility | AWS handles it |
| **Scaling Infrastructure** | You scale EC2 instances | AWS scales automatically |
| **Pricing** | Pay for EC2 instances | Pay per task (vCPU + memory per second) |
| **Instance Types** | Choose any EC2 type (GPU, etc.) | Predefined CPU/memory combos |
| **Docker Storage** | You configure it | AWS manages it |
| **SSH Access** | Yes (you own the instance) | No (serverless) |
| **Max Resources** | Up to the EC2 instance limit | 16 vCPU, 120 GB memory per task |
| **Startup Time** | Faster (if instance exists) | Slightly slower (provisioning) |
| **Cost at Scale** | Cheaper (use Reserved/Spot) | Can be more expensive |
| **Best For** | Cost-conscious, GPU, high control | Simplicity, smaller teams |

### Decision Guide:

```
Want simplicity? No server management headaches?
    → Use FARGATE ✅

Need GPUs? Need specific instance types?
    → Use EC2 Launch Type ✅

Need to save money at scale? (Reserved Instances, Spot)
    → Use EC2 Launch Type ✅

Running occasional/bursty workloads?
    → Use FARGATE ✅ (pay only when running)

For most new projects → Start with FARGATE
```

## 💻 ECS Hands-On — Step by Step

### Step 1: Create an ECS Cluster

```bash
# Create a Fargate cluster (no EC2 instances needed!)
aws ecs create-cluster \
  --cluster-name my-app-cluster \
  --capacity-providers FARGATE FARGATE_SPOT \
  --default-capacity-provider-strategy \
    capacityProvider=FARGATE,weight=1,base=1 \
    capacityProvider=FARGATE_SPOT,weight=3

# Explanation:
# --cluster-name     : Name of your cluster
# --capacity-providers: FARGATE (on-demand) and FARGATE_SPOT (cheaper, can be interrupted)
# --default-capacity-provider-strategy:
#   base=1           : At least 1 task runs on regular FARGATE (guaranteed)
#   weight=1/3       : 1 out of 4 tasks use FARGATE, 3 out of 4 use FARGATE_SPOT
#   This saves money! Spot is up to 70% cheaper.
```

### Step 2: Create a Task Definition

The task definition is a JSON file that describes your containers:

```json
{
  "family": "my-web-app",
  "networkMode": "awsvpc",
  "requiresCompatibilities": ["FARGATE"],
  "cpu": "256",
  "memory": "512",
  "executionRoleArn": "arn:aws:iam::123456789:role/ecsTaskExecutionRole",
  "taskRoleArn": "arn:aws:iam::123456789:role/ecsTaskRole",
  "containerDefinitions": [
    {
      "name": "web-app",
      "image": "123456789.dkr.ecr.us-east-1.amazonaws.com/my-web-app:latest",
      "essential": true,
      "portMappings": [
        {
          "containerPort": 8080,
          "hostPort": 8080,
          "protocol": "tcp"
        }
      ],
      "environment": [
        {
          "name": "NODE_ENV",
          "value": "production"
        },
        {
          "name": "PORT",
          "value": "8080"
        }
      ],
      "secrets": [
        {
          "name": "DB_PASSWORD",
          "valueFrom": "arn:aws:secretsmanager:us-east-1:123456789:secret:db-password-abc123"
        }
      ],
      "logConfiguration": {
        "logDriver": "awslogs",
        "options": {
          "awslogs-group": "/ecs/my-web-app",
          "awslogs-region": "us-east-1",
          "awslogs-stream-prefix": "ecs"
        }
      },
      "healthCheck": {
        "command": ["CMD-SHELL", "curl -f http://localhost:8080/health || exit 1"],
        "interval": 30,
        "timeout": 5,
        "retries": 3,
        "startPeriod": 60
      }
    }
  ]
}
```

Let's break down every field:

```
family              : Name for this task definition (like a project name)
networkMode         : "awsvpc" = each task gets its own ENI (network interface)
                      This is REQUIRED for Fargate
requiresCompatibilities: ["FARGATE"] or ["EC2"] or both
cpu                 : CPU units (256 = 0.25 vCPU, 512 = 0.5 vCPU, 1024 = 1 vCPU)
memory              : Memory in MB (512 = 512 MB, 1024 = 1 GB)
executionRoleArn    : IAM role that ECS uses to pull images and write logs
taskRoleArn         : IAM role that YOUR container uses to access AWS services

containerDefinitions:
    name            : Container name (used in logs and references)
    image           : Docker image to run (from ECR, Docker Hub, etc.)
    essential       : If true, task fails if this container dies
    portMappings    : Which ports to expose
    environment     : Environment variables (non-sensitive)
    secrets         : Sensitive values from Secrets Manager or SSM
    logConfiguration: Where to send logs (CloudWatch Logs)
    healthCheck     : Container-level health check
```

```bash
# Register the task definition
aws ecs register-task-definition \
  --cli-input-json file://task-definition.json
```

### Fargate CPU/Memory Combinations:

Not all CPU/memory combos are valid! Here are the allowed ones:

| CPU (vCPU) | Memory Options |
|-----------|----------------|
| 0.25 vCPU (256) | 0.5 GB, 1 GB, 2 GB |
| 0.5 vCPU (512) | 1 GB to 4 GB (1 GB increments) |
| 1 vCPU (1024) | 2 GB to 8 GB (1 GB increments) |
| 2 vCPU (2048) | 4 GB to 16 GB (1 GB increments) |
| 4 vCPU (4096) | 8 GB to 30 GB (1 GB increments) |
| 8 vCPU (8192) | 16 GB to 60 GB (4 GB increments) |
| 16 vCPU (16384) | 32 GB to 120 GB (8 GB increments) |

### Step 3: Create an ECS Service

```bash
# Create a service that runs 3 copies of your task behind an ALB
aws ecs create-service \
  --cluster my-app-cluster \
  --service-name my-web-service \
  --task-definition my-web-app:1 \
  --desired-count 3 \
  --launch-type FARGATE \
  --network-configuration "awsvpcConfiguration={
    subnets=[subnet-0abc1234,subnet-0def5678],
    securityGroups=[sg-0abc1234],
    assignPublicIp=ENABLED
  }" \
  --load-balancers "targetGroupArn=arn:aws:elasticloadbalancing:us-east-1:123456:targetgroup/my-tg/abc123,containerName=web-app,containerPort=8080" \
  --deployment-configuration "maximumPercent=200,minimumHealthyPercent=100" \
  --enable-execute-command

# Let's break this down:
#
# --cluster                  : Which cluster to run in
# --service-name             : Name for this service
# --task-definition          : Which task definition to use (name:revision)
# --desired-count            : How many tasks to keep running (3 copies)
# --launch-type              : FARGATE or EC2
# --network-configuration    : VPC settings for the tasks
#   subnets                  : Which subnets to run tasks in
#   securityGroups           : Which security group controls task traffic
#   assignPublicIp           : ENABLED if tasks need internet access (in public subnet)
# --load-balancers           : Connect to an ALB target group
#   containerName            : Which container in the task receives traffic
#   containerPort            : Which port on the container
# --deployment-configuration :
#   maximumPercent=200        : During deployment, can run up to 2x tasks (6 total)
#   minimumHealthyPercent=100 : Must keep at least 100% tasks healthy during deployment
# --enable-execute-command   : Allow "exec" into running containers (for debugging)
```

### Step 4: Update Your Service (Deploy New Version)

```bash
# Option 1: Update the task definition with a new image tag,
# then update the service to use the new revision

# Register new task definition (with updated image tag)
aws ecs register-task-definition \
  --cli-input-json file://task-definition-v2.json
# Returns: my-web-app:2 (revision 2)

# Update the service to use the new task definition
aws ecs update-service \
  --cluster my-app-cluster \
  --service my-web-service \
  --task-definition my-web-app:2 \
  --force-new-deployment

# ECS will perform a rolling update:
# 1. Start new tasks with the new image
# 2. Wait for new tasks to pass health checks
# 3. Drain connections from old tasks
# 4. Stop old tasks
# 5. Repeat until all tasks are updated
# Zero downtime! 🎉
```

## 🔄 ECS Deployment Strategies

ECS supports different deployment strategies through **deployment controllers**:

### 1. Rolling Update (Default)

```
Rolling Update — how it works:
    
    Desired: 3 tasks, minimumHealthy: 100%, maximum: 200%
    
    Step 1: Start new Task 4 (new version)
    Running: [Old1 ✅] [Old2 ✅] [Old3 ✅] [New4 🟡 starting...]
    
    Step 2: New4 passes health check, stop Old1
    Running: [Old1 🔴 draining] [Old2 ✅] [Old3 ✅] [New4 ✅]
    
    Step 3: Start New5, Old1 stopped
    Running: [Old2 ✅] [Old3 ✅] [New4 ✅] [New5 🟡 starting...]
    
    Step 4: Continue until all are new
    Running: [New4 ✅] [New5 ✅] [New6 ✅]
    
    Done! Zero downtime! 🎉
```

### 2. Blue/Green Deployment (with CodeDeploy)

```
Blue/Green Deployment:
    
    BLUE (current version)     GREEN (new version)
    ┌────────┐                  ┌────────┐
    │ Task 1 │                  │ Task 1 │
    │ v1.0   │                  │ v2.0   │
    └────────┘                  └────────┘
    ┌────────┐                  ┌────────┐
    │ Task 2 │                  │ Task 2 │
    │ v1.0   │                  │ v2.0   │
    └────────┘                  └────────┘
    ┌────────┐                  ┌────────┐
    │ Task 3 │     SWITCH →     │ Task 3 │
    │ v1.0   │     TRAFFIC      │ v2.0   │
    └────────┘                  └────────┘
    
    ALB switches traffic from BLUE to GREEN instantly.
    If something goes wrong, switch back to BLUE.
```

## 🔒 ECS IAM Roles — Understanding the Two Roles

ECS uses **two different IAM roles**, and confusing them is a common mistake:

```
┌──────────────────────────────────────────────────────────────┐
│                  ECS IAM ROLES                                │
│                                                              │
│  1. Task Execution Role (executionRoleArn)                  │
│     → Used by ECS itself (the infrastructure)               │
│     → Permissions needed:                                    │
│       • Pull Docker images from ECR                         │
│       • Write logs to CloudWatch                            │
│       • Read secrets from Secrets Manager / SSM             │
│     → WHO uses it: ECS Agent (AWS infrastructure)           │
│                                                              │
│  2. Task Role (taskRoleArn)                                 │
│     → Used by YOUR application code inside the container    │
│     → Permissions needed:                                    │
│       • Whatever your app needs (S3, DynamoDB, SQS, etc.)  │
│     → WHO uses it: Your application code                    │
│                                                              │
│  Think of it this way:                                       │
│  Execution Role = "ECS, please pull my image and set up logs"│
│  Task Role      = "My app needs to read from S3 and write   │
│                    to DynamoDB"                               │
└──────────────────────────────────────────────────────────────┘
```

### Creating the Roles:

```bash
# Create Task Execution Role
# (This is a standard role — AWS provides a managed policy for it)
aws iam create-role \
  --role-name ecsTaskExecutionRole \
  --assume-role-policy-document '{
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Allow",
      "Principal": { "Service": "ecs-tasks.amazonaws.com" },
      "Action": "sts:AssumeRole"
    }]
  }'

# Attach the AWS managed policy
aws iam attach-role-policy \
  --role-name ecsTaskExecutionRole \
  --policy-arn arn:aws:iam::aws:policy/service-role/AmazonECSTaskExecutionRolePolicy

# This policy allows:
# → ecr:GetAuthorizationToken (pull images from ECR)
# → ecr:BatchGetImage, ecr:GetDownloadUrlForLayer
# → logs:CreateLogStream, logs:PutLogEvents (write to CloudWatch)
```

```bash
# Create Task Role (for your application)
aws iam create-role \
  --role-name myAppTaskRole \
  --assume-role-policy-document '{

    │  │ IP: .10   │  │ IP: .11   │  │ IP: .12   │ │
    │  │ ENI: eni-1│  │ ENI: eni-2│  │ ENI: eni-3│ │
    │  │ SG: sg-x  │  │ SG: sg-x  │  │ SG: sg-x  │ │
    │  └───────────┘  └───────────┘  └───────────┘ │
    │                                                │
    │  Security Groups apply at the TASK level!     │
    │  Each task can have different security groups! │
    └────────────────────────────────────────────────┘

```
Internet

## 📊 ECS Service Auto Scaling

ECS can automatically scale the number of tasks based on metrics:

```
Auto Scaling Options:
    
    1. Target Tracking Scaling
       → "Keep CPU utilization at 50%"
       → ECS adds/removes tasks to maintain the target
    
    2. Step Scaling
       → "If CPU > 70%, add 2 tasks. If CPU > 90%, add 5 tasks."
       → More granular control
    
    3. Scheduled Scaling
       → "Every weekday at 8 AM, scale to 10 tasks. At 8 PM, scale to 3."
       → For predictable traffic patterns
```

```bash
# Register your ECS service as a scalable target
aws application-autoscaling register-scalable-target \
  --service-namespace ecs \
  --resource-id service/my-app-cluster/my-web-service \
  --scalable-dimension ecs:service:DesiredCount \
  --min-capacity 2 \
  --max-capacity 20

# This says: "The service can scale between 2 and 20 tasks"
```

```bash
# Create a Target Tracking scaling policy (keep CPU at 50%)
aws application-autoscaling put-scaling-policy \
  --service-namespace ecs \
  --resource-id service/my-app-cluster/my-web-service \
  --scalable-dimension ecs:service:DesiredCount \
  --policy-name cpu-target-tracking \
  --policy-type TargetTrackingScaling \
  --target-tracking-scaling-policy-configuration '{
    "TargetValue": 50.0,
    "PredefinedMetricSpecification": {
      "PredefinedMetricType": "ECSServiceAverageCPUUtilization"
    },
    "ScaleInCooldown": 300,
    "ScaleOutCooldown": 60
  }'

# TargetValue: 50.0       = Keep average CPU at 50%
# ScaleOutCooldown: 60     = Wait 60 seconds before scaling out again
# ScaleInCooldown: 300     = Wait 5 minutes before scaling in (avoid flapping)
```

## 📋 ECS Service Discovery

Service Discovery lets your containers find each other by **name** instead of IP addresses (which change every time a task restarts).

```
Without Service Discovery:
    Web App needs to call API Service
    → What's the API's IP? It changes every deployment! 😱
    → Hardcoding IPs is BAD

With Service Discovery (using AWS Cloud Map):
    Web App calls: api.my-app.local
    → AWS Cloud Map resolves it to the current task IPs
    → IPs update automatically when tasks change
    → Just like DNS, but for containers! ✅
```

```bash
# Create a namespace for service discovery
aws servicediscovery create-private-dns-namespace \
  --name my-app.local \
  --vpc vpc-0abc1234

# Create a service discovery service
aws servicediscovery create-service \
  --name api \
  --dns-config "NamespaceId=ns-abc123,DnsRecords=[{Type=A,TTL=10}]" \
  --health-check-custom-config FailureThreshold=1

# Now tasks registered with this service can be reached at:
# api.my-app.local → resolves to task IPs
```

## 🐛 ECS Exec — Debug Running Containers

ECS Exec lets you **shell into** a running Fargate container for debugging. It's like `docker exec` but for ECS:

```bash
# Execute a command in a running container
aws ecs execute-command \
  --cluster my-app-cluster \
  --task abc123def456 \
  --container web-app \
  --interactive \
  --command "/bin/sh"

# Now you have a shell inside the running container!
# You can:
# → Check environment variables: env
# → View logs: cat /var/log/app.log
# → Test connectivity: curl http://other-service:8080
# → Debug issues: ps aux, netstat, etc.

# Requirements:
# 1. --enable-execute-command must be set on the service
# 2. Task role needs SSM permissions
# 3. AWS SSM Agent is automatically included in Fargate platform version 1.4.0+
```

## 📝 ECS Logging

By default, you should send container logs to **CloudWatch Logs**:

```json
// In your task definition, add this to each container:
"logConfiguration": {
    "logDriver": "awslogs",
    "options": {
        "awslogs-group": "/ecs/my-web-app",
        "awslogs-region": "us-east-1",
        "awslogs-stream-prefix": "ecs",
        "awslogs-create-group": "true"
    }
}
```

```
Log structure in CloudWatch:
    /ecs/my-web-app                          ← Log Group
    ├── ecs/web-app/task-id-1                ← Log Stream (Task 1)
    ├── ecs/web-app/task-id-2                ← Log Stream (Task 2)
    └── ecs/web-app/task-id-3                ← Log Stream (Task 3)
```

Other log drivers supported:
- `awslogs` — CloudWatch Logs (most common)
- `splunk` — Splunk
- `fluentd` — Fluentd
- `awsfirelens` — FireLens (route to any destination)

## 📋 ECS vs EKS — When to Choose What

```
Choose ECS when:
    → You're all-in on AWS (no multi-cloud plans)
    → You want simplicity (fewer concepts than Kubernetes)
    → Your team is small and doesn't want to learn Kubernetes
    → You're running straightforward containerized apps
    → You want deep AWS integration out of the box

Choose EKS when:
    → You might go multi-cloud in the future
    → Your team already knows Kubernetes
    → You need the rich Kubernetes ecosystem (Helm, Operators, etc.)
    → You have complex networking or scheduling requirements
    → You want portability (same manifests work on any K8s cluster)
```

## 🧠 Important Concepts for Interviews & Exams

### Q: What's the difference between ECS and Fargate?

```
ECS = The orchestration service (manages WHAT containers to run, HOW MANY, etc.)
Fargate = The compute engine (manages WHERE the containers actually run)

ECS is the "brain" — decides what to do
Fargate is the "muscles" — does the actual work

You use ECS WITH Fargate (or ECS WITH EC2 instances)
Fargate is a LAUNCH TYPE within ECS, not a separate service
```

### Q: How does ECS handle secrets?

```
1. AWS Secrets Manager → For database passwords, API keys
   → Referenced in task definition as "secrets"
   → ECS Agent fetches them at task startup
   → Injected as environment variables

2. SSM Parameter Store → For configuration values
   → Similar to Secrets Manager but simpler
   → Can be encrypted (SecureString) or plain (String)

3. NEVER put secrets in:
   → Environment variables in task definition (visible in console!)
   → Docker image
   → Source code
```

### Q: What happens during an ECS rolling deployment?

```
1. ECS creates new tasks with the new task definition
2. New tasks are registered with the load balancer
3. Load balancer runs health checks on new tasks
4. Once healthy, traffic starts flowing to new tasks
5. Old tasks are deregistered from the load balancer
6. Connection draining occurs (deregistration delay)
7. Old tasks are stopped
8. Process repeats until all tasks are updated
9. If any new task fails health checks, deployment can be rolled back
```
