# 15.1 — Observability Fundamentals — Metrics, Logs, Traces (The Three Pillars)

> **One Line Answer:** Observability is the ability to understand **what's happening inside your system** just by looking at the data it produces — and it stands on **three pillars: Metrics, Logs, and Traces.**

## 🤔 Let's Start With a Real-World Analogy

Imagine you're a **doctor** and a patient walks in feeling sick.

How do you figure out what's wrong? You use **three types of information:**

| Type | Doctor's World | DevOps World |
|------|---------------|--------------|
| **Vitals (Numbers)** | Heart rate = 120 bpm, Temperature = 102°F | CPU = 95%, Memory = 87%, Request latency = 3.2s |
| **Patient's Story (Events)** | "I started feeling dizzy at 3 PM after eating fish" | `ERROR: Database connection timeout at 15:04:32` |
| **Full Medical History (Journey)** | First headache → then fever → then vomit → then hospital | User request → API Gateway → Auth Service → DB Query → Timeout |

That's exactly what **Metrics**, **Logs**, and **Traces** are in the software world.

- **Metrics** = The vital signs (numbers over time)
- **Logs** = The patient's story (detailed events)
- **Traces** = The full journey of a request (what happened step by step)

Together, these three let you **observe** your system from the outside and understand what's going on inside — that's **observability**.

## 🏛️ The Three Pillars of Observability

Let's deep dive into each one.

---

### 📊 Pillar 1: Metrics

#### What Are Metrics?

Metrics are **numerical measurements** collected at **regular intervals** over time. They tell you "how much" or "how many" of something is happening.

Think of them like the **dashboard of your car** — speed, fuel level, engine temperature. You don't need to know every detail of the engine; you just glance at the numbers and know if things are OK or not.

#### Examples of Metrics:

| Metric | What It Tells You |
|--------|-------------------|
| `cpu_usage_percent` = 85% | Your server's CPU is working really hard |
| `http_requests_total` = 1,042,389 | Total HTTP requests received so far |
| `http_request_duration_seconds` = 0.25 | Average time to respond to a request |
| `memory_used_bytes` = 3.2 GB | How much RAM is being used |
| `disk_free_bytes` = 10 GB | How much disk space is left |
| `http_errors_total` = 342 | How many requests resulted in errors |
| `container_restart_count` = 5 | How many times a container restarted |

#### Types of Metrics:

| Type | Description | Example |
|------|-------------|---------|
| **Counter** | A value that only goes UP (or resets to zero) | Total requests served, total errors |
| **Gauge** | A value that can go UP or DOWN | Current CPU usage, memory usage, temperature |
| **Histogram** | Groups values into buckets to show distribution | Request latency: how many requests took 0-100ms, 100-500ms, 500ms+ |
| **Summary** | Similar to histogram but calculates percentiles (p50, p99) | 99th percentile latency = 1.2s |

#### How Metrics Are Stored:

Metrics are stored in **time-series databases** — databases designed to store data points with timestamps.

```
metric_name{label1="value1", label2="value2"} value timestamp

# Example:
http_requests_total{method="GET", status="200", path="/api/users"} 1523 1690000000
http_requests_total{method="POST", status="500", path="/api/orders"} 42 1690000000
```

Each entry has:
- **Metric name** — what you're measuring (`http_requests_total`)
- **Labels** — extra context to filter/group (`method="GET"`, `status="200"`)
- **Value** — the actual number (`1523`)
- **Timestamp** — when it was recorded (`1690000000`)

#### Popular Metrics Tools:

| Tool | Description |
|------|-------------|
| **Prometheus** | The king of open-source metrics. Pull-based. PromQL for querying. |
| **Grafana** | Beautiful dashboards to visualize metrics from any source |
| **Datadog** | Commercial SaaS monitoring platform |
| **CloudWatch** | AWS native monitoring (metrics, alarms) |
| **InfluxDB** | Time-series database for metrics |
| **StatsD** | Lightweight metrics collection daemon |

#### When Metrics Are Great:

✅ "Is the system healthy right now?" → Check dashboards  
✅ "Is CPU usage trending upward?" → Plot over time  
✅ "How many 500 errors in the last hour?" → Query the counter  
✅ "Alert me when latency > 2 seconds" → Set up alerting rules  

#### When Metrics Are NOT Enough:

❌ "WHY did latency spike at 3:42 PM?" → Metrics tell you **what** happened, not **why**  
❌ "What was the exact error message?" → You need **logs** for that  
❌ "Which microservice caused the slowdown?" → You need **traces** for that

### 📝 Pillar 2: Logs

#### What Are Logs?

Logs are **timestamped text records** of **events** that happened in your system. Every time something noteworthy happens — a user logs in, an error occurs, a service starts — a log entry is created.

Think of logs like a **diary** that your application keeps. It writes down everything it does, every error it encounters, and every important event.

#### Example Log Entries:

```
# A simple unstructured log
2024-01-15 14:32:05 INFO  UserService - User john@example.com logged in successfully
2024-01-15 14:32:06 ERROR PaymentService - Payment failed for order #12345: insufficient funds
2024-01-15 14:32:07 WARN  DatabasePool - Connection pool usage at 90% (45/50 connections)
2024-01-15 14:32:08 DEBUG APIGateway - Received GET /api/products from 192.168.1.50
```

#### Log Levels:

Every log message has a **severity level** that tells you how important it is:

| Level | When to Use | Example |
|-------|-------------|---------|
| **TRACE** | Super detailed debugging (rarely used in production) | Entering function `calculateTax()` with args: {amount: 100} |
| **DEBUG** | Detailed info for developers during debugging | Database query took 45ms: SELECT * FROM users WHERE id=5 |
| **INFO** | Normal operations — things that are supposed to happen | Server started on port 8080 |
| **WARN** | Something unusual but not broken (yet) | Disk usage at 85%, consider cleanup |
| **ERROR** | Something broke! A specific operation failed | Failed to connect to database: connection refused |
| **FATAL/CRITICAL** | The whole application is crashing | Out of memory! Shutting down! |

#### Unstructured vs Structured Logs:

**Unstructured logs** (old school — hard to search and parse):
```
2024-01-15 14:32:06 ERROR PaymentService - Payment failed for order #12345: insufficient funds
```

**Structured logs** (modern — easy to search, filter, and analyze):
```json
{
  "timestamp": "2024-01-15T14:32:06.123Z",
  "level": "ERROR",
  "service": "PaymentService",
  "message": "Payment failed",
  "orderId": "12345",
  "reason": "insufficient funds",
  "userId": "user-789",
  "traceId": "abc-def-123-456"
}
```

**Why structured logs are better:**
- You can easily **search**: "Show me all logs where `orderId=12345`"
- You can easily **filter**: "Show me only `ERROR` logs from `PaymentService`"
- You can **correlate** with traces using `traceId`
- Machines can parse them automatically

#### Where Do Logs Go?

In a simple setup, logs are just written to files:
```
/var/log/nginx/access.log
/var/log/nginx/error.log
/var/log/syslog
/var/log/application/app.log
```

But in a modern **distributed system** with 50+ microservices running in containers, you can't SSH into every container and `cat` log files. You need **centralized logging** — collecting all logs in one place.

```
                    ┌──────────────┐
App 1 ──logs──►     │              │
App 2 ──logs──►     │  Log         │     ┌──────────────┐
App 3 ──logs──►     │  Aggregator  │────►│  Search &     │
App 4 ──logs──►     │  (Logstash/  │     │  Visualize    │
App 5 ──logs──►     │   Fluentd)   │     │  (Kibana)     │
                    │              │     └──────────────┘
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │  Log Storage  │
                    │ (Elasticsearch│
                    │  / Loki)      │
                    └──────────────┘
```

#### Popular Logging Tools:

| Tool | Description |
|------|-------------|
| **ELK Stack** | Elasticsearch + Logstash + Kibana — the classic logging stack |
| **EFK Stack** | Elasticsearch + Fluentd/Fluent Bit + Kibana — lighter alternative |
| **Loki** | Log aggregation by Grafana Labs — lightweight, pairs with Grafana |
| **Splunk** | Enterprise-grade log management (commercial) |
| **CloudWatch Logs** | AWS native log management |
| **Fluentd / Fluent Bit** | Log collectors/shippers (CNCF projects) |

#### When Logs Are Great:

✅ "What was the exact error message?" → Read the log entry  
✅ "What happened in the last 5 minutes on the payment service?" → Search logs  
✅ "Show me all failed login attempts" → Filter by event type  
✅ Debugging a specific issue → Logs give you the **detailed story**  

#### When Logs Are NOT Enough:

❌ "How many errors per second across all services?" → Metrics are better for aggregation  
❌ "Which service in the chain caused the request to fail?" → Traces show the full journey  
❌ Logs can be **extremely high volume** → Expensive to store and search

### 🔗 Pillar 3: Traces (Distributed Tracing)

#### What Are Traces?

A **trace** follows a single request as it travels through **multiple services** in your system. It shows you the **complete journey** — which services were called, in what order, and how long each one took.

Think of it like a **GPS tracking a delivery package**:
- Package picked up at warehouse (50ms)
- Delivered to sorting center (200ms)  
- Stuck at customs for inspection (3000ms) ← 🐌 **Found the bottleneck!**
- Delivered to local hub (100ms)
- Delivered to customer (150ms)

Similarly, a trace shows:
- User hits API Gateway (5ms)
- API Gateway calls Auth Service (50ms)
- Auth Service calls User Database (200ms)
- API Gateway calls Order Service (100ms)
- Order Service calls Payment Service (3000ms) ← 🐌 **Found the bottleneck!**
- Payment Service calls External Payment API (2800ms) ← 🐌 **Root cause!**

#### Key Concepts:

| Concept | Description |
|---------|-------------|
| **Trace** | The entire journey of one request across all services |
| **Span** | One unit of work within a trace (e.g., one service call). A trace is made up of many spans. |
| **Parent Span** | The span that initiated the current span |
| **Child Span** | A span created by another span |
| **Trace ID** | A unique ID that connects all spans belonging to the same request |
| **Span ID** | A unique ID for each individual span |
| **Context Propagation** | Passing the Trace ID from one service to the next (usually via HTTP headers) |

#### What a Trace Looks Like:

```
Trace ID: abc-123-def-456

├── [Span 1] API Gateway          ──────────────────────────── 3.5s total
│   ├── [Span 2] Auth Service     ────── 250ms
│   │   └── [Span 3] User DB     ──── 200ms
│   ├── [Span 4] Order Service    ──────────────── 3.1s
│   │   ├── [Span 5] Inventory   ──── 100ms
│   │   └── [Span 6] Payment     ──────────── 2.8s    ← SLOW!
│   │       └── [Span 7] Stripe  ────────── 2.7s      ← ROOT CAUSE
│   └── [Span 8] Response        ── 10ms
```

Now you can see:
- The total request took **3.5 seconds**
- Most of the time was spent in **Payment Service** (2.8s)
- The actual bottleneck was the **Stripe API call** (2.7s)

Without tracing, you'd be looking at logs from 6 different services trying to piece together what happened. With tracing, you see the **whole picture at once**.

#### How Context Propagation Works:

When Service A calls Service B, it passes the **Trace ID** in the HTTP headers:

```
# Service A makes an HTTP request to Service B
GET /api/orders HTTP/1.1
Host: order-service:8080
traceparent: 00-abc123def456-span001-01     ← Trace ID is passed along!
```

Service B reads this header, creates a new **span** with the same **Trace ID**, and if it calls another service (Service C), it passes the Trace ID forward again.

This is how all the spans get connected into one single trace.

#### Popular Tracing Tools:

| Tool | Description |
|------|-------------|
| **Jaeger** | Open-source distributed tracing (CNCF project, created by Uber) |
| **Zipkin** | Open-source distributed tracing (created by Twitter) |
| **Tempo** | Distributed tracing backend by Grafana Labs |
| **OpenTelemetry (OTel)** | The universal standard — collects metrics, logs, AND traces |
| **AWS X-Ray** | AWS native distributed tracing |
| **Datadog APM** | Commercial tracing with Datadog |

#### When Traces Are Great:

✅ "Which service caused the slow response?" → Follow the trace  
✅ "What's the full path a request takes through our system?" → Visualize the trace  
✅ "Where is the bottleneck?" → Look for the longest span  
✅ Debugging issues in **microservices architectures** → Traces are a lifesaver  

#### When Traces Are NOT Enough:

❌ "What's the current CPU usage?" → Use metrics  
❌ "What was the exact error message?" → Use logs  
❌ "How many requests per second are we handling?" → Use metrics

## 🤝 How the Three Pillars Work Together

Here's the magic — **you need all three together**. They complement each other perfectly.

### Scenario: Your Website Is Slow

```
Step 1: METRICS alert you
   → "Alert: HTTP response latency > 2 seconds (p99) on api-gateway"
   → You know SOMETHING is wrong

Step 2: TRACES show you WHERE
   → You pick a slow request and look at its trace
   → You see: API Gateway → Order Service → Payment Service (2.8s!) → Stripe API (2.7s!)
   → Now you know the Payment Service calling Stripe is the bottleneck

Step 3: LOGS tell you WHY
   → You search logs for the Payment Service around that time
   → Log: "WARN: Stripe API rate limit reached. Retrying request 3/3. Backoff: 2000ms"
   → Now you know WHY — you're hitting Stripe's rate limit!

Step 4: FIX IT
   → Implement caching, reduce duplicate calls, upgrade Stripe plan
```

### The Three Pillars at a Glance:

| Pillar | Question It Answers | Data Type | Best For |
|--------|-------------------|-----------|----------|
| **Metrics** | What is happening? How much? | Numbers over time | Alerting, dashboards, trends |
| **Logs** | Why did it happen? What exactly? | Text events | Debugging, auditing, details |
| **Traces** | Where in the system did it happen? | Request journey | Finding bottlenecks in distributed systems |

### Analogy — Investigating a Car Accident:

| Pillar | Car Accident Analogy |
|--------|---------------------|
| **Metrics** | "The car was going 120 km/h" (dashboard numbers) |
| **Logs** | "Driver reported: brakes felt soft at 2:15 PM" (event details) |
| **Traces** | "The car went from Highway A → Exit 5 → Sharp turn → Barrier" (the full journey) |

## 📝 Key Terms Quick Reference

| Term | Definition |
|------|------------|
| **Observability** | Ability to understand a system's internal state from its external outputs |
| **Telemetry** | The data produced by your system (metrics, logs, traces) |
| **Instrumentation** | Adding code to your application to produce telemetry data |
| **Time Series** | Data points indexed by timestamp — used for metrics |
| **Cardinality** | The number of unique combinations of labels for a metric (high cardinality = expensive) |
| **SLI** | Service Level Indicator — a measurable metric (e.g., latency, error rate) |
| **SLO** | Service Level Objective — a target for an SLI (e.g., 99.9% availability) |
| **SLA** | Service Level Agreement — a contract with consequences if SLO is breached |
| **MTTD** | Mean Time to Detect — how fast you notice a problem |
| **MTTR** | Mean Time to Recover — how fast you fix a problem |
| **Golden Signals** | The 4 key metrics: Latency, Traffic, Errors, Saturation |
| **RED Method** | Rate, Errors, Duration — for request-driven services |
| **USE Method** | Utilization, Saturation, Errors — for infrastructure resources |

## 🎯 The Golden Signals (Google SRE)

Google's SRE book defines **four golden signals** that every service should monitor:

| Signal | What It Measures | Example |
|--------|-----------------|---------|
| **Latency** | How long it takes to serve a request | p50 = 100ms, p99 = 1.2s |
| **Traffic** | How much demand is on your system | 500 requests/second |
| **Errors** | The rate of failed requests | 2% of requests returning 500 |
| **Saturation** | How full your resources are | CPU at 90%, Memory at 85% |

If you monitor these four things for every service, you'll catch most problems.

## 🔴 RED Method vs 📐 USE Method

These are two complementary frameworks for choosing what to monitor:

### RED Method (for services/applications):

| Letter | Meaning | Example |
|--------|---------|---------|
| **R** — Rate | How many requests per second? | 200 req/s |
| **E** — Errors | How many of those are failing? | 5 req/s (2.5% error rate) |
| **D** — Duration | How long do requests take? | p99 = 800ms |

### USE Method (for infrastructure/resources):

| Letter | Meaning | Example |
|--------|---------|---------|
| **U** — Utilization | How busy is the resource? | CPU at 75% |
| **S** — Saturation | How much work is queued/waiting? | 10 requests waiting in queue |
| **E** — Errors | Are there any hardware/resource errors? | 3 disk I/O errors |

### When to Use Which:

- Monitoring your **API service**? → Use **RED**
- Monitoring your **server hardware**? → Use **USE**
- Want a complete picture? → Use **both** + **Golden Signals**

## ✅ Summary

| Concept | Key Takeaway |
|---------|-------------|
| **Observability** | Understanding system internals from external data |
| **Three Pillars** | Metrics (numbers), Logs (events), Traces (request journeys) |
