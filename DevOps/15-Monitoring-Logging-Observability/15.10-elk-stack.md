## рҹӨ” Let's Start With a Simple Story

Imagine you run a **hospital with 500 rooms.**

Every room has a patient, and every patient has a medical chart. Nurses write observations, doctors write prescriptions, lab techs write test results вҖ” all on **paper**.

Now imagine a patient has a problem. You need to figure out what happened. Where do you go?

**Without centralized logging (no ELK):**
- Walk to Room 101, read the chart рҹ“Ӣ
- Walk to Room 205, read that chart рҹ“Ӣ
- Walk to the lab, find the test results рҹ§Ә
- Walk to the pharmacy, check the prescription рҹ’Ҡ
- **Takes hours. You might miss something critical.**

**With centralized logging (ELK Stack):**
- Sit at your **one computer** рҹ–ҘпёҸ
- Type the patient's name
- **Instantly see EVERYTHING** вҖ” every observation, prescription, test result вҖ” all in one place, searchable, filterable, with pretty graphs and timelines.

**That's exactly what ELK does for your servers and applications.**

Instead of SSHing into 50 servers and running `grep` on log files, you have ONE dashboard where ALL logs flow into, and you can search, filter, and visualize them instantly.

## рҹ“Ұ What is the ELK Stack?

ELK is an acronym for three open-source projects by **Elastic** (the company):

| Letter | Tool | Role | Simple Analogy |
|--------|------|------|----------------|
| **E** | **Elasticsearch** | Stores and searches logs | The **library** вҖ” stores all the books (logs) and lets you search through them lightning-fast |
| **L** | **Logstash** | Collects, transforms, and sends logs | The **mail carrier** вҖ” picks up letters (logs) from everywhere, organizes them, and delivers to the library |
| **K** | **Kibana** | Visualizes logs with dashboards | The **librarian's display board** вҖ” shows you beautiful charts, graphs, and summaries of what's in the library |

### The Flow:

```
Your Servers/Apps вҶ’ Generate Logs
        вҶ“
   Logstash (or Beats)
   - Collects logs from everywhere
   - Parses & transforms them
   - Adds structure to messy data
        вҶ“
   Elasticsearch
   - Stores all the logs
   - Indexes them for fast search
   - Handles massive volumes
        вҶ“
   Kibana
   - Beautiful dashboards
   - Search & filter logs
   - Create alerts & visualizations
        вҶ“
   YOU see everything! рҹҺү
```

### 1пёҸвғЈ Elasticsearch вҖ” The Brain рҹ§ 

**What is it?**

Elasticsearch is a **distributed, RESTful search and analytics engine**. Think of it as a super-fast database that's **optimized for searching text**.

**Why not just use a regular database (like MySQL)?**

| Feature | MySQL | Elasticsearch |
|---------|-------|---------------|
| Full-text search | Slow (LIKE '%error%') | Blazing fast (inverted index) |
| Schema | Rigid (define columns upfront) | Flexible (JSON documents) |
| Scaling | Vertical (bigger server) | Horizontal (add more nodes) |
| Log analysis | Not designed for it | Built for it |
| Real-time | Not really | Near real-time |

**Key Concepts:**

| Concept | What It Is | Analogy |
|---------|-----------|---------|
| **Index** | A collection of similar documents | Like a **database table** |
| **Document** | A single log entry (stored as JSON) | Like a **row** in a table |
| **Shard** | A piece of an index (for distribution) | Like splitting a big book into **chapters** stored on different shelves |
| **Replica** | A copy of a shard (for redundancy) | A **photocopy** of a chapter вҖ” in case the original gets damaged |
| **Node** | A single Elasticsearch server | One **computer** in the cluster |
| **Cluster** | A group of nodes working together | The whole **library system** across multiple buildings |

**How Does Elasticsearch Search So Fast?**

It uses something called an **Inverted Index** вҖ” the same technology Google uses!

**Normal Index (like a book's table of contents):**
```
Page 1 вҶ’ "The server started successfully"
Page 2 вҶ’ "Error: database connection failed"
Page 3 вҶ’ "The server handled 100 requests"
```

To find "error" вҶ’ you have to read EVERY page. Slow! рҹҗў

**Inverted Index (like a book's back-of-the-book index):**
```
"server"     вҶ’ Page 1, Page 3
"started"    вҶ’ Page 1
"error"      вҶ’ Page 2
"database"   вҶ’ Page 2
"connection" вҶ’ Page 2
"failed"     вҶ’ Page 2
"handled"    вҶ’ Page 3
"requests"   вҶ’ Page 3
```

To find "error" вҶ’ instantly go to Page 2! Fast! вҡЎ

**Elasticsearch REST API Examples:**

```bash
# Check if Elasticsearch is running
curl -X GET "http://localhost:9200"

# Response:
{
  "name": "node-1",
  "cluster_name": "my-cluster",
  "version": {
    "number": "8.12.0"
  },
  "tagline": "You Know, for Search"
}
```

```bash
# Create an index
curl -X PUT "http://localhost:9200/application-logs"

# Insert a document (a log entry)
curl -X POST "http://localhost:9200/application-logs/_doc" \
  -H "Content-Type: application/json" \
  -d '{
    "timestamp": "2025-01-15T10:30:00Z",
    "level": "ERROR",
    "service": "payment-service",
    "message": "Payment gateway timeout after 30s",
    "host": "server-03",
    "status_code": 504
  }'
```

```bash
# Search for all ERROR logs
curl -X GET "http://localhost:9200/application-logs/_search" \
  -H "Content-Type: application/json" \
  -d '{
    "query": {
      "match": {
        "level": "ERROR"
      }
    }
  }'
```

```bash
# Search for logs containing "timeout" in the message
curl -X GET "http://localhost:9200/application-logs/_search" \
  -H "Content-Type: application/json" \
  -d '{
    "query": {
      "match": {
        "message": "timeout"
      }
    }
  }'
```

```bash
# Search with multiple conditions (ERROR + payment-service)
curl -X GET "http://localhost:9200/application-logs/_search" \
  -H "Content-Type: application/json" \
  -d '{
    "query": {
      "bool": {
        "must": [
          { "match": { "level": "ERROR" } },
          { "match": { "service": "payment-service" } }
        ],
        "filter": [
          {
            "range": {
              "timestamp": {
                "gte": "2025-01-15T00:00:00Z",
                "lte": "2025-01-15T23:59:59Z"
              }
            }
          }
        ]
      }
    }
  }'
```

**What each part means:**
- `bool` вҶ’ combine multiple conditions
- `must` вҶ’ all these conditions MUST match (like AND)
- `filter` вҶ’ additional filters (range, term) вҖ” doesn't affect scoring
- `range` вҶ’ filter by time range (gte = greater than or equal, lte = less than or equal)

### 2пёҸвғЈ Logstash вҖ” The Pipeline рҹ”§

**What is it?**

Logstash is a **data processing pipeline** that collects data from multiple sources, transforms it, and sends it to a destination (usually Elasticsearch).

Think of Logstash as a **food processing factory**:

```
Raw Materials (Logs)     вҶ’  Factory (Logstash)     вҶ’  Packaged Product (Structured Data)
                              вҶ“
                         1. INPUT:   Receive raw logs
                         2. FILTER:  Clean, parse, transform
                         3. OUTPUT:  Send to Elasticsearch
```

**Logstash has 3 stages:**

| Stage | What It Does | Example |
|-------|-------------|---------|
| **Input** | Where to collect logs FROM | Files, syslog, Kafka, Redis, Beats, TCP/UDP, HTTP |
| **Filter** | How to TRANSFORM the logs | Parse, add fields, remove fields, convert types, geoip lookup |
| **Output** | Where to SEND the processed logs | Elasticsearch, file, stdout, S3, Kafka |

**A Simple Logstash Configuration:**

```ruby
# File: /etc/logstash/conf.d/simple-pipeline.conf

# в”Җв”Җв”Җ STAGE 1: INPUT в”Җв”Җв”Җ
# Where are the logs coming from?
input {
  # Read from a log file
  file {
    path => "/var/log/nginx/access.log"
    start_position => "beginning"    # Start reading from the beginning of the file
    sincedb_path => "/dev/null"      # Don't remember where we left off (for testing)
    type => "nginx-access"           # Tag these logs as nginx-access
  }
}

# в”Җв”Җв”Җ STAGE 2: FILTER в”Җв”Җв”Җ
# How should we transform/parse the logs?
filter {
  # Parse the Nginx access log format
  # Example log line:
  # 192.168.1.100 - - [15/Jan/2025:10:30:00 +0000] "GET /api/users HTTP/1.1" 200 1234
  
  grok {
    match => {
      "message" => "%{IPORHOST:client_ip} - - \[%{HTTPDATE:timestamp}\] \"%{WORD:method} %{URIPATHPARAM:request} HTTP/%{NUMBER:http_version}\" %{NUMBER:status_code:integer} %{NUMBER:bytes:integer}"
    }
  }
  # What grok does:
  # Takes the messy log line and breaks it into structured fields:
  # client_ip = 192.168.1.100
  # timestamp = 15/Jan/2025:10:30:00 +0000
  # method = GET
  # request = /api/users
  # status_code = 200
  # bytes = 1234

  # Convert the timestamp string to a proper date
  date {
    match => ["timestamp", "dd/MMM/yyyy:HH:mm:ss Z"]
    target => "@timestamp"
  }

  # Add geographic information based on the IP address
  geoip {
    source => "client_ip"
  }

  # Remove fields we don't need
  mutate {
    remove_field => ["message", "host", "timestamp"]
  }
}

# в”Җв”Җв”Җ STAGE 3: OUTPUT в”Җв”Җв”Җ
# Where should we send the processed logs?
output {
  # Send to Elasticsearch
  elasticsearch {
    hosts => ["http://localhost:9200"]
    index => "nginx-logs-%{+YYYY.MM.dd}"
    # Creates a new index every day: nginx-logs-2025.01.15, nginx-logs-2025.01.16, etc.
    # This makes it easy to delete old logs (just delete old indices)
  }

  # Also print to console (for debugging вҖ” remove in production)
  stdout {
    codec => rubydebug
  }
}
```

**Breaking down the Grok pattern:**

Grok is Logstash's most powerful filter. It uses **patterns** to extract structured data from unstructured text.

```
%{PATTERN_NAME:field_name}

Common patterns:
%{IPORHOST}   вҶ’ matches an IP address or hostname
%{HTTPDATE}   вҶ’ matches HTTP date format
%{WORD}       вҶ’ matches a single word
%{URIPATHPARAM} вҶ’ matches a URI path
%{NUMBER}     вҶ’ matches a number
%{GREEDYDATA} вҶ’ matches everything (greedy)
```

**Before Grok (raw log):**
```
192.168.1.100 - - [15/Jan/2025:10:30:00 +0000] "GET /api/users HTTP/1.1" 200 1234
```

**After Grok (structured JSON):**
```json
{
  "client_ip": "192.168.1.100",
  "method": "GET",
  "request": "/api/users",
  "http_version": "1.1",
  "status_code": 200,
  "bytes": 1234,
  "@timestamp": "2025-01-15T10:30:00.000Z",
  "geoip": {
    "country_name": "United States",
    "city_name": "New York",
    "latitude": 40.7128,
    "longitude": -74.0060
  }
}
```

**See the magic? One messy line вҶ’ a beautiful, searchable JSON document!**

**Another Example вҖ” Parsing Application Logs:**

```ruby
# Parse application logs that look like:
# 2025-01-15 10:30:00 ERROR [payment-service] PaymentController - Payment failed for order #12345: timeout

input {
  file {
    path => "/var/log/myapp/application.log"
    type => "application"
  }
}

filter {
  grok {
    match => {
      "message" => "%{TIMESTAMP_ISO8601:timestamp} %{LOGLEVEL:level} \[%{DATA:service}\] %{DATA:class} - %{GREEDYDATA:log_message}"
    }
  }
  # Result:
  # timestamp = 2025-01-15 10:30:00
  # level = ERROR
  # service = payment-service
  # class = PaymentController
  # log_message = Payment failed for order #12345: timeout

  date {
    match => ["timestamp", "yyyy-MM-dd HH:mm:ss"]
    target => "@timestamp"
  }

  # Add conditional logic
  if [level] == "ERROR" {
    mutate {
      add_tag => ["alert"]
      add_field => { "severity" => "high" }
    }
  }
}

output {
  elasticsearch {
    hosts => ["http://localhost:9200"]
    index => "app-logs-%{+YYYY.MM.dd}"
  }
}
```

### 3пёҸвғЈ Kibana вҖ” The Eyes рҹ‘Җ

**What is it?**

Kibana is the **visualization layer** of the ELK Stack. It's a web application that lets you create beautiful dashboards, search through logs, and set up alerts вҖ” all from your browser.

**What Can You Do With Kibana?**

| Feature | What It Does | Use Case |
|---------|-------------|----------|
| **Discover** | Search and explore raw logs | "Show me all ERROR logs from the last hour" |
| **Dashboards** | Create visual dashboards with charts, maps, tables | "Overview of all services вҖ” errors, latency, traffic" |
| **Visualize** | Build individual charts (bar, line, pie, heatmap, map) | "Error count per service over time" |
| **Alerting** | Get notified when something goes wrong | "Alert me if error rate > 5% in 5 minutes" |
| **Dev Tools** | Run Elasticsearch queries directly | Testing and debugging queries |
| **Index Management** | Manage your Elasticsearch indices | Delete old indices, set up lifecycle policies |
| **Lens** | Drag-and-drop visualization builder | Easy chart creation without knowing query syntax |

**Kibana Query Language (KQL):**

KQL is the simple query language you use in Kibana's search bar:

```
# Find all ERROR logs
level: "ERROR"

# Find errors in payment-service
level: "ERROR" and service: "payment-service"

# Find logs from a specific IP
client_ip: "192.168.1.100"

# Find 404 status codes
status_code: 404

# Find logs containing "timeout" in the message
log_message: "timeout"

# Wildcard search
service: "payment*"

# Range query (status codes 500-599)
status_code >= 500 and status_code < 600

# NOT query
level: "ERROR" and not service: "test-service"
```

## рҹҡҖ Complete ELK Stack Setup with Docker Compose

Here's a **production-like setup** using Docker Compose. This is the easiest way to get started:

```yaml
# File: docker-compose.yml
# ELK Stack - Complete Setup

version: "3.8"

services:
  # в”Җв”Җв”Җ Elasticsearch в”Җв”Җв”Җ
  elasticsearch:
    image: docker.elastic.co/elasticsearch/elasticsearch:8.12.0
    container_name: elasticsearch
    environment:
      - discovery.type=single-node          # Single node for dev/testing
      - xpack.security.enabled=false        # Disable security for simplicity (enable in production!)
      - "ES_JAVA_OPTS=-Xms1g -Xmx1g"      # Give Elasticsearch 1GB of heap memory
      # Xms = minimum heap, Xmx = maximum heap
      # Rule of thumb: set both the same, and no more than 50% of your total RAM
    volumes:
      - elasticsearch-data:/usr/share/elasticsearch/data   # Persist data
    ports:
      - "9200:9200"    # REST API port
      - "9300:9300"    # Internal cluster communication port
    networks:
      - elk-network
    healthcheck:
      test: ["CMD-SHELL", "curl -f http://localhost:9200 || exit 1"]
      interval: 30s
      timeout: 10s
      retries: 5

  # в”Җв”Җв”Җ Logstash в”Җв”Җв”Җ
  logstash:
    image: docker.elastic.co/logstash/logstash:8.12.0
    container_name: logstash
    volumes:
      - ./logstash/pipeline:/usr/share/logstash/pipeline    # Pipeline config files
      - ./logstash/config/logstash.yml:/usr/share/logstash/config/logstash.yml
      # Mount your log files here if reading from files:
      # - /var/log/nginx:/var/log/nginx:ro
    ports:
      - "5044:5044"    # Beats input (Filebeat sends logs here)
      - "5000:5000"    # TCP input (for syslog or direct log shipping)
      - "9600:9600"    # Logstash monitoring API
    environment:
      - "LS_JAVA_OPTS=-Xms512m -Xmx512m"
    networks:
      - elk-network
    depends_on:
      elasticsearch:
        condition: service_healthy

  # в”Җв”Җв”Җ Kibana в”Җв”Җв”Җ
  kibana:
    image: docker.elastic.co/kibana/kibana:8.12.0
    container_name: kibana
    environment:
      - ELASTICSEARCH_HOSTS=http://elasticsearch:9200
      # Tell Kibana where Elasticsearch is
    ports:
      - "5601:5601"    # Kibana web UI
    networks:
      - elk-network
    depends_on:
      elasticsearch:
        condition: service_healthy

volumes:
  elasticsearch-data:
    driver: local

networks:
  elk-network:
    driver: bridge
```

**Logstash pipeline config:**

```ruby
# File: logstash/pipeline/logstash.conf

input {
  # Accept logs from Beats (Filebeat, Metricbeat, etc.)
  beats {
    port => 5044
  }

  # Accept logs via TCP (for syslog)
  tcp {
    port => 5000
    codec => json
  }
}

filter {
  # If the log is from Nginx
  if [fields][log_type] == "nginx" {
    grok {
      match => {
        "message" => "%{COMBINEDAPACHELOG}"
      }
    }
    mutate {
      convert => {
        "response" => "integer"
        "bytes" => "integer"
      }
    }
  }

  # If the log is JSON already (many modern apps log in JSON)
  if [fields][log_type] == "application" {
    json {
      source => "message"
    }
  }
}

output {
  elasticsearch {
    hosts => ["http://elasticsearch:9200"]
    index => "%{[fields][log_type]}-%{+YYYY.MM.dd}"
    # This creates indices like:
    # nginx-2025.01.15
    # application-2025.01.15
  }
}
```

## рҹ“Ҙ Beats вҖ” The Lightweight Shippers

In modern ELK setups, you often hear about **Beats**. Beats are **lightweight data shippers** that you install on your servers to send logs/metrics to Logstash or Elasticsearch.

The stack is sometimes called **ELK + Beats** or **Elastic Stack**.

```
Your Servers
   в”ңв”Җв”Җ Server 1: Filebeat installed вҶ’ sends logs вҶ’
   в”ңв”Җв”Җ Server 2: Filebeat installed вҶ’ sends logs вҶ’ вҶ’ Logstash вҶ’ Elasticsearch вҶ’ Kibana
   в”ңв”Җв”Җ Server 3: Filebeat installed вҶ’ sends logs вҶ’
   в””в”Җв”Җ Server 4: Metricbeat installed вҶ’ sends metrics вҶ’
```

**Types of Beats:**

| Beat | What It Ships | Use Case |
|------|--------------|----------|
| **Filebeat** | Log files | Ship Nginx, application, syslog logs |
| **Metricbeat** | System & service metrics | CPU, memory, disk, Docker, Kubernetes metrics |
| **Packetbeat** | Network data | Monitor network traffic, HTTP requests |
| **Heartbeat** | Uptime data | Check if services are alive (health checks) |
| **Auditbeat** | Audit data | Monitor file integrity, user activity |
| **Winlogbeat** | Windows event logs | Ship Windows Event Logs |

**Why use Beats instead of Logstash directly?**

| | Logstash | Beats (Filebeat) |
|---|---------|-------------------|
| Resource usage | Heavy (~1GB RAM) | Very light (~50MB RAM) |
| Install on every server? | No (centralized) | Yes (runs on each server) |
| Data transformation | Yes (complex filters) | Basic (simple processing) |
| Role | Central processor | Lightweight shipper |

**Best Practice:** Install Filebeat on each server вҶ’ sends to Logstash (central) вҶ’ Logstash transforms вҶ’ sends to Elasticsearch.

**Filebeat Configuration:**

```yaml
# File: /etc/filebeat/filebeat.yml (on each server)

# в”Җв”Җв”Җ What logs to collect в”Җв”Җв”Җ
filebeat.inputs:
  # Collect Nginx access logs
  - type: filestream
    id: nginx-access
    enabled: true
    paths:
      - /var/log/nginx/access.log
    fields:
      log_type: nginx           # Tag this log so Logstash knows how to parse it
    fields_under_root: false

  # Collect application logs
  - type: filestream
    id: app-logs
    enabled: true
    paths:
      - /var/log/myapp/*.log
    fields:
      log_type: application
    fields_under_root: false
    # Handle multi-line logs (like Java stack traces)
    parsers:
      - multiline:
          type: pattern
          pattern: '^\d{4}-\d{2}-\d{2}'   # New log entry starts with a date
          negate: true                       # Lines NOT matching this pattern...
          match: after                       # ...are appended to the PREVIOUS line

# в”Җв”Җв”Җ Where to send logs в”Җв”Җв”Җ
output.logstash:
  hosts: ["logstash-server:5044"]
  # Send to Logstash for processing

# OR send directly to Elasticsearch (skipping Logstash):
# output.elasticsearch:
#   hosts: ["elasticsearch:9200"]
#   index: "filebeat-%{+yyyy.MM.dd}"

# в”Җв”Җв”Җ Logging в”Җв”Җв”Җ
logging.level: info
logging.to_files: true
logging.files:
  path: /var/log/filebeat
  name: filebeat.log
  keepfiles: 7
```

## рҹҸ—пёҸ ELK Stack Architecture вҖ” From Simple to Production

### Simple Setup (Dev/Testing):

```
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ              Your Single Server                   в”Ӯ
в”Ӯ                                                   в”Ӯ
в”Ӯ  App вҶ’ writes logs вҶ’ /var/log/myapp/app.log      в”Ӯ
в”Ӯ                           вҶ“                       в”Ӯ
в”Ӯ                      Filebeat                     в”Ӯ
в”Ӯ                           вҶ“                       в”Ӯ
в”Ӯ                      Logstash                     в”Ӯ
в”Ӯ                           вҶ“                       в”Ӯ
в”Ӯ                    Elasticsearch                  в”Ӯ
в”Ӯ                           вҶ“                       в”Ӯ
в”Ӯ                       Kibana                      в”Ӯ
```

## рҹ“Ҡ Index Lifecycle Management (ILM)

Logs can eat up disk space FAST. You don't want to keep 2 years of logs at full speed. **ILM** lets you automatically manage the lifecycle of your indices:

```
Hot Phase    вҶ’ Actively writing & searching (fast SSD storage)
  вҶ“ (after 1 day)
Warm Phase   вҶ’ Still searchable but not writing new data (cheaper storage)
  вҶ“ (after 7 days)
Cold Phase   вҶ’ Rarely accessed, compressed (cheapest storage)
  вҶ“ (after 30 days)
Delete Phase вҶ’ Automatically deleted
```

**Setting up ILM via API:**

```bash
# Create an ILM policy
curl -X PUT "http://localhost:9200/_ilm/policy/logs-lifecycle" \
  -H "Content-Type: application/json" \
  -d '{
    "policy": {
      "phases": {
        "hot": {
          "min_age": "0ms",
          "actions": {
            "rollover": {
              "max_size": "10gb",
              "max_age": "1d"
            }
          }
        },
        "warm": {
          "min_age": "2d",
          "actions": {
            "shrink": {
              "number_of_shards": 1
            },
            "forcemerge": {
              "max_num_segments": 1
            }
          }
        },
        "cold": {
          "min_age": "7d",
          "actions": {
            "freeze": {}
          }
        },
        "delete": {
          "min_age": "30d",
          "actions": {
            "delete": {}
          }
        }
      }
    }
  }'
```

**What each phase does:**
- **Hot** вҶ’ New logs go here. Fast storage. Once the index hits 10GB or 1 day, it "rolls over" to a new index.
- **Warm** вҶ’ After 2 days, the index moves here. Shrink to 1 shard, merge segments for efficiency.
- **Cold** вҶ’ After 7 days, freeze the index. Rarely searched.
- **Delete** вҶ’ After 30 days, automatically delete the index. рҹ—‘пёҸ

## рҹӣЎпёҸ ELK Stack Security (Production Must-Haves)

In production, you **MUST** enable security:

```yaml
# In docker-compose.yml, for Elasticsearch:
environment:
  - xpack.security.enabled=true
  - ELASTIC_PASSWORD=your_secure_password
  # Enable TLS for inter-node communication
  - xpack.security.transport.ssl.enabled=true
```

**Key security features:**

| Feature | What It Does |
|---------|-------------|
| **Authentication** | Username/password or API keys to access Elasticsearch |
| **Authorization (RBAC)** | Control who can read/write to which indices |
| **TLS/SSL** | Encrypt communication between nodes and clients |
| **Audit Logging** | Track who did what |
| **IP Filtering** | Restrict access by IP address |

## рҹҗӣ Common Problems & Troubleshooting

| Problem | Cause | Solution |
|---------|-------|----------|
| Elasticsearch won't start | Not enough memory | Increase `ES_JAVA_OPTS` and system RAM |
| `max virtual memory areas too low` | Linux kernel setting | Run: `sysctl -w vm.max_map_count=262144` |
| Kibana shows "No results found" | Index pattern not set up | Go to Kibana вҶ’ Stack Management вҶ’ Index Patterns вҶ’ Create pattern |
| Logstash not parsing logs | Grok pattern doesn't match | Use the Grok Debugger in Kibana Dev Tools |
| Disk space filling up fast | No ILM policy | Set up Index Lifecycle Management (see above) |
| Slow searches | Too many shards | Consolidate small indices, use ILM |
| Beats can't connect | Network/firewall issue | Check ports 5044 (Beats), 9200 (ES), 5601 (Kibana) |

## рҹҺҜ Key Takeaways

1. **ELK Stack = Elasticsearch + Logstash + Kibana** вҖ” the gold standard for centralized logging.
2. **Elasticsearch** stores and searches вҖ” it's the heart of the stack.
3. **Logstash** collects and transforms вҖ” it's the pipeline that cleans up your messy logs.
4. **Kibana** visualizes вҖ” it's the dashboard where you see everything.
5. **Beats** (especially Filebeat) are lightweight shippers that go on each server.
6. Modern setup: **Filebeat вҶ’ Logstash вҶ’ Elasticsearch вҶ’ Kibana**.
7. In production: enable **security**, set up **ILM** for storage management, and run Elasticsearch as a **cluster**.
8. ELK is **resource-heavy** вҖ” plan for at least 4-8GB RAM minimum, more for production.
