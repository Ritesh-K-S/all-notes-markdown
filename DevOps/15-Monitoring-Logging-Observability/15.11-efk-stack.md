# 15.11 — EFK Stack — Elasticsearch, Fluentd/Fluent Bit, Kibana

> **One Line Answer:** The EFK Stack replaces **Logstash** with **Fluentd** (or its lighter sibling **Fluent Bit**) for log collection — it's **the preferred logging stack in Kubernetes** because Fluentd is a CNCF graduated project, uses less resources, and is built for cloud-native environments.

---

## 🤔 Wait, Didn't We Just Learn ELK? Why EFK?

Great question! Let's be real — ELK is fantastic. But it has one problem in modern cloud-native/Kubernetes environments:

**Logstash is heavy.** 🏋️

```
Logstash:
  - Written in Java (JRuby)
  - Needs ~1GB RAM minimum
  - One centralized instance
  - Not natively designed for Kubernetes

Fluentd:
  - Written in C + Ruby
  - Needs ~40-100MB RAM
  - Runs as a DaemonSet on every K8s node
  - CNCF Graduated project (same level as Kubernetes itself!)
  - 500+ plugins
  - Built for cloud-native

Fluent Bit:
  - Written in pure C
  - Needs ~5-15MB RAM (!!!)
  - Ultra-lightweight — perfect for edge/IoT/containers
  - Created by the same team as Fluentd
  - Often used as a "forwarder" that sends to Fluentd
```

**So the modern cloud-native logging stack is:**

```
ELK  = Elasticsearch + Logstash   + Kibana   ← Traditional
EFK  = Elasticsearch + Fluentd    + Kibana   ← Cloud-Native
ELFK = Elasticsearch + Fluent Bit + Fluentd + Kibana  ← Best of both worlds
```

---

## 📦 What is Fluentd?

**Fluentd** is an open-source **unified logging layer** that collects logs from multiple sources, processes them, and sends them to multiple destinations.

Think of Fluentd as a **universal adapter** for logs:

```
                         ┌──────────┐
  Apache logs     ──────→│          │──────→ Elasticsearch
  Nginx logs      ──────→│          │──────→ S3
  App logs        ──────→│ Fluentd  │──────→ Kafka
  Docker logs     ──────→│          │──────→ Datadog
  Syslog          ──────→│          │──────→ CloudWatch
  Kubernetes logs ──────→│          │──────→ Splunk
                         └──────────┘
                    "One ring to rule them all"
```

No matter where your logs come from or where they need to go — Fluentd handles it.

## 🆚 Fluentd vs Fluent Bit — What's the Difference?

This is one of the most common questions, so let's clear it up:

| Feature | Fluentd | Fluent Bit |
|---------|---------|------------|
| **Written in** | C + Ruby | Pure C |
| **Memory footprint** | ~40-100MB | ~5-15MB |
| **Plugins** | 500+ plugins | ~100 built-in plugins |
| **Plugin language** | Ruby (easy to write custom plugins) | C (harder but faster) |
| **Use case** | Central aggregator, complex processing | Lightweight shipper/forwarder |
| **Buffering** | Memory + file-based | Memory + file-based |
| **Best for** | Heavy processing, complex routing | Edge, IoT, sidecar, K8s DaemonSet |
| **Container image size** | ~150MB | ~25MB |

**The common pattern in Kubernetes:**

```
┌────────────────────────────────────────────────────────┐
│                    Kubernetes Cluster                    │
│                                                         │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐             │
│  │  Node 1  │  │  Node 2  │  │  Node 3  │             │
│  │          │  │          │  │          │             │
│  │ Pod  Pod │  │ Pod  Pod │  │ Pod  Pod │             │
│  │  ↓    ↓  │  │  ↓    ↓  │  │  ↓    ↓  │             │
│  │Fluent Bit│  │Fluent Bit│  │Fluent Bit│  ← DaemonSet│
│  └────┬─────┘  └────┬─────┘  └────┬─────┘  (on every  │
│       │              │              │         node)      │
│       └──────────────┼──────────────┘                   │
│                      ↓                                   │
│              ┌──────────────┐                            │
│              │   Fluentd    │  ← Central Aggregator     │
│              │ (Deployment) │     (optional)            │
│              └──────┬───────┘                            │
│                     ↓                                    │
│             ┌───────────────┐                            │
│             │ Elasticsearch │                            │
│             └───────┬───────┘                            │
│                     ↓                                    │
│              ┌──────────┐                                │
│              │  Kibana   │                               │
│              └──────────┘                                │
└────────────────────────────────────────────────────────┘
```

**Why this pattern?**
- **Fluent Bit** on every node = lightweight, collects all container logs
- **Fluentd** as aggregator = does the heavy processing, routing, filtering
- **Saves resources** compared to running full Fluentd on every node

**Or the simpler pattern (Fluent Bit directly to Elasticsearch):**

```
Fluent Bit (DaemonSet) → Elasticsearch → Kibana
```

This works great for smaller clusters where you don't need complex log processing.

## 🔧 Fluentd Architecture — How It Works

Fluentd uses a **tag-based routing system**. Every log event has a **tag**, and Fluentd routes events based on their tags.

```
┌──────────────────────────────────────────────┐
│                   Fluentd                     │
│                                               │
│  ┌─────────┐   ┌─────────┐   ┌─────────┐    │
│  │  INPUT   │ → │ FILTER  │ → │ OUTPUT  │    │
│  │(source)  │   │(process)│   │(match)  │    │
│  └─────────┘   └─────────┘   └─────────┘    │
│                                               │
│  Tag: "nginx.access"                          │
│  Tag: "app.payment-service"                   │
│  Tag: "kubernetes.pod.myapp"                  │
│                                               │
│  Routes events based on tag patterns          │
└──────────────────────────────────────────────┘
```

**Fluentd Directives:**

| Directive | Purpose | Like |
|-----------|---------|------|
| `<source>` | Where to collect logs FROM (input) | Logstash's `input {}` |
| `<filter>` | How to TRANSFORM logs | Logstash's `filter {}` |
| `<match>` | Where to SEND logs (output) | Logstash's `output {}` |
| `<label>` | Group routes together (like sub-pipelines) | Logstash's pipeline-to-pipeline |
| `<system>` | System-wide settings | Logstash's `logstash.yml` |

## 📝 Fluentd Configuration — Step by Step

### Basic Fluentd Config:

```xml
# File: /etc/fluentd/fluent.conf (or /fluentd/etc/fluent.conf in Docker)

# ─── STEP 1: INPUT — Where do logs come from? ───

# Collect logs from files (like Nginx)
<source>
  @type tail                          # "tail" means follow a file (like tail -f)
  path /var/log/nginx/access.log      # Which file to read
  pos_file /var/log/fluentd/nginx.pos # Remember where we left off (survive restarts)
  tag nginx.access                    # Tag these events as "nginx.access"
  
  <parse>
    @type nginx                       # Use built-in Nginx log parser
  </parse>
</source>

# Collect logs from Docker containers
<source>
  @type tail
  path /var/lib/docker/containers/*/*.log
  pos_file /var/log/fluentd/docker.pos
  tag docker.*
  
  <parse>
    @type json                        # Docker logs are JSON by default
  </parse>
</source>

# Accept logs over HTTP (for applications to send directly)
<source>
  @type http
  port 9880
  tag app.http
</source>

# Accept syslog
<source>
  @type syslog
  port 5140
  tag system.syslog
</source>

# ─── STEP 2: FILTER — Transform/enrich the logs ───

# Add hostname to all nginx logs
<filter nginx.**>
  @type record_transformer
  <record>
    hostname "#{Socket.gethostname}"
    environment "production"
  </record>
</filter>

# Parse and structure application logs
<filter app.**>
  @type parser
  key_name log
  reserve_data true                   # Keep original fields
  <parse>
    @type regexp
    expression /^(?<timestamp>\d{4}-\d{2}-\d{2} \d{2}:\d{2}:\d{2}) (?<level>\w+) \[(?<service>[^\]]+)\] (?<message>.*)$/
  </parse>
</filter>

# Remove sensitive data (like passwords or tokens)
<filter **>
  @type record_transformer
  remove_keys password, secret_key, api_token
</filter>

# ─── STEP 3: OUTPUT — Where to send the processed logs ───

# Send Nginx logs to Elasticsearch
<match nginx.**>
  @type elasticsearch
  host elasticsearch
  port 9200
  logstash_format true                # Create daily indices (logstash-2025.01.15)
  logstash_prefix nginx-logs          # Index name prefix: nginx-logs-2025.01.15
  include_tag_key true                # Include the Fluentd tag in the document
  
  # Buffering settings (important for reliability!)
  <buffer>
    @type file                        # Buffer to disk (survive crashes)
    path /var/log/fluentd/buffer/nginx
    flush_interval 5s                 # Flush every 5 seconds
    chunk_limit_size 5MB              # Max chunk size
    total_limit_size 1GB              # Max total buffer size
    retry_max_interval 30s            # Retry interval if Elasticsearch is down
    retry_forever true                # Keep retrying (don't lose logs!)
  </buffer>
</match>

# Send application logs to Elasticsearch
<match app.**>
  @type elasticsearch
  host elasticsearch
  port 9200
  logstash_format true
  logstash_prefix app-logs
  
  <buffer>
    @type file
    path /var/log/fluentd/buffer/app
    flush_interval 5s
  </buffer>
</match>

# Catch-all — send everything else to Elasticsearch too
<match **>
  @type elasticsearch
  host elasticsearch
  port 9200
  logstash_format true
  logstash_prefix other-logs
  
  <buffer>
    @type file
    path /var/log/fluentd/buffer/other
    flush_interval 10s
  </buffer>
</match>
```

**Breaking down the config line by line:**

1. **`<source>`** — Defines WHERE logs come from
   - `@type tail` → follow a file (like running `tail -f`)
   - `path` → which file to read
   - `pos_file` → bookmark file (remembers last read position)
   - `tag` → labels the log events for routing

2. **`<filter>`** — Defines HOW to transform logs
   - Tag pattern (`nginx.**`) → apply this filter only to matching events
   - `record_transformer` → add/modify/delete fields
   - `parser` → re-parse a field into structured data

3. **`<match>`** — Defines WHERE to send logs
   - Tag pattern (`nginx.**`) → route matching events here
   - `@type elasticsearch` → send to Elasticsearch
   - `<buffer>` → how to buffer data before sending (prevents data loss)

## 📝 Fluent Bit Configuration

Fluent Bit uses a simpler config format (INI-style or YAML):

### Classic Format (INI-style):

```ini
# File: /etc/fluent-bit/fluent-bit.conf

# ─── SERVICE SETTINGS ───
[SERVICE]
    Flush        5              # Flush logs every 5 seconds
    Daemon       Off            # Run in foreground (important for containers)
    Log_Level    info           # Logging level: debug, info, warn, error
    Parsers_File parsers.conf   # Path to parsers file
    HTTP_Server  On             # Enable monitoring endpoint
    HTTP_Listen  0.0.0.0
    HTTP_Port    2020           # Monitoring port

# ─── INPUT: Collect container logs in Kubernetes ───
[INPUT]
    Name              tail
    Tag               kube.*
    Path              /var/log/containers/*.log
    Parser            docker         # Parse Docker JSON log format
    DB                /var/log/fluentbit_kube.db    # Position tracking database
    Mem_Buf_Limit     5MB           # Memory buffer limit per input
    Skip_Long_Lines   On            # Skip very long lines (prevent OOM)
    Refresh_Interval  10            # Check for new files every 10 seconds

# ─── INPUT: Collect system logs ───
[INPUT]
    Name              systemd
    Tag               host.*
    Systemd_Filter    _SYSTEMD_UNIT=kubelet.service
    Read_From_Tail    On

# ─── FILTER: Add Kubernetes metadata ───
[FILTER]
    Name                kubernetes
    Match               kube.*
    Kube_URL            https://kubernetes.default.svc:443
    Kube_CA_File        /var/run/secrets/kubernetes.io/serviceaccount/ca.crt
    Kube_Token_File     /var/run/secrets/kubernetes.io/serviceaccount/token
    Kube_Tag_Prefix     kube.var.log.containers.
    Merge_Log           On           # Merge parsed JSON log into main record
    K8S-Logging.Parser  On           # Allow pods to specify their own parser
    K8S-Logging.Exclude On           # Allow pods to exclude themselves from logging

# What this filter does:
# Takes a raw container log and enriches it with Kubernetes metadata:
# - Pod name
# - Namespace
# - Container name
# - Labels
# - Annotations
# This is SUPER useful for filtering logs by namespace, app, etc. in Kibana!

# ─── FILTER: Exclude health check logs (they're noisy!) ───
[FILTER]
    Name    grep
    Match   kube.*
    Exclude log /health|/readiness|/liveness

# ─── OUTPUT: Send to Elasticsearch ───
[OUTPUT]
    Name            es
    Match           *
    Host            elasticsearch
    Port            9200
    Logstash_Format On                    # Create daily indices
    Logstash_Prefix k8s-logs              # Index: k8s-logs-2025.01.15
    Replace_Dots    On                    # Replace dots in field names (ES doesn't like dots)
    Retry_Limit     False                 # Keep retrying forever
    Suppress_Type_Name On                 # Required for ES 8.x+
```

### YAML Format (Fluent Bit 2.x+):

```yaml
# File: /etc/fluent-bit/fluent-bit.yaml

service:
  flush: 5
  daemon: off
  log_level: info
  http_server: on
  http_listen: 0.0.0.0
  http_port: 2020

pipeline:
  inputs:
    - name: tail
      tag: kube.*
      path: /var/log/containers/*.log
      parser: docker
      db: /var/log/fluentbit_kube.db
      mem_buf_limit: 5MB
      skip_long_lines: on
      refresh_interval: 10

  filters:
    - name: kubernetes
      match: kube.*
      kube_url: https://kubernetes.default.svc:443
      kube_ca_file: /var/run/secrets/kubernetes.io/serviceaccount/ca.crt
      kube_token_file: /var/run/secrets/kubernetes.io/serviceaccount/token
      merge_log: on
      k8s-logging.parser: on
      k8s-logging.exclude: on

    - name: grep
      match: kube.*
      exclude: log /health|/readiness|/liveness

  outputs:
    - name: es
      match: "*"
      host: elasticsearch
      port: 9200
      logstash_format: on
      logstash_prefix: k8s-logs
      replace_dots: on
      retry_limit: false
      suppress_type_name: on
```

## 🚀 Complete EFK Stack on Kubernetes

This is the real-world setup that most teams use. Let's deploy the full EFK stack on Kubernetes:

### Step 1: Create a Namespace

```yaml
# File: namespace.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: logging
```

### Step 2: Deploy Elasticsearch

```yaml
# File: elasticsearch.yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: elasticsearch
  namespace: logging
spec:
  serviceName: elasticsearch
  replicas: 1                          # Use 3+ for production
  selector:
    matchLabels:
      app: elasticsearch
  template:
    metadata:
      labels:
        app: elasticsearch
    spec:
      containers:
        - name: elasticsearch
          image: docker.elastic.co/elasticsearch/elasticsearch:8.12.0
          ports:
            - containerPort: 9200
              name: rest
            - containerPort: 9300
              name: inter-node
          resources:
            requests:
              memory: "1Gi"
              cpu: "500m"
            limits:
              memory: "2Gi"
              cpu: "1000m"
          env:
            - name: discovery.type
              value: single-node
            - name: xpack.security.enabled
              value: "false"
            - name: ES_JAVA_OPTS
              value: "-Xms512m -Xmx512m"
          volumeMounts:
            - name: data
              mountPath: /usr/share/elasticsearch/data
      initContainers:
        # Elasticsearch needs this kernel setting
        - name: fix-permissions
          image: busybox
          command: ["sh", "-c", "chown -R 1000:1000 /usr/share/elasticsearch/data"]
          volumeMounts:
            - name: data
              mountPath: /usr/share/elasticsearch/data
        - name: increase-vm-max-map
          image: busybox
          command: ["sysctl", "-w", "vm.max_map_count=262144"]
          securityContext:
            privileged: true
  volumeClaimTemplates:
    - metadata:
        name: data
      spec:
        accessModes: ["ReadWriteOnce"]
        resources:
          requests:
            storage: 10Gi
---
# Elasticsearch Service
apiVersion: v1
kind: Service
metadata:
  name: elasticsearch
  namespace: logging
spec:
  selector:
    app: elasticsearch
  ports:
    - port: 9200
      name: rest
    - port: 9300
      name: inter-node
  clusterIP: None                      # Headless service for StatefulSet
```

### Step 3: Deploy Kibana

```yaml
# File: kibana.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: kibana
  namespace: logging
spec:
  replicas: 1
  selector:
    matchLabels:
      app: kibana

    metadata:
      labels:
        app: kibana
    spec:
      containers:
        - name: kibana
          image: docker.elastic.co/kibana/kibana:8.12.0
          ports:
            - containerPort: 5601
          env:
            - name: ELASTICSEARCH_HOSTS
              value: "http://elasticsearch:9200"
          resources:
            requests:
              memory: "512Mi"
              cpu: "250m"
            limits:
              memory: "1Gi"
              cpu: "500m"
---
apiVersion: v1
kind: Service
metadata:
  name: kibana
  namespace: logging
spec:
  selector:
    app: kibana
  ports:
    - port: 5601
      targetPort: 5601
  type: ClusterIP
  # Use LoadBalancer or Ingress for external access
```

### Step 4: Deploy Fluent Bit as a DaemonSet

```yaml
# File: fluent-bit.yaml

# ConfigMap with Fluent Bit configuration
apiVersion: v1
kind: ConfigMap
metadata:
  name: fluent-bit-config
  namespace: logging
data:
  fluent-bit.conf: |
    [SERVICE]
        Flush         5
        Log_Level     info
        Daemon        off
        Parsers_File  parsers.conf
        HTTP_Server   On
        HTTP_Listen   0.0.0.0
        HTTP_Port     2020

    [INPUT]
        Name              tail
        Tag               kube.*
        Path              /var/log/containers/*.log
        Parser            docker
        DB                /var/log/fluentbit_kube.db
        Mem_Buf_Limit     10MB
        Skip_Long_Lines   On
        Refresh_Interval  10

    [FILTER]
        Name                kubernetes
        Match               kube.*
        Kube_URL            https://kubernetes.default.svc:443
        Kube_CA_File        /var/run/secrets/kubernetes.io/serviceaccount/ca.crt
        Kube_Token_File     /var/run/secrets/kubernetes.io/serviceaccount/token
        Merge_Log           On
        K8S-Logging.Parser  On
        K8S-Logging.Exclude On

    [FILTER]
        Name    grep
        Match   kube.*
        Exclude log /health|/readiness|/liveness

    [OUTPUT]
        Name            es
        Match           *
        Host            elasticsearch.logging.svc.cluster.local
        Port            9200
        Logstash_Format On
        Logstash_Prefix k8s-logs
        Replace_Dots    On
        Retry_Limit     False
        Suppress_Type_Name On

  parsers.conf: |
    [PARSER]
        Name        docker
        Format      json
        Time_Key    time
        Time_Format %Y-%m-%dT%H:%M:%S.%L
        Time_Keep   On

    [PARSER]
        Name        syslog
        Format      regex
        Regex       ^\<(?<pri>[0-9]+)\>(?<time>[^ ]* {1,2}[^ ]* [^ ]*) (?<host>[^ ]*) (?<ident>[a-zA-Z0-9_\/\.\-]*)(?:\[(?<pid>[0-9]+)\])?(?:[^\:]*\:)? *(?<message>.*)$
        Time_Key    time
        Time_Format %b %d %H:%M:%S
---
# ServiceAccount for Fluent Bit (needs access to K8s API for metadata)
apiVersion: v1
kind: ServiceAccount
metadata:
  name: fluent-bit
  namespace: logging
---
# ClusterRole — what permissions Fluent Bit needs
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: fluent-bit-read
rules:
  - apiGroups: [""]
    resources:
      - namespaces
      - pods
    verbs: ["get", "list", "watch"]
---
# Bind the role to the service account
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: fluent-bit-read
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: fluent-bit-read
subjects:
  - kind: ServiceAccount
    name: fluent-bit
    namespace: logging
---
# The DaemonSet — Fluent Bit runs on EVERY node
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: fluent-bit
  namespace: logging
  labels:
    app: fluent-bit
spec:
  selector:
    matchLabels:
      app: fluent-bit
  template:
    metadata:
      labels:
        app: fluent-bit
    spec:
      serviceAccountName: fluent-bit
      tolerations:
        # Allow running on master nodes too
        - key: node-role.kubernetes.io/master
          operator: Exists
          effect: NoSchedule
        - key: node-role.kubernetes.io/control-plane
          operator: Exists
          effect: NoSchedule
      containers:
        - name: fluent-bit
          image: fluent/fluent-bit:2.2
          ports:
            - containerPort: 2020    # Monitoring
          resources:
            requests:
              memory: "64Mi"
              cpu: "50m"
            limits:
              memory: "128Mi"        # Only 128MB! Compare to Logstash's 1GB+
              cpu: "200m"
          volumeMounts:
            - name: varlog
              mountPath: /var/log
            - name: varlibdockercontainers
              mountPath: /var/lib/docker/containers
              readOnly: true
            - name: config
              mountPath: /fluent-bit/etc/
      volumes:
        - name: varlog
          hostPath:
            path: /var/log
        - name: varlibdockercontainers
          hostPath:
            path: /var/lib/docker/containers
        - name: config
          configMap:
            name: fluent-bit-config
```

**Deploy everything:**

```bash
# Apply all manifests
kubectl apply -f namespace.yaml
kubectl apply -f elasticsearch.yaml
kubectl apply -f kibana.yaml
kubectl apply -f fluent-bit.yaml

# Check if everything is running
kubectl get all -n logging

# Expected output:
# NAME                          READY   STATUS    RESTARTS   AGE
# pod/elasticsearch-0           1/1     Running   0          2m
# pod/kibana-xxxxxxx-xxxxx      1/1     Running   0          1m
# pod/fluent-bit-xxxxx          1/1     Running   0          1m   ← one per node
# pod/fluent-bit-yyyyy          1/1     Running   0          1m
# pod/fluent-bit-zzzzz          1/1     Running   0          1m

# Port-forward Kibana to access it
kubectl port-forward svc/kibana 5601:5601 -n logging

# Now open http://localhost:5601 in your browser!
```

## 🚀 Complete EFK Stack with Docker Compose (for Learning)

If you don't have a Kubernetes cluster, here's a Docker Compose setup:

```yaml
# File: docker-compose.yml

version: "3.8"

services:
  # ─── Elasticsearch ───
  elasticsearch:
    image: docker.elastic.co/elasticsearch/elasticsearch:8.12.0
    container_name: elasticsearch
    environment:
      - discovery.type=single-node
      - xpack.security.enabled=false
      - "ES_JAVA_OPTS=-Xms512m -Xmx512m"
    volumes:
      - es-data:/usr/share/elasticsearch/data
    ports:
      - "9200:9200"
    networks:
      - efk-network
    healthcheck:
      test: ["CMD-SHELL", "curl -f http://localhost:9200 || exit 1"]
      interval: 30s
      timeout: 10s
      retries: 5

  # ─── Fluentd ───
  fluentd:
    build:
      context: ./fluentd
      dockerfile: Dockerfile
    container_name: fluentd
    volumes:
      - ./fluentd/conf:/fluentd/etc
      # Mount Docker log directory to collect container logs:
      # - /var/lib/docker/containers:/var/lib/docker/containers:ro
    ports:
      - "24224:24224"       # Forward input (other containers send logs here)
      - "24224:24224/udp"   # UDP forward input
    networks:
      - efk-network
    depends_on:
      elasticsearch:
        condition: service_healthy

  # ─── Kibana ───
  kibana:
    image: docker.elastic.co/kibana/kibana:8.12.0
    container_name: kibana
    environment:
```

```xml
    retry_forever true
```

## 🏷️ Kubernetes Logging — How Container Logs Work

To understand EFK on Kubernetes, you need to know HOW Kubernetes handles logs:

```
┌─────────────────────────────────────────────────────┐
│                   Kubernetes Node                    │
│                                                      │
│  ┌────────────────┐                                  │
│  │     Pod         │                                  │
│  │  ┌───────────┐ │                                  │
│  │  │ Container │ │  → stdout/stderr                 │
│  │  └───────────┘ │       ↓                          │
│  └────────────────┘       │                          │
│                           ↓                          │
│  Container Runtime (containerd/Docker)               │
│         ↓                                            │
│  Writes to: /var/log/containers/<pod>_<ns>_<container>.log
│         ↓                                            │
│  Fluent Bit (DaemonSet) reads these files            │
│         ↓                                            │
│  Sends to Elasticsearch                              │
└─────────────────────────────────────────────────────┘
```

**Key insight:** Every container's `stdout` and `stderr` output gets written to a log file on the node. Fluent Bit/Fluentd just reads those files!

That's why:
- Your apps should log to **stdout/stderr** (not files inside the container)
- Fluent Bit needs access to **`/var/log`** on the host
- Fluent Bit runs as a **DaemonSet** (one on every node)

## ⚡ Fluentd vs Logstash — The Complete Comparison

| Feature | Logstash | Fluentd |
|---------|----------|---------|
| **Written in** | JRuby (Java) | C + Ruby |
| **Memory** | ~1GB+ | ~40-100MB |
| **Startup time** | Slow (JVM warmup) | Fast |
| **Config format** | Custom DSL (like Ruby) | XML-like (tags) |
| **Plugin ecosystem** | 200+ plugins | 500+ plugins |
| **Buffering** | In-memory (can lose data) | Memory + file (more reliable) |
| **Kubernetes native?** | Not really | Yes (CNCF project) |
| **Maintained by** | Elastic (company) | CNCF (community) |
| **Structured logging** | Good | Better (built-in tag routing) |
| **Community** | Large | Very large |
| **Best with** | Elastic Stack | Any backend (more flexible) |

**When to use what:**

- **Use Logstash** → if you're already fully invested in the Elastic ecosystem and need complex event processing
- **Use Fluentd** → if you're in Kubernetes, need multi-destination output, or want lower resource usage
- **Use Fluent Bit** → if you need an ultra-lightweight shipper on every node (K8s DaemonSet)

## 🐛 Common EFK Troubleshooting

| Problem | Cause | Solution |
|---------|-------|----------|
| Fluent Bit pods in CrashLoopBackOff | Config error | Check logs: `kubectl logs <pod> -n logging` |
| No logs in Kibana | Index pattern not created | Create index pattern in Kibana (e.g., `k8s-logs-*`) |
| Logs missing Kubernetes metadata | RBAC not set up | Ensure ServiceAccount + ClusterRole + Binding exist |
| "Connection refused" to Elasticsearch | ES not ready or wrong host | Check ES service name and port, wait for ES to be healthy |
| Buffer overflow warnings | ES can't keep up | Increase buffer size, add more ES nodes, or reduce log volume |
| Duplicate logs | Multiple parsers matching | Check your filter/match patterns for overlaps |
| Missing logs from some pods | Fluent Bit not on that node | Ensure DaemonSet tolerations allow all nodes |

## 🎯 Key Takeaways

1. **EFK = Elasticsearch + Fluentd/Fluent Bit + Kibana** — the cloud-native alternative to ELK.
2. **Fluentd** replaces Logstash — it's lighter, more flexible, and a CNCF graduated project.
3. **Fluent Bit** is even lighter than Fluentd — perfect as a DaemonSet on every Kubernetes node.
4. Common pattern: **Fluent Bit (shipper) → Fluentd (aggregator) → Elasticsearch → Kibana**.
5. In Kubernetes, container logs go to **`/var/log/containers/`** — Fluent Bit just reads them.
6. Fluentd uses **tag-based routing** — every log event gets a tag, and you route based on tags.
7. **Apps should log to stdout/stderr** — not to files inside the container.
8. Fluent Bit needs proper **RBAC** in Kubernetes to access pod metadata.
