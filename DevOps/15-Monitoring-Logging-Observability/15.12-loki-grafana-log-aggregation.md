# 15.12 — Loki + Grafana — Log Aggregation (Lightweight Alternative to ELK)

> **One Line Answer:** Loki is a **log aggregation system inspired by Prometheus** — instead of indexing the full text of every log line (like Elasticsearch does), Loki only indexes **labels/metadata**, making it **massively cheaper, simpler, and faster** while still letting you search your logs through Grafana.

---

## 🤔 Let's Start With a Simple Story

Imagine you have a **library with 10 million books** (logs).

**The Elasticsearch Way (ELK/EFK):**
- Read EVERY book, word by word 📖
- Create an index of EVERY word in EVERY book 📚
- Store the index (which is almost as big as the books themselves!) 💾
- Now you can search for any word instantly ⚡
- **Problem:** The index is HUGE, needs tons of RAM and disk 💰💰💰

**The Loki Way:**
- DON'T read the books 📖❌
- Just look at the **cover** of each book — the title, author, genre, year 🏷️
- Create an index of ONLY those labels
- When someone asks "find books about dragons" → first narrow down by genre (Fantasy), then scan only those books
- **Result:** Index is TINY, uses way less resources 🎯

**The tradeoff?** Searching for a specific word inside a book is slower (because you have to scan the actual book). But narrowing down BY labels first makes it totally manageable.

**In real terms:**

```
ELK:  "Show me all logs containing 'payment timeout'" 
      → Elasticsearch searches its massive index → instant result
      → But storing that index costs $$$ (RAM + disk)

Loki: "Show me all logs from service=payment-service, level=error"
      → Loki finds matching streams by labels → instant
      → Then scans those log lines for "timeout" → fast enough!
      → Index is tiny → costs pennies compared to ELK
```

## 📦 What is Loki?

**Grafana Loki** is a horizontally scalable, highly available, multi-tenant **log aggregation system** created by **Grafana Labs** (the same team that makes Grafana).

It was announced in 2018 and its design philosophy is:

> **"Like Prometheus, but for logs."**

Just like Prometheus stores metrics with labels, Loki stores logs with labels. That's the key insight.

### The Loki Stack (also called PLG Stack):

| Component | Role | Analogy |
|-----------|------|---------|
| **Promtail** (or Grafana Agent) | Collects and ships logs to Loki | Like **Filebeat** in ELK |
| **Loki** | Stores and indexes logs (by labels only) | Like **Elasticsearch** in ELK (but way lighter) |
| **Grafana** | Visualizes and queries logs | Like **Kibana** in ELK |

```
PLG Stack:   Promtail → Loki → Grafana
ELK Stack:   Filebeat → Elasticsearch → Kibana
```

## ⚡ Why Loki Over ELK? — The Big Differences

| Feature | ELK (Elasticsearch) | Loki |
|---------|---------------------|------|
| **What gets indexed** | EVERYTHING (full text of every log) | ONLY labels (metadata) |
| **Index size** | Huge (often bigger than raw logs!) | Tiny (just labels) |
| **Storage cost** | Expensive (fast SSD needed) | Cheap (uses object storage like S3!) |
| **RAM needed** | 4-16GB+ per node | 256MB-2GB |
| **Operational complexity** | High (tune shards, replicas, JVM) | Low (single binary, or microservices) |
| **Full-text search speed** | Lightning fast ⚡ | Slower (scans log content) |
| **Label-based filtering** | Possible but not the primary design | Core design — blazing fast |
| **Multi-tenancy** | Possible with effort | Built-in |
| **Storage backend** | Its own (expensive) | S3, GCS, Azure Blob, local filesystem |
| **Setup difficulty** | Medium-Hard | Easy |
| **Best for** | Complex log analysis, SIEM, compliance | Cost-effective log aggregation, Kubernetes |

**The bottom line:** If you need Google-like full-text search across billions of logs, use Elasticsearch. If you need a cost-effective way to store and query logs (which is 90% of use cases), use Loki.

## 🏗️ How Loki Works — The Architecture

### The Key Insight: Labels, Not Content

In Loki, every log line belongs to a **stream**. A stream is defined by its **labels** (key-value pairs):

```
Stream 1: {job="nginx", host="server-01", env="production"}
  → "192.168.1.1 - GET /api/users 200 45ms"
  → "192.168.1.2 - POST /api/orders 201 120ms"
  → "10.0.0.5 - GET /api/users 500 2000ms"

Stream 2: {job="payment-service", host="server-02", env="production"}
  → "2025-01-15 10:30:00 INFO Processing payment #12345"
  → "2025-01-15 10:30:05 ERROR Payment gateway timeout"
  → "2025-01-15 10:30:10 INFO Retrying payment #12345"

Stream 3: {job="nginx", host="server-01", env="staging"}
  → "192.168.1.1 - GET /test 200 5ms"
```

**What Loki indexes:** `{job="nginx", host="server-01", env="production"}` ← just these labels

**What Loki does NOT index:** The actual log content (the text of each line)

When you query, you first select streams by labels, then optionally filter/search within those streams.

### Loki Architecture Components:

```
┌───────────────────────────────────────────────────────┐
│                      Loki                              │
│                                                        │
│  ┌────────────┐  ┌────────────┐  ┌────────────┐      │
│  │ Distributor │  │  Ingester  │  │   Querier  │      │
│  │             │  │            │  │            │      │
│  │ Receives    │  │ Writes to  │  │ Reads from │      │
│  │ log data    │→ │ chunks &   │  │ storage &  │      │
│  │ from agents │  │ WAL        │  │ ingesters  │      │
│  └────────────┘  └──────┬─────┘  └──────┬─────┘      │
│                         │               │              │
│                    ┌────┴───────────────┴────┐        │
│                    │     Storage Backend      │        │
│                    │  ┌─────────┐ ┌────────┐ │        │
│                    │  │  Index  │ │ Chunks │ │        │
│                    │  │(BoltDB, │ │ (S3,   │ │        │
│                    │  │ DynamoDB│ │  GCS,  │ │        │
│                    │  │  etc.)  │ │  etc.) │ │        │
│                    │  └─────────┘ └────────┘ │        │
│                    └────────────────────────┘        │
└───────────────────────────────────────────────────────┘
```

| Component | What It Does |
|-----------|-------------|
| **Distributor** | Receives incoming log streams, validates them, and forwards to Ingesters |
| **Ingester** | Batches logs into "chunks", compresses them, and writes to storage |
| **Querier** | Handles read queries — fetches data from both Ingesters (recent) and storage (older) |
| **Index** | Stores the mapping of labels → chunks (small, just metadata) |
| **Chunks** | The actual compressed log data (stored cheaply in S3/GCS/etc.) |

**The magic:** Because only labels are indexed, the index is tiny. The actual log data is compressed and stored in cheap object storage (like S3 at $0.023/GB/month!).

## 🔧 Promtail — The Log Collector

**Promtail** is the official agent that ships logs to Loki. It's designed to be paired with Loki like Filebeat is paired with Elasticsearch.

**How Promtail works:**

```
Application → writes to stdout/stderr
    ↓
Container Runtime → writes to /var/log/containers/*.log
    ↓
Promtail (DaemonSet)
    ├── Discovers log files
    ├── Reads them (tail -f style)
    ├── Attaches labels (from K8s metadata)
    ├── Optionally parses/transforms
    └── Pushes to Loki via HTTP API
```

### Promtail Configuration:

```yaml
# File: /etc/promtail/promtail.yaml

# ─── Server settings ───
server:
  http_listen_port: 9080     # Promtail's own metrics/health endpoint
  grpc_listen_port: 0        # Disabled

# ─── Where to store position tracking ───
positions:
  filename: /tmp/positions.yaml
  # This file tracks where Promtail left off in each log file
  # So it doesn't re-read old logs after a restart

# ─── Where to send logs ───
clients:
  - url: http://loki:3100/loki/api/v1/push
    # This is the Loki push API endpoint

# ─── What logs to collect ───
scrape_configs:
  # ─── Job 1: Collect Kubernetes pod logs ───
  - job_name: kubernetes-pods
    kubernetes_sd_configs:
      - role: pod
      # Auto-discover all pods in the cluster!
      # Promtail uses the Kubernetes API to find pods

    relabel_configs:
      # Extract the pod name as a label
      - source_labels: ['__meta_kubernetes_pod_name']
        target_label: 'pod'

      # Extract the namespace as a label
      - source_labels: ['__meta_kubernetes_namespace']
        target_label: 'namespace'

      # Extract the container name as a label
      - source_labels: ['__meta_kubernetes_pod_container_name']
        target_label: 'container'

      # Extract specific pod labels (like app name)
      - source_labels: ['__meta_kubernetes_pod_label_app']
        target_label: 'app'

      # Set the log file path
      - replacement: /var/log/pods/*$1/*.log
        separator: /
        source_labels:
          - __meta_kubernetes_pod_uid
          - __meta_kubernetes_pod_container_name
        target_label: __path__

    pipeline_stages:
      # ─── Parse Docker/CRI log format ───
      - cri: {}
      # Container Runtime Interface (CRI) logs look like:
      # 2025-01-15T10:30:00.123456789Z stdout F actual log message here
      # cri stage parses this and extracts the actual message

      # ─── Parse JSON logs (if your app logs in JSON) ───
      - match:
          selector: '{app="payment-service"}'
          stages:
            - json:
                expressions:
                  level: level
                  msg: message
                  caller: caller
            - labels:
                level:
            # Now you can query: {app="payment-service", level="error"}

      # ─── Extract log level from plain text logs ───
      - match:
          selector: '{app="nginx"}'
          stages:
            - regex:
                expression: '(?P<status_code>\d{3})'
            - labels:
                status_code:
            # Now you can query: {app="nginx", status_code="500"}

  # ─── Job 2: Collect system logs ───
  - job_name: system-logs
    static_configs:
      - targets:
          - localhost
        labels:
          job: syslog
          host: server-01
          __path__: /var/log/syslog

  # ─── Job 3: Collect Nginx access logs ───
  - job_name: nginx
    static_configs:
      - targets:
          - localhost
        labels:
          job: nginx
          env: production
          __path__: /var/log/nginx/access.log

    pipeline_stages:
      - regex:
          expression: '^(?P<ip>\S+) \S+ \S+ \[(?P<timestamp>[^\]]+)\] "(?P<method>\S+) (?P<path>\S+) \S+" (?P<status>\d+) (?P<size>\d+)'
      - labels:
          method:
          status:
      # Now you can query:
      # {job="nginx", method="POST"} — all POST requests
      # {job="nginx", status="500"} — all 500 errors
```

**What each section does:**

1. **`server`** → Promtail's own HTTP server (for health checks and metrics)
2. **`positions`** → Tracks read positions in log files (like Filebeat's registry)
3. **`clients`** → Where to send logs (Loki's push endpoint)
4. **`scrape_configs`** → What to collect and how to label it
5. **`pipeline_stages`** → Parse and transform log lines (extract labels, parse JSON, regex, etc.)

## 📊 LogQL — Loki's Query Language

LogQL is Loki's query language. If you know **PromQL** (Prometheus), LogQL will feel very familiar. If not, don't worry — it's simple!

### Two Types of Queries:

1. **Log Queries** → Return actual log lines
2. **Metric Queries** → Return numbers (counts, rates, etc.)

### Log Queries:

```logql
# ─── BASIC: Select logs by labels ───

# All logs from the nginx job
{job="nginx"}

# All logs from the payment-service in production
{app="payment-service", env="production"}

# All logs from any service in the "backend" namespace
{namespace="backend"}

# Regex matching — all services starting with "payment"
{app=~"payment.*"}

# Negative match — everything EXCEPT the health-check service
{app!="health-check"}

# Regex negative match — exclude all test services
{app!~".*test.*"}
```

```logql
# ─── FILTERING: Search within selected streams ───

# Find ERROR logs from payment-service
{app="payment-service"} |= "ERROR"
# |= means "line contains" (case-sensitive)

# Find logs NOT containing "health"
{app="nginx"} != "health"
# != means "line does NOT contain"

# Regex filter — find lines matching a pattern
{app="nginx"} |~ "status=(4|5)\\d{2}"
# |~ means "line matches regex"

# Negative regex — exclude lines matching a pattern
{app="nginx"} !~ "GET /health"
# !~ means "line does NOT match regex"
```

```logql
# ─── PARSING: Extract fields from log lines ───

# Parse JSON logs and filter by a field
{app="payment-service"} | json | level="error"
# 1. Select streams with label app=payment-service
# 2. Parse each line as JSON
# 3. Filter where the "level" field equals "error"

# Parse with regex and filter
{job="nginx"} | regexp `(?P<ip>\S+) \S+ \S+ \[(?P<ts>[^\]]+)\] "(?P<method>\S+) (?P<path>\S+) \S+" (?P<status>\d+)` | status >= 400
# Extract fields with regex, then filter by status >= 400

# Parse log format and use extracted fields
{app="payment-service"} | json | line_format "{{.level}} — {{.message}}"
# Reformat the output to show only level and message

# Use logfmt parser (for key=value style logs)
# Example log: level=error msg="payment failed" order_id=12345 duration=2.5s
{app="payment-service"} | logfmt | level="error" | duration > 1s
# Parse key=value pairs, filter errors taking longer than 1 second
```

### Metric Queries (the real power!):

```logql
# ─── COUNT: How many logs? ───

# Count of ERROR logs per second from payment-service in the last 5 minutes
count_over_time({app="payment-service"} |= "ERROR" [5m])

# Rate of errors per second (smoothed)
rate({app="payment-service"} |= "ERROR" [5m])

# ─── AGGREGATION: Group and summarize ───

# Total error count per app in the last hour
sum(count_over_time({env="production"} |= "ERROR" [1h])) by (app)
# Groups by the "app" label — shows which app has the most errors

# Top 5 apps by error count
topk(5, sum(count_over_time({env="production"} |= "ERROR" [1h])) by (app))

# Error rate per namespace
sum(rate({env="production"} |= "ERROR" [5m])) by (namespace)

# ─── BYTES: How much log data? ───

# Bytes of logs per service in the last hour
sum(bytes_over_time({env="production"} [1h])) by (app)
# Useful for finding which service is generating the most logs!

# ─── QUANTILES: Latency from logs ───

# If your logs contain duration info:
# Example log: {"level":"info","method":"GET","path":"/api/users","duration_ms":45}
{app="api-gateway"} | json | unwrap duration_ms
# "unwrap" extracts a numeric value from a label

# P99 latency from logs
quantile_over_time(0.99, {app="api-gateway"} | json | unwrap duration_ms [5m]) by (path)
# 99th percentile response time per path — from logs!
```

### LogQL Cheat Sheet:

| Operator | Meaning | Example |
|----------|---------|---------|
| `{label="value"}` | Select by label | `{app="nginx"}` |
| `{label=~"regex"}` | Select by regex | `{app=~"pay.*"}` |
| `{label!="value"}` | Exclude label | `{app!="test"}` |
| `\|=` | Line contains | `\|= "ERROR"` |
| `!=` | Line doesn't contain | `!= "health"` |
| `\|~` | Line matches regex | `\|~ "5\\d{2}"` |
| `!~` | Line doesn't match regex | `!~ "GET /health"` |
| `\| json` | Parse as JSON | `\| json \| level="error"` |
| `\| logfmt` | Parse key=value | `\| logfmt \| duration > 1s` |
| `\| regexp` | Parse with regex | `\| regexp \`(?P<ip>\\S+)\`` |
| `\| line_format` | Reformat output | `\| line_format "{{.level}}: {{.msg}}"` |
| `count_over_time()` | Count log lines | `count_over_time({app="x"} [5m])` |
| `rate()` | Logs per second | `rate({app="x"} [5m])` |
| `sum() by ()` | Group and sum | `sum(...) by (app)` |
| `topk(n, ...)` | Top N | `topk(5, ...)` |

## 🚀 Complete Loki + Grafana Stack with Docker Compose

Here's a full setup to get started locally:

```yaml
# File: docker-compose.yml

version: "3.8"

services:
  # ─── Loki ───
  loki:
    image: grafana/loki:2.9.4
    container_name: loki
    ports:
      - "3100:3100"       # Loki API
    volumes:
      - ./loki-config.yaml:/etc/loki/local-config.yaml
      - loki-data:/loki
    command: -config.file=/etc/loki/local-config.yaml
    networks:
      - loki-network
    healthcheck:
      test: ["CMD-SHELL", "wget --no-verbose --tries=1 --spider http://localhost:3100/ready || exit 1"]
      interval: 15s
      timeout: 5s
      retries: 5

  # ─── Promtail ───
  promtail:
    image: grafana/promtail:2.9.4
    container_name: promtail
    volumes:
      - ./promtail-config.yaml:/etc/promtail/config.yaml
      - /var/log:/var/log:ro                          # Read system logs
      - /var/lib/docker/containers:/var/lib/docker/containers:ro  # Docker logs
    command: -config.file=/etc/promtail/config.yaml
    networks:
      - loki-network
    depends_on:
      loki:
        condition: service_healthy

  # ─── Grafana ───
  grafana:
    image: grafana/grafana:10.3.1
    container_name: grafana
    ports:
      - "3000:3000"       # Grafana web UI
    environment:
      - GF_SECURITY_ADMIN_USER=admin
      - GF_SECURITY_ADMIN_PASSWORD=admin
      - GF_AUTH_ANONYMOUS_ENABLED=true
      - GF_AUTH_ANONYMOUS_ORG_ROLE=Admin
    volumes:
      - grafana-data:/var/lib/grafana
      - ./grafana-datasources.yaml:/etc/grafana/provisioning/datasources/datasources.yaml
    networks:
      - loki-network
    depends_on:
      loki:
        condition: service_healthy

  # ─── Sample Apps (to generate logs) ───
  nginx:
    image: nginx:alpine
    container_name: sample-nginx
    ports:
      - "8080:80"
    networks:
      - loki-network

  log-generator:
    image: mingrammer/flog
    container_name: log-generator
    command: -f json -d 1s -l
    # Generates fake JSON logs every second — perfect for testing!
    networks:
      - loki-network

volumes:
  loki-data:
  grafana-data:

networks:
  loki-network:
    driver: bridge
```

### Loki Configuration:

```yaml
# File: loki-config.yaml

auth_enabled: false                    # Disable multi-tenancy for simplicity

server:
  http_listen_port: 3100               # API port

common:
  path_prefix: /loki
  storage:
    filesystem:
      chunks_directory: /loki/chunks   # Where to store log chunks
      rules_directory: /loki/rules     # Where to store alerting rules
  replication_factor: 1                # 1 for single instance (3 for production)
  ring:
    kvstore:
      store: inmemory                  # In-memory ring for single instance

schema_config:
  configs:
    - from: 2020-10-24                 # Schema version date
      store: tsdb                      # Use TSDB for index (recommended for Loki 2.8+)
      object_store: filesystem         # Store chunks on local filesystem
      schema: v12                      # Schema version
      index:
        prefix: index_
        period: 24h                    # Create a new index table every 24h

limits_config:
  reject_old_samples: true
  reject_old_samples_max_age: 168h     # Reject logs older than 7 days
  max_entries_limit_per_query: 5000    # Safety limit for queries
  ingestion_rate_mb: 10                # Max ingestion rate per tenant
  ingestion_burst_size_mb: 20          # Burst limit

# ─── Storage retention ───
table_manager:
  retention_deletes_enabled: true
  retention_period: 720h               # Keep logs for 30 days (720 hours)

# ─── Compactor (for storage optimization) ───
compactor:
  working_directory: /loki/compactor
  shared_store: filesystem
  retention_enabled: true
```

**What each section means:**
- **`auth_enabled: false`** → No tenant ID needed (for single-user setups)
- **`schema_config`** → Defines how data is stored — TSDB index + filesystem chunks
- **`limits_config`** → Prevents abuse — rate limits, old log rejection, query limits
- **`table_manager`** → Automatically deletes old logs after 30 days

### Promtail Configuration:

```yaml
# File: promtail-config.yaml
```

## 🚀 Loki on Kubernetes with Helm

The easiest way to deploy Loki on Kubernetes is using the official Helm chart:

```bash
# Add the Grafana Helm repository
helm repo add grafana https://grafana.github.io/helm-charts
helm repo update

# ─── Option 1: Loki Stack (all-in-one: Loki + Promtail + Grafana) ───
helm install loki grafana/loki-stack \
  --namespace logging \
  --create-namespace \
  --set grafana.enabled=true \
  --set prometheus.enabled=false \
  --set loki.persistence.enabled=true \
  --set loki.persistence.size=10Gi

# ─── Option 2: Loki (just Loki, bring your own Grafana) ───
helm install loki grafana/loki \
  --namespace logging \
  --create-namespace \
  --values loki-values.yaml

# Check everything is running
kubectl get all -n logging

# Port-forward Grafana
kubectl port-forward svc/loki-grafana 3000:80 -n logging

# Get the Grafana admin password
kubectl get secret loki-grafana -n logging -o jsonpath="{.data.admin-password}" | base64 --decode
```

**Custom Helm values for production:**

```yaml
# File: loki-values.yaml

loki:
  auth_enabled: false
  
  storage:
    type: s3                           # Use S3 for production!
    s3:
      endpoint: s3.amazonaws.com
      region: us-east-1
      bucketnames: my-loki-logs
      # access_key_id and secret_access_key via environment or IAM role
  
  limits_config:
    retention_period: 720h             # 30 days
    max_entries_limit_per_query: 5000
    ingestion_rate_mb: 20
    ingestion_burst_size_mb: 30

  schema_config:
    configs:
      - from: "2024-01-01"

          prefix: loki_index_
          period: 24h

# Promtail runs as DaemonSet on every node

  enabled: true
  config:
    clients:
      - url: http://loki-gateway/loki/api/v1/push

# Grafana

  enabled: true
  persistence:
    enabled: true
    size: 5Gi
```

## 🏗️ Loki Architecture — From Simple to Production

### Single Binary (Simple / Dev):

```
┌────────────────────────────────────────┐
│           Single Loki Binary            │
│                                         │
│  Distributor + Ingester + Querier       │
│  + Compactor + Index Gateway            │
│  (all in one process)                   │
│                                         │
│  Storage: Local Filesystem              │
│                                         │
│  Good for: Dev, testing, small scale    │
└────────────────────────────────────────┘
```

### Simple Scalable (Medium / Most teams):


## 📊 Grafana Dashboards for Loki

Here's how to build useful dashboards in Grafana using Loki:

### Dashboard Panel Examples:

**Panel 1: Log Volume Over Time**
```logql
# Query: Total log lines per minute
sum(count_over_time({job=~".+"} [1m])) by (job)

# Visualization: Time series (stacked bar)
```

**Panel 2: Error Rate by Service**
```logql
# Query: Error logs per minute per service
sum(rate({env="production"} |= "ERROR" [5m])) by (app)

# Visualization: Time series (line chart)
```

**Panel 3: Top 10 Error Messages**
```logql
# Query: Count errors grouped by message
topk(10, sum(count_over_time({env="production"} |= "ERROR" | json | keep message [1h])) by (message))

# Visualization: Table or Bar chart
```

**Panel 4: Log Volume by Namespace (Kubernetes)**
```logql
# Query: Bytes ingested per namespace
sum(bytes_over_time({namespace=~".+"} [1h])) by (namespace)

# Visualization: Pie chart
```

**Panel 5: Live Log Tail**
```logql
# Query: Last logs from a specific service
{app="payment-service", env="production"}

# Visualization: Logs panel (shows raw logs in real-time)
```

### Setting Up Alerts in Grafana with Loki:

```yaml
# Example: Alert when error rate exceeds threshold

# In Grafana UI:
# 1. Create a new alert rule
# 2. Query: sum(rate({app="payment-service"} |= "ERROR" [5m]))
# 3. Condition: IS ABOVE 0.5 (more than 0.5 errors per second)
# 4. For: 5 minutes (sustained for 5 min before alerting)
# 5. Send to: Slack channel #alerts
```

Or define alerts as code using Loki's built-in ruler:

```yaml
# File: loki-alerts.yaml (loaded into Loki's ruler)

groups:
  - name: application-alerts
    rules:
      - alert: HighErrorRate
        expr: |
          sum(rate({app="payment-service"} |= "ERROR" [5m])) > 0.5
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "High error rate in payment-service"
          description: "Error rate is {{ $value }} errors/sec (threshold: 0.5)"

      - alert: NoLogsReceived
        expr: |
          absent_over_time({app="payment-service"} [15m])
        labels:
          severity: warning
        annotations:
          summary: "No logs received from payment-service for 15 minutes"
          description: "The service might be down or Promtail might be misconfigured"
```

## 🏷️ Labels Best Practices (Very Important!)

Labels are the **most important concept** in Loki. Getting them right is crucial for performance.

### ✅ DO:

```yaml
# Use static, low-cardinality labels
{job="nginx", env="production", region="us-east-1"}
{app="payment-service", namespace="backend"}

# These are GOOD labels because:
# - They have a small number of possible values
# - They don't change often
# - They're useful for filtering
```

### ❌ DON'T:

```yaml
# NEVER use high-cardinality labels!
{user_id="12345"}          # ← Millions of unique values!
{request_id="abc-def-123"} # ← Unique per request!
{ip="192.168.1.100"}       # ← Thousands of unique IPs!
{timestamp="2025-01-15"}   # ← Already handled by Loki's time index!

# WHY? Each unique label combination creates a new "stream"
# More streams = more index entries = more memory = slower queries = bad! 💀
```

### The Rule of Thumb:

```
A label should have at most a few hundred unique values.

Good labels: env (2-3 values), region (5-10), app (10-50), level (4-5)
Bad labels:  user_id (millions), request_id (billions), ip (thousands)
```

**If you need to search by user_id, use filter expressions instead:**

```logql
# ❌ BAD: user_id as a label
{app="api", user_id="12345"}

# ✅ GOOD: user_id in the log line, search with filter
{app="api"} |= "user_id=12345"
# OR with JSON parsing:
{app="api"} | json | user_id="12345"
```

## 🐛 Common Loki Troubleshooting

| Problem | Cause | Solution |
|---------|-------|----------|
| "entry too far behind" | Logs older than `reject_old_samples_max_age` | Increase the limit or fix the time on your servers |
| High memory usage | Too many unique label combinations | Reduce label cardinality (see best practices above) |
| Slow queries | Querying too much data without label filters | Always filter by labels FIRST, then by content |
| No logs in Grafana | Promtail not connected or wrong URL | Check Promtail logs, verify Loki URL |
| "stream limit exceeded" | Too many streams (too many unique label combos) | Reduce labels, increase `max_streams_per_user` |
| "per-tenant series limit exceeded" | Too many active streams | Same as above — reduce label cardinality |
| Gaps in logs | Promtail restarted and lost position | Ensure `positions.yaml` is persisted (use a volume!) |

## 🎯 Key Takeaways

1. **Loki indexes LABELS, not log content** — this is its superpower and the source of its efficiency.
2. **PLG Stack = Promtail + Loki + Grafana** — the lightweight alternative to ELK.
3. **Loki is "Prometheus for logs"** — same label-based approach, same ecosystem.
4. **Storage is cheap** — Loki stores compressed chunks in object storage (S3, GCS) instead of expensive Elasticsearch clusters.
5. **LogQL** is the query language — filter by labels first (`{app="x"}`), then by content (`|= "error"`).
6. **Label cardinality matters** — keep labels low-cardinality or you'll kill performance.
7. **Promtail** ships logs to Loki — runs as a DaemonSet on Kubernetes, similar to Filebeat.
8. **Best for 90% of teams** — unless you need complex full-text search or SIEM capabilities, Loki is simpler, cheaper, and easier to operate than ELK.
9. **Native Grafana integration** — Loki was built by the Grafana team, so dashboards and alerts work seamlessly.
10. In production, use **object storage (S3/GCS)** for chunks and run Loki in **Simple Scalable** or **Microservices** mode.

## 📚 Quick Reference — When to Use What?

```
Need full-text search across billions of logs?     → ELK
Running Kubernetes and want cloud-native logging?  → EFK (Fluent Bit + Elasticsearch)
Want cost-effective logging with Grafana?           → Loki + Grafana ✅ (most teams)
Already have Prometheus + Grafana?                  → Loki (same ecosystem, easy add)
Enterprise with big budget?                         → Splunk or Datadog
```
