# 15.13 — Structured Logging — Best Practices

> **One Line Answer:** Structured logging means writing your log messages in a **consistent, machine-readable format** (usually JSON) instead of random free-text strings — so both humans AND machines can easily search, filter, and analyze them.

---

## 🤔 What's Wrong With Normal Logging?

Let's say you have a web application and you're logging things like this:

```
User john logged in at 2025-06-15 10:32:45
Failed to process order 4521 for user alice - insufficient funds
Connection to database lost at 2025-06-15 10:33:01
User bob added item SKU-9921 to cart
```

Looks readable, right? Sure — **to your eyes**. But now imagine you have **10,000 servers**, each generating **millions of log lines per day**. You need to:

- Find all failed orders in the last hour
- Count how many users logged in today
- Filter logs by a specific user
- Alert when database connections drop

**Good luck parsing that with a script!** Every line has a **different format**. There's no consistency. You'd need complex regex patterns for every type of message.

This is called **unstructured logging** — and it's a nightmare at scale.

---

## ✅ Enter Structured Logging

Structured logging means every log entry follows a **consistent, predictable format** — typically **JSON** (but could also be key-value pairs, logfmt, etc.).

Here's the same information, but structured:

```json
{
  "timestamp": "2025-06-15T10:32:45.123Z",
  "level": "INFO",
  "message": "User logged in",
  "service": "auth-service",
  "user_id": "john",
  "source_ip": "192.168.1.42",
  "request_id": "req-abc-123"
}
```

```json
{
  "timestamp": "2025-06-15T10:32:47.456Z",
  "level": "ERROR",
  "message": "Failed to process order",
  "service": "order-service",
  "order_id": "4521",
  "user_id": "alice",
  "reason": "insufficient_funds",
  "request_id": "req-def-456"
}
```

```json
{
  "timestamp": "2025-06-15T10:33:01.789Z",
  "level": "CRITICAL",
  "message": "Database connection lost",
  "service": "order-service",
  "database": "orders-db",
  "host": "db-primary-01",
  "retry_count": 3,
  "request_id": "req-ghi-789"
}
```

### Why Is This Better?

| Aspect | Unstructured | Structured |
|--------|-------------|------------|
| **Readability by machines** | Hard — needs regex/parsing | Easy — just parse JSON |
| **Searchability** | Grep with complex patterns | Query by any field (`user_id = "alice"`) |
| **Filtering** | Painful | Instant (`level = "ERROR"`) |
| **Alerting** | Fragile | Reliable — based on specific fields |
| **Dashboards** | Nearly impossible to build | Easy — every field is a dimension |
| **Cross-service correlation** | Very hard | Easy — use `request_id` to trace across services |
| **Schema evolution** | Nothing to evolve | Add new fields without breaking anything |

---

## 📐 What Makes a Good Structured Log Entry?

Every structured log line should have some **standard fields** (also called a **log schema**). Here's a recommended set:

### Core Fields (Must Have)

| Field | Description | Example |
|-------|-------------|---------|
| `timestamp` | When the event happened (ISO 8601, UTC) | `"2025-06-15T10:32:45.123Z"` |
| `level` | Severity of the log | `"DEBUG"`, `"INFO"`, `"WARN"`, `"ERROR"`, `"FATAL"` |
| `message` | Human-readable description of the event | `"User logged in"` |
| `service` | Name of the microservice/application | `"auth-service"` |

### Context Fields (Should Have)

| Field | Description | Example |
|-------|-------------|---------|
| `request_id` / `trace_id` | Unique ID to correlate logs across services | `"req-abc-123"` |
| `user_id` | The user involved in the action | `"john"` |
| `host` / `hostname` | Which server/pod generated this log | `"web-pod-xyz"` |
| `environment` | Which environment (dev, staging, prod) | `"production"` |
| `version` | Application version | `"2.3.1"` |

### Error-Specific Fields (When Applicable)

| Field | Description | Example |
|-------|-------------|---------|
| `error_type` | The exception/error class | `"TimeoutException"` |
| `error_message` | Detailed error message | `"Connection timed out after 30s"` |
| `stack_trace` | Full stack trace (for ERROR/FATAL) | `"at com.example..."` |
| `error_code` | Application-specific error code | `"ERR_DB_TIMEOUT"` |

---

## 🔤 Log Levels — What They Mean and When to Use Them

Log levels tell you how **serious** a log entry is. Using the right level is crucial.

| Level | When to Use | Example |
|-------|-------------|---------|
| **TRACE** | Super detailed debugging (method entry/exit, variable values). Usually OFF in production. | `"Entering function calculateTax with amount=100"` |
| **DEBUG** | Useful information for developers during debugging. Usually OFF in production. | `"Database query returned 42 rows in 15ms"` |
| **INFO** | Normal operations — things that are supposed to happen. | `"User john logged in"`, `"Order 4521 placed successfully"` |
| **WARN** | Something unexpected happened, but the app can continue. Needs attention. | `"Disk usage at 85%"`, `"Retrying failed API call (attempt 2/3)"` |
| **ERROR** | Something failed. The current operation couldn't complete, but the app is still running. | `"Failed to send email to user alice"`, `"Payment processing failed"` |
| **FATAL** / **CRITICAL** | The application is going down. Immediate attention needed! | `"Out of memory — shutting down"`, `"Cannot connect to database after 10 retries"` |

### Golden Rules for Log Levels:

1. **Don't log everything as INFO** — it drowns out important logs
2. **Don't log everything as ERROR** — "the boy who cried wolf" problem (alert fatigue)
3. **WARN means "investigate soon"** — not right now, but don't ignore it
4. **ERROR means "something broke"** — a specific request/operation failed
5. **FATAL means "we're going down"** — the entire service is crashing

## 🏗️ Structured Logging in Different Languages

Let's see how to implement structured logging in popular languages used in DevOps.

### Python — Using `structlog`

```bash
pip install structlog
```

```python
import structlog
import logging

# Configure structlog for JSON output
structlog.configure(
    processors=[
        structlog.processors.TimeStamper(fmt="iso"),      # Add ISO timestamp
        structlog.processors.add_log_level,                # Add log level
        structlog.processors.JSONRenderer()                # Output as JSON
    ],
    wrapper_class=structlog.BoundLogger,
    context_class=dict,
    logger_factory=structlog.PrintLoggerFactory(),
)

# Create a logger
log = structlog.get_logger()

# Basic logging
log.info("User logged in", user_id="john", source_ip="192.168.1.42")
# Output: {"user_id": "john", "source_ip": "192.168.1.42", "timestamp": "2025-06-15T10:32:45.123456Z", "level": "info", "event": "User logged in"}

# Logging with bound context (context that sticks to the logger)
logger = log.bind(service="auth-service", environment="production")
logger.info("User logged in", user_id="john")
# Output: {"service": "auth-service", "environment": "production", "user_id": "john", ...}

# Error logging
try:
    result = 10 / 0
except ZeroDivisionError as e:
    log.error("Calculation failed", error_type=type(e).__name__, error_message=str(e))
# Output: {"error_type": "ZeroDivisionError", "error_message": "division by zero", "level": "error", "event": "Calculation failed", ...}
```

**What's happening here?**
- `structlog.configure()` — Sets up the log pipeline (what processors transform each log entry)
- `TimeStamper(fmt="iso")` — Automatically adds a timestamp to every log line
- `add_log_level` — Automatically adds the log level (`info`, `error`, etc.)
- `JSONRenderer()` — Outputs everything as JSON
- `log.bind()` — Creates a logger with "permanent" context fields (like `service` name) that appear in every log from that logger

### Python — Using Standard Library (`logging` + `json`)

If you don't want external libraries:

```python
import logging
import json
from datetime import datetime, timezone

class JSONFormatter(logging.Formatter):
    """Custom formatter that outputs logs as JSON."""
    
    def format(self, record):
        log_entry = {
            "timestamp": datetime.now(timezone.utc).isoformat(),
            "level": record.levelname,
            "message": record.getMessage(),
            "logger": record.name,
            "module": record.module,
            "function": record.funcName,
            "line": record.lineno,
        }
        
        # Add any extra fields passed to the log call
        if hasattr(record, "extra_fields"):
            log_entry.update(record.extra_fields)
        
        # Add exception info if present
        if record.exc_info and record.exc_info[0] is not None:
            log_entry["error_type"] = record.exc_info[0].__name__
            log_entry["error_message"] = str(record.exc_info[1])
            log_entry["stack_trace"] = self.formatException(record.exc_info)
        
        return json.dumps(log_entry)

# Setup
handler = logging.StreamHandler()
handler.setFormatter(JSONFormatter())

logger = logging.getLogger("my-app")
logger.addHandler(handler)
logger.setLevel(logging.INFO)

# Usage
logger.info("User logged in", extra={"extra_fields": {"user_id": "john", "source_ip": "192.168.1.42"}})
```

### Node.js — Using `winston`

```bash
npm install winston
```

```javascript
const winston = require('winston');

// Create a logger with JSON format
const logger = winston.createLogger({
  level: 'info',
  format: winston.format.combine(
    winston.format.timestamp({ format: 'YYYY-MM-DDTHH:mm:ss.SSSZ' }),  // ISO timestamp
    winston.format.json()  // Output as JSON
  ),
  defaultMeta: {
    service: 'order-service',         // Added to EVERY log line automatically
    environment: process.env.NODE_ENV  // e.g., "production"
  },
  transports: [
    new winston.transports.Console(),                         // Log to stdout
    new winston.transports.File({ filename: 'error.log', level: 'error' }),  // Errors to file
    new winston.transports.File({ filename: 'combined.log' })  // All logs to file
  ],
});

// Basic logging
logger.info('Order placed successfully', {
  order_id: '4521',
  user_id: 'alice',
  total_amount: 59.99,
  currency: 'USD'
});
// Output: {"level":"info","message":"Order placed successfully","order_id":"4521","user_id":"alice","total_amount":59.99,"currency":"USD","service":"order-service","timestamp":"2025-06-15T10:32:45.123Z"}

// Error logging
logger.error('Payment processing failed', {
  order_id: '4522',
  user_id: 'bob',
  error_type: 'PaymentGatewayError',
  error_message: 'Card declined',
  error_code: 'ERR_CARD_DECLINED'
});
```

**What's happening here?**
- `winston.format.combine()` — Chains multiple formatters together
- `winston.format.timestamp()` — Auto-adds timestamp to every log
- `winston.format.json()` — Outputs everything as JSON
- `defaultMeta` — Fields that appear in EVERY log line (like service name)
- `transports` — Where logs go (console, files, or external services)

```go
package main

import (
    "os"
    "github.com/rs/zerolog"
    "github.com/rs/zerolog/log"
)

func main() {
    // Configure global logger
    zerolog.TimeFieldFormat = zerolog.TimeFormatUnix

    // Create logger that writes JSON to stdout

        Timestamp().
        Str("service", "payment-service").
        Str("environment", "production").
        Logger()

        Str("user_id", "alice").
        Str("order_id", "4521").
```

## 📜 Popular Structured Log Formats

### 1. JSON (Most Popular)

```json
{"timestamp": "2025-06-15T10:32:45Z", "level": "INFO", "message": "User logged in", "user_id": "john"}
```

**Pros:** Universal, supported by every log system, easy to parse  
**Cons:** Verbose (bigger file size), slightly harder for humans to scan

### 2. Logfmt (Key-Value Pairs)

```
timestamp=2025-06-15T10:32:45Z level=INFO message="User logged in" user_id=john
```

**Pros:** More human-readable than JSON, lightweight  
**Cons:** Less standard, some tools don't natively support it  
**Used by:** Heroku, Grafana Loki

### 3. Common Event Format (CEF)

```
CEF:0|MyCompany|AuthService|1.0|LOGIN|User logged in|3|user_id=john src_ip=192.168.1.42
```

**Pros:** Security-industry standard  
**Cons:** Complex format, mainly for SIEM tools  
**Used by:** Security/SIEM tools (Splunk, ArcSight)

### Recommendation

**Use JSON for most cases.** It's the universal format. Every logging platform (ELK, Loki, Datadog, Splunk, CloudWatch) understands JSON natively.

If you're using **Grafana Loki**, `logfmt` is also a great choice because Loki has built-in logfmt parsing.

## 🏆 Structured Logging Best Practices

### 1. Always Use Consistent Field Names

❌ **Bad — Inconsistent naming across services:**

```json
// auth-service logs
{"userId": "john", "ts": "2025-06-15T10:32:45Z"}

// order-service logs  
{"user_name": "john", "timestamp": "Jun 15, 2025 10:32 AM"}

// payment-service logs
{"uid": "john", "time": "1718444565"}
```

✅ **Good — Same field names everywhere:**

```json
// All services use the same schema
{"user_id": "john", "timestamp": "2025-06-15T10:32:45.123Z", "level": "INFO"}
```

**Why?** When all services use the same field names, you can run a single query across ALL services:

```
user_id = "john" AND level = "ERROR"
```

### 2. Use ISO 8601 Timestamps in UTC

❌ **Bad:**
```json
{"time": "Jun 15, 2025 3:32 PM EST"}
{"time": "1718444565"}
{"time": "15/06/2025 10:32:45"}
```

✅ **Good:**
```json
{"timestamp": "2025-06-15T10:32:45.123Z"}
```

**Why?**
- ISO 8601 is universally parseable
- UTC avoids timezone confusion (your servers might be in different zones)
- Include milliseconds for precise ordering

### 3. Include a Correlation/Request ID

When a user makes a request to your system, that request might pass through 5-10 different microservices. How do you trace it?

**Generate a unique ID at the entry point (API gateway) and pass it through every service:**

```json
// API Gateway
{"request_id": "req-abc-123", "service": "api-gateway", "message": "Received request"}

// Auth Service
{"request_id": "req-abc-123", "service": "auth-service", "message": "User authenticated"}

// Order Service  
{"request_id": "req-abc-123", "service": "order-service", "message": "Order created"}

// Payment Service
{"request_id": "req-abc-123", "service": "payment-service", "message": "Payment processed"}
```

Now you can search `request_id = "req-abc-123"` and see the **entire journey** of that request across all services!

### 4. Don't Log Sensitive Data

**NEVER log these:**
- Passwords or password hashes
- API keys, tokens, secrets
- Credit card numbers
- Social Security Numbers / national IDs
- Personal health information
- Full email addresses (in some jurisdictions)

❌ **Bad:**
```json
{"message": "Login attempt", "user": "john", "password": "MyS3cretP@ss!"}
```

✅ **Good:**
```json
{"message": "Login attempt", "user_id": "john", "auth_result": "success"}
```

If you MUST log partial info for debugging:
```json
{"message": "Payment processed", "card_last_four": "4242", "card_type": "visa"}
```

### 5. Log at the Right Level

A very common mistake is logging too much at INFO or too much at ERROR.

```python
# ❌ Bad — this is DEBUG, not INFO
logger.info("Entering function calculateTax", tax_rate=0.08, subtotal=100.00)

# ✅ Good — INFO is for business events
logger.info("Order placed", order_id="4521", total=108.00)

# ❌ Bad — this is a WARN, not ERROR (the app recovered)
logger.error("Cache miss, fetching from database", key="user:john")

# ✅ Good — WARN for recoverable issues
logger.warning("Cache miss, fetching from database", key="user:john")

# ✅ Good — ERROR for actual failures
logger.error("Failed to process payment", order_id="4521", error="timeout")
```

### 6. Keep Messages Short and Static

The `message` field should be a **static string** (no variables baked in). Put variables in separate fields.

❌ **Bad:**
```json
{"message": "User john logged in from 192.168.1.42 at 2025-06-15"}
```

✅ **Good:**
```json
{
  "message": "User logged in",
  "user_id": "john",
  "source_ip": "192.168.1.42",
  "timestamp": "2025-06-15T10:32:45Z"
}
```

**Why?** Static messages are easy to group and count. You can answer "How many login events happened today?" by filtering on `message = "User logged in"`. If the message has variables baked in, every log line is "unique" and you can't aggregate.

### 7. Add Context, Not Just Events

Don't just log WHAT happened — log the CONTEXT around it.

❌ **Bare minimum:**
```json
{"message": "Request failed"}
```

✅ **Rich context:**
```json
{
  "message": "Request failed",
  "service": "order-service",
  "method": "POST",
  "path": "/api/v1/orders",
  "status_code": 503,
  "duration_ms": 30250,
  "error_type": "UpstreamTimeout",
  "upstream_service": "payment-service",
  "retry_count": 3,
  "request_id": "req-abc-123",
  "user_id": "alice"
}
```

With this level of detail, you can answer:
- Which service failed?
- What endpoint was called?
- How long did it take?
- What was the error?
- How many retries were attempted?
- Which user was affected?

### 8. Don't Log Too Much or Too Little

| Problem | Symptom | Solution |
|---------|---------|----------|
| **Too much logging** | Log storage costs explode, noise drowns out important logs | Use appropriate log levels, keep DEBUG/TRACE off in prod |
| **Too little logging** | Can't debug issues, "we have no idea what happened" | Log business events, errors, and key state transitions |
| **Just right** | You can diagnose any production issue within minutes | Follow the "if an incident happens, will this log help?" rule |

**Rule of thumb:** Before adding a log line, ask yourself:

> "If something goes wrong in production at 3 AM, will this log line help me figure out what happened?"

If yes → log it. If no → skip it.

## 🐳 Structured Logging in Docker & Kubernetes

### Docker Logging

By default, Docker captures everything written to `stdout` and `stderr` from your container. So if your app outputs JSON logs to stdout, Docker automatically stores them.

```bash
# View logs from a container
docker logs my-container

# View the raw JSON log file Docker stores
cat /var/lib/docker/containers/<container-id>/<container-id>-json.log
```

Docker wraps your log output in its own JSON:

```json
{
  "log": "{\"timestamp\":\"2025-06-15T10:32:45Z\",\"level\":\"INFO\",\"message\":\"User logged in\",\"user_id\":\"john\"}\n",
  "stream": "stdout",
  "time": "2025-06-15T10:32:45.123456789Z"
}
```

### Kubernetes Logging

In Kubernetes, logs from containers are stored on the node at:

```
/var/log/containers/<pod-name>_<namespace>_<container-name>-<container-id>.log
```

A log collector (like Fluentd, Fluent Bit, or Promtail) picks up these log files and ships them to your logging backend.

**Best practice:** Always log to `stdout`/`stderr` in containers — never to files inside the container. Let the platform (Docker/K8s) handle log collection.

```yaml
# Example: Fluent Bit DaemonSet config for parsing JSON logs
apiVersion: v1
kind: ConfigMap
metadata:
  name: fluent-bit-config
data:
  fluent-bit.conf: |
    [INPUT]
        Name              tail
        Path              /var/log/containers/*.log
        Parser            docker
        Tag               kube.*
        Refresh_Interval  5

    [FILTER]
        Name              kubernetes
        Match             kube.*
        Merge_Log         On          # This parses JSON log lines into structured fields
        Keep_Log          Off

    [OUTPUT]
        Name              es
        Match             *
        Host              elasticsearch
        Port              9200
        Index             logs
```

**What `Merge_Log On` does:** If your application outputs JSON to stdout, Fluent Bit will **parse** that JSON and merge the fields into the log record — so `user_id`, `level`, `message` etc. become top-level searchable fields in Elasticsearch!

## 🧭 Creating a Logging Standard for Your Organization

If you work in a team or organization, you should create a **logging standard** — a document that defines:

### 1. Field Naming Convention

```yaml
# logging-standard.yaml
required_fields:
  - name: timestamp
    format: ISO 8601 UTC
    example: "2025-06-15T10:32:45.123Z"
  
  - name: level
    allowed_values: [TRACE, DEBUG, INFO, WARN, ERROR, FATAL]
  
  - name: message
    description: "Static human-readable message. No variables embedded."
  
  - name: service
    description: "Name of the microservice"
    example: "auth-service"
  
  - name: request_id
    description: "Correlation ID propagated across services"

recommended_fields:
  - name: user_id
  - name: environment
  - name: version
  - name: host
  - name: duration_ms
```

### 2. What to Log and What NOT to Log

```markdown
## MUST Log
- All incoming HTTP requests (method, path, status code, duration)
- Authentication events (login, logout, failed attempts)
- Business transactions (orders, payments, refunds)
- Errors and exceptions (with stack traces)
- External service calls (to databases, APIs, message queues)

## MUST NOT Log
- Passwords, tokens, API keys
- Credit card numbers (use last 4 digits only)
- Personal health information
- Full request/response bodies (unless in DEBUG mode)
```

### 3. Language-Specific Libraries

```markdown
## Approved Logging Libraries
- Python: structlog
- Node.js: pino (preferred) or winston
- Go: zerolog or zap
- Java: Logback + logstash-logback-encoder
```

## 🔄 Migrating From Unstructured to Structured Logging

If you already have a bunch of unstructured logs and want to migrate, here's a step-by-step approach:

### Step 1: Start With New Code

All new code should use structured logging from day one.

### Step 2: Wrap Existing Loggers

Create a wrapper around your existing logger that outputs JSON:

```python
# Before (unstructured)
import logging
logger = logging.getLogger("my-app")
logger.info(f"User {user_id} logged in from {ip}")

# After (structured — just change the formatter, not the log calls)
import logging
import json

class JSONFormatter(logging.Formatter):
    def format(self, record):
        return json.dumps({
            "timestamp": self.formatTime(record),
            "level": record.levelname,
            "message": record.getMessage(),
            "logger": record.name,
        })

handler = logging.StreamHandler()
handler.setFormatter(JSONFormatter())
logger = logging.getLogger("my-app")
logger.handlers = [handler]
logger.setLevel(logging.INFO)

# Now same log call outputs JSON!
logger.info(f"User {user_id} logged in from {ip}")
# Output: {"timestamp": "2025-06-15 10:32:45,123", "level": "INFO", "message": "User john logged in from 192.168.1.42", "logger": "my-app"}
```

### Step 3: Gradually Add Context Fields

Over time, refactor log calls to use structured fields instead of string interpolation:

```python
# Step 2 output (message still has embedded variables):
logger.info(f"User {user_id} logged in from {ip}")

# Step 3 output (variables extracted into fields):
log.info("User logged in", user_id=user_id, source_ip=ip)
```

### Step 4: Configure Your Log Pipeline

Make sure your log collector (Fluentd, Fluent Bit, Logstash) knows how to parse your new JSON logs and ship them to your logging backend (Elasticsearch, Loki, etc.).

## 📊 Quick Reference — Structured Logging Cheat Sheet

```
┌─────────────────────────────────────────────────────────────┐
│                   STRUCTURED LOGGING CHECKLIST               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ✅ Output format: JSON (or logfmt)                        │
│  ✅ Timestamps: ISO 8601 UTC with milliseconds             │
│  ✅ Log levels: TRACE/DEBUG/INFO/WARN/ERROR/FATAL          │
│  ✅ Static messages (no variables baked in)                 │
│  ✅ Context as separate fields (user_id, order_id, etc.)   │
│  ✅ Correlation ID (request_id/trace_id)                   │
│  ✅ Service name in every log entry                        │
│  ✅ Log to stdout/stderr (not files in containers)         │
│  ✅ No sensitive data (passwords, tokens, PII)             │
│  ✅ Consistent field names across all services             │
│  ✅ Right log level (not everything is ERROR)              │
│  ✅ Stack traces for errors                                │
│  ✅ Include duration for performance-sensitive operations   │
│                                                             │
│  ❌ Don't embed variables in message strings               │
│  ❌ Don't log request/response bodies in production        │
│  ❌ Don't use different field names in different services   │
│  ❌ Don't log at DEBUG/TRACE level in production           │
│  ❌ Don't forget to include error context                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 🎯 Summary

| Concept | Key Takeaway |
|---------|-------------|
| **Structured Logging** | Log in a machine-readable format (JSON) with consistent fields |
| **Why?** | Searchable, filterable, alertable, and works at scale |
| **Core Fields** | `timestamp`, `level`, `message`, `service` |
| **Correlation** | Use `request_id` / `trace_id` to trace requests across services |
| **Security** | NEVER log passwords, tokens, or PII |
| **Containers** | Always log to stdout/stderr — let the platform collect logs |
| **Best Format** | JSON for most cases, logfmt if using Loki |
| **Migration** | Start with new code, wrap existing loggers, gradually add fields |
