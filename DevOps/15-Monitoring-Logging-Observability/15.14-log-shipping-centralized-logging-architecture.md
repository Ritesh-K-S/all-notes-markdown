# 15.14 — Log Shipping & Centralized Logging Architecture

> **One Line Answer:** Log shipping is the process of **collecting logs from all your servers, containers, and services** and **sending them to one central place** where you can search, analyze, and alert on them — instead of SSH-ing into 100 servers to find one error.

---

## 🤔 The Problem — Why Can't We Just Use `tail -f`?

Imagine you're running a small app on one server. When something breaks, you just SSH in and run:

```bash
tail -f /var/log/myapp.log
```

Easy, right? Now fast forward — your company grows. You now have:

- 50 microservices
- Running on 200 Kubernetes pods
- Across 3 cloud regions
- Each pod generates thousands of log lines per minute
- Pods are ephemeral — they die and respawn constantly

Now try to find ONE error log for ONE user request. Where do you even start?

```
You: "Something broke for user Alice"
Also you: *stares at 200 servers* "...which one?"
```

**This is why we need centralized logging.**

Instead of logs living scattered across hundreds of servers, we:

1. **Collect** logs from every source
2. **Ship** them to a central location
3. **Store** them in a searchable database
4. **Search & Analyze** from a single dashboard

### The 5 Stages of Centralized Logging:

| Stage | What Happens | Tools |
|-------|-------------|-------|
| **1. Generation** | Your app writes log lines (to stdout, files, syslog) | Your application code |
| **2. Collection** | An agent on each server/node picks up the logs | Fluent Bit, Filebeat, Promtail, Vector |
| **3. Processing** | Logs are parsed, filtered, enriched, and transformed | Logstash, Fluentd, Vector |
| **4. Storage** | Processed logs are stored in a searchable backend | Elasticsearch, Loki, ClickHouse |
| **5. Visualization** | You search and analyze logs through a UI | Kibana, Grafana |

## 🚚 Stage 1 & 2: Log Collection — The Shippers

Log shippers (also called collectors or agents) are lightweight programs that run on every server or Kubernetes node. Their job: **read log files and send them somewhere**.

### Comparison of Popular Log Shippers

| Tool | Weight | Language | Best For | Used With |
|------|--------|----------|----------|-----------|
| **Fluent Bit** | Ultra-lightweight (~1 MB) | C | Kubernetes, IoT, edge | Elasticsearch, Loki, Kafka, S3 |
| **Fluentd** | Moderate (~40 MB) | Ruby + C | Full-featured log routing | Elasticsearch, Kafka, S3, BigQuery |
| **Filebeat** | Lightweight (~30 MB) | Go | Elasticsearch ecosystem | Elasticsearch (ELK Stack) |
| **Promtail** | Lightweight | Go | Grafana Loki ecosystem | Loki only |
| **Vector** | Moderate | Rust | High-performance, flexible | Anything (very versatile) |
| **Logstash** | Heavy (~500 MB, JVM) | Java/Ruby | Complex transformations | Elasticsearch (ELK Stack) |
| **rsyslog** | Ultra-lightweight | C | Traditional Linux syslog | Files, remote syslog, Elasticsearch |

### When to Use What?

```
Using Elasticsearch + Kibana (ELK)?
  → Use Filebeat (collection) + Logstash (processing)
  → Or Fluent Bit (collection) + Fluentd (processing)

Using Grafana Loki?
  → Use Promtail or Fluent Bit

Running Kubernetes?
  → Fluent Bit (as DaemonSet) is the most popular choice

Need maximum flexibility?
  → Vector (handles collection AND processing)

On a tight budget / resource-constrained?
  → Fluent Bit (smallest memory footprint)
```

### How Log Collection Works in Kubernetes

In Kubernetes, every container writes logs to `stdout` and `stderr`. The container runtime (containerd) saves these logs as files on the **node** at:

```
/var/log/containers/<pod-name>_<namespace>_<container-name>-<container-id>.log
```

A log collector runs as a **DaemonSet** (one pod per node) and reads these log files:

```
┌─────────────────────────────────────────────────────┐
│                  Kubernetes Node                     │
│                                                     │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐          │
│  │  App Pod  │  │  App Pod  │  │  App Pod  │         │
│  │  (nginx)  │  │ (api-svc) │  │ (worker)  │         │
│  └─────┬────┘  └─────┬────┘  └─────┬────┘         │
│        │stdout        │stdout       │stdout         │
│        ▼              ▼             ▼               │
│  ┌──────────────────────────────────────────┐       │
│  │     /var/log/containers/*.log            │       │
│  │  (Container runtime saves logs here)     │       │
│  └──────────────────────┬───────────────────┘       │
│                         │                           │
│                         ▼                           │
│  ┌──────────────────────────────────────────┐       │
│  │      Fluent Bit (DaemonSet Pod)          │       │
│  │  - Reads log files                       │       │
│  │  - Adds Kubernetes metadata              │       │
│  │    (pod name, namespace, labels)         │       │
│  │  - Parses JSON log lines                 │       │
│  │  - Ships to backend                      │       │
│  └──────────────────────┬───────────────────┘       │
│                         │                           │
└─────────────────────────┼───────────────────────────┘
                          │
                          ▼
              ┌────────────────────┐
              │  Elasticsearch /   │
              │  Loki / S3 / etc.  │
              └────────────────────┘
```

### Fluent Bit DaemonSet — Complete Kubernetes Example

```yaml
# fluent-bit-daemonset.yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: fluent-bit
  namespace: logging
  labels:
    app: fluent-bit
spec:
  selector:
    matchLabels:
      app: fluent-bit
  template:
    metadata:
      labels:
        app: fluent-bit
    spec:
      serviceAccountName: fluent-bit
      containers:
        - name: fluent-bit
          image: fluent/fluent-bit:2.2
          resources:
            limits:
              memory: 200Mi    # Very lightweight!
              cpu: 200m
            requests:
              memory: 100Mi
              cpu: 100m
          volumeMounts:
            # Mount the node's container log directory
            - name: varlog
              mountPath: /var/log
              readOnly: true
            # Mount the container runtime log directory
            - name: varlibdockercontainers
              mountPath: /var/lib/docker/containers
              readOnly: true
            # Mount the Fluent Bit config
            - name: config
              mountPath: /fluent-bit/etc/
      volumes:
        - name: varlog
          hostPath:
            path: /var/log
        - name: varlibdockercontainers
          hostPath:
            path: /var/lib/docker/containers
        - name: config
          configMap:
            name: fluent-bit-config
```

**What's happening here?**
- `DaemonSet` — Ensures exactly ONE Fluent Bit pod runs on EVERY node in the cluster
- `hostPath: /var/log` — Mounts the node's log directory so Fluent Bit can read container logs
- `readOnly: true` — Fluent Bit only reads logs, never writes (security best practice)
- `resources` — Notice how little memory Fluent Bit needs (100-200 MB)

### Fluent Bit Configuration

```yaml
# fluent-bit-config ConfigMap
apiVersion: v1
kind: ConfigMap
metadata:
  name: fluent-bit-config
  namespace: logging
data:
  fluent-bit.conf: |
    [SERVICE]
        Flush         5          # Send logs every 5 seconds
        Log_Level     info
        Daemon        off
        Parsers_File  parsers.conf

    # INPUT: Read container log files from the node
    [INPUT]
        Name              tail
        Tag               kube.*
        Path              /var/log/containers/*.log
        Parser            docker           # Parse Docker JSON log format
        DB                /var/log/flb_kube.db   # Track file read positions
        Mem_Buf_Limit     50MB
        Skip_Long_Lines   On
        Refresh_Interval  10

    # FILTER: Add Kubernetes metadata (pod name, namespace, labels, etc.)
    [FILTER]
        Name                kubernetes
        Match               kube.*
        Kube_URL            https://kubernetes.default.svc:443
        Kube_CA_File        /var/run/secrets/kubernetes.io/serviceaccount/ca.crt
        Kube_Token_File     /var/run/secrets/kubernetes.io/serviceaccount/token
        Merge_Log           On        # Parse JSON inside the log field
        Keep_Log            Off       # Remove the original unparsed "log" field
        K8S-Logging.Parser  On
        K8S-Logging.Exclude On

    # OUTPUT: Send to Elasticsearch
    [OUTPUT]
        Name            es
        Match           kube.*
        Host            elasticsearch.logging.svc.cluster.local
        Port            9200
        Index           k8s-logs
        Type            _doc
        Logstash_Format On          # Creates daily indices: k8s-logs-2025.06.15
        Retry_Limit     3

  parsers.conf: |
    [PARSER]
        Name        docker
        Format      json
        Time_Key    time
        Time_Format %Y-%m-%dT%H:%M:%S.%LZ
```

**Key configuration explained:**
- `[INPUT]` — Tells Fluent Bit WHERE to read logs from (container log files)
- `tail` — Works like `tail -f` — reads new lines as they're written
- `DB /var/log/flb_kube.db` — Remembers where it stopped reading, so if Fluent Bit restarts, it doesn't re-send old logs
- `[FILTER] kubernetes` — Enriches each log line with Kubernetes info (pod name, namespace, container name, labels)
- `Merge_Log On` — If your app outputs JSON, this parses it so fields become searchable
- `[OUTPUT] es` — Sends processed logs to Elasticsearch
- `Logstash_Format On` — Creates a new index for each day (makes cleanup easy)

## ⚙️ Stage 3: Log Processing — Parse, Filter, Enrich, Transform

Raw logs often need processing before storage. This is where tools like **Logstash**, **Fluentd**, or **Vector** shine.

### What Does Log Processing Do?

| Operation | What It Does | Example |
|-----------|-------------|---------|
| **Parsing** | Convert raw text into structured fields | `"192.168.1.1 - GET /api 200 15ms"` → `{ip: "192.168.1.1", method: "GET", path: "/api", status: 200, duration_ms: 15}` |
| **Filtering** | Drop or keep logs based on conditions | Drop all DEBUG logs, keep only ERROR and above |
| **Enrichment** | Add extra information to logs | Add GeoIP data based on IP address, add Kubernetes labels |
| **Transformation** | Modify field names/values | Rename `msg` to `message`, convert timestamps |
| **Routing** | Send different logs to different places | Errors → Elasticsearch, all logs → S3 for archival |
| **Sampling** | Keep only a percentage of high-volume logs | Keep 10% of DEBUG logs to save storage |

### Logstash Pipeline Example

Logstash is the "L" in the ELK stack. It processes logs using a pipeline of `input → filter → output`.

```ruby
# /etc/logstash/conf.d/pipeline.conf

# INPUT: Where logs come from
input {
  beats {
    port => 5044    # Receive logs from Filebeat
  }
}

# FILTER: Process and transform logs
filter {
  # Parse JSON log messages
  if [message] =~ /^\{/ {
    json {
      source => "message"     # Parse the "message" field as JSON
      target => "app_log"     # Put parsed fields under "app_log"
    }
  }

  # Parse Nginx access logs using a pattern
  if [fields][type] == "nginx-access" {
    grok {
      match => {
        "message" => '%{IPORHOST:client_ip} - %{USER:ident} \[%{HTTPDATE:timestamp}\] "%{WORD:method} %{URIPATHPARAM:request} HTTP/%{NUMBER:http_version}" %{NUMBER:status_code} %{NUMBER:bytes}'
      }
    }
    
    # Convert string fields to numbers
    mutate {
      convert => {
        "status_code" => "integer"
        "bytes" => "integer"
      }
    }

    # Add GeoIP information based on client IP
    geoip {
      source => "client_ip"
      target => "geo"
    }
  }

  # Add a tag for error logs
  if [app_log][level] == "ERROR" or [app_log][level] == "FATAL" {
    mutate {
      add_tag => ["alert_worthy"]
    }
  }

  # Drop health check logs (they're noisy and not useful)
  if [app_log][message] == "Health check passed" {
    drop {}
  }

  # Remove sensitive fields
  mutate {
    remove_field => ["[app_log][password]", "[app_log][token]", "[app_log][api_key]"]
  }
}

# OUTPUT: Where processed logs go
output {
  # Send to Elasticsearch
  elasticsearch {
    hosts => ["http://elasticsearch:9200"]
    index => "logs-%{+YYYY.MM.dd}"    # Daily indices
  }

  # Also send error logs to a separate index for quick access
  if "alert_worthy" in [tags] {
    elasticsearch {
      hosts => ["http://elasticsearch:9200"]
      index => "errors-%{+YYYY.MM.dd}"
    }
  }

  # Archive everything to S3 for long-term storage
  s3 {
    bucket => "my-company-logs"
    region => "us-east-1"
    time_file => 15    # Rotate files every 15 minutes
    codec => "json_lines"
  }
}
```

**What's happening here?**
- `input { beats { port => 5044 } }` — Receives logs from Filebeat agents
- `json { source => "message" }` — Parses JSON log strings into structured fields
- `grok` — Parses unstructured text (like Nginx access logs) using patterns
- `geoip` — Looks up the geographic location of an IP address
- `drop {}` — Completely removes matching log lines (great for noisy health checks)
- `remove_field` — Strips sensitive data before storage
- Multiple `output` blocks — Send different logs to different destinations

### Fluentd Processing Example

```xml
# /etc/fluentd/fluent.conf

# INPUT: Receive logs from Fluent Bit
<source>
  @type forward
  port 24224
  bind 0.0.0.0
</source>

# FILTER: Parse JSON logs
<filter app.**>
  @type parser
  key_name log
  <parse>
    @type json
    time_key timestamp
    time_format %Y-%m-%dT%H:%M:%S.%LZ
  </parse>
</filter>

# FILTER: Add extra fields
<filter app.**>
  @type record_transformer
  <record>
    cluster_name "production-us-east"
    processed_at ${time}
  </record>
</filter>

# FILTER: Drop debug logs in production
<filter app.**>
  @type grep
  <exclude>
    key level
    pattern /^DEBUG$/
  </exclude>
</filter>

# OUTPUT: Send to Elasticsearch
<match app.**>
  @type elasticsearch
  host elasticsearch.logging.svc.cluster.local
  port 9200
  index_name app-logs
  logstash_format true
  
  # Buffer settings for reliability
  <buffer>
    @type file
    path /var/log/fluentd-buffers/app
    flush_interval 10s
    retry_max_interval 30s
    chunk_limit_size 8MB
    queue_limit_length 32
    overflow_action block
  </buffer>
</match>
```

## 💾 Stage 4: Log Storage — Where Do Logs Live?

### Option 1: Elasticsearch (Most Popular for Full-Text Search)

Elasticsearch is a **distributed search engine** that indexes log data for super-fast full-text search.

```
Pros:
  ✅ Incredibly fast full-text search
  ✅ Rich query language (Kibana Query Language / Lucene)
  ✅ Powerful aggregations (count, average, percentiles)
  ✅ Huge ecosystem (ELK/EFK stack)
  ✅ Great for complex log analysis

Cons:
  ❌ Resource-hungry (needs lots of RAM and disk)
  ❌ Expensive at scale
  ❌ Complex to operate (cluster management, shard tuning)
  ❌ Index management can be tricky
```

**Typical resource requirements:**
- Small: 3 nodes, 16 GB RAM each, 500 GB SSD each
- Medium: 5-10 nodes, 32 GB RAM each, 1 TB SSD each
- Large: 20+ nodes, 64 GB RAM each, 2+ TB SSD each

### Option 2: Grafana Loki (Lightweight Alternative)

Loki takes a fundamentally different approach — it **doesn't index the full text** of your logs. It only indexes the **labels** (like pod name, namespace, service name) and stores the log content as compressed chunks.

```
Pros:
  ✅ MUCH cheaper than Elasticsearch (10-50x less storage)
  ✅ Very simple to operate
  ✅ Native Grafana integration
  ✅ Uses the same label concept as Prometheus
  ✅ Perfect for Kubernetes environments

Cons:
  ❌ Slower full-text search (grep-like, not indexed)
  ❌ Less powerful query language than Elasticsearch
  ❌ Not ideal for complex log analytics
```

**Think of it this way:**
- **Elasticsearch** = Google Search for your logs (indexes everything, fast search)
- **Loki** = `grep` on steroids (finds labels fast, then searches within matching streams)

### Option 3: ClickHouse (For Large-Scale Analytics)

ClickHouse is a column-oriented database that's insanely fast for analytical queries on huge datasets.

```
Pros:
  ✅ Extremely fast analytical queries
  ✅ Excellent compression (10-40x)
  ✅ SQL-based queries (familiar!)
  ✅ Great for high-volume logging (petabytes)

Cons:
  ❌ Not designed for full-text search
  ❌ Requires SQL knowledge
  ❌ Less mature logging ecosystem
```

### Option 4: Cloud-Native Solutions

| Service | Cloud | Pros | Cons |
|---------|-------|------|------|
| **CloudWatch Logs** | AWS | Native AWS integration, no infra to manage | Expensive at scale, limited query power |
| **Azure Monitor Logs** | Azure | Native Azure integration, KQL queries | Can get expensive |
| **Cloud Logging** | GCP | Native GCP integration, powerful filtering | Vendor lock-in |
| **Datadog Logs** | SaaS | All-in-one platform, great UI | Very expensive |
| **Splunk** | SaaS/Self-hosted | Most powerful search & analytics | Most expensive |

### Comparison Summary

| Feature | Elasticsearch | Loki | ClickHouse | CloudWatch |
|---------|--------------|------|------------|------------|
| **Full-text search** | ⭐⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐ | ⭐⭐⭐ |
| **Cost** | 💰💰💰 | 💰 | 💰💰 | 💰💰💰💰 |
| **Complexity** | High | Low | Medium | None (managed) |
| **Query power** | Very High | Medium | Very High (SQL) | Medium |
| **Kubernetes-native** | Medium | High | Medium | Low |
| **Storage efficiency** | Medium | High | Very High | Medium |

## 📺 Stage 5: Visualization — Searching and Analyzing Logs

### Kibana (for Elasticsearch)

Kibana is the "K" in the ELK stack. It provides a web UI for searching and visualizing logs stored in Elasticsearch.

**Key features:**
- **Discover** — Search and filter logs in real-time
- **Dashboards** — Create visual dashboards with charts and graphs
- **KQL** — Kibana Query Language for intuitive searching

```
# KQL Examples (Kibana Query Language)

# Find all errors
level: "ERROR"

# Find errors in a specific service
level: "ERROR" AND service: "payment-service"

# Find logs for a specific user
user_id: "alice" AND level: ("ERROR" OR "WARN")

# Find slow requests (over 5 seconds)
duration_ms > 5000

# Wildcard search
message: *timeout*

# Find logs from the last hour with a specific request ID
request_id: "req-abc-123"
```

### Grafana (for Loki)

Grafana works with Loki using **LogQL** — a query language inspired by PromQL (Prometheus).

```
# LogQL Examples

# Show all logs from the auth service
{service="auth-service"}

# Show only error logs
{service="auth-service"} |= "ERROR"

# Parse JSON logs and filter by field
{namespace="production"} | json | level="ERROR"

# Count errors per service in the last hour
sum(count_over_time({level="ERROR"}[1h])) by (service)

# Find logs containing "timeout" in any production service
{namespace="production"} |~ "(?i)timeout"

# Parse and filter by response time
{service="api-gateway"} | json | duration_ms > 5000
```

## 🏛️ Common Logging Architectures

### Architecture 1: ELK Stack (Elasticsearch + Logstash + Kibana)

The classic. Used by thousands of companies.

```
┌────────────────┐     ┌───────────┐     ┌───────────────┐     ┌────────┐
│   Servers /    │     │           │     │               │     │        │
│   Containers   │────▶│ Filebeat  │────▶│   Logstash    │────▶│Elastic │
│   (log files)  │     │ (shipper) │     │  (processor)  │     │search  │
└────────────────┘     └───────────┘     └───────────────┘     └───┬────┘
                                                                    │
                                                               ┌────▼────┐
                                                               │ Kibana  │
                                                               │  (UI)   │
                                                               └─────────┘
```

**When to use:** When you need powerful full-text search and complex log analysis.

### Architecture 2: EFK Stack (Elasticsearch + Fluentd/Fluent Bit + Kibana)

Popular in Kubernetes environments because Fluentd is a CNCF graduated project.

```
┌────────────────┐     ┌───────────┐     ┌───────────┐     ┌───────────────┐     ┌────────┐
│  K8s Pods      │     │ Fluent Bit│     │  Fluentd  │     │               │     │        │
│  (stdout)      │────▶│(DaemonSet)│────▶│(Aggregator│────▶│ Elasticsearch │────▶│Kibana  │
│                │     │  (light)  │     │  (heavy)  │     │               │     │        │
└────────────────┘     └───────────┘     └───────────┘     └───────────────┘     └────────┘
```

**Why two layers (Fluent Bit + Fluentd)?**
- **Fluent Bit** runs on every node (lightweight, ~1 MB) — just collects and forwards
- **Fluentd** runs as a central aggregator (heavier, ~40 MB) — does the complex processing
- This keeps nodes lightweight while still allowing powerful log processing

### Architecture 3: Grafana Loki Stack (Promtail/Fluent Bit + Loki + Grafana)

The modern, lightweight alternative. Perfect for Kubernetes + Grafana users.

```
┌────────────────┐     ┌───────────┐     ┌───────────┐     ┌─────────┐
│  K8s Pods      │     │ Promtail  │     │           │     │         │
│  (stdout)      │────▶│   or      │────▶│   Loki    │────▶│ Grafana │
│                │     │ Fluent Bit│     │           │     │         │
└────────────────┘     └───────────┘     └───────────┘     └─────────┘
```

**When to use:** When you want a cost-effective solution, especially if you already use Grafana for metrics.

### Architecture 4: High-Volume Production Architecture

For large-scale production with **millions of log events per second**:

```
┌────────────────┐     ┌───────────┐     ┌─────────┐     ┌───────────┐     ┌───────────────┐
│  K8s Pods /    │     │ Fluent Bit│     │         │     │           │     │               │
│  Servers       │────▶│ (per node)│────▶│  Kafka  │────▶│ Logstash/ │────▶│ Elasticsearch │
│                │     │           │     │ (buffer) │     │ Fluentd   │     │               │
└────────────────┘     └───────────┘     └────┬────┘     └───────────┘     └───────┬───────┘
                                              │                                     │
                                              │                               ┌─────▼─────┐
                                              │                               │  Kibana    │
                                              ▼                               └───────────┘
                                         ┌─────────┐
                                         │   S3    │  (long-term archival)
                                         │(archive)│
                                         └─────────┘
```

**Why add Kafka in the middle?**

| Without Kafka | With Kafka |
|--------------|-----------|
| If Elasticsearch goes down, logs are **lost** | Kafka buffers logs — nothing is lost |
| Log spikes can overwhelm Elasticsearch | Kafka absorbs spikes, Logstash consumes at its own pace |
| No replay capability | Can replay logs if you need to reprocess |
| Single consumer | Multiple consumers (Elasticsearch + S3 + alerts) |

## 📋 Log Retention & Lifecycle Management

You can't keep logs forever — storage costs money! Set up a **retention policy**.

### Elasticsearch Index Lifecycle Management (ILM)

```json
PUT _ilm/policy/logs-lifecycle
{
  "policy": {
    "phases": {
      "hot": {
        "min_age": "0ms",
        "actions": {
          "rollover": {
            "max_age": "1d",
            "max_size": "50gb"
          }
        }
      },
      "warm": {
        "min_age": "7d",
        "actions": {
          "shrink": { "number_of_shards": 1 },
          "forcemerge": { "max_num_segments": 1 },
          "allocate": { "number_of_replicas": 0 }
        }
      },
      "cold": {
        "min_age": "30d",
        "actions": {
          "freeze": {}
        }
      },
      "delete": {
        "min_age": "90d",
        "actions": {
          "delete": {}
        }
      }
    }
  }
}
```

**What this does:**

| Phase | Age | What Happens |
|-------|-----|-------------|
| **Hot** | 0-7 days | Full performance, active writes and reads |
| **Warm** | 7-30 days | Reduced resources, read-only, fewer replicas |
| **Cold** | 30-90 days | Frozen, minimal resources, slow queries OK |
| **Delete** | 90+ days | Automatically deleted |

### Typical Retention Recommendations

| Log Type | Retention | Reason |
|----------|-----------|--------|
| Application ERROR/FATAL logs | 90 days | Needed for debugging and postmortems |
| Application INFO logs | 30 days | Useful for recent investigation |
| Application DEBUG logs | 7 days (or don't store in prod) | Very high volume, low value |
| Audit / Security logs | 1-7 years | Compliance requirements (SOC2, HIPAA, PCI) |
| Access logs | 30-90 days | Traffic analysis, security investigation |
| Infrastructure logs | 14-30 days | Troubleshooting |

## 🛡️ Best Practices for Centralized Logging

### 1. Always Buffer Your Logs

Never ship logs directly from the collector to the storage backend without buffering.

```
❌ Bad:   App → Fluent Bit → Elasticsearch  (if ES is down, logs are lost)
✅ Good:  App → Fluent Bit → Kafka → Logstash → Elasticsearch  (Kafka buffers)
✅ Also:  App → Fluent Bit (with disk buffer) → Elasticsearch  (local buffer)
```

### 2. Use DaemonSets, Not Sidecars (Usually)

| Pattern | How It Works | When to Use |
|---------|-------------|-------------|
| **DaemonSet** | One log collector per node | Most cases — simpler, lower overhead |
| **Sidecar** | One log collector per pod | When you need per-pod custom parsing or the app writes to files (not stdout) |

### 3. Set Resource Limits on Log Collectors

```yaml
resources:
  limits:
    memory: 200Mi
    cpu: 200m
  requests:
    memory: 100Mi
    cpu: 100m
```

Without limits, a log collector processing a spike could consume all node resources and kill your application pods.

### 4. Drop Noisy, Low-Value Logs Early

```
# Fluent Bit: Drop health check logs before shipping
[FILTER]
    Name    grep
    Match   *
    Exclude log health check
```

This saves bandwidth, processing, and storage costs.

### 5. Separate Hot and Cold Storage

- **Hot storage** (SSD, Elasticsearch) — Recent logs you actively search
- **Cold storage** (S3, GCS, Azure Blob) — Old logs you rarely access but need for compliance

### 6. Monitor Your Logging Pipeline

Yes, you need to **monitor your monitoring system**! If your log pipeline breaks silently, you'll think everything is fine when it's not.

```
Things to monitor:
  - Log collector CPU/memory usage
  - Log shipping lag (are logs arriving with delay?)
  - Elasticsearch cluster health
  - Disk usage on storage nodes
  - Dropped log count (any logs being lost?)
  - Kafka consumer lag (if using Kafka)
```

| **Centralized Logging** | Collect all logs from all servers/pods into one searchable place |
| **5 Stages** | Generate → Collect → Process → Store → Visualize |

| **ELK Stack** | Elasticsearch + Logstash + Kibana — the classic |
| **Loki Stack** | Promtail/Fluent Bit + Loki + Grafana — the modern lightweight |
| **Kafka** | Add Kafka as a buffer for reliability in high-volume setups |
| **Retention** | Don't keep logs forever — use ILM to age and delete old data |
| **K8s Pattern** | Always log to stdout, use DaemonSet collectors |
