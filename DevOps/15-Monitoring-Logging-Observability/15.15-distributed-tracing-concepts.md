# 15.15 — Distributed Tracing — Concepts (Spans, Traces, Context Propagation)

> **One Line Answer:** Distributed tracing lets you **follow a single user request as it travels through multiple microservices** — showing you exactly which service it went through, how long each step took, and where things went wrong.

---

## 🤔 The Problem — Why Do We Need Tracing?

Let's say a user clicks "Place Order" on your e-commerce website. That single click triggers a chain of events across MANY microservices:

```
User clicks "Place Order"
    │
    ▼
┌─────────────┐     ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│ API Gateway │────▶│ Order       │────▶│ Inventory   │────▶│ Payment     │
│             │     │ Service     │     │ Service     │     │ Service     │
└─────────────┘     └──────┬──────┘     └─────────────┘     └──────┬──────┘
                           │                                        │
                           ▼                                        ▼
                    ┌─────────────┐                          ┌─────────────┐
                    │ User        │                          │ Notification│
                    │ Service     │                          │ Service     │
                    └─────────────┘                          └─────────────┘
```

Now the user reports: **"My order took 12 seconds! That's way too slow!"**

You check the logs:

```
# API Gateway logs
INFO: Received POST /api/orders — 200 OK — 12,340ms

# Order Service logs  
INFO: Order created — 45ms

# Inventory Service logs
INFO: Stock checked — 120ms

# Payment Service logs
INFO: Payment processed — 11,800ms   ← 🐌 HERE'S THE BOTTLENECK!

# Notification Service logs
INFO: Email sent — 200ms
```

You found it! The payment service took 11.8 seconds. But here's the thing — you had to:

1. Manually check logs across 5 different services
2. Match timestamps to figure out the order of events
3. Hope that the correlation ID was consistent across all services
4. Do mental math to figure out timing

For ONE request. Now imagine doing this for hundreds of requests per day. **This doesn't scale.**

**Distributed tracing automates this entire process.** It gives you a visual timeline of every step in a request's journey — automatically.

---

## 🎯 What Is Distributed Tracing?

Distributed tracing is a method of **tracking a request as it flows through a distributed system** (multiple microservices, databases, message queues, etc.).

It answers three critical questions:

1. **WHERE** did the request go? (which services did it touch?)
2. **HOW LONG** did each step take? (where's the bottleneck?)
3. **WHERE** did it fail? (which service caused the error?)

Instead of looking at scattered logs, you see something like this:

```
┌───────────────────────────────────────────────────────────────────────┐
│  Trace: "Place Order" (trace_id: abc-123)          Total: 12,340ms  │
├───────────────────────────────────────────────────────────────────────┤
│                                                                       │
│  ├── API Gateway          ████████████████████████████████  12,340ms  │
│  │   ├── Order Service    ██                                   45ms  │
│  │   │   └── User Service █                                    15ms  │
│  │   ├── Inventory Service ████                               120ms  │
│  │   ├── Payment Service  ██████████████████████████████    11,800ms  │
│  │   │   └── Stripe API   █████████████████████████████     11,500ms  │
│  │   └── Notification Svc ██████                              200ms  │
│  │       └── Email Provider ████                              150ms  │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

**Now you can INSTANTLY see:**
- The request went through 6 services
- Payment Service took 11.8 seconds
- Specifically, the Stripe API call inside Payment Service took 11.5 seconds
- Everything else was fast
- **Root cause: Stripe API was slow**

THIS is the power of distributed tracing.

## 📚 Core Concepts — The Building Blocks

Distributed tracing has just a few core concepts. Let's go through each one very carefully.

### 1. Trace

A **trace** represents the **entire journey** of a single request through your system — from start to finish.

Think of it like this:
- **One user action** = **one trace**
- A trace has a unique **Trace ID** (e.g., `trace_id: abc-123-def-456`)
- A trace contains multiple **spans** (the individual steps)

```
Trace (trace_id: abc-123-def-456)
├── Span: API Gateway received request
├── Span: Order Service created order
├── Span: Inventory Service checked stock
├── Span: Payment Service processed payment
│   └── Span: Stripe API charged card
└── Span: Notification Service sent email
    └── Span: Email Provider delivered email
```

**Real-world analogy:** A trace is like a **package tracking number**. When you order something from Amazon, the tracking number follows your package from the warehouse → to the truck → to the sorting facility → to the delivery van → to your door. Every step is logged with a timestamp. That's exactly what a trace does for a request.

### 2. Span

A **span** represents a **single unit of work** within a trace. It's one "step" in the request's journey.

Every span has:

| Field | Description | Example |
|-------|-------------|---------|
| `trace_id` | Which trace this span belongs to | `abc-123-def-456` |
| `span_id` | Unique ID for THIS span | `span-001` |
| `parent_span_id` | The span that called this span (null for root span) | `span-000` or `null` |
| `operation_name` | What this span represents | `"POST /api/orders"` |
| `service_name` | Which service created this span | `"order-service"` |
| `start_time` | When this span started | `2025-06-15T10:32:45.000Z` |
| `duration` | How long this span took | `45ms` |
| `status` | OK, ERROR, or UNSET | `OK` |
| `tags` / `attributes` | Key-value pairs with extra context | `{http.method: "POST", http.status_code: 200}` |
| `events` / `logs` | Timestamped events within the span | `"Database query executed at T+15ms"` |

Here's what a span looks like as data:

```json
{
  "trace_id": "abc-123-def-456",
  "span_id": "span-003",
  "parent_span_id": "span-001",
  "operation_name": "POST /api/payments",
  "service_name": "payment-service",
  "start_time": "2025-06-15T10:32:45.050Z",
  "duration_ms": 11800,
  "status": "OK",
  "attributes": {
    "http.method": "POST",
    "http.url": "/api/payments",
    "http.status_code": 200,
    "payment.method": "credit_card",
    "payment.amount": 59.99,
    "payment.currency": "USD"
  },
  "events": [
    {
      "name": "Stripe API call started",
      "timestamp": "2025-06-15T10:32:45.100Z"
    },
    {
      "name": "Stripe API call completed",
      "timestamp": "2025-06-15T10:32:56.600Z"
    }
  ]
}
```

### 3. Parent-Child Relationship (Span Hierarchy)

Spans have a **parent-child relationship** that forms a tree. This tree shows the **causal relationship** between operations.

```
Root Span (no parent)
    │
    ├── Child Span 1
    │   └── Grandchild Span 1a
    │
    ├── Child Span 2
    │
    └── Child Span 3
        ├── Grandchild Span 3a
        └── Grandchild Span 3b
```

**In our order example:**

```
[Root Span] API Gateway: POST /api/orders (parent: none)
    │
    ├── [Child] Order Service: CreateOrder (parent: API Gateway)
    │   └── [Grandchild] User Service: GetUser (parent: Order Service)
    │
    ├── [Child] Inventory Service: CheckStock (parent: API Gateway)
    │
    ├── [Child] Payment Service: ProcessPayment (parent: API Gateway)
    │   └── [Grandchild] Stripe API: ChargeCard (parent: Payment Service)
    │
    └── [Child] Notification Service: SendEmail (parent: API Gateway)
        └── [Grandchild] Email Provider: Deliver (parent: Notification Service)
```

The `parent_span_id` field links each span to its parent, so the tracing system can reconstruct the entire tree.

### 4. Root Span

The **root span** is the first span in a trace — it has **no parent**. It represents the entry point of the request into your system.

```
Root Span: API Gateway — POST /api/orders
  ├── parent_span_id: null    ← This is what makes it the root
  ├── trace_id: abc-123
  └── span_id: span-000
```

Typically, the root span is created by:
- Your **API Gateway** (most common)
- Your **load balancer**
- The first service that receives the request

### 5. Span Kind

Spans have a "kind" that describes their role in the communication:

| Kind | Description | Example |
|------|-------------|---------|
| **CLIENT** | The span initiates a request to another service | Order Service calling Payment Service |
| **SERVER** | The span handles an incoming request from another service | Payment Service receiving the call |
| **INTERNAL** | The span is an internal operation (no network call) | Database query within a service |
| **PRODUCER** | The span produces a message to a queue | Sending a message to Kafka |
| **CONSUMER** | The span consumes a message from a queue | Processing a Kafka message |

A typical service-to-service call creates TWO spans:

```
Order Service                         Payment Service
┌──────────────────────┐              ┌──────────────────────┐
│ Span: Call Payment   │   HTTP       │ Span: Handle Payment │
│ Kind: CLIENT         │────────────▶ │ Kind: SERVER         │
│ Duration: 11,815ms   │              │ Duration: 11,800ms   │
└──────────────────────┘              └──────────────────────┘
                                      
The 15ms difference = network latency!
```

## 🔗 Context Propagation — The Secret Sauce

**Context propagation is the mechanism that ties everything together.** Without it, distributed tracing wouldn't work.

### The Problem

Each microservice is an independent process — it doesn't know about the other services or the overall trace. So how does the Payment Service know that it's part of trace `abc-123`?

### The Solution: Propagating Context Through HTTP Headers

When Service A calls Service B, it includes the trace information **in the HTTP headers**. Service B reads these headers, creates its own span, and links it to the parent trace.

```
Service A (Order Service)                    Service B (Payment Service)
┌────────────────────────┐                   ┌────────────────────────┐
│ 1. Create span         │                   │                        │
│ 2. Add trace context   │   HTTP Request    │ 3. Extract trace       │
│    to HTTP headers      │─────────────────▶│    context from headers│
│                        │                   │ 4. Create child span   │
│                        │   HTTP Response   │    with same trace_id  │
│ 6. Record response     │◀─────────────────│ 5. Process request     │
└────────────────────────┘                   └────────────────────────┘
```

### What Headers Are Used?

There are two main standards for trace context propagation:

#### 1. W3C Trace Context (The Standard — Use This!)

The **W3C Trace Context** is the official web standard (supported by all modern tracing tools).

```http
GET /api/payments HTTP/1.1
Host: payment-service:8080
traceparent: 00-abc123def456789012345678abcdef01-1234567890abcdef-01
tracestate: vendor1=value1,vendor2=value2
```

Let's break down the `traceparent` header:

```
traceparent: 00-abc123def456789012345678abcdef01-1234567890abcdef-01
             │   │                               │                 │
             │   │                               │                 └── Flags (01 = sampled)
             │   │                               │
             │   │                               └── Parent Span ID (16 hex chars)
             │   │
             │   └── Trace ID (32 hex chars)
             │
             └── Version (always 00 for now)
```

| Part | Size | Description |
|------|------|-------------|
| Version | 2 chars | Always `00` (current version) |
| Trace ID | 32 chars | Unique ID for the entire trace |
| Parent Span ID | 16 chars | ID of the calling span |
| Flags | 2 chars | `01` = this trace is being sampled (recorded) |

#### 2. B3 Propagation (Zipkin's Format — Legacy but Still Common)

The B3 format was created by Zipkin and uses multiple headers:

```http
GET /api/payments HTTP/1.1
Host: payment-service:8080
X-B3-TraceId: abc123def456789012345678abcdef01
X-B3-SpanId: 1234567890abcdef
X-B3-ParentSpanId: abcdef1234567890
X-B3-Sampled: 1
```

Or the compact single-header version:

```http
b3: abc123def456789012345678abcdef01-1234567890abcdef-1-abcdef1234567890
```

### Context Propagation Across Different Transports

Context doesn't only travel via HTTP headers. Here's how it works across different communication methods:

| Transport | How Context Is Propagated |
|-----------|--------------------------|
| **HTTP/REST** | HTTP headers (`traceparent`, `b3`) |
| **gRPC** | Metadata headers |
| **Kafka/Message Queues** | Message headers/properties |
| **Database Queries** | SQL comments (e.g., `/* trace_id=abc123 */`) |
| **AWS Lambda** | Custom environment variables or event payload |

### Context Propagation — Code Example

Here's how context propagation works in practice:

#### Python (with OpenTelemetry)

```python
# Install dependencies
# pip install opentelemetry-api opentelemetry-sdk opentelemetry-instrumentation-requests opentelemetry-instrumentation-flask

from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor, ConsoleSpanExporter
from opentelemetry.instrumentation.flask import FlaskInstrumentor
from opentelemetry.instrumentation.requests import RequestsInstrumentor
from opentelemetry.propagate import inject, extract
import requests
from flask import Flask, request

# ──────────────────────────────────────────────
# SETUP: Configure the tracing SDK
# ──────────────────────────────────────────────

# Create a tracer provider (the "brain" of the tracing system)
provider = TracerProvider()

# Add a span processor that exports spans (here: prints to console)
processor = BatchSpanProcessor(ConsoleSpanExporter())
provider.add_span_processor(processor)

# Set this provider as the global default
trace.set_tracer_provider(provider)

# Get a tracer (the thing you use to create spans)
tracer = trace.get_tracer("order-service")

# ──────────────────────────────────────────────
# AUTO-INSTRUMENTATION: Automatically add tracing to Flask and requests
# ──────────────────────────────────────────────

app = Flask(__name__)
FlaskInstrumentor().instrument_app(app)   # Auto-traces all incoming Flask requests
RequestsInstrumentor().instrument()        # Auto-traces all outgoing HTTP requests

# ──────────────────────────────────────────────
# APPLICATION CODE
# ──────────────────────────────────────────────

@app.route("/api/orders", methods=["POST"])
def create_order():
    # Flask auto-instrumentation already created a SERVER span for this request!
    # The trace_id and parent_span_id were automatically extracted from
    # the incoming request's 'traceparent' header.
    
    # Create a custom span for our business logic
    with tracer.start_as_current_span("create_order_logic") as span:
        # Add attributes (tags) to the span
        span.set_attribute("order.user_id", "alice")
        span.set_attribute("order.total", 59.99)
        
        # Call another service — requests auto-instrumentation automatically
        # injects the traceparent header into the outgoing request!
        response = requests.post(
            "http://payment-service:8080/api/payments",
            json={"amount": 59.99, "user_id": "alice"}
        )
        # ↑ The outgoing request will have headers like:
        # traceparent: 00-<same-trace-id>-<new-span-id>-01
        # So the Payment Service can continue the same trace!
        
        if response.status_code != 200:
            span.set_attribute("error", True)
            span.set_attribute("error.message", "Payment failed")
        
        return {"order_id": "4521", "status": "created"}, 201

if __name__ == "__main__":
    app.run(port=8080)
```

**What's happening here (step by step):**

1. `TracerProvider` — The main configuration object. It decides how spans are created and exported.
2. `ConsoleSpanExporter` — Sends completed spans to the console (for debugging). In production, you'd use `JaegerExporter` or `OTLPSpanExporter`.
3. `BatchSpanProcessor` — Batches spans and sends them in bulk (more efficient than one-by-one).
4. `FlaskInstrumentor` — **Auto-instrumentation magic!** It automatically:
   - Creates a SERVER span for every incoming HTTP request
   - Extracts the `traceparent` header to link to the parent trace
   - Records HTTP method, URL, status code as span attributes
5. `RequestsInstrumentor` — Automatically:
   - Creates a CLIENT span for every outgoing HTTP request
   - **Injects** the `traceparent` header into outgoing requests (THIS IS CONTEXT PROPAGATION!)
6. `tracer.start_as_current_span()` — Creates a new span as a child of the current span. Everything inside this `with` block is "inside" the span.

#### Node.js (with OpenTelemetry)

```javascript
// Install dependencies:
// npm install @opentelemetry/sdk-node @opentelemetry/auto-instrumentations-node
// npm install @opentelemetry/exporter-trace-otlp-http

const { NodeSDK } = require('@opentelemetry/sdk-node');
const { getNodeAutoInstrumentations } = require('@opentelemetry/auto-instrumentations-node');
const { OTLPTraceExporter } = require('@opentelemetry/exporter-trace-otlp-http');

// ──────────────────────────────────────────────
// SETUP: Initialize OpenTelemetry SDK
// ──────────────────────────────────────────────

const sdk = new NodeSDK({
  serviceName: 'order-service',           // Name of this service
  traceExporter: new OTLPTraceExporter({
    url: 'http://jaeger:4318/v1/traces',  // Send traces to Jaeger
  }),
  instrumentations: [
    getNodeAutoInstrumentations(),  // Auto-instrument Express, HTTP, gRPC, etc.
  ],
});

sdk.start();  // Start the SDK — must be called BEFORE requiring other modules!

// ──────────────────────────────────────────────
// APPLICATION CODE (after SDK is started)
// ──────────────────────────────────────────────

const express = require('express');
const axios = require('axios');
const { trace } = require('@opentelemetry/api');

const app = express();
const tracer = trace.getTracer('order-service');

app.post('/api/orders', async (req, res) => {
  // Auto-instrumentation already created a span for this Express route!
  
  // Create a custom span for business logic
  const span = tracer.startSpan('process_order');
  span.setAttribute('order.user_id', 'alice');
  span.setAttribute('order.total', 59.99);
  
  try {
    // axios is auto-instrumented — traceparent header is injected automatically!
    const paymentResponse = await axios.post('http://payment-service:8080/api/payments', {
      amount: 59.99,
      user_id: 'alice'
    });
    
    span.setAttribute('payment.status', paymentResponse.status);
    span.end();  // Don't forget to end the span!
    
    res.status(201).json({ order_id: '4521', status: 'created' });
  } catch (error) {
    span.recordException(error);           // Record the error in the span
    span.setStatus({ code: 2 });           // 2 = ERROR status
    span.end();
    
    res.status(500).json({ error: 'Order processing failed' });
  }
});

app.listen(8080, () => console.log('Order service running on port 8080'));
```

#### Go (with OpenTelemetry)

```go
package main

import (
    "context"
    "net/http"
    "go.opentelemetry.io/otel"
    "go.opentelemetry.io/otel/attribute"
    "go.opentelemetry.io/otel/exporters/otlp/otlptrace/otlptracehttp"
    "go.opentelemetry.io/otel/sdk/resource"
    sdktrace "go.opentelemetry.io/otel/sdk/trace"
    semconv "go.opentelemetry.io/otel/semconv/v1.21.0"
    "go.opentelemetry.io/contrib/instrumentation/net/http/otelhttp"
)

func initTracer() func() {
    exporter, _ := otlptracehttp.New(context.Background(),
        otlptracehttp.WithEndpoint("jaeger:4318"),
        otlptracehttp.WithInsecure(),
    )

    tp := sdktrace.NewTracerProvider(
        sdktrace.WithBatcher(exporter),
        sdktrace.WithResource(resource.NewWithAttributes(
            semconv.SchemaURL,
            semconv.ServiceName("order-service"),
        )),
    )

    otel.SetTracerProvider(tp)

    return func() { tp.Shutdown(context.Background()) }
}

func createOrderHandler(w http.ResponseWriter, r *http.Request) {
    tracer := otel.Tracer("order-service")

    // Create a span for business logic
    ctx, span := tracer.Start(r.Context(), "process_order")
    defer span.End()

    span.SetAttributes(
        attribute.String("order.user_id", "alice"),
        attribute.Float64("order.total", 59.99),
    )

    // Make an outgoing HTTP call — otelhttp automatically propagates trace context!
    client := http.Client{Transport: otelhttp.NewTransport(http.DefaultTransport)}
    req, _ := http.NewRequestWithContext(ctx, "POST", "http://payment-service:8080/api/payments", nil)
    resp, err := client.Do(req)
    if err != nil {
        span.RecordError(err)
    } else {
        span.SetAttributes(attribute.Int("payment.status_code", resp.StatusCode))
    }

    w.WriteHeader(http.StatusCreated)
    w.Write([]byte(`{"order_id": "4521", "status": "created"}`))
}

func main() {
    cleanup := initTracer()
    defer cleanup()

    // Wrap the handler with OpenTelemetry auto-instrumentation
    http.Handle("/api/orders", otelhttp.NewHandler(http.HandlerFunc(createOrderHandler), "POST /api/orders"))
    http.ListenAndServe(":8080", nil)
}
```

## 🎲 Sampling — You Don't Trace Everything

In a production system handling thousands of requests per second, tracing EVERY request would:
- Generate massive amounts of data
- Cost a fortune to store
- Slow down your services (overhead)

So we **sample** — only trace a percentage of requests.

### Sampling Strategies

| Strategy | How It Works | Use When |
|----------|-------------|----------|
| **Head-based sampling** | Decide at the START whether to trace this request | Most common. Simple. Low overhead. |
| **Tail-based sampling** | Decide at the END — keep interesting traces (errors, slow requests) | When you want to always capture errors |
| **Rate limiting** | Trace at most N requests per second | To control costs |
| **Probabilistic** | Trace X% of requests randomly | Simple and predictable |
| **Always On** | Trace everything | Dev/staging only. Never in production at scale! |

### Head-Based Sampling (Most Common)

```
Request arrives at API Gateway
    │
    ├── Random number: 0.15
    │   Threshold: 0.10 (10% sampling)
    │   0.15 > 0.10 → NOT sampled ❌
    │   (No trace created, traceparent flag = 00)
    │
    └── Random number: 0.07
        Threshold: 0.10 (10% sampling)
        0.07 < 0.10 → SAMPLED ✅
        (Full trace created, traceparent flag = 01)
```

**The sampling decision travels with the trace!** Once the API Gateway decides to sample a request (flag = `01`), ALL downstream services also sample it. This ensures you get a **complete trace**, not fragments.

```http
# Sampled (all downstream services will also trace this):
traceparent: 00-abc123...-span123...-01
                                      ↑ 01 = sampled

# Not sampled (downstream services skip tracing):
traceparent: 00-abc123...-span123...-00
                                      ↑ 00 = not sampled
```

### Tail-Based Sampling

```
                        All requests traced initially
                                    │
                                    ▼
                          ┌──────────────────┐
                          │  Sampling Service │
                          │   (at the tail)   │
                          └────────┬─────────┘
                                   │
                    ┌──────────────┼──────────────┐
                    │              │              │
                    ▼              ▼              ▼
              Keep errors     Keep slow       Drop normal
              & exceptions    requests         requests
              (always)       (> 5 sec)        (90% of them)
```

**Tail-based sampling is smarter** because it can look at the RESULT before deciding. But it's more complex and resource-intensive (you have to temporarily store ALL traces before deciding).

### Configuring Sampling in OpenTelemetry

```python
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.sampling import TraceIdRatioBased, ParentBased

# Sample 10% of traces
sampler = TraceIdRatioBased(0.10)

# ParentBased: if the parent is sampled, always sample the child
# This ensures complete traces (not fragments)
parent_sampler = ParentBased(root=sampler)

provider = TracerProvider(sampler=parent_sampler)
```

```yaml
# OpenTelemetry Collector config for tail-based sampling
processors:
  tail_sampling:
    decision_wait: 10s       # Wait 10s for all spans to arrive
    num_traces: 100000       # Keep up to 100k traces in memory
    policies:
      - name: errors-policy
        type: status_code
        status_code: {status_codes: [ERROR]}  # Always keep errors
      - name: slow-policy
        type: latency
        latency: {threshold_ms: 5000}          # Keep traces > 5 seconds
      - name: default-policy
        type: probabilistic
        probabilistic: {sampling_percentage: 10}  # Keep 10% of the rest
```

## 🔭 OpenTelemetry — The Universal Standard

**OpenTelemetry (OTel)** is the industry-standard framework for distributed tracing (and metrics and logs). It's a CNCF project and is supported by virtually every tracing backend.

### Why OpenTelemetry?

Before OpenTelemetry, every tracing tool had its own SDK:
- Jaeger had its own client libraries
- Zipkin had its own client libraries  
- Datadog had its own client libraries
- New Relic had its own client libraries

If you wanted to switch from Jaeger to Datadog, you had to **rewrite all your instrumentation code**. Nightmare!

**OpenTelemetry solves this by providing ONE SDK that works with ALL backends:**

```
┌───────────────────────────────────────────────────┐
│                  Your Application                  │
│                                                   │
│   ┌─────────────────────────────────────────┐     │
│   │        OpenTelemetry SDK                │     │
│   │   (Same code, regardless of backend)    │     │
│   └──────────────────┬──────────────────────┘     │
│                      │                            │
└──────────────────────┼────────────────────────────┘
                       │
        ┌──────────────┼──────────────┐
        │              │              │
        ▼              ▼              ▼
   ┌─────────┐   ┌──────────┐   ┌──────────┐

   Switch backends by changing ONE config line!
```

### OpenTelemetry Components

```
┌──────────────────────────────────────────────────┐
│              OpenTelemetry Architecture            │
│                                                    │
│  ┌────────────────────────────────────────────┐   │
│  │              OTel SDK                      │   │
│  │  (installed in your app)                   │   │
│  │                                            │   │
│  │  ┌──────────────┐  ┌──────────────┐        │   │
│  │  │   Tracer     │  │ Auto-        │        │   │
│  │  │   API        │  │ Instrumentation│       │   │
│  │  └──────────────┘  └──────────────┘        │   │
│  └──────────────────────┬─────────────────────┘   │
│                         │ OTLP (protocol)         │
│                         ▼                         │
│  ┌────────────────────────────────────────────┐   │
│  │          OTel Collector                    │   │
│  │  (optional but recommended)                │   │
│  │                                            │   │
│  │  ┌──────────┐ ┌──────────┐ ┌──────────┐   │   │
│  │  │Receivers │▶│Processors│▶│Exporters │   │   │
│  │  └──────────┘ └──────────┘ └──────────┘   │   │
│  └──────────────────────┬─────────────────────┘   │
│                         │                         │
│              ┌──────────┼──────────┐              │
│              ▼          ▼          ▼              │
│         ┌────────┐ ┌────────┐ ┌────────┐         │
│         │Jaeger  │ │ Tempo  │ │Datadog │         │
│         └────────┘ └────────┘ └────────┘         │
└──────────────────────────────────────────────────┘
```

| Component | What It Does |
|-----------|-------------|
| **OTel API** | The interfaces you use in your code (`tracer.startSpan()`) |
| **OTel SDK** | The implementation that creates and exports spans |
| **Auto-Instrumentation** | Automatically traces HTTP, gRPC, DB calls without code changes |
| **OTLP** | OpenTelemetry Protocol — the standard protocol for sending telemetry data |
| **OTel Collector** | A proxy/agent that receives, processes, and exports trace data |

### Auto-Instrumentation vs Manual Instrumentation

| Type | What It Does | Effort | Coverage |
|------|-------------|--------|----------|
| **Auto-instrumentation** | Automatically traces HTTP, DB, gRPC, message queue calls | Zero code changes | Framework-level |
| **Manual instrumentation** | You create custom spans for your business logic | Some code changes | Business-logic level |

**Best practice:** Use BOTH!
- Auto-instrumentation catches all the infrastructure calls (HTTP, DB, etc.)
- Manual instrumentation captures your business logic (order processing, payment validation, etc.)

### Deploy OpenTelemetry Collector

```yaml
# otel-collector.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: otel-collector-config
  namespace: tracing
data:
  config.yaml: |
    receivers:
      otlp:
        protocols:
          grpc:
            endpoint: 0.0.0.0:4317
          http:
            endpoint: 0.0.0.0:4318

    processors:
      batch:
        timeout: 5s
        send_batch_size: 1024

    exporters:
      # Send traces to Jaeger
      otlp/jaeger:
        endpoint: jaeger:4317
        tls:
          insecure: true
      
      # Also log traces to stdout (for debugging)
      logging:
        loglevel: info

    service:
      pipelines:
        traces:
          receivers: [otlp]
          processors: [batch]
          exporters: [otlp/jaeger, logging]
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: otel-collector
  namespace: tracing
spec:
  replicas: 1
  selector:
    matchLabels:
      app: otel-collector
  template:
    metadata:
      labels:
        app: otel-collector
    spec:
      containers:
        - name: collector
          image: otel/opentelemetry-collector-contrib:0.91.0
          args: ["--config", "/etc/otel/config.yaml"]
          ports:
            - containerPort: 4317   # OTLP gRPC
            - containerPort: 4318   # OTLP HTTP
          volumeMounts:
            - name: config
              mountPath: /etc/otel
      volumes:
        - name: config
          configMap:
            name: otel-collector-config
---
apiVersion: v1
kind: Service
metadata:
  name: otel-collector
  namespace: tracing
spec:
  selector:
    app: otel-collector
  ports:
    - name: otlp-grpc
      port: 4317
    - name: otlp-http
      port: 4318
```

**What's happening:**
- Your apps send traces to the **OTel Collector** (using OTLP protocol)
- The Collector **batches** spans for efficiency
- The Collector **exports** to Jaeger (and can simultaneously export to other backends)

The real power comes when you connect all three pillars of observability:

```

│                    THE OBSERVABILITY TRIANGLE             │
│                                                          │
│                       METRICS                            │
│                    (What's broken?)                       │
│                    ┌───────────┐                          │
│                   /             \                         │
│                  /               \                        │
│                 /    Connected    \                       │
│                /    by trace_id    \                      │
│               /     & service name  \                    │
│              /                       \                   │
│   LOGS ────/─────────────────────────\──── TRACES        │

└──────────────────────────────────────────────────────────┘
```

### How to Connect Them

**Include the `trace_id` in your log entries!** This way, when you find a slow trace, you can jump directly to the related logs.

```python
# Python example: Adding trace_id to structured logs

from opentelemetry import trace
import structlog

def add_trace_context(logger, method_name, event_dict):
    """Automatically add trace_id and span_id to every log entry."""
    span = trace.get_current_span()
    if span.is_recording():
        ctx = span.get_span_context()
        event_dict["trace_id"] = format(ctx.trace_id, "032x")
        event_dict["span_id"] = format(ctx.span_id, "016x")
    return event_dict

structlog.configure(
    processors=[
        add_trace_context,                              # Add trace context to logs
        structlog.processors.TimeStamper(fmt="iso"),
        structlog.processors.add_log_level,
        structlog.processors.JSONRenderer(),
    ]
)

log = structlog.get_logger()

# Now every log automatically includes trace_id and span_id!
log.info("Order created", order_id="4521", user_id="alice")
# Output: {"trace_id": "abc123def456...", "span_id": "1234567890ab...", "order_id": "4521", "user_id": "alice", "level": "info", "event": "Order created", "timestamp": "..."}
```

**Now in Grafana:**
1. You see a spike in error rate (metrics/dashboard)
2. Click on the time range → see traces with errors
3. Click on a trace → see the trace waterfall
4. Click on a span → see the related logs (matched by `trace_id`)

All connected. All searchable. All correlated.

## 📊 Quick Reference — Distributed Tracing Cheat Sheet

```
┌──────────────────────────────────────────────────────────────┐
│               DISTRIBUTED TRACING CONCEPTS                    │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  TRACE     = The full journey of ONE request                 │
│  SPAN      = One step in that journey                        │
│  ROOT SPAN = The first span (entry point)                    │
│  TRACE ID  = Unique ID for the entire trace                  │
│  SPAN ID   = Unique ID for one span                          │
│  PARENT ID = The span that called this span                  │
│                                                              │
│  CONTEXT PROPAGATION:                                        │
│    Standard: W3C traceparent header                          │
│    Legacy:   B3 headers (Zipkin)                             │
│                                                              │
│  SAMPLING:                                                   │
│    Head-based = decide at start (simple, cheap)              │
│    Tail-based = decide at end (smart, expensive)             │
│    Production = 1-10% sampling is typical                    │
│                                                              │
│  TOOLS:                                                      │
│    SDK:      OpenTelemetry (the standard)                    │
│    Backends: Jaeger, Tempo, Zipkin, Datadog, etc.            │
│    Collector: OTel Collector (proxy/router)                  │
│                                                              │
│  BEST PRACTICES:                                             │
│    1. Use OpenTelemetry SDK (vendor-neutral)                 │
│    2. Use auto-instrumentation first                         │
│    3. Add manual spans for business logic                    │
│    4. Include trace_id in logs                               │
│    5. Use head-based sampling in prod (1-10%)                │
│    6. Deploy OTel Collector as a central proxy               │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

## 🎯 Summary

| Concept | Key Takeaway |
|---------|-------------|
| **Distributed Tracing** | Follow a request across multiple microservices |
| **Trace** | The complete journey of one request (identified by Trace ID) |
| **Span** | One step in the journey (has start time, duration, status) |
| **Parent-Child Spans** | Form a tree showing causal relationships |
| **Context Propagation** | Passing trace_id via HTTP headers (W3C `traceparent`) |
| **Sampling** | Only trace a percentage of requests (head-based or tail-based) |
| **OpenTelemetry** | The universal standard SDK — use this! |
| **Auto-instrumentation** | Zero-code tracing for HTTP, DB, gRPC calls |
| **Jaeger / Tempo** | Popular open-source tracing backends |
| **Logs + Traces** | Include `trace_id` in logs to connect them |
