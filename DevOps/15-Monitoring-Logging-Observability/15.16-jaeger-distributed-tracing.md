## 🤔 Why Do We Need Jaeger? (The Problem)

Let's say a user clicks "Place Order" on your e-commerce website. That single click triggers a **chain of calls** across many microservices:

```
User clicks "Place Order"
    → API Gateway
        → Order Service
            → Inventory Service
            → Payment Service
                → Bank API (external)
            → Notification Service
                → Email Service
                → SMS Service
```

Now imagine the user says: **"My order is taking too long!"**

With a monolithic app, you'd check ONE application log. Easy.

But with microservices? That request touched **7+ services**. Each has its own logs. Where do you even start looking?

**This is the problem Jaeger solves.**

Jaeger gives you a **visual timeline** of that entire request — showing you:
- Which services were involved
- How long each service took
- Where the delay happened
- If any service failed

Think of it like **GPS tracking for your requests** — you can follow the request's journey from start to finish.

## 📚 Key Concepts — Before We Dive In

Before we look at Jaeger itself, let's understand the building blocks of distributed tracing:

### 1. Trace

A **trace** represents the **entire journey of a single request** through your system.

```
One user request = One trace
```

Example: User places an order → the trace captures everything that happened across all services for that one order.

### 2. Span

A **span** is a **single unit of work** within a trace. Each service that handles the request creates a span.

```
Trace
  ├── Span: API Gateway (20ms)
  ├── Span: Order Service (150ms)
  │     ├── Span: Inventory Service (30ms)
  │     ├── Span: Payment Service (100ms)
  │     │     └── Span: Bank API call (80ms)
  │     └── Span: Notification Service (15ms)
```

A span contains:
- **Operation name** — what work was done (e.g., `HTTP GET /orders`)
- **Start time** — when the work started
- **Duration** — how long it took
- **Tags** — key-value metadata (e.g., `http.status_code=200`)
- **Logs** — events that happened during the span (e.g., `"cache miss"`)
- **Span Context** — the trace ID, span ID, and baggage items (data carried across services)

### 3. Span Context & Context Propagation

When Service A calls Service B, how does Service B know it's part of the same trace?

**Context Propagation!** Service A injects a **trace ID** and **span ID** into the HTTP headers (or message headers), and Service B extracts them.

```
Service A → HTTP Request with headers:
    traceparent: 00-abcd1234-spanid5678-01
    
Service B reads those headers → "Ah, I'm part of trace abcd1234!"
    → creates a child span under that trace
```

This is how Jaeger "connects the dots" across services.

### 4. Sampling

In a production system, you might handle **millions of requests per second**. Tracing every single one would be expensive and overwhelming.

**Sampling** means: "Only trace a percentage of requests."

| Sampling Type | Description |
|---|---|
| **Const (constant)** | Trace all requests (`1`) or none (`0`). Good for dev/testing |
| **Probabilistic** | Trace X% of requests (e.g., `0.1` = 10%) |
| **Rate Limiting** | Trace N requests per second (e.g., 2 traces/sec) |
| **Remote** | Jaeger backend tells the client what sampling rate to use (dynamic!) |

## 🏗️ Jaeger Architecture — How It All Fits Together

Jaeger has several components. Let's understand each one:

```
┌─────────────────────────────────────────────────────────────┐
│                        Your Application                      │
│  ┌────────────────┐                                          │
│  │ Jaeger Client   │  ← Instrumentation library in your app  │
│  │ (SDK/Tracer)    │                                          │
│  └───────┬────────┘                                          │
└──────────┼──────────────────────────────────────────────────┘
           │ (sends spans)
           ▼
┌─────────────────────┐
│   Jaeger Agent       │  ← Lightweight daemon (usually sidecar or on each host)
│   (UDP listener)     │  ← Batches spans and forwards them
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│   Jaeger Collector   │  ← Receives spans from agents
│                      │  ← Validates, processes, and stores them
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│   Storage Backend    │  ← Where traces are stored
│   (Elasticsearch,    │
│    Cassandra, Kafka,  │
│    Badger, etc.)      │
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│   Jaeger Query       │  ← API server for reading traces
│   + Jaeger UI        │  ← Beautiful web UI to visualize traces
└─────────────────────┘
```

### Component Breakdown:

| Component | What It Does | Where It Runs |
|---|---|---|
| **Jaeger Client (SDK)** | Instruments your application code. Creates spans and propagates context | Inside your application |
| **Jaeger Agent** | Receives spans via UDP, batches them, and forwards to Collector | Sidecar or DaemonSet on each node |
| **Jaeger Collector** | Receives spans from agents, validates them, and writes to storage | Centralized service |
| **Storage Backend** | Stores trace data persistently | Elasticsearch, Cassandra, Kafka, etc. |
| **Jaeger Query + UI** | Provides API and web interface to search and visualize traces | Centralized service |

> **Note:** Jaeger also supports an **all-in-one** mode where all components run in a single binary. Perfect for development and testing!

## 🚀 Setting Up Jaeger — Hands-On

### Option 1: Jaeger All-in-One (Quick Start with Docker)

This is the fastest way to get started. All components in a single container:

```bash
docker run -d --name jaeger \
  -e COLLECTOR_ZIPKIN_HOST_PORT=:9411 \
  -p 6831:6831/udp \
  -p 6832:6832/udp \
  -p 5778:5778 \
  -p 16686:16686 \
  -p 4317:4317 \
  -p 4318:4318 \
  -p 14250:14250 \
  -p 14268:14268 \
  -p 14269:14269 \
  -p 9411:9411 \
  jaegertracing/all-in-one:1.57
```

**What are all these ports?**

| Port | Protocol | Purpose |
|---|---|---|
| `6831` | UDP | Jaeger Thrift compact protocol (agent) |
| `6832` | UDP | Jaeger Thrift binary protocol (agent) |
| `5778` | HTTP | Agent config endpoint (sampling strategies) |
| `16686` | HTTP | **Jaeger UI** ← This is where you view traces! |
| `4317` | gRPC | **OpenTelemetry OTLP** gRPC receiver |
| `4318` | HTTP | **OpenTelemetry OTLP** HTTP receiver |
| `14250` | gRPC | Collector gRPC (used by agents) |
| `14268` | HTTP | Collector HTTP (accepts spans directly) |
| `14269` | HTTP | Collector admin/health check |
| `9411` | HTTP | Zipkin compatible endpoint |

After running, open your browser: **http://localhost:16686** — you'll see the Jaeger UI!

### Option 2: Jaeger on Kubernetes (Production Setup)

For production, you'd use the **Jaeger Operator** on Kubernetes:

```bash
# Step 1: Install cert-manager (required by Jaeger Operator)
kubectl apply -f https://github.com/cert-manager/cert-manager/releases/download/v1.13.3/cert-manager.yaml

# Step 2: Install Jaeger Operator
kubectl create namespace observability

kubectl apply -f https://github.com/jaegertracing/jaeger-operator/releases/download/v1.57.0/jaeger-operator.yaml -n observability
```

Now create a Jaeger instance:

```yaml
# jaeger-production.yaml
apiVersion: jaegertracing.io/v1
kind: Jaeger
metadata:
  name: my-jaeger
  namespace: observability
spec:
  strategy: production          # Use production strategy (separate components)
  
  collector:
    replicas: 2                 # Scale collectors for high throughput
    resources:
      limits:
        cpu: "1"
        memory: 1Gi
  
  storage:
    type: elasticsearch         # Use Elasticsearch as backend
    options:
      es:
        server-urls: http://elasticsearch:9200
        index-prefix: jaeger
        num-shards: 3
        num-replicas: 1
    
  query:
    replicas: 2                 # Scale query service too
  
  agent:
    strategy: DaemonSet         # Run agent on every node
```

```bash
kubectl apply -f jaeger-production.yaml
```

**Jaeger Strategies on Kubernetes:**

| Strategy | Description | Use Case |
|---|---|---|
| `allInOne` | All components in one pod | Development, testing |
| `production` | Separate Collector, Query, Agent pods | Production workloads |
| `streaming` | Uses Kafka between Collector and Storage | Very high throughput production |

## 💻 Instrumenting Your Application

Jaeger needs your application to **create spans**. This is called **instrumentation**. Let's see how to do it in different languages.

### Python Example (using OpenTelemetry SDK — recommended way)

```bash
# Install dependencies
pip install opentelemetry-api \
            opentelemetry-sdk \
            opentelemetry-exporter-jaeger \
            opentelemetry-instrumentation-flask \
            opentelemetry-instrumentation-requests \
            flask requests
```

```python
# app.py — A simple Flask app with Jaeger tracing
from flask import Flask, jsonify
import requests

# OpenTelemetry imports
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.jaeger.thrift import JaegerExporter
from opentelemetry.instrumentation.flask import FlaskInstrumentor
from opentelemetry.instrumentation.requests import RequestsInstrumentor
from opentelemetry.sdk.resources import Resource

# ─── Step 1: Configure the Tracer ───
# Resource tells Jaeger what service this is
resource = Resource.create({"service.name": "order-service"})

# Create a tracer provider with the resource
tracer_provider = TracerProvider(resource=resource)

# Create a Jaeger exporter — this sends spans to Jaeger
jaeger_exporter = JaegerExporter(
    agent_host_name="localhost",   # Where Jaeger agent is running
    agent_port=6831,               # UDP port for Jaeger agent
)

# BatchSpanProcessor batches spans before sending (more efficient)
tracer_provider.add_span_processor(
    BatchSpanProcessor(jaeger_exporter)
)

# Set this as the global tracer provider
trace.set_tracer_provider(tracer_provider)

# Get a tracer for this module
tracer = trace.get_tracer(__name__)

# ─── Step 2: Create the Flask App ───
app = Flask(__name__)

# Auto-instrument Flask — automatically creates spans for every HTTP request
FlaskInstrumentor().instrument_app(app)

# Auto-instrument requests library — traces outgoing HTTP calls
RequestsInstrumentor().instrument()

# ─── Step 3: Define Routes ───
@app.route("/order/<item>")
def place_order(item):
    # This span is automatically created by FlaskInstrumentor
    # But we can add custom spans for specific operations:
    
    with tracer.start_as_current_span("check-inventory") as span:
        # Add attributes (tags) to the span
        span.set_attribute("item.name", item)
        span.set_attribute("item.quantity", 1)
        
        # Simulate calling inventory service
        inventory_ok = check_inventory(item)
        span.set_attribute("inventory.available", inventory_ok)
    
    with tracer.start_as_current_span("process-payment") as span:
        span.set_attribute("payment.method", "credit_card")
        
        # Simulate calling payment service
        payment_ok = process_payment(item)
        span.set_attribute("payment.success", payment_ok)
    
    return jsonify({
        "status": "order placed",
        "item": item,
        "inventory": inventory_ok,
        "payment": payment_ok
    })

def check_inventory(item):
    """Simulate inventory check — could be a call to another service"""
    with tracer.start_as_current_span("db-query-inventory"):
        # Simulate database query
        import time
        time.sleep(0.05)  # 50ms
        return True

def process_payment(item):
    """Simulate payment processing"""
    with tracer.start_as_current_span("call-payment-gateway"):
        # This would normally call an external payment API
        import time
        time.sleep(0.1)  # 100ms
        return True

if __name__ == "__main__":
    app.run(port=5000, debug=True)
```

**What's happening in this code?**

1. We create a `TracerProvider` that knows how to create spans
2. We configure a `JaegerExporter` that sends spans to the Jaeger agent (UDP port 6831)
3. `FlaskInstrumentor` automatically creates spans for every incoming HTTP request
4. `RequestsInstrumentor` automatically creates spans for every outgoing HTTP request
5. We manually create custom spans for specific operations (inventory check, payment)
6. Each span can have **attributes** (key-value metadata) for searchability

**Run it and test:**
```bash
# Terminal 1: Run the app
python app.py

# Terminal 2: Make a request
curl http://localhost:5000/order/laptop
```

Now open **http://localhost:16686** (Jaeger UI) → Select "order-service" → Find Traces → Click on a trace → You'll see the beautiful timeline!

```javascript
const { NodeSDK } = require("@opentelemetry/sdk-node");

    getNodeAutoInstrumentations(), // Auto-instruments Express, HTTP, etc.

// Start the SDK
sdk.start();

// Graceful shutdown
process.on("SIGTERM", () => {

});
```

### Go Example

```go
// main.go
package main

import (
    "context"
    "fmt"
    "log"
    "net/http"
    "time"

    "go.opentelemetry.io/otel"
    "go.opentelemetry.io/otel/exporters/jaeger"
    "go.opentelemetry.io/otel/sdk/resource"
    sdktrace "go.opentelemetry.io/otel/sdk/trace"
    semconv "go.opentelemetry.io/otel/semconv/v1.21.0"
    "go.opentelemetry.io/otel/attribute"
)

func initTracer() (*sdktrace.TracerProvider, error) {
    // Create Jaeger exporter
    exp, err := jaeger.New(
        jaeger.WithCollectorEndpoint(
            jaeger.WithEndpoint("http://localhost:14268/api/traces"),
        ),
    )
    if err != nil {
        return nil, err
    }

    // Create tracer provider
    tp := sdktrace.NewTracerProvider(
        sdktrace.WithBatcher(exp),
        sdktrace.WithResource(resource.NewWithAttributes(
            semconv.SchemaURL,
            semconv.ServiceNameKey.String("inventory-service"),
        )),
    )

    otel.SetTracerProvider(tp)
    return tp, nil
}

func checkStock(ctx context.Context, item string) bool {
    tracer := otel.Tracer("inventory-service")
    
    // Start a child span
    ctx, span := tracer.Start(ctx, "check-stock-db")
    defer span.End()
    
    span.SetAttributes(
        attribute.String("item.name", item),
        attribute.Int("warehouse.id", 42),
    )
    
    time.Sleep(30 * time.Millisecond) // Simulate DB query
    return true
}

func handler(w http.ResponseWriter, r *http.Request) {
    tracer := otel.Tracer("inventory-service")
    ctx, span := tracer.Start(r.Context(), "handle-inventory-request")
    defer span.End()
    
    item := r.URL.Query().Get("item")
    available := checkStock(ctx, item)
    
    fmt.Fprintf(w, "Item: %s, Available: %v", item, available)
}

func main() {
    tp, err := initTracer()
    if err != nil {
        log.Fatal(err)
    }
    defer tp.Shutdown(context.Background())

    http.HandleFunc("/inventory", handler)
    log.Println("Inventory service on :8080")
    log.Fatal(http.ListenAndServe(":8080", nil))
}
```

## 🖥️ Using the Jaeger UI

The Jaeger UI is your main tool for exploring traces. Let's walk through it:

### Search Page

When you open `http://localhost:16686`, you see the **Search** page:

```
┌────────────────────────────────────────────────────┐
│  Service: [order-service ▼]                         │
│  Operation: [all ▼]                                 │
│  Tags: [http.status_code=500]                       │
│  Lookback: [Last Hour ▼]                            │
│  Min Duration: [100ms]                              │
```

You can filter by:
- **Service** — which microservice to look at
- **Operation** — specific endpoint or function
- **Tags** — search by attributes (e.g., `error=true`, `http.status_code=500`)
- **Duration** — find slow requests
- **Lookback** — time range

### Trace Detail View

When you click on a trace, you see a **timeline (Gantt chart)**:

```
Trace: abc123 — 5 Spans — 250ms
├──────────────────────────────────────────────┤
│ order-service: HTTP GET /order/laptop  [250ms]│
│  ├───────────┤                                │
│  │ check-inventory [50ms]                     │
│  │  ├─────┤                                   │
│  │  │ db-query-inventory [30ms]               │
│  │                                            │
│  │        ├──────────────┤                    │
│  │        │ process-payment [100ms]           │
│  │        │  ├──────────┤                     │
│  │        │  │ call-payment-gateway [80ms]    │
└──────────────────────────────────────────────┘
```

Each bar shows:
- **Service name** and **operation name**
- **Duration** of the span
- **Nesting** — child spans are indented under parent spans
- **Color coding** — different services get different colors

You can click on any span to see its **details**:
- Tags/Attributes
- Logs/Events
- Process info
- Warnings

### Trace Comparison

Jaeger lets you **compare two traces** side by side. This is incredibly useful when debugging:

- "Why was this request slow?" — Compare a slow trace with a fast one
- "What changed?" — Compare a trace before and after a deployment

### Dependency Graph (DAG)

Jaeger automatically builds a **service dependency graph** from trace data:

```
        ┌──────────────┐
        │ API Gateway   │
        └──────┬───────┘
               │
        ┌──────▼───────┐
        │ Order Service │
        └──┬───┬───┬───┘
           │   │   │
    ┌──────▼┐ ┌▼────────┐ ┌▼──────────────┐
    │Inventory│ │Payment  │ │Notification   │
    │Service  │ │Service  │ │Service        │
    └────────┘ └─────────┘ └───────────────┘
```

This shows you **which services talk to which**, based on actual traffic — not documentation that might be outdated!

## ⚙️ Jaeger Configuration Deep Dive

### Sampling Configuration

You control sampling on the **client side** or **remotely** from the Jaeger backend.

**Client-side sampling (in your app):**

```python
from opentelemetry.sdk.trace.sampling import TraceIdRatioBased

# Sample 10% of traces
sampler = TraceIdRatioBased(0.1)

tracer_provider = TracerProvider(
    resource=resource,
    sampler=sampler
)
```

**Remote sampling (Jaeger controls it):**

Create a sampling strategy file on the Jaeger backend:

```json
{
  "service_strategies": [
    {
      "service": "order-service",
      "type": "probabilistic",
      "param": 0.5
    },
    {
      "service": "payment-service",
      "type": "ratelimiting",
      "param": 10
    }
  ],
  "default_strategy": {
    "type": "probabilistic",
    "param": 0.1
  }
}
```

This says:
- `order-service`: Sample 50% of traces
- `payment-service`: Sample max 10 traces per second
- Everything else: Sample 10%

### Storage Configuration

**Elasticsearch (most popular for production):**

```bash
docker run -d --name jaeger \
  -e SPAN_STORAGE_TYPE=elasticsearch \
  -e ES_SERVER_URLS=http://elasticsearch:9200 \
  -e ES_NUM_SHARDS=5 \
  -e ES_NUM_REPLICAS=1 \
  -p 16686:16686 \
  -p 14268:14268 \
  jaegertracing/all-in-one:1.57
```

**Cassandra:**

```bash
docker run -d --name jaeger \
  -e SPAN_STORAGE_TYPE=cassandra \
  -e CASSANDRA_SERVERS=cassandra \
  -e CASSANDRA_KEYSPACE=jaeger_v1_dc1 \
  -p 16686:16686 \
  jaegertracing/all-in-one:1.57
```

**Kafka (Streaming Strategy):**

For very high throughput, use Kafka as a buffer between collectors and storage:

```
Agents → Collectors → Kafka → Ingester → Elasticsearch
```

```bash
# Collector writes to Kafka
docker run -d \
  -e SPAN_STORAGE_TYPE=kafka \
  -e KAFKA_PRODUCER_BROKERS=kafka:9092 \
  -e KAFKA_TOPIC=jaeger-spans \
  jaegertracing/jaeger-collector:1.57

# Ingester reads from Kafka and writes to Elasticsearch
docker run -d \
  -e SPAN_STORAGE_TYPE=elasticsearch \
  -e KAFKA_CONSUMER_BROKERS=kafka:9092 \
  -e KAFKA_CONSUMER_TOPIC=jaeger-spans \
  -e ES_SERVER_URLS=http://elasticsearch:9200 \
  jaegertracing/jaeger-ingester:1.57
```

## 🐳 Full Docker Compose Example (Production-like)

Here's a complete setup with Jaeger + Elasticsearch + a sample app:

```yaml
# docker-compose.yaml
version: "3.8"

services:
  # ─── Elasticsearch (Storage Backend) ───
  elasticsearch:
    image: docker.elastic.co/elasticsearch/elasticsearch:8.11.0
    environment:
      - discovery.type=single-node
      - xpack.security.enabled=false

      - "9200:9200"

      - es-data:/usr/share/elasticsearch/data

      - "14268:14268"   # HTTP collector
      - "14250:14250"   # gRPC collector

      - elasticsearch

    image: jaegertracing/jaeger-agent:1.57

      - "6832:6832/udp"
      - "5778:5778"

      - jaeger-collector

  # ─── Jaeger Query + UI ───
  jaeger-query:
    image: jaegertracing/jaeger-query:1.57
    environment:
      - SPAN_STORAGE_TYPE=elasticsearch
```

## 🔌 Integrating Jaeger with Other Tools

### Jaeger + Prometheus + Grafana

Jaeger exposes its own metrics in Prometheus format:

```
# Jaeger Collector metrics
http://jaeger-collector:14269/metrics

# Key metrics to monitor:
jaeger_collector_spans_received_total        # Total spans received
jaeger_collector_spans_rejected_total        # Spans rejected (errors)
jaeger_collector_queue_length                # Internal queue length
jaeger_collector_save_latency_bucket         # Storage write latency
```

You can create Grafana dashboards to monitor Jaeger's health!

### Jaeger + Service Mesh (Istio)

If you're using Istio, tracing is almost free! Istio's Envoy proxies automatically generate spans. You just need to:

1. Configure Istio to send traces to Jaeger
2. Make sure your app **propagates trace headers** (this is the only thing you must do!)

```yaml
# Istio configuration
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  meshConfig:
    enableTracing: true
    defaultConfig:
      tracing:
        sampling: 100.0  # 100% in dev, lower in prod
        zipkin:
          address: jaeger-collector.observability:9411
```

The headers you need to propagate in your app:

```
x-request-id
x-b3-traceid
x-b3-spanid
x-b3-parentspanid
x-b3-sampled
x-b3-flags
b3
traceparent    # W3C standard
tracestate     # W3C standard
```

## 🏆 Best Practices

### 1. Name Your Spans Meaningfully

```python
# ❌ Bad
tracer.start_as_current_span("span1")
tracer.start_as_current_span("do-stuff")

# ✅ Good
tracer.start_as_current_span("GET /api/orders")
tracer.start_as_current_span("query-orders-db")
tracer.start_as_current_span("validate-payment-card")
```

### 2. Add Useful Attributes

```python
# Add searchable attributes to spans
span.set_attribute("user.id", "user-123")
span.set_attribute("order.id", "order-456")
span.set_attribute("http.method", "POST")
span.set_attribute("http.status_code", 200)
span.set_attribute("db.system", "postgresql")
span.set_attribute("db.statement", "SELECT * FROM orders WHERE id = ?")
```

### 3. Record Errors Properly

```python
from opentelemetry.trace import StatusCode

try:
    result = process_order()
except Exception as e:
    span.set_status(StatusCode.ERROR, str(e))
    span.record_exception(e)  # Records the full stack trace
    raise
```

### 4. Use Sensible Sampling in Production

```
Development  → 100% sampling (trace everything)
Staging      → 50% sampling
Production   → 1-10% sampling (or rate-limited)
```

### 5. Set Up Data Retention

Traces take up storage. Set up automatic cleanup:

```bash
# Elasticsearch ILM (Index Lifecycle Management) — delete traces after 7 days
# Or use Jaeger's built-in ES cleaner:
docker run --rm \
  -e ROLLOVER=true \
  -e ES_SERVER_URLS=http://elasticsearch:9200 \
  jaegertracing/jaeger-es-rollover:latest init

# Run the cleaner as a cron job
docker run --rm \
  -e ES_SERVER_URLS=http://elasticsearch:9200 \
  jaegertracing/jaeger-es-index-cleaner:latest 7
  # ^ keeps only last 7 days
```

## 🧠 Quick Revision

| Concept | What It Means |
|---|---|
| **Trace** | The full journey of one request across all services |
| **Span** | A single unit of work (one service, one operation) |
| **Context Propagation** | Passing trace ID across services via headers |
| **Sampling** | Deciding which requests to trace (not all in production) |
| **Jaeger Agent** | Sidecar/daemon that receives spans and forwards them |
| **Jaeger Collector** | Central service that processes and stores spans |
| **Jaeger Query + UI** | Web interface to search and visualize traces |
| **All-in-One** | Single binary with all components (for dev/test) |
