# 15.17 — OpenTelemetry — Unified Observability (Metrics + Logs + Traces)

> **One Line Answer:** OpenTelemetry (OTel) is a **vendor-neutral, open-source observability framework** that provides a **single set of APIs, SDKs, and tools** to generate, collect, and export **metrics, logs, and traces** — so you instrument your code ONCE and send data to ANY backend (Jaeger, Prometheus, Grafana, Datadog, whatever you want).

---

## 🤔 The Problem Before OpenTelemetry

Before OpenTelemetry, the observability world was a **fragmented mess**:

```
For Tracing:
  → Jaeger had its own client libraries
  → Zipkin had its own client libraries
  → Datadog had its own agent & libraries
  → AWS X-Ray had its own SDK

For Metrics:
  → Prometheus had its own client libraries
  → StatsD had its own protocol
  → Datadog had its own metrics agent

For Logging:
  → Everyone did their own thing (log4j, winston, slog, etc.)
  → No standard format, no correlation with traces
```

**The problem?**

1. **Vendor lock-in** — If you instrumented with Jaeger's SDK, switching to Zipkin meant rewriting all your instrumentation code
2. **Multiple libraries** — You needed separate libraries for traces, metrics, and logs
3. **No correlation** — Your traces, metrics, and logs were completely disconnected
4. **Maintenance burden** — Every observability vendor maintained their own SDKs for every language

It was like having **different electrical plug standards in every country** — nothing was compatible!

---

## ✅ What is OpenTelemetry?

OpenTelemetry is the **universal standard for observability**. It's like USB-C for observability — one standard that works everywhere.

```
Before OpenTelemetry:

App → Jaeger SDK → Jaeger          (tracing only)
App → Prometheus Client → Prometheus  (metrics only)
App → Custom logger → ELK            (logs only)
    
    Three different libraries. Three different formats.
    Want to switch backends? Rewrite everything.

─────────────────────────────────────────────

After OpenTelemetry:

App → OpenTelemetry SDK → OTLP → ANY backend you want!
                              ├── Jaeger
                              ├── Zipkin
                              ├── Prometheus
                              ├── Grafana (Tempo, Loki, Mimir)
                              ├── Datadog
                              ├── New Relic
                              ├── AWS X-Ray
                              ├── Elasticsearch
                              └── ...literally anything

    ONE library. ONE format. Switch backends by changing config.
```

### The Big Idea

| Aspect | What OpenTelemetry Provides |
|---|---|
| **APIs** | Language-specific interfaces for creating telemetry (traces, metrics, logs) |
| **SDKs** | Implementations of those APIs with processing and export capabilities |
| **OTLP Protocol** | A standard wire protocol for sending telemetry data |
| **Collector** | A vendor-agnostic proxy that receives, processes, and exports data |
| **Auto-instrumentation** | Libraries that automatically instrument popular frameworks (zero code changes!) |
| **Semantic Conventions** | Standard names for common attributes (e.g., `http.method`, `db.system`) |

## 📚 Core Concepts

### The Three Signals

OpenTelemetry handles all **three pillars of observability**:

```
┌─────────────────────────────────────────────────────────┐
│                    OpenTelemetry                         │
│                                                          │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐              │
│  │ TRACES  │    │ METRICS │    │  LOGS   │              │
│  │         │    │         │    │         │              │
│  │ What    │    │ What's  │    │ What    │              │
│  │ happened│    │ happening│   │ happened│              │
│  │ to this │    │ right   │    │ (detailed│             │
│  │ request?│    │ now?    │    │  events) │              │
│  └─────────┘    └─────────┘    └─────────┘              │
│                                                          │
│  Span, Trace     Counter,       LogRecord                │
│  SpanContext     Histogram,     Severity,                │
│                  Gauge          Body, Attributes          │
└─────────────────────────────────────────────────────────┘
```

| Signal | What It Captures | Example |
|---|---|---|
| **Traces** | The journey of a request across services | "This API call went through Gateway → Order → Payment → DB" |
| **Metrics** | Numerical measurements over time | "CPU usage is 75%", "Request count: 1500/min", "P99 latency: 200ms" |
| **Logs** | Discrete events with context | "User login failed: invalid password for user@example.com" |

The **magic** is that OpenTelemetry **correlates** all three:
- A log entry is linked to the trace it belongs to
- A metric can be linked to the spans that generated it
- You can go from a spike in error metrics → to the specific traces → to the exact log lines

### Resources

A **Resource** describes the entity producing telemetry:

```python
from opentelemetry.sdk.resources import Resource

resource = Resource.create({
    "service.name": "order-service",
    "service.version": "1.2.3",
    "deployment.environment": "production",
    "host.name": "server-42",
    "cloud.provider": "aws",
    "cloud.region": "us-east-1",
})
```

This metadata is attached to ALL telemetry (traces, metrics, logs) from this service. Super helpful for filtering!

### Semantic Conventions

OpenTelemetry defines **standard attribute names** so everyone uses the same terminology:

```python
# ❌ Without semantic conventions (chaos):
span.set_attribute("method", "GET")        # Team A
span.set_attribute("http_method", "GET")   # Team B
span.set_attribute("request.method", "GET") # Team C

# ✅ With semantic conventions (consistency):
span.set_attribute("http.request.method", "GET")  # Everyone uses this!
span.set_attribute("url.path", "/api/orders")
span.set_attribute("http.response.status_code", 200)
span.set_attribute("server.address", "api.example.com")
span.set_attribute("db.system", "postgresql")
span.set_attribute("db.query.text", "SELECT * FROM orders")
```

Common semantic convention namespaces:

| Namespace | Examples |
|---|---|
| `http.*` | `http.request.method`, `http.response.status_code` |
| `db.*` | `db.system`, `db.query.text`, `db.namespace` |
| `rpc.*` | `rpc.system`, `rpc.method` |
| `messaging.*` | `messaging.system`, `messaging.destination.name` |
| `cloud.*` | `cloud.provider`, `cloud.region` |
| `k8s.*` | `k8s.pod.name`, `k8s.namespace.name` |
| `service.*` | `service.name`, `service.version` |

### Component Breakdown

#### 1. API (The Interface)

The API defines **what** you can do (create spans, record metrics, emit logs) but not **how** it works internally.

```python
from opentelemetry import trace, metrics

# These are API calls — they don't care about the implementation
tracer = trace.get_tracer("my-service")
meter = metrics.get_meter("my-service")
```

**Why separate API from SDK?** Library authors can use the API without depending on the SDK. Your app provides the SDK. This way, libraries don't force a specific implementation on you.

#### 2. SDK (The Implementation)

The SDK is the actual implementation. It handles creating spans, batching, and exporting.

```python
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.metrics import MeterProvider
from opentelemetry.sdk._logs import LoggerProvider
```

#### 3. Exporters (Where Data Goes)

Exporters send your telemetry to backends:

| Exporter | Sends To |
|---|---|
| `OTLPSpanExporter` | Any OTLP-compatible backend (Jaeger, Tempo, etc.) |
| `OTLPMetricExporter` | Any OTLP-compatible backend (Prometheus via OTLP, Mimir, etc.) |
| `ConsoleSpanExporter` | Prints to console (for debugging) |
| `PrometheusExporter` | Exposes `/metrics` endpoint for Prometheus to scrape |
| `JaegerExporter` | Directly to Jaeger |
| `ZipkinExporter` | Directly to Zipkin |

#### 4. OTLP Protocol

**OTLP (OpenTelemetry Protocol)** is the native protocol. It supports:

- **gRPC** (port 4317) — binary, fast, efficient
- **HTTP/protobuf** (port 4318) — works through proxies/load balancers
- **HTTP/JSON** (port 4318) — human-readable, good for debugging

```python
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter

# Send traces via OTLP gRPC
exporter = OTLPSpanExporter(
    endpoint="http://otel-collector:4317",  # gRPC
    insecure=True
)
```

## 🚀 OpenTelemetry Collector — The Swiss Army Knife

The Collector is the **most powerful component** of OpenTelemetry. It's a standalone process that receives, processes, and exports telemetry data.

### Why Use a Collector?

```
Without Collector:
  App1 → Jaeger
  App1 → Prometheus  
  App2 → Jaeger
  App2 → Prometheus
  
  Problem: Every app needs to know about every backend.
           Changing backends = changing every app.

With Collector:
  App1 → Collector → Jaeger
  App2 → Collector → Prometheus
                   → Loki (add new backend? Just change collector config!)
  
  Apps only know about the Collector.
  Collector handles routing, processing, and exporting.
```

### Collector Architecture

```
┌─────────────────────────────────────────────────┐
│              OTel Collector                       │
│                                                   │
│  ┌───────────┐   ┌────────────┐   ┌───────────┐ │
│  │ Receivers  │──▶│ Processors │──▶│ Exporters │ │
│  └───────────┘   └────────────┘   └───────────┘ │
│                                                   │
│  "How data     "What to do      "Where to         │
│   comes in"     with the data"   send it"          │
└─────────────────────────────────────────────────┘
```

| Component | Purpose | Examples |
|---|---|---|
| **Receivers** | Accept data in various formats | OTLP, Jaeger, Zipkin, Prometheus, Kafka, Filelog |
| **Processors** | Transform, filter, batch, sample data | Batch, Filter, Attributes, Tail Sampling, Memory Limiter |
| **Exporters** | Send data to backends | OTLP, Jaeger, Prometheus, Loki, Elasticsearch, File |
| **Connectors** | Connect one pipeline's output to another pipeline's input | Spanmetrics (generate metrics from traces!) |

### Collector Configuration

The Collector uses a YAML configuration file:

```yaml
# otel-collector-config.yaml

# ─── RECEIVERS: How data comes in ───
receivers:
  # Accept OTLP data (from your apps)
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
      http:
        endpoint: 0.0.0.0:4318

  # Scrape Prometheus metrics
  prometheus:
    config:
      scrape_configs:
        - job_name: 'my-app'
          scrape_interval: 15s
          static_configs:
            - targets: ['app:8080']

  # Accept Jaeger data (backward compatibility)
  jaeger:
    protocols:
      thrift_compact:
        endpoint: 0.0.0.0:6831
      grpc:
        endpoint: 0.0.0.0:14250

  # Read log files
  filelog:
    include: [/var/log/app/*.log]
    operators:
      - type: json_parser
        timestamp:
          parse_from: attributes.time
          layout: '%Y-%m-%dT%H:%M:%S'

# ─── PROCESSORS: What to do with data ───
processors:
  # Batch spans before sending (efficiency)
  batch:
    timeout: 5s
    send_batch_size: 1000
    send_batch_max_size: 2000

  # Limit memory usage
  memory_limiter:
    check_interval: 1s
    limit_mib: 512
    spike_limit_mib: 128

  # Add/modify attributes
  attributes:
    actions:
      - key: environment
        value: production
        action: upsert

  # Filter out unwanted data
  filter:
    traces:
      span:
        - 'attributes["http.target"] == "/healthz"'  # Drop health check spans
        - 'attributes["http.target"] == "/readyz"'

  # Tail-based sampling (decide AFTER seeing the full trace)
  tail_sampling:
    decision_wait: 10s
    policies:
      - name: errors-policy
        type: status_code
        status_code: { status_codes: [ERROR] }  # Always keep error traces
      - name: slow-traces
        type: latency
        latency: { threshold_ms: 1000 }  # Keep traces > 1s
      - name: probabilistic
        type: probabilistic
        probabilistic: { sampling_percentage: 10 }  # Sample 10% of the rest

# ─── EXPORTERS: Where to send data ───
exporters:
  # Send traces to Jaeger
  otlp/jaeger:
    endpoint: jaeger:4317
    tls:
      insecure: true

  # Send traces to Tempo
  otlp/tempo:
    endpoint: tempo:4317
    tls:
      insecure: true

  # Send metrics to Prometheus (expose /metrics endpoint)
  prometheus:
    endpoint: 0.0.0.0:8889

  # Send metrics via OTLP to Mimir
  otlphttp/mimir:
    endpoint: http://mimir:9009/otlp

  # Send logs to Loki
  loki:
    endpoint: http://loki:3100/loki/api/v1/push

  # Debug: print to console
  debug:
    verbosity: detailed

# ─── CONNECTORS: Bridge between pipelines ───
connectors:
  # Generate RED metrics (Rate, Errors, Duration) from traces!
  spanmetrics:
    histogram:
      explicit:
        buckets: [5ms, 10ms, 25ms, 50ms, 100ms, 250ms, 500ms, 1s, 5s]
    dimensions:
      - name: http.method
      - name: http.status_code

# ─── SERVICE: Wire everything together ───
service:
  pipelines:
    # Traces pipeline
    traces:
      receivers: [otlp, jaeger]
      processors: [memory_limiter, filter, tail_sampling, batch]
      exporters: [otlp/tempo, debug]

    # Metrics pipeline (from apps)
    metrics:
      receivers: [otlp, prometheus]
      processors: [memory_limiter, batch]
      exporters: [prometheus, otlphttp/mimir]

    # Metrics pipeline (generated from traces via spanmetrics connector)
    metrics/spanmetrics:
      receivers: [spanmetrics]
      processors: [batch]
      exporters: [prometheus]

    # Logs pipeline
    logs:
      receivers: [otlp, filelog]
      processors: [memory_limiter, attributes, batch]
      exporters: [loki, debug]
  
  # Collector's own telemetry
  telemetry:
    logs:
      level: info
    metrics:
      address: 0.0.0.0:8888
```

**What's happening here?**

1. **Receivers** accept data from multiple sources (OTLP, Jaeger, Prometheus scrape, log files)
2. **Processors** batch data, limit memory, filter out health checks, do tail sampling
3. **Exporters** send traces to Tempo, metrics to Prometheus/Mimir, logs to Loki
4. **Connectors** generate metrics from trace data (spanmetrics) — so you get RED metrics for free!
5. **Service** wires everything into pipelines

```python
# otel_setup.py — Centralized OpenTelemetry configuration
from opentelemetry import trace, metrics
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.sdk.metrics import MeterProvider
from opentelemetry.sdk.metrics.export import PeriodicExportingMetricReader
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter
from opentelemetry.exporter.otlp.proto.grpc.metric_exporter import OTLPMetricExporter
from opentelemetry.sdk.resources import Resource
from opentelemetry.sdk._logs import LoggerProvider, LoggingHandler
from opentelemetry.sdk._logs.export import BatchLogRecordProcessor
from opentelemetry.exporter.otlp.proto.grpc._log_exporter import OTLPLogExporter

import logging

def setup_otel(service_name: str, otel_endpoint: str = "http://localhost:4317"):
    """Set up OpenTelemetry for traces, metrics, and logs."""
    
    # ─── Resource (identifies this service) ───
    resource = Resource.create({
        "service.name": service_name,
        "service.version": "1.0.0",
        "deployment.environment": "production",
    })
    
    # ─── TRACES ───
    tracer_provider = TracerProvider(resource=resource)
    tracer_provider.add_span_processor(
        BatchSpanProcessor(
            OTLPSpanExporter(endpoint=otel_endpoint, insecure=True)
        )
    )
    trace.set_tracer_provider(tracer_provider)
    
    # ─── METRICS ───
    metric_reader = PeriodicExportingMetricReader(
        OTLPMetricExporter(endpoint=otel_endpoint, insecure=True),
        export_interval_millis=30000,  # Export every 30 seconds
    )
    meter_provider = MeterProvider(resource=resource, metric_readers=[metric_reader])
    metrics.set_meter_provider(meter_provider)
    
    # ─── LOGS ───
    logger_provider = LoggerProvider(resource=resource)
    logger_provider.add_log_record_processor(
        BatchLogRecordProcessor(
            OTLPLogExporter(endpoint=otel_endpoint, insecure=True)
        )
    )
    
    # Bridge Python's logging module to OpenTelemetry
    handler = LoggingHandler(
        level=logging.INFO,
        logger_provider=logger_provider
    )
    logging.getLogger().addHandler(handler)
    
    return tracer_provider, meter_provider, logger_provider
```

```python
# app.py — Flask application with full OpenTelemetry instrumentation
from flask import Flask, jsonify, request
import logging
from otel_setup import setup_otel

from opentelemetry import trace, metrics
from opentelemetry.instrumentation.flask import FlaskInstrumentor
from opentelemetry.instrumentation.requests import RequestsInstrumentor
from opentelemetry.trace import StatusCode

# ─── Initialize OpenTelemetry ───
tp, mp, lp = setup_otel("order-service", "http://otel-collector:4317")

# Get a tracer and meter
tracer = trace.get_tracer("order-service")
meter = metrics.get_meter("order-service")

# ─── Create Custom Metrics ───
# Counter: how many orders were placed
order_counter = meter.create_counter(
    name="orders.total",
    description="Total number of orders placed",
    unit="1",
)

# Histogram: how long order processing takes
order_duration = meter.create_histogram(
    name="orders.duration",
    description="Time to process an order",
    unit="ms",
)

# UpDownCounter: current orders being processed
orders_in_progress = meter.create_up_down_counter(
    name="orders.in_progress",
    description="Number of orders currently being processed",
    unit="1",
)

# ─── Create Flask App ───
app = Flask(__name__)

# Auto-instrument Flask and requests
FlaskInstrumentor().instrument_app(app)
RequestsInstrumentor().instrument()

logger = logging.getLogger(__name__)

@app.route("/order", methods=["POST"])
def create_order():
    import time
    start_time = time.time()
    
    orders_in_progress.add(1)  # Increment in-progress counter
    
    try:
        data = request.json
        item = data.get("item", "unknown")
        quantity = data.get("quantity", 1)
        
        # Log with trace context (automatically correlated!)
        logger.info(f"Received order for {quantity}x {item}")
        
        # Custom span for business logic
        with tracer.start_as_current_span("validate-order") as span:
            span.set_attribute("order.item", item)
            span.set_attribute("order.quantity", quantity)
            
            if quantity <= 0:
                span.set_status(StatusCode.ERROR, "Invalid quantity")
                logger.error(f"Invalid order: quantity={quantity}")
                return jsonify({"error": "Invalid quantity"}), 400
            
            logger.info(f"Order validated: {item}")
        
        with tracer.start_as_current_span("process-payment") as span:
            span.set_attribute("payment.method", "credit_card")
            time.sleep(0.1)  # Simulate payment processing
            span.add_event("Payment authorized", {
                "payment.transaction_id": "txn-12345",
                "payment.amount": 99.99,
            })
            logger.info("Payment processed successfully")
        
        # Record metrics
        order_counter.add(1, {"item": item, "status": "success"})
        
        duration_ms = (time.time() - start_time) * 1000
        order_duration.record(duration_ms, {"item": item})
        
        logger.info(f"Order completed in {duration_ms:.1f}ms")
        
        return jsonify({"status": "created", "item": item})
    
    except Exception as e:
        order_counter.add(1, {"item": "unknown", "status": "error"})
        logger.exception(f"Order failed: {e}")
        raise
    
    finally:
        orders_in_progress.add(-1)  # Decrement in-progress counter

if __name__ == "__main__":
    app.run(port=5000)
```

**What does this give you?**

- **Traces**: Every HTTP request is traced, with custom spans for validate-order and process-payment
- **Metrics**: Order counts, processing duration (histogram), in-progress orders
- **Logs**: All logs are automatically correlated with the trace they belong to
- **Correlation**: Click a log in Loki → jump to the trace in Tempo → see the exact request flow!

```javascript
// app.js
require("./tracing"); // MUST be first!

const express = require("express");
const { trace, metrics } = require("@opentelemetry/api");

const app = express();
app.use(express.json());

const tracer = trace.getTracer("payment-service");
const meter = metrics.getMeter("payment-service");

// Custom metrics
const paymentCounter = meter.createCounter("payments.total", {
  description: "Total payment attempts",
});
const paymentDuration = meter.createHistogram("payments.duration_ms", {
  description: "Payment processing time in ms",
});

app.post("/pay", (req, res) => {
  const start = Date.now();
  
  const span = tracer.startSpan("process-payment", {
    attributes: {
      "payment.method": req.body.method || "card",
      "payment.amount": req.body.amount,
      "payment.currency": req.body.currency || "USD",
    },
  });

  try {
    // Simulate processing
    setTimeout(() => {
      span.addEvent("Payment authorized");
      span.end();

      paymentCounter.add(1, { status: "success", method: req.body.method });
      paymentDuration.record(Date.now() - start);

      res.json({ status: "paid" });
    }, 100);
  } catch (err) {
    span.setStatus({ code: 2, message: err.message }); // StatusCode.ERROR = 2
    span.recordException(err);
    span.end();

    paymentCounter.add(1, { status: "error" });
    res.status(500).json({ error: err.message });
  }
});

app.listen(3000, () => console.log("Payment service on :3000"));
```

### Zero-Code Instrumentation (No Code Changes!)


```yaml
# otel-collector-k8s.yaml

kind: OpenTelemetryCollector
metadata:
  name: otel
  namespace: observability
spec:
  mode: deployment  # or daemonset, statefulset, sidecar

    processors:
      batch:
        timeout: 5s
      memory_limiter:
        check_interval: 1s
        limit_mib: 256
    
    exporters:
      otlp/tempo:
        endpoint: tempo.observability:4317
        tls:
          insecure: true
      prometheus:
        endpoint: 0.0.0.0:8889
      loki:
        endpoint: http://loki.observability:3100/loki/api/v1/push
    
    service:
      pipelines:
        traces:
          receivers: [otlp]
          processors: [memory_limiter, batch]
          exporters: [otlp/tempo]
        metrics:
          receivers: [otlp]
          processors: [memory_limiter, batch]
          exporters: [prometheus]
        logs:
          receivers: [otlp]
          processors: [memory_limiter, batch]
          exporters: [loki]
```

**Auto-instrument pods (zero code changes!):**

```yaml

apiVersion: opentelemetry.io/v1alpha1
kind: Instrumentation
metadata:
  name: auto-instrumentation
  namespace: default
spec:
  exporter:
    endpoint: http://otel-collector.observability:4317

  # Language-specific config
  python:
    image: ghcr.io/open-telemetry/opentelemetry-operator/autoinstrumentation-python:latest

    image: ghcr.io/open-telemetry/opentelemetry-operator/autoinstrumentation-dotnet:latest
```

Now just **annotate your pods** and the Operator auto-instruments them:

```yaml
# my-app-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: order-service
spec:
  template:
    metadata:
      annotations:
        # Just add ONE annotation and your app is instrumented! 🎉

      containers:
        - name: order-service
          image: my-order-service:latest
```

The Operator injects a sidecar init container that sets up auto-instrumentation. **No code changes needed!**

## 🔗 Environment Variables (Configuration Without Code)

OpenTelemetry can be configured entirely via environment variables:

```bash
# Service identity
OTEL_SERVICE_NAME=order-service
OTEL_RESOURCE_ATTRIBUTES=deployment.environment=production,service.version=1.2.3

# Exporter configuration
OTEL_EXPORTER_OTLP_ENDPOINT=http://otel-collector:4317
OTEL_EXPORTER_OTLP_PROTOCOL=grpc    # or http/protobuf, http/json
OTEL_EXPORTER_OTLP_HEADERS=Authorization=Bearer my-token

# Per-signal endpoints (override the general one)
OTEL_EXPORTER_OTLP_TRACES_ENDPOINT=http://tempo:4317
OTEL_EXPORTER_OTLP_METRICS_ENDPOINT=http://mimir:4317
OTEL_EXPORTER_OTLP_LOGS_ENDPOINT=http://loki:4317

# Sampling
OTEL_TRACES_SAMPLER=parentbased_traceidratio
OTEL_TRACES_SAMPLER_ARG=0.1   # 10%

# Propagation
OTEL_PROPAGATORS=tracecontext,baggage

# Logs
OTEL_LOG_LEVEL=info

# Disable specific instrumentations
OTEL_PYTHON_DISABLED_INSTRUMENTATIONS=urllib3
```

This means you can deploy the SAME Docker image to different environments and just change env vars!

## 🏆 Best Practices

### 1. Always Use the Collector

```
❌ App → Jaeger directly
✅ App → OTel Collector → Jaeger

Why? The Collector gives you:
- Buffering (protects backends from spikes)
- Processing (filtering, sampling, enriching)
- Fan-out (send same data to multiple backends)
- Decoupling (change backends without changing apps)
```

### 2. Use Semantic Conventions

```python
# ❌ Custom attribute names
span.set_attribute("method", "GET")
span.set_attribute("status", 200)

# ✅ Semantic conventions
span.set_attribute("http.request.method", "GET")
span.set_attribute("http.response.status_code", 200)
```

### 3. Set Resource Attributes Properly

```python
resource = Resource.create({
    "service.name": "order-service",       # REQUIRED!
    "service.version": "1.2.3",
    "deployment.environment": "production",
    "service.namespace": "ecommerce",
})
```

### 4. Don't Over-Instrument

```python
# ❌ Too granular — creates noise
with tracer.start_as_current_span("parse-json"):
    data = json.loads(body)
with tracer.start_as_current_span("validate-field-name"):
    assert data["name"]

# ✅ Right level of detail
with tracer.start_as_current_span("validate-order-request") as span:
    data = json.loads(body)
    span.set_attribute("order.items_count", len(data.get("items", [])))
```

### 5. Use Baggage for Cross-Service Context

```python
from opentelemetry import baggage, context

# Service A: Set baggage
ctx = baggage.set_baggage("user.id", "user-123")
ctx = baggage.set_baggage("tenant.id", "acme-corp", context=ctx)

# Service B: Read baggage (automatically propagated!)
user_id = baggage.get_baggage("user.id")
tenant_id = baggage.get_baggage("tenant.id")
```

### 6. Configure Memory Limits on the Collector

```yaml
processors:
  memory_limiter:
    check_interval: 1s
    limit_mib: 512        # Hard limit
    spike_limit_mib: 128  # Soft limit for spikes
```

Without this, the Collector can OOM during traffic spikes!

## 🧠 Quick Revision

| Concept | What It Means |
|---|---|
| **OpenTelemetry** | Vendor-neutral framework for traces + metrics + logs |
| **OTLP** | The standard protocol for sending telemetry data |
| **Collector** | Proxy that receives, processes, and exports telemetry |
| **Receivers** | How the Collector accepts data (OTLP, Jaeger, Prometheus, etc.) |
| **Processors** | Transform/filter/batch data in the Collector |
| **Exporters** | Where the Collector sends data (Tempo, Prometheus, Loki, etc.) |
| **Auto-instrumentation** | Instrument apps without code changes |
| **Semantic Conventions** | Standard attribute names for consistency |
| **Resource** | Metadata about the service producing telemetry |
| **Baggage** | Key-value pairs propagated across services |
