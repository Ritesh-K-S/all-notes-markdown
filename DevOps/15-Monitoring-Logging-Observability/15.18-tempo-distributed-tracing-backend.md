## 🤔 Why Tempo? What Problem Does It Solve?

Let's say you're already using **Jaeger** for tracing. It works great! But then your traffic grows and suddenly:

```
Problem 1: Storage Costs 💸
  Jaeger uses Elasticsearch or Cassandra as storage.
  At scale (millions of traces/day), these databases become EXPENSIVE:
  - Elasticsearch needs lots of RAM (for indexing)
  - Cassandra needs lots of disk (for replication)
  - Both need careful tuning and operational expertise

Problem 2: Operational Complexity 🔧
  Running Elasticsearch/Cassandra clusters:
  - Shard management
  - Index lifecycle policies  
  - Cluster scaling
  - JVM tuning nightmares
  
Problem 3: Do you really need full-text search on traces?
  Think about it — when do you search for traces?
  - By Trace ID (someone gives you a trace ID from logs)
  - By Service Name + Time Range
  - By Duration (find slow traces)
  
  You almost NEVER do full-text search on traces.
  So why pay for a full-text search engine (Elasticsearch)?
```

**Tempo's approach:** Store traces in **cheap object storage** (S3, GCS, Azure Blob) and only index the bare minimum needed to find them. No full-text indexing. No expensive databases.

```
Cost comparison (rough estimates at scale):

Jaeger + Elasticsearch:
  - 10 Elasticsearch nodes × $500/month = $5,000/month
  - Operational overhead: HIGH
  
Tempo + S3:
  - S3 storage: ~$50/month (same data volume)
  - 2-3 Tempo instances: ~$300/month
  - Operational overhead: LOW
  
Savings: ~90% cheaper! 🎉
```

## ✅ What Exactly is Tempo?

Tempo is a **distributed tracing backend** created by **Grafana Labs**. Here's what makes it special:

| Feature | Description |
|---|---|
| **Object Storage Backend** | Uses S3, GCS, Azure Blob, or local filesystem — incredibly cheap |
| **No Indexing Required** | Traces are looked up by Trace ID — no need for full-text indexing |
| **Trace Discovery via Logs** | Find Trace IDs in your logs (Loki), then look them up in Tempo |
| **TraceQL** | A powerful query language for searching traces (like PromQL but for traces) |
| **Massive Scale** | Handles billions of spans with minimal infrastructure |
| **OpenTelemetry Native** | Built from the ground up to work with OTLP |
| **Grafana Integration** | First-class support in Grafana for visualization |
| **Multi-protocol** | Accepts Jaeger, Zipkin, OTLP, and OpenCensus formats |

### The Key Insight

Tempo is built around a simple but powerful idea:

```
Traditional Tracing:  "Index everything so users can search"
                       → Expensive, complex
                       
Tempo's Philosophy:   "Store everything cheaply. Use Trace IDs to find traces.
                       Get Trace IDs from logs, metrics, or exemplars."
                       → Cheap, simple
```

This works because in practice, you almost always find traces through:
1. **Logs** — "This error log has trace_id=abc123, let me look at that trace"
2. **Metrics** — "This P99 latency spike has an exemplar with trace_id=xyz789"
3. **Direct Trace ID** — Someone shares a trace ID in a support ticket
4. **TraceQL** — Search by service name, duration, status, attributes

## 🏗️ Tempo Architecture

### Single-Binary Mode (Simple)

For small deployments, Tempo can run as a **single process**:

```
┌─────────────────────────────────────────┐
│              Tempo (All-in-One)           │
│                                           │
│  ┌───────────┐  ┌───────────┐             │
│  │ Distributor│  │  Ingester │             │
│  └─────┬─────┘  └─────┬─────┘             │
│        │               │                   │
│  ┌─────▼───────────────▼─────┐             │
│  │       Compactor           │             │
│  └───────────┬───────────────┘             │
│              │                             │
│  ┌───────────▼───────────────┐             │
│  │     Query Frontend        │             │
│  │     + Querier             │             │
│  └───────────────────────────┘             │
└──────────────┬────────────────────────────┘
               │
               ▼
        ┌──────────────┐
        │ Object Store │
        │ (S3/GCS/     │
        │  MinIO/Local)│
        └──────────────┘
```

### Microservices Mode (Production Scale)

For large deployments, each component runs independently:

```
                    Incoming Traces (OTLP, Jaeger, Zipkin)
                              │
                              ▼
                    ┌─────────────────┐
                    │   Distributor    │  ← Receives spans
                    │   (Stateless)    │  ← Validates and forwards
                    └────────┬────────┘
                             │ (hash ring)
                             ▼
                    ┌─────────────────┐
                    │    Ingester      │  ← Buffers spans in memory
                    │   (Stateful)     │  ← Creates "blocks" of traces
                    │                  │  ← Flushes blocks to object storage
                    └────────┬────────┘
                             │
                ┌────────────┼────────────┐
                │            │            │
                ▼            ▼            ▼
        ┌──────────┐  ┌──────────┐  ┌──────────┐
        │ Object   │  │ Object   │  │ Object   │
        │ Storage  │  │ Storage  │  │ Storage  │
        │ (blocks) │  │ (blocks) │  │ (blocks) │
        └──────────┘  └──────────┘  └──────────┘
                             │
                    ┌────────▼────────┐
                    │   Compactor      │  ← Merges small blocks
                    │   (Stateless)    │  ← Optimizes storage
                    └─────────────────┘
                             │
                    ┌────────▼────────┐
                    │  Query Frontend  │  ← Receives trace queries
                    │   (Stateless)    │  ← Splits queries, caches results
                    └────────┬────────┘
                             │
                    ┌────────▼────────┐
                    │    Querier       │  ← Reads from object storage
                    │   (Stateless)    │  ← Returns traces
                    └─────────────────┘
                             │
                             ▼
                       Grafana UI
```

### Component Deep Dive

| Component | Role | Scaling |
|---|---|---|
| **Distributor** | Entry point. Receives spans via OTLP/Jaeger/Zipkin. Validates them. Uses a hash ring to route spans for the same trace to the same Ingester | Stateless — scale horizontally |
| **Ingester** | Buffers incoming spans in memory. Groups them into traces. Flushes complete traces as "blocks" to object storage | Stateful — needs consistent hashing |
| **Compactor** | Merges small blocks into larger ones. Reduces the number of objects in storage. Creates bloom filters for faster lookups | Stateless — can be scaled |
| **Query Frontend** | Receives search queries. Splits large time-range queries into smaller ones. Caches results | Stateless — scale horizontally |
| **Querier** | Actually reads blocks from object storage. Searches for traces by ID or TraceQL queries | Stateless — scale horizontally |

### How Tempo Stores Data

```
Object Storage (e.g., S3 bucket):

my-tempo-bucket/
├── single-tenant/
│   ├── blocks/
│   │   ├── block-001/
│   │   │   ├── data            ← Actual trace data (compressed)
│   │   │   ├── meta.json       ← Block metadata
│   │   │   ├── bloom-0         ← Bloom filter (for fast "does this trace exist?" checks)
│   │   │   └── index           ← Minimal index for trace lookup
│   │   ├── block-002/
│   │   └── block-003/
│   └── blocklist/
│       └── ...
```

**Key insight:** Tempo uses **bloom filters** to quickly answer "Does trace X exist in this block?" without reading the entire block. This makes lookups by Trace ID very fast even with millions of blocks.

## 🚀 Setting Up Tempo

### Option 1: Docker (Quick Start)

**Tempo configuration file:**

```yaml
# tempo.yaml
stream_over_http_enabled: true

server:
  http_listen_port: 3200        # Tempo API port
  grpc_listen_port: 9095        # Tempo gRPC port

# ─── How traces come in ───
distributor:
  receivers:
    otlp:
      protocols:
        grpc:
          endpoint: 0.0.0.0:4317   # OTLP gRPC
        http:
          endpoint: 0.0.0.0:4318   # OTLP HTTP
    jaeger:
      protocols:
        thrift_compact:
          endpoint: 0.0.0.0:6831   # Jaeger compact (UDP)
        grpc:
          endpoint: 0.0.0.0:14250  # Jaeger gRPC
    zipkin:
      endpoint: 0.0.0.0:9411      # Zipkin

# ─── Ingester settings ───
ingester:
  max_block_duration: 5m         # Max time before flushing a block
  max_block_bytes: 1073741824    # 1GB max block size
  trace_idle_period: 10s         # Flush trace after 10s of no new spans
  flush_check_period: 10s

# ─── Compactor settings ───
compactor:
  compaction:
    block_retention: 48h         # Keep blocks for 48 hours
    compacted_block_retention: 1h
    compaction_window: 1h

# ─── Storage configuration ───
storage:
  trace:
    backend: local               # For development; use s3/gcs/azure in production
    local:
      path: /var/tempo/traces
    wal:
      path: /var/tempo/wal       # Write-Ahead Log for crash recovery
    block:
      bloom_filter_false_positive: 0.05  # 5% false positive rate
      v2_index_downsample_bytes: 1000
      v2_encoding: zstd          # Compression algorithm

# ─── Metrics generator ───
# Generates RED metrics from trace data (like spanmetrics in OTel Collector)
metrics_generator:
  registry:
    external_labels:
      source: tempo
      cluster: docker-compose
  storage:
    path: /var/tempo/generator/wal
    remote_write:
      - url: http://prometheus:9090/api/v1/write
        send_exemplars: true     # Send exemplars (links from metrics → traces!)

# ─── Overrides ───
overrides:
  defaults:
    metrics_generator:
      processors: [service-graphs, span-metrics]  # Generate service graph + RED metrics
```

**Run Tempo:**

```bash
docker run -d --name tempo \
  -v ./tempo.yaml:/etc/tempo/tempo.yaml \
  -v tempo-data:/var/tempo \
  -p 3200:3200 \
  -p 4317:4317 \
  -p 4318:4318 \
  -p 9095:9095 \
  -p 6831:6831/udp \
  -p 14250:14250 \
  -p 9411:9411 \
  grafana/tempo:2.4.0 \
  -config.file=/etc/tempo/tempo.yaml
```

```yaml
    access: proxy
    url: http://tempo:3200
    uid: tempo
    jsonData:
      httpMethod: GET
      tracesToLogs:
        datasourceUid: loki
        tags: ['service.name', 'k8s.pod.name']

        filterByTraceID: true
        filterBySpanID: false
      tracesToMetrics:
        datasourceUid: prometheus
        tags: [{ key: 'service.name', value: 'service' }]
        queries:
          - name: 'Request Rate'
            query: 'sum(rate(traces_spanmetrics_calls_total{$$__tags}[1m]))'
          - name: 'Error Rate'
            query: 'sum(rate(traces_spanmetrics_calls_total{$$__tags, status_code="STATUS_CODE_ERROR"}[1m]))'
      serviceMap:
        datasourceUid: prometheus
      nodeGraph:
        enabled: true
      lokiSearch:
        datasourceUid: loki

  # ─── Prometheus (Metrics) ───
  - name: Prometheus
    type: prometheus
    access: proxy
    url: http://prometheus:9090
    uid: prometheus
    jsonData:
      exemplarTraceIdDestinations:
        - name: traceID
          datasourceUid: tempo

  # ─── Loki (Logs) ───
  - name: Loki
    type: loki
    access: proxy
    url: http://loki:3100
    uid: loki
    jsonData:
      derivedFields:
        - datasourceUid: tempo
          matcherRegex: '"traceID":"(\w+)"'
          name: TraceID
          url: '$${__value.raw}'
```

**What does this data source config do?**

This is where the **magic** happens! It connects all three signals:

1. **Traces → Logs**: Click a trace in Tempo → see the related logs in Loki
2. **Traces → Metrics**: Click a trace → see the related metrics in Prometheus
3. **Metrics → Traces**: Click a metric exemplar → jump to the exact trace
4. **Logs → Traces**: Click a trace ID in a log → jump to the trace in Tempo
5. **Service Map**: Auto-generated service dependency graph from trace data

## 🔍 TraceQL — The Trace Query Language

TraceQL is Tempo's query language. Think of it like **PromQL but for traces**.

### Basic Syntax

```
{ <spanset filter> } | <pipeline operators>
```

### Finding Traces by Attribute

```traceql
# Find all traces from order-service
{ resource.service.name = "order-service" }

# Find traces with HTTP 500 errors
{ span.http.response.status_code = 500 }

# Find slow database queries (> 1 second)
{ span.db.system = "postgresql" && duration > 1s }

# Find traces with errors
{ status = error }

# Find traces from a specific endpoint
{ span.http.route = "/api/orders" && span.http.request.method = "POST" }
```

### Combining Conditions

```traceql
# Find error traces in the payment service that took more than 500ms
{ resource.service.name = "payment-service" && status = error && duration > 500ms }

# Find traces where any span has a specific user
{ span.user.id = "user-12345" }

# Find traces that touched both order-service AND payment-service
{ resource.service.name = "order-service" } && { resource.service.name = "payment-service" }
```

### Aggregate Queries

```traceql
# Average duration of spans in order-service
{ resource.service.name = "order-service" } | avg(duration)

# Count of error spans per service
{ status = error } | count() by(resource.service.name)

# P95 latency of database queries
{ span.db.system = "postgresql" } | quantile_over_time(duration, 0.95)

# Find the slowest spans
{ resource.service.name = "order-service" } | max(duration)
```

### Structural Queries (Trace Shape)

```traceql
# Find traces where order-service is a PARENT of payment-service
{ resource.service.name = "order-service" } >> { resource.service.name = "payment-service" }

# Find traces where order-service is a DIRECT parent (child) of payment-service
{ resource.service.name = "order-service" } > { resource.service.name = "payment-service" }

# Find traces where a service call took longer than its parent
{ duration > 1s } > { duration > 2s }

# Find traces that are at least 5 spans deep
{ } > { } > { } > { } > { }
```

### Using TraceQL in Grafana

1. Open Grafana → **Explore**
2. Select **Tempo** as the data source
3. Switch to **TraceQL** tab
4. Type your query:

```traceql
{ resource.service.name = "order-service" && span.http.response.status_code >= 500 && duration > 1s }
```

5. Click **Run Query** → See matching traces!

## ☸️ Tempo on Kubernetes (Production Setup)

### Using Helm

```bash
# Add Grafana Helm repo
helm repo add grafana https://grafana.github.io/helm-charts
helm repo update

# Install Tempo (single-binary mode)
helm install tempo grafana/tempo \
  --namespace observability \
  --create-namespace \
  --values tempo-values.yaml
```

**Helm values for single-binary:**

```yaml
# tempo-values.yaml (single-binary mode)
tempo:
  storage:
    trace:
      backend: s3
      s3:
        bucket: my-tempo-traces
        endpoint: s3.amazonaws.com
        region: us-east-1
        access_key: ${AWS_ACCESS_KEY_ID}
        secret_key: ${AWS_SECRET_ACCESS_KEY}
  
  receivers:
    otlp:
      protocols:
        grpc:
          endpoint: 0.0.0.0:4317
        http:
          endpoint: 0.0.0.0:4318

  metricsGenerator:
    enabled: true
    remoteWriteUrl: http://prometheus.observability:9090/api/v1/write

persistence:
  enabled: true
  size: 10Gi

resources:
  requests:
    cpu: 500m
    memory: 1Gi
  limits:
    cpu: 2
    memory: 4Gi
```

**For large-scale production, use Tempo Distributed (microservices mode):**

```bash
helm install tempo grafana/tempo-distributed \
  --namespace observability \
  --create-namespace \
  --values tempo-distributed-values.yaml
```

```yaml

  replicas: 2
  resources:
    requests:
      cpu: 500m
      memory: 512Mi

# ─── Querier ───
querier:
  replicas: 2
  resources:
    requests:
      cpu: 500m
      memory: 1Gi

# ─── Metrics Generator ───
metricsGenerator:
  enabled: true
  replicas: 1
  config:
    registry:
      external_labels:
        cluster: production
    storage:
      remote_write:
        - url: http://mimir.observability:9009/api/v1/push
          send_exemplars: true

# ─── Gateway (optional, for multi-tenancy) ───
gateway:
  enabled: true
  replicas: 2

# ─── Memcached (caching layer) ───
memcached:
  enabled: true
  replicas: 2
```

```yaml
      bucket: my-tempo-bucket
```

**Google Cloud Storage (GCS):**
```yaml
storage:
  trace:
    backend: gcs
    gcs:
      bucket_name: my-tempo-bucket
      # Uses Application Default Credentials
      # On GKE: use Workload Identity
```

**Azure Blob Storage:**
```yaml
storage:
  trace:
    backend: azure
    azure:
      container_name: my-tempo-container
      storage_account_name: myaccount
      storage_account_key: ""
      # Or use Managed Identity
```

**MinIO (self-hosted S3-compatible):**

```yaml
storage:

    backend: s3
    s3:
      bucket: tempo

```

### Retention

```yaml
compactor:
  compaction:
    block_retention: 72h  # Keep traces for 3 days
    
    # More aggressive for high-volume environments:
    # block_retention: 24h  # Keep only 1 day
```

### Multi-Tenancy

Tempo supports multi-tenancy — different teams/environments in the same Tempo instance:

```yaml
# Send tenant ID via HTTP header
multitenancy_enabled: true

# In your OTel Collector config:
exporters:
  otlp/tempo:
    endpoint: tempo:4317
    headers:
      X-Scope-OrgID: "team-backend"  # Tenant identifier
```

### Caching

For better query performance, add caching:

```yaml
storage:
  trace:
    cache: memcached
    memcached:
      host: memcached
      service: memcached-client
      timeout: 500ms
    
    # Alternative: Redis
    # cache: redis
    # redis:
    #   endpoint: redis:6379
    
    # Search performance
    search:
      cache_control:
        footer: true  # Cache block footers (big performance win!)
```

## 🔄 Migrating from Jaeger to Tempo

If you're currently using Jaeger and want to switch to Tempo, here's the migration path:

### Step 1: Keep Jaeger Running, Add Tempo in Parallel

```yaml
# OTel Collector config — send traces to BOTH
exporters:
  otlp/jaeger:
    endpoint: jaeger:4317
  otlp/tempo:
    endpoint: tempo:4317

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch]
      exporters: [otlp/jaeger, otlp/tempo]  # Fan-out to both!
```

### Step 2: Verify Traces in Tempo

- Open Grafana → Add Tempo as a data source
- Query traces and compare with Jaeger UI
- Make sure everything looks correct

### Step 3: Switch Fully to Tempo

```yaml
# Remove Jaeger exporter
exporters:
  otlp/tempo:
    endpoint: tempo:4317

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch]
      exporters: [otlp/tempo]  # Only Tempo now
```

### Step 4: Decommission Jaeger

Turn off Jaeger and its Elasticsearch/Cassandra backend. Enjoy the cost savings!

## 📊 Tempo Metrics Generator

One of Tempo's most powerful features is the **metrics generator** — it automatically creates metrics FROM your traces:

### Service Graphs

Generates metrics about service-to-service communication:

```promql
# Request rate between services
traces_service_graph_request_total{
  client="order-service",
  server="payment-service"
}

# Failed request rate
traces_service_graph_request_failed_total{
  client="order-service",
  server="payment-service"
}

# Request duration
traces_service_graph_request_server_seconds_bucket{
  client="order-service",
  server="payment-service"
}
```

### Span Metrics (RED Metrics)

Generates Rate, Errors, and Duration metrics from spans:

```promql
# Request rate by service and operation
rate(traces_spanmetrics_calls_total{
  service="order-service",
  span_name="POST /api/orders"
}[5m])

# Error rate
rate(traces_spanmetrics_calls_total{
  service="order-service",
  status_code="STATUS_CODE_ERROR"
}[5m])

# Duration histogram
histogram_quantile(0.99,
  rate(traces_spanmetrics_duration_seconds_bucket{
    service="order-service"
  }[5m])
)
```

These metrics come with **exemplars** — links back to the actual traces! So when you see a latency spike, you can click through to the exact trace that caused it.

## 🆚 Tempo vs Jaeger vs Zipkin

| Feature | Tempo | Jaeger | Zipkin |
|---|---|---|---|
| **Created By** | Grafana Labs | Uber | Twitter |
| **Storage** | Object Storage (S3/GCS) | Elasticsearch/Cassandra | Elasticsearch/MySQL/Cassandra |
| **Cost at Scale** | Very Low | High | Moderate |
| **Query Language** | TraceQL | Tags + Duration | Tags + Duration |
| **UI** | Grafana | Built-in Jaeger UI | Built-in Zipkin UI |
| **Indexing** | Minimal (bloom filters) | Full indexing | Full indexing |
| **CNCF** | No (Grafana project) | Yes (Graduated) | No |
| **Multi-tenancy** | Built-in | Limited | No |
| **Metrics from Traces** | Built-in generator | No | No |
| **Logs Correlation** | Native with Loki | Manual | No |
| **Best For** | Cost-effective, Grafana users | Feature-rich, standalone | Simple, lightweight |

### When to Choose Tempo

- You're already using (or plan to use) **Grafana, Loki, or Prometheus**
- You want the **lowest possible cost** at scale
- You care about **traces ↔ logs ↔ metrics correlation**
- You use **object storage** (S3, GCS, etc.)
- You need **multi-tenancy**

### When to Choose Jaeger

- You need a **standalone tracing solution** (no Grafana dependency)
- You need the **Jaeger UI** specifically
- You're already invested in Elasticsearch/Cassandra
- You need **CNCF graduated** project status for compliance

## 🏆 Best Practices

### 1. Use Object Storage, Not Local Disk (in Production)

```yaml
# ❌ Don't do this in production
storage:
  trace:
    backend: local

# ✅ Use object storage
storage:
  trace:
    backend: s3
    s3:
      bucket: my-tempo-traces
```

### 2. Enable the Metrics Generator

```yaml
# Get RED metrics + service graphs for FREE from your traces
metrics_generator:
  storage:
    remote_write:
      - url: http://prometheus:9090/api/v1/write
        send_exemplars: true  # Don't forget exemplars!
  
overrides:
  defaults:
    metrics_generator:
      processors: [service-graphs, span-metrics]
```

### 3. Set Appropriate Retention

```yaml
# Don't keep traces forever — they add up fast
compactor:
  compaction:
    block_retention: 72h  # 3 days is usually enough
```

### 4. Use the OTel Collector in Front of Tempo

```
App → OTel Collector → Tempo

Why?
- Collector can batch, sample, and filter before Tempo
- Protects Tempo from sudden traffic spikes
- Lets you add processing (remove PII, add attributes, etc.)
```

### 5. Configure Grafana Data Source Links

Set up the **Traces → Logs → Metrics** connections in Grafana data sources. This is where Tempo really shines. Without these links, you're missing the biggest benefit.

### 6. Use TraceQL for Investigation

```traceql
# Instead of scrolling through thousands of traces, query smartly:

# Find all errors in the last hour
{ status = error && duration > 500ms }

# Find traces from a specific user
{ span.user.id = "user-12345" }

# Find traces where payment took longer than inventory
{ resource.service.name = "payment-service" && duration > 1s } > { resource.service.name = "inventory-service" }
```

## 🧠 Quick Revision

| Concept | What It Means |
|---|---|
| **Tempo** | Open-source tracing backend that stores traces in cheap object storage |
| **Object Storage** | S3/GCS/Azure Blob — dramatically cheaper than Elasticsearch |
| **Bloom Filters** | Data structure for fast "does trace exist?" lookups without reading all data |
| **TraceQL** | Query language for searching traces (like PromQL for metrics) |
| **Distributor** | Receives spans, routes them to the right Ingester |
| **Ingester** | Buffers spans, creates blocks, flushes to object storage |
| **Compactor** | Merges small blocks into larger ones for efficiency |
| **Querier** | Reads blocks from storage to answer queries |
| **Metrics Generator** | Creates RED metrics and service graphs from trace data |
| **Exemplars** | Links from metrics data points to specific traces |
| **Service Graph** | Auto-generated map of how services communicate |
