## 1. What is Incident Management?

**Incident management** is the process of detecting, responding to, and resolving unplanned disruptions or outages in your services.

Think of it like the emergency response system for your software. When something goes wrong:

1. **Detect** — An alert fires (from Prometheus, Grafana, Datadog, etc.)
2. **Notify** — The right person gets paged (phone call, SMS, push notification)
3. **Respond** — The engineer acknowledges and starts investigating
4. **Resolve** — The issue is fixed, the incident is closed
5. **Learn** — A postmortem is conducted to prevent it from happening again

Without a proper incident management tool, this process looks like:
- Alerts go to a Slack channel
- Nobody notices for 45 minutes
- Three people start investigating the same thing independently
- Nobody updates the stakeholders
- After it's resolved, nobody writes down what happened

**With a tool like PagerDuty or Opsgenie:**
- Alert fires → the on-call engineer's phone rings in 30 seconds
- If they don't respond in 5 minutes → the backup gets paged
- The incident is tracked, timestamped, and documented
- Stakeholders get automatic status updates
- After resolution, a postmortem is automatically generated

## 2. Why You Need an Incident Management Tool

Can't you just use Slack and email? Technically, yes. But here's why dedicated tools are essential:

| Problem | Slack/Email | Incident Management Tool |
|---------|------------|-------------------------|
| **Reliability of notifications** | Messages can be missed, muted, or buried | Phone calls, SMS, push — multiple channels until acknowledged |
| **Escalation** | Manual — "Hey, did anyone see that alert?" | Automatic — if Person A doesn't respond, Person B gets paged |
| **On-call scheduling** | Spreadsheet or Google Calendar | Built-in schedules with overrides, rotations, and time zones |
| **Tracking** | Messages disappear in the Slack stream | Full incident timeline, audit trail, MTTA/MTTR metrics |
| **Coordination** | Chaos in a Slack thread | War rooms, roles (commander, communicator), runbook links |
| **Postmortem** | Rarely happens | Auto-generated timeline, built-in templates |

## 4. Key Concepts (Common to All Tools)

Before we dive into specific tools, let's understand the concepts that all of them share:

### 4.1 Incidents

An **incident** is a record of something going wrong. It captures:

- What triggered it (the alert)
- When it was detected
- Who was notified
- When it was acknowledged
- When it was resolved
- Total duration (MTTR — Mean Time To Resolve)

### 4.2 Alerts vs Incidents

This confuses a lot of people. Here's the difference:

```
ALERT:    A signal from your monitoring system
          "Hey, error rate is above 5%!"

INCIDENT: A tracked event in your incident management tool
          "Incident #4521: Payment API degradation — P1"

Relationship:
  - One ALERT can create one INCIDENT
  - Multiple ALERTS can be grouped into one INCIDENT
    (e.g., 10 different alerts about the same root cause)
  - An INCIDENT can exist without an alert
    (e.g., a customer reports an issue)
```

### 4.3 On-Call Schedule

An on-call schedule defines **who is responsible** for responding to incidents at any given time.

```
Week 1:  Alice   (Mon 9am → Mon 9am)
Week 2:  Bob     (Mon 9am → Mon 9am)
Week 3:  Charlie (Mon 9am → Mon 9am)
Week 4:  Alice   (Mon 9am → Mon 9am)
↻ Rotation repeats
```

**Key features:**
- **Rotation** — People take turns (weekly, daily, etc.)
- **Overrides** — "I'm on vacation Tuesday, Bob covers for me"
- **Time zones** — Follow-the-sun schedules (US team → EU team → APAC team)
- **Layers** — Primary on-call + secondary backup

### 4.4 Escalation Policy

An escalation policy defines **what happens** when nobody responds to an alert.

```
┌─────────────────────────────────────────────────┐
│            ESCALATION POLICY                    │
│            "payments-team"                      │
├─────────────────────────────────────────────────┤
│                                                 │
│  Step 1: Page PRIMARY on-call                   │
│          Wait 5 minutes                         │
│          ↓ (if not acknowledged)                │
│                                                 │
│  Step 2: Page SECONDARY on-call                 │
│          Wait 5 minutes                         │
│          ↓ (if not acknowledged)                │
│                                                 │
│  Step 3: Page TEAM LEAD                         │
│          Wait 10 minutes                        │
│          ↓ (if not acknowledged)                │
│                                                 │
│  Step 4: Page ENGINEERING MANAGER               │
│          + Send to #incidents Slack channel      │
│                                                 │
│  Step 5 (final): Page VP OF ENGINEERING         │
│                                                 │
└─────────────────────────────────────────────────┘
```

### 4.5 Services

A **service** in incident management represents a logical component of your system:

- `payment-service`
- `user-service`
- `database-primary`
- `kubernetes-cluster-prod`

Each service can have its own:
- On-call schedule
- Escalation policy
- Integrations (which monitoring tools feed into it)
- Runbooks

### 4.6 Alert Routing / Event Rules

**Routing rules** decide which service/team an incoming alert goes to:

```
IF alert contains "payment" → Route to payments-team
IF alert severity = "critical" → Route to primary on-call
IF alert contains "database" AND severity = "warning" → Route to DBA team
IF alert source = "CloudWatch" AND region = "us-east-1" → Route to US team
```

### 4.7 Notification Channels

How the on-call person gets notified:

| Channel | Use Case | Urgency |
|---------|----------|---------|
| **Phone call** | Critical incidents, must wake someone up | 🔴 Highest |
| **SMS** | Critical to high, quick glanceable notification | 🟠 High |
| **Push notification** | App-based alert, can include context | 🟡 Medium |
| **Email** | Non-urgent, informational | 🟢 Low |
| **Slack/Teams** | Team awareness, non-paging incidents | 🟢 Low |

### 4.8 MTTA and MTTR

Two critical metrics for incident management:

```
MTTA = Mean Time To Acknowledge
       "How long until someone said 'I'm looking at this'?"
       
MTTR = Mean Time To Resolve
       "How long from detection to resolution?"

Example:
  Alert fired:     2:00 AM
  Acknowledged:    2:03 AM  → MTTA = 3 minutes
  Resolved:        2:45 AM  → MTTR = 45 minutes
```

## 5. PagerDuty — Deep Dive

### What is PagerDuty?

PagerDuty is the **most popular** incident management platform. It's been around since 2009 and is considered the industry standard.

**Website:** pagerduty.com

### Key Features

- On-call scheduling with rotations and overrides
- Multi-level escalation policies
- 700+ integrations (Prometheus, Grafana, AWS, Datadog, etc.)
- Event intelligence (ML-based alert grouping and noise reduction)
- Status pages (communicate with customers during outages)
- Postmortem/Post-incident analysis
- Analytics (MTTA, MTTR, alert volume trends)
- Mobile app (iOS/Android) for on-call management
- Automation actions (run scripts automatically when incidents trigger)

### Setting Up PagerDuty

#### Step 1: Create a Service

A service represents a component of your infrastructure.

Navigate to: **Services → + New Service**

```
Service Name: payment-service-production
Description: Handles all payment processing for the platform
Escalation Policy: payments-team-escalation
Integration: Prometheus AlertManager (Events API v2)
```

#### Step 2: Create an On-Call Schedule

Navigate to: **People → On-Call Schedules → + New Schedule**

```yaml
Schedule Name: payments-primary
Time Zone: America/New_York

Rotation:
  Type: Weekly
  Handoff Time: Monday 9:00 AM

Members:
  - alice@company.com
  - bob@company.com  
  - charlie@company.com

# This means:
# Week 1: Alice is on-call
# Week 2: Bob is on-call
# Week 3: Charlie is on-call
# Then it loops back to Alice
```

#### Step 3: Create an Escalation Policy

Navigate to: **People → Escalation Policies → + New Escalation Policy**

```yaml
Policy Name: payments-team-escalation

Rules:
  - Level 1:
      Target: payments-primary (schedule)
      Timeout: 5 minutes
      
  - Level 2:
      Target: payments-secondary (schedule)
      Timeout: 10 minutes
      
  - Level 3:
      Target: engineering-manager@company.com (user)
      Timeout: 15 minutes

Repeat: Yes (if nobody acknowledges after all levels, start over)
Number of Repeats: 3
```

#### Step 4: Integrate with Prometheus AlertManager

First, get your PagerDuty **Integration Key** (also called routing key):
- Go to your Service → Integrations → + Add Integration → Events API v2
- Copy the **Integration Key**

Then configure AlertManager:

```yaml
# alertmanager.yml

global:
  resolve_timeout: 5m

route:
  # Default receiver — catch-all
  receiver: 'default-slack'
  
  # Group alerts to reduce noise
  group_by: ['alertname', 'service', 'severity']
  group_wait: 30s        # Wait 30s to group related alerts
  group_interval: 5m     # Wait 5m before sending updates for the same group
  repeat_interval: 4h    # Re-send unresolved alerts every 4 hours
  
  routes:
    # Critical alerts → PagerDuty (page someone immediately)
    - match:
        severity: critical
      receiver: 'pagerduty-critical'
      continue: true        # Also send to Slack for visibility
    
    # Warning alerts → Slack only (don't wake anyone up)
    - match:
        severity: warning
      receiver: 'team-slack'

receivers:
  # PagerDuty receiver for critical alerts
  - name: 'pagerduty-critical'
    pagerduty_configs:
      - routing_key: 'YOUR_PAGERDUTY_INTEGRATION_KEY'  # Replace with actual key
        severity: 'critical'
        # Map AlertManager labels to PagerDuty fields
        description: '{{ .CommonAnnotations.summary }}'
        details:
          firing: '{{ .Alerts.Firing | len }}'
          service: '{{ .CommonLabels.service }}'
          runbook: '{{ .CommonAnnotations.runbook_url }}'
  
  # Slack receiver for warnings and general visibility
  - name: 'team-slack'
    slack_configs:
      - api_url: 'https://hooks.slack.com/services/YOUR/SLACK/WEBHOOK'
        channel: '#alerts-payments'
        title: '{{ .CommonAnnotations.summary }}'
        text: '{{ .CommonAnnotations.description }}'

  - name: 'default-slack'
    slack_configs:
      - api_url: 'https://hooks.slack.com/services/YOUR/SLACK/WEBHOOK'
        channel: '#alerts-general'
```

**Explanation of the AlertManager config:**

- `route` — The top-level routing tree. Alerts flow through this like a decision tree.
- `group_by` — Groups alerts with the same `alertname`, `service`, and `severity` together, so you get ONE notification for related alerts, not 50.
- `group_wait: 30s` — Waits 30 seconds before sending the first notification, to collect and group any related alerts that fire around the same time.
- `routes` — Child routes that match specific labels. The first match wins (unless `continue: true`).
- `match: severity: critical` — Only critical alerts go to PagerDuty.
- `continue: true` — Even after matching this route, keep checking other routes too (so critical alerts also go to Slack).
- `pagerduty_configs` — Sends the alert to PagerDuty using their Events API v2.
- `routing_key` — Your PagerDuty integration key (ties this alert to a specific PagerDuty service).

#### Step 5: Using the PagerDuty API

PagerDuty also has a powerful REST API for automation:

```bash
# Trigger an incident via API
curl -X POST https://events.pagerduty.com/v2/enqueue \
  -H "Content-Type: application/json" \
  -d '{
    "routing_key": "YOUR_ROUTING_KEY",
    "event_action": "trigger",
    "dedup_key": "payment-db-connection-failure",
    "payload": {
      "summary": "Payment database connection pool exhausted",
      "severity": "critical",
      "source": "payment-service-prod",
      "component": "database",
      "group": "payments",
      "custom_details": {
        "active_connections": 100,
        "max_connections": 100,
        "affected_users": 1500,
        "runbook": "https://wiki.internal/runbooks/db-connection-pool"
      }
    },
    "links": [
      {
        "href": "https://grafana.internal/d/payments",
        "text": "Grafana Dashboard"
      }
    ]
  }'
```

**What each field does:**

- `routing_key` — Determines which PagerDuty service this alert goes to
- `event_action` — Can be `trigger` (create incident), `acknowledge`, or `resolve`
- `dedup_key` — De-duplication key. If another alert comes in with the same key, PagerDuty adds it to the existing incident instead of creating a new one
- `payload.summary` — The title of the incident
- `payload.severity` — `critical`, `error`, `warning`, or `info`
- `payload.source` — Which system generated this alert
- `custom_details` — Any extra data that will help the engineer investigating

```bash
# Resolve an incident via API (using the same dedup_key)
curl -X POST https://events.pagerduty.com/v2/enqueue \
  -H "Content-Type: application/json" \
  -d '{
    "routing_key": "YOUR_ROUTING_KEY",
    "event_action": "resolve",
    "dedup_key": "payment-db-connection-failure"
  }'
```

### PagerDuty Event Intelligence (AIOps)

PagerDuty uses machine learning to reduce noise:

| Feature | What It Does |
|---------|-------------|
| **Alert Grouping** | Automatically groups related alerts into a single incident (e.g., 20 alerts about the same outage → 1 incident) |
| **Noise Reduction** | Suppresses non-actionable, transient alerts |
| **Similar Incidents** | Shows past incidents similar to the current one (helps with resolution) |
| **Intelligent Triage** | Suggests which team/person should handle the incident based on past patterns |

### Setting Up Opsgenie

#### Step 1: Create a Team

```
Team Name: payments-team
Members:
  - alice@company.com (Admin)
  - bob@company.com (User)
  - charlie@company.com (User)
```

#### Step 2: Create an Integration

Navigate to: **Teams → payments-team → Integrations → Add Integration**

For Prometheus AlertManager:
1. Select "Prometheus" from the integration list
2. Copy the **API Key** generated
3. Note the **API URL** (e.g., `https://api.opsgenie.com/v2/alerts`)

#### Step 3: Configure AlertManager for Opsgenie

```yaml
# alertmanager.yml — Opsgenie configuration

receivers:
  - name: 'opsgenie-critical'
    opsgenie_configs:
      - api_key: 'YOUR_OPSGENIE_API_KEY'       # From Step 2
        message: '{{ .CommonAnnotations.summary }}'
        description: '{{ .CommonAnnotations.description }}'
        priority: 'P1'
        tags: 'production,critical,{{ .CommonLabels.service }}'
        # Map to the right Opsgenie team
        responders:
          - type: team
            name: payments-team
        details:
          service: '{{ .CommonLabels.service }}'
          severity: '{{ .CommonLabels.severity }}'
          runbook: '{{ .CommonAnnotations.runbook_url }}'
          dashboard: '{{ .CommonAnnotations.dashboard_url }}'

  - name: 'opsgenie-warning'
    opsgenie_configs:
      - api_key: 'YOUR_OPSGENIE_API_KEY'
        message: '{{ .CommonAnnotations.summary }}'
        priority: 'P3'
        responders:
          - type: team
            name: payments-team
```

**Explanation:**

- `api_key` — Your Opsgenie API key for authentication.
- `message` — The alert title (pulled from Prometheus annotation `summary`).
- `priority` — Opsgenie uses P1 (highest) to P5 (lowest).
- `tags` — Comma-separated tags for filtering and grouping.
- `responders` — Which team or person should be notified. `type: team` routes to the entire team's on-call schedule.
- `details` — Extra key-value data that appears in the alert details page.

#### Step 4: Opsgenie Alert Priorities

| Priority | Meaning | Default Notification |
|----------|---------|---------------------|
| **P1** | Critical | Phone call + SMS + Push + Email |
| **P2** | High | SMS + Push + Email |
| **P3** | Moderate | Push + Email |
| **P4** | Low | Email only |
| **P5** | Informational | Email only (or suppressed) |

#### Step 5: Using the Opsgenie API

```bash
# Create an alert via Opsgenie API
curl -X POST https://api.opsgenie.com/v2/alerts \
  -H "Content-Type: application/json" \
  -H "Authorization: GenieKey YOUR_OPSGENIE_API_KEY" \
  -d '{
    "message": "Database connection pool exhausted",
    "alias": "db-conn-pool-prod",
    "description": "All 100 database connections are in use. New requests are failing.",
    "responders": [
      {"type": "team", "name": "database-team"}
    ],
    "tags": ["database", "production", "critical"],
    "priority": "P1",
    "details": {
      "active_connections": "100",
      "max_connections": "100",
      "affected_service": "payment-service",
      "runbook": "https://wiki.internal/runbooks/db-pool"
    }
  }'
```

**What each field does:**

- `message` — The alert title (what appears on the phone notification)
- `alias` — A unique identifier for deduplication (like PagerDuty's `dedup_key`). If another alert with the same alias comes in, it's added to the existing alert instead of creating a new one.
- `description` — Detailed information about the alert
- `responders` — Who to notify (can be a team, a user, an escalation policy, or a schedule)
- `tags` — Labels for filtering and searching
- `priority` — P1 through P5
- `details` — Any key-value pairs for additional context

```bash
# Close (resolve) an alert
curl -X POST https://api.opsgenie.com/v2/alerts/db-conn-pool-prod/close \
  -H "Content-Type: application/json" \
  -H "Authorization: GenieKey YOUR_OPSGENIE_API_KEY" \
  -d '{
    "note": "Increased connection pool size from 100 to 200. Monitoring."
  }'
```

## 7. VictorOps / Splunk On-Call — Deep Dive

### What is VictorOps?

VictorOps was an incident management tool that was **acquired by Splunk** and rebranded as **Splunk On-Call**. If your organization uses Splunk for log management, Splunk On-Call integrates tightly with it.

> **Note:** Splunk was subsequently acquired by Cisco in 2024. The product continues to exist but the branding and integration landscape may evolve.

**Website:** splunk.com/en_us/products/on-call.html

### Key Features

### Key Concept: Routing Keys

VictorOps uses **routing keys** to direct alerts to the right team. Think of them as labels that map alerts to escalation policies.

```
Monitoring Tool sends alert with routing_key = "payments"
  → VictorOps looks up "payments" routing key
  → Routes to "payments-team" escalation policy
  → Pages the on-call person on that policy
```

### Setting Up VictorOps / Splunk On-Call

#### Step 1: Create a Routing Key

Navigate to: **Settings → Routing Keys**

```
Routing Key: payments
Escalation Policy: payments-team-escalation
```

#### Step 2: Integrate with Prometheus AlertManager

Get your REST API endpoint and routing key from VictorOps.

```yaml
# alertmanager.yml — VictorOps configuration

receivers:
  - name: 'victorops-critical'
    victorops_configs:
      - api_key: 'YOUR_VICTOROPS_API_KEY'
        routing_key: 'payments'
        entity_display_name: '{{ .CommonAnnotations.summary }}'
        state_message: '{{ .CommonAnnotations.description }}'
        message_type: 'CRITICAL'
```

**Explanation:**

- `api_key` — Your VictorOps API key for authentication.
- `routing_key` — Determines which team/escalation policy handles the alert.
- `entity_display_name` — The title of the incident.
- `state_message` — Detailed description of what's happening.
- `message_type` — Can be `CRITICAL`, `WARNING`, `INFO`, or `ACKNOWLEDGEMENT`.

#### Step 3: Using the VictorOps API

```bash
# Send an alert to VictorOps
curl -X POST \
  'https://alert.victorops.com/integrations/generic/20131114/alert/YOUR_API_KEY/payments' \
  -H "Content-Type: application/json" \
  -d '{
    "message_type": "CRITICAL",
    "entity_id": "db-connection-pool-exhausted",
    "entity_display_name": "Database Connection Pool Exhausted",
    "state_message": "All 100 database connections are in use on production.",
    "monitoring_tool": "custom-script",
    "service": "payment-service",
    "runbook_url": "https://wiki.internal/runbooks/db-pool"
  }'
```

**Fields explained:**

- The URL path includes the routing key (`payments`) at the end
- `message_type` — `CRITICAL` pages the on-call, `WARNING` creates an alert but doesn't page, `INFO` is just informational, `RECOVERY` resolves the incident
- `entity_id` — Deduplication key. Same concept as PagerDuty's `dedup_key`
- `entity_display_name` — What the engineer sees as the incident title

```bash
# Resolve the alert
curl -X POST \
  'https://alert.victorops.com/integrations/generic/20131114/alert/YOUR_API_KEY/payments' \
  -H "Content-Type: application/json" \
  -d '{
    "message_type": "RECOVERY",
    "entity_id": "db-connection-pool-exhausted",
    "entity_display_name": "Database Connection Pool Recovered",
    "state_message": "Connection pool utilization back to normal (45/200)."
  }'
```

## 9. Incident Lifecycle — From Alert to Resolution

Every incident goes through a standard lifecycle. Understanding this helps you set up your tools properly.

```
┌──────────────────────────────────────────────────────────────────┐
│                    INCIDENT LIFECYCLE                            │
└──────────────────────────────────────────────────────────────────┘

   DETECTION         TRIAGE          RESPONSE        RESOLUTION
   ─────────        ──────          ────────        ──────────
       │                │               │               │
   ┌───▼───┐       ┌────▼────┐     ┌────▼────┐     ┌───▼────┐
   │Alert  │       │Prioritize│    │Investigate│    │Fix &   │
   │Fires  │──────▶│& Assign  │───▶│& Diagnose │───▶│Verify  │
   └───────┘       └──────────┘    └──────────┘    └────┬────┘
                                                        │
   ┌──────────────────────────────────────────────────────▼──────┐
   │                      POST-INCIDENT                          │
   │                                                              │
   │  1. Write postmortem                                        │
   │  2. Identify root cause                                     │
   │  3. Create action items                                     │
   │  4. Share learnings with the team                           │
   └──────────────────────────────────────────────────────────────┘


Incident States (in the tools):

  ┌───────────┐     ┌──────────────┐     ┌───────────┐
  │ TRIGGERED │────▶│ ACKNOWLEDGED │────▶│ RESOLVED  │
  └───────────┘     └──────────────┘     └───────────┘
       │                                       │
       │            ┌──────────────┐           │
       └───────────▶│ AUTO-RESOLVED│◀──────────┘
                    │ (if alert    │
                    │  clears)     │
                    └──────────────┘
```

### Detailed Steps

| Phase | What Happens | Who Does It | Tool Feature |
|-------|-------------|-------------|--------------|
| **1. Detection** | Monitoring tool detects an anomaly and fires an alert | Automated (Prometheus, Grafana, etc.) | Integration/webhook |
| **2. Notification** | On-call engineer is paged via phone/SMS/push | Automated (PagerDuty, Opsgenie, etc.) | Escalation policy |
| **3. Acknowledge** | Engineer says "I see this and I'm working on it" | On-call engineer | Ack button (stops escalation) |
| **4. Triage** | Determine severity and impact — P1? P2? P3? | On-call engineer / Incident Commander | Priority assignment |
| **5. Communicate** | Update stakeholders — "We're aware and investigating" | Incident Commander / Communicator | Status page, Slack |
| **6. Investigate** | Find the root cause — check dashboards, logs, traces | On-call engineer(s) | Runbooks, dashboards |
| **7. Mitigate** | Stop the bleeding — rollback, scale up, redirect traffic | On-call engineer(s) | Automation actions |
| **8. Resolve** | Confirm the issue is fixed — close the incident | On-call engineer | Resolve button |
| **9. Postmortem** | Document what happened, why, and how to prevent it | Team | Built-in postmortem tool |

## 10. Escalation Policies — Making Sure Nobody Gets Missed

### Designing Good Escalation Policies

```yaml
# Example: Multi-level escalation

Policy: "production-critical"

Level 1:  Primary On-Call Engineer
          Notify via: Phone + SMS + Push
          Wait: 5 minutes
          If no ack → escalate to Level 2

Level 2:  Secondary On-Call Engineer
          Notify via: Phone + SMS + Push
          Wait: 5 minutes
          If no ack → escalate to Level 3

Level 3:  Team Lead
          Notify via: Phone + SMS
          + Post to #incident-room Slack channel
          Wait: 10 minutes
          If no ack → escalate to Level 4

Level 4:  Engineering Manager
          Notify via: Phone
          Wait: 10 minutes
          If no ack → escalate to Level 5

Level 5:  VP of Engineering
          Notify via: Phone
          (This should basically never happen)

Repeat: Yes, loop 3 times total
After all repeats: Auto-create Jira P1 ticket
```

### Tips for Escalation Policies

1. **Keep Level 1 and 2 tight** — 5 minutes max. If someone doesn't respond in 5 min, they're probably not going to.
2. **Don't skip levels** — Always give the primary on-call the first chance to respond.
3. **Use different notification methods** — Phone call is the most reliable for waking someone up at night.
4. **Include a catch-all** — If nobody in the entire chain responds, something should happen (e.g., auto-post to a Slack channel, create a ticket).
5. **Test your escalation policies** — Run a "fire drill" once a quarter to make sure pages actually reach people.

## 11. On-Call Scheduling

### Common Rotation Patterns

#### Pattern 1: Weekly Rotation (Most Common)

```
Week 1:  Alice    (Mon 9am → Mon 9am)
Week 2:  Bob      (Mon 9am → Mon 9am)
Week 3:  Charlie  (Mon 9am → Mon 9am)
Week 4:  Diana    (Mon 9am → Mon 9am)
↻ Repeat

Best for: Teams of 3-6 people
```

#### Pattern 2: Follow-the-Sun

```
US Team  (9 AM - 5 PM PST):    Alice, Bob
EU Team  (9 AM - 5 PM CET):    Charlie, Diana
APAC Team (9 AM - 5 PM JST):   Eve, Frank

No one is ever on-call outside business hours!

Best for: Global teams that want to avoid nighttime pages
```

#### Pattern 3: Split Day/Night

```
Daytime (8 AM - 8 PM):   Regular rotation (more people)
Nighttime (8 PM - 8 AM):  Dedicated night shift or reduced rotation

Best for: Teams that can't do follow-the-sun
```

#### Pattern 4: Primary + Secondary

```
Primary:    Handles all incoming incidents
Secondary:  Backup if primary doesn't respond
            Also provides a "buddy system" for major incidents

Week 1:  Primary: Alice,   Secondary: Bob
Week 2:  Primary: Bob,     Secondary: Charlie
Week 3:  Primary: Charlie, Secondary: Alice
```

### Handling Overrides

Real life happens — vacations, sick days, personal emergencies. All tools support **overrides**:

```
Original Schedule:
  Week of June 15: Alice is on-call

Override:
  Alice is on vacation June 16-18
  Bob covers June 16 (Tue) to June 18 (Thu)

Result:
  June 15 (Mon): Alice
  June 16-18 (Tue-Thu): Bob (override)
  June 19-21 (Fri-Sun): Alice
```

## 13. Postmortems / Post-Incident Reviews

After every significant incident (P1 and P2), conduct a **blameless postmortem**.

### Blameless Culture

The goal of a postmortem is to **learn and prevent**, NOT to **blame and punish**.

```
❌ "Bob deployed bad code and caused the outage"
✅ "A code change was deployed without sufficient testing, 
    which revealed a gap in our CI/CD pipeline"
```

### Postmortem Template

```markdown
# Postmortem: [Incident Title]

## Metadata
- **Date:** 2024-03-15
- **Duration:** 47 minutes (2:13 AM - 3:00 AM UTC)
- **Severity:** P1
- **Incident Commander:** Alice
- **Authors:** Alice, Bob

## Summary
A database migration script ran during a deployment at 2:10 AM, 
locking the payments table for 47 minutes. All checkout requests 
failed during this period. Approximately 3,200 users were impacted.

## Impact
- 3,200 users unable to complete purchases
- Estimated revenue loss: $48,000
- 100% of checkout API requests returned 500 errors
- SLO breach: Availability dropped to 94.2% (target: 99.95%)
- Error budget consumed: 67% of monthly budget in one incident

## Timeline (all times UTC)
| Time | Event |
|------|-------|
| 2:10 AM | Deployment started (commit abc123) |
| 2:13 AM | Prometheus alert: checkout error rate > 5% |
| 2:13 AM | PagerDuty pages Alice (primary on-call) |
| 2:15 AM | Alice acknowledges the page |
| 2:18 AM | Alice identifies database lock in slow query log |
| 2:22 AM | Bob (author of the change) is pulled in |
| 2:25 AM | Decision to rollback the deployment |
| 2:30 AM | Rollback initiated |
| 2:45 AM | Rollback complete, but DB migration lock still held |
| 2:50 AM | DBA kills the long-running migration query |
| 2:55 AM | Checkout API recovering, error rate dropping |
| 3:00 AM | All clear — error rate back to normal |

## Root Cause
The database migration script included an `ALTER TABLE` command 
that acquired an exclusive lock on the `payments` table. This 
migration was not tested against production-scale data and took 
much longer than expected, blocking all reads and writes.

## What Went Well
- Alert fired within 3 minutes of the issue starting
- On-call engineer acknowledged within 2 minutes
- Team collaborated effectively in the incident channel
- Rollback procedure worked (eventually)

## What Went Wrong
- Migration was not tested with production-like data volumes
- No migration lock timeout was set
- CI/CD pipeline didn't flag long-running migrations
- Deployment happened during peak traffic hours

## Action Items
| Action | Owner | Priority | Due Date |
|--------|-------|----------|----------|
| Add lock_timeout to all migrations | Bob | P1 | 2024-03-20 |
| Add migration testing stage to CI/CD | Alice | P1 | 2024-03-25 |
| Move non-urgent deployments to low-traffic window | Team | P2 | 2024-03-22 |
| Add "schema change" label that triggers extra review | Charlie | P2 | 2024-03-30 |
| Test rollback procedures quarterly | Alice | P3 | 2024-04-15 |

## Lessons Learned
1. Database migrations should be tested against production-scale data
2. Set explicit lock timeouts so migrations fail fast rather than blocking
3. Consider online schema change tools (pt-online-schema-change, gh-ost)
4. Deploy during low-traffic windows when possible
```

## 15. Best Practices

### General Best Practices

```
1. 🎯 ROUTE INTELLIGENTLY
   Don't send all alerts to one person/team.
   Route based on service ownership and severity.

2. 🔇 DEDUPLICATE AGGRESSIVELY
   Use dedup_keys/aliases so that 50 alerts about 
   the same issue = 1 incident, not 50.

3. 📊 TRACK METRICS
   Monitor MTTA, MTTR, and alert volume.
   Aim for: MTTA < 5 min, MTTR < 1 hour for P1s.

4. 📚 ATTACH RUNBOOKS
   Every alert should link to a runbook explaining
   what to do. "Alert without runbook = useless alert."

5. 🔄 REVIEW REGULARLY
   Monthly: review noisy alerts and tune/remove them
   Quarterly: review on-call health (burden distribution)

6. 🧪 TEST YOUR SETUP
   Run incident drills to verify:
   - Pages actually reach people
   - Escalation works correctly
   - Runbooks are up to date
   - Communication channels are set up

7. 🤝 FAIR ON-CALL DISTRIBUTION
   Track who gets paged the most.
   Distribute the burden evenly.
   Compensate on-call fairly.

8. 📝 ALWAYS DO POSTMORTEMS
   For every P1 and P2 incident.
   Blameless. Focus on systems, not people.
```

### Integration Best Practices

```yaml
# 1. Use alert grouping to reduce noise
# PagerDuty: Service → Alert Grouping → Time-based (5 min window)
# Opsgenie: Alert deduplication via alias field

# 2. Map severities consistently across all tools
Mapping:
  Prometheus critical → PagerDuty P1 → Phone call
  Prometheus warning  → PagerDuty P3 → Slack only
  Grafana critical    → Opsgenie P1  → Phone call
  Grafana warning     → Opsgenie P3  → Email

# 3. Always include context in alerts
Required fields for every alert:
  - Summary (what's wrong)
  - Service (which service)
  - Severity (how bad)
  - Runbook URL (what to do)
  - Dashboard URL (where to look)
  - Impact (who is affected)
```
