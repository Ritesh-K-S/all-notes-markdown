## 1. What is On-Call?

**On-call** means you're the person responsible for responding to production incidents during a specific time window. When something goes wrong — your payment system crashes, your database runs out of connections, your Kubernetes cluster starts evicting pods — the on-call engineer is the first person who gets notified and is expected to respond.

Think of it like being the **designated firefighter** for your services. You're not expected to be staring at dashboards 24/7, but when the fire alarm goes off, you need to respond quickly.

### On-Call in Practice

```
Normal Day for an On-Call Engineer:

  9:00 AM   Start work, check dashboards briefly
  9:15 AM   Regular work (coding, reviews, meetings)
  12:30 PM  📱 Alert: "Latency spike on user-service"
            → Check dashboard → transient spike → auto-resolved
            → No action needed (informational alert)
  1:00 PM   Regular work continues
  5:00 PM   End of work day — still on-call though
  8:00 PM   Dinner, relaxing — phone nearby
  11:00 PM  Sleep — phone ringer on, volume up
  2:30 AM   📱📱📱 PHONE CALL: "P1 — Payment API down"
            → Wake up → Acknowledge → Open laptop
            → Follow runbook → Fix issue → Back to sleep by 3:15 AM
  9:00 AM   Next day — brief write-up of the incident
```

## 2. Why On-Call Matters

### The Business Perspective

Every minute of downtime costs money. Here's a rough idea:

| Company Size | Estimated Cost Per Minute of Downtime |
|-------------|--------------------------------------|
| Small startup | $100 - $1,000 |
| Mid-size company | $1,000 - $10,000 |
| Large enterprise | $10,000 - $100,000+ |
| Fortune 500 | $100,000 - $1,000,000+ |

Having an on-call system means the difference between a 5-minute outage (quick response) and a 2-hour outage (nobody noticed until morning).

### The Engineering Perspective

On-call creates a **feedback loop**:

```
You build it → You run it → You fix it → You make it better

This means:
- Engineers who write code also feel the pain of bad code in production
- This naturally leads to writing more reliable, observable, and well-tested code
- The "throw it over the wall to ops" mentality disappears
```

This is the core of the **DevOps philosophy** — the people who build the software are also responsible for running it.

## 3. On-Call Anti-Patterns — What NOT to Do

Before we get into best practices, let's look at common mistakes that make on-call miserable:

### Anti-Pattern 1: The "Hero" Culture

```
❌ Problem:
   One person handles ALL the alerts because they're the 
   "only one who knows the system."
   
   Result: 
   - That person burns out
   - When they leave, nobody knows anything
   - Single point of failure for knowledge

✅ Fix:
   - Document everything (runbooks!)
   - Cross-train team members
   - Rotate on-call among ALL team members
   - No one is "too senior" or "too junior" for on-call
```

### Anti-Pattern 2: Alert Storms

```
❌ Problem:
   On-call engineer gets 50+ alerts per shift.
   Most are non-actionable noise.
   
   Result:
   - Alert fatigue — starts ignoring everything
   - Critical alerts get buried under noise
   - Engineers dread on-call rotations

✅ Fix:
   - Every alert must be actionable
   - Deduplicate and group related alerts
   - Kill noisy alerts aggressively
   - Target: < 2 pages per on-call shift (outside business hours)
```

### Anti-Pattern 3: No Runbooks

```
❌ Problem:
   Engineer gets paged at 3 AM.
   Alert says: "Database connection count high"
   No runbook. No documentation. No idea what to do.
   
   Result:
   - Engineer spends 30 minutes just figuring out where to look
   - Might make things worse by trying random fixes
   - Calls someone else to help, waking up two people

✅ Fix:
   - Every alert has a linked runbook
   - Runbook has step-by-step instructions
   - A sleep-deprived, stressed engineer should be able to follow it
```

### Anti-Pattern 4: "Always On" On-Call

```
❌ Problem:
   On-call rotation is 24/7 for 2 weeks straight.
   No backup. No rest periods. No compensation.
   
   Result:
   - Chronic sleep deprivation
   - Burnout → engineers quit
   - Poor decision-making during incidents

✅ Fix:
   - Maximum 1-week rotations (3-4 days preferred)
   - Mandatory rest after night pages
   - Primary + secondary rotation
   - Compensate on-call time fairly
```

### Anti-Pattern 5: No Follow-Up

```
❌ Problem:
   Incident happens at 2 AM.
   Engineer fixes it.
   Nobody talks about it the next day.
   Same incident happens again next week.
   
   Result:
   - Recurring incidents → recurring pages
   - Engineer loses trust in the system
   - No improvement over time

✅ Fix:
   - Every P1/P2 gets a postmortem
   - Action items are tracked and completed
   - Recurring incidents get prioritized for fixing
   - On-call review meetings (weekly/biweekly)
```

## 4. Building a Healthy On-Call Culture

A healthy on-call culture is one where:

### The Core Principles

```
┌──────────────────────────────────────────────────────────────┐
│               HEALTHY ON-CALL PRINCIPLES                     │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  1. ON-CALL IS EVERYONE'S RESPONSIBILITY                     │
│     Engineers who write code also support it in production    │
│     "You build it, you run it" — Werner Vogels, AWS CTO     │
│                                                              │
│  2. ON-CALL SHOULD BE SUSTAINABLE                            │
│     If on-call is miserable, something is wrong with the     │
│     system, not the person on-call                           │
│                                                              │
│  3. ALERTS MUST BE ACTIONABLE                                │
│     If you get paged and there's nothing to do → delete      │
│     or tune that alert                                       │
│                                                              │
│  4. POSTMORTEMS ARE BLAMELESS                                │
│     "What went wrong with our SYSTEM?" not                   │
│     "WHO did this?"                                          │
│                                                              │
│  5. INVEST IN REDUCING TOIL                                  │
│     If you're doing the same manual fix repeatedly,          │
│     automate it                                              │
│                                                              │
│  6. COMPENSATE FAIRLY                                        │
│     Being available 24/7 has real impact on quality of life  │
│     Acknowledge and compensate that                          │
│                                                              │
│  7. MEASURE AND IMPROVE                                      │
│     Track on-call burden, review it regularly,               │
│     and actively work to reduce it                           │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### Google's On-Call Philosophy (from the SRE Book)

Google's Site Reliability Engineering book recommends:

- **No more than 25% of time on operational work** (on-call, tickets, toil)
- **At least 50% of time on engineering work** (building, improving, automating)
- **If on-call load is too high**, the team should push back and demand reliability improvements before taking on more services
- **Cap pages at 2 per 12-hour shift** — if you're getting more than that, the service needs to be made more reliable
- **8-12 hour on-call shifts** are ideal; 24-hour shifts lead to poor decision-making
- **After a significant night page**, the engineer should be able to take time off the next day

## 5. On-Call Scheduling Best Practices

### Rule 1: Minimum Team Size for On-Call

```
Minimum: 4-5 people in the rotation

Why?
  With 4 people on a weekly rotation:
  - Each person is on-call 1 week out of 4 (25%)
  - Allows for vacations without overloading others
  - One person can take PTO without collapsing the schedule

  With fewer than 4:
  - People are on-call too frequently
  - No room for vacations or sick days
  - Burnout is inevitable
```

### Rule 2: Rotation Length

```
Recommended: 1 week (Monday 9 AM → Monday 9 AM)

Why not longer?
  - 2+ weeks is exhausting
  - Quality of response degrades after day 5-6

Why not shorter (daily)?
  - Too many handoffs
  - Loss of context between shifts
  - Hard to track what happened

Exception: 
  Follow-the-sun rotations use 8-12 hour shifts
  (but you're only on-call during YOUR business hours)
```

### Rule 3: Handoff Time

```
Good handoff times:
  - Monday 9:00 AM (start of work week)
  - Monday 10:00 AM (after morning standup)

Bad handoff times:
  - Friday 5:00 PM (new person starts over a weekend)
  - Sunday midnight (nobody wants to start on-call at midnight)
  - Middle of the day (interrupts work flow)
```

### Rule 4: Primary + Secondary

```
Always have TWO layers:

Primary:   First responder. Gets paged first.
Secondary: Backup. Gets paged if primary doesn't respond.
           Also available for major incidents as a second pair of hands.

Rotation offset:
  Week 1:  Primary: Alice,   Secondary: Bob
  Week 2:  Primary: Bob,     Secondary: Charlie
  Week 3:  Primary: Charlie, Secondary: Diana
  Week 4:  Primary: Diana,   Secondary: Alice

This ensures:
  - Someone is always backup
  - Secondary learns the ropes before becoming primary
```

### Rule 5: Handle Time Zones Gracefully

For distributed teams:

```
Option A: Follow-the-Sun (Best)
  ┌──────────┐   ┌──────────┐   ┌──────────┐
  │ US Team  │──▶│ EU Team  │──▶│APAC Team │──▶ (back to US)
  │ 9am-5pm  │   │ 9am-5pm  │   │ 9am-5pm  │
  │  PST     │   │  CET     │   │  JST     │
  └──────────┘   └──────────┘   └──────────┘
  
  Nobody is EVER on-call outside business hours! 🎉

Option B: Regional Rotation (OK)
  Each region handles their own hours.
  US covers US nighttime, EU covers EU nighttime.
  
  Downside: Night shifts still exist within each region.

Option C: Global Rotation (Avoid)
  One person covers everything regardless of timezone.
  
  ❌ This means someone is ALWAYS working at night.
  ❌ This is unfair and leads to burnout.
```

### Scheduling Configuration Example

Here's how you'd set up a follow-the-sun schedule in PagerDuty:

```
Schedule: "production-follow-the-sun"

Layer 1 — US Team:
  Rotation: Weekly
  Members: Alice, Bob, Charlie
  Restriction: 06:00 - 14:00 UTC (covers US business hours)

Layer 2 — EU Team:
  Rotation: Weekly
  Members: Diana, Eve, Frank
  Restriction: 14:00 - 22:00 UTC (covers EU business hours)

Layer 3 — APAC Team:
  Rotation: Weekly
  Members: Grace, Hiro, Isha
  Restriction: 22:00 - 06:00 UTC (covers APAC business hours)

Final Schedule (rendered):
  00:00 - 06:00 UTC → APAC on-call
  06:00 - 14:00 UTC → US on-call
  14:00 - 22:00 UTC → EU on-call
  22:00 - 24:00 UTC → APAC on-call
```

## 6. On-Call Compensation & Fairness

Being on-call has a real impact on quality of life. Fair compensation is essential for sustainability.

### Common Compensation Models

| Model | Description | Example |
|-------|-------------|---------|
| **Flat stipend** | Fixed pay per on-call shift | $200/day or $1,000/week |
| **Per-page pay** | Extra pay for each page received | $50 per page outside business hours |
| **Comp time** | Time off after on-call | "After a night page, take the next morning off" |
| **Hybrid** | Combination of above | $500/week stipend + comp time after night pages |
| **Nothing** | No extra compensation | ❌ Not recommended — leads to resentment |

### Fairness Metrics to Track

```
Track and review quarterly:

1. Pages per person per month
   → Is one person getting paged significantly more than others?
   
2. Night pages per person per month
   → Night pages are more disruptive than daytime ones
   
3. On-call shifts per person per quarter
   → Is the distribution even?
   
4. Incident duration per person
   → Is one person always stuck with the long incidents?

Red flags:
  ❌ Alice: 15 pages/month, Bob: 2 pages/month
     → Route different services to different rotations
     
  ❌ Charlie always covers holidays
     → Rotate holiday coverage fairly
     
  ❌ New hire never does on-call "because they're too new"
     → Shadow first, then start with secondary, then primary
```

### Rest After Night Pages

```
Policy recommendation:

If paged between 12:00 AM and 6:00 AM:
  → May start work late the next day (or take the morning off)
  
If incident lasts > 1 hour during night:
  → Take at least 4 hours off the next day
  
If incident lasts > 3 hours during night:
  → Take the full next day off

This is not a "perk" — it's a safety measure.
Sleep-deprived engineers make worse decisions and write buggier code.
```

## 7. The On-Call Handoff — Shift Transitions

The handoff between on-call shifts is **critical**. A bad handoff leads to dropped context and missed issues.

### Handoff Checklist

```markdown
# On-Call Handoff Checklist

## Outgoing On-Call (Alice → Bob)

### 1. Active Issues
- [ ] Are there any ongoing incidents or unresolved alerts?
- [ ] List any known issues that might page during Bob's shift:
  - "The database has been running hot. If it crosses 90%, 
     see runbook: [db-high-cpu]"
  - "Deploy is scheduled for Wednesday. Watch for error rate 
     spikes afterward."

### 2. Recent Changes
- [ ] What was deployed in the last 24 hours?
- [ ] Any infrastructure changes (scaling, config updates)?
- [ ] Any upcoming maintenance windows?

### 3. Alert Changes
- [ ] Were any alerts silenced? When do they expire?
- [ ] Were any new alerts added or thresholds changed?

### 4. Runbook Updates
- [ ] Were any runbooks updated based on incidents this week?
- [ ] Any runbook gaps discovered? (Document them!)

### 5. General Notes
- [ ] Anything unusual the incoming on-call should know?
- [ ] Any tips or gotchas from this week's experience?
```

### Handoff Meeting (5-10 minutes)

```
Format: Quick standup at shift change (e.g., Monday 9:00 AM)

Agenda:
  1. Outgoing shares their week (2-3 minutes)
     - "I got paged twice — once for the search index rebuild 
        and once for a flaky health check alert."
     - "I silenced the disk space alert on staging — it expires 
        Wednesday. Staging team is cleaning up."
     
  2. Active issues walkthrough (2-3 minutes)
     - "Watch out for the payment gateway — they notified us 
        about maintenance Thursday at 2 AM UTC."
     
  3. Questions from incoming (1-2 minutes)
     - "Is the disk space alert on prod or staging?"
     - "What's the runbook for the search index issue?"
```

### Handoff Document (Living Document)

Keep a shared document (Notion, Confluence, Google Doc) that each on-call person updates:

```markdown
# On-Call Handoff Log — payments-team

## Current On-Call
- **Primary:** Bob (May 20 - May 27)
- **Secondary:** Charlie

## Active Alerts & Issues
| Alert | Status | Notes |
|-------|--------|-------|
| DB connection pool warning | Silenced until May 22 | DBA team is migrating to larger instance |
| Payment gateway P2 | Monitoring | Stripe reported intermittent issues in EU region |

## Recent Deployments
| Date | Service | Version | Notes |
|------|---------|---------|-------|
| May 19 | user-service | v2.4.1 | Bug fix for login timeout |
| May 18 | payment-service | v3.1.0 | New payment provider integration |

## Upcoming Events

- May 23: Planned load test (10x normal traffic)

| Date | Incident | Resolution | Postmortem |

## 8. What To Do When You Get Paged

You're asleep. Your phone rings. It's 3 AM. Here's exactly what to do:

### The First 5 Minutes

```
Step 1: ACKNOWLEDGE (30 seconds)
  → Open PagerDuty/Opsgenie app on your phone
  → Tap "Acknowledge"
  → This stops the escalation timer

Step 2: BREATHE (30 seconds)
  → Take a breath. You're not going to fix this in 30 seconds
     anyway. Panicking helps nobody.

Step 3: READ THE ALERT (1 minute)
  → What does the alert say?
  → Which service is affected?
  → What is the severity?
  → Is there a runbook link?

Step 4: OPEN YOUR LAPTOP (1 minute)
  → Check the linked dashboard
  → Is the issue still happening? (Sometimes things auto-resolve)
  → If auto-resolved: acknowledge, go back to sleep

Step 5: FOLLOW THE RUNBOOK (2+ minutes)
  → Open the runbook
  → Follow the steps — even if you think you know what to do
  → Runbooks are written when you're awake and thinking clearly.
     Trust them over your 3 AM brain.
```

### Triage Decision Tree

```
┌──────────────────────────────────────────┐
│         ARE USERS AFFECTED?              │
└────────────────┬─────────────────────────┘
                 │
        ┌────────┴────────┐
        │                 │
       YES               NO
        │                 │
   ┌────▼─────┐     ┌────▼─────────┐
   │ How many │     │ Will users   │
   │ users?   │     │ be affected  │
   └────┬─────┘     │ soon?        │
        │           └────┬─────────┘
   ┌────┴────┐     ┌────┴────┐
   │         │     │         │
  Many      Few   YES       NO
   │         │     │         │
   │    ┌────▼──┐  │    ┌────▼──────┐
   │    │ P2    │  │    │ Note it,  │
   │    │ Fix   │  │    │ fix in    │
   │    │ during│  │    │ business  │
   │    │ hours │  │    │ hours     │
   │    └───────┘  │    └───────────┘
   │               │
   ▼               ▼
┌──────────────────────┐
│  P1 — FIX NOW        │
│                      │
│  1. Communicate      │
│  2. Mitigate first   │
│  3. Root cause later │
└──────────────────────┘
```

### Mitigation vs Root Cause

At 3 AM, your goal is **mitigation**, NOT root cause analysis:

```
Mitigation (DO NOW):
  "Stop the bleeding"
  → Rollback the bad deployment
  → Scale up instances
  → Redirect traffic
  → Restart the service
  → Failover to backup
  → Disable the broken feature

Root Cause Analysis (DO LATER):
  "Why did this happen?"
  → Debug the code
  → Find the exact bug
  → Redesign the system
  → Write tests

Example:
  3 AM: Payment service is returning 500s after a deploy.
  
  ✅ Mitigation: Roll back the deployment (3 minutes)
  ❌ Root cause: Debug the code to find the exact bug (2 hours)
  
  Roll back. Go back to sleep. Debug tomorrow.
```

## 9. What Are Runbooks?

A **runbook** is a documented set of step-by-step instructions for handling a specific operational scenario — usually an alert or incident.

Think of it as a **recipe** — but instead of cooking food, you're fixing production systems. And just like a recipe, anyone should be able to follow it, even if they've never made the dish before.

### Why Runbooks Are Essential

```
Without a runbook:
  📱 Alert: "Database replication lag > 30 seconds"
  🤔 Engineer: "Uh... what database? Where do I check lag? 
      What's the threshold? Who do I contact? What can I safely 
      restart? Will it lose data?"
  ⏱️ Time to fix: 45 minutes (mostly spent figuring out WHAT to do)

With a runbook:
  📱 Alert: "Database replication lag > 30 seconds"
  📖 Engineer: *Opens runbook link from alert*
  📋 Step 1: Check lag on Grafana dashboard (link provided)
  📋 Step 2: If lag > 60s, restart replication (command provided)
  📋 Step 3: If restart doesn't help, failover to replica (steps provided)
  📋 Step 4: Escalate to DBA if unresolved after 15 min
  ⏱️ Time to fix: 10 minutes
```

### Types of Runbooks

| Type | Purpose | Example |
|------|---------|---------|
| **Alert runbooks** | How to respond to a specific alert | "What to do when payment error rate > 5%" |
| **Diagnostic runbooks** | How to investigate a type of problem | "How to debug slow API responses" |
| **Maintenance runbooks** | How to perform routine operations | "How to rotate database credentials" |
| **Disaster recovery runbooks** | How to recover from major failures | "How to failover to DR region" |
| **Onboarding runbooks** | How to get access and set up tools | "New on-call engineer setup guide" |

## 10. How To Write a Great Runbook

### The Golden Rules

```
1. WRITE FOR 3 AM BRAIN
   The reader is sleep-deprived, stressed, and possibly new 
   to the team. Use simple words, short sentences, and clear steps.

2. BE SPECIFIC
   ❌ "Check the logs"
   ✅ "Run: kubectl logs -n payments deploy/payment-api --tail=100"

3. INCLUDE EXACT COMMANDS
   Copy-pasteable commands save minutes and prevent typos.

4. EXPLAIN THE "WHY"
   Each step should briefly explain what it does and why.
   This helps the engineer make judgment calls.

5. INCLUDE DECISION POINTS
   "If X, do this. If Y, do that."
   Don't assume a linear path.

6. LINK TO DASHBOARDS AND TOOLS
   Every runbook should link to the relevant:
   - Grafana dashboard
   - Log search query
   - Service configuration
   - Escalation contacts

7. KEEP IT UPDATED
   An outdated runbook is worse than no runbook — it gives 
   false confidence and can lead to wrong actions.
   
   Review runbooks: After every incident where the runbook was used.
```

### Runbook Structure

Every runbook should follow this structure:

```
1. TITLE & METADATA
   What alert/scenario is this for?
   Who owns this runbook?
   When was it last updated?

2. OVERVIEW
   What is this alert about?
   What does it mean?
   How serious is it?

3. IMPACT
   Who/what is affected if this isn't fixed?

4. PREREQUISITES
   What access/tools do you need?

5. DIAGNOSIS STEPS
   How to confirm the problem is real and understand its scope.

6. REMEDIATION STEPS
   How to fix it (step by step).

7. VERIFICATION
   How to confirm the fix worked.

8. ESCALATION
   Who to contact if the runbook doesn't solve it.

9. RELATED RESOURCES
   Links to dashboards, docs, architecture diagrams.
```

## Step 1: Verify the Problem

Check if the alert is real and still firing:

1. Open the Grafana dashboard:
   **[Service Dashboard](https://grafana.internal/d/abc123)**

2. Check the current error rate:
   ```promql
   sum(rate(http_requests_total{job="payment-service",status=~"5.."}[5m]))
   / sum(rate(http_requests_total{job="payment-service"}[5m]))
   ```

3. Check if the issue is ongoing or already resolved:
   - If the error rate has returned to normal → the issue may have 
     auto-resolved. Monitor for 10 minutes, then resolve the alert.
   - If the error rate is still elevated → continue to Step 2.

---

## Step 2: Identify the Cause

### Check recent deployments

```bash
# Check what was deployed recently
kubectl -n payments rollout history deploy/payment-api

# Check when the last deployment happened
kubectl -n payments describe deploy/payment-api | grep -i "last"
```

**If a deployment happened shortly before the alert:**
→ Go to [Step 3a: Rollback Deployment](#step-3a-rollback-deployment)

### Check application logs

```bash
# Get recent error logs
kubectl -n payments logs deploy/payment-api --tail=200 | grep -i "error\|exception\|fatal"

# If using Loki/Grafana, use this LogQL query:
# {namespace="payments", app="payment-api"} |= "error" | logfmt
```

**Common errors and what they mean:**
| Error Message | Likely Cause | Go To |
|--------------|-------------|-------|
| `Connection refused: database` | DB is down or unreachable | Step 3b |
| `Timeout exceeded` | Downstream service slow | Step 3c |
| `OutOfMemoryError` | Memory leak or undersized pod | Step 3d |
| `TLS handshake failed` | Certificate expired | Step 3e |

### Check pod health

```bash
# Check pod status
kubectl -n payments get pods -l app=payment-api

# Look for CrashLoopBackOff, OOMKilled, or Error states
kubectl -n payments describe pod <pod-name> | grep -A 5 "State\|Reason\|Last State"
```

---

## Step 3a: Rollback Deployment

If the issue started after a recent deployment:

```bash
# Roll back to the previous version
kubectl -n payments rollout undo deploy/payment-api

# Watch the rollout progress
kubectl -n payments rollout status deploy/payment-api

# Verify pods are running
kubectl -n payments get pods -l app=payment-api
```

**After rollback:**
→ Go to [Step 4: Verify Resolution](#step-4-verify-resolution)

---

## Step 3b: Database Connection Issues

If logs show database connection errors:

```bash
# Check if the database is reachable from the pod
kubectl -n payments exec deploy/payment-api -- \
  nc -zv payment-db.internal 5432

# Check database connection pool metrics
# Grafana: [DB Connection Pool Dashboard](https://grafana.internal/d/dbpool)

# If connection pool is exhausted, restart the deployment
# (this resets all connections)
kubectl -n payments rollout restart deploy/payment-api
```

**If the database itself is down:**
→ Escalate to the **DBA team**: #dba-team on Slack, or page `dba-primary` schedule

---

## Step 3c: Downstream Service Timeout

If logs show timeout errors calling another service:

```bash
# Check the health of downstream services
kubectl -n payments exec deploy/payment-api -- \
  curl -s -o /dev/null -w "%{http_code}" http://inventory-service:8080/health

# Check if the downstream service is healthy
kubectl -n inventory get pods -l app=inventory-service
```

**If the downstream service is unhealthy:**
→ Page the team that owns that service

---

## Step 3d: Out of Memory (OOMKilled)

If pods are being OOMKilled:

```bash
# Check memory usage
kubectl -n payments top pods -l app=payment-api

# Check if there was an OOMKill event
kubectl -n payments get events --sort-by='.lastTimestamp' | grep -i oom

# Temporary fix: increase memory limit
kubectl -n payments set resources deploy/payment-api \
  --limits=memory=1Gi --requests=memory=512Mi

# NOTE: This is a temporary fix. Create a ticket to investigate 
# the memory leak: [Jira Template](https://jira.internal/create?type=bug)
```

---

## Step 3e: TLS Certificate Expired

If logs show TLS/certificate errors:

```bash
# Check certificate expiry
echo | openssl s_client -connect payment-api.internal:443 2>/dev/null | \
  openssl x509 -noout -dates

# If expired, check cert-manager status
kubectl get certificates -n payments
kubectl describe certificate payment-api-tls -n payments
```

→ Escalate to **Platform team** if cert-manager is not auto-renewing.

---

## Step 4: Verify Resolution

After applying a fix, verify that the issue is resolved:

1. **Check error rate** (should drop below threshold within 5 minutes):
   ```promql
   sum(rate(http_requests_total{job="payment-service",status=~"5.."}[5m]))
   / sum(rate(http_requests_total{job="payment-service"}[5m]))
   ```

2. **Check that pods are healthy:**
   ```bash
   kubectl -n payments get pods -l app=payment-api
   # All pods should be Running with 1/1 READY
   ```

3. **Verify from a user's perspective:**
   ```bash
   curl -s -o /dev/null -w "%{http_code}" https://api.company.com/payments/health
   # Should return 200
   ```

4. **Monitor for 10-15 minutes** to ensure the fix holds.

5. **Resolve the incident** in PagerDuty/Opsgenie with a note about what you did.

## 12. Real-World Runbook Examples

### Example 1: High Memory Usage on Kubernetes Pod

```markdown
# Runbook: Pod Memory Usage > 90%

## Overview
A pod is using more than 90% of its memory limit. If it reaches 
100%, Kubernetes will OOMKill the pod, causing a brief interruption.

## Impact
- If OOMKilled: ~30 seconds of downtime for that pod instance
- If multiple pods: reduced capacity, potential cascading failure

## Diagnosis

### Check current memory usage
```bash
# Get memory usage for all pods in the deployment
kubectl -n <namespace> top pods -l app=<service-name>

# Example output:
# NAME                          CPU    MEMORY
# payment-api-7d4b8c6f4-abc12  150m   890Mi   ← 89% of 1Gi limit
# payment-api-7d4b8c6f4-def34  120m   450Mi   ← Normal
```

### Check if it's a gradual leak or sudden spike
Look at the memory graph in Grafana:
- **Gradual increase** → Memory leak (needs code fix)
- **Sudden spike** → Traffic spike or large request (may be transient)

### Check for OOMKill events
```bash
kubectl -n <namespace> get events --sort-by='.lastTimestamp' | grep OOM
```

## Remediation

### Option A: Restart the pod (quick fix for memory leak)
```bash
# Delete the specific high-memory pod (Kubernetes will recreate it)
kubectl -n <namespace> delete pod <pod-name>

# Or restart the entire deployment
kubectl -n <namespace> rollout restart deploy/<deployment-name>
```

### Option B: Scale up (if traffic-related)
```bash
# Add more replicas to spread the load
kubectl -n <namespace> scale deploy/<deployment-name> --replicas=5
```

### Option C: Increase memory limit (temporary)
```bash
kubectl -n <namespace> set resources deploy/<deployment-name> \
  --limits=memory=2Gi --requests=memory=1Gi
```
**⚠️ Create a Jira ticket to investigate the root cause!**

## Verification
```bash
# Confirm memory is back to normal
kubectl -n <namespace> top pods -l app=<service-name>
# All pods should be well below 90%
```
```

### Example 2: SSL Certificate Expiring Soon

```markdown
# Runbook: SSL Certificate Expires in < 7 Days

## Overview
An SSL/TLS certificate is about to expire. If it expires, 
users will see "Your connection is not secure" errors and 
won't be able to access the service.

## Impact
- **If it expires:** Complete service outage for HTTPS traffic
- **Urgency:** HIGH — fix before expiry!

## Diagnosis

### Check which certificate is expiring
```bash
# Check the certificate details
echo | openssl s_client -connect <domain>:443 2>/dev/null | \
  openssl x509 -noout -subject -dates -issuer

# Example output:
# subject=CN = api.company.com
# notBefore=Jan 15 00:00:00 2024 GMT
# notAfter=Apr 15 23:59:59 2024 GMT    ← Expiring soon!
# issuer=O = Let's Encrypt
```

### If using cert-manager in Kubernetes
```bash
# Check certificate status
kubectl get certificates --all-namespaces

# Check if cert-manager is having issues renewing
kubectl describe certificate <cert-name> -n <namespace>
kubectl logs -n cert-manager deploy/cert-manager --tail=100
```

## Remediation

### If cert-manager is handling renewal:
```bash
# Force a renewal
kubectl delete secret <tls-secret-name> -n <namespace>
# cert-manager will detect the missing secret and recreate it

# Or annotate the certificate to trigger renewal
kubectl annotate certificate <cert-name> -n <namespace> \
  cert-manager.io/renew-before="720h" --overwrite
```

### If manually managed certificates:
1. Generate a new certificate (via your CA or Let's Encrypt)
2. Update the Kubernetes secret:
```bash
kubectl create secret tls <secret-name> \
  --cert=new-cert.pem \
  --key=new-key.pem \
  -n <namespace> \
  --dry-run=client -o yaml | kubectl apply -f -
```
3. Restart the ingress controller if needed:
```bash
kubectl rollout restart deploy/ingress-nginx-controller -n ingress-nginx
```

## Verification
```bash
# Verify the new certificate is serving
echo | openssl s_client -connect <domain>:443 2>/dev/null | \
  openssl x509 -noout -dates

# Should show a new notAfter date (90 days from now for Let's Encrypt)
```
```

### Example 3: Kubernetes Node Not Ready

```markdown
# Runbook: Kubernetes Node NotReady

## Overview
A Kubernetes node has entered the NotReady state, meaning it's not 
accepting new pods and existing pods may be evicted.

## Impact
- Pods on the affected node may be rescheduled to other nodes
- If cluster is at capacity, some pods may be unschedulable
- Reduced cluster capacity

## Diagnosis

### Identify the NotReady node
```bash
kubectl get nodes
# Look for nodes with STATUS = NotReady

kubectl describe node <node-name> | grep -A 20 "Conditions"
# Look for:
#   Ready = False → Node is unhealthy
#   MemoryPressure = True → Node is running out of memory
#   DiskPressure = True → Node is running out of disk
#   PIDPressure = True → Too many processes
```

### Check kubelet status (if you have SSH access)
```bash
ssh <node-ip>
sudo systemctl status kubelet
sudo journalctl -u kubelet --since "30 minutes ago" | tail -50
```

### Check node resources
```bash
kubectl top node <node-name>
# Check if CPU or memory is exhausted
```

## Remediation

### If kubelet is crashed:
```bash
ssh <node-ip>
sudo systemctl restart kubelet
sudo systemctl status kubelet
```

### If disk pressure:
```bash
ssh <node-ip>
# Check disk usage
df -h
# Clean up unused Docker images
sudo docker system prune -a --force
# Or for containerd:
sudo crictl rmi --prune
```

### If the node is unrecoverable:
```bash
# Drain the node (move pods to other nodes)
kubectl drain <node-name> --ignore-daemonsets --delete-emptydir-data

# If using cloud provider, terminate and let auto-scaling group 
# create a new one
# AWS: terminate instance in EC2 console
# The ASG will automatically provision a replacement
```

### If using cluster autoscaler:
- Check if new nodes are being provisioned:
```bash
kubectl -n kube-system logs deploy/cluster-autoscaler --tail=50
```

## Verification
```bash
# Confirm node is Ready
kubectl get nodes

# Confirm all pods are rescheduled and running
kubectl get pods --all-namespaces -o wide | grep <node-name>
# Should be empty if node was drained
```

## 13. Automating Runbooks

As you notice patterns in your runbooks, you should start **automating** the repetitive parts.

### Levels of Automation

```
Level 0: MANUAL
  "Read runbook → type commands manually"
  → Good for: Rare incidents, complex decision-making

Level 1: SEMI-AUTOMATED (Scripts)
  "Run a script that does Steps 1-3 automatically"
  → Good for: Common incidents with predictable steps

Level 2: TRIGGERED AUTOMATION
  "Alert fires → script runs automatically → human verifies"
  → Good for: Well-understood issues with safe remediation

Level 3: SELF-HEALING
  "Alert fires → system fixes itself → human is notified"
  → Good for: Simple, safe, and well-tested remediations
```

### Example: Semi-Automated Runbook Script

```bash
#!/bin/bash
# runbook-payment-high-error-rate.sh
# Semi-automated runbook for payment service high error rate
# Usage: ./runbook-payment-high-error-rate.sh

set -e

# Colors for readability
RED='\033[0;31m'
GREEN='\033[0;32m'
YELLOW='\033[1;33m'
NC='\033[0m' # No Color

NAMESPACE="payments"
DEPLOYMENT="payment-api"

echo -e "${YELLOW}=== Payment Service High Error Rate Runbook ===${NC}"
echo ""

# Step 1: Check current state
echo -e "${YELLOW}Step 1: Checking pod status...${NC}"
kubectl -n $NAMESPACE get pods -l app=$DEPLOYMENT
echo ""

# Step 2: Check recent events
echo -e "${YELLOW}Step 2: Checking recent events...${NC}"
kubectl -n $NAMESPACE get events --sort-by='.lastTimestamp' | tail -10
echo ""

# Step 3: Check recent deployments
echo -e "${YELLOW}Step 3: Checking deployment history...${NC}"
kubectl -n $NAMESPACE rollout history deploy/$DEPLOYMENT | tail -5
echo ""

# Step 4: Check error logs
echo -e "${YELLOW}Step 4: Recent error logs...${NC}"
kubectl -n $NAMESPACE logs deploy/$DEPLOYMENT --tail=50 | \
  grep -i "error\|exception\|fatal" | tail -20
echo ""

# Step 5: Ask the engineer what to do
echo -e "${YELLOW}=== DIAGNOSIS COMPLETE ===${NC}"
echo ""
echo "Based on the above, what would you like to do?"
echo "  1) Rollback to previous version"
echo "  2) Restart pods (if memory/connection issue)"
echo "  3) Scale up (if traffic-related)"
echo "  4) Do nothing (investigating further)"
echo ""
read -p "Enter choice (1-4): " choice

case $choice in
  1)
    echo -e "${RED}Rolling back deployment...${NC}"
    kubectl -n $NAMESPACE rollout undo deploy/$DEPLOYMENT
    kubectl -n $NAMESPACE rollout status deploy/$DEPLOYMENT
    echo -e "${GREEN}Rollback complete!${NC}"
    ;;
  2)
    echo -e "${YELLOW}Restarting pods...${NC}"
    kubectl -n $NAMESPACE rollout restart deploy/$DEPLOYMENT
    kubectl -n $NAMESPACE rollout status deploy/$DEPLOYMENT
    echo -e "${GREEN}Restart complete!${NC}"
    ;;
  3)
    read -p "How many replicas? (current: $(kubectl -n $NAMESPACE get deploy/$DEPLOYMENT -o jsonpath='{.spec.replicas}')): " replicas
    echo -e "${YELLOW}Scaling to $replicas replicas...${NC}"
    kubectl -n $NAMESPACE scale deploy/$DEPLOYMENT --replicas=$replicas
    echo -e "${GREEN}Scaled to $replicas replicas!${NC}"
    ;;
  4)
    echo "OK, continuing investigation. Good luck!"
    ;;
  *)
    echo "Invalid choice"
    exit 1
    ;;
esac

echo ""
echo -e "${YELLOW}Step 6: Verifying fix...${NC}"
sleep 10
echo "Current pod status:"
kubectl -n $NAMESPACE get pods -l app=$DEPLOYMENT
echo ""
echo -e "${GREEN}=== Runbook complete ===${NC}"
echo "Don't forget to:"
echo "  - Resolve the incident in PagerDuty/Opsgenie"
echo "  - Write a brief note about what you did"
echo "  - Create a postmortem if this was a P1/P2"
```

**Explanation:**

- This script automates the **diagnostic** portion of the runbook (checking pods, events, logs) — saving the engineer from typing multiple `kubectl` commands.
- It presents the **remediation options** and lets the engineer choose — keeping a human in the loop for the decision.
- After the fix, it automatically verifies the result.
- It reminds the engineer of next steps (resolve incident, write postmortem).

### Example: Self-Healing with Kubernetes

Kubernetes itself provides some self-healing capabilities:

```yaml
# Liveness probe — Kubernetes automatically restarts 
# unhealthy containers
apiVersion: apps/v1
kind: Deployment
metadata:
  name: payment-api
  namespace: payments
spec:
  replicas: 3
  template:
    spec:
      containers:
        - name: payment-api
          image: payment-api:v3.1.0
          
          # Liveness: "Is the container healthy?"
          # If it fails → Kubernetes restarts the container
          livenessProbe:
            httpGet:
              path: /healthz
              port: 8080
            initialDelaySeconds: 15
            periodSeconds: 10
            failureThreshold: 3
          
          # Readiness: "Can this container handle traffic?"
          # If it fails → removed from Service endpoints (no traffic)
          readinessProbe:
            httpGet:
              path: /ready
              port: 8080
            initialDelaySeconds: 5
            periodSeconds: 5
            failureThreshold: 3
          
          # Startup: "Has the container finished starting up?"
          # Protects slow-starting containers from being killed
          startupProbe:
            httpGet:
              path: /healthz
              port: 8080
            failureThreshold: 30
            periodSeconds: 10
          
          resources:
            requests:
              memory: "512Mi"
              cpu: "250m"
            limits:
              memory: "1Gi"
              cpu: "1000m"
```

### PagerDuty Automation Actions

PagerDuty can run automated remediation when incidents trigger:

```

    #!/bin/bash
    # Delete log files older than 7 days

    echo "Cleaned $(find /var/log/app -name "*.log" -mtime +7 | wc -l) old log files"
    
  Result: 
    If successful → Auto-resolve the incident
    If failed → Page the on-call engineer
```

## 16. Training New On-Call Engineers

New engineers should NOT be thrown into on-call without preparation. Here's a training plan:

### Phase 1: Shadow (Week 1-2)

```
New engineer "shadows" the current on-call:

- Gets added to the alert channel (Slack)
- Can see all alerts and incidents
- Joins incident response calls as an observer
- Reads through all runbooks
- Does NOT get paged
- Does NOT make decisions

Goal: Understand the flow, the tools, and the types of issues
```

### Phase 2: Reverse Shadow (Week 3-4)

```
New engineer is the PRIMARY on-call, but with a safety net:

- Gets paged first
- Acknowledges and starts the runbook
- The experienced engineer is SECONDARY and available immediately
- New engineer leads, experienced engineer advises

Goal: Practice responding with a safety net
```

### Phase 3: Supported On-Call (Week 5-6)

```
New engineer is fully on-call:

- Gets paged and responds independently
- Secondary is available if needed but not actively watching
- Debriefs after each incident with the team

Goal: Build confidence and independence
```

### Phase 4: Independent On-Call (Week 7+)

```
New engineer is in the regular rotation:

- Handles incidents independently
- Knows when to escalate
- Contributes to runbook improvements
- Eventually mentors the next new engineer

Goal: Full participation in on-call rotation
```

## Quick Reference Card

```
On-Call Response Protocol:
  1. Acknowledge (< 1 min)
  2. Assess (< 2 min): Read alert, check dashboard
  3. Communicate (< 5 min): Post in #incidents
  4. Mitigate (ASAP): Follow runbook, stop the bleeding
  5. Verify (after fix): Confirm issue is resolved
  6. Document (after incident): Write up what happened
  7. Postmortem (next business day): For P1/P2 incidents

Runbook Must-Haves:
  ✅ Overview (what is this alert?)
  ✅ Impact (who is affected?)
  ✅ Diagnosis steps (how to figure out what's wrong)
  ✅ Exact commands (copy-paste ready)
  ✅ Decision tree (if X, do Y)
  ✅ Verification (how to confirm the fix worked)
  ✅ Escalation contacts (who to call if stuck)
  ✅ Links (dashboards, logs, architecture docs)

On-Call Health Targets:
  MTTA:             < 5 minutes
  MTTR (P1):        < 1 hour
  Off-hours pages:  < 2 per shift
  False positives:  < 10%
  Team satisfaction: > 7/10
```
