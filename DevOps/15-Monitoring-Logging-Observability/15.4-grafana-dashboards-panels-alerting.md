## 1. What is Grafana?

Okay, let's start with the basics.

**Grafana** is an **open-source visualization and analytics platform**. Think of it as a beautiful, interactive window into your data. It takes raw numbers (metrics, logs, traces) from various sources and turns them into gorgeous, easy-to-understand **dashboards** with charts, graphs, tables, and more.

### The Simplest Way to Think About It

Imagine you have a car with no dashboard — no speedometer, no fuel gauge, no temperature warning light. You'd be driving blind! 😱

Grafana is the **dashboard for your servers, applications, and infrastructure**. It tells you:

- How fast are things running? (CPU, memory)
- Are we running out of fuel? (disk space, bandwidth)
- Is anything overheating? (error rates, response times)

### Key Facts About Grafana

| Property | Detail |
|----------|--------|
| **Type** | Open-source visualization platform |
| **Written in** | Go (backend) + TypeScript/React (frontend) |
| **License** | AGPLv3 (OSS) + Enterprise edition available |
| **Created by** | Torkel Ödegaard (2014) |
| **Company** | Grafana Labs |
| **Default Port** | `3000` |
| **Website** | [grafana.com](https://grafana.com) |

### What Grafana is NOT

- Grafana is **NOT a database** — it doesn't store your metrics. It *reads* from databases.
- Grafana is **NOT a data collector** — it doesn't scrape metrics from your servers. That's Prometheus's job.
- Grafana is a **visualization layer** that sits on top of your data sources.

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  Your Servers │────▶│  Prometheus  │────▶│   Grafana    │
│  (emit data)  │     │ (stores data)│     │ (shows data) │
└──────────────┘     └──────────────┘     └──────────────┘
```

## 2. Why Do We Need Grafana?

You might be thinking — "Prometheus has its own UI, Elasticsearch has Kibana... why do we need another tool?"

Great question! Here's why Grafana is so popular:

### 1. **Unified View Across Multiple Data Sources**

This is the killer feature. Grafana can connect to **dozens of data sources** at the same time:

- Prometheus (for metrics)
- Loki (for logs)
- Elasticsearch (for logs/search)
- InfluxDB, MySQL, PostgreSQL, CloudWatch, Azure Monitor, and many more

You can put a Prometheus chart and an Elasticsearch log panel **side by side on the same dashboard**. No other tool does this as elegantly.

### 2. **Beautiful, Customizable Dashboards**

Let's be honest — Prometheus's built-in UI is... functional but ugly. Grafana makes your data look amazing with:

- Line graphs, bar charts, heatmaps, gauges, stat panels
- Color-coded thresholds (green = good, yellow = warning, red = critical)
- Dark mode and light mode themes

### 3. **Powerful Query Editor**

Grafana gives you a visual query builder for each data source. You don't need to memorize PromQL syntax — you can use dropdowns and auto-complete.

### 4. **Alerting Built In**

Set alerts directly from your dashboards. "If CPU goes above 90% for 5 minutes, send a Slack message." Done.

### 5. **It's Free and Open Source**

The community edition is completely free. The enterprise edition adds things like enhanced RBAC, reporting, and enterprise data source plugins.

### 6. **Huge Community**

Thousands of pre-built dashboards on [grafana.com/grafana/dashboards](https://grafana.com/grafana/dashboards) — just import and use.

## 3. Grafana Architecture — How It Works

Grafana's architecture is straightforward:

```
                         ┌─────────────────────────┐
                         │      Grafana Server      │
                         │                          │
                         │  ┌────────────────────┐  │
                         │  │   Frontend (React)  │  │
                         │  │   - Dashboards UI   │  │
                         │  │   - Panel Rendering │  │
                         │  │   - Query Editor    │  │
                         │  └────────┬───────────┘  │
                         │           │               │
                         │  ┌────────▼───────────┐  │
                         │  │   Backend (Go)      │  │
                         │  │   - API Server      │  │
                         │  │   - Auth/RBAC       │  │
                         │  │   - Alerting Engine │  │
                         │  │   - Plugin System   │  │
                         │  └────────┬───────────┘  │
                         │           │               │
                         └───────────┼───────────────┘
                                     │
                    ┌────────────────┼────────────────┐
                    │                │                 │
              ┌─────▼─────┐  ┌──────▼──────┐  ┌──────▼──────┐
              │ Prometheus │  │    Loki     │  │  InfluxDB   │
              │ (metrics)  │  │   (logs)    │  │  (metrics)  │
              └───────────┘  └─────────────┘  └─────────────┘

        ┌──────────────┐
        │  Grafana DB   │  ← Stores dashboards, users, alerts config
        │ (SQLite/MySQL/│     (NOT your metrics data!)
        │  PostgreSQL)  │
        └──────────────┘
```

### Key Components

| Component | What It Does |
|-----------|-------------|
| **Frontend** | React-based UI — renders dashboards, panels, query editors |
| **Backend** | Go-based API server — handles auth, alerting, data source proxying |
| **Grafana Database** | Stores dashboard definitions, user accounts, alert rules, org settings (uses SQLite by default, can use MySQL/PostgreSQL) |
| **Data Source Plugins** | Connectors that know how to talk to each backend (Prometheus, Loki, etc.) |
| **Panel Plugins** | Visualization types (time series, gauge, table, etc.) |

### Important Point

Grafana **proxies** all data source queries through its backend. Your browser doesn't talk directly to Prometheus — it asks Grafana, and Grafana fetches the data from Prometheus. This is great for security because you don't need to expose Prometheus to the internet.

## 4. Installing Grafana

### Option 1: Ubuntu/Debian

```bash
# Add Grafana APT repository
sudo apt-get install -y apt-transport-https software-properties-common wget

# Add the GPG key
sudo mkdir -p /etc/apt/keyrings/
wget -q -O - https://apt.grafana.com/gpg.key | gpg --dearmor | sudo tee /etc/apt/keyrings/grafana.gpg > /dev/null

# Add the repository
echo "deb [signed-by=/etc/apt/keyrings/grafana.gpg] https://apt.grafana.com stable main" | sudo tee -a /etc/apt/sources.list.d/grafana.list

# Install Grafana
sudo apt-get update
sudo apt-get install grafana

# Start Grafana
sudo systemctl daemon-reload
sudo systemctl start grafana-server
sudo systemctl enable grafana-server    # Start on boot

# Check status
sudo systemctl status grafana-server
```

### Option 2: CentOS/RHEL

```bash
# Create repo file
cat <<EOF | sudo tee /etc/yum.repos.d/grafana.repo
[grafana]
name=grafana
baseurl=https://rpm.grafana.com
repo_gpgcheck=1
enabled=1
gpgcheck=1
gpgkey=https://rpm.grafana.com/gpg.key
sslverify=1
sslcacert=/etc/pki/tls/certs/ca-bundle.crt
EOF

# Install
sudo yum install grafana

# Start
sudo systemctl start grafana-server
sudo systemctl enable grafana-server
```

## 6. Data Sources — Connecting Grafana to Your Data

A **data source** is how you tell Grafana where to get data from. Without a data source, Grafana is just a pretty empty canvas.

### Adding a Data Source

1. Go to **Configuration** (⚙️) → **Data Sources**
2. Click **"Add data source"**
3. Select the type (e.g., Prometheus)
4. Enter the connection details
5. Click **"Save & Test"**

### Common Data Sources

| Data Source | Type | Typical URL |
|------------|------|------------|
| **Prometheus** | Metrics | `http://prometheus:9090` |
| **Loki** | Logs | `http://loki:3100` |
| **Elasticsearch** | Logs/Search | `http://elasticsearch:9200` |
| **InfluxDB** | Metrics | `http://influxdb:8086` |
| **MySQL** | Relational DB | `mysql:3306` |
| **PostgreSQL** | Relational DB | `postgres:5432` |
| **CloudWatch** | AWS Metrics | (uses IAM credentials) |
| **Azure Monitor** | Azure Metrics | (uses service principal) |
| **Jaeger** | Traces | `http://jaeger:16686` |
| **Tempo** | Traces | `http://tempo:3200` |

### Example: Adding Prometheus as a Data Source

Through the UI:
1. Go to **Configuration → Data Sources → Add data source**
2. Select **Prometheus**
3. Set the URL: `http://prometheus:9090` (or `http://localhost:9090` if on the same host)
4. Leave other settings as default
5. Click **Save & Test** — you should see "Data source is working" ✅

### Provisioning Data Sources via Config File (Infrastructure as Code)

Instead of clicking in the UI, you can define data sources in a YAML file. Grafana reads these on startup.

Create a file at `/etc/grafana/provisioning/datasources/datasources.yml`:

```yaml
# /etc/grafana/provisioning/datasources/datasources.yml
apiVersion: 1

datasources:
  # Prometheus — for metrics
  - name: Prometheus
    type: prometheus
    access: proxy              # 'proxy' = Grafana backend fetches data
    url: http://prometheus:9090
    isDefault: true            # This will be the default data source
    editable: true

  # Loki — for logs
  - name: Loki
    type: loki
    access: proxy
    url: http://loki:3100
    editable: true

  # Elasticsearch — for logs/search
  - name: Elasticsearch
    type: elasticsearch
    access: proxy
    url: http://elasticsearch:9200
    jsonData:
      index: "logs-*"
      timeField: "@timestamp"
      esVersion: "8.0.0"
```

### Access Modes Explained

| Mode | How It Works | When to Use |
|------|-------------|-------------|
| **proxy** (Server) | Grafana backend fetches data from the data source | ✅ Recommended — more secure, data source doesn't need to be accessible from browser |
| **direct** (Browser) | Browser fetches data directly from the data source | ⚠️ Only when data source needs to be accessed from user's browser (rare) |

> **Always use `proxy` mode** unless you have a very specific reason not to. It's more secure because your data source URL stays hidden from end users.

## 8. Panels — The Building Blocks

A **panel** is a single visualization on a dashboard. It's the individual chart, graph, table, or stat that shows specific data.

### Panel Types (Visualizations)

Grafana comes with many built-in panel types:

| Panel Type | Best For | Example Use |
|-----------|---------|-------------|
| **Time Series** | Metrics over time | CPU usage, request rate |
| **Stat** | Single big number | Current uptime, total requests |
| **Gauge** | Value within a range | CPU at 73% (with colored arc) |
| **Bar Gauge** | Comparing values | Memory usage per server |
| **Table** | Tabular data | List of pods, log entries |
| **Heatmap** | Distribution over time | Response time distribution |
| **Pie Chart** | Proportions | Traffic by region |
| **Bar Chart** | Categorical comparison | Errors by service |
| **Histogram** | Distribution of values | Request latency distribution |
| **State Timeline** | State changes over time | Service up/down history |
| **Logs** | Log entries | Application logs from Loki |
| **Node Graph** | Service topology | Service dependencies |
| **Geomap** | Geographic data | Server locations on a map |
| **Alert List** | Active alerts | Current firing alerts |
| **Text** | Static markdown text | Dashboard descriptions |

### Creating a Panel — Step by Step

Let's create a CPU usage panel with Prometheus:

1. **Edit mode**: Click "Add panel" → "Add a new panel"
2. **Select Data Source**: Choose "Prometheus" from the dropdown
3. **Write Query**: 

```promql
100 - (avg by(instance) (rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100)
```

This query means:
- `node_cpu_seconds_total{mode="idle"}` — total idle CPU seconds
- `rate(...[5m])` — calculate the per-second rate over 5 minutes
- `avg by(instance)` — average across all CPU cores, grouped by server
- `100 - (... * 100)` — convert to percentage and flip (idle → busy)

4. **Choose Visualization**: Select "Time Series"
5. **Customize**:
   - Set title: "CPU Usage (%)"
   - Set Y-axis: Min = 0, Max = 100
   - Add thresholds: Green (0-70), Yellow (70-90), Red (90-100)

### Panel Configuration Deep Dive

When editing a panel, you have these tabs:

#### Query Tab
```
┌──────────────────────────────────────────────────────┐
│ Data Source: [Prometheus ▾]                           │
│                                                      │
│ Metric: node_cpu_seconds_total                       │
│ Label filters: mode = idle                           │
│ Operations: Rate [5m] → Avg by (instance)            │
│                                                      │
│ Legend: {{instance}}                                  │
│ Min interval: 15s                                    │
│                                                      │
│ [+ Query]  [+ Expression]                            │
└──────────────────────────────────────────────────────┘
```

- **Multiple Queries**: You can add multiple queries (A, B, C...) to one panel
- **Legend Format**: Use `{{label_name}}` to show label values in the legend
- **Min Interval**: Minimum time interval between data points

#### Transform Tab

Transforms let you manipulate data **after** it's been fetched:

| Transform | What It Does |
|-----------|-------------|
| **Reduce** | Collapse a time series to a single value (last, avg, max) |
| **Merge** | Combine multiple queries into one table |
| **Filter by name** | Show only specific series |
| **Organize fields** | Rename, reorder, hide columns |
| **Add field from calculation** | Create new fields (e.g., percentage) |
| **Sort by** | Sort rows |
| **Join by field** | Join multiple results like a SQL JOIN |

#### Overrides Tab

Overrides let you customize the appearance of **specific series**:

```
Override 1:
  Match: Fields with name = "server-01"
  Properties:
    - Color: Red
    - Line width: 3
    - Fill opacity: 20%

Override 2:
  Match: Fields with name = "server-02"  
  Properties:
    - Color: Blue
    - Line style: Dashed
```

### Panel Options (Right Side Panel)

| Section | Options |
|---------|---------|
| **Title** | Panel name, description |
| **Legend** | Show/hide, placement (bottom/right), display mode (list/table) |
| **Tooltip** | Single / All series on hover |
| **Axis** | Min, max, scale (linear/log), label, units |
| **Thresholds** | Color-based thresholds (green/yellow/red) |
| **Value mappings** | Map values to text (e.g., 1 → "Up", 0 → "Down") |
| **Links** | Clickable links (drill down to another dashboard) |

### Setting Thresholds

Thresholds add color to your panels based on values:

```
Thresholds:
  Base color: Green (default)
  
  Step 1: 70 → Yellow    (Warning)
  Step 2: 90 → Red       (Critical)
```

This means:
- 0–70: Green ✅ (Everything's fine)
- 70–90: Yellow ⚠️ (Warning — keep an eye on it)
- 90–100: Red 🔴 (Critical — take action!)

### Value Mappings

Map raw values to human-readable text:

```
Value Mappings:
  1 → "Up"   (color: green)
  0 → "Down" (color: red)
```

This is super useful for status panels where your metric returns 1 or 0.

### Units

Grafana supports tons of units out of the box:

| Category | Units |
|----------|-------|
| **Data** | bytes, KB, MB, GB, TB |
| **Time** | ms, seconds, minutes, hours |
| **Throughput** | bytes/sec, packets/sec, requests/sec |
| **Percentage** | percent (0-100), percent (0.0-1.0) |
| **Data rate** | bits/sec, Mbits/sec |
| **Currency** | USD, EUR, etc. |

Setting the right unit ensures Grafana formats the values nicely (e.g., showing "4.2 GB" instead of "4200000000 bytes").

## 9. Variables — Making Dashboards Dynamic

Variables are **one of the most powerful features** in Grafana. They let you create **interactive, reusable dashboards** where users can filter data using dropdowns.

### Why Variables?

Without variables: You'd need a separate dashboard for each server, each environment, each application.

With variables: One dashboard handles everything. Users just select what they want from a dropdown.


### Creating Variables

1. Go to **Dashboard Settings** (⚙️) → **Variables** → **"New variable"**

### Types of Variables

| Type | Description | Example |
|------|------------|---------|
| **Query** | Populated from a data source query | List of servers from Prometheus |
| **Custom** | Manually defined comma-separated values | `dev, staging, prod` |
| **Constant** | A fixed hidden value | A Prometheus job name |
| **Data source** | Dropdown to select which data source to use | Switch between Prometheus instances |
| **Interval** | Time interval selection | `1m, 5m, 15m, 1h` |
| **Text box** | Free-text input | Custom filter string |
| **Ad hoc filters** | Automatic key-value pair filtering | Filter by any label dynamically |

### Example: Creating a "Server" Variable (Query Type)

**Settings:**
- **Name**: `server`
- **Type**: Query
- **Data Source**: Prometheus
- **Query**: `label_values(node_uname_info, instance)`
- **Multi-value**: ✅ (allows selecting multiple servers)
- **Include All option**: ✅ (adds an "All" option)

This query asks Prometheus: "Give me all unique values of the `instance` label from the `node_uname_info` metric."

**Result**: A dropdown appears at the top of the dashboard with all your server names.

### Using Variables in Queries

Once you've defined a variable, use it in your panel queries with `$variable_name` or `${variable_name}`:

```promql
# Single value variable
rate(node_cpu_seconds_total{instance="$server", mode="idle"}[5m])

# Multi-value variable (use regex match)
rate(node_cpu_seconds_total{instance=~"$server", mode="idle"}[5m])
```

> **Important**: When a variable can have multiple values, use `=~` (regex match) instead of `=` (exact match). Grafana automatically joins multiple values with `|` to create a regex like `server-01|server-02|server-03`.

### Using Variables in Panel Titles

You can use variables in panel titles too:

```
Panel Title: CPU Usage — $server
```

This will dynamically change the title based on the selected server!

### Chained Variables (Dependent Dropdowns)

You can create variables that depend on other variables:

**Variable 1: `environment`**
- Query: `label_values(up, env)`
- Values: `dev`, `staging`, `prod`

**Variable 2: `server`**
- Query: `label_values(up{env="$environment"}, instance)`
- This only shows servers in the selected environment!

Result: When user selects `prod` in the environment dropdown, the server dropdown only shows production servers.

### Interval Variable

Great for letting users control the granularity of data:

- **Name**: `interval`
- **Type**: Interval
- **Values**: `1m, 5m, 15m, 30m, 1h`

Use it in queries:
```promql
rate(http_requests_total[${interval}])
```

### Repeat Panels/Rows by Variable

This is incredibly useful! You can repeat a panel for each value of a variable:

1. Create a multi-value variable `server` (with values server-01, server-02, server-03)
2. Create a panel with the query using `$server`
3. In panel settings → **Repeat options** → Repeat by: `server`
4. Grafana will automatically create one copy of the panel for each server!

## 11. Alerting in Grafana

Grafana's built-in alerting lets you get notified when something goes wrong — without needing a separate alerting tool.

### How Grafana Alerting Works (Grafana 9+)

Grafana has a **unified alerting system** (since version 9) that works with any data source:

```
┌──────────────┐     ┌────────────────┐     ┌─────────────────┐
│  Alert Rule  │────▶│  Evaluation    │────▶│ Contact Points   │
│ (conditions) │     │  Engine        │     │ (Slack, Email,   │
│              │     │  (checks every │     │  PagerDuty, etc.)│
│              │     │   X seconds)   │     │                  │
└──────────────┘     └────────┬───────┘     └─────────────────┘
                              │
                     ┌────────▼───────┐
                     │ Notification   │
                     │ Policies       │
                     │ (routing rules)│
                     └────────────────┘
```

### Key Concepts

| Concept | What It Is |
|---------|-----------|
| **Alert Rule** | The condition that triggers an alert (e.g., "CPU > 90% for 5 minutes") |
| **Contact Point** | Where the notification goes (Slack, Email, PagerDuty, Webhook) |
| **Notification Policy** | Routing rules — which alerts go to which contact point |
| **Silence** | Temporarily mute alerts (during maintenance) |
| **Alert Group** | Group related alerts together |

### Creating an Alert Rule

#### Step 1: Define the Query

```promql
# Query A — Get the CPU usage
100 - (avg(rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100)
```

#### Step 2: Define the Condition

```
# Expression B — Threshold
WHEN  last()  OF  A  IS ABOVE  90
```

#### Step 3: Set Evaluation Behavior

```
Evaluate every: 1m          # Check the condition every 1 minute
For: 5m                     # Must be true for 5 minutes before firing
                            # (prevents flapping alerts)
```

#### Step 4: Add Labels and Annotations

```yaml
Labels:
  severity: critical
  team: infrastructure

Annotations:
  summary: "High CPU usage detected"
  description: "CPU usage is {{ $value }}% on {{ $labels.instance }}"
```

### Setting Up Contact Points

#### Slack

```yaml
Contact Point: Slack Alerts
Type: Slack
Webhook URL: https://hooks.slack.com/services/T00/B00/XXXX
Channel: #alerts
Title: "[{{ .Status }}] {{ .CommonLabels.alertname }}"
Message: "{{ .CommonAnnotations.summary }}"
```

#### Email

```yaml
Contact Point: Email Alerts
Type: Email
Addresses: devops-team@company.com
Subject: "[Grafana Alert] {{ .CommonLabels.alertname }}"
```

#### PagerDuty

```yaml
Contact Point: PagerDuty
Type: PagerDuty
Integration Key: your-pagerduty-integration-key
Severity: critical
```

### Notification Policies

Notification policies define **routing rules** — which alerts go where:

```yaml
# Default policy — catch-all
Default contact point: Email Alerts

# Nested policies
Policies:
  # All critical alerts → PagerDuty
  - Match:
      severity: critical
    Contact point: PagerDuty
    Continue: true           # Also check next policy

  # All infrastructure alerts → Slack #infra-alerts
  - Match:
      team: infrastructure
    Contact point: Slack Infra

  # All application alerts → Slack #app-alerts
  - Match:
      team: application
    Contact point: Slack App
```

### Silences

Use silences during planned maintenance to avoid alert spam:

```
Silence:
  Matchers: severity=warning, team=infrastructure
  Start: 2024-03-15 02:00 UTC
  End:   2024-03-15 06:00 UTC
  Comment: "Planned maintenance — server upgrades"
  Created by: ritesh
```

### Grafana Alerting vs Prometheus Alertmanager

| Feature | Grafana Alerting | Prometheus Alertmanager |
|---------|-----------------|----------------------|
| **Data Sources** | Works with ANY data source | Only Prometheus |
| **UI** | Beautiful, visual rule editor | Config files (YAML) |
| **Grouping** | Yes | Yes |
| **Silencing** | Yes | Yes |
| **Routing** | Yes | Yes |
| **Recommended for** | Multi-data-source environments | Pure Prometheus setups |

> **Pro Tip**: In production, many teams use **Prometheus Alertmanager** for metrics-based alerts and **Grafana Alerting** for alerts that span multiple data sources (e.g., "alert when Prometheus metric X is high AND Elasticsearch shows error logs").

## 12. Dashboard Provisioning — Dashboards as Code

In production, you don't want to create dashboards manually in the UI. You want them **provisioned automatically** — defined as code, stored in Git, and deployed alongside your infrastructure.

### How Provisioning Works

Grafana reads provisioning files from `/etc/grafana/provisioning/`:

```
/etc/grafana/provisioning/
├── dashboards/
│   └── dashboards.yml       # Tells Grafana where to find dashboard JSON files
├── datasources/
│   └── datasources.yml      # Data source definitions
├── alerting/
│   └── alerts.yml           # Alert rules
└── notifiers/
    └── notifiers.yml        # Notification channels
```

### Dashboard Provisioning Config

```yaml
# /etc/grafana/provisioning/dashboards/dashboards.yml
apiVersion: 1

providers:
  - name: "Default"
    orgId: 1
    folder: "Provisioned"
    type: file
    disableDeletion: false
    editable: true
    options:
      path: /var/lib/grafana/dashboards    # Where dashboard JSON files live
      foldersFromFilesStructure: true       # Create folders based on directory structure
```

Then place your dashboard JSON files in `/var/lib/grafana/dashboards/`:

```
/var/lib/grafana/dashboards/
├── infrastructure/
│   ├── node-exporter.json
│   └── docker.json
├── kubernetes/
│   ├── cluster-overview.json
│   └── pod-metrics.json
└── applications/
    └── api-gateway.json
```

### Docker Compose Example with Provisioning

```yaml
# docker-compose.yml
version: "3.8"
services:
  grafana:
    image: grafana/grafana-oss:latest
    ports:
      - "3000:3000"
    volumes:
      - ./grafana/provisioning:/etc/grafana/provisioning
      - ./grafana/dashboards:/var/lib/grafana/dashboards
    environment:
      - GF_SECURITY_ADMIN_USER=admin
      - GF_SECURITY_ADMIN_PASSWORD=admin123
```

## 15. Best Practices

### Dashboard Design

1. **One purpose per dashboard** — Don't cram everything into one dashboard
2. **Use rows to group related panels** — Collapsible rows help organize
3. **Put the most important panels at the top** — Users see them first
4. **Use consistent colors** — Green = good, Yellow = warning, Red = bad
5. **Set appropriate time ranges** — Default to the most useful range
6. **Add text panels** — Describe what the dashboard shows and how to use it

### Variables

7. **Always use variables** — Never hardcode server names or environments
8. **Add an "All" option** — Let users see everything at once
9. **Use chained variables** — Environment → Cluster → Namespace → Pod

### Performance

10. **Avoid too many panels** — 20-30 panels max per dashboard
11. **Set appropriate min intervals** — Don't query at 1s resolution for a 7-day view
12. **Use `$__rate_interval`** — Grafana's smart rate interval variable

```promql
# ✅ Good — uses Grafana's smart interval
rate(http_requests_total[$__rate_interval])

# ❌ Bad — hardcoded interval
rate(http_requests_total[5m])
```

### Alerting

13. **Set a "For" duration** — Prevent flapping (at least 5 minutes)
14. **Use severity labels** — critical, warning, info
15. **Don't alert on everything** — Only alert on actionable conditions
16. **Group related alerts** — Reduce notification noise

## 16. Common Interview Questions

**Q: What is Grafana and why is it used?**
> Grafana is an open-source visualization platform used to create dashboards from various data sources like Prometheus, Loki, and Elasticsearch. It provides a unified view for monitoring, logging, and observability.

**Q: What's the difference between Grafana and Kibana?**
> Kibana is tightly coupled with Elasticsearch, while Grafana is data-source agnostic — it can connect to Prometheus, Loki, Elasticsearch, InfluxDB, MySQL, and many more simultaneously.

**Q: How do you make Grafana dashboards dynamic?**
> Using **template variables**. You define variables that populate from data source queries, and use them in panel queries and titles. This allows one dashboard to serve multiple environments, servers, or applications.

**Q: What access modes does Grafana support for data sources?**
> **Proxy** (server-side, recommended) and **Direct** (browser-side). Proxy mode is preferred because it keeps data source URLs hidden from users.

**Q: How does Grafana alerting work?**
> Grafana evaluates alert rules at defined intervals. When a condition is met for a specified duration, it fires an alert to contact points (Slack, Email, PagerDuty) through notification policies.

**Q: How do you provision Grafana dashboards as code?**
> Using Grafana's provisioning system — define data sources and dashboard providers in YAML files under `/etc/grafana/provisioning/`, and store dashboard JSON files in a specified directory.
