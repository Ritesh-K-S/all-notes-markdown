## 1. What is the Prometheus + Grafana Stack?

The **Prometheus + Grafana stack** is the most popular open-source monitoring solution in the DevOps world. It's the go-to combo for monitoring servers, containers, Kubernetes clusters, and applications.

Think of it like this:

| Component | Role | Analogy |
|-----------|------|---------|
| **Exporters** (Node Exporter, cAdvisor) | Collect raw data from your systems | 🌡️ The thermometers and sensors |
| **Prometheus** | Scrapes, stores, and queries the data | 🗄️ The database + data collector |
| **Alertmanager** | Sends alerts when things go wrong | 📢 The alarm system |
| **Grafana** | Visualizes the data beautifully | 📊 The dashboard / display screen |

### Why This Stack?

- ✅ **100% open source and free**
- ✅ **Battle-tested** — used by CNCF, Uber, DigitalOcean, Slack, and thousands more
- ✅ **Cloud-native** — built for containers and Kubernetes
- ✅ **Highly extensible** — hundreds of exporters available
- ✅ **Active community** — constant development and support
- ✅ **Industry standard** — expected knowledge for any DevOps engineer

### The Flow (Step by Step)

1. **Exporters** run on your servers/containers and expose metrics at HTTP endpoints (e.g., `:9100/metrics`)
2. **Prometheus** scrapes these endpoints at regular intervals (e.g., every 15 seconds)
3. **Prometheus** stores the metrics in its time-series database (TSDB)
4. **Prometheus** evaluates alert rules and sends firing alerts to **Alertmanager**
5. **Alertmanager** routes alerts to the right channels (Slack, Email, PagerDuty)
6. **Grafana** queries Prometheus via PromQL and displays beautiful dashboards

## 6. Step 2: Prometheus Configuration

This is the main config file that tells Prometheus:
- How often to scrape
- Where to find targets
- Which alert rules to load
- Where to send alerts

```yaml
# prometheus/prometheus.yml

# Global configuration — applies to all scrape jobs
global:
  scrape_interval: 15s          # How often to scrape targets (default)
  evaluation_interval: 15s      # How often to evaluate alert rules
  scrape_timeout: 10s           # Timeout for each scrape attempt

  # Labels added to all metrics and alerts from this Prometheus instance
  external_labels:
    monitor: "monitoring-stack"
    environment: "development"

# Alertmanager configuration — where to send alerts
alerting:
  alertmanagers:
    - static_configs:
        - targets:
            - "alertmanager:9093"    # Alertmanager container name and port

# Alert rule files — Prometheus loads these and evaluates them
rule_files:
  - "alert-rules.yml"               # Our alert rules file

# Scrape configurations — define what to monitor
scrape_configs:

  # ─────────────────────────────────────────────
  # Job 1: Monitor Prometheus itself
  # ─────────────────────────────────────────────
  - job_name: "prometheus"
    # Override global scrape_interval for this job
    scrape_interval: 5s
    static_configs:
      - targets: ["localhost:9090"]
        labels:
          instance: "prometheus-server"

  # ─────────────────────────────────────────────
  # Job 2: Monitor host system (Node Exporter)
  # ─────────────────────────────────────────────
  - job_name: "node-exporter"
    scrape_interval: 15s
    static_configs:
      - targets: ["node-exporter:9100"]
        labels:
          instance: "docker-host"

  # ─────────────────────────────────────────────
  # Job 3: Monitor Docker containers (cAdvisor)
  # ─────────────────────────────────────────────
  - job_name: "cadvisor"
    scrape_interval: 15s
    static_configs:
      - targets: ["cadvisor:8080"]
        labels:
          instance: "docker-containers"

  # ─────────────────────────────────────────────
  # Job 4: Monitor Grafana
  # ─────────────────────────────────────────────
  - job_name: "grafana"
    scrape_interval: 15s
    static_configs:
      - targets: ["grafana:3000"]
        labels:
          instance: "grafana-server"

  # ─────────────────────────────────────────────
  # Job 5: Monitor Alertmanager
  # ─────────────────────────────────────────────
  - job_name: "alertmanager"
    scrape_interval: 15s
    static_configs:
      - targets: ["alertmanager:9093"]
        labels:
          instance: "alertmanager-server"
```

### Let's Break Down Each Section

**`global`** — Default settings for all scrape jobs:
- `scrape_interval: 15s` — every 15 seconds, Prometheus pulls metrics from each target
- `evaluation_interval: 15s` — every 15 seconds, Prometheus checks if any alert rules are firing
- `external_labels` — labels added to all metrics (useful when you have multiple Prometheus instances)

**`alerting`** — Where to send alerts. We point to our Alertmanager container.

**`rule_files`** — List of files containing alert rules (we'll create this next).

**`scrape_configs`** — The heart of the config. Each `job_name` defines a group of targets to scrape:
- `static_configs` — manually defined targets (for dynamic targets, you'd use service discovery)
- `targets` — the `host:port` to scrape. Since everything runs in Docker Compose, we use container names as hostnames.
- `labels` — extra labels to add to all metrics from this target

## 7. Step 3: Alertmanager Configuration

Alertmanager receives alerts from Prometheus and routes them to the right notification channel.

```yaml
# alertmanager/alertmanager.yml

# Global settings
global:
  resolve_timeout: 5m           # How long to wait before marking an alert as resolved
  # SMTP settings for email alerts (uncomment and fill in to use)
  # smtp_smarthost: 'smtp.gmail.com:587'
  # smtp_from: 'alertmanager@example.com'
  # smtp_auth_username: 'your-email@gmail.com'
  # smtp_auth_password: 'your-app-password'
  # smtp_require_tls: true

# Templates for notification messages (optional)
templates:
  - '/etc/alertmanager/templates/*.tmpl'

# The routing tree — how alerts get routed to receivers
route:
  # The default receiver — catches everything not matched by child routes
  receiver: 'default-receiver'

  # How long to wait before sending a notification for a new group of alerts
  group_wait: 30s

  # How long to wait before sending notifications about new alerts
  # added to an existing group of alerts
  group_interval: 5m

  # How long to wait before re-sending a notification
  repeat_interval: 4h

  # Group alerts by these labels
  group_by: ['alertname', 'severity']

  # Child routes — more specific routing
  routes:
    # Critical alerts → high-priority channel
    - match:
        severity: critical
      receiver: 'critical-alerts'
      group_wait: 10s           # Send critical alerts faster!
      repeat_interval: 1h

    # Warning alerts → normal channel
    - match:
        severity: warning
      receiver: 'warning-alerts'
      repeat_interval: 4h

# Receivers — where notifications actually go
receivers:
  # Default receiver — just logs to Alertmanager (good for testing)
  - name: 'default-receiver'
    # Webhook receiver — sends JSON POST to a URL
    # Useful for custom integrations or testing
    webhook_configs:
      - url: 'http://localhost:5001/'    # You can set up a simple HTTP server to test
        send_resolved: true

  # Critical alerts receiver
  - name: 'critical-alerts'
    # Slack notification (uncomment and configure)
    # slack_configs:
    #   - api_url: 'https://hooks.slack.com/services/T00/B00/XXXX'
    #     channel: '#critical-alerts'
    #     title: '🚨 [{{ .Status | toUpper }}] {{ .CommonLabels.alertname }}'
    #     text: >-
    #       *Alert:* {{ .CommonLabels.alertname }}
    #       *Severity:* {{ .CommonLabels.severity }}
    #       *Description:* {{ .CommonAnnotations.description }}
    #       *Details:*
    #       {{ range .Alerts }}
    #         - *{{ .Labels.instance }}*: {{ .Annotations.summary }}
    #       {{ end }}
    #     send_resolved: true

    # For testing, just use a webhook
    webhook_configs:
      - url: 'http://localhost:5001/'
        send_resolved: true

  # Warning alerts receiver
  - name: 'warning-alerts'
    webhook_configs:
      - url: 'http://localhost:5001/'
        send_resolved: true

# Inhibition rules — suppress certain alerts when others are firing
inhibit_rules:
  # If a critical alert is firing, suppress the corresponding warning alert
  - source_match:
      severity: 'critical'
    target_match:
      severity: 'warning'
    equal: ['alertname', 'instance']
```

### Let's Break Down the Alertmanager Config

**`route`** — The routing tree decides where each alert goes:
- `group_wait: 30s` — When a new alert comes in, wait 30 seconds to see if more come (so they can be grouped together)
- `group_interval: 5m` — After the first notification, wait 5 minutes before adding new alerts to the same notification
- `repeat_interval: 4h` — Re-send the notification every 4 hours if the alert is still firing
- `group_by` — Group alerts by these labels. Same `alertname` + `severity` → same notification

**`receivers`** — The actual notification channels. You can have Slack, Email, PagerDuty, Webhook, etc.

**`inhibit_rules`** — Smart suppression. If a "critical" CPU alert is firing, don't also send the "warning" CPU alert — that would be noise.

## 8. Step 4: Alert Rules

Alert rules define the conditions that trigger alerts.

```yaml
# prometheus/alert-rules.yml

groups:
  # ─────────────────────────────────────────────
  # Group 1: Instance Availability Alerts
  # ─────────────────────────────────────────────
  - name: instance-alerts
    rules:
      # Alert: A target is down
      - alert: InstanceDown
        expr: up == 0
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "Instance {{ $labels.instance }} is down"
          description: "{{ $labels.instance }} of job {{ $labels.job }} has been down for more than 1 minute."

  # ─────────────────────────────────────────────
  # Group 2: Host System Alerts (Node Exporter)
  # ─────────────────────────────────────────────
  - name: host-alerts
    rules:
      # Alert: High CPU Usage
      - alert: HighCpuUsage
        expr: 100 - (avg by(instance) (rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100) > 80
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "High CPU usage on {{ $labels.instance }}"
          description: "CPU usage is above 80% (current value: {{ $value | printf \"%.2f\" }}%)"

      # Alert: Critical CPU Usage
      - alert: CriticalCpuUsage
        expr: 100 - (avg by(instance) (rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100) > 95
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "Critical CPU usage on {{ $labels.instance }}"
          description: "CPU usage is above 95% (current value: {{ $value | printf \"%.2f\" }}%)"

      # Alert: High Memory Usage
      - alert: HighMemoryUsage
        expr: (1 - (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes)) * 100 > 80
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "High memory usage on {{ $labels.instance }}"
          description: "Memory usage is above 80% (current value: {{ $value | printf \"%.2f\" }}%)"

      # Alert: Disk Space Running Low
      - alert: DiskSpaceLow
        expr: (1 - (node_filesystem_avail_bytes{fstype!~"tmpfs|fuse.lxcfs"} / node_filesystem_size_bytes{fstype!~"tmpfs|fuse.lxcfs"})) * 100 > 85
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Disk space is running low on {{ $labels.instance }}"
          description: "Disk {{ $labels.mountpoint }} is {{ $value | printf \"%.2f\" }}% full"

      # Alert: Disk Space Critical
      - alert: DiskSpaceCritical
        expr: (1 - (node_filesystem_avail_bytes{fstype!~"tmpfs|fuse.lxcfs"} / node_filesystem_size_bytes{fstype!~"tmpfs|fuse.lxcfs"})) * 100 > 95
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "Disk space is critically low on {{ $labels.instance }}"
          description: "Disk {{ $labels.mountpoint }} is {{ $value | printf \"%.2f\" }}% full. Immediate action required!"

      # Alert: High Network Traffic (receiving)
      - alert: HighNetworkReceive
        expr: rate(node_network_receive_bytes_total{device!="lo"}[5m]) / 1024 / 1024 > 100
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "High network receive traffic on {{ $labels.instance }}"
          description: "Network receive rate is above 100 MB/s on {{ $labels.device }}"

  # ─────────────────────────────────────────────
  # Group 3: Container Alerts (cAdvisor)
  # ─────────────────────────────────────────────
  - name: container-alerts
    rules:
      # Alert: Container High CPU
      - alert: ContainerHighCpu
        expr: sum(rate(container_cpu_usage_seconds_total{name!=""}[5m])) by (name) > 0.8
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Container {{ $labels.name }} has high CPU usage"
          description: "Container {{ $labels.name }} CPU usage is above 80% (current: {{ $value | printf \"%.2f\" }})"

      # Alert: Container High Memory
      - alert: ContainerHighMemory
        expr: (container_memory_usage_bytes{name!=""} / container_spec_memory_limit_bytes{name!=""}) * 100 > 80
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Container {{ $labels.name }} has high memory usage"
          description: "Container {{ $labels.name }} memory usage is above 80%"

      # Alert: Container Restarting
      - alert: ContainerRestarting
        expr: increase(container_restart_count{name!=""}[1h]) > 3
        for: 0m
        labels:
          severity: warning
        annotations:
          summary: "Container {{ $labels.name }} is restarting frequently"
          description: "Container {{ $labels.name }} has restarted {{ $value }} times in the last hour"

  # ─────────────────────────────────────────────
  # Group 4: Prometheus Self-Monitoring
  # ─────────────────────────────────────────────
  - name: prometheus-alerts
    rules:
      # Alert: Prometheus config reload failed
      - alert: PrometheusConfigReloadFailed
        expr: prometheus_config_last_reload_successful == 0
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "Prometheus config reload failed"
          description: "Prometheus was unable to reload its configuration file."

      # Alert: Too many scrape failures
      - alert: PrometheusHighScrapeFailures
        expr: rate(prometheus_target_scrapes_exceeded_sample_limit_total[5m]) > 0
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Prometheus is experiencing scrape failures"
          description: "Some targets are exceeding the sample limit."
```

### Understanding Alert Rules

Each alert rule has these parts:

```yaml
- alert: HighCpuUsage                     # Alert name (shows in notifications)
  expr: <PromQL expression> > 80           # The condition (a PromQL query)
  for: 5m                                  # How long the condition must be true
  labels:                                  # Labels to attach to the alert
    severity: warning
  annotations:                             # Human-readable info
    summary: "Short description"
    description: "Detailed description with {{ $value }}"
```

- **`expr`**: A PromQL query that returns a value. If the value meets the condition (> 80), the alert starts.
- **`for`**: The "pending" period. The condition must be true for this duration continuously before firing. This prevents false alarms from brief spikes.
- **`labels`**: Used for routing in Alertmanager (e.g., `severity: critical` → PagerDuty)
- **`annotations`**: Human-readable text. `{{ $value }}` is the current metric value. `{{ $labels.instance }}` shows the instance label.

## 9. Step 5: Grafana Provisioning — Data Sources

This file tells Grafana to automatically configure data sources on startup (no manual UI clicking needed).

```yaml
# grafana/provisioning/datasources/datasources.yml

apiVersion: 1

datasources:
  # ─────────────────────────────────────────────
  # Prometheus — for metrics
  # ─────────────────────────────────────────────
  - name: Prometheus
    type: prometheus
    access: proxy
    url: http://prometheus:9090
    isDefault: true
    editable: true
    jsonData:
      timeInterval: "15s"               # Default scrape interval
      httpMethod: POST                   # Use POST for PromQL queries (handles large queries)
      exemplarTraceIdDestinations:
        - name: traceID
          datasourceUid: tempo           # Link to Tempo if you add tracing later

  # ─────────────────────────────────────────────
  # Alertmanager — for alert state
  # ─────────────────────────────────────────────
  - name: Alertmanager
    type: alertmanager
    access: proxy
    url: http://alertmanager:9093
    editable: true
    jsonData:
      implementation: prometheus
```

### Why `access: proxy`?

When set to `proxy`, Grafana's **backend** makes the request to Prometheus. The browser talks to Grafana → Grafana talks to Prometheus. This is more secure because:

1. Users never see the Prometheus URL
2. Prometheus doesn't need to be publicly accessible
3. Grafana can add authentication headers if needed

## 11. Step 7: Docker Compose — Bringing It All Together

This is the main file that orchestrates all the services:

```yaml
# docker-compose.yml

version: "3.8"

# ─────────────────────────────────────────────────────────
# Named volumes for persistent storage
# ─────────────────────────────────────────────────────────
volumes:
  prometheus-data:       # Prometheus time-series data
  grafana-data:          # Grafana dashboards, users, settings
  alertmanager-data:     # Alertmanager silence data

# ─────────────────────────────────────────────────────────
# Shared network for all services
# ─────────────────────────────────────────────────────────
networks:
  monitoring:
    driver: bridge

# ─────────────────────────────────────────────────────────
# Services
# ─────────────────────────────────────────────────────────
services:

  # ═══════════════════════════════════════════════════════
  # PROMETHEUS — Metrics collection & storage
  # ═══════════════════════════════════════════════════════
  prometheus:
    image: prom/prometheus:latest
    container_name: prometheus
    restart: unless-stopped
    ports:
      - "9090:9090"                          # Prometheus Web UI
    volumes:
      - ./prometheus/prometheus.yml:/etc/prometheus/prometheus.yml:ro    # Config (read-only)
      - ./prometheus/alert-rules.yml:/etc/prometheus/alert-rules.yml:ro # Alert rules
      - prometheus-data:/prometheus                                      # Data storage
    command:
      - '--config.file=/etc/prometheus/prometheus.yml'
      - '--storage.tsdb.path=/prometheus'
      - '--storage.tsdb.retention.time=15d'        # Keep 15 days of data
      - '--web.console.libraries=/etc/prometheus/console_libraries'
      - '--web.console.templates=/etc/prometheus/consoles'
      - '--web.enable-lifecycle'                    # Enable /-/reload endpoint
      - '--web.enable-admin-api'                    # Enable admin API
    networks:
      - monitoring

  # ═══════════════════════════════════════════════════════
  # GRAFANA — Visualization & dashboards
  # ═══════════════════════════════════════════════════════
  grafana:
    image: grafana/grafana-oss:latest
    container_name: grafana
    restart: unless-stopped
    ports:
      - "3000:3000"                          # Grafana Web UI
    volumes:
      - grafana-data:/var/lib/grafana                                    # Persistent storage
      - ./grafana/provisioning:/etc/grafana/provisioning                 # Data sources & dashboard config
      - ./grafana/dashboards:/var/lib/grafana/dashboards                 # Dashboard JSON files
    environment:
      # Admin credentials
      - GF_SECURITY_ADMIN_USER=admin
      - GF_SECURITY_ADMIN_PASSWORD=admin123

      # Server settings
      - GF_SERVER_ROOT_URL=http://localhost:3000

      # Disable analytics (optional)
      - GF_ANALYTICS_REPORTING_ENABLED=false

      # Install plugins (optional — add any you need)
      # - GF_INSTALL_PLUGINS=grafana-piechart-panel,grafana-clock-panel

      # Unified alerting
      - GF_UNIFIED_ALERTING_ENABLED=true
      - GF_ALERTING_ENABLED=false              # Disable legacy alerting
    depends_on:
      - prometheus
    networks:
      - monitoring

  # ═══════════════════════════════════════════════════════
  # ALERTMANAGER — Alert routing & notifications
  # ═══════════════════════════════════════════════════════
  alertmanager:
    image: prom/alertmanager:latest
    container_name: alertmanager
    restart: unless-stopped
    ports:
      - "9093:9093"                          # Alertmanager Web UI
    volumes:
      - ./alertmanager/alertmanager.yml:/etc/alertmanager/alertmanager.yml:ro
      - alertmanager-data:/alertmanager
    command:
      - '--config.file=/etc/alertmanager/alertmanager.yml'
      - '--storage.path=/alertmanager'
      - '--web.external-url=http://localhost:9093'
    networks:
      - monitoring

  # ═══════════════════════════════════════════════════════
  # NODE EXPORTER — Host system metrics
  # ═══════════════════════════════════════════════════════
  node-exporter:
    image: prom/node-exporter:latest
    container_name: node-exporter
    restart: unless-stopped
    ports:
      - "9100:9100"                          # Node Exporter metrics endpoint
    volumes:
      # Mount host filesystem as read-only for system metrics
      - /proc:/host/proc:ro
      - /sys:/host/sys:ro
      - /:/rootfs:ro
    command:
      - '--path.procfs=/host/proc'
      - '--path.sysfs=/host/sys'
      - '--path.rootfs=/rootfs'
      - '--collector.filesystem.mount-points-exclude=^/(sys|proc|dev|host|etc)($$|/)'
    networks:
      - monitoring

  # ═══════════════════════════════════════════════════════
  # cADVISOR — Container metrics
  # ═══════════════════════════════════════════════════════
  cadvisor:
    image: gcr.io/cadvisor/cadvisor:latest
    container_name: cadvisor
    restart: unless-stopped
    ports:
      - "8080:8080"                          # cAdvisor Web UI
    volumes:
      - /:/rootfs:ro
      - /var/run:/var/run:ro
      - /sys:/sys:ro
      - /var/lib/docker/:/var/lib/docker:ro
      - /dev/disk/:/dev/disk:ro
    privileged: true
    devices:
      - /dev/kmsg
    networks:
      - monitoring
```

### Docker Compose Breakdown

Let me explain the important parts of each service:

**Prometheus:**
- `--storage.tsdb.retention.time=15d` — keeps data for 15 days (adjust based on disk space)
- `--web.enable-lifecycle` — lets you reload config without restarting (`curl -X POST http://localhost:9090/-/reload`)
- Config files are mounted as `:ro` (read-only) for security

**Grafana:**
- `GF_SECURITY_ADMIN_PASSWORD` — sets the admin password (change this in production!)
- `depends_on: prometheus` — starts Grafana after Prometheus
- Provisioning directories are mounted to auto-configure data sources and dashboards

**Node Exporter:**
- Mounts `/proc`, `/sys`, and `/` from the host to read system metrics
- These mounts are **read-only** — Node Exporter only reads, never writes

**cAdvisor:**
- Needs access to Docker's socket and filesystem to monitor containers
- `privileged: true` — required to access kernel info (needed for some container metrics)
- Gives you CPU, memory, network, and disk metrics for every container

## 15. Step 11: Useful PromQL Queries for Dashboards

Here's a cheat sheet of PromQL queries you'll use constantly:

### System Metrics (Node Exporter)

```promql
# ──── CPU ────
# CPU usage (percentage)
100 - (avg by(instance) (rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100)

# CPU usage by mode (system, user, iowait, etc.)
avg by(mode) (rate(node_cpu_seconds_total[5m])) * 100

# Number of CPUs
count without(cpu, mode) (node_cpu_seconds_total{mode="idle"})

# ──── MEMORY ────
# Memory usage percentage
(1 - (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes)) * 100

# Total memory in GB
node_memory_MemTotal_bytes / 1024 / 1024 / 1024

# Used memory in bytes
node_memory_MemTotal_bytes - node_memory_MemAvailable_bytes

# ──── DISK ────
# Disk usage percentage (per mount point)
(1 - (node_filesystem_avail_bytes{fstype!~"tmpfs|fuse.lxcfs"} / node_filesystem_size_bytes{fstype!~"tmpfs|fuse.lxcfs"})) * 100

# Disk read/write rate (bytes per second)
rate(node_disk_read_bytes_total[5m])
rate(node_disk_written_bytes_total[5m])

# ──── NETWORK ────
# Network receive rate (bytes per second)
rate(node_network_receive_bytes_total{device!="lo"}[5m])

# Network transmit rate (bytes per second)
rate(node_network_transmit_bytes_total{device!="lo"}[5m])

# ──── SYSTEM ────
# System uptime
node_time_seconds - node_boot_time_seconds

# System load (1 min, 5 min, 15 min)
node_load1
node_load5
node_load15
```

### Container Metrics (cAdvisor)

```promql
# Container CPU usage (percentage)
sum(rate(container_cpu_usage_seconds_total{name!=""}[5m])) by (name) * 100

# Container memory usage (bytes)
container_memory_usage_bytes{name!=""}

# Container memory usage percentage (of limit)
(container_memory_usage_bytes{name!=""} / container_spec_memory_limit_bytes{name!=""}) * 100

# Container network receive (bytes per second)
rate(container_network_receive_bytes_total{name!=""}[5m])

# Container network transmit (bytes per second)
rate(container_network_transmit_bytes_total{name!=""}[5m])

# Number of running containers
count(container_last_seen{name!=""})

# Container restart count
increase(container_restart_count{name!=""}[1h])
```

### Prometheus Self-Monitoring

```promql
# Number of active targets
count(up)

# Number of targets that are UP
count(up == 1)

# Number of targets that are DOWN
count(up == 0)

# Scrape duration (how long each scrape takes)
prometheus_target_scrape_pool_sync_total

# Total time series in Prometheus
prometheus_tsdb_head_series

# Prometheus memory usage
process_resident_memory_bytes{job="prometheus"}
```

## 18. Production Considerations

When running this stack in production, keep these things in mind:

### Storage

| Concern | Recommendation |
|---------|---------------|
| **Prometheus retention** | 15-30 days is typical. For longer retention, use Thanos or Cortex |
| **Disk type** | Use SSD/NVMe for Prometheus storage — it's I/O intensive |
| **Disk size** | ~2 bytes per sample. Calculate: `scrape_targets × metrics_per_target × samples_per_day × retention_days × 2 bytes` |
| **Grafana storage** | Small — only stores dashboards and settings. 1-5 GB is plenty |

### High Availability

```
                    ┌────────────┐
                    │ Load       │
                    │ Balancer   │
                    └──────┬─────┘
                    ┌──────┴─────┐
              ┌─────▼────┐ ┌─────▼────┐
              │Prometheus │ │Prometheus │   ← Two instances scraping same targets
              │  (main)   │ │ (replica) │
              └─────┬─────┘ └─────┬─────┘
                    │              │
              ┌─────▼──────────────▼─────┐
              │        Thanos Query       │   ← Deduplicates and merges
              └───────────┬───────────────┘
                          │
                    ┌─────▼─────┐
                    │  Grafana  │
                    └───────────┘
```

- Run **2 Prometheus instances** scraping the same targets for redundancy
- Use **Thanos** or **Cortex** for long-term storage and HA
- Run **multiple Grafana instances** behind a load balancer
- Use **MySQL/PostgreSQL** instead of SQLite for Grafana's database

### Security

| What to Do | How |
|-----------|-----|
| **Change default passwords** | Never use `admin/admin` in production |
| **Use HTTPS/TLS** | Put a reverse proxy (Nginx) in front with TLS termination |
| **Restrict access** | Use Grafana's RBAC to limit who can view/edit dashboards |
| **Network policies** | In K8s, restrict which pods can talk to Prometheus |
| **Auth proxy** | Use OAuth2 (Google, GitHub) for Grafana authentication |
| **Encrypt secrets** | Don't put Slack webhooks in plain-text config files |

### Performance Tuning

```yaml
# Prometheus flags for production
command:
  - '--storage.tsdb.retention.time=30d'
  - '--storage.tsdb.retention.size=50GB'         # Also limit by size
  - '--storage.tsdb.wal-compression'             # Compress WAL for less disk I/O
  - '--query.max-concurrency=20'                 # Max concurrent queries
  - '--query.timeout=2m'                         # Query timeout
```

## 19. Troubleshooting Common Issues

### Problem: Prometheus Target Shows "DOWN"

```bash
# Check if the exporter container is running
docker compose ps

# Check exporter logs
docker compose logs node-exporter

# Try to reach the exporter from Prometheus container
docker compose exec prometheus wget -qO- http://node-exporter:9100/metrics | head
```

**Common causes:**
- Container not running
- Wrong port or hostname in `prometheus.yml`
- Network issues (containers not on the same Docker network)

### Problem: Grafana Shows "No Data"

1. **Check the data source**: Configuration → Data Sources → Prometheus → "Save & Test"
2. **Check the time range**: Make sure it includes recent data (try "Last 5 minutes")
3. **Test the query in Prometheus first**: Go to `http://localhost:9090` and run the query there
4. **Check the query**: Make sure metric names and label filters are correct

### Problem: Alertmanager Not Receiving Alerts

```bash
# Check Prometheus alert rules
curl http://localhost:9090/api/v1/rules | python3 -m json.tool

# Check if alerts are firing
curl http://localhost:9090/api/v1/alerts | python3 -m json.tool

# Check Alertmanager is reachable from Prometheus
docker compose exec prometheus wget -qO- http://alertmanager:9093/api/v1/status
```

### Problem: High Prometheus Memory Usage

```bash
# Check number of time series
curl http://localhost:9090/api/v1/query?query=prometheus_tsdb_head_series

# Check cardinality (high-cardinality labels cause memory issues)
curl http://localhost:9090/api/v1/label/__name__/values | python3 -m json.tool | wc -l
```

**Solutions:**
- Increase `scrape_interval` (less data points)
- Drop unused metrics with `metric_relabel_configs`
- Use recording rules to pre-compute expensive queries

### Useful Commands

```bash
# Restart everything
docker compose restart

# Reload Prometheus config (without restart)
curl -X POST http://localhost:9090/-/reload

# Reload Alertmanager config (without restart)
curl -X POST http://localhost:9093/-/reload

# View all containers' resource usage
docker stats

# Stop the entire stack
docker compose down

# Stop and remove all data (fresh start)
docker compose down -v
```
