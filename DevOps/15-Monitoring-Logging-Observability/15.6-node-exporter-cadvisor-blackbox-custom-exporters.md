## 1. What Are Exporters?

Okay, let's start with the fundamental question.

Prometheus collects metrics by **pulling** (scraping) data from targets. But here's the thing — your Linux server, Docker container, or database doesn't natively speak Prometheus's language. They don't expose metrics in the format Prometheus expects.

That's where **exporters** come in.

### The Simple Analogy

Think of exporters as **translators**:

```
┌──────────────┐          ┌──────────────┐          ┌──────────────┐
│  Linux Server │          │   Exporter   │          │  Prometheus  │
│              │          │  (translator) │          │              │
│  Speaks:     │─────────▶│  Translates   │─────────▶│  Understands │
│  /proc, /sys │          │  to Prometheus│          │  Prometheus  │
│  kernel APIs │          │  format       │          │  format      │
└──────────────┘          └──────────────┘          └──────────────┘
```

Your Linux server stores CPU info in `/proc/stat`, memory info in `/proc/meminfo`, disk info in `/proc/diskstats`. These are kernel files — not something Prometheus can directly read.

The **Node Exporter** reads these files, converts them into Prometheus metrics format, and serves them on an HTTP endpoint (`:9100/metrics`). Prometheus then scrapes this endpoint.

### The Prometheus Metrics Format

Every exporter serves data in this simple text format:

```
# HELP node_cpu_seconds_total Seconds the CPUs spent in each mode.
# TYPE node_cpu_seconds_total counter
node_cpu_seconds_total{cpu="0",mode="idle"} 36524.89
node_cpu_seconds_total{cpu="0",mode="system"} 1245.67
node_cpu_seconds_total{cpu="0",mode="user"} 8734.12
node_cpu_seconds_total{cpu="1",mode="idle"} 35987.45
```

Each metric has:
- **`# HELP`** — Human-readable description
- **`# TYPE`** — Metric type (counter, gauge, histogram, summary)
- **Metric name** — `node_cpu_seconds_total`
- **Labels** — `{cpu="0", mode="idle"}` — key-value pairs for filtering/grouping
- **Value** — The actual number

### Types of Metrics

| Type | Description | Example | Goes Up/Down? |
|------|------------|---------|---------------|
| **Counter** | Only goes up (resets on restart) | Total HTTP requests, total CPU seconds | ↑ Only up |
| **Gauge** | Can go up or down | Current memory usage, temperature, queue size | ↑↓ Both |
| **Histogram** | Counts values in buckets + sum + count | Request latency distribution | Buckets ↑ |
| **Summary** | Like histogram but calculates percentiles client-side | Request latency p50, p90, p99 | ↑ Only up |

## 2. How Exporters Work — The Big Picture

### Where Exporters Fit in the Architecture

```
┌────────────────────────────────────────────────────────────────┐
│                    WHAT YOU WANT TO MONITOR                     │
├────────────────┬────────────────┬───────────────┬──────────────┤
│ Linux Servers  │ Docker         │ HTTP Endpoints│ Databases    │
│                │ Containers     │ (APIs, sites) │              │
├────────────────┼────────────────┼───────────────┼──────────────┤
│ Node Exporter  │ cAdvisor       │ Blackbox      │ mysqld_exp   │
│ :9100          │ :8080          │ Exporter      │ postgres_exp │
│                │                │ :9115         │ redis_exp    │
├────────────────┴────────────────┴───────────────┴──────────────┤
│                                                                │
│                     PROMETHEUS :9090                            │
│              (scrapes all exporters every 15s)                  │
│                                                                │
├────────────────────────────────────────────────────────────────┤
│                      GRAFANA :3000                             │
│                  (visualizes the data)                          │
└────────────────────────────────────────────────────────────────┘
```

### Two Types of Exporters

| Type | How It Works | Examples |
|------|-------------|---------|
| **Direct Exporter** | Runs on/next to the thing it monitors. Reads system files or APIs directly. | Node Exporter, cAdvisor |
| **Proxy Exporter** | Receives target info from Prometheus, probes the target, returns results. | Blackbox Exporter, SNMP Exporter |

The difference is important:

**Direct Exporter (Node Exporter):**
```
Prometheus ──scrape──▶ Node Exporter (reads /proc, /sys) ──respond──▶ Prometheus
```

**Proxy Exporter (Blackbox Exporter):**
```
Prometheus ──scrape with target param──▶ Blackbox Exporter ──probe──▶ https://google.com
                                         ◀── probe result ──         
                                         ──respond──▶ Prometheus
```

## 3. Node Exporter — Deep Dive

### What is Node Exporter?

**Node Exporter** is the most fundamental exporter. It exposes **hardware and OS-level metrics** from Linux systems:

- CPU usage (per core, per mode)
- Memory (total, available, buffers, cached)
- Disk (space, I/O, latency)
- Network (bytes in/out, packets, errors)
- Filesystem (mount points, space, inodes)
- System load, uptime, context switches
- And much more...

### Installation

#### Option 1: Docker (Recommended for Quick Setup)

```bash
docker run -d \
  --name=node-exporter \
  --net="host" \
  --pid="host" \
  -v "/:/host:ro,rslave" \
  prom/node-exporter:latest \
  --path.rootfs=/host
```

**Why `--net="host"` and `--pid="host"`?**
- `--net="host"` — so Node Exporter sees the host's network interfaces, not the container's
- `--pid="host"` — so it can see host processes
- `-v "/:/host:ro"` — mounts the entire host filesystem read-only

#### Option 2: Binary Installation (For Production Servers)

```bash
# Download the latest release
NODE_EXPORTER_VERSION="1.7.0"
wget https://github.com/prometheus/node_exporter/releases/download/v${NODE_EXPORTER_VERSION}/node_exporter-${NODE_EXPORTER_VERSION}.linux-amd64.tar.gz

# Extract
tar xvfz node_exporter-*.tar.gz
cd node_exporter-${NODE_EXPORTER_VERSION}.linux-amd64

# Move the binary
sudo mv node_exporter /usr/local/bin/

# Create a dedicated user (security best practice — don't run as root)
sudo useradd --no-create-home --shell /bin/false node_exporter

# Create a systemd service
sudo cat <<EOF > /etc/systemd/system/node_exporter.service
[Unit]
Description=Node Exporter
Documentation=https://prometheus.io/docs/guides/node-exporter/
Wants=network-online.target
After=network-online.target

[Service]
User=node_exporter
Group=node_exporter
Type=simple
ExecStart=/usr/local/bin/node_exporter \
  --collector.systemd \
  --collector.processes

[Install]
WantedBy=multi-user.target
EOF

# Start the service
sudo systemctl daemon-reload
sudo systemctl start node_exporter
sudo systemctl enable node_exporter

# Verify it's running
sudo systemctl status node_exporter
curl http://localhost:9100/metrics | head -20
```

### Node Exporter Collectors

Node Exporter is organized into **collectors** — each collector gathers a specific type of metric. Some are enabled by default, some need to be turned on manually.

#### Default Collectors (Enabled Automatically)

| Collector | What It Monitors | Key Metrics |
|-----------|-----------------|-------------|
| **cpu** | CPU usage per core and mode | `node_cpu_seconds_total` |
| **meminfo** | Memory usage | `node_memory_MemTotal_bytes`, `node_memory_MemAvailable_bytes` |
| **diskstats** | Disk I/O | `node_disk_read_bytes_total`, `node_disk_written_bytes_total` |
| **filesystem** | Filesystem space and inodes | `node_filesystem_avail_bytes`, `node_filesystem_size_bytes` |
| **netdev** | Network interface stats | `node_network_receive_bytes_total`, `node_network_transmit_bytes_total` |
| **loadavg** | System load average | `node_load1`, `node_load5`, `node_load15` |
| **uname** | System info (OS, kernel, hostname) | `node_uname_info` |
| **time** | System time | `node_time_seconds` |
| **boottime** | System boot time | `node_boot_time_seconds` |
| **conntrack** | Network connection tracking | `node_nf_conntrack_entries` |
| **entropy** | Available entropy (for crypto) | `node_entropy_available_bits` |
| **filefd** | File descriptor usage | `node_filefd_allocated`, `node_filefd_maximum` |
| **hwmon** | Hardware temperature sensors | `node_hwmon_temp_celsius` |
| **netstat** | TCP/UDP socket stats | `node_netstat_Tcp_CurrEstab` |
| **sockstat** | Socket statistics | `node_sockstat_TCP_alloc` |
| **stat** | System-wide stats | `node_context_switches_total`, `node_forks_total` |
| **vmstat** | Virtual memory stats | `node_vmstat_pgfault`, `node_vmstat_pgmajfault` |

#### Optional Collectors (Need to Be Enabled)

| Collector | What It Monitors | How to Enable |
|-----------|-----------------|--------------|
| **systemd** | Systemd service states | `--collector.systemd` |
| **processes** | Process stats | `--collector.processes` |
| **tcpstat** | TCP connection states | `--collector.tcpstat` |
| **wifi** | WiFi interface stats | `--collector.wifi` |
| **ethtool** | NIC hardware stats | `--collector.ethtool` |

To enable optional collectors:

```bash
# Enable systemd and processes collectors
node_exporter \
  --collector.systemd \
  --collector.processes \
  --collector.tcpstat
```

To disable a specific default collector:

```bash
# Disable the wifi collector (not needed on servers)
node_exporter --no-collector.wifi --no-collector.infiniband
```

### Key Node Exporter Metrics — Explained

Let's go through the most important metrics and what they mean:

#### CPU Metrics

```
# HELP node_cpu_seconds_total Seconds the CPUs spent in each mode.
# TYPE node_cpu_seconds_total counter
node_cpu_seconds_total{cpu="0",mode="idle"} 36524.89
node_cpu_seconds_total{cpu="0",mode="system"} 1245.67
node_cpu_seconds_total{cpu="0",mode="user"} 8734.12
node_cpu_seconds_total{cpu="0",mode="iowait"} 234.56
node_cpu_seconds_total{cpu="0",mode="irq"} 12.34
node_cpu_seconds_total{cpu="0",mode="softirq"} 45.67
node_cpu_seconds_total{cpu="0",mode="nice"} 0.00
node_cpu_seconds_total{cpu="0",mode="steal"} 0.00
```

**CPU Modes Explained:**

| Mode | Meaning | Concerning If High? |
|------|---------|-------------------|
| **idle** | CPU doing nothing | No — this is good! |
| **user** | Running user-space applications | Normal — your apps are working |
| **system** | Running kernel operations | ⚠️ High = too many system calls |
| **iowait** | Waiting for I/O (disk, network) | ⚠️ High = disk bottleneck |
| **irq** | Handling hardware interrupts | ⚠️ High = hardware issues |
| **softirq** | Handling software interrupts | ⚠️ High = network issues |
| **nice** | Running low-priority processes | Normal |
| **steal** | Time stolen by hypervisor (VM) | ⚠️ High = noisy neighbor / undersized VM |

**PromQL for CPU usage:**

```promql
# Overall CPU usage percentage (most common query)
100 - (avg(rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100)

# CPU usage by mode (to see WHERE the CPU time goes)
avg by(mode) (rate(node_cpu_seconds_total{mode!="idle"}[5m])) * 100

# CPU iowait specifically (disk bottleneck indicator)
avg(rate(node_cpu_seconds_total{mode="iowait"}[5m])) * 100

# CPU steal (VM/cloud — are you getting robbed?)
avg(rate(node_cpu_seconds_total{mode="steal"}[5m])) * 100
```

#### Memory Metrics

```
node_memory_MemTotal_bytes      14523265024    # Total RAM: ~14 GB
node_memory_MemFree_bytes       1234567890     # Actually free (unused)
node_memory_MemAvailable_bytes  8234567890     # Available for new processes
node_memory_Buffers_bytes       234567890      # Kernel buffers
node_memory_Cached_bytes        5678901234     # Page cache
node_memory_SwapTotal_bytes     2147483648     # Total swap: ~2 GB
node_memory_SwapFree_bytes      2147483648     # Free swap
```

**Important: MemFree vs MemAvailable**

This trips up a lot of people!

- **MemFree**: Memory that is completely unused. On a healthy Linux system, this is usually LOW — Linux uses free memory for caching. Low MemFree is NOT a problem!
- **MemAvailable**: Memory that is actually available for new applications. This includes MemFree + reclaimable cache. THIS is the number you should monitor.

```promql
# Memory usage percentage (the RIGHT way)
(1 - (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes)) * 100

# Memory usage percentage (the WRONG way — don't use MemFree)
# (1 - (node_memory_MemFree_bytes / node_memory_MemTotal_bytes)) * 100
# This will always show high usage because Linux caches aggressively!
```

#### Disk Metrics

```promql
# Disk space usage percentage
(1 - (node_filesystem_avail_bytes{fstype!~"tmpfs|fuse.lxcfs"} / node_filesystem_size_bytes{fstype!~"tmpfs|fuse.lxcfs"})) * 100

# Disk I/O — read bytes per second
rate(node_disk_read_bytes_total[5m])

# Disk I/O — write bytes per second
rate(node_disk_written_bytes_total[5m])

# Disk I/O — operations per second (IOPS)
rate(node_disk_reads_completed_total[5m])    # Read IOPS
rate(node_disk_writes_completed_total[5m])   # Write IOPS

# Disk I/O latency (average time per operation)
rate(node_disk_read_time_seconds_total[5m]) / rate(node_disk_reads_completed_total[5m])
```

> **Tip**: We filter out `tmpfs` and `fuse.lxcfs` because these are virtual filesystems (in-memory), not real disks.

#### Network Metrics

```promql
# Network bandwidth — receive (bytes per second)
rate(node_network_receive_bytes_total{device!="lo"}[5m])

# Network bandwidth — transmit (bytes per second)
rate(node_network_transmit_bytes_total{device!="lo"}[5m])

# Network errors — receive
rate(node_network_receive_errs_total{device!="lo"}[5m])

# Network errors — transmit
rate(node_network_transmit_errs_total{device!="lo"}[5m])

# Network packets dropped
rate(node_network_receive_drop_total{device!="lo"}[5m])
```

> **Tip**: We filter out `lo` (loopback) because it's just local traffic (127.0.0.1).

### Node Exporter Textfile Collector

This is a really cool feature! The **textfile collector** lets you expose **custom metrics** through Node Exporter without writing a custom exporter.

How it works:
1. Write a `.prom` file with metrics in the Prometheus format
2. Place it in a directory
3. Node Exporter reads it and exposes the metrics

#### Setup

```bash
# Create the textfile directory
sudo mkdir -p /var/lib/node_exporter/textfile_collector

# Start Node Exporter with textfile collector
node_exporter --collector.textfile.directory=/var/lib/node_exporter/textfile_collector
```

#### Example: Track Last Backup Time

```bash
# Create a script that runs after your backup
cat <<'EOF' > /usr/local/bin/backup-metrics.sh
#!/bin/bash

# Run the backup...
# backup_command_here

# Write the metric
echo "# HELP backup_last_success_timestamp_seconds Unix timestamp of last successful backup."
echo "# TYPE backup_last_success_timestamp_seconds gauge"
echo "backup_last_success_timestamp_seconds $(date +%s)"

EOF

chmod +x /usr/local/bin/backup-metrics.sh

# Run it and write to the textfile directory
/usr/local/bin/backup-metrics.sh > /var/lib/node_exporter/textfile_collector/backup.prom
```

Now Node Exporter will expose `backup_last_success_timestamp_seconds` as a metric!

#### Example: Track Certificate Expiry

```bash
# Check SSL certificate expiry
cat <<'EOF' > /usr/local/bin/cert-metrics.sh
#!/bin/bash

DOMAIN="example.com"
EXPIRY=$(echo | openssl s_client -servername "$DOMAIN" -connect "$DOMAIN:443" 2>/dev/null | openssl x509 -noout -enddate | cut -d= -f2)
EXPIRY_EPOCH=$(date -d "$EXPIRY" +%s)

echo "# HELP ssl_certificate_expiry_seconds Unix timestamp when the SSL certificate expires."
echo "# TYPE ssl_certificate_expiry_seconds gauge"
echo "ssl_certificate_expiry_seconds{domain=\"$DOMAIN\"} $EXPIRY_EPOCH"

EOF

chmod +x /usr/local/bin/cert-metrics.sh

# Add to crontab to run every hour
# 0 * * * * /usr/local/bin/cert-metrics.sh > /var/lib/node_exporter/textfile_collector/cert.prom
```

## 4. cAdvisor — Container Metrics Deep Dive

### What is cAdvisor?

**cAdvisor** (Container Advisor) is Google's open-source tool for monitoring **container resource usage and performance**. It provides per-container metrics for:

- CPU usage
- Memory usage (and limits)
- Network I/O
- Disk I/O
- Filesystem usage

### The Key Difference: Node Exporter vs cAdvisor

| | Node Exporter | cAdvisor |
|--|--------------|---------|
| **Monitors** | The host system (the whole machine) | Individual containers |
| **Granularity** | System-wide | Per-container |
| **CPU** | Total CPU of the host | CPU per container |
| **Memory** | Total memory of the host | Memory per container (with limits) |
| **Example metric** | `node_cpu_seconds_total` | `container_cpu_usage_seconds_total{name="nginx"}` |

You need **both**:
- Node Exporter tells you "the server's CPU is at 80%"
- cAdvisor tells you "the nginx container is using 60% and the Redis container is using 15%"

### Installation

#### Docker (Standalone)

```bash
docker run -d \
  --name=cadvisor \
  --volume=/:/rootfs:ro \
  --volume=/var/run:/var/run:ro \
  --volume=/sys:/sys:ro \
  --volume=/var/lib/docker/:/var/lib/docker:ro \
  --volume=/dev/disk/:/dev/disk:ro \
  --publish=8080:8080 \
  --privileged \
  --device=/dev/kmsg \
  gcr.io/cadvisor/cadvisor:latest
```

**Why all these volume mounts?**

| Mount | Why |
|-------|-----|
| `/:/rootfs:ro` | Read host filesystem info |
| `/var/run:/var/run:ro` | Access Docker socket |
| `/sys:/sys:ro` | Access cgroup info (how containers are resource-limited) |
| `/var/lib/docker/:/var/lib/docker:ro` | Access Docker's data directory |
| `/dev/disk/:/dev/disk:ro` | Access disk device info |

#### Docker Compose (Part of a Stack)

```yaml
cadvisor:
  image: gcr.io/cadvisor/cadvisor:latest
  container_name: cadvisor
  restart: unless-stopped
  ports:
    - "8080:8080"
  volumes:
    - /:/rootfs:ro
    - /var/run:/var/run:ro
    - /sys:/sys:ro
    - /var/lib/docker/:/var/lib/docker:ro
    - /dev/disk/:/dev/disk:ro
  privileged: true
  devices:
    - /dev/kmsg
```

#### Kubernetes (DaemonSet)

In Kubernetes, cAdvisor is actually **built into the kubelet**! You don't need to deploy it separately. The kubelet exposes cAdvisor metrics at:

```
https://<node-ip>:10250/metrics/cadvisor
```

However, if you need a standalone cAdvisor in K8s:

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: cadvisor
  namespace: monitoring
spec:
  selector:
    matchLabels:
      app: cadvisor
  template:
    metadata:
      labels:
        app: cadvisor
    spec:
      containers:
        - name: cadvisor
          image: gcr.io/cadvisor/cadvisor:latest
          ports:
            - containerPort: 8080
          volumeMounts:
            - name: rootfs
              mountPath: /rootfs
              readOnly: true
            - name: var-run
              mountPath: /var/run
              readOnly: true
            - name: sys
              mountPath: /sys
              readOnly: true
            - name: docker
              mountPath: /var/lib/docker
              readOnly: true
          resources:
            limits:
              cpu: 300m
              memory: 200Mi
      volumes:
        - name: rootfs
          hostPath:
            path: /
        - name: var-run
          hostPath:
            path: /var/run
        - name: sys
          hostPath:
            path: /sys
        - name: docker
          hostPath:
            path: /var/lib/docker
```

### Key cAdvisor Metrics

#### Container CPU

```
# Total CPU usage (in seconds) — this is a counter, use rate()
container_cpu_usage_seconds_total{name="nginx"}

# CPU usage in system mode
container_cpu_system_seconds_total{name="nginx"}

# CPU usage in user mode
container_cpu_user_seconds_total{name="nginx"}

# CPU CFS (Completely Fair Scheduler) throttled time
# If this is high, the container needs more CPU!
container_cpu_cfs_throttled_seconds_total{name="nginx"}
```

**PromQL queries:**

```promql
# Container CPU usage percentage
sum(rate(container_cpu_usage_seconds_total{name!=""}[5m])) by (name) * 100

# CPU throttling — container is hitting its CPU limit
rate(container_cpu_cfs_throttled_seconds_total{name!=""}[5m])

# Top 5 containers by CPU usage
topk(5, sum(rate(container_cpu_usage_seconds_total{name!=""}[5m])) by (name) * 100)
```

#### Container Memory

```
# Current memory usage (bytes) — includes cache
container_memory_usage_bytes{name="nginx"}

# RSS (Resident Set Size) — actual memory in use (excludes cache)
container_memory_rss{name="nginx"}

# Memory working set — what Kubernetes uses for OOMKill decisions
container_memory_working_set_bytes{name="nginx"}

# Memory cache — page cache (can be reclaimed)
container_memory_cache{name="nginx"}

# Memory limit — the maximum memory the container can use
container_spec_memory_limit_bytes{name="nginx"}
```

**PromQL queries:**

```promql
# Container memory usage percentage (of limit)
(container_memory_working_set_bytes{name!=""} / container_spec_memory_limit_bytes{name!=""}) * 100

# Containers with no memory limit (dangerous!)
container_spec_memory_limit_bytes{name!=""} == 0

# Top 5 containers by memory usage
topk(5, container_memory_working_set_bytes{name!=""})
```

> **Important**: Use `container_memory_working_set_bytes` instead of `container_memory_usage_bytes` for real memory usage. The `usage_bytes` includes cache (which can be freed), and Kubernetes uses `working_set_bytes` for OOM kill decisions.

#### Container Network

```promql
# Network receive rate (bytes per second)
rate(container_network_receive_bytes_total{name!=""}[5m])

# Network transmit rate (bytes per second)
rate(container_network_transmit_bytes_total{name!=""}[5m])

# Network receive errors
rate(container_network_receive_errors_total{name!=""}[5m])

# Network packets dropped
rate(container_network_receive_packets_dropped_total{name!=""}[5m])
```

#### Container Disk I/O

```promql
# Disk read bytes per second
rate(container_fs_reads_bytes_total{name!=""}[5m])

# Disk write bytes per second
rate(container_fs_writes_bytes_total{name!=""}[5m])

# Filesystem usage percentage
(container_fs_usage_bytes{name!=""} / container_fs_limit_bytes{name!=""}) * 100
```

### Filtering Out Noise

cAdvisor collects metrics for ALL containers, including system containers, pause containers, and POD containers. To filter them out:

```promql
# Only show application containers (filter by name)
container_cpu_usage_seconds_total{name!="", name!~"k8s_POD_.*"}

# In Kubernetes, filter by namespace
container_cpu_usage_seconds_total{namespace="production", container!="POD", container!=""}
```

## 5. Blackbox Exporter — Probing External Endpoints

### What is Blackbox Exporter?

The **Blackbox Exporter** is different from Node Exporter and cAdvisor. Instead of monitoring a system from the inside, it monitors things from the **outside** — like a user would.

It can probe endpoints over:
- **HTTP/HTTPS** — Is my website up? How fast is it responding? Is the SSL certificate valid?
- **TCP** — Is my database port open? Can I connect to Redis?
- **ICMP** — Can I ping a server?
- **DNS** — Is DNS resolving correctly?
- **gRPC** — Is my gRPC service responding?

### The Simple Analogy

| Exporter | Perspective | Like... |
|----------|------------|---------|
| Node Exporter | Inside the server | A doctor examining a patient's organs |
| cAdvisor | Inside the container | A doctor examining individual cells |
| Blackbox Exporter | Outside, as a user | A person checking if the hospital is open and responding |

### Why Do You Need It?

Your server might show low CPU, plenty of memory, no disk issues — everything looks fine internally. But your website is still DOWN for users because of:

- A firewall rule blocking port 443
- An SSL certificate that expired
- A DNS record pointing to the wrong IP
- A load balancer misconfiguration

Blackbox Exporter catches these **external-facing** problems.

### Configuration

The Blackbox Exporter config defines **modules** — each module is a type of probe:

```yaml
# blackbox/blackbox.yml

modules:
  # ─────────────────────────────────────────────
  # HTTP/HTTPS probes
  # ─────────────────────────────────────────────
  http_2xx:
    prober: http
    timeout: 10s
    http:
      valid_http_versions: ["HTTP/1.1", "HTTP/2.0"]
      valid_status_codes: [200]           # Expect 200 OK
      method: GET
      follow_redirects: true
      preferred_ip_protocol: "ip4"
      tls_config:
        insecure_skip_verify: false       # Validate SSL certificates

  http_post_2xx:
    prober: http
    timeout: 10s
    http:
      method: POST
      headers:
        Content-Type: application/json
      body: '{"test": "probe"}'
      valid_status_codes: [200, 201]

  # Probe that also checks SSL certificate expiry
  http_2xx_ssl:
    prober: http
    timeout: 10s
    http:
      valid_status_codes: [200]
      method: GET
      fail_if_ssl: false
      fail_if_not_ssl: true              # MUST be HTTPS
      tls_config:
        insecure_skip_verify: false

  # ─────────────────────────────────────────────
  # TCP probes
  # ─────────────────────────────────────────────
  tcp_connect:
    prober: tcp
    timeout: 5s

  # ─────────────────────────────────────────────
  # ICMP probes (ping)
  # ─────────────────────────────────────────────
  icmp_ping:
    prober: icmp
    timeout: 5s
    icmp:
      preferred_ip_protocol: "ip4"

  # ─────────────────────────────────────────────
  # DNS probes
  # ─────────────────────────────────────────────
  dns_lookup:
    prober: dns
    timeout: 5s
    dns:
      query_name: "example.com"
      query_type: "A"
      valid_rcodes:
        - NOERROR
```

### Prometheus Configuration for Blackbox Exporter

This is where it gets interesting. The Blackbox Exporter uses the **proxy pattern** — Prometheus tells it WHAT to probe:

```yaml
# In prometheus.yml — add these scrape configs

scrape_configs:
  # ─────────────────────────────────────────────
  # HTTP probing — website monitoring
  # ─────────────────────────────────────────────
  - job_name: "blackbox-http"
    metrics_path: /probe              # Not the default /metrics!
    params:
      module: [http_2xx]              # Which module to use
    static_configs:
      - targets:                      # The URLs to probe
          - https://www.google.com
          - https://www.github.com
          - https://your-app.example.com
          - http://localhost:8080/health
    relabel_configs:
      # This magic relabeling is required for proxy exporters
      # It takes the target URL and passes it as a parameter to Blackbox Exporter
      - source_labels: [__address__]
        target_label: __param_target
      - source_labels: [__param_target]
        target_label: instance
      - target_label: __address__
        replacement: blackbox-exporter:9115    # Blackbox Exporter address

  # ─────────────────────────────────────────────
  # TCP probing — port checks
  # ─────────────────────────────────────────────
  - job_name: "blackbox-tcp"
    metrics_path: /probe
    params:
      module: [tcp_connect]
    static_configs:
      - targets:
          - db-server:5432             # PostgreSQL
          - redis-server:6379          # Redis
          - rabbitmq-server:5672       # RabbitMQ
    relabel_configs:
      - source_labels: [__address__]
        target_label: __param_target
      - source_labels: [__param_target]
        target_label: instance
      - target_label: __address__
        replacement: blackbox-exporter:9115

  # ─────────────────────────────────────────────
  # ICMP probing — ping checks
  # ─────────────────────────────────────────────
  - job_name: "blackbox-icmp"
    metrics_path: /probe
    params:
      module: [icmp_ping]
    static_configs:
      - targets:
          - 8.8.8.8                    # Google DNS
          - 1.1.1.1                    # Cloudflare DNS
          - gateway-server
    relabel_configs:
      - source_labels: [__address__]
        target_label: __param_target
      - source_labels: [__param_target]
        target_label: instance
      - target_label: __address__
        replacement: blackbox-exporter:9115
```

### Understanding the Relabel Config (Don't Skip This!)

The `relabel_configs` section looks confusing, but it's doing something simple. Let me explain step by step:

```yaml
# Without relabeling, Prometheus would try to scrape:
#   https://www.google.com/probe  ← WRONG! We want to scrape Blackbox Exporter

# With relabeling:
# Step 1: Take the original target (https://www.google.com)
- source_labels: [__address__]
  target_label: __param_target
  # Now: __param_target = "https://www.google.com"
  # Prometheus will add ?target=https://www.google.com to the request

# Step 2: Set the "instance" label to the target URL (for display in Grafana)
- source_labels: [__param_target]
  target_label: instance
  # Now: instance = "https://www.google.com"

# Step 3: Replace the actual scrape address with the Blackbox Exporter
- target_label: __address__
  replacement: blackbox-exporter:9115
  # Now: Prometheus scrapes http://blackbox-exporter:9115/probe?target=https://www.google.com&module=http_2xx
```

So the final request is:
```
GET http://blackbox-exporter:9115/probe?target=https://www.google.com&module=http_2xx
```

Blackbox Exporter then probes `https://www.google.com` and returns the results to Prometheus.

### Key Blackbox Exporter Metrics

```promql
# ──── Is the target UP? (1 = success, 0 = failure) ────
probe_success{instance="https://www.google.com"}

# ──── How long did the probe take? (seconds) ────
probe_duration_seconds{instance="https://www.google.com"}

# ──── HTTP status code ────
probe_http_status_code{instance="https://www.google.com"}

# ──── SSL certificate expiry (seconds until expiry) ────
probe_ssl_earliest_cert_expiry{instance="https://www.google.com"} - time()

# Days until SSL expiry
(probe_ssl_earliest_cert_expiry - time()) / 86400

# ──── DNS lookup time ────
probe_dns_lookup_time_seconds{instance="https://www.google.com"}

# ──── TLS handshake time ────
probe_http_duration_seconds{phase="tls"}

# ──── HTTP response phases ────
probe_http_duration_seconds{phase="resolve"}    # DNS resolution
probe_http_duration_seconds{phase="connect"}    # TCP connection
probe_http_duration_seconds{phase="tls"}        # TLS handshake
probe_http_duration_seconds{phase="processing"} # Server processing
probe_http_duration_seconds{phase="transfer"}   # Response transfer
```

### Essential Alert Rules for Blackbox Exporter

```yaml
# In prometheus/alert-rules.yml

groups:
  - name: blackbox-alerts
    rules:
      # Alert: Endpoint is DOWN
      - alert: EndpointDown
        expr: probe_success == 0
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "Endpoint {{ $labels.instance }} is down"
          description: "{{ $labels.instance }} has been unreachable for 2 minutes."

      # Alert: SSL certificate expiring soon (< 30 days)
      - alert: SSLCertExpiringSoon
        expr: (probe_ssl_earliest_cert_expiry - time()) / 86400 < 30
        for: 1h
        labels:
          severity: warning
        annotations:
          summary: "SSL certificate for {{ $labels.instance }} expires in {{ $value | printf \"%.0f\" }} days"
          description: "The SSL certificate is expiring soon. Please renew it."

      # Alert: SSL certificate expiring VERY soon (< 7 days)
      - alert: SSLCertExpiringCritical
        expr: (probe_ssl_earliest_cert_expiry - time()) / 86400 < 7
        for: 0m
        labels:
          severity: critical
        annotations:
          summary: "URGENT: SSL certificate for {{ $labels.instance }} expires in {{ $value | printf \"%.0f\" }} days!"

      # Alert: High response time (> 2 seconds)
      - alert: HighResponseTime
        expr: probe_duration_seconds > 2
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "High response time for {{ $labels.instance }}"
          description: "Response time is {{ $value | printf \"%.2f\" }} seconds (threshold: 2s)"

      # Alert: HTTP status code is not 200
      - alert: HTTPStatusError
        expr: probe_http_status_code != 200
        for: 2m
        labels:
          severity: warning
        annotations:
          summary: "HTTP error on {{ $labels.instance }}"
          description: "HTTP status code is {{ $value }} instead of 200"
```

```python
# app_exporter.py
"""
Custom Prometheus Exporter — Application Metrics

This exporter exposes custom business metrics for our application.
Run it as a standalone service and point Prometheus at it.
"""

from prometheus_client import start_http_server, Gauge, Counter, Histogram, Info
import time
import random

# ──────────────────────────────────────────────
# Define your metrics
# ──────────────────────────────────────────────

# Gauge — a value that can go up and down
# Good for: current values, temperatures, queue sizes
ACTIVE_USERS = Gauge(
    'app_active_users',                    # Metric name (must be unique)
    'Number of currently active users',     # Help text (description)
    ['environment']                         # Labels (optional)
)

QUEUE_SIZE = Gauge(
    'app_queue_size',
    'Current number of items in the processing queue'
)

# Counter — a value that only goes up (resets on restart)
# Good for: total requests, total errors, total bytes processed
REQUESTS_TOTAL = Counter(
    'app_requests_total',
    'Total number of requests received',
    ['method', 'endpoint', 'status']        # Labels for filtering
)

ERRORS_TOTAL = Counter(
    'app_errors_total',
    'Total number of errors',
    ['error_type']
)

# Histogram — measures the distribution of values (e.g., request latency)
# Good for: response times, request sizes
REQUEST_LATENCY = Histogram(
    'app_request_latency_seconds',
    'Request latency in seconds',
    ['endpoint'],
    buckets=[0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1.0, 2.5, 5.0, 10.0]
    #        10ms  25ms   50ms 100ms 250ms 500ms  1s   2.5s   5s   10s
)

# Info — key-value metadata (doesn't change often)
APP_INFO = Info(
    'app',
    'Application information'
)

# ──────────────────────────────────────────────
# Simulate updating metrics
# (In real life, these would come from your actual application)
# ──────────────────────────────────────────────

def simulate_metrics():
    """Simulate changing metrics — replace this with real data collection."""

    # Set app info (this usually only changes on restart/deploy)
    APP_INFO.info({
        'version': '2.1.0',
        'language': 'python',
        'environment': 'production'
    })

    while True:
        # Simulate active users (gauge — goes up and down)
        ACTIVE_USERS.labels(environment='production').set(random.randint(100, 500))
        ACTIVE_USERS.labels(environment='staging').set(random.randint(5, 30))

        # Simulate queue size
        QUEUE_SIZE.set(random.randint(0, 100))

        # Simulate requests (counter — goes up only)
        REQUESTS_TOTAL.labels(
            method='GET',
            endpoint='/api/users',
            status='200'
        ).inc()  # Increment by 1

        REQUESTS_TOTAL.labels(
            method='POST',
            endpoint='/api/orders',
            status='201'
        ).inc()

        # Simulate occasional errors
        if random.random() < 0.1:  # 10% chance
            ERRORS_TOTAL.labels(error_type='timeout').inc()
        if random.random() < 0.05:  # 5% chance
            ERRORS_TOTAL.labels(error_type='database').inc()

        # Simulate request latency (histogram)
        REQUEST_LATENCY.labels(endpoint='/api/users').observe(
            random.uniform(0.01, 0.5)  # Random latency between 10ms and 500ms
        )
        REQUEST_LATENCY.labels(endpoint='/api/orders').observe(
            random.uniform(0.05, 2.0)  # Orders are slower: 50ms to 2s
        )

        # Wait before next update
        time.sleep(5)


# ──────────────────────────────────────────────
# Start the exporter
# ──────────────────────────────────────────────

if __name__ == '__main__':
    # Start the HTTP server on port 8000
    # Metrics will be available at http://localhost:8000/metrics
    port = 8000
    print(f"Starting exporter on port {port}...")
    print(f"Metrics available at http://localhost:{port}/metrics")
    start_http_server(port)

    # Start simulating metrics
    simulate_metrics()
```


#### Example 2: API Health Check Exporter

```python
# api_health_exporter.py
"""
Custom exporter that monitors the health of external APIs.
Checks response time, status codes, and availability.
"""

from prometheus_client import start_http_server, Gauge, Histogram
import requests
import time
import threading

# ──────────────────────────────────────────────
# Define metrics
# ──────────────────────────────────────────────

API_UP = Gauge(
    'api_up',
    'Whether the API endpoint is reachable (1=up, 0=down)',
    ['api_name', 'url']
)

API_RESPONSE_TIME = Histogram(
    'api_response_time_seconds',
    'API response time in seconds',
    ['api_name'],
    buckets=[0.05, 0.1, 0.25, 0.5, 1.0, 2.5, 5.0]
)

API_STATUS_CODE = Gauge(
    'api_status_code',
    'HTTP status code returned by the API',
    ['api_name', 'url']
)

# ──────────────────────────────────────────────
# APIs to monitor
# ──────────────────────────────────────────────

APIS = [
    {"name": "jsonplaceholder", "url": "https://jsonplaceholder.typicode.com/posts/1"},
    {"name": "httpbin", "url": "https://httpbin.org/get"},
    {"name": "github_api", "url": "https://api.github.com"},
]

# ──────────────────────────────────────────────
# Check APIs
# ──────────────────────────────────────────────

def check_apis():
    """Periodically check all APIs and update metrics."""
    while True:
        for api in APIS:
            name = api["name"]
            url = api["url"]
            try:
                response = requests.get(url, timeout=10)

                # API is up
                API_UP.labels(api_name=name, url=url).set(1)

                # Record response time
                API_RESPONSE_TIME.labels(api_name=name).observe(
                    response.elapsed.total_seconds()
                )

                # Record status code
                API_STATUS_CODE.labels(api_name=name, url=url).set(
                    response.status_code
                )

            except requests.exceptions.RequestException:
                # API is down
                API_UP.labels(api_name=name, url=url).set(0)
                API_STATUS_CODE.labels(api_name=name, url=url).set(0)

        time.sleep(30)  # Check every 30 seconds


if __name__ == '__main__':
    port = 8001
    print(f"Starting API Health Exporter on port {port}...")
    start_http_server(port)

    # Run API checks in a background thread
    checker = threading.Thread(target=check_apis, daemon=True)
    checker.start()

    # Keep the main thread alive
    while True:
        time.sleep(1)
```

### Option 2: Custom Exporter in Go

Go is the language Prometheus itself is written in, so there's excellent library support. Go exporters are also faster and compile to a single binary.

```go
// main.go
// Custom Prometheus Exporter in Go

package main

import (
	"log"
	"math/rand"
	"net/http"
	"time"

	"github.com/prometheus/client_golang/prometheus"
	"github.com/prometheus/client_golang/prometheus/promhttp"
)

// ──────────────────────────────────────────────
// Define metrics
// ──────────────────────────────────────────────

var (
	// Gauge — current active users
	activeUsers = prometheus.NewGaugeVec(
		prometheus.GaugeOpts{
			Name: "app_active_users",
			Help: "Number of currently active users",
		},
		[]string{"environment"}, // Labels
	)

	// Counter — total requests
	requestsTotal = prometheus.NewCounterVec(
		prometheus.CounterOpts{
			Name: "app_requests_total",
			Help: "Total number of requests",
		},
		[]string{"method", "endpoint", "status"},
	)

	// Histogram — request latency
	requestLatency = prometheus.NewHistogramVec(
		prometheus.HistogramOpts{
			Name:    "app_request_latency_seconds",
			Help:    "Request latency in seconds",
			Buckets: []float64{0.01, 0.05, 0.1, 0.25, 0.5, 1.0, 2.5, 5.0},
		},
		[]string{"endpoint"},
	)
)

// ──────────────────────────────────────────────
// Register metrics with Prometheus
// ──────────────────────────────────────────────

func init() {
	prometheus.MustRegister(activeUsers)
	prometheus.MustRegister(requestsTotal)
	prometheus.MustRegister(requestLatency)
}

// ──────────────────────────────────────────────
// Simulate metrics (replace with real data)
// ──────────────────────────────────────────────

func simulateMetrics() {
	for {
		// Update active users
		activeUsers.WithLabelValues("production").Set(float64(rand.Intn(400) + 100))
		activeUsers.WithLabelValues("staging").Set(float64(rand.Intn(30) + 5))

		// Increment request counter
		requestsTotal.WithLabelValues("GET", "/api/users", "200").Inc()
		requestsTotal.WithLabelValues("POST", "/api/orders", "201").Inc()

		// Record latency
		requestLatency.WithLabelValues("/api/users").Observe(rand.Float64() * 0.5)
		requestLatency.WithLabelValues("/api/orders").Observe(rand.Float64() * 2.0)

		time.Sleep(5 * time.Second)
	}
}

// ──────────────────────────────────────────────
// Main — start the HTTP server
// ──────────────────────────────────────────────

func main() {
	// Start simulating metrics in background
	go simulateMetrics()

	// Expose metrics endpoint
	http.Handle("/metrics", promhttp.Handler())

	port := ":8000"
	log.Printf("Starting exporter on port %s...\n", port)
	log.Printf("Metrics at http://localhost%s/metrics\n", port)
	log.Fatal(http.ListenAndServe(port, nil))
}
```

Build and run:

```bash

go mod tidy

# Run
go run main.go

# Or build a binary
go build -o custom-exporter
./custom-exporter
```

### Instrumenting Your Existing Application

Instead of building a separate exporter, you can add Prometheus metrics directly to your application code. This is called **instrumentation**.

#### Python (Flask Example)

```python
from flask import Flask, request
from prometheus_client import make_wsgi_app, Counter, Histogram
from werkzeug.middleware.dispatcher import DispatcherMiddleware
import time

app = Flask(__name__)

# Define metrics
REQUEST_COUNT = Counter(
    'http_requests_total',
    'Total HTTP requests',
    ['method', 'endpoint', 'status']
)

REQUEST_LATENCY = Histogram(
    'http_request_duration_seconds',
    'HTTP request latency',
    ['endpoint']
)

# Middleware to track all requests
@app.before_request
def before_request():
    request.start_time = time.time()

@app.after_request
def after_request(response):
    latency = time.time() - request.start_time
    REQUEST_COUNT.labels(
        method=request.method,
        endpoint=request.path,
        status=response.status_code
    ).inc()
    REQUEST_LATENCY.labels(endpoint=request.path).observe(latency)
    return response

# Your normal routes
@app.route('/')
def hello():
    return 'Hello, World!'

@app.route('/api/users')
def users():
    return '{"users": []}'

# Add /metrics endpoint
app.wsgi_app = DispatcherMiddleware(app.wsgi_app, {
    '/metrics': make_wsgi_app()
})

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=5000)
```

Now your Flask app serves metrics at `http://localhost:5000/metrics` alongside your normal API endpoints!

#### Node.js (Express Example)

```javascript

const client = require('prom-client');

const app = express();

// Create a Registry
const register = new client.Registry();

// Add default metrics (process CPU, memory, event loop lag, etc.)
client.collectDefaultMetrics({ register });

// Custom metrics
const httpRequestsTotal = new client.Counter({
  name: 'http_requests_total',
  help: 'Total HTTP requests',
  labelNames: ['method', 'route', 'status'],
  registers: [register],
});

const httpRequestDuration = new client.Histogram({
  name: 'http_request_duration_seconds',
  help: 'HTTP request duration in seconds',
  labelNames: ['method', 'route'],
  buckets: [0.01, 0.05, 0.1, 0.5, 1, 5],
  registers: [register],
});

// Middleware to track requests
app.use((req, res, next) => {
  const end = httpRequestDuration.startTimer({ method: req.method, route: req.path });
  res.on('finish', () => {
    httpRequestsTotal.inc({ method: req.method, route: req.path, status: res.statusCode });
    end();
  });
  next();
});

// Your routes
app.get('/', (req, res) => {
  res.send('Hello World!');
});

// Metrics endpoint
app.get('/metrics', async (req, res) => {
  res.set('Content-Type', register.contentType);
  res.end(await register.metrics());
});

app.listen(3000, () => {
  console.log('App listening on port 3000');
  console.log('Metrics at http://localhost:3000/metrics');
});
```

## 8. Exporter Best Practices

### Naming Conventions

```
# ✅ Good metric names
node_cpu_seconds_total              # snake_case
http_requests_total                 # include unit (total = counter)
http_request_duration_seconds       # include unit (seconds)
node_memory_MemAvailable_bytes      # include unit (bytes)

# ❌ Bad metric names
CPUUsage                           # No camelCase
http_requests                       # Missing _total for counters
response_time                       # Missing unit
memory_usage_megabytes              # Use base units (bytes, not megabytes)
```

### Metric Design Rules

1. **Use base units** — seconds (not milliseconds), bytes (not megabytes)
2. **Counters must end with `_total`** — `http_requests_total`
3. **Include the unit in the name** — `_seconds`, `_bytes`, `_total`
4. **Use labels for dimensions** — Don't create separate metrics for each endpoint
5. **Avoid high cardinality** — Don't use user IDs, request IDs, or timestamps as labels

```promql
# ✅ Good — uses labels
http_requests_total{method="GET", endpoint="/api/users", status="200"}
http_requests_total{method="POST", endpoint="/api/orders", status="201"}

# ❌ Bad — separate metrics for each endpoint
http_requests_api_users_get_200_total
http_requests_api_orders_post_201_total
```

### Security

- **Don't expose sensitive data** in metrics (passwords, tokens, PII)
- **Use read-only mounts** (`ro`) when mounting host filesystems
- **Don't run exporters as root** unless absolutely necessary
- **Restrict access** to exporter ports (firewall rules)
- **Use TLS** for exporters in production if they cross network boundaries

### Performance

- **Keep scrape intervals reasonable** — 15s-30s is typical. Don't go below 5s.
- **Avoid expensive computations** in the metrics endpoint — pre-calculate if needed
- **Limit the number of time series** — Each unique combination of metric name + labels = one time series
- **Use recording rules** in Prometheus for expensive queries instead of computing them at query time

### High Cardinality Warning

**Cardinality** = the number of unique time series. High cardinality kills Prometheus performance.

```yaml
# ⚠️ DANGER — high cardinality example
http_requests_total{user_id="12345", request_id="abc-def-ghi", ...}
# If you have 1 million users × 10 endpoints × 5 methods = 50 million time series!

# ✅ SAFE — low cardinality
http_requests_total{method="GET", endpoint="/api/users", status="200"}
# Maybe 5 methods × 20 endpoints × 5 status codes = 500 time series. That's fine.
```

**Rule of thumb**: If a label value is unique per request/user/transaction → **DON'T use it as a label**.

## 9. Common Interview Questions

**Q: What is a Prometheus exporter?**
> An exporter is a service that translates metrics from a system that doesn't natively support Prometheus into the Prometheus exposition format. It exposes metrics on an HTTP endpoint that Prometheus can scrape.

**Q: What's the difference between Node Exporter and cAdvisor?**
> Node Exporter monitors the host system (CPU, memory, disk, network at the OS level). cAdvisor monitors individual containers (per-container CPU, memory, network). You need both for complete visibility.

**Q: What is the Blackbox Exporter used for?**
> Blackbox Exporter probes endpoints from the outside — like a user would. It checks HTTP endpoints, TCP ports, ICMP ping, and DNS. It's essential for monitoring external-facing services, SSL certificate expiry, and endpoint availability.

**Q: How does the Blackbox Exporter's relabel config work?**
> Prometheus uses relabel_configs to pass the target URL as a query parameter to the Blackbox Exporter. The exporter then probes that URL and returns the results. This is called the "proxy exporter" pattern.

**Q: What's the difference between `container_memory_usage_bytes` and `container_memory_working_set_bytes`?**
> `usage_bytes` includes the page cache (which can be reclaimed by the OS), so it appears higher than actual usage. `working_set_bytes` is the actual memory that cannot be freed — this is what Kubernetes uses for OOM kill decisions.

**Q: How do you build a custom Prometheus exporter?**
> Use the Prometheus client libraries (available in Go, Python, Java, etc.). Define metrics (Counter, Gauge, Histogram), update them with real data, and expose them on an HTTP `/metrics` endpoint.

**Q: What is high cardinality and why is it dangerous?**
> Cardinality is the number of unique time series. High cardinality (millions of unique label combinations) causes Prometheus to use excessive memory and CPU. Avoid using user IDs, request IDs, or other unbounded values as labels.

**Q: How does the Node Exporter textfile collector work?**
> You write metrics in Prometheus format to a `.prom` file in a designated directory. Node Exporter reads these files and exposes their contents alongside its own metrics. It's useful for exposing custom metrics (like backup timestamps) without building a separate exporter.

1. **Exporters are translators** — they convert system metrics into Prometheus format

5. **Custom exporters** let you monitor anything — use Python for quick prototypes, Go for production
6. **Watch out for high cardinality** — don't use unbounded values as labels
7. **Use the textfile collector** for quick custom metrics without building a full exporter
