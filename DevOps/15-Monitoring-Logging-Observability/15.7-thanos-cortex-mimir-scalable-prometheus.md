# 15.7 — Thanos / Cortex / Mimir — Scalable Prometheus

> **One Line Answer:** Thanos, Cortex, and Mimir are tools that solve Prometheus's biggest weakness — **they make Prometheus scale horizontally, store metrics long-term, and work across multiple clusters** — so you can monitor massive infrastructure without hitting Prometheus's limits.

---

## 🤔 Wait, What's Wrong With Prometheus?

Before we jump into Thanos, Cortex, and Mimir, let's understand **why** we even need them.

Prometheus is **amazing**. It's the gold standard for metrics monitoring. But it has some **fundamental limitations:**

| Problem | What Happens |
|---------|-------------|
| **Single Node** | Prometheus runs as a single server. It can't scale horizontally. If you have millions of metrics, one Prometheus can choke. |
| **Local Storage Only** | Prometheus stores data on local disk. If the disk fills up or the server dies, your metrics are **gone**. |
| **No Long-Term Storage** | By default, Prometheus keeps data for ~15 days. Want to see what happened 6 months ago? Too bad. |
| **No Global View** | If you have 10 Kubernetes clusters, each with its own Prometheus, you can't easily query across all of them from one place. |
| **No High Availability (HA)** | If Prometheus goes down, you have a gap in your data. You can run two instances, but they'll have slightly different data and there's no deduplication. |

### The Bottom Line:

```
Prometheus alone = Great for small-to-medium setups
Prometheus + Thanos/Cortex/Mimir = Great for LARGE, multi-cluster, enterprise setups
```

---

## 🏗️ The Three Solutions — Overview

All three tools solve the same core problems, but they do it differently:

| Feature | Thanos | Cortex | Mimir |
|---------|--------|--------|-------|
| **Created By** | Improbable (open-source community) | Weaveworks + community | Grafana Labs |
| **Approach** | Sidecar added to existing Prometheus | Remote write from Prometheus | Remote write from Prometheus |
| **Storage Backend** | Object Storage (S3, GCS, Azure Blob) | Object Storage + optional NoSQL | Object Storage (S3, GCS, etc.) |
| **Architecture** | Several components, pluggable | Microservices-based | Microservices-based (fork of Cortex) |
| **Multi-Tenancy** | Basic | Full built-in | Full built-in |
| **Complexity** | Medium | High | Medium (simplified Cortex) |
| **Community** | CNCF Incubating | CNCF project | Grafana Labs (open-source, AGPLv3) |
| **Best For** | Teams already running Prometheus who want to extend it | Multi-tenant SaaS/platform teams | Teams wanting the best of Cortex with less complexity |

> **Fun Fact:** Mimir is basically Cortex 2.0. Grafana Labs forked Cortex, simplified it, and made it faster. So if you're starting fresh, **Mimir is usually the recommended choice** today.

## 🔷 1. Thanos — Extend Your Existing Prometheus

### What is Thanos?

Thanos is a set of components that **wraps around** your existing Prometheus setup to give it:
- ✅ Long-term storage (unlimited retention)
- ✅ Global query view (query across multiple Prometheus instances)
- ✅ High availability (deduplication of HA Prometheus pairs)
- ✅ Downsampling (compress old data to save storage)

The name "Thanos" is inspired by the Marvel character — the idea is it makes Prometheus **infinitely powerful** 😄

### How Thanos Works — Architecture

```
┌─────────────────────────────────────────────────────────────────┐
│                        Thanos Architecture                       │
│                                                                   │
│  Cluster A                    Cluster B                          │
│  ┌──────────────┐            ┌──────────────┐                   │
│  │  Prometheus   │            │  Prometheus   │                   │
│  │  + Thanos     │            │  + Thanos     │                   │
│  │    Sidecar    │            │    Sidecar    │                   │
│  └──────┬───────┘            └──────┬───────┘                   │
│         │                           │                             │
│         │  Upload TSDB blocks       │                             │
│         ▼                           ▼                             │
│  ┌─────────────────────────────────────────┐                    │
│  │         Object Storage (S3/GCS)          │  ← Long-term       │
│  └─────────────────────────────────────────┘    storage          │
│         │                                                         │
│         ▼                                                         │
│  ┌──────────────┐     ┌──────────────────┐                      │
│  │  Thanos Store │     │  Thanos Compactor │ ← Downsamples &     │
│  │  Gateway      │     │                    │   compacts old data │
│  └──────┬───────┘     └──────────────────┘                      │
│         │                                                         │
│         ▼                                                         │
│  ┌──────────────┐                                                │
│  │ Thanos Query  │  ← Global query endpoint (speaks PromQL)     │
│  │ (Querier)     │                                                │
│  └──────┬───────┘                                                │
│         │                                                         │
│         ▼                                                         │
│  ┌──────────────┐                                                │
│  │   Grafana     │  ← Dashboards (uses Thanos Query as source)  │
│  └──────────────┘                                                │
└─────────────────────────────────────────────────────────────────┘
```

### Thanos Components Explained

| Component | What It Does | Think of It As... |
|-----------|-------------|-------------------|
| **Sidecar** | Sits next to Prometheus. Uploads metric blocks to object storage. Also serves real-time data to the Querier. | A helper that lives with Prometheus and sends data to the cloud |
| **Store Gateway** | Reads old metric data from object storage and serves it to the Querier | A librarian that fetches historical data from the archive |
| **Query (Querier)** | The single endpoint you query. It fans out queries to Sidecars + Store Gateways and merges results. Speaks PromQL! | The "front desk" — ask it anything, it finds the answer from all sources |
| **Compactor** | Runs in the background. Compacts and downsamples old data in object storage to save space | A janitor that cleans up and compresses old data |
| **Ruler** | Evaluates recording rules and alerting rules against the global view | Like Prometheus's rule evaluator, but for the global view |
| **Query Frontend** | Optional caching layer in front of the Querier. Splits large queries and caches results | A smart receptionist that speeds up repeated questions |

### Setting Up Thanos Sidecar with Prometheus

Here's how you'd deploy Prometheus with the Thanos Sidecar using Docker Compose:

```yaml
# docker-compose.yml
version: "3.8"

services:
  prometheus:
    image: prom/prometheus:latest
    volumes:
      - prometheus-data:/prometheus
      - ./prometheus.yml:/etc/prometheus/prometheus.yml
    command:
      - '--config.file=/etc/prometheus/prometheus.yml'
      - '--storage.tsdb.path=/prometheus'
      - '--storage.tsdb.min-block-duration=2h'   # Important for Thanos!
      - '--storage.tsdb.max-block-duration=2h'   # Important for Thanos!
      - '--web.enable-lifecycle'                  # Allows hot reload
    ports:
      - "9090:9090"

  thanos-sidecar:
    image: thanosio/thanos:latest
    command:
      - 'sidecar'
      - '--tsdb.path=/prometheus'                      # Same path as Prometheus data
      - '--prometheus.url=http://prometheus:9090'       # Prometheus URL
      - '--objstore.config-file=/etc/thanos/bucket.yml' # Object storage config
      - '--grpc-address=0.0.0.0:10901'                 # gRPC endpoint for Querier
    volumes:
      - prometheus-data:/prometheus
      - ./bucket.yml:/etc/thanos/bucket.yml

  thanos-query:
    image: thanosio/thanos:latest
    command:
      - 'query'
      - '--http-address=0.0.0.0:9091'
      - '--store=thanos-sidecar:10901'          # Connect to sidecar
      - '--store=thanos-store:10901'            # Connect to store gateway
    ports:
      - "9091:9091"    # This is your new "Prometheus-like" endpoint

  thanos-store:
    image: thanosio/thanos:latest
    command:
      - 'store'
      - '--objstore.config-file=/etc/thanos/bucket.yml'
      - '--grpc-address=0.0.0.0:10901'
    volumes:
      - ./bucket.yml:/etc/thanos/bucket.yml

  thanos-compactor:
    image: thanosio/thanos:latest
    command:
      - 'compact'
      - '--objstore.config-file=/etc/thanos/bucket.yml'
      - '--data-dir=/tmp/thanos-compact'
      - '--wait'       # Run continuously, not just once
    volumes:
      - ./bucket.yml:/etc/thanos/bucket.yml

volumes:
  prometheus-data:
```

**What's happening here:**
- `prometheus` runs normally but with special TSDB block settings (`min-block-duration=2h`, `max-block-duration=2h`). This ensures Prometheus creates clean 2-hour blocks that Thanos can upload.
- `thanos-sidecar` watches Prometheus's data directory and uploads completed blocks to object storage (like S3).
- `thanos-query` is the endpoint you point Grafana to. It queries both live data (via sidecar) and historical data (via store gateway).
- `thanos-store` reads from object storage and serves old metrics to the query layer.
- `thanos-compactor` runs in the background, compacting and downsampling old data.

### Object Storage Configuration (bucket.yml)

For AWS S3:

```yaml
# bucket.yml
type: S3
config:
  bucket: "my-thanos-metrics"
  endpoint: "s3.us-east-1.amazonaws.com"
  region: "us-east-1"
  access_key: "${AWS_ACCESS_KEY_ID}"      # Use environment variables!
  secret_key: "${AWS_SECRET_ACCESS_KEY}"  # Never hardcode secrets!
```

For Google Cloud Storage:

```yaml
# bucket.yml
type: GCS
config:
  bucket: "my-thanos-metrics"
```

For Azure Blob Storage:

```yaml
# bucket.yml
type: AZURE
config:
  storage_account: "mythanosstorageaccount"
  storage_account_key: "${AZURE_STORAGE_KEY}"
  container: "thanos-metrics"
```

### Thanos in Kubernetes (using Helm)

In a real production setup, you'd deploy Thanos on Kubernetes:

```bash
# Add the Bitnami Helm repo
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo update

# Install Thanos
helm install thanos bitnami/thanos \
  --namespace monitoring \
  --create-namespace \
  --set objstoreConfig="$(cat bucket.yml)" \
  --set query.stores[0]=prometheus-thanos-sidecar:10901 \
  --set compactor.enabled=true \
  --set storegateway.enabled=true
```

And configure the kube-prometheus-stack to include the Thanos sidecar:

```yaml
# values.yaml for kube-prometheus-stack Helm chart
prometheus:
  prometheusSpec:
    thanos:
      enabled: true
      image: thanosio/thanos:v0.34.0
      objectStorageConfig:
        secret:
          type: S3
          config:
            bucket: my-thanos-metrics
            endpoint: s3.us-east-1.amazonaws.com
            region: us-east-1
    # Important! Set retention to match block duration
    retention: 4h
    # These are critical for Thanos sidecar to work
    disableCompaction: true
```

### Querying Across Clusters with Thanos

Once you have Thanos running, you query it exactly like Prometheus — using PromQL!

```promql
# Get CPU usage across ALL clusters
sum(rate(node_cpu_seconds_total{mode!="idle"}[5m])) by (cluster)

# Get memory usage from a specific cluster
node_memory_MemAvailable_bytes{cluster="production-us-east"}

# Compare request rates between clusters
sum(rate(http_requests_total[5m])) by (cluster, service)
```

The magic is that you write the same PromQL you already know, but the **Thanos Querier** automatically fans out to all your Prometheus instances and object storage to get the data.

### Thanos Downsampling

Thanos automatically downsamples old data to save storage:

| Data Age | Resolution | What It Means |
|----------|-----------|---------------|
| 0 - 2 weeks | Raw (original) | Full detail, every scrape point |
| 2 weeks - 6 months | 5-minute averages | Aggregated to 5-minute windows |
| 6+ months | 1-hour averages | Aggregated to 1-hour windows |

This means you can store **years of data** without it costing a fortune!

## 🔶 2. Cortex — Multi-Tenant Prometheus at Scale

### What is Cortex?

Cortex takes a **different approach** than Thanos. Instead of sitting next to Prometheus as a sidecar, Cortex acts as a **remote write destination**. 

Prometheus sends all its metrics to Cortex, and Cortex handles:
- ✅ Long-term storage
- ✅ Horizontal scaling
- ✅ Multi-tenancy (different teams/customers, completely isolated)
- ✅ High availability
- ✅ Global query view

### How Cortex Works — Architecture

```
┌────────────────────────────────────────────────────────────────┐
│                       Cortex Architecture                       │
│                                                                  │
│  ┌────────────┐  ┌────────────┐  ┌────────────┐               │
│  │ Prometheus  │  │ Prometheus  │  │ Prometheus  │               │
│  │ (Cluster A) │  │ (Cluster B) │  │ (Cluster C) │               │
│  └─────┬──────┘  └─────┬──────┘  └─────┬──────┘               │
│        │               │               │                         │
│        │  remote_write  │  remote_write  │  remote_write         │
│        ▼               ▼               ▼                         │
│  ┌──────────────────────────────────────────┐                   │
│  │              Distributor                  │ ← Receives &      │
│  │  (validates, rate-limits, shards data)    │   distributes     │
│  └─────────────────┬────────────────────────┘   samples         │
│                    │                                              │
│                    ▼                                              │
│  ┌──────────────────────────────────────────┐                   │
│  │               Ingester                    │ ← Holds recent    │
│  │  (in-memory buffer, WAL, batches writes)  │   data in memory │
│  └─────────────────┬────────────────────────┘                   │
│                    │                                              │
│                    ▼                                              │
│  ┌──────────────────────────────────────────┐                   │
│  │         Object Storage (S3/GCS)           │ ← Long-term      │
│  │  (stores TSDB blocks)                     │   storage        │
│  └──────────────────────────────────────────┘                   │
│                    │                                              │
│                    ▼                                              │
│  ┌──────────────────────────────────────────┐                   │
│  │          Query Frontend + Querier         │ ← Handles PromQL │
│  └──────────────────────────────────────────┘   queries         │
│                    │                                              │
│                    ▼                                              │
│  ┌──────────────────────────────────────────┐                   │
│  │               Grafana                     │ ← Dashboards     │
│  └──────────────────────────────────────────┘                   │
└────────────────────────────────────────────────────────────────┘
```

### Cortex Components Explained

| Component | What It Does |
|-----------|-------------|
| **Distributor** | First stop for incoming metrics. Validates data, enforces rate limits, and distributes samples to ingesters using consistent hashing. |
| **Ingester** | Holds recent data in memory (last few hours). Periodically flushes data as TSDB blocks to object storage. Has a WAL (Write-Ahead Log) for crash recovery. |
| **Querier** | Handles PromQL queries. Fetches data from both ingesters (recent data) and object storage (old data). |
| **Query Frontend** | Sits in front of the Querier. Splits large queries into smaller ones, caches results, and retries failed queries. |
| **Compactor** | Compacts and downsamples blocks in object storage. |
| **Ruler** | Evaluates recording rules and alerting rules. |
| **Store Gateway** | Reads block metadata from object storage for efficient querying. |

### Configuring Prometheus to Remote Write to Cortex

This is the key configuration — tell Prometheus to send metrics to Cortex:

```yaml
# prometheus.yml
global:
  scrape_interval: 15s

remote_write:
  - url: http://cortex-distributor:8080/api/v1/push
    headers:
      X-Scope-OrgID: "team-backend"   # This is the tenant ID!

scrape_configs:
  - job_name: 'my-app'
    static_configs:
      - targets: ['my-app:8080']
```

**What's happening here:**
- `remote_write` tells Prometheus: "Hey, besides storing data locally, also send everything to Cortex."
- `X-Scope-OrgID` is the **tenant ID**. This is how Cortex isolates data between different teams or customers. Team A can't see Team B's metrics.
- Cortex receives the data at the `/api/v1/push` endpoint.

### Multi-Tenancy in Cortex

This is one of Cortex's killer features. Imagine you're a platform team serving multiple product teams:

```
Team Backend  → X-Scope-OrgID: "team-backend"  → Only sees backend metrics
Team Frontend → X-Scope-OrgID: "team-frontend" → Only sees frontend metrics
Team Data     → X-Scope-OrgID: "team-data"     → Only sees data pipeline metrics
```

Each tenant has:
- **Isolated data** — can't see other tenants' metrics
- **Independent rate limits** — one tenant can't overwhelm the system
- **Separate retention policies** — different teams can keep data for different durations

### Deploying Cortex on Kubernetes

```bash
# Add Cortex Helm repo
helm repo add cortex-helm https://cortexproject.github.io/cortex-helm-chart
helm repo update

# Install Cortex
helm install cortex cortex-helm/cortex \
  --namespace monitoring \
  --create-namespace \
  -f cortex-values.yaml
```

A simplified `cortex-values.yaml`:

```yaml
# cortex-values.yaml
config:
  storage:
    engine: blocks
  blocks_storage:
    backend: s3
    s3:
      bucket_name: cortex-metrics
      endpoint: s3.us-east-1.amazonaws.com
      region: us-east-1
    tsdb:
      dir: /data/tsdb
  
  limits:
    ingestion_rate: 100000       # Max samples per second per tenant
    max_series_per_user: 5000000 # Max active time series per tenant
    max_label_names_per_series: 30

  distributor:
    ring:
      kvstore:
        store: consul    # or etcd, memberlist
  
  ingester:
    ring:
      kvstore:
        store: consul
      replication_factor: 3   # Each sample is stored 3 times for HA
```

**What this config is saying:**
- Use **S3** for long-term block storage
- Limit each tenant to 100,000 samples/second (so one tenant can't kill the system)
- Limit each tenant to 5 million active time series
- Use **Consul** for service discovery (so ingesters and distributors can find each other)
- Replicate data **3 times** across ingesters for high availability

## 🟣 3. Mimir — The Best of Cortex, Simplified

### What is Mimir?

Mimir is Grafana Labs' answer to "Cortex is great but too complex." They **forked Cortex**, simplified the architecture, improved performance dramatically, and released it as **Grafana Mimir**.

Think of it this way:

```
Cortex = Pioneer, proved it could be done
Mimir  = Cortex 2.0, faster, simpler, better maintained
```

### Why Choose Mimir Over Cortex?

| Aspect | Cortex | Mimir |
|--------|--------|-------|
| **Performance** | Good | 40x faster queries (according to Grafana Labs benchmarks) |
| **Complexity** | Many config options, multiple storage engines | Simplified — blocks storage only |
| **Maintenance** | Community-driven, slower updates | Backed by Grafana Labs, frequent releases |
| **Max Scale** | Billions of active series | 1 billion+ active series tested |
| **Split-and-merge Compactor** | No | Yes — allows horizontal scaling of compaction |
| **Shuffle Sharding** | Basic | Advanced — better tenant isolation |
| **Query Scheduler** | No | Yes — better query load distribution |

### Mimir Architecture

Mimir has the same core components as Cortex, but with improvements:

```

│  Prometheus instances (remote_write)                            │

│  │ Distributor  │ ← Validates, rate-limits, shards              │
│  └──────┬──────┘                                                │
│         │                                                        │
│         ▼                                                        │
│  ┌─────────────┐                                                │
```

### Deploying Mimir — Quick Start with Docker

```yaml
# docker-compose.yml for Mimir
version: "3.8"

services:
  mimir:
    image: grafana/mimir:latest
    command:
      - '-config.file=/etc/mimir/mimir.yaml'
      - '-target=all'                    # Run all components in one process (monolithic mode)
    volumes:
      - ./mimir.yaml:/etc/mimir/mimir.yaml
      - mimir-data:/data
    ports:
      - "9009:9009"    # HTTP API
      - "9095:9095"    # gRPC

volumes:
  mimir-data:
```

The Mimir configuration file:

```yaml
# mimir.yaml
multitenancy_enabled: false   # Set to true for multi-tenant mode

# Where to store blocks
blocks_storage:
  backend: filesystem        # Use S3/GCS in production!
  filesystem:
    dir: /data/blocks
  tsdb:
    dir: /data/tsdb

# Compactor configuration
compactor:
  data_dir: /data/compactor
  sharding_ring:
    kvstore:
      store: memberlist      # Use memberlist for peer discovery

# Distributor config
distributor:
  ring:
    kvstore:
      store: memberlist

# Ingester config
ingester:
  ring:
    kvstore:
      store: memberlist
    replication_factor: 1    # Set to 3 in production!

# Store gateway config
store_gateway:
  sharding_ring:
    kvstore:
      store: memberlist

# Limits for tenants
limits:
  max_global_series_per_user: 1500000
  ingestion_rate: 50000
  ingestion_burst_size: 100000

# Server config  
server:
  http_listen_port: 9009
  grpc_listen_port: 9095

# Use memberlist for all component discovery
memberlist:
  join_members:
    - mimir:7946
```

**What this config does:**
- `multitenancy_enabled: false` — keeps it simple for single-tenant setups. Turn it on if you need isolation between teams.
- `backend: filesystem` — stores on local disk for dev/testing. Use `s3` or `gcs` in production.
- `memberlist` — components discover each other using gossip protocol. No need for Consul or etcd!
- `replication_factor: 1` — fine for testing, use `3` in production so data survives node failures.
- `limits` — protects the system from being overwhelmed.

### Configuring Prometheus to Write to Mimir

Just like Cortex, add `remote_write` to your Prometheus config:

```yaml
# prometheus.yml
remote_write:
  - url: http://mimir:9009/api/v1/push
    headers:
      X-Scope-OrgID: "my-team"    # Required if multitenancy is enabled
```

That's it! Prometheus sends metrics to Mimir. Mimir stores them long-term.

## ⚖️ Thanos vs Cortex vs Mimir — When to Use What?

This is the big question. Here's a decision guide:

### Choose Thanos If:

```
✅ You already have Prometheus running and don't want to change much
✅ You want a sidecar approach (Prometheus keeps its own storage + uploads to object storage)
✅ You don't need multi-tenancy (or only basic multi-tenancy)
✅ You want a CNCF-backed, vendor-neutral solution
✅ Your team is comfortable with a component-based architecture
```

### Choose Mimir If:

```
✅ You're starting fresh or willing to use remote_write
✅ You want the simplest setup for a powerful system
✅ You need multi-tenancy (multiple teams, SaaS platform)
✅ You're already in the Grafana ecosystem (Loki, Tempo, Grafana)
✅ You want the fastest query performance
✅ You want active maintenance and frequent releases
```

### Choose Cortex If:

```
✅ You have an existing Cortex deployment (don't break what works!)
✅ You need specific Cortex features not yet in Mimir
❌ For new projects, Mimir is generally recommended over Cortex
```

### Quick Decision Flowchart

```
Do you already have Prometheus running?
├── YES → Do you want to keep Prometheus as-is?
│         ├── YES → Use THANOS (sidecar approach)
│         └── NO  → Use MIMIR (remote_write approach)
└── NO  → Starting fresh?
          └── YES → Use MIMIR (best overall choice for new setups)

Do you need multi-tenancy?
├── YES → Use MIMIR (or Cortex if you already have it)
└── NO  → Thanos or Mimir both work great
```

## 💡 Production Tips & Best Practices

### 1. Object Storage is Cheap — Use It

```
S3 Standard: ~$0.023/GB/month
S3 Infrequent Access: ~$0.0125/GB/month
S3 Glacier: ~$0.004/GB/month

→ Storing 1 year of metrics for a medium company = ~$50-200/month
→ Way cheaper than running massive Prometheus servers with huge SSDs!
```

### 2. Start Simple, Scale Later

```
Day 1:   Monolithic mode (all components in one process)
Week 4:  Read/Write path separation  
Month 3: Full microservices mode (if needed)
```

Don't start with a full distributed deployment unless you actually need it. Monolithic mode is fine for up to ~20 million active series!

### 3. Set Proper Limits

Always configure limits to protect your system:

```yaml
# Mimir/Cortex limits
limits:
  ingestion_rate: 100000              # Don't let one tenant flood you
  max_global_series_per_user: 5000000 # Cap cardinality
  max_label_names_per_series: 30      # Prevent label explosion
  max_query_lookback: 30d             # Don't let users query years of data in one shot
  max_query_parallelism: 16           # Limit concurrent query threads
```

### 4. Monitor Your Monitoring

Yes, you need to monitor Thanos/Mimir/Cortex themselves! Use a separate, small Prometheus instance to scrape metrics from these components.

```
Ironic? Yes. Necessary? Absolutely.
```

## 📋 Summary

| Aspect | Thanos | Cortex | Mimir |
|--------|--------|--------|-------|
| **Approach** | Sidecar on Prometheus | Remote write target | Remote write target |
| **Complexity** | Medium | High | Medium |
| **Multi-tenancy** | Basic | Full | Full |
| **Performance** | Good | Good | Excellent |
| **Backed by** | CNCF Community | CNCF Community | Grafana Labs |
| **Long-term storage** | ✅ Object Storage | ✅ Object Storage | ✅ Object Storage |
| **Global query** | ✅ | ✅ | ✅ |
| **HA / Dedup** | ✅ | ✅ | ✅ |
| **Best for new projects** | ✅ If keeping Prometheus as-is | ❌ Use Mimir instead | ✅ Recommended |

> **The Bottom Line:** If you're starting fresh, go with **Mimir**. If you already have Prometheus and want minimal changes, go with **Thanos**. Cortex is the grandparent — respect it, but new projects should look at Mimir.
