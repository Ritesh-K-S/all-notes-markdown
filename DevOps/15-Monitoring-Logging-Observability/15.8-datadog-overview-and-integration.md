## 🤔 What is Datadog?

Imagine you have servers, containers, Kubernetes clusters, databases, and applications running across AWS, Azure, and GCP. You need to monitor ALL of it. You could set up Prometheus + Grafana + ELK + Jaeger + AlertManager... or you could use **Datadog** and get all of it in one platform.

**Datadog is a SaaS (Software as a Service) monitoring platform** that does:

| Capability | What Datadog Calls It |
|-----------|----------------------|
| Metrics & Monitoring | Infrastructure Monitoring |
| Application Performance | APM (Application Performance Monitoring) |
| Logs | Log Management |
| Distributed Tracing | APM Traces |
| Synthetic Monitoring | Synthetics (test your app from different locations) |
| Real User Monitoring | RUM (track real user experience in browsers/mobile) |
| Security Monitoring | Cloud SIEM |
| Network Monitoring | Network Performance Monitoring (NPM) |
| CI/CD Pipeline Visibility | CI Visibility |
| Database Monitoring | Database Monitoring (query-level insights) |
| Error Tracking | Error Tracking |

**All in one platform. One login. One dashboard. One bill.**

## 🆚 Datadog vs Open-Source Stack (Prometheus + Grafana)

Before we go deeper, let's understand when to use each:

| Aspect | Datadog | Prometheus + Grafana |
|--------|---------|---------------------|
| **Type** | SaaS (managed service) | Self-hosted (you run everything) |
| **Setup Time** | Minutes (install agent, done) | Hours/Days (set up Prometheus, Grafana, Alertmanager, storage...) |
| **Cost** | $15-23/host/month + per feature | Free software, but you pay for servers & your team's time |
| **Maintenance** | Zero (Datadog handles everything) | You maintain it. Updates, scaling, storage, backups... |
| **Scalability** | Automatic (it's SaaS) | You scale it yourself (Thanos/Mimir needed for big setups) |
| **Customization** | Good, but within their platform | Unlimited — it's open source |
| **Vendor Lock-in** | Yes (your data is in their platform) | No (you own everything) |
| **Features** | All-in-one (metrics, logs, traces, APM, RUM, etc.) | Need to set up each component separately |
| **Best For** | Teams that want to focus on code, not monitoring infra | Teams that want full control and don't mind managing infrastructure |

### The Bottom Line:

```
Small/Medium teams with budget → Datadog (saves time, everything works out of the box)
Large teams / cost-conscious  → Open-source stack (cheaper at scale, full control)
Enterprise / compliance needs → Can go either way (Datadog has SOC2, HIPAA, etc.)
```

### The Key Components:

| Component | What It Is |
|-----------|-----------|
| **Datadog Agent** | A lightweight process that runs on your servers/containers. Collects metrics, logs, and traces and sends them to Datadog's cloud. |
| **Integrations** | Pre-built connectors for 750+ technologies (AWS, Kubernetes, PostgreSQL, Redis, Nginx, etc.). Each integration knows what metrics to collect. |
| **Datadog Cloud** | The SaaS backend. Stores your data, processes it, runs alerts, and serves dashboards. |
| **Web UI** | Where you create dashboards, set up alerts, investigate issues, and explore data. |

## 🚀 Getting Started with Datadog

### Step 1: Create a Datadog Account

Go to [datadoghq.com](https://www.datadoghq.com/) and sign up. You get a 14-day free trial with all features.

Once signed up, you'll get an **API Key** — this is what connects your agent to your Datadog account.

### Step 2: Install the Datadog Agent

The agent is how data gets from your infrastructure to Datadog. Here's how to install it on different platforms:

#### On Linux (Ubuntu/Debian):

```bash
# One-line install script (replace DD_API_KEY with your actual key)
DD_API_KEY=<YOUR_API_KEY> DD_SITE="datadoghq.com" bash -c \
  "$(curl -L https://install.datadoghq.com/scripts/install_script_agent7.sh)"
```

**What this does:**
- Downloads the Datadog Agent package
- Installs it as a system service
- Configures it with your API key
- Starts collecting system metrics (CPU, memory, disk, network) immediately

#### On Linux (CentOS/RHEL):

```bash
DD_API_KEY=<YOUR_API_KEY> DD_SITE="datadoghq.com" bash -c \
  "$(curl -L https://install.datadoghq.com/scripts/install_script_agent7.sh)"
```

Same command — the script auto-detects your OS!

#### On Docker:

```bash
docker run -d \
  --name datadog-agent \
  -e DD_API_KEY=<YOUR_API_KEY> \
  -e DD_SITE="datadoghq.com" \
  -e DD_LOGS_ENABLED=true \
  -e DD_APM_ENABLED=true \
  -v /var/run/docker.sock:/var/run/docker.sock:ro \
  -v /proc/:/host/proc/:ro \
  -v /sys/fs/cgroup/:/host/sys/fs/cgroup:ro \
  gcr.io/datadoghq/agent:7
```

**What each flag means:**
- `-e DD_API_KEY` — Your API key to authenticate with Datadog
- `-e DD_LOGS_ENABLED=true` — Also collect logs (not just metrics)
- `-e DD_APM_ENABLED=true` — Enable application tracing
- `-v /var/run/docker.sock` — Lets the agent discover other Docker containers
- `-v /proc/` and `-v /sys/fs/cgroup/` — Lets the agent read system metrics

#### On Kubernetes (DaemonSet via Helm — Recommended):

This is the most common setup in modern DevOps:

```bash
# Add the Datadog Helm repo
helm repo add datadog https://helm.datadoghq.com
helm repo update

# Install the Datadog Agent
helm install datadog-agent datadog/datadog \
  --namespace monitoring \
  --create-namespace \
  --set datadog.apiKey=<YOUR_API_KEY> \
  --set datadog.site="datadoghq.com" \
  --set datadog.logs.enabled=true \
  --set datadog.apm.portEnabled=true \
  --set datadog.processAgent.enabled=true \
  --set datadog.networkMonitoring.enabled=true
```

Or better, use a `values.yaml` file:

```yaml
# datadog-values.yaml
datadog:
  apiKey: <YOUR_API_KEY>        # Or use a Kubernetes secret (recommended!)
  site: "datadoghq.com"
  
  # Enable all the good stuff
  logs:
    enabled: true
    containerCollectAll: true    # Collect logs from ALL containers
  
  apm:
    portEnabled: true            # Enable APM trace collection
  
  processAgent:
    enabled: true                # See running processes
    processCollection: true
  
  networkMonitoring:
    enabled: true                # Network flow monitoring
  
  # Kubernetes-specific features
  clusterAgent:
    enabled: true                # Deploy the Cluster Agent
    metricsProvider:
      enabled: true              # Enable HPA based on Datadog metrics!
  
  # Collect Kubernetes events
  kubeStateMetricsCore:
    enabled: true

# Using a secret for the API key (best practice!)
# First create the secret:
# kubectl create secret generic datadog-api-key --from-literal=api-key=<YOUR_KEY> -n monitoring
```

**Why use a Cluster Agent?**
- Regular agents run on every node (DaemonSet)
- The Cluster Agent runs as a single deployment and handles cluster-level data
- It reduces the load on the Kubernetes API server (instead of every agent querying it, only the Cluster Agent does)
- It enables features like HPA with custom Datadog metrics

## 🔌 Datadog Integrations

Integrations are what make Datadog powerful. Instead of you figuring out "what metrics should I collect from PostgreSQL?", Datadog already knows. You just enable the integration.

### How Integrations Work

```
You have PostgreSQL running
       ↓
You enable the PostgreSQL integration in Datadog
       ↓
The agent automatically starts collecting:
  - Query throughput
  - Connection counts
  - Cache hit ratio
  - Replication lag
  - Lock stats
  - And 50+ other metrics
       ↓
Datadog automatically creates a PostgreSQL dashboard for you!
```

### Example: Setting Up the Nginx Integration

1. **Enable Nginx stub_status** (the agent needs an endpoint to scrape):

```nginx
# In your nginx.conf
server {
    listen 80;
    
    location /nginx_status {
        stub_status on;
        allow 127.0.0.1;      # Only allow local access
        deny all;
    }
}
```

2. **Configure the Datadog integration:**

```yaml
# /etc/datadog-agent/conf.d/nginx.d/conf.yaml
init_config:

instances:
  - nginx_status_url: http://localhost/nginx_status
    tags:
      - env:production
      - service:web-frontend
```

3. **Restart the agent:**

```bash
sudo systemctl restart datadog-agent
```

That's it! Within minutes, you'll see Nginx metrics flowing into Datadog with a beautiful pre-built dashboard.

### Example: Setting Up the PostgreSQL Integration

```yaml
# /etc/datadog-agent/conf.d/postgres.d/conf.yaml
init_config:

instances:
  - host: localhost
    port: 5432
    username: datadog            # Create a dedicated monitoring user!
    password: <PASSWORD>         # Use secrets management in production
    dbname: my_database
    tags:
      - env:production
      - service:user-service
    
    # Enable query-level metrics
    collect_activity_metrics: true
    collect_database_size_metrics: true
```

First, create the monitoring user in PostgreSQL:

```sql
-- Run this in PostgreSQL
CREATE USER datadog WITH PASSWORD 'your_secure_password';
GRANT pg_monitor TO datadog;
GRANT SELECT ON pg_stat_database TO datadog;
```

### Example: Kubernetes Integration (Auto-Discovery)

In Kubernetes, Datadog uses **Autodiscovery** — it automatically detects containers and applies the right integration config based on annotations:

```yaml
# Your deployment with Datadog annotations
apiVersion: apps/v1
kind: Deployment
metadata:
  name: redis
spec:
  template:
    metadata:
      annotations:
        # Tell Datadog: "This is a Redis container, monitor it!"
        ad.datadoghq.com/redis.checks: |
          {
            "redisdb": {
              "instances": [
                {
                  "host": "%%host%%",
                  "port": "6379"
                }
              ]
            }
          }
        # Collect logs from this container
        ad.datadoghq.com/redis.logs: |
          [
            {
              "source": "redis",
              "service": "redis-cache"
            }
          ]
    spec:
      containers:
        - name: redis
          image: redis:7
          ports:
            - containerPort: 6379
```

**What's happening here:**
- The annotations tell the Datadog Agent: "This container is Redis. Use the `redisdb` integration."
- `%%host%%` is a template variable that Datadog auto-replaces with the container's IP
- The `logs` annotation tells the agent to collect Redis logs and tag them with `source:redis`
- **No config files needed on the agent side** — it's all annotation-driven!

## 🔔 Datadog Alerting (Monitors)

Monitors are how Datadog alerts you when something goes wrong.

### Types of Monitors

| Monitor Type | Use Case |
|-------------|----------|
| **Metric Monitor** | Alert when a metric crosses a threshold (e.g., CPU > 90%) |
| **Log Monitor** | Alert on log patterns (e.g., "ERROR" appears more than 10 times in 5 min) |
| **APM Monitor** | Alert on service performance (e.g., p99 latency > 500ms) |
| **Composite Monitor** | Combine multiple monitors with AND/OR logic |
| **Anomaly Monitor** | Alert when a metric deviates from its normal pattern (uses ML!) |
| **Forecast Monitor** | Alert when a metric is predicted to cross a threshold (e.g., disk will be full in 3 days) |
| **SLO Alert** | Alert when your error budget is being consumed too fast |
| **Process Monitor** | Alert when a specific process is not running |
| **Network Monitor** | Alert on network connectivity issues |

### Creating a Metric Monitor

In the Datadog UI, or via API:

```bash
# Create a monitor via API (using curl)
curl -X POST "https://api.datadoghq.com/api/v1/monitor" \
  -H "Content-Type: application/json" \
  -H "DD-API-KEY: <YOUR_API_KEY>" \
  -H "DD-APPLICATION-KEY: <YOUR_APP_KEY>" \
  -d '{
    "name": "High CPU Usage on Web Servers",
    "type": "metric alert",
    "query": "avg(last_5m):avg:system.cpu.user{service:web-api} by {host} > 90",
    "message": "CPU usage is above 90% on {{host.name}}.\n\nPlease investigate.\n\n@slack-devops-alerts @pagerduty-web-team",
    "tags": ["env:production", "team:backend"],
    "options": {
      "thresholds": {
        "critical": 90,
        "warning": 75
      },
      "notify_no_data": true,
      "no_data_timeframe": 10,
      "renotify_interval": 60,
      "escalation_message": "CPU is STILL high! This needs immediate attention. @pagerduty-web-oncall"
    }
  }'
```

**What this monitor does:**
- Watches the `system.cpu.user` metric for hosts tagged `service:web-api`
- If the 5-minute average exceeds **75%** → sends a **Warning**
- If the 5-minute average exceeds **90%** → sends a **Critical** alert
- Sends notifications to Slack (`@slack-devops-alerts`) and PagerDuty (`@pagerduty-web-team`)
- If no data arrives for 10 minutes → alerts (something might be down!)
- Re-notifies every 60 minutes if the issue persists

### Monitor as Code (Terraform)

```hcl

}
```

## 📈 Datadog APM (Application Performance Monitoring)

APM lets you trace requests as they flow through your application — from the frontend through microservices to the database.

### How APM Works

```
User Request → API Gateway → User Service → Database
                                ↓
                           Payment Service → Stripe API
```

APM captures this entire flow as a **trace**, showing:
- How long each step took
- Where the bottleneck is
- Which service is slow
- What errors occurred

### Instrumenting Your Application

You need to add a tracing library to your application code:

#### Python (with `ddtrace`)

```bash
# Install the tracing library
pip install ddtrace
```

```python
# Option 1: Auto-instrumentation (no code changes!)
# Just run your app with ddtrace-run:
# ddtrace-run python app.py

# Option 2: Manual instrumentation
from ddtrace import tracer

# The tracer auto-instruments popular libraries:
# - Flask, Django, FastAPI
# - requests, urllib3
# - psycopg2, pymongo, redis
# - celery, kombu

# Custom spans for your own code
@tracer.wrap(service="my-app", resource="process_order")
def process_order(order_id):
    # Your code here
    validate_order(order_id)
    charge_payment(order_id)
    send_confirmation(order_id)

# Or use context manager
def my_function():
    with tracer.trace("custom.operation", service="my-app") as span:
        span.set_tag("order_id", "12345")
        # Your code here
        result = do_something()
        span.set_tag("result_status", "success")
    return result
```

#### Node.js (with `dd-trace`)

```bash
# Install the tracing library
npm install dd-trace
```

```javascript
// Add this as the FIRST line of your application entry point
// (before any other imports!)
const tracer = require('dd-trace').init({
  service: 'my-node-app',
  env: 'production',
  version: '1.0.0',
  logInjection: true,    // Correlate logs with traces!
});

// That's it! dd-trace auto-instruments:
// - Express, Fastify, Koa
// - http/https
// - pg, mysql, mongodb, redis
// - amqplib (RabbitMQ)
// - And many more!

// For custom spans:
const express = require('express');
const app = express();

app.get('/api/orders/:id', (req, res) => {
  const span = tracer.startSpan('process.order');
  span.setTag('order.id', req.params.id);
  
  try {
    const order = getOrder(req.params.id);
    span.setTag('order.status', order.status);
    res.json(order);
  } catch (error) {
    span.setTag('error', true);
    span.setTag('error.message', error.message);
    res.status(500).json({ error: error.message });
  } finally {
    span.finish();
  }
});
```

#### Java (with `dd-java-agent`)

```bash
# Download the Java agent
curl -Lo dd-java-agent.jar https://dtdg.co/latest-java-tracer

# Run your app with the agent attached
```

No code changes needed! The Java agent auto-instruments:
- Spring Boot, Spring MVC
- JDBC (PostgreSQL, MySQL, Oracle)
- Hibernate, JPA
- OkHttp, Apache HttpClient
- Kafka, RabbitMQ
- And many more

#### In Kubernetes (passing APM config via environment variables):

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  template:

            - name: DD_LOGS_INJECTION
              value: "true"     # Inject trace IDs into logs!
```

## 📝 Datadog Log Management

### Collecting Logs

The Datadog Agent can collect logs from files, containers, and other sources.

#### From Files:

```yaml
# /etc/datadog-agent/conf.d/my-app.d/conf.yaml
logs:
  - type: file
    path: /var/log/my-app/*.log
    service: my-app
    source: python
    tags:
      - env:production
```

#### From Docker Containers:

```yaml
# In datadog-agent docker-compose or Helm values
# Just set DD_LOGS_ENABLED=true and DD_LOGS_CONFIG_CONTAINER_COLLECT_ALL=true
# The agent will auto-collect logs from all containers!
```

#### From Kubernetes (Autodiscovery):

```yaml
# Pod annotation to collect logs
metadata:
  annotations:
    ad.datadoghq.com/my-container.logs: |
      [
        {
          "source": "python",
          "service": "my-app",
          "log_processing_rules": [
            {
              "type": "multi_line",
              "name": "python_traceback",
              "pattern": "Traceback \\(most recent call last\\)"
            }
          ]
        }
      ]
```

### Log Processing Pipelines

Datadog can parse and enrich your logs automatically:

```
Raw log → Parsing (extract fields) → Enrichment (add tags) → Indexing → Alerting
```

Example: If your app logs JSON:

```json
{"timestamp": "2025-01-15T10:30:00Z", "level": "ERROR", "service": "payment", "message": "Payment failed", "order_id": "ORD-123", "error": "Card declined"}
```

Datadog automatically extracts `level`, `service`, `message`, `order_id`, and `error` as searchable fields. You can then:

```
# Search logs in Datadog
service:payment level:ERROR @order_id:ORD-123

# Count errors per service in the last hour
# Use facets and measures to build log analytics
```

### Log → Trace Correlation

One of Datadog's killer features is connecting logs to traces:

```
You see an error log → Click it → See the exact trace/request that caused it
You see a slow trace  → Click it → See all the logs generated during that request
```

This works because the tracer injects trace IDs into your logs. In Python:

```python
import logging
from ddtrace import tracer

# ddtrace automatically injects dd.trace_id and dd.span_id into log records
# when DD_LOGS_INJECTION=true

logging.basicConfig(
    format='%(asctime)s %(levelname)s [dd.trace_id=%(dd.trace_id)s dd.span_id=%(dd.span_id)s] %(message)s'
)

logger = logging.getLogger(__name__)

def process_order(order_id):
    logger.info(f"Processing order {order_id}")
    # This log will automatically include the trace ID!
    # In Datadog, you can click the log and jump to the trace
```

## 💰 Datadog Pricing — Understanding the Costs

This is important because Datadog can get expensive fast if you're not careful!

### Pricing Model (as of 2025 — check their website for current pricing):

| Feature | Price | What Counts |
|---------|-------|-------------|
| **Infrastructure** | ~$15/host/month (Pro) | Per host (server, VM, node) |
| **APM** | ~$31/host/month | Per host running traced services |
| **Log Management** | ~$0.10/GB ingested + $1.70/million events indexed | Per GB of logs |
| **Synthetics** | ~$5/10K test runs | Per synthetic test execution |
| **RUM** | ~$1.50/1K sessions | Per user session |
| **Network Monitoring** | ~$5/host/month | Per host |

### Cost Optimization Tips

```
1. Use log filtering → Don't index DEBUG logs in production!
   
   # In your pipeline, exclude debug logs:
   Exclusion filter: @level:DEBUG
   
   Savings: 50-70% of log costs (most logs are DEBUG/INFO)

2. Use metrics instead of logs where possible
   → A counter metric costs almost nothing
   → A log line costs ~$0.10/GB
   
3. Use log archives → Send logs to S3 for cheap long-term storage
   → Only index what you need to search frequently

4. Tag everything → Proper tags help you track costs by team/service
   
5. Set up usage monitors → Alert when a team exceeds their budget
```

### Cost Control with Terraform:

```hcl
# Alert when log usage is high
resource "datadog_monitor" "log_usage" {
  name    = "Log Ingestion Rate Too High"
  type    = "metric alert"
  message = "Log ingestion is above normal. Check if DEBUG logging was accidentally enabled. @slack-finops"
  
  query = "avg(last_1h):sum:datadog.estimated_usage.logs.ingested_bytes{*}.as_rate() > 10000000000"
  
  monitor_thresholds {
    critical = 10000000000   # 10 GB/hour
    warning  = 5000000000    # 5 GB/hour
  }
}
```

## 🏷️ Unified Service Tagging

This is a **best practice** that makes Datadog 10x more useful. Tag everything with three standard tags:

| Tag | Purpose | Example |
|-----|---------|---------|
| `env` | Which environment | `production`, `staging`, `dev` |
| `service` | Which service/app | `web-api`, `payment-service`, `auth` |
| `version` | Which version | `1.2.3`, `2.0.0-beta` |

When these three tags are consistent across metrics, logs, and traces, you can:

```
"Show me all errors for service:payment-service in env:production for version:2.1.0"

→ Metrics show spike in error rate
→ Logs show the exact error messages  
→ Traces show which requests failed and why
→ All correlated by service + env + version
```

### How to Set It:

```yaml
# Kubernetes labels (automatically picked up by the agent)
metadata:
  labels:
    tags.datadoghq.com/env: "production"
    tags.datadoghq.com/service: "payment-service"
    tags.datadoghq.com/version: "2.1.0"
```

## 📋 Summary

| Aspect | Details |
|--------|---------|
| **What** | Fully managed SaaS monitoring platform |
| **Core Features** | Metrics, Logs, APM/Traces, Dashboards, Alerts |
| **Agent** | Lightweight process on every host/container |
| **Integrations** | 750+ pre-built (AWS, K8s, databases, etc.) |
