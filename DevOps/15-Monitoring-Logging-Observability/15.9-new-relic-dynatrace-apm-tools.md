## 🤔 What Are APM Tools?

Before we dive into New Relic and Dynatrace, let's quickly understand what APM means.

**APM = Application Performance Monitoring**

It's the practice of tracking how your application performs in production:

```
Traditional Monitoring: "Is the server up? Is CPU high?"
APM:                    "Why is the checkout page taking 3 seconds? 
                         Which database query is slow? 
                         Which microservice is the bottleneck?
                         How is the real user experience?"
```

### What APM Tools Track

| What | Why It Matters |
|------|---------------|
| **Response Time** | How fast your app responds to requests |
| **Throughput** | How many requests per second your app handles |
| **Error Rate** | What percentage of requests fail |
| **Dependencies** | Which external services (databases, APIs, caches) your app talks to and how fast they respond |
| **Code-Level Visibility** | Which function/method is slow, which SQL query is expensive |
| **Distributed Traces** | Following a request as it flows through multiple microservices |
| **Real User Experience** | How fast pages load for actual users in their browsers |

### Step 2: Install the New Relic Agent

#### For a Node.js Application:

```bash
# Install the New Relic agent
npm install newrelic
```

```javascript
// IMPORTANT: This MUST be the very first line in your app!
require('newrelic');

// Now the rest of your app
const express = require('express');
const app = express();

app.get('/api/users', async (req, res) => {
  const users = await db.query('SELECT * FROM users');
  res.json(users);
});

app.listen(3000);
```

Create the config file:

```javascript
// newrelic.js (in your project root)
'use strict';

exports.config = {
  app_name: ['my-web-app'],            // Your app name in New Relic
  license_key: process.env.NEW_RELIC_LICENSE_KEY,  // Use env var!
  distributed_tracing: {
    enabled: true                       // Enable distributed tracing
  },
  logging: {
    level: 'info',
    filepath: 'stdout'                  // Log to stdout (good for containers)
  },
  allow_all_headers: true,
  attributes: {
    exclude: [
      'request.headers.cookie',
      'request.headers.authorization',  // Don't send sensitive headers!
      'request.headers.proxyAuthorization',
      'request.headers.setCookie*',
      'request.headers.x*',
      'response.headers.cookie',
      'response.headers.authorization',
      'response.headers.proxyAuthorization',
      'response.headers.setCookie*',
      'response.headers.x*'
    ]
  }
};
```

**What's happening:**
- `require('newrelic')` must be the first line — it hooks into Node.js internals to automatically instrument Express, HTTP calls, database queries, etc.
- `distributed_tracing.enabled: true` — traces requests across microservices
- The `attributes.exclude` section **prevents sensitive data** (cookies, auth headers) from being sent to New Relic. This is a security best practice!

## 📊 NRQL — New Relic Query Language

NRQL is what makes New Relic really powerful. It's like SQL but for your observability data. If you know SQL, you'll pick this up in minutes.

### Basic NRQL Queries

```sql
-- How many transactions in the last hour?
SELECT count(*) FROM Transaction SINCE 1 hour ago

-- Average response time per app
SELECT average(duration) FROM Transaction FACET appName SINCE 1 hour ago

-- P95 response time for a specific app
SELECT percentile(duration, 95) FROM Transaction 
WHERE appName = 'my-web-app' SINCE 1 hour ago

-- Error rate
SELECT percentage(count(*), WHERE error IS true) AS 'Error Rate' 
FROM Transaction SINCE 1 hour ago

-- Top 10 slowest transactions
SELECT average(duration) FROM Transaction 
FACET name SINCE 1 hour ago LIMIT 10

-- Throughput over time (timeseries chart)
SELECT rate(count(*), 1 minute) FROM Transaction 
SINCE 1 hour ago TIMESERIES

-- Compare this week to last week
SELECT average(duration) FROM Transaction 
SINCE 1 week ago COMPARE WITH 1 week ago

-- Find specific errors
SELECT * FROM TransactionError 
WHERE error.message LIKE '%timeout%' SINCE 1 day ago

-- Infrastructure: CPU by host
SELECT average(cpuPercent) FROM SystemSample 
FACET hostname SINCE 1 hour ago TIMESERIES

-- Kubernetes: Pod restarts
SELECT sum(restartCount) FROM K8sContainerSample 
FACET podName SINCE 1 hour ago

-- Logs: Count errors per service
SELECT count(*) FROM Log 
WHERE level = 'ERROR' FACET service SINCE 1 hour ago
```

**What makes NRQL powerful:**
- `FACET` = GROUP BY in SQL (group results by a field)
- `TIMESERIES` = Show data over time (great for charts)
- `COMPARE WITH` = Compare current data with a previous time period
- `SINCE/UNTIL` = Time range
- `WHERE` = Filter data
- Works across ALL data types (metrics, logs, traces, events)

### Custom Events

You can send your own custom data to New Relic:

```python
# Python example: Send custom events
import requests
import json

headers = {
    'Content-Type': 'application/json',
    'Api-Key': 'YOUR_INSERT_API_KEY',   # Insert key, not license key
}

events = [{
    'eventType': 'DeploymentEvent',     # Your custom event type
    'appName': 'my-web-app',
    'version': '2.1.0',
    'deployer': 'jenkins',
    'environment': 'production',
    'commitHash': 'abc123',
    'duration': 45                       # Deploy took 45 seconds
}]

response = requests.post(
    'https://insights-collector.newrelic.com/v1/accounts/YOUR_ACCOUNT_ID/events',
    headers=headers,
    data=json.dumps(events)
)
```

Now query it:

```sql
-- Show all deployments
SELECT * FROM DeploymentEvent SINCE 1 week ago

-- Average deploy duration over time
SELECT average(duration) FROM DeploymentEvent 
FACET appName SINCE 1 month ago TIMESERIES 1 day

-- Correlate deploys with error spikes
SELECT count(*) FROM TransactionError 
SINCE 1 day ago TIMESERIES 5 minutes
```

## 🔵 Dynatrace — Overview

### What is Dynatrace?

Dynatrace is an **AI-powered, full-stack observability platform** that's known for:
1. **OneAgent** — a single agent that instruments everything automatically (app, infrastructure, network, containers)
2. **Davis AI** — an AI engine that automatically detects problems and finds root causes
3. **PurePath** — code-level distributed tracing technology

If New Relic's strength is its query language (NRQL) and generous free tier, Dynatrace's strength is its **AI and automation** — it tries to do the thinking for you.

### History (Quick)

```
2005 — Dynatrace founded in Linz, Austria
2011 — Acquired by Compuware
2014 — "Dynatrace Ruxit" launched (cloud-native rewrite)
2015 — Spun off as independent company
2019 — IPO on NYSE (ticker: DT)
2023 — Revenue $1.4 billion (big enterprise player)
```

### Dynatrace Key Features

| Feature | What It Does | What Makes It Special |
|---------|-------------|----------------------|
| **OneAgent** | Single agent monitors everything on the host | Auto-discovers and instruments apps, containers, services — zero config! |
| **Davis AI** | AI engine for problem detection | Doesn't just alert on thresholds — understands causality and finds root cause |
| **PurePath** | Code-level distributed tracing | Captures every transaction end-to-end, code-level detail |
| **Smartscape** | Visual topology map | Real-time map showing how everything connects |
| **Real User Monitoring** | Browser/mobile monitoring | JavaScript injection, session replay |
| **Synthetic Monitoring** | Automated testing | Click-path monitoring from global locations |
| **Cloud Automation** | SLO-based release validation | Auto-validate deployments based on SLOs |
| **Security** | Runtime vulnerability detection | Detects vulnerabilities in running code (not just scanning) |

### Dynatrace Pricing

Dynatrace uses a **DPS (Dynatrace Platform Subscription)** model:

```
DPS = A pool of credits consumed by different capabilities

You buy a block of DPS units, and they're consumed based on usage:
- Full-stack monitoring: X DPS per host/hour
- Application security: Y DPS per host/hour  
- Log monitoring: Z DPS per GB
- etc.

Typical pricing: Enterprise-level ($$$$)
- Generally more expensive than New Relic and Datadog
- But includes more automation/AI out of the box
- Usually starts at ~$69-74/host/month for full-stack
```

> **Key Difference:** Dynatrace is typically the most expensive of the three (Datadog, New Relic, Dynatrace) but also requires the least manual configuration because of its AI-driven approach.

## 🤖 Davis AI — Automatic Problem Detection

This is Dynatrace's crown jewel. Davis AI doesn't just alert on thresholds — it uses **causal AI** to:

1. **Detect anomalies** — Learns what's "normal" and alerts when things deviate
2. **Find root cause** — Traces the problem back to the exact component that caused it
3. **Correlate events** — Groups related alerts into a single "problem" instead of bombarding you with 100 separate alerts

### How Davis AI Works

```
Traditional Monitoring:
  CPU > 90% → Alert!
  Memory > 85% → Alert!
  Response time > 2s → Alert!
  Error rate > 5% → Alert!
  → You get 4 separate alerts. Which one is the real problem? 🤷

Davis AI:
  "I detected a problem affecting service 'checkout-api'.
   Root cause: The PostgreSQL database on host-3 has high query latency
   due to a missing index on the 'orders' table.
   
   Impact:
   - checkout-api response time increased from 200ms to 3.5s
   - Error rate increased from 0.1% to 8.2%
   - 1,247 users affected in the last 15 minutes
   
   This correlates with deployment 'v2.3.1' deployed 20 minutes ago."
   
   → You get ONE problem card with the root cause and impact. 🎯
```

### Davis AI — Under the Hood

```
Davis uses three types of AI:

1. Deterministic AI
   └── Knows the topology (which service calls which)
   └── Follows the dependency chain to find root cause
   └── Example: If Service A depends on Service B, and B is slow,
       Davis knows A's slowness is CAUSED by B

2. Statistical AI  
   └── Learns baselines (what's normal for each metric)
   └── Detects anomalies (deviations from normal)
   └── Adapts to time-of-day, day-of-week patterns
   └── Example: 90% CPU at 2 AM is abnormal; 90% CPU during Black Friday is normal

3. Predictive AI
   └── Predicts future problems (disk full in 3 days)
   └── Forecasts resource needs
```

### Configuring Davis AI

You can tune Davis's sensitivity:

```
Dynatrace UI → Settings → Anomaly Detection

Service-level settings:
├── Response time degradation
│   ├── Auto (Davis learns the baseline)  ← Recommended
│   ├── Fixed threshold (e.g., > 500ms)
│   └── Sensitivity: Low / Medium / High
├── Error rate increase
│   ├── Auto
│   └── Fixed threshold (e.g., > 5%)
├── Traffic spike/drop
│   └── Alert if traffic deviates > X% from normal
└── Custom anomaly detection (using DQL)
```

## 📊 Dynatrace Query Language (DQL)

Dynatrace has its own query language called DQL (Dynatrace Query Language) for exploring data:

```sql
-- Fetch all log records with errors
fetch logs
| filter loglevel == "ERROR"
| sort timestamp desc
| limit 100

-- Get average response time per service
fetch dt.service.request.response_time
| filter dt.entity.service.name == "checkout-api"
| summarize avg = avg(value), p95 = percentile(value, 95)

-- Find slow database queries
fetch dt.service.database.response_time  
| filter value > 1000     -- slower than 1 second
| sort value desc
| limit 20

-- Kubernetes pod CPU usage
fetch dt.kubernetes.container.cpu_usage
| filter k8s.cluster.name == "production"
| summarize avg_cpu = avg(value) by {k8s.pod.name}
| sort avg_cpu desc

-- Count deployments per day
fetch events
| filter event.type == "CUSTOM_DEPLOYMENT"
| summarize count = count() by {bin(timestamp, 1d)}
```

## 🔗 OpenTelemetry — The Vendor-Neutral Future

No matter which tool you pick, you should know about **OpenTelemetry (OTel)**. It's a CNCF project that provides:

- A **standard API** for instrumentation
- **Language SDKs** for metrics, logs, and traces  
- An **OTel Collector** that can export to ANY backend

```
Your App → OpenTelemetry SDK → OTel Collector → New Relic / Dynatrace / Datadog / Jaeger / etc.
```

Why does this matter?

```
Without OpenTelemetry:
  Your code uses dd-trace (Datadog library)
  Want to switch to New Relic? → Re-instrument everything! 😱

With OpenTelemetry:
  Your code uses OpenTelemetry SDK (vendor-neutral)
  Want to switch? → Just change the Collector config! 🎉
```

### Example: OpenTelemetry with Python

```python
# Install OpenTelemetry
# pip install opentelemetry-api opentelemetry-sdk opentelemetry-exporter-otlp

from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter

# Set up the tracer
provider = TracerProvider()

# Export to ANY backend via OTLP protocol
# Just change the endpoint to switch vendors!
exporter = OTLPSpanExporter(
    endpoint="https://otlp.nr-data.net:4317",  # New Relic
    # endpoint="https://YOUR_ENV.live.dynatrace.com/api/v2/otlp",  # Dynatrace
    # endpoint="http://localhost:4317",  # Local OTel Collector
    headers={"api-key": "YOUR_KEY"}
)

provider.add_span_processor(BatchSpanProcessor(exporter))
trace.set_tracer_provider(provider)

# Use the tracer in your code
tracer = trace.get_tracer(__name__)

@app.route('/api/orders')
def get_orders():
    with tracer.start_as_current_span("get-orders") as span:
        span.set_attribute("order.count", len(orders))
        orders = db.query("SELECT * FROM orders")
        return jsonify(orders)
```

### OpenTelemetry Collector Config

```yaml
# otel-collector-config.yaml
receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
      http:
        endpoint: 0.0.0.0:4318

processors:
  batch:
    timeout: 5s
    send_batch_size: 1000

exporters:
  # Send to New Relic
  otlp/newrelic:
    endpoint: https://otlp.nr-data.net:4317
    headers:
      api-key: YOUR_NEW_RELIC_KEY

  # Send to Dynatrace  
  otlphttp/dynatrace:
    endpoint: https://YOUR_ENV.live.dynatrace.com/api/v2/otlp
    headers:
      Authorization: "Api-Token YOUR_DT_TOKEN"

  # Send to Datadog
  datadog:
    api:
      key: YOUR_DD_API_KEY
      site: datadoghq.com

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch]
      exporters: [otlp/newrelic]    # Change this to switch vendors!
    metrics:
      receivers: [otlp]
      processors: [batch]
      exporters: [otlp/newrelic]
    logs:
      receivers: [otlp]
      processors: [batch]
      exporters: [otlp/newrelic]
```

## 🛠️ Real-World Setup: New Relic for a Microservices App

Here's a complete example of monitoring a microservices application:

```yaml
# docker-compose.yml — Microservices with New Relic monitoring
version: "3.8"

services:
  # API Gateway
  api-gateway:
    image: my-api-gateway:latest
    environment:
      - NEW_RELIC_LICENSE_KEY=${NR_LICENSE_KEY}
      - NEW_RELIC_APP_NAME=api-gateway
      - NEW_RELIC_DISTRIBUTED_TRACING_ENABLED=true
      - NEW_RELIC_LOG_LEVEL=info
    ports:
      - "8080:8080"

  # User Service
  user-service:
    image: my-user-service:latest
    environment:
      - NEW_RELIC_LICENSE_KEY=${NR_LICENSE_KEY}
      - NEW_RELIC_APP_NAME=user-service
      - NEW_RELIC_DISTRIBUTED_TRACING_ENABLED=true

  # Order Service
  order-service:
    image: my-order-service:latest
    environment:
      - NEW_RELIC_LICENSE_KEY=${NR_LICENSE_KEY}
      - NEW_RELIC_APP_NAME=order-service
      - NEW_RELIC_DISTRIBUTED_TRACING_ENABLED=true

  # Payment Service
  payment-service:
    image: my-payment-service:latest
    environment:
      - NEW_RELIC_LICENSE_KEY=${NR_LICENSE_KEY}
      - NEW_RELIC_APP_NAME=payment-service
      - NEW_RELIC_DISTRIBUTED_TRACING_ENABLED=true

  # PostgreSQL
  postgres:
    image: postgres:15
    environment:
      - POSTGRES_DB=myapp
      - POSTGRES_USER=app
      - POSTGRES_PASSWORD=${DB_PASSWORD}

  # Redis
  redis:
    image: redis:7

  # New Relic Infrastructure Agent
  newrelic-infra:
    image: newrelic/infrastructure:latest
    environment:
      - NRIA_LICENSE_KEY=${NR_LICENSE_KEY}
      - NRIA_DISPLAY_NAME=docker-host
    volumes:
      - /:/host:ro
      - /var/run/docker.sock:/var/run/docker.sock:ro
    privileged: true
```

Now in New Relic, you'd see:

```
Service Map:
  api-gateway → user-service → postgres
       ↓
  order-service → postgres
       ↓            ↓
  payment-service  redis

Distributed Trace Example:
  [api-gateway] POST /api/orders (total: 450ms)
    └── [order-service] createOrder() (120ms)
         ├── [postgres] INSERT INTO orders... (25ms)
         ├── [user-service] getUser() (45ms)
         │    └── [postgres] SELECT * FROM users... (12ms)
         └── [payment-service] chargeCard() (250ms)
              └── [External] stripe.com/v1/charges (230ms) ← This is slow!
```

From this trace, you can instantly see that the Stripe API call (230ms) is the bottleneck. No guessing needed!

## 💡 Best Practices for APM Tools

### 1. Instrument Everything

```
Don't just monitor your main API. Monitor:
✅ All microservices
✅ Background workers / job processors
✅ Database queries
✅ External API calls
✅ Message queue consumers
✅ Cron jobs
```

### 2. Use Custom Attributes

```python
# Add business context to your traces
from newrelic import agent

@app.route('/api/orders', methods=['POST'])
def create_order():
    agent.add_custom_attribute('customer_tier', customer.tier)     # Gold, Silver, Bronze
    agent.add_custom_attribute('order_value', order.total)         # $150.00
    agent.add_custom_attribute('payment_method', order.payment)    # credit_card, paypal
    agent.add_custom_attribute('region', customer.region)          # us-east, eu-west
    
    # Now you can query:
    # SELECT average(duration) FROM Transaction 
    # WHERE customer_tier = 'Gold' FACET payment_method
```

### 3. Set Up SLOs (Service Level Objectives)

```
Define what "good" looks like:
├── Availability: 99.9% of requests return a non-error response
├── Latency: 95% of requests complete within 500ms
└── Error Rate: Less than 0.1% of requests return 5xx errors

Both New Relic and Dynatrace can track SLOs and alert when your error budget is depleted.
```

### 4. Don't Send Sensitive Data

```python
# BAD — PII being sent to APM tool!
agent.add_custom_attribute('user_email', 'john@example.com')
agent.add_custom_attribute('credit_card', '4111-1111-1111-1111')

# GOOD — Send only non-sensitive identifiers
agent.add_custom_attribute('user_id', 'usr_abc123')
agent.add_custom_attribute('payment_method', 'visa_ending_1111')
```

### 5. Use Deployment Markers

Mark when you deploy so you can correlate performance changes with code changes:

```bash
# New Relic deployment marker
curl -X POST "https://api.newrelic.com/v2/applications/YOUR_APP_ID/deployments.json" \
  -H "Api-Key: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "deployment": {
      "revision": "v2.3.1",
      "user": "deploy-bot",
      "description": "Added new payment method",
      "changelog": "https://github.com/myorg/myapp/releases/tag/v2.3.1"
    }
  }'
```

```bash
# Dynatrace deployment event
curl -X POST "https://YOUR_ENV.live.dynatrace.com/api/v2/events/ingest" \
  -H "Authorization: Api-Token YOUR_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "eventType": "CUSTOM_DEPLOYMENT",
    "title": "Deployed v2.3.1",
    "properties": {
      "version": "v2.3.1",
      "deployer": "jenkins",
      "environment": "production"
    },
    "entitySelector": "type(SERVICE),tag(service:checkout-api)"
  }'
```

## 📋 Summary

| Aspect | New Relic | Dynatrace |
|--------|-----------|-----------|
| **Type** | Full-stack observability platform | AI-powered observability platform |
| **Best Feature** | NRQL query language + free tier | Davis AI + OneAgent auto-discovery |
| **Pricing** | Per GB ingested (free 100 GB/month) | DPS credits (premium pricing) |
| **Setup** | Install agents per app + infra agent | Install OneAgent (auto-discovers everything) |
| **AI** | Good (anomaly detection) | Best-in-class (causal AI, root cause analysis) |
| **Best For** | Cost-conscious teams, query lovers | Enterprise, complex architectures |
| **Vendor Lock-in Risk** | Medium (use OpenTelemetry to reduce) | Medium-High (OneAgent is proprietary) |
| **OpenTelemetry** | Full support | Full support |

> **The Bottom Line:** Both are excellent tools. New Relic gives you more control and a better price entry point. Dynatrace gives you more automation and AI-powered insights. If you want to avoid vendor lock-in, instrument with **OpenTelemetry** and you can switch between any of these (or open-source tools) by just changing the exporter config.
