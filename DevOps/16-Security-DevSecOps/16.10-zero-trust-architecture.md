# 16.10 — Zero Trust Architecture

> **One Line Answer:** Zero Trust means **"never trust, always verify"** — don't automatically trust anyone or anything, whether they're inside or outside your network. Every single request must prove who it is, every single time.

## 🤔 Let's Start With a Story — Why Do We Need Zero Trust?

Imagine your company is like a **castle with a moat**.

In the **old way** (called the "perimeter security" model), security worked like this:

```
Outside the Castle (Internet)  →  🔒 Firewall/VPN (the Moat)  →  Inside the Castle (Corporate Network)
                                                                         ↓
                                                              Everyone inside is TRUSTED ✅
                                                              Can access anything freely
```

So, if someone got past the moat (the firewall/VPN), they were **automatically trusted**. They could roam freely inside.

### The Problem?

```
🎭 Hacker steals an employee's VPN credentials
         ↓
Gets inside the "castle"
         ↓
Now they're TRUSTED — can access databases, servers, secrets...
         ↓
💥 MASSIVE DATA BREACH
```

This is **exactly** how many famous breaches happened — SolarWinds, Colonial Pipeline, and many more.

**Zero Trust was born to fix this.**

---

## ✅ What Exactly is Zero Trust?

Zero Trust is a **security model** (not a product, not a tool) that says:

> **"Never trust, always verify."**

It doesn't matter if you're:
- Inside the corporate network ❌ Still not trusted
- Using a company laptop ❌ Still not trusted
- The CEO ❌ Still not trusted
- Already authenticated 5 minutes ago ❌ Still verify again

**Every request, every time, must be verified.**

### The Key Mindset Shift:

| Old Model (Castle & Moat) | Zero Trust Model |
|---------------------------|-----------------|
| Trust everyone inside the network | Trust NO ONE by default |
| Verify once at the perimeter (firewall/VPN) | Verify EVERY request, EVERY time |
| Flat network — everything can talk to everything | Micro-segmented — strict access controls |
| "Inside = Safe" | "There is no inside or outside" |
| Perimeter-focused | Identity-focused |

## 🏛️ Core Principles of Zero Trust

Zero Trust is built on a few key principles. Let's go through each one:

### 1. Never Trust, Always Verify

Every user, device, and application must authenticate and be authorized **before** getting access to anything.

```
User wants to access a database
     ↓
Who are you? → Verify identity (MFA, SSO)
     ↓
What device are you on? → Is it managed? Patched? Healthy?
     ↓
Where are you? → Expected location? VPN? Unusual IP?
     ↓
What do you want to access? → Do you have permission?
     ↓
✅ Only THEN — grant access (and only the minimum needed)
```

### 2. Least Privilege Access

Give users and systems the **absolute minimum** access they need to do their job. Nothing more.

```
❌ Wrong: "Hey, you're a developer? Here's admin access to ALL servers."

✅ Right: "You're a developer working on the payment service? 
          Here's read-only access to the payment-service repo 
          and dev environment only. Nothing else."
```

### 3. Assume Breach

Always operate as if an attacker is **already inside** your network. Design your systems so that even if one part is compromised, the blast radius is minimized.

```
Attacker compromises Service A
         ↓
Can they reach Service B? → ❌ NO (micro-segmented)
Can they access the database? → ❌ NO (separate credentials, short-lived tokens)
Can they move laterally? → ❌ NO (every hop requires re-authentication)
         ↓
Blast radius = ONLY Service A ✅
```

### 4. Verify Explicitly

Don't just check username/password. Check **everything**:

| What to Verify | Examples |
|----------------|----------|
| **Identity** | Username, MFA, biometrics |
| **Device** | Is the device managed? Is it healthy? OS patched? |
| **Location** | Where is the request coming from? Is it normal? |
| **Behavior** | Is this user doing something unusual? |
| **Data sensitivity** | How sensitive is the data they're accessing? |
| **Time** | Is this request at an unusual time? |

### 5. Micro-Segmentation

Break your network into tiny, isolated segments. Each segment has its own access controls.

```
OLD (Flat Network):
┌─────────────────────────────────────────────┐
│                                             │
│   Web Server ←→ App Server ←→ Database      │
│       ↕            ↕            ↕           │
│   Everything can talk to everything 😱      │
│                                             │
└─────────────────────────────────────────────┘

ZERO TRUST (Micro-Segmented):
┌──────────┐    ┌──────────┐    ┌──────────┐
│ Web Zone │ →  │ App Zone │ →  │ DB Zone  │
│          │    │          │    │          │
│ Only     │    │ Only     │    │ Only     │
│ port 443 │    │ port 8080│    │ port 5432│
│ allowed  │    │ from Web │    │ from App │
└──────────┘    └──────────┘    └──────────┘
     ↑                               ↑
     │     NO DIRECT ACCESS           │
     └────────────── ❌ ──────────────┘
```

## 🏗️ Zero Trust Architecture — The Components

A Zero Trust Architecture (ZTA) has several key building blocks:

### The Architecture Diagram:

```
                    ┌─────────────────────┐
                    │   Policy Engine     │ ← Makes the access decision
                    │  (The "Brain")      │    (Allow or Deny?)
                    └────────┬────────────┘
                             │
                    ┌────────▼────────────┐
                    │ Policy Enforcement  │ ← Enforces the decision
                    │    Point (PEP)      │    (Acts as the gateway)
                    └────────┬────────────┘
                             │
        ┌────────────────────┼────────────────────┐
        │                    │                    │
   ┌────▼─────┐      ┌──────▼──────┐     ┌──────▼──────┐
   │ Identity │      │   Device    │     │  Network    │
   │ Provider │      │   Trust     │     │  Context    │
   │ (Who?)   │      │  (What?)    │     │  (Where?)   │
   └──────────┘      └─────────────┘     └─────────────┘
```

### Component Breakdown:

| Component | What It Does | Examples |
|-----------|-------------|----------|
| **Policy Engine** | The brain — decides if a request should be allowed or denied | Open Policy Agent (OPA), Azure AD Conditional Access |
| **Policy Enforcement Point (PEP)** | The gateway that blocks or allows traffic based on the decision | Proxies, API gateways, Service mesh sidecars |
| **Identity Provider (IdP)** | Verifies who the user/service is | Okta, Azure AD, Google Workspace, Keycloak |
| **Device Trust** | Checks if the device is healthy and managed | Microsoft Intune, Jamf, CrowdStrike |
| **Network Context** | Provides info about where the request is coming from | IP reputation, geo-location, network segment |
| **Data Classification** | Labels data by sensitivity to determine access levels | Public, Internal, Confidential, Restricted |
| **Continuous Monitoring** | Constantly watches for anomalies | SIEM tools, behavior analytics |

## 🔧 Zero Trust in Practice — Real-World Implementation

Let's see how Zero Trust principles actually get implemented in a DevOps environment.

### 1. Identity-Based Access (Instead of Network-Based)

**Old Way:**
```
If user is on the corporate VPN → Allow access to everything
```

**Zero Trust Way:**
```yaml
# Example: Access policy (conceptual)
policy:
  resource: "production-database"
  conditions:
    - identity: "authenticated via SSO + MFA"
    - role: "database-admin"
    - device: "managed, OS patched, antivirus active"
    - location: "expected geography"
    - time: "business hours"
    - risk_score: "low"
  action: "allow read-only access for 1 hour"
```

### 2. Service-to-Service Authentication (mTLS)

In a microservices architecture, services need to verify each other too — not just users.

**Mutual TLS (mTLS)** ensures both sides of a connection verify each other:

```
Service A wants to talk to Service B

OLD WAY:
Service A → Service B  (no verification, just network proximity)

ZERO TRUST WAY (mTLS):
Service A                    Service B
    │                            │
    ├── "Here's MY certificate" ─►│
    │                            │
    │◄─ "Here's MY certificate" ──┤
    │                            │
    ├── Both verify each other ──►│
    │                            │
    ✅ Encrypted & Authenticated  ✅
```

**How to implement mTLS in Kubernetes (using Istio service mesh):**

```yaml
# Istio PeerAuthentication — enforce mTLS for all services
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: production
spec:
  mtls:
    mode: STRICT  # All traffic MUST be mTLS encrypted
```

This tells Istio: "Every service in the `production` namespace **must** use mutual TLS. No exceptions."

### 3. Short-Lived Credentials (No More Permanent Passwords)

Zero Trust says: **never use permanent credentials**. Use short-lived tokens instead.

```
❌ BAD: Database password stored in a config file forever
   Password: "SuperSecret123" → Never rotated → Attacker finds it → Game over

✅ GOOD: Dynamic, short-lived credentials from HashiCorp Vault
   Token valid for 1 hour → Auto-rotates → Attacker finds it → Already expired
```

**Example: Getting short-lived database credentials from Vault:**

```bash
# Vault generates a temporary database credential
# This credential automatically expires after 1 hour

$ vault read database/creds/my-role

Key                Value
---                -----
lease_id           database/creds/my-role/abc123
lease_duration     1h          # ← Only valid for 1 hour!
username           v-token-my-role-xyz789
password           A1b2-C3d4-E5f6     # ← Auto-generated, unique
```

After 1 hour, this credential **automatically disappears**. Even if an attacker steals it, it's already expired.

### 4. Micro-Segmentation with Kubernetes Network Policies

In Kubernetes, you can implement micro-segmentation using Network Policies:

```yaml
# This Network Policy says:
# "The database pod can ONLY receive traffic from the backend pod on port 5432.
#  Nothing else can talk to it."

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-backend-to-db
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: database          # ← Apply this policy to the database pod
  policyTypes:
    - Ingress                # ← Control incoming traffic
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: backend   # ← ONLY the backend pod can connect
      ports:
        - protocol: TCP
          port: 5432         # ← ONLY on port 5432 (PostgreSQL)
```

**What this does:**
- The `database` pod will **only** accept connections from pods labeled `app: backend`
- Only on port `5432` (PostgreSQL)
- Everything else is **blocked by default**
- Even if an attacker compromises the `frontend` pod, they **cannot** reach the database

### 5. Continuous Verification with Conditional Access

Zero Trust doesn't just verify once — it **continuously** checks:

```
User logs in at 9 AM from New York ✅ → Access granted

Same user tries to access at 9:05 AM from Moscow 🤔
     ↓
Risk score increases → "This is impossible travel!"
     ↓
Force re-authentication + Step-up MFA
     ↓
If fails → Block access + Alert security team 🚨
```

**Azure AD Conditional Access Policy Example:**

```json
{
  "displayName": "Require MFA for sensitive apps",
  "conditions": {
    "applications": {
      "includeApplications": ["production-dashboard", "admin-portal"]
    },
    "users": {
      "includeGroups": ["all-employees"]
    },
    "locations": {
      "excludeLocations": ["trusted-office-ips"]
    },
    "signInRiskLevels": ["medium", "high"]
  },
  "grantControls": {
    "operator": "AND",
    "builtInControls": [
      "mfa",
      "compliantDevice"
    ]
  }
}
```

**What this says:**
- For `production-dashboard` and `admin-portal`
- If the user is NOT coming from a trusted office IP
- AND the sign-in risk is medium or high
- THEN require BOTH MFA AND a compliant device

## 🌐 Google BeyondCorp — The Original Zero Trust

Google pioneered Zero Trust after the [Operation Aurora](https://en.wikipedia.org/wiki/Operation_Aurora) hack in 2009. They built **BeyondCorp**, which became the blueprint for Zero Trust.

### Key Ideas from BeyondCorp:

```
1. Access is determined by IDENTITY, not NETWORK LOCATION
   └── Your desk in the office ≠ More trusted than your home

2. Every device must be INVENTORIED and MANAGED
   └── Unknown devices get ZERO access

3. Access to services is granted on a PER-SESSION basis
   └── Each request is evaluated independently

4. Access is determined by DYNAMIC POLICIES
   └── User role + device health + location + risk = access decision
```

### How BeyondCorp Works:

```
Employee at Home                    Employee at Office
       │                                    │
       ▼                                    ▼
  Same Access Process for BOTH:
       │
       ▼
┌──────────────────────┐
│  Access Proxy        │ ← Single entry point
│  (Identity-Aware)    │
└──────────┬───────────┘
           │
    ┌──────▼──────┐
    │  Check:     │
    │  • Identity │
    │  • Device   │
    │  • Context  │
    │  • Policy   │
    └──────┬──────┘
           │
    ┌──────▼──────────┐
    │ Grant/Deny      │
    │ Access to       │
    │ specific app    │
    └─────────────────┘
```

**The key insight:** It doesn't matter WHERE you are. It only matters WHO you are, WHAT device you're using, and WHAT you're trying to access.

## 📋 Implementing Zero Trust — A Step-by-Step Roadmap

You can't implement Zero Trust overnight. Here's a practical roadmap:

### Phase 1: Foundation (Months 1-3)
```
✅ Inventory all users, devices, applications, and data
✅ Implement strong identity (SSO + MFA for ALL users)
✅ Classify your data (What's sensitive? What's public?)
✅ Map your network flows (Who talks to whom?)
```

### Phase 2: Quick Wins (Months 3-6)
```
✅ Enable MFA everywhere (no exceptions!)
✅ Implement least privilege access (review all permissions)
✅ Start network segmentation (separate dev/staging/production)
✅ Deploy a secrets manager (no more hardcoded passwords)
✅ Set up centralized logging and monitoring
```

### Phase 3: Core Implementation (Months 6-12)
```
✅ Deploy an identity-aware proxy (replace VPN for app access)
✅ Implement mTLS for service-to-service communication
✅ Deploy network policies in Kubernetes
✅ Implement device trust checking
✅ Set up conditional access policies
✅ Rotate all long-lived credentials to short-lived ones
```

### Phase 4: Maturity (Months 12+)
```
✅ Continuous verification (every request, every time)
✅ Behavioral analytics (detect anomalies automatically)
✅ Automated response (block suspicious activity instantly)
✅ Regular red team exercises (test your Zero Trust controls)
✅ Zero Trust for CI/CD pipelines (pipeline identity, signed artifacts)
```

## 🔄 Zero Trust for CI/CD Pipelines

Zero Trust isn't just for users — your CI/CD pipelines need it too!

```
OLD WAY:
Jenkins has a permanent AWS access key with admin permissions
     ↓
Key gets leaked → Attacker has FULL access to your AWS account 💀

ZERO TRUST WAY:
Pipeline uses OIDC to get SHORT-LIVED credentials
     ↓
No permanent keys → Nothing to leak
     ↓
Credentials expire in 15 minutes → Blast radius = minimal
```

**Example: GitHub Actions with OIDC (No permanent AWS keys!):**

```yaml
# .github/workflows/deploy.yml
name: Deploy to AWS

on:
  push:
    branches: [main]

permissions:
  id-token: write    # ← Allows GitHub to create an OIDC token
  contents: read

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Configure AWS Credentials (Zero Trust!)
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789012:role/GitHubActionsRole
          aws-region: us-east-1
          # 🎯 Notice: NO access key! NO secret key!
          # GitHub proves its identity via OIDC token
          # AWS gives back short-lived credentials (15 min)

      - name: Deploy
        run: aws s3 sync ./build s3://my-app-bucket
```

**Why this is Zero Trust:**
- No permanent credentials stored anywhere
- GitHub proves its identity via OIDC
- AWS verifies the identity and gives temporary credentials
- Credentials expire in 15 minutes
- Even if someone steals the token, it's already expired

## ❌ Common Mistakes to Avoid

| Mistake | Why It's Wrong | What to Do Instead |
|---------|---------------|-------------------|
| "We bought a Zero Trust product, we're done!" | Zero Trust is a strategy, not a product | Implement it as an ongoing journey |
| "VPN = Zero Trust" | VPN gives network access, not Zero Trust | Use identity-aware proxies instead |
| "MFA alone = Zero Trust" | MFA is just ONE piece of the puzzle | Add device trust, context, least privilege, etc. |
| "We'll do it all at once" | Big bang approaches fail | Implement in phases |
| "Only for humans" | Services and machines need Zero Trust too | Implement workload identity (SPIFFE/SPIRE) |
| "We can skip the inventory" | You can't protect what you don't know about | Start with a complete asset inventory |

```

    └── Implement in phases
```
