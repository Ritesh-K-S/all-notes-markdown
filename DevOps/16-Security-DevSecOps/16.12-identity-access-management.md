## 🤔 Let's Start With a Story — The Problem IAM Solves

Imagine you work at a company with 50 different tools:

```
Tools you use every day:
├── GitHub (code)
├── Jira (tickets)
├── Slack (chat)
├── AWS Console (cloud)
├── Grafana (monitoring)
├── Jenkins (CI/CD)
├── Confluence (docs)
├── SonarQube (code quality)
└── ... 42 more tools
```

**Without IAM (The Nightmare):**

```
😰 Employee joins the company:
   → Create 50 separate accounts with 50 different passwords
   → Employee forgets passwords constantly
   → Each tool has different password requirements
   → No way to enforce MFA consistently

😱 Employee leaves the company:
   → HR tells IT: "Hey, John left"
   → IT has to MANUALLY disable 50 accounts
   → They forget to disable 3 of them
   → John still has access to AWS, Grafana, and Jenkins 💀
```

**With IAM (The Dream):**

```
😊 Employee joins the company:
   → Create ONE account in the Identity Provider (e.g., Okta)
   → Employee logs in ONCE and gets access to ALL tools (SSO)
   → MFA enforced everywhere automatically

😊 Employee leaves the company:
   → Disable ONE account in Okta
   → Instantly locked out of ALL 50 tools ✅
   → No forgotten accounts, no lingering access
```

**That's the power of IAM.** Let's learn how it all works.

## 🔑 The Two Big Questions — AuthN vs AuthZ

Before we dive into protocols, understand these two fundamental concepts:

### Authentication (AuthN) — "Who are you?"

```
🔑 Authentication = Proving your IDENTITY

"Hi, I'm Alice"
"Prove it!"
"Here's my password... and here's my MFA code"
"Okay, you ARE Alice ✅"
```

Authentication methods:
- **Something you know** → Password, PIN
- **Something you have** → Phone (MFA), security key (YubiKey)
- **Something you are** → Fingerprint, face recognition

### Authorization (AuthZ) — "What are you allowed to do?"

```
🛡️ Authorization = Checking your PERMISSIONS

"Okay, you're Alice. What do you want to do?"
"I want to delete the production database"
"Let me check... Alice has 'developer' role... 
 Developers can READ databases but NOT DELETE them"
"❌ ACCESS DENIED"
```

### How They Work Together:

```
Step 1: Authentication → "Are you who you say you are?"
                              ↓
Step 2: Authorization  → "Are you allowed to do what you're trying to do?"

Example:
  Alice types username + password     → AuthN: "Yes, this is Alice" ✅
  Alice tries to access admin panel   → AuthZ: "Alice is not an admin" ❌
  Alice tries to access developer dashboard → AuthZ: "Alice is a developer" ✅
```

## 🌐 SSO — Single Sign-On (The Big Picture)

### What is SSO?

SSO (Single Sign-On) means **logging in once and getting access to multiple applications** without logging in again for each one.

```
WITHOUT SSO:                          WITH SSO:
                                      
GitHub → Login 🔑                     Login ONCE to Okta 🔑
Jira → Login 🔑                           ↓
Slack → Login 🔑                      Click GitHub → ✅ Logged in!
AWS → Login 🔑                        Click Jira → ✅ Logged in!
Grafana → Login 🔑                    Click Slack → ✅ Logged in!
                                      Click AWS → ✅ Logged in!
5 logins 😫                           Click Grafana → ✅ Logged in!
                                      
                                      1 login 😊
```

### How SSO Works (Simple View):

```
┌──────────┐         ┌──────────────────┐         ┌──────────┐
│          │         │                  │         │          │
│   User   │ ──1──►  │ Identity Provider│         │  GitHub  │
│          │         │    (Okta, etc)   │         │  (App)   │
│          │ ◄──2──  │                  │         │          │
│          │         └──────────────────┘         │          │
│          │                                      │          │
│          │ ──3── (Token/Assertion) ─────────►   │          │
│          │                                      │          │
│          │ ◄──4── "Welcome, Alice!" ────────    │          │
└──────────┘                                      └──────────┘

Step 1: User goes to Identity Provider (IdP) and logs in
Step 2: IdP verifies identity and creates a token/assertion
Step 3: User presents the token to the application (GitHub)
Step 4: Application verifies the token and grants access
```

SSO is a **concept** (log in once, access many). It's **implemented** using protocols like **SAML**, **OAuth + OIDC**, or **Kerberos**.

## 📜 SAML — Security Assertion Markup Language

### What is SAML?

SAML is an **XML-based protocol** for SSO in **enterprise environments**. It's been around since 2005 and is the standard for corporate SSO.

Think of SAML like a **passport**:
- Your government (Identity Provider) issues you a passport
- You show the passport to another country (Service Provider)
- They trust the passport because they trust your government

### Key Terms:

| Term | What It Is | Example |
|------|-----------|---------|
| **Identity Provider (IdP)** | The system that authenticates users and issues SAML assertions | Okta, Azure AD, Google Workspace |
| **Service Provider (SP)** | The application that the user wants to access | GitHub, Jira, AWS Console |
| **SAML Assertion** | An XML document containing the user's identity info | "This is Alice, she's in the admin group" |
| **SAML Request** | SP asks IdP to authenticate a user | "Please verify who this person is" |
| **SAML Response** | IdP sends back the assertion | "This is Alice, verified ✅" |

### How SAML Works — Step by Step:

```
The SAML Flow (SP-Initiated — most common):

 User          Service Provider          Identity Provider
  │             (e.g., Jira)              (e.g., Okta)
  │                  │                         │
  ├──1── "I want     │                         │
  │      to access   │                         │
  │      Jira"  ────►│                         │
  │                  │                         │
  │                  ├──2── "Please             │
  │                  │      authenticate       │
  │                  │      this user"  ──────►│
  │                  │      (SAML Request)      │
  │                  │                         │
  │◄─────────────── 3── Redirect to            │
  │                      Okta login page ──────│
  │                                            │
  ├──4── User logs in                          │
  │      (username + password + MFA) ─────────►│
  │                                            │
  │                                            │ 5── Okta verifies
  │                                            │     credentials
  │                                            │
  │◄──────────── 6── SAML Response ────────────│
  │                  (contains SAML Assertion)  │
  │                  "This IS Alice, she's in   │
  │                   the developers group"     │
  │                                            │
  ├──7── Forward SAML                          │
  │      Response to ──►│                      │
  │      Jira            │                      │
  │                      │ 8── Jira verifies    │
  │                      │     the assertion    │
  │                      │     (checks signature│
  │                      │      from Okta)      │
  │                      │                      │
  │◄──9── "Welcome       │                      │
  │       to Jira,       │                      │
  │       Alice!" ──────│                      │
```

### What a SAML Assertion Looks Like (XML):

```xml
<saml:Assertion xmlns:saml="urn:oasis:names:tc:SAML:2.0:assertion"
    ID="_abc123"
    IssueInstant="2024-01-15T10:30:00Z"
    Version="2.0">
    
    <!-- WHO issued this assertion -->
    <saml:Issuer>https://okta.mycompany.com</saml:Issuer>
    
    <!-- Digital signature (proves this is really from Okta) -->
    <ds:Signature>
        <!-- Cryptographic signature here -->
    </ds:Signature>
    
    <!-- WHO is this person -->
    <saml:Subject>
        <saml:NameID>alice@mycompany.com</saml:NameID>
    </saml:Subject>
    
    <!-- WHEN is this valid -->
    <saml:Conditions NotBefore="2024-01-15T10:30:00Z"
                     NotOnOrAfter="2024-01-15T11:30:00Z">
        <!-- Valid for 1 hour -->
    </saml:Conditions>
    
    <!-- WHAT attributes does this person have -->
    <saml:AttributeStatement>
        <saml:Attribute Name="firstName">
            <saml:AttributeValue>Alice</saml:AttributeValue>
        </saml:Attribute>
        <saml:Attribute Name="email">
            <saml:AttributeValue>alice@mycompany.com</saml:AttributeValue>
        </saml:Attribute>
        <saml:Attribute Name="groups">
            <saml:AttributeValue>developers</saml:AttributeValue>
            <saml:AttributeValue>devops-team</saml:AttributeValue>
        </saml:Attribute>
    </saml:AttributeStatement>
</saml:Assertion>
```

**What this says:**
- Issued by Okta (`saml:Issuer`)
- The user is `alice@mycompany.com` (`saml:NameID`)
- Valid for 1 hour (`saml:Conditions`)
- Alice is in the `developers` and `devops-team` groups (`saml:AttributeStatement`)
- Digitally signed by Okta (so no one can forge it)

## 🔐 OAuth 2.0 — The Authorization Framework

### What is OAuth 2.0?

OAuth 2.0 is an **authorization** framework. It lets an application access resources on behalf of a user **without** needing the user's password.

**The key insight:** OAuth 2.0 is about **authorization** (what you can do), NOT **authentication** (who you are). Although people often confuse the two.

### The Problem OAuth Solves — A Story:

```
You want a third-party app ("PhotoPrinter") to print your Google Photos.

WITHOUT OAuth:
  PhotoPrinter: "Give me your Google username and password"
  You: "Uhh... okay" 😟
  PhotoPrinter now has FULL access to your Google account
  They can read your emails, delete your files, change your password...
  
WITH OAuth:
  PhotoPrinter: "I need to access your photos"
  Google: "Hey User, PhotoPrinter wants to access your photos. Allow?"
  You: "Yes, but ONLY my photos, nothing else"
  Google gives PhotoPrinter a TOKEN that ONLY works for photos
  PhotoPrinter can NEVER access your email, drive, or anything else ✅
```

### Key Terms:

| Term | What It Is | Example |
|------|-----------|---------|
| **Resource Owner** | The user who owns the data | You (you own your Google Photos) |
| **Client** | The app that wants access | PhotoPrinter app |
| **Authorization Server** | Issues tokens after user consent | Google's auth server |
| **Resource Server** | Hosts the protected resources | Google Photos API |
| **Access Token** | A key that grants specific access | `eyJhbGciOiJSUzI1...` |
| **Refresh Token** | Used to get a new Access Token when it expires | Long-lived token |
| **Scope** | What the token is allowed to do | `photos.readonly` |

### OAuth 2.0 Flow — Authorization Code (Most Common & Secure):

```
The Authorization Code Flow:

 User           Client App          Auth Server         Resource Server
(Browser)     (PhotoPrinter)        (Google)           (Google Photos API)
  │                │                     │                     │
  ├──1── "Print    │                     │                     │
  │     my photos" │                     │                     │
  │     ──────────►│                     │                     │
  │                │                     │                     │
  │◄──2── Redirect │                     │                     │
  │       to Google│                     │                     │
  │       login ───│─────────────────────│                     │
  │                                      │                     │
  ├──3── Login to Google ───────────────►│                     │
  │      (username + password + MFA)     │                     │
  │                                      │                     │
  │◄──4── "PhotoPrinter wants access    │                     │
  │       to your photos. Allow?" ──────│                     │
  │                                      │                     │
  ├──5── "Yes, allow" ─────────────────►│                     │
  │                                      │                     │
  │◄──6── Redirect back with            │                     │
  │       AUTHORIZATION CODE ───────────│                     │
  │       (one-time code)                │                     │
  │                                      │                     │
  │──────►│                              │                     │
  │       │──7── Exchange code           │                     │
  │       │      for ACCESS TOKEN ──────►│                     │
  │       │      (server-to-server,      │                     │
  │       │       user never sees this)  │                     │
  │       │                              │                     │
  │       │◄──8── Access Token ─────────│                     │
  │       │       + Refresh Token        │                     │
  │       │                              │                     │
  │       │──9── "Give me photos" ───────────────────────────►│
  │       │      Authorization: Bearer <access_token>         │
  │       │                                                    │
  │       │◄──10── Here are the photos ───────────────────────│
  │       │                                                    │
  │◄──11──│ "Here are your                                     │
  │       │  printed photos!"                                  │
```

**Why is this secure?**
- The user NEVER gives their password to PhotoPrinter
- The access token has LIMITED scope (only photos)
- The access token EXPIRES (short-lived)
- The authorization code is one-time use
- Token exchange happens server-to-server (not visible in the browser)

### OAuth 2.0 Grant Types:

| Grant Type | Use Case | Security |
|-----------|----------|----------|
| **Authorization Code** | Server-side web apps (most common) | Most secure ✅ |
| **Authorization Code + PKCE** | Single-page apps (SPA), mobile apps | Secure for public clients ✅ |
| **Client Credentials** | Machine-to-machine (no user involved) | Secure for services ✅ |
| **Device Code** | Smart TVs, CLI tools (limited input) | Good for limited devices |
| **Implicit** | ⚠️ DEPRECATED — Don't use | Insecure ❌ |
| **Resource Owner Password** | ⚠️ DEPRECATED — Don't use | Insecure ❌ |

### OAuth 2.0 Access Token Example (JWT):

An Access Token is usually a **JWT (JSON Web Token)**. Let's decode one:

```
eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkFsaWNlIiwiaWF0IjoxNjM1NjMyMDAwLCJleHAiOjE2MzU2MzU2MDAsInNjb3BlIjoicGhvdG9zLnJlYWRvbmx5In0.signature
```

This token has **three parts** separated by dots:

```
HEADER.PAYLOAD.SIGNATURE

1. HEADER (how it's signed):
{
  "alg": "RS256",     ← Algorithm used to sign
  "typ": "JWT"        ← Type of token
}

2. PAYLOAD (the actual data):
{
  "sub": "1234567890",           ← Subject (user ID)
  "name": "Alice",              ← User's name
  "iat": 1635632000,            ← Issued At (timestamp)
  "exp": 1635635600,            ← Expires At (1 hour later)
  "scope": "photos.readonly"    ← What this token can do
}

3. SIGNATURE:
RS256(
  base64(header) + "." + base64(payload),
  privateKey
)
← Proves the token hasn't been tampered with
```

**Key point:** The token itself contains all the info needed. The Resource Server can verify it without calling the Authorization Server. This is called a **self-contained token**.

### Client Credentials Flow (Machine-to-Machine):

When two services talk to each other (no human involved), they use the **Client Credentials** flow:

```bash
# Service A wants to call Service B's API
# Service A has a client_id and client_secret

# Step 1: Get an access token
curl -X POST https://auth.example.com/oauth/token \
  -d "grant_type=client_credentials" \
  -d "client_id=service-a-id" \
  -d "client_secret=service-a-secret" \
  -d "scope=api.read"

# Response:
{
  "access_token": "eyJhbGciOi...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "scope": "api.read"
}

# Step 2: Use the token to call Service B
curl -H "Authorization: Bearer eyJhbGciOi..." \
  https://service-b.example.com/api/data
```

## 🆔 OpenID Connect (OIDC) — Authentication on Top of OAuth

### What is OIDC?

OIDC (OpenID Connect) is a **thin identity layer on top of OAuth 2.0** that adds **authentication**.

Remember:
- **OAuth 2.0** = Authorization (what you can do)
- **OIDC** = Authentication (who you are) built ON TOP of OAuth 2.0

```
OAuth 2.0 alone:
  "Here's a token that lets you access photos"
  But... who IS this person? 🤷

OAuth 2.0 + OIDC:
  "Here's a token that lets you access photos"
  AND "This person is Alice, email alice@example.com, verified ✅"
```

### What OIDC Adds to OAuth 2.0:

| OAuth 2.0 | OIDC Adds |
|-----------|-----------|
| Access Token (authorization) | **ID Token** (authentication — who you are) |
| Scopes like `photos.read` | **Standard scopes**: `openid`, `profile`, `email` |
| No user info standard | **/userinfo endpoint** (get user details) |
| No standard for identity | **Standard claims** (name, email, picture, etc.) |

### OIDC Flow — Step by Step:

The OIDC flow is **almost identical** to the OAuth Authorization Code flow, with one addition — the **ID Token**.

```
OIDC Flow:

Step 1: Client redirects user to IdP with scope=openid
        https://auth.example.com/authorize?
          response_type=code&
          client_id=my-app&
          redirect_uri=https://myapp.com/callback&
          scope=openid profile email&     ← OIDC scopes!
          state=random123

Step 2: User logs in at IdP

Step 3: IdP redirects back with authorization code
        https://myapp.com/callback?code=abc123&state=random123

Step 4: Client exchanges code for tokens
        POST https://auth.example.com/token
        {
          "grant_type": "authorization_code",
          "code": "abc123",
          "redirect_uri": "https://myapp.com/callback",
          "client_id": "my-app",
          "client_secret": "my-secret"
        }

Step 5: IdP returns BOTH tokens:
        {
          "access_token": "eyJhbGciOi...",    ← For accessing APIs
          "id_token": "eyJhbGciOi...",         ← NEW! For identifying the user
          "token_type": "Bearer",
          "expires_in": 3600,
          "refresh_token": "dGhpcyBpcyBh..."
        }
```

### The ID Token (JWT):

```json
{
  "iss": "https://auth.example.com",     // Issuer (who created this token)
  "sub": "user-12345",                   // Subject (unique user ID)
  "aud": "my-app",                       // Audience (who this token is for)
  "exp": 1635635600,                     // Expiration time
  "iat": 1635632000,                     // Issued at
  "nonce": "random-nonce-123",           // Prevents replay attacks
  
  // Standard OIDC Claims (user info):
  "name": "Alice Johnson",
  "email": "alice@example.com",
  "email_verified": true,
  "picture": "https://example.com/alice.jpg",
  "groups": ["developers", "devops-team"]
}
```

### OIDC Standard Scopes:

| Scope | What Info You Get |
|-------|------------------|
| `openid` | Required. Returns the `sub` (subject/user ID) claim |
| `profile` | name, family_name, given_name, picture, etc. |
| `email` | email, email_verified |
| `address` | Street address, city, country, etc. |
| `phone` | phone_number, phone_number_verified |

### OIDC in Kubernetes — A Real DevOps Use Case:

Kubernetes can use OIDC to authenticate users without managing its own user database!

```yaml
# kube-apiserver configuration for OIDC
# This tells Kubernetes to trust tokens from your Identity Provider

apiVersion: v1
kind: Config

# API Server flags:
# --oidc-issuer-url=https://accounts.google.com
# --oidc-client-id=kubernetes
# --oidc-username-claim=email
# --oidc-groups-claim=groups
```

**How it works:**

```
Developer                 Identity Provider            Kubernetes
                          (Google, Okta)               API Server
   │                           │                          │
   ├──1── Login ──────────────►│                          │
   │                           │                          │
   │◄──2── ID Token ──────────│                          │
   │   (jwt with email,        │                          │
   │    groups, etc.)           │                          │
   │                           │                          │
   ├──3── kubectl get pods ────────────────────────────►  │
   │      Authorization: Bearer <ID Token>                │
   │                                                      │
   │                                    │ 4── Verify token│
   │                                    │     Check issuer│
   │                                    │     Check groups│
   │                                    │     Apply RBAC  │
   │                                                      │
   │◄──5── Here are your pods ────────────────────────────│
```

**RBAC for OIDC users in Kubernetes:**

```yaml
# ClusterRoleBinding that gives 'developer' access 
# to users in the 'developers' OIDC group

apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: developers-view
subjects:
  - kind: Group
    name: developers          # ← This matches the OIDC 'groups' claim
    apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: view                  # ← Built-in read-only role
  apiGroup: rbac.authorization.k8s.io
```

Now anyone in the `developers` OIDC group can **view** resources in Kubernetes, but can't modify them.

### OIDC for CI/CD — GitHub Actions OIDC:

GitHub Actions uses OIDC to authenticate with cloud providers **without storing long-lived secrets**:

```yaml
# .github/workflows/deploy.yml
name: Deploy

permissions:
  id-token: write       # ← Allow GitHub to create OIDC tokens

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Configure AWS credentials via OIDC
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789012:role/github-actions
          aws-region: us-east-1
          # No access keys needed! GitHub proves identity via OIDC token
```

**What happens behind the scenes:**

```
GitHub Actions                    AWS
     │                             │
     ├──1── "Here's my OIDC token" │
     │      Claims:                │
     │      - iss: token.actions.  │
     │            githubusercontent│
     │      - sub: repo:org/       │
     │            repo:ref:main    │
     │      - aud: sts.amazonaws.  │
     │            com              │
     │      ────────────────────►  │
     │                             │
     │                 2── AWS verifies:
     │                     - Is this really from GitHub?
     │                     - Is this the right repo?
     │                     - Is this the right branch?
     │                             │
     │◄──3── Temporary credentials │
     │       (15 min expiry)       │
     │       ────────────────────  │
```

## 📊 Comparison — SAML vs OAuth 2.0 vs OIDC

| Feature | SAML | OAuth 2.0 | OIDC |
|---------|------|-----------|------|
| **Purpose** | SSO / Authentication | Authorization | Authentication (on top of OAuth) |
| **Format** | XML | JSON | JSON (JWT) |
| **Token Type** | SAML Assertion (XML) | Access Token | ID Token + Access Token |
| **Best For** | Enterprise SSO, Legacy apps | API authorization, Mobile apps | Modern SSO, Web & mobile apps |
| **Complexity** | Complex (XML, certificates) | Medium | Medium |
| **Year Created** | 2005 | 2012 | 2014 |
| **Mobile Friendly** | No (XML is heavy) | Yes | Yes |
| **Still Used?** | Yes (enterprise) | Yes (everywhere) | Yes (everywhere) |

### When to Use What:

```
Decision Tree:
│
├── Need to integrate with ENTERPRISE apps (Salesforce, Workday, etc.)?
│   └── Use SAML ← These apps often only support SAML
│
├── Need to authorize access to an API?
│   └── Use OAuth 2.0 ← It's designed for API authorization
│
├── Need to authenticate users in a MODERN web/mobile app?
│   └── Use OIDC ← Modern, JSON-based, supports both AuthN + AuthZ
│
├── Need machine-to-machine authentication?
│   └── Use OAuth 2.0 Client Credentials
│
└── Building something new from scratch?
    └── Use OIDC ← It's the modern standard that does everything
```

## 🔄 SCIM — Automated User Provisioning

SCIM (System for Cross-domain Identity Management) is a protocol for **automatically** creating, updating, and deleting user accounts across systems.

```
WITHOUT SCIM (Manual):
  HR adds Alice to company → Admin manually creates accounts in:
  ├── Okta ✅
  ├── GitHub ✅
  ├── Jira ✅
  ├── AWS ✅
  └── Slack... forgot! ❌

WITH SCIM (Automatic):
  HR adds Alice to company → Okta AUTOMATICALLY provisions accounts in:
  ├── GitHub ✅ (via SCIM)
  ├── Jira ✅ (via SCIM)
  ├── AWS ✅ (via SCIM)
  └── Slack ✅ (via SCIM)
  
  HR removes Alice → Okta AUTOMATICALLY deprovisions ALL accounts ✅
```

### SCIM API Example:

```bash
# Create a user via SCIM API
curl -X POST https://api.example.com/scim/v2/Users \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/scim+json" \
  -d '{
    "schemas": ["urn:ietf:params:scim:schemas:core:2.0:User"],
    "userName": "alice@example.com",
    "name": {
      "givenName": "Alice",
      "familyName": "Johnson"
    },
    "emails": [
      {
        "value": "alice@example.com",
        "primary": true
      }
    ],
    "active": true,
    "groups": [
      {
        "value": "developers",
        "display": "Developers"
      }
    ]
  }'

# Deactivate a user (when they leave)
curl -X PATCH https://api.example.com/scim/v2/Users/user-id-123 \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/scim+json" \
  -d '{
    "schemas": ["urn:ietf:params:scim:api:messages:2.0:PatchOp"],
    "Operations": [
      {
        "op": "replace",
        "path": "active",
        "value": false
      }
    ]
  }'
```

## 🏢 IAM in Cloud Platforms

### AWS IAM — Quick Overview:

```
AWS IAM Structure:
│
├── Users (human identities)
│   └── alice-developer (has access keys + console password)
│
├── Groups (collection of users)
│   ├── developers (alice, bob)
│   └── admins (charlie)
│
├── Roles (temporary identities — for services/cross-account)
│   ├── ec2-s3-role (EC2 instances can access S3)
│   ├── lambda-dynamodb-role (Lambda can access DynamoDB)
│   └── github-actions-role (GitHub can deploy — OIDC)
│
├── Policies (JSON documents that define permissions)
│   ├── AmazonS3ReadOnlyAccess (managed policy)
│   └── custom-deployment-policy (custom policy)
│
└── Identity Providers (external IdPs)
    ├── Okta (SAML)
    └── GitHub Actions (OIDC)
```

**AWS IAM Policy Example:**

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowS3ReadOnly",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::my-app-bucket",
        "arn:aws:s3:::my-app-bucket/*"
      ]
    },
    {
      "Sid": "DenyDeleteBucket",
      "Effect": "Deny",
      "Action": "s3:DeleteBucket",
      "Resource": "*"
    }
  ]
}
```

**What this policy says:**
- ✅ **Allow** reading objects and listing the `my-app-bucket` bucket
- ❌ **Deny** deleting ANY bucket (explicit deny always wins)

### Kubernetes RBAC — Quick Overview:

```yaml
# Role: What permissions are available
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: production
  name: pod-reader
rules:
  - apiGroups: [""]           # Core API group
    resources: ["pods"]        # Resource type
    verbs: ["get", "list", "watch"]  # Allowed actions (read-only)

---
# RoleBinding: Who gets these permissions
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  namespace: production
  name: read-pods-binding
subjects:
  - kind: User
    name: alice@example.com    # ← From OIDC
    apiGroup: rbac.authorization.k8s.io
  - kind: Group
    name: developers           # ← From OIDC groups claim
    apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: pod-reader             # ← The role defined above
  apiGroup: rbac.authorization.k8s.io
```

## ❌ Common Mistakes to Avoid

| Mistake | Why It's Wrong | What to Do Instead |
|---------|---------------|-------------------|
| Using long-lived API keys | Keys get leaked, never expire | Use OIDC / short-lived tokens |
| Sharing service accounts | No accountability, can't audit | One service account per service |
| Not implementing MFA | Passwords alone are easy to steal | Enforce MFA for ALL users |
| Over-permissive policies | `"Action": "*", "Resource": "*"` = admin | Least privilege — only what's needed |
| Not rotating credentials | Old credentials accumulate risk | Auto-rotate or use short-lived tokens |
| Using SAML for new apps | SAML is complex and XML-based | Use OIDC for new applications |
| Not using SCIM | Manual provisioning leads to orphaned accounts | Automate with SCIM |
| Storing tokens in localStorage | Vulnerable to XSS attacks | Use httpOnly cookies or token handlers |

## 🧠 Quick Recap — IAM in 30 Seconds

```
Identity & Access Management:
│
├── CONCEPTS:
│   ├── Authentication (AuthN): "Who are you?"
│   ├── Authorization (AuthZ): "What can you do?"
│   └── SSO: Login once, access everything
│
├── PROTOCOLS:
│   ├── SAML: Enterprise SSO (XML-based, since 2005)
│   ├── OAuth 2.0: API authorization (JSON, since 2012)
│   ├── OIDC: Authentication on top of OAuth (JWT, since 2014)
│   └── SCIM: Automated user provisioning
│
├── KEY COMPONENTS:
│   ├── Identity Provider (IdP): Okta, Azure AD, Keycloak
│   ├── Access Tokens: Short-lived, scoped
│   ├── ID Tokens: User identity (JWT)
│   ├── RBAC: Role-Based Access Control
│   └── MFA: Multi-Factor Authentication
│
└── BEST PRACTICES:
    ├── Use OIDC for new applications
    ├── Short-lived tokens over permanent credentials
    ├── Least privilege everywhere
    ├── MFA for all users
    ├── Automate provisioning with SCIM
    └── Centralize identity in ONE IdP
```
