# 16.12 вҖ” Identity & Access Management вҖ” OAuth, OIDC, SAML, SSO

> **One Line Answer:** Identity & Access Management (IAM) is about **verifying who someone is** (authentication) and **what they're allowed to do** (authorization) using protocols like **OAuth 2.0** (authorization), **OpenID Connect** (authentication), **SAML** (enterprise SSO), and **SSO** (one login for everything).

---

## рҹӨ” Let's Start With a Story вҖ” The Problem IAM Solves

Imagine you work at a company with 50 different tools:

```
Tools you use every day:
в”ңв”Җв”Җ GitHub (code)
в”ңв”Җв”Җ Jira (tickets)
в”ңв”Җв”Җ Slack (chat)
в”ңв”Җв”Җ AWS Console (cloud)
в”ңв”Җв”Җ Grafana (monitoring)
в”ңв”Җв”Җ Jenkins (CI/CD)
в”ңв”Җв”Җ Confluence (docs)
в”ңв”Җв”Җ SonarQube (code quality)
в””в”Җв”Җ ... 42 more tools
```

**Without IAM (The Nightmare):**

```
рҹҳ° Employee joins the company:
   вҶ’ Create 50 separate accounts with 50 different passwords
   вҶ’ Employee forgets passwords constantly
   вҶ’ Each tool has different password requirements
   вҶ’ No way to enforce MFA consistently

рҹҳұ Employee leaves the company:
   вҶ’ HR tells IT: "Hey, John left"
   вҶ’ IT has to MANUALLY disable 50 accounts
   вҶ’ They forget to disable 3 of them
   вҶ’ John still has access to AWS, Grafana, and Jenkins рҹ’Җ
```

**With IAM (The Dream):**

```
рҹҳҠ Employee joins the company:
   вҶ’ Create ONE account in the Identity Provider (e.g., Okta)
   вҶ’ Employee logs in ONCE and gets access to ALL tools (SSO)
   вҶ’ MFA enforced everywhere automatically

рҹҳҠ Employee leaves the company:
   вҶ’ Disable ONE account in Okta
   вҶ’ Instantly locked out of ALL 50 tools вң…
   вҶ’ No forgotten accounts, no lingering access
```

**That's the power of IAM.** Let's learn how it all works.

## рҹ”‘ The Two Big Questions вҖ” AuthN vs AuthZ

Before we dive into protocols, understand these two fundamental concepts:

### Authentication (AuthN) вҖ” "Who are you?"

```
рҹ”‘ Authentication = Proving your IDENTITY

"Hi, I'm Alice"
"Prove it!"
"Here's my password... and here's my MFA code"
"Okay, you ARE Alice вң…"
```

Authentication methods:
- **Something you know** вҶ’ Password, PIN
- **Something you have** вҶ’ Phone (MFA), security key (YubiKey)
- **Something you are** вҶ’ Fingerprint, face recognition

### Authorization (AuthZ) вҖ” "What are you allowed to do?"

```
рҹӣЎпёҸ Authorization = Checking your PERMISSIONS

"Okay, you're Alice. What do you want to do?"
"I want to delete the production database"
"Let me check... Alice has 'developer' role... 
 Developers can READ databases but NOT DELETE them"
"вқҢ ACCESS DENIED"
```

### How They Work Together:

```
Step 1: Authentication вҶ’ "Are you who you say you are?"
                              вҶ“
Step 2: Authorization  вҶ’ "Are you allowed to do what you're trying to do?"

Example:
  Alice types username + password     вҶ’ AuthN: "Yes, this is Alice" вң…
  Alice tries to access admin panel   вҶ’ AuthZ: "Alice is not an admin" вқҢ
  Alice tries to access developer dashboard вҶ’ AuthZ: "Alice is a developer" вң…
```

## рҹҢҗ SSO вҖ” Single Sign-On (The Big Picture)

### What is SSO?

SSO (Single Sign-On) means **logging in once and getting access to multiple applications** without logging in again for each one.

```
WITHOUT SSO:                          WITH SSO:
                                      
GitHub вҶ’ Login рҹ”‘                     Login ONCE to Okta рҹ”‘
Jira вҶ’ Login рҹ”‘                           вҶ“
Slack вҶ’ Login рҹ”‘                      Click GitHub вҶ’ вң… Logged in!
AWS вҶ’ Login рҹ”‘                        Click Jira вҶ’ вң… Logged in!
Grafana вҶ’ Login рҹ”‘                    Click Slack вҶ’ вң… Logged in!
                                      Click AWS вҶ’ вң… Logged in!
5 logins рҹҳ«                           Click Grafana вҶ’ вң… Logged in!
                                      
                                      1 login рҹҳҠ
```

## рҹ“ң SAML вҖ” Security Assertion Markup Language

### What is SAML?

SAML is an **XML-based protocol** for SSO in **enterprise environments**. It's been around since 2005 and is the standard for corporate SSO.

Think of SAML like a **passport**:
- Your government (Identity Provider) issues you a passport
- You show the passport to another country (Service Provider)
- They trust the passport because they trust your government

### Key Terms:

| Term | What It Is | Example |
|------|-----------|---------|
| **Identity Provider (IdP)** | The system that authenticates users and issues SAML assertions | Okta, Azure AD, Google Workspace |
| **Service Provider (SP)** | The application that the user wants to access | GitHub, Jira, AWS Console |
| **SAML Assertion** | An XML document containing the user's identity info | "This is Alice, she's in the admin group" |
| **SAML Request** | SP asks IdP to authenticate a user | "Please verify who this person is" |
| **SAML Response** | IdP sends back the assertion | "This is Alice, verified вң…" |

### What a SAML Assertion Looks Like (XML):

```xml
<saml:Assertion xmlns:saml="urn:oasis:names:tc:SAML:2.0:assertion"
    ID="_abc123"
    IssueInstant="2024-01-15T10:30:00Z"
    Version="2.0">
    
    <!-- WHO issued this assertion -->
    <saml:Issuer>https://okta.mycompany.com</saml:Issuer>
    
    <!-- Digital signature (proves this is really from Okta) -->
    <ds:Signature>
        <!-- Cryptographic signature here -->
    </ds:Signature>
    
    <!-- WHO is this person -->
    <saml:Subject>
        <saml:NameID>alice@mycompany.com</saml:NameID>
    </saml:Subject>
    
    <!-- WHEN is this valid -->
    <saml:Conditions NotBefore="2024-01-15T10:30:00Z"
                     NotOnOrAfter="2024-01-15T11:30:00Z">
        <!-- Valid for 1 hour -->
    </saml:Conditions>
    
    <!-- WHAT attributes does this person have -->
    <saml:AttributeStatement>
        <saml:Attribute Name="firstName">
            <saml:AttributeValue>Alice</saml:AttributeValue>
        </saml:Attribute>
        <saml:Attribute Name="email">
            <saml:AttributeValue>alice@mycompany.com</saml:AttributeValue>
        </saml:Attribute>
        <saml:Attribute Name="groups">
            <saml:AttributeValue>developers</saml:AttributeValue>
            <saml:AttributeValue>devops-team</saml:AttributeValue>
        </saml:Attribute>
    </saml:AttributeStatement>
</saml:Assertion>
```

**What this says:**
- Issued by Okta (`saml:Issuer`)
- The user is `alice@mycompany.com` (`saml:NameID`)
- Valid for 1 hour (`saml:Conditions`)
- Alice is in the `developers` and `devops-team` groups (`saml:AttributeStatement`)
- Digitally signed by Okta (so no one can forge it)

## рҹ”җ OAuth 2.0 вҖ” The Authorization Framework

### What is OAuth 2.0?

OAuth 2.0 is an **authorization** framework. It lets an application access resources on behalf of a user **without** needing the user's password.

**The key insight:** OAuth 2.0 is about **authorization** (what you can do), NOT **authentication** (who you are). Although people often confuse the two.

### The Problem OAuth Solves вҖ” A Story:

```
You want a third-party app ("PhotoPrinter") to print your Google Photos.

WITHOUT OAuth:
  PhotoPrinter: "Give me your Google username and password"
  You: "Uhh... okay" рҹҳҹ
  PhotoPrinter now has FULL access to your Google account
  They can read your emails, delete your files, change your password...
  
WITH OAuth:
  PhotoPrinter: "I need to access your photos"
  Google: "Hey User, PhotoPrinter wants to access your photos. Allow?"
  You: "Yes, but ONLY my photos, nothing else"
  Google gives PhotoPrinter a TOKEN that ONLY works for photos
  PhotoPrinter can NEVER access your email, drive, or anything else вң…
```

### Key Terms:

| Term | What It Is | Example |
|------|-----------|---------|
| **Resource Owner** | The user who owns the data | You (you own your Google Photos) |
| **Client** | The app that wants access | PhotoPrinter app |
| **Authorization Server** | Issues tokens after user consent | Google's auth server |
| **Resource Server** | Hosts the protected resources | Google Photos API |
| **Access Token** | A key that grants specific access | `eyJhbGciOiJSUzI1...` |
| **Refresh Token** | Used to get a new Access Token when it expires | Long-lived token |
| **Scope** | What the token is allowed to do | `photos.readonly` |

### OAuth 2.0 Access Token Example (JWT):

An Access Token is usually a **JWT (JSON Web Token)**. Let's decode one:

```
eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkFsaWNlIiwiaWF0IjoxNjM1NjMyMDAwLCJleHAiOjE2MzU2MzU2MDAsInNjb3BlIjoicGhvdG9zLnJlYWRvbmx5In0.signature
```

This token has **three parts** separated by dots:

```
HEADER.PAYLOAD.SIGNATURE

1. HEADER (how it's signed):
{
  "alg": "RS256",     вҶҗ Algorithm used to sign
  "typ": "JWT"        вҶҗ Type of token
}

2. PAYLOAD (the actual data):
{
  "sub": "1234567890",           вҶҗ Subject (user ID)
  "name": "Alice",              вҶҗ User's name
  "iat": 1635632000,            вҶҗ Issued At (timestamp)
  "exp": 1635635600,            вҶҗ Expires At (1 hour later)
  "scope": "photos.readonly"    вҶҗ What this token can do
}

3. SIGNATURE:
RS256(
  base64(header) + "." + base64(payload),
  privateKey
)
вҶҗ Proves the token hasn't been tampered with
```

**Key point:** The token itself contains all the info needed. The Resource Server can verify it without calling the Authorization Server. This is called a **self-contained token**.

### Client Credentials Flow (Machine-to-Machine):

When two services talk to each other (no human involved), they use the **Client Credentials** flow:

```bash
# Service A wants to call Service B's API
# Service A has a client_id and client_secret

# Step 1: Get an access token
curl -X POST https://auth.example.com/oauth/token \
  -d "grant_type=client_credentials" \
  -d "client_id=service-a-id" \
  -d "client_secret=service-a-secret" \
  -d "scope=api.read"

# Response:
{
  "access_token": "eyJhbGciOi...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "scope": "api.read"
}

# Step 2: Use the token to call Service B
curl -H "Authorization: Bearer eyJhbGciOi..." \
  https://service-b.example.com/api/data
```

## рҹҶ” OpenID Connect (OIDC) вҖ” Authentication on Top of OAuth

### What is OIDC?

OIDC (OpenID Connect) is a **thin identity layer on top of OAuth 2.0** that adds **authentication**.

Remember:
- **OAuth 2.0** = Authorization (what you can do)
- **OIDC** = Authentication (who you are) built ON TOP of OAuth 2.0

```
OAuth 2.0 alone:
  "Here's a token that lets you access photos"
  But... who IS this person? рҹӨ·

OAuth 2.0 + OIDC:
  "Here's a token that lets you access photos"
  AND "This person is Alice, email alice@example.com, verified вң…"
```

### What OIDC Adds to OAuth 2.0:

| OAuth 2.0 | OIDC Adds |
|-----------|-----------|
| Access Token (authorization) | **ID Token** (authentication вҖ” who you are) |
| Scopes like `photos.read` | **Standard scopes**: `openid`, `profile`, `email` |
| No user info standard | **/userinfo endpoint** (get user details) |
| No standard for identity | **Standard claims** (name, email, picture, etc.) |

### OIDC Flow вҖ” Step by Step:

The OIDC flow is **almost identical** to the OAuth Authorization Code flow, with one addition вҖ” the **ID Token**.

```
OIDC Flow:

Step 1: Client redirects user to IdP with scope=openid
        https://auth.example.com/authorize?
          response_type=code&
          client_id=my-app&
          redirect_uri=https://myapp.com/callback&
          scope=openid profile email&     вҶҗ OIDC scopes!
          state=random123

Step 2: User logs in at IdP

Step 3: IdP redirects back with authorization code
        https://myapp.com/callback?code=abc123&state=random123

Step 4: Client exchanges code for tokens
        POST https://auth.example.com/token
        {
          "grant_type": "authorization_code",
          "code": "abc123",
          "redirect_uri": "https://myapp.com/callback",
          "client_id": "my-app",
          "client_secret": "my-secret"
        }

Step 5: IdP returns BOTH tokens:
        {
          "access_token": "eyJhbGciOi...",    вҶҗ For accessing APIs
          "id_token": "eyJhbGciOi...",         вҶҗ NEW! For identifying the user
          "token_type": "Bearer",
          "expires_in": 3600,
          "refresh_token": "dGhpcyBpcyBh..."
        }
```

### The ID Token (JWT):

```json
{
  "iss": "https://auth.example.com",     // Issuer (who created this token)
  "sub": "user-12345",                   // Subject (unique user ID)
  "aud": "my-app",                       // Audience (who this token is for)
  "exp": 1635635600,                     // Expiration time
  "iat": 1635632000,                     // Issued at
  "nonce": "random-nonce-123",           // Prevents replay attacks
  
  // Standard OIDC Claims (user info):
  "name": "Alice Johnson",
  "email": "alice@example.com",
  "email_verified": true,
  "picture": "https://example.com/alice.jpg",
  "groups": ["developers", "devops-team"]
}
```

### OIDC Standard Scopes:

| Scope | What Info You Get |
|-------|------------------|
| `openid` | Required. Returns the `sub` (subject/user ID) claim |
| `profile` | name, family_name, given_name, picture, etc. |
| `email` | email, email_verified |
| `address` | Street address, city, country, etc. |
| `phone` | phone_number, phone_number_verified |

## рҹ“Ҡ Comparison вҖ” SAML vs OAuth 2.0 vs OIDC

| Feature | SAML | OAuth 2.0 | OIDC |
|---------|------|-----------|------|
| **Purpose** | SSO / Authentication | Authorization | Authentication (on top of OAuth) |
| **Format** | XML | JSON | JSON (JWT) |
| **Token Type** | SAML Assertion (XML) | Access Token | ID Token + Access Token |
| **Best For** | Enterprise SSO, Legacy apps | API authorization, Mobile apps | Modern SSO, Web & mobile apps |
| **Complexity** | Complex (XML, certificates) | Medium | Medium |
| **Year Created** | 2005 | 2012 | 2014 |
| **Mobile Friendly** | No (XML is heavy) | Yes | Yes |
| **Still Used?** | Yes (enterprise) | Yes (everywhere) | Yes (everywhere) |

### When to Use What:

```
Decision Tree:
в”Ӯ
в”ңв”Җв”Җ Need to integrate with ENTERPRISE apps (Salesforce, Workday, etc.)?
в”Ӯ   в””в”Җв”Җ Use SAML вҶҗ These apps often only support SAML
в”Ӯ
в”ңв”Җв”Җ Need to authorize access to an API?
в”Ӯ   в””в”Җв”Җ Use OAuth 2.0 вҶҗ It's designed for API authorization
в”Ӯ
в”ңв”Җв”Җ Need to authenticate users in a MODERN web/mobile app?
в”Ӯ   в””в”Җв”Җ Use OIDC вҶҗ Modern, JSON-based, supports both AuthN + AuthZ
в”Ӯ
в”ңв”Җв”Җ Need machine-to-machine authentication?
в”Ӯ   в””в”Җв”Җ Use OAuth 2.0 Client Credentials
в”Ӯ
в””в”Җв”Җ Building something new from scratch?
    в””в”Җв”Җ Use OIDC вҶҗ It's the modern standard that does everything
```

## рҹ”„ SCIM вҖ” Automated User Provisioning

SCIM (System for Cross-domain Identity Management) is a protocol for **automatically** creating, updating, and deleting user accounts across systems.

```
WITHOUT SCIM (Manual):
  HR adds Alice to company вҶ’ Admin manually creates accounts in:
  в”ңв”Җв”Җ Okta вң…
  в”ңв”Җв”Җ GitHub вң…
  в”ңв”Җв”Җ Jira вң…
  в”ңв”Җв”Җ AWS вң…
  в””в”Җв”Җ Slack... forgot! вқҢ

WITH SCIM (Automatic):
  HR adds Alice to company вҶ’ Okta AUTOMATICALLY provisions accounts in:
  в”ңв”Җв”Җ GitHub вң… (via SCIM)
  в”ңв”Җв”Җ Jira вң… (via SCIM)
  в”ңв”Җв”Җ AWS вң… (via SCIM)
  в””в”Җв”Җ Slack вң… (via SCIM)
  
  HR removes Alice вҶ’ Okta AUTOMATICALLY deprovisions ALL accounts вң…
```

### SCIM API Example:

```bash
# Create a user via SCIM API
curl -X POST https://api.example.com/scim/v2/Users \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/scim+json" \
  -d '{
    "schemas": ["urn:ietf:params:scim:schemas:core:2.0:User"],
    "userName": "alice@example.com",
    "name": {
      "givenName": "Alice",
      "familyName": "Johnson"
    },
    "emails": [
      {
        "value": "alice@example.com",
        "primary": true
      }
    ],
    "active": true,
    "groups": [
      {
        "value": "developers",
        "display": "Developers"
      }
    ]
  }'

# Deactivate a user (when they leave)
curl -X PATCH https://api.example.com/scim/v2/Users/user-id-123 \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/scim+json" \
  -d '{
    "schemas": ["urn:ietf:params:scim:api:messages:2.0:PatchOp"],
    "Operations": [
      {
        "op": "replace",
        "path": "active",
        "value": false
      }
    ]
  }'
```

## вқҢ Common Mistakes to Avoid

| Mistake | Why It's Wrong | What to Do Instead |
|---------|---------------|-------------------|
| Using long-lived API keys | Keys get leaked, never expire | Use OIDC / short-lived tokens |
| Sharing service accounts | No accountability, can't audit | One service account per service |
| Not implementing MFA | Passwords alone are easy to steal | Enforce MFA for ALL users |
| Over-permissive policies | `"Action": "*", "Resource": "*"` = admin | Least privilege вҖ” only what's needed |
| Not rotating credentials | Old credentials accumulate risk | Auto-rotate or use short-lived tokens |
| Using SAML for new apps | SAML is complex and XML-based | Use OIDC for new applications |
| Not using SCIM | Manual provisioning leads to orphaned accounts | Automate with SCIM |
| Storing tokens in localStorage | Vulnerable to XSS attacks | Use httpOnly cookies or token handlers |
