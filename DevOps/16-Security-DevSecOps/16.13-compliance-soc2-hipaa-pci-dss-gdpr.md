## What is Compliance?

Let's make this super simple.

**Compliance** = Following a set of rules and regulations that govern how you handle data, build systems, and run your business.

Think of it like this:
- You run a restaurant → health inspectors check if you follow food safety rules.
- You run a software company → auditors check if you follow data safety rules.

These "rules" come in the form of **compliance frameworks/standards** like SOC2, HIPAA, PCI-DSS, and GDPR. Each one focuses on a different area — some on financial data, some on health data, some on credit card data, and some on personal privacy.

> 💡 **Key Idea:** Compliance is NOT just a checkbox exercise. It's about building **trustworthy systems** that protect users and their data.

---

## Why Should DevOps Engineers Care About Compliance?

You might think, "I'm an engineer, not a lawyer." Fair point. But here's the reality:

1. **You build and manage the infrastructure** where data lives → you're directly responsible for how it's protected.
2. **CI/CD pipelines** can be audited → if your pipeline deploys unscanned code, YOUR pipeline failed compliance.
3. **Cloud configurations** are your job → a misconfigured S3 bucket can cause a million-dollar compliance violation.
4. **Audit logs, access controls, encryption** → all things YOU set up.
5. **Compliance violations = massive fines:**
   - GDPR: up to €20 million or 4% of global annual revenue
   - HIPAA: up to $1.5 million per violation category per year
   - PCI-DSS: $5,000 to $100,000 per month of non-compliance

> 🎯 **Bottom Line:** Modern compliance is deeply technical. DevOps engineers are on the frontlines.

### 1. SOC 2

#### What is SOC 2?

**SOC 2** stands for **Service Organization Control 2**. It was developed by the **AICPA** (American Institute of Certified Public Accountants).

In the simplest terms: SOC 2 is a **trust framework** that proves to your customers that your company handles their data securely and responsibly.

#### Who Needs SOC 2?

- **Any SaaS company** that stores or processes customer data
- Cloud service providers
- Companies selling to enterprise clients (they'll ask for your SOC 2 report before signing a deal)

#### The Five Trust Service Criteria (TSC)

SOC 2 is built around **five principles** (you don't have to implement all five — Security is mandatory, the rest are optional):

| Principle | What It Means | Example |
|-----------|---------------|---------|
| **Security** (mandatory) | Protect against unauthorized access | Firewalls, MFA, encryption, access controls |
| **Availability** | System is available as promised (SLAs) | Uptime monitoring, disaster recovery, redundancy |
| **Processing Integrity** | Data processing is accurate & complete | Input validation, data integrity checks |
| **Confidentiality** | Sensitive data is protected | Encryption at rest & in transit, access restrictions |
| **Privacy** | Personal information is collected, used, & disposed of properly | Privacy policies, data retention rules, consent |

#### SOC 2 Type I vs Type II

| Type | What It Checks | Duration |
|------|----------------|----------|
| **Type I** | Are the right controls **designed** and in place at a **point in time**? | Snapshot (one day) |
| **Type II** | Are those controls **actually working** over a **period of time**? | Typically 3–12 months |

> 🧠 **Think of it this way:** Type I = "You have a fire extinguisher." Type II = "We watched for 6 months, and the fire extinguisher works and is maintained."

#### SOC 2 for DevOps — What You Actually Need to Do

```
✅ Implement role-based access control (RBAC) on all systems
✅ Enable MFA on all cloud accounts, CI/CD tools, and admin panels
✅ Encrypt data at rest (AES-256) and in transit (TLS 1.2+)
✅ Set up centralized audit logging (who did what, when)
✅ Implement change management (PRs, approvals, no direct pushes to main)
✅ Automate vulnerability scanning in CI/CD pipelines
✅ Monitor systems for availability (uptime SLAs)
✅ Have an incident response plan documented and tested
✅ Perform regular access reviews (who has access to what?)
✅ Backup data regularly and test restore procedures
```

#### Example: Enforcing Change Management via Git

```yaml
# GitHub branch protection rules (via Terraform)
resource "github_branch_protection" "main" {
  repository_id = github_repository.app.node_id
  pattern       = "main"

  # Require pull request reviews before merging
  required_pull_request_reviews {
    required_approving_review_count = 2    # At least 2 approvals
    dismiss_stale_reviews           = true # Re-review after new commits
    require_code_owner_reviews      = true # Code owners must approve
  }

  # Require status checks (CI must pass)
  required_status_checks {
    strict   = true
    contexts = ["ci/build", "ci/test", "security/scan"]
  }

  # No force pushes, no branch deletion
  enforce_admins    = true
  allows_force_pushes = false
  allows_deletions    = false
}
```

**What this does:**
- `required_approving_review_count = 2` → Every change needs 2 human approvals (audit trail!)
- `required_status_checks` → CI must pass before merging (automated quality gate)
- `allows_force_pushes = false` → Nobody can rewrite history (integrity)
- This gives auditors evidence that you have a **controlled change process**

### 2. HIPAA

#### What is HIPAA?

**HIPAA** = **Health Insurance Portability and Accountability Act** (1996, USA).

In simple terms: HIPAA is a **US law** that says, "If you deal with health data, you MUST protect it."

#### Key Terms

| Term | Meaning |
|------|---------|
| **PHI** (Protected Health Information) | Any health data that can identify a person — name, DOB, medical records, prescriptions, insurance info |
| **ePHI** | Electronic PHI — PHI stored or transmitted electronically |
| **Covered Entity** | Organizations that directly deal with PHI — hospitals, clinics, insurance companies |
| **Business Associate** | Companies that handle PHI on behalf of covered entities — cloud providers, SaaS apps |
| **BAA** (Business Associate Agreement) | A legal contract between a covered entity and business associate defining PHI responsibilities |

#### HIPAA Rules

| Rule | What It Covers |
|------|----------------|
| **Privacy Rule** | Who can access PHI, how it can be used and shared |
| **Security Rule** | Technical, physical, and administrative safeguards for ePHI |
| **Breach Notification Rule** | You MUST notify affected individuals within 60 days of a breach |
| **Enforcement Rule** | Penalties for violations |

#### HIPAA for DevOps — What You Actually Need to Do

```
✅ Encrypt ePHI at rest (AES-256) AND in transit (TLS 1.2+)
✅ Implement access controls — only authorized personnel can access PHI
✅ Enable audit logging — track ALL access to PHI (who, when, what)
✅ Use dedicated, isolated environments for PHI workloads
✅ Sign a BAA with your cloud provider (AWS, Azure, GCP all offer BAAs)
✅ Implement automatic session timeouts
✅ Regular vulnerability assessments and penetration testing
✅ Have a documented incident response plan
✅ Backup ePHI and test disaster recovery
✅ Train all employees on HIPAA requirements
✅ Implement unique user identification (no shared accounts!)
```

#### Example: HIPAA-Compliant AWS S3 Bucket

```hcl
# Terraform — HIPAA-compliant S3 bucket for ePHI
resource "aws_s3_bucket" "phi_bucket" {
  bucket = "company-ephi-data"
}

# Enable versioning (data integrity & recovery)
resource "aws_s3_bucket_versioning" "phi_versioning" {
  bucket = aws_s3_bucket.phi_bucket.id
  versioning_configuration {
    status = "Enabled"
  }
}

# Server-side encryption with KMS (encryption at rest)
resource "aws_s3_bucket_server_side_encryption_configuration" "phi_encryption" {
  bucket = aws_s3_bucket.phi_bucket.id

  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm     = "aws:kms"
      kms_master_key_id = aws_kms_key.phi_key.arn
    }
    bucket_key_enabled = true
  }
}

# Block ALL public access (critical for PHI!)
resource "aws_s3_bucket_public_access_block" "phi_block" {
  bucket = aws_s3_bucket.phi_bucket.id

  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

# Enable access logging (audit trail)
resource "aws_s3_bucket_logging" "phi_logging" {
  bucket = aws_s3_bucket.phi_bucket.id

  target_bucket = aws_s3_bucket.log_bucket.id
  target_prefix = "phi-access-logs/"
}

# Lifecycle rule — retain data per HIPAA requirements (6 years)
resource "aws_s3_bucket_lifecycle_configuration" "phi_lifecycle" {
  bucket = aws_s3_bucket.phi_bucket.id

  rule {
    id     = "retain-phi-data"
    status = "Enabled"

    transition {
      days          = 90
      storage_class = "STANDARD_IA"  # Move to cheaper storage after 90 days
    }

    transition {
      days          = 365
      storage_class = "GLACIER"  # Archive after 1 year
    }

    # HIPAA requires retention for 6 years
    expiration {
      days = 2190  # ~6 years
    }
  }
}
```

**What each block does:**
- **Versioning** → You can recover accidentally deleted/modified PHI data
- **KMS Encryption** → ePHI is encrypted using a managed encryption key (not just default S3 encryption)
- **Public Access Block** → Absolutely NO public access — this is non-negotiable for PHI
- **Logging** → Every access to this bucket is logged (audit trail for HIPAA auditors)
- **Lifecycle** → Data is retained for 6 years (HIPAA requirement) and moved to cheaper storage over time

### 3. PCI-DSS

#### What is PCI-DSS?

**PCI-DSS** = **Payment Card Industry Data Security Standard**.

In simple terms: If your system **touches credit card data**, you MUST follow PCI-DSS.

It was created by the **PCI Security Standards Council** (founded by Visa, Mastercard, American Express, Discover, and JCB).

#### Key Terms

| Term | Meaning |
|------|---------|
| **CHD** (Cardholder Data) | Primary Account Number (PAN), cardholder name, expiration date, service code |
| **SAD** (Sensitive Authentication Data) | CVV/CVC, PIN, full magnetic stripe — **NEVER store this after authorization** |
| **CDE** (Cardholder Data Environment) | All systems, networks, and processes that store, process, or transmit CHD |
| **PCI Levels** | 1–4 based on transaction volume (Level 1 = 6M+ transactions/year) |

#### The 12 Requirements of PCI-DSS (v4.0)

PCI-DSS has **12 requirements** organized into **6 goals**:

| Goal | # | Requirement |
|------|---|-------------|
| **Build Secure Network** | 1 | Install and maintain network security controls (firewalls) |
| | 2 | Apply secure configurations to all system components |
| **Protect Account Data** | 3 | Protect stored account data (encryption, masking, tokenization) |
| | 4 | Protect cardholder data with strong cryptography during transmission |
| **Vulnerability Management** | 5 | Protect all systems against malware |
| | 6 | Develop and maintain secure systems and software |
| **Access Control** | 7 | Restrict access to cardholder data by business need-to-know |
| | 8 | Identify users and authenticate access to system components |
| | 9 | Restrict physical access to cardholder data |
| **Monitoring & Testing** | 10 | Log and monitor all access to network resources and cardholder data |
| | 11 | Test security of systems and networks regularly |
| **Security Policy** | 12 | Support information security with organizational policies and programs |

#### PCI-DSS for DevOps — What You Actually Need to Do

```
✅ NEVER store CVV/CVC, PIN, or full magnetic stripe data (ever!)
✅ Encrypt cardholder data at rest using strong encryption (AES-256)
✅ Encrypt in transit (TLS 1.2+, disable older SSL/TLS versions)
✅ Segment your network — isolate CDE from the rest of your infrastructure
✅ Implement strict firewall rules for CDE
✅ Use tokenization wherever possible (replace real card data with tokens)
✅ Restrict access to CDE on a need-to-know basis
✅ Enable comprehensive logging of all access to CDE
✅ Run quarterly vulnerability scans (ASV — Approved Scanning Vendor)
✅ Annual penetration testing
✅ Keep all systems patched and up to date
✅ Implement file integrity monitoring (FIM)
✅ Use MFA for all remote access to CDE
```

#### Example: Network Segmentation with Kubernetes NetworkPolicy

```yaml
# NetworkPolicy — Isolate payment processing pods (CDE)
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: restrict-payment-pods
  namespace: payment-processing   # CDE namespace
spec:
  podSelector:
    matchLabels:
      app: payment-service        # Only applies to payment pods
  policyTypes:
    - Ingress
    - Egress

  # INGRESS: Only allow traffic from the API gateway
  ingress:
    - from:
        - namespaceSelector:
            matchLabels:
              zone: dmz            # Only from DMZ namespace
          podSelector:
            matchLabels:
              app: api-gateway     # Only from the API gateway pod
      ports:
        - protocol: TCP
          port: 8443               # Only on the HTTPS port

  # EGRESS: Only allow traffic to the payment processor and database
  egress:
    - to:
        - podSelector:
            matchLabels:
              app: payment-database   # Allow DB access within CDE
      ports:
        - protocol: TCP
          port: 5432                  # PostgreSQL
    - to:
        - ipBlock:
            cidr: 203.0.113.0/24     # Payment processor's IP range
      ports:
        - protocol: TCP
          port: 443                   # HTTPS to external payment processor
    # Allow DNS resolution
    - to: []
      ports:
        - protocol: UDP
          port: 53
        - protocol: TCP
          port: 53
```

**What this policy does:**
- **`podSelector`** → Targets only the payment service pods
- **Ingress rule** → Only the API gateway (in the DMZ namespace) can talk to payment pods, and only on port 8443
- **Egress rule** → Payment pods can ONLY talk to their database (port 5432) and the external payment processor (on HTTPS)
- **DNS** → Allow DNS so pods can resolve service names
- Everything else is **blocked by default** — this is network segmentation!

#### Pro Tip: Use a Payment Processor

The best PCI-DSS strategy? **Don't handle card data yourself.**

Use a third-party payment processor like **Stripe, PayPal, Braintree, or Adyen**. They handle the card data, and you only deal with tokens. This dramatically reduces your PCI scope.

```
Without Stripe: You handle card numbers → Full PCI-DSS compliance needed (12 requirements)
With Stripe:    Stripe handles cards, you get tokens → SAQ-A (simplest compliance level)
```

### 4. GDPR

#### What is GDPR?

**GDPR** = **General Data Protection Regulation** (EU, 2018).

In simple terms: GDPR is an **EU law** that gives people control over their personal data and tells companies, "Respect people's privacy or pay a LOT of money."

> 🌍 GDPR applies to **any company** that processes data of **EU residents**, regardless of where the company is located. A US company with EU users? You need GDPR compliance.

#### Key Terms

| Term | Meaning |
|------|---------|
| **Personal Data** | Any data that can identify a person — name, email, IP address, location, cookies, etc. |
| **Data Subject** | The person whose data is being processed (your users) |
| **Data Controller** | The entity that decides WHY and HOW personal data is processed (usually you) |
| **Data Processor** | The entity that processes data ON BEHALF of the controller (e.g., AWS, a SaaS tool) |
| **DPO** (Data Protection Officer) | A designated person responsible for GDPR compliance in the organization |
| **DPA** (Data Processing Agreement) | Contract between controller and processor defining data responsibilities |

#### Key GDPR Principles

| Principle | What It Means |
|-----------|---------------|
| **Lawfulness, Fairness, Transparency** | You must have a legal basis to process data, and users must know about it |
| **Purpose Limitation** | Collect data only for specific, stated purposes |
| **Data Minimization** | Only collect data you actually need |
| **Accuracy** | Keep data accurate and up to date |
| **Storage Limitation** | Don't keep data longer than necessary |
| **Integrity & Confidentiality** | Protect data with appropriate security measures |
| **Accountability** | You must be able to PROVE you're following these principles |

#### Key User Rights Under GDPR

| Right | What It Means | DevOps Impact |
|-------|---------------|---------------|
| **Right to Access** | Users can request a copy of all their data | You need an API/process to export user data |
| **Right to Rectification** | Users can request correction of their data | Your systems must allow data updates |
| **Right to Erasure** ("Right to be Forgotten") | Users can request deletion of their data | You need to delete data from ALL systems, backups, logs |
| **Right to Data Portability** | Users can request their data in a machine-readable format | Export data as JSON, CSV, etc. |
| **Right to Object** | Users can object to data processing | Opt-out mechanisms |
| **Right to Restrict Processing** | Users can limit how their data is used | Feature flags/controls in your app |

#### GDPR for DevOps — What You Actually Need to Do

```
✅ Know what personal data you collect, where it's stored, and who has access
✅ Implement data encryption at rest and in transit
✅ Build mechanisms for data export, deletion, and rectification
✅ Implement consent management (users must opt-in, not opt-out)
✅ Log data access and processing activities
✅ Sign DPAs with all third-party processors (AWS, Datadog, etc.)
✅ Implement data retention policies — auto-delete old data
✅ Anonymize or pseudonymize data wherever possible
✅ Report data breaches to authorities within 72 hours
✅ Conduct Data Protection Impact Assessments (DPIA) for high-risk processing
✅ Ensure data residency — EU data stays in EU regions
```

#### Example: GDPR Data Deletion API (Right to Erasure)

```python
# Flask API endpoint for GDPR Right to Erasure
from flask import Flask, request, jsonify
from datetime import datetime
import logging

app = Flask(__name__)

# Audit logger for compliance evidence
audit_logger = logging.getLogger('gdpr_audit')
handler = logging.FileHandler('gdpr_audit.log')
handler.setFormatter(logging.Formatter(
    '%(asctime)s - %(message)s'
))
audit_logger.addHandler(handler)
audit_logger.setLevel(logging.INFO)


@app.route('/api/v1/users/<user_id>/data', methods=['DELETE'])
def delete_user_data(user_id):
    """
    GDPR Article 17 — Right to Erasure
    Deletes ALL personal data for a given user across all systems.
    """
    # Step 1: Authenticate the request (verify it's the actual user or authorized)
    auth_token = request.headers.get('Authorization')
    if not verify_user_identity(auth_token, user_id):
        return jsonify({"error": "Unauthorized"}), 401

    # Step 2: Log the deletion request (audit trail)
    audit_logger.info(
        f"GDPR_ERASURE_REQUEST | user_id={user_id} | "
        f"requested_at={datetime.utcnow().isoformat()} | "
        f"ip={request.remote_addr}"
    )

    try:
        # Step 3: Delete from primary database
        delete_from_database(user_id)

        # Step 4: Delete from search indexes (Elasticsearch, etc.)
        delete_from_search_index(user_id)

        # Step 5: Delete from cache (Redis, Memcached)
        delete_from_cache(user_id)

        # Step 6: Delete from object storage (S3 files, uploads)
        delete_from_object_storage(user_id)

        # Step 7: Delete from analytics systems
        delete_from_analytics(user_id)

        # Step 8: Request deletion from third-party processors
        notify_third_party_processors(user_id)

        # Step 9: Queue backup purge (async — backups take time)
        queue_backup_purge(user_id)

        # Step 10: Log successful deletion
        audit_logger.info(
            f"GDPR_ERASURE_COMPLETED | user_id={user_id} | "
            f"completed_at={datetime.utcnow().isoformat()}"
        )

        return jsonify({
            "status": "success",
            "message": "All personal data has been deleted",
            "user_id": user_id,
            "deleted_at": datetime.utcnow().isoformat(),
            "note": "Backup purge is queued and will complete within 30 days"
        }), 200

    except Exception as e:
        audit_logger.error(
            f"GDPR_ERASURE_FAILED | user_id={user_id} | error={str(e)}"
        )
        return jsonify({"error": "Deletion failed. Our team has been notified."}), 500
```

**What this code does:**
- **Authentication** → Verifies the person requesting deletion is authorized
- **Audit Logging** → Logs every step for compliance evidence (auditors love this)
- **Multi-system deletion** → Deletes from DB, cache, search, storage, analytics — GDPR requires deletion from ALL systems
- **Third-party notification** → Tells your processors (Stripe, Mailchimp, etc.) to delete the user's data too
- **Backup purge** → Queues backup cleanup (this is usually async because backups are immutable)

#### Example: Terraform — Ensure Data Stays in EU

```hcl
# Force all resources to be created in EU regions only
provider "aws" {
  region = "eu-west-1"   # Ireland

  # Deny any resource creation outside EU
  default_tags {
    tags = {
      data-residency = "EU"
      compliance     = "GDPR"
      environment    = "production"
    }
  }
}

# S3 bucket with EU-only replication
resource "aws_s3_bucket" "user_data" {
  bucket = "company-eu-user-data"

  # Bucket is in eu-west-1 (set by provider)
}

# SCP (Service Control Policy) — Prevent resource creation outside EU
# Apply this at the AWS Organization level
resource "aws_organizations_policy" "eu_only" {
  name        = "restrict-to-eu-regions"
  description = "GDPR: Restrict resource creation to EU regions only"
  type        = "SERVICE_CONTROL_POLICY"

  content = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Sid       = "DenyNonEURegions"
        Effect    = "Deny"
        Action    = "*"
        Resource  = "*"
        Condition = {
          StringNotEquals = {
            "aws:RequestedRegion" = [
              "eu-west-1",      # Ireland
              "eu-west-2",      # London
              "eu-west-3",      # Paris
              "eu-central-1",   # Frankfurt
              "eu-north-1",     # Stockholm
              "eu-south-1"      # Milan
            ]
          }
        }
      }
    ]
  })
}
```

**What this does:**
- **Provider region** → All resources default to `eu-west-1` (Ireland)
- **SCP** → At the AWS Organization level, this **blocks anyone** from creating resources outside EU regions
- **Tags** → Every resource is tagged for compliance tracking
- This ensures EU user data never leaves the EU — a core GDPR requirement

Here's how compliance fits into your CI/CD pipeline:

```
│   │         │         │         │             │              │       │
│   │         │         │         │             │              │       │
```

#### Example: Compliance Gate in GitHub Actions

```yaml
# .github/workflows/compliance-pipeline.yml
name: Compliance-Aware CI/CD

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  security-scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      # SAST — Static code analysis
      - name: Run Semgrep (SAST)
        uses: returntocorp/semgrep-action@v1
        with:
          config: >-
            p/owasp-top-ten
            p/secrets

      # SCA — Check for vulnerable dependencies
      - name: Run Trivy (SCA)
        uses: aquasecurity/trivy-action@master
        with:
          scan-type: 'fs'
          severity: 'HIGH,CRITICAL'
          exit-code: '1'        # Fail the pipeline on high/critical vulnerabilities

  compliance-check:
    runs-on: ubuntu-latest
    needs: security-scan         # Only run after security passes
    steps:
      - uses: actions/checkout@v4

      # IaC compliance check
      - name: Run Checkov (IaC Compliance)
        uses: bridgecrewio/checkov-action@v12
        with:
          directory: ./terraform
          framework: terraform
          check: >-
            CKV_AWS_18,CKV_AWS_19,CKV_AWS_21,CKV_AWS_145
          # CKV_AWS_18: Ensure S3 access logging is enabled
          # CKV_AWS_19: Ensure S3 encryption is enabled
          # CKV_AWS_21: Ensure S3 versioning is enabled
          # CKV_AWS_145: Ensure S3 uses KMS encryption

      # Generate compliance evidence
      - name: Generate SBOM
        uses: anchore/sbom-action@v0
        with:
          format: spdx-json
          output-file: sbom.spdx.json

      - name: Upload Compliance Artifacts
        uses: actions/upload-artifact@v4
        with:
          name: compliance-evidence
          path: |
            sbom.spdx.json
            checkov-report.json

  deploy:
    runs-on: ubuntu-latest
    needs: [security-scan, compliance-check]  # Both must pass!
    if: github.ref == 'refs/heads/main'
    steps:
      - name: Deploy to Production
        run: |
          echo "Deploying — all compliance checks passed!"
          # Your deployment commands here
```

**What this pipeline does:**
1. **Security scanning** → SAST (code vulnerabilities) and SCA (dependency vulnerabilities)
2. **Compliance check** → Validates infrastructure code against compliance rules
3. **SBOM generation** → Creates a Software Bill of Materials (supply chain compliance)
4. **Gated deployment** → Deploy ONLY if both security AND compliance checks pass

### Example: OPA Policy for PCI-DSS

```rego
# policy/pci_dss.rego
# OPA policy to enforce PCI-DSS requirements on Kubernetes deployments

package pci_dss

# DENY: Containers must not run as root (PCI-DSS Req 7 & 8)
deny[msg] {
    input.kind == "Deployment"
    container := input.spec.template.spec.containers[_]
    not container.securityContext.runAsNonRoot
    msg := sprintf(
        "PCI-DSS Violation: Container '%s' must set runAsNonRoot=true",
        [container.name]
    )
}

# DENY: All containers must have resource limits (availability)
deny[msg] {
    input.kind == "Deployment"
    container := input.spec.template.spec.containers[_]
    not container.resources.limits.memory
    msg := sprintf(
        "PCI-DSS Violation: Container '%s' must have memory limits set",
        [container.name]
    )
}

# DENY: Images must come from approved registries only
deny[msg] {
    input.kind == "Deployment"
    container := input.spec.template.spec.containers[_]
    not startswith(container.image, "company-registry.example.com/")
    msg := sprintf(
        "PCI-DSS Violation: Container '%s' uses unapproved registry. Image: %s",
        [container.name, container.image]
    )
}

# DENY: Sensitive namespaces must have network policies
deny[msg] {
    input.kind == "Deployment"
    input.metadata.namespace == "payment-processing"
    not input.metadata.annotations["network-policy-enforced"]
    msg := "PCI-DSS Violation: Payment namespace must have network policies enforced"
}
```

## Audit Logging & Evidence Collection

Auditors want **evidence**. They want to see logs, screenshots, reports. Here's how to automate evidence collection:

### Centralized Audit Logging Architecture

```
┌─────────────┐  ┌──────────────┐  ┌──────────────┐
│  App Logs   │  │ Infra Logs   │  │  API Logs    │
│  (stdout)   │  │ (CloudTrail) │  │  (Access)    │
└──────┬──────┘  └──────┬───────┘  └──────┬───────┘
       │                │                  │
       ▼                ▼                  ▼
┌──────────────────────────────────────────────────┐
│           Centralized Log Aggregation            │
│      (ELK / Loki / CloudWatch / Splunk)          │
├──────────────────────────────────────────────────┤
│  • Immutable storage (WORM — Write Once Read     │
│    Many)                                          │
│  • Retention: 1-7 years depending on standard    │
│  • Tamper-proof (checksums, signed logs)          │
│  • Access restricted to auditors & security team │
└──────────────────────────────────────────────────┘
```

### What to Log (for ALL compliance frameworks)

```
✅ Authentication events (login, logout, failed login)
✅ Authorization events (access granted, access denied)
✅ Data access (who read/wrote what data, when)
✅ System changes (config changes, deployments, scaling events)
✅ Admin actions (user creation, permission changes, key rotation)
✅ Security events (firewall blocks, vulnerability scan results)
✅ API access (who called what endpoint, with what payload)
```

### Example: AWS CloudTrail for Audit Logging

```hcl
# Enable CloudTrail for audit logging
resource "aws_cloudtrail" "compliance_trail" {
  name                          = "compliance-audit-trail"
  s3_bucket_name                = aws_s3_bucket.audit_logs.id
  include_global_service_events = true    # IAM, CloudFront, etc.
  is_multi_region_trail         = true    # Log ALL regions
  enable_log_file_validation    = true    # Detect log tampering!

  # Log data events (S3 access, Lambda invocations)
  event_selector {
    read_write_type           = "All"
    include_management_events = true

    data_resource {
      type   = "AWS::S3::Object"
      values = ["arn:aws:s3:::company-ephi-data/"]  # Log access to PHI bucket
    }
  }

  # Send logs to CloudWatch for real-time alerting
  cloud_watch_logs_group_arn = "${aws_cloudwatch_log_group.audit.arn}:*"
  cloud_watch_logs_role_arn  = aws_iam_role.cloudtrail_cloudwatch.arn

  tags = {
    compliance = "SOC2,HIPAA,PCI-DSS,GDPR"
    retention  = "7-years"
  }
}

# Immutable log storage (WORM — can't delete or modify)
resource "aws_s3_bucket_object_lock_configuration" "audit_lock" {
  bucket = aws_s3_bucket.audit_logs.id

  rule {
    default_retention {
      mode = "COMPLIANCE"    # Even root can't delete!
      years = 7              # Keep for 7 years
    }
  }
}
```

**What this does:**
- **Multi-region trail** → Captures API calls from ALL AWS regions
- **Log file validation** → Detects if anyone tampers with log files
- **Data events** → Logs every S3 access to the PHI bucket
- **CloudWatch integration** → Real-time alerting on suspicious activity
- **Object Lock (WORM)** → Logs CANNOT be deleted or modified for 7 years — even by root!

## Common Compliance Pitfalls

| Pitfall | Why It's Bad | How to Fix It |
|---------|-------------|---------------|
| Treating compliance as a one-time project | Compliance is **continuous**, not a checkbox | Automate checks in CI/CD |
| Shared/root accounts | Destroys accountability | Unique accounts + MFA + RBAC |
| Unencrypted data | Violation of almost every standard | Encrypt EVERYTHING (at rest + in transit) |
| No audit logs | You can't prove anything to auditors | Centralized, immutable logging |
| Overly permissive access | Violates least privilege | Regular access reviews, automated policies |
| Storing sensitive data you don't need | More data = more risk = more compliance burden | Data minimization, auto-purge |
| Manual compliance checks | Humans miss things, don't scale | Compliance as Code (OPA, Checkov) |
| Forgetting backups in deletion requests | GDPR violation (right to erasure) | Include backups in data lifecycle management |
| No incident response plan | Breach notification deadlines will be missed | Document, test, and drill the plan |

## Real-World Compliance Checklist for DevOps Teams

```
## Universal Compliance Checklist (Covers SOC2 + HIPAA + PCI-DSS + GDPR)

### Access Control
[ ] RBAC implemented on all systems
[ ] MFA enforced on all accounts
[ ] Unique user accounts (no shared credentials)
[ ] Regular access reviews (quarterly)
[ ] Least privilege principle enforced
[ ] Admin access is time-limited (just-in-time access)

### Encryption
[ ] Data encrypted at rest (AES-256)
[ ] Data encrypted in transit (TLS 1.2+)
[ ] Encryption keys managed securely (KMS)
[ ] Key rotation enabled (annual at minimum)

### Logging & Monitoring
[ ] Centralized audit logging enabled
[ ] Logs are immutable (WORM storage)
[ ] Log retention meets requirements (1-7 years)
[ ] Real-time alerting on security events
[ ] Log integrity validation enabled

### Change Management
[ ] All changes go through version control (Git)
[ ] Pull requests with required approvals
[ ] CI/CD pipeline with security gates
[ ] No direct access to production systems
[ ] Deployment rollback capability

### Data Protection
[ ] Data classification in place (public, internal, confidential, restricted)
[ ] Data retention policies defined and automated
[ ] Data deletion/purge processes implemented
[ ] Backup & disaster recovery tested regularly

### Vulnerability Management
[ ] Automated vulnerability scanning in CI/CD
[ ] Regular penetration testing (annual)
[ ] Patch management process in place
[ ] Container image scanning enabled

### Incident Response
[ ] Incident response plan documented
[ ] Breach notification procedures defined
[ ] Regular incident response drills
[ ] Post-incident review process (blameless postmortem)

### Network Security
[ ] Network segmentation implemented
[ ] Firewall rules reviewed regularly
[ ] No unnecessary ports open
[ ] VPN/private connectivity for sensitive systems
```

## Summary

| Concept | Key Takeaway |
|---------|-------------|
| **Compliance** | Following rules that govern data handling — NOT optional |
| **SOC 2** | Trust framework for SaaS — prove you handle customer data securely |
| **HIPAA** | US law protecting health data — encrypt ePHI, sign BAAs, audit everything |
| **PCI-DSS** | Protect credit card data — 12 requirements, network segmentation, or use tokenization |
| **GDPR** | EU privacy law — user rights (access, delete, port), 72-hour breach notification |
| **Compliance as Code** | Automate compliance checks with OPA, Checkov, Kyverno |
| **Audit Logging** | Immutable, centralized logs are your best friend during audits |
| **Encryption** | Encrypt everything, everywhere, always |
| **Key Strategy** | Shift compliance left — build it into CI/CD, not as an afterthought |

> 🔑 **The Golden Rule of Compliance:** If you can't prove it, it didn't happen. Automate evidence collection, and audits become easy.
