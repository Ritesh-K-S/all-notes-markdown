## What is Penetration Testing?

Let's break it down in the simplest way possible.

**Penetration Testing (Pen Testing)** = Trying to hack your own systems **before the bad guys do**.

Think of it like this:
- You have a house with doors, windows, and a lock.
- A pen tester is like a **friendly burglar** you hire to try to break in.
- If they get in, they tell you HOW, and you fix the weakness.
- If they can't get in, you know your defenses are solid.

> 💡 **Key Point:** Pen testing is **authorized** hacking. You have written permission to test the target. Without permission, it's just... hacking (and it's illegal).

### Pen Testing vs Vulnerability Scanning

These are often confused. Here's the difference:

| Feature | Vulnerability Scanning | Penetration Testing |
|---------|----------------------|---------------------|
| **What it does** | Automatically finds known vulnerabilities | Manually + automatically tries to **exploit** vulnerabilities |
| **Depth** | Surface-level (finds issues) | Deep (proves impact) |
| **Automation** | Fully automated | Mix of automated + manual |
| **Output** | List of potential vulnerabilities | Proof of exploitation + business impact |
| **Frequency** | Continuous / weekly | Quarterly / annually |
| **Analogy** | Checking if doors are unlocked | Actually walking through the unlocked door |

> 🎯 **Think of it this way:** A vulnerability scan says "your door might be unlocked." A pen test opens the door, walks in, and takes a photo of your TV to prove it.

## Why DevOps Engineers Need to Know Pen Testing

You might think, "I'm a DevOps engineer, not a security researcher." Here's why it matters:

1. **You build the infrastructure** → You need to know what attackers look for
2. **You manage CI/CD pipelines** → Pipelines are a prime attack vector
3. **You configure cloud resources** → Misconfigurations are the #1 cause of breaches
4. **You handle secrets** → Leaked credentials are every hacker's dream
5. **Compliance requires it** → SOC 2, PCI-DSS, HIPAA all mandate regular pen testing
6. **Shift-left security** → You can catch vulnerabilities BEFORE they reach production

> 🧠 **You don't need to be a pen testing expert**, but you DO need to understand the basics so you can:
> - Build more secure systems from the start
> - Understand pen test reports and fix the findings
> - Automate security testing in your pipelines
> - Have informed conversations with your security team

## Types of Penetration Testing

### By Knowledge Level (How much the tester knows)

| Type | Tester's Knowledge | Real-World Analogy |
|------|-------------------|-------------------|
| **Black Box** | Tester knows NOTHING about the target | Testing like an external hacker |
| **White Box** | Tester has FULL access (source code, architecture, credentials) | Testing like an insider/developer |
| **Gray Box** | Tester has PARTIAL knowledge (some credentials, basic architecture) | Testing like a user with limited access |

### By Target (What is being tested)

| Type | What's Tested | DevOps Relevance |
|------|---------------|-----------------|
| **Network Pen Test** | Firewalls, routers, switches, network services | High — you manage the network |
| **Web Application Pen Test** | Web apps, APIs, authentication | Medium — you deploy the apps |
| **Cloud Pen Test** | AWS/Azure/GCP configurations, IAM, storage | Very High — you configure the cloud |
| **Container Pen Test** | Docker images, Kubernetes clusters | Very High — you manage containers |
| **CI/CD Pipeline Pen Test** | Build systems, artifact repos, deployment processes | Very High — you own the pipeline |
| **Social Engineering** | Phishing, pretexting, physical access | Low — not usually DevOps scope |
| **Wireless Pen Test** | WiFi networks | Low — not usually DevOps scope |

| Methodology | What It Is | Best For |

### Phase 1: Reconnaissance

**Goal:** Gather as much information as possible about the target WITHOUT directly touching it.

This is like a burglar driving past your house, noting the type of locks, alarm system, and when you leave for work.

#### Passive Reconnaissance (No direct interaction)

```bash
# Find subdomains using public sources
# These are OSINT (Open Source Intelligence) tools

# 1. DNS enumeration — find all subdomains
# Using 'dig' to query DNS records
dig example.com ANY
dig example.com MX
dig example.com TXT
dig example.com NS

# 2. Find subdomains from certificate transparency logs
# https://crt.sh — free public tool
curl -s "https://crt.sh/?q=%25.example.com&output=json" | \
  jq -r '.[].name_value' | sort -u

# 3. Check what technologies a website uses
# Wappalyzer (browser extension) or WhatWeb (CLI)
whatweb example.com

# 4. Find exposed files in search engines (Google Dorking)
# These are Google search queries — NOT commands to run on a target
# site:example.com filetype:env          → Find exposed .env files
# site:example.com filetype:sql          → Find database dumps
# site:example.com inurl:admin           → Find admin panels
# site:example.com "password" filetype:log → Find password leaks

# 5. Search for leaked credentials or code
# Check if company repos accidentally leaked secrets
# truffleHog — finds secrets in Git history
trufflehog git https://github.com/company/repo --only-verified

# 6. WHOIS — find domain registration info
whois example.com

# 7. Check for email addresses and social media (theHarvester)
theHarvester -d example.com -b google,linkedin,github
```

**What each command does:**
- `dig` → Queries DNS to find mail servers, name servers, etc.
- `crt.sh` → Searches SSL certificate transparency logs to find subdomains
- `whatweb` → Identifies what tech stack a website uses (Nginx? WordPress? React?)
- `trufflehog` → Scans Git repos for accidentally committed secrets (API keys, passwords)
- `whois` → Shows who registered the domain and when
- `theHarvester` → Collects email addresses and usernames from public sources

#### Active Reconnaissance (Direct interaction — requires permission!)

```bash
# Ping sweep — find live hosts on a network
nmap -sn 192.168.1.0/24

# This sends ICMP packets to every IP in the range
# Output shows which hosts are up (alive)
```

### Phase 2: Scanning & Enumeration

**Goal:** Find open doors (ports, services, vulnerabilities) on the target.

#### Port Scanning with Nmap

Nmap is the **king of network scanning**. Every DevOps/security person should know it.

```bash
# Basic port scan — find open ports
nmap -sV 192.168.1.100

# Breakdown:
# -sV = detect service version (not just port, but what's running on it)
# Output example:
# PORT     STATE SERVICE     VERSION
# 22/tcp   open  ssh         OpenSSH 8.9p1
# 80/tcp   open  http        nginx 1.18.0
# 443/tcp  open  ssl/http    nginx 1.18.0
# 3306/tcp open  mysql       MySQL 8.0.28
# 6379/tcp open  redis       Redis 6.2.6

# Aggressive scan (OS detection, scripts, traceroute)
nmap -A -T4 192.168.1.100

# Scan all 65535 ports (not just common ones)
nmap -p- 192.168.1.100

# Scan for specific vulnerabilities using Nmap scripts
nmap --script vuln 192.168.1.100

# Scan a range of IPs
nmap -sV 192.168.1.0/24

# Output to file for later analysis
nmap -sV -oN scan_results.txt 192.168.1.100
```

**Understanding the output:**

```
PORT     STATE  SERVICE
22/tcp   open   ssh        → SSH is accessible (check for weak auth)
80/tcp   open   http       → Web server running (check for web vulns)
443/tcp  open   https      → HTTPS (check SSL/TLS config)
3306/tcp open   mysql      → Database exposed! (should NOT be public)
6379/tcp open   redis      → Redis exposed! (often no auth — critical finding!)
8080/tcp open   http-proxy → Could be Jenkins, Tomcat, etc.
9090/tcp open   prometheus → Prometheus metrics exposed (data leak!)
```

> 🚨 **DevOps Red Flags:** Databases (MySQL, PostgreSQL, Redis, MongoDB), monitoring tools (Prometheus, Grafana), and admin panels should NEVER be exposed to the internet.

#### Web Application Scanning

```bash
# Nikto — web server vulnerability scanner
nikto -h http://example.com

# What Nikto finds:
# - Outdated server software
# - Default files & configurations
# - Dangerous HTTP methods (PUT, DELETE)
# - Missing security headers
# - Known vulnerabilities

# dirb / gobuster — find hidden directories and files
gobuster dir -u http://example.com -w /usr/share/wordlists/dirb/common.txt

# This brute-forces common paths like:
# /admin, /backup, /config, /.env, /api, /swagger, /debug
# If any exist, they might expose sensitive functionality

# Example output:
# /admin          (Status: 302) [Size: 0] [--> /admin/login]
# /api            (Status: 200) [Size: 1234]
# /backup         (Status: 200) [Size: 5678]  ← DANGER!
# /.env           (Status: 200) [Size: 432]   ← CRITICAL! Secrets exposed!
# /metrics        (Status: 200) [Size: 8901]  ← Prometheus metrics exposed
```

#### SSL/TLS Testing

```bash
# testssl.sh — comprehensive SSL/TLS testing
testssl.sh example.com

# Checks for:
# - Weak cipher suites
# - Expired certificates
# - SSL/TLS version support (SSLv3, TLS 1.0, 1.1 should be DISABLED)
# - Known vulnerabilities (Heartbleed, POODLE, BEAST, etc.)
# - HSTS header presence
# - Certificate chain issues

# Quick SSL check with nmap
nmap --script ssl-enum-ciphers -p 443 example.com

# Check specific TLS version support
openssl s_client -connect example.com:443 -tls1_2
openssl s_client -connect example.com:443 -tls1_3
```

### Phase 3: Exploitation

**Goal:** Actually exploit the vulnerabilities found to prove they're real.

> ⚠️ **IMPORTANT:** Only do this with written authorization! Exploitation without permission is illegal.

#### Common Exploitation Scenarios for DevOps

##### 1. Exploiting Exposed Services

```bash
# Scenario: Redis exposed without authentication
# Found: Port 6379 open with no password

# Connect to Redis
redis-cli -h target-ip

# What an attacker can do:
# 1. Read all data
KEYS *
GET user:1:session
GET user:admin:password

# 2. Write a cron job to get a reverse shell (on Linux hosts)
# This is a well-known Redis exploitation technique
# DO NOT use this on systems you don't own!

# 3. Delete all data
FLUSHALL    # Devastating!

# FIX: Never expose Redis to the internet. Use:
# - Firewall rules (only allow localhost or app servers)
# - Redis AUTH password
# - TLS encryption
# - Network segmentation
```

##### 2. Exploiting Default Credentials

```bash
# Common default credentials that pen testers check:

# Jenkins:     admin / admin
# Grafana:     admin / admin
# Kibana:      elastic / changeme
# MongoDB:     (no auth by default!)
# PostgreSQL:  postgres / postgres
# Tomcat:      tomcat / tomcat  or  admin / admin
# ArgoCD:      admin / (initial password in secret)

# Tool to test: Hydra (brute force login)
hydra -l admin -P /usr/share/wordlists/rockyou.txt \
  target-ip http-post-form \
  "/login:username=^USER^&password=^PASS^:Invalid credentials"

# Breakdown:
# -l admin         → Try username "admin"
# -P rockyou.txt   → Try passwords from this wordlist
# http-post-form   → It's a web login form
# /login           → The login URL
# ^USER^ / ^PASS^  → Placeholders for username/password
# "Invalid..."     → Text that appears on failed login

# FIX: 
# - Change ALL default passwords immediately
# - Enforce strong password policies
# - Use MFA everywhere
# - Implement account lockout after failed attempts
```

##### 3. Exploiting Misconfigured S3 Buckets

```bash
# Check if an S3 bucket allows public access
aws s3 ls s3://target-bucket --no-sign-request

# If this works without credentials, the bucket is PUBLIC!
# This means anyone can read the data

# Check if you can write to it
aws s3 cp test.txt s3://target-bucket/ --no-sign-request

# If this works, anyone can upload files to the bucket
# This could lead to data poisoning or hosting malicious files

# Enumerate S3 buckets for a company
# Common naming patterns:
# company-name-backup
# company-name-logs
# company-name-dev
# company-name-staging
# company-name-assets

# FIX:
# - Block public access at the account level
# - Use bucket policies with explicit deny for public access
# - Enable S3 access logging
# - Use AWS Config rules to detect public buckets
```

##### 4. Exploiting Kubernetes Misconfigurations

```bash
# If you gain access to a pod, check for Kubernetes API access
# This is a common post-exploitation scenario

# 1. Check if service account token exists
cat /var/run/secrets/kubernetes.io/serviceaccount/token

# 2. Try to access the Kubernetes API
KUBE_TOKEN=$(cat /var/run/secrets/kubernetes.io/serviceaccount/token)
curl -sk https://kubernetes.default.svc/api/v1/namespaces \
  -H "Authorization: Bearer $KUBE_TOKEN"

# If this returns namespaces, you have cluster access from inside a pod!

# 3. Try to list secrets
curl -sk https://kubernetes.default.svc/api/v1/secrets \
  -H "Authorization: Bearer $KUBE_TOKEN"

# If this works, you can read ALL secrets in the cluster!

# FIX:
# - Use RBAC — don't give pods cluster-admin
# - Disable automounting of service account tokens when not needed:
#   automountServiceAccountToken: false
# - Use network policies to restrict pod-to-API-server access
# - Use OPA/Gatekeeper to enforce security policies
```

### Phase 4: Post-Exploitation

**Goal:** Determine the full impact of the exploit. What else can the attacker reach?

```
After gaining initial access, pen testers check:

1. LATERAL MOVEMENT
   - Can I reach other servers from here?
   - Can I access other namespaces/clusters?
   - Can I pivot to databases or internal services?

2. PRIVILEGE ESCALATION
   - Can I become root/admin?
   - Can I access more sensitive data?
   - Can I modify system configurations?

3. DATA ACCESS
   - What sensitive data can I access?
   - Customer data? Credentials? Source code?
   - Can I exfiltrate data without detection?

4. PERSISTENCE
   - Could an attacker maintain access if this were real?
   - Are there backdoors or weak points that allow re-entry?
```

#### Example: Checking for Privilege Escalation in Linux

```bash
# These are commands you'd run AFTER gaining access to a system
# Used to check if you can escalate privileges

# 1. Check current user and groups
whoami
id
groups

# 2. Check sudo permissions
sudo -l
# Output: "(ALL : ALL) NOPASSWD: ALL" → You can run ANYTHING as root!

# 3. Find SUID binaries (run as owner, not current user)
find / -type f -perm -4000 2>/dev/null
# Look for unusual SUID binaries that could be exploited

# 4. Check for writable sensitive files
find /etc -writable 2>/dev/null
ls -la /etc/passwd /etc/shadow /etc/crontab

# 5. Check for credentials in environment variables
env | grep -i "pass\|key\|secret\|token\|api"
# You'd be shocked how often secrets are in environment variables!

# 6. Check for credentials in files
find / -name "*.env" -o -name "*.config" -o -name "*.yml" 2>/dev/null | \
  xargs grep -l "password\|secret\|api_key" 2>/dev/null

# 7. Check running processes for exposed credentials
ps aux | grep -i "password\|secret"

# 8. Check Docker socket access
ls -la /var/run/docker.sock
# If accessible, you can escape the container!
docker run -v /:/host -it alpine chroot /host /bin/sh
# This mounts the HOST filesystem into a new container → full host access!

# FIX:
# - Minimize SUID binaries
# - Never run containers as root
# - Never mount Docker socket into containers
# - Use read-only file systems
# - Store secrets in Vault/KMS, not env vars or files
```

### Phase 5: Reporting & Remediation

**Goal:** Document everything found and help fix it.

#### Pen Test Report Structure

```
1. EXECUTIVE SUMMARY
   - High-level overview for leadership
   - Overall risk rating (Critical, High, Medium, Low)
   - Key findings count

2. SCOPE & METHODOLOGY
   - What was tested
   - What was NOT tested (out of scope)
   - Tools and techniques used
   - Testing dates

3. FINDINGS (for each vulnerability)
   - Title: Descriptive name
   - Severity: Critical / High / Medium / Low / Informational
   - CVSS Score: Numerical risk score (0-10)
   - Description: What the issue is
   - Impact: What could happen if exploited
   - Evidence: Screenshots, commands, logs (proof!)
   - Remediation: How to fix it
   - References: CVE numbers, documentation links

4. REMEDIATION PRIORITY MATRIX
   - Quick wins (easy fix, high impact)
   - Short-term (fix within 30 days)
   - Medium-term (fix within 90 days)
   - Long-term (architectural changes)

5. APPENDIX
   - Full tool output
   - Raw scan results
   - Network diagrams
```

#### Example: Finding Entry in a Report

```markdown
### Finding #3: Redis Server Exposed Without Authentication

| Attribute | Detail |
|-----------|--------|
| **Severity** | CRITICAL |
| **CVSS Score** | 9.8 |
| **Affected System** | redis-prod-01 (10.0.1.50:6379) |
| **Category** | Network / Authentication |

**Description:**
The Redis server at 10.0.1.50 is accessible on port 6379 without any
authentication. Any user on the network can connect and execute arbitrary
Redis commands, including reading all stored data and flushing the database.

**Impact:**
- Complete data theft (session tokens, cached credentials, application data)
- Data destruction (FLUSHALL command)
- Remote code execution via Redis module loading
- Potential pivot to other internal systems

**Evidence:**
```
$ redis-cli -h 10.0.1.50
10.0.1.50:6379> KEYS *
1) "session:admin:abc123"
2) "user:1:profile"
3) "cache:api-key:stripe"
10.0.1.50:6379> GET cache:api-key:stripe
"sk_live_xxxxxxxxxxxxx"
```

**Remediation:**
1. Immediately: Add authentication — `requirepass` in redis.conf
2. Short-term: Bind Redis to 127.0.0.1 or private IP only
3. Short-term: Add firewall rules to restrict access to app servers only
4. Medium-term: Enable TLS encryption for Redis connections
5. Long-term: Move to Redis Cluster with proper RBAC (Redis ACLs)
```

### Infrastructure-Specific Vulnerabilities

```
CRITICAL MISCONFIGURATIONS (Found in almost every pen test):

1. ❌ Public S3 buckets / Storage blobs
2. ❌ Open security groups (0.0.0.0/0 on sensitive ports)
3. ❌ Default credentials on admin panels
4. ❌ Exposed metadata endpoints (169.254.169.254)
5. ❌ Overpermissive IAM roles (AdministratorAccess on everything)
6. ❌ Unpatched systems (known CVEs)
7. ❌ Secrets in source code / environment variables
8. ❌ No network segmentation
9. ❌ Docker socket mounted in containers
10. ❌ Kubernetes RBAC not configured (default = overpermissive)
```

## Infrastructure Pen Testing

### Network Infrastructure Testing

```bash
# 1. Discover live hosts
nmap -sn 10.0.0.0/24

# 2. Full port scan on discovered hosts
nmap -sV -sC -p- 10.0.0.1-254 -oN full_scan.txt
# -sV = version detection
# -sC = default scripts
# -p- = all ports
# -oN = output to file

# 3. Check for common misconfigurations

# SSH — Check for password authentication (should be key-only)
nmap --script ssh-auth-methods -p 22 target-ip
# If password auth is enabled, brute force is possible

# DNS — Check for zone transfer (information leak)
dig axfr @dns-server example.com
# If this returns records, the DNS server is leaking all records!

# SNMP — Check for default community strings
nmap -sU -p 161 --script snmp-info target-ip
# Default community string "public" → info disclosure

# NFS — Check for open shares
showmount -e target-ip
# If shares are open, you can mount and read/write files

# SMB — Check for null sessions and open shares (Windows)
smbclient -L //target-ip -N
enum4linux target-ip
```

### Web Server Hardening Verification

```bash
# Check security headers
curl -sI https://example.com | grep -iE \
  "x-frame|x-content|x-xss|strict-transport|content-security|referrer"

# Expected headers:
# X-Frame-Options: DENY or SAMEORIGIN
# X-Content-Type-Options: nosniff
# X-XSS-Protection: 0 (or absent — CSP is preferred)
# Strict-Transport-Security: max-age=31536000; includeSubDomains
# Content-Security-Policy: default-src 'self'
# Referrer-Policy: strict-origin-when-cross-origin

# Check TLS configuration
testssl.sh --severity HIGH example.com

# Check for server information disclosure
curl -sI https://example.com | grep -i server
# "Server: nginx/1.18.0" → Version disclosed (should be hidden)
# Fix: Add "server_tokens off;" in nginx.conf
```

## Container & Kubernetes Pen Testing

### Docker Security Testing

```bash
# 1. Scan Docker images for vulnerabilities
trivy image nginx:latest
trivy image company-app:v1.2.3

# Output shows CVEs with severity:
# CRITICAL: CVE-2023-xxxxx — Remote code execution in libssl
# HIGH:     CVE-2023-yyyyy — Path traversal in libcurl

# 2. Check Docker daemon configuration
docker info 2>/dev/null | grep -E "Security|Root"
# Look for:
# - "Rootless" mode (good!)
# - "userns" (user namespace isolation)
# - Live Restore enabled

# 3. Check for dangerous container configurations
# Look at running containers
docker inspect $(docker ps -q) | jq '.[].HostConfig | {
  Privileged,
  PidMode,
  NetworkMode,
  Binds,
  CapAdd
}'

# RED FLAGS:
# "Privileged": true           → Container has FULL host access
# "PidMode": "host"            → Can see host processes
# Binds: ["/var/run/docker.sock"] → Can control Docker daemon
# CapAdd: ["SYS_ADMIN"]        → Near-root capabilities

# 4. Check for secrets in Docker images
# Look at image history for exposed secrets
docker history company-app:v1.2.3 --no-trunc | grep -iE "env|arg|password|key|secret"

# Scan for secrets in image layers
trivy image --scanners secret company-app:v1.2.3

# 5. Docker Bench Security (CIS Benchmark)
docker run -it --net host --pid host \
  --cap-add audit_control \
  -v /var/lib:/var/lib \
  -v /var/run/docker.sock:/var/run/docker.sock \
  -v /etc:/etc \
  docker/docker-bench-security
```

### Kubernetes Security Testing

```bash
# 1. kube-hunter — Automated Kubernetes pen testing
# Run from outside the cluster (remote scanning)
kube-hunter --remote target-ip

# Run from inside a pod (internal attack simulation)
kubectl run kube-hunter --image=aquasec/kube-hunter --restart=Never \

# These services are accessible from outside the cluster
# Dashboard, Prometheus, Grafana should NOT be exposed
```

## CI/CD Pipeline Pen Testing

CI/CD pipelines are a **goldmine for attackers**. Here's what pen testers look for:

### Common CI/CD Attack Vectors

```
1. POISONED PIPELINE EXECUTION (PPE)
   - Attacker modifies the CI/CD config file (Jenkinsfile, .github/workflows)
   - Pipeline runs attacker's code with elevated privileges

2. SECRET THEFT
   - Secrets exposed in pipeline logs
   - Secrets accessible to all branches/forks
   - Secrets stored as plain text environment variables

3. DEPENDENCY CONFUSION
   - Attacker publishes a malicious package with the same name as an internal one
   - Build system pulls the public (malicious) package instead

4. ARTIFACT POISONING
   - Attacker modifies build artifacts
   - Unsigned artifacts deployed to production

5. RUNNER/AGENT COMPROMISE
   - Shared runners between projects → cross-project access
   - Runners with excessive permissions
   - Persistent runners that aren't cleaned between jobs
```

### CI/CD Security Checklist

```bash
# Check Jenkins for common issues

# 1. Is the Jenkins dashboard accessible without auth?
curl -s http://jenkins.example.com/api/json
# If this returns data, Jenkins has no authentication!

# 2. Can anonymous users see build logs?
curl -s http://jenkins.example.com/job/my-app/lastBuild/consoleText
# Build logs often contain secrets!

# 3. Check for exposed credentials in Jenkins
# Navigate to: /credentials/ or /script (Groovy console)
# Groovy console allows arbitrary code execution!
curl -s http://jenkins.example.com/script
# If accessible, attacker has full server access

# 4. Check GitHub Actions workflow files for secret leaks
# Common mistakes:
# - Using ${{ secrets.TOKEN }} in command arguments (logged!)
# - Not using environment variable masking
# - Allowing workflows to run on pull_request_target (from forks)

# 5. Check for dependency confusion
# Scan package.json / requirements.txt for internal packages
# Verify they're pulled from the correct registry
```

### Secure CI/CD Pipeline Configuration

```yaml
# Example: Hardened GitHub Actions workflow
name: Secure Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

# Restrict permissions at the workflow level
permissions:
  contents: read        # Only read access to repo
  packages: write       # Write access to packages (for publishing)
  # All other permissions are DENIED by default

jobs:
  build:
    runs-on: ubuntu-latest
    
    # Pin runner image to a specific version (avoid supply chain attacks)
    container:
      image: node:20-slim@sha256:abc123...  # Pinned by digest!
    
    steps:
      - uses: actions/checkout@v4  # Pin actions to specific versions
        with:
          persist-credentials: false  # Don't keep git credentials
      
      # Use environment variables, not inline secrets
      - name: Build
        env:
          NPM_TOKEN: ${{ secrets.NPM_TOKEN }}  # Masked in logs
        run: |
          # Don't echo secrets!
          npm ci --ignore-scripts  # --ignore-scripts prevents supply chain attacks
          npm run build
      
      # Sign artifacts
      - name: Sign artifact
        run: |
          cosign sign-blob --key ${{ secrets.COSIGN_KEY }} \
            --output-signature artifact.sig \
            dist/app.tar.gz
```

## Cloud Infrastructure Pen Testing

### AWS Pen Testing

```bash
# Prowler — comprehensive AWS security assessment
prowler aws

# Prowler checks for:
# - IAM misconfigurations (overpermissive policies, no MFA)
# - S3 bucket policies (public access)
# - Security group rules (open ports)
# - CloudTrail logging (is it enabled?)
# - Encryption at rest (EBS, RDS, S3)
# - VPC configuration (default VPCs, flow logs)

# ScoutSuite — multi-cloud security auditing
scout aws

# Check for IMDS v1 (metadata service vulnerability)
# From inside an EC2 instance:
curl http://169.254.169.254/latest/meta-data/iam/security-credentials/
# If this works WITHOUT a token, IMDSv1 is enabled → BAD!
# Attackers can steal IAM credentials via SSRF

# Fix: Enforce IMDSv2
aws ec2 modify-instance-metadata-options \
  --instance-id i-1234567890abcdef0 \
  --http-tokens required \
  --http-endpoint enabled

# Check for public EBS snapshots
aws ec2 describe-snapshots --owner-ids self \
  --query "Snapshots[?contains(to_string(CreateVolumePermissions), 'all')]"

# Check for public RDS instances
aws rds describe-db-instances \
  --query "DBInstances[?PubliclyAccessible==\`true\`].DBInstanceIdentifier"
```

### Cloud Pen Testing Rules

> ⚠️ **Each cloud provider has rules about pen testing:**

| Provider | Rules |
|----------|-------|
| **AWS** | No prior approval needed for most services (EC2, RDS, Lambda, etc.). Some services are prohibited (DDoS, DNS zone walking). Read: [AWS Pen Testing Policy](https://aws.amazon.com/security/penetration-testing/) |
| **Azure** | No prior approval needed. Follow the [Rules of Engagement](https://www.microsoft.com/en-us/msrc/pentest-rules-of-engagement) |
| **GCP** | No prior approval needed. Follow [Acceptable Use Policy](https://cloud.google.com/terms/aup) |

```
Universal Cloud Pen Testing Rules:
✅ Test YOUR OWN resources only
✅ Don't perform DoS/DDoS attacks
✅ Don't test shared infrastructure (S3 service itself, Azure AD)
✅ Don't perform DNS zone walking
✅ Don't send phishing to cloud provider employees
✅ Have written authorization from your organization
```

## Automating Security Testing in CI/CD

The best way to catch vulnerabilities is to test automatically, continuously.

### Complete Security Testing Pipeline

```yaml
# .github/workflows/security-testing.yml
name: Security Testing Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]
  schedule:
    - cron: '0 2 * * 1'  # Weekly scan every Monday at 2 AM

permissions:
  contents: read
  security-events: write

jobs:
  # Job 1: SAST — Static Application Security Testing
  sast:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Run Semgrep
        uses: returntocorp/semgrep-action@v1
        with:
          config: >-
            p/owasp-top-ten
            p/secrets
            p/ci
          generateSarif: true
      
      - name: Upload SARIF
        uses: github/codeql-action/upload-sarif@v3
        with:
          sarif_file: semgrep.sarif

  # Job 2: SCA — Dependency Vulnerability Scanning
  sca:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Run Trivy (dependencies)
        uses: aquasecurity/trivy-action@master
        with:
          scan-type: 'fs'
          severity: 'HIGH,CRITICAL'
          exit-code: '1'
          format: 'sarif'
          output: 'trivy-sca.sarif'

  # Job 3: Container Image Scanning
  container-scan:
    runs-on: ubuntu-latest
    needs: [sast]  # Build after SAST passes
    steps:
      - uses: actions/checkout@v4
      
      - name: Build image
        run: docker build -t app:test .
      
      - name: Scan image with Trivy
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: 'app:test'
          severity: 'HIGH,CRITICAL'
          exit-code: '1'

      - name: Scan for secrets in image
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: 'app:test'
          scanners: 'secret'
          exit-code: '1'

  # Job 4: IaC Security Scanning
  iac-scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Run Checkov
        uses: bridgecrewio/checkov-action@v12
        with:
          directory: ./terraform
          framework: terraform
          output_format: sarif
          output_file_path: checkov.sarif

  # Job 5: DAST — Dynamic Application Security Testing
  dast:
    runs-on: ubuntu-latest
    needs: [container-scan]  # Need a running app to test
    steps:
      - uses: actions/checkout@v4
      
      - name: Start application
        run: |
          docker compose up -d
          sleep 10  # Wait for app to start
      
      - name: Run OWASP ZAP
        uses: zaproxy/action-baseline@v0.9.0
        with:
          target: 'http://localhost:8080'
          cmd_options: '-a'

  # Job 6: Kubernetes manifest scanning
  k8s-scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Run kubeaudit
        run: |
          curl -sL https://github.com/Shopify/kubeaudit/releases/latest/download/kubeaudit_linux_amd64.tar.gz | tar xz
          ./kubeaudit all -f k8s/
```

## Pen Testing Rules & Legal Considerations

### Before You Test — The Checklist

```
LEGAL REQUIREMENTS:

1. ✅ WRITTEN AUTHORIZATION
   - Get a signed "Rules of Engagement" document
   - Clearly define scope (what CAN and CANNOT be tested)
   - Define testing window (dates and times)
   - Define allowed techniques (scanning? exploitation? social engineering?)

2. ✅ SCOPE DEFINITION
   - IP addresses / domains in scope
   - IP addresses / domains OUT of scope (third parties, shared services)
   - Environments (only staging? or production too?)
   - Types of tests allowed

3. ✅ EMERGENCY CONTACTS
   - Who to call if something breaks
   - Escalation path
   - How to pause/stop testing immediately

4. ✅ DATA HANDLING
   - How to handle sensitive data discovered during testing
   - Encryption of test results
   - Secure deletion after project completion

5. ✅ LEGAL REVIEW
   - Have legal/compliance review the engagement
   - Ensure compliance with local laws (CFAA in US, Computer Misuse Act in UK)
   - Insurance considerations
```

### What NOT to Do

```
❌ NEVER test systems you don't have permission to test
❌ NEVER perform DoS/DDoS attacks (unless specifically authorized)
❌ NEVER access, modify, or delete real user data
❌ NEVER test third-party services without their permission
❌ NEVER share vulnerabilities publicly before they're fixed
❌ NEVER keep copies of sensitive data after the engagement
❌ NEVER exceed the agreed scope
```

## Summary

| Concept | Key Takeaway |
|---------|-------------|
| **Pen Testing** | Authorized hacking to find vulnerabilities before attackers do |
| **Vs Vuln Scanning** | Vuln scanning finds issues; pen testing proves exploitation and impact |
| **Phases** | Recon → Scan → Exploit → Post-Exploit → Report |
| **Key Tools** | Nmap (network), Burp/ZAP (web), Trivy (containers), kube-hunter (K8s), Prowler (AWS) |
| **Docker Testing** | Scan images, check for root, check for mounted sockets, use docker-bench |
| **K8s Testing** | Check RBAC, pod security, network policies, service exposure |
| **CI/CD Testing** | Check for secret leaks, pipeline poisoning, unsigned artifacts |
| **Cloud Testing** | Check IAM, public storage, metadata service, security groups |
| **Legal** | ALWAYS get written permission. NEVER exceed scope |
| **Practice** | Use DVWA, Juice Shop, Kubernetes Goat, HackTheBox, TryHackMe |

> 🔑 **The DevOps Pen Testing Mindset:** You don't need to be a hacker. You need to **think like one** so you can build systems that are hard to break.
