## What is Runtime Security?

Let's start from scratch with a simple analogy.

Imagine you're protecting a building:

| Security Phase | Building Analogy | Software Equivalent |
|---------------|-----------------|---------------------|
| **Build time** | Checking construction materials for quality | Scanning code & images for vulnerabilities (SAST, SCA) |
| **Deployment time** | Inspecting the building before opening | Admission controllers, policy checks (OPA, Kyverno) |
| **Runtime** | Security cameras, guards, alarms WHILE people are inside | Monitoring containers/servers WHILE they're running |

**Runtime Security** = Detecting and preventing threats **while your applications are actually running** in production.

It answers the question: **"Is something bad happening RIGHT NOW in my system?"**

### What Can Go Wrong at Runtime?

```
Even if you scanned your images, checked your configs, and deployed securely,
bad things can STILL happen at runtime:

🔴 A compromised dependency starts making unexpected network connections
🔴 An attacker gains shell access inside a container
🔴 A process starts reading sensitive files it shouldn't
🔴 A cryptominer gets deployed in your cluster
🔴 Someone modifies system binaries (rootkit)
🔴 A container tries to escape to the host
🔴 Unusual outbound traffic to known malicious IPs
🔴 Privilege escalation attempts
🔴 Secrets being accessed from unexpected processes
```

> 💡 **Key Insight:** Scanning catches KNOWN vulnerabilities. Runtime security catches UNKNOWN threats and actual exploitation.

## Why Runtime Security Matters

### The Security Gaps in Traditional DevSecOps

```
Traditional DevSecOps Pipeline:

CODE ──► BUILD ──► TEST ──► SCAN ──► DEPLOY ──► ??? (Runtime)
  ✅       ✅        ✅       ✅        ✅          ❌ No visibility!
```

Most DevSecOps pipelines focus on **shift-left** — catching issues early. That's great! But it creates a false sense of security because:

1. **Zero-day vulnerabilities** → You can't scan for what you don't know exists
2. **Supply chain attacks** → Malicious code can be injected after scanning
3. **Misconfigurations at runtime** → Drift from desired state
4. **Insider threats** → A compromised user/account does bad things
5. **Legitimate tools used maliciously** → `curl`, `wget`, `bash` are normal tools, but using them to download malware is not

### Real-World Incidents That Runtime Security Would Catch

| Incident | What Happened | Runtime Detection |
|----------|--------------|-------------------|
| **SolarWinds (2020)** | Backdoor injected into build pipeline | Unusual outbound network connections |
| **Log4Shell (2021)** | Zero-day RCE in Log4j | Shell spawned inside Java container |
| **Codecov (2021)** | CI tool compromised to steal secrets | Unexpected network calls from CI runner |
| **Cryptominers** | Attackers deploy miners in Kubernetes | High CPU usage, connections to mining pools |
| **Container escapes** | Attacker breaks out of container to host | Privileged system calls, mount namespace changes |

## How Runtime Security Works

At a high level, runtime security tools work by:

```
┌─────────────────────────────────────────────────────────┐
│                  How Runtime Security Works               │
├─────────────────────────────────────────────────────────┤
│                                                           │
│   1. INSTRUMENT                                           │
│      Hook into the kernel or container runtime            │
│      to observe system calls, network activity,           │
│      file access, and process execution                   │
│                                                           │
│   2. MONITOR                                              │
│      Continuously collect events:                         │
│      - System calls (open, exec, connect, etc.)           │
│      - File I/O (reads, writes, modifications)            │
│      - Network connections (source, dest, protocol)       │
│      - Process creation (what spawned what)               │
│      - Privilege changes (setuid, capabilities)           │
│                                                           │
│   3. DETECT                                               │
│      Compare observed behavior against:                   │
│      - Predefined rules (signatures)                      │
│      - Behavioral baselines (ML/anomaly detection)        │
│      - Policy violations                                  │
│                                                           │
│   4. RESPOND                                              │
│      When a threat is detected:                           │
│      - Generate alert (Slack, PagerDuty, SIEM)            │
│      - Kill the process                                   │
│      - Kill the container/pod                             │
│      - Block the network connection                       │
│      - Capture forensic data                              │
│                                                           │
└─────────────────────────────────────────────────────────┘
```

### The Technology Under the Hood

Runtime security tools use these mechanisms to observe system behavior:

| Technology | How It Works | Pros | Cons |
|-----------|-------------|------|------|
| **Kernel Modules** | Loadable module inserted into Linux kernel | Deep visibility | Risky (crash = system crash) |
| **eBPF** | Programs that run in the kernel safely | Safe, fast, no kernel module needed | Requires newer kernels (4.14+) |
| **ptrace** | Process tracing system call | Simple | Slow, overhead |
| **Syscall interception** | Hook into system call table | Deep visibility | Can impact performance |
| **Container runtime hooks** | OCI runtime hooks | Container-aware | Limited to container events |

> 🏆 **eBPF is the winner.** Modern tools (Falco, Tetragon, Sysdig) are all moving to eBPF because it's fast, safe, and powerful.

## Runtime Security vs Other Security Approaches

| Approach | When It Runs | What It Catches | Example Tools |
|----------|-------------|-----------------|---------------|
| **SAST** | During coding/build | Code vulnerabilities | SonarQube, Semgrep, CodeQL |
| **SCA** | During build | Vulnerable dependencies | Snyk, Dependabot, Trivy |
| **Container Scanning** | Before deployment | Image vulnerabilities | Trivy, Grype, Anchore |
| **Admission Control** | At deployment time | Policy violations | OPA/Gatekeeper, Kyverno |
| **Runtime Security** | While running | Active threats, anomalies | Falco, Sysdig, Aqua, Tetragon |

> 🎯 **You need ALL of these.** They complement each other. Runtime security is the **last line of defense**.

## Falco — The Cloud-Native Runtime Security Engine

### What is Falco?

**Falco** is an open-source runtime security tool created by **Sysdig** and donated to the **CNCF** (Cloud Native Computing Foundation). It's now a **CNCF Graduated** project — meaning it's production-ready and widely adopted.

Think of Falco as a **security camera for your containers and hosts**. It watches everything happening at the system level and alerts you when something suspicious occurs.

```
What Falco monitors:
├── System calls (open, read, write, exec, connect, etc.)
├── File access (who's reading /etc/shadow?)
├── Process execution (bash spawned inside a container?)
├── Network connections (outbound to suspicious IPs?)
├── Kubernetes audit events (who created a privileged pod?)
└── Cloud events (AWS CloudTrail, GCP Audit, Azure)
```

### How Falco Works

```
┌──────────────────────────────────────────────────────┐
│                   Falco Architecture                  │
├──────────────────────────────────────────────────────┤
│                                                        │
│                    ┌──────────────┐                    │
│                    │  Falco Rules │ ← YAML rule files  │
│                    │   Engine     │   (what to detect)  │
│                    └──────┬───────┘                    │
│                           │                            │
│                    ┌──────▼───────┐                    │
│                    │   Falco      │                    │
│                    │  Userspace   │ ← Decision engine   │
│                    │  Process     │   (match events     │
│                    │              │    against rules)    │
│                    └──────┬───────┘                    │
│                           │                            │
│              ┌────────────▼────────────┐              │
│              │    Event Sources         │              │
│              ├────────────┬────────────┤              │
│              │  Kernel    │ K8s Audit  │              │
│              │  (eBPF/    │ Log /      │              │
│              │  kmod)     │ Plugins    │              │
│              └────────────┴────────────┘              │
│                           │                            │
│              ┌────────────▼────────────┐              │
│              │     Linux Kernel         │              │
│              │   (System Calls)         │              │
│              └─────────────────────────┘              │
│                                                        │
│  OUTPUTS: stdout, syslog, HTTP, gRPC, Slack,          │
│           PagerDuty, Kafka, Elasticsearch, etc.        │
└──────────────────────────────────────────────────────┘
```

**How it works step by step:**
1. **Event Source** captures system calls from the kernel (via eBPF or kernel module)
2. **Events** are sent to the Falco userspace process
3. **Rules Engine** checks each event against the loaded rules
4. If an event **matches a rule** → an **alert** is generated
5. Alerts are sent to configured **outputs** (Slack, PagerDuty, SIEM, etc.)

### Installing Falco

#### On a Linux Host

```bash
# Add the Falco repository
curl -fsSL https://falco.org/repo/falcosecurity-packages.asc | \
  sudo gpg --dearmor -o /usr/share/keyrings/falco-archive-keyring.gpg

echo "deb [signed-by=/usr/share/keyrings/falco-archive-keyring.gpg] \
  https://download.falco.org/packages/deb stable main" | \
  sudo tee /etc/apt/sources.list.d/falcosecurity.list

sudo apt update
sudo apt install -y falco

# Start Falco
sudo systemctl start falco
sudo systemctl enable falco

# Check logs
sudo journalctl -u falco -f
```

#### On Kubernetes (using Helm)

```bash
# Add the Falco Helm chart repo
helm repo add falcosecurity https://falcosecurity.github.io/charts
helm repo update

# Install Falco with eBPF driver (recommended)
helm install falco falcosecurity/falco \
  --namespace falco \
  --create-namespace \
  --set driver.kind=ebpf \
  --set falcosidekick.enabled=true \
  --set falcosidekick.webui.enabled=true

# Check Falco pods
kubectl get pods -n falco

# View Falco logs (alerts)
kubectl logs -n falco -l app.kubernetes.io/name=falco -f
```

**What we just did:**
- Installed Falco as a **DaemonSet** (runs on every node in the cluster)
- Used the **eBPF driver** (safer than kernel module, no kernel module loading needed)
- Enabled **Falcosidekick** (routes alerts to Slack, Teams, PagerDuty, etc.)
- Enabled the **Web UI** to view alerts in a dashboard

### Falco Rules — Deep Dive

Falco rules are written in YAML. There are three components:

| Component | What It Is | Purpose |
|-----------|-----------|---------|
| **Rules** | The actual detection logic | Define WHAT to detect |
| **Macros** | Reusable conditions (like functions) | DRY — don't repeat yourself |
| **Lists** | Named collections of values | Group related items |

#### Default Rules — What Falco Detects Out of the Box

Falco ships with **hundreds of built-in rules**. Here are the most important ones:

```yaml
# 1. Shell spawned inside a container
# This catches attackers getting shell access
- rule: Terminal shell in container
  desc: >
    A shell was used as the entrypoint/exec point into a container 
    with an attached terminal.
  condition: >
    spawned_process and container and shell_procs 
    and proc.tty != 0
  output: >
    A shell was spawned in a container with an attached terminal 
    (user=%user.name container=%container.name 
    shell=%proc.name parent=%proc.pname cmdline=%proc.cmdline 
    terminal=%proc.tty)
  priority: NOTICE
  tags: [container, shell, mitre_execution]

# 2. Sensitive file read
# Catches processes reading password/key files
- rule: Read sensitive file untrusted
  desc: >
    An attempt to read any sensitive file (e.g., /etc/shadow)
  condition: >
    sensitive_files and open_read and not trusted_process
  output: >
    Sensitive file opened for reading 
    (user=%user.name file=%fd.name process=%proc.name 
    command=%proc.cmdline container=%container.name)
  priority: WARNING

# 3. Write to filesystem in container (should be read-only!)
- rule: Write below etc
  desc: An attempt to write below /etc
  condition: >
    write and container and fd.directory = /etc
  output: >
    File below /etc written in container 
    (user=%user.name file=%fd.name container=%container.name)
  priority: ERROR

# 4. Outbound connection to suspicious destination
- rule: Unexpected outbound connection
  desc: Container connecting to external network
  condition: >
    outbound and container and not expected_outbound
  output: >
    Unexpected outbound connection 
    (container=%container.name connection=%fd.name 
    process=%proc.name user=%user.name)
  priority: WARNING

# 5. Privilege escalation attempt
- rule: Container drift detected (new executable)
  desc: New executable not present in the original image
  condition: >
    spawned_process and container 
    and proc.is_exe_from_memfd=true
  output: >
    Drift detected — new binary executed in container 
    (process=%proc.name container=%container.name)
  priority: CRITICAL
```

#### Understanding Rule Conditions

Falco conditions use a **filter syntax** based on system call fields:

```yaml
# Key fields you can use in rules:

# Process fields
proc.name        # Process name (e.g., bash, python, curl)
proc.pname       # Parent process name
proc.cmdline     # Full command line
proc.pid         # Process ID
proc.ppid        # Parent PID
proc.exepath     # Path to the executable
proc.tty         # Terminal (0 = no terminal)

# File descriptor fields
fd.name          # File/connection name
fd.directory     # Directory of the file
fd.type          # Type (file, ipv4, ipv6, unix)
fd.sip           # Source IP (for network)
fd.sport         # Source port
fd.dip           # Destination IP  
fd.dport         # Destination port

# User fields
user.name        # Username
user.uid         # User ID

# Container fields
container.name   # Container name
container.id     # Container ID
container.image  # Container image name
k8s.pod.name     # Kubernetes pod name
k8s.ns.name      # Kubernetes namespace

# System call fields
evt.type         # System call type (open, exec, connect, etc.)
evt.dir          # Direction (< = enter, > = exit)
```

### Writing Custom Falco Rules

Let's write custom rules for real-world scenarios:

#### Rule 1: Detect Cryptocurrency Mining

```yaml
# /etc/falco/rules.d/custom-rules.yaml

# Define a list of known mining pool domains/IPs
- list: mining_pool_ports
  items: [3333, 4444, 5555, 7777, 8888, 9999, 14433, 14444]

- list: mining_domains
  items: [
    "pool.minexmr.com",
    "xmr.pool.minergate.com", 
    "monerohash.com",
    "stratum.slushpool.com",
    "eth.2miners.com"
  ]

# Rule: Detect outbound connections to mining pools
- rule: Detect Cryptocurrency Mining
  desc: >
    Detects connections to known cryptocurrency mining pools.
    This often indicates a compromised container running a cryptominer.
  condition: >
    outbound and container
    and (fd.sport in (mining_pool_ports) 
         or fd.dport in (mining_pool_ports))
  output: >
    🚨 CRYPTOMINER DETECTED! Outbound connection to mining pool 
    (container=%container.name image=%container.image 
    connection=%fd.name process=%proc.name 
    pod=%k8s.pod.name namespace=%k8s.ns.name 
    command=%proc.cmdline)
  priority: CRITICAL
  tags: [cryptomining, network, mitre_impact]
```

**What this rule does:**
- Watches for **outbound network connections** from containers
- Checks if the destination port matches known **mining pool ports**
- If matched → fires a **CRITICAL** alert with container/pod details
- Tags follow **MITRE ATT&CK** framework for correlation

#### Rule 2: Detect Reverse Shell

```yaml
# Rule: Detect reverse shell attempts
- rule: Reverse Shell Detected
  desc: >
    Detects common reverse shell patterns where a shell's
    stdin/stdout is connected to a network socket.
  condition: >
    spawned_process and container
    and proc.name in (bash, sh, zsh, dash, csh)
    and (fd.type = ipv4 or fd.type = ipv6)
    and fd.sip != "127.0.0.1"
  output: >
    🚨 REVERSE SHELL! Shell connected to network socket
    (container=%container.name process=%proc.name 
    connection=%fd.name user=%user.name 
    pod=%k8s.pod.name cmdline=%proc.cmdline)
  priority: CRITICAL
  tags: [shell, network, mitre_execution, mitre_c2]
```

#### Rule 3: Detect Kubectl Exec Into Production Pods

```yaml
# Rule: Alert on kubectl exec in production namespace
- rule: Kubectl Exec in Production
  desc: >
    Detects when someone uses kubectl exec to get a shell
    in a production namespace. This should be rare and investigated.
  condition: >
    spawned_process and container
    and k8s.ns.name = "production"
    and proc.name in (bash, sh, zsh)
    and proc.pname = "runc"
  output: >
    ⚠️ KUBECTL EXEC in production! 
    (user=%user.name pod=%k8s.pod.name 
    namespace=%k8s.ns.name container=%container.name 
    command=%proc.cmdline)
  priority: WARNING
  tags: [kubernetes, exec, mitre_execution]
```

#### Rule 4: Detect Package Installation in Running Containers

```yaml
# Rule: Package manager used in running container
# Containers should be immutable — no installing packages at runtime!
- rule: Package Manager in Container
  desc: >
    Package manager (apt, yum, apk) executed inside a running container.
    Containers should be immutable — packages should only be installed
    during image build.
  condition: >
    spawned_process and container
    and proc.name in (apt, apt-get, yum, dnf, apk, pip, pip3, npm)
    and not proc.pname in (docker-build, buildkit)
  output: >
    ⚠️ Package manager used in running container! 
    (container=%container.name process=%proc.name 
    command=%proc.cmdline image=%container.image 
    pod=%k8s.pod.name)
  priority: ERROR
  tags: [container, drift, mitre_execution]
```

#### Rule 5: Detect Sensitive Mount from Container

```yaml
# Rule: Container accessing host filesystem
- rule: Container Accessing Host Filesystem
  desc: >
    Detects when a container accesses sensitive host paths like
    /etc/shadow, Docker socket, or Kubernetes secrets.
  condition: >
    open_read and container
    and (fd.name startswith /host/etc/ 
         or fd.name = /var/run/docker.sock
         or fd.name startswith /host/var/lib/kubelet/)
  output: >
    🚨 Container accessing host filesystem! 
    (container=%container.name file=%fd.name 
    process=%proc.name user=%user.name 
    pod=%k8s.pod.name)
  priority: CRITICAL
  tags: [container, escape, mitre_privilege_escalation]
```

### Falco Alerts & Integrations

Falco by itself outputs alerts to stdout/syslog. To route alerts to useful destinations, use **Falcosidekick**:

```yaml
# Falcosidekick configuration — route alerts to multiple destinations
# values.yaml for Helm installation

falcosidekick:
  enabled: true
  
  config:
    # Slack integration
    slack:
      webhookurl: "https://hooks.slack.com/services/XXX/YYY/ZZZ"
      channel: "#security-alerts"
      minimumpriority: "warning"      # Only send WARNING and above
      messageformat: |
        *{{ .Priority }}* — {{ .Rule }}
        {{ .Output }}
        Container: {{ index .OutputFields "container.name" }}
        Pod: {{ index .OutputFields "k8s.pod.name" }}
        Namespace: {{ index .OutputFields "k8s.ns.name" }}
    
    # PagerDuty for critical alerts
    pagerduty:
      routingkey: "your-pagerduty-routing-key"
      minimumpriority: "critical"      # Only page for CRITICAL
    
    # Elasticsearch for log storage & analysis
    elasticsearch:
      hostport: "https://elasticsearch:9200"
      index: "falco-alerts"
      minimumpriority: "notice"
    
    # Prometheus for metrics
    prometheus:
      extralabels: "source:falco"
    
    # Webhook (generic HTTP endpoint)
    webhook:
      address: "https://your-siem.example.com/api/alerts"
      minimumpriority: "warning"

  # Web UI dashboard
  webui:
    enabled: true
```

```
Alert flow:

Falco ──► Falcosidekick ──► Slack (WARNING+)
                         ──► PagerDuty (CRITICAL only)
                         ──► Elasticsearch (all events)
                         ──► Prometheus (metrics)
                         ──► SIEM (WARNING+)
```

## Sysdig — Visibility + Security Platform

### What is Sysdig?

**Sysdig** is both an **open-source tool** AND a **commercial platform**:

| Version | What It Is | Cost |
|---------|-----------|------|
| **Sysdig OSS** | System-level troubleshooting & inspection tool | Free |
| **Sysdig Secure** | Full runtime security, compliance, vulnerability management | Paid |
| **Sysdig Monitor** | Prometheus-compatible monitoring platform | Paid |

Fun fact: Sysdig Inc. is the company that **created Falco** and then donated it to CNCF!

### Sysdig Open Source (Inspect)

Sysdig OSS is like `strace` + `tcpdump` + `htop` on steroids. It captures system calls and lets you inspect EVERYTHING happening on a system.

#### Installation

```bash
# Install Sysdig on Linux
curl -s https://s3.amazonaws.com/download.draios.com/stable/install-sysdig | sudo bash

# Or via Docker
docker run -it --privileged \
  -v /var/run/docker.sock:/host/var/run/docker.sock \
  -v /dev:/host/dev \
  -v /proc:/host/proc:ro \
  -v /boot:/host/boot:ro \
  -v /src:/src:ro \
  -v /lib/modules:/host/lib/modules:ro \
  sysdig/sysdig
```

#### Using Sysdig for System Inspection

```bash
# 1. See ALL system calls (overwhelming but complete!)
sudo sysdig

# 2. Filter by container name
sudo sysdig container.name=my-app

# 3. See all network connections from a container
sudo sysdig -p"%container.name %fd.name" container.name=my-app and evt.type=connect

# 4. See all file opens in a container
sudo sysdig -p"%proc.name %fd.name" container.name=my-app and evt.type=open

# 5. See all processes spawned in a container
sudo sysdig -p"%proc.name %proc.args" container.name=my-app and evt.type=execve

# 6. Top processes by CPU
sudo sysdig -c topprocs_cpu

# 7. Top containers by network I/O
sudo sysdig -c topcontainers_net

# 8. See all DNS requests from a container
sudo sysdig -p"%proc.name %fd.name" \
  container.name=my-app and evt.type in (sendto,recvfrom) and fd.port=53

# 9. Capture events to a file for later analysis
sudo sysdig -w capture.scap container.name=my-app

# 10. Read and analyze the capture later
sudo sysdig -r capture.scap "proc.name=bash"
```

**What each command does:**
- `-p` → Custom output format (like printf)
- `container.name=my-app` → Filter by container
- `evt.type=connect` → Only show network connections
- `-c topprocs_cpu` → Run the "top processes by CPU" chisel (built-in script)
- `-w` → Write raw capture to file (for forensics)
- `-r` → Read from a capture file

### Using Sysdig for Troubleshooting & Forensics

#### Scenario: Investigating a Security Incident

```bash
# Someone reported suspicious activity in the "payment" container.
# Let's investigate!

# Step 1: Capture everything from the suspicious container
sudo sysdig -w incident.scap container.name=payment

# Let it run for a few minutes, then stop with Ctrl+C

# Step 2: Analyze — What processes were running?
sudo sysdig -r incident.scap -c topprocs_cpu
# Look for unexpected processes (miners, reverse shells)

# Step 3: What files were accessed?
sudo sysdig -r incident.scap \
  -p"%evt.time %proc.name %fd.name" \
  evt.type=open and fd.name contains /etc/
# Look for access to sensitive files

# Step 4: What network connections were made?
sudo sysdig -r incident.scap \
  -p"%evt.time %proc.name %fd.sip:%fd.sport -> %fd.dip:%fd.dport" \
  evt.type=connect
# Look for connections to unknown/suspicious IPs

# Step 5: What commands were run?
sudo sysdig -r incident.scap \
  -p"%evt.time %user.name %proc.name %proc.args" \
  evt.type=execve
# Full command history — like a security camera replay!

# Step 6: Were any files modified?
sudo sysdig -r incident.scap \
  -p"%evt.time %proc.name %fd.name" \
  evt.type in (write, writev) and fd.name contains /
# Check for unauthorized modifications
```

**Why this is powerful:**
- Sysdig captures **raw system calls** — it's like a DVR for your system
- You can replay events **after the fact** for forensic investigation
- Every process, file access, network connection is recorded
- This is invaluable for post-incident analysis

Aqua's commercial runtime protection works differently from Falco. Instead of just detecting threats, it can **actively prevent** them:

Aqua Runtime Protection Capabilities:

1. CONTAINER DRIFT PREVENTION

## eBPF — The Technology Behind Modern Runtime Security

### What is eBPF?

**eBPF** (extended Berkeley Packet Filter) is a technology that lets you run small programs **inside the Linux kernel** without modifying kernel source code or loading kernel modules.

Think of it as a **safe plugin system for the Linux kernel**.

```
Without eBPF:
  To observe system calls, you need to:
  - Load a kernel module (risky — can crash the system)
  - Use ptrace (slow, high overhead)
  - Modify the kernel (complex, version-specific)

With eBPF:
  - Write a small program
  - The kernel VERIFIES it's safe (won't crash, won't loop forever)
  - Attach it to kernel events (system calls, network packets, etc.)
  - Get data with minimal overhead (< 1% performance impact!)
```

### Why eBPF Matters for Runtime Security

```
eBPF advantages for security:

1. SAFETY — Programs are verified before running (can't crash kernel)
2. SPEED — Runs in kernel space (no context switching overhead)
3. FLEXIBILITY — Can attach to any kernel event
4. NO KERNEL MODULES — Works without modifying the kernel
5. PORTABILITY — Works across kernel versions (with CO-RE/BTF)
6. LOW OVERHEAD — Typically < 1% performance impact
```

| Tool | Creator | Focus |

#### Tetragon — eBPF-Based Runtime Enforcement

```bash
# Tetragon by Cilium is gaining popularity as a Falco alternative
# Key difference: Tetragon can ENFORCE (block) in kernel space

# Install Tetragon on Kubernetes
helm repo add cilium https://helm.cilium.io
helm install tetragon cilium/tetragon \
  --namespace kube-system

# Example: Block execution of specific binaries
kubectl apply -f - <<EOF
apiVersion: cilium.io/v1alpha1
kind: TracingPolicy
metadata:
  name: block-package-managers
spec:
  kprobes:
    - call: "__x64_sys_execve"
      syscall: true
      args:
        - index: 0
          type: "string"
      selectors:
        - matchArgs:
            - index: 0
              operator: "In"
              values:
                - "/usr/bin/apt"
                - "/usr/bin/apt-get"
                - "/usr/bin/yum"
                - "/usr/bin/apk"
                - "/usr/bin/curl"
                - "/usr/bin/wget"
          matchNamespaces:
            - namespace: production
              operator: In
          matchActions:
            - action: Sigkill    # Kill the process immediately!
EOF
```

**What this does:**
- Uses **eBPF kprobes** to hook into the `execve` system call (process execution)
- When someone tries to run `apt`, `curl`, `wget` etc. in the `production` namespace
- **Kills the process instantly** (SIGKILL) — before it even starts!
- This happens **in kernel space** — it's incredibly fast and cannot be bypassed
