# 16.2 — OWASP Top 10 — Web Application Vulnerabilities

> **One Line Answer:** The OWASP Top 10 is a **list of the 10 most critical security risks** to web applications, published by the Open Web Application Security Project (OWASP). It's the **gold standard** that every developer, DevOps engineer, and security professional must know.

---

## 🤔 What is OWASP?

**OWASP** stands for **Open Web Application Security Project**. It's a **non-profit foundation** that works to improve the security of software. They provide:

- Free tools and resources
- Community-driven documentation
- Security standards and guidelines
- The famous **OWASP Top 10** list

> Think of OWASP as the "Wikipedia of web security" — free, open, and incredibly valuable.

The **OWASP Top 10** is updated every few years based on real-world data from hundreds of organizations. The current version is from **2021**, and it represents the most common and dangerous vulnerabilities found in real web applications.

## A01: Broken Access Control 🚪

### What Is It?

Access control means making sure people can only do what they're **allowed** to do. **Broken access control** means those rules are missing, weak, or misconfigured.

> Imagine a hotel where every room key opens every room. That's broken access control.

### Real-World Example

```
Normal behavior:
  User A logs in → sees only THEIR profile at /api/users/101

Attack:
  User A changes the URL to /api/users/102 → sees User B's profile! 😱
  
This is called an IDOR (Insecure Direct Object Reference)
```

### Vulnerable Code (Node.js/Express)

```javascript
// ❌ VULNERABLE — No authorization check!
app.get('/api/users/:id', (req, res) => {
    const userId = req.params.id;
    const user = database.getUserById(userId);  // Returns ANY user's data
    res.json(user);
});
```

**Why is this dangerous?** The code just takes whatever `id` is in the URL and returns that user's data. There's no check to verify if the logged-in user has permission to see that specific profile. Any authenticated user could change the `id` in the URL and access anyone else's data.

### Fixed Code

```javascript
// ✅ SECURE — Check if the logged-in user is authorized
app.get('/api/users/:id', authenticateToken, (req, res) => {
    const requestedUserId = req.params.id;
    const loggedInUserId = req.user.id;  // From JWT/session

    // Only allow users to access their OWN data (or admins)
    if (requestedUserId !== loggedInUserId && !req.user.isAdmin) {
        return res.status(403).json({ error: 'Access denied' });
    }

    const user = database.getUserById(requestedUserId);
    res.json(user);
});
```

**What we fixed:**
- Added `authenticateToken` middleware that verifies the user is logged in and extracts their identity from the JWT token
- Added an authorization check — the code compares the `requestedUserId` (from the URL) with the `loggedInUserId` (from the token)
- If they don't match AND the user is not an admin, it returns a `403 Forbidden` response
- This ensures users can only access their own data

### How to Prevent:

```
✅ Deny access by default — only allow what's explicitly permitted
✅ Implement proper role-based access control (RBAC)
✅ Never trust user input for authorization decisions (user ID from URL)
✅ Validate permissions on the SERVER side (not just hide UI elements)
✅ Use automated tests to verify access control rules
✅ Log access control failures and alert on repeated violations
```

## A02: Cryptographic Failures 🔐

### What Is It?

Previously called "Sensitive Data Exposure." This happens when you fail to properly **protect sensitive data** — either in transit (moving across the network) or at rest (stored in databases).

> Imagine sending your credit card number on a postcard instead of in a sealed envelope. That's a cryptographic failure.

### Common Mistakes

```
❌ Storing passwords in plain text
❌ Using HTTP instead of HTTPS
❌ Using weak/outdated algorithms (MD5, SHA1, DES)
❌ Hardcoding encryption keys in source code
❌ Not encrypting sensitive data in the database
❌ Using the same encryption key for everything
```

### Vulnerable Code (Python)

```python
import hashlib

# ❌ VULNERABLE — Storing password with MD5 (broken algorithm, no salt!)
def store_password(password):
    hashed = hashlib.md5(password.encode()).hexdigest()
    database.save(hashed)

# ❌ VULNERABLE — Storing credit card in plain text
def store_credit_card(card_number):
    database.save(card_number)  # Plain text! Anyone with DB access can see it
```

**Why is this dangerous?**
- **MD5** is a broken hashing algorithm — attackers can crack MD5 hashes in seconds using pre-computed rainbow tables
- There's **no salt** — identical passwords produce identical hashes, making bulk cracking trivial
- The credit card number is stored in **plain text** — if anyone gains database access (SQL injection, insider threat, backup leak), all card numbers are exposed

### Fixed Code

```python
import bcrypt
from cryptography.fernet import Fernet

# ✅ SECURE — Using bcrypt for password hashing (with automatic salting)
def store_password(password):
    # bcrypt automatically generates a random salt and includes it in the hash
    salt = bcrypt.gensalt(rounds=12)  # 12 rounds = good balance of security & speed
    hashed = bcrypt.hashpw(password.encode(), salt)
    database.save(hashed)

def verify_password(password, stored_hash):
    # bcrypt handles the salt extraction automatically
    return bcrypt.checkpw(password.encode(), stored_hash)

# ✅ SECURE — Encrypting sensitive data before storage
def store_credit_card(card_number):
    # Key should come from a secrets manager, NOT hardcoded!
    key = secrets_manager.get_secret('ENCRYPTION_KEY')
    fernet = Fernet(key)
    encrypted_card = fernet.encrypt(card_number.encode())
    database.save(encrypted_card)

def get_credit_card(encrypted_card):
    key = secrets_manager.get_secret('ENCRYPTION_KEY')
    fernet = Fernet(key)
    decrypted_card = fernet.decrypt(encrypted_card).decode()
    return decrypted_card
```

**What we fixed:**
- **bcrypt** is a purpose-built password hashing function — it's intentionally slow (making brute-force attacks impractical), and it automatically adds a random **salt** to each password
- `rounds=12` means bcrypt runs 2^12 = 4096 iterations internally — this makes each hash take time to compute, frustrating attackers
- Credit card numbers are now **encrypted** using Fernet (AES-128-CBC) before storage
- The encryption key comes from a **secrets manager** instead of being hardcoded

### How to Prevent:

```
✅ Use HTTPS everywhere (TLS 1.2+ only)
✅ Hash passwords with bcrypt, scrypt, or Argon2 (NEVER MD5 or SHA1)
✅ Encrypt sensitive data at rest (AES-256)
✅ Store encryption keys in a secrets manager (Vault, AWS KMS)
✅ Don't store sensitive data you don't need
✅ Disable caching for responses containing sensitive data
```

## A03: Injection 💉

### What Is It?

Injection happens when **untrusted user input** is sent to an interpreter (database, OS, LDAP) as part of a command or query. The most common type is **SQL Injection**.

> Imagine someone writes their name on a form as `Robert'; DROP TABLE students;--`. If the system blindly uses that input in a SQL query, it would delete the entire students table!

### The Classic SQL Injection

```
Normal login:
  Username: alice
  Password: password123
  
  Query: SELECT * FROM users WHERE username = 'alice' AND password = 'password123'
  Result: ✅ Returns Alice's record if credentials match

Attack:
  Username: ' OR '1'='1' --
  Password: anything
  
  Query: SELECT * FROM users WHERE username = '' OR '1'='1' --' AND password = 'anything'
  Result: 😱 Returns ALL users! The -- comments out the rest of the query
         '1'='1' is always true, so it matches every row
```

### Vulnerable Code (Python/Flask)

```python
# ❌ VULNERABLE — String concatenation in SQL query
@app.route('/login', methods=['POST'])
def login():
    username = request.form['username']
    password = request.form['password']
    
    # DANGER! User input directly in the query string!
    query = f"SELECT * FROM users WHERE username = '{username}' AND password = '{password}'"
    result = db.execute(query)
    
    if result:
        return "Login successful!"
    return "Login failed!"
```

**Why is this dangerous?** The user's input (`username` and `password`) is directly concatenated into the SQL query string using an f-string. An attacker can craft special input that changes the meaning of the SQL query. For example, entering `' OR '1'='1' --` as the username makes the query always return all users, bypassing authentication entirely.

### Fixed Code

```python
# ✅ SECURE — Using parameterized queries (prepared statements)
@app.route('/login', methods=['POST'])
def login():
    username = request.form['username']
    password = request.form['password']
    
    # Parameterized query — the database treats inputs as DATA, not CODE
    query = "SELECT * FROM users WHERE username = %s AND password = %s"
    result = db.execute(query, (username, password))
    
    if result:
        return "Login successful!"
    return "Login failed!"
```

**What we fixed:**
- Instead of putting user input directly into the query string, we use **`%s` placeholders** (parameterized query)
- The database driver handles the escaping — it treats the user input strictly as **data values**, never as SQL code
- Even if an attacker types `' OR '1'='1' --`, the database will literally search for a user with that exact string as their username (which doesn't exist)

### Other Types of Injection

```
┌──────────────────────┬──────────────────────────────────────────┐
│ Type                 │ What Gets Injected                       │
├──────────────────────┼──────────────────────────────────────────┤
│ SQL Injection        │ SQL queries (most common)                │
│ Command Injection    │ OS commands (ls, rm, cat, etc.)          │
│ LDAP Injection       │ LDAP queries (directory services)        │
│ XPath Injection      │ XML queries                              │
│ NoSQL Injection      │ MongoDB/NoSQL queries                    │
│ Template Injection   │ Server-side template engines (SSTI)      │
│ Header Injection     │ HTTP headers                             │
└──────────────────────┴──────────────────────────────────────────┘
```

### Command Injection Example

```python
# ❌ VULNERABLE — User input in OS command
@app.route('/ping')
def ping():
    host = request.args.get('host')
    # Attacker could send: host=google.com; rm -rf /
    result = os.system(f"ping -c 3 {host}")  
    return result

# ✅ SECURE — Use subprocess with argument list (no shell interpretation)
import subprocess

@app.route('/ping')
def ping():
    host = request.args.get('host')
    # Validate input first
    if not re.match(r'^[a-zA-Z0-9.-]+$', host):
        return "Invalid hostname", 400
    # subprocess with list arguments prevents shell injection
    result = subprocess.run(['ping', '-c', '3', host], capture_output=True, text=True)
    return result.stdout
```

**Why the fix works:**
- The **input is validated** using a regex that only allows letters, numbers, dots, and hyphens — characters that are valid in hostnames
- `subprocess.run()` with a **list of arguments** does NOT invoke a shell — each argument is passed directly to the `ping` command, so special characters like `;`, `|`, `&&` have no special meaning
- Compare this to `os.system()` which runs through the shell and interprets those special characters as command separators

### How to Prevent:

```
✅ Always use parameterized queries / prepared statements
✅ Never concatenate user input into queries or commands
✅ Validate and sanitize all user input
✅ Use ORMs (SQLAlchemy, Hibernate, Prisma) — they use parameterized queries by default
✅ Apply least privilege to database accounts
✅ Use allowlists for input validation, not blocklists
```

## A04: Insecure Design 📐

### What Is It?

This is about **design-level flaws** — problems that exist because security wasn't considered during the design phase. You can't fix an insecure design by just writing better code. You need to go back to the drawing board.

> Imagine building a house without locks on any doors, then trying to make it secure by adding cameras. The fundamental design is flawed.

### Example: No Rate Limiting on Password Reset

```
Insecure Design:
  1. User requests password reset
  2. System sends a 4-digit code to email
  3. User enters the code on the website
  4. No limit on how many times you can try the code!

Attack:
  Attacker tries all 10,000 combinations (0000-9999)
  With automation, this takes minutes.
  
  Result: Attacker resets anyone's password! 😱
```

### Secure Design

```
✅ Use a 6+ digit code (or better: a long random token)
✅ Limit attempts (e.g., 5 tries, then lock)
✅ Make the code expire after 10 minutes
✅ Rate limit the API (e.g., 3 requests per minute)
✅ Notify the user when a reset is requested
```

### Another Example: Business Logic Flaw

```python
# ❌ INSECURE DESIGN — No business logic validation
@app.route('/api/transfer', methods=['POST'])
def transfer_money():
    from_account = request.json['from']
    to_account = request.json['to']
    amount = request.json['amount']
    
    # Just transfer the money — no checks!
    database.deduct(from_account, amount)
    database.credit(to_account, amount)
    return {"status": "success"}

# What's wrong?
# - What if amount is negative? (Attacker sends -1000 to STEAL money)
# - What if from_account doesn't have enough balance?
# - What if from_account belongs to someone else?
# - No transaction logging!
```

**Why is this dangerous?** The code has no business logic validation at all. An attacker could:
- Send a **negative amount** (e.g., `-1000`), which would deduct from the `to_account` and credit the `from_account` — effectively stealing money
- Transfer money from someone else's account since there's no ownership check
- Transfer more than the account balance, creating a negative balance

```python
# ✅ SECURE DESIGN — Proper business logic validation
@app.route('/api/transfer', methods=['POST'])
def transfer_money():
    from_account = request.json['from']
    to_account = request.json['to']
    amount = request.json['amount']
    
    # 1. Validate the amount is positive
    if not isinstance(amount, (int, float)) or amount <= 0:
        return {"error": "Amount must be a positive number"}, 400
    
    # 2. Verify the sender owns the from_account
    if from_account != current_user.account_id:
        return {"error": "You can only transfer from your own account"}, 403
    
    # 3. Check sufficient balance
    balance = database.get_balance(from_account)
    if balance < amount:
        return {"error": "Insufficient funds"}, 400
    
    # 4. Use a database transaction (atomic — all or nothing)
    try:
        with database.transaction():
            database.deduct(from_account, amount)
            database.credit(to_account, amount)
            audit_log.record(from_account, to_account, amount, current_user.id)
    except Exception as e:
        return {"error": "Transfer failed"}, 500
    
    return {"status": "success"}
```

**What we fixed:**
- **Amount validation** — ensures the amount is a positive number, blocking negative transfer tricks
- **Ownership check** — verifies the logged-in user actually owns the source account
- **Balance check** — prevents overdraft/negative balance situations
- **Database transaction** — uses `with database.transaction()` to make the operation atomic (either both deduct and credit happen, or neither does)
- **Audit logging** — records every transfer for accountability and investigation

### How to Prevent:

```
✅ Do threat modeling during the design phase (before coding)
✅ Use secure design patterns (defense in depth, least privilege)
✅ Write abuse cases alongside use cases
✅ Rate limit all sensitive operations
✅ Validate business logic, not just input format
✅ Have security architects review the design
```

## A05: Security Misconfiguration ⚙️

### What Is It?

This is the **most common** vulnerability. It happens when systems, frameworks, or cloud services are not configured securely — using defaults, leaving unnecessary features enabled, or exposing sensitive information.

> Imagine buying a fancy lock for your door but leaving the default password "0000". That's security misconfiguration.

### Common Examples

```
❌ Default admin credentials (admin/admin, admin/password)
❌ Unnecessary ports open (SSH on port 22 open to the whole internet)
❌ Debug mode enabled in production
❌ Directory listing enabled on web server
❌ Stack traces / error details shown to users
❌ Unnecessary services running
❌ Default cloud storage permissions (public S3 buckets!)
❌ CORS set to allow all origins (Access-Control-Allow-Origin: *)
```

### Example: Debug Mode in Production (Django)

```python
# settings.py

# ❌ VULNERABLE — Debug mode ON in production
DEBUG = True  # Shows detailed error pages with source code, SQL queries, and settings!

# When an error occurs, Django shows:
# - Full stack trace with source code
# - All environment variables (including API keys!)
# - Database query details
# - Installed middleware and apps
# All visible to ANYONE who triggers an error!
```

```python
# ✅ SECURE — Debug mode OFF in production
import os

DEBUG = os.environ.get('DJANGO_DEBUG', 'False').lower() == 'true'

# In production: set DJANGO_DEBUG=False (or don't set it at all)
# Custom error pages are shown instead of detailed debug info
```

**What we fixed:**
- `DEBUG` is now controlled by an **environment variable** instead of being hardcoded
- In production, the env var is set to `False` (or not set at all, defaulting to `False`)
- With debug off, Django shows a generic error page, not the detailed debug page that leaks internal information

### Example: Insecure Nginx Configuration

```nginx
# ❌ VULNERABLE — Exposing server version and status
server {
    listen 80;
    server_name example.com;
    
    # Server version visible in response headers (helps attackers identify vulnerabilities)
    # server_tokens on;  (this is the default!)
    
    # Status page accessible to everyone
    location /nginx_status {
        stub_status on;
    }
    
    # Directory listing enabled (attackers can browse your files!)
    location /files/ {
        autoindex on;
    }
}

# ✅ SECURE
server {
    listen 443 ssl http2;
    server_name example.com;
    
    # Hide server version
    server_tokens off;
    
    # Security headers
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-Frame-Options "DENY" always;
    add_header X-XSS-Protection "1; mode=block" always;
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
    add_header Content-Security-Policy "default-src 'self'" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;
    
    # Restrict status page to internal network only
    location /nginx_status {
        stub_status on;
        allow 10.0.0.0/8;      # Internal network only
        deny all;               # Block everyone else
    }
    
    # Disable directory listing
    location /files/ {
        autoindex off;
    }
}
```

**What each security header does:**
- **`server_tokens off`** — Hides the Nginx version number from response headers (attackers use version info to find known vulnerabilities)
- **`X-Content-Type-Options: nosniff`** — Prevents browsers from "guessing" the content type, stopping MIME-type sniffing attacks
- **`X-Frame-Options: DENY`** — Prevents your site from being embedded in iframes, stopping clickjacking attacks
- **`X-XSS-Protection`** — Enables browser's built-in XSS filter
- **`Strict-Transport-Security`** — Forces browsers to always use HTTPS for your domain for the next year
- **`Content-Security-Policy`** — Controls which resources the browser is allowed to load, preventing XSS
- **`Referrer-Policy`** — Controls how much URL info is shared when navigating away from your site

### How to Prevent:

```
✅ Remove default credentials immediately
✅ Disable debug mode in production
✅ Keep software updated and patched
✅ Use IaC scanning tools (Checkov, tfsec) to catch misconfigurations
✅ Harden server configurations (CIS Benchmarks)
✅ Remove unnecessary features, components, and documentation
✅ Automate configuration checks in CI/CD
✅ Use security headers on all responses
✅ Review cloud permissions regularly (AWS, Azure, GCP)
```

## A06: Vulnerable and Outdated Components 📦

### What Is It?

This is about using **libraries, frameworks, or other software components** that have known security vulnerabilities. If you use an old version of a package with a known exploit, attackers can use that exploit against your application.

> Imagine using a lock that was recalled because it can be picked in 2 seconds. You just haven't updated to the newer model.

### Why It's So Dangerous

```
Your Application Code (1,000 lines)
         ↓
Depends on 50 npm packages
         ↓
Those 50 packages depend on 500 more packages (transitive dependencies)
         ↓
ANY one of those 550 packages could have a vulnerability
         ↓
You're only as secure as your weakest dependency!
```

### Real-World Example: Log4Shell (2021)

```
Log4j — A widely used Java logging library
CVE-2021-44228 (Log4Shell)
CVSS Score: 10.0 (MAXIMUM severity)

What happened:
  A single line of code in Log4j allowed attackers to execute 
  arbitrary code on ANY server using Log4j, just by sending a 
  specially crafted log message.

Impact:
  - Affected millions of applications worldwide
  - Minecraft, iCloud, Twitter, Amazon, Tesla, and more
  - One of the worst vulnerabilities in history

Lesson:
  One vulnerable dependency can compromise your entire system.
```

### How to Check for Vulnerable Dependencies

```bash
# JavaScript/Node.js
npm audit                     # Built-in vulnerability scanner
npm audit fix                 # Auto-fix vulnerabilities

# Python
pip-audit                     # Scans installed packages

# Java/Maven
mvn dependency-check:check    # OWASP Dependency-Check

# Universal tools (work with any language)
trivy fs .                    # Scan filesystem for vulnerable packages
snyk test                     # Snyk scanner
```

**What each command does:**
- **`npm audit`** — Checks every npm package in your project against the GitHub Advisory Database and reports known vulnerabilities with severity levels
- **`pip-audit`** — Does the same for Python packages, checking against the Python Advisory Database
- **`trivy fs .`** — Scans your entire project directory and identifies vulnerable packages regardless of the language
- **`snyk test`** — Commercial (free tier available) scanner that provides detailed fix recommendations

### Automate Dependency Updates

```yaml
# .github/dependabot.yml — GitHub Dependabot Configuration
# This file tells GitHub to automatically check for outdated dependencies
# and create Pull Requests to update them

version: 2
updates:
  # Check npm dependencies weekly
  - package-ecosystem: "npm"
    directory: "/"
    schedule:
      interval: "weekly"       # Check every week
    open-pull-requests-limit: 10  # Max 10 PRs at a time
    labels:
      - "dependencies"         # Label PRs for easy filtering
      - "security"

  # Check Docker base images weekly
  - package-ecosystem: "docker"
    directory: "/"
    schedule:
      interval: "weekly"

  # Check GitHub Actions versions
  - package-ecosystem: "github-actions"
    directory: "/"
    schedule:
      interval: "weekly"
```

**How Dependabot works:**
- It runs on GitHub's schedule (weekly in this config)
- It checks each package against vulnerability databases
- When it finds an outdated or vulnerable package, it automatically creates a **Pull Request** with the update
- You review the PR, check the tests pass, and merge — that's it!
- It even handles Docker base images and GitHub Actions versions

### How to Prevent:

```
✅ Regularly scan dependencies (npm audit, pip-audit, Trivy, Snyk)
✅ Enable automated dependency updates (Dependabot, Renovate Bot)
✅ Remove unused dependencies
✅ Pin dependency versions (use lock files: package-lock.json, Pipfile.lock)
✅ Subscribe to security advisories for your key dependencies
✅ Have a process to respond quickly when a critical CVE is published
```

## A07: Identification and Authentication Failures 🔑

### What Is It?

Previously called "Broken Authentication." This covers weaknesses in how your application verifies **who you are** (identification/authentication) — things like weak passwords, session hijacking, missing MFA, and credential stuffing attacks.

> Imagine a building where you can enter by just saying "I'm the CEO" without showing any ID. That's broken authentication.

### Common Mistakes

```
❌ Allowing weak passwords (password123, 123456)
❌ Not implementing multi-factor authentication (MFA)
❌ Exposing session IDs in URLs
❌ Not expiring sessions after logout
❌ Using plain text or weak hashing for passwords
❌ Not rate limiting login attempts (allows brute force)
❌ Default credentials on applications/databases
```

### Vulnerable Code: No Brute Force Protection

```python
# ❌ VULNERABLE — No rate limiting on login attempts
@app.route('/login', methods=['POST'])
def login():
    username = request.form['username']
    password = request.form['password']
    
    user = database.find_user(username)
    if user and bcrypt.checkpw(password.encode(), user.password_hash):
        session['user_id'] = user.id
        return redirect('/dashboard')
    
    return "Invalid credentials", 401

# An attacker can try millions of passwords with no limit!
# This is called a "brute force attack"
```

### Fixed Code: With Rate Limiting and Account Lockout

```python
from flask_limiter import Limiter
import time

limiter = Limiter(app, default_limits=["200 per day"])

# Track failed login attempts in the database or cache
failed_attempts = {}  # In production, use Redis or a database

# ✅ SECURE — Rate limiting + account lockout + logging
@app.route('/login', methods=['POST'])
@limiter.limit("5 per minute")  # Max 5 login attempts per minute per IP
def login():
    username = request.form['username']
    password = request.form['password']
    
    # Check if account is locked
    if username in failed_attempts:
        attempts, lockout_time = failed_attempts[username]
        if attempts >= 5 and time.time() < lockout_time:
            remaining = int(lockout_time - time.time())
            return f"Account locked. Try again in {remaining} seconds.", 429
    
    user = database.find_user(username)
    if user and bcrypt.checkpw(password.encode(), user.password_hash):
        # Reset failed attempts on successful login
        failed_attempts.pop(username, None)
        session['user_id'] = user.id
        session.permanent = True
        app.permanent_session_lifetime = timedelta(hours=1)  # Session expires in 1 hour
        return redirect('/dashboard')
    
    # Track failed attempt
    if username in failed_attempts:
        attempts, _ = failed_attempts[username]
        failed_attempts[username] = (attempts + 1, time.time() + 300)  # Lock for 5 min
    else:
        failed_attempts[username] = (1, time.time() + 300)
    
    # Log the failed attempt (for security monitoring)
    security_log.warning(f"Failed login attempt for user: {username} from IP: {request.remote_addr}")
    
    # Return a generic message — don't reveal if username exists!
    return "Invalid credentials", 401
```

**What we fixed:**
- **`@limiter.limit("5 per minute")`** — Flask-Limiter restricts each IP address to 5 login requests per minute. After that, requests get a `429 Too Many Requests` response.
- **Account lockout** — After 5 failed attempts for the same username, the account is locked for 5 minutes (300 seconds). This stops automated password guessing.
- **Session expiry** — `permanent_session_lifetime = timedelta(hours=1)` means the session cookie expires after 1 hour of inactivity, reducing the window for session hijacking.
- **Security logging** — Failed attempts are logged with the username and IP address, allowing security teams to detect attacks.
- **Generic error message** — `"Invalid credentials"` doesn't reveal whether the username or password was wrong, preventing username enumeration.

### How to Prevent:

```
✅ Enforce strong password policies (min 8 chars, complexity)
✅ Implement MFA (multi-factor authentication)
✅ Rate limit login attempts
✅ Implement account lockout after failed attempts
✅ Use secure session management (HttpOnly, Secure, SameSite cookies)
✅ Never expose session IDs in URLs
✅ Hash passwords with bcrypt/Argon2 (never MD5/SHA1)
✅ Return generic error messages ("Invalid credentials" — not "Wrong password")
```

## A08: Software and Data Integrity Failures 📝

### What Is It?

This happens when your application relies on software, updates, or data from untrusted sources **without verifying their integrity**. Think of it as using a package that arrived with a broken seal — someone might have tampered with it.

> Imagine downloading software from the internet without checking if it's the original version. Someone could have modified it to include malware.

### Example: Supply Chain Attack

```
The SolarWinds Attack (2020):
  1. Attackers compromised SolarWinds' build system
  2. Injected malicious code into a software update
  3. 18,000+ organizations installed the "trusted" update
  4. Attackers had backdoor access to government agencies and Fortune 500 companies

Lesson: Even trusted software updates can be compromised!
```

### Vulnerable: Insecure CI/CD Pipeline

```yaml
# ❌ VULNERABLE — Pulling scripts from the internet without verification
steps:
  - name: Setup
    run: |
      # Downloading and running a random script from the internet!
      # What if someone compromises this URL?
      curl -s https://some-website.com/setup.sh | bash
      
  - name: Build
    # Using a third-party GitHub Action without pinning to a specific SHA
    # If the action owner pushes malicious code, your pipeline runs it!
    uses: some-random-org/build-action@main
```

**Why is this dangerous?**
- `curl | bash` downloads a script and immediately runs it — if the website is compromised or someone performs a man-in-the-middle attack, you execute malicious code
- Using `@main` for GitHub Actions means you always get the latest code — if the action's repository is compromised, your pipeline runs the attacker's code

### Secure CI/CD Pipeline

```yaml
# ✅ SECURE — Verify integrity and pin versions
steps:
  - name: Setup
    run: |
      # Download the script first
      curl -o setup.sh https://some-website.com/setup.sh
      
      # Verify the checksum (hash) to ensure it hasn't been tampered with
      echo "a]c3d4e5f6...expected_sha256_hash  setup.sh" | sha256sum -c -
      
      # Only run if checksum matches
      bash setup.sh
      
  - name: Build
    # Pin to a specific commit SHA — immutable, can't be changed by the owner
    uses: some-random-org/build-action@a1b2c3d4e5f6789012345678abcdef1234567890
```

**What we fixed:**
- The script is **downloaded first** (not piped directly to bash) so we can verify it before running
- **SHA256 checksum verification** — we compare the downloaded file's hash against a known-good hash. If even a single byte was changed, the hash won't match and the script won't run
- GitHub Actions are **pinned to a specific commit SHA** instead of a branch name. Commit SHAs are immutable — even if the repository owner pushes malicious code, your pipeline still uses the exact version you verified

### How to Prevent:

```
✅ Verify checksums/signatures of downloaded software
✅ Pin dependencies to specific versions (not "latest")
✅ Use SBOM (Software Bill of Materials) to track all components
✅ Sign your artifacts and verify signatures before deployment
✅ Use trusted package registries (and mirror them internally)
✅ Review CI/CD pipeline dependencies regularly
✅ Implement code review for infrastructure and pipeline changes
```

## A09: Security Logging and Monitoring Failures 📊

### What Is It?

If you can't **detect** an attack, you can't **respond** to it. This vulnerability is about not having enough logging, monitoring, and alerting to detect security incidents.

> Imagine a bank with no security cameras and no alarm system. Robbers could walk in and out, and nobody would know until morning.

### What Should You Log?

```
✅ All authentication events (login, logout, failed logins)
✅ Access control failures (unauthorized access attempts)
✅ Input validation failures (potential injection attempts)
✅ Application errors and exceptions
✅ Admin activities (user creation, permission changes)
✅ API calls (especially to sensitive endpoints)
✅ Data access to sensitive records

❌ Never log passwords, credit card numbers, or personal data!
❌ Never log session tokens or API keys!
```

### Example: Proper Security Logging (Python)

```python
import logging
from datetime import datetime

# Set up a security-specific logger
security_logger = logging.getLogger('security')
security_logger.setLevel(logging.INFO)

# Log to file with proper formatting
handler = logging.FileHandler('security.log')
formatter = logging.Formatter(
    '%(asctime)s | %(levelname)s | %(message)s',
    datefmt='%Y-%m-%d %H:%M:%S'
)
handler.setFormatter(formatter)
security_logger.addHandler(handler)

# ✅ Log authentication events
def login(username, password):
    user = database.find_user(username)
    
    if user and bcrypt.checkpw(password.encode(), user.password_hash):
        security_logger.info(
            f"LOGIN_SUCCESS | user={username} | ip={request.remote_addr} | "
            f"user_agent={request.headers.get('User-Agent')}"
        )
        return create_session(user)
    else:
        security_logger.warning(
            f"LOGIN_FAILURE | user={username} | ip={request.remote_addr} | "
            f"user_agent={request.headers.get('User-Agent')}"
        )
        return None

# ✅ Log access control failures
def access_admin_panel(user):
    if not user.is_admin:
        security_logger.warning(
            f"ACCESS_DENIED | user={user.username} | resource=/admin | "
            f"ip={request.remote_addr}"
        )
        return "Access Denied", 403
    
    security_logger.info(
        f"ADMIN_ACCESS | user={user.username} | ip={request.remote_addr}"
    )
    return render_admin_panel()
```

**What this logging does:**
- Creates a **dedicated security logger** that writes to `security.log` — separate from application logs for easy analysis
- Logs **successful logins** with username, IP address, and User-Agent (browser info) — useful for detecting account takeovers
- Logs **failed logins** as warnings — a spike in these could indicate a brute-force attack
- Logs **access control failures** — repeated denials from the same user/IP may indicate an attacker probing for unauthorized access
- Includes the **IP address** in every log — essential for tracing attacks back to their source
- **Never logs the password** — only the username and metadata

### Example Log Output

```
2025-01-15 10:23:45 | INFO    | LOGIN_SUCCESS | user=alice | ip=192.168.1.10 | user_agent=Mozilla/5.0
2025-01-15 10:24:01 | WARNING | LOGIN_FAILURE | user=admin | ip=203.0.113.42 | user_agent=python-requests/2.28
2025-01-15 10:24:02 | WARNING | LOGIN_FAILURE | user=admin | ip=203.0.113.42 | user_agent=python-requests/2.28
2025-01-15 10:24:03 | WARNING | LOGIN_FAILURE | user=admin | ip=203.0.113.42 | user_agent=python-requests/2.28
2025-01-15 10:24:04 | WARNING | ACCESS_DENIED | user=bob | resource=/admin | ip=192.168.1.15
```

Looking at this log, you can immediately see:
- Alice logged in normally ✅
- Someone at IP `203.0.113.42` is trying to brute-force the `admin` account 🚨 (3 failures in 3 seconds using Python scripts)
- Bob tried to access the admin panel without permission 🚨

### How to Prevent:

```
✅ Log all security-relevant events
✅ Use structured logging (JSON format) for easy parsing
✅ Centralize logs (ELK Stack, Loki, Splunk, Datadog)
✅ Set up alerts for suspicious patterns (e.g., 10+ failed logins in 1 minute)
✅ Monitor in real-time, don't just collect logs
✅ Have an incident response plan
✅ Test your logging — can you actually detect an attack?
✅ Retain logs for an appropriate period (30-90 days minimum)
```

## A10: Server-Side Request Forgery (SSRF) 🌐

### What Is It?

SSRF happens when an attacker can make **your server** send requests to unintended locations. The attacker tricks your server into accessing internal resources (databases, cloud metadata, admin panels) that should not be accessible from the outside.

> Imagine asking a trusted employee to "go check if the vault is open." They have access to the vault — you don't. But by making them check, you learn the vault's status.

### How SSRF Works

```
Normal Flow:
  User → Your Server → External API (https://api.example.com/data)
  
SSRF Attack:
  User → Your Server → Internal Service (http://169.254.169.254/metadata)
                        ↑
                        This is the AWS metadata endpoint!
                        It contains IAM credentials, instance info, etc.
                        The user can't reach it directly, but YOUR SERVER can!
```

### Vulnerable Code (Node.js)

```javascript
// ❌ VULNERABLE — Fetching a URL provided by the user without validation
app.get('/api/fetch-url', async (req, res) => {
    const url = req.query.url;  // User provides ANY URL
    
    // Your server fetches whatever URL the user gives it!
    const response = await fetch(url);
    const data = await response.text();
    
    res.send(data);
});

// Attack: GET /api/fetch-url?url=http://169.254.169.254/latest/meta-data/iam/security-credentials/
// Result: Your server fetches AWS credentials and sends them to the attacker! 😱
```

**Why is this dangerous?** The code blindly fetches whatever URL the user provides. Since the request comes FROM your server (which is inside the VPC/cloud network), it can access:
- **AWS metadata endpoint** (`169.254.169.254`) — contains IAM credentials that could give the attacker full access to your AWS account
- **Internal services** — databases, admin panels, monitoring dashboards
- **Localhost services** — applications running on the same server

### Fixed Code

```javascript
const { URL } = require('url');
const dns = require('dns').promises;

// ✅ SECURE — Validate and restrict the URL
app.get('/api/fetch-url', async (req, res) => {
    const userUrl = req.query.url;
    
    try {
        const parsed = new URL(userUrl);
        
        // 1. Only allow HTTPS
        if (parsed.protocol !== 'https:') {
            return res.status(400).json({ error: 'Only HTTPS URLs are allowed' });
        }
        
        // 2. Block internal/private IP ranges
        const blockedPatterns = [
            /^localhost$/i,
            /^127\./,
            /^10\./,
            /^172\.(1[6-9]|2[0-9]|3[0-1])\./,
            /^192\.168\./,
            /^169\.254\./,           // AWS metadata endpoint
            /^0\./,
            /^\[::1\]/,             // IPv6 localhost
        ];
        
        // Resolve the hostname to an IP address
        const addresses = await dns.resolve4(parsed.hostname);
        
        for (const addr of addresses) {
            for (const pattern of blockedPatterns) {
                if (pattern.test(addr) || pattern.test(parsed.hostname)) {
                    return res.status(400).json({ error: 'Access to internal resources is not allowed' });
                }
            }
        }
        
        // 3. Only allow specific domains (allowlist approach — even more secure)
        const allowedDomains = ['api.example.com', 'cdn.example.com'];
        if (!allowedDomains.includes(parsed.hostname)) {
            return res.status(400).json({ error: 'Domain not in allowlist' });
        }
        
        // 4. Fetch with timeout to prevent hanging
        const controller = new AbortController();
        const timeout = setTimeout(() => controller.abort(), 5000);
        
        const response = await fetch(userUrl, { signal: controller.signal });
        clearTimeout(timeout);
        
        const data = await response.text();
        res.send(data);
        
    } catch (error) {
        res.status(400).json({ error: 'Invalid URL' });
    }
});
```

**What we fixed:**
- **Protocol check** — only allows `https:`, blocking `http:`, `file:`, `ftp:`, `gopher:`, and other protocols that could be abused
- **DNS resolution check** — resolves the hostname to an IP and checks it against blocked private IP ranges. This catches tricks like using a domain that resolves to `127.0.0.1`
- **Private IP blocking** — blocks all private/internal IP ranges including `169.254.x.x` (AWS metadata), `10.x.x.x`, `192.168.x.x`, `172.16-31.x.x`, and localhost
- **Domain allowlist** — the most secure approach: only allows requests to specifically approved domains
- **Request timeout** — `AbortController` with a 5-second timeout prevents your server from hanging if the target is slow or unresponsive

### How to Prevent:

```
✅ Validate and sanitize all user-supplied URLs
✅ Use an allowlist of permitted domains/IPs
✅ Block requests to private/internal IP ranges
✅ Block requests to cloud metadata endpoints (169.254.169.254)
✅ Use a DNS resolution check (hostname could resolve to internal IP)
✅ Disable unnecessary URL schemes (only allow http/https)
✅ Use network segmentation — application servers shouldn't have access to everything
✅ On AWS: Use IMDSv2 (requires token-based access to metadata)
```
