# 16.3 — SAST — Static Application Security Testing (SonarQube, Semgrep, CodeQL)

> **One Line Answer:** SAST (Static Application Security Testing) is the practice of **analyzing your source code for security vulnerabilities without running the application**. Think of it as a spellchecker, but instead of catching typos, it catches security bugs.

## 🤔 What is SAST?

Let's break it down word by word:

| Word | Meaning |
|------|---------|
| **Static** | Analyzing code **without executing it** (just reading the source code) |
| **Application** | Your software/application code |
| **Security** | Looking specifically for **security vulnerabilities** |
| **Testing** | Automated checking/scanning |

### The Simple Analogy

```
Think of your code as a building blueprint.

SAST = A structural engineer reviewing the blueprint BEFORE construction
       "Hey, this wall won't support the roof. Fix it before you build!"

vs.

DAST = An inspector testing the building AFTER it's built
       "I pushed this wall and it fell over. That's a problem."

SAST catches problems EARLIER and CHEAPER.
```

### How SAST Works (Under the Hood)

```
┌─────────────────────────────────────────────────────────────────┐
│                      How SAST Works                              │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Step 1: Parse the source code into an AST                      │
│          (Abstract Syntax Tree — a structured representation)    │
│                                                                  │
│  Step 2: Build a model of data flow                              │
│          (Track how data moves through the code)                │
│                                                                  │
│  Step 3: Apply security rules/patterns                           │
│          "Is user input going directly into a SQL query?"       │
│          "Is a password stored in plain text?"                  │
│          "Is there a hardcoded API key?"                        │
│                                                                  │
│  Step 4: Report findings with:                                   │
│          - Vulnerability type                                    │
│          - Severity (Critical, High, Medium, Low)               │
│          - Exact file and line number                            │
│          - How to fix it                                        │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### What SAST Can Find

```
✅ SQL Injection                    ✅ Command Injection
✅ Cross-Site Scripting (XSS)       ✅ Hardcoded secrets/passwords
✅ Path Traversal                   ✅ Insecure cryptography
✅ Buffer overflows                 ✅ Race conditions
✅ Insecure deserialization         ✅ SSRF vulnerabilities
✅ Use of unsafe functions          ✅ Missing input validation
✅ Security misconfigurations       ✅ Information leakage
```

### What SAST Cannot Find

```
❌ Runtime vulnerabilities (need DAST for that)
❌ Business logic flaws ("transfers should have limits")
❌ Authentication/authorization issues at the architectural level
❌ Configuration issues in production environments
❌ Vulnerabilities in third-party dependencies (need SCA for that)
```

## 🆚 SAST vs DAST vs SCA — What's the Difference?

| Feature | SAST | DAST | SCA |
|---------|------|------|-----|
| **What it scans** | Source code | Running application | Dependencies/libraries |
| **When it runs** | During development/build | During testing/staging | During build |
| **Needs running app?** | ❌ No | ✅ Yes | ❌ No |
| **Finds what?** | Code-level bugs | Runtime vulnerabilities | Known CVEs in packages |
| **False positives** | Higher | Lower | Very Low |
| **Speed** | Fast | Slower | Fast |
| **Example tools** | SonarQube, Semgrep, CodeQL | OWASP ZAP, Burp Suite | Snyk, Trivy, Dependabot |

> **Best practice:** Use ALL THREE together. They complement each other — each catches things the others miss.

```
         Your Security Scanning Strategy
┌─────────────────────────────────────────────┐
│                                             │
│    SAST ──── Scans your CODE                │
│      +                                      │
│    SCA ───── Scans your DEPENDENCIES        │
│      +                                      │
│    DAST ──── Scans your RUNNING APP         │
│      =                                      │
│    Comprehensive coverage 🛡️                │
│                                             │
└─────────────────────────────────────────────┘
```

## 🛠️ Tool #1: SonarQube

### What is SonarQube?

SonarQube is the **most popular** open-source SAST tool. It does more than just security — it also checks for code quality issues (bugs, code smells, duplications). But we'll focus on its **security scanning** capabilities.

```
SonarQube supports 30+ languages including:
Java, JavaScript, TypeScript, Python, C#, Go, PHP, Ruby, C/C++, Kotlin, Swift, and more
```

### SonarQube Architecture

```
┌──────────────────────────────────────────────────────────┐
│                    SonarQube Setup                        │
│                                                          │
│  Developer's Machine / CI Server                         │
│  ┌─────────────────────┐                                │
│  │   Sonar Scanner     │ ← Analyzes your code locally   │
│  │   (CLI tool)        │                                │

│            │ Sends results                               │
│            ▼                                             │
```

### Setting Up SonarQube with Docker

```bash
# Step 1: Start SonarQube server using Docker
docker run -d \
  --name sonarqube \
  -p 9000:9000 \
  -v sonarqube_data:/opt/sonarqube/data \
  -v sonarqube_logs:/opt/sonarqube/logs \
  -v sonarqube_extensions:/opt/sonarqube/extensions \
  sonarqube:community

# What this does:
# -d                  → Run in background (detached)
# --name sonarqube    → Name the container "sonarqube"
# -p 9000:9000        → Map port 9000 (SonarQube's web UI)
# -v sonarqube_data   → Persist SonarQube data across restarts
# -v sonarqube_logs   → Persist logs
# -v sonarqube_extensions → Persist installed plugins

# Step 2: Wait for it to start (takes ~1-2 minutes)
# Check status:
docker logs -f sonarqube
# Look for: "SonarQube is operational"

# Step 3: Open browser
# Go to http://localhost:9000
# Default credentials: admin / admin (CHANGE THIS IMMEDIATELY!)
```

### Running Your First Scan

```bash
# Step 1: Install Sonar Scanner CLI
# On macOS:
brew install sonar-scanner

# On Linux:
# Download from: https://docs.sonarqube.org/latest/analyzing-source-code/scanners/sonarscanner/

# On Windows:
# Download the ZIP, extract, and add to PATH

# Step 2: Create a configuration file in your project root
```

```properties
# sonar-project.properties (place in your project root)
sonar.projectKey=my-project
sonar.projectName=My Project
sonar.projectVersion=1.0

# Where your source code is
sonar.sources=src

# Language (optional — SonarQube auto-detects)
# sonar.language=javascript

# SonarQube server URL
sonar.host.url=http://localhost:9000

# Authentication token (generate in SonarQube UI → My Account → Security → Tokens)
sonar.token=sqp_your_token_here

# Exclude test files and node_modules from scanning
sonar.exclusions=**/node_modules/**,**/test/**,**/*.test.js
```

```bash
# Step 3: Run the scan
cd /path/to/your/project
sonar-scanner

# Output will look like:
# INFO: Scanner configuration file: /path/to/sonar-scanner.properties
# INFO: Project root configuration file: /path/to/project/sonar-project.properties
# INFO: 127 files indexed
# INFO: Sensor JavaScriptSensor [javascript]
# INFO: 45 source files to be analyzed
# INFO: ANALYSIS SUCCESSFUL, you can find the results at:
#       http://localhost:9000/dashboard?id=my-project
```

**What each configuration option does:**
- **`sonar.projectKey`** — A unique identifier for your project in SonarQube (like a primary key)
- **`sonar.sources`** — Tells the scanner where your source code lives
- **`sonar.host.url`** — The URL of your SonarQube server
- **`sonar.token`** — Authentication token so the scanner can upload results to the server
- **`sonar.exclusions`** — Glob patterns for files to skip (no point scanning `node_modules`)

### SonarQube in CI/CD (GitHub Actions)

```yaml
# .github/workflows/sonarqube.yml
name: SonarQube Analysis

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  sonarqube:
    runs-on: ubuntu-latest
    steps:
      # Step 1: Check out the code
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # Full history needed for accurate blame/change analysis

      # Step 2: Run SonarQube scan
      - name: SonarQube Scan
        uses: SonarSource/sonarqube-scan-action@master
        env:
          SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}    # Auth token (stored as GitHub Secret)
          SONAR_HOST_URL: ${{ secrets.SONAR_HOST_URL }}  # SonarQube server URL

      # Step 3: Check the Quality Gate result
      - name: SonarQube Quality Gate
        uses: SonarSource/sonarqube-quality-gate-action@master
        timeout-minutes: 5
        env:
          SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
        # If Quality Gate FAILS, this step fails, and the PR is blocked!
```

**What this pipeline does:**
- **`fetch-depth: 0`** — Clones the full git history (not just the latest commit). SonarQube uses this to show which developer introduced each issue and to track new vs. existing issues.
- **SonarQube Scan** — Runs the sonar-scanner, analyzes the code, and uploads results to your SonarQube server.
- **Quality Gate** — SonarQube has a concept called "Quality Gate" — a set of conditions your code must pass (e.g., "no new critical vulnerabilities", "code coverage > 80%"). If the gate fails, this step fails, which blocks the PR from being merged.

### SonarQube Quality Gate — What Is It?

```
A Quality Gate is like a bouncer at a nightclub — 
your code doesn't get in unless it meets the standards.

Default Quality Gate conditions:
┌────────────────────────────────────┬──────────────┐
│ Condition                          │ Threshold    │
├────────────────────────────────────┼──────────────┤
│ New Bugs                           │ 0            │
│ New Vulnerabilities                │ 0            │
│ New Security Hotspots reviewed     │ 100%         │
│ New Code Coverage                  │ ≥ 80%        │
│ New Code Duplication               │ ≤ 3%         │
└────────────────────────────────────┴──────────────┘

If any condition fails → Quality Gate = FAILED → PR blocked ❌
If all conditions pass → Quality Gate = PASSED → PR can merge ✅
```

### SonarQube Security Categories

SonarQube categorizes findings into:

| Category | What It Means | Example |
|----------|--------------|---------|
| **Vulnerability** | A confirmed security issue that needs fixing | SQL injection, XSS, hardcoded password |
| **Security Hotspot** | Code that MIGHT be a security issue — needs human review | Using `Math.random()` for security purposes, cookie without `Secure` flag |
| **Bug** | A code defect (not necessarily security-related) | Null pointer dereference, infinite loop |
| **Code Smell** | Maintainability issue (not a bug or vulnerability) | Unused variable, too complex method |

## 🛠️ Tool #2: Semgrep

### What is Semgrep?

Semgrep is a **fast, open-source, lightweight** SAST tool. Unlike SonarQube (which is a full platform with a server), Semgrep runs **as a simple CLI tool** — no server needed.

The killer feature of Semgrep is that you can write **custom rules** that look like the code you're searching for. It uses a pattern-matching approach that feels very natural.

```
SonarQube = Full platform (server, database, web UI)
Semgrep   = Lightweight CLI tool (just install and run)
```

### Why Semgrep is Great

```
✅ Free and open-source (Community rules are free)
✅ No server needed — just a CLI tool
✅ Super fast — can scan millions of lines in seconds
✅ Easy custom rules — write patterns that look like code
✅ 2,000+ pre-built rules covering OWASP Top 10
✅ Supports 30+ languages
✅ Great CI/CD integration
✅ Low false positive rate compared to many SAST tools
```

### Installing and Running Semgrep

```bash
# Install Semgrep
# Using pip (Python)
pip install semgrep

# Using Homebrew (macOS)
brew install semgrep

# Using Docker
docker pull semgrep/semgrep

# Verify installation
semgrep --version
```

### Running Your First Scan

```bash
# Scan your project with the "auto" config (recommended rules)
# "auto" includes rules for OWASP Top 10, best practices, and common bugs
semgrep scan --config auto .

# Scan with specific rulesets
semgrep scan --config p/owasp-top-ten .      # OWASP Top 10 rules
semgrep scan --config p/security-audit .     # Comprehensive security audit
semgrep scan --config p/python .              # Python-specific rules
semgrep scan --config p/javascript .          # JavaScript-specific rules
semgrep scan --config p/docker .              # Dockerfile security
semgrep scan --config p/terraform .           # Terraform/IaC security

# Scan a specific file
semgrep scan --config auto src/app.py

# Output in JSON format (useful for CI/CD)
semgrep scan --config auto --json -o results.json .
```

**What each command does:**
- **`--config auto`** — Uses Semgrep's recommended ruleset that covers the most common and important security issues. It's smart and applies language-appropriate rules based on the files it finds.
- **`--config p/owasp-top-ten`** — Uses a curated ruleset specifically targeting the OWASP Top 10 vulnerabilities
- **`.`** — Scans the current directory and all subdirectories
- **`--json -o results.json`** — Outputs results in JSON format to a file for programmatic processing (useful for dashboards, CI/CD gates, etc.)

### Writing Custom Semgrep Rules

This is where Semgrep really shines. Writing rules feels like writing the code you're looking for:

```yaml
# custom-rules.yml

rules:
  # Rule 1: Detect hardcoded AWS access keys
  - id: hardcoded-aws-key
    patterns:
      - pattern: $KEY = "AKIA..."      # AWS access keys start with "AKIA"
    message: |
      Hardcoded AWS access key detected! 
      Use environment variables or AWS IAM roles instead.
      See: https://docs.aws.amazon.com/IAM/latest/UserGuide/best-practices.html
    severity: ERROR
    languages: [python, javascript, typescript, java, go]

  # Rule 2: Detect eval() usage (code injection risk)
  - id: no-eval
    pattern: eval(...)
    message: |
      eval() is dangerous! It executes arbitrary code.
      If user input reaches eval(), it's a code injection vulnerability.
      Use safer alternatives (JSON.parse for JSON, ast.literal_eval for Python).
    severity: ERROR
    languages: [python, javascript]

  # Rule 3: Detect missing authentication decorator in Flask
  - id: flask-route-no-auth
    patterns:
      - pattern: |
          @app.route(...)
          def $FUNC(...):
              ...
      - pattern-not: |
          @app.route(...)
          @login_required
          def $FUNC(...):
              ...
    message: |
      Flask route without @login_required decorator.
      This endpoint may be accessible without authentication.
      Add @login_required if this route requires authentication.
    severity: WARNING
    languages: [python]

  # Rule 4: Detect HTTP URLs (should be HTTPS)
  - id: use-https
    pattern: '"http://..."'
    message: |
      HTTP URL detected. Use HTTPS instead for secure communication.
    severity: WARNING
    languages: [python, javascript, typescript, java]
```

```bash
# Run your custom rules
semgrep scan --config custom-rules.yml .

# Combine custom rules with built-in rules
semgrep scan --config auto --config custom-rules.yml .
```

**Understanding the rule syntax:**
- **`id`** — A unique name for the rule (used for filtering and suppressing)
- **`pattern`** — The code pattern to search for. `$KEY` is a metavariable that matches any variable name. `...` matches any number of arguments or statements.
- **`patterns`** — When you need multiple conditions (AND logic). Use `pattern-not` for "this pattern but NOT this pattern" (good for checking missing things like decorators).
- **`message`** — What to show the developer when this rule matches. Include the fix!
- **`severity`** — `ERROR` (must fix), `WARNING` (should fix), or `INFO` (nice to know)
- **`languages`** — Which programming languages this rule applies to

### Semgrep in CI/CD (GitHub Actions)

```yaml
# .github/workflows/semgrep.yml
name: Semgrep Security Scan

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  semgrep:
    runs-on: ubuntu-latest
    container:
      image: semgrep/semgrep  # Use the official Semgrep Docker image
    steps:
      - uses: actions/checkout@v4

      # Run Semgrep with auto config
      - name: Run Semgrep
        run: semgrep scan --config auto --error --json -o semgrep-results.json .
        # --error flag makes Semgrep exit with code 1 if findings are found
        # This causes the CI job to FAIL, blocking the PR

      # Upload results as an artifact for review
      - name: Upload Results
        if: always()  # Upload even if scan found issues
        uses: actions/upload-artifact@v4
        with:
          name: semgrep-results
          path: semgrep-results.json
```

**What this pipeline does:**
- Uses the official Semgrep Docker image (everything pre-installed)
- **`--error`** flag is the key — it makes `semgrep` exit with a non-zero exit code if it finds any issues, which causes the GitHub Actions job to **fail**, which blocks the PR from being merged
- **`--json -o semgrep-results.json`** — Saves results in JSON format
- The results file is uploaded as a **build artifact** so developers can download and review them even if the job fails

### Ignoring False Positives in Semgrep

Sometimes Semgrep flags code that you know is safe. Here's how to handle that:

```python
# Method 1: Inline comment to ignore a specific line
password_hash = hashlib.sha256(data.encode()).hexdigest()  # nosemgrep: insecure-hash-sha256
# We're using SHA256 for data integrity, NOT password hashing — this is intentional

# Method 2: Ignore a specific rule for a block
# nosemgrep: flask-route-no-auth
@app.route('/public/health')
def health_check():
    """This endpoint is intentionally public — it's a health check for load balancers"""
    return {"status": "healthy"}, 200
```

```yaml
# Method 3: .semgrepignore file (like .gitignore)
# Place in project root

# Ignore test files (they may have intentionally vulnerable code for testing)
tests/
*_test.py
*.test.js

# Ignore vendored/generated code
vendor/
generated/

# Ignore specific files
docs/examples/vulnerable-demo.py
```

## 🛠️ Tool #3: CodeQL

### What is CodeQL?

CodeQL is **GitHub's own** SAST tool. It's unique because it treats your code like a **database** — you can write SQL-like queries to find vulnerabilities. It's incredibly powerful and deeply integrated with GitHub.

```
CodeQL's approach:
  1. Parse your code into a database (CodeQL database)
  2. Run queries against that database
  3. Each query looks for a specific vulnerability pattern
  4. Results show up directly in GitHub Pull Requests!
```

### Why CodeQL is Special

```
✅ Free for public repositories on GitHub
✅ Deep analysis — understands data flow across functions and files
✅ Finds complex vulnerabilities that simpler tools miss
✅ Results appear directly in GitHub PR reviews
✅ Supports: JavaScript, TypeScript, Python, Java, C/C++, C#, Go, Ruby, Kotlin, Swift
✅ 2,000+ built-in security queries
✅ You can write custom queries (CodeQL language)
```

### How CodeQL Works — Step by Step

```
┌─────────────────────────────────────────────────────────────────┐
│                    How CodeQL Works                               │
│                                                                  │
│  Step 1: CREATE DATABASE                                        │
│  ┌──────────────┐        ┌──────────────────┐                   │
│  │  Your Source  │  ───►  │  CodeQL Database  │                  │
│  │  Code         │        │  (relational DB   │                  │
│  │  (.py, .js)  │        │   of your code)   │                  │
│  └──────────────┘        └──────────────────┘                   │
│                                   │                              │
│  Step 2: RUN QUERIES              │                              │
│  ┌──────────────────┐             │                              │
│  │  Security Queries │  ───────►  │  Query the database          │
│  │  (find SQL injection,│         │  like you'd query SQL        │
│  │   find XSS, etc.)   │         │                              │
│  └──────────────────┘             │                              │
│                                   ▼                              │
│  Step 3: GET RESULTS                                            │
│  ┌──────────────────────────────────────┐                       │
│  │  Results: file.py:42 — SQL Injection  │                      │
│  │  Data flows from request.args['name'] │                      │
│  │  to db.execute() without sanitization │                      │
│  └──────────────────────────────────────┘                       │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### CodeQL Data Flow Analysis — Why It's Powerful

Most SAST tools look for **patterns** in code. CodeQL does something smarter — it tracks how data **flows** through your application.

```python
# Consider this code spread across multiple files:

# file: routes.py
@app.route('/search')
def search():
    query = request.args.get('q')      # SOURCE: User input comes in here
    results = search_service(query)     # Data flows to another function
    return jsonify(results)

# file: services.py
def search_service(search_term):
    cleaned = search_term.strip()       # Data flows through processing
    return database_query(cleaned)      # Data flows to yet another function

# file: database.py
def database_query(term):
    query = f"SELECT * FROM products WHERE name LIKE '%{term}%'"  # SINK: SQL injection!
    return db.execute(query)

# Simple pattern-matching tools might NOT catch this because:
# - The user input (source) and the SQL query (sink) are in DIFFERENT files
# - The data passes through multiple functions

# CodeQL WILL catch this because it traces the data flow:
# request.args.get('q') → search_service(query) → database_query(cleaned) → db.execute(query)
# It sees that user-controlled data reaches a SQL query without sanitization!
```

### Setting Up CodeQL on GitHub (Easiest Method)

The easiest way to use CodeQL is through **GitHub Code Scanning**, which is built right into GitHub.

```yaml
# .github/workflows/codeql.yml
name: CodeQL Analysis

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
  schedule:
    - cron: '0 6 * * 1'  # Run weekly on Mondays at 6 AM (catches new vulnerability patterns)

jobs:
  analyze:
    name: Analyze
    runs-on: ubuntu-latest
    
    permissions:
      actions: read
      contents: read
      security-events: write  # Required to upload results to GitHub Security tab

    strategy:
      fail-fast: false
      matrix:
        # Specify which languages to analyze
        language: ['javascript', 'python']
        # CodeQL supports: javascript, python, java, csharp, cpp, go, ruby, kotlin, swift

    steps:
      # Step 1: Check out the code
      - name: Checkout repository
        uses: actions/checkout@v4

      # Step 2: Initialize CodeQL
      - name: Initialize CodeQL
        uses: github/codeql-action/init@v3
        with:
          languages: ${{ matrix.language }}
          # Use the 'security-and-quality' query suite for comprehensive scanning
          queries: security-and-quality
          # Options:
          #   security-extended  → Security queries only (fast)
          #   security-and-quality → Security + code quality (thorough)

      # Step 3: Auto-build the project (CodeQL needs to understand your code)
      - name: Autobuild
        uses: github/codeql-action/autobuild@v3
        # For compiled languages (Java, C++), CodeQL watches the build process
        # For interpreted languages (Python, JS), it just reads the source files

      # Step 4: Run CodeQL analysis
      - name: Perform CodeQL Analysis
        uses: github/codeql-action/analyze@v3
        with:
          category: "/language:${{ matrix.language }}"
          # Results are automatically uploaded to GitHub's Security tab
```

**What each part does:**
- **`schedule: cron`** — Runs the scan weekly even without code changes, because new vulnerability patterns are constantly being added to CodeQL's query libraries
- **`permissions: security-events: write`** — Allows the workflow to upload scan results to the GitHub Security tab
- **`matrix: language`** — Runs the analysis in parallel for each language (one job for JavaScript, one for Python)
- **`queries: security-and-quality`** — Uses the most comprehensive query set. `security-extended` is faster but finds fewer issues.
- **`autobuild`** — For compiled languages (Java, C#, C++), CodeQL needs to observe the compilation to understand the code. For interpreted languages (Python, JS), it reads files directly.
- **Results appear in:** GitHub repo → Security tab → Code scanning alerts

### Writing Custom CodeQL Queries

CodeQL has its own query language (looks like SQL mixed with logic programming):

```ql
/**
 * @name SQL injection from user input
 * @description Finds cases where user input flows into SQL queries
 * @kind path-problem
 * @problem.severity error
 * @security-severity 9.8
 * @tags security
 *       external/cwe/cwe-089
 */

import python
import semmle.python.dataflow.new.DataFlow
import semmle.python.dataflow.new.TaintTracking
import semmle.python.Concepts

// Define what we're looking for:
// SOURCE = user input (request parameters, form data, etc.)
// SINK = SQL query execution (db.execute, cursor.execute, etc.)

module SqlInjectionConfig implements DataFlow::ConfigSig {
  // Define the SOURCE: where user input enters the application
  predicate isSource(DataFlow::Node source) {
    // request.args, request.form, request.json, etc.
    source instanceof Concepts::RemoteFlowSource
  }

  // Define the SINK: where the data reaches a dangerous function
  predicate isSink(DataFlow::Node sink) {
    // Any SQL query execution
    sink instanceof Concepts::SqlExecution
  }
}

module SqlInjectionFlow = TaintTracking::Global<SqlInjectionConfig>;

from SqlInjectionFlow::PathNode source, SqlInjectionFlow::PathNode sink
where SqlInjectionFlow::flowPath(source, sink)
select sink.getNode(), source, sink,
  "This SQL query depends on a $@.", source.getNode(), "user-provided value"
```

**How this query works (simplified):**
- **`isSource`** — Defines where data enters: any remote/user input (HTTP request parameters, form fields, JSON body, etc.)
- **`isSink`** — Defines where data reaches danger: any SQL execution function
- **`TaintTracking::Global`** — CodeQL's engine that traces data flow from sources to sinks, even across function calls and files
- The `from ... where ... select` at the bottom is like SQL — it says "find all paths where tainted data flows from a source to a sink, and report them"

> You don't need to write custom queries day-to-day. The built-in queries cover 95% of common vulnerabilities. Custom queries are useful for organization-specific patterns.

## 🆚 SonarQube vs Semgrep vs CodeQL — Comparison

| Feature | SonarQube | Semgrep | CodeQL |
|---------|-----------|---------|--------|
| **Type** | Full platform (server + scanner) | Lightweight CLI tool | Database-driven analysis |
| **Setup** | Server + database required | Just install CLI | GitHub integration or CLI |
| **Cost** | Community (free) / Enterprise (paid) | Free (OSS) / Team (paid) | Free on GitHub public repos |
| **Speed** | Medium | Very fast | Slower (deep analysis) |
| **Analysis Depth** | Good | Good (pattern-based) | Excellent (data flow) |
| **Custom Rules** | XML-based (complex) | YAML-based (easy!) | QL language (powerful but complex) |
| **False Positives** | Medium | Low | Very low |
| **Languages** | 30+ | 30+ | 10+ |
| **Best For** | Code quality + security | Quick security checks + custom rules | Deep security analysis on GitHub |
| **CI/CD Integration** | Good | Excellent | Excellent (native GitHub) |
| **Beyond Security** | Yes (bugs, code smells, duplication) | Primarily security | Primarily security |

## 🎯 Practical Setup: Using All Three Together

Here's a real-world setup that uses all three tools at different stages:

```
Developer's Machine (Pre-Commit)
  └── Semgrep (instant feedback — catches obvious issues)

Pull Request (CI Pipeline)
  ├── Semgrep (fast scan — results in ~30 seconds)
  ├── CodeQL (deep scan — results in ~5-15 minutes)
  └── SonarQube (quality + security — results in ~2-5 minutes)

Weekly Scheduled Scan
  └── CodeQL with extended query suite (finds everything)
```

```yaml
# .github/workflows/complete-sast.yml — All tools in one pipeline
name: Complete SAST Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  # Job 1: Semgrep (fastest — runs first)
  semgrep:
    name: Semgrep Scan
    runs-on: ubuntu-latest
    container:
      image: semgrep/semgrep
    steps:
      - uses: actions/checkout@v4
      - run: semgrep scan --config auto --error --json -o semgrep.json .
      - uses: actions/upload-artifact@v4
        if: always()
        with:
          name: semgrep-results
          path: semgrep.json

  # Job 2: CodeQL (deepest analysis)
  codeql:
    name: CodeQL Analysis
    runs-on: ubuntu-latest
    permissions:
      security-events: write
    steps:
      - uses: actions/checkout@v4
      - uses: github/codeql-action/init@v3
        with:
          languages: python, javascript
          queries: security-and-quality
      - uses: github/codeql-action/autobuild@v3
      - uses: github/codeql-action/analyze@v3

  # Job 3: SonarQube (quality + security)
  sonarqube:
    name: SonarQube Analysis
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - uses: SonarSource/sonarqube-scan-action@master
        env:
          SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
          SONAR_HOST_URL: ${{ secrets.SONAR_HOST_URL }}
      - uses: SonarSource/sonarqube-quality-gate-action@master
        env:
          SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
```

**How this pipeline works:**
- All three jobs run **in parallel** (they're independent of each other), so you get results from all tools as fast as possible
- **Semgrep** finishes first (~30 seconds), giving quick feedback
- **SonarQube** finishes next (~2-5 minutes), providing quality gate results
- **CodeQL** finishes last (~5-15 minutes), providing the deepest analysis
- If ANY tool finds critical issues, the pipeline fails and the PR is blocked

## ⚠️ Handling False Positives — A Critical Skill

Every SAST tool will produce some **false positives** (flagging code that's actually safe). Learning to handle them is crucial.

### The False Positive Problem

```
Too many false positives → Developers ignore ALL findings → Real vulnerabilities get missed
                           (This is called "alert fatigue")

The goal: Keep false positive rate below 20%
```

### Strategies for Handling False Positives

```
1. TRIAGE: Review each finding and classify it
   ├── True Positive → Fix it!
   ├── False Positive → Suppress with a comment explaining why
   └── Won't Fix → Document the risk and accept it

2. TUNE YOUR TOOLS:
   ├── Disable rules that consistently produce false positives
   ├── Adjust severity thresholds
   └── Customize rules for your specific codebase

3. USE SUPPRESSION COMMENTS:
   # SonarQube:  // NOSONAR — explanation
   # Semgrep:    # nosemgrep: rule-id — explanation  
   # CodeQL:     // lgtm[rule-id] — explanation

4. USE BASELINE FILES:
   ├── Track existing issues separately from new issues
   └── Focus on "zero new vulnerabilities" rather than fixing everything at once
```
