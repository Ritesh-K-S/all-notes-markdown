# 16.4 — DAST — Dynamic Application Security Testing (OWASP ZAP, Burp Suite)

> **One Line Answer:** DAST is a security testing method where you attack your **running application** from the outside (like a hacker would) to find vulnerabilities — without ever looking at the source code.

## 🤔 Let's Start With a Story

Imagine you're a DevOps engineer at a company that just deployed a brand-new web application. Your developers wrote great code, and it passed all the unit tests. Everything looks perfect... right?

Then one day, a security researcher sends your company an email: *"Hey, your login page is vulnerable to SQL Injection. I can steal all your user passwords."* 😱

How did this happen? The code looked fine! Well, here's the thing — **some vulnerabilities only show up when the application is actually running**. A static code scanner might miss them because it doesn't know how the app behaves at runtime, how it responds to weird inputs, or how different components interact.

That's exactly where **DAST** comes in. It tests your app the way a real attacker would — by poking it, sending bad inputs, and seeing what breaks.

---

## ✅ What is DAST?

**DAST** stands for **Dynamic Application Security Testing**.

Think of it like this:

- **SAST** (Static Analysis) = Reading a recipe and checking if the ingredients look safe (**checks code**)
- **DAST** (Dynamic Analysis) = Actually cooking the dish, tasting it, and checking if it makes you sick (**checks the running app**)

DAST tools **don't need access to your source code**. They work by:

1. Connecting to your **running application** (usually a web app)
2. **Crawling** through all the pages, forms, buttons, and links
3. **Sending malicious inputs** (like SQL injection payloads, XSS scripts, etc.)
4. **Analyzing the responses** to see if the app behaved in a vulnerable way

It's also called **"black-box testing"** because the tool has no idea what's inside the app — it only sees what any external user would see.

---

## 🎯 Why Do We Need DAST?

| Reason | Explanation |
|--------|-------------|
| **Finds runtime vulnerabilities** | Some bugs only appear when the app is actually running (e.g., misconfigured servers, broken auth) |
| **No source code needed** | Great for testing third-party apps, legacy apps, or apps where you don't have the code |
| **Tests real-world attack scenarios** | Simulates what an actual hacker would do |
| **Catches configuration issues** | Missing security headers, exposed admin panels, open redirects |
| **Complements SAST** | SAST finds code-level bugs; DAST finds deployment & runtime bugs |

---

## 📊 SAST vs DAST — Quick Comparison

| Feature | SAST (Static) | DAST (Dynamic) |
|---------|---------------|-----------------|
| **What it tests** | Source code | Running application |
| **When it runs** | During development/build | After deployment (staging/QA) |
| **Needs source code?** | ✅ Yes | ❌ No |
| **Finds code-level bugs?** | ✅ Yes | ❌ Not directly |
| **Finds runtime bugs?** | ❌ No | ✅ Yes |
| **False positives** | Higher | Lower |
| **Testing approach** | White-box | Black-box |
| **Speed** | Fast | Slower (needs running app) |
| **Language dependent?** | ✅ Yes | ❌ No (language agnostic) |

> 💡 **Best Practice:** Use **both SAST and DAST** together. SAST catches problems in the code, DAST catches problems in the running application. Together, they cover way more ground.

---

## 🔍 What Vulnerabilities Does DAST Find?

DAST tools are great at catching these common vulnerabilities:

### 1. **SQL Injection (SQLi)**
The tool sends payloads like `' OR 1=1 --` into input fields and checks if the database responds in unexpected ways.

### 2. **Cross-Site Scripting (XSS)**
It injects scripts like `<script>alert('XSS')</script>` and checks if the app executes them.

### 3. **Broken Authentication**
Tests for weak login mechanisms, session fixation, missing logout functionality.

### 4. **Security Misconfigurations**
Missing HTTPS, exposed debug endpoints, default credentials, directory listing enabled.

### 5. **Missing Security Headers**
Checks for headers like `X-Content-Type-Options`, `X-Frame-Options`, `Content-Security-Policy`, `Strict-Transport-Security`.

### 6. **Insecure Direct Object References (IDOR)**
Tests if you can access other users' data by changing IDs in the URL (e.g., `/user/123` → `/user/124`).

### 7. **Open Redirects**
Checks if the app can be tricked into redirecting users to malicious sites.

### 8. **Server Information Disclosure**
Detects if the server reveals its version (e.g., `Apache/2.4.49`) which helps attackers find known exploits.

## 🐝 OWASP ZAP (Zed Attack Proxy) — Deep Dive

### What is OWASP ZAP?

**OWASP ZAP** is a **free, open-source** DAST tool maintained by the **OWASP Foundation** (the same people who create the OWASP Top 10). It's one of the **most popular** security testing tools in the world.

ZAP works as a **"man-in-the-middle" proxy** — it sits between your browser and the web application, intercepting and analyzing all the traffic.

```
Your Browser  ──►  ZAP Proxy  ──►  Web Application
                      │
              Analyzes traffic
              Sends attack payloads
              Reports vulnerabilities
```

### Installing OWASP ZAP

#### Option 1: Download the Desktop App
Go to [https://www.zaproxy.org/download/](https://www.zaproxy.org/download/) and download for your OS.

#### Option 2: Use Docker (Best for CI/CD)

```bash
# Pull the ZAP Docker image
docker pull zaproxy/zap-stable

# Run ZAP in daemon mode (headless — no GUI)
docker run -u zap -p 8080:8080 zaproxy/zap-stable zap.sh -daemon -host 0.0.0.0 -port 8080
```

#### Option 3: Install via Package Manager

```bash
# On Ubuntu/Debian
sudo snap install zaproxy --classic

# On macOS
brew install --cask zap
```

---

### ZAP Scanning Modes

ZAP has several ways to scan your application:

#### 1. **Spider (Crawl)**
ZAP crawls your website to discover all pages, forms, and endpoints.

```bash
# Using ZAP CLI to spider a target
zap-cli spider http://your-app.example.com
```

#### 2. **Passive Scan**
ZAP silently analyzes the traffic passing through it without sending any extra requests. It just looks at what's already happening.

- Safe to run in production (non-intrusive)
- Finds: missing security headers, cookie issues, information disclosure

#### 3. **Active Scan**
ZAP actively sends **attack payloads** to your application to find vulnerabilities.

- ⚠️ **Never run against production!** It can break things.
- Finds: SQL injection, XSS, command injection, path traversal

```bash
# Using ZAP CLI to run an active scan
zap-cli active-scan http://your-app.example.com
```

#### 4. **Ajax Spider**
For modern single-page applications (React, Angular, Vue) that use JavaScript heavily, the regular spider won't work well. The Ajax Spider uses a real browser to crawl the app.

```bash
# Ajax spider for JavaScript-heavy apps
zap-cli ajax-spider http://your-spa-app.example.com
```

---

### Running ZAP in CI/CD (The DevOps Way) 🚀

This is where DAST gets really powerful — **automating it in your pipeline!**

#### Baseline Scan (Quick, Safe, Great for CI/CD)

The **baseline scan** is a quick passive scan that's safe enough to run on every PR:

```bash
# ZAP Baseline Scan using Docker
# This is a passive scan — safe, quick, finds low-hanging fruit
docker run --rm \
  -v $(pwd)/reports:/zap/wrk:rw \
  zaproxy/zap-stable \
  zap-baseline.py \
  -t http://your-staging-app.example.com \
  -r zap-report.html \
  -J zap-report.json
```

**What's happening here?**
- `--rm` → Remove the container after it finishes
- `-v $(pwd)/reports:/zap/wrk:rw` → Mount a local folder to save the report
- `zap-baseline.py` → Run the baseline (passive) scan script
- `-t` → Target URL to scan
- `-r zap-report.html` → Generate an HTML report
- `-J zap-report.json` → Also generate a JSON report (great for parsing in CI/CD)

#### Full Scan (Thorough, Use on Staging Only)

```bash
# ZAP Full Scan — includes active scanning
# ⚠️ Only run on staging/test environments!
docker run --rm \
  -v $(pwd)/reports:/zap/wrk:rw \
  zaproxy/zap-stable \
  zap-full-scan.py \
  -t http://your-staging-app.example.com \
  -r zap-full-report.html \
  -J zap-full-report.json
```

#### API Scan (For REST APIs)

If your app has APIs (and most do), you can scan them using an OpenAPI/Swagger specification:

```bash
# ZAP API Scan — scans your REST API
docker run --rm \
  -v $(pwd)/reports:/zap/wrk:rw \
  zaproxy/zap-stable \
  zap-api-scan.py \
  -t http://your-api.example.com/openapi.json \
  -f openapi \
  -r zap-api-report.html
```

**What's happening here?**
- `zap-api-scan.py` → Run the API scan script
- `-t` → Point to your OpenAPI/Swagger spec URL
- `-f openapi` → Tell ZAP the format is OpenAPI (can also be `soap` for SOAP APIs)

### ZAP in GitHub Actions

Here's a complete GitHub Actions workflow that runs a ZAP scan on every PR:

```yaml
# .github/workflows/dast-scan.yml
name: DAST Security Scan

on:
  pull_request:         # Run on every PR
    branches: [main]
  schedule:
    - cron: '0 2 * * 1' # Also run weekly on Monday at 2 AM

jobs:
  zap-scan:
    runs-on: ubuntu-latest
    name: ZAP Baseline Scan

    steps:
      # Step 1: Check out the code
      - name: Checkout code
        uses: actions/checkout@v4

      # Step 2: Build and start your application
      - name: Start application
        run: |
          docker compose up -d
          # Wait for the app to be ready
          sleep 30

      # Step 3: Run ZAP Baseline Scan
      - name: Run ZAP Baseline Scan
        uses: zaproxy/action-baseline@v0.12.0
        with:
          target: 'http://localhost:8080'    # Your app's URL
          rules_file_name: 'zap-rules.tsv'   # Optional: custom rules
          fail_action: true                   # Fail the build if vulnerabilities found
          allow_issue_writing: false          # Don't create GitHub issues

      # Step 4: Upload the report as an artifact
      - name: Upload ZAP Report
        if: always()                          # Upload even if scan fails
        uses: actions/upload-artifact@v4
        with:
          name: zap-report
          path: report_html.html
```

---

### ZAP Custom Rules File

You can customize which alerts ZAP reports by creating a rules file:

```tsv
# zap-rules.tsv
# Format: AlertID    Action    (IGNORE, WARN, FAIL)
10021	WARN	    # X-Content-Type-Options Header Missing
10038	IGNORE	  # Content Security Policy (CSP) Header Not Set (might be intentional)
40012	FAIL	    # Cross Site Scripting (Reflected) — this is serious!
40014	FAIL	    # Cross Site Scripting (Persistent) — this is very serious!
90001	WARN	    # Insecure JSF ViewState
```

**What's happening here?**
- `IGNORE` → Don't report this alert at all
- `WARN` → Report it but don't fail the build
- `FAIL` → Report it AND fail the build (this is a deal-breaker!)

### ZAP in Jenkins

```groovy
// Jenkinsfile — ZAP DAST Scan Stage
pipeline {
    agent any

    stages {
        stage('Deploy to Staging') {
            steps {
                // Deploy your app to a staging environment
                sh 'docker compose -f docker-compose.staging.yml up -d'
                sh 'sleep 30' // Wait for app to be ready
            }
        }

        stage('DAST Scan - ZAP') {
            steps {
                // Run ZAP baseline scan
                sh '''
                    docker run --rm \
                      --network host \
                      -v ${WORKSPACE}/reports:/zap/wrk:rw \
                      zaproxy/zap-stable \
                      zap-baseline.py \
                      -t http://localhost:8080 \
                      -r zap-report.html \
                      -J zap-report.json \
                      -x zap-report.xml
                '''
            }
        }

        stage('Publish Report') {
            steps {
                // Archive the report
                archiveArtifacts artifacts: 'reports/*', fingerprint: true

                // Publish HTML report (requires HTML Publisher plugin)
                publishHTML(target: [
                    reportName: 'ZAP Security Report',
                    reportDir: 'reports',
                    reportFiles: 'zap-report.html',
                    keepAll: true
                ])
            }
        }
    }

    post {
        always {
            // Clean up staging
            sh 'docker compose -f docker-compose.staging.yml down'
        }
    }
}
```

## 🔥 Burp Suite — Deep Dive

### What is Burp Suite?

**Burp Suite** is a **professional-grade** security testing tool made by **PortSwigger**. It's the **gold standard** for web application security testing and is used by almost every penetration tester.

It comes in three editions:

| Edition | Cost | Best For |
|---------|------|----------|
| **Community Edition** | Free | Learning, basic testing |
| **Professional** | ~$449/year | Professional pen testing, full scanner |
| **Enterprise** | Custom pricing | Automated CI/CD scanning at scale |

### Key Features of Burp Suite

1. **Proxy** — Intercept and modify HTTP/HTTPS traffic between browser and app
2. **Scanner** — Automated vulnerability scanner (Professional/Enterprise only)
3. **Repeater** — Manually modify and resend individual requests
4. **Intruder** — Automated customized attacks (brute force, fuzzing)
5. **Decoder** — Encode/decode data (Base64, URL encoding, etc.)
6. **Comparer** — Compare two pieces of data (requests, responses)
7. **Sequencer** — Analyze the randomness of session tokens

### How Burp Suite Works

```
                    ┌──────────────┐
Your Browser ──────►│  Burp Suite  │──────► Web Application
                    │   (Proxy)    │
                    │              │
                    │  • Intercept │
                    │  • Modify    │
                    │  • Replay    │
                    │  • Scan      │
                    │  • Analyze   │
                    └──────────────┘
```

### Setting Up Burp Suite

#### Step 1: Download and Install
Download from [https://portswigger.net/burp/releases](https://portswigger.net/burp/releases)

#### Step 2: Configure Your Browser to Use Burp as a Proxy
Burp listens on `127.0.0.1:8080` by default. Configure your browser's proxy settings to point there.

> 💡 **Tip:** Use the **FoxyProxy** browser extension to easily switch between proxy profiles.

#### Step 3: Install Burp's CA Certificate
To intercept HTTPS traffic, you need to install Burp's Certificate Authority certificate in your browser:

1. Navigate to `http://burpsuite` in your browser (while proxy is on)
2. Download the CA certificate
3. Import it into your browser's certificate store

---

### Using Burp Suite for DAST

#### Manual Testing Workflow

```
1. Configure browser proxy → Burp Suite
2. Browse the target application manually
3. Burp captures all requests in the "Site Map"
4. Right-click a request → "Send to Repeater" (modify and resend)
5. Right-click a request → "Send to Intruder" (automate attacks)
6. Right-click a target → "Actively Scan" (automated vulnerability scan)
7. Review findings in the "Issues" tab
```

#### Burp Suite Enterprise for CI/CD

Burp Suite Enterprise Edition can be integrated into CI/CD pipelines:

```bash
# Trigger a Burp Enterprise scan via REST API
curl -X POST "https://burp-enterprise.your-company.com/api/v1/scans" \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "scan_configuration_ids": ["default"],
    "site_id": "your-site-id",
    "urls": ["https://staging.your-app.example.com"]
  }'
```

```bash
# Check scan status
curl -X GET "https://burp-enterprise.your-company.com/api/v1/scans/SCAN_ID" \
  -H "Authorization: Bearer YOUR_API_KEY"
```

## 🏗️ DAST in the DevOps Pipeline — Where Does It Fit?

Here's where DAST fits in your CI/CD pipeline:

```
Code ──► Build ──► SAST ──► Deploy to    ──► DAST   ──► Deploy to
                             Staging          Scan        Production
                                               │
                                     ┌─────────┴──────────┐
                                     │                     │
                                  Pass ✅             Fail ❌
                                     │                     │
                               Continue to           Block deploy,
                               Production            fix vulnerabilities
```

### Important Placement Rules:

1. **DAST runs AFTER deployment to staging** — the app must be running
2. **DAST runs BEFORE production deployment** — catch issues before users are affected
3. **Use baseline (passive) scans on every PR** — fast feedback
4. **Use full (active) scans nightly or weekly** — thorough but slower
5. **Never run active scans against production** — they can cause damage

## 📝 Best Practices for DAST in DevOps

### 1. **Start with Baseline Scans**
Don't try to implement full active scanning on day one. Start with passive/baseline scans and gradually increase coverage.

### 2. **Authenticate Your Scans**
Most of your app is behind a login. If you don't authenticate your DAST scans, you'll only test the login page!

```bash
# ZAP with authentication (using context file)
docker run --rm \
  -v $(pwd):/zap/wrk:rw \
  zaproxy/zap-stable \
  zap-full-scan.py \
  -t http://staging.example.com \
  -r report.html \
  -n app.context  # Context file with auth config
```

### 3. **Handle False Positives**
DAST tools sometimes report things that aren't real vulnerabilities. Create a rules/exclusion file to manage these.

### 4. **Don't Scan Third-Party Sites**
Only scan applications you **own** or have **permission** to scan. Scanning someone else's site is illegal! ⚠️

### 5. **Set Scan Time Limits**
Full DAST scans can take hours. Set time limits in CI/CD to avoid blocking your pipeline forever.

```yaml
# In GitHub Actions — add a timeout
- name: Run ZAP Full Scan
  timeout-minutes: 30  # Kill if it takes more than 30 minutes
  run: |
    docker run --rm zaproxy/zap-stable zap-full-scan.py -t http://staging.example.com -r report.html
```

### 6. **Integrate with Issue Trackers**
Automatically create Jira/GitHub issues for critical vulnerabilities found by DAST.

### 7. **Scan APIs Too**
Don't just scan the web UI. Use API scanning capabilities to test your REST/GraphQL APIs.

## 🧪 Hands-On Example: Complete DAST Setup

Let's put it all together with a complete example:

### Step 1: Create a Vulnerable Test App (for practice)

```bash
# Run OWASP Juice Shop — a purposely vulnerable application for testing
docker run --rm -p 3000:3000 bkimminich/juice-shop
```

> 🎯 **OWASP Juice Shop** is a deliberately vulnerable web app designed for security training. It's perfect for practicing DAST!

### Step 2: Run a ZAP Baseline Scan

```bash
# Scan the Juice Shop with ZAP baseline
docker run --rm \
  --network host \
  -v $(pwd)/reports:/zap/wrk:rw \
  zaproxy/zap-stable \
  zap-baseline.py \
  -t http://localhost:3000 \
  -r juice-shop-baseline.html
```

### Step 3: Review the Report

Open `reports/juice-shop-baseline.html` in your browser. You'll see:

- **PASS** — checks that passed
- **WARN** — potential issues that need review
- **FAIL** — confirmed vulnerabilities

### Step 4: Run a Full Scan (More Thorough)

```bash
# Full active scan — this will find more vulnerabilities
docker run --rm \
  --network host \
  -v $(pwd)/reports:/zap/wrk:rw \
  zaproxy/zap-stable \
  zap-full-scan.py \
  -t http://localhost:3000 \
  -r juice-shop-full.html \
  -m 10  # Scan for max 10 minutes
```

| Concept | Key Point |
|---------|-----------|
| **DAST** | Tests **running applications** from the outside |

| **Never scan production actively** | Active scans can break things |
