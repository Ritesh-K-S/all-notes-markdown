# 16.5 — SCA — Software Composition Analysis (Snyk, Dependabot, Trivy)

> **One Line Answer:** SCA scans the **third-party libraries and dependencies** your application uses to find **known vulnerabilities** — because even if YOUR code is perfect, the packages you depend on might not be.

---

## 🤔 Let's Start With a Story

You're a DevOps engineer, and your team just built an awesome Node.js application. Your developers wrote clean, secure code. You ran SAST — no issues. You ran DAST — all good. You're feeling confident. 🎉

Then three months later, you wake up to a notification: **"CRITICAL: Log4Shell vulnerability (CVE-2021-44228) affects your application!"**

Wait, what?! Your team didn't even write any logging code! But here's the problem — your app uses a library called `log4j`, and THAT library has a critical vulnerability that allows **remote code execution**. An attacker can literally take over your server just by sending a specially crafted string. 😱

This is the **supply chain problem**. Modern applications are like buildings — your team might build the walls, but the **bricks** (third-party libraries) come from someone else. If those bricks are cracked, your whole building is at risk.

**SCA** is the tool that checks those bricks before you build with them.

---

## ✅ What is SCA?

**SCA** stands for **Software Composition Analysis**.

It works by:

1. **Scanning your dependency files** (like `package.json`, `requirements.txt`, `pom.xml`, `go.mod`, etc.)
2. **Building a list** of all third-party libraries your app uses (including indirect/transitive dependencies)
3. **Checking each library** against databases of **known vulnerabilities** (like the **NVD** — National Vulnerability Database, and **GitHub Advisory Database**)
4. **Reporting** which libraries have known issues, how severe they are, and how to fix them

### Think of it Like This 🍕

Imagine you're ordering a pizza:
- **SAST** = Checking if the pizza chef followed the recipe correctly
- **DAST** = Tasting the pizza to see if something's wrong
- **SCA** = Checking if any of the **ingredients** (flour, cheese, sauce) are contaminated or recalled

Your chef might be amazing, but if the flour is contaminated, the pizza is still dangerous!

## 🎯 Why is SCA So Important?

### The Shocking Reality of Modern Software

| Fact | Detail |
|------|--------|
| **Open source everywhere** | 70-90% of modern applications are made up of open-source code |
| **Transitive dependencies** | Your app might use 10 libraries, but those 10 libraries use 200 more |
| **Known vulnerabilities** | Thousands of new CVEs (Common Vulnerabilities and Exposures) are published every year |
| **Supply chain attacks** | Attackers now target popular libraries to compromise thousands of apps at once |
| **License compliance** | Some open-source licenses have restrictions that could legally impact your business |

### Real-World SCA Disasters

| Incident | What Happened |
|----------|---------------|
| **Log4Shell (2021)** | A vulnerability in the Java `log4j` library affected millions of applications worldwide |
| **event-stream (2018)** | An npm package was hijacked to steal cryptocurrency |
| **colors.js (2022)** | A popular npm package was intentionally sabotaged by its own maintainer |
| **ua-parser-js (2021)** | A popular npm package was hijacked to install crypto miners |

## 📊 How SCA Works — Step by Step

```
┌─────────────────────────────────────────────────────────────────┐
│                     SCA Scanning Process                        │
│                                                                 │
│  1. Read Dependency Files                                       │
│     ├── package.json / package-lock.json (Node.js)              │
│     ├── requirements.txt / Pipfile.lock (Python)                │
│     ├── pom.xml / build.gradle (Java)                           │
│     ├── go.mod / go.sum (Go)                                    │
│     ├── Gemfile.lock (Ruby)                                     │
│     └── Cargo.lock (Rust)                                       │
│                                                                 │
│  2. Build Dependency Tree                                       │
│     your-app                                                    │
│     ├── express@4.18.2                                          │
│     │   ├── body-parser@1.20.1                                  │
│     │   │   └── qs@6.11.0  ← (transitive dependency)           │
│     │   └── cookie@0.5.0                                        │
│     └── lodash@4.17.20  ← ⚠️ VULNERABLE!                       │
│                                                                 │
│  3. Check Against Vulnerability Databases                       │
│     ├── NVD (National Vulnerability Database)                   │
│     ├── GitHub Advisory Database                                │
│     ├── OSV (Open Source Vulnerabilities)                       │
│     └── Vendor-specific databases (Snyk, etc.)                  │
│                                                                 │
│  4. Report Findings                                             │
│     ├── CVE ID (e.g., CVE-2021-23337)                          │
│     ├── Severity (Critical / High / Medium / Low)               │
│     ├── Affected versions                                       │
│     ├── Fixed version (upgrade to this!)                        │
│     └── Exploit availability                                    │
└─────────────────────────────────────────────────────────────────┘
```

## 🔍 What Does SCA Find?

### 1. **Known Vulnerabilities (CVEs)**
Libraries with publicly known security bugs.

Example:
```
CRITICAL: lodash@4.17.20
CVE-2021-23337 — Command Injection
Fix: Upgrade to lodash@4.17.21
```

### 2. **Outdated Dependencies**
Libraries that are far behind the latest version (even if no CVE exists yet, older versions are riskier).

### 3. **License Issues**
Some open-source licenses have legal implications:

| License | Type | Implication |
|---------|------|-------------|
| **MIT** | Permissive | ✅ Use freely, even in commercial software |
| **Apache 2.0** | Permissive | ✅ Use freely, must include notice |
| **GPL v3** | Copyleft | ⚠️ If you use it, YOUR code must also be open-source |
| **AGPL** | Strong Copyleft | ⚠️ Even SaaS usage requires open-sourcing |
| **SSPL** | Restrictive | ⚠️ Very restrictive for cloud services |

### 4. **Malicious Packages**
Some packages are deliberately malicious — typosquatting (e.g., `loadsh` instead of `lodash`), hijacked packages, etc.

### 5. **Transitive Vulnerabilities**
Your direct dependencies might be fine, but THEIR dependencies might be vulnerable.

```
your-app
└── safe-library@2.0.0  ← looks fine!
    └── vulnerable-helper@1.0.0  ← ⚠️ THIS has a vulnerability!
```

## 🐙 GitHub Dependabot — Deep Dive

### What is Dependabot?

**Dependabot** is GitHub's **built-in** SCA tool. It's completely **free** for all GitHub repositories. It does two things:

1. **Dependabot Alerts** — Tells you when your dependencies have known vulnerabilities
2. **Dependabot Security Updates** — Automatically creates PRs to fix vulnerable dependencies
3. **Dependabot Version Updates** — Automatically creates PRs to keep dependencies up to date

### Setting Up Dependabot

Create a file at `.github/dependabot.yml` in your repo:

```yaml
# .github/dependabot.yml
version: 2

updates:
  # Keep npm dependencies up to date
  - package-ecosystem: "npm"
    directory: "/"               # Where is package.json?
    schedule:
      interval: "weekly"         # Check weekly for updates
      day: "monday"              # On Mondays
      time: "09:00"              # At 9 AM
      timezone: "America/New_York"
    open-pull-requests-limit: 10 # Max 10 PRs open at once
    reviewers:
      - "your-team"              # Auto-assign reviewers
    labels:
      - "dependencies"           # Add label to PRs
      - "automated"

  # Keep Python dependencies up to date
  - package-ecosystem: "pip"
    directory: "/"
    schedule:
      interval: "weekly"

  # Keep Docker base images up to date
  - package-ecosystem: "docker"
    directory: "/"
    schedule:
      interval: "weekly"

  # Keep GitHub Actions up to date
  - package-ecosystem: "github-actions"
    directory: "/"
    schedule:
      interval: "weekly"
```

**What's happening here?**
- `package-ecosystem` → Which package manager to scan (npm, pip, docker, maven, etc.)
- `directory` → Where the dependency file lives (e.g., `package.json`)
- `schedule.interval` → How often to check (`daily`, `weekly`, `monthly`)
- `open-pull-requests-limit` → Don't overwhelm us with too many PRs
- `reviewers` → Automatically assign reviewers to dependency update PRs
- `labels` → Label the PRs for easy filtering

### What Happens After Setup?

1. Dependabot scans your dependencies on the configured schedule
2. If it finds a vulnerable dependency → it creates a **security update PR**
3. If a newer version exists → it creates a **version update PR**
4. You review and merge the PR → vulnerability fixed! ✅

### Example Dependabot PR

Dependabot will create PRs that look like this:

```
Title: Bump lodash from 4.17.20 to 4.17.21

This PR bumps lodash from 4.17.20 to 4.17.21.
 
Release notes: ...
Changelog: ...
Commits: ...

Vulnerabilities fixed:
- CVE-2021-23337 (Command Injection) — CRITICAL
```

### Grouping Dependabot Updates

To avoid getting 50 separate PRs, you can group updates:

```yaml
# .github/dependabot.yml
version: 2

updates:
  - package-ecosystem: "npm"
    directory: "/"
    schedule:
      interval: "weekly"
    groups:
      # Group all minor and patch updates into one PR
      production-dependencies:
        dependency-type: "production"
        update-types:
          - "minor"
          - "patch"
      dev-dependencies:
        dependency-type: "development"
        update-types:
          - "minor"
          - "patch"
```

## 🐍 Snyk — Deep Dive

### What is Snyk?

**Snyk** (pronounced "snick") is one of the most popular **developer-first security platforms**. It started as an SCA tool but now does much more:

- **Snyk Open Source** — SCA (what we're covering here)
- **Snyk Code** — SAST
- **Snyk Container** — Container image scanning
- **Snyk IaC** — Infrastructure as Code scanning

### Installing Snyk CLI

```bash
# Install via npm
npm install -g snyk

# Or via brew (macOS/Linux)
brew install snyk

# Or via Docker
docker pull snyk/snyk

# Authenticate (links to your Snyk account)
snyk auth
```

### Using Snyk CLI

#### Test Your Project for Vulnerabilities

```bash
# Navigate to your project
cd /your-project

# Scan for vulnerabilities
snyk test
```

**Example output:**

#### Auto-Fix Vulnerabilities

```bash
# Let Snyk try to fix vulnerabilities automatically
snyk fix
```

This command will:
- Update the dependency to a safe version
- Create a `.snyk` policy file if needed
- Handle transitive dependency fixes

#### Monitor Your Project (Continuous Scanning)

```bash
# Take a snapshot and monitor for new vulnerabilities
snyk monitor
```

This sends your dependency info to Snyk's cloud and alerts you via email/Slack when new vulnerabilities are discovered in your dependencies — even after you've deployed!

### Snyk in CI/CD

#### GitHub Actions with Snyk

```yaml
# .github/workflows/snyk-sca.yml
name: Snyk SCA Scan

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  snyk-scan:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Install dependencies
        run: npm ci

      # Run Snyk to check for vulnerabilities
      - name: Run Snyk SCA scan
        uses: snyk/actions/node@master
        env:
          SNYK_TOKEN: ${{ secrets.SNYK_TOKEN }}
        with:
          args: --severity-threshold=high  # Only fail on high/critical
          command: test

      # Optional: Monitor the project (snapshot for continuous monitoring)
      - name: Monitor with Snyk
        if: github.ref == 'refs/heads/main'  # Only on main branch
        uses: snyk/actions/node@master
        env:
          SNYK_TOKEN: ${{ secrets.SNYK_TOKEN }}
        with:
          command: monitor
```

**What's happening here?**
- `snyk/actions/node@master` → Official Snyk GitHub Action for Node.js projects
- `SNYK_TOKEN` → Your Snyk API token stored as a GitHub secret
- `--severity-threshold=high` → Only fail the build for high and critical vulnerabilities
- `command: test` → Run the vulnerability test
- `command: monitor` → Send a snapshot to Snyk for continuous monitoring

#### Snyk for Different Languages

```yaml
# Python project
- uses: snyk/actions/python@master
  env:
    SNYK_TOKEN: ${{ secrets.SNYK_TOKEN }}

# Java (Maven)
- uses: snyk/actions/maven@master
  env:
    SNYK_TOKEN: ${{ secrets.SNYK_TOKEN }}

# Go
- uses: snyk/actions/golang@master
  env:
    SNYK_TOKEN: ${{ secrets.SNYK_TOKEN }}

# Docker image scanning
- uses: snyk/actions/docker@master
  env:
    SNYK_TOKEN: ${{ secrets.SNYK_TOKEN }}
  with:
    image: your-image:latest
```

#### Jenkins with Snyk

```groovy
// Jenkinsfile — Snyk SCA Scan
pipeline {
    agent any

    environment {
        SNYK_TOKEN = credentials('snyk-api-token')  // From Jenkins credentials store
    }

    stages {
        stage('Install Dependencies') {
            steps {
                sh 'npm ci'
            }
        }

        stage('SCA Scan - Snyk') {
            steps {
                sh '''
                    # Install Snyk CLI
                    npm install -g snyk

                    # Run the scan — fail on high severity
                    snyk test --severity-threshold=high

                    # Generate JSON report for further processing
                    snyk test --json > snyk-report.json || true
                '''
            }
        }

        stage('Archive Report') {
            steps {
                archiveArtifacts artifacts: 'snyk-report.json', fingerprint: true
            }
        }
    }
}
```

### Snyk Policy File (`.snyk`)

You can create a `.snyk` file to ignore specific vulnerabilities (with justification):

```yaml
# .snyk
version: v1.5.0
ignore:
  SNYK-JS-LODASH-590103:
    - '*':
        reason: 'No fix available yet, risk accepted by security team'
        expires: 2025-12-31T00:00:00.000Z  # Re-evaluate by this date
  SNYK-JS-MINIMIST-559764:
    - '*':
        reason: 'Dev dependency only, not used in production code'
        expires: 2025-06-01T00:00:00.000Z
```

**What's happening here?**
- `ignore` → List of vulnerabilities to temporarily ignore
- `reason` → You MUST document why you're ignoring it (audit trail!)
- `expires` → The ignore expires on this date — forces you to re-evaluate

## 🔱 Trivy for SCA — Deep Dive

### What is Trivy?

**Trivy** (by Aqua Security) is an **all-in-one open-source security scanner**. While it's famous for container scanning (covered in 16.6), it also does excellent SCA:

```bash
# Install Trivy
# macOS
brew install trivy

# Ubuntu/Debian
sudo apt-get install trivy

# Docker
docker pull aquasec/trivy
```

### Using Trivy for Dependency Scanning

```bash
# Scan a project directory for vulnerable dependencies
trivy fs /path/to/your/project

# Scan only for high and critical vulnerabilities
trivy fs --severity HIGH,CRITICAL /path/to/your/project

# Output as JSON for CI/CD processing
trivy fs --format json --output trivy-report.json /path/to/your/project

# Scan a specific lock file
trivy fs --scanners vuln package-lock.json
```

**Example output:**
```
package-lock.json (npm)

├──────────────┼────────────────┼──────────┼────────┼───────────────────┼───────────────┼──────────────────────────────────────┤
│ lodash       │ CVE-2021-23337 │ CRITICAL │ fixed  │ 4.17.20           │ 4.17.21       │ Command Injection                    │
│ axios        │ CVE-2023-45857 │ HIGH     │ fixed  │ 0.21.1            │ 1.6.0         │ SSRF via proxy                       │
```

### Trivy SCA in GitHub Actions

```yaml
# .github/workflows/trivy-sca.yml
name: Trivy SCA Scan

on:
  push:
    branches: [main]
  pull_request:

jobs:
  trivy-scan:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Run Trivy vulnerability scanner (filesystem mode)
        uses: aquasecurity/trivy-action@master
        with:
          scan-type: 'fs'                    # Filesystem scan (SCA)
          scan-ref: '.'                      # Scan current directory
          format: 'table'                    # Human-readable output
          severity: 'HIGH,CRITICAL'          # Only high and critical
          exit-code: '1'                     # Fail the build if issues found

      - name: Run Trivy and upload SARIF
        uses: aquasecurity/trivy-action@master
        with:
          scan-type: 'fs'
          format: 'sarif'
          output: 'trivy-results.sarif'
          severity: 'HIGH,CRITICAL'

      # Upload results to GitHub Security tab
      - name: Upload Trivy scan results to GitHub Security
        uses: github/codeql-action/upload-sarif@v3
        if: always()
        with:
          sarif_file: 'trivy-results.sarif'
```

**What's happening here?**
- `scan-type: 'fs'` → Filesystem scan (for dependencies in your repo)
- `format: 'sarif'` → Security Analysis Results Interchange Format — GitHub understands this!
- `upload-sarif` → Sends results to the **Security** tab of your GitHub repo

## 📦 Built-in Package Manager Audit Commands

Most package managers have built-in audit capabilities. They're not as powerful as dedicated SCA tools, but they're quick and easy:

### npm audit (Node.js)

```bash
# Check for vulnerabilities
npm audit

# Auto-fix what can be fixed
npm audit fix

# Force fix (may include breaking changes)
npm audit fix --force

# Output as JSON
npm audit --json > npm-audit.json
```

### pip-audit (Python)

```bash
# Install pip-audit
pip install pip-audit

# Scan your Python environment
pip-audit

# Scan a requirements file
pip-audit -r requirements.txt

# Output as JSON
pip-audit --format json --output pip-audit.json
```

### yarn audit (Node.js with Yarn)

```bash
# Check for vulnerabilities
yarn audit

# Output as JSON
yarn audit --json
```

### cargo audit (Rust)

```bash
# Install cargo-audit
cargo install cargo-audit

# Scan your Rust project
cargo audit
```

### bundle-audit (Ruby)

```bash

gem install bundler-audit

```

## 🔄 Renovate — The Dependency Update Bot (Bonus)

**Renovate** is like Dependabot on steroids. It's an open-source tool that automatically creates PRs to update your dependencies:

```json
// renovate.json (place in repo root)
{
  "$schema": "https://docs.renovatebot.com/renovate-schema.json",
  "extends": [
    "config:recommended"
  ],
  "packageRules": [
    {
      "matchUpdateTypes": ["minor", "patch"],
      "automerge": true,
      "automergeType": "pr"
    },
    {
      "matchUpdateTypes": ["major"],
      "automerge": false,
      "labels": ["major-update", "needs-review"]
    }
  ],
  "schedule": ["every weekend"],
  "vulnerabilityAlerts": {
    "enabled": true,
    "labels": ["security"]
  }
}
```

**What's happening here?**
- Minor and patch updates → auto-merge (they're usually safe)
- Major updates → require manual review (might have breaking changes)
- Vulnerability alerts → always enabled with a "security" label
- Schedule → only run on weekends to reduce noise

## 🏗️ SCA in the DevOps Pipeline — Where Does It Fit?

```
Developer       Build          Test           Deploy
  Push    ──►  Install   ──►  SAST    ──►   Staging
  Code         Dependencies   SCA ←──────   DAST
                  │             │
                  │        ┌────┴──────┐
                  │        │           │
                  │     Pass ✅    Fail ❌
                  │        │           │
                  │   Continue    Block build,
                  │                fix deps
                  │
           ┌──────┴──────────────────────┐
           │  SCA checks:                │
           │  • Known CVEs               │
           │  • License compliance       │
           │  • Outdated packages        │
           │  • Malicious packages       │
           └─────────────────────────────┘
```

### Best Placement:
1. **During `npm install` / `pip install`** — right after dependencies are installed
2. **In PR checks** — block merging if critical vulnerabilities exist
3. **Continuous monitoring** — scan regularly even after deployment (new CVEs are published daily!)

## 📝 Best Practices for SCA

### 1. **Don't Ignore Vulnerabilities Without Justification**
Every ignored vulnerability should have a documented reason and an expiration date.

### 2. **Set Severity Thresholds**
Don't block your pipeline for every low-severity issue. Start by blocking on **Critical** and **High** only.

```bash
# Snyk — only fail on high severity or above
snyk test --severity-threshold=high

# Trivy — only show critical and high
trivy fs --severity HIGH,CRITICAL .
```

### 3. **Keep Dependencies Updated**
The best defense is keeping your dependencies current. Use Dependabot or Renovate to automate this.

### 4. **Lock Your Dependencies**
Always commit your lock files (`package-lock.json`, `Pipfile.lock`, `go.sum`). This ensures everyone uses the exact same versions.

### 5. **Minimize Dependencies**
The fewer dependencies you have, the smaller your attack surface. Ask yourself: "Do I really need this library, or can I write the 10 lines of code myself?"

### 6. **Monitor After Deployment**
New vulnerabilities are discovered daily. A dependency that's safe today might have a CVE tomorrow. Use continuous monitoring (Snyk Monitor, Dependabot Alerts).

### 7. **Check Licenses**
Make sure your dependencies' licenses are compatible with your project. A GPL dependency in a commercial product could be a legal problem.
