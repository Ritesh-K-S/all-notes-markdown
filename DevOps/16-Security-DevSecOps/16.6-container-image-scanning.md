## 🤔 Let's Start With a Story

You're a DevOps engineer who just containerized your company's main application. You wrote a beautiful Dockerfile, built the image, pushed it to your registry, and deployed it to Kubernetes. Everything is running smoothly. ✅

Then one morning, your security team runs a scan on your container images and sends you a terrifying report:

```
Image: myapp:latest
Total vulnerabilities: 247
  CRITICAL: 12
  HIGH: 43
  MEDIUM: 89
  LOW: 103
```

**247 vulnerabilities?!** 😱 But YOUR code is fine! Where did all these come from?

Here's the thing — when you write `FROM ubuntu:20.04` in your Dockerfile, you're pulling in an **entire operating system** with hundreds of packages. Each of those packages could have known vulnerabilities. On top of that, when you `RUN pip install flask`, you're pulling in Python libraries that might have their own issues.

Your container image is like a **shipping container** — you packed your app inside, but you also packed a bunch of other stuff (OS, libraries, tools) that could be compromised. **Container image scanning** is like an X-ray machine at the port — it inspects everything inside before you ship it out.

---

## ✅ What is Container Image Scanning?

Container image scanning is the process of **analyzing a Docker/OCI image** to find:

1. **OS Package Vulnerabilities** — Bugs in Ubuntu, Alpine, Debian packages (`apt`, `apk`)
2. **Application Dependency Vulnerabilities** — Bugs in Python (`pip`), Node.js (`npm`), Java (`maven`), Go (`mod`) libraries
3. **Misconfigurations** — Running as root, exposed secrets, insecure permissions
4. **Malware** — Known malicious files or patterns
5. **License Issues** — Software with restrictive licenses

### How Does It Work?

```
┌────────────────────────────────────────────────┐
│           Container Image Scanning             │
│                                                │
│  1. Pull/Load the image                        │
│     docker pull myapp:latest                   │
│                                                │
│  2. Extract all layers                         │
│     Layer 1: Ubuntu base OS                    │
│     Layer 2: Python + pip packages             │
│     Layer 3: Your application code             │
│                                                │
│  3. Identify all packages                      │
│     ├── OS packages (dpkg, apk, rpm)           │
│     ├── Python packages (pip)                  │
│     ├── Node.js packages (npm)                 │
│     ├── Java packages (jar)                    │
│     ├── Go binaries                            │
│     └── Ruby gems, Rust crates, etc.           │
│                                                │
│  4. Match against vulnerability databases      │
│     ├── NVD (National Vulnerability Database)  │
│     ├── Alpine SecDB                           │
│     ├── Debian Security Tracker                │
│     ├── Ubuntu CVE Tracker                     │
│     ├── Red Hat Security Data                  │
│     ├── GitHub Advisory Database               │
│     └── Vendor-specific databases              │
│                                                │
│  5. Generate report                            │
│     ├── CVE ID                                 │
│     ├── Severity (Critical/High/Medium/Low)    │
│     ├── Package name + version                 │
│     ├── Fixed version (if available)           │
│     └── Description                            │
└────────────────────────────────────────────────┘
```

## 🎯 Why Scan Container Images?

| Reason | Explanation |
|--------|-------------|
| **Base images have vulnerabilities** | Even official images like `ubuntu:22.04` have known CVEs |
| **Dependencies are bundled** | Everything is baked in — OS packages, libraries, tools |
| **Images are immutable** | Once built, the image doesn't get OS patches automatically |
| **Images live long** | A 6-month-old image has 6 months of unpatched vulnerabilities |
| **Compliance requirements** | SOC2, PCI-DSS, HIPAA all require vulnerability management |
| **Supply chain security** | Verify that base images and dependencies haven't been tampered with |
| **Shift-left security** | Catch issues before deploying to production |

### The Vulnerability Lifecycle of a Container Image

```
Day 0: Build image (0 known CVEs)
Day 30: 5 new CVEs discovered in base image packages
Day 60: 15 new CVEs — some are CRITICAL
Day 90: 30+ CVEs — image is now a security risk 😱

This is why you need to:
1. Scan images BEFORE deployment
2. Re-scan regularly (even deployed images)
3. Rebuild images frequently with updated base images
```

## 📊 SCA vs Container Scanning — What's the Difference?

You might be thinking: "Wait, didn't we just cover SCA in 16.5? How is this different?"

| Aspect | SCA (16.5) | Container Image Scanning (16.6) |
|--------|-----------|--------------------------------|
| **What it scans** | Dependency files (`package.json`, `requirements.txt`) | The **entire container image** |
| **Scope** | Application dependencies only | OS packages + app dependencies + configs |
| **When it runs** | During build, after `npm install` | After `docker build`, before `docker push` |
| **Input** | Lock files in source code | Docker/OCI image (tar file or registry reference) |
| **Finds** | Library vulnerabilities | Library vulnerabilities + OS vulnerabilities + misconfigs |

> 💡 **Think of it this way:** SCA checks your shopping list (dependency files). Container scanning opens the box and checks everything inside (the built image).

## 🔱 Trivy — Deep Dive (The Swiss Army Knife)

### What is Trivy?

**Trivy** (by **Aqua Security**) is the **most popular open-source container scanner**. It's fast, easy to use, and scans way more than just containers:

- ✅ Container images
- ✅ Filesystems (SCA)
- ✅ Git repositories
- ✅ Kubernetes clusters
- ✅ Infrastructure as Code (Terraform, CloudFormation)
- ✅ SBOM generation

### Installing Trivy

```bash
# macOS
brew install trivy

# Ubuntu/Debian
sudo apt-get install wget apt-transport-https gnupg lsb-release
wget -qO - https://aquasecurity.github.io/trivy-repo/deb/public.key | gpg --dearmor | sudo tee /usr/share/keyrings/trivy.gpg > /dev/null
echo "deb [signed-by=/usr/share/keyrings/trivy.gpg] https://aquasecurity.github.io/trivy-repo/deb $(lsb_release -sc) main" | sudo tee -a /etc/apt/sources.list.d/trivy.list
sudo apt-get update
sudo apt-get install trivy

# RHEL/CentOS
sudo yum install trivy

# Docker (no installation needed!)
docker run aquasec/trivy
```

```bash
# Scan an image from Docker Hub
trivy image nginx:latest

trivy image registry.example.com/myapp:latest
```

#### Filter by Severity

```bash
# Only show HIGH and CRITICAL vulnerabilities
trivy image --severity HIGH,CRITICAL nginx:latest

# Only show CRITICAL vulnerabilities
trivy image --severity CRITICAL myapp:latest
```

#### Different Output Formats

```bash
# Default table format (human-readable)
trivy image nginx:latest

# JSON format (great for CI/CD parsing)
trivy image --format json --output trivy-report.json nginx:latest

# SARIF format (for GitHub Security tab)
trivy image --format sarif --output trivy-results.sarif nginx:latest

# HTML template (for reports)
trivy image --format template --template "@contrib/html.tpl" --output report.html nginx:latest

# CycloneDX SBOM format
trivy image --format cyclonedx --output sbom.json nginx:latest
```

#### Exit Codes for CI/CD

```bash
# Exit with code 1 if any vulnerability is found (strict mode)
trivy image --exit-code 1 nginx:latest

# Exit with code 1 only for CRITICAL and HIGH
trivy image --exit-code 1 --severity HIGH,CRITICAL nginx:latest

# Exit with code 0 for Medium/Low, code 1 for High/Critical
trivy image --exit-code 0 --severity MEDIUM,LOW nginx:latest
trivy image --exit-code 1 --severity HIGH,CRITICAL nginx:latest
```

#### Ignore Specific Vulnerabilities

Create a `.trivyignore` file in your project root:

```
# .trivyignore
# Format: one CVE per line

# This CVE doesn't affect us because we don't use the affected function
CVE-2023-45853

# Accepted risk — no fix available yet, monitoring for updates
CVE-2024-0727

# False positive — confirmed by security team on 2024-03-15
CVE-2023-12345
```

```bash
# Trivy will automatically read .trivyignore from the current directory
trivy image --ignorefile .trivyignore myapp:latest
```

### Trivy in CI/CD Pipelines

#### GitHub Actions

```yaml
# .github/workflows/container-scan.yml
name: Container Image Scan

on:
  push:
    branches: [main]
  pull_request:

jobs:
  build-and-scan:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Build Docker image
        run: docker build -t myapp:${{ github.sha }} .

      # Scan the image with Trivy
      - name: Run Trivy vulnerability scanner
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: 'myapp:${{ github.sha }}'
          format: 'table'
          exit-code: '1'                     # Fail if vulnerabilities found
          severity: 'CRITICAL,HIGH'          # Only fail on critical and high
          ignore-unfixed: true               # Ignore vulns with no fix available

      # Also generate SARIF for GitHub Security tab
      - name: Trivy scan (SARIF)
        uses: aquasecurity/trivy-action@master
        if: always()
        with:
          image-ref: 'myapp:${{ github.sha }}'
          format: 'sarif'
          output: 'trivy-results.sarif'
          severity: 'CRITICAL,HIGH'

      - name: Upload Trivy scan results
        uses: github/codeql-action/upload-sarif@v3
        if: always()
        with:
          sarif_file: 'trivy-results.sarif'

      # Push only if scan passed
      - name: Push image to registry
        if: success()
        run: |
          docker tag myapp:${{ github.sha }} registry.example.com/myapp:${{ github.sha }}
          docker push registry.example.com/myapp:${{ github.sha }}
```

**What's happening here?**
1. Build the Docker image
2. Scan it with Trivy — fail if CRITICAL or HIGH vulns are found
3. Generate a SARIF report and upload to GitHub Security tab
4. Only push the image if the scan passed ✅

#### Jenkins Pipeline

```groovy
// Jenkinsfile — Container Image Scanning
pipeline {
    agent any

    environment {
        IMAGE_NAME = "myapp"
        IMAGE_TAG = "${env.BUILD_NUMBER}"
        REGISTRY = "registry.example.com"
    }

    stages {
        stage('Build Image') {
            steps {
                sh "docker build -t ${IMAGE_NAME}:${IMAGE_TAG} ."
            }
        }

        stage('Scan Image - Trivy') {
            steps {
                // Run Trivy scan
                sh """
                    docker run --rm \
                      -v /var/run/docker.sock:/var/run/docker.sock \
                      -v ${WORKSPACE}/reports:/reports \
                      aquasec/trivy image \
                        --exit-code 1 \
                        --severity HIGH,CRITICAL \
                        --ignore-unfixed \
                        --format json \
                        --output /reports/trivy-report.json \
                        ${IMAGE_NAME}:${IMAGE_TAG}
                """
            }
            post {
                always {
                    // Archive the report
                    archiveArtifacts artifacts: 'reports/trivy-report.json', fingerprint: true
                }
                failure {
                    echo "🚨 Trivy found HIGH/CRITICAL vulnerabilities! Image will NOT be pushed."
                }
            }
        }

        stage('Push Image') {
            steps {
                sh """
                    docker tag ${IMAGE_NAME}:${IMAGE_TAG} ${REGISTRY}/${IMAGE_NAME}:${IMAGE_TAG}
                    docker push ${REGISTRY}/${IMAGE_NAME}:${IMAGE_TAG}
                """
            }
        }
    }
}
```

#### GitLab CI/CD

```yaml
# .gitlab-ci.yml
stages:
  - build
  - scan
  - push
  - deploy

build-image:
  stage: build
  image: docker:latest
  services:
    - docker:dind
  script:
    - docker build -t $CI_REGISTRY_IMAGE:$CI_COMMIT_SHA .
    - docker save $CI_REGISTRY_IMAGE:$CI_COMMIT_SHA > image.tar
  artifacts:
    paths:
      - image.tar

scan-image:
  stage: scan
  image:
    name: aquasec/trivy
    entrypoint: [""]
  script:
    # Load the image from the artifact
    - trivy image
        --exit-code 1
        --severity HIGH,CRITICAL
        --ignore-unfixed
        --format json
        --output trivy-report.json
        --input image.tar
  artifacts:
    when: always
    paths:
      - trivy-report.json
    reports:
      container_scanning: trivy-report.json
  allow_failure: false  # Block pipeline on vulnerabilities

push-image:
  stage: push
  image: docker:latest
  services:
    - docker:dind
  script:
    - docker load < image.tar
    - docker push $CI_REGISTRY_IMAGE:$CI_COMMIT_SHA
  needs:
    - build-image
    - scan-image  # Only push if scan passed
```

## 🐙 Grype — Deep Dive

### What is Grype?

**Grype** (by **Anchore**) is a **fast, lightweight** vulnerability scanner for container images and filesystems. It pairs with **Syft** (a tool that generates SBOMs — Software Bill of Materials).

### Installing Grype

```bash
# macOS
brew install grype

# Linux (install script)
curl -sSfL https://raw.githubusercontent.com/anchore/grype/main/install.sh | sh -s -- -b /usr/local/bin

# Docker
docker run anchore/grype
```

### Using Grype

#### Basic Scan

```bash
# Scan a container image
grype nginx:latest

# Scan a locally built image
grype myapp:v1.2.3

# Scan from a Docker archive (tar file)
grype docker-archive:path/to/image.tar
```

**Example output:**
```
 ✔ Vulnerability DB        [no update available]
 ✔ Loaded image            
 ✔ Parsed image            
 ✔ Cataloged packages      [142 packages]
 ✔ Scanned for vulnerabilities     [47 vulnerabilities]

NAME          INSTALLED         FIXED-IN          TYPE   VULNERABILITY   SEVERITY
curl          7.88.1-10+deb12u5 7.88.1-10+deb12u6 deb    CVE-2024-2398   High
libssl3       3.0.11-1~deb12u2  3.0.13-1~deb12u1  deb    CVE-2024-0727   Critical
zlib1g        1:1.2.13.dfsg-1                     deb    CVE-2023-45853  Critical
```

#### Filter and Control Output

```bash
# Only show high and critical
grype nginx:latest --only-fixed --fail-on high

# Output as JSON
grype nginx:latest -o json > grype-report.json

# Output as table (default)
grype nginx:latest -o table

# Output as CycloneDX SBOM
grype nginx:latest -o cyclonedx-json > grype-sbom.json

# Fail with exit code 1 on specific severity
grype nginx:latest --fail-on critical
```

### Grype + Syft Combo

**Syft** generates an SBOM (a list of everything in your image). **Grype** scans that SBOM for vulnerabilities. Together, they're a powerful combo:

```bash
# Install Syft
brew install syft

# Step 1: Generate SBOM with Syft
syft nginx:latest -o spdx-json > nginx-sbom.json

# Step 2: Scan the SBOM with Grype
grype sbom:nginx-sbom.json
```

**Why is this useful?**
- Generate the SBOM once, scan it many times
- Store the SBOM as an artifact for audit/compliance
- Share the SBOM with your security team

### Grype in GitHub Actions

```yaml
# .github/workflows/grype-scan.yml
name: Container Scan with Grype

on:
  push:
    branches: [main]

jobs:
  scan:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Build image
        run: docker build -t myapp:latest .

      - name: Scan image with Grype
        uses: anchore/scan-action@v4
        id: scan
        with:
          image: "myapp:latest"
          fail-build: true
          severity-cutoff: high   # Fail on high or critical
          output-format: sarif

      - name: Upload SARIF report
        uses: github/codeql-action/upload-sarif@v3
        if: always()
        with:
          sarif_file: ${{ steps.scan.outputs.sarif }}
```

## ⚓ Anchore — Deep Dive

### What is Anchore?

**Anchore** is a **policy-based** container scanning platform. What makes it special is that you can define **custom policies** to enforce your own security rules — not just checking for CVEs.

It comes in two flavors:
- **Anchore Engine** (open-source) — now deprecated in favor of Grype + Syft
- **Anchore Enterprise** — full-featured enterprise product

### How Anchore Enterprise Works

```
┌─────────────────────────────────────────────────┐
│              Anchore Enterprise                  │
│                                                  │
│  Image ──► Analyze ──► Policy Check ──► Report   │
│                            │                     │
│                    ┌───────┴────────┐             │
│                    │                │             │
│                 Pass ✅          Fail ❌          │
│                    │                │             │
│              Allow deploy      Block deploy      │
│                                                  │
│  Policies can check:                             │
│  • CVE severity thresholds                       │
│  • Allowed/denied OS packages                    │
│  • Base image requirements                       │
│  • License compliance                            │
│  • File permissions                              │
│  • Exposed ports                                 │
│  • User settings (no root!)                      │
│  • Secret detection                              │
└─────────────────────────────────────────────────┘
```

### Anchore Policy Example

Anchore policies let you define rules like:

```json
{
  "name": "production-policy",
  "rules": [
    {
      "gate": "vulnerabilities",
      "trigger": "package",
      "action": "stop",
      "params": {
        "severity": "critical",
        "fix_available": "true"
      },
      "comment": "Block any image with a fixable critical vulnerability"
    },
    {
      "gate": "dockerfile",
      "trigger": "instruction",
      "action": "stop",
      "params": {
        "instruction": "USER",
        "check": "not_exists"
      },
      "comment": "Block images that don't set a non-root user"
    },
    {
      "gate": "licenses",
      "trigger": "deny_list",
      "action": "warn",
      "params": {
        "licenses": ["GPL-3.0", "AGPL-3.0"]
      },
      "comment": "Warn about copyleft licenses"
    }
  ]
}
```

**What's happening here?**
- **Rule 1:** STOP (block) if there's a critical vulnerability with a fix available
- **Rule 2:** STOP (block) if the Dockerfile doesn't specify a non-root user
- **Rule 3:** WARN (but don't block) if copyleft licenses are found

## 🏗️ Where Does Container Scanning Fit in the Pipeline?

```
Dockerfile ──► docker build ──► Image ──► SCAN ──► Push to ──► Deploy
                                          HERE!    Registry    to K8s
                                           │
                                   ┌───────┴────────┐
                                   │                │
                                Pass ✅          Fail ❌
                                   │                │
                              Push image        Block push,
                              to registry       fix image
```

### The Complete Image Security Pipeline

```yaml
# Complete container security workflow
# .github/workflows/secure-container.yml
name: Secure Container Pipeline

on:
  push:
    branches: [main]
  pull_request:

jobs:
  build-scan-push:
    runs-on: ubuntu-latest
    steps:
      # 1. Get the code
      - uses: actions/checkout@v4

      # 2. Lint the Dockerfile (catch bad practices)
      - name: Lint Dockerfile
        uses: hadolint/hadolint-action@v3.1.0
        with:
          dockerfile: Dockerfile

      # 3. Build the image
      - name: Build image
        run: docker build -t myapp:${{ github.sha }} .

      # 4. Scan for vulnerabilities (Trivy)
      - name: Scan with Trivy
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: 'myapp:${{ github.sha }}'
          exit-code: '1'
          severity: 'CRITICAL,HIGH'
          ignore-unfixed: true

      # 5. Generate SBOM (Software Bill of Materials)
      - name: Generate SBOM
        uses: anchore/sbom-action@v0
        with:
          image: 'myapp:${{ github.sha }}'
          format: spdx-json
          output-file: sbom.spdx.json

      # 6. Push only if all checks pass
      - name: Push to Registry
        if: github.ref == 'refs/heads/main'
        run: |
          docker tag myapp:${{ github.sha }} ghcr.io/${{ github.repository }}:${{ github.sha }}
          docker push ghcr.io/${{ github.repository }}:${{ github.sha }}

      # 7. Upload SBOM as artifact
      - name: Upload SBOM
        uses: actions/upload-artifact@v4
        with:
          name: sbom
          path: sbom.spdx.json
```

## 📝 Best Practices for Container Image Scanning

### 1. **Use Minimal Base Images**

The smaller your base image, the fewer vulnerabilities it will have:

```dockerfile
# ❌ BAD — Full Ubuntu image (hundreds of packages, many vulnerabilities)
FROM ubuntu:22.04

# ⚠️ BETTER — Slim variant (fewer packages)
FROM python:3.12-slim

# ✅ BEST — Alpine (minimal, few packages)
FROM python:3.12-alpine

# 🏆 ULTIMATE — Distroless (almost nothing except your app)
FROM gcr.io/distroless/python3-debian12
```

**Vulnerability count comparison:**

| Base Image | Approximate Package Count | Typical CVEs |
|-----------|--------------------------|-------------|
| `ubuntu:22.04` | 100+ | 50-200 |
| `python:3.12` | 400+ | 100-400 |
| `python:3.12-slim` | 100+ | 30-100 |
| `python:3.12-alpine` | 30+ | 5-30 |
| `distroless/python3` | ~15 | 0-10 |
| `scratch` | 0 | 0 (but only works for static binaries) |

### 2. **Multi-Stage Builds** (Keep Build Tools Out of Production)

```dockerfile
# Stage 1: Build (has compilers, dev tools — we don't care about CVEs here)
FROM golang:1.22 AS builder
WORKDIR /app
COPY . .
RUN go build -o myapp .

# Stage 2: Production (minimal image — this is what we scan and deploy)
FROM gcr.io/distroless/static-debian12
COPY --from=builder /app/myapp /myapp
ENTRYPOINT ["/myapp"]
```

**What's happening here?**
- The **builder** stage has Go compilers, tools, etc. (lots of packages, lots of CVEs)
- The **production** stage only has your compiled binary — almost zero packages, almost zero CVEs ✅

### 3. **Don't Run as Root**

```dockerfile
# ❌ BAD — running as root
FROM node:20-alpine
WORKDIR /app
COPY . .
RUN npm install
CMD ["node", "server.js"]

# ✅ GOOD — running as non-root user
FROM node:20-alpine
WORKDIR /app
COPY . .
RUN npm install
# Create a non-root user and switch to it
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser
CMD ["node", "server.js"]
```

### 4. **Pin Your Base Image Versions**

```dockerfile
# ❌ BAD — "latest" can change anytime
FROM nginx:latest

# ⚠️ BETTER — pinned to a tag
FROM nginx:1.25

# ✅ BEST — pinned to a specific digest (immutable)
FROM nginx:1.25@sha256:abc123def456...
```

### 5. **Rebuild Images Regularly**

Even if your code hasn't changed, rebuild images weekly to pick up OS security patches:

```yaml
# Scheduled rebuild — every Monday
on:
  schedule:
    - cron: '0 6 * * 1'  # Every Monday at 6 AM
```

### 6. **Scan at Multiple Points**

```
Developer    →  CI/CD Build   →  Registry    →  Kubernetes
Local scan      Pipeline scan    Admission       Runtime scan
(shift left)    (gate)           controller      (continuous)
                                 (gate)
```

- **Local scan:** Developer scans before pushing (fast feedback)
- **CI/CD scan:** Automated scan in pipeline (block bad images)
- **Registry scan:** Scan images already stored in the registry
- **Runtime scan:** Continuously scan running containers for new CVEs

### 7. **Use an Admission Controller in Kubernetes**

Block unscanned or vulnerable images from being deployed:

```yaml
# Example: OPA Gatekeeper policy to require image scanning
apiVersion: constraints.gatekeeper.sh/v1beta1
kind: K8sAllowedRepos
metadata:
  name: require-scanned-images
spec:
  match:
    kinds:
      - apiGroups: [""]
        kinds: ["Pod"]
  parameters:
    repos:
      - "registry.example.com/"  # Only allow images from our scanned registry
```

## 🧪 Hands-On Example: Scan and Fix a Vulnerable Image

Let's practice the full workflow:

### Step 1: Create a Deliberately Vulnerable Dockerfile

```dockerfile
# Dockerfile.vulnerable
FROM python:3.9  # Old version — will have vulnerabilities
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
CMD ["python", "app.py"]
```

### Step 2: Build the Image

```bash
docker build -t vulnerable-app:v1 -f Dockerfile.vulnerable .
```

### Step 3: Scan It

```bash
# Scan with Trivy
trivy image vulnerable-app:v1

# Or with Grype
grype vulnerable-app:v1
```

You'll see tons of vulnerabilities! 😱

### Step 4: Fix It — Create a Secure Dockerfile

```dockerfile
# Dockerfile.secure
FROM python:3.12-slim  # Latest version, slim variant
WORKDIR /app

# Install dependencies first (better layer caching)
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Copy application code
COPY . .

# Don't run as root
RUN useradd -m appuser
USER appuser

# Expose port
EXPOSE 8000

CMD ["python", "app.py"]
```

### Step 5: Rebuild and Rescan

```bash
# Build the secure version
docker build -t secure-app:v1 -f Dockerfile.secure .

# Scan it again
trivy image secure-app:v1
```

You should see **dramatically fewer vulnerabilities**! 🎉

### Step 6: Compare the Results

```bash
# With Docker Scout (if available)
docker scout compare vulnerable-app:v1 secure-app:v1
```

## 🔑 Key Takeaways

| Concept | Key Point |
|---------|-----------|
| **Container image scanning** | Checks images for OS + app vulnerabilities |
| **Why it's needed** | Images bundle everything — base OS, libraries, your code |
| **Trivy** | Best all-in-one free scanner — images, SCA, IaC, K8s |
| **Grype** | Fast, lightweight, pairs with Syft for SBOMs |
| **Anchore** | Enterprise-grade, policy-based scanning |
| **Minimal base images** | Fewer packages = fewer vulnerabilities |
| **Multi-stage builds** | Keep build tools out of production images |
| **Don't run as root** | Always use a non-root user in production |
| **Pin versions** | Use specific tags or digests, not `latest` |
| **Rebuild regularly** | Pick up OS patches even if your code hasn't changed |
| **Scan everywhere** | Local, CI/CD, registry, and runtime |
| **SBOM** | Generate a bill of materials for audit and compliance |
