# 16.7 — Infrastructure Security Scanning — Checkov, tfsec, KICS

> **One Line Answer:** Infrastructure Security Scanning tools like **Checkov**, **tfsec**, and **KICS** automatically analyze your Infrastructure as Code (IaC) files — such as Terraform, CloudFormation, Kubernetes manifests, Dockerfiles, and more — to **find misconfigurations and security issues BEFORE you deploy**, so you catch problems like open security groups, unencrypted databases, or public S3 buckets before they ever reach production.

## 🤔 The Big Picture — Why Scan Infrastructure Code?

Let's say you wrote a Terraform file to create an AWS S3 bucket. You accidentally set it to **public access**. If you deploy this without scanning, your bucket is now open to the entire internet — anyone can read your data. 😱

Infrastructure Security Scanning is like having a **security expert review your code automatically** every time you make a change.

```
Without Scanning:
┌────────────────┐     ┌──────────┐     ┌──────────────┐
│ Write IaC Code │ ──→ │  Deploy  │ ──→ │  OOPS! 💀    │
│ (Terraform,    │     │  (apply) │     │  Public S3!  │
│  K8s YAML)     │     │          │     │  Open Ports! │
└────────────────┘     └──────────┘     └──────────────┘

With Scanning:
┌────────────────┐     ┌──────────────┐     ┌──────────┐     ┌──────────────┐
│ Write IaC Code │ ──→ │  SCAN CODE   │ ──→ │  Fix It  │ ──→ │ Safe Deploy  │
│ (Terraform,    │     │  ✅ Checkov  │     │  🔧      │     │ ✅ Secure!   │
│  K8s YAML)     │     │  ✅ tfsec    │     │          │     │              │
│                │     │  ✅ KICS     │     │          │     │              │
└────────────────┘     └──────────────┘     └──────────┘     └──────────────┘
```

### What Kinds of Problems Do These Tools Find?

| Category | Example Issue | Risk |
|----------|--------------|------|
| **Encryption** | S3 bucket without server-side encryption | Data can be read if accessed |
| **Access Control** | Security group allows 0.0.0.0/0 on port 22 | Anyone on the internet can SSH in |
| **Logging** | CloudTrail logging is disabled | You can't see who did what |
| **Networking** | Database is publicly accessible | Attackers can reach your DB directly |
| **Secrets** | Hard-coded passwords in Terraform | Credentials exposed in source code |
| **Compliance** | No tags on resources | Fails SOC2/CIS benchmarks |
| **Container Security** | Docker container running as root | Attacker gets root if container is compromised |

## 1️⃣ Checkov — The Swiss Army Knife of IaC Scanning

### What is Checkov?

Checkov is a **static analysis tool** that scans your infrastructure-as-code files for misconfigurations. It's like a linter, but instead of checking code style, it checks **security and compliance**.

Think of it as: **"A security-obsessed friend who reads your Terraform/K8s files and tells you what's wrong."**

### What Does Checkov Support?

- ✅ Terraform (HCL and plan files)
- ✅ CloudFormation (JSON/YAML)
- ✅ Kubernetes manifests (YAML)
- ✅ Helm charts
- ✅ Dockerfiles
- ✅ ARM templates (Azure)
- ✅ Serverless Framework
- ✅ Ansible
- ✅ Bicep
- ✅ GitHub Actions workflows
- ✅ And more!

### Installation

```bash
# Using pip (Python package manager)
pip install checkov

# Verify installation
checkov --version

# Using Docker (no installation needed)
docker run --rm -v $(pwd):/tf bridgecrew/checkov -d /tf

# Using Homebrew (macOS)
brew install checkov
```

### Your First Scan — Basic Usage

Let's say you have this Terraform file:

```hcl
# main.tf — A BADLY configured S3 bucket (on purpose!)

resource "aws_s3_bucket" "my_data" {
  bucket = "my-company-data-bucket"
}

resource "aws_s3_bucket_public_access_block" "my_data" {
  bucket = aws_s3_bucket.my_data.id

  block_public_acls       = false    # ❌ Not blocking public ACLs!
  block_public_policy     = false    # ❌ Not blocking public policies!
  ignore_public_acls      = false    # ❌ Not ignoring public ACLs!
  restrict_public_buckets = false    # ❌ Not restricting public buckets!
}

resource "aws_s3_bucket_server_side_encryption_configuration" "my_data" {
  bucket = aws_s3_bucket.my_data.id
  # No encryption configured! ❌
}
```

Now let's scan it:

```bash
# Scan the current directory
checkov -d .

# Scan a specific file
checkov -f main.tf

# Scan with a specific framework
checkov -d . --framework terraform
```

### Useful Checkov Commands

```bash
# Scan and output as JSON (for CI/CD pipelines)
checkov -d . -o json

# Scan and output as JUnit XML (for test reports)
checkov -d . -o junitxml > results.xml

# Scan and output as SARIF (for GitHub Security tab)
checkov -d . -o sarif > results.sarif

# Scan only specific checks
checkov -d . --check CKV_AWS_18,CKV_AWS_19

# Skip specific checks (when you have a reason)
checkov -d . --skip-check CKV_AWS_18

# Show all available checks for a framework
checkov --list --framework terraform

# Scan Kubernetes manifests
checkov -d ./k8s-manifests/ --framework kubernetes

# Scan Dockerfiles
checkov -f Dockerfile --framework dockerfile

# Scan Terraform plan file (more accurate than HCL scanning)
terraform plan -out=plan.tfplan
terraform show -json plan.tfplan > plan.json
checkov -f plan.json

# Set a hard gate — fail if ANY check fails (useful in CI)
checkov -d . --hard-fail-on HIGH,CRITICAL

# Compact output (less verbose)
checkov -d . --compact
```

### Inline Suppressions — Skipping Checks With a Reason

Sometimes you have a **valid reason** to skip a check (e.g., a public website bucket is supposed to be public). You can suppress checks inline:

```hcl
# For Terraform — add a comment above the resource
#checkov:skip=CKV_AWS_19:This bucket is for public website hosting
resource "aws_s3_bucket" "website" {
  bucket = "my-public-website"
}
```

```yaml
# For Kubernetes — add an annotation
apiVersion: v1
kind: Pod
metadata:
  name: my-pod
  annotations:
    checkov.io/skip: "CKV_K8S_40=This pod needs to run as root for init"
```

```dockerfile
# For Dockerfile — add a comment
#checkov:skip=CKV_DOCKER_2:Health check is handled by Kubernetes
FROM python:3.12-slim
```

### Writing Custom Checks in Checkov

You can write your own checks! Checkov supports custom policies in **Python** or **YAML**.

**YAML Custom Check Example:**

```yaml
# custom_checks/require_team_tag.yaml
metadata:
  id: "CUSTOM_AWS_1"
  name: "Ensure all resources have a 'team' tag"
  category: "GENERAL_SECURITY"
  guidelines: "All resources must have a 'team' tag for ownership tracking."

definition:
  cond_type: "attribute"
  resource_types:
    - "aws_s3_bucket"
    - "aws_instance"
    - "aws_rds_cluster"
  attribute: "tags.team"
  operator: "exists"
```

```bash
# Run Checkov with your custom checks
checkov -d . --external-checks-dir ./custom_checks
```

**Python Custom Check Example:**

```python
# custom_checks/require_encryption.py
from checkov.common.models.enums import CheckResult, CheckCategories
from checkov.terraform.checks.resource.base_resource_check import BaseResourceCheck

class EnsureRDSEncryption(BaseResourceCheck):
    def __init__(self):
        name = "Ensure RDS instance has storage encryption enabled"
        id = "CUSTOM_AWS_2"
        supported_resources = ["aws_db_instance"]
        categories = [CheckCategories.ENCRYPTION]
        super().__init__(name=name, id=id, categories=categories,
                        supported_resources=supported_resources)

    def scan_resource_conf(self, conf):
        # Check if storage_encrypted is set to True
        storage_encrypted = conf.get("storage_encrypted", [False])
        if storage_encrypted == [True] or storage_encrypted is True:
            return CheckResult.PASSED
        return CheckResult.FAILED

# Register the check
check = EnsureRDSEncryption()
```

### Checkov in CI/CD — GitHub Actions Example

```yaml
# .github/workflows/checkov.yml
name: Infrastructure Security Scan

on:
  pull_request:
    paths:
      - 'terraform/**'
      - 'kubernetes/**'

jobs:
  checkov:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Run Checkov
        uses: bridgecrewio/checkov-action@v12
        with:
          directory: terraform/
          framework: terraform
          output_format: sarif
          output_file_path: results.sarif
          soft_fail: false           # Fail the pipeline if issues found
          skip_check: CKV_AWS_18     # Skip specific checks if needed

      - name: Upload SARIF results
        if: always()
        uses: github/codeql-action/upload-sarif@v3
        with:
          sarif_file: results.sarif
```

### Checkov Configuration File

Instead of passing flags every time, use a config file:

```yaml
# .checkov.yml (place in your repo root)
branch: main
compact: true
directory:
  - terraform/
  - kubernetes/
download-external-modules: true
evaluate-variables: true
framework:
  - terraform
  - kubernetes
  - dockerfile
output:
  - cli
  - sarif
skip-check:
  - CKV_AWS_18   # Access logging — handled separately
  - CKV_AWS_999  # Not applicable to our setup
soft-fail: false
```

## 2️⃣ tfsec — Terraform-Specific Security Scanner

### What is tfsec?

tfsec is a **fast, focused security scanner specifically for Terraform**. It was built from the ground up to understand HCL (HashiCorp Configuration Language) deeply.

> **Important Update:** tfsec has been **officially merged into Trivy** by Aqua Security. The tfsec project is now in maintenance mode. For new projects, use `trivy config` instead. But the concepts and checks are the same!

Think of it as: **"A Terraform security specialist — it only does one thing, but it does it really well."**

### Installation

```bash
# Using Homebrew (macOS/Linux)
brew install tfsec

# Using Go
go install github.com/aquasecurity/tfsec/cmd/tfsec@latest

# Using Chocolatey (Windows)
choco install tfsec

# Using Docker
docker run --rm -v $(pwd):/src aquasecurity/tfsec /src

# Verify installation
tfsec --version
```

### Using tfsec — The Modern Way (via Trivy)

Since tfsec is now part of Trivy, here's how to use it:

```bash
# Install Trivy
# macOS
brew install trivy

# Ubuntu/Debian
sudo apt-get install trivy

# Using Docker
docker run --rm -v $(pwd):/src aquasecurity/trivy config /src

# Scan Terraform code with Trivy (replaces tfsec)
trivy config .
trivy config ./terraform/
trivy config --severity HIGH,CRITICAL .
```

## 3️⃣ KICS — Keeping Infrastructure as Code Secure

### What is KICS?

KICS (Keeping Infrastructure as Code Secure) is an **open-source IaC scanner** by Checkmarx. It has **the largest library of built-in queries** (2000+) and supports a wide range of IaC platforms.

Think of it as: **"The biggest security checklist for your infrastructure code — it checks EVERYTHING."**

### What Makes KICS Special?

- **2000+ built-in queries** — the largest built-in rule set
- **Supports many platforms** — Terraform, CloudFormation, K8s, Docker, Ansible, Helm, Azure Resource Manager, Google Deployment Manager, gRPC, OpenAPI, and more
- **Rego-based custom queries** — same language used by OPA/Gatekeeper
- **Fast** — written in Go for speed
- **Great HTML reports** — visual, shareable reports

### Installation

```bash
# Using Docker (recommended — no install needed)
docker run --rm -v $(pwd):/path checkmarx/kics:latest scan -p /path

# Using Homebrew
brew install kics

# Using Go
go install github.com/Checkmarx/kics@latest

# Download binary from GitHub releases
# https://github.com/Checkmarx/kics/releases

# Verify installation
kics version
```

### Basic KICS Usage

```bash
# Scan the current directory
kics scan -p .

# Scan a specific directory
kics scan -p ./terraform/

# Scan multiple directories
kics scan -p ./terraform/ -p ./kubernetes/ -p ./docker/

# Scan specific file types only
kics scan -p . --type terraform,kubernetes,dockerfile
```

### Example — Scanning a Kubernetes Manifest

Here's a Kubernetes deployment with security issues:

```yaml
# bad-deployment.yaml — Multiple security issues!
apiVersion: apps/v1
kind: Deployment
metadata:
  name: vulnerable-app
spec:
  replicas: 1
  selector:
    matchLabels:
      app: vulnerable-app
  template:
    metadata:
      labels:
        app: vulnerable-app
    spec:
      containers:
        - name: app
          image: myapp:latest                    # 🚨 Using 'latest' tag!
          ports:
            - containerPort: 8080
          securityContext:
            runAsUser: 0                         # 🚨 Running as root!
            privileged: true                     # 🚨 Privileged container!
            allowPrivilegeEscalation: true       # 🚨 Can escalate privileges!
            readOnlyRootFilesystem: false         # 🚨 Writable filesystem!
          # No resource limits!                  # 🚨 Can consume all node resources!
          # No liveness/readiness probes!        # 🚨 No health checks!
```

Scanning with KICS:

```bash
kics scan -p bad-deployment.yaml
```

## 🏗️ Real-World Setup — Using Multiple Tools Together

In practice, many teams use **multiple scanners** for defense in depth. Here's a complete CI/CD pipeline:

```yaml
# .github/workflows/iac-security.yml
name: IaC Security Pipeline

on:
  pull_request:
    paths:
      - 'infrastructure/**'

jobs:
  # Job 1: Checkov — Broad coverage
  checkov-scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run Checkov
        uses: bridgecrewio/checkov-action@v12
        with:
          directory: infrastructure/
          framework: terraform,kubernetes
          output_format: sarif
          output_file_path: checkov.sarif
          soft_fail: true
      - uses: github/codeql-action/upload-sarif@v3
        if: always()
        with:
          sarif_file: checkov.sarif
          category: checkov

  # Job 2: Trivy (tfsec replacement) — Fast Terraform scan
  trivy-scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run Trivy Config Scan
        uses: aquasecurity/trivy-action@master
        with:
          scan-type: config
          scan-ref: infrastructure/
          format: sarif
          output: trivy.sarif
          severity: HIGH,CRITICAL
      - uses: github/codeql-action/upload-sarif@v3
        if: always()
        with:
          sarif_file: trivy.sarif
          category: trivy

  # Job 3: KICS — Maximum query coverage
  kics-scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run KICS
        uses: checkmarx/kics-github-action@v2.1.0
        with:
          path: infrastructure/
          fail_on: critical
          output_path: kics-results
          output_formats: sarif
      - uses: github/codeql-action/upload-sarif@v3
        if: always()
        with:
          sarif_file: kics-results/results.sarif
          category: kics
```

## 🔧 Fixing Common Issues Found by These Tools

### Fix 1: S3 Bucket — Make It Secure

```hcl
# ❌ BAD — Insecure S3 Bucket
resource "aws_s3_bucket" "data" {
  bucket = "my-data-bucket"
}

# ✅ GOOD — Secure S3 Bucket
resource "aws_s3_bucket" "data" {
  bucket = "my-data-bucket"
}

resource "aws_s3_bucket_public_access_block" "data" {
  bucket = aws_s3_bucket.data.id

  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

resource "aws_s3_bucket_server_side_encryption_configuration" "data" {
  bucket = aws_s3_bucket.data.id

  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "aws:kms"    # Use KMS for encryption
    }
  }
}

resource "aws_s3_bucket_versioning" "data" {
  bucket = aws_s3_bucket.data.id

  versioning_configuration {
    status = "Enabled"    # Enable versioning for data protection
  }
}

resource "aws_s3_bucket_logging" "data" {
  bucket = aws_s3_bucket.data.id

  target_bucket = aws_s3_bucket.logs.id
  target_prefix = "s3-access-logs/"
}
```

### Fix 2: Security Group — Restrict Access

```hcl
# ❌ BAD — SSH open to the world
resource "aws_security_group" "bad" {
  ingress {
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

# ✅ GOOD — SSH restricted to your office IP
resource "aws_security_group" "good" {
  ingress {
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["203.0.113.0/24"]    # Your office IP range only
    description = "SSH access from office network"
  }
}
```

### Fix 3: Kubernetes Deployment — Secure Pod

```yaml
# ❌ BAD — Insecure Pod
spec:
  containers:
    - name: app
      image: myapp:latest
      securityContext:
        runAsUser: 0
        privileged: true

# ✅ GOOD — Secure Pod
spec:
  securityContext:
    runAsNonRoot: true
    seccompProfile:
      type: RuntimeDefault
  containers:
    - name: app
      image: myapp:1.2.3                       # Pinned version
      securityContext:
        runAsUser: 1000                         # Non-root user
        runAsGroup: 1000
        privileged: false                       # Not privileged
        allowPrivilegeEscalation: false         # Can't escalate
        readOnlyRootFilesystem: true            # Read-only filesystem
        capabilities:
          drop:
            - ALL                               # Drop all capabilities
      resources:
        limits:
          memory: "256Mi"
          cpu: "500m"
        requests:
          memory: "128Mi"
          cpu: "250m"
      livenessProbe:
        httpGet:
          path: /healthz
          port: 8080
        initialDelaySeconds: 10
        periodSeconds: 30
      readinessProbe:
        httpGet:
          path: /ready
          port: 8080
        initialDelaySeconds: 5
        periodSeconds: 10
```

## 📊 Pre-commit Hooks — Scan Before You Even Commit

The BEST time to catch issues is **before the code is even committed**:

```yaml
# .pre-commit-config.yaml
repos:
  - repo: https://github.com/bridgecrewio/checkov
    rev: '3.2.0'
    hooks:
      - id: checkov
        args: ['--compact', '--quiet']

  - repo: https://github.com/aquasecurity/tfsec
    rev: 'v1.28.4'
    hooks:
      - id: tfsec

  - repo: https://github.com/antonbabenko/pre-commit-terraform
    rev: 'v1.88.0'
    hooks:
      - id: terraform_tfsec
      - id: terraform_checkov
```

```bash
# Install pre-commit
pip install pre-commit

# Install the hooks
pre-commit install

# Now every time you run 'git commit', the scanners run automatically!
# You can also run them manually:
pre-commit run --all-files
```

## 🎯 Best Practices — Infrastructure Security Scanning

| Practice | Why |
|----------|-----|
| **Scan in CI/CD pipelines** | Catch issues before they reach production |
| **Use pre-commit hooks** | Catch issues before they even get committed |
| **Don't suppress without a reason** | Every skip should have a documented justification |
| **Use multiple tools** | Each tool catches different things (defense in depth) |
| **Scan Terraform plans, not just HCL** | Plan-based scanning is more accurate |
| **Keep tools updated** | New checks are added regularly for new vulnerabilities |
| **Start with soft-fail** | When first adopting, warn but don't block (fix existing issues gradually) |
| **Escalate to hard-fail** | Once existing issues are fixed, fail the pipeline on new issues |
| **Track findings over time** | Use dashboards to see if your security posture is improving |
| **Educate the team** | Developers should understand WHY checks exist, not just fix them blindly |
