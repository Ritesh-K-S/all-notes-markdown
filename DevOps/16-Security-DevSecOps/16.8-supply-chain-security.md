# 16.8 — Supply Chain Security — SBOM, Sigstore, SLSA Framework

> **One Line Answer:** Supply Chain Security is about making sure that **every piece of software you use, build, and deploy** — from open-source libraries to container images to CI/CD pipelines — hasn't been **tampered with, compromised, or injected with malicious code** at any point in its journey from source code to production.

---

## 🤔 The Big Picture — What is a "Software Supply Chain"?

Think of a **real-world supply chain** for making a pizza:

1. A farmer grows wheat → flour mill grinds it → bakery makes dough → pizzeria uses the dough
2. A farmer grows tomatoes → factory makes sauce → pizzeria uses the sauce
3. A dairy farm produces milk → cheese factory makes mozzarella → pizzeria uses the cheese

If **any step** is contaminated (bad flour, spoiled sauce, fake cheese), the pizza is ruined — or worse, someone gets sick. And the pizzeria might not even know!

**Software works the same way:**

```
Software Supply Chain:
┌──────────────┐   ┌────────────────┐   ┌────────────────┐   ┌───────────────┐
│  Open Source  │   │   Your Code    │   │  Build System  │   │  Container    │
│  Libraries   │──→│  (app logic)   │──→│  (CI/CD)       │──→│  Registry     │
│  (npm, pip)  │   │                │   │  (GitHub       │   │  (Docker Hub) │
│              │   │                │   │   Actions)     │   │               │
└──────────────┘   └────────────────┘   └────────────────┘   └───────────────┘
       │                   │                    │                     │
       ▼                   ▼                    ▼                     ▼
   Could have          Could be            Pipeline could       Image could
   malicious code      tampered with       be compromised       be replaced
   (dependency         (unauthorized       (injected build      (registry
    confusion)          commits)            steps)               poisoning)
```

### Real-World Attacks That Changed Everything

| Attack | Year | What Happened | Impact |
|--------|------|---------------|--------|
| **SolarWinds** | 2020 | Attackers injected malicious code into the SolarWinds Orion build process | 18,000+ organizations compromised, including US government |
| **Codecov** | 2021 | Attackers modified the Codecov Bash Uploader script | CI/CD secrets stolen from thousands of repos |
| **ua-parser-js** | 2021 | Popular npm package hijacked with crypto miners | 8 million weekly downloads affected |
| **Log4Shell** | 2021 | Critical vulnerability in Log4j library | Virtually every Java application was vulnerable |
| **3CX** | 2023 | Desktop app's build process compromised via a supply chain attack on a trading app | Millions of users received malware through auto-update |
| **xz Utils** | 2024 | Maintainer social engineering — malicious backdoor inserted into a critical Linux compression utility | Could have affected virtually every Linux system |

These attacks showed that **you can't just trust your code — you need to trust everything your code depends on**.

## 1️⃣ SBOM — Software Bill of Materials

### What is an SBOM?

An SBOM is a **complete list of every component, library, and dependency** in your software — like a **nutrition label on food** or an **ingredients list on a medicine box**.

```
Real World Analogy:

Medication Box:
┌──────────────────────────────┐
│  Aspirin 500mg               │
│  ─────────────────           │
│  Active Ingredients:         │
│   • Acetylsalicylic acid     │
│  Inactive Ingredients:       │
│   • Corn starch              │
│   • Hypromellose             │
│   • Powdered cellulose       │
│   • Triacetin                │
│  Manufactured: 2024-01-15    │
│  Lot: ABC123                 │
│  Supplier: PharmaCo Inc      │
└──────────────────────────────┘

Software SBOM:
┌──────────────────────────────┐
│  MyWebApp v2.5.0             │
│  ─────────────────           │
│  Components:                 │
│   • express 4.18.2           │
│   • lodash 4.17.21           │
│   • axios 1.6.2              │
│   • jsonwebtoken 9.0.2       │
│   • pg 8.11.3                │
│  Build Date: 2024-01-15      │
│  Build ID: abc123            │
│  Author: MyCompany           │
└──────────────────────────────┘
```

### Why Do You Need an SBOM?

| Scenario | Without SBOM | With SBOM |
|----------|-------------|-----------|
| **New vulnerability announced** (like Log4Shell) | "Wait... do we even USE Log4j? In which apps? Which versions?" 😰 | "Let me search our SBOMs... Yes, 3 apps use Log4j 2.14.1. Here they are." ✅ |
| **License compliance audit** | Manually checking every dependency, hoping you don't miss one 😓 | Auto-generated list of all licenses across all components ✅ |
| **Customer asks "what's in your software?"** | "Uh... let me check... give us a week" 😬 | "Here's our SBOM — full transparency" ✅ |
| **Regulatory requirement** (US Executive Order 14028) | Non-compliant ❌ | Compliant ✅ |

### SBOM Formats — CycloneDX vs SPDX

There are two main SBOM formats:

| Feature | CycloneDX | SPDX |
|---------|-----------|------|
| **Created by** | OWASP | Linux Foundation |
| **Focus** | Security-oriented | License compliance + security |
| **Formats** | JSON, XML | JSON, RDF, Tag-Value, YAML |
| **Adoption** | Growing fast, DevSecOps focused | Established, widely adopted |
| **Standard** | OWASP standard | ISO/IEC 5962:2021 (international standard) |
| **Best for** | Vulnerability tracking, DevOps | License compliance, legal teams |

### Generating SBOMs — Tools and Commands

#### Using Syft (by Anchore) — Most Popular

```bash
# Install Syft
curl -sSfL https://raw.githubusercontent.com/anchore/syft/main/install.sh | sh -s -- -b /usr/local/bin

# Generate SBOM for a container image
syft alpine:3.19
syft myapp:latest

# Generate SBOM in CycloneDX JSON format
syft myapp:latest -o cyclonedx-json > sbom.cdx.json

# Generate SBOM in SPDX JSON format
syft myapp:latest -o spdx-json > sbom.spdx.json

# Generate SBOM for a directory (source code)
syft dir:./my-project

# Generate SBOM for a Dockerfile
syft dir:. --catalogers dockerfile

# Generate SBOM from a running container
syft docker:my-running-container
```

#### Using Trivy

```bash
# Generate SBOM with Trivy
trivy image --format cyclonedx --output sbom.cdx.json myapp:latest
trivy image --format spdx-json --output sbom.spdx.json myapp:latest

# Scan a filesystem for SBOM
trivy fs --format cyclonedx --output sbom.cdx.json .
```

#### Using Docker (Built-in since Docker Desktop 4.7+)

```bash
# Docker Scout SBOM generation
docker scout sbom myapp:latest
docker scout sbom --format spdx myapp:latest
docker scout sbom --format cyclonedx myapp:latest --output sbom.json
```

#### Using npm/pip/go (Language-Specific)

```bash
# Node.js — CycloneDX
npx @cyclonedx/cyclonedx-npm --output-file sbom.json

# Python — CycloneDX
pip install cyclonedx-bom
cyclonedx-py environment --output sbom.json

# Go — CycloneDX
go install github.com/CycloneDX/cyclonedx-gomod/cmd/cyclonedx-gomod@latest
cyclonedx-gomod mod -json > sbom.json
```

### Scanning SBOMs for Vulnerabilities

Once you have an SBOM, you can **scan it for known vulnerabilities** without needing the original image or source code:

```bash
# Scan an SBOM with Grype (by Anchore)
# Install Grype
curl -sSfL https://raw.githubusercontent.com/anchore/grype/main/install.sh | sh -s -- -b /usr/local/bin

# Scan the SBOM
grype sbom:./sbom.cdx.json

# Scan with severity filter
grype sbom:./sbom.cdx.json --only-fixed --fail-on high

# Scan an SBOM with Trivy
trivy sbom sbom.cdx.json
```

### SBOM in CI/CD — GitHub Actions

```yaml
# .github/workflows/sbom.yml
name: Generate and Store SBOM

on:
  push:
    branches: [main]

jobs:
  sbom:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Build Docker Image
        run: docker build -t myapp:${{ github.sha }} .

      - name: Generate SBOM with Syft
        uses: anchore/sbom-action@v0
        with:
          image: myapp:${{ github.sha }}
          format: cyclonedx-json
          output-file: sbom.cdx.json

      - name: Scan SBOM for Vulnerabilities
        uses: anchore/scan-action@v4
        with:
          sbom: sbom.cdx.json
          fail-build: true
          severity-cutoff: high

      - name: Upload SBOM as Artifact
        uses: actions/upload-artifact@v4
        with:
          name: sbom
          path: sbom.cdx.json

      - name: Attach SBOM to GitHub Release
        if: startsWith(github.ref, 'refs/tags/')
        uses: softprops/action-gh-release@v2
        with:
          files: sbom.cdx.json
```

## 2️⃣ Sigstore — Signing and Verifying Software Artifacts

### What is Sigstore?

Sigstore is a **free, open-source project** that makes it easy to **digitally sign, verify, and protect** software artifacts (container images, binaries, SBOMs, etc.).

Think of it as: **"A notary for software"** — it proves that an artifact is authentic and hasn't been tampered with.

```
The Problem Sigstore Solves:

Without Signing:
┌──────────┐                    ┌────────────┐
│ Developer │ ─── pushes ──→    │  Registry  │    ← Anyone could have pushed this!
│           │     image         │            │    ← How do you know it's real?
└──────────┘                    └────────────┘    ← Was it tampered with?

With Sigstore:
┌──────────┐    ┌───────────┐    ┌────────────┐
│ Developer │──→│ Signs the │──→ │  Registry  │    ← Signed! We know WHO pushed it
│           │   │ image     │    │  + Signature│    ← We can VERIFY it's untampered
└──────────┘    │ with      │    └────────────┘    ← Transparent and auditable
                │ Sigstore  │
                └───────────┘
```

### Sigstore Components

Sigstore is not a single tool — it's a **collection of tools** that work together:

```
Sigstore Ecosystem:
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌─────────────┐ │
│  │  Cosign   │   │  Fulcio   │   │  Rekor    │   │  Gitsign    │ │
│  │          │   │          │   │          │   │             │ │
│  │ Signs &  │   │ Issues   │   │ Tamper-  │   │ Signs Git   │ │
│  │ verifies │   │ short-   │   │ evident  │   │ commits     │ │
│  │ container│   │ lived    │   │ trans-   │   │ with        │ │
│  │ images   │   │ certs    │   │ parency  │   │ Sigstore    │ │
│  │ & blobs  │   │ (like    │   │ log      │   │             │ │
│  │          │   │ Let's    │   │ (public  │   │             │ │
│  │          │   │ Encrypt  │   │ audit    │   │             │ │
│  │          │   │ for code │   │ trail)   │   │             │ │
│  │          │   │ signing) │   │          │   │             │ │
│  └──────────┘   └──────────┘   └──────────┘   └─────────────┘ │
│                                                                 │
│  Cosign = The main tool you'll use day-to-day                  │
│  Fulcio = Certificate Authority (issues signing certificates)   │
│  Rekor  = Transparency log (public, tamper-proof record)       │
│  Gitsign = For signing Git commits using Sigstore              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### Cosign — The Star of Sigstore

Cosign is the tool you'll use most. It **signs and verifies container images**.

#### Installation

```bash
# Using Homebrew (macOS/Linux)
brew install cosign

# Using Go
go install github.com/sigstore/cosign/v2/cmd/cosign@latest

# Download binary
# https://github.com/sigstore/cosign/releases

# Using Docker
docker run --rm gcr.io/projectsigstore/cosign version

# Verify installation
cosign version
```

#### Keyless Signing (Recommended — The Magic of Sigstore!)

This is what makes Sigstore special. You **don't need to manage keys**! Sigstore uses your **identity** (GitHub, Google, Microsoft account) to sign.

```bash
# Sign a container image (keyless — uses your OIDC identity)
cosign sign myregistry.io/myapp:v1.0.0

# What happens behind the scenes:
# 1. Cosign opens your browser for OIDC login (GitHub/Google/Microsoft)
# 2. Fulcio issues a short-lived certificate tied to your identity
# 3. The image is signed with this certificate
# 4. The signature is recorded in the Rekor transparency log
# 5. The certificate expires (short-lived = more secure!)

# Verify the signed image
cosign verify myregistry.io/myapp:v1.0.0 \
  --certificate-identity=developer@company.com \
  --certificate-oidc-issuer=https://accounts.google.com

# The output shows:
# ✅ The image signature is valid
# ✅ The signer's identity
# ✅ The timestamp of signing
# ✅ The Rekor log entry
```

#### Key-Based Signing (Traditional Approach)

If you prefer managing your own keys:

```bash
# Generate a key pair
cosign generate-key-pair
# This creates:
# - cosign.key (private key — keep this SECRET!)
# - cosign.pub (public key — share this freely)

# Sign an image with your private key
cosign sign --key cosign.key myregistry.io/myapp:v1.0.0

# Verify with the public key
cosign verify --key cosign.pub myregistry.io/myapp:v1.0.0
```

#### Signing and Attaching SBOMs

```bash
# Attach an SBOM to a container image
cosign attach sbom --sbom sbom.cdx.json myregistry.io/myapp:v1.0.0

# Sign the SBOM attachment
cosign sign --attachment sbom myregistry.io/myapp:v1.0.0

# Verify the SBOM
cosign verify --attachment sbom myregistry.io/myapp:v1.0.0 \
  --certificate-identity=ci@company.com \
  --certificate-oidc-issuer=https://token.actions.githubusercontent.com

# Attest an SBOM (more advanced — creates a signed attestation)
cosign attest --predicate sbom.cdx.json \
  --type cyclonedx \
  myregistry.io/myapp:v1.0.0

# Verify the attestation
cosign verify-attestation \
  --type cyclonedx \
  --certificate-identity=ci@company.com \
  --certificate-oidc-issuer=https://token.actions.githubusercontent.com \
  myregistry.io/myapp:v1.0.0
```

#### Signing Blobs (Files, Binaries, etc.)

```bash
# Sign any file
cosign sign-blob --output-signature sig.txt myapp-binary

# Verify a signed file
cosign verify-blob --signature sig.txt myapp-binary \
  --certificate-identity=developer@company.com \
  --certificate-oidc-issuer=https://accounts.google.com
```

### Cosign in CI/CD — GitHub Actions (Keyless)

```yaml
# .github/workflows/sign-image.yml
name: Build, Push & Sign Container Image

on:
  push:
    tags:
      - 'v*'

permissions:
  contents: read
  packages: write
  id-token: write        # ← REQUIRED for keyless signing!

jobs:
  build-sign:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Login to GHCR
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build and Push Image
        id: build
        uses: docker/build-push-action@v5
        with:
          push: true
          tags: ghcr.io/${{ github.repository }}:${{ github.ref_name }}

      - name: Install Cosign
        uses: sigstore/cosign-installer@v3

      - name: Sign the Image (Keyless!)
        env:
          DIGEST: ${{ steps.build.outputs.digest }}
        run: |
          cosign sign --yes ghcr.io/${{ github.repository }}@${DIGEST}
          # --yes skips the confirmation prompt
          # Using digest (not tag) ensures we sign the exact image

      - name: Generate SBOM
        uses: anchore/sbom-action@v0
        with:
          image: ghcr.io/${{ github.repository }}@${{ steps.build.outputs.digest }}
          format: cyclonedx-json
          output-file: sbom.cdx.json

      - name: Attest SBOM
        env:
          DIGEST: ${{ steps.build.outputs.digest }}
        run: |
          cosign attest --yes \
            --predicate sbom.cdx.json \
            --type cyclonedx \
            ghcr.io/${{ github.repository }}@${DIGEST}
```

### Verifying Images in Kubernetes — Policy Enforcement

You can enforce that **only signed images** are allowed in your cluster:

```yaml
# Using Kyverno — Kubernetes-native policy
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: verify-image-signatures
spec:
  validationFailureAction: Enforce   # Block unsigned images!
  rules:
    - name: verify-cosign-signature
      match:
        any:
          - resources:
              kinds:
                - Pod
      verifyImages:
        - imageReferences:
            - "ghcr.io/myorg/*"
          attestors:
            - entries:
                - keyless:
                    subject: "https://github.com/myorg/*"
                    issuer: "https://token.actions.githubusercontent.com"
                    rekor:
                      url: https://rekor.sigstore.dev
```

```yaml
# Using Sigstore Policy Controller
apiVersion: policy.sigstore.dev/v1beta1
kind: ClusterImagePolicy
metadata:
  name: require-signed-images
spec:
  images:
    - glob: "ghcr.io/myorg/**"
  authorities:
    - keyless:
        identities:
          - issuer: "https://token.actions.githubusercontent.com"
            subject: "https://github.com/myorg/myrepo/.github/workflows/build.yml@refs/heads/main"
        ctlog:
          url: https://rekor.sigstore.dev
```

### Rekor — The Transparency Log

Rekor is a **public, tamper-evident log** that records every signing event. Think of it as a **public ledger** (like a blockchain, but simpler).

```bash
# Search Rekor for entries about your image
rekor-cli search --email developer@company.com

# Get details of a specific log entry
rekor-cli get --uuid <entry-uuid>

# Search by artifact hash
rekor-cli search --sha 9f86d081884c7d659a2feaa0c55ad015a3bf4f1b2b0b822cd15d6c15b0f00a08

# You can also browse entries at:
# https://search.sigstore.dev
```

## 3️⃣ SLSA Framework — Supply-chain Levels for Software Artifacts

### What is SLSA?

SLSA (pronounced "salsa") is a **security framework** that defines **levels of security guarantees** for your software build process. It tells you **how trustworthy your build process is**.

Think of it as: **"A food safety rating for software builds"** — like how restaurants get health inspection grades (A, B, C), SLSA gives your build process a security grade.

```
SLSA is like restaurant health ratings:

Restaurant:                        Software Build:
┌──────────────┐                   ┌──────────────┐
│ Grade A  ⭐⭐⭐│                   │ SLSA Level 3 │
│ Clean kitchen│                   │ Secure build │
│ Inspected    │                   │ Verified     │
│ Documented   │                   │ Auditable    │
└──────────────┘                   └──────────────┘

"You can trust this pizza               "You can trust this software
 won't make you sick"                     wasn't tampered with"
```

### SLSA Levels Explained

```
SLSA Levels — From "No Security" to "Maximum Trust"

Level 0 — No guarantees
   "I built this on my laptop, trust me bro" 😅
   ┌──────────────────────────────────────────┐
   │  No provenance. No verification.         │
   │  Anyone could have built this anywhere.  │
   └──────────────────────────────────────────┘

Level 1 — Provenance exists
   "Here's documentation of how it was built" 📝
   ┌──────────────────────────────────────────┐
   │  Build process is documented.            │
   │  Provenance metadata is generated.       │
   │  But it's NOT tamper-proof yet.          │
   └──────────────────────────────────────────┘

Level 2 — Hosted build + signed provenance
   "It was built on a CI system, and provenance is signed" ✍️
   ┌──────────────────────────────────────────┐
   │  Built on a hosted build service         │
   │  (GitHub Actions, Cloud Build, etc.)     │
   │  Provenance is signed and verifiable.    │
   │  Harder to tamper with.                  │
   └──────────────────────────────────────────┘

Level 3 — Hardened build + non-falsifiable provenance
   "The build environment is isolated, and provenance CANNOT be faked" 🔒
   ┌──────────────────────────────────────────┐
   │  Build runs in an isolated, ephemeral    │
   │  environment.                            │
   │  Provenance is generated by the build    │
   │  platform (not the user).                │
   │  Provenance cannot be falsified.         │
   │  Two-person review required for changes. │
   └──────────────────────────────────────────┘
```

### SLSA Key Concepts

#### Provenance — "Where Did This Come From?"

Provenance is a **signed document** that describes:
- **Who** triggered the build
- **What** source code was used
- **Where** it was built (which CI/CD system)
- **How** it was built (build steps, parameters)
- **When** it was built

```json
// Example SLSA Provenance (simplified)
{
  "_type": "https://in-toto.io/Statement/v1",
  "subject": [
    {
      "name": "myapp",
      "digest": {
        "sha256": "9f86d081884c7d659a2feaa0c55ad015a3bf4f1b2b0b822cd15d6c15b0f00a08"
      }
    }
  ],
  "predicateType": "https://slsa.dev/provenance/v1",
  "predicate": {
    "buildDefinition": {
      "buildType": "https://github.com/slsa-framework/slsa-github-generator/generic@v2",
      "externalParameters": {
        "source": {
          "uri": "git+https://github.com/myorg/myapp@refs/tags/v1.0.0",
          "digest": {
            "sha1": "abc123def456..."
          }
        }
      }
    },
    "runDetails": {
      "builder": {
        "id": "https://github.com/slsa-framework/slsa-github-generator/.github/workflows/generator_generic_slsa3.yml@refs/tags/v2.0.0"
      },
      "metadata": {
        "invocationId": "https://github.com/myorg/myapp/actions/runs/12345",
        "startedOn": "2024-01-15T10:30:00Z",
        "finishedOn": "2024-01-15T10:35:00Z"
      }
    }
  }
}
```

The easiest way to achieve SLSA Level 3 is with the **SLSA GitHub Generator**:

#### For Container Images:

```yaml
on:
  push:
    tags:
      - 'v*'

  id-token: write     # For signing

jobs:
  build:
    runs-on: ubuntu-latest
    outputs:
      digest: ${{ steps.build.outputs.digest }}
    steps:
      - uses: actions/checkout@v4

      - name: Login to GHCR
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build and Push
        id: build
        uses: docker/build-push-action@v5
        with:
          push: true
          tags: ghcr.io/${{ github.repository }}:${{ github.ref_name }}

  # This job generates SLSA provenance
  provenance:
    needs: build
    permissions:
      actions: read
      id-token: write
      packages: write
    uses: slsa-framework/slsa-github-generator/.github/workflows/generator_container_slsa3.yml@v2.0.0
    with:
      image: ghcr.io/${{ github.repository }}
      digest: ${{ needs.build.outputs.digest }}
      registry-username: ${{ github.actor }}
    secrets:
      registry-password: ${{ secrets.GITHUB_TOKEN }}
```

### Complete GitHub Actions Pipeline

```yaml
# .github/workflows/secure-supply-chain.yml
name: Secure Supply Chain Pipeline

on:
  push:
    tags:
      - 'v*'

permissions:
  contents: read
  packages: write
  id-token: write
  actions: read

env:
  IMAGE: ghcr.io/${{ github.repository }}

jobs:
  build-sign-attest:
    runs-on: ubuntu-latest
    outputs:
      digest: ${{ steps.build.outputs.digest }}
    steps:
      - uses: actions/checkout@v4

      # Step 1: Build the image
      - name: Login to GHCR
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build and Push
        id: build
        uses: docker/build-push-action@v5
        with:
          push: true
          tags: ${{ env.IMAGE }}:${{ github.ref_name }}

      # Step 2: Generate SBOM
      - name: Generate SBOM
        uses: anchore/sbom-action@v0
        with:
          image: ${{ env.IMAGE }}@${{ steps.build.outputs.digest }}
          format: cyclonedx-json
          output-file: sbom.cdx.json

      # Step 3: Scan SBOM for vulnerabilities
      - name: Scan SBOM
        uses: anchore/scan-action@v4
        with:
          sbom: sbom.cdx.json
          fail-build: true
          severity-cutoff: critical

      # Step 4: Sign the image (keyless)
      - name: Install Cosign
        uses: sigstore/cosign-installer@v3

      - name: Sign Image
        run: cosign sign --yes ${{ env.IMAGE }}@${{ steps.build.outputs.digest }}

      # Step 5: Attest the SBOM
      - name: Attest SBOM
        run: |
          cosign attest --yes \
            --predicate sbom.cdx.json \
            --type cyclonedx \
            ${{ env.IMAGE }}@${{ steps.build.outputs.digest }}

  # Step 6: Generate SLSA provenance
  provenance:
    needs: build-sign-attest
    permissions:
      actions: read
      id-token: write
      packages: write
    uses: slsa-framework/slsa-github-generator/.github/workflows/generator_container_slsa3.yml@v2.0.0
    with:
      image: ghcr.io/${{ github.repository }}
      digest: ${{ needs.build-sign-attest.outputs.digest }}
      registry-username: ${{ github.actor }}
    secrets:
      registry-password: ${{ secrets.GITHUB_TOKEN }}
```

## 🎯 Best Practices — Supply Chain Security

| Practice | Description |
|----------|-------------|
| **Generate SBOMs for every release** | Treat SBOMs like release notes — always create and store them |
| **Sign all container images** | Use Cosign keyless signing in CI/CD — it's free and easy |
| **Achieve SLSA Level 3** | Use GitHub's SLSA generator for automatic provenance |
| **Enforce signatures in Kubernetes** | Use Kyverno or Sigstore Policy Controller to block unsigned images |
| **Pin dependencies** | Use lock files (package-lock.json, go.sum, requirements.txt) |
| **Use digest references** | Reference images by digest, not tag (`myapp@sha256:abc...`) |
| **Scan SBOMs regularly** | New vulnerabilities are discovered daily — rescan your SBOMs |
| **Monitor the Rekor log** | Watch for unauthorized signing events |
| **Enable Dependabot/Renovate** | Auto-update dependencies to patch vulnerabilities |
| **Review third-party actions** | Pin GitHub Actions to specific commit SHAs |

```
┌─────────────────────────────────────────────────────────────────┐
│                    SUPPLY CHAIN SECURITY                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │

│    → Tools: Cosign (sign/verify), Fulcio (certs), Rekor (log)  │

│    → Use: Prove who built it and that it wasn't tampered with   │
│                                                                 │

│    → Security rating for your build process (Levels 0-3)        │
│    → Tools: slsa-github-generator, slsa-verifier               │
│    → Key concept: Provenance (signed build metadata)            │
│    → Use: Prove the build process wasn't compromised            │
│                                                                 │
│  Together they answer:                                          │
│    SBOM    → "What components are in this software?"            │
│    Cosign  → "Who built it? Was it modified?"                   │
│    SLSA    → "Can I trust the build process?"                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```
