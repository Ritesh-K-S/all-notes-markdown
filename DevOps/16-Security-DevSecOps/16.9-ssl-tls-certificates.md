## 🤔 The Big Picture — Why Do We Need SSL/TLS?

Imagine you're sending a **postcard** vs a **sealed letter**:

```
Without SSL/TLS (HTTP):
┌──────────┐                                          ┌──────────┐
│  Browser │ ─── "My password is hunter2" ────────→  │  Server  │
│          │          ↑                                │          │
└──────────┘          │                                └──────────┘
                      │
               🕵️ Attacker
               "I can read
                everything!"

With SSL/TLS (HTTPS):
┌──────────┐                                          ┌──────────┐
│  Browser │ ─── "x8#kQ!m2$vL9@..." ─────────────→  │  Server  │
│          │          ↑                                │          │
└──────────┘          │                                └──────────┘
                      │
               🕵️ Attacker
               "I can't read
                any of this! 😡"
```

### SSL vs TLS — What's the Difference?

| | SSL | TLS |
|---|-----|-----|
| **Full Name** | Secure Sockets Layer | Transport Layer Security |
| **Created by** | Netscape (1995) | IETF (1999) |
| **Current Status** | ❌ DEPRECATED (insecure) | ✅ ACTIVE |
| **Latest Version** | SSL 3.0 (1996) | TLS 1.3 (2018) |
| **Should You Use It?** | ❌ NO! Never! | ✅ Yes! TLS 1.2 or 1.3 |

> **Fun Fact:** Everyone says "SSL certificate" but they actually mean "TLS certificate." SSL is dead! We just can't stop calling it SSL because the name stuck. It's like still calling a "USB flash drive" a "thumb drive" even though it has nothing to do with thumbs.

## 📜 What Exactly Is a Certificate?

A TLS certificate is a **digital document** that contains:

```
TLS Certificate Contents:
┌──────────────────────────────────────────────────────────┐
│                    TLS Certificate                        │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  Subject: CN=www.example.com         ← Who this cert    │
│                                        is for           │
│                                                          │
│  Issuer: CN=Let's Encrypt Authority  ← Who issued it    │
│          O=Let's Encrypt               (Certificate     │
│                                         Authority)      │
│                                                          │
│  Valid From: 2024-01-15               ← When it starts  │
│  Valid To:   2024-04-15               ← When it expires │
│                                        (90 days for     │
│                                         Let's Encrypt)  │
│                                                          │
│  Public Key: MIIBIjANBgkqhki...      ← Used for        │
│              AQAB                       encryption       │
│                                                          │
│  SANs (Subject Alternative Names):   ← Other domains    │
│    - www.example.com                   this cert covers  │
│    - example.com                                         │
│    - api.example.com                                     │
│                                                          │
│  Signature: SHA256withRSA             ← Proves it's     │
│             30:82:01:0a:02:82...        authentic        │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### How TLS Works — The Handshake (Simplified)

When your browser connects to `https://example.com`, here's what happens:

```
TLS Handshake (simplified):

Browser                                          Server
   │                                                │
   │  1. "Hello! I support TLS 1.3,               │
   │      here are my cipher suites"               │
   │ ──────────────────────────────────────────→    │
   │                                                │
   │  2. "Hello! Let's use TLS 1.3 with           │
   │      AES-256-GCM. Here's my certificate"      │
   │ ←──────────────────────────────────────────    │
   │                                                │
   │  3. Browser verifies the certificate:          │
   │     ✅ Is it signed by a trusted CA?           │
   │     ✅ Is the domain name correct?             │
   │     ✅ Is it not expired?                      │
   │     ✅ Is it not revoked?                      │
   │                                                │
   │  4. Both sides generate a shared              │
   │     session key using key exchange             │
   │ ──────────── Key Exchange ────────────→       │
   │                                                │
   │  5. 🔒 All communication is now               │
   │     encrypted with the session key!            │
   │ ←════════ Encrypted Data ═══════════→         │
   │                                                │
```

### Types of Certificates

| Type | Validation | Time to Get | Cost | Use Case |
|------|-----------|-------------|------|----------|
| **DV (Domain Validation)** | Proves you own the domain | Minutes | Free (Let's Encrypt) | Personal sites, APIs, most services |
| **OV (Organization Validation)** | Proves domain + organization identity | Days | $50-200/year | Business websites |
| **EV (Extended Validation)** | Proves domain + thorough org verification | Weeks | $100-500/year | Banks, e-commerce (less common now) |
| **Wildcard** | Covers `*.example.com` | Minutes-Days | Free-$$$ | Multiple subdomains |
| **SAN/Multi-Domain** | Multiple different domains in one cert | Minutes-Days | Free-$$$ | Multiple sites on one server |
| **Self-Signed** | Not verified by anyone | Instant | Free | Development/testing ONLY |

## 🔐 Certificate Authority (CA) — The Trust System

### How Does Trust Work?

Your browser trusts a certificate because it was **signed by a trusted Certificate Authority (CA)**. But how does the browser trust the CA?

```
Chain of Trust:

┌─────────────────────────┐
│     Root CA             │  ← Pre-installed in your browser/OS
│     (DigiCert,          │     (There are ~150 trusted Root CAs)
│      Let's Encrypt,     │
│      GlobalSign)        │
└───────────┬─────────────┘
            │ Signs
            ▼
┌─────────────────────────┐
│  Intermediate CA        │  ← Signed by Root CA
│  (Let's Encrypt R3,     │     (adds a layer of security)
│   DigiCert SHA2)        │
└───────────┬─────────────┘
            │ Signs
            ▼
┌─────────────────────────┐
│  Your Certificate       │  ← Signed by Intermediate CA
│  (www.example.com)      │     (the cert on your server)
└─────────────────────────┘

Browser validates: Your Cert → Intermediate CA → Root CA → "I trust this!" ✅
```

### The Certificate Files You'll Work With

```
File Types You'll Encounter:

┌──────────────────────────────────────────────────────────────────┐
│  File Extension    │ What It Contains              │ Format      │
├──────────────────────────────────────────────────────────────────┤
│  .pem              │ Certificate or key (Base64)   │ PEM (text)  │
│  .crt / .cer       │ Certificate only              │ PEM or DER  │
│  .key              │ Private key only              │ PEM         │
│  .csr              │ Certificate Signing Request   │ PEM         │
│  .p12 / .pfx       │ Cert + key bundled together   │ PKCS#12     │
│  .ca-bundle        │ Intermediate + Root CA certs  │ PEM         │
└──────────────────────────────────────────────────────────────────┘
```

... (many more lines of Base64 encoded data) ...

# Private key file (privkey.pem or private.key)

## 🆓 Let's Encrypt — Free Certificates for Everyone

### What is Let's Encrypt?

Let's Encrypt is a **free, automated, and open Certificate Authority** run by the nonprofit Internet Security Research Group (ISRG). It has **revolutionized** web security by making HTTPS free and easy.

```
Before Let's Encrypt (pre-2016):                After Let's Encrypt:
┌─────────────────────────────┐                 ┌─────────────────────────────┐
│ Pay $50-200/year per cert   │                 │ FREE! $0 forever            │
│ Manual renewal every year   │                 │ Auto-renewal every 90 days  │
│ Wait days for validation    │                 │ Get a cert in seconds       │
│ Complex manual process      │                 │ Fully automated             │
│ Only big companies use HTTPS│                 │ Everyone uses HTTPS         │
└─────────────────────────────┘                 └─────────────────────────────┘

Result: HTTPS adoption went from ~40% to ~95% of web traffic! 🎉
```

### Key Facts About Let's Encrypt

| Feature | Details |
|---------|---------|
| **Cost** | Completely free |
| **Certificate Type** | DV (Domain Validation) only |
| **Validity** | 90 days (auto-renew!) |
| **Rate Limits** | 50 certs per domain per week |
| **Wildcard** | ✅ Supported (requires DNS challenge) |
| **Protocol** | ACME (Automated Certificate Management Environment) |
| **Root CA** | ISRG Root X1 (trusted by all major browsers) |

### How Let's Encrypt Works — The ACME Protocol

ACME (Automated Certificate Management Environment) is the **protocol** that automates the entire certificate lifecycle:

```
ACME Workflow:
┌─────────────┐        ┌─────────────────┐        ┌──────────────┐
│  Your Server│        │  ACME Client    │        │ Let's Encrypt│
│             │        │  (Certbot,      │        │  CA Server   │
│             │        │   cert-manager) │        │              │
└──────┬──────┘        └────────┬────────┘        └──────┬───────┘
       │                       │                         │
       │                       │  1. "I need a cert      │
       │                       │     for example.com"    │
       │                       │ ───────────────────────→│
       │                       │                         │
       │                       │  2. "Prove you own      │
       │                       │     example.com.        │
       │                       │     Put this token at   │
       │                       │     /.well-known/acme-  │
       │                       │     challenge/xyz"      │
       │                       │ ←───────────────────────│
       │                       │                         │
       │  3. Place the token   │                         │
       │ ←─────────────────────│                         │
       │                       │                         │
       │                       │  4. "Token is ready,    │
       │                       │     please verify"      │
       │                       │ ───────────────────────→│
       │                       │                         │
       │  5. Let's Encrypt     │                         │
       │     checks the token  │ ←──── HTTP request ─────│
       │ ─── returns token ───→│                         │
       │                       │                         │
       │                       │  6. "Verified! Here's   │
       │                       │     your certificate"   │
       │                       │ ←───────────────────────│
       │                       │                         │
       │  7. Install the cert  │                         │
       │ ←─────────────────────│                         │
       │                       │                         │
       │  🔒 HTTPS is now      │                         │
       │     working!          │                         │
```

### Challenge Types — How Do You Prove Domain Ownership?

| Challenge | How It Works | When to Use |
|-----------|-------------|-------------|
| **HTTP-01** | Place a file at `http://yourdomain/.well-known/acme-challenge/token` | Standard web servers, most common |
| **DNS-01** | Create a DNS TXT record `_acme-challenge.yourdomain` | Wildcard certs, servers behind firewalls, no HTTP access |
| **TLS-ALPN-01** | Respond on port 443 with a self-signed cert containing the token | When port 80 is blocked |

### Certbot — The Official Let's Encrypt Client

Certbot is the **most popular ACME client** for getting Let's Encrypt certificates.

#### Installation

```bash
# Ubuntu/Debian
sudo apt update
sudo apt install certbot

# With Nginx plugin
sudo apt install certbot python3-certbot-nginx

# With Apache plugin
sudo apt install certbot python3-certbot-apache

# macOS
brew install certbot

# Using Docker
docker run --rm -v /etc/letsencrypt:/etc/letsencrypt \
  -v /var/www/certbot:/var/www/certbot \
  certbot/certbot certonly --help

# Using snap (recommended by Let's Encrypt)
sudo snap install --classic certbot
sudo ln -s /snap/bin/certbot /usr/bin/certbot
```

#### Getting Your First Certificate

```bash
# Method 1: Automatic Nginx configuration (easiest!)
sudo certbot --nginx -d example.com -d www.example.com
# Certbot will:
# 1. Get the certificate
# 2. Automatically configure Nginx to use it
# 3. Set up auto-renewal

# Method 2: Automatic Apache configuration
sudo certbot --apache -d example.com -d www.example.com

# Method 3: Standalone (Certbot runs its own temporary web server)
# Use this when you DON'T have a web server running yet
sudo certbot certonly --standalone -d example.com -d www.example.com

# Method 4: Webroot (use an existing web server's document root)
sudo certbot certonly --webroot -w /var/www/html -d example.com -d www.example.com

# Method 5: DNS challenge (for wildcard certificates)
sudo certbot certonly --manual --preferred-challenges dns -d "*.example.com" -d example.com
# You'll be prompted to create DNS TXT records

# Method 6: DNS challenge with automated DNS provider
sudo certbot certonly --dns-cloudflare \
  --dns-cloudflare-credentials /etc/letsencrypt/cloudflare.ini \
  -d "*.example.com" -d example.com
```

#### Where Are the Certificate Files?

```bash
# After getting a cert, files are stored here:
/etc/letsencrypt/live/example.com/
├── cert.pem       # Your certificate
├── chain.pem      # Intermediate CA certificate
├── fullchain.pem  # cert.pem + chain.pem combined (USE THIS!)
├── privkey.pem    # Your private key (KEEP SECRET!)
└── README

# In your Nginx config, use:
ssl_certificate     /etc/letsencrypt/live/example.com/fullchain.pem;
ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

# In your Apache config, use:
SSLCertificateFile    /etc/letsencrypt/live/example.com/fullchain.pem
SSLCertificateKeyFile /etc/letsencrypt/live/example.com/privkey.pem
```

#### Certificate Renewal

```bash
# Test renewal (dry run — doesn't actually renew)
sudo certbot renew --dry-run

# Actually renew all certificates
sudo certbot renew

# Force renewal of a specific certificate
sudo certbot renew --cert-name example.com --force-renewal

# Certbot auto-renewal is already set up!
# Check the systemd timer:
systemctl list-timers | grep certbot

# Or check the cron job:
cat /etc/cron.d/certbot
# Output: 0 */12 * * * root certbot renew --quiet

# Add a hook to reload Nginx after renewal
sudo certbot renew --deploy-hook "systemctl reload nginx"
```

#### Certbot with Nginx — Complete Example

```bash
# Step 1: Install everything
sudo apt update
sudo apt install nginx certbot python3-certbot-nginx

# Step 2: Create a basic Nginx config
sudo tee /etc/nginx/sites-available/example.com << 'EOF'
server {
    listen 80;
    server_name example.com www.example.com;
    root /var/www/example.com;

    location / {
        try_files $uri $uri/ =404;
    }
}
EOF

# Step 3: Enable the site
sudo ln -s /etc/nginx/sites-available/example.com /etc/nginx/sites-enabled/
sudo nginx -t
sudo systemctl reload nginx

# Step 4: Get the certificate (Certbot modifies Nginx config automatically!)
sudo certbot --nginx -d example.com -d www.example.com

# Step 5: That's it! Certbot has:
# ✅ Obtained the certificate
# ✅ Modified your Nginx config to use HTTPS
# ✅ Added HTTP→HTTPS redirect
# ✅ Set up auto-renewal
```

After Certbot runs, your Nginx config looks like this:

```nginx
# /etc/nginx/sites-available/example.com (auto-modified by Certbot)
server {
    server_name example.com www.example.com;
    root /var/www/example.com;

    location / {
        try_files $uri $uri/ =404;
    }

    # Certbot added everything below this line:
    listen 443 ssl;
    ssl_certificate /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;
    include /etc/letsencrypt/options-ssl-nginx.conf;
    ssl_dhparam /etc/letsencrypt/ssl-dhparams.pem;
}

# Certbot also added this redirect block:
server {
    if ($host = www.example.com) {
        return 301 https://$host$request_uri;
    }
    if ($host = example.com) {
        return 301 https://$host$request_uri;
    }

    listen 80;
    server_name example.com www.example.com;
    return 404;
}
```

## 🎯 cert-manager — Automated Certificates for Kubernetes

### What is cert-manager?

cert-manager is a **Kubernetes add-on** that **automatically provisions, manages, and renews** TLS certificates. It's the **standard way** to handle certificates in Kubernetes.

Think of it as: **"A robot that gets, installs, and renews all your certificates in Kubernetes — automatically."**

```
Without cert-manager:
┌───────────────┐     ┌─────────────────┐     ┌────────────────┐
│ Manually get  │ ──→ │ Create K8s      │ ──→ │ Remember to    │
│ certificates  │     │ Secrets by hand │     │ renew before   │
│ from CA       │     │                 │     │ expiry (oops!) │
└───────────────┘     └─────────────────┘     └────────────────┘

With cert-manager:
┌───────────────────────────────────────────────────────────────┐
│  cert-manager does EVERYTHING automatically:                  │
│                                                               │
│  ✅ Requests certificates from Let's Encrypt (or other CAs)  │
│  ✅ Proves domain ownership (HTTP or DNS challenge)           │
│  ✅ Creates Kubernetes Secrets with the cert + key            │
│  ✅ Auto-renews before expiry                                 │
│  ✅ Works with Ingress resources seamlessly                   │
│                                                               │
│  You just say "I need a cert for example.com" and it         │
│  handles the rest! 🎉                                         │
└───────────────────────────────────────────────────────────────┘
```

### Key Concepts in cert-manager

```
cert-manager Resources:

┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  Issuer / ClusterIssuer                                      │
│  ──────────────────────                                      │
│  "WHERE to get certificates from"                            │
│  • Issuer = works in ONE namespace                           │
│  • ClusterIssuer = works in ALL namespaces                   │
│  • Examples: Let's Encrypt, Vault, self-signed               │
│                                                              │
│  Certificate                                                 │
│  ───────────                                                 │
│  "WHAT certificate to get"                                   │
│  • Domain names (SANs)                                       │
│  • Which Issuer to use                                       │
│  • Which Secret to store it in                               │
│                                                              │
│  Secret (auto-created)                                       │
│  ─────────────────────                                       │
│  "WHERE the cert is stored in K8s"                           │
│  • tls.crt = certificate + chain                             │
│  • tls.key = private key                                     │
│  • ca.crt = CA certificate (optional)                        │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### Step 2: Request a Certificate

There are **two ways** to request certificates:

#### Way 1: Using a Certificate Resource (Explicit)

```yaml
# certificate.yaml
apiVersion: cert-manager.io/v1
kind: Certificate
metadata:
  name: example-com-tls
  namespace: default
spec:
  # The Secret where cert-manager will store the certificate
  secretName: example-com-tls-secret

  # How long the certificate should be valid (default: 2160h = 90 days)
  duration: 2160h

  # When to start renewing (default: 360h = 15 days before expiry)
  renewBefore: 360h

  # The domain names for this certificate
  dnsNames:
    - example.com
    - www.example.com
    - api.example.com

  # Which Issuer to use
  issuerRef:
    name: letsencrypt-prod
    kind: ClusterIssuer
```

```bash
# Apply it
kubectl apply -f certificate.yaml

# Watch cert-manager work its magic
kubectl get certificates
# NAME               READY   SECRET                    AGE
# example-com-tls    True    example-com-tls-secret    60s

# Check the certificate details
kubectl describe certificate example-com-tls

# Check the auto-created secret
kubectl get secret example-com-tls-secret
# NAME                      TYPE                DATA   AGE
# example-com-tls-secret    kubernetes.io/tls   2      60s

# View the certificate data
kubectl get secret example-com-tls-secret -o jsonpath='{.data.tls\.crt}' | base64 -d | openssl x509 -text -noout
```

#### Way 2: Using Ingress Annotations (Automatic — Most Common!)

This is the **easiest way** — just add annotations to your Ingress, and cert-manager handles everything:

```yaml
# ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-app-ingress
  annotations:
    # This annotation tells cert-manager to auto-create a Certificate!
    cert-manager.io/cluster-issuer: "letsencrypt-prod"
spec:
  ingressClassName: nginx
  tls:
    - hosts:
        - myapp.example.com
      secretName: myapp-tls-secret    # cert-manager creates this automatically!
  rules:
    - host: myapp.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: my-app-service
                port:
                  number: 80
```

```bash
# Apply it
kubectl apply -f ingress.yaml

# cert-manager automatically:
# 1. Sees the annotation on the Ingress
# 2. Creates a Certificate resource
# 3. Requests a cert from Let's Encrypt
# 4. Completes the HTTP-01 challenge
# 5. Stores the cert in the Secret
# 6. Your Ingress now has HTTPS! 🎉

# Check the certificate
kubectl get certificates
# NAME               READY   SECRET              AGE
# myapp-tls-secret   True    myapp-tls-secret    45s
```

### Troubleshooting cert-manager

```bash
# Check certificate status
kubectl get certificates -A
kubectl describe certificate <name>

# Check certificate requests
kubectl get certificaterequests -A
kubectl describe certificaterequest <name>

# Check challenges (if cert is stuck)
kubectl get challenges -A
kubectl describe challenge <name>

# Check orders
kubectl get orders -A
kubectl describe order <name>

# Check cert-manager logs
kubectl logs -n cert-manager deployment/cert-manager -f

# Common issues and fixes:

# Issue: Certificate stuck at "Issuing"
# → Check: kubectl describe challenge <name>
# → Common fix: Ensure port 80 is reachable from the internet (for HTTP-01)

# Issue: "dns-01 self check failed"
# → Check: DNS TXT record was created correctly
# → Common fix: Wait for DNS propagation (can take 1-5 minutes)

# Issue: Rate limit exceeded
# → Fix: Use staging issuer for testing first!
# → Let's Encrypt limit: 50 certs per domain per week

# Issue: Certificate not renewing
# → Check: kubectl describe certificate <name> — look at "Renewal Time"
# → cert-manager renews at 2/3 of the certificate's lifetime by default

# Manually trigger a renewal
kubectl cert-manager renew <certificate-name>
# (requires cert-manager kubectl plugin)
```

## 🔧 Useful OpenSSL Commands — The Certificate Swiss Army Knife

```bash
# ═══════════════════════════════════════════════════════════════
# VIEWING CERTIFICATES
# ═══════════════════════════════════════════════════════════════

# View a certificate's details
openssl x509 -in cert.pem -text -noout

# View certificate expiry date
openssl x509 -in cert.pem -enddate -noout

# View certificate subject and issuer
openssl x509 -in cert.pem -subject -issuer -noout

# View SANs (Subject Alternative Names)
openssl x509 -in cert.pem -text -noout | grep -A1 "Subject Alternative Name"

# Check a remote server's certificate
openssl s_client -connect example.com:443 -servername example.com </dev/null 2>/dev/null | openssl x509 -text -noout

# Check certificate expiry of a remote server
openssl s_client -connect example.com:443 -servername example.com </dev/null 2>/dev/null | openssl x509 -enddate -noout

# Check the full certificate chain
openssl s_client -connect example.com:443 -servername example.com -showcerts </dev/null

# ═══════════════════════════════════════════════════════════════
# GENERATING KEYS AND CSRS
# ═══════════════════════════════════════════════════════════════

# Generate a private key (RSA 2048-bit)
openssl genrsa -out private.key 2048

# Generate a private key (RSA 4096-bit — more secure)
openssl genrsa -out private.key 4096

# Generate a private key (ECDSA — modern, faster, smaller)
openssl ecparam -genkey -name prime256v1 -out private.key

# Generate a CSR (Certificate Signing Request)
openssl req -new -key private.key -out request.csr -subj "/CN=example.com"

# Generate a self-signed certificate (for testing only!)
openssl req -x509 -newkey rsa:4096 -keyout private.key -out cert.pem -days 365 -nodes -subj "/CN=localhost"

# ═══════════════════════════════════════════════════════════════
# VERIFYING AND TESTING
# ═══════════════════════════════════════════════════════════════

# Verify a certificate matches a private key
openssl x509 -in cert.pem -noout -modulus | md5sum
openssl rsa -in private.key -noout -modulus | md5sum
# If both md5sums match → they belong together ✅

# Verify a certificate chain
openssl verify -CAfile ca-bundle.pem cert.pem

# Test SSL/TLS connection
openssl s_client -connect example.com:443

# ═══════════════════════════════════════════════════════════════
# CONVERTING FORMATS
# ═══════════════════════════════════════════════════════════════

# PEM to DER
openssl x509 -in cert.pem -outform DER -out cert.der

# DER to PEM
openssl x509 -in cert.der -inform DER -outform PEM -out cert.pem

# PEM to PKCS#12 (combine cert + key into one file)
openssl pkcs12 -export -out cert.p12 -inkey private.key -in cert.pem -certfile ca-bundle.pem

# PKCS#12 to PEM
openssl pkcs12 -in cert.p12 -out combined.pem -nodes
```

## 🛡️ TLS Best Practices

### Nginx — Secure TLS Configuration

```nginx
# /etc/nginx/conf.d/ssl.conf — Secure defaults
server {
    listen 443 ssl http2;
    server_name example.com;

    # Certificate files
    ssl_certificate     /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

    # Modern TLS configuration
    ssl_protocols TLSv1.2 TLSv1.3;          # Only TLS 1.2 and 1.3
    ssl_prefer_server_ciphers on;
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384;

    # OCSP Stapling (faster certificate verification)
    ssl_stapling on;
    ssl_stapling_verify on;
    ssl_trusted_certificate /etc/letsencrypt/live/example.com/chain.pem;
    resolver 1.1.1.1 8.8.8.8 valid=300s;
    resolver_timeout 5s;

    # Security Headers
    add_header Strict-Transport-Security "max-age=63072000; includeSubDomains; preload" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-Frame-Options "SAMEORIGIN" always;
    add_header X-XSS-Protection "1; mode=block" always;

    # Session settings
    ssl_session_timeout 1d;
    ssl_session_cache shared:SSL:50m;
    ssl_session_tickets off;

    # DH parameters (generate with: openssl dhparam -out /etc/nginx/dhparam.pem 2048)
    ssl_dhparam /etc/nginx/dhparam.pem;
}

# Redirect HTTP to HTTPS
server {
    listen 80;
    server_name example.com;
    return 301 https://$host$request_uri;
}
```

### HSTS — HTTP Strict Transport Security

```
What is HSTS?
HSTS tells browsers: "ALWAYS use HTTPS for this domain. Never try HTTP."

Without HSTS:
  User types: example.com
  Browser tries: http://example.com → redirect → https://example.com
  Attacker can intercept the HTTP request! 🚨

With HSTS:
  User types: example.com
  Browser knows: "I must use HTTPS" → goes directly to https://example.com
  No HTTP request = no chance to intercept! ✅
```

```nginx
# Add this header (already included in the config above)
add_header Strict-Transport-Security "max-age=63072000; includeSubDomains; preload" always;
# max-age=63072000 = remember for 2 years
# includeSubDomains = apply to all subdomains too
# preload = submit to browser preload list (hardcoded HTTPS in Chrome/Firefox)
```
