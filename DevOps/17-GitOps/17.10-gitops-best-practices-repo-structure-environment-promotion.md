# 17.10 — GitOps Best Practices — Repo Structure, Environment Promotion

> **One Line Answer:** GitOps best practices are the **proven patterns and guidelines** for organizing your Git repositories, structuring your manifests, and promoting changes safely across environments (dev → staging → production) — all using **Git as the single source of truth.**

---

## 🤔 Why Do Best Practices Matter in GitOps?

Let's be honest — you can set up ArgoCD or Flux in an afternoon. But **scaling GitOps** to handle multiple teams, dozens of microservices, and several environments? That's where things get messy FAST if you don't follow good practices.

Here's what happens when you DON'T follow best practices:

```
Without Best Practices:
├── All manifests dumped in one repo → nobody knows what's where
├── Copy-paste YAML between environments → configs drift apart
├── No clear promotion strategy → "just merge to main and pray" 🙏
├── Secrets committed in plain text → security nightmare 😱
├── 50 teams editing the same repo → merge conflicts everywhere
└── Result: GitOps becomes GitChaos 💥
```

And here's what happens when you DO:

```
With Best Practices:
├── Clear repo structure → everyone knows where things live
├── DRY manifests (Helm/Kustomize) → one change, all environments updated
├── Structured promotion → dev → staging → prod with proper gates
├── Secrets handled securely → Sealed Secrets, SOPS, External Secrets
├── Team autonomy → each team owns their repo/directory
└── Result: GitOps becomes a superpower 🚀
```

## 📂 Repository Structure — The Foundation of GitOps

The **single most important decision** you'll make in GitOps is how you organize your Git repositories. Get this wrong, and everything else becomes painful.

There are **three main approaches:**

### Approach 1: Monorepo (Single Repository)

Everything lives in **one repository** — all applications, all environments, all configs.

```
gitops-repo/
├── apps/
│   ├── frontend/
│   │   ├── base/
│   │   │   ├── deployment.yaml
│   │   │   ├── service.yaml
│   │   │   └── kustomization.yaml
│   │   └── overlays/
│   │       ├── dev/
│   │       │   ├── kustomization.yaml
│   │       │   └── patch-replicas.yaml
│   │       ├── staging/
│   │       │   ├── kustomization.yaml
│   │       │   └── patch-replicas.yaml
│   │       └── prod/
│   │           ├── kustomization.yaml
│   │           └── patch-replicas.yaml
│   ├── backend-api/
│   │   ├── base/
│   │   └── overlays/
│   │       ├── dev/
│   │       ├── staging/
│   │       └── prod/
│   └── payment-service/
│       ├── base/
│       └── overlays/
│           ├── dev/
│           ├── staging/
│           └── prod/
├── infrastructure/
│   ├── namespaces/
│   ├── ingress/
│   ├── cert-manager/
│   └── monitoring/
└── README.md
```

**When to use Monorepo:**

| Pros | Cons |
|------|------|
| Simple to start with | Gets messy with many teams/apps |
| One place to see everything | Access control is hard (everyone can see/edit everything) |
| Easy to set up CI/CD for | Merge conflicts increase with team size |
| Good for small-medium teams (< 10 teams) | Blast radius is high — one bad commit can affect everything |

---

### Approach 2: Repo-Per-Team / Repo-Per-App

Each application (or each team) gets its **own repository** for its Kubernetes manifests.

```
Organization's GitHub:
├── team-frontend/
│   └── k8s-manifests/
│       ├── base/
│       │   ├── deployment.yaml
│       │   ├── service.yaml
│       │   └── kustomization.yaml
│       └── overlays/
│           ├── dev/
│           ├── staging/
│           └── prod/
│
├── team-backend/
│   └── k8s-manifests/
│       ├── base/
│       └── overlays/
│           ├── dev/
│           ├── staging/
│           └── prod/
│
├── team-payments/
│   └── k8s-manifests/
│       ├── base/
│       └── overlays/
│           ├── dev/
│           ├── staging/
│           └── prod/
│
└── platform-infra/          ← Platform team manages cluster-wide stuff
    ├── namespaces/
    ├── rbac/
    ├── ingress-controller/
    ├── cert-manager/
    └── monitoring/
```

**When to use Repo-Per-Team/App:**

| Pros | Cons |
|------|------|
| Team autonomy — each team controls their own repo | Harder to get a "big picture" view |
| Fine-grained access control via Git permissions | More repos to manage |
| Small blast radius — one team's mistake won't affect others | Cross-cutting changes (like updating base images) require changes in many repos |
| Scales well for large organizations (10+ teams) | Need ArgoCD ApplicationSets or Flux to manage many sources |

---

### Approach 3: Repo-Per-Environment (Less Common)

Each **environment** gets its own repo.

```
Organization's GitHub:
├── gitops-dev/
│   ├── frontend/
│   ├── backend/
│   └── payments/
│
├── gitops-staging/
│   ├── frontend/
│   ├── backend/
│   └── payments/
│
└── gitops-prod/
    ├── frontend/
    ├── backend/
    └── payments/
```

| Pros | Cons |
|------|------|
| Very strict access control per environment | Lots of duplication across repos |
| Clear separation — you can lock down prod completely | Keeping repos in sync is painful |
| Good for heavily regulated industries | Promotion between environments is harder to automate |

---

### 🏆 Recommended: The Hybrid Approach

Most mature organizations use a **hybrid** — separate repos for apps vs infrastructure, with environment overlays inside each.

```
Organization's GitHub:

📁 app-configs/                    ← Application team repo (one per team or one shared)
│   ├── frontend/
│   │   ├── base/
│   │   └── overlays/
│   │       ├── dev/
│   │       ├── staging/
│   │       └── prod/
│   ├── backend-api/
│   │   ├── base/
│   │   └── overlays/
│   │       ├── dev/
│   │       ├── staging/
│   │       └── prod/
│   └── ...
│
📁 platform-configs/               ← Platform/SRE team repo
│   ├── cluster-addons/
│   │   ├── ingress-nginx/
│   │   ├── cert-manager/
│   │   ├── external-dns/
│   │   └── monitoring-stack/
│   ├── namespaces/
│   ├── rbac/
│   └── network-policies/
│
📁 frontend-app/                   ← Application source code repo (separate from configs!)
│   ├── src/
│   ├── Dockerfile
│   ├── package.json
│   └── ...
```

**Key Insight:** Always separate **application source code** from **deployment manifests (GitOps configs)**. They have different lifecycles, different owners, and different change frequencies.

## 📁 The Golden Rules of Repo Structure

Here are the **battle-tested rules** that the best GitOps teams follow:

### Rule 1: Separate App Code from Config Repos

```
❌ BAD: App code + K8s manifests in the SAME repo
my-app/
├── src/
├── Dockerfile
├── k8s/                    ← Don't put manifests here for GitOps
│   ├── deployment.yaml
│   └── service.yaml
└── package.json

✅ GOOD: App code and K8s manifests in SEPARATE repos
my-app/                     ← Source code repo
├── src/
├── Dockerfile
└── package.json

my-app-config/              ← GitOps config repo
├── base/
│   ├── deployment.yaml
│   ├── service.yaml
│   └── kustomization.yaml
└── overlays/
    ├── dev/
    ├── staging/
    └── prod/
```

**Why?** Because:
- App code changes don't always need a deployment change
- Different people might have access to configs vs code
- CI pipeline (build app) is different from CD pipeline (deploy via GitOps)
- You don't want a code commit to trigger a deployment accidentally

---

### Rule 2: Use Base + Overlays Pattern (DRY Principle)

Never copy-paste manifests between environments. Use **Kustomize overlays** or **Helm values files** to keep things DRY.

#### With Kustomize:

```yaml
# base/deployment.yaml — The common baseline
apiVersion: apps/v1
kind: Deployment
metadata:
  name: frontend
spec:
  replicas: 1                    # Default — overridden per environment
  selector:
    matchLabels:
      app: frontend
  template:
    metadata:
      labels:
        app: frontend
    spec:
      containers:
        - name: frontend
          image: myregistry/frontend:latest   # Will be overridden
          ports:
            - containerPort: 3000
          resources:
            requests:
              cpu: 100m
              memory: 128Mi
            limits:
              cpu: 200m
              memory: 256Mi
```

```yaml
# base/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - deployment.yaml
  - service.yaml
```

```yaml
# overlays/dev/kustomization.yaml — Dev-specific overrides
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - ../../base
namePrefix: dev-
namespace: dev
patches:
  - path: patch-replicas.yaml
images:
  - name: myregistry/frontend
    newTag: dev-abc1234            # Dev image tag
```

```yaml
# overlays/dev/patch-replicas.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: frontend
spec:
  replicas: 1                     # Dev only needs 1 replica
```

```yaml
# overlays/prod/kustomization.yaml — Prod-specific overrides
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - ../../base
namePrefix: prod-
namespace: production
patches:
  - path: patch-replicas.yaml
  - path: patch-resources.yaml
images:
  - name: myregistry/frontend
    newTag: v1.5.2                 # Prod uses a stable release tag
```

```yaml
# overlays/prod/patch-replicas.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: frontend
spec:
  replicas: 5                     # Prod needs 5 replicas for traffic
```

```yaml
# overlays/prod/patch-resources.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: frontend
spec:
  template:
    spec:
      containers:
        - name: frontend
          resources:
            requests:
              cpu: 500m           # More resources in prod
              memory: 512Mi
            limits:
              cpu: "1"
              memory: 1Gi
```

Now your base stays **clean and common**, and each environment only specifies what's **different**. If you need to add a new container port, you change it **once** in the base, and all environments get it automatically.

#### With Helm (values per environment):

```
my-app-chart/
├── Chart.yaml
├── templates/
│   ├── deployment.yaml
│   ├── service.yaml
│   └── ingress.yaml
├── values.yaml                  ← Default values
├── values-dev.yaml              ← Dev overrides
├── values-staging.yaml          ← Staging overrides
└── values-prod.yaml             ← Prod overrides
```

```yaml
# values.yaml — Defaults
replicaCount: 1
image:
  repository: myregistry/frontend
  tag: latest
resources:
  requests:
    cpu: 100m
    memory: 128Mi
  limits:
    cpu: 200m
    memory: 256Mi
ingress:
  enabled: false
```

```yaml
# values-dev.yaml — Only override what's different
replicaCount: 1
image:
  tag: dev-abc1234
ingress:
  enabled: true
  host: frontend.dev.example.com
```

```yaml
# values-prod.yaml — Production overrides
replicaCount: 5
image:
  tag: v1.5.2
resources:
  requests:
    cpu: 500m
    memory: 512Mi
  limits:
    cpu: "1"
    memory: 1Gi
ingress:
  enabled: true
  host: frontend.example.com
  tls: true
```

### Rule 3: Use Consistent Naming Conventions

Be **boring and predictable** with names. Future-you will thank present-you.

```
✅ Good naming:
apps/
├── user-service/
│   ├── base/
│   └── overlays/
│       ├── dev/
│       ├── staging/
│       └── prod/
├── order-service/
├── payment-service/
└── notification-service/

❌ Bad naming:
apps/
├── UserSVC/                     ← Inconsistent casing
├── orders/                      ← Doesn't match the service name
├── pay-svc-v2-final-FINAL/      ← Please don't
└── notif/                       ← Abbreviations confuse people
```

**Naming rules to follow:**
- Use **lowercase with hyphens**: `user-service`, not `UserService` or `user_service`
- Match names to your **Kubernetes namespace or service name**
- Environment names should be consistent: always `dev/staging/prod` — not `development/stg/production` in some places
- Use the **same directory structure** for every app — don't let teams invent their own

### Rule 4: Have a Clear README and CODEOWNERS

Every GitOps repo should have:

```markdown
# 📁 GitOps Config Repository

## Structure
- `apps/` — Application deployment manifests
- `infrastructure/` — Cluster-wide resources
- `overlays/` — Environment-specific configs

## Environments
| Env | Cluster | Namespace | Branch/Path |
|-----|---------|-----------|-------------|
| Dev | dev-cluster | dev | overlays/dev |
| Staging | staging-cluster | staging | overlays/staging |
| Prod | prod-cluster | production | overlays/prod |

## How to Promote
1. Make changes in `overlays/dev/` → commit → push
2. After testing, update `overlays/staging/` → commit → push
3. After approval, update `overlays/prod/` → commit via PR → requires 2 approvals

## Contact
- Platform Team: @platform-team
- On-call: #oncall-channel
```

And a `CODEOWNERS` file:

```
# CODEOWNERS — who must approve changes
# Prod changes require platform team approval
apps/*/overlays/prod/    @platform-team @sre-team
infrastructure/          @platform-team

# Dev/staging — app teams can manage themselves
apps/frontend/           @frontend-team
apps/backend-api/        @backend-team
apps/payment-service/    @payments-team
```

This way, **Git pull request reviews become your change management process**. A commit to prod requires approval from the right people — enforced by Git, not by humans remembering to ask.

## 🔄 Environment Promotion — Moving Changes Safely Through Environments

Environment promotion is **how you move a change from dev → staging → production** safely. This is the most critical part of GitOps to get right.

### The Promotion Flow

```
┌──────────┐     ┌───────────┐     ┌──────────────┐
│   DEV    │────▶│  STAGING  │────▶│  PRODUCTION  │
│          │     │           │     │              │
│ Auto-sync│     │ Auto-sync │     │ Manual sync  │
│ No gates │     │ Smoke     │     │ PR approval  │
│          │     │ tests     │     │ + manual     │
│          │     │           │     │ approval     │
└──────────┘     └───────────┘     └──────────────┘
     │                │                    │
     │                │                    │
  Cluster: dev     Cluster: staging    Cluster: prod
  Namespace: dev   Namespace: staging  Namespace: production
```

---

### Strategy 1: Directory-Based Promotion (Most Common)

Each environment has its own **directory** (overlay). Promotion = updating the correct directory.

```
apps/frontend/
├── base/
│   ├── deployment.yaml
│   └── kustomization.yaml
└── overlays/
    ├── dev/                    ← Step 1: Change lands here first
    │   └── kustomization.yaml
    ├── staging/                ← Step 2: Promote here after dev testing
    │   └── kustomization.yaml
    └── prod/                   ← Step 3: Promote here after staging approval
        └── kustomization.yaml
```

**How it works:**

```bash
# Step 1: CI pipeline builds the app and pushes a new image
# Image: myregistry/frontend:v1.5.2

# Step 2: CI pipeline (or human) updates the dev overlay
# In overlays/dev/kustomization.yaml, update the image tag:
images:
  - name: myregistry/frontend
    newTag: v1.5.2

# Step 3: Commit and push — ArgoCD auto-syncs to dev cluster
git add overlays/dev/
git commit -m "feat(frontend): deploy v1.5.2 to dev"
git push

# Step 4: After testing in dev, promote to staging
# Update overlays/staging/kustomization.yaml with the same tag
git add overlays/staging/
git commit -m "promote(frontend): v1.5.2 to staging"
git push

# Step 5: After staging approval, promote to prod (via Pull Request!)
# Update overlays/prod/kustomization.yaml with the same tag
# Create a PR → requires review from @platform-team
git checkout -b promote/frontend-v1.5.2
# ... make changes to overlays/prod/ ...
git add overlays/prod/
git commit -m "promote(frontend): v1.5.2 to production"
git push -u origin promote/frontend-v1.5.2
# Open PR → Get approvals → Merge → ArgoCD syncs
```

---

### Strategy 2: Branch-Based Promotion

Each environment maps to a **Git branch**.

```
Branches:
├── dev       ← Dev environment reads from this branch
├── staging   ← Staging environment reads from this branch
└── main      ← Production environment reads from this branch
```

**How it works:**

```bash
# Step 1: Developer pushes changes to 'dev' branch
git checkout dev
# ... make changes ...
git commit -m "deploy frontend v1.5.2"
git push origin dev
# ArgoCD watches 'dev' branch → auto-syncs to dev cluster

# Step 2: Promote to staging — merge dev into staging
git checkout staging
git merge dev
git push origin staging
# ArgoCD watches 'staging' branch → auto-syncs to staging cluster

# Step 3: Promote to prod — merge staging into main (via PR!)
# Create PR: staging → main
# Requires approvals
# Merge → ArgoCD watches 'main' → syncs to prod cluster
```

**Branch-Based Pros & Cons:**

| Pros | Cons |
|------|------|
| Familiar Git workflow (branching + merging) | Merge conflicts can get messy |
| PR reviews control what goes to each env | Hard to have different configs per env (all in one branch) |
| Easy to understand | Doesn't scale well for many apps |
| | Cherry-picking between branches is error-prone |

> **⚠️ Note:** The **GitOps community generally recommends directory-based promotion over branch-based**. Branch-based was popular early on but has proven harder to manage at scale. ArgoCD and Flux both recommend directory/path-based approaches.

### Strategy 3: PR-Based Promotion with Automation

This is the **gold standard** — combining directory-based structure with automated PRs.

```
Automated Promotion Pipeline:

┌─────────────────────────────────────────────────────────┐
│  1. CI Pipeline builds image → pushes myregistry/       │
│     frontend:v1.5.2                                     │
│                                                         │
│  2. CI Pipeline auto-updates dev overlay:               │
│     overlays/dev/kustomization.yaml → tag: v1.5.2       │
│     Commits directly to main branch                     │
│                                                         │
│  3. ArgoCD auto-syncs to dev cluster                    │
│                                                         │
│  4. After dev tests pass → CI creates a PR:             │
│     "Promote frontend v1.5.2 to staging"                │
│     Updates overlays/staging/kustomization.yaml          │
│                                                         │
│  5. Team lead reviews PR → merges                       │
│     ArgoCD auto-syncs to staging                        │
│                                                         │
│  6. After staging validation → CI creates another PR:   │
│     "Promote frontend v1.5.2 to production"             │
│     Updates overlays/prod/kustomization.yaml             │
│                                                         │
│  7. SRE/Platform team reviews → 2 approvals required    │
│     Merge → ArgoCD syncs to production                  │
└─────────────────────────────────────────────────────────┘
```

Here's a **GitHub Actions workflow** that automates the promotion:

```yaml
# .github/workflows/promote.yaml
name: Promote to Next Environment

on:
  workflow_dispatch:
    inputs:
      app:
        description: 'Application to promote'
        required: true
        type: string
      image_tag:
        description: 'Image tag to promote'
        required: true
        type: string
      target_env:
        description: 'Target environment'
        required: true
        type: choice
        options:
          - staging
          - prod

jobs:
  promote:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout config repo
        uses: actions/checkout@v4

      - name: Setup Kustomize
        uses: imranismail/setup-kustomize@v2

      - name: Update image tag for target environment
        run: |
          cd apps/${{ inputs.app }}/overlays/${{ inputs.target_env }}
          kustomize edit set image myregistry/${{ inputs.app }}:${{ inputs.image_tag }}

      - name: Create Pull Request
        uses: peter-evans/create-pull-request@v5
        with:
          token: ${{ secrets.GITHUB_TOKEN }}
          commit-message: "promote(${{ inputs.app }}): ${{ inputs.image_tag }} to ${{ inputs.target_env }}"
          title: "🚀 Promote ${{ inputs.app }} ${{ inputs.image_tag }} → ${{ inputs.target_env }}"
          body: |
            ## Promotion Request
            - **App:** ${{ inputs.app }}
            - **Image Tag:** ${{ inputs.image_tag }}
            - **Target:** ${{ inputs.target_env }}
            
            ### Checklist
            - [ ] Dev testing passed
            - [ ] Staging smoke tests passed (if promoting to prod)
            - [ ] Rollback plan reviewed
          branch: promote/${{ inputs.app }}-${{ inputs.image_tag }}-${{ inputs.target_env }}
          labels: |
            promotion
            ${{ inputs.target_env }}
```

### Strategy 4: ArgoCD ApplicationSet for Multi-Environment

ArgoCD's `ApplicationSet` controller can automatically generate ArgoCD `Application` resources for every environment from a **single template**.

```yaml
# applicationset.yaml — One definition, multiple environments
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: frontend
  namespace: argocd
spec:
  generators:
    - list:
        elements:
          - env: dev
            cluster: https://dev-cluster-api:6443
            namespace: dev
            autoSync: true
          - env: staging
            cluster: https://staging-cluster-api:6443
            namespace: staging
            autoSync: true
          - env: prod
            cluster: https://prod-cluster-api:6443
            namespace: production
            autoSync: false          # Prod requires manual sync!
  template:
    metadata:
      name: frontend-{{env}}
    spec:
      project: default
      source:
        repoURL: https://github.com/myorg/gitops-configs.git
        targetRevision: main
        path: apps/frontend/overlays/{{env}}
      destination:
        server: '{{cluster}}'
        namespace: '{{namespace}}'
      syncPolicy:
        automated:
          prune: '{{autoSync}}'
          selfHeal: '{{autoSync}}'
```

This creates **three ArgoCD Applications** (`frontend-dev`, `frontend-staging`, `frontend-prod`) from one template. Dev and staging auto-sync; prod requires manual sync.

## 🔐 Secrets Management in GitOps

One of the **trickiest parts** of GitOps: how do you store secrets in Git without exposing them?

**Rule: NEVER commit plain-text secrets to Git. EVER.** 🚫

Here are the approved approaches:

### Option 1: Sealed Secrets (Bitnami)

```bash
# Install kubeseal CLI
brew install kubeseal

# Create a regular Kubernetes secret
kubectl create secret generic db-creds \
  --from-literal=username=admin \
  --from-literal=password=supersecret \
  --dry-run=client -o yaml > secret.yaml

# Seal it (encrypt with the cluster's public key)
kubeseal --format yaml < secret.yaml > sealed-secret.yaml

# Now sealed-secret.yaml is SAFE to commit to Git!
# Only the cluster's controller can decrypt it
```

```yaml
# sealed-secret.yaml — This is safe to store in Git
apiVersion: bitnami.com/v1alpha1
kind: SealedSecret
metadata:
  name: db-creds
  namespace: default
spec:
  encryptedData:
    username: AgBy3i...encrypted...data...==
    password: AgCtr8...encrypted...data...==
```

### Option 2: SOPS (Mozilla) + Age/KMS

```bash
# Install sops
brew install sops

# Create a .sops.yaml config in your repo root
cat > .sops.yaml << 'EOF'
creation_rules:
  - path_regex: .*secrets.*\.yaml$
    age: age1ql3z7hjy54pw3hyww5ayyfg7zqgvc7w3j2elw8zmrj2kg5sfn9aq2d7x5a
EOF

# Encrypt a secret file
sops --encrypt secret.yaml > secret.enc.yaml

# Commit the encrypted file to Git
# ArgoCD/Flux has SOPS plugins that decrypt at sync time
```

### Option 3: External Secrets Operator

```yaml
# external-secret.yaml — References secrets from AWS Secrets Manager, Vault, etc.
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: db-creds
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: aws-secrets-manager
    kind: ClusterSecretStore
  target:
    name: db-creds
  data:
    - secretKey: username
      remoteRef:
        key: /myapp/prod/db-username
    - secretKey: password
      remoteRef:
        key: /myapp/prod/db-password
```

This is safe to commit — it only contains **references** to secrets, not the actual values. The External Secrets Operator fetches the real values from your secrets manager at runtime.

## ✅ GitOps Best Practices Checklist

Here's the complete list of best practices, organized by category:

### Repository Structure

| # | Best Practice | Why |
|---|--------------|-----|
| 1 | Separate app source code repos from GitOps config repos | Different lifecycles, different access needs |
| 2 | Use a consistent directory structure across all apps | Predictability reduces cognitive load |
| 3 | Use Kustomize base/overlays or Helm values per environment | DRY — don't repeat yourself |
| 4 | Add README and CODEOWNERS to every repo | Documentation and access control |
| 5 | One repo per team (or small group) for configs | Team autonomy + manageable scope |
| 6 | Keep infrastructure configs separate from app configs | Different change frequency and ownership |

### Branching & Commits

| # | Best Practice | Why |
|---|--------------|-----|
| 7 | Use `main` as the source of truth (single branch) | Avoid branch sprawl |
| 8 | Prefer directory-based promotion over branch-based | Simpler, less merge conflicts |
| 9 | Write meaningful commit messages | Git history IS your audit log |
| 10 | Use conventional commits: `feat(app):`, `fix(app):`, `promote(app):` | Easy to parse and automate |
| 11 | Squash merge promotion PRs | Clean history, easy rollbacks |

**Good commit message examples:**

```
feat(frontend): deploy v1.5.2 to dev
promote(frontend): v1.5.2 → staging (tested in dev for 2 days)
promote(frontend): v1.5.2 → production (approved by @sre-lead)
fix(backend-api): rollback to v2.3.1 — OOM errors in prod
chore(infra): update ingress-nginx to 4.8.3
```

### Environment Promotion

| # | Best Practice | Why |
|---|--------------|-----|
| 12 | Always promote through ALL environments: dev → staging → prod | Catch issues before they hit customers |
| 13 | Auto-sync dev & staging, manual sync for prod | Fast iteration in lower envs, safety in prod |
| 14 | Use PR reviews as the gate for production promotion | Human review + Git audit trail |
| 15 | Require at least 2 approvals for prod changes | Reduce risk of single-person mistakes |
| 16 | Run automated tests in staging before promoting to prod | Confidence in the change |
| 17 | Use the SAME image tag across all environments | What you tested is what you deploy |

### Security

| # | Best Practice | Why |
|---|--------------|-----|
| 18 | Never commit plain-text secrets to Git | Security 101 |
| 19 | Use Sealed Secrets, SOPS, or External Secrets Operator | Encrypt at rest in Git |
| 20 | Enable branch protection rules on the main branch | Prevent direct pushes |
| 21 | Use CODEOWNERS to enforce review requirements | Right people approve right changes |
| 22 | Sign commits with GPG keys | Verify who made the change |
| 23 | Scan manifests for security issues (Kubescape, Datree, Polaris) | Catch misconfigs before they're applied |

### Sync & Drift Management

| # | Best Practice | Why |
|---|--------------|-----|
| 24 | Enable self-healing (ArgoCD) / auto-remediation | Prevent manual kubectl changes from causing drift |
| 25 | Enable prune (delete resources removed from Git) | Git is the source of truth — if it's not in Git, it shouldn't exist |
| 26 | Set sync windows for prod (only deploy during business hours) | Reduce risk |
| 27 | Configure health checks and sync waves | Ensure resources are created in the right order |
| 28 | Alert on sync failures | Know immediately when something goes wrong |

## 🔄 Rollback Strategy

One of the **biggest advantages** of GitOps: rollback = `git revert`.

```bash
# Something broke in production after the last promotion?

# Option 1: Git Revert (Recommended — preserves history)
git revert HEAD
git push
# ArgoCD sees the revert commit → syncs → rolls back the deployment

# Option 2: Revert to a specific commit
git log --oneline
# abc1234 promote(frontend): v1.5.2 → production  ← THIS broke things
# def5678 promote(frontend): v1.5.1 → production  ← THIS was working

git revert abc1234
git push
# Now production is back to v1.5.1

# Option 3: ArgoCD Manual Rollback (faster for emergencies)
argocd app rollback frontend-prod
# This rolls back the ArgoCD app to the previous sync
# ⚠️ But this creates DRIFT from Git — fix it in Git afterward!
```

**Rollback Best Practices:**
- Always prefer `git revert` over ArgoCD manual rollback
- If you do an emergency ArgoCD rollback, immediately commit the revert to Git
- Keep at least 10 revisions in ArgoCD history for quick rollbacks
- Test your rollback procedure BEFORE you need it

## ⚠️ Common Mistakes to Avoid

| Mistake | Why It's Bad | What to Do Instead |
|---------|-------------|-------------------|
| Storing app code and K8s manifests in the same repo | Tight coupling, accidental deploys | Separate repos |
| Using `kubectl apply` alongside GitOps | Creates drift — cluster doesn't match Git | ALL changes through Git only |
| Not using PR reviews for prod changes | No audit trail, risky | Enforce branch protection + CODEOWNERS |
| Copy-pasting YAML between environments | Configs drift apart over time | Use Kustomize overlays or Helm values |
| Committing plain-text secrets | Security breach waiting to happen | Use Sealed Secrets, SOPS, or External Secrets |
| Having no rollback plan | Panic when things break | Practice `git revert`, keep ArgoCD history |
| Skipping environments (dev → prod) | No safety net | Always go through ALL environments |
| Too many manual steps in promotion | Human error, slow process | Automate promotion with CI pipelines |
| Not monitoring sync status | Failures go unnoticed | Set up alerts for sync failures |
| Letting everyone push to main | Chaos, no accountability | Branch protection, require PRs |

## 🎯 Summary — The 10 Commandments of GitOps

```
 1. Git is the SINGLE source of truth — if it's not in Git, it doesn't exist
 2. Separate app source code from deployment configs
 3. Use base + overlays (Kustomize) or values files (Helm) — stay DRY
 4. Use directory-based promotion (not branch-based)
 5. Auto-sync dev/staging, require PRs for production
 6. NEVER commit plain-text secrets
 7. Use CODEOWNERS to enforce review requirements
 8. Write meaningful commit messages — Git history is your audit log
 9. Rollback = git revert — practice it before you need it
10. Monitor sync status and alert on failures
```
