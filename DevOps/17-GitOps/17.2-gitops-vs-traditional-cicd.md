## Quick Recap — What is CI/CD?

Before we compare, let's make sure we're on the same page about what CI/CD means:

| Term | Full Form | What It Does |
|------|-----------|-------------|
| **CI** | Continuous Integration | Automatically build and test code every time someone pushes a commit |
| **CD** | Continuous Delivery | Automatically prepare the code so it's **ready** to deploy at any time |
| **CD** | Continuous Deployment | Automatically **deploy** to production without manual approval |

**A typical CI/CD pipeline looks like this:**

```
Code Push → Build → Test → Package → Deploy
```

The big question is: **HOW does that last step ("Deploy") happen?**

- In **traditional CI/CD**, the pipeline **pushes** the deployment to the cluster. (**Push Model**)
- In **GitOps**, the cluster **pulls** the deployment from Git. (**Pull Model**)

This difference might seem small, but it changes **everything**.

---

## Traditional CI/CD — The Push Model

### How It Works

In the push model, the CI/CD pipeline is responsible for deploying your application. When code is pushed to Git, the pipeline runs and at the end, it **pushes** the changes directly to the target environment (like a Kubernetes cluster).

```
┌──────────┐    push     ┌──────────┐    build    ┌──────────┐    push     ┌──────────────┐
│Developer │───────────▶│  Git     │───────────▶│ CI/CD    │───────────▶│  Kubernetes   │
│          │   code      │  Repo    │  trigger   │ Pipeline │  deploy    │  Cluster      │
└──────────┘             └──────────┘            └──────────┘            └──────────────┘
```

### The Flow — Step by Step

1. **Developer** pushes code to the Git repository (e.g., GitHub).
2. The CI/CD tool (e.g., Jenkins, GitHub Actions) **detects** the push.
3. The pipeline runs:
   - **Build** — Compile code, build Docker image.
   - **Test** — Run unit tests, integration tests.
   - **Package** — Push Docker image to a container registry.
   - **Deploy** — Run `kubectl apply` or `helm upgrade` to push changes to the cluster.
4. The pipeline **needs credentials** (kubeconfig, tokens) to access the cluster.
5. Deployment is done. ✅

### Example: Traditional CI/CD with GitHub Actions (Push Model)

```yaml
# .github/workflows/deploy.yml
name: Build and Deploy

on:
  push:
    branches: [main]

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest

    steps:
      # Step 1: Check out the code
      - name: Checkout code
        uses: actions/checkout@v4

      # Step 2: Build the Docker image
      - name: Build Docker image
        run: |
          docker build -t my-registry.com/my-app:${{ github.sha }} .

      # Step 3: Push the Docker image to a registry
      - name: Push Docker image
        run: |
          echo "${{ secrets.DOCKER_PASSWORD }}" | docker login my-registry.com -u "${{ secrets.DOCKER_USERNAME }}" --password-stdin
          docker push my-registry.com/my-app:${{ github.sha }}

      # Step 4: Deploy to Kubernetes (THE PUSH!)
      - name: Deploy to Kubernetes
        run: |
          # The pipeline needs cluster credentials!
          echo "${{ secrets.KUBECONFIG }}" > kubeconfig.yaml
          export KUBECONFIG=kubeconfig.yaml

          # Pipeline directly talks to the cluster
          kubectl set image deployment/my-app \
            my-app=my-registry.com/my-app:${{ github.sha }} \
            --namespace production
```

**Notice the key things here:**

- The pipeline has **direct access** to the Kubernetes cluster (via `KUBECONFIG` secret).
- The pipeline **pushes** the change directly using `kubectl set image`.
- The cluster **doesn't know about Git** — it just receives commands from the pipeline.
- If someone manually changes the cluster later, **nobody will know** — the pipeline only runs on code push.

## GitOps CI/CD — The Pull Model

### How It Works

In the pull model, the CI/CD pipeline **never touches the cluster**. Instead:

1. The CI pipeline builds and pushes the Docker image.
2. The CI pipeline (or a human) updates the **config repo** with the new image tag.
3. A **GitOps agent** (ArgoCD/Flux) running **inside** the cluster detects the change in Git.
4. The agent **pulls** the new configuration and applies it to the cluster.

```
┌──────────┐    push     ┌──────────┐    build    ┌──────────┐   update   ┌──────────┐
│Developer │───────────▶│ App Repo │───────────▶│ CI       │──────────▶│ Config   │
│          │   code      │ (code)   │  trigger   │ Pipeline │  image    │ Repo     │
└──────────┘             └──────────┘            └──────────┘  tag      └────┬─────┘
                                                                             │
                                                                      watch  │  pull
                                                                             ▼
                                                                     ┌──────────────┐
                                                                     │ GitOps Agent │
                                                                     │ (in cluster) │
                                                                     │    ↓ apply   │
                                                                     │ Kubernetes   │
                                                                     │ Cluster      │
                                                                     └──────────────┘
```

### The Flow — Step by Step

1. **Developer** pushes code to the **application repo**.
2. The CI pipeline builds and tests the code.
3. The CI pipeline pushes a new Docker image to the registry (e.g., `my-app:abc123`).
4. The CI pipeline (or an automated bot) updates the **config repo** — changing the image tag in the Kubernetes manifest.
5. The **GitOps agent** (ArgoCD/Flux) detects the change in the config repo.
6. The agent **pulls** the new manifest and applies it to the cluster.
7. Kubernetes performs a rolling update.

### Example: GitOps with GitHub Actions + ArgoCD (Pull Model)

**Step 1: CI Pipeline (GitHub Actions) — Builds and updates the config repo:**

```yaml
# In the APPLICATION repo: .github/workflows/ci.yml
name: CI Pipeline

on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      # Step 1: Build the Docker image
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Build and push Docker image
        run: |
          docker build -t my-registry.com/my-app:${{ github.sha }} .
          echo "${{ secrets.DOCKER_PASSWORD }}" | docker login my-registry.com -u "${{ secrets.DOCKER_USERNAME }}" --password-stdin
          docker push my-registry.com/my-app:${{ github.sha }}

      # Step 2: Update the CONFIG repo with the new image tag
      # (This is the ONLY thing the CI pipeline does for deployment)
      - name: Update config repo
        run: |
          git clone https://x-access-token:${{ secrets.CONFIG_REPO_TOKEN }}@github.com/my-org/k8s-configs.git
          cd k8s-configs

          # Update the image tag in the deployment manifest
          sed -i "s|image: my-registry.com/my-app:.*|image: my-registry.com/my-app:${{ github.sha }}|" \
            apps/my-app/deployment.yaml

          git config user.name "CI Bot"
          git config user.email "ci-bot@my-org.com"
          git add .
          git commit -m "Update my-app image to ${{ github.sha }}"
          git push
```

**Step 2: ArgoCD (running inside the cluster) does the rest:**

```
ArgoCD watches the config repo:
  "Oh! The image tag changed from abc123 to def456!"
  "Let me apply the new deployment.yaml to the cluster."
  "Rolling update in progress... Done! ✅"
```

**Notice the key differences:**

- The CI pipeline **NEVER touches the cluster**. No `kubectl`. No `KUBECONFIG` secret.
- The CI pipeline only **updates a file in Git**. That's it.
- ArgoCD, running **inside** the cluster, handles the actual deployment.
- The cluster is **self-aware** — it knows what Git says and constantly checks.

## Push vs Pull — Side-by-Side Comparison

| Aspect | Push Model (Traditional CI/CD) | Pull Model (GitOps) |
|--------|-------------------------------|---------------------|
| **Who deploys?** | CI/CD pipeline | GitOps agent inside the cluster |
| **Direction** | Pipeline pushes TO cluster | Agent pulls FROM Git |
| **Cluster credentials** | Stored in CI/CD as secrets | Not needed — agent runs inside cluster |
| **Source of truth** | CI/CD pipeline config (Jenkinsfile, workflow) | Git repository |
| **Drift detection** | ❌ None — once deployed, pipeline doesn't check again | ✅ Continuous — agent always compares Git vs cluster |
| **Self-healing** | ❌ No — manual changes persist | ✅ Yes — manual changes are reverted |
| **Security surface** | Large — CI system has cluster access | Small — only the agent (inside cluster) has access |
| **Rollback** | Re-run old pipeline or write rollback logic | `git revert` — simple and clean |
| **Audit trail** | Spread across CI logs and cluster | Everything in Git |
| **Works offline?** | No — needs CI/CD system running | Agent works independently once deployed |
| **Complexity** | Lower initial setup | Higher initial setup, lower ongoing maintenance |

## Deep Dive: The Push Model

### Advantages of Push Model

1. **Simpler to set up initially** — Just add a deploy step to your existing pipeline.
2. **Works with any environment** — Not just Kubernetes; works with VMs, bare metal, serverless.
3. **More flexible** — Can run imperative commands, database migrations, scripts.
4. **Familiar** — Most developers already know CI/CD pipelines.
5. **Single pipeline** — Build, test, and deploy all in one place.

### Disadvantages of Push Model

1. **Credential sprawl** — Every CI/CD system needs cluster credentials.

   ```
   🔴 Security Risk:
   - Jenkins has your kubeconfig
   - GitHub Actions has your kubeconfig
   - GitLab CI has your kubeconfig
   - Developer laptops might have your kubeconfig

   If ANY of these are compromised, your cluster is compromised.
   ```

2. **No drift detection** — If someone manually changes the cluster, the pipeline doesn't know.

   ```
   Example:
   1. Pipeline deploys version 2.0 with 3 replicas.
   2. Someone runs: kubectl scale deployment my-app --replicas=10
   3. Now the cluster has 10 replicas.
   4. Pipeline doesn't care — it only runs on code push.
   5. Nobody knows why there are 10 replicas. 🤷
   ```

3. **Pipeline is the bottleneck** — If your CI/CD system goes down, you can't deploy.

4. **Audit trail is scattered** — Deployment history is in CI logs, not in Git.

5. **Rollback is complex** — You need to find the old pipeline run or write custom rollback logic.

## Deep Dive: The Pull Model

### Advantages of Pull Model

1. **Superior security** — No cluster credentials outside the cluster.

   ```
   ✅ Security Model:
   - CI/CD only pushes Docker images and updates Git.
   - CI/CD has ZERO access to the cluster.
   - Only the GitOps agent (running INSIDE the cluster) can make changes.
   - Reduced attack surface.
   ```

2. **Continuous drift detection and correction** — The agent always watches.

3. **Git IS the audit trail** — Complete, immutable, reviewable history.

4. **Easy rollback** — Just revert the commit.

5. **Cluster independence** — The cluster manages itself; it doesn't depend on an external CI system.

6. **Multi-cluster management** — Point multiple agents at the same Git repo to deploy consistently.

### Disadvantages of Pull Model

1. **Kubernetes-centric** — Most tools (ArgoCD, Flux) only work with Kubernetes.

2. **Higher initial setup** — You need to install and configure the GitOps agent.

3. **Two repos to manage** — Application code and config are usually in separate repos.

4. **Secrets are tricky** — You can't (and shouldn't) store plain-text secrets in Git. You need tools like SOPS, Sealed Secrets, or Vault.

5. **Not great for imperative tasks** — Database migrations, one-time scripts, and other non-declarative operations don't fit neatly into GitOps.

6. **Learning curve** — Teams need to learn new tools (ArgoCD/Flux) and new workflows.

## Security Differences — Why Pull Wins

This is one of the **biggest reasons** people switch to GitOps. Let's compare the security posture:

### Push Model Security

```
                   ┌─────────────────┐
                   │  CI/CD System   │
                   │  (Jenkins,      │
                   │   GitHub Actions)│
                   │                 │
                   │  Has:           │
                   │  - KUBECONFIG   │ ← 🔴 If compromised, cluster is exposed
                   │  - API tokens   │
                   │  - Deploy keys  │
                   └────────┬────────┘
                            │
                   kubectl apply (over network)
                            │
                            ▼
                   ┌─────────────────┐
                   │  Kubernetes     │
                   │  Cluster        │
                   └─────────────────┘
```

**Attack surface:**
- CI/CD system is **outside** the cluster but has **full access** to it.
- If an attacker compromises your CI/CD (supply chain attack, stolen secrets), they can do **anything** in your cluster.
- Credentials travel **over the network** from CI to cluster.
- Multiple teams might have multiple CI pipelines, all with cluster access.

### Pull Model Security

```
                   ┌─────────────────┐
                   │  CI/CD System   │
                   │  (Jenkins,      │
                   │   GitHub Actions)│
                   │                 │
                   │  Has:           │
                   │  - Docker creds │ ← Only pushes images
                   │  - Git token    │ ← Only updates config repo
                   │  - NO cluster   │ ← ✅ No cluster access!
                   │    access       │
                   └─────────────────┘

                   ┌─────────────────────────────┐
                   │  Kubernetes Cluster          │
                   │                              │
                   │  ┌───────────────────────┐   │
                   │  │  GitOps Agent         │   │
                   │  │  (ArgoCD / Flux)      │   │
                   │  │                       │   │
                   │  │  Has:                 │   │
                   │  │  - Git read access    │   │ ← Only reads Git
                   │  │  - In-cluster access  │   │ ← Already inside!
                   │  └───────────────────────┘   │
                   └─────────────────────────────┘
```

**Attack surface:**
- CI/CD has **no cluster access** at all. Even if compromised, the attacker can't directly mess with the cluster.
- The GitOps agent only has **read access to Git** (pulling configs).
- The agent runs **inside** the cluster — it already has in-cluster permissions through Kubernetes RBAC.
- **Nothing crosses the network boundary** to deploy.

> **Summary:** Push model = "Here, CI system, have the keys to my cluster." Pull model = "My cluster will check Git itself, thanks."

## What Happens When Things Go Wrong?

Let's compare how each model handles failures:

### Scenario 1: Bad Deployment (App Crashes)

**Push Model:**
```
1. Pipeline deploys v2.0.
2. v2.0 crashes immediately.
3. Team scrambles to find the last working version.
4. Someone has to find the right pipeline run and re-trigger it.
   Or manually run: kubectl rollout undo deployment/my-app
5. Hope that the rollback doesn't break anything else.
6. Debug why it happened using CI logs + cluster logs.
```

**Pull Model (GitOps):**
```
1. Config repo is updated with v2.0 image tag.
2. ArgoCD deploys v2.0.
3. v2.0 crashes immediately.
4. Team runs: git revert HEAD (one command!)
5. ArgoCD detects the revert → deploys the previous version.
6. Everything is back to normal.
7. The entire incident is visible in Git history.
```

### Scenario 2: CI/CD System Goes Down

**Push Model:**
```
1. Jenkins/GitHub Actions goes down.
2. Nobody can deploy anything. ❌
3. If a critical fix is needed, someone has to manually kubectl apply. 😱
4. That manual change creates drift that the pipeline won't know about.
```

**Pull Model (GitOps):**
```
1. GitHub Actions goes down.
2. You can't build new images, but...
3. If you need to deploy an existing image, just update the config repo directly.
4. ArgoCD is running INSIDE the cluster — it doesn't care about your CI system.
5. Deployments continue working. ✅
```

### Scenario 3: Someone Makes a Manual Change

**Push Model:**
```
1. Someone runs: kubectl scale deployment my-app --replicas=10
2. The cluster now has 10 replicas.
3. Git still says 3 replicas.
4. Nobody knows about the change.
5. Next pipeline run might or might not reset it (depends on the pipeline).
6. Configuration drift accumulates over time. 💀
```

**Pull Model (GitOps):**
```
1. Someone runs: kubectl scale deployment my-app --replicas=10
2. The cluster now has 10 replicas.
3. ArgoCD detects: "Hey, Git says 3 but cluster has 10 — that's not right!"
4. ArgoCD automatically scales back to 3 replicas.
5. The manual change is reverted. ✅
6. (ArgoCD can also be configured to just alert instead of auto-fix)
```

## Drift — The Silent Killer

**Drift** happens when the actual state of your system doesn't match the intended state. It's one of the biggest operational problems in infrastructure management.

### How Drift Happens

```
Day 1: Deploy app with 3 replicas, 512MB memory.          ← Matches config
Day 5: DevOps engineer manually increases memory to 1GB.   ← Drift begins
Day 12: Someone adds an environment variable via kubectl.  ← More drift
Day 20: A hotfix is applied directly to a pod.             ← Even more drift
Day 30: Nobody knows what the actual config looks like.    ← Config is a mystery 💀
```

### Push Model vs Pull Model on Drift

| Aspect | Push Model | Pull Model (GitOps) |
|--------|-----------|---------------------|
| **Detection** | ❌ No built-in detection | ✅ Continuous detection |
| **Alerting** | ❌ No alerts | ✅ Alerts when drift is detected |
| **Correction** | ❌ Manual only | ✅ Automatic (self-healing) |
| **Prevention** | ❌ Must rely on RBAC and policies | ✅ Agent constantly corrects drift |

### ArgoCD Drift Detection in Action

When ArgoCD detects drift, it shows it clearly in its UI:

```
Application: my-app
Sync Status: OutOfSync ⚠️
Health Status: Healthy ✅

Diff:
  spec:
    replicas:
-     3          ← Git says 3
+     10         ← Cluster has 10
```

You can configure ArgoCD to:

1. **Just alert** — "Hey, there's drift. Want me to fix it?" (Manual sync)
2. **Auto-fix** — Automatically correct any drift. (Auto-sync enabled)

## Hybrid Approach — CI with Push + CD with Pull

In practice, most teams use a **hybrid approach**: 

- **CI (Build + Test) uses the Push model** — Pipeline pushes images and updates Git.
- **CD (Deploy) uses the Pull model** — GitOps agent pulls from Git and deploys.

This gives you the best of both worlds:

```
┌─────────────────────────────── PUSH (CI) ──────────────────────────────┐
│                                                                        │
│  Developer → Push Code → CI Pipeline → Build → Test → Push Image →    │
│  Update Config Repo                                                    │
│                                                                        │
└────────────────────────────────────────────────────────────────────────┘
                                    │
                            Config Repo Updated
                                    │
┌───────────────────────────── PULL (CD) ────────────────────────────────┐
│                                                                        │
│  GitOps Agent (ArgoCD/Flux) → Detect Change → Pull Config →           │
│  Apply to Cluster → Monitor & Self-Heal                                │
│                                                                        │
└────────────────────────────────────────────────────────────────────────┘
```

### This is the Recommended Approach!

| Phase | Model | Tool Examples |
|-------|-------|---------------|
| **Build & Test** | Push | GitHub Actions, Jenkins, GitLab CI |
| **Push Image** | Push | Docker push to ECR/GCR/DockerHub |
| **Update Config** | Push | CI bot commits new image tag to config repo |
| **Deploy to Cluster** | Pull | ArgoCD or Flux CD |
| **Monitor & Self-Heal** | Pull | ArgoCD or Flux CD |

## Real-World Pipeline Comparison

Let's compare a complete real-world pipeline in both models:

### Traditional Push Pipeline (Jenkins)

```groovy
// Jenkinsfile — Push Model
pipeline {
    agent any

    environment {
        REGISTRY = 'my-registry.com'
        IMAGE_NAME = 'my-app'
        KUBECONFIG = credentials('kubeconfig-prod')  // ← 🔴 Cluster creds in Jenkins!
    }

    stages {
        stage('Build') {
            steps {
                sh "docker build -t ${REGISTRY}/${IMAGE_NAME}:${BUILD_NUMBER} ."
            }
        }

        stage('Test') {
            steps {
                sh "docker run ${REGISTRY}/${IMAGE_NAME}:${BUILD_NUMBER} npm test"
            }
        }

        stage('Push Image') {
            steps {
                withCredentials([usernamePassword(credentialsId: 'docker-creds',
                    usernameVariable: 'USER', passwordVariable: 'PASS')]) {
                    sh "echo ${PASS} | docker login ${REGISTRY} -u ${USER} --password-stdin"
                    sh "docker push ${REGISTRY}/${IMAGE_NAME}:${BUILD_NUMBER}"
                }
            }
        }

        stage('Deploy') {
            steps {
                // 🔴 Pipeline DIRECTLY deploys to the cluster
                sh """
                    kubectl --kubeconfig=${KUBECONFIG} set image \
                        deployment/my-app \
                        my-app=${REGISTRY}/${IMAGE_NAME}:${BUILD_NUMBER} \
                        -n production
                """
            }
        }
    }
}
```

### GitOps Pull Pipeline (GitHub Actions + ArgoCD)

```yaml
# In the APPLICATION repo: .github/workflows/ci.yml
# GitOps Pull Model — CI only builds and updates config
name: CI Pipeline

on:
  push:
    branches: [main]

jobs:
  build-and-test:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Build Docker image
        run: docker build -t my-registry.com/my-app:${{ github.sha }} .

      - name: Run tests
        run: docker run my-registry.com/my-app:${{ github.sha }} npm test

      - name: Push Docker image
        run: |
          echo "${{ secrets.DOCKER_PASSWORD }}" | docker login my-registry.com -u "${{ secrets.DOCKER_USERNAME }}" --password-stdin
          docker push my-registry.com/my-app:${{ github.sha }}

      # ✅ No cluster credentials! Just update the config repo.
      - name: Update GitOps config repo
        run: |
          git clone https://x-access-token:${{ secrets.CONFIG_REPO_TOKEN }}@github.com/my-org/k8s-configs.git
          cd k8s-configs
          
          # Update the image tag
          sed -i "s|image: my-registry.com/my-app:.*|image: my-registry.com/my-app:${{ github.sha }}|" \
            apps/my-app/deployment.yaml
          
          git config user.name "CI Bot"
          git config user.email "ci@my-org.com"
          git add .
          git commit -m "chore: update my-app to ${{ github.sha }}"
          git push

# ✅ That's it! ArgoCD handles the rest automatically.
# No kubectl. No kubeconfig. No cluster access.
```

**Config repo (what ArgoCD watches):**

```yaml
# k8s-configs/apps/my-app/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
  namespace: production
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
      - name: my-app
        image: my-registry.com/my-app:abc123def   # ← CI bot updates this
        ports:
        - containerPort: 3000
        resources:
          requests:
            cpu: 100m
            memory: 128Mi
          limits:
            cpu: 500m
            memory: 256Mi
```

## When to Use What?

### Use Push Model When:

| Scenario | Why Push Works Better |
|----------|----------------------|
| Deploying to **non-Kubernetes** environments (VMs, bare metal) | Most GitOps tools are K8s-only |
| Running **imperative tasks** (DB migrations, seed scripts) | GitOps is declarative-only |
| **Small team, simple app** | GitOps setup might be overkill |
| **Prototyping or demos** | Quick and easy setup |
| **Serverless deployments** (Lambda, Cloud Functions) | GitOps tools don't manage serverless well |
| **Legacy systems** that can't be declaratively managed | Push is more flexible |

### Use Pull Model (GitOps) When:

| Scenario | Why GitOps Works Better |
|----------|------------------------|
| Running **Kubernetes** in production | GitOps was built for K8s |
| Managing **multiple clusters** | One Git repo, many clusters |
| Need **compliance and audit trails** | Git = complete audit log |
| Need **self-healing** and drift detection | Continuous reconciliation |
| **Security-conscious** environment | No cluster creds outside the cluster |
| **Multiple teams** deploying to the same cluster | PRs = natural access control |
| Managing **many microservices** | Declarative configs scale well |

### Use Hybrid When:

| Scenario | Why Hybrid Works Best |
|----------|----------------------|
| You want **automated builds** (CI) + **GitOps deployments** (CD) | Best of both worlds |
| You're **migrating from push to pull** gradually | Don't need to change everything at once |
| You need **CI for non-K8s tasks** (testing, scanning) + **GitOps for K8s** | Each tool does what it's best at |

> **Recommendation:** For most Kubernetes-based production environments, the **hybrid approach** (CI = Push, CD = Pull) is the sweet spot. ✅

## Common Misconceptions

### Misconception 1: "GitOps replaces CI/CD"

**Wrong!** GitOps replaces only the **CD (deployment)** part. You still need CI for building, testing, and pushing images.

```
Traditional: CI → CD (push)
GitOps:      CI → Update Git → CD (pull by agent)
```

---

### Misconception 2: "GitOps means everything must be in one Git repo"

**Wrong!** You can (and should) have multiple repos:
- One for app code
- One for Kubernetes configs
- Maybe separate ones per environment or team

---

### Misconception 3: "Push model is insecure — never use it"

**Not exactly.** Push model can be secured with:
- Short-lived tokens
- OIDC authentication
- Network policies
- Pipeline hardening

It's just that pull model is **more secure by design** — fewer credentials, smaller attack surface.

---

### Misconception 4: "GitOps means I can't use kubectl anymore"

**Not entirely.** You can still use `kubectl` for **debugging and reading** (get, describe, logs). You just shouldn't use it for **making changes** (apply, delete, scale). Changes should go through Git.

---

### Misconception 5: "GitOps only works with ArgoCD"

**Wrong!** ArgoCD is the most popular, but there are other options:
- **Flux CD** — CNCF graduated, very popular
- **Rancher Fleet** — For multi-cluster management
- **Jenkins X** — CI/CD + GitOps (less common now)

## Key Takeaways

1. **Push model:** CI/CD pipeline pushes changes TO the cluster. Simple but less secure.
2. **Pull model:** GitOps agent inside the cluster pulls changes FROM Git. More secure, with self-healing.
3. **The biggest difference is WHERE the cluster credentials live** — outside (push) or inside (pull).
4. **Drift detection is exclusive to the pull model** — push model has zero awareness of manual changes.
5. **Hybrid is the sweet spot** — Use CI (push) for building/testing, GitOps (pull) for deploying.
6. **GitOps doesn't replace CI/CD** — it replaces only the deployment (CD) part.
7. **Security matters** — Fewer credentials outside the cluster = smaller attack surface.
8. **Rollback in GitOps is just `git revert`** — clean, auditable, and simple.
