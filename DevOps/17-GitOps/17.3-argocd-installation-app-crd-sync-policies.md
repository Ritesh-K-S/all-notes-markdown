## What is ArgoCD?

**ArgoCD** is the most popular **GitOps continuous delivery tool** for Kubernetes. It's a **CNCF Graduated project** (meaning it's production-ready and widely adopted).

### In Simple Words

ArgoCD is a tool that:

1. **Watches** a Git repository for changes.
2. **Compares** what Git says (desired state) with what's actually in the cluster (actual state).
3. **Applies** the changes to make the cluster match Git.
4. **Continuously monitors** for drift and can auto-correct it.

Think of it as a **robot sitting inside your Kubernetes cluster**, constantly watching Git and saying:

> *"Does the cluster match Git? Yes? Great, I'll just keep watching. No? Let me fix that right now."*

### Key Facts About ArgoCD

| Fact | Detail |
|------|--------|
| **Created by** | Intuit (the company behind TurboTax) |
| **Now maintained by** | CNCF (Cloud Native Computing Foundation) |
| **Status** | CNCF Graduated (highest maturity level) |
| **Written in** | Go |
| **Supports** | Kubernetes manifests, Helm charts, Kustomize, Jsonnet |
| **Has a UI?** | Yes! A beautiful web-based dashboard |
| **Has a CLI?** | Yes! `argocd` command-line tool |
| **GitHub stars** | 18,000+ |

ArgoCD runs as a set of **Kubernetes controllers** inside your cluster. Let's understand each component:

### Component Breakdown

| Component | What It Does |
|-----------|-------------|
| **API Server** (`argocd-server`) | The brain of ArgoCD. Serves the web UI, provides REST and gRPC APIs, handles authentication and authorization. |
| **Application Controller** (`argocd-application-controller`) | The worker. It continuously watches Git repos, compares desired vs actual state, and syncs applications. This is the core of ArgoCD. |
| **Repo Server** (`argocd-repo-server`) | Clones Git repos and generates Kubernetes manifests from Helm charts, Kustomize overlays, plain YAML, or Jsonnet. |
| **Redis** (`argocd-redis`) | In-memory cache for improving performance. Stores application state and manifest cache. |
| **Dex** (`argocd-dex-server`) | Optional. Handles Single Sign-On (SSO) — lets you log in with GitHub, Google, LDAP, etc. |

## Installing ArgoCD

There are multiple ways to install ArgoCD. We'll cover the two most common:

### Method 1: Quick Install (Recommended for Learning)

This installs ArgoCD with all default settings. Perfect for learning and development.

```bash
# Step 1: Create a dedicated namespace for ArgoCD
kubectl create namespace argocd

# Step 2: Install ArgoCD using the official manifest
kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml
```

That's it! Two commands. Let's verify:

```bash
# Step 3: Check that all ArgoCD pods are running
kubectl get pods -n argocd

# Expected output (wait a minute for all pods to be Running):
# NAME                                               READY   STATUS    
# argocd-application-controller-0                    1/1     Running   
# argocd-dex-server-xxxxxxxxxx-xxxxx                 1/1     Running   
# argocd-redis-xxxxxxxxxx-xxxxx                      1/1     Running   
# argocd-repo-server-xxxxxxxxxx-xxxxx                1/1     Running   
# argocd-server-xxxxxxxxxx-xxxxx                     1/1     Running   
# argocd-notifications-controller-xxxxxxxxxx-xxxxx   1/1     Running   
```

```bash
# Check all services
kubectl get svc -n argocd

# Expected output:
# NAME                    TYPE        CLUSTER-IP     PORT(S)
# argocd-server           ClusterIP   10.96.xxx.xx   80/TCP,443/TCP
# argocd-repo-server      ClusterIP   10.96.xxx.xx   8081/TCP
# argocd-redis            ClusterIP   10.96.xxx.xx   6379/TCP
# argocd-dex-server       ClusterIP   10.96.xxx.xx   5556/TCP,5557/TCP
```

### Method 2: Install Using Helm (Recommended for Production)

Helm gives you more control over the configuration.

```bash
# Add the ArgoCD Helm repository
helm repo add argo https://argoproj.github.io/argo-helm
helm repo update

# Install ArgoCD with Helm
helm install argocd argo/argo-cd \
  --namespace argocd \
  --create-namespace \
  --set server.service.type=LoadBalancer   # Optional: expose via LoadBalancer

# Verify
kubectl get pods -n argocd
```

### What Gets Installed?

After installation, ArgoCD creates these resources in the `argocd` namespace:

```bash
# Check all resources
kubectl get all -n argocd

# You'll see:
# - Deployments (argocd-server, argocd-repo-server, argocd-redis, argocd-dex-server, argocd-notifications-controller)
# - StatefulSet (argocd-application-controller)
# - Services
# - Pods
# - ServiceAccounts
# - Roles and RoleBindings
# - ConfigMaps and Secrets
```

## Accessing the ArgoCD UI

ArgoCD has a beautiful web-based dashboard. Let's access it:

### Option 1: Port Forwarding (Easiest — Great for Local Clusters)

```bash
# Forward port 8080 on your machine to the ArgoCD server
kubectl port-forward svc/argocd-server -n argocd 8080:443

# Now open your browser and go to:
# https://localhost:8080
```

> **Note:** You'll get a certificate warning (self-signed cert). Just click "Advanced" → "Proceed" to continue.

### Option 2: Change Service Type to LoadBalancer

```bash
# Patch the ArgoCD server service to use LoadBalancer
kubectl patch svc argocd-server -n argocd -p '{"spec": {"type": "LoadBalancer"}}'

# Get the external IP
kubectl get svc argocd-server -n argocd

# Output:
# NAME            TYPE           CLUSTER-IP     EXTERNAL-IP      PORT(S)
# argocd-server   LoadBalancer   10.96.xx.xx    203.0.113.100    80:31xxx/TCP,443:32xxx/TCP
#
# Access it at: https://203.0.113.100
```

### Option 3: Use NodePort

```bash
# Patch to NodePort
kubectl patch svc argocd-server -n argocd -p '{"spec": {"type": "NodePort"}}'

# Get the NodePort
kubectl get svc argocd-server -n argocd

# Access at: https://<node-ip>:<node-port>
```

### Option 4: Use an Ingress (Production)

```yaml
# argocd-ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: argocd-ingress
  namespace: argocd
  annotations:
    nginx.ingress.kubernetes.io/force-ssl-redirect: "true"
    nginx.ingress.kubernetes.io/ssl-passthrough: "true"
spec:
  ingressClassName: nginx
  rules:
  - host: argocd.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: argocd-server
            port:
              number: 443
```

```bash
kubectl apply -f argocd-ingress.yaml
```

## Logging In — CLI and UI

### Getting the Initial Admin Password

ArgoCD creates a default admin user with an auto-generated password. Let's get it:

```bash
# Get the initial admin password
# (It's stored in a Kubernetes secret)
kubectl -n argocd get secret argocd-initial-admin-secret -o jsonpath="{.data.password}" | base64 -d

# Output will be something like: xYz123AbCdEfG
```

On **Windows PowerShell**:
```powershell
# Get and decode the password
$encoded = kubectl -n argocd get secret argocd-initial-admin-secret -o jsonpath="{.data.password}"
[System.Text.Encoding]::UTF8.GetString([System.Convert]::FromBase64String($encoded))
```

### Login via CLI

```bash
# Make sure port-forward is running (if using local cluster)
kubectl port-forward svc/argocd-server -n argocd 8080:443 &

# Login
argocd login localhost:8080

# It will ask:
# WARNING: server certificate had error: ... Proceed insecurely (y/n)? y
# Username: admin
# Password: <paste the password from above>

# Output: 'admin:login' logged in successfully
```

### Login via UI

1. Open `https://localhost:8080` in your browser.
2. **Username:** `admin`
3. **Password:** The password you got from the command above.

### Change the Admin Password (Recommended!)

```bash
# Change the password via CLI
argocd account update-password

# Current password: <initial password>
# New password: <your new password>
# Confirm new password: <your new password>
```


## Understanding the Application CRD

This is one of the **most important concepts** in ArgoCD.

### What is a CRD?

**CRD = Custom Resource Definition.** It extends Kubernetes with new resource types.

Just like Kubernetes has built-in resources like `Deployment`, `Service`, `Pod` — ArgoCD creates a **custom resource** called `Application`.

```
Built-in resources:          ArgoCD custom resources:
├── Deployment               ├── Application          ← This is the main one!
├── Service                  ├── AppProject
├── Pod                      └── ApplicationSet
├── ConfigMap
└── ...
```

### What is an ArgoCD Application?

An **Application** in ArgoCD is a Kubernetes custom resource that tells ArgoCD:

1. **WHERE** to find the desired state → Git repository URL, branch, path
2. **WHERE** to deploy it → Kubernetes cluster, namespace
3. **HOW** to sync it → Sync policy, options, automated/manual

Think of it like a **contract** between Git and your cluster:

> *"Dear ArgoCD, please take the Kubernetes manifests from THIS Git repo, at THIS path, and deploy them to THIS namespace in THIS cluster. And keep watching for changes."*

### The Simplest Application YAML

```yaml
apiVersion: argoproj.io/v1alpha1      # ArgoCD's API group
kind: Application                      # The custom resource type
metadata:
  name: my-app                         # Name of this ArgoCD application
  namespace: argocd                    # Must be in the argocd namespace
spec:
  project: default                     # ArgoCD project (more on this later)

  source:                              # WHERE to get the manifests (Git)
    repoURL: https://github.com/my-org/k8s-configs.git
    targetRevision: main               # Branch, tag, or commit hash
    path: apps/my-app                  # Folder path inside the repo

  destination:                         # WHERE to deploy (Kubernetes)
    server: https://kubernetes.default.svc   # The cluster (in-cluster)
    namespace: production              # Target namespace
```

That's it! This tells ArgoCD:

- **Source:** Get manifests from `github.com/my-org/k8s-configs`, branch `main`, folder `apps/my-app`.
- **Destination:** Deploy to the `production` namespace in the current cluster.

## Creating Your First ArgoCD Application

Let's create an actual application step by step.

### Step 1: Set Up a Sample Git Repository

For this example, we'll use a public sample repo. But first, let me show you what a config repo looks like:

**Create a simple repo structure:**
```
my-gitops-repo/
└── apps/
    └── nginx-demo/
        ├── deployment.yaml
        ├── service.yaml
        └── namespace.yaml
```

**`namespace.yaml`:**
```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: nginx-demo
```

**`deployment.yaml`:**
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
  namespace: nginx-demo
  labels:
    app: nginx
spec:
  replicas: 2
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.25
        ports:
        - containerPort: 80
        resources:
          requests:
            cpu: 50m
            memory: 64Mi
          limits:
            cpu: 100m
            memory: 128Mi
```

**`service.yaml`:**
```yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx
  namespace: nginx-demo
spec:
  selector:
    app: nginx
  ports:
  - port: 80
    targetPort: 80
  type: ClusterIP
```

Push this to your Git repository.

### Step 2: Create the ArgoCD Application

**Method A: Using the CLI**

```bash
argocd app create nginx-demo \
  --repo https://github.com/your-username/my-gitops-repo.git \
  --path apps/nginx-demo \
  --dest-server https://kubernetes.default.svc \
  --dest-namespace nginx-demo \
  --revision main \
  --project default

# Output: application 'nginx-demo' created
```

**Method B: Using a YAML file (Declarative — Recommended!)**

```yaml
# argocd-app-nginx-demo.yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: nginx-demo
  namespace: argocd
spec:
  project: default

  source:
    repoURL: https://github.com/your-username/my-gitops-repo.git
    targetRevision: main
    path: apps/nginx-demo

  destination:
    server: https://kubernetes.default.svc
    namespace: nginx-demo
```

```bash
# Apply the Application CRD
kubectl apply -f argocd-app-nginx-demo.yaml

# Output: application.argoproj.io/nginx-demo created
```

**Method C: Using the ArgoCD UI**

1. Open the ArgoCD dashboard.
2. Click **"+ NEW APP"**.
3. Fill in:
   - **Application Name:** `nginx-demo`
   - **Project:** `default`
   - **Sync Policy:** Manual (for now)
   - **Repository URL:** `https://github.com/your-username/my-gitops-repo.git`
   - **Revision:** `main`
   - **Path:** `apps/nginx-demo`
   - **Cluster URL:** `https://kubernetes.default.svc`
   - **Namespace:** `nginx-demo`
4. Click **"CREATE"**.

### Step 3: Check the Application Status

```bash
# List all ArgoCD applications
argocd app list

# Output:
# NAME        CLUSTER                         NAMESPACE   STATUS     HEALTH   SYNCPOLICY
# nginx-demo  https://kubernetes.default.svc  nginx-demo  OutOfSync  Missing  <none>
```

The status is `OutOfSync` because we haven't synced yet (we used manual sync policy).

### Step 4: Sync the Application

```bash
# Sync (deploy) the application
argocd app sync nginx-demo

# Output:
# TIMESTAMP                  GROUP        KIND         NAMESPACE    NAME    STATUS   HEALTH
# 2024-01-15T10:30:00+00:00             Namespace    nginx-demo   nginx   Synced
# 2024-01-15T10:30:01+00:00  apps        Deployment   nginx-demo   nginx   Synced   Healthy
# 2024-01-15T10:30:01+00:00             Service      nginx-demo   nginx   Synced   Healthy
```

### Step 5: Verify Everything Is Running

```bash
# Check ArgoCD app status
argocd app get nginx-demo

# Check in Kubernetes directly
kubectl get all -n nginx-demo

# Output:
# NAME                         READY   STATUS    RESTARTS   AGE
# pod/nginx-xxxxxxxxxx-xxxxx   1/1     Running   0          30s
# pod/nginx-xxxxxxxxxx-xxxxx   1/1     Running   0          30s
# 
# NAME            TYPE        CLUSTER-IP     PORT(S)   AGE
# service/nginx   ClusterIP   10.96.xx.xx    80/TCP    30s
# 
# NAME                    READY   UP-TO-DATE   AVAILABLE   AGE
# deployment.apps/nginx   2/2     2            2           30s
```

**Congratulations! You've deployed your first app via GitOps!** 🎉

## Application CRD — Every Field Explained

Let's go through a **complete** Application YAML with all the commonly used fields:

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: my-app                        # Name of the ArgoCD application
  namespace: argocd                    # MUST be in the argocd namespace
  labels:                              # Optional labels for organizing
    team: backend
    env: production
  annotations:
    notifications.argoproj.io/subscribe.on-sync-succeeded.slack: my-channel
  finalizers:                          # Optional: cascade delete
    - resources-finalizer.argocd.argoproj.io   # Deletes K8s resources when app is deleted

spec:
  # ─────────────────────────────────
  # PROJECT
  # ─────────────────────────────────
  project: default
  # ArgoCD Projects restrict what an app can do:
  # - Which repos it can pull from
  # - Which clusters it can deploy to
  # - Which namespaces it can target
  # - Which resource types it can create
  # "default" allows everything.

  # ─────────────────────────────────
  # SOURCE — Where to get the manifests
  # ─────────────────────────────────
  source:
    repoURL: https://github.com/my-org/k8s-configs.git    # Git repo URL
    targetRevision: main                                    # Branch, tag, or commit SHA
    path: apps/my-app                                       # Path inside the repo

    # If using Helm charts:
    # helm:
    #   valueFiles:
    #     - values.yaml
    #     - values-production.yaml
    #   parameters:
    #     - name: image.tag
    #       value: "2.0"
    #   releaseName: my-app

    # If using Kustomize:
    # kustomize:
    #   namePrefix: prod-
    #   commonLabels:
    #     env: production
    #   images:
    #     - my-registry.com/my-app:2.0

    # If using plain directory of YAML files:
    # directory:
    #   recurse: true      # Include subdirectories
    #   jsonnet: {}         # Jsonnet support

  # ─────────────────────────────────
  # DESTINATION — Where to deploy
  # ─────────────────────────────────
  destination:
    server: https://kubernetes.default.svc    # Cluster API server URL
    # OR use cluster name:
    # name: my-production-cluster
    namespace: production                      # Target namespace

  # ─────────────────────────────────
  # SYNC POLICY — How to sync
  # ─────────────────────────────────
  syncPolicy:
    automated:                    # Enable auto-sync
      prune: true                 # Delete resources removed from Git
      selfHeal: true              # Revert manual changes (drift correction)
      allowEmpty: false           # Don't sync if source is empty (safety)

    syncOptions:                  # Fine-tune sync behavior
      - CreateNamespace=true      # Create the namespace if it doesn't exist
      - PrunePropagationPolicy=foreground   # Wait for dependents to be deleted
      - PruneLast=true            # Prune resources after all others are synced
      - ApplyOutOfSyncOnly=true   # Only apply resources that are out of sync
      - Validate=true             # Validate manifests before applying
      - RespectIgnoreDifferences=true

    retry:                        # Retry sync on failure
      limit: 5                    # Max retry attempts
      backoff:
        duration: 5s              # Initial delay
        factor: 2                 # Multiply delay by this factor each retry
        maxDuration: 3m           # Maximum delay between retries

  # ─────────────────────────────────
  # IGNORE DIFFERENCES — Fields to ignore when comparing
  # ─────────────────────────────────
  ignoreDifferences:
    - group: apps
      kind: Deployment
      jsonPointers:
        - /spec/replicas     # Ignore replica count (useful with HPA)
    - group: ""
      kind: Service
      jqPathExpressions:
        - .spec.clusterIP    # Ignore auto-assigned ClusterIP

  # ─────────────────────────────────
  # REVISION HISTORY LIMIT
  # ─────────────────────────────────
  revisionHistoryLimit: 10    # Keep last 10 sync history entries
```

### Quick Reference: Source Types

ArgoCD supports multiple source types. Here's how to configure each:

**Plain Kubernetes YAML:**
```yaml
source:
  repoURL: https://github.com/my-org/k8s-configs.git

  path: apps/my-app
  directory:
    recurse: true     # Include files in subdirectories
```

**Helm Chart from a Git Repo:**
```yaml
source:
  repoURL: https://github.com/my-org/helm-charts.git

    valueFiles:
      - values.yaml
      - values-prod.yaml
```

**Helm Chart from a Helm Repository:**
```yaml
source:
  repoURL: https://charts.bitnami.com/bitnami    # Helm repo URL (not Git!)

```

**Kustomize:**
```yaml

    commonLabels:
      env: production
```

## Sync Policies — Controlling How ArgoCD Deploys

Sync policies tell ArgoCD **when** and **how** to apply changes from Git to the cluster.

### The Two Main Modes

| Mode | How It Works |
|------|-------------|
| **Manual Sync** | ArgoCD detects changes but waits for YOU to click "Sync" or run `argocd app sync` |
| **Auto-Sync** | ArgoCD automatically applies changes as soon as it detects them in Git |

---

## Manual Sync vs Auto-Sync

### Manual Sync (Default)

When you create an application without a sync policy, it defaults to **manual sync**:

```yaml
spec:
  # No syncPolicy section = manual sync
  source:
    repoURL: https://github.com/my-org/k8s-configs.git
    targetRevision: main
    path: apps/my-app
  destination:
    server: https://kubernetes.default.svc
    namespace: production
```

**How it works:**
1. ArgoCD detects a new commit in Git.
2. ArgoCD shows the app as `OutOfSync`.
3. You manually trigger a sync:
   - **CLI:** `argocd app sync my-app`
   - **UI:** Click the "SYNC" button on the app.
4. ArgoCD applies the changes.

**When to use manual sync:**
- **Production environments** where you want a human to approve deployments.
- **Critical applications** that need extra verification.
- **Compliance-heavy environments** (banks, healthcare) where manual approval is required.
- When you're **first learning** ArgoCD and want to see what happens before auto-deploying.

### Auto-Sync

Auto-sync tells ArgoCD to **automatically** apply changes whenever it detects them:

```yaml
spec:
  syncPolicy:
    automated: {}     # Enable auto-sync with default settings
```

Or with more options:

```yaml
spec:
  syncPolicy:
    automated:
      prune: true       # Also delete resources removed from Git
      selfHeal: true    # Auto-correct manual changes
      allowEmpty: false # Don't sync if no resources found
```

**How it works:**
1. ArgoCD detects a new commit in Git.
2. ArgoCD **immediately** applies the changes. No human needed.
3. Done. ✅

**When to use auto-sync:**
- **Development/staging environments** where speed matters.
- **Non-critical apps** where fast deployment is preferred.
- When you fully trust your Git workflow (PRs, code reviews, CI tests).
- **Mature teams** with good testing and rollback strategies.

### Comparison

| Feature | Manual Sync | Auto-Sync |
|---------|------------|-----------|
| **Human approval needed?** | Yes | No |
| **Speed** | Slower (waiting for human) | Instant |
| **Safety** | Higher (human checks) | Depends on your Git workflow |
| **Best for** | Production, critical apps | Dev/staging, fast iteration |
| **Risk** | Low | Medium (bad commit = auto-deployed) |

## Sync Options — Fine-Tuning Behavior

Sync options give you granular control over how ArgoCD applies changes:

### `CreateNamespace=true`

**What it does:** Creates the target namespace if it doesn't exist.

```yaml
syncPolicy:
  syncOptions:
    - CreateNamespace=true
```

**Without this:** If the namespace doesn't exist, the sync will fail with an error.

---

### `ApplyOutOfSyncOnly=true`

**What it does:** Only applies resources that have actually changed, not everything.

```yaml
syncPolicy:
  syncOptions:
    - ApplyOutOfSyncOnly=true
```

**Why it matters:** If you have 50 resources but only 1 changed, ArgoCD will only apply that 1 resource instead of re-applying all 50. This is faster and safer.

---

### `PruneLast=true`

**What it does:** When deleting old resources (pruning), do it AFTER all new resources are applied.

```yaml
syncPolicy:
  syncOptions:
    - PruneLast=true
```

**Why it matters:** Prevents downtime. New resources come up first, then old ones are removed.

---

### `ServerSideApply=true`

**What it does:** Uses Kubernetes server-side apply instead of client-side.

```yaml
syncPolicy:
  syncOptions:
    - ServerSideApply=true
```

**Why it matters:** Better conflict resolution when multiple tools manage the same resource.

---

### `Replace=true`

**What it does:** Uses `kubectl replace` instead of `kubectl apply`.

```yaml
syncPolicy:
  syncOptions:
    - Replace=true
```

**When to use:** When a resource can't be patched (some CRDs have this issue).

---

### `Validate=true` (default)

**What it does:** Validates manifests against the Kubernetes API before applying.

```yaml
syncPolicy:
  syncOptions:
    - Validate=true    # This is the default
```

---

## Pruning — Deleting Resources Removed from Git

**Pruning** is what happens when you **delete** a resource file from Git.

### Without Pruning (Default)

```yaml
syncPolicy:
  automated: {}    # No prune: true
```

If you delete `service.yaml` from your Git repo:
- ArgoCD sees that the Service exists in the cluster but NOT in Git.
- ArgoCD does **nothing**. The orphaned Service stays in the cluster.
- ArgoCD shows the resource as "pruned" but doesn't delete it.

### With Pruning Enabled

```yaml
syncPolicy:
  automated:
    prune: true    # ← Enable pruning
```

If you delete `service.yaml` from your Git repo:
- ArgoCD detects the difference.
- ArgoCD **deletes** the Service from the cluster.
- The cluster now matches Git exactly. ✅

### When to Enable Pruning

| Scenario | Prune? |
|----------|--------|
| Production with strict GitOps | ✅ Yes — cluster must match Git exactly |
| Development/staging | ✅ Yes — keep things clean |
| Learning/experimenting | ❌ No — you might accidentally delete things |
| Migrating to GitOps gradually | ❌ No — some resources might not be in Git yet |

### Safety: The `resources-finalizer`

If you add the ArgoCD finalizer to your Application, deleting the ArgoCD Application will also **delete all the Kubernetes resources it manages**:

```yaml
metadata:
  finalizers:
    - resources-finalizer.argocd.argoproj.io
```

**Without finalizer:** Delete ArgoCD app → Kubernetes resources stay.
**With finalizer:** Delete ArgoCD app → Kubernetes resources are also deleted.

> **Be careful with this in production!** Deleting an ArgoCD Application with the finalizer will delete your actual running workloads.

## Self-Heal — Automatic Drift Correction

Self-heal is one of the most powerful features of GitOps with ArgoCD.

### What is Self-Heal?

**Self-heal** means: If someone (or something) makes a manual change directly in the cluster, ArgoCD will **automatically revert it** to match what Git says.

### Enabling Self-Heal

```yaml
syncPolicy:
  automated:
    selfHeal: true    # ← Enable self-healing
```

### Example

```
1. Git says: replicas: 3, image: nginx:1.25

2. Someone runs:
   kubectl scale deployment nginx --replicas=10 -n production

3. Cluster now has: replicas: 10 (DRIFT!)

4. ArgoCD detects drift:
   "Git says 3, cluster has 10. SelfHeal is enabled."

5. ArgoCD automatically runs the equivalent of:
   kubectl apply -f deployment.yaml
   (which sets replicas back to 3)

6. Cluster is back to: replicas: 3 ✅
```

### Self-Heal Behavior

| Setting | Behavior |
|---------|----------|
| `selfHeal: false` (default) | ArgoCD detects drift and shows `OutOfSync`, but does NOT fix it |
| `selfHeal: true` | ArgoCD detects drift and **automatically** fixes it |

### Ignoring Certain Differences

Sometimes you WANT certain fields to be managed outside of Git (e.g., HPA manages replicas). Use `ignoreDifferences`:

```yaml
spec:
  ignoreDifferences:
    - group: apps
      kind: Deployment
      jsonPointers:
        - /spec/replicas     # Don't treat replica changes as drift
```

Now ArgoCD will ignore replica count differences — perfect when using Horizontal Pod Autoscaler (HPA).

## Sync Waves and Hooks

### Sync Waves

**Sync waves** let you control the **order** in which resources are applied during a sync.

**Why is this useful?** Some resources depend on others:
- Namespace must exist before Deployments.
- ConfigMaps/Secrets must exist before Pods that use them.
- CRDs must exist before Custom Resources.

```yaml
# This resource will be applied FIRST (wave 0 is the default)
apiVersion: v1
kind: Namespace
metadata:
  name: my-app
  annotations:
    argocd.argoproj.io/sync-wave: "-1"    # Negative = applied earlier
```

```yaml
# This resource will be applied SECOND (wave 0)
apiVersion: v1
kind: ConfigMap
metadata:
  name: my-config
  namespace: my-app
  annotations:
    argocd.argoproj.io/sync-wave: "0"     # Default wave
```

```yaml
# This resource will be applied THIRD (wave 1)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
  namespace: my-app
  annotations:
    argocd.argoproj.io/sync-wave: "1"     # Applied after wave 0
```

**The order:** Wave -1 → Wave 0 → Wave 1 → Wave 2 → ...

### Sync Hooks

**Hooks** let you run special jobs at specific points during the sync process:

| Hook | When It Runs |
|------|-------------|
| `PreSync` | Before the sync starts |
| `Sync` | During the sync (alongside other resources) |
| `PostSync` | After all resources are synced and healthy |
| `SyncFail` | When the sync fails |
| `Skip` | Skipped during sync (resource is not applied) |

**Example: Run a database migration before deployment:**

```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: db-migrate
  namespace: my-app
  annotations:
    argocd.argoproj.io/hook: PreSync           # Run before sync
    argocd.argoproj.io/hook-delete-policy: HookSucceeded   # Clean up after success
spec:
  template:
    spec:
      containers:
      - name: migrate
        image: my-app:2.0
        command: ["python", "manage.py", "migrate"]
      restartPolicy: Never
  backoffLimit: 3
```

**Example: Send a Slack notification after deployment:**

```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: notify-slack
  namespace: my-app
  annotations:
    argocd.argoproj.io/hook: PostSync          # Run after sync completes
    argocd.argoproj.io/hook-delete-policy: HookSucceeded
spec:
  template:
    spec:
      containers:
      - name: notify
        image: curlimages/curl:latest
        command:
          - "curl"
          - "-X"
          - "POST"
          - "https://hooks.slack.com/services/XXXXX"
          - "-d"
          - '{"text":"Deployment of my-app completed successfully!"}'
      restartPolicy: Never
```

### Hook Delete Policies

| Policy | Behavior |
|--------|----------|
| `HookSucceeded` | Delete the hook resource after it succeeds |
| `HookFailed` | Delete the hook resource after it fails |
| `BeforeHookCreation` | Delete the previous hook before creating a new one (default) |

## Health Checks — How ArgoCD Knows If Your App Is OK

ArgoCD doesn't just deploy — it also monitors the **health** of your applications.

### Built-In Health Checks

ArgoCD knows how to check the health of standard Kubernetes resources:

| Resource | How ArgoCD Checks Health |
|----------|------------------------|
| **Deployment** | All replicas are ready and updated |
| **StatefulSet** | All replicas are ready and at the correct revision |
| **DaemonSet** | Desired number equals ready number |
| **Service** | Always "Healthy" (services don't have health) |
| **Ingress** | Has an assigned address/IP |
| **PersistentVolumeClaim** | Status is "Bound" |
| **Pod** | Running and ready |
| **Job** | Completed successfully |

### Health Statuses

| Status | Meaning | Icon in UI |
|--------|---------|-----------|
| **Healthy** | Everything is working as expected | ✅ Green heart |
| **Progressing** | Resource is updating (e.g., rolling update in progress) | 🔄 Spinning |
| **Degraded** | Something is wrong (e.g., CrashLoopBackOff) | ❤️ Yellow/Red |
| **Suspended** | Resource is paused (e.g., suspended CronJob) | ⏸️ Paused |
| **Missing** | Resource doesn't exist in the cluster yet | ❓ Missing |
| **Unknown** | ArgoCD can't determine the health | ❔ Unknown |

### Application-Level Statuses

Each ArgoCD Application has two statuses:

```
┌──────────────────────────────────┐
│  Application: my-app             │
│                                  │
│  Sync Status:   Synced ✅        │  ← Does the cluster match Git?
│  Health Status: Healthy ✅       │  ← Is the app actually working?
│                                  │
└──────────────────────────────────┘
```

| Sync Status | Meaning |
|-------------|---------|
| `Synced` | Cluster matches Git exactly |
| `OutOfSync` | Cluster doesn't match Git (there are changes to apply) |
| `Unknown` | ArgoCD hasn't checked yet |

## Practical Example — Full GitOps Setup from Scratch

Let's put everything together with a complete, real-world example.

### The Goal

Deploy a simple **Guestbook application** using ArgoCD with:
- Auto-sync enabled
- Pruning enabled
- Self-heal enabled
- Namespace auto-creation

### Step 1: Verify ArgoCD Is Running

```bash
kubectl get pods -n argocd
# All pods should be Running
```

### Step 2: Create the Application (Using ArgoCD's Demo Repo)

ArgoCD provides a sample repo we can use:

```yaml
# guestbook-app.yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: guestbook
  namespace: argocd
  finalizers:
    - resources-finalizer.argocd.argoproj.io
spec:
  project: default

  source:
    repoURL: https://github.com/argoproj/argocd-example-apps.git
    targetRevision: HEAD
    path: guestbook

  destination:
    server: https://kubernetes.default.svc
    namespace: guestbook

  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
      - CreateNamespace=true

    retry:
      limit: 3
      backoff:
        duration: 5s
        factor: 2
        maxDuration: 1m
```

```bash
# Apply the Application
kubectl apply -f guestbook-app.yaml

# Output: application.argoproj.io/guestbook created
```

### Step 3: Watch ArgoCD Work Its Magic


## Troubleshooting Common Issues

### Issue 1: Application Stuck in "Progressing"

**Symptoms:** App shows "Progressing" for a long time.

**Common causes:**
```bash
# Check the pods
kubectl get pods -n <namespace>

# Common issues:
# - ImagePullBackOff → Wrong image name/tag or no registry access
# - CrashLoopBackOff → App crashes on startup
# - Pending → Not enough resources (CPU/memory) or no matching nodes

# Check pod events
kubectl describe pod <pod-name> -n <namespace>

# Check ArgoCD app events
argocd app get <app-name>
```

---

### Issue 2: "ComparisonError" or "Failed to Load Target State"

**Symptoms:** ArgoCD can't read from the Git repo.

**Fix:**
```bash
# Check if ArgoCD can access the repo
argocd repo list

# Re-add the repo with credentials
argocd repo add https://github.com/my-org/configs.git \
  --username <username> \
  --password <token>
```

---

### Issue 3: "OutOfSync" Even After Syncing

**Symptoms:** App keeps showing `OutOfSync` after every sync.

**Common cause:** Kubernetes adds default fields that aren't in your Git manifests (like default tolerations, service account tokens, etc.).

**Fix:** Use `ignoreDifferences`:
```yaml
spec:
  ignoreDifferences:
    - group: apps
      kind: Deployment
      jsonPointers:
        - /spec/template/metadata/annotations
```

---

### Issue 4: Namespace Not Found

**Symptoms:** Sync fails because the target namespace doesn't exist.

**Fix:**
```yaml
syncPolicy:
  syncOptions:
    - CreateNamespace=true
```

---

### Issue 5: Permission Denied

**Symptoms:** ArgoCD can't create resources in a namespace.

**Fix:** Check RBAC — the ArgoCD application controller needs permissions:
```bash
# Check if the service account has the right permissions
kubectl auth can-i create deployments \
  --as=system:serviceaccount:argocd:argocd-application-controller \
  -n <target-namespace>
```

## Best Practices

### 1. Use Declarative Application Definitions

Don't create apps via CLI or UI. Use YAML files stored in Git:

```yaml
# Store this file in your config repo!
# apps/argocd-apps/my-app.yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: my-app
  namespace: argocd
spec:
  # ... (your app config)
```

> **Meta-GitOps:** Even your ArgoCD Application definitions should be in Git!

---

### 2. Use Projects to Restrict Access

Don't put everything in the `default` project:

```yaml
apiVersion: argoproj.io/v1alpha1
kind: AppProject
metadata:
  name: team-backend
  namespace: argocd
spec:
  description: "Backend team's project"
  sourceRepos:
    - "https://github.com/my-org/backend-configs.git"
  destinations:
    - namespace: "backend-*"
      server: "https://kubernetes.default.svc"
  clusterResourceWhitelist:
    - group: ""
      kind: Namespace
```

---

### 3. Start with Manual Sync, Move to Auto-Sync

- **New apps / production:** Start with manual sync. Get comfortable.
- **After confidence builds:** Enable auto-sync.
- **After more confidence:** Enable auto-sync + prune + self-heal.

---

### 4. Always Use `CreateNamespace=true`

Avoids "namespace not found" errors:

```yaml
syncPolicy:
  syncOptions:
    - CreateNamespace=true
```

---

### 5. Use `ignoreDifferences` Wisely

If you use HPA (Horizontal Pod Autoscaler), ignore replicas:

```yaml
ignoreDifferences:
  - group: apps
    kind: Deployment
    jsonPointers:
      - /spec/replicas
```

---

### 6. Set Up Notifications

ArgoCD has a notifications system. Get alerts on Slack, email, Teams, etc. when things go wrong or deployments succeed.

---

### 7. Use the Finalizer Carefully

```yaml
finalizers:
  - resources-finalizer.argocd.argoproj.io
```

This means: **Deleting the ArgoCD app will delete all Kubernetes resources.** Be very careful in production.
