## 🤔 The Problem: Managing Many Applications

Imagine you have **50 microservices** running in Kubernetes across **3 environments** (dev, staging, prod).

That means you need to create **150 ArgoCD Application manifests** — one for each microservice in each environment.

Now imagine doing this **by hand**:

```yaml
# You'd write this 150 times with tiny differences... 😱
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: user-service-dev
spec:
  project: default
  source:
    repoURL: https://github.com/myorg/k8s-manifests.git
    path: apps/user-service/overlays/dev
  destination:
    server: https://kubernetes.default.svc
    namespace: user-service
```

That's a nightmare. You'd go insane copying, pasting, and modifying YAML files.

**Two solutions exist:**
1. **App of Apps Pattern** — A parent app that manages child apps
2. **ApplicationSets** — A template engine that auto-generates apps

Let's learn both! 🚀

---

## 📁 Part 1: The App of Apps Pattern

### What Is It?

The App of Apps pattern is simple:

> **You create ONE ArgoCD Application (the "parent") that points to a Git directory containing OTHER ArgoCD Application manifests (the "children").**

Think of it like a **folder of folders**:

```
📁 Parent App (root-app)
├── 📄 Application: user-service
├── 📄 Application: order-service
├── 📄 Application: payment-service
├── 📄 Application: inventory-service
└── 📄 Application: notification-service
```

When ArgoCD syncs the parent app, it creates all the child apps automatically. When you add a new child app manifest to the Git folder, ArgoCD picks it up and deploys it. Magic! ✨

---

### How Does It Work? — Step by Step

#### Step 1: Create a Directory for Child Applications

In your Git repo, create a folder that holds all your Application manifests:

```
k8s-manifests/
├── apps/                          # ← This is your "app of apps" directory
│   ├── user-service.yaml          # ArgoCD Application for user-service
│   ├── order-service.yaml         # ArgoCD Application for order-service
│   ├── payment-service.yaml       # ArgoCD Application for payment-service
│   └── notification-service.yaml  # ArgoCD Application for notification-service
├── base/
│   ├── user-service/
│   ├── order-service/
│   └── ...
└── overlays/
    ├── dev/
    ├── staging/
    └── prod/
```

#### Step 2: Create Child Application Manifests

Each file in the `apps/` directory is a standard ArgoCD Application:

**`apps/user-service.yaml`**

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: user-service
  namespace: argocd
  # This label tells ArgoCD that this app is managed by the parent
  labels:
    app.kubernetes.io/part-of: root-app
  # Finalizer ensures proper cleanup when deleted
  finalizers:
    - resources-finalizer.argocd.argoproj.io
spec:
  project: default
  source:
    repoURL: https://github.com/myorg/k8s-manifests.git
    targetRevision: main
    path: base/user-service         # Points to the actual K8s manifests
  destination:
    server: https://kubernetes.default.svc
    namespace: user-service
  syncPolicy:
    automated:                       # Auto-sync enabled
      prune: true                    # Delete resources not in Git
      selfHeal: true                 # Fix drift automatically
    syncOptions:
      - CreateNamespace=true         # Create namespace if it doesn't exist
```

**`apps/order-service.yaml`**

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: order-service
  namespace: argocd
  labels:
    app.kubernetes.io/part-of: root-app
  finalizers:
    - resources-finalizer.argocd.argoproj.io
spec:
  project: default
  source:
    repoURL: https://github.com/myorg/k8s-manifests.git
    targetRevision: main
    path: base/order-service
  destination:
    server: https://kubernetes.default.svc
    namespace: order-service
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
      - CreateNamespace=true
```

#### Step 3: Create the Parent Application (The "Root App")

Now, create the **parent** Application that points to the `apps/` directory:

**`root-app.yaml`**

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: root-app
  namespace: argocd
  finalizers:
    - resources-finalizer.argocd.argoproj.io
spec:
  project: default
  source:
    repoURL: https://github.com/myorg/k8s-manifests.git
    targetRevision: main
    path: apps                       # ← Points to the folder with child apps!
  destination:
    server: https://kubernetes.default.svc
    namespace: argocd                # Child Application CRDs go into argocd namespace
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
```

#### Step 4: Apply the Parent Application

```bash
kubectl apply -f root-app.yaml
```

That's it! ArgoCD will:
1. Read the `apps/` directory in your Git repo
2. Find all the Application YAML files
3. Create each child Application in ArgoCD
4. Each child Application then syncs its own resources (Deployments, Services, etc.)

---

### Visual Flow

```
                    ┌────────────────────┐
                    │     root-app       │  ← You only apply THIS
                    │   (Parent App)     │
                    └──────┬─────────────┘
                           │
                    Points to apps/ dir
                           │
              ┌────────────┼────────────────┐
              │            │                │
              ▼            ▼                ▼
      ┌──────────┐  ┌──────────┐   ┌───────────────┐
      │  user-   │  │  order-  │   │  payment-     │
      │  service │  │  service │   │  service      │
      └────┬─────┘  └────┬─────┘   └───────┬───────┘
           │              │                 │
           ▼              ▼                 ▼
     K8s resources   K8s resources   K8s resources
     (Deploy, Svc)   (Deploy, Svc)   (Deploy, Svc)
```

### Adding a New App — Super Easy!

Want to add a new microservice? Just add a new YAML file to the `apps/` folder:

```yaml
# apps/analytics-service.yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: analytics-service
  namespace: argocd
  labels:
    app.kubernetes.io/part-of: root-app
  finalizers:
    - resources-finalizer.argocd.argoproj.io
spec:
  project: default
  source:
    repoURL: https://github.com/myorg/k8s-manifests.git
    targetRevision: main
    path: base/analytics-service
  destination:
    server: https://kubernetes.default.svc
    namespace: analytics-service
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
      - CreateNamespace=true
```

Commit it, push to Git, and ArgoCD will automatically create the new Application! No `kubectl` commands needed.

---

### App of Apps with Helm

You can also use **Helm** for the parent app instead of raw YAML files. This is powerful because you can use Helm templates to generate child Application manifests from a `values.yaml` file.

**Directory structure:**

```
root-app-chart/
├── Chart.yaml
├── values.yaml
└── templates/
    └── applications.yaml
```

**`Chart.yaml`**

```yaml
apiVersion: v2
name: root-app
version: 1.0.0
description: Parent app that generates child ArgoCD Applications
```

**`values.yaml`**

```yaml
# Define all your apps here — easy to manage!
apps:
  - name: user-service
    path: base/user-service
    namespace: user-service
  - name: order-service
    path: base/order-service
    namespace: order-service
  - name: payment-service
    path: base/payment-service
    namespace: payment-service
  - name: notification-service
    path: base/notification-service
    namespace: notification-service

# Shared settings
repoURL: https://github.com/myorg/k8s-manifests.git
targetRevision: main
destinationServer: https://kubernetes.default.svc
```

**`templates/applications.yaml`**

```yaml
{{- range .Values.apps }}
---
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: {{ .name }}
  namespace: argocd
  finalizers:
    - resources-finalizer.argocd.argoproj.io
spec:
  project: default
  source:
    repoURL: {{ $.Values.repoURL }}
    targetRevision: {{ $.Values.targetRevision }}
    path: {{ .path }}
  destination:
    server: {{ $.Values.destinationServer }}
    namespace: {{ .namespace }}
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
      - CreateNamespace=true
{{- end }}
```

Now, to add a new app, you just add one entry to `values.yaml` — no need to write a full Application manifest!

---

### Pros & Cons of App of Apps

| ✅ Pros | ❌ Cons |
|---------|---------|
| Simple to understand | You still write YAML for each child app (unless using Helm) |
| Works with plain YAML or Helm | No dynamic generation based on Git directories or clusters |
| Easy to add/remove apps | Doesn't scale well to 100s of apps across many clusters |
| Full control over each child app | Each app has its own sync status (can be noisy) |
| Battle-tested pattern | The parent app itself needs to be bootstrapped manually |

## 🚀 Part 2: ApplicationSets

### What Is ApplicationSet?

**ApplicationSet** is a **controller** that comes with ArgoCD. It's like a **template engine on steroids** — you define a template once, and it **automatically generates** ArgoCD Applications based on **generators**.

Think of it like this:

> **App of Apps** = You write each Application by hand and put them in a folder  
> **ApplicationSet** = You write ONE template, and the controller generates Applications automatically

It's the **official, recommended way** to manage many applications at scale.

---

### Installing ApplicationSet

Good news — **ApplicationSet controller is built into ArgoCD** since version 2.3+. You don't need to install anything extra! If you have ArgoCD 2.3 or later, you already have ApplicationSets.

To verify:

```bash
kubectl get pods -n argocd | grep applicationset

# You should see something like:
# argocd-applicationset-controller-xxx   Running
```

---

### How ApplicationSets Work

An ApplicationSet has three main parts:

```yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: my-appset
  namespace: argocd
spec:
  generators:       # ← WHERE to get the data (list, git, cluster, matrix...)
    - ...
  template:         # ← WHAT to create (an Application template)
    ...
```

1. **Generators** — Define the **data source** (a list, a Git directory, registered clusters, etc.)
2. **Template** — Define the **Application template** that gets filled with data from generators

The controller watches the generators, and whenever new data appears (new directory in Git, new cluster registered, new item in a list), it **automatically creates a new ArgoCD Application**.

---

### Generator 1: List Generator

The simplest generator. You provide a **hardcoded list** of values.

**Use case:** You know exactly which apps and environments you want.

```yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: my-apps
  namespace: argocd
spec:
  generators:
    - list:
        elements:
          - appName: user-service
            namespace: user-service
            cluster: https://kubernetes.default.svc
          - appName: order-service
            namespace: order-service
            cluster: https://kubernetes.default.svc
          - appName: payment-service
            namespace: payment-service
            cluster: https://kubernetes.default.svc
  template:
    metadata:
      name: "{{appName}}"           # ← Uses the variable from the list
    spec:
      project: default
      source:
        repoURL: https://github.com/myorg/k8s-manifests.git
        targetRevision: main
        path: "base/{{appName}}"     # ← Dynamic path!
      destination:
        server: "{{cluster}}"
        namespace: "{{namespace}}"
      syncPolicy:
        automated:
          prune: true
          selfHeal: true
        syncOptions:
          - CreateNamespace=true
```

**What happens:** ArgoCD creates 3 Applications (user-service, order-service, payment-service) from this single template.

---

### Generator 2: Git Directory Generator

This is **super powerful**. It scans a Git repository for **directories** and generates an Application for each one.

**Use case:** Each microservice has its own directory. You want ArgoCD to auto-discover them.

**Git repo structure:**

```
k8s-manifests/
├── apps/
│   ├── user-service/
│   │   ├── deployment.yaml
│   │   └── service.yaml
│   ├── order-service/
│   │   ├── deployment.yaml
│   │   └── service.yaml
│   ├── payment-service/
│   │   ├── deployment.yaml
│   │   └── service.yaml
│   └── notification-service/      # ← New service added!
│       ├── deployment.yaml
│       └── service.yaml
```

**ApplicationSet:**

```yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: microservices
  namespace: argocd
spec:
  generators:
    - git:
        repoURL: https://github.com/myorg/k8s-manifests.git
        revision: main
        directories:
          - path: apps/*              # ← Scan all directories under apps/
  template:
    metadata:
      name: "{{path.basename}}"       # ← "user-service", "order-service", etc.
    spec:
      project: default
      source:
        repoURL: https://github.com/myorg/k8s-manifests.git
        targetRevision: main
        path: "{{path}}"              # ← "apps/user-service", "apps/order-service", etc.
      destination:
        server: https://kubernetes.default.svc
        namespace: "{{path.basename}}"
      syncPolicy:
        automated:
          prune: true
          selfHeal: true
        syncOptions:
          - CreateNamespace=true
```

**The magic:** When someone adds a new directory `apps/analytics-service/`, ArgoCD **automatically** creates a new Application for it! No manual work needed.

**Available template variables from the Git Directory generator:**

| Variable | Example Value | Description |
|----------|--------------|-------------|
| `{{path}}` | `apps/user-service` | Full path of the directory |
| `{{path.basename}}` | `user-service` | Just the directory name |
| `{{path[0]}}` | `apps` | First path segment |
| `{{path[1]}}` | `user-service` | Second path segment |

**Excluding directories:**

```yaml
generators:
  - git:
      repoURL: https://github.com/myorg/k8s-manifests.git
      revision: main
      directories:
        - path: apps/*
        - path: apps/legacy-service    # ← Exclude this one
          exclude: true
```

### Generator 3: Git File Generator

Instead of scanning directories, it reads **specific files** from Git and uses their content as input.

**Use case:** You have a config file (JSON/YAML) that lists all apps and their settings.

**Git repo — `config/apps.json`:**

```json
{
  "apps": [
    {
      "name": "user-service",
      "team": "platform",
      "replicas": 3,
      "namespace": "user-service"
    },
    {
      "name": "order-service",
      "team": "commerce",
      "replicas": 2,
      "namespace": "order-service"
    }
  ]
}
```

**ApplicationSet:**

```yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: config-driven-apps
  namespace: argocd
spec:
  generators:
    - git:
        repoURL: https://github.com/myorg/k8s-manifests.git
        revision: main
        files:
          - path: "config/apps.json"
  template:
    metadata:
      name: "{{name}}"
      labels:
        team: "{{team}}"
    spec:
      project: default
      source:
        repoURL: https://github.com/myorg/k8s-manifests.git
        targetRevision: main
        path: "base/{{name}}"
      destination:
        server: https://kubernetes.default.svc
        namespace: "{{namespace}}"
```

---

### Generator 4: Cluster Generator

Generates Applications for **every cluster** registered in ArgoCD.

**Use case:** You want to deploy the same app (like monitoring agents or ingress controllers) to ALL clusters.

```yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: monitoring-everywhere
  namespace: argocd
spec:
  generators:
    - clusters: {}    # ← ALL registered clusters!
  template:
    metadata:
      name: "monitoring-{{name}}"     # {{name}} = cluster name
    spec:
      project: default
      source:
        repoURL: https://github.com/myorg/k8s-manifests.git
        targetRevision: main
        path: monitoring/base
      destination:
        server: "{{server}}"          # {{server}} = cluster API server URL
        namespace: monitoring
      syncPolicy:
        automated:
          prune: true
          selfHeal: true
```

**What happens:** If you have 3 clusters registered (dev-cluster, staging-cluster, prod-cluster), ArgoCD creates 3 Applications:
- `monitoring-dev-cluster`
- `monitoring-staging-cluster`
- `monitoring-prod-cluster`

When you register a **new cluster**, a new Application is created automatically!

**Filtering clusters by labels:**

```yaml
generators:
  - clusters:
      selector:
        matchLabels:
          env: production            # Only production clusters
```

### Generator 5: Matrix Generator (Combining Generators!)

This is where it gets **really powerful**. The Matrix generator **combines two generators** and creates the **Cartesian product** (every combination).

**Use case:** Deploy every app to every cluster/environment.

```yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: all-apps-all-envs
  namespace: argocd
spec:
  generators:
    - matrix:
        generators:
          # Generator 1: List of environments
          - list:
              elements:
                - env: dev
                  cluster: https://dev.k8s.example.com
                - env: staging
                  cluster: https://staging.k8s.example.com
                - env: prod
                  cluster: https://prod.k8s.example.com
          # Generator 2: Git directories (our microservices)
          - git:
              repoURL: https://github.com/myorg/k8s-manifests.git
              revision: main
              directories:
                - path: apps/*
  template:
    metadata:
      name: "{{path.basename}}-{{env}}"    # e.g., "user-service-dev"
    spec:
      project: default
      source:
        repoURL: https://github.com/myorg/k8s-manifests.git
        targetRevision: main
        path: "overlays/{{env}}/{{path.basename}}"
      destination:
        server: "{{cluster}}"
        namespace: "{{path.basename}}"
      syncPolicy:
        automated:
          prune: true
          selfHeal: true
        syncOptions:
          - CreateNamespace=true
```

**If you have 4 apps and 3 environments, this creates 12 Applications automatically!**

```
user-service-dev          user-service-staging       user-service-prod
order-service-dev         order-service-staging      order-service-prod
payment-service-dev       payment-service-staging    payment-service-prod
notification-service-dev  notification-service-staging notification-service-prod
```

### Generator 6: Merge Generator

The Merge generator is like the Matrix generator, but instead of creating combinations, it **merges** data from multiple generators. Later generators can **override** values from earlier ones.

**Use case:** You have default settings for all apps, but some apps need custom overrides.

```yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: merged-apps
  namespace: argocd
spec:
  generators:
    - merge:
        mergeKeys:
          - appName                    # Merge on this key
        generators:
          # Base generator — defaults for all apps
          - list:
              elements:
                - appName: user-service
                  replicas: "2"
                  memory: "256Mi"
                - appName: order-service
                  replicas: "2"
                  memory: "256Mi"
                - appName: payment-service
                  replicas: "2"
                  memory: "256Mi"
          # Override generator — custom values for specific apps
          - list:
              elements:
                - appName: payment-service
                  replicas: "5"        # ← Override: payment needs more replicas
                  memory: "512Mi"      # ← Override: and more memory
  template:
    metadata:
      name: "{{appName}}"
    spec:
      project: default
      source:
        repoURL: https://github.com/myorg/k8s-manifests.git
        targetRevision: main
        path: "base/{{appName}}"
      destination:
        server: https://kubernetes.default.svc
        namespace: "{{appName}}"
```

Result:
- `user-service` → replicas: 2, memory: 256Mi (default)
- `order-service` → replicas: 2, memory: 256Mi (default)
- `payment-service` → replicas: 5, memory: 512Mi (overridden!)

### ApplicationSet Policies

You can control what happens when an ApplicationSet updates or deletes generated Applications.

#### Sync Policy

```yaml
spec:
  syncPolicy:
    preserveResourcesOnDeletion: true   # Don't delete K8s resources when AppSet is deleted
```

This is a **safety net** — if someone accidentally deletes the ApplicationSet, the actual Kubernetes resources (Deployments, Services, etc.) will NOT be deleted.

#### Application Ownership Policy

```yaml
spec:
  # What to do when ApplicationSet no longer generates an app
  # (e.g., directory removed from Git)
  strategy:
    type: create-update               # Only create and update, never delete
    # OR
    type: create-only                 # Only create, never update or delete
    # OR
    type: create-delete               # Create and delete (no updates)
```

### Real-World Example: Complete Multi-Environment Setup

Here's a production-ready example that puts it all together:

```yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: microservices-all-envs
  namespace: argocd
spec:
  # Don't delete K8s resources if this ApplicationSet is removed
  syncPolicy:
    preserveResourcesOnDeletion: true

  generators:
    - matrix:
        generators:
          # Environments
          - list:
              elements:
                - env: dev
                  cluster: https://kubernetes.default.svc
                  autoSync: "true"
                - env: staging
                  cluster: https://staging.k8s.example.com
                  autoSync: "true"
                - env: prod
                  cluster: https://prod.k8s.example.com
                  autoSync: "false"     # ← Manual sync in prod!
          # Apps (auto-discovered from Git)
          - git:
              repoURL: https://github.com/myorg/k8s-manifests.git
              revision: main
              directories:
                - path: apps/*
                - path: apps/experimental-*
                  exclude: true          # Exclude experimental apps

  template:
    metadata:
      name: "{{path.basename}}-{{env}}"
      namespace: argocd
      labels:
        environment: "{{env}}"
        app: "{{path.basename}}"
      annotations:
        notifications.argoproj.io/subscribe.on-sync-failed.slack: devops-alerts
    spec:
      project: "{{env}}"               # Use ArgoCD projects per environment
      source:
        repoURL: https://github.com/myorg/k8s-manifests.git
        targetRevision: main
        path: "overlays/{{env}}/{{path.basename}}"
      destination:
        server: "{{cluster}}"
        namespace: "{{path.basename}}"
      syncPolicy:
        automated:
          prune: true
          selfHeal: true
        syncOptions:
          - CreateNamespace=true
          - PruneLast=true
        retry:
          limit: 3
          backoff:
            duration: 5s
            factor: 2
            maxDuration: 3m
```

## 🥊 App of Apps vs ApplicationSets — When to Use What?

| Feature | App of Apps | ApplicationSets |
|---------|------------|-----------------|
| **Complexity** | Simple — just YAML files in a folder | More powerful but more complex |
| **Auto-discovery** | No — you add each app manually | Yes — scan Git directories, clusters |
| **Scaling** | Good for <30 apps | Great for 100s of apps |
| **Dynamic generation** | Only with Helm templates | Built-in generators |
| **Multi-cluster** | Manual — one app per cluster | Automatic — cluster generator |
| **Official support** | Community pattern | Official ArgoCD feature |
| **Version requirement** | Any ArgoCD version | ArgoCD 2.3+ |
| **When to use** | Small/medium setups, learning | Large-scale, multi-cluster, dynamic |

**My recommendation:**
- **Starting out?** → Use App of Apps. It's simple and you'll understand what's happening.
- **Scaling up?** → Switch to ApplicationSets. They're more maintainable at scale.
- **Many clusters?** → ApplicationSets with Cluster + Matrix generators is the way to go.

## 🎯 Common Gotchas & Best Practices

### 1. Always Use Finalizers on Child Apps

```yaml
finalizers:
  - resources-finalizer.argocd.argoproj.io
```

Without this, when you delete an Application, the Kubernetes resources it created **won't be cleaned up**. You'll have orphaned Deployments, Services, etc. floating around.

### 2. Use `preserveResourcesOnDeletion` in Production

```yaml
spec:
  syncPolicy:
    preserveResourcesOnDeletion: true
```

If someone accidentally deletes the ApplicationSet, you don't want all your production resources to vanish.

### 3. Use ArgoCD Projects to Restrict Access

Don't dump everything into the `default` project. Create separate ArgoCD Projects for each environment:

```yaml
apiVersion: argoproj.io/v1alpha1
kind: AppProject
metadata:
  name: production
  namespace: argocd
spec:
  description: Production environment
  sourceRepos:
    - https://github.com/myorg/k8s-manifests.git
  destinations:
    - server: https://prod.k8s.example.com
      namespace: '*'
  clusterResourceWhitelist:
    - group: ''
      kind: Namespace
```

### 4. Template Variables Are Case-Sensitive

`{{appName}}` and `{{appname}}` are different! Make sure your generator keys match the template variables exactly.

### 5. Test with `argocd appset generate`

Before applying an ApplicationSet, preview what it will generate:

```bash
# Preview generated Applications (dry run)
argocd appset generate my-appset.yaml
```

This shows you all the Applications that would be created without actually creating them.

## 🧠 Quick Recap

| Concept | What It Does |
|---------|-------------|
| **App of Apps** | A parent ArgoCD App that manages child Application manifests from a Git directory |
| **ApplicationSet** | A controller that auto-generates Applications from templates + generators |
| **List Generator** | Creates apps from a hardcoded list of values |
| **Git Directory Generator** | Creates apps from directories found in a Git repo |
| **Git File Generator** | Creates apps from config files (JSON/YAML) in a Git repo |
| **Cluster Generator** | Creates apps for each registered cluster |
| **Matrix Generator** | Combines two generators (Cartesian product = every combination) |
| **Merge Generator** | Merges data from generators with override capability |
