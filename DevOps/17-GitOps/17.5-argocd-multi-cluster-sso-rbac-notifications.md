## 🤔 Why Do You Need This?

When you're just learning ArgoCD, you deploy to a single cluster and log in as admin. Easy.

But in a **real company**, things get more complex:

| Challenge | Solution |
|-----------|----------|
| "We have 5 Kubernetes clusters (dev, staging, prod-us, prod-eu, prod-asia)" | **Multi-Cluster management** |
| "We have 30 developers. They shouldn't all be admin!" | **RBAC — Role-Based Access Control** |
| "Everyone should log in with their company Google/GitHub/Okta account" | **SSO — Single Sign-On** |
| "We want Slack alerts when a deployment fails or succeeds" | **Notifications** |

Let's tackle each one! 🚀

## 🌍 Part 1: Multi-Cluster Management

### The Concept

ArgoCD runs on **one cluster** (the "management cluster") but can deploy applications to **many clusters** (the "target clusters").

```
                    ┌────────────────────────┐
                    │   Management Cluster   │
                    │    (ArgoCD lives here)  │
                    └───────────┬────────────┘
                                │
              ┌─────────────────┼─────────────────┐
              │                 │                  │
              ▼                 ▼                  ▼
     ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
     │  Dev Cluster  │  │ Staging      │  │ Prod Cluster │
     │              │  │ Cluster      │  │              │
     └──────────────┘  └──────────────┘  └──────────────┘
```

ArgoCD uses **ServiceAccount tokens** or **kubeconfig credentials** to authenticate with remote clusters.

---

### Adding a Cluster to ArgoCD

#### Method 1: Using the ArgoCD CLI (Recommended)

The CLI does the heavy lifting — it creates a ServiceAccount on the target cluster and registers it in ArgoCD.

```bash
# Step 1: Make sure your kubeconfig has access to the target cluster
kubectl config get-contexts

# You'll see something like:
# CURRENT   NAME              CLUSTER           
# *         management        management-cluster
#           dev-cluster       dev-cluster       
#           staging-cluster   staging-cluster   
#           prod-cluster      prod-cluster      

# Step 2: Add the cluster to ArgoCD
argocd cluster add dev-cluster

# What this does behind the scenes:
# 1. Creates a ServiceAccount "argocd-manager" in the target cluster
# 2. Creates a ClusterRole with broad permissions
# 3. Creates a ClusterRoleBinding
# 4. Stores the ServiceAccount token in ArgoCD as a secret
```

**What ArgoCD creates on the target cluster:**

```yaml
# ServiceAccount on the target cluster
apiVersion: v1
kind: ServiceAccount
metadata:
  name: argocd-manager
  namespace: kube-system

---
# ClusterRole with broad permissions
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: argocd-manager-role
rules:
  - apiGroups: ["*"]
    resources: ["*"]
    verbs: ["*"]        # Full access — you can restrict this later

---
# Binding
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: argocd-manager-role-binding
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: argocd-manager-role
subjects:
  - kind: ServiceAccount
    name: argocd-manager
    namespace: kube-system
```

#### Method 2: Using a Kubernetes Secret (Declarative)

You can also register a cluster by creating a Secret in the `argocd` namespace. This is useful for GitOps — you can store the cluster registration in Git!

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: dev-cluster-secret
  namespace: argocd
  labels:
    argocd.argoproj.io/secret-type: cluster    # ← This label is REQUIRED
type: Opaque
stringData:
  name: dev-cluster
  server: https://dev.k8s.example.com
  config: |
    {
      "bearerToken": "<service-account-token>",
      "tlsClientConfig": {
        "insecure": false,
        "caData": "<base64-encoded-ca-cert>"
      }
    }
```

> ⚠️ **Security Note:** In production, use **External Secrets Operator** or **Sealed Secrets** to manage the bearer token. Never commit raw tokens to Git!

#### Method 3: Using AWS IAM (for EKS clusters)

For AWS EKS clusters, you can use IAM roles instead of bearer tokens:

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: eks-cluster
  namespace: argocd
  labels:
    argocd.argoproj.io/secret-type: cluster
type: Opaque
stringData:
  name: eks-prod
  server: https://ABCDEF1234.gr7.us-east-1.eks.amazonaws.com
  config: |
    {
      "awsAuthConfig": {
        "clusterName": "my-eks-cluster",
        "roleARN": "arn:aws:iam::123456789012:role/argocd-manager"
      },
      "tlsClientConfig": {
        "insecure": false,
        "caData": "<base64-encoded-ca-cert>"
      }
    }
```


### Deploying to a Remote Cluster

Once a cluster is registered, just use its server URL in the Application's `destination`:

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: user-service-prod
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/myorg/k8s-manifests.git
    targetRevision: main
    path: overlays/prod/user-service
  destination:
    server: https://prod.k8s.example.com    # ← Remote cluster!
    namespace: user-service
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
```

That's it! ArgoCD will deploy to the remote production cluster.

---

### Multi-Cluster with ApplicationSets

The real power comes when you combine multi-cluster with ApplicationSets (covered in 17.4):

```yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: monitoring-all-clusters
  namespace: argocd
spec:
  generators:
    - clusters: {}               # ← Generate for ALL registered clusters
  template:
    metadata:
      name: "monitoring-{{name}}"
    spec:
      project: default
      source:
        repoURL: https://github.com/myorg/k8s-manifests.git
        targetRevision: main
        path: monitoring/base
      destination:
        server: "{{server}}"     # ← Automatically uses each cluster's URL
        namespace: monitoring
```

Register a new cluster → monitoring is automatically deployed there! 🎉

---

### Multi-Cluster Best Practices

| Practice | Why |
|----------|-----|
| Use a **dedicated management cluster** for ArgoCD | Keeps ArgoCD isolated from workload issues |
| **Label your clusters** (env=prod, region=us-east-1) | Makes filtering with ApplicationSets easy |
| Use **ArgoCD Projects** per environment | Prevents dev from deploying to prod |
| **Restrict ClusterRole** on target clusters | Don't give ArgoCD full admin everywhere |
| Use **cloud IAM** (AWS IAM, GCP Workload Identity) | More secure than static tokens |

## 🔐 Part 2: SSO — Single Sign-On

### Why SSO?

By default, ArgoCD has a single `admin` user with a password. That's fine for learning, but in a real company:

- You don't want 50 people sharing one password
- You want people to log in with their company accounts (Google, GitHub, Okta, etc.)
- You want to know WHO did WHAT (audit trail)
- You want to automatically remove access when someone leaves the company

**SSO solves all of this** by delegating authentication to an external **Identity Provider (IdP)**.

---

### How ArgoCD SSO Works

ArgoCD supports SSO through **Dex** (a built-in OIDC provider) or **direct OIDC integration**:

```
User → ArgoCD UI → "Login with Google" → Google authenticates → 
→ Returns token → ArgoCD verifies → User is logged in!
```

Two approaches:

| Approach | How It Works | When to Use |
|----------|-------------|-------------|
| **Dex (built-in)** | ArgoCD ships with Dex, which acts as a middleman to many IdPs | When your IdP doesn't support OIDC natively, or you want multi-IdP support |
| **Direct OIDC** | ArgoCD talks directly to your OIDC provider (Okta, Auth0, Keycloak) | When you have a proper OIDC provider and want simpler setup |

---

### Setting Up SSO with GitHub (via Dex)

This is one of the most common setups. Let's walk through it step by step.

#### Step 1: Create a GitHub OAuth App

1. Go to GitHub → Settings → Developer Settings → OAuth Apps → New OAuth App
2. Fill in:
   - **Application name:** ArgoCD
   - **Homepage URL:** `https://argocd.example.com`
   - **Authorization callback URL:** `https://argocd.example.com/api/dex/callback`
3. Note down the **Client ID** and **Client Secret**

#### Step 2: Configure ArgoCD ConfigMap

Edit the `argocd-cm` ConfigMap:

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: argocd-cm
  namespace: argocd
data:
  # The external URL of ArgoCD
  url: https://argocd.example.com

  # Dex configuration
  dex.config: |
    connectors:
      - type: github
        id: github
        name: GitHub
        config:
          clientID: YOUR_GITHUB_CLIENT_ID
          clientSecret: $dex.github.clientSecret    # Reference to a secret
          orgs:
            - name: myorg                            # Only allow users from this org
              teams:
                - devops                             # Optional: restrict to specific teams
                - platform
```

#### Step 3: Store the Client Secret

Add the secret to `argocd-secret`:

```bash
kubectl -n argocd patch secret argocd-secret \
  --type merge \
  -p '{"stringData": {"dex.github.clientSecret": "YOUR_GITHUB_CLIENT_SECRET"}}'
```

#### Step 4: Restart Dex

```bash
kubectl -n argocd rollout restart deployment argocd-dex-server
```

Now when users visit ArgoCD, they'll see a "Login with GitHub" button! 🎉

---

### Setting Up SSO with Google (via Dex)

```yaml
# argocd-cm ConfigMap
data:
  url: https://argocd.example.com
  dex.config: |
    connectors:
      - type: oidc
        id: google
        name: Google
        config:
          issuer: https://accounts.google.com
          clientID: YOUR_GOOGLE_CLIENT_ID
          clientSecret: $dex.google.clientSecret
          hostedDomains:
            - mycompany.com             # Only allow @mycompany.com emails
```

---

### Setting Up SSO with Okta (Direct OIDC)

For direct OIDC (without Dex):

```yaml
# argocd-cm ConfigMap
data:
  url: https://argocd.example.com
  oidc.config: |
    name: Okta
    issuer: https://mycompany.okta.com/oauth2/default
    clientID: YOUR_OKTA_CLIENT_ID
    clientSecret: $oidc.okta.clientSecret
    requestedScopes:
      - openid
      - profile
      - email
      - groups
    requestedIDTokenClaims:
      groups:
        essential: true
```

---

### Setting Up SSO with Keycloak

```yaml
# argocd-cm ConfigMap
data:
  url: https://argocd.example.com
  oidc.config: |
    name: Keycloak
    issuer: https://keycloak.example.com/realms/myrealm
    clientID: argocd
    clientSecret: $oidc.keycloak.clientSecret
    requestedScopes:
      - openid
      - profile
      - email
      - groups
```

### Disabling the Admin Account

Once SSO is working, you should disable the built-in admin account:

```yaml
# argocd-cm ConfigMap
data:
  admin.enabled: "false"     # ← Disables the admin user
```

## 🛡️ Part 3: RBAC — Role-Based Access Control

### Why RBAC?

SSO tells ArgoCD **WHO you are**. RBAC tells ArgoCD **WHAT you can do**.

Without RBAC, anyone who logs in can do everything — create apps, delete apps, sync to production. That's dangerous!

With RBAC, you can say:
- "Developers can **view** apps but not **sync** to production"
- "The platform team can **manage** all apps"
- "Interns can only **view** the dev environment"

---

### How ArgoCD RBAC Works

ArgoCD uses a **Casbin-based** policy system. Policies are defined in the `argocd-rbac-cm` ConfigMap.

The format is:

```
p, <subject>, <resource>, <action>, <object>, <effect>
```

Where:
- **subject** — who (user, group, or role)
- **resource** — what type of resource (applications, clusters, projects, repositories, etc.)
- **action** — what they can do (get, create, update, delete, sync, override, action)
- **object** — which specific resources (project/app name, or `*` for all)
- **effect** — allow or deny

---

### Built-in Roles

ArgoCD has two built-in roles:

| Role | Permissions |
|------|------------|
| `role:readonly` | Can view everything but change nothing |
| `role:admin` | Full access to everything |

---

### Configuring RBAC

Edit the `argocd-rbac-cm` ConfigMap:

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: argocd-rbac-cm
  namespace: argocd
data:
  # Default policy for authenticated users (if no other policy matches)
  policy.default: role:readonly

  # CSV-formatted policy rules
  policy.csv: |
    # ===== CUSTOM ROLES =====

    # Role: developer — can view and sync apps in dev project
    p, role:developer, applications, get, dev/*, allow
    p, role:developer, applications, sync, dev/*, allow
    p, role:developer, applications, action/*, dev/*, allow
    p, role:developer, logs, get, dev/*, allow

    # Role: senior-dev — can view and sync apps in dev AND staging
    p, role:senior-dev, applications, get, dev/*, allow
    p, role:senior-dev, applications, sync, dev/*, allow
    p, role:senior-dev, applications, get, staging/*, allow
    p, role:senior-dev, applications, sync, staging/*, allow
    p, role:senior-dev, logs, get, */*, allow

    # Role: ops — can do everything in all projects
    p, role:ops, applications, *, */*, allow
    p, role:ops, clusters, *, *, allow
    p, role:ops, repositories, *, *, allow
    p, role:ops, projects, *, *, allow
    p, role:ops, logs, get, */*, allow
    p, role:ops, exec, create, */*, allow

    # ===== MAP SSO GROUPS TO ROLES =====

    # GitHub team "myorg:devops" gets the ops role
    g, myorg:devops, role:ops

    # GitHub team "myorg:developers" gets the developer role
    g, myorg:developers, role:developer

    # GitHub team "myorg:senior-devs" gets the senior-dev role
    g, myorg:senior-devs, role:senior-dev

    # Specific user gets admin
    g, john@mycompany.com, role:admin

  # How to match SSO groups to ArgoCD subjects
  # For GitHub: "org:team" format
  # For OIDC/Okta: the group name from the groups claim
  scopes: '[groups, email]'
```

---

### Understanding the Policy Format

Let's break down a policy line:

```
p, role:developer, applications, sync, dev/*, allow
│  │               │              │     │      │
│  │               │              │     │      └── Effect: allow or deny
│  │               │              │     └── Object: project/app (* = all)
│  │               │              └── Action: what they can do
│  │               └── Resource type
│  └── Subject (role name)
└── "p" = policy rule
```

And a group mapping:

```
g, myorg:devops, role:ops
│  │              │
│  │              └── ArgoCD role to assign
│  └── SSO group name
└── "g" = group mapping
```

---

### Available Resources and Actions

| Resource | Available Actions |
|----------|------------------|
| `applications` | `get`, `create`, `update`, `delete`, `sync`, `override`, `action/*` |
| `clusters` | `get`, `create`, `update`, `delete` |
| `projects` | `get`, `create`, `update`, `delete` |
| `repositories` | `get`, `create`, `update`, `delete` |
| `accounts` | `get`, `update` |
| `gpgkeys` | `get`, `create`, `delete` |
| `logs` | `get` |
| `exec` | `create` (allows `kubectl exec` into pods) |

---

### RBAC Examples for Common Scenarios

#### Scenario 1: Developers Can Only View

```csv
policy.default: role:readonly
# That's it! All authenticated users can only view.
```

#### Scenario 2: Team-Based Access

```csv
# Frontend team can manage frontend apps
p, role:frontend, applications, *, default/frontend-*, allow
g, myorg:frontend-team, role:frontend

# Backend team can manage backend apps  
p, role:backend, applications, *, default/backend-*, allow
g, myorg:backend-team, role:backend
```

#### Scenario 3: Environment-Based Access

```csv
# Anyone can sync in dev
p, role:developer, applications, sync, dev/*, allow

# Only seniors can sync in staging
p, role:senior, applications, sync, staging/*, allow

# Only ops can sync in prod
p, role:ops, applications, sync, prod/*, allow
```

#### Scenario 4: Read-Only for External Teams

```csv
# External team can only view specific apps
p, role:external, applications, get, default/public-api, allow
p, role:external, applications, get, default/docs-site, allow
g, external-team@partner.com, role:external
```

---

### Testing RBAC Policies

You can test if a user/group can perform an action:

```bash
# Check if a user can sync an app
argocd admin settings rbac can role:developer sync applications dev/user-service

# Output: Yes  (or No)

# Check what a user can do
argocd admin settings rbac can role:developer get applications '*/*'

# Validate the entire policy file
argocd admin settings rbac validate --policy-file policy.csv
```

## 🔔 Part 4: Notifications

### Why Notifications?

You want to know when things happen in ArgoCD:
- ✅ App successfully synced
- ❌ Sync failed
- ⚠️ App health is degraded
- 🔄 New deployment started

ArgoCD Notifications sends alerts to **Slack**, **email**, **Microsoft Teams**, **webhooks**, **GitHub**, **Grafana**, and more.

---

### Architecture

ArgoCD Notifications is **built into ArgoCD** (since v2.4). It runs as part of the `argocd-notifications-controller`.

```
ArgoCD Application event → Notifications Controller → 
→ Evaluates triggers → Matches template → Sends to service (Slack, email, etc.)
```

Three concepts:
1. **Services** — WHERE to send (Slack, email, webhook)
2. **Templates** — WHAT to send (message content)
3. **Triggers** — WHEN to send (on sync success, on health degradation, etc.)

---

### Setting Up Slack Notifications

This is the most common setup. Let's do it step by step.

#### Step 1: Create a Slack Bot

1. Go to [api.slack.com/apps](https://api.slack.com/apps) → Create New App
2. Go to **OAuth & Permissions** → Add Bot Token Scope: `chat:write`
3. **Install to workspace** → Copy the **Bot User OAuth Token** (starts with `xoxb-`)
4. **Invite the bot** to your Slack channel: `/invite @ArgoCD-Bot`

#### Step 2: Configure the Notification Service

Edit the `argocd-notifications-cm` ConfigMap:

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: argocd-notifications-cm
  namespace: argocd
data:
  # ===== SERVICES — Where to send notifications =====
  service.slack: |
    token: $slack-token           # References argocd-notifications-secret

  # ===== TEMPLATES — What to send =====
  template.app-sync-succeeded: |
    slack:
      attachments: |
        [{
          "title": "✅ {{.app.metadata.name}} synced successfully!",
          "color": "#18be52",
          "fields": [{
            "title": "Application",
            "value": "{{.app.metadata.name}}",
            "short": true
          }, {
            "title": "Project",
            "value": "{{.app.spec.project}}",
            "short": true
          }, {
            "title": "Revision",
            "value": "{{.app.status.sync.revision}}",
            "short": true
          }, {
            "title": "Environment",
            "value": "{{index .app.metadata.labels \"environment\"}}",
            "short": true
          }]
        }]

  template.app-sync-failed: |
    slack:
      attachments: |
        [{
          "title": "❌ {{.app.metadata.name}} sync FAILED!",
          "color": "#E96D76",
          "fields": [{
            "title": "Application",
            "value": "{{.app.metadata.name}}",
            "short": true
          }, {
            "title": "Error",
            "value": "{{.app.status.operationState.message}}",
            "short": false
          }]
        }]

  template.app-health-degraded: |
    slack:
      attachments: |
        [{
          "title": "⚠️ {{.app.metadata.name}} health is DEGRADED!",
          "color": "#f4c030",
          "fields": [{
            "title": "Application",
            "value": "{{.app.metadata.name}}",
            "short": true
          }, {
            "title": "Health Status",
            "value": "{{.app.status.health.status}}",
            "short": true
          }]
        }]

  # ===== TRIGGERS — When to send =====
  trigger.on-sync-succeeded: |
    - when: app.status.operationState.phase in ['Succeeded']
      send: [app-sync-succeeded]

  trigger.on-sync-failed: |
    - when: app.status.operationState.phase in ['Error', 'Failed']
      send: [app-sync-failed]

  trigger.on-health-degraded: |
    - when: app.status.health.status == 'Degraded'
      send: [app-health-degraded]

  trigger.on-deployed: |
    - when: app.status.operationState.phase in ['Succeeded'] and app.status.health.status == 'Healthy'
      send: [app-sync-succeeded]
```

#### Step 3: Store the Slack Token

```bash
kubectl -n argocd patch secret argocd-notifications-secret \
  --type merge \
  -p '{"stringData": {"slack-token": "xoxb-your-slack-bot-token"}}'
```

#### Step 4: Subscribe Applications to Notifications

Add an annotation to your Application to subscribe to notifications:

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: user-service
  namespace: argocd
  annotations:
    # Subscribe to triggers and send to a Slack channel
    notifications.argoproj.io/subscribe.on-sync-succeeded.slack: deployments
    notifications.argoproj.io/subscribe.on-sync-failed.slack: deployments-alerts
    notifications.argoproj.io/subscribe.on-health-degraded.slack: deployments-alerts
spec:
  # ... rest of the spec
```

The annotation format is:

```
notifications.argoproj.io/subscribe.<trigger>.<service>: <target>
                                     │           │         │
                                     │           │         └── Slack channel name
                                     │           └── Service type (slack, email, etc.)
                                     └── Trigger name (on-sync-succeeded, etc.)
```

```yaml
# argocd-notifications-cm
data:
  service.webhook.custom: |
    url: https://api.mycompany.com/deploy-events
    headers:
      - name: Authorization
        value: Bearer $webhook-token
      - name: Content-Type
        value: application/json

  template.webhook-payload: |
    webhook:
      custom:
        method: POST
        body: |
          {
            "app": "{{.app.metadata.name}}",
            "project": "{{.app.spec.project}}",
            "status": "{{.app.status.operationState.phase}}",
            "health": "{{.app.status.health.status}}",
            "revision": "{{.app.status.sync.revision}}",
            "timestamp": "{{.app.status.operationState.finishedAt}}"
          }
```

### Default Triggers (Built-in)

ArgoCD comes with many **built-in triggers** you can use right away:

| Trigger | When It Fires |
|---------|--------------|
| `on-created` | Application is created |
| `on-deleted` | Application is deleted |
| `on-deployed` | Application is synced and healthy |
| `on-health-degraded` | App health becomes Degraded |
| `on-sync-failed` | Sync operation fails |
| `on-sync-running` | Sync operation starts |
| `on-sync-status-unknown` | Sync status is unknown |
| `on-sync-succeeded` | Sync operation succeeds |

### Subscribing at the ApplicationSet Level

Instead of annotating each Application individually, you can subscribe at the **ApplicationSet** level:

```yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: microservices
  namespace: argocd
spec:
  generators:
    - git:
        repoURL: https://github.com/myorg/k8s-manifests.git
        revision: main
        directories:
          - path: apps/*
  template:
    metadata:
      name: "{{path.basename}}"
      annotations:
        # ALL generated apps will have these notification subscriptions
        notifications.argoproj.io/subscribe.on-sync-failed.slack: devops-alerts
        notifications.argoproj.io/subscribe.on-health-degraded.slack: devops-alerts
        notifications.argoproj.io/subscribe.on-deployed.slack: deployments
    spec:
      # ... rest of spec
```

## 🧠 Quick Recap

| Topic | Key Takeaway |
|-------|-------------|
| **Multi-Cluster** | ArgoCD runs on one cluster, deploys to many. Use `argocd cluster add` or declarative secrets. |
| **SSO** | Delegate login to GitHub/Google/Okta/Keycloak via Dex or direct OIDC. Disable admin after setup. |
| **RBAC** | Define who can do what using Casbin policies in `argocd-rbac-cm`. Map SSO groups to roles. |
| **Notifications** | Configure services (Slack, email), templates (message content), triggers (when to send). Subscribe via annotations. |
