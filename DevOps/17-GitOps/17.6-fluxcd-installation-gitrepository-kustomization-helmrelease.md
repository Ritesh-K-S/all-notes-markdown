# 17.6 — Flux CD — Installation, GitRepository, Kustomization & HelmRelease

> **One Line Answer:** **Flux CD** is a GitOps tool (just like ArgoCD) that runs inside your Kubernetes cluster, watches your Git repositories, and **automatically syncs** your cluster to match what's in Git — but unlike ArgoCD, it's **completely CLI-driven** (no UI), uses **native Kubernetes CRDs** for everything, and follows a **modular, controller-based architecture**.

---

## 🤔 What is Flux CD?

Flux CD (usually just called **Flux**) is a **GitOps toolkit** for Kubernetes. It was created by **Weaveworks** and is a **CNCF Graduated project** (same level as Kubernetes, Prometheus — meaning it's production-grade and battle-tested).

Think of it this way:

> **ArgoCD** = GitOps with a shiny UI, centralized management  
> **Flux CD** = GitOps with pure Kubernetes CRDs, decentralized, CLI-first

Both do the same core job — **keep your Kubernetes cluster in sync with Git** — but they take very different approaches.

---

### Flux vs ArgoCD — Quick Comparison

| Feature | ArgoCD | Flux CD |
|---------|--------|---------|
| **UI** | Beautiful web UI | No built-in UI (CLI + optional Weave GitOps UI) |
| **Architecture** | Centralized server | Distributed controllers |
| **Configuration** | Application CRD | Multiple CRDs (GitRepository, Kustomization, HelmRelease) |
| **Multi-tenancy** | Projects + RBAC | Namespaced CRDs + Kubernetes RBAC |
| **Helm support** | Built-in | HelmRelease CRD (Helm Controller) |
| **Kustomize support** | Built-in | Kustomization CRD (Kustomize Controller) |
| **Image automation** | Not built-in | Built-in (Image Reflector + Image Automation controllers) |
| **Learning curve** | Lower (UI helps) | Higher (all YAML, no UI) |
| **Philosophy** | "One tool does everything" | "Each controller has one job" |

---

## 🏗️ Flux Architecture — How It Works

Flux is made up of **specialized controllers**, each doing one thing well:

```
┌─────────────────────────────────────────────────────────────┐
│                    Kubernetes Cluster                         │
│                                                              │
│  ┌─────────────────┐  ┌─────────────────┐                  │
│  │ Source Controller│  │Kustomize        │                  │
│  │                  │  │Controller       │                  │
│  │ • Fetches Git    │  │                  │                  │
│  │   repos          │──▶│ • Applies K8s   │                  │
│  │ • Fetches Helm   │  │   manifests     │                  │
│  │   charts         │  │ • Runs          │                  │
│  │ • Fetches OCI    │  │   Kustomize     │                  │
│  │   artifacts      │  └─────────────────┘                  │
│  └─────────────────┘                                        │
│                                                              │
│  ┌─────────────────┐  ┌─────────────────┐                  │
│  │ Helm Controller │  │Notification     │                  │
│  │                  │  │Controller       │                  │
│  │ • Installs Helm │  │                  │                  │
│  │   charts         │  │ • Sends alerts  │                  │
│  │ • Manages        │  │   to Slack,     │                  │
│  │   releases       │  │   Teams, etc.   │                  │
│  └─────────────────┘  └─────────────────┘                  │
│                                                              │
│  ┌─────────────────┐  ┌─────────────────┐                  │
│  │Image Reflector  │  │Image Automation │                  │
│  │Controller       │  │Controller       │                  │
│  │                  │  │                  │                  │
│  │ • Scans container│  │ • Updates YAML  │                  │
│  │   registries     │  │   with new image│                  │
│  │   for new tags   │  │   tags          │                  │
│  └─────────────────┘  └─────────────────┘                  │
└─────────────────────────────────────────────────────────────┘
```

Each controller has its own **CRD** (Custom Resource Definition):

| Controller | CRD | What It Does |
|-----------|-----|-------------|
| **Source Controller** | `GitRepository`, `HelmRepository`, `OCIRepository`, `Bucket` | Fetches source code/configs from Git, Helm repos, OCI, S3 |
| **Kustomize Controller** | `Kustomization` | Applies Kubernetes manifests (plain YAML or Kustomize overlays) |
| **Helm Controller** | `HelmRelease` | Installs/upgrades Helm charts |
| **Notification Controller** | `Alert`, `Provider`, `Receiver` | Sends notifications and receives webhooks |
| **Image Reflector Controller** | `ImageRepository`, `ImagePolicy` | Scans container registries for new tags |
| **Image Automation Controller** | `ImageUpdateAutomation` | Updates YAML files with new image tags |

## 📦 Installing Flux CD

### Prerequisites

- A Kubernetes cluster (Minikube, Kind, EKS, AKS, GKE — anything works)
- `kubectl` configured to talk to your cluster
- A GitHub (or GitLab/Bitbucket) account
- A **Personal Access Token** (PAT) with repo permissions

### Step 1: Install the Flux CLI

**macOS:**
```bash
brew install fluxcd/tap/flux
```

**Linux:**
```bash
curl -s https://fluxcd.io/install.sh | sudo bash
```

**Windows (Chocolatey):**
```powershell
choco install flux
```

**Verify installation:**
```bash
flux --version
# flux version 2.x.x
```

### Step 2: Check Prerequisites

Before bootstrapping, check if your cluster meets the requirements:

```bash
flux check --pre

# Output:
# ► checking prerequisites
# ✔ Kubernetes 1.28.0 >=1.25.0-0
# ✔ prerequisites checks passed
```

### Step 3: Bootstrap Flux

The **bootstrap** command does everything:
1. Creates the `flux-system` namespace
2. Deploys all Flux controllers
3. Creates a Git repository for Flux to manage itself
4. Commits the Flux manifests to your Git repo (so Flux manages itself via GitOps!)

**Bootstrap with GitHub:**

```bash
export GITHUB_TOKEN=<your-personal-access-token>

flux bootstrap github \
  --owner=myorg \                      # GitHub org or username
  --repository=fleet-infra \           # Repo name (will be created if doesn't exist)
  --branch=main \
  --path=clusters/my-cluster \         # Path within the repo for this cluster
  --personal                           # Use personal account (omit for org)
```

**What happens behind the scenes:**

1. Flux creates (or uses) a GitHub repo called `fleet-infra`
2. It creates a directory `clusters/my-cluster/` with Flux component manifests
3. It deploys itself to the `flux-system` namespace
4. It starts watching the repo for changes

**Your repo after bootstrap:**

```
fleet-infra/
└── clusters/
    └── my-cluster/
        └── flux-system/
            ├── gotk-components.yaml     # All Flux controllers (auto-generated)
            ├── gotk-sync.yaml           # Flux watches this repo (self-management)
            └── kustomization.yaml       # Kustomize overlay for Flux components
```

**Bootstrap with GitLab:**

```bash
export GITLAB_TOKEN=<your-gitlab-token>

flux bootstrap gitlab \
  --owner=myorg \
  --repository=fleet-infra \
  --branch=main \
  --path=clusters/my-cluster
```

### Step 4: Verify Flux is Running

```bash
# Check all Flux components
flux check

# Output:
# ► checking prerequisites
# ✔ Kubernetes 1.28.0 >=1.25.0-0
# ► checking controllers
# ✔ helm-controller: deployment ready
# ✔ kustomize-controller: deployment ready
# ✔ notification-controller: deployment ready
# ✔ source-controller: deployment ready
# ✔ all checks passed

# See all Flux controllers running
kubectl get pods -n flux-system

# Output:
# NAME                                       READY   STATUS    RESTARTS   AGE
# helm-controller-xxx                        1/1     Running   0          2m
# kustomize-controller-xxx                   1/1     Running   0          2m
# notification-controller-xxx                1/1     Running   0          2m
# source-controller-xxx                      1/1     Running   0          2m
```

## 📂 GitRepository — Telling Flux Where Your Code Is

### What Is a GitRepository?

A `GitRepository` is a Flux CRD that tells the **Source Controller** to **fetch** content from a Git repository at regular intervals.

Think of it as: *"Hey Flux, here's a Git repo. Go check it every minute and tell me if anything changed."*

### Basic GitRepository

```yaml
apiVersion: source.toolkit.fluxcd.io/v1
kind: GitRepository
metadata:
  name: my-app-repo
  namespace: flux-system
spec:
  interval: 1m                              # Check for changes every minute
  url: https://github.com/myorg/k8s-manifests.git
  ref:
    branch: main                             # Which branch to track
```

### GitRepository with Authentication

For **private repos**, you need to provide credentials.

**Using HTTPS + Token:**

```bash
# First, create a secret with your Git credentials
flux create secret git my-app-repo-auth \
  --url=https://github.com/myorg/k8s-manifests.git \
  --username=git \
  --password=$GITHUB_TOKEN
```

This creates a Kubernetes Secret. Then reference it:

```yaml
apiVersion: source.toolkit.fluxcd.io/v1
kind: GitRepository
metadata:
  name: my-app-repo
  namespace: flux-system
spec:
  interval: 1m
  url: https://github.com/myorg/k8s-manifests.git
  ref:
    branch: main
  secretRef:
    name: my-app-repo-auth                  # ← Reference the secret
```

**Using SSH:**

```bash
# Generate an SSH key pair and add public key to GitHub
flux create secret git my-app-repo-ssh \
  --url=ssh://git@github.com/myorg/k8s-manifests.git \
  --ssh-key-algorithm=ecdsa \
  --ssh-ecdsa-curve=p521
```

```yaml
apiVersion: source.toolkit.fluxcd.io/v1
kind: GitRepository
metadata:
  name: my-app-repo
  namespace: flux-system
spec:
  interval: 1m
  url: ssh://git@github.com/myorg/k8s-manifests.git
  ref:
    branch: main
  secretRef:
    name: my-app-repo-ssh
```

### Tracking Different References

```yaml
# Track a specific branch
ref:
  branch: main

# Track a specific tag
ref:
  tag: v1.2.3

# Track a semver range (latest matching tag)
ref:
  semver: ">=1.0.0 <2.0.0"

# Track a specific commit
ref:
  commit: abc123def456
```

### GitRepository with Path Filtering

Only trigger when specific paths change:

```yaml
spec:
  interval: 1m
  url: https://github.com/myorg/k8s-manifests.git
  ref:
    branch: main
  include:
    - path: "apps/**"                        # Only watch the apps/ directory
  ignore: |
    # Ignore these files
    docs/
    README.md
    *.txt
```

### Checking GitRepository Status

```bash
# Using Flux CLI
flux get sources git

# Output:
# NAME           REVISION       SUSPENDED  READY  MESSAGE
# my-app-repo    main@sha1:abc  False      True   stored artifact for revision 'main@sha1:abc'
# fleet-infra    main@sha1:def  False      True   stored artifact for revision 'main@sha1:def'

# Using kubectl
kubectl get gitrepository -n flux-system

# Force a reconciliation (don't wait for the interval)
flux reconcile source git my-app-repo
```

## 📋 Kustomization — Applying Manifests to the Cluster

### What Is a Flux Kustomization?

> ⚠️ **Important:** Flux's `Kustomization` CRD is **NOT the same** as Kubernetes Kustomize's `kustomization.yaml`! They have similar names but are different things.

| Term | What It Is |
|------|-----------|
| **Kustomize** (`kustomization.yaml`) | A Kubernetes tool for customizing YAML manifests (overlays, patches) |
| **Flux Kustomization** (CRD) | A Flux resource that tells the Kustomize Controller to **apply** manifests from a source |

A Flux `Kustomization` CRD says: *"Take the content from this GitRepository, look at this path, and apply it to the cluster."*

It works with both:
- **Plain YAML manifests** (no kustomization.yaml needed!)
- **Kustomize overlays** (if a kustomization.yaml exists in the path)

---

### Basic Kustomization — Plain YAML

Let's say your Git repo has this structure:

```
k8s-manifests/
├── apps/
│   └── user-service/
│       ├── deployment.yaml
│       ├── service.yaml
│       └── configmap.yaml
```

**Step 1: Create a GitRepository (if not already created)**

```yaml
apiVersion: source.toolkit.fluxcd.io/v1
kind: GitRepository
metadata:
  name: my-app-repo
  namespace: flux-system
spec:
  interval: 5m
  url: https://github.com/myorg/k8s-manifests.git
  ref:
    branch: main
```

**Step 2: Create a Kustomization to apply the manifests**

```yaml
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: user-service
  namespace: flux-system
spec:
  interval: 10m                          # Reconcile every 10 minutes
  sourceRef:
    kind: GitRepository
    name: my-app-repo                    # ← Points to the GitRepository above
  path: ./apps/user-service              # ← Which directory to apply
  prune: true                            # Delete resources removed from Git
  targetNamespace: user-service          # Deploy to this namespace
```

**What happens:**
1. Source Controller fetches the Git repo
2. Kustomize Controller reads `apps/user-service/` from the fetched content
3. It applies all YAML files in that directory to the cluster
4. Every 10 minutes, it checks for drift and re-applies if needed

---

### Kustomization with Kustomize Overlays

If your directory has a `kustomization.yaml`, Flux will use it:

```
k8s-manifests/
├── base/
│   └── user-service/
│       ├── deployment.yaml
│       ├── service.yaml
│       └── kustomization.yaml           # ← Kustomize file
├── overlays/
│   ├── dev/
│   │   └── user-service/
│   │       ├── kustomization.yaml       # References base + adds dev patches
│   │       └── patch-replicas.yaml
│   └── prod/
│       └── user-service/
│           ├── kustomization.yaml       # References base + adds prod patches
│           └── patch-replicas.yaml
```

```yaml
# Flux Kustomization pointing to the dev overlay
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: user-service-dev
  namespace: flux-system
spec:
  interval: 10m
  sourceRef:
    kind: GitRepository
    name: my-app-repo
  path: ./overlays/dev/user-service      # ← Points to the overlay
  prune: true
  targetNamespace: user-service-dev
```

---

### Kustomization with Variable Substitution

Flux can substitute variables in your manifests, like a lightweight templating engine:

```yaml
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: user-service
  namespace: flux-system
spec:
  interval: 10m
  sourceRef:
    kind: GitRepository
    name: my-app-repo
  path: ./apps/user-service
  prune: true

  # Inline variable substitution
  postBuild:
    substitute:
      ENVIRONMENT: "production"
      REPLICAS: "3"
      IMAGE_TAG: "v1.2.3"

    # Or from ConfigMaps/Secrets
    substituteFrom:
      - kind: ConfigMap
        name: cluster-config
      - kind: Secret
        name: cluster-secrets
```

In your YAML manifests, use `${VARIABLE}` syntax:

```yaml
# deployment.yaml — uses Flux variable substitution
apiVersion: apps/v1
kind: Deployment
metadata:
  name: user-service
  labels:
    environment: ${ENVIRONMENT}
spec:
  replicas: ${REPLICAS}
  template:
    spec:
      containers:
        - name: user-service
          image: myorg/user-service:${IMAGE_TAG}
```

Flux will replace `${ENVIRONMENT}`, `${REPLICAS}`, and `${IMAGE_TAG}` before applying.

---

### Kustomization Dependencies

You can define **dependencies** between Kustomizations. For example, install the namespace and CRDs before the applications:

```yaml
# Step 1: Apply infrastructure (namespaces, CRDs)
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: infrastructure
  namespace: flux-system
spec:
  interval: 10m
  sourceRef:
    kind: GitRepository
    name: my-app-repo
  path: ./infrastructure
  prune: true

---
# Step 2: Apply apps (depends on infrastructure)
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: apps
  namespace: flux-system
spec:
  interval: 10m
  sourceRef:
    kind: GitRepository
    name: my-app-repo
  path: ./apps
  prune: true
  dependsOn:
    - name: infrastructure              # ← Wait for infrastructure to be ready!
```

This ensures the `infrastructure` Kustomization is healthy before `apps` is applied. Super useful for:
- Installing CRDs before resources that use them
- Creating namespaces before deploying apps into them
- Setting up ingress controllers before creating Ingress resources

---

### Health Checks

Flux can wait for resources to be healthy after applying them:

```yaml
spec:
  interval: 10m
  sourceRef:
    kind: GitRepository
    name: my-app-repo
  path: ./apps/user-service
  prune: true
  wait: true                             # ← Wait for all resources to be ready
  timeout: 5m                            # ← Timeout after 5 minutes

  # Or define specific health checks
  healthChecks:
    - apiVersion: apps/v1
      kind: Deployment
      name: user-service
      namespace: user-service
```

## ⎈ HelmRelease — Managing Helm Charts with Flux

### What Is a HelmRelease?

A `HelmRelease` tells the **Helm Controller** to install or upgrade a Helm chart. Instead of running `helm install` manually, you define a HelmRelease CRD, and Flux manages the Helm lifecycle for you via GitOps.

Think of it as: *"I want this Helm chart installed with these values. Flux, make it happen and keep it in sync."*

---

### HelmRepository — Where to Find Charts

First, you need to tell Flux where to find Helm charts. A `HelmRepository` is like adding a Helm repo (`helm repo add`):

```yaml
apiVersion: source.toolkit.fluxcd.io/v1
kind: HelmRepository
metadata:
  name: bitnami
  namespace: flux-system
spec:
  interval: 1h                           # Check for new chart versions every hour
  url: https://charts.bitnami.com/bitnami
```

**Other repository types:**

```yaml
# OCI-based Helm repository (e.g., ECR, Docker Hub, GHCR)
apiVersion: source.toolkit.fluxcd.io/v1
kind: HelmRepository
metadata:
  name: my-oci-repo
  namespace: flux-system
spec:
  type: oci
  interval: 1h
  url: oci://ghcr.io/myorg/charts

---
# Helm repo with authentication
apiVersion: source.toolkit.fluxcd.io/v1
kind: HelmRepository
metadata:
  name: private-charts
  namespace: flux-system
spec:
  interval: 1h
  url: https://charts.mycompany.com
  secretRef:
    name: helm-repo-credentials          # Secret with username/password
```

---

### Basic HelmRelease

Let's install **nginx** using a Helm chart:

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: nginx
  namespace: flux-system
spec:
  interval: 10m                          # Reconcile every 10 minutes
  chart:
    spec:
      chart: nginx                       # Chart name
      version: "15.x"                    # Semver range — auto-updates within 15.x
      sourceRef:
        kind: HelmRepository
        name: bitnami                    # ← References the HelmRepository above
      interval: 1h                       # Check for new versions hourly
  values:                                # ← Same as Helm values.yaml
    replicaCount: 3
    service:
      type: LoadBalancer
    resources:
      requests:
        memory: 128Mi
        cpu: 100m
      limits:
        memory: 256Mi
        cpu: 200m
```

**What this does:**
1. Helm Controller finds the `nginx` chart from the `bitnami` HelmRepository
2. It installs (or upgrades) the chart with the specified values
3. Every 10 minutes, it checks if the release matches the desired state
4. If someone changes the release manually (drift), Flux reverts it

---

### HelmRelease from a Git Repository

Instead of a Helm repository, you can install a chart directly from your **Git repo**:

```
k8s-manifests/
└── charts/
    └── my-custom-app/
        ├── Chart.yaml
        ├── values.yaml
        └── templates/
            ├── deployment.yaml
            └── service.yaml
```

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-custom-app
  namespace: flux-system
spec:
  interval: 10m
  chart:
    spec:
      chart: ./charts/my-custom-app      # ← Path within the Git repo
      sourceRef:
        kind: GitRepository
        name: my-app-repo               # ← References a GitRepository
      interval: 5m
  values:
    replicaCount: 2
    image:
      repository: myorg/my-app
      tag: v1.0.0
```

---

### Overriding Values from ConfigMaps and Secrets

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: flux-system
spec:
  interval: 10m
  chart:
    spec:
      chart: my-app
      sourceRef:
        kind: HelmRepository
        name: my-charts

  # Values from multiple sources (merged in order)
  valuesFrom:
    - kind: ConfigMap
      name: my-app-base-values          # ← Base values from a ConfigMap
    - kind: Secret
      name: my-app-secret-values        # ← Sensitive values from a Secret
      valuesKey: values.yaml             # ← Key within the Secret

  # Inline values (highest priority — overrides valuesFrom)
  values:
    replicaCount: 3
    image:
      tag: v1.2.3
```

**Merge order (last wins):**
1. Chart's default `values.yaml`
2. `valuesFrom` entries (in order)
3. Inline `values`

---

### HelmRelease with Upgrade and Rollback Policies

```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: my-app
  namespace: flux-system
spec:
  interval: 10m
  chart:
    spec:
      chart: my-app
      sourceRef:
        kind: HelmRepository
        name: my-charts

  # Install configuration
  install:
    createNamespace: true               # Create namespace if it doesn't exist
    remediation:
      retries: 3                        # Retry installation 3 times on failure

  # Upgrade configuration
  upgrade:
    cleanupOnFail: true                 # Delete new resources on failed upgrade
    remediation:
      retries: 3                        # Retry upgrade 3 times
      remediateLastFailure: true        # Remediate the last failed attempt
      strategy: rollback                # Rollback on failure!

  # Rollback configuration
  rollback:
    timeout: 5m
    cleanupOnFail: true

  # Uninstall configuration
  uninstall:
    keepHistory: false                  # Clean up Helm release history

  values:
    replicaCount: 3
```

This means: If an upgrade fails 3 times, **automatically rollback** to the previous working version. Super safe!

---

### HelmRelease with Test Hooks

```yaml
spec:
  test:
    enable: true                         # Run Helm test hooks after install/upgrade
    timeout: 5m
    ignoreFailures: false                # Fail the release if tests fail
```

### Real-World Example: Installing Prometheus + Grafana

```yaml
# Step 1: Add the prometheus-community Helm repo
apiVersion: source.toolkit.fluxcd.io/v1
kind: HelmRepository
metadata:
  name: prometheus-community
  namespace: flux-system
spec:
  interval: 1h
  url: https://prometheus-community.github.io/helm-charts

---
# Step 2: Install kube-prometheus-stack
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: kube-prometheus-stack
  namespace: monitoring
spec:
  interval: 30m
  chart:
    spec:
      chart: kube-prometheus-stack
      version: "55.x"
      sourceRef:
        kind: HelmRepository
        name: prometheus-community
        namespace: flux-system           # Cross-namespace reference
      interval: 1h
  install:
    createNamespace: true
    crds: CreateReplace                  # Install CRDs
    remediation:
      retries: 3
  upgrade:
    crds: CreateReplace
    remediation:
      retries: 3
      strategy: rollback
  values:
    grafana:
      adminPassword: "admin"             # In real setup, use a Secret!
      ingress:
        enabled: true
        hosts:
          - grafana.example.com
    prometheus:
      prometheusSpec:
        retention: 30d
        storageSpec:
          volumeClaimTemplate:
            spec:
              accessModes: ["ReadWriteOnce"]
              resources:
                requests:
                  storage: 50Gi
    alertmanager:
      alertmanagerSpec:
        retention: 720h
```

```bash
# Output:

# monitoring  kube-prometheus-stack   55.1.0    False      True   Helm install succeeded
# flux-system nginx                   15.3.0    False      True   Helm upgrade succeeded

# Get detailed status
flux get helmrelease kube-prometheus-stack -n monitoring

# Force a reconciliation
flux reconcile helmrelease kube-prometheus-stack -n monitoring

# Suspend a HelmRelease
flux suspend helmrelease kube-prometheus-stack -n monitoring

# Resume
flux resume helmrelease kube-prometheus-stack -n monitoring
```

**`clusters/dev/infrastructure.yaml`** — Points Flux to the infrastructure directory:

```yaml
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: infrastructure
  namespace: flux-system
spec:
  interval: 10m
  sourceRef:
    kind: GitRepository
    name: flux-system                    # The bootstrap GitRepository
  path: ./infrastructure
  prune: true
  wait: true
```

**`clusters/dev/apps.yaml`** — Points Flux to the dev apps:

```yaml
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: apps
  namespace: flux-system
spec:
  interval: 10m
  sourceRef:
    kind: GitRepository
    name: flux-system
  path: ./apps/dev
  prune: true
  dependsOn:
    - name: infrastructure               # Wait for infra to be ready
```

## 🧰 Useful Flux CLI Commands

Here's a cheat sheet of the most commonly used Flux commands:

```bash
# ===== INSTALLATION & STATUS =====
flux check --pre                         # Check prerequisites
flux check                               # Check Flux health
flux get all                             # See everything Flux is managing
flux logs                                # View Flux controller logs

# ===== SOURCES =====
flux get sources git                     # List GitRepositories
flux get sources helm                    # List HelmRepositories
flux reconcile source git <name>         # Force re-fetch from Git

# ===== KUSTOMIZATIONS =====
flux get kustomizations                  # List all Kustomizations
flux reconcile kustomization <name>      # Force re-apply
flux suspend kustomization <name>        # Stop syncing
flux resume kustomization <name>         # Resume syncing

# ===== HELM RELEASES =====
flux get helmreleases -A                 # List all HelmReleases (all namespaces)
flux reconcile helmrelease <name>        # Force re-install/upgrade
flux suspend helmrelease <name>          # Stop managing
flux resume helmrelease <name>           # Resume managing

# ===== DEBUGGING =====
flux logs --level=error                  # Show only error logs
flux events                              # Show Flux events
flux trace kustomization <name>          # Trace a Kustomization's source chain

# ===== EXPORT (for backup/migration) =====
flux export source git --all             # Export all GitRepository definitions
flux export kustomization --all          # Export all Kustomization definitions
flux export helmrelease --all -A         # Export all HelmRelease definitions

# ===== UNINSTALL =====
flux uninstall                           # Remove Flux from the cluster
```

## 🔄 Complete Workflow — From Git to Cluster

Let's trace a complete deployment:

```
1. Developer pushes code change to Git repo
                    │
                    ▼
2. Source Controller detects new commit (GitRepository polls every 1m)
                    │
                    ▼
3. Source Controller downloads the new content
                    │
                    ▼
4. Kustomize Controller sees new content available
                    │
                    ▼
5. Kustomize Controller runs kustomize build (if kustomization.yaml exists)
   OR reads plain YAML manifests
                    │
                    ▼
6. Kustomize Controller applies manifests to the cluster (kubectl apply)
                    │
                    ▼
7. Kustomize Controller runs health checks (wait for Deployments to be ready)
                    │
                    ▼
8. Notification Controller sends success/failure alert to Slack
                    │
                    ▼
9. ✅ Deployment complete! Cluster matches Git.
```

For Helm-based deployments, the Helm Controller handles steps 5-7 instead of the Kustomize Controller.

## 🧠 Quick Recap

| Concept | What It Does |
|---------|-------------|
| **Flux CD** | A modular GitOps toolkit with specialized controllers for each task |
| **Bootstrap** | One command that installs Flux and sets up self-management via GitOps |
| **GitRepository** | CRD that tells Source Controller to fetch from a Git repo |
| **HelmRepository** | CRD that tells Source Controller where to find Helm charts |
| **Kustomization** | CRD that tells Kustomize Controller to apply manifests from a source |
| **HelmRelease** | CRD that tells Helm Controller to install/manage a Helm chart |
| **prune: true** | Deletes resources removed from Git (keeps cluster in sync) |
| **dependsOn** | Ensures one Kustomization is healthy before applying another |
| **valuesFrom** | Loads Helm values from ConfigMaps or Secrets |
| **Variable Substitution** | Replaces `${VAR}` in manifests with actual values |
