# 17.7 — Flux CD — Image Automation & Notifications

> **One Line Answer:** Flux's **Image Automation** watches your container registry for new image tags and **automatically updates** your Git repo with the new tag — closing the full GitOps loop. **Notifications** let Flux tell you (via Slack, Teams, webhooks, etc.) whenever something happens — a deployment succeeded, failed, or a new image was detected.

## 🤔 Why Do We Need Image Automation?

Let's think about the normal GitOps workflow:

```
1. Developer pushes code
2. CI pipeline builds a Docker image (e.g., myapp:v1.2.3)
3. CI pushes the image to a container registry (Docker Hub, ECR, etc.)
4. ??? Someone manually updates the Git repo YAML to point to the new image tag ???
5. Flux detects the Git change and deploys it to the cluster
```

Wait — step 4 is **manual**! That breaks the whole automation chain. You'd have to go to your Git repo, open the deployment YAML, change `image: myapp:v1.2.2` to `image: myapp:v1.2.3`, commit, push... every single time.

**Image Automation fixes this.** It makes step 4 fully automatic:

```
WITH Image Automation:

1. Developer pushes code
2. CI pipeline builds Docker image → myapp:v1.2.3
3. CI pushes image to registry
4. Flux Image Automation DETECTS the new tag automatically  ← 🤖
5. Flux UPDATES the Git repo YAML with the new tag          ← 🤖
6. Flux GitOps controller sees the Git change
7. Flux deploys the new version to the cluster

Result: ZERO manual steps from code push to production! 🎉
```

## 🏗️ Architecture — How Image Automation Works

Image Automation consists of **three components** that work together:

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Flux Image Automation                            │
│                                                                     │
│  ┌─────────────────────┐     ┌──────────────────────────┐          │
│  │  Image Repository   │     │  Image Policy            │          │
│  │                     │     │                          │          │
│  │  "Watch this        │────▶│  "Pick the latest tag    │          │
│  │   container         │     │   that matches this      │          │
│  │   registry for      │     │   pattern/semver rule"   │          │
│  │   new tags"         │     │                          │          │
│  └─────────────────────┘     └────────────┬─────────────┘          │
│                                           │                        │
│                                           ▼                        │
│                              ┌──────────────────────────┐          │
│                              │  Image Update            │          │
│                              │  Automation              │          │
│                              │                          │          │
│                              │  "Update the YAML in     │          │
│                              │   Git repo with the      │          │
│                              │   selected tag and       │          │
│                              │   commit the change"     │          │
│                              └──────────────────────────┘          │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### The Three Custom Resources

| Resource | What It Does | Analogy |
|----------|-------------|---------|
| **ImageRepository** | Scans a container registry for available image tags | A **security camera** watching the registry door |
| **ImagePolicy** | Decides WHICH tag to pick based on rules (latest semver, alphabetical, etc.) | A **filter/gatekeeper** that says "only let through tags matching these rules" |
| **ImageUpdateAutomation** | Takes the selected tag and writes it into the Git repo YAML files | A **robot editor** that opens your YAML, updates the image tag, and commits |

## 🛠️ Step-by-Step Setup

### Prerequisites

Before we start, make sure you have:

1. A Kubernetes cluster with Flux already installed (see 17.6)
2. A container registry with images (Docker Hub, ECR, GCR, ACR, etc.)
3. A Git repository that Flux is already managing

### Step 1: Install the Image Automation Controllers

By default, Flux doesn't install the image automation components. You need to explicitly enable them during bootstrap or install them separately.

**Option A: During Flux Bootstrap**

```bash
# When bootstrapping Flux, add the image automation components
flux bootstrap github \
  --owner=your-github-username \
  --repository=your-fleet-repo \
  --path=clusters/my-cluster \
  --personal \
  --components-extra=image-reflector-controller,image-automation-controller
```

Let's break this down:
- `--components-extra` tells Flux to install **additional** controllers beyond the defaults
- `image-reflector-controller` → handles **ImageRepository** and **ImagePolicy** (the "watching" part)
- `image-automation-controller` → handles **ImageUpdateAutomation** (the "updating Git" part)

**Option B: Install on an Existing Flux Setup**

```bash
# If Flux is already running, install the extra components manually
flux install \
  --components-extra=image-reflector-controller,image-automation-controller
```

**Verify they're running:**

```bash
kubectl get pods -n flux-system

# You should see something like:
# NAME                                            READY   STATUS
# image-automation-controller-7f8b9d6c4-x2k8p     1/1     Running
# image-reflector-controller-5d4f8b7a3-m9n2j      1/1     Running
# source-controller-6b8c9d5f7-p4q7r               1/1     Running
# kustomize-controller-8a7b6c5d4-s3t2u            1/1     Running
# ... other Flux controllers ...
```

### Step 2: Create an ImageRepository

The `ImageRepository` tells Flux: **"Hey, keep an eye on this container registry and tell me what tags are available."**

```yaml
# image-repo.yaml
apiVersion: image.toolkit.fluxcd.io/v1beta2
kind: ImageRepository
metadata:
  name: myapp                    # A name you choose for this resource
  namespace: flux-system         # Usually in the flux-system namespace
spec:
  image: docker.io/yourname/myapp   # The container image to watch (WITHOUT the tag)
  interval: 5m                      # How often to scan for new tags (every 5 minutes)
  secretRef:
    name: regcred                   # (Optional) Secret for private registries
```

**Line-by-line explanation:**

- `image: docker.io/yourname/myapp` → This is the full image path **without** any tag. Flux will scan ALL tags for this image.
- `interval: 5m` → Every 5 minutes, Flux will go to the registry and ask: "What tags exist for this image?" Think of it as Flux refreshing the page every 5 minutes.
- `secretRef` → If your registry is private (like a private ECR or private Docker Hub repo), you need to provide credentials. We'll cover this below.

**For private registries (e.g., AWS ECR):**

```yaml
# First, create a Kubernetes secret with your registry credentials
kubectl create secret docker-registry regcred \
  --docker-server=123456789.dkr.ecr.us-east-1.amazonaws.com \
  --docker-username=AWS \
  --docker-password=$(aws ecr get-login-password) \
  -n flux-system
```

**For Docker Hub (public images, no secret needed):**

```yaml
apiVersion: image.toolkit.fluxcd.io/v1beta2
kind: ImageRepository
metadata:
  name: myapp
  namespace: flux-system
spec:
  image: docker.io/yourname/myapp
  interval: 5m
  # No secretRef needed for public images!
```

**Apply and verify:**

```bash
kubectl apply -f image-repo.yaml

# Check if it's scanning successfully
flux get image repository myapp

# Expected output:
# NAME   LAST SCAN                  SUSPENDED   READY   MESSAGE
# myapp  2024-01-15T10:30:00Z       False       True    successful scan: found 25 tags
```

If you see `"successful scan: found X tags"` — it's working! Flux can see your images.

### Step 3: Create an ImagePolicy

The `ImagePolicy` answers the question: **"Out of all the tags found, which one should I use?"**

This is where you define your **tag selection strategy**. Let's look at different strategies:

#### Strategy 1: Semantic Versioning (Most Common)

```yaml
# image-policy-semver.yaml
apiVersion: image.toolkit.fluxcd.io/v1beta2
kind: ImagePolicy
metadata:
  name: myapp
  namespace: flux-system
spec:
  imageRepositoryRef:
    name: myapp                    # Points to the ImageRepository we created above
  policy:
    semver:
      range: ">=1.0.0"            # Accept any tag that's semver >= 1.0.0
```

**What this does:**

```
Available tags in registry: v1.0.0, v1.1.0, v1.2.0, v2.0.0, latest, dev, abc123

Semver filter (>=1.0.0) selects: v1.0.0, v1.1.0, v1.2.0, v2.0.0
(Ignores "latest", "dev", "abc123" because they're not valid semver)

Picks the HIGHEST: v2.0.0 ✅
```

**More semver range examples:**

```yaml
# Only patch updates (e.g., 1.2.x)
policy:
  semver:
    range: ">=1.2.0 <1.3.0"      # Picks latest 1.2.x (e.g., 1.2.15)

# Only minor updates within v1 (e.g., 1.x.x)
policy:
  semver:
    range: ">=1.0.0 <2.0.0"      # Picks latest 1.x.x (e.g., 1.9.3)

# Exact major version
policy:
  semver:
    range: "^1.0.0"              # Same as >=1.0.0 <2.0.0 (caret range)

# Any version with tilde
policy:
  semver:
    range: "~1.2.0"              # Same as >=1.2.0 <1.3.0 (tilde range)
```

#### Strategy 2: Alphabetical (Latest Alphabetically)

```yaml
# image-policy-alpha.yaml
apiVersion: image.toolkit.fluxcd.io/v1beta2
kind: ImagePolicy
metadata:
  name: myapp
  namespace: flux-system
spec:
  imageRepositoryRef:
    name: myapp
  policy:
    alphabetical:
      order: asc                  # ascending = pick the "highest" alphabetically
```

**When to use this:** When your tags aren't semver but are sortable strings, like dates:

```
Tags: 2024-01-01, 2024-01-15, 2024-02-01
Alphabetical (ascending) picks: 2024-02-01 ✅ (it's the "latest" date string)
```

#### Strategy 3: Numerical (Latest Number)

```yaml
# image-policy-numerical.yaml
apiVersion: image.toolkit.fluxcd.io/v1beta2
kind: ImagePolicy
metadata:
  name: myapp
  namespace: flux-system
spec:
  imageRepositoryRef:
    name: myapp
  policy:
    numerical:
      order: asc                  # ascending = pick the highest number
```

**When to use this:** When your tags are simple numbers like build numbers:

```
Tags: 100, 101, 102, 103
Numerical (ascending) picks: 103 ✅
```

#### Filtering Tags Before Selecting

Sometimes your registry has tags you want to ignore. You can use `filterTags` to narrow down:

```yaml
apiVersion: image.toolkit.fluxcd.io/v1beta2
kind: ImagePolicy
metadata:
  name: myapp
  namespace: flux-system
spec:
  imageRepositoryRef:
    name: myapp
  filterTags:
    pattern: '^main-[a-f0-9]+-(?P<ts>[0-9]+)'   # Regex pattern to match tags
    extract: '$ts'                                  # Extract this part for sorting
  policy:
    numerical:
      order: asc
```

**What this does:**

```
Tags in registry: main-abc123-1705312800, main-def456-1705399200, dev-ghi789-1705400000

Filter pattern: ^main-[a-f0-9]+-(?P<ts>[0-9]+)
Only matches tags starting with "main-" → ignores dev-ghi789-1705400000

Extract the timestamp part for comparison:
  main-abc123-1705312800 → 1705312800
  main-def456-1705399200 → 1705399200

Numerical ascending picks the highest timestamp: main-def456-1705399200 ✅
```

**Apply and verify:**

```bash
kubectl apply -f image-policy-semver.yaml

# Check which tag was selected
flux get image policy myapp

# Expected output:
# NAME   LATEST IMAGE                      READY   MESSAGE
# myapp  docker.io/yourname/myapp:v2.0.0   True    Latest image tag for... is v2.0.0
```

### Step 4: Mark Your YAML Files for Update

This is a crucial step! You need to tell Flux **exactly where** in your YAML files to update the image tag. You do this by adding a special **marker comment**.

**Before (your deployment YAML):**

```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
  namespace: default
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: myapp
        image: docker.io/yourname/myapp:v1.0.0  # {"$imagepolicy": "flux-system:myapp"}
```

**The magic is this comment at the end of the image line:**

```
# {"$imagepolicy": "flux-system:myapp"}
```

Let's break it down:
- `flux-system` → the namespace where your ImagePolicy lives
- `myapp` → the name of your ImagePolicy resource
- The comment tells Flux: **"Hey, when you have a new image tag from the 'myapp' ImagePolicy in the 'flux-system' namespace, update THIS line."**

**You can also update just the tag part or the whole image+tag:**

```yaml
# Update the FULL image (registry + name + tag):
image: docker.io/yourname/myapp:v1.0.0  # {"$imagepolicy": "flux-system:myapp"}
# Result: docker.io/yourname/myapp:v2.0.0

# Update ONLY the tag:
image: docker.io/yourname/myapp:v1.0.0  # {"$imagepolicy": "flux-system:myapp:tag"}
# Result: docker.io/yourname/myapp:v2.0.0
# (Notice ":tag" at the end — only the tag portion gets updated)

# Update ONLY the image name (rare):
image: docker.io/yourname/myapp:v1.0.0  # {"$imagepolicy": "flux-system:myapp:name"}
# Result: docker.io/yourname/myapp:v2.0.0
```

> **Important:** This marker comment MUST be in your Git repository files. Without it, Flux doesn't know what to update!

### Step 5: Create an ImageUpdateAutomation

This is the final piece — it tells Flux: **"Take the image selected by the policy and write it into the Git repo."**

```yaml
# image-update-automation.yaml
apiVersion: image.toolkit.fluxcd.io/v1beta2
kind: ImageUpdateAutomation
metadata:
  name: flux-system
  namespace: flux-system
spec:
  interval: 30m                           # How often to check for pending updates
  sourceRef:
    kind: GitRepository
    name: flux-system                     # The GitRepository resource Flux is using
  git:
    checkout:
      ref:
        branch: main                      # The branch to checkout
    commit:
      author:
        name: fluxcdbot                   # Who shows up as the commit author
        email: fluxcdbot@users.noreply.github.com
      messageTemplate: |                  # The commit message template
        Automated image update

        Automation name: {{ .AutomationObject }}

        Files:
        {{ range $filename, $_ := .Changed.FileChanges -}}
        - {{ $filename }}
        {{ end -}}

        Objects:
        {{ range $resource, $changes := .Changed.Objects -}}
        - {{ $resource }}
          {{- range $_, $change := $changes }}
          - {{ $change.OldValue }} -> {{ $change.NewValue }}
          {{- end }}
        {{ end -}}
    push:
      branch: main                        # The branch to push changes to
  update:
    path: ./clusters/my-cluster           # Only update files in this directory
    strategy: Setters                     # The strategy (currently only "Setters")
```

**Line-by-line breakdown:**

- `interval: 30m` → Every 30 minutes, check if any ImagePolicy has a new tag that needs updating
- `sourceRef` → Points to your GitRepository (the same one Flux uses for deployments)
- `git.checkout.ref.branch` → Which branch to work on
- `git.commit.author` → The bot identity for auto-commits (so you know it was automated, not a human)
- `git.commit.messageTemplate` → A Go template for the commit message. The `{{ }}` parts get filled in automatically with details about what changed
- `git.push.branch` → Which branch to push the commit to
- `update.path` → Only scan YAML files in this directory for marker comments. This prevents Flux from accidentally editing files it shouldn't touch
- `update.strategy: Setters` → Uses the marker comment approach we set up in Step 4

**Apply it:**

```bash
kubectl apply -f image-update-automation.yaml

# Verify
flux get image update flux-system

# Expected output:
# NAME          LAST RUN                  SUSPENDED   READY   MESSAGE
# flux-system   2024-01-15T10:45:00Z      False       True    no updates made
```

### Step 6: Watch the Magic Happen! 🎩

Now let's see the full flow in action:

```
1. You push code to your app repo
2. Your CI (GitHub Actions, Jenkins, etc.) builds image: myapp:v2.1.0
3. CI pushes myapp:v2.1.0 to Docker Hub

4. Flux ImageRepository scans Docker Hub → "Found new tag: v2.1.0!"
5. Flux ImagePolicy evaluates → "v2.1.0 matches >=1.0.0 and is the highest. Selected!"
6. Flux ImageUpdateAutomation:
   a. Checks out the Git repo
   b. Finds the marker: # {"$imagepolicy": "flux-system:myapp"}
   c. Updates the line: image: myapp:v1.0.0 → image: myapp:v2.1.0
   d. Commits with message: "Automated image update"
   e. Pushes to main branch

7. Flux GitOps (Kustomize/Helm controller) sees the new commit
8. Flux applies the updated deployment to the cluster
9. Kubernetes rolls out the new pods with myapp:v2.1.0

Total human intervention: ZERO 🎯
```

**Check what's happening:**

```bash
# See the latest image that was selected
flux get image policy myapp

# See if any updates were committed
flux get image update flux-system

# Check your Git repo — you'll see auto-commits from "fluxcdbot"!
git log --oneline -5
# abc1234 Automated image update
# def5678 Automated image update
# ghi9012 feat: add new feature (your commit)
```

## 🔔 Flux Notifications — "Hey, Something Just Happened!"

Alright, now let's talk about the second big topic: **Notifications**.

### Why Notifications?

GitOps is great, but it's largely **silent**. Things happen automatically in the background. That's awesome... until something goes wrong and nobody notices.

You need to know when:
- ✅ A deployment **succeeded**
- ❌ A deployment **failed**
- 🔄 A new image was **detected and updated**
- ⚠️ A reconciliation **error** occurred
- 🔔 A health check **failed**

Flux Notifications solve this with two custom resources:

```
┌─────────────────────────────────────────────────────────┐
│                   Flux Notifications                     │
│                                                          │
│  ┌──────────────┐           ┌──────────────────────┐    │
│  │   Provider    │           │   Alert              │    │
│  │              │           │                      │    │
│  │  WHERE to    │◀──────────│  WHAT events to      │    │
│  │  send        │           │  watch and WHO       │    │
│  │  notifications│           │  to notify           │    │
│  │              │           │                      │    │
│  │  (Slack,     │           │  (Kustomization      │    │
│  │   Teams,     │           │   failed? GitRepo    │    │
│  │   Discord,   │           │   updated? etc.)     │    │
│  │   Webhook)   │           │                      │    │
│  └──────────────┘           └──────────────────────┘    │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

| Resource | What It Does | Analogy |
|----------|-------------|---------|
| **Provider** | Defines WHERE to send notifications (Slack channel, Teams webhook, etc.) | The **mailbox address** |
| **Alert** | Defines WHAT events to listen for and links them to a Provider | The **mail filter rule** — "Send me an email when X happens" |

### Setting Up Slack Notifications (Most Common)

#### Step 1: Create a Slack Incoming Webhook

1. Go to your Slack workspace → Apps → Search for "Incoming Webhooks"
2. Add it to your desired channel (e.g., `#deployments`)
3. Copy the webhook URL (looks like `https://hooks.slack.com/services/T00/B00/xxxx`)

#### Step 2: Create a Secret with the Webhook URL

```bash
# Store the webhook URL as a Kubernetes secret
kubectl create secret generic slack-webhook-url \
  --from-literal=address=https://hooks.slack.com/services/T00/B00/xxxx \
  -n flux-system
```

> **Why a secret?** Because webhook URLs should be treated as sensitive — anyone with the URL can post to your Slack channel.

#### Step 3: Create a Provider

```yaml
# notification-provider.yaml
apiVersion: notification.toolkit.fluxcd.io/v1beta3
kind: Provider
metadata:
  name: slack                         # Name for this provider
  namespace: flux-system
spec:
  type: slack                         # The notification service type
  channel: deployments                # Slack channel name (without #)
  secretRef:
    name: slack-webhook-url           # The secret containing the webhook URL
```

**Supported provider types:**

| Provider | `spec.type` Value | Use Case |
|----------|-------------------|----------|
| Slack | `slack` | Team chat notifications |
| Microsoft Teams | `msteams` | Team chat notifications |
| Discord | `discord` | Team/community notifications |
| Google Chat | `googlechat` | Google Workspace teams |
| Rocket.Chat | `rocket` | Self-hosted chat |
| Matrix | `matrix` | Open-source chat |
| Opsgenie | `opsgenie` | Incident management |
| PagerDuty | `pagerduty` | On-call alerting |
| Sentry | `sentry` | Error tracking |
| Azure Event Hub | `azureeventhub` | Azure event streaming |
| Generic Webhook | `generic` | Any HTTP endpoint |
| Generic HMAC | `generic-hmac` | Webhook with HMAC signing |
| GitHub (Dispatch) | `githubdispatch` | Trigger GitHub Actions |
| Grafana | `grafana` | Grafana annotations |
| Webex | `webex` | Cisco Webex teams |
| Telegram | `telegram` | Telegram bot notifications |

#### Step 4: Create an Alert

```yaml
# notification-alert.yaml
apiVersion: notification.toolkit.fluxcd.io/v1beta3
kind: Alert
metadata:
  name: on-call-webapp
  namespace: flux-system
spec:
  summary: "Production cluster notification"   # A description of this alert
  providerRef:
    name: slack                                 # Points to the Provider above
  eventSeverity: info                           # Minimum severity: info or error
  eventSources:                                 # WHAT to watch
    - kind: GitRepository
      name: '*'                                 # Watch ALL GitRepositories
    - kind: Kustomization
      name: '*'                                 # Watch ALL Kustomizations
    - kind: HelmRelease
      name: '*'                                 # Watch ALL HelmReleases
    - kind: ImageRepository
      name: '*'                                 # Watch ALL ImageRepositories
    - kind: ImagePolicy
      name: '*'                                 # Watch ALL ImagePolicies
    - kind: ImageUpdateAutomation
      name: '*'                                 # Watch ALL ImageUpdateAutomations
```

**Understanding `eventSeverity`:**

```
eventSeverity: info   → Get EVERYTHING (info + error messages)
eventSeverity: error  → Get ONLY error messages (failures, problems)

Recommendation:
- For dev/staging → use "info" (see everything, learn what's happening)
- For production  → use "error" (only get alerted on problems, less noise)
```

**Understanding `eventSources`:**

```yaml
eventSources:
  # Watch a SPECIFIC resource:
  - kind: Kustomization
    name: my-webapp                # Only this specific Kustomization

  # Watch ALL resources of a kind:
  - kind: Kustomization
    name: '*'                      # ALL Kustomizations (wildcard)

  # Watch resources in a specific namespace:
  - kind: HelmRelease
    name: '*'
    namespace: production          # Only HelmReleases in 'production' namespace

  # Watch across ALL namespaces:
  - kind: HelmRelease
    name: '*'
    namespace: '*'                 # Note: cross-namespace requires RBAC setup
```

**Apply everything:**

```bash
kubectl apply -f notification-provider.yaml
kubectl apply -f notification-alert.yaml

# Verify
flux get alert
# NAME              SUSPENDED   READY   MESSAGE
# on-call-webapp    False       True    Initialized

flux get alert-provider
# NAME    READY   MESSAGE
# slack   True    Initialized
```

### Example: Generic Webhook (for Any HTTP Endpoint)

This is super flexible — you can send notifications to any API or service:

```yaml
# generic-provider.yaml
apiVersion: notification.toolkit.fluxcd.io/v1beta3
kind: Provider
metadata:
  name: my-custom-webhook
  namespace: flux-system
spec:
  type: generic
  address: https://my-api.example.com/flux-events   # Your custom endpoint
  headers:                                            # Optional custom headers
    Authorization: Bearer ${TOKEN}
  secretRef:
    name: webhook-secret
```

The generic webhook sends a JSON payload like this to your endpoint:

```json
{
  "involvedObject": {
    "kind": "Kustomization",
    "namespace": "flux-system",
    "name": "my-app",
    "apiVersion": "kustomize.toolkit.fluxcd.io/v1"
  },
  "severity": "info",
  "timestamp": "2024-01-15T10:45:00Z",
  "message": "Reconciliation finished in 5s, next run in 10m",
  "reason": "ReconciliationSucceeded",
  "metadata": {
    "revision": "main@sha1:abc123def456",
    "summary": "configmap/my-config created\ndeployment/my-app configured"
  }
}
```

You can build any custom integration with this — trigger a Jira ticket, send an SMS, update a dashboard, etc.

### Example: GitHub Dispatch (Trigger GitHub Actions from Flux)

This is a powerful pattern — Flux can **trigger a GitHub Actions workflow** when something happens:

```yaml
# github-dispatch-provider.yaml
apiVersion: notification.toolkit.fluxcd.io/v1beta3
kind: Provider
metadata:
  name: github-dispatch
  namespace: flux-system
```

## 🔄 Incoming Webhooks — Receiver (The Other Direction)

So far, we've talked about Flux **sending** notifications OUT. But what about the other direction — **receiving** webhooks to trigger Flux?

The `Receiver` resource allows external systems to trigger Flux reconciliation via webhook:

```
External System → Webhook → Flux Receiver → Triggers Reconciliation

Examples:
- Docker Hub sends a webhook when new image is pushed → Flux re-scans immediately
- GitHub sends a webhook on push → Flux re-syncs immediately (instead of waiting)
- Your CI pipeline sends a webhook → Flux deploys immediately
```

### Setting Up a Receiver

```yaml
# receiver.yaml
apiVersion: notification.toolkit.fluxcd.io/v1
kind: Receiver
metadata:
  name: github-receiver
  namespace: flux-system
spec:
  type: github                        # The webhook source type
  events:
    - "ping"
    - "push"                          # Trigger on push events
  secretRef:
    name: webhook-token               # Shared secret for verification
  resources:
    - kind: GitRepository
      name: flux-system               # Which resource to reconcile when triggered
```

**Create the webhook secret:**

```bash
# Generate a random token
TOKEN=$(head -c 12 /dev/urandom | shasum | cut -d ' ' -f1)

# Create the secret
kubectl create secret generic webhook-token \
  --from-literal=token=$TOKEN \
  -n flux-system

echo "Your webhook token: $TOKEN"
# Save this! You'll need it when configuring the GitHub webhook.
```

**Get the Receiver URL:**

```bash
# After applying the Receiver, get the webhook URL
kubectl get receiver github-receiver -n flux-system

# Output shows the webhook path:
# NAME               READY   STATUS
# github-receiver    True    Receiver initialized for path: /hook/abc123def456

# The full URL would be:
# http://<your-flux-webhook-service>/hook/abc123def456
```

**Expose the Receiver (so GitHub can reach it):**

You need to make the Flux webhook receiver accessible from the internet. Common approaches:

```yaml
# Option 1: Ingress
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: flux-receiver
  namespace: flux-system
spec:
  rules:
    - host: flux-webhook.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: webhook-receiver
                port:
                  number: 80
```

Then configure the webhook in GitHub:
1. Go to your repo → Settings → Webhooks → Add webhook
2. Payload URL: `https://flux-webhook.example.com/hook/abc123def456`
3. Content type: `application/json`
4. Secret: The token you generated above

**Supported Receiver Types:**

| Type | Source | Use Case |
|------|--------|----------|
| `github` | GitHub webhooks | Trigger on push/PR events |
| `gitlab` | GitLab webhooks | Trigger on push/merge events |
| `bitbucket` | Bitbucket webhooks | Trigger on push events |
| `harbor` | Harbor webhooks | Trigger on image push |
| `dockerhub` | Docker Hub webhooks | Trigger on image push |
| `quay` | Quay.io webhooks | Trigger on image push |
| `gcr` | Google Container Registry | Trigger on image push |
| `acr` | Azure Container Registry | Trigger on image push |
| `nexus` | Nexus webhooks | Trigger on artifact push |
| `generic` | Any HTTP webhook | Custom integrations |
| `generic-hmac` | HTTP webhook with HMAC | Verified custom integrations |

## 🔧 Troubleshooting Image Automation

### Common Issues and Fixes

| Problem | Cause | Fix |
|---------|-------|-----|
| "no tags found" | Wrong image path or private registry without creds | Double-check `spec.image` path and `secretRef` |
| "no match for policy" | No tags match your semver/filter pattern | Check your tags — are they actually valid semver? |
| "authentication required" | Private registry, missing or wrong credentials | Create/update the docker-registry secret |
| "push rejected" | Flux bot doesn't have write access to Git repo | Check deploy key or PAT has write access |
| Commits not appearing | `update.path` doesn't match where your YAMLs are | Verify the path matches your repo structure |
| Marker not found | Missing or malformed `# {"$imagepolicy": ...}` comment | Check for typos, must be valid JSON in the comment |

### Useful Debug Commands

```bash
# Check image repository scanning
flux get image repository
kubectl describe imagerepository webapp -n flux-system

# Check image policy selection
flux get image policy
kubectl describe imagepolicy webapp -n flux-system

# Check image update automation status
flux get image update
kubectl describe imageupdateautomation flux-system -n flux-system

# Check controller logs for errors
kubectl logs -n flux-system deploy/image-reflector-controller
kubectl logs -n flux-system deploy/image-automation-controller

# Force an immediate scan (instead of waiting for interval)
flux reconcile image repository webapp

# Force an immediate update check
flux reconcile image update flux-system

# Check alerts and providers
flux get alert
flux get alert-provider
kubectl logs -n flux-system deploy/notification-controller
```

```
│    This tells Flux exactly which line to update                 │

│  The Full Loop:                                                 │
│    CI builds image → Registry → Flux detects →                  │
```
