# 17.8 — ArgoCD vs Flux CD — Comparison & When to Use

> **One Line Answer:** ArgoCD and Flux CD are both **GitOps tools** that sync your Kubernetes cluster with a Git repository, but they take very different approaches — ArgoCD gives you a **beautiful UI and rich features out of the box**, while Flux takes a **modular, composable, CLI-first** approach. Neither is "better" — the right choice depends on your team, your workflow, and your environment.

## 🤔 Why Compare Them?

If you're starting with GitOps, the first question you'll face is: **"Should I use ArgoCD or Flux?"**

Both are:
- ✅ CNCF **Graduated** projects (the highest maturity level — this is a big deal!)
- ✅ Open source and free
- ✅ Production-ready and battle-tested at scale
- ✅ Following the GitOps principles (Git as the source of truth, pull-based model)

But they feel **very different** to use. Let's explore exactly how.

## 🏗️ Architecture Comparison

### ArgoCD Architecture

```
┌──────────────────────────────────────────────────────────────────┐
│                        ArgoCD                                    │
│                                                                  │
│  ┌──────────────┐   ┌──────────────────┐   ┌────────────────┐  │
│  │  API Server  │   │  Repo Server     │   │  Application   │  │
│  │              │   │                  │   │  Controller    │  │
│  │  Handles UI  │   │  Clones & caches │   │                │  │
│  │  + API       │   │  Git repos       │   │  Watches apps  │  │
│  │  + Auth      │   │  Renders manifests│   │  Syncs state   │  │
│  │  + RBAC      │   │  (Helm, Kustomize│   │  Health checks │  │
│  │              │   │   Jsonnet, etc.) │   │                │  │
│  └──────┬───────┘   └────────┬─────────┘   └───────┬────────┘  │
│         │                    │                      │            │
│         ▼                    ▼                      ▼            │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │                    Redis (Cache)                          │   │
│  └──────────────────────────────────────────────────────────┘   │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │                    Dex (SSO/OIDC)                         │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
│  Total: 5-7 pods running in your cluster                        │
└──────────────────────────────────────────────────────────────────┘
```

**Key points:**
- ArgoCD is a **single, monolithic** application (all components come together)
- Has its own **UI, API server, authentication**, and caching layer
- Runs as **5-7 pods** in your cluster
- The `Application` CRD is the central concept — everything is an "Application"

### Flux CD Architecture

```
┌──────────────────────────────────────────────────────────────────┐
│                         Flux CD                                   │
│                                                                   │
│  ┌─────────────────┐  ┌─────────────────┐  ┌──────────────────┐ │
│  │  Source          │  │  Kustomize      │  │  Helm            │ │
│  │  Controller      │  │  Controller      │  │  Controller      │ │
│  │                  │  │                  │  │                  │ │
│  │  Fetches Git     │  │  Applies plain   │  │  Applies Helm    │ │
│  │  repos, Helm     │  │  YAML &          │  │  charts          │ │
│  │  repos, OCI,     │  │  Kustomizations  │  │                  │ │
│  │  S3 buckets      │  │                  │  │                  │ │
│  └─────────────────┘  └─────────────────┘  └──────────────────┘ │
│                                                                   │
│  ┌─────────────────┐  ┌─────────────────┐                       │
│  │  Notification    │  │  Image           │                       │
│  │  Controller      │  │  Automation      │   (Optional extras)  │
│  │                  │  │  Controllers     │                       │
│  │  Alerts &        │  │                  │                       │
│  │  Webhooks        │  │  Image scanning  │                       │
│  │                  │  │  & auto-update   │                       │
│  └─────────────────┘  └─────────────────┘                       │
│                                                                   │
│  Total: 4-6 pods (you choose which controllers to install)       │
└──────────────────────────────────────────────────────────────────┘
```

**Key points:**
- Flux is a collection of **independent, composable controllers** (microservices approach!)
- **No built-in UI** — it's CLI-first, uses `flux` CLI and `kubectl`
- You can install **only the controllers you need** — don't need Helm? Don't install the Helm controller!
- Uses **multiple CRDs** (GitRepository, Kustomization, HelmRelease, etc.) — each controller manages its own CRDs

### Multi-Cluster Management

| Feature | ArgoCD | Flux CD | Notes |
|---------|--------|---------|-------|
| **Multi-cluster from single instance** | ✅ Native (register external clusters) | ❌ Each cluster runs its own Flux | Very different approaches |
| **Hub-spoke model** | ✅ (one ArgoCD manages many clusters) | ❌ (not native) | **ArgoCD** 🏆 |
| **Cluster-per-Flux model** | ❌ Not the default pattern | ✅ (each cluster is independent) | **Flux** 🏆 for isolation |
| **ApplicationSets (generate apps for many clusters)** | ✅ Powerful generators | Uses Kustomize overlays | Different but both work |

## 🎯 Decision Matrix — When to Use Which

### Choose ArgoCD When:

```
✅ You want a beautiful Web UI for your team
✅ Your team is new to GitOps/Kubernetes (the UI helps learning)
✅ You need to manage MULTIPLE clusters from ONE place
✅ You want ApplicationSets for generating apps dynamically
✅ You need SSO/OIDC authentication out of the box
✅ You use Jsonnet for configuration
✅ You want to use Argo Rollouts for progressive delivery
✅ You want a "batteries-included" experience
✅ You need granular sync control (sync waves, selective sync)
✅ Your org has developers who want to see deployment status visually
```

### Choose Flux When:

```
✅ You prefer CLI-first, "everything as code" approach
✅ You want native Kustomize integration (Flux IS Kustomize-native)
✅ You need built-in Image Automation (auto-update image tags)
✅ You want native SOPS integration for secret encryption
✅ You prefer lightweight, modular architecture (install only what you need)
✅ You want each cluster to be fully independent (no central hub)
✅ You prefer standard Kubernetes RBAC (no separate auth system)
✅ You want to use Flagger for progressive delivery (canary, A/B)
✅ You're comfortable with kubectl and don't need a UI
✅ You want deeper integration with Kubernetes itself
```

## 🔄 Side-by-Side: Same Task, Different Approach

Let's see how you'd do the exact same things in both tools:

### Task 1: Deploy an Application from Git

**ArgoCD:**

```yaml
# ArgoCD Application CRD
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: my-webapp
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/myorg/my-webapp.git
    targetRevision: main
    path: k8s/overlays/production
  destination:
    server: https://kubernetes.default.svc
    namespace: production
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
      - CreateNamespace=true
```

**Flux:**

```yaml
# Flux GitRepository + Kustomization
---
apiVersion: source.toolkit.fluxcd.io/v1
kind: GitRepository
metadata:
  name: my-webapp
  namespace: flux-system
spec:
  interval: 5m
  url: https://github.com/myorg/my-webapp.git
  ref:
    branch: main
---
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: my-webapp
  namespace: flux-system
spec:
  interval: 10m
  sourceRef:
    kind: GitRepository
    name: my-webapp
  path: ./k8s/overlays/production
  targetNamespace: production
  prune: true
  healthChecks:
    - kind: Deployment
      name: my-webapp
      namespace: production
```

**Key differences:**
- ArgoCD: **1 resource** (Application) — everything in one place
- Flux: **2 resources** (GitRepository + Kustomization) — source and deployment are separated
- ArgoCD uses `selfHeal: true`, Flux uses `prune: true` (both achieve drift correction)
- Flux explicitly defines health checks; ArgoCD has built-in health assessment

### Task 2: Deploy a Helm Chart

**ArgoCD:**

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: nginx-ingress
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://kubernetes.github.io/ingress-nginx
    chart: ingress-nginx
    targetRevision: 4.7.1
    helm:
      values: |
        controller:
          replicaCount: 2
          service:
            type: LoadBalancer
  destination:
    server: https://kubernetes.default.svc
    namespace: ingress-nginx
  syncPolicy:
    automated:
      prune: true
```

**Flux:**

```yaml
---
apiVersion: source.toolkit.fluxcd.io/v1
kind: HelmRepository
metadata:
  name: ingress-nginx
  namespace: flux-system
spec:
  interval: 1h
  url: https://kubernetes.github.io/ingress-nginx
---
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: nginx-ingress
  namespace: flux-system
spec:
  interval: 10m
  chart:
    spec:
      chart: ingress-nginx
      version: 4.7.1
      sourceRef:
        kind: HelmRepository
        name: ingress-nginx
  targetNamespace: ingress-nginx
  install:
    createNamespace: true
  values:
    controller:
      replicaCount: 2
      service:
        type: LoadBalancer
```

**Key differences:**
- ArgoCD: Still just **1 resource** — the Application CRD handles Helm natively
- Flux: **2 resources** — HelmRepository (source) + HelmRelease (deployment)
- Flux's HelmRelease has more granular control over install/upgrade/rollback behaviors
- ArgoCD renders the Helm template server-side; Flux uses native Helm install

### Task 3: Check Application Status

**ArgoCD:**

```bash
# CLI way
argocd app get my-webapp

# Output:
# Name:               argocd/my-webapp
# Server:             https://kubernetes.default.svc
# Namespace:          production
# URL:                https://argocd.example.com/applications/my-webapp
# Repo:               https://github.com/myorg/my-webapp.git
# Target:             main
# Path:               k8s/overlays/production
# SyncWindow:         Sync Allowed
# Sync Policy:        Automated (Prune)
# Sync Status:        Synced to main (abc1234)
# Health Status:      Healthy
#
# GROUP  KIND        NAMESPACE   NAME       STATUS  HEALTH   HOOK  MESSAGE
#        Service     production  my-webapp  Synced  Healthy        service/my-webapp created
# apps   Deployment  production  my-webapp  Synced  Healthy        deployment/my-webapp configured

# Or just open the UI at https://argocd.example.com 🌐
```

**Flux:**

```bash
# CLI way
flux get kustomizations my-webapp

# Output:
# NAME       REVISION     SUSPENDED  READY  MESSAGE
# my-webapp  main@abc1234 False      True   Applied revision: main@abc1234

# More details
flux get kustomizations my-webapp --verbose

# Check all resources
kubectl get all -n production
```

### Task 4: Manual Sync / Force Reconciliation

**ArgoCD:**

```bash
# CLI
argocd app sync my-webapp

# Or click "SYNC" button in the UI ← this is nice!
```

**Flux:**

```bash
# CLI
flux reconcile kustomization my-webapp --with-source
```

### Task 5: Rollback

**ArgoCD:**

```bash
# Rollback to a previous sync
argocd app rollback my-webapp

# Or in the UI: History → select a revision → Rollback
```

**Flux:**

```bash
# Flux doesn't have a "rollback" command. The GitOps way is:
# Revert the commit in Git, and Flux will sync the reverted state.
git revert HEAD
git push

# Flux will pick up the revert and apply it.
# This is actually MORE GitOps-correct because Git remains the source of truth!
```

> **Interesting philosophical difference:** ArgoCD allows "escape hatches" (manual sync, rollback, resource sync from UI). Flux is more strict about GitOps — **everything must go through Git**. Neither approach is wrong, but they reflect different philosophies.

## 📈 Resource Usage & Performance

| Metric | ArgoCD | Flux CD |
|--------|--------|---------|
| **Memory (idle)** | ~300-500 MB | ~150-250 MB |
| **CPU (idle)** | ~100-200m | ~50-100m |
| **Number of pods** | 5-7 | 4-6 (depends on components) |
| **Handles 1000+ apps** | ✅ (with sharding) | ✅ |
| **Redis required** | Yes | No |
| **DB required** | No (uses Kubernetes) | No (uses Kubernetes) |

> Flux is generally **lighter** because it doesn't run a web server, Redis cache, or Dex auth server. ArgoCD trades resources for the richer feature set.

## 🤝 Can You Use Both Together?

Surprisingly, **yes**! Some teams use both:

```
Common hybrid patterns:

1. ArgoCD for app deployments (teams love the UI)
   + Flux for infrastructure (Terraform controller, image automation)

2. ArgoCD for production (needs UI for visibility)
   + Flux for dev/staging (lightweight, no UI needed)

3. ArgoCD for platform team (manages cluster-wide resources)
   + Flux for app teams (each team manages their own namespace)
```

However, **keep it simple** — for most teams, pick one and stick with it. Running both adds complexity.

## 🧪 Migration Between Tools

### ArgoCD → Flux

```bash
# General approach:
# 1. Install Flux alongside ArgoCD
# 2. Create Flux GitRepository + Kustomization for each ArgoCD Application
# 3. Verify Flux is managing the resources
# 4. Remove ArgoCD Applications one by one
# 5. Uninstall ArgoCD

# There's no automatic migration tool — it's a manual process.
# The good news: both manage the SAME Kubernetes resources,
# so the transition is about swapping the GitOps controller, not the apps.
```

### Flux → ArgoCD

```bash
# Same approach in reverse:
# 1. Install ArgoCD alongside Flux
# 2. Create ArgoCD Applications for each Flux Kustomization/HelmRelease
# 3. Verify ArgoCD is managing the resources
# 4. Remove Flux Kustomizations one by one
# 5. Uninstall Flux
```

## 📋 Summary Table — The Final Verdict

| Aspect | ArgoCD | Flux CD |
|--------|--------|---------|
| **Best for** | Teams wanting UI + multi-cluster | Teams wanting lightweight + CLI-first |
| **UI** | 🏆 Beautiful built-in | ❌ Requires third-party |
| **Multi-cluster** | 🏆 Native hub-spoke | Each cluster runs its own |
| **Image automation** | Requires addon | 🏆 Built-in |
| **Kustomize** | Good support | 🏆 Native, deeper |
| **SOPS encryption** | Via plugin | 🏆 Built-in |
| **Resource usage** | Heavier | 🏆 Lighter |
| **Learning curve** | Easier (UI helps) | Steeper (CLI only) |
| **Philosophy** | Pragmatic (escape hatches) | Pure GitOps (Git is the only way) |
| **SSO** | 🏆 Built-in (Dex) | Kubernetes auth only |
| **Community** | Larger | Growing fast |
