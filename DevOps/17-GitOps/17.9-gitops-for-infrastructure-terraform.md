# 17.9 — GitOps for Infrastructure (Terraform + GitOps)

> **One Line Answer:** GitOps isn't just for Kubernetes apps — you can apply the same "Git as the single source of truth" principle to your **entire infrastructure** using Terraform. This means your VPCs, databases, DNS records, IAM roles, and everything else are managed through Git pull requests, automated plans, and approved applies — no more running `terraform apply` from someone's laptop!

---

## 🤔 Wait, Isn't Terraform Already "Infrastructure as Code"?

Yes! Terraform IS Infrastructure as Code (IaC). But there's a difference between "Infrastructure as Code" and "GitOps for Infrastructure."

Let's understand why:

```
Traditional Terraform Workflow (IaC, but NOT GitOps):

Developer's Laptop:
  $ terraform plan      ← Run locally
  $ terraform apply     ← Run locally
  
  Problems:
  ❌ Who ran this? (No audit trail beyond Git commits)
  ❌ Did they run plan before apply? (Trust-based)
  ❌ Is their local state file in sync? (State drift)
  ❌ What if two people run apply at the same time? (Race condition)
  ❌ Did they use the right credentials? (Credential sprawl)
  ❌ Was the plan reviewed before apply? (No approval gate)

GitOps Terraform Workflow:

  1. Developer creates a PR with Terraform changes
  2. Automated system runs `terraform plan` and posts results on the PR
  3. Team reviews the plan in the PR
  4. On merge (or approval), automated system runs `terraform apply`
  5. State is stored remotely and locked
  6. Everything is audited, reviewed, and traceable

  ✅ Full audit trail (Git history)
  ✅ Plan is always reviewed (PR comments)
  ✅ No local credentials needed (CI/CD handles auth)
  ✅ No race conditions (automated locking)
  ✅ Consistent environment (always runs in CI/CD)
```

### The Key Difference

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  IaC = Your infrastructure is DEFINED in code               │
│        (Terraform files exist in a Git repo)                │
│                                                             │
│  GitOps for Infra = Your infrastructure is MANAGED          │
│                     through Git workflows                   │
│        (No one touches infra except through PRs)            │
│                                                             │
│  IaC answers: "What does my infrastructure look like?"      │
│  GitOps answers: "How do I safely change it?"               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 🏛️ Approach 1: Atlantis — The Most Popular Open-Source Option

### What is Atlantis?

Atlantis is an **open-source, self-hosted** application that listens for Terraform pull requests on GitHub, GitLab, or Bitbucket. When someone opens a PR with Terraform changes, Atlantis automatically runs `terraform plan` and posts the output as a PR comment. When the PR is approved and you comment `atlantis apply`, it runs `terraform apply`.

Think of Atlantis as your **Terraform robot assistant** that lives in your cluster and does all the Terraform work for you, triggered by PR comments.

```
The Atlantis Workflow:

  Developer                    Atlantis                    Cloud Provider
     │                            │                            │
     │  1. Opens PR with          │                            │
     │     Terraform changes      │                            │
     │ ──────────────────────────>│                            │
     │                            │  2. Runs terraform plan    │
     │                            │ ──────────────────────────>│
     │                            │                            │
     │                            │  3. Gets plan output       │
     │                            │ <──────────────────────────│
     │  4. Posts plan as          │                            │
     │     PR comment             │                            │
     │ <──────────────────────────│                            │
     │                            │                            │
     │  5. Team reviews plan      │                            │
     │     in the PR              │                            │
     │                            │                            │
     │  6. Comments               │                            │
     │     "atlantis apply"       │                            │
     │ ──────────────────────────>│                            │
     │                            │  7. Runs terraform apply   │
     │                            │ ──────────────────────────>│
     │                            │                            │
     │                            │  8. Gets apply output      │
     │                            │ <──────────────────────────│
     │  9. Posts apply result     │                            │
     │     as PR comment          │                            │
     │ <──────────────────────────│                            │
     │                            │                            │
     │  10. PR is merged          │                            │
     │                            │                            │
```

### Installing Atlantis

**Option 1: Docker (Quick Start)**

```bash
# Run Atlantis as a Docker container
docker run -d \
  --name atlantis \
  -p 4141:4141 \
  -e ATLANTIS_GH_USER="your-github-username" \
  -e ATLANTIS_GH_TOKEN="ghp_your_github_token" \
  -e ATLANTIS_GH_WEBHOOK_SECRET="your-webhook-secret" \
  -e ATLANTIS_REPO_ALLOWLIST="github.com/your-org/*" \
  -e AWS_ACCESS_KEY_ID="your-access-key" \
  -e AWS_SECRET_ACCESS_KEY="your-secret-key" \
  ghcr.io/runatlantis/atlantis:latest \
  server
```

Let's break down the environment variables:
- `ATLANTIS_GH_USER` → The GitHub username that Atlantis will use to comment on PRs
- `ATLANTIS_GH_TOKEN` → A GitHub Personal Access Token (PAT) with `repo` scope
- `ATLANTIS_GH_WEBHOOK_SECRET` → A secret shared between GitHub and Atlantis for webhook verification
- `ATLANTIS_REPO_ALLOWLIST` → Which repos Atlantis should respond to (security — prevent random repos from using your Atlantis)
- `AWS_*` → Cloud provider credentials (or use IAM roles for better security)

**Option 2: Kubernetes with Helm (Production)**

```bash
# Add the Atlantis Helm repo
helm repo add runatlantis https://runatlantis.github.io/helm-charts
helm repo update

# Create a values file
cat > atlantis-values.yaml << 'EOF'
orgAllowlist: "github.com/your-org/*"
github:
  user: atlantis-bot
  token: ghp_your_github_token
  secret: your-webhook-secret

# Use a ServiceAccount with IRSA (AWS) or Workload Identity (GCP)
# instead of hardcoding credentials!
serviceAccount:
  create: true
  annotations:
    eks.amazonaws.com/role-arn: arn:aws:iam::123456789:role/atlantis-role

# Persistent storage for Terraform state locks
dataStorage: 5Gi

# Resource limits
resources:
  requests:
    cpu: 100m
    memory: 256Mi
  limits:
    cpu: 500m
    memory: 512Mi
EOF

# Install Atlantis
helm install atlantis runatlantis/atlantis \
  -f atlantis-values.yaml \
  -n atlantis \
  --create-namespace
```

### Configuring Atlantis

Atlantis can be configured at two levels:

**1. Server-side config (atlantis.yaml in the Atlantis server):**

This controls server-wide settings and is configured when you deploy Atlantis.

**2. Repo-side config (atlantis.yaml in your Terraform repo):**

```yaml
# atlantis.yaml (in the ROOT of your Terraform repository)
version: 3
automerge: false                    # Don't auto-merge after successful apply
parallel_plan: true                 # Run plans in parallel for different dirs
parallel_apply: false               # Don't apply in parallel (safer)

projects:
  - name: networking
    dir: infrastructure/networking   # Directory containing Terraform code
    workspace: default
    terraform_version: v1.6.0       # Pin Terraform version
    autoplan:
      when_modified:                 # Auto-plan when these files change
        - "**/*.tf"
        - "**/*.tfvars"
      enabled: true
    apply_requirements:              # What's needed before apply is allowed
      - approved                     # PR must be approved
      - mergeable                    # PR must be mergeable (no conflicts)

  - name: compute
    dir: infrastructure/compute
    workspace: default
    terraform_version: v1.6.0
    autoplan:
      when_modified:
        - "**/*.tf"
        - "**/*.tfvars"
      enabled: true
    apply_requirements:
      - approved
      - mergeable

  - name: database
    dir: infrastructure/database
    workspace: default
    terraform_version: v1.6.0
    autoplan:
      when_modified:
        - "**/*.tf"
      enabled: true
    apply_requirements:
      - approved
      - mergeable
      - undiverged                   # Local state must match remote
```

**Line-by-line explanation:**

- `projects` → Define multiple Terraform projects within one repo. Each "project" is a separate Terraform root module (directory with its own state).
- `dir` → Where the Terraform code lives relative to the repo root
- `autoplan.when_modified` → File patterns that trigger an automatic plan when changed in a PR
- `autoplan.enabled: true` → Automatically run plan when a PR is opened/updated (no need to comment `atlantis plan`)
- `apply_requirements` → Gates that must pass before `atlantis apply` is allowed:
  - `approved` → At least one reviewer approved the PR
  - `mergeable` → No merge conflicts, all CI checks passed
  - `undiverged` → The plan output matches the current state (no one else applied between plan and apply)

### Atlantis Commands (PR Comments)

| Command | What It Does |
|---------|-------------|
| `atlantis plan` | Run terraform plan for all modified projects |
| `atlantis plan -d infrastructure/networking` | Plan a specific directory |
| `atlantis apply` | Apply all planned projects |
| `atlantis apply -d infrastructure/networking` | Apply a specific directory |
| `atlantis plan -- -var="env=staging"` | Pass extra args to terraform plan |
| `atlantis unlock` | Unlock the project (if locked due to failed apply) |
| `atlantis help` | Show all available commands |

## ☁️ Approach 2: Terraform Cloud / Terraform Enterprise

### What is Terraform Cloud?

Terraform Cloud is HashiCorp's **managed service** for running Terraform. It provides the same GitOps-like workflow but with a hosted platform — no need to self-host anything like Atlantis.

```
Terraform Cloud vs Atlantis:

┌─────────────────────────┬──────────────────────────────────┐
│       Atlantis          │       Terraform Cloud            │
├─────────────────────────┼──────────────────────────────────┤
│ Self-hosted (you run it)│ Managed service (HashiCorp runs) │
│ Free & open source      │ Free tier + paid plans           │
│ You manage upgrades     │ Always up to date                │
│ You handle HA/scaling   │ Built-in HA & scaling            │
│ You secure it           │ SOC2 compliant                   │
│ Minimal UI              │ Full Web UI                      │
│ Community support       │ Enterprise support available     │
└─────────────────────────┴──────────────────────────────────┘
```

### Setting Up Terraform Cloud

**Step 1: Create an account at app.terraform.io**

**Step 2: Configure your Terraform code to use Terraform Cloud:**

```hcl
# backend.tf
terraform {
  cloud {
    organization = "your-org-name"

    workspaces {
      name = "production-infrastructure"
    }
  }

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}
```

**Step 3: Configure VCS Integration**

In Terraform Cloud UI:
1. Go to Settings → VCS Providers → Add a VCS Provider
2. Connect your GitHub/GitLab account
3. Create a Workspace and link it to your repo

**Step 4: Set Variables**

In the Terraform Cloud workspace settings, add your variables:

```
Environment Variables:
  AWS_ACCESS_KEY_ID     = (sensitive)
  AWS_SECRET_ACCESS_KEY = (sensitive)
  AWS_DEFAULT_REGION    = us-east-1

Terraform Variables:
  environment = "production"
  team        = "platform"
```

### Terraform Cloud Workflow

```
1. Developer pushes to a branch → Opens PR
2. Terraform Cloud automatically runs a "Speculative Plan"
   (a plan that CANNOT be applied — just for review)
3. Plan output appears as a GitHub check on the PR
4. Team reviews the plan
5. PR is merged to main
6. Terraform Cloud runs a "Real Plan" on main
7. The plan requires manual confirmation in the TFC UI
   (or auto-apply if configured)
8. Terraform Cloud applies the changes

The flow:
PR → Speculative Plan → Review → Merge → Real Plan → Confirm → Apply
```

### Terraform Cloud Configuration (in the workspace settings)

```
Workspace Settings:
  ├── General
  │   ├── Execution Mode: Remote (runs on TFC's runners)
  │   ├── Apply Method: Manual (require confirmation) or Auto
  │   └── Terraform Version: 1.6.x
  │
  ├── VCS Integration
  │   ├── Repository: github.com/your-org/infrastructure
  │   ├── Branch: main
  │   ├── Working Directory: infrastructure/production
  │   └── Automatic Speculative Plans: Enabled
  │
  ├── Variables
  │   ├── Terraform Variables: (key-value pairs)
  │   └── Environment Variables: (AWS creds, etc.)
  │
  ├── Team Access
  │   ├── platform-team: Admin
  │   ├── dev-team: Plan (can trigger plans, can't apply)
  │   └── security-team: Read (can view state and plans)
  │
  └── Notifications
      ├── Slack: #infra-changes channel
      └── Email: platform-team@company.com
```

## 🎮 Approach 3: DIY with GitHub Actions (Roll Your Own)

If you don't want to run Atlantis or pay for Terraform Cloud, you can build your own GitOps workflow using **GitHub Actions** (or GitLab CI, Jenkins, etc.).

### Basic GitHub Actions Terraform Pipeline

```yaml
# .github/workflows/terraform.yml
name: Terraform GitOps

on:
  pull_request:
    branches: [main]
    paths:
      - 'infrastructure/**'        # Only trigger when infra files change
  push:
    branches: [main]
    paths:
      - 'infrastructure/**'

# IMPORTANT: Prevent concurrent runs on the same infra
concurrency:
  group: terraform-${{ github.ref }}
  cancel-in-progress: false         # Don't cancel running applies!

permissions:
  contents: read
  pull-requests: write              # Needed to comment on PRs
  id-token: write                   # Needed for OIDC auth with cloud providers

jobs:
  terraform-plan:
    name: Terraform Plan
    runs-on: ubuntu-latest
    if: github.event_name == 'pull_request'   # Only plan on PRs

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Terraform
        uses: hashicorp/setup-terraform@v3
        with:
          terraform_version: 1.6.0

      # Use OIDC for AWS auth (no static credentials!)
      - name: Configure AWS Credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789:role/github-actions-terraform
          aws-region: us-east-1

      - name: Terraform Init
        working-directory: infrastructure
        run: terraform init

      - name: Terraform Format Check
        working-directory: infrastructure
        run: terraform fmt -check -recursive

      - name: Terraform Validate
        working-directory: infrastructure
        run: terraform validate

      - name: Terraform Plan
        id: plan
        working-directory: infrastructure
        run: terraform plan -no-color -out=tfplan
        continue-on-error: true

      # Post the plan as a PR comment (like Atlantis does!)
      - name: Comment Plan on PR
        uses: actions/github-script@v7
        with:
          github-token: ${{ secrets.GITHUB_TOKEN }}
          script: |
            const output = `#### Terraform Plan 📖
            
            \`\`\`
            ${{ steps.plan.outputs.stdout }}
            \`\`\`
            
            *Pushed by: @${{ github.actor }}, Action: \`${{ github.event_name }}\`*`;
            
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: output
            })

      - name: Fail if Plan Failed
        if: steps.plan.outcome == 'failure'
        run: exit 1

  terraform-apply:
    name: Terraform Apply
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'  # Only apply on main
    environment: production         # Requires manual approval in GitHub!

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Terraform
        uses: hashicorp/setup-terraform@v3
        with:
          terraform_version: 1.6.0

      - name: Configure AWS Credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789:role/github-actions-terraform
          aws-region: us-east-1

      - name: Terraform Init
        working-directory: infrastructure
        run: terraform init

      - name: Terraform Apply
        working-directory: infrastructure
        run: terraform apply -auto-approve
```

**Line-by-line walkthrough of the key parts:**

```yaml
# concurrency block:
concurrency:
  group: terraform-${{ github.ref }}   # Group by branch
  cancel-in-progress: false            # NEVER cancel a running terraform apply!
  # If you cancel mid-apply, resources could be in a broken state!

# permissions block:
permissions:
  id-token: write   # This enables OIDC authentication
  # OIDC means GitHub Actions gets a SHORT-LIVED token from AWS
  # No static AWS_ACCESS_KEY_ID needed! Much more secure!

# environment: production
  environment: production
  # GitHub Environments can require MANUAL APPROVAL
  # Go to: Repo Settings → Environments → production → Required reviewers
  # This means: even after PR merge, a human must click "Approve" 
  # before terraform apply runs. Safety net! 🛡️
```

### Adding Security Scanning

A proper GitOps Terraform pipeline should also scan for security issues:

```yaml
  # Add these steps BEFORE terraform plan:

      - name: Run tfsec (Security Scanner)
        uses: aquasecurity/tfsec-action@v1.0.0
        with:
          working_directory: infrastructure

      - name: Run Checkov (Policy Scanner)
        uses: bridgecrewio/checkov-action@v12
        with:
          directory: infrastructure
          framework: terraform
          quiet: true                     # Only show failures

      - name: Run tflint (Linter)
        uses: terraform-linters/setup-tflint@v4
      - run: |
          cd infrastructure
          tflint --init
          tflint
```

These scanners catch things like:
- S3 buckets without encryption
- Security groups open to 0.0.0.0/0
- IAM policies that are too permissive
- Resources without required tags

## 🔧 Approach 4: tf-controller (Kubernetes-Native Terraform with Flux)

### What is tf-controller?

If you're already using Flux for GitOps, you might want to manage Terraform **through Kubernetes** as well. The `tf-controller` (now called **Tofu Controller**) lets you define Terraform runs as Kubernetes custom resources!

```
Traditional approach:
  Git repo → Atlantis/TFC → Cloud Provider

Kubernetes-native approach:
  Git repo → Flux → tf-controller → Cloud Provider

Everything runs INSIDE your Kubernetes cluster!
```

### How It Works

```
┌──────────────────────────────────────────────────────────────┐
│                     Kubernetes Cluster                        │
│                                                              │
│  ┌──────────────┐     ┌──────────────┐     ┌─────────────┐ │
│  │ Flux Source   │     │ Flux         │     │ tf-controller│ │
│  │ Controller    │────▶│ Kustomize    │────▶│             │ │
│  │               │     │ Controller   │     │ Runs        │ │
│  │ Watches Git   │     │              │     │ terraform   │ │
│  │ for changes   │     │ Applies CRDs │     │ plan/apply  │ │
│  └──────────────┘     └──────────────┘     └──────┬──────┘ │
│                                                    │        │
│                                                    ▼        │
│                                            ┌──────────────┐ │
│                                            │ Cloud        │ │
│                                            │ Provider     │ │
│                                            │ (AWS, GCP,   │ │
│                                            │  Azure)      │ │
│                                            └──────────────┘ │
└──────────────────────────────────────────────────────────────┘
```

### Installing tf-controller with Flux

```bash
# Add the tf-controller Helm repository as a Flux source
flux create source helm tf-controller \
  --url=https://flux-iac.github.io/tofu-controller \
  --interval=1h

# Install tf-controller using HelmRelease
flux create helmrelease tf-controller \
  --source=HelmRepository/tf-controller \
  --chart=tf-controller \
  --chart-version="0.16.0-rc.4" \
  --target-namespace=flux-system
```

Or as YAML:

```yaml
# tf-controller-source.yaml
apiVersion: source.toolkit.fluxcd.io/v1
kind: HelmRepository
metadata:
  name: tf-controller
  namespace: flux-system
spec:
  interval: 1h
  url: https://flux-iac.github.io/tofu-controller
  type: oci
---
# tf-controller-release.yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: tf-controller
  namespace: flux-system
spec:
  interval: 10m
  chart:
    spec:
      chart: tf-controller
      version: ">=0.16.0"
      sourceRef:
        kind: HelmRepository
        name: tf-controller
  install:
    createNamespace: true
```

### Using tf-controller — The Terraform CRD

```yaml
# terraform-s3.yaml
apiVersion: infra.contrib.fluxcd.io/v1alpha2
kind: Terraform
metadata:
  name: s3-bucket
  namespace: flux-system
spec:
  interval: 10m
  path: ./infrastructure/s3           # Path to Terraform code in the Git repo
  sourceRef:
    kind: GitRepository
    name: flux-system                  # The Flux GitRepository to use
  approvePlan: auto                    # "auto" = auto-apply, or "disable" = manual
  destroyResourcesOnDeletion: true     # Delete cloud resources when CR is deleted
  writeOutputsToSecret:
    name: s3-bucket-outputs            # Store TF outputs as a K8s Secret
    keys:
      - bucket_arn
      - bucket_name
  vars:                                # Terraform variables
    - name: bucket_name
      value: "my-data-bucket-prod"
    - name: environment
      value: "production"
  varsFrom:                            # Variables from Kubernetes Secrets
    - kind: Secret
      name: aws-credentials
```

**Explanation:**

- `path` → Where in your Git repo the Terraform files are
- `sourceRef` → Points to a Flux GitRepository (same one Flux uses for other resources)
- `approvePlan: auto` → Automatically apply after successful plan. Set to `"disable"` for manual approval
- `destroyResourcesOnDeletion: true` → If you delete this Kubernetes CR, tf-controller will run `terraform destroy`. Be careful with this!
- `writeOutputsToSecret` → Terraform outputs (like bucket ARN) are stored as Kubernetes Secrets. Other apps in the cluster can then use these values!
- `vars` → Terraform input variables, specified inline
- `varsFrom` → Terraform variables pulled from existing Kubernetes Secrets

### Manual Approval with tf-controller

For production infrastructure, you probably don't want auto-apply:

```yaml
apiVersion: infra.contrib.fluxcd.io/v1alpha2
kind: Terraform
metadata:
  name: production-vpc
  namespace: flux-system
spec:
  interval: 10m
  path: ./infrastructure/vpc
  sourceRef:
    kind: GitRepository
    name: flux-system
  approvePlan: "plan-main-abc123def456"    # Approve a SPECIFIC plan
  # When a new plan is generated, its name appears in the status.
  # You review it, then update this field to approve that specific plan.
  # This prevents "approve once, apply everything forever" problems.
```

**The approval workflow:**

```bash
# 1. tf-controller detects a change and generates a plan
kubectl get terraform production-vpc -n flux-system
# STATUS: Plan generated: plan-main-abc123def456

# 2. Review the plan
kubectl describe terraform production-vpc -n flux-system
# Shows the plan output in the status section

# 3. Approve by updating the spec (via Git PR, of course!)
# In your Git repo, update the Terraform CR:
#   approvePlan: "plan-main-abc123def456"
# Commit, push, Flux syncs, tf-controller applies!
```

### Environment Promotion Pattern

```
The GitOps way to promote infrastructure changes across environments:

1. Create a PR with changes in environments/dev/
   → Atlantis plans → Review → Apply to dev

2. Test in dev, verify everything works

3. Create another PR applying same changes to environments/staging/
   → Atlantis plans → Review → Apply to staging

4. Test in staging, verify everything works

5. Create another PR applying same changes to environments/production/
   → Atlantis plans → Team review → Security review → Apply to production

This gives you:
✅ Changes are tested in lower environments first
✅ Each promotion is a reviewed PR
✅ Full audit trail of what changed and when
✅ Easy rollback (revert the PR)

Visual:

  dev ──────▶ staging ──────▶ production
   PR #101      PR #102        PR #103
   (auto plan)  (auto plan)    (manual approval + plan)
```

## 🔐 Security Best Practices for Terraform GitOps

### 1. Never Store Credentials in Git

```
❌ BAD:
  # terraform.tfvars (committed to Git!)
  aws_access_key = "AKIAIOSFODNN7EXAMPLE"
  aws_secret_key = "wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY"

✅ GOOD:
  # Use OIDC / Workload Identity / IAM Roles
  # No static credentials anywhere!
```

### 2. Use OIDC for Cloud Authentication

```yaml
# GitHub Actions OIDC with AWS (no static keys!)
- name: Configure AWS Credentials
  uses: aws-actions/configure-aws-credentials@v4
  with:
    role-to-assume: arn:aws:iam::123456789:role/github-terraform
    aws-region: us-east-1
    # GitHub sends a short-lived JWT to AWS
    # AWS validates it and returns temporary credentials
    # Credentials expire automatically — no rotation needed!
```

### 3. Use Remote State with Encryption and Locking

```hcl
# backend.tf
terraform {
  backend "s3" {
    bucket         = "company-terraform-state"
    key            = "production/infrastructure/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true                    # Encrypt state at rest
    dynamodb_table = "terraform-state-locks" # Prevent concurrent applies
    
    # State bucket should have:
    # - Versioning enabled (recover from corruption)
    # - Access logging enabled (audit trail)
    # - Strict bucket policy (only CI/CD can access)
    # - MFA delete enabled (prevent accidental deletion)
  }
}
```

### 4. Use Policy-as-Code

```yaml
# Run OPA/Sentinel/Checkov policies in your pipeline
# Example: Block public S3 buckets

# Checkov custom check
# .checkov/custom_checks/no_public_buckets.py
from checkov.terraform.checks.resource.base_resource_check import BaseResourceCheck

class NoPublicBuckets(BaseResourceCheck):
    def __init__(self):
        name = "Ensure S3 buckets are not publicly accessible"
        id = "CUSTOM_001"
        supported_resources = ["aws_s3_bucket"]
        super().__init__(name=name, id=id, supported_resources=supported_resources)

    def scan_resource_conf(self, conf):
        # Check for public ACL
        acl = conf.get("acl", ["private"])
        if acl[0] in ["public-read", "public-read-write"]:
            return "FAILED"
        return "PASSED"
```

### 5. Restrict Who Can Approve/Apply

```yaml
# atlantis.yaml — restrict apply to specific teams
projects:
  - name: production
    dir: environments/production
    apply_requirements:
      - approved                    # Must be approved
      - mergeable                   # Must pass all checks
    allowed_overrides:
      - apply_requirements
    # Only certain teams can comment "atlantis apply"
    # Configure this in server-side config:
    # repos:
    #   - id: github.com/your-org/infrastructure
    #     allowed_overrides: [apply_requirements]
    #     apply_requirements: [approved, mergeable]
```

## ⚠️ Common Pitfalls & How to Avoid Them

### Pitfall 1: State File Conflicts

```
Problem: Two PRs modify the same Terraform state simultaneously.

Solution:
  - Use remote state with LOCKING (DynamoDB for S3 backend)
  - Atlantis has built-in locking (only one plan/apply at a time per project)
  - Terraform Cloud has built-in locking
  - NEVER run terraform locally if a CI/CD system is managing it
```

### Pitfall 2: Plan-Apply Mismatch

```
Problem: Plan shows "2 to add" but by the time you apply, 
         someone else changed the infrastructure, causing a different result.

Solution:
  - Use Atlantis's "undiverged" apply requirement
  - Terraform Cloud re-plans before applying
  - Always plan and apply as an atomic operation
  - Use concurrency controls to prevent parallel applies
```

### Pitfall 3: Secrets in State Files

```
Problem: Terraform state contains sensitive values in plain text
         (database passwords, API keys, etc.)

Solution:
  - Encrypt state at rest (S3 encryption, GCS encryption)
  - Restrict access to state backend (IAM policies)
  - Use data sources instead of storing secrets in state
  - Use external secret managers (AWS Secrets Manager, Vault)
  - NEVER commit state files to Git!
```

### Pitfall 4: Blast Radius Too Large

```
Problem: One Terraform project manages ALL infrastructure.
         A single bad apply could destroy everything.

Solution:
  - Break infrastructure into smaller, independent state files
  - Use the directory structure shown above (per-service or per-layer)
  - Use terraform_remote_state data source to reference between projects
  
  Example:
    infrastructure/
    ├── networking/    → State: networking.tfstate    (VPC, subnets)
    ├── compute/       → State: compute.tfstate       (EKS, EC2)
    ├── database/      → State: database.tfstate      (RDS, Redis)
    └── monitoring/    → State: monitoring.tfstate     (CloudWatch)
    
    Each is independent. Breaking networking won't affect the database state.
```
