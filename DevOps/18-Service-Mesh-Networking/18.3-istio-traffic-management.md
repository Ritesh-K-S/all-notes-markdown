# 18.3 — Istio Traffic Management — VirtualService, DestinationRule, Gateway

## 🤔 Why Do You Need Traffic Management?

Imagine you have a `payment-service` with two versions running:

- **v1** — the stable, current version
- **v2** — a new version you want to test

In plain Kubernetes, when you create a Service, it load balances **equally** across all Pods matching the selector — you can't control the ratio:

```
Plain Kubernetes Service:
                        ┌──────────────────┐
                   ┌───→│ payment-svc v1   │  50%
                   │    └──────────────────┘
  ┌──────────┐     │
  │ Service  │─────┤
  └──────────┘     │    ┌──────────────────┐
                   └───→│ payment-svc v2   │  50%
                        └──────────────────┘
  
  You can't say "send 95% to v1 and 5% to v2"!
  You can't route based on headers or user identity!
  You can't add retries, timeouts, or fault injection!
```

**Istio's traffic management resources give you fine-grained control over ALL of this.**

## 🧩 The Three Amigos — Gateway, VirtualService, DestinationRule

These three resources work together like a team:

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Traffic Management Flow                           │
│                                                                      │
│   External                                                           │
│   User/Client                                                        │
│       │                                                              │
│       ▼                                                              │
│  ┌──────────┐    "Which hosts and ports                              │
│  │ GATEWAY  │     should I accept traffic for?"                      │
│  │          │     (Entry point — like a front door)                   │
│  └────┬─────┘                                                        │
│       │                                                              │
│       ▼                                                              │
│  ┌──────────────┐  "WHERE should I route this traffic?"              │
│  │VIRTUAL       │   (Routing rules — like a GPS)                     │
│  │SERVICE       │   • Route to v1 or v2?                             │
│  │              │   • Split traffic 90/10?                            │
│  │              │   • Match by header, URI, method?                   │
│  │              │   • Add retries, timeouts, fault injection?         │
│  └────┬─────────┘                                                    │
│       │                                                              │
│       ▼                                                              │
│  ┌──────────────┐  "HOW should I connect to the destination?"        │
│  │DESTINATION   │   (Policies — like road conditions)                │
│  │RULE          │   • Define subsets (v1, v2, v3)                    │
│  │              │   • Load balancing algorithm                        │
│  │              │   • Connection pool settings                        │
│  │              │   • Circuit breaking                                │
│  │              │   • TLS settings                                    │
│  └──────────────┘                                                    │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

## 🚪 Gateway — The Front Door

A **Gateway** configures the Istio Ingress Gateway (the Envoy proxy sitting at the edge of your mesh) to accept incoming traffic on specific **hosts** and **ports**.

> **Analogy:** Think of a Gateway as the **front door of a hotel**. It defines which entrances are open, what protocols they accept (HTTP, HTTPS), and which domain names they respond to.

### Basic Gateway Example:

```yaml
# gateway.yaml
apiVersion: networking.istio.io/v1beta1
kind: Gateway
metadata:
  name: my-gateway
  namespace: default
spec:
  # Which Istio gateway pod should use this config?
  selector:
    istio: ingressgateway       # Use the default Istio ingress gateway
  
  # Define the entry points (servers)
  servers:
    - port:
        number: 80              # Listen on port 80
        name: http
        protocol: HTTP          # Accept HTTP traffic
      hosts:
        - "myapp.example.com"   # Only for this hostname
```

**What this does:**

```
Internet → DNS (myapp.example.com) → Istio Ingress Gateway (port 80)
                                          │
                                     Gateway says:
                                     "Yes, I accept traffic for
                                      myapp.example.com on port 80"
                                          │
                                          ▼
                                     Next: VirtualService decides
                                     WHERE to route this traffic
```

### Gateway with HTTPS/TLS:

```yaml
# gateway-https.yaml
apiVersion: networking.istio.io/v1beta1
kind: Gateway
metadata:
  name: my-secure-gateway
spec:
  selector:
    istio: ingressgateway
  servers:
    # HTTPS on port 443
    - port:
        number: 443
        name: https
        protocol: HTTPS
      hosts:
        - "myapp.example.com"
        - "api.example.com"        # Multiple hosts allowed!
      tls:
        mode: SIMPLE               # One-way TLS (standard HTTPS)
        credentialName: my-tls-cert # Kubernetes Secret with the TLS cert
    
    # HTTP on port 80 — redirect to HTTPS
    - port:
        number: 80
        name: http
        protocol: HTTP
      hosts:
        - "myapp.example.com"
      tls:
        httpsRedirect: true        # Automatically redirect HTTP → HTTPS
```

```bash
# Create the TLS secret (using cert-manager or manually)
kubectl create secret tls my-tls-cert \
  --cert=path/to/cert.pem \
  --key=path/to/key.pem \
  -n istio-system              # Must be in the same namespace as the gateway pod!
```

### TLS Modes Explained:

| Mode | Description | Use Case |
|------|-------------|----------|
| `SIMPLE` | Standard one-way TLS (server presents certificate) | Regular HTTPS websites |
| `MUTUAL` | Mutual TLS (both client and server present certificates) | API-to-API, high security |
| `PASSTHROUGH` | Don't terminate TLS, pass encrypted traffic through | When the backend handles TLS |
| `AUTO_PASSTHROUGH` | Like PASSTHROUGH but with SNI-based routing | Multi-cluster setups |
| `ISTIO_MUTUAL` | Istio's internal mTLS (automatic cert management) | Service-to-service within mesh |

### Gateway with Multiple Servers:

```yaml
# Handle different domains on different ports/protocols
apiVersion: networking.istio.io/v1beta1
kind: Gateway
metadata:
  name: multi-gateway
spec:
  selector:
    istio: ingressgateway
  servers:
    # Frontend app — HTTPS
    - port:
        number: 443
        name: https-frontend
        protocol: HTTPS
      hosts:
        - "www.myapp.com"
      tls:
        mode: SIMPLE
        credentialName: frontend-cert
    
    # API — HTTPS
    - port:
        number: 443
        name: https-api
        protocol: HTTPS
      hosts:
        - "api.myapp.com"
      tls:
        mode: SIMPLE
        credentialName: api-cert
    
    # gRPC service — on port 8443
    - port:
        number: 8443
        name: grpc
        protocol: GRPC
      hosts:
        - "grpc.myapp.com"
```

## 🗺️ VirtualService — The GPS / Router

A **VirtualService** defines the **routing rules** — it tells Istio WHERE to send traffic and WHAT to do with it along the way.

> **Analogy:** If the Gateway is the front door, the VirtualService is the **receptionist** inside the hotel. When a guest (request) arrives, the receptionist checks their name (URL, headers, etc.) and directs them to the right room (service/version).

### Basic VirtualService Example:

```yaml
# virtualservice-basic.yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: my-app-vs
spec:
  # Which hostnames does this VirtualService apply to?
  hosts:
    - "myapp.example.com"
  
  # Attach to which Gateway? (for external traffic)
  gateways:
    - my-gateway                # References the Gateway we created above
  
  # Define the routing rules
  http:
    - route:
        - destination:
            host: my-app-service    # Kubernetes Service name
            port:
              number: 8080          # Service port
```

**Traffic flow:**

```
Client → Gateway (myapp.example.com:80)
              → VirtualService (routing rules)
                   → my-app-service:8080 (Kubernetes Service)
                        → Pod(s)
```

### 🎯 Feature 1: Traffic Splitting (Canary Deployments)

This is one of the MOST powerful features. You can split traffic between different versions:

```yaml
# virtualservice-canary.yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: payment-service-vs
spec:
  hosts:
    - payment-service            # Internal service name (mesh traffic)
  http:
    - route:
        - destination:
            host: payment-service
            subset: v1           # Defined in DestinationRule (see below)
          weight: 90             # 90% of traffic → v1 (stable)
        - destination:
            host: payment-service
            subset: v2           # New version
          weight: 10             # 10% of traffic → v2 (canary)
```

**Gradual canary rollout strategy:**

```
Day 1:   v1: 95%  |  v2: 5%     ← Testing with tiny traffic
Day 2:   v1: 90%  |  v2: 10%    ← Looking good, increase
Day 3:   v1: 75%  |  v2: 25%    ← Monitoring closely
Day 4:   v1: 50%  |  v2: 50%    ← Half and half
Day 5:   v1: 0%   |  v2: 100%   ← Full rollout!

All done with just YAML changes — no code changes, no redeployments!
Just: kubectl apply -f virtualservice-canary.yaml
```

### 🎯 Feature 2: Header-Based Routing

Route traffic based on HTTP headers — perfect for testing:

```yaml
# virtualservice-header-routing.yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: reviews-vs
spec:
  hosts:
    - reviews
  http:
    # Rule 1: If the user is "jason", send to v2 (black stars)
    - match:
        - headers:
            end-user:
              exact: jason
      route:
        - destination:
            host: reviews
            subset: v2
    
    # Rule 2: If the header "x-test-version" is "v3", send to v3
    - match:
        - headers:
            x-test-version:
              exact: "v3"
      route:
        - destination:
            host: reviews
            subset: v3
    
    # Rule 3: Default — everyone else goes to v1
    - route:
        - destination:
            host: reviews
            subset: v1
```

**How teams use this in practice:**

```bash
# QA team testing v3 — they add a header to their requests:
curl -H "x-test-version: v3" http://myapp.example.com/reviews

# Normal users — get v1 (stable version)
curl http://myapp.example.com/reviews

# Specific test user "jason" — always gets v2
# (login as jason → header automatically set by the app)
```

> **This means you can test new versions in production with REAL traffic — but only for specific users or with specific headers!** Other users are completely unaffected.

### 🎯 Feature 3: URI-Based Routing (Path-Based)

Route traffic based on the URL path:

```yaml
# virtualservice-uri-routing.yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: my-app-vs
spec:
  hosts:
    - "myapp.example.com"
  gateways:
    - my-gateway
  http:
    # /api/v1/* → api-service-v1
    - match:
        - uri:
            prefix: /api/v1
      route:
        - destination:
            host: api-service-v1
            port:
              number: 8080
    
    # /api/v2/* → api-service-v2
    - match:
        - uri:
            prefix: /api/v2
      route:
        - destination:
            host: api-service-v2
            port:
              number: 8080
    
    # /static/* → static-content-service
    - match:
        - uri:
            prefix: /static
      route:
        - destination:
            host: static-content-service
            port:
              number: 80
    
    # Everything else → frontend
    - route:
        - destination:
            host: frontend-service
            port:
              number: 3000
```

### URI Match Types:

| Match Type | Example | Matches |
|-----------|---------|---------|
| `exact` | `/api/users` | Only exactly `/api/users` |
| `prefix` | `/api/` | `/api/users`, `/api/orders`, `/api/anything` |
| `regex` | `"/api/v[0-9]+/.*"` | `/api/v1/users`, `/api/v2/orders` |

### 🎯 Feature 4: URL Rewriting

Change the URL before sending it to the backend:

```yaml
# virtualservice-rewrite.yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: rewrite-example
spec:
  hosts:
    - "myapp.example.com"
  gateways:
    - my-gateway
  http:
    - match:
        - uri:
            prefix: /reviews        # Client sends: /reviews/123
      rewrite:
        uri: /api/v1/reviews         # Backend receives: /api/v1/reviews/123
      route:
        - destination:
            host: reviews-service
```

### 🎯 Feature 5: Retries

Automatically retry failed requests:

```yaml
# virtualservice-retries.yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: payment-vs
spec:
  hosts:
    - payment-service
  http:
    - route:
        - destination:
            host: payment-service
            subset: v1
      retries:
        attempts: 3                 # Retry up to 3 times
        perTryTimeout: 2s           # Each retry has a 2-second timeout
        retryOn: 5xx,reset,connect-failure,retriable-4xx
        # retryOn options:
        #   5xx              — retry on any 5xx error
        #   reset            — retry on connection reset
        #   connect-failure  — retry on connection failure
        #   retriable-4xx    — retry on 409 Conflict
        #   gateway-error    — retry on 502, 503, 504
```

**How retries work:**

```
Request → payment-service
              │
              ├─ Attempt 1: ❌ 503 (service unavailable)
              │   (wait, retry)
              ├─ Attempt 2: ❌ Connection reset
              │   (wait, retry)
              ├─ Attempt 3: ✅ 200 OK!
              │
         Response returned to client: 200 OK
         
Client has NO IDEA retries happened — it's transparent!
```

### 🎯 Feature 6: Timeouts

Set maximum time to wait for a response:

```yaml
# virtualservice-timeout.yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: slow-service-vs
spec:
  hosts:
    - slow-service
  http:
    - route:
        - destination:
            host: slow-service
      timeout: 5s                  # If no response in 5 seconds → 504 Gateway Timeout
```

**Combining retries + timeout:**

```yaml
http:
  - route:
      - destination:
          host: payment-service
    timeout: 10s                     # Total timeout for the entire request (including retries)
    retries:
      attempts: 3
      perTryTimeout: 3s             # Each individual retry: 3 seconds max
      # So: 3 retries × 3s = 9s max (fits within the 10s total timeout)
```

### 🎯 Feature 7: Fault Injection (Chaos Engineering)

Intentionally inject faults to test your system's resilience:

```yaml
# virtualservice-fault-injection.yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: ratings-vs
spec:
  hosts:
    - ratings
  http:
    - fault:
        # Inject a delay for 50% of requests
        delay:
          percentage:
            value: 50.0              # 50% of requests
          fixedDelay: 5s             # will be delayed by 5 seconds
        
        # Inject HTTP 500 errors for 10% of requests
        abort:
          percentage:
            value: 10.0              # 10% of requests
          httpStatus: 500            # will get a 500 error
      
      route:
        - destination:
            host: ratings
            subset: v1
```

**Why would you intentionally break things?**

```
Scenario: "What happens if the ratings service is slow?"

Without fault injection:
  You have to wait for a REAL outage to find out. 😰

With fault injection:
  Inject a 5-second delay → See if the UI shows a loading spinner
  or if it crashes. Fix it BEFORE the real outage happens. 😌

This is the core idea of CHAOS ENGINEERING!
```

### 🎯 Feature 8: Traffic Mirroring (Shadow Traffic)

Send a copy of live traffic to another service for testing — **without affecting the real users**:

```yaml
# virtualservice-mirroring.yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: payment-vs
spec:
  hosts:
    - payment-service
  http:
    - route:
        - destination:
            host: payment-service
            subset: v1              # Real traffic goes to v1
      mirror:
        host: payment-service
        subset: v2                  # A COPY goes to v2 (shadow traffic)
      mirrorPercentage:
        value: 100.0                # Mirror 100% of traffic
```

**How mirroring works:**

```
Client → payment-service v1 (REAL traffic — response goes back to client)
              │
              └──→ payment-service v2 (SHADOW — response is DISCARDED)

Benefits:
  ✅ Test v2 with REAL production traffic
  ✅ Compare v2's behavior with v1
  ✅ Zero risk — v2's response is thrown away
  ✅ Find bugs before actual rollout
```

## 📦 DestinationRule — The Road Conditions

A **DestinationRule** defines policies that apply **after** routing has happened. It controls HOW traffic reaches the destination.

> **Analogy:** If the VirtualService is the GPS telling you which road to take, the DestinationRule is the **road conditions and driving rules** — speed limits (connection pools), detour signs (circuit breakers), and road types (TLS settings).

### Key Responsibilities:

```
┌─────────────────────────────────────────────────────────────┐
│                   DestinationRule                             │
│                                                              │
│  1. SUBSETS     — Define versions (v1, v2, v3) using labels  │
│  2. LOAD BAL.   — Algorithm (round-robin, random, least req) │
│  3. CONN. POOL  — Max connections, requests per connection   │
│  4. OUTLIER DET.— Circuit breaking (eject unhealthy hosts)   │
│  5. TLS         — Configure TLS mode for the destination     │
└─────────────────────────────────────────────────────────────┘
```

---

### 🏷️ Feature 1: Subsets — Defining Versions

Subsets let you group Pod instances by **labels** (like version). This is what VirtualService references when routing to `v1` or `v2`:

```yaml
# destinationrule-subsets.yaml
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: payment-service-dr
spec:
  host: payment-service              # The Kubernetes Service name
  subsets:
    - name: v1                        # Subset name (referenced by VirtualService)
      labels:
        version: v1                   # Pods with label version=v1
    - name: v2
      labels:
        version: v2                   # Pods with label version=v2
    - name: v3
      labels:
        version: v3                   # Pods with label version=v3
```

**How this connects with Kubernetes Deployments:**

```yaml
# deployment-v1.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: payment-service-v1
spec:
  replicas: 3
  selector:
    matchLabels:
      app: payment-service
      version: v1                     # ← This label is what the subset matches!
  template:
    metadata:
      labels:
        app: payment-service
        version: v1                   # ← Must match!
    spec:
      containers:
        - name: payment
          image: myapp/payment:1.0.0

---
# deployment-v2.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: payment-service-v2
spec:
  replicas: 1
  selector:
    matchLabels:
      app: payment-service
      version: v2                     # ← Different version label
  template:
    metadata:
      labels:
        app: payment-service
        version: v2
    spec:
      containers:
        - name: payment
          image: myapp/payment:2.0.0

---
# A single Kubernetes Service for BOTH versions
apiVersion: v1
kind: Service
metadata:
  name: payment-service
spec:
  selector:
    app: payment-service              # Selects BOTH v1 and v2 Pods
  ports:
    - port: 8080
```

**The complete picture:**

```
                         VirtualService
                         (90% v1, 10% v2)
                              │
                    ┌─────────┴─────────┐
                    │                   │
              DestinationRule       DestinationRule
              subset: v1            subset: v2
              (version=v1)          (version=v2)
                    │                   │
              ┌─────┴─────┐            │
              │     │     │            │
            Pod1  Pod2  Pod3         Pod4
           (v1)  (v1)  (v1)         (v2)
```

### ⚖️ Feature 2: Load Balancing

Control how traffic is distributed across instances:

```yaml
# destinationrule-loadbalancing.yaml
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: payment-service-dr
spec:
  host: payment-service
  trafficPolicy:
    loadBalancer:
      simple: LEAST_REQUEST          # Send to the instance with fewest active requests
  subsets:
    - name: v1
      labels:
        version: v1
    - name: v2
      labels:
        version: v2
      trafficPolicy:                  # You can override per subset!
        loadBalancer:
          simple: ROUND_ROBIN
```

**Load Balancing Algorithms:**

| Algorithm | How It Works | Best For |
|-----------|-------------|----------|
| `ROUND_ROBIN` | Rotates through instances one by one | Default, general use |
| `LEAST_REQUEST` | Sends to instance with fewest active requests | Uneven processing times |
| `RANDOM` | Picks a random instance | Simple, works well at scale |
| `PASSTHROUGH` | Doesn't do any load balancing | When you handle it yourself |

**Consistent Hashing (Sticky Sessions):**

```yaml
# Route requests from the same user to the same Pod (session affinity)
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: sticky-sessions-dr
spec:
  host: my-service
  trafficPolicy:
    loadBalancer:
      consistentHash:
        httpHeaderName: x-user-id      # Hash based on user ID header
        # OR:
        # httpCookie:                  # Hash based on cookie
        #   name: JSESSIONID
        #   ttl: 3600s
        # OR:
        # useSourceIp: true           # Hash based on client IP
```

### 🔌 Feature 3: Connection Pool Settings

Control the maximum number of connections and requests:

```yaml
# destinationrule-connection-pool.yaml
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: payment-service-dr
spec:
  host: payment-service
  trafficPolicy:
    connectionPool:
      tcp:
        maxConnections: 100            # Max TCP connections to this service
        connectTimeout: 5s             # TCP connection timeout
      http:
        h2UpgradePolicy: DEFAULT       # Use HTTP/2 when possible
        http1MaxPendingRequests: 100   # Max queued HTTP/1.1 requests
        http2MaxRequests: 1000         # Max active HTTP/2 requests
        maxRequestsPerConnection: 10   # Max requests per TCP connection
        maxRetries: 3                  # Max retries across all hosts
```

**Why is this important?**

```
Without connection pool limits:
  If payment-service is slow, thousands of connections pile up
  → Memory exhaustion → Cascading failures → Everything crashes! 💥

With connection pool limits:
  Max 100 connections. If exceeded:
  → New requests get 503 (service unavailable)
  → Other services are protected
  → The problem is CONTAINED 🛡️
```

### 🔒 Feature 4: Circuit Breaking (Outlier Detection)

Automatically stop sending traffic to unhealthy instances:

```yaml
# destinationrule-circuit-breaker.yaml
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: payment-service-dr
spec:
  host: payment-service
  trafficPolicy:
    outlierDetection:
      consecutive5xxErrors: 3          # After 3 consecutive 5xx errors...
      interval: 10s                    # ...checked every 10 seconds...
      baseEjectionTime: 30s           # ...eject the host for 30 seconds
      maxEjectionPercent: 50          # Never eject more than 50% of hosts
      minHealthPercent: 30            # Keep at least 30% hosts in pool
```

**How circuit breaking works:**

```
                    ┌──────────────────┐
              ┌────→│ Pod 1 (healthy)  │ ✅ Getting traffic
              │     └──────────────────┘
              │
  Envoy ──────┤     ┌──────────────────┐
  (Load       │     │ Pod 2 (failing)  │ ❌ 3 consecutive 5xx errors
  Balancer)   │     └──────────────────┘    → EJECTED for 30 seconds!
              │                               (no traffic sent here)
              │     ┌──────────────────┐
              └────→│ Pod 3 (healthy)  │ ✅ Getting more traffic now
                    └──────────────────┘

After 30 seconds:
  Pod 2 is allowed back → if it returns 5xx again → ejected again (for longer)
  This is called the "circuit breaker" pattern!
```

### 🔐 Feature 5: TLS Settings

Control how Envoy connects to the upstream service:

```yaml
# destinationrule-tls.yaml
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: external-api-dr
spec:
  host: external-api.example.com       # An external service
  trafficPolicy:
    tls:
      mode: SIMPLE                      # Use TLS when connecting
      # Options:
      # DISABLE     — No TLS (plain text)
      # SIMPLE      — One-way TLS (client verifies server)
      # MUTUAL      — mTLS (both sides verify each other)
      # ISTIO_MUTUAL — Use Istio's auto-managed mTLS certificates
```

### Complete DestinationRule Example:

```yaml
# destinationrule-complete.yaml
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: payment-service-dr
spec:
  host: payment-service

        connectionPool:
          http:
            http2MaxRequests: 50       # v2 is new, be more cautious
```

## 🔗 How Everything Connects — Full Example

Let's put **Gateway + VirtualService + DestinationRule** together for a complete setup:

### Scenario:

- E-commerce app accessible at `shop.example.com`
- Frontend and API services
- Canary deployment for the API (v1: 90%, v2: 10%)
- Circuit breaking enabled

```yaml
# 1. GATEWAY — Accept external traffic
apiVersion: networking.istio.io/v1beta1
kind: Gateway
metadata:
  name: shop-gateway
spec:
  selector:
    istio: ingressgateway
  servers:
    - port:
        number: 443
        name: https
        protocol: HTTPS
      hosts:
        - "shop.example.com"
      tls:
        mode: SIMPLE
        credentialName: shop-tls-cert

---
# 2. VIRTUALSERVICE — Define routing rules
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: shop-vs
spec:
  hosts:
    - "shop.example.com"
  gateways:
    - shop-gateway
  http:
    # Route /api/* to the API service (with canary split)
    - match:
        - uri:
            prefix: /api
      route:
        - destination:
            host: api-service
            subset: v1
          weight: 90
        - destination:
            host: api-service
            subset: v2
          weight: 10
      timeout: 10s
      retries:
        attempts: 3
        perTryTimeout: 3s
        retryOn: 5xx,connect-failure
    
    # Route everything else to the frontend
    - route:
        - destination:
            host: frontend-service
            port:
              number: 3000

---
# 3. DESTINATIONRULE — Define subsets and policies for the API
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: api-service-dr
spec:
  host: api-service
  trafficPolicy:
    loadBalancer:
      simple: LEAST_REQUEST
    connectionPool:
      tcp:
        maxConnections: 200
      http:
        http2MaxRequests: 500
    outlierDetection:
      consecutive5xxErrors: 3
      interval: 10s
      baseEjectionTime: 30s
      maxEjectionPercent: 50
    tls:
      mode: ISTIO_MUTUAL
  subsets:
    - name: v1
      labels:
        version: v1
    - name: v2
      labels:
        version: v2

---
# 4. DESTINATIONRULE — Policies for the frontend
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: frontend-service-dr
spec:
  host: frontend-service
  trafficPolicy:
    loadBalancer:
      simple: ROUND_ROBIN
    outlierDetection:
      consecutive5xxErrors: 5
      interval: 15s
      baseEjectionTime: 60s
```

**Complete traffic flow:**

```
User (Browser)
     │
     │  HTTPS (shop.example.com)
     ▼
┌────────────────────────┐
│  Istio Ingress Gateway │  ← Gateway config: accept HTTPS on shop.example.com
│  (Envoy)               │
└────────┬───────────────┘
         │
         ▼
┌────────────────────────┐
│   VirtualService       │  ← Routing rules
│                        │
│  /api/* → api-service  │     90% → v1, 10% → v2 (canary)
│  /*     → frontend     │     retries: 3, timeout: 10s
└────┬──────────┬────────┘
     │          │
     ▼          ▼
┌──────────┐  ┌──────────────┐
│ frontend │  │ api-service  │  ← DestinationRule: subsets, load balancing,
│ (v1)     │  │ v1(90%) v2   │    circuit breaking, mTLS
│ 3 pods   │  │ 3pods  1pod  │
└──────────┘  └──────────────┘
```

## 🛠️ Debugging Traffic Management

When things don't work as expected, use these commands:

```bash
# 1. Analyze configuration for errors
istioctl analyze

# Output examples:
# ✔ No validation issues found
# or
# ⚠ [IST0101] VirtualService "payment-vs" references undefined subset "v4"

# 2. Check what routes a specific proxy knows about
istioctl proxy-config routes deploy/productpage-v1

# 3. Check what clusters (upstream services) a proxy knows about
istioctl proxy-config clusters deploy/productpage-v1

# 4. Check endpoints
istioctl proxy-config endpoints deploy/productpage-v1 \
  --cluster "outbound|8080|v1|payment-service.default.svc.cluster.local"

# 5. Check if traffic policies are applied
istioctl proxy-config cluster deploy/productpage-v1 \
  -o json | grep -A 20 "payment-service"

# 6. View Envoy access logs (see actual request routing)
kubectl logs deploy/productpage-v1 -c istio-proxy --tail=20

# Example log line:
# [2024-01-15T10:30:00.000Z] "GET /api/payment HTTP/1.1" 200 
#   - via_upstream - "-" 0 178 12 11 
#   "-" "curl/7.68.0" "abc-123-trace-id" 
#   "payment-service.default.svc.cluster.local:8080" 
#   "10.244.0.15:8080" 
#   outbound|8080|v1|payment-service.default.svc.cluster.local

# 7. Verify subsets are correctly defined
kubectl get destinationrule payment-service-dr -o yaml

# 8. Check VirtualService
kubectl get virtualservice payment-vs -o yaml
```

## ⚠️ Common Mistakes and How to Fix Them

### Mistake 1: VirtualService References a Subset That Doesn't Exist

```yaml
# ❌ WRONG — subset "v2" referenced but not defined in DestinationRule
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: my-vs
spec:
  hosts:
    - my-service
  http:
    - route:
        - destination:
            host: my-service
            subset: v2              # ← This subset must exist in DestinationRule!
```

```bash
# Fix: Make sure the DestinationRule defines the subset
istioctl analyze
# Output: ⚠ [IST0101] Referenced subset "v2" not found in DestinationRule
```

### Mistake 2: Forgetting to Label Pods with Version

```yaml
# ❌ WRONG — Pods don't have version label
spec:
  template:
    metadata:
      labels:
        app: my-service
        # Missing: version: v1    ← DestinationRule subset can't match!
```

### Mistake 3: Gateway and VirtualService Host Mismatch

```yaml
# ❌ WRONG — Gateway listens on "myapp.com" but VirtualService uses "api.myapp.com"
# Gateway:
  hosts:
    - "myapp.com"

# VirtualService:
  hosts:
    - "api.myapp.com"           # ← Doesn't match! Traffic won't route.
```

### Mistake 4: Weights Don't Add Up to 100

```yaml
# ❌ WRONG — Weights must total 100
http:
  - route:
      - destination:
          host: my-service
          subset: v1
        weight: 80
      - destination:
          host: my-service
          subset: v2
        weight: 30              # 80 + 30 = 110 ← ERROR!
```

### Mistake 5: Namespace Issues

```yaml
# If your VirtualService is in namespace "prod" but the service is in "default":
http:
  - route:
      - destination:
          host: payment-service.default.svc.cluster.local   # ← Use FQDN!
          # NOT just "payment-service" (that would look in the "prod" namespace)
```

## 📋 Quick Reference — All Match Conditions

VirtualService supports these match conditions:

```yaml
http:
  - match:
      - uri:                          # URL path matching
          exact: /api/v1/users
          prefix: /api/
          regex: "/api/v[0-9]+/.*"
        
        headers:                      # HTTP header matching
          x-custom-header:
            exact: "my-value"
            prefix: "test-"
            regex: "v[0-9]+"
        
        method:                       # HTTP method matching
          exact: GET
        
        queryParams:                  # Query parameter matching
          user:
            exact: "admin"
            regex: "user-[0-9]+"
        
        sourceLabels:                 # Source pod labels
          app: frontend
          version: v2
        
        port: 8080                    # Specific port
```

## 📋 Summary — Key Takeaways

| Resource | Purpose | Analogy |
|----------|---------|---------|
| **Gateway** | Entry point for external traffic (hosts, ports, TLS) | Hotel front door |
| **VirtualService** | Routing rules (where to send traffic) | GPS / Receptionist |
| **DestinationRule** | Policies for destination (how to send traffic) | Road conditions / Driving rules |

| Feature | Resource | Example |
|---------|----------|---------|
| Traffic splitting (canary) | VirtualService | `weight: 90` / `weight: 10` |
| Header-based routing | VirtualService | `match.headers.x-test: exact: canary` |
| URI/path routing | VirtualService | `match.uri.prefix: /api` |
| Retries | VirtualService | `retries.attempts: 3` |
| Timeouts | VirtualService | `timeout: 10s` |
| Fault injection | VirtualService | `fault.delay` / `fault.abort` |
| Traffic mirroring | VirtualService | `mirror.host: service-v2` |
| Subsets (versions) | DestinationRule | `subsets.labels.version: v1` |
| Load balancing | DestinationRule | `loadBalancer.simple: LEAST_REQUEST` |
| Circuit breaking | DestinationRule | `outlierDetection.consecutive5xxErrors: 3` |
| Connection pools | DestinationRule | `connectionPool.tcp.maxConnections: 100` |
| TLS settings | DestinationRule | `tls.mode: ISTIO_MUTUAL` |
