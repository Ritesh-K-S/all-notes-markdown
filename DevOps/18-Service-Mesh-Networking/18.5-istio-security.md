## 1. Why Do We Need Security in a Service Mesh?

Let's say you have 20 microservices running inside your Kubernetes cluster. You might think: "They're inside the cluster, so they're safe, right?"

**Wrong.** Here's why:

### The Problem: Trust No One (Zero Trust)

```
Traditional thinking (WRONG):
"Everything inside our network is trusted"

┌──────────────────────────────────────────────┐
│  Kubernetes Cluster (Trusted Zone?)          │
│                                              │
│  Service A ──── plain text ────▶ Service B   │
│                                              │
│  Anyone who compromises ONE pod can:         │
│  ❌ Sniff traffic between services           │
│  ❌ Impersonate any service                  │
│  ❌ Access any other service freely           │
│  ❌ Perform man-in-the-middle attacks        │
└──────────────────────────────────────────────┘

Zero Trust thinking (CORRECT):
"Never trust, always verify — even inside the cluster"

┌──────────────────────────────────────────────┐
│  Kubernetes Cluster (Zero Trust)             │
│                                              │
│  Service A ──── encrypted (mTLS) ──▶ Service B│
│                                              │
│  ✅ All traffic is encrypted                 │
│  ✅ Every service proves its identity        │
│  ✅ Access is explicitly granted (deny by default)│
│  ✅ Even compromised pods can't sniff traffic│
└──────────────────────────────────────────────┘
```

### What Can Go Wrong Without Service Mesh Security?

| Threat | Description | Impact |
|--------|-------------|--------|
| **Eavesdropping** | Attacker sniffs network traffic between pods | Sensitive data (passwords, tokens, PII) leaked |
| **Spoofing** | Attacker pretends to be a legitimate service | Unauthorized access to data |
| **Man-in-the-Middle** | Attacker intercepts and modifies traffic | Data tampering, injecting malicious responses |
| **Lateral movement** | Compromised pod freely accesses all other services | One breach = total breach |
| **Privilege escalation** | Frontend directly calls admin APIs | Unauthorized actions |

Istio solves ALL of these problems.

## 2. Istio Security Architecture — The Big Picture

Istio's security has three main components:

```
┌───────────────────────────────────────────────────────────┐
│                    ISTIO SECURITY                          │
├───────────────────┬──────────────────┬────────────────────┤
│    IDENTITY       │  ENCRYPTION      │  ACCESS CONTROL    │
│   (Who are you?)  │ (Protect data)   │ (Are you allowed?) │
├───────────────────┼──────────────────┼────────────────────┤
│ SPIFFE identity   │ Mutual TLS       │ AuthorizationPolicy│
│ X.509 certificates│ (mTLS)           │ (allow/deny rules) │
│ Automatic cert    │ Auto-encryption  │ Fine-grained:      │
│ rotation          │ of all traffic   │ - Source            │
│                   │                  │ - Path              │
│ "I am Service A   │ "Our conversation│ - Method           │
│  and I can prove  │  is encrypted    │ - Headers          │
│  it"              │  end-to-end"     │ - JWT claims       │
└───────────────────┴──────────────────┴────────────────────┘

Components:
┌──────────┐         ┌──────────┐         ┌──────────┐
│  Istiod  │         │  Envoy   │         │  Envoy   │
│(CA, cert │────────▶│  Proxy   │◀──mTLS─▶│  Proxy   │
│ issuer)  │ issues  │(Service A)│        │(Service B)│
└──────────┘ certs   └──────────┘         └──────────┘
```

### Security Resources in Istio

| Resource | What It Does | Think of It As |
|----------|-------------|----------------|
| `PeerAuthentication` | Configures mTLS between services | "How do services prove who they are to each other?" |
| `RequestAuthentication` | Validates JWT tokens on incoming requests | "How do end-users prove who they are?" |
| `AuthorizationPolicy` | Controls access (allow/deny rules) | "Who is allowed to do what?" |

## 3. Identity in Istio — SPIFFE

### What is SPIFFE?

**SPIFFE** (Secure Production Identity Framework for Everyone) is a standard for assigning identities to services. Instead of using IP addresses (which change all the time in Kubernetes), Istio gives each service a **cryptographic identity**.

### How SPIFFE IDs Look

Every service in the mesh gets a SPIFFE ID in this format:

```
spiffe://<trust-domain>/ns/<namespace>/sa/<service-account>
```

**Examples:**

```
spiffe://cluster.local/ns/default/sa/frontend
spiffe://cluster.local/ns/payments/sa/payment-service
spiffe://cluster.local/ns/admin/sa/admin-api
```

Let me break this down:

```
spiffe://cluster.local/ns/default/sa/frontend
         ─────────────  ── ─────── ── ────────
              │           │    │     │    │
              │           │    │     │    └── Kubernetes Service Account name
              │           │    │     └── "sa" = service account
              │           │    └── Kubernetes namespace
              │           └── "ns" = namespace
              └── Trust domain (usually "cluster.local")
```

### Why Is This Important?

The identity is baked into an **X.509 certificate** that Istio automatically creates for every workload. This means:

- Service A can **prove** it's Service A (not someone pretending to be Service A)
- Service B can **verify** that the request actually came from Service A
- All of this happens automatically — no manual certificate management!

### The Certificate Lifecycle

```
1. Pod starts up
   │
   ▼
2. Envoy sidecar sends a Certificate Signing Request (CSR) to Istiod
   │
   ▼
3. Istiod validates the request (checks Kubernetes Service Account)
   │
   ▼
4. Istiod signs the certificate and sends it back to Envoy
   │
   ▼
5. Envoy uses this certificate for all mTLS connections
   │
   ▼
6. Certificate auto-rotates before expiry (default: every 24 hours)
   │
   ▼
7. Zero downtime — Envoy hot-swaps the new certificate
```

## 4. Mutual TLS (mTLS) — The Foundation

### Regular TLS vs Mutual TLS

You probably know **regular TLS** from HTTPS websites:

```
Regular TLS (one-way):
┌────────┐                    ┌────────┐
│ Client │───────────────────▶│ Server │
│        │  "Show me your     │        │
│        │   certificate"     │        │
│        │◀───────────────────│        │
│        │  "Here it is"      │        │
│        │                    │        │
│ Client │  ✅ Verifies server │        │
│        │  ❌ Server doesn't  │        │
│        │     verify client  │        │
└────────┘                    └────────┘

Mutual TLS (two-way):
┌────────┐                    ┌────────┐
│Service │───────────────────▶│Service │
│   A    │  "Show me YOUR     │   B    │
│        │   certificate"     │        │
│        │◀───────────────────│        │
│        │  "Here's mine"     │        │
│        │───────────────────▶│        │
│        │  "And here's MINE" │        │
│        │◀───────────────────│        │
│        │  "Verified! ✅"    │        │
│        │                    │        │
│  ✅ A verifies B            │        │
│  ✅ B verifies A            │        │
│  ✅ Traffic is encrypted     │        │
└────────┘                    └────────┘
```

**In plain English:**
- **Regular TLS**: Only the server proves who it is (like a website showing its SSL certificate)
- **Mutual TLS**: BOTH sides prove who they are (like two people showing each other their passports)

### What mTLS Gives You

| Benefit | Description |
|---------|-------------|
| **Encryption** | All traffic between services is encrypted (even inside the cluster) |
| **Authentication** | Both services verify each other's identity using certificates |
| **Integrity** | Traffic can't be modified in transit (tamper-proof) |
| **Identity** | Each service gets a verifiable SPIFFE identity |

### mTLS Modes in Istio

Istio supports different mTLS modes:

| Mode | Behavior | When to Use |
|------|----------|-------------|
| `STRICT` | Only mTLS traffic is accepted. Plain text is rejected. | Production (after migration is complete) |
| `PERMISSIVE` | Accepts both mTLS and plain text traffic | During migration (some services don't have sidecars yet) |
| `DISABLE` | No mTLS — plain text only | Testing, or for services that can't do mTLS |
| `UNSET` | Inherit from parent (namespace → mesh) | Default — uses the parent policy |

Think of it like a building's security:
- **STRICT** = "You MUST show your badge to enter. No exceptions."
- **PERMISSIVE** = "We prefer badges, but visitors can enter too."
- **DISABLE** = "Door is open, anyone can walk in."

## 5. PeerAuthentication — Configuring mTLS

`PeerAuthentication` controls how services authenticate each other (peer = another service in the mesh).

### Mesh-Wide mTLS (Enable for Everything)

```yaml
# Enable STRICT mTLS for the entire mesh
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: istio-system       # istio-system = applies to entire mesh
spec:
  mtls:
    mode: STRICT                # Only encrypted traffic allowed
```

**What this does:** Every service in every namespace MUST use mTLS. Any plain-text connection will be rejected.

### Namespace-Level mTLS

```yaml
# Enable STRICT mTLS for the "payments" namespace only
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: payments           # Only applies to the "payments" namespace
spec:
  mtls:
    mode: STRICT
```

### Workload-Level mTLS

```yaml
# Set PERMISSIVE mode for a specific workload
# (e.g., a service that also receives traffic from outside the mesh)
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: legacy-service-policy
  namespace: default
spec:
  selector:
    matchLabels:
      app: legacy-service       # Only applies to pods with this label
  mtls:
    mode: PERMISSIVE            # Accept both mTLS and plain text
```

### Port-Level mTLS

```yaml
# Different mTLS settings for different ports on the same service
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: multi-port-policy
  namespace: default
spec:
  selector:
    matchLabels:
      app: my-service
  mtls:
    mode: STRICT                # Default: STRICT for all ports
  portLevelMtls:
    8080:
      mode: PERMISSIVE          # Port 8080: also accepts plain text
    9090:
      mode: DISABLE             # Port 9090: plain text only (e.g., health checks)
```

### Migration Strategy: Gradually Enabling STRICT mTLS

```
Step 1: Start with PERMISSIVE (accepts both mTLS and plain text)
        → This won't break anything

Step 2: Check that all services have sidecars
        → kubectl get pods --all-namespaces -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.spec.containers[*].name}{"\n"}{end}' | grep istio-proxy

Step 3: Monitor traffic in Kiali
        → Look for any connections that are NOT using mTLS (shown as unlocked icons)

Step 4: Switch to STRICT mode
        → Now all services MUST use mTLS

Step 5: If something breaks, temporarily set that specific workload to PERMISSIVE
        → Fix the root cause, then go back to STRICT
```

### Verifying mTLS is Working

```bash
# Method 1: Check with istioctl
istioctl x describe pod <pod-name>
# Look for: "mTLS: STRICT" or "mTLS: PERMISSIVE"

# Method 2: Check the actual TLS connection
istioctl proxy-config secret <pod-name> -o json
# This shows the certificates being used

# Method 3: Check in Kiali
# Kiali shows a 🔒 lock icon on connections using mTLS

# Method 4: Try to send plain text to a STRICT service
kubectl exec <pod-without-sidecar> -- curl http://my-service:8080
# Should get: "connection reset" (because plain text is rejected)
```

### How to Check mTLS Status for All Services

```bash
# Check authentication policies
kubectl get peerauthentication --all-namespaces

# Check destination rules related to mTLS
kubectl get destinationrule --all-namespaces

# Use istioctl to analyze
istioctl analyze --all-namespaces
```

## 6. RequestAuthentication — JWT Validation

While `PeerAuthentication` handles service-to-service identity, `RequestAuthentication` handles **end-user identity** using JWT tokens.

### What This Does

```
Without RequestAuthentication:
┌──────┐    "I'm user John"    ┌──────────┐
│ User │──────────────────────▶│ Service  │
│      │   (anyone can claim   │          │
│      │    to be John!)       │          │
└──────┘                       └──────────┘

With RequestAuthentication:
┌──────┐   JWT: "I'm John,    ┌──────────┐     ┌──────────┐
│ User │── signed by Auth0" ─▶│  Envoy   │────▶│ Service  │
│      │                      │  Proxy   │     │          │
└──────┘                      │          │     │          │
                              │ ✅ Verify │     │ Gets     │
                              │ JWT sig   │     │ verified │
                              │ ✅ Check  │     │ claims   │
                              │ expiry    │     │          │
                              │ ✅ Extract│     │          │
                              │ claims    │     │          │
                              └──────────┘     └──────────┘
```

### Basic JWT Validation

```yaml
apiVersion: security.istio.io/v1beta1
kind: RequestAuthentication
metadata:
  name: jwt-auth
  namespace: default
spec:
  selector:
    matchLabels:
      app: frontend               # Apply to the frontend service
  jwtRules:
    - issuer: "https://accounts.google.com"        # Who issued the token
      jwksUri: "https://www.googleapis.com/oauth2/v3/certs"  # Where to get public keys
```

**What this does:**
- Envoy intercepts incoming requests
- If a JWT token is present, it validates:
  - ✅ Signature is valid (using the public key from `jwksUri`)
  - ✅ Token hasn't expired
  - ✅ Issuer matches
- If the token is **invalid** → request is rejected with `401 Unauthorized`
- If **no token** is present → request is allowed (use AuthorizationPolicy to require tokens)

### Multiple JWT Issuers

```yaml
apiVersion: security.istio.io/v1beta1
kind: RequestAuthentication
metadata:
  name: multi-jwt
  namespace: default
spec:
  selector:
    matchLabels:
      app: api-gateway
  jwtRules:
    # Accept tokens from Auth0
    - issuer: "https://my-app.auth0.com/"
      jwksUri: "https://my-app.auth0.com/.well-known/jwks.json"
      forwardOriginalToken: true          # Forward the token to the service

    # Also accept tokens from Google
    - issuer: "https://accounts.google.com"
      jwksUri: "https://www.googleapis.com/oauth2/v3/certs"

    # Also accept tokens from your own auth service
    - issuer: "https://auth.mycompany.com"
      jwksUri: "https://auth.mycompany.com/.well-known/jwks.json"
      audiences:                           # Only accept tokens for this audience
        - "api.mycompany.com"
```

### JWT from Different Locations

By default, Istio looks for JWT in the `Authorization: Bearer <token>` header. You can change this:

```yaml
jwtRules:
  - issuer: "https://my-app.auth0.com/"
    jwksUri: "https://my-app.auth0.com/.well-known/jwks.json"
    # Look for token in a custom header
    fromHeaders:
      - name: x-jwt-token
        prefix: "Bearer "
    # Or look for it in a query parameter
    fromParams:
      - "access_token"
```

## 7. AuthorizationPolicy — Who Can Talk to Whom

This is where the real power is. `AuthorizationPolicy` lets you define fine-grained access control rules — essentially a firewall for your microservices.

### The Three Actions

| Action | Behavior |
|--------|----------|
| `ALLOW` | Allow requests that match the rules (deny everything else) |
| `DENY` | Deny requests that match the rules |
| `CUSTOM` | Delegate to an external authorizer (e.g., OPA) |

### How Authorization Works (Order of Evaluation)

```
Incoming Request
      │
      ▼
┌─────────────┐     Match?     ┌──────────────┐
│ CUSTOM       │──── Yes ──────▶│ Delegate to  │
│ policies     │                │ ext. authz   │
│              │──── No ───┐   └──────────────┘
└─────────────┘            │
                           ▼
                   ┌─────────────┐     Match?     ┌──────────────┐
                   │ DENY         │──── Yes ──────▶│ DENIED (403) │
                   │ policies     │                └──────────────┘
                   │              │──── No ───┐
                   └─────────────┘            │
                                              ▼
                                      ┌─────────────┐     Match?     ┌──────────────┐
                                      │ ALLOW        │──── Yes ──────▶│ ALLOWED ✅   │
                                      │ policies     │                └──────────────┘
                                      │              │──── No ───┐
                                      └─────────────┘            │
                                                                 ▼
                                                         ┌──────────────┐
                                                         │ No ALLOW      │
                                                         │ policies?     │
                                                         │ → ALLOWED ✅  │
                                                         │               │
                                                         │ ALLOW policies│
                                                         │ exist but     │
                                                         │ didn't match? │
                                                         │ → DENIED ❌   │
                                                         └──────────────┘
```

**Key rule:** If ANY `ALLOW` policy exists for a workload but the request doesn't match any of them → **DENIED**.

### Example 1: Deny All Traffic (Start Secure)

```yaml
# Deny all traffic to all services in the namespace
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: deny-all
  namespace: default
spec:
  {}                              # Empty spec = match all workloads
  # No rules = deny all
```

**What this does:** No service in the `default` namespace can receive any traffic. This is the starting point for a "deny by default" approach.

### Example 2: Allow Specific Traffic

```yaml
# Allow frontend to call the backend API
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: allow-frontend-to-backend
  namespace: default
spec:
  selector:
    matchLabels:
      app: backend-api            # This policy applies to the backend-api service
  action: ALLOW
  rules:
    - from:
        - source:
            principals:
              - "cluster.local/ns/default/sa/frontend"   # Only frontend can call
      to:
        - operation:
            methods: ["GET", "POST"]       # Only GET and POST methods
            paths: ["/api/*"]              # Only /api/* paths
```

Let me break this down:

```yaml
spec:
  selector:
    matchLabels:
      app: backend-api            # WHO is being protected? → backend-api
  action: ALLOW                   # What action? → Allow matching requests
  rules:
    - from:                       # WHO is the caller?
        - source:
            principals:           # SPIFFE identity of the caller
              - "cluster.local/ns/default/sa/frontend"
      to:                         # WHAT are they trying to do?
        - operation:
            methods: ["GET", "POST"]   # HTTP methods allowed
            paths: ["/api/*"]          # URL paths allowed
```

### Example 3: Allow Based on Namespace

```yaml
# Allow any service in the "frontend" namespace to call services in "backend" namespace
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: allow-frontend-namespace
  namespace: backend
spec:
  action: ALLOW
  rules:
    - from:
        - source:
            namespaces: ["frontend"]       # Any service in the "frontend" namespace
```

### Example 4: Deny Specific Paths

```yaml
# Deny access to admin paths from non-admin services
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: deny-admin-paths
  namespace: default
spec:
  selector:
    matchLabels:
      app: backend-api
  action: DENY
  rules:
    - from:
        - source:
            notPrincipals:
              - "cluster.local/ns/admin/sa/admin-service"   # NOT the admin service
      to:
        - operation:
            paths: ["/admin/*", "/internal/*"]              # Trying to access admin paths
```

**In plain English:** "If you're NOT the admin-service and you try to access /admin/* or /internal/*, you're DENIED."

### Example 5: Authorization Based on JWT Claims

```yaml
# Only allow users with "admin" role to access admin endpoints
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: admin-only
  namespace: default
spec:
  selector:
    matchLabels:
      app: admin-panel
  action: ALLOW
  rules:
    - from:
        - source:
            requestPrincipals: ["https://auth.mycompany.com/*"]   # Must have a valid JWT
      to:
        - operation:
            paths: ["/admin/*"]
      when:
        - key: request.auth.claims[role]     # JWT must have "role" claim
          values: ["admin"]                  # And the value must be "admin"
```

### Example 6: Allow Health Checks

```yaml
# Always allow health check endpoints (even with deny-all in place)
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: allow-health-checks
  namespace: default
spec:
  selector:
    matchLabels:
      app: my-service
  action: ALLOW
  rules:
    - to:
        - operation:
            paths: ["/healthz", "/readyz", "/livez"]
            methods: ["GET"]
      # No "from" = allow from anyone
```

### All Available Source Conditions

| Field | Description | Example |
|-------|-------------|---------|
| `principals` | SPIFFE ID of the source | `["cluster.local/ns/default/sa/frontend"]` |
| `notPrincipals` | Exclude these SPIFFE IDs | `["cluster.local/ns/default/sa/attacker"]` |
| `requestPrincipals` | JWT subject (iss/sub) | `["https://auth0.com/user123"]` |
| `namespaces` | Source namespace | `["frontend", "api"]` |
| `notNamespaces` | Exclude these namespaces | `["untrusted"]` |
| `ipBlocks` | Source IP ranges | `["10.0.0.0/8"]` |
| `notIpBlocks` | Exclude these IPs | `["10.0.1.0/24"]` |

### All Available Operation (To) Conditions

| Field | Description | Example |
|-------|-------------|---------|
| `hosts` | Destination host | `["api.example.com"]` |
| `notHosts` | Exclude these hosts | `["internal.example.com"]` |
| `ports` | Destination port | `["8080", "443"]` |
| `notPorts` | Exclude these ports | `["9090"]` |
| `methods` | HTTP methods | `["GET", "POST"]` |
| `notMethods` | Exclude these methods | `["DELETE"]` |
| `paths` | URL paths (supports wildcards) | `["/api/*", "/public"]` |
| `notPaths` | Exclude these paths | `["/internal/*"]` |

### All Available "When" Conditions

| Key | Description | Example |
|-----|-------------|---------|
| `request.headers[name]` | HTTP header value | `request.headers[x-api-key]` |
| `request.auth.claims[name]` | JWT claim value | `request.auth.claims[role]` |
| `request.auth.presenter` | JWT authorized party | — |
| `request.auth.audiences` | JWT audience | — |
| `source.ip` | Source IP address | — |
| `source.namespace` | Source namespace | — |
| `destination.port` | Destination port | — |
| `connection.sni` | TLS SNI | — |

## 8. Real-World Authorization Scenarios

### Scenario 1: E-Commerce Application

```yaml
# Architecture:
# web-frontend → api-gateway → [product-service, order-service, payment-service]
# order-service → payment-service
# admin-dashboard → api-gateway

---
# 1. Deny all by default
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: deny-all
  namespace: ecommerce
spec:
  {}

---
# 2. Allow web-frontend → api-gateway
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: allow-web-to-gateway
  namespace: ecommerce
spec:
  selector:
    matchLabels:
      app: api-gateway
  action: ALLOW
  rules:
    - from:
        - source:
            principals: ["cluster.local/ns/ecommerce/sa/web-frontend"]
      to:
        - operation:
            methods: ["GET", "POST", "PUT"]
            paths: ["/api/products/*", "/api/orders/*", "/api/cart/*"]

---
# 3. Allow api-gateway → product-service (read only)
```

## 9. Certificate Management in Istio

### How Certificates Work in Istio

```
┌────────────────────────────────────────────────────────┐
│                  Certificate Flow                       │
│                                                        │
│  ┌──────────┐  1. CSR (Certificate    ┌──────────────┐│
│  │  Envoy   │─── Signing Request) ──▶│   Istiod     ││
│  │  Sidecar │                        │   (acts as   ││
│  │          │  2. Signed Certificate  │   Certificate││
│  │          │◀── (X.509) ────────────│   Authority) ││
│  └──────────┘                        └──────────────┘│
│                                                        │
│  Certificate contains:                                 │
│  - SPIFFE ID (identity)                               │
│  - Public key                                         │
│  - Validity period (default: 24 hours)                │
│  - Issuer (Istiod CA)                                 │
└────────────────────────────────────────────────────────┘
```

### Default Certificate Settings

| Setting | Default Value | Description |
|---------|---------------|-------------|
| Certificate lifetime | 24 hours | How long a cert is valid |
| Root CA certificate | Self-signed (auto-generated) | Trust anchor |
| Certificate rotation | Automatic | Envoy renews before expiry |
| Key type | RSA 2048 / EC P256 | Cryptographic algorithm |

### Using Custom Root CA (Production)

For production, you should use your own root CA instead of Istio's self-signed one:

```bash
# Create a secret with your custom CA certificates
kubectl create secret generic cacerts -n istio-system \
  --from-file=ca-cert.pem \
  --from-file=ca-key.pem \
  --from-file=root-cert.pem \
  --from-file=cert-chain.pem
```

### Integrating with cert-manager

```yaml
# Use cert-manager as the CA for Istio
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  values:
    pilot:
      env:
        EXTERNAL_CA: ISTIOD_RA_KUBERNETES_API
  meshConfig:
    caCertificates:
      - certSigners:
          - clusterissuers.cert-manager.io/istio-ca
```

## 10. Debugging Security Issues

### Problem: "Connection Refused" or "Connection Reset"

```bash
# Step 1: Check PeerAuthentication policy
kubectl get peerauthentication --all-namespaces

# Step 2: Check if sidecar is injected in both pods
kubectl get pod <source-pod> -o jsonpath='{.spec.containers[*].name}'
kubectl get pod <dest-pod> -o jsonpath='{.spec.containers[*].name}'
# Both should show "istio-proxy"

# Step 3: Check mTLS status
istioctl x describe pod <dest-pod>

# Step 4: Check Envoy proxy logs
kubectl logs <dest-pod> -c istio-proxy | grep "TLS error"
```

### Problem: "403 Forbidden" (RBAC Access Denied)

```bash
# Step 1: Check what AuthorizationPolicies apply
kubectl get authorizationpolicy -n <namespace>

# Step 2: Look at the Envoy RBAC debug logs
kubectl logs <dest-pod> -c istio-proxy | grep "rbac"

# Step 3: Enable debug logging for RBAC
istioctl proxy-config log <dest-pod> --level rbac:debug

# Step 4: Retry the request and check logs again
kubectl logs <dest-pod> -c istio-proxy | grep "enforced"

# Step 5: Use istioctl analyze to find misconfigurations
istioctl analyze -n <namespace>
```

### Problem: "401 Unauthorized" (JWT Issues)

```bash
# Step 1: Check RequestAuthentication policy
kubectl get requestauthentication -n <namespace> -o yaml

# Step 2: Verify your JWT token
# Paste your token at https://jwt.io to decode it
# Check: issuer matches? Token expired? Audience correct?

# Step 3: Check if Envoy can reach the JWKS URI
kubectl exec <pod> -c istio-proxy -- curl -sS <jwks-uri>

# Step 4: Check Envoy logs for JWT errors
kubectl logs <pod> -c istio-proxy | grep "jwt"
```

### The Ultimate Debug Command

```bash
# This gives you a comprehensive overview of security for a pod
istioctl x describe pod <pod-name>

# Output includes:
# - mTLS mode (STRICT/PERMISSIVE)
# - PeerAuthentication policies applied
# - AuthorizationPolicies applied
# - Destination rules
# - Certificate status
```

## 11. Security Best Practices

### 1. Enable STRICT mTLS Mesh-Wide

```yaml
# Always do this in production
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: istio-system
spec:
  mtls:
    mode: STRICT
```

### 2. Use "Deny by Default" Authorization

```yaml
# Start with deny-all, then explicitly allow what's needed
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: deny-all
  namespace: my-namespace
spec:
  {}
```

### 3. Use Least Privilege
Only allow the **minimum required** access:
- Restrict HTTP methods (don't allow DELETE if only GET is needed)
- Restrict paths (don't allow `/admin/*` for regular services)
- Restrict source services (only the services that NEED to call)

### 4. Monitor and Alert on Security Events

```promql
# Alert on 403s (authorization failures) — might indicate an attack
sum(rate(istio_requests_total{response_code="403"}[5m])) by (source_workload, destination_workload)

# Alert on 401s (authentication failures)
sum(rate(istio_requests_total{response_code="401"}[5m])) by (destination_workload)
```

### 5. Use Custom Root CA in Production
Don't rely on Istio's self-signed CA for production. Use your organization's PKI or a tool like cert-manager.

### 6. Regularly Rotate Certificates
Istio handles this automatically, but verify it's working:
```bash
istioctl proxy-config secret <pod-name> -o json | jq '.dynamicActiveSecrets[0].secret.tlsCertificate.certificateChain.inlineBytes' | base64 -d | openssl x509 -noout -dates
```

### 7. Validate Policies with `istioctl analyze`
```bash
# Always run this after applying new policies
istioctl analyze --all-namespaces
```

## 12. Common Mistakes to Avoid

| Mistake | Impact | Fix |
|---------|--------|-----|
| Jumping straight to STRICT mTLS | Services without sidecars break | Start with PERMISSIVE, migrate, then switch to STRICT |
| Forgetting deny-all when creating ALLOW policies | Without deny-all, services with no ALLOW policy remain open | Always start with a deny-all policy |
| Putting AuthorizationPolicy in wrong namespace | Policy doesn't apply | Policy goes in the TARGET service's namespace |
| Not handling health check endpoints | Kubernetes health checks fail, pods restart | Always allow `/healthz`, `/readyz`, `/livez` |
| Using IP-based rules | IPs change constantly in Kubernetes | Use SPIFFE principals or namespace-based rules |
| Not testing policies in staging first | Production outage | Always test in a non-production environment first |
| Forgetting about egress traffic | Services can reach external endpoints freely | Apply authorization policies for egress too |
| Not monitoring 403/401 responses | Missing security events | Set up Prometheus alerts for auth failures |
