## 1. What is Linkerd?

Linkerd (pronounced "linker-dee") is a **lightweight, ultra-fast service mesh** for Kubernetes. It was actually the **first** service mesh ever created (before Istio!), built by the folks at Buoyant.

### The Philosophy: Simplicity First

While Istio tries to be a Swiss Army knife with tons of features, Linkerd takes a different approach:

> "Do the most important things perfectly, and skip the rest."

Think of it like this:
- **Istio** = A fully loaded luxury car with 500 buttons and features (powerful, but complex)
- **Linkerd** = A Tesla with a clean interface (does what you need, simple to use)

### What Linkerd Does

| Feature | Description |
|---------|-------------|
| **Automatic mTLS** | Encrypts all traffic between services — enabled by default! |
| **Observability** | Golden metrics (success rate, latency, request volume) for every service |
| **Reliability** | Retries, timeouts, load balancing (EWMA algorithm) |
| **Authorization** | Policy to control who can talk to whom |
| **Multi-cluster** | Connect services across multiple Kubernetes clusters |

### What Linkerd Intentionally Doesn't Do

| Feature | Linkerd's Approach |
|---------|-------------------|
| Ingress/Egress gateway | Use existing solutions (Nginx, Traefik, etc.) |
| Advanced traffic shaping | Basic canary via TrafficSplit; no complex routing rules |
| Rate limiting | Not built-in (use your ingress controller) |
| JWT validation | Not built-in (handle at the application or gateway level) |
| Custom Envoy filters | Uses its own proxy (linkerd2-proxy), not Envoy |

## 2. Linkerd vs Istio — When to Use What

### Quick Comparison

| Aspect | Linkerd | Istio |
|--------|---------|-------|
| **Complexity** | Simple — "it just works" | Complex — many knobs to turn |
| **Proxy** | linkerd2-proxy (Rust) | Envoy (C++) |
| **Resource usage** | ~10MB RAM per sidecar | ~50-100MB RAM per sidecar |
| **Latency overhead** | Sub-millisecond (<1ms) | ~3-5ms |
| **mTLS** | On by default! | Need to configure |
| **Learning curve** | Hours to learn | Days/weeks to learn |
| **Traffic management** | Basic (TrafficSplit) | Advanced (VirtualService, DestinationRule) |
| **Community** | CNCF Graduated project | CNCF Graduated project |
| **Configuration** | Minimal YAML | Lots of CRDs and YAML |
| **JWT/AuthN** | Not built-in | Built-in |
| **Egress control** | Not built-in | Built-in |

### When to Choose Linkerd

Choose Linkerd when:
- You want mTLS and observability with **minimal effort**
- You're running a **small to medium** number of services
- You want the **lowest possible resource overhead**
- Your team doesn't have deep service mesh experience
- You value **simplicity over features**
- You want something that works in **5 minutes**, not 5 days

### When to Choose Istio

Choose Istio when:
- You need **advanced traffic management** (complex routing, traffic mirroring, fault injection)
- You need **JWT validation** at the mesh level
- You need **egress control** (control what external services your apps can reach)
- You have a **large, complex** microservices architecture
- You need **custom Envoy filters** or WASM plugins
- Your team has the expertise to manage Istio's complexity

### The Honest Truth

For **80% of teams**, Linkerd gives you everything you actually need with a fraction of the complexity. Many teams start with Istio because it's more famous, then switch to Linkerd because Istio was overkill for their needs.

## 3. Linkerd Architecture

### High-Level Architecture

```
┌──────────────────────────────────────────────────────┐
│                 LINKERD ARCHITECTURE                  │
├──────────────────────────────────────────────────────┤
│                                                      │
│  CONTROL PLANE (in linkerd namespace)                │
│  ┌────────────────────────────────────────────────┐  │
│  │                                                │  │
│  │  ┌──────────────┐  ┌───────────────────────┐  │  │
│  │  │ Destination  │  │ Identity              │  │  │
│  │  │ (service     │  │ (certificate          │  │  │
│  │  │  discovery,  │  │  authority for mTLS)  │  │  │
│  │  │  profiles,   │  │                       │  │  │
│  │  │  policies)   │  │                       │  │  │
│  │  └──────────────┘  └───────────────────────┘  │  │
│  │                                                │  │
│  │  ┌──────────────┐                             │  │
│  │  │ Proxy        │                             │  │
│  │  │ Injector     │                             │  │
│  │  │ (adds sidecar│                             │  │
│  │  │  to pods)    │                             │  │
│  │  └──────────────┘                             │  │
│  └────────────────────────────────────────────────┘  │
│                                                      │
│  DATA PLANE (in every meshed pod)                    │
│  ┌────────────────────────────────────────────────┐  │
│  │  ┌──────────────┐    ┌──────────────────────┐ │  │
│  │  │ Your App     │◀──▶│ linkerd2-proxy       │ │  │
│  │  │ Container    │    │ (Rust-based sidecar)  │ │  │
│  │  └──────────────┘    └──────────────────────┘ │  │
│  └────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────┘
```

### Control Plane Components

| Component | What It Does | Analogy |
|-----------|-------------|---------|
| **Destination** | Service discovery — tells proxies where to send traffic. Also manages service profiles, policies, and traffic splits. | Like a GPS navigation system |
| **Identity** | Issues TLS certificates to proxies for mTLS. Acts as the Certificate Authority (CA). | Like a passport office |
| **Proxy Injector** | Mutating webhook — automatically injects the sidecar proxy into pods. | Like a security checkpoint that gives everyone a bodyguard |

### The linkerd2-proxy (The Secret Weapon)

This is what makes Linkerd special. Instead of using Envoy (like Istio does), Linkerd uses its **own proxy written in Rust**:

| Feature | linkerd2-proxy | Envoy |
|---------|---------------|-------|
| Language | Rust | C++ |
| Memory usage | ~10MB | ~50-100MB |
| Latency overhead | <1ms (P99) | ~3-5ms |
| Configuration | Automatic (no manual config) | Complex YAML/xDS |
| Purpose-built | Yes, specifically for Linkerd | General-purpose proxy |
| CVE history | Very few | Many (larger attack surface) |

Why Rust? Because Rust gives you C++-level performance with memory safety — no buffer overflows, no null pointer crashes. This means fewer security vulnerabilities.

## 5. Adding Services to the Mesh (Meshing)

"Meshing" a service means injecting the Linkerd sidecar proxy into your pods.

### Method 1: Annotate the Namespace (Recommended)

```bash
# Add annotation to automatically inject sidecar into ALL pods in a namespace
kubectl annotate namespace default linkerd.io/inject=enabled

# Any NEW pods in this namespace will automatically get the sidecar
# Existing pods need to be restarted
kubectl rollout restart deployment -n default
```

### Method 2: Annotate a Specific Deployment

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  template:
    metadata:
      annotations:
        linkerd.io/inject: enabled       # Add sidecar to this deployment only
    spec:
      containers:
        - name: my-app
          image: my-app:v1
```

### Method 3: Use the CLI to Inject

```bash
# Inject into an existing deployment YAML
kubectl get deployment my-app -o yaml | linkerd inject - | kubectl apply -f -

# Or inject from a file
linkerd inject my-deployment.yaml | kubectl apply -f -
```

### Verify Injection

```bash
# Check that the proxy sidecar was added
kubectl get pods -n default

# Before injection:
NAME                      READY   STATUS    RESTARTS   AGE
my-app-7d8f5b6c4d-k9n2p   1/1     Running   0          5m

# After injection:
NAME                      READY   STATUS    RESTARTS   AGE
my-app-7d8f5b6c4d-x2m4r   2/2     Running   0          30s
                          ^^^
                          2 containers now! (app + linkerd-proxy)
```

### What Gets Injected?

```yaml
# Linkerd adds these containers/volumes to your pod:
spec:
  containers:
    - name: my-app                 # Your original container (unchanged)
      image: my-app:v1

    - name: linkerd-proxy          # The injected sidecar
      image: cr.l5d.io/linkerd/proxy:stable-2.14.0
      ports:
        - containerPort: 4143      # Inbound proxy port
        - containerPort: 4140      # Outbound proxy port
        - containerPort: 4191      # Metrics port
      resources:
        requests:
          cpu: 100m                # Very lightweight!
          memory: 20Mi             # Only ~20MB of memory!

  initContainers:
    - name: linkerd-init           # Sets up iptables to redirect traffic
      image: cr.l5d.io/linkerd/proxy-init:v2.2.0
```

Notice the resource usage: **100m CPU and 20Mi memory**. Compare that to Istio's Envoy sidecar which typically uses 100m CPU and 128Mi memory.

### Opt Out Specific Pods

```yaml
# If you DON'T want the sidecar on a specific pod
metadata:
  annotations:
    linkerd.io/inject: disabled
```

## 6. Automatic mTLS — Zero Config Encryption

This is one of Linkerd's best features. **mTLS is enabled by default with zero configuration.**

### How It Works

```
When two meshed pods communicate:

┌──────────────────┐                    ┌──────────────────┐
│ Pod A            │                    │ Pod B            │
│                  │                    │                  │
│ App  → linkerd   │───── mTLS ────────│ linkerd → App    │
│        proxy     │  (encrypted!)     │ proxy            │
│                  │                    │                  │
│ App sends plain  │                    │ App receives     │
│ HTTP to proxy    │                    │ plain HTTP from  │
│                  │                    │ proxy            │
└──────────────────┘                    └──────────────────┘

Your app never needs to know about TLS!
It sends plain HTTP → Linkerd encrypts it → Linkerd decrypts it → App receives plain HTTP
```

### Verify mTLS is Working

```bash
# Check mTLS status for all meshed pods
linkerd edges deployment -n default

# Output:
SRC          DST          SRC_NS    DST_NS    SECURED
frontend     backend      default   default   √
backend      database     default   default   √
frontend     auth-service default   default   √
```

The `√` under `SECURED` means mTLS is active.

```bash
# More detailed check
linkerd identity -n default

# Shows certificate details for each pod
```

### What About Non-Meshed Services?

If one side has the Linkerd proxy and the other doesn't:

```
Meshed Pod → Non-meshed Pod:
  linkerd-proxy sends plain text (falls back gracefully)
  ⚠️ Traffic is NOT encrypted

Non-meshed Pod → Meshed Pod:
  linkerd-proxy accepts plain text
  ⚠️ Traffic is NOT encrypted

Meshed Pod → Meshed Pod:
  linkerd-proxy ↔ linkerd-proxy = mTLS
  ✅ Traffic IS encrypted
```

**Key takeaway:** Both sides need the sidecar for mTLS to work. Linkerd handles this gracefully — no crashes, just falls back to plain text.

## 7. Traffic Management

### Load Balancing (EWMA — Better Than Round Robin)

Linkerd uses **EWMA (Exponentially Weighted Moving Average)** load balancing instead of simple round robin.

```
Round Robin (what most tools use):
Request 1 → Pod A (fast, 10ms response)
Request 2 → Pod B (slow, 500ms response — it's overloaded!)
Request 3 → Pod A (fast, 10ms response)
Request 4 → Pod B (slow, 500ms response — still overloaded!)
  ↑ Round robin doesn't care that Pod B is slow. It keeps sending traffic.

EWMA (what Linkerd uses):
Request 1 → Pod A (10ms response) — nice!
Request 2 → Pod B (500ms response) — hmm, B is slow
Request 3 → Pod A (10ms response) — A is fast, send more here
Request 4 → Pod A (12ms response) — keep sending to A
Request 5 → Pod B (50ms response) — B recovered, try again
  ↑ EWMA tracks response times and sends more traffic to faster pods
```

**You don't need to configure anything.** EWMA is the default, and it just works.

### Traffic Splitting (Canary Deployments)

Linkerd uses the **SMI TrafficSplit** resource for canary deployments:

```yaml
# Split traffic: 90% to v1, 10% to v2 (canary)
apiVersion: split.smi-spec.io/v1alpha2
kind: TrafficSplit
metadata:
  name: backend-split
  namespace: default
spec:
  service: backend                 # The original service
  backends:
    - service: backend-v1          # Stable version
      weight: 900                  # 90% of traffic
    - service: backend-v2          # Canary version
      weight: 100                  # 10% of traffic
```

**Prerequisites:** You need separate services for each version:

```yaml
# Service for v1
apiVersion: v1
kind: Service
metadata:
  name: backend-v1
spec:
  selector:
    app: backend
    version: v1
  ports:
    - port: 8080

---
# Service for v2
apiVersion: v1
kind: Service
metadata:
  name: backend-v2
spec:
  selector:
    app: backend
    version: v2
  ports:
    - port: 8080

---
# The "main" service (clients use this name)
apiVersion: v1
kind: Service
metadata:
  name: backend
spec:
  selector:
    app: backend
  ports:
    - port: 8080
```

### Gradual Canary Rollout

```bash
# Start with 5% canary
# backend-split.yaml: weight v1=950, v2=50
kubectl apply -f backend-split.yaml

# Check metrics — is v2 healthy?
linkerd stat trafficsplit -n default

# Output:
NAME            APEX      LEAF         WEIGHT   SUCCESS   RPS   LATENCY_P50   LATENCY_P99
backend-split   backend   backend-v1   950      100.00%   45    5ms           20ms
backend-split   backend   backend-v2   50       99.50%    2.5   6ms           25ms

# Looks good! Increase to 20%
# Update: weight v1=800, v2=200
kubectl apply -f backend-split.yaml

# Continue increasing until 100% on v2
```

### Retries and Timeouts (via Service Profiles)

```yaml
# ServiceProfile for the backend service
apiVersion: linkerd.io/v1alpha2
kind: ServiceProfile
metadata:
  name: backend.default.svc.cluster.local
  namespace: default
spec:
  routes:
    - name: GET /api/products
      condition:
        method: GET
        pathRegex: /api/products
      timeout: 5s                    # Timeout after 5 seconds
      isRetryable: true              # Retry on failure

    - name: POST /api/orders
      condition:
        method: POST
        pathRegex: /api/orders
      timeout: 10s                   # Longer timeout for order creation
      isRetryable: false             # DON'T retry POSTs (not idempotent!)
```

**Important:** Only set `isRetryable: true` for **idempotent** operations (GET, PUT, DELETE). Never retry a POST that creates an order — you might create duplicate orders!

## 8. Observability in Linkerd

### Golden Metrics — Automatic for Every Service

Linkerd automatically tracks three key metrics for every meshed service:

| Metric | What It Tells You |
|--------|-------------------|
| **Success Rate** | What percentage of requests are succeeding? (non-5xx) |
| **Request Volume** | How many requests per second is this service handling? |
| **Latency** | How fast is this service responding? (P50, P95, P99) |

**You get these with ZERO configuration.** The moment you mesh a service, metrics start flowing.

### Using the CLI for Observability

#### Overall Stats

```bash
# See stats for all deployments in a namespace
linkerd stat deployment -n default

# Output:
NAME         MESHED   SUCCESS   RPS    LATENCY_P50   LATENCY_P95   LATENCY_P99
frontend     1/1      100.00%   10.5   5ms           25ms          50ms
backend      2/2      99.85%    45.2   3ms           15ms          80ms
auth-svc     1/1      100.00%   30.1   2ms           8ms           20ms
database     1/1      98.50%    50.0   10ms          50ms          200ms
```

At a glance, you can see:
- `database` has a 98.50% success rate — something's wrong there!
- `database` also has the highest latency (P99: 200ms)

#### Stats by Resource Type

```bash
# Stats for all pods
linkerd stat pod -n default

# Stats for all services
linkerd stat service -n default

# Stats for all namespaces
linkerd stat namespace

# Stats for specific deployment
linkerd stat deployment/backend -n default
```

#### Traffic Between Services

```bash
# See who's calling whom
linkerd stat deployment -n default --from deployment/frontend

# Output:
NAME       MESHED   SUCCESS   RPS    LATENCY_P50   LATENCY_P95   LATENCY_P99
backend    2/2      99.85%    35.0   3ms           15ms          80ms
auth-svc   1/1      100.00%   10.5   2ms           8ms           20ms

# This shows: frontend is calling backend (35 rps) and auth-svc (10.5 rps)
```

```bash
# See who's calling the backend
linkerd stat deployment -n default --to deployment/backend

# Output:
NAME       MESHED   SUCCESS   RPS    LATENCY_P50   LATENCY_P95   LATENCY_P99
frontend   1/1      100.00%   35.0   3ms           15ms          80ms
worker     1/1      100.00%   10.2   5ms           20ms          60ms

# This shows: frontend and worker are both calling backend
```

#### Live Traffic Monitoring

```bash
# Watch live requests to a deployment (like tcpdump but for HTTP)
linkerd tap deployment/backend -n default

# Output (real-time stream):
req id=0:0 proxy=in  src=10.244.0.5:48234 dst=10.244.1.3:8080 tls=true :method=GET :path=/api/products
rsp id=0:0 proxy=in  src=10.244.0.5:48234 dst=10.244.1.3:8080 tls=true :status=200 latency=3ms
req id=0:1 proxy=in  src=10.244.0.5:48234 dst=10.244.1.3:8080 tls=true :method=GET :path=/api/products/42
rsp id=0:1 proxy=in  src=10.244.0.5:48234 dst=10.244.1.3:8080 tls=true :status=200 latency=5ms
```

This shows **every request** in real-time: source, destination, method, path, TLS status, status code, and latency.

```bash
# Filter tap to show only errors
linkerd tap deployment/backend -n default --path=/api/orders --method=POST | grep ":status=5"

# Filter to show only slow requests (use linkerd top for aggregated view)
linkerd top deployment/backend -n default

# Output:
Source                Destination           Method   Path             Count   Best   Worst   Last   Success
frontend-xyz          backend-abc           GET      /api/products    150     1ms    50ms    3ms    100.00%
frontend-xyz          backend-abc           POST     /api/orders      42      10ms   200ms   45ms   97.62%

```

### Prometheus Integration

Linkerd exposes metrics in Prometheus format:

```bash
# If you have the viz extension installed, Prometheus comes bundled
# Linkerd metrics are available at the proxy's metrics port (:4191)

# Key Linkerd metrics:
# request_total — total requests (counter)
# response_total — total responses (counter)  
# response_latency_ms — response latency (histogram)
```

Example Prometheus queries:

```promql
# Success rate for the backend deployment
sum(rate(response_total{deployment="backend", classification="success"}[5m])) 
/ 
sum(rate(response_total{deployment="backend"}[5m]))

# P99 latency for the backend
histogram_quantile(0.99, 
  sum(rate(response_latency_ms_bucket{deployment="backend"}[5m])) by (le)
)

# Request rate by source → destination
sum(rate(request_total[5m])) by (deployment, dst_deployment)
```

## 10. Service Profiles — Per-Route Metrics & Retries

### What Are Service Profiles?

Service Profiles are a Linkerd CRD that lets you configure **per-route behavior** — metrics, retries, and timeouts for specific API endpoints.

### Creating Service Profiles

#### Method 1: Auto-generate from OpenAPI/Swagger

```bash
# Generate a Service Profile from a Swagger file
linkerd profile --open-api swagger.yaml backend.default.svc.cluster.local \
  | kubectl apply -f -
```

#### Method 2: Auto-generate from live traffic (tap)

```bash
# Watch live traffic and generate a profile
linkerd profile --tap deployment/backend -n default --tap-duration 30s \
  backend.default.svc.cluster.local | kubectl apply -f -
```

#### Method 3: Write manually

```yaml
apiVersion: linkerd.io/v1alpha2
kind: ServiceProfile
metadata:
  name: backend.default.svc.cluster.local     # Must match the FQDN
  namespace: default
spec:
  routes:
    # GET /api/products — list all products
    - name: GET /api/products
      condition:
        method: GET
        pathRegex: /api/products
      isRetryable: true                        # Safe to retry (read-only)
      timeout: 5s

    # GET /api/products/{id} — get a specific product
    - name: GET /api/products/{id}
      condition:
        method: GET
        pathRegex: /api/products/[0-9]+
      isRetryable: true
      timeout: 3s

    # POST /api/orders — create a new order
    - name: POST /api/orders
      condition:
        method: POST
        pathRegex: /api/orders
      isRetryable: false                       # NOT safe to retry (creates data!)
      timeout: 10s

    # DELETE /api/orders/{id} — cancel an order
    - name: DELETE /api/orders/{id}
      condition:
        method: DELETE
        pathRegex: /api/orders/[0-9]+
      isRetryable: true                        # Idempotent, safe to retry
      timeout: 5s
```

### How Retries Work

```
Without retries:
Client → Backend (fails with 503) → Client gets error ❌

With retries (isRetryable: true):
Client → Backend (fails with 503) 
       → Backend (retry #1 — fails with 503)
       → Backend (retry #2 — succeeds! 200) → Client gets success ✅

Retry budget: Linkerd adds at most 20% extra requests
  If your service gets 100 rps, Linkerd will retry up to 20 rps max
  This prevents "retry storms" that can overwhelm a struggling service
```

### Viewing Per-Route Metrics

```bash
# After applying a ServiceProfile, you get per-route metrics
linkerd routes deployment/backend -n default

# Output:
ROUTE                         SERVICE   SUCCESS   RPS    LATENCY_P50   LATENCY_P99
GET /api/products             backend   100.00%   20.0   3ms           15ms
GET /api/products/{id}        backend   100.00%   15.2   5ms           20ms
POST /api/orders              backend   99.50%    10.0   15ms          80ms
DELETE /api/orders/{id}       backend   100.00%   2.0    8ms           25ms
[DEFAULT]                     backend   100.00%   3.0    4ms           12ms
```

`[DEFAULT]` catches any routes not matched by the profile.

## 11. Multi-Cluster with Linkerd

### What is Multi-Cluster?

Linkerd can connect services across **multiple Kubernetes clusters**, with mTLS still active across clusters. This is useful for:

- Disaster recovery (services in two regions)
- Hybrid cloud (on-prem + cloud)
- Separation of concerns (production workloads in different clusters)

### How It Works

```
┌──────────────────────┐          ┌──────────────────────┐
│   Cluster: us-east   │          │   Cluster: eu-west   │
│                      │          │                      │
│  ┌──────────┐        │   mTLS   │        ┌──────────┐ │
│  │ frontend │───────────────────────────▶│ backend  │ │
│  │          │        │   over   │        │          │ │
│  └──────────┘        │ internet │        └──────────┘ │
│                      │          │                      │
│  ┌──────────────┐    │          │    ┌──────────────┐ │
│  │ Linkerd      │◀──────────────────▶│ Linkerd      │ │
│  │ Gateway      │    │ service  │    │ Gateway      │ │
│  └──────────────┘    │ mirroring│    └──────────────┘ │
└──────────────────────┘          └──────────────────────┘
```

### Setting Up Multi-Cluster

```bash
# Install the multicluster extension on both clusters
linkerd multicluster install | kubectl apply -f -

# Link the clusters (run from cluster 1, targeting cluster 2)
linkerd multicluster link --cluster-name eu-west \
  --api-server-address https://eu-west.k8s.example.com:6443 \
  | kubectl apply -f -

# Verify the link
linkerd multicluster check

# Export a service from cluster 2 to be available in cluster 1
kubectl --context eu-west label svc backend \
  mirror.linkerd.io/exported=true

# Now in cluster 1, you can access:
# backend.default.svc.cluster.local        → local
# backend-eu-west.default.svc.cluster.local → remote (mirrored)
```

## 12. Authorization Policy in Linkerd

### Server and ServerAuthorization (Linkerd's RBAC)

Starting from Linkerd 2.12+, you can control access using `Server` and `AuthorizationPolicy` resources.

### Step 1: Define a Server (What You're Protecting)

```yaml
# Define the server (port + pod selection)
apiVersion: policy.linkerd.io/v1beta2
kind: Server
metadata:
  name: backend-http
  namespace: default
spec:
  podSelector:
    matchLabels:
      app: backend
  port: 8080
  proxyProtocol: HTTP/1      # HTTP/1.1, HTTP/2, gRPC, opaque, TLS
```

### Step 2: Create an Authorization Policy

```yaml
# Allow only frontend to access the backend
apiVersion: policy.linkerd.io/v1alpha1
kind: AuthorizationPolicy
metadata:
  name: allow-frontend
  namespace: default
spec:
  targetRef:
    group: policy.linkerd.io
    kind: Server
    name: backend-http            # Protect the backend server
  requiredAuthentication:
    - meshTLS:
        identities:
          - "*.default.serviceaccount.identity.linkerd.cluster.local"
          # Or be specific:
          # - "frontend.default.serviceaccount.identity.linkerd.cluster.local"
```

### Default Policy Modes

You can set a default policy for what happens when no authorization policy matches:

```bash
# Install Linkerd with a default deny policy
linkerd install --set policyController.defaultPolicy=deny | kubectl apply -f -
```

Available default policies:

| Policy | Behavior |
|--------|----------|
| `all-unauthenticated` | Allow all traffic (default — most permissive) |
| `all-authenticated` | Allow only meshed (mTLS) traffic |
| `cluster-unauthenticated` | Allow traffic from within the cluster |
| `cluster-authenticated` | Allow only meshed traffic from within the cluster |
| `deny` | Deny all traffic unless explicitly allowed (most secure) |

### Example: Lock Down a Production Namespace

```yaml
# 1. Set default deny for the namespace
apiVersion: v1
kind: Namespace
metadata:
  name: production
  annotations:
    config.linkerd.io/default-inbound-policy: deny

---
# 2. Define the server
apiVersion: policy.linkerd.io/v1beta2
kind: Server
metadata:
  name: api-server
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: api
  port: 8080
  proxyProtocol: HTTP/1

---
# 3. Allow only the gateway to call the API
apiVersion: policy.linkerd.io/v1alpha1
kind: AuthorizationPolicy
metadata:
  name: allow-gateway-to-api
  namespace: production
spec:
  targetRef:
    group: policy.linkerd.io
    kind: Server
    name: api-server
  requiredAuthentication:
    - meshTLS:
        identities:
          - "gateway.production.serviceaccount.identity.linkerd.cluster.local"

---
# 4. Always allow health checks (from kubelet — not meshed)
apiVersion: policy.linkerd.io/v1beta2
kind: Server
metadata:
  name: health-check
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: api
  port: 8080
  proxyProtocol: HTTP/1

---
apiVersion: policy.linkerd.io/v1alpha1
kind: AuthorizationPolicy
metadata:
  name: allow-health-probes
  namespace: production
spec:
  targetRef:
    group: policy.linkerd.io
    kind: Server
    name: health-check
  requiredAuthentication:
    - network:
        cidr: 0.0.0.0/0          # Allow from any IP (kubelet)
  # Note: This allows probes. Combine with HTTPRoute to limit to just /healthz
```

## 14. Production Deployment Best Practices

### 1. Use High Availability Mode

```bash
# Install Linkerd in HA mode (3 replicas of each control plane component)
linkerd install --ha | kubectl apply -f -

# This ensures the control plane survives node failures
```

### 2. Use a Trust Anchor from Your PKI

```bash
# Generate your own trust anchor (do this once, store securely!)
step certificate create root.linkerd.cluster.local ca.crt ca.key \
  --profile root-ca --no-password --insecure

step certificate create identity.linkerd.cluster.local issuer.crt issuer.key \
  --profile intermediate-ca --not-after 8760h --no-password --insecure \
  --ca ca.crt --ca-key ca.key

# Install with your own certificates
linkerd install \
  --identity-trust-anchors-file ca.crt \
  --identity-issuer-certificate-file issuer.crt \
  --identity-issuer-key-file issuer.key \
  | kubectl apply -f -
```

### 3. Set Resource Limits

```yaml
# Configure proxy resources for your workload
metadata:
  annotations:
    config.linkerd.io/proxy-cpu-request: "100m"
    config.linkerd.io/proxy-cpu-limit: "500m"
    config.linkerd.io/proxy-memory-request: "20Mi"
    config.linkerd.io/proxy-memory-limit: "250Mi"
```

### 4. Monitor Linkerd Health

```bash
# Regular health checks
linkerd check

# Check control plane
linkerd check --proxy -n linkerd

# Set up alerts for:
# - Control plane pod restarts
# - Certificate expiry (linkerd check warns 60 days before)
# - Proxy injection failures
```

### 5. Keep Linkerd Updated

```bash
# Check for updates
linkerd upgrade | kubectl apply -f -

# Linkerd supports zero-downtime upgrades
# Proxies are updated when pods restart
```

## 15. Common Troubleshooting

### Problem: Pods Won't Start After Injection

```bash
# Check proxy injector logs
kubectl logs -n linkerd -l linkerd.io/control-plane-component=proxy-injector

# Common cause: init container fails to set up iptables
# Fix: Check if your cluster supports NET_ADMIN capability
kubectl describe pod <pod-name> | grep -A 5 "Init Containers"
```

### Problem: mTLS Not Working

```bash
# Verify both pods are meshed
linkerd stat deployment -n default
# Check MESHED column — should show 2/2

# Check edges for TLS status
linkerd edges deployment -n default
# SECURED column should show √

# If not secured, check that the identity component is healthy
linkerd check | grep identity
```

### Problem: High Latency After Meshing

```bash
# Check proxy resource usage
kubectl top pod <pod-name> --containers

# If proxy is CPU-starved, increase limits:
# config.linkerd.io/proxy-cpu-request: "200m"

# Check if Service Profiles are causing excessive retries
linkerd routes deployment/backend -n default
# Look for high EFFECTIVE_RPS vs actual RPS
```

### Problem: `linkerd check` Fails

```bash
# Run with verbose output
linkerd check --verbose

# Common issues:
# ❌ Certificate expiry — regenerate issuer certificate
# ❌ Clock skew — sync NTP on all nodes
# ❌ Webhook failure — restart proxy-injector
kubectl rollout restart deployment linkerd-proxy-injector -n linkerd
```

### General Debug Commands

```bash
# Check proxy logs for a specific pod
kubectl logs <pod-name> -c linkerd-proxy

# Check control plane logs
kubectl logs -n linkerd -l linkerd.io/control-plane-component=destination

# Check proxy configuration
linkerd diagnostics proxy-metrics -n default <pod-name>

# Comprehensive diagnostics
linkerd diagnostics -n default
```
