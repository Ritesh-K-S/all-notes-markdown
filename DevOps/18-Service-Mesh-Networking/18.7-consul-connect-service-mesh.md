# 18.7 — Consul Connect — Service Mesh by HashiCorp

> **One Line Answer:** Consul Connect is HashiCorp's service mesh solution that provides **secure service-to-service communication** using mutual TLS (mTLS), traffic management, and service discovery — all managed through the same Consul tool you might already use for service discovery and configuration.

---

## 🤔 Wait, What Even is Consul?

Before we dive into "Consul Connect," let's first understand **Consul** itself.

**Consul** is a tool made by **HashiCorp** (the same folks who made Terraform, Vault, Vagrant, etc.). Consul was originally built for:

1. **Service Discovery** — "Where is my payment service running?"
2. **Health Checking** — "Is my payment service healthy?"
3. **Key-Value Store** — Store configuration data
4. **Multi-Datacenter** — Work across multiple data centers

Think of Consul as a **phone book + health checker** for your microservices.

```
Without Consul:
  Service A: "Hmm, where is Service B? What's its IP? Is it even alive?"
  
With Consul:
  Service A: "Hey Consul, where's Service B?"
  Consul: "It's at 10.0.2.15:8080, and it's healthy ✅"
```

---

## 🔌 So What is Consul Connect?

**Consul Connect** is the **service mesh layer** built on top of Consul. It was added later to give Consul superpowers:

| Feature | Plain Consul | Consul Connect (Service Mesh) |
|---------|-------------|-------------------------------|
| Service Discovery | ✅ Yes | ✅ Yes |
| Health Checking | ✅ Yes | ✅ Yes |
| KV Store | ✅ Yes | ✅ Yes |
| Encrypted Communication (mTLS) | ❌ No | ✅ Yes |
| Traffic Control (routing, splitting) | ❌ No | ✅ Yes |
| Authorization (who can talk to whom) | ❌ No | ✅ Yes |
| Sidecar Proxies | ❌ No | ✅ Yes |

In simple words:

> **Consul Connect = Consul + Service Mesh features (mTLS, traffic management, access control)**

## 🏗️ Architecture — How Does Consul Connect Work?

Let's break down the architecture piece by piece.

### Key Components

```
┌──────────────────────────────────────────────────────┐
│                   CONSUL CLUSTER                      │
│                                                       │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐  │
│  │   Server 1   │  │   Server 2   │  │   Server 3   │  │
│  │   (Leader)   │  │  (Follower)  │  │  (Follower)  │  │
│  └─────────────┘  └─────────────┘  └─────────────┘  │
│         ▲                 ▲                ▲          │
│         │     Raft Consensus Protocol      │          │
│         └─────────────┼────────────────────┘          │
│                       │                               │
└───────────────────────┼───────────────────────────────┘
                        │
          ┌─────────────┼─────────────┐
          │             │             │
    ┌─────▼─────┐ ┌─────▼─────┐ ┌─────▼─────┐
    │  Agent     │ │  Agent     │ │  Agent     │
    │  (Client)  │ │  (Client)  │ │  (Client)  │
    │            │ │            │ │            │
    │ ┌────────┐ │ │ ┌────────┐ │ │ ┌────────┐ │
    │ │Sidecar │ │ │ │Sidecar │ │ │ │Sidecar │ │
    │ │ Proxy  │ │ │ │ Proxy  │ │ │ │ Proxy  │ │
    │ └───┬────┘ │ │ └───┬────┘ │ │ └───┬────┘ │
    │     │      │ │     │      │ │     │      │
    │ ┌───▼────┐ │ │ ┌───▼────┐ │ │ ┌───▼────┐ │
    │ │Service │ │ │ │Service │ │ │ │Service │ │
    │ │   A    │ │ │ │   B    │ │ │ │   C    │ │
    │ └────────┘ │ │ └────────┘ │ │ └────────┘ │
    └────────────┘ └────────────┘ └────────────┘
       Node 1          Node 2          Node 3
```

### Let's Understand Each Part:

#### 1. Consul Servers (The Brain)

- These are the **central coordination** servers.
- Usually you run **3 or 5** of them (odd number for consensus).
- They use the **Raft consensus protocol** — one is the **Leader**, others are **Followers**.
- They store all service information, health status, and configuration.

Think of them like the **central management office** that knows everything about every service.

#### 2. Consul Agents (Client Mode)

- Each **node** (server/VM/container) runs a lightweight **Consul agent** in client mode.
- The agent talks to the Consul servers and reports the health of local services.
- It's like a **local representative** of Consul on each machine.

#### 3. Sidecar Proxy (Envoy or Built-in)

- This is the **heart of Consul Connect**.
- Each service gets a **sidecar proxy** running next to it (usually **Envoy Proxy**).
- All traffic goes **through the proxy** — the service itself doesn't need to know about mTLS or routing.

```
Without Sidecar:
  Service A ──────────────────────→ Service B
  (plain HTTP, no encryption, no control)

With Sidecar (Consul Connect):
  Service A → [Sidecar Proxy A] ══mTLS══ [Sidecar Proxy B] → Service B
                                  (encrypted, authorized, observed)
```

## 🔐 Feature 1: Mutual TLS (mTLS) — Automatic Encryption

One of the biggest features of Consul Connect is **automatic mTLS**.

### What is mTLS?

Regular TLS (like HTTPS) = Only the **server** proves its identity.
Mutual TLS (mTLS) = **Both** the client AND the server prove their identity to each other.

```
Regular TLS:
  Client: "Hey Server, prove you are who you say you are"
  Server: "Here's my certificate" ✅
  Client: "OK, I trust you"
  (But the server never verified who the CLIENT is!)

Mutual TLS (mTLS):
  Client: "Hey Server, prove you are who you say you are"
  Server: "Here's my certificate" ✅
  Server: "Now YOU prove who YOU are"
  Client: "Here's MY certificate" ✅
  Both: "We trust each other. Let's encrypt our conversation." 🔒
```

### How Consul Connect Does mTLS:

1. Consul has a **built-in Certificate Authority (CA)**.
2. It automatically **generates and rotates TLS certificates** for every service.
3. The sidecar proxies handle all the mTLS — your service code doesn't change at all!

```
Consul CA
   │
   ├── Issues certificate for Service A → Sidecar A uses it
   ├── Issues certificate for Service B → Sidecar B uses it
   └── Issues certificate for Service C → Sidecar C uses it

All certificates are automatically rotated (renewed) before they expire!
```

**The beauty?** You get encryption between ALL services with **zero code changes**. Your app still talks plain HTTP to `localhost`, and the sidecar handles the rest.

## 🚦 Feature 2: Intentions — Who Can Talk to Whom?

**Intentions** are Consul Connect's way of defining **access control** — which service is allowed to communicate with which other service.

### Think of It Like a Bouncer at a Club:

```
Without Intentions:
  Any service can talk to any other service
  Service A → Service B ✅
  Service A → Database ✅  ← DANGEROUS! What if Service A is compromised?

With Intentions:
  You define explicit rules:
  ✅ "Web frontend" CAN talk to "API Gateway"
  ✅ "API Gateway" CAN talk to "Payment Service"
  ❌ "Web frontend" CANNOT talk to "Database" directly
  ❌ "Payment Service" CANNOT talk to "User Service"
```

### Defining Intentions via CLI:

```bash
# Allow "web" to talk to "api"
consul intention create -allow web api

# Deny "web" from talking to "database"
consul intention create -deny web database

# List all intentions
consul intention list

# Delete an intention
consul intention delete web api
```

### Defining Intentions via Configuration (HCL):

```hcl
# intentions.hcl
Kind = "service-intentions"
Name = "api"          # The DESTINATION service

Sources = [
  {
    Name   = "web"    # The SOURCE service
    Action = "allow"  # Allow web → api
  },
  {
    Name   = "payment"
    Action = "allow"  # Allow payment → api
  },
  {
    Name   = "*"      # Everything else
    Action = "deny"   # DENY by default
  }
]
```

Apply it:

```bash
consul config write intentions.hcl
```

### What happens when a denied service tries to connect?

```
Service "web" tries to connect to "database"
     ↓
Sidecar Proxy checks with Consul: "Is web allowed to talk to database?"
     ↓
Consul: "NOPE ❌ — there's a DENY intention"
     ↓
Connection REFUSED. Traffic never reaches the database.
```

**This is so powerful!** You're controlling service-to-service access without touching any firewall rules or network configurations.

## 🌐 Feature 3: Service Discovery (Built-In)

Unlike Istio or Linkerd where you need a separate service discovery mechanism, Consul **comes with built-in service discovery**.

### How Services Register:

Services can register in multiple ways:

#### Option 1: Configuration File

```json
{
  "service": {
    "name": "web",
    "port": 8080,
    "tags": ["v1", "production"],
    "check": {
      "http": "http://localhost:8080/health",
      "interval": "10s",
      "timeout": "3s"
    },
    "connect": {
      "sidecar_service": {}
    }
  }
}
```

Save this as `web-service.json` in Consul's config directory.

**Let's break this down:**
- `name`: The name other services use to find this one
- `port`: Which port the service listens on
- `tags`: Labels for organizing and filtering
- `check`: Health check — Consul will hit `/health` every 10 seconds
- `connect.sidecar_service`: This tells Consul to spin up a sidecar proxy for this service (this is what makes it part of the mesh!)

#### Option 2: HTTP API

```bash
curl -X PUT http://localhost:8500/v1/agent/service/register \
  -H "Content-Type: application/json" \
  -d '{
    "Name": "payment",
    "Port": 9090,
    "Check": {
      "HTTP": "http://localhost:9090/health",
      "Interval": "10s"
    },
    "Connect": {
      "SidecarService": {}
    }
  }'
```

### Discovering Services:

```bash
# Find all instances of the "web" service
consul catalog services

# Get details about a specific service
consul catalog nodes -service=web

# DNS-based discovery (Consul runs a DNS server!)
dig @localhost -p 8600 web.service.consul

# Result: 10.0.1.5  ← IP of the web service
```

**DNS-based discovery is super cool** — any application that can do DNS lookups can find services without any special client library!

```
Your app: "I need to connect to the payment service"
Your app: connects to "payment.service.consul:9090"
Consul DNS: resolves it to the actual healthy instance → 10.0.2.15:9090
```

## 🚀 Feature 4: Traffic Management

Consul Connect gives you fine-grained control over how traffic flows between services.

### Service Splitters — Canary Deployments

Split traffic between different versions of a service:

```hcl
# service-splitter.hcl
Kind = "service-splitter"
Name = "api"

Splits = [
  {
    Weight        = 90
    ServiceSubset = "v1"
  },
  {
    Weight        = 10
    ServiceSubset = "v2"
  }
]
```

This sends **90% of traffic to v1** and **10% to v2** — perfect for canary deployments!

```
                    ┌──── 90% ────→ API v1 (stable)
Incoming Traffic ───┤
                    └──── 10% ────→ API v2 (canary — testing)
```

### Service Resolvers — Define Subsets

```hcl
# service-resolver.hcl
Kind = "service-resolver"
Name = "api"

DefaultSubset = "v1"

Subsets = {
  "v1" = {
    Filter = "Service.Meta.version == v1"
  }
  "v2" = {
    Filter = "Service.Meta.version == v2"
  }
}
```

### Service Routers — Route Based on Headers, Paths, etc.

```hcl
# service-router.hcl
Kind = "service-router"
Name = "api"

Routes = [
  {
    Match {
      HTTP {
        PathPrefix = "/api/v2"
      }
    }
    Destination {
      ServiceSubset = "v2"
    }
  },
  {
    Match {
      HTTP {
        Header = [
          {
            Name  = "x-canary"
            Exact = "true"
          }
        ]
      }
    }
    Destination {
      ServiceSubset = "v2"
    }
  }
  # Default: everything else goes to v1
]
```

**What this does:**
- Requests to `/api/v2/*` → go to API v2
- Requests with header `x-canary: true` → go to API v2
- Everything else → goes to API v1 (default)

## ☸️ Consul on Kubernetes

Consul Connect works beautifully with Kubernetes. HashiCorp provides an official **Helm chart** to deploy Consul on K8s.

### Installation via Helm:

```bash
# Add the HashiCorp Helm repo
helm repo add hashicorp https://helm.releases.hashicorp.com
helm repo update

# Create a values file
cat > consul-values.yaml <<EOF
global:
  name: consul
  datacenter: dc1

server:
  replicas: 3          # 3 Consul servers
  storage: 10Gi        # Persistent storage

client:
  enabled: true        # Run Consul agent on every node

connectInject:
  enabled: true        # Enable Consul Connect sidecar injection
  default: false       # Don't inject by default (opt-in per pod)

controller:
  enabled: true        # Enable the config entry controller

ui:
  enabled: true        # Enable the Consul Web UI
  service:
    type: LoadBalancer
EOF

# Install Consul
helm install consul hashicorp/consul -f consul-values.yaml -n consul --create-namespace
```

**Let's break down the key settings:**

| Setting | What It Does |
|---------|-------------|
| `server.replicas: 3` | Runs 3 Consul servers for high availability |
| `connectInject.enabled: true` | Enables automatic sidecar proxy injection |
| `connectInject.default: false` | Services must opt-in to the mesh (safer) |
| `controller.enabled: true` | Allows Kubernetes CRDs for Consul config |
| `ui.enabled: true` | Gives you a nice web dashboard |

### Adding a Service to the Mesh:

To add a Kubernetes service to the Consul mesh, just add an **annotation** to your Pod:

```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web-app
  template:
    metadata:
      labels:
        app: web-app
      annotations:
        # This single annotation adds the service to Consul Connect mesh!
        consul.hashicorp.com/connect-inject: "true"
    spec:
      containers:
        - name: web-app
          image: myapp:v1
          ports:
            - containerPort: 8080
```

That's it! When this pod starts, Consul will automatically:
1. Register the service in Consul
2. Inject a sidecar proxy (Envoy) into the pod
3. Set up mTLS certificates
4. Enforce any intentions you've defined

### What the Pod Looks Like After Injection:

```
┌─────────────────────────────────────────────┐
│                    POD                       │
│                                              │
│  ┌──────────────┐    ┌───────────────────┐  │
│  │              │    │                   │  │
│  │   web-app    │◄──►│   Envoy Sidecar   │◄────── Incoming mTLS traffic
│  │  (your app)  │    │   (auto-injected) │──────► Outgoing mTLS traffic
│  │              │    │                   │  │
│  │ Port: 8080   │    │ Port: 20000       │  │
│  └──────────────┘    └───────────────────┘  │
│                                              │
│  ┌──────────────────────────────────────┐   │
│  │   consul-connect-inject-init         │   │
│  │   (init container - sets up config)  │   │
│  └──────────────────────────────────────┘   │
└─────────────────────────────────────────────┘
```

### Intentions on Kubernetes (CRD):

```yaml
# intention.yaml
apiVersion: consul.hashicorp.com/v1alpha1
kind: ServiceIntentions
metadata:
  name: api-intentions
spec:
  destination:
    name: api
  sources:
    - name: web
      action: allow
    - name: payment
      action: allow
    - name: "*"
      action: deny      # Deny everything else
```

```bash
kubectl apply -f intention.yaml
```

## 🌍 Multi-Datacenter & Multi-Cloud — Consul's Superpower

This is where Consul truly shines compared to Istio or Linkerd. Consul was **built from day one** for multi-datacenter support.

### The Problem:

```
You have services in:
  - AWS (us-east-1)     → Datacenter "dc1"
  - Azure (westeurope)  → Datacenter "dc2"
  - On-premise          → Datacenter "dc3"

You need services across ALL of these to:
  - Discover each other
  - Communicate securely (mTLS)
  - Be health-checked
```

### How Consul Handles This:

```
┌──────────────┐    WAN Gossip     ┌──────────────┐    WAN Gossip     ┌──────────────┐
│   DC1 (AWS)  │◄─────────────────►│  DC2 (Azure) │◄─────────────────►│  DC3 (On-Prem)│
│              │                    │              │                    │              │
│ Consul       │                    │ Consul       │                    │ Consul       │
│ Servers      │                    │ Servers      │                    │ Servers      │
│              │                    │              │                    │              │
│ Services:    │                    │ Services:    │                    │ Services:    │
│  - web       │                    │  - api       │                    │  - database  │
│  - auth      │                    │  - payment   │                    │  - cache     │
└──────────────┘                    └──────────────┘                    └──────────────┘
```

Services in DC1 can discover and connect to services in DC2 or DC3 seamlessly:

```bash
# From DC1, discover the "api" service in DC2:
dig @localhost -p 8600 api.service.dc2.consul

# It just works! Consul handles cross-datacenter DNS.
```

### Mesh Gateways — The Bridge Between Datacenters:

```hcl
# mesh-gateway config
Kind = "mesh"
Peering {
  PeerThroughMeshGateways = true
}
```

```
DC1                                                   DC2
┌────────────────────┐                    ┌────────────────────┐
│                    │                    │                    │
│  Service A ──► Sidecar ──► Mesh    ══════►  Mesh    ──► Sidecar ──► Service B
│                          Gateway   (mTLS) Gateway                    │
│                    │                    │                    │
└────────────────────┘                    └────────────────────┘

Mesh Gateways handle cross-datacenter mTLS traffic.
Services don't need to know they're talking across datacenters!
```

## 🆚 Consul Connect vs Istio vs Linkerd

Let's do a fair comparison:

| Feature | Consul Connect | Istio | Linkerd |
|---------|---------------|-------|---------|
| **Complexity** | Medium | High | Low |
| **Sidecar Proxy** | Envoy (or built-in) | Envoy | linkerd2-proxy (Rust) |
| **mTLS** | ✅ Built-in CA | ✅ Yes | ✅ Yes |
| **Traffic Management** | ✅ Good | ✅ Excellent | ✅ Good |
| **Multi-Cluster** | ✅ Excellent (built-in) | ✅ Good (complex setup) | ✅ Good |
| **Multi-Platform** | ✅ K8s + VMs + Bare Metal | ⚠️ Primarily K8s | ❌ K8s only |
| **Service Discovery** | ✅ Built-in | ❌ Uses K8s DNS | ❌ Uses K8s DNS |
| **KV Store** | ✅ Built-in | ❌ No | ❌ No |
| **Learning Curve** | Medium | Steep | Easy |
| **Resource Usage** | Medium | Heavy | Light |
| **Best For** | Multi-cloud, VMs + K8s | Kubernetes-heavy | Simple K8s mesh |

### When to Choose Consul Connect:

✅ You have a **mix of Kubernetes AND VMs/bare metal**
✅ You need **multi-datacenter or multi-cloud** support
✅ You already use other **HashiCorp tools** (Vault, Terraform, Nomad)
✅ You want **built-in service discovery** (not just K8s)
✅ You need to mesh services that are **NOT in Kubernetes**

### When NOT to Choose Consul Connect:

❌ You're purely Kubernetes — Linkerd might be simpler
❌ You need the most advanced traffic management — Istio might be better
❌ You want the lightest possible resource footprint — Linkerd wins

## 🛠️ Hands-On: Complete Setup on Kubernetes

Let's walk through a full working example.

### Step 1: Install Consul

```bash
# Add Helm repo
helm repo add hashicorp https://helm.releases.hashicorp.com
helm repo update

# Install
helm install consul hashicorp/consul \
  --set global.name=consul \
  --set server.replicas=1 \
  --set connectInject.enabled=true \
  --set connectInject.default=false \
  --set controller.enabled=true \
  --set ui.enabled=true \
  -n consul --create-namespace
```

### Step 2: Deploy Two Services

**Frontend Service:**

```yaml
# frontend.yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: frontend
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: frontend
spec:
  replicas: 2
  selector:
    matchLabels:
      app: frontend
  template:
    metadata:
      labels:
        app: frontend
      annotations:
        consul.hashicorp.com/connect-inject: "true"     # Join the mesh
    spec:
      serviceAccountName: frontend
      containers:
        - name: frontend
          image: hashicorp/http-echo
          args:
            - "-text=Hello from frontend"
            - "-listen=:8080"
          ports:
            - containerPort: 8080
---
apiVersion: v1
kind: Service
metadata:
  name: frontend
spec:
  selector:
    app: frontend
  ports:
    - port: 8080
      targetPort: 8080
```

**Backend Service:**

```yaml
# backend.yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: backend
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: backend
spec:
  replicas: 2
  selector:
    matchLabels:
      app: backend
  template:
    metadata:
      labels:
        app: backend
      annotations:
        consul.hashicorp.com/connect-inject: "true"     # Join the mesh
    spec:
      serviceAccountName: backend
      containers:
        - name: backend
          image: hashicorp/http-echo
          args:
            - "-text=Hello from backend"
            - "-listen=:9090"
          ports:
            - containerPort: 9090
---
apiVersion: v1
kind: Service
metadata:
  name: backend
spec:
  selector:
    app: backend
  ports:
    - port: 9090
      targetPort: 9090
```

```bash
kubectl apply -f frontend.yaml
kubectl apply -f backend.yaml
```

### Step 3: Define Intentions

```yaml
# intentions.yaml
apiVersion: consul.hashicorp.com/v1alpha1
kind: ServiceIntentions
metadata:
  name: backend
spec:
  destination:
    name: backend
  sources:
    - name: frontend
      action: allow         # frontend CAN talk to backend
    - name: "*"
      action: deny          # everything else is DENIED
```

```bash
kubectl apply -f intentions.yaml
```

### Step 4: Test the Connection

```bash
# Get a shell in the frontend pod
kubectl exec -it deployment/frontend -c frontend -- sh

# Try to reach the backend through the mesh
# Consul sets up an upstream on localhost
curl http://localhost:9090

# Expected output: "Hello from backend" ✅
```

### Step 5: Verify mTLS is Working

```bash
# Check that the sidecar proxy is running
kubectl get pods
# You'll see 2 containers per pod (app + envoy-sidecar)

# Check proxy configuration
kubectl exec deployment/frontend -c consul-sidecar -- consul connect proxy status
```

## 📊 Consul Connect Observability

Consul Connect provides observability features through its proxy layer:

### Metrics

The Envoy sidecar proxies emit detailed metrics. You can scrape them with Prometheus:

```hcl
# Enable Prometheus metrics in Helm values
connectInject:
  enabled: true
  metrics:
    defaultEnabled: true      # Enable metrics for all injected services
    defaultPrometheusScrapePort: 20200
    defaultPrometheusScrapePath: "/metrics"
```

### Useful Metrics:

| Metric | What It Tells You |
|--------|------------------|
| `envoy_http_downstream_rq_total` | Total incoming requests |
| `envoy_http_downstream_rq_xx` | Requests by status code (2xx, 4xx, 5xx) |
| `envoy_cluster_upstream_cx_connect_timeout` | Connection timeouts to upstream |
| `envoy_cluster_upstream_rq_time` | Request duration (latency) |

## 🔑 Key Takeaways

1. **Consul Connect = Consul + Service Mesh** — It adds mTLS, traffic management, and access control to Consul's service discovery and health checking.

2. **mTLS is automatic** — Consul's built-in CA generates and rotates certificates. Zero code changes needed.

3. **Intentions = Access Control** — Simple allow/deny rules for service-to-service communication. No firewall rules needed.

4. **Multi-platform** — Works on Kubernetes, VMs, bare metal, and across multiple datacenters/clouds. This is its biggest differentiator.

5. **HashiCorp Ecosystem** — Integrates beautifully with Vault (for secrets), Terraform (for infrastructure), and Nomad (for orchestration).

6. **Envoy-based** — Uses the same battle-tested Envoy proxy that Istio uses, so you get all the proxy-level features.

7. **DNS-based discovery** — Any app can discover services using simple DNS queries (`myservice.service.consul`).
