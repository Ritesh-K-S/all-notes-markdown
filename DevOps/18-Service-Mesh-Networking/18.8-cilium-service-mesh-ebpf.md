## 🤔 Before We Start — What's the Problem with Traditional Service Meshes?

Traditional service meshes (Istio, Linkerd, Consul Connect) all work using the **sidecar proxy pattern**:

```
Traditional Service Mesh (Sidecar Pattern):

┌─────────────────────────────────────────┐
│                  POD                     │
│                                          │
│  ┌──────────────┐  ┌─────────────────┐  │
│  │  Your App     │  │  Sidecar Proxy  │  │
│  │  Container    │  │  (Envoy)        │  │
│  │              │  │                  │  │
│  │  Uses ~200MB  │  │  Uses ~50-100MB │  │
│  │  RAM          │  │  RAM EXTRA!     │  │
│  └──────────────┘  └─────────────────┘  │
│                                          │
│  Every pod has this extra container!     │
└─────────────────────────────────────────┘
```

### The Pain Points:

| Problem | Why It Hurts |
|---------|-------------|
| **Extra resource usage** | Every single pod gets a sidecar = huge RAM/CPU overhead at scale |
| **Added latency** | Every request goes: App → Sidecar → Network → Sidecar → App (extra hops) |
| **Complexity** | Managing thousands of sidecar proxies is a nightmare |
| **Slow startup** | Pods take longer to start because the sidecar needs to initialize |
| **Upgrade pain** | Upgrading the mesh means restarting every pod to update every sidecar |

### Let's Do Some Math:

```
Say you have 500 pods.
Each sidecar uses ~70MB of RAM.

500 pods × 70MB = 35 GB of RAM just for sidecars! 😱

That's RAM not being used for your actual applications.
```

**Cilium's approach:** What if we could do all of this **inside the Linux kernel** — no sidecars needed?

## 🧬 What is eBPF? (The Technology Behind Cilium)

Before understanding Cilium, you **must** understand eBPF. It's the secret sauce.

### eBPF in Simple Terms:

**eBPF** stands for **Extended Berkeley Packet Filter**.

Think of eBPF as the ability to **run small programs directly inside the Linux kernel** — without modifying the kernel itself or loading kernel modules.

```
Without eBPF:
  Your application lives in "user space"
  The Linux kernel lives in "kernel space"
  
  To do networking, security, monitoring:
    App (user space) ←→ System Call ←→ Kernel (kernel space)
    
  This crossing between user space and kernel space is SLOW.

With eBPF:
  You inject tiny programs DIRECTLY into the kernel
  These programs run at kernel speed — no crossing needed!
  
  ┌────────────────────────────────────┐
  │          User Space                │
  │   ┌─────────────────────────┐     │
  │   │     Your Application    │     │
  │   └─────────────────────────┘     │
  ├────────────────────────────────────┤
  │          Kernel Space              │
  │   ┌─────────────────────────┐     │
  │   │  eBPF Programs          │     │
  │   │  - Networking hooks     │     │
  │   │  - Security hooks       │     │
  │   │  - Observability hooks  │     │
  │   └─────────────────────────┘     │
  │                                    │
  │   These run at KERNEL SPEED! 🚀   │
  └────────────────────────────────────┘
```

### A Real-World Analogy:

Imagine a highway toll booth.

**Without eBPF (Traditional Sidecar):**
```
Every car (packet) must:
  1. Exit the highway
  2. Stop at a separate toll booth (sidecar proxy)
  3. Pay the toll (get inspected, encrypted, logged)
  4. Get back on the highway
  
This is SLOW — every car has to take a detour!
```

**With eBPF (Cilium):**
```
The toll collection is built INTO the road itself:
  1. Car drives normally
  2. Sensors in the road automatically scan, charge, log
  3. Car never slows down!
  
This is FAST — no detour needed!
```

### eBPF Hooks — Where Can It Attach?

eBPF programs can attach to various points in the kernel:

```
┌────────────────────────────────────────────────┐
│                Linux Kernel                     │
│                                                 │
│   Network Packet Arrives                        │
│         │                                       │
│         ▼                                       │
│   ┌─── XDP (eXpress Data Path) ◄── eBPF Hook   │
│   │    (Earliest point — even before the        │
│   │     kernel's network stack!)                 │
│   │                                              │
│   ▼                                              │
│   ┌─── TC (Traffic Control) ◄── eBPF Hook       │
│   │    (After basic kernel processing)           │
│   │                                              │
│   ▼                                              │
│   ┌─── Socket Layer ◄── eBPF Hook               │
│   │    (Before reaching the application)         │
│   │                                              │
│   ▼                                              │
│   Application receives the packet                │
│                                                  │
│   (eBPF can also hook into:)                     │
│   - System calls                                 │
│   - Kernel functions (kprobes)                   │
│   - User-space functions (uprobes)               │
│   - Tracepoints                                  │
│   - Cgroups                                      │
└──────────────────────────────────────────────────┘
```

**Key takeaway:** eBPF can intercept, inspect, modify, redirect, or drop network packets **at kernel speed** — without any sidecar proxy!

## 🐝 What is Cilium?

**Cilium** is an open-source project that uses eBPF to provide:

1. **Networking** — Pod-to-pod, service-to-service, and external connectivity
2. **Security** — Network policies, identity-based security, encryption
3. **Observability** — Deep visibility into network flows without instrumenting apps
4. **Service Mesh** — L4/L7 traffic management without sidecars (or with optional Envoy)

Cilium was created by **Isovalent** (now part of Cisco) and is a **CNCF Graduated project** — the highest level of maturity in the Cloud Native Computing Foundation.

```
Cilium = eBPF + Kubernetes Networking + Service Mesh + Security + Observability

All in one package! 🎁
```

### Key Components:

#### 1. Cilium Agent (DaemonSet)

- Runs on **every node** in the cluster.
- Loads **eBPF programs** into the Linux kernel.
- Manages networking, security policies, and load balancing.
- Monitors pod lifecycle and updates eBPF programs accordingly.

#### 2. Cilium Operator

- Runs as a **Deployment** (usually 2 replicas).
- Handles cluster-wide operations.
- Manages IP address allocation (IPAM).
- Synchronizes state across the cluster.

#### 3. Hubble — Observability Layer

- Provides **deep network visibility**.
- Shows you traffic flows between pods/services.
- Has a beautiful **UI** for visualizing service maps.
- CLI for real-time flow monitoring.

Think of Hubble as having **X-ray vision** into your cluster's network.

#### 4. eBPF Programs (The Real Magic)

- Tiny programs injected into the **kernel** by the Cilium Agent.
- They handle packet processing, policy enforcement, load balancing, and metrics collection.
- Run at **near-native kernel speed**.

## 🚀 Cilium Service Mesh — How It Works WITHOUT Sidecars

This is the revolutionary part. Let's compare:

### Traditional Service Mesh (Istio/Linkerd):

```
Pod A                                              Pod B
┌──────────────────────┐                   ┌──────────────────────┐
│  App  →  Envoy Proxy ─────network─────── Envoy Proxy  →  App   │
│         (sidecar)    │                   │  (sidecar)           │
└──────────────────────┘                   └──────────────────────┘

Packet path: App → localhost → Envoy → kernel → network → kernel → Envoy → localhost → App
Number of context switches: MANY 🐌
```

### Cilium Service Mesh (eBPF-Based):

```
Pod A                                              Pod B
┌──────────────┐                           ┌──────────────┐
│     App      │                           │     App      │
│  (no sidecar!)│                          │  (no sidecar!)│
└──────┬───────┘                           └──────▲───────┘
       │                                          │
═══════╪══════════════════════════════════════════╪════════
       │           Linux Kernel                    │
       │    ┌─────────────────────────────────┐   │
       └───►│  eBPF Programs (Cilium)          │───┘
            │                                   │
            │  ✅ Encrypt (WireGuard/IPsec)     │
            │  ✅ Apply network policy           │
            │  ✅ Load balance                   │
            │  ✅ Collect metrics                 │
            │  ✅ Route traffic                   │
            └─────────────────────────────────┘
═══════════════════════════════════════════════════════════

Packet path: App → kernel (eBPF handles everything) → App
Number of context switches: MINIMAL 🚀
```

### The Performance Difference:

```
Scenario: 1000 requests per second

Traditional Sidecar Mesh:
  Latency per request: ~1-5ms overhead (sidecar processing)
  Extra memory: ~70MB per pod × 500 pods = 35GB
  Extra CPU: significant

Cilium eBPF Mesh:
  Latency per request: ~0.1-0.5ms overhead (kernel processing)
  Extra memory: ~0 per pod (eBPF is in the kernel, shared per node)
  Extra CPU: minimal

That's up to 10x less latency and dramatically less resource usage! 📊
```

## 🔧 Installing Cilium

### Prerequisites:

- A Kubernetes cluster (without another CNI — Cilium replaces your CNI!)
- Linux kernel 4.19+ (5.10+ recommended for all features)
- Helm 3.x

### Method 1: Cilium CLI (Easiest)

```bash
# Install Cilium CLI
# On Linux:
CILIUM_CLI_VERSION=$(curl -s https://raw.githubusercontent.com/cilium/cilium-cli/main/stable.txt)
CLI_ARCH=amd64
curl -L --fail --remote-name-all \
  https://github.com/cilium/cilium-cli/releases/download/${CILIUM_CLI_VERSION}/cilium-linux-${CLI_ARCH}.tar.gz
sudo tar xzvf cilium-linux-${CLI_ARCH}.tar.gz -C /usr/local/bin
rm cilium-linux-${CLI_ARCH}.tar.gz

# Install Cilium in your cluster
cilium install

# Verify the installation
cilium status

# Expected output:
#     /¯¯\
#  /¯¯\__/¯¯\    Cilium:          OK
#  \__/¯¯\__/    Operator:        OK
#  /¯¯\__/¯¯\    Hubble Relay:    disabled
#  \__/¯¯\__/    ClusterMesh:     disabled
#  \__/

# Enable Hubble (observability)
cilium hubble enable --ui

# Run connectivity test
cilium connectivity test
```

### Method 2: Helm (More Control)

```bash
# Add Cilium Helm repo
helm repo add cilium https://helm.cilium.io/
helm repo update

# Install Cilium with custom values
helm install cilium cilium/cilium \
  --version 1.15.0 \
  --namespace kube-system \
  --set hubble.enabled=true \
  --set hubble.relay.enabled=true \
  --set hubble.ui.enabled=true \
  --set hubble.metrics.enableOpenMetrics=true \
  --set encryption.enabled=true \
  --set encryption.type=wireguard \
  --set kubeProxyReplacement=true \
  --set loadBalancer.algorithm=maglev
```

**Let's break down those flags:**

| Flag | What It Does |
|------|-------------|
| `hubble.enabled=true` | Enables Hubble observability |
| `hubble.relay.enabled=true` | Enables Hubble Relay (aggregates data from all nodes) |
| `hubble.ui.enabled=true` | Enables the Hubble web UI |
| `encryption.enabled=true` | Enables transparent encryption |
| `encryption.type=wireguard` | Uses WireGuard for encryption (fast!) |
| `kubeProxyReplacement=true` | Replaces kube-proxy with eBPF (faster!) |
| `loadBalancer.algorithm=maglev` | Uses Maglev consistent hashing for load balancing |

### Verify Installation:

```bash
# Check Cilium pods
```

## 🔐 Feature 1: Network Policies (On Steroids!)

Cilium supports standard **Kubernetes NetworkPolicy** but also provides its own **CiliumNetworkPolicy** with WAY more features.

### Standard Kubernetes NetworkPolicy:

```yaml
# Basic K8s network policy (works with Cilium)
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-backend
spec:
  podSelector:
    matchLabels:
      app: backend
  policyTypes:
    - Ingress
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: frontend
      ports:
        - protocol: TCP
          port: 8080
```

### CiliumNetworkPolicy — The Supercharged Version:

```yaml
# Cilium-specific network policy with L7 (HTTP) rules!
apiVersion: cilium.io/v2
kind: CiliumNetworkPolicy
metadata:
  name: l7-policy
spec:
  endpointSelector:
    matchLabels:
      app: backend-api
  ingress:
    - fromEndpoints:
        - matchLabels:
            app: frontend
      toPorts:
        - ports:
            - port: "8080"
              protocol: TCP
          rules:
            http:                              # L7 HTTP rules!
              - method: "GET"                  # Only allow GET
                path: "/api/products"          # Only to this path
              - method: "GET"
                path: "/api/health"
              - method: "POST"
                path: "/api/orders"            # Allow POST to orders
                headers:
                  - 'Content-Type: application/json'  # Must be JSON!
```

**This is incredibly powerful!** With standard Kubernetes NetworkPolicy, you can only say "frontend can talk to backend on port 8080." With CiliumNetworkPolicy, you can say "frontend can only make GET requests to `/api/products` and POST requests to `/api/orders` with JSON content type."

### Comparison of Policy Capabilities:

```
Standard K8s NetworkPolicy:
  ✅ Allow/deny by pod labels
  ✅ Allow/deny by namespace
  ✅ Allow/deny by IP range (CIDR)
  ✅ Allow/deny by port
  ❌ Cannot filter by HTTP method
  ❌ Cannot filter by HTTP path
  ❌ Cannot filter by headers
  ❌ Cannot filter by DNS name
  ❌ Cannot filter by Kafka topic
  ❌ Cannot filter by gRPC method

CiliumNetworkPolicy:
  ✅ Everything above, PLUS:
  ✅ Filter by HTTP method (GET, POST, PUT, DELETE)
  ✅ Filter by HTTP path (/api/v1/users)
  ✅ Filter by HTTP headers
  ✅ Filter by DNS name (allow traffic to *.example.com)
  ✅ Filter by Kafka topic and API key
  ✅ Filter by gRPC service/method
  ✅ Identity-based (not just IP-based)
  ✅ Cluster-wide policies (CiliumClusterwideNetworkPolicy)
```

### DNS-Based Policy (Restrict Egress by Domain):

```yaml
# Only allow pods to access specific external domains
apiVersion: cilium.io/v2
kind: CiliumNetworkPolicy
metadata:
  name: allow-specific-external
spec:
  endpointSelector:
    matchLabels:
      app: my-app
  egress:
    - toEndpoints:
        - matchLabels:
            io.kubernetes.pod.namespace: kube-system
            k8s-app: kube-dns
      toPorts:
        - ports:
            - port: "53"
              protocol: ANY
          rules:
            dns:
              - matchPattern: "*.example.com"    # Allow DNS queries for example.com
              - matchPattern: "api.github.com"   # Allow DNS queries for GitHub API
    - toFQDNs:
        - matchPattern: "*.example.com"          # Allow traffic to example.com
        - matchName: "api.github.com"            # Allow traffic to GitHub API
    # Everything else is DENIED by default
```

**This means:** Your pod can ONLY make outbound connections to `*.example.com` and `api.github.com`. Any attempt to connect to other domains will be blocked. This is amazing for security!

## 🔒 Feature 2: Transparent Encryption

Cilium supports transparent encryption between pods — just like mTLS in traditional service meshes, but done in the kernel!

### Option 1: WireGuard Encryption (Recommended)

```bash
# Enable WireGuard encryption via Helm
helm upgrade cilium cilium/cilium \
  --namespace kube-system \
  --set encryption.enabled=true \
  --set encryption.type=wireguard \
  --set encryption.nodeEncryption=true   # Also encrypt node-to-node traffic
```

Or in `values.yaml`:

```yaml
encryption:
  enabled: true
  type: wireguard
  nodeEncryption: true    # Encrypt all node-to-node traffic too
```

### Option 2: IPsec Encryption

```yaml
encryption:
  enabled: true
  type: ipsec
```

### WireGuard vs IPsec:

| Feature | WireGuard | IPsec |
|---------|-----------|-------|
| **Performance** | Very fast (modern crypto) | Slower (more overhead) |
| **Simplicity** | Simple | Complex |
| **Code size** | ~4,000 lines | ~400,000 lines |
| **Setup** | Automatic key exchange | Requires PSK or certificates |
| **Kernel support** | Linux 5.6+ | Linux 2.6+ |

### How It Works:

```
Without Encryption:
  Pod A ──── plain text ────► Pod B
  (Anyone sniffing the network can read this!)

With Cilium WireGuard:
  Pod A ──── encrypted tunnel ────► Pod B
  
  Done in the KERNEL via eBPF + WireGuard
  No sidecar needed
  No application changes needed
  Near zero performance impact!
```

### Verify Encryption:

```bash
# Check encryption status
cilium encrypt status

# Expected output:
# Encryption: Wireguard
#   Listening port: 51871
#   Interface: cilium_wg0
#   Peer count: 3         (one per other node)

# Check WireGuard stats per node
kubectl exec -n kube-system ds/cilium -- cilium encrypt status

# You can also verify with tcpdump — you should see encrypted WireGuard packets
kubectl exec -n kube-system ds/cilium -- tcpdump -i eth0 port 51871
```

```
                    ┌───────────────────────────┐

               │              │              │
```

### Powerful Hubble Filters:

```bash
# Watch traffic FROM a specific pod

# Watch traffic TO a specific service

```

## ⚡ Feature 4: kube-proxy Replacement

Cilium can completely **replace kube-proxy** with a faster eBPF-based implementation!

### What Does kube-proxy Do?

kube-proxy manages Kubernetes **Services** — it makes sure when you connect to a Service IP, your request gets routed to one of the backend Pods.

**Traditional kube-proxy uses iptables:**

```
iptables rules for a Service with 3 pods:

-A KUBE-SERVICES -d 10.96.0.100/32 -p tcp -m tcp --dport 80 -j KUBE-SVC-XXXXX
-A KUBE-SVC-XXXXX -m statistic --mode random --probability 0.333 -j KUBE-SEP-AAA
-A KUBE-SVC-XXXXX -m statistic --mode random --probability 0.500 -j KUBE-SEP-BBB
-A KUBE-SVC-XXXXX -j KUBE-SEP-CCC
-A KUBE-SEP-AAA -p tcp -j DNAT --to-destination 10.0.1.5:8080
-A KUBE-SEP-BBB -p tcp -j DNAT --to-destination 10.0.1.6:8080
-A KUBE-SEP-CCC -p tcp -j DNAT --to-destination 10.0.2.3:8080

Multiply this by hundreds of services = THOUSANDS of iptables rules!
Every packet traverses ALL rules linearly = O(n) = SLOW at scale 🐌
```

**Cilium's eBPF replacement:**

```
eBPF hash map:
  ServiceIP:Port → [Pod1:Port, Pod2:Port, Pod3:Port]
  10.96.0.100:80 → [10.0.1.5:8080, 10.0.1.6:8080, 10.0.2.3:8080]

Lookup is O(1) = constant time regardless of number of services! 🚀
```

### Enable kube-proxy Replacement:

```bash
# During installation:
helm install cilium cilium/cilium \
  --namespace kube-system \
  --set kubeProxyReplacement=true \
  --set k8sServiceHost=<API_SERVER_IP> \
  --set k8sServicePort=<API_SERVER_PORT>
```

### Performance Comparison:

```
Scenario: Cluster with 5000 Services

kube-proxy (iptables):
  - 20,000+ iptables rules
  - Service routing latency: ~5ms (linear rule traversal)
  - Rule update time: ~5-10 seconds (full chain rebuild)
  - Memory: significant for rule storage

Cilium (eBPF):
  - Single eBPF hash map
  - Service routing latency: ~0.1ms (O(1) lookup)
  - Rule update time: ~1ms (just update the map entry)
  - Memory: minimal
```

## 🌐 Feature 5: Cilium Service Mesh (L7 Traffic Management)

For advanced L7 (HTTP, gRPC) traffic management, Cilium offers two modes:

### Mode 1: Sidecar-Free (eBPF only) — For L3/L4 + Basic L7

```
All L3/L4 policies + basic L7 (HTTP path/method matching)
are handled purely in eBPF. No sidecar needed!

Pod ──► eBPF (kernel) ──► Network ──► eBPF (kernel) ──► Pod
```

### Mode 2: Per-Node Envoy Proxy — For Advanced L7

For advanced L7 features (header manipulation, retries, circuit breaking), Cilium runs **one Envoy proxy per node** instead of one per pod:

```
Traditional (Istio):
  500 pods = 500 Envoy sidecars 😰

Cilium:
  500 pods across 10 nodes = 10 Envoy instances 😊
  (One shared Envoy per node, managed by Cilium)
```

### CiliumEnvoyConfig — L7 Traffic Management:

```yaml
apiVersion: cilium.io/v2
kind: CiliumEnvoyConfig
metadata:
  name: envoy-lb-listener
spec:
  services:
    - name: backend
      namespace: default
  backendServices:
    - name: backend
      namespace: default
  resources:
    - "@type": type.googleapis.com/envoy.config.listener.v3.Listener
      name: envoy-lb-listener
      filter_chains:
        - filters:
            - name: envoy.filters.network.http_connection_manager
              typed_config:
                "@type": type.googleapis.com/envoy.extensions.filters.network.http_connection_manager.v3.HttpConnectionManager
                stat_prefix: envoy-lb-listener
                route_config:
                  virtual_hosts:
                    - name: "backend-route"
                      domains: ["*"]
                      routes:
                        - match:
                            prefix: "/api/v2"
                          route:
                            cluster: "default/backend"
                          request_headers_to_add:
                            - header:
                                key: "x-version"
                                value: "v2"
                              append_action: OVERWRITE_IF_EXISTS_OR_ADD
                        - match:
                            prefix: "/"
                          route:
                            cluster: "default/backend"
```

### Ingress with Cilium:

Cilium can also act as an **Ingress Controller**, replacing Nginx Ingress:

```yaml
# Standard Kubernetes Ingress — handled by Cilium!
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-ingress
spec:
  ingressClassName: cilium    # Use Cilium as the ingress controller
  rules:
    - host: app.example.com
      http:
        paths:
          - path: /api
            pathType: Prefix
            backend:
              service:
                name: backend
                port:
                  number: 8080
          - path: /
            pathType: Prefix
            backend:
              service:
                name: frontend
                port:
                  number: 3000
```

Enable Cilium Ingress:

```bash
helm upgrade cilium cilium/cilium \
  --namespace kube-system \
  --set ingressController.enabled=true \
  --set ingressController.default=true   # Make it the default ingress controller
```

## 🌍 Feature 6: Cluster Mesh — Multi-Cluster Connectivity

Cilium can connect multiple Kubernetes clusters together:

```
┌─── Cluster 1 (us-east) ─────┐    ┌─── Cluster 2 (eu-west) ─────┐
│                               │    │                               │
│  ┌────────┐  ┌────────┐     │    │  ┌────────┐  ┌────────┐     │
│  │ Pod A  │  │ Pod B  │     │    │  │ Pod C  │  │ Pod D  │     │
│  └────────┘  └────────┘     │    │  └────────┘  └────────┘     │
│                               │    │                               │
│  ┌─────────────────────────┐ │    │ ┌─────────────────────────┐  │
│  │    Cluster Mesh Agent    │◄├────┤►│    Cluster Mesh Agent    │  │
│  └─────────────────────────┘ │    │ └─────────────────────────┘  │
└───────────────────────────────┘    └───────────────────────────────┘

Pod A in Cluster 1 can talk to Pod C in Cluster 2 transparently!
With encryption, network policies, and full observability!
```

### Enable Cluster Mesh:

```bash
# On Cluster 1:
cilium clustermesh enable --service-type LoadBalancer

# On Cluster 2:
cilium clustermesh enable --service-type LoadBalancer

# Connect the clusters:
cilium clustermesh connect --destination-context cluster2

# Verify:
cilium clustermesh status
```

### Shared Services Across Clusters:

```yaml
# Annotate a service to be shared across clusters
apiVersion: v1
kind: Service
metadata:
  name: backend
  annotations:
    service.cilium.io/global: "true"        # Share this service across clusters
    service.cilium.io/shared: "true"        # Include local and remote endpoints
spec:
  selector:
    app: backend
  ports:
    - port: 8080
```

Now pods in Cluster 1 can reach `backend.default.svc.cluster.local` and Cilium will load-balance across backends in **both** clusters!

## 🆚 Cilium vs Traditional Service Meshes

| Feature | Cilium | Istio | Linkerd |
|---------|--------|-------|---------|
| **Sidecar Required** | ❌ No (eBPF) | ✅ Yes (Envoy) | ✅ Yes (linkerd-proxy) |
| **Resource Overhead** | Very Low | High | Medium |
| **Latency Impact** | Minimal | Noticeable | Low |
| **L3/L4 Policies** | ✅ eBPF (fast) | ✅ iptables/Envoy | ✅ iptables/proxy |
| **L7 Policies** | ✅ eBPF + per-node Envoy | ✅ Per-pod Envoy | ✅ Per-pod proxy |
| **mTLS** | ✅ WireGuard/IPsec (kernel) | ✅ Envoy mTLS | ✅ linkerd-proxy mTLS |
| **Observability** | ✅ Hubble (amazing!) | ✅ Kiali + Jaeger | ✅ Viz dashboard |
| **kube-proxy replacement** | ✅ Yes | ❌ No | ❌ No |
| **CNI** | ✅ IS the CNI | ❌ Needs separate CNI | ❌ Needs separate CNI |
| **Ingress Controller** | ✅ Built-in | ❌ Separate | ❌ Separate |
| **Gateway API** | ✅ Native support | ✅ Yes | ✅ Yes |
| **Multi-Cluster** | ✅ Cluster Mesh | ✅ Multi-cluster | ✅ Multi-cluster |
| **DNS Policies** | ✅ FQDN-based egress | ❌ No | ❌ No |
| **Kafka/gRPC aware** | ✅ Protocol-aware policies | ✅ Via Envoy | ❌ No |
| **Kernel Version Req.** | 4.19+ (5.10+ recommended) | No special req. | No special req. |
| **Learning Curve** | Medium-High | High | Low |
| **Maturity** | CNCF Graduated | CNCF Graduated | CNCF Graduated |

### When to Choose Cilium:

✅ You want the **highest performance** service mesh
✅ You want to **eliminate sidecar overhead**
✅ You need **advanced network policies** (L7, DNS-based, Kafka-aware)
✅ You want to **replace kube-proxy** for better performance
✅ You need a **CNI + Service Mesh + Ingress** all-in-one solution
✅ You care about deep **kernel-level observability**
✅ You're running at **large scale** (hundreds/thousands of pods)

### When NOT to Choose Cilium:

❌ You need the simplest possible service mesh → Use Linkerd
❌ Your kernel is too old (below 4.19) → Can't use eBPF
❌ You're on a managed Kubernetes where you can't control the CNI
❌ Your team is already deeply invested in Istio

## 🔬 Feature 7: Tetragon — Runtime Security with eBPF

Cilium has a companion project called **Tetragon** for runtime security observability:

```bash
# Install Tetragon
helm install tetragon cilium/tetragon -n kube-system
```

Tetragon can detect and alert on:
- Process execution inside pods
- File access patterns
- Network connections
- Privilege escalation attempts
- Kernel capability usage

```yaml
# Example: Alert when any pod runs a shell
apiVersion: cilium.io/v1alpha1
kind: TracingPolicy
metadata:
  name: detect-shells
spec:
  kprobes:
    - call: "__x64_sys_execve"
      syscall: true
      args:
        - index: 0
          type: "string"
      selectors:
        - matchArgs:
            - index: 0
              operator: "Equal"
              values:
                - "/bin/sh"
                - "/bin/bash"
                - "/bin/zsh"
```

This will alert you whenever someone runs a shell inside any pod — extremely useful for detecting potential intrusions!

## 🔑 Key Takeaways

1. **eBPF is the game-changer** — By running networking logic directly in the Linux kernel, Cilium avoids the overhead of sidecar proxies entirely.

2. **No sidecars needed** — Traditional service meshes add a proxy container to every pod. Cilium handles everything in the kernel, saving massive resources at scale.

3. **Cilium is more than a service mesh** — It's a CNI + Service Mesh + Ingress Controller + kube-proxy replacement + Observability platform all in one.

4. **CiliumNetworkPolicy is incredibly powerful** — L7 (HTTP path/method), DNS-based, Kafka-aware network policies that standard Kubernetes NetworkPolicy can't do.

5. **Hubble gives you X-ray vision** — See every packet, every flow, every dropped connection in real-time with beautiful dashboards.

6. **WireGuard encryption is automatic and fast** — Encrypted pod-to-pod communication at near-zero overhead, done entirely in the kernel.

7. **Cluster Mesh enables multi-cluster** — Connect multiple clusters with shared services, network policies, and observability.

8. **It requires a modern kernel** — You need Linux 4.19+ (ideally 5.10+) for all features. This is the main tradeoff.

9. **Big players trust it** — Google (GKE), AWS (EKS), and Azure (AKS) all use or support Cilium as their networking layer.
