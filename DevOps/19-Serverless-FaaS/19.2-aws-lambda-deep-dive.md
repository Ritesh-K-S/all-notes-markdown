## 🤔 What is AWS Lambda?

Think of Lambda like a **vending machine for code execution**:

1. You put your code into the machine (deploy a function)
2. Someone presses a button (an event triggers the function)
3. The machine gives the result (your function runs and returns output)
4. You pay only for the snack dispensed (you pay only for the compute time used)

No maintaining the machine. No stocking it. No keeping the lights on. AWS does all that.

### Lambda at a Glance:

| Feature | Details |
|---------|---------|
| **Supported Languages** | Python, Node.js, Java, Go, C# (.NET), Ruby, Rust, Custom Runtimes |
| **Max Execution Time** | 15 minutes (900 seconds) |
| **Max Memory** | 128 MB to 10,240 MB (10 GB) |
| **Max Deployment Package** | 50 MB (zipped), 250 MB (unzipped), 10 GB (container image) |
| **Max Concurrent Executions** | 1,000 (default, can be increased) |
| **Pricing** | $0.20 per 1M requests + $0.0000166667 per GB-second |
| **Free Tier** | 1M requests + 400,000 GB-seconds per month (forever free!) |
| **Ephemeral Storage (/tmp)** | 512 MB (default), up to 10 GB |

## 🏗️ Lambda Architecture — How It Works Under the Hood

```
┌────────────────────────────────────────────────────────────────┐
│                     AWS LAMBDA ARCHITECTURE                      │
│                                                                  │
│  ┌────────────┐     ┌───────────────┐     ┌─────────────────┐  │
│  │  EVENT      │────→│  LAMBDA       │────→│  EXECUTION      │  │
│  │  SOURCE     │     │  SERVICE      │     │  ENVIRONMENT    │  │
│  │             │     │               │     │                 │  │
│  │ - API GW    │     │ - Receives    │     │ - Micro VM      │  │
│  │ - S3        │     │   event       │     │   (Firecracker) │  │
│  │ - SQS       │     │ - Routes to   │     │ - Your runtime  │  │
│  │ - DynamoDB  │     │   function    │     │ - Your code     │  │
│  │ - Schedule  │     │ - Manages     │     │ - Dependencies  │  │
│  │ - SNS       │     │   scaling     │     │ - /tmp storage  │  │
│  └────────────┘     └───────────────┘     └─────────────────┘  │
│                                                                  │
│  Key: Lambda uses Firecracker microVMs — lightweight, fast,      │
│       and secure isolation for each function execution.          │
└────────────────────────────────────────────────────────────────┘
```

### The Lifecycle of a Lambda Invocation:

```
1. EVENT arrives (HTTP request, S3 upload, SQS message, etc.)
        ↓
2. Lambda Service receives the event
        ↓
3. Is there a WARM execution environment?
        ├── YES → Reuse it (warm start — fast! 🚀)
        └── NO  → Create new one (cold start — slower 🥶)
                    ├── Download code
                    ├── Start runtime (Python, Node.js, etc.)
                    ├── Run initialization code (outside handler)
                    └── Ready!
        ↓
4. HANDLER function is called with the event data
        ↓
5. Your code processes the event
        ↓
6. Response is returned to the caller
        ↓
7. Execution environment is kept warm for a while (~5-15 min)
   or frozen and eventually destroyed
```

## 🛠️ Creating Your First Lambda Function

### Method 1: AWS Console (Quick & Easy)

1. Go to AWS Console → Lambda → Create Function
2. Choose "Author from scratch"
3. Give it a name, choose runtime (e.g., Python 3.12)
4. Click "Create Function"
5. Write your code in the inline editor

### Method 2: AWS CLI

```bash
# Step 1: Create a Python file
cat > lambda_function.py << 'EOF'
import json

def lambda_handler(event, context):
    name = event.get('name', 'World')
    return {
        'statusCode': 200,
        'body': json.dumps(f'Hello, {name}! Welcome to Lambda!')
    }
EOF

# Step 2: Zip the code
zip function.zip lambda_function.py

# Step 3: Create the Lambda function
aws lambda create-function \
    --function-name my-first-function \
    --runtime python3.12 \
    --role arn:aws:iam::123456789012:role/lambda-execution-role \
    --handler lambda_function.lambda_handler \
    --zip-file fileb://function.zip

# Step 4: Invoke (test) the function
aws lambda invoke \
    --function-name my-first-function \
    --payload '{"name": "Ritesh"}' \
    output.json

# Check the result
cat output.json
# Output: {"statusCode": 200, "body": "\"Hello, Ritesh! Welcome to Lambda!\""}
```

### Method 3: Infrastructure as Code (Terraform)

```hcl
# main.tf

# IAM Role for Lambda
resource "aws_iam_role" "lambda_role" {
  name = "my-lambda-role"

  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Action = "sts:AssumeRole"
        Effect = "Allow"
        Principal = {
          Service = "lambda.amazonaws.com"
        }
      }
    ]
  })
}

# Attach basic execution policy (for CloudWatch Logs)
resource "aws_iam_role_policy_attachment" "lambda_basic" {
  role       = aws_iam_role.lambda_role.name
  policy_arn = "arn:aws:iam::aws:policy/service-role/AWSLambdaBasicExecutionRole"
}

# Lambda function
resource "aws_lambda_function" "my_function" {
  filename         = "function.zip"
  function_name    = "my-first-function"
  role             = aws_iam_role.lambda_role.arn
  handler          = "lambda_function.lambda_handler"
  runtime          = "python3.12"
  timeout          = 30         # seconds
  memory_size      = 256        # MB
  source_code_hash = filebase64sha256("function.zip")

  environment {
    variables = {
      ENVIRONMENT = "production"
      LOG_LEVEL   = "INFO"
    }
  }
}
```

## 📥 The Handler Function — Anatomy Explained

Every Lambda function has a **handler** — the entry point that Lambda calls when your function is triggered.

### Python Handler:

```python
def lambda_handler(event, context):
    # event  → The input data (what triggered the function)
    # context → Runtime information about the execution
    
    # Your logic here
    
    return {
        'statusCode': 200,
        'body': 'Hello!'
    }
```

### Node.js Handler:

```javascript
exports.handler = async (event, context) => {
    // event   → The input data (what triggered the function)
    // context → Runtime information about the execution
    
    // Your logic here
    
    return {
        statusCode: 200,
        body: JSON.stringify('Hello!')
    };
};
```

### Go Handler:

```go
package main

import (
    "context"
    "fmt"
    "github.com/aws/aws-lambda-go/lambda"
)

type Event struct {
    Name string `json:"name"`
}

type Response struct {
    StatusCode int    `json:"statusCode"`
    Body       string `json:"body"`
}

func handler(ctx context.Context, event Event) (Response, error) {
    message := fmt.Sprintf("Hello, %s!", event.Name)
    return Response{StatusCode: 200, Body: message}, nil
}

func main() {
    lambda.Start(handler)
}
```

### Understanding the `event` Object:

The `event` object is different depending on **what triggered** the function. Here are some examples:

**API Gateway trigger:**
```json
{
  "httpMethod": "GET",
  "path": "/users/123",
  "queryStringParameters": { "format": "json" },
  "headers": { "Authorization": "Bearer abc123" },
  "body": null
}
```

**S3 trigger (file upload):**
```json
{
  "Records": [
    {
      "s3": {
        "bucket": { "name": "my-photos-bucket" },
        "object": { "key": "uploads/photo.jpg", "size": 1024 }
      }
    }
  ]
}
```

**SQS trigger (message):**
```json
{
  "Records": [
    {
      "body": "{\"orderId\": \"12345\", \"amount\": 99.99}",
      "messageId": "msg-abc-123",
      "receiptHandle": "AQEBwJ..."
    }
  ]
}
```

### Understanding the `context` Object:

```python
def lambda_handler(event, context):
    print(context.function_name)        # "my-first-function"
    print(context.function_version)     # "$LATEST" or "1", "2", etc.
    print(context.memory_limit_in_mb)   # 256
    print(context.aws_request_id)       # Unique ID for this invocation
    
    # How much time is left before timeout (in milliseconds)
    remaining_time = context.get_remaining_time_in_millis()
    print(f"Time remaining: {remaining_time}ms")
```

## ⚡ Lambda Triggers — All Event Sources Explained

Triggers are what **cause** your Lambda function to run. Lambda supports a massive number of event sources.

### Trigger Categories:

```
┌─────────────────────────────────────────────────────────┐
│              LAMBDA TRIGGER CATEGORIES                    │
│                                                          │
│  1. SYNCHRONOUS (caller waits for response)              │
│     ├── API Gateway  ← Most common for APIs              │
│     ├── Application Load Balancer (ALB)                  │
│     ├── CloudFront (Lambda@Edge)                         │
│     ├── Cognito                                          │
│     └── Alexa Skills                                     │
│                                                          │
│  2. ASYNCHRONOUS (fire and forget)                       │
│     ├── S3 Events                                        │
│     ├── SNS                                              │
│     ├── EventBridge                                      │
│     ├── CloudWatch Events                                │
│     ├── CodeCommit                                       │
│     ├── CodePipeline                                     │
│     └── IoT Rules                                        │
│                                                          │
│  3. POLL-BASED (Lambda polls the source)                 │
│     ├── SQS                                              │
│     ├── DynamoDB Streams                                 │

└─────────────────────────────────────────────────────────┘
```

### Trigger 1: API Gateway (Most Popular)

Makes your Lambda accessible via HTTP/REST/WebSocket.

```
Client → HTTPS Request → API Gateway → Lambda → Response → Client
```

**Setting up with AWS CLI:**
```bash
# Create a REST API
aws apigateway create-rest-api \
    --name "MyServerlessAPI" \
    --description "My first serverless API"
```

**Setting up with Terraform:**
```hcl
# API Gateway
resource "aws_apigatewayv2_api" "api" {
  name          = "my-serverless-api"
  protocol_type = "HTTP"
}

# Integration with Lambda
resource "aws_apigatewayv2_integration" "lambda_integration" {
  api_id             = aws_apigatewayv2_api.api.id
  integration_type   = "AWS_PROXY"
  integration_uri    = aws_lambda_function.my_function.invoke_arn
  payload_format_version = "2.0"
}

# Route
resource "aws_apigatewayv2_route" "route" {
  api_id    = aws_apigatewayv2_api.api.id
  route_key = "GET /hello"
  target    = "integrations/${aws_apigatewayv2_integration.lambda_integration.id}"
}

# Stage (deployment)
resource "aws_apigatewayv2_stage" "stage" {
  api_id      = aws_apigatewayv2_api.api.id
  name        = "prod"
  auto_deploy = true
}

# Permission for API Gateway to invoke Lambda
resource "aws_lambda_permission" "api_gw" {
  statement_id  = "AllowAPIGateway"
  action        = "lambda:InvokeFunction"
  function_name = aws_lambda_function.my_function.function_name
  principal     = "apigateway.amazonaws.com"
  source_arn    = "${aws_apigatewayv2_api.api.execution_arn}/*/*"
}
```

### Trigger 2: S3 (File Events)

Lambda runs when files are uploaded, deleted, or modified in an S3 bucket.

```python
import boto3
import json

s3_client = boto3.client('s3')

def lambda_handler(event, context):
    # Get file info from the S3 event
    for record in event['Records']:
        bucket = record['s3']['bucket']['name']
        key = record['s3']['object']['key']
        size = record['s3']['object']['size']
        
        print(f"New file uploaded!")
        print(f"  Bucket: {bucket}")
        print(f"  File: {key}")
        print(f"  Size: {size} bytes")
        
        # Example: Read the file
        response = s3_client.get_object(Bucket=bucket, Key=key)
        content = response['Body'].read().decode('utf-8')
        
        # Process the file content...
        print(f"  Content preview: {content[:100]}")
    
    return {'statusCode': 200, 'body': 'File processed successfully!'}
```

**Terraform setup for S3 trigger:**
```hcl
# Allow S3 to invoke Lambda
resource "aws_lambda_permission" "s3_trigger" {
  statement_id  = "AllowS3Invoke"
  action        = "lambda:InvokeFunction"
  function_name = aws_lambda_function.my_function.function_name
  principal     = "s3.amazonaws.com"
  source_arn    = aws_s3_bucket.my_bucket.arn
}

# S3 bucket notification → triggers Lambda on file upload
resource "aws_s3_bucket_notification" "bucket_notification" {
  bucket = aws_s3_bucket.my_bucket.id

  lambda_function {
    lambda_function_arn = aws_lambda_function.my_function.arn
    events              = ["s3:ObjectCreated:*"]      # Trigger on any file creation
    filter_prefix       = "uploads/"                   # Only files in uploads/ folder
    filter_suffix       = ".jpg"                       # Only .jpg files
  }
}
```

### Trigger 3: SQS (Message Queue)

Lambda processes messages from an SQS queue.

```python
import json

def lambda_handler(event, context):
    for record in event['Records']:
        # Parse the message body
        message = json.loads(record['body'])
        
        print(f"Processing order: {message['orderId']}")
        print(f"  Amount: ${message['amount']}")
        print(f"  Customer: {message['customer']}")
        
        # Process the order...
        # If this function fails, the message goes back to the queue
        # and will be retried automatically!
    
    return {'statusCode': 200, 'body': 'Messages processed!'}
```

### Trigger 4: EventBridge (Scheduled/Cron Jobs)

Run Lambda on a schedule — like cron but serverless.

```hcl
# EventBridge rule — runs every day at 9 AM UTC
resource "aws_cloudwatch_event_rule" "daily_cleanup" {
  name                = "daily-cleanup"
  description         = "Triggers Lambda every day at 9 AM UTC"
  schedule_expression = "cron(0 9 * * ? *)"
}

# Connect the rule to Lambda
resource "aws_cloudwatch_event_target" "lambda_target" {
  rule      = aws_cloudwatch_event_rule.daily_cleanup.name
  target_id = "cleanup-function"
  arn       = aws_lambda_function.my_function.arn
}

# Permission
resource "aws_lambda_permission" "eventbridge" {
  statement_id  = "AllowEventBridge"
  action        = "lambda:InvokeFunction"
  function_name = aws_lambda_function.my_function.function_name
  principal     = "events.amazonaws.com"
  source_arn    = aws_cloudwatch_event_rule.daily_cleanup.arn
}
```

**Schedule expression examples:**
```
rate(5 minutes)          → Every 5 minutes
rate(1 hour)             → Every hour
rate(1 day)              → Every day
cron(0 9 * * ? *)        → Every day at 9:00 AM UTC
cron(0 12 ? * MON-FRI *) → Weekdays at noon UTC
cron(0 0 1 * ? *)        → First day of every month at midnight
```

### Trigger 5: DynamoDB Streams

Lambda runs when data changes in a DynamoDB table.

```python
def lambda_handler(event, context):
    for record in event['Records']:
        event_name = record['eventName']  # INSERT, MODIFY, or REMOVE
        
        if event_name == 'INSERT':
            new_item = record['dynamodb']['NewImage']
            print(f"New user created: {new_item['username']['S']}")
            # Send welcome email, update analytics, etc.
        
        elif event_name == 'MODIFY':
            old_item = record['dynamodb']['OldImage']
            new_item = record['dynamodb']['NewImage']
            print(f"User updated: {new_item['username']['S']}")
        
        elif event_name == 'REMOVE':
            old_item = record['dynamodb']['OldImage']
            print(f"User deleted: {old_item['username']['S']}")
    
    return {'statusCode': 200}
```

## 📦 Lambda Layers — Share Code Across Functions

### The Problem:

Imagine you have 10 Lambda functions that all use the same libraries (e.g., `requests`, `boto3`, `Pillow`). Without layers, you'd have to include these libraries in **every single function's deployment package**. That's:

- Wasteful (same code duplicated 10 times)
- Hard to update (change a library version → redeploy all 10 functions)
- Slower deployments

### The Solution: Lambda Layers

> A **Lambda Layer** is a ZIP archive containing libraries, custom runtimes, or other dependencies that you can **share across multiple functions**.

```
WITHOUT Layers:                          WITH Layers:
┌──────────────┐ ┌──────────────┐       ┌──────────────┐ ┌──────────────┐
│ Function A   │ │ Function B   │       │ Function A   │ │ Function B   │
│ ┌──────────┐ │ │ ┌──────────┐ │       │ (just code)  │ │ (just code)  │
│ │ Your Code│ │ │ │ Your Code│ │       └──────┬───────┘ └──────┬───────┘
│ ├──────────┤ │ │ ├──────────┤ │              │                │
│ │ requests │ │ │ │ requests │ │              ▼                ▼
│ │ Pillow   │ │ │ │ Pillow   │ │       ┌──────────────────────────────┐
│ │ boto3    │ │ │ │ boto3    │ │       │        SHARED LAYER           │
│ └──────────┘ │ │ └──────────┘ │       │   requests, Pillow, boto3    │
└──────────────┘ └──────────────┘       └──────────────────────────────┘
  50MB each        50MB each             2MB each       One 48MB layer
  = 100MB total                          = 52MB total — saves space!
```

### How Layers Work:

- You can attach **up to 5 layers** to a single function
- Layers are extracted to `/opt` in the execution environment
- Total unzipped size (function + all layers) must be ≤ 250 MB
- Layers can be **versioned** independently
- Layers can be **shared** across accounts and even made public

### Creating a Layer — Step by Step:

```bash
# Step 1: Create directory structure
# For Python, libraries must be in python/ folder
mkdir -p my-layer/python

# Step 2: Install libraries into the layer directory
pip install requests Pillow -t my-layer/python/

# Step 3: Zip the layer
cd my-layer
zip -r ../my-layer.zip .
cd ..

# Step 4: Publish the layer to AWS
aws lambda publish-layer-version \
    --layer-name my-shared-libraries \
    --description "Common libraries: requests, Pillow" \
    --zip-file fileb://my-layer.zip \
    --compatible-runtimes python3.11 python3.12

# Output includes LayerVersionArn like:
# arn:aws:lambda:us-east-1:123456789012:layer:my-shared-libraries:1

# Step 5: Attach the layer to a function
aws lambda update-function-configuration \
    --function-name my-function \
    --layers arn:aws:lambda:us-east-1:123456789012:layer:my-shared-libraries:1
```

### Layer Directory Structure by Runtime:

| Runtime | Layer Path |
|---------|-----------|
| Python | `python/` or `python/lib/python3.x/site-packages/` |
| Node.js | `nodejs/node_modules/` |
| Java | `java/lib/` |
| Ruby | `ruby/gems/3.2.0/` |
| All Runtimes | `bin/` (executables), `lib/` (shared libraries) |

### Terraform — Layer Example:

```hcl
# Create the layer
resource "aws_lambda_layer_version" "shared_libs" {
  filename            = "my-layer.zip"
  layer_name          = "shared-libraries"
  description         = "Common Python libraries"
  compatible_runtimes = ["python3.11", "python3.12"]
  source_code_hash    = filebase64sha256("my-layer.zip")
}

# Use the layer in a function
resource "aws_lambda_function" "my_function" {
  filename         = "function.zip"
  function_name    = "my-function"
  role             = aws_iam_role.lambda_role.arn
  handler          = "lambda_function.lambda_handler"
  runtime          = "python3.12"
  source_code_hash = filebase64sha256("function.zip")
  
  # Attach the layer! (up to 5 layers)
  layers = [
    aws_lambda_layer_version.shared_libs.arn
  ]
}
```

## 🔢 Lambda Versions — Immutable Snapshots of Your Code

### The Problem:

When you update your Lambda function code, the `$LATEST` version changes immediately. But what if:
- Your new code has a bug?
- You want to test the new version before sending all traffic to it?
- You need to roll back to the previous working version?

### The Solution: Versions

> A **Version** is an **immutable snapshot** of your function's code AND configuration at a specific point in time.

```
$LATEST (mutable — always points to current code)
  │
  ├── Version 1 (immutable snapshot) — "First release"
  ├── Version 2 (immutable snapshot) — "Added validation"
  ├── Version 3 (immutable snapshot) — "Fixed bug #123"
  └── Version 4 (immutable snapshot) — "Performance improvement"
```

### Key Facts About Versions:

| Property | Value |
|----------|-------|
| **$LATEST** | Special version — always the most recent code. Mutable. |
| **Published versions** | Numbered (1, 2, 3, ...). Immutable after creation. |
| **What's frozen** | Code, runtime, memory, timeout, environment variables, layers |
| **What's NOT frozen** | Concurrency settings, triggers (can be changed) |
| **ARN format** | `arn:aws:lambda:region:account:function:name:VERSION` |

### Publishing a Version:

```bash
# Publish a new version of your function
aws lambda publish-version \
    --function-name my-function \
    --description "v2 - added input validation"

# Output:
# {
#     "Version": "2",
#     "FunctionArn": "arn:aws:lambda:us-east-1:123456789012:function:my-function:2",
#     "Description": "v2 - added input validation"
# }

# Invoke a specific version
aws lambda invoke \
    --function-name my-function \
    --qualifier 2 \
    output.json
```

### Terraform — Publishing Versions:

```hcl
resource "aws_lambda_function" "my_function" {
  filename         = "function.zip"
  function_name    = "my-function"
  role             = aws_iam_role.lambda_role.arn
  handler          = "lambda_function.lambda_handler"
  runtime          = "python3.12"
  publish          = true    # ← This publishes a new version on every change!
  source_code_hash = filebase64sha256("function.zip")
}

# Output the version number
output "function_version" {
  value = aws_lambda_function.my_function.version
}
```

## 🏷️ Lambda Aliases — Named Pointers to Versions

### The Problem:

Versions are great, but referring to functions by number (version 1, 2, 3) is:
- Not meaningful (what does "version 7" mean?)
- Hard to manage in triggers (you'd need to update all triggers when deploying a new version)

### The Solution: Aliases

> An **Alias** is a **named pointer** (like a friendly bookmark) that points to a specific version. You can move the alias to point to a different version at any time.

```
                        Aliases              Versions
                      ┌─────────┐          ┌──────────┐
                      │  PROD   │─────────→│ Version 5│  ← Production traffic
                      ├─────────┤          ├──────────┤
                      │  STAGE  │─────────→│ Version 6│  ← Staging/testing
                      ├─────────┤          ├──────────┤
                      │  DEV    │─────────→│ $LATEST  │  ← Development
                      └─────────┘          ├──────────┤
                                           │ Version 4│
                                           │ Version 3│
                                           │ Version 2│
                                           │ Version 1│
                                           └──────────┘
```

### Why Aliases Are Powerful:

| Benefit | Explanation |
|---------|-------------|
| **Stable endpoint** | API Gateway points to the `PROD` alias. When you deploy, just move the alias — no API Gateway changes needed! |
| **Easy rollback** | New version has a bug? Move `PROD` alias back to the previous version. Instant rollback! |
| **Environment separation** | `DEV`, `STAGE`, `PROD` aliases for different environments |
| **Traffic shifting** | Send 10% of traffic to new version, 90% to old (canary deployment!) |

### Creating and Managing Aliases:

```bash
# Create a PROD alias pointing to version 5
aws lambda create-alias \
    --function-name my-function \
    --name PROD \
    --function-version 5 \
    --description "Production traffic"

# Create a STAGING alias pointing to version 6
aws lambda create-alias \
    --function-name my-function \
    --name STAGING \
    --function-version 6

# Deploy new version and move PROD alias (deployment!)
aws lambda publish-version --function-name my-function
# Returns version 7

# Update PROD to point to version 7
aws lambda update-alias \
    --function-name my-function \
    --name PROD \
    --function-version 7

# Invoke using alias (not version number)
aws lambda invoke \
    --function-name my-function \
    --qualifier PROD \
    output.json
```

### Traffic Shifting with Aliases (Canary Deployments):

This is one of the coolest features! You can split traffic between two versions:

```bash
# Send 90% of traffic to version 5, 10% to version 7 (canary)
aws lambda update-alias \
    --function-name my-function \
    --name PROD \
    --function-version 5 \
    --routing-config '{"AdditionalVersionWeights": {"7": 0.1}}'

# Monitor errors and latency for version 7...
# Everything looks good? Send 100% to version 7

aws lambda update-alias \
    --function-name my-function \
    --name PROD \
    --function-version 7 \
    --routing-config '{}'

# Something broke? Roll back to version 5
aws lambda update-alias \
    --function-name my-function \
    --name PROD \
    --function-version 5
```

```
CANARY DEPLOYMENT FLOW:

  Step 1: 90% → Version 5 (old)    Step 2: 50/50 split        Step 3: 100% → Version 7 (new)
          10% → Version 7 (new)

  ████████████░░   (monitor)    ██████████████   (all good!)   ████████████████████████████
  90% v5  10% v7                50% v5  50% v7                 100% v7 ← fully deployed!
```

## 🔧 Lambda Environment Variables

Environment variables let you configure your function **without changing code**. Perfect for things like database URLs, API keys, feature flags, etc.

```python
import os
import boto3

# Read environment variables
DB_TABLE = os.environ['DB_TABLE_NAME']
ENVIRONMENT = os.environ.get('ENVIRONMENT', 'dev')
LOG_LEVEL = os.environ.get('LOG_LEVEL', 'INFO')

dynamodb = boto3.resource('dynamodb')
table = dynamodb.Table(DB_TABLE)

def lambda_handler(event, context):
    print(f"Running in {ENVIRONMENT} environment")
    print(f"Using table: {DB_TABLE}")
    
    # Your logic here...
    return {'statusCode': 200, 'body': f'Hello from {ENVIRONMENT}!'}
```

```bash
# Set environment variables via CLI
aws lambda update-function-configuration \
    --function-name my-function \
    --environment "Variables={DB_TABLE_NAME=users-prod,ENVIRONMENT=production,LOG_LEVEL=WARNING}"
```

> ⚠️ **Security Note:** Never put plain-text secrets (passwords, API keys) in environment variables! Use **AWS Secrets Manager** or **SSM Parameter Store** and fetch them at runtime, or use **encrypted environment variables** with KMS.

```python
# ✅ GOOD — Fetch secrets from Secrets Manager
import boto3
import json
import os

secrets_client = boto3.client('secretsmanager')

def get_secret(secret_name):
    response = secrets_client.get_secret_value(SecretId=secret_name)
    return json.loads(response['SecretString'])

# Fetch once during cold start
db_credentials = get_secret(os.environ['DB_SECRET_NAME'])

def lambda_handler(event, context):
    # Use db_credentials['username'] and db_credentials['password']
    pass
```

## 📊 Lambda Concurrency — How Scaling Works

### How Lambda Scales:

```
Request 1 arrives → Lambda creates Execution Env 1 → processes request
Request 2 arrives (while 1 is busy) → Lambda creates Execution Env 2 → processes request
Request 3 arrives (while 1 & 2 are busy) → Lambda creates Execution Env 3 → processes request
Request 4 arrives (Env 1 is now free) → Reuses Execution Env 1 → processes request (warm start!)
```

Each concurrent request gets its own execution environment. Lambda **auto-scales horizontally**.

### Concurrency Types:

| Type | What It Does | Default |
|------|-------------|---------|
| **Unreserved Concurrency** | Shared pool across all functions in your account | 1,000 total |
| **Reserved Concurrency** | Guarantees a specific number of concurrent executions for a function | Not set |
| **Provisioned Concurrency** | Pre-warms a specific number of execution environments (eliminates cold starts) | Not set |

### Reserved Concurrency:

```bash
# Reserve 100 concurrent executions for this function
# This means: max 100 simultaneous executions, AND these 100 are guaranteed
# (other functions can't steal them)
aws lambda put-function-concurrency \
    --function-name my-critical-function \
    --reserved-concurrent-executions 100
```

```
Account limit: 1,000 concurrent executions

Without reservation:
  All functions share 1,000 → one noisy function could starve others!

With reservation:
  my-critical-function: 100 reserved (guaranteed, capped at 100)
  all-other-functions: share remaining 900
```

### Provisioned Concurrency (Eliminate Cold Starts):

```bash
# Pre-warm 50 execution environments for the PROD alias
aws lambda put-provisioned-concurrency-config \
    --function-name my-function \
    --qualifier PROD \
    --provisioned-concurrent-executions 50
```

```
Without Provisioned Concurrency:
  Cold request → 🥶 cold start (200ms-2s delay) → then execute

With Provisioned Concurrency (50 instances):
  First 50 requests → 🔥 instant! (no cold start, environments are pre-warmed)
  Request 51+ → normal scaling (may get cold start)
```

> 💡 **Cost Warning:** Provisioned Concurrency costs money even when not processing requests (you're paying to keep environments warm). Use it only for latency-sensitive workloads.

## 📝 Lambda Logging & Monitoring

### CloudWatch Logs:

Every Lambda function automatically logs to CloudWatch. Just use `print()` or your language's logging library.

```python
import logging

# Set up structured logging
logger = logging.getLogger()
logger.setLevel(logging.INFO)

def lambda_handler(event, context):
    logger.info("Function started", extra={
        "request_id": context.aws_request_id,
        "function_name": context.function_name
    })
    
    try:
        # Your logic
        result = process_event(event)
        logger.info(f"Successfully processed event: {result}")
        return {'statusCode': 200, 'body': 'Success'}
    
    except Exception as e:
        logger.error(f"Error processing event: {str(e)}", exc_info=True)
        return {'statusCode': 500, 'body': 'Internal Server Error'}
```

### Key CloudWatch Metrics for Lambda:

| Metric | What It Measures |
|--------|-----------------|
| **Invocations** | Number of times your function was called |
| **Duration** | How long each invocation took |
| **Errors** | Number of invocations that resulted in errors |
| **Throttles** | Number of invocations that were throttled (hit concurrency limit) |
| **ConcurrentExecutions** | Number of simultaneous executions |
| **IteratorAge** | For stream-based triggers — how far behind real-time you are |

## 🚀 Lambda Best Practices

### 1. Keep Functions Small & Focused

```python
# ❌ BAD — One function doing everything
def lambda_handler(event, context):
    validate_input(event)
    process_payment(event)
    update_inventory(event)
    send_email(event)
    generate_report(event)
    # If any step fails, everything fails!

# ✅ GOOD — Each function does ONE thing
# Function 1: validate-order
# Function 2: process-payment  
# Function 3: update-inventory
# Function 4: send-notification
# Orchestrate with Step Functions or EventBridge
```

### 2. Initialize Outside the Handler

```python
# ✅ GOOD — Reuse connections across warm invocations
import boto3

# These run ONCE during cold start, then reuse
dynamodb = boto3.resource('dynamodb')
table = dynamodb.Table('my-table')
s3_client = boto3.client('s3')

def lambda_handler(event, context):
    # Just use the pre-initialized clients
    response = table.get_item(Key={'id': event['id']})
    return response
```

### 3. Minimize Deployment Package Size

```bash
# Only include what you need
pip install requests -t package/ --no-cache-dir

# Exclude unnecessary files
cd package
zip -r ../function.zip . -x "*.pyc" "*.pyo" "__pycache__/*" "*.dist-info/*" "tests/*"
cd ..
zip function.zip lambda_function.py
```

### 4. Use Environment Variables for Configuration

```python
import os

# ✅ GOOD — Easy to change without redeploying code
TABLE_NAME = os.environ['TABLE_NAME']
API_URL = os.environ['API_URL']
MAX_RETRIES = int(os.environ.get('MAX_RETRIES', '3'))
```

### 5. Handle Errors Gracefully

```python
import json
import logging

logger = logging.getLogger()

def lambda_handler(event, context):
    try:
        # Validate input
        if 'user_id' not in event:
            return {
                'statusCode': 400,
                'body': json.dumps({'error': 'user_id is required'})
            }
        
        result = process_user(event['user_id'])
        
        return {
            'statusCode': 200,
            'body': json.dumps(result)
        }
    
    except UserNotFoundException as e:
        return {
            'statusCode': 404,
            'body': json.dumps({'error': f'User not found: {str(e)}'})
        }
    
    except Exception as e:
        logger.error(f"Unexpected error: {str(e)}", exc_info=True)
        return {
            'statusCode': 500,
            'body': json.dumps({'error': 'Internal server error'})
        }
```

### 6. Set Appropriate Timeout and Memory

```
Rule of Thumb:
  - Timeout: Set to 2-3x your expected execution time
  - Memory: Start at 256MB, increase if needed
  
  More memory = More CPU power (they scale together in Lambda)
  Sometimes increasing memory makes your function FASTER and CHEAPER!
  
  Example:
    128MB, 3 seconds  → 3,000ms × 0.125 GB = 375 GB-ms   ($X)
    256MB, 1.5 seconds → 1,500ms × 0.25 GB = 375 GB-ms   ($X)  ← Same cost, faster!
    512MB, 0.8 seconds → 800ms × 0.5 GB = 400 GB-ms       (~$X) ← Slightly more, way faster!
```
