## 🤔 What is Azure Functions?

If AWS Lambda is the Uber of serverless (as we said in 19.1), then Azure Functions is **Lyft** — same concept, different ride, with its own set of unique features and advantages.

Azure Functions lets you:
1. Write small pieces of code (functions)
2. Define what triggers them (HTTP request, timer, queue message, etc.)
3. Deploy them — Azure handles all the servers, scaling, and maintenance
4. Pay only for what you use (or choose an always-on plan)

### Azure Functions vs AWS Lambda — Quick Comparison:

| Feature | Azure Functions | AWS Lambda |
|---------|----------------|------------|
| **Provider** | Microsoft Azure | Amazon Web Services |
| **Supported Languages** | C#, JavaScript, TypeScript, Python, Java, PowerShell, Go, Rust, Custom | Python, Node.js, Java, Go, C#, Ruby, Rust, Custom |
| **Max Execution Time** | 5 min (Consumption), 230 sec (HTTP), unlimited (Premium/Dedicated) | 15 minutes |
| **Stateful Functions** | Yes! (Durable Functions) | No (need Step Functions separately) |
| **Local Development** | Excellent (Azure Functions Core Tools) | Okay (SAM CLI) |
| **Unique Feature** | Durable Functions, Bindings | Lambda@Edge, Layers |
| **Free Tier** | 1M executions + 400,000 GB-seconds/month | 1M requests + 400,000 GB-seconds/month |

### What Makes Azure Functions Special:

1. **Input/Output Bindings** — Declarative connections to services (no SDK code needed!)
2. **Durable Functions** — Built-in support for stateful, long-running workflows
3. **Multiple Hosting Plans** — Consumption (pay-per-use), Premium (always warm), Dedicated (App Service)
4. **Excellent Local Development** — Run and debug functions on your machine before deploying
5. **Deep Azure Integration** — Works seamlessly with Azure services (Cosmos DB, Service Bus, Event Grid, etc.)

## 🏗️ Azure Functions Architecture

```
┌──────────────────────────────────────────────────────────────┐
│                  AZURE FUNCTIONS ARCHITECTURE                  │
│                                                                │
│  ┌─────────────┐    ┌──────────────────┐    ┌──────────────┐ │
│  │  TRIGGER     │───→│  AZURE FUNCTIONS │───→│  OUTPUT       │ │
│  │              │    │  RUNTIME         │    │  BINDING      │ │
│  │ - HTTP       │    │                  │    │               │ │
│  │ - Timer      │    │  ┌────────────┐  │    │ - Cosmos DB   │ │
│  │ - Queue      │    │  │ Your Code  │  │    │ - Blob Storage│ │
│  │ - Blob       │    │  │ (Function) │  │    │ - Queue       │ │
│  │ - Event Hub  │    │  └────────────┘  │    │ - HTTP Resp   │ │
│  │ - Cosmos DB  │    │                  │    │ - Event Hub   │ │
│  │ - Service Bus│    │  Input Bindings  │    │ - SendGrid    │ │
│  │ - Event Grid │    │  (auto-fetches   │    │ - SignalR     │ │
│  └─────────────┘    │   data for you)  │    └──────────────┘ │
│                      └──────────────────┘                      │
│                                                                │
│  KEY CONCEPT: Triggers START your function.                    │
│               Bindings CONNECT your function to services       │
│               (both input and output) WITHOUT boilerplate code.│
└──────────────────────────────────────────────────────────────┘
```

### Core Components:

| Component | What It Is |
|-----------|-----------|
| **Function App** | A container that groups one or more functions. Like a "project" or "microservice". Shares settings, hosting plan, and deployment. |
| **Function** | A single piece of code that runs in response to a trigger. |
| **Trigger** | The event that causes a function to run. **Every function has exactly ONE trigger.** |
| **Input Binding** | Declarative connection to read data from a service (optional, 0 or more). |
| **Output Binding** | Declarative connection to write data to a service (optional, 0 or more). |
| **host.json** | Global configuration for the entire Function App (logging, extensions, etc.). |
| **function.json** | Configuration for a specific function (trigger, bindings). Auto-generated in C#. |

## 💻 Hosting Plans — Choose Your Ride

Azure Functions gives you **three hosting options** (unlike Lambda which is consumption-only):

| Plan | How It Works | Scaling | Cost | Best For |
|------|-------------|---------|------|----------|
| **Consumption** | Runs only when triggered. Scales automatically. | 0 to 200 instances | Pay-per-execution ($0 when idle) | Variable workloads, cost optimization |
| **Premium (Elastic Premium)** | Always-warm instances + auto-scaling. No cold starts. | 1 to 100 instances (pre-warmed) | Per-second billing (always paying for min instances) | Production APIs needing low latency |
| **Dedicated (App Service)** | Runs on your App Service plan (regular VMs). | Manual or auto-scale | Monthly (always running) | Existing App Service plan, predictable workloads |
| **Container Apps** | Run functions in containers on Azure Container Apps. | 0 to many | Per-second billing | Containerized workloads, microservices |

### Which Plan Should You Choose?

```
Start here:
  │
  ├── Do you need $0 idle cost? → Consumption Plan ✅
  │
  ├── Do you need NO cold starts? → Premium Plan ✅
  │
  ├── Do you already have an App Service Plan? → Dedicated Plan ✅
  │
  └── Do you need custom containers or KEDA-based scaling? → Container Apps ✅
```

## 📝 Function Code — All Languages

### Python (v2 Programming Model — Recommended):

```python
# function_app.py — This is the new Python v2 model (decorator-based)

import azure.functions as func
import json
import logging

app = func.FunctionApp()

# HTTP Trigger — Simple API endpoint
@app.route(route="hello", auth_level=func.AuthLevel.ANONYMOUS)
def hello_function(req: func.HttpRequest) -> func.HttpResponse:
    logging.info('Python HTTP trigger function processed a request.')

    # Get 'name' from query string or request body
    name = req.params.get('name')
    if not name:
        try:
            req_body = req.get_json()
            name = req_body.get('name')
        except ValueError:
            pass

    if name:
        return func.HttpResponse(
            json.dumps({"message": f"Hello, {name}! Welcome to Azure Functions!"}),
            mimetype="application/json",
            status_code=200
        )
    else:
        return func.HttpResponse(
            json.dumps({"error": "Please pass a name in the query string or request body"}),
            mimetype="application/json",
            status_code=400
        )


# Timer Trigger — Runs every 5 minutes (like a cron job)
@app.timer_trigger(schedule="0 */5 * * * *", arg_name="myTimer",
                   run_on_startup=False)
def timer_function(myTimer: func.TimerRequest) -> None:
    logging.info('Timer trigger function ran!')
    
    if myTimer.past_due:
        logging.warning('The timer is past due!')
    
    # Clean up expired sessions, send reports, etc.
    logging.info('Cleanup completed successfully.')


# Queue Trigger — Processes messages from Azure Queue Storage
@app.queue_trigger(arg_name="msg", queue_name="order-queue",
                   connection="AzureWebJobsStorage")
def queue_function(msg: func.QueueMessage) -> None:
    logging.info(f'Processing queue message: {msg.get_body().decode("utf-8")}')
    
    order = json.loads(msg.get_body().decode('utf-8'))
    logging.info(f'Order ID: {order["orderId"]}, Amount: ${order["amount"]}')
    
    # Process the order...


# Blob Trigger — Runs when a file is uploaded to Blob Storage
@app.blob_trigger(arg_name="myblob", path="uploads/{name}",
                  connection="AzureWebJobsStorage")
def blob_function(myblob: func.InputStream) -> None:
    logging.info(f'Blob trigger processed blob: {myblob.name}')
    logging.info(f'  Size: {myblob.length} bytes')
    
    # Read blob content
    content = myblob.read()
    logging.info(f'  Content preview: {content[:100]}')
```

### C# (Isolated Worker Model — Recommended):

```csharp
// HelloFunction.cs
using Microsoft.Azure.Functions.Worker;
using Microsoft.Azure.Functions.Worker.Http;
using Microsoft.Extensions.Logging;
using System.Net;

public class HelloFunction
{
    private readonly ILogger _logger;

    public HelloFunction(ILoggerFactory loggerFactory)
    {
        _logger = loggerFactory.CreateLogger<HelloFunction>();
    }

    // HTTP Trigger
    [Function("Hello")]
    public HttpResponseData Run(
        [HttpTrigger(AuthorizationLevel.Anonymous, "get", "post")] HttpRequestData req)
    {
        _logger.LogInformation("C# HTTP trigger function processed a request.");

        var name = req.Query["name"] ?? "World";

        var response = req.CreateResponse(HttpStatusCode.OK);
        response.Headers.Add("Content-Type", "application/json");
        response.WriteString($"{{\"message\": \"Hello, {name}!\"}}");

        return response;
    }

    // Timer Trigger
    [Function("DailyCleanup")]
    public void TimerRun([TimerTrigger("0 0 9 * * *")] TimerInfo myTimer)
    {
        _logger.LogInformation($"Timer triggered at: {DateTime.Now}");
        if (myTimer.ScheduleStatus is not null)
        {
            _logger.LogInformation($"Next timer schedule: {myTimer.ScheduleStatus.Next}");
        }
    }
}
```

## 🔗 Triggers and Bindings — The Magic of Azure Functions

This is Azure Functions' **killer feature** compared to other serverless platforms. Bindings let you connect to services **declaratively** — without writing SDK boilerplate code.

### What Are Bindings?

Think of bindings as **automatic plumbing**:

```
WITHOUT Bindings (like AWS Lambda):
  1. Your code receives event
  2. You import the SDK (boto3, etc.)
  3. You create a client
  4. You make API calls to read/write data
  5. You handle errors, retries, connections
  6. Lots of boilerplate code!

WITH Bindings (Azure Functions):
  1. You declare: "I want data from Cosmos DB" in configuration
  2. Azure Functions automatically fetches it and passes it to your function
  3. You declare: "I want to write to Queue Storage" as output
  4. Azure Functions automatically sends it when your function returns
  5. Zero boilerplate! 🎉
```

|---------|---------|---------------|----------------|
| **HTTP** | ✅ | — | ✅ (response) |
| **Timer** | ✅ | — | — |

| **Queue Storage** | ✅ | — | ✅ |

### Example: Combining Triggers and Bindings

**Scenario:** When a user uploads an image to Blob Storage, read their profile from Cosmos DB, and send a notification to a Queue.

```python
# Python v2 model — Look how clean this is!

import azure.functions as func
import json
import logging

app = func.FunctionApp()

@app.blob_trigger(arg_name="inputBlob", path="uploads/{userId}/{filename}",
                  connection="AzureWebJobsStorage")
@app.cosmos_db_input(arg_name="userProfile",
                     database_name="mydb",
                     container_name="users",
                     id="{userId}",                    # Auto-extracted from blob path!
                     partition_key="{userId}",
                     connection="CosmosDBConnection")
@app.queue_output(arg_name="notification",
                  queue_name="notifications",
                  connection="AzureWebJobsStorage")
def process_upload(inputBlob: func.InputStream,
                   userProfile: func.DocumentList,
                   notification: func.Out[str]) -> None:
    
    logging.info(f"New file uploaded: {inputBlob.name}")
    logging.info(f"File size: {inputBlob.length} bytes")
    
    # userProfile is automatically fetched from Cosmos DB!
    # No SDK code, no client creation, no API calls!
    if userProfile:
        user = userProfile[0]
        user_name = user['name']
        user_email = user['email']
        logging.info(f"Uploaded by: {user_name} ({user_email})")
    
    # Write to notification queue — just set the output binding!
    notification.set(json.dumps({
        "userId": user['id'],
        "message": f"File {inputBlob.name} uploaded successfully!",
        "email": user_email
    }))
    # That's it! The message is automatically sent to the queue!
```

**Without bindings**, the same function would require:
- Import `azure.cosmos` SDK
- Create a Cosmos DB client
- Handle connection strings
- Make a query to fetch the user
- Import `azure.storage.queue` SDK
- Create a queue client
- Send the message manually
- Handle errors and retries

**With bindings**, all of that is done declaratively in the decorators. Your function only contains **business logic**. This is the beauty of Azure Functions!

## ⏰ Timer Trigger — CRON Expressions in Azure Functions

Azure Functions uses **NCrontab expressions** (6 fields instead of Unix's 5 fields):

```
Timer Expression Format:
{second} {minute} {hour} {day} {month} {day-of-week}

Field:          Allowed Values:
second          0-59
minute          0-59
hour            0-23
day             1-31
month           1-12
day-of-week     0-6 (0 = Sunday)
```

### Common Timer Expressions:

| Expression | Meaning |
|-----------|---------|
| `0 */5 * * * *` | Every 5 minutes |
| `0 0 * * * *` | Every hour (at minute 0) |
| `0 0 */2 * * *` | Every 2 hours |
| `0 30 9 * * *` | Every day at 9:30 AM |
| `0 0 9 * * 1-5` | Weekdays at 9:00 AM |
| `0 0 0 * * *` | Every day at midnight |
| `0 0 0 1 * *` | First day of every month at midnight |
| `0 0 0 * * 0` | Every Sunday at midnight |

### Timer Trigger Example with Error Handling:

```python
import azure.functions as func
import logging
import datetime

app = func.FunctionApp()

@app.timer_trigger(schedule="0 0 */6 * * *", arg_name="myTimer",
                   run_on_startup=False)
def cleanup_expired_data(myTimer: func.TimerRequest) -> None:
    """Runs every 6 hours to clean up expired records."""
    
    utc_timestamp = datetime.datetime.utcnow().isoformat()
    
    if myTimer.past_due:
        logging.warning('The timer is past due! This execution was delayed.')
    
    logging.info(f'Cleanup function started at {utc_timestamp}')
    
    try:
        # Your cleanup logic here
        expired_count = delete_expired_sessions()
        logging.info(f'Deleted {expired_count} expired sessions')
        
        stale_count = archive_stale_records()
        logging.info(f'Archived {stale_count} stale records')
        
    except Exception as e:
        logging.error(f'Cleanup failed: {str(e)}', exc_info=True)
        raise  # Re-raise to mark the invocation as failed
    
    logging.info(f'Cleanup completed at {datetime.datetime.utcnow().isoformat()}')
```

## 🔄 Durable Functions — Stateful Serverless Workflows

This is Azure Functions' **most unique and powerful feature**. Nothing like this exists natively in AWS Lambda (you'd need AWS Step Functions as a separate service).

### The Problem:

Regular serverless functions are **stateless** — they run, return, and forget everything. But what about workflows that:
- Take hours or days?
- Need to wait for human approval?
- Run multiple steps in sequence?
- Fan out to many parallel tasks?
- Need to retry individual steps?

### The Solution: Durable Functions

> **Durable Functions** is an extension of Azure Functions that lets you write **stateful workflows** in code. The runtime manages state, checkpoints, and restarts automatically.

### Durable Functions Patterns:

```
Pattern 1: FUNCTION CHAINING (Sequential Steps)
┌────────┐    ┌────────┐    ┌────────┐    ┌────────┐
│ Step 1 │───→│ Step 2 │───→│ Step 3 │───→│ Step 4 │
│ Validate│   │ Process │   │ Approve│   │ Notify │
└────────┘    └────────┘    └────────┘    └────────┘

Pattern 2: FAN-OUT / FAN-IN (Parallel Execution)
                ┌────────────┐
           ┌───→│ Process A  │───┐
┌────────┐ │    └────────────┘   │    ┌──────────┐
│ Split  │─┤    ┌────────────┐   ├───→│ Combine  │
│ Work   │ ├───→│ Process B  │───┤    │ Results  │
└────────┘ │    └────────────┘   │    └──────────┘
           └───→┌────────────┐───┘
                │ Process C  │
                └────────────┘

Pattern 3: HUMAN INTERACTION (Wait for Approval)
┌────────┐    ┌───────────────────┐    ┌────────┐
│ Submit │───→│ Wait for Approval │───→│ Process│
│ Request│    │ (can wait days!)  │    │ Order  │
└────────┘    └───────────────────┘    └────────┘

Pattern 4: MONITOR (Periodic Polling)
┌─────────┐    ┌──────────┐    ┌──────────┐
│ Start   │───→│ Check    │───→│ Done?    │
│ Monitor │    │ Status   │    │ No → Wait│──→ (loop back to Check)
└─────────┘    └──────────┘    │ Yes → ✅ │
                               └──────────┘
```

### Durable Functions Code Example — Function Chaining:

```python
# Python Durable Functions

import azure.functions as func
import azure.durable_functions as df
import json
import logging

app = func.FunctionApp()

# STEP 1: HTTP Starter — Kicks off the orchestration
@app.route(route="start-order-processing")
@app.durable_client_input(client_name="client")
async def http_start(req: func.HttpRequest, client) -> func.HttpResponse:
    order_data = req.get_json()
    
    # Start the orchestration
    instance_id = await client.start_new("order_orchestrator", None, order_data)
    
    logging.info(f"Started orchestration with ID = '{instance_id}'")
    
    # Return URLs to check status, terminate, etc.
    return client.create_check_status_response(req, instance_id)


# STEP 2: Orchestrator — Defines the workflow (the "conductor")
@app.orchestration_trigger(context_name="context")
def order_orchestrator(context: df.DurableOrchestrationContext):
    order = context.get_input()
    
    # Step 1: Validate the order
    validation_result = yield context.call_activity("validate_order", order)
    
    if not validation_result['is_valid']:
        return {"status": "failed", "reason": "Validation failed"}
    
    # Step 2: Process payment
    payment_result = yield context.call_activity("process_payment", order)
    
    # Step 3: Update inventory
    inventory_result = yield context.call_activity("update_inventory", order)
    
    # Step 4: Send confirmation
    yield context.call_activity("send_confirmation", {
        "order": order,
        "payment": payment_result
    })
    
    return {
        "status": "completed",
        "orderId": order['orderId'],
        "paymentId": payment_result['paymentId']
    }


# STEP 3: Activity Functions — Do the actual work
@app.activity_trigger(input_name="order")
def validate_order(order: dict) -> dict:
    logging.info(f"Validating order: {order['orderId']}")
    # Validation logic...
    return {"is_valid": True, "message": "Order is valid"}


@app.activity_trigger(input_name="order")
def process_payment(order: dict) -> dict:
    logging.info(f"Processing payment for order: {order['orderId']}")
    # Payment processing logic...
    return {"paymentId": "PAY-12345", "status": "success"}


@app.activity_trigger(input_name="order")
def update_inventory(order: dict) -> dict:
    logging.info(f"Updating inventory for order: {order['orderId']}")
    # Inventory update logic...
    return {"status": "updated"}


@app.activity_trigger(input_name="data")
def send_confirmation(data: dict) -> None:
    logging.info(f"Sending confirmation for order: {data['order']['orderId']}")
    # Send email, push notification, etc.
```

### Durable Functions — Fan-Out / Fan-In Example:

```python
@app.orchestration_trigger(context_name="context")
def parallel_processing(context: df.DurableOrchestrationContext):
    """Process multiple items in parallel, then combine results."""
    
    items = context.get_input()  # e.g., ["file1.jpg", "file2.jpg", "file3.jpg"]
    
    # Fan-out: Start all tasks in parallel
    parallel_tasks = []
    for item in items:
        task = context.call_activity("process_image", item)
        parallel_tasks.append(task)
    
    # Fan-in: Wait for ALL tasks to complete
    results = yield context.task_all(parallel_tasks)
    
    # Combine results
    total_processed = len(results)
    return {"total_processed": total_processed, "results": results}
```

### Durable Functions — Human Approval Example:

```python
@app.orchestration_trigger(context_name="context")
def approval_workflow(context: df.DurableOrchestrationContext):
    """Wait for human approval with a timeout."""
    
    request = context.get_input()
    
    # Send approval request (email, Slack, etc.)
    yield context.call_activity("send_approval_request", request)
    
    # Wait for external event (approval/rejection) with 72-hour timeout
    import datetime
    timeout = context.current_utc_datetime + datetime.timedelta(hours=72)
    
    approval_event = context.wait_for_external_event("ApprovalResponse")
    timeout_event = context.create_timer(timeout)
    
    # Wait for either approval or timeout
    winner = yield context.task_any([approval_event, timeout_event])
    
    if winner == approval_event:
        approval = approval_event.result
        if approval['approved']:
            yield context.call_activity("process_approved_request", request)
            return {"status": "approved"}
        else:
            return {"status": "rejected", "reason": approval.get('reason')}
    else:
        # Timeout — escalate!
        yield context.call_activity("escalate_request", request)
        return {"status": "escalated", "reason": "No response within 72 hours"}
```

### Application Insights Integration:

import logging

def my_function(req):

        }
    })

### Key Metrics to Monitor:

| Metric | What It Tells You |
|--------|-------------------|
| **Function Execution Count** | How many times your function ran |
| **Execution Duration** | How long each execution took |
| **Failure Rate** | Percentage of failed executions |
| **Success Rate** | Percentage of successful executions |
| **Server Response Time** | End-to-end response time (for HTTP functions) |
| **Active Instance Count** | Number of running function instances (scaling) |
| **Function Execution Units** | Total consumption (for billing estimation) |

### KQL Queries in Application Insights:

```kql
// Find all failed function executions in the last 24 hours
requests
| where timestamp > ago(24h)
| where success == false
| project timestamp, name, duration, resultCode
| order by timestamp desc

// Average execution duration by function name
requests
| where timestamp > ago(7d)
| summarize avg_duration = avg(duration), count = count() by name
| order by avg_duration desc

// Find slow executions (> 5 seconds)
requests
| where timestamp > ago(24h)
| where duration > 5000
| project timestamp, name, duration, customDimensions
| order by duration desc
```

## 🔒 Security Best Practices

### 1. Authentication & Authorization:

```python
# Auth levels for HTTP triggers:

# ANONYMOUS — No authentication required (public API)
@app.route(route="public", auth_level=func.AuthLevel.ANONYMOUS)
def public_api(req): ...

# FUNCTION — Requires a function key in the URL or header
@app.route(route="internal", auth_level=func.AuthLevel.FUNCTION)
def internal_api(req): ...
# Call with: https://myapp.azurewebsites.net/api/internal?code=YOUR_FUNCTION_KEY

# ADMIN — Requires the master key (highest privilege)
@app.route(route="admin", auth_level=func.AuthLevel.ADMIN)
def admin_api(req): ...
```

### 2. Managed Identity (No Credentials in Code!):

```python
# Instead of storing connection strings, use Managed Identity
from azure.identity import DefaultAzureCredential
from azure.cosmos import CosmosClient

# This automatically uses the Function App's managed identity
credential = DefaultAzureCredential()
client = CosmosClient("https://mycosmosdb.documents.azure.com:443/", credential)
```

### 3. Key Vault References:

```json
// In App Settings, reference secrets from Key Vault instead of storing them directly:
{
    "DB_PASSWORD": "@Microsoft.KeyVault(SecretUri=https://myvault.vault.azure.net/secrets/db-password/)"
}
```

## 🆚 Azure Functions vs AWS Lambda — Detailed Comparison

| Feature | Azure Functions | AWS Lambda |
|---------|----------------|------------|
| **Max Timeout** | 10 min (Consumption), Unlimited (Premium) | 15 minutes |
| **Bindings** | Rich declarative bindings | Manual SDK integration |
| **Stateful Workflows** | Durable Functions (built-in) | Step Functions (separate service) |
| **Local Dev Experience** | Excellent (Core Tools + VS Code) | Good (SAM CLI) |
| **Hosting Options** | Consumption, Premium, Dedicated, Container Apps | Consumption only |
| **VNet Integration** | Premium Plan | Yes (all plans) |
| **Container Support** | Yes (Container Apps hosting) | Yes (container images) |
| **Cold Start** | Slower on Consumption, eliminated on Premium | Can be reduced with Provisioned Concurrency |
| **Ecosystem** | Azure (Cosmos DB, Service Bus, Event Grid) | AWS (DynamoDB, SQS, EventBridge) |
| **Pricing** | Similar ($0.20/1M executions) | Similar ($0.20/1M requests) |
