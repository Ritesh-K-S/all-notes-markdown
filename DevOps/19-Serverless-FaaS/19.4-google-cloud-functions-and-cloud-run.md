## What Are We Going to Learn?

Google Cloud Platform (GCP) gives you **two awesome serverless compute options**:

1. **Google Cloud Functions** — Write a small function, GCP runs it for you when something happens.
2. **Google Cloud Run** — Package your app in a container, GCP runs it serverlessly.

Both let you **run code without managing any servers**. No patching, no scaling headaches, no SSH-ing into VMs at 2 AM.

Let's learn both, step by step! 🎯

## Google Cloud Functions

### What is Google Cloud Functions?

Google Cloud Functions is Google's **Function-as-a-Service (FaaS)** platform. You write a single function (a small piece of code), upload it, and Google:

- **Runs it** whenever an event happens (HTTP request, file upload, Pub/Sub message, etc.)
- **Scales it** automatically — from 0 to thousands of instances
- **Charges you** only when the function is running (pay-per-invocation)

Think of it like this:

> "Hey Google, here's my function. Run it whenever someone calls this URL, or whenever a file lands in this bucket. I don't want to manage any servers."

That's it. That's Cloud Functions.

---

### How Does It Work? (The Big Picture)

```
                    ┌──────────────────────┐
   Event Trigger    │  Google Cloud         │
   ──────────────►  │  Functions Runtime    │ ──► Your Function Executes
   (HTTP, Pub/Sub,  │  (Auto-provisioned)   │
    GCS, Firestore) └──────────────────────┘
                              │
                              ▼
                    Function scales to 0 when idle
                    (you pay nothing when idle)
```

**Flow:**

1. An **event** happens (someone hits a URL, a file is uploaded, a message arrives)
2. GCP **spins up** a lightweight container to run your function
3. Your function **executes** and returns a result
4. If no more events come, the container **scales down to zero**

---

### Cloud Functions Generations — 1st Gen vs 2nd Gen

Google Cloud Functions has two generations:

| Feature | 1st Gen | 2nd Gen (Recommended) |
|---------|---------|----------------------|
| **Built on** | Custom infrastructure | Cloud Run + Eventarc |
| **Max timeout** | 9 minutes (540s) | 60 minutes (3600s) |
| **Max memory** | 8 GB | 32 GB |
| **Max vCPUs** | 2 | 8 |
| **Concurrency** | 1 request per instance | Up to 1000 requests per instance |
| **Traffic splitting** | ❌ | ✅ (for canary deployments) |
| **Min instances** | ✅ | ✅ |
| **Event sources** | Limited | Eventarc (90+ event sources) |

**Bottom line:** Always use **2nd Gen** for new projects. It's faster, more powerful, and more flexible.

### Writing Your First Cloud Function

Let's write a simple HTTP function in **Python** and **Node.js**:

#### Python Example

Create a file called `main.py`:

```python
import functions_framework

@functions_framework.http
def hello_world(request):
    """
    This function handles HTTP requests.
    
    Args:
        request: The HTTP request object (Flask request)
    
    Returns:
        A friendly greeting message
    """
    # Get the 'name' from query string, default to 'World'
    name = request.args.get('name', 'World')
    
    return f'Hello, {name}! Welcome to Google Cloud Functions! 🎉'
```

And a `requirements.txt`:

```
functions-framework==3.*
```

**What's happening here?**
- `functions_framework` is Google's library that handles the plumbing (receiving HTTP requests, routing, etc.)
- `@functions_framework.http` tells Google "this is an HTTP-triggered function"
- `request` is a standard Flask request object — you can access query params, body, headers, etc.
- We grab the `name` parameter from the URL and return a greeting

#### Node.js Example

Create a file called `index.js`:

```javascript
const functions = require('@google-cloud/functions-framework');

// Register an HTTP function
functions.http('helloWorld', (req, res) => {
    // Get the 'name' from query string, default to 'World'
    const name = req.query.name || 'World';
    
    res.send(`Hello, ${name}! Welcome to Google Cloud Functions! 🎉`);
});
```

And a `package.json`:

```json
{
    "name": "hello-cloud-function",
    "version": "1.0.0",
    "dependencies": {
        "@google-cloud/functions-framework": "^3.0.0"
    }
}
```

**What's happening here?**
- We import `@google-cloud/functions-framework` — Google's framework for Cloud Functions
- `functions.http('helloWorld', ...)` registers an HTTP function named `helloWorld`
- `req` and `res` work just like Express.js — because under the hood, it IS Express

#### Event-Driven Function Example (Cloud Storage Trigger)

When a file is uploaded to a GCS bucket, this function runs:

```python
import functions_framework

@functions_framework.cloud_event
def process_file(cloud_event):
    """
    Triggered when a file is uploaded to a Cloud Storage bucket.
    
    Args:
        cloud_event: The CloudEvent object containing event data
    """
    data = cloud_event.data
    
    bucket = data["bucket"]        # Which bucket?
    file_name = data["name"]       # What file was uploaded?
    file_size = data["size"]       # How big is it?
    content_type = data["contentType"]  # What type of file?
    
    print(f"📁 New file detected!")
    print(f"   Bucket: {bucket}")
    print(f"   File: {file_name}")
    print(f"   Size: {file_size} bytes")
    print(f"   Type: {content_type}")
    
    # You could now:
    # - Process the file (resize image, parse CSV, etc.)
    # - Move it to another bucket
    # - Store metadata in a database
    # - Send a notification
```

**What's happening here?**
- `@functions_framework.cloud_event` tells Google "this function is triggered by events, not HTTP"
- `cloud_event.data` contains all the details about what happened
- We're just printing info, but in real life you'd do something useful (resize images, process CSVs, etc.)

### Deploying Cloud Functions (gcloud CLI)

First, install the [Google Cloud SDK](https://cloud.google.com/sdk/docs/install) and authenticate:

```bash
# Login to GCP
gcloud auth login

# Set your project
gcloud config set project YOUR_PROJECT_ID
```

#### Deploy an HTTP Function (2nd Gen)

```bash
gcloud functions deploy hello-world \
    --gen2 \                           # Use 2nd generation
    --runtime python312 \              # Python 3.12
    --region us-central1 \             # Deploy region
    --source . \                       # Source code directory
    --entry-point hello_world \        # Function name in code
    --trigger-http \                   # HTTP trigger
    --allow-unauthenticated            # Anyone can call it (for testing)
```

**What each flag means:**
- `--gen2` → Use the newer, better 2nd gen runtime
- `--runtime python312` → Which language/version to use
- `--region` → Where to deploy (pick a region close to your users)
- `--source .` → Where your code lives (current directory)
- `--entry-point` → The function name to call (must match the function name in your code)
- `--trigger-http` → Make it an HTTP function
- `--allow-unauthenticated` → No auth required (use carefully in production!)

#### Deploy an Event-Driven Function (GCS Trigger)

```bash
gcloud functions deploy process-file \
    --gen2 \
    --runtime python312 \
    --region us-central1 \
    --source . \
    --entry-point process_file \
    --trigger-event-filters="type=google.cloud.storage.object.v1.finalized" \
    --trigger-event-filters="bucket=my-upload-bucket"
```

This function will run **every time a file is uploaded** to the `my-upload-bucket` bucket.

#### Test Your Deployed Function

```bash
# For HTTP functions — just curl the URL
curl https://us-central1-YOUR_PROJECT_ID.cloudfunctions.net/hello-world?name=DevOps

# Or use gcloud
gcloud functions call hello-world --gen2 --data '{"name": "DevOps"}'
```

### Environment Variables & Secrets

#### Setting Environment Variables

```bash
gcloud functions deploy my-function \
    --gen2 \
    --runtime python312 \
    --source . \
    --entry-point my_function \
    --trigger-http \
    --set-env-vars DATABASE_URL=postgres://...,API_KEY=abc123
```

Access them in code:

```python
import os

db_url = os.environ.get('DATABASE_URL')
api_key = os.environ.get('API_KEY')
```

#### Using Secret Manager (Better for Secrets!)

Don't put real secrets in environment variables. Use **Secret Manager** instead:

```bash
# First, create a secret
echo -n "my-super-secret-password" | gcloud secrets create db-password --data-file=-

# Then reference it in your function
gcloud functions deploy my-function \
    --gen2 \
    --runtime python312 \
    --source . \
    --entry-point my_function \
    --trigger-http \
    --set-secrets 'DB_PASSWORD=db-password:latest'
```

Now `DB_PASSWORD` will be available as an environment variable, but the actual secret is safely stored in Secret Manager.

### Cold Starts in Cloud Functions

**Cold start** = The delay when your function is invoked for the first time (or after being idle). GCP has to:

1. Find a server
2. Start a container
3. Load your code
4. Initialize dependencies
5. THEN run your function

**How long are cold starts?**

| Language | Typical Cold Start |
|----------|--------------------|
| Node.js | 200ms — 800ms |
| Python | 300ms — 1s |
| Go | 100ms — 500ms |
| Java | 1s — 5s (JVM startup) |

**How to reduce cold starts:**

```bash
# Set minimum instances to keep functions warm
gcloud functions deploy my-function \
    --gen2 \
    --min-instances 1 \   # Always keep at least 1 instance ready
    --max-instances 100   # But don't go crazy with scaling
```

Other tips:
- **Use lazy initialization** — don't load everything at startup
- **Keep dependencies minimal** — fewer packages = faster startup
- **Use Go or Node.js** — they have the fastest cold starts
- **Global variables** — initialize database connections outside the function handler (they persist between invocations)

```python
import functions_framework
from google.cloud import firestore

# This runs ONCE when the container starts (shared between invocations)
db = firestore.Client()

@functions_framework.http
def my_function(request):
    # This runs on EVERY request
    # 'db' is already initialized — no cold start penalty!
    doc = db.collection('users').document('user1').get()
    return doc.to_dict()
```

### Connecting to Other GCP Services

Cloud Functions integrate seamlessly with other GCP services:

```python
import functions_framework
from google.cloud import storage
from google.cloud import pubsub_v1

@functions_framework.http
def process_and_notify(request):
    # Talk to Cloud Storage
    storage_client = storage.Client()
    bucket = storage_client.bucket('my-bucket')
    blob = bucket.blob('output.txt')
    blob.upload_from_string('Hello from Cloud Functions!')
    
    # Publish a message to Pub/Sub
    publisher = pubsub_v1.PublisherClient()
    topic_path = publisher.topic_path('my-project', 'my-topic')
    publisher.publish(topic_path, data=b'File processed successfully!')
    
    return 'Done! File saved and notification sent.'
```

### IAM & Security

#### Authentication for HTTP Functions

By default (2nd Gen), HTTP functions are **private**. To call them, you need authentication:

```bash
# Call an authenticated function
curl -H "Authorization: Bearer $(gcloud auth print-identity-token)" \
    https://REGION-PROJECT.cloudfunctions.net/my-function
```

#### Granting Access

```bash
# Allow a specific service account to invoke the function
gcloud functions add-invoker-policy-binding my-function \
    --gen2 \
    --region us-central1 \
    --member="serviceAccount:my-sa@my-project.iam.gserviceaccount.com"

# Allow anyone (public) — be careful!
gcloud functions add-invoker-policy-binding my-function \
    --gen2 \
    --region us-central1 \
    --member="allUsers"
```

#### Using a Custom Service Account

```bash
gcloud functions deploy my-function \
    --gen2 \
    --service-account my-function-sa@my-project.iam.gserviceaccount.com
```

**Best practice:** Create a dedicated service account for each function with **least privilege** permissions.

### Pricing

Cloud Functions pricing is based on:

| Component | Price |
|-----------|-------|
| **Invocations** | $0.40 per million (first 2M free/month) |
| **Compute time** | Based on memory + CPU allocated (per 100ms) |
| **Networking** | Outbound data transfer charges |

**Example:** A function with 256MB memory running for 200ms:
- ~$0.000000463 per invocation
- 2 million free invocations per month
- It's **ridiculously cheap** for light workloads!

## Google Cloud Run

### What is Cloud Run?

Cloud Run is Google's **container-based serverless platform**. Instead of writing a single function, you:

1. **Package your entire app** in a Docker container
2. **Push it** to Google Container Registry (or Artifact Registry)
3. **Deploy it** to Cloud Run
4. Cloud Run **runs your container**, scales it up/down, and gives you an HTTPS URL

Think of it like this:

> "Hey Google, here's my containerized app. Run it for me. Scale it up when traffic comes, scale to zero when it's idle. I don't want to manage anything."

Cloud Run is like a **managed Kubernetes for a single service** — without the complexity.

---

### Cloud Functions vs Cloud Run — When to Use What?

| Criteria | Cloud Functions | Cloud Run |
|----------|----------------|-----------|
| **Use case** | Small, single-purpose functions | Full applications, APIs, microservices |
| **Code structure** | Single function | Any app (Express, Flask, FastAPI, etc.) |
| **Container** | Managed by Google | You bring your own container |
| **Languages** | Only supported runtimes | Anything that runs in a container |
| **Max timeout** | 60 min (2nd gen) | 60 min |
| **Concurrency** | Up to 1000/instance (2nd gen) | Up to 1000/instance |
| **Use when...** | "I just want to run this function" | "I have a full app/API" |

**Rule of thumb:**
- Small event-driven tasks → **Cloud Functions**
- Full apps, APIs, microservices → **Cloud Run**
- Need a specific runtime not in Cloud Functions → **Cloud Run**
- Already have a Dockerfile → **Cloud Run**

---

### How Cloud Run Works

```
                      ┌────────────────────┐
    HTTPS Request     │                    │
    ───────────────►  │   Cloud Run        │
                      │   (Your Container) │
    Auto-scales       │                    │
    0 → N instances   └────────────────────┘
                              │
                              ▼
                    Scales to 0 when no traffic
                    (you pay nothing!)
```

Cloud Run:
1. Takes your **container image**
2. Runs it on Google's infrastructure
3. Gives you an **HTTPS URL** automatically (with valid TLS cert!)
4. **Auto-scales** from 0 to thousands of instances based on traffic
5. **Charges** only for the time your container is processing requests

### Writing a Cloud Run Service

Let's create a simple Python web app with **Flask**:

#### Project Structure

```
my-cloud-run-app/
├── app.py                # Main application
├── requirements.txt      # Python dependencies
└── Dockerfile            # Container definition
```

#### app.py

```python
import os
from flask import Flask, request, jsonify

app = Flask(__name__)

@app.route('/')
def home():
    """Home route — just a friendly greeting."""
    return jsonify({
        "message": "Hello from Cloud Run! 🚀",
        "service": os.environ.get('K_SERVICE', 'unknown'),
        "revision": os.environ.get('K_REVISION', 'unknown')
    })

@app.route('/api/greet', methods=['POST'])
def greet():
    """API endpoint that accepts a name and returns a greeting."""
    data = request.get_json()
    name = data.get('name', 'World')
    return jsonify({
        "greeting": f"Hello, {name}! Welcome to Cloud Run!"
    })

@app.route('/health')
def health():
    """Health check endpoint — useful for monitoring."""
    return jsonify({"status": "healthy"}), 200

if __name__ == '__main__':
    # Cloud Run sets the PORT environment variable
    port = int(os.environ.get('PORT', 8080))
    app.run(host='0.0.0.0', port=port)
```

**Key things to notice:**
- `PORT` environment variable — Cloud Run tells your container which port to listen on
- `host='0.0.0.0'` — Must listen on all interfaces, not just localhost
- `K_SERVICE` and `K_REVISION` — Cloud Run automatically sets these env vars

#### requirements.txt

```
Flask==3.0.0
gunicorn==21.2.0
```

---

### Creating the Dockerfile

```dockerfile
# Use an official Python runtime as the base image
FROM python:3.12-slim

# Set the working directory inside the container
WORKDIR /app

# Copy requirements first (for better Docker layer caching)
COPY requirements.txt .

# Install Python dependencies
RUN pip install --no-cache-dir -r requirements.txt

# Copy the rest of the application code
COPY . .

# Cloud Run uses the PORT environment variable
ENV PORT=8080

# Use gunicorn for production (better than Flask's dev server)
# --bind :$PORT → Listen on the port Cloud Run assigns
# --workers 1 → One worker (Cloud Run handles scaling via instances)
# --threads 8 → 8 threads per worker for concurrency
CMD exec gunicorn --bind :$PORT --workers 1 --threads 8 --timeout 0 app:app
```

**Why gunicorn?**
- Flask's built-in server is for development only
- Gunicorn is a production-grade WSGI server
- `--workers 1` is fine because Cloud Run scales by adding more container instances
- `--threads 8` allows handling 8 concurrent requests per instance

### Deploying to Cloud Run

#### Step 1: Build and Push the Container Image

```bash
# Option A: Build locally and push to Artifact Registry
# First, create a repository
gcloud artifacts repositories create my-repo \
    --repository-format=docker \
    --location=us-central1

# Configure Docker to use gcloud authentication
gcloud auth configure-docker us-central1-docker.pkg.dev

# Build the image
docker build -t us-central1-docker.pkg.dev/MY_PROJECT/my-repo/my-app:v1 .

# Push to Artifact Registry
docker push us-central1-docker.pkg.dev/MY_PROJECT/my-repo/my-app:v1
```

```bash
# Option B: Use Cloud Build (no local Docker needed!)
gcloud builds submit --tag us-central1-docker.pkg.dev/MY_PROJECT/my-repo/my-app:v1
```

#### Step 2: Deploy to Cloud Run

```bash
gcloud run deploy my-app \
    --image us-central1-docker.pkg.dev/MY_PROJECT/my-repo/my-app:v1 \
    --region us-central1 \
    --platform managed \
    --port 8080 \
    --memory 512Mi \
    --cpu 1 \
    --min-instances 0 \
    --max-instances 10 \
    --allow-unauthenticated
```

**What each flag means:**
- `--image` → Your container image
- `--region` → Where to deploy
- `--platform managed` → Fully managed by Google (not on GKE)
- `--port 8080` → The port your app listens on
- `--memory 512Mi` → RAM for each instance
- `--cpu 1` → CPU per instance
- `--min-instances 0` → Scale to zero when idle (saves money)
- `--max-instances 10` → Don't scale beyond 10 instances
- `--allow-unauthenticated` → Public access

**Output:**

```
Service [my-app] revision [my-app-00001-abc] has been deployed and is serving
100 percent of traffic.

Service URL: https://my-app-abc123-uc.a.run.app
```

That's it! Your app is live with HTTPS! 🎉

### Scaling & Concurrency

#### Auto-Scaling Configuration

```bash
gcloud run deploy my-app \
    --image my-image \
    --min-instances 0 \     # Scale to zero (cost-effective)
    --max-instances 100 \   # Maximum instances
    --concurrency 80 \      # Max requests per instance
    --cpu 2 \               # CPU per instance
    --memory 1Gi            # Memory per instance
```

#### How Scaling Works

```
Traffic: 0 requests/sec
Instances: 0 (scaled to zero — you pay nothing!)

Traffic: 10 requests/sec, concurrency=80
Instances: 1 (one instance can handle up to 80 concurrent requests)

Traffic: 200 requests/sec, concurrency=80
Instances: 3 (200/80 ≈ 3 instances needed)

Traffic: 1000 requests/sec, concurrency=80
Instances: 13 (1000/80 ≈ 13 instances)
```

#### CPU Allocation Options

```bash
# CPU is allocated only during request processing (cheaper, default)
gcloud run deploy my-app --no-cpu-throttling=false

# CPU is always allocated (for background work, websockets)
gcloud run deploy my-app --cpu-boost --no-cpu-throttling
```

### Cloud Run Jobs (Batch Tasks)

Cloud Run isn't just for web services! **Cloud Run Jobs** let you run batch tasks:

```bash
# Create a job
gcloud run jobs create my-batch-job \
    --image us-central1-docker.pkg.dev/MY_PROJECT/my-repo/my-batch:v1 \
    --region us-central1 \
    --tasks 10 \           # Run 10 parallel tasks
    --max-retries 3 \      # Retry failed tasks up to 3 times
    --task-timeout 600     # Each task has 10 minutes

# Execute the job
gcloud run jobs execute my-batch-job --region us-central1
```

Use cases for Cloud Run Jobs:
- Data processing / ETL
- Database migrations
- Sending batch emails
- Generating reports
- Cleanup tasks

### Cloud Run + CI/CD

#### Deploying from GitHub with Cloud Build

Create a `cloudbuild.yaml` in your repo:

```yaml
steps:
  # Step 1: Build the container image
  - name: 'gcr.io/cloud-builders/docker'
    args: ['build', '-t', 'us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/my-app:$COMMIT_SHA', '.']
  
  # Step 2: Push the image to Artifact Registry
  - name: 'gcr.io/cloud-builders/docker'
    args: ['push', 'us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/my-app:$COMMIT_SHA']
  
  # Step 3: Deploy to Cloud Run
  - name: 'gcr.io/google.com/cloudsdktool/cloud-sdk'
    entrypoint: gcloud
    args:
      - 'run'
      - 'deploy'
      - 'my-app'
      - '--image'
      - 'us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/my-app:$COMMIT_SHA'
      - '--region'
      - 'us-central1'
      - '--platform'
      - 'managed'

images:
  - 'us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/my-app:$COMMIT_SHA'
```

Then set up a **Cloud Build trigger** that runs this on every push to `main`.

## Cloud Functions vs Cloud Run — Side-by-Side Comparison

| Feature | Cloud Functions | Cloud Run |
|---------|----------------|-----------|
| **Input** | Source code (single function) | Container image (full app) |
| **Languages** | Supported runtimes only | Anything in a container |
| **Trigger types** | HTTP + 90+ event sources | HTTP, Pub/Sub, Scheduler, gRPC |
| **Scale to zero** | ✅ | ✅ |
| **Max timeout** | 60 min (2nd gen) | 60 min |
| **Max memory** | 32 GB | 32 GB |
| **Concurrency** | Up to 1000/instance | Up to 1000/instance |
| **WebSockets** | ❌ | ✅ |
| **gRPC** | ❌ | ✅ |
| **Custom binaries** | Limited | ✅ (full container control) |
| **Deployment** | gcloud deploy | Build image → deploy |
| **Pricing** | Per invocation + compute | Per request + compute |
| **Best for** | Event-driven small tasks | APIs, microservices, full apps |

## Best Practices

### For Cloud Functions
1. **Keep functions small and focused** — One function, one job
2. **Use 2nd Gen** — Always. More features, better performance
3. **Set min instances** for latency-sensitive functions (reduces cold starts)
4. **Initialize clients globally** — Database connections, API clients, etc.
5. **Use Secret Manager** for secrets — never hardcode them
6. **Set appropriate timeouts** — Don't leave the default if your function is fast
7. **Use structured logging** — Makes debugging much easier

### For Cloud Run
1. **Keep container images small** — Use slim/distroless base images
2. **Listen on `PORT` env var** — Cloud Run sets this dynamically
3. **Handle graceful shutdown** — Listen for SIGTERM signals
4. **Set concurrency appropriately** — Based on your app's capacity
5. **Use health checks** — Add a `/health` endpoint
6. **Use Cloud Build** for CI/CD — Automate your deployments
7. **Set CPU allocation wisely** — "Request-only" saves money, "always on" for background work

## Summary

| Topic | Key Takeaway |
|-------|-------------|
| **Cloud Functions** | Write a function, Google runs it on events. Perfect for small, event-driven tasks. |
| **Cloud Run** | Bring a container, Google runs it serverlessly. Perfect for full apps and APIs. |
| **2nd Gen Functions** | Always use 2nd Gen — longer timeouts, more concurrency, more event sources. |
| **Cold Starts** | Minimize with min instances, global initialization, lightweight dependencies. |
| **Scaling** | Both scale from 0 → N automatically. You only pay for what you use. |
| **Deployment** | `gcloud functions deploy` for Functions, `gcloud run deploy` for Cloud Run. |
| **Best choice** | Small event tasks → Functions. Full apps → Cloud Run. Already have Docker → Cloud Run. |
