## What Are We Going to Learn?

So far we've learned about AWS Lambda, Azure Functions, and Google Cloud Functions. Each cloud has its own:

- CLI tool
- Configuration format
- Deployment process
- Syntax and quirks

**What if you could use ONE tool to deploy serverless functions to ANY cloud?**

That's exactly what the **Serverless Framework** does. Let's learn it!

---

## What is the Serverless Framework?

The **Serverless Framework** is an **open-source tool** that makes it easy to build, deploy, and manage serverless applications. It's:

- **Cloud-agnostic** — Works with AWS, Azure, GCP, and more
- **Infrastructure as Code** — Define everything in a `serverless.yml` file
- **Plugin ecosystem** — Extend with 1000+ community plugins
- **Developer-friendly** — Simple CLI commands to deploy, test, and manage

Think of it like this:

> "I don't want to learn the deployment CLI for every cloud. Give me ONE tool that works everywhere."

```
                    ┌─────────────────┐
                    │                 │
   serverless.yml   │   Serverless    │──────► AWS Lambda
   ────────────────►│   Framework     │──────► Azure Functions
                    │   (CLI Tool)    │──────► Google Cloud Functions
                    │                 │──────► Other providers
                    └─────────────────┘
```

## Why Use the Serverless Framework?

### Without Serverless Framework (The Pain)

```bash
# AWS — You need CloudFormation/SAM, package code, upload to S3, create IAM roles...
aws cloudformation deploy --template-file template.yaml --stack-name my-stack ...

# Azure — You need ARM templates, function apps, storage accounts...
az functionapp create --name myapp --resource-group mygroup ...

# GCP — You need gcloud commands, service accounts...
gcloud functions deploy my-function --runtime python312 ...
```

Each cloud has its own way. Different syntax, different tools, different headaches.

### With Serverless Framework (The Joy)

```bash
# Same command for ANY cloud!
serverless deploy
```

One YAML file. One CLI. Every cloud. Done.

### Key Benefits

| Benefit | Description |
|---------|-------------|
| **Simplicity** | Write a YAML file, run `serverless deploy`. That's it. |
| **Multi-cloud** | Same workflow for AWS, Azure, GCP |
| **Infrastructure as Code** | Everything is defined in `serverless.yml` |
| **Auto-provisions** | Creates API Gateway, IAM roles, CloudWatch logs, etc. automatically |
| **Plugin ecosystem** | 1000+ plugins for offline testing, custom domains, warming, etc. |
| **Fast iterations** | `serverless deploy function -f myFunc` deploys a single function in seconds |
| **Rollback** | Easy rollback to previous versions |

## Core Concepts

Before diving in, let's understand the key concepts:

| Concept | What It Is | Example |
|---------|-----------|---------|
| **Service** | Your entire serverless application (a project) | `my-api` |
| **Function** | A single serverless function | `createUser`, `getUsers` |
| **Event** | What triggers a function | HTTP request, S3 upload, cron schedule |
| **Resource** | Additional cloud resources your functions need | DynamoDB table, S3 bucket, SQS queue |
| **Provider** | The cloud platform | AWS, Azure, GCP |
| **Stage** | Environment/deployment stage | `dev`, `staging`, `prod` |
| **Plugin** | Extension that adds functionality | offline testing, custom domains |

```
Service (my-api)
├── Function: createUser
│   └── Event: POST /users (HTTP)
├── Function: getUsers
│   └── Event: GET /users (HTTP)
├── Function: processImage
│   └── Event: S3 bucket upload
├── Resource: UsersTable (DynamoDB)
└── Resource: ImagesBucket (S3)
```

## The serverless.yml File — The Heart of Everything

Everything in the Serverless Framework revolves around `serverless.yml`. This single file defines:

- Which cloud provider to use
- What functions you have
- What triggers (events) each function responds to
- What cloud resources to create
- Environment variables, IAM permissions, and more

Here's the basic structure:

```yaml
# Service name — your project's identity
service: my-awesome-api

# Cloud provider configuration
provider:
  name: aws                    # Which cloud? (aws, azure, google)
  runtime: nodejs20.x          # Language runtime
  region: us-east-1            # Deployment region
  stage: ${opt:stage, 'dev'}   # Environment (dev, staging, prod)

# Your serverless functions
functions:
  hello:
    handler: handler.hello     # File.functionName
    events:
      - httpApi:               # Trigger: HTTP API
          path: /hello
          method: get

# Additional cloud resources (CloudFormation for AWS)
resources:
  Resources:
    MyTable:
      Type: AWS::DynamoDB::Table
      Properties:
        TableName: my-table
        # ... table configuration

# Plugins to extend functionality
plugins:
  - serverless-offline
```

## Your First Serverless Project (AWS Lambda)

### Step 1: Create a New Project

```bash
# Create a new service from a template
serverless create --template aws-nodejs --path my-first-service

# Move into the directory
cd my-first-service
```

This creates:

```
my-first-service/
├── handler.js          # Your function code
└── serverless.yml      # Configuration
```

### Step 2: Look at the Generated Code

**handler.js:**

```javascript
'use strict';

// This is your serverless function
// 'event' contains the incoming request data
// 'context' has runtime information about the function
module.exports.hello = async (event) => {
    return {
        statusCode: 200,
        body: JSON.stringify(
            {
                message: 'Hello from Serverless Framework! 🚀',
                input: event,  // Echo back the event for debugging
            },
            null,
            2  // Pretty-print JSON with 2-space indentation
        ),
    };
};
```

**serverless.yml:**

```yaml
service: my-first-service

provider:
  name: aws
  runtime: nodejs20.x
  region: us-east-1

functions:
  hello:
    handler: handler.hello   # handler.js → export.hello function
    events:
      - httpApi:
          path: /hello
          method: get
```

### Step 3: Deploy!

```bash
serverless deploy

# Output:
# Deploying "my-first-service" to stage "dev" (us-east-1)
# 
# ✔ Service deployed to stack my-first-service-dev
# 
# endpoint: GET - https://abc123.execute-api.us-east-1.amazonaws.com/hello
# functions:
#   hello: my-first-service-dev-hello (1.2 kB)
```

### Step 4: Test It!

```bash
# Call the deployed function via curl
curl https://abc123.execute-api.us-east-1.amazonaws.com/hello

# Or invoke it directly via Serverless CLI
serverless invoke -f hello

# Or invoke locally (no deployment needed!)
serverless invoke local -f hello
```

### Step 5: Check Logs

```bash
# View function logs
serverless logs -f hello

# Tail logs in real-time
serverless logs -f hello --tail
```

### Step 6: Remove Everything

```bash
# Delete all deployed resources
serverless remove
```

## Deep Dive into serverless.yml

### Functions

Functions are the core of your serverless app. Each function maps to a Lambda function (or equivalent on other clouds):

```yaml
functions:
  # Function 1: Create a user
  createUser:
    handler: src/users/create.handler   # File path + export name
    description: "Creates a new user"
    memorySize: 256        # Memory in MB (default: 1024)
    timeout: 10            # Timeout in seconds (default: 6)
    events:
      - httpApi:
          path: /users
          method: post
  
  # Function 2: Get all users
  getUsers:
    handler: src/users/list.handler
    memorySize: 128
    timeout: 5
    events:
      - httpApi:
          path: /users
          method: get
  
  # Function 3: Process uploaded images
  processImage:
    handler: src/images/process.handler
    memorySize: 1024       # Image processing needs more memory
    timeout: 60
    events:
      - s3:
          bucket: my-uploads
          event: s3:ObjectCreated:*
  
  # Function 4: Scheduled cleanup
  cleanup:
    handler: src/jobs/cleanup.handler
    timeout: 300           # 5 minutes
    events:
      - schedule:
          rate: rate(1 day)  # Run once a day
          enabled: true
```

**Key properties for each function:**

| Property | Description | Default |
|----------|-------------|---------|
| `handler` | Path to the function code (file.exportedFunction) | Required |
| `description` | Human-readable description | — |
| `memorySize` | Memory allocation in MB | 1024 |
| `timeout` | Max execution time in seconds | 6 |
| `events` | List of event triggers | — |
| `environment` | Function-specific env vars | — |
| `layers` | Lambda Layers to attach | — |
| `reservedConcurrency` | Max concurrent executions | — |

### Events (Triggers)

Events define **what triggers your function**. Here are the most common ones:

#### HTTP API (API Gateway v2 — Recommended)

```yaml
functions:
  api:
    handler: handler.api
    events:
      - httpApi:
          path: /users/{id}      # Path with parameter
          method: get
      - httpApi:
          path: /users
          method: post
```

#### REST API (API Gateway v1 — More features but heavier)

```yaml
functions:
  api:
    handler: handler.api
    events:
      - http:
          path: /users
          method: get
          cors: true                # Enable CORS
          authorizer:               # Add authentication
            name: myAuthorizer
```

#### S3 (File Uploads)

```yaml
functions:
  processFile:
    handler: handler.processFile
    events:
      - s3:
          bucket: my-uploads
          event: s3:ObjectCreated:*     # Any file created
          rules:
            - prefix: uploads/         # Only files in uploads/ folder
            - suffix: .jpg             # Only .jpg files
```

#### Schedule (Cron Jobs)

```yaml
functions:
  dailyReport:
    handler: handler.dailyReport
    events:
      - schedule:
          rate: cron(0 9 * * ? *)      # Every day at 9 AM UTC
          enabled: true
          input:                       # Pass data to the function
            reportType: daily
  
  everyFiveMinutes:
    handler: handler.checkHealth
    events:
      - schedule:
          rate: rate(5 minutes)        # Every 5 minutes
```

#### SQS (Message Queue)

```yaml
functions:
  processMessage:
    handler: handler.processMessage
    events:
      - sqs:
          arn: !GetAtt MyQueue.Arn
          batchSize: 10                # Process 10 messages at a time
```

#### DynamoDB Streams

```yaml
functions:
  onUserChange:
    handler: handler.onUserChange
    events:
      - stream:
          type: dynamodb
          arn: !GetAtt UsersTable.StreamArn
          batchSize: 100
          startingPosition: LATEST
```

#### SNS (Notifications)

```yaml
functions:
  sendEmail:
    handler: handler.sendEmail
    events:
      - sns:
          topicName: user-notifications
          displayName: User Notifications
```

### Resources

Resources let you create **additional cloud infrastructure** alongside your functions. For AWS, this uses CloudFormation syntax:

```yaml
resources:
  Resources:
    # DynamoDB Table
    UsersTable:
      Type: AWS::DynamoDB::Table
      Properties:
        TableName: ${self:service}-users-${sls:stage}
        AttributeDefinitions:
          - AttributeName: id
            AttributeType: S
        KeySchema:
          - AttributeName: id
            KeyType: HASH
        BillingMode: PAY_PER_REQUEST    # On-demand pricing
    
    # S3 Bucket
    UploadsBucket:
      Type: AWS::S3::Bucket
      Properties:
        BucketName: ${self:service}-uploads-${sls:stage}
    
    # SQS Queue
    ProcessingQueue:
      Type: AWS::SQS::Queue
      Properties:
        QueueName: ${self:service}-processing-${sls:stage}
        VisibilityTimeout: 60
  
  # Outputs — values you can reference later
  Outputs:
    UsersTableArn:
      Value: !GetAtt UsersTable.Arn
    UploadsBucketName:
      Value: !Ref UploadsBucket
```

### Custom Variables

Variables make your `serverless.yml` dynamic and reusable:

```yaml
custom:
  tableName: ${self:service}-users-${sls:stage}
  bucketName: ${self:service}-uploads-${sls:stage}
  
  # Stage-specific settings
  memorySize:
    dev: 128
    staging: 256
    prod: 512

provider:
  name: aws
  runtime: nodejs20.x
  environment:
    USERS_TABLE: ${self:custom.tableName}
    UPLOADS_BUCKET: ${self:custom.bucketName}

functions:
  createUser:
    handler: handler.createUser
    memorySize: ${self:custom.memorySize.${sls:stage}}
```

**Variable reference sources:**

| Syntax | Source | Example |
|--------|--------|---------|
| `${self:...}` | Values in this file | `${self:provider.stage}` |
| `${sls:stage}` | Current stage | `dev`, `prod` |
| `${opt:...}` | CLI options | `${opt:stage, 'dev'}` |
| `${env:...}` | Environment variables | `${env:API_KEY}` |
| `${file(...):...}` | External files | `${file(./config.json):db.host}` |
| `${ssm:...}` | AWS SSM Parameter Store | `${ssm:/my-app/db-url}` |
| `${aws:accountId}` | AWS Account ID | `123456789012` |
| `${aws:region}` | AWS Region | `us-east-1` |

### Stages & Environments

Stages let you deploy the same service to different environments:

```bash
# Deploy to dev (default)
serverless deploy

# Deploy to staging
serverless deploy --stage staging

# Deploy to production
serverless deploy --stage prod
```

Each stage creates **completely separate** resources:

```
my-api-dev-createUser       # Dev Lambda function
my-api-staging-createUser   # Staging Lambda function
my-api-prod-createUser      # Prod Lambda function

my-api-users-dev            # Dev DynamoDB table
my-api-users-staging        # Staging DynamoDB table
my-api-users-prod           # Prod DynamoDB table
```

#### Stage-specific configuration:

```yaml
provider:
  name: aws
  runtime: nodejs20.x
  stage: ${opt:stage, 'dev'}
  
  # Different settings per stage
  environment:
    LOG_LEVEL: ${self:custom.logLevel.${sls:stage}}

custom:
  logLevel:
    dev: debug
    staging: info
    prod: warn
```

## Deploying to AWS

This is the most common and best-supported provider:

```yaml
# serverless.yml for AWS
service: my-aws-api

provider:
  name: aws
  runtime: nodejs20.x
  region: us-east-1
  stage: ${opt:stage, 'dev'}
  
  # IAM permissions for your functions
  iam:
    role:
      statements:
        - Effect: Allow
          Action:
            - dynamodb:PutItem
            - dynamodb:GetItem
            - dynamodb:Scan
          Resource: !GetAtt UsersTable.Arn

functions:
  createUser:
    handler: src/create.handler
    events:
      - httpApi:
          path: /users
          method: post
  
  getUsers:
    handler: src/list.handler
    events:
      - httpApi:
          path: /users
          method: get

resources:
  Resources:
    UsersTable:
      Type: AWS::DynamoDB::Table
      Properties:
        TableName: ${self:service}-users-${sls:stage}
        AttributeDefinitions:
          - AttributeName: id
            AttributeType: S
        KeySchema:
          - AttributeName: id
            KeyType: HASH
        BillingMode: PAY_PER_REQUEST
```

**What happens when you deploy:**

```bash
serverless deploy
```

The Serverless Framework automatically:
1. ✅ Packages your code into a ZIP
2. ✅ Uploads it to an S3 bucket
3. ✅ Creates/updates a CloudFormation stack
4. ✅ Creates Lambda functions
5. ✅ Creates API Gateway endpoints
6. ✅ Sets up IAM roles and permissions
7. ✅ Creates DynamoDB table
8. ✅ Wires everything together

All from that one YAML file!

## Working with Plugins

Plugins are one of the most powerful features of the Serverless Framework. Here are the most useful ones:

### serverless-offline — Run Locally

The most popular plugin! Run your functions locally without deploying:

```bash
# Install
npm install --save-dev serverless-offline
```

```yaml
# serverless.yml
plugins:
  - serverless-offline

custom:
  serverless-offline:
    httpPort: 3000           # Local port
    lambdaPort: 3002         # Lambda port
```

```bash
# Start local server
serverless offline

# Output:
# ┌─────────────────────────────────────────────┐
# │   GET  | http://localhost:3000/hello         │
# │   POST | http://localhost:3000/users         │
# └─────────────────────────────────────────────┘
#
# Server ready: http://localhost:3000
```

Now you can test your functions locally before deploying!

### serverless-dotenv-plugin — Load .env Files

```bash
npm install --save-dev serverless-dotenv-plugin
```

```yaml
plugins:
  - serverless-dotenv-plugin

# .env file variables are automatically loaded into provider.environment
```

Create a `.env` file:

```
DATABASE_URL=postgres://localhost:5432/mydb
API_KEY=my-secret-key
```

### serverless-prune-plugin — Clean Old Versions

Lambda keeps old function versions. This cleans them up:

```bash
npm install --save-dev serverless-prune-plugin
```

```yaml
plugins:
  - serverless-prune-plugin

custom:
  prune:
    automatic: true
    number: 3              # Keep only the last 3 versions
```

### serverless-domain-manager — Custom Domains

```bash
npm install --save-dev serverless-domain-manager
```

```yaml
plugins:
  - serverless-domain-manager

custom:
  customDomain:
    domainName: api.mycompany.com
    basePath: ''
    stage: ${sls:stage}
    createRoute53Record: true
    certificateArn: arn:aws:acm:...
```

## Environment Variables & Secrets

### Setting Environment Variables

```yaml
provider:
  name: aws
  runtime: nodejs20.x
  
  # Global env vars (available to ALL functions)
  environment:
    NODE_ENV: ${sls:stage}
    TABLE_NAME: ${self:custom.tableName}

functions:
  myFunction:
    handler: handler.myFunction
    # Function-specific env vars (override global ones)
    environment:
      SPECIAL_CONFIG: some-value
```

### Using SSM Parameter Store (Recommended for Secrets)

```yaml
provider:
  environment:
    # Pull secrets from AWS SSM Parameter Store
    DB_PASSWORD: ${ssm:/my-app/${sls:stage}/db-password}
    API_KEY: ${ssm:/my-app/${sls:stage}/api-key}
```

Store secrets in SSM first:

```bash
# Store a secret
aws ssm put-parameter \
    --name "/my-app/dev/db-password" \
    --type SecureString \
    --value "super-secret-password"
```

### Using Secrets Manager

```yaml
provider:
  environment:
    DB_CREDENTIALS: ${aws:secretsmanager:/my-app/db-credentials}
```

## Packaging & Dependencies

### Controlling What Gets Packaged

By default, the Serverless Framework packages **everything** in your project. Control it:

```yaml
package:
  # Exclude files from deployment package
  patterns:
    - '!node_modules/**'      # Exclude node_modules (if using layers)
    - '!tests/**'             # Exclude test files
    - '!docs/**'              # Exclude documentation
    - '!.env'                 # Never deploy .env files!
    - '!.git/**'              # Exclude git directory
    - 'src/**'                # Include source code
    - 'package.json'          # Include package.json
```

### Individual Packaging

Package each function separately for smaller deployments:

```yaml
package:
  individually: true     # Each function gets its own ZIP

functions:
  createUser:
    handler: src/users/create.handler
    package:
      patterns:
        - 'src/users/**'     # Only include user-related code
        - 'src/shared/**'    # And shared utilities
  
  processImage:
    handler: src/images/process.handler
    package:
      patterns:
        - 'src/images/**'
        - 'src/shared/**'
```

### Using Lambda Layers

```yaml
functions:
  myFunction:
    handler: handler.myFunction
    layers:
      - arn:aws:lambda:us-east-1:123456789:layer:my-dependencies:3

layers:
  dependencies:
    path: layer
    compatibleRuntimes:
      - nodejs20.x
    description: "Shared dependencies layer"
```

### Unit Testing (with Jest)

```javascript
// handler.test.js
const { hello } = require('./handler');

describe('hello function', () => {
```

## CI/CD with Serverless Framework

### GitHub Actions Example

```yaml
# .github/workflows/deploy.yml
name: Deploy Serverless

on:
  push:
    branches:
      - main        # Deploy to prod on push to main
      - develop     # Deploy to dev on push to develop

jobs:
  deploy:
    runs-on: ubuntu-latest
    
    steps:
      - uses: actions/checkout@v4
      
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
      
      - name: Install dependencies
        run: npm ci
      
      - name: Deploy to Dev
        if: github.ref == 'refs/heads/develop'
        run: npx serverless deploy --stage dev
        env:
          AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
          AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
      
      - name: Deploy to Prod
        if: github.ref == 'refs/heads/main'
        run: npx serverless deploy --stage prod
        env:
          AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
          AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
```

## Serverless Alternatives

The Serverless Framework isn't the only tool. Here's how it compares:

| Tool | Description | Best For |
|------|-------------|----------|
| **Serverless Framework** | Multi-cloud, plugin ecosystem | General serverless development |
| **AWS SAM** | AWS-native, CloudFormation-based | Pure AWS shops |
| **AWS CDK** | IaC with real programming languages | Complex AWS infrastructure |
| **Terraform** | Multi-cloud IaC (not serverless-specific) | Full infrastructure management |
| **SST (v3)** | Modern serverless framework for AWS | TypeScript-first AWS projects |
| **Architect** | Convention-based serverless | Simple projects |
| **Claudia.js** | Easy Lambda + API Gateway | Small Node.js projects |

### Quick Comparison — Serverless Framework vs AWS SAM

| Feature | Serverless Framework | AWS SAM |
|---------|---------------------|---------|
| **Multi-cloud** | ✅ AWS, Azure, GCP | ❌ AWS only |
| **Config format** | serverless.yml | template.yaml (CloudFormation) |
| **Plugin ecosystem** | 1000+ plugins | Limited |
| **Local testing** | serverless-offline plugin | sam local (built-in) |
| **Learning curve** | Easy | Medium |
| **Community** | Very large | Large (AWS-backed) |
| **Pricing** | Free tier + paid | Free (open-source) |

## Best Practices

### 1. Project Structure

```
my-serverless-app/
├── src/
│   ├── functions/
│   │   ├── users/
│   │   │   ├── create.js
│   │   │   ├── list.js
│   │   │   └── delete.js
│   │   └── images/
│   │       └── process.js
│   ├── lib/                # Shared utilities
│   │   ├── db.js
│   │   └── response.js
│   └── middleware/         # Shared middleware
│       └── auth.js
├── tests/
│   ├── unit/
│   └── integration/
├── events/                 # Test event payloads
│   └── createUser.json
├── serverless.yml
├── package.json
└── .env.example
```

### 2. Use Middy for Middleware (Node.js)

```javascript
// Instead of repeating boilerplate in every function...
const middy = require('@middy/core');
const jsonBodyParser = require('@middy/http-json-body-parser');
const httpErrorHandler = require('@middy/http-error-handler');
const validator = require('@middy/validator');

const createUser = async (event) => {
    // event.body is already parsed as JSON (thanks to middy!)
    const { name, email } = event.body;
    
    // Your business logic here...
    
    return {
        statusCode: 201,
        body: JSON.stringify({ id: '123', name, email })
    };
};

// Wrap your function with middleware
module.exports.handler = middy(createUser)
    .use(jsonBodyParser())       // Parse JSON body
    .use(httpErrorHandler())     // Handle errors gracefully
    .use(validator({ inputSchema }));  // Validate input
```

### 3. Keep Functions Small

```javascript
// ❌ BAD — One giant function doing everything
module.exports.handler = async (event) => {
    // 200 lines of code handling create, read, update, delete...
};

// ✅ GOOD — One function per action
// create.js
module.exports.handler = async (event) => { /* create logic */ };

// list.js
module.exports.handler = async (event) => { /* list logic */ };

// delete.js
module.exports.handler = async (event) => { /* delete logic */ };
```

### 4. Use Stages Properly

```bash
# Never deploy directly to prod
serverless deploy --stage dev     # For development
serverless deploy --stage staging # For testing
serverless deploy --stage prod    # For production (via CI/CD only!)
```

### 5. Monitor Costs

```yaml
provider:
  name: aws
  # Set limits to prevent bill shock
  timeout: 10              # Don't let functions run forever
  memorySize: 256          # Don't over-allocate memory

functions:
  myFunction:
    handler: handler.myFunction
    reservedConcurrency: 10    # Limit concurrent executions
```
