## What Are We Going to Learn?

So far we've learned about cloud-managed serverless platforms (AWS Lambda, Azure Functions, Google Cloud Functions). They're great, but they have a big limitation:

**You're locked into a specific cloud provider.**

What if you want:
- Serverless on **your own Kubernetes cluster**?
- The ability to **move between clouds** without rewriting everything?
- **Scale-to-zero** on Kubernetes (something Kubernetes doesn't do by default)?
- **Serverless at the edge** or **on-premises**?

Enter **Knative** — the project that brings serverless capabilities to Kubernetes!

---

## The Problem: Why Serverless on Kubernetes?

Let's understand why Knative exists:

### Regular Kubernetes Deployment

```yaml
# A typical Kubernetes deployment — always running
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-api
spec:
  replicas: 3    # Always 3 pods running (even at 3 AM with zero traffic!)
```

**Problems:**
- Pods are **always running** — even when no one is using them 💸
- You have to **manually configure HPA** for auto-scaling
- No built-in **traffic splitting** for canary deployments
- No built-in **revision management**
- Setting up **Ingress, TLS, DNS** is a lot of work

### Knative on Kubernetes

```yaml
# A Knative service — scales to zero automatically!
apiVersion: serving.knative.dev/v1
kind: Service
metadata:
  name: my-api
spec:
  template:
    spec:
      containers:
        - image: my-api:latest
# That's it! Knative handles:
# ✅ Auto-scaling (including to zero)
# ✅ Revision management
# ✅ Traffic routing
# ✅ HTTPS/TLS
# ✅ DNS/domain mapping
```

**Knative gives you the serverless experience on top of Kubernetes.**

## What is Knative?

**Knative** (pronounced "kay-native") is an open-source platform that adds serverless capabilities to Kubernetes. It was created by Google, with contributions from IBM, Red Hat, VMware, and many others.

Think of it like this:

> "Kubernetes is great, but it doesn't know how to scale to zero or handle events natively. Knative adds those superpowers."

```
┌──────────────────────────────────────────────────────┐
│                    Knative                            │
│  ┌─────────────┐  ┌──────────────┐  ┌────────────┐  │
│  │   Serving    │  │   Eventing    │  │  Functions  │ │
│  │             │  │              │  │            │  │
│  │ Scale-to-0  │  │ Event-driven │  │ Write      │  │
│  │ Revisions   │  │ Brokers      │  │ functions  │  │
│  │ Traffic     │  │ Triggers     │  │ not YAML   │  │

├──────────────────────────────────────────────────────┤
```

## Knative Architecture — The Big Picture

Knative sits on top of Kubernetes and extends it with Custom Resource Definitions (CRDs):

```
                    User Request
                         │
                         ▼
              ┌──────────────────────┐
              │   Networking Layer   │
              │   (Kourier/Istio/    │
              │    Contour)          │
              └──────────┬───────────┘
                         │
              ┌──────────▼───────────┐
              │  Knative Serving     │
              │  ┌──────────────┐   │
              │  │  Activator   │   │  ← Wakes up scaled-to-zero pods
              │  │  (lives here │   │
              │  │   when pods  │   │
              │  │   are at 0)  │   │
              │  └──────────────┘   │
              │  ┌──────────────┐   │
              │  │ Autoscaler   │   │  ← Scales pods up/down
              │  │ (KPA/HPA)   │   │
              │  └──────────────┘   │
              └──────────┬───────────┘
                         │
              ┌──────────▼───────────┐
              │  Your Container Pods  │  ← Your actual application
              │  (0 to N replicas)    │
              └──────────────────────┘
```

**Key components:**
1. **Activator** — Receives requests when pods are scaled to zero, triggers scale-up, and buffers requests until a pod is ready
2. **Autoscaler** — Watches request metrics and decides how many pods to run
3. **Networking Layer** — Routes traffic (Kourier, Istio, or Contour)
4. **Controller** — Manages Knative resources (Services, Revisions, Routes)

---

## Knative Components

Knative has three main components:

### Knative Serving — Run Serverless Containers

**What it does:** Deploys and auto-scales your containerized applications, including scale-to-zero.

**Key features:**
- ⚡ **Scale to zero** — No traffic? No pods running. No cost.
- 📈 **Auto-scaling** — Automatically scales based on request concurrency or CPU
- 🔄 **Revisions** — Every deployment creates an immutable snapshot (revision)
- 🚦 **Traffic splitting** — Route traffic between revisions (canary, blue/green)
- 🔒 **HTTPS** — Automatic TLS certificate management
- 🌐 **Custom domains** — Map your own domains

### Knative Eventing — Event-Driven Architecture

**What it does:** Provides a platform for building event-driven applications with loose coupling.

**Key features:**
- 📨 **Event Sources** — Consume events from external systems (Kafka, GitHub, Cron, etc.)
- 📬 **Brokers & Triggers** — Route events to the right services based on filters
- 📡 **Channels & Subscriptions** — Fan-out events to multiple consumers
- 📋 **CloudEvents** — Standard event format (CNCF standard)

### Knative Functions — Write Functions, Not Containers

**What it does:** Lets you write simple functions (like Lambda) and deploy them on Knative without writing Dockerfiles.

**Key features:**
- 📝 Write functions in Node.js, Python, Go, Java, Rust, etc.
- 🐳 Auto-builds container images
- ⚡ Deploys to Knative Serving automatically

## Prerequisites & Installation

### What You Need

- A **Kubernetes cluster** (v1.28+)
  - Options: Minikube, Kind, k3s, EKS, AKS, GKE, or any Kubernetes cluster
- **kubectl** installed and configured
- **Helm** (optional but recommended)

### Install Knative on a Kubernetes Cluster

Let's install Knative step by step:

#### Step 1: Install Knative Serving

```bash
# Install Knative Serving CRDs (Custom Resource Definitions)
kubectl apply -f https://github.com/knative/serving/releases/download/knative-v1.14.0/serving-crds.yaml

# Install Knative Serving core components
kubectl apply -f https://github.com/knative/serving/releases/download/knative-v1.14.0/serving-core.yaml
```

#### Step 2: Install a Networking Layer

Knative needs a networking layer for routing traffic. **Kourier** is the simplest:

```bash
# Install Kourier (lightweight, recommended for getting started)
kubectl apply -f https://github.com/knative/net-kourier/releases/download/knative-v1.14.0/kourier.yaml

# Configure Knative to use Kourier
kubectl patch configmap/config-network \
    --namespace knative-serving \
    --type merge \
    --patch '{"data":{"ingress-class":"kourier.ingress.networking.knative.dev"}}'
```

Other options:
- **Istio** — Full service mesh features (heavy but powerful)
- **Contour** — Envoy-based, good balance of features and simplicity

#### Step 3: Configure DNS

For local development, use **sslip.io** (a free DNS service that maps IPs to domains):

```bash
# Get the external IP of Kourier
kubectl get svc kourier -n kourier-system
# Note the EXTERNAL-IP (e.g., 35.192.x.x)

# Configure DNS with sslip.io (magic DNS — no setup needed)
kubectl patch configmap/config-domain \
    --namespace knative-serving \
    --type merge \
    --patch '{"data":{"35.192.x.x.sslip.io":""}}'
```

With sslip.io, your services will get URLs like:
```
http://my-service.default.35.192.x.x.sslip.io
```

#### Step 4: Install Knative Eventing (Optional)

```bash
# Install Eventing CRDs
kubectl apply -f https://github.com/knative/eventing/releases/download/knative-v1.14.0/eventing-crds.yaml

# Install Eventing core
kubectl apply -f https://github.com/knative/eventing/releases/download/knative-v1.14.0/eventing-core.yaml

# Install the in-memory channel (for development — use Kafka for production)
kubectl apply -f https://github.com/knative/eventing/releases/download/knative-v1.14.0/in-memory-channel.yaml

# Install the MT (multi-tenant) broker
kubectl apply -f https://github.com/knative/eventing/releases/download/knative-v1.14.0/mt-channel-broker.yaml
```

#### Step 5: Verify Installation

```bash
# Check Knative Serving pods
kubectl get pods -n knative-serving

# Expected output:
# NAME                                    READY   STATUS    RESTARTS   AGE
# activator-xxxxx                         1/1     Running   0          2m
# autoscaler-xxxxx                        1/1     Running   0          2m
# controller-xxxxx                        1/1     Running   0          2m
# webhook-xxxxx                           1/1     Running   0          2m

# Check Knative Eventing pods (if installed)
kubectl get pods -n knative-eventing

# Check the CRDs
kubectl get crd | grep knative
# services.serving.knative.dev
# routes.serving.knative.dev
# revisions.serving.knative.dev

# brokers.eventing.knative.dev

# ...
```

## Knative Serving — Deep Dive

### Your First Knative Service

Let's deploy a simple app. You can do it with **YAML** or with the **kn CLI**:

#### Option 1: Using YAML

```yaml
# hello-service.yaml
apiVersion: serving.knative.dev/v1
kind: Service
metadata:
  name: hello
  namespace: default
spec:
  template:
    metadata:
      # Annotations for autoscaler configuration
      annotations:
        autoscaling.knative.dev/minScale: "0"    # Scale to zero!
        autoscaling.knative.dev/maxScale: "10"   # Max 10 pods
    spec:
      containers:
        - image: gcr.io/knative-samples/helloworld-go
          ports:
            - containerPort: 8080
          env:
            - name: TARGET
              value: "Knative"
```

```bash
# Deploy it
kubectl apply -f hello-service.yaml

# Check the service
kubectl get ksvc hello
# NAME    URL                                        READY   REASON
# hello   http://hello.default.35.192.x.x.sslip.io  True

# Test it!
curl http://hello.default.35.192.x.x.sslip.io
# Hello Knative!
```

#### Option 2: Using kn CLI (Much Easier!)

```bash
# Create a service with one command
kn service create hello \
    --image gcr.io/knative-samples/helloworld-go \
    --port 8080 \
    --env TARGET=Knative

# Output:
# Creating service 'hello' in namespace 'default':
#   0.032s The Route is still working to reflect the latest desired specification.
#   0.059s Configuration "hello" is waiting for a Revision to become ready.
#   3.752s Ingress has not yet been reconciled.
#   3.879s Waiting for load balancer to be ready
#   4.126s Ready to serve.
# 
# Service 'hello' created to latest revision 'hello-00001'
# URL: http://hello.default.35.192.x.x.sslip.io
```

That's it! Your serverless app is running on Kubernetes!

### Understanding Knative Service Resources

When you create a Knative Service, it automatically creates several Kubernetes resources:

```
Knative Service (ksvc)
├── Configuration
│   └── Revision (hello-00001)     ← Immutable snapshot of your app
│       └── Deployment             ← Standard K8s Deployment
│           └── Pods               ← Your actual containers
└── Route
    └── Ingress                    ← Routes traffic to the right revision
```

```bash
# See all the resources
kubectl get ksvc                    # Knative Service
kubectl get configuration           # Configuration
kubectl get revision                # Revisions (each deploy = new revision)
kubectl get route                   # Route (traffic routing rules)

# Detailed view
kubectl get ksvc hello -o yaml
```

**Key concepts:**

| Resource | Description |
|----------|-------------|
| **Service (ksvc)** | The top-level resource. Manages Configuration + Route. |
| **Configuration** | Defines the desired state of your app (image, env, etc.) |
| **Revision** | An immutable snapshot. Every change creates a new revision. Like a Git commit for deployments. |
| **Route** | Defines how traffic is split between revisions |

### Scale to Zero — The Magic

This is Knative's **killer feature**. When no traffic is coming, your pods scale to ZERO:

```bash
# Deploy a service
kn service create hello --image gcr.io/knative-samples/helloworld-go

# Check pods — they're running!
kubectl get pods
# NAME                                      READY   STATUS    RESTARTS   AGE
# hello-00001-deployment-xxxxx              2/2     Running   0          10s

# Wait ~60 seconds with no traffic...
kubectl get pods
# No resources found in default namespace.
# PODS ARE GONE! Scaled to zero! 🎉

# Now send a request
curl http://hello.default.35.192.x.x.sslip.io
# Hello World!     ← Response comes (after a brief cold start)

# Check pods again
kubectl get pods
# NAME                                      READY   STATUS    RESTARTS   AGE
# hello-00001-deployment-xxxxx              2/2     Running   0          5s
# POD IS BACK! 🚀
```

**How does it work?**

```
1. No traffic → Autoscaler scales to 0 pods
2. Request comes in → Hits the Activator (always running)
3. Activator holds the request → Tells Autoscaler to scale up
4. Pod starts → Activator forwards the buffered request to the pod
5. Response sent back to the user
```

**The cold start:** The time between step 2 and step 5. Typically 1-5 seconds depending on your container image size and startup time.

### Auto-Scaling Configuration

Knative provides two autoscalers:

| Autoscaler | Type | Description |
|------------|------|-------------|
| **KPA** (Knative Pod Autoscaler) | Default | Scale based on request concurrency. Supports scale-to-zero. |
| **HPA** (Horizontal Pod Autoscaler) | Optional | Standard K8s HPA. Scale based on CPU/memory. Does NOT support scale-to-zero. |

#### Configuring KPA (Default)

```yaml
apiVersion: serving.knative.dev/v1
kind: Service
metadata:
  name: my-service
spec:
  template:
    metadata:
      annotations:
        # Autoscaler type
        autoscaling.knative.dev/class: "kpa.autoscaling.knative.dev"   # Default
        
        # Target concurrency per pod
        # "Each pod should handle 10 concurrent requests"
        autoscaling.knative.dev/target: "10"
        
        # Scaling bounds
        autoscaling.knative.dev/minScale: "0"     # Allow scale to zero
        autoscaling.knative.dev/maxScale: "100"   # Max 100 pods
        
        # Scale-down delay (how long to wait before scaling down)
        autoscaling.knative.dev/scale-down-delay: "60s"
        
        # Initial scale (pods to start with on a new revision)
        autoscaling.knative.dev/initial-scale: "1"
    spec:
      # Per-container concurrency limit
      containerConcurrency: 0   # 0 = unlimited (let the autoscaler decide)
      containers:
        - image: my-app:latest
```

**What does `target: "10"` mean?**

```
If your service is handling 10 requests concurrently → 1 pod
If your service is handling 50 requests concurrently → 5 pods (50/10)
If your service is handling 200 requests concurrently → 20 pods (200/10)
If your service is handling 0 requests → 0 pods (after scale-down delay)
```

#### Using kn CLI for Scaling Config

```bash
# Create with scaling annotations
kn service create my-service \
    --image my-app:latest \
    --scale-min 0 \          # Scale to zero
    --scale-max 50 \         # Max 50 pods
    --concurrency-target 10  # 10 concurrent requests per pod
```

#### Prevent Scale to Zero (Always Keep Warm)

```yaml
annotations:
  autoscaling.knative.dev/minScale: "1"   # Always keep at least 1 pod
```

Or with kn:

```bash
kn service update my-service --scale-min 1
```

### Traffic Splitting & Canary Deployments

This is where Knative really shines. You can split traffic between revisions with a simple command:

#### Step 1: Deploy v1

```bash
kn service create my-app --image my-app:v1 --env VERSION=v1
# Creates revision: my-app-00001 (gets 100% traffic)
```

#### Step 2: Deploy v2

```bash
kn service update my-app --image my-app:v2 --env VERSION=v2
# Creates revision: my-app-00002 (gets 100% traffic, v1 gets 0%)
```

#### Step 3: Split Traffic (Canary)

```bash
# Send 80% to v1 (old) and 20% to v2 (new) — canary deployment!
kn service update my-app \
    --traffic my-app-00001=80 \
    --traffic my-app-00002=20
```

#### Step 4: Gradually Increase

```bash
# 50/50 split
kn service update my-app \
    --traffic my-app-00001=50 \
    --traffic my-app-00002=50

# All traffic to v2 (v2 is stable, promote it!)
kn service update my-app \
    --traffic my-app-00002=100
```

#### Using YAML for Traffic Splitting

```yaml
apiVersion: serving.knative.dev/v1
kind: Service
metadata:
  name: my-app
spec:
  template:
    spec:
      containers:
        - image: my-app:v2
  traffic:
    - revisionName: my-app-00001    # Old version
      percent: 80                    # Gets 80% of traffic
    - revisionName: my-app-00002    # New version
      percent: 20                    # Gets 20% of traffic (canary)
    - latestRevision: true           # Tag the latest revision
      tag: latest                    # Accessible at latest-my-app.default.x.x.x.x.sslip.io
```

#### Tag-Based Routing

You can give revisions named tags that create separate URLs:

```bash
# Tag a revision
kn service update my-app --tag my-app-00002=staging

# Now you can access it at:
# http://staging-my-app.default.x.x.x.x.sslip.io
# (Separate from the main URL — great for testing!)
```

### Private Services (Cluster-Local)

```yaml
apiVersion: serving.knative.dev/v1

  name: internal-service
  labels:

        - image: my-internal-app:latest
```

```bash
# Or with kn CLI
kn service create internal-service \
    --image my-internal-app:latest \
    --cluster-local
```

Internal services get URLs like: `http://internal-service.default.svc.cluster.local`

### Container Configuration

You can configure your containers just like regular Kubernetes pods:

```yaml
apiVersion: serving.knative.dev/v1
kind: Service
metadata:
  name: my-app
spec:
  template:
    spec:
      containers:
        - image: my-app:latest
          ports:
            - containerPort: 8080

          env:
            - name: DATABASE_URL
              value: "postgres://db:5432/mydb"
            - name: SECRET_KEY
              valueFrom:
                secretKeyRef:
                  name: my-secrets
                  key: secret-key
          
          # Resource limits
          resources:
            requests:
              cpu: 100m
              memory: 128Mi
            limits:
              cpu: 500m
              memory: 512Mi
          
          # Readiness probe
          readinessProbe:
            httpGet:
              path: /health
              port: 8080
            initialDelaySeconds: 3
            periodSeconds: 5
          
          # Volume mounts
          volumeMounts:
            - name: config
              mountPath: /etc/config

      volumes:
        - name: config
          configMap:
            name: my-config
      
      # Service account
      serviceAccountName: my-service-account
      
      # Timeout for requests (seconds)
      timeoutSeconds: 300
```

## Knative Eventing — Deep Dive

### Core Concepts

Knative Eventing provides a way to build **event-driven architectures** on Kubernetes:

```
Event Source ──► Broker ──► Trigger ──► Service (Consumer)
                  │
                  ├──► Trigger ──► Service B
                  │
                  └──► Trigger ──► Service C

(Events flow through the Broker, Triggers filter and route them)
```

| Concept | What It Is | Analogy |
|---------|-----------|---------|
| **Event Source** | Produces events | A sensor that detects something |
| **Broker** | Receives and stores events | A post office |
| **Trigger** | Filters events and routes to consumers | A mail filter/rule |
| **Sink** | The destination for events | The mailbox |
| **Channel** | Event transport mechanism | A conveyor belt |
| **Subscription** | Links a Channel to a consumer | A subscription to a topic |

---

### Event Sources

Event Sources bring events INTO your Knative cluster from external systems:

#### PingSource (Cron-like Scheduler)

```yaml
apiVersion: sources.knative.dev/v1
kind: PingSource
metadata:
  name: every-minute
spec:
  schedule: "*/1 * * * *"         # Cron expression: every 1 minute
  contentType: "application/json"
  data: '{"message": "Hello from PingSource!"}'
  sink:
    ref:
      apiVersion: serving.knative.dev/v1
      kind: Service
      name: my-event-handler       # Send events to this service
```

**What's happening?** Every minute, PingSource sends a JSON event to the `my-event-handler` Knative Service.

#### ApiServerSource (Kubernetes API Events)

```yaml
apiVersion: sources.knative.dev/v1
kind: ApiServerSource
metadata:
  name: pod-events
spec:
  serviceAccountName: events-sa
  mode: Resource
  resources:
    - apiVersion: v1
      kind: Pod                    # Watch for Pod events
  sink:
    ref:
      apiVersion: serving.knative.dev/v1
      kind: Service
      name: pod-watcher            # Send events here
```

**What's happening?** Every time a pod is created, updated, or deleted, an event is sent to the `pod-watcher` service.

#### KafkaSource (Consume Kafka Messages)

```yaml
apiVersion: sources.knative.dev/v1beta1
kind: KafkaSource
metadata:
  name: kafka-orders
spec:
  bootstrapServers:
    - my-kafka-broker:9092
  topics:
    - orders                       # Kafka topic to consume from
  consumerGroup: knative-group
  sink:
    ref:
      apiVersion: serving.knative.dev/v1
      kind: Service
      name: order-processor        # Process Kafka messages
```

#### Other Available Sources

### Brokers and Triggers

**Broker + Trigger** is the recommended pattern for routing events:

#### Step 1: Create a Broker

```yaml
# broker.yaml
apiVersion: eventing.knative.dev/v1
kind: Broker
metadata:
  name: default
  namespace: default
```

```bash
kubectl apply -f broker.yaml

# Or with kn CLI
kn broker create default
```

#### Step 2: Create Services that Consume Events

```yaml
# order-service.yaml — Handles order events
apiVersion: serving.knative.dev/v1
kind: Service
metadata:
  name: order-processor
spec:
  template:
    spec:
      containers:
        - image: my-order-processor:latest

---
# email-service.yaml — Sends confirmation emails
apiVersion: serving.knative.dev/v1
kind: Service
metadata:
  name: email-sender
spec:
  template:
    spec:
      containers:
        - image: my-email-sender:latest
```

#### Step 3: Create Triggers (Event Filters)

```yaml
# trigger-orders.yaml
# Route order events to the order processor
apiVersion: eventing.knative.dev/v1
kind: Trigger
metadata:
  name: order-trigger
spec:
  broker: default
  filter:
    attributes:
      type: com.myapp.order.created       # Only events of this type
      source: /api/orders                  # From this source
  subscriber:
    ref:
      apiVersion: serving.knative.dev/v1
      kind: Service
      name: order-processor

---
# trigger-emails.yaml
# Route email events to the email sender
apiVersion: eventing.knative.dev/v1
kind: Trigger
metadata:
  name: email-trigger
spec:
  broker: default
  filter:
    attributes:
      type: com.myapp.email.send           # Only email events
  subscriber:
    ref:
      apiVersion: serving.knative.dev/v1
      kind: Service
      name: email-sender
```

Now events sent to the Broker are automatically routed:
- `com.myapp.order.created` events → `order-processor`
- `com.myapp.email.send` events → `email-sender`

### Channels and Subscriptions

Channels are an alternative to Brokers for simpler fan-out scenarios:

```yaml
# Channel — a pipeline for events
apiVersion: messaging.knative.dev/v1
kind: Channel
metadata:
  name: orders-channel
spec:
  channelTemplate:
    apiVersion: messaging.knative.dev/v1
    kind: InMemoryChannel    # For production, use KafkaChannel

---
# Subscription 1 — Send to order processor
apiVersion: messaging.knative.dev/v1
kind: Subscription
metadata:
  name: order-sub
spec:
  channel:
    apiVersion: messaging.knative.dev/v1
    kind: Channel
    name: orders-channel
  subscriber:
    ref:
      apiVersion: serving.knative.dev/v1
      kind: Service
      name: order-processor

---
# Subscription 2 — Also send to analytics service (fan-out!)
apiVersion: messaging.knative.dev/v1
kind: Subscription
metadata:
  name: analytics-sub
spec:
  channel:
    apiVersion: messaging.knative.dev/v1
    kind: Channel
    name: orders-channel
  subscriber:
    ref:
      apiVersion: serving.knative.dev/v1
      kind: Service
      name: analytics-service
```

**Broker vs Channel — When to use what?**

| Pattern | Use Case |
|---------|----------|
| **Broker + Trigger** | When you need to filter events by type/source. Most common pattern. |
| **Channel + Subscription** | When you want to fan-out ALL events to multiple consumers (no filtering). |

### CloudEvents — The Standard

All Knative events use the **CloudEvents** specification (a CNCF standard). This makes events portable and interoperable:

```json
{
    "specversion": "1.0",
    "type": "com.myapp.order.created",
    "source": "/api/orders",
    "id": "A234-1234-1234",
    "time": "2024-01-15T10:30:00Z",
    "datacontenttype": "application/json",
    "data": {
        "orderId": "12345",
        "customer": "Alice",
        "total": 99.99
    }
}
```

Your service receives these as HTTP requests with special headers:

```
POST / HTTP/1.1
ce-specversion: 1.0
ce-type: com.myapp.order.created
ce-source: /api/orders
ce-id: A234-1234-1234
ce-time: 2024-01-15T10:30:00Z
Content-Type: application/json

{"orderId": "12345", "customer": "Alice", "total": 99.99}
```

#### Handling CloudEvents in Your Code

**Python:**

```python
from flask import Flask, request, jsonify
from cloudevents.http import from_http

app = Flask(__name__)

@app.route('/', methods=['POST'])
def handle_event():
    # Parse the CloudEvent
    event = from_http(request.headers, request.get_data())
    
    # Access event attributes
    print(f"Event Type: {event['type']}")
    print(f"Event Source: {event['source']}")
    print(f"Event Data: {event.data}")
    
    # Process the event
    if event['type'] == 'com.myapp.order.created':
        order = event.data
        print(f"Processing order {order['orderId']} for {order['customer']}")
        # ... process the order
    
    return jsonify({"status": "processed"}), 200

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=8080)
```

**Node.js:**

```javascript
const express = require('express');
const { HTTP } = require('cloudevents');

const app = express();
app.use(express.json());

app.post('/', (req, res) => {
    // Parse the CloudEvent
    const event = HTTP.toEvent({ headers: req.headers, body: req.body });
    
    console.log(`Event Type: ${event.type}`);
    console.log(`Event Source: ${event.source}`);
    console.log(`Event Data:`, event.data);
    
    // Process the event
    if (event.type === 'com.myapp.order.created') {
        console.log(`Processing order ${event.data.orderId}`);
        // ... process the order
    }
    
    res.status(200).json({ status: 'processed' });
});

app.listen(8080, () => console.log('Listening on port 8080'));
```

### Building an Event-Driven Pipeline

Let's build a complete event-driven pipeline:

```
User places order
       │
       ▼
┌─────────────┐    ┌──────────┐    ┌────────────────┐
│ Order API   │───►│  Broker  │───►│ Order Processor │
│ (source)    │    │          │    │ (validates,     │
└─────────────┘    │          │    │  saves to DB)   │
                   │          │    └────────────────┘
                   │          │
                   │          │───►┌────────────────┐
                   │          │    │ Email Sender    │
                   │          │    │ (confirmation)  │
                   │          │    └────────────────┘
                   │          │
                   │          │───►┌────────────────┐
                   └──────────┘    │ Analytics       │
                                   │ (track metrics) │
                                   └────────────────┘
```

This entire pipeline can scale to zero when there are no orders! When an order comes in, only the relevant services wake up.

**func.py** (auto-generated):

```python
from parliament import Context

def main(context: Context):
    """
    Main function handler.
    Called when an event is received or an HTTP request is made.
    """
    if context.cloud_event:
        # Handle CloudEvent
        return handle_event(context)
    else:
        # Handle HTTP request
        return handle_http(context)

def handle_http(context: Context):
    """Handle a regular HTTP request."""
    body = context.request.json if context.request.is_json else {}
    name = body.get("name", "World")
    return {"message": f"Hello, {name}!"}

def handle_event(context: Context):
    """Handle a CloudEvent."""
    event_data = context.cloud_event.data
    print(f"Received event: {event_data}")
    return {"status": "event processed"}
```

```bash
# Invoke directly
func invoke --data '{"name": "DevOps"}'
# {"message": "Hello, DevOps!"}
```

## Knative vs Cloud Serverless Platforms

| Feature | Knative | AWS Lambda | Google Cloud Functions | Azure Functions |
|---------|---------|------------|----------------------|-----------------|
| **Runs on** | Any Kubernetes cluster | AWS only | GCP only | Azure only |
| **Vendor lock-in** | ❌ None | ✅ High | ✅ High | ✅ High |
| **Scale to zero** | ✅ | ✅ | ✅ | ✅ |
| **Container support** | ✅ Any container | ✅ (with limitations) | ✅ (Cloud Run) | ✅ |
| **Language support** | Any (it's containers) | Limited runtimes | Limited runtimes | Limited runtimes |
| **Max timeout** | Configurable | 15 min | 60 min | 10 min (Consumption) |
| **Event system** | CloudEvents + Eventing | EventBridge | Eventarc | Event Grid |
| **Ops overhead** | You manage K8s | Zero | Zero | Zero |
| **Cost** | You pay for K8s infra | Pay per invocation | Pay per invocation | Pay per invocation |
| **Best for** | Multi-cloud, on-prem, no vendor lock-in | AWS-native apps | GCP-native apps | Azure-native apps |

**When to choose Knative:**
- You already have Kubernetes clusters
- You want to avoid vendor lock-in
- You need serverless on-premises or at the edge
- You want consistent developer experience across environments
- You need more control over the runtime environment

**When NOT to choose Knative:**
- You don't want to manage Kubernetes
- You want zero operational overhead
- Your team is small and you want the simplest option
- You're fully committed to one cloud provider

## Best Practices

### 1. Container Image Best Practices

```dockerfile
# ✅ Use small base images for faster cold starts
FROM python:3.12-slim
# Or even better:
FROM gcr.io/distroless/python3

# ✅ Multi-stage builds
FROM node:20 AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --production
COPY . .
RUN npm run build

FROM node:20-slim
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
CMD ["node", "dist/index.js"]
```

### 2. Reduce Cold Start Time

```yaml
annotations:
  # Keep at least 1 pod warm for critical services
  autoscaling.knative.dev/minScale: "1"
  
  # Start with more pods for expected traffic spikes
  autoscaling.knative.dev/initial-scale: "3"
```

### 3. Set Proper Resource Limits

```yaml
containers:
  - image: my-app:latest
    resources:
      requests:
        cpu: 100m
        memory: 128Mi
      limits:
        cpu: 1000m
        memory: 512Mi
```

### 4. Use Readiness Probes

```yaml
containers:
  - image: my-app:latest
    readinessProbe:
      httpGet:
        path: /health
        port: 8080
      initialDelaySeconds: 0
      periodSeconds: 3
```

### 5. Use Structured Logging

```python
import json
import logging

logging.basicConfig(
    format='{"timestamp":"%(asctime)s","level":"%(levelname)s","message":"%(message)s"}',
```

## Troubleshooting Common Issues

### Service Not Ready

```bash
# Check the service status
kubectl get ksvc my-service -o yaml

# Check events
kubectl describe ksvc my-service

# Common issues:
# - Image pull errors → Check image name and registry access
# - Container crash → Check logs: kubectl logs -l serving.knative.dev/service=my-service
# - Port mismatch → Ensure containerPort matches your app's port
```

### Pods Not Scaling to Zero

```bash
# Check autoscaler config
kubectl get configmap config-autoscaler -n knative-serving -o yaml

# Check annotations on your service
kubectl get ksvc my-service -o jsonpath='{.spec.template.metadata.annotations}'

# Make sure minScale is 0
kn service update my-service --scale-min 0
```

### Cold Start Too Slow

```bash
# Reduce image size
docker images my-app  # Check image size

# Keep min instances
kn service update my-service --scale-min 1

# Use smaller base images (alpine, slim, distroless)
```

### Events Not Being Delivered

```bash
# Check broker
kubectl get broker default

# Check triggers
kubectl get trigger

# Check event source
kubectl get pingsource,apiserversource,kafkasource

# Check logs of the broker
kubectl logs -l eventing.knative.dev/broker=default -n knative-eventing
```
