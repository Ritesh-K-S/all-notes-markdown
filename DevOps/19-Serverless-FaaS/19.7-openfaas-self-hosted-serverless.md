## What Are We Going to Learn?

We just learned about **Knative** — which is powerful but can be complex to set up and manage. What if you want something **simpler**?

**OpenFaaS** (Open Function as a Service) is a **developer-friendly, easy-to-use** serverless platform that runs on Kubernetes (or even Docker Swarm). It's designed with one goal:

> **Make serverless simple.**

If Knative is the "enterprise" option, OpenFaaS is the "developer happiness" option. Let's learn it!

---

## What is OpenFaaS?

**OpenFaaS** is an open-source framework for building and deploying serverless functions and microservices on **Kubernetes** or **Docker Swarm**.

The core idea is dead simple:

1. **Write a function** in any language
2. **Package it** as a Docker container (OpenFaaS does this for you!)
3. **Deploy it** with one command
4. **Call it** via HTTP
5. It **auto-scales** based on demand

```
┌──────────────────────────────────────────────────────┐
│                    OpenFaaS                          │
│                                                      │
│   ┌─────────┐    ┌───────────┐    ┌──────────────┐  │
│   │ faas-cli │───►│  Gateway  │───►│  Functions   │  │
│   │ (CLI)    │    │  (API +   │    │  (Your Code  │  │
│   └─────────┘    │   UI +    │    │   in Docker   │  │
│                  │   Proxy)  │    │   Containers) │  │
│   ┌─────────┐    │           │    │              │  │
│   │ UI      │───►│           │    │  func-1 ──►  │  │
│   │ (Portal)│    └───────────┘    │  func-2 ──►  │  │
│   └─────────┘                     │  func-3 ──►  │  │
│                                   └──────────────┘  │
│                                                      │
│   ┌──────────────┐    ┌──────────────────────────┐  │
│   │  Prometheus  │    │   NATS (Async Queue)     │  │
│   │  (Metrics)   │    │   (for async calls)       │  │
│   └──────────────┘    └──────────────────────────┘  │
├──────────────────────────────────────────────────────┤
│                  Kubernetes                          │
└──────────────────────────────────────────────────────┘
```

## Why OpenFaaS?

### The OpenFaaS Philosophy

OpenFaaS follows these principles:

1. **Developer-first** — Simple CLI, quick feedback loops
2. **Any language** — If it runs in a container, it runs on OpenFaaS
3. **Any infrastructure** — Kubernetes, Docker Swarm, on-prem, cloud, edge
4. **No vendor lock-in** — It's just containers. You own everything.
5. **Community-driven** — Large, active open-source community

### Why Choose OpenFaaS?

| Reason | Details |
|--------|---------|
| **Simplicity** | Easier to set up and use than Knative |
| **Language agnostic** | Write functions in Python, Node.js, Go, Java, C#, Rust, PHP, Ruby, or ANY language |
| **Portable** | Runs on any Kubernetes cluster — cloud, on-prem, edge |
| **Quick start** | From zero to deployed function in under 5 minutes |
| **Built-in UI** | Web portal to manage, deploy, and invoke functions |
| **Async support** | Fire-and-forget async invocations via NATS |
| **Auto-scaling** | Automatic scaling based on requests per second |
| **Scale to zero** | Available in OpenFaaS Pro |
| **Great docs** | Excellent documentation and community |

## OpenFaaS Architecture — How It All Fits Together

Let's break down each component:

```
                    HTTP Request
                         │
                         ▼
              ┌──────────────────────┐
              │      Gateway         │
              │  (API + UI + Proxy)  │
              │                      │
              │  - Routes requests   │
              │  - Collects metrics  │
              │  - Serves UI         │
              │  - Auth/rate limit   │
              └──────────┬───────────┘
                         │
            ┌────────────┼────────────┐
            │            │            │
            ▼            ▼            ▼
     ┌───────────┐ ┌───────────┐ ┌───────────┐
     │ Function  │ │ Function  │ │ Function  │
     │ Pod(s)    │ │ Pod(s)    │ │ Pod(s)    │
     │           │ │           │ │           │
     │ watchdog  │ │ watchdog  │ │ watchdog  │
     │ + your    │ │ + your    │ │ + your    │
     │   code    │ │   code    │ │   code    │
     └───────────┘ └───────────┘ └───────────┘
```

### Key Components

| Component | What It Does |
|-----------|-------------|
| **Gateway** | The front door. Routes HTTP requests to the right function. Also serves the UI and API. |
| **faas-cli** | Command-line tool to create, build, deploy, and manage functions |
| **Function Watchdog** | A tiny HTTP server inside each function container. Receives requests and passes them to your code. |
| **Prometheus** | Collects metrics (invocation count, duration, errors) for auto-scaling |
| **NATS Streaming** | Message queue for async function invocations |
| **AlertManager** | Triggers auto-scaling based on Prometheus metrics |
| **faas-netes** | The Kubernetes provider — manages function pods on K8s |

### The Function Watchdog — The Secret Sauce

Every OpenFaaS function container has a **watchdog** process:

```
┌─────────────────────────────────┐
│         Function Container       │
│                                  │
│  ┌────────────┐   ┌──────────┐  │
│  │  Watchdog  │──►│ Your     │  │
│  │  (HTTP     │   │ Function │  │
│  │   server)  │◄──│ Code     │  │
│  └────────────┘   └──────────┘  │
│                                  │
│  Port 8080                       │
└──────────────────────────────────┘
```

There are two watchdog modes:

| Mode | How It Works | Best For |
|------|-------------|----------|
| **Classic Watchdog** | Forks a new process for each request (stdin/stdout) | Simple scripts, CLI tools |
| **of-watchdog** (recommended) | Keeps your process running, forwards HTTP requests | Web frameworks, APIs, better performance |

## Core Concepts

| Concept | Description |
|---------|-------------|
| **Function** | Your code packaged in a container. Receives HTTP, returns HTTP. |

## Installation & Setup

### Prerequisites

- **Kubernetes cluster** (Minikube, Kind, k3s, or cloud-managed)
- **kubectl** configured
- **Helm** (v3+)
- **Docker** (for building function images)

### Install OpenFaaS on Kubernetes with Helm

```bash
# Step 1: Add the OpenFaaS Helm chart repository
helm repo add openfaas https://openfaas.github.io/faas-netes/
helm repo update

# Step 2: Create the namespaces
kubectl apply -f https://raw.githubusercontent.com/openfaas/faas-netes/master/namespaces.yml

# This creates two namespaces:
# - openfaas        → OpenFaaS core components (Gateway, Prometheus, NATS)
# - openfaas-fn     → Your functions run here
```

```bash
# Step 3: Install OpenFaaS
helm install openfaas openfaas/openfaas \
    --namespace openfaas \
    --set functionNamespace=openfaas-fn \
    --set generateBasicAuth=true \
    --set serviceType=LoadBalancer
```

**What each flag means:**
- `--namespace openfaas` → Install core components in the `openfaas` namespace
- `--set functionNamespace=openfaas-fn` → Functions go in the `openfaas-fn` namespace
- `--set generateBasicAuth=true` → Auto-generate a password for the Gateway
- `--set serviceType=LoadBalancer` → Expose the Gateway via a LoadBalancer (use `NodePort` for local clusters)

```bash
# Step 4: Verify the installation
kubectl get pods -n openfaas

# Expected output:
# NAME                            READY   STATUS    RESTARTS   AGE
# alertmanager-xxxxx              1/1     Running   0          1m
# gateway-xxxxx                   2/2     Running   0          1m
# nats-xxxxx                      1/1     Running   0          1m
# prometheus-xxxxx                1/1     Running   0          1m
# queue-worker-xxxxx              1/1     Running   0          1m
```

### Access the OpenFaaS Gateway

```bash
# Get the auto-generated password
PASSWORD=$(kubectl get secret -n openfaas basic-auth -o jsonpath="{.data.basic-auth-password}" | base64 --decode)

echo "Your OpenFaaS password is: $PASSWORD"

# Get the Gateway URL
# For LoadBalancer:
GATEWAY_URL=$(kubectl get svc gateway-external -n openfaas -o jsonpath='{.status.loadBalancer.ingress[0].ip}')
echo "Gateway URL: http://$GATEWAY_URL:8080"

# For local clusters (port-forward):
kubectl port-forward svc/gateway -n openfaas 8080:8080 &
GATEWAY_URL="127.0.0.1"
echo "Gateway URL: http://127.0.0.1:8080"

# Login to OpenFaaS
echo -n $PASSWORD | faas-cli login --username admin --password-stdin --gateway http://$GATEWAY_URL:8080

# Set the gateway URL as an environment variable (so you don't type it every time)
export OPENFAAS_URL=http://$GATEWAY_URL:8080
```

## Your First Function — Step by Step

### Step 1: Create a Function from a Template

```bash
# List available templates
faas-cli template store list

# Pull the templates you want
faas-cli template store pull python3-http

# Create a new function using the python3-http template
faas-cli new hello-python --lang python3-http
```

### Step 2: Understand the Project Structure

After running `faas-cli new`, you get:

```
.
├── hello-python/           # Your function directory
│   ├── handler.py          # Your function code (this is where you write logic!)
│   └── requirements.txt    # Python dependencies
└── hello-python.yml        # Stack file (configuration)
```

### Step 3: Write Your Function Logic

Edit `hello-python/handler.py`:

```python
def handle(event, context):
    """
    Handle an HTTP request.
    
    Args:
        event: The request object
            - event.body        → Request body (string)
            - event.headers     → Request headers (dict)
            - event.method      → HTTP method (GET, POST, etc.)
            - event.query       → Query parameters (dict)
            - event.path        → Request path (string)
        
        context: Runtime context
            - context.hostname  → Function hostname
    
    Returns:
        A response object with body, status_code, and headers
    """
    import json
    
    # Parse the incoming data
    if event.body:
        try:
            data = json.loads(event.body)
            name = data.get("name", "World")
        except json.JSONDecodeError:
            name = event.body  # If it's plain text, use it as the name
    else:
        name = "World"
    
    # Build the response
    result = {
        "message": f"Hello, {name}! Welcome to OpenFaaS! 🚀",
        "method": event.method,
        "path": event.path,
        "function": context.hostname
    }
    
    return {
        "statusCode": 200,
        "body": json.dumps(result, indent=2),
        "headers": {
            "Content-Type": "application/json"
        }
    }
```

**What's happening here?**
- `event` gives you the HTTP request details (body, headers, method, query params, path)
- `context` gives you runtime info
- You return a dict with `statusCode`, `body`, and optional `headers`
- It's just a simple Python function — nothing fancy!

### Step 4: Build, Push, and Deploy

```bash
# Set your Docker Hub username (or private registry)
export DOCKER_USER=yourusername

# Edit hello-python.yml to use your registry
# Change: image: hello-python:latest
# To:     image: yourusername/hello-python:latest

# Build the function (creates a Docker image)
faas-cli build -f hello-python.yml

# Push to your registry
faas-cli push -f hello-python.yml

# Deploy to OpenFaaS
faas-cli deploy -f hello-python.yml

# Or do all three in one command! 🎉
faas-cli up -f hello-python.yml
# 'up' = build + push + deploy (the most common workflow)
```

**What `faas-cli up` does behind the scenes:**
1. **Build** → Creates a Docker image with your function code + the watchdog
2. **Push** → Pushes the image to your Docker registry
3. **Deploy** → Tells OpenFaaS to create/update a Kubernetes Deployment for your function

### Step 5: Invoke Your Function

```bash
# Method 1: Using faas-cli
faas-cli invoke hello-python
# Type some input and press Ctrl+D
# Or pipe input:
echo '{"name": "DevOps"}' | faas-cli invoke hello-python

# Method 2: Using curl
curl http://$OPENFAAS_URL/function/hello-python \
    -d '{"name": "DevOps"}' \
    -H "Content-Type: application/json"

# Method 3: Using the UI
# Go to http://YOUR_GATEWAY:8080/ui/ and click "Invoke"

# Output:
# {
#   "message": "Hello, DevOps! Welcome to OpenFaaS! 🚀",
#   "method": "POST",
#   "path": "/",
#   "function": "hello-python-xxxxx"
# }
```

## Writing Functions in Different Languages

### Python Function (with Dependencies)

```bash
faas-cli new image-analyzer --lang python3-http
```

**image-analyzer/handler.py:**

```python
import json
import requests
from PIL import Image
from io import BytesIO

def handle(event, context):
    """
    Analyze an image from a URL.
    Expects JSON body: {"url": "https://example.com/image.jpg"}
    """
    try:
        data = json.loads(event.body)
        image_url = data.get("url")
        
        if not image_url:
            return {
                "statusCode": 400,
                "body": json.dumps({"error": "Please provide an 'url' field"})
            }
        
        # Download the image
        response = requests.get(image_url, timeout=10)
        img = Image.open(BytesIO(response.content))
        
        # Analyze it
        result = {
            "format": img.format,
            "mode": img.mode,
            "width": img.size[0],
            "height": img.size[1],
            "megapixels": round((img.size[0] * img.size[1]) / 1_000_000, 2)
        }
        
        return {
            "statusCode": 200,
            "body": json.dumps(result, indent=2),
            "headers": {"Content-Type": "application/json"}
        }
    
    except Exception as e:
        return {
            "statusCode": 500,
            "body": json.dumps({"error": str(e)})
        }
```

```

```

### Go Function

```bash
faas-cli template store pull golang-http
faas-cli new hello-go --lang golang-http
```

**hello-go/handler.go:**

```go
package function

import (
    "encoding/json"
    "fmt"
    "net/http"
    "runtime"
)

// Handle processes incoming HTTP requests
func Handle(w http.ResponseWriter, r *http.Request) {
    // Parse request body
    var input struct {
        Name string `json:"name"`
    }
    
    if r.Body != nil {
        defer r.Body.Close()
        json.NewDecoder(r.Body).Decode(&input)
    }
    
    if input.Name == "" {
        input.Name = "World"
    }
    
    // Build response
    response := map[string]string{
        "message":    fmt.Sprintf("Hello, %s! From OpenFaaS Go! 🔵", input.Name),
        "goVersion":  runtime.Version(),
        "arch":       runtime.GOARCH,
    }
    
    w.Header().Set("Content-Type", "application/json")
    w.WriteHeader(http.StatusOK)
    json.NewEncoder(w).Encode(response)
}
```

### Dockerfile Function (Any Language!)

The `dockerfile` template lets you bring **any** language or runtime:

```bash
faas-cli new my-custom-function --lang dockerfile
```

This creates a directory with a `Dockerfile` that you fully control:

**my-custom-function/Dockerfile:**

```dockerfile
FROM --platform=${TARGETPLATFORM:-linux/amd64} ghcr.io/openfaas/of-watchdog:0.10.1 AS watchdog

FROM python:3.12-slim

# Copy the watchdog binary
COPY --from=watchdog /fwatchdog /usr/bin/fwatchdog
RUN chmod +x /usr/bin/fwatchdog

WORKDIR /app

# Install dependencies
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Copy function code
COPY . .

# Configure the watchdog
ENV fprocess="python app.py"          # Command to run your app
ENV mode="http"                       # HTTP mode (recommended)
ENV upstream_url="http://127.0.0.1:5000"  # Your app's URL
ENV exec_timeout="30s"                # Function timeout

# Health check
HEALTHCHECK --interval=3s CMD [ -e /tmp/.lock ] || exit 1

EXPOSE 8080

CMD ["fwatchdog"]
```

This is great when you need:
- A specific base image
- System-level packages
- Custom build steps
- An existing application that you want to run as a function

## The stack.yml File — Deep Dive

The `stack.yml` (or `function-name.yml`) is your function's configuration:

```yaml
version: 1.0
provider:
  name: openfaas
  gateway: http://127.0.0.1:8080     # OpenFaaS Gateway URL

functions:
  # Function 1: A Python API
  user-api:
    lang: python3-http
    handler: ./user-api                # Directory with your code
    image: yourusername/user-api:latest
    
    # Environment variables
    environment:
      DATABASE_URL: "postgres://db:5432/users"
      LOG_LEVEL: "info"
      write_timeout: "60s"            # Function timeout
      read_timeout: "60s"
      exec_timeout: "60s"
    
    # Kubernetes-specific settings
    labels:
      com.openfaas.scale.min: "1"     # Minimum replicas
      com.openfaas.scale.max: "20"    # Maximum replicas
      com.openfaas.scale.factor: "20" # Scale by 20% each time
      com.openfaas.scale.zero: "true" # Allow scale to zero (Pro only)
    
    # Resource limits
    limits:
      cpu: "200m"
      memory: "256Mi"
    requests:
      cpu: "100m"
      memory: "128Mi"
    
    # Secrets (mounted as files in /var/openfaas/secrets/)
    secrets:
      - db-password
      - api-key
    
    # Annotations (for Kubernetes)
    annotations:
      prometheus.io/scrape: "true"
    
    # Read-only filesystem (security best practice)
    readonly_root_filesystem: true

  # Function 2: A Node.js webhook handler
  webhook-handler:
    lang: node20
    handler: ./webhook-handler
    image: yourusername/webhook-handler:latest
    environment:
      SLACK_WEBHOOK_URL: "https://hooks.slack.com/..."
    labels:
      com.openfaas.scale.min: "1"
      com.openfaas.scale.max: "5"

  # Function 3: A Go image processor
  image-processor:
```

### Async Invocations (Fire and Forget)

Sometimes you don't want to wait for a function to finish. Use **async invocations**:

```bash
# Synchronous (wait for result)
curl http://localhost:8080/function/heavy-task

# Asynchronous (fire and forget — returns immediately with 202 Accepted)
curl http://localhost:8080/async-function/heavy-task

# Async with callback URL (get notified when done)
curl http://localhost:8080/async-function/heavy-task \
    -H "X-Callback-Url: http://my-service/callback"
```

**How async works:**

```
1. Request hits /async-function/heavy-task
2. Gateway puts the request in NATS queue → Returns 202 Accepted immediately
3. Queue worker picks up the request → Invokes the function
4. Function finishes → Result sent to callback URL (if provided)
```

This is perfect for:
- Long-running tasks (image processing, report generation)
- Tasks where you don't need the result immediately
- Decoupling request handling from processing

### Webhook Triggers

OpenFaaS functions are perfect for handling webhooks:

```python
# webhook-handler/handler.py
import json
import hmac
import hashlib

def handle(event, context):
    """Handle a GitHub webhook."""
    
    # Verify the webhook signature (security!)
    signature = event.headers.get("X-Hub-Signature-256", "")
    secret = get_secret("github-webhook-secret")
    
    expected = "sha256=" + hmac.new(
        secret.encode(), event.body.encode(), hashlib.sha256
    ).hexdigest()
    
    if not hmac.compare_digest(signature, expected):
        return {"statusCode": 403, "body": "Invalid signature"}
    
    # Parse the webhook payload
    payload = json.loads(event.body)
    event_type = event.headers.get("X-GitHub-Event")
    
    if event_type == "push":
        branch = payload["ref"].split("/")[-1]
        commits = len(payload.get("commits", []))
        pusher = payload["pusher"]["name"]
        
        message = f"🔔 {pusher} pushed {commits} commit(s) to {branch}"
        # Send to Slack, Discord, etc.
        print(message)
    
    return {
        "statusCode": 200,
        "body": json.dumps({"status": "processed"})
    }

def get_secret(name):
    """Read a secret from the OpenFaaS secrets mount."""
    with open(f"/var/openfaas/secrets/{name}") as f:
        return f.read().strip()
```

## Auto-Scaling

### How Auto-Scaling Works

OpenFaaS uses **Prometheus metrics** + **AlertManager** to auto-scale functions:

```
Prometheus ──► AlertManager ──► Gateway ──► Scale Up/Down
(metrics)      (rules)          (API)       (adjust replicas)
```

1. Prometheus collects `gateway_function_invocation_total` metrics
2. AlertManager fires an alert when the invocation rate exceeds a threshold
3. Gateway receives the alert and adjusts the function's replica count

### Scaling Configuration

```yaml
functions:
  my-function:
    lang: python3-http
    handler: ./my-function
    image: yourusername/my-function:latest
    labels:
      # Minimum replicas (always keep at least 1 running)
      com.openfaas.scale.min: "1"
      
      # Maximum replicas (don't go above 20)
      com.openfaas.scale.max: "20"
      
      # Scaling factor — how aggressively to scale (percentage)
      # 20 means increase by 20% of max each time
      com.openfaas.scale.factor: "20"
      
      # Scale type: rps (requests per second) or capacity
      com.openfaas.scale.type: "rps"
      
      # Target RPS per replica (scale up when exceeded)
      com.openfaas.scale.target: "50"
```

**Example scaling behavior (with above config):**

```
RPS = 0  → Replicas = 1   (min)
RPS = 30 → Replicas = 1   (under target of 50)
RPS = 60 → Replicas = 2   (60/50 = 1.2, round up)
RPS = 200 → Replicas = 4  (200/50 = 4)
RPS = 1000 → Replicas = 20 (capped at max)
```

### Scale to Zero

Scale to zero is available in **OpenFaaS Pro**:

```yaml
functions:
  my-function:
    labels:
      com.openfaas.scale.zero: "true"                  # Enable scale to zero
      com.openfaas.scale.zero-duration: "15m"           # Scale down after 15 min of inactivity
```

When a request comes in for a function with 0 replicas:
1. Gateway detects 0 replicas
2. Scales the function to 1 replica
3. Buffers the request until the pod is ready
4. Forwards the request

- You want a **gentle learning curve**
- You need **self-hosted serverless** without complexity

**Choose Knative when:**
- You need **advanced traffic management** (canary, A/B testing)
- You need **built-in eventing** with CloudEvents

## Best Practices

### 1. Use the `of-watchdog` (HTTP Mode)

```yaml
# The of-watchdog keeps your process running (no fork per request)
# This is the DEFAULT in modern templates — just use them!
functions:
  my-function:
    lang: python3-http   # ← Uses of-watchdog in HTTP mode
```

### 2. Set Proper Timeouts

```yaml
environment:
  read_timeout: "30s"    # Don't set too high — prevents hung connections
  write_timeout: "30s"
  exec_timeout: "30s"
```

### 3. Use Health Checks

```python
# Add a health check endpoint
@app.route('/health')
def health():
    return {"status": "ok"}, 200
```

### 4. Keep Images Small

```dockerfile
# Use slim/alpine base images
FROM python:3.12-slim   # ~120MB instead of ~900MB for python:3.12

# Use multi-stage builds for Go
FROM golang:1.22 AS builder
WORKDIR /app
COPY . .
RUN CGO_ENABLED=0 go build -o handler .

FROM alpine:3.19
COPY --from=builder /app/handler /handler
CMD ["/handler"]
```

### 5. Use Read-Only Filesystem

```yaml
functions:
  my-function:
    readonly_root_filesystem: true   # Prevents writing to the container filesystem
    # Use /tmp for temporary files (it's writable by default)
```

### 6. One Function, One Purpose

```bash
# ❌ BAD — One function doing everything
faas-cli new everything-api --lang python3-http

# ✅ GOOD — Separate functions for separate concerns
faas-cli new create-user --lang python3-http
faas-cli new send-email --lang python3-http
faas-cli new process-payment --lang python3-http
```

```bash
faas-cli describe my-function
```

```bash
# Check function logs
faas-cli logs my-function --follow
```
