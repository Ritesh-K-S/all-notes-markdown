# 20.1 — What are Artifacts? — Types (Docker Images, Helm Charts, JARs, npm Packages)

> **One Line Answer:** An **artifact** is any **output produced by a build process** — it's the "finished product" of your code that is ready to be stored, versioned, shared, and deployed.

## 🤔 Let's Start With a Simple Analogy

Think of a **bakery**:

```
Raw Ingredients (Flour, Sugar, Eggs)
        ↓
    Baking Process
        ↓
    Finished Cake 🎂  ← This is the "artifact"
```

In software, it works exactly the same way:

```
Source Code (your .java, .py, .js files)
        ↓
    Build Process (compile, test, package)
        ↓
    Artifact (JAR file, Docker image, npm package, etc.)  ← This is what gets deployed
```

**You never deploy source code directly to production.** You always deploy **artifacts** — the packaged, ready-to-run versions of your software.

---

## ✅ So What Exactly is an Artifact?

An **artifact** is:

> Any **file or package** produced as a result of the **build/compilation/packaging** process that can be **deployed**, **distributed**, or **consumed** by other systems or applications.

### Key Characteristics of Artifacts:

| Property | What It Means |
|----------|---------------|
| **Immutable** | Once built, it should NEVER be changed. If you need changes, build a new one. |
| **Versioned** | Every artifact has a unique version (e.g., `v1.2.3`, `latest`, `sha256:abc123`) |
| **Reproducible** | Same source code + same build process = same artifact every time |
| **Portable** | Can be moved between environments (dev → staging → production) without changes |
| **Stored** | Kept in an **artifact repository** (like JFrog Artifactory, Nexus, Docker Hub, etc.) |

## 📦 Types of Artifacts (The Complete Picture)

Different programming languages, frameworks, and tools produce different types of artifacts. Let's go through all of them:

---

### 1. 🐳 Docker Images (Container Images)

**What:** A packaged application with everything it needs to run — code, runtime, libraries, OS tools — all bundled into one image.

**Format:** OCI (Open Container Initiative) image format

**File Extension:** No traditional extension — identified by name + tag (e.g., `myapp:v1.2.3`)

**Where They Live:** Container Registries — Docker Hub, AWS ECR, Azure ACR, Google GCR, Harbor, GHCR

**Example — Building a Docker Image:**

```dockerfile
# Dockerfile
FROM node:18-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci --production
COPY . .
EXPOSE 3000
CMD ["node", "server.js"]
```

```bash
# Build the image (this produces the artifact)
docker build -t mycompany/myapp:v1.2.3 .

# Push it to a registry (store the artifact)
docker push mycompany/myapp:v1.2.3

# Pull it anywhere and run (consume the artifact)
docker pull mycompany/myapp:v1.2.3
docker run -p 3000:3000 mycompany/myapp:v1.2.3
```

**Why Docker Images Are So Popular:**
- Contains EVERYTHING needed — no more "it works on my machine"
- Runs the same on any machine that has Docker installed
- Extremely lightweight compared to VMs
- Easy to version and roll back

---

### 2. ☕ JAR / WAR / EAR Files (Java)

**What:** Packaged Java applications

| Type | Full Name | Used For |
|------|-----------|----------|
| **JAR** | Java Archive | Standalone Java apps, libraries |
| **WAR** | Web Application Archive | Java web applications (deployed on Tomcat, JBoss, etc.) |
| **EAR** | Enterprise Application Archive | Large enterprise Java apps (contains JARs + WARs) |

**How They're Built:**

```bash
# Using Maven (most common Java build tool)
mvn clean package
# Output: target/myapp-1.0.0.jar

# Using Gradle
gradle build
# Output: build/libs/myapp-1.0.0.jar
```

**What's Inside a JAR File:**

```
myapp-1.0.0.jar
├── META-INF/
│   └── MANIFEST.MF          ← Metadata (main class, version, etc.)
├── com/
│   └── mycompany/
│       └── app/
│           ├── Main.class    ← Compiled Java bytecode
│           └── Utils.class
└── application.properties    ← Configuration
```

**Running a JAR:**

```bash
java -jar myapp-1.0.0.jar
```

---

### 3. 📦 npm Packages (Node.js / JavaScript)

**What:** Packaged JavaScript/TypeScript libraries or applications

**Where They Live:** npm Registry (npmjs.com), private registries (Artifactory, Verdaccio)

**Key File:** `package.json` — the "identity card" of every npm package

```json
{
  "name": "@mycompany/my-library",
  "version": "2.1.0",
  "main": "dist/index.js",
  "scripts": {
    "build": "tsc",
    "test": "jest"
  }
}
```

**Publishing an npm Package:**

```bash
# Build the package
npm run build

# Login to registry
npm login

# Publish to npm registry (this uploads the artifact)
npm publish

# Others can now install it
npm install @mycompany/my-library@2.1.0
```

**What Gets Published:**

```
my-library-2.1.0.tgz         ← This is the actual artifact (tarball)
├── package.json
├── dist/
│   ├── index.js
│   └── index.d.ts
└── README.md
```

---

### 4. 🐍 Python Packages (Wheels / sdist)

**What:** Packaged Python libraries or applications

**Formats:**

| Format | Extension | What It Is |
|--------|-----------|------------|
| **Wheel** | `.whl` | Pre-built binary package (fast to install) |
| **sdist** | `.tar.gz` | Source distribution (needs to be built during install) |

**Where They Live:** PyPI (pypi.org), private registries (Artifactory, Nexus, DevPI)

**Building Python Packages:**

```bash
# Build the package
python -m build

# Output:
# dist/mypackage-1.0.0-py3-none-any.whl    ← Wheel (preferred)
# dist/mypackage-1.0.0.tar.gz              ← Source distribution
```

**Publishing to PyPI:**

```bash
# Upload to PyPI
twine upload dist/*

# Others can install it
pip install mypackage==1.0.0
```

---

### 5. ⎈ Helm Charts (Kubernetes)

**What:** Packaged Kubernetes application definitions — a collection of YAML templates that define all the Kubernetes resources your app needs.

**Format:** `.tgz` (a tarball containing the chart directory)

**Where They Live:** Helm Repositories (ChartMuseum, OCI registries, Artifact Hub)

**Structure of a Helm Chart:**

```
mychart/
├── Chart.yaml          ← Metadata (name, version, description)
├── values.yaml         ← Default configuration values
├── templates/
│   ├── deployment.yaml ← Kubernetes Deployment template
│   ├── service.yaml    ← Kubernetes Service template
│   ├── ingress.yaml    ← Kubernetes Ingress template
│   └── _helpers.tpl    ← Template helper functions
└── charts/             ← Dependencies (sub-charts)
```

**Packaging and Publishing:**

```bash
# Package the chart (creates the artifact)
helm package mychart/
# Output: mychart-1.0.0.tgz

# Push to a Helm repository
helm push mychart-1.0.0.tgz oci://registry.mycompany.com/charts

# Others can install it
helm install my-release oci://registry.mycompany.com/charts/mychart --version 1.0.0
```

## 🔄 Artifacts in the DevOps Lifecycle

Here's where artifacts fit in the CI/CD pipeline:

```
┌─────────────────────────────────────────────────────────────────────┐
│                        CI/CD Pipeline                               │
│                                                                     │
│  Source Code    Build &      Store         Deploy        Monitor    │
│  (Git Repo) → Compile  →  Artifact   →  to Target  →  in Prod    │
│     │           │        Repository        │              │        │
│     │           │            │              │              │        │
│  git push    maven/npm/   Artifactory/   Kubernetes/     Prometheus│
│              docker build  Nexus/ECR      EC2/ECS        Grafana   │
│                              │                                     │
│                     ┌────────┴────────┐                            │
│                     │   Artifacts     │                            │
│                     │  stored here    │                            │
│                     │  with versions  │                            │
│                     └─────────────────┘                            │
└─────────────────────────────────────────────────────────────────────┘
```

### Step-by-Step Flow:

| Step | What Happens | Example |
|------|-------------|---------|
| 1. **Code** | Developer pushes code to Git | `git push origin main` |
| 2. **Build** | CI server compiles/builds the code | `docker build -t myapp:v1.2.3 .` |
| 3. **Test** | Automated tests run | `npm test`, `pytest` |
| 4. **Package** | Build output is packaged into an artifact | `myapp:v1.2.3` image created |
| 5. **Store** | Artifact is pushed to a repository | `docker push registry.com/myapp:v1.2.3` |
| 6. **Deploy** | Artifact is pulled and deployed to an environment | `helm upgrade --install myapp` |
| 7. **Promote** | Same artifact moves dev → staging → production | **Same image**, different config |

## 🏪 What is an Artifact Repository?

An **artifact repository** (also called artifact manager or registry) is a specialized storage system designed to:

| Feature | What It Does |
|---------|-------------|
| **Store** | Keep all your artifacts in one central place |
| **Version** | Track every version of every artifact |
| **Distribute** | Serve artifacts to CI/CD pipelines and deployment targets |
| **Secure** | Control who can upload, download, or delete artifacts |
| **Cache** | Proxy and cache artifacts from public registries (npm, PyPI, Docker Hub) |
| **Scan** | Check artifacts for security vulnerabilities |
| **Cleanup** | Automatically delete old or unused artifacts to save storage |

### Popular Artifact Repositories:

| Repository | Type | Best For |
|------------|------|----------|
| **JFrog Artifactory** | Universal | Supports ALL artifact types (Docker, Maven, npm, PyPI, Helm, etc.) |
| **Sonatype Nexus** | Universal | Open-source, supports most artifact types |
| **Docker Hub** | Container only | Public Docker images |
| **AWS ECR** | Container only | AWS-native container registry |
| **Azure ACR** | Container only | Azure-native container registry |
| **Google GCR / GAR** | Container + others | Google Cloud artifact storage |
| **GitHub Packages (GHCR)** | Multi-format | npm, Docker, Maven, NuGet, RubyGems |
| **Harbor** | Container + Helm | Open-source, enterprise container registry |
| **Verdaccio** | npm only | Lightweight private npm registry |
| **PyPI** | Python only | Public Python packages |

## 🎯 Why Artifact Management Matters

### Without Proper Artifact Management:

```
❌ "Which version of the app is running in production?"  → Nobody knows
❌ "Let's roll back to the last working version"         → Can't find it
❌ "Did the artifact pass security scanning?"             → Never scanned
❌ "The build is slow because it downloads everything"    → No caching
❌ "Someone uploaded a malicious package"                 → No access control
❌ "We're out of storage"                                 → No cleanup policy
```

### With Proper Artifact Management:

```
✅ Every artifact has a version, a build ID, and metadata
✅ You can roll back to ANY previous version in seconds
✅ Every artifact is scanned for vulnerabilities before deployment
✅ Builds are fast because dependencies are cached locally
✅ Only authorized users/pipelines can push artifacts
✅ Old artifacts are automatically cleaned up based on retention policies
```

## 🔑 Key Concepts You Must Know

### 1. Immutability

> **Once an artifact is built and versioned, NEVER overwrite it.**

```
✅ Good:  myapp:v1.0.0, myapp:v1.0.1, myapp:v1.1.0 (new version = new artifact)
❌ Bad:   myapp:latest (overwriting the same tag over and over — you can't tell what's deployed)
```

Why? Because if you overwrite `v1.0.0` with new code, you can't reproduce what was running in production at a specific time.

### 2. Semantic Versioning (SemVer)

```
MAJOR.MINOR.PATCH
  │      │     │
  │      │     └── Bug fixes (backward compatible)
  │      └──────── New features (backward compatible)
  └─────────────── Breaking changes

Examples:
  1.0.0 → First stable release
  1.1.0 → New feature added (no breaking changes)
  1.1.1 → Bug fix
  2.0.0 → Breaking change (upgrade with caution!)
```

### 3. Artifact Promotion

> Move the **same artifact** through environments — don't rebuild for each environment.

```
                    SAME ARTIFACT
                         │
    ┌────────────────────┼────────────────────┐
    │                    │                    │
    ▼                    ▼                    ▼
┌──────────┐      ┌──────────┐      ┌──────────┐
│   DEV    │ ───→ │ STAGING  │ ───→ │   PROD   │
│          │      │          │      │          │
│ v1.2.3   │      │ v1.2.3   │      │ v1.2.3   │
└──────────┘      └──────────┘      └──────────┘
     ✅ Same image, different configuration (env variables)
     ❌ Never rebuild for a different environment
```

### 4. Artifact Metadata

Every artifact should carry metadata:

```json
{
  "name": "myapp",
  "version": "1.2.3",
  "build_number": "456",
  "git_commit": "a1b2c3d",
  "git_branch": "main",
  "built_by": "jenkins-pipeline",
  "built_at": "2025-01-15T10:30:00Z",
  "vulnerabilities_scanned": true,
  "scan_result": "clean"
}
```

This metadata helps you answer:
- **What** is this artifact?
- **When** was it built?
- **Who** (or what pipeline) built it?
- **Which** commit does it correspond to?
- **Is it safe** to deploy?

## 🛡️ Artifact Security Best Practices

| Practice | Why |
|----------|-----|
| **Scan every artifact** | Check for known vulnerabilities (CVEs) before deployment |
| **Sign artifacts** | Use digital signatures to prove authenticity (cosign, Notary) |
| **Use private registries** | Don't pull everything from public internet in production |
| **Minimal base images** | Use `alpine`, `distroless`, or `scratch` to reduce attack surface |
| **No secrets in artifacts** | Never bake passwords, API keys, or tokens into images |
| **Retention policies** | Auto-delete old artifacts to save storage and reduce exposure |
| **RBAC** | Control who can push/pull which artifacts |

## 📝 Quick Summary

| Concept | Key Takeaway |
|---------|-------------|
| **Artifact** | Output of a build process — the deployable unit |
| **Types** | Docker images, JARs, npm packages, Python wheels, Helm charts, DEBs, RPMs, etc. |
| **Repository** | Where artifacts are stored, versioned, and distributed |
| **Immutability** | Never modify a published artifact — always create a new version |
| **Promotion** | Move the SAME artifact across environments — don't rebuild |
| **Metadata** | Tag artifacts with build info, commit hash, scan results |
| **Security** | Scan, sign, use private registries, apply RBAC |
