## 🤔 What is Sonatype Nexus?

Sonatype Nexus (often just called "Nexus") is a **repository manager** — a tool that stores your build artifacts and proxies external package registries.

Think of it like this:

```
Without Nexus:
┌──────────────┐         🌐 Internet
│  Your Build  │ ──────→ Download from Maven Central, npm, Docker Hub every time
│   Server     │ ──────→ Slow, unreliable, no control
└──────────────┘

With Nexus:
┌──────────────┐     ┌─────────────┐         🌐 Internet
│  Your Build  │ ──→ │   Nexus     │ ──────→ First time: fetches from internet
│   Server     │ ←── │  (Proxy +   │ ←────── Caches it locally
│              │     │   Storage)  │         Next time: serves from cache ⚡
└──────────────┘     └─────────────┘
```

## 🏗️ Nexus Architecture

```
┌─────────────────────────────────────────────────────────────────┐
│                    Sonatype Nexus Repository Manager             │
│                                                                  │
│  ┌────────────────────────────────────────────────────────┐     │
│  │                  Web UI (Port 8081)                     │     │
│  │          Browser-based administration panel             │     │
│  └────────────────────────┬───────────────────────────────┘     │
│                           │                                      │
│  ┌────────────────────────▼───────────────────────────────┐     │
│  │              Repository Manager Engine                  │     │
│  │                                                         │     │
│  │  ┌──────────┐  ┌──────────┐  ┌──────────┐            │     │
│  │  │  Hosted   │  │  Proxy   │  │  Group   │            │     │
│  │  │  Repos    │  │  Repos   │  │  Repos   │            │     │
│  │  └──────────┘  └──────────┘  └──────────┘            │     │
│  └────────────────────────────────────────────────────────┘     │
│                                                                  │
│  ┌────────────────────────────────────────────────────────┐     │
│  │              Blob Store (Storage Layer)                 │     │
│  │     File System  |  S3  |  Azure Blob  |  Google GCS  │     │
│  └────────────────────────────────────────────────────────┘     │
│                                                                  │
│  ┌────────────────────────────────────────────────────────┐     │
│  │          OrientDB (Embedded Metadata Database)         │     │
│  │       (Nexus 3.x uses embedded DB by default)          │     │
│  └────────────────────────────────────────────────────────┘     │
└─────────────────────────────────────────────────────────────────┘
```

### Key Components:

| Component | What It Does |
|-----------|-------------|
| **Web UI** | Browser dashboard for managing repos, users, and settings (runs on port 8081) |
| **REST API** | Programmatic access for CI/CD pipelines and automation |
| **Repository Engine** | Manages the three repo types: Hosted, Proxy, Group |
| **Blob Store** | Physical storage where artifact binary data lives |
| **OrientDB** | Internal embedded database for metadata, indexes, and config |

## 📂 Repository Types in Nexus

Nexus has **3 repository types** (similar to Artifactory but with different names):

### 1. Hosted Repository

> **Same as Artifactory's "Local" repo.** This is where YOU store YOUR artifacts.

```
┌───────────────────────┐
│   Hosted Repository   │
│                       │
│  Your CI/CD pipeline  │
│  uploads artifacts    │───→  myapp-1.0.0.jar
│  here                 │───→  myapp:v1.2.3 (Docker)
│                       │───→  my-lib-2.0.0.tgz (npm)
└───────────────────────┘
```

**Common Hosted Repos:**
- `maven-releases` — released Java artifacts
- `maven-snapshots` — development/snapshot Java artifacts
- `docker-hosted` — your Docker images
- `npm-hosted` — your npm packages
- `pypi-hosted` — your Python packages

### 2. Proxy Repository

> **Same as Artifactory's "Remote" repo.** Caches artifacts from external public registries.

```
┌───────────────────────┐         ┌──────────────────┐
│   Proxy Repository    │         │  Public Registry  │
│   (Cache + Proxy)     │ ──────→ │  (Maven Central,  │
│                       │ ←────── │   npmjs.com,      │
│   First time:         │         │   Docker Hub)     │
│   fetches + caches    │         └──────────────────┘
│                       │
│   Next time:          │
│   serves from cache ⚡│
└───────────────────────┘
```

**Common Proxy Repos:**
- `maven-central` — proxy for Maven Central
- `npm-proxy` — proxy for npmjs.com
- `docker-hub-proxy` — proxy for Docker Hub
- `pypi-proxy` — proxy for pypi.org

### 3. Group Repository

> **Same as Artifactory's "Virtual" repo.** Combines multiple repos under one URL.

```
┌───────────────────────────────┐
│      Group Repository         │
│      (maven-public)           │
│                               │
│  Combines:                    │
│  ├── maven-releases (hosted)  │
│  ├── maven-snapshots (hosted) │
│  └── maven-central (proxy)    │
│                               │
│  One URL for everything:      │
│  http://nexus:8081/repository/│
│         maven-public/         │
└───────────────────────────────┘
```

### Comparison with Artifactory Terms:

| Nexus Term | Artifactory Term | Purpose |
|------------|-----------------|---------|
| **Hosted** | **Local** | Store your own artifacts |
| **Proxy** | **Remote** | Cache external public repos |
| **Group** | **Virtual** | Combine multiple repos into one URL |

## 🚀 Installing Nexus

### Option 1: Docker (Easiest — Perfect for Learning)

```bash
# Pull Nexus 3
docker pull sonatype/nexus3:latest

# Run Nexus
docker run -d \
  --name nexus \
  -p 8081:8081 \
  -v nexus-data:/nexus-data \
  sonatype/nexus3:latest
```

```
After starting (takes 1-2 minutes to fully boot):
- Web UI:   http://localhost:8081
- Username: admin
- Password: Found inside the container:
  docker exec nexus cat /nexus-data/admin.password
  
  (You'll be prompted to change it on first login)
```

### Option 2: Docker Compose (Production-like)

```yaml
# docker-compose.yml
version: '3'
services:
  nexus:
    image: sonatype/nexus3:latest
    container_name: nexus
    ports:
      - "8081:8081"     # Web UI + Maven/npm/PyPI
      - "8082:8082"     # Docker registry (hosted)
      - "8083:8083"     # Docker registry (group)
    volumes:
      - nexus-data:/nexus-data
    environment:
      - INSTALL4J_ADD_VM_PARAMS=-Xms512m -Xmx2g -XX:MaxDirectMemorySize=2g
    restart: always

volumes:
  nexus-data:
```

```bash
docker compose up -d

# Wait for it to start, then get the admin password
docker exec nexus cat /nexus-data/admin.password
```

> ⚠️ **Note:** Nexus needs **separate ports** for Docker registries. Port 8081 is for the Web UI and Maven/npm/PyPI. You need additional ports (8082, 8083) for Docker push/pull. We'll configure this below.

### Option 3: Kubernetes (Helm)

```bash
# Add Sonatype Helm repo
helm repo add sonatype https://sonatype.github.io/helm3-charts/
helm repo update

# Install Nexus
helm install nexus sonatype/nexus-repository-manager \
  --namespace nexus \
  --create-namespace \
  --set persistence.enabled=true \
  --set persistence.storageSize=50Gi
```

### Option 4: Direct Installation (Linux)

```bash
# Download Nexus
wget https://download.sonatype.com/nexus/3/nexus-3.68.0-04-unix.tar.gz

# Extract
tar -xzf nexus-3.68.0-04-unix.tar.gz

# Create a dedicated user
sudo useradd -r -m -d /opt/nexus -s /bin/bash nexus

# Move files
sudo mv nexus-3.68.0-04 /opt/nexus
sudo mv sonatype-work /opt/sonatype-work
sudo chown -R nexus:nexus /opt/nexus /opt/sonatype-work

# Configure to run as nexus user
echo 'run_as_user="nexus"' | sudo tee /opt/nexus/bin/nexus.rc

# Start Nexus
sudo -u nexus /opt/nexus/bin/nexus start

# Check status
sudo -u nexus /opt/nexus/bin/nexus status
```

**Create a systemd Service (for auto-start):**

```ini
# /etc/systemd/system/nexus.service
[Unit]
Description=Sonatype Nexus Repository Manager
After=network.target

[Service]
Type=forking
User=nexus
Group=nexus
ExecStart=/opt/nexus/bin/nexus start
ExecStop=/opt/nexus/bin/nexus stop
Restart=on-failure
LimitNOFILE=65536

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl daemon-reload
sudo systemctl enable nexus
sudo systemctl start nexus
```

## 📦 Setting Up Repositories — Step by Step

### Creating a Maven Hosted Repository (Web UI)

1. Go to **⚙️ Server Administration → Repositories → Create Repository**
2. Select **maven2 (hosted)**
3. Configure:
   - **Name:** `maven-releases`
   - **Version Policy:** `Release` (only release versions allowed)
   - **Layout Policy:** `Strict`
   - **Deployment Policy:** `Disable Redeploy` (immutability!)
   - **Blob Store:** `default`
4. Click **Create Repository**

Repeat for snapshots:
- **Name:** `maven-snapshots`
- **Version Policy:** `Snapshot`
- **Deployment Policy:** `Allow Redeploy`

### Creating a Maven Proxy Repository

1. Select **maven2 (proxy)**
2. Configure:
   - **Name:** `maven-central`
   - **Remote URL:** `https://repo1.maven.org/maven2/`
   - **Cache Timeout:** `1440 minutes` (24 hours)
3. Click **Create Repository**

### Creating a Maven Group Repository

1. Select **maven2 (group)**
2. Configure:
   - **Name:** `maven-public`
   - **Member Repos:** Add `maven-releases`, `maven-snapshots`, `maven-central`
   - **Order matters!** Hosted repos first, then proxy repos
3. Click **Create Repository**

## 🐳 Setting Up Docker Registry in Nexus

Docker in Nexus requires a bit more configuration because Docker needs its own dedicated port (it can't share the main Nexus port).

### Step 1: Create a Docker Hosted Repository

1. **Create Repository → docker (hosted)**
2. Configure:
   - **Name:** `docker-hosted`
   - **HTTP Port:** `8082` (dedicated port for Docker push)
   - **Enable Docker V1 API:** ❌ (leave unchecked, V2 is standard)
   - **Blob Store:** `default`
3. Click **Create Repository**

### Step 2: Create a Docker Proxy Repository (for Docker Hub)

1. **Create Repository → docker (proxy)**
2. Configure:
   - **Name:** `docker-hub-proxy`
   - **Remote URL:** `https://registry-1.docker.io`
   - **Docker Index:** `Use Docker Hub`
   - **Enable Docker V1 API:** ❌
3. Click **Create Repository**

### Step 3: Create a Docker Group Repository

1. **Create Repository → docker (group)**
2. Configure:
   - **Name:** `docker-group`
   - **HTTP Port:** `8083` (dedicated port for Docker pull)
   - **Member Repos:** `docker-hosted`, `docker-hub-proxy`
3. Click **Create Repository**

### Step 4: Configure Docker Client

```bash
# For HTTP (non-TLS — development only!)
# Add to /etc/docker/daemon.json
{
  "insecure-registries": ["nexus.mycompany.com:8082", "nexus.mycompany.com:8083"]
}

# Restart Docker
sudo systemctl restart docker
```

### Step 5: Use Docker with Nexus

```bash
# Login to Nexus Docker registry
docker login nexus.mycompany.com:8082
# Enter your Nexus username and password

# Tag your image
docker tag myapp:v1.0.0 nexus.mycompany.com:8082/myapp:v1.0.0

# Push to Nexus (uses hosted repo on port 8082)
docker push nexus.mycompany.com:8082/myapp:v1.0.0

# Pull from Nexus (uses group repo on port 8083)
docker pull nexus.mycompany.com:8083/myapp:v1.0.0

# Pull public images THROUGH Nexus (cached from Docker Hub)
docker pull nexus.mycompany.com:8083/library/nginx:latest
```

### Why Separate Ports?

```
Port 8081 — Nexus Web UI + Maven/npm/PyPI APIs
Port 8082 — Docker PUSH (hosted repository)
Port 8083 — Docker PULL (group repository — combines hosted + proxy)

Docker protocol doesn't support path-based routing well,
so each Docker repo needs its own port or subdomain.
```

## 📦 Setting Up npm Registry in Nexus

### Create npm Repositories

```
1. npm (hosted)  → Name: npm-hosted
2. npm (proxy)   → Name: npm-proxy, Remote URL: https://registry.npmjs.org
3. npm (group)   → Name: npm-group, Members: npm-hosted + npm-proxy
```

### Configure npm Client

```bash
# Set Nexus as your npm registry
npm config set registry http://nexus.mycompany.com:8081/repository/npm-group/

# Login for publishing
npm login --registry=http://nexus.mycompany.com:8081/repository/npm-hosted/

# Install packages (goes through group repo)
npm install express    # Fetched via npm-proxy (cached from npmjs.com)

# Publish your package (goes to hosted repo)
npm publish --registry=http://nexus.mycompany.com:8081/repository/npm-hosted/
```

### Using `.npmrc` File (Project-Level Config)

```ini
# .npmrc (in your project root)
registry=http://nexus.mycompany.com:8081/repository/npm-group/
//nexus.mycompany.com:8081/repository/npm-hosted/:_auth=BASE64_ENCODED_USER:PASS
always-auth=true
```

## 🐍 Setting Up PyPI Registry in Nexus

### Create PyPI Repositories

```
1. pypi (hosted) → Name: pypi-hosted
2. pypi (proxy)  → Name: pypi-proxy, Remote URL: https://pypi.org
3. pypi (group)  → Name: pypi-group, Members: pypi-hosted + pypi-proxy
```

### Configure pip

```ini
# ~/.pip/pip.conf (Linux) or %APPDATA%\pip\pip.ini (Windows)
[global]
index-url = http://nexus.mycompany.com:8081/repository/pypi-group/simple
trusted-host = nexus.mycompany.com
```

### Upload Python Packages

```ini
# ~/.pypirc
[distutils]
index-servers =
    nexus

[nexus]
repository = http://nexus.mycompany.com:8081/repository/pypi-hosted/
username = your-username
password = your-password
```

```bash
# Upload using twine
twine upload -r nexus dist/*

# Install from Nexus
pip install mypackage --index-url http://nexus.mycompany.com:8081/repository/pypi-group/simple
```

## ⎈ Setting Up Helm Repository in Nexus

### Create Helm Repository

```
1. helm (hosted) → Name: helm-hosted

```

### Use Helm with Nexus

```bash
# Add Nexus as a Helm repo

helm repo update
helm search repo nexus/mychart
helm install myrelease nexus/mychart --version 1.0.0
```

## 🔐 Security & Access Control

### Roles and Privileges

Nexus uses a **Role-Based Access Control (RBAC)** model:

```
Privileges → Roles → Users
    │           │        │
    │           │        └── Individual accounts
    │           └──────────── Collection of privileges
    └──────────────────────── Individual permissions (read, write, delete)
```

### Creating a User (Web UI)

1. Go to **⚙️ Administration → Security → Users**
2. Click **Create Local User**
3. Fill in:
   - **ID:** `john`
   - **First Name:** `John`
   - **Email:** `john@company.com`
   - **Password:** (set a strong password)
   - **Status:** `Active`
   - **Roles:** Assign appropriate roles

### Built-in Roles:

| Role | What It Can Do |
|------|---------------|
| `nx-admin` | Full admin access — everything |
| `nx-anonymous` | Read-only access (for anonymous browsing) |

### Creating Custom Roles:

1. Go to **⚙️ Administration → Security → Roles**
2. Click **Create Role**
3. Add privileges like:
   - `nx-repository-view-docker-docker-hosted-*` — access to Docker hosted repo
   - `nx-repository-view-maven2-*-read` — read access to all Maven repos
   - `nx-repository-view-npm-*-*` — full access to all npm repos

### Using REST API for User Management:

```bash
# Create a user via API
curl -X POST \
  -H "Content-Type: application/json" \
  -u admin:password \
  "http://localhost:8081/service/rest/v1/security/users" \
  -d '{
    "userId": "john",
    "firstName": "John",
    "lastName": "Doe",
    "emailAddress": "john@company.com",
    "password": "secure-password",
    "status": "active",
    "roles": ["nx-developer"]
  }'
```

### Realms (Authentication Methods)

Nexus supports multiple authentication "realms":

| Realm | Purpose |
|-------|---------|
| **Local Authenticating Realm** | Username/password stored in Nexus (default) |
| **LDAP Realm** | Authenticate against Active Directory / LDAP |
| **Docker Bearer Token Realm** | Required for Docker authentication — **enable this!** |
| **npm Bearer Token Realm** | Token-based auth for npm |
| **Rut Auth Realm** | Authenticate via HTTP headers (for reverse proxy SSO) |

> ⚠️ **Important:** For Docker to work, you MUST enable the **Docker Bearer Token Realm** under Administration → Security → Realms.

## 🔍 Nexus REST API

Nexus has a comprehensive REST API for automation:

### Common API Endpoints:

```bash
# List all repositories
curl -u admin:password \
  "http://localhost:8081/service/rest/v1/repositories"

# Get repository details
curl -u admin:password \
  "http://localhost:8081/service/rest/v1/repositories/maven-releases"

# Search for components (artifacts)
curl -u admin:password \
  "http://localhost:8081/service/rest/v1/search?repository=maven-releases&name=myapp"

# Search for components by Maven coordinates
curl -u admin:password \
  "http://localhost:8081/service/rest/v1/search?repository=maven-releases&maven.groupId=com.mycompany&maven.artifactId=myapp"

# List components in a repository
curl -u admin:password \
  "http://localhost:8081/service/rest/v1/components?repository=maven-releases"

# Upload a component (Maven)
curl -u admin:password \
  -X POST \
  "http://localhost:8081/service/rest/v1/components?repository=maven-releases" \
  -F "maven2.groupId=com.mycompany" \
  -F "maven2.artifactId=myapp" \
  -F "maven2.version=1.0.0" \
  -F "maven2.asset1=@target/myapp-1.0.0.jar" \
  -F "maven2.asset1.extension=jar"

# Delete a component
curl -u admin:password \
  -X DELETE \
  "http://localhost:8081/service/rest/v1/components/COMPONENT_ID"

# System status
curl "http://localhost:8081/service/rest/v1/status"

# System health check
curl "http://localhost:8081/service/rest/v1/status/check"
```

### API via Script (Groovy Scripting — Advanced)

Nexus also supports Groovy scripts for advanced automation:

```bash
# Upload a script
curl -u admin:password \
  -X POST \
  -H "Content-Type: application/json" \
  "http://localhost:8081/service/rest/v1/script" \
  -d '{
    "name": "list-repos",
    "type": "groovy",
    "content": "repository.repositoryManager.browse().collect { it.name }"
  }'

# Run the script
curl -u admin:password \
  -X POST \
  "http://localhost:8081/service/rest/v1/script/list-repos/run"
```

## 🧹 Cleanup & Maintenance

### Cleanup Policies

Nexus has built-in cleanup policies to automatically delete old artifacts:

1. Go to **⚙️ Administration → Repository → Cleanup Policies**
2. Click **Create Cleanup Policy**
3. Configure:
   - **Name:** `delete-old-snapshots`
   - **Format:** `maven2`
   - **Criteria:**
     - Component Age: older than `30 days`
     - Last Downloaded: more than `15 days` ago
4. Attach the policy to a repository (edit the repo and select the cleanup policy)

### Running Cleanup Tasks

1. Go to **⚙️ Administration → System → Tasks**
2. Find **"Admin - Cleanup repositories using their associated policies"**
3. Run it manually or schedule it (e.g., daily at 2 AM)

### Compact Blob Store (Reclaim Disk Space)

After cleanup, Nexus marks artifacts as deleted but doesn't immediately free disk space. You need to compact:

1. Go to **Tasks**
2. Find **"Admin - Compact blob store"**
3. Run it after cleanup to actually free disk space

### Common Scheduled Tasks:

| Task | Purpose | Schedule |
|------|---------|----------|
| **Cleanup repositories** | Delete artifacts matching cleanup policies | Daily, 2 AM |
| **Compact blob store** | Reclaim disk space after cleanup | Daily, 3 AM |
| **Docker - Delete unused manifests** | Clean up Docker layers | Weekly |
| **Docker - Delete incomplete uploads** | Remove failed uploads | Daily |
| **Rebuild repository index** | Fix search/browse issues | Weekly |

## 🔗 Nexus in CI/CD Pipelines

### Jenkins Integration

```groovy
// Jenkinsfile
pipeline {
    agent any
    
    environment {
        NEXUS_URL = 'http://nexus.mycompany.com:8081'
        NEXUS_CREDENTIALS = credentials('nexus-credentials')
    }
    
    stages {
        stage('Build') {
            steps {
                sh 'mvn clean package -DskipTests'
            }
        }
        
        stage('Publish to Nexus') {
            steps {
                sh """
```

## 🔧 Nexus Configuration Tips & Best Practices

### 1. Blob Store Configuration

By default, all repos share one blob store. For better management, create separate blob stores:

```
blob-store-docker   → For Docker images (can get very large)
blob-store-maven    → For Maven artifacts
blob-store-npm      → For npm packages
default             → For everything else
```

**Why?** Different blob stores can use different storage backends (local disk for small repos, S3 for Docker images).

### 2. Content Selectors (Access Control)

Content selectors let you create fine-grained access rules:

```
Example: Allow team-a to only access their own namespace

Content Selector:
  Name: team-a-docker
  Type: CSEL
  Expression: path =^ "/v2/team-a/"
  
Then create a privilege using this selector and assign it to a role.
```

### 3. Reverse Proxy with Nginx (Production Setup)

In production, put Nginx in front of Nexus:

```nginx
# /etc/nginx/conf.d/nexus.conf
server {
    listen 443 ssl;
    server_name nexus.mycompany.com;
    
    ssl_certificate /etc/ssl/certs/nexus.crt;
    ssl_certificate_key /etc/ssl/private/nexus.key;
    
    # Nexus Web UI + Maven/npm/PyPI
    location / {
        proxy_pass http://localhost:8081;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        client_max_body_size 1G;  # Important for large uploads!
    }
}

# Docker Registry
server {
    listen 443 ssl;
    server_name docker.mycompany.com;
    
    ssl_certificate /etc/ssl/certs/nexus.crt;
    ssl_certificate_key /etc/ssl/private/nexus.key;
    
    location / {
        proxy_pass http://localhost:8082;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        client_max_body_size 10G;  # Docker images can be large!
    }
}
```

### 4. Memory Configuration

Nexus needs adequate memory. Configure JVM settings:

```bash
# /opt/nexus/bin/nexus.vmoptions (or via INSTALL4J_ADD_VM_PARAMS env var)
-Xms1g          # Minimum heap size
-Xmx4g          # Maximum heap size (recommended: 4GB+)
-XX:MaxDirectMemorySize=4g
-XX:+UseG1GC
-Djava.util.prefs.userRoot=/nexus-data/javaprefs
```

## 📝 Quick Summary

| Concept | Key Takeaway |
|---------|-------------|
| **What is Nexus?** | Open-source artifact repository manager — stores, proxies, and distributes artifacts |
| **Hosted Repo** | Your private storage (= Artifactory's Local Repo) |
| **Proxy Repo** | Cache for public registries (= Artifactory's Remote Repo) |
| **Group Repo** | Combines Hosted + Proxy under one URL (= Artifactory's Virtual Repo) |
| **Docker Setup** | Needs dedicated ports (separate from Web UI port) |
| **Free OSS Version** | Supports Docker, Maven, npm, PyPI, Helm — great value! |
| **Cleanup** | Use Cleanup Policies + Compact Blob Store tasks |
| **Security** | RBAC with Privileges → Roles → Users + Realms for auth methods |
| **CI/CD** | REST API + native tool configs (Maven settings.xml, .npmrc, pip.conf) |

## 🆚 When to Choose Nexus vs Artifactory?

| Choose Nexus When... | Choose Artifactory When... |
|---------------------|--------------------------|
| You need a **free** tool with Docker/npm/PyPI support | You need **30+ package formats** |
| Budget is a concern | Enterprise-grade features are required |
| Simple setup for small-medium teams | Multi-site replication (federated repos) |
| You prefer open-source software | You need built-in security scanning (Xray) |
| Standard CI/CD integration is enough | Deep CI/CD integration (build info, promotion) |
