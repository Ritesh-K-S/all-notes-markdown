# 20.4 — Container Registries — Docker Hub, Harbor, ECR, ACR, GCR, GHCR

> **One Line Answer:** A Container Registry is a **storage and distribution system** for container images — it's basically the "warehouse" where your Docker images live so that anyone (or any server) can **pull and run** them.

---

## 🤔 What is a Container Registry?

Okay, let's start super simple.

When you build a Docker image, that image **lives only on your machine**. But what if your Kubernetes cluster running on AWS needs to pull that image? Or your colleague needs to test it? Or your CI/CD pipeline needs to deploy it?

You need a **central place** to store and share your images. That place is called a **Container Registry**.

Think of it like this:

```
GitHub  → stores your SOURCE CODE
Container Registry → stores your CONTAINER IMAGES
```

That's it! A container registry is **GitHub but for Docker images**.

---

## 🏗️ How Container Registries Work

Here's the basic flow:

```
Developer builds an image
        ↓
docker build -t myapp:v1.0 .
        ↓
Tags the image for the registry
        ↓
docker tag myapp:v1.0 registry.example.com/myapp:v1.0
        ↓
Pushes the image to the registry
        ↓
docker push registry.example.com/myapp:v1.0
        ↓
Registry stores the image (with layers, manifests, tags)
        ↓
Any authorized user/server can now pull the image
        ↓
docker pull registry.example.com/myapp:v1.0
```

### What Actually Gets Stored?

A container image is **not** one big file. It's made up of **layers**. The registry stores:

| Component | What It Is |
|-----------|------------|
| **Image Manifest** | A JSON file that describes the image — what layers it has, what config it uses |
| **Image Layers** | The actual filesystem layers (each `RUN`, `COPY`, etc. in Dockerfile creates a layer) |
| **Image Config** | Metadata like ENV variables, CMD, ENTRYPOINT, architecture (amd64/arm64) |
| **Tags** | Human-readable labels like `v1.0`, `latest`, `production` |
| **Digest** | A unique SHA256 hash — the **true identity** of an image (tags can change, digests don't) |

### The OCI Standard

All modern registries follow the **OCI (Open Container Initiative) Distribution Specification**. This means:

- Images built by Docker can be stored in ANY OCI-compliant registry
- You're not locked into a single registry vendor
- Other artifacts (Helm charts, Wasm modules) can also be stored as OCI artifacts

## 1️⃣ Docker Hub — The Original Registry

Docker Hub is the **default** registry. When you do `docker pull nginx`, Docker pulls from Docker Hub automatically.

### Key Features

- **Public repositories** — unlimited and free (this is where all the official images live)
- **Private repositories** — 1 free, more with paid plans
- **Official Images** — curated, verified images (nginx, postgres, node, python, etc.)
- **Automated Builds** — connect GitHub repo → auto-build image on push (legacy feature)
- **Docker Verified Publisher** — images from trusted software vendors
- **Docker Scout** — image vulnerability scanning

### Working with Docker Hub

#### Login:

```bash
# Login to Docker Hub
docker login

# You'll be prompted for username and password/token
# Username: yourusername
# Password: your-access-token (NOT your password — use access tokens!)
```

> ⚠️ **Security Tip:** Never use your Docker Hub password directly. Go to Docker Hub → Account Settings → Security → New Access Token.

#### Push an Image:

```bash
# Step 1: Build your image
docker build -t myapp:v1.0 .

# Step 2: Tag it with your Docker Hub username
docker tag myapp:v1.0 yourusername/myapp:v1.0

# Step 3: Push it
docker push yourusername/myapp:v1.0
```

#### Pull an Image:

```bash
# Pull a public image (no login needed)
docker pull nginx:latest

# Pull your private image (login needed)
docker pull yourusername/myapp:v1.0
```

### Docker Hub Rate Limits

Docker Hub has **pull rate limits** for free users:

| User Type | Rate Limit |
|-----------|------------|
| Anonymous (no login) | 100 pulls per 6 hours (per IP) |
| Free authenticated user | 200 pulls per 6 hours |
| Pro/Team/Business | Unlimited |

This is a **big deal** in CI/CD pipelines. If your pipeline pulls images heavily, you'll hit rate limits fast!

**Solutions:**
1. Use a **registry mirror/cache** (like Harbor as a proxy cache)
2. Use a **paid Docker Hub plan**
3. Switch to **another registry** (ECR, GHCR, etc.)
4. **Authenticate** in your CI/CD pipelines (at least doubles your limit)

### Docker Hub YAML Example (Kubernetes):

```yaml
# Using a Docker Hub image in a Kubernetes Pod
apiVersion: v1
kind: Pod
metadata:
  name: myapp
spec:
  containers:
    - name: myapp
      image: yourusername/myapp:v1.0   # Docker Hub image
  imagePullSecrets:
    - name: dockerhub-secret           # For private images
```

Creating the pull secret:

```bash
kubectl create secret docker-registry dockerhub-secret \
  --docker-server=https://index.docker.io/v1/ \
  --docker-username=yourusername \
  --docker-password=your-access-token \
  --docker-email=you@example.com
```

## 2️⃣ Harbor — Enterprise Self-Hosted Registry

Harbor is an **open-source** container registry from the **CNCF** (Cloud Native Computing Foundation). It's what you use when you want to run **your own** registry with enterprise-grade features.

### Why Harbor?

| Feature | Description |
|---------|-------------|
| **Self-hosted** | Runs on YOUR infrastructure — full control over your images |
| **Vulnerability Scanning** | Built-in scanning with Trivy (finds CVEs in your images) |
| **RBAC** | Fine-grained access control (who can push/pull what) |
| **Image Signing** | Verify image integrity with Notary/Cosign |
| **Replication** | Replicate images between Harbor instances or to/from other registries |
| **Proxy Cache** | Cache images from Docker Hub (avoids rate limits!) |
| **Garbage Collection** | Clean up old/unused image layers to save disk space |
| **Audit Logs** | Track who pushed/pulled what and when |
| **Multi-tenancy** | Projects with separate access controls |
| **OCI Compliance** | Supports OCI artifacts (Helm charts, OPA bundles, etc.) |

### Installing Harbor

Harbor can be installed with Docker Compose (for testing) or Helm (for Kubernetes):

#### Docker Compose Install (Quick Start):

```bash
# Download the Harbor installer
wget https://github.com/goharbor/harbor/releases/download/v2.10.0/harbor-online-installer-v2.10.0.tgz

# Extract it
tar xvf harbor-online-installer-v2.10.0.tgz
cd harbor

# Copy the config file
cp harbor.yml.tmpl harbor.yml

# Edit the config
nano harbor.yml
```

Key settings in `harbor.yml`:

```yaml
# The hostname of your Harbor instance
hostname: harbor.yourdomain.com

# HTTP/HTTPS config
# For production, always use HTTPS!
https:
  port: 443
  certificate: /your/cert/path/server.crt
  private_key: /your/cert/path/server.key

# Admin password (CHANGE THIS!)
harbor_admin_password: Harbor12345

# Database config
database:
  password: your-db-password
  max_idle_conns: 100
  max_open_conns: 900

# Storage backend (local filesystem by default, can use S3)
storage_service:
  filesystem:
    maxthreads: 100
```

```bash
# Run the installer
sudo ./install.sh --with-trivy

# --with-trivy enables vulnerability scanning
# Harbor is now running at https://harbor.yourdomain.com
```

#### Helm Install (Kubernetes):

```bash
# Add Harbor Helm repo
helm repo add harbor https://helm.goharbor.io
helm repo update

# Install Harbor
helm install harbor harbor/harbor \
  --namespace harbor \
  --create-namespace \
  --set expose.type=ingress \
  --set expose.ingress.hosts.core=harbor.yourdomain.com \
  --set externalURL=https://harbor.yourdomain.com \
  --set harborAdminPassword=StrongPassword123!
```

### Using Harbor

```bash
# Login to Harbor
docker login harbor.yourdomain.com

# Tag and push an image
docker tag myapp:v1.0 harbor.yourdomain.com/myproject/myapp:v1.0
docker push harbor.yourdomain.com/myproject/myapp:v1.0

# Pull an image
docker pull harbor.yourdomain.com/myproject/myapp:v1.0
```

### Harbor as a Docker Hub Proxy Cache

This is one of Harbor's **killer features**. Instead of every server pulling from Docker Hub (and hitting rate limits), they pull from your Harbor cache:

```
Before:  Server A → Docker Hub    (counts toward rate limit)
         Server B → Docker Hub    (counts toward rate limit)
         Server C → Docker Hub    (counts toward rate limit)

After:   Server A → Harbor Cache → Docker Hub  (only Harbor hits Docker Hub)
         Server B → Harbor Cache                (served from cache!)
         Server C → Harbor Cache                (served from cache!)
```

To set this up:
1. In Harbor UI → **Registries** → **New Endpoint** → add Docker Hub
2. Create a **Proxy Cache Project** pointing to that endpoint
3. Configure Docker daemon to use Harbor as a mirror

### Harbor Vulnerability Scanning

```bash
# Harbor scans images automatically after push (if configured)
# You can also trigger a manual scan from the UI

# Or via the API:
curl -X POST "https://harbor.yourdomain.com/api/v2.0/projects/myproject/repositories/myapp/artifacts/v1.0/scan" \
  -H "Authorization: Basic $(echo -n 'admin:password' | base64)"
```

The scan results show:

```
┌─────────────┬──────────┬──────────────────────────────────┐
│  Severity   │  Count   │  Example                         │
├─────────────┼──────────┼──────────────────────────────────┤
│  Critical   │    2     │  CVE-2024-1234 (openssl 3.0.1)   │
│  High       │    5     │  CVE-2024-5678 (curl 7.88)       │
│  Medium     │   12     │  CVE-2024-9012 (libxml2)         │
│  Low        │    8     │  ...                             │
└─────────────┴──────────┴──────────────────────────────────┘
```

You can configure Harbor to **block pulls** for images with Critical/High vulnerabilities — super useful for enforcing security policies!

## 3️⃣ Amazon ECR (Elastic Container Registry)

ECR is AWS's fully managed container registry. If you're running containers on AWS (ECS, EKS, Fargate, Lambda), ECR is the natural choice.

### Key Features

| Feature | Description |
|---------|-------------|
| **Fully Managed** | No servers to maintain — AWS handles everything |
| **IAM Integration** | Access control through AWS IAM policies (no separate auth system) |
| **Image Scanning** | Built-in vulnerability scanning (basic + enhanced with Inspector) |
| **Lifecycle Policies** | Automatically clean up old images based on rules |
| **Cross-Region Replication** | Replicate images to other AWS regions |
| **Cross-Account Access** | Share images between AWS accounts |
| **Encryption** | Images encrypted at rest with KMS |
| **ECR Public** | Host public images on `public.ecr.aws` |

### Setting Up ECR

#### Create a Repository:

```bash
# Create a private repository
aws ecr create-repository \
  --repository-name myapp \
  --region us-east-1 \
  --image-scanning-configuration scanOnPush=true \
  --encryption-configuration encryptionType=KMS
```

This gives you a repository URL like:
```
123456789012.dkr.ecr.us-east-1.amazonaws.com/myapp
```

#### Authenticate Docker with ECR:

```bash
# Get a login token (valid for 12 hours)
aws ecr get-login-password --region us-east-1 | \
  docker login --username AWS --password-stdin \
  123456789012.dkr.ecr.us-east-1.amazonaws.com
```

> **What's happening here?** 
> - `aws ecr get-login-password` gets a temporary auth token from AWS
> - That token is piped into `docker login` 
> - `--username AWS` is always "AWS" for ECR (it's a fixed value)
> - This token expires in 12 hours, so you need to re-authenticate in CI/CD pipelines

#### Push an Image:

```bash
# Build the image
docker build -t myapp:v1.0 .

# Tag it for ECR
docker tag myapp:v1.0 123456789012.dkr.ecr.us-east-1.amazonaws.com/myapp:v1.0

# Push it
docker push 123456789012.dkr.ecr.us-east-1.amazonaws.com/myapp:v1.0
```

#### Pull an Image:

```bash
docker pull 123456789012.dkr.ecr.us-east-1.amazonaws.com/myapp:v1.0
```

### ECR Lifecycle Policies

One of ECR's best features. Automatically delete old images so you don't pay for storage you don't need.

```json
{
  "rules": [
    {
      "rulePriority": 1,
      "description": "Keep only the last 10 tagged images",
      "selection": {
        "tagStatus": "tagged",
        "tagPrefixList": ["v"],
        "countType": "imageCountMoreThan",
        "countNumber": 10
      },
      "action": {
        "type": "expire"
      }
    },
    {
      "rulePriority": 2,
      "description": "Delete untagged images older than 7 days",
      "selection": {
        "tagStatus": "untagged",
        "countType": "sinceImagePushed",
        "countUnit": "days",
        "countNumber": 7
      },
      "action": {
        "type": "expire"
      }
    }
  ]
}
```

```bash
# Apply the lifecycle policy
aws ecr put-lifecycle-policy \
  --repository-name myapp \
  --lifecycle-policy-text file://lifecycle-policy.json
```

**What this does:**
- Rule 1: Keep only the **last 10** images tagged with `v` prefix (like v1.0, v2.0). Older ones get deleted.
- Rule 2: Delete any **untagged** images (dangling images) that are older than 7 days.

This saves you money on ECR storage costs!

### ECR IAM Policy Example

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowPushPull",
      "Effect": "Allow",
      "Action": [
        "ecr:GetAuthorizationToken",
        "ecr:BatchCheckLayerAvailability",
        "ecr:GetDownloadUrlForLayer",
        "ecr:BatchGetImage",
        "ecr:PutImage",
        "ecr:InitiateLayerUpload",
        "ecr:UploadLayerPart",
        "ecr:CompleteLayerUpload"
      ],
      "Resource": "arn:aws:ecr:us-east-1:123456789012:repository/myapp"
    },
    {
      "Sid": "AllowAuth",
      "Effect": "Allow",
      "Action": "ecr:GetAuthorizationToken",
      "Resource": "*"
    }
  ]
}
```

### ECR in Kubernetes (EKS)

If you're running EKS, ECR authentication is handled automatically through **IAM Roles for Service Accounts (IRSA)** or **node instance roles**. No need for `imagePullSecrets`!

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: myapp
spec:
  containers:
    - name: myapp
      image: 123456789012.dkr.ecr.us-east-1.amazonaws.com/myapp:v1.0
  # No imagePullSecrets needed if using IRSA or node role!
```

### ECR Cross-Region Replication

```bash
# Enable replication to another region
aws ecr put-replication-configuration \
  --replication-configuration '{
    "rules": [
      {
        "destinations": [
          {
            "region": "eu-west-1",
            "registryId": "123456789012"
          }
        ]
      }
    ]
  }'
```

Now every image pushed to us-east-1 is **automatically replicated** to eu-west-1. Great for multi-region deployments!

## 4️⃣ Azure ACR (Azure Container Registry)

ACR is Azure's managed container registry. It integrates deeply with Azure services like AKS, App Service, and Azure DevOps.

### Key Features

| Feature | Description |
|---------|-------------|
| **SKUs** | Basic, Standard, Premium (each with different storage/throughput limits) |
| **Geo-replication** | Replicate across Azure regions (Premium tier) |
| **ACR Tasks** | Build images in the cloud (no local Docker needed!) |
| **Content Trust** | Image signing with Notary |
| **Managed Identity** | Passwordless auth using Azure Managed Identities |
| **Private Link** | Access ACR over private network (no public internet) |
| **Vulnerability Scanning** | Microsoft Defender for Containers |
| **Retention Policies** | Auto-delete untagged manifests |

### Setting Up ACR

```bash
# Create ACR
az acr create \
  --resource-group myResourceGroup \
  --name myacrregistry \
  --sku Standard \
  --admin-enabled false

# Your registry URL will be: myacrregistry.azurecr.io
```

### Authenticate with ACR

```bash
# Method 1: Azure CLI login (for developers)
az acr login --name myacrregistry

# Method 2: Service Principal (for CI/CD)
# First, create a service principal
az ad sp create-for-rbac \
  --name acr-push-sp \
  --scopes /subscriptions/<sub-id>/resourceGroups/myResourceGroup/providers/Microsoft.ContainerRegistry/registries/myacrregistry \
  --role AcrPush

# Then use the credentials
docker login myacrregistry.azurecr.io \
  --username <appId> \
  --password <password>

# Method 3: Managed Identity (best for AKS)
# Attach ACR to AKS — no credentials needed!
az aks update \
  --resource-group myResourceGroup \
  --name myAKSCluster \
  --attach-acr myacrregistry
```

### Push and Pull

```bash
# Tag the image for ACR
docker tag myapp:v1.0 myacrregistry.azurecr.io/myapp:v1.0

# Push
docker push myacrregistry.azurecr.io/myapp:v1.0

# Pull
docker pull myacrregistry.azurecr.io/myapp:v1.0
```

### ACR Tasks — Build in the Cloud!

This is a **unique** ACR feature. You can build images in Azure's cloud — no Docker installed locally!

```bash
# Quick build — send Dockerfile + context to ACR, it builds and pushes
az acr build \
  --registry myacrregistry \
  --image myapp:v1.0 \
  .

# This:
# 1. Uploads your source code to ACR
# 2. Builds the image in Azure
# 3. Pushes it to your ACR registry
# All in one command!
```

Multi-step ACR Task (defined in YAML):

```yaml
# acr-task.yaml
version: v1.1.0
steps:
  - build: -t {{.Run.Registry}}/myapp:{{.Run.ID}} .
  - push:
    - {{.Run.Registry}}/myapp:{{.Run.ID}}
  - cmd: {{.Run.Registry}}/myapp:{{.Run.ID}}
    args: ["--version"]
```

```bash
# Run the multi-step task
az acr run --registry myacrregistry -f acr-task.yaml .
```

### ACR in Kubernetes (AKS)

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: myapp
spec:
  containers:
    - name: myapp
      image: myacrregistry.azurecr.io/myapp:v1.0
  # If ACR is attached to AKS (via az aks update --attach-acr),
  # no imagePullSecrets needed!
```

## 5️⃣ Google Cloud — GCR & Artifact Registry

Google Cloud has **two** registries:
- **GCR (Google Container Registry)** — the old one (being deprecated)
- **Artifact Registry** — the new one (replaces GCR, supports more artifact types)

> ⚠️ **Important:** Google recommends using **Artifact Registry** for all new projects. GCR is in maintenance mode.

### Artifact Registry Key Features

| Feature | Description |
|---------|-------------|
| **Multi-format** | Docker images, npm, Maven, Python, Go, Helm, etc. |
| **IAM Integration** | Google Cloud IAM for access control |
| **Vulnerability Scanning** | Built-in with Google Container Analysis |
| **VPC Service Controls** | Restrict access to VPC networks |
| **Cleanup Policies** | Auto-delete old images |
| **Regional/Multi-regional** | Choose where your artifacts are stored |

### Setting Up Artifact Registry

```bash
# Enable the API
gcloud services enable artifactregistry.googleapis.com

# Create a Docker repository
gcloud artifacts repositories create my-docker-repo \
  --repository-format=docker \
  --location=us-central1 \
  --description="My Docker images"

# Your registry URL will be:
# us-central1-docker.pkg.dev/my-project/my-docker-repo
```

### Authenticate

```bash
# Configure Docker to use gcloud as credential helper
gcloud auth configure-docker us-central1-docker.pkg.dev
```

This adds a config entry to `~/.docker/config.json` so Docker knows to use `gcloud` for authentication when pushing/pulling from `us-central1-docker.pkg.dev`.

### Push and Pull

```bash
# Tag the image
docker tag myapp:v1.0 us-central1-docker.pkg.dev/my-project/my-docker-repo/myapp:v1.0

# Push
docker push us-central1-docker.pkg.dev/my-project/my-docker-repo/myapp:v1.0

# Pull
docker pull us-central1-docker.pkg.dev/my-project/my-docker-repo/myapp:v1.0
```

### Cleanup Policies

```bash
# Delete images older than 30 days
gcloud artifacts repositories set-cleanup-policies my-docker-repo \
  --location=us-central1 \
  --policy=cleanup-policy.json
```

```json
[
  {
    "name": "delete-old-images",
    "action": {"type": "Delete"},
    "condition": {
      "olderThan": "2592000s",
      "tagState": "ANY"
    }
  },
  {
    "name": "keep-minimum-versions",
    "action": {"type": "Keep"},
    "mostRecentVersions": {
      "keepCount": 5
    }
  }
]
```

### Artifact Registry in Kubernetes (GKE)

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: myapp
spec:
  containers:
    - name: myapp
      image: us-central1-docker.pkg.dev/my-project/my-docker-repo/myapp:v1.0
  # GKE nodes have automatic access to Artifact Registry in the same project
  # No imagePullSecrets needed!
```

## 6️⃣ GHCR (GitHub Container Registry)

GHCR is GitHub's container registry, tightly integrated with GitHub repos, Actions, and packages.

### Key Features

| Feature | Description |
|---------|-------------|
| **Free for public repos** | Unlimited storage and transfer for public packages |
| **GitHub Actions integration** | Seamless auth in Actions workflows using `GITHUB_TOKEN` |
| **RBAC** | Tied to GitHub users, teams, and organizations |
| **Visibility control** | Public, private, or internal packages |
| **OCI compliant** | Supports Docker images and other OCI artifacts |

### Authenticate with GHCR

```bash
# Login using a Personal Access Token (PAT)
# PAT needs 'read:packages', 'write:packages', 'delete:packages' scopes
echo $GITHUB_TOKEN | docker login ghcr.io -u USERNAME --password-stdin
```

### Push and Pull

```bash
# Tag the image for GHCR
docker tag myapp:v1.0 ghcr.io/yourusername/myapp:v1.0

# Push
docker push ghcr.io/yourusername/myapp:v1.0

# Pull (public images — no auth needed)
docker pull ghcr.io/yourusername/myapp:v1.0
```

### GHCR in GitHub Actions

This is where GHCR **really shines** — seamless integration with GitHub Actions:

```yaml
# .github/workflows/build-push.yml
name: Build and Push to GHCR

on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write        # Required for pushing to GHCR

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Login to GHCR
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}   # Auto-provided! No manual secret needed!

      - name: Build and Push
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: |
            ghcr.io/${{ github.repository }}:latest
            ghcr.io/${{ github.repository }}:${{ github.sha }}
```

**What's amazing here:**
- `${{ secrets.GITHUB_TOKEN }}` is **automatically available** in every workflow — no manual secret setup!
- `${{ github.repository }}` gives you `owner/repo` — perfect for image naming
- `${{ github.sha }}` gives you the commit hash — great for unique tags

### GHCR in Kubernetes

```bash
# Create a pull secret for GHCR
kubectl create secret docker-registry ghcr-secret \
  --docker-server=ghcr.io \
  --docker-username=yourusername \
  --docker-password=your-github-pat \
  --docker-email=you@example.com
```

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: myapp
spec:
  containers:
    - name: myapp
      image: ghcr.io/yourusername/myapp:v1.0
  imagePullSecrets:
    - name: ghcr-secret
```

## 🔐 Registry Security Best Practices

No matter which registry you use, follow these security practices:

### 1. Never Use `latest` Tag in Production

```yaml
# ❌ BAD — what version is "latest"? It changes with every push!
image: myapp:latest

# ✅ GOOD — pinned to a specific version
image: myapp:v1.2.3

# ✅ BEST — pinned to digest (immutable, can't be overwritten)
image: myapp@sha256:abc123def456...
```

### 2. Enable Image Scanning

```bash
# ECR — scan on push
aws ecr put-image-scanning-configuration \
  --repository-name myapp \
  --image-scanning-configuration scanOnPush=true

# Harbor — scanning is built-in with Trivy

# ACR — use Microsoft Defender for Containers

# GHCR — use GitHub's Dependabot / CodeQL
```

### 3. Use Image Signing

Sign your images so you can verify they haven't been tampered with:

```bash
# Using Cosign (from Sigstore project)
# Install cosign
brew install cosign   # or download from GitHub releases

# Generate a key pair
cosign generate-key-pair

# Sign an image
cosign sign --key cosign.key ghcr.io/yourusername/myapp:v1.0

# Verify an image
cosign verify --key cosign.pub ghcr.io/yourusername/myapp:v1.0
```

### 4. Use Minimal Base Images

```dockerfile
# ❌ BAD — full Ubuntu image (77MB+, hundreds of packages, large attack surface)
FROM ubuntu:22.04

# ✅ BETTER — Alpine (5MB, minimal packages)
FROM alpine:3.19

# ✅ BEST — Distroless (no shell, no package manager, minimal attack surface)
FROM gcr.io/distroless/static-debian12
```

## 🔄 Registry Migration — Moving Between Registries

Sometimes you need to move images between registries. Here's how:

### Method 1: Pull and Push (Simple)

```bash
# Pull from source registry
docker pull old-registry.com/myapp:v1.0

# Tag for destination registry
docker tag old-registry.com/myapp:v1.0 new-registry.com/myapp:v1.0

# Push to destination
docker push new-registry.com/myapp:v1.0
```

### Method 2: Skopeo (Better — No Docker Daemon Needed!)

```bash
# Install skopeo
sudo apt install skopeo

# Copy image directly between registries (no local pull/push!)
skopeo copy \
  docker://old-registry.com/myapp:v1.0 \
  docker://new-registry.com/myapp:v1.0

# Copy all tags of an image
skopeo copy --all \
  docker://old-registry.com/myapp \
  docker://new-registry.com/myapp
```

**Why Skopeo is better:**
- No Docker daemon needed
- Direct registry-to-registry transfer (faster)
- Supports copying all tags at once
- Can inspect images without pulling them

### Method 3: Harbor Replication (For Harbor Users)

Harbor has built-in replication rules:
1. Harbor UI → **Replications** → **New Replication Rule**
2. Source: Docker Hub / ECR / GCR / etc.
3. Destination: Your Harbor instance (or vice versa)
4. Set filters (by repository name, tag, etc.)
5. Set trigger (manual, scheduled, event-driven)

## 📊 Choosing the Right Registry — Decision Guide

```
Are you using a cloud provider?
├── AWS → Use ECR
├── Azure → Use ACR
├── GCP → Use Artifact Registry
└── No / Multi-cloud?
    ├── Need self-hosted? → Use Harbor
    ├── Using GitHub for code? → Use GHCR
    └── Public images / quick start? → Use Docker Hub
```

| Factor | Docker Hub | Harbor | ECR | ACR | GCR/AR | GHCR |
|--------|-----------|--------|-----|-----|--------|------|
| **Cost** | Free (limited) | Free (self-hosted) | Pay per storage/transfer | Pay per tier | Pay per storage | Free (public) |
| **Setup effort** | None | Medium-High | Low | Low | Low | None |
| **Maintenance** | None | You manage it | None | None | None | None |
| **Scanning** | Limited | Trivy (built-in) | Inspector | Defender | Container Analysis | Dependabot |
| **CI/CD integration** | Manual | Manual | CodePipeline/Actions | Azure DevOps | Cloud Build | Actions (native!) |
| **Best for** | Open source | Enterprise/air-gapped | AWS shops | Azure shops | GCP shops | GitHub-centric teams |

## 🎯 Summary

| Concept | Key Takeaway |
|---------|-------------|
| **Container Registry** | A storage system for container images (like GitHub for Docker images) |
| **Docker Hub** | Default registry, great for public images, has rate limits |
| **Harbor** | Self-hosted, enterprise-grade, CNCF project, proxy cache, scanning |
| **ECR** | AWS-managed, IAM auth, lifecycle policies, cross-region replication |
| **ACR** | Azure-managed, ACR Tasks for cloud builds, Managed Identity auth |
| **Artifact Registry** | Google's next-gen registry, multi-format, replaces GCR |
| **GHCR** | GitHub's registry, free for public, amazing GitHub Actions integration |
| **Security** | Always scan, sign, use RBAC, avoid `latest`, use minimal base images |
| **OCI Standard** | All modern registries follow OCI spec — images are portable |
