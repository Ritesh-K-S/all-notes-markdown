# 20.5 — Helm Chart Repositories — ChartMuseum, OCI Registries

> **One Line Answer:** A Helm Chart Repository is a **place to store and share Helm charts** — the packages that define how applications are deployed on Kubernetes. You can use traditional **HTTP-based repos** (like ChartMuseum) or modern **OCI-based registries** (like ECR, ACR, Harbor) to host your charts.

---

## 🤔 Wait, What's a Helm Chart Again? (Quick Refresher)

Before we dive into repositories, let's make sure we're on the same page.

A **Helm chart** is a bundle of Kubernetes YAML templates + configuration that describes how to deploy an application. Think of it like an "installer package" for Kubernetes.

```
mychart/
├── Chart.yaml          # Chart metadata (name, version, description)
├── values.yaml         # Default configuration values
├── templates/          # Kubernetes YAML templates
│   ├── deployment.yaml
│   ├── service.yaml
│   ├── ingress.yaml
│   └── _helpers.tpl
└── charts/             # Dependencies (sub-charts)
```

Instead of doing `kubectl apply -f deployment.yaml -f service.yaml -f ingress.yaml ...`, you just do:

```bash
helm install myapp ./mychart
```

And Helm creates everything for you, with your custom values applied. Beautiful!

Now the question is: **where do you store these charts so others can use them?**

---

## 📦 Two Ways to Store Helm Charts

There are **two** approaches to hosting Helm charts, and this is important to understand:

| Approach | How It Works | Standard |
|----------|-------------|----------|
| **Traditional Chart Repository** | A simple HTTP server that hosts an `index.yaml` file + chart `.tgz` packages | Helm v2/v3 native |
| **OCI Registry** | Store charts as OCI artifacts in any container registry (ECR, ACR, Harbor, GHCR, etc.) | Helm v3.8+ (GA) |

```
Traditional:  helm repo add myrepo https://charts.example.com
              helm install myapp myrepo/mychart

OCI:          helm install myapp oci://registry.example.com/charts/mychart --version 1.0.0
```

Let's explore both!

---

## 🏛️ Traditional Helm Chart Repositories

### How They Work

A traditional Helm chart repository is just a **web server** with two things:

1. An `index.yaml` file — lists all charts, their versions, and download URLs
2. The `.tgz` chart packages — the actual packaged charts

That's it! It's literally just a file server.

```
https://charts.example.com/
├── index.yaml                    # Index of all charts
├── mychart-1.0.0.tgz           # Chart package v1.0.0
├── mychart-1.1.0.tgz           # Chart package v1.1.0
└── another-chart-2.0.0.tgz     # Another chart
```

### The `index.yaml` File

This is what Helm uses to discover charts and their versions:

```yaml
apiVersion: v1
entries:
  mychart:
    - name: mychart
      version: 1.1.0
      description: My awesome application
      appVersion: "2.0.0"
      created: "2024-06-15T10:00:00Z"
      digest: sha256:abc123...
      urls:
        - https://charts.example.com/mychart-1.1.0.tgz
    - name: mychart
      version: 1.0.0
      description: My awesome application
      appVersion: "1.0.0"
      created: "2024-05-01T10:00:00Z"
      digest: sha256:def456...
      urls:
        - https://charts.example.com/mychart-1.0.0.tgz
  another-chart:
    - name: another-chart
      version: 2.0.0
      description: Another cool app
      ...
generated: "2024-06-15T10:00:00Z"
```

### Using Traditional Repos

```bash
# Add a repository
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo add stable https://charts.helm.sh/stable    # Deprecated but still used

# Update the local index cache
helm repo update

# Search for charts
helm search repo nginx
helm search repo bitnami/postgresql

# Install a chart from a repo
helm install my-nginx bitnami/nginx --version 15.0.0

# List your configured repos
helm repo list

# Remove a repo
helm repo remove bitnami
```

### Hosting Your Own — Simple Options

#### Option 1: GitHub Pages (Free!)

You can host a Helm repo for free using GitHub Pages:

```bash
# Step 1: Create a GitHub repo (e.g., my-helm-charts)

# Step 2: Package your chart
helm package mychart/
# Creates: mychart-1.0.0.tgz

# Step 3: Generate the index
helm repo index . --url https://yourusername.github.io/my-helm-charts

# Step 4: Commit and push to the gh-pages branch
git add .
git commit -m "Add mychart 1.0.0"
git push origin gh-pages

# Step 5: Enable GitHub Pages for the repo

# Step 6: Use it!
helm repo add myrepo https://yourusername.github.io/my-helm-charts
helm install myapp myrepo/mychart
```

#### Option 2: S3 Bucket (AWS)

```bash
# Install the helm-s3 plugin
helm plugin install https://github.com/hypnoglow/helm-s3.git

# Initialize an S3 bucket as a Helm repo
helm s3 init s3://my-helm-charts-bucket/charts

# Add the S3 repo
helm repo add myrepo s3://my-helm-charts-bucket/charts

# Push a chart to S3
helm s3 push mychart-1.0.0.tgz myrepo

# Install from S3
helm install myapp myrepo/mychart
```

#### Option 3: GCS Bucket (Google Cloud)

```bash
# Install the helm-gcs plugin
helm plugin install https://github.com/hayorov/helm-gcs.git

# Initialize
helm gcs init gs://my-helm-charts-bucket

# Add repo
helm repo add myrepo gs://my-helm-charts-bucket

# Push chart
helm gcs push mychart-1.0.0.tgz myrepo
```

## 🏗️ ChartMuseum — Dedicated Helm Chart Server

ChartMuseum is an **open-source Helm chart repository server** written in Go. It's the most popular tool for hosting your own Helm charts.

### Why ChartMuseum?

| Feature | Description |
|---------|-------------|
| **Simple** | Just run a single binary — it's a Helm repo server |
| **Multiple backends** | Store charts on local filesystem, S3, GCS, Azure Blob, MinIO, etc. |
| **API** | RESTful API for uploading/managing charts |
| **Chart upload** | Push charts via API (no need to manually manage index.yaml) |
| **Multi-tenancy** | Support multiple repos (orgs/teams) |
| **Basic auth** | Built-in basic authentication |
| **Chart provenance** | Support for Helm chart provenance files (`.prov`) |
| **Helm plugin** | `helm cm-push` plugin for easy chart pushing |

### Installing ChartMuseum

#### Docker:

```bash
# Run ChartMuseum with Docker
docker run -d \
  --name chartmuseum \
  -p 8080:8080 \
  -e DEBUG=true \
  -e STORAGE=local \
  -e STORAGE_LOCAL_ROOTDIR=/charts \
  -v $(pwd)/charts:/charts \
  ghcr.io/helm/chartmuseum:v0.16.0
```

**What's happening here?**
- `-p 8080:8080` — expose port 8080
- `STORAGE=local` — store charts on local filesystem
- `STORAGE_LOCAL_ROOTDIR=/charts` — where to store them inside the container
- `-v $(pwd)/charts:/charts` — mount a local directory for persistence

#### With S3 Backend:

```bash
docker run -d \
  --name chartmuseum \
  -p 8080:8080 \
  -e STORAGE=amazon \
  -e STORAGE_AMAZON_BUCKET=my-helm-charts \
  -e STORAGE_AMAZON_REGION=us-east-1 \
  -e AWS_ACCESS_KEY_ID=your-key \
  -e AWS_SECRET_ACCESS_KEY=your-secret \
  ghcr.io/helm/chartmuseum:v0.16.0
```

#### With Google Cloud Storage Backend:

```bash
docker run -d \
  --name chartmuseum \
  -p 8080:8080 \
  -e STORAGE=google \
  -e STORAGE_GOOGLE_BUCKET=my-helm-charts \
  -e STORAGE_GOOGLE_PREFIX= \
  -v /path/to/service-account.json:/gcs-key.json \
  -e GOOGLE_APPLICATION_CREDENTIALS=/gcs-key.json \
  ghcr.io/helm/chartmuseum:v0.16.0
```

#### Helm Install (on Kubernetes):

```bash
helm repo add chartmuseum https://chartmuseum.github.io/charts
helm repo update

helm install chartmuseum chartmuseum/chartmuseum \
  --namespace chartmuseum \
  --create-namespace \
  --set env.open.STORAGE=local \
  --set persistence.enabled=true \
  --set persistence.size=10Gi \
  --set ingress.enabled=true \
  --set ingress.hosts[0].name=charts.yourdomain.com \
  --set ingress.hosts[0].path=/
```

### Using ChartMuseum

#### Add the ChartMuseum Repo:

```bash
helm repo add myrepo http://localhost:8080
helm repo update
```

#### Push Charts to ChartMuseum:

**Method 1: Using the helm cm-push plugin (recommended)**

```bash
# Install the plugin
helm plugin install https://github.com/chartmuseum/helm-push

# Package and push your chart
helm cm-push mychart/ myrepo

# Or push a specific .tgz file
helm cm-push mychart-1.0.0.tgz myrepo
```

**Method 2: Using curl (API)**

```bash
# Package the chart first
helm package mychart/

# Upload via API
curl --data-binary "@mychart-1.0.0.tgz" http://localhost:8080/api/charts
```

#### Managing Charts:

```bash
# List all charts
curl http://localhost:8080/api/charts | jq

# Get info about a specific chart
curl http://localhost:8080/api/charts/mychart | jq

# Get a specific version
curl http://localhost:8080/api/charts/mychart/1.0.0 | jq

# Delete a chart version
curl -X DELETE http://localhost:8080/api/charts/mychart/1.0.0
```

#### Install Charts:

```bash
# Update repo index to see new charts
helm repo update

# Search for charts
helm search repo myrepo/

# Install a chart
helm install myapp myrepo/mychart --version 1.0.0
```

### ChartMuseum with Authentication

```bash
# Run with basic auth
docker run -d \
  --name chartmuseum \
  -p 8080:8080 \
  -e STORAGE=local \
  -e STORAGE_LOCAL_ROOTDIR=/charts \
  -e BASIC_AUTH_USER=admin \
  -e BASIC_AUTH_PASS=secretpassword \
  -e AUTH_ANONYMOUS_GET=true \
  -v $(pwd)/charts:/charts \
  ghcr.io/helm/chartmuseum:v0.16.0
```

**What each auth setting does:**
- `BASIC_AUTH_USER` / `BASIC_AUTH_PASS` — credentials for pushing charts
- `AUTH_ANONYMOUS_GET=true` — anyone can **pull** charts without auth, but **push** requires login

```bash
# Add repo with credentials
helm repo add myrepo http://localhost:8080 --username admin --password secretpassword

# Push with credentials
helm cm-push mychart/ myrepo --username admin --password secretpassword
```

### ChartMuseum Multi-Tenancy

You can run ChartMuseum with a "depth" setting that creates separate repos per path:

```bash
docker run -d \
  -e STORAGE=local \
  -e STORAGE_LOCAL_ROOTDIR=/charts \
  -e DEPTH=1 \
  -p 8080:8080 \
  ghcr.io/helm/chartmuseum:v0.16.0
```

With `DEPTH=1`, each top-level directory becomes a separate repo:

```bash
# Team A's repo
helm repo add team-a http://localhost:8080/team-a
curl --data-binary "@myapp-1.0.0.tgz" http://localhost:8080/api/team-a/charts

# Team B's repo
helm repo add team-b http://localhost:8080/team-b
curl --data-binary "@otherapp-1.0.0.tgz" http://localhost:8080/api/team-b/charts
```

## 📦 OCI-Based Helm Chart Registries (The Modern Way)

Starting with **Helm v3.8.0**, OCI support became **GA (Generally Available)**. This means you can store Helm charts in **any OCI-compliant container registry** — the same registries you use for Docker images!

### Why OCI for Helm Charts?

| Traditional Repos | OCI Registries |
|-------------------|----------------|
| Need a separate server (ChartMuseum, GitHub Pages, etc.) | Use your **existing** container registry |
| Custom `index.yaml` format | Standard OCI distribution spec |
| `helm repo add` + `helm repo update` workflow | Direct `helm push` / `helm pull` |
| Separate auth system | Same auth as your container images |
| Can become stale (index not updated) | Always current (no index to update) |
| Limited features | Inherits registry features (scanning, replication, RBAC) |

### How OCI Helm Charts Work

When you push a Helm chart to an OCI registry, it's stored as an **OCI artifact** — just like a Docker image, but with a different media type:

```
Docker Image:    application/vnd.oci.image.manifest.v1+json
Helm Chart:      application/vnd.cncf.helm.chart.content.v1.tar+gzip
```

The registry doesn't care — it stores both the same way!

```
my-registry.com/
├── myapp           ← Docker image
├── nginx           ← Docker image
└── charts/
    ├── mychart     ← Helm chart (OCI artifact)
    └── mylib       ← Helm chart (OCI artifact)
```

### Using OCI Helm Charts

#### Login to Registry:

```bash
# Login works the same as Docker
helm registry login registry.example.com

# Or use specific credentials
helm registry login ghcr.io --username myuser --password mytoken
```

#### Package and Push:

```bash
# Step 1: Package the chart
helm package mychart/
# Creates: mychart-1.0.0.tgz

# Step 2: Push to OCI registry
helm push mychart-1.0.0.tgz oci://registry.example.com/charts
```

**What happens:**
- The chart `mychart-1.0.0.tgz` is pushed to `registry.example.com/charts/mychart`
- The version `1.0.0` becomes the **tag**
- So the full reference is: `oci://registry.example.com/charts/mychart:1.0.0`

#### Pull a Chart:

```bash
# Pull a chart (downloads the .tgz)
helm pull oci://registry.example.com/charts/mychart --version 1.0.0
```

#### Install Directly from OCI:

```bash
# Install directly — no "helm repo add" needed!
helm install myapp oci://registry.example.com/charts/mychart --version 1.0.0

# With custom values
helm install myapp oci://registry.example.com/charts/mychart \
  --version 1.0.0 \
  --values my-values.yaml \
  --namespace myapp \
  --create-namespace
```

Notice the big difference: **no `helm repo add` step!** You just reference the chart by its full OCI URL. No stale index, no repo update. Just direct access.

#### Show Chart Info:

```bash
# Show chart metadata
helm show chart oci://registry.example.com/charts/mychart --version 1.0.0

# Show default values
helm show values oci://registry.example.com/charts/mychart --version 1.0.0

# Show everything
helm show all oci://registry.example.com/charts/mychart --version 1.0.0
```

#### Template Locally:

```bash
# Render the templates without installing
helm template myapp oci://registry.example.com/charts/mychart --version 1.0.0
```

### GitHub Actions — Push Helm Chart to GHCR:

```yaml
name: Publish Helm Chart

on:
  push:
    paths:
      - 'charts/**'
    branches: [main]

jobs:
  publish:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write

    steps:
      - uses: actions/checkout@v4

      - name: Install Helm

      - name: Push Chart
        run: helm push mychart-*.tgz oci://ghcr.io/${{ github.repository_owner }}/charts
```

## 🔗 Using OCI Charts as Dependencies

You can reference OCI-hosted charts as dependencies in your `Chart.yaml`:

```yaml
# Chart.yaml
apiVersion: v2
name: my-app
version: 1.0.0

dependencies:
  # Traditional repo dependency
  - name: postgresql
    version: "12.1.0"
    repository: "https://charts.bitnami.com/bitnami"

  # OCI registry dependency
  - name: redis
    version: "17.0.0"
    repository: "oci://registry.example.com/charts"
```

```bash
# Build dependencies (downloads sub-charts)
helm dependency build mychart/

# Or update dependencies
helm dependency update mychart/
```

## 🧹 Chart Repository Housekeeping

### Versioning Your Charts

Always use **Semantic Versioning** for charts:

```yaml
# Chart.yaml
apiVersion: v2
name: mychart
version: 1.2.3        # Chart version (the package version)
appVersion: "4.5.6"    # Version of the app being deployed
```

| Field | Meaning | When to Bump |
|-------|---------|-------------|
| `version` | Chart version | When chart templates/values change |
| `appVersion` | Application version | When the deployed app version changes |

```
version format: MAJOR.MINOR.PATCH

MAJOR: Breaking changes (new required values, removed templates)
MINOR: New features (new templates, new optional values)
PATCH: Bug fixes (template fixes, documentation)
```

### Cleaning Up Old Chart Versions

#### ChartMuseum:

```bash
# Delete a specific version
curl -X DELETE http://localhost:8080/api/charts/mychart/1.0.0

# Tip: Automate cleanup in CI/CD — keep only last N versions
```

#### ECR:

```bash
# ECR lifecycle policies work for Helm charts too!
# Same lifecycle policy as Docker images (see 20.4)
```

#### OCI Registries (General):

```bash
# Use skopeo to list tags
skopeo list-tags docker://registry.example.com/charts/mychart

# Delete specific tags with registry-specific CLI tools
```

## 🆚 ChartMuseum vs OCI — Which Should You Use?

| Factor | ChartMuseum | OCI Registries |
|--------|-------------|----------------|
| **Setup** | Extra server to deploy/maintain | Use existing registry |
| **Complexity** | Simple, purpose-built | Slightly different workflow |
| **Standards** | Helm-specific | Industry-standard OCI |
| **Auth** | Basic auth / custom | Same as container registry |
| **Future-proof** | Maintenance mode (less active) | The Helm community direction |
| **Ecosystem** | Helm-only | Multi-artifact (images, charts, Wasm, etc.) |
| **`helm search`** | ✅ Works with `helm search repo` | ❌ No `helm search` for OCI |
| **Index** | `index.yaml` (can become stale) | No index needed |

### The Verdict

> 🏆 **For new projects: Use OCI registries.** They're the future, simpler (no extra server), and leverage your existing infrastructure.
>
> ⚙️ **For existing setups with ChartMuseum:** No rush to migrate, but plan for OCI migration when convenient.

## 🔄 Complete Workflow Example — CI/CD with Helm OCI Charts

Here's a complete real-world workflow:

```yaml
# .github/workflows/helm-release.yml
name: Release Helm Chart

on:
  push:
    tags:
      - 'chart-v*'    # Trigger on tags like chart-v1.0.0

jobs:
  release:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write

    steps:
      - uses: actions/checkout@v4

      - name: Install Helm
        uses: azure/setup-helm@v3

      - name: Extract version from tag
        id: version
        run: echo "VERSION=${GITHUB_REF_NAME#chart-v}" >> $GITHUB_OUTPUT

      - name: Update Chart.yaml version
        run: |
          sed -i "s/^version:.*/version: ${{ steps.version.outputs.VERSION }}/" charts/myapp/Chart.yaml

      - name: Lint Chart
        run: helm lint charts/myapp/

      - name: Package Chart
        run: helm package charts/myapp/

      - name: Login to GHCR
        run: |
          echo "${{ secrets.GITHUB_TOKEN }}" | \
            helm registry login ghcr.io --username ${{ github.actor }} --password-stdin

      - name: Push Chart
        run: helm push myapp-${{ steps.version.outputs.VERSION }}.tgz oci://ghcr.io/${{ github.repository_owner }}/charts

      - name: Summary
        run: |
          echo "### Chart Published! 🎉" >> $GITHUB_STEP_SUMMARY
          echo "" >> $GITHUB_STEP_SUMMARY
          echo "**Chart:** myapp" >> $GITHUB_STEP_SUMMARY
          echo "**Version:** ${{ steps.version.outputs.VERSION }}" >> $GITHUB_STEP_SUMMARY
          echo "**Registry:** oci://ghcr.io/${{ github.repository_owner }}/charts/myapp" >> $GITHUB_STEP_SUMMARY
```

**How to use:**

```bash
# Create a tag to trigger the release
git tag chart-v1.0.0
git push origin chart-v1.0.0

# GitHub Actions will:
# 1. Lint the chart
# 2. Package it
# 3. Push to GHCR

# Then anyone can install it:
helm install myapp oci://ghcr.io/yourorg/charts/myapp --version 1.0.0
```

## 🎯 Summary

| Concept | Key Takeaway |

| **Helm Chart Repository** | A place to store and share Helm charts |
