# 20.6 — Artifact Promotion Strategies

> **One Line Answer:** Artifact Promotion is the practice of **moving a tested, verified artifact (Docker image, Helm chart, JAR, etc.) through a series of environments** (dev → staging → production) WITHOUT rebuilding it — ensuring that the exact same artifact that passed your tests is the one running in production.

---

## 🤔 What Problem Does Artifact Promotion Solve?

Let's paint a picture that happens in too many teams:

### The Anti-Pattern (Building per Environment):

```
Developer pushes code to Git
        ↓
CI pipeline builds image for DEV          → myapp:dev-abc123
        ↓
Deploy to DEV → Test → ✅ Passes!
        ↓
CI pipeline builds ANOTHER image for STAGING  → myapp:staging-abc123
        ↓
Deploy to STAGING → Test → ✅ Passes!
        ↓
CI pipeline builds YET ANOTHER image for PROD  → myapp:prod-abc123
        ↓
Deploy to PROD → 💥 BREAKS!
```

**Wait... why did it break?** All three environments used the same source code. But here's the dirty truth:

- **Different build time** = different dependency versions could be resolved
- **Different environment variables** during build
- **Different base image** if `latest` tag was updated between builds
- **Non-deterministic builds** — even the same Dockerfile can produce different images at different times
- **Build cache** differences between environments

**The core problem: You're NOT deploying what you tested!**

### The Right Way (Artifact Promotion):

```
Developer pushes code to Git
        ↓
CI pipeline builds ONE image              → myapp:v1.2.3 (sha256:abc123)
        ↓
Push to registry
        ↓
PROMOTE to DEV → Deploy → Test → ✅ Passes!
        ↓
PROMOTE the SAME image to STAGING → Deploy → Test → ✅ Passes!
        ↓
PROMOTE the SAME image to PROD → Deploy → ✅ Works!
```

**The same exact image (same SHA256 digest) flows through ALL environments.** This is artifact promotion.

---

## 🏗️ How Artifact Promotion Works

The key principle is: **Build once, deploy everywhere.**

There are several strategies to implement this. Let's explore each one.

---

## Strategy 1: Tag-Based Promotion

The simplest approach. You add environment-specific tags to the same image digest.

### How It Works:

```
Build Phase:
  docker build → myapp:v1.2.3   (digest: sha256:abc123)
  docker push  → registry.com/myapp:v1.2.3

Promote to Dev:
  docker tag registry.com/myapp:v1.2.3 registry.com/myapp:dev
  docker push registry.com/myapp:dev

Promote to Staging (after dev tests pass):
  docker tag registry.com/myapp:v1.2.3 registry.com/myapp:staging
  docker push registry.com/myapp:staging

Promote to Production (after staging tests pass):
  docker tag registry.com/myapp:v1.2.3 registry.com/myapp:production
  docker push registry.com/myapp:production
```

All four tags (`v1.2.3`, `dev`, `staging`, `production`) point to the **exact same image digest**. No rebuild!

### Implementation:

```bash
#!/bin/bash
# promote.sh — Simple tag-based promotion script

IMAGE="registry.com/myapp"
SOURCE_TAG=$1      # e.g., "v1.2.3"
TARGET_ENV=$2      # e.g., "staging" or "production"

echo "Promoting $IMAGE:$SOURCE_TAG to $TARGET_ENV..."

# Get the digest of the source image
DIGEST=$(docker inspect --format='{{index .RepoDigests 0}}' "$IMAGE:$SOURCE_TAG" 2>/dev/null)

if [ -z "$DIGEST" ]; then
    # If not local, pull it
    docker pull "$IMAGE:$SOURCE_TAG"
fi

# Retag for the target environment
docker tag "$IMAGE:$SOURCE_TAG" "$IMAGE:$TARGET_ENV"

# Push the new tag
docker push "$IMAGE:$TARGET_ENV"

echo "✅ Promoted $IMAGE:$SOURCE_TAG → $IMAGE:$TARGET_ENV"
echo "   Digest: $DIGEST"
```

### Without Docker (Using Skopeo):

```bash
# Skopeo can copy tags without pulling the full image!
skopeo copy \
  docker://registry.com/myapp:v1.2.3 \
  docker://registry.com/myapp:staging

# This is MUCH faster because:
# - No Docker daemon needed
# - No local pull/push
# - Direct registry-to-registry copy
```

### GitHub Actions Example:

```yaml
# .github/workflows/promote.yml
name: Promote Artifact

on:
  workflow_dispatch:
    inputs:
      version:
        description: 'Image version to promote (e.g., v1.2.3)'
        required: true
      target_env:
        description: 'Target environment'
        required: true
        type: choice
        options:
          - staging
          - production

jobs:
  promote:
    runs-on: ubuntu-latest
    permissions:
      packages: write

    steps:
      - name: Login to Registry
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Install Skopeo
        run: sudo apt-get install -y skopeo

      - name: Promote Image
        run: |
          skopeo copy \
            docker://ghcr.io/${{ github.repository }}/myapp:${{ inputs.version }} \
            docker://ghcr.io/${{ github.repository }}/myapp:${{ inputs.target_env }}

      - name: Verify Promotion
        run: |
          # Verify both tags point to the same digest
          SOURCE_DIGEST=$(skopeo inspect docker://ghcr.io/${{ github.repository }}/myapp:${{ inputs.version }} | jq -r '.Digest')
          TARGET_DIGEST=$(skopeo inspect docker://ghcr.io/${{ github.repository }}/myapp:${{ inputs.target_env }} | jq -r '.Digest')

          echo "Source digest: $SOURCE_DIGEST"
          echo "Target digest: $TARGET_DIGEST"

          if [ "$SOURCE_DIGEST" = "$TARGET_DIGEST" ]; then
            echo "✅ Promotion verified — digests match!"
          else
            echo "❌ ERROR: Digests don't match!"
            exit 1
          fi
```

### Pros & Cons:

| Pros | Cons |
|------|------|
| Simple to implement | Environment tags get overwritten (lose history) |
| Works with any registry | Hard to track "what version is in production?" |
| No extra infrastructure | Tags are mutable — can be confusing |
| Easy to understand | Need discipline to never rebuild |

## Strategy 2: Repository-Based Promotion

Instead of different tags, you use **different repositories** (or registry projects) for each environment.

### How It Works:

```
Build Phase:
  Build → registry.com/dev/myapp:v1.2.3

Promote to Staging:
  Copy → registry.com/staging/myapp:v1.2.3

Promote to Production:
  Copy → registry.com/prod/myapp:v1.2.3
```

Each environment has its own "namespace" in the registry.

### With Harbor Projects:

```
harbor.company.com/
├── dev/            ← Dev images
│   └── myapp:v1.2.3
├── staging/        ← Staging images (promoted from dev)
│   └── myapp:v1.2.3
└── prod/           ← Production images (promoted from staging)
    └── myapp:v1.2.3
```

```bash
# Build and push to dev
docker build -t harbor.company.com/dev/myapp:v1.2.3 .
docker push harbor.company.com/dev/myapp:v1.2.3

# Promote to staging (using skopeo — no rebuild!)
skopeo copy \
  docker://harbor.company.com/dev/myapp:v1.2.3 \
  docker://harbor.company.com/staging/myapp:v1.2.3

# Promote to production
skopeo copy \
  docker://harbor.company.com/staging/myapp:v1.2.3 \
  docker://harbor.company.com/prod/myapp:v1.2.3
```

### With ECR:

```bash
# Create repos for each environment
aws ecr create-repository --repository-name dev/myapp
aws ecr create-repository --repository-name staging/myapp
aws ecr create-repository --repository-name prod/myapp

# Build and push to dev
docker build -t 123456789012.dkr.ecr.us-east-1.amazonaws.com/dev/myapp:v1.2.3 .
docker push 123456789012.dkr.ecr.us-east-1.amazonaws.com/dev/myapp:v1.2.3

# Promote to staging
skopeo copy \
  docker://123456789012.dkr.ecr.us-east-1.amazonaws.com/dev/myapp:v1.2.3 \
  docker://123456789012.dkr.ecr.us-east-1.amazonaws.com/staging/myapp:v1.2.3

# Promote to production
skopeo copy \
  docker://123456789012.dkr.ecr.us-east-1.amazonaws.com/staging/myapp:v1.2.3 \
  docker://123456789012.dkr.ecr.us-east-1.amazonaws.com/prod/myapp:v1.2.3
```

### With RBAC Per Repository:

This is the **killer advantage** of repository-based promotion. You can set up **different permissions per environment**:

```
dev/myapp:
  - Developers: Push + Pull
  - CI/CD: Push + Pull

staging/myapp:
  - Developers: Pull only
  - CI/CD: Push + Pull (automated promotion)
  - QA Team: Pull

prod/myapp:
  - Developers: NO ACCESS
  - CI/CD: Push (promotion only)
  - Production K8s nodes: Pull only
  - SRE Team: Pull
```

This enforces a gate: **developers can't accidentally push directly to production!**

### Pros & Cons:

| Pros | Cons |
|------|------|
| Clear separation of environments | More repos to manage |
| Environment-specific RBAC | Slightly more complex CI/CD |
| Easy to audit (what's in prod?) | Registry storage duplication (same image, multiple repos) |
| Version history preserved per env | Need tooling to copy between repos |

## Strategy 3: Registry-Based Promotion (Separate Registries)

The most secure approach — each environment has its **own registry** (or even its own AWS account).

### How It Works:

```
Build:      dev-registry.company.com/myapp:v1.2.3
Staging:    staging-registry.company.com/myapp:v1.2.3
Production: prod-registry.company.com/myapp:v1.2.3
```

### Multi-Account AWS Example:

```
AWS Account 111111111111 (Dev):
  ECR: 111111111111.dkr.ecr.us-east-1.amazonaws.com/myapp

AWS Account 222222222222 (Staging):
  ECR: 222222222222.dkr.ecr.us-east-1.amazonaws.com/myapp

AWS Account 333333333333 (Production):
  ECR: 333333333333.dkr.ecr.us-east-1.amazonaws.com/myapp
```

```bash
# Promote from Dev account to Staging account
# First, set up cross-account ECR permissions

# In the Staging account's ECR, add this policy:
aws ecr set-repository-policy \
  --repository-name myapp \
  --policy-text '{
    "Version": "2012-10-17",
    "Statement": [
      {
        "Sid": "AllowDevAccountPush",
        "Effect": "Allow",
        "Principal": {
          "AWS": "arn:aws:iam::111111111111:root"
        },
        "Action": [
          "ecr:BatchCheckLayerAvailability",
          "ecr:GetDownloadUrlForLayer",
          "ecr:BatchGetImage",
          "ecr:PutImage"
        ]
      }
    ]
  }'

# Then copy the image
skopeo copy \
  docker://111111111111.dkr.ecr.us-east-1.amazonaws.com/myapp:v1.2.3 \
  docker://222222222222.dkr.ecr.us-east-1.amazonaws.com/myapp:v1.2.3
```

### Pros & Cons:

| Pros | Cons |
|------|------|
| Maximum security isolation | Complex setup (cross-account/cross-registry auth) |
| Production registry is completely separate | Harder to manage credentials |
| Blast radius minimization | More infrastructure to maintain |
| Compliance-friendly | Higher cost (multiple registries) |

## Strategy 4: GitOps-Based Promotion

Instead of moving artifacts, you **change the Git manifests** to point to the promoted artifact. The artifact itself stays in ONE registry — you just update which tag/digest each environment uses.

### How It Works:

```
Registry (single):
  registry.com/myapp:v1.2.3    (digest: sha256:abc123)

Git Repo (environment configs):
├── environments/
│   ├── dev/
│   │   └── values.yaml        ← image.tag: v1.2.3  ✅ deployed
│   ├── staging/
│   │   └── values.yaml        ← image.tag: v1.2.3  ✅ deployed
│   └── production/
│       └── values.yaml        ← image.tag: v1.1.0  (not yet promoted)
```

To promote to production, you simply update the Git manifest:

```yaml
# environments/production/values.yaml
image:
  repository: registry.com/myapp
  tag: "v1.2.3"     # Changed from v1.1.0 → v1.2.3
```

Then ArgoCD / Flux CD detects the change and deploys it!

### Implementation with ArgoCD:

```
git-repo/
├── charts/
│   └── myapp/                 # The Helm chart (shared)
│       ├── Chart.yaml
│       ├── values.yaml        # Base values
│       └── templates/
└── environments/
    ├── dev/
    │   └── values.yaml        # image.tag: v1.2.3
    ├── staging/
    │   └── values.yaml        # image.tag: v1.2.3
    └── production/
        └── values.yaml        # image.tag: v1.2.3 ← PR to change this
```

ArgoCD Applications:

```yaml
# dev-app.yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: myapp-dev
spec:
  source:
    repoURL: https://github.com/company/k8s-manifests.git
    path: charts/myapp
    helm:
      valueFiles:
        - ../../environments/dev/values.yaml
  destination:
    server: https://dev-cluster.example.com
    namespace: myapp

---
# production-app.yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: myapp-production
spec:
  source:
    repoURL: https://github.com/company/k8s-manifests.git
    path: charts/myapp
    helm:
      valueFiles:
        - ../../environments/production/values.yaml
  destination:
    server: https://prod-cluster.example.com
    namespace: myapp
```

### Automated GitOps Promotion with GitHub Actions:

```yaml
# .github/workflows/promote-to-production.yml
name: Promote to Production

on:
  workflow_dispatch:
    inputs:
      version:
        description: 'Version to promote (e.g., v1.2.3)'
        required: true

jobs:
  promote:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Verify image exists and was tested
        run: |
          # Check that the image exists in the registry
          skopeo inspect docker://registry.com/myapp:${{ inputs.version }}

          # Check that it passed staging tests (you could check a status file, API, etc.)
          echo "Verifying staging tests passed..."

      - name: Update production values
        run: |
          # Update the image tag in the production values file
          sed -i 's/tag: ".*"/tag: "${{ inputs.version }}"/' environments/production/values.yaml

      - name: Create Pull Request
        uses: peter-evans/create-pull-request@v5
        with:
          title: "Promote ${{ inputs.version }} to production"
          body: |
            ## Artifact Promotion

            **Version:** ${{ inputs.version }}
            **Target:** Production

            This PR promotes the tested artifact to production.

            ### Checklist
            - [ ] Dev tests passed
            - [ ] Staging tests passed
            - [ ] Security scan clean
            - [ ] Product owner approved
          branch: promote/${{ inputs.version }}-to-prod
          commit-message: "chore: promote ${{ inputs.version }} to production"
```

**The beauty of this approach:**
- Promotion is a **Git commit** — fully auditable
- Production changes go through **Pull Request review**
- ArgoCD automatically deploys after merge
- **Rollback** = revert the Git commit!

### Pros & Cons:

| Pros | Cons |
|------|------|
| Full audit trail in Git | Requires GitOps tool (ArgoCD/Flux) |
| PR-based approvals | More complex initial setup |
| Easy rollback (git revert) | Image must be accessible from all clusters |
| Declarative & version-controlled | Learning curve for GitOps |
| Best for Kubernetes | Less applicable outside K8s |

## 🔒 Promotion Gates — Quality Checks Before Promotion

You shouldn't promote blindly! Add **gates** between environments:

### Common Promotion Gates:

```
BUILD
  ↓
  Gate 1: Unit Tests          ← Must pass 100%
  Gate 2: Linting             ← Code quality
  Gate 3: SAST Scan           ← Security (static analysis)
  Gate 4: Image Scan          ← Vulnerability check (Trivy, Grype)
  ↓
DEV ENVIRONMENT
  ↓
  Gate 5: Integration Tests   ← API tests, DB tests
  Gate 6: Smoke Tests         ← Basic health checks
  ↓
STAGING ENVIRONMENT
  ↓
  Gate 7: E2E Tests           ← Full end-to-end tests
  Gate 8: Performance Tests   ← Load testing (k6, JMeter)
  Gate 9: Security Scan       ← DAST (dynamic analysis)
  Gate 10: Manual QA          ← Human review (optional)
  ↓
PRODUCTION
```

### Implementation Example:

```yaml
# .github/workflows/promotion-pipeline.yml
name: Build, Test & Promote

on:
  push:
    branches: [main]

env:
  IMAGE: ghcr.io/${{ github.repository }}/myapp

jobs:
  # ─── Stage 1: Build ────────────────────────────────────
  build:
    runs-on: ubuntu-latest
    permissions:
      packages: write
    outputs:
      version: ${{ steps.version.outputs.VERSION }}
      digest: ${{ steps.build.outputs.digest }}
    steps:
      - uses: actions/checkout@v4

      - name: Generate Version
        id: version
        run: echo "VERSION=$(date +%Y%m%d)-${GITHUB_SHA::8}" >> $GITHUB_OUTPUT

      - name: Build & Push Image
        id: build
        uses: docker/build-push-action@v5
        with:
          push: true
          tags: ${{ env.IMAGE }}:${{ steps.version.outputs.VERSION }}

  # ─── Stage 2: Security Scan (Gate) ─────────────────────
  security-scan:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - name: Run Trivy vulnerability scanner
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: ${{ env.IMAGE }}:${{ needs.build.outputs.version }}
          exit-code: '1'              # Fail if vulnerabilities found
          severity: 'CRITICAL,HIGH'   # Block on Critical and High

  # ─── Stage 3: Deploy to Dev ────────────────────────────
  deploy-dev:
    needs: [build, security-scan]
    runs-on: ubuntu-latest
    environment: dev                  # GitHub Environment (optional approval)
    steps:
      - name: Deploy to Dev
        run: |
          echo "Deploying ${{ env.IMAGE }}:${{ needs.build.outputs.version }} to Dev"
          # kubectl set image deployment/myapp myapp=$IMAGE:$VERSION
          # or update Helm values, ArgoCD sync, etc.

  # ─── Stage 4: Integration Tests (Gate) ─────────────────
  integration-tests:
    needs: deploy-dev
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run Integration Tests
        run: |
          echo "Running integration tests against dev environment..."
          # pytest tests/integration/ --base-url=https://dev.example.com
          # npm run test:integration

  # ─── Stage 5: Promote to Staging ───────────────────────
  promote-staging:
    needs: [build, integration-tests]
    runs-on: ubuntu-latest
    environment: staging              # Requires approval if configured
    steps:
      - name: Promote to Staging
        run: |
          # Option A: Tag-based promotion
          skopeo copy \
            docker://${{ env.IMAGE }}:${{ needs.build.outputs.version }} \
            docker://${{ env.IMAGE }}:staging

          # Option B: Repo-based promotion
          # skopeo copy \
          #   docker://registry.com/dev/myapp:$VERSION \
          #   docker://registry.com/staging/myapp:$VERSION

  # ─── Stage 6: E2E Tests (Gate) ─────────────────────────
  e2e-tests:
    needs: promote-staging
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run E2E Tests
        run: |
          echo "Running E2E tests against staging..."
          # npx playwright test
          # cypress run --config baseUrl=https://staging.example.com

  # ─── Stage 7: Promote to Production ────────────────────
  promote-production:
    needs: [build, e2e-tests]
    runs-on: ubuntu-latest
    environment: production           # MUST require manual approval!
    steps:
      - name: Promote to Production
        run: |
          skopeo copy \
            docker://${{ env.IMAGE }}:${{ needs.build.outputs.version }} \
            docker://${{ env.IMAGE }}:production

          echo "✅ ${{ env.IMAGE }}:${{ needs.build.outputs.version }} promoted to production!"
```

**Key things to notice:**
- The image is built **once** in the `build` job
- Every subsequent stage uses the **same image** (same digest)
- **GitHub Environments** (`dev`, `staging`, `production`) can require manual approvals
- The production promotion **requires explicit approval** — no accidental deploys!

## 🏷️ Tagging Strategies for Promotion

How you tag your images matters. Here are common patterns:

### Strategy A: Semantic Version + Environment Tags

```
myapp:v1.2.3           ← Immutable version tag (never changes)
myapp:dev              ← Points to current dev image (mutable)
myapp:staging          ← Points to current staging image (mutable)
myapp:production       ← Points to current production image (mutable)
```

### Strategy B: Git SHA + Environment

```
myapp:abc123f          ← Git commit SHA (immutable)
myapp:dev-abc123f      ← Deployed to dev
myapp:staging-abc123f  ← Promoted to staging
myapp:prod-abc123f     ← Promoted to production
```

### Strategy C: Timestamp + Build Number

```
myapp:20240615-42      ← Date + build number
myapp:latest-dev       ← Current dev
myapp:latest-staging   ← Current staging
myapp:latest-prod      ← Current prod
```

### Recommended: Version + Digest

```yaml
# In your Kubernetes manifests or Helm values:

# ❌ Don't use mutable tags for production
image: myapp:latest
image: myapp:production

# ✅ Use immutable references
image: myapp:v1.2.3
image: myapp@sha256:abc123def456789...   # Best — digest is truly immutable
```

## 🛠️ Promotion with JFrog Artifactory

JFrog Artifactory has **built-in promotion** as a first-class feature:

### Artifactory Promotion API:

```bash
# Promote a Docker image from dev-docker-local to staging-docker-local
curl -X POST "https://artifactory.company.com/api/docker/dev-docker-local/v2/promote" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $JFROG_TOKEN" \
  -d '{
    "targetRepo": "staging-docker-local",
    "dockerRepository": "myapp",
    "tag": "v1.2.3",
    "copy": true
  }'
```

### Artifactory Build Promotion:

```bash
# Promote an entire build (all artifacts produced by a CI build)
curl -X POST "https://artifactory.company.com/api/build/promote/myapp-build/42" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $JFROG_TOKEN" \
  -d '{
    "status": "staged",
    "comment": "Promoted after passing integration tests",
    "ciUser": "jenkins",
    "targetRepo": "staging-docker-local",
    "copy": true,
    "artifacts": true,
    "dependencies": false
  }'
```

**What makes Artifactory special for promotion:**
- Promotion is a **metadata operation** — no actual data copy (fast!)
- Full **promotion history** — who promoted what, when, and why
- **Build info** — tracks which source code, dependencies, and build steps produced the artifact
- **Xray integration** — block promotion if vulnerabilities found

A critical part of promotion is knowing **the full story of an artifact**:

### What to Track:

```yaml
# Artifact Promotion Record
artifact:
  name: myapp
  version: v1.2.3
  digest: sha256:abc123def456...
  
build:
  source_repo: https://github.com/company/myapp
  commit: abc123f
  branch: main
  build_number: 42
  build_time: "2024-06-15T10:30:00Z"
  builder: GitHub Actions
  
security:
  trivy_scan: passed
  snyk_scan: passed
  critical_vulns: 0
  high_vulns: 0
  
tests:
  unit_tests: 142/142 passed
  integration_tests: 38/38 passed
  e2e_tests: 25/25 passed
  
promotion_history:
  - env: dev
    promoted_at: "2024-06-15T10:35:00Z"
    promoted_by: ci-bot
  - env: staging
    promoted_at: "2024-06-15T14:20:00Z"
    promoted_by: ci-bot
    approved_by: jane@company.com
  - env: production
    promoted_at: "2024-06-16T09:00:00Z"
    promoted_by: ci-bot
    approved_by: john@company.com

```

### OCI Annotations for Metadata:

You can attach metadata directly to OCI artifacts:

```bash
# When building with Docker BuildKit
```

## 🆚 Promotion Strategies Comparison

| Strategy | Complexity | Security | Traceability | Best For |
|----------|-----------|----------|-------------|----------|
| **Tag-Based** | ⭐ Simple | ⭐⭐ Medium | ⭐⭐ Medium | Small teams, simple setups |
| **Repository-Based** | ⭐⭐ Medium | ⭐⭐⭐ Good | ⭐⭐⭐ Good | Medium teams, single registry |
| **Registry-Based** | ⭐⭐⭐ Complex | ⭐⭐⭐⭐ Excellent | ⭐⭐⭐ Good | Enterprise, compliance-heavy |
| **GitOps-Based** | ⭐⭐ Medium | ⭐⭐⭐⭐ Excellent | ⭐⭐⭐⭐ Excellent | Kubernetes-native teams |

### Decision Guide:

```
How many environments do you have?
├── 2 (dev + prod) → Tag-Based promotion
├── 3+ (dev + staging + prod)
│   ├── Using Kubernetes + GitOps? → GitOps-Based promotion
│   ├── Need strict RBAC per env? → Repository-Based promotion
│   └── Enterprise / compliance? → Registry-Based promotion
└── Complex multi-region / multi-cloud → Registry-Based + GitOps combo
```

## ✅ Artifact Promotion Best Practices

### 1. Build Once, Promote Everywhere

```
✅ Build image ONCE → promote through environments
❌ Rebuild for each environment
```

### 2. Use Immutable Tags

```
✅ myapp:v1.2.3 (never overwrite this tag)
✅ myapp@sha256:abc123... (digest — truly immutable)
❌ myapp:latest (what version is this?!)
```

### 3. Separate Config from Artifact

```
✅ Same image, different config per environment
   myapp:v1.2.3 + dev-config.yaml
   myapp:v1.2.3 + staging-config.yaml
   myapp:v1.2.3 + prod-config.yaml

❌ Baking environment config INTO the image
   myapp:v1.2.3-dev    (different image per env)
   myapp:v1.2.3-prod   (defeats the purpose!)
```

### 4. Require Approval for Production

```
✅ Production promotion requires human approval (PR review, manual gate)
❌ Fully automated promotion to production (risky!)
```

### 5. Verify Digest After Promotion

```bash
# Always verify the digest matches after promotion
SOURCE_DIGEST=$(skopeo inspect docker://registry.com/dev/myapp:v1.2.3 | jq -r '.Digest')
TARGET_DIGEST=$(skopeo inspect docker://registry.com/prod/myapp:v1.2.3 | jq -r '.Digest')

[ "$SOURCE_DIGEST" = "$TARGET_DIGEST" ] && echo "✅ Match!" || echo "❌ Mismatch!"
```

### 6. Sign Promoted Artifacts

```bash
# Sign with Cosign after promotion to production
cosign sign --key cosign.key registry.com/prod/myapp:v1.2.3

# Kubernetes admission controller verifies the signature before allowing deployment
```

### 7. Maintain an Audit Trail

```
Every promotion should record:
- WHO promoted (user or CI pipeline)
- WHAT was promoted (image + digest)
- WHEN it was promoted (timestamp)
- WHERE it was promoted from/to (source → target env)
- WHY it was promoted (linked to ticket/PR/approval)
```

## 🎯 Summary

| Concept | Key Takeaway |
|---------|-------------|
| **Artifact Promotion** | Moving a tested artifact through environments WITHOUT rebuilding |
| **Build Once** | Build the image ONCE, promote the same digest everywhere |
| **Tag-Based** | Add environment tags to the same image — simple but limited |
| **Repository-Based** | Copy images to env-specific repos — good RBAC isolation |
| **Registry-Based** | Separate registries per environment — maximum security |
| **GitOps-Based** | Update Git manifests to point to promoted artifact — best for K8s |
| **Promotion Gates** | Quality checks (tests, scans) must pass before promotion |
| **Immutable Tags** | Use version tags or digests, never overwrite tags |
| **Config Separation** | Same image + different config per environment |
| **Audit Trail** | Track who, what, when, where, why for every promotion |
| **Image Signing** | Sign production artifacts with Cosign/Notary |
| **Skopeo** | Best tool for copying images between registries without Docker daemon |
