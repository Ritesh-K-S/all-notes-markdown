# 21.2 — HashiCorp Vault — Architecture, Secrets Engines, Auth Methods

> **One Line Answer:** **HashiCorp Vault** is an open-source tool for **securely storing, accessing, and managing secrets** (passwords, API keys, certificates, encryption keys) — with powerful features like **dynamic secrets**, **encryption as a service**, **identity-based access**, and **detailed audit logging.**

---

## 🤔 What is HashiCorp Vault? (In Simple Words)

Think of Vault as a **super-secure digital safe** for all your secrets.

But unlike a regular safe, this one:
- **Knows who you are** before opening (authentication)
- **Only shows you the stuff you're allowed to see** (authorization / policies)
- **Logs every single time someone opens a drawer** (audit logging)
- **Can create temporary keys that self-destruct** (dynamic secrets)
- **Can encrypt your data for you** without you handling the keys (encryption as a service)

```
Without Vault:
┌──────────────────────────────────────────────┐
│  Secrets scattered everywhere:               │
│                                               │
│  📁 .env files on servers                    │
│  📝 Confluence pages with passwords          │
│  💬 Slack messages: "Hey, DB pass is abc123" │
│  🐳 Docker ENV variables                    │
│  ☁️ Cloud console copy-paste                 │
│                                               │
│  No audit trail. No rotation. No control.    │
└──────────────────────────────────────────────┘

With Vault:
┌──────────────────────────────────────────────┐
│  One secure, centralized place:              │
│                                               │
│  🔐 All secrets in Vault (encrypted)        │
│  👤 Apps authenticate → get only their stuff │
│  📊 Full audit log of every access           │
│  🔄 Auto-rotation of credentials             │
│  ⏰ Dynamic secrets (expire automatically)   │
│                                               │
│  Total control. Total visibility.            │
└──────────────────────────────────────────────┘
```

## 🧩 Component 1: The API Server

Everything in Vault happens through its **HTTP API**. Whether you use:
- The **CLI** (`vault` command)
- The **Web UI** (browser)
- An **application** (using SDK/library)
- **curl** commands

...they ALL talk to the same API server.

```bash
# All of these do the same thing:

# CLI
vault kv get secret/myapp/db-password

# curl (API)
curl -H "X-Vault-Token: s.xyz123" \
     https://vault.example.com/v1/secret/data/myapp/db-password

# Web UI → Just click in the browser
```

**Default Port:** Vault listens on port **8200** by default.

## 🧩 Component 2: The Seal / Unseal Mechanism

This is one of the most unique things about Vault. When Vault starts, it's **sealed** — meaning it **cannot read or write any secrets**, even though the data is right there in storage.

### Why?

Because the data in storage is **encrypted**. To decrypt it, Vault needs a **master key**. But Vault **never stores the master key** — that would be too risky.

### How Unsealing Works:

```
When Vault is first initialized:
├── It generates a Master Key
├── The Master Key is split into N "key shares" using Shamir's Secret Sharing
├── You need K out of N shares to reconstruct the master key (e.g., 3 out of 5)
└── The master key decrypts the encryption key that protects the data

Example: 5 key shares distributed to 5 different people
         Need 3 of them to unseal Vault
         No single person can unseal alone
```

```
┌───────────────────────────────────────────────┐
│              VAULT SEALED STATE               │
│                                                │
│   Storage: [████ ENCRYPTED DATA ████]          │
│   Master Key: NOT available                    │
│   Status: Cannot read or write anything        │
│                                                │
│   To unseal: Provide 3 of 5 key shares         │
└───────────────────────────────────────────────┘

    Key Share 1 ──┐
    Key Share 3 ──┼──▶ Master Key Reconstructed
    Key Share 5 ──┘

┌───────────────────────────────────────────────┐
│             VAULT UNSEALED STATE              │
│                                                │
│   Storage: [Decrypted & Accessible]            │
│   Master Key: In memory (never on disk)        │
│   Status: Fully operational                    │
│                                                │
└───────────────────────────────────────────────┘
```

### Vault Init & Unseal Commands:

```bash
# Initialize Vault (only done ONCE, ever)
vault operator init -key-shares=5 -key-threshold=3

# Output:
# Unseal Key 1: abc123...
# Unseal Key 2: def456...
# Unseal Key 3: ghi789...
# Unseal Key 4: jkl012...
# Unseal Key 5: mno345...
# Initial Root Token: s.pqr678...
#
# ⚠️ SAVE THESE KEYS SECURELY! You will NEVER see them again!

# Unseal Vault (need 3 keys)
vault operator unseal abc123...
vault operator unseal def456...
vault operator unseal ghi789...

# Check seal status
vault status
```

### Auto-Unseal (Production Recommended):

In production, manually unsealing is impractical. You can use **Auto-Unseal** with a cloud KMS:

```hcl
# vault-config.hcl — Auto-Unseal with AWS KMS
seal "awskms" {
  region     = "us-east-1"
  kms_key_id = "alias/vault-unseal-key"
}
```

This means Vault uses AWS KMS (or Azure Key Vault, or GCP KMS) to automatically unseal itself when it starts.

## 🧩 Component 3: Storage Backend

This is where Vault **stores all its encrypted data**. The data is **always encrypted** before it reaches the storage backend, so the storage itself doesn't need to be secure — Vault handles the encryption.

### Common Storage Backends:

| Backend | Type | Best For |
|---------|------|----------|
| **Raft** (Integrated Storage) | Built-in | Production (recommended for most setups) |
| **Consul** | External | Production (HashiCorp's other tool) |
| **S3** | External | Simple storage (not HA) |
| **MySQL / PostgreSQL** | External | If you already have a DB |
| **In-Memory** | Built-in | Development only (data lost on restart) |
| **File** | Built-in | Development only (single node) |

### Raft Storage (Recommended):

```hcl
# vault-config.hcl
storage "raft" {
  path    = "/opt/vault/data"
  node_id = "vault-node-1"
}

listener "tcp" {
  address     = "0.0.0.0:8200"
  tls_disable = false
  tls_cert_file = "/opt/vault/tls/vault-cert.pem"
  tls_key_file  = "/opt/vault/tls/vault-key.pem"
}

api_addr     = "https://vault.example.com:8200"
cluster_addr = "https://vault-node-1:8201"
```

**Why Raft is Recommended:**
- No external dependencies (no need for Consul or a database)
- Built-in High Availability (HA) with leader election
- Data is replicated across all Vault nodes
- Easier to operate and manage

## 🧩 Component 4: Auth Methods — "Who Are You?"

Before anyone (human or machine) can access secrets, they must **authenticate** — prove who they are. Vault supports MANY auth methods:

### Auth Methods at a Glance:

| Auth Method | Who Uses It | How It Works |
|-------------|-------------|--------------|
| **Token** | Everyone (default) | Direct token-based auth. Every other method eventually gives you a token. |
| **Userpass** | Humans | Username + Password (simple, for dev/testing) |
| **LDAP** | Humans | Authenticate against Active Directory / LDAP |
| **OIDC / JWT** | Humans | SSO via Okta, Auth0, Azure AD, Google, etc. |
| **AppRole** | Applications | Role ID + Secret ID (designed for machines) |
| **Kubernetes** | Pods in K8s | K8s Service Account Token → Vault Token |
| **AWS IAM** | EC2/Lambda/ECS | AWS IAM role → Vault Token |
| **Azure** | Azure VMs/Functions | Azure Managed Identity → Vault Token |
| **GCP** | GCE/GKE | GCP Service Account → Vault Token |
| **TLS Certificates** | Machines | Client TLS cert → Vault Token |
| **GitHub** | Developers | GitHub personal access token → Vault Token |

### Let's Set Up a Few Auth Methods:

#### 1. Token Auth (Default — Always Enabled)

```bash
# Login with root token (only for initial setup!)
vault login s.roottoken123

# Create a new token with specific policies
vault token create -policy="app-readonly" -ttl=1h

# Output:
# Key                  Value
# ---                  -----
# token                s.newtoken456
# token_accessor       abc123
# token_duration       1h
# token_renewable      true
# token_policies       ["app-readonly" "default"]
```

#### 2. Userpass Auth (For Humans)

```bash
# Enable userpass auth method
vault auth enable userpass

# Create a user
vault write auth/userpass/users/ritesh \
    password="MySecurePass123" \
    policies="dev-team"

# Login as that user
vault login -method=userpass username=ritesh password="MySecurePass123"

# Output:
# Key                    Value
# ---                    -----
# token                  s.usertoken789
# token_policies         ["default" "dev-team"]
# token_duration         768h
```

#### 3. AppRole Auth (For Applications — Most Common for Automation)

This is the **recommended method for applications and CI/CD pipelines**. It's like a username/password system but designed for machines.

```bash
# Enable AppRole auth method
vault auth enable approle

# Create a role for your application
vault write auth/approle/role/my-web-app \
    token_ttl=1h \
    token_max_ttl=4h \
    token_policies="web-app-policy" \
    secret_id_ttl=10m \
    secret_id_num_uses=1

# Get the Role ID (like a username — stable, doesn't change)
vault read auth/approle/role/my-web-app/role-id
# role_id: 1234-5678-abcd-efgh

# Generate a Secret ID (like a password — one-time use)
vault write -f auth/approle/role/my-web-app/secret-id
# secret_id: ijkl-mnop-qrst-uvwx

# Application logs in using both:
vault write auth/approle/login \
    role_id="1234-5678-abcd-efgh" \
    secret_id="ijkl-mnop-qrst-uvwx"

# Output:
# Key                     Value
# ---                     -----
# token                   s.apptoken012
# token_policies          ["default" "web-app-policy"]
# token_duration          1h
```

**How AppRole works in practice:**

```
┌──────────────┐     1. Deploy Role ID        ┌──────────────┐
│   CI/CD      │◄─────────────────────────────►│    Vault      │
│   Pipeline   │     2. Generate Secret ID      │              │
│              │     3. Login (RoleID+SecretID) │              │
│              │     4. Receive Token            │              │
└──────┬───────┘                                └──────────────┘
       │
       │ 5. Pass token to app
       ▼
┌──────────────┐     6. Use token to read       ┌──────────────┐
│  Application │        secrets                  │    Vault      │
│              │────────────────────────────────►│              │
│              │◄───────────────────────────────│              │
│              │     7. Receives secrets          │              │
└──────────────┘                                └──────────────┘
```

#### 4. Kubernetes Auth (For K8s Pods)

```bash
# Enable Kubernetes auth
vault auth enable kubernetes

# Configure it to talk to the K8s API server
vault write auth/kubernetes/config \
    kubernetes_host="https://kubernetes.default.svc:443" \
    kubernetes_ca_cert=@/var/run/secrets/kubernetes.io/serviceaccount/ca.crt \
    token_reviewer_jwt=@/var/run/secrets/kubernetes.io/serviceaccount/token

# Create a role that maps K8s service accounts to Vault policies
vault write auth/kubernetes/role/my-app \
    bound_service_account_names=my-app-sa \
    bound_service_account_namespaces=production \
    policies=my-app-policy \
    ttl=1h
```

Now any pod running with the `my-app-sa` service account in the `production` namespace can authenticate to Vault and get secrets. We'll cover the full Kubernetes integration in chapter 21.4.

## 🧩 Component 5: Policies — "Are You Allowed to Do This?"

After authenticating (proving WHO you are), Vault checks **policies** to determine WHAT you're allowed to do. Policies are written in **HCL (HashiCorp Configuration Language)** and are **path-based**.

### Policy Syntax:

```hcl
# policy: dev-team.hcl

# Allow reading secrets under secret/data/dev/*
path "secret/data/dev/*" {
  capabilities = ["read", "list"]
}

# Allow full access to the dev team's own space
path "secret/data/dev/my-team/*" {
  capabilities = ["create", "update", "read", "delete", "list"]
}

# Deny access to production secrets (explicit deny)
path "secret/data/prod/*" {
  capabilities = ["deny"]
}
```

### Available Capabilities:

| Capability | HTTP Method | What It Does |
|------------|-------------|--------------|
| `create` | POST | Create new secrets |
| `read` | GET | Read existing secrets |
| `update` | POST/PUT | Update existing secrets |
| `delete` | DELETE | Delete secrets |
| `list` | LIST | List secrets at a path |
| `deny` | — | Explicitly deny access (overrides everything) |
| `sudo` | — | Allows access to root-protected paths |

### Writing and Applying Policies:

```bash
# Write a policy file
cat > web-app-policy.hcl << EOF
# Web app can only read its own secrets
path "secret/data/apps/web-app/*" {
  capabilities = ["read", "list"]
}

# Web app can read database credentials
path "database/creds/web-app-role" {
  capabilities = ["read"]
}

# Web app can encrypt/decrypt using transit engine
path "transit/encrypt/web-app-key" {
  capabilities = ["update"]
}
path "transit/decrypt/web-app-key" {
  capabilities = ["update"]
}
EOF

# Apply the policy to Vault
vault policy write web-app-policy web-app-policy.hcl

# List all policies
vault policy list

# Read a specific policy
vault policy read web-app-policy
```

### Built-in Policies:

| Policy | Description |
|--------|-------------|
| `root` | God mode — can do absolutely everything. Use ONLY for initial setup. |
| `default` | Attached to every token. Allows token self-management (lookup, renew). |

**⚠️ Rule of Thumb:** Never use the root token for daily operations. Create specific policies with **least privilege**.

## 🧩 Component 6: Secrets Engines — "Where Secrets Live"

Secrets engines are the **core of Vault** — they're pluggable components that store, generate, or encrypt data. Each engine is mounted at a **path** and handles a specific type of secret.

### Types of Secrets Engines:

```
┌─────────────────────────────────────────────────────────────┐
│                    SECRETS ENGINES                            │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  📦 KV (Key-Value)     → Store static secrets (passwords,   │
│                          API keys, configs)                   │
│                                                              │
│  🗄️ Database           → Generate dynamic DB credentials    │
│                          (MySQL, PostgreSQL, MongoDB, etc.)  │
│                                                              │
│  ☁️ AWS / Azure / GCP  → Generate dynamic cloud credentials │
│                                                              │
│  🔑 PKI                → Generate TLS certificates          │
│                          (your own Certificate Authority!)   │
│                                                              │
│  🔐 Transit            → Encryption as a Service            │
│                          (encrypt/decrypt without seeing key)│
│                                                              │
│  🐰 RabbitMQ           → Dynamic RabbitMQ credentials       │
│                                                              │
│  🔒 SSH                → Sign SSH keys / generate OTPs      │
│                                                              │
│  📝 TOTP               → Generate TOTP codes (like Google   │
│                          Authenticator)                      │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

Let's deep dive into the most important ones:

---

### Secrets Engine 1: KV (Key-Value) — Static Secrets

This is the **simplest and most commonly used** secrets engine. It stores key-value pairs — like a dictionary or a JSON object.

**Two versions exist:**
- **KV v1** — Simple, no versioning
- **KV v2** — With versioning (recommended) — keeps history of changes

```bash
# Enable KV v2 secrets engine at path "secret/"
vault secrets enable -path=secret -version=2 kv

# ─── Write a secret ───
vault kv put secret/myapp/config \
    db_host="db.prod.example.com" \
    db_port="5432" \
    db_user="app_user" \
    db_password="SuperSecret123!"

# Output:
# === Secret Path ===
# secret/data/myapp/config
#
# ======= Metadata =======
# Key                Value
# ---                -----
# created_time       2025-01-15T10:30:00.000Z
# version            1

# ─── Read a secret ───
vault kv get secret/myapp/config

# Output:
# === Secret Path ===
# secret/data/myapp/config
#
# ======= Metadata =======
# Key                Value
# ---                -----
# created_time       2025-01-15T10:30:00.000Z
# version            1
#
# ====== Data ======
# Key            Value
# ---            -----
# db_host        db.prod.example.com
# db_port        5432
# db_user        app_user
# db_password    SuperSecret123!

# ─── Read only a specific field ───
vault kv get -field=db_password secret/myapp/config
# SuperSecret123!

# ─── Get output as JSON (great for scripts) ───
vault kv get -format=json secret/myapp/config

# ─── Update the secret (creates version 2) ───
vault kv put secret/myapp/config \
    db_host="db.prod.example.com" \
    db_port="5432" \
    db_user="app_user" \
    db_password="NewPassword456!"

# ─── Read a specific version ───
vault kv get -version=1 secret/myapp/config   # Gets the old password
vault kv get -version=2 secret/myapp/config   # Gets the new password

# ─── List secrets at a path ───
vault kv list secret/myapp/
# Keys
# ----
# config

# ─── Delete a secret (soft delete — can be recovered) ───
vault kv delete secret/myapp/config

# ─── Undelete (recover) ───
vault kv undelete -versions=2 secret/myapp/config

# ─── Permanently destroy a version ───
vault kv destroy -versions=1 secret/myapp/config
```

#### Using KV Secrets in Applications:

**Python Example:**

```python
import hvac  # pip install hvac

# Connect to Vault
client = hvac.Client(
    url='https://vault.example.com:8200',
    token='s.mytoken123'
)

# Read a secret
secret = client.secrets.kv.v2.read_secret_version(
    path='myapp/config',
    mount_point='secret'
)

db_password = secret['data']['data']['db_password']
db_host = secret['data']['data']['db_host']

print(f"Connecting to {db_host} with password {db_password}")
```

**curl Example (REST API):**

```bash
# Read a secret via API
curl -s \
    -H "X-Vault-Token: s.mytoken123" \
    https://vault.example.com:8200/v1/secret/data/myapp/config | jq .

# Write a secret via API
curl -s \
    -H "X-Vault-Token: s.mytoken123" \
    -X POST \
    -d '{"data": {"username": "admin", "password": "secret123"}}' \
    https://vault.example.com:8200/v1/secret/data/myapp/new-secret
```

### Secrets Engine 2: Database — Dynamic Credentials

This is where Vault gets **really powerful**. Instead of storing a static database password, Vault **creates a temporary database user on-demand** with a **time-limited password** that expires automatically.

We'll cover this in detail in chapter **21.3**, but here's a quick taste:

```bash
# Enable database secrets engine
vault secrets enable database

# Configure connection to PostgreSQL
vault write database/config/my-postgres \
    plugin_name=postgresql-database-plugin \
    connection_url="postgresql://{{username}}:{{password}}@db.example.com:5432/mydb" \
    allowed_roles="my-app-role" \
    username="vault_admin" \
    password="vault_admin_password"

# Create a role that defines what the dynamic user looks like
vault write database/roles/my-app-role \
    db_name=my-postgres \
    creation_statements="CREATE ROLE \"{{name}}\" WITH LOGIN PASSWORD '{{password}}' VALID UNTIL '{{expiration}}'; GRANT SELECT ON ALL TABLES IN SCHEMA public TO \"{{name}}\";" \
    default_ttl="1h" \
    max_ttl="24h"

# Now any app can request temporary credentials:
vault read database/creds/my-app-role

# Output:
# Key                Value
# ---                -----
# lease_id           database/creds/my-app-role/abc123
# lease_duration     1h
# username           v-approle-my-app-r-abc123xyz
# password           A1b2C3d4E5f6G7h8-randomized
#
# ⏰ This user will be AUTOMATICALLY DELETED after 1 hour!
```

### Secrets Engine 3: Transit — Encryption as a Service

The Transit engine lets your applications **encrypt and decrypt data** without ever seeing or managing encryption keys. Vault handles all the cryptography.

```bash
# Enable transit secrets engine
vault secrets enable transit

# Create an encryption key
vault write -f transit/keys/my-app-key

# Encrypt data (must be base64 encoded)
vault write transit/encrypt/my-app-key \
    plaintext=$(echo -n "my secret credit card: 4111-1111-1111-1111" | base64)

# Output:
# Key           Value
# ---           -----
# ciphertext    vault:v1:abc123encrypted456data789...

# Decrypt data
vault write transit/decrypt/my-app-key \
    ciphertext="vault:v1:abc123encrypted456data789..."

# Output:
# Key          Value
# ---          -----
# plaintext    bXkgc2VjcmV0IGNyZWRpdCBjYXJkOiA0MTExLTExMTEtMTExMS0xMTEx

# Decode the base64 result
echo "bXkgc2VjcmV0IGNyZWRpdCBjYXJkOiA0MTExLTExMTEtMTExMS0xMTEx" | base64 --decode
# my secret credit card: 4111-1111-1111-1111
```

**Why is this amazing?**
- Your app **never touches** the encryption key
- You can **rotate keys** without re-encrypting data
- Vault handles the **cryptography** — no chance of your devs implementing it wrong
- Full **audit trail** of every encrypt/decrypt operation

## 🧩 Component 7: Audit Logging — "I'm Recording Everything"

Every single request to Vault (successful or failed) can be recorded in an audit log. This is **critical** for compliance and security.

```bash
# Enable file-based audit logging
vault audit enable file file_path=/var/log/vault/audit.log

# Enable syslog-based audit logging
vault audit enable syslog tag="vault" facility="AUTH"

# List enabled audit devices
vault audit list
```

### What an Audit Log Entry Looks Like:

```json
{
  "time": "2025-01-15T10:30:00.000Z",
  "type": "response",
  "auth": {
    "client_token": "hmac-sha256:abc123...",
    "accessor": "hmac-sha256:def456...",
    "display_name": "userpass-ritesh",
    "policies": ["default", "dev-team"],
    "token_type": "service"
  },
  "request": {
    "id": "request-uuid-123",
    "operation": "read",
    "path": "secret/data/myapp/config",
    "remote_address": "10.0.1.50"
  },
  "response": {
    "data": {
      "db_password": "hmac-sha256:ghi789..."
    }
  }
}
```

**Notice:** The actual secret values are **HMAC'd** (hashed) in the audit log — so even the audit log doesn't contain plaintext secrets! You can see WHO accessed WHAT, but not the actual values.

## 🚀 Hands-On: Setting Up Vault (Dev Mode)

Let's get Vault running and try everything we learned. We'll use **dev mode** for learning (never use dev mode in production!).

### Step 1: Install Vault

```bash
# macOS
brew install vault

# Ubuntu/Debian
curl -fsSL https://apt.releases.hashicorp.com/gpg | sudo apt-key add -
sudo apt-add-repository "deb [arch=amd64] https://apt.releases.hashicorp.com $(lsb_release -cs) main"
sudo apt update && sudo apt install vault

# Docker (easiest for trying out)
docker run -d \
    --name vault \
    -p 8200:8200 \
    -e 'VAULT_DEV_ROOT_TOKEN_ID=myroot' \
    -e 'VAULT_DEV_LISTEN_ADDRESS=0.0.0.0:8200' \
    hashicorp/vault:latest

# Verify installation
vault --version
# Vault v1.15.4
```

### Step 2: Start Vault in Dev Mode

```bash
# Start dev server (auto-unsealed, in-memory storage, root token provided)
vault server -dev -dev-root-token-id="myroot"

# In another terminal, set environment variables:
export VAULT_ADDR='http://127.0.0.1:8200'
export VAULT_TOKEN='myroot'

# Check status
vault status
# Key             Value
# ---             -----
# Seal Type       shamir
# Initialized     true
# Sealed          false    ← Already unsealed (dev mode)
# ...
```

### Step 3: Store and Retrieve Secrets

```bash
# Write a secret
vault kv put secret/hello greeting="world" name="DevOps Engineer"

# Read it back
vault kv get secret/hello

# Output:
# ====== Data ======
# Key         Value
# ---         -----
# greeting    world
# name        DevOps Engineer

# Read as JSON
vault kv get -format=json secret/hello
```

### Step 4: Create a Policy and User

```bash
# Create a policy file
cat > my-policy.hcl << EOF
path "secret/data/hello" {
  capabilities = ["read"]
}
path "secret/data/myapp/*" {
  capabilities = ["read", "list"]
}
EOF

# Apply the policy
vault policy write my-policy my-policy.hcl

# Enable userpass auth and create a user with this policy
vault auth enable userpass
vault write auth/userpass/users/testuser \
    password="testpass123" \
    policies="my-policy"

# Login as the new user
vault login -method=userpass username=testuser password="testpass123"

# Try to read the allowed secret
vault kv get secret/hello
# ✅ Works!

# Try to read something not allowed
vault kv get secret/admin/config
# ❌ Error: permission denied

# Try to write (not allowed by policy)
vault kv put secret/hello greeting="hacked"
# ❌ Error: permission denied
```

## 🏢 Production Architecture — High Availability (HA)

In production, you want Vault to be **highly available** so it doesn't become a single point of failure:

```
                        ┌─────────────────┐
                        │   Load Balancer  │
                        │  (Vault:8200)    │
                        └────────┬────────┘
                                 │
                 ┌───────────────┼───────────────┐
                 │               │               │
                 ▼               ▼               ▼
          ┌──────────┐   ┌──────────┐   ┌──────────┐
          │ Vault #1 │   │ Vault #2 │   │ Vault #3 │
          │ (Active) │   │(Standby) │   │(Standby) │
          │ Leader   │   │          │   │          │
          └────┬─────┘   └────┬─────┘   └────┬─────┘
               │              │              │
               └──────────────┼──────────────┘
                              │
                    ┌─────────────────┐
                    │  Raft Storage   │
                    │  (Replicated    │
                    │   across all    │
                    │   3 nodes)      │
                    └─────────────────┘
```

**Key Points:**
- Only **one node** is the **active/leader** — handles all requests
- **Standby nodes** forward requests to the active node
- If the active node goes down, a standby node is automatically **elected as the new leader**
- All data is **replicated** across nodes via Raft
- Minimum **3 nodes** for HA (to maintain quorum)

### Production Config Example:

```hcl
# /etc/vault.d/vault.hcl

# Storage — Raft (integrated HA)
storage "raft" {
  path    = "/opt/vault/data"
  node_id = "vault-node-1"

  retry_join {
    leader_api_addr = "https://vault-node-2:8200"
  }
  retry_join {
    leader_api_addr = "https://vault-node-3:8200"
  }
}

# Listener — TLS enabled
listener "tcp" {
  address       = "0.0.0.0:8200"
  tls_cert_file = "/opt/vault/tls/vault.crt"
  tls_key_file  = "/opt/vault/tls/vault.key"
}

# Auto-unseal with AWS KMS
seal "awskms" {
  region     = "us-east-1"
  kms_key_id = "alias/vault-unseal-key"
}

# API and cluster addresses
api_addr     = "https://vault-node-1.example.com:8200"
cluster_addr = "https://vault-node-1.example.com:8201"

# Telemetry (for monitoring)
telemetry {
  prometheus_retention_time = "30s"
  disable_hostname          = true
}

# UI
ui = true
```

## 🧠 Key Takeaways

| Concept | Remember |
|---------|----------|
| **Vault = secure secrets store** | Centralized, encrypted, audited, access-controlled |
| **Seal/Unseal** | Vault starts sealed — needs unseal keys to operate |
| **Shamir's Secret Sharing** | Master key split into N shares, need K to unseal |
| **Auto-Unseal** | Use cloud KMS in production (AWS KMS, Azure Key Vault, GCP KMS) |
| **Auth Methods** | How clients prove who they are (Token, AppRole, K8s, LDAP, etc.) |
| **Policies** | Path-based ACLs — define WHO can access WHAT |
| **Secrets Engines** | Pluggable backends (KV, Database, Transit, PKI, etc.) |
| **KV Engine** | Store static key-value secrets with versioning |
| **Transit Engine** | Encryption as a Service — apps never see the key |
| **Database Engine** | Dynamic credentials — auto-generated, auto-expired |
| **Audit Logging** | Every request logged — secrets are HMAC'd |
| **HA with Raft** | 3+ nodes, automatic leader election, replicated storage |
