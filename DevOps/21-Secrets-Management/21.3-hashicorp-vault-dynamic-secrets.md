# 21.3 — HashiCorp Vault — Dynamic Secrets, PKI, Transit Encryption

> **One Line Answer:** Vault's killer features are **dynamic secrets** (auto-generated credentials that expire on their own), **PKI** (your own Certificate Authority to issue TLS certificates), and **Transit encryption** (encrypt/decrypt data without your app ever touching the encryption keys).

---

## 🤔 Why Dynamic Secrets Are a Game-Changer

In chapter 21.2, we used the **KV engine** to store static secrets — passwords that someone manually created and put into Vault. That's useful, but it has problems:

```
Static Secrets — The Old Way:
┌─────────────────────────────────────────────────────┐
│                                                      │
│  1. Human creates DB password: "MyPassword123!"      │
│  2. Stores it in Vault                               │
│  3. 50 apps read the SAME password                   │
│  4. Password lives forever (unless manually rotated) │
│  5. If leaked → ALL 50 apps are compromised          │
│  6. Can't tell which app leaked it                   │
│                                                      │
│  Problems: Shared password, no expiry, no tracing    │
└─────────────────────────────────────────────────────┘

Dynamic Secrets — The Vault Way:
┌─────────────────────────────────────────────────────┐
│                                                      │
│  1. App requests DB credentials from Vault           │
│  2. Vault creates a UNIQUE user/password on the DB   │
│  3. Credentials are valid for only 1 hour            │
│  4. After 1 hour → Vault DELETES the user from DB   │
│  5. Each app gets its OWN credentials                │
│  6. If leaked → revoke just that one credential      │
│                                                      │
│  Benefits: Unique per app, auto-expire, traceable    │
└─────────────────────────────────────────────────────┘
```

**The big idea:** Instead of storing secrets, Vault **generates** them on-demand and **destroys** them when they're no longer needed.

## 🗄️ Part 1: Dynamic Database Secrets

This is the most popular use of dynamic secrets. Vault connects to your database and creates temporary users with temporary passwords.

### How It Works (The Flow):

```
┌──────────┐          ┌──────────┐          ┌──────────────┐
│   App    │    1.    │  Vault   │    2.    │  PostgreSQL  │
│          │─────────▶│          │─────────▶│              │
│          │ "I need  │          │ CREATE   │              │
│          │  DB      │          │ ROLE     │              │
│          │  creds"  │          │ "v-app-  │              │
│          │          │          │  xyz123" │              │
│          │◀─────────│◀─────────│          │              │
│          │    3.    │    4.    │          │              │
│          │ username:│ Role     │          │              │
│          │ v-app-   │ created! │          │              │
│          │  xyz123  │          │          │              │
│          │ password:│          │          │              │
│          │ A1b2C3d4 │          │          │              │
│          │ TTL: 1h  │          │          │              │
└──────────┘          └──────────┘          └──────────────┘

        ⏰ After 1 hour...

┌──────────┐          ┌──────────┐          ┌──────────────┐
│          │          │  Vault   │─────────▶│  PostgreSQL  │
│          │          │          │  DROP    │              │
│          │          │          │  ROLE    │              │
│          │          │          │ "v-app-  │              │
│          │          │          │  xyz123" │              │
│          │          │          │          │              │
└──────────┘          └──────────┘          └──────────────┘
                    Credential auto-deleted!
```

### Step-by-Step Setup: PostgreSQL Dynamic Secrets

#### Step 1: Enable the Database Secrets Engine

```bash
vault secrets enable database
```

#### Step 2: Configure the Database Connection

Vault needs a **privileged** connection to the database so it can create and delete users.

```bash
vault write database/config/my-postgres-db \
    plugin_name="postgresql-database-plugin" \
    allowed_roles="readonly-role,readwrite-role" \
    connection_url="postgresql://{{username}}:{{password}}@db.example.com:5432/myapp?sslmode=require" \
    username="vault_admin" \
    password="VaultAdminPassword123!"
```

**What's happening:**
- `plugin_name` — tells Vault which database plugin to use
- `allowed_roles` — which Vault roles can use this connection
- `connection_url` — how to connect to the database (the `{{username}}` and `{{password}}` are Vault template variables)
- `username/password` — the privileged account Vault uses (must have `CREATEROLE` permission)

**Important:** After configuring, Vault will **rotate** this root password so even you don't know it anymore:

```bash
# Rotate the root password (recommended)
vault write -f database/rotate-root/my-postgres-db
# Now even the admin who set it up can't use the old password!
```

#### Step 3: Create a Role (Template for Dynamic Users)

A **role** defines what kind of database user Vault will create:

```bash
# Read-only role — can only SELECT data
vault write database/roles/readonly-role \
    db_name="my-postgres-db" \
    creation_statements="
        CREATE ROLE \"{{name}}\" WITH LOGIN PASSWORD '{{password}}' VALID UNTIL '{{expiration}}';
        GRANT USAGE ON SCHEMA public TO \"{{name}}\";
        GRANT SELECT ON ALL TABLES IN SCHEMA public TO \"{{name}}\";
    " \
    revocation_statements="
        DROP ROLE IF EXISTS \"{{name}}\";
    " \
    default_ttl="1h" \
    max_ttl="24h"
```

```bash
# Read-write role — can SELECT, INSERT, UPDATE, DELETE
vault write database/roles/readwrite-role \
    db_name="my-postgres-db" \
    creation_statements="
        CREATE ROLE \"{{name}}\" WITH LOGIN PASSWORD '{{password}}' VALID UNTIL '{{expiration}}';
        GRANT USAGE ON SCHEMA public TO \"{{name}}\";
        GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO \"{{name}}\";
    " \
    revocation_statements="
        REVOKE ALL PRIVILEGES ON ALL TABLES IN SCHEMA public FROM \"{{name}}\";
        DROP ROLE IF EXISTS \"{{name}}\";
    " \
    default_ttl="1h" \
    max_ttl="24h"
```

**Template variables Vault replaces:**
| Variable | What It Becomes |
|----------|----------------|
| `{{name}}` | Auto-generated username (e.g., `v-approle-readonly-abc123xyz`) |
| `{{password}}` | Auto-generated strong password |
| `{{expiration}}` | Timestamp when the credential expires |

#### Step 4: Request Dynamic Credentials

Now any authenticated app can request credentials:

```bash
# Request read-only credentials
vault read database/creds/readonly-role

# Output:
# Key                Value
# ---                -----
# lease_id           database/creds/readonly-role/7f3a2b1c-4d5e-6f7g-8h9i-0j1k2l3m4n5o
# lease_duration     1h
# lease_renewable    true
# password           A1b-C3d4E5f6G7h8
# username           v-token-readonly-r-1234567890abcdef
```

```bash
# Request read-write credentials
vault read database/creds/readwrite-role

# Output:
# Key                Value
# ---                -----
# lease_id           database/creds/readwrite-role/9p8o7n6m-5l4k-3j2i-1h0g-fedcba098765
# lease_duration     1h
# lease_renewable    true
# password           X9y-Z0a1B2c3D4e5
# username           v-token-readwrite-1234567890abcdef
```

**Verify the user was created on PostgreSQL:**

```sql
-- Run this on your PostgreSQL server
SELECT usename, valuntil FROM pg_user WHERE usename LIKE 'v-%';

-- Output:
-- usename                              | valuntil
-- ------------------------------------+------------------------
-- v-token-readonly-r-1234567890abcdef  | 2025-01-15 11:30:00+00
-- v-token-readwrite-1234567890abcdef   | 2025-01-15 11:30:00+00
```

After the TTL expires, Vault automatically runs the `revocation_statements` and the user is **gone**.

#### Step 5: Using Dynamic Credentials in an Application

```python
import hvac
import psycopg2

# Connect to Vault
vault_client = hvac.Client(
    url='https://vault.example.com:8200',
    token='s.my-app-token'
)

# Request dynamic database credentials
creds = vault_client.secrets.database.generate_credentials(
    name='readonly-role',
    mount_point='database'
)

db_username = creds['data']['username']
db_password = creds['data']['password']
lease_id = creds['lease_id']
lease_duration = creds['lease_duration']

print(f"Got temporary credentials:")
print(f"  Username: {db_username}")
print(f"  Password: {db_password}")
print(f"  Valid for: {lease_duration} seconds")

# Connect to the database with temporary credentials
conn = psycopg2.connect(
    host="db.example.com",
    database="myapp",
    user=db_username,
    password=db_password
)

# Do your queries
cursor = conn.cursor()
cursor.execute("SELECT * FROM users LIMIT 10")
rows = cursor.fetchall()
for row in rows:
    print(row)

# When done, you can optionally revoke the credential early
vault_client.sys.revoke_lease(lease_id)
print("Credential revoked early — no longer valid!")
```

### Lease Management — Renewing and Revoking

Every dynamic secret has a **lease** — a timer that says "this credential is valid for X time."

```bash
# Renew a lease (extend the TTL)
vault lease renew database/creds/readonly-role/7f3a2b1c-4d5e-6f7g-8h9i-0j1k2l3m4n5o

# Renew with a specific increment
vault lease renew -increment=2h database/creds/readonly-role/7f3a2b1c...

# Revoke a specific lease (delete the DB user immediately)
vault lease revoke database/creds/readonly-role/7f3a2b1c-4d5e-6f7g-8h9i-0j1k2l3m4n5o

# ⚠️ EMERGENCY: Revoke ALL leases for a role (revoke everyone's access)
vault lease revoke -prefix database/creds/readonly-role/

# ⚠️ NUCLEAR: Revoke ALL database leases
vault lease revoke -prefix database/
```

**This is incredibly powerful for incident response:**
```
🚨 BREACH DETECTED!
   ↓
vault lease revoke -prefix database/
   ↓
💥 ALL dynamic database credentials are instantly revoked
   ↓
All apps lose DB access immediately
   ↓
Only legitimate apps can request new credentials
   ↓
Attacker is locked out
```

### Dynamic Secrets for MySQL

The setup is very similar to PostgreSQL:

```bash
# Configure MySQL connection
vault write database/config/my-mysql-db \
    plugin_name="mysql-database-plugin" \
    allowed_roles="mysql-readonly" \
    connection_url="{{username}}:{{password}}@tcp(mysql.example.com:3306)/" \
    username="vault_admin" \
    password="VaultMySQLAdmin123!"

# Create a role
vault write database/roles/mysql-readonly \
    db_name="my-mysql-db" \
    creation_statements="CREATE USER '{{name}}'@'%' IDENTIFIED BY '{{password}}'; GRANT SELECT ON mydb.* TO '{{name}}'@'%';" \
    revocation_statements="DROP USER IF EXISTS '{{name}}'@'%';" \
    default_ttl="1h" \
    max_ttl="24h"

# Request credentials
vault read database/creds/mysql-readonly
```

## ☁️ Part 2: Dynamic Cloud Credentials

Vault can also generate **temporary cloud credentials** — AWS IAM keys, Azure service principals, GCP service account keys — that expire automatically.

### AWS Dynamic Credentials

```bash
# Enable AWS secrets engine
vault secrets enable aws

# Configure with your root/admin AWS credentials
# (Vault will use these to create temporary IAM users)
vault write aws/config/root \
    access_key="AKIAIOSFODNN7EXAMPLE" \
    secret_key="wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY" \
    region="us-east-1"

# Create a role — defines what permissions the temporary IAM user gets
vault write aws/roles/s3-readonly \
    credential_type=iam_user \
    policy_document=-<<EOF
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::my-bucket",
        "arn:aws:s3:::my-bucket/*"
      ]
    }
  ]
}
EOF

# Request temporary AWS credentials
vault read aws/creds/s3-readonly

# Output:
# Key                Value
# ---                -----
# lease_id           aws/creds/s3-readonly/abc123...
# lease_duration     768h
# access_key         AKIAI4EXAMPLE12345
# secret_key         wJalrXUt...EXAMPLEKEY
# security_token     (none for iam_user type)
#
# These credentials will be automatically deleted when the lease expires!
```

#### STS Assumed Role (Even Better — No IAM Users Created):

```bash
# Using STS assumed role (temporary tokens, no IAM user creation needed)
vault write aws/roles/deploy-role \
    credential_type=assumed_role \
    role_arns="arn:aws:iam::123456789012:role/deploy-role" \
    default_sts_ttl="1h" \
    max_sts_ttl="4h"

vault read aws/creds/deploy-role

# Output:
# access_key         ASIATEMP12345
# secret_key         TempSecret...
# security_token     FwoGZXIv...  ← STS session token (temporary)
# ttl                3600
```

## 🔑 Part 3: PKI Secrets Engine — Your Own Certificate Authority

The PKI (Public Key Infrastructure) secrets engine lets you run your **own Certificate Authority (CA)** inside Vault. This means you can **issue TLS certificates** for your services — on demand, automatically, and with proper expiration.

### Why Do You Need This?

```
Without Vault PKI:
┌─────────────────────────────────────────────────────┐
│  1. Submit CSR to an external CA (DigiCert, etc.)   │
│  2. Wait hours/days for approval                    │
│  3. Download the cert, manually install it          │
│  4. Cert valid for 1-2 years                        │
│  5. Forget to renew → 💥 SERVICE OUTAGE             │
│  6. Managing 500 certs manually? Good luck.         │
└─────────────────────────────────────────────────────┘

With Vault PKI:
┌─────────────────────────────────────────────────────┐
│  1. App requests a cert from Vault                  │
│  2. Vault issues it INSTANTLY (milliseconds)        │
│  3. Cert valid for 24 hours (short-lived!)          │
│  4. Auto-renews before expiry                       │
│  5. If compromised → just wait 24h, it expires      │
│  6. Managing 5000 certs? Vault handles it.          │
└─────────────────────────────────────────────────────┘
```

### Step-by-Step: Setting Up a Certificate Authority

#### Step 1: Enable PKI Secrets Engine

```bash
# Enable PKI engine for the Root CA
vault secrets enable -path=pki pki

# Configure maximum TTL (10 years for root CA)
vault secrets tune -max-lease-ttl=87600h pki
```

#### Step 2: Generate the Root CA Certificate

```bash
vault write -field=certificate pki/root/generate/internal \
    common_name="My Company Root CA" \
    issuer_name="root-2025" \
    ttl=87600h > root_ca.crt

# View the root CA cert
openssl x509 -in root_ca.crt -text -noout
```

#### Step 3: Configure CA URLs

```bash
vault write pki/config/urls \
    issuing_certificates="https://vault.example.com:8200/v1/pki/ca" \
    crl_distribution_points="https://vault.example.com:8200/v1/pki/crl"
```

#### Step 4: Create an Intermediate CA (Best Practice)

In real-world PKI, you **never issue end-entity certificates from the root CA**. You create an **Intermediate CA** and issue from that:

```
Root CA (kept super safe, rarely used)
  └── Intermediate CA (issues day-to-day certificates)
        ├── web-server.example.com cert
        ├── api.example.com cert
        └── internal-service.example.com cert
```

```bash
# Enable a second PKI engine for the Intermediate CA
vault secrets enable -path=pki_int pki
vault secrets tune -max-lease-ttl=43800h pki_int

# Generate Intermediate CA CSR
vault write -format=json pki_int/intermediate/generate/internal \
    common_name="My Company Intermediate CA" \
    issuer_name="intermediate-2025" \
    | jq -r '.data.csr' > intermediate.csr

# Sign the Intermediate CA with the Root CA
vault write -format=json pki/root/sign-intermediate \
    csr=@intermediate.csr \
    format=pem_bundle \
    ttl=43800h \
    | jq -r '.data.certificate' > intermediate_ca.crt

# Import the signed Intermediate CA cert back
vault write pki_int/intermediate/set-signed \
    certificate=@intermediate_ca.crt
```

#### Step 5: Create a Role (Template for Issuing Certs)

```bash
vault write pki_int/roles/example-dot-com \
    allowed_domains="example.com" \
    allow_subdomains=true \
    allow_bare_domains=false \
    max_ttl=720h \
    key_type="rsa" \
    key_bits=2048 \
    require_cn=true \
    generate_lease=true
```

**What this means:**
- Can issue certs for `*.example.com` (any subdomain)
- Cannot issue for `example.com` itself (bare domain)
- Max validity: 30 days
- Uses RSA 2048-bit keys

#### Step 6: Issue a Certificate!

```bash
vault write -format=json pki_int/issue/example-dot-com \
    common_name="web-server.example.com" \
    alt_names="www.example.com,api.example.com" \
    ttl=24h

# Output:
# {
#   "data": {
#     "certificate": "-----BEGIN CERTIFICATE-----\nMIID...\n-----END CERTIFICATE-----",
#     "issuing_ca": "-----BEGIN CERTIFICATE-----\nMIID...\n-----END CERTIFICATE-----",
#     "ca_chain": ["-----BEGIN CERTIFICATE-----\n..."],
#     "private_key": "-----BEGIN RSA PRIVATE KEY-----\nMIIE...\n-----END RSA PRIVATE KEY-----",
#     "serial_number": "3a:b1:c2:d3:e4:f5:...",
#     "expiration": 1705406400
#   }
# }
```

#### Save the Certificate and Key:

```bash
# Save to files for use in Nginx, Apache, etc.
vault write -format=json pki_int/issue/example-dot-com \
    common_name="web-server.example.com" \
    ttl=24h | tee \
    >(jq -r '.data.certificate' > server.crt) \
    >(jq -r '.data.private_key' > server.key) \
    >(jq -r '.data.ca_chain[]' > ca-chain.crt)

# Use in Nginx
# nginx.conf:
# ssl_certificate     /etc/nginx/ssl/server.crt;
# ssl_certificate_key /etc/nginx/ssl/server.key;
# ssl_trusted_certificate /etc/nginx/ssl/ca-chain.crt;
```

#### Revoke a Certificate (If Compromised):

```bash
vault write pki_int/revoke \
    serial_number="3a:b1:c2:d3:e4:f5:..."
```

### Real-World Use Case: mTLS (Mutual TLS) Between Microservices

With Vault PKI, every microservice can get its **own TLS certificate**, enabling **mTLS** (both sides verify each other):

```
┌──────────────┐                        ┌──────────────┐
│  Service A   │◄──── mTLS ────────────▶│  Service B   │
│              │                        │              │
│  Has cert:   │  Both sides verify     │  Has cert:   │
│  svc-a.cert  │  each other's cert     │  svc-b.cert  │
│  svc-a.key   │  using the same CA     │  svc-b.key   │
└──────────────┘                        └──────────────┘
       │                                       │
       │        Both certs issued by           │
       └──────────── Vault PKI ────────────────┘
```

```bash
# Cert for Service A
vault write pki_int/issue/example-dot-com \
    common_name="service-a.internal.example.com" \
    ttl=24h

# Cert for Service B
vault write pki_int/issue/example-dot-com \
    common_name="service-b.internal.example.com" \
    ttl=24h
```

## 🔐 Part 4: Transit Secrets Engine — Encryption as a Service

The Transit engine is like having a **cryptography expert built into Vault**. Your apps send data to Vault, Vault encrypts it, and sends back the encrypted version. Your app **never sees** the encryption key.

### Why This is Important

```
Without Transit (DIY Encryption):
┌─────────────────────────────────────────────────────┐
│  1. Developer generates an AES key                  │
│  2. Stores the key... somewhere? (Config file? DB?) │
│  3. Writes encryption code (probably wrong)          │
│  4. Uses ECB mode instead of GCM (insecure!)        │
│  5. Key gets leaked in a commit                     │
│  6. Can't rotate the key without re-encrypting      │
│     ALL data                                         │
│  7. Different apps use different encryption methods  │
│                                                      │
│  Result: Inconsistent, insecure, nightmare           │
└─────────────────────────────────────────────────────┘

With Transit (Encryption as a Service):
┌─────────────────────────────────────────────────────┐
│  1. App sends plaintext to Vault                    │
│  2. Vault encrypts with AES-256-GCM (best practice) │
│  3. Vault returns ciphertext                         │
│  4. App stores ciphertext in its own database       │
│  5. Key never leaves Vault (app never sees it)      │
│  6. Key rotation is automatic — old data still works │
│  7. All apps use the same proven encryption          │
│                                                      │
│  Result: Consistent, secure, simple                  │
└─────────────────────────────────────────────────────┘
```

### Step-by-Step: Setting Up Transit Encryption

#### Step 1: Enable Transit Engine

```bash
vault secrets enable transit
```

#### Step 2: Create Named Encryption Keys

```bash
# Create a key for encrypting user data
vault write -f transit/keys/user-data

# Create a key for encrypting payment info (with auto-rotation)
vault write transit/keys/payment-data \
    type="aes256-gcm96" \
    auto_rotate_period="720h"   # Auto-rotate every 30 days

# View key info
vault read transit/keys/user-data

# Output:
# Key                       Value
# ---                       -----
# allow_plaintext_backup    false
# auto_rotate_period        0s
# deletion_allowed          false
# keys                      map[1:123456789]  ← Version 1 of the key
# latest_version            1
# min_decryption_version    1
# min_encryption_version    0
# name                      user-data
# type                      aes256-gcm96
```

#### Step 3: Encrypt Data

```bash
# Encrypt some data (must be base64 encoded first)
vault write transit/encrypt/user-data \
    plaintext=$(echo -n "SSN: 123-45-6789" | base64)

# Output:
# Key           Value
# ---           -----
# ciphertext    vault:v1:kL8mF3j2pQ9xR4wS5tU6vW7yZ0aB1cD2eF3gH4iJ5kL6mN7oP8qR9sT0=
# key_version   1
```

**The ciphertext format:** `vault:v1:encrypted_data`
- `vault` — prefix identifying this as Vault-encrypted
- `v1` — which version of the key was used
- The rest — the actual encrypted data

#### Step 4: Decrypt Data

```bash
vault write transit/decrypt/user-data \
    ciphertext="vault:v1:kL8mF3j2pQ9xR4wS5tU6vW7yZ0aB1cD2eF3gH4iJ5kL6mN7oP8qR9sT0="

# Output:
# Key          Value
# ---          -----
# plaintext    U1NOOiAxMjMtNDUtNjc4OQ==

# Decode from base64
echo "U1NOOiAxMjMtNDUtNjc4OQ==" | base64 --decode
# SSN: 123-45-6789
```

#### Step 5: Key Rotation (The Magic)

```bash
# Rotate the encryption key (creates version 2)
vault write -f transit/keys/user-data/rotate

# Now new encryptions use version 2
vault write transit/encrypt/user-data \
    plaintext=$(echo -n "New secret data" | base64)

# Output:
# ciphertext    vault:v2:aB3cD4eF5gH6iJ7kL8mN9oP0qR1sT2uV3wX4yZ5=
#                    ^^
#                  Now v2!

# But OLD data encrypted with v1 can STILL be decrypted!
vault write transit/decrypt/user-data \
    ciphertext="vault:v1:kL8mF3j2pQ9xR4wS5tU6vW7yZ0aB1cD2eF3gH4iJ5kL6mN7oP8qR9sT0="
# ✅ Still works! Vault keeps old key versions for decryption
```

#### Step 6: Re-wrap (Upgrade Old Ciphertext to New Key Version)

If you want old data to use the new key version without decrypting it first:

```bash
# Re-wrap: upgrade ciphertext from v1 to v2 without exposing plaintext
vault write transit/rewrap/user-data \
    ciphertext="vault:v1:kL8mF3j2pQ9xR4wS5tU6vW7yZ0aB1cD2eF3gH4iJ5kL6mN7oP8qR9sT0="

# Output:
# ciphertext    vault:v2:xY9zA0bC1dE2fG3hI4jK5lM6nO7pQ8rS9tU0vW1xY2zA3=
#                    ^^
#                  Now v2! (same plaintext, but encrypted with new key)
```

**This is amazing because:**
- Your app reads the v1 ciphertext from the database
- Sends it to Vault for re-wrapping
- Gets back v2 ciphertext
- Writes v2 back to the database
- **The plaintext never left Vault!**

### Using Transit in a Real Application (Python)

```python
import hvac
import base64
import json

# Connect to Vault
client = hvac.Client(
    url='https://vault.example.com:8200',
    token='s.my-app-token'
)

class VaultEncryption:
    """Helper class for encrypting/decrypting with Vault Transit."""
    
    def __init__(self, vault_client, key_name='user-data'):
        self.client = vault_client
        self.key_name = key_name
    
    def encrypt(self, plaintext: str) -> str:
        """Encrypt a string using Vault Transit."""
        # Base64 encode the plaintext (required by Vault)
        encoded = base64.b64encode(plaintext.encode()).decode()
        
        result = self.client.secrets.transit.encrypt_data(
            name=self.key_name,
            plaintext=encoded
        )
        return result['data']['ciphertext']
    
    def decrypt(self, ciphertext: str) -> str:
        """Decrypt a Vault Transit ciphertext."""
        result = self.client.secrets.transit.decrypt_data(
            name=self.key_name,
            ciphertext=ciphertext
        )
        # Decode from base64
        return base64.b64decode(result['data']['plaintext']).decode()
    
    def encrypt_dict(self, data: dict) -> str:
        """Encrypt a dictionary as JSON."""
        json_str = json.dumps(data)
        return self.encrypt(json_str)
    
    def decrypt_dict(self, ciphertext: str) -> dict:
        """Decrypt a ciphertext back to a dictionary."""
        json_str = self.decrypt(ciphertext)
        return json.loads(json_str)


# Usage Example
enc = VaultEncryption(client)

# Encrypt sensitive user data
user_data = {
    "ssn": "123-45-6789",
    "credit_card": "4111-1111-1111-1111",
    "date_of_birth": "1990-01-15"
}

# Encrypt it
ciphertext = enc.encrypt_dict(user_data)
print(f"Encrypted: {ciphertext}")
# vault:v1:kL8mF3j2pQ9xR4wS5...

# Store ciphertext in your database (it's safe — it's encrypted!)
# db.execute("UPDATE users SET sensitive_data = %s WHERE id = %s", (ciphertext, user_id))

# Later, decrypt it when needed
decrypted = enc.decrypt_dict(ciphertext)
print(f"Decrypted: {decrypted}")
# {'ssn': '123-45-6789', 'credit_card': '4111-1111-1111-1111', 'date_of_birth': '1990-01-15'}
```

## 📋 Quick Reference: When to Use What

| Scenario | Engine | Why |
|----------|--------|-----|
| Store an API key or password | **KV v2** | Simple key-value storage with versioning |
| App needs database access | **Database** | Dynamic creds, auto-expire, unique per app |
| App needs AWS/Azure/GCP access | **AWS/Azure/GCP** | Temporary cloud credentials |
| Need TLS certificates | **PKI** | Instant cert issuance, auto-renewal |
| Need to encrypt user data (PII, cards) | **Transit** | Encrypt without managing keys |
| Need SSH access to servers | **SSH** | Signed SSH keys or one-time passwords |
| Need to verify data integrity | **Transit (HMAC)** | Hash-based message authentication |
| Need to sign documents/tokens | **Transit (Sign)** | Digital signatures |
| Need random numbers/tokens | **Transit (Random)** | Cryptographically secure randomness |

## ⚠️ Common Gotchas and Best Practices

### 1. Always Set Appropriate TTLs

```bash
# ❌ Too long — defeats the purpose of dynamic secrets
vault write database/roles/my-role \
    default_ttl="8760h"   # 1 year — way too long!

# ✅ Short-lived is better
vault write database/roles/my-role \
    default_ttl="1h" \
    max_ttl="24h"
```

### 2. Handle Lease Renewal in Your App

```python
import threading

def renew_lease_periodically(vault_client, lease_id, interval_seconds=1800):
    """Renew the lease every 30 minutes."""
    def renew():
        while True:
            try:
                vault_client.sys.renew_lease(lease_id)
                print(f"Lease {lease_id} renewed successfully")
            except Exception as e:
                print(f"Failed to renew lease: {e}")
                # Request new credentials
                break
            threading.Event().wait(interval_seconds)
    
    thread = threading.Thread(target=renew, daemon=True)
    thread.start()
```

### 3. Use Vault Agent for Automatic Token and Secret Management

Instead of handling Vault tokens and lease renewal in your app code, use **Vault Agent** — a client-side daemon that handles authentication and secret caching:

```hcl
# vault-agent-config.hcl
auto_auth {
  method "kubernetes" {
    mount_path = "auth/kubernetes"
    config = {
      role = "my-app"
    }
  }

  sink "file" {
    config = {
      path = "/tmp/vault-token"
    }
  }
}

# Template — render secrets to a file
template {
  source      = "/etc/vault-agent/templates/db-creds.ctmpl"
  destination = "/app/config/db-creds.json"
}
```

```gotemplate
{{/* /etc/vault-agent/templates/db-creds.ctmpl */}}
{{ with secret "database/creds/readonly-role" }}
{
  "db_username": "{{ .Data.username }}",
  "db_password": "{{ .Data.password }}"
}
{{ end }}
```

Vault Agent will:
1. Authenticate to Vault using Kubernetes auth
2. Request dynamic DB credentials
3. Render them to `/app/config/db-creds.json`
4. Auto-renew the lease before expiry
5. Re-render the file with new credentials when they rotate

### 4. Always Rotate Root Credentials

```bash
# After configuring a database connection, rotate the root password
vault write -f database/rotate-root/my-postgres-db

# Now even the Vault admin doesn't know the DB root password
# Only Vault can manage it
```

### 5. Set Minimum Encryption Version After Rotation

```bash
# After rotating transit keys, set minimum encryption version
# This forces all new encryptions to use the latest key
vault write transit/keys/user-data \
    min_encryption_version=2

# And eventually, after re-wrapping all old data:
vault write transit/keys/user-data \
    min_decryption_version=2
# Now old key versions are no longer usable
```

## 🧠 Key Takeaways

| Concept | Remember |
|---------|----------|
| **Dynamic secrets** | Generated on-demand, unique per app, auto-expire |
| **Database engine** | Creates temporary DB users — no more shared passwords |
| **Lease management** | Every dynamic secret has a TTL — renew or revoke |
| **Emergency revocation** | `vault lease revoke -prefix` kills all credentials instantly |
| **PKI engine** | Your own CA — issue TLS certs in milliseconds |
| **Intermediate CA** | Always issue from an intermediate, never the root CA |
| **Short-lived certs** | 24h certs are more secure than 1-year certs |
| **Transit engine** | Encrypt/decrypt without your app seeing the key |
| **Key rotation** | Rotate transit keys without re-encrypting all data |
| **Re-wrap** | Upgrade old ciphertext to new key version (plaintext never exposed) |
| **Vault Agent** | Client-side daemon that handles auth and secrets for your app |
