# 21.4 — HashiCorp Vault — Kubernetes Integration (Vault Agent, CSI Driver)

> **One Line Answer:** Vault + Kubernetes integration lets your pods **automatically fetch secrets from Vault** without hardcoding anything — using methods like Vault Agent Sidecar Injector and the Secrets Store CSI Driver.

---

## 🤔 Let's Start With a Story (The Problem)

Imagine you're running a microservices-based e-commerce app on Kubernetes. You've got 20+ services — a payment service, user service, inventory service, notification service… and **each one needs secrets** — database passwords, API keys, encryption keys, etc.

Now, where do you store them?

### The Painful Options Without Vault:

```
Option 1: Kubernetes Secrets (base64 encoded — NOT encrypted!)
    └── Anyone with access to etcd or the namespace can read them
    └── No audit trail — who accessed what?
    └── No automatic rotation

Option 2: Environment Variables
    └── Visible in pod spec
    └── Logged by accident
    └── Visible to anyone who can "kubectl describe pod"

Option 3: Hardcoded in Code or Config Files
    └── 😱 Absolutely terrible. Don't even think about it.
    └── Ends up in Git history forever
```

**The real question is:** How do you give your Kubernetes pods access to secrets stored in a **secure, centralized vault** — without the pod even knowing the nitty-gritty details?

That's exactly what **Vault + Kubernetes integration** solves.

---

## ✅ What Are the Ways to Integrate Vault with Kubernetes?

There are **three main methods** to get secrets from Vault into your Kubernetes pods:

| Method | How It Works | Complexity | Best For |
|--------|-------------|------------|----------|
| **Vault Agent Sidecar Injector** | A sidecar container runs alongside your app, fetches secrets, writes them to a shared volume | Medium | Most common. Great for apps that read secrets from files |
| **Vault CSI Provider** | Uses the Kubernetes Secrets Store CSI Driver to mount Vault secrets as volumes | Medium | Teams already using CSI Driver for other providers |
| **Vault SDK / API (Direct)** | Your app directly calls Vault's API using a client library | High | Apps that need dynamic secrets or fine-grained control |

Let's go through each one in detail. 👇

---

## 🔐 Prerequisites — What You Need Before We Start

Before diving in, you need:

1. **A running Kubernetes cluster** (Minikube, Kind, EKS, AKS, GKE — anything works)
2. **Vault installed and running** (can be inside or outside the cluster)
3. **Helm** installed (we'll use it for deploying Vault)
4. **kubectl** configured and connected to your cluster

### Step 1: Install Vault on Kubernetes Using Helm

The easiest way to run Vault on Kubernetes is via the official Helm chart:

```bash
# Add the HashiCorp Helm repo
helm repo add hashicorp https://helm.releases.hashicorp.com
helm repo update

# Install Vault in dev mode (for learning — NOT for production!)
helm install vault hashicorp/vault \
  --set "server.dev.enabled=true" \
  --set "injector.enabled=true" \
  --namespace vault \
  --create-namespace
```

**What this does:**
- `server.dev.enabled=true` → Runs Vault in **dev mode** (auto-unsealed, in-memory storage — great for learning)
- `injector.enabled=true` → Installs the **Vault Agent Sidecar Injector** (we'll use this!)
- Creates a `vault` namespace for all Vault components

### Verify the installation:

```bash
kubectl get pods -n vault
```

**Expected output:**

```
NAME                                    READY   STATUS    RESTARTS   AGE
vault-0                                 1/1     Running   0          60s
vault-agent-injector-5945fb98b5-abc12   1/1     Running   0          60s
```

You should see two pods:
- `vault-0` → The Vault server itself
- `vault-agent-injector-*` → The sidecar injector controller (it watches for pods that need secrets)

## 🧩 Method 1: Vault Agent Sidecar Injector (Most Popular)

This is the **most commonly used** method. Here's how it works:

### How It Works — The Big Picture

```
┌─────────────────────────────────────────────────────────────────┐
│                    KUBERNETES CLUSTER                            │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │                YOUR POD                                  │    │
│  │                                                         │    │
│  │  ┌──────────────────┐    ┌───────────────────────────┐  │    │
│  │  │  Your App         │    │  Vault Agent (Sidecar)    │  │    │
│  │  │  Container        │    │  Container                │  │    │
│  │  │                  │    │                           │  │    │
│  │  │  Reads secrets   │    │  1. Authenticates to      │  │    │
│  │  │  from            │◄───│     Vault using K8s SA    │  │    │
│  │  │  /vault/secrets/ │    │  2. Fetches secrets       │  │    │
│  │  │                  │    │  3. Writes to shared      │  │    │
│  │  │                  │    │     volume                │  │    │
│  │  │                  │    │  4. Keeps them updated    │  │    │
│  │  └──────────────────┘    └───────────┬───────────────┘  │    │
│  │         ▲                            │                  │    │
│  │         │      Shared Volume         │                  │    │
│  │         └────── /vault/secrets/ ◄────┘                  │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                          │                      │
│                                          │ Authenticates &      │
│                                          │ Fetches Secrets      │
│                                          ▼                      │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │                  VAULT SERVER                              │  │
│  │  Stores all secrets securely (encrypted, access-controlled)│  │
│  └───────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

**In simple terms:**
1. You add special **annotations** to your pod
2. The Vault Injector sees these annotations and **automatically adds a sidecar container** to your pod
3. The sidecar **authenticates to Vault** using the pod's Kubernetes Service Account
4. It **fetches the secrets** and writes them to a **shared volume** (`/vault/secrets/`)
5. Your app simply **reads the secrets from files** — that's it!

### Step-by-Step Setup

#### Step 1: Enable Kubernetes Auth Method in Vault

Vault needs to know how to verify that a Kubernetes pod is who it says it is. We do this by enabling the **Kubernetes auth method**.

```bash
# Exec into the Vault pod
kubectl exec -it vault-0 -n vault -- /bin/sh

# Inside the Vault pod, enable K8s auth
vault auth enable kubernetes

# Configure it to talk to the Kubernetes API
vault write auth/kubernetes/config \
    kubernetes_host="https://$KUBERNETES_PORT_443_TCP_ADDR:443"
```

**What's happening here?**
- `vault auth enable kubernetes` → Tells Vault: "Hey, Kubernetes pods can now authenticate with you"
- `vault write auth/kubernetes/config` → Tells Vault where the Kubernetes API server is, so it can verify pod identities

#### Step 2: Create a Secret in Vault

Let's store a database password that our app will need:

```bash
# Still inside the Vault pod

# Enable a KV (Key-Value) secrets engine at path "secret"
vault secrets enable -path=secret kv-v2

# Store a secret
vault kv put secret/myapp/config \
    db_username="admin" \
    db_password="SuperSecret123!" \
    api_key="ak_live_abc123xyz789"
```

**What we just did:**
- Created a KV v2 secrets engine (versioned key-value store)
- Stored 3 secrets at the path `secret/myapp/config`

#### Step 3: Create a Vault Policy

A **policy** defines what secrets a particular identity (our app) can access:

```bash
# Create a policy that allows reading our app's secrets
vault policy write myapp-policy - <<EOF
path "secret/data/myapp/config" {
  capabilities = ["read"]
}
EOF
```

**What this does:**
- Creates a policy called `myapp-policy`
- This policy only allows **reading** the secrets at `secret/data/myapp/config`
- Note: KV v2 adds `data/` to the path automatically — so `secret/myapp/config` becomes `secret/data/myapp/config` in the policy

#### Step 4: Create a Kubernetes Auth Role

This ties it all together — it maps a Kubernetes Service Account to a Vault policy:

```bash
vault write auth/kubernetes/role/myapp-role \
    bound_service_account_names=myapp-sa \
    bound_service_account_namespaces=default \
    policies=myapp-policy \
    ttl=24h
```

**What this does:**
- Creates a role called `myapp-role`
- Says: "If a pod is running with the service account `myapp-sa` in the `default` namespace, give it the `myapp-policy` permissions"
- Tokens expire after 24 hours (`ttl=24h`)

Now exit the Vault pod:

```bash
exit
```

#### Step 5: Create the Kubernetes Service Account

```bash
kubectl create serviceaccount myapp-sa -n default
```

#### Step 6: Deploy Your App with Vault Annotations

Here's the magic part — you just add **annotations** to your pod spec, and the Vault Injector does the rest!

```yaml
# myapp-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
  namespace: default
spec:
  replicas: 1
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
      annotations:
        # ──── Vault Agent Injector Annotations ────
        vault.hashicorp.com/agent-inject: "true"
        vault.hashicorp.com/role: "myapp-role"
        vault.hashicorp.com/agent-inject-secret-config.txt: "secret/data/myapp/config"
        vault.hashicorp.com/agent-inject-template-config.txt: |
          {{- with secret "secret/data/myapp/config" -}}
          DB_USERNAME={{ .Data.data.db_username }}
          DB_PASSWORD={{ .Data.data.db_password }}
          API_KEY={{ .Data.data.api_key }}
          {{- end -}}
    spec:
      serviceAccountName: myapp-sa
      containers:
        - name: myapp
          image: busybox
          command: ["sh", "-c", "while true; do cat /vault/secrets/config.txt; sleep 5; done"]
```

**Let's break down every annotation:**

| Annotation | What It Does |
|-----------|-------------|
| `vault.hashicorp.com/agent-inject: "true"` | Tells the injector: "Hey, this pod needs Vault secrets!" |
| `vault.hashicorp.com/role: "myapp-role"` | Which Vault role to use for authentication |
| `vault.hashicorp.com/agent-inject-secret-config.txt` | The Vault path to fetch secrets from. The part after `agent-inject-secret-` becomes the **filename** (`config.txt`) |
| `vault.hashicorp.com/agent-inject-template-config.txt` | A Go template that controls the **format** of the secret file |

**The template explained:**

```
{{- with secret "secret/data/myapp/config" -}}   ← Fetch the secret
DB_USERNAME={{ .Data.data.db_username }}            ← Access the "db_username" field
DB_PASSWORD={{ .Data.data.db_password }}            ← Access the "db_password" field
API_KEY={{ .Data.data.api_key }}                    ← Access the "api_key" field
{{- end -}}                                        ← End the block
```

#### Step 7: Deploy and Verify

```bash
# Deploy the app
kubectl apply -f myapp-deployment.yaml

# Wait for the pod to be ready
kubectl get pods -n default -w

# Check the secrets file inside the pod
kubectl exec -it $(kubectl get pod -l app=myapp -o jsonpath='{.items[0].metadata.name}') \
  -c myapp -- cat /vault/secrets/config.txt
```

**Expected output:**

```
DB_USERNAME=admin
DB_PASSWORD=SuperSecret123!
API_KEY=ak_live_abc123xyz789
```

🎉 **Your app now reads secrets from Vault without any hardcoding!**

### What Happens Behind the Scenes?

When you deploy that YAML, this is the sequence of events:

```
1. You run: kubectl apply -f myapp-deployment.yaml
         │
         ▼
2. Kubernetes API Server receives the pod spec
         │
         ▼
3. Vault Mutating Webhook intercepts the request
   (It checks: "Does this pod have vault.hashicorp.com/agent-inject: true?")
         │
         ▼
4. YES! → Webhook MUTATES the pod spec:
   - Adds an init container (vault-agent-init) → fetches secrets BEFORE your app starts
   - Adds a sidecar container (vault-agent) → keeps secrets UPDATED while your app runs
   - Adds a shared volume (vault-secrets) → where secrets are written
         │
         ▼
5. Init container runs first:
   - Authenticates to Vault using the pod's Service Account token
   - Fetches secrets
   - Writes them to /vault/secrets/config.txt
         │
         ▼
6. Your app container starts:
   - Secrets are already there in /vault/secrets/config.txt
   - Your app just reads the file — done!
         │
         ▼
7. Sidecar container keeps running:
   - Watches Vault for changes
   - If secrets are rotated in Vault, the sidecar automatically updates the file
```

## 🔌 Method 2: Vault CSI Provider (Secrets Store CSI Driver)

The **Secrets Store CSI Driver** is a Kubernetes-native way to mount secrets from external providers (Vault, AWS, Azure, GCP) as volumes.

### How It Works

```
┌────────────────────────────────────────────────────────────────┐
│                    KUBERNETES CLUSTER                           │
│                                                                │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │                YOUR POD                                   │  │
│  │                                                          │  │
│  │  ┌──────────────────┐                                    │  │
│  │  │  Your App         │                                    │  │
│  │  │  Container        │                                    │  │
│  │  │                  │                                    │  │
│  │  │  Reads secrets   │                                    │  │
│  │  │  from mounted    │                                    │  │
│  │  │  volume          │                                    │  │
│  │  │  /mnt/secrets/   │                                    │  │
│  │  └──────────────────┘                                    │  │
│  └──────────┬───────────────────────────────────────────────┘  │
│             │                                                  │
│             │ CSI Volume Mount                                 │
│             ▼                                                  │
│  ┌──────────────────────────────┐                              │
│  │ Secrets Store CSI Driver     │                              │
│  │ (DaemonSet on every node)    │──────────────────────┐       │
│  └──────────────────────────────┘                      │       │
│                                                        ▼       │
│                                              ┌─────────────┐   │
│                                              │ Vault CSI    │   │
│                                              │ Provider     │   │
│                                              └──────┬──────┘   │
│                                                     │          │
│                                                     ▼          │
│                                              ┌─────────────┐   │
│                                              │ VAULT SERVER │   │
│                                              └─────────────┘   │
└────────────────────────────────────────────────────────────────┘
```

**Key difference from Agent Injector:**
- **No sidecar container** — the CSI driver runs as a DaemonSet on each node
- Secrets are mounted as **volumes** using the standard Kubernetes CSI mechanism
- Can optionally **sync secrets to Kubernetes Secrets** (so you can use them as env vars too)

### Step-by-Step Setup

#### Step 1: Install the Secrets Store CSI Driver

```bash
# Install the Secrets Store CSI Driver
helm repo add secrets-store-csi-driver https://kubernetes-sigs.github.io/secrets-store-csi-driver/charts
helm repo update

helm install csi-secrets-store secrets-store-csi-driver/secrets-store-csi-driver \
  --namespace kube-system \
  --set syncSecret.enabled=true \
  --set enableSecretRotation=true
```

**What this installs:**
- `syncSecret.enabled=true` → Allows syncing Vault secrets to Kubernetes Secrets (optional but handy)
- `enableSecretRotation=true` → Automatically rotates secrets when they change in Vault

#### Step 2: Install the Vault CSI Provider

```bash
helm install vault hashicorp/vault \
  --set "server.enabled=false" \
  --set "injector.enabled=false" \
  --set "csi.enabled=true" \
  --namespace vault \
  --create-namespace
```

**What this does:**
- Disables the Vault server and injector (assuming Vault is already running elsewhere)
- Enables only the **CSI provider**

#### Step 3: Create a SecretProviderClass

This is a Kubernetes custom resource that tells the CSI driver **where to fetch secrets from**:

```yaml
# secret-provider-class.yaml
apiVersion: secrets-store.csi.x-k8s.io/v1
kind: SecretProviderClass
metadata:
  name: vault-secrets
  namespace: default
spec:
  provider: vault
  parameters:
    vaultAddress: "http://vault.vault.svc.cluster.local:8200"
    roleName: "myapp-role"
    objects: |
      - objectName: "db_username"
        secretPath: "secret/data/myapp/config"
        secretKey: "db_username"
      - objectName: "db_password"
        secretPath: "secret/data/myapp/config"
        secretKey: "db_password"
      - objectName: "api_key"
        secretPath: "secret/data/myapp/config"
        secretKey: "api_key"
  # Optional: Sync to a Kubernetes Secret
  secretObjects:
    - secretName: myapp-k8s-secret
      type: Opaque
      data:
        - objectName: db_username
          key: DB_USERNAME
        - objectName: db_password
          key: DB_PASSWORD
        - objectName: api_key
          key: API_KEY
```

**What each part does:**

| Field | Purpose |
|-------|---------|
| `provider: vault` | Use Vault as the secrets backend |
| `vaultAddress` | Where to reach Vault (K8s service DNS) |
| `roleName` | The Vault Kubernetes auth role to use |
| `objects` | List of individual secrets to fetch from Vault |
| `secretObjects` | (Optional) Sync these secrets to a regular K8s Secret |

#### Step 4: Deploy Your App with CSI Volume

```yaml
# myapp-csi-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-csi
  namespace: default
spec:
  replicas: 1
  selector:
    matchLabels:
      app: myapp-csi
  template:
    metadata:
      labels:
        app: myapp-csi
    spec:
      serviceAccountName: myapp-sa
      containers:
        - name: myapp
          image: busybox
          command: ["sh", "-c", "while true; do echo '--- Secrets ---'; cat /mnt/secrets-store/db_username; echo; cat /mnt/secrets-store/db_password; echo; sleep 5; done"]
          volumeMounts:
            - name: secrets-store
              mountPath: "/mnt/secrets-store"
              readOnly: true
          # Optional: Use synced K8s secret as env vars
          env:
            - name: DB_USERNAME
              valueFrom:
                secretKeyRef:
                  name: myapp-k8s-secret
                  key: DB_USERNAME
            - name: DB_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: myapp-k8s-secret
                  key: DB_PASSWORD
      volumes:
        - name: secrets-store
          csi:
            driver: secrets-store.csi.k8s.io
            readOnly: true
            volumeAttributes:
              secretProviderClass: "vault-secrets"
```

**What's happening:**
- The `volumes` section uses the CSI driver to create a volume from the `vault-secrets` SecretProviderClass
- Secrets are mounted at `/mnt/secrets-store/` — each secret becomes a file
- Optionally, the synced K8s Secret is used for environment variables

#### Step 5: Deploy and Verify

```bash
kubectl apply -f secret-provider-class.yaml
kubectl apply -f myapp-csi-deployment.yaml

# Check secrets are mounted
kubectl exec -it $(kubectl get pod -l app=myapp-csi -o jsonpath='{.items[0].metadata.name}') \
  -- ls /mnt/secrets-store/

# Output:
# api_key  db_password  db_username

# Read a secret
kubectl exec -it $(kubectl get pod -l app=myapp-csi -o jsonpath='{.items[0].metadata.name}') \
  -- cat /mnt/secrets-store/db_password

# Output:
# SuperSecret123!
```

## ⚖️ Agent Injector vs CSI Driver — Which One Should You Pick?

| Feature | Vault Agent Injector | CSI Driver |
|---------|---------------------|------------|
| **Sidecar needed?** | Yes (adds a container to your pod) | No (runs as a DaemonSet) |
| **Resource overhead** | Higher (each pod gets a sidecar) | Lower (shared DaemonSet) |
| **Secret format** | Templated files (you control the format) | One file per secret |
| **Auto-rotation** | ✅ Built-in (sidecar watches for changes) | ✅ With `enableSecretRotation=true` |
| **K8s Secret sync** | ❌ Not directly | ✅ Can sync to K8s Secrets |
| **Env var support** | Via files only | Via synced K8s Secrets |
| **Multi-provider** | Vault only | Vault, AWS, Azure, GCP |
| **Complexity** | Simple annotations | More YAML (SecretProviderClass) |
| **Best for** | Most teams, simple setups | Multi-cloud, no-sidecar preference |

### 🏆 Recommendation:

- **Use Agent Injector** if you're a Vault-only shop and want the simplest setup
- **Use CSI Driver** if you use multiple secret providers or want to avoid sidecars

## 🔧 Method 3: Direct API Access (Vault SDK)

For applications that need **dynamic secrets** or **fine-grained control**, you can call Vault's API directly from your code.

### Python Example

```python
import hvac
import os

# Create a Vault client
client = hvac.Client(
    url='http://vault.vault.svc.cluster.local:8200'
)

# Authenticate using Kubernetes Service Account token
# This token is automatically mounted at this path in every K8s pod
with open('/var/run/secrets/kubernetes.io/serviceaccount/token', 'r') as f:
    jwt_token = f.read()

# Login to Vault using K8s auth
client.auth.kubernetes.login(
    role='myapp-role',
    jwt=jwt_token
)

# Now read secrets!
secret = client.secrets.kv.v2.read_secret_version(
    path='myapp/config',
    mount_point='secret'
)

# Access individual values
db_username = secret['data']['data']['db_username']
db_password = secret['data']['data']['db_password']
api_key = secret['data']['data']['api_key']

print(f"DB Username: {db_username}")
print(f"DB Password: {db_password}")
print(f"API Key: {api_key}")
```

**What this code does:**

| Line | Explanation |
|------|------------|
| `import hvac` | Import the official HashiCorp Vault Python client |
| `hvac.Client(url=...)` | Create a Vault client pointing to the Vault server |
| `open('/var/run/secrets/...')` | Read the K8s Service Account JWT token (auto-mounted in every pod) |
| `client.auth.kubernetes.login(...)` | Authenticate to Vault using the K8s auth method |
| `client.secrets.kv.v2.read_secret_version(...)` | Fetch the secret from Vault's KV v2 engine |
| `secret['data']['data']['db_username']` | Access the actual secret value (KV v2 nests data under `data.data`) |

### Go Example

```go
package main

import (
    "fmt"
    "os"

    vault "github.com/hashicorp/vault/api"
    auth "github.com/hashicorp/vault/api/auth/kubernetes"
)

func main() {
    // Create Vault client
    config := vault.DefaultConfig()
    config.Address = "http://vault.vault.svc.cluster.local:8200"

    client, err := vault.NewClient(config)
    if err != nil {
        panic(err)
    }

    // Authenticate using Kubernetes auth
    k8sAuth, err := auth.NewKubernetesAuth(
        "myapp-role",
        auth.WithServiceAccountTokenPath(
            "/var/run/secrets/kubernetes.io/serviceaccount/token",
        ),
    )
    if err != nil {
        panic(err)
    }

    authInfo, err := client.Auth().Login(context.TODO(), k8sAuth)
    if err != nil {
        panic(err)
    }

    // Read secret
    secret, err := client.KVv2("secret").Get(context.TODO(), "myapp/config")
    if err != nil {
        panic(err)
    }

    // Access values
    dbUsername := secret.Data["db_username"].(string)
    dbPassword := secret.Data["db_password"].(string)

    fmt.Printf("DB Username: %s\n", dbUsername)
    fmt.Printf("DB Password: %s\n", dbPassword)
}
```

## 🔄 Secret Rotation — Keeping Secrets Fresh

One of the **biggest advantages** of Vault integration is **automatic secret rotation**.

### With Agent Injector:

The sidecar runs continuously and checks Vault for updates. If you rotate a secret in Vault:

```bash
# Rotate the database password in Vault
kubectl exec -it vault-0 -n vault -- vault kv put secret/myapp/config \
    db_username="admin" \
    db_password="NewRotatedPassword456!" \
    api_key="ak_live_abc123xyz789"
```

The sidecar will detect the change and **automatically update** `/vault/secrets/config.txt` in your pod — **no restart needed!**

### With CSI Driver:

Enable rotation in the Helm values:

```bash
helm upgrade csi-secrets-store secrets-store-csi-driver/secrets-store-csi-driver \
  --namespace kube-system \
  --set enableSecretRotation=true \
  --set rotationPollInterval=30s
```

This checks Vault every 30 seconds and updates the mounted files if secrets changed.

## 🛡️ Production Best Practices

Here are the things you should **always** do when running Vault with Kubernetes in production:

### 1. Never Run Vault in Dev Mode in Production

```bash
# ❌ Dev mode (in-memory, auto-unsealed, root token is "root")
--set "server.dev.enabled=true"

# ✅ Production mode (encrypted storage, manual unseal, proper auth)
--set "server.ha.enabled=true"
--set "server.ha.replicas=3"
```

### 2. Use HA (High Availability) Mode

```yaml
# values-production.yaml
server:
  ha:
    enabled: true
    replicas: 3
    raft:
      enabled: true    # Use integrated Raft storage
      setNodeId: true
  
  # Use proper storage
  dataStorage:
    enabled: true
    size: 10Gi
    storageClass: gp3
```

### 3. Limit Policy Permissions (Least Privilege)

```hcl
# ❌ Too broad — allows access to ALL secrets
path "secret/*" {
  capabilities = ["read", "list"]
}

# ✅ Specific — only allows access to this app's secrets
path "secret/data/myapp/config" {
  capabilities = ["read"]
}
```

### 4. Use Namespaces for Multi-Tenant Environments

```bash
# Create separate Vault namespaces for different teams
vault namespace create team-payments
vault namespace create team-users
vault namespace create team-inventory
```

### 5. Enable Audit Logging

```bash
# Log every secret access to a file
vault audit enable file file_path=/vault/logs/audit.log

# Or send audit logs to stdout (great for K8s + Fluentd/Loki)
vault audit enable file file_path=stdout
```

### 6. Auto-Unseal with Cloud KMS

Instead of manually unsealing Vault, use a cloud KMS:

```yaml
# AWS KMS Auto-Unseal
server:
  extraEnvironmentVars:
    VAULT_SEAL_TYPE: awskms
    VAULT_AWSKMS_SEAL_KEY_ID: "your-kms-key-id"
```

## 🧪 Troubleshooting Common Issues

| Problem | Cause | Fix |
|---------|-------|-----|
| Pod stuck in `Init:0/1` | Vault Agent can't authenticate | Check Service Account name matches the Vault role |
| `permission denied` error | Policy doesn't match the secret path | Remember KV v2 adds `/data/` to paths |
| Secrets file is empty | Template syntax error | Check Go template syntax in annotations |
| Injector not adding sidecar | Webhook not running or annotations missing | `kubectl get pods -n vault` — is the injector running? |
| CSI volume mount fails | SecretProviderClass misconfigured | Check `vaultAddress` and `roleName` |
| `service account not allowed` | SA not bound to the Vault role | Verify `bound_service_account_names` and `bound_service_account_namespaces` |

### Useful Debug Commands:

```bash
# Check Vault Agent logs in the sidecar
kubectl logs <pod-name> -c vault-agent

# Check init container logs
kubectl logs <pod-name> -c vault-agent-init

# Verify Vault K8s auth config
kubectl exec -it vault-0 -n vault -- vault read auth/kubernetes/config

# List all roles
kubectl exec -it vault-0 -n vault -- vault list auth/kubernetes/role

# Test authentication manually
kubectl exec -it vault-0 -n vault -- vault write auth/kubernetes/login \
  role=myapp-role \
  jwt=$(cat /var/run/secrets/kubernetes.io/serviceaccount/token)
```

## 📝 Quick Reference — Key Annotations for Agent Injector

| Annotation | Purpose | Example |
|-----------|---------|---------|
| `vault.hashicorp.com/agent-inject` | Enable injection | `"true"` |
| `vault.hashicorp.com/role` | Vault auth role | `"myapp-role"` |

| `vault.hashicorp.com/agent-pre-populate-only` | Only inject once (no sidecar) | `"true"` |
| `vault.hashicorp.com/agent-inject-token` | Also inject the Vault token | `"true"` |
| `vault.hashicorp.com/agent-revoke-on-shutdown` | Revoke token on pod shutdown | `"true"` |
| `vault.hashicorp.com/agent-cache-enable` | Enable response caching | `"true"` |
| `vault.hashicorp.com/tls-skip-verify` | Skip TLS verification (dev only!) | `"true"` |

## 🎯 Summary

| Concept | Key Takeaway |
|---------|-------------|
| **Why integrate Vault + K8s?** | K8s Secrets are base64-encoded (not encrypted), no rotation, no audit trail |
| **Vault Agent Injector** | Most popular. Adds a sidecar to your pod that fetches and refreshes secrets |
| **CSI Driver** | No sidecar. Uses K8s-native CSI to mount secrets as volumes. Multi-provider support |
| **Direct API** | Most flexible. Use Vault SDK in your app code for dynamic secrets |
| **K8s Auth Method** | Vault verifies pod identity using K8s Service Account tokens |
| **Policies** | Always use least-privilege. Only grant access to the exact secret paths needed |
| **Production** | HA mode, auto-unseal, audit logging, never dev mode |
