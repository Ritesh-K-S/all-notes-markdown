## 🤔 Let's Start With a Story (The Problem)

You're a DevOps engineer at a startup. Your team just deployed a Node.js app on AWS (running on EC2 or ECS or Lambda). The app connects to an RDS database, calls a Stripe API for payments, and sends emails via SendGrid.

That means your app needs:
- Database hostname, username, and password
- Stripe API key
- SendGrid API key
- JWT signing secret

Now, where do you put all these? 🤔

### What People Actually Do (The Wrong Way):

```
Attempt 1: Hardcode in source code
    └── "mongodb://admin:P@ssw0rd@db.example.com/mydb"
    └── 😱 Now it's in Git history FOREVER
    └── Anyone with repo access can see it

Attempt 2: .env file
    └── Better, but still a file that can be leaked
    └── What if someone commits it by accident?
    └── How do you share it across 10 servers?

Attempt 3: Environment variables in EC2 User Data
    └── Visible in AWS console
    └── No encryption, no versioning, no rotation
    └── Terrible for compliance (SOC2, HIPAA)

Attempt 4: AWS Secrets Manager / SSM Parameter Store ✅
    └── Encrypted at rest with KMS
    └── Access controlled via IAM
    └── Automatic rotation
    └── Audit trail via CloudTrail
    └── THIS is the right way!
```

**AWS gives you two services for this. Let's understand both.**

## 🏗️ The Two Services — Overview

Before diving deep, let's understand what each service is and when to use which:

| Feature | AWS Secrets Manager | SSM Parameter Store |
|---------|-------------------|-------------------|
| **Primary purpose** | Store and rotate **secrets** (passwords, API keys, tokens) | Store **configuration data** and secrets |
| **Encryption** | Always encrypted (KMS) | Optional — SecureString uses KMS |
| **Auto-rotation** | ✅ Built-in (Lambda-based) | ❌ Not built-in (you'd build it yourself) |
| **Cost** | 💰 $0.40 per secret/month + $0.05 per 10K API calls | 🆓 Free for Standard tier (up to 10K params) |
| **Max size** | 64 KB per secret | 4 KB (Standard) / 8 KB (Advanced) |
| **Versioning** | ✅ Automatic (staging labels) | ✅ Built-in (version history) |
| **Cross-account sharing** | ✅ Via resource policies | ✅ Via IAM and sharing |
| **Best for** | Database creds, API keys, certificates | Feature flags, config values, non-sensitive + sensitive params |

### The Simple Rule:

```
🔐 Sensitive secret that needs rotation? → Secrets Manager
⚙️ Configuration value or simple secret? → SSM Parameter Store
💰 Budget is tight?                       → SSM Parameter Store (it's free!)
```

### What Is It?

AWS Secrets Manager is a fully managed service that helps you **store, retrieve, manage, and rotate** secrets like database credentials, API keys, and other sensitive data.

### Creating Secrets — Multiple Ways

#### Method 1: AWS Console (UI)

1. Go to AWS Console → Secrets Manager
2. Click "Store a new secret"
3. Choose secret type (e.g., "Credentials for RDS database")
4. Enter key-value pairs
5. Name the secret (e.g., `prod/myapp/db-creds`)
6. Optionally configure rotation
7. Click "Store"

#### Method 2: AWS CLI

```bash
# Create a simple key-value secret
aws secretsmanager create-secret \
    --name "prod/myapp/db-creds" \
    --description "Production database credentials" \
    --secret-string '{"username":"admin","password":"SuperSecret123!","host":"mydb.cluster-xyz.us-east-1.rds.amazonaws.com","port":"5432","dbname":"myapp"}'
```

**What each flag does:**

| Flag | Purpose |
|------|---------|
| `--name` | The name/path of the secret. Use `/` for organizing (like folders) |
| `--description` | Human-readable description |
| `--secret-string` | The actual secret value. Can be plain text or JSON |

```bash
# Create a secret from a file
aws secretsmanager create-secret \
    --name "prod/myapp/tls-cert" \
    --secret-binary fileb://certificate.pem
```

#### Method 3: Terraform (Infrastructure as Code)

```hcl
# Create a secret in Secrets Manager using Terraform
resource "aws_secretsmanager_secret" "db_creds" {
  name        = "prod/myapp/db-creds"
  description = "Production database credentials"

  # Optional: Use a custom KMS key for encryption
  kms_key_id  = aws_kms_key.secrets_key.arn

  # Optional: Recovery window (days before permanent deletion)
  recovery_window_in_days = 7

  tags = {
    Environment = "production"
    Team        = "backend"
  }
}

# Set the actual secret value
resource "aws_secretsmanager_secret_version" "db_creds_value" {
  secret_id     = aws_secretsmanager_secret.db_creds.id
  secret_string = jsonencode({
    username = "admin"
    password = "SuperSecret123!"
    host     = "mydb.cluster-xyz.us-east-1.rds.amazonaws.com"
    port     = 5432
    dbname   = "myapp"
  })
}
```

**What each part does:**
- `aws_secretsmanager_secret` → Creates the secret **container** (name, description, encryption key)
- `aws_secretsmanager_secret_version` → Sets the actual **value** inside the secret
- `recovery_window_in_days` → If you delete the secret, AWS keeps it for 7 days before permanent deletion (safety net!)
- `kms_key_id` → Use your own KMS key instead of the default `aws/secretsmanager` key

> ⚠️ **Important Terraform Warning:** The secret value will be stored in your Terraform state file. Make sure your state file is encrypted (use S3 backend with encryption enabled).

### Retrieving Secrets — In Your Application

#### Python (boto3)

```python
import boto3
import json

def get_secret(secret_name, region_name="us-east-1"):
    """Retrieve a secret from AWS Secrets Manager."""
    
    # Create a Secrets Manager client
    client = boto3.client(
        service_name='secretsmanager',
        region_name=region_name
    )
    
    # Fetch the secret
    response = client.get_secret_value(SecretId=secret_name)
    
    # Parse the secret (it's stored as a JSON string)
    secret = json.loads(response['SecretString'])
    
    return secret

# Usage
creds = get_secret("prod/myapp/db-creds")
print(f"Host: {creds['host']}")
print(f"Username: {creds['username']}")
print(f"Password: {creds['password']}")
print(f"Database: {creds['dbname']}")
```

**Line-by-line explanation:**

| Line | What It Does |
|------|-------------|
| `boto3.client('secretsmanager')` | Creates an AWS Secrets Manager client using your IAM credentials |
| `get_secret_value(SecretId=...)` | Calls the API to fetch the secret. The IAM role/user must have permission |
| `response['SecretString']` | The raw secret value (a JSON string in our case) |
| `json.loads(...)` | Parse the JSON string into a Python dictionary |

#### Node.js

```javascript
const { SecretsManagerClient, GetSecretValueCommand } = require("@aws-sdk/client-secrets-manager");

async function getSecret(secretName) {
    const client = new SecretsManagerClient({ region: "us-east-1" });
    
    const command = new GetSecretValueCommand({ SecretId: secretName });
    const response = await client.send(command);
    
    // Parse the JSON secret
    return JSON.parse(response.SecretString);
}

// Usage
async function main() {
    const creds = await getSecret("prod/myapp/db-creds");
    console.log(`Connecting to ${creds.host} as ${creds.username}`);
    
    // Use creds to connect to database
    // const pool = new Pool({
    //     host: creds.host,
    //     user: creds.username,
    //     password: creds.password,
    //     database: creds.dbname,
    //     port: creds.port
    // });
}

main();
```

#### Go

```go
package main

import (
    "context"
    "encoding/json"
    "fmt"
    "log"

    "github.com/aws/aws-sdk-go-v2/config"
    "github.com/aws/aws-sdk-go-v2/service/secretsmanager"
)

type DBCreds struct {
    Username string `json:"username"`
    Password string `json:"password"`
    Host     string `json:"host"`
    Port     int    `json:"port"`
    DBName   string `json:"dbname"`
}

func getSecret(secretName string) (*DBCreds, error) {
    cfg, err := config.LoadDefaultConfig(context.TODO(), config.WithRegion("us-east-1"))
    if err != nil {
        return nil, err
    }

    client := secretsmanager.NewFromConfig(cfg)
    
    result, err := client.GetSecretValue(context.TODO(), &secretsmanager.GetSecretValueInput{
        SecretId: &secretName,
    })
    if err != nil {
        return nil, err
    }

    var creds DBCreds
    err = json.Unmarshal([]byte(*result.SecretString), &creds)
    if err != nil {
        return nil, err
    }

    return &creds, nil
}

func main() {
    creds, err := getSecret("prod/myapp/db-creds")
    if err != nil {
        log.Fatal(err)
    }
    fmt.Printf("Connecting to %s as %s\n", creds.Host, creds.Username)
}
```

### 🔄 Secret Rotation — The Killer Feature

This is the **#1 reason** to choose Secrets Manager over SSM Parameter Store. Secrets Manager can **automatically rotate secrets** on a schedule.

#### How Rotation Works:

```
Step 1: Rotation Lambda is triggered (on schedule or manually)
              │
              ▼
Step 2: Lambda creates a NEW password
              │
              ▼
Step 3: Lambda updates the database with the new password
              │
              ▼
Step 4: Lambda stores the new password in Secrets Manager
              │
              ▼
Step 5: Lambda marks the new version as "AWSCURRENT"
              │
              ▼
Step 6: Old password is labeled "AWSPREVIOUS" (kept as fallback)
              │
              ▼
🎉 Your app automatically gets the new password next time it calls GetSecretValue!
```

#### Setting Up Rotation for RDS:

```bash
# Enable rotation for an RDS secret (AWS provides built-in Lambda functions for RDS)
aws secretsmanager rotate-secret \
    --secret-id "prod/myapp/db-creds" \
    --rotation-lambda-arn "arn:aws:lambda:us-east-1:123456789:function:SecretsManagerRDSRotation" \
    --rotation-rules '{"AutomaticallyAfterDays": 30}'
```

**What this does:**
- Every **30 days**, the Lambda function runs
- It generates a new password
- Updates the RDS database password
- Updates the secret in Secrets Manager
- Zero downtime. Zero manual work.

#### Rotation with Terraform:

```hcl
# Enable rotation for the secret
resource "aws_secretsmanager_secret_rotation" "db_creds_rotation" {
  secret_id           = aws_secretsmanager_secret.db_creds.id
  rotation_lambda_arn = aws_lambda_function.rotation_lambda.arn

  rotation_rules {
    automatically_after_days = 30
  }
}
```

#### Rotation Strategies:

| Strategy | How It Works | Downtime Risk |
|----------|-------------|--------------|
| **Single-user rotation** | Updates the password for the same user | Tiny window where the old password is invalid |
| **Alternating-user rotation** | Alternates between two users (e.g., `admin` and `admin_clone`) | ✅ Zero downtime — always one valid user |

For **zero-downtime rotation**, use the alternating-user strategy:

```
Time 0:   App uses "admin" user (password: "OldPass")
Time 1:   Rotation creates "admin_clone" with new password
Time 2:   Secret now points to "admin_clone"
Time 3:   Next rotation updates "admin" with a new password
Time 4:   Secret now points back to "admin"
... and so on, alternating forever
```

### 🔑 IAM Permissions for Secrets Manager

Your EC2 instance, Lambda function, or ECS task needs IAM permissions to access secrets.

#### IAM Policy — Least Privilege:

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": [
                "secretsmanager:GetSecretValue"
            ],
            "Resource": [
                "arn:aws:secretsmanager:us-east-1:123456789012:secret:prod/myapp/*"
            ]
        },
        {
            "Effect": "Allow",
            "Action": [
                "kms:Decrypt"
            ],
            "Resource": [
                "arn:aws:kms:us-east-1:123456789012:key/your-kms-key-id"
            ],
            "Condition": {
                "StringEquals": {
                    "kms:ViaService": "secretsmanager.us-east-1.amazonaws.com"
                }
            }
        }
    ]
}
```

**What each statement does:**

| Statement | Purpose |
|-----------|---------|
| `secretsmanager:GetSecretValue` | Allow reading secrets |
| `Resource: prod/myapp/*` | Only secrets under the `prod/myapp/` path |
| `kms:Decrypt` | Needed to decrypt the secret (encrypted with KMS) |
| `kms:ViaService` condition | Only allow KMS decryption when called via Secrets Manager (extra security) |

> 🛡️ **Best Practice:** Never use `"Resource": "*"`. Always scope to specific secret ARNs or paths.

## ⚙️ Part 2: SSM Parameter Store — Deep Dive

### What Is It?

**AWS Systems Manager Parameter Store** (commonly called SSM Parameter Store) is a free, hierarchical store for configuration data and secrets.

Think of it as a **simple key-value store** that lives in AWS and can optionally encrypt values.

### Parameter Types

| Type | Encryption | Use Case | Max Size |
|------|-----------|----------|----------|
| **String** | ❌ None | Non-sensitive config (feature flags, URLs) | 4 KB / 8 KB |
| **StringList** | ❌ None | Comma-separated values | 4 KB / 8 KB |
| **SecureString** | ✅ KMS encrypted | Passwords, API keys, tokens | 4 KB / 8 KB |

### Parameter Tiers

| Feature | Standard (Free) | Advanced (Paid) |
|---------|----------------|-----------------|
| **Max parameters** | 10,000 | 100,000 |
| **Max value size** | 4 KB | 8 KB |
| **Parameter policies** | ❌ | ✅ (expiration, notifications) |
| **Cost** | 🆓 Free! | $0.05 per advanced param/month |
| **Higher throughput** | 1,000 TPS | 1,000 TPS (with higher limit available) |

---

### Creating Parameters

#### AWS CLI:

```bash
# Create a simple String parameter (NOT encrypted)
aws ssm put-parameter \
    --name "/myapp/prod/app-url" \
    --value "https://api.myapp.com" \
    --type "String" \
    --description "Production API URL"

# Create a StringList parameter
aws ssm put-parameter \
    --name "/myapp/prod/allowed-origins" \
    --value "https://myapp.com,https://www.myapp.com,https://admin.myapp.com" \
    --type "StringList"

# Create a SecureString parameter (ENCRYPTED with KMS)
aws ssm put-parameter \
    --name "/myapp/prod/db-password" \
    --value "SuperSecret123!" \
    --type "SecureString" \
    --description "Production database password"

# Use a custom KMS key for encryption
aws ssm put-parameter \
    --name "/myapp/prod/api-key" \
    --value "sk_live_abc123xyz" \
    --type "SecureString" \
    --key-id "arn:aws:kms:us-east-1:123456789:key/your-key-id"

# Update an existing parameter (use --overwrite)
aws ssm put-parameter \
    --name "/myapp/prod/db-password" \
    --value "NewPassword456!" \
    --type "SecureString" \
    --overwrite
```

**Key things to notice:**
- Use **hierarchical paths** with `/` (like a file system) — e.g., `/myapp/prod/db-password`
- `SecureString` encrypts with KMS (uses the default `aws/ssm` key unless you specify `--key-id`)
- `--overwrite` is needed to update an existing parameter

#### Terraform:

```hcl

  }

# Sensitive parameter (encrypted)

    new_checkout_flow    = false
    max_upload_size_mb   = 50
  })
}
```

### Retrieving Parameters

#### AWS CLI:

```bash
# Get a single parameter
aws ssm get-parameter \
    --name "/myapp/prod/app-url"

# Get a SecureString parameter (decrypted)
aws ssm get-parameter \
    --name "/myapp/prod/db-password" \
    --with-decryption

# Get ONLY the value
aws ssm get-parameter \
    --name "/myapp/prod/db-password" \
    --with-decryption \
    --query 'Parameter.Value' \
    --output text

# Get ALL parameters under a path (like listing a directory!)
aws ssm get-parameters-by-path \
    --path "/myapp/prod" \
    --recursive \
    --with-decryption
```

**Important:** For `SecureString` parameters, you **must** add `--with-decryption` or you'll get the encrypted blob.

#### Python (boto3):

```python
import boto3

ssm = boto3.client('ssm', region_name='us-east-1')

def get_parameter(name, decrypt=True):
    """Get a single parameter from SSM Parameter Store."""
    response = ssm.get_parameter(
        Name=name,
        WithDecryption=decrypt
    )
    return response['Parameter']['Value']

def get_parameters_by_path(path, decrypt=True):
    """Get all parameters under a path (like a directory listing)."""
    parameters = {}
    paginator = ssm.get_paginator('get_parameters_by_path')
    
    for page in paginator.paginate(
        Path=path,
        Recursive=True,
        WithDecryption=decrypt
    ):
        for param in page['Parameters']:
            # Extract just the parameter name (remove the path prefix)
            key = param['Name'].split('/')[-1]
            parameters[key] = param['Value']
    
    return parameters

# Usage — Single parameter
db_password = get_parameter("/myapp/prod/db-password")
print(f"DB Password: {db_password}")

# Usage — All parameters under a path
config = get_parameters_by_path("/myapp/prod")
print(f"All config: {config}")
# Output: {'app-url': 'https://api.myapp.com', 'db-password': 'SuperSecret123!', ...}
```

**Why `get_parameters_by_path` is awesome:**
- You can grab **all config for an app** in one call
- It's like doing `ls /myapp/prod/` — you get everything under that path
- Perfect for loading all config at app startup

#### Node.js:

```javascript
const { SSMClient, GetParameterCommand, GetParametersByPathCommand } = require("@aws-sdk/client-ssm");

const ssm = new SSMClient({ region: "us-east-1" });

async function getParameter(name) {
    const command = new GetParameterCommand({
        Name: name,
        WithDecryption: true
    });
    const response = await ssm.send(command);
    return response.Parameter.Value;
}

async function getAllConfig(path) {
    const command = new GetParametersByPathCommand({
        Path: path,
        Recursive: true,
        WithDecryption: true
    });
    const response = await ssm.send(command);
    
    const config = {};
    for (const param of response.Parameters) {
        const key = param.Name.split('/').pop();
        config[key] = param.Value;
    }
    return config;
}

// Usage
async function main() {
    const dbPassword = await getParameter("/myapp/prod/db-password");
    console.log(`DB Password: ${dbPassword}`);
    
    const allConfig = await getAllConfig("/myapp/prod");
    console.log("All config:", allConfig);
}

main();
```

### 🌳 Organizing Parameters with Hierarchies

One of the best features of Parameter Store is the **hierarchical path structure**:

```
/
├── myapp/
│   ├── prod/
│   │   ├── db-host          → "prod-db.xyz.rds.amazonaws.com"
│   │   ├── db-password      → "ProdSecret123!" (SecureString)
│   │   ├── api-key          → "sk_live_abc123" (SecureString)
│   │   ├── log-level        → "warn"
│   │   └── feature-flags    → '{"dark_mode": true}'
│   ├── staging/
│   │   ├── db-host          → "staging-db.xyz.rds.amazonaws.com"
│   │   ├── db-password      → "StagingSecret!" (SecureString)
│   │   ├── api-key          → "sk_test_xyz789" (SecureString)
│   │   ├── log-level        → "debug"
│   │   └── feature-flags    → '{"dark_mode": true, "new_checkout": true}'
│   └── dev/
│       ├── db-host          → "localhost"
│       ├── db-password      → "devpass" (SecureString)
│       └── log-level        → "debug"
└── another-app/
    ├── prod/
    │   └── ...
    └── staging/
        └── ...
```

**Benefits of this structure:**
- **Environment isolation** — `/myapp/prod/` vs `/myapp/staging/`
- **Bulk retrieval** — `GetParametersByPath("/myapp/prod")` gets ALL prod config
- **IAM scoping** — Allow access to `/myapp/prod/*` only for production workloads
- **Same code, different config** — Your app reads from `/myapp/{ENV}/db-password`

```json
                {
                    "name": "DB_PASSWORD",
                    "valueFrom": "arn:aws:ssm:us-east-1:123456789:parameter/myapp/prod/db-password"
                },
                {
                    "name": "API_KEY",
                    "valueFrom": "arn:aws:secretsmanager:us-east-1:123456789:secret:prod/myapp/api-key-AbCdEf"
                }
            ],
            "environment": [
                {
                    "name": "LOG_LEVEL",
                    "value": "warn"
                }
            ]
        }
    ],
    "executionRoleArn": "arn:aws:iam::123456789:role/ecsTaskExecutionRole"
}
```

**What's happening:**
- `secrets` → Values are fetched at container start from SSM or Secrets Manager
- `environment` → Plain text env vars (for non-sensitive config)
- The **execution role** needs permission to read from SSM/Secrets Manager

### 🔗 Using SSM Parameters in Lambda

```python
import boto3
import json
import os

# Cache the SSM client (reused across Lambda invocations)
ssm = boto3.client('ssm')

# Cache parameters (loaded once, reused across warm invocations)
_cached_config = None

def get_config():
    """Load all config from SSM (with caching for Lambda warm starts)."""
    global _cached_config
    if _cached_config is None:
        env = os.environ.get('ENVIRONMENT', 'dev')
        response = ssm.get_parameters_by_path(
            Path=f'/myapp/{env}',
            Recursive=True,
            WithDecryption=True
        )
        _cached_config = {
            p['Name'].split('/')[-1]: p['Value']
            for p in response['Parameters']
        }
    return _cached_config

def lambda_handler(event, context):
    config = get_config()
    
    db_host = config['db-host']
    db_password = config['db-password']
    
    # Connect to DB and do your thing...
    return {
        'statusCode': 200,
        'body': json.dumps({'message': 'Success!'})
    }
```

**Why we cache:**
- Lambda keeps the execution environment warm between invocations
- Loading config from SSM on every invocation adds latency and costs API calls
- Cache it once, reuse on warm starts — much faster!

## ⚖️ Secrets Manager vs SSM Parameter Store — Decision Guide

Here's a detailed comparison to help you choose:

| Scenario | Use This |
|----------|---------|
| Database credentials that need auto-rotation | **Secrets Manager** ✅ |
| Feature flags and config values | **SSM Parameter Store** ✅ |
| API keys (no rotation needed) | **Either** (SSM is free though!) |
| Budget is tight | **SSM Parameter Store** (free tier) |
| Need to store files/certs > 4KB | **Secrets Manager** (up to 64KB) |
| Cross-account secret sharing | **Secrets Manager** (resource policies) |
| Thousands of config values | **SSM Parameter Store** (designed for this) |
| Compliance requires rotation | **Secrets Manager** ✅ |
| Simple key-value config store | **SSM Parameter Store** ✅ |
| ECS/Lambda secrets injection | **Both work equally well** |

### 💡 The Best Practice: Use BOTH Together!

Most teams use **both** services together:

```
SSM Parameter Store          Secrets Manager
─────────────────────        ─────────────────────
/myapp/prod/app-url          prod/myapp/db-creds (with rotation)
/myapp/prod/log-level        prod/myapp/stripe-key
/myapp/prod/feature-flags    prod/myapp/jwt-secret
/myapp/prod/max-retries      prod/myapp/tls-certificate
/myapp/prod/cache-ttl
```

- **SSM** for non-sensitive and simple config → Free!
- **Secrets Manager** for sensitive secrets that need rotation → Worth the cost

## 🛡️ Security Best Practices

### 1. Use IAM Policies with Least Privilege

```json
{
    "Effect": "Allow",
    "Action": ["ssm:GetParameter"],
    "Resource": "arn:aws:ssm:*:*:parameter/myapp/prod/*"
}
```

Never use `"Resource": "*"` — always scope to specific paths.

### 2. Use Custom KMS Keys (Not Default)

```bash
# Create a dedicated KMS key for your app's secrets
aws kms create-key --description "KMS key for myapp secrets"

# Use it when creating parameters
aws ssm put-parameter \
    --name "/myapp/prod/db-password" \
    --value "secret" \
    --type "SecureString" \
    --key-id "your-custom-kms-key-id"
```

### 3. Enable CloudTrail for Audit Logging

CloudTrail automatically logs all API calls to Secrets Manager and SSM. You can see:
- **Who** accessed a secret
- **When** they accessed it
- **From where** (IP address)

### 4. Tag Everything

```bash
aws ssm put-parameter \
    --name "/myapp/prod/db-password" \
    --value "secret" \
    --type "SecureString" \
    --tags "Key=Environment,Value=production" "Key=Team,Value=backend"
```

Tags help with:
- Cost allocation
- Access control (tag-based IAM policies)
- Organization

### 5. Use Resource Policies for Cross-Account Access

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Principal": {
                "AWS": "arn:aws:iam::987654321098:root"
            },
            "Action": "secretsmanager:GetSecretValue",
            "Resource": "*"
        }
    ]
}
```

## 🧪 Useful CLI Commands — Quick Reference

```bash
# ──── Secrets Manager ────

# List all secrets
aws secretsmanager list-secrets

# Get a secret value
aws secretsmanager get-secret-value --secret-id "prod/myapp/db-creds"

# Update a secret
aws secretsmanager update-secret --secret-id "prod/myapp/db-creds" \
    --secret-string '{"username":"admin","password":"NewPass!"}'

# Delete a secret (with recovery window)
aws secretsmanager delete-secret --secret-id "prod/myapp/db-creds" \
    --recovery-window-in-days 7

# Manually trigger rotation
aws secretsmanager rotate-secret --secret-id "prod/myapp/db-creds"

# ──── SSM Parameter Store ────

# List all parameters
aws ssm describe-parameters

# Get a parameter
aws ssm get-parameter --name "/myapp/prod/db-password" --with-decryption

# Get all parameters under a path
aws ssm get-parameters-by-path --path "/myapp/prod" --recursive --with-decryption

# Delete a parameter
aws ssm delete-parameter --name "/myapp/prod/db-password"

# Get parameter history (all versions)
aws ssm get-parameter-history --name "/myapp/prod/db-password" --with-decryption
```

## 🎯 Summary

| Concept | Key Takeaway |
|---------|-------------|
| **Secrets Manager** | Best for **secrets that need rotation**. Costs $0.40/secret/month. Built-in rotation for RDS, Redshift, DocumentDB |
| **SSM Parameter Store** | Best for **general config and simple secrets**. **Free** for standard tier. Hierarchical paths |
| **SecureString** | SSM parameter type that encrypts with KMS. Always use for passwords/keys |
| **Hierarchy** | Organize with paths: `/app/env/key`. Enables bulk retrieval and IAM scoping |
| **ECS/Lambda integration** | Both services integrate natively — secrets injected as env vars |
| **Use both together** | SSM for config, Secrets Manager for secrets. Best of both worlds |
| **IAM least privilege** | Always scope permissions to specific secret/parameter ARNs |
| **CloudTrail** | Automatically logs all access — essential for compliance |
