## 🤔 Let's Start With a Story (The Problem)

You're a DevOps engineer at a company that runs everything on Azure. You've got:
- A .NET API running on **Azure App Service**
- A microservices platform on **AKS** (Azure Kubernetes Service)
- A CI/CD pipeline in **Azure DevOps**
- Databases on **Azure SQL** and **Cosmos DB**
- Storage accounts, Redis Cache, and Event Hubs

Every single one of these services needs **secrets** — connection strings, API keys, certificates, and encryption keys.

### Where NOT to Put Your Secrets:

```
❌ In appsettings.json (committed to Git)
    └── "ConnectionStrings": { "DB": "Server=prod;Password=P@ssw0rd" }
    └── Now every developer and every fork has your production password

❌ In Azure DevOps Pipeline Variables (unprotected)
    └── Visible in logs if you're not careful
    └── No versioning, no rotation

❌ In environment variables on the VM
    └── Anyone who SSHs in can see them
    └── No audit trail

✅ In Azure Key Vault
    └── Encrypted at rest with HSM-backed keys
    └── Access controlled via Azure RBAC + Access Policies
    └── Full audit trail via Azure Monitor
    └── Automatic certificate rotation
    └── Native integration with ALL Azure services
```

## ✅ What Is Azure Key Vault?

Azure Key Vault is a cloud service that provides a **secure store** for three types of objects:

| Object Type | What It Stores | Example |
|------------|---------------|---------|
| **Secrets** | Any string value (passwords, connection strings, API keys) | Database password, JWT signing key |
| **Keys** | Cryptographic keys (RSA, EC) for encrypt/decrypt/sign/verify | Data encryption key, signing key |
| **Certificates** | X.509 certificates (TLS/SSL) with automatic renewal | HTTPS certificates, client certificates |

### Key Vault Architecture:

## 🛠️ Creating a Key Vault

### Method 1: Azure Portal (UI)

1. Go to Azure Portal → Create a resource → Search "Key Vault"
2. Fill in:
   - **Subscription**: Your subscription
   - **Resource Group**: Your RG
   - **Key Vault Name**: `myapp-prod-kv` (globally unique!)
   - **Region**: Same as your app
   - **Pricing tier**: Standard (or Premium for HSM-backed keys)
3. Configure **Access Configuration** (RBAC recommended)
4. Configure **Networking** (Private Endpoint recommended for production)
5. Click "Create"

### Method 2: Azure CLI

```bash
# Create a resource group (if you don't have one)
az group create \
    --name myapp-rg \
    --location eastus

# Create the Key Vault
az keyvault create \
    --name myapp-prod-kv \
    --resource-group myapp-rg \
    --location eastus \
    --sku standard \
    --enable-rbac-authorization true \
    --enable-soft-delete true \
    --soft-delete-retention-days 90 \
    --enable-purge-protection true
```

**What each flag does:**

| Flag | Purpose |
|------|---------|
| `--name` | Globally unique name for the Key Vault |
| `--sku standard` | Standard tier (use `premium` for HSM-backed keys) |
| `--enable-rbac-authorization` | Use Azure RBAC instead of access policies (recommended!) |
| `--enable-soft-delete` | Deleted secrets are recoverable (default: enabled) |
| `--soft-delete-retention-days 90` | Keep deleted secrets for 90 days before permanent deletion |
| `--enable-purge-protection` | Prevents permanent deletion even by admins (critical for compliance!) |

### Method 3: Terraform

```hcl
# Get current Azure client info (for access policies)
data "azurerm_client_config" "current" {}

# Create the Key Vault
resource "azurerm_key_vault" "main" {
  name                        = "myapp-prod-kv"
  location                    = azurerm_resource_group.main.location
  resource_group_name         = azurerm_resource_group.main.name
  tenant_id                   = data.azurerm_client_config.current.tenant_id
  sku_name                    = "standard"
  
  # Security settings
  enable_rbac_authorization   = true
  soft_delete_retention_days  = 90
  purge_protection_enabled    = true
  
  # Network rules (restrict access)
  network_acls {
    default_action = "Deny"
    bypass         = "AzureServices"    # Allow Azure services to access
    
    # Allow specific VNets
    virtual_network_subnet_ids = [
      azurerm_subnet.app_subnet.id
    ]
    
    # Allow specific IPs (e.g., your CI/CD agent)
    ip_rules = ["203.0.113.0/24"]
  }

  tags = {
    Environment = "production"
    Team        = "platform"
  }
}
```

**What each part does:**
- `enable_rbac_authorization = true` → Uses Azure RBAC (modern approach, recommended over access policies)
- `purge_protection_enabled = true` → Even an admin can't permanently delete a secret during the retention period
- `network_acls` → Only allow access from specific VNets and IPs (defense in depth)
- `bypass = "AzureServices"` → Allow Azure services (App Service, AKS, etc.) to access even with network restrictions

## 🔐 Managing Secrets

### Adding Secrets

#### Azure CLI:

```bash
# Add a simple secret
az keyvault secret set \
    --vault-name myapp-prod-kv \
    --name "db-password" \
    --value "SuperSecret123!"

# Add a secret with content type (helps identify what it is)
az keyvault secret set \
    --vault-name myapp-prod-kv \
    --name "db-connection-string" \
    --value "Server=mydb.database.windows.net;Database=myapp;User=admin;Password=SuperSecret123!" \
    --content-type "application/x-connection-string"

# Add a secret with expiration date
az keyvault secret set \
    --vault-name myapp-prod-kv \
    --name "api-key" \
    --value "sk_live_abc123xyz" \
    --expires "2025-12-31T23:59:59Z"

# Add a secret from a file (great for certificates or long values)
az keyvault secret set \
    --vault-name myapp-prod-kv \
    --name "tls-private-key" \
    --file ./private-key.pem
```

#### Terraform:

```hcl
resource "azurerm_key_vault_secret" "db_password" {
  name         = "db-password"
  value        = "SuperSecret123!"
  key_vault_id = azurerm_key_vault.main.id
  content_type = "password"

  # Optional: Set expiration
  expiration_date = "2025-12-31T23:59:59Z"

  tags = {
    Environment = "production"
    ManagedBy   = "terraform"
  }
}

resource "azurerm_key_vault_secret" "db_connection_string" {
  name         = "db-connection-string"
  value        = "Server=${azurerm_mssql_server.main.fqdn};Database=myapp;User=admin;Password=${var.db_password}"
  key_vault_id = azurerm_key_vault.main.id
  content_type = "application/x-connection-string"
}
```

> ⚠️ **Terraform Warning:** Just like with AWS, the secret value will be in your Terraform state file. Always encrypt your state backend!

### Retrieving Secrets

#### Azure CLI:

```bash
# Get a secret (shows metadata + value)
az keyvault secret show \
    --vault-name myapp-prod-kv \
    --name "db-password"

# Get ONLY the value
az keyvault secret show \
    --vault-name myapp-prod-kv \
    --name "db-password" \
    --query "value" \
    --output tsv

# List all secrets (names only, not values)
az keyvault secret list \
    --vault-name myapp-prod-kv \
    --query "[].name" \
    --output tsv

# Get a specific version of a secret
az keyvault secret show \
    --vault-name myapp-prod-kv \
    --name "db-password" \
    --version "abc123def456"

# List all versions of a secret
az keyvault secret list-versions \
    --vault-name myapp-prod-kv \
    --name "db-password"
```

## 🔑 Access Control — Two Models

Azure Key Vault supports **two access control models**. You should pick one (don't mix them!):

### Model 1: Azure RBAC (Recommended ✅)

Uses standard Azure role-based access control. This is the **modern, recommended approach**.

| Built-in Role | What It Can Do |
|--------------|---------------|
| **Key Vault Administrator** | Full access — manage vault settings, secrets, keys, certificates |
| **Key Vault Secrets Officer** | Read, write, delete secrets |
| **Key Vault Secrets User** | Read secrets only |
| **Key Vault Crypto Officer** | Manage keys |
| **Key Vault Certificates Officer** | Manage certificates |
| **Key Vault Reader** | Read metadata (but NOT secret values) |

#### Assigning a role:

```bash
# Give a Managed Identity read access to secrets
az role assignment create \
    --role "Key Vault Secrets User" \
    --assignee <managed-identity-principal-id> \
    --scope /subscriptions/<sub-id>/resourceGroups/myapp-rg/providers/Microsoft.KeyVault/vaults/myapp-prod-kv
```

#### Terraform:

```hcl
# Give the App Service Managed Identity access to read secrets
resource "azurerm_role_assignment" "app_kv_access" {
  scope                = azurerm_key_vault.main.id
  role_definition_name = "Key Vault Secrets User"
  principal_id         = azurerm_linux_web_app.main.identity[0].principal_id
}
```

### Model 2: Access Policies (Legacy)

The older model — still works, but RBAC is better.

```bash
# Grant an app access to get and list secrets
az keyvault set-policy \
    --name myapp-prod-kv \
    --object-id <app-principal-id> \
    --secret-permissions get list
```

### Why RBAC Is Better:

| Feature | Access Policies | Azure RBAC |
|---------|----------------|------------|
| **Granularity** | Vault-level only | Individual secret level |
| **Management** | Per-vault configuration | Centralized Azure IAM |
| **Limit** | Max 1024 policies per vault | No limit |
| **Condition-based** | ❌ | ✅ (time-based, IP-based, etc.) |
| **Inheritance** | ❌ | ✅ (from subscription/RG) |

## 🤖 Managed Identity — The Magic of Passwordless Authentication

**Managed Identity** is the **best way** to authenticate to Key Vault from Azure services. No passwords, no secrets, no credentials to manage!

### What Is Managed Identity?

```
Traditional Way:
    App → stores a client_id + client_secret → uses them to call Key Vault
    └── 🤦 Now you have a secret to access your secrets! Chicken-and-egg problem!

Managed Identity Way:
    App → "Hey Azure, I'm this App Service" → Azure verifies → Access granted
    └── ✅ No credentials stored anywhere! Azure handles everything!
```

### Types of Managed Identity:

| Type | What It Is | When to Use |
|------|-----------|-------------|
| **System-assigned** | Azure creates an identity tied to the resource. Deleted when resource is deleted | Most cases — one resource, one identity |
| **User-assigned** | You create the identity separately. Can be shared across resources | When multiple resources need the same identity |

### Setting Up Managed Identity:

#### For Azure App Service:

```bash
# Enable system-assigned managed identity
az webapp identity assign \
    --name myapp-webapp \
    --resource-group myapp-rg

# Output: principalId: "aaaabbbb-cccc-dddd-eeee-ffff00001111"
```

#### For AKS (Azure Kubernetes Service):

```bash
# Create a user-assigned managed identity for workloads
az identity create \
    --name myapp-identity \
    --resource-group myapp-rg

# Enable workload identity on AKS
az aks update \
    --name myaks-cluster \
    --resource-group myapp-rg \
    --enable-oidc-issuer \
    --enable-workload-identity
```

#### For VMs:

```bash
# Enable system-assigned managed identity on a VM
az vm identity assign \
    --name myvm \
    --resource-group myapp-rg
```

#### Then Grant Access to Key Vault:

```bash
# Get the principal ID of the managed identity
PRINCIPAL_ID=$(az webapp identity show --name myapp-webapp --resource-group myapp-rg --query principalId -o tsv)

# Grant "Key Vault Secrets User" role
az role assignment create \
    --role "Key Vault Secrets User" \
    --assignee $PRINCIPAL_ID \
    --scope $(az keyvault show --name myapp-prod-kv --query id -o tsv)
```

## 💻 Accessing Key Vault from Applications

### Python

```python
from azure.identity import DefaultAzureCredential
from azure.keyvault.secrets import SecretClient

# DefaultAzureCredential automatically uses:
# - Managed Identity (when running in Azure)
# - Azure CLI credentials (when developing locally)
# - Environment variables (for CI/CD)
credential = DefaultAzureCredential()

# Create the Key Vault client
vault_url = "https://myapp-prod-kv.vault.azure.net"
client = SecretClient(vault_url=vault_url, credential=credential)

# Get a secret
db_password = client.get_secret("db-password")
print(f"Secret value: {db_password.value}")
print(f"Secret version: {db_password.properties.version}")
print(f"Created on: {db_password.properties.created_on}")

# List all secrets (names only)
secrets = client.list_properties_of_secrets()
for secret in secrets:
    print(f"Secret: {secret.name}, Enabled: {secret.enabled}")
```

**Line-by-line explanation:**

| Line | What It Does |
|------|-------------|
| `DefaultAzureCredential()` | Smart credential that automatically picks the right auth method based on where your code is running |
| `SecretClient(vault_url, credential)` | Creates a client to talk to your specific Key Vault |
| `client.get_secret("db-password")` | Fetches the secret. Returns an object with `.value` and metadata |
| `client.list_properties_of_secrets()` | Lists all secret names (but NOT values — for security) |

**Install the required packages:**

```bash
pip install azure-identity azure-keyvault-secrets
```

### .NET / C#

```csharp
using Azure.Identity;
using Azure.Security.KeyVault.Secrets;

// Create the client with Managed Identity
var client = new SecretClient(
    new Uri("https://myapp-prod-kv.vault.azure.net"),
    new DefaultAzureCredential()
);

// Get a secret
KeyVaultSecret secret = await client.GetSecretAsync("db-password");
string dbPassword = secret.Value;
Console.WriteLine($"DB Password: {dbPassword}");

// List all secrets
await foreach (SecretProperties secret in client.GetPropertiesOfSecretsAsync())
{
    Console.WriteLine($"Secret: {secret.Name}, Enabled: {secret.Enabled}");
}
```

### Node.js / JavaScript

```javascript
const { DefaultAzureCredential } = require("@azure/identity");
const { SecretClient } = require("@azure/keyvault-secrets");

async function main() {
    // Create credential and client
    const credential = new DefaultAzureCredential();
    const vaultUrl = "https://myapp-prod-kv.vault.azure.net";
    const client = new SecretClient(vaultUrl, credential);

    // Get a secret
    const secret = await client.getSecret("db-password");
    console.log(`Secret value: ${secret.value}`);

    // List all secrets
    for await (const secretProperties of client.listPropertiesOfSecrets()) {
        console.log(`Secret: ${secretProperties.name}`);
    }
}

main();
```

### Go

```go
package main

import (
    "context"
    "fmt"
    "log"

    "github.com/Azure/azure-sdk-for-go/sdk/azidentity"

    }

}
```

## 🔗 Integration with Azure Services

### 1. App Service — Key Vault References

Azure App Service can **directly reference** Key Vault secrets in its configuration — no code changes needed!

```bash
# Set an app setting that references a Key Vault secret
az webapp config appsettings set \
    --name myapp-webapp \
    --resource-group myapp-rg \
    --settings "DB_PASSWORD=@Microsoft.KeyVault(VaultName=myapp-prod-kv;SecretName=db-password)"
```

**The magic syntax:**

```
@Microsoft.KeyVault(VaultName=myapp-prod-kv;SecretName=db-password)
```

This tells App Service: "Don't use this literal string. Instead, fetch the value from Key Vault!"

**Your app just reads it like a normal environment variable:**

```python
import os
db_password = os.environ.get("DB_PASSWORD")
# This will be the actual secret value, not the @Microsoft.KeyVault(...) reference!
```

#### Terraform:

```hcl
resource "azurerm_linux_web_app" "main" {
  name                = "myapp-webapp"
  resource_group_name = azurerm_resource_group.main.name
  location            = azurerm_resource_group.main.location
  service_plan_id     = azurerm_service_plan.main.id

  identity {
    type = "SystemAssigned"
  }

  app_settings = {
    "DB_PASSWORD" = "@Microsoft.KeyVault(VaultName=${azurerm_key_vault.main.name};SecretName=db-password)"
    "API_KEY"     = "@Microsoft.KeyVault(VaultName=${azurerm_key_vault.main.name};SecretName=api-key)"
    "LOG_LEVEL"   = "warn"    # Non-sensitive values can be plain text
  }

  site_config {
    # ...
  }
}
```

### 2. AKS — Secrets Store CSI Driver

AKS has native support for the **Secrets Store CSI Driver** with the Azure Key Vault provider.

#### Enable the CSI Driver Add-on:

```bash
# Enable the Key Vault CSI driver on AKS
az aks enable-addons \
    --name myaks-cluster \
    --resource-group myapp-rg \
    --addons azure-keyvault-secrets-provider
```

#### Create a SecretProviderClass:

```yaml
# secret-provider-class.yaml
apiVersion: secrets-store.csi.x-k8s.io/v1
kind: SecretProviderClass
metadata:
  name: azure-kv-secrets
  namespace: default
spec:
  provider: azure
  parameters:
    usePodIdentity: "false"
    useVMManagedIdentity: "false"
    clientID: "<managed-identity-client-id>"     # Workload Identity client ID
    keyvaultName: "myapp-prod-kv"
    tenantId: "<your-azure-tenant-id>"
    objects: |
      array:
        - |
          objectName: db-password
          objectType: secret
        - |
          objectName: api-key
          objectType: secret
        - |
          objectName: tls-cert
          objectType: secret
  # Optional: Sync to Kubernetes Secrets
  secretObjects:
    - secretName: myapp-secrets
      type: Opaque
      data:
        - objectName: db-password
          key: DB_PASSWORD
        - objectName: api-key
          key: API_KEY
```

#### Mount in Your Pod:

```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 1
  selector:
    matchLabels:
      app: myapp
  template:

            driver: secrets-store.csi.k8s.io
            readOnly: true
            volumeAttributes:
              secretProviderClass: "azure-kv-secrets"
```

### 3. Azure DevOps Pipelines

Use Key Vault secrets in your CI/CD pipelines:

```yaml
# azure-pipelines.yml

# Method 1: Use the AzureKeyVault task
steps:
  - task: AzureKeyVault@2
    inputs:
      azureSubscription: 'MyAzureServiceConnection'
      KeyVaultName: 'myapp-prod-kv'
      SecretsFilter: 'db-password,api-key'    # Comma-separated list, or '*' for all
      RunAsPreJob: true                        # Fetch before any other task

  - script: |
      echo "Using secrets in script..."
      # Secrets are available as pipeline variables
      # (They're masked in logs automatically)
      echo "DB Password is set: $(db-password)"
    displayName: 'Use Key Vault Secrets'

# Method 2: Use Variable Groups linked to Key Vault
# (Configure in Azure DevOps UI: Pipelines → Library → Variable Groups → Link to Key Vault)
```

**How the `AzureKeyVault` task works:**
1. Authenticates to Key Vault using the service connection
2. Downloads the specified secrets
3. Sets them as **pipeline variables** (automatically masked in logs)
4. You reference them as `$(secret-name)` in subsequent steps

### 4. Azure Functions

```python
# Azure Function with Key Vault integration

import azure.functions as func
from azure.identity import DefaultAzureCredential
from azure.keyvault.secrets import SecretClient
import os

# Initialize Key Vault client (cached across invocations)
credential = DefaultAzureCredential()
kv_client = SecretClient(
    vault_url="https://myapp-prod-kv.vault.azure.net",
    credential=credential
)

def main(req: func.HttpRequest) -> func.HttpResponse:
    # Get secret from Key Vault
    db_password = kv_client.get_secret("db-password").value
    
    # Or use App Settings with Key Vault references (simpler!)
    # db_password = os.environ["DB_PASSWORD"]  # Set via Key Vault reference
    
    return func.HttpResponse(f"Connected to database successfully!", status_code=200)
```

## 📜 Certificates in Key Vault

Key Vault can **store, manage, and auto-renew** TLS/SSL certificates.

### Import an Existing Certificate:

```bash
# Import a PFX certificate
az keyvault certificate import \
    --vault-name myapp-prod-kv \
    --name "myapp-tls-cert" \
    --file ./certificate.pfx \
    --password "pfx-password"
```

### Create a Self-Signed Certificate (for testing):

```bash
az keyvault certificate create \
    --vault-name myapp-prod-kv \
    --name "myapp-self-signed" \
    --policy "$(az keyvault certificate get-default-policy)"
```

### Auto-Renew with DigiCert/GlobalSign:

Key Vault can **automatically renew** certificates from integrated CAs:

```bash
# Create a certificate with auto-renewal policy
az keyvault certificate create \
    --vault-name myapp-prod-kv \
    --name "myapp-production-cert" \
    --policy '{
        "issuerParameters": {
            "name": "DigiCert"
        },
        "x509CertificateProperties": {
            "subject": "CN=myapp.example.com",
            "validityInMonths": 12
        },
        "lifetimeActions": [
            {
                "action": { "actionType": "AutoRenew" },
                "trigger": { "daysBeforeExpiry": 30 }
            }
        ]
    }'
```

**This means:**
- Certificate is issued by DigiCert
- Valid for 12 months
- Auto-renews 30 days before expiry
- **No manual certificate management ever again!** 🎉

## 🔐 Encryption Keys in Key Vault

Key Vault can manage **cryptographic keys** for encryption, decryption, signing, and verification.

### Why Use Key Vault for Keys?

```
Without Key Vault:
    Your app generates a key → stores it in a config file → 😱
    Key rotation? Manual process → 😱
    Key access audit? None → 😱

With Key Vault:
    Key Vault generates the key → stored in HSM → ✅
    Encryption/decryption happens INSIDE Key Vault → key never leaves → ✅
    Full audit trail → ✅
    Automatic rotation possible → ✅
```

### Create an Encryption Key:

```bash
# Create an RSA key
az keyvault key create \
    --vault-name myapp-prod-kv \
    --name "data-encryption-key" \
    --kty RSA \
    --size 2048

# Create an EC key (for signing)
az keyvault key create \
    --vault-name myapp-prod-kv \
    --name "signing-key" \
    --kty EC \
    --curve P-256
```

### Encrypt/Decrypt with Key Vault (Python):

```python
from azure.identity import DefaultAzureCredential
from azure.keyvault.keys import KeyClient
from azure.keyvault.keys.crypto import CryptographyClient, EncryptionAlgorithm

credential = DefaultAzureCredential()
vault_url = "https://myapp-prod-kv.vault.azure.net"

# Get the key
key_client = KeyClient(vault_url, credential)
key = key_client.get_key("data-encryption-key")

# Create a crypto client for this key
crypto_client = CryptographyClient(key, credential)

# Encrypt data
plaintext = b"This is my secret data that needs encryption"
result = crypto_client.encrypt(EncryptionAlgorithm.rsa_oaep, plaintext)
ciphertext = result.ciphertext
print(f"Encrypted: {ciphertext[:50]}...")

# Decrypt data
result = crypto_client.decrypt(EncryptionAlgorithm.rsa_oaep, ciphertext)
decrypted = result.plaintext
print(f"Decrypted: {decrypted.decode()}")
```

**The key never leaves Key Vault!** The encryption/decryption happens server-side. Your app sends the data, Key Vault encrypts it and sends back the result.

## 🔄 Secret Versioning and Rotation

### How Versioning Works:

Every time you update a secret, Key Vault creates a **new version**:

```bash
# Version 1
az keyvault secret set --vault-name myapp-prod-kv --name "db-password" --value "Password_v1"

# Version 2 (automatically created)
az keyvault secret set --vault-name myapp-prod-kv --name "db-password" --value "Password_v2"

# Version 3
az keyvault secret set --vault-name myapp-prod-kv --name "db-password" --value "Password_v3"

# Get the LATEST version (default)
az keyvault secret show --vault-name myapp-prod-kv --name "db-password"
# Returns: Password_v3

# List all versions
az keyvault secret list-versions --vault-name myapp-prod-kv --name "db-password"

# Get a SPECIFIC version
az keyvault secret show --vault-name myapp-prod-kv --name "db-password" --version "abc123..."
```

### Setting Up Secret Rotation with Azure Event Grid:

You can automate secret rotation using **Azure Functions + Event Grid**:

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  Key Vault    │     │  Event Grid   │     │  Azure       │     │  Key Vault   │
│  Secret       │────▶│  (Near Expiry │────▶│  Function    │────▶│  (New Secret │
│  Near Expiry  │     │   Event)      │     │  (Rotation)  │     │   Version)   │
└──────────────┘     └──────────────┘     └──────────────┘     └──────────────┘
```

#### Step 1: Create an Event Grid Subscription

```bash
# Subscribe to Key Vault events
az eventgrid event-subscription create \
    --name "secret-rotation-sub" \
    --source-resource-id $(az keyvault show --name myapp-prod-kv --query id -o tsv) \
    --endpoint $(az functionapp function show --name myapp-rotation-func --function-name RotateSecret --query id -o tsv) \
    --endpoint-type azurefunction \
    --included-event-types "Microsoft.KeyVault.SecretNearExpiry"
```

#### Step 2: Create the Rotation Function (Python)

```python
import azure.functions as func
from azure.identity import DefaultAzureCredential
from azure.keyvault.secrets import SecretClient
import secrets
import string

def main(event: func.EventGridEvent):
    """Triggered when a Key Vault secret is near expiry."""
    
    # Parse the event
    event_data = event.get_json()
    secret_name = event_data.get("ObjectName")
    vault_name = event_data.get("VaultName")
    
    # Create Key Vault client
    credential = DefaultAzureCredential()
    vault_url = f"https://{vault_name}.vault.azure.net"
    client = SecretClient(vault_url=vault_url, credential=credential)
    
    # Generate a new password
    alphabet = string.ascii_letters + string.digits + string.punctuation
    new_password = ''.join(secrets.choice(alphabet) for _ in range(32))
    
    # Update the secret in Key Vault
    client.set_secret(secret_name, new_password)
    
    # TODO: Also update the password in the actual service
    # (e.g., update the database password, rotate the API key, etc.)
    
    print(f"Successfully rotated secret: {secret_name}")
```

## 🌐 Networking — Securing Key Vault Access

Instead of accessing Key Vault over the public internet, use a **private endpoint** that keeps traffic within your VNet:

```bash

    --name myapp-kv-pe \
    --resource-group myapp-rg \
    --vnet-name myapp-vnet \

    --private-connection-resource-id $(az keyvault show --name myapp-prod-kv --query id -o tsv) \
    --group-ids vault \
    --connection-name kv-connection
```

## 🛡️ Production Best Practices

| Practice | What to Do |
|----------|-----------|
| **Use RBAC over Access Policies** | RBAC is more granular, centralized, and supports conditions |
| **Enable Soft Delete + Purge Protection** | Prevents accidental permanent deletion |
| **Use Managed Identity** | No credentials to manage. Zero secrets to access secrets |
| **Private Endpoints** | Keep Key Vault traffic within your VNet |
| **Separate Vaults per Environment** | `myapp-dev-kv`, `myapp-staging-kv`, `myapp-prod-kv` |
| **Enable Diagnostic Logging** | Know who accessed what, when |
| **Set Expiry on Secrets** | Force rotation by setting expiration dates |
| **Use Key Vault References** | For App Service — no code changes needed |
| **Tag Everything** | Environment, team, application, managed-by |
| **Backup Your Vault** | Use `az keyvault backup` for disaster recovery |

### Vault Backup:

```bash
# Backup individual secrets
az keyvault secret backup \
    --vault-name myapp-prod-kv \
    --name "db-password" \
    --file db-password.backup

# Restore from backup (to same or different vault in same subscription)
az keyvault secret restore \
    --vault-name myapp-dr-kv \
    --file db-password.backup
```

## 🧪 Troubleshooting Common Issues

| Problem | Cause | Fix |
|---------|-------|-----|
| `AccessDenied` or `Forbidden` | Missing RBAC role or access policy | Assign "Key Vault Secrets User" role to the identity |
| `SecretNotFound` | Wrong secret name or vault name | Check name, vault, and subscription |
| `NetworkError` / timeout | Key Vault blocked by network rules | Add your IP/VNet to Key Vault firewall, or use Private Endpoint |
| App Service shows `@Microsoft.KeyVault(...)` as literal value | Managed Identity not configured, or no access | Enable Managed Identity and grant RBAC access |
| `SecretDisabled` | Secret was manually disabled | Re-enable it: `az keyvault secret set-attributes --enabled true` |
| `Throttled` (429) | Too many requests | Implement caching, reduce call frequency |

## ⚖️ Azure Key Vault vs AWS Secrets Manager vs HashiCorp Vault

| Feature | Azure Key Vault | AWS Secrets Manager | HashiCorp Vault |
|---------|----------------|-------------------|----------------|
| **Provider** | Azure | AWS | Multi-cloud / Self-hosted |
| **Managed?** | ✅ Fully managed | ✅ Fully managed | Self-managed (or HCP Vault) |
| **Secrets** | ✅ | ✅ | ✅ |
| **Encryption Keys** | ✅ (HSM-backed) | ✅ (KMS) | ✅ (Transit engine) |
| **Certificates** | ✅ (with auto-renew) | ❌ (use ACM) | ✅ (PKI engine) |
| **Auto-rotation** | Via Event Grid + Functions | ✅ Built-in (Lambda) | ✅ Dynamic secrets |
| **Cost** | ~$0.03/10K operations | $0.40/secret/month | Free (OSS) / paid (HCP) |
| **Best for** | Azure-native workloads | AWS-native workloads | Multi-cloud, self-hosted |
