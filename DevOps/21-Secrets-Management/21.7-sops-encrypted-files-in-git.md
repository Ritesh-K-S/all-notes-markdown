# 21.7 — SOPS — Encrypted Files in Git

> **One Line Answer:** SOPS (Secrets OPerationS) is a tool by Mozilla that lets you **encrypt secret values inside configuration files** (YAML, JSON, ENV, INI) so you can **safely store them in Git** — only the values are encrypted, the keys (field names) stay readable.

## 🤔 The Problem — Why Do We Need SOPS?

Let's say you have a Kubernetes secret file like this:

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: my-app-secrets
type: Opaque
stringData:
  DB_PASSWORD: "super-secret-password-123"
  API_KEY: "sk-abc123def456"
  JWT_SECRET: "my-jwt-secret-token"
```

Now you want to push this to Git so your team can use it, your CI/CD pipeline can deploy it, and everything is version-controlled.

**But wait!** 😰

If you push this file as-is to Git, **anyone who has access to the repo can see your passwords in plain text!** Even if the repo is private, that's a huge security risk. What if someone's laptop gets stolen? What if a former employee still has access?

### Common (Bad) Solutions People Try:

| Approach | Why It's Bad |
|----------|-------------|
| Put secrets in `.gitignore` | Now they're not version-controlled, and every dev needs to get them manually |
| Share secrets via Slack/email | No audit trail, secrets scattered everywhere |
| Use environment variables only | Hard to manage across environments, not version-controlled |
| Base64 encode them | Base64 is NOT encryption! Anyone can decode it instantly |

### The SOPS Solution:

What if you could **encrypt just the secret values** in the file, keep the structure readable, store it in Git, and only people with the right encryption key could decrypt it?

**That's exactly what SOPS does!** ✅

## ✅ What is SOPS?

**SOPS** stands for **Secrets OPerationS**. It was created by Mozilla and is now maintained as an open-source project under the CNCF.

Here's the magic — SOPS encrypts **only the values**, not the keys:

### Before Encryption (Plain text):

```yaml
db:
  username: admin
  password: super-secret-123
api:
  key: sk-abc123def456
```

### After SOPS Encryption:

```yaml
db:
  username: ENC[AES256_GCM,data:kP3l2w==,iv:abc...,tag:def...,type:str]
  password: ENC[AES256_GCM,data:x9Kj3mN4pQ==,iv:ghi...,tag:jkl...,type:str]
api:
  key: ENC[AES256_GCM,data:mN0pQ7rS8t==,iv:mno...,tag:pqr...,type:str]
sops:
  # metadata about encryption (which key was used, etc.)
```

**See what happened?**
- The structure (`db.username`, `db.password`, `api.key`) is still **readable** — you can tell what secrets exist
- The actual **values** are encrypted — nobody can read them without the decryption key
- There's a `sops` metadata section that stores encryption details

This is incredibly useful because:
- You can **review diffs** in Git (you can see which fields changed, even if you can't see the values)
- You can **store everything in Git** safely
- You know **what secrets exist** without seeing their values

## 🔐 How Does SOPS Encrypt? — Supported Backends

SOPS doesn't have its own encryption. Instead, it uses **external encryption providers** (backends). Think of SOPS as a smart wrapper that knows how to work with config files, and it delegates the actual encryption to these providers:

| Backend | What It Is | Best For |
|---------|-----------|----------|
| **AWS KMS** | Amazon's Key Management Service | Teams using AWS |
| **GCP KMS** | Google Cloud's Key Management Service | Teams using GCP |
| **Azure Key Vault** | Microsoft's key management | Teams using Azure |
| **age** | A modern, simple encryption tool | Local development, open-source projects |
| **PGP/GPG** | The classic encryption standard | Legacy setups (age is preferred now) |
| **HashiCorp Vault Transit** | Vault's encryption-as-a-service | Teams already using Vault |

### Which Backend Should You Choose?

- **Using a cloud provider?** → Use that cloud's KMS (AWS KMS, GCP KMS, Azure Key Vault)
- **Open-source / local / no cloud?** → Use **age** (simpler and more modern than PGP)
- **Already using Vault?** → Use Vault Transit

## 🎯 Using SOPS with `age` — Step-by-Step Tutorial

`age` is the simplest and most recommended backend for getting started. Let's walk through the entire workflow.

### Step 1: Generate an `age` Key Pair

```bash
age-keygen -o ~/.sops/age/keys.txt
```

This creates a file like:

```
# created: 2024-01-15T10:30:00Z
# public key: age1ql3z7hjy54pw3hyww5ayyfg7zqgvc7w3j2elw8zmrj2kg5sfn9aqmcac8p
AGE-SECRET-KEY-1QQQQQQQQQQQQQQQQQQQQQQQQQQQQQQQQQQQQQQQQQQQQQQQQQQQQQQQQQQ
```

- **Public key** (`age1...`) — used to **encrypt**. You can share this with your team.
- **Secret key** (`AGE-SECRET-KEY-...`) — used to **decrypt**. Keep this private! Never commit this to Git.

### Step 2: Tell SOPS Where Your Key Is

SOPS needs to know where your `age` key file is. Set an environment variable:

```bash
export SOPS_AGE_KEY_FILE=~/.sops/age/keys.txt
```

Add this to your `~/.bashrc` or `~/.zshrc` so it's always set:

```bash
echo 'export SOPS_AGE_KEY_FILE=~/.sops/age/keys.txt' >> ~/.bashrc
source ~/.bashrc
```

### Step 3: Create a `.sops.yaml` Configuration File

This is a file you put in the **root of your Git repository**. It tells SOPS which keys to use for which files:

```yaml
# .sops.yaml
creation_rules:
  # Encrypt all YAML files in the secrets/ directory with this age key
  - path_regex: secrets/.*\.yaml$
    age: "age1ql3z7hjy54pw3hyww5ayyfg7zqgvc7w3j2elw8zmrj2kg5sfn9aqmcac8p"

  # Encrypt all JSON files in the config/ directory
  - path_regex: config/.*\.json$
    age: "age1ql3z7hjy54pw3hyww5ayyfg7zqgvc7w3j2elw8zmrj2kg5sfn9aqmcac8p"
```

**What's happening here?**
- `path_regex` — a regex pattern that matches file paths. SOPS will use the corresponding key for files matching this pattern.
- `age` — the **public key** to encrypt with. You can list multiple keys (comma-separated) for multiple recipients.

### Step 4: Create a Secrets File

Create a plain-text secrets file:

```bash
mkdir secrets
cat > secrets/database.yaml << 'EOF'
db:
  host: "my-database.example.com"
  port: 5432
  username: "app_user"
  password: "super-secret-password-123"
  connection_string: "postgresql://app_user:super-secret-password-123@my-database.example.com:5432/mydb"
redis:
  host: "redis.example.com"
  password: "redis-secret-456"
EOF
```

### Step 5: Encrypt the File

```bash
sops encrypt secrets/database.yaml > secrets/database.enc.yaml
```

Or encrypt the file **in place** (replaces the original):

```bash
sops encrypt -i secrets/database.yaml
```

The `-i` flag means "in-place" — it overwrites the file with the encrypted version.

**After encryption, the file looks something like this:**

```yaml
db:
  host: ENC[AES256_GCM,data:4kR7mN...,iv:...,tag:...,type:str]
  port: ENC[AES256_GCM,data:8jQ==,iv:...,tag:...,type:int]
  username: ENC[AES256_GCM,data:3pL9w==,iv:...,tag:...,type:str]
  password: ENC[AES256_GCM,data:x9Kj3m...,iv:...,tag:...,type:str]
  connection_string: ENC[AES256_GCM,data:mN0pQ7r...,iv:...,tag:...,type:str]
redis:
  host: ENC[AES256_GCM,data:5tY2x...,iv:...,tag:...,type:str]
  password: ENC[AES256_GCM,data:kL8nP...,iv:...,tag:...,type:str]
sops:
  age:
    - recipient: age1ql3z7hjy54pw3hyww5ayyfg7zqgvc7w3j2elw8zmrj2kg5sfn9aqmcac8p
      enc: |
        -----BEGIN AGE ENCRYPTED FILE-----
        ...encrypted data key...
        -----END AGE ENCRYPTED FILE-----
  lastmodified: "2024-01-15T10:35:00Z"
  mac: ENC[AES256_GCM,data:...,iv:...,tag:...,type:str]
  version: 3.9.4
```

**Now this file is safe to commit to Git!** 🎉

### Step 6: Decrypt the File

```bash
# Decrypt and print to stdout
sops decrypt secrets/database.yaml

# Decrypt in place (overwrite with plaintext — be careful!)
sops decrypt -i secrets/database.yaml

# Decrypt to a different file
sops decrypt secrets/database.yaml > secrets/database.plain.yaml
```

### Step 7: Edit an Encrypted File

This is one of SOPS's coolest features. You can **edit an encrypted file directly**:

```bash
sops secrets/database.yaml
```

This will:
1. Decrypt the file in memory
2. Open it in your `$EDITOR` (vim, nano, etc.)
3. You edit the plain text
4. When you save and quit, SOPS **re-encrypts** the file automatically

You never have to manually decrypt → edit → re-encrypt!

## ☁️ Using SOPS with AWS KMS

For production environments, you'll often use a cloud KMS. Here's how to set it up with AWS KMS.

### Step 1: Create a KMS Key in AWS

```bash
aws kms create-key --description "SOPS encryption key for my-project"
```

This returns a key ARN like:

```
arn:aws:kms:us-east-1:123456789012:key/abcd1234-ef56-gh78-ij90-klmnopqrstuv
```

### Step 2: Configure `.sops.yaml`

```yaml
# .sops.yaml
creation_rules:
  - path_regex: secrets/production/.*\.yaml$
    kms: "arn:aws:kms:us-east-1:123456789012:key/abcd1234-ef56-gh78-ij90-klmnopqrstuv"

  - path_regex: secrets/staging/.*\.yaml$
    kms: "arn:aws:kms:us-east-1:123456789012:key/wxyz5678-ab12-cd34-ef56-ghijklmnopqr"
```

**Notice**: Different environments can use different KMS keys! This means:
- Production secrets can only be decrypted by people/services with access to the production KMS key
- Staging secrets use a different key with broader access

### Step 3: Encrypt & Decrypt (Same as Before)

```bash
# Encrypt
sops encrypt -i secrets/production/database.yaml

# Decrypt
sops decrypt secrets/production/database.yaml

# Edit
sops secrets/production/database.yaml
```

AWS KMS handles the encryption/decryption automatically as long as your AWS credentials have permission to use the KMS key.

## 🔑 Using Multiple Keys (Key Groups)

You can encrypt a file with **multiple keys** so that multiple people or systems can decrypt it:

```yaml
# .sops.yaml
creation_rules:
  - path_regex: secrets/.*\.yaml$
    key_groups:
      - age:
          - age1ql3z7hjy54pw3hyww5ayyfg7zqgvc7w3j2elw8zmrj2kg5sfn9aqmcac8p  # Developer 1
          - age1abc123def456ghi789jkl012mno345pqr678stu901vwx234yz567abc890  # Developer 2
        kms:
          - arn: "arn:aws:kms:us-east-1:123456789012:key/abcd1234-..."       # CI/CD pipeline
```

With this setup:
- **Developer 1** can decrypt with their `age` key
- **Developer 2** can decrypt with their `age` key
- **CI/CD pipeline** can decrypt using AWS KMS (with the right IAM permissions)

All of them can decrypt the same encrypted file!

## 🚫 Encrypting Only Specific Fields

Sometimes you don't want to encrypt everything. For example, the database `host` and `port` aren't really secrets — only the `password` is.

SOPS lets you control **which fields get encrypted** using `--encrypted-regex` or `--unencrypted-regex`:

### Method 1: Encrypt Only Fields Matching a Pattern

```bash
sops encrypt --encrypted-regex '^(password|secret|key|token)$' -i secrets/database.yaml
```

This only encrypts fields whose names are `password`, `secret`, `key`, or `token`. Everything else stays in plain text.

### Method 2: Put It in `.sops.yaml`

```yaml
# .sops.yaml
creation_rules:
  - path_regex: secrets/.*\.yaml$
    age: "age1ql3z7hjy54pw3hyww5ayyfg7zqgvc7w3j2elw8zmrj2kg5sfn9aqmcac8p"
    encrypted_regex: "^(password|secret|key|token|api_key|connection_string)$"
```

### Result:

```yaml
db:
  host: my-database.example.com          # NOT encrypted (readable!)
  port: 5432                              # NOT encrypted (readable!)
  username: app_user                      # NOT encrypted (readable!)
  password: ENC[AES256_GCM,data:x9K...,iv:...,tag:...,type:str]  # ENCRYPTED!
  connection_string: ENC[AES256_GCM,data:mN0...,iv:...,tag:...,type:str]  # ENCRYPTED!
redis:
  host: redis.example.com                # NOT encrypted
  password: ENC[AES256_GCM,data:kL8...,iv:...,tag:...,type:str]  # ENCRYPTED!
```

This makes Git diffs even more useful — you can see all the non-secret configuration changes clearly.

## 🔄 SOPS in CI/CD Pipelines

The real power of SOPS comes when you integrate it into your CI/CD workflow. Here's how secrets flow from Git to your application.

### GitHub Actions Example:

```yaml
# .github/workflows/deploy.yaml
name: Deploy Application
on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install SOPS
        run: |
          curl -LO https://github.com/getsops/sops/releases/download/v3.9.4/sops-v3.9.4.linux.amd64
          sudo mv sops-v3.9.4.linux.amd64 /usr/local/bin/sops
          sudo chmod +x /usr/local/bin/sops

      - name: Decrypt secrets
        env:
          SOPS_AGE_KEY: ${{ secrets.SOPS_AGE_SECRET_KEY }}
        run: |
          # Decrypt the secrets file
          sops decrypt secrets/production/database.yaml > /tmp/database-secrets.yaml

          # Use the decrypted secrets (example: create Kubernetes secret)
          kubectl create secret generic db-secrets \
            --from-file=/tmp/database-secrets.yaml \
            --dry-run=client -o yaml | kubectl apply -f -

          # Clean up the decrypted file
          rm /tmp/database-secrets.yaml

      - name: Deploy to Kubernetes
        run: |
          kubectl apply -f k8s/
```

**Key points:**
- The `SOPS_AGE_KEY` environment variable contains the **secret key** (stored as a GitHub secret)
- Decrypted files are written to `/tmp/` (not to the workspace) and cleaned up immediately
- The encrypted files in Git are safe — only the CI/CD pipeline can decrypt them

### With AWS KMS in CI/CD:

If you're using AWS KMS, you don't need to pass any key — just give the CI/CD runner the right **IAM permissions** to access the KMS key:

```yaml
      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789012:role/github-actions-deploy
          aws-region: us-east-1

      - name: Decrypt secrets
        run: |
          # AWS KMS decryption happens automatically via IAM role
          sops decrypt secrets/production/database.yaml > /tmp/secrets.yaml
```

## 🔄 SOPS with Helm & Kubernetes

A very common pattern is using SOPS with Helm charts to manage secret values.

### Using `helm-secrets` Plugin:

```bash
# Install the helm-secrets plugin
helm plugin install https://github.com/jkroepke/helm-secrets
```

Now you can have encrypted Helm values files:

```bash
# Create an encrypted values file
sops encrypt values-secret.yaml > values-secret.enc.yaml

# Use it directly with Helm
helm secrets upgrade my-app ./my-chart \
  -f values.yaml \
  -f secrets://values-secret.enc.yaml
```

### Example — Encrypted Values File:

```yaml
# values-secret.yaml (before encryption)
database:
  password: "super-secret-password-123"
redis:
  password: "redis-secret-456"
externalApi:
  apiKey: "sk-abc123def456"
```

After encryption with SOPS, commit the encrypted version. The `helm-secrets` plugin automatically decrypts it during `helm install` or `helm upgrade`.

## 🔄 Key Rotation

Security best practice says you should **rotate encryption keys** periodically. SOPS makes this easy:

```bash
# Update the key in .sops.yaml to the new key, then:
sops updatekeys secrets/database.yaml
```

This re-encrypts the file's data key with the new encryption key without changing the secret values themselves.

### To Rotate Keys for All Files:

```bash
# Find all encrypted files and rotate their keys
find secrets/ -name "*.yaml" -exec sops updatekeys {} \;
```

## ✅ Best Practices for SOPS

| Practice | Why |
|----------|-----|
| **Use `.sops.yaml`** in repo root | So everyone uses the same encryption rules automatically |
| **Use cloud KMS for production** | Better access control, audit logging, automatic key rotation |
| **Use `age` for local dev** | Simple, fast, no cloud dependency |
| **Use `encrypted_regex`** | Only encrypt what's actually secret — makes diffs more useful |
| **Never commit the secret/private key** | Add `keys.txt` to `.gitignore` |
| **Use key groups for team access** | Multiple people can decrypt without sharing keys |
| **Rotate keys periodically** | Limits the damage if a key is compromised |
| **Use different keys per environment** | Production and staging should have different access controls |
| **Add a pre-commit hook** | Prevent accidental commits of unencrypted secrets |

### Pre-commit Hook Example:

```bash
#!/bin/bash
# .git/hooks/pre-commit
# Prevent committing unencrypted secret files

for file in $(git diff --cached --name-only | grep "^secrets/"); do
  if ! grep -q "sops" "$file" 2>/dev/null; then
    echo "ERROR: $file appears to be unencrypted! Encrypt it with sops first."
    exit 1
  fi
done
```

## 🆚 SOPS vs Other Secret Tools

| Feature | SOPS | Vault | Sealed Secrets | External Secrets |
|---------|------|-------|----------------|-----------------|
| **Approach** | Encrypt files in Git | Centralized secret server | Encrypt K8s secrets for Git | Sync from external stores |
| **Complexity** | Low | High | Medium | Medium |
| **K8s Required?** | No | No | Yes | Yes |
| **GitOps Friendly** | ✅ Excellent | ⚠️ Needs integration | ✅ Excellent | ✅ Good |
| **Cloud KMS Support** | ✅ Built-in | ✅ Built-in | ❌ | ✅ Via providers |
| **Dynamic Secrets** | ❌ | ✅ | ❌ | ❌ |
| **Best For** | Any project, any size | Enterprise, dynamic secrets | Kubernetes-only | Multi-cloud K8s |

## 🧪 Quick Reference — Common Commands

```bash
# Encrypt a file in-place
sops encrypt -i secrets/myfile.yaml

# Decrypt a file to stdout
sops decrypt secrets/myfile.yaml

# Decrypt a file in-place
sops decrypt -i secrets/myfile.yaml

# Edit an encrypted file (opens in $EDITOR)
sops secrets/myfile.yaml

# Encrypt with a specific age key (overriding .sops.yaml)
sops encrypt --age "age1..." -i secrets/myfile.yaml

# Encrypt with AWS KMS
sops encrypt --kms "arn:aws:kms:..." -i secrets/myfile.yaml

# Encrypt only specific fields
sops encrypt --encrypted-regex '^(password|secret|key)$' -i secrets/myfile.yaml

# Rotate keys (after updating .sops.yaml)
sops updatekeys secrets/myfile.yaml

# Check file status (encrypted or not)
sops filestatus secrets/myfile.yaml

# Extract a single value from an encrypted file
sops decrypt --extract '["db"]["password"]' secrets/database.yaml
```

## 📝 Summary

| Concept | Key Point |
|---------|-----------|
| **What** | SOPS encrypts values in config files (YAML, JSON, ENV, INI) |
| **Why** | Store secrets safely in Git without exposing them |
| **How** | Uses external encryption backends (age, AWS KMS, GCP KMS, Azure Key Vault, PGP) |
| **Key Feature** | Encrypts only values, keeps keys/structure readable |
| **Best Backend** | Cloud KMS for production, `age` for local dev |
| **GitOps** | Perfect for GitOps — encrypted secrets live in Git alongside your code |
| **CI/CD** | Decrypt in pipeline using environment variables or IAM roles |
| **Edit** | `sops filename.yaml` opens a decrypted editor, re-encrypts on save |

> **Bottom Line:** SOPS is the simplest, most GitOps-friendly way to manage secrets in version control. If you want to keep secrets in Git (and you should, for traceability), SOPS is your best friend. 🔐
