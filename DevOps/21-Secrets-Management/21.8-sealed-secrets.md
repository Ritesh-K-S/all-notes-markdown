# 21.8 — Sealed Secrets — Kubernetes-Native Secret Encryption

> **One Line Answer:** Sealed Secrets is a Kubernetes controller + CLI tool by Bitnami that lets you **encrypt Kubernetes Secrets into "SealedSecrets"** that are safe to store in Git — only the Sealed Secrets controller running **inside your cluster** can decrypt them.

## 🤔 The Problem — Kubernetes Secrets Are NOT Secret!

Let's talk about a dirty little secret in Kubernetes (pun intended). When you create a Kubernetes Secret, the data is stored as **base64-encoded** — NOT encrypted.

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: my-database-creds
type: Opaque
data:
  username: YWRtaW4=          # base64 of "admin"
  password: c3VwZXItc2VjcmV0  # base64 of "super-secret"
```

Anyone can decode base64 in 2 seconds:

```bash
echo "c3VwZXItc2VjcmV0" | base64 -d
# Output: super-secret
```

**Base64 is an encoding, NOT encryption.** It provides ZERO security.

### The GitOps Problem:

If you're doing GitOps (storing all your Kubernetes manifests in Git), you have a problem:

```
Your Git Repo:
├── deployments/
│   └── my-app.yaml          ✅ Safe to commit
├── services/
│   └── my-service.yaml      ✅ Safe to commit
└── secrets/
    └── my-database-creds.yaml  ❌ NOT safe! Contains base64 passwords!
```

You **can't** commit Kubernetes Secret manifests to Git because anyone with repo access can read the passwords.

But if you **don't** commit secrets to Git, then your Git repo is not the complete source of truth — you have to manage secrets separately, outside of Git, which breaks the GitOps model.

### How Sealed Secrets Fixes This:

Sealed Secrets gives you a **new custom resource** called `SealedSecret`. It's an **encrypted version** of a Kubernetes Secret that is **safe to store in Git**.

```
The Flow:
                                                    
  Developer creates a Secret  →  kubeseal encrypts it  →  SealedSecret (safe for Git!)
                                                                    ↓
                                                           Push to Git repo
                                                                    ↓
                                                    GitOps tool (ArgoCD/Flux) applies it
                                                                    ↓
                                                    Sealed Secrets controller decrypts it
                                                                    ↓
                                                    Regular K8s Secret is created in cluster
```

Only the **controller running inside the cluster** has the private key to decrypt. Nobody else can decrypt it — not even the person who encrypted it!

## 🏗️ Architecture — How It Works

Sealed Secrets has two components:

### 1. The Controller (Server-side)

- Runs as a **Deployment** in your Kubernetes cluster (usually in the `kube-system` namespace)
- Generates a **public/private key pair** (RSA 4096-bit)
- Watches for `SealedSecret` resources
- When it sees one, it **decrypts** it and creates a regular `Secret`
- The **private key never leaves the cluster**

### 2. `kubeseal` CLI (Client-side)

- A command-line tool you install on your laptop
- Fetches the **public key** from the controller (or you provide it)
- Encrypts regular `Secret` manifests into `SealedSecret` resources
- **Cannot decrypt** — it only has the public key

### The Cryptography (Simplified):

```
                        PUBLIC KEY (everyone has this)
                              ↓
Plain Secret  →  kubeseal encrypts  →  SealedSecret
                                            ↓
                                     Stored in Git
                                            ↓
                                    Applied to cluster
                                            ↓
                        PRIVATE KEY (only controller has this)
                              ↓
                    Controller decrypts  →  Regular K8s Secret
```

This is **asymmetric encryption** (like SSH keys):
- The **public key** encrypts (anyone can do this)
- The **private key** decrypts (only the controller can do this)

## 📦 Installation

### Step 1: Install the Controller in Your Cluster

```bash
# Using kubectl
kubectl apply -f https://github.com/bitnami-labs/sealed-secrets/releases/download/v0.27.3/controller.yaml
```

Or using **Helm** (recommended for production):

```bash
helm repo add sealed-secrets https://bitnami-labs.github.io/sealed-secrets
helm repo update

helm install sealed-secrets sealed-secrets/sealed-secrets \
  --namespace kube-system \
  --set fullnameOverride=sealed-secrets-controller
```

### Verify the controller is running:

```bash
kubectl get pods -n kube-system -l app.kubernetes.io/name=sealed-secrets

# NAME                                          READY   STATUS    RESTARTS   AGE
# sealed-secrets-controller-5b4c7f8b9d-x2k7j   1/1     Running   0          1m
```

### Step 2: Install `kubeseal` CLI

```bash
# macOS
brew install kubeseal

# Linux
KUBESEAL_VERSION=0.27.3
curl -OL "https://github.com/bitnami-labs/sealed-secrets/releases/download/v${KUBESEAL_VERSION}/kubeseal-${KUBESEAL_VERSION}-linux-amd64.tar.gz"
tar -xvzf kubeseal-${KUBESEAL_VERSION}-linux-amd64.tar.gz kubeseal
sudo install -m 755 kubeseal /usr/local/bin/kubeseal

# Windows (via Chocolatey)
choco install kubeseal
```

### Verify:

```bash
kubeseal --version
# kubeseal version: 0.27.3
```

## 🎯 Using Sealed Secrets — Step-by-Step Tutorial

### Step 1: Create a Regular Kubernetes Secret (Don't Apply It!)

First, create a normal Secret manifest — but **don't apply it to the cluster**. This is just the input for `kubeseal`:

```bash
# Method 1: Create from the command line
kubectl create secret generic my-database-creds \
  --from-literal=username=admin \
  --from-literal=password=super-secret-password-123 \
  --dry-run=client \
  -o yaml > my-secret.yaml
```

This generates:

```yaml
# my-secret.yaml
apiVersion: v1
kind: Secret
metadata:
  name: my-database-creds
  namespace: default
type: Opaque
data:
  username: YWRtaW4=
  password: c3VwZXItc2VjcmV0LXBhc3N3b3JkLTEyMw==
```

### Step 2: Seal (Encrypt) the Secret

```bash
kubeseal --format yaml < my-secret.yaml > my-sealed-secret.yaml
```

**What happened:**
- `kubeseal` contacted the Sealed Secrets controller in your cluster
- Fetched the public certificate
- Encrypted the secret data
- Created a `SealedSecret` resource

The output looks like this:

```yaml
# my-sealed-secret.yaml
apiVersion: bitnami.com/v1alpha1
kind: SealedSecret
metadata:
  name: my-database-creds
  namespace: default
spec:
  encryptedData:
    username: AgBf2k3j8Kx9mN4pQ7rS8tUvWxYz... (long encrypted string)
    password: AgCd5e6f7Gh8iJ9kL0mN1oP2qR3sT... (long encrypted string)
  template:
    metadata:
      name: my-database-creds
      namespace: default
    type: Opaque
```

**This file is safe to commit to Git!** 🎉

### Step 3: Delete the Plain Secret File!

```bash
rm my-secret.yaml
```

Never keep the unencrypted version around. Only the SealedSecret should exist in your repo.

### Step 4: Commit and Push the SealedSecret

```bash
git add my-sealed-secret.yaml
git commit -m "Add encrypted database credentials"
git push
```

### Step 5: Apply the SealedSecret to the Cluster

```bash
kubectl apply -f my-sealed-secret.yaml
```

Or if you're using GitOps (ArgoCD/Flux), it will automatically apply when it detects the new file in Git.

### Step 6: Verify — The Controller Creates the Real Secret

```bash
# Check the SealedSecret
kubectl get sealedsecret my-database-creds
# NAME                 STATUS   SYNCED   AGE
# my-database-creds              True     30s

# Check the real Secret (auto-created by the controller)
kubectl get secret my-database-creds
# NAME                 TYPE     DATA   AGE
# my-database-creds    Opaque   2      30s

# Verify the values
kubectl get secret my-database-creds -o jsonpath='{.data.password}' | base64 -d
# super-secret-password-123  ✅
```

The controller saw the `SealedSecret`, decrypted it, and created a regular `Secret` that your Pods can use normally!

## 📋 Scopes — Controlling Where Secrets Can Be Used

Sealed Secrets has three **scopes** that control how tightly the encrypted secret is bound:

### 1. `strict` (Default) — Bound to Name AND Namespace

```bash
kubeseal --scope strict --format yaml < secret.yaml > sealed-secret.yaml
```

- The SealedSecret can **only** create a Secret with the **exact same name** in the **exact same namespace**
- Most secure option
- If you rename the Secret or move it to another namespace, decryption **fails**

### 2. `namespace-wide` — Bound to Namespace Only

```bash
kubeseal --scope namespace-wide --format yaml < secret.yaml > sealed-secret.yaml
```

- The SealedSecret can create a Secret with **any name** but only in the **same namespace**
- Useful when you want to rename the Secret later

### 3. `cluster-wide` — Not Bound to Anything

```bash
kubeseal --scope cluster-wide --format yaml < secret.yaml > sealed-secret.yaml
```

- The SealedSecret can be used in **any namespace** with **any name**
- Least secure — use sparingly
- Useful for shared secrets across namespaces

### Which Scope Should You Use?

| Scope | Security | Flexibility | Best For |
|-------|----------|------------|----------|
| `strict` | 🔒 Highest | Low | Production secrets (recommended default) |
| `namespace-wide` | 🔒 Medium | Medium | Secrets that might be renamed |
| `cluster-wide` | ⚠️ Lowest | High | Shared secrets across namespaces |

## 🔄 Updating a Sealed Secret

Need to change a password? Here's how:

### Method 1: Re-create and Re-seal

```bash
# Create a new secret with updated values
kubectl create secret generic my-database-creds \
  --from-literal=username=admin \
  --from-literal=password=new-even-more-secret-password \
  --dry-run=client -o yaml | kubeseal --format yaml > my-sealed-secret.yaml

# Commit and push
git add my-sealed-secret.yaml
git commit -m "Rotate database password"
git push

# Apply (or let GitOps handle it)
kubectl apply -f my-sealed-secret.yaml
```

### Method 2: Merge a Single Value (Add/Update One Key)

```bash
# Create a secret with just the field you want to update
echo -n "new-password-value" | kubectl create secret generic my-database-creds \
  --from-file=password=/dev/stdin \
  --dry-run=client -o yaml | \
  kubeseal --format yaml --merge-into my-sealed-secret.yaml
```

The `--merge-into` flag is super useful — it **adds or replaces** a single encrypted field without touching the others.

## 🔐 Offline Sealing (Without Cluster Access)

Sometimes you want to seal secrets on a machine that doesn't have access to the cluster (like a developer's laptop). You can do this by saving the controller's public certificate:

### Step 1: Fetch the Certificate (On a Machine With Cluster Access)

```bash
kubeseal --fetch-cert > sealed-secrets-cert.pem
```

### Step 2: Share the Certificate

Distribute `sealed-secrets-cert.pem` to your team. This is a **public** certificate — it's safe to share.

You can even commit it to your Git repo:

```bash
git add sealed-secrets-cert.pem
git commit -m "Add Sealed Secrets public certificate"
```

### Step 3: Seal Secrets Offline

```bash
kubeseal --cert sealed-secrets-cert.pem --format yaml < my-secret.yaml > my-sealed-secret.yaml
```

No cluster connection needed! 🎉

## 🔄 Key Management & Rotation

### How the Controller Manages Keys:

- The controller generates a **new key pair every 30 days** by default
- **Old keys are kept** — the controller can still decrypt secrets sealed with old keys
- The **latest key** is used for new encryptions
- All keys are stored as Kubernetes Secrets in the `kube-system` namespace

### Viewing Keys:

```bash
kubectl get secrets -n kube-system -l sealedsecrets.bitnami.com/sealed-secrets-key
```

### Backup the Private Keys!

This is **critical**. If you lose the private keys (e.g., cluster is destroyed), you **cannot decrypt any SealedSecrets**.

```bash
# Backup all sealing keys
kubectl get secret -n kube-system -l sealedsecrets.bitnami.com/sealed-secrets-key \
  -o yaml > sealed-secrets-keys-backup.yaml

# Store this backup securely (e.g., in a password manager or a cloud KMS-encrypted bucket)
# NEVER commit this to Git!
```

### Restore Keys (After Cluster Recreation):

```bash
# Apply the backup before installing the controller
kubectl apply -f sealed-secrets-keys-backup.yaml

# Then install the controller — it will pick up the existing keys
helm install sealed-secrets sealed-secrets/sealed-secrets \
  --namespace kube-system
```

### Manual Key Rotation:

```bash
# Generate a new key (the controller does this automatically every 30 days)
# But you can force it by restarting the controller:
kubectl delete pod -n kube-system -l app.kubernetes.io/name=sealed-secrets

# The controller generates a new key on startup
# Old keys are preserved for backward compatibility
```

### Re-encrypt Existing SealedSecrets with New Keys:

After key rotation, you might want to re-encrypt existing SealedSecrets to use the latest key:

```bash
# Fetch the latest certificate
kubeseal --fetch-cert > new-cert.pem

# For each SealedSecret, decrypt the original Secret and re-seal it
# (You need the original plain secrets or to read them from the cluster)

# Read the current secret from the cluster
kubectl get secret my-database-creds -o yaml | \
  kubeseal --format yaml > my-sealed-secret-rotated.yaml
```

## 🔄 Sealed Secrets with ArgoCD (GitOps)

The most common use case for Sealed Secrets is with **ArgoCD** for GitOps:

### Repository Structure:

```
my-app/
├── base/
│   ├── deployment.yaml
│   ├── service.yaml
│   └── kustomization.yaml
└── overlays/
    ├── staging/
    │   ├── sealed-secret.yaml    # Encrypted! Safe in Git ✅
    │   └── kustomization.yaml
    └── production/
        ├── sealed-secret.yaml    # Encrypted! Safe in Git ✅
        └── kustomization.yaml
```

### Kustomization Example:

```yaml
# overlays/production/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - ../../base
  - sealed-secret.yaml
```

### ArgoCD Application:

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: my-app
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/myorg/my-app.git
    targetRevision: main
    path: overlays/production
  destination:
    server: https://kubernetes.default.svc
    namespace: production
```

**Flow:**
1. Developer creates a Secret → seals it with `kubeseal` → pushes SealedSecret to Git
2. ArgoCD detects the change and applies the `SealedSecret` to the cluster
3. Sealed Secrets controller decrypts it and creates the real `Secret`
4. Pods use the `Secret` normally

**The entire workflow stays in Git!** Full GitOps with encrypted secrets. 🎉

## ⚠️ Common Pitfalls & Troubleshooting

### 1. "Error: cannot fetch certificate"

```bash
# Problem: kubeseal can't connect to the controller
kubeseal --fetch-cert
# error: cannot fetch certificate: services "sealed-secrets-controller" not found

# Fix: Check controller name and namespace
kubectl get svc -n kube-system | grep sealed

# If the service name is different, specify it:
kubeseal --controller-name sealed-secrets --controller-namespace kube-system
```

### 2. "SealedSecret is not syncing"

```bash
# Check the controller logs
kubectl logs -n kube-system -l app.kubernetes.io/name=sealed-secrets

# Common causes:
# - SealedSecret sealed for a different cluster (different key pair)
# - Namespace mismatch (strict scope)
# - Name mismatch (strict scope)
```

### 3. "Sealed Secret was encrypted with a different key"

This happens when you recreate the cluster without restoring the backup keys.

```bash
# Solution 1: Restore the key backup (if you have it)
kubectl apply -f sealed-secrets-keys-backup.yaml
kubectl delete pod -n kube-system -l app.kubernetes.io/name=sealed-secrets

# Solution 2: Re-seal all secrets with the new key
kubeseal --fetch-cert > new-cert.pem
# Then re-seal all your secrets
```

### 4. Sealed Secret Created But No Secret Appears

```bash
# Check events on the SealedSecret
kubectl describe sealedsecret my-database-creds

# Look for errors like:
# "no key could decrypt secret"
# "namespace mismatch"
```

## ✅ Best Practices

| Practice | Why |
|----------|-----|
| **Always backup sealing keys** | If lost, you can't decrypt any SealedSecrets |
| **Use `strict` scope by default** | Most secure; prevents secrets from being moved |
| **Delete plain Secret files immediately** | Only SealedSecrets should be in your repo |
| **One Secret per SealedSecret** | Easier to manage and rotate |
| **Rotate secrets, not just keys** | Key rotation protects future secrets; rotate actual passwords too |
| **Use different clusters for different environments** | Each cluster has its own key pair = natural isolation |
| **Add a Makefile or script for sealing** | Makes it easy for the team to create SealedSecrets consistently |
| **Monitor controller logs** | Catch decryption failures early |

### Handy Makefile:

```makefile
# Makefile
KUBESEAL_CERT ?= sealed-secrets-cert.pem

.PHONY: fetch-cert seal-secret

fetch-cert:
	kubeseal --fetch-cert > $(KUBESEAL_CERT)

seal-secret:
	@read -p "Secret name: " name; \
	read -p "Namespace: " ns; \
	read -p "Key: " key; \
	read -sp "Value: " value; echo; \
	kubectl create secret generic $$name \
		--namespace $$ns \
		--from-literal=$$key=$$value \
		--dry-run=client -o yaml | \
	kubeseal --cert $(KUBESEAL_CERT) --format yaml > sealed-$$name.yaml; \
	echo "Created sealed-$$name.yaml"
```

## 🆚 Sealed Secrets vs SOPS vs External Secrets

| Feature | Sealed Secrets | SOPS | External Secrets |
|---------|---------------|------|-----------------|
| **Approach** | Encrypt K8s Secrets for Git | Encrypt any config file for Git | Sync from external stores |
| **K8s Required?** | ✅ Yes | ❌ No | ✅ Yes |
| **Encryption Key** | Managed by controller in cluster | Managed externally (KMS, age, PGP) | N/A (reads from stores) |
| **Supported Formats** | K8s Secrets only | YAML, JSON, ENV, INI | K8s Secrets only |
| **Ease of Setup** | Easy | Easy | Medium |
| **Key Backup Needed?** | ✅ Critical | Depends on backend | No (keys in external store) |
| **Multi-cluster** | One key per cluster | One key for all | One operator per cluster |
| **Best For** | K8s-only GitOps | Any project, any platform | Multi-cloud, external secret stores |

## 📝 Summary

| Concept | Key Point |
|---------|-----------|
| **What** | A Kubernetes controller that decrypts `SealedSecret` resources into regular `Secrets` |
| **Why** | Safely store encrypted Kubernetes Secrets in Git for GitOps workflows |
| **How** | `kubeseal` encrypts with a public key; only the in-cluster controller has the private key |
| **Scopes** | `strict` (name+namespace), `namespace-wide`, `cluster-wide` |
| **Key Feature** | Asymmetric encryption — encrypters can't decrypt; only the cluster can |
| **Critical** | Always backup the controller's private keys! |
| **GitOps** | Perfect companion for ArgoCD or Flux |
| **Best Practice** | Use `strict` scope, backup keys, delete plain secrets, one secret per SealedSecret |

> **Bottom Line:** Sealed Secrets is the go-to solution for storing Kubernetes secrets in Git. It's simple, Kubernetes-native, and works perfectly with GitOps tools like ArgoCD and Flux. Just remember to backup those private keys! 🔐
