## 🤔 Why Backups Matter (The Horror Stories)

Before we get into the technical details, let's understand WHY backups are so important — with real stories:

```
🔥 GitLab (2017):
   An engineer accidentally ran "rm -rf" on the PRODUCTION database.
   5 out of 5 backup methods had FAILED silently.
   They lost 6 hours of data.
   Lesson: Test your backups. Verify they actually work.

🔥 Toy Story 2 (1998):
   Someone ran "rm -rf /" on the Pixar server.
   90% of the movie was deleted.
   Backups? Failed. But a director had a copy at home.
   Lesson: Off-site backups saved a $200 million movie.

🔥 MySpace (2019):
   Lost 12 YEARS of user-uploaded music (50 million songs)
   during a server migration.
   Lesson: Migration ≠ Backup. Always have a backup BEFORE you migrate.
```

### The Backup Golden Rule:

> **If your data doesn't exist in at least 3 places, it doesn't exist.**

## 📊 RTO and RPO — The Two Most Important Metrics

These two metrics drive ALL backup and recovery decisions. Understanding them is absolutely essential.

### RPO — Recovery Point Objective

> **RPO = How much data can you afford to LOSE?**

RPO answers the question: **"If disaster strikes RIGHT NOW, how far back in time can we go to restore data?"**

```
                    Last Backup         Disaster
                        │                  │
  ──────────────────────┼──────────────────┼────────
          Data saved    │    Data LOST     │
          (recovered)   │    (gone!)       │
                        │                  │
                        │←─── RPO ────────→│
                        
  RPO = The maximum acceptable gap between your 
        last backup and the disaster.
```

### RPO Examples:

| RPO | Meaning | Example |
|-----|---------|---------|
| **RPO = 0** | Zero data loss | Financial transactions, banking |
| **RPO = 1 minute** | Lose at most 1 minute of data | E-commerce orders |
| **RPO = 1 hour** | Lose at most 1 hour of data | User activity logs |
| **RPO = 24 hours** | Lose at most 1 day of data | Internal wiki, documentation |
| **RPO = 1 week** | Lose at most 1 week of data | Archived logs |

### How to Achieve Different RPOs:

| RPO Target | Backup Method | Cost |
|------------|---------------|------|
| 0 (zero loss) | Synchronous replication | $$$$ |
| Seconds | Asynchronous replication (streaming) | $$$ |
| Minutes | WAL (Write-Ahead Log) shipping | $$ |
| Hours | Periodic automated snapshots | $ |
| Days | Daily full backups | $ |

---

### RTO — Recovery Time Objective

> **RTO = How FAST must you recover?**

RTO answers the question: **"After a disaster, how long can we be down before it's unacceptable?"**

```
        Disaster        Recovery Complete
           │                   │
  ─────────┼───────────────────┼────────────────
           │   System is DOWN  │   Back Online!
           │                   │
           │←───── RTO ───────→│
           
  RTO = The maximum acceptable time from disaster 
        to full recovery.
```

### RTO Examples:

| RTO | Meaning | Example |
|-----|---------|---------|
| **RTO = 0** | Zero downtime | Stock trading platform |
| **RTO = 5 min** | Back up in 5 minutes | Payment processing |
| **RTO = 1 hour** | Back up within an hour | E-commerce website |
| **RTO = 4 hours** | Back up within 4 hours | Internal business apps |
| **RTO = 24 hours** | Back up within a day | Non-critical internal tools |
| **RTO = 72 hours** | Back up within 3 days | Archival systems |

### How to Achieve Different RTOs:

| RTO Target | Recovery Method | Cost |
|------------|----------------|------|
| 0 (zero downtime) | Active-Active multi-region | $$$$ |
| Minutes | Warm standby with auto-failover | $$$ |
| 10-60 min | Pilot light with scripted recovery | $$ |
| Hours | Restore from snapshots | $ |
| Days | Restore from cold backups (tape/glacier) | $ |

### RTO and RPO Together — The Big Picture

```
┌──────────────────────────────────────────────────────────┐
│                                                          │
│    Past ◄──────────────────────────────────► Future       │
│                                                          │
│    ◄──── RPO ────►           ◄──── RTO ────►            │
│    (Data you lose)           (Time to recover)           │
│                                                          │
│    Last good      Disaster    Recovery                   │
│    backup         happens     complete                   │
│      │               │           │                       │
│  ────┼───────────────┼───────────┼──────────────         │
│      │  Lost data    │  Downtime │                       │
│      │               │           │                       │
│                                                          │
│  GOAL: Minimize BOTH RPO and RTO                        │
│  REALITY: Lower RPO/RTO = Higher cost and complexity    │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### How to Set RTO and RPO:

```
Step 1: List all your systems/services

Step 2: For each service, answer:
  - How critical is this to the business?
  - How much revenue is lost per hour of downtime?
  - How much data loss is acceptable?
  - What do compliance requirements say?

Step 3: Classify into tiers:

┌────────────────────────────────────────────────────────┐
│ Tier │ Criticality  │ RPO       │ RTO       │ Example │
├──────┼──────────────┼───────────┼───────────┼─────────┤
│  1   │ Mission-     │ 0 - 1 min │ 0 - 5 min │ Payment │
│      │ Critical     │           │           │ system  │
│  2   │ Business-    │ 1 - 60    │ 5 - 60   │ E-comm  │
│      │ Critical     │ min       │ min       │ website │
│  3   │ Important    │ 1 - 24   │ 1 - 4    │ Internal│
│      │              │ hours     │ hours     │ tools   │
│  4   │ Non-         │ 1 - 7    │ 24 - 72  │ Dev/    │
│      │ Critical     │ days      │ hours     │ staging │
└────────────────────────────────────────────────────────┘

Step 4: Choose backup and recovery strategies based on tier
```

## 🗂️ Types of Backups

### 1. Full Backup

A complete copy of ALL data.

```
Day 1: Full Backup (100 GB)    → Copy EVERYTHING
Day 2: Full Backup (102 GB)    → Copy EVERYTHING again
Day 3: Full Backup (105 GB)    → Copy EVERYTHING again

Pros:
  ✅ Simplest to restore (just restore the one backup)
  ✅ Complete copy of data
  ✅ Self-contained — doesn't depend on other backups

Cons:
  ❌ Slow to create (copies everything every time)
  ❌ Uses the most storage
  ❌ Takes the most network bandwidth
```

### 2. Incremental Backup

Only backs up data that **changed since the LAST backup** (full OR incremental).

```
Day 1: Full Backup      (100 GB)  → Copy everything
Day 2: Incremental       (2 GB)   → Only changes since Day 1
Day 3: Incremental       (3 GB)   → Only changes since Day 2
Day 4: Incremental       (1 GB)   → Only changes since Day 3
Day 5: Incremental       (4 GB)   → Only changes since Day 4

Total storage: 100 + 2 + 3 + 1 + 4 = 110 GB

To restore Day 5:
  Must restore: Full (Day 1) + Incr (Day 2) + Incr (Day 3) + 
                Incr (Day 4) + Incr (Day 5)
  → Need ALL pieces! If Day 3 is corrupt, you can only restore to Day 2.

Pros:
  ✅ Very fast backups (only copies changes)
  ✅ Uses minimal storage
  ✅ Low network bandwidth

Cons:
  ❌ Slow to restore (need the full + ALL incrementals in order)
  ❌ If ANY incremental is corrupt, you lose all subsequent data
  ❌ More complex to manage
```

### 3. Differential Backup

Only backs up data that **changed since the LAST FULL backup**.

```
Day 1: Full Backup       (100 GB)  → Copy everything
Day 2: Differential       (2 GB)   → Changes since Day 1 (full)
Day 3: Differential       (5 GB)   → Changes since Day 1 (full)
Day 4: Differential       (7 GB)   → Changes since Day 1 (full)
Day 5: Differential       (10 GB)  → Changes since Day 1 (full)

Notice: Each differential gets BIGGER because it includes ALL 
changes since the last FULL backup.

Total storage: 100 + 2 + 5 + 7 + 10 = 124 GB

To restore Day 5:
  Must restore: Full (Day 1) + Differential (Day 5)
  → Only need 2 pieces! Much simpler than incremental.

Pros:
  ✅ Faster restore than incremental (only need full + latest diff)
  ✅ If one differential is corrupt, others still work
  
Cons:
  ❌ Differential backups get larger over time
  ❌ Uses more storage than incremental
  ❌ Slower backups than incremental
```

### Comparison:

```
┌──────────────────────────────────────────────────────────┐
│                                                          │
│  Backup Type    │ Backup Speed │ Storage │ Restore Speed │
│  ───────────────┼──────────────┼─────────┼───────────────│
│  Full           │ Slowest      │ Most    │ Fastest       │
│  Differential   │ Medium       │ Medium  │ Medium        │
│  Incremental    │ Fastest      │ Least   │ Slowest       │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### The Typical Strategy — Combine Them:

```
Monday:    FULL backup         (everything)
Tuesday:   Incremental backup  (changes since Monday)
Wednesday: Incremental backup  (changes since Tuesday)
Thursday:  Incremental backup  (changes since Wednesday)
Friday:    Incremental backup  (changes since Thursday)
Saturday:  Incremental backup  (changes since Friday)
Sunday:    Incremental backup  (changes since Saturday)
Monday:    FULL backup         (start fresh!)
```

## 🗄️ The 3-2-1 Backup Rule

This is the **gold standard** for backup strategy:

```
┌──────────────────────────────────────────────────────┐
│                                                      │
│              The 3-2-1 Backup Rule                   │
│                                                      │
│   3  → Keep at least 3 copies of your data           │
│   2  → Store on at least 2 different types of media  │
│   1  → Keep at least 1 copy off-site                 │
│                                                      │
│   Example:                                           │
│   Copy 1: Production database (primary)              │
│   Copy 2: Backup on local SSD/NAS (different media)  │
│   Copy 3: Backup in cloud storage (off-site)         │
│                                                      │
└──────────────────────────────────────────────────────┘
```

### Modern 3-2-1-1-0 Rule:

Some organizations extend this:

```
3 — Three copies of data
2 — Two different storage types
1 — One copy off-site
1 — One copy offline (air-gapped, can't be hit by ransomware)
0 — Zero errors (verified backups with regular testing!)
```

## 🛠️ Backup Strategies by System Type

### Database Backups

#### PostgreSQL Backup Methods:

```bash
# Method 1: pg_dump (Logical Backup)
# Best for: Smaller databases, selective backup, cross-version restore

# Full database backup
pg_dump -h localhost -U postgres -d myapp -F c -f /backups/myapp_$(date +%Y%m%d_%H%M%S).dump

# Specific tables only
pg_dump -h localhost -U postgres -d myapp -t orders -t users -F c -f /backups/tables.dump

# All databases
pg_dumpall -h localhost -U postgres -f /backups/all_databases.sql

# Restore from pg_dump
pg_restore -h localhost -U postgres -d myapp -F c /backups/myapp_20250315.dump
```

```bash
# Method 2: pg_basebackup (Physical Backup)
# Best for: Large databases, faster backup/restore, PITR

# Take a physical backup
pg_basebackup -h localhost -U replicator \
  -D /backups/base_$(date +%Y%m%d) \
  -Ft -z -P \
  --wal-method=stream

# This creates a tar file of the entire data directory
# Combined with WAL archiving, allows Point-in-Time Recovery (PITR)
```

```bash
# Method 3: WAL Archiving (Continuous Archiving for PITR)
# Best for: Near-zero RPO, point-in-time recovery

# postgresql.conf settings
archive_mode = on
archive_command = 'aws s3 cp %p s3://my-wal-archive/%f'
wal_level = replica

# This continuously ships WAL files to S3
# Combined with a base backup, you can restore to ANY point in time
```

#### PostgreSQL Backup with pgBackRest (Industry Standard):

```bash
# pgBackRest is the most popular PostgreSQL backup tool
# It supports full, differential, and incremental backups

# Install pgBackRest
sudo apt install pgbackrest

# Configuration (/etc/pgbackrest/pgbackrest.conf)
[global]
repo1-type=s3
repo1-s3-bucket=my-pg-backups
repo1-s3-endpoint=s3.amazonaws.com
repo1-s3-region=us-east-1
repo1-path=/backups
repo1-retention-full=4          # Keep 4 full backups
repo1-retention-diff=14         # Keep 14 differential backups
repo1-cipher-type=aes-256-cbc   # Encrypt backups at rest!

[myapp]
pg1-path=/var/lib/postgresql/16/main

# Take a full backup
pgbackrest --stanza=myapp --type=full backup

# Take a differential backup
pgbackrest --stanza=myapp --type=diff backup

# Take an incremental backup
pgbackrest --stanza=myapp --type=incr backup

# Restore (full restore)
pgbackrest --stanza=myapp restore

# Point-in-Time Recovery — restore to a specific time
pgbackrest --stanza=myapp \
  --type=time \
  --target="2025-03-15 14:30:00" \
  restore
```

#### MySQL Backup:

```bash
# Logical backup with mysqldump
mysqldump -u root -p --all-databases --single-transaction \
  --routines --triggers > /backups/all_databases_$(date +%Y%m%d).sql

# Restore
mysql -u root -p < /backups/all_databases_20250315.sql

# Physical backup with Percona XtraBackup (industry standard for MySQL)
# Full backup
xtrabackup --backup --target-dir=/backups/full

# Incremental backup
xtrabackup --backup --target-dir=/backups/incr1 \
  --incremental-basedir=/backups/full

# Prepare and restore
xtrabackup --prepare --target-dir=/backups/full
xtrabackup --copy-back --target-dir=/backups/full
```

#### MongoDB Backup:

```bash
# mongodump (logical backup)
mongodump --uri="mongodb://user:pass@localhost:27017" \
  --db=myapp \
  --out=/backups/mongo_$(date +%Y%m%d)

# mongorestore
mongorestore --uri="mongodb://user:pass@localhost:27017" \
  --db=myapp \
  /backups/mongo_20250315/myapp
```

#### Velero — The Standard Kubernetes Backup Tool:

```bash
# Install Velero CLI

  --backup-location-config region=us-east-1 \
  --snapshot-location-config region=us-east-1 \
  --secret-file ./credentials-velero
```

```bash
# Create a one-time backup of everything
velero backup create full-backup-$(date +%Y%m%d)

# Backup specific namespace
velero backup create payments-backup \
  --include-namespaces payments

# Backup specific resources
velero backup create deployments-backup \
  --include-resources deployments,services,configmaps

# Backup with label selector
velero backup create critical-backup \
  --selector tier=critical
```

```bash
# Schedule automatic backups
# Daily backup of everything, retain for 30 days
velero schedule create daily-full \
  --schedule="0 2 * * *" \
  --ttl 720h

# Hourly backup of critical namespace, retain for 48 hours
velero schedule create hourly-payments \
  --schedule="0 * * * *" \
  --include-namespaces payments \
  --ttl 48h
```

```bash
# Restore from backup
velero restore create --from-backup full-backup-20250315

# Restore specific namespace only
velero restore create --from-backup full-backup-20250315 \
  --include-namespaces payments

# Restore to a different namespace
velero restore create --from-backup full-backup-20250315 \
  --include-namespaces payments \
  --namespace-mappings payments:payments-restored

# Check restore status
velero restore describe payments-restore-20250315
```

```yaml
      - default
```

#### AWS Backup (Managed Service):

```hcl
# Terraform — AWS Backup Plan
resource "aws_backup_plan" "production" {
  name = "production-backup-plan"

  rule {
    rule_name         = "daily-backup"
    target_vault_name = aws_backup_vault.production.name

      delete_after = 30  # Retain for 30 days
    }

      destination_vault_arn = aws_backup_vault.dr_region.arn  # Cross-region copy!

        delete_after = 90  # DR copy retained for 90 days

    rule_name         = "weekly-backup"
    target_vault_name = aws_backup_vault.production.name
    schedule          = "cron(0 3 ? * SUN *)"  # Sundays at 3 AM
    
    lifecycle {
      cold_storage_after = 30   # Move to cold storage after 30 days
      delete_after       = 365  # Keep for 1 year
    }
  }
```

## ✅ Backup Verification — Don't Skip This!

### Why Verification is Critical:

```
A backup you haven't verified is NOT a backup.
It's a hope. And hope is not a strategy.

Common ways backups silently fail:
  ❌ Backup file is corrupted
  ❌ Backup is incomplete (missing tables/files)
  ❌ Backup password/encryption key is lost
  ❌ Backup restore process has changed
  ❌ Storage is full — new backups aren't created
  ❌ Permissions changed — backup can't read data
```

### Automated Backup Verification:

```bash
#!/bin/bash
# backup-verify.sh — Automated backup verification script

set -euo pipefail

BACKUP_FILE=$1
VERIFY_DB="verify_$(date +%Y%m%d_%H%M%S)"
SLACK_WEBHOOK="https://hooks.slack.com/services/YOUR/WEBHOOK/URL"

echo "=== Starting Backup Verification ==="
echo "Backup: $BACKUP_FILE"
echo "Verify DB: $VERIFY_DB"

# Step 1: Create a temporary database
echo "Creating temporary database..."
createdb -h localhost -U postgres "$VERIFY_DB"

# Step 2: Restore the backup into the temp database
echo "Restoring backup..."
pg_restore -h localhost -U postgres -d "$VERIFY_DB" "$BACKUP_FILE" 2>/dev/null

# Step 3: Run verification queries
echo "Running verification queries..."

# Check row counts
USERS_COUNT=$(psql -h localhost -U postgres -d "$VERIFY_DB" -t -c "SELECT COUNT(*) FROM users;")
ORDERS_COUNT=$(psql -h localhost -U postgres -d "$VERIFY_DB" -t -c "SELECT COUNT(*) FROM orders;")

echo "Users: $USERS_COUNT"
echo "Orders: $ORDERS_COUNT"

# Check data integrity
INTEGRITY=$(psql -h localhost -U postgres -d "$VERIFY_DB" -t -c "
  SELECT CASE 
    WHEN COUNT(*) > 0 THEN 'OK' 
    ELSE 'EMPTY' 
  END FROM users;
")

# Step 4: Clean up
echo "Dropping temporary database..."
dropdb -h localhost -U postgres "$VERIFY_DB"

# Step 5: Report results
if [ "$(echo $INTEGRITY | xargs)" == "OK" ] && [ "$USERS_COUNT" -gt 0 ]; then
  echo "✅ Backup verification PASSED"
  
  # Send success notification
  curl -s -X POST "$SLACK_WEBHOOK" \
    -H 'Content-type: application/json' \
    -d "{\"text\": \"✅ Backup verification PASSED\nBackup: $BACKUP_FILE\nUsers: $USERS_COUNT | Orders: $ORDERS_COUNT\"}"
else
  echo "❌ Backup verification FAILED"
  
  # Send failure notification (URGENT)
  curl -s -X POST "$SLACK_WEBHOOK" \
    -H 'Content-type: application/json' \
    -d "{\"text\": \"🚨 BACKUP VERIFICATION FAILED!\nBackup: $BACKUP_FILE\nCheck immediately!\"}"
  
  exit 1
fi
```

### Automate Verification with Cron:


## 🔐 Backup Security

### Encrypt Your Backups!

```bash
# Encrypt a backup with GPG
pg_dump -h localhost -U postgres myapp | \
  gpg --encrypt --recipient backup@company.com > /backups/myapp_encrypted.dump.gpg

# Decrypt and restore
gpg --decrypt /backups/myapp_encrypted.dump.gpg | \
  pg_restore -h localhost -U postgres -d myapp
```

```bash
# Encrypt with OpenSSL (simpler)
# Encrypt
pg_dump -h localhost -U postgres myapp -F c | \
  openssl enc -aes-256-cbc -salt -pbkdf2 -out /backups/myapp_encrypted.dump

# Decrypt and restore
openssl enc -d -aes-256-cbc -pbkdf2 -in /backups/myapp_encrypted.dump | \
  pg_restore -h localhost -U postgres -d myapp
```

### Backup Security Best Practices:

```
1. ENCRYPT backups at rest (AES-256)
2. ENCRYPT backups in transit (TLS/SSL)
3. RESTRICT access (only backup service account)
4. ROTATE encryption keys regularly
5. STORE keys separately from backups (use KMS)
6. IMMUTABLE backups (prevent ransomware from deleting them)
7. AIR-GAP one copy (offline, disconnected from network)
8. AUDIT backup access logs
```

### Immutable Backups (Ransomware Protection):

```hcl
# AWS S3 with Object Lock (prevents deletion/modification)
resource "aws_s3_bucket" "backup" {
  bucket = "my-immutable-backups"
  
  object_lock_configuration {
    object_lock_enabled = "Enabled"
  }
}

resource "aws_s3_bucket_object_lock_configuration" "backup" {
  bucket = aws_s3_bucket.backup.id

  rule {
    default_retention {
      mode = "COMPLIANCE"   # Even root user can't delete!
      days = 30
    }
  }
}
```

## 📊 Monitoring Your Backups

### What to Monitor:

| Metric | Alert If | Why |
|--------|----------|-----|
| Last successful backup age | > 2× backup interval | Backups might be failing |
| Backup size | Changes > 50% from previous | Data corruption or mass deletion |
| Backup duration | > 2× normal duration | Performance degradation |
| Available storage | < 20% remaining | Backups will stop when full |
| Restore test result | Failed | Your backups are useless |
| Replication lag | > RPO threshold | You might lose more data than acceptable |

### Prometheus Alerts for Backups:

```yaml
groups:
  - name: backup-alerts
    rules:
      # Alert if no successful backup in 25 hours (daily backup)
      - alert: BackupStale
        expr: |
          time() - backup_last_success_timestamp_seconds > 90000
        for: 10m
        labels:
          severity: critical
        annotations:
          summary: "No successful backup in over 25 hours"
          description: "Last backup for {{ $labels.database }} was {{ $value | humanizeDuration }} ago"

      # Alert if backup size dropped dramatically
      - alert: BackupSizeDrop
        expr: |
          backup_size_bytes / backup_size_bytes offset 1d < 0.5
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Backup size dropped by more than 50%"
          description: "Backup for {{ $labels.database }} is significantly smaller than yesterday"

      # Alert if backup verification failed
      - alert: BackupVerificationFailed
        expr: backup_verification_success == 0
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "Backup verification FAILED"
          description: "Backup restore test for {{ $labels.database }} failed. Backups may be corrupted!"
```

## 📚 Key Takeaways

```
✅ RPO = How much data you can afford to lose
✅ RTO = How fast you must recover
✅ Lower RPO/RTO = Higher cost and complexity
✅ Three backup types: Full, Incremental, Differential
✅ Follow the 3-2-1 rule: 3 copies, 2 media types, 1 off-site
✅ Full backups: Slowest backup, fastest restore
✅ Incremental backups: Fastest backup, slowest restore
✅ ALWAYS encrypt backups (AES-256)
✅ ALWAYS verify backups (automated restore tests)
✅ An unverified backup is NOT a backup
✅ Use immutable storage to protect against ransomware
✅ Use Velero for Kubernetes backups
✅ Use pgBackRest for PostgreSQL backups
✅ Monitor backup health (age, size, duration, verification)
✅ Test DR recovery quarterly (Game Days!)
```
