# 22.2 — Chaos Monkey — Netflix's Approach

> **One Line Answer:** Chaos Monkey is a tool created by Netflix that **randomly terminates servers/instances in production** to ensure that engineers build systems that can survive unexpected failures.

---

## 🤔 The Origin Story — Why Netflix Built Chaos Monkey

Let's go back to 2008. Netflix made one of the biggest decisions in their history — they decided to **migrate everything from their own data centers to Amazon Web Services (AWS)**.

### The Problem They Faced:

```
In their OWN data center:
  ✅ They controlled the hardware
  ✅ Servers rarely died unexpectedly
  ✅ They could physically walk to a machine and fix it
  ✅ Hardware was reliable (expensive, enterprise-grade)

In AWS (the cloud):
  ❌ Servers are virtual machines (VMs) — they CAN and WILL disappear
  ❌ AWS doesn't guarantee individual instance uptime
  ❌ Network issues between instances are common
  ❌ You don't control the underlying hardware
  ❌ If you depend on a SINGLE server, you're in trouble
```

Netflix engineers realized something important:

> **"In the cloud, failure is not a possibility — it's a certainty. The question isn't IF a server will die, but WHEN."**

So they asked themselves a brilliant question:

> **"What if we INTENTIONALLY kill servers during the day, so our engineers are FORCED to build systems that can survive it?"**

And that's how **Chaos Monkey** was born in **2010**.

---

## 🐒 What Exactly is Chaos Monkey?

Chaos Monkey is a software tool that:

```
1. Runs during business hours (NOT at 2 AM!)
2. Randomly picks a server/instance in production
3. Terminates it (kills it, shuts it down)
4. Engineers watch what happens
5. If something breaks → they fix it so it won't break next time
6. If nothing breaks → great, the system is resilient! ✅
```

### Why the Name "Chaos Monkey"?

> Imagine a **wild monkey** running loose in your data center, randomly pulling cables and pressing power buttons. That's essentially what Chaos Monkey does — but in a controlled, software-driven way.

### The Key Insight:

```
WITHOUT Chaos Monkey:
  Engineers: "This server probably won't die. Let's not worry about redundancy."
  Result: When the server DOES die → 💥 OUTAGE

WITH Chaos Monkey:
  Engineers: "Chaos Monkey WILL kill this server today. I MUST build redundancy."
  Result: When any server dies → System keeps running ✅
```

### The Process Flow:

```
1. Chaos Monkey runs on a schedule (e.g., every weekday between 9 AM - 3 PM)
2. It queries the cloud provider API to find all running instances
3. It filters instances based on configuration (opt-in/opt-out groups)
4. It randomly selects one instance from the eligible pool
5. It terminates that instance using the cloud provider API
6. It sends a notification (Slack, email, etc.) to the team
7. The team observes the impact on their monitoring dashboards
8. If the system self-healed → ✅ Resilient!
9. If something broke → 🐛 Fix it and make it resilient
```

## 🔧 Setting Up Chaos Monkey (Spinnaker-Based)

The modern version of Chaos Monkey is designed to work with **Spinnaker** (Netflix's open-source continuous delivery platform). Let's walk through the setup:

### Prerequisites:

```
✅ A cloud environment (AWS is the most common)
✅ Spinnaker installed and configured
✅ Applications deployed with auto-scaling groups
✅ Monitoring in place (you NEED to see what happens!)
```

### Installation:

Chaos Monkey is now part of Netflix's **Simian Army** and has been rewritten as a standalone application.

**Step 1: Clone the Repository**

```bash
# Clone the Chaos Monkey repository
git clone https://github.com/Netflix/chaosmonkey.git
cd chaosmonkey
```

**Step 2: Build Chaos Monkey**

```bash
# Chaos Monkey is written in Go
# Make sure Go is installed (1.17+)
go build -o chaosmonkey ./cmd/chaosmonkey
```

**Step 3: Configure Chaos Monkey**

Create a configuration file called `chaosmonkey.toml`:

```toml
# chaosmonkey.toml — Main Configuration File

# ─────────────────────────────────────────
# Spinnaker Configuration
# ─────────────────────────────────────────
[spinnaker]
# Spinnaker API endpoint
endpoint = "https://spinnaker.example.com"

# ─────────────────────────────────────────
# Chaos Monkey Behavior
# ─────────────────────────────────────────
[chaosmonkey]
# Enable or disable Chaos Monkey globally
enabled = true

# How often to run (schedule)
# "weekdays" = Monday through Friday
schedule_enabled = true

# Kill probability (1.0 = always kill, 0.5 = 50% chance)
# Start with a low probability!
mean_time_between_kills_in_work_days = 2
# ↑ This means on average, each group will lose an instance every 2 work days

# Time window for kills (24-hour format)
# Only kill during business hours when the team is awake!
start_hour = 9    # 9 AM
end_hour = 15     # 3 PM

# ─────────────────────────────────────────
# Cloud Provider
# ─────────────────────────────────────────
[cloud]
provider = "aws"

# ─────────────────────────────────────────
# Database (for tracking kills)
# ─────────────────────────────────────────
[database]
driver = "mysql"
host = "chaosmonkey-db.example.com"
port = 3306
name = "chaosmonkey"
user = "chaosmonkey"
# password should be passed via environment variable: CM_DATABASE_PASSWORD
```

Let's understand every configuration option:

```
enabled = true
  → Master switch. Set to false to disable ALL chaos experiments.

mean_time_between_kills_in_work_days = 2
  → Controls how frequently instances are killed.
  → 2 means each auto-scaling group will lose an instance every ~2 work days.
  → Start with a higher number (5-10) when you're new!

start_hour = 9, end_hour = 15
  → Chaos Monkey only kills instances between 9 AM and 3 PM.
  → WHY? So the team is awake and ready to respond!
  → Never run chaos at night when nobody's watching.

provider = "aws"
  → Which cloud provider to use.
  → Chaos Monkey uses the AWS API to terminate instances.
```

**Step 4: Run Chaos Monkey**

```bash
# Start Chaos Monkey
./chaosmonkey migrate   # Set up the database tables
./chaosmonkey schedule  # Schedule the next kill
./chaosmonkey terminate # Manually trigger a termination (for testing)
```

## 🦧 The Simian Army — Chaos Monkey's Family

Netflix didn't stop at Chaos Monkey. They built an entire **family of chaos tools** called the **Simian Army**:

| Tool | What It Does | What It Tests |
|------|-------------|---------------|
| **Chaos Monkey** | Kills random instances/VMs | Instance-level resilience |
| **Latency Monkey** | Injects artificial delays in network calls | Timeout handling, retry logic |
| **Conformity Monkey** | Finds instances that don't follow best practices | Compliance, standards |
| **Doctor Monkey** | Health checks on instances, removes unhealthy ones | Health monitoring |
| **Janitor Monkey** | Finds and deletes unused resources | Cost optimization, cleanup |
| **Security Monkey** | Finds security issues (open ports, bad configs) | Security posture |
| **10-18 Monkey** | Detects internationalization/localization issues | i18n/l10n compliance |
| **Chaos Gorilla** | Takes down an entire AWS Availability Zone | Multi-AZ resilience |
| **Chaos Kong** | Takes down an entire AWS Region | Multi-region resilience |

### The Hierarchy of Chaos:

```
Chaos Monkey   → Kills a single instance      (smallest blast radius)
Chaos Gorilla  → Kills an entire AZ            (medium blast radius)
Chaos Kong     → Kills an entire Region        (largest blast radius)

Most companies start with Chaos Monkey and never need Chaos Kong.
Only companies like Netflix that need multi-region resilience go that far.
```

## 💻 Building Your Own Simple Chaos Monkey

You don't need Netflix's exact tool. Let's build a simplified chaos monkey in different ways to understand the concept:

### Version 1: Simple Bash Script (for learning)

```bash
#!/bin/bash
# simple-chaos-monkey.sh
# ⚠️ FOR LEARNING ONLY — NOT FOR PRODUCTION USE!
# This script kills a random Docker container

echo "🐒 Simple Chaos Monkey Starting..."
echo "Time: $(date)"

# Get a list of all running containers (excluding essential ones)
CONTAINERS=$(docker ps --format "{{.Names}}" | grep -v "monitoring\|prometheus\|grafana")

# Count how many containers we have
COUNT=$(echo "$CONTAINERS" | wc -l)

if [ "$COUNT" -eq 0 ]; then
    echo "❌ No eligible containers found. Nothing to kill."
    exit 0
fi

# Pick a random container
RANDOM_CONTAINER=$(echo "$CONTAINERS" | shuf -n 1)

echo "🎯 Target selected: $RANDOM_CONTAINER"
echo "💀 Killing container: $RANDOM_CONTAINER"

# Kill the container
docker kill "$RANDOM_CONTAINER"

echo "✅ Container $RANDOM_CONTAINER has been killed!"
echo "📊 Check your monitoring dashboard to see the impact."
echo ""
echo "Remaining running containers:"
docker ps --format "table {{.Names}}\t{{.Status}}"
```

Let's understand every line:

```
docker ps --format "{{.Names}}"
  → Lists all running containers by name

grep -v "monitoring\|prometheus\|grafana"
  → Excludes monitoring containers (you NEVER want to kill your monitoring!)
  → The -v flag means "invert match" = exclude matching lines

shuf -n 1
  → Randomly selects one line from the input
  → This is how we pick a random container

docker kill "$RANDOM_CONTAINER"
  → Sends SIGKILL to the container (immediate termination)
  → This simulates a sudden crash (the container gets no warning)
```

### Version 2: Python Script for Kubernetes

```python
#!/usr/bin/env python3
"""
simple_chaos_monkey_k8s.py
A simple Chaos Monkey for Kubernetes environments.
⚠️ FOR LEARNING/STAGING ONLY — Add safety controls before using in production!
"""

import random
import time
import logging
from datetime import datetime
from kubernetes import client, config

# ──────────────────────────────────────────
# Configuration
# ──────────────────────────────────────────
NAMESPACE = "default"                    # Which namespace to target
EXCLUDED_LABELS = ["app=monitoring"]     # Pods with these labels won't be killed
DRY_RUN = True                          # Set to False to actually kill pods
KILL_INTERVAL_SECONDS = 300             # Time between kills (5 minutes)
MAX_KILLS_PER_RUN = 1                   # How many pods to kill per run

# ──────────────────────────────────────────
# Setup
# ──────────────────────────────────────────
logging.basicConfig(
    level=logging.INFO,
    format='%(asctime)s - %(levelname)s - %(message)s'
)
logger = logging.getLogger("chaos-monkey")


def get_eligible_pods(v1_api, namespace):
    """
    Get all pods that are eligible for termination.
    We exclude pods that match our exclusion labels.
    """
    # List all running pods in the namespace
    pods = v1_api.list_namespaced_pod(
        namespace=namespace,
        field_selector="status.phase=Running"  # Only target running pods
    )

    eligible_pods = []

    for pod in pods.items:
        pod_name = pod.metadata.name
        pod_labels = pod.metadata.labels or {}

        # Check if this pod should be excluded
        should_exclude = False
        for exclusion in EXCLUDED_LABELS:
            key, value = exclusion.split("=")
            if pod_labels.get(key) == value:
                should_exclude = True
                logger.info(f"  Excluding pod: {pod_name} (matches exclusion: {exclusion})")
                break

        if not should_exclude:
            eligible_pods.append(pod)

    return eligible_pods


def kill_pod(v1_api, pod, namespace, dry_run=True):
    """
    Kill (delete) a specific pod.
    If the pod is managed by a Deployment/ReplicaSet,
    Kubernetes will automatically create a new one!
    """
    pod_name = pod.metadata.name

    if dry_run:
        logger.info(f"  🔍 [DRY RUN] Would kill pod: {pod_name}")
        return True

    try:
        # Delete the pod
        # grace_period_seconds=0 means immediate termination (like a crash)
        v1_api.delete_namespaced_pod(
            name=pod_name,
            namespace=namespace,
            grace_period_seconds=0  # Immediate kill, no graceful shutdown
        )
        logger.info(f"  💀 Killed pod: {pod_name}")
        return True
    except client.exceptions.ApiException as e:
        logger.error(f"  ❌ Failed to kill pod {pod_name}: {e}")
        return False


def run_chaos_monkey():
    """
    Main chaos monkey loop.
    """
    logger.info("🐒 Chaos Monkey for Kubernetes Starting!")
    logger.info(f"   Namespace: {NAMESPACE}")
    logger.info(f"   Dry Run: {DRY_RUN}")
    logger.info(f"   Kill Interval: {KILL_INTERVAL_SECONDS}s")
    logger.info("")

    # Load Kubernetes configuration
    # In-cluster: uses the service account token
    # Out-of-cluster: uses ~/.kube/config
    try:
        config.load_incluster_config()    # Try in-cluster first
        logger.info("   Using in-cluster Kubernetes config")
    except config.ConfigException:
        config.load_kube_config()          # Fall back to kubeconfig
        logger.info("   Using local kubeconfig")

    # Create the Kubernetes API client
    v1_api = client.CoreV1Api()

    # ── Main Loop ──
    while True:
        logger.info(f"\n🔄 Chaos Monkey Run — {datetime.now().strftime('%Y-%m-%d %H:%M:%S')}")

        # Step 1: Get eligible pods
        eligible_pods = get_eligible_pods(v1_api, NAMESPACE)
        logger.info(f"   Found {len(eligible_pods)} eligible pods")

        if not eligible_pods:
            logger.info("   No eligible pods found. Skipping this run.")
        else:
            # Step 2: Randomly select pod(s) to kill
            targets = random.sample(
                eligible_pods,
                min(MAX_KILLS_PER_RUN, len(eligible_pods))
            )

            # Step 3: Kill the selected pods
            for target in targets:
                logger.info(f"\n   🎯 Target: {target.metadata.name}")
                logger.info(f"      Labels: {target.metadata.labels}")
                logger.info(f"      Node: {target.spec.node_name}")
                kill_pod(v1_api, target, NAMESPACE, dry_run=DRY_RUN)

        # Step 4: Wait before next run
        logger.info(f"\n   ⏰ Next run in {KILL_INTERVAL_SECONDS} seconds...")
        time.sleep(KILL_INTERVAL_SECONDS)


if __name__ == "__main__":
    run_chaos_monkey()
```

Let's break down every important part of this script:

```python
config.load_incluster_config()
# ↑ When running INSIDE a Kubernetes cluster (as a pod),
#   this loads the service account token automatically.
#   Kubernetes mounts the token at /var/run/secrets/kubernetes.io/serviceaccount/

config.load_kube_config()
# ↑ When running OUTSIDE the cluster (your laptop),
#   this loads ~/.kube/config (your kubectl config)

v1_api.list_namespaced_pod(namespace=namespace, field_selector="status.phase=Running")
# ↑ Lists all pods in the given namespace that are currently running.
#   We don't want to kill pods that are already terminating or pending.

v1_api.delete_namespaced_pod(name=pod_name, namespace=namespace, grace_period_seconds=0)
# ↑ Deletes (kills) the pod.
#   grace_period_seconds=0 means NO graceful shutdown — instant kill.
#   This simulates a sudden crash (OOMKill, node failure, etc.)
#   If we set grace_period_seconds=30, the pod gets 30 seconds to clean up.

random.sample(eligible_pods, min(MAX_KILLS_PER_RUN, len(eligible_pods)))
# ↑ Randomly picks MAX_KILLS_PER_RUN pods from the eligible list.
#   random.sample() picks without replacement (won't pick the same pod twice).
```

### Deploying as a Kubernetes CronJob:

You can deploy the chaos monkey as a CronJob that runs on a schedule:

```yaml
# chaos-monkey-cronjob.yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: chaos-monkey
  namespace: chaos-engineering    # Run in its own namespace
spec:
  schedule: "0 10 * * 1-5"       # Run at 10 AM, Monday through Friday
  # ↑ Cron syntax: minute hour day-of-month month day-of-week
  #   0     = minute 0
  #   10    = 10 AM
  #   *     = every day of month
  #   *     = every month
  #   1-5   = Monday through Friday (no weekends!)
  
  concurrencyPolicy: Forbid       # Don't run multiple chaos monkeys at once!
  successfulJobsHistoryLimit: 5   # Keep last 5 successful runs for debugging
  failedJobsHistoryLimit: 3       # Keep last 3 failed runs for debugging
  
  jobTemplate:
    spec:
      template:
        spec:
          serviceAccountName: chaos-monkey-sa    # Service account with pod delete permission
          containers:
          - name: chaos-monkey
            image: python:3.11-slim
            command: ["python", "/scripts/simple_chaos_monkey_k8s.py"]
            env:
            - name: NAMESPACE
              value: "default"
            - name: DRY_RUN
              value: "false"       # Set to "true" for testing!
            volumeMounts:
            - name: scripts
              mountPath: /scripts
          volumes:
          - name: scripts
            configMap:
              name: chaos-monkey-scripts
          restartPolicy: OnFailure     # Retry if the script crashes

---
# Service Account for Chaos Monkey
apiVersion: v1
kind: ServiceAccount
metadata:
  name: chaos-monkey-sa
  namespace: chaos-engineering

---
# RBAC: Give Chaos Monkey permission to list and delete pods
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: chaos-monkey-role
rules:
- apiGroups: [""]          # Core API group
  resources: ["pods"]      # Only pods — nothing else!
  verbs: ["list", "delete", "get"]   # Can list, get, and delete pods

---
# Bind the role to the service account
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: chaos-monkey-binding
subjects:
- kind: ServiceAccount
  name: chaos-monkey-sa
  namespace: chaos-engineering
roleRef:
  kind: ClusterRole
  name: chaos-monkey-role
  apiGroup: rbac.authorization.k8s.io
```

Let's explain the RBAC (Role-Based Access Control) part:

```yaml
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["list", "delete", "get"]

# This follows the "least privilege" principle:
# - Chaos Monkey can ONLY access pods (not secrets, services, etc.)
# - It can ONLY list, get, and delete (not create or update)
# - This limits what Chaos Monkey can do if something goes wrong
```

## 📊 What Happens When Chaos Monkey Kills Something?

Let's walk through a real scenario:


## ⚙️ Netflix's Production Practices

Here's how Netflix actually runs Chaos Monkey in their production environment:

### Netflix's Rules:

```
1. Chaos Monkey runs ONLY during business hours (Mon-Fri, 9 AM - 3 PM)
   → So engineers are awake and can respond

2. Every service is opted IN by default
   → If you deploy to Netflix, Chaos Monkey WILL kill your instances
   → You can opt out, but you have to justify why

3. Teams are expected to build for failure
   → If Chaos Monkey breaks your service, YOUR team fixes it
   → No blaming Chaos Monkey — it's YOUR responsibility to be resilient

4. Monitoring is mandatory
   → Every service must have dashboards and alerts
   → If you can't see the impact, you can't learn from it

5. Auto-scaling is expected
   → Services should auto-scale to replace killed instances
   → If you don't have auto-scaling, you're not ready for the cloud
```

### Netflix's Chaos Engineering Culture:

```
"We've found that the best way to avoid failure is to fail constantly."
    — Netflix Tech Blog

What this means:
  → By killing instances EVERY DAY, engineers are FORCED to build resilience
  → It becomes muscle memory — resilience is built into every service by default
  → When a REAL failure happens (AWS outage), Netflix barely notices
     because every service is already designed to survive instance failures
```

## 🧰 Alternative: Using AWS Fault Injection Simulator (FIS)

If you're on AWS, Amazon provides its own chaos engineering service called **FIS** (Fault Injection Simulator). It's a managed alternative to Chaos Monkey:

```
AWS FIS can:
  ✅ Terminate EC2 instances
  ✅ Throttle API calls
  ✅ Inject network latency
  ✅ Cause CPU/memory stress
  ✅ Simulate AZ failures
  ✅ Stop/start RDS databases

Benefits over Chaos Monkey:
  ✅ No infrastructure to manage (it's a managed service)
  ✅ Works with many AWS services (not just EC2)
  ✅ Built-in safety controls (stop conditions)
  ✅ IAM integration for access control
  ✅ Integrated with CloudWatch for monitoring
```

### Quick FIS Example (AWS CLI):

```bash
# Create an experiment template that stops an EC2 instance
aws fis create-experiment-template \
  --description "Kill a random EC2 instance" \
  --targets '{
    "myInstances": {
      "resourceType": "aws:ec2:instance",
      "selectionMode": "COUNT(1)",
      "resourceTags": {
        "chaos-eligible": "true"
      }
    }
  }' \
  --actions '{
    "stopInstances": {
      "actionId": "aws:ec2:stop-instances",
      "parameters": {},
      "targets": {
        "Instances": "myInstances"
      }
    }
  }' \
  --stop-conditions '[{
    "source": "aws:cloudwatch:alarm",
    "value": "arn:aws:cloudwatch:us-east-1:123456789:alarm:high-error-rate"
  }]' \
  --role-arn "arn:aws:iam::123456789:role/FISRole"

# ↑ Let's understand each part:
# targets: Which instances to affect (tagged with "chaos-eligible=true")
# selectionMode: "COUNT(1)" means pick 1 random instance
# actions: What to do (stop the instance)
# stop-conditions: Safety! If the "high-error-rate" alarm fires, STOP the experiment
# role-arn: IAM role that gives FIS permission to stop instances
```

## 📋 Best Practices for Running Chaos Monkey

```
✅ DO:
  1. Always run during business hours (when the team is awake)
  2. Start with non-critical services first
  3. Have monitoring dashboards open when running experiments
  4. Have a kill switch (ability to stop Chaos Monkey instantly)
  5. Notify the team before and after each kill
  6. Start in staging before moving to production
  7. Ensure services have multiple replicas
  8. Document every experiment and its results
  9. Use it as a learning tool, not a blame tool

❌ DON'T:
  1. Run Chaos Monkey at night or on weekends
  2. Kill instances without monitoring in place
  3. Target single-replica services (you'll just cause an outage)
  4. Run without telling anyone
  5. Kill monitoring/logging infrastructure
  6. Use it to "test" other teams without their knowledge
  7. Start with production on day one
```

## 📚 Key Takeaways

```
1. Chaos Monkey was created by Netflix in 2010 to test cloud resilience
2. It randomly kills server instances in production during business hours
3. The goal: FORCE engineers to build systems that survive failures
4. It's part of a larger family called the Simian Army
5. Modern alternatives: Litmus Chaos, Chaos Mesh, Gremlin, AWS FIS
6. Always start small: staging first, then production
7. You MUST have monitoring before running ANY chaos experiments
8. The culture matters more than the tool — embrace failure as learning
```
