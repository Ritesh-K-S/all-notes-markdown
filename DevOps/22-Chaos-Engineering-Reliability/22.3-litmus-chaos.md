# 22.3 — Litmus Chaos — Kubernetes Chaos Engineering

> **One Line Answer:** Litmus Chaos is an **open-source, Kubernetes-native chaos engineering platform** (part of CNCF) that lets you define, run, and manage chaos experiments on your Kubernetes clusters using **ChaosExperiments as code**.

---

## 🤔 Why Litmus Chaos? (What Problem Does It Solve?)

We already learned about Chaos Monkey — it kills cloud instances randomly. But here's the thing:

```
Chaos Monkey was built for the VM/cloud instance era.
Today, most modern applications run on KUBERNETES.

In Kubernetes:
  - Your apps run in PODS (not VMs)
  - You have Services, Deployments, StatefulSets
  - Networking is complex (CNI, Network Policies, Service Mesh)
  - Storage is abstracted (PV, PVC, StorageClass)
  - There's a control plane (API server, etcd, scheduler)

You need chaos tools that understand Kubernetes natively.
That's where Litmus Chaos comes in!
```

### What Makes Litmus Special:

| Feature | Why It Matters |
|---------|---------------|
| **Kubernetes-Native** | It runs AS Kubernetes resources (CRDs). It speaks Kubernetes. |
| **CNCF Project** | Part of the Cloud Native Computing Foundation (Incubating stage) — trusted by the community |
| **Declarative** | Define chaos experiments as YAML — just like your Kubernetes manifests |
| **ChaosHub** | A library of 50+ pre-built chaos experiments you can use right away |
| **Web UI (ChaosCenter)** | A beautiful dashboard to create, schedule, and monitor experiments |
| **GitOps Friendly** | Store your chaos experiments in Git, trigger them from CI/CD |
| **Observability Built-in** | Integrates with Prometheus, Grafana for monitoring experiments |
| **Multi-cluster** | Manage chaos experiments across multiple clusters from one place |

## 🧪 Core Concepts — CRDs (Custom Resource Definitions)

Litmus uses three main CRDs. Think of them as the building blocks of every chaos experiment:

### 1. ChaosExperiment

This defines **WHAT** chaos to inject and **HOW**.

```yaml
# This is like a "recipe" — it describes the chaos experiment
apiVersion: litmuschaos.io/v1alpha1
kind: ChaosExperiment
metadata:
  name: pod-delete                # Name of the experiment
  namespace: litmus
spec:
  definition:
    scope: Namespaced             # Scope: Namespaced or Cluster
    permissions:                  # What permissions does this experiment need?
      - apiGroups: [""]
        resources: ["pods"]
        verbs: ["delete", "list", "get"]
    image: "litmuschaos/go-runner:latest"    # The container that runs the chaos
    imagePullPolicy: Always
    args:
    - -c
    - ./experiments -name pod-delete         # Which experiment to run
    command:
    - /bin/bash
    env:
    # ─── Experiment-Specific Variables ───
    - name: TOTAL_CHAOS_DURATION
      value: '30'                 # How long to run the chaos (in seconds)
    - name: CHAOS_INTERVAL
      value: '10'                 # Interval between chaos actions (seconds)
    - name: FORCE
      value: 'true'              # Force delete (no grace period)
    labels:
      name: pod-delete
      app.kubernetes.io/part-of: litmus
```

```
Understanding the ChaosExperiment:

scope: Namespaced
  → The experiment affects resources in a specific namespace only.
  → "Cluster" scope would affect resources across all namespaces.

permissions:
  → What Kubernetes RBAC permissions the experiment needs.
  → pod-delete needs permission to delete pods (obviously!).

image: "litmuschaos/go-runner:latest"
  → The container image that contains the chaos logic.
  → Litmus provides pre-built images for each experiment type.

TOTAL_CHAOS_DURATION: '30'
  → How long the chaos experiment runs (30 seconds).
  → During this time, pods will be killed repeatedly.

CHAOS_INTERVAL: '10'
  → How often to perform the chaos action.
  → Every 10 seconds, a pod will be deleted.
  → So in 30 seconds, ~3 pods will be killed.

FORCE: 'true'
  → Force-delete pods (grace period = 0).
  → Simulates a sudden crash, not a graceful shutdown.
```

### 2. ChaosEngine

This defines **WHERE** to run the experiment and connects it to your application.

```yaml
# This is the "trigger" — it says "run THIS experiment on THAT application"
apiVersion: litmuschaos.io/v1alpha1
kind: ChaosEngine
metadata:
  name: order-service-chaos       # Name of this chaos engine
  namespace: default              # The namespace where your app lives
spec:
  # ─── Application Under Test ───
  appinfo:
    appns: 'default'              # Namespace of the target application
    applabel: 'app=order-service' # Label selector to find target pods
    appkind: 'deployment'         # Kind of workload (deployment, statefulset, etc.)
  
  # ─── Chaos Settings ───
  engineState: 'active'           # 'active' = run the experiment, 'stop' = don't run
  chaosServiceAccount: litmus-admin  # ServiceAccount with permissions to run chaos
  
  # ─── Monitoring ───
  monitoring: true                # Enable Prometheus metrics for this experiment
  
  # ─── Annotation Check ───
  annotationCheck: 'true'         # Only target pods that have the chaos annotation
  # ↑ This means pods MUST have the annotation:
  #   litmuschaos.io/chaos: "true"
  # This is a safety mechanism — only opted-in pods get chaos!
  
  # ─── Experiments to Run ───
  experiments:
  - name: pod-delete              # Reference the ChaosExperiment we defined above
    spec:
      components:
        env:
        # Override experiment defaults here
        - name: TOTAL_CHAOS_DURATION
          value: '60'             # Override: run for 60 seconds instead of 30
        - name: CHAOS_INTERVAL
          value: '15'             # Override: kill every 15 seconds
        - name: PODS_AFFECTED_PERC
          value: '50'             # Kill 50% of matching pods
      probe:                      # Health checks to validate steady state
      - name: "check-order-service-health"
        type: "httpProbe"
        httpProbe/inputs:
          url: "http://order-service.default.svc:8080/health"
          method:
            get:
              criteria: "=="
              responseCode: "200"
        mode: "Continuous"        # Check continuously during the experiment
        runProperties:
          probeTimeout: 5s
          interval: 5s
          retry: 3
```

Let's break down every important field:

```yaml
appinfo:
  applabel: 'app=order-service'
  # ↑ This is HOW Litmus finds your application pods.
  # It uses Kubernetes label selectors.
  # All pods with label "app=order-service" are potential targets.

annotationCheck: 'true'
  # ↑ SAFETY FEATURE!
  # When true, Litmus will ONLY target pods that have this annotation:
  #   litmuschaos.io/chaos: "true"
  # So you must explicitly opt-in each deployment.
  # This prevents accidentally chaos-testing the wrong pods.

PODS_AFFECTED_PERC: '50'
  # ↑ Blast radius control!
  # Instead of killing ALL matching pods, only kill 50%.
  # If you have 4 pods, only 2 will be killed.
  # Start with a low percentage and increase gradually.

probe:
  # ↑ PROBES are Litmus's way of checking if your system is healthy
  #   during and after the chaos experiment.
  # This HTTP probe hits /health every 5 seconds.
  # If it gets a 200 response, the system is still healthy.
  # If the probe FAILS, the experiment result = FAIL
  # (meaning your system couldn't handle the chaos).
```

### 3. ChaosResult

This is automatically created by Litmus after an experiment completes:

```yaml
# You don't create this — Litmus creates it automatically
apiVersion: litmuschaos.io/v1alpha1
kind: ChaosResult
metadata:
  name: order-service-chaos-pod-delete
  namespace: default
spec:
  engine: order-service-chaos
  experiment: pod-delete
status:
  experimentStatus:
    phase: Completed              # Current phase: Running, Completed, Stopped
    verdict: Pass                 # Result: Pass, Fail, or Awaited
    failStep: "N/A"              # If failed, which step failed
  probeSuccessPercentage: "100"  # What % of probes passed
  history:
    passedRuns: 3                # How many times this experiment passed
    failedRuns: 0                # How many times it failed
    stoppedRuns: 0               # How many times it was stopped manually
```

```
Understanding the result:

verdict: Pass
  → The system survived the chaos! Your resilience patterns work!
  → All probes passed during the experiment.

verdict: Fail
  → The system couldn't handle the chaos.
  → Some probes failed — meaning the service went down or degraded.
  → Action needed: fix the resilience issue and re-run.

probeSuccessPercentage: "100"
  → 100% of health checks passed during chaos.
  → If this was 60%, it means 40% of health checks failed
  → (the service was unhealthy 40% of the time during chaos).
```

## 🧪 Complete Hands-On: Your First Litmus Chaos Experiment

Let's walk through a complete example from start to finish.

### Step 1: Deploy a Sample Application

```yaml
# sample-app.yaml
# A simple nginx deployment that we'll use as our target
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-app
  namespace: default
  labels:
    app: nginx
  annotations:
    litmuschaos.io/chaos: "true"   # ← OPT-IN for chaos testing!
spec:
  replicas: 3                      # 3 replicas for redundancy
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
      annotations:
        litmuschaos.io/chaos: "true"  # ← Annotation on pods too!
    spec:
      containers:
      - name: nginx
        image: nginx:latest
        ports:
        - containerPort: 80
        readinessProbe:             # Health check — is the pod ready?
          httpGet:
            path: /
            port: 80
          initialDelaySeconds: 5
          periodSeconds: 5
        livenessProbe:              # Health check — is the pod alive?
          httpGet:
            path: /
            port: 80
          initialDelaySeconds: 10
          periodSeconds: 10
---
apiVersion: v1
kind: Service
metadata:
  name: nginx-service
  namespace: default
spec:
  selector:
    app: nginx
  ports:
  - port: 80
    targetPort: 80
  type: ClusterIP
```

### Step 2: Set Up the Chaos ServiceAccount & RBAC

```yaml
# chaos-rbac.yaml
# Litmus needs permissions to perform chaos operations
apiVersion: v1
kind: ServiceAccount
metadata:
  name: litmus-admin
  namespace: default
  labels:
    name: litmus-admin

---
# This role gives Litmus permission to:
# - List, get, delete pods (for pod-delete experiment)
# - List, get deployments (to understand the target)
# - Create events (to log what happened)
# - Manage ChaosEngine and ChaosResult resources
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: litmus-admin-role
  namespace: default
rules:
# Pod operations (needed for pod-delete experiment)
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["create", "delete", "get", "list", "patch", "update", "deletecollection"]

# Needed to check deployment status
- apiGroups: ["apps"]
  resources: ["deployments", "replicasets"]
  verbs: ["list", "get"]

# Event creation (for logging)
- apiGroups: [""]
  resources: ["events"]
  verbs: ["create", "get", "list", "patch", "update"]

# ChaosEngine and ChaosResult operations
- apiGroups: ["litmuschaos.io"]
  resources: ["chaosengines", "chaosresults", "chaosexperiments"]
  verbs: ["create", "list", "get", "patch", "update", "delete"]

# Needed to run the experiment pods
- apiGroups: ["batch"]
  resources: ["jobs"]
  verbs: ["create", "list", "get", "delete", "deletecollection"]

# Pod logs (for observability)
- apiGroups: [""]
  resources: ["pods/log"]
  verbs: ["get", "list"]

---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: litmus-admin-binding
  namespace: default
subjects:
- kind: ServiceAccount
  name: litmus-admin
  namespace: default
roleRef:
  kind: Role
  name: litmus-admin-role
  apiGroup: rbac.authorization.k8s.io
```

```bash
# Apply the RBAC configuration
kubectl apply -f chaos-rbac.yaml
```

### Example: Network Latency Experiment

```yaml
# network-latency-experiment.yaml
# Inject 2 seconds of network latency into the order-service
apiVersion: litmuschaos.io/v1alpha1
kind: ChaosEngine
metadata:
  name: order-service-network-chaos
  namespace: default
spec:
  appinfo:
    appns: 'default'
    applabel: 'app=order-service'
    appkind: 'deployment'
  engineState: 'active'
  chaosServiceAccount: litmus-admin
  experiments:
  - name: pod-network-latency
    spec:
      components:
        env:
        # How long to run the experiment
        - name: TOTAL_CHAOS_DURATION
          value: '60'             # 60 seconds of network chaos
        
        # How much latency to inject
        - name: NETWORK_LATENCY
          value: '2000'           # 2000ms = 2 seconds of delay!
        
        # Which direction to affect
        - name: NETWORK_INTERFACE
          value: 'eth0'           # The pod's network interface
        
        # Optional: Only affect traffic to specific destination
        - name: DESTINATION_IPS
          value: '10.96.0.1'     # Only slow down traffic to this IP
        
        # Jitter — randomness in the latency
        - name: JITTER
          value: '500'           # ±500ms jitter
          # ↑ This makes the latency more realistic.
          #   Real network issues have variable latency, not fixed.
          #   Actual latency will be 1500ms to 2500ms (2000 ± 500)
```

```
What this experiment tests:

BEFORE chaos:
  User → API Gateway → Order Service → Database
                        ↑ responds in 50ms (normal)

DURING chaos (2s latency injected):
  User → API Gateway → Order Service → Database
                        ↑ responds in 2050ms (50ms + 2000ms latency)

Questions to answer:
  ✅ Does the API Gateway have proper timeout settings?
  ✅ Does the user see a loading indicator (not a frozen page)?
  ✅ Does the circuit breaker kick in?
  ✅ Is there a fallback response for slow downstream services?
  ✅ Are retries configured properly (with backoff)?
```

```yaml
  experiments:
  - name: pod-cpu-hog
    spec:
      components:
        env:
        - name: TOTAL_CHAOS_DURATION
          value: '120'           # Run for 2 minutes
        
        - name: CPU_CORES
          value: '2'             # Consume 2 CPU cores
          # ↑ If your pod has a CPU limit of 2 cores,
          #   this will push it to 100% CPU usage!
        
        - name: PODS_AFFECTED_PERC
          value: '50'            # Affect 50% of pods
          # ↑ If you have 4 pods, 2 will experience CPU stress
```

```
What this experiment tests:

✅ Does HPA (Horizontal Pod Autoscaler) trigger when CPU goes high?
✅ Do CPU resource limits prevent the pod from affecting other pods?
✅ Does the service degrade gracefully under CPU pressure?
✅ Are requests properly load-balanced away from stressed pods?
✅ Do health checks (liveness/readiness probes) detect the stressed pod?
```

## 📊 Litmus Probes — Validating System Health

Probes are Litmus's way of checking if your system maintains its **steady state** during chaos. Think of them as automated health checks that run throughout the experiment.

### Types of Probes:

| Probe Type | What It Does | Example |
|-----------|-------------|---------|
| **httpProbe** | Makes HTTP requests and checks the response | Check if `/health` returns 200 |
| **cmdProbe** | Runs a shell command and checks the output | Run `curl` or custom script |
| **k8sProbe** | Checks Kubernetes resource status | Verify deployment has 3 ready replicas |
| **promProbe** | Queries Prometheus metrics | Check if error rate < 1% |

### Probe Modes:

```
SOT (Start of Test):   Run probe BEFORE chaos starts
                       → Verifies steady state before experiment

EOT (End of Test):     Run probe AFTER chaos ends
                       → Verifies system recovered after chaos

Edge:                  Run probe BEFORE and AFTER chaos
                       → Compares state before and after

Continuous:            Run probe THROUGHOUT the chaos
                       → Checks if system stays healthy during chaos
                       → Most thorough mode!
```

### Example: Using Probes in an Experiment

```yaml
apiVersion: litmuschaos.io/v1alpha1
kind: ChaosEngine
metadata:
  name: nginx-chaos-with-probes
  namespace: default
spec:
  appinfo:
    appns: 'default'
    applabel: 'app=nginx'
    appkind: 'deployment'
  engineState: 'active'
  chaosServiceAccount: litmus-admin
  experiments:
  - name: pod-delete
    spec:
      components:
        env:
        - name: TOTAL_CHAOS_DURATION
          value: '30'
      probe:
      # ─── Probe 1: HTTP Health Check ───
      - name: "nginx-health-check"
        type: "httpProbe"
        httpProbe/inputs:
          url: "http://nginx-service.default.svc:80/"
          method:
            get:
              criteria: "=="           # Response code should EQUAL 200
              responseCode: "200"
        mode: "Continuous"             # Check throughout the experiment
        runProperties:
          probeTimeout: 5s             # Timeout for each probe check
          interval: 3s                 # Check every 3 seconds
          retry: 2                     # Retry 2 times before marking as failed
          probePollingInterval: 2s     # Time between retries
      
      # ─── Probe 2: Kubernetes Deployment Check ───
      - name: "nginx-deployment-check"
        type: "k8sProbe"
        k8sProbe/inputs:
          group: "apps"
          version: "v1"
          resource: "deployments"
          namespace: "default"
          fieldSelector: "metadata.name=nginx-app"
          operation: "present"         # The deployment should exist
        mode: "EOT"                    # Check at end of test
        runProperties:
          probeTimeout: 5s
          interval: 2s
          retry: 3
      
      # ─── Probe 3: Prometheus Metrics Check ───
      - name: "error-rate-check"
        type: "promProbe"
        promProbe/inputs:
          endpoint: "http://prometheus.monitoring.svc:9090"
          query: "sum(rate(http_requests_total{status=~\"5..\"}[1m])) / sum(rate(http_requests_total[1m])) * 100"
          # ↑ This PromQL query calculates the error rate percentage
          comparator:
            type: "float"
            criteria: "<="            # Error rate should be <= 1%
            value: "1"
        mode: "Continuous"
        runProperties:
          probeTimeout: 10s
          interval: 10s
          retry: 2
```

```
How probes determine Pass/Fail:

ALL probes must pass → Experiment Result = Pass ✅
ANY probe fails     → Experiment Result = Fail ❌

Example:
  HTTP probe (Continuous): 20 checks, 20 passed → ✅
  K8s probe (EOT): Deployment exists and healthy → ✅
  Prom probe (Continuous): Error rate always < 1% → ✅
  Overall: PASS ✅

Example (failure):
  HTTP probe (Continuous): 20 checks, 15 passed, 5 failed → ❌
  K8s probe (EOT): Deployment exists and healthy → ✅
  Prom probe (Continuous): Error rate spiked to 15% → ❌
  Overall: FAIL ❌
  → Your system couldn't maintain steady state during chaos!
```

## 🔄 Litmus Chaos Workflows (ChaosScenarios)

A **Workflow** (also called ChaosScenario) lets you chain multiple experiments together:

```yaml
# chaos-workflow.yaml
# Run multiple chaos experiments in sequence
apiVersion: argoproj.io/v1alpha1         # Uses Argo Workflows under the hood!
kind: Workflow
metadata:
  name: full-chaos-scenario
  namespace: litmus
spec:
  entrypoint: chaos-workflow
  templates:
  - name: chaos-workflow
    steps:
    # Step 1: Delete pods first
    - - name: pod-delete-step
        template: pod-delete
    # Step 2: After pod-delete completes, add network latency
    - - name: network-latency-step
        template: network-latency
    # Step 3: After network latency, stress the CPU
    - - name: cpu-hog-step
        template: cpu-hog
  
  # ─── Templates for each experiment ───
  - name: pod-delete
    container:
      image: litmuschaos/go-runner:latest
      args: ["-c", "./experiments -name pod-delete"]
      command: ["/bin/bash"]
      env:
      - name: TOTAL_CHAOS_DURATION
        value: "30"
  
  - name: network-latency
    container:
      image: litmuschaos/go-runner:latest
      args: ["-c", "./experiments -name pod-network-latency"]
      command: ["/bin/bash"]
      env:
      - name: TOTAL_CHAOS_DURATION
        value: "60"
      - name: NETWORK_LATENCY
        value: "1000"
  
  - name: cpu-hog
    container:
      image: litmuschaos/go-runner:latest
      args: ["-c", "./experiments -name pod-cpu-hog"]
      command: ["/bin/bash"]
      env:
      - name: TOTAL_CHAOS_DURATION
        value: "60"
      - name: CPU_CORES
        value: "1"
```

```
Workflow Execution:

Step 1: pod-delete (30s)
  → Kills pods, verifies they recover
  ↓ (waits for completion)

Step 2: network-latency (60s)
  → Adds 1 second latency to network
  → Tests timeout handling
  ↓ (waits for completion)

Step 3: cpu-hog (60s)
  → Maxes out 1 CPU core
  → Tests HPA and resource limits

Total time: ~2.5 minutes of controlled chaos
```

## 🔗 Integrating Litmus with CI/CD

One of the most powerful features — run chaos experiments as part of your deployment pipeline:

### GitHub Actions Example:

```yaml
# .github/workflows/chaos-test.yaml
name: Chaos Engineering Tests

on:
  push:
    branches: [main]
  schedule:
    - cron: '0 10 * * 1-5'    # Run at 10 AM, weekdays

jobs:
  chaos-test:
    runs-on: ubuntu-latest
    steps:
    - name: Checkout Code
      uses: actions/checkout@v4

    - name: Set up kubectl
      uses: azure/setup-kubectl@v3
      with:
        version: 'latest'

    - name: Configure kubeconfig
      run: |
        echo "${{ secrets.KUBE_CONFIG }}" | base64 -d > kubeconfig.yaml
        export KUBECONFIG=kubeconfig.yaml

    - name: Install Litmus Operator
      run: |
        kubectl apply -f https://litmuschaos.github.io/litmus/litmus-operator-v3.0.0.yaml
        # Wait for operator to be ready
        kubectl wait --for=condition=ready pod -l name=chaos-operator -n litmus --timeout=120s

    - name: Install Chaos Experiments
      run: |
        kubectl apply -f https://hub.litmuschaos.io/api/chaos/3.0.0?file=charts/generic/pod-delete/experiment.yaml

    - name: Run Chaos Experiment
      run: |
        kubectl apply -f chaos/pod-delete-engine.yaml
        # Wait for the experiment to complete
        kubectl wait --for=jsonpath='{.status.experimentStatus.phase}'=Completed \
          chaosresult/nginx-chaos-pod-delete -n default --timeout=300s

    - name: Check Results
      run: |
        VERDICT=$(kubectl get chaosresult nginx-chaos-pod-delete -n default \
          -o jsonpath='{.status.experimentStatus.verdict}')
        echo "Chaos Experiment Verdict: $VERDICT"
        if [ "$VERDICT" != "Pass" ]; then
          echo "❌ Chaos experiment FAILED! System is not resilient enough."
          exit 1   # Fail the pipeline!
        fi
        echo "✅ Chaos experiment PASSED! System is resilient."

    - name: Cleanup
      if: always()
      run: |
        kubectl delete chaosengine nginx-chaos -n default --ignore-not-found
```

```
What this CI/CD pipeline does:

1. On every push to main (or on a schedule):
2. Sets up kubectl and connects to the cluster
3. Installs the Litmus operator
4. Installs the pod-delete experiment
5. Runs the chaos experiment
6. Checks the result
7. If the experiment FAILS → the pipeline FAILS
   → The deployment is blocked until resilience is fixed!
8. If the experiment PASSES → the pipeline continues
   → Deployment proceeds with confidence
```

## 📋 Best Practices for Litmus Chaos

```
✅ DO:
  1. Always use annotationCheck: 'true' — opt-in pods explicitly
  2. Start with pod-delete (simplest experiment) before trying network chaos
  3. Use probes to validate steady state — don't just observe manually
  4. Set PODS_AFFECTED_PERC low at first (10-20%), increase gradually
  5. Store chaos experiments in Git (they're just YAML!)
  6. Integrate with your monitoring stack (Prometheus + Grafana)
  7. Run experiments on a schedule (CronJob or CI/CD)
  8. Document results and share with your team
  9. Use namespaced RBAC — limit chaos permissions to specific namespaces

❌ DON'T:
  1. Skip the RBAC setup — always use least-privilege service accounts
  2. Run chaos without monitoring — you need to SEE what happens
  3. Target all pods at once (PODS_AFFECTED_PERC: 100) on your first try
  4. Forget to clean up — delete ChaosEngines after experiments
  5. Run node-level chaos without understanding the blast radius
  6. Ignore failed experiments — every failure is a learning opportunity
```
