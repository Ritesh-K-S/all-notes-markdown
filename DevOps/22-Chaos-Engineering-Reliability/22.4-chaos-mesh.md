## 🤔 What is Chaos Mesh?

Okay, let's start simple.

Imagine you have a beautiful Kubernetes cluster running 50 microservices. Everything works perfectly in testing. But what happens when:

- A pod suddenly crashes?
- The network between two services becomes super slow?
- A disk fills up on a node?
- DNS stops working for 30 seconds?

**Will your system survive?** Or will it fall apart like a house of cards?

**Chaos Mesh** helps you answer this question by **intentionally injecting these failures** into your Kubernetes cluster — in a safe, controlled, and automated way.

Think of it like a **fire drill** for your cluster. You don't wait for a real fire to find out if the exits work. You **practice** beforehand.

## 🏗️ Chaos Mesh Architecture — How Does It Work?

Chaos Mesh runs **inside your Kubernetes cluster** and has three main components:

```
┌─────────────────────────────────────────────────────┐
│                  Chaos Mesh Architecture              │
│                                                       │
│  ┌─────────────────┐    ┌──────────────────────┐     │
│  │  Chaos Dashboard │    │   chaos-controller    │     │
│  │   (Web UI)       │───▶│   -manager            │     │
│  │                  │    │   (Brain of Chaos      │     │
│  │ Create & Monitor │    │    Mesh — schedules    │     │
│  │ experiments      │    │    & manages chaos)    │     │
│  └─────────────────┘    └──────────┬───────────┘     │
│                                     │                  │
│                          Sends instructions            │
│                                     │                  │
│                          ┌──────────▼───────────┐     │
│                          │   chaos-daemon        │     │
│                          │   (Runs on EVERY node │     │
│                          │    as a DaemonSet —   │     │
│                          │    actually injects   │     │
│                          │    the chaos)          │     │
│                          └──────────────────────┘     │
│                                                       │
│  Uses Kubernetes CRDs (Custom Resource Definitions)   │
│  to define chaos experiments as YAML                  │
└─────────────────────────────────────────────────────┘
```

### Breaking It Down:

| Component | What It Does |
|-----------|-------------|
| **Chaos Dashboard** | A nice web UI where you can create, view, manage, and monitor chaos experiments visually |
| **Chaos Controller Manager** | The brain — it reads your chaos experiment definitions (CRDs) and decides what to do, when to do it, and for how long |
| **Chaos Daemon** | The worker — runs on every node as a DaemonSet. It's the one that actually injects faults (kills pods, adds network latency, corrupts files, etc.) using Linux kernel features like `iptables`, `tc`, `cgroups`, etc. |

### Key Design Decisions:

- **CRD-based**: Every chaos experiment is a Kubernetes Custom Resource. You define it in YAML, apply it with `kubectl`, and Kubernetes manages the lifecycle.
- **Runs as DaemonSet**: `chaos-daemon` runs on every node so it can inject failures on any pod, anywhere.
- **Uses Linux namespaces**: Chaos Daemon enters the target container's Linux namespace to inject faults — super precise, doesn't affect other containers.

## 📦 Installing Chaos Mesh

### Prerequisites:
- A running Kubernetes cluster (Minikube, Kind, EKS, AKS, GKE — anything works)
- `kubectl` configured
- Helm 3 installed

### Method 1: Install Using Helm (Recommended)

```bash
# Step 1: Add the Chaos Mesh Helm repo
helm repo add chaos-mesh https://charts.chaos-mesh.org

# Step 2: Update your Helm repos
helm repo update

# Step 3: Create a namespace for Chaos Mesh
kubectl create namespace chaos-mesh

# Step 4: Install Chaos Mesh
helm install chaos-mesh chaos-mesh/chaos-mesh \
  --namespace chaos-mesh \
  --set chaosDaemon.runtime=containerd \
  --set chaosDaemon.socketPath=/run/containerd/containerd.sock \
  --version 2.7.0
```

**What's happening here:**
- We add the official Chaos Mesh Helm chart repository
- We create a dedicated namespace `chaos-mesh` to keep things organized
- We install it with Helm, telling it which container runtime we use (`containerd` is the default for modern K8s clusters)
- The `socketPath` tells the chaos daemon where to find the container runtime socket

> **Note:** If you're using Docker as your runtime (older clusters), use `--set chaosDaemon.runtime=docker` and `--set chaosDaemon.socketPath=/var/run/docker.sock`

### Method 2: Install Using the Quick Script

```bash
# One-liner install (good for quick testing)
curl -sSL https://mirrors.chaos-mesh.org/v2.7.0/install.sh | bash
```

### Verify the Installation:

```bash
# Check that all Chaos Mesh pods are running
kubectl get pods -n chaos-mesh

# You should see something like:
# NAME                                       READY   STATUS    RESTARTS   AGE
# chaos-controller-manager-xxxxx-xxxxx       1/1     Running   0          2m
# chaos-daemon-xxxxx                         1/1     Running   0          2m
# chaos-daemon-yyyyy                         1/1     Running   0          2m
# chaos-dashboard-xxxxx-xxxxx               1/1     Running   0          2m
```

### Access the Dashboard:

```bash
# Port-forward the dashboard to your local machine
kubectl port-forward -n chaos-mesh svc/chaos-dashboard 2333:2333

# Now open http://localhost:2333 in your browser
```

## 🧪 Types of Chaos Experiments

This is where Chaos Mesh really shines. It supports a **huge variety** of fault types:

### Overview Table:

| Chaos Type | What It Does | Real-World Scenario |
|-----------|-------------|-------------------|
| **PodChaos** | Kill pods, cause container failures | "What if a pod suddenly dies?" |
| **NetworkChaos** | Add latency, packet loss, corrupt packets, partition network | "What if the network between services becomes unreliable?" |
| **StressChaos** | Stress CPU and memory | "What if a node runs out of CPU/memory?" |
| **IOChaos** | Add latency to disk I/O, return errors | "What if disk reads become really slow?" |
| **TimeChaos** | Skew the system clock | "What if the clock drifts on some pods?" |
| **DNSChaos** | Make DNS resolution fail or return wrong IPs | "What if DNS stops working temporarily?" |
| **HTTPChaos** | Inject faults into HTTP requests/responses | "What if an API starts returning 500 errors?" |
| **KernelChaos** | Inject kernel-level faults | "What if system calls start failing?" |
| **JVMChaos** | Inject faults into Java applications | "What if a Java method throws unexpected exceptions?" |
| **AWSChaos** | Stop/restart AWS EC2 instances, detach EBS volumes | "What if an EC2 instance goes down?" |
| **GCPChaos** | Stop/reset GCP VM instances | "What if a GCP VM goes down?" |
| **AzureChaos** | Stop/restart Azure VM instances | "What if an Azure VM goes down?" |
| **BlockChaos** | Inject block device faults | "What if the block storage has issues?" |
| **PhysicalMachineChaos** | Inject faults on physical/bare-metal machines | "What if physical hardware has issues?" |

That's a LOT of chaos options! Let's now look at each major type with real YAML examples.

## 🔥 PodChaos — Killing and Disrupting Pods

PodChaos is the simplest and most common type. It lets you:
- **Kill pods** (pod-kill)
- **Make containers fail** (container-kill)
- **Cause pod failures** (pod-failure)

### Example 1: Kill Random Pods

```yaml
# pod-kill-experiment.yaml
apiVersion: chaos-mesh.org/v1alpha1
kind: PodChaos
metadata:
  name: pod-kill-example
  namespace: default
spec:
  action: pod-kill          # Action: kill the pod
  mode: one                 # Kill just ONE random pod matching the selector
  selector:
    namespaces:
      - default             # Look for pods in the 'default' namespace
    labelSelectors:
      app: my-web-app       # Only target pods with label app=my-web-app
  duration: "30s"           # The experiment lasts 30 seconds
  gracePeriod: 0            # Kill immediately (no graceful shutdown)
```

**What this does:**
- Finds all pods in the `default` namespace with the label `app: my-web-app`
- Picks ONE random pod from those matches
- Kills it immediately (like `kill -9`)
- The experiment runs for 30 seconds (it may kill the pod again if it comes back)

**Apply it:**
```bash
kubectl apply -f pod-kill-experiment.yaml
```

**Check the status:**
```bash
kubectl describe podchaos pod-kill-example -n default
```

**Delete it (stop the chaos):**
```bash
kubectl delete -f pod-kill-experiment.yaml
```

### Example 2: Kill a Percentage of Pods

```yaml
# pod-kill-percentage.yaml
apiVersion: chaos-mesh.org/v1alpha1
kind: PodChaos
metadata:
  name: kill-50-percent
  namespace: default
spec:
  action: pod-kill
  mode: fixed-percent       # Kill a percentage of matching pods
  value: "50"               # Kill 50% of them
  selector:
    namespaces:
      - default
    labelSelectors:
      app: my-web-app
  duration: "60s"
```

**What this does:** Kills **50% of all pods** that match the label `app: my-web-app`. If you have 10 pods, 5 will be killed. This is great for testing if your app can handle losing half its instances.

### Example 3: Pod Failure (Make Pods Unavailable)

```yaml
# pod-failure-experiment.yaml
apiVersion: chaos-mesh.org/v1alpha1
kind: PodChaos
metadata:
  name: pod-failure-example
  namespace: default
spec:
  action: pod-failure       # Make the pod "fail" without actually killing it
  mode: one
  selector:
    namespaces:
      - default
    labelSelectors:
      app: my-web-app
  duration: "60s"           # Pod will be unavailable for 60 seconds
```

**What this does:** Makes a pod unavailable for 60 seconds without actually killing it. The pod enters a failed state, and after 60 seconds, Chaos Mesh cleans up and the pod recovers.

### Mode Options Explained:

| Mode | Description |
|------|-------------|
| `one` | Select ONE random target |
| `all` | Select ALL matching targets |
| `fixed` | Select a FIXED NUMBER of targets (set `value: "3"` for 3 pods) |
| `fixed-percent` | Select a PERCENTAGE of targets (set `value: "50"` for 50%) |
| `random-max-percent` | Select a RANDOM percentage up to the value (set `value: "50"` for 0-50%) |

## 🌐 NetworkChaos — Messing With the Network

Network failures are the **#1 most common failure** in distributed systems. NetworkChaos lets you simulate:
- Network **latency** (slow network)
- Packet **loss** (dropped packets)
- Packet **duplication**
- Packet **corruption**
- Network **partition** (complete isolation)
- **Bandwidth** throttling

### Example 1: Add Network Latency

```yaml
# network-delay.yaml
apiVersion: chaos-mesh.org/v1alpha1
kind: NetworkChaos
metadata:
  name: network-delay-example
  namespace: default
spec:
  action: delay              # Add network delay
  mode: all                  # Apply to ALL matching pods
  selector:
    namespaces:
      - default
    labelSelectors:
      app: payment-service   # Target the payment service
  delay:
    latency: "200ms"         # Add 200 milliseconds of delay
    correlation: "50"        # 50% correlation (makes it more realistic)
    jitter: "50ms"           # +/- 50ms random variation
  direction: to              # Apply delay to OUTGOING traffic
  duration: "120s"           # Run for 2 minutes
```

**What this does:**
- Targets all pods with label `app: payment-service`
- Adds **200ms of delay** to every outgoing network packet
- The `jitter` of `50ms` means the actual delay will be 150ms–250ms (randomly varying, more realistic)
- `correlation: 50` means the current delay is 50% correlated with the previous one (simulates real network behavior where delays tend to cluster)
- Only affects **outgoing** traffic (`direction: to`)

### Example 2: Packet Loss

```yaml
# network-loss.yaml
apiVersion: chaos-mesh.org/v1alpha1
kind: NetworkChaos
metadata:
  name: packet-loss-example
  namespace: default
spec:
  action: loss
  mode: all
  selector:
    namespaces:
      - default
    labelSelectors:
      app: order-service
  loss:
    loss: "30"               # Drop 30% of packets
    correlation: "25"        # 25% correlation
  duration: "60s"
```

**What this does:** Drops **30% of network packets** going to/from the `order-service`. This simulates an unreliable network — think of a flaky Wi-Fi connection but for your pods.

### Example 3: Network Partition (Complete Isolation)

```yaml
# network-partition.yaml
apiVersion: chaos-mesh.org/v1alpha1
kind: NetworkChaos
metadata:
  name: network-partition-example
  namespace: default
spec:
  action: partition            # Completely cut off network
  mode: all
  selector:
    namespaces:
      - default
    labelSelectors:
      app: database
  direction: both              # Cut both incoming and outgoing traffic
  target:
    selector:
      namespaces:
        - default
      labelSelectors:
        app: backend-api       # Cut network BETWEEN database and backend-api
    mode: all
  duration: "30s"
```

**What this does:** Creates a **network partition** between your `database` pods and your `backend-api` pods for 30 seconds. They **cannot communicate** at all during this time. This is one of the most important chaos tests — it simulates what happens when a network split occurs (a very real scenario in distributed systems).

### Example 4: Bandwidth Throttling

```yaml
# network-bandwidth.yaml
apiVersion: chaos-mesh.org/v1alpha1
kind: NetworkChaos
metadata:
  name: bandwidth-limit
  namespace: default
spec:
  action: bandwidth
  mode: all
  selector:
    namespaces:
      - default
    labelSelectors:
      app: file-service
  bandwidth:
    rate: "1mbps"             # Limit bandwidth to 1 Mbps
    limit: 20971520           # Queue limit in bytes (20 MB)
    buffer: 10000             # Buffer size in bytes
  duration: "300s"            # Run for 5 minutes
```

**What this does:** Throttles the network bandwidth of `file-service` to just **1 Mbps** for 5 minutes. Great for testing how your file upload/download service handles slow networks.

## 💻 StressChaos — CPU & Memory Stress

Want to see what happens when a pod runs out of CPU or memory? StressChaos is your friend.

### Example 1: CPU Stress

```yaml
# cpu-stress.yaml
apiVersion: chaos-mesh.org/v1alpha1
kind: StressChaos
metadata:
  name: cpu-stress-example
  namespace: default
spec:
  mode: one
  selector:
    namespaces:
      - default
    labelSelectors:
      app: my-web-app
  stressors:
    cpu:
      workers: 2             # Use 2 CPU-stressing workers
      load: 80               # Each worker consumes 80% of a CPU core
  duration: "120s"
```

**What this does:** Spins up **2 worker processes** inside the target pod, each consuming **80% of a CPU core**. This simulates a CPU spike — maybe a noisy neighbor or a CPU-intensive task. Your app will suddenly have much less CPU available. Will it still respond to requests?

### Example 2: Memory Stress

```yaml
# memory-stress.yaml
apiVersion: chaos-mesh.org/v1alpha1
kind: StressChaos
metadata:
  name: memory-stress-example
  namespace: default
spec:
  mode: one
  selector:
    namespaces:
      - default
    labelSelectors:
      app: my-web-app
  stressors:
    memory:
      workers: 1              # One memory-stressing worker
      size: "512MB"           # Consume 512 MB of memory
  duration: "60s"
```

**What this does:** Allocates **512 MB of memory** inside the target pod. If your pod has a memory limit of 1 GB, this suddenly eats half of it. Will the OOM killer step in? Will the app crash or degrade gracefully?

## 💾 IOChaos — Disk I/O Faults

### Example: Slow Disk Reads

```yaml
# io-delay.yaml
apiVersion: chaos-mesh.org/v1alpha1
kind: IOChaos
metadata:
  name: io-delay-example
  namespace: default
spec:
  action: latency
  mode: one
  selector:
    namespaces:
      - default
    labelSelectors:
      app: database
  delay: "100ms"              # Add 100ms delay to I/O operations
  volumePath: /var/lib/data   # Target this specific volume mount
  path: "*"                   # Apply to all files in the path
  percent: 50                 # Affect 50% of I/O operations
  duration: "120s"
```

**What this does:** Adds **100ms delay to 50% of disk I/O operations** on the `/var/lib/data` volume inside the `database` pod. Databases are very sensitive to disk performance — this test reveals how your database handles slow disks.

### Example: I/O Errors

```yaml
# io-fault.yaml
apiVersion: chaos-mesh.org/v1alpha1
kind: IOChaos
metadata:
  name: io-fault-example
  namespace: default
spec:
  action: fault
  mode: one
  selector:
    namespaces:
      - default
    labelSelectors:
      app: database
  errno: 5                    # Return EIO (I/O error) — error number 5
  volumePath: /var/lib/data
  path: "*.log"               # Only affect .log files
  percent: 100                # Every I/O operation on .log files will fail
  duration: "60s"
```

**What this does:** Makes **every I/O operation** on `.log` files in the database pod return an **I/O error** (`EIO`). This simulates a failing disk or corrupted filesystem — does your database handle write failures gracefully?

## ⏰ TimeChaos — Clock Skew

Clock drift can cause serious issues in distributed systems (think: certificate expiration, token validation, database timestamp conflicts).

### Example: Time Shift

```yaml
# time-chaos.yaml
apiVersion: chaos-mesh.org/v1alpha1
kind: TimeChaos
metadata:
  name: time-shift-example
  namespace: default
spec:
  mode: one
  selector:
    namespaces:
      - default
    labelSelectors:
      app: auth-service
  timeOffset: "-2h"           # Shift the clock BACK by 2 hours
  duration: "120s"
```

**What this does:** Shifts the system clock **2 hours into the past** for the `auth-service` pod. Imagine the auth service suddenly thinks it's 2 hours ago — will JWT tokens suddenly appear to be "from the future"? Will session cookies expire prematurely? This is a sneaky but very real failure mode.

## 🌐 DNSChaos — DNS Failures

### Example: DNS Errors

```yaml
# dns-chaos.yaml
apiVersion: chaos-mesh.org/v1alpha1
kind: DNSChaos
metadata:
  name: dns-error-example
  namespace: default
spec:
  action: error               # Make DNS queries return errors
  mode: all
  selector:
    namespaces:
      - default
    labelSelectors:
      app: frontend
  duration: "30s"
```

**What this does:** Makes **all DNS queries** from the `frontend` pods **fail** for 30 seconds. The frontend can't resolve any hostnames. Can it handle DNS failures gracefully? Does it cache DNS results? Does it retry?

### Example: DNS Spoofing (Return Wrong IP)

```yaml
# dns-random.yaml
apiVersion: chaos-mesh.org/v1alpha1
kind: DNSChaos
metadata:
  name: dns-random-example
  namespace: default
spec:
  action: random               # Return random IP addresses for DNS queries
  mode: all
  selector:
    namespaces:
      - default
    labelSelectors:
      app: frontend
  patterns:
    - "payment-service.*"      # Only affect DNS resolution for payment-service
  duration: "60s"
```

**What this does:** When `frontend` pods try to resolve `payment-service`, DNS returns a **random IP address** instead of the real one. This tests what happens when service discovery goes wrong.

## 🌐 HTTPChaos — HTTP Fault Injection

### Example: Inject HTTP 500 Errors

```yaml
# http-chaos.yaml
apiVersion: chaos-mesh.org/v1alpha1
kind: HTTPChaos
metadata:
  name: http-500-example
  namespace: default
spec:
  mode: all
  selector:
    namespaces:
      - default
    labelSelectors:
      app: backend-api
  target: Response             # Modify the response (not the request)
  port: 8080                   # Target port 8080
  path: "/api/orders*"         # Only affect paths matching /api/orders*
  method: GET                  # Only affect GET requests
  replace:
    code: 500                  # Replace the response code with 500
    body: '{"error": "Internal Server Error - Chaos Test"}'
  duration: "60s"
```

**What this does:** Every **GET request** to `/api/orders*` on the `backend-api` service will return a **500 Internal Server Error** instead of the real response. This tests if upstream services handle downstream failures properly — do they retry? Do they show a friendly error to the user? Or do they just crash?

## 📅 Scheduling Chaos Experiments

You don't always want to run chaos experiments manually. Chaos Mesh lets you **schedule** them!

### Example: Schedule Chaos with Cron

```yaml
# scheduled-chaos.yaml
apiVersion: chaos-mesh.org/v1alpha1
kind: Schedule
metadata:
  name: daily-pod-kill
  namespace: default
spec:
  schedule: "0 10 * * 1-5"    # Run at 10:00 AM, Monday to Friday
  type: PodChaos               # The type of chaos to run
  historyLimit: 5               # Keep last 5 experiment records
  concurrencyPolicy: Forbid     # Don't overlap experiments
  podChaos:
    action: pod-kill
    mode: one
    selector:
      namespaces:
        - default
      labelSelectors:
        app: my-web-app
    duration: "30s"
```

**What this does:** Automatically kills a random `my-web-app` pod at **10:00 AM every weekday**. This is like having a continuous fire drill — your team will be forced to make the app resilient because they know chaos is coming!

## 🔄 Workflows — Running Multiple Experiments Together

What if you want to run a sequence of chaos experiments, or multiple experiments at the same time? That's what **Workflows** are for.

### Example: Sequential Chaos Workflow

```yaml
# chaos-workflow.yaml
apiVersion: chaos-mesh.org/v1alpha1
kind: Workflow
metadata:
  name: chaos-workflow-example
  namespace: default
spec:
  entry: the-entry             # Starting point of the workflow
  templates:
    # Step 1: The entry point — run steps in sequence
    - name: the-entry
      templateType: Serial     # Run steps one after another
      deadline: "600s"         # Max total time: 10 minutes
      children:
        - network-delay-step
        - pod-kill-step
        - stress-step
    
    # Step 2: First, add network delay
    - name: network-delay-step
      templateType: NetworkChaos
      deadline: "120s"
      networkChaos:
        action: delay
        mode: one
        selector:
          namespaces:
            - default
          labelSelectors:
            app: my-web-app
        delay:
          latency: "200ms"
        duration: "60s"
    
    # Step 3: Then kill a pod
    - name: pod-kill-step
      templateType: PodChaos
      deadline: "60s"
      podChaos:
        action: pod-kill
        mode: one
        selector:
          namespaces:
            - default
          labelSelectors:
            app: my-web-app
        duration: "30s"
    
    # Step 4: Then stress the CPU
    - name: stress-step
      templateType: StressChaos
      deadline: "120s"
      stressChaos:
        mode: one
        selector:
          namespaces:
            - default
          labelSelectors:
            app: my-web-app
        stressors:
          cpu:
            workers: 1
            load: 90
        duration: "60s"
```

**What this does:**
1. First, adds 200ms network latency for 60 seconds
2. Then, kills a pod
3. Then, stresses the CPU at 90% for 60 seconds

This simulates a cascading failure scenario — which is how real outages often happen (one thing goes wrong, then another, then another).

### Parallel Workflow Example:

```yaml
# Change the entry template type to run steps simultaneously
- name: the-entry
  templateType: Parallel     # Run ALL steps at the same time
  deadline: "300s"
  children:
    - network-delay-step
    - pod-kill-step
    - stress-step
```

Use `Parallel` instead of `Serial` to run all chaos experiments at the same time — this is a more aggressive test!

## 🛡️ Safety Features — Controlling the Blast Radius

Chaos engineering should be **controlled**, not reckless. Chaos Mesh gives you several safety mechanisms:

### 1. Namespace Scoping

Only allow chaos in specific namespaces:

```yaml
selector:
  namespaces:
    - staging            # Only target the staging namespace
    # NEVER target production directly without proper controls
```

### 2. Label Selectors

Target only specific pods:

```yaml
selector:
  labelSelectors:
    app: my-web-app
    chaos-enabled: "true"    # Only pods that explicitly opt-in to chaos
```

### 3. Annotation Selectors

```yaml
selector:
  annotationSelectors:
    chaos-mesh.org/allow: "true"   # Only target pods with this annotation
```

### 4. Protected Namespaces

When installing Chaos Mesh, you can specify namespaces that should NEVER be targeted:

```bash
helm install chaos-mesh chaos-mesh/chaos-mesh \
  --namespace chaos-mesh \
  --set controllerManager.enableFilterNamespace=true
```

Then add a label to namespaces you want to protect:

```bash
kubectl label namespace kube-system chaos-mesh.org/inject=disabled
kubectl label namespace chaos-mesh chaos-mesh.org/inject=disabled
```

### 5. RBAC Integration

Chaos Mesh integrates with Kubernetes RBAC. You can control:
- Who can create chaos experiments
- Which namespaces they can target
- Which types of chaos they can use

```yaml
# Example: Allow user to only create PodChaos in staging
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: chaos-experimenter
  namespace: staging
rules:
  - apiGroups: ["chaos-mesh.org"]
    resources: ["podchaos"]
    verbs: ["create", "get", "list", "delete"]
```

## 🏢 Real-World Use Case — Complete Example

Let's put it all together. Imagine you have a typical microservices app:

```
User → Frontend → Backend API → Payment Service → Database
```

Here's a complete chaos testing plan:

```yaml
# full-chaos-test.yaml
---
# Test 1: What if the payment service pod dies?
apiVersion: chaos-mesh.org/v1alpha1
kind: PodChaos
metadata:
  name: payment-pod-kill
  namespace: default
spec:
  action: pod-kill
  mode: one
  selector:
    namespaces: [default]
    labelSelectors:
      app: payment-service
  duration: "30s"
---
# Test 2: What if the network between backend and database is slow?
apiVersion: chaos-mesh.org/v1alpha1
kind: NetworkChaos
metadata:
  name: db-network-delay
  namespace: default
spec:
  action: delay
  mode: all
  selector:
    namespaces: [default]
    labelSelectors:
      app: backend-api
  delay:
    latency: "500ms"
    jitter: "100ms"
  target:
    selector:
      namespaces: [default]
      labelSelectors:
        app: database
    mode: all
  direction: to
  duration: "120s"
---
# Test 3: What if the database disk is slow?
apiVersion: chaos-mesh.org/v1alpha1
kind: IOChaos
metadata:
  name: db-io-delay
  namespace: default
spec:
  action: latency
  mode: one
  selector:
    namespaces: [default]
    labelSelectors:
      app: database
  delay: "200ms"
  volumePath: /var/lib/postgresql/data
  path: "*"
  percent: 80
  duration: "120s"
```

**Apply all tests at once:**
```bash
kubectl apply -f full-chaos-test.yaml
```

**Clean up all tests:**
```bash
kubectl delete -f full-chaos-test.yaml
```

## 📝 Best Practices for Using Chaos Mesh

| Practice | Why |
|----------|-----|
| **Start in staging, NOT production** | Always test chaos experiments in a safe environment first |
| **Start small** | Begin with `mode: one` before testing with `mode: all` |
| **Set short durations initially** | Use `duration: "10s"` first, then increase |
| **Use label selectors carefully** | Make sure you're targeting ONLY the intended pods |
| **Have monitoring in place FIRST** | You need to see the impact of chaos — without monitoring, chaos experiments are blind |
| **Use the dashboard** | It's much easier to manage experiments visually |
| **Document your experiments** | Write down what you tested, what happened, and what you learned |
| **Integrate with CI/CD** | Run chaos experiments as part of your pipeline (after deployment to staging) |
| **Review results as a team** | Chaos engineering is a team sport — discuss findings together |
| **Fix what you find** | Finding weaknesses is useless if you don't fix them! |

## 🔑 Key Takeaways

1. **Chaos Mesh** is a powerful, CNCF-hosted, open-source chaos engineering platform built for Kubernetes.
2. It uses **CRDs** — you define chaos experiments as YAML and apply them with `kubectl`.
3. It supports a **massive variety of faults** — pod kills, network chaos, disk I/O, CPU/memory stress, DNS, HTTP, time skew, and even cloud provider failures.
4. **Workflows** let you chain multiple experiments together (serial or parallel).
5. **Scheduling** lets you automate chaos experiments on a cron schedule.
6. **Safety features** like namespace scoping, label selectors, and RBAC keep chaos controlled.
7. The **Dashboard** makes it easy to create, monitor, and manage experiments visually.
8. Always **start small**, **monitor the impact**, and **fix what you find**.
