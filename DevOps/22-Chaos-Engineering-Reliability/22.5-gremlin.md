# 22.5 — Gremlin — Enterprise Chaos Engineering

> **One Line Answer:** Gremlin is a **commercial, enterprise-grade chaos engineering platform** that lets you safely inject failures into your infrastructure — from Kubernetes to VMs to bare-metal servers to cloud services — with a super easy-to-use web UI, built-in safety controls, and **no YAML required**.

---

## 🤔 What is Gremlin?

Think of Gremlin as the **"easy mode" of chaos engineering**.

While tools like Chaos Mesh and Litmus Chaos require you to write YAML files and apply them to Kubernetes with `kubectl`, Gremlin gives you a **beautiful web interface** where you can literally click a few buttons and inject chaos into your systems.

But don't let the easy UI fool you — Gremlin is **incredibly powerful** under the hood. It's used by companies like **JP Morgan Chase, Target, Expedia, and Charter Communications** to test their production systems.

### The Gremlin Philosophy

Gremlin follows the principle of **"Thoughtful Chaos"** — meaning:

1. **Don't just break things randomly** — have a hypothesis about what will happen
2. **Start small** — test one thing at a time
3. **Scale gradually** — increase the blast radius as you gain confidence
4. **Stop immediately** if things go wrong — Gremlin has a built-in "halt all" button

## 🏗️ Gremlin Architecture — How It Works

```

│                     Gremlin Architecture                    │
│                                                             │
│  ┌─────────────────────┐                                    │
│  │   Gremlin Web App   │  ◄── You (the operator)           │

│             │                                               │

│  │  - Manages agents   │                                    │
│  │  - Sends commands   │                                    │
│  │  - Stores results   │                                    │
│  └──────────┬──────────┘                                    │
│             │                                               │
│     Secure connection (outbound only from agent)            │
│             │                                               │
│  ┌──────────▼──────────┐  ┌──────────────────┐              │
│  │  Gremlin Agent      │  │  Gremlin Agent   │              │

│  │  - Receives attack  │  │  - Same           │              │
│  │    instructions     │  │                   │              │
│  │  - Executes attacks │  │                   │              │
│  │  - Reports results  │  │                   │              │
│  └─────────────────────┘  └──────────────────┘              │
│                                                             │
│  OR for Kubernetes:                                         │
│  ┌─────────────────────────────┐                            │
│  │  Gremlin Kubernetes Agent   │                            │
│  │  (DaemonSet in your cluster)│                            │
│  │  - Can target individual    │                            │
│  │    pods, deployments,       │                            │
│  │    or nodes                 │                            │
│  └─────────────────────────────┘                            │
└────────────────────────────────────────────────────────────┘
```

### Key Components:

| Component | What It Does |
|-----------|-------------|
| **Gremlin Web App** | The main UI at `app.gremlin.com` — where you create attacks, view results, manage teams, and set up scenarios |
| **Gremlin Control Plane** | Hosted by Gremlin (SaaS) — manages communication between the web app and your agents |
| **Gremlin Agent** | Installed on your hosts/VMs/containers — the software that actually injects faults. Connects **outbound** to the control plane (no inbound firewall rules needed!) |
| **Gremlin Kubernetes Operator** | For K8s environments — deploys agents as a DaemonSet and can target individual pods/deployments |

### Important Security Detail:

The Gremlin agent connects **outbound** to the Gremlin control plane. This means:
- You don't need to open any inbound ports on your firewall
- The agent "phones home" to get instructions
- All communication is encrypted with TLS
- You authenticate agents using team certificates or API keys

## 📦 Installing Gremlin

### Step 1: Create a Gremlin Account

Go to [https://app.gremlin.com/signup](https://app.gremlin.com/signup) and sign up. Gremlin offers a **free tier** for individuals.

### Step 2A: Install on Linux/VMs

```bash
# Add the Gremlin repo (Debian/Ubuntu)
echo "deb https://deb.gremlin.com/ release non-free" | sudo tee /etc/apt/sources.list.d/gremlin.list

# Add the GPG key
sudo apt-key adv --keyserver keyserver.ubuntu.com --recv-keys 9CDB294B29A5B1E2E00C24C022E8EF3461A50EF6

# Install Gremlin
sudo apt-get update && sudo apt-get install -y gremlin gremlind
```

```bash
# Initialize Gremlin with your team credentials
sudo gremlin init
# It will ask you for:
# - Team ID (from the Gremlin web app)
# - Team Secret (from the Gremlin web app)
# OR you can use certificate-based authentication
```

### Step 2B: Install on Kubernetes (Helm)

```bash
# Add the Gremlin Helm repo
helm repo add gremlin https://helm.gremlin.com

# Create a namespace
kubectl create namespace gremlin

# Create a secret with your Gremlin credentials
kubectl create secret generic gremlin-team-secret \
  --namespace gremlin \
  --from-literal=GREMLIN_TEAM_ID=your-team-id \
  --from-literal=GREMLIN_TEAM_SECRET=your-team-secret

# Install Gremlin
helm install gremlin gremlin/gremlin \
  --namespace gremlin \
  --set gremlin.secret.managed=true \
  --set gremlin.secret.type=secret \
  --set gremlin.secret.teamID=your-team-id \
  --set gremlin.secret.clusterID=my-k8s-cluster
```

**What each flag means:**
- `gremlin.secret.managed=true` — Gremlin manages the secret
- `gremlin.secret.type=secret` — Use Kubernetes secret for auth
- `gremlin.secret.teamID` — Your Gremlin team ID (from the web app)
- `gremlin.secret.clusterID` — A friendly name for your cluster (shows up in the UI)

### Step 2C: Install Using Docker

```bash
# Run Gremlin agent as a Docker container
docker run -d \
  --name gremlin-agent \
  --net=host \
  --pid=host \
  --cap-add=NET_ADMIN \
  --cap-add=SYS_BOOT \
  --cap-add=SYS_TIME \
  --cap-add=KILL \
  -e GREMLIN_TEAM_ID="your-team-id" \
  -e GREMLIN_TEAM_SECRET="your-team-secret" \
  -v /var/run/docker.sock:/var/run/docker.sock \
  -v /var/log/gremlin:/var/log/gremlin \
  -v /var/lib/gremlin:/var/lib/gremlin \
  gremlin/gremlin daemon
```

**Why all those flags?**
- `--net=host` — Access the host network (needed for network attacks)
- `--pid=host` — Access host processes (needed for process/CPU attacks)
- `--cap-add=NET_ADMIN` — Permission to modify network settings
- `--cap-add=SYS_BOOT` — Permission to simulate reboots
- `--cap-add=SYS_TIME` — Permission to change system time
- `--cap-add=KILL` — Permission to kill processes

### Verify Installation:

After installing, your agents should appear in the Gremlin web app under **Clients**. You'll see:
- Hostname
- OS type
- IP address
- Agent version
- Status (active/inactive)

## 🧪 Types of Gremlin Attacks

Gremlin organizes its attacks into **four categories**:

### 1. 🔧 Resource Attacks

These consume resources on the target system.

| Attack | What It Does | Use Case |
|--------|-------------|----------|
| **CPU** | Consumes CPU cores | "What happens when CPU maxes out?" |
| **Memory** | Consumes RAM | "What if we run out of memory?" |
| **Disk** | Fills up disk space or does heavy I/O | "What if the disk fills up?" |
| **IO** | Stresses disk I/O | "What if disk is super busy?" |

### 2. 🌐 Network Attacks

These mess with network connectivity.

| Attack | What It Does | Use Case |
|--------|-------------|----------|
| **Blackhole** | Drops all network traffic | "What if a host goes completely offline?" |
| **Latency** | Adds delay to network packets | "What if the network is slow?" |
| **Packet Loss** | Drops a percentage of packets | "What if the network is unreliable?" |
| **DNS** | Blocks or redirects DNS queries | "What if DNS stops working?" |

### 3. 🖥️ State Attacks

These change the state of the target.

| Attack | What It Does | Use Case |
|--------|-------------|----------|
| **Shutdown** | Shuts down or reboots the host | "What if a server goes down?" |
| **Process Killer** | Kills specific processes | "What if this process crashes?" |
| **Time Travel** | Shifts the system clock | "What if the clock drifts?" |

### 4. 📦 Application-Level Attacks (Advanced)

| Attack | What It Does | Use Case |
|--------|-------------|----------|
| **Certificate Expiry** | Simulates expired TLS certificates | "What if a cert expires?" |
| **HTTP Faults** | Injects HTTP errors/latency | "What if the API returns errors?" |

## 🖱️ Running Your First Attack — Step by Step (Web UI)

This is where Gremlin shines. No YAML, no CLI, just a clean web interface.

### Example: CPU Attack via Web UI

1. **Log in** to [app.gremlin.com](https://app.gremlin.com)
2. Click **"Attacks"** in the left sidebar
3. Click **"New Attack"**
4. **Choose targets:**
   - Select **Hosts** or **Kubernetes** (containers/pods)
   - Pick the specific hosts/containers you want to target
5. **Choose the attack:**
   - Category: **Resource**
   - Attack type: **CPU**
   - **Cores:** 1 (start small!)
   - **CPU capacity (%):** 80 (consume 80% of one core)
   - **Length:** 60 seconds
6. Click **"Unleash Gremlin"** 🚀
7. **Watch** the attack progress in real-time on the dashboard
8. If anything goes wrong, click the big red **"Halt"** button

That's it! No YAML, no kubectl, no CLI. Just click and go.

## 💻 Running Attacks via CLI

If you prefer the command line (or want to automate), Gremlin has a CLI too.

### CPU Attack

```bash
# Run a CPU attack on the local machine
gremlin attack cpu \
  --cores 1 \
  --percent 80 \
  --length 60
```

**What this does:** Consumes 80% of 1 CPU core for 60 seconds on the machine where the command is run.

### Memory Attack

```bash
# Consume 512 MB of memory for 120 seconds
gremlin attack memory \
  --amount 512 \
  --type mb \
  --length 120
```

### Network Latency

```bash
# Add 200ms latency to traffic on port 443
gremlin attack latency \
  --ms 200 \
  --port 443 \
  --length 60
```

### Blackhole (Drop All Traffic)

```bash
# Drop all traffic to port 3306 (MySQL) for 30 seconds
gremlin attack blackhole \
  --port 3306 \
  --length 30
```

**What this does:** Completely blocks all network traffic to MySQL (port 3306) for 30 seconds. This simulates your database becoming unreachable — does your app handle it gracefully or does it crash?

### Packet Loss

```bash
# Drop 30% of packets for 60 seconds
gremlin attack packet_loss \
  --percent 30 \
  --length 60
```

### Process Killer

```bash
# Kill the nginx process
gremlin attack process_killer \
  --process nginx \
  --interval 5 \
  --length 60
```

**What this does:** Kills the `nginx` process every 5 seconds for 60 minutes. This tests if your process supervisor (systemd, Docker restart policy, Kubernetes) can restart the process quickly enough.

### DNS Attack

```bash
# Block DNS resolution for 30 seconds
gremlin attack dns \
  --length 30
```

### Shutdown (Reboot)

```bash
# Reboot the host after 10 seconds
gremlin attack shutdown \
  --delay 10 \
  --reboot
```

### Time Travel

```bash
# Shift the clock forward by 1 day
gremlin attack time_travel \
  --offset 86400 \
  --length 60
```

**What this does:** Moves the system clock forward by 86400 seconds (1 day). Think about it — what if suddenly your server thinks it's tomorrow? Certificates might expire, cron jobs might fire, tokens might become invalid.

## 🎯 Running Attacks via API

Gremlin has a full REST API, perfect for automation and CI/CD integration.

### API Authentication

```bash
# Get an API token
curl -X POST https://api.gremlin.com/v1/users/auth/mfa/auth \
  -H "Content-Type: application/json" \
  -d '{
    "email": "you@example.com",
    "password": "your-password",
    "companyName": "YourCompany"
  }'

# The response gives you a token — use it in subsequent requests
```

### Create an Attack via API

```bash
# Run a CPU attack via API
curl -X POST https://api.gremlin.com/v1/attacks/new \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer YOUR_API_TOKEN" \
  -d '{
    "target": {
      "type": "Random",
      "containers": {
        "multiSelectLabels": {
          "app": ["my-web-app"]
        }
      },
      "exact": 1
    },
    "command": {
      "type": "cpu",
      "args": ["-c", "1", "-p", "80", "-l", "60"]
    }
  }'
```

**What each part means:**
- `target.type: "Random"` — Pick random target(s) from matches
- `target.containers.multiSelectLabels` — Target containers with label `app=my-web-app`
- `target.exact: 1` — Target exactly 1 container
- `command.type: "cpu"` — Run a CPU attack
- `command.args` — `-c 1` (1 core), `-p 80` (80%), `-l 60` (60 seconds)

### Halt All Attacks via API

```bash
# Emergency stop — halt everything!
curl -X DELETE https://api.gremlin.com/v1/attacks \
  -H "Authorization: Bearer YOUR_API_TOKEN"
```

This is the API equivalent of the big red "Halt" button. It immediately stops ALL running attacks.

## 📋 Scenarios — Automated Chaos Test Suites

Gremlin **Scenarios** are like test suites — they let you define a sequence of attacks that run automatically. Think of them as chaos engineering "recipes".

### Built-In Scenarios (Recommended Scenarios)

Gremlin comes with pre-built scenarios based on common failure modes:

| Scenario | What It Tests |
|----------|-------------|
| **Validate Auto-Scaling** | Generates CPU/memory load and checks if auto-scaling kicks in |
| **Validate Redundancy** | Shuts down instances to check if traffic fails over correctly |
| **Validate DNS Failover** | Blocks DNS to test if DNS failover works |
| **Validate Container Recovery** | Kills containers to test restart policies |
| **Validate Certificate Expiry** | Checks what happens when TLS certificates expire |

### Creating a Custom Scenario

In the Gremlin web app:

1. Go to **Scenarios** → **New Scenario**
2. Give it a name: "Payment Service Resilience Test"
3. Add steps:
   - **Step 1:** Network latency (200ms) on payment-service for 60s
   - **Step 2:** (Wait for Step 1 to complete)
   - **Step 3:** Kill payment-service pod
   - **Step 4:** (Wait 30s — observe recovery)
   - **Step 5:** CPU stress on payment-service for 120s
4. Set **Health Checks:**
   - After each step, check if `https://my-app.com/health` returns 200 OK
   - If the health check fails, automatically **halt** the scenario
5. Click **Run**

### Scenario Health Checks

This is a killer feature. You can define health checks that Gremlin runs **during** the chaos experiment:

```
Health Check Configuration:
- Endpoint: https://api.myapp.com/health
- Method: GET
- Expected Status: 200
- Check Interval: 10 seconds
- If health check fails → HALT all attacks immediately
```

This means if your system starts actually breaking (not just being stressed), Gremlin automatically stops the chaos. **Safety first!**

## 📊 Status Checks — Validating System Health

Gremlin **Status Checks** let you define what "healthy" means for your system and continuously monitor it during chaos experiments.

### Types of Status Checks:

| Type | What It Monitors |
|------|-----------------|
| **Endpoint** | HTTP health check — monitors a URL for expected status code |
| **Datadog** | Checks Datadog monitors — "Are all monitors green?" |
| **PagerDuty** | Checks PagerDuty — "Are there any active incidents?" |
| **New Relic** | Checks New Relic alerts |
| **Custom Webhook** | Calls your custom webhook — you define what "healthy" means |

### Example Flow with Status Checks:

```
Start Attack → Status Check passes → Continue Attack
                                         ↓
                              Status Check FAILS → Auto-Halt! 🛑
                                         ↓
                              Alert the team, save results
```

## 🔐 Safety Features — Why Enterprises Love Gremlin

Gremlin is built for enterprises, which means safety is a **first-class citizen**:

### 1. The Halt Button

Every attack has a **Halt** button. You can also halt ALL attacks across ALL targets with one click. There's also a keyboard shortcut and an API endpoint for halting.

### 2. Attack Magnitude Limits

You can set team-wide limits:
- Max CPU attack: 50% (nobody can run more than 50% CPU attack)
- Max memory attack: 1 GB
- Max attack duration: 300 seconds
- Max targets: 5 hosts at a time

### 3. Role-Based Access Control (RBAC)

```
Team Admin:
  - Can manage team settings, users, and attack limits
  - Can create and run any attack

Operator:
  - Can create and run attacks
  - Cannot change team settings or limits

Observer:
  - Can view attacks and results
  - Cannot create or run attacks
```

### 4. Blast Radius Controls

- Target specific hosts, containers, or Kubernetes resources
- Limit the number of targets per attack
- Use tags to group and select targets

### 5. Audit Logs

Every action is logged:
- Who created the attack?
- When was it run?
- What were the parameters?
- What was the outcome?
- Who halted it?

This is critical for compliance (SOC 2, HIPAA, etc.).

### 6. SSO Integration

Gremlin supports Single Sign-On via:
- SAML 2.0
- Google Workspace
- Okta
- Azure AD
- LDAP

## 🔄 Gremlin + CI/CD — Automating Chaos in Your Pipeline

One of Gremlin's superpowers is integrating chaos experiments into your CI/CD pipeline.

### Example: GitHub Actions Integration

```yaml
# .github/workflows/chaos-test.yaml
name: Chaos Engineering Test

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to Staging
        run: |
          # Your deployment steps here
          kubectl apply -f deployment.yaml
          kubectl rollout status deployment/my-app

      - name: Wait for Deployment to Stabilize
        run: sleep 60

      - name: Run Gremlin Chaos Test
        env:
          GREMLIN_API_KEY: ${{ secrets.GREMLIN_API_KEY }}
          GREMLIN_TEAM_ID: ${{ secrets.GREMLIN_TEAM_ID }}
        run: |
          # Run a CPU stress test
          curl -X POST https://api.gremlin.com/v1/attacks/new \
            -H "Content-Type: application/json" \
            -H "Authorization: Key ${GREMLIN_API_KEY}" \
            -d '{
              "target": {
                "type": "Random",
                "containers": {
                  "multiSelectLabels": {
                    "app": ["my-app"],
                    "env": ["staging"]
                  }
                },
                "exact": 1
              },
              "command": {
                "type": "cpu",
                "args": ["-c", "1", "-p", "80", "-l", "60"]
              }
            }'

      - name: Validate System Health
        run: |
          # Wait for chaos to start
          sleep 10
          # Check if the app is still healthy during chaos
          for i in {1..12}; do
            STATUS=$(curl -s -o /dev/null -w "%{http_code}" https://staging.myapp.com/health)
            if [ "$STATUS" != "200" ]; then
              echo "Health check failed! Status: $STATUS"
              # Halt all attacks
              curl -X DELETE https://api.gremlin.com/v1/attacks \
                -H "Authorization: Key ${GREMLIN_API_KEY}"
              exit 1
            fi
            echo "Health check $i passed (Status: $STATUS)"
            sleep 5
          done
          echo "All health checks passed during chaos!"
```

**What this pipeline does:**
1. Deploys to staging
2. Waits for the deployment to stabilize
3. Runs a Gremlin CPU stress attack on the deployed app
4. Continuously checks the app's health during the chaos
5. If the app becomes unhealthy → halts the attack and fails the pipeline
6. If the app stays healthy → chaos test passes!

## ⚡ Gremlin vs Open-Source Tools — When to Use Gremlin?

| Aspect | Gremlin | Open-Source (Chaos Mesh/Litmus) |
|--------|---------|-------------------------------|
| **Cost** | Paid (free tier available) | Free |
| **Ease of Use** | Very easy — web UI, no YAML | Requires YAML & kubectl knowledge |
| **Setup Time** | Minutes | 30+ minutes |
| **Support** | Enterprise support, SLAs | Community support |
| **Multi-Platform** | K8s + VMs + Bare Metal + Cloud | Mostly K8s-focused |
| **Safety Features** | Enterprise-grade (RBAC, limits, SSO, audit logs) | Basic |
| **Scenarios** | Built-in recommended scenarios | Build your own |
| **Status Checks** | Built-in (Datadog, PagerDuty, etc.) | Build your own |
| **Compliance** | SOC 2 certified | Self-managed |
| **CI/CD Integration** | REST API, native integrations | REST API, kubectl |

### When to Choose Gremlin:
- Your organization is **enterprise/large** and needs compliance, RBAC, and audit logs
- You want **quick time-to-value** — get started in minutes, not hours
- You need to test **non-Kubernetes infrastructure** (VMs, bare metal)
- Your team is **new to chaos engineering** and needs guided scenarios
- You need **status check integrations** with your existing monitoring tools
- You need **enterprise support** and SLAs

### When to Choose Open-Source:
- You're **cost-sensitive** and can invest time in setup
- You're **100% Kubernetes** and don't need VM/bare-metal support
- Your team is **technically strong** and comfortable with YAML/kubectl
- You want **full control** over the chaos engineering platform
- You want to **contribute to the project** or customize it

## 🏢 Real-World Use Cases

### Use Case 1: E-Commerce — Black Friday Preparation

```
Problem: "Will our system handle Black Friday traffic?"

Gremlin Approach:
1. Run CPU stress on web servers (simulate high traffic CPU usage)
2. Add 500ms latency between microservices (simulate network congestion)
3. Kill 30% of pods (simulate scaling issues)
4. Fill disk on logging servers (simulate log overflow)

Result: Found 3 critical issues:
- Payment service didn't have proper circuit breakers
- Log rotation wasn't configured (disk would fill up in 2 hours under load)
- Auto-scaling was too slow (3 minutes to spin up new pods)
```

### Use Case 2: Financial Services — Region Failover

```
Problem: "If us-east-1 goes down, will traffic fail over to us-west-2?"

Gremlin Approach:
1. Use Blackhole attack to cut off all network to us-east-1 services
2. Monitor if Route 53 health checks detect the failure
3. Verify traffic fails over to us-west-2
4. Check data consistency after failover

Result: Failover worked, but took 4 minutes instead of expected 30 seconds.
Root cause: Route 53 health check interval was set to 30 seconds with a
threshold of 3 failures → 90 seconds minimum detection time + DNS TTL.
```

### Use Case 3: Healthcare — Compliance Testing

```
Problem: "Can we prove our system meets uptime SLAs for HIPAA compliance?"

Gremlin Approach:
1. Run weekly automated scenarios against staging
2. Document all tests, results, and remediation
3. Use Gremlin's audit logs as evidence for auditors
4. Prove that single points of failure have been identified and fixed

Result: Passed HIPAA audit with chaos engineering evidence.
```

## 📝 Best Practices for Using Gremlin

| Practice | Why It Matters |
|----------|---------------|
| **Start with the Recommended Scenarios** | Gremlin's built-in scenarios cover the most common failure modes — start there |
| **Use Status Checks** | Always define what "healthy" means before running chaos |
| **Set Attack Limits** | Configure team-wide limits to prevent accidental disasters |
| **Run in Staging First** | Gain confidence in staging before touching production |
| **Keep the Blast Radius Small** | Start with 1 target, then scale up |
| **Integrate with Your Monitoring** | Connect Gremlin with Datadog/New Relic/PagerDuty for automated safety |
| **Schedule Regular Chaos** | Don't just run chaos once — make it a habit (weekly Game Days) |
| **Document Everything** | Write down hypotheses, results, and lessons learned |
| **Get Buy-In from Leadership** | Chaos engineering needs organizational support — not just engineering |
| **Celebrate Findings** | Every bug found by chaos engineering is a bug that didn't hit production |
