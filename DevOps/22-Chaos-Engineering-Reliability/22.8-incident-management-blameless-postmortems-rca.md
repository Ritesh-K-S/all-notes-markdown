## 🤔 What is an Incident?

Let's start simple.

> **An incident is any unplanned event that disrupts or degrades a service for users.**

### Examples of Incidents:

| Incident | Impact |
|----------|--------|
| Website returns 500 errors | Users can't access the application |
| Payment processing is down | Revenue loss, customer frustration |
| Database is running out of disk | Service will crash soon if not fixed |
| Latency spikes to 10 seconds | Users experience extremely slow pages |
| Security breach detected | Customer data potentially exposed |
| Deployment broke production | New version is crashing |

### Not Everything is an Incident:

| Not an Incident | Why |
|-----------------|-----|
| A single failed request | Normal — systems have small error rates |
| A background job retry | Expected behavior, self-healing |
| A dev server going down | No user impact |
| A known/planned maintenance | Users were informed, it's expected |

---

## 📊 Incident Severity Levels

Not all incidents are equal. You need a **severity classification** so your team knows how urgently to respond.

### Common Severity Scale:

| Severity | Name | Description | Example | Response Time |
|----------|------|-------------|---------|---------------|
| **SEV-1** (P1) | Critical | Complete service outage, major revenue impact | Payment system is completely down | Immediate (< 5 min) |
| **SEV-2** (P2) | Major | Significant degradation, many users affected | Homepage loads but is extremely slow | < 15 minutes |
| **SEV-3** (P3) | Minor | Partial degradation, some users affected | One region is experiencing errors | < 1 hour |
| **SEV-4** (P4) | Low | Minor issue, minimal impact | A non-critical dashboard is broken | Next business day |

### How to Decide Severity:

```
Ask these questions:
1. How many users are affected?        → More users = Higher severity
2. Is there revenue impact?            → Money loss = Higher severity
3. Is there data loss/security risk?   → Data issues = Always high severity
4. Is there a workaround?              → No workaround = Higher severity
5. How long has it been going on?      → Long-running = May escalate severity
```

## 🚨 The Incident Response Lifecycle

When an incident happens, you don't just randomly scramble. There's a structured process:

```
┌──────────────────────────────────────────────────────────────┐
│               Incident Response Lifecycle                    │
│                                                              │
│  1. DETECTION    → Alert fires, user reports, monitoring     │
│        ↓                                                     │
│  2. TRIAGE       → Assess severity, assign roles             │
│        ↓                                                     │
│  3. RESPONSE     → Investigate, communicate, mitigate        │
│        ↓                                                     │
│  4. MITIGATION   → Stop the bleeding (even if it's a hack)  │
│        ↓                                                     │
│  5. RESOLUTION   → Fix the root cause properly               │
│        ↓                                                     │
│  6. POSTMORTEM   → Learn from it, prevent recurrence         │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

Let's go through each phase in detail.

---

## Phase 1: Detection

### How Incidents Are Detected:

| Detection Method | Description | Example |
|-----------------|-------------|---------|
| **Monitoring Alerts** | Automated alerts from Prometheus, Datadog, etc. | Alert: "Error rate > 5% for 5 minutes" |
| **User Reports** | Customers report issues via support | "I can't log in!" |
| **Synthetic Monitoring** | Automated health checks | Uptime monitor shows service is down |
| **Internal Discovery** | An engineer notices something wrong | "Hey, the dashboard looks weird" |
| **Error Budget Burn** | Error budget is depleting too fast | Burn rate alert fires |

### Best Practice — Alerting Rules:

```yaml
# Prometheus Alert Example
groups:
  - name: payment-service-alerts
    rules:
      # High Error Rate Alert
      - alert: PaymentHighErrorRate
        expr: |
          sum(rate(http_requests_total{service="payment", status=~"5.."}[5m]))
          /
          sum(rate(http_requests_total{service="payment"}[5m]))
          > 0.05
        for: 5m
        labels:
          severity: critical
          team: payments
        annotations:
          summary: "Payment service error rate is above 5%"
          description: "Error rate is {{ $value | humanizePercentage }} for the last 5 minutes"
          runbook: "https://wiki.company.com/runbooks/payment-high-error-rate"

      # High Latency Alert
      - alert: PaymentHighLatency
        expr: |
          histogram_quantile(0.99, 
            sum(rate(http_request_duration_seconds_bucket{service="payment"}[5m])) by (le)
          ) > 2
        for: 5m
        labels:
          severity: warning
          team: payments
        annotations:
          summary: "Payment service p99 latency is above 2 seconds"
```

## Phase 2: Triage & Roles

When an incident is declared, you need **clear roles**. Without roles, everyone runs around confused.

### Incident Response Roles:

| Role | Responsibility | Who? |
|------|---------------|------|
| **Incident Commander (IC)** | Runs the show. Makes decisions. Coordinates everyone. | Senior on-call engineer or manager |
| **Technical Lead** | Digs into the technical issue. Investigates root cause. | Most knowledgeable engineer for the affected system |
| **Communications Lead** | Updates stakeholders, customers, status page. | PM, support lead, or designated person |
| **Scribe** | Documents everything that happens during the incident (timeline). | Anyone available |

### Why Roles Matter:

```
Without roles:                          With roles:
─────────────                          ──────────
Everyone investigates                   IC coordinates
Nobody communicates                     Comms lead updates
People duplicate work                   Tech lead focuses
Things get missed                       Scribe captures all
Chaos! 🔥                              Organized! ✅
```

### Example Incident Declaration (Slack/Chat):

```
🚨 INCIDENT DECLARED — SEV-1 🚨

What: Payment service returning 500 errors for all users
Impact: No payments can be processed
When detected: 2025-03-15 14:32 UTC
Detection method: Prometheus alert — PaymentHighErrorRate

Roles:
  - Incident Commander: @alice
  - Technical Lead: @bob
  - Communications Lead: @charlie
  - Scribe: @diana

War Room: #incident-20250315-payment
Status Page: Being updated

Next update in: 15 minutes
```

## Phase 3 & 4: Response & Mitigation

### The Golden Rule of Incident Response:

> **Mitigate first. Investigate later.**

Your first priority is to **stop the bleeding** — restore service for users. Understanding WHY it happened comes AFTER.

### Common Mitigation Strategies:

| Strategy | When to Use | Example |
|----------|------------|---------|
| **Rollback** | Bad deployment broke things | Roll back to the previous version |
| **Restart** | Service is in a bad state | Restart the pods/service |
| **Scale Up** | System is overloaded | Add more replicas/instances |
| **Failover** | Primary system is down | Switch to backup/secondary |
| **Feature Flag** | New feature is causing issues | Disable the feature flag |
| **Traffic Shift** | One region/AZ is unhealthy | Route traffic away from it |
| **Hotfix** | Quick code fix can solve it | Deploy a targeted fix |

### Example Mitigation — Kubernetes Rollback:

```bash
# Check the deployment history
kubectl rollout history deployment/payment-service -n production

# Rollback to the previous version
kubectl rollout undo deployment/payment-service -n production

# Check rollback status
kubectl rollout status deployment/payment-service -n production

# Verify pods are healthy
kubectl get pods -n production -l app=payment-service
```

### Communication During Incidents:

Regular updates are CRITICAL. Even if you don't have new information, say so.

```
📢 INCIDENT UPDATE — 14:45 UTC (13 min after detection)

Status: INVESTIGATING
Impact: Payment processing remains down for all users
Actions taken:
  - Identified recent deployment as potential cause
  - Initiating rollback to v2.3.4
  - Database connections look healthy
  
Next update: 15:00 UTC
```

```
📢 INCIDENT UPDATE — 15:02 UTC

Status: MITIGATED ✅
Impact: Payment processing is recovering
Actions taken:
  - Rollback to v2.3.4 completed
  - Error rate dropping: 45% → 2% → 0.1%
  - Monitoring closely

Next update: 15:15 UTC (confirmation of full recovery)
```

## Phase 5: Resolution

Once the immediate fire is out, you properly fix the root cause.

```
Mitigation ≠ Resolution

Mitigation: "We rolled back the bad deployment"
            → Service is working, but we haven't fixed the bug

Resolution: "We found the bug, fixed it, added tests, 
             and deployed the fix"
            → The actual problem is solved
```

## Phase 6: Postmortem — The Most Important Phase

### What is a Postmortem?

A postmortem is a **structured document** written after an incident that captures:
1. What happened
2. Why it happened
3. How we responded
4. What we'll do to prevent it from happening again

> **The goal of a postmortem is LEARNING, not BLAME.**

---

## 🤝 Blameless Postmortems — The Heart of SRE Culture

### What Does "Blameless" Mean?

> **Blameless does NOT mean "no accountability." It means we focus on the SYSTEM that allowed the failure, not the PERSON who made the mistake.**

### The Wrong Way (Blame Culture):

```
❌ "Bob pushed a bad deployment and broke production. 
    Bob should have been more careful."

Result:
  - Bob feels terrible
  - Next time, Bob will be afraid to deploy
  - Others learn to hide their mistakes
  - Nobody learns anything useful
  - The SYSTEM that allowed this is never fixed
```

### The Right Way (Blameless Culture):

```
✅ "A deployment went out that caused a production outage. 
    Let's understand:
    - Why did the deployment pass all tests but still break production?
    - Why didn't our canary deployment catch this?
    - Why didn't we have better monitoring for this failure mode?
    - What changes to our SYSTEM can prevent this?"

Result:
  - Nobody feels punished
  - People freely share information
  - We find the REAL causes (system gaps)
  - We make meaningful improvements
  - The same mistake can't happen again
```

### Why Blameless Works:

```
In aviation, when a plane crashes:
  - Pilots are NOT punished for reporting issues
  - The ENTIRE system is analyzed
  - This is why flying is incredibly safe

In medicine, when a mistake happens:
  - If we blame the nurse, they won't report errors
  - If we fix the system, we prevent future errors

Same principle applies to software:
  - Blame the person → They hide future mistakes
  - Fix the system → Future mistakes are prevented
```

### The Blameless Postmortem Rule:

> **We assume that everyone involved had good intentions and made the best decisions they could with the information they had at the time.**

## Root Cause Analysis

### What Happened:
The deployment v2.3.5 included a change to the database connection 
pooling library (upgraded from v3.2.1 to v4.0.0). The new version 
changed the default connection timeout from 30 seconds to 5 seconds. 
Under normal load, connections that took 6-10 seconds (common during 
peak) were being terminated, causing cascading failures.

### Why It Wasn't Caught:
1. **Unit tests passed** — they don't test with production-like latency
2. **Integration tests passed** — test database is fast (< 1ms)
3. **Canary deployment** — canary only ran for 2 minutes, not enough 
   time for the issue to manifest under load
4. **No load testing** — we don't run load tests before production 
   deployments

### The 5 Whys:
1. **Why did payments fail?**
   → Database connections were being terminated prematurely
   
2. **Why were connections terminated?**
   → Connection timeout was set to 5 seconds (changed by library upgrade)
   
3. **Why didn't we know about the timeout change?**
   → We didn't review the library's CHANGELOG for breaking changes
   
4. **Why didn't testing catch it?**
   → Our test environments don't simulate production-like latency
   
5. **Why don't we simulate production latency in tests?**
   → We never set up load testing as part of our CI/CD pipeline

### What Didn't Go Well:

- No data corruption occurred

## 🔍 Root Cause Analysis (RCA) — Deep Dive

### What is RCA?

> **Root Cause Analysis (RCA) is a systematic process of identifying the UNDERLYING cause of an incident — not just the surface-level trigger.**

### Why Surface-Level Analysis is Dangerous:

```
Surface Level: "The server crashed because it ran out of memory."
Fix: "Add more memory."
Result: It crashes again next month. 😞

Root Cause: "The application has a memory leak in the session 
            handling code that accumulates over 2 weeks."
Fix: "Fix the memory leak, add memory monitoring alerts."
Result: Problem is permanently solved. ✅
```

### RCA Techniques:

#### 1. The 5 Whys Method

The simplest and most popular technique. Just keep asking "Why?" until you reach the root cause.

```
Incident: Website went down for 2 hours

Why 1: Why did the website go down?
  → The web server ran out of memory and crashed.

Why 2: Why did it run out of memory?
  → A memory leak in the user session module.

Why 3: Why was there a memory leak?
  → Sessions were never being cleaned up after user logout.

Why 4: Why weren't sessions cleaned up?
  → The cleanup code was removed during a refactor 3 months ago.

Why 5: Why wasn't this caught?
  → We don't have tests for session cleanup, and our monitoring 
    doesn't track memory trends over time.

ROOT CAUSE: Missing test coverage for session lifecycle + 
            no long-term memory trend monitoring.

ACTION ITEMS:
  1. Fix the session cleanup code
  2. Add tests for session lifecycle
  3. Add memory trend monitoring and alerting
```

#### 2. Fishbone Diagram (Ishikawa)

This technique categorizes potential causes into groups:

```
                          INCIDENT: Website Down
                                |
        ┌───────────┬───────────┼───────────┬───────────┐
        |           |           |           |           |
    People      Process     Technology  Environment  Data
        |           |           |           |           |
   - No backup  - No review  - Memory    - Traffic   - Corrupt
     on-call      process      leak        spike       cache
   - Junior     - No load    - Old        - DNS       - Bad
     engineer     testing      library     failure     migration
```

#### 3. Timeline Analysis

Create a detailed timeline and look for the point where things went wrong:

```
13:00 - Normal operations ✅
13:15 - Deploy v2.5 started ✅
13:18 - Deploy complete, health checks pass ✅
13:25 - Memory usage starts climbing ⚠️  ← First sign of trouble
13:45 - Memory at 80%, no alert triggered ⚠️
14:00 - Memory at 95%, OOM killer invoked ❌ ← Failure point
14:02 - Alert fires (too late!)
14:15 - Rollback initiated

Key findings:
- Memory started climbing at 13:25 (7 min after deploy)
- Alert threshold was set at OOM (too late)
- Should have had an alert at 70% memory
```

#### 4. Contributing Factors Analysis

Most incidents don't have a single root cause — they have multiple **contributing factors** that combined to cause the failure:

```
Contributing Factors for Payment Outage:

1. TRIGGER: Library upgrade changed default timeout
   (This started the chain of events)

2. INSUFFICIENT TESTING: No load tests in CI/CD
   (This allowed the trigger to reach production)

3. SHORT CANARY: 2-minute canary window
   (This didn't give enough time to detect the issue)

4. IMPLICIT CONFIGURATION: Relying on library defaults
   (This made us vulnerable to upstream changes)

5. MONITORING GAP: No alert on connection pool metrics
   (This delayed detection)

All 5 factors had to be present for the outage to occur.
Fixing ANY ONE of them would have prevented or reduced the impact.
```

## 🔄 Incident Management Process — Complete Workflow

Here's a complete incident management workflow that you can adopt:

### Pre-Incident Preparation:

```yaml
# What to have BEFORE incidents happen

on_call_rotation:
  tool: PagerDuty / Opsgenie
  schedule: Weekly rotation
  escalation:
    - level_1: Primary on-call (5 min response)
    - level_2: Secondary on-call (15 min response)
    - level_3: Team lead / Manager (30 min response)

runbooks:
  location: "https://wiki.company.com/runbooks/"
  template: Standardized format
  requirements:
    - One runbook per alert
    - Step-by-step mitigation steps
    - Escalation contacts
    - Links to dashboards

communication:
  incident_channel: "#incidents" (auto-created per incident)
  status_page: StatusPage.io / Atlassian Statuspage
  stakeholder_updates: Every 15-30 minutes during SEV-1/SEV-2

tools:
  alerting: Prometheus + AlertManager
  paging: PagerDuty
  chat: Slack (#incident-YYYYMMDD-description)
  video: Zoom / Google Meet (war room)
  status: StatusPage.io
  postmortem: Google Docs / Notion
```

### Runbook Example:

```markdown
# Runbook: Payment Service — High Error Rate

## Alert
- Name: PaymentHighErrorRate
- Condition: Error rate > 5% for 5 minutes
- Severity: SEV-1

## Quick Diagnosis

### Step 1: Check current error rate
```bash
# Check error rate in Prometheus
curl -s "http://prometheus:9090/api/v1/query?query=sum(rate(http_requests_total{service='payment',status=~'5..'}[5m]))/sum(rate(http_requests_total{service='payment'}[5m]))"
```

### Step 2: Check recent deployments
```bash
kubectl rollout history deployment/payment-service -n production
```

### Step 3: Check pod health
```bash
kubectl get pods -n production -l app=payment-service
kubectl describe pod <pod-name> -n production
kubectl logs <pod-name> -n production --tail=100
```

### Step 4: Check database connectivity
```bash
kubectl exec -it <pod-name> -n production -- \
  pg_isready -h payment-db.internal -p 5432
```

## Mitigation Options

### Option A: Rollback (if recent deployment)
```bash
kubectl rollout undo deployment/payment-service -n production
```

### Option B: Restart pods (if in bad state)
```bash
kubectl rollout restart deployment/payment-service -n production
```

### Option C: Scale up (if overloaded)
```bash
kubectl scale deployment/payment-service -n production --replicas=10
```

## Escalation
- Primary: @payments-oncall
- Secondary: @payments-team-lead  
- Database issues: @database-team
```

## 📊 Measuring Incident Management

Track these metrics to improve your incident response over time:

### Key Metrics:

| Metric | What It Measures | Target |
|--------|-----------------|--------|
| **MTTD** (Mean Time to Detect) | How quickly we discover incidents | < 5 minutes |
| **MTTA** (Mean Time to Acknowledge) | How quickly someone starts looking | < 10 minutes |
| **MTTR** (Mean Time to Resolve) | How quickly we fix incidents | < 1 hour (SEV-1) |
| **MTBF** (Mean Time Between Failures) | How often incidents happen | Increasing over time |
| **Incident Count** | Total incidents per period | Decreasing over time |
| **Postmortem Completion Rate** | % of incidents with postmortems | 100% for SEV-1/SEV-2 |
| **Action Item Completion** | % of postmortem actions completed | > 90% |

```
MTTD + MTTA + MTTR = Total Incident Duration

Timeline:
──────────────────────────────────────────────────────────→
│         │              │                    │
Incident  Detected      Acknowledged          Resolved
Starts                   
│←─ MTTD ─→│←── MTTA ──→│←───── MTTR ───────→│
```

## 🏢 Building an Incident Management Culture

### Do's and Don'ts:

```
DO:
✅ Declare incidents early (when in doubt, declare it)
✅ Communicate frequently during incidents
✅ Focus on mitigation first, investigation later
✅ Write postmortems for every significant incident
✅ Follow up on ALL action items
✅ Practice incident response (game days)
✅ Celebrate good incident responses
✅ Share postmortems widely (transparency)

DON'T:
❌ Blame individuals for causing incidents
❌ Skip postmortems because you're "too busy"
❌ Let action items die in a backlog
❌ Wait too long to declare an incident
❌ Have heroes who do everything alone
❌ Keep incident details secret
❌ Punish people for making mistakes
❌ Treat postmortems as a bureaucratic checkbox
```

## 📚 Key Takeaways

```
✅ An incident is any unplanned event that disrupts service

✅ Clear roles (IC, Tech Lead, Comms, Scribe) prevent chaos
✅ MITIGATE FIRST, investigate later
✅ Blameless postmortems focus on the SYSTEM, not the person
✅ The 5 Whys technique digs to the root cause
✅ Most incidents have multiple contributing factors
✅ MTTD, MTTA, MTTR are key metrics to track
✅ Runbooks and on-call rotations are essential preparation
✅ Action items from postmortems MUST be completed
✅ Practice incident response before real incidents happen
