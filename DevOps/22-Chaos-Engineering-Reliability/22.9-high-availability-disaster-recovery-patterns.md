## 🤔 What's the Difference Between HA and DR?

Before we dive deep, let's understand the difference — because people confuse these ALL the time.

| | High Availability (HA) | Disaster Recovery (DR) |
|---|---|---|
| **Goal** | Keep the system running despite failures | Recover the system after a major disaster |
| **When it kicks in** | Component failure (server crash, disk failure) | Catastrophic failure (data center fire, region outage) |
| **Scope** | Usually within a data center/region | Across data centers/regions |
| **Downtime** | Seconds to minutes (or zero!) | Minutes to hours |
| **Cost** | Moderate — redundancy within a location | Higher — duplicate infrastructure in another location |
| **Automatic?** | Usually yes (failover is automatic) | Can be automatic or manual |

### The Simple Analogy:

```
High Availability = Having a spare tire in your car
  → A tire blows out, you swap it and keep driving
  → Minimal disruption

Disaster Recovery = Having a second car at home
  → Your car gets totaled in an accident
  → You go home and get the other car
  → More disruption, but you can still get to work
```

## 🏗️ High Availability (HA) — Deep Dive

### What is High Availability?

> **High Availability means designing a system so that it continues to operate (with acceptable performance) even when individual components fail.**

The key insight: **individual components WILL fail** — that's guaranteed. HA is about designing so that **no single component failure brings down the whole system**.

### Measuring Availability — The Nines:

Availability is measured in "nines":

| Availability | Downtime/Year | Downtime/Month | Common Name |
|-------------|---------------|----------------|-------------|
| 99% | 3.65 days | 7.3 hours | "Two nines" |
| 99.9% | 8.76 hours | 43.2 minutes | "Three nines" |
| 99.95% | 4.38 hours | 21.6 minutes | "Three and a half nines" |
| 99.99% | 52.6 minutes | 4.32 minutes | "Four nines" |
| 99.999% | 5.26 minutes | 26.3 seconds | "Five nines" |
| 99.9999% | 31.5 seconds | 2.6 seconds | "Six nines" |

### Availability Formula:

```
Availability = Uptime / (Uptime + Downtime) × 100%

Or alternatively:
Availability = MTBF / (MTBF + MTTR)

Where:
  MTBF = Mean Time Between Failures
  MTTR = Mean Time To Repair
```

**Key Insight:** You can improve availability two ways:
1. **Increase MTBF** — Make failures happen less often
2. **Decrease MTTR** — Fix failures faster

## HA Pattern 1: Redundancy

### The Fundamental Principle:

> **Never have a Single Point of Failure (SPOF).** If any single component failure can bring down your system, you don't have HA.

### Types of Redundancy:

#### Active-Active (All nodes handle traffic simultaneously)

```
                    ┌─────────────────┐
                    │  Load Balancer   │
                    └────────┬────────┘
                    ┌────────┴────────┐
               ┌────▼────┐      ┌────▼────┐
               │ Server 1 │      │ Server 2 │
               │ (Active) │      │ (Active) │
               └────┬─────┘      └────┬─────┘
                    │                  │
                    └────────┬─────────┘
                        ┌────▼────┐
                        │Database │
                        │ Cluster │
                        └─────────┘

Both servers handle traffic at the same time.
If Server 1 fails → Server 2 handles ALL traffic.
No downtime at all! ✅
```

**Pros:**
- Zero downtime during failures
- Better resource utilization (both servers working)
- Better performance (traffic is distributed)

**Cons:**
- More complex to set up
- Need to handle data consistency
- State management can be tricky

#### Active-Passive (Standby node takes over on failure)

```
                    ┌─────────────────┐
                    │  Load Balancer   │
                    └────────┬────────┘
                             │
               ┌─────────────┴──────────────┐
          ┌────▼────┐                  ┌────▼────┐
          │ Server 1 │                  │ Server 2 │
          │ (Active) │ ──── Heartbeat ──│(Standby) │
          │ Handles  │                  │  Idle    │
          │ traffic  │                  │  Ready   │
          └──────────┘                  └──────────┘

Server 2 sits idle, monitoring Server 1.
If Server 1 fails → Server 2 takes over.
Small downtime during switchover (seconds to minutes).
```

**Pros:**
- Simpler than active-active
- No data consistency issues
- Easy to understand

**Cons:**
- Wasted resources (standby server is idle)
- Some downtime during failover
- Standby may have stale data

#### N+1 Redundancy

```
If you need 3 servers to handle normal load,
run 4 servers (3 + 1 spare).

Normal:  [S1] [S2] [S3] [S4]  — Load shared across 4
Failure: [S1] [XX] [S3] [S4]  — 3 remaining handle the load

The "+1" gives you room for 1 failure without degradation.
```

## HA Pattern 2: Load Balancing

Load balancers distribute traffic across multiple servers. If one server fails, traffic goes to the healthy ones.

### Layer 4 vs Layer 7 Load Balancing:

```
Layer 4 (Transport Layer):
  - Works with TCP/UDP
  - Faster, simpler
  - Based on IP + Port
  - Example: AWS NLB, HAProxy (TCP mode)

Layer 7 (Application Layer):
  - Works with HTTP/HTTPS
  - Can make smarter decisions
  - Based on URL path, headers, cookies
  - Example: AWS ALB, Nginx, Traefik
```

### Load Balancing Algorithms:

| Algorithm | How It Works | Best For |
|-----------|-------------|----------|
| **Round Robin** | Requests go to servers in order: 1, 2, 3, 1, 2, 3... | Equal-capacity servers |
| **Weighted Round Robin** | Like Round Robin but some servers get more traffic | Mixed-capacity servers |
| **Least Connections** | Send to server with fewest active connections | Variable-length requests |
| **IP Hash** | Same client IP always goes to same server | Session persistence |
| **Random** | Random server selection | Simple, surprisingly effective |

### Health Checks:

Load balancers need to know which servers are healthy:

```yaml
# Nginx health check example
upstream backend {
    server 10.0.1.10:8080;
    server 10.0.1.11:8080;
    server 10.0.1.12:8080;
}

server {
    location / {
        proxy_pass http://backend;
    }

    # Health check endpoint
    location /health {
        proxy_pass http://backend/health;
    }
}
```

```yaml
# Kubernetes Service (built-in load balancing)
apiVersion: v1
kind: Service
metadata:
  name: my-app
spec:
  selector:
    app: my-app
  ports:
    - port: 80
      targetPort: 8080
  type: ClusterIP
  # K8s automatically load balances across healthy pods
```

### Health Checks in Kubernetes:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
        - name: my-app
          image: my-app:v1.0
          ports:
            - containerPort: 8080
          
          # Liveness Probe: Is the container alive?
          # If this fails, K8s RESTARTS the container
          livenessProbe:
            httpGet:
              path: /healthz
              port: 8080
            initialDelaySeconds: 10
            periodSeconds: 10
            failureThreshold: 3
          
          # Readiness Probe: Is the container ready to serve traffic?
          # If this fails, K8s STOPS sending traffic to it
          readinessProbe:
            httpGet:
              path: /ready
              port: 8080
            initialDelaySeconds: 5
            periodSeconds: 5
            failureThreshold: 3
          
          # Startup Probe: Has the container started?
          # Protects slow-starting containers
          startupProbe:
            httpGet:
              path: /healthz
              port: 8080
            initialDelaySeconds: 0
            periodSeconds: 5
            failureThreshold: 30  # 30 × 5s = 150s max startup time
```

## HA Pattern 3: Database High Availability

Databases are often the hardest part to make highly available because they hold STATE (data).

### Primary-Replica (Master-Slave) Replication:

```
┌────────────────┐     Replication      ┌────────────────┐
│    Primary     │ ──────────────────▶ │    Replica     │
│  (Read/Write)  │     (async/sync)     │  (Read Only)   │
└────────────────┘                      └────────────────┘
        │                                       │
   Writes go here                        Reads can go here
   Reads can also                        (offloads the primary)
   go here

If Primary fails → Promote Replica to Primary
```

#### PostgreSQL Streaming Replication Example:

```bash
# On Primary server (postgresql.conf)
wal_level = replica
max_wal_senders = 5
wal_keep_size = 1024   # Keep 1GB of WAL files for replicas

# On Primary server (pg_hba.conf) — Allow replica to connect
host replication replicator 10.0.1.20/32 md5
```

```bash
# On Replica server — Set up streaming replication
# First, take a base backup from primary
pg_basebackup -h 10.0.1.10 -D /var/lib/postgresql/data -U replicator -P -R

# The -R flag creates standby.signal and sets connection info
# Replica will automatically start streaming WAL from primary
```

### Multi-Primary (Master-Master) Replication:

```
┌────────────────┐     Replication      ┌────────────────┐
│   Primary A    │ ◀────────────────▶ │   Primary B    │
│  (Read/Write)  │   (bi-directional)  │  (Read/Write)  │
└────────────────┘                      └────────────────┘

Both nodes accept reads AND writes.
Changes are replicated in both directions.

⚠️ WARNING: Conflict resolution is HARD!
What happens if both nodes update the same row at the same time?
```

### Database Clustering Solutions:

| Solution | Database | Type | Notes |
|----------|----------|------|-------|
| **PostgreSQL Patroni** | PostgreSQL | Primary-Replica with auto-failover | Most popular for PostgreSQL HA |
| **MySQL Group Replication** | MySQL | Multi-Primary | Built into MySQL 8.0+ |
| **Galera Cluster** | MySQL/MariaDB | Synchronous Multi-Primary | Popular, but adds write latency |
| **MongoDB Replica Set** | MongoDB | Primary-Secondary with auto-failover | Built into MongoDB |
| **Redis Sentinel** | Redis | Primary-Replica with auto-failover | For Redis HA |
| **Redis Cluster** | Redis | Sharded + Replicated | For large-scale Redis |
| **CockroachDB** | CockroachDB | Distributed SQL, built-in HA | NewSQL, auto-sharding |

### PostgreSQL HA with Patroni (Kubernetes):

```yaml
# Simplified Patroni cluster on Kubernetes using Zalando operator
apiVersion: "acid.zalan.do/v1"
kind: postgresql
metadata:
  name: my-app-db
  namespace: production
spec:
  teamId: "my-team"
  volume:
    size: 100Gi
    storageClass: gp3
  numberOfInstances: 3       # 1 primary + 2 replicas
  
  postgresql:
    version: "16"
    parameters:
      max_connections: "200"
      shared_buffers: "2GB"
  
  resources:
    requests:
      cpu: "1"
      memory: 4Gi
    limits:
      cpu: "2"
      memory: 8Gi

  patroni:
    failsafe_mode: true      # Prevent unnecessary failovers
    synchronous_mode: true   # Synchronous replication (no data loss)
```

## HA Pattern 4: Multi-AZ (Availability Zone) Deployment

Cloud providers have **Availability Zones (AZs)** — physically separate data centers within a region.

```
┌─────────── Region: us-east-1 ───────────────────┐
│                                                   │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│  │    AZ-1a      │  │    AZ-1b      │  │    AZ-1c      │
│  │               │  │               │  │               │
│  │  [App-1]      │  │  [App-2]      │  │  [App-3]      │
│  │  [DB-Primary] │  │  [DB-Replica] │  │  [DB-Replica] │
│  │  [Cache-1]    │  │  [Cache-2]    │  │  [Cache-3]    │
│  │               │  │               │  │               │
│  └──────────────┘  └──────────────┘  └──────────────┘
│                                                   │
│  If AZ-1a goes down entirely:                     │
│  → AZ-1b and AZ-1c continue serving traffic      │
│  → DB replica in AZ-1b is promoted to primary     │
│  → Users experience minimal or zero downtime       │
│                                                   │
└───────────────────────────────────────────────────┘
```

### Kubernetes Multi-AZ Deployment:

```yaml
# Pod Anti-Affinity — Spread pods across AZs
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 6
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      topologySpreadConstraints:
        # Spread evenly across AZs
        - maxSkew: 1
          topologyKey: topology.kubernetes.io/zone
          whenUnsatisfiable: DoNotSchedule
          labelSelector:
            matchLabels:
              app: my-app
      
      affinity:
        podAntiAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:
            - weight: 100
              podAffinityTerm:
                labelSelector:
                  matchLabels:
                    app: my-app
                topologyKey: kubernetes.io/hostname
      
      containers:
        - name: my-app
          image: my-app:v1.0
          resources:
            requests:
              cpu: "500m"
              memory: "512Mi"
```

## HA Pattern 5: Circuit Breaker

When a downstream service is failing, keep calling it will just make things worse. A circuit breaker **stops calling** the failing service and returns a fallback response.

```
States of a Circuit Breaker:

CLOSED (normal)     OPEN (tripped)      HALF-OPEN (testing)
      │                    │                    │
  All requests         All requests         Some requests
  go through           are rejected         go through
      │                    │                    │
  If failures          After timeout        If test succeeds
  exceed threshold →   period passes →      → Back to CLOSED
                                            If test fails →
                                            → Back to OPEN

┌──────────┐     Failures exceed      ┌──────────┐
│          │     threshold            │          │
│  CLOSED  │ ──────────────────────▶ │   OPEN   │
│  (OK)    │                          │ (Reject) │
│          │ ◀─────────────────────── │          │
└──────────┘     Test succeeds        └────┬─────┘
      ▲                                     │
      │            Timeout                  │
      │            expires                  ▼
      │                              ┌──────────┐
      └───────────────────────────── │HALF-OPEN │
              Test succeeds          │ (Testing)│
                                     └──────────┘
```

### Why Circuit Breakers Matter:

```
Without Circuit Breaker:
  Service A calls Service B (which is down)
  → Service A waits... and waits... timeout after 30s
  → Service A's threads are all stuck waiting
  → Service A itself becomes unresponsive!
  → Now Service C, which calls Service A, also breaks!
  → CASCADING FAILURE — Everything is down! 💥

With Circuit Breaker:
  Service A calls Service B (which is down)
  → After 5 failures, circuit breaker OPENS
  → Subsequent calls immediately return a fallback response
  → Service A stays healthy!
  → No cascading failure ✅
```

## HA Pattern 6: Graceful Degradation

When part of your system fails, the rest should keep working — even if with reduced functionality.

```
Full System Working:
┌────────────────────────────────────┐
│  [Product Catalog] ✅              │
│  [Search] ✅                       │
│  [Recommendations] ✅             │
│  [Reviews] ✅                      │
│  [Payment] ✅                      │
│  [User Profiles] ✅               │
└────────────────────────────────────┘

Recommendations Service Down:
┌────────────────────────────────────┐
│  [Product Catalog] ✅              │
│  [Search] ✅                       │
│  [Recommendations] ❌ → Show       │
│    "Popular Products" instead      │
│  [Reviews] ✅                      │
│  [Payment] ✅                      │
│  [User Profiles] ✅               │
└────────────────────────────────────┘

The site still works! Users can still browse and buy.
They just don't get personalized recommendations.
```

## 🌍 Disaster Recovery (DR) — Deep Dive

### What is Disaster Recovery?

> **Disaster Recovery is a set of policies, tools, and procedures to recover your technology infrastructure and systems after a catastrophic event.**

### What Kind of Disasters Are We Talking About?

| Disaster Type | Example |
|--------------|---------|
| **Natural Disaster** | Earthquake, flood, hurricane destroys data center |
| **Infrastructure Failure** | Entire cloud region goes offline |
| **Cyber Attack** | Ransomware encrypts all your data |
| **Human Error** | Someone accidentally deletes the production database |
| **Power Outage** | Extended power failure at data center |
| **Software Bug** | A bug corrupts data across the entire system |

---

## DR Strategy Tiers

There are 4 main DR strategies, ranging from cheapest (but slowest recovery) to most expensive (but fastest recovery):

### Tier 1: Backup & Restore (Cold DR)

```
┌──────────────────────────────────────────────────────┐
│                                                      │
│  PRIMARY REGION (us-east-1)     DR REGION (us-west-2)│
│                                                      │
│  ┌─────────┐                    ┌─────────┐         │
│  │   App   │                    │  Empty  │         │
│  │ Servers │                    │  (No    │         │
│  └────┬────┘                    │  infra) │         │
│       │                         └─────────┘         │
│  ┌────▼────┐   Backup to S3    ┌─────────┐         │
│  │Database │ ────────────────▶ │ Backups │         │
│  │         │   (periodic)      │ (S3)    │         │
│  └─────────┘                    └─────────┘         │
│                                                      │
│  When disaster strikes:                              │
│  1. Provision infrastructure in DR region             │
│  2. Restore data from backups                        │
│  3. Deploy applications                              │
│  4. Update DNS to point to DR region                 │
│                                                      │
│  RTO: Hours to Days                                  │
│  RPO: Hours (depends on backup frequency)            │
│  Cost: 💰 (cheapest — just backup storage)           │
│                                                      │
└──────────────────────────────────────────────────────┘
```

### Tier 2: Pilot Light

```
┌──────────────────────────────────────────────────────┐
│                                                      │
│  PRIMARY REGION (us-east-1)     DR REGION (us-west-2)│
│                                                      │
│  ┌─────────┐                    ┌─────────┐         │
│  │   App   │                    │  Empty  │         │
│  │ Servers │                    │ (No app │         │
│  │ (live)  │                    │ servers)│         │
│  └────┬────┘                    └─────────┘         │
│       │                              │              │
│  ┌────▼────┐   Replication     ┌────▼────┐         │
│  │Database │ ────────────────▶ │Database │         │
│  │(Primary)│   (continuous)    │(Replica)│         │
│  └─────────┘                    └─────────┘         │
│                                                      │
│  The "pilot light" = Core data is always replicated  │
│  But app servers are NOT running in DR                │
│                                                      │
│  When disaster strikes:                              │
│  1. Provision/start app servers in DR region          │
│  2. Promote DB replica to primary                    │
│  3. Update DNS to point to DR region                 │
│                                                      │
│  RTO: 10 minutes to 1 hour                           │
│  RPO: Near-zero (continuous replication)             │
│  Cost: 💰💰 (DB replica running, but no app servers) │
│                                                      │
└──────────────────────────────────────────────────────┘
```

### Tier 3: Warm Standby

```
┌──────────────────────────────────────────────────────┐
│                                                      │
│  PRIMARY REGION (us-east-1)     DR REGION (us-west-2)│
│                                                      │
│  ┌─────────┐                    ┌─────────┐         │
│  │   App   │                    │   App   │         │
│  │ Servers │                    │ Servers │         │
│  │(10 pods)│                    │(2 pods) │         │
│  └────┬────┘                    └────┬────┘         │
│       │                              │              │
│  ┌────▼────┐   Replication     ┌────▼────┐         │
│  │Database │ ────────────────▶ │Database │         │
│  │(Primary)│   (continuous)    │(Replica)│         │
│  └─────────┘                    └─────────┘         │
│                                                      │
│  DR region has MINIMUM app servers running           │
│  (scaled down, but functional)                       │
│                                                      │
│  When disaster strikes:                              │
│  1. Scale UP app servers in DR region (2 → 10 pods)  │
│  2. Promote DB replica to primary                    │
│  3. Update DNS to point to DR region                 │
│                                                      │
│  RTO: Minutes                                        │
│  RPO: Near-zero (continuous replication)             │
│  Cost: 💰💰💰 (reduced infra running in DR)          │
│                                                      │
└──────────────────────────────────────────────────────┘
```

### Tier 4: Multi-Region Active-Active (Hot DR)

```
┌──────────────────────────────────────────────────────┐
│                                                      │
│  REGION 1 (us-east-1)          REGION 2 (us-west-2) │
│                                                      │
│  ┌─────────┐                    ┌─────────┐         │
│  │   App   │ ◀── Global DNS ──▶ │   App   │         │
│  │ Servers │    (Route 53)      │ Servers │         │
│  │(10 pods)│                    │(10 pods)│         │
│  └────┬────┘                    └────┬────┘         │
│       │                              │              │
│  ┌────▼────┐   Bi-directional  ┌────▼────┐         │
│  │Database │ ◀──────────────▶ │Database │         │
│  │(Primary)│   replication     │(Primary)│         │
│  └─────────┘                    └─────────┘         │
│                                                      │
│  BOTH regions serve traffic simultaneously!           │
│  Users are routed to the nearest region.              │
│                                                      │
│  When disaster strikes:                              │
│  1. DNS automatically routes ALL traffic to           │
│     the healthy region                               │
│  2. That's it! Nothing else to do! 🎉                │
│                                                      │
│  RTO: Near-zero (automatic failover)                 │
│  RPO: Near-zero (synchronous/near-sync replication)  │
│  Cost: 💰💰💰💰 (full duplicate infrastructure)      │
│                                                      │
└──────────────────────────────────────────────────────┘
```

### DR Strategy Comparison:

| Strategy | RTO | RPO | Cost | Complexity |
|----------|-----|-----|------|------------|
| **Backup & Restore** | Hours to Days | Hours | $ | Low |
| **Pilot Light** | 10 min - 1 hour | Near-zero | $$ | Medium |
| **Warm Standby** | Minutes | Near-zero | $$$ | Medium-High |
| **Active-Active** | Near-zero | Near-zero | $$$$ | High |

### How to Choose:

```
Ask yourself:
1. How critical is this service?
   → Payment system → Active-Active or Warm Standby
   → Internal wiki → Backup & Restore is fine

2. How much downtime can you afford?
   → "No downtime" → Active-Active
   → "Up to 1 hour" → Warm Standby or Pilot Light
   → "A few hours is OK" → Backup & Restore

3. How much money loss per hour of downtime?
   → $100K/hour → Active-Active (the DR cost is worth it)
   → $100/hour → Backup & Restore

4. What does your SLA require?
   → 99.99% → You need at least Warm Standby
   → 99.9% → Pilot Light might be enough
```

## 🛠️ Implementing DR — Practical Examples

### DNS-Based Failover (Route 53):

```
┌──────────┐     Healthy?     ┌──────────────┐
│  Route   │ ───────────────▶ │  Primary     │
│   53     │   Health check   │  us-east-1   │
│  (DNS)   │                  │  IP: 1.2.3.4 │
│          │     Healthy?     └──────────────┘
│          │ ───────────────▶ ┌──────────────┐
│          │   Health check   │  Secondary   │
└──────────┘                  │  us-west-2   │
                              │  IP: 5.6.7.8 │
                              └──────────────┘

If Primary fails health check → Route 53 sends 
all traffic to Secondary automatically.
```

```json
// AWS Route 53 Failover Routing Policy (Terraform)
resource "aws_route53_health_check" "primary" {
  fqdn              = "primary.example.com"
  port               = 443
  type               = "HTTPS"
  resource_path      = "/health"
  failure_threshold  = 3
  request_interval   = 10
}

resource "aws_route53_record" "app" {
  zone_id = aws_route53_zone.main.zone_id
  name    = "app.example.com"
  type    = "A"

  failover_routing_policy {
    type = "PRIMARY"
  }

  set_identifier  = "primary"
  health_check_id = aws_route53_health_check.primary.id
  
  alias {
    name                   = aws_lb.primary.dns_name
    zone_id                = aws_lb.primary.zone_id
    evaluate_target_health = true
  }
}

resource "aws_route53_record" "app_secondary" {
  zone_id = aws_route53_zone.main.zone_id
  name    = "app.example.com"
  type    = "A"

  failover_routing_policy {
    type = "SECONDARY"
  }

  set_identifier = "secondary"

  alias {
    name                   = aws_lb.secondary.dns_name
    zone_id                = aws_lb.secondary.zone_id
    evaluate_target_health = true
  }
}
```

### Cross-Region Database Replication (AWS RDS):

```hcl
# Primary database in us-east-1
resource "aws_db_instance" "primary" {
  identifier     = "myapp-primary"
  engine         = "postgres"
  engine_version = "16.1"
  instance_class = "db.r6g.xlarge"
  
  allocated_storage     = 100
  max_allocated_storage = 500
  storage_encrypted     = true
  
  multi_az             = true    # HA within the region
  backup_retention_period = 7
  
  db_name  = "myapp"
  username = "admin"
  password = var.db_password     # Use Secrets Manager in production!
}

# Cross-region read replica in us-west-2
resource "aws_db_instance" "replica" {
  provider = aws.us-west-2       # Different region!
  
  identifier          = "myapp-replica"
  replicate_source_db = aws_db_instance.primary.arn
  instance_class      = "db.r6g.xlarge"
  
  storage_encrypted = true
  multi_az          = true       # HA within DR region too
  
  # Can be promoted to standalone primary during DR
}
```

## 🔄 DR Testing — Game Days

### Why You MUST Test DR:

> **A disaster recovery plan that hasn't been tested is NOT a plan — it's a hope.**

```
Untested DR plans fail because:
  - Backups might be corrupted (you never checked!)
  - Restore process might take 10x longer than expected
  - Credentials for DR environment might be expired
  - Documentation might be outdated
  - Team might not know the procedures
  - Network configs might have changed since setup
```

### How to Run a DR Test (Game Day):

```yaml
# DR Game Day Plan
name: "DR Game Day — Q1 2025"
date: "2025-03-20"
duration: "4 hours"
scope: "Full DR failover for payment service"

objectives:
  - Validate RTO is under 15 minutes
  - Validate RPO is under 1 minute
  - Verify team knows DR procedures
  - Identify gaps in runbooks

participants:
  - role: Game Day Lead
    person: Alice (SRE Lead)
  - role: Technical Executor
    person: Bob (Senior SRE)
  - role: Observer/Scribe
    person: Charlie (Junior SRE)
  - role: Stakeholder
    person: Diana (Product Manager)

steps:
  1. Pre-checks:
     - Verify DR environment is healthy
     - Verify backups are recent
     - Notify stakeholders
     
  2. Simulate Failure:
     - Option A: Disable primary region (stop primary instances)
     - Option B: Block traffic to primary (network ACL)
     
  3. Execute DR:
     - Follow runbook to failover to DR region
     - Time each step
     - Note any issues or deviations
     
  4. Validate:
     - Verify application works in DR region
     - Run smoke tests
     - Check data integrity
     
  5. Failback:
     - Restore primary region
     - Resync data from DR to primary
     - Switch traffic back to primary
     
  6. Debrief:
     - Document actual RTO and RPO achieved
     - List all issues discovered
     - Create action items for improvements
```

## 📊 HA & DR Summary — Decision Matrix

```
┌──────────────────────────────────────────────────────────────┐
│                  Which Strategy Do You Need?                 │
│                                                              │
```

## 📚 Key Takeaways

```
✅ HA = Survive component failures (within a region)
✅ DR = Recover from catastrophic failures (across regions)
✅ Never have a Single Point of Failure (SPOF)
✅ Active-Active = Best availability, highest cost
✅ Active-Passive = Simpler, some downtime during failover
✅ Multi-AZ deployment is the MINIMUM for production workloads
✅ Circuit breakers prevent cascading failures
✅ Graceful degradation = Keep partial functionality when components fail
✅ DR has 4 tiers: Backup/Restore → Pilot Light → Warm Standby → Active-Active
✅ RTO = How fast you recover, RPO = How much data you lose
✅ ALWAYS test your DR plan (Game Days!)
✅ An untested DR plan is not a plan — it's a hope
```
