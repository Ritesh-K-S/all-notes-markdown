# 23.3 — Backstage — Developer Portal by Spotify

> **One Line Answer:** Backstage is an **open-source developer portal platform** originally built by Spotify, now a **CNCF Incubating project**, that helps organizations build a **single pane of glass** for all their services, documentation, infrastructure, and developer tools — it's the most popular foundation for building an Internal Developer Platform.

---

## 🤔 Let's Start With a Story — Spotify's Problem

It's 2016 at Spotify. The company has grown to over **1,600 engineers** working on more than **1,000 microservices**.

Everything looks great from the outside — Spotify is the world's most popular music streaming service. But inside, developers are struggling:

```
Developer A: "I need to call the playlist service API. Who owns it? 
              Where are the docs? Is it still maintained?"
              → Spends 2 hours searching Slack, wikis, and asking around

Developer B: "I'm new and I need to deploy my first service. 
              Where do I start? Which template? What tools?"
              → Spends 3 DAYS figuring out the deployment workflow

Developer C: "There's a bug in production. Which team owns this service? 
              How do I reach them? What's their on-call rotation?"
              → Searches 5 different tools to find the answer

Developer D: "I built something similar to what Team X already has. 
              Nobody told me they had already built it."
              → Duplicate work, wasted effort
```

Spotify's engineering leadership realized they had a **discovery problem** and a **developer experience problem**.

The solution? They built **Backstage** — a single place where developers can find everything they need, create new services from templates, and manage the entire software lifecycle.

It was so successful internally that in 2020, **Spotify open-sourced it**. In 2022, it became a **CNCF Incubating Project**, joining the same ecosystem as Kubernetes, Prometheus, and Helm.

---

## ✅ What Exactly is Backstage?

Backstage is an **open-source platform** for building **developer portals**. It provides a framework and a set of core features that you can customize and extend to create your own Internal Developer Platform.

### Think of Backstage Like This:

```
Backstage is to Developer Portals 
    what WordPress is to Websites

WordPress gives you:
    → A framework for building websites
    → Themes to customize the look
    → Plugins to add features
    → You build YOUR website on top of it

Backstage gives you:
    → A framework for building developer portals
    → A UI shell to customize
    → Plugins to add features (100+ available)
    → You build YOUR developer portal on top of it
```

### What Backstage is NOT:

| Backstage is NOT... | It IS... |
|---------------------|----------|
| A ready-to-use product you install and it works | A framework you build YOUR portal on top of |
| A CI/CD tool | A portal that can integrate WITH your CI/CD tools |
| A replacement for Kubernetes, Terraform, etc. | A UI layer that sits on TOP of these tools |
| A SaaS service (by default) | Self-hosted, open-source software |
| Something you deploy in 5 minutes | Something that requires setup, customization, and maintenance |

## 📋 Feature 1: Software Catalog

The **Software Catalog** is the heart of Backstage. It's a **centralized registry** of all software in your organization — services, websites, libraries, data pipelines, ML models, infrastructure components — everything.

### Why It Matters:

```
Without a catalog:
    "Does anyone know who owns the payment service?"      → 😕 Ask around
    "Is there an existing notification library?"          → 🤷 Nobody knows
    "What depends on the user-auth service?"              → 😰 No idea

With Backstage catalog:
    Search "payment" → Found! Owner: Team Payments. Docs: linked. API: documented.
    Everything discoverable in ONE place. ✅
```

### How It Works:

Every team registers their software components by creating a **catalog-info.yaml** file in their Git repository:

```yaml
# catalog-info.yaml — placed in the ROOT of your Git repository
# This file tells Backstage about your service

apiVersion: backstage.io/v1alpha1
kind: Component
metadata:
  name: user-auth-service
  description: |
    Handles user authentication and authorization using JWT tokens.
    Supports OAuth2, SAML, and local authentication.
  labels:
    tier: critical
  annotations:
    # These annotations connect Backstage to external tools
    github.com/project-slug: mycompany/user-auth-service
    backstage.io/techdocs-ref: dir:.
    pagerduty.com/service-id: PABC123
    grafana/dashboard-url: https://grafana.internal/d/auth-dashboard
    sonarqube.org/project-key: user-auth-service
  tags:
    - python
    - fastapi
    - authentication
    - jwt
  links:
    - url: https://grafana.internal/d/auth-dashboard
      title: Grafana Dashboard
      icon: dashboard
    - url: https://runbooks.internal/auth-service
      title: Runbook
      icon: docs
spec:
  type: service              # Type: service, website, library, etc.
  lifecycle: production       # production, experimental, deprecated
  owner: team-identity        # Which team owns this (links to a Group entity)
  system: user-management     # Which system this belongs to
  providesApis:
    - user-auth-api           # APIs this service provides
  consumesApis:
    - email-notification-api  # APIs this service depends on
  dependsOn:
    - resource:postgresql-auth-db    # Database dependency
    - component:redis-session-cache  # Cache dependency
```

### Entity Kinds in Backstage:

Backstage supports different **kinds** of entities:

| Kind | What It Represents | Example |
|------|-------------------|---------|
| **Component** | A piece of software (service, website, library) | `user-auth-service` |
| **API** | An API interface (REST, gRPC, GraphQL, async) | `user-auth-api` |
| **Resource** | Infrastructure (database, S3 bucket, message queue) | `postgresql-auth-db` |
| **System** | A collection of components that work together | `user-management-system` |
| **Domain** | A business domain grouping | `identity-domain` |
| **Group** | A team or organizational unit | `team-identity` |
| **User** | An individual person | `john.doe` |
| **Location** | A pointer to where catalog files can be found | `github.com/myorg/*/catalog-info.yaml` |

### The Entity Hierarchy:

```

    └── System: "User Management"
            ├── Component: "user-auth-service" (service)
            │       ├── Provides: "user-auth-api" (API)

            │       └── Depends on: "redis-session-cache" (component)
            │
            ├── Component: "user-profile-service" (service)
            │       ├── Provides: "user-profile-api" (API)
            │       └── Consumes: "user-auth-api" (API)
            │
            └── Component: "user-management-frontend" (website)
                    └── Consumes: "user-profile-api" (API)

Owned by: Group: "team-identity"
    ├── User: "alice@company.com" (Tech Lead)
    ├── User: "bob@company.com" (Senior Dev)
    └── User: "charlie@company.com" (SRE)
```

## 🧰 Feature 2: Software Templates (Scaffolder)

The **Scaffolder** is Backstage's built-in templating engine. It lets you create **Software Templates** — pre-built blueprints for creating new projects, services, or resources.

### Why It Matters:

```
Without templates:
    Developer: "I need to create a new microservice."
    → Creates a Git repo manually
    → Copies Dockerfile from another project (probably outdated)
    → Writes Kubernetes YAML from scratch (probably wrong)
    → Sets up CI/CD manually (probably missing security scans)
    → Time: 2-5 days. Quality: inconsistent. Security: questionable.

With Backstage templates:
    Developer: Click "Create" → Select "Node.js Microservice" → Fill form → Click "Create"
    → Everything generated automatically with best practices
    → Time: 5 minutes. Quality: standardized. Security: built-in. ✅
```

### How Templates Work:

A template is defined using a YAML file with these parts:

```yaml
# template.yaml — Defines a Software Template in Backstage

apiVersion: scaffolder.backstage.io/v1beta3
kind: Template
metadata:
  name: nodejs-microservice
  title: Node.js Microservice
  description: |
    Creates a new Node.js microservice with Express, Docker, 
    Kubernetes manifests, CI/CD pipeline, and monitoring.
  tags:
    - recommended
    - nodejs
    - microservice
spec:
  owner: platform-team
  type: service

  # ──────────────────────────────────────────────
  # STEP 1: PARAMETERS — What the developer fills in
  # ──────────────────────────────────────────────
  parameters:
    - title: Service Information
      required:
        - name
        - description
        - owner
      properties:
        name:
          title: Service Name
          type: string
          description: Unique name for the service (lowercase, hyphens only)
          pattern: '^[a-z][a-z0-9-]*$'
          ui:autofocus: true
        description:
          title: Description
          type: string
          description: A short description of what this service does
        owner:
          title: Owner Team
          type: string
          description: The team that will own this service
          ui:field: OwnerPicker
          ui:options:
            catalogFilter:
              kind: Group

    - title: Infrastructure Options
      properties:
        database:
          title: Database
          type: string
          description: Select a database if needed
          enum:
            - none
            - postgresql
            - mongodb
            - redis
          default: none
        environment:
          title: Initial Environment
          type: string
          enum:
            - staging
            - production
          default: staging
        port:
          title: Application Port
          type: number
          default: 3000

  # ──────────────────────────────────────────────
  # STEP 2: STEPS — What happens when they click "Create"
  # ──────────────────────────────────────────────
  steps:
    # Step 1: Generate code from skeleton template
    - id: fetch-skeleton
      name: Fetch Skeleton
      action: fetch:template
      input:
        url: ./skeleton     # Path to template files
        values:
          name: ${{ parameters.name }}
          description: ${{ parameters.description }}
          owner: ${{ parameters.owner }}
          port: ${{ parameters.port }}
          database: ${{ parameters.database }}

    # Step 2: Create GitHub repository
    - id: publish-github
      name: Create GitHub Repository
      action: publish:github
      input:
        allowedHosts: ['github.com']
        repoUrl: github.com?owner=mycompany&repo=${{ parameters.name }}
        description: ${{ parameters.description }}
        defaultBranch: main
        repoVisibility: internal
        protectDefaultBranch: true

    # Step 3: Register in Backstage catalog
    - id: register-catalog
      name: Register in Catalog
      action: catalog:register
      input:
        repoContentsUrl: ${{ steps['publish-github'].output.repoContentsUrl }}
        catalogInfoPath: /catalog-info.yaml

    # Step 4: Create Kubernetes namespace
    - id: create-namespace
      name: Create Kubernetes Namespace
      action: http:backstage:request
      input:
        method: POST
        path: /api/kubernetes/namespaces
        body:
          name: ${{ parameters.name }}
          team: ${{ parameters.owner }}

    # Step 5: Provision database (if selected)
    - id: provision-database
      name: Provision Database
      if: ${{ parameters.database !== 'none' }}
      action: http:backstage:request
      input:
        method: POST
        path: /api/infrastructure/databases
        body:
          name: ${{ parameters.name }}-db
          engine: ${{ parameters.database }}
          environment: ${{ parameters.environment }}

  # ──────────────────────────────────────────────
  # STEP 3: OUTPUT — Links shown after creation
  # ──────────────────────────────────────────────
  output:
    links:
      - title: Repository
        url: ${{ steps['publish-github'].output.remoteUrl }}
      - title: Open in Catalog
        icon: catalog
        entityRef: ${{ steps['register-catalog'].output.entityRef }}
      - title: CI/CD Pipeline
        url: ${{ steps['publish-github'].output.remoteUrl }}/actions
```

## 📖 Feature 3: TechDocs — Documentation That Lives With Code

**TechDocs** is Backstage's built-in documentation system. It follows the **"docs-like-code"** approach — documentation is written in Markdown, stored alongside your code, and automatically rendered in Backstage.

### Why It Matters:

```
Without TechDocs:
    → Documentation scattered across Confluence, Google Docs, wikis, READMEs
    → Docs are always outdated (nobody updates them)
    → New developers can't find anything
    → "The docs for that service? I think they're in some Confluence space... maybe."

With TechDocs:
    → Docs written in Markdown next to your code (in the same Git repo)
    → Automatically rendered and searchable in Backstage
    → Docs update when code updates (same PR = code + docs)
    → Everything discoverable from the service catalog
```

### How to Set Up TechDocs:

**Step 1:** Add an `mkdocs.yml` file to your service repository:

```yaml
# mkdocs.yml — in the root of your service repository
site_name: User Auth Service Documentation
nav:
  - Home: index.md
  - Getting Started: getting-started.md
  - API Reference: api-reference.md
  - Architecture: architecture.md
  - Runbook: runbook.md
  - FAQ: faq.md

plugins:
  - techdocs-core
```

**Step 2:** Create your docs in a `docs/` directory:

```
my-service/
├── src/                    # Application code
├── docs/                   # Documentation (Markdown)
│   ├── index.md            # Home page
│   ├── getting-started.md  # How to get started
│   ├── api-reference.md    # API documentation
│   ├── architecture.md     # Architecture decisions
│   └── runbook.md          # Operational runbook
├── mkdocs.yml              # MkDocs configuration
└── catalog-info.yaml       # Backstage catalog entry
```

**Step 3:** Add the TechDocs annotation to your `catalog-info.yaml`:

```yaml
metadata:
  annotations:
    backstage.io/techdocs-ref: dir:.   # Tells Backstage where to find docs
```

Now when anyone visits the `user-auth-service` in Backstage and clicks the **Docs** tab, they see beautifully rendered documentation — always up-to-date!

## 🚀 Getting Started — Setting Up Backstage (Step by Step)

### Prerequisites:

```
Node.js 18 or 20 (LTS)
yarn (package manager)
Docker (for running dependencies)
Git
PostgreSQL (for production; SQLite for development)
```

### Step 1: Create a New Backstage App

```bash
# Create a new Backstage application
npx @backstage/create-app@latest

# You'll be prompted:
? Enter a name for the app [my-backstage-app]

# This generates the project structure
```

### Step 2: Understand the Project Structure

```
my-backstage-app/
├── app-config.yaml          # Main configuration file
├── app-config.local.yaml    # Local overrides (gitignored)
├── catalog-info.yaml        # Backstage's own catalog entry
├── package.json
├── packages/
│   ├── app/                 # Frontend (React)
│   │   ├── src/
│   │   │   ├── App.tsx      # Main app component
│   │   │   └── components/  # Custom components
│   │   └── package.json
│   └── backend/             # Backend (Node.js)
│       ├── src/
│       │   └── index.ts     # Backend entry point
│       └── package.json
├── plugins/                 # Custom plugins (your own)
└── examples/                # Example catalog entities
```

### Step 3: Configure Backstage

The main configuration is in `app-config.yaml`:

```yaml
# app-config.yaml

app:
  title: My Company Developer Portal
  baseUrl: http://localhost:3000

organization:
  name: My Company

backend:
  baseUrl: http://localhost:7007
  database:
    client: pg                          # PostgreSQL for production
    connection:
      host: ${POSTGRES_HOST}
      port: ${POSTGRES_PORT}
      user: ${POSTGRES_USER}
      password: ${POSTGRES_PASSWORD}

# ──────────────────────────────────────────
# Authentication (GitHub example)
# ──────────────────────────────────────────
auth:
  environment: development
  providers:
    github:
      development:
        clientId: ${AUTH_GITHUB_CLIENT_ID}
        clientSecret: ${AUTH_GITHUB_CLIENT_SECRET}

# ──────────────────────────────────────────
# Integrations (connects to GitHub for catalog & templates)
# ──────────────────────────────────────────
integrations:
  github:
    - host: github.com
      token: ${GITHUB_TOKEN}

# ──────────────────────────────────────────
# Catalog — Where to find catalog entities
# ──────────────────────────────────────────
catalog:
  import:
    entityFilename: catalog-info.yaml
    pullRequestBranchName: backstage-integration
  rules:
    - allow: [Component, System, API, Resource, Location, Domain, Group, User]
  locations:
    # Example: Register all repos in your GitHub org
    - type: github-discovery
      target: https://github.com/mycompany/*/blob/main/catalog-info.yaml

    # Example: Register specific templates
    - type: file
      target: ../../examples/template/template.yaml
      rules:
        - allow: [Template]

# ──────────────────────────────────────────
# TechDocs configuration
# ──────────────────────────────────────────
techdocs:
  builder: 'local'        # or 'external' for CI/CD-built docs
  generator:
    runIn: 'local'         # or 'docker'
  publisher:
    type: 'local'          # or 'awsS3', 'googleGcs', 'azureBlobStorage'

# ──────────────────────────────────────────
# Kubernetes plugin configuration
# ──────────────────────────────────────────
kubernetes:
  serviceLocatorMethod:
    type: 'multiTenant'
  clusterLocatorMethods:
    - type: 'config'
      clusters:
        - url: https://k8s-cluster.internal
          name: production
          authProvider: 'serviceAccount'
          serviceAccountToken: ${K8S_SERVICE_ACCOUNT_TOKEN}
```

### Step 4: Start Backstage

```bash
# Start in development mode
yarn dev

# This starts:
# → Frontend on http://localhost:3000
# → Backend on http://localhost:7007
```

### Step 5: Register Your First Service

Create a `catalog-info.yaml` in one of your existing repos:

```yaml
apiVersion: backstage.io/v1alpha1
kind: Component
metadata:
  name: my-first-service
  description: My first service registered in Backstage
spec:
  type: service
  lifecycle: production
  owner: my-team
```

Then register it in Backstage:
1. Go to `http://localhost:3000`
2. Click "Register Existing Component"
3. Enter the URL of your `catalog-info.yaml`
4. Click "Analyze" → "Import"

Your service now appears in the catalog! 🎉

## 🆚 Backstage vs Alternatives

Backstage isn't the only developer portal option. Here's how it compares:

| Feature | Backstage (Spotify) | Port | Cortex | OpsLevel | Compass (Atlassian) |
|---------|---------------------|------|--------|----------|---------------------|
| **Type** | Open-source | Commercial | Commercial | Commercial | Commercial |
| **Cost** | Free (self-hosted) | Paid SaaS | Paid SaaS | Paid SaaS | Paid SaaS |
| **Hosting** | Self-hosted | SaaS | SaaS | SaaS | SaaS |
| **Customization** | Extremely high (plugins) | High (no-code) | Medium | Medium | Medium |
| **Setup Effort** | High (DIY) | Low | Low | Low | Low |
| **Maintenance** | You maintain it | Vendor maintains | Vendor maintains | Vendor maintains | Vendor maintains |
| **Community** | Huge (CNCF) | Growing | Growing | Growing | Growing |
| **Plugin Ecosystem** | 100+ plugins | Built-in integrations | Built-in | Built-in | Atlassian ecosystem |
| **Best For** | Large orgs with engineering capacity | Orgs wanting quick setup | Orgs focused on reliability | Orgs focused on ownership | Atlassian shops |

### When to Choose Backstage:

```
✅ Choose Backstage when:
    → You have engineering capacity to set up and maintain it
    → You want full customization and control
    → You're already in the CNCF ecosystem
    → You don't want vendor lock-in
    → You have 50+ developers (worth the investment)
    → You want to build custom plugins for your specific needs

❌ Consider alternatives when:
    → You're a small team (< 20 developers) — setup overhead may not be worth it
    → You don't have engineers to maintain it
    → You want something working in hours, not weeks
    → You prefer SaaS over self-hosted
```

## ⚠️ Common Challenges with Backstage

### Challenge 1: Initial Setup Complexity

```
Problem: Backstage is a framework, not a product. 
         Setting it up takes real engineering effort.

Solution: 
    → Start with a minimal setup (catalog + one template)
    → Add plugins incrementally
    → Assign a dedicated engineer for the first month
    → Use the Backstage community and docs (they're excellent)
```

### Challenge 2: Keeping the Catalog Up-to-Date

```
Problem: If teams don't maintain their catalog-info.yaml, data gets stale.

Solution:
    → Make catalog-info.yaml part of the service creation template (automated)
    → Use automated discovery (GitHub org discovery, K8s discovery)
    → Add catalog completeness checks to CI/CD pipelines
    → Show "catalog health score" on dashboards
```

### Challenge 3: Plugin Compatibility and Upgrades

```
Problem: Backstage moves fast. Plugins can break during upgrades.

Solution:
    → Pin Backstage versions
    → Test upgrades in staging first
    → Use the official upgrade helper tool
    → Don't install plugins you don't actively use
```

### Challenge 4: Adoption — Getting Developers to Use It

```
Problem: "We built it, but nobody uses it."

Solution:
    → Make it the default way to create services (golden path)
    → Show clear value (save time, find info faster)
    → Get buy-in from team leads first
    → Make it the homepage in new developer onboarding
    → Celebrate early adopters
```

## 📝 Key Takeaways

```
✅ Backstage is an open-source framework for building developer portals (not a ready product)
✅ Originally built by Spotify, now a CNCF Incubating project
✅ Three core features: Software Catalog, Software Templates (Scaffolder), TechDocs

✅ Best for medium-to-large organizations (50+ developers)
```
