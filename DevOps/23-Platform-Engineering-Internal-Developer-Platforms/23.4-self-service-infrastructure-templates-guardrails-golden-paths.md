# 23.4 — Self-Service Infrastructure — Templates, Guardrails & Golden Paths

> **One Line Answer:** Self-service infrastructure means developers can **provision the infrastructure they need** (databases, clusters, storage, environments) **by themselves**, without filing a ticket and waiting for the ops team — all while staying within **safe boundaries (guardrails)** set by the platform team, and following **recommended best-practice paths (golden paths)** that make the "right way" the "easy way."

## 🤔 The Problem: Why Do We Need Self-Service?

Let's paint a very common picture that happens in most companies:

**Without Self-Service:**

```
Developer: "Hey Ops team, I need a PostgreSQL database for my new microservice."

Ops Team:   "Sure, let us file a ticket..."

             ⏳ 3 days later...

Ops Team:   "What size do you need? What region? What backup policy?"

Developer: "Ummm... not sure. Standard stuff?"

             ⏳ 5 more days later...

Ops Team:   "Done! Here are the credentials."

Developer: "Thanks! But I actually needed a Redis cache too..."

             😩 Repeat the whole process again.
```

**Total time wasted:** 1–2 weeks. Developer is blocked. Ops is overwhelmed. Everyone is frustrated.

**With Self-Service:**

```
Developer: Opens the developer portal → Clicks "New Database" 
           → Picks PostgreSQL, selects "small", clicks "Create"

             ⏳ 5 minutes later...

Platform:  "Here's your database! Connection string is in your namespace's Secret.
            Backups are automatic. Monitoring is pre-configured."

Developer: "That was easy. Let me also spin up a Redis cache..."
           → Clicks "New Cache" → Done in 2 minutes.
```

**Total time:** 10 minutes. No tickets. No waiting. No back-and-forth.

**That's the power of self-service infrastructure.**

## 📖 Key Concepts — The Three Pillars

Self-service infrastructure is built on three pillars:

| Pillar | What It Means | Analogy |
|--------|---------------|---------|
| **Templates** | Pre-built, reusable blueprints for infrastructure | Like a **menu** at a restaurant — pick what you want |
| **Guardrails** | Rules and limits that prevent bad things | Like **lane markings** on a road — you can drive freely but can't go off the road |
| **Golden Paths** | The recommended, best-practice way to do things | Like a **GPS route** — you CAN take any road, but this one is the fastest and safest |

Let's dive deep into each one.

## 🧩 Pillar 1: Templates — Pre-Built Blueprints

### What Are Templates?

Templates are **pre-defined, parameterized infrastructure blueprints** that developers can use to create resources without writing IaC code from scratch.

Think of it like ordering from a menu:

```
🍕 Infrastructure Menu:

1. PostgreSQL Database
   - Size: [small / medium / large]
   - Region: [us-east-1 / eu-west-1 / ap-south-1]
   - Backup: [daily / hourly]

2. Redis Cache
   - Size: [small / medium / large]
   - Eviction Policy: [allkeys-lru / volatile-ttl]

3. S3 Bucket
   - Access: [private / public-read]
   - Versioning: [enabled / disabled]

4. Kubernetes Namespace with RBAC
   - Team: [frontend / backend / data]
   - Environment: [dev / staging / prod]
```

Developers just pick from the menu and fill in a few parameters. The template handles everything else — networking, security, monitoring, tagging, compliance.

---

### How Do Templates Work?

Here's the general flow:

```
Developer                      Platform Team                    Cloud/Kubernetes
    |                               |                                |
    | 1. Picks template             |                                |
    |   (e.g., "PostgreSQL small")  |                                |
    |------------------------------>|                                |
    |                               | 2. Template fills in           |
    |                               |    default values, applies     |
    |                               |    company standards           |
    |                               |------------------------------->|
    |                               |                                | 3. Provisions the
    |                               |                                |    actual resource
    |                               |                                |
    | 4. Gets back connection info  |<-------------------------------|
    |<------------------------------|                                |
```

### Example: Terraform Module as a Template

The platform team creates a **Terraform module** that becomes a template:

**`modules/postgresql/main.tf`** (Created by Platform Team)

```hcl
# This module is the "template" — it hides all the complexity
# Developers just pass in a few simple parameters

variable "name" {
  description = "Name of the database"
  type        = string
}

variable "size" {
  description = "Size of the database"
  type        = string
  default     = "small"

  validation {
    condition     = contains(["small", "medium", "large"], var.size)
    error_message = "Size must be small, medium, or large."
  }
  # ☝️ This is a GUARDRAIL — developers can't pick "xxxl-super-expensive"
}

variable "environment" {
  description = "Environment (dev, staging, prod)"
  type        = string
}

# Map sizes to actual instance types — developers don't need to know this!
locals {
  size_map = {
    small  = "db.t3.micro"    # 2 vCPU, 1 GB RAM  — ~$15/month
    medium = "db.t3.medium"   # 2 vCPU, 4 GB RAM  — ~$70/month
    large  = "db.r5.large"    # 2 vCPU, 16 GB RAM — ~$200/month
  }
}

resource "aws_db_instance" "this" {
  identifier     = "${var.name}-${var.environment}"
  engine         = "postgres"
  engine_version = "15.4"                          # ← Standardized version
  instance_class = local.size_map[var.size]

  allocated_storage     = 20
  max_allocated_storage = 100                       # ← Auto-scaling enabled
  storage_encrypted     = true                      # ← Always encrypted (guardrail)

  db_name  = var.name
  username = "app_user"
  password = random_password.db_password.result

  # Security best practices baked in
  publicly_accessible    = false                    # ← Never public (guardrail)
  deletion_protection    = var.environment == "prod" # ← Prod can't be deleted accidentally
  skip_final_snapshot    = var.environment != "prod"
  backup_retention_period = var.environment == "prod" ? 30 : 7

  # Monitoring baked in
  performance_insights_enabled = true
  monitoring_interval          = 60

  # Standard tags — always applied
  tags = {
    Name        = var.name
    Environment = var.environment
    ManagedBy   = "platform-team"
    Template    = "postgresql-v2"
  }
}

# Auto-generate a secure password
resource "random_password" "db_password" {
  length  = 32
  special = true
}

# Store the password in AWS Secrets Manager — not in plain text!
resource "aws_secretsmanager_secret" "db_password" {
  name = "${var.name}-${var.environment}-db-password"
}

resource "aws_secretsmanager_secret_version" "db_password" {
  secret_id     = aws_secretsmanager_secret.db_password.id
  secret_string = random_password.db_password.result
}

# Outputs that the developer needs
output "endpoint" {
  value = aws_db_instance.this.endpoint
}

output "secret_arn" {
  value       = aws_secretsmanager_secret.db_password.arn
  description = "ARN of the secret containing the database password"
}
```

**What the developer writes** (using the template):

```hcl
# Developer's code — super simple!
module "my_database" {
  source      = "git::https://github.com/mycompany/platform-modules//postgresql"
  name        = "user-service-db"
  size        = "small"
  environment = "dev"
}

# That's it! 4 lines! 🎉
# The template handles encryption, backups, monitoring, security, tagging...
```

**See the difference?** The developer writes 4 lines. The template handles 60+ lines of best practices behind the scenes.

## 🛡️ Pillar 2: Guardrails — Safe Boundaries

### What Are Guardrails?

Guardrails are **automated rules and policies** that prevent developers from doing things that could cause problems — without blocking them or slowing them down.

The key idea is:

> **Freedom within boundaries.**

Developers are free to provision and deploy whatever they need, but guardrails ensure they can't accidentally:
- Create a $50,000/month database instance
- Deploy to production without tests passing
- Make an S3 bucket publicly accessible
- Use an outdated, vulnerable base image
- Skip encryption for sensitive data
- Deploy without proper tagging

---

### Types of Guardrails

| Type | What It Does | Example |
|------|-------------|---------|
| **Cost Guardrails** | Prevent overspending | Max instance size = `large`, budget alerts |
| **Security Guardrails** | Enforce security standards | Encryption required, no public access |
| **Compliance Guardrails** | Meet regulatory requirements | Data residency, audit logging, retention |
| **Operational Guardrails** | Prevent outages | Must have health checks, resource limits required |
| **Naming/Tagging Guardrails** | Ensure consistency | Standard naming convention, required tags |

---

### Implementing Guardrails — Real Examples

#### 1. Terraform Validation (Preventive Guardrail)

```hcl
variable "instance_type" {
  type = string

  validation {
    condition = contains([
      "t3.micro", "t3.small", "t3.medium",   # Allowed types
      "m5.large", "m5.xlarge"                 # Nothing bigger!
    ], var.instance_type)
    error_message = "Instance type not allowed. Contact platform team for exceptions."
  }
}

variable "environment" {
  type = string

  validation {
    condition     = contains(["dev", "staging", "prod"], var.environment)
    error_message = "Environment must be dev, staging, or prod."
  }
}
```

#### 2. OPA/Gatekeeper Policy (Kubernetes Guardrail)

This prevents anyone from deploying containers running as root:

```yaml
# ConstraintTemplate — defines the rule
apiVersion: templates.gatekeeper.sh/v1
kind: ConstraintTemplate
metadata:
  name: k8s-no-root-containers
spec:
  crd:
    spec:
      names:
        kind: K8sNoRootContainers
  targets:
    - target: admission.k8s.gatekeeper.sh
      rego: |
        package k8snorootcontainers

        violation[{"msg": msg}] {
          container := input.review.object.spec.containers[_]
          container.securityContext.runAsUser == 0
          msg := sprintf("Container '%v' is running as root (UID 0). This is not allowed.", [container.name])
        }

        violation[{"msg": msg}] {
          container := input.review.object.spec.containers[_]
          not container.securityContext.runAsNonRoot
          msg := sprintf("Container '%v' must set securityContext.runAsNonRoot = true", [container.name])
        }
---
# Constraint — applies the rule
apiVersion: constraints.gatekeeper.sh/v1beta1
kind: K8sNoRootContainers
metadata:
  name: no-root-containers
spec:
  match:
    kinds:
      - apiGroups: [""]
        kinds: ["Pod"]
    excludedNamespaces:
      - kube-system       # System namespaces are excluded
      - gatekeeper-system
```

**What happens when someone tries to deploy a root container?**

```bash
$ kubectl apply -f my-pod.yaml

Error from server (Forbidden): admission webhook "validation.gatekeeper.sh" denied the request:
  Container 'nginx' must set securityContext.runAsNonRoot = true
```

**The guardrail stopped the bad deployment automatically!** No human reviewer needed.

#### 3. Kyverno Policy (Another Kubernetes Guardrail)

This enforces that all deployments MUST have resource limits:

```yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: require-resource-limits
  annotations:
    policies.kyverno.io/title: Require Resource Limits
    policies.kyverno.io/description: >-
      Every container must define CPU and memory limits.
      This prevents any single app from consuming all cluster resources.
spec:
  validationFailureAction: Enforce    # Block the deployment if it fails
  background: true
  rules:
    - name: check-resource-limits
      match:
        any:
          - resources:
              kinds:
                - Pod
      validate:
        message: >-
          All containers must have CPU and memory limits defined.
          Example:
            resources:
              limits:
                cpu: "500m"
                memory: "512Mi"
        pattern:
          spec:
            containers:
              - resources:
                  limits:
                    memory: "?*"    # Must not be empty
                    cpu: "?*"       # Must not be empty
```

#### 4. AWS Service Control Policy (Cloud Account Guardrail)

This prevents anyone in the organization from creating resources in unapproved regions:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyUnapprovedRegions",
      "Effect": "Deny",
      "Action": "*",
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "aws:RequestedRegion": [
            "us-east-1",
            "eu-west-1",
            "ap-south-1"
          ]
        }
      }
    }
  ]
}
```

Even if someone has admin access, they literally **cannot** create resources in unapproved regions. That's a guardrail at the organizational level.

#### 5. CI/CD Pipeline Guardrails

```yaml
# GitHub Actions workflow with guardrails
name: Deploy

on:
  push:
    branches: [main]

jobs:
  guardrails:
    runs-on: ubuntu-latest
    steps:
      # Guardrail 1: Security scan must pass
      - name: Run Trivy Security Scan
        uses: aquasecurity/trivy-action@master
        with:
          scan-type: 'fs'
          exit-code: '1'               # ← Fail the pipeline if vulnerabilities found

      # Guardrail 2: Infrastructure cost check
      - name: Infracost Cost Estimate
        uses: infracost/actions/setup@v2
      - run: |
          infracost breakdown --path=. --format=json --out-file=/tmp/infracost.json
          MONTHLY_COST=$(jq -r '.totalMonthlyCost' /tmp/infracost.json)
          if (( $(echo "$MONTHLY_COST > 500" | bc -l) )); then
            echo "❌ Monthly cost ($MONTHLY_COST) exceeds $500 limit!"
            echo "Please get approval from the platform team."
            exit 1
          fi

      # Guardrail 3: Policy check
      - name: Checkov Policy Scan
        uses: bridgecrewio/checkov-action@master
        with:
          directory: .
          framework: terraform
          soft_fail: false             # ← Fail the pipeline if policy violations found

  deploy:
    needs: guardrails                   # ← Can't deploy unless guardrails pass!
    runs-on: ubuntu-latest
    steps:
      - name: Deploy Infrastructure
        run: terraform apply -auto-approve
```

---

### Guardrails Philosophy — Soft vs Hard

| Type | Behavior | When to Use |
|------|----------|-------------|
| **Hard Guardrail** | **Blocks** the action completely | Security requirements, compliance, cost limits |
| **Soft Guardrail** | **Warns** but allows the action | Best practices, recommendations, non-critical standards |

```
Hard Guardrail:    ❌ BLOCKED — "You cannot deploy without encryption."
Soft Guardrail:    ⚠️ WARNING — "You're not using the recommended instance type. Proceed anyway?"
```

**Start with soft guardrails** when rolling out platform engineering. Developers hate being suddenly blocked. Warn first, educate, then gradually make critical ones hard.

## 🛤️ Pillar 3: Golden Paths — The "Right Way Made Easy"

### What Are Golden Paths?

A golden path is the **recommended, opinionated way** to accomplish a common task. It's a pre-paved road that's well-lit, well-maintained, and well-documented.

The key philosophy:

> **"Make the right thing the easy thing."**

If the best-practice way is also the easiest way, developers will naturally follow it.

---

### Golden Path vs No Golden Path

**Without Golden Path:**

```
Developer wants to deploy a new microservice.

"Hmm, which programming language should I use?"
"What's our standard project structure?"
"How do I set up CI/CD?"
"What Docker base image should I use?"
"How do I connect to the database?"
"What monitoring do I need?"
"How do I handle secrets?"
"What testing framework should I use?"

→ Spends 2 weeks figuring this out
→ Does it differently than every other team
→ Misses security best practices
→ No monitoring set up
→ Creates technical debt from day 1
```

**With Golden Path:**

```
Developer wants to deploy a new microservice.

Opens developer portal → Clicks "Create New Microservice"
→ Picks language (Node.js) and framework (Express)
→ Template generates:
   ✅ Standard project structure
   ✅ Pre-configured CI/CD pipeline
   ✅ Dockerfile with approved base image
   ✅ Kubernetes manifests with resource limits
   ✅ Monitoring (Prometheus metrics endpoint)
   ✅ Health check endpoints
   ✅ Secrets management integration
   ✅ Unit test framework setup
   ✅ README with team standards
   ✅ Git repo created and linked

→ Developer writes business logic from minute 1
→ Deploys to dev environment in under 30 minutes
→ Everything follows company standards automatically
```

---

### Anatomy of a Golden Path

A golden path typically includes:

```
🛤️ Golden Path: "Deploy a New Microservice"

1. 📋 Service Catalog Entry
   └── Register your service in Backstage with ownership info

       │   ├── service.yaml
       │   ├── hpa.yaml
       │   └── network-policy.yaml
       ├── .github/workflows/     # CI/CD pipeline
       │   ├── ci.yaml
       │   └── cd.yaml
       ├── monitoring/
       │   ├── dashboard.json     # Grafana dashboard
       │   └── alerts.yaml        # Alert rules
       └── docs/
           ├── README.md
           ├── API.md
           └── RUNBOOK.md

3. 🔄 CI/CD Pipeline
   └── Pre-configured pipeline:
       Build → Test → Scan → Build Image → Deploy to Dev → Integration Tests → Promote

4. 📊 Observability
   └── Auto-configured:
       Prometheus metrics | Grafana dashboard | Log aggregation | Tracing

5. 🔐 Security
   └── Built-in:
       Image scanning | SAST | Secret detection | Network policies

6. 📖 Documentation
   └── Generated:
       API docs | Runbook template | Architecture diagram
```

### Example: Golden Path Starter Template

Here's a real-world example of a **golden path project template** for a Node.js microservice:

**Generated `Dockerfile`** (best practices baked in):

```dockerfile
# Golden Path: Multi-stage build with distroless base image
# This is the RECOMMENDED way to build containers at our company

# Stage 1: Build
FROM node:20-alpine AS builder
WORKDIR /app

# Copy package files first (better layer caching)
COPY package*.json ./
RUN npm ci --only=production

COPY . .
RUN npm run build

# Stage 2: Production (distroless = minimal attack surface)
FROM gcr.io/distroless/nodejs20-debian12
WORKDIR /app

COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/package.json ./

# Non-root user (security best practice)
USER nonroot

EXPOSE 3000

CMD ["dist/server.js"]
```

**Generated `k8s/deployment.yaml`** (guardrails built in):

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ service_name }}
  labels:
    app: {{ service_name }}
    team: {{ team_name }}
    golden-path: "microservice-nodejs-v2"
spec:
  replicas: 2                              # ← Minimum 2 for availability
  selector:
    matchLabels:
      app: {{ service_name }}
  template:
    metadata:
      labels:
        app: {{ service_name }}
      annotations:
        prometheus.io/scrape: "true"       # ← Auto-discovered by Prometheus
        prometheus.io/port: "3000"
        prometheus.io/path: "/metrics"
    spec:
      securityContext:
        runAsNonRoot: true                 # ← Security guardrail
        fsGroup: 1000
      containers:
        - name: {{ service_name }}
          image: {{ image_registry }}/{{ service_name }}:{{ image_tag }}
          ports:
            - containerPort: 3000
          resources:                       # ← Resource guardrail — always set!
            requests:
              cpu: "100m"
              memory: "128Mi"
            limits:
              cpu: "500m"
              memory: "512Mi"
          livenessProbe:                   # ← Health check guardrail
            httpGet:
              path: /health/live
              port: 3000
            initialDelaySeconds: 10
            periodSeconds: 15
          readinessProbe:
            httpGet:
              path: /health/ready
              port: 3000
            initialDelaySeconds: 5
            periodSeconds: 10
          env:
            - name: NODE_ENV
              value: "production"
            - name: LOG_FORMAT
              value: "json"                # ← Structured logging (golden path standard)
```

**Generated CI/CD pipeline** (`.github/workflows/ci.yaml`):

```yaml
name: CI Pipeline (Golden Path v2)

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js

        with:
          node-version: '20'

      - name: Lint

      - name: Unit Tests
        run: npm test -- --coverage

      - name: Upload Coverage

  security-scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Run Trivy (Vulnerability Scan)

          scan-type: 'fs'

    steps:
      - uses: actions/checkout@v4

      - name: Build & Push Docker Image
        uses: docker/build-push-action@v5

          push: true
          tags: |
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:latest
```

### Golden Path ≠ The Only Path

This is a critical point:

> **A golden path is a recommendation, not a mandate.**

```
Golden Path:   "We recommend Node.js + Express for microservices"
Reality:       "If your team has strong Python skills and a good reason, 
                you can use FastAPI instead — just follow the Python golden path"

Golden Path:   "Use PostgreSQL for relational data"
Reality:       "If you need a document store, MongoDB is the approved alternative"
```

The idea is to have a **paved road** that 80% of teams will happily use. The other 20% can go off-road, but they'll need to handle more things themselves.

## ⚠️ Common Mistakes to Avoid

| Mistake | Why It's Bad | Better Approach |
|---------|-------------|-----------------|
| **Too many guardrails too fast** | Developers feel blocked and frustrated | Start with soft guardrails (warnings), gradually harden |
| **Templates are too rigid** | Developers can't customize anything | Allow overrides for non-critical settings |
| **Golden path is the only path** | Innovation is stifled | Make it the default, not the mandate |
| **No documentation** | Nobody knows the templates exist | Invest in docs, training, and demos |
| **Building everything custom** | Huge maintenance burden | Use existing tools (Backstage, Crossplane) |
| **Ignoring developer feedback** | Platform nobody wants to use | Treat developers as your customers |

---

## 🔑 Key Takeaways

1. **Templates** = Pre-built blueprints that hide complexity. Developers pick from a menu instead of writing from scratch.

2. **Guardrails** = Automated safety boundaries. They prevent mistakes without slowing developers down. Use a mix of hard (blocking) and soft (warning) guardrails.

3. **Golden Paths** = The recommended, easiest way to do common tasks. "Make the right thing the easy thing."

4. **Self-service ≠ Wild West.** It's freedom within safe boundaries.

5. **Start small.** Begin with the most common requests (database, namespace, new service) and expand from there.

6. **Treat developers as customers.** Your platform is a product. If developers don't want to use it, it has failed.
