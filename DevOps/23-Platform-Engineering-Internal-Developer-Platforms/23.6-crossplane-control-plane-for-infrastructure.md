# 23.6 — Crossplane — Control Plane for Infrastructure

> **One Line Answer:** Crossplane is a **Kubernetes add-on** that turns your Kubernetes cluster into a **universal control plane** for managing infrastructure across ANY cloud provider (AWS, Azure, GCP, etc.). Instead of using Terraform CLI or cloud consoles, developers can create cloud resources (databases, buckets, VPCs) by simply applying **Kubernetes YAML manifests** — just like they create Pods and Services. It's the ultimate self-service infrastructure tool for platform engineering.

## 🤔 What Problem Does Crossplane Solve?

Let's say you're a platform engineer. Your developers need cloud resources — databases, storage buckets, caches, DNS records, etc.

**Option 1: Terraform** (the traditional way)

```
Developer: "I need a PostgreSQL database"

Platform Team: 
  1. Developer writes Terraform code (or requests it)
  2. Someone reviews the PR
  3. CI/CD runs `terraform plan` and `terraform apply`
  4. State file is stored in S3
  5. Changes are tracked separately from Kubernetes
  
Problems:
  ❌ Developers need to learn Terraform
  ❌ State management is complex
  ❌ Terraform and Kubernetes are separate worlds
  ❌ No automatic drift detection (unless you set up extra tooling)
  ❌ Credentials scattered between Terraform and Kubernetes
```

**Option 2: Cloud Console** (the click-ops way)

```
Developer: "I need a PostgreSQL database"

Platform Team:
  1. Someone logs into AWS Console
  2. Clicks through 15 screens
  3. Maybe forgets to enable encryption
  4. No record of what was created
  5. No way to reproduce it

Problems:
  ❌ Not repeatable
  ❌ No version control
  ❌ Human error
  ❌ No audit trail
  ❌ Snowflake infrastructure
```

**Option 3: Crossplane** (the Kubernetes-native way)

```
Developer: "I need a PostgreSQL database"

Developer:
  1. Writes a simple YAML file (just like creating a Kubernetes Service)
  2. Runs `kubectl apply -f database.yaml`
  3. Crossplane provisions the database in AWS/Azure/GCP
  4. Connection details are automatically stored in a Kubernetes Secret
  5. Crossplane continuously ensures the database matches the desired state

Benefits:
  ✅ Developers already know kubectl and YAML
  ✅ Infrastructure is managed like any other Kubernetes resource
  ✅ Automatic drift detection and correction
  ✅ GitOps-compatible (ArgoCD/Flux can manage Crossplane resources)
  ✅ Self-service with guardrails
```

## 🏗️ Crossplane Architecture — How Does It Work?

```
┌─────────────────────────────────────────────────────────────────┐
│                    Kubernetes Cluster                           │
│                    (Your Control Plane)                         │
│                                                                 │
│   ┌──────────────────────────────────────────┐                 │
│   │           Crossplane Core                │                 │
│   │                                          │                 │
│   │   ┌────────────┐   ┌────────────┐       │                 │
│   │   │  Provider   │   │  Provider  │       │                 │
│   │   │    AWS      │   │   Azure    │       │                 │
│   │   └──────┬─────┘   └──────┬─────┘       │                 │
│   │          │                │              │                 │
│   │   ┌──────┴────────────────┴──────┐       │                 │
│   │   │    Managed Resources (MRs)   │       │                 │
│   │   │  (CRDs for cloud resources)  │       │                 │
│   │   └──────────────────────────────┘       │                 │
│   │                                          │                 │
│   │   ┌──────────────────────────────┐       │                 │
│   │   │    Compositions (XRs)        │       │                 │
│   │   │  (Templates that combine     │       │                 │
│   │   │   multiple resources)        │       │                 │
│   │   └──────────────────────────────┘       │                 │
│   │                                          │                 │
│   │   ┌──────────────────────────────┐       │                 │
│   │   │    Claims (XRCs)             │       │                 │
│   │   │  (What developers use —      │       │                 │
│   │   │   simple, abstracted API)    │       │                 │
│   │   └──────────────────────────────┘       │                 │
│   │                                          │                 │
│   └──────────────────────────────────────────┘                 │
│                                                                 │
└──────────────┬──────────────────┬──────────────────┬───────────┘
               │                  │                  │
        ┌──────▼──────┐   ┌──────▼──────┐   ┌──────▼──────┐
        │     AWS     │   │    Azure    │   │     GCP     │
        │             │   │             │   │             │
        │  RDS, S3,   │   │  SQL DB,    │   │  Cloud SQL, │
        │  VPC, EKS   │   │  Blob,      │   │  GCS, GKE   │
        │  ...        │   │  AKS ...    │   │  ...        │
        └─────────────┘   └─────────────┘   └─────────────┘
```

### Key Components Explained

| Component | What It Is | Analogy |
|-----------|-----------|---------|
| **Provider** | A plugin that knows how to talk to a cloud API (AWS, Azure, GCP, etc.) | Like a **database driver** — it connects Crossplane to a specific cloud |
| **Managed Resource (MR)** | A CRD that represents a single cloud resource (e.g., an RDS instance, an S3 bucket) | Like a Kubernetes **Pod** but for cloud resources |
| **Composition (XR)** | A template that combines multiple managed resources into a single unit | Like a Kubernetes **Deployment** (it manages multiple things as one) |
| **Claim (XRC)** | A simplified API that developers use to request resources | Like ordering from a **menu** — "I want a database, size medium" |

## 🛠️ Getting Started — Installing Crossplane

### Step 1: Install Crossplane into Your Cluster

```bash
# Add the Crossplane Helm repo
helm repo add crossplane-stable https://charts.crossplane.io/stable
helm repo update

# Install Crossplane
helm install crossplane \
  crossplane-stable/crossplane \
  --namespace crossplane-system \
  --create-namespace

# Verify it's running
kubectl get pods -n crossplane-system

# Expected output:
# NAME                                      READY   STATUS    RESTARTS   AGE
# crossplane-7c88c45998-abc12               1/1     Running   0          30s
# crossplane-rbac-manager-8d6c4b869-xyz34   1/1     Running   0          30s
```

### Step 2: Install a Provider (e.g., AWS)

```yaml
# provider-aws.yaml
apiVersion: pkg.crossplane.io/v1
kind: Provider
metadata:
  name: provider-aws
spec:
  package: xpkg.upbound.io/upbound/provider-family-aws:v1.7.0
```

```bash
kubectl apply -f provider-aws.yaml

# Wait for it to be healthy
kubectl get providers

# Expected output:
# NAME           INSTALLED   HEALTHY   PACKAGE                                          AGE
# provider-aws   True        True      xpkg.upbound.io/upbound/provider-family-aws      2m
```

### Step 3: Configure Cloud Credentials

```yaml
# aws-credentials.yaml
# First, create a Kubernetes Secret with your AWS credentials
apiVersion: v1
kind: Secret
metadata:
  name: aws-credentials
  namespace: crossplane-system
type: Opaque
stringData:
  credentials: |
    [default]
    aws_access_key_id = YOUR_ACCESS_KEY_ID
    aws_secret_access_key = YOUR_SECRET_ACCESS_KEY
---
# Then, create a ProviderConfig that references the secret
apiVersion: aws.upbound.io/v1beta1
kind: ProviderConfig
metadata:
  name: default
spec:
  credentials:
    source: Secret
    secretRef:
      namespace: crossplane-system
      name: aws-credentials
      key: credentials
```

```bash
kubectl apply -f aws-credentials.yaml
```

> **Security Note:** In production, use IAM Roles for Service Accounts (IRSA) instead of static credentials! We'll show that later.

## 📝 Level 1: Creating Cloud Resources Directly (Managed Resources)

Now that Crossplane is installed, you can create cloud resources using kubectl.

### Example 1: Create an S3 Bucket

```yaml
# s3-bucket.yaml
apiVersion: s3.aws.upbound.io/v1beta1
kind: Bucket
metadata:
  name: my-app-data-bucket
spec:
  forProvider:
    region: us-east-1
    tags:
      Environment: dev
      Team: backend
      ManagedBy: crossplane
  providerConfigRef:
    name: default
```

```bash
kubectl apply -f s3-bucket.yaml

# Check status
kubectl get bucket my-app-data-bucket

# NAME                  READY   SYNCED   EXTERNAL-NAME          AGE
# my-app-data-bucket    True    True     my-app-data-bucket     2m

# Describe for more details
kubectl describe bucket my-app-data-bucket
```

**That's it!** You just created an S3 bucket in AWS using `kubectl apply`. No Terraform, no AWS console, no CLI. Just YAML.

And here's the magic — if someone goes to the AWS console and **manually changes** something about this bucket, Crossplane will **automatically detect the drift and fix it** (reconciliation loop, just like how Kubernetes ensures Pods match their desired state).

### Example 2: Create an RDS PostgreSQL Database

```yaml
# rds-database.yaml
apiVersion: rds.aws.upbound.io/v1beta1
kind: Instance
metadata:
  name: my-app-database
spec:
  forProvider:
    region: us-east-1
    engine: postgres
    engineVersion: "15"
    instanceClass: db.t3.micro
    allocatedStorage: 20
    
    dbName: myappdb
    username: admin
    
    # Auto-generate a password and store it in a Kubernetes Secret
    autoGeneratePassword: true
    passwordSecretRef:
      name: db-password
      namespace: default
      key: password

    publiclyAccessible: false          # Security guardrail!
    storageEncrypted: true             # Always encrypted!
    
    skipFinalSnapshot: true            # For dev only!
    
    tags:
      Environment: dev
      Team: backend
      ManagedBy: crossplane
  
  # Write connection details to a Kubernetes Secret
  writeConnectionSecretToRef:
    name: db-connection-details
    namespace: default

  providerConfigRef:
    name: default
```

```bash
kubectl apply -f rds-database.yaml

# Watch the database being created (takes a few minutes)
kubectl get instance my-app-database -w

# NAME              READY   SYNCED   EXTERNAL-NAME      AGE
# my-app-database   False   True     my-app-database    30s
# my-app-database   True    True     my-app-database    5m

# Connection details are automatically in a Kubernetes Secret!
kubectl get secret db-connection-details -o yaml
```

Now your application can use the database connection details directly from the Kubernetes Secret:

```yaml
# app-deployment.yaml — using the database
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  template:
    spec:
      containers:
        - name: my-app
          image: myapp:latest
          env:
            - name: DB_HOST
              valueFrom:
                secretKeyRef:
                  name: db-connection-details    # ← Auto-created by Crossplane!
                  key: endpoint
            - name: DB_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: db-connection-details
                  key: password
```

## 📝 Level 2: Compositions — Building Templates for Developers

Managed Resources are powerful, but they're **too detailed** for developers. A developer shouldn't need to know about VPC subnet groups, security groups, parameter groups, etc.

**Compositions** let the platform team create **simplified, opinionated APIs** that hide all that complexity.

### The Idea

```
WITHOUT Compositions:
  Developer needs to create:
  ├── VPC Subnet Group
  ├── Security Group
  ├── Security Group Rule (ingress)
  ├── Security Group Rule (egress)
  ├── RDS Parameter Group
  ├── RDS Instance
  ├── Secret for password
  └── IAM Policy for access
  
  → 8 YAML files, 200+ lines 😩

WITH Compositions:
  Developer writes:
  └── "I want a database, size small, environment dev"
  
  → 1 YAML file, 10 lines 😊
  → Composition creates all 8 resources behind the scenes
```

### Step 1: Define a Composite Resource Definition (XRD)

The XRD defines the **simplified API** that developers will use:

```yaml
# xrd-database.yaml
# This defines WHAT developers can request (the API/schema)
apiVersion: apiextensions.crossplane.io/v1
kind: CompositeResourceDefinition
metadata:
  name: xdatabases.platform.mycompany.com
spec:
  group: platform.mycompany.com
  names:
    kind: XDatabase                    # The "composite resource" name
    plural: xdatabases
  
  claimNames:
    kind: Database                     # The "claim" name (what developers use)
    plural: databases
  
  versions:
    - name: v1alpha1
      served: true
      referenceable: true
      schema:
        openAPIV3Schema:
          type: object
          properties:
            spec:
              type: object
              properties:
                # These are the ONLY parameters developers need to provide
                size:
                  type: string
                  description: "Database size: small, medium, or large"
                  enum:
                    - small
                    - medium
                    - large
                  default: small
                
                engine:
                  type: string
                  description: "Database engine"
                  enum:
                    - postgres
                    - mysql
                  default: postgres
                
                environment:
                  type: string
                  description: "Environment"
                  enum:
                    - dev
                    - staging
                    - prod
                  default: dev
              
              required:
                - size
                - environment
            
            status:
              type: object
              properties:
                endpoint:
                  type: string
                  description: "Database connection endpoint"
                ready:
                  type: boolean
                  description: "Whether the database is ready"
```

```bash
kubectl apply -f xrd-database.yaml
```

### Step 2: Create the Composition

The Composition defines **HOW** the simplified API maps to actual cloud resources:

```yaml
# composition-database.yaml
# This defines HOW a Database claim gets turned into real cloud resources
apiVersion: apiextensions.crossplane.io/v1
kind: Composition
metadata:
  name: database-aws
  labels:
    provider: aws
    crossplane.io/xrd: xdatabases.platform.mycompany.com
spec:
  compositeTypeRef:
    apiVersion: platform.mycompany.com/v1alpha1
    kind: XDatabase
  
  writeConnectionSecretsToNamespace: crossplane-system
  
  resources:
    # Resource 1: Security Group for the database
    - name: security-group
      base:
        apiVersion: ec2.aws.upbound.io/v1beta1
        kind: SecurityGroup
        spec:
          forProvider:
            region: us-east-1
            description: "Security group for Crossplane-managed database"
            ingress:
              - fromPort: 5432
                toPort: 5432
                protocol: tcp
                cidrBlocks:
                  - "10.0.0.0/8"             # Only internal traffic
            tags:
              ManagedBy: crossplane
      patches:
        - type: FromCompositeFieldPath
          fromFieldPath: spec.environment
          toFieldPath: spec.forProvider.tags.Environment
    
    # Resource 2: The actual RDS database
    - name: rds-instance
      base:
        apiVersion: rds.aws.upbound.io/v1beta1
        kind: Instance
        spec:
          forProvider:
            region: us-east-1
            engine: postgres
            engineVersion: "15"
            allocatedStorage: 20
            publiclyAccessible: false        # ALWAYS false (guardrail)
            storageEncrypted: true           # ALWAYS encrypted (guardrail)
            autoGeneratePassword: true
            tags:
              ManagedBy: crossplane
          writeConnectionSecretToRef:
            namespace: crossplane-system
      
      patches:
        # Map "size" to actual instance class
        - type: FromCompositeFieldPath
          fromFieldPath: spec.size
          toFieldPath: spec.forProvider.instanceClass
          transforms:
            - type: map
              map:
                small: db.t3.micro          # $15/month
                medium: db.t3.medium        # $70/month
                large: db.r5.large          # $200/month
        
        # Map "engine" to RDS engine
        - type: FromCompositeFieldPath
          fromFieldPath: spec.engine
          toFieldPath: spec.forProvider.engine
        
        # Set environment tag
        - type: FromCompositeFieldPath
          fromFieldPath: spec.environment
          toFieldPath: spec.forProvider.tags.Environment
        
        # Set database name based on claim name
        - type: FromCompositeFieldPath
          fromFieldPath: metadata.name
          toFieldPath: spec.forProvider.dbName
          transforms:
            - type: string
              string:
                type: Convert
                convert: ToLower
        
        # Set backup retention based on environment
        - type: FromCompositeFieldPath
          fromFieldPath: spec.environment
          toFieldPath: spec.forProvider.backupRetentionPeriod
          transforms:
            - type: map
              map:
                dev: "1"
                staging: "7"
                prod: "30"
        
        # Enable deletion protection for prod
        - type: FromCompositeFieldPath
          fromFieldPath: spec.environment
          toFieldPath: spec.forProvider.deletionProtection
          transforms:
            - type: map
              map:
                dev: "false"
                staging: "false"
                prod: "true"
        
        # Write connection details to status
        - type: ToCompositeFieldPath
          fromFieldPath: status.atProvider.endpoint
          toFieldPath: status.endpoint
      
      connectionDetails:
        - name: endpoint
          fromFieldPath: status.atProvider.endpoint
        - name: port
          fromFieldPath: status.atProvider.port
        - name: username
          fromFieldPath: spec.forProvider.username
        - name: password
          fromConnectionSecretKey: attribute.password
```

```bash
kubectl apply -f composition-database.yaml
```

### Step 3: Developer Creates a Database (The Claim)

Now here's the beautiful part. This is ALL a developer needs to write:

```yaml
# my-database.yaml (Developer writes this — it's called a "Claim")
apiVersion: platform.mycompany.com/v1alpha1
kind: Database
metadata:
  name: user-service-db
  namespace: backend-team
spec:
  size: small
  engine: postgres
  environment: dev
  
  writeConnectionSecretToRef:
    name: user-service-db-conn
```

```bash
# Developer runs this
kubectl apply -f my-database.yaml

# Check status
kubectl get database -n backend-team

# NAME              SIZE    ENGINE     ENV   READY   AGE
# user-service-db   small   postgres   dev   True    5m

# Connection details are in a Secret in the developer's namespace!
kubectl get secret user-service-db-conn -n backend-team
```

**Look at that!** The developer wrote **7 lines of YAML** and got:
- A security group with proper rules
- An RDS instance with encryption, backups, and proper sizing
- Connection details automatically stored in their namespace as a Secret
- Drift detection and auto-correction
- Everything following company standards

The developer didn't need to know anything about security groups, parameter groups, subnet groups, encryption settings, backup policies, etc. The **composition handled all of that** with sensible defaults and guardrails.

## 🔄 How Crossplane Reconciliation Works

This is a key concept. Crossplane uses the same **reconciliation loop** as Kubernetes:

```
     ┌──────────────────────────────────────────┐
     │                                          │
     │     1. Developer applies YAML            │
     │        (desired state)                   │
     │                                          │
     └──────────────┬───────────────────────────┘
                    │
                    ▼
     ┌──────────────────────────────────────────┐
     │                                          │
     │     2. Crossplane reads desired state    │
     │        and compares with actual state    │
     │                                          │
     └──────────────┬───────────────────────────┘
                    │
          ┌─────────┴─────────┐
          │                   │
          ▼                   ▼
     ┌──────────┐       ┌──────────┐
     │  Match?  │       │ No Match │
     │  ✅ Do   │       │ ❌ Fix   │
     │  nothing │       │  it!     │
     └──────────┘       └────┬─────┘
                             │
                             ▼
                    ┌──────────────┐
                    │ Call cloud   │
                    │ API to make  │
                    │ actual =     │
                    │ desired      │
                    └──────────────┘
                             │
                             │
                    (Loop every 1-10 minutes)
```

**Example of drift correction:**

```bash
# Someone manually changes the database instance type in AWS Console
# from db.t3.micro to db.t3.xlarge (maybe trying to be "helpful")

# Crossplane detects the drift within minutes:
# "Hey, the actual state (db.t3.xlarge) doesn't match the desired state (db.t3.micro)"
# "Let me fix that..."

# Crossplane changes it back to db.t3.micro automatically! 🛡️
```

This is **incredibly powerful** for preventing configuration drift and ensuring your infrastructure always matches what's defined in Git.

## 🔐 Production-Ready Setup: Using IRSA (IAM Roles for Service Accounts)

In production, **don't use static AWS credentials.** Use IRSA instead:

```yaml
# provider-config-irsa.yaml
apiVersion: aws.upbound.io/v1beta1
kind: ProviderConfig
metadata:
  name: default
spec:
  credentials:
    source: IRSA    # ← Use IAM Role instead of static keys!
---
# The Crossplane provider's ServiceAccount needs an IAM role annotation
# This is set up when installing the provider
apiVersion: v1
kind: ServiceAccount
metadata:
  name: provider-aws
  namespace: crossplane-system
  annotations:
    eks.amazonaws.com/role-arn: arn:aws:iam::123456789012:role/crossplane-provider-aws
```

## 🆚 Crossplane vs Terraform — When to Use What?

This is one of the most common questions, so let's break it down:

| Feature | Crossplane | Terraform |
|---------|-----------|-----------|
| **Paradigm** | Kubernetes-native (YAML, kubectl) | CLI-based (HCL, terraform apply) |
| **State management** | Stored in Kubernetes etcd (no separate state file) | Requires remote state backend (S3, Consul, etc.) |
| **Drift detection** | Continuous, automatic (reconciliation loop) | Manual (`terraform plan` must be run) |
| **Self-healing** | Yes — automatically fixes drift | No — you must run `terraform apply` |
| **Learning curve for K8s teams** | Low (they already know YAML/kubectl) | Medium (must learn HCL) |
| **Learning curve for non-K8s teams** | High (must know Kubernetes) | Low-Medium |
| **GitOps integration** | Native (ArgoCD/Flux manage Crossplane resources) | Possible but requires extra tooling (Atlantis) |
| **Multi-tenancy** | Namespace-based (Claims are namespaced) | Workspace-based |
| **Ecosystem maturity** | Growing (younger project) | Very mature (huge ecosystem) |
| **Best for** | Platform teams building self-service on K8s | General IaC, teams without Kubernetes |

### When to Use Crossplane

- ✅ You're already running Kubernetes
- ✅ You want self-service infrastructure for developers
- ✅ You want automatic drift detection and correction
- ✅ You're building an Internal Developer Platform
- ✅ You use GitOps (ArgoCD/Flux)

### When to Use Terraform

- ✅ You're managing infrastructure that ISN'T in Kubernetes
- ✅ Your team isn't Kubernetes-native
- ✅ You need the massive Terraform provider ecosystem
- ✅ You're doing initial cluster bootstrapping (creating the K8s cluster itself)
- ✅ You need modules that already exist in the Terraform Registry

### Can You Use Both?

**Yes!** Many teams use both:

```
Terraform → Bootstraps the Kubernetes cluster, VPC, core networking
Crossplane → Manages application-level resources (databases, caches, buckets) 
             within the cluster as self-service for developers
```

## 🧩 Advanced: Crossplane Functions (Compositions v2)

Crossplane recently introduced **Composition Functions** — a way to write composition logic in real programming languages (Go, Python) instead of just YAML patches:

```yaml
# composition-with-functions.yaml
apiVersion: apiextensions.crossplane.io/v1
kind: Composition
metadata:
  name: database-smart
spec:
  compositeTypeRef:
    apiVersion: platform.mycompany.com/v1alpha1
    kind: XDatabase
  mode: Pipeline                    # ← Use function pipeline!
  pipeline:
    - step: create-resources
      functionRef:
        name: function-go-templating
      input:
        apiVersion: gotemplating.fn.crossplane.io/v1beta1
        kind: GoTemplate
        source: Inline
        inline:
          template: |
            {{ $size := .observed.composite.resource.spec.size }}
            {{ $env := .observed.composite.resource.spec.environment }}
            
            {{ $instanceClass := "" }}
            {{ if eq $size "small" }}{{ $instanceClass = "db.t3.micro" }}{{ end }}
            {{ if eq $size "medium" }}{{ $instanceClass = "db.t3.medium" }}{{ end }}
            {{ if eq $size "large" }}{{ $instanceClass = "db.r5.large" }}{{ end }}
            
            apiVersion: rds.aws.upbound.io/v1beta1
            kind: Instance
            metadata:
              name: {{ .observed.composite.resource.metadata.name }}-rds
              annotations:
                gotemplating.fn.crossplane.io/composition-resource-name: rds
            spec:
              forProvider:
                region: us-east-1
                engine: postgres
                instanceClass: {{ $instanceClass }}
                storageEncrypted: true
                publiclyAccessible: false
                backupRetentionPeriod: {{ if eq $env "prod" }}30{{ else }}7{{ end }}
    
    - step: validate
      functionRef:
        name: function-cel-filter      # ← Validate using CEL expressions
      input:
        apiVersion: celfilter.fn.crossplane.io/v1beta1
        kind: CelFilter
        rules:
          - name: no-large-in-dev
            condition: |
              observed.composite.resource.spec.size == "large" && 
              observed.composite.resource.spec.environment == "dev"
            message: "Large databases are not allowed in dev environment"
```

## 📊 Crossplane with GitOps (ArgoCD)

Since Crossplane resources are just Kubernetes objects, they work perfectly with GitOps:

```yaml
# argocd-application.yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: team-backend-infrastructure
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/mycompany/infrastructure.git
    targetRevision: main
    path: teams/backend/resources     # ← Crossplane Claims live here
  destination:
    server: https://kubernetes.default.svc
    namespace: backend-team
  syncPolicy:
    automated:
      selfHeal: true
      prune: true
```

**Git repo structure:**

```
infrastructure/
├── teams/
│   ├── backend/
│   │   └── resources/
│   │       ├── database.yaml         # Crossplane Database Claim
│   │       ├── redis.yaml            # Crossplane Redis Claim
│   │       └── s3-bucket.yaml        # Crossplane S3 Bucket Claim
│   ├── frontend/
│   │   └── resources/
│   │       ├── cdn.yaml
│   │       └── s3-static-site.yaml
│   └── data-team/
│       └── resources/
│           ├── data-warehouse.yaml
│           └── kafka-cluster.yaml
└── platform/
    ├── compositions/                  # Platform team manages these
    │   ├── database-composition.yaml
    │   ├── redis-composition.yaml
    │   └── s3-composition.yaml
    └── xrds/
        ├── database-xrd.yaml
        └── redis-xrd.yaml
```

**Workflow:**

```
1. Developer wants a database
2. Developer adds a Database Claim YAML to their team's folder in Git
3. Creates a PR → reviewed by platform team (or auto-approved with guardrails)
4. PR is merged
5. ArgoCD syncs the Claim to the cluster
6. Crossplane provisions the database in AWS
7. Connection Secret appears in the developer's namespace
8. Developer's app uses the Secret

Everything is in Git. Everything is auditable. Everything is automated. 🎉
```

## ⚠️ Crossplane Gotchas & Best Practices

| Gotcha | Solution |
|--------|----------|
| **Provider CRDs are huge** | Use Provider Families to install only the CRDs you need |
| **Debugging is hard** | Use `kubectl describe` on the Composite Resource and check events |
| **Slow provisioning** | Cloud APIs are slow (RDS takes 5-10 min). Set expectations |
| **CRD conflicts** | Be careful with CRD versions across providers |
| **RBAC complexity** | Use Claims (namespace-scoped) instead of Composite Resources (cluster-scoped) |

**Debugging tips:**

```bash
# Check the composite resource
kubectl describe xdatabase user-service-db

# Check individual managed resources
kubectl get managed

# Check Crossplane events
kubectl get events --field-selector involvedObject.kind=Instance

# Check Crossplane provider logs
kubectl logs -n crossplane-system -l pkg.crossplane.io/revision -c package-runtime
```

## 🔑 Key Takeaways

1. **Crossplane turns Kubernetes into a universal infrastructure control plane.** Create cloud resources with `kubectl apply`.

2. **Managed Resources** are 1:1 mappings to cloud resources (like Terraform resources).

3. **Compositions** let platform teams create simplified, opinionated APIs that hide complexity from developers.

4. **Claims** are what developers actually use — simple, namespace-scoped requests for infrastructure.

5. **Automatic drift detection** — Crossplane continuously reconciles desired state vs actual state. If someone changes something manually, Crossplane fixes it.

6. **GitOps-native** — Crossplane resources are Kubernetes objects, so ArgoCD/Flux manage them seamlessly.

7. **Crossplane + Terraform** is a valid combo — Terraform bootstraps the cluster, Crossplane manages app-level resources.

8. **Start small** — begin with one composition (e.g., databases), prove value, then expand.
