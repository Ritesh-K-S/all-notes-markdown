# 23.7 — Port & Humanitec — Platform Orchestration

> **One Line Answer:** **Port** and **Humanitec** are platform orchestration tools that help platform teams build **Internal Developer Platforms (IDPs)** without building everything from scratch. Port gives you a **developer portal** with self-service actions, scorecards, and a software catalog. Humanitec provides a **Platform Orchestrator** that dynamically generates infrastructure and deployment configs based on simple developer inputs. Both solve the same core problem: **"How do we let developers self-serve without drowning in YAML?"**

---

## 🤔 Why Do We Need Platform Orchestration Tools?

So far in this chapter, we've talked about:
- **Backstage** (23.3) — Developer portal for documentation and service catalog
- **Self-service infrastructure** (23.4) — Templates, guardrails, golden paths
- **DevEx** (23.5) — Making developers productive and happy
- **Crossplane** (23.6) — Kubernetes-native infrastructure provisioning

But here's the problem: **putting all these pieces together is REALLY hard.**

```
Building a Platform WITHOUT Orchestration Tools:

Platform Team needs to:
├── Set up Backstage for service catalog
├── Write Crossplane Compositions for each resource type
├── Create ArgoCD ApplicationSets for GitOps
├── Build CI/CD pipeline templates
├── Create Terraform modules for non-K8s resources
├── Build custom scripts for environment management
├── Create custom dashboards for service health
├── Write policies (OPA/Kyverno) for guardrails
├── Build a scoring system for service maturity
├── Create custom webhooks for self-service actions
├── Wire everything together with glue code
└── Maintain all of this forever

→ Easily 6-12 months of work for a team of 4-5 engineers
→ Lots of custom code that needs maintenance
→ Every company reinvents the same wheel 😩
```

**Platform orchestration tools** like Port and Humanitec give you a **pre-built foundation** so you don't have to build everything from scratch.

```
Building a Platform WITH Orchestration Tools:

Platform Team needs to:
├── Configure Port/Humanitec (not build it)
├── Define resource types and relationships
├── Set up self-service actions
├── Create scoring rules
└── Connect to existing tools (K8s, Git, CI/CD, cloud)

→ Weeks instead of months
→ Less custom code to maintain
→ Focus on YOUR platform's unique needs, not rebuilding common features
```

## 🌊 Part 1: Port — The Internal Developer Portal

### What Is Port?

**Port** (getport.io) is an **Internal Developer Portal** that gives you:

1. **Software Catalog** — A single place to see all your services, environments, infrastructure, and their relationships
2. **Self-Service Actions** — Buttons that developers click to do things (create a service, deploy, rollback, etc.)
3. **Scorecards** — Track how well services follow best practices
4. **Automations** — Trigger workflows based on events
5. **Dashboards** — Visualize your platform's health

Think of Port as **Backstage on steroids** — with a focus on **actions** and **scorecards**, not just documentation.

---


When they click "Create Service", Port triggers a GitHub Actions workflow:

```yaml
# .github/workflows/create-microservice.yaml
name: Create Microservice

on:
  workflow_dispatch:
    inputs:
      service_name:
        required: true
      language:
        required: true
      framework:
        required: true
      team:
        required: true
      needs_database:
        required: true
      database_type:
        required: false
      port_run_id:
        required: true       # Port passes this to track the action

jobs:
  create-service:
    runs-on: ubuntu-latest
    steps:
      # Step 1: Report action started to Port
      - name: Report to Port - In Progress
        uses: port-labs/port-github-action@v1
        with:
          clientId: ${{ secrets.PORT_CLIENT_ID }}
          clientSecret: ${{ secrets.PORT_CLIENT_SECRET }}
          operation: PATCH_RUN
          runId: ${{ inputs.port_run_id }}
          status: "IN_PROGRESS"
          logMessage: "🚀 Creating microservice '${{ inputs.service_name }}'..."

      # Step 2: Create repo from template
      - name: Create Repository from Template
        uses: actions/github-script@v7
        with:
          github-token: ${{ secrets.ORG_TOKEN }}
          script: |
            const templateMap = {
              'nodejs-express': 'microservice-template-nodejs-express',
              'python-fastapi': 'microservice-template-python-fastapi',
              'go-gin': 'microservice-template-go-gin'
            };
            const template = templateMap[`${{ inputs.language }}-${{ inputs.framework }}`];
            
            await github.rest.repos.createUsingTemplate({
              template_owner: 'mycompany',
              template_repo: template,
              owner: 'mycompany',
              name: '${{ inputs.service_name }}',
              private: true,
              description: 'Created via Platform Portal'
            });

      # Step 3: Create database if requested
      - name: Create Database (if needed)
        if: inputs.needs_database == 'true'
        run: |
          # Create a Crossplane Claim for the database
          cat <<EOF > database-claim.yaml
          apiVersion: platform.mycompany.com/v1alpha1
          kind: Database
          metadata:
            name: ${{ inputs.service_name }}-db
            namespace: ${{ inputs.team }}
          spec:
            size: small
            engine: ${{ inputs.database_type }}
            environment: dev
            writeConnectionSecretToRef:
              name: ${{ inputs.service_name }}-db-conn
          EOF
          
          kubectl apply -f database-claim.yaml

      # Step 4: Register service in Port's catalog
      - name: Register in Port Catalog
        uses: port-labs/port-github-action@v1
        with:
          clientId: ${{ secrets.PORT_CLIENT_ID }}
          clientSecret: ${{ secrets.PORT_CLIENT_SECRET }}
          operation: UPSERT
          blueprint: service
          identifier: ${{ inputs.service_name }}
          properties: |
            {
              "language": "${{ inputs.language }}",
              "tier": "Tier 3 - Internal",
              "lifecycle": "Development",
              "repository": "https://github.com/mycompany/${{ inputs.service_name }}"
            }
          relations: |
            {
              "team": "${{ inputs.team }}"
            }

      # Step 5: Report success to Port
      - name: Report to Port - Success
        uses: port-labs/port-github-action@v1
        with:
          clientId: ${{ secrets.PORT_CLIENT_ID }}
          clientSecret: ${{ secrets.PORT_CLIENT_SECRET }}
          operation: PATCH_RUN
          runId: ${{ inputs.port_run_id }}
          status: "SUCCESS"
          logMessage: |
            ✅ Microservice '${{ inputs.service_name }}' created!
            📁 Repo: https://github.com/mycompany/${{ inputs.service_name }}
            🐘 Database: ${{ inputs.needs_database == 'true' && 'Created!' || 'Not requested' }}
```

**The developer experience:**

```
1. Click "Create Microservice" in Port
2. Fill in the form (30 seconds)
3. Click "Create"
4. Watch the progress in real-time in Port's UI
5. 2 minutes later: repo is created, database is provisioned, 
   service is registered in the catalog
6. Start coding! 🚀
```

## 🤖 Part 2: Humanitec — The Platform Orchestrator

### What Is Humanitec?

**Humanitec** takes a fundamentally different approach than Port or Backstage. Instead of being a portal, Humanitec is a **Platform Orchestrator** — it sits between developers and infrastructure and **dynamically generates** the right configuration for each deployment.

The key idea:

> **Developers declare WHAT they need (e.g., "my app needs a database and a DNS record"). Humanitec figures out HOW to create it based on the environment and company policies.**

---

### The Problem Humanitec Solves

```
Traditional Approach:

Developer's app needs:
├── A container running their code
├── A PostgreSQL database  
├── A Redis cache
├── A DNS record
├── An SSL certificate
└── Environment variables connecting everything

For EACH environment, someone needs to manually create:
├── Different Kubernetes manifests (dev vs prod have different settings)
├── Different database configs (RDS for prod, local PostgreSQL for dev)
├── Different DNS records (dev.myapp.com vs myapp.com)
├── Different secrets
└── Different resource sizes

3 services × 4 environments = 12 different configurations to maintain 😩
And every change needs to be applied to all of them!
```

```
Humanitec Approach:

Developer says: "My app needs a database and a cache"
                (This is called a "Score" workload spec)

Humanitec says: "OK! Let me figure out the details for each environment."

For dev:    → Local PostgreSQL in K8s, small Redis, dev.myapp.com
For staging: → RDS small, ElastiCache small, staging.myapp.com  
For prod:   → RDS large with Multi-AZ, ElastiCache cluster, myapp.com

All generated automatically based on rules the platform team set up.
Developer doesn't need to know ANY of these details. 🎉
```

### Score — The Developer's Workload Spec

**Score** is an open-source, platform-agnostic workload specification. It's incredibly simple:

```yaml
# score.yaml — This is ALL the developer writes
apiVersion: score.dev/v1b1

metadata:
  name: order-service

containers:
  main:
    image: .                          # Built from this repo
    variables:
      DB_HOST: ${resources.db.host}
      DB_PORT: ${resources.db.port}
      DB_NAME: ${resources.db.name}
      DB_PASSWORD: ${resources.db.password}
      REDIS_URL: ${resources.cache.url}
      LOG_LEVEL: info

resources:
  db:
    type: postgres                    # "I need a PostgreSQL database"
  cache:
    type: redis                       # "I need a Redis cache"
  dns:
    type: dns                         # "I need a DNS record"
  route:
    type: route                       # "I need an HTTP route"
    params:
      host: ${resources.dns.host}
      path: /api/orders
      port: 3000

service:
  ports:
    http:
      port: 3000
      protocol: TCP
```

**Notice what's NOT in this file:**
- No Kubernetes YAML (no Deployments, Services, Ingresses)
- No Terraform code
- No cloud-specific configuration
- No environment-specific settings
- No resource sizes, regions, or instance types

The developer just says "I need a database and a cache" — that's it. Humanitec handles everything else based on the platform team's rules.

### Resource Definitions — How Things Get Built

The platform team creates **Resource Definitions** that tell Humanitec how to provision each resource type:

```yaml
# resource-definition: PostgreSQL for development
# In dev, use a simple PostgreSQL container in Kubernetes
apiVersion: entity.humanitec.io/v1b1
kind: Definition
metadata:
  id: postgres-dev
entity:
  name: PostgreSQL (Dev - In-Cluster)
  type: postgres
  driver_type: humanitec/k8s-cluster-postgres
  driver_inputs:
    values:
      version: "15"
      size: small
      storage: 5Gi
  criteria:
    - env_type: development           # ← Only used in dev environments
```

```yaml
# resource-definition: PostgreSQL for production
# In prod, use a managed AWS RDS instance
apiVersion: entity.humanitec.io/v1b1
kind: Definition
metadata:
  id: postgres-prod
entity:
  name: PostgreSQL (Prod - AWS RDS)
  type: postgres
  driver_type: humanitec/terraform
  driver_inputs:
    values:
      source:
        path: modules/rds-postgresql
        url: https://github.com/mycompany/terraform-modules.git
        rev: v2.1.0
      variables:
        instance_class: db.r5.large
        multi_az: true
        storage_encrypted: true
        backup_retention_period: 30
        deletion_protection: true
  criteria:
    - env_type: production             # ← Only used in prod environments
```

**The magic:** When a developer deploys their Score file:
- **In dev:** Humanitec uses the `postgres-dev` definition → creates a simple PostgreSQL Pod in the cluster
- **In prod:** Humanitec uses the `postgres-prod` definition → creates a fully managed RDS instance with encryption, Multi-AZ, and backups

**The developer's Score file is EXACTLY THE SAME in both cases.** The platform team controls what happens behind the scenes.

### Humanitec's Dynamic Configuration

The most powerful feature of Humanitec is **dynamic configuration resolution**. Here's what happens:

```
Developer's Score file says:
  DB_HOST: ${resources.db.host}

Humanitec resolves this DIFFERENTLY per environment:

Dev environment:
  DB_HOST: postgresql.order-service-dev.svc.cluster.local

Staging environment:
  DB_HOST: order-service-staging.abc123.us-east-1.rds.amazonaws.com

Production environment:
  DB_HOST: order-service-prod.xyz789.us-east-1.rds.amazonaws.com

The developer NEVER needs to know these values!
Humanitec figures them out automatically and injects them.
```

## 🆚 Port vs Humanitec — Comparison

| Feature | Port | Humanitec |
|---------|------|-----------|
| **Primary focus** | Developer Portal (catalog + self-service) | Platform Orchestrator (config generation) |
| **What it replaces** | Backstage (partially), custom portals | Custom deployment scripts, manual YAML management |
| **Software Catalog** | ✅ Built-in, very powerful | ❌ Not a portal (pairs with Backstage/Port for UI) |
| **Self-Service Actions** | ✅ Rich UI-driven actions | ✅ Through Score + CI/CD |
| **Scorecards** | ✅ Built-in | ❌ Not a feature |
| **Dynamic Config Generation** | ❌ Not its focus | ✅ Core feature |
| **Resource Provisioning** | Via actions (triggers external tools) | Built-in (drivers for cloud resources) |
| **Environment Management** | Basic | ✅ Core feature (dynamic per-env config) |
| **Open Source** | No (commercial, free tier available) | Score spec is open source, orchestrator is commercial |
| **Best For** | Teams wanting a developer portal with actions | Teams wanting automatic config generation |
| **Pricing** | Free tier + paid plans | Enterprise pricing |

### When to Use Port

- ✅ You want a **developer portal** with a rich UI
- ✅ You want **scorecards** to track service maturity
- ✅ You want to build **self-service actions** (buttons developers click)
- ✅ You want a **software catalog** to track everything
- ✅ You already have Kubernetes, Terraform, etc., and want to wrap them in a nice UI

### When to Use Humanitec

- ✅ You're drowning in **environment-specific YAML** configurations
- ✅ You want to **abstract infrastructure completely** from developers
- ✅ You have many environments and the config differences are killing you
- ✅ You want **dynamic resource provisioning** that adapts per environment
- ✅ You want developers to just describe WHAT they need, not HOW

### Can You Use Both?

**Yes!** They solve different problems and complement each other:

```
Port (Portal Layer)          — What developers see and interact with
                ↕
Humanitec (Orchestration)    — How deployments are configured and executed
                ↕
Crossplane / Terraform       — How infrastructure is actually provisioned
                ↕
Kubernetes / Cloud APIs      — Where things run
```

## 📋 Getting Started — Practical Steps

### If You're Just Starting Platform Engineering:

```
Month 1-2:  Start with Backstage (free, open-source)
            → Set up service catalog
            → Create basic templates

Month 3-4:  Add Port or a scorecard solution
            → Define production readiness criteria
            → Create self-service actions for common tasks

Month 5-6:  Add Crossplane or Terraform modules for self-service infra
            → Databases, caches, storage as self-service

Month 7+:   Consider Humanitec if config management is painful
            → Especially if you have 4+ environments per service
```

### If You Already Have a Mature Platform:

```
Evaluate:
1. Where are developers still waiting for tickets?
   → Add self-service actions (Port)
   
2. Are environment configs a mess?
   → Consider Humanitec for dynamic config

3. Are services not following best practices?
   → Add scorecards (Port, OpsLevel)

4. Is infrastructure provisioning a bottleneck?
   → Add Crossplane or standardized Terraform modules
```

## ⚠️ Common Mistakes

| Mistake | Better Approach |
|---------|-----------------|
| **Buying a tool before understanding the problem** | Interview developers first, map pain points, then evaluate tools |
| **Trying to adopt everything at once** | Start with one tool, prove value, then expand |
| **Not getting developer buy-in** | Involve developers from day 1 — they're your customers |
| **Over-customizing the portal** | Start with defaults, customize only what's needed |
| **Ignoring the "boring" stuff** | Good docs, fast CI/CD, and a working local dev setup matter more than a fancy portal |
| **Not measuring outcomes** | Track DORA metrics, survey satisfaction, measure self-service ratio |

## 🔑 Key Takeaways

1. **Port** is a developer portal that gives you a software catalog, self-service actions, and scorecards. It's great for giving developers visibility and self-service capabilities.

2. **Humanitec** is a platform orchestrator that dynamically generates deployment configurations based on what developers need and which environment they're deploying to. It eliminates environment-specific YAML sprawl.

3. **Score** (by Humanitec) is an open-source, platform-agnostic workload specification. Developers describe WHAT their app needs, not HOW to deploy it.

4. **Port and Humanitec complement each other** — Port handles the portal/UI layer, Humanitec handles the orchestration/config layer.

5. **Start simple.** You don't need all these tools on day 1. Start with the biggest pain point and solve it first.

6. **The goal is always the same:** Let developers focus on building features, not fighting infrastructure.
