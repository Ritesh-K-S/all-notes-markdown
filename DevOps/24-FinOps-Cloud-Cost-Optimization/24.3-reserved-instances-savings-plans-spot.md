# 24.3 — Reserved Instances, Savings Plans & Spot Instances

> **One Line Answer:** AWS offers three ways to **pay less than On-Demand pricing** — **Reserved Instances** (commit to a specific instance type for 1-3 years, save up to 72%), **Savings Plans** (commit to a dollar amount of compute per hour, save up to 72% with more flexibility), and **Spot Instances** (use spare AWS capacity at up to 90% discount, but AWS can take it back with 2 minutes notice).

---

## 🤔 Why Does This Matter?

Here's a shocking fact: if you're running EC2 instances **24/7 on On-Demand pricing**, you're probably paying **2-3x more than you need to**.

Let's look at the math:

```
One m5.xlarge instance running 24/7 for a year:

On-Demand:                $0.192/hour × 8,760 hours = $1,682/year
1-Year Reserved (No Up):  $0.124/hour × 8,760 hours = $1,086/year  → Save $596 (35%)
3-Year Reserved (All Up): $0.079/hour × 8,760 hours = $692/year   → Save $990 (59%)
Spot Instance:            ~$0.040/hour × 8,760 hours = $350/year   → Save $1,332 (80%)
```

**Now multiply that by 50 instances:**

```
50 × m5.xlarge On-Demand:    $84,100/year
50 × m5.xlarge 1-Year RI:    $54,300/year  → Save $29,800/year
50 × m5.xlarge 3-Year RI:    $34,600/year  → Save $49,500/year
```

**That's $30K-$50K saved per year** — just by committing to what you're already using. And that's only 50 instances. Imagine the savings at scale.

Before we dive deep, let's understand all the pricing options at a high level:

```
AWS Compute Pricing Models:
═══════════════════════════

┌──────────────┬───────────┬───────────┬────────────────────────────┐
│ Model        │ Discount  │ Commitment│ Best For                   │
├──────────────┼───────────┼───────────┼────────────────────────────┤

Spot         = Using someone's leftover minutes (cheapest, might get cut off)
```

## 🔒 Part 1: Reserved Instances (RIs)

### What are Reserved Instances?

A **Reserved Instance** is a **billing discount** applied to On-Demand instances. You commit to using a **specific instance type**, in a **specific region**, for **1 or 3 years** — and in return, AWS gives you a big discount.

> ⚠️ **Important:** RIs are NOT separate instances. They're a **billing construct**. You still launch normal EC2 instances — the RI discount is automatically applied to matching instances.

```
How it works:
─────────────
1. You buy a Reserved Instance: m5.xlarge, us-east-1, Linux, 1-year
2. Any running m5.xlarge in us-east-1 (Linux) automatically gets the RI discount
3. If no matching instance is running → You're still paying for the RI (wasted money!)
4. If more instances than RIs → Extra instances billed at On-Demand rates
```

### RI Payment Options

There are three payment options, each with different discount levels:

| Payment Option | How You Pay | Discount Level | Best For |
|----------------|-------------|----------------|----------|
| **No Upfront** | Monthly payments only | Lowest (~36% off) | Companies that prefer monthly billing |
| **Partial Upfront** | Part upfront + monthly | Medium (~42% off) | Balance of savings and cash flow |
| **All Upfront** | Pay everything upfront | Highest (~59-72% off) | Maximum savings, have cash available |

```
Example: m5.xlarge in us-east-1, 1-Year Term

On-Demand:        $0.192/hour   →   $1,682/year
No Upfront RI:    $0.124/hour   →   $1,086/year   →   Save 35%
Partial Upfront:  $595 upfront + $0.068/hour  →   $1,191/year  →   Save 29%
                  (Wait — that's more expensive? No! $595 + $0.068×8760 = $1,191)
                  Actually: Partial = $595 + ($0.034×8760) = $893/year → Save 47%
All Upfront:      $987 upfront  →   $987/year    →   Save 41%

3-Year Term:
All Upfront:      $1,687 upfront → $562/year     →   Save 67%
```

> **Note:** Exact prices vary by instance type, region, and change over time. Always check the [AWS Pricing page](https://aws.amazon.com/ec2/pricing/reserved-instances/pricing/) for current rates.

### Types of Reserved Instances

| Type | Flexibility | Discount |
|------|------------|----------|
| **Standard RI** | Locked to instance type, can change AZ and size within family | Higher discount |
| **Convertible RI** | Can change instance type, OS, tenancy during the term | Lower discount (~45-54%) |

```
Standard RI:
  ✅ Buy: m5.xlarge, us-east-1
  ✅ Can change to: m5.large (same family, smaller) — uses "instance size flexibility"
  ❌ Cannot change to: c5.xlarge (different family)
  ❌ Cannot change to: r5.xlarge (different family)

Convertible RI:
  ✅ Buy: m5.xlarge, us-east-1
  ✅ Can exchange to: c5.2xlarge (different family!) 
  ✅ Can exchange to: r5.xlarge (different family!)
  ✅ Can change OS: Linux → Windows
  ⚠️ Can only exchange to EQUAL or HIGHER value
```

### Instance Size Flexibility (Standard RIs)

This is a really useful feature. Within the same instance family, your RI can cover different sizes:

```
You bought: 1x m5.xlarge RI

Instance size equivalence (normalization factor):
─────────────────────────────────────────────────
m5.xlarge    = 8 units
m5.large     = 4 units
m5.medium    = 2 units
m5.small     = 1 unit
m5.2xlarge   = 16 units

Your 1x m5.xlarge RI (8 units) can cover:
  ✅ 1x m5.xlarge (8 units)
  ✅ 2x m5.large (4+4 = 8 units)
  ✅ 4x m5.medium (2+2+2+2 = 8 units)
  ✅ 1x m5.large + 2x m5.medium (4+2+2 = 8 units)
  ❌ 1x m5.2xlarge (16 units — not enough!)
```

### RI Utilization — The Most Important Metric

Buying RIs is only half the battle. You MUST make sure they're being used:

```
RI Utilization = (Hours Used / Hours Available) × 100%

Good:   95-100% utilization → You're using what you paid for ✅
OK:     80-95% utilization  → Some waste, investigate
Bad:    < 80% utilization   → You're paying for RIs that aren't being used! 🔴

Example:
  You bought 10 × m5.xlarge RIs (8,760 hours each = 87,600 total hours/year)
  But you only run 8 instances most of the time
  
  Utilization: 8/10 = 80%
  Wasted: 2 RIs × $1,086/year = $2,172/year thrown away!
```

Check utilization in:
- **Cost Explorer** → RI Utilization Report
- **AWS Budgets** → RI Budget (set alerts if utilization drops below 90%)

```bash
# Check RI utilization via CLI
aws ce get-reservation-utilization \
  --time-period Start=2025-01-01,End=2025-04-01 \
  --granularity MONTHLY \
  --output table
```

### RI Coverage — Are You Missing Savings?

**RI Coverage** shows what percentage of your On-Demand usage **could be** covered by RIs:

```
RI Coverage = (RI Hours / Total Hours) × 100%

Example:
  Total EC2 hours: 100,000 hours
  Covered by RIs: 70,000 hours
  On-Demand: 30,000 hours

  Coverage: 70%
  
  → You're paying On-Demand for 30% of usage
  → Opportunity: Buy more RIs for steady-state workloads!
```

```bash
# Check RI coverage
aws ce get-reservation-coverage \
  --time-period Start=2025-01-01,End=2025-04-01 \
  --granularity MONTHLY \
  --output table
```

Made a mistake? Workload changed? You can sell Standard RIs on the **Reserved Instance Marketplace**:

```

Rules:

  ❌ Convertible RIs CANNOT be sold

```

## 💚 Part 2: Savings Plans — The Modern Way

### What are Savings Plans?

**Savings Plans** are the **newer, more flexible alternative** to Reserved Instances. Instead of committing to a specific instance type, you commit to a **dollar amount of compute per hour** for 1 or 3 years.

```
Reserved Instance:     "I'll use m5.xlarge in us-east-1 for 1 year"
Savings Plan:          "I'll spend $10/hour on ANY compute for 1 year"
```

### Types of Savings Plans

| Type | Flexibility | Discount |
|------|------------|----------|
| **Compute Savings Plans** | Any instance family, region, OS, tenancy. Also covers Fargate and Lambda! | Up to 66% off |
| **EC2 Instance Savings Plans** | Locked to instance family + region, but flexible on size, OS, tenancy | Up to 72% off |
| **SageMaker Savings Plans** | For ML workloads on SageMaker | Up to 64% off |

Let's compare:

```
Compute Savings Plan ($10/hour commitment):
  ✅ Covers m5.xlarge in us-east-1
  ✅ Covers c5.2xlarge in eu-west-1 (different family AND region!)
  ✅ Covers Fargate tasks
  ✅ Covers Lambda invocations
  ✅ Change anytime — as long as you spend $10/hour on compute
  
EC2 Instance Savings Plan ($10/hour commitment, m5 family, us-east-1):
  ✅ Covers m5.xlarge in us-east-1
  ✅ Covers m5.large in us-east-1 (same family, different size)
  ✅ Covers m5.xlarge Windows in us-east-1 (same family, different OS)
  ❌ Does NOT cover c5 instances
  ❌ Does NOT cover m5 in eu-west-1 (different region)
  ❌ Does NOT cover Fargate or Lambda
```

### How Savings Plans Work — Step by Step

```
Step 1: Look at your On-Demand compute usage
  → You spend about $15/hour on EC2 compute consistently

Step 2: Decide your commitment
  → You commit to $10/hour (covering ~67% of your usage)
  → The committed $10/hour gets the Savings Plan discount
  → The remaining $5/hour stays at On-Demand rates

Step 3: AWS automatically applies the discount
  → Every hour, AWS looks at your usage
  → It applies the SP discount to the first $10/hour of usage
  → Anything above $10/hour is billed at On-Demand

Visual:
  Total hourly usage: $15/hour On-Demand
  
  ┌──────────────────────────────────────┐
  │  $10/hour → Savings Plan discount    │  ← 66% off = $3.40/hour
  ├──────────────────────────────────────┤
  │  $5/hour → On-Demand pricing         │  ← Full price = $5/hour
  └──────────────────────────────────────┘
  
  Without SP: $15/hour × 8,760 = $131,400/year
  With SP:    ($3.40 + $5) × 8,760 = $73,584/year
  Savings:    $57,816/year (44% savings on total, 66% on committed portion)
```

### Savings Plans Payment Options

Same as RIs — three payment options:

| Payment Option | Discount |
|----------------|----------|
| No Upfront | Lowest (~50-60% off) |
| Partial Upfront | Medium (~55-66% off) |
| All Upfront | Highest (~60-72% off) |

### Savings Plans Utilization & Coverage

Just like RIs, you need to monitor utilization:

```
Savings Plan: $10/hour commitment

Good: You consistently use $10+/hour of compute
  → 100% utilization ✅
  → SP covers $10/hour, rest is On-Demand

Bad: Some hours you only use $6/hour of compute
  → 60% utilization for those hours 🔴
  → You're paying for $10/hour but only using $6
  → $4/hour wasted during those periods

Solution: Right-size your commitment!
  → If you consistently use $8-15/hour, commit to $8/hour (the minimum baseline)
  → This ensures near 100% utilization
```

### Reserved Instances vs Savings Plans — When to Use Which?

```
┌─────────────────────────────────┬──────────────────┬──────────────────┐
│ Feature                         │ Reserved Instance │ Savings Plans    │
├─────────────────────────────────┼──────────────────┼──────────────────┤
│ Commitment                      │ Instance type    │ $/hour           │
│ Flexibility                     │ Limited          │ High             │
│ Covers EC2                      │ ✅               │ ✅               │
│ Covers Fargate                  │ ❌               │ ✅ (Compute SP)  │
│ Covers Lambda                   │ ❌               │ ✅ (Compute SP)  │
│ Cross-region                    │ ❌ (Standard)    │ ✅ (Compute SP)  │
│ Cross-family                    │ ❌ (Standard)    │ ✅ (Compute SP)  │
│ Change OS                       │ ❌ (Standard)    │ ✅               │
│ Max discount                    │ ~72%             │ ~72%             │
│ Sellable on marketplace         │ ✅ (Standard)    │ ❌               │
│ Capacity reservation            │ ✅ (Zonal)       │ ❌               │
│ Ease of management              │ Complex          │ Simple           │
│ AWS recommendation              │ Being phased out │ ✅ Preferred     │
└─────────────────────────────────┴──────────────────┴──────────────────┘
```

**AWS's recommendation:** Use **Savings Plans** for most cases. Use **RIs** only when you need:
- Capacity reservation (guarantee an instance is available)
- To sell unused commitments on the marketplace

```
Decision flowchart:
────────────────────
Do you need capacity reservation?
  YES → Use Zonal Reserved Instances
  NO  → Continue ↓

Do you use Fargate or Lambda significantly?
  YES → Use Compute Savings Plans (covers everything)
  NO  → Continue ↓

Are your workloads in a single instance family and region?
  YES → Use EC2 Instance Savings Plans (highest discount)
  NO  → Use Compute Savings Plans (most flexible)
```

## ⚡ Part 3: Spot Instances — The Biggest Savings (With a Catch)

### What are Spot Instances?

**Spot Instances** let you use **spare (unused) EC2 capacity** at up to **90% discount** compared to On-Demand. The catch? **AWS can take them back with a 2-minute warning** whenever they need the capacity.

```
Think of it like standby flights:
───────────────────────────────────
Regular ticket (On-Demand):  $500  → Guaranteed seat
Standby ticket (Spot):       $50  → 90% cheaper! But if the flight fills up,
                                    you get bumped and have to wait for the next one.
```

### How Spot Pricing Works

Spot prices fluctuate based on **supply and demand**:

```
Spot Price History (m5.xlarge, us-east-1a):
──────────────────────────────────────────
Time          Spot Price    On-Demand    Savings
06:00 AM      $0.038/hr     $0.192/hr    80%
12:00 PM      $0.042/hr     $0.192/hr    78%
06:00 PM      $0.039/hr     $0.192/hr    80%
12:00 AM      $0.036/hr     $0.192/hr    81%

Average:      ~$0.039/hr    $0.192/hr    ~80% savings
```

> **Important:** Unlike the old Spot bidding model, you no longer set a bid price. You either pay the current Spot price or you don't get the instance. If the price exceeds On-Demand, AWS won't launch your Spot instance.

### Spot Interruptions — The Big Catch

AWS can reclaim your Spot Instance with **2 minutes warning** when:
- The Spot price exceeds your maximum price (if you set one)
- AWS needs the capacity back for On-Demand or RI customers
- The available capacity drops

```
Interruption notice:
  → AWS posts a notice to instance metadata
  → You have 2 minutes to gracefully shut down
  → Your instance is stopped or terminated (depending on your config)

Check for interruption (from inside the instance):
  $ curl -s http://169.254.169.254/latest/meta-data/spot/instance-action
  
  No interruption:  404 (not found)
  Interruption!:    {"action": "terminate", "time": "2025-03-15T12:00:00Z"}
```

### When to Use Spot (and When NOT To)

```
✅ GREAT for Spot:
──────────────────
• Batch processing jobs (data analytics, ETL, video encoding)
• CI/CD build agents (Jenkins agents, GitHub Actions runners)
• Containerized workloads (Kubernetes pods, ECS tasks)
• Stateless web servers (behind a load balancer with multiple instances)
• Machine learning training (can checkpoint and resume)
• Big data (EMR, Spark clusters)
• Testing and development environments
• Queue-based workers (SQS consumers)

❌ BAD for Spot:
──────────────────
• Single-instance databases (if it's interrupted, you're down!)
• Stateful applications that can't handle interruptions
• Critical real-time services with no redundancy
• Long-running jobs that can't checkpoint/resume
• Applications with strict SLAs (99.99% uptime)
```

#### Option 3: EC2 Auto Scaling with Mixed Instances (Recommended)

The **best practice** is to use an Auto Scaling Group with a mix of On-Demand and Spot:

```bash
# This is the modern, recommended approach
# Terraform example below
```

```hcl
# Terraform — Auto Scaling Group with Mixed Instances (On-Demand + Spot)
resource "aws_autoscaling_group" "web_servers" {
  name                = "web-servers-asg"
  desired_capacity    = 10
  min_size            = 4     # Minimum On-Demand baseline
  max_size            = 20
  vpc_zone_identifier = [subnet_a, subnet_b, subnet_c]

  mixed_instances_policy {
    # Instance types to use (multiple for Spot diversification)
    launch_template {
      launch_template_specification {
        launch_template_id = aws_launch_template.web.id
        version            = "$Latest"
      }

      # Specify multiple instance types for Spot availability
      override {
        instance_type = "m5.xlarge"
      }
      override {
        instance_type = "m5a.xlarge"    # AMD variant (often cheaper)
      }
      override {
        instance_type = "m4.xlarge"     # Previous generation
      }
      override {
        instance_type = "m5d.xlarge"    # With local NVMe storage
      }
      override {
        instance_type = "c5.xlarge"     # Compute-optimized alternative
      }
    }

    instances_distribution {
      # On-Demand settings
      on_demand_base_capacity                  = 4    # First 4 instances are always On-Demand
      on_demand_percentage_above_base_capacity = 25   # 25% of additional capacity is On-Demand
      
      # Spot settings
      spot_allocation_strategy = "capacity-optimized"  # AWS picks the pool with most capacity
      spot_max_price           = ""                     # Empty = pay up to On-Demand price
    }
  }

  tag {
    key                 = "Name"
    value               = "web-server"
    propagate_at_launch = true
  }
  
  tag {
    key                 = "Team"
    value               = "platform"
    propagate_at_launch = true
  }
}

resource "aws_launch_template" "web" {
  name_prefix   = "web-server-"
  image_id      = "ami-0c55b159cbfafe1f0"
  
  # Don't specify instance_type here — it's in the ASG overrides
  
  user_data = base64encode(<<-EOF
    #!/bin/bash
    # Install your application
    yum update -y
    yum install -y httpd
    systemctl start httpd
    systemctl enable httpd
    
    # Set up Spot interruption handler
    # (monitors for interruption and drains connections gracefully)
    cat > /usr/local/bin/spot-handler.sh << 'SCRIPT'
    #!/bin/bash
    while true; do
      STATUS=$(curl -s -o /dev/null -w "%{http_code}" \
        http://169.254.169.254/latest/meta-data/spot/instance-action)
      if [ "$STATUS" -eq 200 ]; then
        echo "Spot interruption notice received! Draining..."
        # Deregister from load balancer
        INSTANCE_ID=$(curl -s http://169.254.169.254/latest/meta-data/instance-id)
        aws autoscaling set-instance-health \
          --instance-id $INSTANCE_ID \
          --health-status Unhealthy
        # Wait for connections to drain
        sleep 90
        exit 0
      fi
      sleep 5
    done
    SCRIPT
    chmod +x /usr/local/bin/spot-handler.sh
    nohup /usr/local/bin/spot-handler.sh &
  EOF
  )

  tag_specifications {
    resource_type = "instance"
    tags = {
      Name = "web-server"
    }
  }
}
```

**What this does:**

```
Auto Scaling Group: 10 instances desired

Instance allocation:
─────────────────────
First 4 instances:      On-Demand (guaranteed, no interruptions)
Remaining 6 instances:  75% Spot + 25% On-Demand
                        = ~4-5 Spot + ~1-2 On-Demand

If any Spot instance is interrupted:
  → ASG automatically launches a replacement
  → Tries different instance types if needed
  → Your capacity is maintained
```

### Spot Best Practices

#### 1. Diversify Instance Types

Don't use just one instance type. If that type's Spot pool runs out, you get interrupted:

```
❌ Bad:  Only m5.xlarge
✅ Good: m5.xlarge, m5a.xlarge, m4.xlarge, m5d.xlarge, c5.xlarge, r5.xlarge

More instance types = more Spot pools = fewer interruptions
```

#### 2. Diversify Across Availability Zones

```
❌ Bad:  All instances in us-east-1a
✅ Good: Spread across us-east-1a, us-east-1b, us-east-1c

Different AZs have different Spot capacity
```

#### 3. Use Capacity-Optimized Allocation Strategy

```
Allocation strategies:
──────────────────────
lowest-price:         Picks the cheapest pool → Higher interruption risk
capacity-optimized:   Picks the pool with most spare capacity → Fewer interruptions ✅
price-capacity-optimized: Balances price and capacity → Best default choice ✅
diversified:          Spreads evenly across pools
```

#### 4. Handle Interruptions Gracefully

```python
# Spot Interruption Handler (Python example for a worker service)
import requests
import signal
import sys
import time

METADATA_URL = "http://169.254.169.254/latest/meta-data/spot/instance-action"

def check_spot_interruption():
    """Check if this Spot Instance is being interrupted."""
    try:
        response = requests.get(METADATA_URL, timeout=2)
        if response.status_code == 200:
            action = response.json()
            print(f"⚠️ SPOT INTERRUPTION: {action['action']} at {action['time']}")
            return True
    except requests.exceptions.RequestException:
        pass  # Metadata not available or timeout
    return False

def graceful_shutdown(signum=None, frame=None):
    """Gracefully shut down the application."""
    print("🛑 Shutting down gracefully...")
    # 1. Stop accepting new work
    # 2. Finish current work (or checkpoint it)
    # 3. Save state if needed
    # 4. Deregister from load balancer / service discovery
    sys.exit(0)

# Register signal handlers
signal.signal(signal.SIGTERM, graceful_shutdown)
signal.signal(signal.SIGINT, graceful_shutdown)

# Main loop
while True:
    if check_spot_interruption():
        graceful_shutdown()
    
    # Do your work here
    process_next_job()
    
    time.sleep(5)  # Check every 5 seconds
```

## 🎯 Part 4: The Optimal Strategy — Combining All Three

The **best practice** is to combine On-Demand, Reserved/Savings Plans, and Spot in a layered approach:

```
Compute Strategy (The "Wedding Cake" Model):
════════════════════════════════════════════

                    ┌─────────────────┐
                    │     SPOT        │  ← Burst/fault-tolerant workloads
                    │   (up to 90%    │     (cheapest but interruptible)
                    │    discount)    │
                ┌───┴─────────────────┴───┐
                │      ON-DEMAND          │  ← Variable workloads
                │   (full price, but      │     (no commitment, full flexibility)
                │    no commitment)       │
            ┌───┴─────────────────────────┴───┐
            │     SAVINGS PLANS / RIs         │  ← Steady-state baseline
            │   (up to 72% discount)          │     (committed, guaranteed)
            │   (predictable, always-on       │
            │    workloads)                   │
            └─────────────────────────────────┘
```

### Real-World Example

```
Company: E-commerce platform
Total compute need: ~100 instances equivalent

Layer 1 — Savings Plans (50 instances worth):
  → These services run 24/7: API servers, databases, cache
  → Buy Compute Savings Plan: $15/hour commitment
  → Savings: ~60% on this layer
  → Monthly cost: ~$5,400 (instead of $13,500)

Layer 2 — On-Demand (20 instances):
  → Variable workloads: business-hours-only services, new services
  → No commitment needed — scale up and down
  → Monthly cost: ~$2,700

Layer 3 — Spot (30 instances):
  → Batch jobs, CI/CD, data processing, non-critical workers
  → Monthly cost: ~$900 (instead of $4,050 on-demand)
  → Handle interruptions with queues and checkpointing

Total WITHOUT optimization:   100 instances × $135/month = $13,500/month
Total WITH optimization:      $5,400 + $2,700 + $900 = $9,000/month
Savings:                      $4,500/month = $54,000/year (33% total savings)
```

## 🎯 Key Takeaways

```
1. On-Demand is the most EXPENSIVE way to run in AWS — use it only when needed
2. Savings Plans are the EASIEST way to save — commit $/hour, get up to 72% off
3. Compute Savings Plans are most FLEXIBLE — cover EC2, Fargate, and Lambda
4. Spot Instances offer up to 90% savings — but only for fault-tolerant workloads
5. Diversify Spot across instance types and AZs to reduce interruptions
6. Use the "wedding cake" approach: Savings Plans base + On-Demand variable + Spot burst
7. Monitor utilization and coverage — unused commitments are wasted money
8. Start conservative with commitments (70-80% of baseline) — you can always add more
9. Auto Scaling Groups with mixed instances are the BEST way to use Spot
10. Always handle Spot interruptions gracefully — you only get 2 minutes!
```
