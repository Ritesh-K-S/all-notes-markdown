## 1. Why is Kubernetes Cost Optimization Hard?

Kubernetes is **amazing** at running containers — but it's also **amazing** at burning money if you're not careful. Here's why:

### The Visibility Problem

In traditional cloud (EC2, VMs), it's simple:
- 1 VM = 1 application = 1 bill line item

In Kubernetes, it's messy:
- 1 node runs **many pods** from **many teams**
- Pods from different namespaces share the same node
- You see the **node cost** on your cloud bill, but NOT the **pod cost**

> 🧩 Imagine a shared apartment. The electricity bill comes in for the whole flat. But who used the most power? The person mining crypto in their room or the one who just charged their phone? That's Kubernetes cost allocation in a nutshell.

### The Over-Provisioning Problem

| Resource | What Teams Request | What They Actually Use |
|---|---|---|
| CPU | 4 cores per pod | 200m (0.2 cores) per pod |
| Memory | 8Gi per pod | 512Mi per pod |
| Replicas | 10 | Could run fine with 3 |

Studies show that **Kubernetes clusters typically run at 20-30% utilization**, meaning **70-80% of what you pay for is wasted**.

### The Shared Cost Problem

Some costs are "shared" and hard to divide:
- **Control plane** (API server, etcd, scheduler)
- **System pods** (kube-dns, monitoring agents)
- **Networking** (load balancers, data transfer)
- **Storage** (persistent volumes)

## 2. Where Does K8s Money Go?

Let's break down a typical Kubernetes bill:

```
┌──────────────────────────────────────────────────┐
│           Kubernetes Cost Breakdown               │
├──────────────────────────────────────────────────┤
│  Compute (Nodes/VMs)          60-70%             │
│  ├── Worker Nodes                                │
│  ├── Control Plane (managed)                     │
│  └── GPU Nodes (if any)                          │
│                                                  │
│  Storage                      10-15%             │
│  ├── Persistent Volumes (EBS, Azure Disk)        │
│  ├── Container Image Storage (ECR, ACR)          │
│  └── Snapshots & Backups                         │
│                                                  │
│  Networking                   10-20%             │
│  ├── Load Balancers                              │
│  ├── NAT Gateway (AWS — this is EXPENSIVE)       │
│  ├── Cross-AZ Data Transfer                      │
│  └── Ingress/Egress Traffic                      │
│                                                  │
│  Management & Overhead        5-10%              │
│  ├── Managed K8s Fee (EKS=$73/mo, AKS=free)     │
│  ├── Monitoring & Logging                        │
│  └── DNS (Route 53, etc.)                        │
└──────────────────────────────────────────────────┘
```

### Surprise Costs in Kubernetes

1. **NAT Gateway (AWS):** Each GB through NAT costs $0.045. Containers pulling images or calling external APIs rack up huge NAT bills.
2. **Cross-AZ Traffic:** Pods talking to pods in different availability zones cost $0.01/GB each way.
3. **Load Balancers:** Each Kubernetes `Service type: LoadBalancer` creates a cloud load balancer (~$16-25/month each).
4. **Persistent Volume Snapshots:** Old snapshots pile up and cost money.

## 3. Cost Allocation in Kubernetes

Before you can optimize, you need to know **who's spending what**. This is called **cost allocation**.

### The Challenge

Your cloud bill shows: `3 x m5.xlarge = $420/month`

But you need to know:
- How much does **Team A's API** cost?
- How much does **Team B's database** cost?
- How much is **shared infrastructure**?

### Using Labels for Cost Allocation

The foundation of K8s cost allocation is **labels**. Always label your resources:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: payment-service
  labels:
    app: payment-service
    team: payments                  # Which team owns this
    environment: production         # prod, staging, dev
    cost-center: CC-1234           # Finance department code
    project: checkout-v2           # Which project
spec:
  replicas: 3
  template:
    metadata:
      labels:
        app: payment-service
        team: payments
        environment: production
        cost-center: CC-1234
    spec:
      containers:
      - name: payment-service
        image: payment:v2
        resources:
          requests:
            cpu: "250m"
            memory: "512Mi"
```

### Enforcing Labels with OPA/Kyverno

You can **force** all deployments to have cost-related labels:

```yaml
# Kyverno policy: require cost labels on all deployments
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: require-cost-labels
spec:
  validationFailureAction: Enforce
  rules:
  - name: check-cost-labels
    match:
      any:
      - resources:
          kinds:
          - Deployment
          - StatefulSet
          - DaemonSet
    validate:
      message: "All workloads must have 'team', 'environment', and 'cost-center' labels."
      pattern:
        metadata:
          labels:
            team: "?*"
            environment: "?*"
            cost-center: "?*"
```

Now if someone tries to deploy without these labels, Kubernetes will **reject** the deployment.

### How Cost Allocation Works

```
Node Cost = $200/month (m5.xlarge: 4 CPUs, 16GB RAM)

Pod A requests: 1 CPU, 4GB RAM  → Uses 25% CPU, 25% RAM → Cost share: ~$50
Pod B requests: 2 CPU, 8GB RAM  → Uses 50% CPU, 50% RAM → Cost share: ~$100
Pod C requests: 0.5 CPU, 2GB → Uses 12.5% CPU, 12.5% RAM → Cost share: ~$25
Shared/System: 0.5 CPU, 2GB → System pods                → Cost share: ~$25
```

This is exactly what tools like Kubecost and OpenCost calculate for you — **automatically**.

## 4. OpenCost — The Open-Source Standard

### What is OpenCost?

**OpenCost** is a **free, open-source** project (CNCF Sandbox) that provides real-time cost monitoring for Kubernetes. It's actually the **core engine** that powers Kubecost's free tier.

Think of OpenCost as the "Prometheus of cost" — it collects cost data and exposes it as metrics.

### How OpenCost Works

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  Cloud Bill  │────▶│   OpenCost   │────▶│  Prometheus  │
│  (Pricing)   │     │   (Engine)   │     │  (Storage)   │
└──────────────┘     └──────┬───────┘     └──────────────┘
                            │
                     ┌──────▼───────┐
                     │   Grafana    │
                     │ (Dashboard)  │
                     └──────────────┘
```

OpenCost combines:
1. **Cloud pricing data** (how much nodes, storage, etc. cost)
2. **Kubernetes resource usage** (from metrics-server / Prometheus)
3. = **Per-pod, per-namespace, per-label cost breakdown**

### Installing OpenCost

```bash
# Prerequisites: Prometheus must be running in your cluster

# Install OpenCost using Helm
helm repo add opencost https://opencost.github.io/opencost-helm-chart
helm repo update

helm install opencost opencost/opencost \
  --namespace opencost \
  --create-namespace \
  --set opencost.prometheus.internal.serviceName=prometheus-server \
  --set opencost.prometheus.internal.namespaceName=monitoring
```

### Accessing the OpenCost UI

```bash
# Open http://localhost:9090 in your browser
```

## 6. Right-Sizing K8s Workloads

This is the **#1 way to save money** in Kubernetes. Most pods request way more resources than they need.

### Step 1: Find Over-Provisioned Pods

```bash
# Check what pods are requesting vs using
kubectl top pods -n production --sort-by=cpu

# Compare with their requests
kubectl get pods -n production -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.spec.containers[0].resources.requests.cpu}{"\t"}{.spec.containers[0].resources.requests.memory}{"\n"}{end}'
```

### Step 2: Use VPA to Get Recommendations

```yaml
# Install VPA
# Clone the VPA repo
# git clone https://github.com/kubernetes/autoscaler.git
# cd autoscaler/vertical-pod-autoscaler
# ./hack/vpa-up.sh

# Create a VPA for your deployment (recommendation mode)
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: my-app-vpa
  namespace: production
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-app
  updatePolicy:
    updateMode: "Off"   # Just give recommendations, don't auto-change
```

```bash
# Check VPA recommendations after a few hours/days
kubectl describe vpa my-app-vpa -n production

# Output:
#   Recommendation:
#     Container Recommendations:
#       Container Name:  my-app
#       Lower Bound:
#         Cpu:     25m
#         Memory:  128Mi
#       Target:
#         Cpu:     80m        ← This is what your pod actually needs
#         Memory:  256Mi      ← This is what your pod actually needs
#       Upper Bound:
#         Cpu:     200m
#         Memory:  512Mi
```

### Step 3: Apply Right-Sized Resources

```yaml
# Before: Over-provisioned
resources:
  requests:
    cpu: "2"
    memory: "4Gi"
  limits:
    cpu: "4"
    memory: "8Gi"

# After: Right-sized (based on VPA + 20% headroom)
resources:
  requests:
    cpu: "100m"       # VPA target was 80m + 20% headroom
    memory: "300Mi"   # VPA target was 256Mi + ~20% headroom
  limits:
    cpu: "400m"       # 4x the request for burst capacity
    memory: "512Mi"   # Upper bound from VPA
```

### Automating with Goldilocks

```bash
# Install Goldilocks (creates VPA for every deployment automatically)
helm install goldilocks fairwinds-stable/goldilocks \
  --namespace goldilocks \
  --create-namespace

# Enable for a namespace
kubectl label ns production goldilocks.fairwinds.com/enabled=true

# Access the dashboard
kubectl port-forward -n goldilocks svc/goldilocks-dashboard 8080:80
# Open http://localhost:8080
```

Goldilocks shows you **before/after** resource recommendations for every single workload.

## 7. Node Optimization Strategies

### a) Use the Right Node Sizes

**Small nodes vs. large nodes:**

| Strategy | Pros | Cons |
|---|---|---|
| Many small nodes | Less waste per node, better fault tolerance | More overhead (kubelet, OS on each node) |
| Fewer large nodes | Less overhead, better bin-packing | One node failure impacts more pods |

**General guideline:** Use nodes that are 2-4x the size of your largest pod. This gives good bin-packing without excessive waste.

### b) Mix Node Types

```yaml
# Use a mix of instance families for different workloads
# EKS node groups example

# General purpose (for most workloads)
apiVersion: eksctl.io/v1alpha5
kind: ClusterConfig
metadata:
  name: my-cluster
  region: us-east-1
managedNodeGroups:
  - name: general
    instanceTypes: ["m5.xlarge", "m5a.xlarge", "m6i.xlarge"]
    minSize: 2
    maxSize: 10
    labels:
      workload-type: general

  # Compute optimized (for CPU-heavy workloads)
  - name: compute
    instanceTypes: ["c5.xlarge", "c5a.xlarge", "c6i.xlarge"]
    minSize: 0
    maxSize: 5
    labels:
      workload-type: compute-intensive

  # Memory optimized (for caches, in-memory databases)
  - name: memory
    instanceTypes: ["r5.xlarge", "r5a.xlarge", "r6i.xlarge"]
    minSize: 0
    maxSize: 3
    labels:
      workload-type: memory-intensive
```

### c) Node Utilization Target

Aim for **60-80% utilization** across your nodes:
- Below 60% → you're wasting money
- Above 80% → you risk performance issues during spikes

```bash
# Check node utilization
kubectl top nodes

# Example output:
# NAME             CPU(cores)   CPU%   MEMORY(bytes)   MEMORY%
# node-1           1200m        30%    4096Mi          25%     ← Under-utilized!
# node-2           3500m        87%    12288Mi         75%     ← Good
# node-3           800m         20%    2048Mi          12%     ← Wasting money!
```

## 8. Namespace-Level Cost Management

### Setting Resource Quotas

Prevent teams from over-provisioning by setting **namespace-level quotas**:

```yaml
# Resource quota for the development namespace
apiVersion: v1
kind: ResourceQuota
metadata:
  name: dev-quota
  namespace: development
spec:
  hard:
    requests.cpu: "10"           # Total CPU requests: max 10 cores
    requests.memory: "20Gi"      # Total memory requests: max 20GB
    limits.cpu: "20"             # Total CPU limits: max 20 cores
    limits.memory: "40Gi"        # Total memory limits: max 40GB
    pods: "50"                   # Max 50 pods
    persistentvolumeclaims: "10" # Max 10 PVCs
    services.loadbalancers: "2"  # Max 2 load balancers (they're expensive!)
```

### Setting LimitRange Defaults

Ensure every pod has resource requests (prevents unlimited resource consumption):

```yaml
# Default resource limits for pods in this namespace
apiVersion: v1
kind: LimitRange
metadata:
  name: default-limits
  namespace: development
spec:
  limits:
  - default:               # Default limits (if not specified)
      cpu: "500m"
      memory: "512Mi"
    defaultRequest:         # Default requests (if not specified)
      cpu: "100m"
      memory: "128Mi"
    max:                    # Maximum allowed
      cpu: "2"
      memory: "4Gi"
    min:                    # Minimum allowed
      cpu: "50m"
      memory: "64Mi"
    type: Container
```

## 9. Spot / Preemptible Nodes for Kubernetes

Spot instances (AWS) / Preemptible VMs (GCP) / Spot VMs (Azure) are **60-90% cheaper** than on-demand instances. They can be reclaimed by the cloud provider with short notice.

### When to Use Spot Nodes

| ✅ Good for Spot | ❌ Bad for Spot |
|---|---|
| Stateless web apps | Databases (StatefulSets) |
| CI/CD build agents | Single-replica critical services |
| Batch processing | etcd nodes |
| Dev/test environments | Control plane components |
| Worker pods with >2 replicas | Services with long startup times |

### AWS EKS with Spot Instances

```yaml
# eksctl config with spot node groups
apiVersion: eksctl.io/v1alpha5
kind: ClusterConfig
metadata:
  name: my-cluster
  region: us-east-1
managedNodeGroups:
  # On-demand nodes for critical workloads
  - name: on-demand
    instanceTypes: ["m5.xlarge"]
    minSize: 2
    maxSize: 5
    labels:
      lifecycle: on-demand
    taints:
      - key: CriticalAddonsOnly
        value: "true"
        effect: PreferNoSchedule

  # Spot nodes for cost savings (70% cheaper!)
  - name: spot
    instanceTypes: ["m5.xlarge", "m5a.xlarge", "m5d.xlarge", "m4.xlarge"]
    minSize: 0
    maxSize: 20
    spot: true
    labels:
      lifecycle: spot
```

### Scheduling Pods on Spot Nodes

```yaml
# Schedule tolerant workloads on spot nodes
apiVersion: apps/v1
kind: Deployment
metadata:
  name: worker
spec:
  replicas: 5
  template:
    spec:
      affinity:
        nodeAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:
          - weight: 100
            preference:
              matchExpressions:
              - key: lifecycle
                operator: In
                values:
                - spot
      tolerations:
      - key: "spot"
        operator: "Equal"
        value: "true"
        effect: "NoSchedule"
      containers:
      - name: worker
        image: my-worker:v1
        resources:
          requests:
            cpu: "250m"
            memory: "512Mi"
```

### Handling Spot Interruptions

```yaml
# Use Pod Disruption Budgets to handle spot interruptions gracefully
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: worker-pdb
spec:
  minAvailable: 3       # Always keep at least 3 pods running
  selector:
    matchLabels:
      app: worker
```

## 10. Cluster Autoscaler & Karpenter

### Cluster Autoscaler

Cluster Autoscaler **adds or removes nodes** based on pending pods and node utilization.

```bash
# Install Cluster Autoscaler on EKS
helm repo add autoscaler https://kubernetes.github.io/autoscaler
helm install cluster-autoscaler autoscaler/cluster-autoscaler \
  --namespace kube-system \
  --set autoDiscovery.clusterName=my-cluster \
  --set awsRegion=us-east-1 \
  --set extraArgs.scale-down-utilization-threshold=0.5 \
  --set extraArgs.scale-down-delay-after-add=5m \
  --set extraArgs.scale-down-unneeded-time=5m
```

Key parameters for cost optimization:
- `scale-down-utilization-threshold=0.5` — Scale down nodes below 50% utilization
- `scale-down-delay-after-add=5m` — Wait 5 min after adding a node before considering scale-down
- `scale-down-unneeded-time=5m` — Remove unneeded nodes after 5 min

### Karpenter (AWS — Next-Gen Autoscaler)

**Karpenter** is AWS's replacement for Cluster Autoscaler. It's **faster, smarter, and cheaper** because it picks the optimal instance type for each workload.

```yaml
# Karpenter NodePool (v1)
apiVersion: karpenter.sh/v1
kind: NodePool
metadata:
  name: default
spec:
  template:
    spec:
      requirements:
      - key: karpenter.sh/capacity-type
        operator: In
        values: ["spot", "on-demand"]     # Prefer spot!
      - key: kubernetes.io/arch
        operator: In
        values: ["amd64"]
      - key: karpenter.k8s.aws/instance-category
        operator: In
        values: ["c", "m", "r"]           # Compute, general, memory families
      - key: karpenter.k8s.aws/instance-size
        operator: In
        values: ["large", "xlarge", "2xlarge"]
  limits:
    cpu: "100"                            # Max 100 CPUs total
    memory: "400Gi"                       # Max 400GB RAM total
  disruption:
    consolidationPolicy: WhenEmptyOrUnderutilized
    consolidateAfter: 1m                  # Consolidate underutilized nodes quickly
---
apiVersion: karpenter.k8s.aws/v1
kind: EC2NodeClass
metadata:
  name: default
spec:
  amiSelectorTerms:
  - alias: al2023@latest
  subnetSelectorTerms:
  - tags:
      karpenter.sh/discovery: my-cluster
  securityGroupSelectorTerms:
  - tags:
      karpenter.sh/discovery: my-cluster
```

**Why Karpenter saves more money:**
1. **Right-sizes nodes** — picks the cheapest instance that fits pending pods
2. **Consolidation** — automatically moves pods to fewer nodes and terminates empty ones
3. **Spot-first** — intelligently uses spot instances with automatic fallback to on-demand

## 11. Pod Scheduling Strategies for Cost

### a) Pod Topology Spread (Reduce Cross-AZ Traffic)

Cross-AZ traffic costs $0.01/GB on AWS. Keeping pods in the same AZ saves money:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-api
spec:
  replicas: 6
  template:
    spec:
      topologySpreadConstraints:
      - maxSkew: 1
        topologyKey: topology.kubernetes.io/zone
        whenUnsatisfiable: ScheduleAnyway
        labelSelector:
          matchLabels:
            app: my-api
      containers:
      - name: my-api
        image: my-api:v1
```

### b) Schedule Dev/Test Workloads on Cheap Nodes

```yaml
# Taint cheap (spot) nodes for dev workloads
kubectl taint nodes dev-node-1 environment=dev:NoSchedule

# Schedule dev workloads with matching toleration
apiVersion: apps/v1
kind: Deployment
metadata:
  name: dev-app
  namespace: development
spec:
  template:
    spec:
      tolerations:
      - key: "environment"
        operator: "Equal"
        value: "dev"
        effect: "NoSchedule"
      nodeSelector:
        lifecycle: spot
```

### c) Pod Priority — Protect Important Workloads

```yaml
# High priority for production
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: production-high
value: 1000000
globalDefault: false
description: "Production critical workloads"

---
# Low priority for batch jobs (can be evicted)
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: batch-low
value: 100
globalDefault: false
description: "Batch jobs - can be preempted"
```

## 12. Network & Storage Cost Optimization

### Network Cost Savings

```yaml
# 1. Use ClusterIP instead of LoadBalancer when possible
# BAD (costs $16-25/month per service):
apiVersion: v1
kind: Service
metadata:
  name: internal-api
spec:
  type: LoadBalancer    # ❌ Creates a cloud load balancer!
  ports:
  - port: 80

# GOOD (free):
apiVersion: v1
kind: Service
metadata:
  name: internal-api
spec:
  type: ClusterIP       # ✅ Internal only, no cloud LB created
  ports:
  - port: 80
```

```yaml
# 2. Use a single Ingress Controller instead of many LoadBalancer services
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: main-ingress
spec:
  rules:
  - host: api.example.com
    http:
      paths:
      - path: /payments
        pathType: Prefix
        backend:
          service:
            name: payment-service
            port:
              number: 80
      - path: /users
        pathType: Prefix
        backend:
          service:
            name: user-service
            port:
              number: 80
# One Ingress LB handles ALL services! Instead of one LB per service.
```

### Storage Cost Savings

```bash
# Find unused PVCs (Persistent Volume Claims)
kubectl get pvc --all-namespaces | grep -v Bound

# Find PVCs not attached to any pod
kubectl get pvc --all-namespaces -o json | jq -r '
  .items[] |
  select(.status.phase == "Bound") |
  "\(.metadata.namespace)/\(.metadata.name)"
' | while read pvc; do
  ns=$(echo $pvc | cut -d'/' -f1)
  name=$(echo $pvc | cut -d'/' -f2)
  pods=$(kubectl get pods -n $ns -o json | jq -r ".items[].spec.volumes[]?.persistentVolumeClaim.claimName" 2>/dev/null | grep "^${name}$")
  if [ -z "$pods" ]; then
    echo "UNUSED PVC: $pvc"
  fi
done
```

```yaml
# Use appropriate storage classes
# GP3 is 20% cheaper than GP2 on AWS
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: gp3
provisioner: ebs.csi.aws.com
parameters:
  type: gp3
  fsType: ext4
reclaimPolicy: Delete
volumeBindingMode: WaitForFirstConsumer
```

## 13. Idle Resource Detection

### Finding Abandoned Workloads

```bash
# Find deployments with 0 replicas
kubectl get deployments --all-namespaces -o json | jq -r '
  .items[] |
  select(.spec.replicas == 0) |
  "\(.metadata.namespace)/\(.metadata.name) - Replicas: 0"
'

# Find pods with no network traffic (requires Prometheus)
# PromQL query:
# sum(rate(container_network_receive_bytes_total{pod!=""}[7d])) by (pod, namespace)
# == 0

# Find services with no endpoints
kubectl get endpoints --all-namespaces -o json | jq -r '
  .items[] |
  select((.subsets == null) or (.subsets | length == 0)) |
  "\(.metadata.namespace)/\(.metadata.name) - NO ENDPOINTS"
'
```

### Schedule Dev/Test Shutdown

Save money by shutting down non-production workloads outside business hours:

```yaml
# Using a CronJob to scale down dev at night
apiVersion: batch/v1
kind: CronJob
metadata:
  name: scale-down-dev
  namespace: kube-system
spec:
  schedule: "0 20 * * 1-5"    # 8 PM Mon-Fri
  jobTemplate:
    spec:
      template:
        spec:
          serviceAccountName: scaler
          containers:
          - name: kubectl
            image: bitnami/kubectl:latest
            command:
            - /bin/sh
            - -c
            - |
              # Scale down all deployments in dev namespace
              for deploy in $(kubectl get deploy -n development -o name); do
                kubectl scale $deploy --replicas=0 -n development
              done
          restartPolicy: OnFailure

---
# Scale back up in the morning
apiVersion: batch/v1
kind: CronJob
metadata:
  name: scale-up-dev
  namespace: kube-system
spec:
  schedule: "0 8 * * 1-5"     # 8 AM Mon-Fri
  jobTemplate:
    spec:
      template:
        spec:
          serviceAccountName: scaler
          containers:
          - name: kubectl
            image: bitnami/kubectl:latest
            command:
            - /bin/sh
            - -c
            - |
              # Scale up deployments in dev namespace
              kubectl scale deploy --all --replicas=2 -n development
          restartPolicy: OnFailure
```

This alone can save **60%+ on dev/test environments** (running only 12 hours on weekdays = ~35% of the time).

## 14. Cost Governance & Policies

### Setting Up Cost Alerts with Kubecost

```yaml
# Kubecost alerts configuration (values.yaml)
kubecostModel:
  alerts:
    # Alert when a namespace exceeds budget
    - type: budget
      threshold: 5000
      window: 30d
      aggregation: namespace
      filter: production
      ownerContact:
        - owner: platform-team
          slackWebhook: "https://hooks.slack.com/xxx"
    
    # Alert when daily spend spikes
    - type: spendChange
      relativeThreshold: 0.5  # 50% increase
      window: 1d
      baselineWindow: 7d
      aggregation: cluster
      ownerContact:
        - owner: finops-team
          slackWebhook: "https://hooks.slack.com/yyy"
    
    # Alert on high efficiency loss
    - type: efficiency
      threshold: 0.4  # Alert when efficiency drops below 40%
      window: 48h
      aggregation: namespace
```
