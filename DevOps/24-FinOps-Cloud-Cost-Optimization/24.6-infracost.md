## 1. What is Infracost?

**Infracost** is a free, open-source tool that shows you the **cost of your infrastructure changes before you deploy them**.

Think of it like this:

> 🛒 When you shop online, you see the total price in your cart **before** you click "Buy Now." Infracost does the same for your cloud infrastructure — it shows you the price tag **before** you run `terraform apply`.

### The Problem It Solves

Without Infracost:
1. Developer writes Terraform code
2. `terraform plan` shows what resources will be created
3. `terraform apply` creates the resources
4. **30 days later...** surprise! The cloud bill is $5,000 higher than expected
5. "Who deployed that massive RDS instance?!" 😱

With Infracost:
1. Developer writes Terraform code
2. **Infracost shows:** "This change will cost **+$1,200/month**"
3. Developer thinks: "Wait, maybe I should use a smaller instance..."
4. Developer adjusts, bill stays under control ✅

### Key Features

| Feature | Description |
|---|---|
| **Cost Breakdown** | Shows monthly cost of each resource |
| **Cost Diff** | Shows cost difference between current and planned state |
| **PR Comments** | Posts cost estimate as a comment on pull requests |
| **Usage Estimation** | Estimates usage-based costs (data transfer, API calls) |
| **Cost Policies** | Set thresholds to block expensive changes |
| **Multi-Cloud** | Supports AWS, Azure, GCP |
| **CI/CD Integration** | Works with GitHub Actions, GitLab CI, Azure DevOps, etc. |
| **Free & Open Source** | Core CLI is completely free |

## 2. Why Do You Need IaC Cost Estimation?

### The Scary Truth About Cloud Bills

| Scenario | Without Infracost | With Infracost |
|---|---|---|
| Dev creates `db.r5.4xlarge` RDS | Nobody notices until bill arrives | PR comment: "+$2,400/month" |
| Someone adds 10 NAT Gateways | 10 × $32/month = hidden cost | Shows: "+$320/month for NAT" |
| Forgot to set storage to GP3 | Paying 20% more than needed | Shows cheaper alternative |
| PR changes instance from t3 to m5 | Reviewer doesn't check pricing | PR shows exact cost increase |

### Who Benefits?

- **Developers** — see the cost impact of their infra changes
- **Reviewers** — catch expensive changes during code review
- **FinOps teams** — enforce cost guardrails automatically
- **Management** — get early visibility into cost changes

## 3. How Infracost Works

```
┌──────────────────┐     ┌──────────────────┐     ┌──────────────────┐
│   Your Terraform │────▶│   Infracost CLI  │────▶│   Cost Output    │
│   Code (.tf)     │     │                  │     │   (Table/JSON)   │
└──────────────────┘     └────────┬─────────┘     └──────────────────┘
                                  │
                         ┌────────▼─────────┐
                         │  Cloud Pricing   │
                         │  API (Infracost  │
                         │  hosted / cached)│
                         └──────────────────┘
```

**Step by step:**

1. **Infracost reads** your Terraform files (or `terraform plan` output)
2. **Maps resources** to cloud pricing (e.g., `aws_instance "m5.xlarge"` → $0.192/hour)
3. **Calculates** monthly cost estimates
4. **Outputs** a nice table, JSON, or PR comment

### What It Supports

| Provider | Resources Covered |
|---|---|
| **AWS** | 260+ resource types (EC2, RDS, S3, Lambda, EKS, etc.) |
| **Azure** | 170+ resource types |
| **GCP** | 100+ resource types |

**Important:** Infracost reads your Terraform code — it does NOT need cloud credentials or access to your actual cloud account.

## 7. Infracost with Terraform Projects

### Multi-Directory Projects

If your Terraform code is split across multiple directories:

```bash
# Project structure:
# infrastructure/
# ├── networking/
# │   └── main.tf
# ├── compute/
# │   └── main.tf
# └── database/
#     └── main.tf

# Option 1: Run on each directory
infracost breakdown --path infrastructure/networking/
infracost breakdown --path infrastructure/compute/
infracost breakdown --path infrastructure/database/

# Option 2: Use a config file for all at once
```

### Infracost Config File

Create `infracost.yml` in your project root:

```yaml
# infracost.yml
version: 0.1

projects:
  - path: infrastructure/networking
    name: Networking

  - path: infrastructure/compute
    name: Compute
    terraform_var_files:
      - terraform.tfvars
      - production.tfvars

  - path: infrastructure/database
    name: Database
    terraform_vars:
      environment: production
      instance_class: db.r5.xlarge
```

```bash
# Run with config file
infracost breakdown --config-file infracost.yml
```

### Using with Terraform Workspaces

```bash
# Estimate cost for a specific workspace
infracost breakdown --path . --terraform-workspace production
infracost breakdown --path . --terraform-workspace staging
```

### Using with Terraform Variables

```bash
# Pass variables
infracost breakdown --path . \
  --terraform-var "instance_type=m5.xlarge" \
  --terraform-var "environment=production"

# Use var files
infracost breakdown --path . \
  --terraform-var-file production.tfvars

# Compare different configurations
infracost breakdown --path . --terraform-var "instance_type=m5.xlarge"
# vs
infracost breakdown --path . --terraform-var "instance_type=m5.large"
```

## 8. Cost Diff — Before vs. After

This is where Infracost really shines — showing you the **cost difference** when you change your infrastructure.

### How Cost Diff Works

```bash
# Step 1: Generate baseline (current state)
infracost breakdown --path . --format json --out-file baseline.json

# Step 2: Make your changes to the Terraform code
# (e.g., change instance type from m5.large to m5.xlarge)

# Step 3: Generate diff against baseline
infracost diff --path . --compare-to baseline.json
```

### Example: Upgrading an Instance

Let's say you change `instance_type = "m5.large"` to `instance_type = "m5.xlarge"`:

```bash
infracost diff --path . --compare-to baseline.json
```

**Output:**

```
Project: .

+ aws_instance.web
  +$70.08 ($70.08 → $140.16)

    ~ Instance usage (Linux/UNIX, on-demand, m5.large → m5.xlarge)
      +$70.08 ($70.08 → $140.16)

Monthly cost change: +$70.08
Percent change: +100%

──────────────────────────────────
Key: ~ changed, + added, - removed
```

**Instantly you see:** "This change will add $70.08/month to our bill."

### Example: Adding a New Resource

```bash
# You add a new RDS instance to your Terraform code
infracost diff --path . --compare-to baseline.json
```

```
Project: .

+ aws_db_instance.new_database
  +$838.80

    + Database instance (db.r5.xlarge, Multi-AZ)
      +$838.80 ($0.00 → $838.80)

    + Storage (gp3, 200 GB)
      +$46.00 ($0.00 → $46.00)

Monthly cost change: +$884.80
Percent change: +83%

──────────────────────────────────
```

The reviewer sees "+$884.80/month" and might ask: "Do we really need Multi-AZ for this database?"

## 9. Infracost in CI/CD Pipelines

### GitHub Actions

```yaml
# .github/workflows/infracost.yml
name: Infracost Cost Estimation

on:
  pull_request:
    paths:
      - '**.tf'
      - '**.tfvars'

jobs:
  infracost:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      pull-requests: write     # Needed to post PR comments
    
    env:
      INFRACOST_API_KEY: ${{ secrets.INFRACOST_API_KEY }}
    
    steps:
      - name: Checkout base branch
        uses: actions/checkout@v4
        with:
          ref: ${{ github.event.pull_request.base.ref }}

      - name: Setup Infracost
        uses: infracost/actions/setup@v3
        with:
          api-key: ${{ secrets.INFRACOST_API_KEY }}

      - name: Generate baseline cost
        run: |
          infracost breakdown --path . \
            --format json \
            --out-file /tmp/infracost-base.json

      - name: Checkout PR branch
        uses: actions/checkout@v4

      - name: Generate cost diff
        run: |
          infracost diff --path . \
            --compare-to /tmp/infracost-base.json \
            --format json \
            --out-file /tmp/infracost-diff.json

      - name: Post PR comment
        run: |
          infracost comment github \
            --path /tmp/infracost-diff.json \
            --repo ${{ github.repository }} \
            --pull-request ${{ github.event.pull_request.number }} \
            --github-token ${{ github.token }} \
            --behavior update
```

### GitLab CI

```yaml
# .gitlab-ci.yml
infracost:
  image: infracost/infracost:ci-latest
  stage: plan
  
  variables:
    INFRACOST_API_KEY: ${INFRACOST_API_KEY}
  
  script:
    # Generate baseline from main branch
    - git checkout $CI_MERGE_REQUEST_TARGET_BRANCH_NAME
    - infracost breakdown --path . --format json --out-file /tmp/base.json
    
    # Generate diff from MR branch
    - git checkout $CI_MERGE_REQUEST_SOURCE_BRANCH_NAME
    - infracost diff --path . --compare-to /tmp/base.json --format json --out-file /tmp/diff.json
    
    # Post comment on MR
    - infracost comment gitlab
        --path /tmp/diff.json
        --repo $CI_PROJECT_PATH
        --merge-request $CI_MERGE_REQUEST_IID
        --gitlab-token $GITLAB_TOKEN
        --behavior update
  
  rules:
    - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'
      changes:
        - "**/*.tf"
        - "**/*.tfvars"
```

### Azure DevOps Pipeline

```yaml
# azure-pipelines.yml
trigger: none

pr:
  branches:
    include:
      - main
  paths:
    include:
      - '**/*.tf'

pool:
  vmImage: 'ubuntu-latest'

steps:
  - task: InfracostSetup@2
    inputs:
      apiKey: $(INFRACOST_API_KEY)
      version: '0.10.x'

  - bash: |
      # Checkout base branch and generate baseline
      git checkout $(System.PullRequest.TargetBranch)
      infracost breakdown --path . --format json --out-file /tmp/base.json
      
      # Checkout PR branch and generate diff
      git checkout $(System.PullRequest.SourceBranch)
      infracost diff --path . --compare-to /tmp/base.json --format json --out-file /tmp/diff.json
      
      # Post comment on PR
      infracost comment azure-repos \
        --path /tmp/diff.json \
        --azure-access-token $(System.AccessToken) \
        --pull-request $(System.PullRequest.PullRequestId) \
        --repo-url $(Build.Repository.Uri) \
        --behavior update
    displayName: 'Run Infracost'
```

## 12. Usage-Based Cost Estimation

Some cloud resources have **usage-based pricing** (like data transfer, API calls, storage reads). Infracost can estimate these too with a **usage file**.

### Creating a Usage File

```yaml
# infracost-usage.yml
version: 0.1
resource_usage:

  # S3 bucket usage estimates
  aws_s3_bucket.data:
    standard:
      storage_gb: 500                    # 500 GB stored
      monthly_tier_1_requests: 1000000   # 1M PUT/POST requests
      monthly_tier_2_requests: 5000000   # 5M GET requests

  # NAT Gateway data transfer
  aws_nat_gateway.nat:
    monthly_data_processed_gb: 100       # 100 GB through NAT

  # Lambda function usage
  aws_lambda_function.processor:
    monthly_requests: 10000000           # 10M invocations
    request_duration_ms: 200             # 200ms per invocation

  # Data transfer (general)
  aws_instance.web:
    monthly_data_transfer_out_gb: 50     # 50 GB outbound

  # RDS backup storage
  aws_db_instance.database:
    additional_backup_storage_gb: 100    # 100 GB backup storage

  # CloudWatch logs
  aws_cloudwatch_log_group.app_logs:
    monthly_data_ingested_gb: 10         # 10 GB/month of logs
    monthly_data_scanned_gb: 5           # 5 GB/month queried
```

### Using the Usage File

```bash
# Include usage-based estimates
infracost breakdown --path . --usage-file infracost-usage.yml
```

### Output with Usage

```
 aws_s3_bucket.data
 └─ Standard
    ├─ Storage                                 500  GB                $11.50
    ├─ PUT, COPY, POST, LIST requests        1,000  1k requests       $5.00
    └─ GET, SELECT requests                  5,000  1k requests       $2.00

 aws_nat_gateway.nat
 ├─ NAT gateway                               730  hours             $32.85
 └─ Data processed                             100  GB                $4.50

 aws_lambda_function.processor
 ├─ Requests                            10,000,000  1M requests       $2.00
 └─ Duration (128 MB, 200ms)            10,000,000  GB-seconds       $16.67
```

Now you get a much more accurate cost estimate!

### Auto-Generating Usage Files

```bash

```

This creates a usage file with all the usage-based parameters pre-filled (with zeros). Just fill in your estimated numbers.

## 13. Cost Policies — Setting Guardrails

### Using Infracost with OPA (Open Policy Agent)

You can write policies to automatically **block** or **warn** about expensive changes:

```rego
# policy.rego — Block changes that increase monthly cost by more than $500

package infracost

deny[msg] {
    maxDiff = 500.0
    msg := sprintf(
        "Total monthly cost increase of $%.2f exceeds the maximum allowed ($%.2f). Please get FinOps approval.",
        [to_number(input.diffTotalMonthlyCost), maxDiff]
    )
    to_number(input.diffTotalMonthlyCost) > maxDiff
}
```

```rego
# Block expensive instance types
package infracost

deny[msg] {
    resource := input.projects[_].breakdown.resources[_]
    resource.resourceType == "aws_instance"
    
    contains(resource.metadata.calls[0].blockName, "m5.4xlarge")
    
    msg := sprintf(
        "Instance %s uses m5.4xlarge which is too expensive. Use m5.xlarge or smaller.",
        [resource.name]
    )
}
```

### Running Policy Checks in CI/CD

```bash
# Generate JSON output
infracost breakdown --path . --format json --out-file infracost.json

# Run OPA policy check
opa eval \
  --input infracost.json \
  --data policy.rego \
  --format raw \
  "data.infracost.deny"

# If any policies are violated, OPA returns the deny messages
# Your CI/CD pipeline can fail the build based on this
```

### Simple Bash-Based Guardrails

If you don't want to use OPA, here's a simpler approach:

```bash
#!/bin/bash
# cost-guardrail.sh — Fail CI if cost increase exceeds threshold

MAX_MONTHLY_INCREASE=500

# Get the cost diff
DIFF=$(infracost diff --path . \
  --compare-to baseline.json \
  --format json | jq -r '.diffTotalMonthlyCost')

# Compare with threshold
if (( $(echo "$DIFF > $MAX_MONTHLY_INCREASE" | bc -l) )); then
  echo "❌ BLOCKED: Monthly cost increase of \$$DIFF exceeds maximum of \$$MAX_MONTHLY_INCREASE"
  echo "Please get approval from the FinOps team before merging."
  exit 1
else
  echo "✅ Cost change (\$$DIFF/month) is within the allowed threshold."
  exit 0
fi
```

```yaml
# Use in GitHub Actions
- name: Check cost guardrails
  run: |
    chmod +x cost-guardrail.sh
    ./cost-guardrail.sh
```

## 15. VS Code Extension

Infracost has a **VS Code extension** that shows costs **right inside your editor** as you write Terraform code!

### Installing

1. Open VS Code
2. Go to Extensions (Ctrl+Shift+X)
3. Search for "Infracost"
4. Install the extension

### What It Does

When you open a `.tf` file, the extension shows:
- **Inline cost annotations** next to each resource
- **Hover tooltips** with detailed cost breakdown
- **CodeLens** showing monthly cost above each resource block

```hcl
# In your editor, you'll see something like:
# 💰 $140.16/month
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "m5.xlarge"     # 💰 $0.192/hour
  
  root_block_device {
    volume_size = 100             # 💰 $8.00/month
    volume_type = "gp3"
  }
}

# 💰 $884.80/month
resource "aws_db_instance" "database" {
  instance_class = "db.r5.xlarge"  # 💰 $838.80/month
  allocated_storage = 200          # 💰 $46.00/month
  multi_az = true                  # Doubles the instance cost!
}
```

This is incredibly useful — you see the cost impact **as you type**, not after you push.

## 16. Best Practices

### ✅ Do This

1. **Add Infracost to every PR pipeline** — make cost visibility automatic
2. **Use usage files** for accurate estimates (especially data transfer, S3, Lambda)
3. **Set cost guardrails** — block PRs that exceed a threshold
4. **Review cost comments in PRs** — make it part of code review culture
5. **Track costs over time** — use Infracost Cloud dashboard
6. **Combine with tagging** — ensure resources have cost-center tags
7. **Run on all environments** — staging and dev add up too!
8. **Update usage files quarterly** — traffic patterns change

### ❌ Avoid This

1. **Don't ignore the PR comment** — if Infracost says +$1,000, investigate
2. **Don't treat estimates as exact bills** — they're estimates, actual usage varies
3. **Don't set guardrails too tight initially** — start with warnings, then enforce
4. **Don't forget about usage-based costs** — S3, data transfer, and Lambda can be expensive
5. **Don't skip development environments** — "it's just dev" adds up across teams

### Infracost Workflow Summary

```
Developer writes Terraform → Push to PR → CI runs Infracost →
PR shows cost diff → Reviewer checks cost → Merge or adjust →
Apply to cloud → Actual bill matches estimate ✅
```
