## 🤔 Why Kubeflow? (The Problem It Solves)

So you've learned about ML pipelines in the previous section. You know you need to automate data ingestion, training, evaluation, and serving. But here's the question:

> **Where do you actually RUN all of this?**

Let's look at the challenges:

```
Running ML Workloads Without Kubeflow:
├── Training needs GPUs → How do you manage GPU clusters?
├── Different experiments need different environments → Dependency hell!
├── Need to run 100 experiments in parallel → How do you schedule them?
├── Models need to be served with auto-scaling → Who manages the infrastructure?
├── Data scientists use Jupyter Notebooks → How do they access the cluster?
├── Need reproducible pipelines → How do you define and version them?
└── All of this needs to be production-grade → That's a LOT of work!
```

**Enter Kubeflow.** It takes all these problems and solves them using Kubernetes — the same platform you already know for running containers.

Think of it this way:

```
Kubernetes = Operating system for containers
Kubeflow   = Machine learning platform running ON that operating system

Just like:
Linux runs on hardware → Apps run on Linux
Kubernetes runs on servers → Kubeflow runs on Kubernetes → ML runs on Kubeflow
```

## ✅ What Exactly is Kubeflow?

Kubeflow is NOT a single tool. It's a **collection of tools and components** that together form a complete ML platform on Kubernetes.

> **Kubeflow** = A composable, portable, scalable ML platform that leverages Kubernetes to run every stage of the ML lifecycle.

| Aspect | Description |
|--------|-------------|
| **Open Source** | Free, community-driven, backed by Google |
| **Kubernetes-Native** | Runs entirely on Kubernetes, uses K8s primitives |
| **Composable** | Use all components or just the ones you need |
| **Portable** | Works on any Kubernetes cluster (cloud or on-prem) |
| **End-to-End** | Covers the entire ML lifecycle |

### The Name "Kubeflow"

```
Kube  = Kubernetes
Flow  = ML workflow / pipeline

Kubeflow = ML workflows on Kubernetes
```

The project started at **Google** in 2017 when they wanted to make it easier to run TensorFlow on Kubernetes. It quickly grew to support all ML frameworks and the entire ML lifecycle.

### 2. Kubeflow Notebooks (Jupyter)

> Run Jupyter Notebooks directly on Kubernetes with GPU access.

This is huge for data scientists! Instead of running notebooks on their laptops, they can spin up a notebook server on the Kubernetes cluster with:
- Access to GPUs
- Shared data volumes
- Team-specific environments
- More compute power than a laptop

```yaml
# Notebook Server Custom Resource
apiVersion: kubeflow.org/v1
kind: Notebook
metadata:
  name: my-notebook
  namespace: ritesh-ns
spec:
  template:
    spec:
      containers:
        - name: notebook
          image: kubeflownotebookswg/jupyter-tensorflow-cuda:v1.8.0
          resources:
            requests:
              cpu: "2"
              memory: "8Gi"
              nvidia.com/gpu: "1"    # Request 1 GPU!
            limits:
              cpu: "4"
              memory: "16Gi"
              nvidia.com/gpu: "1"
          volumeMounts:
            - name: workspace
              mountPath: /home/jovyan
      volumes:
        - name: workspace
          persistentVolumeClaim:
            claimName: my-notebook-pvc
```

**What This Does:**
- Creates a Jupyter Notebook server as a Kubernetes Pod
- Gives it 2 CPUs, 8GB RAM, and 1 GPU
- Mounts a persistent volume so your work is saved even if the pod restarts
- Uses a pre-built image with TensorFlow and CUDA (GPU support)

**How Data Scientists Use It:**

```
1. Open Kubeflow Dashboard → Click "Notebooks"
2. Click "New Notebook Server"
3. Choose:
   - Name: "my-experiments"
   - Image: tensorflow-gpu or pytorch-gpu
   - CPU: 4 cores
   - Memory: 16 GB
   - GPUs: 1 × NVIDIA T4
   - Storage: 50 GB
4. Click "Launch" → Notebook spins up in ~30 seconds
5. Click "Connect" → Opens Jupyter in the browser
6. Start coding! 🎉
```

### 3. Kubeflow Pipelines (KFP) ⭐ (The Heart of Kubeflow)

> Define, run, and manage end-to-end ML workflows as code.

Kubeflow Pipelines is the **most important and most used** component. It lets you define your entire ML workflow as a **Python-defined DAG (Directed Acyclic Graph)** where each step runs in its own container.

**Key Concepts:**

```
Kubeflow Pipelines Concepts:
├── Pipeline: The entire workflow definition (DAG of steps)
├── Component: A single step in the pipeline (runs in a container)
├── Run: One execution of a pipeline
├── Experiment: A group of related runs
├── Artifact: Output of a step (data, model, metrics)
└── Pipeline Parameters: Inputs you can change per run
```

**Example: Building a Kubeflow Pipeline (KFP v2)**

```python
# churn_pipeline.py — A complete ML pipeline using Kubeflow Pipelines SDK v2

from kfp import dsl
from kfp import compiler

# ============================================================
# Step 1: Define Pipeline Components (each runs in its own container)
# ============================================================

@dsl.component(
    base_image="python:3.11",
    packages_to_install=["pandas", "sqlalchemy", "psycopg2-binary"]
)
def ingest_data(
    db_connection: str,
    output_path: dsl.OutputPath("csv")
):
    """Extract data from the database."""
    import pandas as pd
    from sqlalchemy import create_engine
    
    engine = create_engine(db_connection)
    df = pd.read_sql("SELECT * FROM customers WHERE active = true", engine)
    df.to_csv(output_path, index=False)
    print(f"Ingested {len(df)} rows")


@dsl.component(
    base_image="python:3.11",
    packages_to_install=["pandas", "great_expectations"]
)
def validate_data(
    input_path: dsl.InputPath("csv"),
    output_path: dsl.OutputPath("csv")
):
    """Validate data quality."""
    import pandas as pd
    
    df = pd.read_csv(input_path)
    
    # Basic validation checks
    assert len(df) > 100, f"Too few rows: {len(df)}"
    assert df["customer_id"].nunique() == len(df), "Duplicate customer IDs!"
    assert df["age"].between(0, 150).all(), "Invalid age values!"
    
    null_pct = df.isnull().mean().max()
    assert null_pct < 0.1, f"Too many nulls: {null_pct:.1%}"
    
    df.to_csv(output_path, index=False)
    print(f"Validation passed! {len(df)} rows OK")


@dsl.component(
    base_image="python:3.11",
    packages_to_install=["pandas", "scikit-learn"]
)
def prepare_features(
    input_path: dsl.InputPath("csv"),
    train_path: dsl.OutputPath("csv"),
    test_path: dsl.OutputPath("csv")
):
    """Engineer features and split data."""
    import pandas as pd
    from sklearn.model_selection import train_test_split
    
    df = pd.read_csv(input_path)
    
    # Feature engineering
    df["tenure_years"] = df["tenure_months"] / 12
    df["charge_per_month_ratio"] = df["total_charges"] / (df["tenure_months"] + 1)
    
    # Split
    train_df, test_df = train_test_split(df, test_size=0.2, random_state=42)
    
    train_df.to_csv(train_path, index=False)
    test_df.to_csv(test_path, index=False)
    print(f"Train: {len(train_df)}, Test: {len(test_df)}")


@dsl.component(
    base_image="python:3.11",
    packages_to_install=["pandas", "scikit-learn", "mlflow", "joblib"]
)
def train_model(
    train_path: dsl.InputPath("csv"),
    n_estimators: int,
    max_depth: int,
    model_path: dsl.OutputPath("joblib")
):
    """Train a Random Forest model."""
    import pandas as pd
    from sklearn.ensemble import RandomForestClassifier
    import joblib
    
    df = pd.read_csv(train_path)
    feature_cols = ["age", "tenure_months", "monthly_charges", "total_charges",
                    "num_support_tickets", "tenure_years", "charge_per_month_ratio"]
    
    X = df[feature_cols]
    y = df["churned"]
    
    model = RandomForestClassifier(
        n_estimators=n_estimators,
        max_depth=max_depth,
        random_state=42
    )
    model.fit(X, y)
    
    joblib.dump(model, model_path)
    print(f"Model trained with {n_estimators} estimators, depth={max_depth}")


@dsl.component(
    base_image="python:3.11",
    packages_to_install=["pandas", "scikit-learn", "joblib"]
)
def evaluate_model(
    model_path: dsl.InputPath("joblib"),
    test_path: dsl.InputPath("csv"),
    min_accuracy: float
) -> float:
    """Evaluate model and return accuracy."""
    import pandas as pd
    from sklearn.metrics import accuracy_score, f1_score
    import joblib
    
    model = joblib.load(model_path)
    df = pd.read_csv(test_path)
    
    feature_cols = ["age", "tenure_months", "monthly_charges", "total_charges",
                    "num_support_tickets", "tenure_years", "charge_per_month_ratio"]
    
    X = df[feature_cols]
    y = df["churned"]
    
    predictions = model.predict(X)
    accuracy = accuracy_score(y, predictions)
    f1 = f1_score(y, predictions)
    
    print(f"Accuracy: {accuracy:.4f}")
    print(f"F1 Score: {f1:.4f}")
    
    if accuracy < min_accuracy:
        raise ValueError(f"Model accuracy {accuracy:.4f} below threshold {min_accuracy}")
    
    return accuracy


# ============================================================
# Step 2: Define the Pipeline (how components connect)
# ============================================================

@dsl.pipeline(
    name="Churn Prediction Pipeline",
    description="End-to-end pipeline for customer churn prediction"
)
def churn_pipeline(
    db_connection: str = "postgresql://user:pass@db:5432/customers",
    n_estimators: int = 200,
    max_depth: int = 15,
    min_accuracy: float = 0.90
):
    # Step 1: Ingest data
    ingest_task = ingest_data(db_connection=db_connection)
    
    # Step 2: Validate data
    validate_task = validate_data(input_path=ingest_task.outputs["output_path"])
    
    # Step 3: Prepare features
    prepare_task = prepare_features(input_path=validate_task.outputs["output_path"])
    
    # Step 4: Train model
    train_task = train_model(
        train_path=prepare_task.outputs["train_path"],
        n_estimators=n_estimators,
        max_depth=max_depth
    )
    
    # Step 5: Evaluate model
    evaluate_task = evaluate_model(
        model_path=train_task.outputs["model_path"],
        test_path=prepare_task.outputs["test_path"],
        min_accuracy=min_accuracy
    )


# ============================================================
# Step 3: Compile the pipeline to YAML
# ============================================================

if __name__ == "__main__":
    compiler.Compiler().compile(
        pipeline_func=churn_pipeline,
        package_path="churn_pipeline.yaml"
    )
    print("Pipeline compiled to churn_pipeline.yaml")
```

**What Happens When You Run This Pipeline:**

```
Kubeflow Pipeline Execution:

1. ingest_data Pod spins up
   ├── Container: python:3.11
   ├── Installs: pandas, sqlalchemy
   ├── Runs: SQL query
   ├── Output: CSV file → saved to shared storage
   └── Pod terminates ✅

2. validate_data Pod spins up
   ├── Input: CSV from step 1
   ├── Runs: quality checks
   ├── Output: validated CSV
   └── Pod terminates ✅

3. prepare_features Pod spins up
   ├── Input: validated CSV from step 2
   ├── Runs: feature engineering + split
   ├── Output: train.csv + test.csv
   └── Pod terminates ✅

4. train_model Pod spins up
   ├── Input: train.csv from step 3
   ├── Runs: RandomForest training
   ├── Output: model.joblib
   └── Pod terminates ✅

5. evaluate_model Pod spins up
   ├── Input: model.joblib + test.csv
   ├── Runs: accuracy check
   ├── Output: accuracy score
   ├── If accuracy < threshold → FAIL ❌
   └── Pod terminates ✅
```

**Running the Pipeline:**

```bash
# Option 1: Compile and upload via CLI
pip install kfp

# Compile pipeline to YAML
python churn_pipeline.py
# Output: churn_pipeline.yaml

# Upload and run via CLI
kfp pipeline upload --pipeline-name "churn-prediction" churn_pipeline.yaml

# Create a run

  --pipeline-name "churn-prediction" \
  --experiment-name "churn-experiments" \
  --run-name "run-20250115" \
  --parameter db_connection="postgresql://user:pass@db:5432/customers" \
  --parameter n_estimators=300 \
  --parameter max_depth=20
```

```python

client = Client(host="https://kubeflow.your-cluster.com")

# Create an experiment
experiment = client.create_experiment(name="churn-experiments")

    pipeline_func=churn_pipeline,
    experiment_name="churn-experiments",
    run_name="run-20250115",
    arguments={
        "db_connection": "postgresql://user:pass@db:5432/customers",
        "n_estimators": 300,
        "max_depth": 20,

    }
)

print(f"Run submitted: {run.run_id}")
```

### 4. Training Operators

> Run distributed training jobs on Kubernetes for different ML frameworks.

Kubeflow provides **Custom Resource Definitions (CRDs)** for popular ML frameworks so you can run distributed training jobs natively on Kubernetes.

```
Training Operators:
├── TFJob     → Distributed TensorFlow training
├── PyTorchJob → Distributed PyTorch training
├── MXNetJob  → Distributed MXNet training
├── XGBoostJob → Distributed XGBoost training
├── MPIJob    → MPI-based distributed training (Horovod)
└── PaddleJob → PaddlePaddle training
```

**Example: PyTorch Distributed Training Job**

```yaml
# pytorch-training-job.yaml
apiVersion: kubeflow.org/v1
kind: PyTorchJob
metadata:
  name: churn-model-training
  namespace: ml-team
spec:
  pytorchReplicaSpecs:
    # Master node (coordinates training)
    Master:
      replicas: 1
      restartPolicy: OnFailure
      template:
        spec:
          containers:
            - name: pytorch
              image: my-registry/churn-trainer:v1.0
              command: ["python", "train.py"]
              args:
                - "--epochs=50"
                - "--batch-size=256"
                - "--learning-rate=0.001"
              resources:
                requests:
                  cpu: "4"
                  memory: "16Gi"
                  nvidia.com/gpu: "1"
                limits:
                  cpu: "8"
                  memory: "32Gi"
                  nvidia.com/gpu: "1"
              volumeMounts:
                - name: data
                  mountPath: /data
                - name: output
                  mountPath: /output
          volumes:
            - name: data
              persistentVolumeClaim:
                claimName: training-data-pvc
            - name: output
              persistentVolumeClaim:
                claimName: model-output-pvc
    
    # Worker nodes (do the heavy computation)
    Worker:
      replicas: 3  # 3 worker nodes for distributed training
      restartPolicy: OnFailure
      template:
        spec:
          containers:
            - name: pytorch
              image: my-registry/churn-trainer:v1.0
              command: ["python", "train.py"]
              args:
                - "--epochs=50"
                - "--batch-size=256"
                - "--learning-rate=0.001"
              resources:
                requests:
                  cpu: "4"
                  memory: "16Gi"
                  nvidia.com/gpu: "1"
                limits:
                  cpu: "8"
                  memory: "32Gi"
                  nvidia.com/gpu: "1"
              volumeMounts:
                - name: data
                  mountPath: /data
          volumes:
            - name: data
              persistentVolumeClaim:
                claimName: training-data-pvc
```

**What This Does:**

```
Creates 4 Pods on Kubernetes:
├── 1 Master Pod (coordinates)
├── 3 Worker Pods (compute)
│
├── Each Pod gets 1 GPU
├── Total: 4 GPUs working together
├── Data is shared via PVC
│
├── PyTorch's DistributedDataParallel (DDP) handles
│   splitting the work across GPUs automatically
│
└── When training completes, Pods terminate
    and model is saved to the output volume
```

### 5. Katib — Automated Hyperparameter Tuning (AutoML)

> Automatically find the best hyperparameters for your model.

Remember hyperparameter tuning from the previous section? Katib does this **automatically on Kubernetes**, running many experiments in parallel.

```
Katib Concepts:
├── Experiment: The overall tuning job
├── Trial: One specific combination of hyperparameters
├── Suggestion: Algorithm that proposes new hyperparameters to try
└── Objective: The metric to optimize (e.g., maximize accuracy)
```

```yaml
# katib-experiment.yaml
apiVersion: kubeflow.org/v1beta1
kind: Experiment
metadata:
  name: churn-hyperparam-tuning
  namespace: ml-team
spec:
  # What we want to optimize
  objective:
    type: maximize
    goal: 0.97                    # Stop when we reach 97% accuracy
    objectiveMetricName: accuracy
    additionalMetricNames:
      - f1-score
      - loss
  
  # Search algorithm
  algorithm:
    algorithmName: bayesianoptimization  # Smart search
    # Other options: random, grid, hyperband, cmaes, tpe
  
  # How many trials to run
  parallelTrialCount: 4    # Run 4 trials at the same time
  maxTrialCount: 30         # Try up to 30 combinations
  maxFailedTrialCount: 5    # Allow up to 5 failures
  
  # Hyperparameter search space
  parameters:
    - name: learning-rate
      parameterType: double
      feasibleSpace:
        min: "0.001"
        max: "0.1"
    
    - name: n-estimators
      parameterType: int
      feasibleSpace:
        min: "50"
        max: "500"
    
    - name: max-depth
      parameterType: int
      feasibleSpace:
        min: "3"
        max: "30"
    
    - name: batch-size
      parameterType: categorical
      feasibleSpace:
        list: ["32", "64", "128", "256"]
  
  # What to run for each trial
  trialTemplate:
    primaryContainerName: training
    trialParameters:
      - name: learningRate
        reference: learning-rate
      - name: nEstimators
        reference: n-estimators
      - name: maxDepth
        reference: max-depth
      - name: batchSize
        reference: batch-size
    trialSpec:
      apiVersion: batch/v1
      kind: Job
      spec:
        template:
          spec:
            containers:
              - name: training
                image: my-registry/churn-trainer:v1.0
                command:
                  - "python"
                  - "train.py"
                  - "--lr=${trialParameters.learningRate}"
                  - "--n-estimators=${trialParameters.nEstimators}"
                  - "--max-depth=${trialParameters.maxDepth}"
                  - "--batch-size=${trialParameters.batchSize}"
                resources:
                  requests:
                    cpu: "2"
                    memory: "4Gi"
                  limits:
                    cpu: "4"
                    memory: "8Gi"
            restartPolicy: Never
```

**How Katib Works:**

```
Katib Experiment: churn-hyperparam-tuning

Trial 1: lr=0.01, n_est=100, depth=10  → accuracy: 0.91 ❌
Trial 2: lr=0.05, n_est=200, depth=15  → accuracy: 0.93 ❌
Trial 3: lr=0.03, n_est=300, depth=20  → accuracy: 0.95 ✅ (best so far!)
Trial 4: lr=0.02, n_est=250, depth=18  → accuracy: 0.94 ❌
    ↓
Bayesian Optimization learns from results and suggests better parameters
    ↓
Trial 5: lr=0.028, n_est=320, depth=22 → accuracy: 0.96 ✅ (new best!)
Trial 6: lr=0.031, n_est=310, depth=21 → accuracy: 0.965 ✅ (even better!)
    ...
Trial 18: lr=0.029, n_est=315, depth=22 → accuracy: 0.971 ✅ (GOAL REACHED!)
    ↓
Experiment Complete! Best: lr=0.029, n_est=315, depth=22, accuracy=0.971
```

```bash
# Apply the experiment
kubectl apply -f katib-experiment.yaml

# Watch the experiment
kubectl get experiment churn-hyperparam-tuning -n ml-team -o wide

# Get the optimal parameters when done
kubectl get experiment churn-hyperparam-tuning -n ml-team -o jsonpath='{.status.currentOptimalTrial}'
```

### 6. KServe — Model Serving

> Deploy models as scalable, production-ready inference services on Kubernetes.

KServe (formerly KFServing) is Kubeflow's component for **serving models in production**. It supports autoscaling, canary deployments, and multiple ML frameworks.

```yaml
# kserve-inference-service.yaml
apiVersion: serving.kserve.io/v1beta1
kind: InferenceService
metadata:
  name: churn-predictor
  namespace: ml-serving
spec:
  predictor:
    # Option 1: Use a built-in model server
    sklearn:
      storageUri: "s3://models/churn/v2"
      resources:
        requests:
          cpu: "1"
          memory: "2Gi"
        limits:
          cpu: "2"
          memory: "4Gi"
    
    # Auto-scaling configuration
    minReplicas: 1       # Minimum pods (can scale to 0 too!)
    maxReplicas: 10      # Maximum pods
    scaleTarget: 10      # Target concurrent requests per pod
    scaleMetric: concurrency
```

```yaml
# Canary deployment — gradually shift traffic to new model
apiVersion: serving.kserve.io/v1beta1
kind: InferenceService
metadata:
  name: churn-predictor
  namespace: ml-serving
spec:
  predictor:
    # Current production model (90% traffic)
    sklearn:
      storageUri: "s3://models/churn/v2"
    canaryTrafficPercent: 10  # Send 10% to canary
  
  # Canary model (10% traffic) — new version being tested
  canaryPredictor:
    sklearn:
      storageUri: "s3://models/churn/v3"
```

```bash
# Deploy the model
kubectl apply -f kserve-inference-service.yaml

# Check status
kubectl get inferenceservice churn-predictor -n ml-serving

# Get the prediction URL
kubectl get inferenceservice churn-predictor -n ml-serving -o jsonpath='{.status.url}'
# Output: https://churn-predictor.ml-serving.example.com

# Make a prediction
curl -X POST \
  https://churn-predictor.ml-serving.example.com/v1/models/churn-predictor:predict \
  -H "Content-Type: application/json" \
  -d '{
    "instances": [
      [35, 24, 65.5, 1572.0, 3, 2.0, 65.5]
    ]
  }'

# Response:
# {"predictions": [0.87]}  → 87% chance of churning!
```

**KServe Features:**

```
KServe Capabilities:
├── Autoscaling (including scale-to-zero!)
├── Canary deployments (gradual traffic shifting)
├── A/B testing (compare model versions)
├── Multi-framework support (sklearn, TF, PyTorch, XGBoost, ONNX)
├── GPU support
├── Request batching (group multiple requests for efficiency)
├── Pre/Post processing (transformers)
├── Explainability (why did the model predict this?)
├── Model monitoring hooks
└── gRPC and REST API support
```

## 🛠️ Installing Kubeflow

There are several ways to install Kubeflow depending on your environment:

### Option 1: Local Development — Kubeflow on Kind/Minikube

```bash
# 1. Create a Kubernetes cluster with Kind
kind create cluster --name kubeflow --config - <<EOF
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
nodes:
  - role: control-plane
  - role: worker
  - role: worker
EOF

# 2. Install Kubeflow using manifests
git clone https://github.com/kubeflow/manifests.git
cd manifests

# 3. Install with kustomize (this takes a while)
while ! kustomize build example | kubectl apply -f -; do
  echo "Retrying to apply resources..."
  sleep 10
done

# 4. Wait for all pods to be ready
kubectl get pods -n kubeflow --watch

# 5. Access the dashboard
kubectl port-forward svc/istio-ingressgateway -n istio-system 8080:80

# Open browser: http://localhost:8080
# Default credentials: user@example.com / 12341234
```

### Option 2: Cloud Managed Kubeflow

```
Cloud-Managed Options:
├── AWS: Kubeflow on EKS (using AWS distribution)
├── GCP: Kubeflow on GKE (best supported, Google-backed)
├── Azure: Kubeflow on AKS
└── Any cloud: Use kubeflow manifests on managed K8s
```

```bash
# Example: Kubeflow on AWS EKS

# 1. Install prerequisites
pip install awscli eksctl

# 2. Create an EKS cluster
eksctl create cluster \
  --name kubeflow-cluster \
  --region us-west-2 \
  --nodegroup-name ml-workers \
  --node-type p3.2xlarge \   # GPU instance!
  --nodes 3 \
  --nodes-min 1 \
  --nodes-max 5

# 3. Install Kubeflow
export KUBEFLOW_RELEASE_VERSION=v1.8.0
export AWS_CLUSTER_NAME=kubeflow-cluster

git clone https://github.com/awslabs/kubeflow-manifests.git
cd kubeflow-manifests

make deploy-kubeflow INSTALLATION_OPTION=kustomize
```

### Option 3: Kubeflow Components Only (Pick What You Need)

You don't have to install ALL of Kubeflow. You can install individual components:

```bash
# Install ONLY Kubeflow Pipelines
export PIPELINE_VERSION=2.1.0

kubectl apply -k \
  "github.com/kubeflow/pipelines/manifests/kustomize/cluster-scoped-resources?ref=$PIPELINE_VERSION"

kubectl wait --for condition=established \
  --timeout=60s crd/applications.app.k8s.io

kubectl apply -k \
  "github.com/kubeflow/pipelines/manifests/kustomize/env/platform-agnostic?ref=$PIPELINE_VERSION"

# Install ONLY KServe
kubectl apply -f https://github.com/kserve/kserve/releases/download/v0.12.0/kserve.yaml
kubectl apply -f https://github.com/kserve/kserve/releases/download/v0.12.0/kserve-cluster-resources.yaml

# Install ONLY Katib
kubectl apply -k "github.com/kubeflow/katib/manifests/v1beta1/installs/katib-standalone?ref=v0.16.0"
```

## 🔄 End-to-End Kubeflow Workflow

Let's see how all the components work together in a real scenario:

```
End-to-End MLOps Workflow with Kubeflow:

1️⃣  Data Scientist opens Kubeflow Dashboard
    └── Launches a Jupyter Notebook (with GPU)
    └── Explores data, builds initial model prototype

2️⃣  Data Scientist creates a Kubeflow Pipeline
    └── Defines steps: ingest → validate → train → evaluate
    └── Compiles pipeline to YAML
    └── Uploads to Kubeflow Pipelines

3️⃣  Katib runs hyperparameter tuning
    └── Spawns 30 trial pods in parallel
    └── Each trial trains with different hyperparameters
    └── Finds the best configuration

4️⃣  Best model is trained with Training Operator
    └── PyTorchJob with 4 GPUs (distributed training)
    └── Uses the best hyperparameters from Katib
    └── Saves model to storage

5️⃣  Model is evaluated against quality gates
    └── Accuracy > 95%? ✅
    └── F1 Score > 0.90? ✅
    └── No bias detected? ✅
    └── Model approved for deployment

6️⃣  Model deployed with KServe
    └── InferenceService created
    └── Canary deployment: 10% traffic to new model
    └── Gradually increase to 100%
    └── Autoscaling configured

7️⃣  Pipeline runs automatically on schedule (weekly)
    └── New data triggers retraining
    └── New model compared to current production model
    └── If better → automatic promotion
    └── If worse → alert the team, keep current model
```

**Visual Flow:**

```
┌───────────┐    ┌───────────┐    ┌──────────┐    ┌──────────┐
│ Notebook  │───▶│ Pipeline  │───▶│  Katib   │───▶│ Training │
│ (Explore) │    │  (DAG)    │    │  (Tune)  │    │(Distrib.)│
└───────────┘    └───────────┘    └──────────┘    └──────────┘
                                                       │
                 ┌───────────┐    ┌──────────┐         │
                 │  Monitor  │◀───│  KServe  │◀────────┘
                 │  (Drift)  │    │  (Serve) │
                 └───────────┘    └──────────┘
                       │
                       ▼
                 Retrain if needed (loop back to Pipeline)
```

## ⚠️ Kubeflow Challenges & Considerations

Let's be honest about the challenges:

### 1. Complexity

```
Kubeflow requires knowledge of:
├── Kubernetes (pods, services, PVCs, RBAC, namespaces)
├── Docker (container images)
├── Istio (service mesh, used by Kubeflow for networking)
├── Knative (used by KServe for serverless)
├── ML frameworks (TensorFlow, PyTorch, etc.)
└── Pipeline SDKs (KFP Python SDK)

That's a LOT of things to know!
```

### 2. Resource Requirements

```
Minimum Cluster for Production Kubeflow:
├── Control Plane: 3 nodes (4 CPU, 16GB RAM each)
├── Worker Nodes: 3-10 nodes
├── GPU Nodes: 1-4 nodes (for training)
├── Storage: 500GB+ (for data, models, artifacts)
├── Istio overhead: ~2GB RAM
└── Total: Not cheap! Plan for significant infra costs
```

### 3. Installation & Upgrades

```
Pain Points:
├── Installation has many components and dependencies
├── Version compatibility between components
├── Upgrades can be breaking (especially across major versions)
├── Istio upgrades are particularly tricky
└── Documentation can be outdated or incomplete
```

### Best Practices to Manage Complexity:

```
1. Start small — Install only what you need (Pipelines + KServe)
2. Use managed distributions (AWS, GCP, or Arrikto distributions)
3. Have a dedicated platform team to manage Kubeflow
4. Invest in documentation and training for your ML team
5. Use GitOps (ArgoCD/Flux) to manage Kubeflow configuration
6. Start with a dev/staging environment before production
```
