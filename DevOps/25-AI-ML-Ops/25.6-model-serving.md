## 1. What is Model Serving?

You've trained a brilliant ML model. It has 95% accuracy. It's sitting in a Jupyter notebook. Now what?

**Model Serving** is the process of making your trained ML model **available for predictions** in a production environment — so real users, applications, and systems can send data to it and get predictions back.

### Simple Analogy:

Think of a restaurant:

| Restaurant | Model Serving |
|---|---|
| Chef (cooking) | Model training (data scientists) |
| Recipe (the dish) | Trained model |
| Kitchen | Training infrastructure (GPUs, notebooks) |
| **Serving the dish to customers** | **Model serving** |
| Waiter | API endpoint |
| Menu | API documentation |
| Multiple tables served simultaneously | Handling concurrent requests |
| Fast service | Low-latency predictions |

**Training a model is like cooking a dish. Serving a model is like running a restaurant.**

Training is done once (or periodically). Serving runs **24/7**, handling thousands or millions of requests.

### What Does a Model Server Do?

```
Client Request                  Model Server                    Response
─────────────                  ─────────────                    ────────
{                    ───▶     1. Receive request      ───▶    {
  "features": [              2. Validate input                 "prediction": 0.94,
    5.1, 3.5,                3. Preprocess data                "class": "fraud",
    1.4, 0.2                 4. Run inference                  "confidence": 0.94,
  ]                          5. Postprocess output             "latency_ms": 12
}                            6. Return response               }
```

A model server handles:

| Responsibility | What It Does |
|---|---|
| **Loading models** | Load model files (SavedModel, ONNX, PyTorch, etc.) into memory |
| **API endpoint** | Expose REST or gRPC endpoints for prediction requests |
| **Input validation** | Check that input data matches expected schema |
| **Preprocessing** | Transform raw input into model-ready format |
| **Inference** | Run the actual prediction (forward pass) |
| **Postprocessing** | Transform model output into user-friendly format |
| **Batching** | Group multiple requests together for efficient GPU use |
| **Scaling** | Handle traffic spikes by scaling horizontally |
| **Versioning** | Serve multiple model versions simultaneously |
| **Monitoring** | Track latency, throughput, errors, and data drift |

## 2. Model Serving Challenges

Serving models in production is **much harder** than training them. Here's why:

### 2.1 — Latency Requirements

```
Training:    "It took 3 hours to train? That's fine!"
Serving:     "The prediction took 200ms? TOO SLOW! Our SLA is 50ms!"
```

| Use Case | Latency Requirement |
|---|---|
| Fraud detection (payment processing) | < 50ms |
| Recommendation engine (web page) | < 100ms |
| Ad bidding (real-time) | < 10ms |
| Autonomous driving | < 5ms |
| Batch prediction (daily report) | Minutes to hours (OK) |

### 2.2 — Scalability

```
Low traffic:    10 requests/second → 1 server is fine
Normal:         1,000 requests/second → Need multiple replicas
Black Friday:   100,000 requests/second → Need autoscaling + load balancing
```

### 2.3 — Model Management

```
"We have 50 ML models in production."
"Model A needs version 2.3, Model B needs version 1.7"
"Roll back Model C to last week's version — it's making bad predictions"
"A/B test the new recommendation model with 10% of traffic"
```

### 2.4 — Hardware Optimization

```
CPU models:    Cheap, easy to scale, good for simple models
GPU models:    Expensive, but 10-100x faster for deep learning
               Need to batch requests to maximize GPU utilization
```

### 2.5 — The Complete Challenge Map

```
┌────────────────────────────────────────────────────────┐
│              Model Serving Challenges                   │
│                                                        │
│  Performance         Operations        ML-Specific     │
│  ───────────         ──────────        ───────────     │
│  • Low latency       • Monitoring      • A/B testing   │
│  • High throughput   • Auto-scaling    • Canary deploy │
│  • GPU utilization   • Health checks   • Model drift   │
│  • Request batching  • Logging         • Version mgmt  │
│  • Caching           • Alerting        • Rollbacks     │
│  • Memory mgmt       • CI/CD          • Multi-model    │
└────────────────────────────────────────────────────────┘
```

## 3. Model Serving Patterns

### 3.1 — Embedded Model (Simplest)

The model runs **inside** your application:

```python
# The model is just a Python object in your Flask/FastAPI app
from fastapi import FastAPI
import pickle

app = FastAPI()

# Load model at startup
with open("model.pkl", "rb") as f:
    model = pickle.load(f)

@app.post("/predict")
async def predict(features: list[float]):
    prediction = model.predict([features])
    return {"prediction": prediction[0]}
```

| Pros | Cons |
|---|---|
| Simple to implement | Model and app are tightly coupled |
| No extra infrastructure | Can't scale model independently |
| Low latency (in-process) | Hard to update model without redeploying app |
| | No specialized ML optimizations |

**Best for:** Simple models, prototypes, small-scale applications.

### 3.2 — Model as a Service (MaaS)

The model runs in a **separate service** with its own API:

```
┌──────────────┐        ┌──────────────┐
│  Application │──HTTP──│  Model       │
│  (FastAPI)   │  /gRPC │  Server      │
│              │───────▶│  (TF Serving │
│              │        │   KServe,    │
│              │◀───────│   Seldon)    │
└──────────────┘        └──────────────┘
```

| Pros | Cons |
|---|---|
| Independent scaling | Network latency overhead |
| Update model without touching app | More infrastructure to manage |
| Specialized ML optimizations | Slightly more complex |
| Multiple apps can share the model | |

**Best for:** Production deployments, microservices architecture.

### 3.3 — Batch Serving

Predictions are computed in **bulk** on a schedule:

```
┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
│  Data    │────▶│  Model   │────▶│  Results  │────▶│  Database│
│  (batch) │     │  (batch  │     │  (batch)  │     │  / Cache │
│          │     │  predict)│     │           │     │          │
└──────────┘     └──────────┘     └──────────┘     └──────────┘

Schedule: Run daily at 2 AM
```

```python
# Batch prediction script
import pandas as pd
import mlflow

# Load model
model = mlflow.pyfunc.load_model("models:/recommender/Production")

# Load all users
users = pd.read_parquet("s3://data/users.parquet")

# Predict for ALL users at once
predictions = model.predict(users)

# Save results
results = users.assign(recommendation=predictions)
results.to_parquet("s3://data/predictions/today.parquet")
```

| Pros | Cons |
|---|---|
| Simple, no real-time infra needed | Predictions are stale (not real-time) |
| Efficient (batch processing) | Doesn't handle new users immediately |
| Easy to debug and audit | Can't personalize based on current context |

**Best for:** Recommendations, daily reports, non-time-critical predictions.

### 3.4 — Streaming Serving

Predictions triggered by **events** in real-time:

```
┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
│  Event   │────▶│  Stream  │────▶│  Model   │────▶│  Action  │
│  (Kafka) │     │  Processor│    │  (predict)│     │  (alert, │
│          │     │  (Flink)  │     │          │     │  store)  │
└──────────┘     └──────────┘     └──────────┘     └──────────┘
```

**Best for:** Real-time fraud detection, IoT anomaly detection.

## 4. TensorFlow Serving

### 4.1 — What is TensorFlow Serving?

**TensorFlow Serving** is a **high-performance model serving system** built by Google specifically for serving TensorFlow models in production.

Think of it as a **production-grade waiter** that only serves TensorFlow dishes — it's extremely efficient and fast at what it does.

### Key Facts:

| Property | Detail |
|---|---|
| Created by | Google |
| Language | C++ (core), Python (client) |
| Model format | TensorFlow SavedModel |
| API | REST + gRPC |
| Focus | High performance, low latency |
| Best for | TensorFlow/Keras models in production |

### 4.2 — Architecture

```
│                  TensorFlow Serving                      │
│                                                          │
│  ┌──────────────┐     ┌──────────────┐                   │

│  │  (port 8501)  │     │  (port 8500)  │                  │
│  └──────┬───────┘     └──────┬───────┘                   │
│         │                    │                            │
│  ┌──────▼────────────────────▼───────┐                   │
│  │        Serving Core               │                   │
│  │                                   │                   │
```

### 4.3 — Saving a Model for TF Serving

First, you need to save your model in TensorFlow's **SavedModel** format:

```python
import tensorflow as tf
from tensorflow import keras

# Build and train a model
model = keras.Sequential([
    keras.layers.Dense(128, activation='relu', input_shape=(10,)),
    keras.layers.Dropout(0.3),
    keras.layers.Dense(64, activation='relu'),
    keras.layers.Dense(1, activation='sigmoid')
])

model.compile(optimizer='adam', loss='binary_crossentropy', metrics=['accuracy'])
model.fit(X_train, y_train, epochs=10, validation_split=0.2)

# Save in SavedModel format
# The directory structure MUST follow this convention:
# /models/<model_name>/<version_number>/
model.save("/models/fraud_detector/1")  # Version 1
```

The directory structure must look like this:

```
/models/
└── fraud_detector/
    ├── 1/                          ← Version 1
    │   ├── saved_model.pb          ← Model graph
    │   ├── fingerprint.pb          ← Model fingerprint
    │   └── variables/
    │       ├── variables.data-00000-of-00001
    │       └── variables.index
    └── 2/                          ← Version 2 (just add a new folder!)
        ├── saved_model.pb
        ├── fingerprint.pb
        └── variables/
            ├── variables.data-00000-of-00001
            └── variables.index
```

**Important:** The version numbers are just integers (1, 2, 3...). TF Serving automatically serves the **highest numbered version** by default.

### 4.4 — Running TF Serving with Docker

The easiest way to run TF Serving:

```bash
# Pull the TF Serving image
docker pull tensorflow/serving:latest

# Run TF Serving with your model
docker run -d \
  --name tf-serving \
  -p 8501:8501 \
  -p 8500:8500 \
  -v "/path/to/models:/models" \
  -e MODEL_NAME=fraud_detector \
  tensorflow/serving:latest

# What each flag means:
# -p 8501:8501  → REST API port
# -p 8500:8500  → gRPC API port
# -v            → Mount your model directory
# -e MODEL_NAME → Tell TF Serving which model to serve
```

### 4.5 — Making Predictions

**REST API:**

```bash
# Check model status
curl http://localhost:8501/v1/models/fraud_detector

# Response:
# {
#   "model_version_status": [{
#     "version": "2",
#     "state": "AVAILABLE",
#     "status": { "error_code": "OK" }
#   }]
# }

# Make a prediction (single example)
curl -X POST http://localhost:8501/v1/models/fraud_detector:predict \
  -H "Content-Type: application/json" \
  -d '{
    "instances": [[0.5, 1.2, 3.4, 0.1, 2.3, 0.7, 1.1, 0.3, 2.1, 0.9]]
  }'

# Response:
# {
#   "predictions": [[0.8723]]
# }

# Predict using a specific version
curl -X POST http://localhost:8501/v1/models/fraud_detector/versions/1:predict \
  -H "Content-Type: application/json" \
  -d '{
    "instances": [[0.5, 1.2, 3.4, 0.1, 2.3, 0.7, 1.1, 0.3, 2.1, 0.9]]
  }'
```

**gRPC (Python client):**

```python
import grpc
import numpy as np
import tensorflow as tf
from tensorflow_serving.apis import predict_pb2
from tensorflow_serving.apis import prediction_service_pb2_grpc

# Create gRPC channel
channel = grpc.insecure_channel('localhost:8500')
stub = prediction_service_pb2_grpc.PredictionServiceStub(channel)

# Create prediction request
request = predict_pb2.PredictRequest()
request.model_spec.name = 'fraud_detector'
request.model_spec.signature_name = 'serving_default'

# Add input data
input_data = np.array([[0.5, 1.2, 3.4, 0.1, 2.3, 0.7, 1.1, 0.3, 2.1, 0.9]])
request.inputs['input_1'].CopyFrom(
    tf.make_tensor_proto(input_data, dtype=tf.float32)
)

# Make prediction
response = stub.Predict(request, timeout=10.0)

# Get results
output = tf.make_ndarray(response.outputs['output_1'])
print(f"Prediction: {output}")
```

### 4.6 — Model Configuration (Serving Multiple Models)

Create a config file to serve multiple models:

```protobuf
# models.config
model_config_list {
  config {
    name: "fraud_detector"
    base_path: "/models/fraud_detector"
    model_platform: "tensorflow"
    model_version_policy {
      specific {
        versions: 1    # Serve version 1
        versions: 2    # AND version 2 (for A/B testing)
      }
    }
  }
  config {
    name: "recommender"
    base_path: "/models/recommender"
    model_platform: "tensorflow"
    model_version_policy {
      latest {
        num_versions: 1    # Only serve the latest version
      }
    }
  }
}
```

```bash
# Run with config file
docker run -d \
  -p 8501:8501 \
  -p 8500:8500 \
  -v "/path/to/models:/models" \
  -v "/path/to/models.config:/config/models.config" \
  tensorflow/serving:latest \
  --model_config_file=/config/models.config
```

### 4.7 — Request Batching

Batching groups multiple requests together for efficient GPU utilization:

```protobuf
# batching.config
max_batch_size { value: 32 }          # Max requests in a batch
batch_timeout_micros { value: 5000 }  # Wait up to 5ms for more requests
num_batch_threads { value: 4 }        # Number of batching threads
max_enqueued_batches { value: 100 }   # Max pending batches
```

```bash
# Run with batching enabled
docker run -d \
  -p 8501:8501 \
  -v "/path/to/models:/models" \
  -e MODEL_NAME=fraud_detector \
  tensorflow/serving:latest \
  --enable_batching=true \
  --batching_parameters_file=/config/batching.config
```

Why batching matters:

```
Without batching (GPU):                With batching (GPU):
Request 1 → predict → done            Requests 1-32 → predict → done
Request 2 → predict → done               (all at once on GPU)
Request 3 → predict → done
...
Request 32 → predict → done

Total time: 32 × 5ms = 160ms          Total time: 10ms + 5ms wait = 15ms
GPU utilization: ~3%                   GPU utilization: ~90%
```

## 5. Seldon Core

### 5.1 — What is Seldon Core?

**Seldon Core** is an **open-source platform** for deploying ML models on Kubernetes. It's **framework-agnostic** — meaning it works with any ML framework (TensorFlow, PyTorch, Scikit-learn, XGBoost, etc.).

Think of TF Serving as a **specialist** (only TensorFlow), while Seldon Core is a **generalist** (any framework, any language).

### Key Facts:

| Property | Detail |
|---|---|
| Created by | Seldon (UK company) |
| License | Business Source License (open source with restrictions) |
| Runs on | Kubernetes (required) |
| Model formats | Any (TF, PyTorch, sklearn, ONNX, custom) |
| Language support | Python, Java, Go, R, C++ |
| Key feature | Inference graphs (chain models together) |

### 5.2 — Architecture

```
┌──────────────────────────────────────────────────────────────┐
│                    Kubernetes Cluster                         │
│                                                              │
│  ┌────────────────────────────────────────────────────────┐  │
│  │              Seldon Core Operator                      │  │
│  │                                                        │  │
│  │  Watches for SeldonDeployment CRDs and creates:        │  │
│  │  - Pods (model containers)                             │  │
│  │  - Services (networking)                               │  │
│  │  - Istio VirtualServices (if using Istio)              │  │
│  │  - HPA (autoscaling)                                   │  │
│  └────────────────────────────────────────────────────────┘  │
│                                                              │
│  ┌────────────────────────────────────────────────────────┐  │

│  │  │  Predictor                                       │  │  │
```

### 5.4 — Creating a Model Server (Python)

Seldon uses a simple Python interface. You just need to implement a `predict` method:

```python
# Model.py — Your model wrapper

import pickle
import numpy as np
import logging

logger = logging.getLogger(__name__)

class FraudDetector:
    """
    A Seldon model wrapper for fraud detection.
    Seldon will call __init__() once at startup, then predict() for each request.
    """

    def __init__(self):
        """Called once when the model is loaded."""
        logger.info("Loading fraud detection model...")
        with open("model.pkl", "rb") as f:
            self.model = pickle.load(f)
        logger.info("Model loaded successfully!")

    def predict(self, X, features_names=None):
        """
        Called for every prediction request.
        
        Args:
            X: numpy array of shape (n_samples, n_features)
            features_names: optional list of feature names
        
        Returns:
            numpy array of predictions
        """
        logger.info(f"Received prediction request with shape: {X.shape}")
        predictions = self.model.predict_proba(X)
        return predictions

    def health_status(self):
        """Optional: Custom health check."""
        return {"status": "healthy", "model_loaded": self.model is not None}
```

## 6. KServe (formerly KFServing)

### 6.1 — What is KServe?

**KServe** is a **Kubernetes-native model serving platform** that provides a standard, cloud-native way to deploy ML models. It was originally part of Kubeflow (as KFServing) and is now a standalone project.

Think of KServe as the **"Kubernetes Ingress for ML models"** — just like Ingress standardizes how you expose web apps, KServe standardizes how you expose ML models.

### Key Facts:

| Property | Detail |
|---|---|
| Originally | KFServing (part of Kubeflow) |
| Current name | KServe |
| License | Apache 2.0 (fully open source) |
| Runs on | Kubernetes (with Knative or raw K8s) |
| Key features | Autoscaling (including scale-to-zero), multi-framework, canary deployments |
| Supported frameworks | TensorFlow, PyTorch, Scikit-learn, XGBoost, ONNX, Triton, HuggingFace, LightGBM |
| API standard | V2 Inference Protocol (Open Inference Protocol) |

### 6.2 — Why KServe?

| Problem | KServe Solution |
|---|---|
| "I need to deploy 50 models" | Standard CRD — deploy any model the same way |
| "Traffic varies — sometimes 0, sometimes 10K rps" | Autoscaling + scale-to-zero with Knative |
| "I need canary deployments for models" | Built-in traffic splitting |
| "Each model needs preprocessing" | Transformer + Predictor + Explainer pattern |
| "I need low-latency GPU inference" | NVIDIA Triton integration |
| "I want a standard API" | V2 Inference Protocol (works across frameworks) |

### 6.3 — Architecture


### 6.7 — Making Predictions

KServe uses the **V2 Inference Protocol** (also called Open Inference Protocol):

```bash
# Determine the Ingress URL
SERVICE_HOSTNAME=$(kubectl get inferenceservice sklearn-iris -n ml-models \
  -o jsonpath='{.status.url}' | cut -d "/" -f 3)

INGRESS_HOST=$(kubectl -n istio-system get service istio-ingressgateway \
  -o jsonpath='{.status.loadBalancer.ingress[0].ip}')

INGRESS_PORT=$(kubectl -n istio-system get service istio-ingressgateway \
  -o jsonpath='{.spec.ports[?(@.name=="http2")].port}')

# V1 Protocol — Simple JSON
curl -X POST "http://${INGRESS_HOST}:${INGRESS_PORT}/v1/models/sklearn-iris:predict" \
  -H "Host: ${SERVICE_HOSTNAME}" \
  -H "Content-Type: application/json" \
  -d '{
    "instances": [
      [6.8, 2.8, 4.8, 1.4],
      [6.0, 3.4, 4.5, 1.6]
    ]
  }'

# V2 Protocol — Open Inference Protocol (standard)
curl -X POST "http://${INGRESS_HOST}:${INGRESS_PORT}/v2/models/sklearn-iris/infer" \
  -H "Host: ${SERVICE_HOSTNAME}" \
  -H "Content-Type: application/json" \
  -d '{
    "inputs": [
      {
        "name": "input-0",
        "shape": [2, 4],
        "datatype": "FP32",
        "data": [
          [6.8, 2.8, 4.8, 1.4],
          [6.0, 3.4, 4.5, 1.6]
        ]
      }
    ]
  }'

# Response (V2):
# {
#   "model_name": "sklearn-iris",
#   "model_version": "v1",
#   "id": "abc-123",
#   "outputs": [
#     {
#       "name": "predict",
#       "shape": [2],
#       "datatype": "INT64",
#       "data": [1, 1]
#     }
#   ]
# }
```

### 6.8 — Autoscaling & Scale-to-Zero

This is KServe's killer feature when used with Knative:

```yaml
# autoscaling-example.yaml
apiVersion: serving.kserve.io/v1beta1
kind: InferenceService
metadata:
  name: fraud-detector
  annotations:
    # Autoscaling settings
    autoscaling.knative.dev/target: "10"              # Target 10 concurrent requests per pod
    autoscaling.knative.dev/minScale: "0"             # Scale to zero when idle!
    autoscaling.knative.dev/maxScale: "20"            # Max 20 pods
    autoscaling.knative.dev/scaleDownDelay: "5m"      # Wait 5 min before scaling down
spec:
  predictor:
    model:
      modelFormat:
        name: sklearn
      storageUri: "s3://models/fraud-detector"
      resources:
        requests:
          cpu: "1"
          memory: "2Gi"
```

What happens:

```
No traffic → 0 pods (scale to zero, save money!)
     ↓
First request arrives → 1 pod spins up (cold start: ~10-30 seconds)
     ↓
Traffic increases to 50 rps → 5 pods (autoscale up)
     ↓
Traffic spike to 200 rps → 20 pods (max scale)
     ↓
Traffic drops → Pods gradually scale down
     ↓
No traffic for 5 minutes → 0 pods again
```

The Transformer code:

```python
# transformer.py
import kserve
from typing import Dict
import numpy as np

class FraudPreprocessor(kserve.Model):
    def __init__(self, name: str, predictor_host: str):
        super().__init__(name)
        self.predictor_host = predictor_host
        self.ready = False

    def load(self):
        # Load any preprocessing artifacts (scalers, encoders, etc.)
        self.scaler = load_scaler("scaler.pkl")
        self.ready = True

    def preprocess(self, inputs: Dict, headers: Dict = None) -> Dict:
        """Transform raw input before sending to model."""
        instances = inputs["instances"]
        
        # Apply scaling
        scaled = self.scaler.transform(instances)
        
        # Add derived features
        enhanced = self.add_features(scaled)
        
        return {"instances": enhanced.tolist()}

    def postprocess(self, outputs: Dict, headers: Dict = None) -> Dict:
        """Transform model output before returning to client."""
        predictions = outputs["predictions"]
        
        # Convert to human-readable format
        results = []
        for pred in predictions:
            results.append({
                "fraud_probability": pred[1],
                "is_fraud": pred[1] > 0.5,
                "confidence": max(pred),
            })
        
        return {"results": results}

    def add_features(self, data):
        """Add derived features."""
        # Example: add interaction features
        return np.column_stack([
            data,
            data[:, 0] * data[:, 1],  # feature interaction
        ])
```

Flow:
```
Request → [Transformer (preprocess)] → [Predictor (model)] → [Transformer (postprocess)] → Response
```

### 6.11 — Custom Serving Runtime

```python
# custom_model.py
import kserve
from typing import Dict
import torch

class MyPyTorchModel(kserve.Model):
    def __init__(self, name: str):
        super().__init__(name)
        self.name = name
        self.ready = False
        self.model = None

    def load(self):
        """Load the model from storage."""
        self.model = torch.load("/mnt/models/model.pt")
        self.model.eval()
        self.ready = True

    def predict(self, payload: Dict, headers: Dict = None) -> Dict:
        """Run inference."""
        inputs = payload["instances"]
        tensor_input = torch.tensor(inputs, dtype=torch.float32)
        
        with torch.no_grad():
            outputs = self.model(tensor_input)
        
        return {"predictions": outputs.numpy().tolist()}


if __name__ == "__main__":
    model = MyPyTorchModel("my-pytorch-model")
    model.load()
    kserve.ModelServer().start([model])
```

## 7. TensorFlow Serving vs Seldon vs KServe — Comparison

| Feature | TF Serving | Seldon Core | KServe |
|---|---|---|---|
| **Framework Support** | TensorFlow only | Any framework | Any framework |
| **Language** | C++ (fast) | Python/Java/Go | Python |
| **Runs on** | Anywhere (Docker, K8s, bare metal) | Kubernetes only | Kubernetes only |
| **Autoscaling** | Manual (with K8s HPA) | HPA | Knative (scale-to-zero!) |
| **Canary/A/B testing** | Manual config | Built-in | Built-in |
| **Inference Graphs** | No | Yes (Seldon's superpower) | Transformer pattern |
| **Model Explainability** | No | Alibi (optional) | Built-in (Explainer) |
| **API Protocol** | Custom REST + gRPC | Custom REST + gRPC | V2 Inference Protocol (standard) |
| **Scale-to-Zero** | No | No | Yes (with Knative) |
| **GPU Support** | Excellent | Good | Good (Triton integration) |
| **Performance** | Excellent (C++) | Good (Python) | Good |
| **Learning Curve** | Low | Medium | Medium |
| **Community** | Large (Google) | Medium | Large (CNCF) |
| **License** | Apache 2.0 | BSL (see terms) | Apache 2.0 |
| **Best For** | TensorFlow in production | Complex ML pipelines | Kubernetes-native, multi-framework |

### Decision Guide:

```
"I only use TensorFlow"
  → TensorFlow Serving (fastest, simplest)

"I need complex inference pipelines (preprocessing → model → postprocessing)"
  → Seldon Core (inference graphs)

"I want Kubernetes-native with scale-to-zero and standard API"
  → KServe (most cloud-native)

"I need maximum performance with GPU"
  → NVIDIA Triton (via KServe or standalone)

"I want the simplest possible setup"
  → TensorFlow Serving (Docker, no K8s needed)

"I want open-source, no vendor lock-in"
  → KServe (Apache 2.0, CNCF project)
```

### End-to-End Request Flow:

```
1. Client sends request: POST /predict {features: [...]}
2. API Gateway validates API key, rate limits
3. Request reaches KServe/Seldon InferenceService
4. Transformer fetches features from Feast (if needed)
5. Transformer preprocesses input
6. Predictor loads model and runs inference
7. Postprocessor formats output
8. Response returned to client
9. Metrics logged to Prometheus
10. Predictions logged for monitoring (data drift detection)
```

### 10.1 — Model A/B Testing with Metrics

```yaml
# Full A/B test setup with KServe
```

### 10.2 — Multi-Model Serving

Serve multiple models from a single server (save resources):

```yaml
# KServe multi-model serving
apiVersion: serving.kserve.io/v1beta1
kind: InferenceService
metadata:
  name: multi-model-server
spec:
  predictor:
    model:
      modelFormat:
        name: sklearn
      storageUri: "s3://models/"
      # This loads ALL models from the directory
    # Enable multi-model serving
    annotations:
      serving.kserve.io/multiModelServer: "true"
```

## 11. Best Practices

### 11.1 — Always Use Health Checks

```yaml
# KServe handles this automatically, but for custom deployments:
readinessProbe:
  httpGet:
    path: /v2/health/ready
    port: 8080
  initialDelaySeconds: 10

livenessProbe:
  httpGet:
    path: /v2/health/live
    port: 8080
  initialDelaySeconds: 30
```

### 11.2 — Set Resource Limits

```yaml
resources:
  requests:
    cpu: "1"
    memory: "2Gi"
  limits:
    cpu: "2"
    memory: "4Gi"
```

Without limits, one model can starve other pods on the same node.

### 11.3 — Use Model Signatures

Always define input/output schemas so clients know exactly what to send:

```python
# With MLflow
from mlflow.models.signature import ModelSignature
from mlflow.types.schema import Schema, ColSpec

input_schema = Schema([
    ColSpec("double", "feature_1"),
    ColSpec("double", "feature_2"),
    ColSpec("integer", "feature_3"),
])
output_schema = Schema([ColSpec("double", "fraud_probability")])
signature = ModelSignature(inputs=input_schema, outputs=output_schema)
```

### 11.4 — Monitor Everything

Key metrics to monitor:

| Metric | Why |
|---|---|
| **Prediction latency (p50, p95, p99)** | SLA compliance |
| **Throughput (requests/sec)** | Capacity planning |
| **Error rate** | Model health |
| **Model version** | Track which version is serving |
| **Input data distribution** | Detect data drift |
| **Prediction distribution** | Detect concept drift |
| **GPU utilization** | Cost optimization |
| **Memory usage** | Prevent OOM crashes |

### 11.5 — Use Canary Deployments (Always!)

Never deploy a new model to 100% of traffic immediately:

```
Day 1: New model → 5% traffic    (monitor metrics)
Day 2: If good → 25% traffic     (monitor more)
Day 3: If good → 50% traffic     (compare performance)
Day 4: If good → 100% traffic    (full rollout)
       If bad  → 0% traffic      (instant rollback!)
```

### 11.6 — Keep Model Size Small

| Strategy | How |
|---|---|
| Quantization | Reduce precision (FP32 → FP16 → INT8) |
| Pruning | Remove unimportant weights |
| Knowledge distillation | Train a smaller model to mimic a large one |
| ONNX conversion | Optimize with ONNX Runtime |
| TensorRT | NVIDIA-specific optimization |

## 12. Summary

| Concept | Key Takeaway |
|---|---|
| **Model Serving** | Making trained models available for real-time or batch predictions |
| **Serving Patterns** | Embedded, Model-as-a-Service, Batch, Streaming |
| **TF Serving** | High-performance, C++, TensorFlow-only, simple setup |
| **Seldon Core** | Kubernetes-native, any framework, inference graphs, A/B testing |
| **KServe** | Kubernetes-native, any framework, scale-to-zero, canary deployments, V2 protocol |
| **NVIDIA Triton** | Maximum performance, GPU optimization, multi-framework |
| **V2 Inference Protocol** | Standard API across all serving frameworks |
| **Scale-to-Zero** | KServe + Knative can scale models to 0 when idle (save costs) |
| **Canary Deployments** | Gradually roll out new models with traffic splitting |
| **Model Monitoring** | Track latency, errors, data drift, prediction drift |
