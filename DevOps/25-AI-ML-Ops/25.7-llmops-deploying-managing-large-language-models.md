# 25.7 — LLMOps — Deploying & Managing Large Language Models

> **One Line Answer:** LLMOps is a specialized branch of MLOps focused on the **unique challenges** of deploying, serving, monitoring, and managing **Large Language Models** (like GPT, LLaMA, Mistral, etc.) in production — because LLMs are a completely different beast compared to traditional ML models.

---

## 🤔 Why Do We Need a Separate "LLMOps"? Can't We Just Use Regular MLOps?

Great question! Traditional MLOps was designed for models like:
- A fraud detection model (a few MBs)
- A recommendation engine (maybe 100 MB)
- An image classifier (a few hundred MBs)

But LLMs are **monsters** in comparison:

```
Traditional ML Model vs LLM — Size Comparison:

┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  Fraud Detection Model:      ~5 MB          🐜             │
│  Image Classifier (ResNet):  ~100 MB        🐕             │
│  BERT (NLP):                 ~440 MB        🐎             │
│  LLaMA 7B:                   ~13 GB         🐘             │
│  LLaMA 70B:                  ~130 GB        🦕             │
│  GPT-4 (estimated):          ~1.7 TB+       🌍             │
│                                                             │
│  That's a 300,000x difference from fraud model to GPT-4!   │
└─────────────────────────────────────────────────────────────┘
```

This massive scale difference means:
- **You can't just `pip install` and serve them** — you need GPUs, special infrastructure, and careful memory management.
- **Inference is expensive** — every API call costs real money (GPU time).
- **Latency matters** — users expect fast responses, but generating text token by token is slow.
- **Prompts are the new code** — managing prompts, versions, and guardrails is a whole new discipline.
- **Hallucinations are real** — LLMs make stuff up, and you need systems to catch that.

## 1️⃣ Model Selection — Choosing the Right LLM

The first big decision: **which model to use?**

### Option A: Use an API (Managed LLM)

You call someone else's model through an API.

| Provider | Models | Pros | Cons |
|----------|--------|------|------|
| **OpenAI** | GPT-4, GPT-4o, GPT-3.5 | Best quality, easy to start | Expensive, data leaves your infra |
| **Anthropic** | Claude 3.5, Claude 4 | Great at reasoning, long context | Expensive, vendor lock-in |
| **Google** | Gemini Pro, Gemini Ultra | Good multimodal, integrated with GCP | Still maturing |
| **AWS Bedrock** | Claude, Titan, LLaMA, Mistral | Multi-model, stays in AWS | AWS-specific |
| **Azure OpenAI** | GPT-4, GPT-3.5 | Enterprise compliance, Azure integration | Azure lock-in |

**When to use APIs:**
- You're getting started quickly
- You don't want to manage GPU infrastructure
- Data privacy isn't a critical concern
- You need the very best quality (GPT-4, Claude 4)

### Option B: Self-Host an Open-Source Model

You download and run the model on your own infrastructure.

| Model | Parameters | License | Good For |
|-------|-----------|---------|----------|
| **LLaMA 3** | 8B, 70B, 405B | Meta License | General purpose, very capable |
| **Mistral** | 7B, 8x7B (Mixtral) | Apache 2.0 | Efficient, great for smaller infra |
| **Falcon** | 7B, 40B, 180B | Apache 2.0 | Open, multilingual |
| **Phi-3** | 3.8B, 14B | MIT | Small but surprisingly good |
| **Qwen 2** | 7B, 72B | Apache 2.0 | Strong multilingual, coding |
| **CodeLlama** | 7B, 13B, 34B | Meta License | Code generation |
| **Gemma** | 2B, 7B | Google License | Lightweight, good for edge |

**When to self-host:**
- Data must stay in your infrastructure (privacy, compliance)
- You want to fine-tune the model on your data
- You want to avoid per-token API costs
- You need offline / air-gapped deployment

## 2️⃣ Customizing the Model — Fine-Tuning vs RAG

Once you've picked a model, you usually need to make it work better for **your specific use case**. There are two main approaches:

### Approach A: Fine-Tuning

**What is it?** You take a pre-trained model and train it further on **your own data** so it learns your domain, tone, and patterns.

```
Fine-Tuning Process:

  Pre-trained Model (general knowledge)
          │
          │  + Your Custom Training Data
          │    (e.g., 10,000 customer support conversations)
          │
          ▼
  Fine-Tuned Model (knows YOUR domain)
```

**When to fine-tune:**
- You need the model to follow a very specific output format
- You want it to learn specialized knowledge (medical, legal, financial)
- You need to change the model's behavior/tone consistently
- You have enough quality training data (at least hundreds of examples)

**Example: Fine-tuning with Hugging Face & LoRA (Low-Rank Adaptation):**

```python
# Fine-tuning a model using LoRA (Parameter-Efficient Fine-Tuning)
# LoRA is genius — instead of training ALL parameters (billions!),
# it only trains a tiny adapter layer (a few million parameters).

from transformers import AutoModelForCausalLM, AutoTokenizer
from peft import LoraConfig, get_peft_model, TaskType
from datasets import load_dataset

# Step 1: Load the base model
model_name = "meta-llama/Llama-3-8b"
model = AutoModelForCausalLM.from_pretrained(
    model_name,
    load_in_4bit=True,       # Quantize to 4-bit to save GPU memory
    device_map="auto"        # Automatically spread across available GPUs
)
tokenizer = AutoTokenizer.from_pretrained(model_name)

# Step 2: Configure LoRA
# Instead of training 8 billion parameters, we only train ~4 million!
lora_config = LoraConfig(
    task_type=TaskType.CAUSAL_LM,   # We're doing text generation
    r=16,                            # Rank of the adapter (lower = fewer params)
    lora_alpha=32,                   # Scaling factor
    lora_dropout=0.1,                # Dropout for regularization
    target_modules=[                 # Which layers to adapt
        "q_proj", "v_proj",          # Attention layers
        "k_proj", "o_proj"
    ]
)

# Step 3: Apply LoRA to the model
model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
# Output: trainable params: 4,194,304 || all params: 8,030,261,248
# That's only 0.05% of the total parameters! Super efficient.

# Step 4: Load your training data
# Format: instruction + input → output
dataset = load_dataset("json", data_files="my_training_data.jsonl")

# Step 5: Train!
from transformers import TrainingArguments, Trainer

training_args = TrainingArguments(
    output_dir="./fine-tuned-model",
    num_train_epochs=3,
    per_device_train_batch_size=4,
    gradient_accumulation_steps=4,    # Simulate larger batch size
    learning_rate=2e-4,
    fp16=True,                        # Use mixed precision for speed
    save_strategy="epoch",
    logging_steps=10,
)

trainer = Trainer(
    model=model,
    args=training_args,
    train_dataset=dataset["train"],
)

trainer.train()

# Step 6: Save the fine-tuned adapter (just a few MB!)
model.save_pretrained("./my-fine-tuned-adapter")
```

**What the code does:**
- **Line 1-3:** We import the necessary libraries — `transformers` for the model, `peft` for LoRA (efficient fine-tuning), and `datasets` for loading data.
- **Step 1:** We load a pre-trained LLaMA 3 8B model. `load_in_4bit=True` compresses the model to use less GPU memory. `device_map="auto"` automatically distributes the model across your GPUs.
- **Step 2:** LoRA config defines how we'll fine-tune. Instead of updating all 8B parameters, we add small trainable "adapter" layers. `r=16` means the adapter matrices are rank-16 (smaller = fewer params, faster training).
- **Step 3:** We wrap the model with LoRA. Now only 0.05% of parameters are trainable!
- **Step 4:** Load your custom data in JSONL format.
- **Step 5:** Standard training with Hugging Face Trainer. `fp16=True` uses half-precision floats for 2x speed.
- **Step 6:** Save just the LoRA adapter (a few MB), not the whole model.

### Approach B: RAG (Retrieval-Augmented Generation)

**What is it?** Instead of changing the model itself, you **give it relevant context** at query time by searching a knowledge base.

```
RAG Flow:

  User Question: "What's our refund policy?"
        │
        ▼
  ┌─────────────────┐
  │ Embed the query  │ ──→ Convert question to a vector
  └────────┬────────┘
           │
           ▼
  ┌─────────────────┐
  │ Search Vector DB │ ──→ Find similar documents
  │ (Pinecone/Weaviate│     from your knowledge base
  │  /ChromaDB)       │
  └────────┬────────┘
           │  Returns: "Our refund policy is 30 days..."
           ▼
  ┌─────────────────┐
  │ Build Prompt     │
  │ "Given this      │
  │  context: [docs] │
  │  Answer: [query]"│
  └────────┬────────┘
           │
           ▼
  ┌─────────────────┐
  │ LLM generates    │ ──→ "Based on our policy, you can
  │ response         │      get a refund within 30 days..."
  └─────────────────┘
```

**When to use RAG:**
- Your knowledge base changes frequently (docs, FAQs, product info)
- You don't have enough data or budget for fine-tuning
- You need the model to cite sources
- You want to avoid hallucinations (the model answers from your documents)

**Example: Simple RAG Pipeline with LangChain:**

```python
# A simple RAG pipeline using LangChain + ChromaDB

from langchain.document_loaders import DirectoryLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.embeddings import HuggingFaceEmbeddings
from langchain.vectorstores import Chroma
from langchain.llms import Ollama
from langchain.chains import RetrievalQA

# Step 1: Load your documents
# This loads all PDFs from a folder
loader = DirectoryLoader("./company_docs/", glob="**/*.pdf")
documents = loader.load()
print(f"Loaded {len(documents)} documents")

# Step 2: Split documents into chunks
# LLMs have a context window limit, so we split big docs into small pieces
splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,       # Each chunk is ~1000 characters
    chunk_overlap=200,     # Overlap so we don't lose context at boundaries
)
chunks = splitter.split_documents(documents)
print(f"Created {len(chunks)} chunks")

# Step 3: Create embeddings and store in a vector database
# Embeddings = convert text to numbers (vectors) that capture meaning
embeddings = HuggingFaceEmbeddings(
    model_name="sentence-transformers/all-MiniLM-L6-v2"
)
vector_store = Chroma.from_documents(
    documents=chunks,
    embedding=embeddings,
    persist_directory="./chroma_db"
)

# Step 4: Create a retrieval chain
# This combines: search vector DB → build prompt → call LLM
llm = Ollama(model="llama3")  # Using a local LLM via Ollama
qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    chain_type="stuff",           # "stuff" = put all retrieved docs into prompt
    retriever=vector_store.as_retriever(
        search_kwargs={"k": 4}    # Retrieve top 4 most relevant chunks
    ),
    return_source_documents=True   # Show which documents were used
)

# Step 5: Ask a question!
result = qa_chain({"query": "What is our refund policy?"})
print(result["result"])
print("Sources:", [doc.metadata["source"] for doc in result["source_documents"]])
```

**What the code does:**
- **Step 1:** Loads all PDF documents from a folder. These are your company's knowledge base (policies, docs, manuals).
- **Step 2:** Splits the documents into smaller chunks of ~1000 characters. This is needed because LLMs can only process a limited amount of text at once (context window).
- **Step 3:** Converts each chunk into a mathematical vector (embedding) and stores it in ChromaDB, a vector database. Similar texts will have similar vectors.
- **Step 4:** Creates a "chain" that connects everything: when you ask a question, it searches the vector DB for relevant chunks, stuffs them into a prompt, and sends that to the LLM.
- **Step 5:** Ask your question and get an answer that's grounded in your actual documents!

### Fine-Tuning vs RAG — Quick Comparison

```
┌──────────────────┬──────────────────────┬──────────────────────┐
│ Aspect           │ Fine-Tuning          │ RAG                  │
├──────────────────┼──────────────────────┼──────────────────────┤
│ Changes model?   │ Yes                  │ No                   │
│ Needs GPU?       │ Yes (for training)   │ No (just inference)  │
│ Knowledge update │ Retrain the model    │ Update the vector DB │
│ Cost             │ High (training)      │ Low (just retrieval) │
│ Best for         │ Style, format, tone  │ Facts, docs, FAQs    │
│ Hallucination    │ Can still hallucinate│ Much less (grounded) │
│ Setup complexity │ High                 │ Medium               │
│ Time to deploy   │ Days to weeks        │ Hours to days        │
└──────────────────┴──────────────────────┴──────────────────────┘

Pro tip: You can COMBINE both!
Fine-tune for style + RAG for knowledge = Best of both worlds 🎯
```

## 4️⃣ LLM Evaluation — How Do You Know If Your LLM Is Good?

This is one of the **hardest problems** in LLMOps. Unlike traditional ML (where you can calculate accuracy, precision, recall), LLM outputs are free-form text. How do you evaluate "good"?

### Evaluation Approaches

```
LLM Evaluation Methods:

┌─────────────────────────────────────────────────────┐
│                                                     │
│  1. AUTOMATED METRICS                               │
│     - BLEU, ROUGE (text similarity scores)          │
│     - Perplexity (how "surprised" the model is)     │
│     - Exact match (for structured outputs)          │
│                                                     │
│  2. LLM-AS-A-JUDGE                                  │
│     - Use a strong LLM (GPT-4) to evaluate          │
│       outputs of your model                         │
│     - Surprisingly effective!                       │
│                                                     │
│  3. HUMAN EVALUATION                                │
│     - Manual review by domain experts               │
│     - Gold standard but expensive and slow          │
│                                                     │
│  4. BENCHMARK DATASETS                              │
│     - MMLU (general knowledge)                      │
│     - HumanEval (coding)                            │
│     - TruthfulQA (factual accuracy)                 │
│     - MT-Bench (conversational ability)             │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### Example: Automated Evaluation with LLM-as-a-Judge

```python
# Using an LLM to evaluate another LLM's output
# This is called "LLM-as-a-Judge"

import openai

def evaluate_response(question, response, reference_answer):
    """
    Use GPT-4 to score the quality of an LLM response.
    Returns a score from 1-5 and an explanation.
    """
    evaluation_prompt = f"""
    You are an expert evaluator. Score the following response on a scale of 1-5.
    
    Criteria:
    - Accuracy (does it match the reference answer?)
    - Completeness (does it cover all key points?)
    - Clarity (is it easy to understand?)
    - Relevance (does it actually answer the question?)
    
    Question: {question}
    Reference Answer: {reference_answer}
    Model Response: {response}
    
    Respond in this exact JSON format:
    {{
        "score": <1-5>,
        "accuracy": <1-5>,
        "completeness": <1-5>,
        "clarity": <1-5>,
        "explanation": "<brief explanation>"
    }}
    """
    
    client = openai.OpenAI()
    result = client.chat.completions.create(
        model="gpt-4",
        messages=[{"role": "user", "content": evaluation_prompt}],
        temperature=0,        # Low temperature = more consistent evaluation
        response_format={"type": "json_object"}
    )
    
    return result.choices[0].message.content

# Run evaluation on a batch of test cases
test_cases = [
    {
        "question": "What is Kubernetes?",
        "reference": "Kubernetes is an open-source container orchestration platform...",
        "model_response": "K8s is a tool for managing containers at scale..."
    },
    # ... more test cases
]

for test in test_cases:
    score = evaluate_response(
        test["question"], 
        test["model_response"], 
        test["reference"]
    )
    print(f"Question: {test['question']}")
    print(f"Score: {score}")
    print("---")
```

**What this code does:**
- We use GPT-4 as a "judge" to evaluate the quality of responses from our LLM.
- The judge scores on 4 criteria: accuracy, completeness, clarity, relevance.
- `temperature=0` makes the evaluation deterministic (same input = same output).
- `response_format={"type": "json_object"}` ensures we get structured output we can parse.
- We loop through test cases and get automated quality scores — much faster than human review!

## 5️⃣ Deploying LLMs — Serving Infrastructure

This is where DevOps expertise really shines. Deploying LLMs is **very different** from deploying a web app.

### LLM Serving Frameworks

| Framework | Best For | Key Features |
|-----------|----------|-------------|
| **vLLM** | High-throughput serving | PagedAttention, continuous batching, fastest inference |
| **TGI (Text Generation Inference)** | Hugging Face ecosystem | Easy setup, good defaults, production-ready |
| **Ollama** | Local development | Dead simple, `ollama run llama3` and you're done |
| **TensorRT-LLM** | Maximum NVIDIA GPU performance | NVIDIA-optimized, complex setup |
| **Triton Inference Server** | Multi-model serving | Supports any framework, GPU scheduling |
| **Ray Serve** | Scalable distributed serving | Auto-scaling, multi-model, Python-native |
| **KServe** | Kubernetes-native serving | Integrates with K8s, auto-scaling, canary |

### Example: Deploying LLaMA 3 with vLLM

**vLLM** is currently the gold standard for serving LLMs. It's fast, memory-efficient, and production-ready.

```bash
# Install vLLM
pip install vllm

# Start a vLLM server — it's that simple!
# This serves LLaMA 3 8B on port 8000 with an OpenAI-compatible API
python -m vllm.entrypoints.openai.api_server \
    --model meta-llama/Llama-3-8b-instruct \
    --host 0.0.0.0 \
    --port 8000 \
    --tensor-parallel-size 2 \    # Split model across 2 GPUs
    --max-model-len 8192 \        # Max context length
    --gpu-memory-utilization 0.9  # Use 90% of GPU memory
```

**What each flag does:**
- `--model`: Which model to load from Hugging Face
- `--tensor-parallel-size 2`: Split the model across 2 GPUs (for models too large for one GPU)
- `--max-model-len 8192`: Maximum number of tokens in a single request
- `--gpu-memory-utilization 0.9`: Use 90% of GPU memory for the model (leave 10% for overhead)

**Now you can call it like an OpenAI API:**

```python
import openai

# Point to your local vLLM server instead of OpenAI
client = openai.OpenAI(
    base_url="http://localhost:8000/v1",
    api_key="not-needed"    # Local server doesn't need a key
)

response = client.chat.completions.create(
    model="meta-llama/Llama-3-8b-instruct",
    messages=[
        {"role": "system", "content": "You are a helpful DevOps assistant."},
        {"role": "user", "content": "Explain Kubernetes in 3 sentences."}
    ],
    temperature=0.7,
    max_tokens=200
)

print(response.choices[0].message.content)
```

### Deploying vLLM on Kubernetes

```yaml
# vllm-deployment.yaml
# Deploy vLLM as a Kubernetes service with GPU support

apiVersion: apps/v1
kind: Deployment
metadata:
  name: vllm-llama3
  namespace: ml-serving
  labels:
    app: vllm-llama3
spec:
  replicas: 2                    # 2 replicas for high availability
  selector:
    matchLabels:
      app: vllm-llama3
  template:
    metadata:
      labels:
        app: vllm-llama3
    spec:
      containers:
      - name: vllm
        image: vllm/vllm-openai:latest
        ports:
        - containerPort: 8000
        env:
        - name: MODEL_NAME
          value: "meta-llama/Llama-3-8b-instruct"
        - name: HUGGING_FACE_HUB_TOKEN    # Needed for gated models
          valueFrom:
            secretKeyRef:
              name: hf-token
              key: token
        args:
        - "--model"
        - "$(MODEL_NAME)"
        - "--tensor-parallel-size"
        - "1"
        - "--max-model-len"
        - "4096"
        - "--gpu-memory-utilization"
        - "0.9"
        resources:
          limits:
            nvidia.com/gpu: 1     # Request 1 NVIDIA GPU
            memory: "32Gi"
            cpu: "8"
          requests:
            nvidia.com/gpu: 1
            memory: "24Gi"
            cpu: "4"
        readinessProbe:            # Don't send traffic until model is loaded
          httpGet:
            path: /health
            port: 8000
          initialDelaySeconds: 120  # Model loading takes time!
          periodSeconds: 10
        livenessProbe:
          httpGet:
            path: /health
            port: 8000
          initialDelaySeconds: 120
          periodSeconds: 30
        volumeMounts:
        - name: model-cache
          mountPath: /root/.cache/huggingface
      volumes:
      - name: model-cache
        persistentVolumeClaim:
          claimName: model-cache-pvc   # Cache models to avoid re-downloading
      nodeSelector:
        gpu-type: a100                  # Schedule on nodes with A100 GPUs
      tolerations:
      - key: nvidia.com/gpu
        operator: Exists
        effect: NoSchedule

---
apiVersion: v1
kind: Service
metadata:
  name: vllm-llama3-service
  namespace: ml-serving
spec:
  selector:
    app: vllm-llama3

  name: vllm-hpa

        name: gpu_utilization         # Scale based on GPU usage

        type: AverageValue

```

**What this Kubernetes manifest does:**
- **Deployment:** Runs 2 replicas of vLLM, each with 1 NVIDIA GPU. The model is cached on a PVC so it doesn't re-download on every restart.
- **readinessProbe:** Waits 120 seconds before considering the pod ready (LLM loading is slow!).
- **nodeSelector:** Only schedule on nodes with A100 GPUs.
- **Service:** Exposes vLLM internally in the cluster on port 8000.
- **HPA:** Auto-scales from 2 to 8 replicas based on GPU utilization.

```python
# Example: Custom Prometheus metrics for LLM serving

from prometheus_client import Histogram, Counter, Gauge, start_http_server
import time

# Define metrics
REQUEST_LATENCY = Histogram(
    'llm_request_duration_seconds',
    'Time spent processing LLM request',
    ['model', 'endpoint'],
    buckets=[0.1, 0.5, 1.0, 2.0, 5.0, 10.0, 30.0, 60.0]
)

TOKENS_GENERATED = Counter(
    'llm_tokens_generated_total',
    'Total number of tokens generated',
    ['model']
)

TOKEN_THROUGHPUT = Gauge(
    'llm_tokens_per_second',
    'Current token generation throughput',
    ['model']
)

REQUEST_COST = Counter(
    'llm_request_cost_dollars',
    'Estimated cost of LLM requests',
    ['model']
)

GPU_MEMORY_USAGE = Gauge(
    'llm_gpu_memory_usage_bytes',
    'GPU memory usage in bytes',
    ['gpu_id']
)

# Instrument your LLM calls
def call_llm(prompt, model="llama3"):
    start_time = time.time()
    
    # Make the actual LLM call
    response = llm.generate(prompt)
    
    # Record metrics
    duration = time.time() - start_time
    tokens = response.usage.total_tokens
    
    REQUEST_LATENCY.labels(model=model, endpoint="/generate").observe(duration)
    TOKENS_GENERATED.labels(model=model).inc(tokens)
    TOKEN_THROUGHPUT.labels(model=model).set(tokens / duration)
    
    # Estimate cost (example: $0.002 per 1K tokens)
    cost = (tokens / 1000) * 0.002
    REQUEST_COST.labels(model=model).inc(cost)
    
    return response

# Start Prometheus metrics server on port 9090
start_http_server(9090)
```

**What this code does:**
- Defines custom Prometheus metrics specific to LLM serving: latency, token counts, throughput, cost, GPU memory.
- Every time you call the LLM, it records how long it took, how many tokens were generated, and estimated cost.
- These metrics can be scraped by Prometheus and visualized in Grafana dashboards.

## 8️⃣ LLM Cost Optimization

LLM inference is **expensive**. Here are proven strategies to reduce costs:

### Cost Reduction Strategies

```
LLM Cost Optimization Strategies:

┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  1. MODEL SELECTION                                         │
│     Use the smallest model that works for your task         │
│     GPT-4 ($30/M tokens) vs GPT-3.5 ($0.50/M tokens)      │
│     = 60x cost difference!                                  │
│                                                             │
│  2. CACHING                                                 │
│     Cache identical or similar prompts                      │
│     If 30% of queries repeat → 30% cost savings             │
│                                                             │
│  3. PROMPT OPTIMIZATION                                     │
│     Shorter prompts = fewer input tokens = lower cost       │
│     Remove unnecessary instructions                         │
│                                                             │
│  4. QUANTIZATION                                            │
│     Run models in 4-bit or 8-bit precision                  │
│     50-75% less GPU memory, minimal quality loss            │
│                                                             │
│  5. BATCHING                                                │
│     Process multiple requests together                      │
│     Better GPU utilization = lower cost per request         │
│                                                             │
│  6. MODEL ROUTING                                           │
│     Simple queries → small/cheap model                      │
│     Complex queries → large/expensive model                 │
│     Use a classifier to route automatically                 │
│                                                             │
│  7. SELF-HOSTING                                            │
│     At scale, self-hosting open-source models is cheaper    │
│     Break-even: usually ~$10K-20K/month in API costs        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Example: Semantic Caching

```python
# Semantic caching — cache not just exact matches,
# but also semantically similar queries!

import hashlib
import numpy as np
from sentence_transformers import SentenceTransformer
import redis

class SemanticCache:
    def __init__(self, similarity_threshold=0.95):
        self.encoder = SentenceTransformer("all-MiniLM-L6-v2")
        self.redis = redis.Redis(host="localhost", port=6379)
        self.threshold = similarity_threshold
    
    def _get_embedding(self, text):
        """Convert text to a vector embedding."""
        return self.encoder.encode(text)
    
    def get(self, query):
        """Check if a similar query exists in cache."""
        query_embedding = self._get_embedding(query)
        
        # Check all cached embeddings for similarity
        for key in self.redis.scan_iter("cache:*"):
            cached = self.redis.hgetall(key)
            cached_embedding = np.frombuffer(cached[b"embedding"], dtype=np.float32)
            
            # Cosine similarity
            similarity = np.dot(query_embedding, cached_embedding) / (
                np.linalg.norm(query_embedding) * np.linalg.norm(cached_embedding)
            )
            
            if similarity >= self.threshold:
                print(f"Cache hit! Similarity: {similarity:.3f}")
                return cached[b"response"].decode()
        
        return None  # Cache miss
    
    def set(self, query, response, ttl=3600):
        """Store a query-response pair in cache."""
        embedding = self._get_embedding(query)
        cache_key = f"cache:{hashlib.md5(query.encode()).hexdigest()}"
        
        self.redis.hset(cache_key, mapping={
            "query": query,
            "response": response,
            "embedding": embedding.tobytes()
        })
        self.redis.expire(cache_key, ttl)

# Usage
cache = SemanticCache(similarity_threshold=0.92)

# First call — cache miss, call the LLM
query1 = "What is Kubernetes?"
result = cache.get(query1)
if result is None:
    result = call_llm(query1)    # Expensive LLM call
    cache.set(query1, result)

# Second call — different wording, but same meaning!
query2 = "Can you explain what K8s is?"
result = cache.get(query2)       # Cache hit! Same meaning, saves $$$
```

**What this code does:**
- **Traditional caching** only matches exact same queries. But "What is Kubernetes?" and "Can you explain K8s?" are different strings with the same meaning.
- **Semantic caching** converts queries to vectors (embeddings) and checks if a cached query is "close enough" in meaning (cosine similarity > 0.92).
- This means you avoid calling the expensive LLM for rephrased versions of the same question — huge cost savings!

## 📝 Key Takeaways

| Concept | Remember This |
|---------|---------------|
| **LLMOps ≠ MLOps** | LLMs need specialized infra, monitoring, and tooling |
| **API vs Self-Host** | APIs for quick start; self-host for privacy, cost control at scale |
| **Fine-Tuning vs RAG** | Fine-tune for behavior/style; RAG for knowledge/facts |
| **Prompt = Code** | Version, test, and manage prompts like source code |
| **Evaluation is hard** | Use LLM-as-a-judge + automated metrics + human review |
| **Guardrails are essential** | Never deploy without input/output safety checks |
| **Cost management is critical** | Caching, routing, quantization can cut costs 50-90% |
| **GPU infrastructure** | vLLM + Kubernetes + auto-scaling = production-ready serving |
