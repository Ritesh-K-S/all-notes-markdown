# 25.8 — GPU Workloads on Kubernetes — NVIDIA Device Plugin

> **One Line Answer:** Running AI/ML workloads on Kubernetes requires **GPU support**, and the NVIDIA Device Plugin is the bridge that lets Kubernetes **discover, schedule, and manage NVIDIA GPUs** on your cluster nodes — without it, Kubernetes has no idea GPUs even exist.

---

## 🤔 Why Do We Need GPUs for AI/ML?

Before we jump into Kubernetes, let's understand WHY GPUs are so important for AI workloads.

### CPU vs GPU — The Core Difference

```
CPU vs GPU — Think of it like this:

CPU (Central Processing Unit):
┌──────────────────────────────┐
│  A really smart professor     │
│  who can solve ANY problem    │
│  but works on ONE thing       │
│  at a time.                   │
│                               │
│  Cores: 8-64                  │
│  Good at: Complex logic,      │
│  sequential tasks             │
└──────────────────────────────┘

GPU (Graphics Processing Unit):
┌──────────────────────────────┐
│  10,000 students who can      │
│  each solve a SIMPLE math     │
│  problem, but they all work   │
│  AT THE SAME TIME.            │
│                               │
│  Cores: 5,000-16,000+         │
│  Good at: Parallel math,      │
│  matrix multiplication        │
└──────────────────────────────┘

Why does this matter for AI?

Neural networks = MASSIVE matrix multiplications
Matrix multiplication = Perfectly parallelizable
GPUs = Massively parallel processors

Training a model on CPU:   30 days  😫
Training a model on GPU:   8 hours  🚀

That's a 90x speedup!
```

## Step 1: GPU Node Setup — Installing NVIDIA Drivers

First, your Kubernetes worker nodes that have GPUs need the NVIDIA driver installed.

### Option A: Install Manually on the Node

```bash
# Check if GPUs are detected by the system
lspci | grep -i nvidia
# Output: 00:1e.0 3D controller: NVIDIA Corporation A100 (rev a1)
# Great! The hardware is there.

# Install NVIDIA drivers on Ubuntu
sudo apt-get update
sudo apt-get install -y linux-headers-$(uname -r)

# Add NVIDIA repository
distribution=$(. /etc/os-release; echo $ID$VERSION_ID)
curl -fsSL https://nvidia.github.io/libnvidia-container/gpgkey | \
    sudo gpg --dearmor -o /usr/share/keyrings/nvidia-container-toolkit-keyring.gpg
curl -s -L https://nvidia.github.io/libnvidia-container/$distribution/libnvidia-container.list | \
    sed 's#deb https://#deb [signed-by=/usr/share/keyrings/nvidia-container-toolkit-keyring.gpg] https://#g' | \
    sudo tee /etc/apt/sources.list.d/nvidia-container-toolkit.list

# Install the driver
sudo apt-get update
sudo apt-get install -y nvidia-driver-550   # Use latest stable version
sudo reboot

# Verify installation
nvidia-smi
```

## Step 2: NVIDIA Container Toolkit — GPUs Inside Containers

Regular Docker containers can't see GPUs. The **NVIDIA Container Toolkit** (formerly nvidia-docker) bridges this gap.

```

With NVIDIA Container Toolkit:
┌─────────────────────┐
│     Container       │
│                     │
│  PyTorch code:      │
│  torch.cuda.is_     │
│  available()        │
```

### Manual Installation (if not using GPU Operator)

```bash
# Install NVIDIA Container Toolkit
sudo apt-get install -y nvidia-container-toolkit

# Configure Docker to use NVIDIA runtime
sudo nvidia-ctk runtime configure --runtime=docker
sudo systemctl restart docker

# Test it! Run nvidia-smi inside a container
docker run --rm --gpus all nvidia/cuda:12.4.0-base-ubuntu22.04 nvidia-smi
# You should see the same nvidia-smi output as on the host!
```

## Step 3: NVIDIA Device Plugin — The Star of the Show

The **NVIDIA Device Plugin** is a DaemonSet that runs on every GPU node in your Kubernetes cluster. Its job is simple but critical:

> **"Hey Kubernetes! This node has GPUs. Here they are. You can schedule workloads on them."**

Without the device plugin, Kubernetes simply doesn't know GPUs exist on the node. It's like having a car with no steering wheel — the engine is there, but you can't use it.

### How the Device Plugin Works

```
How NVIDIA Device Plugin Works:

  Node with 4x A100 GPUs
  ┌──────────────────────────────────────────────────┐
  │                                                  │
  │  NVIDIA Device Plugin Pod (DaemonSet)            │
  │  ┌──────────────────────────────────────────┐    │
  │  │                                          │    │
  │  │  1. Discovers GPUs on the node           │    │
  │  │     "I found 4 NVIDIA A100 GPUs!"        │    │
  │  │                                          │    │
  │  │  2. Registers them with kubelet          │    │
  │  │     Advertises: nvidia.com/gpu = 4       │    │
  │  │                                          │    │
  │  │  3. Handles GPU allocation               │    │
  │  │     Pod requests 1 GPU → assigns GPU #2  │    │
  │  │                                          │    │
  │  │  4. Health monitoring                    │    │
  │  │     GPU #3 has errors? Mark unhealthy    │    │
  │  │                                          │    │
  │  └──────────────────────────────────────────┘    │
  │                                                  │
  │  kubelet now knows:                              │
  │  ┌──────────────────────────────────┐            │
  │  │ Allocatable Resources:           │            │
  │  │   cpu:           64              │            │
  │  │   memory:        256Gi           │            │
  │  │   nvidia.com/gpu: 4   ← NEW!    │            │
  │  └──────────────────────────────────┘            │
  │                                                  │
  └──────────────────────────────────────────────────┘

  Now the Kubernetes scheduler can say:
  "This pod needs 2 GPUs → this node has 4 → schedule it here!"
```

### Installing the Device Plugin (Standalone)

If you're NOT using the GPU Operator, you can install the device plugin directly:

```bash
# Deploy NVIDIA Device Plugin as a DaemonSet
kubectl apply -f https://raw.githubusercontent.com/NVIDIA/k8s-device-plugin/v0.15.0/deployments/static/nvidia-device-plugin.yml
```

**Let's look at what this DaemonSet actually does:**

```yaml
# nvidia-device-plugin.yml (simplified for understanding)

apiVersion: apps/v1
kind: DaemonSet                    # Runs on EVERY matching node
metadata:
  name: nvidia-device-plugin-daemonset
  namespace: kube-system
spec:
  selector:
    matchLabels:
      name: nvidia-device-plugin-ds
  template:
    metadata:
      labels:
        name: nvidia-device-plugin-ds
    spec:
      tolerations:
      - key: nvidia.com/gpu         # Tolerate GPU-specific taints
        operator: Exists
        effect: NoSchedule
      priorityClassName: system-node-critical  # Don't evict this!
      containers:
      - name: nvidia-device-plugin
        image: nvcr.io/nvidia/k8s-device-plugin:v0.15.0
        securityContext:
          allowPrivilegeEscalation: false
          capabilities:
            drop: ["ALL"]
        volumeMounts:
        - name: device-plugin         # Socket for kubelet communication
          mountPath: /var/lib/kubelet/device-plugins
      volumes:
      - name: device-plugin
        hostPath:
          path: /var/lib/kubelet/device-plugins
      nodeSelector:                    # Only run on GPU nodes
        feature.node.kubernetes.io/pci-10de.present: "true"
        # 10de = NVIDIA's PCI vendor ID
```

**What each part does:**
- **DaemonSet:** Ensures this pod runs on every GPU node (and only GPU nodes).
- **tolerations:** GPU nodes often have a `nvidia.com/gpu` taint to prevent non-GPU pods from landing there. This toleration lets the device plugin run anyway.
- **priorityClassName: system-node-critical:** This is a critical system component — don't evict it even under resource pressure.
- **volumeMounts:** Mounts the kubelet device-plugins directory so the plugin can register itself with kubelet via a Unix socket.
- **nodeSelector:** Only run on nodes that have an NVIDIA PCI device (detected by Node Feature Discovery).

## Step 4: Requesting GPUs in Your Pods

Now that Kubernetes knows about GPUs, here's how you actually **use** them in your workloads:

### Basic GPU Pod

```yaml
# gpu-test-pod.yaml
# Simple pod that requests 1 GPU and runs nvidia-smi

apiVersion: v1
kind: Pod
metadata:
  name: gpu-test
spec:
  restartPolicy: Never
  containers:
  - name: cuda-test
    image: nvidia/cuda:12.4.0-base-ubuntu22.04
    command: ["nvidia-smi"]
    resources:
      limits:
        nvidia.com/gpu: 1    # Request exactly 1 GPU
        # That's it! Just add this one line!
```

```bash
# Run it
kubectl apply -f gpu-test-pod.yaml

# Check the output
kubectl logs gpu-test
# Shows nvidia-smi output proving the GPU is available inside the pod!
```

### Important Rules for GPU Resources in Kubernetes

```
GPU Scheduling Rules:

┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  1. GPUs are WHOLE numbers only                             │
│     ✅ nvidia.com/gpu: 1                                    │
│     ✅ nvidia.com/gpu: 4                                    │
│     ❌ nvidia.com/gpu: 0.5   (can't do half a GPU!)        │
│                                                             │
│  2. GPUs CANNOT be shared between pods (by default)         │
│     One GPU → One Pod. Period.                              │
│     (Unless you use MIG or GPU sharing — see below)         │
│                                                             │
│  3. You MUST set limits (not just requests)                 │
│     nvidia.com/gpu goes in resources.limits                 │
│     The request is automatically set equal to limit         │
│                                                             │
│  4. GPUs don't overcommit                                   │
│     If you have 4 GPUs and 4 pods each want 1,             │
│     a 5th pod will stay Pending until one finishes          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Real-World Example: PyTorch Training Job

```yaml
# pytorch-training-job.yaml
# A distributed PyTorch training job using multiple GPUs

apiVersion: batch/v1
kind: Job
metadata:
  name: pytorch-training
  namespace: ml-training
spec:
  backoffLimit: 3                   # Retry up to 3 times on failure
  template:
    metadata:
      labels:
        app: pytorch-training
    spec:
      restartPolicy: OnFailure
      containers:
      - name: trainer
        image: pytorch/pytorch:2.3.0-cuda12.1-cudnn8-runtime
        command:
        - python
        - -m
        - torch.distributed.launch
        - --nproc_per_node=4        # Use all 4 GPUs on this node
        - train.py
        - --epochs=50
        - --batch-size=64
        - --learning-rate=0.001
        env:
        - name: NCCL_DEBUG
          value: "INFO"              # NCCL = NVIDIA's multi-GPU communication library
        - name: CUDA_VISIBLE_DEVICES
          value: "all"
        resources:
          limits:
            nvidia.com/gpu: 4        # Request 4 GPUs
            memory: "128Gi"
            cpu: "32"
          requests:
            memory: "64Gi"
            cpu: "16"
        volumeMounts:
        - name: training-data
          mountPath: /data
        - name: model-output
          mountPath: /output
        - name: shared-memory        # Required for PyTorch DataLoader
          mountPath: /dev/shm
      volumes:
      - name: training-data
        persistentVolumeClaim:
          claimName: training-data-pvc
      - name: model-output
        persistentVolumeClaim:
          claimName: model-output-pvc
      - name: shared-memory          # PyTorch needs this for multi-worker data loading
        emptyDir:
          medium: Memory
          sizeLimit: "16Gi"
      nodeSelector:
        gpu-type: a100               # Schedule on A100 nodes
      tolerations:
      - key: nvidia.com/gpu
        operator: Exists
        effect: NoSchedule
```

**What each important section does:**
- **`nvidia.com/gpu: 4`**: Requests 4 GPUs for this training job.
- **`torch.distributed.launch --nproc_per_node=4`**: PyTorch's distributed training launcher. Starts 4 processes, one per GPU.
- **`NCCL_DEBUG: INFO`**: NCCL (NVIDIA Collective Communications Library) handles GPU-to-GPU communication. INFO level helps debug multi-GPU issues.
- **`/dev/shm` volume**: PyTorch DataLoader uses shared memory for fast data loading with multiple workers. Without this, you'll get "bus error" crashes.
- **`nodeSelector: gpu-type: a100`**: Only schedule on nodes labeled with A100 GPUs.

## Step 5: GPU Monitoring — DCGM Exporter + Prometheus + Grafana

### Install DCGM Exporter

```bash
# If using GPU Operator, DCGM Exporter is already installed!
```

### Prometheus ServiceMonitor for GPU Metrics

```yaml
# gpu-service-monitor.yaml
# Tell Prometheus to scrape GPU metrics

apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: dcgm-exporter
  namespace: gpu-monitoring
  labels:
    release: prometheus        # Must match your Prometheus operator selector
spec:
  selector:
    matchLabels:
      app: dcgm-exporter
  endpoints:
  - port: metrics
    interval: 15s              # Scrape every 15 seconds
    path: /metrics
```

### Grafana Dashboard — Sample PromQL Queries

```
Useful PromQL Queries for GPU Monitoring:

# Average GPU utilization across all GPUs
avg(DCGM_FI_DEV_GPU_UTIL)

# GPU memory usage percentage per GPU
DCGM_FI_DEV_FB_USED / (DCGM_FI_DEV_FB_USED + DCGM_FI_DEV_FB_FREE) * 100

# GPU temperature per node
DCGM_FI_DEV_GPU_TEMP{gpu=~".*"}

# GPUs with utilization below 20% (underutilized — wasting money!)
DCGM_FI_DEV_GPU_UTIL < 20

# Total power consumption of all GPUs
sum(DCGM_FI_DEV_POWER_USAGE)

# GPU errors (XID errors — hardware problems!)
increase(DCGM_FI_DEV_XID_ERRORS[1h]) > 0

# Alert: GPU temperature too high
# (should trigger alert if sustained above 85°C)
DCGM_FI_DEV_GPU_TEMP > 85
```

### Alerting Rules for GPUs

```yaml
# gpu-alerts.yaml
# PrometheusRule for GPU alerting

apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: gpu-alerts
  namespace: gpu-monitoring
spec:
  groups:
  - name: gpu.rules
    rules:
    
    # Alert if GPU temperature is too high
    - alert: GPUTemperatureCritical
      expr: DCGM_FI_DEV_GPU_TEMP > 90
      for: 5m
      labels:
        severity: critical
      annotations:
        summary: "GPU temperature critical on {{ $labels.node }}"
        description: "GPU {{ $labels.gpu }} on node {{ $labels.node }} is at {{ $value }}°C"
    
    # Alert if GPU has hardware errors
    - alert: GPUXIDErrors
      expr: increase(DCGM_FI_DEV_XID_ERRORS[1h]) > 0
      labels:
        severity: warning
      annotations:
        summary: "GPU XID errors detected on {{ $labels.node }}"
        description: "GPU {{ $labels.gpu }} has {{ $value }} XID errors in the last hour"
    
    # Alert if GPU memory is almost full (OOM risk)
    - alert: GPUMemoryAlmostFull
      expr: (DCGM_FI_DEV_FB_USED / (DCGM_FI_DEV_FB_USED + DCGM_FI_DEV_FB_FREE)) * 100 > 95
      for: 5m
      labels:
        severity: warning
      annotations:
        summary: "GPU memory almost full on {{ $labels.node }}"
        description: "GPU {{ $labels.gpu }} memory is {{ $value }}% full"
    
    # Alert if GPUs are idle (wasting money!)
    - alert: GPUUnderutilized
      expr: DCGM_FI_DEV_GPU_UTIL < 10
      for: 30m
      labels:
        severity: info
      annotations:
        summary: "GPU underutilized on {{ $labels.node }}"
        description: "GPU {{ $labels.gpu }} has been below 10% utilization for 30 minutes. Consider scaling down."
```

## Advanced: Multi-Instance GPU (MIG)

**Problem:** A single A100 GPU has 80GB of memory. But what if your workload only needs 10GB? You're wasting 70GB!

**Solution:** MIG lets you **split one physical GPU into multiple smaller virtual GPUs**.

```
MIG — Multi-Instance GPU:

Without MIG:
┌──────────────────────────────────────┐
│           A100 (80 GB)               │
│                                      │
│   Pod A needs 10GB                   │
│   Gets entire 80GB GPU              │
│   70GB WASTED! 💸                    │
│                                      │
└──────────────────────────────────────┘

With MIG:
┌──────────────────────────────────────┐
│           A100 (80 GB)               │
│  ┌────────┬────────┬────────┐        │
│  │ MIG 1  │ MIG 2  │ MIG 3  │       │
│  │ 20GB   │ 20GB   │ 40GB   │       │
│  │        │        │        │       │
│  │ Pod A  │ Pod B  │ Pod C  │       │
│  │ 10GB   │ 15GB   │ 35GB   │       │
│  └────────┴────────┴────────┘        │
│  No waste! Every GB is used! 🎯      │
└──────────────────────────────────────┘
```

### MIG Profiles for A100

| Profile | GPU Memory | GPU Compute | Best For |
|---------|-----------|-------------|----------|
| **1g.10gb** | 10 GB | 1/7 compute | Small inference |
| **2g.20gb** | 20 GB | 2/7 compute | Medium inference |
| **3g.40gb** | 40 GB | 3/7 compute | Large inference |
| **4g.40gb** | 40 GB | 4/7 compute | Training |
| **7g.80gb** | 80 GB | Full compute | Large training |

### Enable MIG

```bash
# Enable MIG mode on GPU 0
sudo nvidia-smi -i 0 -mig 1

# Create MIG instances
# Split into 3 instances: 2x 20GB + 1x 40GB
sudo nvidia-smi mig -i 0 -cgi 9,9,14 -C
#   9  = 2g.20gb profile
#   14 = 3g.40gb profile

# Verify MIG instances
nvidia-smi -L
# GPU 0: NVIDIA A100-SXM4-80GB (UUID: GPU-xxxxx)
#   MIG 2g.20gb  Device  0: (UUID: MIG-xxxxx)
#   MIG 2g.20gb  Device  1: (UUID: MIG-xxxxx)
#   MIG 3g.40gb  Device  2: (UUID: MIG-xxxxx)
```

### Using MIG in Kubernetes

```yaml
# When MIG is enabled, you request specific MIG profiles instead of full GPUs

apiVersion: v1
kind: Pod
metadata:
  name: mig-inference-pod
spec:
  containers:
  - name: inference
    image: my-ml-app:latest
    resources:
      limits:
        # Instead of nvidia.com/gpu: 1 (full GPU)
        # Request a specific MIG profile:
        nvidia.com/mig-2g.20gb: 1    # Request one 20GB MIG instance
```

## Advanced: GPU Sharing — Time-Slicing

**MIG is only available on A100/H100.** For older GPUs (T4, V100, A10G), you can use **time-slicing** to share a GPU between multiple pods.

```
Time-Slicing:
┌──────────────────────────────────────┐
│           T4 GPU (16 GB)             │
│                                      │
│  Time 0-100ms:  Pod A runs          │
│  Time 100-200ms: Pod B runs         │
│  Time 200-300ms: Pod C runs         │
│  Time 300-400ms: Pod A runs again   │
│  ... and so on                      │
│                                      │
│  Each pod THINKS it has its own GPU │
│  but they're taking turns!          │
│  ⚠️ No memory isolation though!     │
└──────────────────────────────────────┘
```

### Configure Time-Slicing

```yaml
# time-slicing-config.yaml
# ConfigMap for NVIDIA Device Plugin time-slicing

apiVersion: v1
kind: ConfigMap
metadata:
  name: nvidia-device-plugin-config
  namespace: kube-system
data:
  config.yaml: |
    version: v1
    sharing:
      timeSlicing:
        renameByDefault: false
        failRequestsGreaterThanOne: false
        resources:
        - name: nvidia.com/gpu
          replicas: 4           # Each physical GPU appears as 4 virtual GPUs
          # So a node with 2 GPUs now shows 8 nvidia.com/gpu!
```

```bash
# Apply the config and restart the device plugin
kubectl apply -f time-slicing-config.yaml
kubectl -n kube-system rollout restart daemonset nvidia-device-plugin-daemonset

# Verify — node should now show 4x more GPUs
kubectl describe node gpu-node | grep nvidia.com/gpu
# Allocatable:  nvidia.com/gpu: 8  (was 2, now 2×4=8)
```

**Important caveats of time-slicing:**
- ⚠️ **No memory isolation** — one pod can OOM and crash another pod's GPU work
- ⚠️ **No compute guarantee** — all pods share compute equally
- ✅ **Great for development** — multiple developers can share GPUs
- ✅ **Great for light inference** — multiple small models on one GPU

## Node Labeling & Scheduling — Putting It All Together

In a real cluster, you'll have different types of GPU nodes. You need to label them properly so the right workloads land on the right GPUs.

### GPU Node Labels (from GPU Feature Discovery)

```bash
# GPU Feature Discovery automatically labels your nodes!
kubectl describe node gpu-node-1 | grep nvidia

# Labels you'll see:
#   nvidia.com/gpu.product=NVIDIA-A100-SXM4-80GB
#   nvidia.com/gpu.memory=81920                    # MB
#   nvidia.com/gpu.count=4
#   nvidia.com/gpu.family=ampere
#   nvidia.com/cuda.driver.major=550
#   nvidia.com/cuda.runtime.major=12
#   nvidia.com/mig.capable=true
#   nvidia.com/gpu.compute.major=8
```

### Scheduling Strategy

```yaml
# Example: Schedule training on A100, inference on T4

# Training job — needs big GPUs
apiVersion: batch/v1
kind: Job
metadata:
  name: model-training
spec:
  template:
    spec:
      nodeSelector:
        nvidia.com/gpu.product: NVIDIA-A100-SXM4-80GB
      containers:
      - name: trainer
        resources:
          limits:
            nvidia.com/gpu: 4

---
# Inference — smaller GPU is fine
apiVersion: apps/v1
kind: Deployment
metadata:
  name: inference-server
spec:
  template:
    spec:
      nodeSelector:
        nvidia.com/gpu.product: Tesla-T4
      containers:
      - name: inference
        resources:
          limits:
            nvidia.com/gpu: 1
```

### Taints and Tolerations for GPU Nodes

```bash
# Taint GPU nodes so only GPU workloads run there
# This prevents regular pods from wasting expensive GPU node resources
kubectl taint nodes gpu-node-1 nvidia.com/gpu=present:NoSchedule

# Now only pods with the matching toleration can be scheduled here
```

```yaml
# Pod that can run on tainted GPU nodes
spec:
  tolerations:
  - key: nvidia.com/gpu
    operator: Exists
    effect: NoSchedule
  containers:
  - name: ml-workload
    resources:
      limits:
        nvidia.com/gpu: 1
```

### Debugging Commands

```bash
# 1. Check if device plugin is running
kubectl -n kube-system get pods | grep nvidia
kubectl -n kube-system logs <nvidia-device-plugin-pod>

# 2. Check GPU resources on nodes
kubectl get nodes -o json | jq '.items[] | {
  name: .metadata.name, 
  gpus: .status.allocatable["nvidia.com/gpu"]
}'

# 3. Check which pods are using GPUs
kubectl get pods --all-namespaces -o json | jq '.items[] | 
  select(.spec.containers[].resources.limits["nvidia.com/gpu"] != null) | {
    namespace: .metadata.namespace,
    name: .metadata.name,
    gpus: .spec.containers[].resources.limits["nvidia.com/gpu"]
  }'

# 4. Run nvidia-smi inside a pod
kubectl exec -it <pod-name> -- nvidia-smi

# 5. Check GPU operator health
kubectl -n gpu-operator get pods
kubectl -n gpu-operator logs <gpu-operator-pod>

# 6. Check CUDA availability in a pod
kubectl exec -it <pod-name> -- python -c "import torch; print(torch.cuda.is_available()); print(torch.cuda.device_count())"
```

## 📝 Key Takeaways

| Concept | Remember This |
|---------|---------------|
| **GPUs are essential for AI** | 90x faster than CPUs for matrix math |
| **NVIDIA Device Plugin** | Makes GPUs visible to Kubernetes as `nvidia.com/gpu` |
| **GPU Operator** | Automates entire GPU stack installation (recommended!) |
| **Container Toolkit** | Makes GPUs accessible inside containers |
| **GPUs are whole numbers** | Can't request 0.5 GPU (use MIG or time-slicing instead) |
| **MIG** | Split one A100 into multiple virtual GPUs (memory isolation) |
| **Time-slicing** | Share GPU between pods via time-sharing (no isolation) |
| **DCGM Exporter** | GPU metrics for Prometheus/Grafana monitoring |
| **Node labels** | Use `nodeSelector` to schedule on right GPU type |
| **Taints** | Keep non-GPU pods off expensive GPU nodes |
