# 25.9 — AI for DevOps — AIOps, Predictive Monitoring, Auto-Remediation

> **One Line Answer:** AIOps is the practice of using **Artificial Intelligence and Machine Learning** to enhance DevOps operations — from **detecting anomalies before they become incidents**, to **predicting failures before they happen**, to **automatically fixing problems without human intervention**.

---

## 🤔 What is AIOps?

AIOps stands for **Artificial Intelligence for IT Operations**. It was coined by Gartner in 2016, and the idea is simple:

> **"We have too much data, too many alerts, and too few humans. Let's use AI to help."**

Think about a modern DevOps environment:

```
The Data Overload Problem:

A typical production environment generates:

  📊 Metrics:     ~1 million data points per minute
  📝 Logs:        ~10 GB per hour
  🔗 Traces:      ~500,000 spans per minute
  🚨 Alerts:      ~500-2000 per day
  📈 Events:      ~10,000 per hour (deploys, config changes, scaling)

One human can meaningfully process:
  ~50 alerts per day
  ~100 log entries per investigation
  ~5-10 dashboards at a time

That's a MASSIVE gap! 

AIOps bridges this gap by using ML to:
  ✅ Find the needle in the haystack (anomaly detection)
  ✅ Predict problems before they happen (predictive analytics)
  ✅ Reduce noise (alert correlation & deduplication)
  ✅ Fix known issues automatically (auto-remediation)
  ✅ Find root causes faster (causal analysis)
```

## 1️⃣ Anomaly Detection — "Something Looks Wrong"

Anomaly detection is the **bread and butter** of AIOps. Instead of setting static thresholds (like "alert if CPU > 80%"), you use ML to learn **what's normal** and flag anything **unusual**.

### Why Static Thresholds Suck

```
The Problem with Static Thresholds:

CPU Alert Threshold: > 80%

  Monday    : Normal CPU is 30%   → 80% threshold = reasonable
  Friday    : Normal CPU is 75%   → 80% threshold = too sensitive!
  Black Friday: Normal CPU is 95%  → 80% threshold = CONSTANT ALERTS! 🚨🚨🚨

With ML-based anomaly detection:
  The model LEARNS the pattern:
    Monday normal: 25-35%
    Friday normal: 70-80%
    Black Friday normal: 90-97%
  
  It only alerts when something is TRULY abnormal for THAT time/day.
  
  Example: 50% CPU on Monday at 3am? → ANOMALY! (normally 10% at 3am)
           95% CPU on Black Friday?    → NORMAL (expected traffic spike)
```

### Types of Anomaly Detection

```
Types of Anomalies:

1. POINT ANOMALY
   A single data point that's way off.
   Example: Response time suddenly spikes to 30 seconds.
   
   Normal: ──────────────────────────────────
                              ↑
                          Spike! 🚨

2. CONTEXTUAL ANOMALY
   A data point that's normal in one context but abnormal in another.
   Example: High CPU at 3am (normally idle) vs high CPU at noon (normal).
   
   Noon:   ████████████ (80% CPU = normal)
   3am:    ████████████ (80% CPU = ANOMALY! Why is CPU high at 3am?)

3. COLLECTIVE ANOMALY  
   A series of data points that together form an unusual pattern.
   Example: Memory slowly increasing over days (memory leak).
   
   Day 1: ██░░░░░░░░ (20%)
   Day 2: ████░░░░░░ (40%)
   Day 3: ██████░░░░ (60%)
   Day 4: ████████░░ (80%)  ← Each point is "normal" but the TREND is anomalous!
```

### Building an Anomaly Detector — Practical Example

Let's build a real anomaly detector for server metrics using Python:

```python
# anomaly_detector.py
# A production-ready anomaly detector for time-series metrics
# Uses Isolation Forest (great for real-time anomaly detection)

import numpy as np
import pandas as pd
from sklearn.ensemble import IsolationForest
from sklearn.preprocessing import StandardScaler
from prometheus_api_client import PrometheusConnect
import warnings
warnings.filterwarnings('ignore')

class MetricAnomalyDetector:
    """
    Detects anomalies in time-series metrics from Prometheus.
    
    How it works:
    1. Pulls historical metric data from Prometheus
    2. Extracts features (rolling mean, std, time-of-day, day-of-week)
    3. Trains an Isolation Forest model to learn "normal" patterns
    4. Scores new data points as normal or anomalous
    """
    
    def __init__(self, prometheus_url="http://prometheus:9090"):
        self.prom = PrometheusConnect(url=prometheus_url)
        self.model = None
        self.scaler = StandardScaler()
    
    def fetch_metrics(self, query, hours=168):
        """
        Fetch metric data from Prometheus for the last N hours.
        Default: 168 hours = 1 week (to capture weekly patterns).
        """
        metric_data = self.prom.custom_query_range(
            query=query,
            start_time=pd.Timestamp.now() - pd.Timedelta(hours=hours),
            end_time=pd.Timestamp.now(),
            step="5m"   # One data point every 5 minutes
        )
        
        # Convert to DataFrame
        timestamps = [pd.Timestamp(point[0], unit='s') for point in metric_data[0]['values']]
        values = [float(point[1]) for point in metric_data[0]['values']]
        
        df = pd.DataFrame({'timestamp': timestamps, 'value': values})
        df.set_index('timestamp', inplace=True)
        return df
    
    def extract_features(self, df):
        """
        Create features that help the model understand patterns.
        
        Why these features?
        - Rolling mean/std: captures recent trend
        - Hour of day: CPU at 3am vs 3pm is different
        - Day of week: weekends vs weekdays are different
        - Rate of change: sudden spikes are suspicious
        """
        features = pd.DataFrame(index=df.index)
        
        # The raw metric value
        features['value'] = df['value']
        
        # Rolling statistics (30-minute window = 6 data points at 5m intervals)
        features['rolling_mean_30m'] = df['value'].rolling(6).mean()
        features['rolling_std_30m'] = df['value'].rolling(6).std()
        
        # Rolling statistics (2-hour window)
        features['rolling_mean_2h'] = df['value'].rolling(24).mean()
        features['rolling_std_2h'] = df['value'].rolling(24).std()
        
        # Time-based features (captures daily/weekly patterns)
        features['hour'] = df.index.hour
        features['day_of_week'] = df.index.dayofweek
        features['is_weekend'] = (df.index.dayofweek >= 5).astype(int)
        features['is_business_hours'] = ((df.index.hour >= 9) & (df.index.hour <= 17)).astype(int)
        
        # Rate of change (how fast the metric is changing)
        features['rate_of_change'] = df['value'].diff()
        features['rate_of_change_abs'] = df['value'].diff().abs()
        
        # Deviation from rolling mean (z-score-like)
        features['deviation'] = (df['value'] - features['rolling_mean_2h']) / (features['rolling_std_2h'] + 0.001)
        
        # Drop NaN rows (from rolling calculations)
        features.dropna(inplace=True)
        
        return features
    
    def train(self, query, hours=168):
        """
        Train the anomaly detection model on historical data.
        
        Isolation Forest works by:
        1. Randomly selecting a feature
        2. Randomly selecting a split value
        3. Points that are "easy to isolate" (few splits needed) are anomalies
        
        Think of it like a game of 20 questions:
        If you can identify a data point in just 2-3 questions, it's unusual!
        Normal points need many questions to isolate.
        """
        print(f"Fetching {hours} hours of data for: {query}")
        df = self.fetch_metrics(query, hours)
        features = self.extract_features(df)
        
        # Scale features (important for Isolation Forest)
        feature_columns = [col for col in features.columns]
        scaled_features = self.scaler.fit_transform(features[feature_columns])
        
        # Train Isolation Forest
        self.model = IsolationForest(
            n_estimators=200,          # Number of trees (more = more accurate)
            contamination=0.05,        # Expect ~5% of data to be anomalous
            max_samples='auto',
            random_state=42,
            n_jobs=-1                  # Use all CPU cores
        )
        self.model.fit(scaled_features)
        
        print(f"Model trained on {len(features)} data points")
        return self
    
    def detect(self, query, hours=1):
        """
        Check the last N hours for anomalies.
        Returns anomalous data points with scores.
        
        Score interpretation:
        - Negative scores = anomalous (more negative = more anomalous)
        - Positive scores = normal
        """
        df = self.fetch_metrics(query, hours)
        features = self.extract_features(df)
        
        feature_columns = [col for col in features.columns]
        scaled_features = self.scaler.transform(features[feature_columns])
        
        # Predict: 1 = normal, -1 = anomaly
        predictions = self.model.predict(scaled_features)
        # Score: more negative = more anomalous
        scores = self.model.decision_function(scaled_features)
        
        features['is_anomaly'] = predictions == -1
        features['anomaly_score'] = scores
        
        anomalies = features[features['is_anomaly']]
        
        if len(anomalies) > 0:
            print(f"🚨 Found {len(anomalies)} anomalies!")
            for idx, row in anomalies.iterrows():
                print(f"  Time: {idx}")
                print(f"  Value: {row['value']:.2f}")
                print(f"  Score: {row['anomaly_score']:.4f}")
                print(f"  Context: hour={row['hour']}, day={row['day_of_week']}")
                print()
        else:
            print("✅ No anomalies detected")
        
        return anomalies


# Usage Example:

# 1. Create detector
detector = MetricAnomalyDetector("http://prometheus:9090")

# 2. Train on 1 week of historical data
detector.train(
    query='rate(http_requests_total{job="api-server"}[5m])',
    hours=168    # 1 week
)

# 3. Check last hour for anomalies
anomalies = detector.detect(
    query='rate(http_requests_total{job="api-server"}[5m])',
    hours=1
)

# 4. If anomalies found, send alert
if len(anomalies) > 0:
    send_alert_to_slack(
        channel="#alerts",
        message=f"🚨 Anomaly detected in API request rate!\n"
                f"Time: {anomalies.index[0]}\n"
                f"Value: {anomalies.iloc[0]['value']:.2f}\n"
                f"Anomaly Score: {anomalies.iloc[0]['anomaly_score']:.4f}"
    )
```

**What this code does in simple terms:**
- **`fetch_metrics()`**: Pulls metric data from Prometheus (e.g., HTTP request rate).
- **`extract_features()`**: Creates "smart" features from the raw data — rolling averages, time of day, rate of change, etc. These help the model understand context (high CPU at 3am is different from high CPU at noon).
- **`train()`**: Uses Isolation Forest algorithm to learn what "normal" looks like over a week of data. The algorithm randomly isolates data points — anomalies are easy to isolate, normal points are hard.
- **`detect()`**: Checks recent data against the trained model. Points with negative anomaly scores are flagged as anomalies.

### Example: Disk Space Prediction

```python
# disk_space_predictor.py
# Predict when disk space will run out using linear regression

import numpy as np
from prometheus_api_client import PrometheusConnect
from datetime import datetime, timedelta

class DiskSpacePredictor:
    """
    Predicts when a disk will be full based on current usage trends.
    
    How it works:
    1. Fetch disk usage data from the last N days
    2. Fit a linear regression (find the trend line)
    3. Extrapolate to find when it hits 100%
    
    It's like drawing a straight line through your data points
    and seeing where it hits the ceiling.
    """
    
    def __init__(self, prometheus_url="http://prometheus:9090"):
        self.prom = PrometheusConnect(url=prometheus_url)
    
    def predict_disk_full(self, instance, mountpoint="/", lookback_days=7):
        """
        Predict when a disk will be 100% full.
        
        Args:
            instance: The server to check (e.g., "web-server-1:9100")
            mountpoint: Which disk to check (default: root /)
            lookback_days: How many days of history to analyze
        
        Returns:
            Dictionary with prediction details
        """
        
        # Query: disk used percentage over time
        query = f'''
            100 - (
                node_filesystem_avail_bytes{{instance="{instance}", mountpoint="{mountpoint}"}}
                /
                node_filesystem_size_bytes{{instance="{instance}", mountpoint="{mountpoint}"}}
                * 100
            )
        '''
        
        # Fetch historical data
        data = self.prom.custom_query_range(
            query=query,
            start_time=datetime.now() - timedelta(days=lookback_days),
            end_time=datetime.now(),
            step="1h"    # One data point per hour
        )
        
        if not data or not data[0]['values']:
            return {"error": "No data available"}
        
        # Extract timestamps and values
        values = data[0]['values']
        timestamps = np.array([float(v[0]) for v in values])
        usage_pct = np.array([float(v[1]) for v in values])
        
        # Normalize timestamps (start from 0 for easier math)
        t_start = timestamps[0]
        t_normalized = timestamps - t_start
        
        # Fit linear regression: usage = slope * time + intercept
        # slope = how fast disk usage is growing (% per second)
        # intercept = starting usage %
        slope, intercept = np.polyfit(t_normalized, usage_pct, 1)
        
        current_usage = usage_pct[-1]
        growth_rate_per_day = slope * 86400  # Convert from per-second to per-day
        
        if slope <= 0:
            return {
                "instance": instance,
                "mountpoint": mountpoint,
                "current_usage_pct": round(current_usage, 2),
                "growth_rate_per_day": round(growth_rate_per_day, 4),
                "prediction": "Disk usage is stable or decreasing. No risk.",
                "days_until_full": None,
                "risk_level": "low"
            }
        
        # When will it hit 100%?
        remaining_pct = 100 - current_usage
        seconds_until_full = remaining_pct / slope
        hours_until_full = seconds_until_full / 3600
        days_until_full = hours_until_full / 24
        
        # When will it hit 90%? (warning threshold)
        remaining_to_90 = max(0, 90 - current_usage)
        if remaining_to_90 > 0:
            hours_until_90 = (remaining_to_90 / slope) / 3600
        else:
            hours_until_90 = 0  # Already above 90%
        
        # Determine risk level
        if days_until_full < 1:
            risk_level = "critical"
        elif days_until_full < 3:
            risk_level = "high"
        elif days_until_full < 7:
            risk_level = "medium"
        else:
            risk_level = "low"
        
        predicted_full_date = datetime.now() + timedelta(hours=hours_until_full)
        
        result = {
            "instance": instance,
            "mountpoint": mountpoint,
            "current_usage_pct": round(current_usage, 2),
            "growth_rate_per_day": round(growth_rate_per_day, 2),
            "days_until_full": round(days_until_full, 1),
            "predicted_full_date": predicted_full_date.strftime("%Y-%m-%d %H:%M"),
            "hours_until_90_pct": round(hours_until_90, 1),
            "risk_level": risk_level
        }
        
        return result


# Usage:
predictor = DiskSpacePredictor("http://prometheus:9090")

# Check all instances
instances = ["web-1:9100", "web-2:9100", "db-1:9100", "db-2:9100"]

for instance in instances:
    prediction = predictor.predict_disk_full(instance, mountpoint="/data")
    
    print(f"\n{'='*50}")
    print(f"Server: {prediction['instance']}")
    print(f"Current Usage: {prediction['current_usage_pct']}%")
    print(f"Growth Rate: {prediction['growth_rate_per_day']}% per day")
    
    if prediction['days_until_full']:
        print(f"Days Until Full: {prediction['days_until_full']}")
        print(f"Predicted Full Date: {prediction['predicted_full_date']}")
        print(f"Risk Level: {prediction['risk_level'].upper()}")
        
        if prediction['risk_level'] in ['critical', 'high']:
            print("⚠️ ACTION REQUIRED!")
            # Trigger auto-remediation
            trigger_disk_cleanup(instance)
            # Or trigger volume expansion
            expand_volume(instance, mountpoint="/data")
```

**What this code does in simple terms:**
- It pulls disk usage data from Prometheus for the last 7 days.
- It fits a straight line through the data (linear regression) to find the growth trend.
- It calculates: "If usage keeps growing at this rate, when will it hit 100%?"
- It categorizes the risk: critical (< 1 day), high (< 3 days), medium (< 7 days), low.
- If the risk is high/critical, it triggers automatic cleanup or volume expansion.

## 3️⃣ Alert Correlation & Noise Reduction

One of the biggest pain points in DevOps is **alert fatigue**. When one thing breaks, it can trigger a cascade of alerts:

```
The Alert Storm Problem:

  Database goes down at 2:00 AM
       │
       ├── Alert: Database connection failed (x5 replicas)
       ├── Alert: API server error rate > 5%
       ├── Alert: API server latency > 2s
       ├── Alert: Frontend 500 errors increasing
       ├── Alert: User login failures increasing
       ├── Alert: Payment processing down
       ├── Alert: Background job queue growing
       ├── Alert: Health check failed (x10 services)
       ├── Alert: SLA breach risk
       └── Alert: Customer complaints spike

  Total: 20+ alerts for ONE root cause!
  
  On-call engineer at 2 AM:
  😱 "WHAT IS HAPPENING?! EVERYTHING IS ON FIRE!"
  
  With AIOps:
  🧠 "Database is down. 18 related alerts grouped. 
      Root cause: DB connection pool exhausted.
      Suggested fix: Restart connection pool."
```

### Alert Correlation Algorithm

```python
# alert_correlator.py
# Groups related alerts and identifies root causes

from datetime import datetime, timedelta
from collections import defaultdict
import re

class AlertCorrelator:
    """
    Correlates related alerts to reduce noise and identify root causes.
    
    How it works:
    1. Collect all alerts within a time window
    2. Group alerts that are:
       - Close in time (within 5 minutes)
       - From related services (based on dependency map)
       - Have similar labels (same namespace, cluster, etc.)
    3. Identify the most likely root cause
    4. Present ONE grouped alert instead of 20 individual ones
    """
    
    def __init__(self):
        # Define service dependency graph
        # "A depends on B" means if B is down, A will also be affected
        self.dependency_graph = {
            "frontend": ["api-gateway"],
            "api-gateway": ["user-service", "payment-service", "order-service"],
            "user-service": ["postgres-primary", "redis-cache"],
            "payment-service": ["postgres-primary", "stripe-api"],
            "order-service": ["postgres-primary", "rabbitmq"],
            "postgres-primary": [],
            "redis-cache": [],
            "rabbitmq": [],
        }
        
        self.active_alerts = []
    
    def add_alert(self, alert):
        """
        Add a new alert to the correlator.
        
        alert format: {
            "name": "HighErrorRate",
            "service": "api-gateway",
            "severity": "critical",
            "timestamp": datetime,
            "labels": {"namespace": "production", "cluster": "us-east-1"},
            "message": "Error rate above 5% for 5 minutes"
        }
        """
        self.active_alerts.append(alert)
    
    def find_root_cause(self, affected_services):
        """
        Walk the dependency graph to find which downstream service
        is causing all the problems.
        
        Logic: The service that is:
        1. Affected (has alerts)
        2. Deepest in the dependency chain
        3. Has the most services depending on it
        = Most likely root cause
        """
        
        # Find services that are affected AND are dependencies of other affected services
        root_causes = []
        
        for service in affected_services:
            # Check if this service is a dependency of other affected services
            dependent_count = 0
            for other_service, deps in self.dependency_graph.items():
                if service in deps and other_service in affected_services:
                    dependent_count += 1
            
            # A root cause has affected dependents but doesn't depend on other affected services
            own_deps_affected = [
                dep for dep in self.dependency_graph.get(service, [])
                if dep in affected_services
            ]
            
            if dependent_count > 0 and len(own_deps_affected) == 0:
                root_causes.append({
                    "service": service,
                    "affected_dependents": dependent_count
                })
        
        # Sort by most dependents affected
        root_causes.sort(key=lambda x: x["affected_dependents"], reverse=True)
        
        return root_causes
    
    def correlate(self, time_window_minutes=5):
        """
        Group related alerts and identify root causes.
        
        Returns a list of alert groups, each with:
        - Root cause
        - Related alerts
        - Suggested actions
        """
        
        if not self.active_alerts:
            return []
        
        # Sort alerts by time
        sorted_alerts = sorted(self.active_alerts, key=lambda a: a["timestamp"])
        
        # Group alerts within the time window
        groups = []
        current_group = [sorted_alerts[0]]
        
        for alert in sorted_alerts[1:]:
            time_diff = (alert["timestamp"] - current_group[-1]["timestamp"]).total_seconds()
            
            if time_diff <= time_window_minutes * 60:
                current_group.append(alert)
            else:
                groups.append(current_group)
                current_group = [alert]
        
        groups.append(current_group)
        
        # Analyze each group
        results = []
        for group in groups:
            affected_services = list(set(a["service"] for a in group))
            root_causes = self.find_root_cause(affected_services)
            
            max_severity = "info"
            severity_order = {"info": 0, "warning": 1, "critical": 2}
            for alert in group:
                if severity_order.get(alert["severity"], 0) > severity_order.get(max_severity, 0):
                    max_severity = alert["severity"]
            
            result = {
                "total_alerts": len(group),
                "affected_services": affected_services,
                "root_causes": root_causes,
                "severity": max_severity,
                "time_range": {
                    "start": group[0]["timestamp"],
                    "end": group[-1]["timestamp"]
                },
                "alerts": group,
                "summary": self._generate_summary(group, root_causes)
            }
            results.append(result)
        
        return results
    
    def _generate_summary(self, alerts, root_causes):
        """Generate a human-readable summary of the alert group."""
        
        if root_causes:
            rc = root_causes[0]["service"]
            return (
                f"🚨 {len(alerts)} related alerts detected. "
                f"Likely root cause: {rc}. "
                f"Affected services: {', '.join(set(a['service'] for a in alerts))}. "
                f"First alert: {alerts[0]['timestamp'].strftime('%H:%M:%S')}."
            )
        else:
            return (
                f"⚠️ {len(alerts)} alerts in the same time window. "
                f"Services affected: {', '.join(set(a['service'] for a in alerts))}."
            )
```

## 4️⃣ Auto-Remediation — "Fix It Without Waking Me Up"

Auto-remediation is the **holy grail** of AIOps. The system detects a problem AND fixes it automatically.

### Auto-Remediation Levels

```
Levels of Auto-Remediation:

Level 0: MANUAL
  Alert fires → Human reads it → Human investigates → Human fixes it
  Time to resolve: 30 minutes to hours
  ❌ Requires human intervention

Level 1: SUGGESTED REMEDIATION
  Alert fires → System suggests a fix → Human approves → System executes
  Time to resolve: 5-15 minutes
  ⚠️ Human in the loop (approval needed)

Level 2: AUTOMATED REMEDIATION (with guardrails)
  Alert fires → System executes known fix → Verifies it worked → Notifies human
  Time to resolve: 1-5 minutes
  ✅ Automatic but with safety checks

Level 3: PREDICTIVE AUTO-REMEDIATION
  System predicts problem → Fixes it BEFORE it impacts users → Logs action
  Time to resolve: 0 minutes (prevented!)
  🚀 The dream!
```

### Example: Kubernetes Auto-Remediation Controller

```python
# auto_remediator.py
# A Kubernetes-native auto-remediation system

import kubernetes
from kubernetes import client, config, watch
from prometheus_api_client import PrometheusConnect
import logging
import time

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger("auto-remediator")

class KubernetesAutoRemediator:
    """
    Watches for common Kubernetes issues and fixes them automatically.
    
    Remediation rules:
    1. CrashLoopBackOff → Check logs, restart with backoff, notify
    2. OOMKilled → Increase memory limits, restart
    3. High pod restarts → Rollback to previous deployment version
    4. Node not ready → Drain and cordon the node
    5. Disk pressure → Clean up old images and logs
    6. High CPU throttling → Increase CPU limits
    """
    
    def __init__(self):
        # Load kubeconfig (in-cluster or local)
        try:
            config.load_incluster_config()     # Running inside K8s
        except kubernetes.config.ConfigException:
            config.load_kube_config()           # Running locally
        
        self.v1 = client.CoreV1Api()
        self.apps_v1 = client.AppsV1Api()
        self.prom = PrometheusConnect(url="http://prometheus:9090")
        
        # Define remediation rules
        self.rules = {
            "CrashLoopBackOff": self.handle_crash_loop,
            "OOMKilled": self.handle_oom_killed,
            "HighRestarts": self.handle_high_restarts,
            "DiskPressure": self.handle_disk_pressure,
        }
        
        # Safety limits — prevent runaway automation
        self.max_actions_per_hour = 10
        self.actions_taken = []
    
    def _can_take_action(self):
        """Safety check: Don't take too many actions too quickly."""
        one_hour_ago = time.time() - 3600
        recent_actions = [a for a in self.actions_taken if a > one_hour_ago]
        self.actions_taken = recent_actions   # Clean up old entries
        
        if len(recent_actions) >= self.max_actions_per_hour:
            logger.warning(f"Rate limit reached! {len(recent_actions)} actions in last hour.")
            return False
        return True
    
    def _log_action(self, action_type, target, details):
        """Log every automated action for audit trail."""
        self.actions_taken.append(time.time())
        logger.info(f"AUTO-REMEDIATION: {action_type} | Target: {target} | Details: {details}")
        
        # Also create a Kubernetes Event so it shows up in `kubectl describe`
        event = client.CoreV1Event(
            metadata=client.V1ObjectMeta(
                generate_name="auto-remediation-",
                namespace=target.get("namespace", "default")
            ),
            reason="AutoRemediation",
            message=f"{action_type}: {details}",
            type="Normal",
            action=action_type,
            first_timestamp=client.V1Time(),
            involved_object=client.V1ObjectReference(
                kind="Pod",
                name=target.get("name", "unknown"),
                namespace=target.get("namespace", "default")
            )
        )
    
    def handle_crash_loop(self, pod_name, namespace):
        """
        Fix: CrashLoopBackOff
        
        Strategy:
        1. Check pod logs for known error patterns
        2. If known issue → apply specific fix
        3. If unknown → delete pod (let the controller recreate it)
        4. If it keeps crashing after 3 restarts → notify human
        """
        if not self._can_take_action():
            return
        
        logger.info(f"Handling CrashLoopBackOff for {namespace}/{pod_name}")
        
        # Get pod details
        pod = self.v1.read_namespaced_pod(pod_name, namespace)
        restart_count = pod.status.container_statuses[0].restart_count
        
        # Check logs for known patterns
        try:
            logs = self.v1.read_namespaced_pod_log(
                pod_name, namespace, tail_lines=50
            )
        except Exception:
            logs = ""
        
        # Pattern matching on logs
        if "connection refused" in logs.lower() or "cannot connect" in logs.lower():
            logger.info("Root cause: Connection issue. Checking dependencies...")
            # Don't restart — the issue is upstream
            self._log_action("SKIP_RESTART", 
                           {"name": pod_name, "namespace": namespace},
                           "Connection issue detected — upstream dependency may be down")
            
        elif "out of memory" in logs.lower() or "oom" in logs.lower():
            logger.info("Root cause: OOM. Delegating to OOM handler...")
            self.handle_oom_killed(pod_name, namespace)
            
        elif restart_count < 3:
            # Simple restart — delete pod and let controller recreate
            logger.info(f"Deleting pod {pod_name} (restart #{restart_count + 1})")
            self.v1.delete_namespaced_pod(pod_name, namespace)
            self._log_action("POD_RESTART",
                           {"name": pod_name, "namespace": namespace},
                           f"Deleted crashing pod (restart #{restart_count + 1})")
            
        else:
            # Too many restarts — escalate to human
            logger.warning(f"Pod {pod_name} has restarted {restart_count} times. Escalating!")
            self._log_action("ESCALATE",
                           {"name": pod_name, "namespace": namespace},
                           f"Pod has restarted {restart_count} times. Human intervention needed.")
            send_pagerduty_alert(
                service="auto-remediator",
                message=f"Pod {namespace}/{pod_name} in CrashLoopBackOff after {restart_count} restarts. "
                       f"Auto-remediation exhausted. Last logs:\n{logs[-500:]}"
            )
    
    def handle_oom_killed(self, pod_name, namespace):
        """
        Fix: OOMKilled (Out of Memory)
        
        Strategy:
        1. Find the deployment that owns this pod
        2. Increase memory limit by 50% (up to a maximum)
        3. The deployment controller will recreate the pod with more memory
        """
        if not self._can_take_action():
            return
        
        logger.info(f"Handling OOMKilled for {namespace}/{pod_name}")
        
        # Find the owner deployment
        pod = self.v1.read_namespaced_pod(pod_name, namespace)
        owner_ref = next(
            (ref for ref in (pod.metadata.owner_references or []) 
             if ref.kind == "ReplicaSet"),
            None
        )
        
        if not owner_ref:
            logger.warning("Cannot find owner deployment. Skipping.")
            return
        
        # Get the ReplicaSet to find the Deployment
        rs = self.apps_v1.read_namespaced_replica_set(owner_ref.name, namespace)
        deploy_ref = next(
            (ref for ref in (rs.metadata.owner_references or [])
             if ref.kind == "Deployment"),
            None
        )
        
        if not deploy_ref:
            return
        
        # Get current memory limit
        deployment = self.apps_v1.read_namespaced_deployment(deploy_ref.name, namespace)
        container = deployment.spec.template.spec.containers[0]
        current_limit = container.resources.limits.get("memory", "512Mi")
        
        # Parse and increase by 50%
        current_mb = self._parse_memory(current_limit)
        new_mb = int(current_mb * 1.5)
        max_mb = 8192   # Safety cap: never go above 8Gi
        
        if new_mb > max_mb:
            logger.warning(f"Memory limit would exceed {max_mb}Mi. Escalating to human.")
            return
        
        new_limit = f"{new_mb}Mi"
        
        # Patch the deployment
        container.resources.limits["memory"] = new_limit
        container.resources.requests["memory"] = f"{int(new_mb * 0.8)}Mi"
        
        self.apps_v1.patch_namespaced_deployment(
            deploy_ref.name, namespace, deployment
        )
        
        self._log_action("INCREASE_MEMORY",
                        {"name": deploy_ref.name, "namespace": namespace},
                        f"Memory limit increased from {current_limit} to {new_limit}")
        
        logger.info(f"✅ Increased memory for {deploy_ref.name}: {current_limit} → {new_limit}")
    
    def handle_high_restarts(self, deployment_name, namespace):
        """
        Fix: Too many restarts (bad deployment)
        
        Strategy:
        1. Check if a recent deployment happened
        2. If yes → rollback to the previous version
        3. Notify the team
        """
        if not self._can_take_action():
            return
        
        logger.info(f"Handling high restarts for deployment {namespace}/{deployment_name}")
        
        # Rollback to previous revision
        # This is equivalent to: kubectl rollout undo deployment/<name>
        deployment = self.apps_v1.read_namespaced_deployment(deployment_name, namespace)
        
        # Create a rollback by patching with the previous revision
        self.apps_v1.patch_namespaced_deployment(
            deployment_name,
            namespace,
            body={
                "spec": {
                    "template": {
                        "metadata": {
                            "annotations": {
                                "kubectl.kubernetes.io/restartedAt": 
                                    time.strftime("%Y-%m-%dT%H:%M:%SZ")
                            }
                        }
                    }
                }
            }
        )
        
        self._log_action("ROLLBACK",
                        {"name": deployment_name, "namespace": namespace},
                        "Rolled back deployment due to high restart count")
    
    def handle_disk_pressure(self, node_name, namespace=""):
        """
        Fix: Node has disk pressure
        
        Strategy:
        1. Clean up unused container images
        2. Clean up old logs
        3. If still not enough → cordon the node (prevent new pods)
        """
        if not self._can_take_action():
            return
        
        logger.info(f"Handling disk pressure on node {node_name}")
        
        # In practice, you'd SSH or use a DaemonSet to clean up
        # Here's the Kubernetes API approach:
        
        # Option 1: Cordon the node (prevent new pods from being scheduled)
        body = {
            "spec": {
                "unschedulable": True
            }
        }
        self.v1.patch_node(node_name, body)
        
        self._log_action("CORDON_NODE",
                        {"name": node_name, "namespace": ""},
                        f"Cordoned node {node_name} due to disk pressure")
        
        logger.info(f"⚠️ Node {node_name} cordoned. Needs manual disk cleanup or replacement.")
    
    def _parse_memory(self, memory_str):
        """Parse Kubernetes memory string to megabytes."""
        memory_str = str(memory_str)
        if memory_str.endswith("Gi"):
            return int(float(memory_str[:-2]) * 1024)
        elif memory_str.endswith("Mi"):
            return int(float(memory_str[:-2]))
        elif memory_str.endswith("Ki"):
            return int(float(memory_str[:-2]) / 1024)
        else:
            return int(float(memory_str) / (1024 * 1024))
    
    def watch_and_remediate(self):
        """
        Main loop: Watch for pod events and trigger remediation.
        Runs forever as a controller.
        """
        logger.info("🚀 Auto-Remediator started. Watching for issues...")
        
        w = watch.Watch()
        
        for event in w.stream(self.v1.list_pod_for_all_namespaces):
            pod = event['object']
            event_type = event['type']
            
            if event_type != 'MODIFIED':
                continue
            
            if not pod.status.container_statuses:
                continue
            
            for container_status in pod.status.container_statuses:
                # Check for CrashLoopBackOff
                if (container_status.state.waiting and 
                    container_status.state.waiting.reason == "CrashLoopBackOff"):
                    self.handle_crash_loop(
                        pod.metadata.name,
                        pod.metadata.namespace
                    )
                
                # Check for OOMKilled
                if (container_status.last_state.terminated and
                    container_status.last_state.terminated.reason == "OOMKilled"):
                    self.handle_oom_killed(
                        pod.metadata.name,
                        pod.metadata.namespace
                    )


# Run the auto-remediator
if __name__ == "__main__":
    remediator = KubernetesAutoRemediator()
    remediator.watch_and_remediate()
```

**What this code does:**
- It runs as a controller inside Kubernetes, constantly watching for pod events.
- When it detects a `CrashLoopBackOff`, it checks the logs, and either restarts the pod, handles the root cause, or escalates to a human after 3 failed attempts.
- When it detects `OOMKilled`, it automatically increases the memory limit by 50% (up to 8Gi max).
- It has **safety guardrails**: max 10 actions per hour, memory caps, escalation after retries.
- Every action is logged as a Kubernetes Event for audit trail.

## 8️⃣ Best Practices for AIOps Adoption

### Do's and Don'ts

```
AIOps Best Practices:

DO:
┌────────────────────────────────────────────────────────────────┐
│                                                                │
│  ✅ Start small — pick ONE use case (e.g., anomaly detection   │
│     on your top 5 critical metrics)                            │
│                                                                │
│  ✅ Start with DRY RUN mode — log what the system WOULD do     │
│     before letting it take action                              │
│                                                                │
│  ✅ Build a dependency map of your services first              │
│     (AIOps is only as good as your understanding of            │
│      how services relate to each other)                        │
│                                                                │
│  ✅ Keep humans in the loop initially                          │
│     (approve auto-remediation before it runs)                  │
│                                                                │
│  ✅ Measure the impact — track MTTR, alert noise, false        │
│     positive rate before and after AIOps                       │
│                                                                │
│  ✅ Have a kill switch — be able to disable auto-remediation   │
│     instantly if something goes wrong                          │
│                                                                │
│  ✅ Version your ML models and retrain regularly               │
│     (patterns change as your system evolves)                   │
│                                                                │
└────────────────────────────────────────────────────────────────┘

DON'T:
┌────────────────────────────────────────────────────────────────┐
│                                                                │
│  ❌ Don't try to automate everything at once                   │
│     (start with high-frequency, well-understood issues)        │
│                                                                │
│  ❌ Don't skip the monitoring fundamentals                     │
│     (AIOps enhances good monitoring, it doesn't replace it)    │
│                                                                │
│  ❌ Don't trust the AI blindly                                 │
│     (always have audit logs and rollback capability)           │
│                                                                │
│  ❌ Don't forget about edge cases                              │
│     (what if the auto-remediator has a bug?)                   │
│                                                                │
│  ❌ Don't ignore data quality                                  │
│     (garbage in = garbage out, even with AI)                   │
│                                                                │
│  ❌ Don't auto-remediate destructive actions                   │
│     (never auto-delete data, nodes, or PVCs)                   │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

## 📝 Key Takeaways

| Concept | Remember This |
|---------|---------------|
| **AIOps** | Using AI/ML to enhance IT operations — detect, predict, remediate |
| **Anomaly Detection** | ML learns "normal" patterns and flags deviations (better than static thresholds) |
| **Predictive Monitoring** | Forecast future problems before they happen (disk full, memory leak, traffic spike) |
| **Alert Correlation** | Group related alerts, reduce noise, identify root cause from cascade |
| **Auto-Remediation** | Automatically fix known issues (OOM → increase memory, crash → restart) |
| **Safety First** | Always start in dry-run mode, have rate limits, keep humans in the loop |
| **k8sGPT** | Uses LLMs to explain Kubernetes issues in plain English — game changer |
| **Start Small** | Pick one use case, prove value, then expand |
| **Data Quality** | AIOps is only as good as your monitoring data |
