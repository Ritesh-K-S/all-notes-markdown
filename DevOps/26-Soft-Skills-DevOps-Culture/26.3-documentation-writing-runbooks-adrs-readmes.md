# 26.3 — Documentation — Writing Runbooks, ADRs, READMEs

> **One Line Answer:** Great documentation is the **difference between a 5-minute fix and a 3-hour panic** at 2 AM. As a DevOps engineer, you'll write runbooks (how to fix things), ADRs (why we made decisions), and READMEs (how to use things) — and all of them will save your team countless hours.

---

## 🤔 Why Documentation Matters So Much in DevOps

Let's paint a picture:

```
🕐 2:47 AM — Your phone rings. PagerDuty alert.

"CRITICAL: Payment service is down in production"

You wake up, groggy, heart racing.

Scenario A (No documentation):
  → You SSH into the server
  → You have no idea what this service does or how it's deployed
  → You search Slack history for 30 minutes
  → You wake up 2 other people
  → 3 hours later, you figure out the fix
  → Customers are furious 😡

Scenario B (Good documentation):
  → You open the runbook: "Payment Service — Troubleshooting"
  → Step 1: Check pod status → kubectl get pods -n payments
  → Step 2: If CrashLoopBackOff → Check DB connection
  → Step 3: Restart command: kubectl rollout restart deployment/payment-svc
  → 15 minutes later, service is back up ✅
  → You go back to sleep 😴
```

**That's the power of documentation.** It turns tribal knowledge into shared knowledge. It turns chaos into calm.

## 📚 The 3 Types of Documentation Every DevOps Team Needs

| Type | What It Is | When You Use It | Who Reads It |
|------|-----------|-----------------|-------------|
| **Runbooks** | Step-by-step guides for operational tasks | During incidents, maintenance, deployments | On-call engineers, Ops team |
| **ADRs** | Records of WHY important decisions were made | When revisiting past decisions, onboarding | Entire engineering team |
| **READMEs** | Introduction and setup guide for a project | When starting with a new project/service | Developers, new team members |

Let's dive deep into each one.

## 📖 Part 1: Runbooks — Your 2 AM Best Friend

A **runbook** is a step-by-step guide that tells someone **exactly what to do** in a specific situation — like troubleshooting an incident, performing maintenance, or deploying a service.

### Why Runbooks Are Critical:

```
Without runbooks:
- Only "that one person" knows how to fix things
- New on-call engineers are terrified
- Incident response takes hours
- Same problems get solved differently each time

With runbooks:
- ANYONE on the team can handle common incidents
- Consistent, repeatable procedures
- Faster incident response
- Knowledge survives when people leave the team
```

### Runbook Template

Here's a battle-tested template you can use for any runbook:

```markdown
# Runbook: [Service Name] — [Scenario]

## Overview
- **Service:** Payment Service
- **Owner:** Checkout Team
- **Last Updated:** 2025-03-15
- **On-Call Slack Channel:** #payments-oncall

## When to Use This Runbook
Use this when you see any of the following alerts:
- `PaymentServiceDown`
- `PaymentLatencyHigh`
- `PaymentErrorRateAbove5Percent`

## Prerequisites
- Access to the `production` Kubernetes cluster
- `kubectl` configured for the payments namespace
- Access to Grafana dashboards

## Steps

### Step 1: Assess the Situation
```bash
# Check pod status
kubectl get pods -n payments

# Check recent events
kubectl get events -n payments --sort-by=.lastTimestamp | tail -20
```

**If pods are `Running` → Go to Step 2**
**If pods are `CrashLoopBackOff` → Go to Step 3**
**If pods are `Pending` → Go to Step 4**

### Step 2: Check Service Health
```bash
# Check logs for errors
kubectl logs -n payments deployment/payment-svc --tail=100

# Check metrics in Grafana
# Dashboard: https://grafana.internal/d/payments-overview
```

**Look for:**
- Database connection errors → Go to Step 5
- Memory issues → Go to Step 6
- External API timeout → Go to Step 7

### Step 3: Fix CrashLoopBackOff
```bash
# Check why the pod is crashing
kubectl logs -n payments deployment/payment-svc --previous

# Common fix: Restart the deployment
kubectl rollout restart deployment/payment-svc -n payments

# Watch the rollout
kubectl rollout status deployment/payment-svc -n payments
```

### Step 4: Fix Pending Pods
```bash
# Check if there are resource issues
kubectl describe pod <pod-name> -n payments

# Common causes:
# - Insufficient CPU/Memory → Scale the node group
# - PVC not bound → Check storage class
```

### Step 5: Database Connection Issues
```bash
# Test database connectivity from a debug pod
kubectl run -it --rm debug --image=postgres:15 \
  --restart=Never -n payments -- \
  pg_isready -h payments-db.internal -p 5432

# Check DB status in AWS RDS console:
# https://console.aws.amazon.com/rds/home?region=us-east-1
```

## Escalation
If the above steps don't resolve the issue within **30 minutes**:
1. Page the Payments Team lead: @alice (PagerDuty)
2. Post in #incident-response with timeline so far
3. Create an incident ticket in Jira (Project: OPS)

## Related Links
- Architecture Diagram: https://wiki.internal/payments-architecture
- Grafana Dashboard: https://grafana.internal/d/payments-overview
- Service Repository: https://github.com/company/payment-service
- Last Postmortem: https://wiki.internal/postmortem-2025-02-28
```

### Real-World Runbook Examples

Here are some common runbooks every DevOps team should have:

#### Runbook: Disk Space Alert

```markdown
# Runbook: Disk Space Alert (>85% Usage)

## Alert
`DiskUsageHigh` — Disk usage exceeds 85% on a node.

## Quick Fix (< 5 minutes)

### Step 1: Identify the node
```bash
# Find which node has the alert
kubectl get nodes -o wide

# SSH into the node (if needed)
ssh admin@<node-ip>
```

### Step 2: Find what's using disk space
```bash
# Check overall disk usage
df -h

# Find largest directories
du -sh /* 2>/dev/null | sort -rh | head -10

# Common culprits:
# /var/log          → Log files not being rotated
# /var/lib/docker   → Old Docker images
# /tmp              → Temp files not cleaned up
```

### Step 3: Clean up
```bash
# Clean old Docker images (safe on Kubernetes nodes)
docker system prune -af

# Clean old journal logs (keep last 2 days)
journalctl --vacuum-time=2d

# Clean old log files
find /var/log -name "*.gz" -mtime +7 -delete

# Check disk space again
df -h
```

### Step 4: Verify alert clears
Wait 5 minutes, then check if the alert has resolved in Grafana/PagerDuty.

## Prevention
- Ensure log rotation is configured (logrotate)
- Set up Docker image cleanup cron job
- Consider increasing disk size in Terraform if this happens frequently
```

#### Runbook: SSL Certificate Renewal

```markdown
# Runbook: SSL Certificate Expiry

## Alert
`SSLCertExpiringSoon` — Certificate expires in < 30 days.

## Step 1: Check certificate details
```bash
# Check certificate expiry date
echo | openssl s_client -connect myapp.example.com:443 2>/dev/null | \
  openssl x509 -noout -dates

# Output example:
# notBefore=Jan 15 00:00:00 2025 GMT
# notAfter=Apr 15 23:59:59 2025 GMT
```

## Step 2: Renew the certificate

### If using Let's Encrypt (cert-manager in K8s):
```bash
# Check cert-manager status
kubectl get certificates -A

# Force renewal
kubectl delete certificate myapp-tls -n default
# cert-manager will automatically request a new certificate

# Verify new certificate
kubectl get certificates -A
kubectl describe certificate myapp-tls -n default
```

### If using AWS ACM:
```bash
# ACM certificates auto-renew if validated via DNS
# Check in AWS console: ACM → Certificates
# If stuck, re-validate DNS records
```

## Step 3: Verify
```bash
# Verify the new certificate is being served
echo | openssl s_client -connect myapp.example.com:443 2>/dev/null | \
  openssl x509 -noout -dates -subject

# Test HTTPS connection
curl -vI https://myapp.example.com 2>&1 | grep "expire"
```
```

### Tips for Writing Great Runbooks:

```
✅ DO:
- Use numbered steps (so someone can say "I'm on step 3")
- Include exact commands (copy-paste ready)
- Add decision points ("If X → Go to Step Y")
- Include expected output for commands
- Add escalation paths ("If not fixed in 30 min, call...")
- Keep them updated (add a "Last Updated" date)
- Test your runbooks! Have someone else follow them

❌ DON'T:
- Write paragraphs of explanation in a runbook (save that for docs)
- Assume the reader knows the system well
- Use vague instructions ("check the logs" → WHERE? WHICH logs?)
- Let runbooks go stale (outdated runbooks are worse than none)
- Put runbooks only on one person's laptop
```

## 📝 Part 2: ADRs — Architecture Decision Records

An **ADR (Architecture Decision Record)** is a short document that captures **an important technical decision**, why it was made, and what alternatives were considered.

### Why ADRs Are Important:

```
6 months from now:
  New team member: "Why are we using PostgreSQL instead of MySQL?"
  
  Without ADR:
    → Nobody remembers
    → "I think Bob chose it... but Bob left the company"
    → Team debates the same decision again
    → Wastes days of discussion

  With ADR:
    → "Check ADR-005"
    → "Oh, we chose PostgreSQL because of JSON support and our team's experience"
    → Decision is clear, context is preserved
    → 5-minute conversation instead of 5-day debate
```

### The Golden Rule of ADRs:

> **ADRs are not about documenting the decision. They're about documenting the CONTEXT and REASONING behind the decision.**

Because decisions that seemed obvious at the time become mysterious 6 months later when the context is forgotten.

### ADR Template (Michael Nygard Format)

This is the most widely used ADR format, created by Michael Nygard:

```markdown
# ADR-[NUMBER]: [TITLE]

## Status
[Proposed | Accepted | Deprecated | Superseded by ADR-XXX]

## Date
YYYY-MM-DD

## Context
What is the issue or situation that is forcing us to make this decision?
What are the constraints and forces at play?

## Decision
What is the change that we're proposing or have agreed to?

## Consequences
What becomes easier or more difficult because of this decision?
What are the trade-offs?
```

### Real-World ADR Examples

#### ADR for Choosing a CI/CD Tool

```markdown
# ADR-012: Use GitHub Actions for CI/CD

## Status
Accepted

## Date
2025-02-15

## Context
We are setting up CI/CD for our microservices platform (12 services). 
We need a CI/CD solution that:
- Integrates with our GitHub repositories
- Supports Docker builds and Kubernetes deployments
- Has a reasonable learning curve for the team
- Fits within our budget ($0-500/month)

Our code is already on GitHub. Team has experience with Jenkins (3 engineers) 
and basic GitHub Actions (5 engineers).

### Options Considered

| Option | Pros | Cons |
|--------|------|------|
| **Jenkins** | Very flexible, team has some experience | Self-hosted (maintenance overhead), complex setup, plugins are fragile |
| **GitHub Actions** | Native GitHub integration, easy to start, marketplace actions, generous free tier | Vendor lock-in to GitHub, less flexible than Jenkins for complex workflows |
| **GitLab CI** | Good features, built-in registry | Would require migrating repos to GitLab |
| **CircleCI** | Fast, good Docker support | Additional cost, another vendor to manage |

## Decision
We will use **GitHub Actions** as our CI/CD platform.

## Reasoning
1. Our code is already on GitHub — native integration means zero setup overhead
2. More team members already know GitHub Actions (5 vs 3 for Jenkins)
3. Free tier covers our current needs (2000 minutes/month)
4. GitHub-hosted runners eliminate maintenance burden
5. Marketplace has pre-built actions for Docker, K8s, Terraform, etc.

## Consequences

### Positive
- No CI/CD infrastructure to maintain (GitHub-hosted runners)
- Pipeline config lives with the code (.github/workflows/)
- Team can start immediately with minimal training
- Free for our current scale

### Negative
- Complex multi-stage pipelines are less intuitive than Jenkins Pipeline
- Vendor lock-in to GitHub ecosystem
- Self-hosted runners needed for private network access (future consideration)
- Some Jenkins plugins have no GitHub Actions equivalent

### Follow-up Actions
- [ ] Create reusable workflow templates for common patterns
- [ ] Set up self-hosted runner for database migration jobs
- [ ] Document workflow conventions in team wiki
```

#### ADR for Choosing a Monitoring Stack

```markdown
# ADR-008: Use Prometheus + Grafana for Monitoring

## Status
Accepted

## Date
2025-01-20

## Context
We currently have no centralized monitoring. Each service has its own 
ad-hoc logging. When incidents happen, we SSH into servers and grep logs.

We need:
- Metrics collection from Kubernetes pods and nodes
- Alerting based on thresholds
- Dashboards for real-time visibility
- Reasonable cost (we're a startup with limited budget)

### Options Considered

| Option | Pros | Cons |
|--------|------|------|
| **Prometheus + Grafana** | Free, industry standard for K8s, huge community, PromQL is powerful | Self-hosted (maintenance), storage management, steep PromQL learning curve |
| **Datadog** | Easy setup, great UX, all-in-one | Very expensive at scale ($15-23/host/month), vendor lock-in |
| **New Relic** | Good APM, free tier available | Free tier is limited, expensive for full features |
| **ELK Stack** | Great for logs, free | Resource-heavy, complex to maintain, not ideal for metrics |

## Decision
We will use **Prometheus** for metrics collection and alerting, and 
**Grafana** for dashboards and visualization.

## Reasoning
1. Prometheus is the de-facto standard for Kubernetes monitoring
2. Zero licensing cost — critical for our startup budget
3. Kubernetes has native Prometheus support (ServiceMonitor, PodMonitor)
4. Grafana's dashboard ecosystem has thousands of pre-built dashboards
5. Large community means help is always available
6. We can migrate to a managed Prometheus (like Amazon Managed Prometheus) 
   later if maintenance becomes a burden

## Consequences

### Positive
- Industry-standard tooling (easy to hire for)
- Free and open-source
- Native Kubernetes integration
- Flexible alerting with Alertmanager

### Negative
- Self-hosted means we're responsible for uptime and storage
- PromQL has a learning curve
- Long-term metric storage requires Thanos or Cortex
- Need separate solution for logs (will address in ADR-009)
```

#### ADR for a DevOps-Specific Decision

```markdown
# ADR-015: Use Terraform Over CloudFormation for IaC

## Status
Accepted

## Date
2025-03-01

## Context
We're moving our infrastructure to code. Currently, everything is 
manually created in the AWS console. We need an IaC tool.

Our infrastructure:
- Primarily AWS (90%), some GCP for ML workloads (10%)
- 3 environments: dev, staging, production
- ~50 resources to manage (VPCs, EKS, RDS, S3, IAM, etc.)

### Options Considered

| Option | Pros | Cons |
|--------|------|------|
| **Terraform** | Multi-cloud, huge provider ecosystem, HCL is readable, large community | State file management, no rollback, HashiCorp license change (BSL) |
| **CloudFormation** | AWS-native, no state file to manage, automatic rollback | AWS-only, verbose YAML/JSON, slow updates, limited error messages |
| **Pulumi** | Real programming languages, good for complex logic | Smaller community, steeper learning curve, less mature |
| **CDK** | AWS-native, uses real languages (TypeScript/Python) | AWS-only, generates CloudFormation (adds complexity layer) |

## Decision
We will use **Terraform** with the **AWS provider** as our primary IaC tool.

## Reasoning
1. We use both AWS and GCP — Terraform supports both
2. HCL is easier to read than CloudFormation YAML
3. Largest community and module ecosystem
4. Team has 2 engineers with Terraform experience
5. State can be managed remotely in S3 + DynamoDB
6. Extensive documentation and learning resources

## Consequences

### Positive
- One tool for all our cloud infrastructure
- Reusable modules for common patterns
- Plan before apply — see changes before they happen
- Large pool of engineers who know Terraform (hiring advantage)

### Negative
- Must manage state file (S3 backend + DynamoDB locking)
- No automatic rollback (must be handled manually or via CI/CD)
- BSL license may impact future tooling choices
- Need to establish module conventions and code review practices
```

### Where to Store ADRs

```
Most common approaches:

1. In the code repository (RECOMMENDED):
   repo/
   ├── docs/
   │   └── adr/
   │       ├── 0001-use-github-actions.md
   │       ├── 0002-use-postgresql.md
   │       ├── 0003-use-terraform.md
   │       └── template.md
   ├── src/
   └── ...

2. In a dedicated documentation repo:
   company-docs/
   └── adr/
       ├── platform/
       │   ├── 0001-use-kubernetes.md
       │   └── 0002-use-prometheus.md
       └── application/
           ├── 0001-use-react.md
           └── 0002-use-graphql.md

3. In a wiki (Confluence, Notion):
   Less ideal because it's separated from the code,
   but better than nothing!
```

### ADR Lifecycle

```
Proposed → Accepted → [Lives forever as a record]
                   ↓
              Deprecated (if the decision is no longer relevant)
                   or
              Superseded by ADR-XXX (if a new decision replaces this one)

IMPORTANT: Never delete ADRs! Even deprecated ones are valuable history.
Mark them as "Superseded by ADR-XXX" so people can follow the trail.
```

### Managing ADRs with a CLI Tool

There's a handy tool called `adr-tools` that makes managing ADRs easy:

```bash
# Install adr-tools (macOS)
brew install adr-tools

# Initialize ADR directory
adr init docs/adr

# Create a new ADR
adr new "Use GitHub Actions for CI/CD"
# Creates: docs/adr/0012-use-github-actions-for-ci-cd.md

# Supersede an old ADR
adr new -s 5 "Switch from MySQL to PostgreSQL"
# Creates a new ADR and marks ADR-5 as superseded

# List all ADRs
adr list
```

## 📄 Part 3: READMEs — The Front Door of Your Project

A **README** is the **first thing** anyone sees when they look at your repository. It should answer:

1. **What** is this project?
2. **Why** does it exist?
3. **How** do I set it up and use it?

### Why READMEs Matter in DevOps:

```
You have 30 microservices. A new engineer joins the team.

Without READMEs:
  → "What does this service do?"
  → "How do I run it locally?"
  → "How do I deploy it?"
  → They ask 50 questions over 2 weeks
  → Productivity: 📉

With READMEs:
  → They read the README
  → They have the app running locally in 30 minutes
  → They understand the architecture
  → They're productive in 2 days
  → Productivity: 📈
```

### README for a CI/CD Pipeline

```markdown
# 🔄 CI/CD Pipelines

> Shared CI/CD workflows and pipeline configurations.

## Workflows

| Workflow | Trigger | What It Does |
|----------|---------|-------------|
| `ci.yml` | PR to main | Lint, test, build Docker image |
| `deploy-staging.yml` | Merge to main | Deploy to staging |
| `deploy-production.yml` | Git tag (v*) | Deploy to production |
| `terraform-plan.yml` | PR with infra changes | Run `terraform plan` |

## Usage

### For Application Repos:

Add to `.github/workflows/ci.yml`:

```yaml
name: CI
on:
  pull_request:
    branches: [main]

jobs:
  build:
    uses: company/ci-cd-pipelines/.github/workflows/ci.yml@main
    with:
      app-name: my-service
      dockerfile: ./Dockerfile
    secrets: inherit
```

### For Infrastructure Repos:

Add to `.github/workflows/terraform.yml`:

```yaml
name: Terraform
on:
  pull_request:
    branches: [main]
    paths:
      - 'infrastructure/**'

jobs:
  plan:

## ✍️ Writing Tips — How to Write Documentation That People Actually Read

### The KISS Principle: Keep It Simple, Stupid

```
❌ Bad:
"In order to facilitate the deployment of the application to the 
production environment, it is necessary to first ensure that all 
prerequisite conditions have been satisfactorily met, including 
but not limited to the configuration of the appropriate environment 
variables as specified in the configuration documentation."

✅ Good:
"Before deploying to production:
1. Copy `.env.example` to `.env`
2. Fill in the required values
3. Run `make deploy-prod`"
```

### Use the Inverted Pyramid

Put the most important information first:

```
1. WHAT (summary — 1-2 sentences)
2. HOW (quick start — get running in 5 minutes)
3. WHY (context and details)
4. REFERENCE (full API docs, configuration options)
```

### Make It Scannable

```
✅ Use:
- Headers (##, ###)
- Bullet points
- Tables
- Code blocks
- Bold for key terms
- Short paragraphs (3-4 lines max)

❌ Avoid:
- Long paragraphs
- Walls of text
- No formatting
- Jargon without explanation
```

### The "New Person" Test

> **If a new team member can follow your documentation and complete the task without asking anyone for help, your documentation is good.**

Try this:
1. Write the doc
2. Give it to someone who's never done the task
3. Watch them (don't help!)
4. Note where they get stuck
5. Update the doc
6. Repeat

## 📂 Documentation as Code — Keep Docs Next to Code

The best practice in DevOps is to treat documentation like code:

```
my-service/
├── .github/
│   └── workflows/
├── docs/
│   ├── adr/
│   │   ├── 0001-use-postgresql.md
│   │   └── 0002-use-github-actions.md
│   ├── runbooks/
│   │   ├── incident-response.md
│   │   ├── database-failover.md
│   │   └── disk-space-alert.md
│   └── architecture.md
├── src/
├── Dockerfile
├── Makefile
├── README.md              ← Root README (project overview)
└── docker-compose.yml
```

### Why Keep Docs with Code?

| Benefit | Explanation |
|---------|-------------|
| **Version controlled** | Docs change with the code — you can see what the docs said at any point in time |
| **Review process** | Docs go through the same PR review as code |
| **Discoverable** | New team members find docs right in the repo |
| **Always updated** | When code changes, docs are updated in the same PR |
| **No separate tool** | No Confluence page that nobody updates |

### Automating Documentation

You can even automate some documentation with your CI/CD pipeline:

```yaml
# .github/workflows/docs.yml
name: Update Documentation

on:
  push:
    branches: [main]

jobs:
  update-docs:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Generate API docs from OpenAPI spec
        run: |
          npx redocly build-docs api/openapi.yaml -o docs/api.html

      - name: Generate Terraform docs
        uses: terraform-docs/gh-actions@v1
        with:
          working-dir: infrastructure/modules
          output-file: README.md
      
      - name: Commit updated docs
        run: |
          git config user.name "docs-bot"
          git config user.email "docs-bot@company.com"
          git add docs/
          git diff --staged --quiet || git commit -m "docs: auto-update documentation"
          git push
```

**What this pipeline does:**
- Every time code is pushed to `main`, it automatically regenerates documentation
- It builds API documentation from an OpenAPI specification file
- It generates Terraform module documentation using `terraform-docs`
- It commits the updated docs back to the repository
- The `git diff --staged --quiet ||` part ensures it only commits if there are actual changes

## 🎯 Key Takeaways

```
1. Runbooks save lives (and sleep) — write step-by-step guides for every 
   common operational task
2. ADRs preserve context — document WHY decisions were made, not just WHAT
3. READMEs are the front door — they should get someone productive in 
   30 minutes or less
4. Keep docs next to code — version control, PR reviews, always discoverable
5. Write for the "2 AM on-call engineer" — clear, concise, copy-paste ready
6. Never delete ADRs — mark them as superseded, they're historical records
7. Test your docs — if someone can't follow them alone, they need improvement
8. Use the Inverted Pyramid — most important info first
9. Automate where possible — generate docs from code/specs
10. Stale docs are worse than no docs — keep them updated or delete them
```
