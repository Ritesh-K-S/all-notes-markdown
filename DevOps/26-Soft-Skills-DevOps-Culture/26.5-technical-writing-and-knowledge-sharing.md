# 26.5 — Technical Writing & Knowledge Sharing

> **One Line Answer:** Technical writing is the art of explaining complex things in simple, clear, and structured ways — and knowledge sharing is the practice of making sure that knowledge doesn't stay trapped in one person's brain.

---

## 🤔 Why Should a DevOps Engineer Care About Writing?

You might be thinking, *"I'm an engineer, not a writer. Why do I need this?"*

Here's the truth:

> **The best engineers are not just the ones who can solve problems — they're the ones who can explain HOW they solved them so others can learn too.**

Think about it:

```
Scenario: You spent 3 hours debugging a weird Kubernetes networking issue.
You finally fixed it. 🎉

Option A: You close the ticket and move on.
→ 6 months later, your teammate hits the SAME issue.
→ They spend 3 hours debugging it too.
→ Total time wasted: 6 hours (and counting, for every future person)

Option B: You write a 10-minute doc explaining the problem and fix.
→ 6 months later, your teammate hits the same issue.
→ They search the docs, find your write-up, fix it in 5 minutes.
→ Total time wasted: 3 hours + 10 minutes + 5 minutes = 3h 15m
→ And EVERY future person benefits too.

That 10-minute doc keeps saving time FOREVER.
```

### The "Bus Factor"

There's a concept called the **Bus Factor** — it asks:

> *"How many people would need to get hit by a bus (or quit, or go on vacation) before the team is completely stuck?"*

```
Bus Factor = 1 (BAD 🚨)
→ Only ONE person knows how the deployment pipeline works
→ Only ONE person knows the database password rotation process
→ Only ONE person knows how to debug the payment service

Bus Factor = 5+ (GOOD ✅)
→ Multiple people know each critical system
→ Everything is documented
→ Anyone on the team can handle any system in an emergency
```

**Technical writing and knowledge sharing increase your Bus Factor.** They make your team resilient.

## 🧱 Part 1: Technical Writing Fundamentals

### The Golden Rule of Technical Writing

> **Write for your reader, not for yourself.**

When you write documentation, you already understand the topic. Your reader doesn't. Your job is to **bridge that gap**.

```
❌ Writing for yourself:
   "Configure the nginx reverse proxy with upstream 
    directives and ensure proper header forwarding."
   
   → You know what this means. Your reader might not.

✅ Writing for your reader:
   "Set up Nginx to forward incoming requests to your 
    application server. We'll configure it so that when 
    someone visits your website, Nginx sends the request 
    to the right backend service."
   
   → Clear. Simple. Anyone can understand it.
```

### The 5 Principles of Good Technical Writing

```
┌────────────────────────────────────────────────────────────────┐
│                                                                │
│  1. CLEAR   → Use simple words. Short sentences. No jargon    │
│               unless you explain it first.                     │
│                                                                │
│  2. CONCISE → Say what needs to be said. Nothing more.        │
│               Cut filler words. Get to the point.              │
│                                                                │
│  3. STRUCTURED → Use headings, lists, and tables.             │
│                  Make it scannable. Nobody reads                │
│                  wall-of-text documentation.                   │
│                                                                │
│  4. ACCURATE → Test your instructions. If you say             │
│                "run this command," make sure it actually works. │
│                                                                │
│  5. MAINTAINABLE → Write docs that are easy to UPDATE.        │
│                    Outdated docs are worse than no docs.        │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### Writing Style Tips

#### Tip 1: Use Active Voice

```
❌ Passive Voice:
   "The configuration file should be edited by the user."
   "The server was restarted by the admin."
   "The pipeline is triggered when code is pushed."

✅ Active Voice:
   "Edit the configuration file."
   "The admin restarted the server."
   "The pipeline triggers when you push code."

Active voice is clearer, shorter, and more direct.
```

#### Tip 2: Use Second Person ("you")

```
❌ "The user should run the following command..."
❌ "One must ensure that the service is running..."

✅ "Run the following command..."
✅ "Make sure the service is running..."

Speaking directly to the reader is friendlier and clearer.
```

#### Tip 3: Define Jargon Before Using It

```
❌ "Configure the HPA with a target CPU utilization of 80%."
   → What's an HPA? What does target CPU utilization mean?

✅ "Configure the Horizontal Pod Autoscaler (HPA) — this is a 
    Kubernetes feature that automatically adds or removes pods 
    based on how much CPU they're using. We'll set it to add 
    more pods when CPU usage goes above 80%."
   → Now even a beginner understands.
```

#### Tip 4: Use Code Blocks Generously

Always put commands, code, and file content in code blocks with proper syntax highlighting:

````
❌ Without code block:
   Run kubectl get pods -n production to see all running pods.

✅ With code block:
   Run the following command to see all running pods in the production namespace:
   
   ```bash
   kubectl get pods -n production
   ```
   
   Expected output:
   ```
   NAME                        READY   STATUS    RESTARTS   AGE
   web-app-6d8f5c7b9-abc12    1/1     Running   0          2h
   api-server-7f9d3c4e2-xyz34 1/1     Running   0          1h
   ```
````

#### Tip 5: Add Context — The "Why" Before the "How"

```
❌ Just instructions (no context):
   1. Run: kubectl apply -f deployment.yaml
   2. Run: kubectl apply -f service.yaml
   3. Run: kubectl apply -f ingress.yaml

✅ Instructions with context:
   We need to deploy three resources to get the app running:
   
   1. **Deployment** — This creates and manages the pods running your app.
      ```bash
      kubectl apply -f deployment.yaml
      ```
   
   2. **Service** — This creates a stable network endpoint for the pods.
      ```bash
      kubectl apply -f service.yaml
      ```
   
   3. **Ingress** — This exposes the service to the outside world via a URL.
      ```bash
      kubectl apply -f ingress.yaml
      ```
```

#### Tip 6: Use Visuals and Diagrams

A picture is worth a thousand words. Use ASCII diagrams, Mermaid diagrams, or screenshots:

```
Traffic Flow:

  User → Load Balancer → Nginx (Reverse Proxy) → App Server → Database
                                    │
                                    └→ Static Files (S3 / CDN)
```

## 🧱 Part 2: Types of Documentation in DevOps

Different documents serve different purposes. Here's a complete guide to each type:

### 1. README Files

The **front door** of any project. It's the first thing people read.

```markdown
# Project Name

One-line description of what this project does.

## Quick Start

How to get this running in 5 minutes or less.

## Prerequisites

What you need before you begin:
- Docker 20.10+
- kubectl 1.25+
- AWS CLI configured

## Installation

Step-by-step installation instructions.

## Usage

Common commands and how to use the project.

## Configuration

What environment variables / config files exist and what they do.

## Architecture

High-level diagram of how the system works.

## Contributing

How to contribute (branch naming, PR process, etc.)

## Troubleshooting

Common problems and their solutions.
```

**Tips for great READMEs:**
- Start with a **Quick Start** — people want to try it immediately
- Include **actual commands** they can copy-paste
- Keep it **up to date** — an outdated README is worse than none
- Add a **Troubleshooting** section — save people from Googling

### 2. Runbooks

A **runbook** is a step-by-step guide for handling operational tasks or incidents. Think of it as a recipe — anyone should be able to follow it, even at 3 AM when half-asleep.

```markdown
# Runbook: Database Disk Space Alert

## When to Use This Runbook
When you receive an alert: "Database disk usage > 85%"

## Severity
- > 85%: Warning — follow steps below
- > 95%: Critical — follow steps below + escalate immediately

## Step 1: Check Current Disk Usage

SSH into the database server and check disk usage:

```bash
df -h /var/lib/postgresql/data
```

Expected output:
```
Filesystem      Size  Used  Avail  Use%  Mounted on
/dev/sda1       100G  87G   13G    87%   /var/lib/postgresql/data
```

## Step 2: Identify What's Using Space

```bash
# Check for large log files
du -sh /var/lib/postgresql/data/pg_log/*

# Check for temp files
du -sh /var/lib/postgresql/data/base/pgsql_tmp/
```

## Step 3: Clean Up (Safe Actions)

### Option A: Clean old WAL files (safe)
```bash
# Check WAL file count
ls -la /var/lib/postgresql/data/pg_wal/ | wc -l

# If more than 100 files, archive old ones
pg_archivecleanup /var/lib/postgresql/data/pg_wal/ <latest_checkpoint>
```

### Option B: Vacuum the database (safe)
```bash
psql -U postgres -c "VACUUM FULL VERBOSE;"
```

### Option C: Expand disk (if cleaning isn't enough)
→ Follow the "Disk Expansion Runbook" (link)
→ Requires infrastructure team approval

## Step 4: Verify

After cleanup, verify disk usage is back below 80%:
```bash
df -h /var/lib/postgresql/data
```

## Escalation
If none of the above works, escalate to:
- On-call DBA: Check PagerDuty schedule
- Infrastructure Team: #infra-oncall Slack channel

## Related Runbooks
- [Database Backup Runbook](link)
- [Disk Expansion Runbook](link)
```

**Tips for great runbooks:**
- Write them so someone who has **never seen this system** can follow them
- Include **expected output** for every command
- Include **what NOT to do** (dangerous actions)
- Include **escalation paths** — who to call when things are really bad
- **Test your runbooks** — actually run through them to make sure they work
- **Date them** — add a "Last verified" date so people know if they're fresh

### 3. Architecture Decision Records (ADRs)

An **ADR** documents WHY you made a specific technical decision. This is incredibly valuable because 6 months later, nobody remembers WHY you chose PostgreSQL over MongoDB, or why you picked ArgoCD over Flux.

```markdown
# ADR-007: Use ArgoCD for GitOps

## Status
Accepted

## Date
2026-03-15

## Context
We need a GitOps tool to manage Kubernetes deployments across 
3 clusters (dev, staging, production). Our requirements:
- Must support Helm and Kustomize
- Must have a UI for visibility
- Must support RBAC
- Must integrate with our SSO (Okta)

## Decision
We will use ArgoCD as our GitOps tool.

## Alternatives Considered

### Flux CD
- Pros: Lightweight, Kubernetes-native, CNCF graduated
- Cons: No built-in UI, steeper learning curve for our team

### Jenkins + kubectl
- Pros: Team already knows Jenkins
- Cons: Push-based (not true GitOps), harder to manage drift

### Spinnaker
- Pros: Feature-rich, supports multiple cloud providers
- Cons: Very complex to operate, overkill for our needs

## Reasons for Choosing ArgoCD
1. Built-in UI — our team needs visibility without CLI expertise
2. Native Helm + Kustomize support — matches our current tooling
3. SSO integration — works with Okta out of the box
4. Large community — 14k+ GitHub stars, active development
5. CNCF graduated project — proven at scale

## Consequences
- Team needs training on ArgoCD (estimated: 1 week)
- Need to set up ArgoCD HA deployment in production
- All deployment manifests must be in Git (this is the goal anyway)
- We lose some Jenkins pipeline flexibility (acceptable tradeoff)
```

**Tips for great ADRs:**
- Keep them **short** — one page is ideal
- Always document **alternatives you considered** and why you rejected them
- Include the **context** — what problem were you solving?
- Record the **consequences** — what tradeoffs did you accept?
- Number them sequentially (ADR-001, ADR-002, etc.)
- Never delete old ADRs — if you change a decision, create a new ADR that supersedes the old one

### 4. How-To Guides

Step-by-step instructions for accomplishing a specific task:

```markdown
# How To: Set Up a New Microservice

## Overview
This guide walks you through creating and deploying a new microservice 
in our infrastructure. Expected time: ~30 minutes.

## Prerequisites
- Access to the GitHub organization
- kubectl configured for your cluster
- Helm installed (v3.10+)

## Steps

### 1. Create the Repository
Use our service template:
```bash
gh repo create myorg/new-service --template myorg/service-template --private
git clone git@github.com:myorg/new-service.git
cd new-service
```

### 2. Configure the Service
Edit `config/settings.yaml`:
```yaml
service:
  name: new-service
  port: 8080
  replicas: 2
  resources:
    cpu: 250m
    memory: 256Mi
```

### 3. Set Up CI/CD
The template includes a GitHub Actions workflow. You only need to add secrets:

1. Go to Settings → Secrets → Actions
2. Add: `DOCKER_REGISTRY` = `123456789.dkr.ecr.us-east-1.amazonaws.com`
3. Add: `KUBE_CONFIG` = (get from 1Password → DevOps vault)

### 4. Deploy to Dev
```bash
# Build and push Docker image
docker build -t new-service:latest .
docker push $REGISTRY/new-service:latest

# Deploy to dev cluster
helm upgrade --install new-service ./helm/new-service \
  --namespace dev \
  --values helm/new-service/values-dev.yaml
```

### 5. Verify
```bash
kubectl get pods -n dev -l app=new-service
# Should show 2 pods in Running state

# Test the service
curl http://new-service.dev.internal:8080/health
# Should return: {"status": "ok"}
```

## Common Issues
| Problem | Solution |
|---------|----------|
| Pods stuck in ImagePullBackOff | Check ECR login: `aws ecr get-login-password` |
| Service unreachable | Check NetworkPolicy allows ingress on port 8080 |
| Health check failing | Ensure /health endpoint is implemented |
```

### 5. Troubleshooting Guides

Documents that help diagnose and fix common problems:

```markdown
# Troubleshooting: Pod CrashLoopBackOff

## What is CrashLoopBackOff?
It means your container keeps crashing and Kubernetes keeps restarting it.
Each restart has an increasing delay (10s, 20s, 40s... up to 5 minutes).

## Step 1: Check Pod Logs
```bash
# Current logs
kubectl logs <pod-name> -n <namespace>

# Previous crash logs (most useful!)
kubectl logs <pod-name> -n <namespace> --previous
```

## Step 2: Check Pod Events
```bash
kubectl describe pod <pod-name> -n <namespace>
```
Look at the "Events" section at the bottom.

## Common Causes and Fixes

### Cause 1: Application Error
**Symptoms:** Logs show stack trace or error message
**Fix:** Fix the application code or configuration

### Cause 2: Missing Config/Secret
**Symptoms:** Logs say "config file not found" or "env var not set"  
**Fix:** 
```bash
# Check if ConfigMap exists
kubectl get configmap <name> -n <namespace>

# Check if Secret exists  
kubectl get secret <name> -n <namespace>
```

### Cause 3: Out of Memory (OOMKilled)
**Symptoms:** `kubectl describe pod` shows "OOMKilled"
**Fix:** Increase memory limits:
```yaml
resources:
  limits:
    memory: "512Mi"  # Increase this
```

### Cause 4: Liveness Probe Failing
**Symptoms:** Events show "Liveness probe failed"
**Fix:** Check your health endpoint works and adjust timing:
```yaml
livenessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 30  # Give app time to start
  periodSeconds: 10
```
```

## 🧱 Part 3: Knowledge Sharing Practices

Writing great docs is only half the battle. You also need to make sure **people actually find and use them**. That's where knowledge sharing practices come in.

### 1. Write-Up Culture

Make writing a **natural part of work**, not an extra task:

```
When to Write Things Down:

✅ After fixing a tricky bug
   → "How I fixed the intermittent timeout issue in the payment service"

✅ After learning something new
   → "TIL: Kubernetes DNS resolution and why ndots:5 matters"

✅ After making a decision
   → ADR documenting why you chose Tool A over Tool B

✅ After an incident
   → Blameless postmortem (see section 26.4)

✅ After setting something up
   → Step-by-step guide so the next person doesn't struggle

✅ When you answer the same question twice
   → Write it down. The third person won't need to ask.
```

### 2. "Document As You Go" vs "Document After"

```
❌ "We'll write docs later"
   → "Later" never comes. You forget the details.
   → Docs never get written.

✅ "Document as you go"
   → Write docs WHILE you're doing the work
   → The details are fresh in your mind
   → It actually gets done
   
Practical approach:
1. Open a blank doc before you start
2. Write down each step as you do it
3. Copy-paste commands and outputs
4. Clean it up slightly when you're done
5. Publish it — imperfect docs are better than no docs
```

### 3. Internal Tech Talks / Brown Bag Sessions

```
What: Short (30-45 min) presentations where someone teaches 
      the team about a topic they know well.

Examples:
→ "How our CI/CD pipeline works (end to end)"
→ "Intro to eBPF and why it matters for observability"
→ "How I debugged the production memory leak last week"
→ "Terraform modules — our patterns and anti-patterns"

Format:
→ 20-30 min presentation
→ 10-15 min Q&A
→ Record it (for people who can't attend)
→ Share slides + recording in a central location

Tips for speakers:
→ You don't have to be an expert. Teaching is learning.
→ Live demos > Slides (but have a backup if the demo fails)
→ Focus on the "why" and "how", not just the "what"
→ Share mistakes you made — it's the most valuable part
```

### 4. Pair Programming / Pair Operations

```
What: Two people work together on the same task.
      One drives (types), one navigates (reviews, suggests).

Why:
→ Knowledge transfers naturally through conversation
→ You learn the OTHER person's shortcuts, tools, and mental models
→ It's real-time code review
→ It's more fun than working alone

Best Practices:
→ Swap roles every 30 minutes
→ Pair senior + junior for learning
→ Pair cross-team for breaking silos (Dev + Ops)
→ Don't force it every day — 2-3 times a week is plenty
→ Remote? Use VS Code Live Share or screen sharing
```

### 6. Code Reviews as Knowledge Sharing

Code reviews aren't just about finding bugs — they're one of the best knowledge sharing opportunities:

```
How to make code reviews a learning experience:

For the Reviewer:
─────────────────
✅ Explain the WHY behind your suggestions
   ❌ "Use a map here instead of a list"
   ✅ "A map would be better here because lookup is O(1) vs O(n) 
      for a list, and we're searching in a loop"

✅ Ask questions instead of making demands
   ❌ "This is wrong"
   ✅ "I'm curious — what happens if this value is nil?"

✅ Point out good things too
   ✅ "Nice! I didn't know about this library. TIL."
   ✅ "Great error handling here."

For the Author:
───────────────
✅ Write good PR descriptions
   → What does this change do? Why?
   → How to test it?
   → Any risks?

✅ Keep PRs small (< 400 lines ideally)
   → Smaller PRs = better reviews = faster feedback

✅ Respond to feedback gracefully
   → "Good catch, fixed!" not "Well actually..."
```

## ⚠️ Common Pitfalls in Technical Writing & Knowledge Sharing

```
Pitfall #1: "Perfect is the enemy of good"
──────────────────────────────────────────
Don't wait until docs are perfect to publish them.
Imperfect docs > No docs. Always.
You can always improve them later.

Pitfall #2: "Write once, never update"
──────────────────────────────────────
Outdated documentation is DANGEROUS.
Someone follows old instructions → breaks something.
Add "Last Updated" dates and review docs quarterly.

Pitfall #3: "Too much detail"
──────────────────────────────
Write for your audience. A runbook for on-call doesn't 
need to explain what Kubernetes is. Get to the point.

Pitfall #4: "Documentation is someone else's job"
──────────────────────────────────────────────────
If you build it, document it. Period.
Don't wait for a "technical writer" to do it.
The person who knows the system writes the docs.

Pitfall #5: "Just read the code"
─────────────────────────────────
Code tells you WHAT it does. Docs tell you WHY.
Code can't explain business context, design decisions, 
or operational procedures. You need both.

Pitfall #6: "Knowledge sharing takes too much time"
────────────────────────────────────────────────────
It takes time NOW but saves 10x time LATER.
Every question you answer twice that could be a doc 
is time wasted — for you AND the person asking.
```

## 📊 Measuring Knowledge Sharing Effectiveness

```
Metrics to Track:

1. Onboarding Time
   → How quickly can a new person become productive?
   → If it takes 2+ weeks, your docs need work.

2. Repeat Questions
   → Are people asking the same questions in Slack?
   → If yes, write it down and link to it.

3. Documentation Coverage
   → Do all critical systems have runbooks?
   → Do all services have READMEs?
   → Are all major decisions documented as ADRs?

4. Doc Freshness
   → When were docs last updated?
   → Are they still accurate?

5. Search Success Rate
   → When people search the wiki, do they find what they need?
   → If not, the docs exist but aren't discoverable.

6. Bus Factor
   → For each critical system, how many people can operate it?
   → Target: at least 2-3 people per system.
```

## 🧠 Key Takeaways

```
1. Writing is a superpower.
   → The engineer who can explain complex things simply is 
     10x more valuable than one who can't.

2. Document as you go.
   → "Later" never comes. Write while it's fresh.

3. Focus on the reader.
   → They don't have your context. Bridge the gap.

4. Imperfect docs > No docs.
   → Don't let perfect be the enemy of good. Ship it.

5. Knowledge shared = knowledge multiplied.
   → One person's 3-hour debugging session can save everyone 
     else from the same pain.

6. Increase your Bus Factor.
   → If only one person knows something, write it down TODAY.

7. Make it findable.
   → Docs that nobody can find are docs that don't exist.
     Use a central, searchable knowledge base.
```
