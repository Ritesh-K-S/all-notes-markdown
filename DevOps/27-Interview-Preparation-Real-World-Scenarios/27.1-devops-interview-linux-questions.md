### Q1. What is Linux? Why is it used so much in DevOps?

**Answer:**

Linux is a **free, open-source operating system** based on Unix. It was created by **Linus Torvalds** in 1991.

Why DevOps engineers love Linux:

| Reason | Explanation |
|--------|-------------|
| **Free & Open Source** | No licensing costs. You can modify it however you want. |
| **Stability** | Linux servers run for YEARS without rebooting. |
| **Security** | Strong permission model, SELinux, minimal attack surface. |
| **Automation-Friendly** | Everything can be done via command line — perfect for scripting. |
| **Dominates the Server World** | 96%+ of the world's top servers run Linux. |
| **Container Foundation** | Docker, Kubernetes — all built on Linux (namespaces, cgroups). |

💡 **Pro Tip:** Mention specific distributions — "I mostly work with Ubuntu for development, and RHEL/CentOS for production environments."

### Q2. What is the Linux File System Hierarchy? Explain key directories.

**Answer:**

In Linux, everything starts from `/` (root). Here's what each important directory does:

```
/
├── /bin      → Essential commands (ls, cp, mv, cat) — available to all users
├── /sbin     → System admin commands (fdisk, iptables, reboot)
├── /etc      → Configuration files (nginx.conf, ssh config, passwd)
├── /home     → Home directories for regular users (/home/john, /home/ritesh)
├── /root     → Home directory for the root user
├── /var      → Variable data — logs (/var/log), mail, spool, temp data
├── /tmp      → Temporary files — cleared on reboot
├── /opt      → Optional/third-party software (Jenkins, Tomcat, custom apps)
├── /usr      → User programs and libraries
│   ├── /usr/bin    → Non-essential user commands
│   ├── /usr/lib    → Libraries for /usr/bin programs
│   └── /usr/local  → Locally compiled software
├── /dev      → Device files (disks, terminals, USB)
├── /proc     → Virtual filesystem — live kernel/process info
├── /sys      → Virtual filesystem — hardware/device info
├── /boot     → Boot loader files (kernel, GRUB)
├── /lib      → Shared libraries (like DLLs in Windows)
└── /mnt      → Temporary mount points
```

💡 **Interview Gold:** If they ask "Where are Nginx config files?" → `/etc/nginx/`. "Where are logs?" → `/var/log/`. Know the pattern — config goes in `/etc`, logs go in `/var/log`, binaries go in `/usr/bin` or `/usr/local/bin`.

### Q3. What is the difference between absolute path and relative path?

**Answer:**

| Type | Starts From | Example |
|------|-------------|---------|
| **Absolute Path** | Always starts from `/` (root) | `/home/ritesh/projects/app.py` |
| **Relative Path** | Starts from your current directory | `./projects/app.py` or `../config/settings.yml` |

```bash
# Absolute — works from anywhere
cat /etc/nginx/nginx.conf

# Relative — depends on where you are right now
cd /etc/nginx
cat nginx.conf          # same file, relative path
cat ../ssh/sshd_config  # go up one level, then into ssh/
```

**Key symbols:**
- `.` → current directory
- `..` → parent directory (one level up)
- `~` → home directory of current user

### Q4. What are hard links and soft links (symbolic links)?

**Answer:**

Think of it like this:

- **Hard Link** = Another name for the SAME file. Both names point to the same data on disk. If you delete one, the other still works.
- **Soft Link (Symlink)** = A shortcut/pointer to another file. If you delete the original, the symlink breaks.

```bash
# Create a regular file
echo "Hello World" > original.txt

# Create a hard link
ln original.txt hardlink.txt

# Create a soft link (symlink)
ln -s original.txt softlink.txt

# Check them
ls -li
# Notice: original.txt and hardlink.txt have the SAME inode number
# softlink.txt has a DIFFERENT inode and shows -> original.txt
```

| Feature | Hard Link | Soft Link (Symlink) |
|---------|-----------|-------------------|
| **Points to** | Same inode (same data) | Path of another file |
| **Delete original** | Hard link still works | Symlink breaks (dangling) |
| **Cross filesystem** | ❌ No | ✅ Yes |
| **Link to directory** | ❌ No (usually) | ✅ Yes |
| **Command** | `ln file link` | `ln -s file link` |

💡 **Real-world use:** Symlinks are everywhere in DevOps — `/etc/nginx/sites-enabled/mysite` is usually a symlink to `/etc/nginx/sites-available/mysite`.

### Q5. What is an inode in Linux?

**Answer:**

An **inode** is like an **ID card** for a file. It stores all the metadata about a file EXCEPT the filename and the actual content.

What's stored in an inode:
- File size
- Owner (UID) and Group (GID)
- Permissions (read/write/execute)
- Timestamps (created, modified, accessed)
- Number of hard links
- Pointers to the actual data blocks on disk

```bash
# See the inode number of a file
ls -i myfile.txt
# Output: 1234567 myfile.txt

# See detailed inode info
stat myfile.txt
```

💡 **Key insight:** The filename is NOT stored in the inode — it's stored in the directory entry. A directory is basically a table that maps filenames → inode numbers.

## 👤 Section 2: Users, Groups & Permissions

---

### Q6. Explain Linux file permissions. What does `chmod 755` mean?

**Answer:**

Every file in Linux has three sets of permissions for three types of users:

```
-rwxr-xr-x  →  This is what you see with ls -l
│├─┤├─┤├─┤
│ │   │  └── Others (everyone else)
│ │   └───── Group (users in the file's group)
│ └───────── Owner (the user who owns the file)
└──────────── File type (- = file, d = directory, l = link)
```

Each permission has a number:
- `r` (read) = **4**
- `w` (write) = **2**
- `x` (execute) = **1**

So `chmod 755`:
- **7** (owner) = 4+2+1 = read + write + execute
- **5** (group) = 4+0+1 = read + execute
- **5** (others) = 4+0+1 = read + execute

```bash
# Give full permissions to owner, read+execute to group and others
chmod 755 script.sh

# Same thing in symbolic mode
chmod u=rwx,g=rx,o=rx script.sh

# Make a file readable by everyone, writable only by owner
chmod 644 config.txt

# Remove all permissions for others
chmod o= secretfile.txt
```

**Common permission numbers you should memorize:**

| Number | Meaning | Used For |
|--------|---------|----------|
| `777` | Everyone can do everything | ⚠️ NEVER use in production |
| `755` | Owner: full, Others: read+execute | Scripts, directories |
| `644` | Owner: read+write, Others: read | Config files, documents |
| `600` | Only owner can read+write | SSH keys, secrets |
| `700` | Only owner can do everything | Private scripts, `.ssh/` directory |

### Q7. What is the difference between `chmod`, `chown`, and `chgrp`?

**Answer:**

| Command | What It Changes | Example |
|---------|----------------|---------|
| `chmod` | **Permissions** (read/write/execute) | `chmod 755 file.sh` |
| `chown` | **Owner** (and optionally group) | `chown ritesh file.sh` |
| `chgrp` | **Group** only | `chgrp devops file.sh` |

```bash
# Change owner to 'ritesh'
chown ritesh myfile.txt

# Change owner AND group at the same time
chown ritesh:devops myfile.txt

# Change ownership recursively (all files in a directory)
chown -R ritesh:devops /var/www/html/

# Change group only
chgrp docker myfile.txt
```

💡 **Interview follow-up:** "What's the difference between `chown user:group file` and `chown user.group file`?" → Both work, but `:` is the modern/correct way. The `.` syntax is legacy.

### Q8. What is `umask`? How does it work?

**Answer:**

`umask` sets the **default permissions** for newly created files and directories.

Think of it as: "What permissions should I REMOVE from new files?"

```
Default permissions for files:       666 (rw-rw-rw-)
Default permissions for directories:  777 (rwxrwxrwx)

If umask is 022:
  Files:       666 - 022 = 644 (rw-r--r--)
  Directories: 777 - 022 = 755 (rwxr-xr-x)
```

```bash
# Check current umask
umask
# Output: 0022

# Change umask for current session
umask 027
# Now: Files get 640, Directories get 750

# Make it permanent — add to ~/.bashrc or /etc/profile
echo "umask 027" >> ~/.bashrc
```

💡 **Common umask values:**
- `022` → Default on most systems (files: 644, dirs: 755)
- `027` → More secure (files: 640, dirs: 750) — group can read, others get nothing
- `077` → Very restrictive (files: 600, dirs: 700) — only owner has access

### Q9. What is the difference between `su` and `sudo`?

**Answer:**

| Feature | `su` | `sudo` |
|---------|------|--------|
| **Full form** | Switch User | Superuser Do |
| **What it does** | Switches to another user entirely | Runs ONE command as root (or another user) |
| **Password needed** | Password of the TARGET user | Password of the CURRENT user |
| **Logging** | Minimal | Every sudo command is logged in `/var/log/auth.log` |
| **Security** | Less secure (you share root password) | More secure (no need to share root password) |

```bash
# Switch to root user (need root's password)
su -
# You are now root. Every command runs as root until you type 'exit'.

# Run a single command as root (need YOUR password)
sudo apt update
# Runs just this one command as root, then you're back to normal.

# Switch to another user
su - john

# Run a command as another user using sudo
sudo -u john whoami
# Output: john
```

💡 **Best Practice:** In DevOps, we almost always use `sudo` instead of `su`. Why? Because `sudo` is logged, auditable, and you can control exactly which commands each user can run (via `/etc/sudoers`).

---

### Q10. How do you add a user to the sudoers file?

**Answer:**

There are two ways:

**Method 1: Add user to the `sudo` group (Recommended)**
```bash
# On Ubuntu/Debian
usermod -aG sudo ritesh

# On RHEL/CentOS
usermod -aG wheel ritesh
```

**Method 2: Edit the sudoers file directly**
```bash
# ALWAYS use visudo (it validates syntax before saving!)
sudo visudo

# Add this line at the bottom:
ritesh ALL=(ALL) ALL
# Meaning: user 'ritesh' can run ALL commands as ALL users on ALL hosts

# For no-password sudo (useful for automation, less secure):
ritesh ALL=(ALL) NOPASSWD: ALL
```

⚠️ **NEVER edit `/etc/sudoers` directly with `vim` or `nano`!** If you make a syntax error, you can lock yourself out of sudo entirely. Always use `visudo`.

## ⚙️ Section 3: Process Management

---

### Q11. How do you check running processes in Linux?

**Answer:**

```bash
# Show all running processes (snapshot)
ps aux
# a = all users, u = user-oriented format, x = include processes without a terminal

# Show processes in a tree format (see parent-child relationships)
ps auxf
# or
pstree

# Real-time, interactive process viewer
top

# Better version of top (if installed)
htop

# Show only processes matching a name
ps aux | grep nginx

# Count how many nginx processes are running
ps aux | grep nginx | grep -v grep | wc -l
```

**Understanding `ps aux` output:**

```
USER       PID  %CPU %MEM    VSZ   RSS TTY      STAT START   TIME COMMAND
root         1   0.0  0.1 169532 13300 ?        Ss   May29   0:03 /sbin/init
nginx     1234   0.1  0.5 234567 45678 ?        S    10:00   0:15 nginx: worker
```

| Column | Meaning |
|--------|---------|
| `PID` | Process ID — unique number for each process |
| `%CPU` | CPU usage percentage |
| `%MEM` | Memory usage percentage |
| `VSZ` | Virtual memory size |
| `RSS` | Resident Set Size — actual physical memory used |
| `STAT` | Process state (S=sleeping, R=running, Z=zombie, T=stopped) |

### Q12. What is the difference between a process and a daemon?

**Answer:**

| Feature | Process | Daemon |
|---------|---------|--------|
| **What is it** | Any running program | A special process that runs in the background |
| **Interaction** | Can be interactive (connected to terminal) | Runs silently — no terminal attached |
| **Lifecycle** | Starts and stops with your session | Usually starts at boot and runs forever |
| **Naming** | Any name | Often ends with `d` (sshd, httpd, dockerd, systemd) |
| **Examples** | `vim`, `python app.py`, `ls` | `sshd`, `nginx`, `dockerd`, `crond` |

```bash
# See all daemons managed by systemd
systemctl list-units --type=service

# Check if a specific daemon is running
systemctl status nginx
systemctl status sshd

# Start/Stop/Restart a daemon
sudo systemctl start nginx
sudo systemctl stop nginx
sudo systemctl restart nginx

# Enable daemon to start on boot
sudo systemctl enable nginx
```

### Q13. What are zombie and orphan processes?

**Answer:**

**Zombie Process:**
- A process that has **finished executing** but its **entry still exists** in the process table.
- Why? Because the parent process hasn't read its exit status yet (using `wait()` system call).
- It's **dead but not cleaned up** — like a ghost 👻.
- Shows as `Z` in `ps aux` output.

**Orphan Process:**
- A process whose **parent has died/exited** before it finished.
- The orphan gets **adopted by PID 1** (`init` or `systemd`), which will eventually clean it up.
- Orphans are generally harmless.

```bash
# Find zombie processes
ps aux | awk '$8=="Z" {print}'

# Or
ps aux | grep -w Z

# You can't kill a zombie directly (it's already dead!)
# You need to kill its parent process, or the parent needs to call wait()
kill -9 <parent_PID>
```

💡 **Interview nugget:** "A few zombies are normal and harmless. But hundreds of zombies indicate a buggy parent process that's not calling wait() to clean up its children."

### Q14. What signals can you send to a process? Explain `kill` command.

**Answer:**

Signals are like **messages** you send to a process to tell it what to do.

```bash
# See all available signals
kill -l
```

**Most important signals:**

| Signal | Number | Meaning | Can Be Caught? |
|--------|--------|---------|---------------|
| `SIGHUP` | 1 | Hangup — often used to reload config | Yes |
| `SIGINT` | 2 | Interrupt — same as Ctrl+C | Yes |
| `SIGQUIT` | 3 | Quit with core dump | Yes |
| `SIGKILL` | 9 | Force kill — IMMEDIATELY terminates | ❌ No |
| `SIGTERM` | 15 | Graceful termination (default) | Yes |
| `SIGSTOP` | 19 | Pause/freeze the process | ❌ No |
| `SIGCONT` | 18 | Resume a stopped process | Yes |

```bash
# Gracefully terminate (SIGTERM — this is the default)
kill 1234
# or explicitly:
kill -15 1234

# Force kill (when graceful doesn't work)
kill -9 1234

# Reload config without restarting (e.g., Nginx)
kill -HUP $(cat /var/run/nginx.pid)
# or
sudo systemctl reload nginx

# Kill all processes by name
killall nginx
pkill nginx

# Kill all processes by a specific user
pkill -u john
```

💡 **Best Practice:** Always try `SIGTERM` (15) first. Give the process a chance to clean up (close files, save data, release ports). Only use `SIGKILL` (9) as a last resort — it doesn't let the process clean up anything.

### Q15. What is the difference between `nohup`, `&`, and `disown`?

**Answer:**

All three help you run processes that survive after you close your terminal — but they work differently:

| Command | What It Does |
|---------|-------------|
| `&` | Runs process in background, but it DIES when terminal closes |
| `nohup` | Runs process immune to hangup signal — survives terminal close |
| `disown` | Removes a running background job from the shell's job table |

```bash
# Run in background (dies when terminal closes)
python3 server.py &

# Run in background AND survive terminal close
nohup python3 server.py &
# Output goes to nohup.out by default

# Run in background with custom log file
nohup python3 server.py > app.log 2>&1 &

# Already running in background? Disown it:
python3 server.py &
disown
# Now it survives terminal close
```

💡 **In real DevOps:** We rarely use `nohup` anymore. Instead, we use **systemd services** to manage long-running processes properly — with auto-restart, logging, and dependency management.

## 📦 Section 4: Package Management

### Q16. What is a package manager? Explain the differences between `apt`, `yum`, and `dnf`.

**Answer:**

A **package manager** is a tool that helps you install, update, and remove software — handling all dependencies automatically.

Think of it like an **app store for Linux**, but via command line.

| Feature | `apt` (Debian/Ubuntu) | `yum` (RHEL/CentOS 7) | `dnf` (RHEL/CentOS 8+, Fedora) |
|---------|----------------------|----------------------|-------------------------------|
| **Distro** | Debian, Ubuntu | RHEL 7, CentOS 7 | RHEL 8+, CentOS 8+, Fedora |
| **Package format** | `.deb` | `.rpm` | `.rpm` |
| **Install** | `apt install nginx` | `yum install nginx` | `dnf install nginx` |
| **Remove** | `apt remove nginx` | `yum remove nginx` | `dnf remove nginx` |
| **Update all** | `apt update && apt upgrade` | `yum update` | `dnf update` |
| **Search** | `apt search nginx` | `yum search nginx` | `dnf search nginx` |
| **List installed** | `apt list --installed` | `yum list installed` | `dnf list installed` |

```bash
# Ubuntu/Debian — Install a package
sudo apt update             # Refresh package list (always do this first!)
sudo apt install nginx -y   # Install nginx (-y = auto-confirm)

# RHEL/CentOS — Install a package
sudo yum install nginx -y   # CentOS 7
sudo dnf install nginx -y   # CentOS 8+

# Check if a package is installed
dpkg -l | grep nginx        # Debian/Ubuntu
rpm -qa | grep nginx        # RHEL/CentOS
```

---

### Q17. What is the difference between `apt update` and `apt upgrade`?

**Answer:**

This confuses a LOT of people. Here's the simple explanation:

| Command | What It Does |
|---------|-------------|
| `apt update` | Downloads the **latest package list** from repositories. Doesn't install anything! Think of it as "refreshing the menu". |
| `apt upgrade` | Actually **installs newer versions** of your installed packages. This is the real update. |

```bash
# Step 1: Refresh the package list
sudo apt update

# Step 2: Upgrade all installed packages to their latest versions
sudo apt upgrade -y

# Do both in one line (very common in DevOps)
sudo apt update && sudo apt upgrade -y

# Full upgrade (also handles dependency changes, removes old packages)
sudo apt full-upgrade -y
```

💡 **In Dockerfiles**, you often see:
```dockerfile
RUN apt-get update && apt-get install -y nginx && rm -rf /var/lib/apt/lists/*
```
The `rm -rf /var/lib/apt/lists/*` at the end cleans up the package cache to keep the Docker image small.

## 💾 Section 5: Disk & Storage

### Q18. How do you check disk usage in Linux?

**Answer:**

```bash
# Check disk space of all mounted filesystems
df -h
# -h = human readable (shows GB, MB instead of bytes)

# Check disk space of a specific directory
du -sh /var/log
# -s = summary (total), -h = human readable

# Check disk usage of all subdirectories (sorted by size)
du -sh /var/* | sort -rh | head -20

# Check which disk/partition a file is on
df -h /var/log/syslog

# See all block devices (disks and partitions)
lsblk

# Detailed partition info
sudo fdisk -l
```

**Sample `df -h` output:**
```
Filesystem      Size  Used  Avail Use%  Mounted on
/dev/sda1        50G   32G    16G  67%  /
/dev/sdb1       100G   45G    50G  48%  /data
tmpfs            16G     0    16G   0%  /dev/shm
```

💡 **When disk is full — quick debugging steps:**
```bash
# 1. Find what's eating up space
du -sh /* | sort -rh | head -10

# 2. Usually it's logs
du -sh /var/log/*  | sort -rh | head -10

# 3. Find large files (> 100MB)
find / -type f -size +100M -exec ls -lh {} \;

# 4. Check for deleted files still held open
lsof | grep deleted
```

### Q19. What is LVM? Why is it used?

**Answer:**

**LVM (Logical Volume Manager)** is a layer between your physical disks and file systems that gives you **flexibility** to resize, extend, and manage storage without downtime.

**Without LVM:** Your partition is 50GB. It's full. You need to unmount, repartition, and reformat. Painful. 😭

**With LVM:** Your volume is 50GB. It's full. You just extend it online. No downtime. 🎉

```
Physical Layer:    /dev/sda    /dev/sdb    /dev/sdc     (Physical Disks)
                      ↓            ↓           ↓
PV (Physical       /dev/sda1   /dev/sdb1   /dev/sdc1    (Physical Volumes)
Volumes):              ↓            ↓           ↓
                  ┌────────────────────────────────┐
VG (Volume        │         my_volume_group         │    (Volume Group — pool of storage)
Group):           └────────────────────────────────┘
                      ↓            ↓           ↓
LV (Logical       lv_root     lv_data     lv_logs      (Logical Volumes — like virtual partitions)
Volumes):          50GB        100GB        30GB
                      ↓            ↓           ↓
Filesystem:        /           /data       /var/log
```

```bash
# Create a Physical Volume
sudo pvcreate /dev/sdb1

# Create a Volume Group
sudo vgcreate my_vg /dev/sdb1

# Create a Logical Volume (50GB)
sudo lvcreate -L 50G -n my_lv my_vg

# Format it
sudo mkfs.ext4 /dev/my_vg/my_lv

# Mount it
sudo mount /dev/my_vg/my_lv /data

# EXTEND a logical volume (add 20GB more)
sudo lvextend -L +20G /dev/my_vg/my_lv
sudo resize2fs /dev/my_vg/my_lv    # For ext4
# or
sudo xfs_growfs /data               # For XFS
```

## 🌐 Section 6: Networking Commands

### Q20. How do you check network configuration and troubleshoot connectivity?

**Answer:**

```bash
# Check your IP address
ip addr show
# or older (deprecated but still works):
ifconfig

# Check default gateway
ip route show
# Look for the line: default via 10.0.0.1

# Check DNS configuration
cat /etc/resolv.conf

# Test connectivity to a host
ping google.com
ping -c 4 google.com    # Send only 4 packets

# Trace the route to a host (see every hop)
traceroute google.com
# or
mtr google.com    # Better — real-time, combines ping + traceroute

# Check if a specific port is open
nc -zv 192.168.1.10 22     # Check SSH port
nc -zv google.com 443      # Check HTTPS port

# Check which ports are listening on your machine
ss -tlnp
# t=TCP, l=listening, n=numeric, p=process name

# Older alternative (still common in interviews)
netstat -tlnp

# DNS lookup
nslookup google.com
dig google.com
dig +short google.com     # Just the IP
```

---

### Q21. What is the difference between `ss` and `netstat`?

**Answer:**

| Feature | `ss` | `netstat` |
|---------|------|-----------|
| **Speed** | Very fast (reads from kernel directly) | Slower (reads from /proc) |
| **Status** | Modern — actively maintained | Deprecated (but still widely used) |
| **Package** | Built into `iproute2` | Requires `net-tools` package |
| **Output** | More detailed | Less detailed |

```bash
# Show all TCP listening ports with process names
ss -tlnp
netstat -tlnp    # Same thing, older way

# Show all established connections
ss -t state established

# Show connections to a specific port
ss -t dst :443

# Count connections per state
ss -s
```

💡 **In interviews**, say: "I prefer `ss` over `netstat` because it's faster, more modern, and `netstat` is deprecated. But I'm comfortable with both."

### Q22. How do you check which process is using a specific port?

**Answer:**

This is a VERY common question and a VERY common real-world task.

```bash
# Method 1: Using ss
sudo ss -tlnp | grep :80
# Output: LISTEN 0 511 0.0.0.0:80 ... users:(("nginx",pid=1234,fd=6))

# Method 2: Using lsof
sudo lsof -i :80
# Output shows PID, user, and command using port 80

# Method 3: Using fuser
sudo fuser 80/tcp
```

### Q26. How does SSH work? Explain SSH key-based authentication.

**Answer:**

**SSH (Secure Shell)** lets you securely connect to a remote machine over an encrypted connection.

**Two types of authentication:**

| Type | How It Works | Security |
|------|-------------|----------|
| **Password-based** | You type your password each time | ❌ Less secure (can be brute-forced) |
| **Key-based** | You use a public-private key pair | ✅ Much more secure |

**How SSH key-based authentication works (step by step):**

```
1. You generate a key pair on YOUR machine:
   - Private key (id_rsa) → stays on YOUR machine. NEVER share this!
   - Public key (id_rsa.pub) → you copy this to the remote server

2. You copy your public key to the server:
   → It gets added to ~/.ssh/authorized_keys on the server

3. When you connect:
   Client says: "Hey, I want to log in."
   Server says: "Prove you have the private key for this public key."
   Client: *does cryptographic math using private key*
   Server: *verifies using the stored public key*
   Server: "You're in!" ✅
```

```bash
# Step 1: Generate SSH key pair
ssh-keygen -t ed25519 -C "ritesh@company.com"
# ed25519 is modern and secure. RSA also works: ssh-keygen -t rsa -b 4096

# Step 2: Copy public key to remote server
ssh-copy-id user@remote-server
# This adds your public key to ~/.ssh/authorized_keys on the server

# Step 3: Connect (no password needed!)
ssh user@remote-server

# Connect with a specific key file
ssh -i ~/.ssh/my-custom-key user@remote-server

# SSH with a specific port
ssh -p 2222 user@remote-server
```

**Important file permissions for SSH:**
```bash
chmod 700 ~/.ssh              # .ssh directory
chmod 600 ~/.ssh/id_rsa       # Private key (MUST be 600 or SSH refuses!)
chmod 644 ~/.ssh/id_rsa.pub   # Public key
chmod 600 ~/.ssh/authorized_keys
```

### Q27. What is SSH tunneling/port forwarding?

**Answer:**

SSH tunneling creates an **encrypted tunnel** between your machine and a remote machine, letting you access services as if they were local.

**Three types:**

**1. Local Port Forwarding** — Access a remote service through a local port:
```bash
# Access remote server's MySQL (port 3306) via your localhost:3307
ssh -L 3307:localhost:3306 user@remote-server

# Now connect to localhost:3307 and it reaches remote MySQL
mysql -h 127.0.0.1 -P 3307 -u dbuser -p
```

**2. Remote Port Forwarding** — Expose your local service to the remote server:
```bash
# Let the remote server access your local app on port 3000
ssh -R 8080:localhost:3000 user@remote-server

# Now remote-server:8080 reaches your local port 3000
```

**3. Dynamic Port Forwarding (SOCKS Proxy):**
```bash
# Create a SOCKS proxy through the remote server
ssh -D 1080 user@remote-server

# Configure your browser to use localhost:1080 as a SOCKS proxy
# All traffic goes through the encrypted SSH tunnel
```

💡 **Real use case:** You need to access a database in a private subnet. You can't reach it directly. But you CAN SSH into a bastion/jump host that's in the same VPC. Use local port forwarding to reach the database through the bastion.

### Q29. How do you create a custom systemd service?

**Answer:**

Let's say you have a Python app you want to run as a service:

```bash
# Step 1: Create a service file
sudo vim /etc/systemd/system/myapp.service
```

```ini
[Unit]
Description=My Python Application
After=network.target
Wants=network-online.target

[Service]
Type=simple
User=appuser
Group=appuser
WorkingDirectory=/opt/myapp
ExecStart=/usr/bin/python3 /opt/myapp/app.py
Restart=always
RestartSec=5
Environment=ENV=production
Environment=PORT=8080
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target
```

```bash
# Step 2: Reload systemd to pick up the new service
sudo systemctl daemon-reload

# Step 3: Start the service
sudo systemctl start myapp

# Step 4: Enable it to start on boot
sudo systemctl enable myapp

# Step 5: Check status
sudo systemctl status myapp

# Step 6: View logs
journalctl -u myapp -f
```

**Key fields explained:**

| Field | Meaning |
|-------|---------|
| `After=network.target` | Start this service AFTER network is ready |
| `Type=simple` | The process started by ExecStart is the main process |
| `Restart=always` | Restart the service if it crashes |
| `RestartSec=5` | Wait 5 seconds before restarting |
| `WantedBy=multi-user.target` | Start in normal multi-user mode (like runlevel 3) |

### Q30. What is the difference between `cron` and `systemd timers`?

**Answer:**

Both are used to schedule tasks, but systemd timers are more powerful:

| Feature | Cron | Systemd Timers |
|---------|------|---------------|
| **Setup** | Simple (`crontab -e`) | Requires two files (.timer + .service) |
| **Logging** | Must configure manually | Built-in (`journalctl`) |
| **Missed runs** | Missed = skipped forever | Can run missed tasks on next boot (`Persistent=true`) |
| **Dependencies** | None | Can depend on other services |
| **Randomized delay** | No | Yes (`RandomizedDelaySec`) |
| **Accuracy** | 1 minute minimum | Microsecond precision |

**Cron example:**
```bash
# Edit crontab

# Run a backup every day at 2:00 AM
0 2 * * * /opt/scripts/backup.sh >> /var/log/backup.log 2>&1

# Cron format:
# ┌─────── minute (0-59)
# │ ┌───── hour (0-23)
# │ │ ┌─── day of month (1-31)
# │ │ │ ┌─ month (1-12)
# │ │ │ │ ┌ day of week (0-7, 0 and 7 = Sunday)
# * * * * * command

# Every 5 minutes
*/5 * * * * /opt/scripts/health-check.sh

# Every Monday at 9 AM
0 9 * * 1 /opt/scripts/weekly-report.sh

# List all cron jobs
crontab -l
```

**Systemd timer example:**
```bash
# /etc/systemd/system/backup.timer
[Unit]
Description=Daily Backup Timer

[Timer]
OnCalendar=*-*-* 02:00:00
Persistent=true

[Install]
WantedBy=timers.target
```

```bash
# Enable and start the timer

# List all timers
systemctl list-timers
```

### Q32. Write a script to monitor disk usage and send an alert if it exceeds 80%.

**Answer:**

```bash
#!/bin/bash

THRESHOLD=80
ALERT_EMAIL="admin@company.com"

# Check each mounted filesystem
df -h --output=pcent,target | tail -n +2 | while read -r usage mount; do
    # Remove the % sign to get the number
    usage_num=${usage%\%}
```

**Explanation line by line:**

- `tail -n +2` → Skip the header line
- `${usage%\%}` → Remove the `%` character from the string (bash string manipulation)
- `-gt` → Greater than (numeric comparison in bash)

### Q33. What is the difference between `$()` and backticks `` ` ` `` in bash?

**Answer:**

Both are used for **command substitution** — running a command and capturing its output.

```bash
# Using backticks (old way)
today=`date +%Y-%m-%d`

# Using $() (modern way — recommended)
today=$(date +%Y-%m-%d)

# Both do the same thing — store today's date in a variable
echo "Today is: $today"
```

**Why `$()` is better:**

| Feature | Backticks `` ` ` `` | `$()` |
|---------|---------------------|-------|
| **Nesting** | Very ugly and confusing | Clean and easy |
| **Readability** | Hard to read, especially with quotes | Clear and obvious |
| **Can be confused with** | Single quotes `'` | Nothing |

```bash
# Nesting with backticks — nightmare 😱
result=`echo \`date\``

# Nesting with $() — clean and clear 😊
result=$(echo $(date))

# Real-world example with nesting:
latest_log=$(ls -t /var/log/$(hostname)/*.log | head -1)
```

💡 **Rule:** Always use `$()`. Forget backticks exist.

### Q34. What is the difference between `"double quotes"` and `'single quotes'` in bash?

**Answer:**

| Feature | `"Double Quotes"` | `'Single Quotes'` |
|---------|-------------------|-------------------|
| **Variable expansion** | ✅ Yes — `$VAR` gets replaced | ❌ No — `$VAR` stays as literal text |
| **Command substitution** | ✅ Yes — `$(cmd)` runs | ❌ No — `$(cmd)` stays as literal text |
| **Escape sequences** | ✅ Yes — `\n`, `\t` work | ❌ No — `\n` stays as `\n` |
| **Use when** | You want variables to be replaced | You want everything treated literally |

```bash
name="Ritesh"

# Double quotes — variable is expanded
echo "Hello, $name"      # Output: Hello, Ritesh

# Single quotes — everything is literal
echo 'Hello, $name'      # Output: Hello, $name

# This matters a LOT in DevOps:
echo "The hostname is $(hostname)"    # Output: The hostname is web-server-01
echo 'The hostname is $(hostname)'    # Output: The hostname is $(hostname)
```

💡 **Practical rule:** Use double quotes when you need variable substitution. Use single quotes when you want to preserve the exact text (like in `awk` or `grep` patterns).

### Q36. What happens when you type a URL in a browser? (Linux/Networking perspective)

**Answer:**

This is asked to test your understanding of networking + Linux together:

```
1. DNS Resolution
   └── Browser checks its cache → OS cache → /etc/hosts → /etc/resolv.conf DNS server
   └── DNS server returns the IP address (e.g., 93.184.216.34)

2. TCP Connection (Three-Way Handshake)
   └── Client sends SYN → Server replies SYN-ACK → Client sends ACK
   └── Connection established on port 443 (HTTPS)

3. TLS Handshake (if HTTPS)
   └── Client and server agree on encryption method
   └── Server sends its SSL certificate
   └── Both sides generate session keys for encrypted communication

4. HTTP Request
   └── Client sends: GET / HTTP/1.1, Host: example.com, headers...

5. Server Processing
   └── Web server (Nginx/Apache) receives the request
   └── Routes to the right application (reverse proxy)
   └── Application processes request, queries database if needed

6. HTTP Response
   └── Server sends back: HTTP/1.1 200 OK, HTML content, headers...

7. Browser Renders
   └── Browser parses HTML, loads CSS/JS, renders the page

8. TCP Connection Close (Four-Way Handshake)
   └── FIN → ACK → FIN → ACK (or kept alive for HTTP keep-alive)
```

### Q37. What is the `/proc` filesystem?

**Answer:**

`/proc` is a **virtual filesystem** — it doesn't exist on disk. It's created by the kernel in memory and gives you a window into the running system.

### Q38. What are Linux namespaces and cgroups? (Container foundation)

**Answer:**

This is what makes Docker/containers work under the hood!

**Namespaces** = **Isolation** (what you can SEE)
**Cgroups** = **Resource Limits** (what you can USE)

**Namespaces — Each container gets its own isolated view of:**

| Namespace | What It Isolates | Example |
|-----------|-----------------|---------|
| **PID** | Process IDs | Container sees PID 1 as its main process |
| **NET** | Network stack | Container has its own IP, ports, interfaces |
| **MNT** | Mount points / filesystem | Container has its own `/`, `/etc`, etc. |
| **UTS** | Hostname | Container has its own hostname |
| **IPC** | Inter-process communication | Shared memory, semaphores are isolated |
| **USER** | User/Group IDs | Root inside container ≠ root on host |

**Cgroups — Limit resource usage:**

```bash
# See cgroup info for a process
cat /proc/<PID>/cgroup

# Docker uses cgroups to limit containers:
docker run --memory=512m --cpus=1.5 nginx
# This sets cgroup limits:
# - Max 512MB RAM
# - Max 1.5 CPU cores
```

```bash
# See cgroup limits for a running container
# CPU limit
cat /sys/fs/cgroup/cpu/docker/<container-id>/cpu.cfs_quota_us

# Memory limit
cat /sys/fs/cgroup/memory/docker/<container-id>/memory.limit_in_bytes
```

💡 **The simple way to explain it:** "Namespaces make a process THINK it's alone on the machine. Cgroups make sure it BEHAVES and doesn't hog all the resources."

### Q39. What is `iptables`? How do you configure a basic firewall?

**Answer:**

`iptables` is the traditional Linux **firewall** that controls network traffic based on rules.

**Concept:**
- Traffic flows through **chains**: INPUT (incoming), OUTPUT (outgoing), FORWARD (routed)
- Each chain has **rules** that say: ACCEPT, DROP, or REJECT

```bash
# See current rules
sudo iptables -L -n -v

# Allow incoming SSH (port 22)
sudo iptables -A INPUT -p tcp --dport 22 -j ACCEPT

# Allow incoming HTTP (port 80) and HTTPS (port 443)
sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 443 -j ACCEPT

# Allow established connections (responses to your outgoing requests)
sudo iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# Allow loopback (localhost)
sudo iptables -A INPUT -i lo -j ACCEPT

# Drop everything else
sudo iptables -A INPUT -j DROP

# Save the rules (Ubuntu)
sudo iptables-save > /etc/iptables/rules.v4

# Delete a specific rule
sudo iptables -D INPUT -p tcp --dport 8080 -j ACCEPT

# Flush (delete) ALL rules
sudo iptables -F
```

💡 **Modern alternatives:**
- `ufw` (Ubuntu) — simpler frontend for iptables: `sudo ufw allow 22/tcp`
- `firewalld` (RHEL/CentOS) — zone-based firewall: `sudo firewall-cmd --add-port=22/tcp --permanent`
- `nftables` — the successor to iptables (newer kernels)

```bash
# -a = archive (preserves permissions, ownership, timestamps)
# -v = verbose
# -z = compress during transfer

# rsync — Sync entire directories
rsync -avz --delete /local/folder/ user@remote:/remote/folder/
# --delete = delete files on remote that don't exist locally (mirror)

# rsync — With progress bar
rsync -avz --progress /data/ user@remote:/backup/
```

💡 **Why `rsync` is better than `scp`:**
- Only transfers CHANGED parts of files (incremental)
- Can resume interrupted transfers
- Can compress data during transfer
- Can delete extraneous files (mirror mode)
- Can exclude files/patterns

### Q44. What is swap space? How do you configure it?

**Answer:**

**Swap** is disk space used as "emergency RAM." When your RAM is full, Linux moves less-used data from RAM to swap to free up memory.

```bash
# Check current swap
free -h
swapon --show

# Create a swap file (2GB)
sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile       # Only root should access it
sudo mkswap /swapfile          # Format as swap
sudo swapon /swapfile          # Activate it

# Make it permanent (survives reboot)
echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab

# Check swappiness (how eagerly Linux uses swap)
cat /proc/sys/vm/swappiness
# Default: 60 (range: 0-100)
# Lower = less swap usage, Higher = more swap usage

# For servers/databases, set lower:
sudo sysctl vm.swappiness=10
# Make permanent:
echo "vm.swappiness=10" | sudo tee -a /etc/sysctl.conf
```

💡 **Kubernetes note:** In Kubernetes, swap should be DISABLED on worker nodes (it's required by kubelet):
```bash
sudo swapoff -a
# Remove swap entry from /etc/fstab to make permanent
```

### Q45. How do you check and manage boot process in Linux?

**Answer:**

```bash
# See boot messages
dmesg | less
journalctl -b       # Current boot logs
journalctl -b -1    # Previous boot logs

# See how long boot took
systemd-analyze
systemd-analyze blame    # Shows which services took the longest

# See default boot target (runlevel)
systemctl get-default
# graphical.target = GUI (runlevel 5)
# multi-user.target = CLI only (runlevel 3)

# Change default boot target
sudo systemctl set-default multi-user.target    # CLI only (servers)

# See boot process:
# BIOS/UEFI → GRUB bootloader → Kernel → initramfs → systemd (PID 1) → Services
```

### Q47. What is the `/etc/hosts` file? When would you use it?

**Answer:**

The `/etc/hosts` file is a **local DNS** file. It maps hostnames to IP addresses **before** DNS is consulted.

```bash
# View the file
cat /etc/hosts

# Example content:
127.0.0.1       localhost
127.0.1.1       my-hostname
192.168.1.100   db-server
192.168.1.101   web-server
10.0.0.50       api.internal.company.com
```

**When you'd use it:**
1. **Testing:** Point a domain to a different server without changing DNS
2. **Local development:** Map `myapp.local` to `127.0.0.1`
3. **Blocking sites:** Map unwanted domains to `0.0.0.0`
4. **No DNS available:** In isolated networks or during DNS outages
5. **Kubernetes pods:** K8s uses `/etc/hosts` for pod hostname resolution via `hostAliases`

```yaml
# Kubernetes — adding entries to pod's /etc/hosts
apiVersion: v1
kind: Pod
spec:
  hostAliases:
    - ip: "10.0.0.50"
      hostnames:
        - "db-server"
        - "database.internal"
```

### Q48. What is `sysctl`? Give some important tunable parameters.

**Answer:**

`sysctl` lets you **read and modify kernel parameters at runtime** — without rebooting.

```bash
# See all kernel parameters
sysctl -a

# Get a specific parameter
sysctl net.ipv4.ip_forward

# Set a parameter (temporary — lost on reboot)
sudo sysctl -w net.ipv4.ip_forward=1

# Set permanently — add to /etc/sysctl.conf or /etc/sysctl.d/*.conf
echo "net.ipv4.ip_forward = 1" | sudo tee -a /etc/sysctl.conf
sudo sysctl -p    # Reload the config
```

**Important parameters DevOps engineers should know:**

| Parameter | What It Does | Common Value |
|-----------|-------------|-------------|
| `net.ipv4.ip_forward` | Enable IP routing (needed for Docker, K8s) | `1` |
| `vm.swappiness` | How aggressively to use swap | `10` (servers) |
| `net.core.somaxconn` | Max queued connections | `65535` (high traffic) |
| `net.ipv4.tcp_max_syn_backlog` | Max SYN queue size | `65535` |
| `fs.file-max` | Max open files system-wide | `2097152` |
| `vm.max_map_count` | Max memory map areas (needed for Elasticsearch) | `262144` |
| `net.bridge.bridge-nf-call-iptables` | Bridge traffic through iptables (needed for K8s) | `1` |

### Q49. What is the difference between TCP Wrappers and Firewall?

**Answer:**

| Feature | TCP Wrappers (`/etc/hosts.allow`, `/etc/hosts.deny`) | Firewall (iptables/firewalld) |
|---------|------------------------------------------------------|------------------------------|
| **Level** | Application level | Network/Kernel level |
| **How it works** | Checks rules for services linked with libwrap | Filters packets at the kernel |
| **Scope** | Only services that support TCP wrappers (sshd, vsftpd) | ALL network traffic |
| **Performance** | Slower (application level check) | Faster (kernel level) |
| **Modern usage** | Mostly deprecated | Standard and recommended |

```bash
# TCP Wrappers example (legacy but still asked)
# /etc/hosts.allow — who IS allowed
sshd: 192.168.1.0/24
sshd: 10.0.0.0/8

# /etc/hosts.deny — who is NOT allowed
sshd: ALL
# Meaning: Allow SSH from 192.168.1.x and 10.x.x.x, deny everyone else
```

💡 **In modern systems**, firewalls (iptables, firewalld, ufw, or cloud security groups) are the standard. TCP wrappers are considered legacy.

### Q50. Explain `top` command output. What do you look for?

**Answer:**

```
top - 14:23:45 up 30 days, 2:15, 3 users, load average: 2.45, 1.89, 1.56
Tasks: 256 total,   1 running, 254 sleeping,   0 stopped,   1 zombie
%Cpu(s): 25.3 us,  5.2 sy,  0.0 ni, 68.5 id,  0.8 wa,  0.0 hi,  0.2 si
MiB Mem :  16384.0 total,   2048.0 free,  12288.0 used,   2048.0 buff/cache
MiB Swap:   4096.0 total,   3584.0 free,    512.0 used.   3584.0 avail Mem

  PID USER      PR  NI    VIRT    RES    SHR S  %CPU  %MEM     TIME+ COMMAND
 1234 www-data  20   0  456789  12345   6789 S  45.2   3.0  123:45.67 nginx
```

**What to look for:**

| Section | What to Check | Warning Signs |
|---------|--------------|---------------|
| **Load Average** | 3 numbers: 1min, 5min, 15min | Higher than CPU count = overloaded |
| **CPU - us** | User space CPU (your apps) | > 80% = app using too much CPU |
| **CPU - sy** | System/kernel CPU | > 20% = too many system calls |
| **CPU - wa** | I/O Wait | > 10% = disk is the bottleneck |
| **CPU - id** | Idle | < 10% = system is fully loaded |
| **Memory** | Used vs Available | Available very low = memory pressure |
| **Swap used** | How much swap is in use | Any swap usage on servers = warning |
| **Zombie** | Zombie process count | > 0 = parent not cleaning up children |

**Useful `top` shortcuts:**
- `P` → Sort by CPU
- `M` → Sort by Memory
- `k` → Kill a process (enter PID)
- `1` → Show per-CPU breakdown
- `c` → Show full command path
