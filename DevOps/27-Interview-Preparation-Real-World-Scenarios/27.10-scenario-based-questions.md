# 27.10 — DevOps Interview — Scenario-Based Questions

> **Why This Matters:** In real DevOps interviews, you won't just be asked "What is Kubernetes?" — you'll be thrown into a **real-world scenario** and asked "How would you solve this?" Scenario-based questions test your **thinking process, troubleshooting skills, and practical experience.** This is where great candidates shine.

---

## 🎯 How to Approach Scenario-Based Questions

Before we dive into the scenarios, here's a framework for answering ANY scenario question:

```
Step 1: CLARIFY — Ask clarifying questions (shows maturity)
Step 2: IDENTIFY — Identify the root cause or the core problem
Step 3: PLAN — Outline your approach (don't jump to solutions)
Step 4: EXECUTE — Describe the steps you would take
Step 5: PREVENT — Explain how you'd prevent this in the future
Step 6: COMMUNICATE — Mention how you'd communicate with stakeholders
```

> 💡 **Pro Tip:** Interviewers care more about your **thought process** than the exact answer. Talk out loud. Show your reasoning.

### Scenario 1: "It's Friday afternoon. Your team just deployed a new release to production, and users are reporting 500 errors. What do you do?"

**Great Answer:**

**Step 1 — Don't Panic. Assess the situation.**

```
First things first:
1. Check monitoring dashboards (Grafana, Datadog, CloudWatch)
   - Are error rates spiking?
   - Which endpoints are failing?
   - When did it start? (Does it correlate with the deployment?)

2. Check application logs
   kubectl logs <pod-name> --tail=100
   # or
   Check ELK/Loki for error patterns

3. Check if it's ALL users or just SOME users
   - If some: might be region-specific or feature-flag related
   - If all: likely a deployment issue
```

**Step 2 — Rollback if necessary.**

```bash
# If the errors clearly started after deployment, ROLLBACK immediately
# Don't debug in production under pressure — rollback first, investigate later

# Kubernetes rollback:
kubectl rollout undo deployment/my-app

# Or if using Helm:
helm rollback my-app <previous-revision>

# Or if using ArgoCD:
# Revert the Git commit and let ArgoCD sync
```

**Step 3 — Communicate.**
- Notify the team in Slack/Teams: "We've identified 500 errors post-deployment. Rolling back now."
- Update the incident channel
- If customer-facing: notify support team

**Step 4 — Investigate the root cause (after rollback).**
```
- Compare the new release with the previous one (git diff)
- Check for database migration issues
- Check for environment variable changes
- Check for dependency updates
- Review the CI/CD pipeline logs
```

**Step 5 — Prevent it next time.**
```
- Add better health checks and readiness probes
- Implement canary deployments (deploy to 5% of users first)
- Add smoke tests in the pipeline
- Implement feature flags for risky features
- Never deploy on Friday! (or at least have a rollback plan)
```

### Scenario 2: "You need to deploy a critical hotfix to production, but the CI/CD pipeline is broken. What do you do?"

**Great Answer:**

```
Priority #1: Get the hotfix out (but safely)

Option A — Fix the pipeline first (if it's a quick fix)
  - Check pipeline logs → identify the failure
  - Common issues: expired credentials, flaky test, runner unavailable
  - If fixable in 5-10 minutes → fix it and deploy normally

Option B — Manual deployment (if pipeline fix takes too long)
  - Build the image locally or on a build server
  - Push to container registry manually
  - Deploy using kubectl or helm manually
  - ⚠️ DOCUMENT everything you did manually

  # Build manually
  docker build -t myapp:hotfix-v1.2.1 .
  docker push registry.example.com/myapp:hotfix-v1.2.1

  # Deploy manually
  kubectl set image deployment/myapp myapp=registry.example.com/myapp:hotfix-v1.2.1

Important considerations:
  - Get peer review on the code (even a quick one)
  - Ensure the same change goes through the pipeline later
  - Create an incident ticket for the broken pipeline
  - Never skip testing entirely — at minimum run unit tests locally

After the hotfix:
  - Fix the CI/CD pipeline
  - Retroactively push the changes through the pipeline
  - Write a post-mortem: why was the pipeline broken?
  - Add pipeline health monitoring and alerts
```

**Great Answer:**

```

│   ├── Start with rolling updates

├── Set up automatic rollback on failure

├── Blameless post-mortems
├── On-call rotation
├── "You build it, you run it" ownership

- Deployment Frequency: monthly → daily → multiple per day
- Lead Time for Changes: weeks → hours

- Mean Time to Recovery: < 1 hour
```

### Scenario 4: "A Kubernetes pod is stuck in CrashLoopBackOff. How do you troubleshoot it?"

**Great Answer:**

```bash
# Step 1 — Check the pod status and events
kubectl describe pod <pod-name> -n <namespace>
# Look at:
#   - Events section (at the bottom) → tells you WHY it's crashing
#   - Container status → exit codes, restart count
#   - Resource limits → is it getting OOMKilled?

# Step 2 — Check the logs
kubectl logs <pod-name> -n <namespace>
# If it already crashed, check previous container logs:
kubectl logs <pod-name> -n <namespace> --previous

# Step 3 — Common causes and fixes:

# Cause 1: Application error (bad code, missing config)
# → Fix: Check logs for stack traces, fix the code/config

# Cause 2: Missing environment variables or ConfigMaps/Secrets
kubectl get configmap <name> -n <namespace>
kubectl get secret <name> -n <namespace>
# → Fix: Create/update the missing ConfigMap or Secret

# Cause 3: OOMKilled (Out of Memory)
kubectl describe pod <pod-name> | grep -i oom
# → Fix: Increase memory limits in the deployment spec
# resources:
#   limits:
#     memory: "512Mi"  # increase this

# Cause 4: Liveness probe failing
# → Fix: Adjust probe timing (initialDelaySeconds, periodSeconds)
# livenessProbe:
#   httpGet:
#     path: /healthz
#     port: 8080
#   initialDelaySeconds: 30   # give app time to start
#   periodSeconds: 10

# Cause 5: Image pull error (wrong image tag)
kubectl describe pod <pod-name> | grep -i "image"
# → Fix: Verify the image exists and tag is correct

# Cause 6: Database/dependency not reachable
# → Fix: Check network policies, DNS, service endpoints

# Step 4 — Quick debug with an ephemeral container (K8s 1.23+)
kubectl debug -it <pod-name> --image=busybox --target=<container-name>

# Step 5 — If nothing works, run the container locally
docker run -it <image-name> /bin/sh
# Manually test the startup process
```

### Scenario 5: "Your website is extremely slow. Users are complaining. Walk me through your troubleshooting process."

**Great Answer:**

```
I'd approach this systematically — from the outside in:

Layer 1 — Is it really slow? (Confirm the problem)
├── Check from my own browser (clear cache, different network)
├── Use external tools: Pingdom, GTmetrix, curl
│   curl -w "@curl-format.txt" -o /dev/null -s https://myapp.com
│   # This shows DNS lookup time, TLS handshake time, TTFB, etc.
├── Check status page / uptime monitor
└── Check: Is it slow for ALL users or specific regions?

Layer 2 — Check the infrastructure metrics
├── CPU usage → Are servers at 90%+ CPU?
│   kubectl top pods
│   kubectl top nodes
├── Memory usage → Is anything swapping or OOMKilled?
├── Disk I/O → Is disk throughput saturated?
├── Network → Is there packet loss or high latency?
└── Are there enough pods/instances? Check HPA status.
    kubectl get hpa

Layer 3 — Check the application
├── Application logs → Any errors? Slow queries? Timeouts?
├── APM traces (Datadog, New Relic, Jaeger)
│   → Find the slowest endpoints
│   → Identify bottlenecks (DB queries, external API calls)
├── Thread/connection pool exhaustion?
└── Memory leaks? Garbage collection pauses?

Layer 4 — Check the database
├── Slow query log → Any queries taking > 1 second?
├── Connection pool → Are connections maxed out?
├── Database CPU/memory → Is it overloaded?
├── Missing indexes → EXPLAIN ANALYZE on slow queries
├── Replication lag (if using read replicas)
└── Lock contention → Are queries blocking each other?

Layer 5 — Check external dependencies
├── Third-party APIs → Are they responding slowly?
├── CDN → Is it caching properly?
├── DNS → Is resolution slow?
└── Load balancer → Is it distributing traffic correctly?

Quick wins to try:
1. Scale up pods/instances (horizontal scaling)
2. Add caching (Redis/Memcached) for hot queries
3. Enable CDN for static assets
4. Optimize slow database queries (add indexes)
5. Enable gzip/brotli compression
```

### Scenario 6: "Your application suddenly starts consuming 100% CPU on all nodes. What's your approach?"

**Great Answer:**

```bash
# Step 1 — Identify WHICH pods/processes are consuming CPU

# On Kubernetes:
kubectl top pods --all-namespaces --sort-by=cpu
kubectl top nodes

# On a Linux server:
top -c              # or htop
ps aux --sort=-%cpu | head -20

# Step 2 — Is it the application or something else?
# Look for:
#   - Is it your app pods?
#   - Is it system processes (kubelet, docker)?
#   - Is it a DaemonSet (logging agent, monitoring)?
#   - Did something get deployed recently?

kubectl get events --sort-by='.lastTimestamp' | tail -20

# Step 3 — Common causes:

# Cause 1: Infinite loop or busy wait in code
# → Check recent code changes (git log)
# → Profile the application (Java: jstack, Python: cProfile, Node: --prof)

# Cause 2: Spike in traffic (DDoS or legitimate)
# → Check request rates in load balancer / ingress
# → If DDoS: enable WAF, rate limiting
# → If legitimate: scale out (add more pods/nodes)
kubectl scale deployment myapp --replicas=10

# Cause 3: Resource limits not set
# → Pods without limits can consume ALL available CPU
# → Fix: Add resource limits
# resources:
#   requests:
#     cpu: "250m"
#   limits:
#     cpu: "500m"

# Cause 4: Memory leak causing excessive GC
# → Check memory usage alongside CPU
# → JVM: excessive garbage collection burns CPU

# Cause 5: Crypto mining malware 😱
# → Check for unknown processes
# → Check for unknown containers
kubectl get pods --all-namespaces | grep -v "Running\|Completed"

# Step 4 — Immediate mitigation
# Option A: Scale horizontally
kubectl scale deployment myapp --replicas=20

# Option B: Apply resource limits (if missing)
kubectl patch deployment myapp -p '{"spec":{"template":{"spec":{"containers":[{"name":"myapp","resources":{"limits":{"cpu":"500m"}}}]}}}}'

# Option C: Rollback if it started after a deployment
kubectl rollout undo deployment/myapp

# Step 5 — Long-term fixes
# - Set resource requests and limits for ALL pods
# - Implement LimitRange and ResourceQuota per namespace
# - Set up CPU usage alerts (Prometheus + AlertManager)
# - Profile the application regularly
# - Add auto-scaling (HPA)
```

```
│   ├── Build → Test → Scan → Push Image → Deploy to Staging → Deploy to Prod
│   ├── Use Helm charts or Kustomize for deployments
│   └── GitOps with ArgoCD for deployments

Phase 4 — Operationalize
├── On-call rotation per service team
├── Runbooks for each service
├── Chaos engineering (test failure scenarios)
├── Cost monitoring per service (Kubecost)
└── Regular architecture reviews
```

### Scenario 8: "The CEO says the application must have 99.99% uptime. How do you achieve this?"

**Great Answer:**

```
99.99% uptime = 52.6 minutes of downtime per YEAR
That's only 4.38 minutes per month. This is VERY aggressive.

Here's my multi-layer approach:

Layer 1 — Application Level
├── Run multiple replicas (minimum 3 pods)
├── Implement health checks (liveness + readiness probes)
├── Graceful shutdown handling
├── Circuit breaker pattern (Hystrix, Resilience4j)
├── Retry logic with exponential backoff
├── Horizontal Pod Autoscaler (HPA) for auto-scaling
└── Pod Disruption Budgets (PDB) — never all pods down at once
    apiVersion: policy/v1
    kind: PodDisruptionBudget
    metadata:
      name: myapp-pdb
    spec:
      minAvailable: 2    # Always keep at least 2 pods running
      selector:
        matchLabels:
          app: myapp

Layer 2 — Infrastructure Level
├── Multi-AZ deployment (spread across availability zones)
│   topologySpreadConstraints:
│   - maxSkew: 1
│     topologyKey: topology.kubernetes.io/zone
│     whenUnsatisfiable: DoNotSchedule
├── Multi-region deployment (for disaster recovery)
├── Auto-scaling groups for nodes (Cluster Autoscaler)
├── Load balancer with health checks
├── DNS failover (Route 53 health checks + failover routing)
└── Database: Multi-AZ RDS with read replicas

Layer 3 — Deployment Level
├── Blue/Green or Canary deployments (zero-downtime)
├── Rolling updates with maxUnavailable: 0
├── Database migrations must be backward-compatible
├── Feature flags for risky changes
├── Automated rollback on failure
└── Maintenance windows for planned changes

Layer 4 — Monitoring & Response
├── Real-time alerting (PagerDuty, Opsgenie)
├── SLI/SLO tracking
│   SLI: Request success rate
│   SLO: 99.99% of requests succeed
│   Error budget: 0.01% = ~4.38 min/month
├── Synthetic monitoring (simulated user requests)
├── On-call rotation (24/7 coverage)
├── Runbooks for every alert
└── Automated remediation for known issues

Layer 5 — Disaster Recovery
├── Regular backups (database, etcd, config)
├── Test disaster recovery quarterly
├── RTO (Recovery Time Objective): < 15 minutes
├── RPO (Recovery Point Objective): < 5 minutes
├── Document and rehearse failover procedures
└── Chaos engineering (Litmus, Chaos Mesh)

Key Math:
┌──────────┬───────────────────────────┐
│ Uptime   │ Downtime per Year         │
├──────────┼───────────────────────────┤
│ 99%      │ 3.65 days                 │
│ 99.9%    │ 8.76 hours                │
│ 99.99%   │ 52.6 minutes              │
│ 99.999%  │ 5.26 minutes              │
└──────────┴───────────────────────────┘
```

### Scenario 10: "A developer accidentally committed AWS credentials to a public GitHub repository. What do you do?"

**Great Answer:**

```
🚨 THIS IS A CRITICAL SECURITY INCIDENT. Act FAST.

Immediate Actions (within minutes):

Step 1 — Rotate the credentials IMMEDIATELY
├── Go to AWS Console → IAM
├── Deactivate the exposed access key
├── Create a new access key
├── Update all services using the old key
└── Do NOT just delete the key yet (you need it for audit)

Step 2 — Check for unauthorized access
├── AWS CloudTrail → Check for unusual API calls
│   aws cloudtrail lookup-events \
│     --lookup-attributes AttributeKey=AccessKeyId,AttributeValue=AKIA... \
│     --start-time 2024-01-01
├── Look for:
│   ├── New IAM users or roles created?
│   ├── New EC2 instances launched? (crypto mining!)
│   ├── S3 buckets accessed or data downloaded?
│   ├── Security groups modified?
│   └── Lambda functions created?
└── If unauthorized access found → escalate to security team

Step 3 — Remove from Git history
├── Just deleting the file in a new commit is NOT enough!
├── The secret is still in git history
├── Use BFG Repo Cleaner or git filter-branch:
    # Using BFG (recommended — much faster)
    bfg --replace-text passwords.txt my-repo.git
    git reflog expire --expire=now --all
    git gc --prune=now --aggressive
    git push --force

├── Or if it's a public repo:
│   ├── Contact GitHub support to purge caches
│   └── Consider the key compromised regardless

Step 4 — Notify stakeholders
├── Security team
├── Engineering manager
├── Compliance team (if applicable — GDPR, SOC2)
└── Document the incident

Prevention (so this never happens again):

1. Pre-commit hooks to detect secrets:
   # Install git-secrets or detect-secrets
   pip install detect-secrets
   detect-secrets scan > .secrets.baseline
   # Add to pre-commit hooks

2. CI/CD scanning:
   # Add secret scanning to pipeline
   - uses: trufflesecurity/trufflehog@main  # GitHub Actions
   # Or use GitLeaks, detect-secrets

3. Use a secrets manager:
   ├── HashiCorp Vault
   ├── AWS Secrets Manager
   ├── Environment variables in CI/CD (never in code)
   └── Kubernetes Secrets / External Secrets Operator

4. GitHub settings:
   ├── Enable GitHub Secret Scanning (free for public repos)
   ├── Enable GitHub Push Protection (blocks pushes with secrets)
   └── Set up branch protection rules

5. Education:
   ├── Train developers on secrets management
   ├── Create a "secrets handling" runbook
   └── Add to onboarding checklist

.gitignore should include:
   .env
   *.pem
   *.key
   credentials.json
   *secret*
```

### Scenario 11: "Your container images have critical CVEs (vulnerabilities). How do you handle this?"

**Great Answer:**

```
Step 1 — Assess the severity
├── Run a vulnerability scan:
│   trivy image myapp:latest
│   # or
│   grype myapp:latest
│
├── Categorize findings:
│   ├── CRITICAL — Fix immediately (RCE, privilege escalation)
│   ├── HIGH — Fix within 24-48 hours
│   ├── MEDIUM — Fix within 1-2 weeks
│   └── LOW — Track and fix in next sprint
│
└── Check if the CVE is actually exploitable in your context
    (not all CVEs in an image are actually reachable by your code)

Step 2 — Fix the vulnerabilities

# Fix 1: Update the base image
# BEFORE (vulnerable):
FROM node:16-buster
# AFTER (patched):
FROM node:20-bookworm-slim

# Fix 2: Use minimal/distroless images
# BEFORE:
FROM ubuntu:22.04        # ~77MB, tons of packages
# AFTER:
FROM gcr.io/distroless/nodejs:20   # ~30MB, minimal attack surface

# Fix 3: Update application dependencies
npm audit fix
pip install --upgrade <vulnerable-package>

# Fix 4: Multi-stage builds (don't ship build tools)
FROM node:20 AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM node:20-slim AS runtime
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
CMD ["node", "dist/index.js"]

Step 3 — Automate scanning in CI/CD

# In GitHub Actions:
- name: Scan Docker image
  uses: aquasecurity/trivy-action@master
  with:
    image-ref: myapp:${{ github.sha }}
    severity: CRITICAL,HIGH
    exit-code: 1        # Fail the pipeline on CRITICAL/HIGH

# In Jenkins:
stage('Security Scan') {
    steps {
        sh 'trivy image --severity CRITICAL,HIGH --exit-code 1 myapp:latest'
    }
}

Step 4 — Implement admission control

# Block vulnerable images from running in Kubernetes
# Using OPA/Gatekeeper or Kyverno:

apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: block-unscanned-images
spec:
  validationFailureAction: Enforce
  rules:
  - name: require-image-scan
    match:
      resources:
        kinds:
        - Pod
    verifyImages:
    - imageReferences:
      - "*"
      attestors:
      - entries:
        - keys:
            publicKeys: |-
              -----BEGIN PUBLIC KEY-----
              ...
              -----END PUBLIC KEY-----

Step 5 — Ongoing practices
├── Scan images nightly (not just during builds)
├── Set up Dependabot/Renovate for automatic dependency updates
├── Use image signing (cosign) to verify images
├── Maintain a list of approved base images
├── Regular patching schedule
└── Track vulnerability metrics over time
```

### Scenario 12: "Your CI/CD pipeline takes 45 minutes to complete. Developers are frustrated. How do you speed it up?"

**Great Answer:**

```
First, I need to understand WHERE the time is being spent.

Step 1 — Profile the pipeline
├── Break down the 45 minutes:
│   ├── Checkout code: 2 min
│   ├── Install dependencies: 8 min    ← 🐌
│   ├── Build: 5 min
│   ├── Unit tests: 10 min             ← 🐌
│   ├── Integration tests: 12 min      ← 🐌
│   ├── Docker build: 5 min
│   └── Deploy: 3 min

Step 2 — Apply optimizations for each stage

Optimization 1: CACHING
├── Cache dependencies (node_modules, .m2, pip cache)
│   # GitHub Actions example:
│   - uses: actions/cache@v3
│     with:
│       path: ~/.npm
│       key: ${{ runner.os }}-npm-${{ hashFiles('**/package-lock.json') }}
│
├── Cache Docker layers
│   # Use BuildKit cache mounts
│   docker build --cache-from=registry.example.com/myapp:latest .
│
└── Cache test results (only re-run tests for changed code)

Optimization 2: PARALLELIZATION
├── Run independent jobs in parallel:
│   jobs:
│     lint:         # ┐
│       ...         # │ All run in parallel
│     unit-test:    # │
│       ...         # │
│     security-scan:# ┘
│       ...
│     deploy:       # Runs AFTER all above pass
│       needs: [lint, unit-test, security-scan]
│
├── Split tests across multiple runners:
│   strategy:
│     matrix:
│       shard: [1, 2, 3, 4]
│   steps:
│     - run: npm test -- --shard=${{ matrix.shard }}/4
│
└── Run unit tests and integration tests in parallel

Optimization 3: FASTER BUILDS
├── Use multi-stage Docker builds
├── Use smaller base images (alpine, slim, distroless)
├── Order Dockerfile layers properly (least-changing first)
│   COPY package*.json ./          # Changes rarely
│   RUN npm ci                     # Cached if package.json didn't change
│   COPY . .                       # Changes often (but npm ci is cached)
│   RUN npm run build
└── Use Buildx with remote cache

Optimization 4: SMARTER TESTING
├── Run only affected tests (monorepo: use nx, turborepo)
├── Run fast tests first, slow tests later
├── Use test impact analysis
├── Move flaky tests to a separate non-blocking job
└── Consider: do ALL tests need to run on every PR?
    ├── Unit tests: every PR (must pass)
    ├── Integration tests: every PR (must pass)
    ├── E2E tests: maybe only on main branch?
    └── Performance tests: nightly?

Optimization 5: BETTER INFRASTRUCTURE
├── Use faster CI runners (4 CPU → 8 CPU)
├── Use self-hosted runners with local caching
├── Use ARM-based runners (cheaper and faster for some workloads)
└── Pre-bake CI images with common dependencies

Goal:
├── 45 minutes → 15 minutes (with caching + parallelism)
├── 15 minutes → 8 minutes (with smarter testing)
└── 8 minutes → 5 minutes (with faster infrastructure)
```

```yaml
name: CI/CD Pipeline
on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

env:
  IMAGE_NAME: myorg/payment-service

jobs:
  # ============ Stage 1: Code Quality ============
  lint-and-format:

      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'
      - run: npm ci
      - run: npm run lint
      - run: npm run format:check

  # ============ Stage 2: Testing ============
  unit-tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'
      - run: npm ci
      - run: npm test -- --coverage
      - uses: actions/upload-artifact@v4
        with:
          name: coverage
          path: coverage/

  integration-tests:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:16
        env:
          POSTGRES_PASSWORD: testpass
        ports: ['5432:5432']
      redis:
        image: redis:7
        ports: ['6379:6379']
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run test:integration
        env:
          DATABASE_URL: postgresql://postgres:testpass@localhost:5432/test

  # ============ Stage 3: Security Scanning ============
  security-scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: SAST Scan
        uses: returntocorp/semgrep-action@v1
      - name: Dependency Scan
        run: npm audit --audit-level=high

  # ============ Stage 4: Build & Push ============
  build:
    needs: [lint-and-format, unit-tests, integration-tests, security-scan]
    runs-on: ubuntu-latest
    outputs:
      image-tag: ${{ steps.meta.outputs.tags }}
    steps:
      - uses: actions/checkout@v4
      - uses: docker/setup-buildx-action@v3
      - uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
      - id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
          tags: |
            type=sha
            type=ref,event=branch
      - uses: docker/build-push-action@v5
        with:
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

      # Scan the built image
      - name: Scan Docker Image
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:sha-${{ github.sha }}
          severity: CRITICAL,HIGH
          exit-code: 1

  # ============ Stage 5: Deploy to Staging ============
  deploy-staging:
    needs: [build]
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    environment: staging
    steps:
      - uses: actions/checkout@v4
      - name: Deploy to staging
        run: |
          # Update image tag in Helm values or Kustomize
          # Then let ArgoCD sync automatically
          cd k8s/overlays/staging
          kustomize edit set image $IMAGE_NAME=$REGISTRY/$IMAGE_NAME:sha-${{ github.sha }}
          git commit -am "Deploy sha-${{ github.sha }} to staging"
          git push

  # ============ Stage 6: Deploy to Production ============
  deploy-production:
    needs: [deploy-staging]
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    environment: production    # Requires manual approval
    steps:
      - uses: actions/checkout@v4
      - name: Deploy to production (canary)
        run: |
          cd k8s/overlays/production
          kustomize edit set image $IMAGE_NAME=$REGISTRY/$IMAGE_NAME:sha-${{ github.sha }}
          git commit -am "Deploy sha-${{ github.sha }} to production"
          git push
```

### Scenario 19: "How would you implement zero-downtime deployments?"

**Great Answer:**

```
There are several strategies. Let me explain each:

Strategy 1: Rolling Update (Simplest)
├── Replace old pods one at a time
├── Built into Kubernetes by default
│
│   spec:
│     replicas: 5
│     strategy:
│       type: RollingUpdate
│       rollingUpdate:
│         maxSurge: 1          # Add 1 extra pod during update
│         maxUnavailable: 0    # Never let pods go below 5
│
│   Flow:
│   Old: [v1] [v1] [v1] [v1] [v1]
│   Step 1: [v1] [v1] [v1] [v1] [v1] [v2]  ← new pod added
│   Step 2: [v1] [v1] [v1] [v1] [v2]        ← old pod removed
│   ... continues until all are v2
│   Final: [v2] [v2] [v2] [v2] [v2]
│
├── ✅ Simple, built-in
├── ❌ Both versions run simultaneously (need backward compatibility)
└── ❌ Slow rollback

Strategy 2: Blue/Green Deployment
├── Run TWO complete environments
│
│   ┌──────────────┐    ┌──────────────┐
│   │  Blue (v1)   │    │  Green (v2)  │
│   │  LIVE ✅     │    │  Staging     │
│   │  5 pods      │    │  5 pods      │
│   └──────┬───────┘    └──────────────┘
│          │
│    Load Balancer
│
│   After testing Green:
│   ┌──────────────┐    ┌──────────────┐
│   │  Blue (v1)   │    │  Green (v2)  │
│   │  Standby     │    │  LIVE ✅     │
│   └──────────────┘    └──────┬───────┘
│                              │
│                        Load Balancer
│
├── ✅ Instant rollback (switch back to Blue)
├── ✅ Full testing before switching
├── ❌ Requires 2x resources
└── ❌ Database migrations need to work for both versions

# Implementation with Kubernetes:
# Use two deployments + switch the service selector
kubectl patch service myapp -p \
  '{"spec":{"selector":{"version":"green"}}}'

Strategy 3: Canary Deployment (Safest)
├── Deploy new version to a SMALL percentage of users
│
│   ┌──────────────┐  ┌──────────────┐
│   │  v1 (95%)    │  │  v2 (5%)     │
│   │  19 pods     │  │  1 pod       │
│   └──────┬───────┘  └──────┬───────┘
│          └────────┬─────────┘
│             Load Balancer
│
│   If v2 looks good → gradually increase to 100%
│   If v2 has errors → rollback (only 5% affected!)
│
├── ✅ Minimal blast radius
├── ✅ Real user testing
├── ❌ More complex to set up
└── ❌ Need good monitoring to detect issues

# With Argo Rollouts:
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: myapp
spec:
  replicas: 20
  strategy:
    canary:
      steps:
      - setWeight: 5          # 5% traffic
      - pause: {duration: 5m} # Wait 5 minutes
      - setWeight: 25         # 25% traffic
      - pause: {duration: 5m}
      - setWeight: 50         # 50% traffic
      - pause: {duration: 10m}
      - setWeight: 100        # Full rollout

CRITICAL REQUIREMENTS for all strategies:

1. Readiness Probes (don't send traffic until ready)
   readinessProbe:
     httpGet:
       path: /healthz
       port: 8080
     initialDelaySeconds: 10
     periodSeconds: 5

2. Graceful Shutdown (handle in-flight requests)
   lifecycle:
     preStop:
       exec:
         command: ["sh", "-c", "sleep 10"]
   terminationGracePeriodSeconds: 30

3. Backward-Compatible Database Migrations
   # Never drop columns in the same release
   # Step 1: Add new column, migrate data
   # Step 2: Deploy code that uses new column
   # Step 3: Drop old column (next release)

4. Connection Draining
   # Load balancer should finish existing connections
   # before removing the old pod
```

**Great Answer:**

```
Manual rotation at scale is impossible. Here's the automated approach:

Architecture:

┌────────────────┐     ┌────────────────────┐
│ Vault          │────▶│ Vault Agent Injector│
│ (Secrets Store)│     │ (Sidecar in pods)   │
└────────┬───────┘     └────────────────────┘
         │
         │ Dynamic Secrets (auto-rotating)
         │
┌────────▼───────┐     ┌────────────────────┐
│ AWS Secrets    │────▶│ External Secrets    │
│ Manager        │     │ Operator            │
└────────────────┘     └────────┬────────────┘
                                │
                    ┌───────────▼───────────┐
                    │ Kubernetes Secrets     │
                    │ (auto-synced)          │
                    └───────────────────────┘

Approach 1: HashiCorp Vault Dynamic Secrets (Best)
├── Vault generates SHORT-LIVED credentials on demand
├── No manual rotation needed — credentials expire automatically
│
│   # Example: Dynamic database credentials
│   # Vault generates a unique DB username/password for each pod
│   # Credentials auto-expire after 1 hour
│
│   vault write database/roles/myapp-role \
│     db_name=mydb \
│     creation_statements="CREATE ROLE '{{name}}' WITH LOGIN PASSWORD '{{password}}';" \
│     default_ttl="1h" \
│     max_ttl="24h"
│
│   # Pod gets unique credentials via Vault Agent sidecar:
│   annotations:
│     vault.hashicorp.com/agent-inject: "true"
│     vault.hashicorp.com/role: "myapp-role"
│     vault.hashicorp.com/agent-inject-secret-db-creds: "database/creds/myapp-role"
│
├── ✅ Automatic rotation
├── ✅ Unique credentials per pod (easy to revoke)
├── ✅ Audit trail for all secret access
└── ✅ No secrets stored in Kubernetes

Approach 2: External Secrets Operator + Auto-Rotation
├── Store secrets in AWS Secrets Manager / Azure Key Vault
├── External Secrets Operator syncs to Kubernetes Secrets
├── AWS Secrets Manager handles auto-rotation
│
│   # ExternalSecret resource:
│   apiVersion: external-secrets.io/v1beta1
│   kind: ExternalSecret
│   metadata:
│     name: db-credentials
│   spec:
│     refreshInterval: 1h    # Check for new values every hour
│     secretStoreRef:
│       name: aws-secrets-manager
│       kind: ClusterSecretStore
│     target:
│       name: db-credentials  # Kubernetes Secret name
│     data:
│     - secretKey: password
│       remoteRef:
│         key: production/db-password
│
│   # AWS Secrets Manager rotation:
│   aws secretsmanager rotate-secret \
│     --secret-id production/db-password \
│     --rotation-lambda-arn arn:aws:lambda:...:function:rotate-secret \
│     --rotation-rules '{"ScheduleExpression":"rate(30 days)"}'

Approach 3: Application-Level Secret Reload
├── Apps should watch for secret changes and reload without restart
│
│   # Option A: File-based secrets (Vault Agent writes to file)
│   # App watches the file for changes
│   const fs = require('fs');
│   fs.watchFile('/vault/secrets/db-password', () => {
│     const newPassword = fs.readFileSync('/vault/secrets/db-password');
│     reconnectDB(newPassword);
│   });
│
│   # Option B: Kubernetes Reloader
│   # Automatically restarts pods when ConfigMap/Secret changes
│   # Install: https://github.com/stakater/Reloader
│   annotations:
│     reloader.stakater.com/auto: "true"

Key Practices:
├── Never store secrets in Git (use SOPS if you must)
├── Use short-lived dynamic secrets whenever possible
├── Automate rotation — humans forget, machines don't
├── Monitor secret usage and alert on anomalies
├── Test rotation in staging before production
└── Have a "break glass" procedure for emergency rotation
```

## 💡 Quick-Fire Scenario Questions (Short Answers)

Here are some rapid-fire scenario questions you might get:

| Scenario | Quick Answer |
|----------|-------------|
| "Disk space is 95% full on a production server" | Check what's using space (`du -sh /*`), clear old logs, rotate logs, expand volume if possible, set up monitoring alerts |
| "DNS changes aren't propagating" | Check TTL values, flush DNS cache, use `dig` to trace resolution, check if the update was applied to the correct hosted zone |
| "SSL certificate expired in production" | Renew ASAP (Let's Encrypt or vendor), implement cert-manager for auto-renewal, set up expiry monitoring with blackbox exporter |
| "Git merge conflict in CI/CD pipeline" | Don't auto-resolve — fail the pipeline, notify the developer, they should resolve locally and push again |
| "Terraform plan shows 50 resources will be destroyed" | STOP. Don't apply. Check what changed. Likely a state issue or backend change. Review carefully before proceeding |
| "Kubernetes cluster ran out of IP addresses" | Check CIDR range, consider CNI-specific solutions, plan for larger subnets, use VPC CNI secondary CIDRs (EKS) |
| "Build artifacts are 2GB in size" | Use multi-stage Docker builds, remove build tools from final image, use .dockerignore, use slim/alpine base images |
| "Developers need access to production logs but not the servers" | Set up centralized logging (ELK/Loki), give access to Kibana/Grafana with RBAC, use tools like Teleport for audited access |

## 🏆 Tips for Acing Scenario-Based Questions

```
1. THINK OUT LOUD
   - Interviewers want to see your thought process
   - Silence is the worst thing in a scenario question

2. ASK CLARIFYING QUESTIONS
   - "What's the current infrastructure?"
   - "How many users are affected?"
   - "Do we have monitoring in place?"
   - This shows maturity and real-world experience

3. PRIORITIZE
   - Always address the IMMEDIATE impact first
   - Then investigate root cause
   - Then prevent recurrence

4. MENTION COMMUNICATION
   - "I'd notify the team..."
   - "I'd update the status page..."
   - "I'd write a post-mortem..."
   - This shows you understand DevOps is about PEOPLE, not just tech

5. DON'T FORGET PREVENTION
   - After solving the problem, always explain how to prevent it
   - This shows you think about long-term improvements

6. USE REAL TOOL NAMES
   - Don't say "I'd use some monitoring tool"
   - Say "I'd check Prometheus/Grafana for metrics and Loki for logs"
   - This shows hands-on experience

7. BE HONEST
   - If you don't know something, say:
   - "I haven't dealt with this exact scenario, but here's how I'd approach it..."
   - Honesty + good reasoning > bullshitting
```
