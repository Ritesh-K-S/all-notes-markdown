## 🎯 How to Approach System Design Questions

Before we dive into the designs, here's a framework for ANY system design question:

```
Step 1: REQUIREMENTS — Clarify functional and non-functional requirements
Step 2: SCOPE — Define what's in scope and what's not
Step 3: HIGH-LEVEL DESIGN — Draw the big picture (boxes and arrows)
Step 4: DEEP DIVE — Go deeper into critical components
Step 5: TRADE-OFFS — Discuss alternatives and justify your choices
Step 6: SCALABILITY — How does it scale? What are the limits?
Step 7: MONITORING — How do you know it's working?
```

> 💡 **Pro Tip:** Always start by asking questions. "What's the team size?", "What's the expected traffic?", "What cloud provider?" — This shows you think before you build.

## 📋 System Design 1: Design a CI/CD Pipeline

### The Question:
> "Design a complete CI/CD pipeline for a company with 10 engineering teams, 50+ microservices, deploying to Kubernetes on AWS. The pipeline should support multiple environments (dev, staging, production)."

---

### Step 1 — Clarify Requirements

Before designing, I'd ask:
```
Functional Requirements:
- What languages/frameworks do teams use? (Node.js, Go, Java, Python)
- What's the Git workflow? (trunk-based, feature branches, Git flow?)
- How often should teams deploy? (multiple times per day)
- Do we need manual approval for production?
- Any compliance requirements? (SOC2, HIPAA)

Non-Functional Requirements:
- Pipeline should complete in < 15 minutes
- Must support 50+ concurrent pipelines
- Must be secure (no secrets in code, image scanning)
- Must be self-service (teams can onboard without DevOps help)
- Must be auditable (who deployed what, when)
```

### Step 3 — Detailed Pipeline Stages

```yaml
# Shared CI workflow: .github/workflows/ci.yml (reusable)
# Every microservice calls this shared workflow

name: Microservice CI Pipeline

on:
  workflow_call:
    inputs:
      service-name:
        required: true
        type: string
      language:
        required: true
        type: string    # node, go, java, python

jobs:
  # ═══════════════════════════════════════════
  # STAGE 1: Code Quality & Linting
  # ═══════════════════════════════════════════
  code-quality:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Lint
        run: make lint
      - name: Format Check
        run: make format-check
      - name: SonarQube Analysis
        uses: sonarsource/sonarqube-scan-action@master
        env:
          SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}

  # ═══════════════════════════════════════════
  # STAGE 2: Unit Tests (parallel with linting)
  # ═══════════════════════════════════════════
  unit-tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run Unit Tests
        run: make test-unit
      - name: Upload Coverage
        uses: codecov/codecov-action@v4

  # ═══════════════════════════════════════════
  # STAGE 3: Integration Tests
  # ═══════════════════════════════════════════
  integration-tests:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:16
        env:
          POSTGRES_PASSWORD: test
    steps:
      - uses: actions/checkout@v4
      - name: Run Integration Tests
        run: make test-integration

  # ═══════════════════════════════════════════
  # STAGE 4: Security Scanning
  # ═══════════════════════════════════════════
  security:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: SAST (Static Analysis)
        uses: returntocorp/semgrep-action@v1
      - name: Dependency Scan (SCA)
        run: make security-scan
      - name: License Check
        run: make license-check

  # ═══════════════════════════════════════════
  # STAGE 5: Build Docker Image
  # ═══════════════════════════════════════════
  build:
    needs: [code-quality, unit-tests, integration-tests, security]
    runs-on: ubuntu-latest
    outputs:
      image-tag: ${{ steps.build.outputs.tag }}
    steps:
      - uses: actions/checkout@v4
      - uses: docker/setup-buildx-action@v3
      - uses: docker/login-action@v3
        with:
          registry: ${{ secrets.ECR_REGISTRY }}
      - id: build
        uses: docker/build-push-action@v5
        with:
          push: true
          tags: |
            ${{ secrets.ECR_REGISTRY }}/${{ inputs.service-name }}:${{ github.sha }}
            ${{ secrets.ECR_REGISTRY }}/${{ inputs.service-name }}:latest
          cache-from: type=gha
          cache-to: type=gha,mode=max

  # ═══════════════════════════════════════════
  # STAGE 6: Container Image Scanning
  # ═══════════════════════════════════════════
  image-scan:
    needs: [build]
    runs-on: ubuntu-latest
    steps:
      - name: Trivy Scan
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: ${{ secrets.ECR_REGISTRY }}/${{ inputs.service-name }}:${{ github.sha }}
          severity: CRITICAL,HIGH
          exit-code: 1
      - name: Sign Image (Cosign)
        run: cosign sign ${{ secrets.ECR_REGISTRY }}/${{ inputs.service-name }}:${{ github.sha }}

  # ═══════════════════════════════════════════
  # STAGE 7: Deploy to Dev (automatic)
  # ═══════════════════════════════════════════
  deploy-dev:
    needs: [image-scan]
    runs-on: ubuntu-latest
    steps:
      - name: Update GitOps repo (dev overlay)
        run: |
          git clone https://github.com/myorg/gitops-repo.git
          cd gitops-repo/services/${{ inputs.service-name }}/overlays/dev
          kustomize edit set image ${{ inputs.service-name }}=${{ secrets.ECR_REGISTRY }}/${{ inputs.service-name }}:${{ github.sha }}
          git commit -am "Deploy ${{ inputs.service-name }}:${{ github.sha }} to dev"
          git push
      # ArgoCD automatically syncs dev cluster

  # ═══════════════════════════════════════════
  # STAGE 8: Deploy to Staging (automatic after dev)
  # ═══════════════════════════════════════════
  deploy-staging:
    needs: [deploy-dev]
    runs-on: ubuntu-latest
    environment: staging    # May have protection rules
    steps:
      - name: Run smoke tests on dev
        run: make test-smoke ENV=dev
      - name: Update GitOps repo (staging overlay)
        run: |
          # Similar to dev but targeting staging overlay
          cd gitops-repo/services/${{ inputs.service-name }}/overlays/staging
          kustomize edit set image ...
          git push

  # ═══════════════════════════════════════════
  # STAGE 9: Deploy to Production (manual approval)
  # ═══════════════════════════════════════════
  deploy-production:
    needs: [deploy-staging]
    runs-on: ubuntu-latest
    environment: production    # Requires manual approval
    steps:
      - name: Run smoke tests on staging
        run: make test-smoke ENV=staging
      - name: Update GitOps repo (production overlay)
        run: |
          cd gitops-repo/services/${{ inputs.service-name }}/overlays/production
          kustomize edit set image ...
          git push
      # ArgoCD deploys as canary to production
```

### Step 5 — Key Design Decisions & Trade-offs

```
Decision 1: GitHub Actions vs Jenkins
├── Chose: GitHub Actions
├── Why: Managed service, no infrastructure to maintain, 
│        native GitHub integration, great marketplace
├── Alternative: Jenkins (more customizable, but heavy to maintain)
└── Trade-off: Vendor lock-in with GitHub

Decision 2: GitOps (ArgoCD) vs Push-based CD
├── Chose: ArgoCD (Pull-based GitOps)
├── Why: Git as single source of truth, audit trail, 
│        easy rollback (git revert), self-healing
├── Alternative: Push-based (pipeline deploys directly via kubectl)
└── Trade-off: Additional repository to manage, learning curve

Decision 3: Kustomize vs Helm
├── Chose: Kustomize (with overlays per environment)
├── Why: Simpler, built into kubectl, no templating complexity
├── Alternative: Helm (more powerful for complex charts)
└── Trade-off: Helm is better for third-party apps, 
    Kustomize is better for internal services

Decision 4: Mono-repo vs Multi-repo
├── Chose: Multi-repo (one repo per service)
├── Why: Independent CI/CD, clear ownership, simpler pipelines
├── Alternative: Mono-repo (easier code sharing, atomic changes)
└── Trade-off: Harder to make cross-service changes

Decision 5: Shared Workflows
├── Chose: GitHub Actions reusable workflows
├── Why: Standardization across 50+ services, one place to update
├── Teams call the shared workflow with service-specific inputs
└── Teams can extend (not replace) the shared workflow
```

### Step 6 — Self-Service Onboarding

```
How does a new team onboard a service?

1. Create a new repo from the service template
   gh repo create myorg/new-service --template=myorg/service-template

   Template includes:
   ├── Dockerfile (multi-stage, optimized)
   ├── .github/workflows/ci.yml (calls shared workflow)
   ├── Makefile (standard targets: build, test, lint, scan)
   ├── k8s/ directory (Kustomize base + overlays)
   └── README.md with onboarding instructions

2. Register in the GitOps repo
   # Add to gitops-repo/services/new-service/
   # ArgoCD ApplicationSet auto-discovers new services

3. That's it! Push code → pipeline runs → deploys automatically

No DevOps team involvement needed!
```

## 📋 System Design 2: Design a Monitoring & Observability Stack

### The Question:
> "Design a complete monitoring and observability stack for a company running 100+ microservices on Kubernetes across 3 AWS regions. The system should provide metrics, logs, traces, and alerting."

---

### Step 1 — Clarify Requirements

```
Questions I'd ask:
- How much log data per day? (~500GB/day)
- How long to retain metrics? (30 days hot, 1 year cold)
- How long to retain logs? (7 days hot, 90 days cold)
- How many developers need access? (~100)
- Do we need compliance/audit logging?
- What's the budget sensitivity?
- Any existing tools we must keep?

### Step 3 — Detailed Component Design

#### 3A — Metrics (Prometheus + Thanos)

```
Why Thanos instead of standalone Prometheus?
- Prometheus is per-cluster — we have 3 clusters across 3 regions
- Thanos provides a global view across all clusters
- Thanos provides long-term storage (S3)

Architecture:

  Region 1 (us-east-1)    Region 2 (eu-west-1)    Region 3 (ap-south-1)
  ┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
  │ Prometheus       │     │ Prometheus       │     │ Prometheus       │
  │ + Thanos Sidecar │     │ + Thanos Sidecar │     │ + Thanos Sidecar │
  └────────┬────────┘     └────────┬────────┘     └────────┬────────┘
           │                       │                       │
           └───────────┬───────────┴───────────┬───────────┘
                       │                       │
              ┌────────▼────────┐    ┌─────────▼────────┐
              │  Thanos Query   │    │  Thanos Store     │
              │  (Global view)  │    │  Gateway (S3)     │
              └────────┬────────┘    └──────────────────┘
                       │
              ┌────────▼────────┐
              │    Grafana       │
              │  (Dashboards)    │
              └─────────────────┘

Key Exporters (what we monitor):
├── node-exporter          → Node-level metrics (CPU, memory, disk, network)
├── kube-state-metrics     → Kubernetes object metrics (pods, deployments)
├── cAdvisor               → Container-level metrics
├── blackbox-exporter      → Endpoint availability (HTTP, TCP, DNS)
├── Application metrics    → Custom metrics via /metrics endpoint (RED method)
└── AWS CloudWatch Exporter → AWS service metrics (RDS, ELB, etc.)
```

#### 3B — Logging (Loki + Fluent Bit)

```
Why Loki over ELK?
- ELK is powerful but resource-HEAVY (Elasticsearch needs lots of memory)
- Loki only indexes metadata (labels), stores logs in object storage (S3)
- Much cheaper at scale (500GB/day of logs)
- Native Grafana integration (same tool for metrics and logs)
- BUT: ELK is better for full-text search across log content

Architecture:

  ┌──────────────────────────────────────────────┐
  │  Every Node (DaemonSet)                       │
  │  ┌──────────────┐                             │
  │  │  Fluent Bit   │  ← Collects logs from all  │
  │  │  (log shipper) │    containers on the node  │
  │  └──────┬───────┘                             │
  └─────────┼─────────────────────────────────────┘
            │
  ┌─────────▼─────────────────────────────────────┐
  │  Loki (Write Path)                             │
  │  ┌────────────┐  ┌───────────┐  ┌──────────┐  │
  │  │ Distributor │→ │ Ingester  │→ │ S3       │  │
  │  │             │  │ (in-memory│  │ (long-   │  │
  │  │             │  │  + WAL)   │  │  term)   │  │
  │  └────────────┘  └───────────┘  └──────────┘  │
  └────────────────────────────────────────────────┘
            │
  ┌─────────▼─────────────────────────────────────┐
  │  Loki (Read Path)                              │
  │  ┌──────────┐  ┌────────────┐                  │
  │  │ Querier  │← │ Query      │                  │
  │  │          │  │ Frontend   │ ← Grafana        │
  │  └──────────┘  └────────────┘                  │
  └────────────────────────────────────────────────┘
```

#### 3C — Tracing (Tempo + OpenTelemetry)

```
Why Tempo?
- Part of the Grafana ecosystem (metrics + logs + traces in one UI)
- Uses object storage (S3) — very cost effective
- Scales horizontally
- Integrates with OpenTelemetry natively

### Step 4 — Alerting Architecture

```
Alerting Flow:

  Prometheus Alert Rules
         │
         ▼
  AlertManager
  ├── Route by severity:
  │   ├── CRITICAL → PagerDuty (pages on-call)
  │   ├── WARNING  → Slack #alerts-warning
  │   ├── INFO     → Slack #alerts-info
  │   └── CUSTOM   → Team-specific channels
  │
  ├── Grouping:
  │   Group related alerts to avoid alert fatigue
  │   (e.g., if 10 pods are failing, send 1 alert not 10)
  │
  ├── Silencing:
  │   During maintenance windows
  │
  └── Inhibition:
      If cluster is down, suppress pod-level alerts

Key Alerts to Configure:

Application Alerts:
├── Error rate > 1% for 5 minutes (SLO breach)
├── Latency p99 > 500ms for 5 minutes
├── Request rate dropped > 50% (possible outage)
└── Pod restart count > 3 in 10 minutes

Infrastructure Alerts:
├── Node CPU > 85% for 10 minutes
├── Node memory > 90% for 5 minutes
├── Disk usage > 85%
├── Node NotReady
└── PersistentVolume > 80% full

Kubernetes Alerts:
├── Pod CrashLoopBackOff
├── Deployment replicas mismatch (desired != actual)
├── HPA at max replicas for 30 minutes
├── PodDisruptionBudget violated
└── Certificate expiring in < 30 days

Business Alerts:
├── Order processing rate dropped > 50%
├── Payment failure rate > 5%
└── User signup rate anomaly
```

      expr: |

        severity: warning
      annotations:

### Step 5 — Dashboards Strategy

```
Dashboard Hierarchy:

Level 1 — Executive Dashboard (1 dashboard)
├── Overall system health (green/yellow/red)
├── Key business metrics (orders, revenue, users)
├── SLO compliance (are we meeting our targets?)
└── Cost overview

Level 2 — Platform Dashboard (3-5 dashboards)
├── Kubernetes cluster health (all regions)
├── Node resource usage
├── Deployment status
├── Database health
└── CI/CD pipeline health

Level 3 — Service Dashboard (per service)
├── RED metrics (Rate, Errors, Duration)
│   ├── Request rate (requests/second)
│   ├── Error rate (% of 5xx responses)
│   └── Duration (p50, p95, p99 latency)
├── Resource usage (CPU, memory)
├── Pod status and restarts
├── Dependencies health
└── Recent deployments overlay

Level 4 — Debug Dashboard (on-demand)
├── Specific service deep-dive
├── Database query performance
├── Cache hit ratios
├── Connection pool status
└── Garbage collection metrics

USE vs RED Method:
- USE Method (for infrastructure): Utilization, Saturation, Errors
- RED Method (for services): Rate, Errors, Duration
```

### Step 6 — Multi-Region Considerations

```
Challenge: 3 regions, each with its own cluster

Solution:

Option A — Centralized Observability (Chosen)
├── Each region has local Prometheus + Fluent Bit + OTel Collector
├── Thanos provides global query across all regions
├── Loki and Tempo configured to write to centralized S3
├── One Grafana instance queries everything
│
│   Pros: Single pane of glass, easier to manage
│   Cons: Cross-region query latency, data transfer costs

Option B — Federated Observability
├── Each region has full independent stack
├── Central Grafana with multi-data-source support
│
│   Pros: No cross-region dependency
│   Cons: Harder to correlate across regions

Storage strategy:
├── Hot storage (< 7 days): Local SSD in cluster
├── Warm storage (7-30 days): S3 Standard
├── Cold storage (30-365 days): S3 Glacier
└── After 1 year: Delete (unless compliance requires longer)

Cost estimation (rough):
├── Metrics: ~$500/month (Thanos + S3)
├── Logs: ~$2000/month (Loki + S3 for 500GB/day)
├── Traces: ~$500/month (Tempo + S3, sampling at 10%)
├── Grafana: ~$200/month (or Grafana Cloud)
└── Total: ~$3200/month (MUCH cheaper than Datadog at this scale)
```

### Step 7 — SLI/SLO Framework

```
For each critical service, define:

SLIs (Service Level Indicators) — What we measure:
├── Availability: % of successful requests
├── Latency: p99 response time
├── Throughput: requests per second
└── Error rate: % of errors

SLOs (Service Level Objectives) — Our targets:
├── Availability: 99.95% (22 minutes downtime/month)
├── Latency: p99 < 300ms
├── Error rate: < 0.1%
└── Error budget: 0.05% = 22 minutes/month

Error Budget Policy:
├── If error budget > 50% remaining → ship features freely
├── If error budget 25-50% remaining → slow down, extra testing
├── If error budget < 25% remaining → freeze features, focus on reliability
└── If error budget exhausted → stop ALL feature work until recovered

SLO Dashboards in Grafana:
├── Burn rate alerts (if burning error budget too fast)
├── Monthly SLO tracking
├── Error budget remaining visualization
└── Historical SLO compliance
```

### Capacity Planning

```
Calculating resources for 10,000 RPS:

Assumptions:
├── Each pod handles ~500 RPS
├── We want 50% headroom for spikes
├── Target: 10,000 RPS × 1.5 (headroom) = 15,000 RPS capacity

Pod calculation:
├── 15,000 RPS ÷ 500 RPS/pod = 30 pods needed
├── Spread across 3 AZs = 10 pods per AZ
├── With HPA: min=30, max=100

Node calculation:
├── Each pod: 500m CPU, 512Mi memory
├── 30 pods × 500m = 15 CPU cores needed
├── m6i.xlarge = 4 vCPU, 16GB memory
├── 15 cores ÷ 4 cores/node = ~4 nodes (minimum)
├── With overhead: 6 nodes minimum, 15 desired, 30 max

Database calculation:
├── Assume 20% of requests hit DB = 2,000 QPS
├── db.r6g.2xlarge: handles ~5,000 QPS easily
├── Add read replica for read-heavy queries
├── Connection pooling: PgBouncer (max 200 connections)

Cache calculation:
├── Assume 60% cache hit ratio
├── 6,000 requests served from Redis
├── cache.r6g.xlarge: handles 100,000+ ops/sec (overkill, but HA)
```

## 🧠 System Design Tips for Interviews

```
1. ALWAYS START WITH REQUIREMENTS
   - Don't jump into drawing boxes
   - Ask: "What are the most important requirements?"
   - Non-functional requirements matter MORE than functional

2. DRAW AS YOU TALK
   - Use whiteboard/paper/Excalidraw
   - Boxes for services, arrows for data flow
   - Label everything

3. DISCUSS TRADE-OFFS
   - Every decision has a trade-off
   - "I chose X over Y because..."
   - Shows depth of understanding

4. MENTION FAILURE SCENARIOS
   - "What happens if this component fails?"
   - "How does the system recover?"
   - Shows production mindset

5. THINK ABOUT SECURITY
   - Network segmentation (public/private subnets)
   - Encryption (at rest and in transit)
   - IAM and RBAC
   - Secret management

6. THINK ABOUT COST
   - "This would cost approximately $X/month"
   - Show you can balance performance with budget

7. DON'T OVER-ENGINEER
   - Start simple, add complexity as needed
   - "We could add this later as we scale"
   - YAGNI — You Ain't Gonna Need It (yet)

8. KNOW THE NUMBERS
   ├── 1 web server: ~1000-5000 RPS (depending on framework)
   ├── 1 PostgreSQL: ~5000-20000 QPS
   ├── 1 Redis: ~100,000 ops/sec
   ├── 1 Kafka partition: ~10MB/s
   ├── S3 PUT: ~3500 req/s per prefix
   ├── DNS propagation: up to 48 hours (but usually minutes)
   └── TLS handshake: ~100-300ms
```
