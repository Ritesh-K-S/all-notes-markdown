### Q1. Explain the OSI Model. Why does it matter?

**Answer:**

The **OSI (Open Systems Interconnection) Model** breaks down network communication into **7 layers**. Think of it like sending a letter — each layer adds an envelope around your data.

```
Layer 7 — Application    → What the USER interacts with (HTTP, HTTPS, DNS, FTP, SSH)
Layer 6 — Presentation   → Data formatting & encryption (SSL/TLS, JPEG, JSON)
Layer 5 — Session        → Manages connections/sessions (keeps track of "who's talking")
Layer 4 — Transport      → Reliable delivery (TCP, UDP) — port numbers live here
Layer 3 — Network        → Routing & IP addressing (IP, ICMP) — routers work here
Layer 2 — Data Link      → Local delivery via MAC addresses — switches work here
Layer 1 — Physical       → Actual cables, signals, electricity — hardware level
```

**Easy way to remember (top to bottom):**
> **A**ll **P**eople **S**eem **T**o **N**eed **D**ata **P**rocessing

**Why DevOps engineers care:**

| Problem | Which Layer? | Tool to Debug |
|---------|-------------|---------------|
| "Website returns 500 error" | Layer 7 (Application) | `curl -v`, browser dev tools |
| "SSL certificate expired" | Layer 6 (Presentation) | `openssl s_client` |
| "Connection timeout" | Layer 4 (Transport) | `ss`, `netstat`, `telnet` |
| "Can't reach the server" | Layer 3 (Network) | `ping`, `traceroute`, `ip route` |
| "Network interface is down" | Layer 2/1 | `ip link`, `ethtool` |

💡 **In practice**, most DevOps work happens at **Layers 3, 4, and 7**. But understanding all layers helps you debug systematically — start from the bottom and work your way up.

### Q2. What is the difference between TCP and UDP?

**Answer:**

Think of **TCP** as a **phone call** and **UDP** as a **postcard**.

| Feature | TCP (Transmission Control Protocol) | UDP (User Datagram Protocol) |
|---------|--------------------------------------|------------------------------|
| **Connection** | Connection-oriented (handshake first) | Connectionless (just send) |
| **Reliability** | ✅ Guaranteed delivery (acknowledgments, retransmissions) | ❌ No guarantee — packets can be lost |
| **Ordering** | ✅ Data arrives in order | ❌ No ordering guarantee |
| **Speed** | Slower (overhead of reliability) | Faster (no overhead) |
| **Header size** | 20 bytes | 8 bytes |
| **Use cases** | HTTP, HTTPS, SSH, FTP, Email (SMTP) | DNS, DHCP, Video streaming, VoIP, Gaming |

**TCP Three-Way Handshake (how a connection starts):**

```
Client                    Server
  │                         │
  │ ── SYN ──────────────→  │  Step 1: "Hey, I want to connect"
  │                         │
  │ ←──────── SYN-ACK ──  │  Step 2: "Sure, I'm ready too"
  │                         │
  │ ── ACK ──────────────→  │  Step 3: "Great, let's go!"
  │                         │
  │ ←── DATA TRANSFER ──→  │  Connection established!
```

**TCP Four-Way Termination (how a connection ends):**

```
Client                    Server
  │ ── FIN ──────────────→  │  "I'm done sending"
  │ ←──────── ACK ────────  │  "OK, noted"
  │ ←──────── FIN ────────  │  "I'm done too"
  │ ── ACK ──────────────→  │  "OK, goodbye!"
```

💡 **DevOps context:** "In load balancers, L4 (TCP) load balancing is faster because it just routes packets without inspecting content. L7 (HTTP) load balancing is smarter — it can route based on URL path, headers, cookies — but it's slightly slower because it has to read the HTTP request."

### Q3. What is the difference between IPv4 and IPv6?

**Answer:**

| Feature | IPv4 | IPv6 |
|---------|------|------|
| **Address length** | 32 bits | 128 bits |
| **Format** | Dotted decimal: `192.168.1.1` | Hex with colons: `2001:0db8:85a3::8a2e:0370:7334` |
| **Total addresses** | ~4.3 billion | ~340 undecillion (practically infinite) |
| **NAT needed?** | Yes (because addresses ran out) | No (enough for every device) |
| **Header** | Variable length, more complex | Fixed 40-byte, simpler |
| **Built-in security** | Optional (IPSec) | Mandatory (IPSec) |

```bash
# Check your IPv4 address
ip -4 addr show

# Check your IPv6 address
ip -6 addr show

# Test IPv6 connectivity
ping6 google.com
```

💡 **Real-world:** Most cloud providers (AWS, Azure, GCP) support **dual-stack** (both IPv4 and IPv6). Kubernetes supports IPv6 and dual-stack since v1.23.

### Q4. What is subnetting? Explain CIDR notation.

**Answer:**

**Subnetting** = Dividing a large network into smaller, manageable sub-networks (subnets).

**CIDR (Classless Inter-Domain Routing)** = A way to describe IP address ranges using a **prefix length**.

```
Format: IP_ADDRESS / PREFIX_LENGTH

Example: 192.168.1.0/24
         └──────────┘ └┘
          Network part  How many bits are for the network

/24 means: First 24 bits = network, Last 8 bits = hosts
           That gives us 2^8 = 256 addresses (254 usable hosts)
```

**Common CIDR blocks you should memorize:**

| CIDR | Subnet Mask | Total IPs | Usable Hosts | Used For |
|------|-------------|-----------|--------------|----------|
| `/32` | 255.255.255.255 | 1 | 1 | Single host |
| `/24` | 255.255.255.0 | 256 | 254 | Small networks |
| `/16` | 255.255.0.0 | 65,536 | 65,534 | Medium networks, VPCs |
| `/8` | 255.0.0.0 | 16,777,216 | 16,777,214 | Huge networks |
| `/0` | 0.0.0.0 | All IPs | All | "Anywhere" (default route) |

**Private IP ranges (RFC 1918) — you'll see these everywhere:**

| Range | CIDR | Commonly Used For |
|-------|------|-------------------|
| `10.0.0.0 – 10.255.255.255` | `10.0.0.0/8` | Cloud VPCs (AWS, Azure) |
| `172.16.0.0 – 172.31.255.255` | `172.16.0.0/12` | Docker default bridge network |
| `192.168.0.0 – 192.168.255.255` | `192.168.0.0/16` | Home networks, labs |

```bash
# Calculate subnet info
ipcalc 192.168.1.0/24
# Shows: network address, broadcast, host range, etc.

# AWS VPC example:
# VPC CIDR: 10.0.0.0/16 (65,536 IPs)
#   Public Subnet:  10.0.1.0/24 (256 IPs)
#   Private Subnet: 10.0.2.0/24 (256 IPs)
#   DB Subnet:      10.0.3.0/24 (256 IPs)
```

💡 **Interview tip:** If they ask you to design a VPC, start with `/16` for the VPC, and use `/24` subnets. Always leave room for growth.

### Q5. What is DNS? How does DNS resolution work?

**Answer:**

**DNS (Domain Name System)** translates human-friendly names (like `google.com`) into IP addresses (like `142.250.190.14`) that computers understand. It's like the **phone book of the internet**.

**DNS Resolution — Step by Step:**

```
You type: www.google.com in your browser

Step 1: Browser Cache
   └── "Have I looked this up before?" → If yes, use cached IP. Done!

Step 2: OS Cache (stub resolver)
   └── Check /etc/hosts first (local override)
   └── Then check OS DNS cache

Step 3: Recursive Resolver (usually your ISP's DNS or 8.8.8.8)
   └── This is the "worker" that does the heavy lifting
   └── If it has the answer cached → return it. Done!

Step 4: Root Name Server (.)
   └── Resolver asks: "Where can I find .com domains?"
   └── Root says: "Go ask the .com TLD server at 192.5.6.30"

Step 5: TLD (Top-Level Domain) Server (.com)
   └── Resolver asks: "Where can I find google.com?"
   └── TLD says: "Go ask Google's authoritative server at ns1.google.com"

Step 6: Authoritative Name Server (ns1.google.com)
   └── Resolver asks: "What's the IP for www.google.com?"
   └── Authoritative says: "It's 142.250.190.14"

Step 7: Resolver returns the IP to your browser
   └── Browser connects to 142.250.190.14
   └── Result is cached for future use (based on TTL)
```

**DNS Record Types (MUST know for interviews):**

| Record | Purpose | Example |
|--------|---------|---------|
| `A` | Maps domain to IPv4 address | `example.com → 93.184.216.34` |
| `AAAA` | Maps domain to IPv6 address | `example.com → 2606:2800:220:1:...` |
| `CNAME` | Alias — points to another domain | `www.example.com → example.com` |
| `MX` | Mail server for the domain | `example.com → mail.example.com` |
| `TXT` | Text records (SPF, DKIM, verification) | `v=spf1 include:_spf.google.com` |
| `NS` | Nameserver for the domain | `example.com → ns1.example.com` |
| `SOA` | Start of Authority — zone info | Primary NS, admin email, serial, TTL |
| `SRV` | Service location (port + host) | Used by K8s, LDAP, SIP |
| `PTR` | Reverse DNS — IP to domain | `34.216.184.93 → example.com` |

```bash
# DNS lookup tools
nslookup google.com
dig google.com
dig +short google.com           # Just the IP
dig google.com MX               # Query MX records
dig google.com NS               # Query nameservers
dig @8.8.8.8 google.com         # Query using specific DNS server
host google.com                 # Simple lookup

# Trace the full DNS resolution path
dig +trace google.com

# Check local DNS config
cat /etc/resolv.conf

# Clear DNS cache (varies by OS)
sudo systemd-resolve --flush-caches    # Linux (systemd)
ipconfig /flushdns                      # Windows
```

### Q6. What is the difference between a domain, FQDN, and URL?

**Answer:**

| Term | What It Is | Example |
|------|-----------|---------|
| **Domain** | The name registered in DNS | `google.com` |
| **FQDN (Fully Qualified Domain Name)** | The COMPLETE domain name ending with a dot | `www.google.com.` |
| **URL (Uniform Resource Locator)** | The full web address including protocol, path, etc. | `https://www.google.com/search?q=devops` |

```
URL breakdown:
https://www.google.com:443/search?q=devops#results
└─┬──┘ └─┬─┘└───┬────┘└┬┘ └──┬──┘└───┬───┘└──┬───┘
  │      │      │      │     │       │        │
Protocol │   Domain    Port  Path    Query   Fragment
      Subdomain              string
```

💡 **FQDN vs Domain:** The trailing dot in FQDN (`google.com.`) represents the root of the DNS hierarchy. In practice, browsers and most tools add it automatically.

### Q7. Explain HTTP methods. What is the difference between GET, POST, PUT, PATCH, and DELETE?

**Answer:**

HTTP methods (also called verbs) tell the server **what action** you want to perform:

| Method | Purpose | Idempotent? | Body? | Example |
|--------|---------|-------------|-------|---------|
| `GET` | Read/retrieve data | ✅ Yes | ❌ No | Get user profile |
| `POST` | Create new data | ❌ No | ✅ Yes | Create a new user |
| `PUT` | Replace entire resource | ✅ Yes | ✅ Yes | Replace all user fields |
| `PATCH` | Update part of resource | ✅ Yes | ✅ Yes | Update just the email |
| `DELETE` | Remove a resource | ✅ Yes | ❌ No | Delete a user |
| `HEAD` | Same as GET but no body | ✅ Yes | ❌ No | Check if resource exists |
| `OPTIONS` | What methods are allowed? | ✅ Yes | ❌ No | CORS preflight check |

**Idempotent** = calling it multiple times gives the same result (safe to retry).

```bash
# GET — Retrieve data
curl -X GET https://api.example.com/users/123

# POST — Create new resource
curl -X POST https://api.example.com/users \
  -H "Content-Type: application/json" \
  -d '{"name": "Ritesh", "email": "ritesh@example.com"}'

# PUT — Replace entire resource
curl -X PUT https://api.example.com/users/123 \
  -H "Content-Type: application/json" \
  -d '{"name": "Ritesh Singh", "email": "ritesh@newmail.com"}'

# PATCH — Update partial resource
curl -X PATCH https://api.example.com/users/123 \
  -H "Content-Type: application/json" \
  -d '{"email": "newemail@example.com"}'

# DELETE — Remove resource
curl -X DELETE https://api.example.com/users/123
```

### Q8. What are HTTP status codes? Explain the main categories.

**Answer:**

HTTP status codes are **3-digit numbers** the server sends back to tell you what happened:

| Range | Category | Meaning |
|-------|----------|---------|
| `1xx` | Informational | Request received, processing... |
| `2xx` | Success | Request was successful ✅ |
| `3xx` | Redirection | Client needs to go somewhere else |
| `4xx` | Client Error | YOU (the client) did something wrong ❌ |
| `5xx` | Server Error | The SERVER messed up 💥 |

**Status codes you MUST memorize:**

| Code | Meaning | When You See It |
|------|---------|----------------|
| `200` | OK | Request succeeded |
| `201` | Created | New resource was created (POST) |
| `204` | No Content | Success, but nothing to return (DELETE) |
| `301` | Moved Permanently | URL changed permanently (SEO redirect) |
| `302` | Found (Temporary Redirect) | URL temporarily changed |
| `304` | Not Modified | Use your cached version |
| `400` | Bad Request | Invalid request (wrong JSON, missing fields) |
| `401` | Unauthorized | Not authenticated (no/invalid token) |
| `403` | Forbidden | Authenticated but not authorized (no permission) |
| `404` | Not Found | Resource doesn't exist |
| `405` | Method Not Allowed | Wrong HTTP method (e.g., POST to a GET-only endpoint) |
| `408` | Request Timeout | Server waited too long for your request |
| `429` | Too Many Requests | Rate limited |
| `500` | Internal Server Error | Something crashed on the server |
| `502` | Bad Gateway | Reverse proxy got an invalid response from upstream |
| `503` | Service Unavailable | Server is overloaded or under maintenance |
| `504` | Gateway Timeout | Reverse proxy timed out waiting for upstream |

💡 **DevOps context:**
- **502** and **504** are your BEST FRIENDS in debugging. They usually mean your backend app is either down (502) or too slow (504).
- When Nginx shows 502 → your upstream app crashed.
- When Nginx shows 504 → your upstream app is too slow and the proxy timed out.

```bash
# Check HTTP status code of a URL
curl -o /dev/null -s -w "%{http_code}" https://example.com
# Output: 200

# Verbose output (see headers + status)
curl -v https://example.com

# Follow redirects
curl -L https://example.com
```

### Q9. What is HTTPS? How does TLS/SSL work?

**Answer:**

**HTTPS = HTTP + TLS/SSL encryption.** It encrypts all data between client and server so nobody in the middle can read or tamper with it.

**TLS Handshake (simplified):**

```
Client (Browser)                    Server (Nginx)
     │                                    │
     │ ── Client Hello ──────────────→    │  "Hi! I support TLS 1.3, these ciphers..."
     │                                    │
     │ ←──────────── Server Hello ────    │  "OK! Let's use TLS 1.3 with this cipher"
     │                                    │  "Here's my SSL certificate"
     │                                    │
     │ Verify certificate                 │  Client checks:
     │   - Is it expired?                 │    ✅ Valid CA
     │   - Is the CA trusted?             │    ✅ Not expired
     │   - Does domain match?             │    ✅ Domain matches
     │                                    │
     │ ── Key Exchange ──────────────→    │  Both sides generate a shared secret
     │ ←──────────────────────────────    │  (using Diffie-Hellman or similar)
     │                                    │
     │ ══ Encrypted Communication ═══    │  All data is now encrypted! 🔒
```

**Key concepts:**

| Term | What It Is |
|------|-----------|
| **SSL** | Old name (SSL 3.0 is deprecated). People still say "SSL" but mean TLS |
| **TLS** | Current standard. TLS 1.2 and TLS 1.3 are what you should use |
| **Certificate** | A file that proves "I am really example.com". Issued by a CA |
| **CA (Certificate Authority)** | Trusted organization that issues certificates (Let's Encrypt, DigiCert) |
| **Private Key** | Server's secret key. NEVER share this! |
| **Public Key** | Included in the certificate. Safe to share |

```bash
# Check SSL certificate of a website
openssl s_client -connect google.com:443 -servername google.com </dev/null 2>/dev/null | \
  openssl x509 -noout -dates -subject -issuer

# Check certificate expiry
echo | openssl s_client -connect example.com:443 2>/dev/null | \
  openssl x509 -noout -enddate

# Generate a self-signed certificate (for testing)
openssl req -x509 -newkey rsa:4096 -keyout key.pem -out cert.pem -days 365 -nodes

# In Kubernetes, use cert-manager for automatic certificate management:
# It gets free certificates from Let's Encrypt and auto-renews them
```

💡 **DevOps must-know:** Use **Let's Encrypt** for free certificates. In Kubernetes, use **cert-manager** to auto-issue and auto-renew. Never let certificates expire — set up monitoring alerts for expiry.

### Q10. What is the difference between HTTP/1.1, HTTP/2, and HTTP/3?

**Answer:**

| Feature | HTTP/1.1 | HTTP/2 | HTTP/3 |
|---------|----------|--------|--------|
| **Year** | 1997 | 2015 | 2022 |
| **Protocol** | TCP | TCP | QUIC (UDP-based) |
| **Connections** | One request per connection (or keep-alive) | Multiplexing — multiple requests on ONE connection | Same multiplexing, but faster |
| **Header compression** | None | HPACK compression | QPACK compression |
| **Server Push** | No | Yes (server can send resources before asked) | Yes |
| **Head-of-Line blocking** | Yes (one slow request blocks others) | Solved at HTTP level, but TCP still has it | Fully solved (UDP-based) |
| **TLS required?** | No (optional) | Practically yes (browsers require it) | Yes (built into QUIC) |

```
HTTP/1.1 — One request at a time (or pipelining, which nobody uses):
────[Request 1]────[Response 1]────[Request 2]────[Response 2]────

HTTP/2 — Multiple requests at the same time on ONE connection:
────[Request 1]──[Request 2]──[Request 3]────
────[Response 2]──[Response 1]──[Response 3]──   (any order!)

HTTP/3 — Same as HTTP/2, but on QUIC (UDP), so even packet loss doesn't slow other streams
```

```bash
# Check which HTTP version a site uses
curl -sI https://google.com | head -1
# HTTP/2 200

# Force HTTP/2
curl --http2 https://example.com

# Check in Nginx config
# In nginx.conf:
server {
    listen 443 ssl http2;    # Enable HTTP/2
    # ...
}
```

### Q11. What is a Load Balancer? Explain L4 vs L7 Load Balancing.

**Answer:**

A **load balancer** distributes incoming traffic across multiple servers so no single server gets overwhelmed.

```
                        ┌─── Server 1 (app)
Client ──→ Load ────────┤
           Balancer ────┼─── Server 2 (app)
                        │
                        └─── Server 3 (app)
```

**L4 (Transport Layer) vs L7 (Application Layer):**

| Feature | L4 Load Balancing | L7 Load Balancing |
|---------|-------------------|-------------------|
| **Works at** | TCP/UDP level | HTTP/HTTPS level |
| **Sees** | IP addresses and port numbers | Full HTTP request (URL, headers, cookies) |
| **Routing based on** | IP + Port | URL path, hostname, headers, cookies |
| **Speed** | Faster (less processing) | Slightly slower (inspects content) |
| **SSL termination** | Can pass through (not terminate) | Usually terminates SSL |
| **Examples** | AWS NLB, HAProxy (TCP mode) | AWS ALB, Nginx, HAProxy (HTTP mode), Traefik |

**L7 routing example:**
```
Request: GET /api/users → routes to → API servers
Request: GET /images/logo.png → routes to → Static file servers
Request: Host: admin.example.com → routes to → Admin servers
```

**Common Load Balancing Algorithms:**

| Algorithm | How It Works | Best For |
|-----------|-------------|----------|
| **Round Robin** | Send to each server in turn (1→2→3→1→2→3) | Equal servers, general use |
| **Least Connections** | Send to the server with fewest active connections | Varying request durations |
| **IP Hash** | Hash the client IP to always route to the same server | Session persistence |
| **Weighted Round Robin** | Like Round Robin but servers have weights | Servers with different capacities |
| **Random** | Pick a random server | Simple, works surprisingly well |

```nginx
# Nginx load balancer configuration example
upstream backend {
    least_conn;                    # Use least connections algorithm
    server 10.0.0.1:8080 weight=3;  # This server gets 3x traffic
    server 10.0.0.2:8080 weight=1;
    server 10.0.0.3:8080 backup;    # Only used if others are down
}

server {
    listen 80;
    location / {
        proxy_pass http://backend;
    }
}
```

### Q12. What is the difference between a Forward Proxy and a Reverse Proxy?

**Answer:**

| Feature | Forward Proxy | Reverse Proxy |
|---------|--------------|---------------|
| **Sits in front of** | Clients | Servers |
| **Who knows about it** | Client knows, server doesn't | Server knows, client doesn't |
| **Purpose** | Client privacy, content filtering, caching | Load balancing, SSL termination, security |
| **Direction** | Client → Proxy → Internet | Internet → Proxy → Backend servers |
| **Examples** | Squid, corporate proxy, VPN | Nginx, HAProxy, Traefik, AWS ALB |

```
Forward Proxy:
Client ──→ [Forward Proxy] ──→ Internet ──→ Server
           "I'll go to the    (Server sees proxy's
            internet for you"  IP, not client's)

Reverse Proxy:
Client ──→ Internet ──→ [Reverse Proxy] ──→ Backend Server
                        "I'll distribute     (Client sees proxy,
                         requests to          not backend)
                         my backends"
```

**Why Reverse Proxies are essential in DevOps:**

```
                    ┌─ SSL Termination (handle HTTPS)
                    ├─ Load Balancing (distribute traffic)
Reverse Proxy does: ├─ Caching (serve static content fast)
                    ├─ Compression (gzip responses)
                    ├─ Rate Limiting (prevent abuse)
                    ├─ Security Headers (XSS, CSRF protection)
                    └─ Routing (URL path → correct microservice)
```

```nginx
# Nginx as reverse proxy
server {
    listen 443 ssl;
    server_name api.example.com;

    # SSL termination
    ssl_certificate /etc/ssl/cert.pem;
    ssl_certificate_key /etc/ssl/key.pem;

    # Proxy to backend
    location / {
        proxy_pass http://localhost:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

### Q13. What is NAT (Network Address Translation)? Why is it used?

**Answer:**

**NAT** translates **private IP addresses** to **public IP addresses** (and vice versa). It's why your home can have 20 devices but only needs ONE public IP from your ISP.

```
Private Network (192.168.1.x)              Public Internet
┌─────────────────────┐
│ Phone: 192.168.1.10 │
│ Laptop: 192.168.1.11│──→ [NAT Router] ──→  203.0.113.5 ──→  Google
│ TV: 192.168.1.12    │    (translates       (your ONE         (sees your
└─────────────────────┘     private→public)   public IP)        public IP)
```

**Types of NAT:**

| Type | What It Does | Used In |
|------|-------------|---------|
| **SNAT (Source NAT)** | Changes source IP (private → public) | Outbound traffic from VPC |
| **DNAT (Destination NAT)** | Changes destination IP (public → private) | Port forwarding, load balancing |
| **PAT (Port Address Translation)** | Multiple private IPs share one public IP using different ports | Home routers, NAT Gateway |

**In AWS (very common in interviews):**

```
Public Subnet                          Private Subnet
┌──────────────┐                      ┌──────────────┐
│ EC2 Instance │                      │ EC2 Instance │
│ (has public  │                      │ (NO public   │
│  IP)         │                      │  IP)         │
│              │                      │    │         │
└──────────────┘                      └────┼─────────┘
                                           │
                                      [NAT Gateway]
                                      (in public subnet)
                                           │
                                      [Internet Gateway]
                                           │
                                        Internet
```

The **NAT Gateway** lets private instances access the internet (for updates, downloading packages) WITHOUT being directly reachable from the internet.

```bash
# On Linux, enable NAT (for routing between networks):
sudo iptables -t nat -A POSTROUTING -o eth0 -j MASQUERADE
sudo sysctl -w net.ipv4.ip_forward=1
```

## 🔐 Section 4: Firewalls & Security

---

### Q14. What is a Firewall? Explain Security Groups vs NACLs in AWS.

**Answer:**

A **firewall** controls what traffic is ALLOWED in and out of your network/server.

**Linux firewalls:**
```bash
# UFW (Ubuntu — simplest)
sudo ufw allow 22/tcp       # Allow SSH
sudo ufw allow 80/tcp       # Allow HTTP
sudo ufw allow 443/tcp      # Allow HTTPS
sudo ufw deny 3306/tcp      # Block MySQL
sudo ufw enable             # Turn on the firewall
sudo ufw status             # Check rules

# firewalld (RHEL/CentOS)
sudo firewall-cmd --add-service=http --permanent
sudo firewall-cmd --add-port=8080/tcp --permanent
sudo firewall-cmd --reload

# iptables (low-level, universal)
sudo iptables -A INPUT -p tcp --dport 22 -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT
sudo iptables -A INPUT -j DROP
```

**AWS — Security Groups vs NACLs:**

| Feature | Security Group | NACL (Network ACL) |
|---------|---------------|-------------------|
| **Level** | Instance (ENI) level | Subnet level |
| **State** | Stateful (return traffic auto-allowed) | Stateless (must allow return traffic explicitly) |
| **Rules** | Allow rules ONLY | Allow AND Deny rules |
| **Evaluation** | All rules evaluated together | Rules evaluated in ORDER (by number) |
| **Default** | Deny all inbound, Allow all outbound | Allow all (default NACL) |
| **Scope** | Applies to specific instances | Applies to ALL instances in the subnet |

```
Internet → [NACL] → Subnet → [Security Group] → EC2 Instance
            │                       │
            │ Stateless             │ Stateful
            │ Has DENY rules       │ Only ALLOW rules
            │ Processed in order   │ All rules evaluated
```

💡 **Best Practice:** Use Security Groups as your primary firewall. Use NACLs as a backup layer for subnet-level rules (like blocking an IP range).

### Q15. What is the difference between Stateful and Stateless firewalls?

**Answer:**

| Feature | Stateful | Stateless |
|---------|----------|-----------|
| **Remembers connections?** | ✅ Yes — tracks connection state | ❌ No — treats each packet independently |
| **Return traffic** | Automatically allowed | Must explicitly allow return traffic |
| **Performance** | Slightly slower (tracks state) | Faster (no state tracking) |
| **Examples** | Security Groups, iptables (with `-m state`) | NACLs, basic packet filters |

**Example to understand:**

You allow **outbound HTTP (port 80)** from your server:

```
Stateful Firewall:
  → Outbound rule: Allow port 80 ✅
  ← Return traffic: Auto-allowed ✅ (because it remembers you started this connection)

Stateless Firewall:
  → Outbound rule: Allow port 80 ✅
  ← Return traffic: ❌ BLOCKED (it doesn't remember!)
  → You ALSO need: Inbound rule allowing ephemeral ports (1024-65535)
```

### Q16. What is a VPC? Explain VPC architecture with subnets.

**Answer:**

**VPC (Virtual Private Cloud)** = Your own **isolated network** in the cloud. Think of it as your own private data center inside AWS/Azure/GCP.

```
VPC (10.0.0.0/16 — Your isolated network)
│
├── Availability Zone: us-east-1a
│   ├── Public Subnet (10.0.1.0/24)
│   │   ├── NAT Gateway
│   │   ├── Bastion Host (jump server)
│   │   └── ALB (Application Load Balancer)
│   │
│   └── Private Subnet (10.0.2.0/24)
│       ├── App Server 1
│       ├── App Server 2
│       └── Database (RDS)
│
├── Availability Zone: us-east-1b
│   ├── Public Subnet (10.0.3.0/24)
│   │   └── ALB (multi-AZ)
│   │
│   └── Private Subnet (10.0.4.0/24)
│       ├── App Server 3
│       └── Database (RDS standby)
│
├── Internet Gateway (connects VPC to internet)
├── Route Tables (define traffic routing)
├── Security Groups (instance-level firewall)
└── NACLs (subnet-level firewall)
```

**Key components:**

| Component | Purpose |
|-----------|---------|
| **Internet Gateway (IGW)** | Connects VPC to the internet |
| **NAT Gateway** | Lets private instances access internet (outbound only) |
| **Route Table** | Rules that decide where traffic goes |
| **Public Subnet** | Has a route to the IGW (instances get public IPs) |
| **Private Subnet** | No direct route to IGW (instances are hidden from internet) |
| **Bastion Host** | A "jump server" in the public subnet for SSH access to private instances |

**Public vs Private Subnet:**

```bash
# Public Subnet Route Table:
Destination     Target
10.0.0.0/16     local          (traffic within VPC stays in VPC)
0.0.0.0/0       igw-12345      (everything else → Internet Gateway)

# Private Subnet Route Table:
Destination     Target
10.0.0.0/16     local          (traffic within VPC stays in VPC)
0.0.0.0/0       nat-12345      (everything else → NAT Gateway)
```

### Q17. What is a VPN? How does Site-to-Site VPN work?

**Answer:**

**VPN (Virtual Private Network)** creates an **encrypted tunnel** between two networks, making them act as if they're on the same local network.

```
Your Office Network              AWS VPC
(10.1.0.0/16)                   (10.0.0.0/16)
      │                               │
[VPN Device/Router] ═══════════ [VPN Gateway]
      │              encrypted        │
      │              tunnel           │
   Employees      over internet    EC2 Instances

Now office machines can directly reach EC2 instances
via private IPs (10.0.x.x) through the encrypted tunnel!
```

**Types of VPN:**

| Type | Connects | Use Case |
|------|----------|----------|
| **Site-to-Site VPN** | Two networks (office ↔ cloud) | Connect office to AWS VPC |
| **Client VPN** | Individual user → network | Remote employees accessing internal resources |
| **VPC Peering** | Two VPCs | Connect VPCs in same/different regions |
| **Transit Gateway** | Multiple VPCs + on-prem | Hub-and-spoke network architecture |

💡 **For production traffic with high bandwidth**, use **AWS Direct Connect** (or Azure ExpressRoute) instead of VPN — it's a dedicated physical connection, not over the internet.

### Q19. How does Kubernetes networking work? What are the key rules?

**Answer:**

Kubernetes has **3 fundamental networking rules:**

1. **Every Pod gets its own IP address**
2. **All Pods can communicate with all other Pods without NAT**
3. **All Nodes can communicate with all Pods without NAT**

```
Kubernetes Cluster Networking:
┌────────── Node 1 (10.0.1.10) ──────────┐    ┌────────── Node 2 (10.0.1.11) ──────────┐
│                                          │    │                                          │
│  Pod A (10.244.0.5)  Pod B (10.244.0.6) │    │  Pod C (10.244.1.3)  Pod D (10.244.1.4) │
│       │                    │             │    │       │                    │             │
│       └─── Pod Network (CNI) ───────────┼────┼───────┘                    │             │
│                                          │    │                                          │
└──────────────────────────────────────────┘    └──────────────────────────────────────────┘
                    │                                            │
                    └──────── Node Network ──────────────────────┘
```

**Key networking concepts in K8s:**

| Concept | What It Does |
|---------|-------------|
| **Pod Network** | Assigns IPs to Pods. Managed by CNI plugins (Calico, Flannel, Cilium) |
| **Service** | Stable IP/DNS for a group of Pods (since Pod IPs change) |
| **ClusterIP** | Internal-only service IP (only accessible within cluster) |
| **NodePort** | Exposes service on every node's IP at a specific port (30000-32767) |
| **LoadBalancer** | Creates a cloud load balancer pointing to your service |
| **Ingress** | HTTP/HTTPS routing rules (path-based, host-based) |
| **CoreDNS** | Internal DNS — `my-service.my-namespace.svc.cluster.local` |
| **Network Policy** | Firewall rules between Pods |

```yaml
# ClusterIP Service (internal only)
apiVersion: v1
kind: Service
metadata:
  name: my-api
spec:
  selector:
    app: my-api
  ports:
    - port: 80         # Service port
      targetPort: 8080  # Container port
  type: ClusterIP      # Only accessible within cluster

---
# NodePort Service (accessible from outside on node IP)
apiVersion: v1
kind: Service
metadata:
  name: my-api-nodeport
spec:
  selector:
    app: my-api
  ports:
    - port: 80
      targetPort: 8080
      nodePort: 30080    # Access via <NodeIP>:30080
  type: NodePort

---
# Ingress (HTTP routing)
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-ingress
spec:
  rules:
    - host: api.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: my-api
                port:
                  number: 80
```

**DNS in Kubernetes:**
```bash
# Any pod can reach a service by its name:
curl http://my-api                                    # Same namespace
curl http://my-api.default                            # Explicit namespace
curl http://my-api.default.svc.cluster.local          # FQDN

# DNS format: <service>.<namespace>.svc.cluster.local
```

### Q20. What is a CNI plugin? Name some popular ones.

**Answer:**

**CNI (Container Network Interface)** is a standard that defines how networking is set up for containers. Kubernetes delegates all networking to CNI plugins.

Think of it this way: Kubernetes says "I need Pod A to have an IP and reach Pod B." The CNI plugin makes it happen.

| CNI Plugin | Key Features | Used By |
|-----------|-------------|---------|
| **Calico** | L3 networking, Network Policies, BGP routing | Most popular. Used by EKS, AKS |
| **Flannel** | Simple overlay network. Easy setup | Learning/simple clusters |
| **Cilium** | eBPF-based, very fast, advanced Network Policies | High-performance, security-focused |
| **Weave** | Easy setup, encryption support | Small clusters |
| **AWS VPC CNI** | Pods get real VPC IPs (not overlay) | EKS (AWS) |
| **Azure CNI** | Pods get real Azure VNet IPs | AKS (Azure) |

💡 **Interview tip:** "In production, I prefer Calico for its robust Network Policy support, or Cilium for its eBPF performance. For AWS EKS, the VPC CNI is used by default since pods get real VPC IPs."

### Q21. How do you troubleshoot a "connection refused" error?

**Answer:**

"Connection refused" means the target machine is reachable, but **nothing is listening** on the requested port.

```bash
# Systematic debugging approach:

# Step 1: Is the service actually running?
sudo systemctl status nginx
sudo ss -tlnp | grep :80
# If nothing shows → service isn't listening. Start it!

# Step 2: Is it listening on the right interface?
ss -tlnp | grep :80
# "127.0.0.1:80" → only listening on localhost (can't be reached remotely!)
# "0.0.0.0:80" → listening on ALL interfaces ✅

# Step 3: Is the firewall blocking it?
sudo iptables -L -n | grep 80
sudo ufw status
# Or in AWS: check Security Group inbound rules

# Step 4: Can you reach the port from the client?
nc -zv server-ip 80
telnet server-ip 80
curl -v http://server-ip:80

# Step 5: Is there a network path?
traceroute server-ip
ping server-ip
```

**Common causes:**
1. Service is not running
2. Service is listening on `127.0.0.1` instead of `0.0.0.0`
3. Firewall (iptables/ufw/security group) is blocking the port
4. Wrong port number
5. Service crashed and hasn't been restarted

### Q22. How do you troubleshoot a "connection timed out" error?

**Answer:**

"Connection timed out" means the client **didn't get any response** — not even a rejection.

```bash
# Step 1: Can you reach the machine at all?
ping server-ip
# If ping fails → network issue (routing, firewall blocking ICMP)

# Step 2: Can you reach the specific port?
nc -zv -w 5 server-ip 80    # -w 5 = 5 second timeout
telnet server-ip 80

# Step 3: Trace the network path
traceroute server-ip
mtr server-ip    # Better — real-time

# Step 4: Check DNS resolution
dig server-hostname
nslookup server-hostname
# Is it resolving to the correct IP?

# Step 5: Check routing
ip route get server-ip
# Does the route make sense?

# Step 6: Check firewall on BOTH sides
# On the server:
sudo iptables -L -n
# On your machine:
sudo iptables -L -n
# In cloud: check NACL + Security Group + Route Tables
```

**Common causes:**
1. Firewall silently DROP-ping (not rejecting) traffic
2. Wrong IP address or DNS resolution issue
3. Routing problem (no route to host)
4. Server in a private subnet without proper NAT/routing
5. Security Group has no inbound rule for the port
6. NACL blocking the traffic

💡 **Key difference:**
- **Connection refused** = machine is reachable, port is closed
- **Connection timed out** = machine can't be reached OR firewall is silently dropping

### Q23. Explain the `tcpdump` command. How do you capture packets?

**Answer:**

`tcpdump` is a **packet sniffer** — it captures and displays network packets in real-time. It's the command-line version of Wireshark.

```bash
# Capture all traffic on a specific interface
sudo tcpdump -i eth0

# Capture only TCP traffic on port 80
sudo tcpdump -i eth0 tcp port 80

# Capture traffic to/from a specific host
sudo tcpdump -i eth0 host 192.168.1.100

# Capture only the first 10 packets
sudo tcpdump -i eth0 -c 10

# Save capture to a file (for Wireshark analysis)
sudo tcpdump -i eth0 -w capture.pcap

# Read a saved capture file
tcpdump -r capture.pcap

# Show ASCII content (useful for HTTP debugging)
sudo tcpdump -i eth0 -A port 80

# Capture DNS queries
sudo tcpdump -i eth0 port 53

# Capture traffic between two hosts
sudo tcpdump -i eth0 src 10.0.0.1 and dst 10.0.0.2

# Human-readable timestamps
sudo tcpdump -i eth0 -tttt port 443
```

**Real debugging example — "Why can't my app connect to the database?"**
```bash
# On the app server, capture traffic to DB port
sudo tcpdump -i eth0 tcp port 3306 -nn

# What you might see:
# SYN sent but no SYN-ACK → firewall blocking
# SYN-ACK received, RST sent → app rejecting
# No packets at all → wrong IP/routing issue
```

### Q24. What is MTU? What happens when MTU is wrong?

**Answer:**

**MTU (Maximum Transmission Unit)** is the **maximum packet size** that can be sent over a network without being fragmented.

| Network | Default MTU |
|---------|------------|
| Ethernet | 1500 bytes |
| VPN/Tunnels | ~1400 bytes (overhead from encapsulation) |
| Jumbo Frames | 9000 bytes (within data centers) |
| Docker overlay | 1450 bytes |

**When MTU is wrong, you get weird problems:**
- Small requests work, large ones fail
- SSH works, SCP hangs
- Ping works, web pages don't load
- `curl` for small pages works, large downloads fail

```bash
# Check current MTU
ip link show eth0
# Look for: mtu 1500

# Test with specific packet sizes
ping -M do -s 1472 google.com    # 1472 + 28 bytes header = 1500 (should work)
ping -M do -s 1473 google.com    # 1473 + 28 = 1501 → too big! → will fail

# Change MTU
sudo ip link set eth0 mtu 1400

# In Docker, set MTU in daemon.json:
# /etc/docker/daemon.json
{
    "mtu": 1450
}
```

💡 **Common interview scenario:** "After setting up a VPN, large file transfers fail but small requests work." → Answer: MTU is too high. VPN adds encapsulation overhead (40-60 bytes). Solution: Lower MTU to 1400.

### Q25. What is CORS? Why does it matter?

**Answer:**

**CORS (Cross-Origin Resource Sharing)** is a browser security feature that blocks web pages from making requests to a different domain than the one that served the page.

```
Your web app: https://myapp.com
Your API:     https://api.myapp.com   ← Different origin!

Browser says: "Hmm, myapp.com is trying to reach api.myapp.com.
              That's a different origin! Let me check if the API allows this..."

Browser sends a PREFLIGHT request (OPTIONS):
  OPTIONS /users
  Origin: https://myapp.com
  Access-Control-Request-Method: GET

API must respond with:
  Access-Control-Allow-Origin: https://myapp.com
  Access-Control-Allow-Methods: GET, POST
  Access-Control-Allow-Headers: Content-Type, Authorization

If the API responds correctly → browser allows the request ✅
If not → browser blocks it ❌ (CORS error)
```

**Fixing CORS in Nginx:**
```nginx
server {
    location /api/ {
        # Allow specific origin
        add_header 'Access-Control-Allow-Origin' 'https://myapp.com';
        add_header 'Access-Control-Allow-Methods' 'GET, POST, PUT, DELETE, OPTIONS';
        add_header 'Access-Control-Allow-Headers' 'Content-Type, Authorization';
        
        # Handle preflight requests
        if ($request_method = 'OPTIONS') {
            add_header 'Access-Control-Max-Age' 86400;
            add_header 'Content-Length' 0;
            return 204;
        }
        
        proxy_pass http://backend;
    }
}
```

💡 **DevOps tip:** CORS issues are VERY common when deploying frontend + backend separately. The fix is always on the **server side** (backend or reverse proxy), never the client.

### Q26. What is a CDN (Content Delivery Network)? How does it work?

**Answer:**

A **CDN** is a network of servers distributed **globally** that caches your content close to users — making your website faster.

```
Without CDN:
User in Tokyo ──── 200ms ────→ Server in Virginia

With CDN:

                  (fast!)        │
                                 │ (if not cached, fetch from origin)
                                 └── 200ms ──→ Origin Server in Virginia
                                                    │
                                              (CDN caches the response)
                                              (next request from Tokyo is instant!)
```

**What a CDN caches:**
- Static files (images, CSS, JavaScript, fonts)
- HTML pages (if configured)
- API responses (if cache headers allow)
- Video content

**Popular CDNs:**

| CDN | Provider | Notes |
|-----|----------|-------|
| **CloudFront** | AWS | Integrates with S3, ALB, EC2 |
| **Cloudflare** | Cloudflare | Also provides DDoS protection, DNS, WAF |
| **Akamai** | Akamai | Enterprise — used by largest companies |
| **Azure CDN** | Microsoft | Integrates with Azure services |
| **Cloud CDN** | Google | Integrates with GCP |

💡 **Cache control headers matter:**
```bash
# Tell CDN to cache for 1 hour

curl -I https://example.com | grep -i cache
```

### Q27. What is a VLAN? How is it different from a subnet?

**Answer:**

| Feature | VLAN (Virtual LAN) | Subnet |
|---------|-------------------|--------|
| **Layer** | Layer 2 (Data Link) | Layer 3 (Network) |
| **What it separates** | Broadcast domains (which devices can "hear" each other) | IP address ranges |
| **How** | Tagging Ethernet frames (802.1Q) | IP addressing and routing |
| **Where** | On switches | On routers/firewalls |
| **Purpose** | Physical network segmentation | Logical IP address organization |

```
Without VLANs (one big network):
All 100 devices on one switch can see each other's broadcast traffic.
→ Noisy, insecure, poor performance.

With VLANs:
VLAN 10 (Engineering): 20 devices → only see each other
VLAN 20 (HR): 15 devices → only see each other
VLAN 30 (Servers): 10 devices → isolated from user VLANs

Same physical switch, but logically separated!
```

💡 **In cloud environments**, VLANs are less relevant because cloud providers handle network isolation through VPCs and subnets. But understanding VLANs is important for hybrid cloud and on-premise setups.

### Q28. What is BGP (Border Gateway Protocol)? Why is it important?

**Answer:**

**BGP** is the routing protocol that makes the **internet work**. It decides how data travels from one network to another across the entire internet.

Think of it as the **GPS of the internet** — it finds the best path for your data.

```
Your network (AS 12345)
       │
       │ BGP announces: "I own 203.0.113.0/24"
       │
    [Router] ──── BGP session ──── [ISP Router] ──── [Internet backbone]
                                        │
                                   BGP decides the
                                   best path to reach
                                   any destination
```

**Key terms:**

| Term | Meaning |
|------|---------|
| **AS (Autonomous System)** | A network owned by one organization (e.g., AWS, Google, your company) |
| **AS Number (ASN)** | Unique ID for each AS (e.g., Google = AS 15169) |
| **BGP Peering** | Two ASes agreeing to exchange routing information |
| **BGP Announcement** | "Hey, I own this IP range and you can reach it through me" |

**Where DevOps engineers encounter BGP:**
1. **AWS Direct Connect** — uses BGP to exchange routes between your network and AWS
2. **Kubernetes with Calico** — uses BGP to distribute Pod routes between nodes
3. **Multi-cloud networking** — BGP for routing between cloud providers
4. **CDN providers** — use BGP Anycast to route users to nearest edge server

```bash
# Check BGP routes (on a BGP-enabled router)
show ip bgp summary
show ip bgp neighbors

# With Calico in Kubernetes
calicoctl node status
# Shows BGP peering status between nodes
```

### Q29. Explain the concept of DNS-based service discovery. How does it work in microservices?

**Answer:**

In microservices, services need to **find each other**. DNS-based service discovery means services register their location in DNS, and other services look them up by name.

```
Traditional (bad way):
  App config: DATABASE_HOST=10.0.2.45:5432
  → If database moves to a new IP, everything breaks!

DNS-based discovery (good way):
  App config: DATABASE_HOST=database.production.svc.cluster.local
  → DNS always resolves to the current IP, even if it changes!
```

**How it works in Kubernetes:**

```yaml
# When you create a Service:
apiVersion: v1
kind: Service
metadata:
  name: user-service
  namespace: production
spec:
  selector:
    app: user-service
  ports:
    - port: 80

# Kubernetes automatically creates DNS records:
# user-service.production.svc.cluster.local → ClusterIP (10.96.45.67)

# Any pod can reach it by:
# curl http://user-service                           (same namespace)
# curl http://user-service.production                (cross-namespace)
# curl http://user-service.production.svc.cluster.local  (FQDN)
```

**Other service discovery methods:**

| Method | How It Works | Example |
|--------|-------------|---------|
| **DNS-based** | Services registered in DNS | Kubernetes Services, Consul DNS |
| **Registry-based** | Services register in a central registry | Consul, Eureka, etcd |
| **Mesh-based** | Sidecar proxy handles discovery | Istio, Linkerd |

### Q30. A website is slow. How do you debug it from a networking perspective?

**Answer:**

This is a **scenario-based question** — interviewers want to see your systematic approach:

```bash
# Step 1: Is it a DNS issue?
time dig +short example.com
# If this takes > 100ms → DNS resolution is slow
# Fix: Check /etc/resolv.conf, use faster DNS (8.8.8.8, 1.1.1.1)

# Step 2: Is it a network latency issue?
ping -c 10 example.com
# Check average round-trip time
# < 50ms for same region
# 100-200ms for cross-continent

# Step 3: Is there packet loss?
mtr example.com
# Look for any hop with > 1% packet loss

# Step 4: Is the TCP connection slow?
curl -o /dev/null -w "\
  DNS:        %{time_namelookup}s\n\
  Connect:    %{time_connect}s\n\
  TLS:        %{time_appconnect}s\n\
  TTFB:       %{time_starttransfer}s\n\
  Total:      %{time_total}s\n\
" https://example.com

# This breaks down the request into stages:
# DNS: Time to resolve the domain
# Connect: Time to establish TCP connection
# TLS: Time for TLS handshake
# TTFB: Time to First Byte (server processing time)
# Total: Total time

# Step 5: Is the server overloaded?
# SSH into the server and check:
top           # CPU
free -h       # Memory
iostat -x     # Disk I/O
ss -s         # Connection count

# Step 6: Is the bandwidth saturated?
iftop -i eth0    # Real-time bandwidth usage

# Step 7: Is the problem at a specific network hop?
traceroute example.com
# Look for sudden latency jumps between hops
```

**Decision tree:**

```
Website is slow
    ├── DNS slow? → Use faster DNS, add caching
    ├── High latency? → Use CDN, deploy closer to users
    ├── Packet loss? → Network issue (ISP, cable, router)
    ├── Slow TLS? → Use TLS 1.3, session resumption
    ├── High TTFB? → Server is slow (CPU/memory/DB issue)
    ├── High bandwidth? → Compress content, use CDN
    └── Too many connections? → Connection pooling, load balancing
```

💡 **The `curl` timing breakdown is interview GOLD.** Memorize it. It immediately tells you WHERE the slowness is — DNS, network, TLS, or server processing.

## 🏆 Networking Commands Cheat Sheet

```bash
# IP & Interface
ip addr show                    # Show IP addresses
ip route show                   # Show routing table
ip link show                    # Show interfaces

# DNS
dig +short example.com          # Quick DNS lookup
dig example.com MX              # Query MX records
cat /etc/resolv.conf            # DNS config

# Connectivity
ping -c 4 host                  # Test reachability
traceroute host                 # Trace route
mtr host                        # Real-time traceroute
nc -zv host port                # Test port connectivity

# Ports & Connections
ss -tlnp                        # Show listening ports
ss -t state established         # Show established connections
lsof -i :80                     # What's using port 80

# Packet Capture
tcpdump -i eth0 port 80         # Capture HTTP traffic
tcpdump -i eth0 -w file.pcap   # Save to file

# HTTP Testing
curl -v https://example.com     # Verbose HTTP request
curl -o /dev/null -w "%{time_total}" url  # Time a request

# Firewall
iptables -L -n                  # List firewall rules
ufw status                      # UFW status
```
