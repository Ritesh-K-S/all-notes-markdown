### Q1. What is Docker? Why is it used?

**Answer:**

**Docker** is a platform that lets you package your application along with **everything it needs** (code, libraries, dependencies, config) into a single unit called a **container**. This container runs the same way everywhere — your laptop, a test server, or production.

**The problem Docker solves:**

```
Before Docker:
Developer: "It works on my machine!" 🤷
Ops team:  "But it doesn't work on the server!" 😡

The problem:
- Dev machine has Python 3.9, server has Python 3.6
- Dev has certain libraries installed, server doesn't
- Different OS, different config, different everything

After Docker:
Developer: "Here's a container with EVERYTHING included."
Ops team:  "It works exactly the same everywhere!" ✅
```

**Key benefits:**

| Benefit | Explanation |
|---------|-------------|
| **Consistency** | Same environment everywhere — dev, test, staging, production |
| **Isolation** | Each app runs in its own container — no conflicts |
| **Lightweight** | Containers share the host OS kernel — much lighter than VMs |
| **Fast startup** | Containers start in seconds (VMs take minutes) |
| **Portable** | Build once, run anywhere (any Linux system, any cloud) |
| **Scalable** | Spin up 100 containers in seconds |

💡 **Don't say:** "Docker is a lightweight VM." **Say:** "Docker uses OS-level virtualization (namespaces and cgroups) to isolate processes, sharing the host kernel, which makes it much more efficient than traditional VMs."

### Q2. What is the difference between a Container and a Virtual Machine?

**Answer:**

```
Virtual Machine:                        Container:
┌─────────────────────┐                ┌──────────────────────┐
│   App A    │  App B  │                │   App A   │   App B  │
├────────────┼─────────┤                ├───────────┼──────────┤
│  Bins/Libs │Bins/Libs│                │ Bins/Libs │ Bins/Libs│
├────────────┼─────────┤                ├───────────┴──────────┤
│  Guest OS  │Guest OS │                │    Container Runtime │
│  (Ubuntu)  │(CentOS) │                │    (Docker Engine)   │
├────────────┴─────────┤                ├──────────────────────┤
│     Hypervisor       │                │      Host OS         │
│  (VMware/KVM/Hyper-V)│                │     (Linux)          │
├──────────────────────┤                ├──────────────────────┤
│   Physical Hardware  │                │   Physical Hardware  │
└──────────────────────┘                └──────────────────────┘

Each VM has its OWN OS                  All containers share the
(heavy! 1-10 GB each)                  HOST OS kernel (light! MBs)
```

| Feature | Virtual Machine | Container |
|---------|----------------|-----------|
| **Size** | Gigabytes (full OS) | Megabytes (just app + dependencies) |
| **Startup time** | Minutes | Seconds |
| **OS** | Each VM has its own OS | Shares host OS kernel |
| **Isolation** | Strong (hardware-level) | Process-level (lighter) |
| **Resource usage** | Heavy (runs full OS) | Lightweight (shares kernel) |
| **Portability** | Less portable (tied to hypervisor) | Very portable (runs anywhere Docker runs) |
| **Use case** | Different OS needed, strong isolation | Microservices, CI/CD, quick deployments |

💡 **When to use VMs over containers:** When you need different operating systems, stronger security isolation, or when running Windows and Linux workloads on the same machine.

### Q3. Explain Docker Architecture. What are the main components?

**Answer:**

Docker uses a **client-server architecture**:

```
┌──────────────────────────────────────────────────────┐
│                    Docker Host                        │
│                                                       │
│  ┌────────────┐    ┌──────────────────────────────┐  │
│  │Docker Client│───→│      Docker Daemon            │  │
│  │(docker CLI) │    │      (dockerd)                │  │
│  │             │    │                                │  │
│  │ docker run  │    │  ┌────────┐  ┌────────┐      │  │
│  │ docker build│    │  │Container│  │Container│      │  │
│  │ docker pull │    │  │  (App1) │  │  (App2) │      │  │
│  │ docker ps   │    │  └────────┘  └────────┘      │  │
│  └────────────┘    │                                │  │
│                     │  ┌─────────┐  ┌─────────┐    │  │
│                     │  │ Image 1 │  │ Image 2 │    │  │
│                     │  └─────────┘  └─────────┘    │  │
│                     └──────────────────────────────┘  │
└──────────────────────────┼───────────────────────────┘
                           │
                    ┌──────┴──────┐
                    │   Docker    │
                    │  Registry   │
                    │(Docker Hub) │
                    └─────────────┘
```

| Component | What It Does |
|-----------|-------------|
| **Docker Client** (`docker`) | The CLI tool you type commands into. Talks to the daemon |
| **Docker Daemon** (`dockerd`) | The background service that does the actual work — builds, runs, manages containers |
| **Docker Images** | Read-only templates with your app + dependencies. Like a "blueprint" |
| **Docker Containers** | Running instances of images. The actual "living" application |
| **Docker Registry** | Storage for images (Docker Hub, ECR, GCR, Harbor) |
| **Dockerfile** | A text file with instructions to BUILD an image |

```bash
# The flow:
# 1. Write a Dockerfile
# 2. Build an image: docker build -t myapp .
# 3. Push to registry: docker push myregistry/myapp:v1
# 4. Pull on another machine: docker pull myregistry/myapp:v1
# 5. Run a container: docker run myregistry/myapp:v1
```

### Q4. What is a Docker Image? How is it different from a Container?

**Answer:**

Think of it like this:

| Concept | Real-World Analogy |
|---------|-------------------|
| **Docker Image** | A recipe (instructions + ingredients list) |
| **Docker Container** | A dish made from that recipe (the actual running thing) |

Or in programming terms:
- **Image** = Class (definition/blueprint)
- **Container** = Object/Instance (running instance of that class)

```bash
# An IMAGE is a read-only template
docker images
# REPOSITORY    TAG       IMAGE ID       SIZE
# nginx         latest    a8758716bb6a   187MB
# python        3.11      12345abc       920MB

# A CONTAINER is a running instance of an image
docker ps
# CONTAINER ID   IMAGE    STATUS         NAMES
# abc123         nginx    Up 5 minutes   my-web-server

# You can create MULTIPLE containers from ONE image
docker run -d --name web1 nginx
docker run -d --name web2 nginx
docker run -d --name web3 nginx
# Three containers, all from the same nginx image!
```

**Key difference:**

| Feature | Image | Container |
|---------|-------|-----------|
| **State** | Read-only (immutable) | Read-write (has a writable layer on top) |
| **Persistence** | Stored on disk permanently | Ephemeral — data is lost when removed (unless volumes are used) |
| **Created by** | `docker build` or `docker pull` | `docker run` or `docker create` |
| **Can be shared** | Yes — push to registry | No — it's a running process |

### Q5. What is a Dockerfile? Explain common instructions.

**Answer:**

A **Dockerfile** is a text file containing step-by-step instructions to build a Docker image. Think of it as a **recipe**.

```dockerfile
# Start from a base image
FROM python:3.11-slim

# Set metadata
LABEL maintainer="ritesh@company.com"
LABEL version="1.0"

# Set the working directory inside the container
WORKDIR /app

# Copy dependency file first (for caching!)
COPY requirements.txt .

# Install dependencies
RUN pip install --no-cache-dir -r requirements.txt

# Copy the rest of the application code
COPY . .

# Set environment variables
ENV FLASK_ENV=production
ENV PORT=5000

# Expose the port (documentation — doesn't actually publish)
EXPOSE 5000

# Create a non-root user (security best practice)
RUN adduser --disabled-password --gecos "" appuser
USER appuser

# Default command to run when container starts
CMD ["python", "app.py"]
```

**All important Dockerfile instructions:**

| Instruction | Purpose | Example |
|-------------|---------|---------|
| `FROM` | Base image to start from | `FROM ubuntu:22.04` |
| `WORKDIR` | Set working directory inside container | `WORKDIR /app` |
| `COPY` | Copy files from host to container | `COPY . /app` |
| `ADD` | Like COPY but can extract tar files and download URLs | `ADD app.tar.gz /app` |
| `RUN` | Execute a command during BUILD time | `RUN apt-get update && apt-get install -y curl` |
| `CMD` | Default command when container STARTS (can be overridden) | `CMD ["python", "app.py"]` |
| `ENTRYPOINT` | Main executable (harder to override than CMD) | `ENTRYPOINT ["python"]` |
| `ENV` | Set environment variables | `ENV NODE_ENV=production` |
| `ARG` | Build-time variables (not available at runtime) | `ARG VERSION=1.0` |
| `EXPOSE` | Document which port the app listens on | `EXPOSE 8080` |
| `VOLUME` | Create a mount point for persistent data | `VOLUME /data` |
| `USER` | Switch to a non-root user | `USER appuser` |
| `LABEL` | Add metadata to the image | `LABEL version="1.0"` |
| `HEALTHCHECK` | Define a health check command | `HEALTHCHECK CMD curl -f http://localhost/` |

### Q6. What is the difference between `CMD` and `ENTRYPOINT`?

**Answer:**

This is one of the **most asked** Docker questions. Here's the clear difference:

| Feature | `CMD` | `ENTRYPOINT` |
|---------|-------|-------------|
| **Purpose** | Default command + arguments | The main executable |
| **Override** | Easily overridden with `docker run ... <new command>` | NOT easily overridden (need `--entrypoint` flag) |
| **Use when** | You want users to be able to change the command | You want the container to ALWAYS run a specific program |

**Example 1: Using CMD (can be overridden)**
```dockerfile
FROM ubuntu
CMD ["echo", "Hello World"]
```

```bash
docker run myimage
# Output: Hello World

docker run myimage echo "Goodbye"
# Output: Goodbye  ← CMD was replaced!
```

**Example 2: Using ENTRYPOINT (fixed executable)**
```dockerfile
FROM ubuntu
ENTRYPOINT ["echo"]
CMD ["Hello World"]
```

```bash
docker run myimage
# Output: Hello World

docker run myimage "Goodbye"
# Output: Goodbye  ← Only the CMD part (argument) was replaced!
# ENTRYPOINT (echo) is still there

docker run --entrypoint /bin/bash myimage
# Only way to override ENTRYPOINT: use --entrypoint flag
```

**Best practice — use both together:**
```dockerfile
# ENTRYPOINT = the program to run
# CMD = default arguments for that program
ENTRYPOINT ["python"]
CMD ["app.py"]

# docker run myimage          → runs: python app.py
# docker run myimage test.py  → runs: python test.py
```

💡 **Real-world pattern:** Use `ENTRYPOINT` for the main application binary and `CMD` for default flags/arguments.

### Q7. What is the difference between `COPY` and `ADD`?

**Answer:**

| Feature | `COPY` | `ADD` |
|---------|--------|-------|
| **Basic copy** | ✅ Copies files from host to container | ✅ Same |
| **Extract tar files** | ❌ No — copies tar as-is | ✅ Auto-extracts tar/gzip |
| **Download URLs** | ❌ No | ✅ Can download from URLs |
| **Best practice** | ✅ Recommended — predictable and simple | ⚠️ Only use when you need tar extraction |

```dockerfile
# COPY — simple and predictable (preferred)
COPY app.py /app/
COPY requirements.txt /app/

# ADD — auto-extracts tar files
ADD archive.tar.gz /app/
# The contents of archive.tar.gz will be extracted into /app/

# ADD — can download from URLs (but don't do this in production)
ADD https://example.com/file.txt /app/
# Better approach: use RUN with curl
RUN curl -o /app/file.txt https://example.com/file.txt
```

💡 **Rule of thumb:** Always use `COPY` unless you specifically need tar extraction. The Docker documentation itself recommends `COPY` over `ADD`.

### Q8. What are the most essential Docker commands?

**Answer:**

```bash
# ═══════════ IMAGE COMMANDS ═══════════

# List all images
docker images
docker image ls

# Pull an image from registry
docker pull nginx:1.25

# Build an image from Dockerfile
docker build -t myapp:v1 .
docker build -t myapp:v1 -f Dockerfile.prod .    # Custom Dockerfile

# Tag an image
docker tag myapp:v1 myregistry.com/myapp:v1

# Push an image to registry
docker push myregistry.com/myapp:v1

# Remove an image
docker rmi nginx:1.25
docker image prune         # Remove unused images
docker image prune -a      # Remove ALL unused images

# Inspect an image
docker inspect nginx
docker history nginx       # See image layers

# ═══════════ CONTAINER COMMANDS ═══════════

# Run a container
docker run -d --name myapp -p 8080:80 nginx
# -d = detached (background)
# --name = give it a name
# -p 8080:80 = map host port 8080 to container port 80

# List running containers
docker ps

# List ALL containers (including stopped)
docker ps -a

# Stop a container
docker stop myapp

# Start a stopped container
docker start myapp

# Restart a container
docker restart myapp

# Remove a container
docker rm myapp
docker rm -f myapp         # Force remove (even if running)

# View logs
docker logs myapp
docker logs -f myapp       # Follow (real-time)
docker logs --tail 100 myapp   # Last 100 lines

# Execute a command inside a running container
docker exec -it myapp /bin/bash     # Interactive shell
docker exec myapp cat /etc/nginx/nginx.conf   # Run single command

# Copy files between host and container
docker cp myapp:/app/logs ./local-logs/    # Container → Host
docker cp ./config.yml myapp:/app/config/  # Host → Container

# View container resource usage
docker stats
docker stats myapp

# Inspect a container (detailed info)
docker inspect myapp

# View container processes
docker top myapp

# ═══════════ CLEANUP COMMANDS ═══════════

# Remove ALL stopped containers
docker container prune

# Remove ALL unused data (containers, images, networks, volumes)
docker system prune
docker system prune -a     # Also remove unused images
docker system prune --volumes   # Also remove unused volumes

# Check disk usage
docker system df
```

### Q9. What does `docker run -it` mean? Explain each flag.

**Answer:**

```bash
docker run -it ubuntu /bin/bash
```

| Flag | Meaning | Without It |
|------|---------|-----------|
| `-i` | **Interactive** — keeps STDIN open (you can type input) | Container ignores your keyboard input |
| `-t` | **TTY** — allocates a pseudo-terminal (gives you a prompt) | No terminal formatting, raw output |
| `-it` (together) | Interactive terminal — you get a shell you can type in | You can't interact with the container |

```bash
# Interactive mode — you get a bash shell inside the container
docker run -it ubuntu /bin/bash
root@abc123:/# ls        # You're inside the container!
root@abc123:/# exit       # Leave the container

# Detached mode — runs in background
docker run -d nginx
# Container runs in background, you get your terminal back

# Common combinations:
docker run -d --name web -p 80:80 nginx          # Background, named, port mapped
docker run -it --rm ubuntu bash                    # Interactive, auto-remove when done
docker run -d --restart=always --name app myapp    # Background, auto-restart
```

### Q10. What is the difference between `docker stop` and `docker kill`?

**Answer:**

| Command | Signal Sent | Behavior |
|---------|------------|----------|
| `docker stop` | SIGTERM (15), then SIGKILL after 10s | Graceful — gives the process time to clean up |
| `docker kill` | SIGKILL (9) immediately | Forceful — kills instantly, no cleanup |

```bash
# Graceful stop (recommended)
docker stop myapp
# Docker sends SIGTERM → app has 10 seconds to finish
# If not stopped after 10s → sends SIGKILL

# Custom timeout
docker stop -t 30 myapp     # Wait 30 seconds before force killing

# Immediate kill
docker kill myapp
# App is killed INSTANTLY — no cleanup, no saving state

# Send a specific signal
docker kill --signal=SIGHUP myapp     # Reload config
docker kill --signal=SIGUSR1 myapp    # Custom signal
```

💡 **Best Practice:** Always use `docker stop` first. Only use `docker kill` if the container is stuck and doesn't respond to `docker stop`. Your app needs to handle SIGTERM properly for graceful shutdown (close connections, finish ongoing requests, flush data).

### Q11. How do you check container logs? What are logging drivers?

**Answer:**

```bash
# Basic log viewing
docker logs myapp                    # All logs
docker logs -f myapp                 # Follow (real-time) — like tail -f
docker logs --tail 50 myapp          # Last 50 lines
docker logs --since "2024-01-01" myapp  # Logs since a date
docker logs --since 30m myapp        # Logs from last 30 minutes
docker logs --until "2024-01-02" myapp  # Logs until a date

# Combine flags
docker logs -f --tail 100 myapp     # Follow, starting from last 100 lines
```

**Logging Drivers — where should container logs go?**

| Driver | Where Logs Go | Use Case |
|--------|-------------|----------|
| `json-file` (default) | JSON files on host (`/var/lib/docker/containers/`) | Default, simple |
| `syslog` | System syslog | Centralized logging with syslog |
| `fluentd` | Fluentd collector | Kubernetes / EFK stack |
| `awslogs` | AWS CloudWatch | AWS environments |
| `gcplogs` | Google Cloud Logging | GCP environments |
| `splunk` | Splunk | Enterprise logging |
| `none` | Nowhere (logs disabled) | When you handle logging inside the app |

```bash
# Run container with specific logging driver
docker run -d --log-driver=syslog nginx

# Set log driver globally in /etc/docker/daemon.json
{
    "log-driver": "json-file",
    "log-opts": {
        "max-size": "10m",     # Max log file size
        "max-file": "3"         # Keep 3 rotated files
    }
}

# ⚠️ IMPORTANT: Without log rotation, logs can fill up your disk!
# Always set max-size and max-file in production
```

### Q12. What is Docker Layer Caching? How do you optimize Dockerfiles?

**Answer:**

Each instruction in a Dockerfile creates a **layer**. Docker caches these layers and reuses them if nothing changed — making subsequent builds much faster.

```dockerfile
# ❌ BAD — cache is busted every time you change any code
FROM python:3.11-slim
COPY . .                        # Layer 1: Copy EVERYTHING
RUN pip install -r requirements.txt  # Layer 2: Install deps
# Problem: Any code change invalidates the COPY layer,
# so pip install runs EVERY TIME even if dependencies didn't change!

# ✅ GOOD — dependencies are cached separately
FROM python:3.11-slim
COPY requirements.txt .          # Layer 1: Copy ONLY requirements
RUN pip install -r requirements.txt  # Layer 2: Install deps (CACHED if requirements.txt unchanged)
COPY . .                        # Layer 3: Copy application code
# Now: Code changes only invalidate Layer 3. Dependencies stay cached!
```

**How caching works:**

```
Instruction 1: FROM python:3.11   → Check cache → Cached? Use it! ✅
Instruction 2: COPY req.txt .     → Check cache → req.txt unchanged? Use cache! ✅
Instruction 3: RUN pip install    → Check cache → Previous layer cached? Use cache! ✅
Instruction 4: COPY . .           → Check cache → Code changed? REBUILD! 🔄
                                     ↓
                                   All subsequent layers are rebuilt too
```

**More optimization tips:**

```dockerfile
# ✅ Combine RUN commands to reduce layers
# ❌ BAD:
RUN apt-get update
RUN apt-get install -y curl
RUN apt-get install -y wget
RUN rm -rf /var/lib/apt/lists/*

# ✅ GOOD:
RUN apt-get update && \
    apt-get install -y curl wget && \
    rm -rf /var/lib/apt/lists/*

# ✅ Use .dockerignore to exclude unnecessary files
# .dockerignore:
node_modules
.git
*.md
.env
docker-compose*.yml
```

### Q13. What is a Multi-Stage Build? Why is it important?

**Answer:**

A **multi-stage build** uses multiple `FROM` statements in one Dockerfile. Each `FROM` starts a new stage. You can copy artifacts from one stage to another, leaving behind all the build tools and intermediate files.

**The Problem:**
```
Build image with Go compiler: 1.2 GB 😱
Your actual Go binary:         10 MB
→ You're shipping 1.19 GB of JUNK with your app!
```

**The Solution — Multi-Stage Build:**

```dockerfile
# ═══════ Stage 1: BUILD (has all build tools) ═══════
FROM golang:1.21 AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -o myapp .

# ═══════ Stage 2: RUNTIME (minimal image) ═══════
FROM alpine:3.18
RUN apk --no-cache add ca-certificates
WORKDIR /app
COPY --from=builder /app/myapp .
EXPOSE 8080
CMD ["./myapp"]

# Stage 1 image: ~1.2 GB (has Go compiler, source code, etc.)
# Stage 2 image: ~15 MB (just Alpine + your binary!) 🎉
```

**Another example — Node.js:**

```dockerfile
# Stage 1: Build
FROM node:20 AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Stage 2: Production
FROM node:20-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY package*.json ./
EXPOSE 3000
CMD ["node", "dist/index.js"]
```

**Benefits of multi-stage builds:**

| Benefit | Explanation |
|---------|-------------|
| **Smaller images** | Final image only has what's needed to RUN the app |
| **Better security** | No compilers, build tools, or source code in production image |
| **Faster deployments** | Smaller images = faster push/pull |
| **Single Dockerfile** | No need for separate build and runtime Dockerfiles |

### Q14. What are Distroless images? When should you use them?

**Answer:**

**Distroless images** contain ONLY your application and its runtime dependencies — no package manager, no shell, no OS utilities. Created by Google.

```
Regular image (Ubuntu):     ~80 MB + app  → has bash, apt, curl, etc.
Alpine image:               ~5 MB + app   → has sh, apk, basic utils
Distroless image:           ~2 MB + app   → has NOTHING except your app
Scratch image:              0 MB + app    → completely empty (for static binaries)
```

```dockerfile
# Using distroless for a Go app
FROM golang:1.21 AS builder
WORKDIR /app
COPY . .
RUN CGO_ENABLED=0 go build -o myapp .

FROM gcr.io/distroless/static:nonroot
COPY --from=builder /app/myapp /
USER nonroot:nonroot
ENTRYPOINT ["/myapp"]

# Using distroless for a Java app
FROM maven:3.9 AS builder
WORKDIR /app
COPY . .
RUN mvn package -DskipTests

FROM gcr.io/distroless/java17-debian11
COPY --from=builder /app/target/myapp.jar /app.jar
CMD ["app.jar"]
```

**When to use what:**

| Image | Size | Shell? | Use Case |
|-------|------|--------|----------|
| Ubuntu/Debian | 80-200 MB | Yes | Development, debugging |
| Alpine | 5-8 MB | Yes (sh) | Production (good balance) |
| Distroless | 2-20 MB | No | Production (high security) |
| Scratch | 0 MB | No | Static binaries (Go, Rust) |

💡 **Tradeoff:** Distroless images can't be `docker exec`-ed into (no shell!), making debugging harder. Use them in production but have a way to debug (like ephemeral containers in K8s).

### Q15. What are Docker network types? Explain each one.

**Answer:**

```bash
# List all networks
docker network ls
# NETWORK ID     NAME      DRIVER    SCOPE
# abc123         bridge    bridge    local
# def456         host      host      local
# ghi789         none      null      local
```

| Network | How It Works | Use Case |
|---------|-------------|----------|
| **bridge** (default) | Creates an isolated network. Containers get private IPs | Default for single-host apps |
| **host** | Container shares the host's network stack directly | Maximum performance (no NAT overhead) |
| **none** | No networking at all | Security-sensitive containers |
| **overlay** | Spans multiple Docker hosts | Docker Swarm, multi-host |
| **macvlan** | Container gets its own MAC address on physical network | Legacy apps needing direct LAN access |

```bash
# Bridge Network (default)
docker run -d --name app1 nginx
docker run -d --name app2 nginx
# app1 and app2 can reach each other by IP (but NOT by name on default bridge)

# Custom Bridge Network (recommended — supports DNS name resolution!)
docker network create my-network
docker run -d --name api --network my-network my-api
docker run -d --name db --network my-network postgres
```

### Q16. How does Docker port mapping work?

**Answer:**

Port mapping connects a port on the **host** to a port on the **container**, allowing external access.

```bash
# Syntax: -p HOST_PORT:CONTAINER_PORT
docker run -d -p 8080:80 nginx
# Host port 8080 → Container port 80
# Access via: http://localhost:8080

# Map to a specific host interface
docker run -d -p 127.0.0.1:8080:80 nginx
# Only accessible from localhost (not from other machines)

# Map multiple ports
docker run -d -p 8080:80 -p 8443:443 nginx

# Random host port (Docker picks one)
docker run -d -p 80 nginx
docker port <container-id>
# 80/tcp -> 0.0.0.0:32768

# UDP port mapping
docker run -d -p 5353:53/udp dns-server

# Expose ALL ports defined in the image
docker run -d -P nginx    # Capital P = publish all exposed ports
```

```
Port Mapping Visualization:

External Request → localhost:8080
                        │
                   ┌────┴────┐
                   │  Host   │
                   │  :8080  │   ← iptables NAT rule
                   └────┬────┘
                        │
                   ┌────┴────┐
                   │Container│
                   │  :80    │   ← Nginx listening here
                   └─────────┘
```

💡 **Common mistake:** `EXPOSE` in a Dockerfile does NOT publish a port! It's just documentation. You MUST use `-p` flag when running to actually map ports.

## 💾 Section 5: Docker Volumes & Storage

---

### Q17. What are Docker Volumes? Explain the three types of mounts.

**Answer:**

Containers are **ephemeral** — when you remove a container, all data inside it is lost. Volumes solve this by providing **persistent storage**.

| Mount Type | What It Is | Data Location | Managed By |
|-----------|-----------|---------------|------------|
| **Volume** | Docker-managed storage | `/var/lib/docker/volumes/` | Docker |
| **Bind Mount** | Maps a host directory to container | Anywhere on host | You |
| **tmpfs** | In-memory storage (RAM) | Memory (never on disk) | Kernel |

```bash
# ═══════ Docker Volumes (Recommended) ═══════
# Create a volume
docker volume create mydata

# Run container with volume
docker run -d -v mydata:/app/data nginx
# Data persists in 'mydata' even if container is removed

# List volumes
docker volume ls

# Inspect a volume
docker volume inspect mydata

# Remove a volume
docker volume rm mydata

# Remove unused volumes
docker volume prune

# ═══════ Bind Mounts (Host directory) ═══════
# Mount current directory into container
docker run -d -v $(pwd)/config:/app/config nginx
# or using --mount (more explicit):
docker run -d --mount type=bind,source=$(pwd)/config,target=/app/config nginx

# Read-only bind mount
docker run -d -v $(pwd)/config:/app/config:ro nginx

# ═══════ tmpfs Mounts (In-memory) ═══════
docker run -d --tmpfs /app/temp nginx
# Data stored in RAM — fast but lost when container stops
# Use case: temporary files, secrets in memory
```

**When to use what:**

| Scenario | Best Choice |
|----------|------------|
| Database data (MySQL, PostgreSQL) | Docker Volume |
| Share code between host and container (development) | Bind Mount |
| Config files | Bind Mount (read-only) |
| Temporary sensitive data | tmpfs |
| Data shared between containers | Docker Volume |

### Q18. What happens to data when a container is removed?

**Answer:**

```
Container with NO volumes:
┌─────────────────┐
│    Container     │
│  Writable Layer  │  ← DATA IS HERE
│  (ephemeral)     │  
├─────────────────┤
│   Image Layers   │  ← Read-only
│   (shared)       │
└─────────────────┘
docker rm container → Writable layer DELETED → DATA LOST! 💀

Container WITH volume:
┌─────────────────┐     ┌──────────────┐
│    Container     │     │   Volume     │
│  Writable Layer  │────→│  /app/data   │  ← DATA IS HERE
│  (ephemeral)     │     │  (persistent)│
├─────────────────┤     └──────────────┘
│   Image Layers   │
└─────────────────┘
docker rm container → Volume SURVIVES → DATA SAFE! ✅
```

```bash
# Prove it:
# Without volume — data is lost
docker run --name test1 alpine sh -c "echo 'hello' > /data.txt"
docker rm test1
# /data.txt is GONE

# With volume — data persists
docker run --name test2 -v mydata:/data alpine sh -c "echo 'hello' > /data/data.txt"
docker rm test2
docker run --rm -v mydata:/data alpine cat /data/data.txt
# Output: hello ← Data survived!
```

## 🐳 Section 6: Docker Compose

### Q19. What is Docker Compose? Why is it used?

**Answer:**

**Docker Compose** is a tool for defining and running **multi-container applications** using a single YAML file.

**Without Compose (pain!):**
```bash
docker network create myapp
docker run -d --name db --network myapp -e POSTGRES_PASSWORD=secret postgres:15
docker run -d --name redis --network myapp redis:7
docker run -d --name api --network myapp -p 8080:3000 -e DB_HOST=db -e REDIS_HOST=redis myapi
docker run -d --name frontend --network myapp -p 80:80 myfrontend
# Imagine typing all this every time... 😩
```

**With Compose (clean!):**
```yaml
# docker-compose.yml
version: '3.8'

services:
  db:
    image: postgres:15
    environment:
      POSTGRES_PASSWORD: secret
      POSTGRES_DB: myapp
    volumes:
      - db-data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 10s
      timeout: 5s
      retries: 5

  redis:
    image: redis:7-alpine
    volumes:
      - redis-data:/data

  api:
    build: ./backend
    ports:
      - "8080:3000"
    environment:
      DB_HOST: db
      REDIS_HOST: redis
    depends_on:
      db:
        condition: service_healthy
      redis:
        condition: service_started

  frontend:
    build: ./frontend
    ports:
      - "80:80"
    depends_on:
      - api

volumes:
  db-data:
  redis-data:
```

```bash
# Start everything
docker compose up -d

# Stop everything
docker compose down

# Stop and remove volumes too
docker compose down -v

# View logs
docker compose logs -f api

# Rebuild images
docker compose build
docker compose up -d --build

# Scale a service
docker compose up -d --scale api=3

# List running services
docker compose ps
```

### Q20. What is the difference between `depends_on` and `healthcheck` in Docker Compose?

**Answer:**

| Feature | `depends_on` (basic) | `depends_on` + `healthcheck` |
|---------|---------------------|------------------------------|
| **Waits for** | Container to START | Container to be HEALTHY |
| **Guarantees app is ready?** | ❌ No — container started ≠ app is ready | ✅ Yes — confirmed the app is responding |
| **Example** | MySQL container started, but MySQL server isn't ready yet | MySQL container started AND responding to queries |

```yaml
services:
  db:
    image: postgres:15
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 10s      # Check every 10 seconds
      timeout: 5s         # Timeout after 5 seconds
      retries: 5          # Retry 5 times before marking unhealthy
      start_period: 30s   # Wait 30s before first check

  api:
    build: .
    depends_on:
      db:
        condition: service_healthy    # ← Wait until db is HEALTHY!
```

💡 **Why this matters:** Without healthcheck, your API might start before the database is ready, causing connection errors on startup. With healthcheck, Compose waits until the database is actually accepting connections.

### Q21. What are Docker security best practices?

**Answer:**

```dockerfile
# ═══════ 1. Don't run as root ═══════
# ❌ BAD: Container runs as root (default)
FROM ubuntu
RUN apt-get update && apt-get install -y python3
CMD ["python3", "app.py"]

# ✅ GOOD: Create and use a non-root user
FROM ubuntu
RUN apt-get update && apt-get install -y python3 && \
    adduser --disabled-password --gecos "" appuser
USER appuser
CMD ["python3", "app.py"]

# ═══════ 2. Use specific image tags (not :latest) ═══════
# ❌ BAD:
FROM python:latest      # Could change anytime!

# ✅ GOOD:
FROM python:3.11.7-slim  # Pinned and minimal

# ═══════ 3. Use multi-stage builds ═══════
# Don't ship build tools in production images

# ═══════ 4. Scan images for vulnerabilities ═══════
# Using Trivy:
# trivy image myapp:v1

# Using Docker Scout:
# docker scout cves myapp:v1

# ═══════ 5. Don't store secrets in images ═══════
# ❌ BAD:
ENV DATABASE_PASSWORD=supersecret

# ✅ GOOD: Pass secrets at runtime
# docker run -e DATABASE_PASSWORD=$DB_PASS myapp
# Or use Docker secrets / external secret managers

# ═══════ 6. Use .dockerignore ═══════
# .dockerignore:
.git
.env
node_modules
*.md
docker-compose.yml
```

**More security practices:**

| Practice | Why |
|----------|-----|
| Use minimal base images (Alpine, distroless) | Fewer packages = fewer vulnerabilities |
| Set `--read-only` flag | Prevents writing to filesystem |
| Drop Linux capabilities | Remove unneeded privileges |
| Set resource limits | Prevent container from hogging all resources |
| Use `--no-new-privileges` | Prevent privilege escalation |
| Scan images in CI/CD | Catch vulnerabilities before deployment |

```bash
# Run with security flags
docker run -d \
  --read-only \                          # Read-only filesystem
  --tmpfs /tmp \                          # Writable tmp directory
  --security-opt=no-new-privileges \      # No privilege escalation
  --cap-drop=ALL \                        # Drop all capabilities
  --cap-add=NET_BIND_SERVICE \            # Add only what's needed
  --memory=512m \                         # Limit memory
  --cpus=1.0 \                            # Limit CPU
  --user 1000:1000 \                      # Run as non-root
  myapp:v1
```

### Q22. What is the difference between Docker secrets and environment variables?

**Answer:**

| Feature | Environment Variables | Docker Secrets |
|---------|----------------------|---------------|
| **Visibility** | Visible in `docker inspect`, `docker exec env`, `/proc/*/environ` | Only available as files inside the container |
| **Storage** | Stored in clear text | Encrypted at rest (in Swarm) |
| **Scope** | Any container | Swarm services only (or Compose with files) |
| **Security** | ❌ Not secure for sensitive data | ✅ Designed for sensitive data |

```bash
# Environment variables — visible everywhere (NOT for secrets!)
docker run -e DB_PASSWORD=secret123 myapp
docker inspect myapp | grep DB_PASSWORD
# Output: "DB_PASSWORD=secret123" ← Anyone can see this! 😱

# Docker secrets (Swarm mode)
echo "supersecretpassword" | docker secret create db_password -

docker service create --name api \
  --secret db_password \
  myapp

# Inside the container, the secret is available as a file:
# /run/secrets/db_password

# In Docker Compose (using files):
# docker-compose.yml
services:
  api:
    image: myapp
    secrets:
      - db_password

secrets:
  db_password:
    file: ./secrets/db_password.txt
```

💡 **In production (Kubernetes):** Use Kubernetes Secrets (or better: HashiCorp Vault, AWS Secrets Manager) instead of Docker secrets. Environment variables are fine for non-sensitive config like `NODE_ENV=production`.

### Q23. How do Docker image layers work?

**Answer:**

A Docker image is made up of **stacked read-only layers**. Each instruction in the Dockerfile creates one layer.

```dockerfile
FROM ubuntu:22.04          # Layer 1: Base Ubuntu image
RUN apt-get update         # Layer 2: Package list update
RUN apt-get install nginx  # Layer 3: Nginx installed
COPY app/ /var/www/html/   # Layer 4: Your application code
CMD ["nginx", "-g", "daemon off;"]  # Layer 5: No new layer (metadata only)
```

```
Image Layers (Read-Only):
┌─────────────────────────┐
│ Layer 5: CMD metadata   │  ← No filesystem change
├─────────────────────────┤
│ Layer 4: COPY app/      │  ← Your app files
├─────────────────────────┤
│ Layer 3: Install nginx  │  ← Nginx binary + config
├─────────────────────────┤
│ Layer 2: apt-get update │  ← Package lists
├─────────────────────────┤
│ Layer 1: Ubuntu base    │  ← Base OS files
└─────────────────────────┘

When you run the container:
┌─────────────────────────┐
│ Writable Layer (R/W)    │  ← Container-specific changes
├─────────────────────────┤
│ Layer 5: CMD            │  ↑
│ Layer 4: COPY           │  │ All read-only
│ Layer 3: nginx          │  │ Shared between containers
│ Layer 2: apt-get        │  │ using same image!
│ Layer 1: Ubuntu         │  ↓
└─────────────────────────┘
```

```bash
# See layers of an image
docker history nginx
# IMAGE          CREATED       SIZE    COMMAND
# a8758716bb6a   2 days ago    0B      CMD ["nginx" "-g" "daemon off;"]
# <missing>      2 days ago    61.1MB  # nginx installed
# <missing>      2 days ago    80.4MB  # ubuntu base

# See detailed layer info
docker inspect nginx | jq '.[0].RootFS.Layers'
```

**Key insights:**
- Layers are **shared** between images and containers (saves disk space)
- Layers are **cached** — unchanged layers don't need to be rebuilt
- Only `RUN`, `COPY`, and `ADD` create new layers with filesystem changes
- `CMD`, `ENV`, `EXPOSE`, `LABEL` only add metadata (no new layer)

### Q24. How do you reduce Docker image size?

**Answer:**

```dockerfile
# ═══════ Technique 1: Use smaller base images ═══════
# ❌ Ubuntu: ~80MB
FROM ubuntu:22.04

# ✅ Alpine: ~5MB
FROM python:3.11-alpine

# ✅✅ Distroless: ~2MB
FROM gcr.io/distroless/python3

# ═══════ Technique 2: Multi-stage builds ═══════
FROM node:20 AS builder
WORKDIR /app
COPY . .
RUN npm ci && npm run build

FROM node:20-alpine
COPY --from=builder /app/dist ./dist
CMD ["node", "dist/index.js"]

# ═══════ Technique 3: Minimize layers and clean up ═══════
# ❌ BAD (3 layers, cache not cleaned):
RUN apt-get update
RUN apt-get install -y curl wget
RUN apt-get clean

# ✅ GOOD (1 layer, cache cleaned):
RUN apt-get update && \
    apt-get install -y --no-install-recommends curl wget && \
    rm -rf /var/lib/apt/lists/*

# ═══════ Technique 4: Use .dockerignore ═══════
# .dockerignore:
node_modules
.git
*.md
.env
test/
docs/

# ═══════ Technique 5: Only install what you need ═══════
# ❌ BAD:
RUN pip install -r requirements.txt

# ✅ GOOD:
RUN pip install --no-cache-dir -r requirements.txt
# --no-cache-dir prevents pip from caching downloaded packages
```

**Size comparison example:**

```
python:3.11             →  920 MB
python:3.11-slim        →  130 MB
python:3.11-alpine      →   50 MB
gcr.io/distroless/python3 → 52 MB
```

### Q25. What is the difference between `docker save/load` and `docker export/import`?

**Answer:**

| Command | Works With | Preserves | Format | Use Case |
|---------|-----------|-----------|--------|----------|
| `docker save` | Images | All layers, tags, metadata | tar | Transfer images between machines |
| `docker load` | Images | All layers, tags, metadata | tar | Load saved images |
| `docker export` | Containers | Single flat filesystem | tar | Backup a container's filesystem |
| `docker import` | Creates Image | Single layer, no metadata | tar | Create image from exported filesystem |

```bash
# Save an image to a file (preserves layers, history, tags)
docker save -o nginx.tar nginx:latest
# Transfer this file to another machine

# Load the image on another machine
docker load -i nginx.tar

# Export a container's filesystem (flat, no layers)
docker export mycontainer > container.tar

# Import as a new image (single layer)
docker import container.tar my-imported-image:v1
```

💡 **When to use which:**
- `save/load` → Moving images between disconnected environments (air-gapped networks)
- `export/import` → Creating a snapshot of a container's current state
- In most cases, use a **registry** (Docker Hub, ECR) instead — it's much better for image distribution.

### Q26. What is Docker BuildKit? What are its advantages?

**Answer:**

**BuildKit** is Docker's modern, improved build engine. It replaces the old builder with a much faster and more feature-rich one.

```bash
# Enable BuildKit
export DOCKER_BUILDKIT=1
docker build -t myapp .

# Or set it permanently in /etc/docker/daemon.json:
{
    "features": {
        "buildkit": true
    }
}

# BuildKit is the DEFAULT in Docker 23.0+
```

**Advantages of BuildKit:**

| Feature | Old Builder | BuildKit |
|---------|------------|----------|
| **Parallel builds** | Sequential stages | ✅ Parallel independent stages |
| **Cache management** | Basic layer cache | ✅ Advanced cache (registry, local) |
| **Build secrets** | ❌ None | ✅ `--secret` flag (secrets not stored in layers) |
| **SSH forwarding** | ❌ None | ✅ `--ssh` flag (for private repo access) |
| **Multi-platform** | ❌ No | ✅ `docker buildx build --platform linux/amd64,linux/arm64` |
| **Progress output** | All output | ✅ Clean, structured output |

```dockerfile
# BuildKit secret mounting (secret is NOT stored in the image!)
RUN --mount=type=secret,id=npm_token \
    NPM_TOKEN=$(cat /run/secrets/npm_token) \
    npm install

# Build with secret:
docker build --secret id=npm_token,src=.npm_token -t myapp .

# BuildKit cache mounting (speeds up dependency installs)
RUN --mount=type=cache,target=/root/.cache/pip \
    pip install -r requirements.txt
```

### Q27. What is `docker buildx`? How do you build multi-platform images?

**Answer:**

`docker buildx` is an extended build command that supports **multi-platform builds** — building the same image for different CPU architectures (AMD64, ARM64, etc.).

```bash
# Create a new builder instance
docker buildx create --name mybuilder --use

# Build for multiple platforms
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  -t myregistry/myapp:v1 \
  --push \
  .

# This creates ONE image tag that works on both:
# - x86_64 machines (most servers, Intel/AMD)
# - ARM64 machines (AWS Graviton, Apple M1/M2, Raspberry Pi)

# List available platforms
docker buildx ls

# Inspect the builder
docker buildx inspect mybuilder
```

💡 **Why this matters:** AWS Graviton (ARM) instances are 20-40% cheaper than x86. Multi-platform builds let you run the same image on both, choosing the cheapest option.

### Q31. How do you debug a container that keeps crashing (CrashLoop)?

**Answer:**

```bash
# Step 1: Check container status and exit code
docker ps -a
# STATUS: Exited (1) 5 seconds ago → non-zero exit code means error

# Step 2: Check container logs
docker logs myapp
docker logs --tail 50 myapp    # Last 50 lines
# This is usually where you find the error!

# Step 3: If logs aren't helpful, run interactively
docker run -it myapp /bin/sh
# or
docker run -it --entrypoint /bin/sh myapp
# Now you're inside the container — investigate!

# Step 4: Check if the process is even starting
docker inspect myapp | jq '.[0].State'
# Look at: ExitCode, Error, OOMKilled

# Step 5: Check resource limits
docker stats myapp
# Is it running out of memory? (OOMKilled)

# Step 6: Check if it's a permission issue
docker run -it myapp ls -la /app/
# Are files owned by the right user?

# Step 7: Check if dependencies are reachable
docker exec -it myapp sh
ping db           # Can it reach the database?
nc -zv db 5432    # Is the port open?
curl http://api   # Can it reach the API?
```

**Common causes and fixes:**

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| Exit code 1 | Application error | Check logs, fix the code |
| Exit code 137 | OOM Killed (out of memory) | Increase `--memory` limit |
| Exit code 126 | Permission denied on command | Check file permissions, `chmod +x` |
| Exit code 127 | Command not found | Check `CMD`/`ENTRYPOINT`, binary path |
| Exit code 0 but restarts | Container has no foreground process | Use `CMD` that runs in foreground |

### Q34. How do you set resource limits on containers?

**Answer:**

```bash
# Limit memory
docker run -d --memory=512m nginx         # Hard limit: 512MB

  --cpus=0.5 \
  --name limited-app \

    image: myapp
    deploy:
      resources:
        limits:
          cpus: '1.0'
          memory: 512M
        reservations:
          cpus: '0.5'
          memory: 256M
```

💡 **What happens when limits are exceeded?**
- **Memory limit exceeded:** Container is killed by OOM Killer (exit code 137)
- **CPU limit exceeded:** Container is throttled (slowed down, not killed)

### Q35. What is Docker-in-Docker (DinD)? When would you use it?

**Answer:**

**Docker-in-Docker** means running Docker inside a Docker container. This is used when your CI/CD pipeline needs to build Docker images inside a container.

```
Without DinD:
┌──── Host ──────────────────────┐
│                                 │
│  ┌── Jenkins Container ──┐     │
│  │                        │     │
│  │  Can't build Docker    │     │
│  │  images! No Docker     │     │
│  │  inside here.          │     │
│  └────────────────────────┘     │
│                                 │
│  Docker Daemon                  │
└─────────────────────────────────┘

With DinD (Method 1: Docker Socket mounting):
┌──── Host ──────────────────────┐
│                                 │
│  ┌── Jenkins Container ──┐     │
│  │                        │     │
│  │  docker build works!  │─────┼── /var/run/docker.sock (mounted from host)
│  │  Uses HOST's Docker   │     │
│  │  daemon                │     │
│  └────────────────────────┘     │
│                                 │
│  Docker Daemon                  │
└─────────────────────────────────┘

With DinD (Method 2: Actual DinD):
┌──── Host ──────────────────────┐
│                                 │
│  ┌── Jenkins Container ──┐     │
│  │                        │     │
│  │  ┌── Docker Daemon ──┐│     │
│  │  │  Has its OWN       ││     │   ← Runs its own Docker daemon
│  │  │  Docker daemon     ││     │      inside the container!
│  │  └────────────────────┘│     │
│  └────────────────────────┘     │
│                                 │
│  Docker Daemon (host)           │
└─────────────────────────────────┘
```

**Method 1: Mount Docker socket (simpler, less secure)**
```bash
docker run -v /var/run/docker.sock:/var/run/docker.sock jenkins/jenkins
# ⚠️ Security risk: container has full access to host's Docker daemon
```

**Method 2: Actual DinD (isolated)**
```bash
docker run --privileged -d docker:dind
# ⚠️ Requires --privileged mode (security concern)
```

| Method | Pros | Cons |
|--------|------|------|
| Socket mounting | Simple, fast, uses cache | Security risk — full host access |
| True DinD | Isolated | Needs `--privileged`, performance overhead |
| Kaniko | No Docker daemon needed | Google's tool, works in K8s, secure |
| Buildah | Rootless builds | No daemon needed, OCI compliant |

💡 **In Kubernetes CI/CD:** Use **Kaniko** or **Buildah** instead of DinD — they don't need a Docker daemon or privileged mode.

### Q36. What is the difference between `docker run --restart` policies?

**Answer:**

| Policy | Behavior |
|--------|----------|
| `no` (default) | Never restart automatically |
| `always` | Always restart, no matter what (even after daemon restart) |
| `on-failure` | Restart only if exit code is non-zero (error) |
| `on-failure:5` | Restart on failure, max 5 retries |
| `unless-stopped` | Like `always`, but doesn't restart if you manually stopped it |

```bash
# Always restart (good for production services)
docker run -d --restart=always --name web nginx

# Restart only on failure (max 3 attempts)
docker run -d --restart=on-failure:3 --name worker myworker

# Unless manually stopped
docker run -d --restart=unless-stopped --name api myapi

# Check restart policy of a running container
docker inspect myapp --format '{{.HostConfig.RestartPolicy}}'

# Update restart policy of existing container
docker update --restart=always myapp
```

💡 **Use cases:**
- `always` → Web servers, APIs, databases in production
- `on-failure:3` → Workers, batch jobs that might fail
- `unless-stopped` → Services you might need to manually stop for maintenance

### Q37. How do you tag and version Docker images properly?

**Answer:**

```bash
# Basic tagging
docker build -t myapp:v1.0.0 .
docker build -t myapp:latest .

# Multiple tags for the same image
docker tag myapp:v1.0.0 myapp:latest
docker tag myapp:v1.0.0 myapp:v1.0
docker tag myapp:v1.0.0 myapp:v1

# With registry prefix
docker tag myapp:v1.0.0 myregistry.com/myapp:v1.0.0

# CI/CD tagging strategies
docker build -t myapp:${GIT_COMMIT_SHA} .         # Git commit hash
docker build -t myapp:${BUILD_NUMBER} .             # Build number
docker build -t myapp:${GIT_TAG} .                  # Git release tag
docker build -t myapp:${BRANCH_NAME}-${BUILD_ID} .  # Branch + build
```

**Recommended tagging strategy:**

```bash
# For every build, create multiple tags:
IMAGE=myregistry.com/myapp

docker build -t ${IMAGE}:${GIT_SHA} .           # Immutable, unique
docker tag ${IMAGE}:${GIT_SHA} ${IMAGE}:v1.2.3   # Semantic version
docker tag ${IMAGE}:${GIT_SHA} ${IMAGE}:latest    # Latest (for convenience)

docker push ${IMAGE}:${GIT_SHA}
docker push ${IMAGE}:v1.2.3
docker push ${IMAGE}:latest
```

💡 **NEVER use `latest` in production deployments.** Always use specific version tags or SHA digests. `latest` can change at any time and you won't know what version you're running.

### Q40. How do you handle Docker image vulnerability scanning in CI/CD?

**Answer:**

```bash
# ═══════ Trivy (most popular, free) ═══════
# Scan an image
trivy image myapp:v1

# Scan with severity filter
trivy image --severity HIGH,CRITICAL myapp:v1

# Fail if critical vulnerabilities found (for CI/CD)
trivy image --exit-code 1 --severity CRITICAL myapp:v1

# Scan a Dockerfile
trivy config ./Dockerfile

# ═══════ Docker Scout (Docker's built-in) ═══════
docker scout cves myapp:v1
docker scout recommendations myapp:v1

# ═══════ Grype ═══════
grype myapp:v1
```

**In a GitHub Actions CI/CD pipeline:**
```yaml
# .github/workflows/build.yml
jobs:
  build-and-scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Build Docker image
        run: docker build -t myapp:${{ github.sha }} .
      
      - name: Run Trivy vulnerability scanner
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: myapp:${{ github.sha }}
          format: 'table'
          exit-code: '1'                    # Fail pipeline if vulnerabilities found
          severity: 'CRITICAL,HIGH'
          ignore-unfixed: true              # Ignore vulnerabilities without fixes
      
      - name: Push image (only if scan passes)
        if: success()
        run: |
          docker tag myapp:${{ github.sha }} myregistry/myapp:${{ github.sha }}
          docker push myregistry/myapp:${{ github.sha }}
```

### Q45. How do you update a running container's image with zero downtime?

**Answer:**

Docker alone doesn't support zero-downtime updates. You need either Docker Compose or an orchestrator:

```bash
# Method 1: Docker Compose (rolling update)
docker compose build api          # Build new image
docker compose up -d --no-deps api  # Recreate only the api service

```

💡 **For true zero-downtime deployments**, use Kubernetes, Docker Swarm, or a load balancer in front of multiple containers.

### Q46. What is Docker overlay filesystem?

**Answer:** Docker uses a **layered filesystem** (overlay2 by default) where each image layer is stacked on top of the previous one using a union mount.

```
┌─── Writable Layer (Container) ───┐   ← Changes go here
├──────────────────────────────────┤
│ Layer 4: COPY app/ → /app/      │   ← Read-only
│ Layer 3: RUN pip install         │   ← Read-only
│ Layer 2: COPY requirements.txt   │   ← Read-only
│ Layer 1: FROM python:3.11-slim   │   ← Read-only
└──────────────────────────────────┘

How overlay2 merges them:
All layers are combined (union mounted) into a single view.
If a file exists in multiple layers, the TOPMOST version wins.
If a container modifies a file, it's copied to the writable layer (Copy-on-Write).
```

```bash
# Check storage driver
docker info | grep "Storage Driver"
# Storage Driver: overlay2

# See layers of a container
docker inspect mycontainer | jq '.[0].GraphDriver'
```

### Q48. What are Docker healthchecks? How do you implement them?

**Answer:**

Healthchecks let Docker know if your application inside the container is actually **working**, not just running.

```dockerfile
# In Dockerfile
HEALTHCHECK --interval=30s --timeout=10s --start-period=5s --retries=3 \
  CMD curl -f http://localhost:3000/health || exit 1
```

```bash
# Check health status
docker inspect --format='{{.State.Health.Status}}' myapp
# Output: healthy | unhealthy | starting

# See health check logs
docker inspect --format='{{json .State.Health}}' myapp | jq

# Override healthcheck at runtime
docker run -d \
  --health-cmd="curl -f http://localhost:3000/health || exit 1" \
  --health-interval=30s \
  --health-timeout=10s \
  --health-retries=3 \
  myapp

# Disable healthcheck
docker run -d --no-healthcheck myapp
```

| Parameter | Meaning | Default |
|-----------|---------|---------|
| `--interval` | Time between health checks | 30s |
| `--timeout` | Max time for check to complete | 30s |
| `--start-period` | Grace period for app to start | 0s |
| `--retries` | Consecutive failures before "unhealthy" | 3 |

### Q50. Explain the complete lifecycle of a Docker container.

**Answer:**

```
┌─── docker create ───┐
│                      │
▼                      │
Created ──── docker start ──→ Running ──── docker pause ──→ Paused
   │                            │  │                          │
   │                            │  │    docker unpause ◄──────┘
   │                            │  │
   │            docker stop ◄───┘  │──── docker restart ──→ Running
   │                │               │
   │                ▼               │
   │             Stopped ──────────┘
   │                │
   │    docker rm ◄─┘
   │        │
   ▼        ▼
   Removed (gone forever)
```

**State transitions:**

| State | Description | Command |
|-------|-------------|---------|
| **Created** | Container exists but hasn't started | `docker create` |
| **Running** | Container is executing | `docker start` or `docker run` |
| **Paused** | Container processes are frozen | `docker pause` |
| **Stopped** | Container has exited (still exists) | `docker stop` |
| **Removed** | Container is deleted | `docker rm` |

```bash
# Full lifecycle example
docker create --name myapp nginx      # Created
docker start myapp                      # Running
docker pause myapp                      # Paused
docker unpause myapp                    # Running again
docker stop myapp                       # Stopped
docker start myapp                      # Running again
docker restart myapp                    # Stopped → Running
docker stop myapp                       # Stopped
docker rm myapp                         # Removed (gone)
```
