## Core Concepts

### Q1. What is Kubernetes, and why do we need it?

**Answer:**

Kubernetes (K8s) is an open-source **container orchestration platform** originally developed by Google. Think of it as a **manager** for your containers.

**Why do we need it?**

Imagine you have 100 containers running across 20 servers. Now ask yourself:
- What happens if a container crashes? → Someone needs to restart it.
- What if traffic spikes? → You need to spin up more containers.
- How do you distribute traffic evenly? → You need load balancing.
- How do you update your app without downtime? → You need rolling updates.

Doing all this **manually** is impossible at scale. Kubernetes does all of this **automatically**.

**Key things Kubernetes does for you:**
- **Self-healing** — If a container crashes, K8s restarts it automatically.
- **Scaling** — It can scale up/down based on load.
- **Load balancing** — It distributes traffic across containers.
- **Rolling updates & rollbacks** — Deploy new versions without downtime.
- **Service discovery** — Containers can find and talk to each other.
- **Secret management** — Manages sensitive data like passwords and API keys.

### Q2. Explain the Kubernetes architecture.

**Answer:**

Kubernetes follows a **master-worker** architecture (now called **control plane** and **worker nodes**).

```
┌──────────────────────────────────────────────┐
│              CONTROL PLANE (Master)          │
│                                              │
│  ┌──────────┐  ┌────────────┐  ┌──────────┐ │
│  │ API      │  │ Controller │  │Scheduler │ │
│  │ Server   │  │ Manager    │  │          │ │
│  └──────────┘  └────────────┘  └──────────┘ │
│  ┌──────────┐  ┌────────────┐               │
│  │   etcd   │  │ Cloud      │               │
│  │          │  │ Controller │               │
│  └──────────┘  └────────────┘               │
└──────────────────────────────────────────────┘
            │
            │  (communicates via API)
            │
┌───────────┴──────────────────────────────────┐
│              WORKER NODES                     │
│                                              │
│  ┌──────────┐  ┌────────────┐  ┌──────────┐ │
│  │ kubelet  │  │ kube-proxy │  │Container │ │
│  │          │  │            │  │ Runtime  │ │
│  └──────────┘  └────────────┘  └──────────┘ │
│  ┌────────────────────────────────────────┐  │
│  │         Pods (your containers)         │  │
│  └────────────────────────────────────────┘  │
└──────────────────────────────────────────────┘
```

**Control Plane Components:**

| Component | What It Does |
|-----------|-------------|
| **API Server** | The "front door" of Kubernetes. Everything talks to the API Server. |
| **etcd** | A key-value store that holds ALL cluster data (the "brain" of K8s). |
| **Scheduler** | Decides which node a new pod should run on. |
| **Controller Manager** | Runs controllers that watch the cluster state and make corrections (e.g., "there should be 3 replicas — there are only 2 — start one more"). |
| **Cloud Controller Manager** | Integrates with cloud providers (AWS, Azure, GCP) for load balancers, storage, etc. |

**Worker Node Components:**

| Component | What It Does |
|-----------|-------------|
| **kubelet** | An agent on each node that makes sure containers are running as expected. |
| **kube-proxy** | Handles networking rules so pods can communicate with each other and the outside world. |
| **Container Runtime** | The software that actually runs containers (containerd, CRI-O). |

### Q3. What is a Pod in Kubernetes?

**Answer:**

A **Pod** is the **smallest deployable unit** in Kubernetes. It's NOT a container — it's a **wrapper around one or more containers**.

Think of a Pod as a **tiny house** where containers live together. Containers in the same Pod:
- Share the same **network** (same IP address)
- Share the same **storage** (volumes)
- Are always **scheduled together** on the same node

**Most common pattern:** One container per Pod (the main app).

**Multi-container Pod patterns:**
- **Sidecar** — Helper container (e.g., log shipper)
- **Ambassador** — Proxy container (e.g., connection to external service)
- **Adapter** — Transforms output format (e.g., reformats logs)

**Example Pod YAML:**

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
  labels:
    app: web
spec:
  containers:
    - name: web-server
      image: nginx:1.25
      ports:
        - containerPort: 80
```

**What each field means:**
- `apiVersion: v1` — The API version (Pods are part of core v1)
- `kind: Pod` — We're creating a Pod
- `metadata.name` — The name of the Pod
- `metadata.labels` — Key-value tags for organizing and selecting Pods
- `spec.containers` — List of containers in the Pod
- `image` — The Docker image to use
- `ports.containerPort` — Port the container listens on

### Q4. What is a Namespace in Kubernetes?

**Answer:**

A **Namespace** is like a **virtual cluster** inside your actual cluster. It's a way to **organize and isolate** resources.

Think of Namespaces like **folders on your computer** — they keep things separated and organized.

**Default Namespaces in every cluster:**

| Namespace | Purpose |
|-----------|---------|
| `default` | Where your stuff goes if you don't specify a namespace |
| `kube-system` | Kubernetes system components (API server, DNS, etc.) |
| `kube-public` | Publicly accessible data (rarely used) |
| `kube-node-lease` | Node heartbeat data |

**Why use Namespaces?**
- **Environment separation** — `dev`, `staging`, `production` in the same cluster
- **Team isolation** — Team A and Team B can have their own namespaces
- **Resource quotas** — Limit CPU/memory per namespace
- **RBAC** — Give different teams access to different namespaces

```bash
# Create a namespace
kubectl create namespace dev

# List all namespaces
kubectl get namespaces

# Deploy something in a specific namespace
kubectl apply -f deployment.yaml -n dev

# Set default namespace (so you don't need -n every time)
kubectl config set-context --current --namespace=dev
```

### Q5. What is the difference between a Pod and a Container?

**Answer:**

| Aspect | Container | Pod |
|--------|-----------|-----|
| **What** | A single running process (e.g., Nginx) | A group of one or more containers |
| **Managed by** | Container runtime (containerd) | Kubernetes |
| **Network** | Has its own network stack (in Docker) | All containers in a Pod share the SAME IP |
| **Storage** | Has its own filesystem | Containers in a Pod can share volumes |
| **Scheduling** | N/A | Kubernetes schedules Pods, not containers |
| **Smallest unit** | Yes (in Docker world) | Yes (in Kubernetes world) |

**Simple analogy:**
- A **container** is a person.
- A **Pod** is an apartment — one or more people (containers) living together, sharing the same address (IP) and kitchen (volumes).

### Q6. What is kubectl? List some commonly used commands.

**Answer:**

`kubectl` (pronounced "kube-control" or "kube-C-T-L") is the **command-line tool** to interact with Kubernetes clusters. It talks to the **API Server**.

**Commonly used commands:**

```bash
# --- Cluster Info ---
kubectl cluster-info                    # Show cluster details
kubectl get nodes                       # List all nodes
kubectl version                         # Show client & server versions

# --- Viewing Resources ---
kubectl get pods                        # List pods in current namespace
kubectl get pods -A                     # List pods in ALL namespaces
kubectl get pods -o wide                # More details (IP, Node)
kubectl get deployments                 # List deployments
kubectl get services                    # List services
kubectl get all                         # List all resources

# --- Describing Resources ---
kubectl describe pod <pod-name>         # Detailed info about a pod
kubectl describe node <node-name>       # Detailed info about a node

# --- Creating & Applying ---
kubectl apply -f manifest.yaml          # Create/update resources from file
kubectl create deployment nginx --image=nginx   # Quick create

# --- Debugging ---
kubectl logs <pod-name>                 # View pod logs
kubectl logs <pod-name> -c <container>  # Logs from specific container
kubectl logs <pod-name> --previous      # Logs from crashed container
kubectl exec -it <pod-name> -- /bin/sh  # Shell into a pod
kubectl port-forward <pod> 8080:80      # Forward port to local machine

# --- Deleting ---
kubectl delete pod <pod-name>           # Delete a pod
kubectl delete -f manifest.yaml         # Delete resources from file

# --- Scaling ---
kubectl scale deployment nginx --replicas=5  # Scale to 5 replicas
```

### Q7. What is etcd? Why is it so important?

**Answer:**

**etcd** is a **distributed key-value store** that acts as the **brain and memory** of the Kubernetes cluster. Every piece of information about your cluster is stored in etcd.

**What's stored in etcd?**
- All cluster configuration
- State of all Pods, Services, Deployments
- Secrets and ConfigMaps
- RBAC policies
- Namespace information

**Why is it important?**
- If etcd goes down → **the entire cluster stops working** (no new deployments, no scaling, nothing)
- If etcd data is lost → **you lose your entire cluster state**

**Key characteristics:**
- Uses the **Raft consensus algorithm** (ensures data consistency across replicas)
- Usually deployed as an **odd number** (3, 5, 7) for quorum
- Supports **watch** functionality — other components can "watch" for changes

**Best practices:**
- Always **backup etcd** regularly
- Run etcd on **dedicated, fast SSD disks**
- Keep etcd cluster size at **3 or 5 nodes**

```bash
# Backup etcd
ETCDCTL_API=3 etcdctl snapshot save /backup/etcd-snapshot.db \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key

# Verify backup
ETCDCTL_API=3 etcdctl snapshot status /backup/etcd-snapshot.db --write-table

# Restore etcd
ETCDCTL_API=3 etcdctl snapshot restore /backup/etcd-snapshot.db \
  --data-dir=/var/lib/etcd-restored
```

### Q8. What is the role of the kube-scheduler?

**Answer:**

The **kube-scheduler** watches for newly created Pods that have no node assigned and **decides which node** they should run on.

**How does it decide?** It goes through two steps:

1. **Filtering** — Removes nodes that DON'T meet the Pod's requirements:
   - Does the node have enough CPU/memory?
   - Does the node match nodeSelector/affinity rules?
   - Does the node have taints that the Pod doesn't tolerate?

2. **Scoring** — Ranks the remaining nodes and picks the best one:
   - Which node has the most resources available?
   - Which node already has the image cached?
   - Which node balances the load best?

**Example:** If a Pod needs 2 CPU and 4GB RAM, and Node A has only 1 CPU free but Node B has 4 CPU free, the scheduler picks Node B.

### Q9. What happens when you run `kubectl apply -f deployment.yaml`?

**Answer:**

This is a GREAT interview question because it tests your understanding of the entire Kubernetes flow:

```
kubectl apply → API Server → etcd → Controller Manager → Scheduler → kubelet → Container
```

**Step-by-step:**

1. **kubectl** reads the YAML file and sends an HTTP POST request to the **API Server**.
2. **API Server** authenticates and authorizes the request (RBAC check).
3. **API Server** validates the YAML (is the schema correct?).
4. **API Server** stores the desired state in **etcd**.
5. **Controller Manager** detects the new Deployment object:
   - Creates a **ReplicaSet** (if it's a Deployment)
   - The ReplicaSet controller creates **Pod** objects
6. **Scheduler** notices Pods with no node assigned → assigns each Pod to a node.
7. **kubelet** on the assigned node detects the new Pod → pulls the container image → starts the container.
8. **kube-proxy** sets up networking rules for the Pod.

The entire process is **declarative** — you tell Kubernetes WHAT you want, not HOW to do it.

### Q10. What is the difference between `kubectl create` and `kubectl apply`?

**Answer:**

| Feature | `kubectl create` | `kubectl apply` |
|---------|-----------------|-----------------|
| **Approach** | Imperative (do this NOW) | Declarative (make it look like THIS) |
| **If resource exists** | ❌ Error: "already exists" | ✅ Updates the resource |
| **If resource doesn't exist** | ✅ Creates it | ✅ Creates it |
| **Tracks changes** | No | Yes (stores last-applied config) |
| **Use case** | Quick one-off creation | Managing resources over time |

```bash
# Imperative — creates, fails if exists
kubectl create deployment nginx --image=nginx

# Declarative — creates if new, updates if exists
kubectl apply -f deployment.yaml
```

**Best practice:** Always use `kubectl apply` for production. It's declarative and idempotent.

### Q11. What is a ReplicaSet?

**Answer:**

A **ReplicaSet** ensures that a **specified number of pod replicas** are running at any given time. It's the controller responsible for **self-healing**.

**How it works:**
- You say: "I want 3 replicas of my app."
- ReplicaSet ensures exactly 3 Pods are running.
- If a Pod crashes → ReplicaSet creates a new one.
- If there are too many Pods → ReplicaSet deletes extras.

```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: my-app-rs
spec:
  replicas: 3                    # I want 3 Pods
  selector:
    matchLabels:
      app: my-app                # Manage Pods with this label
  template:
    metadata:
      labels:
        app: my-app              # Label for the Pods
    spec:
      containers:
        - name: my-app
          image: my-app:v1
```

**Important:** You almost **NEVER create ReplicaSets directly**. You create **Deployments**, which automatically create and manage ReplicaSets for you.

### Q12. What is the difference between a ReplicaSet and a Deployment?

**Answer:**

| Feature | ReplicaSet | Deployment |
|---------|------------|------------|
| **What it does** | Maintains desired number of Pods | Manages ReplicaSets + adds update capabilities |
| **Rolling updates** | ❌ No | ✅ Yes |
| **Rollbacks** | ❌ No | ✅ Yes (keeps history of ReplicaSets) |
| **Update strategy** | None | RollingUpdate or Recreate |
| **Use directly?** | Rarely | Almost always |

**Think of it like this:**
- **ReplicaSet** = "Keep 3 copies running"
- **Deployment** = "Keep 3 copies running" + "Update them smoothly" + "Roll back if something breaks"

```
Deployment
  └── ReplicaSet (current) → Pod1, Pod2, Pod3
  └── ReplicaSet (old)     → (kept for rollback)
```

### Q13. What are Labels and Selectors in Kubernetes?

**Answer:**

**Labels** are **key-value pairs** attached to Kubernetes objects. They're like **tags** or **sticky notes** you put on resources.

**Selectors** are how you **search for** and **group** resources based on their labels.

```yaml
# Adding labels to a Pod
metadata:
  labels:
    app: frontend
    environment: production
    team: payments
```

```bash
# Using selectors to find resources
kubectl get pods -l app=frontend                    # Equality-based
kubectl get pods -l 'environment in (production, staging)'   # Set-based
kubectl get pods -l app=frontend,team=payments      # Multiple labels
```

**Why are they so important?**
- **Services** use selectors to find which Pods to send traffic to
- **Deployments** use selectors to know which Pods they manage
- **Network Policies** use selectors to define rules
- **Node affinity** uses labels to schedule Pods on specific nodes

### Q14. What are Annotations in Kubernetes?

**Answer:**

**Annotations** are like Labels, but they're meant for **non-identifying metadata** — information that isn't used for selecting resources but is still useful.

**Labels** = for selection and grouping (short, structured)
**Annotations** = for storing extra info (can be long, freeform)

```yaml
metadata:
  annotations:
    description: "Main frontend application"
    owner: "team-alpha@company.com"
    kubernetes.io/ingress.class: "nginx"
    prometheus.io/scrape: "true"
    prometheus.io/port: "9090"
```

**Common uses:**
- Build/release info (git commit hash, build timestamp)
- Contact info (team email, Slack channel)
- Tool configuration (Prometheus scraping, Ingress settings)
- Deployment tracking (last deploy timestamp, deployed by)

### Q15. What is the difference between Imperative and Declarative approach in Kubernetes?

**Answer:**

| Aspect | Imperative | Declarative |
|--------|-----------|-------------|
| **How** | You tell K8s **what to do step by step** | You tell K8s **what you want the end result to be** |
| **Command** | `kubectl create`, `kubectl run`, `kubectl expose` | `kubectl apply -f file.yaml` |
| **Trackable** | Hard to track changes | Easy — YAML files in Git |
| **Repeatable** | No — running twice might fail | Yes — idempotent |
| **Best for** | Quick tests, learning | Production, GitOps |

```bash
# Imperative — step by step
kubectl create deployment nginx --image=nginx
kubectl scale deployment nginx --replicas=3
kubectl expose deployment nginx --port=80 --type=LoadBalancer

# Declarative — desired state in YAML
kubectl apply -f deployment.yaml   # Contains all of the above
```

**Golden rule for interviews:** "In production, always use the **declarative** approach because it's trackable, repeatable, and works with GitOps."

## Workloads & Controllers

### Q16. What is a Deployment? How do you create one?

**Answer:**

A **Deployment** is the most common way to deploy applications on Kubernetes. It manages ReplicaSets which manage Pods.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
  labels:
    app: my-app
spec:
  replicas: 3                         # Run 3 instances
  selector:
    matchLabels:
      app: my-app                     # Match Pods with this label
  strategy:
    type: RollingUpdate               # Update strategy
    rollingUpdate:
      maxSurge: 1                     # Max extra Pods during update
      maxUnavailable: 0               # Never go below desired count
  template:                           # Pod template
    metadata:
      labels:
        app: my-app
    spec:
      containers:
        - name: my-app
          image: my-app:v1.0
          ports:
            - containerPort: 8080
          resources:
            requests:
              cpu: "100m"
              memory: "128Mi"
            limits:
              cpu: "500m"
              memory: "512Mi"
          livenessProbe:
            httpGet:
              path: /healthz
              port: 8080
            initialDelaySeconds: 10
            periodSeconds: 5
          readinessProbe:
            httpGet:
              path: /ready
              port: 8080
            initialDelaySeconds: 5
            periodSeconds: 3
```

**Line by line:**
- `replicas: 3` — Keep 3 Pod copies running
- `strategy: RollingUpdate` — Update one by one (no downtime)
- `maxSurge: 1` — During update, allow 1 extra Pod (4 total temporarily)
- `maxUnavailable: 0` — During update, never go below 3 running Pods
- `resources.requests` — Minimum resources guaranteed to the Pod
- `resources.limits` — Maximum resources the Pod can use
- `livenessProbe` — "Is the app alive?" If not → restart the container
- `readinessProbe` — "Is the app ready for traffic?" If not → remove from Service

### Q17. What are rolling updates and rollbacks?

**Answer:**

**Rolling Update** = Gradually replacing old Pods with new Pods, one at a time. Zero downtime!

```
Before:  [v1] [v1] [v1]
Step 1:  [v1] [v1] [v2]  ← One new Pod starts
Step 2:  [v1] [v2] [v2]  ← Another old Pod replaced
Step 3:  [v2] [v2] [v2]  ← All updated!
```

```bash
# Update the image (triggers rolling update)
kubectl set image deployment/my-app my-app=my-app:v2.0

# Watch the rollout progress
kubectl rollout status deployment/my-app

# View rollout history
kubectl rollout history deployment/my-app
```

**Rollback** = Going back to a previous version if the new one breaks.

```bash
# Rollback to the previous version
kubectl rollout undo deployment/my-app

# Rollback to a specific revision
kubectl rollout undo deployment/my-app --to-revision=2
```

### Q18. What is a StatefulSet? When do you use it?

**Answer:**

A **StatefulSet** is like a Deployment, but for **stateful applications** — apps that need **stable identity** and **persistent storage**.

**When to use StatefulSet:**
- Databases (MySQL, PostgreSQL, MongoDB)
- Message queues (Kafka, RabbitMQ)
- Distributed systems (Elasticsearch, ZooKeeper)

**What makes it special compared to Deployments:**

| Feature | Deployment | StatefulSet |
|---------|------------|-------------|
| Pod names | Random (my-app-abc123) | Ordered (my-app-0, my-app-1, my-app-2) |
| Pod creation | All at once | One at a time, in order |
| Pod deletion | Any order | Reverse order (2 → 1 → 0) |
| Storage | Shared or none | Each Pod gets its own persistent volume |
| Network identity | No stable hostname | Stable DNS (my-app-0.my-svc.ns.svc.cluster.local) |

```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: mysql
spec:
  serviceName: "mysql"                # Headless service for DNS
  replicas: 3
  selector:
    matchLabels:
      app: mysql
  template:
    metadata:
      labels:
        app: mysql
    spec:
      containers:
        - name: mysql
          image: mysql:8.0
          ports:
            - containerPort: 3306
          volumeMounts:
            - name: data
              mountPath: /var/lib/mysql
  volumeClaimTemplates:               # Each Pod gets its own PVC
    - metadata:
        name: data
      spec:
        accessModes: ["ReadWriteOnce"]
        resources:
          requests:
            storage: 10Gi
```

### Q19. What is a DaemonSet?

**Answer:**

A **DaemonSet** ensures that **one copy of a Pod runs on every node** (or a subset of nodes).

**Think of it as:** "I need this thing on EVERY server."

**Common use cases:**
- **Log collection** — Fluentd/Fluent Bit on every node
- **Monitoring** — Node Exporter (Prometheus) on every node
- **Networking** — kube-proxy, Calico, Cilium
- **Storage** — CSI drivers

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: fluentd
spec:
  selector:
    matchLabels:
      app: fluentd
  template:
    metadata:
      labels:
        app: fluentd
    spec:
      containers:
        - name: fluentd
          image: fluentd:v1.16
          volumeMounts:
            - name: varlog
              mountPath: /var/log
      volumes:
        - name: varlog
          hostPath:
            path: /var/log
```

**Key behavior:**
- When a new node is added to the cluster → DaemonSet automatically creates a Pod on it
- When a node is removed → the Pod is garbage collected
- You can limit it to specific nodes using `nodeSelector` or `affinity`

### Q20. What are Jobs and CronJobs?

**Answer:**

**Job** = Run a task **once** and make sure it completes successfully.

**CronJob** = Run a Job on a **schedule** (like Linux cron).

**Job example:**

```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: backup-db
spec:
  completions: 1                # Run once
  backoffLimit: 3               # Retry up to 3 times if it fails
  template:
    spec:
      containers:
        - name: backup
          image: backup-tool:latest
          command: ["./backup.sh"]
      restartPolicy: Never      # Don't restart, let the Job controller handle retries
```

**CronJob example:**

```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: nightly-backup
spec:
  schedule: "0 2 * * *"        # Every day at 2 AM
  jobTemplate:
    spec:
      template:
        spec:
          containers:
            - name: backup
              image: backup-tool:latest
              command: ["./backup.sh"]
          restartPolicy: OnFailure
  successfulJobsHistoryLimit: 3   # Keep last 3 successful jobs
  failedJobsHistoryLimit: 1       # Keep last 1 failed job
```

**CronJob schedule format:** `minute hour day-of-month month day-of-week`

```
┌───────────── minute (0 - 59)
│ ┌───────────── hour (0 - 23)
│ │ ┌───────────── day of the month (1 - 31)
│ │ │ ┌───────────── month (1 - 12)
│ │ │ │ ┌───────────── day of the week (0 - 6, Sunday=0)
│ │ │ │ │
* * * * *
```

### Q21. What are Init Containers?

**Answer:**

**Init Containers** are special containers that run **before** the main containers in a Pod start. They run to completion, one at a time, in order.

**Use cases:**
- Wait for a database to be ready before starting the app
- Download config files or secrets
- Run database migrations
- Set up permissions on shared volumes

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  initContainers:
    - name: wait-for-db
      image: busybox
      command: ['sh', '-c', 'until nc -z mysql-service 3306; do echo waiting for db; sleep 2; done']
    - name: download-config
      image: busybox
      command: ['wget', '-O', '/config/app.conf', 'http://config-server/app.conf']
      volumeMounts:
        - name: config
          mountPath: /config
  containers:
    - name: my-app
      image: my-app:v1
      volumeMounts:
        - name: config
          mountPath: /config
  volumes:
    - name: config
      emptyDir: {}
```

**How it flows:**
1. `wait-for-db` init container starts → waits until MySQL is reachable → exits
2. `download-config` init container starts → downloads config file → exits
3. `my-app` main container starts (config file is already there!)

### Q22. What are Probes in Kubernetes? (Liveness, Readiness, Startup)

**Answer:**

Probes are **health checks** that Kubernetes uses to know if your container is working properly.

| Probe | Question It Asks | What Happens If It Fails |
|-------|-----------------|-------------------------|
| **Liveness** | "Is the app alive?" | Container is **restarted** |
| **Readiness** | "Is the app ready to serve traffic?" | Pod is **removed from Service** (no traffic) |
| **Startup** | "Has the app started yet?" | Liveness/Readiness probes are paused until startup succeeds |

**Three ways to check:**

```yaml
# 1. HTTP GET — Most common for web apps
livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
  initialDelaySeconds: 10    # Wait 10s before first check
  periodSeconds: 5           # Check every 5 seconds
  failureThreshold: 3        # Fail after 3 consecutive failures

# 2. TCP Socket — Good for databases
readinessProbe:
  tcpSocket:
    port: 3306
  initialDelaySeconds: 5
  periodSeconds: 10

# 3. Command — Run a command inside the container
livenessProbe:
  exec:
    command:
      - cat
      - /tmp/healthy
  periodSeconds: 5
```

**Startup Probe (for slow-starting apps):**

```yaml
startupProbe:
  httpGet:
    path: /healthz
    port: 8080
  failureThreshold: 30       # Allow up to 30 x 10 = 300 seconds to start
  periodSeconds: 10
```

### Q23. What is the difference between `Recreate` and `RollingUpdate` deployment strategies?

**Answer:**

| Strategy | How It Works | Downtime? | Use When |
|----------|-------------|-----------|----------|
| **Recreate** | Kill ALL old Pods first, then start new Pods | ✅ Yes (brief downtime) | App can't run two versions simultaneously (e.g., DB schema changes) |
| **RollingUpdate** | Replace Pods one by one gradually | ❌ No downtime | Most applications (default) |

```yaml
# Recreate Strategy
spec:
  strategy:
    type: Recreate            # Kill all → Start all

# RollingUpdate Strategy (default)
spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 25%           # Max extra Pods during update
      maxUnavailable: 25%     # Max Pods that can be unavailable
```

### Q24. How does Horizontal Pod Autoscaler (HPA) work?

**Answer:**

HPA automatically **scales the number of Pod replicas** based on CPU usage, memory usage, or custom metrics.

```
                     ┌─────────┐
  Metrics Server ──→ │   HPA   │ ──→ Scale Deployment
                     └─────────┘
                     
  CPU > 80%? → Scale UP (add more Pods)
  CPU < 20%? → Scale DOWN (remove Pods)
```

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: my-app-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-app
  minReplicas: 2                # Never go below 2
  maxReplicas: 10               # Never exceed 10
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70    # Scale when CPU > 70%
    - type: Resource
      resource:
        name: memory
        target:
          type: Utilization
          averageUtilization: 80    # Scale when Memory > 80%
```

```bash
# Quick way to create HPA
kubectl autoscale deployment my-app --min=2 --max=10 --cpu-percent=70

# Check HPA status
kubectl get hpa
```

**Prerequisite:** **Metrics Server** must be installed in the cluster for HPA to work!

### Q25. What is a Headless Service?

**Answer:**

A **Headless Service** is a Service with `clusterIP: None`. Instead of providing a single virtual IP for load balancing, it returns the **individual Pod IPs** via DNS.

**Normal Service:** `my-svc` → one IP (10.96.0.1) → load balances to Pods
**Headless Service:** `my-svc` → returns all Pod IPs directly

**Why use it?**
- StatefulSets need to address individual Pods (e.g., `mysql-0.mysql.default.svc.cluster.local`)
- Clients need to know ALL Pod IPs (e.g., for peer discovery in Kafka, Elasticsearch)

```yaml
apiVersion: v1
kind: Service
metadata:
  name: mysql
spec:
  clusterIP: None              # This makes it headless!
  selector:
    app: mysql
  ports:
    - port: 3306
```

```bash
# DNS lookup — returns individual Pod IPs
nslookup mysql.default.svc.cluster.local
# → mysql-0.mysql.default.svc.cluster.local → 10.244.1.5
# → mysql-1.mysql.default.svc.cluster.local → 10.244.2.8
# → mysql-2.mysql.default.svc.cluster.local → 10.244.3.3
```

### Q26. What is a PodDisruptionBudget (PDB)?

**Answer:**

A **PDB** defines the **minimum number of Pods that must be available** during voluntary disruptions (node drains, cluster upgrades, etc.).

It does NOT protect against involuntary disruptions (node crashes, hardware failures).

```yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: my-app-pdb
spec:
  minAvailable: 2              # At least 2 Pods must be running
  # OR
  # maxUnavailable: 1          # At most 1 Pod can be down
  selector:
    matchLabels:
      app: my-app
```

**Real-world scenario:** You have 3 replicas, and someone runs `kubectl drain node-1`. Without PDB, Kubernetes might drain all Pods on that node at once. With `minAvailable: 2`, Kubernetes ensures at least 2 Pods are always running — it drains one at a time.

### Q28. How do you troubleshoot a Pod stuck in `CrashLoopBackOff`?

**Answer:**

`CrashLoopBackOff` means the container keeps **crashing and Kubernetes keeps restarting it**, with increasing delays between restarts.

**Troubleshooting steps:**

```bash
# Step 1: Check the Pod events
kubectl describe pod <pod-name>
# Look at the "Events" section at the bottom

# Step 2: Check logs from the current container
kubectl logs <pod-name>

# Step 3: Check logs from the PREVIOUS crashed container
kubectl logs <pod-name> --previous

# Step 4: Check if the image exists and is correct
kubectl get pod <pod-name> -o jsonpath='{.spec.containers[*].image}'

# Step 5: Try running the container interactively
kubectl run debug --image=<image-name> -it --rm -- /bin/sh
```

**Common causes:**
1. **Application error** — The app crashes on startup (check logs)
2. **Missing config** — Environment variables, ConfigMaps, or Secrets not found
3. **Wrong command** — The CMD/entrypoint is incorrect
4. **Resource limits** — Container gets OOMKilled (Out of Memory)
5. **Liveness probe failing** — Container is restarted even though it's fine (probe misconfigured)
6. **Permission issues** — Container can't write to a volume or bind to a port

### Q29. What are Pod Lifecycle phases?

**Answer:**

```
Pending → Running → Succeeded/Failed
                        ↑
                    (or Unknown)
```

| Phase | Meaning |
|-------|---------|
| **Pending** | Pod is accepted but containers aren't running yet (scheduling, image pulling) |
| **Running** | Pod is bound to a node and at least one container is running |
| **Succeeded** | All containers exited successfully (exit code 0) — common for Jobs |
| **Failed** | All containers have terminated, and at least one failed (non-zero exit) |
| **Unknown** | Pod state can't be determined (usually a node communication issue) |

**Container states within a Pod:**
- **Waiting** — Container is not running yet (pulling image, waiting for init containers)
- **Running** — Container is executing
- **Terminated** — Container has exited (either successfully or with an error)

### Q30. What is a ReplicationController? Is it still used?

**Answer:**

A **ReplicationController** was the **original way** to ensure a desired number of Pod replicas were running. It's the **predecessor** of ReplicaSet.

**Difference:**

| Feature | ReplicationController | ReplicaSet |
|---------|----------------------|------------|
| Selector type | Equality-based only (`app=nginx`) | Equality + Set-based (`app in (nginx, web)`) |
| Used by | Nothing (legacy) | Deployments |
| Status | **Deprecated** | Current |

**Bottom line:** ReplicationController is **legacy** and should NOT be used. Use **Deployments** (which create ReplicaSets automatically).

### Q31. What is a Service in Kubernetes? What are the types?

**Answer:**

A **Service** is a stable way to **expose and access Pods**. Since Pods are temporary (they can be created, destroyed, and get new IPs), you need a Service to provide a **consistent endpoint**.

**Think of a Service as a phone number:** Pods (people) change, but the phone number (Service) stays the same.

**Types of Services:**

| Type | What It Does | Accessible From |
|------|-------------|----------------|
| **ClusterIP** | Internal-only IP within the cluster | Inside the cluster only |
| **NodePort** | Opens a port (30000-32767) on every node | Outside the cluster via `NodeIP:NodePort` |
| **LoadBalancer** | Creates an external load balancer (cloud only) | Outside via a public IP |
| **ExternalName** | Maps to an external DNS name (CNAME) | Inside the cluster |

```yaml
# ClusterIP (default) — internal only
apiVersion: v1
kind: Service
metadata:
  name: my-app-svc
spec:
  type: ClusterIP
  selector:
    app: my-app
  ports:
    - port: 80              # Service port
      targetPort: 8080      # Container port

---
# NodePort — accessible from outside
apiVersion: v1
kind: Service
metadata:
  name: my-app-nodeport
spec:
  type: NodePort
  selector:
    app: my-app
  ports:
    - port: 80
      targetPort: 8080
      nodePort: 30080       # Accessible at NodeIP:30080

---
# LoadBalancer — gets a public IP
apiVersion: v1
kind: Service
metadata:
  name: my-app-lb
spec:
  type: LoadBalancer
  selector:
    app: my-app
  ports:
    - port: 80
      targetPort: 8080
```

### Q32. How does DNS work inside Kubernetes?

**Answer:**

Kubernetes runs **CoreDNS** (a DNS server) that gives every Service and Pod a DNS name.

**DNS naming format for Services:**

```
<service-name>.<namespace>.svc.cluster.local
```

**Examples:**

```bash
# Service "mysql" in namespace "default"
mysql.default.svc.cluster.local

# Short forms (within same namespace)
mysql
mysql.default

# Headless Service — individual Pod DNS
mysql-0.mysql.default.svc.cluster.local
mysql-1.mysql.default.svc.cluster.local
```

**How it works internally:**
1. A Pod makes a DNS query (e.g., "where is mysql?")
2. The request goes to **CoreDNS** (running as a Pod in `kube-system` namespace)
3. CoreDNS looks up the Service → returns the ClusterIP
4. The Pod connects using that IP

```bash
# Check DNS from inside a pod
kubectl exec -it my-pod -- nslookup mysql.default.svc.cluster.local
```

### Q33. What is an Ingress? How is it different from a Service?

**Answer:**

An **Ingress** manages **external HTTP/HTTPS access** to Services in your cluster. It provides:
- **URL routing** — Route different paths to different Services
- **SSL/TLS termination** — Handle HTTPS certificates
- **Name-based virtual hosting** — Route by hostname

**Service vs Ingress:**

| Feature | Service (LoadBalancer) | Ingress |
|---------|----------------------|---------|
| **Protocol** | TCP/UDP (any) | HTTP/HTTPS only |
| **Routing** | 1 Service = 1 Load Balancer | Many Services behind 1 Load Balancer |
| **Path-based routing** | ❌ No | ✅ Yes |
| **SSL termination** | ❌ No (manual) | ✅ Yes |
| **Cost** | 1 Load Balancer per Service ($$$) | 1 Load Balancer for everything ($) |

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  tls:
    - hosts:
        - myapp.example.com
      secretName: tls-secret
  rules:
    - host: myapp.example.com
      http:
        paths:
          - path: /api
            pathType: Prefix
            backend:
              service:
                name: api-service
                port:
                  number: 80
          - path: /
            pathType: Prefix
            backend:
              service:
                name: frontend-service
                port:
                  number: 80
```

**Important:** Ingress needs an **Ingress Controller** (like Nginx Ingress, Traefik) to actually work. Ingress is just the rules; the controller enforces them.

### Q35. What are Network Policies?

**Answer:**

**Network Policies** control **which Pods can talk to which other Pods** and to external endpoints. By default, all Pods in Kubernetes can communicate with all other Pods — Network Policies restrict this.

Think of them as **firewall rules** for your Pods.

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-backend
  namespace: default
spec:
  podSelector:
    matchLabels:
      app: backend               # Apply to backend Pods
  policyTypes:
    - Ingress                     # Control incoming traffic
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: frontend       # Only allow traffic FROM frontend Pods
      ports:
        - protocol: TCP
          port: 8080              # Only on port 8080
```

**This policy says:** "Only allow traffic to `backend` Pods from `frontend` Pods on port 8080. Block everything else."

**Important:** Network Policies require a **CNI plugin** that supports them (Calico, Cilium, Weave). Basic flannel does NOT support Network Policies!

```
                    Internet
                       │
```

### Q37. What is kube-proxy? How does it work?

**Answer:**

**kube-proxy** runs on every node and manages **networking rules** to route traffic to the right Pods.

When you create a Service, kube-proxy creates rules so that traffic sent to the Service IP gets forwarded to one of the backend Pods.

| **userspace** | Proxies in user space | Slow, legacy, don't use |

### Q39. How do Pods communicate with each other?

**Answer:**

Kubernetes networking follows these fundamental rules:
1. Every Pod gets its own **unique IP address**
2. Pods on the **same node** can communicate directly
3. Pods on **different nodes** can communicate without NAT
4. All Pods can reach all other Pods (unless Network Policies restrict it)

**Same Pod:** Containers use `localhost` (they share the network namespace)
**Same Node:** Via the virtual bridge (cbr0 or similar)
**Different Nodes:** Via the overlay network (managed by the CNI plugin)
**Via Services:** Using DNS names (e.g., `mysql.default.svc.cluster.local`)

### Q40. What is a Gateway API? How is it different from Ingress?

**Answer:**

**Gateway API** is the next-generation replacement for Ingress. It's more flexible, more expressive, and better designed.

| Feature | Ingress | Gateway API |
|---------|---------|-------------|
| **Protocol support** | HTTP/HTTPS only | HTTP, HTTPS, TCP, UDP, gRPC |

| **Multi-tenancy** | Limited | Built-in (different teams manage different layers) |
| **Status** | Stable, mature | Graduating, becoming the standard |

```yaml
# Gateway API splits concerns:
# 1. Infrastructure team: manages GatewayClass + Gateway
# 2. Application team: manages Routes

    - name: my-gateway

            type: PathPrefix

        - name: api-service
          port: 80
          weight: 90              # 90% traffic
        - name: api-service-canary
          port: 80
          weight: 10              # 10% traffic (canary!)
```

### Q41. What is the difference between port, targetPort, nodePort, and containerPort?

**Answer:**

This confuses everyone! Here's a clear breakdown:

```
Internet → NodePort (30080) → Service Port (80) → targetPort (8080) = containerPort (8080)
```

| Port | Where It Lives | What It Does |
|------|---------------|-------------|
| **containerPort** | Pod spec | The port your app listens on inside the container |
| **port** | Service spec | The port the Service exposes (what clients connect to) |
| **targetPort** | Service spec | The port on the Pod where traffic is forwarded to (usually = containerPort) |
| **nodePort** | Service spec (NodePort type) | The port opened on every node (30000-32767) |

```yaml
# Pod
spec:
  containers:
    - name: my-app
      ports:
        - containerPort: 8080     # App listens on 8080

# Service
spec:
  type: NodePort
  ports:
    - port: 80                    # Service listens on 80
      targetPort: 8080            # Forwards to container's 8080
      nodePort: 30080             # Node opens port 30080
```

### Q43. What are Volumes in Kubernetes?

**Answer:**

Containers are **ephemeral** — when they restart, all data inside them is lost. **Volumes** provide persistent or shared storage for Pods.

**Common Volume Types:**

| Type | Description | Persistent? |
|------|------------|-------------|
| `emptyDir` | Empty directory, created when Pod starts | ❌ Deleted when Pod is deleted |

| `configMap` | Mount a ConfigMap as files | ❌ Read-only data |
| `secret` | Mount a Secret as files | ❌ Read-only data |
| `persistentVolumeClaim` | Claim a PersistentVolume | ✅ Yes |
| `nfs` | Network File System | ✅ Yes |

```yaml
# emptyDir — shared storage between containers in same Pod
spec:
  containers:
    - name: app
      image: my-app
      volumeMounts:
        - name: shared-data
          mountPath: /data
    - name: sidecar
      image: log-processor
      volumeMounts:
        - name: shared-data
          mountPath: /data
  volumes:
    - name: shared-data
      emptyDir: {}
```

### Q44. What is the difference between PV and PVC?

**Answer:**

| Concept | PersistentVolume (PV) | PersistentVolumeClaim (PVC) |
|---------|----------------------|---------------------------|
| **Who creates it** | Admin (or dynamic provisioning) | Developer |
| **What it is** | Actual storage resource (like a disk) | A request for storage |
| **Analogy** | A hotel room | A hotel reservation |

```yaml
# PV — Created by admin (or dynamically by StorageClass)
apiVersion: v1
kind: PersistentVolume
metadata:
  name: my-pv
spec:
  capacity:
    storage: 10Gi
  accessModes:
    - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain
  storageClassName: standard
  hostPath:
    path: /data/my-pv

---
# PVC — Created by developer
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-pvc
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 5Gi                  # I need 5Gi
  storageClassName: standard

---
# Pod using the PVC
spec:
  containers:
    - name: app
      volumeMounts:
        - name: data
          mountPath: /app/data
  volumes:
    - name: data
      persistentVolumeClaim:
        claimName: my-pvc            # Use this PVC
```

**Access Modes:**
- `ReadWriteOnce (RWO)` — Can be mounted as read-write by ONE node
- `ReadOnlyMany (ROX)` — Can be mounted as read-only by MANY nodes
- `ReadWriteMany (RWX)` — Can be mounted as read-write by MANY nodes

### Q45. What is a StorageClass?

**Answer:**

A **StorageClass** enables **dynamic provisioning** of PersistentVolumes. Instead of an admin pre-creating PVs, the StorageClass automatically creates them when a PVC is made.

**Without StorageClass:** Admin manually creates PVs → Developers create PVCs → Kubernetes matches them.
**With StorageClass:** Developers create PVCs → StorageClass automatically creates PVs.

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-ssd
provisioner: kubernetes.io/aws-ebs      # AWS EBS provisioner
parameters:
  type: gp3                              # SSD type
  fsType: ext4
reclaimPolicy: Delete                    # Delete volume when PVC is deleted
volumeBindingMode: WaitForFirstConsumer  # Create volume only when Pod needs it
```

```yaml
# PVC using this StorageClass
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-fast-storage
spec:
  storageClassName: fast-ssd             # Use the fast SSD class
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 20Gi
```

### Q46. What is the difference between `emptyDir` and `hostPath`?

**Answer:**

| Feature | emptyDir | hostPath |
|---------|---------|---------|
| **Created when** | Pod is assigned to a node | Mounted from existing host directory |
| **Lifetime** | Same as the Pod — deleted when Pod dies | Persists on the node even after Pod dies |
| **Shared between** | Containers in the SAME Pod | Any Pod scheduled on that node |
| **Use case** | Temp storage, caching, sharing between containers | Accessing host files (logs, Docker socket) |
| **Production use** | ✅ Common | ⚠️ Avoid (not portable, security risk) |

### Q47. What happens when a PersistentVolume is deleted?

**Answer:**

It depends on the **Reclaim Policy**:

| Policy | What Happens |
|--------|-------------|
| **Retain** | PV is kept but marked as "Released". Data is preserved. Admin must manually clean up. |
| **Delete** | PV AND the underlying storage (EBS volume, Azure Disk) are deleted. Data is GONE. |
| **Recycle** | Data is deleted (`rm -rf /volume/*`), PV becomes available again. **Deprecated!** |

```yaml
spec:
  persistentVolumeReclaimPolicy: Retain    # Keep my data!
```

**Best practice:** Use `Retain` for important data (databases). Use `Delete` for temporary/reproducible data.

### Q48. What is a CSI Driver?

**Answer:**

**CSI (Container Storage Interface)** is a standard interface that allows Kubernetes to work with **any storage system** without built-in plugins.

**Before CSI:** Storage drivers were built INTO Kubernetes (hard to update, hard to add new ones).
**After CSI:** Storage vendors write their own drivers as separate components.

**Popular CSI Drivers:**
- **AWS EBS CSI** — Amazon Elastic Block Store

- **GCE PD CSI** — Google Persistent Disk

### Q49. How do you resize a PersistentVolume?

**Answer:**

You can resize a PVC (expand it) if the StorageClass allows it:

```yaml
# StorageClass must allow expansion
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: expandable
provisioner: kubernetes.io/aws-ebs
allowVolumeExpansion: true          # This must be true!
```

```bash
# Edit the PVC and increase the storage
kubectl edit pvc my-pvc
# Change: storage: 10Gi → storage: 20Gi

# Or use patch
kubectl patch pvc my-pvc -p '{"spec":{"resources":{"requests":{"storage":"20Gi"}}}}'
```

**Important:** You can only **expand** — you cannot **shrink** a PVC!

### Q50. What is an ephemeral volume?

**Answer:**

An **ephemeral volume** is created and deleted along with the Pod. It exists only for the Pod's lifetime.

**Types of ephemeral volumes:**
- `emptyDir` — Empty directory for scratch space
- `configMap` / `secret` — Project configuration data
- `downwardAPI` — Expose Pod metadata
- **Generic ephemeral volumes** (Kubernetes 1.23+) — Dynamically provisioned temporary volumes

```yaml
# Generic ephemeral volume
spec:
  containers:
    - name: app
      volumeMounts:
        - name: scratch
          mountPath: /tmp/data
  volumes:
    - name: scratch
      ephemeral:
        volumeClaimTemplate:
          spec:
            accessModes: ["ReadWriteOnce"]
            storageClassName: fast-ssd
            resources:
              requests:
                storage: 10Gi
```

**Use case:** When your app needs temporary high-performance storage (e.g., ML model training, video processing) that's bigger than `emptyDir` but you don't need to keep it.

## Configuration & Secrets

### Q51. What is a ConfigMap?

**Answer:**

A **ConfigMap** stores **non-sensitive configuration data** as key-value pairs. It decouples configuration from your container images so you can change config without rebuilding images.

```bash
# Create from literal values
kubectl create configmap app-config \
  --from-literal=DB_HOST=mysql.default.svc.cluster.local \
  --from-literal=DB_PORT=3306 \
  --from-literal=LOG_LEVEL=info

# Create from a file
kubectl create configmap nginx-config --from-file=nginx.conf
```

```yaml
# ConfigMap YAML
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  DB_HOST: "mysql.default.svc.cluster.local"
  DB_PORT: "3306"
  LOG_LEVEL: "info"
  app.properties: |
    server.port=8080
    spring.profiles.active=production
```

**Using ConfigMap in Pods:**

```yaml
# Method 1: As environment variables
spec:
  containers:
    - name: app
      envFrom:
        - configMapRef:
            name: app-config      # All keys become env vars

# Method 2: Individual env vars
      env:
        - name: DATABASE_HOST
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: DB_HOST

# Method 3: As a mounted volume (files)
      volumeMounts:
        - name: config
          mountPath: /etc/config
  volumes:
    - name: config
      configMap:
        name: app-config
```

### Q52. What is a Secret? How is it different from ConfigMap?

**Answer:**

A **Secret** is similar to a ConfigMap but is intended for **sensitive data** — passwords, tokens, certificates, API keys.

| Feature | ConfigMap | Secret |
|---------|----------|--------|
| **Data type** | Non-sensitive config | Sensitive data |
| **Storage** | Plain text in etcd | Base64-encoded in etcd |
| **Size limit** | 1MB | 1MB |
| **Encrypted at rest** | Usually not | Can be (with encryption provider) |

```bash
# Create a Secret
kubectl create secret generic db-credentials \
  --from-literal=username=admin \
  --from-literal=password=supersecret123
```

```yaml
# Secret YAML (values must be base64-encoded)
apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
type: Opaque
data:
  username: YWRtaW4=               # echo -n "admin" | base64
  password: c3VwZXJzZWNyZXQxMjM=   # echo -n "supersecret123" | base64

# Or use stringData (plain text — Kubernetes encodes it for you)
apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
type: Opaque
stringData:
  username: admin
  password: supersecret123
```

**⚠️ Important:** Base64 is NOT encryption! Anyone can decode it. For real security:
- Enable **encryption at rest** for etcd
- Use **RBAC** to restrict who can read Secrets
- Consider **external secret managers** (Vault, AWS Secrets Manager)

## Scheduling & Resource Management

### Q58. What are Resource Requests and Limits?

**Answer:**

**Requests** = Minimum resources **guaranteed** to a container (used for scheduling).
**Limits** = Maximum resources a container can use (enforced at runtime).

```yaml
spec:
  containers:
    - name: app
      resources:
        requests:
          cpu: "250m"          # 250 millicores = 0.25 CPU
          memory: "256Mi"      # 256 MiB RAM
        limits:
          cpu: "500m"          # Max 0.5 CPU
          memory: "512Mi"      # Max 512 MiB RAM
```

**What happens when limits are exceeded?**
- **CPU limit exceeded** → Container is **throttled** (slowed down, not killed)
- **Memory limit exceeded** → Container is **OOMKilled** (killed and restarted)

**CPU units:**
- `1` = 1 full CPU core
- `500m` = 0.5 CPU = half a core
- `100m` = 0.1 CPU = 10% of a core

**Memory units:**
- `Mi` = Mebibytes (binary: 1Mi = 1,048,576 bytes)
- `Gi` = Gibibytes
- `M` = Megabytes (decimal: 1M = 1,000,000 bytes)

---

### Q59. What are QoS Classes in Kubernetes?

**Answer:**

Kubernetes assigns a **Quality of Service (QoS) class** to each Pod based on its resource configuration. This determines which Pods get killed first when a node runs out of resources.

| QoS Class | When Assigned | Eviction Priority |
|-----------|--------------|------------------|
| **Guaranteed** | requests = limits (for ALL containers) | Last to be killed |
| **Burstable** | requests < limits (or only requests set) | Killed after BestEffort |
| **BestEffort** | No requests or limits set | First to be killed |

```yaml
# Guaranteed (safest)
resources:
  requests:
    cpu: "500m"
    memory: "512Mi"
  limits:
    cpu: "500m"           # Same as request
    memory: "512Mi"       # Same as request

# Burstable
resources:
  requests:
    cpu: "100m"
    memory: "128Mi"
  limits:
    cpu: "500m"           # Different from request
    memory: "512Mi"       # Different from request

# BestEffort (no resources specified at all)
# Just don't set resources section
```

### Q60. What are Taints and Tolerations?

**Answer:**

**Taints** are applied to **nodes** — they repel Pods that don't have matching tolerations.
**Tolerations** are applied to **Pods** — they allow Pods to be scheduled on tainted nodes.

Think of it like: **Taints** = "No entry unless you have a pass." **Tolerations** = "I have the pass."

```bash
# Taint a node
kubectl taint nodes node1 environment=production:NoSchedule

# Remove a taint
kubectl taint nodes node1 environment=production:NoSchedule-
```

```yaml
# Pod with toleration
spec:
  tolerations:
    - key: "environment"
      operator: "Equal"
      value: "production"
      effect: "NoSchedule"
```

**Taint Effects:**

| Effect | What Happens |
|--------|-------------|
| `NoSchedule` | New Pods won't be scheduled (existing Pods stay) |
| `PreferNoSchedule` | Try to avoid scheduling, but do it if no other option |
| `NoExecute` | New Pods aren't scheduled + existing Pods are evicted |

**Real-world use:** Reserve GPU nodes only for ML workloads by tainting them with `gpu=true:NoSchedule` and adding tolerations only to ML Pods.

### Q61. What are Node Affinity and Anti-Affinity?

**Answer:**

**Node Affinity** = "Schedule this Pod on nodes with specific labels" (like an advanced `nodeSelector`).
**Pod Anti-Affinity** = "Don't schedule this Pod on the same node as another specific Pod."

```yaml
# Node Affinity — schedule on nodes in us-east-1a or us-east-1b
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:     # HARD rule
        nodeSelectorTerms:
          - matchExpressions:
              - key: topology.kubernetes.io/zone
                operator: In
                values:
                  - us-east-1a
                  - us-east-1b
      preferredDuringSchedulingIgnoredDuringExecution:    # SOFT rule
        - weight: 80
          preference:
            matchExpressions:
              - key: node-type
                operator: In
                values:
                  - high-memory
```

```yaml
# Pod Anti-Affinity — spread replicas across nodes
spec:
  affinity:
    podAntiAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        - labelSelector:
            matchExpressions:
              - key: app
                operator: In
                values:
                  - my-app
          topologyKey: kubernetes.io/hostname   # Don't put 2 Pods on same node
```

### Q62. What is a LimitRange?

**Answer:**

A **LimitRange** sets **default and maximum resource limits** for containers in a namespace. It prevents any single Pod from consuming too many resources.

```yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: default-limits
  namespace: dev
spec:
  limits:
    - type: Container
      default:                    # Default limits (if not specified)
        cpu: "500m"
        memory: "512Mi"
      defaultRequest:             # Default requests (if not specified)
        cpu: "100m"
        memory: "128Mi"
      max:                        # Maximum allowed
        cpu: "2"
        memory: "4Gi"
      min:                        # Minimum required
        cpu: "50m"
        memory: "64Mi"
```

If a developer creates a Pod without specifying resources, the LimitRange automatically adds the default values.

### Q63. What is a ResourceQuota?

**Answer:**

A **ResourceQuota** limits the **total amount of resources** a namespace can consume. Unlike LimitRange (per-Pod), ResourceQuota is for the **entire namespace**.

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: dev-quota
  namespace: dev
spec:
  hard:
    pods: "20"                   # Max 20 Pods
    requests.cpu: "10"           # Max 10 CPU cores total
    requests.memory: "20Gi"      # Max 20Gi memory total
    limits.cpu: "20"
    limits.memory: "40Gi"
    persistentvolumeclaims: "10" # Max 10 PVCs
    services.loadbalancers: "2"  # Max 2 LoadBalancer services
```

```bash
# Check quota usage
kubectl describe resourcequota dev-quota -n dev
```

### Q64. What is Pod Priority and Preemption?

**Answer:**

**Pod Priority** lets you assign importance levels to Pods. **Preemption** means if a high-priority Pod can't be scheduled, Kubernetes can **evict lower-priority Pods** to make room.

```yaml
# Step 1: Create PriorityClass
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: high-priority
value: 1000000
globalDefault: false
description: "High priority for production workloads"

---
# Step 2: Use it in a Pod
spec:
  priorityClassName: high-priority
  containers:
    - name: critical-app
      image: my-app
```

### Q65. What is the Cluster Autoscaler?

**Answer:**

The **Cluster Autoscaler** automatically adjusts the **number of nodes** in your cluster:
- **Scales UP** — When Pods are pending because no node has enough resources
- **Scales DOWN** — When nodes are underutilized for a period of time

```
Pod can't be scheduled (no resources)
        │
        ▼
Cluster Autoscaler detects pending Pods
        │
        ▼
Requests cloud provider to add new node(s)
        │
        ▼
New node joins cluster → Pod is scheduled
```

It works with cloud providers:
- **AWS** — Auto Scaling Groups (ASG)
- **Azure** — Virtual Machine Scale Sets (VMSS)
- **GCP** — Managed Instance Groups (MIG)

**Newer alternative:** **Karpenter** (AWS) — faster, more flexible node provisioning.

## Security

### Q66. What is RBAC in Kubernetes?

**Answer:**

**RBAC (Role-Based Access Control)** controls **who can do what** in your cluster.

Four key objects:

| Object | Scope | Purpose |
|--------|-------|---------|
| **Role** | Namespace | Defines permissions within a namespace |
| **ClusterRole** | Cluster-wide | Defines permissions across the entire cluster |
| **RoleBinding** | Namespace | Assigns a Role to a user/group/service account |
| **ClusterRoleBinding** | Cluster-wide | Assigns a ClusterRole to a user/group/service account |

```yaml
# Role — allows read access to Pods in "dev" namespace
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: dev
rules:
  - apiGroups: [""]                # Core API group
    resources: ["pods"]
    verbs: ["get", "list", "watch"]

---
# RoleBinding — assigns Role to a user
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-pods-binding
  namespace: dev
subjects:
  - kind: User
    name: jane
    apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
```

```bash
# Check if you have permission
kubectl auth can-i create pods --namespace=dev
kubectl auth can-i delete deployments --namespace=production --as=jane
```

### Q67. What is a Service Account?

**Answer:**

A **Service Account** provides an identity for **Pods** (not humans) to interact with the Kubernetes API.

Every namespace has a `default` Service Account. Every Pod uses it unless you specify another.

```yaml
# Create a Service Account
apiVersion: v1
kind: ServiceAccount
metadata:
  name: my-app-sa
  namespace: default

---
# Use it in a Pod
spec:
  serviceAccountName: my-app-sa
  containers:
    - name: app
      image: my-app
```

**Why custom Service Accounts?**
- **Least privilege** — Give each app only the permissions it needs
- **Image pull secrets** — Attach registry credentials to a Service Account
- **RBAC integration** — Bind Roles to Service Accounts

```yaml
# Give the Service Account permission to read Pods
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: my-app-binding
subjects:
  - kind: ServiceAccount
    name: my-app-sa
    namespace: default
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
```

### Q68. What are Pod Security Standards (PSS)?

**Answer:**

**Pod Security Standards** define three security levels for Pods:

| Level | Description | Use Case |
|-------|------------|----------|
| **Privileged** | No restrictions at all | System-level workloads (CNI, storage drivers) |
| **Baseline** | Prevents known privilege escalations | Most workloads |
| **Restricted** | Highly restrictive best practices | Security-sensitive workloads |

**Pod Security Admission (PSA)** enforces these standards at the **namespace level**:

```yaml
# Label a namespace with a security level
apiVersion: v1
kind: Namespace
metadata:
  name: production
  labels:
    pod-security.kubernetes.io/enforce: restricted   # Block violations
    pod-security.kubernetes.io/audit: restricted      # Log violations
    pod-security.kubernetes.io/warn: restricted       # Warn on violations
```

**Restricted level requirements:**
- Must run as non-root
- Must drop ALL capabilities
- Must use a read-only root filesystem
- Cannot use hostNetwork, hostPID, hostIPC
- Cannot use privileged containers

### Q69. What is OPA Gatekeeper?

**Answer:**

**OPA (Open Policy Agent) Gatekeeper** is a policy engine for Kubernetes. It lets you define custom rules (policies) that are enforced when resources are created or updated.

**Example policies:**
- "All containers must have resource limits"
- "No container can run as root"
- "All images must come from our private registry"
- "All resources must have a `team` label"

```yaml
# ConstraintTemplate — defines the policy logic
apiVersion: templates.gatekeeper.sh/v1
kind: ConstraintTemplate
metadata:
  name: k8srequiredlabels
spec:
  crd:
    spec:
      names:
        kind: K8sRequiredLabels
      validation:
        openAPIV3Schema:
          type: object
          properties:
            labels:
              type: array
              items:
                type: string
  targets:
    - target: admission.k8s.gatekeeper.sh
      rego: |
        package k8srequiredlabels
        violation[{"msg": msg}] {
          provided := {label | input.review.object.metadata.labels[label]}
          required := {label | label := input.parameters.labels[_]}
          missing := required - provided
          count(missing) > 0
          msg := sprintf("Missing labels: %v", [missing])
        }

---
# Constraint — applies the policy
apiVersion: constraints.gatekeeper.sh/v1beta1
kind: K8sRequiredLabels
metadata:
  name: require-team-label
spec:
  match:
    kinds:
      - apiGroups: ["apps"]
        kinds: ["Deployment"]
  parameters:
    labels: ["team", "environment"]
```

### Q70. How do you secure the Kubernetes API Server?

Key security measures:

1. **Authentication** — Who are you?

### Q71. What is Kyverno?

**Answer:**

**Kyverno** is a **Kubernetes-native policy engine** (alternative to OPA Gatekeeper). It uses standard Kubernetes YAML — no new language to learn (unlike Rego in OPA).

```yaml
# Policy: Require all Pods to have resource limits
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: require-limits
spec:
  validationFailureAction: Enforce   # Block or Audit
  rules:
    - name: check-limits
      match:
        any:
          - resources:
              kinds:
                - Pod
      validate:
        message: "CPU and memory limits are required!"
        pattern:
          spec:
            containers:
              - resources:
                  limits:
                    memory: "?*"
                    cpu: "?*"
```

**Kyverno vs OPA Gatekeeper:**

| Feature | Kyverno | OPA Gatekeeper |
|---------|---------|---------------|
| **Language** | YAML (K8s-native) | Rego (custom language) |
| **Learning curve** | Low | Higher |
| **Generate resources** | ✅ Yes | ❌ No |
| **Mutate resources** | ✅ Yes | ✅ Yes |
| **Community** | Growing fast | Well-established |

### Q72. How do you handle image security in Kubernetes?

**Answer:**

1. **Use trusted registries:**

   ```yaml
   # Only allow images from your private registry
   ```

### Q73. What is Helm? Why do we use it?

**Answer:**

**Helm** is the **package manager for Kubernetes** — like apt for Linux or npm for Node.js.

**Problem it solves:** Deploying an app on Kubernetes requires many YAML files (Deployment, Service, ConfigMap, Secret, Ingress, etc.). Helm packages all of them into a single **Chart** with customizable values.

```bash
# Install an app with one command
helm install my-release bitnami/nginx

# Install with custom values
helm install my-app ./my-chart -f values-prod.yaml

# Upgrade
helm upgrade my-app ./my-chart -f values-prod.yaml

# Rollback
helm rollback my-app 1

# List releases
helm list

# Uninstall
helm uninstall my-app
```

**Helm Chart structure:**
```
my-chart/
  Chart.yaml          # Chart metadata (name, version)
  values.yaml         # Default configuration values
  templates/          # Kubernetes YAML templates
    deployment.yaml
    service.yaml
    ingress.yaml
    _helpers.tpl      # Template helpers
  charts/             # Sub-charts (dependencies)
```

**Templating example:**
```yaml
# templates/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ .Release.Name }}-{{ .Chart.Name }}
spec:
  replicas: {{ .Values.replicas }}
  template:
    spec:
      containers:
        - name: {{ .Chart.Name }}
          image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
```

```yaml
# values.yaml
replicas: 3
image:
  repository: nginx
  tag: "1.25"
```

### Q74. What is Kustomize?

**Answer:**

**Kustomize** is a template-free way to customize Kubernetes YAML. Instead of Go templating (like Helm), it uses **patches and overlays** to modify base configurations.

It's built into `kubectl` — no extra tools needed!

```
base/
  deployment.yaml
  service.yaml
  kustomization.yaml
overlays/
  dev/
    kustomization.yaml
    replicas-patch.yaml
  prod/
    kustomization.yaml
    replicas-patch.yaml
```

```yaml
# base/kustomization.yaml
resources:
  - deployment.yaml
  - service.yaml

# overlays/prod/kustomization.yaml
resources:
  - ../../base
namePrefix: prod-
patches:
  - replicas-patch.yaml

# overlays/prod/replicas-patch.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 10      # Override to 10 replicas in prod
```

```bash
# Apply with kustomize
kubectl apply -k overlays/prod/
```

### Q75. What are Custom Resource Definitions (CRDs)?

**Answer:**

CRDs let you **extend Kubernetes** with your own custom resource types. You're basically creating new API objects.

**Example:** Kubernetes doesn't have a "Database" resource. But with a CRD, you can create one!

```yaml
# Define the CRD
apiVersion: apiextensions.k8s.io/v1
kind: CustomResourceDefinition
metadata:
  name: databases.example.com
spec:
  group: example.com
  versions:
    - name: v1
      served: true
      storage: true
      schema:
        openAPIV3Schema:
          type: object
          properties:
            spec:
              type: object
              properties:
                engine:
                  type: string
                  enum: ["mysql", "postgres"]
                version:
                  type: string
                storage:
                  type: string
  scope: Namespaced
  names:
    plural: databases
    singular: database
    kind: Database
    shortNames:
      - db

---
# Now use your custom resource!
apiVersion: example.com/v1
kind: Database
metadata:
  name: my-database
spec:
  engine: postgres
  version: "15.4"
  storage: "100Gi"
```

```bash
kubectl get databases
kubectl describe database my-database
```

CRDs are the foundation of **Operators** — controllers that automate complex application management.

### Q76. What is a Kubernetes Operator?

**Answer:**

An **Operator** is a **custom controller** that uses CRDs to automate the management of complex applications. It encodes human operational knowledge into software.

**Think of it as:** An SRE who never sleeps — constantly watching your application and taking action.

**What Operators do:**
- Install and configure complex apps (databases, Kafka, Elasticsearch)
- Handle upgrades and migrations
- Manage backups and restores
- Scale based on application-specific metrics
- Self-heal when things go wrong

**Examples:**
- **Prometheus Operator** — Manages Prometheus, Alertmanager, ServiceMonitor
- **Strimzi** — Manages Apache Kafka clusters
- **CloudNativePG** — Manages PostgreSQL clusters
- **cert-manager** — Manages TLS certificates
- **ArgoCD** — Manages GitOps deployments

**Operator = CRD + Custom Controller**

```
User creates Custom Resource (e.g., "I want a PostgreSQL cluster with 3 replicas")
        │
        ▼
Operator Controller watches for changes
        │
        ▼
Operator creates/manages Pods, Services, PVCs, ConfigMaps
        │
        ▼
Operator continuously reconciles desired state vs actual state
```

### Q77. How do you upgrade a Kubernetes cluster?

**Answer:**

**Golden rule:** Always upgrade **one minor version at a time** (1.27 → 1.28 → 1.29, NOT 1.27 → 1.29).

**Steps for kubeadm clusters:**

```bash
# Step 1: Upgrade the control plane (one node at a time)
# On the first control plane node:

# Update kubeadm
apt-get update && apt-get install -y kubeadm=1.28.0-00

# Check upgrade plan
kubeadm upgrade plan

# Apply upgrade
kubeadm upgrade apply v1.28.0

# Upgrade kubelet and kubectl
apt-get install -y kubelet=1.28.0-00 kubectl=1.28.0-00
systemctl daemon-reload && systemctl restart kubelet

# Step 2: Upgrade worker nodes (one at a time)
# Drain the node (move Pods to other nodes)
kubectl drain node-1 --ignore-daemonsets --delete-emptydir-data

# On the worker node:
apt-get update && apt-get install -y kubeadm=1.28.0-00
kubeadm upgrade node
apt-get install -y kubelet=1.28.0-00
systemctl daemon-reload && systemctl restart kubelet

# Uncordon the node (allow Pods back)
kubectl uncordon node-1
```

**For managed K8s (EKS, AKS, GKE):**
- Use the cloud console or CLI
- Control plane is upgraded first (often with one click)
- Worker nodes are upgraded by updating the node group/pool

### Q78. How do you debug Pods in Kubernetes?

**Answer:**

**Debugging toolkit:**

```bash
# 1. Check Pod status
kubectl get pods -o wide

# 2. Describe the Pod (shows events, conditions)
kubectl describe pod <pod-name>

# 3. Check logs
kubectl logs <pod-name>                    # Current logs
kubectl logs <pod-name> --previous         # Previous container logs
kubectl logs <pod-name> -c <container>     # Specific container
kubectl logs <pod-name> --tail=100         # Last 100 lines
kubectl logs <pod-name> -f                 # Follow (stream) logs

# 4. Shell into the Pod
kubectl exec -it <pod-name> -- /bin/sh

# 5. Debug with ephemeral containers (K8s 1.25+)
kubectl debug <pod-name> -it --image=busybox --target=my-container

# 6. Run a debug Pod on a specific node
kubectl debug node/node-1 -it --image=ubuntu

# 7. Check events
kubectl get events --sort-by=.metadata.creationTimestamp

# 8. Check resource usage
kubectl top pods
kubectl top nodes

# 9. Check DNS
kubectl exec -it <pod> -- nslookup kubernetes.default

# 10. Check networking
kubectl exec -it <pod> -- wget -qO- http://service-name:port
```

**Common issues and what to look for:**

| Status | Likely Issue | Check |
|--------|-------------|-------|
| `Pending` | No node can schedule it | `kubectl describe pod` → Events |
| `ImagePullBackOff` | Wrong image name/tag or no pull credentials | Image name, imagePullSecrets |
| `CrashLoopBackOff` | App keeps crashing | `kubectl logs --previous` |
| `OOMKilled` | Out of memory | Increase memory limits |
| `Evicted` | Node under resource pressure | Check node resources |

### Q79. What is the difference between `kubectl drain`, `kubectl cordon`, and `kubectl uncordon`?

**Answer:**

| Command | What It Does |
|---------|-------------|
| `kubectl cordon <node>` | Marks node as **unschedulable** — no NEW Pods will be placed on it. Existing Pods keep running. |
| `kubectl drain <node>` | Cordons the node AND **evicts all Pods** from it (moves them to other nodes). |
| `kubectl uncordon <node>` | Marks node as **schedulable** again — new Pods can be placed on it. |

```bash
# Preparing a node for maintenance
kubectl cordon node-1            # Stop new Pods from being scheduled
kubectl drain node-1 \
  --ignore-daemonsets \          # Don't evict DaemonSet Pods
  --delete-emptydir-data \      # Delete emptyDir data
  --force                        # Force evict Pods not managed by controllers

# After maintenance
kubectl uncordon node-1          # Allow Pods to be scheduled again
```

**Use case:** Node maintenance, OS updates, kernel upgrades, cluster upgrades.

### Q80. Explain multi-cluster Kubernetes. When and why?

**Answer:**

**Multi-cluster** means running **multiple Kubernetes clusters** and managing them together.

**Why multi-cluster?**

| Reason | Example |
|--------|---------|
| **High Availability** | If one cluster dies, others handle traffic |
| **Geographic distribution** | Clusters in different regions for low latency |
| **Environment isolation** | Separate clusters for dev, staging, prod |
| **Compliance** | Data sovereignty requirements (data must stay in specific regions) |
| **Blast radius** | Limit the impact of failures or misconfigurations |
| **Scale** | Single cluster has limits (~5,000 nodes) |

**Tools for multi-cluster management:**

| Tool | What It Does |
|------|-------------|
| **Rancher** | Centralized multi-cluster management UI |
| **Cluster API** | Declarative cluster lifecycle management |
| **ArgoCD** | Deploy apps across multiple clusters via GitOps |
| **Admiralty** | Multi-cluster scheduling |
| **Submariner** | Cross-cluster networking |
| **Liqo** | Virtual nodes for multi-cluster |
| **Karmada** | Multi-cluster workload management |

```
┌─────────────────┐    ┌─────────────────┐    ┌─────────────────┐
│  Cluster: US    │    │  Cluster: EU    │    │  Cluster: APAC  │
│  (us-east-1)    │    │  (eu-west-1)    │    │  (ap-south-1)   │
│                 │    │                 │    │                 │
```

## 🎯 Quick Revision — Key Differences Cheat Sheet

| Comparison | Key Difference |
|-----------|---------------|
| Pod vs Container | Pod is a wrapper around one or more containers |
| ReplicaSet vs Deployment | Deployment adds rolling updates & rollbacks on top of ReplicaSet |
| StatefulSet vs Deployment | StatefulSet gives stable identity and persistent storage |
| DaemonSet vs Deployment | DaemonSet runs one Pod per node |
| ConfigMap vs Secret | Secret is for sensitive data (base64-encoded) |
| PV vs PVC | PV is the actual storage; PVC is a request for storage |
| Role vs ClusterRole | Role is namespace-scoped; ClusterRole is cluster-wide |
| Ingress vs Service | Ingress is for HTTP routing; Service is for L4 load balancing |
| Taints vs Tolerations | Taints repel Pods; Tolerations allow Pods on tainted nodes |
| Node Affinity vs nodeSelector | Affinity is more expressive (supports In, NotIn, Exists operators) |
| Liveness vs Readiness Probe | Liveness restarts; Readiness removes from Service |
| Helm vs Kustomize | Helm uses templating; Kustomize uses patching |
| HPA vs VPA | HPA scales Pod count; VPA scales Pod resources |
| cordon vs drain | Cordon stops scheduling; Drain also evicts existing Pods |

## 🏆 Interview Tips

1. **Always explain the WHY** — Don't just say what something does, explain why you'd use it.
2. **Use real examples** — "In my project, we used StatefulSets for our PostgreSQL cluster because..."
3. **Draw diagrams** — On whiteboards, always draw the architecture.
4. **Mention trade-offs** — "We chose Helm over Kustomize because... but the downside is..."
5. **Talk about debugging** — Show that you can troubleshoot, not just deploy.
6. **Know `kubectl` well** — Be able to type commands without thinking.

8. **Practice with a real cluster** — Use Minikube, Kind, or k3s locally.
