### Q1. What is CI/CD? Explain each part.

**Answer:**

CI/CD stands for **Continuous Integration**, **Continuous Delivery**, and **Continuous Deployment**. These are three related but different practices.

```
в”Ӯ   Integration    в”Ӯ  (Happens on every commit)

в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ

в”Ӯ   Continuous     в”Ӯ  Package вҶ’ Stage вҶ’ Manual approval вҶ’ Deploy

в”Ӯ   Continuous     в”Ӯ  Package вҶ’ Stage вҶ’ Deploy automatically
```

| Practice | What It Does | Human Approval? |
|----------|-------------|----------------|
| **Continuous Integration** | Developers merge code frequently; each merge triggers automated build + tests | No вҖ” fully automated |

| **Continuous Deployment** | Every change that passes all tests is deployed to production **automatically** | No вҖ” fully automated |

**Simple way to remember:**
- CI = "Does my code work? Build it, test it, every time."
- Continuous Delivery = "My code is READY to go to production anytime."
- Continuous Deployment = "My code GOES to production automatically."

### Q2. What is a CI/CD Pipeline? What are the typical stages?

### Q3. What is the difference between Continuous Delivery and Continuous Deployment?

**Answer:**

This is one of the most commonly confused concepts:

| Aspect | Continuous Delivery | Continuous Deployment |
|--------|--------------------|--------------------|
| **Deploy to production** | Manual approval required | Fully automatic |
| **Risk** | Lower (human reviews first) | Higher (but caught by tests) |
| **Speed** | Slower (waiting for approval) | Fastest possible |
| **Requirements** | Good test coverage | Excellent test coverage + monitoring |
| **Common in** | Most companies | Very mature teams (Netflix, Amazon) |

```
Continuous Delivery:
  Code вҶ’ Build вҶ’ Test вҶ’ Scan вҶ’ Stage вҶ’ [MANUAL APPROVAL] вҶ’ Production

Continuous Deployment:
  Code вҶ’ Build вҶ’ Test вҶ’ Scan вҶ’ Stage вҶ’ Production (automatic!)
```

**Interview tip:** Most companies use **Continuous Delivery** (not Deployment). Say: "We use Continuous Delivery with a manual approval gate before production, because we want a human to verify before impacting customers."

### Q4. What is Jenkins? Why is it popular?

**Answer:**

**Jenkins** is an open-source automation server that is the **most widely used CI/CD tool** in the world. It's been around since 2011 (originally called Hudson).

**Why it's popular:**
- **Free and open-source** вҖ” No license costs
- **1,800+ plugins** вҖ” Integrates with everything (Docker, K8s, AWS, Git, etc.)
- **Highly customizable** вҖ” You can do literally anything
- **Large community** вҖ” Tons of documentation and support
- **Pipeline as Code** вҖ” Jenkinsfile stored in your repo

**Jenkins Architecture:**

```
в”Ӯ   Jenkins Master    в”Ӯ  вҶҗ Manages jobs, schedules builds, serves UI
в”Ӯ   (Controller)      в”Ӯ
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
           в”Ӯ
     в”Ңв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”җ
     в”Ӯ           в”Ӯ
в”Ңв”Җв”Җв”Җв”Җв–јв”Җв”Җв”Җв”Җв”җ в”Ңв”Җв”Җв”Җв–јв”Җв”Җв”Җв”Җв”Җв”җ

```

**Jenkinsfile example (Declarative Pipeline):**

```groovy
pipeline {
    agent any
    
    stages {
        stage('Build') {
            steps {
                sh 'mvn clean package'
            }
        }
        stage('Test') {
            steps {
                sh 'mvn test'
            }
        }
        stage('Docker Build') {
            steps {
                sh 'docker build -t my-app:${BUILD_NUMBER} .'
            }
        }
        stage('Deploy') {
            steps {
                sh 'kubectl apply -f k8s/deployment.yaml'
            }
        }
    }
    
    post {
        success {
            echo 'Pipeline succeeded!'
        }
        failure {
            echo 'Pipeline failed!'
        }
    }
}
```

### Q5. What is the difference between Declarative and Scripted Pipeline in Jenkins?

**Answer:**

| Feature | Declarative Pipeline | Scripted Pipeline |
|---------|---------------------|-------------------|
| **Syntax** | Structured YAML-like (simpler) | Full Groovy scripting |
| **Starts with** | `pipeline { }` | `node { }` |
| **Flexibility** | Limited (opinionated) | Unlimited (full programming) |
| **Learning curve** | Easy | Harder |
| **Error handling** | Built-in `post` section | Manual try-catch |
| **Best for** | 90% of pipelines | Complex, dynamic pipelines |

```groovy
// DECLARATIVE вҖ” Simple, structured
pipeline {
    agent any
    stages {
        stage('Build') {
            steps {
                sh 'make build'
            }
        }
    }
    post {
        always {
            cleanWs()
        }
    }
}

// SCRIPTED вҖ” Full Groovy control
node {
    try {
        stage('Build') {
            sh 'make build'
        }
    } catch (Exception e) {
        currentBuild.result = 'FAILURE'
        throw e
    } finally {
        cleanWs()
    }
}
```

**Interview tip:** "I prefer Declarative pipelines because they're easier to read and maintain. I only use Scripted when I need complex logic like dynamic stages or advanced error handling."

### Q6. What is GitHub Actions? How does it work?

**Answer:**

**GitHub Actions** is a CI/CD platform built directly into GitHub. It lets you automate workflows triggered by GitHub events (push, pull request, issue, schedule, etc.).

**Key concepts:**

| Concept | What It Is |
|---------|-----------|
| **Workflow** | A YAML file that defines the automation (`.github/workflows/`) |
| **Event** | What triggers the workflow (push, PR, cron, etc.) |
| **Job** | A set of steps that run on the same runner |
| **Step** | A single task (run a command or use an Action) |
| **Action** | A reusable unit of code (from GitHub Marketplace or custom) |
| **Runner** | The machine that runs the job (GitHub-hosted or self-hosted) |

```yaml
# .github/workflows/ci.yaml
name: CI Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  build-and-test:
    runs-on: ubuntu-latest           # GitHub-hosted runner
    
    steps:
      - name: Checkout code
        uses: actions/checkout@v4     # Reusable Action from Marketplace
      
      - name: Set up Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
      
      - name: Install dependencies
        run: npm ci                   # Shell command
      
      - name: Run tests
        run: npm test
      
      - name: Build
        run: npm run build

  deploy:
    needs: build-and-test             # Depends on previous job
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'    # Only on main branch
    
    steps:
      - name: Deploy to production
        run: echo "Deploying..."
```

**Why GitHub Actions is popular:**
- **Zero setup** вҖ” Built into GitHub, no separate server needed
- **Free for public repos** вҖ” Generous free tier for private repos too
- **Marketplace** вҖ” Thousands of pre-built Actions
- **Matrix builds** вҖ” Test across multiple OS/language versions easily

### Q7. What are Build Artifacts?

**Answer:**

A **build artifact** is the **output** of your build process вҖ” the thing that actually gets deployed. It's the finished product.

**Examples of artifacts:**

| Application Type | Artifact |
|-----------------|---------|
| Java app | JAR or WAR file |
| Node.js app | Docker image |
| Python app | Docker image or wheel package |
| Go app | Binary executable |
| Frontend app | Static files (HTML, CSS, JS bundle) |
| Infrastructure | Terraform plan, Helm chart |

**Where artifacts are stored:**
- **Docker images** вҶ’ Container registries (ECR, DockerHub, Harbor)
- **JAR/WAR** вҶ’ Nexus, JFrog Artifactory
- **npm packages** вҶ’ npm registry, Artifactory
- **Helm charts** вҶ’ Helm repo, OCI registry

```yaml
# GitHub Actions вҖ” upload artifact
- name: Build
  run: npm run build

- name: Upload artifact
  uses: actions/upload-artifact@v4
  with:
    name: build-output
    path: dist/
```

**Best practices:**
- Tag artifacts with version/build number (never `latest` in production)
- Store artifacts in a proper registry (not just on the CI server)
- Scan artifacts for vulnerabilities before deploying

### Q8. What is Webhook? How is it used in CI/CD?

**Answer:**

A **Webhook** is an HTTP callback вҖ” when an event happens in one system, it sends an HTTP POST request to another system.

**In CI/CD:** GitHub/GitLab sends a webhook to Jenkins/other CI tool whenever code is pushed.

```
Developer pushes code to GitHub
        в”Ӯ
        в–ј
GitHub sends HTTP POST to Jenkins webhook URL
(https://jenkins.company.com/github-webhook/)
        в”Ӯ
        в–ј
Jenkins receives the request вҶ’ starts the pipeline
```

**Why webhooks?**
- **Real-time** вҖ” Pipeline starts immediately after push
- **Efficient** вҖ” No polling needed (Jenkins doesn't keep asking "any new commits?")

**Alternative: Polling (less efficient)**
```
Jenkins checks GitHub every 5 minutes: "Any new commits?"
  вҶ’ No
  вҶ’ No
  вҶ’ No
  вҶ’ Yes! вҶ’ Start pipeline
```

Polling wastes resources. Webhooks are **event-driven** вҖ” much better.

### Q9. What is the role of version control in CI/CD?

**Answer:**

Version control (Git) is the **foundation** of CI/CD. Without it, CI/CD doesn't work.

**How Git enables CI/CD:**

1. **Trigger** вҖ” Every `git push` or PR triggers the pipeline
2. **Source of truth** вҖ” Pipeline code (Jenkinsfile, `.github/workflows/`) is stored IN the repo
3. **Branching strategy** вҖ” Different branches trigger different actions:
   - `feature/*` вҶ’ Run tests only
   - `develop` вҶ’ Deploy to staging
   - `main` вҶ’ Deploy to production
4. **Traceability** вҖ” Every deployment links back to a specific commit
5. **Collaboration** вҖ” PR reviews ensure code quality before merging

```
feature/login в”Җв”ҖPRв”Җв”ҖвҶ’ develop в”Җв”Җmergeв”Җв”ҖвҶ’ main
     в”Ӯ                   в”Ӯ                 в”Ӯ
     в–ј                   в–ј                 в–ј
  Run tests         Deploy to           Deploy to
  only              staging             production
```

### Q10. What are environment variables in CI/CD? How do you manage secrets?

**Answer:**

**Environment variables** store configuration that changes between environments (dev, staging, prod) and sensitive data like API keys.

```yaml
# GitHub Actions вҖ” environment variables
env:
  APP_ENV: production           # Plain text вҖ” OK for non-sensitive

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Deploy
        env:
          AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}         # From secrets store
          AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}  # NEVER hardcode
        run: aws s3 sync ./dist s3://my-bucket
```

**How to manage secrets (DO's and DON'Ts):**

| вқҢ DON'T | вң… DO |
|----------|------|
| Hardcode secrets in code | Use CI/CD secrets management |
| Commit `.env` files to Git | Add `.env` to `.gitignore` |
| Print secrets in logs | Mask secrets in pipeline output |
| Share secrets in Slack/email | Use a secrets manager (Vault, AWS SM) |

**Secrets management in CI/CD tools:**
- **Jenkins** вҶ’ Credentials plugin (username/password, secret text, SSH key)
- **GitHub Actions** вҶ’ Repository/Organization/Environment secrets
- **GitLab CI** вҶ’ CI/CD Variables (masked, protected)
- **External** вҶ’ HashiCorp Vault, AWS Secrets Manager

3. **Cache dependencies** вҖ” Don't download npm/maven packages on every run
4. **Immutable artifacts** вҖ” Build the Docker image ONCE, deploy the SAME image everywhere
5. **Environment parity** вҖ” Staging should mirror production
6. **Automated rollback** вҖ” If deploy fails, roll back automatically
7. **Notifications** вҖ” Alert the team on failures (Slack, email)

### Q13. What is a Jenkins Agent? What are the types?

**Answer:**

A Jenkins **Agent** (formerly "slave") is a machine that actually **runs the build jobs**. The Jenkins Master (controller) only orchestrates вҖ” agents do the heavy lifting.

**Types of agents:**

| Type | Description | Best For |
|------|------------|---------|

```groovy
// Using a Docker agent
pipeline {

            steps {

                      command: ['cat']
```

### Q14. What is a Jenkins Shared Library?

**Answer:**

A **Shared Library** is a way to **reuse pipeline code** across multiple Jenkins pipelines. Instead of copy-pasting the same Groovy code everywhere, you put it in a shared library.

**Think of it as:** Functions/modules that any Jenkinsfile can import and use.

**Structure:**

```

    deployToK8s.groovy
    notifySlack.groovy
  src/                     # Groovy classes (optional)
    com/company/
      Pipeline.groovy
  resources/               # Non-Groovy files (config, scripts)
    scripts/
      deploy.sh
```

```groovy
// vars/buildDocker.groovy
def call(Map config) {
    sh """
        docker build -t ${config.registry}/${config.image}:${config.tag} .
        docker push ${config.registry}/${config.image}:${config.tag}
    """
}
```

```groovy
// Jenkinsfile вҖ” using the shared library
@Library('my-shared-library') _

pipeline {
    agent any
    stages {
        stage('Build & Push') {
            steps {
                buildDocker(
                    registry: 'registry.company.com',
                    image: 'my-app',
                    tag: env.BUILD_NUMBER

            }
        }
        stage('Deploy') {
            steps {
                deployToK8s(environment: 'production')

        }

        failure {

    }
```

### Q16. What is a Matrix Build?

**Answer:**

A **matrix build** runs the same job with **multiple combinations** of parameters вҖ” like testing your app on different OS versions and language versions simultaneously.

```yaml
# GitHub Actions matrix
jobs:
  test:
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
        node-version: [18, 20, 22]
        # This creates 3 Г— 3 = 9 parallel jobs!
    
    runs-on: ${{ matrix.os }}
    
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
      - run: npm ci
      - run: npm test
```

This runs **9 jobs in parallel:**
- Ubuntu + Node 18, Ubuntu + Node 20, Ubuntu + Node 22
- Windows + Node 18, Windows + Node 20, Windows + Node 22
- macOS + Node 18, macOS + Node 20, macOS + Node 22

**Why?** Ensures your app works across all platforms and versions.

### Q18. What is Caching in CI/CD pipelines? Why is it important?

**Answer:**

**Caching** stores files (dependencies, build outputs) between pipeline runs so you don't have to download or rebuild them every time.

**Without caching:** Every pipeline run downloads ALL dependencies from scratch (slow!).
**With caching:** Dependencies are cached вҖ” if nothing changed, use the cache (fast!).

```yaml
# GitHub Actions вҖ” cache npm dependencies
- name: Cache node modules
  uses: actions/cache@v4
  with:
    path: ~/.npm
    key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }}
    restore-keys: |
      ${{ runner.os }}-node-

- name: Install dependencies
  run: npm ci      # Uses cache if package-lock.json hasn't changed
```

```groovy
// Jenkins вҖ” cache with stash/unstash
stage('Build') {
    steps {
        sh 'npm ci'
        stash includes: 'node_modules/**', name: 'node-modules'
    }
}
stage('Test') {
    steps {
        unstash 'node-modules'    // Reuse from previous stage
        sh 'npm test'
    }
}
```

**What to cache:**
- `node_modules` / `.npm` (Node.js)
- `~/.m2/repository` (Maven)
- `~/.gradle/caches` (Gradle)
- Docker layers
- Go modules

### Q20. What is Tekton? How is it different from Jenkins?

### Q21. What are deployment strategies? Explain Blue/Green, Canary, and Rolling.

**Answer:**

These are strategies for deploying new versions of your application to production **with minimal risk**.

**1. Rolling Update (Default in Kubernetes):**

Gradually replaces old pods with new ones, one at a time.

```
Before:  [v1] [v1] [v1] [v1]
Step 1:  [v1] [v1] [v1] [v2]
Step 2:  [v1] [v1] [v2] [v2]
Step 3:  [v1] [v2] [v2] [v2]
Step 4:  [v2] [v2] [v2] [v2]
```

- вң… No downtime, gradual rollout
- вқҢ Two versions running simultaneously (might cause issues)

**2. Blue/Green Deployment:**

Run two identical environments вҖ” Blue (current) and Green (new). Switch traffic all at once.

```
                         в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
                    в”Ңв”Җв”ҖвҶ’ в”Ӯ Blue (v1)   в”Ӯ  вҶҗ Currently serving traffic
Load Balancer в”Җв”Җв”Җв”Җв”Җв”Җв”Ө    в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
                    в”Ӯ    в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
                    в””в”Җв”ҖвҶ’ в”Ӯ Green (v2)  в”Ӯ  вҶҗ New version (testing)
                         в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ

After testing:
                         в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
                    в”Ңв”Җв”ҖвҶ’ в”Ӯ Blue (v1)   в”Ӯ  вҶҗ Idle (kept for rollback)
Load Balancer в”Җв”Җв”Җв”Җв”Җв”Җв”Ө    в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
                    в”Ӯ    в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
                    в””в”Җв”ҖвҶ’ в”Ӯ Green (v2)  в”Ӯ  вҶҗ Now serving traffic!
                         в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

- вң… Instant rollback (switch back to Blue)
- вң… Test the new version before switching
- вқҢ Requires double infrastructure ($$)

**3. Canary Deployment:**

Send a small percentage of traffic to the new version. Gradually increase if things look good.

```
Step 1: 95% вҶ’ v1,  5% вҶ’ v2    (test with 5%)
Step 2: 80% вҶ’ v1, 20% вҶ’ v2    (looking good, increase)
Step 3: 50% вҶ’ v1, 50% вҶ’ v2    (still good, increase more)
Step 4:  0% вҶ’ v1, 100% вҶ’ v2   (full rollout!)
```

- вң… Lowest risk (only a small percentage of users see the new version)
- вң… Real production testing
- вқҢ More complex to implement
- вқҢ Need good observability to monitor canary traffic

### Q22. What are Feature Flags?

**Answer:**

**Feature flags** (feature toggles) let you **enable or disable features** without deploying new code. It's like having a switch for each feature.

```python
# Without feature flags
def get_checkout_page():
    return render_new_checkout()    # New feature вҖ” risky!

# With feature flags
def get_checkout_page():
    if feature_flags.is_enabled("new_checkout"):    # Toggle!
        return render_new_checkout()
    else:
        return render_old_checkout()
```

**Why feature flags?**
- **Decouple deployment from release** вҖ” Deploy code but don't activate it yet
- **Gradual rollout** вҖ” Enable for 10% of users, then 50%, then 100%
- **Kill switch** вҖ” Instantly disable a broken feature without redeploying
- **A/B testing** вҖ” Show different features to different user groups

**Popular tools:**
- **LaunchDarkly** вҖ” Enterprise feature flag platform
- **Unleash** вҖ” Open-source
- **Flagsmith** вҖ” Open-source
- **Split.io** вҖ” A/B testing focused
- **ConfigCat** вҖ” Simple and affordable

### Q24. What is a Multi-Branch Pipeline in Jenkins?

### Q25. How do you handle database migrations in CI/CD?

**Answer:**

Database migrations (schema changes) are one of the **trickiest parts** of CI/CD because databases are stateful вҖ” you can't just "restart" them like a container.

**Best practices:**

1. **Use migration tools:**
   - **Flyway** вҖ” Version-based migrations (SQL files like `V1__create_users.sql`)
   - **Liquibase** вҖ” Changelog-based migrations (XML, YAML, SQL)
   - **Alembic** вҖ” For Python (SQLAlchemy)
   - **Rails Migrations** вҖ” For Ruby on Rails
   - **Prisma Migrate** вҖ” For Node.js

2. **Run migrations as part of the pipeline:**

```yaml
# GitHub Actions
jobs:
  migrate:
    runs-on: ubuntu-latest
    steps:
      - name: Run database migration
        run: flyway -url=jdbc:postgresql://db:5432/myapp migrate

  deploy:
    needs: migrate          # Deploy AFTER migration succeeds
    steps:
      - name: Deploy app
        run: kubectl apply -f deployment.yaml
```

3. **Make migrations backward-compatible:**
   - Add new columns with default values (don't break old code)
   - Don't rename/delete columns in the same release
   - Use **expand and contract pattern:**
     1. Deploy migration (add new column)
     2. Deploy new code (writes to both old and new columns)
     3. Migrate data
     4. Deploy final code (reads from new column only)
     5. Drop old column

4. **Always backup before migrating!**

```
в”Ӯ      в”Ӯ   в”Ӯ (Static) в”Ӯ   в”Ӯ (Deps)   в”Ӯ   в”Ӯ (Dynamic) в”Ӯ   в”Ӯ (Container) в”Ӯ
в””в”Җв”Җв”Җв”Җв”Җв”Җв”ҳ   в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ   в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ   в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ   в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

|---------------|--------------|------|-------|
| **SAST** (Static Analysis) | Source code for vulnerabilities | At build time | SonarQube, Semgrep, CodeQL |
| **SCA** (Software Composition) | Dependencies for known CVEs | At build time | Snyk, Dependabot, Trivy |
| **DAST** (Dynamic Analysis) | Running app for vulnerabilities | After deployment to staging | OWASP ZAP, Burp Suite |

| **IaC Scanning** | Terraform/K8s configs for misconfigurations | At PR/build time | Checkov, tfsec, KICS |
| **Secret Scanning** | Code for hardcoded secrets | At commit time (pre-commit) | GitLeaks, TruffleHog |

### Q27. How do you implement rollback in CI/CD?

**Answer:**

**Rollback** = Going back to a previous working version when something goes wrong.

**Approaches:**

**1. Kubernetes rollback:**
```bash
# Rollback to previous version
kubectl rollout undo deployment/my-app

# Rollback to specific revision
kubectl rollout undo deployment/my-app --to-revision=3
```

**2. Re-deploy previous artifact:**
```yaml
# In your pipeline, keep track of the previous working image
# If deploy fails, deploy the previous image
- name: Rollback on failure
  if: failure()
  run: |
    kubectl set image deployment/my-app \
      my-app=registry.company.com/my-app:${{ env.PREVIOUS_VERSION }}
```

**3. GitOps rollback (with ArgoCD):**
```bash
# Just revert the commit in Git вҖ” ArgoCD auto-syncs
git revert HEAD
git push
# ArgoCD detects the change вҶ’ deploys previous version
```

**4. Blue/Green rollback:**
```bash
# Just switch the load balancer back to the blue environment
# Instant rollback!
```

**5. Automated rollback with health checks:**
```yaml
# ArgoCD вҖ” automatic rollback if health checks fail
spec:
  syncPolicy:
    automated:
      selfHeal: true
    rollbackOnError: true
```

**Best practices:**
- Always keep at least 3-5 previous versions/images
- Test rollback procedures regularly
- Have runbooks for manual rollbacks
- Monitor after deployment вҖ” automated rollback on error spike

### Q28. What is the difference between `docker build` in CI vs CD?

**Answer:**

| Phase | What Happens | Example |
|-------|-------------|---------|
| **CI (Build phase)** | Docker image is **built, tested, and pushed** to a registry | `docker build -t my-app:v1.2 . && docker push` |
| **CD (Deploy phase)** | The **same image** is pulled and deployed to environments | `kubectl set image deployment/my-app my-app=my-app:v1.2` |

**Critical rule:** **Build once, deploy everywhere.** The same image deployed to staging MUST be the exact same image deployed to production. Never rebuild for production!

```
CI:
  Build image в”Җв”ҖвҶ’ Push to registry в”Җв”ҖвҶ’ my-app:abc123

CD:
  Staging:    Pull my-app:abc123 в”Җв”ҖвҶ’ Deploy
  Production: Pull my-app:abc123 в”Җв”ҖвҶ’ Deploy  (SAME image!)
```

**Configuration differences** (dev vs prod) should come from:
- Environment variables
- ConfigMaps / Secrets
- Helm values files

NOT from different Docker images.

### Q29. What is Trunk-Based Development and how does it affect CI/CD?

**Answer:**

**Trunk-Based Development** is a branching strategy where all developers commit to a **single branch** (main/trunk) вҖ” no long-lived feature branches.

```
# Traditional Git Flow (long-lived branches)
main в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
  в””в”Җв”Җ develop в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
         в”ңв”Җв”Җ feature/login (lives for 2 weeks)
         в””в”Җв”Җ feature/cart (lives for 3 weeks)

# Trunk-Based Development (short-lived branches)
main в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
  в”ңв”Җв”Җ feat/login (lives for 1-2 days max)
  в”ңв”Җв”Җ feat/cart (lives for 1 day)
  в””в”Җв”Җ feat/search (lives for a few hours)
```

**How it affects CI/CD:**
- CI runs on **every commit** to main
- Merges happen **multiple times per day**
- Feature flags are used to hide incomplete features
- Small, incremental changes = lower risk
- Pipeline must be **fast** (developers merge frequently)

**Requirements for Trunk-Based Development:**
- Strong CI pipeline (tests must be reliable)
- Feature flags
- Good code review process (short PRs)
- Automated testing (high coverage)

### Q30. How do you measure CI/CD pipeline performance?

**Answer:**

**Key metrics (DORA metrics):**

The **DORA (DevOps Research and Assessment)** metrics are the industry standard:

| Metric | What It Measures | Elite Target |
|--------|-----------------|-------------|
| **Deployment Frequency** | How often you deploy to production | Multiple times per day |
| **Lead Time for Changes** | Time from commit to production | Less than 1 hour |
| **Change Failure Rate** | % of deployments causing failures | 0-15% |
| **Mean Time to Recovery (MTTR)** | How fast you recover from failures | Less than 1 hour |

**Pipeline-specific metrics:**

| Metric | What It Measures | Good Target |
|--------|-----------------|-------------|
| **Build time** | Time to build the application | < 5 minutes |
| **Test time** | Time to run all tests | < 10 minutes |
| **Total pipeline time** | End-to-end pipeline duration | < 15 minutes |
| **Pipeline success rate** | % of pipeline runs that succeed | > 95% |
| **Flaky test rate** | % of tests that fail intermittently | < 1% |
| **Queue time** | Time waiting for an available runner | < 2 minutes |

**How to measure:**
- **Jenkins** вҶ’ Build History, Blue Ocean, Prometheus plugin
- **GitHub Actions** вҶ’ Workflow run duration, Actions metrics
- **GitLab** вҶ’ CI/CD Analytics, DORA metrics dashboard
- **External** вҶ’ LinearB, Sleuth, Faros, Jellyfish

```yaml
# GitHub Actions вҖ” add timing to your pipeline
- name: Report pipeline metrics
  run: |
    echo "Pipeline Duration: ${{ github.event.workflow_run.run_duration }}"
    # Send to Prometheus/Datadog for tracking
```

**Tips to improve pipeline speed:**
1. **Cache dependencies** вҖ” Don't download everything each time
2. **Parallelize stages** вҖ” Run tests and scans simultaneously
3. **Use faster runners** вҖ” More CPU/memory = faster builds
4. **Optimize Docker builds** вҖ” Multi-stage builds, layer caching
5. **Split large test suites** вҖ” Run test groups in parallel
6. **Fail fast** вҖ” Put quick checks (lint, format) first
7. **Only build what changed** вҖ” Skip unchanged services in monorepos
