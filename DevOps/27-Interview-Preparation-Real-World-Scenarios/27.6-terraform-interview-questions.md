## Core Concepts

### Q1. What is Terraform? Why do we use it?

**Answer:**

**Terraform** is an open-source **Infrastructure as Code (IaC)** tool created by HashiCorp. It lets you **define, provision, and manage infrastructure** using code (instead of clicking around in the AWS/Azure/GCP console).

**Why Terraform?**

Imagine you need to set up:
- 3 EC2 instances
- 1 VPC with 4 subnets
- 2 Security Groups
- 1 RDS database
- 1 Load Balancer

**Without Terraform:** Click through the AWS console, remember every setting, and do it again for staging and production. 😩

**With Terraform:** Write it ONCE in code, and create identical infrastructure with one command. 🎉

**Key benefits:**
- **Reproducible** — Same code = same infrastructure every time
- **Version controlled** — Infrastructure changes tracked in Git
- **Multi-cloud** — Works with AWS, Azure, GCP, Kubernetes, and 3,000+ providers
- **Plan before apply** — See what will change before making changes
- **Declarative** — You describe WHAT you want, not HOW to do it

```hcl
# This creates an EC2 instance — that's it!
resource "aws_instance" "web_server" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.micro"
  
  tags = {
    Name        = "web-server"
    Environment = "production"
  }
}
```

```bash
terraform init     # Initialize — download providers
terraform plan     # Preview — show what will happen
terraform apply    # Execute — create the infrastructure
terraform destroy  # Cleanup — delete everything
```

### Q2. Explain Terraform's architecture. How does it work?

**Answer:**

Terraform has a simple but powerful workflow:

```
┌─────────────┐     ┌────────────┐     ┌──────────────┐
│  .tf files  │────→│  Terraform │────→│   Provider   │
│  (your code)│     │  Core      │     │  (AWS, Azure)│
└─────────────┘     └─────┬──────┘     └──────┬───────┘
                          │                    │
                    ┌─────▼──────┐      ┌─────▼───────┐
                    │   State    │      │  Real       │
                    │   File    │      │  Infrastructure│
                    │(.tfstate) │      │  (AWS, etc.)  │
                    └────────────┘      └──────────────┘
```

**Components:**

| Component | What It Does |
|-----------|-------------|
| **Terraform Core** | Reads `.tf` files, builds a dependency graph, determines what changes to make |
| **Providers** | Plugins that know how to talk to specific services (AWS, Azure, GCP, K8s) |
| **State File** | Records what infrastructure Terraform has created (maps code to real resources) |
| **.tf Files** | Your infrastructure code (HCL — HashiCorp Configuration Language) |

**The Terraform Workflow:**

```
Write ──→ Init ──→ Plan ──→ Apply
  │         │        │        │
  │         │        │        └─ Create/update/delete real resources
  │         │        └─ Show a preview of changes (diff)
  │         └─ Download providers and initialize backend
  └─ Write .tf files describing desired infrastructure
```

### Q3. What is a Terraform Provider?

**Answer:**

A **Provider** is a **plugin** that lets Terraform interact with a specific service or platform. It translates your Terraform code into API calls.

**Think of it as:** A translator between Terraform and the cloud platform.

```hcl
# Configure the AWS provider
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"       # Use version 5.x
    }
  }
}

provider "aws" {
  region = "us-east-1"
}

# Now you can create AWS resources!
resource "aws_s3_bucket" "my_bucket" {
  bucket = "my-unique-bucket-name"
}
```

**Popular providers:**

| Provider | What It Manages |
|----------|----------------|
| `hashicorp/aws` | AWS resources (EC2, S3, RDS, etc.) |
| `hashicorp/azurerm` | Azure resources |
| `hashicorp/google` | Google Cloud resources |
| `hashicorp/kubernetes` | Kubernetes resources |
| `hashicorp/helm` | Helm charts |
| `hashicorp/vault` | HashiCorp Vault |
| `integrations/github` | GitHub repos, teams, etc. |

**Version constraints:**
- `"= 5.0.0"` → Exactly this version
- `"~> 5.0"` → Any 5.x version (5.0, 5.1, 5.99 but NOT 6.0)
- `">= 5.0, < 6.0"` → Range

### Q4. What are Resources and Data Sources in Terraform?

**Answer:**

| Concept | What It Does | Keyword |
|---------|-------------|---------|
| **Resource** | **Creates** and manages infrastructure | `resource` |
| **Data Source** | **Reads** existing infrastructure (doesn't create anything) | `data` |

```hcl
# RESOURCE — Creates a new VPC
resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"
  tags = {
    Name = "main-vpc"
  }
}

# DATA SOURCE — Reads an existing VPC (created outside Terraform)
data "aws_vpc" "existing" {
  id = "vpc-abc123"    # Look up this specific VPC
}

# DATA SOURCE — Look up the latest Amazon Linux AMI
data "aws_ami" "amazon_linux" {
  most_recent = true
  owners      = ["amazon"]
  
  filter {
    name   = "name"
    values = ["amzn2-ami-hvm-*-x86_64-gp2"]
  }
}

# Use the data source result
resource "aws_instance" "web" {
  ami           = data.aws_ami.amazon_linux.id    # Use the AMI we found
  instance_type = "t3.micro"
  subnet_id     = data.aws_vpc.existing.id
}
```

**When to use Data Sources:**
- Reference existing resources not managed by Terraform
- Look up dynamic values (latest AMI, availability zones)
- Read information from other Terraform states

### Q5. What are Variables in Terraform? What types are there?

**Answer:**

Variables make your Terraform code **reusable and flexible**. Instead of hardcoding values, you use variables.

**Three types of variables:**

| Type | Purpose | Keyword |
|------|---------|---------|
| **Input Variables** | Accept values from the user | `variable` |
| **Output Variables** | Display/export values after apply | `output` |
| **Local Variables** | Define computed values used within the module | `locals` |

```hcl
# --- INPUT VARIABLES (variables.tf) ---
variable "instance_type" {
  description = "EC2 instance type"
  type        = string
  default     = "t3.micro"
}

variable "environment" {
  description = "Environment name"
  type        = string
  
  validation {
    condition     = contains(["dev", "staging", "prod"], var.environment)
    error_message = "Environment must be dev, staging, or prod."
  }
}

variable "allowed_ports" {
  description = "List of allowed ports"
  type        = list(number)
  default     = [80, 443, 8080]
}

variable "tags" {
  description = "Resource tags"
  type        = map(string)
  default     = {
    Team    = "devops"
    Project = "web-app"
  }
}

# --- USING VARIABLES ---
resource "aws_instance" "web" {
  instance_type = var.instance_type      # Use with var.<name>
  tags          = var.tags
}

# --- OUTPUT VARIABLES (outputs.tf) ---
output "instance_ip" {
  description = "The public IP of the instance"
  value       = aws_instance.web.public_ip
}

output "instance_id" {
  value = aws_instance.web.id
}

# --- LOCAL VARIABLES ---
locals {
  name_prefix = "${var.environment}-${var.project}"
  common_tags = merge(var.tags, {
    Environment = var.environment
    ManagedBy   = "terraform"
  })
}

resource "aws_instance" "web" {
  tags = local.common_tags    # Use with local.<name>
}
```

**Variable types:**
- `string` — `"hello"`
- `number` — `42`
- `bool` — `true` or `false`
- `list(type)` — `["a", "b", "c"]`
- `map(type)` — `{key = "value"}`
- `set(type)` — Like list but unique values
- `object({...})` — Complex structure
- `tuple([...])` — Mixed-type list

**Ways to set variable values (in order of precedence):**
1. `-var` flag: `terraform apply -var="instance_type=t3.large"`
2. `-var-file` flag: `terraform apply -var-file="prod.tfvars"`
3. `terraform.tfvars` or `*.auto.tfvars` files (auto-loaded)
4. `TF_VAR_` environment variables: `export TF_VAR_instance_type=t3.large`
5. Default value in the variable block
6. Interactive prompt (if no default and no value provided)

### Q6. What is `terraform init`? What does it do?

**Answer:**

`terraform init` is the **first command** you run in any Terraform project. It initializes the working directory.

**What it does:**

1. **Downloads providers** — Fetches the provider plugins (AWS, Azure, etc.) specified in your code
2. **Initializes backend** — Sets up where state will be stored (local or remote)
3. **Downloads modules** — Fetches any referenced modules
4. **Creates `.terraform` directory** — Stores downloaded plugins and modules
5. **Creates `.terraform.lock.hcl`** — Locks provider versions (like package-lock.json)

```bash
$ terraform init

Initializing the backend...
Initializing provider plugins...
- Finding hashicorp/aws versions matching "~> 5.0"...
- Installing hashicorp/aws v5.31.0...
- Installed hashicorp/aws v5.31.0

Terraform has been successfully initialized!
```

**When to re-run `terraform init`:**
- Added a new provider
- Changed the backend configuration
- Added a new module
- Changed provider version constraints

### Q7. What is `terraform plan`? Why is it important?

**Answer:**

`terraform plan` creates an **execution plan** — it shows you what Terraform WILL do **without actually doing it**. It's a **dry run**.

```bash
$ terraform plan

Terraform will perform the following actions:

  # aws_instance.web will be created
  + resource "aws_instance" "web" {
      + ami           = "ami-0c55b159cbfafe1f0"
      + instance_type = "t3.micro"
      + id            = (known after apply)
      + public_ip     = (known after apply)
      + tags          = {
          + "Name" = "web-server"
        }
    }

Plan: 1 to add, 0 to change, 0 to destroy.
```

**Understanding the plan output:**

| Symbol | Meaning |
|--------|---------|
| `+` (green) | Resource will be **created** |
| `-` (red) | Resource will be **destroyed** |
| `~` (yellow) | Resource will be **modified in-place** |
| `-/+` (red/green) | Resource will be **destroyed and recreated** (replacement) |

**Why it's critical:**
- **Safety** — See what will change before making changes
- **Review** — In CI/CD, plan output can be reviewed before apply
- **No surprises** — Catch accidental deletions or replacements
- **Save plan** — `terraform plan -out=plan.tfplan` saves the plan for later apply

```bash
# Save plan to a file
terraform plan -out=plan.tfplan

# Apply the saved plan (no confirmation needed — it's exactly what was planned)
terraform apply plan.tfplan
```

### Q8. What is `terraform apply` and `terraform destroy`?

**Answer:**

**`terraform apply`** — Creates or updates infrastructure to match your code.
**`terraform destroy`** — Deletes ALL infrastructure managed by Terraform.

```bash
# Apply — creates/updates resources
terraform apply                     # Interactive (asks for confirmation)
terraform apply -auto-approve       # Skip confirmation (use in CI/CD)
terraform apply plan.tfplan         # Apply a saved plan

# Destroy — deletes everything
terraform destroy                   # Interactive
terraform destroy -auto-approve     # Skip confirmation

# Destroy specific resource only
terraform destroy -target=aws_instance.web
```

**The apply process:**

```
terraform apply
      │
      ├─ Reads .tf files
      ├─ Reads state file
      ├─ Compares desired vs actual state
      ├─ Creates execution plan
      ├─ Shows plan and asks for confirmation
      ├─ Makes API calls to create/update/delete resources
      └─ Updates state file
```

### Q9. What is HCL (HashiCorp Configuration Language)?

**Answer:**

**HCL** is the language you write Terraform code in. It's designed to be **human-readable** while being machine-friendly.

**Basic syntax:**

```hcl
# This is a comment

# Resource block
resource "aws_instance" "example" {    # resource TYPE "NAME"
  ami           = "ami-abc123"          # argument = value
  instance_type = "t3.micro"
  
  tags = {                              # Map
    Name = "example"
  }
}

# Strings
name = "hello"

# Numbers
count = 3

# Booleans
enabled = true

# Lists
ports = [80, 443, 8080]

# Maps
tags = {
  env  = "prod"
  team = "devops"
}

# Multi-line string (heredoc)
user_data = <<-EOF
  #!/bin/bash
  echo "Hello World"
  apt-get update
EOF

# String interpolation
name = "server-${var.environment}"

# Conditional
instance_type = var.environment == "prod" ? "t3.large" : "t3.micro"
```

### Q10. What is the difference between `terraform.tfvars` and `variables.tf`?

**Answer:**

| File | Purpose | Contains |
|------|---------|----------|
| `variables.tf` | **Declares** variables (name, type, description, default) | Variable definitions |
| `terraform.tfvars` | **Assigns values** to variables | Variable values |

**Think of it as:** `variables.tf` = "What questions can you ask?" `terraform.tfvars` = "Here are the answers."

```hcl
# variables.tf — DECLARES the variables
variable "region" {
  description = "AWS region"
  type        = string
}

variable "instance_type" {
  description = "EC2 instance type"
  type        = string
  default     = "t3.micro"
}

variable "environment" {
  description = "Environment name"
  type        = string
}
```

```hcl
# terraform.tfvars — ASSIGNS values
region        = "us-east-1"
instance_type = "t3.large"
environment   = "production"
```

```hcl
# You can have multiple .tfvars files for different environments
# dev.tfvars
region        = "us-east-1"
instance_type = "t3.micro"
environment   = "dev"

# prod.tfvars
region        = "us-east-1"
instance_type = "t3.large"
environment   = "production"
```

```bash
# Use a specific tfvars file
terraform apply -var-file="prod.tfvars"
```

**Auto-loaded files:**
- `terraform.tfvars` — Always loaded automatically
- `*.auto.tfvars` — Always loaded automatically
- Any other `.tfvars` files — Must be passed with `-var-file`

### Q11. What is the Terraform dependency graph?

**Answer:**

Terraform automatically builds a **dependency graph** of all resources and processes them in the correct order.

```hcl
# Terraform knows to create the VPC first, then the subnet, then the instance
resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"
}

resource "aws_subnet" "public" {
  vpc_id     = aws_vpc.main.id          # Depends on VPC
  cidr_block = "10.0.1.0/24"
}

resource "aws_instance" "web" {
  subnet_id = aws_subnet.public.id      # Depends on subnet
  ami       = "ami-abc123"
  instance_type = "t3.micro"
}
```

**Dependency order:**
```
aws_vpc.main → aws_subnet.public → aws_instance.web
```

Terraform figures this out **automatically** by looking at references (`aws_vpc.main.id`).

**Explicit dependency (when there's no direct reference):**

```hcl
resource "aws_instance" "web" {
  ami           = "ami-abc123"
  instance_type = "t3.micro"
  
  depends_on = [aws_iam_role_policy.s3_access]    # Explicit dependency
}
```

```bash
# Visualize the dependency graph
terraform graph | dot -Tpng > graph.png
```

### Q12. What are Provisioners in Terraform? Should you use them?

**Answer:**

**Provisioners** execute scripts or commands on resources **after they're created** (or before they're destroyed).

**Types:**

| Provisioner | What It Does |
|------------|-------------|
| `local-exec` | Runs a command on YOUR machine (where Terraform runs) |
| `remote-exec` | Runs a command on the REMOTE resource (via SSH or WinRM) |
| `file` | Copies files from your machine to the remote resource |

```hcl
resource "aws_instance" "web" {
  ami           = "ami-abc123"
  instance_type = "t3.micro"

  # Run on the remote server after creation
  provisioner "remote-exec" {
    inline = [
      "sudo apt-get update",
      "sudo apt-get install -y nginx",
      "sudo systemctl start nginx"
    ]
    
    connection {
      type        = "ssh"
      user        = "ubuntu"
      private_key = file("~/.ssh/id_rsa")
      host        = self.public_ip
    }
  }

  # Run on YOUR local machine
  provisioner "local-exec" {
    command = "echo ${self.public_ip} >> inventory.txt"
  }

  # Run before destroying the resource
  provisioner "local-exec" {
    when    = destroy
    command = "echo 'Server ${self.id} is being destroyed!'"
  }
}
```

**⚠️ Should you use provisioners? USUALLY NO.**

HashiCorp themselves say provisioners are a **last resort**. Better alternatives:

| Instead of | Use |
|-----------|-----|
| `remote-exec` to configure servers | **Ansible**, **user_data**, **cloud-init** |
| `local-exec` to run scripts | **null_resource** with `triggers`, or CI/CD pipeline |
| `file` provisioner | **Packer** to bake AMIs with everything pre-installed |

```hcl
# Better approach: use user_data instead of provisioners
resource "aws_instance" "web" {
  ami           = "ami-abc123"
  instance_type = "t3.micro"
  
  user_data = <<-EOF
    #!/bin/bash
    apt-get update
    apt-get install -y nginx
    systemctl start nginx
  EOF
}
```

## State Management

### Q13. What is Terraform State? Why is it important?

**Answer:**

The **state file** (`terraform.tfstate`) is a JSON file that maps your Terraform code to real-world resources. It's how Terraform knows **what it has already created**.

**What's in the state file:**
- Every resource Terraform manages
- Resource IDs, attributes, and metadata
- Dependencies between resources
- Provider configuration

**Why is it critical?**

Without the state file, Terraform has NO idea what exists. It would try to create everything from scratch every time.

```
Your Code (.tf files)     State File (.tfstate)     Real World (AWS)
─────────────────────     ─────────────────────     ─────────────────
aws_instance "web"   ──→  id: "i-abc123"       ──→  EC2 Instance
aws_s3_bucket "logs" ──→  bucket: "my-logs"    ──→  S3 Bucket
```

**What happens during `terraform plan`:**

```
1. Read .tf files → "I WANT 3 EC2 instances"
2. Read state file → "I currently HAVE 2 EC2 instances"  
3. Compare → "I need to CREATE 1 more instance"
```

**⚠️ Critical rules:**
- NEVER edit the state file manually
- NEVER commit the state file to Git (it may contain secrets!)
- ALWAYS use remote state in a team environment
- ALWAYS backup your state file

### Q14. What is Remote State? Why should you use it?

**Answer:**

By default, Terraform stores state **locally** in a `terraform.tfstate` file. **Remote state** stores it in a **shared, remote location**.

**Problems with local state:**
- Only YOU have the state file — teammates can't use Terraform
- If your laptop dies, the state is gone
- No locking — two people can run `terraform apply` at the same time and corrupt things
- State might contain secrets — sitting on your local machine is risky

**Remote state solves all of this!**

```hcl
# Backend configuration — store state in AWS S3
terraform {
  backend "s3" {
    bucket         = "my-terraform-state"
    key            = "production/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true                          # Encrypt state at rest
    dynamodb_table = "terraform-lock"              # State locking!
  }
}
```

**Common remote backends:**

| Backend | Provider | Locking |
|---------|----------|---------|
| **S3 + DynamoDB** | AWS | ✅ Yes (DynamoDB) |
| **Azure Blob Storage** | Azure | ✅ Yes (built-in) |
| **GCS** | Google Cloud | ✅ Yes (built-in) |
| **Terraform Cloud** | HashiCorp | ✅ Yes (built-in) |
| **Consul** | HashiCorp | ✅ Yes (built-in) |
| **PostgreSQL** | Any | ✅ Yes |

### Q15. What is State Locking? Why is it important?

**Answer:**

**State locking** prevents **multiple people from running Terraform at the same time** on the same state file. Without it, two concurrent `terraform apply` commands could corrupt your state.

```
Without Locking:
  Alice runs terraform apply ─────┐
  Bob runs terraform apply ───────┤  BOTH modify state simultaneously
                                  └─ STATE CORRUPTED! 💥

With Locking:
  Alice runs terraform apply ──→ LOCKS state ──→ Makes changes ──→ UNLOCKS
  Bob runs terraform apply ──→ "State is locked, please wait..." ──→ Retries later
```

**How to enable locking (AWS):**

```hcl
# Create a DynamoDB table for locking
resource "aws_dynamodb_table" "terraform_lock" {
  name         = "terraform-lock"
  billing_mode = "PAY_PER_REQUEST"
  hash_key     = "LockID"
  
  attribute {
    name = "LockID"
    type = "S"
  }
}

# Use it in backend configuration
terraform {
  backend "s3" {
    bucket         = "my-terraform-state"
    key            = "prod/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-lock"     # This enables locking!
  }
}
```

**If someone's lock is stuck:**
```bash
# Force unlock (use with caution!)
terraform force-unlock <LOCK_ID>
```

### Q16. How do you import existing infrastructure into Terraform?

**Answer:**

If you have resources that were created **manually** (through the console) and you want Terraform to manage them, you use **`terraform import`**.

**Steps:**

```hcl
# Step 1: Write the resource block in your .tf file
resource "aws_instance" "existing_server" {
  # Leave empty for now — we'll fill in after import
}
```

```bash
# Step 2: Import the real resource into Terraform state
terraform import aws_instance.existing_server i-0abc123def456
```

```bash
# Step 3: Run terraform plan to see what's different
terraform plan
# This shows what attributes the real resource has

# Step 4: Fill in the .tf file to match the real resource
# (So that plan shows no changes)
```

**Terraform 1.5+ — Import blocks (better approach):**

```hcl
# New way — import block in code
import {
  to = aws_instance.existing_server
  id = "i-0abc123def456"
}

resource "aws_instance" "existing_server" {
  ami           = "ami-abc123"
  instance_type = "t3.micro"
  # ... fill in to match
}
```

```bash
# Generate the configuration automatically!
terraform plan -generate-config-out=generated.tf
```

### Q17. What is `terraform state` command? What can you do with it?

**Answer:**

The `terraform state` command lets you **inspect and manipulate** the state file.

```bash
# List all resources in state
terraform state list
# Output:
# aws_instance.web
# aws_s3_bucket.logs
# aws_vpc.main

# Show details of a specific resource
terraform state show aws_instance.web
# Output: shows all attributes of the resource

# Move a resource (rename in state)
terraform state mv aws_instance.web aws_instance.web_server
# Use when: You renamed a resource in your code and don't want to destroy/recreate

# Remove a resource from state (Terraform "forgets" about it)
terraform state rm aws_instance.web
# Use when: You want to stop managing a resource but NOT delete it

# Pull remote state to local
terraform state pull > state_backup.json

# Push local state to remote (dangerous!)
terraform state push state_backup.json
```

**Common scenarios:**

| Scenario | Command |
|----------|---------|
| Renamed a resource | `terraform state mv old_name new_name` |
| Moving resource to a module | `terraform state mv aws_instance.web module.web.aws_instance.this` |
| Stop managing a resource | `terraform state rm resource_name` |
| Check what's in state | `terraform state list` |

### Q18. What happens if the Terraform state file is lost or corrupted?

**Answer:**

**If state is lost:** Terraform has NO idea what resources exist. It thinks nothing has been created.

**Consequences:**
- `terraform plan` will show it wants to create EVERYTHING again
- Running `terraform apply` would create **duplicate resources**
- You've essentially lost the mapping between code and reality

**How to recover:**

1. **From backup** (best option):
   ```bash
   # If using S3 with versioning enabled
   aws s3api list-object-versions --bucket my-terraform-state --key prod/terraform.tfstate
   # Download a previous version
   ```

2. **Re-import resources:**
   ```bash
   # Import each resource back into state
   terraform import aws_instance.web i-abc123
   terraform import aws_vpc.main vpc-xyz789
   # ... repeat for every resource
   ```

3. **From Terraform Cloud:** Terraform Cloud keeps state history — you can restore from any previous version.

**Prevention:**
- Always use **remote state** with **versioning enabled**
- Enable **S3 versioning** (for AWS backend)
- Take **regular backups** of state
- Enable **state locking** to prevent corruption

### Q19. What is the difference between `terraform taint` and `terraform untaint`?

**Answer:**

> **Note:** `terraform taint` is deprecated in Terraform 1.5+. Use `terraform apply -replace` instead.

**Taint** marks a resource for **recreation** on the next apply. The resource will be destroyed and recreated.

**Untaint** removes the taint mark — the resource will NOT be recreated.

```bash
# Old way (deprecated)
terraform taint aws_instance.web        # Mark for recreation
terraform untaint aws_instance.web      # Remove the mark

# New way (Terraform 1.5+)
terraform apply -replace="aws_instance.web"    # Recreate this resource
```

**When to use:**
- A resource is in a bad state (e.g., EC2 instance is corrupted)
- You need to force a recreation (e.g., apply updated user_data)
- Provisioner failed and you want to re-run it

### Q20. What is `terraform refresh`?

**Answer:**

`terraform refresh` updates the state file to match the **real-world infrastructure**. It reads the actual state of resources from the cloud provider and updates the state file.

```bash
# Standalone (deprecated in newer versions)
terraform refresh

# Now built into plan and apply
terraform plan -refresh-only        # Show what changed in real world
terraform apply -refresh-only       # Update state to match reality
```

**When it's useful:**
- Someone manually changed a resource outside Terraform
- You want to check if reality matches your state
- Before running plan, to get an accurate comparison

**Example:**
```
Your state says:  instance_type = "t3.micro"
Someone manually changed it in AWS to:  instance_type = "t3.large"

terraform refresh → Updates state to say "t3.large"
terraform plan → Shows it wants to change back to "t3.micro" (to match your code)
```

### Q21. What are Terraform Modules?

**Answer:**

A **Module** is a **reusable package** of Terraform code. It's like a function in programming — write it once, use it many times.

**Why modules?**
- **Reusability** — Use the same VPC module across all projects
- **Organization** — Break large infrastructure into manageable pieces
- **Standardization** — Enforce consistent patterns across teams
- **Abstraction** — Hide complexity behind a simple interface

```
project/
├── main.tf              # Root module
├── variables.tf
├── outputs.tf
├── modules/
│   ├── vpc/             # VPC module
│   │   ├── main.tf
│   │   ├── variables.tf
│   │   └── outputs.tf
│   ├── ec2/             # EC2 module
│   │   ├── main.tf
│   │   ├── variables.tf
│   │   └── outputs.tf
│   └── rds/             # RDS module
│       ├── main.tf
│       ├── variables.tf
│       └── outputs.tf
```

```hcl
# modules/vpc/main.tf
resource "aws_vpc" "this" {
  cidr_block = var.cidr_block
  tags       = var.tags
}

resource "aws_subnet" "public" {
  count      = length(var.public_subnets)
  vpc_id     = aws_vpc.this.id
  cidr_block = var.public_subnets[count.index]
}

# modules/vpc/variables.tf
variable "cidr_block" {
  type = string
}
variable "public_subnets" {
  type = list(string)
}
variable "tags" {
  type    = map(string)
  default = {}
}

# modules/vpc/outputs.tf
output "vpc_id" {
  value = aws_vpc.this.id
}
output "subnet_ids" {
  value = aws_subnet.public[*].id
}
```

module "production_vpc" {

  subnet_id = module.production_vpc.subnet_ids[0]
  # ...
}

### Q24. What is the difference between `count` and `for_each`?

**Answer:**

Both create **multiple instances** of a resource, but they work differently.

**`count`** — Create N copies (uses index numbers):

```hcl
resource "aws_instance" "web" {
  count         = 3
  ami           = "ami-abc123"
  instance_type = "t3.micro"
  
  tags = {
    Name = "web-${count.index}"    # web-0, web-1, web-2
  }
}

# Access: aws_instance.web[0], aws_instance.web[1]
```

**`for_each`** — Create one for each item in a map or set (uses keys):

```hcl
resource "aws_instance" "web" {
  for_each      = toset(["app", "api", "worker"])
  ami           = "ami-abc123"
  instance_type = "t3.micro"
  
  tags = {
    Name = "web-${each.key}"       # web-app, web-api, web-worker
  }
}

# Access: aws_instance.web["app"], aws_instance.web["api"]
```

**Key differences:**

| Feature | `count` | `for_each` |
|---------|---------|-----------|
| **Input** | A number | A map or set |
| **Index** | Numeric (0, 1, 2) | String keys ("app", "api") |
| **Removing middle item** | Shifts all higher indexes (causes recreation!) | Only removes that item (no shifts!) |
| **Best for** | Identical resources | Resources with unique identifiers |

**Why `for_each` is usually better:**

```hcl
# With count — if you remove "web-1", "web-2" becomes "web-1" → RECREATED!
count = 3  →  [web-0, web-1, web-2]
# Remove web-1:
count = 2  →  [web-0, web-1]  # web-2 was RENAMED to web-1 → destroyed and recreated!

# With for_each — if you remove "api", only "api" is destroyed
for_each = toset(["app", "api", "worker"])
# Remove "api":
for_each = toset(["app", "worker"])  # Only "api" is destroyed. "app" and "worker" untouched!
```

### Q25. What are Dynamic Blocks?

**Answer:**

**Dynamic blocks** let you generate **repeating nested blocks** based on a variable. Instead of copy-pasting the same block multiple times, you use a `dynamic` block.

```hcl
# WITHOUT dynamic blocks — lots of repetition
resource "aws_security_group" "web" {
  name = "web-sg"
  
  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
  
  ingress {
    from_port   = 443
    to_port     = 443
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
  
  ingress {
    from_port   = 8080
    to_port     = 8080
    protocol    = "tcp"
    cidr_blocks = ["10.0.0.0/8"]
  }
}

# WITH dynamic blocks — clean and DRY
variable "ingress_rules" {
  default = [
    { port = 80,   cidr = "0.0.0.0/0" },
    { port = 443,  cidr = "0.0.0.0/0" },
    { port = 8080, cidr = "10.0.0.0/8" },
  ]
}

resource "aws_security_group" "web" {
  name = "web-sg"
  
  dynamic "ingress" {
    for_each = var.ingress_rules
    content {
      from_port   = ingress.value.port
      to_port     = ingress.value.port
      protocol    = "tcp"
      cidr_blocks = [ingress.value.cidr]
    }
  }
}
```

**How it works:**
- `dynamic "ingress"` — The block name to repeat
- `for_each` — The collection to iterate over
- `content {}` — The actual block content
- `ingress.value` — The current item in the iteration
- `ingress.key` — The current index/key

### Q28. What is `terraform plan -target`? When should you use it?

**Answer:**

`-target` lets you plan/apply only a **specific resource** instead of the entire configuration.

```bash
# Only plan for a specific resource
terraform plan -target=aws_instance.web

# Only apply a specific resource
terraform apply -target=aws_instance.web

# Target a module
terraform apply -target=module.vpc

# Target multiple resources
terraform apply -target=aws_instance.web -target=aws_security_group.web
```

**When to use:**
- Debugging a specific resource issue
- Very large infrastructure where you only want to change one thing
- Breaking a chicken-and-egg dependency problem
- Emergency fix to a single resource

**⚠️ When NOT to use:**
- Regular workflow — always plan/apply the full configuration
- As a habit — it can lead to state drift

**Interview answer:** "I use `-target` only as a last resort for debugging or emergencies. Normal workflow should always be a full `terraform plan` and `terraform apply`."

### Q29. What are `lifecycle` rules in Terraform?

**Answer:**

`lifecycle` blocks control how Terraform manages resource creation, update, and deletion.

```hcl
resource "aws_instance" "web" {
  ami           = "ami-abc123"
  instance_type = "t3.micro"
  
  lifecycle {
    # 1. create_before_destroy — Create new resource BEFORE destroying the old one
    create_before_destroy = true
    # Use when: You can't afford downtime (like a database)
    
    # 2. prevent_destroy — Terraform will ERROR if you try to destroy this
    prevent_destroy = true
    # Use when: Critical resources (production database, S3 bucket with data)
    
    # 3. ignore_changes — Ignore changes to specific attributes
    ignore_changes = [
      tags,           # Ignore tag changes (maybe another tool manages tags)
      ami,            # Ignore AMI changes (maybe auto-updated by AWS)
    ]
    # Use when: External processes modify the resource
    
    # 4. replace_triggered_by — Recreate when another resource changes
    replace_triggered_by = [
      aws_ami.custom.id     # Recreate this instance when AMI changes
    ]
  }
}
```

**Common interview follow-up:** "What happens if you try to `terraform destroy` a resource with `prevent_destroy = true`?"

**Answer:** Terraform will throw an error and refuse to destroy it. You must first remove the lifecycle block, apply, and then destroy.

### Q30. What is `null_resource`? When would you use it?

**Answer:**

A `null_resource` is a **resource that doesn't create any infrastructure**. It's used to run provisioners or trigger actions that don't map to any real resource.

```hcl
resource "null_resource" "run_ansible" {
  # Triggers — recreate this resource when these values change
  triggers = {
    instance_id = aws_instance.web.id
    always_run  = timestamp()        # Run every time
  }
  
  provisioner "local-exec" {
    command = "ansible-playbook -i '${aws_instance.web.public_ip},' playbook.yml"
  }
  
  depends_on = [aws_instance.web]
}
```

**Use cases:**
- Run a script after infrastructure is created
- Trigger external tools (Ansible, shell scripts)
- Create a dependency that doesn't map to a resource
- Run commands that don't have a Terraform provider

**Modern alternative — `terraform_data` resource (Terraform 1.4+):**

```hcl
resource "terraform_data" "run_ansible" {
  triggers_replace = [aws_instance.web.id]
  
  provisioner "local-exec" {
    command = "ansible-playbook playbook.yml"
  }
}
```

### Q32. What is `for` expression in Terraform?

**Answer:**

`for` expressions let you **transform** collections (lists, maps).

```hcl
# Transform a list
variable "names" {
  default = ["alice", "bob", "charlie"]
}

# Uppercase all names — returns a list
output "upper_names" {
  value = [for name in var.names : upper(name)]
  # ["ALICE", "BOB", "CHARLIE"]
}

# Filter — only names starting with "a"
output "a_names" {
  value = [for name in var.names : name if substr(name, 0, 1) == "a"]
  # ["alice"]
}

# Create a map from a list
output "name_map" {
  value = { for name in var.names : name => upper(name) }
  # { alice = "ALICE", bob = "BOB", charlie = "CHARLIE" }
}

# Transform a map
variable "instances" {
  default = {
    web    = "t3.micro"
    api    = "t3.small"
    worker = "t3.medium"
  }
}

output "instance_info" {
  value = { for name, type in var.instances : name => "Instance: ${type}" }
  # { web = "Instance: t3.micro", api = "Instance: t3.small", ... }
}
```

### Q33. What is `terraform plan -out` and why should you use it in CI/CD?

**Answer:**

`terraform plan -out` saves the execution plan to a **binary file**. This ensures that what you reviewed is EXACTLY what gets applied.

```bash
# In CI/CD pipeline:

# Step 1: Plan and save
terraform plan -out=plan.tfplan

# Step 2: Review (human or automated)
terraform show plan.tfplan      # Show the plan in human-readable format
terraform show -json plan.tfplan # JSON format for automated analysis

# Step 3: Apply the EXACT plan
terraform apply plan.tfplan      # No confirmation needed — applies exactly what was planned
```

**Why it matters in CI/CD:**

Without `-out`:
```
terraform plan    ← Shows 3 changes
# ... 30 minutes later, someone approves ...
terraform apply   ← Might apply 5 changes (infrastructure changed between plan and apply!)
```

With `-out`:
```
terraform plan -out=plan.tfplan   ← Plans 3 changes, saves to file
# ... 30 minutes later, someone approves ...
terraform apply plan.tfplan       ← Applies EXACTLY those 3 changes. Safe!
```

### Q35. What is Sentinel? How does it work with Terraform?

**Answer:**

**Sentinel** is HashiCorp's **Policy as Code** framework. It enforces rules BEFORE `terraform apply` executes — like a guard at the gate.

**Example policies:**
- "No EC2 instance larger than t3.xlarge in dev"
- "All S3 buckets must have encryption enabled"
- "All resources must have a `team` tag"
- "No public security groups (0.0.0.0/0 on port 22)"

```python
# Sentinel policy example
import "tfplan"

# Rule: All EC2 instances must be t3.micro or t3.small in dev
main = rule {
    all tfplan.resources.aws_instance as _, instances {
        all instances as _, r {
            r.applied.instance_type in ["t3.micro", "t3.small"]
        }
    }
}
```

**How it fits in the workflow:**

```
terraform plan → Sentinel policy check → terraform apply
                      │
                      ├─ PASS → Apply proceeds
                      └─ FAIL → Apply is BLOCKED
```

**Enforcement levels:**
- **Advisory** — Warn but allow (soft fail)
- **Soft mandatory** — Block, but override with approval
- **Hard mandatory** — Block, no override possible

**Open-source alternative:** **OPA (Open Policy Agent)** with **Conftest** or **Checkov/tfsec** for static analysis.

### Q37. How do you handle secrets in Terraform?

**Answer:**

**❌ NEVER do this:**

```hcl
# DON'T hardcode secrets in .tf files!
resource "aws_db_instance" "main" {
  username = "admin"
  password = "SuperSecret123!"    # ❌ THIS IS IN GIT!
}
```

**✅ Correct approaches:**

**1. Variable with no default (prompt at runtime):**
```hcl
variable "db_password" {
  type      = string
  sensitive = true          # Marks it as sensitive (hidden in output)
}

# Pass via: terraform apply -var="db_password=secret"
# Or: export TF_VAR_db_password=secret
```

**2. From a secrets manager:**
```hcl
data "aws_secretsmanager_secret_version" "db_password" {
  secret_id = "prod/database/password"
}

resource "aws_db_instance" "main" {
  password = data.aws_secretsmanager_secret_version.db_password.secret_string
}
```

**3. From HashiCorp Vault:**
```hcl
data "vault_generic_secret" "db" {
  path = "secret/database"
}

resource "aws_db_instance" "main" {
  password = data.vault_generic_secret.db.data["password"]
}
```

**4. Environment variables:**
```bash
export TF_VAR_db_password="SuperSecret123!"
terraform apply
```

**5. SOPS-encrypted files:**
```hcl
data "sops_file" "secrets" {
  source_file = "secrets.enc.yaml"
}
```

**Also remember:**
- Mark variables as `sensitive = true`
- Enable **encryption at rest** on your state backend (state contains secrets!)
- Use `.gitignore` to exclude `.tfvars` files with secrets

### Q38. How do you test Terraform code?

**Answer:**

| Tool | What It Tests | Type |
|------|-------------|------|
| `terraform validate` | Syntax and config correctness | Static |
| `terraform plan` | What will change (diff) | Static |
| `terraform fmt -check` | Code formatting | Static |
| **tflint** | Terraform-specific linting rules | Static |
| **Checkov** | Security best practices | Static |
| **tfsec** | Security misconfigurations | Static |
| **Terratest** | Full integration tests (create + verify + destroy) | Integration |
| **Kitchen-Terraform** | Integration testing framework | Integration |
| **terraform test** (native) | Built-in testing (Terraform 1.6+) | Integration |

**1. Static analysis (fast, no infrastructure created):**

```bash
# Lint with tflint
tflint --init
tflint

# Security scan with Checkov
checkov -d .

# Security scan with tfsec
tfsec .
```

**2. Terraform native tests (Terraform 1.6+):**

```hcl
# tests/vpc_test.tftest.hcl
run "create_vpc" {
  command = apply
  
  variables {
    cidr_block = "10.0.0.0/16"
  }
  
  assert {
    condition     = aws_vpc.main.cidr_block == "10.0.0.0/16"
    error_message = "VPC CIDR block is incorrect"
  }
  
  assert {
    condition     = aws_vpc.main.enable_dns_support == true
    error_message = "DNS support should be enabled"
  }
}
```

```bash
terraform test
```

**3. Integration tests with Terratest (Go):**

```go
package test

import (
    "testing"
    "github.com/gruntwork-io/terratest/modules/terraform"
    "github.com/stretchr/testify/assert"
)

func TestVPC(t *testing.T) {
    opts := &terraform.Options{
        TerraformDir: "../modules/vpc",
        Vars: map[string]interface{}{
            "cidr_block": "10.0.0.0/16",
        },
    }
    
    // Clean up after test
    defer terraform.Destroy(t, opts)
    
    // Create infrastructure
    terraform.InitAndApply(t, opts)
    
    // Verify
    vpcID := terraform.Output(t, opts, "vpc_id")
    assert.NotEmpty(t, vpcID)
}
```

### Q39. How do you use Terraform in a CI/CD pipeline?

**Answer:**

```yaml
# GitHub Actions — Terraform CI/CD pipeline
name: Terraform

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  terraform:
    runs-on: ubuntu-latest
    
    steps:
      - uses: actions/checkout@v4
      
      - name: Setup Terraform
        uses: hashicorp/setup-terraform@v3
        with:
          terraform_version: 1.7.0
      
      # Format check
      - name: Terraform Format
        run: terraform fmt -check -recursive
      
      # Initialize
      - name: Terraform Init
        run: terraform init
        env:
          AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
          AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
      
      # Validate
      - name: Terraform Validate
        run: terraform validate
      
      # Security scan
      - name: Run Checkov
        uses: bridgecrewio/checkov-action@master
        with:
          directory: .
      
      # Plan (on PRs)
      - name: Terraform Plan
        if: github.event_name == 'pull_request'
        run: terraform plan -no-color -out=plan.tfplan
      
      # Comment plan on PR
      - name: Comment Plan on PR
        if: github.event_name == 'pull_request'
        uses: actions/github-script@v7
        with:
          script: |
            // Post plan output as PR comment
      
      # Apply (only on main branch merge)
      - name: Terraform Apply
        if: github.ref == 'refs/heads/main' && github.event_name == 'push'
        run: terraform apply -auto-approve
```

**Pipeline flow:**

```
PR opened → Format Check → Init → Validate → Security Scan → Plan → Comment on PR
                                                                         │
                                                              PR Approved & Merged
                                                                         │
Main branch → Format Check → Init → Validate → Plan → Apply (auto-approve)
```

**Best practices for Terraform in CI/CD:**
1. Always run `terraform fmt -check` (enforce formatting)
2. Always run `terraform validate` (catch errors early)
3. Run security scans (Checkov, tfsec)
4. Show plan output in PR comments (so reviewers can see the impact)
5. Use `-out=plan.tfplan` to save the plan
6. Auto-approve only on main branch (after human review via PR)
7. Use remote state with locking
8. Pin Terraform and provider versions

### Q40. What is OpenTofu? How is it different from Terraform?

**Answer:**

**OpenTofu** is an **open-source fork** of Terraform, created after HashiCorp changed Terraform's license from open-source (MPL 2.0) to the more restrictive **BSL (Business Source License)** in August 2023.

**Why was it created?**
- HashiCorp's license change meant competitors couldn't use Terraform code
- The community (led by companies like Gruntwork, Spacelift, env0) created OpenTofu
- It's maintained by the **Linux Foundation** — truly open-source forever

**Differences:**

| Feature | Terraform | OpenTofu |
|---------|-----------|---------|
| **License** | BSL 1.1 (restrictive) | MPL 2.0 (open-source) |
| **Maintained by** | HashiCorp | Linux Foundation + Community |
| **Command** | `terraform` | `tofu` |
| **Compatibility** | N/A | Drop-in replacement for Terraform 1.6 |
| **State format** | Same | Same (compatible!) |
| **Providers** | Terraform Registry | OpenTofu Registry (same providers) |
| **New features** | HashiCorp decides | Community-driven |

**Migration from Terraform to OpenTofu:**

```bash
# It's basically a drop-in replacement
# Step 1: Install OpenTofu
brew install opentofu    # macOS
# or
snap install opentofu    # Linux

# Step 2: Replace terraform commands with tofu
tofu init
tofu plan
tofu apply

# That's it! Same syntax, same state, same providers.
```

**Interview tip:** Mention OpenTofu to show you're up-to-date with the ecosystem. "I'm aware of the Terraform license change and OpenTofu as an alternative. For new projects, I'd consider OpenTofu, but existing Terraform projects can continue as-is since the syntax is identical."

## 🎯 Quick Revision — Terraform Cheat Sheet

| Command | What It Does |
|---------|-------------|
| `terraform init` | Initialize, download providers and modules |
| `terraform plan` | Preview changes (dry run) |
| `terraform apply` | Create/update infrastructure |
| `terraform destroy` | Delete all managed infrastructure |
| `terraform fmt` | Format code |
| `terraform validate` | Check syntax |
| `terraform state list` | List resources in state |
| `terraform state show` | Show resource details |
| `terraform state mv` | Rename a resource in state |
| `terraform state rm` | Remove resource from state |
| `terraform import` | Import existing resource into state |
| `terraform output` | Show output values |
| `terraform workspace` | Manage workspaces |
| `terraform graph` | Generate dependency graph |
| `terraform refresh` | Sync state with real world |

---

| Concept | One-Line Definition |
|---------|-------------------|
| Provider | Plugin that talks to a cloud platform (AWS, Azure) |
| Resource | Creates and manages infrastructure |
| Data Source | Reads existing infrastructure |
| State | JSON file mapping code to real resources |
| Module | Reusable package of Terraform code |
| Backend | Where state is stored (local, S3, Terraform Cloud) |
| Workspace | Isolated state for the same code (dev, staging, prod) |
| Variable | Input parameters for your code |
| Output | Exported values from your code |
| Locals | Computed values used within a module |
| Provisioner | Runs scripts on resources (last resort!) |
| Dynamic Block | Generates repeating nested blocks |
| `count` | Create N identical copies |
| `for_each` | Create one per item in a collection |
| `depends_on` | Explicit dependency between resources |
| `lifecycle` | Control create/destroy behavior |
