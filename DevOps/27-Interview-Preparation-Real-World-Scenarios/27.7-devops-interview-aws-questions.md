## 🔐 IAM (Identity & Access Management)

---

### Q1. What is IAM in AWS? Why is it important?

**Answer:**

IAM stands for **Identity and Access Management**. It's the service that controls **WHO** can access **WHAT** in your AWS account.

Think of IAM as the **security guard** of your AWS account. Without IAM, either everyone has full access (dangerous!) or no one can do anything.

```
IAM lets you manage:
├── Users       → Individual people (developers, admins)
├── Groups      → Collections of users (dev-team, ops-team)
├── Roles       → Temporary permissions (for services, apps, cross-account)
├── Policies    → JSON documents that define permissions (allow/deny)
```

**Key points for the interview:**
- IAM is **global** — it's not region-specific
- The **root account** should never be used for daily tasks
- Always follow the **Principle of Least Privilege** — give only the permissions needed, nothing more
- IAM is **free** — you don't pay for creating users, groups, or roles

---

### Q2. What is the difference between an IAM Role and an IAM User?

**Answer:**

| Feature | IAM User | IAM Role |
|---------|----------|----------|
| **Who uses it?** | A specific person or application | Any trusted entity (service, user, another account) |
| **Credentials** | Has permanent credentials (password, access keys) | Has **temporary** credentials (auto-rotated) |
| **Best for** | Human users who need console/CLI access | EC2 instances, Lambda functions, cross-account access |
| **Security** | Less secure (long-lived keys can leak) | More secure (credentials expire automatically) |

**Real-world example:**
```
Bad practice:  Create an IAM User with access keys → hardcode them in your EC2 app
Good practice: Create an IAM Role → attach it to the EC2 instance → app gets temporary creds automatically
```

> **Interview gold:** Always recommend IAM Roles over IAM Users for applications. Roles use the **AWS Security Token Service (STS)** to provide temporary credentials that auto-expire.

---

### Q3. What is an IAM Policy? Can you explain the structure?

**Answer:**

An IAM Policy is a **JSON document** that defines permissions. It says what actions are allowed or denied on which resources.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowS3ReadAccess",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::my-bucket",
        "arn:aws:s3:::my-bucket/*"
      ],
      "Condition": {
        "IpAddress": {
          "aws:SourceIp": "203.0.113.0/24"
        }
      }
    }
  ]
}
```

**Breaking it down:**

| Field | Meaning |
|-------|---------|
| `Version` | Policy language version (always use `"2012-10-17"`) |
| `Statement` | Array of permission rules |
| `Sid` | Statement ID — a friendly name (optional) |
| `Effect` | `Allow` or `Deny` |
| `Action` | What API calls are permitted (e.g., `s3:GetObject`) |
| `Resource` | Which AWS resources this applies to (ARN format) |
| `Condition` | Optional — extra conditions (IP range, time, MFA, etc.) |

> **Key rule:** If there's an explicit `Deny`, it **always** overrides an `Allow`. This is called the **explicit deny wins** rule.

---

### Q4. What is the difference between AWS Managed Policies and Customer Managed Policies?

**Answer:**

| Type | Description | Example |
|------|-------------|---------|
| **AWS Managed** | Pre-built by AWS, maintained by AWS | `AmazonS3ReadOnlyAccess`, `AdministratorAccess` |
| **Customer Managed** | Created by YOU, maintained by YOU | Custom policy for your specific needs |
| **Inline** | Embedded directly into a user/group/role | One-off permissions for a specific entity |

**Best practice:**
- Use **AWS Managed** policies when they fit your needs (less maintenance)
- Use **Customer Managed** policies when you need fine-grained control
- **Avoid inline policies** — they're hard to manage and can't be reused

## 🖥️ EC2 (Elastic Compute Cloud)

---

### Q5. What is EC2? Explain the different instance types.

**Answer:**

EC2 is AWS's **virtual server** service. It lets you run virtual machines (called **instances**) in the cloud. You choose the OS, CPU, memory, storage, and networking.

**Instance type families:**

```
Instance Types (think of them like car types):
├── t-series (t3, t3a, t4g)  → 🚗 Economy car   → General purpose, burstable, good for dev/test
├── m-series (m5, m6i, m7g)  → 🚙 Sedan         → General purpose, balanced CPU/memory
├── c-series (c5, c6i, c7g)  → 🏎️ Sports car    → Compute optimized (CPU-heavy tasks)
├── r-series (r5, r6i, r7g)  → 🚌 Bus           → Memory optimized (databases, caching)
├── i-series (i3, i4i)       → 🚚 Truck          → Storage optimized (high I/O)
├── p-series (p4d, p5)       → 🚀 Rocket         → GPU instances (ML/AI training)
└── g-series (g5, g6)        → 🎮 Gaming PC      → GPU instances (graphics/inference)
```

**Naming convention:**
```
Example: m5.xlarge

m  = Family (General Purpose)
5  = Generation (5th generation)
.  = Separator
xlarge = Size (determines CPU, RAM, network)

Sizes: nano < micro < small < medium < large < xlarge < 2xlarge < ... < metal
```

---

### Q6. What are the different EC2 pricing models?

**Answer:**

This is a **very common** interview question. Know these cold:

| Pricing Model | Discount | Commitment | Best For |
|---------------|----------|------------|----------|
| **On-Demand** | 0% (full price) | None — pay per second/hour | Unpredictable workloads, short-term, testing |
| **Reserved Instances (RI)** | Up to 72% off | 1 or 3 year commitment | Steady-state workloads (databases, production servers) |
| **Savings Plans** | Up to 72% off | 1 or 3 year $/hour commitment | Flexible — applies across instance families |
| **Spot Instances** | Up to 90% off | None — but AWS can reclaim with 2-min notice | Batch jobs, CI/CD runners, fault-tolerant workloads |
| **Dedicated Hosts** | Varies | Optional | Compliance requirements, license-bound software |

**Interview scenario:**
```
Interviewer: "How would you reduce EC2 costs for a production environment?"

Your answer:
1. Use Reserved Instances or Savings Plans for baseline (always-on) workloads → 40-72% savings
2. Use Spot Instances for CI/CD build agents and batch processing → up to 90% savings
3. Right-size instances using AWS Compute Optimizer recommendations
4. Use Auto Scaling to match capacity with demand
5. Schedule dev/staging instances to shut down at night and weekends
```

### Q7. What is an AMI? How do you create a custom AMI?

**Answer:**

**AMI** = Amazon Machine Image. It's a **template** that contains the OS, applications, and configuration needed to launch an EC2 instance.

Think of it like a **snapshot of a fully configured server** that you can use to create identical copies.

```
AMI contains:
├── Root volume (OS + installed software)
├── Launch permissions (who can use this AMI)
├── Block device mapping (attached EBS volumes)
```

**How to create a custom AMI:**

```bash
# Step 1: Launch an EC2 instance with a base AMI (e.g., Amazon Linux 2023)
# Step 2: SSH in and install your software
sudo yum install -y nginx java-17 docker
sudo systemctl enable nginx docker

# Step 3: Configure everything (config files, environment variables, etc.)

# Step 4: Create the AMI from the running instance
aws ec2 create-image \
  --instance-id i-0abc123def456 \
  --name "my-app-server-v1.0" \
  --description "Nginx + Java 17 + Docker pre-installed" \
  --no-reboot

# Step 5: Use this AMI in your Auto Scaling Group or Launch Template
```

**DevOps best practice:** Use tools like **Packer** (by HashiCorp) to automate AMI creation in your CI/CD pipeline. This is called **immutable infrastructure** — instead of updating servers in place, you build a new AMI and replace the old instances.

### Q8. What is an EC2 Security Group? How is it different from a NACL?

**Answer:**

Both are **firewalls**, but they work at different levels:

| Feature | Security Group | NACL (Network ACL) |
|---------|---------------|---------------------|
| **Level** | Instance level (attached to ENI) | Subnet level |
| **State** | **Stateful** — if inbound is allowed, outbound response is auto-allowed | **Stateless** — must explicitly allow both inbound AND outbound |
| **Rules** | Allow rules only (no deny rules) | Both Allow and Deny rules |
| **Rule evaluation** | All rules evaluated together | Rules evaluated in **order** (lowest number first) |
| **Default** | Denies all inbound, allows all outbound | Allows all inbound and outbound |
| **Use case** | Fine-grained instance-level control | Broad subnet-level guardrails |

```
Real-world analogy:

Security Group = The lock on your apartment door (per-unit security)
NACL           = The security gate at the building entrance (building-level security)

You typically use BOTH:
Internet → NACL (subnet level) → Security Group (instance level) → Your EC2
```

## 🌐 VPC (Virtual Private Cloud)

---

### Q9. What is a VPC? Explain its core components.

**Answer:**

A VPC is your **own private network** inside AWS. It's like having your own isolated section of the AWS cloud where you define the network topology.

```
VPC Architecture:

┌─────────────────────────────────────────────────────────┐
│  VPC: 10.0.0.0/16  (65,536 IP addresses)               │
│                                                         │
│  ┌──────────────────┐    ┌──────────────────┐           │
│  │ Public Subnet     │    │ Public Subnet     │          │
│  │ 10.0.1.0/24       │    │ 10.0.2.0/24       │          │
│  │ AZ: us-east-1a    │    │ AZ: us-east-1b    │          │
│  │                    │    │                    │          │
│  │ ┌──────────────┐  │    │ ┌──────────────┐  │          │
│  │ │ NAT Gateway  │  │    │ │ ALB           │  │          │
│  │ └──────────────┘  │    │ └──────────────┘  │          │
│  └────────┬──────────┘    └──────────────────┘           │
│           │                                              │
│  ┌────────▼──────────┐    ┌──────────────────┐           │
│  │ Private Subnet     │    │ Private Subnet     │         │
│  │ 10.0.3.0/24        │    │ 10.0.4.0/24        │         │
│  │ AZ: us-east-1a     │    │ AZ: us-east-1b     │         │
│  │                     │    │                     │         │
│  │ ┌──────────────┐   │    │ ┌──────────────┐   │         │
│  │ │ App Servers   │   │    │ │ App Servers   │   │        │
│  │ └──────────────┘   │    │ └──────────────┘   │         │
│  └─────────────────────┘    └─────────────────────┘       │
│                                                           │
│  Internet Gateway (IGW) ← connects VPC to the internet   │
└───────────────────────────────────────────────────────────┘
```

**Core components:**

| Component | Purpose |
|-----------|---------|
| **VPC** | The overall private network (you define the CIDR block) |
| **Subnets** | Subdivisions of VPC, each in a specific AZ |
| **Internet Gateway (IGW)** | Allows public subnets to access the internet |
| **NAT Gateway** | Allows private subnets to access the internet (outbound only) |
| **Route Tables** | Rules that determine where network traffic goes |
| **Security Groups** | Instance-level firewalls (stateful) |
| **NACLs** | Subnet-level firewalls (stateless) |
| **Elastic IP** | Static public IP address |
| **VPC Peering** | Connect two VPCs privately |
| **Transit Gateway** | Hub to connect multiple VPCs and on-prem networks |

---

### Q10. What is the difference between a Public Subnet and a Private Subnet?

**Answer:**

| Feature | Public Subnet | Private Subnet |
|---------|--------------|----------------|
| **Internet access** | Yes — via Internet Gateway | No direct access — uses NAT Gateway for outbound |
| **Route table** | Has a route to IGW (`0.0.0.0/0 → igw-xxx`) | Has a route to NAT Gateway (`0.0.0.0/0 → nat-xxx`) |
| **Public IP** | Instances can have public IPs | Instances only have private IPs |
| **What goes here?** | Load balancers, bastion hosts, NAT Gateways | App servers, databases, internal services |
| **Security** | More exposed | More secure (no inbound from internet) |

```
Traffic flow for private instances:

Outbound (Private → Internet):
  App Server (private subnet) → NAT Gateway (public subnet) → IGW → Internet
  
Inbound (Internet → Private):
  Internet → ALB (public subnet) → App Server (private subnet)
  (Direct inbound from internet to private subnet is NOT possible)
```

## 📦 S3 (Simple Storage Service)

---

### Q11. What is S3? What are Storage Classes?

**Answer:**

S3 is AWS's **object storage** service. It stores files (called **objects**) in containers (called **buckets**). It's designed for **99.999999999% (11 nines)** durability.

```
S3 is NOT a file system — it's object storage:
├── No folders (the "/" in paths is just a prefix for organization)
├── Each object can be up to 5 TB
├── Unlimited total storage
├── Accessed via HTTP/HTTPS (REST API)
```

**Storage Classes** (choose based on access frequency and cost):

| Storage Class | Access Pattern | Cost | Use Case |
|---------------|----------------|------|----------|
| **S3 Standard** | Frequent access | $$$ | Active data, websites, apps |
| **S3 Intelligent-Tiering** | Unknown/changing | $$-$$$ | Auto-moves data between tiers |
| **S3 Standard-IA** | Infrequent (once/month) | $$ | Backups, disaster recovery |
| **S3 One Zone-IA** | Infrequent, non-critical | $ | Reproducible data, secondary backups |
| **S3 Glacier Instant** | Rare but needs instant access | $ | Archived data needing millisecond access |
| **S3 Glacier Flexible** | Rare (minutes to hours retrieval) | ¢ | Long-term archives |
| **S3 Glacier Deep Archive** | Very rare (12-48 hour retrieval) | ¢¢ | Compliance archives, 7+ year retention |

---

### Q12. How do you secure an S3 bucket?

**Answer:**

S3 security is a **favorite interview topic** because so many data breaches happen due to misconfigured S3 buckets.

```
S3 Security Layers:

1. Block Public Access (Account/Bucket level)
   └── ALWAYS enable this unless you specifically need public access

2. Bucket Policy (Resource-based)
   └── JSON policy attached to the bucket

3. IAM Policies (Identity-based)
   └── Permissions attached to users/roles

4. ACLs (Legacy — avoid using these)
   └── Old way of managing access — use policies instead

5. Encryption
   ├── SSE-S3 (AWS managed keys — simplest)
   ├── SSE-KMS (Customer managed keys via KMS — more control, audit trail)
   └── SSE-C (Customer provided keys — you manage the keys)

6. VPC Endpoints
   └── Access S3 privately without going over the internet

7. Access Logging & CloudTrail
   └── Track who accessed what and when

8. Object Lock & Versioning
   └── Prevent accidental deletion (WORM compliance)

9. Pre-signed URLs
   └── Temporary, time-limited access to specific objects
```

**Example — Secure bucket policy:**

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyUnencryptedUploads",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::my-secure-bucket/*",
      "Condition": {
        "StringNotEquals": {
          "s3:x-amz-server-side-encryption": "aws:kms"
        }
      }
    },
    {
      "Sid": "DenyHTTP",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:*",
      "Resource": "arn:aws:s3:::my-secure-bucket/*",
      "Condition": {
        "Bool": {
          "aws:SecureTransport": "false"
        }
      }
    }
  ]
}
```

This policy does two things:
1. **Denies** any upload that isn't encrypted with KMS
2. **Denies** any request that doesn't use HTTPS

## 🔄 Auto Scaling & Load Balancing

---

### Q13. What is Auto Scaling? Explain the components.

**Answer:**

Auto Scaling automatically **adjusts the number of EC2 instances** based on demand. More traffic? Add instances. Less traffic? Remove instances.

```
Auto Scaling Components:

1. Launch Template (or Launch Configuration)
   └── WHAT to launch (AMI, instance type, security groups, key pair, user data)

2. Auto Scaling Group (ASG)
   └── HOW MANY to launch and WHERE
       ├── Min capacity  (e.g., 2)   → Never go below this
       ├── Desired capacity (e.g., 4) → Target number of instances
       └── Max capacity  (e.g., 10)  → Never exceed this

3. Scaling Policies
   └── WHEN to scale
       ├── Target Tracking  → "Keep average CPU at 60%"
       ├── Step Scaling      → "If CPU > 80%, add 2 instances"
       ├── Simple Scaling    → "If CPU > 80%, add 1 instance" (legacy)
       └── Scheduled         → "Scale up to 10 at 9 AM on weekdays"
```

**Real-world example:**

```
Normal day:     2 instances handling 100 requests/sec
Black Friday:   Auto Scaling detects high CPU → scales to 20 instances
After sale ends: Traffic drops → scales back down to 2 instances

You only pay for what you use! 💰
```

### Q14. Explain the different types of AWS Load Balancers.

**Answer:**

| Type | Layer | Protocol | Best For |
|------|-------|----------|----------|
| **ALB (Application LB)** | Layer 7 (HTTP/HTTPS) | HTTP, HTTPS, WebSocket, gRPC | Web apps, microservices, path/host-based routing |
| **NLB (Network LB)** | Layer 4 (TCP/UDP) | TCP, UDP, TLS | High performance, low latency, static IPs, gaming |
| **CLB (Classic LB)** | Layer 4 & 7 | HTTP, HTTPS, TCP | **Legacy** — don't use for new projects |
| **GWLB (Gateway LB)** | Layer 3 (IP) | IP | Third-party virtual appliances (firewalls, IDS) |

**Key differences that interviewers love:**

```
ALB Features:
├── Path-based routing  → /api/* → API servers, /web/* → web servers
├── Host-based routing  → api.example.com → API, www.example.com → Web
├── Weighted target groups → Canary deployments (90% v1, 10% v2)
├── Sticky sessions → Same user → same server
├── Authentication integration → Cognito, OIDC
└── HTTP/2 and gRPC support

NLB Features:
├── Ultra-low latency (millions of requests per second)
├── Static IP / Elastic IP support
├── Preserves source IP address
├── TCP/UDP passthrough
└── Used for non-HTTP workloads (databases, gaming, IoT)
```

---

### Q15. What is a Target Group?

**Answer:**

A Target Group is a **collection of targets** (EC2 instances, IP addresses, Lambda functions, or containers) that a Load Balancer routes traffic to.

```
ALB → Target Group → Targets

Example setup:
├── Target Group: "api-servers"    → EC2 instances running API
├── Target Group: "web-servers"    → EC2 instances running frontend
└── Target Group: "lambda-target"  → Lambda function for serverless endpoints

Health Checks:
├── The LB regularly checks if targets are healthy
├── Unhealthy targets stop receiving traffic
├── Configurable: protocol, path, interval, thresholds
```

## 🗄️ RDS & Databases

---

### Q16. What is RDS? What databases does it support?

**Answer:**

RDS = **Relational Database Service**. It's a managed database service — AWS handles the heavy lifting (backups, patching, replication, failover) so you can focus on your data.

```
Supported Engines:
├── Amazon Aurora (MySQL & PostgreSQL compatible — AWS's own, 5x faster)
├── MySQL
├── PostgreSQL
├── MariaDB
├── Oracle
├── Microsoft SQL Server
```

**What RDS manages for you:**

| Task | Without RDS (Self-managed) | With RDS (Managed) |
|------|---------------------------|---------------------|
| Provisioning | You install the DB on EC2 | Click a button or run Terraform |
| Patching | You manually apply patches | Automated during maintenance windows |
| Backups | You set up cron jobs | Automated daily snapshots, PITR |
| High Availability | You configure replication | Enable Multi-AZ with one checkbox |
| Scaling | You resize the server manually | Modify instance class, enable read replicas |
| Monitoring | You install monitoring tools | Built-in CloudWatch metrics, Performance Insights |

---

### Q17. What is RDS Multi-AZ vs Read Replicas?

**Answer:**

This is one of the **most asked** AWS database questions:

| Feature | Multi-AZ | Read Replicas |
|---------|----------|---------------|
| **Purpose** | **High Availability** (disaster recovery) | **Performance** (scale reads) |
| **How it works** | Synchronous replication to standby in another AZ | Asynchronous replication to read-only copies |
| **Failover** | Automatic failover to standby (60-120 seconds) | No automatic failover (must promote manually) |
| **Read traffic** | Standby is NOT used for reads (it's just waiting) | Read replicas actively serve read queries |
| **Regions** | Same region, different AZs | Can be cross-region |
| **Number** | 1 standby | Up to 5-15 read replicas (varies by engine) |
| **Use case** | Production databases that can't afford downtime | Heavy read workloads (reports, dashboards) |

```
Multi-AZ (High Availability):

  Primary DB (us-east-1a) ──sync──► Standby DB (us-east-1b)
        ↑                                    │
    App writes here                    If primary dies,
                                       standby takes over
                                       (automatic failover)


Read Replicas (Performance):

  Primary DB ──async──► Read Replica 1 (same region)
      ↑                 Read Replica 2 (same region)
      │                 Read Replica 3 (different region!)
  App writes here       
                        App reads from replicas
                        (distributes read load)
```

## 🚀 ECS & EKS (Container Services)

---

### Q18. What is ECS? Explain the difference between EC2 launch type and Fargate.

**Answer:**

ECS = **Elastic Container Service**. It's AWS's own container orchestration platform (alternative to Kubernetes).

```
ECS Core Concepts:

Cluster → The top-level grouping (a logical boundary)
  └── Service → Manages running tasks (desired count, load balancer, scaling)
       └── Task → A running instance of your containers (like a K8s Pod)
            └── Task Definition → Blueprint for your task (like a K8s Deployment YAML)
                 └── Container Definition → Docker image, ports, env vars, resources
```

**EC2 Launch Type vs Fargate:**

| Feature | EC2 Launch Type | Fargate Launch Type |
|---------|----------------|---------------------|
| **Infrastructure** | YOU manage the EC2 instances | AWS manages everything (serverless) |
| **Control** | Full control over instances | No access to underlying instances |
| **Scaling** | Must scale EC2 instances + tasks | Just scale tasks — infrastructure is automatic |
| **Pricing** | Pay for EC2 instances (even if underused) | Pay per task (vCPU + memory per second) |
| **Maintenance** | YOU patch, update, monitor the hosts | AWS handles all host maintenance |
| **GPU support** | Yes | Limited |
| **Best for** | Large, predictable workloads; need host access | Variable workloads, small teams, simplicity |

```
When to choose what:

Fargate → "I just want to run containers. I don't want to manage servers."
EC2     → "I need GPU, specific instance types, or cost optimization at scale."
```

---

### Q19. What is EKS? How is it different from ECS?

**Answer:**

EKS = **Elastic Kubernetes Service**. It's AWS's managed Kubernetes service.

| Feature | ECS | EKS |
|---------|-----|-----|
| **Orchestrator** | AWS proprietary | Kubernetes (open-source standard) |
| **Learning curve** | Easier (AWS-native) | Steeper (Kubernetes complexity) |
| **Portability** | AWS locked-in | Multi-cloud portable (same K8s everywhere) |
| **Ecosystem** | AWS tools only | Massive K8s ecosystem (Helm, ArgoCD, Istio, etc.) |
| **Pricing** | No control plane cost | $0.10/hour for the control plane (~$73/month) |
| **Best for** | Simple container workloads, AWS-only shops | Complex microservices, multi-cloud, existing K8s teams |

**Interview tip:**
```
"If the team already knows Kubernetes → go with EKS
 If the team is small and AWS-native → go with ECS
 Both work great — it depends on your team's skills and needs."
```

### Q20. What is ECR?

**Answer:**

ECR = **Elastic Container Registry**. It's AWS's **private Docker image registry** — like Docker Hub, but private and integrated with AWS.

```
ECR Features:
├── Private repositories for your Docker images
├── Image scanning for vulnerabilities (uses Trivy/Clair under the hood)
├── Image lifecycle policies (auto-delete old/untagged images)
├── Cross-region and cross-account replication
├── Integrates with ECS, EKS, Lambda, and CodeBuild
├── Supports OCI artifacts (Helm charts, Sigstore signatures)
```

**Common DevOps workflow:**

```bash
# Authenticate Docker to ECR
aws ecr get-login-password --region us-east-1 | \
  docker login --username AWS --password-stdin 123456789.dkr.ecr.us-east-1.amazonaws.com

# Build image
docker build -t my-app:v1.0 .

# Tag image for ECR
docker tag my-app:v1.0 123456789.dkr.ecr.us-east-1.amazonaws.com/my-app:v1.0

# Push to ECR
docker push 123456789.dkr.ecr.us-east-1.amazonaws.com/my-app:v1.0
```

## 🛣️ Route 53

---

### Q21. What is Route 53? What routing policies does it support?

**Answer:**

Route 53 is AWS's **DNS service**. It translates domain names (like `www.example.com`) to IP addresses. The name "53" comes from port 53, the standard DNS port.

**Routing Policies:**

| Policy | Description | Use Case |
|--------|-------------|----------|
| **Simple** | One record, one or more values (random selection) | Single server or random distribution |
| **Weighted** | Distribute traffic by percentage (e.g., 70/30) | Canary deployments, A/B testing |
| **Latency-based** | Route to the region with lowest latency | Global apps, multi-region |
| **Failover** | Primary/secondary — auto-switch on health check failure | Disaster recovery |
| **Geolocation** | Route based on user's geographic location | Country-specific content, compliance |
| **Geoproximity** | Route based on geographic distance + bias | Fine-grained geographic routing |
| **Multi-value** | Return multiple healthy IPs (like simple + health checks) | Simple load balancing with health checks |

```
Real-world example (Blue/Green Deployment with Weighted Routing):

Route 53 Weighted Records:
├── app.example.com → Blue environment (ALB-1)   Weight: 90
└── app.example.com → Green environment (ALB-2)  Weight: 10

Gradually shift traffic:
Step 1: 90/10  →  Step 2: 70/30  →  Step 3: 50/50  →  Step 4: 0/100 (fully on Green)
```

## 🔒 KMS & Encryption

---

### Q24. What is KMS? How does encryption work in AWS?

**Answer:**

KMS = **Key Management Service**. It lets you create and manage **encryption keys** used to encrypt your data across AWS services.

```
How KMS Encryption Works (Envelope Encryption):

Step 1: You create a Customer Master Key (CMK) in KMS
Step 2: When encrypting data:
        KMS generates a Data Encryption Key (DEK)
        ├── Plaintext DEK → used to encrypt your data
        └── Encrypted DEK → stored alongside your encrypted data
Step 3: Plaintext DEK is deleted from memory (never stored)
Step 4: To decrypt:
        ├── Send encrypted DEK to KMS
        ├── KMS decrypts it using the CMK
        └── Use the plaintext DEK to decrypt your data

Why? Because encrypting/decrypting large data directly with KMS
would be slow and expensive. Envelope encryption is fast and cheap.
```

**Types of KMS Keys:**

| Type | Managed By | Cost | Rotation |
|------|-----------|------|----------|
| **AWS Managed Key** | AWS | Free (within service usage) | Auto every year |
| **Customer Managed Key** | You | $1/month + API calls | Optional (auto or manual) |
| **AWS Owned Key** | AWS | Free | AWS decides |

## 📡 SNS, SQS & EventBridge

### Q27. What is the difference between SNS and SQS?

**Answer:**

| Feature | SNS (Simple Notification Service) | SQS (Simple Queue Service) |
|---------|-----------------------------------|----------------------------|
| **Model** | **Pub/Sub** (publish-subscribe) | **Queue** (message queue) |
| **Delivery** | Push — sends to all subscribers immediately | Pull — consumers poll the queue for messages |
| **Consumers** | Multiple subscribers receive the same message | Typically one consumer processes each message |
| **Persistence** | No persistence (if subscriber is down, message is lost) | Messages persist in queue until processed |
| **Use case** | Fan-out notifications, alerts, email/SMS | Decoupling services, work queues, buffering |

```
SNS = Radio broadcast 📻 (everyone listening hears the same message)
SQS = Post office mailbox 📬 (messages wait until someone picks them up)

Common Pattern — SNS + SQS (Fan-Out):

  Producer → SNS Topic → SQS Queue 1 (Order Service)
                       → SQS Queue 2 (Email Service)
                       → SQS Queue 3 (Analytics Service)
                       → Lambda Function (Real-time processing)

Each subscriber gets its own copy of the message!
```

Event Source → Event Bus → Rules (filter + route) → Targets

Event Sources:                    Targets:

**Key advantage over SNS:** EventBridge supports **content-based filtering** — you can match on any field in the event JSON, not just message attributes.

### Q30. What is a Cold Start in Lambda? How do you minimize it?

**Answer:**

A cold start happens when Lambda has to create a **new execution environment** from scratch. This adds latency to the first request.

```
Cold Start (first request or after scaling):
  Download code → Create container → Initialize runtime → Run init code → Run handler
  ├──────────── This part is the "cold start" ────────────┤├── Your code ──┤
  
Warm Start (subsequent requests — container is reused):
  Run handler
  ├── Your code (no overhead) ──┤
```

**How to minimize cold starts:**

| Technique | How It Helps |
|-----------|-------------|
| **Provisioned Concurrency** | Pre-warms N containers (no cold starts, but you pay for it) |
| **Keep functions small** | Smaller deployment = faster download and init |
| **Use lighter runtimes** | Python, Node.js start faster than Java, .NET |
| **Avoid heavy SDKs** | Import only what you need |
| **Initialize outside handler** | Code outside the handler runs once (during cold start) and is reused |
| **SnapStart (Java)** | AWS takes a snapshot of initialized memory — restores it instantly |

```python
# Example — Initialize OUTSIDE the handler (reused in warm starts)
import boto3

# This runs ONCE during cold start and is reused for subsequent invocations
dynamodb = boto3.resource('dynamodb')
table = dynamodb.Table('my-table')

def lambda_handler(event, context):
    # This runs on EVERY invocation
    response = table.get_item(Key={'id': event['id']})
    return response['Item']
```

### Q32. What is the difference between AWS Secrets Manager and SSM Parameter Store?

**Answer:**

Both store secrets and configuration, but they're designed for different use cases:

| Feature | Secrets Manager | SSM Parameter Store |
|---------|----------------|---------------------|
| **Purpose** | Specifically for secrets (passwords, API keys, DB creds) | General config + secrets |
| **Auto-rotation** | ✅ Built-in rotation for RDS, Redshift, DocumentDB | ❌ No built-in rotation |
| **Cross-account** | ✅ Easily share across accounts | Requires more setup |
| **Cost** | $0.40/secret/month + $0.05/10K API calls | Free tier (Standard) or $0.05/advanced param/month |

| **Versioning** | ✅ Automatic | ✅ Available |
| **Encryption** | ✅ Always encrypted (KMS) | Optional (SecureString uses KMS) |

```
Rule of thumb:
├── Database passwords that need auto-rotation → Secrets Manager
├── API keys, tokens                           → Either one works
├── Feature flags, config values               → SSM Parameter Store
└── Cost-sensitive, lots of parameters         → SSM Parameter Store (it's cheaper/free)
```

**Example — Retrieve a secret in code:**

response = client.get_secret_value(SecretId='prod/database/mysql')

# SSM Parameter Store
ssm = boto3.client('ssm')
response = ssm.get_parameter(Name='/prod/api/key', WithDecryption=True)

## 🌐 Elastic Load Balancing — Advanced

### Q40. What is Cross-Zone Load Balancing?

**Answer:**

By default, each load balancer node distributes traffic only to targets in its own Availability Zone. **Cross-Zone Load Balancing** distributes traffic evenly across ALL targets in ALL AZs.

```
WITHOUT Cross-Zone LB:

AZ-a (LB Node) → 2 instances → Each gets 25% traffic
AZ-b (LB Node) → 8 instances → Each gets 6.25% traffic

Traffic is 50/50 between AZs, but instances in AZ-a are overloaded!

WITH Cross-Zone LB:

AZ-a (LB Node) ──┐
                   ├──→ All 10 instances → Each gets 10% traffic evenly
AZ-b (LB Node) ──┘

Much better distribution!
```

| Feature | ALB | NLB |
|---------|-----|-----|
| Cross-Zone LB | **Enabled by default** (free) | Disabled by default (costs extra if enabled) |

## 📡 AWS Networking — Advanced

---

### Q41. What is VPC Peering? What are its limitations?

**Answer:**

VPC Peering creates a **private network connection** between two VPCs. Traffic stays within AWS's network (doesn't go over the internet).

```
VPC Peering:

VPC-A (10.0.0.0/16) ←── Peering Connection ──→ VPC-B (172.16.0.0/16)

Properties:
├── Works across accounts ✅
├── Works across regions ✅ (inter-region peering)
├── Traffic is encrypted (inter-region) ✅
├── No bandwidth bottleneck ✅
```

**Limitations (interview favorite):**

```
❌ No transitive peering!

VPC-A ←→ VPC-B ←→ VPC-C

VPC-A can talk to VPC-B ✅
VPC-B can talk to VPC-C ✅
VPC-A CANNOT talk to VPC-C through VPC-B ❌

To connect A to C, you need a separate peering connection.

Solution for many VPCs: Use Transit Gateway instead!
```

---

### Q42. What is AWS Transit Gateway?

**Answer:**

Transit Gateway is a **central hub** that connects multiple VPCs, on-premises networks, and VPN connections through a single gateway.

```
Without Transit Gateway (mesh peering — gets messy fast):

VPC-A ←→ VPC-B
VPC-A ←→ VPC-C
VPC-A ←→ VPC-D
VPC-B ←→ VPC-C
VPC-B ←→ VPC-D
VPC-C ←→ VPC-D
= 6 peering connections for 4 VPCs! (n*(n-1)/2)

With Transit Gateway (hub-and-spoke — clean and scalable):

         VPC-A
           ↕
VPC-D ←→ TGW ←→ VPC-B
           ↕
         VPC-C
= 4 connections for 4 VPCs! (n connections)

TGW also supports:
├── VPN connections (site-to-site)
├── Direct Connect gateways
├── Cross-region peering (TGW-to-TGW)
├── Route tables for traffic control
└── Multicast support
```

| **Vendor** | AWS (native) | HashiCorp (third-party) |

| **Cloud support** | AWS only | Multi-cloud (AWS, Azure, GCP, etc.) |
| **State** | Managed by AWS (no state file to worry about) | You manage the state file (S3 + DynamoDB) |
| **Drift detection** | Built-in | Requires `terraform plan` |
| **Rollback** | Automatic rollback on failure | No automatic rollback |
| **Community** | AWS only | Huge community, lots of modules |
| **Cost** | Free | Free (open source) or paid (Terraform Cloud) |

```
When to choose:
CloudFormation → Pure AWS shop, want zero state management hassle, need rollback
Terraform      → Multi-cloud, team already knows HCL, need reusable modules
```

### Q44. What is AWS EFS? When would you use it over EBS?

**Answer:**

EFS = **Elastic File System**. It's a fully managed, scalable **NFS file system** that can be mounted by multiple EC2 instances simultaneously.

```
Use EFS when:
├── Multiple EC2 instances need to read/write the same files
├── You need a shared file system (e.g., WordPress media, ML training data)
├── Storage needs to grow/shrink automatically
├── You need cross-AZ access

Use EBS when:
├── Single instance needs fast, low-latency storage
├── Databases (RDS, self-managed MySQL/Postgres)
├── Boot volumes
├── Applications that need high IOPS
```

**EFS Storage Classes:**

| Class | Use Case | Cost |
|-------|----------|------|
| **Standard** | Frequently accessed files | $$$  |
| **Infrequent Access (IA)** | Files accessed less often | $ |
| **One Zone** | Non-critical, single-AZ | $$ (cheaper than Standard) |
| **One Zone-IA** | Non-critical, rarely accessed | ¢ (cheapest) |

### Q45. What is AWS Systems Manager (SSM)?

**Answer:**

SSM is a **management toolbox** for your infrastructure. It provides operational tools for EC2 instances, on-prem servers, and edge devices.

```
SSM Key Features:

1. Session Manager
   └── SSH-like access to EC2 WITHOUT opening port 22
       (No SSH keys, no bastion host, no open inbound ports!)
       All sessions are logged to CloudWatch/S3

2. Parameter Store
   └── Store configuration and secrets (covered in Q32)

3. Run Command
   └── Run commands on multiple instances simultaneously
       (e.g., "Install patch on all 100 servers")

4. Patch Manager
   └── Automate OS and application patching
       Define patch baselines and maintenance windows

5. State Manager
   └── Keep instances in a desired state
       (ensure a package is always installed, a service always running)

6. Automation
   └── Runbooks for common tasks
       (stop/start instances, create AMIs, rotate keys)

7. Inventory
   └── Collect metadata from all managed instances
       (installed software, network config, Windows updates)
```

**Interview tip:** When asked "How do you SSH into an EC2 instance in a private subnet?", the modern answer is **SSM Session Manager** — no bastion host needed, no port 22, fully auditable.

### Q46. How would you design a highly available architecture on AWS?

**Answer:**

Highly Available Architecture:

                         Route 53 (DNS failover)
                              │
                    ┌─────────┴─────────┐
                    ▼                   ▼
              CloudFront          CloudFront
              (Edge cache)        (Edge cache)

          │   │   └── Cache (Redis) │  │   │   └── Cache (Redis)  │

5. **Loose coupling** — use SQS/EventBridge between services
6. **Data replication** — RDS Multi-AZ + cross-region read replicas
7. **Caching** — ElastiCache (Redis/Memcached) to reduce database load
8. **CDN** — CloudFront for static content and global performance

### Q47. How would you migrate an on-premises application to AWS?

**Answer:**

AWS recommends the **6 R's of Migration:**

```
6 R's of Migration:

1. Rehost ("Lift and Shift")
   └── Move as-is to EC2. No code changes.
       Quick but doesn't leverage cloud benefits.

2. Replatform ("Lift, Tinker, and Shift")
   └── Move with minor optimizations.
       e.g., Move from self-managed MySQL to RDS.

3. Refactor / Re-architect
   └── Redesign for the cloud (microservices, serverless).
       Most effort but most cloud-native benefits.

4. Repurchase
   └── Move to a SaaS solution.
       e.g., Replace self-hosted CRM with Salesforce.

5. Retire
   └── Turn off apps that are no longer needed.
       (You'd be surprised how many zombie apps exist!)

6. Retain
   └── Keep on-premises (not everything needs to move).
       e.g., Legacy mainframes, compliance-restricted systems.
```

**Migration tools:**

| Tool | Purpose |
|------|---------|
| **AWS Migration Hub** | Central dashboard to track migration progress |
| **AWS Application Discovery Service** | Discover on-prem servers and dependencies |
| **AWS Database Migration Service (DMS)** | Migrate databases (even between engines) |
| **AWS Server Migration Service (SMS)** | Migrate VMs to EC2 AMIs |
| **AWS DataSync** | Transfer large amounts of data to S3/EFS |
| **AWS Snow Family** | Physical devices for massive data transfers (Snowball, Snowmobile) |

### Q48. Your application is running slow on AWS. How would you troubleshoot?

**Answer:**

```
Troubleshooting Steps (systematic approach):

1. CHECK METRICS (CloudWatch)
   ├── EC2: CPU, memory, disk I/O, network
   ├── ALB: Request count, latency, 5xx errors, target response time
   ├── RDS: CPU, connections, read/write latency, free storage
   └── Lambda: Duration, errors, throttles, concurrent executions

2. CHECK LOGS
   ├── Application logs (CloudWatch Logs Insights)
   ├── ALB access logs (S3)
   ├── VPC Flow Logs (network issues)
   └── CloudTrail (API errors)

3. COMMON BOTTLENECKS
   ├── CPU bound     → Scale up (bigger instance) or scale out (more instances)
   ├── Memory bound  → Instance right-sizing, check for memory leaks
   ├── Disk I/O      → Switch to gp3/io2, check IOPS limits
   ├── Network       → Check security groups, NACLs, bandwidth limits
   ├── Database      → Slow queries, missing indexes, connection pool exhaustion
   └── DNS           → Route 53 health check failures, TTL too high

4. PERFORMANCE TOOLS
   ├── AWS X-Ray            → Distributed tracing (find slow microservice)
   ├── RDS Performance Insights → Find slow database queries
   ├── CloudWatch Contributor Insights → Find top talkers
   ├── VPC Reachability Analyzer → Debug network connectivity
   └── AWS Compute Optimizer → Right-sizing recommendations
```

### Q49. How do you implement disaster recovery on AWS?

**Answer:**

AWS defines 4 DR strategies (ordered by RTO/RPO and cost):

| Strategy | RTO | RPO | Cost | Description |
|----------|-----|-----|------|-------------|
| **Backup & Restore** | Hours | Hours | $ | Backup data to S3, restore when needed |
| **Pilot Light** | Minutes-Hours | Minutes | $$ | Core systems always running, scale up on disaster |
| **Warm Standby** | Minutes | Seconds-Minutes | $$$ | Scaled-down version always running in DR region |
| **Multi-Site Active/Active** | Near-zero | Near-zero | $$$$ | Full production in multiple regions simultaneously |

```
RTO = Recovery Time Objective  → How fast you need to recover
RPO = Recovery Point Objective → How much data loss is acceptable

Backup & Restore:   RTO: hours    RPO: hours    → "We can be down for a day"
Pilot Light:        RTO: minutes  RPO: minutes  → "We need to be back in 30 minutes"
Warm Standby:       RTO: minutes  RPO: seconds  → "We need to be back in 5 minutes"
Active/Active:      RTO: ~0       RPO: ~0       → "We can NEVER be down"
```

**Real-world implementation:**

```
Warm Standby Example:

Primary Region (us-east-1):
├── Full production workload
├── RDS Primary (Multi-AZ within region)
├── S3 with cross-region replication → DR Region
└── Route 53 health checks

DR Region (us-west-2):
├── Scaled-down infrastructure (smaller instances, fewer of them)
├── RDS Read Replica (can promote to Primary)
├── S3 replica bucket
└── Auto Scaling configured (can scale UP quickly)

On Disaster:
1. Route 53 detects primary is down (health check fails)
2. Route 53 fails over DNS to DR region
3. Promote RDS read replica to primary
4. Auto Scaling scales UP instances in DR region
5. Monitor and verify
```

### Q50. What is the AWS Well-Architected Framework?

**Answer:**

The Well-Architected Framework provides **best practices** for designing and operating reliable, secure, efficient, cost-effective, and sustainable systems in the cloud.

```
6 Pillars of the Well-Architected Framework:

1. 🔒 Operational Excellence
   └── Run and monitor systems, continuously improve
       (Automate, IaC, observability, incident response)

2. 🛡️ Security
   └── Protect data, systems, and assets
       (IAM, encryption, network security, logging)

3. ⚡ Reliability
   └── Recover from failures, meet demand
       (Multi-AZ, Auto Scaling, DR, health checks)

4. 🚀 Performance Efficiency
   └── Use resources efficiently
       (Right-sizing, caching, CDN, serverless)

5. 💰 Cost Optimization
   └── Avoid unnecessary costs
       (Right-sizing, Reserved Instances, Spot, S3 tiers)

6. 🌱 Sustainability
   └── Minimize environmental impact
       (Efficient workloads, right-sized resources, managed services)
```

**Interview tip:** When designing any architecture, mention which pillars you're addressing. For example: "I'm using Multi-AZ for **reliability**, IAM roles for **security**, Auto Scaling for **performance efficiency**, and Spot Instances for **cost optimization**."
