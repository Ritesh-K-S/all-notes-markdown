### Q1. What is Ansible? Why is it popular in DevOps?

**Answer:**

Ansible is an **open-source automation tool** used for configuration management, application deployment, and task automation. It's developed by **Red Hat**.

Think of Ansible as a **remote control for your servers** — you write instructions once, and Ansible executes them on hundreds of servers simultaneously.

```
Why Ansible is so popular:

1. Agentless    → No software to install on target servers (uses SSH)
2. Simple       → Written in YAML (human-readable, not code)
3. Powerful     → Can manage servers, cloud resources, network devices, containers
4. Idempotent   → Run the same playbook 10 times, result is always the same
5. Free         → Open source (with paid version: Ansible Automation Platform)
```

**Real-world analogy:**
```
Without Ansible:
  You SSH into 100 servers, one by one, and run the same commands.
  Takes hours. Easy to make mistakes. Impossible to track.

With Ansible:
  You write a playbook ONCE.
  Run it against 100 servers simultaneously.
  Takes minutes. Consistent. Repeatable. Documented.
```

### Q2. What is the architecture of Ansible? How does it work?

**Answer:**

```
Ansible Architecture:

┌───────────────────────────────────────────────────┐
│                CONTROL NODE                        │
│  (Your laptop or a dedicated Ansible server)       │
│                                                    │
│  ┌──────────┐  ┌──────────┐  ┌────────────────┐  │
│  │ Playbook │  │ Inventory│  │ ansible.cfg    │  │
│  │ (YAML)   │  │ (hosts)  │  │ (configuration)│  │
│  └──────────┘  └──────────┘  └────────────────┘  │
│                                                    │
│  Ansible Engine                                    │
│  ├── Reads playbook                                │
│  ├── Reads inventory                               │
│  ├── Generates Python scripts (modules)            │
│  └── Sends them over SSH to managed nodes          │
└──────────────────┬────────────────────────────────┘
                   │ SSH (port 22)
                   │ (No agent needed on target!)
          ┌────────┼────────────┐
          ▼        ▼            ▼
   ┌──────────┐ ┌──────────┐ ┌──────────┐
   │ Server 1 │ │ Server 2 │ │ Server 3 │
   │ (Managed │ │ (Managed │ │ (Managed │
   │  Node)   │ │  Node)   │ │  Node)   │
   └──────────┘ └──────────┘ └──────────┘
   
   Requirements on managed nodes:
   ├── Python installed (Python 3)
   └── SSH access from control node
```

**Key components:**

| Component | Purpose |
|-----------|---------|
| **Control Node** | The machine where Ansible is installed and playbooks run from |
| **Managed Nodes** | The target servers being managed (no Ansible needed on them) |
| **Inventory** | List of managed nodes (static file or dynamic from cloud APIs) |
| **Playbook** | YAML file with automation instructions |
| **Module** | Reusable units of work (e.g., `apt`, `yum`, `copy`, `service`) |
| **Role** | Organized collection of tasks, handlers, files, templates |
| **Plugins** | Extend Ansible (connection, callback, lookup, filter plugins) |

### Q3. What is an Ansible Inventory? Explain the different types.

**Answer:**

An inventory is the **list of servers** that Ansible manages. It tells Ansible WHAT to manage and HOW to connect.

**Static Inventory (INI format):**

```ini
# inventory/hosts.ini

[webservers]
web1.example.com ansible_host=10.0.1.10
web2.example.com ansible_host=10.0.1.11

[dbservers]
db1.example.com ansible_host=10.0.2.10 ansible_user=dbadmin
db2.example.com ansible_host=10.0.2.11

[appservers]
app[1:5].example.com    # Expands to app1, app2, app3, app4, app5

[production:children]    # Group of groups
webservers
dbservers
appservers

[production:vars]        # Variables for all production hosts
ansible_user=deploy
ansible_ssh_private_key_file=~/.ssh/prod_key.pem
environment=production
```

**Static Inventory (YAML format):**

```yaml
# inventory/hosts.yml
all:
  children:
    production:
      children:
        webservers:
          hosts:
            web1.example.com:
              ansible_host: 10.0.1.10
            web2.example.com:
              ansible_host: 10.0.1.11
        dbservers:
          hosts:
            db1.example.com:
              ansible_host: 10.0.2.10
      vars:
        ansible_user: deploy
        environment: production
```

**Dynamic Inventory:**

Instead of a static file, Ansible can **query cloud APIs** (AWS, Azure, GCP) to get the current list of servers automatically.

```bash
# Using the AWS EC2 plugin
# inventory/aws_ec2.yml
plugin: amazon.aws.aws_ec2
regions:
  - us-east-1
  - us-west-2
filters:
  tag:Environment: production
keyed_groups:
  - key: tags.Role
    prefix: role
  - key: placement.availability_zone
    prefix: az
```

```bash
# Verify dynamic inventory
ansible-inventory -i inventory/aws_ec2.yml --list
ansible-inventory -i inventory/aws_ec2.yml --graph
```

### Q4. What is the difference between an Ansible Module and a Plugin?

**Answer:**

| Feature | Module | Plugin |
|---------|--------|--------|
| **Where it runs** | On the **managed node** (target server) | On the **control node** (your machine) |
| **Purpose** | Perform specific tasks on target servers | Extend Ansible's core functionality |
| **Examples** | `apt`, `yum`, `copy`, `service`, `docker_container` | Connection, callback, lookup, filter plugins |
| **Written in** | Python | Python |
| **How used** | In playbook tasks | Behind the scenes or in Jinja2 templates |

```
Module Examples:
├── apt          → Install packages on Debian/Ubuntu
├── yum          → Install packages on RHEL/CentOS
├── copy         → Copy files to target
├── template     → Render Jinja2 templates and copy
├── service      → Start/stop/restart services
├── user         → Manage users
├── file         → Manage files and directories
├── command      → Run raw commands (not idempotent!)
├── shell        → Run shell commands (not idempotent!)
├── docker_container → Manage Docker containers
└── k8s          → Manage Kubernetes resources

Plugin Examples:
├── connection: ssh, winrm, local, docker
├── callback: json, yaml, minimal, timer
├── lookup: file, env, password, aws_ssm
└── filter: to_json, to_yaml, regex_replace, ipaddr
```

### Q5. What is Idempotency in Ansible? Why is it important?

**Answer:**

**Idempotency** means that running the same operation multiple times produces the **exact same result** — no matter how many times you run it.

```
Idempotent (GOOD):
  Run playbook 1st time → Nginx installed ✅
  Run playbook 2nd time → Nginx already installed, no change ✅ (status: "ok")
  Run playbook 3rd time → Still no change ✅ (status: "ok")

Non-Idempotent (BAD):
  Run script 1st time → Adds a line to config ✅
  Run script 2nd time → Adds the SAME line AGAIN ❌ (duplicate!)
  Run script 3rd time → Adds it AGAIN ❌ (config is now broken!)
```

**Why it matters:**
```
1. Safety    → You can run playbooks without fear of breaking things
2. Recovery  → If a playbook fails halfway, just re-run it
3. Drift fix → If someone manually changes a server, Ansible puts it back
4. CI/CD     → You can run Ansible in pipelines repeatedly
```

**Idempotent vs Non-Idempotent modules:**

| Idempotent (Preferred) | Not Idempotent (Use with caution) |
|------------------------|-----------------------------------|
| `apt`, `yum` (state: present) | `command` |
| `copy`, `template` | `shell` |
| `service` (state: started) | `raw` |
| `user`, `group` | `script` |
| `file` (state: directory) | |

```yaml
# Idempotent — safe to run multiple times
- name: Install Nginx
  apt:
    name: nginx
    state: present
  # If Nginx is already installed, this does NOTHING (ok, no change)

# NOT idempotent — DANGER if run multiple times
- name: Add config line (BAD WAY)
  shell: echo "server_name example.com;" >> /etc/nginx/nginx.conf
  # This APPENDS the line EVERY TIME you run it! 💥

# Idempotent alternative
- name: Add config line (GOOD WAY)
  lineinfile:
    path: /etc/nginx/nginx.conf
    line: "server_name example.com;"
    state: present
  # This checks if the line exists first, only adds it if missing ✅
```

### Q6. What is an Ansible Playbook? Explain its structure.

**Answer:**

A playbook is a **YAML file** containing a set of automation instructions (plays). Each play maps a group of hosts to a set of tasks.

```yaml
# deploy-app.yml — Complete playbook example
---
# Play 1: Configure web servers
- name: Configure Web Servers
  hosts: webservers             # Which hosts to target
  become: yes                   # Run as root (sudo)
  gather_facts: yes             # Collect system info before running tasks
  vars:                         # Variables for this play
    app_port: 8080
    app_user: appuser

  pre_tasks:                    # Tasks that run BEFORE roles
    - name: Update apt cache
      apt:
        update_cache: yes
        cache_valid_time: 3600

  roles:                        # Include roles
    - nginx
    - nodejs

  tasks:                        # Main tasks
    - name: Create application user
      user:
        name: "{{ app_user }}"
        shell: /bin/bash
        state: present

    - name: Deploy application code
      copy:
        src: ./app/
        dest: /opt/myapp/
        owner: "{{ app_user }}"
        mode: '0755'
      notify: Restart application     # Trigger handler if this task changes something

    - name: Ensure application is running
      service:
        name: myapp
        state: started
        enabled: yes

  post_tasks:                   # Tasks that run AFTER everything else
    - name: Verify application health
      uri:
        url: "http://localhost:{{ app_port }}/health"
        status_code: 200
      register: health_check
      retries: 5
      delay: 10

  handlers:                     # Only run when notified (triggered by "notify")
    - name: Restart application
      service:
        name: myapp
        state: restarted

# Play 2: Configure database servers
- name: Configure Database Servers
  hosts: dbservers
  become: yes
  roles:
    - postgresql
```

**Playbook structure breakdown:**

```
Playbook
├── Play 1
│   ├── hosts          → Which servers to target
│   ├── become         → Privilege escalation (sudo)
│   ├── vars           → Variables
│   ├── pre_tasks      → Tasks before roles
│   ├── roles          → Reusable role collections
│   ├── tasks          → Main tasks
│   ├── post_tasks     → Tasks after everything
│   └── handlers       → Actions triggered by "notify"
├── Play 2
│   └── ...
└── Play N
```

### Q7. What are Handlers in Ansible? How are they different from Tasks?

**Answer:**

Handlers are **special tasks** that only run when they are **notified** by another task. They're typically used to restart services after a configuration change.

```yaml
# Without handlers (BAD):
- name: Update Nginx config
  template:
    src: nginx.conf.j2
    dest: /etc/nginx/nginx.conf

- name: Restart Nginx
  service:
    name: nginx
    state: restarted
  # ❌ This restarts Nginx EVERY TIME, even if the config didn't change!


# With handlers (GOOD):
tasks:
  - name: Update Nginx config
    template:
      src: nginx.conf.j2
      dest: /etc/nginx/nginx.conf
    notify: Restart Nginx
    # ✅ Only notifies the handler IF the config actually changed

  - name: Update SSL certificate
    copy:
      src: ssl/cert.pem
      dest: /etc/nginx/ssl/cert.pem
    notify: Restart Nginx
    # ✅ Same handler can be notified by multiple tasks

handlers:
  - name: Restart Nginx
    service:
      name: nginx
      state: restarted
    # ✅ Runs ONCE at the end of the play, even if notified multiple times
```

**Key differences:**

| Feature | Task | Handler |
|---------|------|---------|
| **When it runs** | Always (in order) | Only when notified |
| **How many times** | Every playbook run | Once per play (even if notified multiple times) |
| **When in the play** | In sequence with other tasks | At the END of the play (after all tasks) |
| **Use case** | Install, configure, deploy | Restart/reload services after config changes |

### Q8. What is `become` in Ansible? How does privilege escalation work?

**Answer:**

`become` tells Ansible to **run tasks with elevated privileges** (like `sudo`).

```yaml
# At the play level — ALL tasks run as root
- name: Configure servers
  hosts: webservers
  become: yes                    # All tasks in this play run as root
  become_method: sudo            # How to escalate (default is sudo)
  become_user: root              # Which user to become (default is root)

  tasks:
    - name: Install package (needs root)
      apt:
        name: nginx
        state: present

    - name: Deploy app as specific user
      copy:
        src: app.jar
        dest: /opt/app/app.jar
      become_user: appuser       # Override — become appuser, not root

    - name: Read user file (no root needed)
      command: cat /home/deploy/notes.txt
      become: no                 # Override — DON'T use sudo for this task
```

**Privilege escalation methods:**

| Method | Description |
|--------|-------------|
| `sudo` | Most common (default) |
| `su` | Switch user |
| `pbrun` | PowerBroker |
| `pfexec` | Solaris privilege |
| `doas` | BSD systems |
| `runas` | Windows |

### Q9. How do you use Variables in Ansible? What is variable precedence?

**Answer:**

Variables make your playbooks **dynamic and reusable**. They can come from many sources:

```yaml
# 1. In the playbook (vars section)
- hosts: webservers
  vars:
    http_port: 80
    app_name: myapp

# 2. In separate variable files (vars_files)
- hosts: webservers
  vars_files:
    - vars/common.yml
    - vars/{{ environment }}.yml

# 3. In inventory (host_vars / group_vars)
# inventory/group_vars/webservers.yml
http_port: 80
max_connections: 1000

# inventory/host_vars/web1.example.com.yml
http_port: 8080    # Override for this specific host

# 4. Registered variables (from task output)
- name: Check disk space
  command: df -h /
  register: disk_output

- name: Show disk space
  debug:
    msg: "{{ disk_output.stdout }}"

# 5. Facts (automatically gathered system info)
- name: Show OS info
  debug:
    msg: "OS: {{ ansible_distribution }} {{ ansible_distribution_version }}"

# 6. Command line (-e or --extra-vars)
# ansible-playbook deploy.yml -e "version=2.0 environment=prod"
```

**Variable precedence (lowest to highest — higher wins):**

```
Variable Precedence (simplified — 22 levels exist, here are the most important):

Lowest priority:
  1.  Role defaults (roles/myrole/defaults/main.yml)
  2.  Inventory group_vars
  3.  Inventory host_vars
  4.  Playbook group_vars
  5.  Playbook host_vars
  6.  Play vars
  7.  Play vars_files
  8.  Role vars (roles/myrole/vars/main.yml)
  9.  Task vars (set_fact, register)
  10. Extra vars (-e on command line)           ← ALWAYS WINS
Highest priority:

Memory trick: "Extra vars (-e) ALWAYS win. Role defaults ALWAYS lose."
```

### Q10. What is the difference between `copy`, `template`, and `file` modules?

**Answer:**

| Module | Purpose | Example |
|--------|---------|---------|
| `copy` | Copy a **static file** from control node to managed node | Copy a binary, script, or config file as-is |
| `template` | Copy a **Jinja2 template** (with variable substitution) | Generate config files with dynamic values |
| `file` | Manage file/directory **properties** (doesn't copy content) | Create dirs, set permissions, create symlinks |

```yaml
# copy — Static file (no variable substitution)
- name: Copy static config file
  copy:
    src: files/my-script.sh          # File on control node
    dest: /opt/scripts/my-script.sh  # Destination on managed node
    owner: root
    group: root
    mode: '0755'

# template — Dynamic file (Jinja2 variables are replaced)
- name: Generate Nginx config from template
  template:
    src: templates/nginx.conf.j2     # Jinja2 template on control node
    dest: /etc/nginx/nginx.conf      # Rendered file on managed node
    owner: root
    group: root
    mode: '0644'
  notify: Reload Nginx

# file — Manage file/directory properties
- name: Create application directory
  file:
    path: /opt/myapp
    state: directory
    owner: appuser
    group: appuser
    mode: '0755'

- name: Create a symbolic link
  file:
    src: /opt/myapp/current
    dest: /var/www/html
    state: link
```

**Jinja2 template example:**

```jinja2
{# templates/nginx.conf.j2 #}
server {
    listen {{ http_port }};
    server_name {{ server_name }};
    
    location / {
        proxy_pass http://{{ app_host }}:{{ app_port }};
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
    
    {% if ssl_enabled %}
    listen 443 ssl;
    ssl_certificate {{ ssl_cert_path }};
    ssl_certificate_key {{ ssl_key_path }};
    {% endif %}
    
    {% for location in custom_locations %}
    location {{ location.path }} {
        proxy_pass {{ location.backend }};
    }
    {% endfor %}
}
```

### Q11. What is an Ansible Role? Explain the directory structure.

**Answer:**

A Role is a **structured, reusable way** to organize your Ansible code. Instead of putting everything in one giant playbook, you break it into roles.

```
Think of roles like:
  Playbook = An entire recipe book
  Role     = A single recipe (e.g., "How to set up Nginx")
  
Each role is self-contained with everything it needs:
  tasks, handlers, files, templates, variables, defaults
```

**Role directory structure:**

```
roles/
└── nginx/                    # Role name
    ├── tasks/
    │   └── main.yml          # Main list of tasks (entry point)
    ├── handlers/
    │   └── main.yml          # Handlers (restart, reload, etc.)
    ├── templates/
    │   └── nginx.conf.j2     # Jinja2 templates
    ├── files/
    │   └── custom-error.html # Static files to copy
    ├── vars/
    │   └── main.yml          # High-priority variables (DON'T override easily)
    ├── defaults/
    │   └── main.yml          # Default variables (easily overridable)
    ├── meta/
    │   └── main.yml          # Role metadata (dependencies, author, license)
    ├── tests/
    │   ├── inventory
    │   └── test.yml          # Test playbook for this role
    └── README.md             # Documentation
```

**How to use a role:**

```yaml
# Method 1: Simple
- hosts: webservers
  roles:
    - nginx
    - nodejs

# Method 2: With variables
- hosts: webservers
  roles:
    - role: nginx
      vars:
        http_port: 8080
        worker_processes: 4

# Method 3: With conditions and tags
- hosts: webservers
  roles:
    - role: nginx
      when: install_nginx | default(true)
      tags: ['nginx', 'webserver']

# Method 4: Using include_role (dynamic, inside tasks)
- hosts: webservers
  tasks:
    - name: Include nginx role
      include_role:
        name: nginx
      when: ansible_os_family == "Debian"
```

**Creating a role from scratch:**

```bash
# Use ansible-galaxy to create the directory structure
ansible-galaxy role init my-new-role

# This creates the full directory structure automatically:
# my-new-role/
#   ├── defaults/main.yml
#   ├── files/
#   ├── handlers/main.yml
#   ├── meta/main.yml
#   ├── tasks/main.yml
#   ├── templates/
#   ├── tests/
#   └── vars/main.yml
```

### Q12. What is Ansible Galaxy?

**Answer:**

Ansible Galaxy is a **community hub** for sharing and downloading Ansible roles and collections. Think of it as the **npm/pip for Ansible**.

```bash
# Search for roles
ansible-galaxy search nginx

# Install a role from Galaxy
ansible-galaxy install geerlingguy.nginx

# Install from a requirements file
# requirements.yml:
# - name: geerlingguy.nginx
#   version: "3.1.0"
# - name: geerlingguy.docker
#   version: "6.0.0"

ansible-galaxy install -r requirements.yml

# Install a collection (newer format — contains roles + modules + plugins)
ansible-galaxy collection install community.docker
ansible-galaxy collection install amazon.aws

# List installed roles
ansible-galaxy list
```

**Roles vs Collections:**

| Feature | Role | Collection |
|---------|------|------------|
| **Contains** | Tasks, handlers, files, templates, vars | Roles + Modules + Plugins + Playbooks |
| **Scope** | Single purpose (e.g., install Nginx) | Broader scope (e.g., all AWS modules) |
| **Namespace** | `author.rolename` | `namespace.collectionname` |
| **Install command** | `ansible-galaxy role install` | `ansible-galaxy collection install` |
| **Example** | `geerlingguy.nginx` | `amazon.aws`, `community.docker` |

## 🔐 Ansible Vault

### Q13. What is Ansible Vault? How do you use it?

**Answer:**

Ansible Vault is a feature that lets you **encrypt sensitive data** (passwords, API keys, certificates) so you can safely store them in version control (Git).

```bash
# Create a new encrypted file
ansible-vault create secrets.yml
# Opens your editor — type your secrets, save, and the file is encrypted

# Encrypt an existing file
ansible-vault encrypt vars/prod-secrets.yml

# Decrypt a file (back to plaintext)
ansible-vault decrypt vars/prod-secrets.yml

# Edit an encrypted file (decrypts, opens editor, re-encrypts on save)
ansible-vault edit vars/prod-secrets.yml

# View contents without decrypting the file
ansible-vault view vars/prod-secrets.yml

# Change the vault password
ansible-vault rekey vars/prod-secrets.yml

# Encrypt a single string (inline)
ansible-vault encrypt_string 'SuperSecretP@ss!' --name 'db_password'
# Output:
# db_password: !vault |
#   $ANSIBLE_VAULT;1.1;AES256
#   62613966356565323...
```

**Using vault in playbooks:**

```yaml
# vars/secrets.yml (encrypted with ansible-vault)
db_password: MySecretPassword123
api_key: sk-abc123def456

# playbook.yml
- hosts: dbservers
  vars_files:
    - vars/secrets.yml
  tasks:
    - name: Configure database
      mysql_user:
        name: admin
        password: "{{ db_password }}"
```

**Running playbooks with vault:**

```bash
# Prompt for vault password
ansible-playbook deploy.yml --ask-vault-pass

# Use a password file (for CI/CD — DON'T commit this file!)
ansible-playbook deploy.yml --vault-password-file ~/.vault_pass.txt

# Use environment variable
export ANSIBLE_VAULT_PASSWORD_FILE=~/.vault_pass.txt
ansible-playbook deploy.yml

# Multiple vault IDs (different passwords for different environments)
ansible-playbook deploy.yml \
  --vault-id dev@~/.vault_pass_dev.txt \
  --vault-id prod@~/.vault_pass_prod.txt
```

### Q14. How do you use Conditionals in Ansible?

**Answer:**

Use `when` to run tasks only when certain conditions are true:

```yaml
tasks:
  # Simple condition
  - name: Install Nginx on Debian-based systems
    apt:
      name: nginx
      state: present
    when: ansible_os_family == "Debian"

  - name: Install Nginx on RedHat-based systems
    yum:
      name: nginx
      state: present
    when: ansible_os_family == "RedHat"

  # Multiple conditions (AND)
  - name: Restart app only in production and if config changed
    service:
      name: myapp
      state: restarted
    when:
      - environment == "production"
      - config_changed | bool

  # OR condition
  - name: Install package on either Debian or Ubuntu
    apt:
      name: curl
      state: present
    when: ansible_distribution == "Debian" or ansible_distribution == "Ubuntu"

  # Check if variable is defined
  - name: Set custom port if provided
    template:
      src: app.conf.j2
      dest: /etc/app.conf
    when: custom_port is defined

  # Check command result
  - name: Check if Docker is installed
    command: docker --version
    register: docker_check
    ignore_errors: yes
    changed_when: false

  - name: Install Docker if not present
    include_role:
      name: docker
    when: docker_check.rc != 0
```

### Q15. How do you use Loops in Ansible?

**Answer:**

Ansible provides several ways to loop over data:

```yaml
tasks:
  # Simple loop (list of items)
  - name: Install multiple packages
    apt:
      name: "{{ item }}"
      state: present
    loop:
      - nginx
      - git
      - curl
      - htop
      - vim

  # Even simpler — pass a list directly to the module
  - name: Install multiple packages (better way)
    apt:
      name:
        - nginx
        - git
        - curl
        - htop
      state: present
    # ✅ This is faster because it runs apt ONCE, not once per package

  # Loop over a list of dictionaries
  - name: Create multiple users
    user:
      name: "{{ item.name }}"
      groups: "{{ item.groups }}"
      shell: "{{ item.shell }}"
      state: present
    loop:
      - { name: 'alice', groups: 'developers', shell: '/bin/bash' }
      - { name: 'bob', groups: 'operations', shell: '/bin/zsh' }
      - { name: 'charlie', groups: 'developers,operations', shell: '/bin/bash' }

  # Loop with index
  - name: Create numbered config files
    template:
      src: worker.conf.j2
      dest: "/etc/workers/worker-{{ ansible_loop.index }}.conf"
    loop:
      - { port: 8001, name: "worker-1" }
      - { port: 8002, name: "worker-2" }
      - { port: 8003, name: "worker-3" }
    loop_control:
      extended: yes

  # Loop with dict2items (loop over a dictionary)
  - name: Set sysctl parameters
    sysctl:
      name: "{{ item.key }}"
      value: "{{ item.value }}"
      state: present
    loop: "{{ sysctl_params | dict2items }}"
    vars:
      sysctl_params:
        net.core.somaxconn: 65535
        vm.swappiness: 10
        fs.file-max: 2097152

  # Loop with fileglob (loop over files)
  - name: Copy all config files
    copy:
      src: "{{ item }}"
      dest: /etc/myapp/conf.d/
    loop: "{{ lookup('fileglob', 'files/configs/*.conf') | list }}"
```

### Q16. How do you handle errors in Ansible?

**Answer:**

```yaml
tasks:
  # 1. ignore_errors — Continue even if this task fails
  - name: Try to stop a service that might not exist
    service:
      name: old-service
      state: stopped
    ignore_errors: yes

  # 2. failed_when — Custom failure condition
  - name: Run health check
    uri:
      url: http://localhost:8080/health
    register: health
    failed_when: health.status != 200 and health.status != 201

  # 3. changed_when — Control when a task is reported as "changed"
  - name: Check application version
    command: /opt/app/bin/version
    register: app_version
    changed_when: false    # This command never changes anything

  # 4. block/rescue/always — Try/Catch/Finally for Ansible
  - name: Deploy with rollback capability
    block:
      - name: Deploy new version
        copy:
          src: app-v2.jar
          dest: /opt/app/app.jar
        
      - name: Restart application
        service:
          name: myapp
          state: restarted
        
      - name: Health check
        uri:
          url: http://localhost:8080/health
          status_code: 200
        retries: 5
        delay: 10
    
    rescue:    # Runs if ANY task in "block" fails (like "catch")
      - name: Rollback to previous version
        copy:
          src: app-v1.jar
          dest: /opt/app/app.jar
        
      - name: Restart with old version
        service:
          name: myapp
          state: restarted
        
      - name: Send alert
        slack:
          token: "{{ slack_token }}"
          msg: "⚠️ Deployment failed! Rolled back to v1."
          channel: "#deployments"
    
    always:    # Runs no matter what (like "finally")
      - name: Clean up temp files
        file:
          path: /tmp/deploy-artifacts
          state: absent

  # 5. any_errors_fatal — Stop ALL hosts if ONE host fails
  - name: Critical deployment
    hosts: webservers
    any_errors_fatal: true    # If one server fails, stop deploying to all
    tasks:
      - name: Deploy
        copy:
          src: app.jar
          dest: /opt/app/app.jar

  # 6. max_fail_percentage — Allow some failures
  - name: Rolling deployment
    hosts: webservers
    max_fail_percentage: 25    # Allow up to 25% of hosts to fail
    serial: 5                  # Deploy 5 servers at a time
    tasks:
      - name: Deploy
        copy:
          src: app.jar
          dest: /opt/app/app.jar
```

### Q17. What is the difference between `include_*` and `import_*`?

**Answer:**

This is a **subtle but important** distinction:

| Feature | `import_*` (Static) | `include_*` (Dynamic) |
|---------|---------------------|----------------------|
| **When processed** | At **parse time** (before execution) | At **runtime** (during execution) |
| **Conditionals** | Applied to EACH imported task individually | Applied to the include itself |
| **Loops** | Cannot be used with loops | CAN be used with loops |
| **Tags** | Tags on imported tasks work as expected | Tasks inside `include_*` can't be tagged from outside |
| **Handlers** | Can notify handlers inside imported files | Handlers inside included files may not be visible |
| **Use when** | You know WHAT to include at parse time | You need to DECIDE at runtime what to include |

```yaml
# import_tasks — STATIC (processed at parse time)
- name: Always import these tasks
  import_tasks: tasks/setup.yml
  when: setup_needed    # ⚠️ This condition is applied to EACH task inside setup.yml

# include_tasks — DYNAMIC (processed at runtime)
- name: Conditionally include tasks
  include_tasks: "tasks/{{ ansible_os_family | lower }}.yml"
  # ✅ The filename is determined at runtime!

# Dynamic include with loop
- name: Include role-specific tasks
  include_tasks: "tasks/{{ item }}.yml"
  loop:
    - firewall
    - monitoring
    - logging
  when: item in enabled_features
```

**Rule of thumb:**
```
Use import_tasks when: You ALWAYS want to include the same file
Use include_tasks when: You need to DECIDE at runtime (dynamic filenames, loops)
```

### Q18. What is `serial` in Ansible? How does rolling deployment work?

**Answer:**

`serial` controls how many hosts Ansible processes at a time. By default, Ansible runs tasks on ALL hosts simultaneously. With `serial`, you process hosts in **batches** — this is how you do **rolling deployments**.

```yaml
# Without serial — ALL hosts at once (downtime!)
- name: Deploy to web servers
  hosts: webservers    # If you have 10 servers
  tasks:               # ALL 10 go down at the same time 💥
    - name: Stop app
      service: name=myapp state=stopped

# With serial — Rolling deployment (zero downtime!)
- name: Rolling deployment
  hosts: webservers
  serial: 2            # Process 2 servers at a time
  # Batch 1: web1, web2  (other 8 still serving traffic)
  # Batch 2: web3, web4  (web1, web2 back up, others still serving)
  # ...continues until all servers are updated

  tasks:
    - name: Pull latest code
      git:
        repo: https://github.com/myorg/myapp.git
        dest: /opt/myapp
        version: "{{ deploy_version }}"

    - name: Restart application
      service:
        name: myapp
        state: restarted

    - name: Wait for health check
      uri:
        url: "http://localhost:8080/health"
        status_code: 200
      retries: 10
      delay: 5
      register: health
      until: health.status == 200

# Serial can also be a percentage
- name: Canary deployment
  hosts: webservers
  serial:
    - 1           # First, deploy to just 1 server (canary)
    - 25%         # Then 25% of remaining servers
    - 100%        # Then all remaining servers
```

### Q21. What is the difference between Ansible and Terraform?

**Answer:**

This is a **super common** interview question. Know this well:

| Feature | Ansible | Terraform |
|---------|---------|-----------|
| **Primary purpose** | Configuration management (configure servers) | Infrastructure provisioning (create servers) |
| **Approach** | Procedural (do this, then this, then this) | Declarative (this is what I want, figure it out) |
| **State** | Stateless (no state file) | Stateful (tracks everything in state file) |
| **Agent** | Agentless (SSH) | Agentless (API calls) |
| **Language** | YAML (playbooks) | HCL (HashiCorp Configuration Language) |
| **Strength** | Server configuration, app deployment, orchestration | Cloud infrastructure, networking, IAM |
| **Idempotency** | Most modules are idempotent | Built-in (declarative by nature) |
| **Dry run** | `--check` mode | `terraform plan` |
| **Community** | Huge (Ansible Galaxy) | Huge (Terraform Registry) |

```
Real-world combo (they complement each other!):

Terraform creates the infrastructure:
  ├── VPC, Subnets, Security Groups
  ├── EC2 instances, RDS, S3
  └── Load Balancers, Auto Scaling Groups

Ansible configures what's inside:
  ├── Install packages (Nginx, Java, Docker)
  ├── Deploy application code
  ├── Configure monitoring agents
  └── Manage users, firewall rules, cron jobs

Think of it as:
  Terraform = Builds the house (foundation, walls, roof)
  Ansible   = Furnishes the house (furniture, appliances, decorations)
```

### Q22. What is Ansible Dynamic Inventory? When would you use it?

**Answer:**

Dynamic inventory **automatically discovers** your servers by querying cloud APIs instead of maintaining a static file.

```
Static Inventory (manual, gets outdated):
  You add/remove servers from a file manually.
  Problem: In auto-scaling environments, servers come and go constantly!

Dynamic Inventory (automatic, always current):
  Ansible queries AWS/Azure/GCP APIs to get the current list of servers.
  It always has the LATEST server list.
```

**AWS EC2 Dynamic Inventory example:**

```yaml
# inventory/aws_ec2.yml
plugin: amazon.aws.aws_ec2
regions:
  - us-east-1
  - us-west-2

# Filter — only get production servers
filters:
  tag:Environment:
    - production
  instance-state-name:
    - running

# Create groups based on tags
keyed_groups:
  - key: tags.Role
    prefix: role            # Creates groups like: role_webserver, role_database
    separator: "_"
  - key: tags.Environment
    prefix: env             # Creates groups like: env_production, env_staging
  - key: placement.region
    prefix: region          # Creates groups like: region_us_east_1

# Set connection variables
compose:
  ansible_host: private_ip_address    # Connect via private IP
  ansible_user: "'ec2-user'"          # SSH user

# Caching (avoid hitting AWS API every time)
cache: true
cache_plugin: jsonfile
cache_timeout: 300                    # Cache for 5 minutes
cache_connection: /tmp/aws_inventory_cache
```

```bash
# Test your dynamic inventory
ansible-inventory -i inventory/aws_ec2.yml --graph
ansible-inventory -i inventory/aws_ec2.yml --list

# Use it with a playbook
ansible-playbook -i inventory/aws_ec2.yml deploy.yml

# Target specific dynamic groups
ansible-playbook -i inventory/aws_ec2.yml deploy.yml --limit role_webserver
```

### Q23. How do you test Ansible code?

**Answer:**

```
Testing Pyramid for Ansible:

          ┌─────────────┐
          │  Integration │  ← Molecule (full lifecycle test)
          │    Tests     │
          ├─────────────┤
          │   Linting   │  ← ansible-lint, yamllint
          ├─────────────┤
          │  Syntax     │  ← ansible-playbook --syntax-check
          │  Check      │
          └─────────────┘
```

**Testing methods:**

```bash
# 1. Syntax check — Verify YAML and Ansible syntax
ansible-playbook deploy.yml --syntax-check

# 2. Dry run (Check mode) — Show what WOULD change without actually doing it
ansible-playbook deploy.yml --check --diff

# 3. Linting — Check for best practices and common mistakes
pip install ansible-lint yamllint
ansible-lint playbooks/
yamllint playbooks/

# 4. Molecule — Full integration testing framework
pip install molecule molecule-docker
cd roles/nginx
molecule init scenario --driver-name docker
molecule test    # Create → Converge → Verify → Destroy
```

**Molecule test lifecycle:**

```
molecule test runs this sequence:

1. dependency   → Install role dependencies
2. lint         → Run linters
3. cleanup      → Clean up from previous runs
4. destroy      → Destroy existing test instances
5. syntax       → Check syntax
6. create       → Create test instances (Docker containers)
7. prepare      → Pre-configure test instances
8. converge     → Run your role/playbook (the actual test!)
9. idempotence  → Run AGAIN — verify nothing changes (idempotency check)
10. verify      → Run Testinfra/Ansible assertions
11. cleanup     → Clean up
12. destroy     → Destroy test instances
```

**Molecule scenario example (`molecule/default/molecule.yml`):**

```yaml
  name: docker

  - name: ubuntu-test
```

### Q24. How do you optimize Ansible performance?

**Answer:**

```yaml
# ansible.cfg — Performance tuning
[defaults]
# 1. Increase parallelism (default is 5)
forks = 20        # Run tasks on 20 hosts simultaneously

# 2. Use pipelining (reduces SSH connections)
[ssh_connection]
pipelining = True
# ⚠️ Requires "requiretty" to be disabled in sudoers

# 3. Enable SSH multiplexing (reuse SSH connections)
ssh_args = -o ControlMaster=auto -o ControlPersist=60s

# 4. Disable fact gathering if not needed
[defaults]
gathering = smart    # Cache facts, only gather if not cached
fact_caching = jsonfile
fact_caching_connection = /tmp/ansible_facts_cache
fact_caching_timeout = 3600

# 5. Use callback plugins to identify slow tasks
callback_whitelist = timer, profile_tasks, profile_roles
```

**In playbooks:**

```yaml
# Disable fact gathering if you don't need system facts
- hosts: webservers
  gather_facts: no     # Saves 2-5 seconds per host
  tasks:
    - name: Quick task that doesn't need facts
      copy:
        content: "Hello World"
        dest: /tmp/hello.txt

# Use free strategy (tasks run as fast as each host allows)
- hosts: webservers
  strategy: free       # Default is "linear" (waits for all hosts per task)
  tasks:
    - name: Install packages
      apt:
        name: nginx
        state: present

# Use async for long-running tasks
- name: Long running database migration
  command: /opt/app/migrate.sh
  async: 3600          # Maximum runtime: 1 hour
  poll: 0              # Don't wait — fire and forget

- name: Wait for migration to complete
  async_status:
    jid: "{{ migration_job.ansible_job_id }}"
  register: job_result
  until: job_result.finished
  retries: 120
  delay: 30
```

### Q28. How do you manage Windows servers with Ansible?

Ansible manages Windows servers using **WinRM** (Windows Remote Management) instead of SSH.


```yaml
- name: Configure Windows Servers

        name: Web-Server
        state: present
        include_management_tools: yes

    - name: Create a directory
      win_file:
        path: C:\inetpub\myapp
        state: directory

    - name: Copy website files
      win_copy:
        src: files/website/
        dest: C:\inetpub\myapp\

    - name: Install software with Chocolatey
      win_chocolatey:
        name:
          - git
          - notepadplusplus
          - 7zip
        state: present

    - name: Ensure Windows service is running
      win_service:
        name: W3SVC      # IIS service
        state: started
        start_mode: auto

    - name: Run PowerShell command
      win_shell: |
        Get-Process | Where-Object { $_.CPU -gt 100 }
      register: heavy_processes

    - name: Set registry key
      win_regedit:
        path: HKLM:\SOFTWARE\MyApp
        name: Version
        data: "2.0.0"
        type: string
```

### Q29. How do you debug Ansible playbooks?

**Answer:**

```yaml
# 1. Verbose output (more -v = more detail)
# ansible-playbook deploy.yml -v      # Basic verbose
# ansible-playbook deploy.yml -vv     # More detail
# ansible-playbook deploy.yml -vvv    # Connection debugging
# ansible-playbook deploy.yml -vvvv   # Maximum detail (shows SSH commands)

# 2. Debug module — Print variables and messages
- name: Debug a variable
  debug:
    var: my_variable

- name: Debug with a message
  debug:
    msg: "The server {{ inventory_hostname }} has IP {{ ansible_host }}"

- name: Debug with verbosity control
  debug:
    msg: "This only shows with -vv or higher"
    verbosity: 2

# 3. Register + Debug — See task output
- name: Run a command
  command: uptime
  register: uptime_result
  changed_when: false

- name: Show command output
  debug:
    var: uptime_result.stdout

# 4. assert — Validate conditions
- name: Verify enough disk space
  assert:
    that:
      - ansible_mounts | selectattr('mount', 'equalto', '/') | map(attribute='size_available') | first > 1073741824
    fail_msg: "Less than 1GB free disk space on /"
    success_msg: "Sufficient disk space available"

# 5. Pause — Stop and wait for input
- name: Pause before continuing
  pause:
    prompt: "Press Enter to continue deploying to production (Ctrl+C to abort)"

# 6. Step mode — Run one task at a time
# ansible-playbook deploy.yml --step
# Ansible will ask "Perform task: <task name> (N)o/(y)es/(c)ontinue"

# 7. Start at a specific task
# ansible-playbook deploy.yml --start-at-task="Deploy application"

# 8. List tasks without running them
# ansible-playbook deploy.yml --list-tasks
# ansible-playbook deploy.yml --list-hosts
# ansible-playbook deploy.yml --list-tags
```

### Q30. What are Ansible Best Practices?

**Answer:**

```
Ansible Best Practices Checklist:

📁 Project Structure:
├── ✅ Use roles for reusable code (not monolithic playbooks)
├── ✅ Separate inventories per environment (dev/staging/prod)
├── ✅ Use group_vars and host_vars (not inline variables)
├── ✅ Use requirements.yml for dependencies
└── ✅ Keep playbooks in version control (Git)

🔒 Security:
├── ✅ Use Ansible Vault for ALL secrets
├── ✅ Never commit plaintext passwords or keys
├── ✅ Use SSH keys, not passwords
├── ✅ Use become (sudo) only when needed
└── ✅ Limit vault access with multiple vault IDs

📝 Writing Playbooks:
├── ✅ Name EVERY task (makes debugging 10x easier)
├── ✅ Use native modules instead of command/shell
├── ✅ Use handlers for service restarts (not tasks)
├── ✅ Use tags for partial runs
├── ✅ Use block/rescue for error handling
└── ✅ Test with --check --diff before applying

🔄 Idempotency & Reliability:
├── ✅ Ensure all tasks are idempotent
├── ✅ Use changed_when and failed_when appropriately
├── ✅ Use serial for rolling deployments
├── ✅ Set any_errors_fatal for critical tasks
└── ✅ Include health checks after deployments

📊 Performance:
├── ✅ Increase forks (default 5 is too low for large inventories)
├── ✅ Enable pipelining in ansible.cfg
├── ✅ Disable gather_facts when not needed
├── ✅ Use fact caching for large inventories
└── ✅ Use free strategy when task order doesn't matter

🧪 Testing:
├── ✅ Lint with ansible-lint and yamllint
├── ✅ Test roles with Molecule
├── ✅ Use --syntax-check in CI/CD
├── ✅ Run --check in CI before actual deployment
└── ✅ Test on multiple OS versions
```

## 🎯 Quick-Fire Questions (Bonus)

| # | Question | Quick Answer |
|---|----------|-------------|
| 1 | What port does Ansible use? | 22 (SSH) for Linux, 5985/5986 (WinRM) for Windows |
| 2 | Where is the default Ansible config file? | `/etc/ansible/ansible.cfg` or `./ansible.cfg` (project-level, highest priority) |
| 3 | What is a fact? | Auto-gathered system information (`ansible_os_family`, `ansible_hostname`, etc.) |
| 4 | What is `register`? | Captures the output of a task into a variable |
| 5 | Can Ansible manage network devices? | Yes — via `network_cli`, `netconf`, or `httpapi` connection plugins |
| 6 | What is `delegate_to`? | Run a task on a DIFFERENT host than the current target |
| 7 | What is `run_once`? | Run a task only ONCE, even if multiple hosts are targeted |
| 8 | What is `set_fact`? | Dynamically set a variable during playbook execution |
| 9 | How do you skip a task? | Use `when: false` or `--skip-tags` |
| 10 | What is the `local_action` module? | Runs a task on the control node (equivalent to `delegate_to: localhost`) |

## 🏁 Final Interview Tips for Ansible

```
Before the Interview:
├── Practice writing playbooks from scratch (no copying!)
├── Build a small project: Ansible + Vagrant or Docker
├── Know the Ansible vs Terraform distinction cold
├── Read the official Ansible documentation (it's excellent)
└── Be ready to whiteboard a role structure

During the Interview:
├── Start simple — show you can write a basic playbook
├── Mention idempotency — it shows you understand the philosophy
├── Talk about real projects — even a simple "I automated my home lab"
├── Discuss error handling — block/rescue, serial, max_fail_percentage
└── Mention testing — Molecule, ansible-lint, --check mode

Common Mistakes to Avoid:
├── ❌ Using command/shell for everything (use native modules!)
├── ❌ Hardcoding passwords in playbooks (use Vault!)
├── ❌ Not naming tasks (makes debugging impossible)
├── ❌ Running on all hosts at once in production (use serial!)
└── ❌ Not using roles (monolithic playbooks don't scale)
```
