## 🎯 What Are We Building?

We're going to deploy a **static website** (HTML, CSS, JavaScript) on a **Linux Virtual Machine** using **Nginx** as the web server.

By the end of this project, you'll have a working website that anyone can visit through a browser. This is the **most fundamental DevOps project** — if you can do this, you understand the foundation of how the internet works.

### What is a Static Website?

A static website is a website that has **fixed content** — the files (HTML, CSS, JS, images) are served **exactly as they are** to every visitor. There's no database, no server-side processing, no dynamic content.

**Examples of static websites:**
- Portfolio websites
- Landing pages
- Documentation sites
- Blogs (built with static site generators like Hugo, Jekyll)

### What is Nginx?

Nginx (pronounced "Engine-X") is one of the most popular **web servers** in the world. It's used by companies like Netflix, Airbnb, and Dropbox. It can:

- Serve static files (HTML, CSS, JS, images) super fast
- Act as a reverse proxy
- Do load balancing
- Handle SSL/TLS (HTTPS)

For this project, we'll use it to **serve our static website**.

### Step 2: Update Your System

The first thing you should ALWAYS do on a fresh Linux system is update the packages. Think of it like updating your phone's apps — you want the latest versions with bug fixes and security patches.

```bash
# Update the package list (checks what updates are available)
sudo apt update

# Upgrade all installed packages to their latest versions
sudo apt upgrade -y
```

**What's happening here?**

| Command | What It Does |
|---------|-------------|
| `sudo` | Run as superuser (admin privileges). Most system-level commands need this |
| `apt update` | Refreshes the list of available packages from Ubuntu's repositories |
| `apt upgrade -y` | Actually installs the updates. `-y` means "yes, do it without asking me" |

### Step 3: Install Nginx

Now let's install Nginx:

```bash
# Install Nginx
sudo apt install nginx -y
```

That's it! One command. Ubuntu's package manager (`apt`) downloads and installs Nginx for you.

#### Verify Nginx is installed:

```bash
# Check Nginx version
nginx -v
```

**Output:**

```
nginx version: nginx/1.18.0 (Ubuntu)
```

#### Start and Enable Nginx:

```bash
# Start Nginx (starts it RIGHT NOW)
sudo systemctl start nginx

# Enable Nginx (makes it start automatically when the server boots up)
sudo systemctl enable nginx

# Check the status (make sure it's running)
sudo systemctl status nginx
```

**What do `start` and `enable` mean?**

| Command | What It Does |
|---------|-------------|
| `systemctl start nginx` | Starts Nginx right now, in this session |
| `systemctl enable nginx` | Tells the system: "Hey, start Nginx automatically every time this server reboots" |
| `systemctl status nginx` | Shows you if Nginx is running or not, and recent logs |

**You should see output like:**

```
● nginx.service - A high performance web server and a reverse proxy server
     Loaded: loaded (/lib/systemd/system/nginx.service; enabled; vendor preset: enabled)
     Active: active (running) since Mon 2024-01-15 10:30:00 UTC; 5s ago
```

The keyword is **`active (running)`** — that means Nginx is alive and listening! 🎉

### Step 5: Configure the Firewall

A firewall controls which network traffic is allowed in and out of your server. We need to make sure Port 80 (HTTP) and Port 443 (HTTPS) are open.

```bash
# Check firewall status
sudo ufw status

# If the firewall is inactive, enable it (but first allow SSH so you don't lock yourself out!)
sudo ufw allow ssh        # Allow SSH (port 22) — CRITICAL! Don't skip this!
sudo ufw allow 'Nginx Full'  # Allow both HTTP (80) and HTTPS (443)

# Enable the firewall
sudo ufw enable

# Verify the rules
sudo ufw status
```

**Output:**

```
Status: active

To                         Action      From
--                         ------      ----
22/tcp                     ALLOW       Anywhere
Nginx Full                 ALLOW       Anywhere
22/tcp (v6)                ALLOW       Anywhere (v6)
Nginx Full (v6)            ALLOW       Anywhere (v6)
```

**⚠️ CRITICAL WARNING:** Always allow SSH (`sudo ufw allow ssh`) BEFORE enabling the firewall. If you enable the firewall without allowing SSH, you'll **lock yourself out** of your own server and won't be able to SSH back in. You'd have to access the server through the cloud provider's console.

### Step 6: Understand Nginx File Structure

Before we create our website, let's understand where Nginx keeps its stuff:

```
/etc/nginx/                   ← Main Nginx configuration directory
├── nginx.conf                ← The MAIN config file (global settings)
├── sites-available/          ← All your website configs go here (available but not active)
│   └── default              ← The default website config
├── sites-enabled/            ← Symlinks to configs that are ACTIVE
│   └── default → ../sites-available/default
├── conf.d/                   ← Additional config files
├── snippets/                 ← Reusable config snippets
└── mime.types                ← File type definitions

/var/www/                     ← Default web content directory
└── html/                    ← Default website files
    └── index.nginx-debian.html  ← The default welcome page you saw
```

**Key Concept: `sites-available` vs `sites-enabled`**

| Directory | Purpose |
|-----------|---------|
| `sites-available/` | Contains ALL website configuration files — like a shelf of books |
| `sites-enabled/` | Contains **symlinks** (shortcuts) to the configs that are actually ACTIVE — like the books you're currently reading |

To **activate** a website config, you create a symlink from `sites-available` to `sites-enabled`. To **deactivate** it, you remove the symlink. This pattern lets you keep configs around without them being active.

### Step 8: Set Proper File Permissions

File permissions are **super important** for security. Nginx needs to be able to **read** your website files, but you don't want just anyone to be able to **modify** them.

```bash
# Set the owner to www-data (the user Nginx runs as)
sudo chown -R www-data:www-data /var/www/mywebsite

# Set proper permissions:
# - Directories: 755 (owner can read/write/execute, everyone else can read/execute)
# - Files: 644 (owner can read/write, everyone else can only read)
sudo chmod -R 755 /var/www/mywebsite
sudo find /var/www/mywebsite -type f -exec chmod 644 {} \;
```

**What do these permission numbers mean?**

```
Permission: 755
  Owner:  7 = read(4) + write(2) + execute(1) = full access
  Group:  5 = read(4) + execute(1) = can read and enter directories
  Others: 5 = read(4) + execute(1) = can read and enter directories

Permission: 644
  Owner:  6 = read(4) + write(2) = can read and modify
  Group:  4 = read(4) = can only read
  Others: 4 = read(4) = can only read
```

**Why `www-data`?** Nginx runs as a user called `www-data` on Ubuntu. This is a special system user with limited privileges — it can only read web files, not mess with your system. It's a security best practice.

### Step 9: Configure Nginx to Serve Your Website

Now we need to tell Nginx: "Hey, serve files from `/var/www/mywebsite/` instead of the default location."

```bash
# Create a new Nginx config file for our website
sudo nano /etc/nginx/sites-available/mywebsite
```

Paste this configuration:

```nginx
server {
    # Listen on port 80 (HTTP)
    listen 80;
    listen [::]:80;    # Also listen on IPv6

    # Server name — your domain or IP
    # For now, use _ (underscore) which matches ANY hostname
    server_name _;

    # Root directory — where your website files are
    root /var/www/mywebsite;

    # Default file to serve when someone visits the root URL
    index index.html;

    # How to handle requests
    location / {
        # Try to serve the exact file requested
        # If not found, try it as a directory
        # If still not found, return a 404 error
        try_files $uri $uri/ =404;
    }

    # Optional: Custom error page
    error_page 404 /404.html;

    # Optional: Deny access to hidden files (files starting with .)
    location ~ /\. {
        deny all;
    }

    # Optional: Enable gzip compression for faster loading
    gzip on;
    gzip_types text/plain text/css application/json application/javascript text/xml;
}
```

**Let's break down every line of this config:**

| Directive | What It Does |
|-----------|-------------|
| `listen 80` | Tells Nginx to listen for HTTP requests on port 80 |
| `listen [::]:80` | Same but for IPv6 connections |
| `server_name _` | Matches any hostname. In production, you'd put your domain like `example.com` |
| `root /var/www/mywebsite` | The folder where your website files live |
| `index index.html` | When someone visits `/`, serve `index.html` |
| `try_files $uri $uri/ =404` | Try to find the requested file → if not found, try as directory → if still not found, show 404 |
| `location ~ /\.` | A regex location that matches hidden files (starting with `.`) and denies access |
| `gzip on` | Compresses responses to make your site load faster |

### Step 10: Enable the Website

Now we need to **activate** our website config and **deactivate** the default one:

```bash
# Remove the default website symlink (deactivate default site)
sudo rm /etc/nginx/sites-enabled/default

# Create a symlink for our website (activate our site)
sudo ln -s /etc/nginx/sites-available/mywebsite /etc/nginx/sites-enabled/

# Test the Nginx configuration for syntax errors
sudo nginx -t
```

**You should see:**

```
nginx: the configuration file /etc/nginx/nginx.conf syntax is ok
nginx: configuration file /etc/nginx/nginx.conf test is successful
```

**`nginx -t` is your best friend!** Always run it before reloading Nginx. It checks your config files for errors WITHOUT affecting the running server. If there's a typo or mistake, it'll tell you exactly where.

Now reload Nginx to apply the changes:

```bash
# Reload Nginx (applies new config without dropping active connections)
sudo systemctl reload nginx
```

**`reload` vs `restart`:**

| Command | What It Does |
|---------|-------------|
| `reload` | Reads the new config and applies it gracefully. Active connections are NOT dropped. **Use this one.** |
| `restart` | Completely stops and starts Nginx. Active connections ARE dropped. Use only if reload doesn't work |

## 🔍 Verification & Debugging

If something doesn't work, here's how to debug:

### Check if Nginx is running:

```bash
sudo systemctl status nginx
```

### Check Nginx error logs:

```bash
# The error log shows you exactly what went wrong
sudo tail -f /var/log/nginx/error.log

# The access log shows every request that comes in
sudo tail -f /var/log/nginx/access.log
```

**`tail -f`** means "show me the last few lines of this file AND keep watching for new lines." It's like a live feed. Press `Ctrl + C` to stop watching.

### Check if port 80 is being used:

```bash
sudo ss -tlnp | grep :80
```

This command shows which process is listening on port 80. You should see Nginx listed there.

### Check file permissions:

```bash
ls -la /var/www/mywebsite/
```

Make sure the files are owned by `www-data` and have proper permissions.

### Test Nginx config:

```bash
sudo nginx -t
```

Always run this after any config change!

### Common Issues & Fixes:

| Problem | Cause | Fix |
|---------|-------|-----|
| Can't access from browser | Firewall blocking port 80 | `sudo ufw allow 80/tcp` |
| 403 Forbidden | Wrong file permissions | `sudo chown -R www-data:www-data /var/www/mywebsite` |
| 404 Not Found | Wrong `root` path in config | Check the `root` directive in your Nginx config |
| 502 Bad Gateway | Nginx can't reach backend | Only applies to reverse proxy setups, not static sites |
| Nginx won't start | Config syntax error | Run `sudo nginx -t` to find the error |

## 🎓 Bonus: Add a Custom Domain (Optional)

If you have a domain name, you can point it to your server:

### Step 1: Update DNS Records

Go to your domain registrar (GoDaddy, Namecheap, Cloudflare, etc.) and add an **A Record**:

```
Type: A
Name: @ (or your subdomain)
Value: your-server-public-ip
TTL: 300
```

### Step 2: Update Nginx Config

```bash
sudo nano /etc/nginx/sites-available/mywebsite
```

Change the `server_name` line:

```nginx
server_name yourdomain.com www.yourdomain.com;
```

### Step 3: Reload Nginx

```bash
sudo nginx -t
sudo systemctl reload nginx
```

### Step 4: Add HTTPS with Let's Encrypt (Free SSL!)

```bash
# Install Certbot (Let's Encrypt client)
sudo apt install certbot python3-certbot-nginx -y

# Get an SSL certificate (Certbot will automatically configure Nginx)
sudo certbot --nginx -d yourdomain.com -d www.yourdomain.com

# Certbot will ask you a few questions:
# - Enter your email
# - Agree to terms
# - Choose whether to redirect HTTP to HTTPS (choose YES!)
```

Certbot will automatically:
1. Get a free SSL certificate
2. Modify your Nginx config to use HTTPS
3. Set up auto-renewal (certificates expire every 90 days, but Certbot renews them automatically)
