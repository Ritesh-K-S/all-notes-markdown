## 🏗️ Architecture — What We're Building

```
┌──────────────────────────────────────────────────────────────────────┐
│                        Kubernetes Cluster                            │
│                                                                      │
│  ┌─────────────┐    scrapes     ┌──────────────────┐                │
│  │ Node Exporter│◄──────────────│                  │                │
│  │ (DaemonSet)  │               │                  │                │
│  └─────────────┘                │                  │                │
│                                 │   Prometheus     │   alerts       │
│  ┌─────────────────┐  scrapes   │   Server         │──────────┐    │
│  │ kube-state-     │◄───────────│                  │          │    │
│  │ metrics         │            │                  │          ▼    │
│  └─────────────────┘            │                  │  ┌────────────┐│
│                                 └────────┬─────────┘  │ Alert      ││
│  ┌─────────────┐    scrapes              │            │ Manager    ││
│  │ Your App    │◄────────────────────────┘            │            ││
│  │ (metrics    │                                      └──┬───┬─────┘│
│  │  endpoint)  │            ┌──────────────┐             │   │     │
│  └─────────────┘            │              │             │   │     │
│                       ┌─────│   Grafana    │          Slack Email   │
│                       │     │  Dashboards  │                       │
│                       │     │              │                       │
│                       │     └──────────────┘                       │
│                       │           queries                          │
│                       └──────► Prometheus                          │
└──────────────────────────────────────────────────────────────────────┘
```

**In simple words:**

1. **Prometheus** goes around asking services: "Hey, how are your metrics?" (this is called **scraping**)
2. **Node Exporter** sits on every node and reports CPU, memory, disk usage
3. **kube-state-metrics** reports Kubernetes-specific info (pod status, deployments, replicas)
4. **Grafana** makes all this data look beautiful with charts and dashboards
5. **AlertManager** sends you a Slack message or email when something goes wrong

## 🚀 Step 3 — Deploy Prometheus (The Metrics Collector)

We'll do this **two ways** — first manually (to understand everything), then with Helm (the production way).

### Method A: Manual Deployment (Learn the internals)

#### 3a. Prometheus RBAC (ClusterRole & ServiceAccount)

Prometheus needs permission to discover and scrape targets across the cluster:

```yaml
# prometheus-rbac.yaml
---
apiVersion: v1
kind: ServiceAccount
metadata:
  name: prometheus
  namespace: monitoring

---
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: prometheus
rules:
  - apiGroups: [""]
    resources:
      - nodes
      - nodes/proxy
      - nodes/metrics
      - services
      - endpoints
      - pods
    verbs: ["get", "list", "watch"]
  - apiGroups: ["extensions", "networking.k8s.io"]
    resources:
      - ingresses
    verbs: ["get", "list", "watch"]
  - nonResourceURLs: ["/metrics"]
    verbs: ["get"]

---
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: prometheus
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: prometheus
subjects:
  - kind: ServiceAccount
    name: prometheus
    namespace: monitoring
```

**What's happening here?**

| Resource | Why Prometheus Needs It |
|----------|----------------------|
| `nodes` | To discover all nodes in the cluster |
| `pods` | To find pods that expose metrics |
| `services/endpoints` | To discover services and their endpoints |
| `/metrics` | To actually scrape the metrics endpoints |

```bash
kubectl apply -f prometheus-rbac.yaml
```

#### 3b. Prometheus ConfigMap (Configuration)

This is the brain of Prometheus — it tells it **what to scrape** and **how often**:

```yaml
# prometheus-configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: prometheus-config
  namespace: monitoring
data:
  prometheus.yml: |
    # ============================================
    # Global Configuration
    # ============================================
    global:
      scrape_interval: 15s        # How often to scrape targets
      evaluation_interval: 15s    # How often to evaluate alert rules
      scrape_timeout: 10s         # Timeout for each scrape

    # ============================================
    # Alerting Configuration
    # ============================================
    alerting:
      alertmanagers:
        - static_configs:
            - targets:
                - alertmanager:9093    # AlertManager service

    # ============================================
    # Alert Rules
    # ============================================
    rule_files:
      - /etc/prometheus/rules/*.yml

    # ============================================
    # Scrape Configurations
    # ============================================
    scrape_configs:

      # --- Job 1: Scrape Prometheus itself ---
      - job_name: 'prometheus'
        static_configs:
          - targets: ['localhost:9090']

      # --- Job 2: Scrape Kubernetes nodes ---
      - job_name: 'kubernetes-nodes'
        scheme: https
        tls_config:
          ca_file: /var/run/secrets/kubernetes.io/serviceaccount/ca.crt
        bearer_token_file: /var/run/secrets/kubernetes.io/serviceaccount/token
        kubernetes_sd_configs:
          - role: node
        relabel_configs:
          - action: labelmap
            regex: __meta_kubernetes_node_label_(.+)
          - target_label: __address__
            replacement: kubernetes.default.svc:443
          - source_labels: [__meta_kubernetes_node_name]
            regex: (.+)
            target_label: __metrics_path__
            replacement: /api/v1/nodes/${1}/proxy/metrics

      # --- Job 3: Scrape Node Exporter ---
      - job_name: 'node-exporter'
        kubernetes_sd_configs:
          - role: pod
        relabel_configs:
          - source_labels: [__meta_kubernetes_pod_label_app]
            regex: node-exporter
            action: keep
          - source_labels: [__meta_kubernetes_pod_ip]
            target_label: __address__
            replacement: ${1}:9100

      # --- Job 4: Scrape kube-state-metrics ---
      - job_name: 'kube-state-metrics'
        static_configs:
          - targets: ['kube-state-metrics:8080']

      # --- Job 5: Scrape pods with prometheus annotations ---
      - job_name: 'kubernetes-pods'
        kubernetes_sd_configs:
          - role: pod
        relabel_configs:
          # Only scrape pods with annotation: prometheus.io/scrape: "true"
          - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_scrape]
            action: keep
            regex: true
          # Use custom path if specified
          - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_path]
            action: replace
            target_label: __metrics_path__
            regex: (.+)
          # Use custom port if specified
          - source_labels: [__address__, __meta_kubernetes_pod_annotation_prometheus_io_port]
            action: replace
            regex: ([^:]+)(?::\d+)?;(\d+)
            replacement: $1:$2
            target_label: __address__
          # Add pod labels
          - action: labelmap
            regex: __meta_kubernetes_pod_label_(.+)
          - source_labels: [__meta_kubernetes_namespace]
            action: replace
            target_label: kubernetes_namespace
          - source_labels: [__meta_kubernetes_pod_name]
            action: replace
            target_label: kubernetes_pod_name

  # ============================================
  # Alert Rules
  # ============================================
  alert-rules.yml: |
    groups:
      # --- Infrastructure Alerts ---
      - name: infrastructure
        rules:
          - alert: HighCPUUsage
            expr: 100 - (avg by(instance) (irate(node_cpu_seconds_total{mode="idle"}[5m])) * 100) > 80
            for: 5m
            labels:
              severity: warning
            annotations:
              summary: "High CPU usage on {{ $labels.instance }}"
              description: "CPU usage is above 80% for more than 5 minutes. Current: {{ $value }}%"

          - alert: HighMemoryUsage
            expr: (1 - (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes)) * 100 > 85
            for: 5m
            labels:
              severity: warning
            annotations:
              summary: "High memory usage on {{ $labels.instance }}"
              description: "Memory usage is above 85%. Current: {{ $value }}%"

          - alert: DiskSpaceRunningLow
            expr: (1 - (node_filesystem_avail_bytes{mountpoint="/"} / node_filesystem_size_bytes{mountpoint="/"})) * 100 > 85
            for: 10m
            labels:
              severity: critical
            annotations:
              summary: "Disk space critical on {{ $labels.instance }}"
              description: "Disk usage on / is above 85%. Current: {{ $value }}%"

          - alert: NodeDown
            expr: up{job="node-exporter"} == 0
            for: 2m
            labels:
              severity: critical
            annotations:
              summary: "Node {{ $labels.instance }} is down"
              description: "Node exporter is not reachable for 2 minutes"

      # --- Kubernetes Alerts ---
      - name: kubernetes
        rules:
          - alert: PodCrashLooping
            expr: rate(kube_pod_container_status_restarts_total[15m]) * 60 * 5 > 0
            for: 5m
            labels:
              severity: critical
            annotations:
              summary: "Pod {{ $labels.namespace }}/{{ $labels.pod }} is crash looping"
              description: "Pod is restarting frequently"

          - alert: PodNotReady
            expr: kube_pod_status_ready{condition="true"} == 0
            for: 10m
            labels:
              severity: warning
            annotations:
              summary: "Pod {{ $labels.namespace }}/{{ $labels.pod }} not ready"
              description: "Pod has been in not-ready state for 10 minutes"

          - alert: DeploymentReplicasMismatch
            expr: kube_deployment_spec_replicas != kube_deployment_status_ready_replicas
            for: 10m
            labels:
              severity: warning
            annotations:
              summary: "Deployment {{ $labels.namespace }}/{{ $labels.deployment }} replica mismatch"
              description: "Desired replicas don't match available replicas"

          - alert: HighPodMemoryUsage
            expr: container_memory_usage_bytes{container!=""} / container_spec_memory_limit_bytes{container!=""} * 100 > 90
            for: 5m
            labels:
              severity: warning
            annotations:
              summary: "Pod {{ $labels.pod }} memory usage high"
              description: "Container {{ $labels.container }} is using >90% of its memory limit"
```

**Let me explain the scrape configs in plain English:**

| Job Name | What It Scrapes | How It Finds Targets |
|----------|----------------|---------------------|
| `prometheus` | Prometheus itself | Static IP (localhost:9090) |
| `kubernetes-nodes` | Kubelet metrics on each node | Auto-discovers nodes via K8s API |
| `node-exporter` | CPU, memory, disk on nodes | Finds pods labeled `app: node-exporter` |
| `kube-state-metrics` | K8s object metrics | Static service name |
| `kubernetes-pods` | Any pod with prometheus annotations | Looks for `prometheus.io/scrape: "true"` annotation |

```bash
kubectl apply -f prometheus-configmap.yaml
```

Prometheus needs persistent storage so metrics survive pod restarts:

```yaml

  name: prometheus-data

  accessModes:

  # storageClassName: standard  # Uncomment for GKE
```

#### 3d. Prometheus Deployment

```yaml
# prometheus-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: prometheus
  namespace: monitoring
  labels:
    app: prometheus
spec:
  replicas: 1
  selector:
    matchLabels:
      app: prometheus
  template:
    metadata:
      labels:
        app: prometheus
    spec:
      serviceAccountName: prometheus
      # Init container to fix permissions on the data directory
      initContainers:
        - name: fix-permissions
          image: busybox:1.36
          command: ['sh', '-c', 'chown -R 65534:65534 /prometheus']
          volumeMounts:
            - name: prometheus-data
              mountPath: /prometheus
      containers:
        - name: prometheus
          image: prom/prometheus:v2.51.0
          args:
            - '--config.file=/etc/prometheus/prometheus.yml'
            - '--storage.tsdb.path=/prometheus'
            - '--storage.tsdb.retention.time=15d'     # Keep 15 days of data
            - '--storage.tsdb.retention.size=8GB'      # Max 8GB storage
            - '--web.enable-lifecycle'                  # Allow config reload via API
            - '--web.console.libraries=/etc/prometheus/console_libraries'
            - '--web.console.templates=/etc/prometheus/consoles'
          ports:
            - containerPort: 9090
              name: web
          resources:
            requests:
              cpu: 500m
              memory: 512Mi
            limits:
              cpu: 1000m
              memory: 1Gi
          volumeMounts:
            - name: prometheus-config
              mountPath: /etc/prometheus/prometheus.yml
              subPath: prometheus.yml
            - name: prometheus-config
              mountPath: /etc/prometheus/rules/alert-rules.yml
              subPath: alert-rules.yml
            - name: prometheus-data
              mountPath: /prometheus
          # Health checks
          livenessProbe:
            httpGet:
              path: /-/healthy
              port: 9090
            initialDelaySeconds: 30
            periodSeconds: 15
          readinessProbe:
            httpGet:
              path: /-/ready
              port: 9090
            initialDelaySeconds: 5
            periodSeconds: 10
      volumes:
        - name: prometheus-config
          configMap:
            name: prometheus-config
        - name: prometheus-data
          persistentVolumeClaim:
            claimName: prometheus-data
```

**What each arg means:**

| Argument | Explanation |
|----------|-------------|
| `--storage.tsdb.retention.time=15d` | Keep metrics data for 15 days, then auto-delete |
| `--storage.tsdb.retention.size=8GB` | Don't use more than 8GB of disk space |
| `--web.enable-lifecycle` | Allows reloading config without restarting the pod |

```bash
kubectl apply -f prometheus-deployment.yaml
```

## 🚀 Step 4 — Deploy Node Exporter (System Metrics)

Node Exporter runs on **every node** and reports hardware/OS metrics (CPU, RAM, disk, network).

```yaml
# node-exporter-daemonset.yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: node-exporter
  namespace: monitoring
  labels:
    app: node-exporter
spec:
  selector:
    matchLabels:
      app: node-exporter
  template:
    metadata:
      labels:
        app: node-exporter
      annotations:
        prometheus.io/scrape: "true"
        prometheus.io/port: "9100"
    spec:
      hostPID: true           # Access host process info
      hostNetwork: true       # Access host network metrics
      containers:
        - name: node-exporter
          image: prom/node-exporter:v1.7.0
          args:
            - '--path.procfs=/host/proc'
            - '--path.sysfs=/host/sys'
            - '--path.rootfs=/host/root'
            - '--collector.filesystem.mount-points-exclude=^/(sys|proc|dev|host|etc)($$|/)'
          ports:
            - containerPort: 9100
              hostPort: 9100
              name: metrics
          resources:
            requests:
              cpu: 100m
              memory: 64Mi
            limits:
              cpu: 200m
              memory: 128Mi
          volumeMounts:
            - name: proc
              mountPath: /host/proc
              readOnly: true
            - name: sys
              mountPath: /host/sys
              readOnly: true
            - name: root
              mountPath: /host/root
              readOnly: true
              mountPropagation: HostToContainer
      tolerations:
        - effect: NoSchedule
          operator: Exists    # Run on ALL nodes, even tainted ones
      volumes:
        - name: proc
          hostPath:
            path: /proc
        - name: sys
          hostPath:
            path: /sys
        - name: root
          hostPath:
            path: /
```

**Why a DaemonSet?**

> A DaemonSet guarantees that **one copy of Node Exporter runs on every node** in the cluster. When you add a new node, it automatically gets a Node Exporter pod. Perfect for collecting per-node metrics!

```bash
kubectl apply -f node-exporter-daemonset.yaml
```

Verify:

```bash
kubectl get pods -n monitoring -l app=node-exporter -o wide
```

```

node-exporter-abc12   1/1     Running   worker-1
node-exporter-def34   1/1     Running   worker-2
node-exporter-ghi56   1/1     Running   control-plane
```

One pod per node — exactly what we want! ✅

## 🚀 Step 6 — Deploy AlertManager (Alert Routing & Notifications)

AlertManager receives alerts from Prometheus and routes them to Slack, email, PagerDuty, etc.

```yaml
# alertmanager-configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: alertmanager-config
  namespace: monitoring
data:
  alertmanager.yml: |
    # ============================================
    # Global Configuration
    # ============================================
    global:
      resolve_timeout: 5m

      # SMTP Settings (for email alerts)
      smtp_smarthost: 'smtp.gmail.com:587'
      smtp_from: 'alerts@yourdomain.com'
      smtp_auth_username: 'alerts@yourdomain.com'
      smtp_auth_password: 'your-app-password'     # Use App Password, not regular password!
      smtp_require_tls: true

    # ============================================
    # Alert Templates
    # ============================================
    templates:
      - '/etc/alertmanager/templates/*.tmpl'

    # ============================================
    # Route Tree — How alerts are routed
    # ============================================
    route:
      # Default receiver for all alerts
      receiver: 'slack-critical'

      # Group alerts by these labels
      group_by: ['alertname', 'namespace', 'severity']

      # Wait 30s to group alerts together before sending
      group_wait: 30s

      # Wait 5 min before sending updates for the same group
      group_interval: 5m

      # Wait 4 hours before re-sending the same alert
      repeat_interval: 4h

      # Child routes — more specific routing
      routes:
        # Critical alerts → Slack + Email
        - match:
            severity: critical
          receiver: 'slack-critical'
          continue: true     # Also send to email

        - match:
            severity: critical
          receiver: 'email-critical'

        # Warning alerts → Slack only
        - match:
            severity: warning
          receiver: 'slack-warning'

    # ============================================
    # Receivers — Where alerts go
    # ============================================
    receivers:
      # Slack — Critical channel
      - name: 'slack-critical'
        slack_configs:
          - api_url: 'https://hooks.slack.com/services/YOUR/SLACK/WEBHOOK'
            channel: '#alerts-critical'
            title: '🚨 [{{ .Status | toUpper }}] {{ .CommonLabels.alertname }}'
            text: >-
              *Alert:* {{ .CommonLabels.alertname }}
              *Severity:* {{ .CommonLabels.severity }}
              *Namespace:* {{ .CommonLabels.namespace }}
              *Description:* {{ .CommonAnnotations.description }}
            send_resolved: true    # Also notify when alert resolves

      # Slack — Warning channel
      - name: 'slack-warning'
        slack_configs:
          - api_url: 'https://hooks.slack.com/services/YOUR/SLACK/WEBHOOK'
            channel: '#alerts-warning'
            title: '⚠️ [{{ .Status | toUpper }}] {{ .CommonLabels.alertname }}'
            text: >-
              *Alert:* {{ .CommonLabels.alertname }}
              *Description:* {{ .CommonAnnotations.description }}
            send_resolved: true

      # Email — Critical only
      - name: 'email-critical'
        email_configs:
          - to: 'oncall-team@yourdomain.com'
            send_resolved: true

    # ============================================
    # Inhibition Rules — Suppress noisy alerts
    # ============================================
    inhibit_rules:
      # If a critical alert fires, suppress the warning for the same alertname
      - source_match:
          severity: 'critical'
        target_match:
          severity: 'warning'
        equal: ['alertname', 'namespace']
```

**Let me explain the routing in plain English:**

```
Alert fires in Prometheus
        ↓
    What severity?
        ↓
  ┌─────┴─────┐
  │           │
critical   warning
  │           │
  ├→ Slack    └→ Slack
  │  #alerts-     #alerts-
  │  critical     warning
  │
  └→ Email
     oncall-team
```

### Create a Custom Dashboard (Your Own!)

Let's create a simple dashboard using PromQL:

#### Panel 1: CPU Usage Per Node

```promql
100 - (avg by (instance) (irate(node_cpu_seconds_total{mode="idle"}[5m])) * 100)
```

**What this query does:**
1. `node_cpu_seconds_total{mode="idle"}` — get idle CPU time
2. `irate(...[5m])` — calculate the rate of change over 5 minutes
3. `avg by (instance)` — average across all CPU cores per node
4. `100 - ...` — convert idle % to usage % (100 - idle = used)

#### Panel 2: Memory Usage Per Node

```promql
(1 - (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes)) * 100
```

**What this query does:**
1. `node_memory_MemAvailable_bytes` — how much memory is available
2. `node_memory_MemTotal_bytes` — total memory
3. `available / total` — what percentage is available
4. `1 - ...` — flip to get usage percentage
5. `* 100` — convert to percentage

#### Panel 3: Pod Count by Namespace

```promql
count by (namespace) (kube_pod_info)
```

#### Panel 4: Pod Restart Count (Last 1 Hour)

```promql
increase(kube_pod_container_status_restarts_total[1h]) > 0
```

#### Panel 5: Disk Usage

```promql
(1 - (node_filesystem_avail_bytes{mountpoint="/"} / node_filesystem_size_bytes{mountpoint="/"})) * 100
```

#### Panel 6: Network Traffic (Received)

```promql
rate(node_network_receive_bytes_total{device!="lo"}[5m]) * 8
```

> This gives you **bits per second** received on each network interface (excluding loopback).

## 🚀 Step 9 — Deploy a Sample App with Custom Metrics

Let's deploy a sample app that exposes its own Prometheus metrics:

```yaml
# sample-app.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: sample-app
  namespace: default
  labels:
    app: sample-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: sample-app
  template:
    metadata:
      labels:
        app: sample-app
      annotations:
        prometheus.io/scrape: "true"     # Tell Prometheus to scrape this pod
        prometheus.io/port: "8080"        # On this port
        prometheus.io/path: "/metrics"    # At this path
    spec:
      containers:
        - name: sample-app
          image: quay.io/brancz/prometheus-example-app:v0.5.0
          ports:
            - containerPort: 8080
          resources:
            requests:
              cpu: 50m
              memory: 64Mi
            limits:
              cpu: 100m
              memory: 128Mi

---
apiVersion: v1
kind: Service
metadata:
  name: sample-app
  namespace: default
spec:
  ports:
    - port: 8080
      targetPort: 8080
  selector:
    app: sample-app
```

```bash
kubectl apply -f sample-app.yaml
```

**The magic of annotations:**
- `prometheus.io/scrape: "true"` → Prometheus auto-discovers this pod
- `prometheus.io/port: "8080"` → Scrape on port 8080
- `prometheus.io/path: "/metrics"` → Hit the `/metrics` endpoint

No need to manually add scrape targets — Prometheus finds it automatically!

#### Verify Prometheus is scraping the app

1. Go to **Prometheus UI** → **Status** → **Targets**
2. You should see `kubernetes-pods` with your sample-app listed
3. Status should be **UP** ✅

## 🚀 Step 11 — Essential PromQL Queries Cheat Sheet

Here are the most useful PromQL queries for day-to-day monitoring:

### CPU Queries

```promql
# Overall cluster CPU usage (%)
100 - (avg(irate(node_cpu_seconds_total{mode="idle"}[5m])) * 100)

# CPU usage per node
100 - (avg by(instance) (irate(node_cpu_seconds_total{mode="idle"}[5m])) * 100)

# Top 5 pods by CPU usage
topk(5, sum by(pod) (rate(container_cpu_usage_seconds_total{container!=""}[5m])))
```

### Memory Queries

```promql
# Cluster memory usage (%)
(1 - sum(node_memory_MemAvailable_bytes) / sum(node_memory_MemTotal_bytes)) * 100

# Memory usage per pod (MB)
sum by(pod) (container_memory_usage_bytes{container!=""}) / 1024 / 1024

# Pods near their memory limit (>80%)
container_memory_usage_bytes{container!=""} / container_spec_memory_limit_bytes{container!=""} * 100 > 80
```

### Disk Queries

```promql
# Disk usage per node (%)
(1 - node_filesystem_avail_bytes{mountpoint="/"} / node_filesystem_size_bytes{mountpoint="/"}) * 100

# Disk I/O (reads per second)
rate(node_disk_reads_completed_total[5m])
```

### Kubernetes Queries

```promql
# Number of pods by status
count by(phase) (kube_pod_status_phase)

# Pods not running
kube_pod_status_phase{phase!="Running"} == 1

# Deployments with unavailable replicas
kube_deployment_status_replicas_unavailable > 0

# Container restarts in last hour
increase(kube_pod_container_status_restarts_total[1h]) > 0
```

### Network Queries

```promql
# Network traffic received (bytes/sec per node)
rate(node_network_receive_bytes_total{device!="lo"}[5m])

# Network traffic sent
rate(node_network_transmit_bytes_total{device!="lo"}[5m])
```

## 🚨 Common Issues & Troubleshooting

| Problem | Solution |
|---------|----------|
| Prometheus pod `Pending` | Check PVC — is storage provisioner available? Use `emptyDir` for testing |
| Targets show `DOWN` | Check network policies, verify service names, check port numbers |
| Grafana "No data" on dashboards | Verify Prometheus data source URL is correct (`http://prometheus:9090`) |
| AlertManager not sending alerts | Check webhook URLs, verify SMTP credentials, check AlertManager logs |
| High memory usage by Prometheus | Reduce retention, add resource limits, use recording rules |
| `permission denied` on PVC | Use init container to fix ownership (shown in deployment above) |
