## 🧠 Quick Recap — What is GitOps?

Before we get our hands dirty, let's understand what GitOps actually is in super simple terms:

**Traditional Deployment:**
```
Developer → writes code → builds image → runs kubectl apply → deploys to cluster
```
The problem? Anyone with `kubectl` access can change things. No audit trail. No consistency.

**GitOps Deployment:**
```
Developer → writes code → pushes to Git → ArgoCD detects change → auto-deploys to cluster
```
Everything goes through Git. Git IS your deployment tool. If it's not in Git, it doesn't exist in your cluster.

**Think of it like this:**
- Git = your **desired state** (what you WANT your cluster to look like)
- Kubernetes = your **actual state** (what your cluster actually looks like)
- ArgoCD = the **bridge** that makes actual state match desired state. Always. Automatically.


**Why change the password?**
- The default password is stored as a Kubernetes secret — anyone with cluster access could read it

## 📁 Part 3 — Setting Up the Git Repository Structure

This is the most important part. Your Git repo IS your deployment configuration.

### Step 3.1 — Create the GitOps Repository

```bash
# Create a new directory for our GitOps repo
mkdir gitops-demo && cd gitops-demo

```

### Step 3.2 — Design the Repository Structure

```
gitops-demo/
├── apps/                          # ArgoCD Application definitions
│   ├── dev/
│   │   └── my-app.yaml           # App definition for dev environment
│   ├── staging/
│   │   └── my-app.yaml           # App definition for staging
│   └── production/
│       └── my-app.yaml           # App definition for production
│
├── manifests/                     # Kubernetes manifests (the actual config)
│   ├── base/                     # Base manifests (shared across environments)
│   │   ├── deployment.yaml
│   │   ├── service.yaml
│   │   ├── configmap.yaml
│   │   └── kustomization.yaml
│   │
│   └── overlays/                 # Environment-specific overrides
│       ├── dev/
│       │   ├── kustomization.yaml
│       │   └── patch-replicas.yaml
│       ├── staging/
│       │   ├── kustomization.yaml
│       │   └── patch-replicas.yaml
│       └── production/
│           ├── kustomization.yaml
│           ├── patch-replicas.yaml
│           └── patch-resources.yaml
│
├── app-of-apps/                  # App of Apps pattern
│   └── root-app.yaml
│
└── README.md
```

**Why this structure?**
- **base/** — contains the "default" configuration that all environments share
- **overlays/** — each environment can override specific things (like replicas, resource limits, image tags)
- **apps/** — ArgoCD Application CRDs that tell ArgoCD what to watch and where to deploy
- **Kustomize** is used to layer environment-specific changes on top of the base

### Step 3.3 — Create the Base Kubernetes Manifests

**Base Deployment — `manifests/base/deployment.yaml`:**

```yaml
# manifests/base/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
  labels:
    app: my-app
spec:
  replicas: 1                    # Default: 1 replica (overridden per environment)
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
        - name: my-app
          image: nginx:1.25       # We'll use nginx as our sample app
          ports:
            - containerPort: 80
          resources:
            requests:
              cpu: "100m"         # Request 0.1 CPU cores
              memory: "128Mi"     # Request 128 MB RAM
            limits:
              cpu: "250m"         # Max 0.25 CPU cores
              memory: "256Mi"     # Max 256 MB RAM
          livenessProbe:          # Is the container alive?
            httpGet:
              path: /
              port: 80
            initialDelaySeconds: 5
            periodSeconds: 10
          readinessProbe:         # Is the container ready to serve traffic?
            httpGet:
              path: /
              port: 80
            initialDelaySeconds: 3
            periodSeconds: 5
```

**Let me explain every part of this deployment:**

| Field | What It Does |
|-------|-------------|
| `replicas: 1` | Run 1 copy of our app (base default) |
| `selector.matchLabels` | How the Deployment finds its Pods — by label `app: my-app` |
| `image: nginx:1.25` | The container image to use |
| `resources.requests` | Minimum resources the Pod needs to be scheduled |
| `resources.limits` | Maximum resources the Pod can consume |
| `livenessProbe` | K8s checks if the container is alive; restarts if it fails |
| `readinessProbe` | K8s checks if the container can serve traffic; removes from Service if it fails |

**Base Service — `manifests/base/service.yaml`:**

```yaml
# manifests/base/service.yaml
apiVersion: v1
kind: Service
metadata:
  name: my-app
  labels:
    app: my-app
spec:
  type: ClusterIP               # Internal-only access (default)
  selector:
    app: my-app                 # Route traffic to pods with label app=my-app
  ports:
    - port: 80                  # Service listens on port 80
      targetPort: 80            # Forward to container port 80
      protocol: TCP
```

**What does this Service do?**
- Creates a stable internal DNS name (`my-app.dev.svc.cluster.local`)
- Load balances traffic across all Pods with the label `app: my-app`
- `ClusterIP` means it's only accessible within the cluster (not from the internet)

**Base ConfigMap — `manifests/base/configmap.yaml`:**

```yaml
# manifests/base/configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: my-app-config
data:
  APP_ENV: "base"               # Will be overridden per environment
  LOG_LEVEL: "info"
  APP_NAME: "GitOps Demo App"
```

**What is a ConfigMap?**
- A way to store non-sensitive configuration data (environment variables, config files)
- Your app can read these values as environment variables or mounted files
- Each environment will override `APP_ENV` to its own value (dev, staging, production)

**Base Kustomization — `manifests/base/kustomization.yaml`:**

```yaml
# manifests/base/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

resources:
  - deployment.yaml
  - service.yaml
  - configmap.yaml
```

**What is Kustomization?**
- It's a "bill of materials" — lists all the resources that make up your app
- Kustomize (built into kubectl) uses this file to know which manifests to include
- Overlays can reference this base and add their own patches on top

### Step 3.4 — Create Environment-Specific Overlays

**Dev Overlay — `manifests/overlays/dev/kustomization.yaml`:**

```yaml
# manifests/overlays/dev/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

namespace: dev                   # Deploy everything into the 'dev' namespace

resources:
  - ../../base                  # Include all base manifests

patches:
  - path: patch-replicas.yaml   # Override the replica count

configMapGenerator:
  - name: my-app-config
    behavior: merge             # Merge with base ConfigMap
    literals:
      - APP_ENV=development     # Override APP_ENV for dev
      - LOG_LEVEL=debug         # More verbose logging in dev
```

**Dev Replica Patch — `manifests/overlays/dev/patch-replicas.yaml`:**

```yaml
# manifests/overlays/dev/patch-replicas.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 1                   # Dev only needs 1 replica
```

**Staging Overlay — `manifests/overlays/staging/kustomization.yaml`:**

```yaml
# manifests/overlays/staging/kustomization.yaml

kind: Kustomization

namespace: staging

resources:
  - ../../base

patches:
  - path: patch-replicas.yaml

configMapGenerator:
  - name: my-app-config
    behavior: merge
    literals:
      - APP_ENV=staging
      - LOG_LEVEL=info
```

**Staging Replica Patch — `manifests/overlays/staging/patch-replicas.yaml`:**

```yaml

apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 2                   # Staging gets 2 replicas for testing
```

**Production Overlay — `manifests/overlays/production/kustomization.yaml`:**

```yaml
# manifests/overlays/production/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

namespace: production

resources:
  - ../../base

patches:
  - path: patch-replicas.yaml
  - path: patch-resources.yaml   # Production gets more resources

configMapGenerator:
  - name: my-app-config
    behavior: merge
    literals:
      - APP_ENV=production
      - LOG_LEVEL=warn           # Less verbose in production
```

**Production Replica Patch — `manifests/overlays/production/patch-replicas.yaml`:**

```yaml
# manifests/overlays/production/patch-replicas.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3                   # Production gets 3 replicas for high availability
```

**Production Resource Patch — `manifests/overlays/production/patch-resources.yaml`:**

```yaml
# manifests/overlays/production/patch-resources.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  template:
    spec:
      containers:
        - name: my-app
          resources:
            requests:
              cpu: "250m"        # More CPU for production
              memory: "256Mi"
            limits:
              cpu: "500m"
              memory: "512Mi"    # More memory for production
```

## ⚙️ Part 4 — Creating ArgoCD Applications

Now we tell ArgoCD what to watch and where to deploy.

### Step 4.1 — Create ArgoCD Application for Dev

**`apps/dev/my-app.yaml`:**

```yaml
# apps/dev/my-app.yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: my-app-dev                    # Name of this ArgoCD Application
  namespace: argocd                   # ArgoCD Applications live in argocd namespace
  labels:
    environment: dev
  finalizers:
    - resources-finalizer.argocd.argoproj.io   # Clean up K8s resources when app is deleted
spec:
  project: default                    # ArgoCD project (default is fine for now)
  
  source:
    repoURL: https://github.com/YOUR_USERNAME/gitops-demo.git   # Your Git repo URL
    targetRevision: main              # Which branch to watch
    path: manifests/overlays/dev      # Path to the dev overlay
  
  destination:
    server: https://kubernetes.default.svc   # Deploy to the local cluster
    namespace: dev                           # Target namespace
  
  syncPolicy:
    automated:                        # Enable automatic syncing
      prune: true                     # Delete resources that are no longer in Git
      selfHeal: true                  # If someone manually changes something, revert it
      allowEmpty: false               # Don't sync if no resources are found
    syncOptions:
      - CreateNamespace=true          # Create the namespace if it doesn't exist
      - PruneLast=true                # Delete removed resources after new ones are healthy
    retry:
      limit: 5                        # Retry sync up to 5 times on failure
      backoff:
        duration: 5s
        factor: 2
        maxDuration: 3m
```

**Let me explain the key fields:**

| Field | What It Does | Why It Matters |
|-------|-------------|----------------|
| `source.repoURL` | The Git repository ArgoCD watches | This is your single source of truth |
| `source.targetRevision` | Which branch/tag to follow | Usually `main` for dev, specific tags for prod |
| `source.path` | Which folder in the repo contains the manifests | Points to the Kustomize overlay |
| `destination.server` | Which K8s cluster to deploy to | Local cluster or remote |
| `syncPolicy.automated` | Auto-sync when Git changes | No manual `kubectl apply` needed! |
| `syncPolicy.automated.prune` | Delete resources removed from Git | Keeps cluster clean |
| `syncPolicy.automated.selfHeal` | Revert manual cluster changes | Prevents config drift |

### Step 4.2 — Create ArgoCD Application for Staging

**`apps/staging/my-app.yaml`:**

```yaml
# apps/staging/my-app.yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: my-app-staging
  namespace: argocd
  labels:
    environment: staging
  finalizers:
    - resources-finalizer.argocd.argoproj.io
spec:
  project: default
  
  source:
    repoURL: https://github.com/YOUR_USERNAME/gitops-demo.git
    targetRevision: main
    path: manifests/overlays/staging
  
  destination:
    server: https://kubernetes.default.svc
    namespace: staging
  
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
      - CreateNamespace=true
    retry:
      limit: 5
      backoff:
        duration: 5s
        factor: 2
        maxDuration: 3m
```

### Step 4.3 — Create ArgoCD Application for Production

**`apps/production/my-app.yaml`:**

```yaml
# apps/production/my-app.yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: my-app-production
  namespace: argocd
  labels:
    environment: production
  finalizers:
    - resources-finalizer.argocd.argoproj.io
spec:
  project: default
  
  source:
    repoURL: https://github.com/YOUR_USERNAME/gitops-demo.git
    targetRevision: main              # In real production, you might use a tag like v1.2.3
    path: manifests/overlays/production
  
  destination:
    server: https://kubernetes.default.svc
    namespace: production
  
  syncPolicy:
    # NOTE: For production, we intentionally DON'T enable automated sync
    # We want manual approval before deploying to production!
    syncOptions:
      - CreateNamespace=true
    retry:
      limit: 3
      backoff:
        duration: 10s
        factor: 2
        maxDuration: 5m
```

**🔑 Notice something important?** Production does NOT have `automated` sync enabled. In real-world setups:
- **Dev & Staging** auto-sync (push to Git → auto-deploy)
- **Production** requires manual sync (push to Git → review in ArgoCD UI → click Sync)

This gives you a safety net before deploying to real users.

## 🔄 Part 6 — The GitOps Workflow in Action

This is where the magic happens. Let's see the full lifecycle.

### Step 6.1 — Making a Change (Update Image Version)

Let's update the nginx image from `1.25` to `1.26`:

```bash
# Edit the base deployment
# Change: image: nginx:1.25
# To:     image: nginx:1.26
```

**In `manifests/base/deployment.yaml`, change:**
```yaml
      containers:
        - name: my-app
          image: nginx:1.26       # Updated from 1.25 to 1.26
```

```bash
# Commit and push the change
git add .
git commit -m "Upgrade nginx from 1.25 to 1.26"
git push origin main
```

**What happens now (automatically):**

```
1. GitHub receives the push
2. ArgoCD polls the repo (every 3 minutes by default)
   OR receives a webhook notification (instant)
3. ArgoCD detects the image change in the base deployment
4. Since dev & staging have auto-sync enabled:
   → ArgoCD performs a rolling update in dev namespace
   → ArgoCD performs a rolling update in staging namespace
5. Production shows "OutOfSync" in the UI
6. You review the change and manually sync production
```

```bash
# Watch the rollout happen in real-time
kubectl rollout status deployment/my-app -n dev -w

# Verify the new image is deployed
kubectl get deployment my-app -n dev -o jsonpath='{.spec.template.spec.containers[0].image}'
# Output: nginx:1.26
```

### Step 6.2 — Setting Up Webhooks for Instant Sync

By default, ArgoCD polls Git every 3 minutes. For instant deployments, set up a webhook:

```bash
# In GitHub → Your repo → Settings → Webhooks → Add webhook
# Payload URL: https://your-argocd-url/api/webhook
# Content type: application/json
# Secret: (set a secret and configure it in ArgoCD)
# Events: Just the push event
```

### Step 6.3 — Performing a Rollback

Uh oh! The new nginx version has a bug. Let's rollback:

**Option 1 — Git Revert (RECOMMENDED — the GitOps way):**
```bash
# Find the commit to revert
git log --oneline
# abc1234 Upgrade nginx from 1.25 to 1.26
# def5678 Initial GitOps setup with ArgoCD

# Revert the commit
git revert HEAD
git push origin main

# ArgoCD will automatically deploy the previous version!
```

**Option 2 — ArgoCD History (quick but less clean):**
```bash
# View deployment history
argocd app history my-app-dev

# Rollback to a previous version
argocd app rollback my-app-dev <HISTORY_ID>
```

**Why Git Revert is better:**
- It creates a new commit (audit trail is preserved)
- Every other developer can see what happened and why
- ArgoCD syncs the reverted state to all environments
- The "self-heal" feature won't fight with you

### Step 6.4 — Handling Drift Detection

What if someone manually changes something in the cluster?

```bash
# Naughty developer manually scales dev deployment
kubectl scale deployment my-app -n dev --replicas=5

# Watch what happens...
kubectl get pods -n dev -w
```

**With `selfHeal: true` enabled:**
```
1. ArgoCD detects the cluster state (5 replicas) differs from Git state (1 replica)
2. ArgoCD automatically reverts the change back to 1 replica
3. This happens within seconds!
```

This is the power of GitOps — **Git ALWAYS wins**. No one can make shadow changes to your cluster.

## 🎯 Part 7 — App of Apps Pattern

When you have many applications, creating ArgoCD Application CRDs one by one is tedious. The **App of Apps** pattern solves this.

### Step 7.1 — Create a Root Application

**`app-of-apps/root-app.yaml`:**

```yaml
# app-of-apps/root-app.yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: root-app
  namespace: argocd
  finalizers:
    - resources-finalizer.argocd.argoproj.io
spec:
  project: default
  source:
    repoURL: https://github.com/YOUR_USERNAME/gitops-demo.git
    targetRevision: main
    path: apps                       # Points to the folder containing ALL app definitions
    directory:
      recurse: true                  # Recursively find all YAML files in subdirectories
  destination:
    server: https://kubernetes.default.svc
    namespace: argocd
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
```

**How it works:**

```
┌──────────────────────────────────────────┐
│           ROOT APP (App of Apps)          │
│                                          │
│  Watches: apps/ directory (recursively)  │
│                                          │
│  ┌──────────┐  ┌──────────┐  ┌────────┐ │
│  │ my-app   │  │ my-app   │  │ my-app │ │
│  │ dev      │  │ staging  │  │ prod   │ │
│  └────┬─────┘  └────┬─────┘  └───┬────┘ │
│       │              │            │      │
└───────┼──────────────┼────────────┼──────┘
        │              │            │
        ▼              ▼            ▼
   Dev Cluster   Staging Cluster  Prod Cluster
   (namespace)   (namespace)      (namespace)
```

```bash
# Apply only the root app — it manages everything else!
kubectl apply -f app-of-apps/root-app.yaml

# Now ArgoCD will automatically discover and create all apps
# defined in the apps/ directory
argocd app list
```

**Why use App of Apps?**
- One `kubectl apply` to manage ALL your apps
- Add a new app? Just create a YAML file in the `apps/` folder and push to Git
- Remove an app? Delete the YAML file and push — ArgoCD cleans everything up
- It's GitOps all the way down! 🐢

## 🔒 Part 8 — ArgoCD RBAC and Projects

### Step 8.1 — Create ArgoCD Projects

ArgoCD Projects let you restrict which repos, clusters, and namespaces each team can access:

```yaml
# projects/team-frontend.yaml
apiVersion: argoproj.io/v1alpha1
kind: AppProject
metadata:
  name: frontend-team
  namespace: argocd
spec:
  description: "Frontend team's applications"
  
  # Only allow this Git repo
  sourceRepos:
    - 'https://github.com/YOUR_ORG/frontend-apps.git'
  
  # Only deploy to these namespaces
  destinations:
    - namespace: 'dev'
      server: 'https://kubernetes.default.svc'
    - namespace: 'staging'
      server: 'https://kubernetes.default.svc'
  
  # Only allow these Kubernetes resource types
  clusterResourceWhitelist:
    - group: ''
      kind: Namespace
  
  namespaceResourceWhitelist:
    - group: 'apps'
      kind: Deployment
    - group: ''
      kind: Service
    - group: ''
      kind: ConfigMap
```

**What this project does:**
- The frontend team can ONLY deploy from their own Git repo
- They can ONLY deploy to `dev` and `staging` namespaces (not production!)
- They can ONLY create Deployments, Services, and ConfigMaps (not create RBAC roles or mess with cluster configs)

## 📊 Part 9 — Monitoring ArgoCD

### Step 9.1 — ArgoCD Notifications

ArgoCD can notify you via Slack, email, or webhooks when things happen:

```yaml
# argocd-notifications-cm ConfigMap
apiVersion: v1
kind: ConfigMap
metadata:
  name: argocd-notifications-cm
  namespace: argocd
data:
  service.slack: |
    token: $slack-token
  
  template.app-sync-succeeded: |
    message: |
      ✅ Application {{.app.metadata.name}} has been successfully synced!
      Revision: {{.app.status.sync.revision}}
      Environment: {{.app.metadata.labels.environment}}
  
  template.app-sync-failed: |
    message: |
      ❌ Application {{.app.metadata.name}} sync FAILED!
      Error: {{range .app.status.conditions}}{{.message}}{{end}}
  
  trigger.on-sync-succeeded: |
    - when: app.status.operationState.phase in ['Succeeded']
      send: [app-sync-succeeded]
  
  trigger.on-sync-failed: |
    - when: app.status.operationState.phase in ['Error', 'Failed']
      send: [app-sync-failed]
```

Then annotate your ArgoCD Applications to enable notifications:

```yaml
metadata:
  annotations:
    notifications.argoproj.io/subscribe.on-sync-succeeded.slack: my-channel
    notifications.argoproj.io/subscribe.on-sync-failed.slack: my-channel
```

### Step 9.2 — ArgoCD Metrics for Prometheus

ArgoCD exposes Prometheus metrics out of the box:

## 🏆 Part 11 — Production Best Practices

### What You Should Do in Real Production

| Practice | Why |
|----------|-----|
| **Separate repos for app code and GitOps config** | Keeps CI (build) and CD (deploy) independent |
| **Use Sealed Secrets or External Secrets** | Never store plain secrets in Git |
| **Enable webhook for instant sync** | Don't wait 3 minutes for deployments |
| **Use ArgoCD Projects for multi-team** | Restrict who can deploy where |
| **Tag production releases** | Use `targetRevision: v1.2.3` instead of `main` for prod |
| **Enable notifications** | Know instantly when deploys succeed or fail |
| **Use ApplicationSets** | Generate apps dynamically for many environments/clusters |
| **Health checks on all apps** | ArgoCD monitors app health, not just sync status |
| **Regular ArgoCD upgrades** | Keep ArgoCD itself updated for security patches |

### Common Mistakes to Avoid

| ❌ Don't Do This | ✅ Do This Instead |
|------------------|-------------------|
| `kubectl apply` directly to cluster | Push to Git and let ArgoCD sync |
| Auto-sync production | Manual sync with approval for production |
| Store secrets in Git (plain text) | Use Sealed Secrets, Vault, or External Secrets |
| One giant repo for everything | Separate app code repo and GitOps config repo |
| Ignore ArgoCD health status | Set up alerts for unhealthy/degraded apps |
