## 🧠 Why Dynamic Secrets? — The Problem with Static Secrets

### The Old Way (Static Secrets) 😬

```yaml
# ❌ DON'T DO THIS — hardcoded secrets in YAML
apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
type: Opaque
data:
  username: bXlhcHA=           # "myapp" base64 encoded
  password: c3VwZXJzZWNyZXQ=  # "supersecret" base64 encoded
```

**What's wrong with this?**

| Problem | Impact |
|---------|--------|
| Secrets are in Git | Anyone with repo access can decode them (base64 is NOT encryption!) |
| Same password everywhere | Dev, staging, and prod all use the same credentials |
| Never rotated | The password "supersecret" might be 3 years old |
| No audit trail | Who accessed these credentials? When? You'll never know |
| Shared credentials | 50 pods all use the SAME username/password |
| Revocation is manual | Someone leaves the company? Good luck changing every secret |

### The New Way (Dynamic Secrets with Vault) 🎉

```
Pod starts → authenticates to Vault → gets UNIQUE credentials → uses them → 
credentials auto-expire → Pod gets new ones when needed
```

**What's better?**

| Benefit | How |
|---------|-----|
| **Unique per pod** | Every pod gets its own username/password |
| **Short-lived** | Credentials expire automatically (e.g., after 1 hour) |
| **Auto-rotated** | New credentials are generated before old ones expire |
| **Full audit trail** | Vault logs who accessed what, when |
| **Easy revocation** | Disable one pod's access without affecting others |
| **No secrets in Git** | Secrets are generated at runtime, never stored |

### Step 1.3 — Deploy PostgreSQL

We need a database that Vault will manage credentials for:

```yaml
# postgres.yaml
---
apiVersion: v1
kind: Secret
metadata:
  name: postgres-secret
  namespace: database
type: Opaque
stringData:
  POSTGRES_USER: "admin"            # This is the admin user Vault will use
  POSTGRES_PASSWORD: "adminpassword" # Vault needs admin access to create users
  POSTGRES_DB: "myappdb"
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: postgres
  namespace: database
spec:
  replicas: 1
  selector:
    matchLabels:
      app: postgres
  template:
    metadata:
      labels:
        app: postgres
    spec:
      containers:
        - name: postgres
          image: postgres:15
          ports:
            - containerPort: 5432
          envFrom:
            - secretRef:
                name: postgres-secret
          volumeMounts:
            - mountPath: /var/lib/postgresql/data
              name: postgres-data
      volumes:
        - name: postgres-data
          emptyDir: {}             # Use persistent storage in production!
---
apiVersion: v1
kind: Service
metadata:
  name: postgres
  namespace: database
spec:
  selector:
    app: postgres
  ports:
    - port: 5432
      targetPort: 5432
```

```bash
# Deploy PostgreSQL
kubectl apply -f postgres.yaml

# Wait for it to be ready
kubectl wait --for=condition=ready pod -l app=postgres -n database --timeout=120s

# Verify PostgreSQL is running
kubectl get pods -n database
# Expected:
# NAME                        READY   STATUS    RESTARTS   AGE
# postgres-abc123-def45       1/1     Running   0          30s
```

**What we just did:**
1. Created a PostgreSQL database with an admin user (`admin` / `adminpassword`)
2. Vault will use this admin account to dynamically create/delete database users
3. Our app won't know the admin password — it'll get its own unique credentials from Vault

## ⚙️ Part 3 — Configuring Vault Secrets Engines

Now we configure Vault to generate dynamic database credentials.

### Step 3.1 — Enable the Database Secrets Engine

```bash
# Exec into the Vault pod for all configuration
kubectl exec -it vault-0 -n vault -- /bin/sh

# Enable the database secrets engine
vault secrets enable database
```

**What is a Secrets Engine?**
- Think of it like a "plugin" for Vault
- Each secrets engine handles a different type of secret
- The **database** engine can dynamically create credentials for PostgreSQL, MySQL, MongoDB, etc.
- Other engines include: `kv` (key-value), `pki` (certificates), `transit` (encryption), `aws` (IAM keys)

### Step 3.2 — Configure the PostgreSQL Connection

```bash
# Tell Vault how to connect to PostgreSQL
vault write database/config/myappdb \
  plugin_name="postgresql-database-plugin" \
  allowed_roles="myapp-role" \
  connection_url="postgresql://{{username}}:{{password}}@postgres.database.svc.cluster.local:5432/myappdb?sslmode=disable" \
  username="admin" \
  password="adminpassword"
```

**Let me break this down line by line:**

| Parameter | What It Does |
|-----------|-------------|
| `database/config/myappdb` | Name for this database connection in Vault |
| `plugin_name` | Which database plugin to use (PostgreSQL in our case) |
| `allowed_roles` | Which Vault roles can use this connection |
| `connection_url` | How to connect to PostgreSQL (uses Kubernetes DNS!) |
| `username` / `password` | Admin credentials Vault uses to create new users |

**Notice the connection URL:** `postgres.database.svc.cluster.local:5432`
- This is Kubernetes internal DNS
- `postgres` = Service name
- `database` = Namespace
- `svc.cluster.local` = Kubernetes DNS suffix

### Step 3.3 — Create a Database Role

```bash
# Create a role that defines HOW dynamic credentials are created
vault write database/roles/myapp-role \
  db_name="myappdb" \
  creation_statements="CREATE ROLE \"{{name}}\" WITH LOGIN PASSWORD '{{password}}' VALID UNTIL '{{expiration}}'; GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO \"{{name}}\";" \
  default_ttl="1h" \
  max_ttl="24h"
```

**What each parameter means:**

| Parameter | What It Does |
|-----------|-------------|
| `db_name` | Which database connection to use (the one we configured above) |
| `creation_statements` | SQL that Vault runs to CREATE a new user |
| `default_ttl` | How long the credentials are valid (1 hour) |
| `max_ttl` | Maximum possible lifetime (24 hours, even with renewals) |

**The `creation_statements` SQL explained:**
```sql
-- Vault replaces {{name}}, {{password}}, and {{expiration}} at runtime
CREATE ROLE "v-k8s-myapp-role-abc123xyz"      -- Unique username
  WITH LOGIN 
  PASSWORD 'randomlyGeneratedPassword123!'     -- Random strong password
  VALID UNTIL '2024-01-15 10:00:00';           -- Auto-expires!

-- Give the new user permission to work with tables
GRANT SELECT, INSERT, UPDATE, DELETE 
  ON ALL TABLES IN SCHEMA public 
  TO "v-k8s-myapp-role-abc123xyz";
```

### Step 3.4 — Test Dynamic Credential Generation

```bash
# Generate a dynamic credential — right now, manually
vault read database/creds/myapp-role
```

**Expected Output:**
```
Key                Value
---                -----
lease_id           database/creds/myapp-role/abc123xyz456
lease_duration     1h
lease_renewable    true
password           A1b2C3d4E5-randomPassword
username           v-root-myapp-ro-abc123xyz
```

🎉 **Vault just created a BRAND NEW PostgreSQL user!** This user:
- Has a unique, random username
- Has a strong, random password
- Will be automatically DELETED from PostgreSQL after 1 hour
- Can be renewed (up to 24 hours max)

```bash
# Generate another one — it'll be completely different!
vault read database/creds/myapp-role

# Key                Value
# ---                -----
# password           X9y8Z7w6V5-differentPassword
# username           v-root-myapp-ro-def456uvw    ← Different username!
```

Each pod in your cluster will get its own unique credentials. If one pod is compromised, you revoke its credentials without affecting any other pod.

```bash
# Exit the Vault pod
exit
```

## 🔑 Part 4 — Kubernetes Authentication

Now we need to let Kubernetes pods authenticate to Vault. We use the **Kubernetes Auth Method** — pods prove their identity using their ServiceAccount token.

### Step 4.1 — Enable Kubernetes Auth in Vault

```bash
kubectl exec -it vault-0 -n vault -- /bin/sh

# Enable Kubernetes auth method
vault auth enable kubernetes

# Configure it to talk to the Kubernetes API
vault write auth/kubernetes/config \
  kubernetes_host="https://$KUBERNETES_PORT_443_TCP_ADDR:443"
```

**How does this work?**

```
┌─────────────────────────────────────────────────────┐
│              Kubernetes Auth Flow                     │
│                                                      │
│  1. Pod starts with a ServiceAccount                 │
│  2. Pod sends its ServiceAccount JWT token to Vault  │
│  3. Vault verifies the token with Kubernetes API     │
│  4. If valid, Vault issues a Vault token             │
│  5. Pod uses the Vault token to request secrets      │
│                                                      │
│   Pod ──JWT Token──▶ Vault ──Verify──▶ K8s API      │
│                        │                             │
│                  "Token is valid!                     │
│                   This pod is                        │
│                   'my-app' in                        │
│                   namespace 'app'"                   │
│                        │                             │
│                  Vault Token ◀──                     │
│                        │                             │
│                  Secrets ◀──                         │
└─────────────────────────────────────────────────────┘
```

### Step 4.2 — Create a Vault Policy

A policy defines WHAT a authenticated entity can access:

```bash
# Create a policy that allows reading database credentials
vault policy write myapp-policy - <<EOF
# Allow reading dynamic database credentials
path "database/creds/myapp-role" {
  capabilities = ["read"]
}

# Allow renewing leases (to extend credential lifetime)
path "sys/leases/renew" {
  capabilities = ["create"]
}

# Allow looking up token info
path "auth/token/lookup-self" {
  capabilities = ["read"]
}
EOF
```

**What does this policy do?**
- The app can ONLY read credentials from `database/creds/myapp-role`
- The app can renew its credential leases (extend expiry time)
- The app CANNOT read any other secrets, configure Vault, or do anything else
- This is the **principle of least privilege** — minimum permissions needed

### Step 4.3 — Create a Kubernetes Auth Role

This links a Kubernetes ServiceAccount to a Vault policy:

```bash
# Create a role that maps K8s identity to Vault policy
vault write auth/kubernetes/role/myapp \
  bound_service_account_names=myapp-sa \
  bound_service_account_namespaces=app \
  policies=myapp-policy \
  ttl=24h
```

**What each parameter means:**

| Parameter | What It Does |
|-----------|-------------|
| `bound_service_account_names` | Only pods using ServiceAccount `myapp-sa` can authenticate |
| `bound_service_account_namespaces` | Only pods in namespace `app` can authenticate |
| `policies` | When authenticated, apply the `myapp-policy` we created above |
| `ttl` | The Vault token expires after 24 hours |

```bash
# Exit the Vault pod
exit
```

### Step 4.4 — Create the Kubernetes ServiceAccount

```yaml
# serviceaccount.yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: myapp-sa
  namespace: app
```

```bash
kubectl apply -f serviceaccount.yaml
```

**The connection chain:**
```
ServiceAccount (myapp-sa) 
  → K8s Auth Role (myapp) 
    → Vault Policy (myapp-policy) 
      → Database Credentials (database/creds/myapp-role)
```

## 💉 Part 5 — Vault Agent Injector — Automatic Secret Injection

This is the coolest part. The Vault Agent Injector **automatically** injects secrets into your pods — no code changes needed!

### Step 5.1 — How the Injector Works

```
┌─────────────────────────────────────────────────────────────┐
│              Vault Agent Injector Flow                       │
│                                                             │
│  1. You deploy a Pod with special annotations               │
│  2. The Vault Agent Injector (mutating webhook) catches it  │
│  3. It adds a "sidecar" container (vault-agent) to your Pod │
│  4. The sidecar authenticates to Vault                      │
│  5. The sidecar fetches secrets and writes them to a        │
│     shared volume (/vault/secrets/)                         │
│  6. Your app reads secrets from /vault/secrets/             │
│  7. The sidecar keeps renewing secrets automatically!       │
│                                                             │
│   ┌─────────────────────────────────────────┐               │
│   │              Your Pod                    │               │
│   │                                          │               │
│   │  ┌──────────────┐  ┌──────────────┐     │               │
│   │  │   Your App   │  │ Vault Agent  │     │               │
│   │  │  Container   │  │  (sidecar)   │     │               │
│   │  │              │  │              │     │               │
│   │  │ reads from:  │  │ writes to:   │     │               │
│   │  │ /vault/      │◀─│ /vault/      │     │               │
│   │  │  secrets/    │  │  secrets/    │     │               │
│   │  └──────────────┘  └──────┬───────┘     │               │
│   │                           │              │               │
│   └───────────────────────────┼──────────────┘               │
│                               │                              │
│                               ▼                              │
│                         Vault Server                         │
│                        (vault-0 pod)                         │
└─────────────────────────────────────────────────────────────┘
```

### Step 5.2 — Deploy Your Application with Vault Annotations

```yaml
# app-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
  namespace: app
spec:
  replicas: 1
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
      annotations:
        # ═══════════════════════════════════════════════════
        # These annotations tell the Vault Agent Injector
        # what to do with this pod
        # ═══════════════════════════════════════════════════
        
        vault.hashicorp.com/agent-inject: "true"
        # ↑ "Hey Vault Injector, please inject secrets into this pod!"
        
        vault.hashicorp.com/role: "myapp"
        # ↑ "Use the 'myapp' auth role we created in Vault"
        
        vault.hashicorp.com/agent-inject-secret-db-creds: "database/creds/myapp-role"
        # ↑ "Fetch credentials from this Vault path and save to:
        #    /vault/secrets/db-creds"
        
        vault.hashicorp.com/agent-inject-template-db-creds: |
          {{- with secret "database/creds/myapp-role" -}}
          {
            "db_host": "postgres.database.svc.cluster.local",
            "db_port": "5432",
            "db_name": "myappdb",
            "db_username": "{{ .Data.username }}",
            "db_password": "{{ .Data.password }}"
          }
          {{- end -}}
        # ↑ "Format the secret as JSON with these specific fields"
        
    spec:
      serviceAccountName: myapp-sa       # MUST match the Vault auth role
      containers:
        - name: myapp
          image: busybox:latest
          command: ["/bin/sh", "-c"]
          args:
            - |
              echo "=== Waiting for secrets to be injected ==="
              while [ ! -f /vault/secrets/db-creds ]; do
                echo "Waiting for /vault/secrets/db-creds..."
                sleep 2
              done
              echo "=== Secrets are available! ==="
              echo "--- Contents of /vault/secrets/db-creds ---"
              cat /vault/secrets/db-creds
              echo ""
              echo "=== App is running with dynamic credentials ==="
              # In a real app, you'd read this file and connect to the database
              # For now, let's just keep the pod running and print credentials
              while true; do
                echo "--- Current credentials (may rotate!) ---"
                cat /vault/secrets/db-creds
                echo ""
                sleep 30
              done
```

**Let me explain every annotation:**

| Annotation | What It Does |
|-----------|-------------|
| `agent-inject: "true"` | Activates the Vault Agent Injector for this pod |
| `role: "myapp"` | Which Vault Kubernetes auth role to use |
| `agent-inject-secret-db-creds` | Vault path → writes to `/vault/secrets/db-creds` |
| `agent-inject-template-db-creds` | Custom template to format the secret output |

## 🔄 Part 6 — Secret Rotation in Action

Let's see what happens when credentials expire and get rotated.

### Step 6.1 — Watch Credentials Rotate

```bash
# Watch the secrets file — it updates automatically!
kubectl exec -it deployment/myapp -n app -c myapp -- /bin/sh -c \
  'while true; do echo "$(date): $(cat /vault/secrets/db-creds | grep db_username)"; sleep 60; done'
```

**What happens over time:**
```
Mon 10:00:00: "db_username": "v-kubernet-myapp-ro-abc123"   ← Original
Mon 10:30:00: "db_username": "v-kubernet-myapp-ro-abc123"   ← Same (renewed)
Mon 10:55:00: "db_username": "v-kubernet-myapp-ro-def456"   ← New credentials!
Mon 11:00:00: "db_username": "v-kubernet-myapp-ro-def456"   ← Still new
```

The Vault Agent automatically:
1. Renews credentials before they expire
2. If renewal fails (e.g., max TTL reached), gets brand new credentials
3. Updates the file at `/vault/secrets/db-creds`
4. Your app just needs to re-read the file!

### Step 6.2 — Verify in PostgreSQL

```bash
# Connect to PostgreSQL and check which users exist
kubectl exec -it deployment/postgres -n database -- psql -U admin -d myappdb -c "\du"
```

**Expected Output:**
```
                             List of roles
         Role name          |  Attributes  | Member of
----------------------------+--------------+-----------
 admin                      | Superuser    | {}
 v-kubernet-myapp-ro-abc123 | Login        | {}    ← Vault-created user!
 v-kubernet-myapp-ro-def456 | Login        | {}    ← Another Vault-created user!
```

When these users' TTLs expire, Vault will automatically run a `DROP ROLE` statement to clean them up!

### Step 6.3 — Manual Credential Revocation

```bash
# If a pod is compromised, you can immediately revoke its credentials
kubectl exec -it vault-0 -n vault -- vault lease revoke -prefix database/creds/myapp-role

# This instantly:
# 1. Invalidates ALL current credentials for this role
# 2. Drops the users from PostgreSQL
# 3. Pods will get new credentials on next rotation
```

## 🔏 Part 7 — KV Secrets Engine (Static Secrets)

Not everything needs dynamic secrets. For configuration values, API keys, etc., use the KV (Key-Value) engine:

### Step 7.1 — Store Static Secrets in Vault

```bash
kubectl exec -it vault-0 -n vault -- /bin/sh

# Enable KV v2 secrets engine
vault secrets enable -path=secret kv-v2

# Store some application secrets
vault kv put secret/myapp/config \
  api_key="sk-abc123def456" \
  smtp_host="smtp.gmail.com" \
  smtp_port="587" \
  feature_flag_dark_mode="true"

# Read it back
vault kv get secret/myapp/config
```

**Expected Output:**
```
====== Metadata ======
Key                Value
---                -----
created_time       2024-01-15T10:00:00Z
custom_metadata    <nil>
deletion_time      n/a
destroyed          false
version            1

====== Data ======
Key                       Value
---                       -----
api_key                   sk-abc123def456
feature_flag_dark_mode    true
smtp_host                 smtp.gmail.com
smtp_port                 587
```

### Step 7.2 — Update the Vault Policy

```bash
# Update policy to include KV access
vault policy write myapp-policy - <<EOF
# Dynamic database credentials
path "database/creds/myapp-role" {
  capabilities = ["read"]
}

# Static KV secrets
path "secret/data/myapp/config" {
  capabilities = ["read"]
}

# Lease renewal
path "sys/leases/renew" {
  capabilities = ["create"]
}

path "auth/token/lookup-self" {
  capabilities = ["read"]
}
EOF

exit
```

### Step 7.3 — Inject Both Dynamic and Static Secrets

```yaml
# app-deployment-v2.yaml — Updated with both secret types
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
  namespace: app
spec:
  replicas: 1
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
      annotations:
        vault.hashicorp.com/agent-inject: "true"
        vault.hashicorp.com/role: "myapp"
        
        # Secret 1: Dynamic database credentials
        vault.hashicorp.com/agent-inject-secret-db-creds: "database/creds/myapp-role"
        vault.hashicorp.com/agent-inject-template-db-creds: |
          {{- with secret "database/creds/myapp-role" -}}
          export DB_HOST="postgres.database.svc.cluster.local"
          export DB_PORT="5432"
          export DB_NAME="myappdb"
          export DB_USERNAME="{{ .Data.username }}"
          export DB_PASSWORD="{{ .Data.password }}"
          {{- end -}}
        
        # Secret 2: Static app configuration
        vault.hashicorp.com/agent-inject-secret-app-config: "secret/data/myapp/config"
        vault.hashicorp.com/agent-inject-template-app-config: |
          {{- with secret "secret/data/myapp/config" -}}
          export API_KEY="{{ .Data.data.api_key }}"
          export SMTP_HOST="{{ .Data.data.smtp_host }}"
          export SMTP_PORT="{{ .Data.data.smtp_port }}"
          export FEATURE_DARK_MODE="{{ .Data.data.feature_flag_dark_mode }}"
          {{- end -}}
        
    spec:
      serviceAccountName: myapp-sa
      containers:
        - name: myapp
          image: busybox:latest
          command: ["/bin/sh", "-c"]
          args:
            - |
              # Source the secrets as environment variables
              while [ ! -f /vault/secrets/db-creds ]; do sleep 2; done
              source /vault/secrets/db-creds
              source /vault/secrets/app-config
              
              echo "=== Application Started ==="
              echo "DB Host: $DB_HOST"
              echo "DB Username: $DB_USERNAME"
              echo "API Key: $API_KEY (first 10 chars)"
              echo "SMTP Host: $SMTP_HOST"
              echo "Feature Dark Mode: $FEATURE_DARK_MODE"
              
              # Keep running
              while true; do sleep 30; done
```

**Notice the template format:**
- For dynamic secrets: `{{ .Data.username }}` — directly in the secret data
- For KV v2 secrets: `{{ .Data.data.api_key }}` — note the extra `.data` (KV v2 wraps data in a metadata envelope)

```bash
kubectl apply -f app-deployment-v2.yaml
```

## 🔒 Part 8 — Transit Secrets Engine (Encryption as a Service)

Vault can encrypt/decrypt data for you without your app ever seeing the encryption key!

### Step 8.1 — Enable Transit Engine

```bash
kubectl exec -it vault-0 -n vault -- /bin/sh

# Enable transit secrets engine
vault secrets enable transit

# Create an encryption key
vault write -f transit/keys/myapp-encryption

# Encrypt some data
vault write transit/encrypt/myapp-encryption \
  plaintext=$(echo -n "My super secret credit card: 4111-1111-1111-1111" | base64)
```

**Expected Output:**
```
Key            Value
---            -----
ciphertext     vault:v1:abc123encrypted456data789...
key_version    1
```

```bash
# Decrypt it back
vault write transit/decrypt/myapp-encryption \
  ciphertext="vault:v1:abc123encrypted456data789..."

# The result is base64 encoded — decode it:
echo "base64_result_here" | base64 -d
# Output: My super secret credit card: 4111-1111-1111-1111
```

**Why is this useful?**
- Your app sends data to Vault for encryption → gets ciphertext back
- Your app stores the ciphertext in the database
- When needed, sends ciphertext to Vault → gets plaintext back
- Your app NEVER handles encryption keys
- Vault manages key rotation, and you can re-encrypt data with new keys without downtime

```bash
# Rotate the encryption key
vault write -f transit/keys/myapp-encryption/rotate

# Now new encryptions use key version 2
# Old ciphertext (v1) can still be decrypted

# Re-encrypt old data with the new key
vault write transit/rewrap/myapp-encryption \
  ciphertext="vault:v1:old_ciphertext_here"
# Returns: vault:v2:new_ciphertext_here

exit
```

## 📜 Part 9 — PKI Secrets Engine (Auto-Generated TLS Certificates)

Vault can be your internal Certificate Authority (CA) and auto-generate TLS certificates!

### Step 9.1 — Set Up a PKI Engine

```bash
kubectl exec -it vault-0 -n vault -- /bin/sh

# Enable PKI secrets engine
vault secrets enable pki

# Configure maximum TTL for certificates (1 year)
vault secrets tune -max-lease-ttl=8760h pki

# Generate a root certificate
vault write pki/root/generate/internal \
  common_name="My Company Internal CA" \
  ttl=8760h

# Create a role for issuing certificates
vault write pki/roles/internal-services \
  allowed_domains="svc.cluster.local,internal.mycompany.com" \
  allow_subdomains=true \
  max_ttl=720h \
  generate_lease=true
```

### Step 9.2 — Generate a TLS Certificate

```bash
# Issue a certificate for a service
vault write pki/issue/internal-services \
  common_name="myapp.app.svc.cluster.local" \
  ttl=24h
```

**Expected Output:**
```
Key                 Value
---                 -----
certificate         -----BEGIN CERTIFICATE-----
                    MIIDGzCCA...
                    -----END CERTIFICATE-----
issuing_ca          -----BEGIN CERTIFICATE-----
                    MIIDFzCCA...
                    -----END CERTIFICATE-----
private_key         -----BEGIN RSA PRIVATE KEY-----
                    MIIEpAIBA...
                    -----END RSA PRIVATE KEY-----
serial_number       1a:2b:3c:4d:5e:6f
```

🎉 **Vault just generated a TLS certificate that expires in 24 hours!** No more dealing with Let's Encrypt or manual certificate management for internal services.

## 🏆 Part 10 — Production Best Practices

### What to Do Differently in Production

| Lab Setup | Production Setup |
|-----------|-----------------|
| Dev mode (auto-unseal) | HA mode with auto-unseal via AWS KMS / Azure Key Vault |
| In-memory storage | Persistent storage (Consul, Raft, or cloud storage) |
| Root token: "root" | Root token generated once, then revoked |
| No TLS | Full TLS encryption |
| Single node | 3 or 5 node HA cluster |
| No audit logging | Audit logging to file and SIEM |
| No backup | Regular Raft snapshots / Consul backups |

### Production Vault Architecture

```
┌─────────────────────────────────────────────────────┐
│               Production Vault Setup                 │
│                                                      │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐             │
│  │ Vault-0 │  │ Vault-1 │  │ Vault-2 │   3 replicas│
│  │ (Active)│  │(Standby)│  │(Standby)│   for HA    │
│  └────┬────┘  └────┬────┘  └────┬────┘             │
│       │             │            │                   │
│       └─────────────┼────────────┘                   │
│                     │                                │
│              ┌──────▼──────┐                         │
│              │ Raft Storage│  Integrated storage     │
│              │ (replicated)│  (or Consul)            │
│              └─────────────┘                         │
│                                                      │
│  Auto-unseal: AWS KMS / Azure Key Vault / GCP KMS   │
│  Audit: Enabled (file + syslog)                      │
│  TLS: Full end-to-end encryption                     │
│  Backup: Automated Raft snapshots                    │
└─────────────────────────────────────────────────────┘
```

### Common Mistakes to Avoid

| ❌ Don't Do This | ✅ Do This Instead |
|------------------|-------------------|
| Use dev mode in production | Use HA mode with auto-unseal |
| Keep the root token around | Generate, use once, then revoke |
| Skip audit logging | Enable audit to file + SIEM |
| Long TTLs (30 days) | Short TTLs (1-24 hours) |
| One policy for everything | Granular policies per service |
| No backup strategy | Automated Raft snapshot backups |
| Share Vault admin access | Individual admin accounts with MFA |
