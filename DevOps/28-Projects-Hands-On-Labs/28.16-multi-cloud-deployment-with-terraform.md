# 28.16 — Multi-Cloud Deployment with Terraform

> **Project Goal:** Deploy the **same application** across **AWS and Azure** (or GCP) using Terraform — with a single codebase, modular structure, and real-world best practices.

> **Level:** Advanced

> **What You'll Learn:** How to write reusable Terraform modules, use workspaces, manage multiple providers, and deploy infrastructure on more than one cloud — just like big companies do in production.

---

## 🤔 Why Multi-Cloud?

Before we jump into code, let's understand **why** companies even bother with multi-cloud:

| Reason | Explanation |
|--------|-------------|
| **Avoid Vendor Lock-in** | If you only use AWS and they raise prices or have an outage, you're stuck |
| **High Availability** | If AWS goes down in a region, your app is still running on Azure |
| **Best-of-Breed** | Use the best service from each provider (e.g., AWS Lambda + GCP BigQuery) |
| **Compliance** | Some regulations require data to be in specific clouds or regions |
| **Cost Optimization** | Use the cheapest cloud for each workload |

### Real-World Example

Netflix, Spotify, and most Fortune 500 companies run on **multiple clouds**. They don't put all their eggs in one basket.

## 🚀 Step 1 — Set Up Providers

The **provider** block tells Terraform which cloud(s) to talk to.

### `providers.tf`

```hcl
# ─────────────────────────────────────────────
# Terraform Settings — Required Providers
# ─────────────────────────────────────────────
terraform {
  required_version = ">= 1.5.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
    azurerm = {
      source  = "hashicorp/azurerm"
      version = "~> 3.0"
    }
  }
}

# ─────────────────────────────────────────────
# AWS Provider
# ─────────────────────────────────────────────
provider "aws" {
  region = var.aws_region

  default_tags {
    tags = {
      Project     = var.project_name
      Environment = var.environment
      ManagedBy   = "Terraform"
    }
  }
}

# ─────────────────────────────────────────────
# Azure Provider
# ─────────────────────────────────────────────
provider "azurerm" {
  features {}

  # Azure doesn't have default_tags like AWS,
  # so we'll pass tags to each resource manually
}
```

**What's happening here?**

- `required_providers` — We're telling Terraform: "Hey, I need the AWS plugin and the Azure plugin."
- `provider "aws"` — Configures AWS. The `region` comes from a variable. The `default_tags` block automatically adds tags to every AWS resource we create — super handy!
- `provider "azurerm"` — Configures Azure. The `features {}` block is required (even if empty) by the Azure provider.

## 🚀 Step 3 — AWS Networking Module

### `modules/aws/networking/variables.tf`

```hcl
variable "project_name" {
  type = string
}

variable "environment" {
  type = string
}

variable "vpc_cidr" {
  description = "CIDR block for the VPC"
  type        = string
}

variable "public_subnet_cidrs" {
  description = "List of CIDR blocks for public subnets"
  type        = list(string)
  default     = ["10.0.1.0/24", "10.0.2.0/24"]
}

variable "availability_zones" {
  description = "List of AZs to use"
  type        = list(string)
  default     = ["us-east-1a", "us-east-1b"]
}
```

### `modules/aws/networking/main.tf`

```hcl
# ─────────────────────────────────────────────
# VPC — Virtual Private Cloud
# Think of this as your own private network in AWS
# ─────────────────────────────────────────────
resource "aws_vpc" "main" {
  cidr_block           = var.vpc_cidr
  enable_dns_hostnames = true
  enable_dns_support   = true

  tags = {
    Name = "${var.project_name}-${var.environment}-vpc"
  }
}

# ─────────────────────────────────────────────
# Internet Gateway — Allows resources to talk to the internet
# Without this, your VPC is completely isolated
# ─────────────────────────────────────────────
resource "aws_internet_gateway" "main" {
  vpc_id = aws_vpc.main.id

  tags = {
    Name = "${var.project_name}-${var.environment}-igw"
  }
}

# ─────────────────────────────────────────────
# Public Subnets — Where our EC2 instances will live
# We create 2 subnets in 2 different AZs for high availability
# ─────────────────────────────────────────────
resource "aws_subnet" "public" {
  count = length(var.public_subnet_cidrs)

  vpc_id                  = aws_vpc.main.id
  cidr_block              = var.public_subnet_cidrs[count.index]
  availability_zone       = var.availability_zones[count.index]
  map_public_ip_on_launch = true

  tags = {
    Name = "${var.project_name}-${var.environment}-public-${count.index + 1}"
  }
}

# ─────────────────────────────────────────────
# Route Table — Tells traffic where to go
# This one sends all internet-bound traffic (0.0.0.0/0) to the IGW
# ─────────────────────────────────────────────
resource "aws_route_table" "public" {
  vpc_id = aws_vpc.main.id

  route {
    cidr_block = "0.0.0.0/0"
    gateway_id = aws_internet_gateway.main.id
  }

  tags = {
    Name = "${var.project_name}-${var.environment}-public-rt"
  }
}

# ─────────────────────────────────────────────
# Associate each public subnet with the route table
# ─────────────────────────────────────────────
resource "aws_route_table_association" "public" {
  count = length(aws_subnet.public)

  subnet_id      = aws_subnet.public[count.index].id
  route_table_id = aws_route_table.public.id
}

# ─────────────────────────────────────────────
# Security Group — Firewall rules for our instances
# Allows HTTP (80), HTTPS (443), and SSH (22)
# ─────────────────────────────────────────────
resource "aws_security_group" "web" {
  name_prefix = "${var.project_name}-${var.environment}-web-"
  vpc_id      = aws_vpc.main.id
  description = "Security group for web servers"

  ingress {
    description = "HTTP"
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  ingress {
    description = "HTTPS"
    from_port   = 443
    to_port     = 443
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  ingress {
    description = "SSH"
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]  # ⚠️ In production, restrict this to your IP!
  }

  egress {
    description = "Allow all outbound"
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }

  tags = {
    Name = "${var.project_name}-${var.environment}-web-sg"
  }
}
```

**Let me explain every block:**

1. **VPC** — Think of it as building a fence around your own plot of land in the cloud. Everything you create goes inside this fence.

2. **Internet Gateway** — This is the door in your fence that connects to the internet. Without it, nothing inside your VPC can reach the outside world.

3. **Subnets** — We split our VPC into smaller sections. We use 2 subnets in 2 different data centers (Availability Zones) so if one data center has a fire, the other keeps running.

4. **Route Table** — This is like a GPS for your network traffic. It says: "If you want to go to the internet (0.0.0.0/0), use the Internet Gateway."

5. **Security Group** — This is your firewall. We allow web traffic (port 80/443) and SSH (port 22) in, and allow all traffic out.

### `modules/aws/compute/main.tf`

```hcl
# ─────────────────────────────────────────────
# Get the latest Amazon Linux 2023 AMI
# Instead of hardcoding an AMI ID (which changes per region),
# we dynamically look it up — much cleaner!
# ─────────────────────────────────────────────
data "aws_ami" "amazon_linux" {
  most_recent = true
  owners      = ["amazon"]

  filter {
    name   = "name"
    values = ["al2023-ami-*-x86_64"]
  }

  filter {
    name   = "virtualization-type"
    values = ["hvm"]
  }
}

# ─────────────────────────────────────────────
# EC2 Instance — Our web server
# ─────────────────────────────────────────────
resource "aws_instance" "web" {
  ami                    = data.aws_ami.amazon_linux.id
  instance_type          = var.instance_type
  subnet_id              = var.subnet_id
  vpc_security_group_ids = [var.security_group_id]

  # Startup script — runs when the instance first boots
  user_data = var.user_data != "" ? var.user_data : <<-EOF
    #!/bin/bash
    yum update -y
    yum install -y httpd
    systemctl start httpd
    systemctl enable httpd
    echo "<h1>Hello from AWS! 🌩️</h1><p>Instance: $(hostname)</p><p>Region: $(curl -s http://169.254.169.254/latest/meta-data/placement/region)</p>" > /var/www/html/index.html
  EOF

  root_block_device {
    volume_size = 20
    volume_type = "gp3"
    encrypted   = true   # Always encrypt your disks!
  }

  tags = {
    Name = "${var.project_name}-${var.environment}-web-server"
  }
}
```

**Breaking this down:**

- **`data "aws_ami"`** — Instead of hardcoding an AMI ID like `ami-0abcdef1234567890` (which is different in every AWS region), we tell Terraform: "Find me the latest Amazon Linux 2023 AMI." This makes our code work in ANY region.

- **`user_data`** — This is a startup script. When the EC2 instance boots for the first time, it runs this script to install Apache (httpd) and create a simple web page.

- **`root_block_device`** — This configures the disk. We use 20GB of `gp3` storage (fast SSD) and **encrypt it** — always encrypt your disks in production!

### `modules/aws/loadbalancer/main.tf`

```hcl
# ─────────────────────────────────────────────
# Application Load Balancer (ALB)
# This sits in front of your EC2 instances and distributes traffic
# Think of it as a traffic cop for your web servers
# ─────────────────────────────────────────────
resource "aws_lb" "main" {
  name               = "${var.project_name}-${var.environment}-alb"
  internal           = false          # false = internet-facing
  load_balancer_type = "application"  # Layer 7 (HTTP/HTTPS)
  security_groups    = [var.security_group_id]
  subnets            = var.subnet_ids

  tags = {
    Name = "${var.project_name}-${var.environment}-alb"
  }
}

# ─────────────────────────────────────────────
# Target Group — Defines WHERE to send traffic
# The ALB sends traffic to targets (our EC2 instances) in this group
# ─────────────────────────────────────────────
resource "aws_lb_target_group" "web" {
  name     = "${var.project_name}-${var.environment}-tg"
  port     = 80
  protocol = "HTTP"
  vpc_id   = var.vpc_id

  health_check {
    path                = "/"
    protocol            = "HTTP"
    healthy_threshold   = 2
    unhealthy_threshold = 3
    timeout             = 5
    interval            = 30
  }
}

# ─────────────────────────────────────────────
# Register our EC2 instance with the target group
# ─────────────────────────────────────────────
resource "aws_lb_target_group_attachment" "web" {
  target_group_arn = aws_lb_target_group.web.arn
  target_id        = var.instance_id
  port             = 80
}

# ─────────────────────────────────────────────
# Listener — Defines HOW to receive traffic
# "Listen on port 80 and forward to the target group"
# ─────────────────────────────────────────────
resource "aws_lb_listener" "http" {
  load_balancer_arn = aws_lb.main.arn
  port              = 80
  protocol          = "HTTP"

  default_action {
    type             = "forward"
    target_group_arn = aws_lb_target_group.web.arn
  }
}
```

**How an ALB works (simple version):**

```
User's Browser
      │
      ▼
┌─────────────┐     ┌─────────────────┐     ┌──────────┐
│  ALB        │────▶│  Target Group    │────▶│   EC2    │
│  (Port 80)  │     │  (Health Check)  │     │ (Apache) │
└─────────────┘     └─────────────────┘     └──────────┘
```

1. User hits the ALB's URL
2. ALB checks the Target Group to see which instances are healthy
3. ALB forwards the request to a healthy EC2 instance

### `modules/azure/networking/main.tf`

```hcl
# ─────────────────────────────────────────────
# Resource Group — Azure's way of organizing resources
# Think of it like a folder — everything related goes in here
# AWS doesn't have this concept; Azure requires it
# ─────────────────────────────────────────────
resource "azurerm_resource_group" "main" {
  name     = "${var.project_name}-${var.environment}-rg"
  location = var.location

  tags = {
    Project     = var.project_name
    Environment = var.environment
    ManagedBy   = "Terraform"
  }
}

# ─────────────────────────────────────────────
# Virtual Network (VNet) — Azure's version of AWS VPC
# Same concept: your own private network in the cloud
# ─────────────────────────────────────────────
resource "azurerm_virtual_network" "main" {
  name                = "${var.project_name}-${var.environment}-vnet"
  address_space       = [var.vnet_cidr]
  location            = azurerm_resource_group.main.location
  resource_group_name = azurerm_resource_group.main.name

  tags = {
    Project     = var.project_name
    Environment = var.environment
  }
}

# ─────────────────────────────────────────────
# Subnets — Just like AWS subnets
# ─────────────────────────────────────────────
resource "azurerm_subnet" "public" {
  count = length(var.subnet_cidrs)

  name                 = "${var.project_name}-${var.environment}-subnet-${count.index + 1}"
  resource_group_name  = azurerm_resource_group.main.name
  virtual_network_name = azurerm_virtual_network.main.name
  address_prefixes     = [var.subnet_cidrs[count.index]]
}

# ─────────────────────────────────────────────
# Network Security Group (NSG) — Azure's version of AWS Security Group
# Same idea: define what traffic is allowed in and out
# ─────────────────────────────────────────────
resource "azurerm_network_security_group" "web" {
  name                = "${var.project_name}-${var.environment}-web-nsg"
  location            = azurerm_resource_group.main.location
  resource_group_name = azurerm_resource_group.main.name

  security_rule {
    name                       = "AllowHTTP"
    priority                   = 100
    direction                  = "Inbound"
    access                     = "Allow"
    protocol                   = "Tcp"
    source_port_range          = "*"
    destination_port_range     = "80"
```

### AWS vs Azure — Side-by-Side Comparison

```hcl
# ─────────────────────────────────────────────
# Public IP — So our VM is accessible from the internet
# AWS EC2 gets a public IP automatically (if configured)
# Azure requires you to create one explicitly
# ─────────────────────────────────────────────
resource "azurerm_public_ip" "web" {
  name                = "${var.project_name}-${var.environment}-web-pip"
  location            = var.location
  resource_group_name = var.resource_group_name
  allocation_method   = "Static"
  sku                 = "Standard"

  tags = {
    Project     = var.project_name
    Environment = var.environment
  }
}

# ─────────────────────────────────────────────
# Network Interface (NIC) — Connects the VM to the network
# Think of it as the "network card" for your virtual machine
# AWS does this automatically; Azure requires it explicitly
# ─────────────────────────────────────────────
resource "azurerm_network_interface" "web" {
  name                = "${var.project_name}-${var.environment}-web-nic"
  location            = var.location
  resource_group_name = var.resource_group_name

  ip_configuration {
    name                          = "internal"
    subnet_id                     = var.subnet_id
    private_ip_address_allocation = "Dynamic"
    public_ip_address_id          = azurerm_public_ip.web.id
  }

  tags = {
    Project     = var.project_name
    Environment = var.environment
  }
}

# ─────────────────────────────────────────────
# SSH Key — We'll generate one for the VM
# Never use passwords for VMs — always use SSH keys!
# ─────────────────────────────────────────────
resource "tls_private_key" "ssh" {
  algorithm = "RSA"
  rsa_bits  = 4096
}

# ─────────────────────────────────────────────
# Azure Linux VM — Our web server
# ─────────────────────────────────────────────
resource "azurerm_linux_virtual_machine" "web" {
  name                  = "${var.project_name}-${var.environment}-web-vm"
  location              = var.location
  resource_group_name   = var.resource_group_name
  size                  = var.vm_size
  admin_username        = var.admin_username
  network_interface_ids = [azurerm_network_interface.web.id]

  admin_ssh_key {
    username   = var.admin_username
    public_key = tls_private_key.ssh.public_key_openssh
  }

  os_disk {
    caching              = "ReadWrite"
    storage_account_type = "Standard_LRS"
  }

  source_image_reference {
    publisher = "Canonical"
    offer     = "0001-com-ubuntu-server-jammy"
    sku       = "22_04-lts"
    version   = "latest"
  }

  # Startup script — install nginx and create a welcome page
  custom_data = base64encode(<<-EOF
    #!/bin/bash
    apt-get update -y
    apt-get install -y nginx
    systemctl start nginx
    systemctl enable nginx
    echo "<h1>Hello from Azure! ☁️</h1><p>VM: $(hostname)</p><p>Region: ${var.location}</p>" > /var/www/html/index.html
  EOF
  )

  tags = {
    Project     = var.project_name
    Environment = var.environment
  }
}
```

**Key Differences from AWS:**

| Aspect | AWS | Azure |
|--------|-----|-------|
| Public IP | Automatic (with setting) | Must create separately |
| Network Card | Automatic | Must create NIC explicitly |
| Startup Script | `user_data` (plain text) | `custom_data` (must be base64 encoded) |
| SSH Key | Key Pair (separate resource) | Inline in VM config |
| AMI/Image | AMI ID or data source | `source_image_reference` block |

```hcl
  request_path    = "/"
```

**The magic of `count = var.enable_aws ? 1 : 0`:**

This is a Terraform conditional. It reads as:
- "If `enable_aws` is true, create 1 instance of this module"
- "If `enable_aws` is false, create 0 instances (skip it entirely)"

This lets you toggle entire clouds on/off with a single variable!

## 🏗️ Advanced: Adding Terraform Workspaces

Workspaces let you manage **multiple environments** (dev, staging, prod) with the **same code**:

```bash
# Create workspaces
terraform workspace new dev
terraform workspace new staging
terraform workspace new prod

# Switch between them
terraform workspace select dev

# List all workspaces
terraform workspace list

# Apply to a specific workspace
terraform workspace select staging
terraform apply -var-file="environments/staging.tfvars"
```

Each workspace has its **own state file**, so dev and prod don't interfere with each other.

## 🧠 Key Concepts Recap

| Concept | What It Means |
|---------|--------------|
| **Provider** | Plugin that lets Terraform talk to a cloud (AWS, Azure, GCP) |
| **Module** | Reusable group of Terraform resources (like a function in programming) |
| **State** | Terraform's memory — it remembers what it created |
| **Variable** | Input parameters to make your code flexible |
| **Output** | Values that Terraform shows you after running (like IPs, URLs) |
| **Conditional** | `count = var.x ? 1 : 0` — create or skip resources based on conditions |
| **Backend** | Where state is stored (local file or remote like S3) |
| **Workspace** | Separate state files for different environments |

## ⚠️ Common Mistakes & How to Avoid Them

| Mistake | Fix |
|---------|-----|
| Committing `.tfvars` to git | Add to `.gitignore` — it may contain secrets |
| Committing `.tfstate` to git | Use remote backend (S3) — state contains sensitive data |
| Hardcoding AMI IDs | Use `data` sources to look them up dynamically |
| Hardcoding regions | Use variables for everything |
| Not using modules | Always modularize — or your `main.tf` becomes 1000+ lines of mess |
| Not encrypting disks | Always set `encrypted = true` on EBS volumes |
| Open SSH to 0.0.0.0/0 | Restrict SSH to your IP in production |
| Forgetting `terraform destroy` | You'll get a surprise cloud bill! |
