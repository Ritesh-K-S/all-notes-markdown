# 28.17 — Chaos Engineering — Inject Failures in Kubernetes with Litmus

> **Project Goal:** Set up **LitmusChaos** on a Kubernetes cluster and run real chaos experiments — kill pods, inject network latency, stress CPU/memory — to test how resilient your applications really are.

> **Level:** Advanced

> **What You'll Learn:** What chaos engineering is, why it matters, how to install LitmusChaos, create chaos experiments, observe the impact, and build confidence that your system can handle real-world failures.

---

## 🤔 What is Chaos Engineering?

Let me explain it with a simple story.

Imagine you build a beautiful house. It looks perfect. But have you tested it against a storm? An earthquake? A flood? You **don't know** if your house is strong until it faces real problems.

**Chaos Engineering is the same thing — but for software.**

> **Chaos Engineering** = Intentionally breaking things in a controlled way to find weaknesses BEFORE they cause real outages.

### The Netflix Origin Story

In 2010, Netflix moved to AWS. They were worried: "What happens if an AWS server dies randomly?" Instead of worrying, they built **Chaos Monkey** — a tool that **randomly kills production servers** during business hours.

If their system survived Chaos Monkey, it could survive anything. And it did! This is how Chaos Engineering was born.

---

## 📋 Why Should You Care?

| Without Chaos Engineering | With Chaos Engineering |
|--------------------------|----------------------|
| "I think our app is resilient" | "I **know** our app handles pod failures" |
| Find bugs when customers complain | Find bugs before they hit production |
| Firefighting at 3 AM | Confidence that things will recover |
| "It works on my machine" | "It works when half the cluster is down" |

### Real-World Failures Chaos Engineering Can Simulate

```
Pod gets killed randomly               → Does Kubernetes restart it?
Network becomes slow (latency)          → Does the app timeout gracefully?
A database becomes unreachable          → Does the app show a nice error?
CPU goes to 100%                        → Does auto-scaling kick in?
A whole availability zone goes down     → Does traffic shift to another zone?
DNS stops resolving                     → Does the app cache DNS properly?
Disk fills up                           → Does the app handle it gracefully?
```

## 🛠️ What is LitmusChaos?

**LitmusChaos** is an open-source, cloud-native chaos engineering framework built specifically for Kubernetes. It's a CNCF (Cloud Native Computing Foundation) project — the same foundation that manages Kubernetes itself.

### Why Litmus?

| Feature | Description |
|---------|-------------|
| **Kubernetes-Native** | Built as CRDs (Custom Resource Definitions) — feels natural in K8s |
| **ChaosHub** | 50+ pre-built experiments ready to use |
| **Web UI** | Beautiful dashboard (LitmusChaos Center) to manage experiments |
| **GitOps Ready** | Store experiments in Git, trigger via CI/CD |
| **Observability** | Built-in monitoring with Prometheus metrics |
| **CNCF Project** | Backed by the cloud-native community |

## 🚀 Step 2 — Deploy a Sample Application

We need an app to test against. Let's deploy a simple nginx app with multiple replicas:

### `sample-app.yaml`

```yaml
# ─────────────────────────────────────────────
# Namespace for our target application
# ─────────────────────────────────────────────
apiVersion: v1
kind: Namespace
metadata:
  name: demo-app

---
# ─────────────────────────────────────────────
# Deployment — 3 replicas of nginx
# 3 replicas means if 1 pod dies, 2 others keep serving traffic
# ─────────────────────────────────────────────
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-demo
  namespace: demo-app
  labels:
    app: nginx-demo
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx-demo
  template:
    metadata:
      labels:
        app: nginx-demo
    spec:
      containers:
        - name: nginx
          image: nginx:1.25
          ports:
            - containerPort: 80
          resources:
            requests:
              cpu: "50m"
              memory: "64Mi"
            limits:
              cpu: "100m"
              memory: "128Mi"
          # Readiness probe — K8s uses this to know if the pod is ready
          readinessProbe:
            httpGet:
              path: /
              port: 80
            initialDelaySeconds: 5
            periodSeconds: 10
          # Liveness probe — K8s uses this to know if the pod is alive
          livenessProbe:
            httpGet:
              path: /
              port: 80
            initialDelaySeconds: 10
            periodSeconds: 15

---
# ─────────────────────────────────────────────
# Service — Expose the deployment
# ─────────────────────────────────────────────
apiVersion: v1
kind: Service
metadata:
  name: nginx-demo-service
  namespace: demo-app
spec:
  selector:
    app: nginx-demo
  ports:
    - port: 80
      targetPort: 80
  type: ClusterIP
```

```bash
# Deploy the sample app
kubectl apply -f sample-app.yaml

# Verify
kubectl get pods -n demo-app
kubectl get svc -n demo-app
```

**Expected output:**
```

nginx-demo-abc123-ghi67       1/1     Running   0          30s
```

All 3 replicas running.

## 🚀 Step 4 — Create a Service Account for Chaos

Chaos experiments need permissions to do things like killing pods. We need to create a ServiceAccount with the right RBAC:

### `chaos-rbac.yaml`

```yaml
# ─────────────────────────────────────────────
# Service Account for Chaos experiments
# This gives the chaos runner permission to kill/modify pods
# ─────────────────────────────────────────────
apiVersion: v1
kind: ServiceAccount
metadata:
  name: chaos-sa
  namespace: demo-app
  labels:
    app.kubernetes.io/part-of: litmus

---
# ─────────────────────────────────────────────
# ClusterRole — What permissions does chaos need?
# It needs to list, get, delete, and create pods
# ─────────────────────────────────────────────
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: chaos-cluster-role
  labels:
    app.kubernetes.io/part-of: litmus
rules:
  # Need to manage pods (the main target of chaos)
  - apiGroups: [""]
    resources: ["pods", "pods/log", "pods/exec", "events", "services"]
    verbs: ["get", "list", "watch", "create", "delete", "deletecollection", "patch", "update"]
  # Need to manage deployments, replicasets
  - apiGroups: ["apps"]
    resources: ["deployments", "replicasets", "statefulsets", "daemonsets"]
    verbs: ["get", "list", "watch", "update", "patch"]
  # Need to manage jobs (chaos runner creates jobs)
  - apiGroups: ["batch"]
    resources: ["jobs"]
    verbs: ["get", "list", "watch", "create", "delete", "deletecollection"]
  # Need access to Litmus CRDs
  - apiGroups: ["litmuschaos.io"]
    resources: ["chaosengines", "chaosexperiments", "chaosresults"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
  # Need to read nodes (for node-level experiments)
  - apiGroups: [""]
    resources: ["nodes"]
    verbs: ["get", "list", "watch"]

---
# ─────────────────────────────────────────────
# Bind the role to the service account
# ─────────────────────────────────────────────
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: chaos-cluster-role-binding
  labels:
    app.kubernetes.io/part-of: litmus
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: chaos-cluster-role
subjects:
  - kind: ServiceAccount
    name: chaos-sa
    namespace: demo-app
```

```bash
# Apply the RBAC
kubectl apply -f chaos-rbac.yaml

# Verify
kubectl get sa chaos-sa -n demo-app
```

## 🚀 Step 5 — Experiment 1: Pod Delete

This is the most common chaos experiment. We'll randomly kill pods and see if Kubernetes brings them back.

### `experiment-pod-delete.yaml`

```yaml
# ─────────────────────────────────────────────
# ChaosEngine — This is what triggers an experiment
#
# Think of it like this:
#   ChaosExperiment = The recipe (what to do)
#   ChaosEngine     = The chef (who executes it)
# ─────────────────────────────────────────────
apiVersion: litmuschaos.io/v1alpha1
kind: ChaosEngine
metadata:
  name: nginx-pod-delete
  namespace: demo-app
spec:
  # What app are we targeting?
  appinfo:
    appns: demo-app               # Namespace of the target app
    applabel: "app=nginx-demo"     # Label selector — which pods to target
    appkind: deployment            # Type of workload

  # Should the experiment be annotated on the target?
  annotationCheck: "false"

  # What state should the engine be in?
  engineState: "active"            # "active" = run now, "stop" = don't run

  # What happens if chaos fails?
  chaosServiceAccount: chaos-sa    # ServiceAccount with permissions

  # The experiments to run
  experiments:
    - name: pod-delete
      spec:
        components:
          env:
            # ─── Experiment Tuning Parameters ───

            # How many pods to kill at once?
            - name: TOTAL_CHAOS_DURATION
              value: "30"          # Run chaos for 30 seconds

            # Time interval between pod deletions
            - name: CHAOS_INTERVAL
              value: "10"          # Kill a pod every 10 seconds

            # Should we force delete? (like kill -9)
            - name: FORCE
              value: "false"       # Graceful deletion

            # How many pods to kill? (as a percentage)
            - name: PODS_AFFECTED_PERC
              value: "50"          # Kill 50% of pods
```

**Let's understand each parameter:**

| Parameter | Value | Meaning |
|-----------|-------|---------|
| `TOTAL_CHAOS_DURATION` | `30` | The experiment runs for 30 seconds total |
| `CHAOS_INTERVAL` | `10` | Every 10 seconds, another pod gets killed |
| `FORCE` | `false` | Graceful shutdown (pod gets SIGTERM first, then SIGKILL after 30s) |
| `PODS_AFFECTED_PERC` | `50` | Kill 50% of the target pods (so 1-2 out of 3) |

### Run the Experiment

```bash
# Before chaos — check pods
echo "=== BEFORE CHAOS ==="
kubectl get pods -n demo-app

# Start the chaos!
kubectl apply -f experiment-pod-delete.yaml

# Watch what happens in real-time!
# Open a new terminal and run:
kubectl get pods -n demo-app -w
```

### What You'll See

```
=== BEFORE CHAOS ===
NAME                          READY   STATUS    RESTARTS   AGE
nginx-demo-abc123-def45       1/1     Running   0          10m
nginx-demo-abc123-ghi67       1/1     Running   0          10m
nginx-demo-abc123-jkl89       1/1     Running   0          10m

=== DURING CHAOS ===
nginx-demo-abc123-def45       1/1     Terminating   0          10m    ← Pod being killed!
nginx-demo-abc123-mnop1       0/1     Pending       0          0s     ← K8s creating new one
nginx-demo-abc123-mnop1       1/1     Running       0          5s     ← New pod is up! ✅

=== AFTER CHAOS ===
nginx-demo-abc123-ghi67       1/1     Running   0          11m
nginx-demo-abc123-jkl89       1/1     Running   0          11m
nginx-demo-abc123-mnop1       1/1     Running   0          30s    ← Replaced pod
```

**What happened?**
1. Litmus killed 50% of the pods (1-2 pods)
2. Kubernetes noticed the pod was gone
3. The ReplicaSet controller said "Hey, I need 3 replicas, but only 2 are running!"
4. Kubernetes created a new pod to replace the killed one
5. Service recovered automatically! ✅

### Check Chaos Results

```bash
# View the ChaosResult — did the experiment pass or fail?
kubectl get chaosresult -n demo-app

# Get detailed result
kubectl get chaosresult nginx-pod-delete-pod-delete -n demo-app -o yaml
```

**Expected output:**
```yaml
spec:
  experimentStatus:
    verdict: Pass        # ← The experiment passed!
    phase: Completed
    failStep: ""
```

A "Pass" verdict means: **your app survived the chaos.** The pods were killed, but Kubernetes recovered them successfully.

## 🚀 Step 6 — Experiment 2: Pod Network Latency

Now let's make things harder. Instead of killing pods, we'll inject **network latency** — making network responses slow.

### `experiment-network-latency.yaml`

```yaml
apiVersion: litmuschaos.io/v1alpha1
kind: ChaosEngine
metadata:
  name: nginx-network-latency
  namespace: demo-app
spec:
  appinfo:
    appns: demo-app
    applabel: "app=nginx-demo"
    appkind: deployment
  annotationCheck: "false"
  engineState: "active"
  chaosServiceAccount: chaos-sa

  experiments:
    - name: pod-network-latency
      spec:
        components:
          env:
            # How long to run the chaos
            - name: TOTAL_CHAOS_DURATION
              value: "60"          # 60 seconds of network chaos

            # How much latency to add (in milliseconds)
            - name: NETWORK_LATENCY
              value: "2000"        # Add 2000ms (2 seconds) of delay!

            # What percentage of pods to affect
            - name: PODS_AFFECTED_PERC
              value: "100"         # Affect ALL pods

            # Which network interface to target
            - name: NETWORK_INTERFACE
              value: "eth0"

            # Which destination IPs to affect (empty = all traffic)
            - name: DESTINATION_IPS
              value: ""

            # Container runtime
            - name: CONTAINER_RUNTIME
              value: "containerd"  # Use "docker" if using Docker runtime
            
            - name: SOCKET_PATH
              value: "/run/containerd/containerd.sock"
```

### Run and Observe

```bash
# Clean up previous experiment
kubectl delete chaosengine nginx-pod-delete -n demo-app

# Apply network latency experiment
kubectl apply -f experiment-network-latency.yaml

# Test the latency! From another terminal:
# Before chaos, response time should be ~1-5ms
# During chaos, it should be ~2000ms+

# Port forward the service
kubectl port-forward svc/nginx-demo-service 8080:80 -n demo-app &

# Time the response
time curl http://localhost:8080
```

### Expected Result

**Before chaos:**
```
real    0m0.015s    ← 15 milliseconds (fast!)
```

**During chaos:**
```
real    0m2.234s    ← 2+ seconds (slow because of injected latency!)
```

**After chaos ends:**
```
real    0m0.012s    ← Back to normal!
```

This is incredibly useful for testing:
- Does your app have proper **timeouts**?
- Does your app show a **loading spinner** when things are slow?
- Does your **circuit breaker** kick in?
- Do your **SLOs** detect the degradation?

## 🚀 Step 7 — Experiment 3: Pod CPU Hog

What happens when a pod's CPU goes to 100%? Does your app crash? Does Kubernetes throttle it? Let's find out!

### `experiment-cpu-hog.yaml`

```yaml
apiVersion: litmuschaos.io/v1alpha1
kind: ChaosEngine
metadata:
  name: nginx-cpu-hog
  namespace: demo-app
spec:
  appinfo:
    appns: demo-app
    applabel: "app=nginx-demo"
    appkind: deployment
  annotationCheck: "false"
  engineState: "active"
  chaosServiceAccount: chaos-sa

  experiments:
    - name: pod-cpu-hog
      spec:
        components:
          env:
            # How long to stress the CPU
            - name: TOTAL_CHAOS_DURATION
              value: "60"          # 60 seconds

            # Number of CPU cores to consume
            - name: CPU_CORES
              value: "1"           # Hog 1 CPU core

            # What percentage of pods to affect
            - name: PODS_AFFECTED_PERC
              value: "50"          # Stress 50% of pods

            # Container runtime
            - name: CONTAINER_RUNTIME
              value: "containerd"
            
            - name: SOCKET_PATH
              value: "/run/containerd/containerd.sock"
```

### Run and Monitor

```bash
# Clean up previous experiment
kubectl delete chaosengine nginx-network-latency -n demo-app

# Apply CPU hog experiment
kubectl apply -f experiment-cpu-hog.yaml

# Watch CPU usage in real-time
kubectl top pods -n demo-app --containers

# Or use kubectl describe to see events
kubectl describe pod -l app=nginx-demo -n demo-app
```

### What to Look For

```bash
# Before chaos:
NAME                      CPU(cores)   MEMORY(bytes)
nginx-demo-abc-def45      1m           5Mi         ← Normal: 1 millicore

# During chaos:
NAME                      CPU(cores)   MEMORY(bytes)
nginx-demo-abc-def45      100m         5Mi         ← Throttled at 100m (our limit!)
nginx-demo-abc-ghi67      1m           5Mi         ← Unaffected pod
```

**Key Insight:** Because we set `resources.limits.cpu: "100m"` in our deployment, Kubernetes **throttles** the container. It can't use more than 100 millicores even though the chaos experiment is trying to consume a full core. This is why **resource limits are so important!**

## 🚀 Step 8 — Experiment 4: Pod Memory Hog

### `experiment-memory-hog.yaml`

```yaml
apiVersion: litmuschaos.io/v1alpha1
kind: ChaosEngine
metadata:
  name: nginx-memory-hog
  namespace: demo-app
spec:
  appinfo:
    appns: demo-app
    applabel: "app=nginx-demo"
    appkind: deployment
  annotationCheck: "false"
  engineState: "active"
  chaosServiceAccount: chaos-sa

  experiments:
    - name: pod-memory-hog
      spec:
        components:
          env:
            - name: TOTAL_CHAOS_DURATION
              value: "60"

            # How much memory to consume (in MB)
            - name: MEMORY_CONSUMPTION
              value: "500"         # Try to consume 500MB

            - name: PODS_AFFECTED_PERC
              value: "30"          # Affect 30% of pods

            - name: CONTAINER_RUNTIME
              value: "containerd"
            
            - name: SOCKET_PATH
              value: "/run/containerd/containerd.sock"
```

### What Happens?

Our pods have a memory limit of `128Mi`. The experiment tries to consume `500MB`. This will trigger an **OOMKill** (Out of Memory Kill):

```
nginx-demo-abc-def45    0/1    OOMKilled    1    2m    ← Kubernetes killed it!
nginx-demo-abc-qrs12    0/1    Pending      0    0s    ← New pod starting
nginx-demo-abc-qrs12    1/1    Running      0    5s    ← Back to normal
```

**Key Learning:** Kubernetes kills pods that exceed their memory limits. The ReplicaSet controller then creates replacement pods. This is exactly how production systems should behave!

## 🚀 Step 9 — Create a Complete Chaos Workflow

Instead of running experiments one at a time, let's create a **workflow** that runs multiple experiments in sequence:

### `chaos-workflow.yaml`

```yaml
# ─────────────────────────────────────────────
# This is a complete chaos workflow that:
# 1. First kills pods (test self-healing)
# 2. Then adds network latency (test timeout handling)
# 3. Then stresses CPU (test resource limits)
#
# You can also define this through the ChaosCenter UI!
# ─────────────────────────────────────────────
apiVersion: argoproj.io/v1alpha1
kind: Workflow
metadata:
  name: nginx-chaos-workflow
  namespace: litmus
spec:
  entrypoint: chaos-workflow
  serviceAccountName: argo-chaos
  templates:
    # ─── Main Workflow ───
    - name: chaos-workflow
      steps:
        # Step 1: Verify app is healthy before starting
        - - name: check-app-health
            template: health-check

        # Step 2: Run pod-delete chaos
        - - name: pod-delete-chaos
            template: run-pod-delete

        # Step 3: Wait and verify recovery
        - - name: verify-recovery
            template: health-check

        # Step 4: Run network latency chaos
        - - name: network-latency-chaos
            template: run-network-latency

        # Step 5: Final health check
        - - name: final-health-check
            template: health-check

    # ─── Health Check Template ───
    - name: health-check
      container:
        image: curlimages/curl:latest
        command: ["/bin/sh", "-c"]
        args:
          - |
            echo "Checking application health..."
            for i in $(seq 1 10); do
              response=$(curl -s -o /dev/null -w "%{http_code}" http://nginx-demo-service.demo-app.svc.cluster.local:80 || echo "000")
              echo "Attempt $i: HTTP $response"
              if [ "$response" = "200" ]; then
                echo "✅ Application is healthy!"
                exit 0
              fi
              sleep 5
            done
            echo "❌ Application health check failed!"
            exit 1

    # ─── Pod Delete Template ───
    - name: run-pod-delete
      container:
        image: litmuschaos/litmus-checker:latest
        command: ["/bin/sh", "-c"]
        args:
          - |
            echo "🔥 Starting Pod Delete Chaos..."
            # The actual ChaosEngine would be applied here
            # For simplicity, we're using kubectl
            kubectl delete pod -l app=nginx-demo -n demo-app --wait=false
            echo "Pod deleted. Waiting 30 seconds for recovery..."
            sleep 30
            echo "✅ Pod Delete chaos completed!"

    # ─── Network Latency Template ───
    - name: run-network-latency
      container:
        image: litmuschaos/litmus-checker:latest
        command: ["/bin/sh", "-c"]
        args:
          - |
            echo "🌐 Starting Network Latency Chaos..."
            # Apply network latency experiment
            sleep 60
            echo "✅ Network Latency chaos completed!"
```

## 🚀 Step 11 — Chaos in CI/CD Pipeline

The real power of chaos engineering is running it **automatically** in your CI/CD pipeline:

### GitHub Actions Example

```yaml
# .github/workflows/chaos-test.yaml
name: Chaos Engineering Tests

on:
  # Run chaos tests on every PR to main
  pull_request:
    branches: [main]
  # Also run weekly (catch regressions)
  schedule:
    - cron: "0 6 * * 1"   # Every Monday at 6 AM

jobs:
  chaos-test:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Set up Kind cluster
        uses: helm/kind-action@v1.8.0
        with:
          cluster_name: chaos-ci

      - name: Install Litmus
        run: |
          helm repo add litmuschaos https://litmuschaos.github.io/litmus-helm/
          helm install litmus litmuschaos/litmus --namespace litmus --create-namespace
          kubectl wait --for=condition=ready pod -l app=litmus -n litmus --timeout=120s

      - name: Deploy Application
        run: |
          kubectl apply -f k8s/deployment.yaml
          kubectl wait --for=condition=ready pod -l app=nginx-demo -n demo-app --timeout=60s

      - name: Run Pod Delete Chaos
        run: |
          kubectl apply -f chaos/experiment-pod-delete.yaml
          # Wait for experiment to complete
          sleep 60
          # Check result
          VERDICT=$(kubectl get chaosresult -n demo-app -o jsonpath='{.items[0].spec.experimentStatus.verdict}')
          echo "Chaos Result: $VERDICT"
          if [ "$VERDICT" != "Pass" ]; then
            echo "❌ Chaos experiment FAILED! Application is not resilient."
            exit 1
          fi
          echo "✅ Chaos experiment PASSED! Application is resilient."

      - name: Run Network Latency Chaos
        run: |
          kubectl apply -f chaos/experiment-network-latency.yaml
          sleep 90
          VERDICT=$(kubectl get chaosresult -n demo-app -o jsonpath='{.items[0].spec.experimentStatus.verdict}')
          if [ "$VERDICT" != "Pass" ]; then
            echo "❌ Network latency test FAILED!"
            exit 1
          fi
          echo "✅ Network latency test PASSED!"

      - name: Cleanup
        if: always()
        run: |
          kind delete cluster --name chaos-ci
```

**This means:** Every time someone creates a PR, the CI pipeline:
1. Creates a test Kubernetes cluster
2. Deploys the app
3. Runs chaos experiments
4. If the app doesn't recover → **PR is blocked** ❌
5. If the app recovers → **PR can be merged** ✅

Here's a quick reference of all the experiments we covered and more:

| Experiment | What It Does | Use Case |

## 🧠 Key Takeaways

| Lesson | Detail |
|--------|--------|
| **Chaos is not random destruction** | It's controlled, measured, and has clear hypotheses |
| **Start small** | Begin with pod-delete in staging, not network partition in production |
| **Always have a rollback plan** | Know how to stop the experiment instantly |
| **Monitor everything** | Chaos without observability is just breaking things |
| **Automate chaos** | Run it in CI/CD to catch regressions automatically |
| **Resource limits matter** | They protect your cluster from runaway processes |
| **Replicas save you** | Multiple replicas = your app survives pod failures |
| **Chaos builds confidence** | If your system passes chaos tests, you can sleep at night |

## ⚠️ Important Safety Rules

1. **NEVER run chaos in production** without approval and a rollback plan
2. **Start in dev/staging** — prove your experiments work before going to production
3. **Always monitor** — run chaos experiments while watching your dashboards
4. **Communicate** — tell your team before running chaos experiments
5. **Small blast radius** — start with 1 pod, not the entire cluster
6. **Have a kill switch** — know how to stop chaos instantly: `kubectl delete chaosengine <name> -n <namespace>`
7. **Document everything** — record hypotheses, results, and learnings
