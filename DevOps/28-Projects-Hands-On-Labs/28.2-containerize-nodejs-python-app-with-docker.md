## 🎯 What Are We Building?

We're going to take a simple application (we'll build both a **Node.js** and a **Python** version — pick your favorite!) and **containerize** it using Docker.

By the end of this project, your application will run inside a Docker container, completely isolated from your host machine. You'll be able to share it with anyone, and it'll work **exactly the same** on their machine. No more "it works on my machine" problems!

### What Does "Containerize" Mean?

**Containerizing** an app means packaging it with **everything it needs** to run — code, runtime, libraries, dependencies — into a single, portable unit called a **container**.

Think of it like packing a lunchbox 🍱:
- Without Docker: "Here's the recipe, go buy ingredients, cook it yourself" (hope you have the right kitchen!)
- With Docker: "Here's the complete meal, ready to eat" (works in ANY kitchen!)

```
Before Containers:
  "Hey, can you run my app?"
  "Sure... wait, I need Node.js 18, not 16"
  "I also need this specific library version"
  "Why is it crashing? Oh, my OS is different"
  😡 Hours wasted on setup

With Containers:
  "Hey, can you run my app?"
  "docker run myapp"
  "It's running! That took 3 seconds"
  🎉 Everyone is happy
```

## 🟢 Option A: Containerize a Node.js Application

Let's build a simple Node.js web API and containerize it.

### Step 1: Create the Project

```bash
# Create project directory
mkdir my-docker-app-node
cd my-docker-app-node
```

### Step 2: Initialize Node.js Project

```bash
# Create package.json
npm init -y
```

This creates a `package.json` file — it's the configuration file for Node.js projects. It tracks your app's name, version, and dependencies (like a shopping list for your app).

### Step 3: Install Express

```bash
# Install Express (a lightweight web framework for Node.js)
npm install express
```

**What is Express?** Express is the most popular web framework for Node.js. It makes it super easy to create web servers and APIs. Think of it like the "Nginx" of Node.js — it handles HTTP requests and sends responses.

### Step 4: Create the Application

Create a file called `app.js`:

```javascript
// app.js — A simple Node.js web application

const express = require('express');
const os = require('os');

const app = express();

// Define the port — use environment variable or default to 3000
const PORT = process.env.PORT || 3000;

// Root route — the main page
app.get('/', (req, res) => {
    res.json({
        message: '🚀 Hello from Docker!',
        app: 'My Dockerized Node.js App',
        version: '1.0.0',
        hostname: os.hostname(),     // Shows the container ID
        platform: os.platform(),      // Shows 'linux' (inside container)
        uptime: `${Math.floor(process.uptime())} seconds`,
        timestamp: new Date().toISOString()
    });
});

// Health check endpoint — very important in production!
app.get('/health', (req, res) => {
    res.json({
        status: 'healthy',
        uptime: `${Math.floor(process.uptime())} seconds`,
        timestamp: new Date().toISOString()
    });
});

// Environment info route — shows environment variables
app.get('/env', (req, res) => {
    res.json({
        nodeVersion: process.version,
        environment: process.env.NODE_ENV || 'development',
        port: PORT
    });
});

// Start the server
app.listen(PORT, '0.0.0.0', () => {
    console.log(`✅ Server is running on port ${PORT}`);
    console.log(`📍 Open http://localhost:${PORT} in your browser`);
    console.log(`🐳 Running inside container: ${os.hostname()}`);
});
```

**Let's understand the code:**

| Part | What It Does |
|------|-------------|
| `require('express')` | Imports the Express framework |
| `process.env.PORT \|\| 3000` | Uses the PORT environment variable if set, otherwise defaults to 3000. This is important for containers! |
| `app.get('/', ...)` | When someone visits the root URL (`/`), send back a JSON response with app info |
| `app.get('/health', ...)` | A health check endpoint — tools like Kubernetes use this to check if your app is alive |
| `os.hostname()` | Returns the hostname — inside a container, this is the **container ID** (a cool way to verify you're in Docker!) |
| `app.listen(PORT, '0.0.0.0', ...)` | Start listening on all network interfaces (`0.0.0.0`). This is important inside containers! |

**Why `0.0.0.0` and not `localhost`?**

Inside a container, `localhost` refers to the container itself. If you listen on `localhost`, only processes **inside the container** can reach your app. By listening on `0.0.0.0`, you're saying "accept connections from ANY network interface" — which means connections from outside the container (like your browser) can reach it.

### Step 5: Test Locally (Optional)

```bash
# Run the app locally first (without Docker)
node app.js

# Visit http://localhost:3000 in your browser
# Press Ctrl+C to stop
```

### Step 6: Create the Dockerfile

Now for the main event — the **Dockerfile**! This is the recipe that tells Docker how to build your container image.

Create a file called `Dockerfile` (no extension!):

```dockerfile
# ============================================
# Dockerfile for Node.js Application
# ============================================

# Stage 1: Use the official Node.js image as the base
# "alpine" is a tiny Linux distribution (~5MB vs ~900MB for Ubuntu)
# This keeps our image small and secure
FROM node:18-alpine

# Set metadata labels (optional but good practice)
LABEL maintainer="your-email@example.com"
LABEL description="A simple Dockerized Node.js application"
LABEL version="1.0.0"

# Set the working directory INSIDE the container
# All following commands (COPY, RUN, CMD) will run from this directory
# If the directory doesn't exist, Docker creates it automatically
WORKDIR /app

# Copy package.json and package-lock.json FIRST
# Why? Docker caches layers. If these files haven't changed,
# Docker won't re-run "npm install" — saving LOTS of time!
COPY package*.json ./

# Install dependencies
# --only=production: Don't install devDependencies (we don't need them in production)
# This keeps the image smaller
RUN npm install --only=production

# NOW copy the rest of the application code
# We do this AFTER npm install to take advantage of Docker's layer caching
COPY . .

# Create a non-root user for security
# Running as root inside a container is a security risk!
RUN addgroup -S appgroup && adduser -S appuser -G appgroup

# Switch to the non-root user
USER appuser

# Tell Docker this container will listen on port 3000
# This is documentation — it doesn't actually publish the port
EXPOSE 3000

# Set environment variables
ENV NODE_ENV=production

# The command to run when the container starts
# Use array syntax (exec form) — it's more reliable than shell form
CMD ["node", "app.js"]
```

**Let's break down every Dockerfile instruction:**

| Instruction | What It Does | Why It Matters |
|-------------|-------------|----------------|
| `FROM node:18-alpine` | Starts with a Node.js 18 image on Alpine Linux | Alpine is tiny (~5MB). Smaller image = faster builds, faster deploys, less attack surface |
| `WORKDIR /app` | Sets the working directory to `/app` inside the container | Keeps things organized. All commands run from here |
| `COPY package*.json ./` | Copies `package.json` and `package-lock.json` first | **Layer caching trick!** Dependencies change rarely, code changes often. By copying package files first, Docker caches the `npm install` layer |
| `RUN npm install` | Installs Node.js dependencies | Builds the `node_modules` folder inside the container |
| `COPY . .` | Copies ALL your application code into the container | This is the layer that changes most often, so it's last |
| `RUN addgroup... adduser...` | Creates a non-root user | **Security!** Never run apps as root in containers |
| `USER appuser` | Switches to the non-root user | All subsequent commands (including CMD) run as this user |
| `EXPOSE 3000` | Documents that the app listens on port 3000 | It's just metadata — you still need `-p` when running |
| `ENV NODE_ENV=production` | Sets an environment variable | Tells Node.js to run in production mode (disables debugging, enables optimizations) |
| `CMD ["node", "app.js"]` | The command that runs when the container starts | This is what keeps the container alive |

### Step 7: Create .dockerignore

Just like `.gitignore` tells Git what to ignore, `.dockerignore` tells Docker what NOT to copy into the image.

Create a file called `.dockerignore`:

```
# Don't copy these into the Docker image
node_modules          # We install fresh inside the container
npm-debug.log         # Debug logs
.git                  # Git history (not needed in container)
.gitignore            # Not needed in container
.dockerignore         # Not needed in container
Dockerfile            # The recipe doesn't need to be in the meal!
README.md             # Documentation not needed at runtime
.env                  # Environment files (might contain secrets!)
.vscode               # Editor settings
*.md                  # Markdown files
```

**Why ignore `node_modules`?** Because:
1. They might be built for YOUR machine's OS (e.g., Windows), but the container runs Linux
2. We run `npm install` inside the container, which builds them for the right platform
3. It makes the build context smaller and the build faster

### Step 8: Build the Docker Image

Now let's build our image!

```bash
# Build the Docker image
# -t = "tag" — give our image a name
# . = "build context" — use the current directory
docker build -t my-node-app:1.0 .
```

**What's happening during the build?**

```
Step 1/10 : FROM node:18-alpine          ← Downloads the base image
Step 2/10 : WORKDIR /app                 ← Creates /app directory
Step 3/10 : COPY package*.json ./        ← Copies package files
Step 4/10 : RUN npm install              ← Installs dependencies
Step 5/10 : COPY . .                     ← Copies your code
Step 6/10 : RUN addgroup...              ← Creates non-root user
Step 7/10 : USER appuser                 ← Switches user
Step 8/10 : EXPOSE 3000                  ← Documents port
Step 9/10 : ENV NODE_ENV=production      ← Sets env var
Step 10/10: CMD ["node", "app.js"]       ← Sets start command
```

**Verify the image was created:**

```bash
docker images

# Output:
# REPOSITORY      TAG       IMAGE ID       CREATED          SIZE
# my-node-app     1.0       abc123def456   10 seconds ago   120MB
```

Notice the size — only ~120MB! That's because we used Alpine Linux. If we had used the full `node:18` image, it would be ~900MB.

### Step 9: Run the Container

Let's run it!

```bash
# Run the container
docker run -d -p 3000:3000 --name my-app my-node-app:1.0
```

**Let's understand every flag:**

| Flag | What It Does |
|------|-------------|
| `-d` | **Detached mode** — runs in the background (you get your terminal back) |
| `-p 3000:3000` | **Port mapping** — maps port 3000 on your machine to port 3000 in the container. Format: `host_port:container_port` |
| `--name my-app` | Gives the container a friendly name instead of a random one |
| `my-node-app:1.0` | The image to create the container from |

**Port Mapping Explained:**

```
Your Computer                        Docker Container
┌──────────────┐                    ┌──────────────┐
│              │    -p 3000:3000    │              │
│  Port 3000   │ ←──────────────→  │  Port 3000   │
│  (host)      │                    │  (container) │
│              │                    │  Node.js app │
└──────────────┘                    └──────────────┘

When you visit http://localhost:3000:
1. Your browser sends a request to localhost:3000 (your machine's port 3000)
2. Docker forwards it to the container's port 3000
3. Your Node.js app inside the container handles it
4. Response comes back the same way
```

You could also map to a different host port:

```bash
# Map host port 8080 to container port 3000
docker run -d -p 8080:3000 --name my-app my-node-app:1.0
# Now visit http://localhost:8080
```

Open your browser and visit:

## 🐍 Option B: Containerize a Python Application

Prefer Python? Let's do the same thing with Python and Flask!

### Step 1: Create the Project

```bash
mkdir my-docker-app-python
cd my-docker-app-python
```

### Step 2: Create requirements.txt

```
flask==3.0.0
gunicorn==21.2.0
```

**What are these?**

| Package | What It Does |
|---------|-------------|
| `flask` | A lightweight Python web framework (like Express for Python) |
| `gunicorn` | A production-grade WSGI server for Python. Flask's built-in server is only for development — gunicorn is what you use in production |

### Step 3: Create the Application

Create a file called `app.py`:

```python
# app.py — A simple Python Flask application

import os
import socket
import datetime
from flask import Flask, jsonify

app = Flask(__name__)

# Root route — the main page
@app.route('/')
def home():
    return jsonify({
        'message': '🚀 Hello from Docker!',
        'app': 'My Dockerized Python App',
        'version': '1.0.0',
        'hostname': socket.gethostname(),      # Container ID
        'platform': os.name,                    # 'posix' on Linux
        'python_version': os.popen('python --version').read().strip(),
        'timestamp': datetime.datetime.now().isoformat()
    })

# Health check endpoint
@app.route('/health')
def health():
    return jsonify({
        'status': 'healthy',
        'timestamp': datetime.datetime.now().isoformat()
    })

# Environment info
@app.route('/env')
def env_info():
    return jsonify({
        'environment': os.environ.get('FLASK_ENV', 'production'),
        'debug': app.debug,
        'port': os.environ.get('PORT', '5000')
    })

if __name__ == '__main__':
    port = int(os.environ.get('PORT', 5000))
    app.run(host='0.0.0.0', port=port)
```

**Code explanation:**

| Part | What It Does |
|------|-------------|
| `Flask(__name__)` | Creates a Flask application |
| `@app.route('/')` | Decorator that maps the `/` URL to the function below it |
| `jsonify(...)` | Returns a JSON response (like Express's `res.json()`) |
| `socket.gethostname()` | Returns the hostname — inside a container, this is the container ID |
| `host='0.0.0.0'` | Listen on all interfaces (same reason as Node.js — so traffic from outside the container can reach it) |

### Step 4: Create the Dockerfile

```dockerfile
# ============================================
# Dockerfile for Python Flask Application
# ============================================

# Use Python 3.11 on Alpine Linux (small and secure)
FROM python:3.11-alpine

# Set metadata
LABEL maintainer="your-email@example.com"
LABEL description="A simple Dockerized Python Flask application"

# Set working directory
WORKDIR /app

# Copy requirements first (layer caching — same trick as Node.js!)
COPY requirements.txt .

# Install Python dependencies
# --no-cache-dir: Don't store pip's cache (saves space)
# --no-compile: Don't compile .pyc files (saves space)
RUN pip install --no-cache-dir -r requirements.txt

# Copy the rest of the application
COPY . .

# Create a non-root user (security best practice)
RUN addgroup -S appgroup && adduser -S appuser -G appgroup

# Switch to non-root user
USER appuser

# Document the port
EXPOSE 5000

# Set environment variables
ENV FLASK_ENV=production
ENV PORT=5000

# Run with gunicorn (production server) instead of Flask's dev server
# --bind 0.0.0.0:5000 — Listen on all interfaces, port 5000
# --workers 2 — Use 2 worker processes (good for a small app)
# --access-logfile - — Print access logs to stdout (Docker captures stdout)
CMD ["gunicorn", "--bind", "0.0.0.0:5000", "--workers", "2", "--access-logfile", "-", "app:app"]
```

**Why use gunicorn instead of Flask's built-in server?**

Flask's built-in server (what you get with `app.run()`) is designed for **development only**. It:
- Handles one request at a time
- Isn't secure
- Isn't optimized for performance

Gunicorn is a **production-grade** server that:
- Handles multiple requests simultaneously (multiple workers)
- Is battle-tested and secure
- Is the standard way to deploy Flask apps in production

## 🔧 Essential Docker Commands — Your Cheat Sheet

Now that your app is running, here are the commands you'll use every day:

### Container Management

```bash
# List running containers
docker ps

# List ALL containers (including stopped ones)
docker ps -a

# Stop a container
docker stop my-app

# Start a stopped container
docker start my-app

# Restart a container
docker restart my-app

# Remove a container (must be stopped first)
docker rm my-app

# Force remove a running container
docker rm -f my-app

# View container logs
docker logs my-app

# Follow logs in real-time (like tail -f)
docker logs -f my-app

# View last 50 lines of logs
docker logs --tail 50 my-app
```

### Debugging Containers

```bash
# Get inside a running container (open a shell)
docker exec -it my-app /bin/sh

# For Alpine images, use sh (not bash — Alpine doesn't have bash by default)
docker exec -it my-app sh

# Run a specific command inside the container
docker exec my-app ls /app

# Inspect container details (JSON output with EVERYTHING about the container)
docker inspect my-app

# View resource usage (CPU, memory, network)
docker stats my-app

# View the processes running inside the container
docker top my-app
```

### Image Management

```bash
# List all images
docker images

# Remove an image
docker rmi my-node-app:1.0

# Remove all unused images (dangling images)
docker image prune

# View image layers and history
docker history my-node-app:1.0

# Get detailed info about an image
docker inspect my-node-app:1.0
```

## 🔍 Understanding Docker Layers & Caching

One of the most important concepts in Docker is **layer caching**. Let's understand it.

Every instruction in your Dockerfile creates a **layer**. Docker **caches** these layers. When you rebuild, Docker checks: "Has anything changed in this layer?" If not, it uses the cached version.

```
Dockerfile                     Layers (cached)
──────────                     ────────────────
FROM node:18-alpine        →  Layer 1: Base image     ✅ Cached (rarely changes)
WORKDIR /app               →  Layer 2: Work dir       ✅ Cached (never changes)
COPY package*.json ./      →  Layer 3: Package files  ✅ Cached (changes rarely)
RUN npm install            →  Layer 4: Dependencies   ✅ Cached (changes rarely)
COPY . .                   →  Layer 5: App code       ❌ REBUILDS (changes often)
```

**The trick:** We copy `package.json` and run `npm install` BEFORE copying the rest of the code. This way, if you only change your application code (which happens constantly), Docker reuses the cached `npm install` layer. Without this trick, every code change would trigger a full `npm install`, making builds slow.

```
Without the trick: Change 1 line of code → Rebuild takes 2 minutes (npm install runs again)
With the trick:    Change 1 line of code → Rebuild takes 5 seconds (npm install is cached!)
```

## 🏋️ Using Environment Variables

Environment variables let you configure your app **without changing the code**. Super important for containers because the same image might run in dev, staging, and production.

```bash
# Pass environment variables when running a container
docker run -d \
  -p 3000:3000 \
  -e PORT=3000 \
  -e NODE_ENV=production \
  -e APP_NAME="My Cool App" \
  --name my-app \
  my-node-app:1.0
```

Inside your app, you read them with:
- **Node.js:** `process.env.APP_NAME`
- **Python:** `os.environ.get('APP_NAME')`

**Common environment variables in containers:**

| Variable | Purpose |
|----------|---------|
| `PORT` | What port the app should listen on |
| `NODE_ENV` / `FLASK_ENV` | Production or development mode |
| `DATABASE_URL` | Database connection string |
| `API_KEY` | API keys (careful with security!) |
| `LOG_LEVEL` | How verbose the logs should be |

## 📝 Dockerfile Best Practices Summary

Here's a quick reference of all the best practices we followed:

| Practice | Why |
|----------|-----|
| Use Alpine-based images | Smaller size (~5MB vs ~900MB), less attack surface |
| Copy dependency files first | Layer caching — `npm install` / `pip install` doesn't re-run unless dependencies change |
| Use `.dockerignore` | Keeps build context small and fast, avoids copying unnecessary files |
| Run as non-root user | Security — limits what a compromised container can do |
| Use `EXPOSE` | Documents which port your app uses |
| Use exec form for `CMD` | `CMD ["node", "app.js"]` is more reliable than `CMD node app.js` |
| Pin dependency versions | `flask==3.0.0` not just `flask` — ensures reproducible builds |
| Use `--no-cache-dir` for pip | Saves space by not caching downloaded packages |
| Use production servers | Gunicorn for Python, not Flask dev server |
| Set `NODE_ENV=production` | Optimizes Node.js for production |

## 🔍 Verification & Debugging

### Check if your container is running:

```bash
docker ps
```

### View logs:

```bash
# View all logs
docker logs my-app

# Follow logs in real-time
docker logs -f my-app
```

### Get inside the container:

```bash
# Open a shell inside the container
docker exec -it my-app sh

# Look around
ls /app
cat /app/app.js
whoami           # Should show "appuser", not "root"

# Exit the container shell
exit
```

### Common Issues:

| Problem | Cause | Fix |
|---------|-------|-----|
| Container exits immediately | App crashes on startup | Check logs: `docker logs my-app` |
| Port already in use | Another process using port 3000 | Use a different port: `-p 3001:3000` |
| `COPY failed: file not found` | File doesn't exist or is in `.dockerignore` | Check your `.dockerignore` and file paths |
| `npm install` fails | Network issues or wrong Node version | Check `FROM` image version and try again |
| Image too large | Not using Alpine, not ignoring `node_modules` | Use Alpine images and proper `.dockerignore` |
| Permission denied | Running as wrong user | Check `USER` directive and file permissions |

## 🎓 Key Concepts Recap

| Concept | What You Learned |
|---------|-----------------|
| **Docker Image** | A read-only template containing your app + everything it needs. Created from a Dockerfile |
| **Docker Container** | A running instance of an image. You can run multiple containers from the same image |
| **Dockerfile** | The recipe/blueprint for building an image. Each instruction creates a layer |
| **Layer Caching** | Docker reuses unchanged layers to speed up builds. Order your Dockerfile instructions wisely! |
| **Port Mapping** | `-p host:container` connects your machine's port to the container's port |
| **Non-root User** | Security best practice — never run as root in containers |
| **.dockerignore** | Tells Docker which files to skip when building |
| **Docker Hub** | A registry for storing and sharing Docker images |
| **Environment Variables** | Configure your app without changing code — essential for containers |
| **exec form CMD** | `CMD ["node", "app.js"]` — the recommended way to set the container's start command |
