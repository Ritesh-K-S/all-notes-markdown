## 🤔 Why Do We Need This?

Let's start with a scary but common story.

A company builds an amazing CI/CD pipeline:

```
Developer pushes code → Build → Test → Deploy to Production ✅

Everything looks great! Fast deployments! Happy developers!

But then... 💥

One day, a hacker finds:
  ❌ A vulnerable npm package (Log4Shell-style attack)
  ❌ Hardcoded API keys in the code
  ❌ A Docker image running as root
  ❌ An open API endpoint with no authentication
  ❌ A misconfigured Kubernetes deployment with privileged containers

The company gets breached. Customer data is stolen. 😱
```

**The problem?** Security was an **afterthought**. The pipeline was fast, but it had **zero security gates**. Anything could get deployed to production — including vulnerabilities.

**DevSecOps = Security is NOT a separate phase. It's integrated into EVERY phase.**

```
Traditional Pipeline:
  Code → Build → Test → Deploy → ... → Security Audit (way too late!) ❌

DevSecOps Pipeline:
  Code → [Security] → Build → [Security] → Test → [Security] → Deploy → [Security] ✅
       ↑                     ↑                    ↑                     ↑
    Pre-commit            Image scan          DAST scan          Runtime security
    SAST scan             SBOM generation     API testing        Monitoring
    Secret detection      Signing             Compliance         Alerting
```

## 🎯 What We're Building

By the end of this project, you'll have a pipeline with **8 security gates** that catch vulnerabilities at every stage:

| Gate # | Stage | What It Catches | Tool |
|--------|-------|----------------|------|
| 1 | **Pre-Commit** | Secrets, credentials, API keys in code | git-secrets, detect-secrets |
| 2 | **SAST** | Code vulnerabilities, bugs, code smells | SonarQube, Semgrep |
| 3 | **SCA** | Vulnerable dependencies (npm, pip, etc.) | Trivy, Snyk |
| 4 | **Container Scan** | Vulnerable base images, misconfigurations | Trivy, Grype |
| 5 | **Image Signing** | Ensures images haven't been tampered with | Cosign (Sigstore) |

| 7 | **DAST** | Runtime vulnerabilities, API security issues | OWASP ZAP |
| 8 | **Admission Control** | Block non-compliant deployments to K8s | OPA Gatekeeper, Kyverno |

### Gate 1: Pre-Commit Security Hooks (Catch Secrets Before They Leave Your Machine)

**The first line of defense.** Before code even reaches Git, we check for secrets, credentials, and common security mistakes.

**Why is this critical?** Once a secret is pushed to Git, it's in the history **forever** (even if you delete it). GitHub has bots that scan public repos for secrets — and attackers use them within **minutes**.

#### Install pre-commit framework:

```bash
# Install pre-commit (Python-based)
pip install pre-commit

# Install detect-secrets (by Yelp)
pip install detect-secrets

# Install git-secrets (by AWS)
# Linux:
git clone https://github.com/awslabs/git-secrets.git
cd git-secrets
sudo make install

# Mac:
brew install git-secrets
```

#### Create pre-commit configuration:

```yaml
# .pre-commit-config.yaml (put this in your project root)

repos:
  # ═══════════════════════════════════════════
  # Secret Detection — Catch hardcoded credentials
  # ═══════════════════════════════════════════
  - repo: https://github.com/Yelp/detect-secrets
    rev: v1.4.0
    hooks:
      - id: detect-secrets
        args: ['--baseline', '.secrets.baseline']
        # This scans ALL staged files for patterns that look like:
        # - API keys (long random strings)
        # - Passwords in config files
        # - AWS access keys (AKIA...)
        # - Private keys (-----BEGIN RSA PRIVATE KEY-----)

  # ═══════════════════════════════════════════
  # Git-secrets — AWS-specific credential scanning
  # ═══════════════════════════════════════════
  - repo: https://github.com/awslabs/git-secrets
    rev: master
    hooks:
      - id: git-secrets
        # Catches: AWS access keys, secret keys, account IDs

  # ═══════════════════════════════════════════
  # General Code Quality Hooks
  # ═══════════════════════════════════════════
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v4.5.0
    hooks:
      - id: check-yaml          # Validate YAML syntax
      - id: check-json          # Validate JSON syntax
      - id: check-merge-conflict # Detect unresolved merge conflicts
      - id: detect-private-key   # Catch private keys
      - id: end-of-file-fixer   # Ensure files end with newline
      - id: trailing-whitespace # Remove trailing whitespace
      - id: no-commit-to-branch # Prevent direct commits to main
        args: ['--branch', 'main', '--branch', 'master']

  # ═══════════════════════════════════════════
  # Dockerfile Linting
  # ═══════════════════════════════════════════
  - repo: https://github.com/hadolint/hadolint
    rev: v2.12.0
    hooks:
      - id: hadolint
        # Catches: Running as root, using ADD instead of COPY,
        # not pinning package versions, etc.

  # ═══════════════════════════════════════════
  # Kubernetes Manifest Linting
  # ═══════════════════════════════════════════
  - repo: https://github.com/stackrox/kube-linter
    rev: v0.6.5
    hooks:
      - id: kube-linter
        # Catches: No resource limits, running as root,
        # no health checks, writable root filesystem
```

#### Set it up in your project:

```bash
# Initialize pre-commit in your repo
cd your-project
pre-commit install

# Create a secrets baseline (whitelist known false positives)
detect-secrets scan > .secrets.baseline

# Test it! Try committing a file with a fake secret:
echo 'AWS_SECRET_KEY=wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY' > test-secret.txt
git add test-secret.txt
git commit -m "test"  # ← This will be BLOCKED! 🛑
```

**What happens when you try to commit secrets:**

```
$ git commit -m "Add config"
Detect Secrets...................................................Failed
- hook id: detect-secrets
- exit code: 1

ERROR: Potential secret detected in config.py:
  Line 42: API_KEY = "sk-proj-abc123xyz456..."

  Please remove the secret and use environment variables instead.
  If this is a false positive, update .secrets.baseline

Commit BLOCKED! 🛑
```

**The developer is forced to fix it before they can commit.** The secret NEVER reaches Git.

### Gate 2: SAST — Static Application Security Testing

**SAST scans your source code** (without running it) to find security vulnerabilities, bugs, and code quality issues.

Think of it as a **super-powered code review** that catches things humans miss.

#### Set up SonarQube:

```bash
# Run SonarQube using Docker Compose
cat <<'EOF' > docker-compose.sonarqube.yaml
version: '3.8'
services:
  sonarqube:
    image: sonarqube:community
    container_name: sonarqube
    ports:
      - "9000:9000"
    environment:
      - SONAR_ES_BOOTSTRAP_CHECKS_DISABLE=true
    volumes:
      - sonarqube_data:/opt/sonarqube/data
      - sonarqube_extensions:/opt/sonarqube/extensions
      - sonarqube_logs:/opt/sonarqube/logs
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:9000/api/system/status"]
      interval: 30s
      timeout: 10s
      retries: 5

volumes:
  sonarqube_data:
  sonarqube_extensions:
  sonarqube_logs:
EOF

docker compose -f docker-compose.sonarqube.yaml up -d
```

After it starts (takes ~2 minutes), open `http://localhost:9000`:
- Default login: `admin` / `admin`
- You'll be asked to change the password on first login

#### Create a SonarQube project configuration:

```properties
# sonar-project.properties (put this in your project root)

sonar.projectKey=my-app
sonar.projectName=My Application
sonar.projectVersion=1.0

# Source code location
sonar.sources=src
sonar.tests=tests

# Language-specific settings
sonar.javascript.lcov.reportPaths=coverage/lcov.info

# Exclude test files and node_modules
sonar.exclusions=**/node_modules/**,**/test/**,**/coverage/**

# Quality gate settings
sonar.qualitygate.wait=true  # Fail the build if quality gate fails
```

#### Also set up Semgrep (lightweight, fast, and free):

```yaml
# .semgrep.yml (custom rules for your project)
rules:
  # Rule 1: Detect hardcoded passwords
  - id: hardcoded-password
    patterns:
      - pattern: |
          password = "..."
      - pattern: |
          passwd = "..."
      - pattern: |
          pwd = "..."
    message: "Hardcoded password detected! Use environment variables instead."
    severity: ERROR
    languages: [javascript, typescript, python]

  # Rule 2: Detect SQL injection
  - id: sql-injection
    patterns:
      - pattern: |
          $QUERY = "SELECT ... " + $INPUT
      - pattern: |
          $QUERY = `SELECT ... ${$INPUT}`
    message: "Potential SQL injection! Use parameterized queries."
    severity: ERROR
    languages: [javascript, typescript]

  # Rule 3: Detect insecure HTTP usage
  - id: insecure-http
    pattern: |
      http://...
    message: "Use HTTPS instead of HTTP for security."
    severity: WARNING
    languages: [javascript, typescript, python, yaml]
    paths:
      exclude:
        - "*.test.*"
        - localhost
```

**How SAST works in the pipeline:**

```
Developer pushes code
         ↓
SAST Scanner analyzes the source code (without running it)
         ↓
Checks for:
  ├── SQL Injection          → "You're concatenating user input in SQL!"
  ├── XSS (Cross-Site Scripting) → "You're not escaping HTML output!"
  ├── Hardcoded Credentials  → "Is that a password on line 42?"
  ├── Insecure Crypto        → "MD5 is not secure for passwords!"
  ├── Path Traversal         → "User can access files outside the app!"
  ├── Code Smells            → "This function is 500 lines long..."
  └── Bug Patterns           → "This variable is always null here"
         ↓
If CRITICAL or HIGH issues found → Pipeline FAILS 🛑
If only LOW/INFO issues → Pipeline passes with warnings ⚠️
```

### Gate 3: SCA — Software Composition Analysis

**Your code is only as secure as the libraries you use.** SCA scans all your dependencies (npm packages, pip packages, etc.) for known vulnerabilities.

Remember **Log4Shell**? That was a vulnerability in a widely-used Java library. SCA would have caught it.

#### Using Trivy for dependency scanning:

```bash
# Install Trivy
# Linux:
curl -sfL https://raw.githubusercontent.com/aquasecurity/trivy/main/contrib/install.sh | sh -s -- -b /usr/local/bin

# Mac:
brew install trivy

# Scan your project's dependencies
trivy fs --scanners vuln --severity HIGH,CRITICAL .
```

**Example output:**

```
$ trivy fs --scanners vuln --severity HIGH,CRITICAL .

package-lock.json (npm)
========================
Total: 3 (HIGH: 2, CRITICAL: 1)

┌──────────────────┬────────────────┬──────────┬─────────────────────┬──────────────────────┐
│     Library      │ Vulnerability  │ Severity │ Installed Version   │   Fixed Version      │
├──────────────────┼────────────────┼──────────┼─────────────────────┼──────────────────────┤
│ express          │ CVE-2024-1234  │ HIGH     │ 4.17.1              │ 4.18.2               │
│ jsonwebtoken     │ CVE-2024-5678  │ CRITICAL │ 8.5.1               │ 9.0.0                │
│ lodash           │ CVE-2024-9012  │ HIGH     │ 4.17.20             │ 4.17.21              │
└──────────────────┴────────────────┴──────────┴─────────────────────┴──────────────────────┘
```

**What this tells us:**
- `express` version 4.17.1 has a HIGH vulnerability → Update to 4.18.2
- `jsonwebtoken` version 8.5.1 has a CRITICAL vulnerability → Update to 9.0.0
- `lodash` version 4.17.20 has a HIGH vulnerability → Update to 4.17.21

#### Generate an SBOM (Software Bill of Materials):

An SBOM is like an "ingredients list" for your software. It tells you **every single package** your app uses, so if a new vulnerability is discovered, you can instantly check if you're affected.

```bash
# Generate SBOM in CycloneDX format
trivy fs --format cyclonedx --output sbom.json .

# Generate SBOM in SPDX format
trivy fs --format spdx-json --output sbom-spdx.json .
```

**Why SBOM matters:**

```
Without SBOM:
  New vulnerability announced → "Do we use this library? 🤷"
  → Manual search through 200 repos → Takes days

With SBOM:
  New vulnerability announced → Search SBOM database → "Yes, 3 services use it" 
  → Fix in minutes ✅
```

### Gate 4: Container Image Scanning

Even if your code is secure, your **Docker base image** might have vulnerabilities. This gate scans the final built image.

#### Scan Docker images with Trivy:

```bash
# Build your Docker image
docker build -t my-app:latest .

# Scan it
trivy image --severity HIGH,CRITICAL my-app:latest
```

**Example output:**

```
$ trivy image --severity HIGH,CRITICAL my-app:latest

my-app:latest (alpine 3.18.4)
=============================
Total: 5 (HIGH: 3, CRITICAL: 2)

┌──────────────┬────────────────┬──────────┬────────────────┬────────────────┐
│   Library    │ Vulnerability  │ Severity │    Installed   │     Fixed      │
├──────────────┼────────────────┼──────────┼────────────────┼────────────────┤
│ openssl      │ CVE-2024-1111  │ CRITICAL │ 3.1.2-r0       │ 3.1.4-r0       │
│ curl         │ CVE-2024-2222  │ CRITICAL │ 8.4.0-r0       │ 8.5.0-r0       │
│ zlib         │ CVE-2024-3333  │ HIGH     │ 1.2.13-r0      │ 1.3.1-r0       │
└──────────────┴────────────────┴──────────┴────────────────┴────────────────┘
```

#### Use distroless images for smaller attack surface:

```dockerfile
# BEFORE: Regular Alpine base (has shell, package manager, etc.)
FROM node:20-alpine    # 180+ packages, shell access

# AFTER: Distroless base (only your app, nothing else)
FROM gcr.io/distroless/nodejs20-debian12   # Only Node.js runtime
# No shell, no package manager, no curl — nothing for attackers to use!
```

**Why distroless?**
- Regular images: ~180 packages → 180 potential vulnerabilities
- Distroless images: ~15 packages → much smaller attack surface
- **No shell** → even if an attacker gets in, they can't run commands

#### Create a Trivy policy file:

```yaml
# trivy-policy.yaml
# Define what severity levels should fail the build

namespaces:
  - builtin.aws
  - builtin.kubernetes
  - user

policies:
  - name: container-scan-policy
    rules:
      # Block if any CRITICAL vulnerabilities found
      - severity: CRITICAL
        action: FAIL
      # Warn for HIGH vulnerabilities
      - severity: HIGH
        action: WARN
        # But fail if there are more than 5 HIGH vulns
        max_count: 5
      # Ignore MEDIUM and below
      - severity: MEDIUM
        action: IGNORE
      - severity: LOW
        action: IGNORE
```

### Gate 5: Image Signing with Cosign

**How do you know the Docker image running in production is the same one your pipeline built?** What if someone tampered with it in the registry?

**Image signing** solves this. You cryptographically sign the image after building it, and then verify the signature before deploying it.

#### Install Cosign:

```bash
# Linux
curl -O -L "https://github.com/sigstore/cosign/releases/latest/download/cosign-linux-amd64"
sudo mv cosign-linux-amd64 /usr/local/bin/cosign
sudo chmod +x /usr/local/bin/cosign

# Mac
brew install cosign
```

#### Sign images in your pipeline:

```bash
# Generate a key pair (do this ONCE, store the private key securely)
cosign generate-key-pair

# This creates:
# cosign.key  ← Private key (KEEP THIS SECRET — store in vault)
# cosign.pub  ← Public key  (distribute this for verification)

# Sign an image after building it
cosign sign --key cosign.key ghcr.io/myorg/my-app:v1.0.0

# Verify an image before deploying it
cosign verify --key cosign.pub ghcr.io/myorg/my-app:v1.0.0
```

**Keyless signing with Sigstore (even better — no keys to manage!):**

```bash
# Keyless signing uses OIDC identity (GitHub, Google, etc.)
# No private keys to manage! 🎉

# Sign (will open browser for identity verification)
cosign sign ghcr.io/myorg/my-app:v1.0.0

# Verify
cosign verify \
  --certificate-identity=user@email.com \
  --certificate-oidc-issuer=https://accounts.google.com \
  ghcr.io/myorg/my-app:v1.0.0
```

**What image signing prevents:**

```
Without signing:
  Attacker compromises registry → Replaces your image with a malicious one
  → K8s pulls the malicious image → Deployed to production 💀

With signing:
  Attacker compromises registry → Replaces your image
  → K8s checks the signature → SIGNATURE MISMATCH → Deployment BLOCKED 🛑
```

### Gate 6: Infrastructure as Code (IaC) Scanning

Your Terraform files and Kubernetes manifests can have security misconfigurations too. This gate catches them.

#### Using Checkov:

```bash
# Install Checkov
pip install checkov

# Scan Terraform files
checkov -d terraform/ --framework terraform

# Scan Kubernetes manifests
checkov -d k8s/ --framework kubernetes

# Scan Dockerfiles
checkov -f Dockerfile --framework dockerfile
```

**Example Checkov output:**

```
$ checkov -d k8s/ --framework kubernetes

Passed checks: 12, Failed checks: 5, Skipped checks: 0

Check: CKV_K8S_1: "Do not admit containers with the NET_RAW capability"
  FAILED for resource: Deployment.my-app
  File: /k8s/deployment.yaml:1-45

Check: CKV_K8S_6: "Do not admit containers with root access"
  FAILED for resource: Deployment.my-app
  File: /k8s/deployment.yaml:1-45
  Guide: https://docs.bridgecrew.io/docs/bc_k8s_6

Check: CKV_K8S_8: "Liveness probe should be configured"
  FAILED for resource: Deployment.my-app
  File: /k8s/deployment.yaml:1-45

Check: CKV_K8S_20: "Containers should not run with allowPrivilegeEscalation"
  FAILED for resource: Deployment.my-app
  File: /k8s/deployment.yaml:1-45

Check: CKV_K8S_28: "Minimize the admission of containers with resource limits"
  FAILED for resource: Deployment.my-app
  File: /k8s/deployment.yaml:1-45
```

**What Checkov catches in Kubernetes manifests:**

| Check | What It Catches | Why It Matters |
|-------|----------------|----------------|
| **Running as root** | Container runs as UID 0 | Root in container ≈ root on host (container escape) |
| **No resource limits** | CPU/memory not limited | One pod can starve the entire node |
| **Privileged mode** | Container has all Linux capabilities | Can access host kernel, devices, etc. |
| **No liveness probe** | K8s can't check if the app is alive | Dead containers keep receiving traffic |
| **Writable root filesystem** | Attacker can modify files inside container | Persistence of malicious code |

#### Fix the issues — secure Kubernetes manifest:

```yaml
# k8s/deployment.yaml (SECURE version)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 2
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      # Don't use the default service account
      automountServiceAccountToken: false

      # Security context for the entire pod
      securityContext:
        runAsNonRoot: true              # NEVER run as root
        runAsUser: 1001                 # Specific non-root user
        runAsGroup: 1001                # Specific group
        fsGroup: 1001                   # File system group
        seccompProfile:
          type: RuntimeDefault          # Use the default seccomp profile

      containers:
        - name: my-app
          image: ghcr.io/myorg/my-app:v1.0.0
          ports:
            - containerPort: 3000

          # Container-level security context
          securityContext:
            allowPrivilegeEscalation: false  # Can't gain more privileges
            readOnlyRootFilesystem: true     # Can't write to container filesystem
            capabilities:
              drop:
                - ALL                        # Drop ALL Linux capabilities
              # add:                         # Only add what you NEED
              #   - NET_BIND_SERVICE         # If you need to bind to port < 1024

          # Resource limits (prevent resource abuse)
          resources:
            requests:
              cpu: 100m
              memory: 128Mi
            limits:
              cpu: 500m
              memory: 256Mi

          # Health checks
          livenessProbe:
            httpGet:
              path: /health
              port: 3000
            initialDelaySeconds: 10
            periodSeconds: 30
          readinessProbe:
            httpGet:
              path: /ready
              port: 3000
            initialDelaySeconds: 5
            periodSeconds: 10

          # Volume mount for writable temp directory
          # (since root filesystem is read-only)
          volumeMounts:
            - name: tmp
              mountPath: /tmp

      volumes:
        - name: tmp
          emptyDir: {}
```

### Gate 7: DAST — Dynamic Application Security Testing

**DAST is different from SAST.** SAST looks at your code. DAST **actually runs your application** and tries to attack it — just like a real hacker would.

#### Set up OWASP ZAP:

```bash
# Run OWASP ZAP as a Docker container
docker pull ghcr.io/zaproxy/zaproxy:stable

# Quick scan (baseline scan — passive only)
docker run --rm -t ghcr.io/zaproxy/zaproxy:stable zap-baseline.py \
  -t http://your-staging-app.com \
  -r zap-report.html

# Full scan (active — actually tries to exploit vulnerabilities)
docker run --rm -t ghcr.io/zaproxy/zaproxy:stable zap-full-scan.py \
  -t http://your-staging-app.com \
  -r zap-full-report.html

# API scan (for REST APIs)
docker run --rm -t ghcr.io/zaproxy/zaproxy:stable zap-api-scan.py \
  -t http://your-staging-app.com/api/openapi.json \
  -f openapi \
  -r zap-api-report.html
```

**What DAST checks:**

| Attack | What It Tests | Example |
|--------|--------------|---------|
| **SQL Injection** | Sending `' OR 1=1 --` in input fields | Can attacker dump your database? |
| **XSS** | Injecting `<script>alert('hacked')</script>` | Can attacker run JavaScript in users' browsers? |
| **CSRF** | Making unauthorized requests from another site | Can attacker perform actions on behalf of a user? |
| **Broken Auth** | Testing default credentials, session hijacking | Can attacker access other users' accounts? |
| **Sensitive Data Exposure** | Checking response headers, error messages | Does the app leak internal information? |
| **Security Headers** | Checking for CSP, HSTS, X-Frame-Options | Are proper security headers set? |

#### Configure ZAP for your API:

### Gate 8: Admission Control (The Final Gatekeeper)

This is the **last gate before production**. Even if everything else passes, the Kubernetes admission controller checks the deployment one more time.

We'll use **Kyverno** (Kubernetes-native, easier than OPA Gatekeeper).

#### Install Kyverno:

```bash
# Install Kyverno using Helm
helm repo add kyverno https://kyverno.github.io/kyverno/
helm repo update

helm install kyverno kyverno/kyverno \
  --namespace kyverno \
  --create-namespace \
  --set replicaCount=3
```

#### Create security policies:

```yaml
# policies/require-signed-images.yaml
# Only allow images that have been signed with Cosign
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: require-signed-images
  annotations:
    policies.kyverno.io/title: Require Signed Images
    policies.kyverno.io/description: >
      All container images must be signed using Cosign.
      Unsigned images will be rejected.
spec:
  validationFailureAction: Enforce  # Block non-compliant deployments
  background: false
  rules:
    - name: verify-image-signature
      match:
        any:
          - resources:
              kinds:
                - Pod
      verifyImages:
        - imageReferences:
            - "ghcr.io/myorg/*"
          attestors:
            - entries:
                - keys:
                    publicKeys: |-
                      -----BEGIN PUBLIC KEY-----
                      MFkwEwYHKoZIzj0CAQYIKoZIzj0DAQcDQgAE...
                      (your cosign public key)
                      -----END PUBLIC KEY-----
```


## 🔄 The Complete GitHub Actions Pipeline

Now let's put **ALL 8 gates** together into one GitHub Actions workflow:

```yaml
# .github/workflows/devsecops-pipeline.yaml
name: DevSecOps Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  # ═══════════════════════════════════════════════════════════
  # GATE 2: SAST — Static Application Security Testing
  # ═══════════════════════════════════════════════════════════
  sast-scan:
    name: "🔍 SAST Scan"
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4
        with:
          fetch-depth: 0  # Full history for SonarQube

      # Semgrep — Fast, lightweight SAST
      - name: Run Semgrep
        uses: returntocorp/semgrep-action@v1
        with:
          config: >-
            p/security-audit
            p/secrets
            p/owasp-top-ten
            p/nodejs
          generateSarif: true

      # Upload results to GitHub Security tab
      - name: Upload SARIF
        uses: github/codeql-action/upload-sarif@v3
        if: always()
        with:
          sarif_file: semgrep.sarif

      # SonarQube scan
      - name: SonarQube Scan
        uses: sonarsource/sonarqube-scan-action@master
        env:
          SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
          SONAR_HOST_URL: ${{ secrets.SONAR_HOST_URL }}

      - name: SonarQube Quality Gate
        uses: sonarsource/sonarqube-quality-gate-action@master
        timeout-minutes: 5
        env:
          SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}

  # ═══════════════════════════════════════════════════════════
  # GATE 3: SCA — Software Composition Analysis
  # ═══════════════════════════════════════════════════════════
  sca-scan:
    name: "📦 SCA Scan"
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      # Scan dependencies for known vulnerabilities
      - name: Run Trivy SCA scan
        uses: aquasecurity/trivy-action@master
        with:
          scan-type: 'fs'
          scan-ref: '.'
          scanners: 'vuln'
          severity: 'HIGH,CRITICAL'
          exit-code: '1'               # Fail if vulnerabilities found
          format: 'sarif'
          output: 'trivy-sca-results.sarif'

      - name: Upload Trivy results
        uses: github/codeql-action/upload-sarif@v3
        if: always()
        with:
          sarif_file: trivy-sca-results.sarif

      # Generate SBOM
      - name: Generate SBOM
        uses: aquasecurity/trivy-action@master
        with:
          scan-type: 'fs'
          format: 'cyclonedx'
          output: 'sbom.json'

      - name: Upload SBOM
        uses: actions/upload-artifact@v4
        with:
          name: sbom
          path: sbom.json

  # ═══════════════════════════════════════════════════════════
  # GATE 6: IaC Scan — Infrastructure as Code Security
  # ═══════════════════════════════════════════════════════════
  iac-scan:
    name: "🏗️ IaC Scan"
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      # Checkov — Scan Terraform, K8s, Docker
      - name: Run Checkov
        uses: bridgecrewio/checkov-action@master
        with:
          directory: .
          framework: terraform,kubernetes,dockerfile
          soft_fail: false      # Fail the build on issues
          output_format: sarif
          output_file_path: checkov-results.sarif

      - name: Upload Checkov results
        uses: github/codeql-action/upload-sarif@v3
        if: always()
        with:
          sarif_file: checkov-results.sarif

  # ═══════════════════════════════════════════════════════════
  # Unit Tests
  # ═══════════════════════════════════════════════════════════
  unit-tests:
    name: "🧪 Unit Tests"
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Run tests
        run: npm test -- --coverage

      - name: Upload coverage
        uses: actions/upload-artifact@v4
        with:
          name: coverage
          path: coverage/

  # ═══════════════════════════════════════════════════════════
  # Build Docker Image
  # ═══════════════════════════════════════════════════════════
  build-image:
    name: "🐳 Build Image"
    needs: [sast-scan, sca-scan, iac-scan, unit-tests]
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
      id-token: write   # Needed for keyless signing
    outputs:
      image-digest: ${{ steps.build.outputs.digest }}
      image-tag: ${{ steps.meta.outputs.tags }}
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Log in to GitHub Container Registry
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Extract metadata
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
          tags: |
            type=sha,prefix=
            type=ref,event=branch
            type=semver,pattern={{version}}

      - name: Build and push
        id: build
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

  # ═══════════════════════════════════════════════════════════
  # GATE 4: Container Image Scan
  # ═══════════════════════════════════════════════════════════
  container-scan:
    name: "🔒 Container Scan"
    needs: build-image
    runs-on: ubuntu-latest
    steps:
      - name: Run Trivy image scan
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: ${{ needs.build-image.outputs.image-tag }}
          severity: 'HIGH,CRITICAL'
          exit-code: '1'
          format: 'sarif'
          output: 'trivy-image-results.sarif'

      - name: Upload Trivy results
        uses: github/codeql-action/upload-sarif@v3
        if: always()
        with:
          sarif_file: trivy-image-results.sarif

  # ═══════════════════════════════════════════════════════════
  # GATE 5: Sign the Image with Cosign
  # ═══════════════════════════════════════════════════════════
  sign-image:
    name: "✍️ Sign Image"
    needs: [build-image, container-scan]
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
      id-token: write
    steps:
      - name: Install Cosign
        uses: sigstore/cosign-installer@v3

      - name: Log in to registry
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      # Keyless signing using GitHub OIDC
      - name: Sign image with Cosign
        env:
          COSIGN_EXPERIMENTAL: "true"
        run: |
          cosign sign --yes \
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}@${{ needs.build-image.outputs.image-digest }}

  # ═══════════════════════════════════════════════════════════
  # Deploy to Staging
  # ═══════════════════════════════════════════════════════════
  deploy-staging:
    name: "🚀 Deploy to Staging"
    needs: sign-image
    runs-on: ubuntu-latest
    environment: staging
    steps:
      - name: Checkout GitOps repo
        uses: actions/checkout@v4
        with:
          repository: ${{ github.repository_owner }}/gitops-manifests
          token: ${{ secrets.GITOPS_TOKEN }}
          path: gitops

      - name: Update image tag in staging
        run: |
          cd gitops/environments/staging
          # Update the image tag
          sed -i "s|image: .*|image: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}@${{ needs.build-image.outputs.image-digest }}|" kustomization.yaml
          
          git config user.name "GitHub Actions"
          git config user.email "actions@github.com"
          git add .
          git commit -m "Deploy to staging: ${{ github.sha }}"
          git push

      - name: Wait for ArgoCD sync
        run: |
          echo "Waiting for ArgoCD to sync staging..."
          sleep 60  # In production, use argocd CLI to wait for sync

  # ═══════════════════════════════════════════════════════════
  # GATE 7: DAST — Dynamic Application Security Testing
  # ═══════════════════════════════════════════════════════════
  dast-scan:
    name: "🌐 DAST Scan"
    needs: deploy-staging
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      # Run OWASP ZAP against the staging environment
      - name: OWASP ZAP Baseline Scan
        uses: zaproxy/action-baseline@v0.10.0
        with:
          target: ${{ vars.STAGING_URL }}
          rules_file_name: 'zap-rules.tsv'
          cmd_options: '-a'

      - name: OWASP ZAP API Scan
        uses: zaproxy/action-api-scan@v0.7.0
        with:
          target: ${{ vars.STAGING_URL }}/api/openapi.json
          format: openapi
          cmd_options: '-a'

  # ═══════════════════════════════════════════════════════════
  # Deploy to Production (only if ALL gates pass)
  # ═══════════════════════════════════════════════════════════
  deploy-production:
    name: "🏭 Deploy to Production"
    needs: [dast-scan]
    runs-on: ubuntu-latest
    environment: production  # Requires manual approval in GitHub
    steps:
      - name: Checkout GitOps repo
        uses: actions/checkout@v4
        with:
          repository: ${{ github.repository_owner }}/gitops-manifests
          token: ${{ secrets.GITOPS_TOKEN }}
          path: gitops

      - name: Update image tag in production
        run: |
          cd gitops/environments/production
          sed -i "s|image: .*|image: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}@${{ needs.build-image.outputs.image-digest }}|" kustomization.yaml
          
          git config user.name "GitHub Actions"
          git config user.email "actions@github.com"
          git add .
          git commit -m "Deploy to production: ${{ github.sha }}"
          git push

      - name: Notify team
        uses: slackapi/slack-github-action@v1.25.0
        with:
          channel-id: '#deployments'
          slack-message: "✅ ${{ env.IMAGE_NAME }} deployed to production (commit: ${{ github.sha }})"
        env:
          SLACK_BOT_TOKEN: ${{ secrets.SLACK_BOT_TOKEN }}
```

## 🛡️ Runtime Security with Falco (Bonus Gate)

Even after deployment, you need runtime monitoring. **Falco** watches what your containers actually **do** at runtime and alerts on suspicious behavior.

#### Install Falco:

```bash
helm repo add falcosecurity https://falcosecurity.github.io/charts
helm repo update

helm install falco falcosecurity/falco \
  --namespace falco \
  --create-namespace \
  --set falcosidekick.enabled=true \
  --set falcosidekick.config.slack.webhookurl=$SLACK_WEBHOOK_URL
```

#### What Falco detects at runtime:

```

│ FALCO RUNTIME DETECTION EXAMPLES                         │
├──────────────────────────────────────────────────────────┤
│                                                          │

│                                                          │

│    → "Someone downloaded and ran a new binary!"          │
│                                                          │

│    → "Container is trying to access the K8s API!"        │
```

#### Custom Falco rules:

```yaml
# falco-custom-rules.yaml
customRules:
  custom-rules.yaml: |-
    # Alert when someone exec's into a production container
    - rule: Shell in Production Container
      desc: Detect shell execution in production namespaces
      condition: >
        spawned_process and container and
        proc.name in (bash, sh, zsh, ash) and
        k8s.ns.name in (production, prod)
      output: >
        Shell spawned in production container
        (user=%user.name pod=%k8s.pod.name ns=%k8s.ns.name
         container=%container.name command=%proc.cmdline)
      priority: WARNING
      tags: [container, shell, production]

    # Alert when sensitive files are accessed
    - rule: Sensitive File Access
      desc: Detect access to sensitive files
      condition: >
        open_read and container and
        fd.name in (/etc/shadow, /etc/passwd, /root/.ssh)
      output: >
        Sensitive file accessed in container
        (file=%fd.name user=%user.name container=%container.name)
      priority: ERROR
      tags: [container, filesystem, security]
```

## ❌ Common Mistakes and How to Avoid Them

| Mistake | Problem | Solution |
|---------|---------|----------|
| **Too many false positives** | Developers ignore all alerts | Tune rules, create baselines, suppress known FPs |
| **Blocking everything** | Pipeline never passes, devs work around it | Start with WARN mode, gradually move to ENFORCE |
| **Not fixing findings** | Scanning without remediation is useless | Set SLAs: CRITICAL = 24h, HIGH = 7 days |
| **Scanning only in CI** | Pre-commit and runtime are missed | Cover ALL stages: pre-commit → CI → deploy → runtime |
| **Ignoring SBOM** | Can't respond quickly to new CVEs | Generate SBOM for every build, store centrally |
| **No security dashboard** | No visibility into security posture | Grafana dashboard with security metrics |
| **One-time setup** | Security tools get outdated | Update scanning rules and tools regularly |
