## 🎯 What Are We Building?

We're building a **real-world multi-container application** that consists of three services working together:

| Service | Technology | Purpose |
|---------|-----------|---------|
| **App** | Node.js (Express) | The web application — handles HTTP requests, talks to the database and cache |
| **Database** | PostgreSQL | Stores persistent data (users, tasks, etc.) |
| **Cache** | Redis | In-memory cache for fast data retrieval and session storage |

This is how **real production applications** are structured. Almost every app you use daily (Instagram, Twitter, Uber) has this exact pattern: an application server, a database, and a cache.

### Why Docker Compose?

In Project 28.2, we ran a single container with `docker run`. But real apps have **multiple services** that need to work together. Imagine running each one manually:

```bash
# Without Docker Compose — the painful way 😫
docker network create mynetwork
docker run -d --name postgres --network mynetwork -e POSTGRES_PASSWORD=secret postgres:16
docker run -d --name redis --network mynetwork redis:7-alpine
docker run -d --name app --network mynetwork -p 3000:3000 \
  -e DATABASE_URL=postgresql://... \
  -e REDIS_URL=redis://redis:6379 \
  my-app:1.0
```

Three long commands, and you have to remember the exact order, network names, environment variables... What a mess!

**Docker Compose** lets you define ALL of this in a single YAML file and manage everything with one command:

```bash
# With Docker Compose — the easy way 😎
docker compose up -d
```

That's it. One command starts everything. One command stops everything. Beautiful.

#### 4.2 — Database Connection (db.js)

Create `app/src/db.js`:

```javascript
// db.js — PostgreSQL Database Connection

const { Pool } = require('pg');

// Create a connection pool
// A pool maintains multiple database connections and reuses them
// This is MUCH more efficient than creating a new connection for every query
const pool = new Pool({
    connectionString: process.env.DATABASE_URL,
    // Pool configuration
    max: 10,            // Maximum 10 connections in the pool
    idleTimeoutMillis: 30000,  // Close idle connections after 30 seconds
    connectionTimeoutMillis: 5000,  // Give up connecting after 5 seconds
});

// Test the connection
pool.on('connect', () => {
    console.log('📦 Connected to PostgreSQL database');
});

pool.on('error', (err) => {
    console.error('❌ PostgreSQL pool error:', err.message);
});

// Helper function to run queries
async function query(text, params) {
    const start = Date.now();
    const result = await pool.query(text, params);
    const duration = Date.now() - start;
    console.log(`📊 Query executed in ${duration}ms | Rows: ${result.rowCount}`);
    return result;
}

// Export the pool and query function
module.exports = { pool, query };
```

**What's a connection pool?**

Imagine a database connection is like a phone line to a call center:

```
Without a pool:
  Need to make a query → Pick up phone → Dial → Wait for connection → Talk → Hang up
  Need another query → Pick up phone → Dial → Wait → Talk → Hang up
  (Calling and hanging up every time is SLOW)

With a pool:
  Start app → Open 10 phone lines → Keep them open
  Need a query → Use an available line → Talk → Line goes back to pool
  Need another query → Use another available line → Talk → Back to pool
  (Much faster because connections are already established!)
```

#### 4.3 — Redis Cache Connection (cache.js)

Create `app/src/cache.js`:

```javascript
// cache.js — Redis Cache Connection

const redis = require('redis');

// Create Redis client
const client = redis.createClient({
    url: process.env.REDIS_URL || 'redis://redis:6379'
});

// Handle connection events
client.on('connect', () => {
    console.log('🔴 Connected to Redis cache');
});

client.on('error', (err) => {
    console.error('❌ Redis error:', err.message);
});

client.on('reconnecting', () => {
    console.log('🔄 Reconnecting to Redis...');
});

// Connect to Redis
async function connectRedis() {
    try {
        await client.connect();
        console.log('✅ Redis connection established');
    } catch (err) {
        console.error('❌ Failed to connect to Redis:', err.message);
        // Don't crash the app — Redis is optional (cache miss is okay)
    }
}

// Cache helper functions

// GET: Retrieve a value from cache
async function getCache(key) {
    try {
        const value = await client.get(key);
        if (value) {
            console.log(`🎯 Cache HIT for key: ${key}`);
            return JSON.parse(value);
        }
        console.log(`❌ Cache MISS for key: ${key}`);
        return null;
    } catch (err) {
        console.error('Redis GET error:', err.message);
        return null;
    }
}

// SET: Store a value in cache with optional expiration
async function setCache(key, value, ttlSeconds = 60) {
    try {
        await client.set(key, JSON.stringify(value), { EX: ttlSeconds });
        console.log(`💾 Cached key: ${key} (TTL: ${ttlSeconds}s)`);
    } catch (err) {
        console.error('Redis SET error:', err.message);
    }
}

// DELETE: Remove a value from cache
async function deleteCache(key) {
    try {
        await client.del(key);
        console.log(`🗑️ Deleted cache key: ${key}`);
    } catch (err) {
        console.error('Redis DEL error:', err.message);
    }
}

// INCREMENT: Increment a counter (great for tracking visits)
async function incrementCounter(key) {
    try {
        const count = await client.incr(key);
        return count;
    } catch (err) {
        console.error('Redis INCR error:', err.message);
        return 0;
    }
}

module.exports = { client, connectRedis, getCache, setCache, deleteCache, incrementCounter };
```

**What is Redis and why do we use it?**

Redis is an **in-memory data store** — it stores data in RAM instead of on disk. This makes it incredibly fast (microseconds vs milliseconds for a database).

**Common uses of Redis:**
1. **Caching:** Store database query results so you don't hit the database every time
2. **Session storage:** Store user sessions (login state)
3. **Rate limiting:** Track how many requests a user has made
4. **Counters:** Count page views, likes, etc.
5. **Queues:** Job queues for background processing

```
Without Redis:
  User request → App → Database (100ms) → Response
  Same request → App → Database (100ms) → Response
  Same request → App → Database (100ms) → Response
  (Hitting the database EVERY time)

With Redis:
  User request → App → Database (100ms) → Store in Redis → Response
  Same request → App → Redis (1ms) → Response  ← MUCH FASTER!
  Same request → App → Redis (1ms) → Response  ← Still fast!
  (Database only hit once, Redis serves the rest)
```

#### 4.4 — API Routes (routes.js)

Create `app/src/routes.js`:

```javascript
// routes.js — API Route Handlers

const express = require('express');
const { query } = require('./db');
const { getCache, setCache, deleteCache, incrementCounter } = require('./cache');
const os = require('os');

const router = express.Router();

// ============================================
// HOME — Basic info about the app
// ============================================
router.get('/', async (req, res) => {
    // Track total visits using Redis counter
    const visitCount = await incrementCounter('total_visits');

    res.json({
        message: '🚀 Multi-Container App is running!',
        version: '1.0.0',
        services: {
            app: 'Node.js + Express',
            database: 'PostgreSQL 16',
            cache: 'Redis 7'
        },
        container: {
            hostname: os.hostname(),
            platform: os.platform(),
            uptime: `${Math.floor(process.uptime())} seconds`
        },
        stats: {
            totalVisits: visitCount
        },
        endpoints: {
            tasks: '/api/tasks',
            taskById: '/api/tasks/:id',
            health: '/health',
            stats: '/api/stats'
        },
        timestamp: new Date().toISOString()
    });
});

// ============================================
// HEALTH CHECK — Used by Docker & Kubernetes
// ============================================
router.get('/health', async (req, res) => {
    const health = {
        status: 'healthy',
        uptime: `${Math.floor(process.uptime())} seconds`,
        checks: {}
    };

    // Check database connection
    try {
        await query('SELECT 1');
        health.checks.database = 'connected';
    } catch (err) {
        health.checks.database = 'disconnected';
        health.status = 'unhealthy';
    }

    // Check Redis connection
    try {
        const { client } = require('./cache');
        if (client.isReady) {
            health.checks.redis = 'connected';
        } else {
            health.checks.redis = 'disconnected';
        }
    } catch (err) {
        health.checks.redis = 'disconnected';
    }

    const statusCode = health.status === 'healthy' ? 200 : 503;
    res.status(statusCode).json(health);
});

// ============================================
// GET ALL TASKS — with Redis caching!
// ============================================
router.get('/api/tasks', async (req, res) => {
    try {
        // Step 1: Check Redis cache first
        const cachedTasks = await getCache('all_tasks');
        if (cachedTasks) {
            // Cache HIT — return cached data (super fast!)
            return res.json({
                source: 'cache (Redis)',
                data: cachedTasks
            });
        }

        // Step 2: Cache MISS — query the database
        const result = await query('SELECT * FROM tasks ORDER BY created_at DESC');

        // Step 3: Store the result in Redis for 30 seconds
        await setCache('all_tasks', result.rows, 30);

        // Step 4: Return the data
        res.json({
            source: 'database (PostgreSQL)',
            data: result.rows
        });
    } catch (err) {
        console.error('Error fetching tasks:', err.message);
        res.status(500).json({ error: 'Failed to fetch tasks', details: err.message });
    }
});

// ============================================
// GET SINGLE TASK — by ID
// ============================================
router.get('/api/tasks/:id', async (req, res) => {
    try {
        const { id } = req.params;

        // Check cache first
        const cached = await getCache(`task_${id}`);
        if (cached) {
            return res.json({ source: 'cache', data: cached });
        }

        // Query database (using parameterized query to prevent SQL injection!)
        const result = await query('SELECT * FROM tasks WHERE id = $1', [id]);

        if (result.rows.length === 0) {
            return res.status(404).json({ error: 'Task not found' });
        }

        // Cache for 60 seconds
        await setCache(`task_${id}`, result.rows[0], 60);

        res.json({ source: 'database', data: result.rows[0] });
    } catch (err) {
        console.error('Error fetching task:', err.message);
        res.status(500).json({ error: 'Failed to fetch task', details: err.message });
    }
});

// ============================================
// CREATE A NEW TASK
// ============================================
router.post('/api/tasks', async (req, res) => {
    try {
        const { title, description, priority } = req.body;

        if (!title) {
            return res.status(400).json({ error: 'Title is required' });
        }

        const result = await query(
            'INSERT INTO tasks (title, description, priority) VALUES ($1, $2, $3) RETURNING *',
            [title, description || null, priority || 'medium']
        );

        // Invalidate the tasks cache (because we added a new task)
        await deleteCache('all_tasks');

        res.status(201).json({
            message: 'Task created successfully',
            data: result.rows[0]
        });
    } catch (err) {
        console.error('Error creating task:', err.message);
        res.status(500).json({ error: 'Failed to create task', details: err.message });
    }
});

// ============================================
// UPDATE A TASK
// ============================================
router.put('/api/tasks/:id', async (req, res) => {
    try {
        const { id } = req.params;
        const { title, description, status, priority } = req.body;

        const result = await query(
            `UPDATE tasks 
             SET title = COALESCE($1, title),
                 description = COALESCE($2, description),
                 status = COALESCE($3, status),
                 priority = COALESCE($4, priority),
                 updated_at = CURRENT_TIMESTAMP
             WHERE id = $5 RETURNING *`,
            [title, description, status, priority, id]
        );

        if (result.rows.length === 0) {
            return res.status(404).json({ error: 'Task not found' });
        }

        // Invalidate caches
        await deleteCache('all_tasks');
        await deleteCache(`task_${id}`);

        res.json({
            message: 'Task updated successfully',
            data: result.rows[0]
        });
    } catch (err) {
        console.error('Error updating task:', err.message);
        res.status(500).json({ error: 'Failed to update task', details: err.message });
    }
});

// ============================================
// DELETE A TASK
// ============================================
router.delete('/api/tasks/:id', async (req, res) => {
    try {
        const { id } = req.params;

        const result = await query('DELETE FROM tasks WHERE id = $1 RETURNING *', [id]);

        if (result.rows.length === 0) {
            return res.status(404).json({ error: 'Task not found' });
        }

        // Invalidate caches
        await deleteCache('all_tasks');
        await deleteCache(`task_${id}`);

        res.json({
            message: 'Task deleted successfully',
            data: result.rows[0]
        });
    } catch (err) {
        console.error('Error deleting task:', err.message);
        res.status(500).json({ error: 'Failed to delete task', details: err.message });
    }
});

// ============================================
// STATS — Application statistics
// ============================================
router.get('/api/stats', async (req, res) => {
    try {
        // Get task statistics from database
        const totalResult = await query('SELECT COUNT(*) as total FROM tasks');
        const statusResult = await query(
            `SELECT status, COUNT(*) as count 
             FROM tasks 
             GROUP BY status 
             ORDER BY count DESC`
        );
        const priorityResult = await query(
            `SELECT priority, COUNT(*) as count 
             FROM tasks 
             GROUP BY priority 
             ORDER BY count DESC`
        );

        res.json({
            tasks: {
                total: parseInt(totalResult.rows[0].total),
                byStatus: statusResult.rows,
                byPriority: priorityResult.rows
            },
            container: {
                hostname: os.hostname(),
                memoryUsage: `${Math.round(process.memoryUsage().heapUsed / 1024 / 1024)}MB`,
                uptime: `${Math.floor(process.uptime())} seconds`
            }
        });
    } catch (err) {
        console.error('Error fetching stats:', err.message);
        res.status(500).json({ error: 'Failed to fetch stats', details: err.message });
    }
});

module.exports = router;
```

**Key patterns in the code:**

1. **Cache-aside pattern:** Check Redis first → if cached, return it → if not, query database → store in Redis → return result
2. **Parameterized queries:** `$1, $2, $3` instead of string concatenation — prevents SQL injection attacks!
3. **Cache invalidation:** When data changes (create/update/delete), we delete the cached version so stale data isn't served
4. **Health checks:** The `/health` endpoint checks if the database and Redis are accessible — this is used by Docker and Kubernetes to know if the app is healthy

#### 4.5 — Main Application Entry Point (index.js)

Create `app/src/index.js`:

```javascript
// index.js — Application Entry Point

const express = require('express');
const routes = require('./routes');
const { connectRedis } = require('./cache');
const { pool } = require('./db');

const app = express();
const PORT = process.env.PORT || 3000;

// Middleware: Parse JSON request bodies
app.use(express.json());

// Middleware: Simple request logging
app.use((req, res, next) => {
    console.log(`📨 ${req.method} ${req.path}`);
    next();
});

// Mount all routes
app.use(routes);

// Global error handler
app.use((err, req, res, next) => {
    console.error('💥 Unhandled error:', err.message);
    res.status(500).json({ error: 'Internal Server Error' });
});

// Start the server
async function startServer() {
    try {
        // Connect to Redis
        await connectRedis();

        // Test database connection
        const dbResult = await pool.query('SELECT NOW()');
        console.log(`📦 Database connected at: ${dbResult.rows[0].now}`);

        // Start Express server
        app.listen(PORT, '0.0.0.0', () => {
            console.log('═══════════════════════════════════════');
            console.log(`🚀 Server running on port ${PORT}`);
            console.log(`📍 http://localhost:${PORT}`);
            console.log(`📦 Database: PostgreSQL`);
            console.log(`🔴 Cache: Redis`);
            console.log('═══════════════════════════════════════');
        });
    } catch (err) {
        console.error('❌ Failed to start server:', err.message);
        // Retry after 5 seconds (database might not be ready yet)
        console.log('🔄 Retrying in 5 seconds...');
        setTimeout(startServer, 5000);
    }
}

// Handle graceful shutdown
process.on('SIGTERM', async () => {
    console.log('👋 SIGTERM received. Shutting down gracefully...');
    await pool.end();         // Close database connections
    process.exit(0);
});

process.on('SIGINT', async () => {
    console.log('👋 SIGINT received. Shutting down...');
    await pool.end();
    process.exit(0);
});

// Start!
startServer();
```

**Important concepts in this file:**

| Concept | What It Does |
|---------|-------------|
| `express.json()` | Middleware that parses JSON request bodies (so we can read `req.body`) |
| Request logging middleware | Logs every incoming request — great for debugging |
| Retry logic | If the database isn't ready yet (common with Docker), the app retries after 5 seconds |
| Graceful shutdown | When the container is stopped (`SIGTERM`), we properly close database connections instead of just killing everything |

**Why graceful shutdown matters:**

When Docker stops a container, it sends a `SIGTERM` signal first and waits 10 seconds. If the app is still running after 10 seconds, Docker sends `SIGKILL` (force kill). By handling `SIGTERM`, we can:
- Close database connections properly (prevents "connection leaked" errors)
- Finish processing in-flight requests
- Clean up resources

```dockerfile
# ============================================
# Dockerfile for the Node.js Application
# ============================================

FROM node:18-alpine

# Set metadata
LABEL maintainer="devops-learner"
LABEL description="Multi-container app — Node.js service"

# Set working directory

# Copy package files first (layer caching!)

# Install dependencies

# Copy application code
COPY . .

# Create non-root user
RUN addgroup -S appgroup && adduser -S appuser -G appgroup

# Switch to non-root user
USER appuser

# Document the port
EXPOSE 3000

# Health check — Docker will periodically check if the app is healthy
# --interval=30s: Check every 30 seconds
# --timeout=10s: Wait up to 10 seconds for a response
# --start-period=40s: Wait 40 seconds before starting checks (gives the app time to start)
# --retries=3: Mark as unhealthy after 3 consecutive failures
HEALTHCHECK --interval=30s --timeout=10s --start-period=40s --retries=3 \
    CMD wget --no-verbose --tries=1 --spider http://localhost:3000/health || exit 1

# Start the application
CMD ["node", "src/index.js"]
```

**New instruction — HEALTHCHECK:**

The `HEALTHCHECK` instruction tells Docker how to check if your container is healthy. Docker will periodically run the command specified and mark the container as `healthy` or `unhealthy`.

| Parameter | Meaning |
|-----------|---------|
| `--interval=30s` | Run the health check every 30 seconds |
| `--timeout=10s` | If the check takes more than 10 seconds, consider it failed |
| `--start-period=40s` | Wait 40 seconds after the container starts before running checks (startup time) |
| `--retries=3` | After 3 consecutive failures, mark as `unhealthy` |

The check itself: `wget --spider http://localhost:3000/health` — sends an HTTP request to the `/health` endpoint. If it returns 200, the container is healthy. If it fails, the container is unhealthy.

### Step 7: Create the Docker Compose File (The Star of the Show! ⭐)

Create `docker-compose.yml` in the project root:

```yaml
# ============================================
# Docker Compose Configuration
# Multi-Container App: Node.js + PostgreSQL + Redis
# ============================================

# Compose file version (optional in v2, but good practice)
version: '3.8'

services:
  # ==========================================
  # SERVICE 1: Node.js Application
  # ==========================================
  app:
    # Build the image from the ./app directory using the Dockerfile there
    build:
      context: ./app
      dockerfile: Dockerfile

    # Container name (optional — makes it easier to reference)
    container_name: devops-app

    # Port mapping: host_port:container_port
    ports:
      - "3000:3000"

    # Environment variables
    # These override the values from the .env file
    environment:
      - NODE_ENV=production
      - PORT=3000
      - DATABASE_URL=postgresql://${POSTGRES_USER}:${POSTGRES_PASSWORD}@postgres:5432/${POSTGRES_DB}
      - REDIS_URL=redis://redis:6379

    # Dependencies — don't start the app until these services are running
    depends_on:
      postgres:
        condition: service_healthy    # Wait until PostgreSQL is actually HEALTHY
      redis:
        condition: service_healthy    # Wait until Redis is actually HEALTHY

    # Connect to our custom network
    networks:
      - app-network

    # Restart policy — what to do if the container crashes
    restart: unless-stopped

  # ==========================================
  # SERVICE 2: PostgreSQL Database
  # ==========================================
  postgres:
    # Use the official PostgreSQL 16 image from Docker Hub
    image: postgres:16-alpine

    container_name: devops-postgres

    # Environment variables for PostgreSQL
    # These are read from the .env file automatically!
    environment:
      - POSTGRES_USER=${POSTGRES_USER}
      - POSTGRES_PASSWORD=${POSTGRES_PASSWORD}
      - POSTGRES_DB=${POSTGRES_DB}

    # Expose port 5432 to the host (optional — useful for debugging with a DB tool)
    ports:
      - "5432:5432"

    # Volume: Persist database data even if the container is removed
    volumes:
      # Named volume for database data
      - pgdata:/var/lib/postgresql/data
      # Mount the init script — PostgreSQL runs .sql files in /docker-entrypoint-initdb.d/ on first start
      - ./db/init.sql:/docker-entrypoint-initdb.d/init.sql:ro

    # Health check for PostgreSQL
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${POSTGRES_USER} -d ${POSTGRES_DB}"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 30s

    # Connect to our network
    networks:
      - app-network

    # Restart policy
    restart: unless-stopped

  # ==========================================
  # SERVICE 3: Redis Cache
  # ==========================================
  redis:
    # Use the official Redis 7 image (Alpine for small size)
    image: redis:7-alpine

    container_name: devops-redis

    # Redis port (optional — useful for debugging with Redis CLI)
    ports:
      - "6379:6379"

    # Persist Redis data
    volumes:
      - redisdata:/data

    # Custom Redis command with some production settings
    command: redis-server --appendonly yes --maxmemory 256mb --maxmemory-policy allkeys-lru

    # Health check for Redis
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 10s

    # Connect to our network
    networks:
      - app-network

    # Restart policy
    restart: unless-stopped

# ============================================
# VOLUMES — Persistent Storage
# ============================================
volumes:
  pgdata:
    driver: local       # Store on the local filesystem
  redisdata:
    driver: local

# ============================================
# NETWORKS — Service Communication
# ============================================
networks:
  app-network:
    driver: bridge      # Default Docker network driver
```

**This is the most important file. Let's break down every section:**

---

#### Understanding `services:`

Each service is a container. We have three:

| Service | Image | Purpose |
|---------|-------|---------|
| `app` | Built from `./app/Dockerfile` | Our Node.js application |
| `postgres` | `postgres:16-alpine` (from Docker Hub) | PostgreSQL database |
| `redis` | `redis:7-alpine` (from Docker Hub) | Redis cache |

#### Understanding `depends_on:`

```yaml
depends_on:
  postgres:
    condition: service_healthy
```

This tells Docker: "Don't start the `app` service until `postgres` is **healthy** (not just started, but actually ready to accept connections)."

Without `condition: service_healthy`, Docker would only wait for the container to START, not for PostgreSQL to be READY. The app might try to connect to the database before it's ready and crash.

#### Understanding `volumes:`

```yaml
volumes:
  - pgdata:/var/lib/postgresql/data     # Named volume
  - ./db/init.sql:/docker-entrypoint-initdb.d/init.sql:ro   # Bind mount
```

| Volume Type | Syntax | What It Does |
|-------------|--------|-------------|
| **Named volume** | `pgdata:/var/lib/postgresql/data` | Docker manages the storage. Data survives container removal. Best for databases |
| **Bind mount** | `./db/init.sql:/path/in/container:ro` | Maps a file/folder from your host into the container. `:ro` means read-only |

**Why volumes matter for databases:**

```
Without volumes:
  docker compose down → ALL DATABASE DATA IS GONE! 💀
  docker compose up → Fresh database, no data

With volumes:
  docker compose down → Container is removed, but volume still exists
  docker compose up → Container starts, finds existing data in volume → All data is preserved! ✅
```

#### Understanding `networks:`

```yaml
networks:
  app-network:
    driver: bridge
```

All three services are on the `app-network` network. This means:

1. They can talk to each other using **service names** as hostnames
2. `app` can reach PostgreSQL at `postgres:5432` (the service name!)
3. `app` can reach Redis at `redis:6379`
4. Services are isolated from other Docker containers on your machine

```
Inside app-network:
  app → postgres:5432  ✅ (can connect)
  app → redis:6379     ✅ (can connect)
  app → google.com     ✅ (can access internet)
  
Outside app-network:
  other-container → postgres:5432  ❌ (can't connect — not on this network)
```

#### Understanding Redis `command:`

```yaml
command: redis-server --appendonly yes --maxmemory 256mb --maxmemory-policy allkeys-lru
```

| Flag | What It Does |
|------|-------------|
| `--appendonly yes` | Enable persistence — Redis saves data to disk (so it survives restarts) |
| `--maxmemory 256mb` | Limit Redis to 256MB of RAM |
| `--maxmemory-policy allkeys-lru` | When memory is full, remove the **Least Recently Used** keys to make room |

#### Understanding `restart: unless-stopped`

| Policy | Behavior |
|--------|----------|
| `no` | Never restart (default) |
| `always` | Always restart, even after `docker compose stop` |
| `on-failure` | Only restart if the container crashes (non-zero exit code) |
| `unless-stopped` | Restart unless you explicitly stop it with `docker compose stop`. Best for production! |

### Common Issues:

| Problem | Cause | Fix |
|---------|-------|-----|
| App keeps restarting | Database not ready yet | `depends_on` with `service_healthy` should handle this. Check DB health check |
| "Connection refused" to postgres | Wrong hostname | Use `postgres` (the service name), not `localhost` |
| "Connection refused" to redis | Wrong hostname | Use `redis` (the service name), not `localhost` |
| Port already in use | Another service on port 3000/5432/6379 | Change the host port: `"3001:3000"` |
| Database data lost | Used `docker compose down -v` | Don't use `-v` if you want to keep data! |
| Init script doesn't run | Database already initialized | Remove the volume: `docker compose down -v` then `docker compose up -d` |
| Build fails | Cache issue | `docker compose build --no-cache app` |

## 📝 Understanding the Data Flow

Let's trace what happens when a user visits `http://localhost:3000/api/tasks`:

```
Step 1: Browser → http://localhost:3000/api/tasks
        ↓
Step 2: Docker forwards port 3000 to the app container
        ↓
Step 3: Express receives the request → routes.js → GET /api/tasks handler
        ↓
Step 4: Check Redis cache → getCache('all_tasks')
        ↓
        ├── Cache HIT → Return cached data (1ms) → Done!
        │
        └── Cache MISS → Continue to Step 5
                ↓
Step 5: Query PostgreSQL → SELECT * FROM tasks ORDER BY created_at DESC
        ↓
Step 6: Store result in Redis → setCache('all_tasks', data, 30)  [TTL: 30 seconds]
        ↓
Step 7: Return data to browser with source: "database"
```

```
On subsequent requests (within 30 seconds):
Step 1-3: Same as above
Step 4: Cache HIT! → Return instantly (1ms vs 50-100ms from database)
```

**This is the Cache-Aside pattern** — one of the most common patterns in production applications. It dramatically reduces database load and improves response times.
