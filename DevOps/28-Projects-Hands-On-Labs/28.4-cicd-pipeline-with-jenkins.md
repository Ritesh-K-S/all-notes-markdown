# 28.4 — CI/CD Pipeline with Jenkins — Build → Test → Docker → Deploy

> **One Line Summary:** Build a complete Jenkins pipeline that automatically builds your app, runs tests, creates a Docker image, and deploys it — every time you push code.

---

## 🎯 What Are We Building?

Imagine this — you write code, push it to GitHub, and **within minutes** your app is automatically:

1. ✅ Built (compiled / dependencies installed)
2. ✅ Tested (unit tests, lint checks)
3. ✅ Packaged into a Docker image
4. ✅ Pushed to a Docker registry
5. ✅ Deployed to a server (running as a container)

No manual steps. No "it works on my machine." **That's CI/CD with Jenkins.**

---

```
Developer pushes code to GitHub
        │
        ▼
   ┌─────────┐

        │

   │  Production  │  ← App running in a Docker container

   └──────────────┘
```

## 🛠️ Step 1 — Create a Simple Node.js Application

Let's start by creating a tiny app that we'll use throughout this project.

### 📁 Project Structure

```
my-jenkins-project/
├── app.js
├── package.json
├── test/
│   └── app.test.js
├── Dockerfile
└── Jenkinsfile
```

### 📄 `app.js` — A Simple Express Server

```javascript
const express = require('express');
const app = express();
const PORT = process.env.PORT || 3000;

app.get('/', (req, res) => {
  res.json({ 
    message: 'Hello from Jenkins CI/CD Pipeline!',
    version: '1.0.0',
    timestamp: new Date().toISOString()
  });
});

app.get('/health', (req, res) => {
  res.json({ status: 'healthy' });
});

// Only start the server if this file is run directly (not during testing)
if (require.main === module) {
  app.listen(PORT, () => {
    console.log(`Server running on port ${PORT}`);
  });
}

module.exports = app;
```

**What's happening here?**

- We create a basic Express.js web server
- It has two routes: `/` (home) and `/health` (health check)
- The `if (require.main === module)` trick allows us to import this in tests without actually starting the server
- The `module.exports = app` exports the app so our tests can use it

### 📄 `package.json` — Project Dependencies

```json
{
  "name": "my-jenkins-project",
  "version": "1.0.0",
  "description": "A demo app for Jenkins CI/CD pipeline",
  "main": "app.js",
  "scripts": {
    "start": "node app.js",
    "test": "jest --verbose"
  },
  "dependencies": {
    "express": "^4.18.2"
  },
  "devDependencies": {
    "jest": "^29.7.0",
    "supertest": "^6.3.3"
  }
}
```

**What's happening here?**

- `express` — our web framework
- `jest` — a popular testing framework
- `supertest` — lets us test HTTP endpoints without starting the server
- `npm test` will run `jest --verbose` (shows detailed test results)

### 📄 `test/app.test.js` — Unit Tests

```javascript
const request = require('supertest');
const app = require('../app');

describe('App Endpoints', () => {

  // Test 1: Check if the home route works
  test('GET / should return welcome message', async () => {
    const res = await request(app).get('/');
    expect(res.statusCode).toBe(200);
    expect(res.body.message).toBe('Hello from Jenkins CI/CD Pipeline!');
  });

  // Test 2: Check if health endpoint works
  test('GET /health should return healthy status', async () => {
    const res = await request(app).get('/health');
    expect(res.statusCode).toBe(200);
    expect(res.body.status).toBe('healthy');
  });

  // Test 3: Check if unknown routes return 404
  test('GET /unknown should return 404', async () => {
    const res = await request(app).get('/unknown');
    expect(res.statusCode).toBe(404);
  });
});
```

**What's happening here?**

- We import `supertest` to make HTTP requests to our app **without starting the server**
- We write 3 tests:
  - Does the `/` route return the right message?
  - Does `/health` return "healthy"?
  - Does an unknown route return 404?
- If any of these fail, Jenkins will **stop the pipeline** and alert you

## 🐳 Step 2 — Create the Dockerfile

```dockerfile
# Stage 1: Build stage
FROM node:18-alpine AS builder

WORKDIR /app

# Copy package files first (for better Docker layer caching)
COPY package*.json ./

# Install only production dependencies
RUN npm ci --only=production

# Copy the rest of the application code
COPY . .

# Stage 2: Production stage (smaller image)
FROM node:18-alpine

WORKDIR /app

# Copy from builder stage
COPY --from=builder /app .

# Create a non-root user for security
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

EXPOSE 3000

CMD ["node", "app.js"]
```

**What's happening here? (Line by line)**

| Line | Explanation |
|---|---|
| `FROM node:18-alpine AS builder` | Start with a lightweight Node.js base image. Alpine Linux = super small (~5MB). We name this stage "builder". |
| `WORKDIR /app` | Set `/app` as our working directory inside the container |
| `COPY package*.json ./` | Copy **only** package.json and package-lock.json first. This way, Docker caches the `npm install` layer — if your code changes but dependencies don't, it skips reinstalling! |
| `RUN npm ci --only=production` | Install dependencies. `npm ci` is faster and more reliable than `npm install` for CI environments. `--only=production` skips devDependencies (we don't need jest in production). |
| `COPY . .` | Now copy the rest of the source code |
| `FROM node:18-alpine` | Start a fresh, clean image for production (multi-stage build) |
| `COPY --from=builder /app .` | Copy the built app from the builder stage |
| `RUN addgroup ... adduser ...` | Create a non-root user (security best practice — never run as root in containers!) |
| `USER appuser` | Switch to the non-root user |
| `EXPOSE 3000` | Document that the app listens on port 3000 |
| `CMD ["node", "app.js"]` | The command to start the app when the container runs |

## ⚙️ Step 3 — Install and Set Up Jenkins

### 3.1 — Install Jenkins on Ubuntu

```bash
# Update your system
sudo apt update && sudo apt upgrade -y

# Install Java (Jenkins needs it)
sudo apt install -y openjdk-17-jre

# Verify Java is installed
java -version

# Add Jenkins repository key and source
curl -fsSL https://pkg.jenkins.io/debian-stable/jenkins.io-2023.key | sudo tee \
  /usr/share/keyrings/jenkins-keyring.asc > /dev/null

echo "deb [signed-by=/usr/share/keyrings/jenkins-keyring.asc] \
  https://pkg.jenkins.io/debian-stable binary/" | sudo tee \
  /etc/apt/sources.list.d/jenkins.list > /dev/null

# Install Jenkins
sudo apt update
sudo apt install -y jenkins

# Start Jenkins and enable it to start on boot
sudo systemctl start jenkins
sudo systemctl enable jenkins

# Check Jenkins is running
sudo systemctl status jenkins
```

**What's happening here?**

- Jenkins is a Java application, so we install Java first
- We add the official Jenkins repository (so we get the latest stable version)
- We install Jenkins and start it as a system service
- `enable` means Jenkins will auto-start whenever the server reboots

### 3.2 — Install Docker on the Jenkins Server

```bash
# Install Docker
sudo apt install -y docker.io

# Start and enable Docker
sudo systemctl start docker
sudo systemctl enable docker

# Add Jenkins user to the Docker group
# (This lets Jenkins run Docker commands without sudo)
sudo usermod -aG docker jenkins

# Restart Jenkins so it picks up the new group
sudo systemctl restart jenkins

# Verify Docker works
docker --version
```

> **⚠️ Important:** The `usermod -aG docker jenkins` command is critical! Without this, Jenkins won't be able to run Docker commands and your pipeline will fail.

### 3.3 — Access Jenkins Web UI

1. Open your browser and go to: `http://<your-server-ip>:8080`
2. Jenkins will ask for an **initial admin password**. Get it with:

```bash
sudo cat /var/lib/jenkins/secrets/initialAdminPassword
```

3. Paste the password in the browser
4. Click **"Install suggested plugins"** — this installs the most common plugins
5. Create your admin user
6. Jenkins is ready!

## 🔐 Step 4 — Configure Credentials in Jenkins

We need to store our Docker Hub credentials securely in Jenkins. **Never hardcode passwords in your Jenkinsfile!**

### 4.1 — Add Docker Hub Credentials

1. Go to **Manage Jenkins → Credentials → System → Global credentials**
2. Click **"Add Credentials"**
3. Fill in:

| Field | Value |
|---|---|
| Kind | Username with password |
| Scope | Global |
| Username | Your Docker Hub username |
| Password | Your Docker Hub password (or Access Token — recommended!) |
| ID | `dockerhub-credentials` |
| Description | Docker Hub Login |

4. Click **Save**

> **💡 Pro Tip:** Use a Docker Hub **Access Token** instead of your actual password. Go to Docker Hub → Account Settings → Security → New Access Token. This is more secure because you can revoke it without changing your password.

### 4.2 — Add GitHub Credentials (if your repo is private)

Same steps as above, but:

| Field | Value |
|---|---|
| Kind | Username with password |
| Username | Your GitHub username |
| Password | GitHub Personal Access Token (PAT) |
| ID | `github-credentials` |

## 📝 Step 5 — Write the Jenkinsfile (The Heart of Everything)

This is where the magic happens. The **Jenkinsfile** defines your entire pipeline — what to build, test, and deploy.

Create a file called `Jenkinsfile` in the **root of your project**:

```groovy
pipeline {
    // Run on any available Jenkins agent
    agent any

    // Environment variables used across the pipeline
    environment {
        DOCKER_IMAGE = 'yourdockerhubusername/my-jenkins-app'
        DOCKER_TAG = "${BUILD_NUMBER}"
        DOCKER_CREDENTIALS = 'dockerhub-credentials'
    }

    // Define stages of the pipeline
    stages {
        
        // ============================================
        // STAGE 1: Checkout the code from GitHub
        // ============================================
        stage('Checkout') {
            steps {
                echo '📥 Pulling code from GitHub...'
                checkout scm
            }
        }

        // ============================================
        // STAGE 2: Install dependencies
        // ============================================
        stage('Build') {
            steps {
                echo '🔨 Installing dependencies...'
                sh 'npm ci'
            }
        }

        // ============================================
        // STAGE 3: Run tests
        // ============================================
        stage('Test') {
            steps {
                echo '🧪 Running tests...'
                sh 'npm test'
            }
        }

        // ============================================
        // STAGE 4: Build Docker image
        // ============================================
        stage('Build Docker Image') {
            steps {
                echo '🐳 Building Docker image...'
                script {
                    dockerImage = docker.build("${DOCKER_IMAGE}:${DOCKER_TAG}")
                }
            }
        }

        // ============================================
        // STAGE 5: Push Docker image to Docker Hub
        // ============================================
        stage('Push to Docker Hub') {
            steps {
                echo '📤 Pushing image to Docker Hub...'
                script {
                    docker.withRegistry('https://registry.hub.docker.com', DOCKER_CREDENTIALS) {
                        dockerImage.push("${DOCKER_TAG}")
                        dockerImage.push('latest')
                    }
                }
            }
        }

        // ============================================
        // STAGE 6: Deploy — Run the container
        // ============================================
        stage('Deploy') {
            steps {
                echo '🚀 Deploying the application...'
                sh '''
                    # Stop and remove the old container (if it exists)
                    docker stop my-jenkins-app || true
                    docker rm my-jenkins-app || true
                    
                    # Run the new container
                    docker run -d \
                        --name my-jenkins-app \
                        -p 3000:3000 \
                        --restart unless-stopped \
                        ${DOCKER_IMAGE}:${DOCKER_TAG}
                '''
            }
        }
    }

    // ============================================
    // POST ACTIONS — Run after all stages
    // ============================================
    post {
        success {
            echo '✅ Pipeline completed successfully!'
            echo "🌐 App is running at http://<your-server-ip>:3000"
        }
        failure {
            echo '❌ Pipeline failed! Check the logs above.'
        }
        always {
            // Clean up workspace to save disk space
            cleanWs()
        }
    }
}
```

### 🔍 Jenkinsfile Explained — Section by Section

Let's break down every part of this file:

#### `pipeline { }` — The Wrapper

Everything goes inside `pipeline { }`. This tells Jenkins "hey, this is a declarative pipeline."

#### `agent any`

```groovy
agent any
```

This means: **run this pipeline on any available Jenkins agent (worker)**. If you have multiple servers connected to Jenkins, it'll pick whichever one is free.

Other options:
- `agent { label 'linux' }` — run only on agents labeled "linux"
- `agent { docker { image 'node:18' } }` — run inside a Docker container

#### `environment { }` — Variables

```groovy
environment {
    DOCKER_IMAGE = 'yourdockerhubusername/my-jenkins-app'
    DOCKER_TAG = "${BUILD_NUMBER}"
    DOCKER_CREDENTIALS = 'dockerhub-credentials'
}
```

- `DOCKER_IMAGE` — the name for your Docker image on Docker Hub
- `DOCKER_TAG` — we use `BUILD_NUMBER` (Jenkins auto-increments this: 1, 2, 3...) so every build gets a unique tag
- `DOCKER_CREDENTIALS` — the ID of the credentials we stored in Step 4

#### `stages { }` — The Pipeline Stages

Each `stage('Name') { }` block is one step in your pipeline. They run **in order**, one after another. If any stage fails, the whole pipeline stops.

#### Stage 1 — Checkout

```groovy
stage('Checkout') {
    steps {
        checkout scm
    }
}
```

`checkout scm` means: "Pull the code from wherever this Jenkinsfile came from." If the job is connected to a GitHub repo, it pulls from there.

#### Stage 2 — Build

```groovy
stage('Build') {
    steps {
        sh 'npm ci'
    }
}
```

`sh` runs a shell command. `npm ci` installs all dependencies from `package-lock.json`. It's the CI-friendly version of `npm install` — faster and more deterministic.

#### Stage 3 — Test

```groovy
stage('Test') {
    steps {
        sh 'npm test'
    }
}
```

Runs our Jest tests. If any test fails, this stage fails, and **the pipeline stops here** — no Docker image, no deployment. This is the safety gate!

#### Stage 4 — Build Docker Image

```groovy
stage('Build Docker Image') {
    steps {
        script {
            dockerImage = docker.build("${DOCKER_IMAGE}:${DOCKER_TAG}")
        }
    }
}
```

- `script { }` — allows us to use Groovy scripting (needed for the Docker Pipeline plugin)
- `docker.build(...)` — builds a Docker image from the Dockerfile in the current directory
- The image is tagged as `yourusername/my-jenkins-app:5` (where 5 is the build number)

#### Stage 5 — Push to Docker Hub

```groovy
stage('Push to Docker Hub') {
    steps {
        script {
            docker.withRegistry('https://registry.hub.docker.com', DOCKER_CREDENTIALS) {
                dockerImage.push("${DOCKER_TAG}")
                dockerImage.push('latest')
            }
        }
    }
}
```

- `docker.withRegistry(...)` — logs into Docker Hub using our saved credentials
- We push the image **twice** with two tags:
  - `yourusername/my-jenkins-app:5` — specific version
  - `yourusername/my-jenkins-app:latest` — always points to the most recent build

#### Stage 6 — Deploy

```groovy
stage('Deploy') {
    steps {
        sh '''
            docker stop my-jenkins-app || true
            docker rm my-jenkins-app || true
            docker run -d --name my-jenkins-app -p 3000:3000 --restart unless-stopped ${DOCKER_IMAGE}:${DOCKER_TAG}
        '''
    }
}
```

- `docker stop ... || true` — stop the old container. `|| true` means "don't fail if it doesn't exist" (first deployment won't have a running container)
- `docker rm ...` — remove the stopped container
- `docker run -d` — run the new container in detached (background) mode
- `-p 3000:3000` — map port 3000 on the host to port 3000 in the container
- `--restart unless-stopped` — auto-restart the container if it crashes or the server reboots

#### `post { }` — After the Pipeline

```groovy
post {
    success { echo '✅ Pipeline completed successfully!' }
    failure { echo '❌ Pipeline failed!' }
    always { cleanWs() }
}
```

- `success` — runs only if **all** stages passed
- `failure` — runs only if any stage failed
- `always` — runs no matter what (cleanup)
- `cleanWs()` — deletes the workspace files to save disk space

## 🔔 Step 7 — Set Up GitHub Webhook (Auto-Trigger)

Right now, you have to click "Build Now" manually. Let's make Jenkins **automatically** build whenever you push code.

### 7.1 — Configure Jenkins

1. Go to your pipeline job → **Configure**
2. Under **Build Triggers**, check **"GitHub hook trigger for GITScm polling"**
3. Save

### 7.2 — Add Webhook in GitHub

1. Go to your GitHub repo → **Settings → Webhooks → Add webhook**
2. Fill in:

| Field | Value |
|---|---|
| Payload URL | `http://<your-jenkins-server-ip>:8080/github-webhook/` |
| Content type | `application/json` |
| Events | Just the push event |
| Active | ✅ Checked |

3. Click **"Add webhook"**

> **⚠️ Note:** Your Jenkins server must be accessible from the internet for GitHub to reach it. If you're running Jenkins locally, you can use tools like **ngrok** to create a temporary public URL:
> ```bash
> ngrok http 8080
> ```
> Then use the ngrok URL as the Payload URL.

### 7.3 — Test It!

```bash
# Make a small change
echo "// Updated" >> app.js

# Push to GitHub
git add .
git commit -m "Test webhook trigger"
git push
```

Within seconds, check Jenkins — a new build should start automatically! 🎉

## 🛡️ Step 8 — Improvements & Best Practices

### 8.1 — Add a Lint Stage

Add a code quality check before testing:

```groovy
stage('Lint') {
    steps {
        echo '🔍 Running code linter...'
        sh 'npx eslint . --ext .js || true'
    }
}
```

### 8.2 — Add Build Notifications (Slack/Email)

Install the **Slack Notification** plugin and add to `post { }`:

```groovy
post {
    success {
        slackSend(
            channel: '#deployments',
            color: 'good',
            message: "✅ Build #${BUILD_NUMBER} succeeded! App deployed."
        )
    }
    failure {
        slackSend(
            channel: '#deployments',
            color: 'danger',
            message: "❌ Build #${BUILD_NUMBER} failed! Check Jenkins."
        )
    }
}
```

### 8.3 — Use a Dedicated Deploy Server

In production, you wouldn't deploy on the same server as Jenkins. Instead, use SSH:

```groovy
stage('Deploy to Production') {
    steps {
        sshagent(['prod-server-ssh-key']) {
            sh '''
                ssh -o StrictHostKeyChecking=no user@production-server << 'EOF'
                    docker pull ${DOCKER_IMAGE}:${DOCKER_TAG}
                    docker stop my-jenkins-app || true
                    docker rm my-jenkins-app || true
                    docker run -d --name my-jenkins-app -p 3000:3000 ${DOCKER_IMAGE}:${DOCKER_TAG}
                EOF
            '''
        }
    }
}
```

### 8.4 — Parallel Testing

If you have multiple test suites, run them in parallel to save time:

```groovy
stage('Tests') {
    parallel {
        stage('Unit Tests') {
            steps {
                sh 'npm run test:unit'
            }
        }
        stage('Integration Tests') {
            steps {
                sh 'npm run test:integration'
            }
        }
    }
}
```

### 8.5 — Docker Image Cleanup

Docker images pile up fast. Add cleanup:

```groovy
stage('Cleanup') {
    steps {
        sh '''
            docker image prune -f
            docker rmi ${DOCKER_IMAGE}:${DOCKER_TAG} || true
        '''
    }
}
```

## 🐛 Common Issues & Troubleshooting

| Problem | Cause | Solution |
|---|---|---|
| `docker: permission denied` | Jenkins user can't access Docker | Run `sudo usermod -aG docker jenkins` and restart Jenkins |
| `npm: command not found` | Node.js not installed on Jenkins agent | Install Node.js or use `agent { docker { image 'node:18' } }` |
| `Cannot connect to Docker daemon` | Docker service not running | Run `sudo systemctl start docker` |
| Pipeline stalls at "Push" | Wrong Docker Hub credentials | Re-check credentials in Jenkins → Manage Credentials |
| Webhook not triggering | Jenkins not reachable from internet | Use ngrok or ensure firewall allows port 8080 |
| `EACCES: permission denied` | npm can't write to workspace | Add `sh 'chmod -R 777 .'` before npm install |
