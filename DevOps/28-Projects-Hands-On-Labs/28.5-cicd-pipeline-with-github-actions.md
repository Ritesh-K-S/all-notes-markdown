# 28.5 — CI/CD Pipeline with GitHub Actions — Build → Test → Push to ECR → Deploy to EKS

> **One Line Summary:** Build a production-grade GitHub Actions pipeline that automatically builds your app, runs tests, pushes a Docker image to AWS ECR, and deploys it to a Kubernetes cluster on AWS EKS.

---

## 🎯 What Are We Building?

Picture this — you merge a pull request into `main`, and without lifting a finger:

1. ✅ GitHub Actions picks up the change
2. ✅ Builds your application
3. ✅ Runs all tests
4. ✅ Builds a Docker image
5. ✅ Pushes it to **AWS ECR** (Elastic Container Registry — AWS's private Docker registry)
6. ✅ Deploys it to **AWS EKS** (Elastic Kubernetes Service — managed Kubernetes)

No Jenkins server to maintain. No infrastructure to manage. It's all built into GitHub. **That's the beauty of GitHub Actions.**

### 📄 `app.py` — A Simple Flask App

```python
from flask import Flask, jsonify
import os
from datetime import datetime

app = Flask(__name__)

@app.route('/')
def home():
    return jsonify({
        'message': 'Hello from GitHub Actions + EKS!',
        'version': os.getenv('APP_VERSION', '1.0.0'),
        'environment': os.getenv('ENVIRONMENT', 'development'),
        'timestamp': datetime.utcnow().isoformat()
    })

@app.route('/health')
def health():
    return jsonify({'status': 'healthy'}), 200

@app.route('/ready')
def ready():
    return jsonify({'status': 'ready'}), 200

if __name__ == '__main__':
    port = int(os.getenv('PORT', 5000))
    app.run(host='0.0.0.0', port=port)
```

**What's happening here?**

- A basic Flask web server with three endpoints
- `/` — returns app info (message, version, environment)
- `/health` — health check (Kubernetes uses this to know if the app is alive)
- `/ready` — readiness check (Kubernetes uses this to know if the app can receive traffic)
- We read `APP_VERSION` and `ENVIRONMENT` from environment variables — this makes the app configurable without code changes

### 📄 `requirements.txt` — Python Dependencies

```

pytest==7.4.3
```

- `flask` — our web framework
- `gunicorn` — production-grade WSGI server (don't use Flask's built-in server in production!)
- `pytest` — testing framework

### 📄 `tests/test_app.py` — Unit Tests

```python
import pytest
from app import app

@pytest.fixture
def client():
    """Create a test client for our Flask app."""
    app.config['TESTING'] = True
    with app.test_client() as client:
        yield client

def test_home_endpoint(client):
    """Test that the home endpoint returns correct data."""
    response = client.get('/')
    data = response.get_json()
    
    assert response.status_code == 200
    assert 'message' in data
    assert data['message'] == 'Hello from GitHub Actions + EKS!'

def test_health_endpoint(client):
    """Test that the health endpoint returns healthy."""
    response = client.get('/health')
    data = response.get_json()
    
    assert response.status_code == 200
    assert data['status'] == 'healthy'

def test_ready_endpoint(client):
    """Test that the readiness endpoint returns ready."""
    response = client.get('/ready')
    data = response.get_json()
    
    assert response.status_code == 200
    assert data['status'] == 'ready'

def test_not_found(client):
    """Test that unknown routes return 404."""
    response = client.get('/nonexistent')
    assert response.status_code == 404
```

**What's happening here?**

- `@pytest.fixture` — creates a reusable test client. Think of it as a "fake browser" that talks to our app
- Each `test_*` function tests one thing:
  - Does `/` return the right message?
  - Does `/health` say "healthy"?
  - Does `/ready` say "ready"?
  - Does a random URL return 404?
- If any assertion fails, the CI pipeline will fail and stop deployment

## 🐳 Step 2 — Create the Dockerfile

```dockerfile
# Build stage
FROM python:3.11-slim AS builder

WORKDIR /app

# Copy requirements first for better layer caching
COPY requirements.txt .
RUN pip install --no-cache-dir --user -r requirements.txt

# Copy source code
COPY . .

# Production stage
FROM python:3.11-slim

WORKDIR /app

# Copy installed packages from builder
COPY --from=builder /root/.local /root/.local
COPY --from=builder /app .

# Make sure scripts in .local are usable
ENV PATH=/root/.local/bin:$PATH

# Expose port
EXPOSE 5000

# Run with gunicorn (production server)
# - 4 worker processes
# - bind to all interfaces on port 5000
CMD ["gunicorn", "--workers", "4", "--bind", "0.0.0.0:5000", "app:app"]
```

**What's happening here?**

| Line | Explanation |
|---|---|
| `FROM python:3.11-slim AS builder` | Slim Python image (~150MB vs ~900MB for full). We name this "builder". |
| `pip install --no-cache-dir --user` | Install packages without caching (saves space) and into user directory |
| `FROM python:3.11-slim` | Fresh image for production (multi-stage build = smaller final image) |
| `COPY --from=builder /root/.local /root/.local` | Copy only the installed packages from builder |
| `ENV PATH=/root/.local/bin:$PATH` | Add the packages to the system PATH so `gunicorn` command works |
| `CMD ["gunicorn", ...]` | Use gunicorn (production server) instead of Flask's dev server. 4 workers = handles 4 requests simultaneously |

## ☁️ Step 3 — Set Up AWS Infrastructure

Before our pipeline can push and deploy, we need to set up AWS resources.

### 3.1 — Create an ECR Repository

ECR (Elastic Container Registry) is AWS's private Docker registry. It's where your Docker images will live.

```bash
# Create the ECR repository
aws ecr create-repository \
    --repository-name my-github-actions-app \
    --region us-east-1 \
    --image-scanning-configuration scanOnPush=true

# The output will include a repositoryUri like:
# 123456789012.dkr.ecr.us-east-1.amazonaws.com/my-github-actions-app
# Save this! You'll need it.
```

**What this does:**

- Creates a private Docker registry called `my-github-actions-app` in AWS
- `scanOnPush=true` — automatically scans images for security vulnerabilities when you push them (free!)
- The `repositoryUri` is the full address where your images will be stored

### 3.2 — Create an EKS Cluster

EKS (Elastic Kubernetes Service) is AWS's managed Kubernetes. AWS handles the control plane (API server, etcd, etc.) — you just manage your worker nodes.

```bash
# Install eksctl (the official EKS CLI tool)
# On Linux:
curl --silent --location "https://github.com/weaveworks/eksctl/releases/latest/download/eksctl_$(uname -s)_amd64.tar.gz" | tar xz -C /tmp
sudo mv /tmp/eksctl /usr/local/bin

# Create the EKS cluster (this takes 15-20 minutes)
eksctl create cluster \
    --name my-eks-cluster \
    --region us-east-1 \
    --version 1.28 \
    --nodegroup-name standard-workers \
    --node-type t3.medium \
    --nodes 2 \
    --nodes-min 1 \
    --nodes-max 3 \
    --managed

# Verify the cluster is ready
kubectl get nodes
```

**What each flag means:**

| Flag | Meaning |
|---|---|
| `--name` | Name of your cluster |
| `--region` | AWS region to create the cluster in |
| `--version 1.28` | Kubernetes version |
| `--nodegroup-name` | Name for the group of worker nodes |
| `--node-type t3.medium` | Instance type (2 vCPU, 4GB RAM — good for learning) |
| `--nodes 2` | Start with 2 worker nodes |
| `--nodes-min 1 / --nodes-max 3` | Auto-scaling range |
| `--managed` | AWS manages the nodes (updates, patching, etc.) |

> **💰 Cost Warning:** EKS costs ~$0.10/hour for the control plane + EC2 instance costs. Two `t3.medium` instances cost ~$0.08/hour. **Remember to delete the cluster when you're done learning!**

### 3.3 — Create an IAM User for GitHub Actions

GitHub Actions needs AWS credentials to push to ECR and deploy to EKS.

```bash
# Create a policy file
cat > github-actions-policy.json << 'EOF'
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": [
                "ecr:GetAuthorizationToken",
                "ecr:BatchCheckLayerAvailability",
                "ecr:GetDownloadUrlForLayer",
                "ecr:BatchGetImage",
                "ecr:PutImage",
                "ecr:InitiateLayerUpload",
                "ecr:UploadLayerPart",
                "ecr:CompleteLayerUpload"
            ],
            "Resource": "*"
        },
        {
            "Effect": "Allow",
            "Action": [
                "eks:DescribeCluster",
                "eks:ListClusters"
            ],
            "Resource": "*"
        },
        {
            "Effect": "Allow",
            "Action": [
                "sts:GetCallerIdentity"
            ],
            "Resource": "*"
        }
    ]
}
EOF

# Create the IAM policy
aws iam create-policy \
    --policy-name GitHubActionsECREKSPolicy \
    --policy-document file://github-actions-policy.json

# Create the IAM user
aws iam create-user --user-name github-actions-user

# Attach the policy to the user
aws iam attach-user-policy \
    --user-name github-actions-user \
    --policy-arn arn:aws:iam::YOUR_ACCOUNT_ID:policy/GitHubActionsECREKSPolicy

# Create access keys
aws iam create-access-key --user-name github-actions-user
# SAVE the AccessKeyId and SecretAccessKey — you'll need them!
```

**What this does:**

- Creates a limited IAM user that can **only** push to ECR and read EKS cluster info
- This follows the **principle of least privilege** — the user can't do anything else in your AWS account
- The access keys are what GitHub Actions will use to authenticate with AWS

> **🔒 Better Alternative — OIDC:** Instead of storing long-lived access keys, you can use **OpenID Connect (OIDC)** to let GitHub Actions assume an IAM role directly. This is more secure because there are no permanent credentials. We'll show this setup at the end as a bonus.

GitHub Secrets are encrypted environment variables that your workflows can use. **Never put credentials in your code!**

## ☸️ Step 5 — Create Kubernetes Manifests

These files tell Kubernetes **how to run your app**.

### 📄 `k8s/deployment.yaml` — How to Run the App

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
  labels:
    app: my-app
spec:
  replicas: 3                    # Run 3 copies of our app
  selector:
    matchLabels:
      app: my-app
  strategy:
    type: RollingUpdate          # Zero-downtime deployments
    rollingUpdate:
      maxSurge: 1                # Create 1 extra pod during update
      maxUnavailable: 0          # Never have 0 pods running
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
        - name: my-app
          image: IMAGE_PLACEHOLDER    # This will be replaced by the pipeline
          ports:
            - containerPort: 5000
          env:
            - name: ENVIRONMENT
              value: "production"
            - name: APP_VERSION
              value: "VERSION_PLACEHOLDER"    # Replaced by pipeline
          resources:
            requests:
              cpu: "100m"           # Minimum CPU (0.1 core)
              memory: "128Mi"       # Minimum memory (128MB)
            limits:
              cpu: "500m"           # Maximum CPU (0.5 core)
              memory: "256Mi"       # Maximum memory (256MB)
          livenessProbe:
            httpGet:
              path: /health
              port: 5000
            initialDelaySeconds: 10
            periodSeconds: 15
          readinessProbe:
            httpGet:
              path: /ready
              port: 5000
            initialDelaySeconds: 5
            periodSeconds: 10
```

**What's happening here? Let me explain every section:**

#### `replicas: 3`
Run 3 copies (pods) of our app. If one crashes, Kubernetes automatically creates a new one. Load is distributed across all 3.

#### `strategy: RollingUpdate`
When deploying a new version:
- `maxSurge: 1` — create 1 new pod before killing an old one
- `maxUnavailable: 0` — never have fewer than 3 pods running

This means: during deployment, you go from 3 old pods → 4 pods (3 old + 1 new) → 3 pods (2 old + 1 new) → ... → 3 new pods. **Zero downtime!**

#### `image: IMAGE_PLACEHOLDER`
We use a placeholder that our GitHub Actions pipeline will replace with the actual ECR image URL. This is a common pattern.

#### `resources`
- `requests` — the **minimum** resources guaranteed to your pod. Kubernetes uses this for scheduling.
- `limits` — the **maximum** resources your pod can use. If it exceeds memory limit, Kubernetes kills it.

#### `livenessProbe`
Kubernetes checks `/health` every 15 seconds. If it fails, Kubernetes **restarts** the pod. This catches crashed apps.

#### `readinessProbe`
Kubernetes checks `/ready` every 10 seconds. If it fails, Kubernetes **stops sending traffic** to this pod (but doesn't restart it). This is useful during startup or when the app is temporarily overloaded.

### 📄 `k8s/service.yaml` — How to Access the App

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-app-service
  labels:
    app: my-app
spec:
  type: LoadBalancer         # Creates an AWS Load Balancer
  selector:
    app: my-app              # Routes traffic to pods with label "app: my-app"
  ports:
    - protocol: TCP
      port: 80               # External port (what users access)
      targetPort: 5000        # Container port (where our app listens)
```

**What's happening here?**

- `type: LoadBalancer` — on AWS EKS, this automatically creates an **AWS Elastic Load Balancer (ELB)**. Users access your app through this load balancer.
- `port: 80` — external port. Users go to `http://load-balancer-url:80`
- `targetPort: 5000` — Kubernetes forwards traffic to port 5000 inside the container (where Flask is listening)
- `selector: app: my-app` — routes traffic to all pods with the label `app: my-app` (our 3 replicas)

## 🎬 Step 6 — Write the GitHub Actions Workflow (The Main Event!)

This is the heart of everything. Create this file at `.github/workflows/cicd.yaml`:

```yaml
name: CI/CD Pipeline - Build, Test, Push to ECR, Deploy to EKS

# ============================================
# WHEN does this pipeline run?
# ============================================
on:
  push:
    branches: [main]          # Run on push to main branch
  pull_request:
    branches: [main]          # Run on PRs targeting main (tests only)

# ============================================
# ENVIRONMENT VARIABLES (used across all jobs)
# ============================================
env:
  AWS_REGION: ${{ secrets.AWS_REGION }}
  ECR_REPOSITORY: ${{ secrets.ECR_REPOSITORY }}
  EKS_CLUSTER_NAME: ${{ secrets.EKS_CLUSTER_NAME }}

# ============================================
# JOBS — The actual work
# ============================================
jobs:

  # ==========================================
  # JOB 1: Build and Test
  # ==========================================
  build-and-test:
    name: 🔨 Build & Test
    runs-on: ubuntu-latest

    steps:
      # Step 1: Get the code
      - name: 📥 Checkout code
        uses: actions/checkout@v4

      # Step 2: Set up Python
      - name: 🐍 Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'
          cache: 'pip'           # Cache pip packages for faster builds

      # Step 3: Install dependencies
      - name: 📦 Install dependencies
        run: |
          python -m pip install --upgrade pip
          pip install -r requirements.txt

      # Step 4: Run linting (code quality)
      - name: 🔍 Run linter
        run: |
          pip install flake8
          flake8 app.py --max-line-length=120 --statistics
        continue-on-error: true    # Don't fail the pipeline on lint warnings

      # Step 5: Run tests
      - name: 🧪 Run tests
        run: |
          pytest tests/ -v --tb=short

  # ==========================================
  # JOB 2: Build Docker Image & Push to ECR
  # ==========================================
  build-and-push:
    name: 🐳 Build & Push to ECR
    runs-on: ubuntu-latest
    needs: build-and-test         # Only runs if Job 1 passes!
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    
    outputs:
      image-tag: ${{ steps.build-image.outputs.image-tag }}

    steps:
      # Step 1: Get the code
      - name: 📥 Checkout code
        uses: actions/checkout@v4

      # Step 2: Configure AWS credentials
      - name: 🔑 Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: ${{ env.AWS_REGION }}

      # Step 3: Login to ECR
      - name: 🔐 Login to Amazon ECR
        id: login-ecr
        uses: aws-actions/amazon-ecr-login@v2

      # Step 4: Build, tag, and push the Docker image
      - name: 🏗️ Build, tag, and push image to ECR
        id: build-image
        env:
          ECR_REGISTRY: ${{ steps.login-ecr.outputs.registry }}
          IMAGE_TAG: ${{ github.sha }}
        run: |
          # Build the Docker image
          docker build \
            --build-arg APP_VERSION=${{ github.sha }} \
            -t $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG \
            -t $ECR_REGISTRY/$ECR_REPOSITORY:latest \
            .

          # Push both tags to ECR
          docker push $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG
          docker push $ECR_REGISTRY/$ECR_REPOSITORY:latest

          # Save the full image URI for the deploy job
          echo "image-tag=$IMAGE_TAG" >> $GITHUB_OUTPUT
          echo "✅ Image pushed: $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG"

  # ==========================================
  # JOB 3: Deploy to EKS
  # ==========================================
  deploy:
    name: 🚀 Deploy to EKS
    runs-on: ubuntu-latest
    needs: build-and-push         # Only runs if Job 2 passes!
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'

    steps:
      # Step 1: Get the code (we need the k8s/ manifests)
      - name: 📥 Checkout code
        uses: actions/checkout@v4

      # Step 2: Configure AWS credentials
      - name: 🔑 Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: ${{ env.AWS_REGION }}

      # Step 3: Update kubeconfig to connect to EKS
      - name: ☸️ Update kubeconfig
        run: |
          aws eks update-kubeconfig \
            --name ${{ env.EKS_CLUSTER_NAME }} \
            --region ${{ env.AWS_REGION }}

      # Step 4: Replace placeholders in Kubernetes manifests
      - name: 📝 Update Kubernetes manifests
        env:
          ECR_REGISTRY: ${{ secrets.AWS_ACCOUNT_ID }}.dkr.ecr.${{ env.AWS_REGION }}.amazonaws.com
          IMAGE_TAG: ${{ needs.build-and-push.outputs.image-tag }}
        run: |
          # Replace the image placeholder with the actual ECR image URI
          sed -i "s|IMAGE_PLACEHOLDER|$ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG|g" k8s/deployment.yaml
          
          # Replace the version placeholder
          sed -i "s|VERSION_PLACEHOLDER|$IMAGE_TAG|g" k8s/deployment.yaml
          
          # Show what the final manifest looks like (for debugging)
          echo "📋 Final deployment manifest:"
          cat k8s/deployment.yaml

      # Step 5: Apply the Kubernetes manifests
      - name: 🚀 Deploy to EKS
        run: |
          kubectl apply -f k8s/deployment.yaml
          kubectl apply -f k8s/service.yaml
          
          # Wait for the deployment to complete
          kubectl rollout status deployment/my-app --timeout=300s
          
          echo "✅ Deployment successful!"

      # Step 6: Show deployment info
      - name: 📊 Show deployment status
        run: |
          echo "=== Pods ==="
          kubectl get pods -l app=my-app
          
          echo ""
          echo "=== Service ==="
          kubectl get svc my-app-service
          
          echo ""
          echo "=== Deployment ==="
          kubectl describe deployment my-app | head -30
```

## 🔍 GitHub Actions Workflow Explained — Section by Section

Let's break down every piece of this workflow:

### The Trigger (`on:`)

```yaml
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
```

- **Push to main** → runs the FULL pipeline (build, test, push, deploy)
- **PR to main** → runs ONLY the build & test job (we don't deploy PRs to production!)

The `if` condition on Job 2 and 3 (`github.ref == 'refs/heads/main' && github.event_name == 'push'`) makes sure deploy only happens on actual pushes to main, not PRs.

### Job Dependencies (`needs:`)

```yaml
jobs:
  build-and-test:     # Runs first
    ...
  
  build-and-push:
    needs: build-and-test    # Waits for build-and-test to pass
    ...
  
  deploy:
    needs: build-and-push    # Waits for build-and-push to pass
    ...
```

This creates a chain: **Test → Build Image → Deploy**. If tests fail, nothing else runs.

```
┌─────────────────┐     ┌──────────────────┐     ┌──────────────┐
│  Build & Test   │────▶│  Build & Push    │────▶│   Deploy     │
│   (Job 1)       │     │  to ECR (Job 2)  │     │  to EKS      │
└─────────────────┘     └──────────────────┘     │  (Job 3)     │
  Runs on: PRs +          Runs on: push to         └──────────────┘
  push to main            main ONLY                Runs on: push to
                                                   main ONLY
```

### Passing Data Between Jobs (`outputs:`)

```yaml
build-and-push:
  outputs:
    image-tag: ${{ steps.build-image.outputs.image-tag }}

# In the build step:
echo "image-tag=$IMAGE_TAG" >> $GITHUB_OUTPUT

# In the deploy job:
IMAGE_TAG: ${{ needs.build-and-push.outputs.image-tag }}
```

Jobs run on **different machines**, so they can't share data directly. We use `outputs` to pass the image tag from Job 2 to Job 3.

### Image Tagging with Git SHA

```yaml
IMAGE_TAG: ${{ github.sha }}
```

We use the **Git commit SHA** as the image tag (e.g., `a1b2c3d4e5...`). Why?

- Every commit gets a unique tag
- You can always trace an image back to the exact code that produced it
- Rolling back is as simple as deploying a previous SHA

### The `sed` Trick for Manifests

```bash
sed -i "s|IMAGE_PLACEHOLDER|$ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG|g" k8s/deployment.yaml
```

This replaces `IMAGE_PLACEHOLDER` in our Kubernetes manifest with the actual ECR image URL. It's a simple approach — in production, you might use Helm or Kustomize instead.

## 🏃 Step 7 — Run the Pipeline!

### 7.1 — Push Everything to GitHub

```bash
cd my-github-actions-project
```

## 🔒 Bonus — Use OIDC Instead of Access Keys (More Secure)

Instead of storing long-lived AWS access keys in GitHub, you can use **OIDC (OpenID Connect)** federation. This is the **recommended approach** for production.

### How it works:

```
GitHub Actions  ──── "I am repo X, running workflow Y" ────▶  AWS IAM
                                                                │
                                                    Checks OIDC trust policy
                                                                │
                ◀──── "OK, here's a temporary role" ────────────┘
```

No permanent credentials stored anywhere!

### Setup:

```bash
# 1. Create an OIDC identity provider in AWS
aws iam create-open-id-connect-provider \
    --url "https://token.actions.githubusercontent.com" \
    --thumbprint-list "6938fd4d98bab03faadb97b34396831e3780aea1" \
    --client-id-list "sts.amazonaws.com"

# 2. Create a trust policy
cat > trust-policy.json << 'EOF'
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Principal": {
                "Federated": "arn:aws:iam::YOUR_ACCOUNT_ID:oidc-provider/token.actions.githubusercontent.com"
            },
            "Action": "sts:AssumeRoleWithWebIdentity",
            "Condition": {
                "StringEquals": {
                    "token.actions.githubusercontent.com:aud": "sts.amazonaws.com"
                },
                "StringLike": {
                    "token.actions.githubusercontent.com:sub": "repo:yourusername/my-github-actions-project:*"
                }
            }
        }
    ]
}
EOF

# 3. Create the IAM role
aws iam create-role \
    --role-name GitHubActionsRole \
    --assume-role-policy-document file://trust-policy.json

# 4. Attach permissions to the role
aws iam attach-role-policy \
    --role-name GitHubActionsRole \
    --policy-arn arn:aws:iam::YOUR_ACCOUNT_ID:policy/GitHubActionsECREKSPolicy
```

Then update your workflow:

```yaml
- name: 🔑 Configure AWS credentials (OIDC)
  uses: aws-actions/configure-aws-credentials@v4
  with:
    role-to-assume: arn:aws:iam::YOUR_ACCOUNT_ID:role/GitHubActionsRole
    aws-region: us-east-1
```

And add this permission to the job:

```yaml
permissions:
  id-token: write
  contents: read
```

No more `AWS_ACCESS_KEY_ID` or `AWS_SECRET_ACCESS_KEY` secrets needed!

## 🐛 Common Issues & Troubleshooting

| Problem | Cause | Solution |
|---|---|---|
| `Error: No credentials configured` | AWS secrets not set or misspelled | Double-check GitHub Secrets names (case-sensitive!) |
| `AccessDeniedException` on ECR push | IAM policy missing ECR permissions | Verify the IAM policy has all ECR actions listed |
| `error: You must be logged in` (kubectl) | kubeconfig not updated | Make sure `aws eks update-kubeconfig` runs before `kubectl` |
| `ImagePullBackOff` in pods | EKS can't pull from ECR | Ensure EKS node role has `AmazonEC2ContainerRegistryReadOnly` policy |
| Deployment timeout | App crashing or health check failing | Check pod logs: `kubectl logs -l app=my-app` |
| Workflow not triggering | Branch name mismatch | Ensure you push to `main` (not `master`) |

## 🆚 GitHub Actions vs Jenkins — When to Use What?

| Aspect | GitHub Actions | Jenkins |
|---|---|---|
| **Hosting** | Managed by GitHub (no server needed) | Self-hosted (you manage the server) |
| **Setup** | Zero setup — just add a YAML file | Install Java, Jenkins, plugins, etc. |
| **Cost** | Free for public repos, 2000 min/month free for private | Free (open source) but you pay for server |
| **Config** | YAML (`.github/workflows/`) | Groovy (Jenkinsfile) |
| **Extensibility** | Actions Marketplace (pre-built) | 1800+ plugins |
| **Best For** | GitHub-hosted projects, cloud-native | Complex pipelines, on-premise, enterprise |
| **Learning Curve** | Low | Medium-High |
