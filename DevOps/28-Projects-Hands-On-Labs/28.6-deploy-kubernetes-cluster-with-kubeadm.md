## 🎯 What Are We Building?

Forget managed Kubernetes (EKS, AKS, GKE). In this project, you're going to **build a Kubernetes cluster from the ground up** — the way it's done in production environments where you manage your own infrastructure.

By the end, you'll have:

```
┌─────────────────────────────────────────────────────────┐
│                  Kubernetes Cluster                      │
│                                                          │
│  ┌──────────────────┐                                   │
│  │   Master Node     │  ← Control Plane                 │
│  │   (192.168.1.10)  │  ← API Server, etcd, Scheduler   │
│  └──────────────────┘  ← Controller Manager              │
│           │                                              │
│     ┌─────┴─────┐                                       │
│     │           │                                       │
│  ┌──┴───────┐ ┌─┴────────┐                              │
│  │ Worker 1 │ │ Worker 2  │  ← Your apps run here       │
│  │ (..1.11) │ │ (..1.12)  │  ← kubelet + container      │
│  └──────────┘ └───────────┘  ← runtime on each          │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

This is how companies like banks, telecom providers, and government agencies run Kubernetes — **on their own hardware**, not in the cloud.

## 🤔 Why kubeadm? Why Not Just Use Minikube or EKS?

Great question! Here's the difference:

| Tool | What It Does | Use Case |
|---|---|---|
| **Minikube** | Single-node cluster on your laptop | Learning, local development |
| **kind** | Kubernetes-in-Docker (single machine) | CI testing |
| **EKS/AKS/GKE** | Cloud-managed Kubernetes | Production in cloud |
| **kubeadm** | Build a real multi-node cluster yourself | Production on-premise, deep learning |

**kubeadm** is the official Kubernetes tool for bootstrapping clusters. When you use kubeadm, you understand **exactly** what's happening — every component, every certificate, every configuration. This knowledge is invaluable and frequently tested in interviews (especially CKA — Certified Kubernetes Administrator).

## 📋 Prerequisites — What You Need

### Hardware / VMs

You need **3 machines** (virtual or physical). Here's the minimum:

| Node | Role | Minimum Specs | IP Address (example) |
|---|---|---|---|
| `k8s-master` | Control Plane (Master) | 2 CPU, 2GB RAM, 20GB disk | 192.168.1.10 |
| `k8s-worker-1` | Worker Node | 2 CPU, 2GB RAM, 20GB disk | 192.168.1.11 |
| `k8s-worker-2` | Worker Node | 2 CPU, 2GB RAM, 20GB disk | 192.168.1.12 |

### How to Get These Machines

**Option 1: VirtualBox / VMware** (Free, runs on your laptop)
- Create 3 Ubuntu 22.04 VMs
- Set each to **Bridged Adapter** networking (so they can talk to each other)
- Allocate resources as above

**Option 2: Vagrant** (Automated VM creation)

```ruby

  # Master Node

    master.vm.network "private_network", ip: "192.168.56.10"
    master.vm.provider "virtualbox" do |vb|
      vb.memory = 2048

      vb.name = "k8s-master"

  end

  # Worker Node 1
  config.vm.define "k8s-worker-1" do |worker1|
    worker1.vm.box = "ubuntu/jammy64"
    worker1.vm.hostname = "k8s-worker-1"
    worker1.vm.network "private_network", ip: "192.168.56.11"
    worker1.vm.provider "virtualbox" do |vb|
      vb.memory = 2048
      vb.cpus = 2
      vb.name = "k8s-worker-1"
    end
  end

  # Worker Node 2
  config.vm.define "k8s-worker-2" do |worker2|
    worker2.vm.box = "ubuntu/jammy64"
    worker2.vm.hostname = "k8s-worker-2"
    worker2.vm.network "private_network", ip: "192.168.56.12"

      vb.memory = 2048
```

```bash

```

- Create 3 `t3.medium` (or similar) instances
- Make sure they're in the same network and can reach each other

## 🛠️ Step 1 — Prepare All 3 Nodes (Run on ALL Nodes)

> **⚠️ Important:** Steps 1-4 must be run on **ALL 3 nodes** (master + worker1 + worker2). You can SSH into each node in separate terminal tabs.

### 1.1 — Set Hostnames

On each node, set the correct hostname:

```bash
# On master:
sudo hostnamectl set-hostname k8s-master

# On worker 1:
sudo hostnamectl set-hostname k8s-worker-1

# On worker 2:
sudo hostnamectl set-hostname k8s-worker-2
```

### 1.2 — Update /etc/hosts (So Nodes Can Find Each Other by Name)

Run on **all 3 nodes**:

```bash
cat << EOF | sudo tee -a /etc/hosts

# Kubernetes cluster nodes
192.168.56.10  k8s-master
192.168.56.11  k8s-worker-1
192.168.56.12  k8s-worker-2
EOF
```

> **What's this doing?** It adds name-to-IP mappings so you can type `ping k8s-master` instead of `ping 192.168.56.10`. Like a mini DNS for your cluster.

### 1.3 — Disable Swap (Kubernetes Requirement)

Kubernetes **requires** swap to be disabled. Why? Because Kubernetes manages memory itself through resource limits and requests. Swap makes memory management unpredictable.

```bash
# Disable swap immediately
sudo swapoff -a

# Disable swap permanently (survives reboot)
sudo sed -i '/ swap / s/^/#/' /etc/fstab

# Verify swap is off (should show 0)
free -h | grep Swap
```

**Expected output:**
```
Swap:            0B          0B          0B
```

### 1.4 — Load Required Kernel Modules

Kubernetes networking needs these kernel modules to work:

```bash
# Load modules now
sudo modprobe overlay
sudo modprobe br_netfilter

# Make them load on boot
cat << EOF | sudo tee /etc/modules-load.d/k8s.conf
overlay
br_netfilter
EOF
```

**What are these modules?**

| Module | Purpose |
|---|---|
| `overlay` | Required for the overlay filesystem used by containers (Docker/containerd store image layers as overlays) |
| `br_netfilter` | Allows iptables to see bridged traffic. Without this, Kubernetes networking (Services, NetworkPolicies) won't work |

### 1.5 — Configure Networking (sysctl Parameters)

```bash
# Set required sysctl parameters
cat << EOF | sudo tee /etc/sysctl.d/k8s.conf
net.bridge.bridge-nf-call-iptables  = 1
net.bridge.bridge-nf-call-ip6tables = 1
net.ipv4.ip_forward                 = 1
EOF

# Apply the settings without rebooting
sudo sysctl --system
```

**What do these settings do?**

| Setting | Explanation |
|---|---|
| `bridge-nf-call-iptables = 1` | Makes bridged IPv4 traffic go through iptables rules. Kubernetes Services use iptables for load balancing. |
| `bridge-nf-call-ip6tables = 1` | Same thing but for IPv6 |
| `ip_forward = 1` | Allows this machine to forward network packets between interfaces. Essential for pod-to-pod communication across nodes. |

### 1.6 — Verify Networking

```bash
# All should return "1"
sysctl net.bridge.bridge-nf-call-iptables
sysctl net.bridge.bridge-nf-call-ip6tables
sysctl net.ipv4.ip_forward
```

## 🐳 Step 2 — Install Container Runtime (containerd) — All Nodes

Kubernetes needs a container runtime to actually run containers. We'll use **containerd** (the industry standard since Docker was deprecated as a Kubernetes runtime in v1.24).

### 2.1 — Install containerd

```bash
# Update package list
sudo apt update

# Install dependencies
sudo apt install -y apt-transport-https ca-certificates curl gnupg lsb-release

# Add Docker's official GPG key (containerd is distributed through Docker's repo)
sudo install -m 0755 -d /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
sudo chmod a+r /etc/apt/keyrings/docker.gpg

# Add the repository
echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] \
  https://download.docker.com/linux/ubuntu \
  $(lsb_release -cs) stable" | sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

# Install containerd
sudo apt update
sudo apt install -y containerd.io
```

### 2.2 — Configure containerd for Kubernetes

The default containerd config needs a small tweak to work with Kubernetes:

```bash
# Generate the default config
sudo mkdir -p /etc/containerd
containerd config default | sudo tee /etc/containerd/config.toml > /dev/null

# Enable SystemdCgroup (CRITICAL for Kubernetes!)
sudo sed -i 's/SystemdCgroup = false/SystemdCgroup = true/' /etc/containerd/config.toml

# Restart containerd to apply changes
sudo systemctl restart containerd
sudo systemctl enable containerd

# Verify containerd is running
sudo systemctl status containerd
```

**Why `SystemdCgroup = true`?**

This is super important and often missed! Here's the deal:

- Linux uses **cgroups** (control groups) to limit resources (CPU, memory) for processes
- There are two cgroup drivers: **cgroupfs** and **systemd**
- Modern Ubuntu uses **systemd** as the init system
- Kubernetes (kubelet) uses **systemd** cgroup driver by default
- If containerd uses **cgroupfs** while kubelet uses **systemd**, they'll fight over resource management, and your cluster will be unstable!

So we set `SystemdCgroup = true` to make containerd use the same driver as kubelet. **Both must match.**

## 📦 Step 3 — Install kubeadm, kubelet, and kubectl — All Nodes

These are the three Kubernetes tools:

| Tool | Purpose |
|---|---|
| **kubeadm** | Bootstraps the cluster (init master, join workers) |
| **kubelet** | The agent that runs on every node, manages pods |
| **kubectl** | CLI tool to interact with the cluster |

### 3.1 — Add Kubernetes Repository

```bash
# Add Kubernetes signing key
curl -fsSL https://pkgs.k8s.io/core:/stable:/v1.29/deb/Release.key | sudo gpg --dearmor -o /etc/apt/keyrings/kubernetes-apt-keyring.gpg

# Add Kubernetes repository
echo 'deb [signed-by=/etc/apt/keyrings/kubernetes-apt-keyring.gpg] https://pkgs.k8s.io/core:/stable:/v1.29/deb/ /' | sudo tee /etc/apt/sources.list.d/kubernetes.list
```

### 3.2 — Install the Tools

```bash
# Update and install
sudo apt update
sudo apt install -y kubelet kubeadm kubectl

# Prevent automatic updates (you want to control K8s version upgrades!)
sudo apt-mark hold kubelet kubeadm kubectl

# Verify installation
kubeadm version
kubelet --version
kubectl version --client
```

**Why `apt-mark hold`?**

Kubernetes version upgrades need to be done carefully (master first, then workers, one at a time). If `apt upgrade` accidentally updates kubelet on one node but not others, your cluster could break. `hold` prevents this.

### 3.3 — Enable kubelet

```bash
sudo systemctl enable kubelet
```

> **Note:** kubelet will keep restarting and crashing at this point — that's normal! It's waiting for kubeadm to give it configuration. Don't worry, it'll stabilize after we initialize the cluster.

## 👑 Step 4 — Initialize the Master Node (Run ONLY on Master)

> **⚠️ This step runs ONLY on the master node (k8s-master)!**

This is the big moment — we're creating the Kubernetes control plane!

### 4.1 — Run kubeadm init

```bash
sudo kubeadm init \
    --apiserver-advertise-address=192.168.56.10 \
    --pod-network-cidr=10.244.0.0/16 \
    --node-name=k8s-master
```

**What each flag does:**

| Flag | Meaning |
|---|---|
| `--apiserver-advertise-address` | The IP address the API server will listen on. Use the master node's IP. Workers will connect to this address. |
| `--pod-network-cidr` | The IP range for pod networking. `10.244.0.0/16` is the default for **Flannel** (the network plugin we'll install next). Different plugins use different ranges (Calico uses `192.168.0.0/16`). |
| `--node-name` | The name this node will have in the cluster |

### 4.2 — What Happens During `kubeadm init`?

When you run this command, a LOT happens behind the scenes. Let's trace it:

```
kubeadm init
    │
    ├── 1. Pre-flight checks
    │       ├── Is swap disabled? ✓
    │       ├── Are required ports open? ✓
    │       ├── Is containerd running? ✓
    │       ├── Is the CPU count >= 2? ✓
    │       └── Is there enough RAM? ✓
    │
    ├── 2. Generate certificates
    │       ├── CA certificate (root of trust)
    │       ├── API server certificate
    │       ├── kubelet client certificate
    │       ├── Front proxy certificate
    │       └── etcd server + peer certificates
    │       (All stored in /etc/kubernetes/pki/)
    │
    ├── 3. Generate kubeconfig files
    │       ├── admin.conf (for cluster admin)
    │       ├── kubelet.conf (for kubelet)
    │       ├── controller-manager.conf
    │       └── scheduler.conf
    │       (All stored in /etc/kubernetes/)
    │
    ├── 4. Generate static pod manifests
    │       ├── kube-apiserver.yaml
    │       ├── kube-controller-manager.yaml
    │       ├── kube-scheduler.yaml
    │       └── etcd.yaml
    │       (All stored in /etc/kubernetes/manifests/)
    │
    ├── 5. Start the control plane
    │       ├── kubelet reads manifests and starts pods
    │       ├── API server comes up first
    │       ├── etcd starts and API server connects
    │       ├── Controller manager starts
    │       └── Scheduler starts
    │
    ├── 6. Upload cluster configuration to etcd
    │
    └── 7. Generate bootstrap token
            └── This token is used by workers to join!
```

### 4.3 — Save the Output!

The output will look something like this:

```
Your Kubernetes control-plane has initialized successfully!

To start using your cluster, you need to run the following as a regular user:

  mkdir -p $HOME/.kube
  sudo cp -i /etc/kubernetes/admin.conf $HOME/.kube/config
  sudo chown $(id -u):$(id -g) $HOME/.kube/config

Then you can join any number of worker nodes by running the following on each as root:

  kubeadm join 192.168.56.10:6443 --token abcdef.0123456789abcdef \
      --discovery-token-ca-cert-hash sha256:1234567890abcdef1234567890abcdef1234567890abcdef1234567890abcdef
```

> **🚨 SAVE THAT `kubeadm join` COMMAND!** You'll need it to add worker nodes. If you lose it, you can regenerate it (shown later).

### 4.4 — Set Up kubectl for the Regular User

```bash
# Create the .kube directory
mkdir -p $HOME/.kube

# Copy the admin config
sudo cp -i /etc/kubernetes/admin.conf $HOME/.kube/config

# Set the correct ownership
sudo chown $(id -u):$(id -g) $HOME/.kube/config
```

**What's happening here?**

- `admin.conf` contains the credentials (certificate) to talk to the Kubernetes API server
- kubectl looks for config at `$HOME/.kube/config` by default
- Without this, every kubectl command would fail with "The connection to the server was refused"

## 🌐 Step 5 — Install a Pod Network Plugin (Run ONLY on Master)

Kubernetes doesn't come with networking built in. You need a **CNI (Container Network Interface) plugin** to enable pod-to-pod communication.

Popular options:

| Plugin | Best For | Complexity |
|---|---|---|
| **Flannel** | Simple clusters, learning | Easy |
| **Calico** | Production, network policies | Medium |
| **Cilium** | eBPF-based, advanced features | Advanced |
| **Weave** | Multi-cloud | Medium |

We'll use **Flannel** because it's the simplest and perfect for learning.

### 5.1 — Install Flannel

```bash
kubectl apply -f https://raw.githubusercontent.com/flannel-io/flannel/master/Documentation/kube-flannel.yml
```

That's it! One command. Flannel deploys as a DaemonSet (runs on every node).

### 5.2 — Wait for Everything to Come Up

```bash
# Watch the pods (Ctrl+C to stop watching)
kubectl get pods -n kube-system -w
```

Wait until **all pods** show `Running` and `1/1` Ready. This usually takes 1-2 minutes.

### 🔍 How Does Flannel Work?

```
       Node 1 (10.244.0.0/24)                Node 2 (10.244.1.0/24)
    ┌──────────────────────┐              ┌──────────────────────┐
    │  Pod A (10.244.0.5)  │              │  Pod B (10.244.1.8)  │
    │         │             │              │         │             │
    │    ┌────┴────┐        │              │    ┌────┴────┐        │
    │    │ veth    │        │              │    │  veth   │        │
    │    └────┬────┘        │              │    └────┬────┘        │
    │    ┌────┴────┐        │              │    ┌────┴────┐        │
    │    │ cni0    │        │              │    │  cni0   │        │
    │    │ bridge  │        │              │    │  bridge │        │
    │    └────┬────┘        │              │    └────┬────┘        │
    │    ┌────┴────┐        │              │    ┌────┴────┐        │
    │    │flannel.1│◄───────┼──────────────┼───▶│flannel.1│        │
    │    │ VXLAN   │        │   VXLAN      │    │ VXLAN   │        │
    │    └────┬────┘        │   Tunnel     │    └────┬────┘        │
    │    ┌────┴────┐        │              │    ┌────┴────┐        │
    │    │  eth0   │        │              │    │  eth0   │        │
    │    └─────────┘        │              │    └─────────┘        │
    └──────────────────────┘              └──────────────────────┘
```

- Each node gets its own subnet (e.g., Node 1 gets 10.244.0.0/24, Node 2 gets 10.244.1.0/24)
- Pods on the same node communicate through the `cni0` bridge
- Pods on different nodes communicate through a **VXLAN tunnel** (encapsulated in UDP packets)
- Flannel manages all of this automatically

## 🔗 Step 6 — Join Worker Nodes (Run on Worker Nodes ONLY)

Now let's add the two worker nodes to the cluster!

### 6.1 — Run the Join Command on Each Worker

SSH into **k8s-worker-1** and run the join command from the `kubeadm init` output:

```bash
sudo kubeadm join 192.168.56.10:6443 \
    --token abcdef.0123456789abcdef \
    --discovery-token-ca-cert-hash sha256:1234567890abcdef... \
    --node-name k8s-worker-1
```

Then SSH into **k8s-worker-2** and run the same command (but change the `--node-name`):

```bash
sudo kubeadm join 192.168.56.10:6443 \
    --token abcdef.0123456789abcdef \
    --discovery-token-ca-cert-hash sha256:1234567890abcdef... \
    --node-name k8s-worker-2
```

**What happens during `kubeadm join`?**

```
kubeadm join
    │
    ├── 1. Connect to API server at 192.168.56.10:6443
    │
    ├── 2. Verify the CA cert hash (prevents man-in-the-middle attacks)
    │
    ├── 3. Use the token to authenticate with the API server
    │
    ├── 4. Download cluster configuration
    │
    ├── 5. Generate kubelet config and certificates for this node
    │
    ├── 6. Start kubelet
    │       ├── Registers the node with the API server
    │       └── Starts kube-proxy (for Service networking)
    │
    └── 7. Download and start flannel (from DaemonSet)
```

### 6.2 — If You Lost the Join Command

Don't panic! You can regenerate it on the **master node**:

```bash
# Generate a new token
kubeadm token create

# Get the CA cert hash
openssl x509 -pubkey -in /etc/kubernetes/pki/ca.crt | openssl rsa -pubin -outform der 2>/dev/null | openssl dgst -sha256 -hex | sed 's/^.* //'

# Or, the easy way — print the full join command:
kubeadm token create --print-join-command
```

That last command gives you the complete join command, ready to copy-paste!

## 🔍 Step 8 — Understanding What's Running Where

Let's take a deep look at what's running on each node:

### Master Node Components

```bash
# See control plane static pods
ls /etc/kubernetes/manifests/
```

Output:
```
etcd.yaml
kube-apiserver.yaml
kube-controller-manager.yaml
kube-scheduler.yaml
```

These are **static pods** — kubelet on the master node reads these YAML files directly and runs them. They're not managed by the API server — they ARE the API server (and friends).

| Component | What It Does |
|---|---|
| **etcd** | Key-value store. ALL cluster data lives here (deployments, pods, secrets, everything). Think of it as the cluster's brain/database. |
| **kube-apiserver** | The front door. Every kubectl command, every internal request goes through here. It validates and processes API requests. |
| **kube-controller-manager** | Runs controllers (loops that watch for changes). For example: "Deployment says 3 replicas, but only 2 pods exist → create 1 more." |
| **kube-scheduler** | Decides which node a new pod should run on. Considers CPU, memory, affinity rules, taints, etc. |

### Worker Node Components

```bash
# On a worker node:
systemctl status kubelet
```

| Component | What It Does |
|---|---|
| **kubelet** | The agent on each node. Receives instructions from the API server ("run this pod") and makes it happen using containerd. Reports node status back. |
| **kube-proxy** | Handles networking for Services. Sets up iptables rules so that traffic to a Service IP gets forwarded to the right pod. |
| **containerd** | The container runtime. Actually pulls images and runs containers. |
| **flannel** | CNI plugin. Manages the network overlay so pods on different nodes can communicate. |

## 🛡️ Step 9 — Essential Cluster Operations

### 9.1 — How to Remove a Node

If you need to take a worker node out of the cluster:

```bash
# On master: drain the node (safely move pods off it)
kubectl drain k8s-worker-2 --ignore-daemonsets --delete-emptydir-data

# On master: remove the node from the cluster
kubectl delete node k8s-worker-2

# On the worker node being removed: reset kubeadm
sudo kubeadm reset
sudo rm -rf /etc/cni/net.d
sudo iptables -F && sudo iptables -t nat -F
```

**What `drain` does:**
- Marks the node as unschedulable (no new pods will be placed here)
- Safely moves existing pods to other nodes
- Waits for pods to terminate gracefully

### 9.2 — How to Re-add a Node

```bash
# On master: generate a new join command
kubeadm token create --print-join-command

# On the worker: run the join command
sudo kubeadm join 192.168.56.10:6443 --token NEW_TOKEN --discovery-token-ca-cert-hash sha256:HASH
```

### 9.3 — Backup etcd (Critical for Production!)

etcd holds ALL cluster data. If it's lost, your cluster is gone. **Always back it up.**

```bash
# Install etcdctl
sudo apt install -y etcd-client

# Create a backup
sudo ETCDCTL_API=3 etcdctl snapshot save /backup/etcd-snapshot.db \
    --endpoints=https://127.0.0.1:2379 \
    --cacert=/etc/kubernetes/pki/etcd/ca.crt \
    --cert=/etc/kubernetes/pki/etcd/server.crt \
    --key=/etc/kubernetes/pki/etcd/server.key

# Verify the backup
sudo ETCDCTL_API=3 etcdctl snapshot status /backup/etcd-snapshot.db --write-table
```

### 9.4 — Check Certificate Expiry

kubeadm certificates expire after 1 year by default:

```bash
kubeadm certs check-expiration
```

To renew:
```bash
sudo kubeadm certs renew all
sudo systemctl restart kubelet
```

## 🐛 Common Issues & Troubleshooting

| Problem | Cause | Solution |
|---|---|---|
| `kubeadm init` fails with "swap" error | Swap is enabled | Run `sudo swapoff -a` and update `/etc/fstab` |
| Node stuck in `NotReady` | CNI plugin not installed | Install Flannel/Calico (Step 5) |
| `kube-proxy` in CrashLoopBackOff | Network module not loaded | Run `modprobe br_netfilter` and check sysctl settings |
| Workers can't join | Token expired (24h default) | Generate new token: `kubeadm token create --print-join-command` |
| CoreDNS pods stuck in `Pending` | No CNI plugin installed | Install Flannel (Step 5) |
| Pods can't reach each other across nodes | Firewall blocking VXLAN | Open UDP port 8472 (Flannel) between nodes |
| `kubectl` not working on master | kubeconfig not set up | Run the `mkdir -p $HOME/.kube` commands from Step 4.4 |
| `The connection to the server was refused` | API server not running | Check: `sudo systemctl status kubelet` and container runtime |
| `error: unable to recognize` | Old kubectl version | Update kubectl: `sudo apt install -y kubectl` |

### Required Ports Checklist

Make sure these ports are open between nodes:

**Master Node:**

| Port | Component | Protocol |
|---|---|---|
| 6443 | API Server | TCP |
| 2379-2380 | etcd | TCP |
| 10250 | kubelet | TCP |
| 10259 | kube-scheduler | TCP |
| 10257 | kube-controller-manager | TCP |

**Worker Nodes:**

| Port | Component | Protocol |
|---|---|---|
| 10250 | kubelet | TCP |
| 30000-32767 | NodePort Services | TCP |
| 8472 | Flannel VXLAN | UDP |
