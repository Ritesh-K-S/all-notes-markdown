## 1. Project Overview

In this project, we'll build and deploy a **simple e-commerce microservices app** on Kubernetes. Think of it like this — instead of one big application doing everything, we break it into small, independent services:

| Service | What It Does | Port |
|---------|-------------|------|
| **frontend** | A simple web UI (Nginx serving static files) | 80 |
| **product-service** | Returns a list of products (Node.js API) | 3001 |
| **order-service** | Handles orders (Node.js API) | 3002 |
| **mongodb** | Database for storing products & orders | 27017 |

Each service runs in its own container, has its own Deployment, its own Service, and they all talk to each other **inside** the Kubernetes cluster. External users access everything through an **Ingress**.

## 4. Architecture Overview

Here's the big picture of what we're building:

```
                    ┌─────────────────────────┐
                    │       INTERNET           │
                    └───────────┬──────────────┘
                                │
                    ┌───────────▼──────────────┐
                    │    INGRESS CONTROLLER     │
                    │    (Nginx Ingress)        │
                    └───────────┬──────────────┘
                                │
              ┌─────────────────┼─────────────────┐
              │                 │                  │
    /         │     /api/products│    /api/orders   │
              │                 │                  │
    ┌─────────▼──┐   ┌─────────▼──┐   ┌──────────▼──┐
    │  Frontend  │   │  Product   │   │   Order     │
    │  Service   │   │  Service   │   │   Service   │
    │  (ClusterIP)   │  (ClusterIP)   │  (ClusterIP) │
    └─────┬──────┘   └─────┬──────┘   └──────┬──────┘
          │                │                  │
    ┌─────▼──────┐   ┌─────▼──────┐   ┌──────▼──────┐
    │  Frontend  │   │  Product   │   │   Order     │
    │  Pods (2)  │   │  Pods (2)  │   │   Pods (2)  │
    └────────────┘   └─────┬──────┘   └──────┬──────┘
                           │                  │
                    ┌──────▼──────────────────▼──┐
                    │      MongoDB Service       │
                    │      (ClusterIP)           │
                    └───────────┬────────────────┘
                    ┌───────────▼────────────────┐
                    │      MongoDB Pod           │
                    └────────────────────────────┘
```

**How the traffic flows:**
1. User hits `myapp.local/api/products` in their browser
2. The **Ingress Controller** receives the request
3. Based on the **path** (`/api/products`), it routes to the correct **Service**
4. The **Service** load-balances across all healthy **Pods**
5. The **Pod** processes the request and talks to **MongoDB** if needed
6. Response flows back to the user

## Step 1 — Set Up Your Kubernetes Cluster

### Option A: Using Minikube (Recommended for This Lab)

```bash
# Start Minikube with enough resources
minikube start --cpus=2 --memory=4096 --driver=docker

# Enable the Ingress addon (this installs Nginx Ingress Controller for us!)
minikube addons enable ingress

# Enable metrics-server (needed for HPA later)
minikube addons enable metrics-server

# Verify everything is enabled
minikube addons list | grep -E "ingress|metrics"
```

You should see:
```
| ingress                     | minikube | enabled ✅  |
| metrics-server              | minikube | enabled ✅  |
```

### Option B: Using Kind

```bash
# Create a Kind cluster with port mapping for Ingress
cat <<EOF | kind create cluster --config=-
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
nodes:
- role: control-plane
  kubeadmConfigPatches:
  - |
    kind: InitConfiguration
    nodeRegistration:
      kubeletExtraArgs:
        node-labels: "ingress-ready=true"
  extraPortMappings:
  - containerPort: 80
    hostPort: 80
    protocol: TCP
  - containerPort: 443
    hostPort: 443
    protocol: TCP
EOF

# Install Nginx Ingress Controller for Kind
kubectl apply -f https://raw.githubusercontent.com/kubernetes/ingress-nginx/main/deploy/static/provider/kind/deploy.yaml

# Wait for it to be ready
kubectl wait --namespace ingress-nginx \
  --for=condition=ready pod \
  --selector=app.kubernetes.io/component=controller \
  --timeout=90s
```

### 2.1 — Product Service (Node.js)

**File: `product-service/server.js`**

```javascript
// product-service/server.js
// A simple REST API that serves product data

const express = require('express');
const app = express();
const PORT = 3001;

// In a real app, this would come from MongoDB
// For simplicity, we'll use in-memory data first, then add MongoDB
const products = [
  { id: 1, name: 'Laptop', price: 999, category: 'Electronics' },
  { id: 2, name: 'Headphones', price: 149, category: 'Electronics' },
  { id: 3, name: 'Backpack', price: 79, category: 'Accessories' },
  { id: 4, name: 'Keyboard', price: 129, category: 'Electronics' },
  { id: 5, name: 'Mouse', price: 49, category: 'Electronics' }
];

// Health check endpoint — Kubernetes uses this to know if the pod is healthy
app.get('/health', (req, res) => {
  res.status(200).json({ status: 'healthy', service: 'product-service' });
});

// Get all products
app.get('/api/products', (req, res) => {
  console.log(`[${new Date().toISOString()}] GET /api/products — serving ${products.length} products`);
  res.json({
    service: 'product-service',
    pod: process.env.HOSTNAME,  // This shows which pod handled the request!
    data: products
  });
});

// Get a single product by ID
app.get('/api/products/:id', (req, res) => {
  const product = products.find(p => p.id === parseInt(req.params.id));
  if (!product) {
    return res.status(404).json({ error: 'Product not found' });
  }
  res.json({ service: 'product-service', pod: process.env.HOSTNAME, data: product });
});

app.listen(PORT, () => {
  console.log(`🛒 Product Service running on port ${PORT}`);
  console.log(`   Pod name: ${process.env.HOSTNAME}`);
});
```

### 2.2 — Order Service (Node.js)

**File: `order-service/server.js`**

```javascript
// order-service/server.js
// A simple REST API that handles orders

const express = require('express');
const app = express();
const PORT = 3002;

app.use(express.json()); // Parse JSON request bodies

// In-memory orders (in production, this would be in a database)
let orders = [
  { id: 1, productId: 1, quantity: 2, status: 'completed', total: 1998 },
  { id: 2, productId: 3, quantity: 1, status: 'pending', total: 79 }
];
let nextOrderId = 3;

// Health check
app.get('/health', (req, res) => {
  res.status(200).json({ status: 'healthy', service: 'order-service' });
});

// Get all orders
app.get('/api/orders', (req, res) => {
  console.log(`[${new Date().toISOString()}] GET /api/orders — serving ${orders.length} orders`);
  res.json({
    service: 'order-service',
    pod: process.env.HOSTNAME,
    data: orders
  });
});

// Create a new order
app.post('/api/orders', (req, res) => {
  const { productId, quantity, total } = req.body;

  if (!productId || !quantity) {
    return res.status(400).json({ error: 'productId and quantity are required' });
  }

  const newOrder = {
    id: nextOrderId++,
    productId,
    quantity,
    total: total || 0,
    status: 'pending'
  };

  orders.push(newOrder);
  console.log(`[${new Date().toISOString()}] POST /api/orders — created order #${newOrder.id}`);

  res.status(201).json({
    service: 'order-service',
    pod: process.env.HOSTNAME,
    data: newOrder
  });
});

// Get a single order
app.get('/api/orders/:id', (req, res) => {
  const order = orders.find(o => o.id === parseInt(req.params.id));
  if (!order) {
    return res.status(404).json({ error: 'Order not found' });
  }
  res.json({ service: 'order-service', pod: process.env.HOSTNAME, data: order });
});

app.listen(PORT, () => {
  console.log(`📦 Order Service running on port ${PORT}`);
  console.log(`   Pod name: ${process.env.HOSTNAME}`);
});
```

## Step 5 — Create Deployments

A **Deployment** tells Kubernetes: "I want X copies of this container running at all times." If a pod crashes, the Deployment automatically creates a new one. Magic! ✨

### 5.1 — Product Service Deployment

**File: `k8s/01-product-deployment.yaml`**

```yaml
# k8s/01-product-deployment.yaml

apiVersion: apps/v1       # API version for Deployments
kind: Deployment          # What kind of resource? A Deployment!
metadata:
  name: product-service                # Name of this Deployment
  namespace: microservices-demo        # Put it in our namespace
  labels:
    app: product-service               # Labels help us identify resources
    tier: backend
spec:
  replicas: 2             # Run 2 copies of this pod (for high availability!)
  
  selector:
    matchLabels:
      app: product-service  # The Deployment manages pods with THIS label
  
  template:                # This is the Pod template — what goes inside each pod
    metadata:
      labels:
        app: product-service  # Pods get this label (MUST match selector above!)
        tier: backend
    spec:
      containers:
      - name: product-service          # Container name
        image: product-service:v1      # Docker image we built earlier
        imagePullPolicy: Never         # Don't try to pull from Docker Hub (it's local!)
        
        ports:
        - containerPort: 3001          # The port our app listens on
          name: http                   # Give the port a name (best practice)
        
        # Environment variables for our app
        env:
        - name: NODE_ENV
          value: "production"
        - name: PORT
          value: "3001"
        
        # Resource limits — VERY important in production!
        resources:
          requests:                    # Minimum resources guaranteed
            cpu: "100m"                # 100 millicores = 0.1 CPU
            memory: "128Mi"            # 128 MB RAM
          limits:                      # Maximum resources allowed
            cpu: "250m"                # 250 millicores = 0.25 CPU
            memory: "256Mi"            # 256 MB RAM
        
        # Liveness probe — "Is the container still alive?"
        # If this fails, Kubernetes RESTARTS the container
        livenessProbe:
          httpGet:
            path: /health              # Hit our health endpoint
            port: 3001
          initialDelaySeconds: 10      # Wait 10s after container starts
          periodSeconds: 15            # Check every 15 seconds
          failureThreshold: 3          # 3 failures = restart
        
        # Readiness probe — "Is the container ready to receive traffic?"
        # If this fails, the pod is REMOVED from the Service (no traffic sent)
        readinessProbe:
          httpGet:
            path: /health
            port: 3001
          initialDelaySeconds: 5       # Wait 5s before first check
          periodSeconds: 10            # Check every 10 seconds
          failureThreshold: 3          # 3 failures = stop sending traffic
```

**Let's break down the key parts:**

| Section | What It Does |
|---------|-------------|
| `replicas: 2` | Runs 2 copies of the pod. If one crashes, the other keeps serving traffic |
| `selector.matchLabels` | Tells the Deployment which pods it "owns" |
| `template` | The blueprint for each pod |
| `imagePullPolicy: Never` | Uses the local image we built (not from Docker Hub) |
| `resources.requests` | "Give me at least this much CPU/memory" |
| `resources.limits` | "Never let me use more than this" |
| `livenessProbe` | Kubernetes restarts the container if it's stuck/dead |
| `readinessProbe` | Kubernetes stops sending traffic if the pod isn't ready |

## Step 6 — Create Services

Pods are **temporary** — they come and go. So how do other pods find them? That's what a **Service** does. A Service gives pods a **stable network address**.

Think of it like this:
- **Pods** = Workers (they might leave, new ones join)
- **Service** = The reception desk (always the same address, routes you to available workers)

### 6.1 — Product Service (ClusterIP)

**File: `k8s/04-product-service.yaml`**

```yaml
# k8s/04-product-service.yaml

apiVersion: v1
kind: Service
metadata:
  name: product-service              # This becomes a DNS name inside the cluster!
  namespace: microservices-demo       # Other pods can reach it at:
  labels:                             # product-service.microservices-demo.svc.cluster.local
    app: product-service
spec:
  type: ClusterIP                    # Only accessible INSIDE the cluster
  
  selector:
    app: product-service             # Route traffic to pods with this label
  
  ports:
  - name: http
    port: 80                         # Port the Service listens on
    targetPort: 3001                 # Port on the pod to forward to
    protocol: TCP
```

**Wait, `port: 80` but our app runs on `3001`?**

Yes! Here's how it works:
- Other pods call `product-service:80` (the Service port)
- The Service forwards traffic to `pod:3001` (the targetPort)
- This way, consumers don't need to know the actual port of the app

```
Other Pod → product-service:80 → [Service] → pod-ip:3001
```

## Step 8 — Configure Ingress Rules

Now let's define **how** external traffic gets routed to our services.

**File: `k8s/07-ingress.yaml`**

```yaml
# k8s/07-ingress.yaml

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: microservices-ingress
  namespace: microservices-demo
  annotations:
    # These annotations configure the Nginx Ingress Controller
    nginx.ingress.kubernetes.io/rewrite-target: /$2    # Important! See explanation below
    nginx.ingress.kubernetes.io/use-regex: "true"       # Enable regex in paths
spec:
  ingressClassName: nginx              # Use the Nginx Ingress Controller
  
  rules:
  - host: myapp.local                  # The hostname (we'll add this to /etc/hosts)
    http:
      paths:
      # Route /api/products/* → product-service
      - path: /api/products(/|$)(.*)
        pathType: ImplementationSpecific
        backend:
          service:
            name: product-service
            port:
              number: 80
      
      # Route /api/orders/* → order-service
      - path: /api/orders(/|$)(.*)
        pathType: ImplementationSpecific
        backend:
          service:
            name: order-service
            port:
              number: 80
      
      # Route everything else → frontend
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend
            port:
              number: 80
```

**Understanding the `rewrite-target` annotation:**

Without rewrite, when someone hits `myapp.local/api/products`, the Ingress sends `/api/products` to the product-service. But our product-service expects just `/api/products` — so it works fine in this case.

But if you had different path prefixes, `rewrite-target: /$2` would strip the prefix using the regex capture group. Here's how it works:

```
User requests:            What the service receives:
/api/products             → /api/products  (captured by $2 = api/products... wait)
```

Actually, let's simplify our Ingress. Since our services already handle `/api/products` and `/api/orders` paths, we can use a simpler config:

**Simpler version of `k8s/07-ingress.yaml`:**

```yaml
# k8s/07-ingress.yaml (simpler version — use this one!)

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: microservices-ingress
  namespace: microservices-demo
  annotations:
    nginx.ingress.kubernetes.io/ssl-redirect: "false"   # No HTTPS for local dev
spec:
  ingressClassName: nginx
  
  rules:
  - host: myapp.local
    http:
      paths:
      # /api/products → product-service
      - path: /api/products
        pathType: Prefix
        backend:
          service:
            name: product-service
            port:
              number: 80
      
      # /api/orders → order-service  
      - path: /api/orders
        pathType: Prefix
        backend:
          service:
            name: order-service
            port:
              number: 80
      
      # Everything else → frontend
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend
            port:
              number: 80
```

## Step 9 — Set Up Horizontal Pod Autoscaler (HPA)

The **HPA** automatically scales your pods based on CPU (or memory, or custom metrics). When traffic goes up, it adds more pods. When traffic goes down, it removes them.

### 9.1 — Make Sure Metrics Server is Running

HPA needs the Metrics Server to know how much CPU/memory pods are using.

```bash

minikube addons enable metrics-server

# Verify it's running
kubectl get pods -n kube-system | grep metrics
```

```
metrics-server-7b97f8cd5-abc12   1/1     Running   0   5m
```

```bash
# Check if metrics are being collected (might take 1-2 minutes to start)
kubectl top pods -n microservices-demo
```

```
NAME                               CPU(cores)   MEMORY(bytes)
frontend-7b9c4d5f8-abc12          1m           5Mi
frontend-7b9c4d5f8-def34          1m           5Mi
order-service-6f8d4c3b7-ghi56     2m           30Mi
order-service-6f8d4c3b7-jkl78     2m           30Mi
product-service-5a7b3c2d9-mno90   2m           28Mi
product-service-5a7b3c2d9-pqr12   3m           29Mi
```

### 9.2 — Create HPA for Product Service

**File: `k8s/08-product-hpa.yaml`**

```yaml
# k8s/08-product-hpa.yaml

apiVersion: autoscaling/v2          # v2 gives us more options than v1
kind: HorizontalPodAutoscaler
metadata:
  name: product-service-hpa
  namespace: microservices-demo
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: product-service             # Which Deployment to scale
  
  minReplicas: 2                      # Never go below 2 pods
  maxReplicas: 10                     # Never go above 10 pods
  
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 50        # Scale up when average CPU > 50%
  
  behavior:                           # Fine-tune scaling behavior
    scaleUp:
      stabilizationWindowSeconds: 30  # Wait 30s before scaling up again
      policies:
      - type: Pods
        value: 2                      # Add max 2 pods at a time
        periodSeconds: 60
    scaleDown:
      stabilizationWindowSeconds: 120 # Wait 2 minutes before scaling down
      policies:
      - type: Pods
        value: 1                      # Remove max 1 pod at a time
        periodSeconds: 60
```

**What does this mean in plain English?**

- "Hey Kubernetes, watch the `product-service` Deployment"
- "Keep at least 2 pods, max 10"
- "If the **average CPU** across all pods goes above **50%**, add more pods"
- "When scaling up, add max 2 pods every 60 seconds"
- "When scaling down, remove max 1 pod every 60 seconds"
- "Wait 2 minutes before scaling down (so we don't yo-yo)"

### 9.5 — Test the Autoscaler — Generate Load!

Let's generate some load and watch the HPA in action:

```bash
# In Terminal 1: Watch the HPA (updates every 15 seconds)
kubectl get hpa -n microservices-demo --watch

# In Terminal 2: Watch the pods
kubectl get pods -n microservices-demo --watch

# In Terminal 3: Generate load with a load test
kubectl run load-generator --rm -it \
  --image=busybox \
  -n microservices-demo \
  -- sh -c "while true; do wget -q -O- http://product-service/api/products > /dev/null 2>&1; done"
```

After 1-2 minutes, you should see:

```
NAME                  REFERENCE                    TARGETS    MINPODS   MAXPODS   REPLICAS
product-service-hpa   Deployment/product-service   78%/50%    2         10        2
# ... wait a bit ...
product-service-hpa   Deployment/product-service   78%/50%    2         10        4
# ... it scaled up! ...
product-service-hpa   Deployment/product-service   45%/50%    2         10        4
```

Stop the load generator (Ctrl+C) and after ~2 minutes, it'll scale back down to 2 pods.

> **💡 HPA Decision Formula:**
> ```
> desiredReplicas = ceil(currentReplicas × (currentMetricValue / desiredMetricValue))
> 
> Example: If CPU is 78% and target is 50%:
> desiredReplicas = ceil(2 × (78/50)) = ceil(2 × 1.56) = ceil(3.12) = 4
> ```

## Common Mistakes & Troubleshooting

### ❌ Pods stuck in `ImagePullBackOff`

**Cause:** Kubernetes can't find the Docker image.

**Fix:**
```bash
# If using Minikube, make sure you're using Minikube's Docker:
eval $(minikube docker-env)

# Rebuild the image
docker build -t product-service:v1 ./product-service/

# Make sure imagePullPolicy is set to Never in your Deployment
```

### ❌ Pods stuck in `CrashLoopBackOff`

**Cause:** Your app is crashing.

**Fix:**
```bash
# Check the logs — this will tell you WHY it's crashing
kubectl logs <pod-name> -n microservices-demo

# Common causes:
# - Wrong port in the container
# - Missing environment variables
# - Application errors
```

### ❌ Ingress returns 404

**Cause:** Ingress rules don't match, or the Ingress Controller isn't running.

**Fix:**
```bash
# Check if Ingress Controller is running
kubectl get pods -n ingress-nginx

# Check Ingress configuration
kubectl describe ingress microservices-ingress -n microservices-demo

# Make sure the host in /etc/hosts matches the Ingress host
```

### ❌ HPA shows `<unknown>/50%`

**Cause:** Metrics Server isn't running or resource requests aren't set.

**Fix:**
```bash
# Check Metrics Server
kubectl get pods -n kube-system | grep metrics

# Make sure your Deployment has resource requests defined!
# HPA can't calculate percentages without knowing the "100%"
```

### ❌ Services can't reach pods

**Cause:** Labels don't match between Service selector and Pod labels.

**Fix:**
```bash
# Check the Service selector
kubectl describe service product-service -n microservices-demo

# Check pod labels
kubectl get pods -n microservices-demo --show-labels

# The Service selector MUST match the pod labels!
```
