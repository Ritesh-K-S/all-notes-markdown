## 1. Project Overview

Remember how in project 28.7 we had 9 separate YAML files for our Kubernetes deployment? Imagine having 50+ YAML files for a real production app, and you need to deploy the same app to dev, staging, and production with different configurations...

That's where **Helm** comes in!

We'll create a Helm chart that:
- Packages all Kubernetes resources into a single, reusable chart
- Uses **templates** with variables so the same chart works for any environment
- Supports **one-command deployment** and **easy rollbacks**
- Can be shared and versioned like any software package

## 4. What is Helm? — The Big Picture

### The Problem Without Helm

```
Without Helm:
├── Deploy to dev    → Edit 10 YAML files, change values, kubectl apply
├── Deploy to staging → Edit 10 YAML files AGAIN, change values, kubectl apply
├── Deploy to prod   → Edit 10 YAML files AGAIN, pray nothing is wrong
├── Need to rollback → "Which version was running before? 😰"
└── Share with team  → "Here's a ZIP of YAML files, good luck"
```

### The Solution With Helm

```
With Helm:
├── Deploy to dev     → helm install myapp ./chart -f dev-values.yaml
├── Deploy to staging → helm install myapp ./chart -f staging-values.yaml
├── Deploy to prod    → helm install myapp ./chart -f prod-values.yaml
├── Need to rollback  → helm rollback myapp 1
└── Share with team   → helm repo add myrepo https://charts.example.com
```

### Helm Terminology

| Term | What It Is | Real-World Analogy |
|------|-----------|-------------------|
| **Chart** | A package of Kubernetes templates + values | Like an `apt` package or `npm` module |
| **Release** | A running instance of a chart in your cluster | Like an installed app on your phone |
| **Repository** | A place to store and share charts | Like Docker Hub, but for Helm charts |
| **Values** | Configuration parameters for a chart | Like `.env` files for your app |
| **Template** | A Kubernetes YAML with placeholders (Go templates) | Like a form with blank fields to fill in |
| **Revision** | A version of a release (each upgrade creates one) | Like Git commits for your deployment |

## Step 3 — Understand the Chart Structure

Here's what each file does:

```
mywebapp/
│
├── Chart.yaml            ← WHO am I? (name, version, description)
│
├── values.yaml           ← WHAT are my defaults? (replicas, image, ports)
│
├── charts/               ← WHO do I depend on? (MySQL, Redis charts)
│
├── templates/            ← HOW do I create Kubernetes resources?
│   │
│   ├── _helpers.tpl      ← Reusable template snippets (like functions)
│   │                        Files starting with _ are NOT rendered as K8s resources
│   │
│   ├── deployment.yaml   ← Template for Kubernetes Deployment
│   ├── service.yaml      ← Template for Kubernetes Service
│   ├── ingress.yaml      ← Template for Kubernetes Ingress
│   ├── configmap.yaml    ← Template for ConfigMap
│   ├── hpa.yaml          ← Template for HPA
│   │
│   ├── NOTES.txt         ← Message shown after `helm install`
│   │
│   └── tests/            ← Helm test templates
│       └── test-connection.yaml
│
└── .helmignore           ← Files to ignore when packaging (like .gitignore)
```

### The Template Rendering Pipeline

```
values.yaml          Chart.yaml
     │                   │
     ▼                   ▼
┌─────────────────────────────┐
│     Template Engine          │
│   (Go Template + Sprig)      │
│                              │
│  deployment.yaml (template)  │──▶  deployment.yaml (rendered K8s manifest)
│  service.yaml    (template)  │──▶  service.yaml    (rendered K8s manifest)
│  ingress.yaml    (template)  │──▶  ingress.yaml    (rendered K8s manifest)
└─────────────────────────────┘
```

Helm takes your templates, fills in the values from `values.yaml`, and produces regular Kubernetes YAML files. Then it applies them to the cluster. Simple!

## Step 4 — Customize Chart.yaml

**File: `mywebapp/Chart.yaml`**

```yaml
# Chart.yaml — Metadata about this Helm chart

apiVersion: v2                    # v2 = Helm 3 (v1 = old Helm 2, don't use)

name: mywebapp                    # Chart name (used in helm install)
description: A Helm chart for deploying a web application with backend API

type: application                 # "application" = deployable app
                                  # "library" = reusable template snippets only

version: 0.1.0                   # Chart version (follows Semantic Versioning)
                                  # Bump this when you change the CHART itself
                                  # (templates, values structure, etc.)

appVersion: "1.0.0"              # Version of the APPLICATION being deployed
                                  # This is informational — shown in `helm list`

# Optional metadata
keywords:
  - web
  - microservices
  - demo

maintainers:
  - name: Your Name
    email: your.email@example.com

home: https://github.com/yourusername/mywebapp
sources:
  - https://github.com/yourusername/mywebapp
```

> **💡 version vs appVersion:**
> - `version: 0.1.0` — The chart's version. Bump when you change templates/values
> - `appVersion: "1.0.0"` — Your app's version. Bump when you release a new app version
> - They're independent. You can have chart v3.2.1 deploying app v1.0.0

## Step 5 — Master values.yaml

This is the **most important file** in your chart. It defines all the configurable parameters.

**File: `mywebapp/values.yaml`**

```yaml
# values.yaml — Default configuration values
# Users can override these with:
#   --set key=value
#   -f custom-values.yaml

# ============================================
# APPLICATION SETTINGS
# ============================================

# Number of pod replicas
replicaCount: 2

# Container image configuration
image:
  repository: nginx                # Docker image name
  tag: "1.25-alpine"               # Image tag (defaults to Chart.appVersion if empty)
  pullPolicy: IfNotPresent         # Always, Never, or IfNotPresent

# Image pull secrets (for private registries)
imagePullSecrets: []
# Example:
# imagePullSecrets:
#   - name: my-registry-secret

# Override the chart name (usually leave empty)
nameOverride: ""
fullnameOverride: ""

# ============================================
# SERVICE ACCOUNT
# ============================================

serviceAccount:
  create: true                     # Create a ServiceAccount?
  automount: true                  # Auto-mount API credentials?
  annotations: {}                  # Extra annotations
  name: ""                         # Override name (auto-generated if empty)

# ============================================
# POD CONFIGURATION
# ============================================

# Extra labels for pods
podLabels: {}

# Pod annotations (useful for Prometheus scraping, etc.)
podAnnotations: {}
# Example:
# podAnnotations:
#   prometheus.io/scrape: "true"
#   prometheus.io/port: "8080"

# Pod security context (runs at the pod level)
podSecurityContext: {}
# Example:
# podSecurityContext:
#   fsGroup: 1000
#   runAsNonRoot: true

# Container security context
securityContext: {}
# Example:
# securityContext:
#   readOnlyRootFilesystem: true
#   runAsUser: 1000
#   allowPrivilegeEscalation: false

# ============================================
# CONTAINER SETTINGS
# ============================================

# Container port
containerPort: 80

# Environment variables
env: []
# Example:
# env:
#   - name: NODE_ENV
#     value: "production"
#   - name: DB_HOST
#     value: "mysql.default.svc.cluster.local"

# ============================================
# SERVICE
# ============================================

service:
  type: ClusterIP                  # ClusterIP, NodePort, or LoadBalancer
  port: 80                         # Service port
  targetPort: 80                   # Container port to forward to

# ============================================
# INGRESS
# ============================================

ingress:
  enabled: false                   # Set to true to create an Ingress
  className: "nginx"               # Ingress class name
  annotations: {}
  # Example:
  # annotations:
  #   nginx.ingress.kubernetes.io/ssl-redirect: "false"
  hosts:
    - host: mywebapp.local
      paths:
        - path: /
          pathType: Prefix
  tls: []
  # Example:
  # tls:
  #   - secretName: mywebapp-tls
  #     hosts:
  #       - mywebapp.local

# ============================================
# RESOURCES
# ============================================

resources:
  requests:
    cpu: 100m
    memory: 128Mi
  limits:
    cpu: 250m
    memory: 256Mi

# ============================================
# HEALTH CHECKS
# ============================================

livenessProbe:
  enabled: true
  path: /
  port: http
  initialDelaySeconds: 10
  periodSeconds: 15
  failureThreshold: 3

readinessProbe:
  enabled: true
  path: /
  port: http
  initialDelaySeconds: 5
  periodSeconds: 10
  failureThreshold: 3

# ============================================
# AUTOSCALING (HPA)
# ============================================

autoscaling:
  enabled: false                   # Set to true to create HPA
  minReplicas: 2
  maxReplicas: 10
  targetCPUUtilizationPercentage: 70
  targetMemoryUtilizationPercentage: 80

# ============================================
# CONFIGMAP
# ============================================

config:
  enabled: true
  data:
    APP_NAME: "My Web Application"
    LOG_LEVEL: "info"
    ENVIRONMENT: "development"

# ============================================
# NODE SCHEDULING
# ============================================

nodeSelector: {}
tolerations: []
affinity: {}
```

> **💡 Why so many options?**
> The beauty of `values.yaml` is that it makes your chart flexible. The same chart can deploy:
> - A tiny dev setup: `replicaCount: 1`, `resources.requests.cpu: 50m`
> - A beefy production setup: `replicaCount: 10`, HPA enabled, Ingress with TLS

## Step 6 — Create Deployment Template

Now the fun part — writing templates! Templates use **Go template syntax** (`{{ }}`) to insert values.

**File: `mywebapp/templates/deployment.yaml`**

```yaml
# templates/deployment.yaml
# This template creates a Kubernetes Deployment

apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ include "mywebapp.fullname" . }}
  # ↑ This calls a helper function that generates a unique name
  # The "." (dot) passes the current context (all values + chart info)
  
  labels:
    {{- include "mywebapp.labels" . | nindent 4 }}
    # ↑ Include common labels (defined in _helpers.tpl)
    # nindent 4 = add newline + indent 4 spaces
    # The "-" after {{ removes whitespace before the tag

spec:
  {{- if not .Values.autoscaling.enabled }}
  replicas: {{ .Values.replicaCount }}
  # ↑ Only set replicas if HPA is NOT enabled
  # (HPA controls replicas when enabled)
  {{- end }}

  selector:
    matchLabels:
      {{- include "mywebapp.selectorLabels" . | nindent 6 }}

  template:
    metadata:
      {{- with .Values.podAnnotations }}
      annotations:
        {{- toYaml . | nindent 8 }}
      {{- end }}
      # ↑ "with" changes the scope — inside this block, "." refers to podAnnotations
      
      labels:
        {{- include "mywebapp.labels" . | nindent 8 }}
        {{- with .Values.podLabels }}
        {{- toYaml . | nindent 8 }}
        {{- end }}

    spec:
      {{- with .Values.imagePullSecrets }}
      imagePullSecrets:
        {{- toYaml . | nindent 8 }}
      {{- end }}

      serviceAccountName: {{ include "mywebapp.serviceAccountName" . }}

      {{- with .Values.podSecurityContext }}
      securityContext:
        {{- toYaml . | nindent 8 }}
      {{- end }}

      containers:
        - name: {{ .Chart.Name }}
          # ↑ .Chart.Name comes from Chart.yaml

          {{- with .Values.securityContext }}
          securityContext:
            {{- toYaml . | nindent 12 }}
          {{- end }}

          image: "{{ .Values.image.repository }}:{{ .Values.image.tag | default .Chart.AppVersion }}"
          # ↑ If image.tag is empty, use appVersion from Chart.yaml
          # Example result: "nginx:1.25-alpine"

          imagePullPolicy: {{ .Values.image.pullPolicy }}

          ports:
            - name: http
              containerPort: {{ .Values.containerPort }}
              protocol: TCP

          {{- if .Values.env }}
          env:
            {{- toYaml .Values.env | nindent 12 }}
          {{- end }}

          # Mount ConfigMap as environment variables
          {{- if .Values.config.enabled }}
          envFrom:
            - configMapRef:
                name: {{ include "mywebapp.fullname" . }}-config
          {{- end }}

          {{- if .Values.livenessProbe.enabled }}
          livenessProbe:
            httpGet:
              path: {{ .Values.livenessProbe.path }}
              port: {{ .Values.livenessProbe.port }}
            initialDelaySeconds: {{ .Values.livenessProbe.initialDelaySeconds }}
            periodSeconds: {{ .Values.livenessProbe.periodSeconds }}
            failureThreshold: {{ .Values.livenessProbe.failureThreshold }}
          {{- end }}

          {{- if .Values.readinessProbe.enabled }}
          readinessProbe:
            httpGet:
              path: {{ .Values.readinessProbe.path }}
              port: {{ .Values.readinessProbe.port }}
            initialDelaySeconds: {{ .Values.readinessProbe.initialDelaySeconds }}
            periodSeconds: {{ .Values.readinessProbe.periodSeconds }}
            failureThreshold: {{ .Values.readinessProbe.failureThreshold }}
          {{- end }}

          resources:
            {{- toYaml .Values.resources | nindent 12 }}

      {{- with .Values.nodeSelector }}
      nodeSelector:
        {{- toYaml . | nindent 8 }}
      {{- end }}

      {{- with .Values.affinity }}
      affinity:
        {{- toYaml . | nindent 8 }}
      {{- end }}

      {{- with .Values.tolerations }}
      tolerations:
        {{- toYaml . | nindent 8 }}
      {{- end }}
```

### Understanding Template Syntax

Here's a quick reference of the Go template syntax used above:

| Syntax | What It Does | Example |
|--------|-------------|---------|
| `{{ .Values.replicaCount }}` | Insert a value from values.yaml | `2` |
| `{{ .Chart.Name }}` | Insert from Chart.yaml | `mywebapp` |
| `{{ .Release.Name }}` | Name given during `helm install` | `my-release` |
| `{{ .Release.Namespace }}` | Namespace deployed to | `default` |
| `{{ include "template" . }}` | Include a named template | (helper output) |
| `{{- ... }}` | Trim whitespace before | Cleaner YAML |
| `{{ ... -}}` | Trim whitespace after | Cleaner YAML |
| `{{ toYaml .Values.x \| nindent 4 }}` | Convert to YAML + indent | Formatted block |
| `{{ if ... }} ... {{ end }}` | Conditional | Include/exclude sections |
| `{{ with ... }} ... {{ end }}` | Change scope | Simplify nested access |
| `{{ range ... }} ... {{ end }}` | Loop | Iterate over lists/maps |
| `{{ default "val" .Values.x }}` | Default value | Fallback if not set |

## Step 7 — Create Service Template

**File: `mywebapp/templates/service.yaml`**

```yaml
# templates/service.yaml

apiVersion: v1
kind: Service
metadata:
  name: {{ include "mywebapp.fullname" . }}
  labels:
    {{- include "mywebapp.labels" . | nindent 4 }}
spec:
  type: {{ .Values.service.type }}
  ports:
    - port: {{ .Values.service.port }}
      targetPort: {{ .Values.service.targetPort }}
      protocol: TCP
      name: http
      {{- if eq .Values.service.type "NodePort" }}
      {{- if .Values.service.nodePort }}
      nodePort: {{ .Values.service.nodePort }}
      {{- end }}
      {{- end }}
  selector:
    {{- include "mywebapp.selectorLabels" . | nindent 4 }}
```

**What's happening:**
- The Service name is auto-generated by our helper function
- If `service.type` is `NodePort`, it optionally sets a `nodePort`
- The `selector` matches pods created by our Deployment

## Step 8 — Create Ingress Template

**File: `mywebapp/templates/ingress.yaml`**

```yaml
# templates/ingress.yaml
# Only created if ingress.enabled = true

{{- if .Values.ingress.enabled -}}
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: {{ include "mywebapp.fullname" . }}
  labels:
    {{- include "mywebapp.labels" . | nindent 4 }}
  {{- with .Values.ingress.annotations }}
  annotations:
    {{- toYaml . | nindent 4 }}
  {{- end }}
spec:
  {{- if .Values.ingress.className }}
  ingressClassName: {{ .Values.ingress.className }}
  {{- end }}

  {{- if .Values.ingress.tls }}
  tls:
    {{- range .Values.ingress.tls }}
    - hosts:
        {{- range .hosts }}
        - {{ . | quote }}
        {{- end }}
      secretName: {{ .secretName }}
    {{- end }}
  {{- end }}

  rules:
    {{- range .Values.ingress.hosts }}
    - host: {{ .host | quote }}
      http:
        paths:
          {{- range .paths }}
          - path: {{ .path }}
            pathType: {{ .pathType }}
            backend:
              service:
                name: {{ include "mywebapp.fullname" $ }}
                port:
                  number: {{ $.Values.service.port }}
          {{- end }}
    {{- end }}
{{- end }}
```

**New concepts here:**

| Syntax | Explanation |
|--------|------------|
| `{{- if .Values.ingress.enabled -}}` | Entire file is skipped if `ingress.enabled = false` |
| `{{- range .Values.ingress.hosts }}` | Loops over each host in the hosts list |
| `{{ . \| quote }}` | Wraps the value in quotes ("mywebapp.local") |
| `{{ include "mywebapp.fullname" $ }}` | `$` refers to the root context (since `.` changed inside `range`) |

> **💡 Why `$` instead of `.`?**
> Inside a `range` or `with` block, `.` (dot) changes to the current item. To access the root context (Values, Chart, Release), use `$`.

## Step 9 — Create ConfigMap Template

**File: `mywebapp/templates/configmap.yaml`**

```yaml
# templates/configmap.yaml
# Creates a ConfigMap from values in config.data

{{- if .Values.config.enabled }}
apiVersion: v1
kind: ConfigMap
metadata:
  name: {{ include "mywebapp.fullname" . }}-config
  labels:
    {{- include "mywebapp.labels" . | nindent 4 }}
data:
  {{- range $key, $value := .Values.config.data }}
  {{ $key }}: {{ $value | quote }}
  {{- end }}
{{- end }}
```

**How this works:**

Given this `values.yaml`:
```yaml
config:
  enabled: true
  data:
    APP_NAME: "My Web App"
    LOG_LEVEL: "info"
```

It renders to:
```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: my-release-mywebapp-config
data:
  APP_NAME: "My Web App"
  LOG_LEVEL: "info"
```

The `range $key, $value := .Values.config.data` iterates over the map, giving us both the key and value.

## Step 10 — Create Helper Templates (_helpers.tpl)

The `_helpers.tpl` file contains **reusable template snippets**. Files starting with `_` are NOT rendered as Kubernetes resources — they're just helper libraries.

**File: `mywebapp/templates/_helpers.tpl`**

```yaml
{{/*
----------------------------------------------------------------------
  _helpers.tpl — Reusable template definitions for the mywebapp chart
----------------------------------------------------------------------
*/}}

{{/*
Expand the name of the chart.
This is used as a base for generating resource names.
*/}}
{{- define "mywebapp.name" -}}
{{- default .Chart.Name .Values.nameOverride | trunc 63 | trimSuffix "-" }}
{{- end }}

{{/*
Create a default fully qualified app name.
We truncate at 63 chars because some Kubernetes name fields are limited to this.
If the release name contains the chart name, don't repeat it.

Example outputs:
  Release: "my-app",  Chart: "mywebapp"  → "my-app-mywebapp"
  Release: "mywebapp", Chart: "mywebapp" → "mywebapp" (no repeat)
  fullnameOverride: "custom-name"        → "custom-name"
*/}}
{{- define "mywebapp.fullname" -}}
{{- if .Values.fullnameOverride }}
{{- .Values.fullnameOverride | trunc 63 | trimSuffix "-" }}
{{- else }}
{{- $name := default .Chart.Name .Values.nameOverride }}
{{- if contains $name .Release.Name }}
{{- .Release.Name | trunc 63 | trimSuffix "-" }}
{{- else }}
{{- printf "%s-%s" .Release.Name $name | trunc 63 | trimSuffix "-" }}
{{- end }}
{{- end }}
{{- end }}

{{/*
Create chart name and version as used by the chart label.
Example: "mywebapp-0.1.0"
*/}}
{{- define "mywebapp.chart" -}}
{{- printf "%s-%s" .Chart.Name .Chart.Version | replace "+" "_" | trunc 63 | trimSuffix "-" }}
{{- end }}

{{/*
Common labels — applied to ALL resources.
These follow the Kubernetes recommended label conventions.
*/}}
{{- define "mywebapp.labels" -}}
helm.sh/chart: {{ include "mywebapp.chart" . }}
{{ include "mywebapp.selectorLabels" . }}
{{- if .Chart.AppVersion }}
app.kubernetes.io/version: {{ .Chart.AppVersion | quote }}
{{- end }}
app.kubernetes.io/managed-by: {{ .Release.Service }}
{{- end }}

{{/*
Selector labels — used in Deployment selectors and Service selectors.
These MUST NOT change after initial deployment (or pods won't match!).
*/}}
{{- define "mywebapp.selectorLabels" -}}
app.kubernetes.io/name: {{ include "mywebapp.name" . }}
app.kubernetes.io/instance: {{ .Release.Name }}
{{- end }}

{{/*
Create the name of the service account to use.
*/}}
{{- define "mywebapp.serviceAccountName" -}}
{{- if .Values.serviceAccount.create }}
{{- default (include "mywebapp.fullname" .) .Values.serviceAccount.name }}
{{- else }}
{{- default "default" .Values.serviceAccount.name }}
{{- end }}
{{- end }}
```

**How helpers work — think of them as functions:**

```
DEFINE:
  {{- define "mywebapp.fullname" -}}
    ... logic ...
  {{- end }}

CALL:
  {{ include "mywebapp.fullname" . }}
```

It's like writing a function and calling it. The `.` passes the current context (all Values, Chart info, etc.)

### Why `trunc 63`?

Kubernetes names have a **63-character limit**. If your release name is really long, we truncate it to stay within limits. `trimSuffix "-"` makes sure we don't end with a dash.

## Step 11 — Create HPA Template

**File: `mywebapp/templates/hpa.yaml`**

```yaml
# templates/hpa.yaml

{{- if .Values.autoscaling.enabled }}
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: {{ include "mywebapp.fullname" . }}

    {{- include "mywebapp.labels" . | nindent 4 }}

    apiVersion: apps/v1
    kind: Deployment
    name: {{ include "mywebapp.fullname" . }}
  minReplicas: {{ .Values.autoscaling.minReplicas }}

          type: Utilization

    {{- end }}
    {{- if .Values.autoscaling.targetMemoryUtilizationPercentage }}
```

## Step 12 — Add Lifecycle Hooks

Helm **hooks** let you run Jobs at specific points in a release lifecycle.

**File: `mywebapp/templates/hooks/pre-install-job.yaml`**

```yaml
# templates/hooks/pre-install-job.yaml
# This Job runs BEFORE the main chart resources are installed

apiVersion: batch/v1
kind: Job
metadata:
  name: {{ include "mywebapp.fullname" . }}-pre-install
  labels:
    {{- include "mywebapp.labels" . | nindent 4 }}
  annotations:
    # These annotations make it a Helm hook
    "helm.sh/hook": pre-install          # When to run: before install
    "helm.sh/hook-weight": "0"           # Order (lower = first)
    "helm.sh/hook-delete-policy": hook-succeeded  # Delete the Job after success
spec:
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: pre-install
          image: busybox
          command:
            - sh
            - -c
            - |
              echo "🔧 Running pre-install checks..."
              echo "Chart: {{ .Chart.Name }}"
              echo "Version: {{ .Chart.Version }}"
              echo "Release: {{ .Release.Name }}"
              echo "Namespace: {{ .Release.Namespace }}"
              echo "✅ Pre-install checks complete!"
  backoffLimit: 1
```

**File: `mywebapp/templates/hooks/post-install-job.yaml`**

```yaml
# templates/hooks/post-install-job.yaml
# This Job runs AFTER the main chart resources are installed

apiVersion: batch/v1
kind: Job
metadata:
  name: {{ include "mywebapp.fullname" . }}-post-install
  labels:
    {{- include "mywebapp.labels" . | nindent 4 }}
  annotations:
    "helm.sh/hook": post-install,post-upgrade  # Runs after install AND upgrade
    "helm.sh/hook-weight": "0"
    "helm.sh/hook-delete-policy": hook-succeeded
spec:
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: post-install
          image: busybox
          command:
            - sh
            - -c
            - |
              echo "🎉 Deployment complete!"
              echo "Application {{ .Chart.Name }} v{{ .Chart.AppVersion }} is now running."
              echo "Release: {{ .Release.Name }}"
              echo ""
              echo "Running post-deployment health check..."
              # In a real scenario, you might:
              # - Run database migrations
              # - Warm up caches
              # - Send a Slack notification
              # - Run smoke tests
              echo "✅ All good!"
  backoffLimit: 1
```

### Available Hook Types

| Hook | When It Runs |
|------|-------------|
| `pre-install` | Before any resources are created |
| `post-install` | After all resources are created |
| `pre-upgrade` | Before an upgrade |
| `post-upgrade` | After an upgrade |
| `pre-delete` | Before deletion |
| `post-delete` | After deletion |
| `pre-rollback` | Before a rollback |
| `post-rollback` | After a rollback |
| `test` | When `helm test` is run |

### Hook Delete Policies

| Policy | What Happens |
|--------|-------------|
| `hook-succeeded` | Delete the hook resource after it succeeds |
| `hook-failed` | Delete the hook resource if it fails |
| `before-hook-creation` | Delete any existing hook before creating a new one |

### Quick Reference — Common Template Functions

```yaml
# ============ STRING FUNCTIONS ============

{{ "hello" | upper }}                  → "HELLO"
{{ "HELLO" | lower }}                  → "hello"
{{ "hello world" | title }}            → "Hello World"
{{ "hello" | quote }}                  → "\"hello\""
{{ "hello" | repeat 3 }}              → "hellohellohello"
{{ "hello world" | replace " " "-" }} → "hello-world"
{{ "  hello  " | trim }}              → "hello"
{{ "hello" | trunc 3 }}               → "hel"
{{ "hello" | contains "ell" }}         → true
{{ "hello" | hasPrefix "hel" }}        → true

# ============ DEFAULT VALUES ============

{{ .Values.missing | default "fallback" }}    → "fallback" (if missing is nil)
{{ empty .Values.x }}                         → true if nil, "", 0, false, [], {}

# ============ TYPE CONVERSION ============

{{ 123 | toString }}                   → "123"
{{ "123" | atoi }}                     → 123 (string to int)
{{ .Values.data | toYaml }}            → Converts to YAML format
{{ .Values.data | toJson }}            → Converts to JSON format

# ============ CONDITIONALS ============

{{ if eq .Values.env "prod" }}
  # production config
{{ else if eq .Values.env "staging" }}
  # staging config
{{ else }}
  # default config
{{ end }}

# Operators: eq, ne, lt, gt, le, ge, and, or, not

# ============ LOOPS ============

# Loop over a list
{{ range .Values.hosts }}
  - {{ . }}
{{ end }}

# Loop with index
{{ range $index, $value := .Values.hosts }}
  {{ $index }}: {{ $value }}
{{ end }}

# Loop over a map
{{ range $key, $val := .Values.config }}
  {{ $key }}: {{ $val }}
{{ end }}

# ============ INDENT FUNCTIONS ============

{{ toYaml .Values.x | indent 4 }}     → Indents (no newline before)
{{ toYaml .Values.x | nindent 4 }}    → Newline + indent
```

## Common Mistakes & Troubleshooting

### ❌ Error: "template: no function 'xyz'"

**Cause:** Typo in template function name.

**Fix:** Double-check the function name. Common ones: `toYaml`, `nindent`, `include`, `default`, `quote`

### ❌ YAML indentation errors

**Cause:** Wrong indent level in templates.

**Fix:**
```yaml
# Wrong — nindent 4 when it should be nindent 8
    labels:
    {{- include "mywebapp.labels" . | nindent 4 }}

# Right
    labels:
      {{- include "mywebapp.labels" . | nindent 6 }}
```

**Debug tip:** Use `helm template` to see the actual rendered output and check indentation!

### ❌ "Error: INSTALLATION FAILED: cannot re-use a name"

**Cause:** A release with that name already exists.

**Fix:**
```bash
# Use a different name, or uninstall first
helm uninstall my-release
helm install my-release ./mywebapp

# Or use upgrade --install (idempotent)
helm upgrade --install my-release ./mywebapp
```

### ❌ Values not being applied

**Cause:** Wrong key path or typo in values.

**Fix:**
```bash
# Check what values are actually being used
helm get values my-release --all

# Test with template first
helm template my-release ./mywebapp --set your.key=value
```

### ❌ "Error: chart requires kubeVersion X"

**Cause:** Your cluster version doesn't match the chart requirement.

**Fix:** Check `kubeVersion` in Chart.yaml and update if needed.
