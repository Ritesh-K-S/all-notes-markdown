## 1. Project Overview

In this project, we're going to build **real AWS infrastructure using code**. No clicking around the AWS console — everything is defined in `.tf` files, versioned in Git, and reproducible.

Here's what we'll create:

| Resource | What It Is | Why We Need It |
|----------|-----------|----------------|
| **VPC** | Virtual Private Cloud — your own isolated network | Everything lives inside a VPC |
| **Subnets** | Public & private subnets across 2 Availability Zones | Public = internet-facing, Private = database/backend |
| **Internet Gateway** | Connects VPC to the internet | So public resources can reach the internet |
| **NAT Gateway** | Lets private resources reach the internet | So private EC2/RDS can download updates |
| **Route Tables** | Traffic rules for subnets | Direct traffic to the right gateway |
| **Security Groups** | Firewall rules | Control what traffic is allowed |
| **EC2 Instances** | Virtual servers | Run our application |
| **RDS (MySQL)** | Managed database | Store application data |
| **S3 Bucket** | Object storage | Store files, logs, backups |

## Step 3 — Configure the Provider

The **provider** tells Terraform which cloud platform to use and how to authenticate.

**File: `main.tf`**

```hcl
# main.tf — Provider configuration and Terraform settings

# ============================================
# TERRAFORM SETTINGS
# ============================================

terraform {
  # Minimum Terraform version required
  required_version = ">= 1.5.0"

  # Provider requirements
  required_providers {
    aws = {
      source  = "hashicorp/aws"    # Official AWS provider
      version = "~> 5.0"           # Use version 5.x (any minor/patch)
    }
  }

  # Backend configuration — where to store state
  # (We'll set up remote state in Step 18)
  # For now, state is stored locally in terraform.tfstate
}

# ============================================
# AWS PROVIDER
# ============================================

provider "aws" {
  region = var.aws_region    # Region from variables.tf

  # Tags applied to ALL resources (very useful for cost tracking!)
  default_tags {
    tags = {
      Project     = var.project_name
      Environment = var.environment
      ManagedBy   = "Terraform"
      Owner       = var.owner
    }
  }
}
```

**What's happening here:**

| Block | Purpose |
|-------|---------|
| `terraform {}` | Global settings — version constraints, backend, required providers |
| `required_version` | Prevents using an old Terraform version that might break things |
| `required_providers` | Which providers we need and from where |
| `provider "aws"` | Configures the AWS provider — region, authentication, default tags |
| `default_tags` | Tags automatically applied to every resource (no more forgetting to tag!) |

> **💡 How does authentication work?**
> Terraform uses the same credentials as AWS CLI. It checks (in order):
> 1. Environment variables (`AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`)
> 2. Shared credentials file (`~/.aws/credentials`)
> 3. IAM instance profile (if running on EC2)
> 4. IAM role (if using assume role)
>
> Since we already ran `aws configure`, Terraform will use those credentials automatically!

## Step 4 — Define Variables

Variables make your code reusable. Instead of hardcoding `us-east-1` everywhere, you use `var.aws_region`.

**File: `variables.tf`**

```hcl
# variables.tf — All variable definitions
# Variables are like function parameters — they make your code reusable!

# ============================================
# GENERAL
# ============================================

variable "aws_region" {
  description = "AWS region to deploy resources in"
  type        = string
  default     = "us-east-1"

  # You can even validate inputs!
  validation {
    condition     = can(regex("^(us|eu|ap|sa|ca|me|af)-(north|south|east|west|central|northeast|southeast)-[0-9]$", var.aws_region))
    error_message = "Must be a valid AWS region (e.g., us-east-1, eu-west-2)."
  }
}

variable "project_name" {
  description = "Name of the project (used in resource names and tags)"
  type        = string
  default     = "my-terraform-project"
}

variable "environment" {
  description = "Environment name (dev, staging, prod)"
  type        = string
  default     = "dev"

  validation {
    condition     = contains(["dev", "staging", "prod"], var.environment)
    error_message = "Environment must be dev, staging, or prod."
  }
}

variable "owner" {
  description = "Owner of the resources (for tagging)"
```

## Step 5 — Create the VPC

A **VPC** (Virtual Private Cloud) is your own isolated network in AWS. Think of it as your own data center in the cloud.

**File: `vpc.tf`**

```hcl
# vpc.tf — Virtual Private Cloud

# ============================================
# VPC
# ============================================

resource "aws_vpc" "main" {
  cidr_block = var.vpc_cidr    # The IP range for this VPC (e.g., 10.0.0.0/16)

  # Enable DNS support — allows instances to resolve domain names
  enable_dns_support   = true

  # Enable DNS hostnames — gives EC2 instances public DNS names
  enable_dns_hostnames = true

  tags = {
    Name = "${var.project_name}-vpc"
    # The ${} syntax is string interpolation
    # Result: "my-terraform-project-vpc"
  }
}
```

**Let's understand what `10.0.0.0/16` means:**

```
10.0.0.0/16
│  │ │ │  │
│  │ │ │  └── /16 = first 16 bits are fixed (the network part)
│  │ │ └── 0 = this can vary (0-255)
│  │ └── 0 = this can vary (0-255)  
│  └── 0 = this is fixed
└── 10 = this is fixed

So our VPC owns all IPs from 10.0.0.0 to 10.0.255.255
That's 65,536 addresses! More than enough for our project.
```

> **💡 CIDR Quick Reference:**
> | CIDR | IPs Available | Use Case |
> |------|--------------|----------|
> | /16 | 65,536 | Large VPC |
> | /20 | 4,096 | Medium subnet |
> | /24 | 256 | Small subnet (most common for subnets) |
> | /28 | 16 | Very small (e.g., NAT Gateway subnet) |

## Step 6 — Create Subnets

Subnets divide your VPC into smaller networks. We create both **public** and **private** subnets.

- **Public subnets:** Resources here CAN be reached from the internet (web servers)
- **Private subnets:** Resources here CANNOT be reached from the internet (databases)

**File: `subnets.tf`**

```hcl
# subnets.tf — Public and Private Subnets

# ============================================
# PUBLIC SUBNETS
# ============================================

# We use "count" to create multiple subnets from a list
# count = length(["10.0.1.0/24", "10.0.2.0/24"]) = 2

resource "aws_subnet" "public" {
  count = length(var.public_subnet_cidrs)    # Creates 2 subnets

  vpc_id            = aws_vpc.main.id                    # Reference the VPC we created
  cidr_block        = var.public_subnet_cidrs[count.index]   # 10.0.1.0/24, then 10.0.2.0/24
  availability_zone = var.availability_zones[count.index]     # us-east-1a, then us-east-1b

  # Auto-assign public IPs to instances launched in this subnet
  map_public_ip_on_launch = true

  tags = {
    Name = "${var.project_name}-public-subnet-${count.index + 1}"
    Type = "Public"
    # Result: "my-terraform-project-public-subnet-1"
    #         "my-terraform-project-public-subnet-2"
  }
}

# ============================================
# PRIVATE SUBNETS
# ============================================

resource "aws_subnet" "private" {
  count = length(var.private_subnet_cidrs)

  vpc_id            = aws_vpc.main.id
  cidr_block        = var.private_subnet_cidrs[count.index]
  availability_zone = var.availability_zones[count.index]

  # Private subnets do NOT auto-assign public IPs
  map_public_ip_on_launch = false

  tags = {
    Name = "${var.project_name}-private-subnet-${count.index + 1}"
    Type = "Private"
  }
}
```

**Understanding `count` and `count.index`:**

```
count = 2 means Terraform creates TWO copies of this resource.

count.index = 0 for the first copy
count.index = 1 for the second copy

So:
  var.public_subnet_cidrs[0] = "10.0.1.0/24"  → in us-east-1a
  var.public_subnet_cidrs[1] = "10.0.2.0/24"  → in us-east-1b
```

> **💡 Why 2 Availability Zones?**
> Availability Zones (AZs) are physically separate data centers. If one goes down (fire, power outage), the other keeps running. This is called **high availability**. Always spread across at least 2 AZs!

## Step 7 — Set Up Internet Gateway & NAT Gateway

**File: `gateways.tf`**

```hcl
# gateways.tf — Internet Gateway & NAT Gateway

# ============================================
# INTERNET GATEWAY
# ============================================
# An Internet Gateway lets resources in public subnets
# communicate with the internet (both inbound and outbound)

resource "aws_internet_gateway" "main" {
  vpc_id = aws_vpc.main.id    # Attach to our VPC

  tags = {
    Name = "${var.project_name}-igw"
  }
}

# Think of it as the "front door" of your VPC — 
# without it, nothing in your VPC can reach the internet!

# ============================================
# ELASTIC IP FOR NAT GATEWAY
# ============================================
# NAT Gateway needs a static public IP (Elastic IP)

resource "aws_eip" "nat" {
  domain = "vpc"    # Allocate in VPC scope

  tags = {
    Name = "${var.project_name}-nat-eip"
  }

  # Ensure the Internet Gateway exists before creating the EIP
  depends_on = [aws_internet_gateway.main]
}

# ============================================
# NAT GATEWAY
# ============================================
# NAT Gateway lets resources in PRIVATE subnets reach the internet
# (outbound only — no one can initiate a connection TO the private subnet)
#
# Use case: Your RDS in a private subnet needs to download security patches,
# but you don't want anyone from the internet to access it directly.

resource "aws_nat_gateway" "main" {
  allocation_id = aws_eip.nat.id              # Use the Elastic IP we created
  subnet_id     = aws_subnet.public[0].id     # NAT Gateway lives in a PUBLIC subnet

  tags = {
    Name = "${var.project_name}-nat-gw"
  }

  # Ensure Internet Gateway exists first
  depends_on = [aws_internet_gateway.main]
}
```

**Internet Gateway vs NAT Gateway:**

```
INTERNET GATEWAY (IGW):                   NAT GATEWAY:
┌──────────────────────┐                  ┌──────────────────────┐
│ Allows:              │                  │ Allows:              │
│  ✅ Outbound traffic │                  │  ✅ Outbound traffic │
│  ✅ Inbound traffic  │                  │  ❌ Inbound traffic  │
│                      │                  │                      │
│ Used by:             │                  │ Used by:             │
│  Public subnets      │                  │  Private subnets     │
│  (web servers, etc.) │                  │  (databases, etc.)   │
└──────────────────────┘                  └──────────────────────┘
```

> **💡 Cost alert:** NAT Gateways cost ~$0.045/hour (~$32/month) + data processing charges. For a dev/test environment, you might skip the NAT Gateway to save costs. Private instances just won't be able to reach the internet.

## Step 8 — Create Route Tables

Route tables tell your subnets **where to send traffic**. Think of them as a GPS for network traffic.

**File: `route-tables.tf`**

```hcl
# route-tables.tf — Route Tables & Associations

# ============================================
# PUBLIC ROUTE TABLE
# ============================================
# "If the traffic is going to the internet (0.0.0.0/0),
#  send it through the Internet Gateway"

resource "aws_route_table" "public" {
  vpc_id = aws_vpc.main.id

  # Route all internet traffic (0.0.0.0/0) to the Internet Gateway
  route {
    cidr_block = "0.0.0.0/0"                        # All traffic
    gateway_id = aws_internet_gateway.main.id        # → goes to IGW
  }

  # Local routes (traffic within the VPC) are added automatically!
  # 10.0.0.0/16 → local (implicit, no need to define)

  tags = {
    Name = "${var.project_name}-public-rt"
  }
}

# Associate public subnets with the public route table
resource "aws_route_table_association" "public" {
  count = length(var.public_subnet_cidrs)

  subnet_id      = aws_subnet.public[count.index].id
  route_table_id = aws_route_table.public.id
}

# ============================================
# PRIVATE ROUTE TABLE
# ============================================
# "If the traffic is going to the internet (0.0.0.0/0),
#  send it through the NAT Gateway (one-way only!)"

resource "aws_route_table" "private" {
  vpc_id = aws_vpc.main.id

  # Route internet-bound traffic through the NAT Gateway
  route {
    cidr_block     = "0.0.0.0/0"                     # All traffic
    nat_gateway_id = aws_nat_gateway.main.id          # → goes to NAT GW
  }

  tags = {
    Name = "${var.project_name}-private-rt"
  }
}

# Associate private subnets with the private route table
resource "aws_route_table_association" "private" {
  count = length(var.private_subnet_cidrs)

  subnet_id      = aws_subnet.private[count.index].id
  route_table_id = aws_route_table.private.id
}
```

**How routing works — a visual:**

```
EC2 in public subnet wants to reach google.com:

EC2 (10.0.1.50) → Route Table says: "0.0.0.0/0 → Internet Gateway"
                → Internet Gateway → google.com ✅

RDS in private subnet wants to download updates:

RDS (10.0.3.25) → Route Table says: "0.0.0.0/0 → NAT Gateway"
                → NAT Gateway (in public subnet) → Internet Gateway → internet ✅

Someone from internet tries to reach RDS:

Internet → Internet Gateway → ??? 
    No route to 10.0.3.25 from outside → BLOCKED ❌
```

## Step 9 — Create Security Groups

Security Groups are **virtual firewalls** that control what traffic can come in (ingress) and go out (egress) of your resources.

**File: `security-groups.tf`**

```hcl
# security-groups.tf — Security Groups (Firewalls)

# ============================================
# EC2 SECURITY GROUP
# ============================================
# Controls what traffic can reach our EC2 instances

resource "aws_security_group" "ec2" {
  name        = "${var.project_name}-ec2-sg"
  description = "Security group for EC2 web servers"
  vpc_id      = aws_vpc.main.id

  # --- INGRESS RULES (Inbound) ---

  # Allow HTTP from anywhere
  ingress {
    description = "HTTP from anywhere"
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]    # From any IP
  }

  # Allow HTTPS from anywhere
  ingress {
    description = "HTTPS from anywhere"
    from_port   = 443
    to_port     = 443
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  # Allow SSH from anywhere (⚠️ In production, restrict to your IP!)
  ingress {
    description = "SSH access"
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]    # In production, use YOUR IP: ["203.0.113.50/32"]
  }

  # --- EGRESS RULES (Outbound) ---

  # Allow ALL outbound traffic
  egress {
    description = "Allow all outbound"
    from_port   = 0
    to_port     = 0
    protocol    = "-1"             # -1 means all protocols
    cidr_blocks = ["0.0.0.0/0"]
  }

  tags = {
    Name = "${var.project_name}-ec2-sg"
  }
}

# ============================================
# RDS SECURITY GROUP
# ============================================
# Controls what can access our database
# Key: ONLY allow connections from EC2 instances!

resource "aws_security_group" "rds" {
  name        = "${var.project_name}-rds-sg"
  description = "Security group for RDS database"
  vpc_id      = aws_vpc.main.id

  # Allow MySQL connections ONLY from EC2 security group
  ingress {
    description     = "MySQL from EC2 instances"
    from_port       = 3306
    to_port         = 3306
    protocol        = "tcp"
    security_groups = [aws_security_group.ec2.id]    # ← Only from EC2 SG!
    # This is much more secure than allowing from any IP!
  }

  # Allow outbound traffic
  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }

  tags = {
    Name = "${var.project_name}-rds-sg"
  }
}
```

**Security Group Rules Explained:**

```
EC2 Security Group:
  INBOUND:
    Port 80  (HTTP)  ← Anyone on the internet can access
    Port 443 (HTTPS) ← Anyone on the internet can access
    Port 22  (SSH)   ← Anyone can SSH (restrict in production!)
  OUTBOUND:
    All ports ← EC2 can connect to anything

RDS Security Group:
  INBOUND:
    Port 3306 (MySQL) ← ONLY from EC2 instances (identified by SG)
  OUTBOUND:
    All ports ← RDS can connect to anything

This means:
  ✅ Internet → EC2 (port 80, 443) 
  ✅ EC2 → RDS (port 3306)
  ❌ Internet → RDS (BLOCKED — no rule allows this!)
```

> **💡 Best practice:** Always use security group references (`security_groups = [aws_security_group.ec2.id]`) instead of IP ranges for internal communication. This is more dynamic and secure.

## Step 10 — Launch EC2 Instances

**File: `ec2.tf`**

```hcl
# ec2.tf — EC2 Instances

# ============================================
# FIND THE LATEST AMAZON LINUX 2023 AMI
# ============================================
# Instead of hardcoding an AMI ID (which changes per region),
# we dynamically look up the latest one!

data "aws_ami" "amazon_linux" {
  most_recent = true
  owners      = ["amazon"]    # Only AMIs owned by Amazon

  filter {
    name   = "name"
    values = ["al2023-ami-*-x86_64"]    # Amazon Linux 2023, 64-bit
  }

  filter {
    name   = "virtualization-type"
    values = ["hvm"]    # Hardware Virtual Machine (standard)
  }

  filter {
    name   = "state"
    values = ["available"]
  }
}

# ============================================
# EC2 INSTANCES
# ============================================

resource "aws_instance" "web" {
  count = var.instance_count    # Create 2 instances (one per AZ)

  ami           = data.aws_ami.amazon_linux.id     # The AMI we looked up above
  instance_type = var.instance_type                 # t2.micro (free tier)

  # Place each instance in a different public subnet (different AZ)
  subnet_id = aws_subnet.public[count.index % length(aws_subnet.public)].id
  # count.index % length → distributes instances across subnets
  # Instance 0 → subnet[0] (us-east-1a)
  # Instance 1 → subnet[1] (us-east-1b)

  # Attach security group
  vpc_security_group_ids = [aws_security_group.ec2.id]

  # SSH key pair (optional — only if you want SSH access)
  key_name = var.key_name != "" ? var.key_name : null

  # User data script — runs when the instance first boots
  user_data = base64encode(<<-EOF
    #!/bin/bash
    # Update the system
    yum update -y

    # Install Nginx
    yum install -y nginx

    # Create a custom homepage
    cat > /usr/share/nginx/html/index.html << 'HTMLEOF'
    <!DOCTYPE html>
    <html>
    <head><title>Terraform Demo</title></head>
    <body>
      <h1>Hello from EC2! 🚀</h1>
      <p>This server was provisioned by <strong>Terraform</strong>.</p>
      <p>Instance: $(hostname)</p>
      <p>AZ: $(curl -s http://169.254.169.254/latest/meta-data/placement/availability-zone)</p>
      <p>Time: $(date)</p>
    </body>
    </html>
    HTMLEOF

    # Start Nginx and enable it on boot
    systemctl start nginx
    systemctl enable nginx
  EOF
  )

  # Root volume (disk)
  root_block_device {
    volume_size = 20           # 20 GB
    volume_type = "gp3"        # General Purpose SSD (latest generation)
    encrypted   = true         # Always encrypt volumes!

    tags = {
      Name = "${var.project_name}-ec2-volume-${count.index + 1}"
    }
  }

  tags = {
    Name = "${var.project_name}-web-${count.index + 1}"
    Role = "web-server"
  }
}
```

**Understanding the `data` block:**

```hcl
data "aws_ami" "amazon_linux" { ... }
```

This is a **data source** — it reads information from AWS without creating anything. It's like running `aws ec2 describe-images` to find the latest AMI. The result is used in `data.aws_ami.amazon_linux.id`.

**Understanding `user_data`:**

User data is a script that runs **once** when the EC2 instance first boots. It's how you automatically install software, configure services, etc. without needing to SSH in manually. The `base64encode()` function is required because AWS expects user data in base64 format.

> **💡 Pro tip:** For anything more complex than a few lines, use Ansible or a configuration management tool instead of user_data. Long user_data scripts are hard to debug.

```hcl
# rds.tf — RDS MySQL Database

# ============================================
# DB SUBNET GROUP
# ============================================
# RDS needs to know which subnets it can use.

resource "aws_db_subnet_group" "main" {

# RDS MYSQL INSTANCE

  # --- Instance Configuration ---

  engine         = var.db_engine                      # "mysql"
  engine_version = var.db_engine_version              # "8.0"
  instance_class = var.db_instance_class              # "db.t3.micro" (free tier)

  # --- Storage ---
  allocated_storage     = var.db_allocated_storage    # 20 GB
  max_allocated_storage = 100                         # Auto-scale up to 100 GB
  storage_type          = "gp3"                       # General Purpose SSD
  storage_encrypted     = true                        # Always encrypt!

  # --- Database ---
  db_name  = var.db_name        # Create a database called "myappdb"
  username = var.db_username    # Master username
  password = var.db_password    # Master password (from tfvars, marked sensitive)

  # --- Networking ---
  db_subnet_group_name   = aws_db_subnet_group.main.name
  vpc_security_group_ids = [aws_security_group.rds.id]
  publicly_accessible    = false    # NOT accessible from internet!
  port                   = 3306

  # --- High Availability ---
  multi_az = var.environment == "prod" ? true : false
  # Multi-AZ only in production (creates a standby in another AZ)
  # In dev, single-AZ is fine (and cheaper)

  # --- Backup ---
  backup_retention_period = 7              # Keep backups for 7 days
  backup_window           = "03:00-04:00"  # Backup between 3-4 AM UTC
  maintenance_window      = "Mon:04:00-Mon:05:00"  # Maintenance window

  # --- Deletion Protection ---
  deletion_protection = var.environment == "prod" ? true : false
  # Prevent accidental deletion in production!

  skip_final_snapshot       = var.environment == "prod" ? false : true
  final_snapshot_identifier = var.environment == "prod" ? "${var.project_name}-final-snapshot" : null
  # Take a final snapshot before deletion in production

  # --- Performance ---
  performance_insights_enabled = true
  # Free tier includes Performance Insights!

  # --- Parameters ---
  parameter_group_name = "default.mysql8.0"

  tags = {
    Name = "${var.project_name}-mysql"
  }
}
```

**Let's break down the key decisions:**

| Setting | Dev Value | Prod Value | Why |
|---------|-----------|-----------|-----|
| `multi_az` | false | true | HA in prod, save money in dev |
| `deletion_protection` | false | true | Prevent accidents in prod |
| `skip_final_snapshot` | true | false | Always snapshot before deleting in prod |
| `backup_retention` | 7 days | 7+ days | Point-in-time recovery |

> **💡 The `? :` syntax is a ternary (conditional) operator:**
> ```hcl
> condition ? value_if_true : value_if_false
> 
> multi_az = var.environment == "prod" ? true : false
> # If environment is "prod", multi_az = true
> # Otherwise, multi_az = false
> ```

> **⚠️ Security note:** The `db_password` is marked as `sensitive = true` in variables.tf. Terraform won't show it in plan output or logs. But it IS stored in the state file — that's why we use remote state with encryption (Step 18)!

## Step 12 — Create the S3 Bucket

**File: `s3.tf`**

```hcl
# s3.tf — S3 Bucket for application files

# ============================================
# S3 BUCKET
# ============================================

resource "aws_s3_bucket" "main" {
  bucket = var.s3_bucket_name    # Must be globally unique across ALL of AWS!

  # Prevent accidental deletion of this bucket
  force_destroy = var.environment == "prod" ? false : true
  # In dev: allow force destroy (easier cleanup)
  # In prod: prevent accidental deletion

  tags = {
    Name = "${var.project_name}-bucket"
  }
}

# ============================================
# BUCKET VERSIONING
# ============================================
# Keep old versions of files — great for recovery!

resource "aws_s3_bucket_versioning" "main" {
  bucket = aws_s3_bucket.main.id

  versioning_configuration {
    status = "Enabled"
  }
}

# ============================================
# SERVER-SIDE ENCRYPTION
# ============================================
# Encrypt all objects at rest (stored on disk)

resource "aws_s3_bucket_server_side_encryption_configuration" "main" {
  bucket = aws_s3_bucket.main.id

  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "AES256"    # Free AWS-managed encryption
    }
    bucket_key_enabled = true      # Reduces KMS costs if using KMS
  }
}

# ============================================
# BLOCK PUBLIC ACCESS
# ============================================
# VERY IMPORTANT! Block all public access by default.
# This prevents data leaks — one of the most common S3 misconfigurations!

resource "aws_s3_bucket_public_access_block" "main" {
  bucket = aws_s3_bucket.main.id

  block_public_acls       = true    # Block public ACLs
  block_public_policy     = true    # Block public bucket policies
  ignore_public_acls      = true    # Ignore existing public ACLs
  restrict_public_buckets = true    # Restrict public bucket policies
}

# ============================================
# LIFECYCLE RULES
# ============================================
# Automatically manage objects to save costs

resource "aws_s3_bucket_lifecycle_configuration" "main" {
  bucket = aws_s3_bucket.main.id

  rule {
    id     = "move-to-ia"
    status = "Enabled"

    # Move objects to Infrequent Access after 30 days
    transition {
      days          = 30
      storage_class = "STANDARD_IA"    # Cheaper storage for less-accessed files
    }

    # Move to Glacier after 90 days (very cheap, slow retrieval)
    transition {
      days          = 90
      storage_class = "GLACIER"
    }

    # Delete after 365 days
    expiration {
      days = 365
    }
  }

  rule {
    id     = "delete-old-versions"
    status = "Enabled"

    # Delete old file versions after 30 days
    noncurrent_version_expiration {
      noncurrent_days = 30
    }
  }
}
```

**S3 Storage Classes & Costs (approximate):**

```
Storage Class      │ Cost/GB/Month │ Best For
───────────────────┼───────────────┼──────────────────────────
Standard           │ $0.023        │ Frequently accessed files
Standard-IA        │ $0.0125       │ Less frequent (>30 days old)
Glacier Instant    │ $0.004        │ Rare access, instant retrieval
Glacier Flexible   │ $0.0036       │ Rare access, minutes-hours retrieval
Glacier Deep       │ $0.00099      │ Archive, 12-48 hour retrieval
```

> **💡 The lifecycle rules automatically move files to cheaper storage as they age. This is a huge cost saver in production!**


**This is like a "dry run."** It shows you everything Terraform will create, modify, or destroy:

```

      + cidr_block           = "10.0.0.0/16"
      + enable_dns_hostnames = true
      + enable_dns_support   = true
      + id                   = (known after apply)

          + "Name" = "my-terraform-project-vpc"
        }
    }

  # aws_subnet.public[0] will be created
  + resource "aws_subnet" "public" {
      ...
    }

```

**Reading the plan:**
- `+` means CREATE (new resource)
- `~` means UPDATE (modify existing resource)
- `-` means DESTROY (delete resource)
- `-/+` means REPLACE (destroy and recreate)

## Step 18 — Remote State with S3 Backend

By default, Terraform stores state in a local file (`terraform.tfstate`). For teams, you need **remote state** so everyone shares the same state.

### 18.1 — Create the Backend Bucket (one-time setup)

```bash
# Create the S3 bucket for state (do this manually or with a separate Terraform config)
aws s3api create-bucket \
  --bucket my-terraform-state-bucket-12345 \
  --region us-east-1

# Enable versioning (so you can recover old state files)
aws s3api put-bucket-versioning \
  --bucket my-terraform-state-bucket-12345 \
  --versioning-configuration Status=Enabled

# Enable encryption
aws s3api put-bucket-encryption \
  --bucket my-terraform-state-bucket-12345 \
  --server-side-encryption-configuration '{"Rules":[{"ApplyServerSideEncryptionByDefault":{"SSEAlgorithm":"AES256"}}]}'

# Create DynamoDB table for state locking (prevents concurrent modifications)
aws dynamodb create-table \
  --table-name terraform-state-lock \
  --attribute-definitions AttributeName=LockID,AttributeType=S \
  --key-schema AttributeName=LockID,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST \
  --region us-east-1
```

### 18.2 — Configure the Backend

Update `main.tf`:

```hcl
terraform {
  required_version = ">= 1.5.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }

  # Remote state configuration
  backend "s3" {
    bucket         = "my-terraform-state-bucket-12345"    # S3 bucket name
    key            = "dev/terraform.tfstate"               # Path inside bucket
    region         = "us-east-1"
    encrypt        = true                                  # Encrypt state file
    dynamodb_table = "terraform-state-lock"                # Lock table
  }
}
```

### 18.3 — Migrate State

```bash
# Re-initialize to migrate from local to remote state
terraform init -migrate-state
```

```
Terraform will ask:
Do you want to copy existing state to the new backend? → yes
```

Now your state is safely stored in S3 with:
- **Encryption** — state is encrypted at rest
- **Versioning** — you can recover previous states
- **Locking** — prevents two people from applying at the same time

## Common Mistakes & Troubleshooting

### ❌ Error: "Error creating VPC: VpcLimitExceeded"

**Cause:** AWS has a default limit of 5 VPCs per region.

**Fix:** Delete unused VPCs, or request a limit increase in the AWS Console.

### ❌ Error: "InvalidParameterValue: Value for DB Instance Identifier is invalid"

**Cause:** RDS names can only contain lowercase letters, numbers, and hyphens.

**Fix:** Change the identifier to use lowercase and hyphens only.

### ❌ Error: "Bucket name already exists"

**Cause:** S3 bucket names are globally unique across ALL AWS accounts.

**Fix:** Add random numbers or your account ID to the bucket name.

### ❌ Error: "Error launching source instance: UnauthorizedOperation"

**Cause:** Your AWS credentials don't have permission.

**Fix:** Ensure your IAM user/role has the necessary policies (EC2, VPC, RDS, S3).

### ❌ State file is out of sync

**Cause:** Someone made changes in the console.

**Fix:**
```bash
# Refresh state to match reality
terraform refresh

# Or import an existing resource
terraform import aws_vpc.main vpc-0abc123
```

### ❌ "Error: Cycle" (circular dependency)

**Cause:** Resource A depends on B, and B depends on A.

**Fix:** Restructure your resources or use `depends_on` explicitly.

## Terraform Commands Cheat Sheet

### Core Workflow

```bash
terraform init                    # Download providers & initialize
terraform validate                # Check syntax
terraform fmt                     # Format code
terraform plan                    # Preview changes
terraform apply                   # Create/update resources
terraform destroy                 # Destroy everything
```

### State Management

```bash
terraform state list              # List all resources in state
terraform state show <resource>   # Show details of one resource
terraform state mv <old> <new>    # Rename a resource in state
terraform state rm <resource>     # Remove from state (doesn't destroy!)
terraform import <resource> <id>  # Import existing resource into state
terraform refresh                 # Sync state with real infrastructure
```

### Workspace Management

```bash
terraform workspace list          # List workspaces
terraform workspace new dev       # Create "dev" workspace
terraform workspace select dev    # Switch to "dev" workspace
terraform workspace show          # Show current workspace
```

### Other Useful Commands

```bash
terraform output                  # Show all outputs
terraform output -json            # Outputs as JSON
terraform graph | dot -Tpng > graph.png  # Visualize dependencies
terraform providers               # Show required providers
terraform version                 # Show Terraform version
```
